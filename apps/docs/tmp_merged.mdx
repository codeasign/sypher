## Summary

Binary Search on Answer solves a different kind of problem: instead of searching for a value hiding in a list, you search for the best number that satisfies some rule тАФ here, the slowest eating speed that still lets Koko finish all her banana piles in time. The trick is to guess a speed, check if it works, and use that yes/no answer to guess higher or lower next time, cutting the range of possible speeds in half each round, just like guessing a number between 1 and 100. It's much faster than trying every possible speed one at a time.

## Pseudocode

```text
1. canFinish(piles, hours, speed): check if eating at this speed finishes all piles within the hour limit.
   1a. Add up the hours needed for each pile at this speed (round up any partial hour).
   1b. If the total ever goes over the hour limit, this speed is too slow тАФ report "no."
   1c. Otherwise, report "yes."
2. Set the slowest possible speed to 1, and the fastest possible speed to the size of the biggest pile.
3. While the slow guess is still less than the fast guess:
4.     Guess the middle speed between them.
5.     Check with canFinish whether that middle speed works.
6.     If it works, it might be possible to go even slower тАФ lower the fast guess to this middle speed.
7.     If it doesn't work, you need to go faster тАФ raise the slow guess just above this middle speed.
8. When the two guesses meet, that number is the slowest speed that still works.
```

### Recursive Version

```text
1. search(low, high, piles, hours):
2.     If low is no longer less than high, there's only one candidate speed left тАФ return low.
3.     Guess the middle speed between low and high.
4.     Check with canFinish whether that middle speed works.
5.     If it works, call search again with the same low but high lowered to the middle speed.
6.     If it doesn't work, call search again with low raised just above the middle speed and the same high.
7. Call search(1, size of the biggest pile, piles, hours) to get the final answer.
```

## Worked Example

This only works because the yes/no check is "one-directional": if a speed is fast enough to finish in time, every faster speed also finishes in time, and if a speed is too slow, every slower speed is too slow too. That means as you slide from slow to fast, the answer flips from "no" to "yes" exactly once тАФ which is what lets you throw away half the guesses every round. Take piles `[3, 6, 7, 11]` with an `8`-hour limit, and find the slowest eating speed that still finishes in time.

## Complexity

- **Best case:** if the range of candidate answers only has one number in it to begin with, the loop never runs and that number is the answer immediately тАФ effectively instant beyond a single feasibility check (`O(1)`).
- **Average case:** every round throws away half of whatever candidate answers are left, but before it can decide which half to keep, it has to run a full feasibility check тАФ like adding up hours for every pile. So the total work is "how expensive one feasibility check is" multiplied by "how many times the range of candidates can be cut in half" (`O(c log m)`, where `c` is the cost of one feasibility check and `m` is the width of the range of candidate answers).
- **Worst case:** there's no early exit here тАФ the loop always keeps narrowing until only one candidate is left, so it always takes the full `O(c log m)`.
- **Space:** only a low, mid, and high guess are kept, no matter how wide the range of candidate answers is тАФ a small, constant amount of extra memory (`O(1)`), on top of whatever memory the feasibility check itself needs.

## Applications

- **Shipping and logistics capacity planning** тАФ the classic "minimum ship or truck capacity to deliver all packages within D days" family of problems used in warehouse and freight scheduling.
- **Cloud autoscaling** тАФ picking the smallest instance size or replica count that still keeps request latency under a target service-level agreement during a load test.
- **Adaptive bitrate video streaming** тАФ choosing the lowest bitrate that still keeps a video player's buffer from running dry within a playback time budget.
- **Parametric search in computational geometry and optimization**, a technique named by Nimrod Megiddo, that turns "find the optimal value" problems into repeated yes/no feasibility checks over a range of candidate values, exactly like this pattern.
- **Competitive programming judges** like LeetCode and Codeforces classify this as "binary search on the answer," a named technique for any problem with a monotonic feasibility check тАФ one bad guess in a direction means every guess further in that direction is also bad.
