## Summary

Heap Sort imagines the list as a pyramid shape, called a heap, where every "parent" number is bigger than its two "children" numbers below it. Because of that rule, the single biggest number always ends up at the very top of the pyramid. Heap Sort keeps pulling the top (biggest) number off, putting it at the end of the sorted list, and then patching the pyramid back into shape — over and over until nothing is left. It's dependably medium-fast no matter what order the numbers started in.

## Pseudocode

```text
1. Arrange the numbers into a pyramid shape where every parent is bigger than its two children — this is called building the heap.
2. Take the number at the very top of the pyramid (it's always the biggest one left) and swap it with the last number still in the pyramid.
3. Set that last spot aside — it now holds a correctly placed, sorted number, and shrink the pyramid by one.
4. Fix the pyramid: let the number that just moved to the top sink down by swapping with its biggest child, repeating until every parent is bigger than its children again.
5. Repeat steps 2-4 until the pyramid has no numbers left in it — the list is now fully sorted from smallest to biggest.
```

## Worked Example

We'll trace `[6, 3, 9, 2, 8, 1]` all the way through: first turning it into a pyramid, then pulling the top number off five times until it's fully sorted.

Laid out as a tree, index 0 is the root, and each parent at index `i` has children at `2i + 1` and `2i + 2`. Right now `3` sits above `8`, which breaks the heap rule that every parent must be bigger than its children. Heap Sort works entirely inside this same list — it never copies the numbers into a second list — which is why it's called an **in-place** sort.

Building the heap works from the bottom of the tree upward. First `3` and its child `8` swap because `8` is bigger, giving `[6, 8, 9, 2, 3, 1]`. Then `6` and its child `9` swap because `9` is bigger, giving `[9, 8, 6, 2, 3, 1]`. Now every parent is bigger than its children, so `9` — the biggest number in the whole list — sits on top, ready to be pulled off.

The last two numbers, `2` and `1`, swap, and with only one number left in the heap there's nothing left to sink. The list is now fully sorted: `[1, 2, 3, 6, 8, 9]`. Notice Heap Sort never guarantees that equal numbers keep their original order — the swapping and sinking can shuffle them past each other, which is why Heap Sort is **not stable**.

## Complexity

- **Best case:** even if the list happens to already be sorted, Heap Sort still has to build the pyramid and sink numbers back into place after every extraction, so it doesn't get to skip much work (`O(n log n)`).
- **Average case:** on a typically shuffled list, it's the same story — build the pyramid once, then pull the top number off and re-fix the pyramid for every remaining number, and each fix takes about `log n` swaps (`O(n log n)`).
- **Worst case:** no starting order makes Heap Sort slower than this, because sinking a number back into place after each extraction only ever takes about `log n` steps, no matter how the list started (`O(n log n)`).
- **Space:** unlike Merge Sort, which needs a whole second list to merge into, Heap Sort only swaps numbers around inside the same list you gave it — no second list is ever built, so it needs no extra memory beyond a couple of temporary variables (`O(1)`).

## Applications

- **Priority queues** — the "always know the biggest (or smallest) item without re-scanning everything" trick behind a heap is exactly how priority queues are built, used in things like task schedulers and Dijkstra's shortest-path algorithm.
- **Finding the k largest or smallest items** in a big pile of data without sorting the whole thing — leaderboard "top scores" and "top-k" style queries lean on this.
- **Guaranteed worst-case performance with no extra memory** — systems that can't risk Quick Sort's rare `O(n²)` slowdown and can't spare Merge Sort's extra memory reach for Heap Sort instead.
- **Introsort**, the hybrid sort behind C++'s `std::sort` and .NET's `Array.Sort`, falls back to Heap Sort whenever Quick Sort's recursion goes too deep — that fallback guarantees the sort still finishes in `O(n log n)` even on Quick Sort's worst-case inputs.
- **"Next most urgent task" scheduling** — operating system kernels and job queues that repeatedly need to grab the highest-priority waiting task use heap-based structures for this.

<CodeBlock language="python">{`def heap_sort(array: list[int]) -> list[int]:
    n = len(array)

    def sift_down(heap_size: int, i: int) -> None:
        largest = i
        left = 2 * i + 1
        right = 2 * i + 2

        if left < heap_size and array[left] > array[largest]:
            largest = left
        if right < heap_size and array[right] > array[largest]:
            largest = right

        if largest != i:
            array[i], array[largest] = array[largest], array[i]
            sift_down(heap_size, largest)

    for i in range(n // 2 - 1, -1, -1):
        sift_down(n, i)

    for end in range(n - 1, 0, -1):
        array[0], array[end] = array[end], array[0]
        sift_down(end, 0)

    return array`}</CodeBlock>
