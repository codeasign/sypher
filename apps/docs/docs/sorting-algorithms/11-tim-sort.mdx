## Summary

Tim Sort is a "best of both worlds" method that mixes two other ideas from this course: it finds small stretches of the list that are already in order, tidies up any short stretch that isn't, and then merges all those stretches together the same way Merge Sort does. It's not just a classroom idea — it's the real method used behind the scenes by many popular programming languages' built-in "sort" button, including Python and Java, because it's genuinely fast on real-world data. It's reliably quick even in the worst case, and often much faster than that when the data already has some order to it.

## Pseudocode

```text
1. Look through the list and find short stretches that are already in
   increasing order — call each one a "run."
2. If a stretch isn't already in order (or is too short), tidy it up using
   Insertion Sort (the same one-at-a-time method from an earlier module) so
   it becomes one clean, sorted run.
3. Now you have a collection of sorted runs. Merge two runs together at a
   time, the same way Merge Sort does: compare the front of each run, take
   the smaller one, repeat until both runs are used up.
4. If one run keeps "winning" the comparison over and over, skip ahead and
   copy a whole block from it at once instead of comparing one item at a
   time — this speeds things up when one run is clearly running ahead.
5. Keep merging runs together, two at a time, until only one big sorted run
   is left. That's your sorted list.
```

## Worked Example

Follow one list through the whole process: `[5, 8, 3, 1, 9, 5, 7, 2]`. It has eight numbers and a repeated value (`5` appears twice), which is enough to show every stage — splitting into runs, tidying each run, and merging.

Neither half is naturally in order, so Tim Sort treats each half as its own "run" of four elements — run A on the left, run B on the right — and fixes each one up separately before it worries about combining them.

insert 1 at the very front ──▶ [1, 3, 5, 8]
────────────────────────────────────────
run A after: [1, 3, 5, 8]`}
/>

Run A isn't sorted, so Tim Sort tidies it up with Insertion Sort — the same one-at-a-time method from an earlier module, where each number is slid into its correct spot among the numbers already placed. After three insertions, run A becomes a clean sorted run: `1, 3, 5, 8`.

<AsciiDiagram
  id="sorting-algorithms/tim-sort/step-3"
  alt="Run B nine five seven two being insertion sorted one number at a time into the sorted order two five seven nine"
  caption="Step 3: insertion-sort run B"
  content={`run B before: [9, 5, 7, 2]
────────────────────────────────────────
insert 5 before 9 ──▶ [5, 9, 7, 2]
insert 7 between 5 and 9 ──▶ [5, 7, 9, 2]
insert 2 at the very front ──▶ [2, 5, 7, 9]
────────────────────────────────────────
run B after: [2, 5, 7, 9]`}
/>

Run B gets the same treatment. Once both runs are sorted on their own, there's nothing left to do except combine them — this is the "merge" half of Tim Sort, and it works exactly like Merge Sort.

<AsciiDiagram
  id="sorting-algorithms/tim-sort/step-4"
  alt="Run A one three five eight and run B two five seven nine being merged element by element, comparing the front of each run and taking the smaller value each time, with a tie at the two matching fives resolved in favor of the left run"
  caption="Step 4: merge run A and run B"
  content={`run A: [1, 3, 5, 8]      run B: [2, 5, 7, 9]
────────────────────────────────────────
1 vs 2 ──▶ 1 wins (from A)
3 vs 2 ──▶ 2 wins (from B)
3 vs 5 ──▶ 3 wins (from A)
5 vs 5 ──▶ tie, left run wins ──▶ 5 from A
8 vs 5 ──▶ 5 wins (from B)
8 vs 7 ──▶ 7 wins (from B)
8 vs 9 ──▶ 8 wins (from A)
(run A used up) ──▶ add what's left: 9
────────────────────────────────────────
merged so far: [1, 2, 3, 5, 5, 7, 8, 9]`}
/>

Merging compares the front of each run and always takes the smaller value. When both runs offer the same value — the two `5`s — Tim Sort always takes from the left run first. That rule is what makes it a **stable** sort: equal values never swap places relative to each other. This step also isn't **in-place**: building the merged list needs a separate temporary list to hold the combined result while it's being built, rather than rearranging the original list where it sits.

<AsciiDiagram
  id="sorting-algorithms/tim-sort/step-5"

/>

Every number now sits in its final position, and the two `5`s stayed in the same left-to-right order they started in — the mark of a stable sort. The whole list is sorted: `1, 2, 3, 5, 5, 7, 8, 9`.

## Complexity

- **Best:** If the list is already sorted, or made up of a few long naturally sorted stretches, Tim Sort barely has to do anything beyond spotting those runs and stitching them together (`O(n)`). This is much better than plain Merge Sort's best case — plain Merge Sort always splits and merges everything regardless of existing order, while Tim Sort checks for order first and skips the work it doesn't need.
- **Average:** On typical, randomly shuffled data, it behaves like Merge Sort: split into runs, tidy each one up, then merge them together, all in about (`O(n log n)`).
- **Worst:** Even with a list that has no helpful order in it at all, Tim Sort still finishes in (`O(n log n)`) — the same guarantee as Merge Sort. It never slows down to the (`O(n²)`) pace that plain Insertion Sort can hit on bad input.
- **Space:** Merging runs needs a second temporary list to build the combined result in, so Tim Sort needs extra memory proportional to the size of the list (`O(n)`).

## Applications

- Python's built-in `sorted()` and `list.sort()` — CPython's actual sorting algorithm is Tim Sort, named after its author, Tim Peters.
- Java's `Collections.sort()` and `Arrays.sort()` for objects (arrays of primitives like `int[]` use a different algorithm instead).
- Android's SDK sorting utilities, since Android apps run on the same Java standard library implementation.
- The V8 engine that powers Google Chrome and Node.js uses Tim Sort for `Array.prototype.sort()`.
