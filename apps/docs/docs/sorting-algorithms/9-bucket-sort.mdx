## Summary

Bucket Sort sorts a list by first tossing each number into one of a few rough groups, kind of like sorting mail by zip code region before a mail carrier deals with any of it individually. Once the numbers are split into these small groups ("buckets"), it tidies up each group on its own using a simple method, then lines the groups up in order to read out the final sorted list. This only works well when the numbers are spread out fairly evenly across the range — if too many numbers land in the same group, that group turns into one big pile again, and sorting it takes a while.

## Pseudocode

```text
1. If the list is empty, you're done — return an empty list.
2. Find the biggest number in the list.
3. Decide how many buckets to use — use one bucket for every number in the list.
4. For each number, work out which bucket it belongs in: multiply the number by
   the bucket count, then divide by (biggest number + 1). This sends small
   numbers to early buckets and large numbers to later buckets.
5. Drop each number into its bucket.
6. Go through the buckets one at a time and tidy up the few numbers inside each
   one (using a simple method called insertion sort — build up a sorted mini-list
   one number at a time).
7. Read the buckets out in order, front to back — that's your sorted list.
```

## Worked Example

We'll sort the list `[0.78, 0.17, 0.39, 0.26, 0.72, 0.94, 0.15, 0.12]` — eight floating-point numbers, all between 0 and 1, which is the classic situation bucket sort is built for.

Each number's bucket comes from multiplying it by 8 and rounding down. `0.17` and `0.15` both round down to bucket `1`, since they're close together and both fall in the `0.125-0.250` slice — they're added to that bucket in the order they appear in the list, `0.17` first, then `0.15`. Bucket `4` ends up empty because no number in this particular list falls between `0.500` and `0.625`. This step needs extra room to hold all these small buckets — it isn't sorting the original list directly in place the way some other methods do.

### Step 3 — tidy up the numbers inside bucket 1

  caption="Step 3: bucket 1 is the only bucket with more than one number, so it's the only one that needs sorting"
  content={`Only bucket 1 has more than one number, so it's the only one that needs work:
bucket 1 before: ┌──────┬──────┐
                  │ 0.17 │ 0.15 │

bucket 1 after:  ┌──────┬──────┐

Bucket `1` holds `[0.17, 0.15]`, out of order. Insertion sort — the simple method that builds up a sorted mini-list one number at a time — compares `0.15` against `0.17`, sees it's smaller, and slides it in front, giving `[0.15, 0.17]`. Since every other bucket has zero or one number in it, there's nothing to compare, so they're already sorted by default.

Because the buckets themselves are already in size order (bucket `0` before bucket `1` before bucket `2`, and so on), and each bucket is sorted inside itself, simply reading them out front to back produces the finished, sorted list: `[0.12, 0.15, 0.17, 0.26, 0.39, 0.72, 0.78, 0.94]`. If two numbers land in the same bucket and are exactly equal, they keep the left-to-right order they started in, since they're added to the bucket in that order and insertion sort preserves it — that's what makes bucket sort a "stable" sort.

## Complexity

- **Best case**: When the numbers are spread out evenly, one per bucket, there's barely any sorting left to do inside each bucket — just the one pass to drop numbers in and read them back out (`O(n)`).
- **Average case**: This assumes the input is spread out roughly evenly across the range, which keeps every bucket small. On average, tidying up all the buckets together costs about the same as one pass through the list (`O(n)`).
- **Worst case**: If every number happens to land in the same bucket — say they're all clustered in a narrow range — bucket sort collapses into sorting one big pile with insertion sort, which gets slow fast (`O(n²)`).
- **Space**: It needs extra room to hold all the buckets alongside the original list, since it isn't sorting in place (`O(n)`).

## Applications

- Sorting large sets of measurements known to be spread evenly across a range, such as normalized sensor readings or probabilities between 0 and 1.
- Color quantization and histogram-based operations in image processing, where pixel intensity values are roughly uniform across their range and get bucketed before further work.
- The partitioning phase of external sorting — sorting data too big to fit in memory — when the data's range is known and roughly uniform, so each bucket becomes its own manageable chunk to sort and merge.
- A stepping stone toward radix sort, which repeatedly buckets numbers by digit — bucket sort's distribute-then-sort idea is the core mechanism radix sort applies one digit at a time.

int *bucket_sort(int *array, int n) {
    int *result = (int *)malloc(sizeof(int) * (n > 0 ? n : 1));
    if (n == 0) {
        return result;
    }
    int max_value = array[0];
    for (int i = 1; i < n; i++) {
        if (array[i] > max_value) {
            max_value = array[i];
        }
    }
    int num_buckets = n;
    int **buckets = (int **)malloc(sizeof(int *) * num_buckets);
    int *bucket_counts = (int *)calloc(num_buckets, sizeof(int));
    for (int i = 0; i < num_buckets; i++) {
        buckets[i] = (int *)malloc(sizeof(int) * n);
    }
    for (int i = 0; i < n; i++) {
        int index = (int)((long)array[i] * num_buckets / (max_value + 1));
        buckets[index][bucket_counts[index]++] = array[i];
    }
    int pos = 0;
    for (int i = 0; i < num_buckets; i++) {
        insertion_sort(buckets[i], bucket_counts[i]);
        for (int j = 0; j < bucket_counts[i]; j++) {
            result[pos++] = buckets[i][j];
        }
        free(buckets[i]);
    }
    free(buckets);
    free(bucket_counts);
    return result;
}`}</CodeBlock>

        return {};

        result.insert(result.end(), bucket.begin(), bucket.end());

public class BucketSort {
    private static void insertionSort(List<Integer> bucket) {

            int j = i - 1;
            while (j >= 0 && bucket.get(j) > key) {
                bucket.set(j + 1, bucket.get(j));

    for i in 1..bucket.len() {
        let key = bucket[i];
        let mut j = i as i32 - 1;
        while j >= 0 && bucket[j as usize] > key {
            bucket[(j + 1) as usize] = bucket[j as usize];
            j -= 1;
        }
        bucket[(j + 1) as usize] = key;
    }
}

pub fn bucket_sort(array: &Vec<i32>) -> Vec<i32> {
    let n = array.len();
    if n == 0 {
        return Vec::new();
    }
    let max_value = *array.iter().max().unwrap();
    let num_buckets = n;
    let mut buckets: Vec<Vec<i32>> = vec![Vec::new(); num_buckets];
    for &value in array.iter() {
        let index = (value as i64 * num_buckets as i64 / (max_value as i64 + 1)) as usize;
        buckets[index].push(value);
    }
    let mut result: Vec<i32> = Vec::with_capacity(n);
    for mut bucket in buckets {
        insertion_sort(&mut bucket);
        result.extend(bucket);
    }
    result
