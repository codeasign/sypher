## Summary

Shell Sort is like Insertion Sort's faster cousin: instead of only ever comparing numbers that sit right next to each other, it starts by comparing numbers that are far apart, then gradually narrows the gap until it's comparing neighbors like normal. Comparing far-apart numbers first lets badly-placed numbers jump most of the way to where they belong in just one move, instead of shuffling one step at a time. It's usually a fair bit faster than plain Insertion Sort, though it can still slow down a lot on some inputs.

## Pseudocode

```text
1. Start with a "gap" about half the length of the list.
2. While the gap is bigger than zero:
   a. Go through the list, comparing each number to the one `gap` positions
      before it (not the one right next to it).
   b. If the earlier number is bigger, shift it forward and keep comparing
      further back, until you find where the current number fits — this is
      the same idea as Insertion Sort, just done at a distance.
   c. Once you've gone through the whole list at this gap, cut the gap in
      half.
3. When the gap finally shrinks to zero, stop — the list is sorted.
```

## How It Works

- Look at the diagram: with a gap of 3, position 0 (`8`) is compared to position 3 (`1`) — far apart, not neighbors. Since `8` is bigger, they swap, and `1` jumps straight to the front in one move instead of shuffling past `3` and `5` one step at a time.
- The same happens for position 1 vs 4 (`3` vs `9`, no swap needed) and position 2 vs 5 (`5` vs `2`, they swap).
- After this one pass, the list is `[1, 3, 2, 8, 9, 5]` — not fully sorted yet, but the badly-placed numbers have already jumped most of the way home.
- The gap then shrinks (3 to 1), and the final pass compares actual neighbors, cleaning up whatever's left. Because the big jumps already happened, this last cleanup pass has much less work to do than if you'd started with neighbors from the beginning.
- This method rearranges the list directly without needing extra storage space, so it doesn't need any extra room to work.
- If two numbers are exactly equal, this method doesn't promise to keep them in their original left-to-right order, because they might get compared and moved across different gap rounds.
- How fast is it? If the list is close to sorted already, this finishes quickly. If it's arranged in a particularly unlucky way, it can end up doing about as much work as comparing almost every number to every other number (worst case around `O(n²)`, though picking the gap sizes more cleverly can improve that).
