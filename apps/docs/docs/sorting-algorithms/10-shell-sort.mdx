## Summary

Shell Sort is like Insertion Sort's faster cousin: instead of only ever comparing numbers that sit right next to each other, it starts by comparing numbers that are far apart, then gradually narrows the gap until it's comparing neighbors like normal. Comparing far-apart numbers first lets badly-placed numbers jump most of the way to where they belong in just one move, instead of shuffling one step at a time. It's usually a fair bit faster than plain Insertion Sort, though it can still slow down a lot on some inputs.

## Pseudocode

```text
1. Start with a "gap" about half the length of the list.
2. While the gap is bigger than zero:
   a. Go through the list, comparing each number to the one `gap` positions
      before it (not the one right next to it).
   b. If the earlier number is bigger, shift it forward and keep comparing
      further back, until you find where the current number fits — this is
      the same idea as Insertion Sort, just done at a distance.
   c. Once you've gone through the whole list at this gap, cut the gap in
      half.
3. When the gap finally shrinks to zero, stop — the list is sorted.
```

## Worked Example

We'll trace `[29, 10, 14, 37, 5, 22, 3, 41]` through every gap Shell Sort uses on an 8-element list: gap 4, then gap 2, then gap 1. Watch how the far-apart comparisons move numbers most of the way home before the final cleanup pass even starts.

With a gap of 4, position 0 (`29`) only ever gets compared to position 4 (`5`) — nowhere near each other in the list, but exactly 4 slots apart. Since `29` is bigger, they swap, and `5` jumps straight from the middle of the list to the very front in one move. The same happens for positions 2 and 6 (`14` and `3` swap); positions 1/5 and 3/7 are already in the right order, so nothing moves there. This rearranges the same list directly with no second list involved, which is what people mean when they call a sort **in-place** — Shell Sort never needs extra storage the size of the input.

When the gap shrinks to 2, the list splits into two separate chains that never touch each other: every even position (0, 2, 4, 6) and every odd position (1, 3, 5, 7). Looking only at the even chain — `5, 3, 29, 14` — position 0 and 2 swap (`5 > 3`), and position 4 and 6 swap (`29 > 14`). Each chain gets fully insertion-sorted at this gap before moving on.

The odd chain — `10, 37, 22, 41` — only needs one move: position 3 and 5 swap (`37 > 22`), since `22` belongs before `37`. Positions 1 and 7 were already correct within this chain, so they stay put. Put both chains back together and the full list is now `[3, 10, 5, 22, 14, 37, 29, 41]` — every number is already much closer to its final resting place, even though no neighbor has been compared yet.

The gap finally drops to 1, which just means comparing plain neighbors — exactly what Insertion Sort does on its own. Because the earlier gap-4 and gap-2 passes already moved everything close to where it belongs, this last pass only needs three small swaps (`10`/`5`, `22`/`14`, `37`/`29`) to finish, far less work than a from-scratch Insertion Sort pass would need on the original unsorted list. The list is now fully sorted: `[3, 5, 10, 14, 22, 29, 37, 41]`.

One thing to flag: because far-apart passes can swap numbers before they ever meet a tied value, Shell Sort does not guarantee two equal numbers keep their original left-to-right order — it is not a **stable** sort.

## Complexity

- **Best case:** if the list is already sorted, every gap pass finds nothing to shift and barely does any work (`O(n log n)`).
- **Average case:** across most shuffled lists, the shrinking gaps clear out disorder faster than plain Insertion Sort, landing somewhere around (`O(n^1.5)`) for the classic gap sequence used here.
- **Worst case:** with an unlucky combination of starting order and gap choices, it can degrade to doing about as much work as comparing almost every number to every other number (`O(n²)`).
- **Space:** it only ever rearranges the same list in place, so it needs no extra memory beyond a couple of temporary variables (`O(1)`).
- All of this depends heavily on the **gap sequence** — the list of gap sizes used (here: half the length, then half again, down to 1). Different gap sequences give provably different worst-case guarantees; picking a better sequence is the main way people speed Shell Sort up further.

## Applications

- It's a good fit for **embedded systems and firmware** with tight memory limits, since it sorts in place and needs no recursion stack or second array the way Merge Sort or Quick Sort do.
- Some **small-to-medium in-memory datasets** favor it over Quick Sort or Merge Sort when the overhead of recursion or extra allocations isn't worth it for a list that isn't huge.
- The **uClibc** C standard library (used in many embedded Linux systems) has historically shipped a Shell Sort implementation for its `qsort()` function.
- It shows up in some **hardware sorting circuits and microcontroller libraries**, where its simple, in-place, iterative structure is easy to implement without a heap or call stack.

<CodeBlock language="python">{`def shell_sort(array: list[int]) -> list[int]:
    n = len(array)
    gap = n // 2
    while gap > 0:
        for i in range(gap, n):
            temp = array[i]
            j = i
            while j >= gap and array[j - gap] > temp:
                array[j] = array[j - gap]
                j -= gap
            array[j] = temp
        gap //= 2
    return array`}</CodeBlock>
