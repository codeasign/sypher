## Summary

Insertion Sort builds a sorted list one number at a time, the same way you'd sort playing cards in your hand. It picks up the next number and slides it left past any bigger numbers already in the sorted part, until it lands in the right spot. It's fast when the list is already close to sorted, but it slows down the more out of order the list is.

## Pseudocode

```text
1. Treat the first number as a sorted list of one.
2. Pick up the next number in line — call it the "key".
3. Compare the key to the number just before it in the sorted part.
4. If that number is bigger than the key, slide it one spot to the right.
5. Keep sliding numbers right, moving one spot left each time, as long as they're bigger than the key.
6. Once you hit a number that's smaller than the key (or you reach the start of the list), drop the key into that empty spot.
7. Repeat steps 2-6 for every remaining number in the list.
```

The new key is the second `2`. It slides past `8` and `5`, but when it reaches the first `2` the comparison is "equal", not "bigger", so sliding stops immediately — the key never hops over a number that's tied with it. That's what makes a sort **stable**: equal values keep their original left-to-right order. The sorted part is now `[2, 2, 5, 8]`.

## Complexity

- **Best case:** if the list is already sorted (or nearly sorted), every key only needs one quick comparison before it stays put, so it barely does any work (`O(n)`).
- **Average case:** on a typically shuffled list, each new key slides past roughly half of the sorted part on its way in (`O(n²)`).
- **Worst case:** if the list is in reverse order, every key has to slide all the way past everything already placed (`O(n²)`).
- **Space:** it only ever slides numbers inside the same list — no second list is built — so it needs no extra memory beyond a couple of temporary variables (`O(1)`).

## Applications

- Standard library sort functions often fall back to Insertion Sort for small lists or small sub-lists — for example, Python's and Java's built-in sorts (**Tim Sort**) switch to insertion sort for runs below a small size threshold because the overhead of fancier algorithms isn't worth it on so few elements.
- It's a natural fit for **online sorting**, where numbers arrive one at a time and each new one needs to be placed into an already-sorted list right away, like keeping a live leaderboard in order as new scores come in.
- It performs very well on **nearly-sorted data**, such as a mostly-ordered log file that only has a few entries out of place, since each key needs hardly any sliding.
- Some embedded and real-time systems favor it for small, mostly-sorted datasets because the code is tiny and its best-case behavior is predictable.
