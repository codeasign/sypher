# Comparison & Selection Guide

A friendly cheat sheet across all 11 sorting methods covered in this course. Use it to
pick a reasonable method for the situation you're in — how big the list is, how much
extra memory you can spare, whether equal items need to keep their original order —
instead of just always reaching for the first one you learned.

## Quick Refresher: What the Speed Notation Means

Every table below lists rough speed using the same shorthand you've seen in each
module, written as a plain phrase first and the technical notation in parentheses:

- **Barely changes as the list grows (`O(1)`)** — same amount of work no matter how big
  the list is.
- **Grows in slow steps (`O(log n)`)** — work grows very slowly because each step cuts
  the remaining problem roughly in half.
- **Grows in direct proportion (`O(n)`)** — double the list, double the work.
- **Grows a bit faster than the list size (`O(n log n)`)** — slightly more than direct
  proportion, but still comfortably fast even on large lists. Most of the "good"
  general-purpose sorts land here.
- **Gets much slower with big lists (`O(n²)`)** — ten times the list means roughly a
  hundred times the work. Fine for small lists, painful for large ones.

## Speed, Memory, and Order-Keeping at a Glance

| Algorithm | Best Case | Typical Case | Worst Case | Extra Memory | Keeps Equal Items in Order? | Sorts in Place? |
|---|---|---|---|---|---|---|
| Bubble Sort | already sorted, barely any work (O(n)) | gets much slower with big lists (O(n²)) | gets much slower with big lists (O(n²)) | none needed (O(1)) | Yes | Yes |
| Selection Sort | gets much slower with big lists (O(n²)) | gets much slower with big lists (O(n²)) | gets much slower with big lists (O(n²)) | none needed (O(1)) | No | Yes |
| Insertion Sort | already sorted, barely any work (O(n)) | gets much slower with big lists (O(n²)) | gets much slower with big lists (O(n²)) | none needed (O(1)) | Yes | Yes |
| Merge Sort | comfortably fast, always (O(n log n)) | comfortably fast, always (O(n log n)) | comfortably fast, always (O(n log n)) | needs a second list (O(n)) | Yes | No |
| Quick Sort | comfortably fast, usually (O(n log n)) | comfortably fast, usually (O(n log n)) | gets much slower on unlucky input (O(n²)) | tiny bit of extra room (O(log n)) | No | Yes |
| Heap Sort | comfortably fast, always (O(n log n)) | comfortably fast, always (O(n log n)) | comfortably fast, always (O(n log n)) | none needed (O(1)) | No | Yes |
| Counting Sort | grows with list size + value range (O(n + k)) | grows with list size + value range (O(n + k)) | grows with list size + value range (O(n + k)) | grows with list size + value range (O(n + k)) | Yes | No |
| Radix Sort | grows with digits × list size (O(d(n + b))) | grows with digits × list size (O(d(n + b))) | grows with digits × list size (O(d(n + b))) | grows with list size + digit base (O(n + b)) | Yes | No |
| Bucket Sort | fast if spread out evenly (O(n + k)) | fast if spread out evenly (O(n + k)) | slow if lopsided (O(n²)) | needs room for the groups (O(n + k)) | Yes | No |
| Shell Sort | comfortably fast (O(n log n)) | grows a bit faster than that (O(n^1.3)) | gets much slower with big lists (O(n²)) | none needed (O(1)) | No | Yes |
| Tim Sort | already sorted, barely any work (O(n)) | comfortably fast, always (O(n log n)) | comfortably fast, always (O(n log n)) | needs a second list (O(n)) | Yes* | No |

`k` = how wide the range of values is (Counting Sort, Bucket Sort). `d` = number of
digits and `b` = the base you're counting in, e.g. 10 for decimal digits (Radix Sort).
*Tim Sort keeps equal items in order in the languages where the built-in sort really is
Tim Sort under the hood (Python, Java's `Object[]`/`Integer[]`, JavaScript/V8, Rust) —
see module 11 for the languages where the built-in uses a different hybrid that doesn't
promise this (C++, C#, Go, C).

## Best Use Case Per Algorithm

| Algorithm | Best Use Case |
|---|---|
| Bubble Sort | Mostly for learning the idea — fine for a tiny, nearly-sorted list where simplicity matters more than speed. |
| Selection Sort | When you want to keep the number of times you move/write data as low as possible (e.g. writing to memory that wears out with too many writes). |
| Insertion Sort | Small lists, or lists that are already almost in order — also the simple method other, fancier sorts fall back on for their smallest pieces. |
| Merge Sort | When you need a guarantee it will never slow down no matter the input, and equal items must keep their order — good for huge datasets that don't fit in memory, or sorting linked lists. |
| Quick Sort | General everyday sorting in memory, when typical speed matters more than a guarantee about the worst possible case. |
| Heap Sort | When you need a guarantee it will never slow down, but also need to keep extra memory use to almost nothing — good for memory-tight devices or priority queues. |
| Counting Sort | Sorting a small, known range of whole numbers (like exam scores from 0 to 100) — faster than any method that compares items one by one. |
| Radix Sort | Sorting lots of fixed-length values, like phone numbers or ID numbers, where the range of possible values is too big for Counting Sort alone. |
| Bucket Sort | Data that's spread out fairly evenly across a known range (like decimal numbers between 0 and 1). |
| Shell Sort | Simple devices that need better speed than the really basic methods, without the extra memory or complexity of Merge Sort or Quick Sort. |
| Tim Sort | Everyday real-world sorting — this is what you'd actually reach for in practice, since it's what most popular programming languages already use behind their built-in sort. |

## Decision Checklist

- **Need to make sure it never gets slow no matter what?** Use Merge Sort or Heap Sort — both promise comfortably fast sorting every time, no matter how messy the input is. Quick Sort and Shell Sort can hit a slow patch on unlucky input.
- **Trying to use almost no extra memory?** Use Heap Sort, Shell Sort, Insertion Sort, Selection Sort, or Bubble Sort. Merge Sort, Counting Sort, Radix Sort, Bucket Sort, and Tim Sort all need some extra room to work.
- **Need equal items to keep their original order?** Use Merge Sort, Insertion Sort, Bubble Sort, Counting Sort, Radix Sort, Bucket Sort, or Tim Sort. Quick Sort, Heap Sort, Selection Sort, and Shell Sort don't promise this.
- **Sorting whole numbers within a small, known range?** Counting Sort is your best bet — it doesn't even compare items to each other, it just counts them.
- **Sorting fixed-length values with a big range** (like 10-digit ID numbers)? Use Radix Sort — it applies Counting Sort one digit at a time, which avoids Counting Sort's problem with very wide ranges.
- **Is your data spread out evenly across a known range?** Bucket Sort works well here — but check that assumption first, since lopsided data makes it slow.
- **Writing real code in a language that already has a built-in sort?** Just use the built-in one (it's Tim Sort or something similarly well-tuned — see module 11). Building your own version of modules 1-10 is great for learning, not for production code.
- **Studying for a coding interview and asked for an in-place sort with minimal memory and no order-keeping requirement?** Heap Sort and Quick Sort are the two most commonly expected answers — Quick Sort for its typical speed, Heap Sort for its guarantee against slow cases.
