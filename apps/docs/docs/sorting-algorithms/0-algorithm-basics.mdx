# Algorithm Basics

New to algorithms? Start here. This page explains the handful of ideas every
module in this course assumes you already know — how we talk about "fast"
and "slow," and what words like **stable** and **in-place** actually mean.
None of this is sorting-specific yet; it's the vocabulary the rest of the
course is written in.

If you've taken a data structures class before, skim this page and jump to
[Module 1: Bubble Sort](/docs/sorting-algorithms/bubble-sort).

## What Does "Sorting" Actually Mean?

Sorting means rearranging a collection of items into a defined order —
usually smallest to largest. Every algorithm in this course solves the exact
same problem (turn `[5, 3, 8, 1]` into `[1, 3, 5, 8]`); what changes between
modules is *how* they get there, and what that approach costs in time and
memory.

## Why Speed Matters: Reading Big-O Notation

You'll see notation like `O(n^2)` or `O(n log n)` in every module. This is
called **Big-O notation**, and all it measures is: *if the list gets bigger,
how much more work does the algorithm have to do?* It's a growth pattern, not
a stopwatch time.

Here's what the ones you'll actually see mean, with an everyday comparison:

- **`O(n log n)` — the "good" sorting speed.** Think of sorting a deck of
  cards by repeatedly splitting it in half, sorting each half, and merging
  the results back together (this is exactly what Merge Sort, Module 4,
  does). Each split roughly halves the work, so the total cost grows much
  slower than comparing every card to every other card.
- **`O(n^2)` — the "naive" sorting speed.** This shows up when an algorithm
  compares every item to every other item, like checking every pair of
  people in a room to see who's taller. Ten times the people means a hundred
  times the comparisons.
- **`O(n + k)` — the "cheat code" speed.** A few algorithms in this course
  (Counting Sort, Radix Sort, Bucket Sort) don't compare items to each other
  at all — they exploit something they already know about the *values*
  (like "these are all whole numbers between 0 and 100") to skip comparisons
  entirely. `k` here means the size of that known range.

The table below makes the difference concrete for sorting 1,000,000 items:

| Growth pattern | Steps at `n = 1,000,000` |
|---|---|
| `O(n log n)` | ~20,000,000 |
| `O(n^2)` | 1,000,000,000,000 |

The lesson: as lists get bigger, the difference between an `O(n log n)` sort
and an `O(n^2)` sort stops being "a little slower" and becomes "unusably
slow." This is why the first few modules (Bubble, Selection, Insertion —
all `O(n^2)`) are taught as foundations, while real software almost always
reaches for an `O(n log n)` sort (or better) once lists get large.

## Two Words You'll See in Every Module

**Stable.** A sort is stable if items that compare as *equal* keep their
original relative order after sorting. Imagine sorting a stack of exam papers
by grade — if two students both scored 85, and student A's paper was above
student B's before sorting, a stable sort guarantees A's paper is still above
B's after sorting. An unstable sort makes no such promise; equal items might
end up in either order.

**In-place.** A sort is in-place if it rearranges the items directly within
the original array, using only a small, fixed amount of extra scratch space
(like one temporary swap variable) — it never needs to build a second,
same-sized array to hold the result. An out-of-place sort does need that
second array.

Both properties matter in practice: stability matters whenever you're
sorting by one field but want to preserve order from a previous sort (e.g.
sorting by last name after already sorting by first name); in-place matters
when memory is limited, since it avoids doubling the space the data needs.
Every module in this course states both properties explicitly, and Module 12
(the Comparison & Selection Guide) tabulates them across all eleven
algorithms side by side.

## How This Course Is Organized

Every module from here on follows the same shape:

1. **Summary** — a plain-English, one-paragraph description of the idea,
   with an everyday analogy and a quick mention of how fast it is and how
   much memory it needs.
2. **Pseudocode** — the algorithm as a short, language-neutral recipe.
3. **Worked Example** — a numbered sequence of pictures walking one real
   list through every pass, with a sentence or two after each picture
   explaining what just happened and why.
4. **Complexity** — best, average, and worst case speed, plus memory use,
   each explained in plain English first with the Big-O notation alongside
   it.
5. **Applications** — a handful of concrete, real places this exact
   algorithm actually gets used.
6. **Solutions** — a complete, runnable reference implementation in nine
   languages (Python, C, C++, Java, C#, JavaScript, TypeScript, Rust, and
   Go), so you can read the whole thing in whichever language you're most
   comfortable with.

Read the summary and pseudocode first, then follow the worked example
picture by picture until it genuinely makes sense — *then* look at the code.
Module 12 at the end is different — it's a plain decision guide with no
code, meant to be read after you've worked through the algorithms
themselves.
