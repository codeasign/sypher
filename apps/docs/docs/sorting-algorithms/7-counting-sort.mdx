## Summary

Counting Sort never actually compares two numbers to each other. Instead, it counts how many times each value shows up in the list, and uses those counts to figure out exactly where every number belongs in the finished, sorted list. It works great when the numbers are all whole numbers in a small, known range — like ages, or exam scores. Because it skips comparing altogether, it can be faster than other sorts, but only when that range of possible values is small.

## Pseudocode

```text
1. If the list is empty, stop — there's nothing to sort.
2. Find the biggest number in the list, so you know how many "slots" you'll need to count.
3. Make a tally: for every number in the list, add one to that number's count.
4. Turn the tallies into running totals — each slot now holds "how many numbers are this value or smaller."
5. Go through the original list from the end to the beginning. For each number, look up its running total, place it there in the final list, and then lower that running total by one.
6. Once every number has been placed, the final list is fully sorted.
```

## How It Works

- Instead of comparing items to each other, Counting Sort counts how many of each value there are. In the diagram, there's one `0`, one `1`, and two `2`s.
- Those counts get turned into running totals, which tell you exactly how many spots in the final list come before (or at) each value. For example, the running total for `1` is `2`, meaning "2 numbers are 1 or smaller."
- Counting Sort then walks through the original list and drops each number straight into its correct spot using that running total — no side-by-side comparisons needed.
- Counting Sort is not "in-place" — it needs extra space for the counts and for a separate output list, rather than just rearranging your original list.
- Counting Sort is "stable," meaning if two numbers in the list are equal, they end up in the same order they started in — as long as you go through the list from end to beginning while placing numbers, which this method does.
- How fast is it? Counting Sort is fast and steady — it takes about the same amount of time no matter the starting order, because it never compares numbers (this is `O(n + k)` time, where `k` is how wide the range of possible values is). The catch: if that range of possible values is huge compared to how many numbers you have, Counting Sort wastes a lot of space and time counting slots you don't need.
