## Summary

Counting Sort never actually compares two numbers to each other. Instead, it counts how many times each value shows up in the list, and uses those counts to figure out exactly where every number belongs in the finished, sorted list. It works great when the numbers are all whole numbers in a small, known range — like ages, or exam scores. Because it skips comparing altogether, it can be faster than other sorts, but only when that range of possible values is small.

## Pseudocode

```text
1. If the list is empty, stop — there's nothing to sort.
2. Find the biggest number in the list, so you know how many "slots" you'll need to count.
3. Make a tally: for every number in the list, add one to that number's count.
4. Turn the tallies into running totals — each slot now holds "how many numbers are this value or smaller."
5. Go through the original list from the end to the beginning. For each number, look up its running total, place it there in the final list, and then lower that running total by one.
6. Once every number has been placed, the final list is fully sorted.
```

## Worked Example

We'll trace `[4, 2, 2, 5, 0, 3]` through every step. The two `2`s will show what happens to equal values along the way.

Six numbers, values `0` through `5`. Counting Sort's first move is to stop thinking about this list as a sequence and start thinking about it as a set of counts.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-2"
  alt="A tally table showing how many times each value from 0 to 5 appears in the input list: one 0, zero 1s, two 2s, one 3, one 4, one 5"
  caption="Step 2: one tally slot for every possible value from 0 to 5"
  content={`step 2 ──> tally how many times each value appears
─────────────────────────────────────────────────────
 value   0     1     2     3     4     5
       ┌─────┬─────┬─────┬─────┬─────┬─────┐
 count │  1  │  0  │  2  │  1  │  1  │  1  │
       └─────┴─────┴─────┴─────┴─────┴─────┘`}
/>

A counting array with one slot per possible value gets built by walking the input once and adding one to the matching slot. Value `2` shows a count of `2`, because it appears twice. This counting array — along with the output list built later — is extra memory the original list didn't need, which is why Counting Sort is not **in-place**: it can't finish the job by only rearranging numbers inside the original list.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-3"
  alt="The tally counts converted into cumulative running totals: 1, 1, 3, 4, 5, 6 for values 0 through 5"
  caption="Step 3: each slot now means how many numbers are that value or smaller"
  content={`step 3 ──> turn tallies into running totals (cumulative counts)
───────────────────────────────────────────────────────────────
 value   0     1     2     3     4     5
       ┌─────┬─────┬─────┬─────┬─────┬─────┐
 total │  1  │  1  │  3  │  4  │  5  │  6  │
       └─────┴─────┴─────┴─────┴─────┴─────┘`}
/>

Each slot now adds up every count up to and including that value. The slot for `2` reads `3`, meaning "3 numbers in the whole list are 2 or smaller." Those running totals are about to become exact destination addresses in the sorted output.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-4"
  alt="Placing the values 3 and 0 from the end of the input list into output slots 3 and 0, using their running totals as the destination index"
  caption="Step 4: place the last two input values, working from the end of the list backward"
  content={`step 4 ──> walk the input backward, starting from the last value
─────────────────────────────────────────────────────────────────
 i=5 value=3 ──> total[3]=4 ──> place in slot 3, total[3]-- (now 3)
 i=4 value=0 ──> total[0]=1 ──> place in slot 0, total[0]-- (now 0)
 output: ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
         │ 0 │ │ · │ │ · │ │ 3 │ │ · │ │ · │
         └───┘ └───┘ └───┘ └───┘ └───┘ └───┘
 slot:      0     1     2     3     4     5`}
/>

Counting Sort now walks the original list from the last number to the first. `3` looks up its running total (`4`), lands in output slot `3` (one before its total, since slots start at zero), and its total drops by one for next time. `0` does the same and lands in slot `0`.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-5"
  alt="Placing the value 5 from index 3 of the input into output slot 5 using its running total"
  caption="Step 5: 5 is the largest value, so it lands in the last slot"
  content={`step 5 ──> continue backward: place 5
───────────────────────────────────────
 i=3 value=5 ──> total[5]=6 ──> place in slot 5, total[5]-- (now 5)
 output: ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
         │ 0 │ │ · │ │ · │ │ 3 │ │ · │ │ 5 │
         └───┘ └───┘ └───┘ └───┘ └───┘ └───┘
 slot:      0     1     2     3     4     5`}
/>

`5` is the biggest value in the list, so its running total of `6` sends it straight to the last slot, index `5`. Three of six numbers are now placed.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-6"
  alt="Placing the two equal 2 values from indices 2 and 1 of the input into output slots 2 and 1, preserving their original left-to-right order"
  caption="Step 6: walking backward keeps equal values in their original order"
  content={`step 6 ──> place both 2s — this is where stability matters
─────────────────────────────────────────────────────────────
 i=2 value=2 ──> total[2]=3 ──> place in slot 2, total[2]-- (now 2)
 i=1 value=2 ──> total[2]=2 ──> place in slot 1, total[2]-- (now 1)
 output: ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
         │ 0 │ │ 2 │ │ 2 │ │ 3 │ │ · │ │ 5 │
         └───┘ └───┘ └───┘ └───┘ └───┘ └───┘
 slot:      0     1     2     3     4     5`}
/>

Here's the payoff of walking backward: the `2` at index `2` gets read first and takes the higher slot, `2`, then the `2` at index `1` takes the lower slot, `1`. The `2` that started further left in the input (index `1`) still ends up further left in the output. That's what **stable** means — equal values never swap past each other, so their original left-to-right order survives the sort.

<AsciiDiagram
  id="sorting-algorithms/counting-sort/step-7"
  alt="The final placement of value 4 into output slot 4, producing the fully sorted list 0, 2, 2, 3, 4, 5"
  caption="Step 7: every value placed — the list is sorted"
  content={`step 7 ──> place the last value, 4 — sort complete
──────────────────────────────────────────────────────
 i=0 value=4 ──> total[4]=5 ──> place in slot 4, total[4]-- (now 4)
 output: ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
         │ 0 │ │ 2 │ │ 2 │ │ 3 │ │ 4 │ │ 5 │
         └───┘ └───┘ └───┘ └───┘ └───┘ └───┘
                    fully sorted list`}
/>

The last number, `4`, drops into slot `4`, and every slot is filled: `[0, 2, 2, 3, 4, 5]`. No two numbers were ever directly compared — the whole sort came from counting and addition.

## Complexity

Counting Sort is **not comparison-based** — it never asks "is this number bigger than that one?" — so its speed depends only on the size of the list (`n`) and the range of possible values (`k`), never on how shuffled the list starts out.

- **Best case:** even a list that's already sorted still needs a full count-and-place pass, so there's no shortcut for lucky orderings (`O(n + k)`).
- **Average case:** the runtime doesn't change with a typical shuffled list, because nothing about the order affects the counting or placing steps (`O(n + k)`).
- **Worst case:** even the most out-of-order list possible takes exactly the same amount of work, since there are no comparisons for a bad order to slow down (`O(n + k)`).
- **Space:** it needs a separate counting array sized to the range of values, plus a separate output list the same size as the input, so it always uses extra memory instead of just rearranging the original list (`O(n + k)`).

The catch is `k`. If the range of possible values is huge compared to how many numbers you actually have, Counting Sort wastes a lot of time and memory building count slots that mostly stay at zero.

## Applications

- Sorting values you know sit in a small, fixed range ahead of time — exam scores from 0 to 100, ages, or single-digit ratings — where building one count slot per possible value is cheap.
- It's the core building block inside **Radix Sort**: each pass of Radix Sort sorts the list by one digit or character at a time, and that per-digit sort is Counting Sort underneath.
- Histogram-style operations in image processing, such as counting how many pixels have each intensity value (0 to 255) before reordering them for histogram equalization.
- Sorting data that maps onto a small number of categories, like grading students by letter grade or bucketing orders by shirt size, once those categories are converted to small integers.

    if not array:
        return []
    max_val = max(array)
    count = [0] * (max_val + 1)
    for value in array:
        count[value] += 1
    for i in range(1, len(count)):
        count[i] += count[i - 1]
    output = [0] * len(array)
    for i in range(len(array) - 1, -1, -1):
        value = array[i]
        count[value] -= 1
        output[count[value]] = value
    return output`}</CodeBlock>

int *counting_sort(int *array, int n) {
    if (n == 0) {
        return NULL;
    }
    int max_val = array[0];
    for (int i = 1; i < n; i++) {
        if (array[i] > max_val) {
            max_val = array[i];
        }
    }
    int *count = (int *)calloc(max_val + 1, sizeof(int));
    for (int i = 0; i < n; i++) {
        count[array[i]]++;
    }
    for (int i = 1; i <= max_val; i++) {
        count[i] += count[i - 1];
    }
    int *output = (int *)malloc(n * sizeof(int));
    for (int i = n - 1; i >= 0; i--) {
        int value = array[i];
        count[value]--;
        output[count[value]] = value;
    }
    free(count);
    return output;
}`}</CodeBlock>

    if array.is_empty() {
        return Vec::new();
    }
    let max_val = *array.iter().max().unwrap();
    let mut count = vec![0usize; (max_val + 1) as usize];
    for &value in array.iter() {
        count[value as usize] += 1;
    }
    for i in 1..count.len() {

    for i in (0..array.len()).rev() {
        let value = array[i];
        count[value as usize] -= 1;
        output[count[value as usize]] = value;
    }
    output
