# Trapping Rain Water — Solution

## Problem Overview

Given `n` non-negative integers representing an elevation map where each bar has width 1, compute how much water it can trap after raining.

## Brute Force Approach

**Idea:** For each position, find the maximum height to its left and right. The water trapped at that position is `min(maxLeft, maxRight) - height[i]`.

**Algorithm:**
1. Initialize `water = 0`
2. For each index `i` from 0 to n-1:
   - Find the maximum height to the left of `i` (including `i`)
   - Find the maximum height to the right of `i` (including `i`)
   - Add `min(maxLeft, maxRight) - height[i]` to `water`
3. Return `water`

**Complexity Analysis:**
- **Time:** O(n²) — for each of n elements, we scan left and right (O(n) each)
- **Space:** O(1) — no extra space

**Why this is suboptimal:** The nested scans are redundant. We can precompute maxLeft and maxRight arrays, or better yet, use two pointers to track them on the fly.

## Better Approach: Dynamic Programming (Precomputed Arrays)

**Idea:** Precompute the maximum height from the left and from the right for each position, then compute water in a single pass.

**Algorithm:**
1. Create `leftMax` array of size n: `leftMax[i] = max(leftMax[i-1], height[i])`
2. Create `rightMax` array of size n: `rightMax[i] = max(rightMax[i+1], height[i])`
3. For each position `i`: `water += min(leftMax[i], rightMax[i]) - height[i]`

**Complexity Analysis:**
- **Time:** O(n) — three passes over the array
- **Space:** O(n) — two arrays of size n

**Why this is better but not optimal:** O(n) time is good, but the O(n) space can be eliminated.

## Optimal Approach: Two Pointers

**Idea:** Maintain two pointers (left and right) and track the running maximum from each side. At each step, process the side with the smaller height — that side's max determines the water level because the water cannot be held higher than the lower bounding wall.

**Algorithm:**
1. Initialize `left = 0`, `right = n - 1`, `maxLeft = 0`, `maxRight = 0`, `water = 0`
2. While `left < right`:
   - If `height[left] < height[right]`:
     - If `height[left] >= maxLeft`: update `maxLeft = height[left]`
     - Else: `water += maxLeft - height[left]`
     - Increment `left`
   - Else:
     - If `height[right] >= maxRight`: update `maxRight = height[right]`
     - Else: `water += maxRight - height[right]`
     - Decrement `right`
3. Return `water`

**Why This Works:** The key insight is that water level at any position is determined by the smaller of the two maximum heights on either side. By always processing the side with the smaller height, we know that the other side has a taller wall (or equal), so the water level is bounded by the current side's max. This eliminates the need to precompute arrays.

**Complexity Analysis:**
- **Time:** O(n) — each element is visited exactly once
- **Space:** O(1) — only a few integer variables

## Deep Dive: How the Two-Pointer Water Trapping Works

### Visual Walkthrough

For `height = [0, 1, 0, 2, 1, 0, 1, 3, 2, 1, 2, 1]`:

```
Initial: left=0, right=11, maxLeft=0, maxRight=0, water=0
height: [0, 1, 0, 2, 1, 0, 1, 3, 2, 1, 2, 1]
         ↑                                        ↑
       left                                     right

Step 1: height[0]=0 < height[11]=1
        height[0] >= maxLeft(0)? Yes → maxLeft=0
        water += 0 - 0 = 0, left=1

Step 2: height[1]=1 < height[11]=1? No (equal)
        height[11] >= maxRight(0)? Yes → maxRight=1
        water += 0, right=10

Step 3: height[1]=1 < height[10]=2? Yes
        height[1] >= maxLeft(0)? Yes → maxLeft=1
        water += 0, left=2

Step 4: height[2]=0 < height[10]=2? Yes
        height[2] >= maxLeft(1)? No → water += 1-0 = 1
        water=1, left=3

... continues ...
```

### The Core Insight

The amount of water at any position is determined by the **shorter** of the two max heights on either side. By always processing the side with the smaller current height, we know the other side has a taller (or equal) wall, so the water level is bounded by the current side's max.

### Three Approaches Compared

| Approach | Time | Space | Notes |
|----------|------|-------|-------|
| Brute Force | O(n²) | O(1) | For each position, scan left and right for max |
| DP Arrays | O(n) | O(n) | Precompute leftMax and rightMax arrays |
| Two Pointers | O(n) | O(1) | Track running maxes on the fly — optimal |

### Edge Cases

- **Ascending/descending arrays**: No water trapped (water runs off)
- **Flat arrays**: No water trapped
- **Single/two elements**: No water trapped
- **All zeros**: No water trapped
- **Symmetrical valleys**: Maximum water trapped in the center

class Solution {
    public int trap(int[] height) {
        if (height.length == 0) return 0;           // Edge case: empty array
        int left = 0;                                 // Left pointer starts at beginning
        int right = height.length - 1;                // Right pointer starts at end
        int maxLeft = 0;                              // Track max height seen from left
        int maxRight = 0;                             // Track max height seen from right
        int water = 0;                                // Total water accumulated

        while (left < right) {
            // Process the side with the smaller height
            if (height[left] < height[right]) {
                // Left side is the limiting factor
                if (height[left] >= maxLeft) {
                    // Update maxLeft — no water can be held here
                    maxLeft = height[left];
                } else {
                    // Water trapped = maxLeft - current height
                    water += maxLeft - height[left];
                }
                left++;  // Move left pointer inward
            } else {
                // Right side is the limiting factor (or equal)
                if (height[right] >= maxRight) {
                    // Update maxRight — no water can be held here
                    maxRight = height[right];
                } else {
                    // Water trapped = maxRight - current height
                    water += maxRight - height[right];
                }
                right--;  // Move right pointer inward
            }
        }
        return water;
    }
}

## Key Takeaways

- **Two pointers from opposite ends** with running max tracking is the optimal O(n) time, O(1) space solution
- The **shorter side determines the water level** — this is the core insight
- Always process the side with the smaller height; the running max on that side is the water level at that position
- The two-pointer approach avoids the O(n) space needed for precomputed arrays
- This is a classic problem that tests understanding of how water accumulates between boundaries
