# Design

The Design pattern focuses on creating data structures and classes that satisfy specific constraints using optimal time and space complexity.

## What You Will Learn

- Object-oriented design principles for coding problems
- Data structure design patterns (LRU, LFU, Min Stack)
- Trade-offs between time and space efficiency
- Common design interview problems

## The Concept

Design problems ask you to implement a data structure or class that supports specific operations within given time and space constraints. Unlike algorithm problems that focus on processing input to produce output, design problems require you to architect a solution from the ground up, choosing the right combination of data structures to meet all performance requirements.

## Key Patterns

### 1. Min Stack

**Problem:** Design a stack that supports push, pop, top, and retrieving the minimum element in O(1) time.

**Solution:** Maintain two stacks — one for all elements and one that tracks the current minimum. When pushing, push the new min onto the min stack. When popping, also pop from the min stack if the top was the minimum.

```
push(5)    stack: [5]    min_stack: [5]
push(2)    stack: [5,2]  min_stack: [5,2]
push(8)    stack: [5,2,8] min_stack: [5,2]  (8 > 2, no change)
push(1)    stack: [5,2,8,1] min_stack: [5,2,1]
pop()      → pops 1     stack: [5,2,8]   min_stack: [5,2]
getMin()   → 2
```

### 2. LRU Cache

**Problem:** Design a cache with O(1) get and put operations that evicts the least recently used item when capacity is exceeded.

**Solution:** Combine a hash map (for O(1) lookup) with a doubly linked list (for O(1) reordering). The hash map maps keys to list nodes. When a key is accessed, its node is moved to the head of the list. When the cache is full, the tail node (least recently used) is removed.

```
Linked List (most recent ← head ... tail → least recent):
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ 3 │<-->│ 1 │<-->│ 2 │<-->│ 4 │
  └───┘    └───┘    └───┘    └───┘
  head                          tail

HashMap: {1: node1, 2: node2, 3: node3, 4: node4}

get(1) → move 1 to head:
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ 1 │<-->│ 3 │<-->│ 2 │<-->│ 4 │
  └───┘    └───┘    └───┘    └───┘
  head                          tail
```

### 3. LFU Cache

**Problem:** Design a cache that evicts the least frequently used item. If there is a tie, evict the least recently used among them.

**Solution:** Use a hash map for O(1) key lookup, plus a frequency map where each frequency bucket has its own doubly linked list tracking recency. Also maintain a `minFreq` counter. When a key is accessed, its frequency increases and it moves to the next frequency bucket. When evicting, remove from the `minFreq` bucket's least recent position.

```
Frequency Map:
  freq=1: [key2 <--> key5]    (LRU order within bucket)
  freq=2: [key1 <--> key3]
  freq=3: [key4]

minFreq = 1
evict → key2 (least recently used among lowest frequency)
```

## Common Design Problems

| Problem | Core Challenge | Key Insight |
|---------|---------------|-------------|
| LRU Cache | O(1) get/put + eviction | HashMap + doubly linked list |
| LFU Cache | O(1) operations + frequency tracking | HashMap + frequency buckets |
| Min Stack | O(1) getMin | Auxiliary min tracking stack |
| Browser History | O(1) back/forward | Two stacks or doubly linked list |
| HashMap | O(1) average operations | Array + chaining |
| Insert Delete GetRandom O(1) | O(1) random access | HashMap + array |
| Design Tic-Tac-Toe | Win detection efficiency | Row/col/diagonal counters |
| Design Snake Game | Movement and collision | Deque + coordinate set |
| Design Hit Counter | Sliding window counting | Queue + timestamp buckets |
| Design Twitter | Feed generation + follow | HashMap + heaps (merge) |

## Designing for Time and Space Efficiency

### Strategy 1: Combine Complementary Structures

No single data structure is optimal for all operations. The key insight is combining structures that excel at different aspects:

- **HashMap** → O(1) lookups, but no ordering
- **Linked List** → O(1) insert/delete at known position, but O(n) lookup
- **Stack/Queue** → O(1) push/pop, but limited access
- **Heap** → O(log n) min/max retrieval

By combining them, you get the best of both worlds. For example, LRU Cache uses a hash map for O(1) lookup and a doubly linked list for O(1) ordering.

### Strategy 2: Track Metadata Alongside Data

When you need to answer questions about your data (like "what's the minimum?" or "how many times has this been accessed?"), maintain parallel structures:

- **Min Stack** → Maintain a separate stack tracking the current minimum
- **LFU Cache** → Maintain a frequency counter for each key
- **Randomized Set** → Maintain both a hash map (value→index) and an array (values)

### Strategy 3: Trade Space for Time

Most design problems require optimizing for time at the cost of memory. This is generally acceptable:

- Use extra hash maps to avoid O(n) lookups
- Use auxiliary stacks to track statistics
- Use linked lists to track ordering with O(1) reordering

The typical interview constraint is that O(n) extra space is acceptable if operations are O(1).

## Worked Example: Design an LRU Cache

**Problem:** Design a data structure that supports get(key) and put(key, value) in O(1) average time. The cache has a fixed capacity. When full, evict the least recently used item.

**Step 1: Identify requirements**
- `get(key)` → return value, mark as recently used
- `put(key, value)` → insert or update, mark as recently used
- Evict LRU item when capacity exceeded
- All operations must be O(1)

**Step 2: Choose data structures**

For O(1) operations, we need:
- **HashMap** for O(1) key lookup
- **Doubly Linked List** for O(1) reordering (move to front) and O(1) tail removal (for eviction)

**Step 3: Design the interface**

```python
class LRUCache:
    # capacity: maximum number of items
    # get(key): return value, move to front
    # put(key, value): insert/update, move to front, evict if needed
```

**Step 4: Implement the linked list wrapper**

Each node stores key, value, prev, next. The list has dummy head (most recent) and tail (least recent) sentinel nodes to avoid null checks.

```python
class _Node:
    def __init__(self, key=0, value=0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None
```

**Step 5: Implement get**

1. Look up key in hash map → O(1)
2. If found: move node to head (remove from current position, add after dummy head) → O(1)
3. If not found: return -1

**Step 6: Implement put**

1. If key exists: update value, move node to head
2. If key does not exist:
   a. Create new node
   b. Add to hash map
   c. Add to head of linked list
   d. If size > capacity: remove tail node, delete from hash map

**Complexity:** O(1) time per operation, O(capacity) space.
