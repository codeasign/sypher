# Sum of Two Integers — Solution

## Problem Overview

Given two integers `a` and `b`, return their sum without using the `+` or `-` operators. You must implement addition using only bit manipulation.

## Brute Force Approach

There is no meaningful brute force approach for this problem — the entire challenge is implementing addition without arithmetic operators. Any solution must use bit manipulation.

## Optimal Approach: Bitwise Addition with Carry

**Idea:** Break addition into two parts — sum without carry (XOR) and carry (AND with shift). Repeat until the carry is 0.

**Why This Works:**
Binary addition at the bit level works as follows:
- `0 + 0 = 0`, carry 0
- `0 + 1 = 1`, carry 0
- `1 + 0 = 1`, carry 0
- `1 + 1 = 0`, carry 1

This is exactly:
- **Sum without carry:** `a ^ b` (XOR)
- **Carry:** `(a & b) << 1` (AND, then shift left)

By repeating these two operations until the carry is 0, we perform full binary addition.

**Algorithm:**
1. While `b != 0`:
   - Compute `carry = (a & b) << 1` — the bits that need to carry over
   - Compute `a = a ^ b` — the sum without carry
   - Set `b = carry` — the carry to add in the next iteration
2. Return `a`

**Trace for `a = 2, b = 3`:**
```
Iteration 1:
  a = 2 (010), b = 3 (011)
  carry = (010 & 011) << 1 = 010 << 1 = 100 (4)
  a = 010 ^ 011 = 001 (1)
  b = 100 (4)

Iteration 2:
  a = 1 (001), b = 4 (100)
  carry = (001 & 100) << 1 = 000 << 1 = 000 (0)
  a = 001 ^ 100 = 101 (5)
  b = 0

Result: a = 5
```

**Complexity Analysis:**
- **Time:** O(1) — at most 32 iterations for 32-bit integers, or 64 iterations for 64-bit
- **Space:** O(1) — only two integer variables

## Deep Dive: Why This Works for Negative Numbers

### Two's Complement

Negative numbers are represented in two's complement form. For example, in 8 bits:
- `-1` is `11111111`
- `-2` is `11111110`

The XOR-and-carry algorithm works correctly with two's complement because the bitwise operations operate on the underlying binary representation, which naturally handles signed arithmetic.

**Trace for `a = -1, b = 1` (in 8-bit representation):**
```
a = -1 (11111111), b = 1 (00000001)
carry = (11111111 & 00000001) << 1 = 00000001 << 1 = 00000010
a = 11111111 ^ 00000001 = 11111110 (-2)
b = 00000010 (2)

carry = (11111110 & 00000010) << 1 = 00000010 << 1 = 00000100
a = 11111110 ^ 00000010 = 11111100 (-4)
b = 00000100 (4)

... pattern continues until the carry propagates off the high end ...
Final: a = 0
```

### Number of Iterations

The algorithm converges because each iteration moves the carry bits one position to the left. After at most 32 iterations (for 32-bit ints), all carries have propagated and `b` becomes 0.

### Edge Cases

- **Both numbers zero:** The loop doesn't execute; returns 0
- **One number zero:** XOR returns the other number; AND gives 0; loop ends; sum is correct
- **Both numbers negative:** Works identically since two's complement handles negative addition the same way

class Solution {
    public int getSum(int a, int b) {
        while (b != 0) {                 // Repeat until no carry remains
            int carry = (a & b) << 1;    // Compute carry: bits that overflow
            a = a ^ b;                   // Sum without carry (XOR)
            b = carry;                   // Assign carry for next iteration
        }

def get_sum(a: int, b: int) -> int:
    # Python integers are arbitrarily large, so we need to mask to 32 bits
    MASK = 0xFFFFFFFF
    MAX_INT = 0x7FFFFFFF

    while b != 0:                    # Loop until no carry remains
        carry = (a & b) << 1         # Compute carry bits
        a = (a ^ b) & MASK           # Sum without carry, mask to 32 bits
        b = carry & MASK             # Mask carry to 32 bits

    # If a is negative (bit 31 is set), convert to Python negative
    return a if a <= MAX_INT else ~(a ^ MASK)

## Key Takeaways

- **XOR gives sum without carry**, AND gives carry — the fundamental insight for bitwise addition
- The loop converges because carry bits always shift left, eventually falling off the high end
- The algorithm works for negative numbers because of two's complement representation
- Python requires extra masking because its integers are arbitrarily large
- This pattern appears in hardware (full adders) and is the basis for ALU addition circuits
