# Counting Bits — Solution

## Problem Overview

Given an integer `n`, return an array of length `n + 1` where `ans[i]` is the number of 1 bits in the binary representation of `i` for all `0 ≤ i ≤ n`.

## Brute Force Approach

**Idea:** For each number from 0 to n, count the number of 1 bits by iterating through all 32 bits.

**Algorithm:**
1. Create an array `ans` of size `n + 1`
2. For each `i` from 0 to n:
   - Count bits by checking each bit position (e.g., `(i >> k) & 1`)
   - Store the count in `ans[i]`
3. Return `ans`

**Complexity Analysis:**
- **Time:** O(n * log n) — for each of n numbers, we check O(log n) bits
- **Space:** O(n) — for the output array (required by the problem)

**Why this could be better:** The brute force approach redoes work for each number. The optimal approach uses a recurrence relation to compute the answer in O(1) per number.

## Optimal Approach: Dynamic Programming with Bit Pattern

**Idea:** Observe that the number of 1 bits in `i` has a relationship with `i >> 1` (integer division by 2):

- If `i` is even: `countBits(i) = countBits(i >> 1)` — shifting right drops a 0
- If `i` is odd: `countBits(i) = countBits(i >> 1) + 1` — shifting right drops a 1

More concisely: `countBits(i) = countBits(i >> 1) + (i & 1)`

**Algorithm:**
1. Create an array `ans` of size `n + 1`
2. Set `ans[0] = 0`
3. For each `i` from 1 to n:
   - `ans[i] = ans[i >> 1] + (i & 1)`
4. Return `ans`

**Why This Works:**
- `i >> 1` is the same as `i // 2`, which is a smaller number whose bit count we already computed
- `i & 1` is 1 if `i` is odd, 0 if `i` is even
- This gives us O(1) computation per number using previously computed results

**Trace for `n = 5`:**
```
ans[0] = 0
ans[1] = ans[0] + 1 = 1  (1 is odd)
ans[2] = ans[1] + 0 = 1  (2 is even, binary: 10)
ans[3] = ans[1] + 1 = 2  (3 is odd, binary: 11)
ans[4] = ans[2] + 0 = 1  (4 is even, binary: 100)
ans[5] = ans[2] + 1 = 2  (5 is odd, binary: 101)
```

**Complexity Analysis:**
- **Time:** O(n) — single pass from 1 to n
- **Space:** O(n) — for the output array (required by the problem)

## Deep Dive: Alternative DP Formulation

### Using the Lowest Set Bit

Another recurrence: `ans[i] = ans[i & (i - 1)] + 1`

- `i & (i - 1)` removes the lowest set bit from `i`
- This number is smaller than `i`, so its bit count is already computed
- We add 1 for the removed bit

This is slightly more intuitive as it directly relates to Kernighan's algorithm.

### Trace for `n = 5` with this formulation:
```
ans[0] = 0
ans[1] = ans[1 & 0] + 1 = ans[0] + 1 = 1
ans[2] = ans[2 & 1] + 1 = ans[0] + 1 = 1
ans[3] = ans[3 & 2] + 1 = ans[2] + 1 = 2
ans[4] = ans[4 & 3] + 1 = ans[0] + 1 = 1
ans[5] = ans[5 & 4] + 1 = ans[4] + 1 = 2
```

### Edge Cases

- `n = 0`: Returns `[0]` — the loop from 1 to n doesn't execute
- `n = 1`: Returns `[0, 1]` — only one iteration of the loop

## Key Takeaways

- **DP with bit patterns** reduces the per-element computation from O(log n) to O(1)
- The recurrence `ans[i] = ans[i >> 1] + (i & 1)` is derived from the observation that shifting right preserves all bits except the LSB
- A similar recurrence `ans[i] = ans[i & (i - 1)] + 1` uses the "remove lowest set bit" trick
- Both approaches achieve O(n) total time complexity
