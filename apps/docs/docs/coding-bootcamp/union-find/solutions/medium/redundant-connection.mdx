# Redundant Connection — Solution

## Problem Overview

In a graph of `n` nodes labeled from `1` to `n`, you are given an array `edges` where `edges[i] = [u, v]` indicates an undirected edge between `u` and `v`. The graph started as a tree (connected, `n` nodes, `n - 1` edges), but one extra edge was added, creating a cycle. Find the edge that can be removed to restore the graph to a tree. If there are multiple answers, return the edge that occurs last in the input.

**Example:**
```
Input: edges = [[1,2],[1,3],[2,3]]
Output: [2,3]
```
The tree has nodes 1, 2, 3. The edge `[2,3]` creates a cycle among `1-2-3-1`.

## Brute Force Approach

For each edge in reverse order, try removing it and check if the remaining graph is a tree (connected and acyclic using DFS).

**Algorithm:**
1. Iterate over edges from right to left.
2. Temporarily remove the current edge.
3. Run DFS from node 1 to see if all nodes are reachable without the edge.
4. If the graph remains connected and acyclic, return the removed edge.
5. Restore the edge and continue.

**Complexity:** O(E * (V + E)) time (for each edge, we run a DFS over the entire graph). O(V + E) space for the adjacency list.

**Why Union-Find is better:** Union-Find detects cycles in O(E * alpha(V)) time with a single pass — no need to repeatedly check connectivity. When we attempt to union two nodes that are already in the same set, the edge is a redundant connection.

## Optimal Approach: Union-Find

Union-Find is the ideal data structure for cycle detection in undirected graphs. As we process edges in order, we union the two endpoints. If `find(u) == find(v)` before the union, then `u` and `v` are already connected — adding this edge creates a cycle, so it is the redundant connection.

**Algorithm:**
1. Initialize Union-Find with `n + 1` elements (1-indexed).
2. For each edge `[u, v]` in `edges`:
   - If `find(u) == find(v)`, return `[u, v]` (this edge creates a cycle).
   - Otherwise, `union(u, v)`.
3. Return an empty array (should not happen per the problem constraints).

**Complexity:** O(n + E * alpha(n)) time, where alpha(n) is the inverse Ackermann function. O(n) space for the `parent` and `rank` arrays.

## Deep Dive

Walk through the example `edges = [[1,2],[2,3],[3,4],[1,4],[1,5]]`:

```
Initial: parent = [0, 1, 2, 3, 4, 5]  (index 0 unused)

Edge [1,2]: find(1)=1, find(2)=2 → different, union
  parent = [0, 1, 1, 3, 4, 5]

Edge [2,3]: find(2)=1, find(3)=3 → different, union
  parent = [0, 1, 1, 1, 4, 5]

Edge [3,4]: find(3)=1, find(4)=4 → different, union
  parent = [0, 1, 1, 1, 1, 5]

Edge [1,4]: find(1)=1, find(4)=1 → SAME root → cycle!
  Return [1, 4] ✅

The edge [1,4] is redundant because nodes 1 and 4 are already
connected through 1-2-3-4.
```

class Solution:
    def findRedundantConnection(self, edges: list[list[int]]) -> list[int]:
        n = len(edges)
        parent = list(range(n + 1))  # 1-indexed
        rank = [0] * (n + 1)

        def find(x: int) -> int:
            if parent[x] != x:
                parent[x] = find(parent[x])
            return parent[x]

        def union(x: int, y: int) -> bool:
            px, py = find(x), find(y)
            if px == py:
                return False  # Already connected — cycle detected
            if rank[px] < rank[py]:
                parent[px] = py
            elif rank[px] > rank[py]:
                parent[py] = px
            else:
                parent[py] = px
                rank[px] += 1
            return True

        for u, v in edges:
            if not union(u, v):
                return [u, v]

        return []  # Should not happen

class Solution {

        int n = edges.length;
        int[] parent = new int[n + 1];
        int[] rank = new int[n + 1];
        for (int i = 1; i <= n; i++) parent[i] = i;

impl Solution {
    pub fn find_redundant_connection(edges: Vec<Vec<i32>>) -> Vec<i32> {
        let n = edges.len();
        let mut parent: Vec<usize> = (0..=n).collect();
        let mut rank = vec![0; n + 1];

        fn find(parent: &mut Vec<usize>, x: usize) -> usize {
            if parent[x] != x {
                parent[x] = find(parent, parent[x]);
            }
            parent[x]
        }

        fn union(parent: &mut Vec<usize>, rank: &mut Vec<i32>, x: usize, y: usize) -> bool {
            let px = find(parent, x);
            let py = find(parent, y);
            if px == py { return false; }  // Already connected
            if rank[px] < rank[py] { parent[px] = py; }
            else if rank[px] > rank[py] { parent[py] = px; }
            else { parent[py] = px; rank[px] += 1; }
            true
        }

        for edge in &edges {
            let u = edge[0] as usize;
            let v = edge[1] as usize;
            if !union(&mut parent, &mut rank, u, v) {
                return vec![u as i32, v as i32];  // Cycle detected
            }

## Key Takeaways

- Union-Find is the optimal tool for cycle detection in undirected graphs — a single pass with nearly O(1) operations per edge.
- The key insight: before unioning two nodes, check if they already share a root. If they do, the current edge creates a cycle.
- This problem is a direct application of the union-find cycle-detection algorithm. The same approach works for finding minimum spanning trees (Kruskal's algorithm).
- Processing edges in input order and returning the first redundant edge naturally satisfies the "return the last edge in the input" requirement.
