# Coin Change — Solution

## Problem Overview

Given coin denominations and a target amount, find the minimum number of coins needed to make that amount. You have an unlimited supply of each coin denomination. This is the classic **unbounded knapsack** problem: each coin can be used any number of times, and we want to minimize the count.

## Brute Force Approach: Recursion

**Idea:** For each amount, recursively try every coin denomination and find the minimum.

**Algorithm:**
1. If `amount == 0`, return 0
2. If `amount < 0`, return infinity (impossible)
3. For each coin, recursively compute `1 + coin_change(coins, amount - coin)` and take the minimum

**Complexity Analysis:**
- **Time:** O(n^amount) where n is the number of coins — exponential explosion
- **Space:** O(amount) — recursion stack depth

**Why this is inefficient:** The recursion explores every possible combination of coins, leading to massive redundancy. For coins=[1,2,5] and amount=11, the recursion tree has thousands of nodes for a problem that only needs 12 DP states.

## Optimal Approach: Dynamic Programming (Bottom-Up)

**Idea:** Build a DP array where `dp[i]` is the minimum number of coins needed to make amount `i`. Start from 0 and work upward.

**Algorithm:**
1. Create an array `dp` of size `amount + 1`, initialized to `amount + 1` (a sentinel larger than any possible answer)
2. Set `dp[0] = 0` (zero coins needed to make amount 0)
3. For each amount `i` from 1 to `amount`:
   - For each coin denomination `c`:
     - If `c <= i`, then `dp[i] = min(dp[i], dp[i - c] + 1)`
4. If `dp[amount] > amount`, return `-1`; otherwise return `dp[amount]`

**Why This Works:** The recurrence `dp[i] = min(dp[i], dp[i - c] + 1)` captures the idea that the optimal way to make amount `i` is to take one coin of denomination `c` and add it to the optimal way to make the remaining amount `i - c`. Since we try all coins and all amounts in increasing order, every subproblem is solved before it is needed.

**Complexity Analysis:**
- **Time:** O(amount * n) — nested loops over amounts and coins
- **Space:** O(amount) — the DP array of size amount + 1

## Deep Dive: Understanding the DP Table

### Visual Walkthrough for coins=[1,2,5], amount=11

```
Initial: dp = [0, INF, INF, INF, INF, INF, INF, INF, INF, INF, INF, INF]

Processing i=1:
  coin 1: dp[1] = min(INF, dp[0]+1) = 1
  coin 2: 2 > 1, skip
  coin 5: 5 > 1, skip
  dp[1] = 1

Processing i=2:
  coin 1: dp[2] = min(INF, dp[1]+1) = 2
  coin 2: dp[2] = min(2, dp[0]+1) = 1  ← better!
  coin 5: 5 > 2, skip
  dp[2] = 1

Processing i=3:
  coin 1: dp[3] = min(INF, dp[2]+1) = 2
  coin 2: dp[3] = min(2, dp[1]+1) = 2
  coin 5: 5 > 3, skip
  dp[3] = 2

... (continues) ...

Final dp[11] = 3  (5 + 5 + 1)
```

### Edge Cases to Consider

- **amount = 0**: Return 0 immediately. No coins needed.
- **No combination possible**: If `dp[amount]` remains at the sentinel value (`amount + 1`), return -1.
- **Single coin denomination**: If amount is a multiple of the coin, the answer is `amount / coin`. Otherwise return -1.
- **Large amounts (up to 10^4)**: The DP array is at most 10,001 elements, well within memory limits.

### The Greedy Trap

A common mistake is to try a greedy approach: use the largest coin as many times as possible, then move to the next largest. This fails for some coin systems. For example, coins=[1,3,4] and amount=6: greedy gives 4+1+1=3 coins, but the optimal is 3+3=2 coins. DP correctly finds the minimum.

### Optimization: Sorting Coins

While not required for correctness, sorting coins in ascending order can help with early termination in the inner loop (break when `coin > i`). This is a minor optimization that doesn't change the worst-case complexity.

```java
class Solution {
    public int coinChange(int[] coins, int amount) {
        // dp[i] = minimum coins to make amount i
        int[] dp = new int[amount + 1];
        // Initialize with a value larger than the maximum possible answer
        // amount+1 works because you can't use more than 'amount' coins of value 1
        java.util.Arrays.fill(dp, amount + 1);
        dp[0] = 0;  // Base: 0 coins needed to make amount 0

        // Build up from amount 1 to the target
        for (int i = 1; i <= amount; i++) {
            // Try each coin denomination
            for (int coin : coins) {
                if (coin <= i) {
                    // Take one 'coin' and add to the optimal solution for the remainder
                    dp[i] = Math.min(dp[i], dp[i - coin] + 1);
                }
            }
        }

        // If dp[amount] is still the sentinel, no combination was found
        return dp[amount] > amount ? -1 : dp[amount];
    }
}
```

```python
def coin_change(coins: list[int], amount: int) -> int:
    # dp[i] = minimum coins needed to make amount i
    # Use 'amount + 1' as a sentinel (larger than any possible answer)
    dp = [amount + 1] * (amount + 1)
    dp[0] = 0  # Base case: 0 coins for amount 0

    # Build bottom-up from 1 to amount
    for i in range(1, amount + 1):
        for coin in coins:
            if coin <= i:
                # dp[i] = min(dp[i], dp[i - coin] + 1)
                dp[i] = min(dp[i], dp[i - coin] + 1)

    # If dp[amount] is still the sentinel, return -1
    return dp[amount] if dp[amount] <= amount else -1
```

```rust
fn coin_change(coins: Vec<i32>, amount: i32) -> i32 {
    let amt = amount as usize;
    // DP array: minimum coins for each amount, sentinel = amount + 1
    let mut dp = vec![amount + 1; amt + 1];
    dp[0] = 0;  // Base case

    // Compute minimum coins for amounts 1..amount
    for i in 1..=amt {
        for &coin in &coins {
            let c = coin as usize;
            if c <= i {
                // Recurrence: 1 + best for remaining amount
                dp[i] = std::cmp::min(dp[i], dp[i - c] + 1);
            }
        }
    }

    // Return -1 if impossible, otherwise the answer
    if dp[amt] > amount { -1 } else { dp[amt] }
}
```

```c
int coinChange(int* coins, int coinsSize, int amount) {
    // Allocate DP array on the heap (amount may be up to 10^4)
    int* dp = (int*)calloc(amount + 1, sizeof(int));
    // Initialize with sentinel value
    for (int i = 1; i <= amount; i++) dp[i] = amount + 1;
    dp[0] = 0;  // Base case

    // Compute minimum coins for each amount
    for (int i = 1; i <= amount; i++) {
        for (int j = 0; j < coinsSize; j++) {
            if (coins[j] <= i) {
                // Recurrence: dp[i] = min(dp[i], dp[i - coin] + 1)
                int candidate = dp[i - coins[j]] + 1;
                if (candidate < dp[i]) dp[i] = candidate;
            }
        }
    }

    int result = dp[amount] > amount ? -1 : dp[amount];
    free(dp);  // Clean up heap allocation
    return result;
}
```

## Key Takeaways

- **Unbounded knapsack pattern**: Coins can be reused any number of times. The DP iterates over amounts (outer loop) and coins (inner loop).
- **Sentinel initialization**: Using `amount + 1` as a sentinel avoids the need for a separate `impossible` flag.
- **The recurrence `dp[i] = min(dp[i], dp[i - coin] + 1)`** is the heart of the algorithm.
- **Greedy does NOT work** in general for coin systems. DP guarantees the optimal answer.
- Time complexity O(amount * n) is acceptable for amount up to 10^4 and n up to 100.
- This problem is the minimization version of the unbounded knapsack. The maximization version (counting combinations) uses the same DP structure but sums instead of taking the minimum.
