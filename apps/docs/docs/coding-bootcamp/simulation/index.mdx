# Simulation

Simulation problems require you to **follow a set of rules exactly** and model the behavior of a system step by step. There is no clever math trick or elegant algorithm — the challenge is in faithfully translating the rules into code and managing the state correctly.

## Key Characteristics

- **Rule-following**: The problem describes a process; you implement it precisely.
- **State management**: You track variables, data structures, or environment state as the simulation progresses.
- **No hidden trick**: The difficulty lies in correctly handling edge cases and complex rules, not in finding an optimized algorithm.
- **Iterative or event-driven**: Simulations often run in a loop (time steps) or process events from a queue.

## Complexity

The time and space complexity of simulation problems varies by problem. Most run in O(n) or O(n log n) time, but some can reach O(n²) or more depending on the rules being simulated.

## Common Applications

- **Game state**: Tracking scores, board positions, player turns
- **Process simulation**: Modeling workflows, pipelines, or physical processes
- **Rule following**: Implementing a spec exactly (e.g., text justification rules)
- **Event-driven simulation**: Processing events in chronological order (e.g., collision detection)

## Worked Example: Robot Return to Origin

**Problem**: A robot starts at (0, 0) and receives a string of moves: 'U' (up), 'D' (down), 'L' (left), 'R' (right). Does it return to the origin after all moves?

**Simulation approach**:

1. **Initialize state**: `x = 0, y = 0`
2. **Process each move**:
   - 'U' → y += 1
   - 'D' → y -= 1
   - 'L' → x -= 1
   - 'R' → x += 1
3. **Check final state**: Return `x == 0 and y == 0`

```
Moves: "UDLR"
Start: (0, 0)
  U → (0, 1)
  D → (0, 0)
  L → (-1, 0)
  R → (0, 0)
Result: true (back at origin)
```
