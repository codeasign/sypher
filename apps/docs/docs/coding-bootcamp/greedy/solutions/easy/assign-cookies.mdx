# Assign Cookies — Solution

## Problem Overview

You are a parent with children who each have a **greed factor** `g[i]` (the minimum cookie size they will accept). You have cookies of various sizes `s[j]`. Each child gets at most one cookie. A child is content if `s[j] >= g[i]`. Maximize the number of content children.

## Brute Force Approach

**Idea:** Try every possible assignment of cookies to children and track the maximum number of content children.

**Algorithm:**
1. Generate all subsets and permutations of assigning cookies to children
2. For each valid assignment, count how many children get a cookie `s[j] >= g[i]`
3. Return the maximum count found

**Complexity Analysis:**
- **Time:** O(m! / (m-n)!) — factorial complexity from trying all assignments
- **Space:** O(n) — recursion stack depth

**Why this could be better:** The brute force approach is computationally infeasible for inputs up to 3 * 10^4. The greedy approach solves this in O(n log n) time.

## Optimal Approach: Greedy with Two Pointers

**Idea:** Sort both arrays. Assign the **smallest cookie that satisfies each child** — this leaves larger cookies for greedier children later.

**Algorithm:**
1. Sort `g` (greed factors) and `s` (cookie sizes) in ascending order
2. Initialize `i = 0` (child pointer) and `j = 0` (cookie pointer)
3. While both pointers are within bounds:
   - If `s[j] >= g[i]`, the child is content — advance both pointers
   - Otherwise, the cookie is too small — advance only the cookie pointer
4. Return `i` (number of content children)

**Why This Works:** This is a classic greedy matching problem. By sorting both arrays and always assigning the smallest adequate cookie, we never waste a large cookie on a less greedy child. The two-pointer technique efficiently scans both sorted arrays in a single pass. This greedy choice is optimal because any valid assignment can be transformed into this greedy one without reducing the count (exchange argument).

**Complexity Analysis:**
- **Time:** O(n log n + m log m) — dominated by sorting the two arrays
- **Space:** O(1) — sorting may use O(log n) stack space depending on the implementation

## Deep Dive: How the Greedy Strategy Works

### Visual Walkthrough

For `g = [1, 2, 3]`, `s = [1, 1]`:

```
Sorted: g = [1, 2, 3],  s = [1, 1]

Step 1: i=0, j=0  →  g[0]=1, s[0]=1  →  1 >= 1 ✓  →  i=1, j=1
Step 2: i=1, j=1  →  g[1]=2, s[1]=1  →  1 >= 2 ✗  →  j=2 (out of bounds)
Result: i = 1 content child
```

For `g = [1, 2]`, `s = [1, 2, 3]`:

```
Sorted: g = [1, 2],  s = [1, 2, 3]

Step 1: i=0, j=0  →  g[0]=1, s[0]=1  →  1 >= 1 ✓  →  i=1, j=1
Step 2: i=1, j=1  →  g[1]=2, s[1]=2  →  2 >= 2 ✓  →  i=2, j=2
Result: i = 2 content children (all satisfied)
```

### Edge Cases

- **Empty children array (`g = []`)**: Loop never executes, `i = 0` is returned
- **Empty cookies array (`s = []`)**: Loop never executes, `i = 0` is returned
- **No cookie large enough**: Cookie pointer advances past all cookies without matching any child, `i` stays at 0
- **All cookies satisfy all children**: Both pointers advance together, `i` equals the smaller of the two array lengths
- **Greed factors equal to cookie sizes**: Each child gets exactly the cookie they need — perfect match

### Why This Greedy Choice Is Optimal

**Exchange argument:** Suppose an optimal solution assigns a large cookie to a child with greed factor `g[a]`, while the greedy approach would give that child a smaller cookie `s[b]` (still satisfying `s[b] >= g[a]`) and give the large cookie to a greedier child `g[c]` (where `g[c] > g[a]`). Swapping these assignments never reduces the number of content children because the large cookie satisfies the greedier child at least as well. Therefore, the greedy strategy is optimal.

def find_content_children(g: list[int], s: list[int]) -> int:
    # Sort both arrays so we can greedily match smallest adequate cookies
    g.sort()
    s.sort()

    i = 0  # pointer for children (greed factors)
    j = 0  # pointer for cookies (sizes)

    # Iterate through both arrays simultaneously
    while i < len(g) and j < len(s):
        # If the current cookie is large enough for the current child
        if s[j] >= g[i]:
            i += 1  # content child — move to next child
        j += 1  # always advance cookie pointer (used or too small)

    return i  # number of content children

import java.util.Arrays;

class Solution {
    public int findContentChildren(int[] g, int[] s) {
        // Sort both arrays for greedy matching
        Arrays.sort(g);
        Arrays.sort(s);

        int i = 0;  // child index
        int j = 0;  // cookie index

        // Scan through both arrays
        while (i < g.length && j < s.length) {
            // If this cookie satisfies this child's greed
            if (s[j] >= g[i]) {
                i++;  // assign the cookie and move to next child
            }
            j++;  // move to next cookie regardless
        }

        return i;  // total content children
    }
}

## Key Takeaways

- **Sorting enables greedy matching** — without sorting, we cannot efficiently find the smallest adequate cookie
- **Two-pointer technique** efficiently scans both sorted arrays in one pass
- **Greedy choice is optimal** because wasting a large cookie on a less greedy child never helps — proven by an exchange argument
- The problem demonstrates the **"assign smallest sufficient resource"** pattern, common in scheduling and matching problems
