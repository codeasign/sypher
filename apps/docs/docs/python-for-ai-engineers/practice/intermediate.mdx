# AI Development & Production Python — Intermediate Practice

⏱ 30 minutes · Difficulty: Intermediate

## Exercise 1: Structured Output Extraction

**Problem:** You have a batch of raw product descriptions from an e-commerce site. Write a function `extract_products(descriptions: list[str]) -> list[dict]` that uses an LLM with structured output to extract product name, price (float), category, and availability status from each description.

Given these descriptions:

```python
descriptions = [
    "The new MacBook Pro 16-inch with M4 chip, priced at $2499, in stock. Category: Electronics.",
    "Organic Arabica coffee beans, 12oz bag for $18.99, currently out of stock. Category: Food & Beverage.",
    "Ergonomic office chair with lumbar support, available for $349.50, ships within 5 business days. Category: Furniture.",
]
```

Your function should return a list of dicts with keys: `product_name`, `price`, `category`, `in_stock`.

**Expected output example:**

```python
[
    {"product_name": "MacBook Pro 16-inch", "price": 2499.0, "category": "Electronics", "in_stock": True},
    {"product_name": "Organic Arabica Coffee Beans", "price": 18.99, "category": "Food & Beverage", "in_stock": False},
    {"product_name": "Ergonomic Office Chair", "price": 349.50, "category": "Furniture", "in_stock": True},
]
```

**Hints:**
- Define a Pydantic `Product` model with the four fields.
- Use Anthropic tool use or OpenAI structured output with the schema.
- Parse the LLM response into a `Product` instance and validate it.
- Handle the case where the description is ambiguous — the model may not always produce clean prices.

**Constraints:** Use Pydantic for schema definition. Use structured output (tool use or response_format), not free-text prompting + regex.

## Exercise 2: RAG Pipeline for Code Documentation

**Problem:** You have a set of technical documentation pages for a Python library. Build a RAG pipeline that answers questions based on these documents. The documents are:

```python
docs = {
    "auth": "The authentication module provides API key and OAuth2 support. "
            "Use Authenticator(api_key='...') for API key auth. "
            "For OAuth2, use Authenticator(oauth_token='...', provider='google'). "
            "Tokens expire after 3600 seconds and are refreshed automatically.",
    "rate_limits": "The API enforces rate limits: 100 requests per minute for free tier, "
                   "1000 requests per minute for pro tier. "
                   "Rate limit headers are returned in every response: "
                   "X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset.",
    "errors": "The API returns standard HTTP errors. 400 means bad request, "
              "401 means unauthorized, 403 means forbidden, 429 means rate limited, "
              "500 means internal server error. "
              "Error responses include a JSON body with 'error' and 'message' fields.",
}
```

Write a function `answer_from_docs(question: str) -> str` that:
1. Chunks each doc into passages
2. Embeds them with `text-embedding-3-small`
3. Stores them in Chroma
4. Retrieves the most relevant passage for the question
5. Generates an answer using Claude or GPT-4o

**Expected input:** `answer_from_docs("How do I authenticate?")`

**Expected output:** An answer that cites the relevant passage about the `Authenticator` class.

**Hints:**
- Each document is already short — you may not need chunking for this small set.
- Use the `openai` embeddings API and Chroma (or an in-memory list).
- Include the retrieved passage in the LLM prompt as context.
- If no relevant passage is found, return "I don't have information about that."

**Constraints:** Must use embeddings for retrieval (not keyword search). Must include the source document name in the answer. Must handle the case where no relevant document is found.

## Exercise 3: CI/CD Pipeline for Testing

**Problem:** Write a GitHub Actions workflow that:
1. Runs on every push and pull request to `main`
2. Installs Python 3.12 and dependencies from `requirements.txt`
3. Runs Ruff linting
4. Runs pytest with coverage (minimum 75%)
5. Only runs tests if linting passes
6. Caches pip packages for speed

You can write the workflow YAML inline. Assume `requirements.txt` exists with `pytest`, `pytest-cov`, `ruff`, `anthropic`, and `openai`.

**Expected output:** A complete `.github/workflows/ci.yml` file.

**Hints:**
- Use `actions/checkout@v4` and `actions/setup-python@v5`.
- Use `actions/cache@v4` for pip caching.
- Use `pytest --cov=. --cov-fail-under=75` for the coverage gate.
- Use `needs:` to make tests depend on linting.

**Constraints:** The workflow must be valid YAML. Every dependency must be installed. Linting must run before tests. Coverage threshold is 75%.
