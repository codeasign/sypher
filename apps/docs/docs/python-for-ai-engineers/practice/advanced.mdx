# AI Development & Production Python — Advanced Practice

⏱ 45 minutes · Difficulty: Advanced

## Exercise 1: Multi-Stage Document Analysis Pipeline

**Problem:** Design and implement a multi-stage AI pipeline that processes customer support tickets. Each ticket contains free-text from a customer. The pipeline must:

1. **Classify** the ticket into one of: `billing`, `technical`, `account`, `feature_request`, `other`
2. **Extract** structured data: customer ID (or None), priority (low/medium/high/urgent), product mentioned, and a list of key phrases
3. **Summarise** the ticket in under 50 words
4. **Route** the result: urgent billing tickets go to a "urgent_billing" queue, all technical tickets go to a "technical" queue, everything else goes to a "general" queue
5. **Store** the result with a timestamp, status, and queue assignment

Write the pipeline as a Python module with:

- A `PipelineStage` decorator or base class with retry logic (max 2 retries, exponential backoff)
- A `TicketPipeline` orchestrator that runs all stages and collects per-ticket results
- Pydantic models for the intermediate output of each stage
- Error handling per stage (one stage failing does not crash the whole pipeline)
- A `PipelineResult` dataclass tracking: ticket ID, success/failure, stages completed, error message, result data

Test data:

```python
tickets = [
    {"id": "T001", "body": "I was charged twice for my subscription last month. Please refund the extra charge immediately!", "customer_id": "C12345"},
    {"id": "T002", "body": "The dashboard is not loading after I click 'View Reports'. I get a blank white screen.", "customer_id": None},
    {"id": "T003", "body": "I would love to see a dark mode option in the mobile app. Also, please add CSV export.", "customer_id": "C67890"},
    {"id": "T004", "body": "Can you reset my password? I forgot it and cannot log in.", "customer_id": "C11111"},
]
```

**Expected output:** When `run_pipeline(tickets)` is called, it returns a list of `PipelineResult` objects, one per ticket, with all stages completed. A summary printout shows:

```
Pipeline complete: 4 succeeded, 0 failed
  T001: OK (stages: 5) → queue: urgent_billing
  T002: OK (stages: 5) → queue: technical
  T003: OK (stages: 5) → queue: general
  T004: OK (stages: 5) → queue: general
```

**Hints:**
- Each stage should be a separate function with the `@pipeline_stage` decorator.
- Use Pydantic models for typed data passing between stages.
- Use an LLM (Anthropic or OpenAI) for the classification, extraction, and summarisation stages.
- Simulate the queue routing — no actual queue system needed, just assign the queue name.
- The `store` stage can be a simple in-memory list or dict.

**Constraints:** All five stages must be implemented. Each stage must have its own error handling. The orchestrator must handle partial failures (a ticket that fails at stage 3 should report 2 stages completed, not crash the pipeline). Use valid Python with no syntax errors.

## Exercise 2: Production-Ready Python Package with CI/CD

**Problem:** You need to publish the pipeline from Exercise 1 as a pip-installable package with automated CI/CD. Design and implement:

1. **Package structure** with `pyproject.toml`:
   - Package name: `ticket-pipeline`
   - Dependencies: `anthropic>=0.30.0`, `pydantic>=2.0.0`, `python-dotenv>=1.0.0`
   - Optional dependency group `[dev]` with `pytest`, `pytest-cov`, `ruff`, `mypy`
   - A console script entry point `ticket-process` that runs the pipeline on a JSON file
   - Python 3.11+ required

2. **Tests** that cover:
   - Each pipeline stage in isolation (with mocked LLM calls)
   - The full pipeline with test tickets
   - Error handling (simulate an API failure, verify the pipeline handles it gracefully)
   - Edge cases: empty ticket body, missing customer ID, all classifications

3. **CI/CD workflow** (`.github/workflows/main.yml`):
   - On push/PR to main: lint (Ruff), type check (mypy), test (pytest with 80% coverage)
   - On tagged release (`v*`): build wheel, publish to PyPI

**Expected output:** A complete project directory with:
- `src/ticket_pipeline/__init__.py`, `stages.py`, `orchestrator.py`, `cli.py`
- `src/ticket_pipeline/models.py` with all Pydantic models
- `tests/test_stages.py`, `tests/test_orchestrator.py`
- `pyproject.toml`
- `.github/workflows/main.yml`

**Hints:**
- Use `unittest.mock.MagicMock()` to simulate LLM responses in tests.
- The CLI entry point should read JSON from stdin or a file path argument.
- Use `pytest --cov=src --cov-fail-under=80` for coverage enforcement.
- The publish job should only run on tags matching `v*`.

**Constraints:** Every file must be syntactically valid. The package must be installable with `pip install -e .`. Tests must pass with mocked LLM calls (no real API keys needed). The YAML must be valid GitHub Actions syntax.
