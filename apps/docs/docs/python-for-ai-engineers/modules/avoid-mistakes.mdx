# Modules — Common Mistakes

⏱ 6 minutes

## When NOT to Use a Module

- **One-off scripts:** If a function is only used in one file and is less than 20 lines, keep it in that file. Extracting every helper into a separate module adds complexity without benefit.
- **Configuration:** Use JSON, YAML, or TOML files for configuration, not Python modules that get imported. Configuration as Python code creates security considerations when loading untrusted configs.
- **Very small projects:** A 3-file project with 50 lines total does not need a package structure.

## The Pattern to Remember

- **Pattern:** Each `.py` file is a module. A directory of modules with `__init__.py` is a package. Use `from package import module` for clean namespacing.
- **When to use:** Any project with more than 2 files, any reusable code, any code that needs testing.
- **When NOT to use:** Single-file scripts, static configuration (use data files instead).
- **Mental model:** A module is a toolbox. You keep related tools together, put a label on the box, and reach for it when you need those tools.

## Common Mistakes

### Mistake 1: Circular Imports

```python
# a.py
from b import bar

def foo():
    print("foo")
    bar()

# b.py
from a import foo

def bar():
    print("bar")
    foo()
```

**Why it happens:** Module `a` imports from `b` and `b` imports from `a`. Python starts loading `a`, hits `from b import bar`, starts loading `b`, hits `from a import foo`, and finds `a` partially loaded — so `foo` is not yet defined.

**Symptoms:** `ImportError: cannot import name 'bar' from partially initialized module 'a'`.

**Fix:** Restructure so that one module does not depend on the other. Common solutions:

- Move the shared dependency into a third module.
- Use lazy imports (import inside a function).
- Change the design so the circular dependency does not exist.

### Mistake 2: Name Shadowing Standard Library Modules

```python
# Don't name your files after stdlib modules!
# json.py, csv.py, sys.py, os.py, math.py

# If you create json.py in your project:
import json  # Your json.py, not the standard library!
```

**Why it happens:** Python looks in the script's directory before the standard library. Your `json.py` shadows the real `json` module.

**Symptoms:** `AttributeError` when you call `json.loads()` — your module does not have that function, or it is not the one you expected.

**Fix:** Never name a `.py` file after a standard library module. Use descriptive names: `my_json_utils.py`, `data_export.py`, etc.

### Mistake 3: Importing at the Wrong Place

**Mistake:** Putting imports inside functions when they could be at the top:

```python
def process_data(path):
    import json  # ❌ Import inside function — unnecessary
    with open(path) as f:
        return json.load(f)
```

**Why it happens:** You think it is cleaner to import where used. Or you are trying to fix a circular import by moving imports inwards.

**Symptoms:** Every call to the function pays a small overhead to check if `json` is already loaded. This is minor, but the bigger issue is readability — imports at the top of the file tell the reader what dependencies the module has.

**Fix:** Put all imports at the top of the file, one per line or grouped by stdlib/third-party/local:

```python
import json
import os
import sys
from collections import defaultdict
import numpy as np
from openai import OpenAI
```

Exception: Lazy imports inside a function are acceptable for optional dependencies that may not be installed, or to break a circular import.

### Mistake 4: Using `from module import *`

```python
from os import *  # ❌ Pollutes namespace
from math import *  # Now you have 40+ names without knowing where they came from
```

**Why it happens:** It seems convenient to import everything.

**Symptoms:** You cannot tell where a function came from. Two modules that both use `*` may silently override each other's names. Tools like linters and type checkers cannot resolve the names.

**Fix:** Import specific names or use the module namespace:

```python
import os
print(os.getcwd())  # Clear origin

# Or
from os import getcwd, listdir  # Explicit about what you use
print(getcwd())
```

### Mistake 5: Running the Wrong Python (Module Not Found)

```python
# You installed pandas in a virtual environment
pip install pandas
# But ran the script with the system Python
python3 script.py  # ModuleNotFoundError: No module named 'pandas'
```

**Why it happens:** The Python interpreter you are running is not the one whose `pip` you used. This happens when you forget to activate the virtual environment.

**Symptoms:** `ModuleNotFoundError` for packages you know you installed.

**Fix:** Check `which python3` or `which pip` to confirm you are in the right environment. Always run `pip list` to verify a package is installed in the current environment.
