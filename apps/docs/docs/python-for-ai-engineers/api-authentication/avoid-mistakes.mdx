# API Authentication — Common Mistakes

⏱ 5 minutes

## When NOT to Use Manual Authentication

- **Shared secrets in code:** Never. Use environment variables or a secrets manager.
- **Long-lived tokens in client-side code:** If your app runs in a browser, never embed API keys. Use a backend proxy.
- **Password-based auth for APIs:** Most AI APIs use API keys or OAuth2, not username/password. If you see password auth, verify it is the intended flow.
- **OAuth2 for simple single-user scripts:** For a personal script calling one API, an API key is simpler and sufficient.

## The Pattern to Remember

- **Pattern:** Load credentials from environment variables → inject into session headers → handle 401 with refresh → never log or print credentials.
- **When to use:** Any API that requires authentication — which is virtually every AI model API.
- **When NOT to use:** Public APIs with no auth (rare), internal-only services on a trusted network.
- **Mental model:** An API key is a password for your code. Treat it with the same care as your personal passwords.

## Common Mistakes

### Mistake 1: Hardcoding API Keys in Source Code

```python
# Wrong — visible to everyone with repo access
API_KEY = "sk-abc123def456ghi789"

# Wrong — committed to version control
# config.py: API_KEY = "sk-..."
```

**Why it happens:** It is fast and convenient during development. You commit "just this once" and the key is in the git history forever — even if you remove it later.

**Symptoms:** The key appears in a public repository. You receive a notification about unusual API usage. Your account is charged for unauthorized calls.

**Fix:** Use environment variables exclusively. Add `.env` and `*.env` to your `.gitignore`. Load credentials at runtime:

```python
import os
API_KEY = os.environ["OPENAI_API_KEY"]  # Raises KeyError if not set
```

### Mistake 2: Logging or Printing Authentication Headers

```python
import requests

headers = {"Authorization": f"Bearer {API_KEY}"}
response = requests.get(url, headers=headers)

# Wrong — prints the key to logs!
print(f"Request headers: {response.request.headers}")
print(f"Response: {response.json()}")
```

**Why it happens:** You add debug logging to understand API calls. The log output includes all request headers — including the `Authorization` header.

**Symptoms:** API keys appear in log files, terminal history, or monitoring dashboards. Anyone with log access now has your credentials.

**Fix:** Sanitize logs explicitly:

```python
safe_headers = {
    k: (v[:8] + "..." if k.lower() == "authorization" else v)
    for k, v in response.request.headers.items()
}
print(f"Request headers: {safe_headers}")
```

### Mistake 3: Not Handling Token Expiry

```python
import requests

# Get a token that expires in 1 hour
token = get_token()  # {"access_token": "...", "expires_in": 3600}

# Use it an hour later — will fail with 401
headers = {"Authorization": f"Bearer {token['access_token']}"}
response = requests.get("https://api.example.com/data", headers=headers)
# 401 Unauthorized — token expired
```

**Why it happens:** You obtain a token once and reuse it indefinitely. Many tokens have a short lifetime (15–60 minutes). After expiry, the server rejects the token.

**Symptoms:** Requests fail with 401 after a period of success. The failure happens at unpredictable times.

**Fix:** Track expiry time and refresh tokens proactively:

```python
import time

expires_at = time.time() + token["expires_in"] - 60  # 60s buffer
if time.time() >= expires_at:
    token = refresh_token()  # Get a new token
```

### Mistake 4: Using GET Requests for Operations That Change State

```python
import requests

# WRONG — GET should be idempotent
response = requests.get(
    "https://api.example.com/models/gpt-4/delete",
    headers={"Authorization": f"Bearer {API_KEY}"},
)
```

**Why it happens:** You are in a hurry and GET is simpler to test (works in a browser URL bar). But GET requests should never change server state — and many API proxies and CDNs cache GET responses.

**Symptoms:** The operation succeeds in testing but fails in production because a CDN cached an earlier GET response. Or the API documentation explicitly says to use DELETE.

**Fix:** Use the correct HTTP method: POST for creating, PUT for updating, DELETE for deleting. The method is part of the authentication contract — some APIs enforce method-specific permissions.

### Mistake 5: Sending API Keys in Query Parameters

```python
import requests

# WRONG — key in URL
response = requests.get(
    "https://api.example.com/data?api_key=sk-abc123..."
)

# The URL may be:
# 1. Logged by web servers and proxies
# 2. Visible in browser history
# 3. Sent in the Referer header to other sites
# 4. Cached by CDNs
```

**Why it happens:** Some APIs document query-parameter authentication for convenience. But URLs are routinely logged by web servers, proxies, and CDNs, leaking the key.

**Symptoms:** API keys appear in server access logs. The key is visible in the URL when sharing a screen or debugging.

**Fix:** Always send API keys in the `Authorization` header. If the API only supports query parameters, be aware of the security implications and use HTTPS exclusively.
