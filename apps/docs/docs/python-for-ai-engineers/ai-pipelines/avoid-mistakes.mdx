# AI Pipelines — Common Mistakes

⏱ 5 minutes

## When NOT to Use a Pipeline Architecture

- **Single-stage tasks.** If your task is one LLM call followed by storing the result, a pipeline framework is unnecessary overhead. Use a simple function.
- **Real-time streaming.** Pipelines with multiple stages add latency per stage. For real-time applications (chat, streaming), minimise pipeline depth or run stages in parallel.
- **Tasks that are better served by a dedicated workflow engine.** If you need complex branching, human-in-the-loop approval, or long-running workflows (hours to days), use a dedicated workflow engine like Temporal or Prefect instead of a custom pipeline.

## The Pattern to Remember

- **Pattern:** Stage decorator for retry/logging, Pydantic models for typed intermediate data, orchestrator that collects per-item results, async concurrency for I/O-bound stages.
- **When to use:** Any multi-step AI workflow where each step can fail independently and you need resilience, observability, and composability.
- **When NOT to use:** Single-step tasks, real-time streaming, or workflows that need human-in-the-loop or multi-hour execution.
- **Mental model:** Each pipeline stage is a function with a defined input and output type. The orchestrator is a for-loop with error boundaries around each function call.

## Common Mistakes

### Mistake 1: Cascading Failures

One stage raises an exception, the entire pipeline stops, and all work is lost.

**Why it happens:** No try/except around individual stages. An exception propagates up and aborts the batch.

**Symptoms:** A single malformed input crashes the pipeline after hours of processing. You restart from scratch.

**Fix:** Wrap each item's pipeline run in a try/except. Collect successes and failures separately. Log the failure and continue to the next item.

### Mistake 2: Untyped Stage Output

Stage 1 returns a dict with key `email_body`. Stage 2 expects `body`. The pipeline silently breaks when the keys do not match.

**Symptoms:** Intermittent bugs that are hard to trace. A stage works in isolation but fails when chained.

**Fix:** Use Pydantic models for stage inputs and outputs. Validate at stage boundaries so type mismatches are caught immediately with a clear error message.

### Mistake 3: No Concurrency Limit

Running 100 concurrent LLM calls without a semaphore hits rate limits, returns 429 errors, and wastes retries.

**Symptoms:** Many rate-limit errors in the logs. Pipeline throughput is much lower than expected due to constant retries.

**Fix:** Use `asyncio.Semaphore` to limit concurrent API calls. Start with 5 concurrent calls and increase gradually while monitoring for 429s.

### Mistake 4: Silent Data Skipping

When a stage fails, the orchestrator silently skips the item. The user does not know their request was never processed.

**Symptoms:** Users report missing responses weeks later. The pipeline logs show failures, but no one monitors them.

**Fix:** Every pipeline result — success or failure — goes to a results table. Set up alerts for failure rates above a threshold (e.g., >5% failure rate triggers a notification).

### Mistake 5: Tight Coupling to One Provider

All stages call Claude Sonnet 4. If the provider has an outage, the entire pipeline stops.

**Symptoms:** Zero throughput during provider incidents. No fallback path.

**Fix:** Add provider abstraction. Stage functions accept a client parameter. Configure a fallback provider that uses a different model when the primary returns 5xx errors.

```python
def classify_email(email: dict, client: Anthropic | OpenAI) -> ClassifiedEmail:
    """Classify email using any supported client."""
    # Implementation uses the provided client
    ...
```
