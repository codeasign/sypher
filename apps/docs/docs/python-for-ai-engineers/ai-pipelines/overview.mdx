## Why This Matters

Real-world AI applications are never a single LLM call. They are sequences of operations: load data, transform it, call an LLM, parse the output, transform again, call another LLM, store the result. Each step can fail, time out, or produce unexpected output.

A pipeline architecture gives you resilience: if one stage fails, the pipeline can retry, fall back to a simpler model, or skip that item and continue. It gives you observability: you can log each stage's latency, token usage, and success rate. And it gives you composability: you can add new stages (validation, enrichment, translation) without rewriting existing ones.

## The Problem

You write a script that: reads customer emails, classifies them by intent, extracts key information, generates a draft response, and stores everything in a database. The first version is a linear script:

```python
emails = read_emails("inbox.csv")
for email in emails:
    intent = classify(email["body"])
    info = extract(email["body"])
    draft = generate_response(intent, info)
    store(email, intent, info, draft)
```

If one email has malformed content, everything crashes. If the LLM rate-limits halfway through, the script fails and you restart from the beginning. If one stage takes too long, the entire loop blocks. You need each stage to handle its own failures and the pipeline to know which steps succeeded and which did not.

## The Concept

An AI pipeline is a directed acyclic graph of stages. Each stage is a function that transforms input data and passes the result to the next stage. Stages can be:

- **Deterministic** — pure Python transformations (parsing, filtering, formatting)
- **AI** — LLM calls for generation, classification, or extraction
- **I/O** — reading/writing files, databases, or APIs

Each stage has its own error handling, retry policy, and timeout. The pipeline orchestrator tracks the state of each item through every stage and reports failures without stopping the entire batch.
