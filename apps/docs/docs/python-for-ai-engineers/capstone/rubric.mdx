# Self-Assessment Rubric

Use this to evaluate your own build. There is no single right answer — this rubric tells you what "portfolio-worthy" looks like, not what exact code to write.

## Functionality (40%)

| Criteria | What "excellent" looks like |
|---|---|
| Health endpoint | Returns `{"status": "ok"}` with 200 — always, regardless of auth state |
| Document ingestion | Accepts text + metadata, chunks correctly (correct token count per chunk, overlap applied), embeds via OpenAI, stores in Chroma, returns chunk count. Handles empty documents gracefully (returns 0 chunks, no crash) |
| RAG answer generation | Retrieves 3–5 relevant chunks, builds a prompt with context, calls the LLM with structured output, returns answer + citations. If zero chunks retrieved, returns a clear no-context message — never hallucinates |
| Structured output | LLM response conforms exactly to the defined schema (answer string + citations array). Schema violations (e.g., missing citations) are caught and reported, not silently passed through |
| Caching | Identical question within TTL returns cached response in under 50ms with `X-Cache-Hit: true` header. Cache persists across service restarts (disk-backed). TTL is configurable |
| Authentication | Bearer token required for `/ask` and `/ingest`. Missing token → 401. Invalid token → 403. Health endpoint is always public |
| Error handling | Every error returns structured JSON. Coverage includes: empty question (422), LLM timeout (408), rate limit (429), missing auth (401), invalid auth (403), internal server error (500 with no stacktrace leak) |

## Code and configuration quality (25%)

| Criteria | What "excellent" looks like |
|---|---|
| Project structure | Follows the standard layout: `src/research_assistant/` package, `tests/` mirroring source, `configs/` with YAML, `scripts/` entry points, `pyproject.toml` with dev extras |
| Type hints | Every function signature is annotated with types. Pydantic models define all API schemas. No bare `dict` or `Any` as return types for public functions |
| Clean code | Functions have single responsibility. Names are descriptive. Docstrings explain *why*, not *what*. No commented-out code. No single-letter variable names |
| Configuration | All tunable values (chunk size, overlap, top-K, cache TTL, model names, port, auth token) are in `configs/default.yaml`. Environment variables override for secrets. No hardcoded magic numbers in source code |
| Logging | Two handlers: console (INFO, concise) and file (DEBUG, detailed). Every request is logged with latency, cache status, chunk count, and token usage. Pipeline stages log start/completion/error |

## Engineering judgment (20%)

This section evaluates the tradeoff decisions you made. There is no single correct answer for any of these — what matters is that you recognised the tradeoff and made a deliberate choice.

| Decision point | What a strong answer looks like | What a weak answer looks like |
|---|---|---|
| **Chunk size and overlap strategy** | You chose a chunk size and overlap (e.g., 500/100) with reasoning: small enough for granular retrieval, large enough for coherent context. You tested it and adjusted if needed | You used the default without thinking, or chose extreme values (50 tokens — too small for context; 2000 — too large, wastes LLM context window) |
| **Caching granularity** | You cache by exact question hash, cache the full response (not just the LLM output), and set a sensible TTL. You considered what happens when new documents are ingested — the cache is stale but acceptable within TTL | You cache by truncated/partial question, or use no TTL (cache lives forever), or cache only the LLM call but not the retrieval step (so every request still hits Chroma) |
| **Error handling philosophy** | You distinguish transient errors (timeout, rate limit — retry with backoff) from permanent errors (bad input, invalid auth — return immediately). Each pipeline stage has stage-appropriate retry | Every error gets the same handling. All errors bubble up as 500. Retries are not implemented or applied to permanent errors |
| **Auth scheme simplicity** | You chose bearer token auth (simple, stateless, good for service-to-service). The token is configurable via environment variable. No user-management complexity | You over-engineered (full OAuth2 flow with scopes) or under-engineered (no auth at all, or hardcoded token in source code) |
| **Docker layering and image size** | Multi-stage build under 300MB. Dependencies installed before code copy (maximises cache). Non-root user. Health check. `.dockerignore` excludes build artifacts | Single-stage build over 1GB. All files copied before pip install (cache miss on every code change). Runs as root. No health check |

## Production readiness (15%)

| Criteria | What "excellent" looks like |
|---|---|
| Testing | 80%+ line coverage. Tests mock external APIs (OpenAI, Anthropic, Chroma). Fixtures provide reusable test data. Parametrised tests cover edge cases (empty question, special characters, very long questions, missing auth). Tests run in under 10 seconds |
| Containerisation | Multi-stage Dockerfile, non-root user, health check, `.dockerignore`, `docker-compose.yml` for local development with volume mounts for persistent Chroma data and logs |
| CI/CD | GitHub Actions workflow runs on push and PR: Ruff lint → mypy → pytest with coverage gate. On merge to main: Docker build and push. Pipeline is under 5 minutes |
| Security | API keys are runtime env vars only. No secrets in the image. Bearer token is configurable. Error responses do not leak stack traces or internal paths. Input validation rejects oversized or malformed requests early |

## How to use this rubric

Score yourself honestly against each section. A portfolio-worthy result scores **strong on Functionality** and **Engineering judgment** even if Production readiness is partial — that is expected for a learning project. Use weak scores to identify which course lessons to revisit.

| Score band | What it means |
|---|---|
| 85–100% | Interview-ready. You could deploy this with minor additions |
| 65–84% | Solid capstone. A few gaps to close — review the relevant lesson |
| Below 65% | Some pieces are missing. Focus on the weakest section first |
