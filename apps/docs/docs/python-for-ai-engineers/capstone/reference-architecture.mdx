**Key flow for `/ask`:**

1. FastAPI receives `POST /ask` with `{"question": "..."}`
2. The auth dependency checks the bearer token — 401/403 if invalid
3. The cache is checked — if the question hash exists and is within TTL, return cached response with `X-Cache-Hit: true`
4. On cache miss: the RAG pipeline embeds the query via OpenAI, retrieves top-K chunks from Chroma, builds a prompt with the context, calls Anthropic with structured output (tool use forcing the answer schema), parses the result
5. The response (answer + citations + latency) is cached and returned to the client

## Key design decisions and why

### Decision: Cache at the API level (question hash → full response), not at the LLM call level

**One reasonable choice:** Hash the entire question string with SHA256 and cache the complete `AnswerResponse` (including citations) keyed by that hash. Use `diskcache` with a configurable TTL (default 3600 seconds).

**Why:** Caching the full response means a cache hit avoids *both* the LLM call and the Chroma query. The retrieval step is fast (~20ms) but it still costs money (embedding tokens) and time. A full-response cache returns in under 5ms. The tradeoff is that after new documents are ingested, a cached question might return a stale answer that does not include the new information. This is acceptable because the TTL bounds the staleness — after one hour, the next request fetches fresh data.

**A valid alternative:** Cache only the LLM call (skip generation on cache hit but still run the retrieval step). This uses slightly more time per request (the embedding + Chroma query) but means the answer always reflects the latest ingested documents. Choose this if your documents change frequently.

### Decision: Structured output via Anthropic tool use with forced tool choice

**One reasonable choice:** Use Anthropic's `tool_choice={"type": "tool", "name": "answer_question"}` with a `tools` array containing one tool whose `input_schema` is the Pydantic model's JSON schema. This forces the model to return JSON matching your schema on every call — no parsing ambiguity.

**Why:** This is the approach taught in the Structured Output lesson. It eliminates the need to regex-parse the LLM response or handle markdown-wrapped JSON. If the model produces invalid output, the SDK raises an error immediately — you never silently propagate malformed data. The forced tool choice ensures the model always uses the tool instead of sometimes responding with free text.

**A valid alternative:** Use OpenAI's `response_format` parameter with `strict: true` which also guarantees schema-conforming output. Or use few-shot prompting to ask the LLM to return JSON and validate with Pydantic (`model_validate_json`) — this works but is less reliable because the model can still produce malformed output.

### Decision: Bearer token for auth, not API keys on every request

**One reasonable choice:** A single, configurable bearer token passed as an environment variable (`AUTH_TOKEN`). FastAPI dependency checks the `Authorization: Bearer <token>` header against the configured value.

**Why:** For a service-to-service API, bearer token auth is simple, stateless, and requires no database or session management. It matches the pattern used by the LLM providers themselves (as taught in the API Authentication lesson). The token is injected at runtime, never baked into the image.

**A valid alternative:** No authentication at all (simple for local development but not production-ready). Or API key per client (requires a key management database — out of scope for this project but valid for multi-tenant services).

### Decision: Console + file logging with the standard `logging` module

**One reasonable choice:** Two handlers: a console handler at INFO level with a concise format (`[%(levelname)s] %(message)s`) and a file handler at DEBUG level with a detailed format (`%(asctime)s [%(levelname)s] %(name)s:%(lineno)d: %(message)s`). The `PipelineLogger` class from the Logging lesson wraps this configuration.

**Why:** Console output during development and in Docker logs should be concise — you only need to see request-level INFO (question hash, latency, cache status, token count). The file log is the debug trail: every pipeline stage, every API call attempt, every retry. Separating concerns by handler is the standard pattern. The file handler uses `encoding="utf-8"` to handle non-ASCII questions.

**A valid alternative:** Structured JSON logging (outputting each log entry as a JSON line). This is better for production log aggregation (ELK, Datadog) but adds complexity. The standard module approach is sufficient for this capstone.

## What this reference does NOT include

This is an architecture reference, not a full implementation. The files you create will contain your own code, not a copy of this document. Specifically, this reference does not include:

- The actual FastAPI route definitions and dependency injection wiring
- The chunking function implementation (that is in the RAG lesson's `chunk_text()`)
- The Chroma query implementation (that is in the Embeddings lesson's `PersistentVectorSearch`)
- The LLM client wrapper with retry logic (that is in the LLM API Basics lesson's `LLMClient`)
- The cache class implementation
- The test files with mocked clients
- The Dockerfile or CI/CD YAML

All of those are yours to build, guided by the lessons you have already completed and the requirements in this capstone.

## Common pitfalls at this scale

1. **Import errors from package layout.** You put code in `src/` but run the tests from the project root without installing the package. Solution: always `pip install -e ".[dev]"` first. The Project Structure lesson covers this exactly.

2. **Chroma database path confusion.** Your Chroma persistent client points to a relative path like `./rag_db`. When you run from different directories (project root vs Docker), the path resolves differently. Solution: make the Chroma path configurable and always use an absolute path derived from the config file's location, or use an environment variable override.

3. **Pydantic schema vs JSON Schema mismatch.** The `tools[].input_schema` in the Anthropic API expects a JSON Schema, but Pydantic v2's `model_json_schema()` may produce `$defs` for nested models that the API does not support inline. Solution: use `model_json_schema()` with `ref_template` or flatten nested schemas. Test with a small example before building the full pipeline.

4. **Mocking depth in tests.** You mock the entire Chroma client, which means your tests never exercise the real retrieval logic. Solution: mock at the embedding API level (return a fake vector), not at the Chroma query level. Let the real Chroma distance computation run — it is fast and tests the actual retrieval math. The Pytest lesson's mock patterns show the right balance.

5. **Docker image with baked-in credentials.** You write `ENV ANTHROPIC_API_KEY=sk-...` in the Dockerfile for convenience, then push the image to a registry. Anyone who pulls the image can extract the key. Solution: never set secrets in the Dockerfile. Use `--env-file` at `docker run` or environment in `docker-compose.yml`. The Docker lesson demonstrates this.
