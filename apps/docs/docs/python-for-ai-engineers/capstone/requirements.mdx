# Requirements

## Functional requirements

1. **Health endpoint.** `GET /health` returns `{"status": "ok"}` with HTTP 200.
2. **Ask endpoint.** `POST /ask` accepts `{"question": "..."}` and returns `{"answer": "...", "citations": [...], "latency_ms": N}`.
3. **Document ingestion.** A `POST /ingest` endpoint accepts a document body with metadata (`doc_id`, `source`) and ingests it into the vector store — chunked, embedded, and stored.
4. **RAG retrieval.** Before answering, the service retrieves the top 3–5 most relevant document chunks from the vector store using cosine similarity on embeddings.
5. **Structured output.** The LLM response is constrained to a JSON schema that includes the answer text and an array of citation objects (each with `source_doc_id`, `relevance_score`).
6. **Answer grounded in context.** If no relevant documents are found, the service returns a message saying so — it never hallucinates answers without context.
7. **Caching.** Identical questions answered within the last hour return the cached response instead of calling the LLM again. The cache is persisted across restarts.
8. **Authentication.** All endpoints except `/health` require a bearer token passed in the `Authorization` header. The token is configured via an environment variable.

## Non-functional requirements

- **Performance:** A cached question returns in under 50ms. An uncached question with 3 retrieved chunks completes in under 15 seconds (including LLM generation).
- **Error handling:** Every API error returns a structured JSON error body: `{"error": "...", "detail": "..."}`. LLM timeouts, rate limits, and malformed questions each produce appropriate HTTP status codes (408, 429, 422).
- **Configuration:** All configuration lives in a YAML file and environment variables. No hardcoded strings. Sensible defaults for chunk size (500 tokens), overlap (100 tokens), top-K retrieval (3), and cache TTL (3600 seconds).
- **Observability:** Every request is logged with timestamp, latency, question hash, number of chunks retrieved, token usage, and cache hit/miss. Logs go to both console (INFO level) and a rotating file (DEBUG level).
- **Test coverage:** At least 80% line coverage on business logic (pipeline, retrieval, response generation). External API calls are mocked.
- **Reproducibility:** The project is installable via `pip install -e .` and runs identically in a Docker container. All dependencies are pinned.

## Constraints

- You may only use Python libraries already introduced in this course: `anthropic`, `openai`, `pydantic`, `chromadb`, `httpx`, `python-dotenv`, `pyyaml`, `diskcache`, `pytest`, `pytest-cov`, `ruff`, `mypy`, `numpy`, `tiktoken`, and the Python standard library.
- FastAPI and `uvicorn` are the only additions for the web layer — `fastapi>=0.110.0` and `uvicorn>=0.29.0`.
- The vector store must be Chroma (persistent client). No other vector databases.
- API keys must never appear in code, configuration files, or the Docker image. They are injected at runtime via environment variables.

## What is explicitly out of scope

- A frontend or UI — this is an API-only service. You can test it with `curl` or an HTTP client.
- Streaming responses — the `/ask` endpoint returns the complete answer at once. Streaming is a future enhancement.
- User management, rate limiting per user, or multi-tenancy. The single bearer token is sufficient.
- A web-based document management UI — documents are ingested programmatically via the `/ingest` endpoint.
- Fine-tuning or model training — this is a RAG system, not a training pipeline.
- Deployment to a cloud provider — the Docker image and CI/CD pipeline demonstrate deployability; actual deployment is environment-specific.
