# Virtual Environments

⏱ 10 minutes · Difficulty: Beginner

## What You Will Learn

By the end of this lesson you will be able to:

- Create and activate a Python virtual environment
- Install packages into an isolated environment with `pip`
- Understand why virtual environments are essential for AI projects
- Generate a `requirements.txt` file and recreate an environment from it
- Deactivate and delete environments cleanly

## Prerequisites

You should already understand:

- [How to install Python and run a script](https://docs.python.org/3/using/cmdline.html) — covered in the first two lessons
- What a Python package is — a reusable library installed via `pip`

## Why This Matters

AI projects have notoriously complex dependency trees. Your project needs `numpy==1.26.0`, another needs `numpy==1.24.3`, and your system Python cannot hold both versions simultaneously in the same site-packages directory. Without isolation, you face version conflicts that manifest as cryptic errors — or worse, silent behavioural differences that produce wrong results. Virtual environments solve this by giving each project its own package directory.

Every AI engineering team uses virtual environments. They are the standard way to manage dependencies in development, CI, and deployment. When you clone a project from GitHub, the first instruction in its README is almost always "create a virtual environment." Mastering this workflow is mandatory for collaborating on any Python project.

## The Problem

You `pip install transformers` and it installs version 4.30.0 along with 15 dependencies. A week later, you start a new project and `pip install transformers` installs version 4.45.0. Now your first project may break because the API changed between versions. You cannot have both projects use their preferred versions simultaneously — there is only one global site-packages directory.

The naive solution — manually uninstall and reinstall each time — does not scale past one project. You need a mechanism to create separate package directories per project and switch between them. Python's built-in `venv` module provides exactly that.

## The Concept

A virtual environment is a self-contained directory that contains its own Python interpreter binary and its own `site-packages` directory. When you activate it, your shell's PATH is changed so that `python3` and `pip` point to the environment's copies, not the system-wide ones. Packages you install while the environment is active go into that environment's `site-packages`, invisible to every other environment and to the system Python.
