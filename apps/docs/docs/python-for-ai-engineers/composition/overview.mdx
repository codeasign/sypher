# Composition

- Apply the Delegation pattern through composition

## Prerequisites

You should already understand:

- [Classes](../classes/overview) — composition uses instances of classes
- [Inheritance](../inheritance/overview) — composition is an alternative to inheritance

## Why This Matters

Composition is the most important design principle in object-oriented programming. Nearly every practical AI system is built by composing smaller components: a pipeline has a data loader, a preprocessor, a model, and an evaluator. A model has a tokeniser, a configuration, and a backend. An API client has a HTTP session, a retry handler, and a logger.

Because composition models "has-a" rather than "is-a," it is more flexible than inheritance. You can swap components, add new ones, and test each part independently. The advice "prefer composition over inheritance" (from the Gang of Four *Design Patterns* book) is one of the most widely adopted guidelines in software engineering.

## The Problem

You want to build a text preprocessing pipeline. You could create a single class with all the logic — but that class would be rigid, hard to test, and impossible to reconfigure.

With composition, you define small classes for each preprocessing step (`LowercaseTransformer`, `PunctuationRemover`, `StopwordFilter`), then compose them into a `PreprocessingPipeline` that holds instances of each and delegates to them in order. You can add, remove, or reorder steps without changing the pipeline class itself.

## The Concept

Composition means one object **contains** another object and **delegates** work to it. Instead of saying "a Pipeline is a kind of Transformer" (inheritance), you say "a Pipeline **has** a list of Transformers" (composition).

The contained objects are typically injected through the constructor — this is called **dependency injection** — and the composed object delegates to them as needed.
