## Why This Matters

Dictionaries are the single most important data structure in Python for organising and accessing data by name. In AI engineering, you will use dictionaries constantly:

- **API responses** from OpenAI, Anthropic, and other providers are parsed into dictionaries.
- **Model configuration** — hyperparameters, model names, API keys — lives in dictionaries.
- **Data records** — a single sample with features, label, and metadata is naturally a dict.
- **Token counts**, evaluation metrics, and experiment results are stored in dicts.

Unlike lists, which store values by position, dictionaries store values by a meaningful key. This makes code more readable and resilient to reordering.

## The Problem

You have a configuration with 10 parameters: model name, temperature, max tokens, top-p, stop sequences, etc. You could store them in parallel lists:

```python
param_names = ["model", "temperature", "max_tokens"]
param_values = ["gpt-4", 0.7, 2048]
```

But this is fragile — `param_values[1]` does not tell you what the parameter is. If you reorder the lists or add a parameter, everything breaks. You need a structure that maps names directly to values.

## The Concept

A dictionary stores **key-value pairs**. Each key is unique and maps to exactly one value. Keys must be **hashable** — typically strings, numbers, or tuples of those. Values can be any Python object.

Internally, Python implements dictionaries as **hash tables**. When you look up `config["model"]`, Python hashes the key `"model"` and uses that hash to locate the entry in O(1) time — regardless of how many keys the dictionary has.
