# CI/CD

⏱ 12 minutes · Difficulty: Intermediate

## Prerequisites

You should already understand:

- [Pytest](../pytest/overview) — CI runs your tests automatically
- [Docker](../docker/overview) — CI builds your Docker image
- [Git basics: commit, push, branch, PR](https://docs.github.com/en/get-started/using-git/about-git)

## Why This Matters

Manual testing and deployment do not scale. You make a change, run tests locally (maybe), push to GitHub, SSH into the server, pull the code, restart the service. One day you forget a step, deploy broken code, and spend two hours debugging.

CI/CD automates this. Every push runs tests, linting, type checks, and builds. Every merge to main deploys the new version automatically. If a test fails, the deploy never happens. The process is auditable, repeatable, and does not depend on anyone remembering the right sequence of commands.

## The Problem

You push a change that renames a function used in three places. You updated two of them. The third is in an API endpoint that only runs in production. The tests pass because they mock the endpoint. You merge. The API crashes at 2 AM.

A CI pipeline that runs the full test suite, checks test coverage (not just pass/fail), and requires a review before merging would have caught the missing update. Without CI/CD, you rely on human memory and discipline — both unreliable under pressure.

## The Concept

CI/CD has two phases:

1. **Continuous Integration (CI).** Every push triggers an automated pipeline: check out code, install dependencies, run tests, run linters, check coverage, build the Docker image. If any step fails, the pipeline fails — and the PR cannot merge.
2. **Continuous Deployment (CD).** After CI passes, automatically deploy the artifact (Docker image, package) to the target environment: staging, production, or both.
