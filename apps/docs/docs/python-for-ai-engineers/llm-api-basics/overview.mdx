# LLM API Basics

⏱ 12 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Call an LLM API (Anthropic and OpenAI) from Python
- Send messages with system prompts, user messages, and conversation history
- Handle streaming responses for real-time output
- Manage API keys securely using environment variables

## Prerequisites

You should already understand:

- [HTTPX](../httpx/overview) and [Requests](../requests/overview) — LLM APIs are HTTP endpoints
- [Environment variables](/docs/python-for-ai-engineers/setup/overview) — API keys are never hardcoded
- [JSON](../json/overview) — request and response payloads are JSON

## Why This Matters

Every AI feature you build — whether it is a chatbot, a code assistant, a summariser, or an agent — starts with a call to an LLM API. Companies like Anthropic, OpenAI, Google, and Mistral expose their models through HTTP endpoints that you call from Python. Understanding how to call these APIs correctly, handle errors, stream tokens, and manage authentication is the foundational skill of AI engineering.

The difference between a prototype and a production integration is not the model you choose. It is how you handle retries, streaming, token tracking, error recovery, and API key security. This lesson gives you those patterns.

## The Problem

You want to send a prompt to an LLM and get a completion back. The naive approach is to copy a curl command from a README and wrap it in `os.system()`. That breaks in a dozen ways: no error handling, no streaming, no timeout, hardcoded keys, no conversation history.

You need a structured way to interact with LLM APIs that handles serialisation, authentication, streaming, errors, and token accounting. The official Python SDKs from Anthropic and OpenAI give you exactly that.

## The Concept

An LLM API call has four parts:

1. **Client** — an authenticated SDK instance configured with your API key
2. **Messages** — a list of turns in the conversation, each with a role (`user`, `assistant`, `system`)
3. **Parameters** — model name, max tokens, temperature, and other generation settings
4. **Response** — the model's output, optionally streamed token by token

The client serialises your messages into JSON, sends them over HTTPS to the provider's endpoint, and deserialises the response back into a Python object. Streaming works via server-sent events (SSE), where tokens arrive one chunk at a time over an open HTTP connection.
