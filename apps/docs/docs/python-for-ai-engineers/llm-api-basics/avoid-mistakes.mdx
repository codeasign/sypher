# LLM API Basics — Common Mistakes

⏱ 5 minutes

## When NOT to Use LLM APIs

- **Local or offline workloads:** If your application runs offline or processes sensitive data that cannot leave your network, use a local model with Ollama or Hugging Face Transformers instead of a cloud API.
- **High-throughput, low-latency classification:** For tasks like sentiment analysis or topic labelling that require thousands of calls per second, a fine-tuned small model (or even a logistic regression on embeddings) is faster and cheaper than an LLM API call.
- **Deterministic logic:** If the task is a pure function (sort this list, compute this aggregation), write Python code, not a prompt.

## The Pattern to Remember

- **Pattern:** Create a client once, reuse it across requests, wrap calls in retry logic with exponential backoff, track token usage, and stream when latency matters.
- **When to use:** Any feature that requires natural language understanding, generation, or reasoning that you cannot achieve with deterministic code.
- **When NOT to use:** Offline environments, high-throughput classification, deterministic logic, or when the data cannot leave your network.
- **Mental model:** An LLM API call is an HTTP request with exponential backoff and per-token billing. Design for failure — the API will return 429 (rate limited) and 503 (unavailable) regularly.

## Common Mistakes

### Mistake 1: Exceeding the Context Window

The model has a hard token limit (e.g., 200K for Claude Sonnet 4). Sending a conversation that exceeds this produces an error or silent truncation.

**Why it happens:** You append every message to the conversation history without pruning. Over many turns, the token count grows past the limit.

**Symptoms:** `anthropic.BadRequestError: ... maximum token length exceeded` or truncated responses with no warning.

**Fix:** Implement a token budget. When the history exceeds a threshold, summarise older messages or drop them, keeping only the most recent N turns.

```python
MAX_CONTEXT_TOKENS = 100_000

def trim_messages(messages: list, max_tokens: int = MAX_CONTEXT_TOKENS) -> list:
    """Drop oldest messages until under the token budget."""
    # Rough estimate: 1 token ≈ 4 characters for English text
    total = sum(len(m["content"]) // 4 for m in messages)
    while total > max_tokens and len(messages) > 1:
        total -= len(messages[0]["content"]) // 4
        messages.pop(0)
    return messages
```

### Mistake 2: No Retry Logic

API calls fail. The network flakes, the provider has an outage, or you hit a rate limit. Without retries, your application throws an exception and the user sees an error.

**Symptoms:** Intermittent failures that correlate with high traffic or provider incidents. Stack traces in production logs.

**Fix:** Wrap every API call in retry logic with exponential backoff as shown in the Build It client wrapper. A minimum of 3 retries with 1s, 2s, 4s backoff catches the vast majority of transient failures.

### Mistake 3: Forgetting to Set max_tokens

If you omit `max_tokens`, the API uses a provider default — often a very large number. An unexpectedly long response can take many seconds and cost significantly more than expected.

**Symptoms:** Occasional very slow responses, unexpectedly high bills, or responses that go on far longer than needed.

**Fix:** Always set `max_tokens` explicitly. For structured outputs (classification, extraction), use 128–256 tokens. For generation, 1024–4096 is typical.

### Mistake 4: Hardcoding API Keys

Committing an API key to version control is the most common security incident in AI projects. Tools crawl GitHub for exposed keys constantly.

**Symptoms:** You receive a notification from your provider that your key was compromised. Or you find your key in a GitHub secret scanning alert.

**Fix:** Use environment variables and `.env` files. Add `.env` to `.gitignore`. Use a secrets manager in production. Rotate keys immediately if exposed.

### Mistake 5: Not Handling Streaming Errors Gracefully

A streaming connection can drop mid-response. The partial output is lost, and the application may hang waiting for more tokens.

**Symptoms:** The chat UI stops responding mid-sentence. No error is logged. The user refreshes and loses context.

**Fix:** Set a timeout on the stream. Buffer the partial response and surface it even if the stream fails. Log the failure but do not crash.

```python
import time

def stream_with_timeout(client, messages, timeout=30):
    """Stream with a hard timeout and partial result recovery."""
    buffer = []
    start = time.perf_counter()
    try:
        with client.messages.stream(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            messages=messages,
        ) as stream:
            for text in stream.text_stream:
                if time.perf_counter() - start > timeout:
                    break
                buffer.append(text)
                print(text, end="", flush=True)
    except Exception as e:
        logger.error("Stream failed after %d chars: %s", len("".join(buffer)), e)
    return "".join(buffer)
```
