# File Handling — Common Mistakes

⏱ 6 minutes

## When NOT to Use Manual File Handling

- **Large datasets (10GB+):** Reading the entire file into memory with `read()` or `readlines()` will crash your program. Use incremental reading (line-by-line iteration) or libraries like `pandas.read_csv(chunksize=...)`.
- **Binary data (images, audio):** Use dedicated libraries — `PIL` for images, `soundfile` for audio, `torch.save()` for model weights. Raw binary file handling is error-prone.
- **Databases:** Do not build your own persistence with CSV/JSON files for data that needs querying, concurrency, or transactions. Use SQLite or a proper database.

## The Pattern to Remember

- **Pattern:** Use `with open(path, mode, encoding="utf-8") as f:` — the `with` block guarantees the file is closed even on errors.
- **When to use:** Reading/writing text files, JSON configs, CSV data, log files.
- **When NOT to use:** Large files (stream instead), binary data (use libraries), data needing queries (use a database).
- **Mental model:** A file handle is a connection to the disk. The `with` statement is a safety belt that closes it automatically when you are done.

## Common Mistakes

### Mistake 1: Forgetting to Close a File

```python
f = open("data.txt", "r", encoding="utf-8")
data = f.read()
# f is never closed!
```

**Why it happens:** You used `open()` without `with`. The file handle stays open until the garbage collector runs or the program exits.

**Symptoms:** On Windows, you may get permission errors when trying to delete or modify the file while the program is still running. On any OS, you leak file descriptors — eventually the program hits the OS limit (`IOError: too many open files`).

**Fix:** Always use a context manager:

```python
with open("data.txt", "r", encoding="utf-8") as f:
    data = f.read()
# Automatically closed here
```

### Mistake 2: Writing in "w" Mode When You Meant "a" or "x"

```python
with open("results.csv", "w") as f:  # Overwrites existing file!
    f.write("model,accuracy\n")
```

**Why it happens:** "w" mode opens for writing and truncates the file immediately. If the file had 100 rows of data, they are gone the moment you open it.

**Symptoms:** You accidentally deleted data. The file exists but is empty or contains only what you wrote in the current session.

**Fix:** Use "a" (append) to add to an existing file, or "x" (exclusive create) to fail if the file already exists. Check if the file exists first:

```python
from pathlib import Path

if Path("results.csv").exists():
    append_or_merge = input("File exists. Overwrite? (y/n): ")
    mode = "w" if append_or_merge.lower() == "y" else "a"
else:
    mode = "w"
```

### Mistake 3: Not Specifying Encoding

```python
# On Windows, this may fail with UnicodeDecodeError
with open("data.txt", "r") as f:  # Uses system default encoding!
    data = f.read()
```

**Why it happens:** Python uses the system's default encoding (`utf-8` on macOS/Linux, `cp1252` on Windows in some locales). A file created on one system may not be readable on another.

**Symptoms:** `UnicodeDecodeError` when reading a file on a different OS. Characters like `é`, `ü`, or emoji are corrupted.

**Fix:** Always specify `encoding="utf-8"` explicitly — on both read and write. UTF-8 is the universal standard:

```python
with open("data.txt", "r", encoding="utf-8") as f:
    data = f.read()
```

### Mistake 4: Using read() on an Entire Large File

```python
with open("large_dataset_5gb.json", "r", encoding="utf-8") as f:
    data = f.read()  # 5 GB string in memory!
```

**Why it happens:** `read()` without an argument reads the entire file into a single string. For a 5 GB file, this either exhausts memory or makes the OS swap heavily.

**Symptoms:** The program runs out of memory (`MemoryError`) or becomes extremely slow due to swapping.

**Fix:** Process the file incrementally:

```python
# Line by line
with open("large_file.txt", "r", encoding="utf-8") as f:
    for line in f:
        process(line)

# Chunk by chunk
while True:
    chunk = f.read(8192)  # 8 KB at a time
    if not chunk:
        break
    process(chunk)
```

For JSON specifically, use `ijson` for streaming, or store data in a format that supports incremental processing like JSON Lines (one JSON object per line):

```python
# JSON Lines format — each line is a complete JSON object
with open("data.jsonl", "r", encoding="utf-8") as f:
    for line in f:
        record = json.loads(line)
        process(record)
```

### Mistake 5: Using String Methods for Path Joining

```python
# Fragile — breaks on Windows with backslashes
path = data_dir + "/" + filename

# Better — works cross-platform
from pathlib import Path
path = Path(data_dir) / filename
```

**Why it happens:** You manually concatenate paths with `/`. This works on macOS and Linux but breaks on Windows, which uses `\` as the path separator.

**Symptoms:** `FileNotFoundError` on Windows. Or the path looks correct but contains a mix of `/` and `\`.

**Fix:** Use `pathlib.Path` for all path operations — it handles `/` and `\` correctly on any OS:

```python
from pathlib import Path

config_dir = Path.home() / ".config" / "myapp"
config_dir.mkdir(parents=True, exist_ok=True)
config_path = config_dir / "settings.json"
```
