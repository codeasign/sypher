## Why This Matters

Type hints are documentation that computers can check. They tell readers (and IDEs) what type a variable should be, what a function expects, and what it returns. In a codebase of any size — and especially in AI engineering where functions pass around complex nested data — type hints catch an entire class of bugs at analysis time rather than runtime.

Tools like `mypy` and `pyright` check your annotations against the actual code, flagging mismatches before you run the program. IDEs use them for autocomplete, inline documentation, and refactoring. Dataclasses require them for field declarations. Type hints are not optional in modern Python — they are the expected standard.

## The Problem

You have a function that processes predictions:

```python
def process_result(result):
    # What is result? A dict? A dataclass? A string?
    # What does it return?
    return result["label"]
```

The caller does not know what to pass. The function author has to remember the structure. When `result` changes, no error is raised until runtime. A bug introduced today may not surface until production tomorrow. Type hints make the contract explicit:

```python
def process_result(result: dict) -> str:
    return result["label"]  # mypy now knows result is a dict
```

## The Concept

Type hints are **syntactic annotations** that document the expected types of variables, parameters, and return values. They are ignored by the Python interpreter at runtime (no performance cost) but checked by external tools.
