# Packaging — Common Mistakes

⏱ 5 minutes

## When NOT to Create a Package

- **Application, not library.** If your project is a web app, a CLI tool, or a data pipeline, it does not need to be a pip-installable package. Docker is the right distribution mechanism.
- **Single file.** If your entire project is one Python file, a package structure adds unnecessary directory nesting.
- **Experimental code.** Package quality takes effort. Wait until the API stabilises and someone else wants to use it.

## The Pattern to Remember

- **Pattern:** `pyproject.toml` with metadata and dependencies, `src/` layout, `__init__.py` exports, `python -m build` for distribution, `twine` for publishing.
- **When to use:** Any reusable Python library that more than one project will depend on.
- **When NOT to use:** Single-file scripts, applications better distributed via Docker, or experimental code.
- **Mental model:** A package is a contract. `pyproject.toml` is the terms (name, version, dependencies, entry points). Once published, changing the contract breaks your users' builds.

## Common Mistakes

### Mistake 1: No __init__.py in the Package Directory

Without `__init__.py`, Python does not recognise the directory as a package. `from llm_tools import client` raises `ModuleNotFoundError`.

**Symptoms:** Imports work during development (because of the path setup) but fail after installing from PyPI.

**Fix:** Every directory that should be a package must have an `__init__.py` — even if it is empty.

### Mistake 2: Version Conflicts with Upper Bounds

`anthropic>=0.30.0` is a lower bound. When Anthropic releases 1.0.0 with breaking changes, your package breaks.

**Symptoms:** A working package breaks after `pip install --upgrade` when a dependency releases a new major version.

**Fix:** Add an upper bound only when you know the next major version breaks compatibility:

```toml
dependencies = [
    "anthropic>=0.30.0,<1.0.0",
]
```

Be conservative with upper bounds — too strict and your package conflicts with other packages that need a newer version.

### Mistake 3: Publishing Development Versions to PyPI

`twine upload dist/*` publishes every file in `dist/`. If you built a version during development, it is on PyPI forever.

**Symptoms:** Users pip-install your package and get an old development version with a bug you already fixed.

**Fix:** Clean `dist/` before building for release. Use `test.pypi.org` for testing. Automate publishing in CI so it only happens on tagged releases.

```bash
rm -rf dist/
python -m build
twine upload dist/*
```

### Mistake 4: Missing or Incorrect Entry Points

The `[project.entry-points.console_scripts]` section defines CLI commands. If the function path is wrong, `pip install` succeeds but running the command fails with `ModuleNotFoundError`.

**Symptoms:** The command is installed (it appears in `PATH`) but crashes with an import error when run.

**Fix:** Test the entry point immediately after installing:

```bash
pip install -e .
your-command --help  # Verify this works before publishing
```

### Mistake 5: Forgetting to Update the Version

You fix a bug, rebuild, and upload to PyPI. Users who run `pip install --upgrade` get the old version because you forgot to bump the version number.

**Symptoms:** A fix is merged and published but users report the bug is still present.

**Fix:** Use automated versioning. Tools like `bumpver` or `setuptools-scm` derive the version from git tags:

```toml
[project]
dynamic = ["version"]

[tool.hatchling.version]
source = "vcs"
```

With this setup, `git tag v0.2.0 && git push --tags` triggers a build with version `0.2.0`.
