## Why This Matters

Sharing your AI code should be as simple as `pip install your-package`. Whether you are distributing an LLM client wrapper, a set of prompt templates, a data preprocessing library, or an evaluation framework, a properly packaged project is the standard way to share Python code.

Packaging also gives you: version pinning (users depend on exactly the version they tested with), dependency specification (when they install your package, pip installs its dependencies too), entry points (users can run your CLI tool from anywhere), and documentation (your docstrings become `help()` output).

## The Problem

You built a reusable LLM client wrapper with structured output validation, retry logic, and prompt templates. Your colleague wants to use it. You tell them to copy the files, set up the paths, install the dependencies manually. They miss a dependency, the import path is wrong, and the version they copied is outdated.

A package solves this. `pip install llm-client-wrapper` installs the code, all dependencies, a CLI entry point, and you can version it. When you release `2.0.0`, your colleague runs `pip install --upgrade llm-client-wrapper`.

## The Concept

A Python package is a directory with an `__init__.py` file and a `pyproject.toml` that describes it. The `pyproject.toml` contains:

1. **Metadata** — name, version, description, author, license
2. **Dependencies** — packages required to use yours
3. **Entry points** — CLI commands that `pip install` makes available globally
4. **Build system** — the tool that builds the package (setuptools, hatchling, flit)

When you run `pip install -e .` in the project directory, pip installs it in editable mode — changes to your source are immediately reflected without reinstalling.
