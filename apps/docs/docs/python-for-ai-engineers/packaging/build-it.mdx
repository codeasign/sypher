# Packaging — Hands On

⏱ 15 minutes

## Breaking It Down

### Step 1: Project Structure

A package needs a specific directory layout:

```
llm-tools/
├── pyproject.toml
├── README.md
├── LICENSE
├── src/
│   └── llm_tools/
│       ├── __init__.py
│       ├── client.py
│       ├── prompts.py
│       └── schemas.py
└── tests/
    ├── test_client.py
    └── test_prompts.py
```

The `src/` layout keeps your package code separate from project-level files (tests, config). It prevents import confusion during development.

### Step 2: pyproject.toml

This is the single configuration file for modern Python packages:

```toml
# pyproject.toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "llm-tools"
version = "0.1.0"
description = "Reusable LLM client wrapper with structured output and retry logic"
readme = "README.md"
license = {text = "MIT"}
requires-python = ">=3.11"

authors = [
    {name = "Your Name", email = "you@example.com"},
]

dependencies = [
    "anthropic>=0.30.0",
    "openai>=1.0.0",
    "pydantic>=2.0.0",
    "python-dotenv>=1.0.0",
]

[project.optional-dependencies]
dev = [
    "pytest>=8.0.0",
    "pytest-cov>=5.0.0",
    "ruff>=0.3.0",
    "mypy>=1.8.0",
]

[project.urls]
Homepage = "https://github.com/your-username/llm-tools"
Documentation = "https://llm-tools.readthedocs.io"

[project.entry-points.console_scripts]
extract = "llm_tools.cli:extract_main"
classify = "llm_tools.cli:classify_main"
```

Key sections:

- `[build-system]` — declares the build tool. Hatchling is the modern default; setuptools also works.
- `[project]` — metadata, dependencies, Python version requirement.
- `[project.optional-dependencies]` — extras that users install with `pip install llm-tools[dev]`.
- `[project.entry-points.console_scripts]` — CLI commands installed globally.

```python
# src/llm_tools/client.py
"""LLM API client with retry logic and structured output."""

import os
import time
import logging
from dataclasses import dataclass
from typing import Optional

logger = logging.getLogger(__name__)


@dataclass
class LLMResponse:
    """Standardised response from any LLM provider."""
    text: str
    model: str
    input_tokens: int
    output_tokens: int
    latency_ms: float


class LLMClient:
    """Unified LLM client supporting Anthropic and OpenAI."""

    def __init__(self, provider: str = "anthropic"):
        self.provider = provider
        if provider == "anthropic":
            from anthropic import Anthropic
            self._client = Anthropic(api_key=os.environ.get("ANTHROPIC_API_KEY"))
        elif provider == "openai":
            from openai import OpenAI
            self._client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))
        else:
            raise ValueError(f"Unsupported provider: {provider}")

    def generate(
        self,
        messages: list[dict],
        system: Optional[str] = None,
        model: Optional[str] = None,
        max_tokens: int = 1024,
        temperature: float = 0.7,
    ) -> LLMResponse:
        """Send a completion request with automatic retry.

        Args:
            messages: Conversation history as role/content dicts.
            system: Optional system prompt.
            model: Model identifier (provider default if omitted).
            max_tokens: Maximum output tokens.
            temperature: Sampling temperature.

        Returns:
            LLMResponse with text, token usage, and latency.
        """
        start = time.perf_counter()

        if self.provider == "anthropic":
            kwargs = dict(
                model=model or "claude-sonnet-4-20250514",
                max_tokens=max_tokens,
                temperature=temperature,
                messages=messages,
            )
            if system:
                kwargs["system"] = system
            response = self._client.messages.create(**kwargs)
            text = response.content[0].text
            input_tokens = response.usage.input_tokens
            output_tokens = response.usage.output_tokens
        else:
            if system:
                messages = [{"role": "system", "content": system}] + messages
            response = self._client.chat.completions.create(
                model=model or "gpt-4o",
                max_tokens=max_tokens,
                temperature=temperature,
                messages=messages,
            )
            text = response.choices[0].message.content
            input_tokens = response.usage.prompt_tokens
            output_tokens = response.usage.completion_tokens

        elapsed = time.perf_counter() - start
        return LLMResponse(
            text=text,
            model=response.model,
            input_tokens=input_tokens,
            output_tokens=output_tokens,
            latency_ms=round(elapsed * 1000),
        )
```

### Step 4: Install Locally in Editable Mode

```bash
# From the project root (where pyproject.toml lives)
pip install -e .

# Verify it works
python -c "from llm_tools import LLMClient; print('Package imported successfully')"

# Install with dev dependencies
pip install -e ".[dev]"
```

Editable mode means changes to `src/llm_tools/` are reflected without reinstalling.

### Step 5: Build Distributions

```bash
# Install build tool
pip install build

# Build wheel and source distribution
python -m build

# Output in dist/:
# dist/llm_tools-0.1.0-py3-none-any.whl
# dist/llm_tools-0.1.0.tar.gz
```

The wheel (`.whl`) is the binary distribution — pip installs it directly. The source distribution (`.tar.gz`) is used for platforms that need to compile extensions.

### Step 6: Publish to PyPI

```bash
# Install twine for uploading
pip install twine

# Upload to Test PyPI first
twine upload --repository-url https://test.pypi.org/legacy/ dist/*

# Upload to PyPI (requires PyPI account and API token)
twine upload dist/*
```

For production, automate publishing in CI:

```yaml
# .github/workflows/publish.yml
name: Publish to PyPI

on:
  release:
    types: [published]

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install build twine
      - run: python -m build
      - run: twine upload dist/*
        env:
          TWINE_USERNAME: __token__
          TWINE_PASSWORD: ${{ secrets.PYPI_API_TOKEN }}
```

## Build Something Real — Package Your AI Client Library

Package the LLM client wrapper you built in Lesson 1 as a proper installable library:

```toml
# pyproject.toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "llm-tools"
version = "0.1.0"
description = "Reusable LLM client with retry, structured output, and prompt templates"
readme = "README.md"
requires-python = ">=3.11"
license = {text = "MIT"}

dependencies = [
    "anthropic>=0.30.0",
    "pydantic>=2.0.0",
]

[project.optional-dependencies]
dev = ["pytest>=8.0", "ruff>=0.3.0"]
openai = ["openai>=1.0.0"]

[project.urls]
Source = "https://github.com/your-org/llm-tools"

[project.entry-points.console_scripts]
llm-complete = "llm_tools.cli:complete"
```

Users can install the base package (Anthropic only) or with OpenAI support:

```bash
pip install llm-tools
pip install llm-tools[openai]  # Includes OpenAI SDK
```

## Performance Considerations

- **Dependency bloat.** Keep dependencies minimal. Each dependency adds download time and potential conflicts. Use optional dependency groups for optional features.
- **C extension compatibility.** Packages with C extensions (NumPy, Pandas) must be built per-platform on PyPI. If your package depends on them, ensure they have published wheels for your target platforms.

## Security Considerations

- **PyPI API tokens.** Never hardcode your PyPI token. Use `${{ secrets.PYPI_API_TOKEN }}` in CI. Set token permissions to a specific project, not all projects.
- **Dependency review.** Before adding a dependency, audit it: how many maintainers, how active, any reported vulnerabilities. A compromised dependency in your package becomes a compromise of every user who installs it.
- **Supply chain.** Pin exact versions in your package metadata where practical. Users can always override. Tools like `pip-audit` scan for known vulnerabilities.
