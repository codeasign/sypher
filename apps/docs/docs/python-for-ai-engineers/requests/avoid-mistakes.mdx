# Requests — Common Mistakes

⏱ 6 minutes

## When NOT to Use requests

- **Asynchronous code:** `requests` blocks the thread during I/O. In async applications, use `httpx` with async support or `aiohttp`.
- **Very high throughput (thousands of requests/second):** Use an async HTTP library or a dedicated HTTP client with connection pooling tuned for your workload.
- **Web scraping with JavaScript:** `requests` does not execute JavaScript. Use Playwright or Selenium for pages that render content dynamically.
- **Streaming large file downloads with progress:** Use `httpx` streaming with progress callbacks, or use `urllib.request.urlretrieve()` with a report hook.

## The Pattern to Remember

- **Pattern:** Create a `Session`, set default headers, call `session.get()` or `session.post()` with `json=` for JSON bodies, check `response.ok`, call `response.json()` to parse, handle errors with `raise_for_status()`.
- **When to use:** REST API calls, LLM inference requests, data fetching from web services, authentication flows.
- **When NOT to use:** Async code, JavaScript rendering, extremely high throughput, file downloads with progress.
- **Mental model:** An HTTP request has method + URL + headers + body; a response has status code + headers + body. The `requests` library wraps all of this into clean method calls.

## Common Mistakes

### Mistake 1: Not Setting a Timeout

```python
import requests

# Without timeout — can hang forever
response = requests.get("https://api.example.com/slow-endpoint")

# With timeout — raises after 10 seconds
response = requests.get("https://api.example.com/slow-endpoint", timeout=10)
```

**Why it happens:** `requests` does not set a default timeout. If the server hangs, your request hangs indefinitely — blocking a thread, consuming a connection, and never returning.

**Symptoms:** The program appears to freeze on a network call. Processes accumulate hung requests. Deployment health checks fail because threads are exhausted.

**Fix:** Always pass `timeout=N`. Use a value appropriate for your API: 5–10 seconds for typical LLM calls, 30–60 seconds for long-running operations. For very conservative safety, use a tuple `(connect_timeout, read_timeout)`:

```python
response = requests.get(url, timeout=(3.05, 10))  # 3s connect, 10s read
```

### Mistake 2: Using `data=` Instead of `json=` for JSON Payloads

```python
import requests

# Wrong — sends form-encoded data, not JSON
response = requests.post(url, data={"model": "gpt-4"})
# Content-Type: application/x-www-form-urlencoded
# The server may not parse this correctly

# Correct — sends JSON
response = requests.post(url, json={"model": "gpt-4"})
# Content-Type: application/json
```

**Why it happens:** `data=` sends the dict as form-encoded key-value pairs. `json=` serializes it to JSON and sets the correct content type. Both are valid HTTP, but API servers expect JSON.

**Symptoms:** The server returns a 400 Bad Request or ignores some fields. The API responds with `"unexpected body format"` or similar.

**Fix:** Use `json=` for any API that expects a JSON body. Use `data=` only when submitting HTML forms or `application/x-www-form-urlencoded` data.

### Mistake 3: Ignoring `raise_for_status()`

```python
import requests

response = requests.get("https://httpbin.org/status/404")
# No error raised — response.status_code is 404, but the code continues
data = response.json()  # May fail or return unexpected content
```

**Why it happens:** `requests` returns a Response object for every HTTP response, including errors. It does not raise an exception for 4xx or 5xx status codes — you must check explicitly.

**Symptoms:** You process an error response as if it were successful. A 404 response body is parsed as valid data. A 500 error body does not have the fields you expect, causing `KeyError`.

**Fix:** Always call `response.raise_for_status()` after every request, or check `response.ok` before proceeding:

```python
response = requests.get(url, timeout=10)
response.raise_for_status()  # Raises HTTPError for 4xx/5xx
data = response.json()
```

### Mistake 4: Creating a New Connection for Every Request

```python
import requests

# Creates a new TCP connection for every request — slow
for model in ["gpt-4", "claude-3", "llama-3"]:
    response = requests.post(
        "https://api.example.com/chat",
        json={"model": model, "prompt": "Hello"},
        headers={"Authorization": "Bearer ..."},
    )
```

**Why it happens:** Each call to `requests.get()` or `requests.post()` opens a new connection, performs the TLS handshake, sends the request, reads the response, and closes the connection. For multiple requests to the same server, this is 2–5x slower than reusing a connection.

**Symptoms:** API calls are slower than expected, especially over HTTPS where TLS negotiation adds significant overhead.

**Fix:** Use a `Session` for all requests to the same server:

```python
with requests.Session() as session:
    session.headers.update({"Authorization": "Bearer ..."})
    for model in ["gpt-4", "claude-3", "llama-3"]:
        response = session.post(
            "https://api.example.com/chat",
            json={"model": model, "prompt": "Hello"},
        )
```

### Mistake 5: Hardcoding API Keys in Source Code

```python
# Wrong — key visible in source control
API_KEY = "sk-abc123..."

# Wrong — key in a config file committed to git
# config.json: {"api_key": "sk-abc123..."}
```

**Why it happens:** You hardcode the key during development, then commit the file to version control. The key is now accessible to everyone who has access to the repository.

**Symptoms:** A security scan flags exposed credentials. Your API key is used by an unauthorized party. You receive a bill for unexpected usage.

**Fix:** Use environment variables or a secrets manager:

```python
import os

API_KEY = os.environ.get("OPENAI_API_KEY")
if not API_KEY:
    raise ValueError("OPENAI_API_KEY environment variable is not set")
```
