# Generators — Common Mistakes

⏱ 5 minutes

## When NOT to Use Generators

- **Multiple passes:** If you need to iterate over the data more than once, materialise it into a list. Generators are single-use.
- **Random access:** Generators do not support indexing (`gen[5]` raises a `TypeError`). If you need random access, use a list.
- **Parallel processing:** Generators are single-threaded by nature. For parallel processing, use `multiprocessing` or `concurrent.futures` with materialised data.

## The Pattern to Remember

- **Pattern:** Generator functions `yield` values and preserve state between calls. Generator expressions `(expr for ...)` are lazy comprehensions.
- **When to use:** Large datasets, streaming data, infinite sequences, processing pipelines.
- **When NOT to use:** Multiple iteration passes, random access, data that fits easily in memory and is iterated many times.
- **Mental model:** A generator is a paused loop that hands you one value each time you ask — it never holds more than one.

## Common Mistakes

### Mistake 1: Using a Generator When You Need Random Access

```python
gen = (x ** 2 for x in range(10))
print(gen[3])  # TypeError: 'generator' object is not subscriptable
print(len(gen))  # TypeError: object of type 'generator' has no len()
```

**Fix:** Materialise to a list if you need indexing or length: `data = list(gen)`.

### Mistake 2: Iterating a Generator More Than Once

```python
gen = (x * 2 for x in [1, 2, 3])
print(list(gen))  # [2, 4, 6]
print(list(gen))  # [] — exhausted, nothing left
```

**Why it happens:** A generator produces each value exactly once. After the last value, further iteration yields nothing.

**Fix:** Materialise to a list if you need multiple passes, or recreate the generator.

### Mistake 3: Returning Instead of Yielding

```python
# ❌ Returns a single value — not a generator
def get_numbers(n):
    return [x ** 2 for x in range(n)]  # Returns a list

# ✅ Yields values lazily — this is a generator
def get_numbers(n):
    for x in range(n):
        yield x ** 2

# ⚠ Also a generator expression
def get_numbers(n):
    return (x ** 2 for x in range(n))  # Returns a generator
```

### Mistake 4: Exhausting a Generator with list() Unnecessarily

```python
gen = some_large_data_generator()
data = list(gen)          # Loads everything into memory — defeats the purpose
for item in data:         # Then iterates over the list
    process(item)

# Better — iterate the generator directly
for item in some_large_data_generator():
    process(item)
```

### Mistake 5: Forgetting That `yield from` Returns the Last Value of the Subgenerator

```python
def inner():
    yield 1
    yield 2
    return "done"

def outer():
    result = yield from inner()
    print(f"Inner returned: {result}")  # Inner returned: done

list(outer())  # [1, 2]
```

The `return` value of a generator is only accessible through `yield from` — you usually do not need it, but it is available for advanced patterns like coroutines.
