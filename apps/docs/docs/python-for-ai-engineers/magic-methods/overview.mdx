## Why This Matters

Magic methods — also called dunder methods (double underscore) — are what make Python objects integrate with the language's built-in features. They define how your objects behave with `len()`, `str()`, `==`, `in`, `[]`, `for`, `+`, `hash()`, and dozens of other operations.

Without magic methods, every object is opaque — `print(obj)` shows `<__main__.MyObject object at 0x...>`, `obj == other` always returns `False` (identity check), and `len(obj)` raises `TypeError`. With magic methods, your objects behave like native Python types — predictable, intuitive, and pleasant to use.

## The Problem

```python
class Prediction:
    def __init__(self, text, label, confidence):
        self.text = text
        self.label = label
        self.confidence = confidence

p = Prediction("Great!", "positive", 0.95)
print(p)            # <__main__.Prediction object at 0x...>
len(p)              # TypeError — no __len__
p[0]                # TypeError — no __getitem__
p == p              # False — identity check, not value equality
```

Without magic methods, the class is a black box. Users must guess the internal structure. With magic methods, it behaves naturally.

## The Concept

Magic methods are predefined method names that Python looks up when you use a built-in operation. When you write `str(obj)`, Python calls `obj.__str__()`. When you write `obj[key]`, Python calls `obj.__getitem__(key)`. When you write `obj + other`, Python calls `obj.__add__(other)`.

You never call these methods directly by name — Python calls them for you when you use the corresponding syntax.
