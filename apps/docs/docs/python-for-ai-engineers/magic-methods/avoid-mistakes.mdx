# Magic Methods — Common Mistakes

⏱ 5 minutes

## When NOT to Use Magic Methods

- **Simple dataclasses:** `@dataclass` auto-generates `__init__`, `__repr__`, `__eq__`, and `__hash__` — no need to write them manually.
- **Unexpected behaviour:** Only implement magic methods when the behaviour matches user expectations. `__add__` on a `Model` class would be confusing.
- **Performance-critical `__getitem__`:** If your `__getitem__` does heavy computation, document it — users expect O(1) indexing from brackets.

## The Pattern to Remember

- **Pattern:** Implement `__repr__` (always), `__str__` (if user-facing), `__eq__` + `__hash__` (together, for comparability), `__len__` + `__getitem__` (for containers).
- **When to use:** When you want your objects to work naturally with Python's built-in functions and operators.
- **When NOT to use:** When the behaviour would violate user expectations, for simple dataclasses that auto-generate them.
- **Mental model:** Magic methods are the hooks that connect your class to Python's syntax. Implement the ones that make your class feel native.

## Common Mistakes

### Mistake 1: Implementing `__eq__` Without `__hash__`

```python
class Bad:
    def __init__(self, x):
        self.x = x
    def __eq__(self, other):
        return self.x == other.x

b1 = Bad(1)
b2 = Bad(1)
print(b1 == b2)  # True
# But:
# set([b1, b2])  # Works because object's __hash__ still works
# However, if you set __hash__ to None, this breaks.
```

**Python's rule:** If you define `__eq__` but not `__hash__`, `__hash__` is set to `None` (making the object unhashable) if the object is mutable. For immutable objects, it is derived from `__eq__`. Always define both together when you mean value equality.

### Mistake 2: Returning `NotImplemented` Instead of Raising `TypeError`

When `__eq__` (or `__add__`, etc.) receives an incompatible type, return `NotImplemented` (the singleton), not `NotImplementedError` or `TypeError`. Python then tries the reflected operation on the other operand:

```python
def __eq__(self, other):
    if not isinstance(other, MyClass):
        return NotImplemented  # Let Python try other.__eq__
    return self.val == other.val
```

### Mistake 3: `__contains__` Without `__iter__`

If you implement `__iter__` but not `__contains__`, Python falls back to iterating and checking each element — which works but is slow for large collections. If you have a fast membership test, implement `__contains__` explicitly.

### Mistake 4: Mutable Object with `__hash__`

If you define `__hash__` on a mutable object, the hash changes when the object mutates, breaking dict and set lookups:

```python
obj = MyMutableClass(42)
d = {obj: "value"}
obj.x = 100  # Hash changed!
d[obj]  # KeyError — object not found at its new hash
```

**Fix:** Only define `__hash__` on immutable objects (enforced with `frozen=True` in dataclasses, or by design).

### Mistake 5: Forgetting `__bool__` Affects `if obj`

If you do not define `__bool__`, Python uses `__len__`. If `__len__` returns 0, the object is falsy. This can surprise:

```python
class Container:
    def __init__(self, items):
        self.items = items
    def __len__(self):
        return len(self.items)

c = Container([])
if not c:  # True — len(c) is 0
    print("Empty container is falsy")
```

Be explicit about your object's truthiness — define `__bool__` if the default does not match expectations.
