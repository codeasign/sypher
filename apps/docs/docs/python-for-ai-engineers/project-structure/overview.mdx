# Project Structure

⏱ 10 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Design a scalable project layout for AI and data science work
- Separate configuration, source code, tests, and data into logical directories
- Use Python's package structure to keep imports clean across a project
- Set up a project that is reproducible, testable, and deployable

## Prerequisites

You should already understand:

- [Modules and packages](../modules/overview) — how Python resolves imports
- [Virtual environments](../virtual-environments/overview) — isolating dependencies per project
- Basic familiarity with the command line

## Why This Matters

A well-structured project is the difference between code you can pick up six months later and code you delete out of frustration. In AI engineering, you will work across data ingestion scripts, model definitions, training loops, evaluation pipelines, configuration files, experiment tracking, and deployment manifests. If every file sits in the same flat directory with names like `test_final_v2_real.py`, the project becomes unmaintainable within weeks.

Companies that ship AI products enforce project structure standards because structure communicates intent. A new engineer joining your team should be able to open the repository and find the source code in `src/`, the tests in `tests/`, and the configuration in `configs/` — without guessing. This consistency is what makes collaboration possible at scale.

## The Problem

You start a new project: a sentiment classifier using an LLM API. You create one file called `sentiment.py`. Then you add a configuration file, a data-loading script, a prompt template file, and a test script. Soon your directory looks like this:

```
sentiment_project/
├── sentiment.py
├── config.py
├── data_loader.py
├── prompts.py
├── test_sentiment.py
├── test_data_loader.py
├── requirements.txt
├── README.md
├── training_data.csv
├── evaluation_results.json
├── model_cache.pkl
├── experiment_1_results.txt
├── experiment_2_results.txt
└── .env
```

This flat structure has several problems:
- You cannot tell which files are production code and which are experiments.
- Imports become fragile as the flat namespace grows.
- Configuration is mixed with business logic, so changing a parameter means editing source code.
- There is no obvious place to add a second model or a third dataset.

## The Concept

A scalable project structure uses **separation by concern**. Every directory has one job:

- `src/` or `project_name/` — the installable package containing your production code
- `tests/` — mirrors `src/` structure, each module has a corresponding test file
- `configs/` — configuration files (YAML, TOML, JSON) kept separate from code
- `data/` — raw and processed data (often in `.gitignore`)
- `notebooks/` — exploratory Jupyter notebooks
- `scripts/` — one-off automation and entry points
- `pyproject.toml` — modern project metadata and dependency specification

The root `src/` directory is structured as an installable package so you can run tests and imports from anywhere without path manipulation tricks.
