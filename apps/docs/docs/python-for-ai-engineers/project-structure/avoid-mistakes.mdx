# Project Structure — Common Mistakes

⏱ 7 minutes

## When NOT to Use a Full Project Structure

- **Single-script analysis:** A Jupyter notebook or single Python script for one-off data exploration does not need `src/`, `tests/`, and `configs/`. Adding ceremony to exploratory work slows iteration.
- **Prototypes under 200 lines:** If the entire project fits in one file and has no external consumers, a flat structure is fine. Add structure when you add tests or a second consumer.
- **Tiny microservices:** A service with exactly one endpoint and no data processing pipeline often works better as a thin module than a full project tree.

## The Pattern to Remember

- **Pattern:** `src/project_name/` as the installable package, `tests/` mirroring the source, `configs/` for parameters, `data/` for datasets, `notebooks/` for exploration, `scripts/` for entry points.
- **When to use:** Any project that will be tested, shared, deployed, or revisited after more than a few days.
- **When NOT to use:** Single-script analysis, throwaway prototypes, or services with fewer than three source files.
- **Mental model:** Think of your project as a library you are publishing — even if you never publish it. The structure tells every future reader what each part is for.

## Common Mistakes

### Mistake 1: Running Everything from the Project Root Without an Installable Package

```python
# scripts/run.py
import sys
sys.path.insert(0, "../src")  # ❌ Path manipulation hack
from my_project import config  # Fragile — breaks depending on cwd
```

**Why it happens:** You want to import from `src/` but running `python scripts/run.py` does not add `src/` to `sys.path` by default.

**Symptoms:** `ModuleNotFoundError` when running from a different directory. You resort to `sys.path.insert(0, ...)` which makes the project brittle.

**Fix:** Install the project in editable mode with `pip install -e .` so `src/my_project` is always importable regardless of working directory.

### Mistake 2: Putting Data and Logs Inside the Package

```python
# src/my_project/config.py — data mixed into source
DATA_DIR = Path(__file__).parent / "data"  # ❌ Installed package dir
LOG_DIR = Path(__file__).parent / "logs"   # ❌ Writes to site-packages
```

**Why it happens:** It seems convenient to keep everything under one tree. But installing the package copies it into `site-packages`, and writing data there causes permission issues and breaks clean uninstalls.

**Symptoms:** You reinstall the package and lose all your data. Or you get `PermissionError` when trying to write logs because `site-packages` is owned by root.

**Fix:** Use environment variables or config files to point to data and log directories outside the package tree:

```python
import os
DATA_DIR = os.environ.get("PROJECT_DATA_DIR", "data")
```

### Mistake 3: One Giant Tests Directory Without Mirroring

```
tests/
├── test_this.py
├── test_that.py
└── utils.py  # ❌ No relation to src/ structure
```

**Why it happens:** You add tests incrementally without planning the layout.

**Symptoms:** As the test suite grows, you cannot find which test corresponds to which module. Test utility code gets duplicated.

**Fix:** Mirror `src/` layout inside `tests/`:

```
tests/
├── data/
│   └── test_loader.py
├── models/
│   └── test_classifier.py
└── conftest.py  # Shared fixtures, not utilities
```

### Mistake 4: Storing API Keys and Secrets in Config Files Committed to Git

```yaml
# configs/default.yaml — ❌ Committed secrets
api_key: sk-123456789abcdef
database_url: postgres://user:password@localhost:5432/db
```

**Why it happens:** It is the fastest way to make credentials available to your code.

**Symptoms:** Your credentials end up in your public GitHub repository. Automated scanners flag them, and you have to rotate every leaked key.

**Fix:** Use environment variables or a `.env` file (listed in `.gitignore`) and load them with `os.getenv()` or a library like `python-dotenv`.

```python
import os
api_key = os.getenv("OPENAI_API_KEY")
if not api_key:
    raise ValueError("OPENAI_API_KEY environment variable not set")
```

### Mistake 5: No `__init__.py` or Forgetting to Export Public API

```python
# src/my_project/__init__.py is empty — ❌

# Elsewhere
from my_project import load_config  # AttributeError
```

**Why it happens:** You assume empty `__init__.py` is always sufficient. While namespace packages work, an empty file does not export anything for a clean public API.

**Symptoms:** Users of your package have to know the full module path (`from my_project.config import load_config`) instead of a clean `from my_project import load_config`.

**Fix:** Use `__init__.py` to define your public API:

```python
# src/my_project/__init__.py
from my_project.config import load_config
from my_project.pipelines.train_pipeline import run_pipeline

__all__ = ["load_config", "run_pipeline"]
```
