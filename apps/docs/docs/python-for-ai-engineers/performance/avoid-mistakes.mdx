# Performance — Common Mistakes

⏱ 7 minutes

## When NOT to Optimise

- **Before profiling:** Optimising without data is guessing. You will optimise the wrong part 9 times out of 10.
- **During initial development:** Write clear, correct code first. Performance problems that seem obvious during design often disappear after profiling reveals the real bottleneck.
- **For one-off scripts:** A script that processes 1,000 documents once does not need caching, async, or vectorisation. It needs to be correct and readable.
- **On very small data:** Optimising a loop that runs 100 times saves at most a few microseconds. The maintenance cost of the optimised version often exceeds the gain.

## The Pattern to Remember

- **Pattern:** Profile → Identify bottleneck → Apply technique → Measure again. Repeat until the performance is acceptable.
- **When to use:** When the pipeline runs too slowly in production, when a batch job times out, when compute costs exceed budget.
- **When NOT to use:** During initial development, on one-off scripts, on tiny datasets, or when the bottleneck is a network call you cannot parallelise.
- **Mental model:** Performance is a budget, not a goal. Know your budget (max latency, max cost per document) and optimise only until you meet it.

## Common Mistakes

### Mistake 1: Premature Optimisation

```python
# ❌ Optimised before it was needed — hard to read, no faster in practice
def process(items):
    return [x * 2 for x in items if x is not None]  # Fine

# But before writing the function, someone spent 2 hours
# trying different comprehension styles — and the bottleneck
# was actually the database query, not the list processing.
```

**Why it happens:** The instinct to write efficient code is strong. But without profiling, you do not know what is slow.

**Symptoms:** Complex code that is hard to read, with no measurable performance gain over a simple version.

**Fix:** Write simple code first. Profile it. If a section uses less than 10% of total runtime, do not optimise it — the complexity is not worth the gain.

### Mistake 2: Misunderstanding What Vectorisation Actually Requires

```python
import numpy as np

# ❌ Still a Python loop — not actually vectorised
def compute(arrays: List[np.ndarray]) -> float:
    total = 0.0
    for arr in arrays:  # Python for-loop — one element at a time
        total += arr.sum()
    return total
```

**Why it happens:** You import NumPy and assume that using arrays automatically means fast C-level operations. But iterating over arrays in a Python for-loop defeats the purpose.

**Symptoms:** Code that uses NumPy arrays but is as slow as plain Python. Profiling shows the loop itself as the bottleneck.

**Fix:** Use NumPy operations on the entire array, not on slices in a loop. If you must iterate, consider `numpy.vectorize` or `numpy.apply_along_axis` — but even those are wrappers around loops. True vectorisation means no Python-level loop at all:

```python
# ✅ True vectorisation — no Python loop
def compute_fast(arrays: np.ndarray) -> float:
    return arrays.sum()  # Operates on the entire array at C speed
```

### Mistake 3: Caching Without Considering Memory

```python
from functools import lru_cache

@lru_cache(maxsize=None)  # ❌ Unlimited cache — memory leak risk
def compute_embedding(text: str) -> List[float]:
    ...
```

**Why it happens:** You focus on speed and forget that every cached result consumes memory. For embeddings, each entry can be kilobytes. For large datasets, an unlimited cache can consume gigabytes.

**Symptoms:** The process grows in memory usage until it is killed by the OS. The profiler shows the cache as the top memory consumer.

**Fix:** Set a `maxsize` based on your working set. Use `@lru_cache(maxsize=1024)` for moderate caches, or use `diskcache` for larger datasets that do not fit in memory. Monitor cache hit rate:

```python
cache_info = compute_embedding.cache_info()
hit_rate = cache_info.hits / (cache_info.hits + cache_info.misses)
print(f"Cache hit rate: {hit_rate:.1%}")
```

If the hit rate is below 10%, the cache is wasting space — consider whether caching is the right technique at all.

### Mistake 4: Too Much Concurrency (Thundering Herd)

```python
import asyncio

# ❌ Spawns 500 concurrent requests — rate limits, timeouts, OOM
async def send_all(texts):
    async with httpx.AsyncClient() as client:
        tasks = [client.post(URL, json={"text": t}) for t in texts]
        responses = await asyncio.gather(*tasks)  # All at once!
        return responses
```

**Why it happens:** More concurrency should be faster, right? In practice, APIs enforce rate limits, and the client machine has finite connection pool and memory.

**Symptoms:** You see `429 Too Many Requests` errors. The client runs out of file descriptors. Responses time out because the connection pool is saturated.

**Fix:** Use a semaphore to limit concurrency:

```python
async def send_limited(texts: List[str], max_concurrent: int = 10):
    sem = asyncio.Semaphore(max_concurrent)

    async def send_one(text: str):
        async with sem:
            async with httpx.AsyncClient() as client:
                resp = await client.post(URL, json={"text": text})
                return resp

    tasks = [send_one(t) for t in texts]
    return await asyncio.gather(*tasks)
```

Start with `max_concurrent=10` and increase based on the API's rate limits.

### Mistake 5: Optimising Cold Paths (The Wrong 10%)

```python
# ❌ Optimising this — it runs once at startup
def load_config(path):
    with open(path) as f:
        return yaml.safe_load(f)

# But ignoring this — it runs 10,000 times per batch
def preprocess_document(doc):
    # A slow inner loop that goes unoptimised
    ...
```

**Why it happens:** You look for the most obviously "slow-looking" code — nested loops, complex expressions — and optimise that. But obvious slow code is often on a cold path (called once), while a hidden bottleneck on the hot path (called thousands of times) goes unnoticed.

**Symptoms:** The code looks faster in isolation, but the overall runtime barely changes. Profiling shows the real bottleneck was somewhere else.

**Fix:** Profile first. Sort by cumulative time. Fix the top entry on the cumulative list, not the most complex-looking function. Every optimisation must start with a measurement showing it is worth doing.
