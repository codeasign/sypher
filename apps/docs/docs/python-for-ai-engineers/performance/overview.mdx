# Performance

- Understand when optimisation is worth the effort and when it is premature
- Write data-processing pipelines that scale from thousands to millions of records

## Why This Matters

AI engineering is bottleneck-driven work. Your training pipeline might process 50,000 documents through an embedding model. Your evaluation script might run 10,000 LLM API calls. A preprocessing step that takes 2 seconds per file becomes 30 minutes at 1,000 files. Performance is not an afterthought — it is a design constraint.

In production AI systems, performance directly affects cost. Every unnecessary API call, every redundant computation, every slow loop burns money. An embedding pipeline that takes 3 hours instead of 30 minutes costs 6x more in compute. Optimising the hot path is often the highest-ROI activity you can do.

## The Problem

You write a preprocessing function that cleans text data, splits it into chunks, computes embeddings, and stores the results. On your laptop with a sample of 100 documents it runs in 3 seconds. Fine. You deploy it to process 50,000 documents. Now it takes 25 minutes. The batch job times out. Your users are waiting. You do not know which part is slow.

This is the universal performance story: the code worked on small data, and nobody profiled it before scaling up. The fix is not "write faster code" — it is "find the bottleneck and fix only that."

## The Concept

Performance optimisation follows a simple rule: **measure before you change**. Guessing is almost always wrong. The Python ecosystem provides two built-in tools for measuring:

- `time` module — simple wall-clock timing for rough estimates
- `cProfile` — deterministic profiler that shows call counts and per-function time

Beyond profiling, three techniques solve most performance problems in AI code:

**Vectorisation** replaces Python loops with C-level array operations (NumPy, pandas). A loop over 100,000 elements that takes 200ms in pure Python takes 2ms when vectorised.

**Caching** (memoization) stores the results of expensive function calls so repeated calls with the same arguments return instantly. Essential for API calls, file reads, and computation-heavy functions.

**Concurrency** overlaps I/O-bound work — multiple API calls, file reads, database queries — so total wall-clock time is the slowest single operation, not the sum of all operations.
