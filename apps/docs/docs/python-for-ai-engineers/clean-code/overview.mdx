# Clean Code

⏱ 10 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Apply naming conventions, function design principles, and documentation standards
- Refactor monolithic code into small, focused functions with clear responsibilities
- Use type hints and docstrings to make code self-documenting
- Write code that is easy to test, debug, and modify

## Prerequisites

You should already understand:

- [Functions](../functions/overview) — defining and calling functions in Python
- [Type hints](../type-hints/overview) — annotating function signatures
- [Error handling](../error-handling/overview) — using try/except blocks

## Why This Matters

AI code is particularly prone to messiness. You iterate rapidly: tweak a prompt, add a preprocessing step, dump results into a notebook cell, extract it to a script, add another model, compare outputs. Without deliberate structure, the codebase becomes a collection of scripts where nobody is sure which version was used to produce a given result.

Clean code is not about aesthetics — it is about economics. Every hour you spend writing clean code saves days of debugging later. In a team setting, clean code is the difference between a PR that gets reviewed in ten minutes and one that sits for days while reviewers try to understand what it does. The Python ecosystem has strong conventions (PEP 8, PEP 257, PEP 484), and following them means any Python developer can read your code immediately.

## The Problem

Consider this real-world AI script that has grown organically:

```python
import openai, os, json, csv
from pathlib import Path

d = Path("data")
f = d / "reviews.csv"
r = []
with open(f) as file:
    reader = csv.DictReader(file)
    for row in reader:
        r.append(row)

res = []
for item in r:
    x = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": "Is this positive or negative? " + item["text"]}]
    )
    res.append({"text": item["text"], "label": item["label"], "pred": x.choices[0].message.content})

good = 0
for item in res:
    if item["label"].strip().lower() == item["pred"].strip().lower():
        good += 1
print(good / len(res))
```

This code works, but it has every clean-code problem at once:
- Single-letter variable names (`d`, `f`, `r`, `x`, `res`)
- No functions — everything runs at module level
- Mixed concerns — file I/O, API calls, and evaluation in one block
- No error handling, no type hints, no documentation
- Hardcoded file paths and model names

You cannot test this code, you cannot reuse parts of it, and you cannot safely modify it without breaking something.

## The Concept

Clean code follows a small set of principles:

**Name things what they are.** A variable holding reviews should be called `reviews`, not `r`. A function that loads data should be called `load_reviews`, not `process`.

**One function, one job.** A function should do one thing and do it well. If a function loads data, it should not also classify it. If a function computes accuracy, it should not also print results.

**Return values, not side effects.** Functions should return data rather than printing or writing to files. The caller decides what to do with the result.

**Document intent, not mechanics.** A docstring should explain *why* and *what*, not regurgitate the code. Type hints tell the reader what types flow through the interface without reading the body.
