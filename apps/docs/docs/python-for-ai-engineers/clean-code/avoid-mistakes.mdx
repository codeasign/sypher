# Clean Code — Common Mistakes

⏱ 7 minutes

## When NOT to Use Strict Clean-Code Patterns

- **Data science exploration:** In a Jupyter notebook or during initial exploration, write messy code. The goal is understanding the data, not production quality. Refactor only when you move code into scripts.
- **Performance-critical inner loops:** Sometimes a single-letter variable in a hot loop is fine — `for i in range(n):` is idiomatic, and the extra characters of `for index in range(total_iterations):` add no clarity.
- **One-off scripts:** If you will run a script twice and delete it, skip the docstrings and type hints.
- **Glue code:** A 10-line function that calls three APIs in sequence does not need a class hierarchy. Keep it simple.

## The Pattern to Remember

- **Pattern:** Small functions with clear names, type hints, and docstrings that explain *why*. One function, one job. Return values, not side effects.
- **When to use:** Any code that will be tested, reviewed by others, or used for more than a week.
- **When NOT to use:** Exploratory analysis, throwaway scripts, or the innermost loop of a performance-critical section.
- **Mental model:** Write code for the person who will debug it at 2 AM — that person might be you six months from now.

## Common Mistakes

### Mistake 1: Using Comments Instead of Better Names

```python
# ❌ Comment explains what a better name would convey
# Check if the model response contains an error
r = getattr(resp, "error", None)  # r is the error object
if r:
    ...

# ✅ No comment needed — the name says it all
response_error = getattr(api_response, "error", None)
if response_error:
    ...
```

**Why it happens:** You write code quickly, name variables for yourself in the moment, and add a comment as a shortcut instead of renaming.

**Symptoms:** Comments and code drift apart. You update the code but forget to update the comment, so the comment becomes misleading.

**Fix:** If you need a comment to explain what a variable or function does, rename it instead. Good names eliminate the need for most inline comments.

### Mistake 2: Functions That Do Too Much

```python
# ❌ One function does everything
def process_data(path: str) -> float:
    data = pd.read_csv(path)                 # Loading
    data = data.dropna()                     # Cleaning
    predictions = model.predict(data)         # Inference
    accuracy = (predictions == data.label).mean()  # Evaluation
    pd.DataFrame({"pred": predictions}).to_csv("results.csv")  # Saving
    return accuracy                           # Return

# ✅ Each concern has its own function
def load_data(path: str) -> pd.DataFrame: ...
def clean_data(df: pd.DataFrame) -> pd.DataFrame: ...
def evaluate_model(df: pd.DataFrame, model) -> float: ...
def save_results(predictions: list, path: str) -> None: ...
def run_pipeline(config: dict) -> float: ...  # Orchestrates the others
```

**Why it happens:** It is faster to add one more line to an existing function than to create a new function. The threshold for extracting a new function feels higher than it is.

**Symptoms:** The function has multiple stages separated by blank-line comments like `# Load data`, `# Clean data`, `# Predict`. The function name uses vague verbs like `process`, `handle`, or `do_work`.

**Fix:** Extract every stage that could be tested or reused independently into its own function. A good rule: if you cannot name the function with a single precise verb and noun (`compute_accuracy`, `load_reviews`), it does too much.

### Mistake 3: Mutable Default Arguments

```python
# ❌ Mutable default — surprise behaviour
def add_tag(tags: list = []) -> list:
    tags.append("new_tag")
    return tags

print(add_tag())   # ['new_tag']
print(add_tag())   # ['new_tag', 'new_tag']  ❌ Unexpected!
```

**Why it happens:** The default argument is evaluated once at function definition time, not each call. Every invocation that uses the default shares the same list object.

**Symptoms:** Results accumulate across calls when you expect a fresh start each time.

**Fix:** Use `None` and create a new list inside the function:

```python
def add_tag(tags: Optional[list] = None) -> list:
    if tags is None:
        tags = []
    tags.append("new_tag")
    return tags
```

### Mistake 4: Bare Except Clauses

```python
# ❌ Swallows every error including KeyboardInterrupt
try:
    result = api_call()
except:  # Bare except — catches everything
    result = None

# ✅ Catch only what you expect
try:
    result = api_call()
except (ConnectionError, TimeoutError) as e:
    logger.warning("API call failed: %s", e)
    result = None
```

**Why it happens:** You want to make the code robust against crashes, so you wrap everything in a blanket `try/except`.

**Symptoms:** The code never crashes — but it silently produces wrong results because errors you did not expect (like `ValueError` from bad data) are also caught.

**Fix:** Always specify the exception type. Log the exception so you know when it happens. Never use bare `except:`.

### Mistake 5: Not Using `functools.wraps` in Decorators

```python
# ❌ Decorator without wraps — loses function metadata
def log_call(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log_call
def predict(text: str) -> str: ...

print(predict.__name__)  # 'wrapper' — not 'predict'!

# ✅ With wraps — preserves metadata
from functools import wraps

def log_call(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper
```

**Why it happens:** You focus on the decorator's behaviour and forget that `wrapper` replaces the original function object.

**Symptoms:** Debugging tools, profilers, and documentation generators show `wrapper` instead of the real function name. Stack traces are harder to read.

**Fix:** Always apply `@wraps` in your decorators. It copies `__name__`, `__doc__`, `__module__`, and `__annotations__` from the original function to the wrapper.
