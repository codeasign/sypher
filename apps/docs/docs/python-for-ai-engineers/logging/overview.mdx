# Logging

⏱ 10 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Configure the `logging` module with levels, handlers, and formatters
- Write log messages at appropriate levels: DEBUG, INFO, WARNING, ERROR, CRITICAL
- Add contextual information to log entries
- Route different loggers to different outputs (console, file)
- Understand when to use logging versus `print()` versus structured logging

## Prerequisites

You should already understand:

- [Modules and imports](../modules/overview) — logging is a standard library module
- [File handling](../file-handling/overview) — logging to files requires file I/O

## Why This Matters

`print()` is for exploration. Logging is for production. In AI engineering, you need to track: which model version was used, how long each pipeline step took, how many API calls succeeded or failed, what parameters were passed, and where errors occurred.

Logging lets you control verbosity independently of the code — you can run at DEBUG level during development and ERROR level in production. It adds timestamps, source locations, and structured fields that make debugging vastly easier. When something goes wrong at 3 AM, well-structured logs are what let you diagnose the problem without reproducing it.

## The Problem

You have a pipeline with data loading, inference, and evaluation. You add `print()` statements everywhere to track progress. In production, the output is a wall of text with no timestamps, no severity levels, and no way to filter. When an error occurs, you cannot find it in the noise. You comment out some `print()` calls, but the next developer adds new ones. It is unmanageable.

You need a system where: you can set a verbosity level, messages have timestamps and severity tags, different parts of the pipeline can be controlled separately, and output can go to files and the console simultaneously.

## The Concept

Python's `logging` module provides hierarchy of loggers, each with a level and a set of handlers. A **logger** is what you write messages to. A **handler** decides where the output goes (console, file, network). A **formatter** controls what each message looks like.

Log levels, in increasing severity: `DEBUG < INFO < WARNING < ERROR < CRITICAL`. A logger set to `WARNING` discards all `DEBUG` and `INFO` messages.
