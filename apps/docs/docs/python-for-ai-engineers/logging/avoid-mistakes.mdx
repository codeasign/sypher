# Logging — Common Mistakes

⏱ 5 minutes

## When NOT to Use Logging

- **User-facing output:** Use `print()` or a proper CLI library for output the user should see (e.g., CLI results, prompts). Logging is for operational records.
- **Performance-critical hot paths:** String formatting in log messages has a cost even if the message is not output. Use lazy formatting with `logger.debug("value: %s", expensive_func())` — the function is only called if the level is enabled.
- **Structured machine-readable output:** For data pipelines that produce output consumed by other programs, use JSON or CSV files, not log files.

## The Pattern to Remember

- **Pattern:** Create a module-level logger with `logger = logging.getLogger(__name__)`, configure it in your app's entry point, and use `logger.info()`, `logger.error()`, etc. throughout.
- **When to use:** All production code, all long-running scripts, any code that runs unattended.
- **When NOT to use:** User-facing output, performance-critical hot paths without lazy formatting.
- **Mental model:** Logging is the black box of your application. When something crashes, the log is where you look for the sequence of events that led to the crash.

## Common Mistakes

### Mistake 1: Using `print()` Instead of `logging`

`print()` goes to stdout and cannot be filtered, formatted, or redirected independently. In production code, never use `print()` for operational messages — only for CLI output that the user explicitly requested.

### Mistake 2: Setting the Level on a Handler but Not the Logger

```python
logger = logging.getLogger("myapp")
logger.setLevel(logging.DEBUG)           # This is required!

handler = logging.StreamHandler()
handler.setLevel(logging.DEBUG)          # Also set here
logger.addHandler(handler)

logger.debug("test")  # Works — both logger and handler allow DEBUG
```

If you forget `logger.setLevel(logging.DEBUG)`, the logger's default level is `WARNING` and the handler's DEBUG setting never comes into play.

### Mistake 3: Logging Sensitive Information

```python
logger.info(f"User credentials: {password}")  # ❌ Never log secrets!
logger.info(f"Full API response: {response}")  # May contain API keys
```

**Fix:** Sanitise sensitive fields before logging. Log that authentication occurred, not the credentials themselves.

### Mistake 4: Creating a New Logger Instance Every Call

```python
def process():
    logger = logging.getLogger(__name__)  # OK — module-level is standard
    # But if done inside a hot loop:
    for i in range(10000):
        logger = logging.getLogger(f"batch_{i}")  # ❌ Creates 10,000 loggers
```

**Fix:** Create loggers once, typically at module level, and reuse them.

### Mistake 5: Not Using `logger.exception()` in Exception Handlers

```python
try:
    risky_operation()
except Exception as e:
    logger.error(str(e))  # Loses the traceback!
```

**Fix:** Use `logger.exception()` inside `except` blocks — it includes the full traceback automatically:

```python
try:
    risky_operation()
except Exception:
    logger.exception("Operation failed")
```
