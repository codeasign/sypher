## Why This Matters

LLMs have a knowledge cutoff — they know nothing after their training date and nothing about your private documents. Fine-tuning can teach new facts but is expensive, slow to update, and cannot be changed per query.

RAG solves this by giving the model relevant documents alongside the question at inference time. The model reads the retrieved passages and answers based on that context, not its training data. This means your system can answer questions about documents that did not exist when the model was trained — your company's internal wiki, yesterday's product changelog, or a PDF you uploaded five minutes ago.

Every major production LLM application uses some form of RAG: customer support bots retrieve from knowledge bases, code assistants retrieve from internal repositories, research tools retrieve from uploaded papers. It is the standard architecture for grounding LLMs in your data.

## The Problem

You ask an LLM "What is our company's policy on remote work?" If the policy was written after the model's training cutoff, the model either hallucinates an answer or says it does not know.

The naive solution is to dump every document into a single massive prompt. That breaks on context window limits, cost (every token of every document is billed per query), and relevance (the model struggles to find the relevant needle in a haystack of thousands of tokens).

You need a system that: stores all documents in a searchable index, retrieves only the most relevant passages for a given question, and feeds those passages as context to the LLM.

## The Concept

RAG has two phases:

1. **Ingestion.** Documents are split into chunks, each chunk is embedded, and the chunk-embedding pairs are stored in a vector database.
2. **Generation.** A query comes in. It is embedded with the same model. The vector database returns the most similar chunks. Those chunks are inserted into a prompt as context, and the LLM generates an answer grounded in that context.

The key insight: the LLM never sees the full document. It only sees the top-K most relevant chunks. This keeps costs low and focus high.
