## Why This Matters

AI code is particularly prone to silent failures: the model returns wrong output but does not crash, the embedding dimension changes with a new model version, the pipeline stage produces a different shape than expected. Without tests, these failures propagate to production where they corrupt data or produce wrong answers that look plausible.

Testing is what separates a prototype from a production system. Pytest makes it easy to write small, focused tests that run in milliseconds. You run them on every change, every PR, and every CI build. When a bug surfaces in production, you write a test that reproduces it and add it to your suite so it never comes back.

## The Problem

You refactored the `extract_info` stage of your email pipeline. It still runs without errors. But the output format changed slightly — a field name changed from `customer_name` to `name`. The downstream `generate_response` stage now receives None for the name field and generates a generic greeting instead of the customer's name.

No error was thrown. No log looks suspicious. But customer satisfaction drops because every email starts with "Dear Valued Customer." A test that asserts the exact output schema would have caught this in CI before deployment.

## The Concept

A test in pytest is a function whose name starts with `test_` and contains assertions. Pytest discovers these functions automatically, runs them, and reports which passed and which failed.

Key concepts:

- **Assertions** — plain Python `assert` statements. No special API needed.
- **Fixtures** — functions marked with `@pytest.fixture` that provide test data or dependencies. They run before the test and clean up after.
- **Parametrization** — `@pytest.mark.parametrize` runs the same test with multiple inputs.
- **Mocking** — `unittest.mock` replaces real API calls with fake ones during tests.
