# CSV — Common Mistakes

⏱ 5 minutes

## When NOT to Use CSV

- **Nested or hierarchical data:** CSV has one level — columns. For nested data like API responses (dicts within lists), use JSON.
- **Data with many columns (100+):** CSV becomes unwieldy. Consider Parquet or a database.
- **Data needing schemas and types:** CSV has no type system — every value is a string. For typed data, use Parquet, Arrow, or a database table.
- **Large datasets (1 GB+):** CSV is slow to parse and takes more space than binary formats. Use Parquet or HDF5.

## The Pattern to Remember

- **Pattern:** Open with `open(path, "r", newline="", encoding="utf-8")`, use `csv.reader` or `csv.DictReader`, iterate rows, convert string values to appropriate types.
- **When to use:** Tabular datasets, experiment logs, evaluation results, configuration tables, data interchange with spreadsheets and databases.
- **When NOT to use:** Nested data, wide tables, large datasets, typed data needing a schema.
- **Mental model:** Every cell is a string. CSV handles the parsing; you handle the validation and type conversion.

## Common Mistakes

### Mistake 1: Not Using `newline=""` When Opening CSV Files

```python
import csv

# Wrong — causes blank lines on Windows when writing
with open("data.csv", "w", encoding="utf-8") as f:
    writer = csv.writer(f)
    writer.writerow(["a", "b"])

# Correct
with open("data.csv", "w", newline="", encoding="utf-8") as f:
    writer = csv.writer(f)
```

**Why it happens:** On Windows, `\n` is translated to `\r\n`. The csv module does its own newline handling, and when the OS also adds newline translation, you get doubled `\r\r\n` sequences resulting in blank lines between rows.

**Symptoms:** Blank lines appear between every row when writing CSV files on Windows.

**Fix:** Always pass `newline=""` when opening any file for the csv module, on any platform.

### Mistake 2: Forgetting That All Values Are Strings

```python
import csv

with open("data.csv", "w", newline="", encoding="utf-8") as f:
    f.write("value\n0.94\n0.93\n")

with open("data.csv", "r", newline="", encoding="utf-8") as f:
    reader = csv.reader(f)
    next(reader)  # skip header
    for row in reader:
        total = row[0] + 0.05  # TypeError: can only concatenate str (not "float") to str
```

**Why it happens:** Every value from a CSV file is a string, even if it looks like a number. You must convert explicitly.

**Symptoms:** `TypeError` when doing arithmetic on CSV values. Unexpected string concatenation instead of addition.

**Fix:** Convert values to the correct types when you read them:

```python
value = float(row[0])  # Convert to float
```

### Mistake 3: Assuming the Delimiter Is Always a Comma

```python
import csv

# Some exports use tab, semicolon, or pipe
data = "model|accuracy|latency_ms\ngpt-4|0.94|1200\n"

# Wrong — comma is the default delimiter
reader = csv.reader(data.splitlines())
for row in reader:
    print(row)  # ['model|accuracy|latency_ms'] — entire row as one field!

# Correct — specify the delimiter
reader = csv.reader(data.splitlines(), delimiter="|")
for row in reader:
    print(row)  # ['model', 'accuracy', 'latency_ms']
```

**Why it happens:** The csv module defaults to comma delimiter. Files from different regions and systems use different delimiters — semicolons are common in European locales where comma is the decimal separator.

**Symptoms:** The entire row appears as a single column, or columns are not split correctly.

**Fix:** Inspect the file before parsing. Check for tabs, semicolons, pipes, or other delimiters. Use `csv.Sniffer()` to detect the dialect automatically, or specify the delimiter explicitly.

### Mistake 4: Not Handling Quote Characters Correctly

```python
import csv

# A field with an embedded double quote
data = 'model,description\ngpt-4,"Model with 8"" context"\n'
# The correct CSV writes "" to escape a quote

reader = csv.DictReader(data.splitlines())
for row in reader:
    print(row)
    # {'model': 'gpt-4', 'description': 'Model with 8" context'}
```

**Why it happens:** CSV fields containing the quote character, delimiter, or newlines must be quoted. The quote character itself is escaped by doubling it (`""`). If you generate CSV by hand with simple string formatting instead of using the csv module, you will produce invalid files.

**Symptoms:** Parsing errors when reading a CSV file. Fields are split at wrong positions. Quotes appear in unexpected places.

**Fix:** Always use the csv module to both read and write CSV files — it handles quoting, escaping, and newlines correctly. Never generate CSV by hand with string formatting.

### Mistake 5: Ignoring Byte Order Marks (BOM) in Excel-Exported CSV

```python
import csv

# Excel on Windows often saves CSV with a UTF-8 BOM (\\ufeff)
# This makes the first field of the first row include the BOM
data = "﻿model,accuracy\ngpt-4,0.94\n"

reader = csv.DictReader(data.splitlines())
for row in reader:
    print(list(row.keys()))
    # ['\\ufeffmodel', 'accuracy'] — note the BOM prefix!
```

**Why it happens:** Excel adds a BOM character (`U+FEFF`) at the start of UTF-8 CSV files. The csv module does not strip it, so the first column name includes the BOM.

**Symptoms:** The first column name has an invisible prefix. `df["model"]` raises `KeyError` because the actual column name is `"﻿model"`. Comparisons and lookups fail silently.

**Fix:** Strip the BOM when reading:

```python
with open("data.csv", "r", newline="", encoding="utf-8-sig") as f:
    reader = csv.DictReader(f)
    for row in reader:
        print(row)  # BOM is stripped by utf-8-sig encoding
```

Use `encoding="utf-8-sig"` instead of `encoding="utf-8"` to handle the BOM transparently.
