The **Single Responsibility Principle (SRP)** states that "a class should have one, and only one, reason to change." Coined by Robert C. Martin (Uncle Bob), this is the first of the five SOLID principles of object-oriented design. It asserts that every class, module, or function should be responsible for a single part of the program's functionality and should encapsulate that part completely. When a class has more than one responsibility, those responsibilities become coupled — changes to one can affect the other, creating fragility throughout the system.

## Why It Exists

Before SRP became codified as a design principle, software engineering suffered from what were colloquially called "God classes" — monolithic classes that tried to do everything. In early enterprise Java applications (circa late 1990s), it was common to find classes like `OrderManager` that handled database connections, business logic, email notifications, log writing, and UI formatting all in one file.

The problem was practical, not theoretical. Consider a team shipping a new e-commerce platform. The `OrderProcessor` class handles validation, persistence, email, and logging. When the marketing team asks for a new email template, a developer edits `OrderProcessor`. When the database schema changes for a new feature, another developer edits `OrderProcessor`. When the ops team wants a different log format, yet another developer edits `OrderProcessor`. The same class has four different reasons to change, owned by four different stakeholders. The merge conflicts alone become a project risk, and every change risks breaking functionality maintained by someone else.

The industry learned this lesson the hard way through projects that became unmaintainable within 18 months. SRP emerged as a direct response to that pain, formalizing what great developers already practiced: separate the concerns.

## The Violation

Here is a realistic violation of SRP in an order processing system. The `OrderProcessor` class handles every concern in one place:

```python
# VIOLATION: OrderProcessor has 4 reasons to change
# 1. Validation rules change
# 2. Database schema/storage changes
# 3. Email template/provider changes
# 4. Logging format/destination changes

class OrderProcessor:
    def __init__(self):
        self.db = DatabaseConnection()
        self.logger = Logger()

    def process_order(self, order_data: dict) -> bool:
        # Responsibility 1: Validation
        if not order_data.get("items"):
            raise ValueError("Order must have items")
        if order_data["total"] < 0:
            raise ValueError("Total cannot be negative")

        # Responsibility 2: Database persistence
        order_id = self.db.insert("orders", order_data)
        for item in order_data["items"]:
            self.db.insert("order_items", {
                "order_id": order_id,
                "product": item["product"],
                "quantity": item["quantity"],
                "price": item["price"]
            })

        # Responsibility 3: Email notification
        email_server = EmailServer(
            host="smtp.example.com",
            port=587
        )
        email_server.send(
            to=order_data["customer_email"],
            subject=f"Order {order_id} Confirmed",
            body=f"Thank you for your order of ${order_data['total']:.2f}"
        )

        # Responsibility 4: Logging
        self.logger.write(f"Order {order_id} processed successfully")
        return True
```

What breaks when each responsibility changes:

| Change | Impact |
|---|---|
| New validation rule (e.g., tax ID required) | Edit `OrderProcessor.process_order` |
| Switch from MySQL to PostgreSQL | Edit `OrderProcessor.__init__` and `process_order` |
| New email provider (e.g., SendGrid) | Edit `OrderProcessor.process_order` |
| Change log format to JSON | Edit `OrderProcessor.__init__` and `process_order` |
| Add unit test | Must mock DB, email server, and logger simultaneously |

The process_order method mixes business logic (validation, total calculation) with infrastructure concerns (database I/O, network I/O, file I/O). A unit test for a simple validation rule requires mocking a database, an email server, and a log file — three infrastructure concerns that have nothing to do with the business rule you are testing.

## The Correct Implementation

The SRP-correct version splits each responsibility into its own class. Note that the high-level orchestration still needs to compose these, but each piece has exactly one reason to change:

```python
# SRP: Each class has exactly one responsibility

class OrderValidator:
    """Responsibility: validate order data — one reason to change."""
    def validate(self, order_data: dict) -> None:
        if not order_data.get("items"):
            raise ValueError("Order must have items")
        if order_data["total"] < 0:
            raise ValueError("Total cannot be negative")

class OrderRepository:
    """Responsibility: persist order data — one reason to change."""
    def __init__(self, db: DatabaseConnection):
        self.db = db

    def save(self, order_data: dict) -> int:
        order_id = self.db.insert("orders", order_data)
        for item in order_data["items"]:
            self.db.insert("order_items", {
                "order_id": order_id,
                "product": item["product"],
                "quantity": item["quantity"],
                "price": item["price"]
            })
        return order_id

class EmailService:
    """Responsibility: send email notifications — one reason to change."""
    def send_confirmation(self, to: str, order_id: int, total: float) -> None:
        email_server = EmailServer(
            host="smtp.example.com",
            port=587
        )
        email_server.send(
            to=to,
            subject=f"Order {order_id} Confirmed",
            body=f"Thank you for your order of ${total:.2f}"
        )

class OrderLogger:
    """Responsibility: record order events — one reason to change."""
    def __init__(self):
        self.logger = Logger()

    def log_success(self, order_id: int) -> None:
        self.logger.write(f"Order {order_id} processed successfully")
```

The orchestration class delegates to each single-responsibility component:

```python
class OrderProcessor:
    """Orchestration — composes the SRP components."""
    def __init__(
        self,
        validator: OrderValidator,
        repository: OrderRepository,
        email_service: EmailService,
        logger: OrderLogger
    ):
        self.validator = validator
        self.repository = repository
        self.email_service = email_service
        self.logger = logger

    def process_order(self, order_data: dict) -> bool:
        self.validator.validate(order_data)
        order_id = self.repository.save(order_data)
        self.email_service.send_confirmation(
            to=order_data["customer_email"],
            order_id=order_id,
            total=order_data["total"]
        )
        self.logger.log_success(order_id)
        return True
```

Now the benefits are clear:

| Aspect | Violation | SRP-Correct |
|---|---|---|
| Testability | Mock 3 infrastructure deps to test validation | Test `OrderValidator` in isolation |
| Change impact | One change risks 4 stakeholders | Each class is independent |
| Reusability | Can't reuse validation without DB | `OrderValidator` works anywhere |
| Team workflow | Merge conflicts on one file | Each class can be owned by different devs |

## How to Spot Violations in Code Review

Watch for these concrete signals in every pull request:

- **Class names ending in `Manager`, `Util`, `Helper`** — these are admissions that the class has no single responsibility. They exist to dump orphan methods.
- **Constructor taking 5+ parameters from unrelated domains** — a class receiving a database connection, an email client, a logger, a file system handler, and an HTTP client likely violates SRP.
- **Import statements from unrelated packages** — a file importing `java.sql`, `javax.mail`, `java.io`, and `java.net` simultaneously is a red flag.
- **Methods named `process`, `handle`, `doEverything`, `execute`** — generic verbs that don't name what is being done.
- **Classes over 200 lines with no obvious grouping** — if you can't name the single responsibility in under 10 words, the class does too much.
- **Mix of business logic and I/O in the same method** — a method that validates data and writes to a file in the same block has at least two responsibilities.
- **Any class with a `throws` clause listing multiple exception types from different packages** — each exception type corresponds to a different domain concern.

## Real-World Occurrences

SRP is baked into many well-designed frameworks and libraries you use daily:

| Framework | Application of SRP |
|---|---|
| **React** | Components render UI; hooks manage state and side effects; event handlers are separate. A `UserProfile` component does not also make API calls — that is delegated to a `useUser` hook or a separate data layer. |
| **Express.js** | Middleware handles one concern (auth, logging, parsing). Routes dispatch to controllers. Controllers delegate to services. Each layer has one job. |
| **Spring Boot** | Controller (HTTP), Service (business logic), Repository (data access) are rigidly separated. A `@Service` never contains `@GetMapping` or `@Repository` annotations. |
| **Django** | Models define schema, Views handle HTTP, Templates render HTML, Forms validate input, Admin registers UI. Each component has a clear single scope. |
| **Rust std library** | The `Iterator` trait handles only iteration. `Read` handles only byte input. `Write` handles only byte output. These are composed rather than merged into a single `Stream` interface. |
| **.NET Core** | Middleware pipeline, controllers, services, and data access layers each have a single focus. ASP.NET Core's DI container enforces this separation. |

## When Strict Adherence Hurts

SRP is a design principle, not a law of nature. Over-applying it leads to its own problems:

**Premature decomposition.** A 50-line script that reads a file, processes data, and writes output does not need four separate classes with interfaces. The overhead of file organization, naming, and navigation is a real cost. Apply SRP when the component demonstrates a need for independent change, testing, or reuse.

**Excessive indirection.** If every one-line method has its own class, callers navigate a maze of tiny files to understand what should be a straightforward operation. Pragmatic SRP groups related small operations into cohesive units rather than scattering them into individual classes.

**Prototyping and exploration.** When you are still discovering the shape of the solution, premature SRP gets in the way. Refactor toward SRP as the design stabilizes and you can identify the actual seams in the system. Trying to guess the right boundaries upfront leads to churn.

**Performance-critical code.** Splitting concerns across classes can introduce virtual dispatch overhead, cache misses from object traversal, and allocation pressure from many small objects. In hot paths, it can be pragmatic to temporarily merge responsibilities and document the tradeoff.

The rule of thumb: extract a new class when the existing class has two different stakeholders, two different rates of change, or two different infrastructure concerns. If you are speculating about future reasons to change, keep the code together until you have real evidence.
