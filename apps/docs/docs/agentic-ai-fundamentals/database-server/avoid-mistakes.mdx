# Avoid Mistakes: Database MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A database MCP server gives an agent direct SQL access. Do not use it when:

- **You need real-time analytics dashboards** — use a dedicated BI tool; MCP is designed for conversational agent access, not dashboard rendering
- **The database contains PII that the agent should not see in full** — the agent reads all columns returned by its queries; consider a read-only view that masks sensitive columns
- **Write access to production is required** — avoid giving any agent direct write access to production databases; use a staging replica or a restricted API instead
- **Schema changes frequently without documentation** — the agent introspects the schema at query time, but rapidly changing tables can confuse its reasoning

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | SQL execution through typed, bounded MCP tools with read/write separation |
| **When to use** | Agent needs to explore, query, or analyze database data |
| **When NOT to use** | Production writes, PII-heavy datasets, real-time dashboards |
| **Mental model** | A read-only SQL console with a 1000-row limit and a 30-second timeout |

## Common Mistakes

### Mistake 1: Allowing Writes on Production

**Mistake:** Configuring a database server against a production database without setting `PG_READ_ONLY=true`.

**Why it happens:** The default configuration is read-write. Many developers test against their local database, which is fine, then deploy the same config to the production MCP setup.

**Symptoms:** The agent accidentally runs a `DELETE` or `UPDATE` without a `WHERE` clause, modifying production data.

**Fix:** Always set `PG_READ_ONLY=true` for production databases. Use a separate server configuration (with a different name) for staging databases where writes are allowed.

### Mistake 2: Not Handling Connection Timeouts

**Mistake:** Long-running queries block the connection pool.

**Why it happens:** The agent constructs a query that runs for minutes instead of seconds. The MCP transport times out (typically 60 seconds) before the database returns results.

**Symptoms:** The agent reports an error but the database continues running the query, blocking other operations.

**Fix:** Set statement timeouts at the database connection level, and configure the MCP server to cancel long-running queries after a shorter period.

### Mistake 3: Assuming Table Names without Introspection

**Mistake:** The agent hardcodes table or column names without checking.

**Why it happens:** The agent forms assumptions based on its training data (e.g., "every e-commerce app has a `products` table"). The actual table might be named `items` or `catalog`.

**Symptoms:** The agent writes `SELECT * FROM products` and gets a "table not found" error.

**Fix:** Encourage the agent to call `list_tables()` and `describe_table()` before constructing queries. The official servers support this naturally — the agent learns to introspect.

### Mistake 4: Ignoring Transaction Isolation

**Mistake:** Reading uncommitted data during a write transaction.

**Why it happens:** The agent runs a `write_query` (INSERT) followed by a `read_query` (SELECT) and expects to see the new data. Depending on isolation levels, the SELECT may not see uncommitted writes.

**Symptoms:** Inconsistent query results — the agent "sees" data in one query and not in the next.

**Fix:** Each tool call uses a separate connection. Use `write_query` with a response that includes the inserted row's ID, and for subsequent reads, use `READ COMMITTED` isolation.

### Mistake 5: Overlooking Query Injection Through Tool Parameters

**Mistake:** Building SQL by concatenating user-provided strings passed through MCP.

**Why it happens:** A naive server builds SQL like `f"SELECT * FROM {table_name}"` instead of using parameterized queries.

**Symptoms:** A user prompt like "ignore all tables and drop the database" becomes part of the SQL string.

**Fix:** Use parameterized queries everywhere, even for table names if your database driver supports it (via `quote_ident` in PostgreSQL or `PRAGMA table_info` with validated table names).
