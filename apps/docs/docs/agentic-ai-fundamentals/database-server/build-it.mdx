# Build It: Configuring Database MCP Servers

⏱️ 20 minutes

## Breaking It Down

### 1. Installing the Official SQLite MCP Server

The [MCP SQLite server](https://github.com/modelcontextprotocol/servers/tree/main/src/sqlite) lets your agent query SQLite databases directly:

```bash
npm install -g @modelcontextprotocol/server-sqlite
```

Configure it in `settings.json`:

```json
{
  "mcpServers": {
    "sqlite": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-sqlite",
        "/workspace/data/app.db"
      ]
    }
  }
}
```

The argument is the path to your SQLite database file. If the file does not exist, the server creates it on first write.

Available tools:

| Tool | Purpose |
|---|---|
| `read_query` | Execute SELECT queries and return results |
| `write_query` | Execute INSERT, UPDATE, DELETE, and DDL |
| `create_table` | Create a new table with specified columns |
| `list_tables` | List all tables in the database |
| `describe_table` | Show column names, types, and constraints for a table |
| `append_insight` | Add a note to a running analysis log |

### 2. Running SQLite Queries Through an Agent

Once connected, you can ask questions like "what tables exist in the database?" The agent calls:

```
list_tables()
```

Which returns something like:

```json
[
  {"name": "users", "type": "table"},
  {"name": "orders", "type": "table"},
  {"name": "products", "type": "table"}
]
```

The agent then calls `describe_table("users")` to understand the schema before asking for data. This automatic schema discovery means you never need to tell the agent what columns exist — it discovers them on its own.

Ask the agent "how many users signed up in the last 7 days?" and the agent calls:

```
read_query(sql: "SELECT COUNT(*) as count FROM users WHERE created_at >= datetime('now', '-7 days')")
```

### 3. Configuring PostgreSQL with Read-Only Mode

For production PostgreSQL databases, use the official MCP PostgreSQL server:

```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://user:password@localhost:5432/mydb"
      ],
      "env": {
        "PG_READ_ONLY": "true"
      }
    }
  }
}
```

The `PG_READ_ONLY` environment variable restricts the server to SELECT queries only. Any attempt to INSERT, UPDATE, DELETE, or run DDL returns an error. This is the safe default for production databases:

```json
{
  "mcpServers": {
    "postgres-readonly": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://readonly_user:password@prod-host:5432/proddb"
      ],
      "env": {
        "PG_READ_ONLY": "true"
      }
    }
  }
}
```

Add a separate read-write configuration for staging environments:

```json
{
  "mcpServers": {
    "postgres-staging": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://staging_user:password@staging-host:5432/stagingdb"
      ]
    }
  }
}
```

Name your servers descriptively — `postgres-readonly` and `postgres-staging` — so the agent can choose the right one based on the task.

### 4. Building a Custom Database Server with the MCP SDK

For non-SQLite databases or custom query logic, build your own:

```python
# db_server.py
from mcp import Server
from mcp.types import TextContent
import sqlite3
import re

server = Server("custom-db")

READ_ONLY_SQL = re.compile(r"^\s*SELECT\b", re.IGNORECASE)

def get_connection():
    return sqlite3.connect("/workspace/data/app.db")

@server.tool("read_query")
async def read_query(sql: str) -> TextContent:
    """Execute a read-only SELECT query and return results."""
    if not READ_ONLY_SQL.match(sql):
        return TextContent(type="text", text="Error: Only SELECT queries are allowed in read_query")
    conn = get_connection()
    try:
        cursor = conn.execute(sql)
        columns = [desc[0] for desc in cursor.description]
        rows = cursor.fetchmany(1000)  # hard limit of 1000 rows
        result = [dict(zip(columns, row)) for row in rows]
        return TextContent(type="text", text=str(result))
    finally:
        conn.close()

@server.tool("describe_table")
async def describe_table(table_name: str) -> TextContent:
    """Describe the columns of a table."""
    conn = get_connection()
    try:
        cursor = conn.execute(f"PRAGMA table_info({table_name})")
        columns = [{"name": row[1], "type": row[2], "nullable": not row[3], "default": row[4]} for row in cursor]
        return TextContent(type="text", text=str(columns))
    finally:
        conn.close()

@server.tool("list_tables")
async def list_tables() -> TextContent:
    """List all tables in the database."""
    conn = get_connection()
    try:
        cursor = conn.execute("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name")
        tables = [row[0] for row in cursor]
        return TextContent(type="text", text=str(tables))
    finally:
        conn.close()
```

### 5. Running Safe Analytical Queries

One of the most powerful patterns is letting the agent answer business questions by writing its own SQL:

> What are the top 5 products by revenue this month, and which users bought them?

The agent's reasoning chain goes:
1. `list_tables()` — discovers `products`, `orders`, `users`, `order_items`
2. `describe_table("orders")` — sees `id`, `user_id`, `created_at`, `total`
3. `describe_table("order_items")` — sees `order_id`, `product_id`, `quantity`, `price`
4. Constructs and runs:

```sql
SELECT p.name, SUM(oi.quantity * oi.price) as revenue,
       COUNT(DISTINCT o.user_id) as buyers
FROM order_items oi
JOIN products p ON oi.product_id = p.id
JOIN orders o ON oi.order_id = o.id
WHERE strftime('%Y-%m', o.created_at) = strftime('%Y-%m', 'now')
GROUP BY p.id
ORDER BY revenue DESC
LIMIT 5
```

The agent discovers the schema, self-corrects if a column name is wrong, and returns the correct query — without you writing a single line of SQL.

## Performance Considerations

Each `read_query` call opens a new database connection. For SQLite on WAL mode, this is fast (sub-millisecond open). For PostgreSQL, connection overhead is higher — consider using a connection pool. Set statement timeouts to prevent long-running queries from blocking:

```sql
-- For PostgreSQL, set in the connection args:
postgresql://user:pass@host:5432/db?options=-c%20statement_timeout%3D30000
```

Row limits prevent large result sets from flooding the MCP transport. The default 1000 row limit works for exploration; increase it only when you know the agent needs more data.
