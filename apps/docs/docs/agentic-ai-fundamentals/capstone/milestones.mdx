# Milestones

Suggested checkpoints. These are not mandatory steps — they exist so you have a way to know if you're on track. Skip ahead if you prefer.

## Milestone 1 — Core Agent Loop with Plan-and-Execute

**Goal:** Build the basic agent that accepts a question, generates a plan, and runs through the Plan-and-Execute loop — without any real MCP servers yet. Use mock tools (two simple Python functions: `mock_search` and `mock_fetch`) to prove the loop works.

**You'll know this works when:** You give the agent a question like "Find the population of France and the capital of Japan." It generates a plan, calls `mock_search` for each step, reports results, and marks steps complete. You can see the plan document evolve in the logs.

**Tasks:**
- Implement the system prompt for the research assistant persona
- Implement the Plan-and-Execute loop: generate plan → execute steps in order → update plan → detect completion
- Build a mock tool dispatcher that routes `search` and `fetch` calls to stub functions
- Add structured JSON logging for every loop iteration
- Add a max-turns safety guard (stop after 25 turns)

## Milestone 2 — MCP Server Integration

**Goal:** Replace mock tools with real MCP servers. Connect the agent to a Web Search MCP server, a GitHub MCP server, and a Database MCP server. The agent calls real tools and processes live results.

**You'll know this works when:** You ask "What are the latest releases from the python/cpython repo and summarize the top 3 news articles about Python 3.13?" The agent searches GitHub via the GitHub server, searches the web via the Web Search server, and stores the conversation in the Database server — all in one session.

**Tasks:**
- Build or configure the Web Search MCP server (Brave Search or equivalent)
- Build or configure the GitHub MCP server with a personal access token
- Build a Database MCP server (SQLite) with `read_query` and `write_query` tools
- Write an MCP client module that discovers tools from each server and translates them to your agent's tool format
- Add retry logic (exponential backoff, 3 attempts) to every tool call
- Add tool-level timeout enforcement (30s default)

## Milestone 3 — Memory, Logging, and Tracing

**Goal:** Add conversation memory (short-term and long-term), structured logging with session trace IDs, and a trace viewer that lets you inspect a complete agent run.

**You'll know this works when:** You ask a follow-up question that references a previous answer ("In that last search, what was the second result about?") and the agent retrieves the context correctly. Every session produces a trace file with the complete timeline of LLM calls, tool calls, errors, and retries.

**Tasks:**
- Implement short-term memory: sliding window of the last N turns (default 20) in the LLM context
- Implement long-term memory: store key findings in the database server and retrieve them via semantic or keyword matching
- Add a `trace_id` (UUID) to every session and attach it to every log record
- Implement structured JSON logging: every LLM call, tool call, plan update, retry, and reflection step produces a log entry with `trace_id`, `timestamp`, `event_type`, and `payload`
- Write a `trace_viewer.py` script that reads a trace file and prints a human-readable timeline
- Add cost tracking: log token usage and estimated cost per LLM call, accumulate per session

## Milestone 4 — Self-Evaluation, Citations, and Confidence Scores

**Goal:** The agent reflects on its own output, generates per-claim confidence scores with citations, flags low-confidence claims, and revises before delivering the final answer.

**You'll know this works when:** The agent answers a research question and the final output includes inline citations like `[1]` mapped to a citation list, each claim has a score in parentheses like `(confidence: 0.85)`, and claims below 0.6 are flagged with a review warning. The agent also produces a self-evaluation section that scores its own completeness, accuracy, and efficiency.

**Tasks:**
- Add a reflection phase after all plan steps are complete: the agent reviews its draft output for missing citations, weak claims, and contradictions
- Implement structured output prompting for the reflection result (JSON schema with `claims: [{claim, confidence, citation, flag}]`)
- Build a citation tracker that maps source URLs to numbered citations
- Add a revision loop: if reflection finds issues, the agent revises and reflects again (max 3 revision cycles)
- Add the self-evaluation step: prompt the agent to rate its own output on completeness, accuracy, and efficiency (1–10 scale with justification)
- Write a test that verifies the agent flags a deliberately fabricated claim

## Milestone 5 — Report Export and Safety

**Goal:** The agent exports a professional report in Markdown and HTML formats, with a human approval gate before writing to disk, and defense-in-depth guardrails against prompt injection.

**You'll know this works when:** The agent generates a complete report with executive summary, methodology, findings (with confidence scores and citations), self-evaluation, and limitations. Before the export writes a file, the agent pauses and asks for human approval. You can cancel the export and the agent returns the report as printed text instead.

**Tasks:**
- Design the report template: executive summary, methodology, findings table, confidence summary, citations, self-evaluation appendix
- Implement the Markdown report renderer
- Implement the HTML report renderer (a clean, readable single-page HTML document)
- Add a human-in-the-loop gate: before writing the export file, the agent prints a summary and asks for confirmation (`y`/`n`/`cancel`)
- Implement input sanitization on user questions (strip control characters, detect embedded instructions)
- Implement output validation: verify that the export does not contain raw API keys or system prompt text
- Add tool-level guardrails: the write-to-file tool refuses paths outside the project directory

**Relevant lessons:**
- [Human in the Loop](../human-in-the-loop/overview)
- [Prompt Injection and Guardrails](../prompt-injection-and-guardrails/overview)
- [Configuration Management](../configuration-management/overview)

## Milestone 6 — Integration, Tests, and Polish

**Goal:** Everything works end-to-end. All components are wired together. Automated tests pass. The agent handles edge cases gracefully.

**You'll know this works when:** You run a full end-to-end test with a realistic research question and get a complete, cited, scored report in under 4 minutes. You run the test suite and all tests pass. You ask an empty question and the agent asks for clarification. You provoke a tool timeout and the agent retries, logs it, and continues with best-available data.

**Tasks:**
- Write integration test: full research question → report (use mock MCP servers for the test)
- Write unit tests: plan generation, tool dispatch, reflection, citation formatting, confidence scoring
- Write edge-case test: empty input, rate-limited tool (simulated), all tools failing, contradictory sources
- Wire up configuration management: `config/default.yaml`, `config/dev.yaml`, `config/prod.yaml` with layered merging
- Add docstrings to every module and function
- Create a `README.md` with setup instructions, architecture overview, and usage examples
- (Optional) Dockerize the agent and MCP servers for reproducible runs

## Final milestone — Done

**You'll know the capstone is complete when:**

- [ ] The agent accepts a natural-language research question
- [ ] A structured plan is generated and visible in the logs
- [ ] At least 3 MCP servers are connected and called during execution
- [ ] The agent retries failed tool calls with exponential backoff
- [ ] Short-term and long-term memory work across turns
- [ ] Every session produces structured logs with a trace ID
- [ ] The agent reflects on its output before finalizing
- [ ] Every claim has a confidence score and a citation
- [ ] Low-confidence claims are flagged
- [ ] The agent exports a professional report (Markdown + HTML)
- [ ] The export has a human approval gate
- [ ] The agent self-evaluates its completeness, accuracy, and efficiency
- [ ] All automated tests pass
- [ ] A `README.md` documents the project
- [ ] The agent handles edge cases (empty input, tool failures, contradictions) without crashing
