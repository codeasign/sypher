# Self-Assessment Rubric

Use this to evaluate your own build. There is no single right answer — this rubric tells you what "portfolio-worthy" looks like, not what exact code to write.

## Functionality (40%)

| Criteria | What "excellent" looks like |
|---|---|
| Plan-and-Execute loop | A structured plan is generated before any tool executes, stored as a mutable document outside the message list, and updated after every step. Failed steps trigger replanning, not a crash or restart. The plan is visible in the trace output. |
| MCP server integration | At least 3 MCP servers are connected via stdio transport, discovered at startup through `tools/list`, and called dynamically during execution. No hardcoded tool wrappers — tools come from the MCP protocol. A custom MCP server is built for any additional capability. |
| Retry and recovery | Every retriable tool failure triggers exponential backoff (min 3 attempts). The agent continues with best-available data if a non-critical step fails after all retries. Circuit-breaker logic prevents cascading retries on persistent failures. |
| Memory | Short-term memory uses a sliding window (configurable depth). Long-term memory stores key findings in the database server and retrieves them on follow-up questions. The agent successfully references earlier findings in later turns. |
| Reflection and self-correction | The agent reflects on its draft output before finalizing. The reflection checks citations, contradictions, and completeness. Issues trigger a revision loop (up to 3 cycles). The final output is measurably better than the first draft. |
| Citations and confidence | Every claim has an inline citation anchor and a confidence score. Each citation maps to a URL (for web sources) or a query identifier (for database sources). Scores below 0.6 are flagged with a visible review warning. A complete citation list appears in the report footer. |
| Report export | The agent produces a Markdown report and an HTML report. Both include executive summary, methodology, findings with scores and citations, confidence summary, limitations, and self-evaluation appendix. The report is grammatically correct, well-structured, and ready to share. |
| Self-evaluation | The agent scores itself on completeness, accuracy, and efficiency with numeric ratings and written justifications. The evaluation is honest — it acknowledges limitations and missing data. The evaluation appears in the report appendix. |

**Excellent:** 8/8 criteria met fully. **Good:** 6/8 met, one partially. **Needs work:** Fewer than 5 fully met.

## Code/Configuration quality (25%)

| Criteria | What "excellent" looks like |
|---|---|
| Project structure | Clear directory layout: `src/` for agent code, `servers/` for MCP servers, `config/` for configuration files, `tests/` for tests, `traces/` for session traces. A `README.md` documents setup, usage, and architecture. |
| Code quality | All functions have type annotations and docstrings. The agent loop is readable — a developer unfamiliar with the code can follow the observe-think-act cycle in under 5 minutes. No dead code, no commented-out blocks, no magic numbers. |
| Configuration management | Environment-specific values (API keys, model names, server commands, thresholds) live in a layered config system. Defaults are in `config/default.yaml`, environment overrides in `config/{env}.yaml`. Secrets are never hardcoded. |
| Testing | At least 8 automated tests covering: plan generation, tool dispatch, retry logic, reflection, citation formatting, confidence scoring, report rendering, and a full end-to-end integration test (with mock MCP servers). Tests run with a single `pytest` command. |
| Logging and observability | Every agent action produces a structured JSON log entry. Every session has a unique trace ID. A trace viewer script reconstructs a readable timeline from any trace file. Token usage and estimated cost are logged per LLM call and accumulated per session. |

**Excellent:** All 5 criteria met. **Good:** 4 met. **Needs work:** 2 or fewer met.

## Engineering judgment (20%)

This project forces you to make real tradeoffs. Below are the key decision points and what a strong versus weak answer looks like for each.

### Single model vs. separated reasoning and execution

**Strong answer:** You separate reasoning (Claude Sonnet for planning) from execution (Claude Haiku for tool calls) and justify the choice with measured cost/latency data. You discuss the overhead of the separation (two round-trips per step) and show it pays off in total session cost.

**Weak answer:** You use the same model for everything without discussing why, or you separate them without measuring the impact.

### Plan format: free text vs. structured JSON

**Strong answer:** You choose a structured plan format (JSON with steps, status, dependencies) and justify why the structure helps with parsing, updating, and logging. You show how your plan updater handles step failure, reordering, and completion detection.

**Weak answer:** The plan is free text in the system prompt with no way to programmatically update or log it.

### Source reconciliation strategy

**Strong answer:** You handle contradictory sources (e.g., two web pages with different numbers) by noting the discrepancy, preferring the more authoritative source, and lowering the confidence score for contested claims. Your approach is documented and visible in the output.

**Weak answer:** You pick the first result and ignore contradictions, or you present contradictory data without acknowledgment.

### Memory granularity

**Strong answer:** You choose what to store in long-term memory (key findings, source URLs, resolved contradictions) and what to leave in short-term memory (raw tool outputs, intermediate reasoning). Your choice is based on token budget and retrieval accuracy, not convenience.

**Weak answer:** You store everything in both memories, or you only implement short-term memory without retrieval.

## Production readiness (15%)

| Criteria | What "excellent" looks like |
|---|---|
| Error handling | Every external call (LLM API, MCP tool, file write) is wrapped in error handling. The agent never terminates with an unhandled exception. Errors are logged with context, not silently swallowed. |
| Safety | Defense-in-depth is implemented: input sanitization strips embedded instructions from user input; output validation prevents API keys or system prompt leaks in exports; tool-level guardrails block writes outside the project directory; the export export requires human approval. |
| Observability | Traces are complete enough to debug any session. Given a trace ID, you can reconstruct the exact plan, every tool call with input/output, every LLM call with token counts, every retry, and the reflection results. |
| Configuration | The agent works in at least two environments (development and production) with different config files. Changing the model, confidence threshold, or log level does not require code changes. |

**Excellent:** 4/4 criteria met. **Good:** 3 met. **Needs work:** 1 or fewer met.

## How to use this rubric

Score yourself honestly against each section. A portfolio-worthy result scores **strong on Functionality** and **Engineering judgment** even if Production readiness is partial — that's expected for a learning project. Use weak scores to identify which course lessons to revisit. If your engineering judgment section scores low, review the lessons on planning, reflection, and trajectory evaluation to understand the tradeoffs you missed.
