The system has five top-level modules:

- **Orchestrator** contains the Planner (generates the plan from the user question), Executor (runs the observe-think-act loop, dispatches tool calls, handles retries), Reflector (critiques the draft output), and Exporter (renders the final report).
- **Tool Dispatcher** wraps every MCP tool call with retry logic, timeout enforcement, and a circuit breaker. It translates between the agent's internal tool format and the MCP protocol.
- **MCP Servers** are standalone processes spawned by the agent at startup. Each server registers its tools via the MCP protocol. The agent discovers them via `tools/list` and calls them via `tools/call`.
- **Cross-cutting modules** handle logging/tracing (every event is logged with a trace ID), memory (short-term in context, long-term in the database server), and configuration (layered YAML files merged at startup).
- **Reports** are rendered by the Exporter as Markdown and HTML, after a human approval gate.

## Key design decisions and why

### Decision: Plan format — structured JSON steps

**One reasonable choice:** Store the plan as a JSON array of objects, each with `id`, `description`, `status` ("pending" / "in_progress" / "completed" / "failed" / "skipped"), `depends_on`, and `result`. The Planner generates this array, the Executor reads and updates it after each step, and the Reflector reads the completed plan to evaluate coverage.

**Why:** JSON is trivial to log, parse, update, and render. A plan step with `status: "failed"` triggers replanning logic without any natural-language parsing. The Logger can snapshot the plan after every change, giving you a complete audit trail of how the plan evolved.

**A valid alternative:** A markdown checklist embedded in the system prompt, updated by the agent rewriting the checklist each turn. Simpler but harder to parse programmatically and more prone to formatting drift over long sessions.

### Decision: Source reconciliation — prefer authoritative, flag discrepancies

**One reasonable choice:** When two sources disagree, the agent prefers the more authoritative source (official documentation over a blog post, a primary source over a summary, a more recent publication over an older one). The discrepancy is noted in the finding text, and the confidence score for the contested claim is reduced (e.g., from 0.90 to 0.65). The citation list includes both sources with a note about the contradiction.

**Why:** This gives the user the most useful information (the best available answer plus awareness of uncertainty) without hiding the ambiguity. The flagged claim invites the user to investigate further, which is the right behavior for a research assistant.

**A valid alternative:** Present both views neutrally with equal confidence and let the user decide. This is more honest but less useful for users who want a clear recommendation.

### Decision: Memory strategy — store key findings, not raw tool output

**One reasonable choice:** Long-term memory stores structured findings extracted by the agent after each plan step completes — not the raw tool output. Each finding is a JSON object with `claim`, `sources: [urls]`, `confidence`, and `timestamp`. The database MCP server stores these in a `findings` table. On follow-up questions, the agent queries the database with keyword matching to retrieve relevant prior findings.

**Why:** Raw tool output is verbose (a web search result may be 10 KB of HTML). Storing extracted findings saves tokens and retrieval time. The structured format makes it easy to include prior findings in the context window for follow-up questions.

**A valid alternative:** Store everything (raw tool output + extracted findings) in the database and rely on the LLM to retrieve and filter. More comprehensive but uses more tokens per turn and may introduce retrieval noise.

### Decision: Export gate — require human approval before writing

**One reasonable choice:** After the report is rendered, the agent prints a preview summary (file name, format, size estimate, number of sources) and waits for a `y` / `n` / `cancel` input. Only `y` writes the file. Any other input returns the report as printed text.

**Why:** Writing a file is an irreversible action. A human approval gate prevents the agent from overwriting an existing report, writing to the wrong directory, or exporting sensitive data to disk without oversight. This is the same pattern used in production agent systems for any side-effecting operation.

**A valid alternative:** Auto-export to a timestamped file in a dedicated `reports/` directory without human approval. Simpler UX but removes the safety check. If you choose this, ensure your tool-level guardrails prevent path traversal and overwrites.

## What this reference does NOT include

This reference shows architecture and design rationale, not full code. Structural snippets are fine for illustration, but this is not a working implementation. You must write the actual agent loop, MCP clients, tool dispatch, memory system, logger, reflector, and exporter yourself. The reference is meant to show ONE plausible decomposition so you can compare your own approach, not to give you a codebase to copy.

Key pieces deliberately omitted:
- The exact system prompt and tool descriptions for the research assistant persona
- The plan generation prompt and structured output schema
- The reflection prompt and confidence-scoring rubric
- The report template rendering code (Markdown and HTML)
- The retry loop implementation (timeout, backoff, circuit breaker state machine)
- The MCP client initialization and tool discovery sequence

## Common pitfalls at this scale

1. **Over-engineering the plan format.** A common mistake is designing a plan schema with dependencies, deadlines, priorities, and parallel execution before the basic loop works. Start with a flat list of steps and a single `status` field. Add complexity only when the flat plan proves insufficient.

2. **Wrapping MCP tools inside custom functions instead of calling them through the MCP protocol.** If you write a Python function `def web_search(query)` that imports a library and makes an HTTP call, you've bypassed the MCP server entirely. The agent should call the web search tool through the MCP protocol (`tools/call`) via the server you configured. If you're not going through MCP, you're not using what the course taught.

3. **Letting the reflection loop run forever.** A draft that never passes the reflection check can trigger infinite revision cycles. Always set a max revision limit (3 is a good default) and output the best available draft when the limit is reached.

4. **Storing everything in long-term memory without a retrieval strategy.** Dumping every raw tool output into the database is not memory — it's a data dump. The agent cannot find relevant information among thousands of rows without a retrieval strategy (keyword search, embedding similarity, or time-based filtering). Design your retrieval before you design your storage.

5. **Skipping the edge case tests.** Your agent will receive empty questions, contradictory sources, rate-limited APIs, and servers that crash mid-session. If you only test the happy path, the agent will fail in production-like scenarios. Write the edge case tests in Milestone 6 — they will save you hours of debugging.
