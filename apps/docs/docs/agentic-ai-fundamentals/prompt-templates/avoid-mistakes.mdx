# Prompt Templates — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Prompt templates add structure but also complexity. They are not always needed:

- **One-off prompts**: If a prompt is used in exactly one place and never changes, a template adds overhead without benefit
- **Very simple prompts**: `"Translate to French: {text}"` does not need a template system — an f-string is fine
- **Static prompts with no variables**: If the prompt never changes, hardcode it. Templates are for dynamic content
- **Early prototyping**: In the exploration phase, templates slow you down. Use inline strings and refactor to templates later

## The Pattern to Remember

**Pattern**: Centralize prompt definitions in a template library. Application code calls templates by name, not by constructing prompts inline.

When to use:
- Multiple features that each need their own prompt
- Prompts that change independently from application code
- Templates that are tested and reviewed separately

When NOT to use:
- Single-use or throwaway prompts
- Prototypes where speed of iteration matters more than structure

**Mental model**: Templates are the view layer of your AI application. Just as you separate HTML templates from server code, separate prompt templates from application logic.

## Common Mistakes

### Mistake 1: Template Injection

**Why it happens**: User input contains template syntax (e.g., `${injection}` or `{{malicious}}`) that gets interpreted by the template engine.

**Symptoms**: The rendered prompt contains unexpected content. The model receives instructions injected through user input.

**Fix**: Escape user input before passing it to the template engine. Use safe substitution methods that treat variables as literal text, not template code. Never use user input as template variable names.

### Mistake 2: Using Unsafe Substitution That Breaks on Missing Variables

**Why it happens**: A variable is not provided, and the template engine raises a KeyError or leaves the placeholder in the rendered text.

**Symptoms**: The API receives a prompt with `${undefined_variable}` still in it. The model is confused by the placeholder text.

**Fix**: Use safe_substitute (Python Template) or provide default values for every variable. Validate that all required variables are present before rendering.

### Mistake 3: Not Validating Rendered Prompt Length

**Why it happens**: Variables like document context or conversation history vary in size, and the rendered prompt can exceed the context window.

**Symptoms**: Sporadic API errors when certain inputs produce long prompts. Production failures that are hard to reproduce.

**Fix**: Estimate the token count of the rendered prompt before sending. Truncate variable content to fit within a budget.

### Mistake 4: Storing Templates in Application Code

**Why it happens**: It is the fastest way to get started — templates are strings in the same file as the business logic.

**Symptoms**: Changing a prompt requires a code deployment. Non-technical team members cannot review or edit prompts.

**Fix**: Store templates in external files (JSON, YAML, or database) loaded at runtime. This lets you change prompts without deploying code.

### Mistake 5: Not Versioning Templates

**Why it happens**: Templates are treated as code, but without explicit versioning.

**Symptoms**: A prompt change breaks the application, and you cannot roll back because the old version was overwritten.

**Fix**: Include a version field in each template. Store old versions. When deploying a prompt change, keep the previous version available for rollback.
