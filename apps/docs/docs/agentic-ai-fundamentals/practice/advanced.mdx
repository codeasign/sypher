# Agentic AI Fundamentals — Advanced Practice

⏱️ 45 minutes · Difficulty: Advanced

## Exercise 1: Multi-Agent Customer Support System with Guardrails and Observability

**Problem:** Design and build a multi-agent customer support system with the following architecture:

**Architecture:**
- A **router agent** that classifies incoming tickets into categories (billing, technical, account, general) and assigns them to the appropriate specialist agent
- A **billing specialist agent** that handles refunds, invoices, and payment issues
- A **technical support agent** that troubleshoots product issues
- An **account management agent** that handles profile changes, password resets, and account upgrades
- A **supervisor agent** that monitors all conversations and escalates risky situations

**Requirements:**

1. **Router agent:**
   - Receives the raw user message
   - Uses an LLM call to classify into one of the 4 categories (with structured output)
   - Routes to the correct specialist agent
   - If confidence is below 0.7, routes to a human handoff instead

2. **Each specialist agent:**
   - Has its own tools (e.g., billing has `process_refund`, `lookup_invoice`, `apply_credit`)
   - Runs the observe-think-act loop (see Section 3)
   - Returns structured output: `{"resolution": str, "status": "resolved" | "escalated" | "needs_human", "tool_calls_made": list[str]}`

3. **Supervisor agent:**
   - Receives the full conversation transcript after the specialist finishes
   - Checks for: refund amounts above $500, password reset requests, account deletion requests, user frustration indicators
   - If any risk flag is raised, sets `requires_approval: true` and generates an escalation summary
   - Otherwise, marks the conversation as approved

4. **Output:** A single `process_ticket(ticket: dict) -> dict` function that:
   - Runs the full pipeline (router -> specialist -> supervisor)
   - Tracks timing for each stage
   - Logs every decision to a structured log
   - Returns a `TicketResult` with: `ticket_id`, `category`, `resolution`, `status`, `requires_approval`, `stages_completed`, `total_duration_ms`, `tool_calls_made`

**Test data:**
```python
tickets = [
    {"id": "TK-001", "message": "I was charged twice for my Pro plan. Please refund the extra $50.", "user_tier": "pro"},
    {"id": "TK-002", "message": "The export feature keeps crashing when I try to download large files over 100MB.", "user_tier": "enterprise"},
    {"id": "TK-003", "message": "I need to delete my account and all associated data immediately.", "user_tier": "free"},
    {"id": "TK-004", "message": "Can you upgrade my plan from basic to premium? I want the extra storage.", "user_tier": "basic"},
]
```

**Expected output example:**
```python
{
    "ticket_id": "TK-001",
    "category": "billing",
    "resolution": "Processed refund of $50 for duplicate charge on Pro plan.",
    "status": "resolved",
    "requires_approval": False,
    "stages_completed": ["classification", "billing_agent", "supervisor_review"],
    "total_duration_ms": 3450,
    "tool_calls_made": ["lookup_invoice", "process_refund"],
}
```

**Hints:**
- The router can use a simple LLM call with structured output — no vector DB needed.
- Each specialist agent is a function that receives the ticket and runs its own loop with mocked tool responses.
- Mock all external tools (refund API, invoice lookup, etc.) with realistic behavior.
- Track durations with `time.perf_counter()` at the start of each stage.
- The supervisor should check for keywords like "delete", "refund > 500", and repeated frustration words.

**Constraints:** Must implement all 3 stages (router, specialist, supervisor). Must include structured logging of every decision point. Must handle escalation conditions. Mock all external APIs — no real LLM calls needed for the tools, but the routing and supervisor should simulate LLM decisions (use a simple rule-based classifier if needed for simulation).

## Exercise 2: Production Agent Deployment with Caching, Rate Limiting, and Health Monitoring

**Problem:** Design and build a deployable agent service that serves a `generate_report` function to users, with production-grade infrastructure.

**Architecture:**

1. **FastAPI service** with a single endpoint `POST /api/report`:
   - Accepts `{"topic": str, "format": "markdown" | "html", "max_length": int}`
   - Calls an LLM to generate the report (simulated)
   - Returns the generated report text

2. **LLM response cache** (see Section 8):
   - Cache key is a hash of `(topic, format)`
   - TTL of 1 hour
   - On cache hit: return cached response (log "cache_hit")
   - On cache miss: simulate LLM call, store in cache, return (log "cache_miss")
   - Include a `X-Cache-Status` header in the response (`HIT` or `MISS`)

3. **Rate limiter** (see Section 8):
   - 10 requests per minute per IP address
   - Uses a token bucket algorithm (in-memory)
   - Returns HTTP 429 with `Retry-After` header when exceeded
   - Rate limit info included in response headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`

4. **Health monitoring** (see Section 7 and 8):
   - `GET /health` endpoint returning:
     ```json
     {
       "status": "ok",
       "uptime_seconds": 3600,
       "requests_total": 150,
       "cache_hit_rate": 0.65,
       "rate_limit_throttled": 3,
       "avg_response_time_ms": 420
     }
     ```
   - Track: total requests, cache hits/misses, rate-limited requests, average response time
   - Reset counters on server restart (in-memory)

5. **Configuration** (see Section 8):
   - Load from environment variables with sensible defaults:
     - `CACHE_TTL_SECONDS` (default: 3600)
     - `RATE_LIMIT_PER_MINUTE` (default: 10)
     - `SIMULATED_LLM_LATENCY_MS` (default: 500)
     - `LOG_LEVEL` (default: INFO)

**Expected behavior:**
```bash
# curl request
POST /api/report
{"topic": "Q4 2025 sales performance", "format": "markdown", "max_length": 500}

# Response
200 OK
X-Cache-Status: MISS
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 9
X-RateLimit-Reset: 42

# After same request again:
200 OK
X-Cache-Status: HIT
```

**Hints:**
- Use FastAPI with a global dict for the cache (or a simple `@lru_cache` variant with TTL).
- The token bucket can be a dict keyed by IP with `{"tokens": N, "last_refill": timestamp}`.
- The simulated LLM call uses `asyncio.sleep(SIMULATED_LLM_LATENCY_MS / 1000)` then returns a template string.
- The health endpoint reads from module-level counters — use `asyncio.Lock` for thread safety.
- Use Pydantic `BaseSettings` for config loading from env vars.

**Constraints:** Must be valid FastAPI code. Must implement all three cross-cutting concerns (caching, rate limiting, monitoring). Mock the LLM call (do not call a real API). Must include startup timestamp for uptime calculation. Use `asyncio` throughout (no blocking calls). Must handle concurrent requests safely.
