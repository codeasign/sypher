# Trajectory Evaluation

## Why This Matters

Two agents can both answer "what is the capital of France?" correctly. One calls a single `web_search` tool, gets the answer, and returns it in 2 seconds. The other calls `list_directory`, then `read_file` on three wrong files, then `web_search`, then `read_file` again, and finally returns the answer after 30 seconds and 6 tool calls. Both succeeded — but one is clearly worse.

Trajectory evaluation catches this difference. It measures not just whether the agent succeeded, but how it got there. A bad trajectory wastes tokens (costing money), takes longer (frustrating users), and often correlates with higher error rates on subsequent tasks. In production, trajectory quality directly impacts your operating costs and user satisfaction.

Companies running agents at scale report that trajectory optimization is one of their highest-leverage activities. Reducing average tool calls per task from 7 to 4 cuts latency by 40-50%, reduces token costs by 30-40%, and lowers the probability of cascading errors. Trajectory evaluation is the tool that makes these improvements measurable.

## The Problem

Your agent's conversation log shows it took 12 tool calls to answer a simple question. Was that necessary? Some of those calls were wasted — the agent searched for the same information twice, or read a file, got confused, and read it again. But which ones? Without trajectory evaluation, you have no systematic way to identify and fix these inefficiencies.

The root cause is that LLMs do not plan entire trajectories upfront. They plan one step at a time, and each step depends on the output of the previous one. This greedy, local decision making naturally produces suboptimal global trajectories — the agent takes a locally reasonable action that leads into a dead end, then backtracks.

## The Concept

Trajectory evaluation treats the agent's sequence of tool calls as a path through a decision space. Each tool call is a step. A good trajectory is one that reaches the goal in the minimum number of safe, correct steps.

You evaluate trajectories along three axes:

- **Correctness** — did each step use the right tool with the right parameters?
- **Efficiency** — were there redundant calls, unnecessary loops, or excessive repeats?
- **Safety** — did any step violate constraints, even if the final answer was correct?

A trajectory score is a weighted combination of these three axes. A perfect score goes to an agent that reaches the correct answer in exactly the expected number of safe steps. A poor score goes to an agent that wastes calls, loops, or takes unsafe actions — even if it eventually gets the right answer.
