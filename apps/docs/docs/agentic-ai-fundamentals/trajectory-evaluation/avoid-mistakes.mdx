# Avoid Mistakes: Trajectory Evaluation Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Trajectory-level evaluation adds significant engineering overhead. Do not use it when:

- **You only have a few eval tasks (under 10)** — trajectory statistics from a small sample are misleading; stick to outcome-level evaluation
- **Your agent's tasks are highly diverse** — if every task requires a completely different tool sequence, the "expected" trajectory is meaningless
- **You are iterating rapidly on the agent prompt** — trajectory scores will fluctuate wildly between prompt changes; fix the prompt first, then optimize trajectories
- **The agent's tools have no meaningful ordering constraints** — for single-tool agents (just "answer question"), trajectory evaluation adds no information

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Multi-axis scoring of tool call sequences against expected trajectories |
| **When to use** | Multi-step agents where tool call efficiency and correctness matter |
| **When NOT to use** | Single-tool agents, `<10` eval tasks, highly diverse task sets |
| **Mental model** | A path-finding quality score: shortest safe path to the goal is best |

## Common Mistakes

### Mistake 1: Expecting Exact Tool Sequence Matches

**Mistake:** Defining the expected trajectory as a rigid sequence of tool names and penalizing any deviation.

**Why it happens:** It is easier to write `["search_web", "read_file", "answer"]` than to define equivalence classes.

**Symptoms:** The agent finds a more efficient path (e.g., answering from cached data without searching) but gets penalized for not following the expected sequence.

**Fix:** Define expected tool sequences as unordered sets or allow multiple valid orderings. Use partial credit — the tool `search_web` appearing anywhere in the first three calls is a pass.

### Mistake 2: Ignoring Argument Quality

**Mistake:** Scoring only the tool name, not the arguments passed to it.

**Why it happens:** Argument schema parsing is more complex than tool name matching.

**Symptoms:** An agent that calls `read_query` with the wrong table name receives the same score as one that calls it with the correct query.

**Fix:** Add argument-level scoring. Compare the actual arguments against expected arguments using a similarity metric or LLM judge:

```python
def score_tool_call_arguments(actual: dict, expected: dict) -> float:
    """Score how closely actual arguments match expected ones (0-1)."""
    matching_keys = set(actual.keys()) & set(expected.keys())
    if not matching_keys:
        return 0.0
    similarity = sum(1 for k in matching_keys if actual[k] == expected[k])
    return similarity / max(len(expected), 1)
```

### Mistake 3: Not Handling Parallel Tool Calls

**Mistake:** Assuming tool calls are sequential and penalizing valid parallel call patterns.

**Why it happens:** Most trajectory evaluation code assumes a linear sequence of tool calls.

**Symptoms:** An agent that calls `search_web` and `read_file` in parallel (legitimate for independent operations) gets scored lower than one that calls them sequentially.

**Fix:** Allow the expected trajectory to specify parallel call groups. Tool calls within the same "parallel group" can appear in any order.

### Mistake 4: Confusing Trajectory with Outcome

**Mistake:** A bad trajectory (many wasted calls) is treated as an evaluation failure, which conflates efficiency with correctness.

**Why it happens:** If your scoring function gives 40 points to correctness and 35 to efficiency, a task that succeeds but wastes calls gets a mediocre score.

**Symptoms:** The team focuses on reducing tool calls at the expense of correctness, optimizing the trajectory score but breaking real task completion.

**Fix:** Report correctness and efficiency as separate metrics, not a single composite. Set a minimum correctness threshold before efficiency matters.

### Mistake 5: Ignoring Tool Call Durations

**Mistake:** Scoring trajectories based only on tool call count, ignoring how long each call takes.

**Why it happens:** Tool call count is easy to count; duration requires timestamps on every call.

**Symptoms:** An agent that makes 3 fast calls (300ms each, 900ms total) is scored the same as one that makes 3 slow calls (3 seconds each, 9 seconds total).

**Fix:** Include duration in your trajectory scoring. A simple formula: `efficiency_score = expected_time / actual_time * 35`.
