# Avoid Mistakes: Custom MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Building a custom MCP server is the right approach for unique APIs. Do not build one when:

- **An official server already exists** — check the [MCP server repository](https://github.com/modelcontextprotocol/servers) first; the community maintains servers for databases, version control, search, and more
- **A simple script wrapper would suffice** — if your API has one or two endpoints, consider wrapping them as a single tool in an existing server rather than building a new one
- **You need client-side logic** — MCP is server-driven; tools run on the server. If the logic must run in the client (browser, mobile app), MCP is not the right abstraction
- **You have no async-capable Python runtime** — the MCP Python SDK uses `asyncio` throughout; a synchronous-only environment requires workarounds

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | SDK-scaffolded async server wrapping one backend API |
| **When to use** | Unique internal API, no off-the-shelf MCP server exists |
| **When NOT to use** | Official server exists, single endpoint, synchronous-only runtime |
| **Mental model** | A thin typed bridge layer between MCP and your API |

## Common Mistakes

### Mistake 1: Forgetting to Handle API Errors Gracefully

**Mistake:** Letting exceptions from the backend API propagate to the MCP transport unhandled.

**Why it happens:** The tool function calls the backend API and assumes it always succeeds. When the API returns an error, the raw exception crashes the server.

**Symptoms:** The MCP client shows a generic "server disconnected" error with no details about what went wrong.

**Fix:** Wrap all backend calls in try/except blocks and return user-friendly error messages:

```python
@server.tool("deploy")
async def deploy(service: str) -> TextContent:
    try:
        result = await backend.deploy(service)
        return TextContent(type="text", text=result)
    except httpx.HTTPStatusError as e:
        return TextContent(type="text", text=f"API error: {e.response.status_code} - {e.response.text[:500]}")
    except httpx.TimeoutException:
        return TextContent(type="text", text="Deploy API timed out. The service may still be deploying.")
    except Exception as e:
        return TextContent(type="text", text=f"Unexpected error: {str(e)}")
```

### Mistake 2: Not Closing HTTP Clients

**Mistake:** Creating a new `httpx.AsyncClient()` in every tool call without closing it.

**Why it happens:** It is convenient to create a client inline. Over many tool calls, the unclosed clients leak file descriptors.

**Symptoms:** The server crashes after several hours with "too many open files" or a similar OS error.

**Fix:** Create the client once in the server's lifetime:

```python
import httpx

client = httpx.AsyncClient(timeout=30)

@server.tool("deploy")
async def deploy(service: str) -> TextContent:
    resp = await client.post(f"{API}/deploy", json={"service": service})
    ...

# Clean up on shutdown
import atexit
atexit.register(lambda: anyio.run(client.aclose))
```

### Mistake 3: Defining Tools with Overly Broad Schemas

**Mistake:** Using `**kwargs` or a single `dict` parameter instead of typed parameters.

**Why it happens:** It is easy to write `async def my_tool(**kwargs)` and pass the dictionary to the backend.

**Symptoms:** The agent does not know what parameters to pass — it sees "kwargs: dict" with no description, no types, and no defaults. It either guesses wrong or asks the user for help.

**Fix:** Define explicit typed parameters for every tool:

```python
# Bad — the agent has no idea what to pass
@server.tool("deploy")
async def deploy(**kwargs) -> TextContent: ...

# Good — the agent sees the full schema
@server.tool("deploy")
async def deploy(service: str, version: str, environment: str = "staging") -> TextContent: ...
```

### Mistake 4: Ignoring Server Lifecycle Events

**Mistake:** Not handling shutdown signals, so connections and resources are not cleaned up.

**Why it happens:** The server runs as a long-lived process. When the MCP client exits, the server process is killed unceremoniously.

**Symptoms:** Database connections stay open, temp files accumulate, and port bindings remain occupied.

**Fix:** Register cleanup handlers:

```python
@server.on_shutdown
async def cleanup():
    await client.aclose()
    print("Server shutting down — resources cleaned up", flush=True)
```

### Mistake 5: Confusing stdio and SSE Transport Code

**Mistake:** Writing SSE-specific logic but running the server in stdio mode, or vice versa.

**Why it happens:** The transport mode changes how the server reads and writes messages. Code that works in SSE mode (which uses Starlette request objects) crashes in stdio mode (which uses stdin/stdout).

**Symptoms:** The server starts but the client reports "connection failed" or "invalid message format."

**Fix:** Check the transport mode at startup and use the correct handler. Provide a `--transport` flag as shown in the Build It section rather than hardcoding one transport.
