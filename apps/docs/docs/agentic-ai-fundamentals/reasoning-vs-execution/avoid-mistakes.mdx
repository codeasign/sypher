# Reasoning vs. Execution — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Separating reasoning and execution is not always beneficial:

- **Short agent runs**: If the agent completes in 1-2 turns, the separation overhead is not worth it.
- **Simple tool calls**: When every tool call is straightforward and the model never needs to deliberate, fused execution is simpler.
- **When using a single model for everything**: If you are already committed to one model, separating the phases in code adds complexity without cost savings.
- **Latency-sensitive tasks**: The two-phase approach adds an extra API call per turn, increasing latency.

## The Pattern to Remember

**Pattern**: Reasoning decides what to do. Execution does it. They can use different models, different budgets, and different error handling.

When to use separation:
- Multi-turn agents where reasoning tokens dominate
- Tasks where different steps need different capability levels
- When you want to audit or log reasoning separately from actions

When NOT to use:
- Quick 1-2 turn interactions
- Simple lookup agents
- Prototypes

**Mental model**: Reasoning is the architect drawing blueprints. Execution is the construction crew following them. You do not need an architect to hang a picture frame.

## Common Mistakes

### Mistake 1: The Execution Model Cannot Follow the Plan

**Why it happens**: You use a much smaller model for execution, but the small model does not understand the plan format.

**Symptoms**: The small model misinterprets the reasoning output, calls wrong tools, or produces malformed responses.

**Fix**: Make the plan output extremely structured — a JSON list of tool calls with clear fields. The smaller the execution model, the more structured its input must be.

### Mistake 2: Losing Context Between Phases

**Why it happens**: The reasoning phase produces a plan, and the execution phase starts with a fresh context that does not include the original user request.

**Symptoms**: The execution phase acts on the plan without understanding why. It calls tools correctly but does not relate the results back to the user's goal.

**Fix**: Always pass the original user request and any prior results into the execution context. The executor needs the full picture.

### Mistake 3: Over-Engineering the Separation

**Why it happens**: You build an elaborate two-phase system with different models, queues, and monitoring before measuring whether the fused approach actually has problems.

**Symptoms**: The separation infrastructure is more complex than the agent logic itself. Debugging the separation takes longer than debugging the agent.

**Fix**: Start fused. Measure the ratio of reasoning tokens to execution tokens. Only separate when the measurement justifies the complexity.

### Mistake 4: Hardcoding Model Assignments

**Why it happens**: You assign "Sonnet for reasoning, Haiku for execution" at design time and never revisit.

**Symptoms**: A task that needs more execution capability (complex tool calls) suffers because the execution model is too weak. A task that needs less reasoning (simple decisions) wastes money on Sonnet.

**Fix**: Make model assignments configurable per task or per request. Different tasks have different reasoning-to-execution ratios.

### Mistake 5: Not Handling Execution Failures in the Reasoning Phase

**Why it happens**: The execution phase fails (tool error, timeout), and there is no feedback path back to the reasoning phase.

**Symptoms**: The reasoning phase continues as if the execution succeeded. The plan proceeds based on incorrect assumptions about what happened.

**Fix**: Execution failures must be returned to the reasoning phase as structured feedback. The reasoning phase should explicitly consider failure recovery.
