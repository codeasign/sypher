# Reasoning vs. Execution

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Separate the reasoning step (what to do) from the execution step (doing it)
- Design systems where reasoning and execution use different models or resources
- Measure and optimize the reasoning-to-execution ratio
- Identify when fused reasoning-and-execution is better than separated

## Prerequisites

You should already understand:

- How chain-of-thought reasoning works (covered in [Chain of Thought and Reasoning](../chain-of-thought-and-reasoning/overview))
- The components of an agent (covered in [Anatomy of an Agent](../anatomy-of-an-agent/overview))
- How tool calling is implemented in LLM APIs (covered in [LLM APIs in Practice](../llm-apis-in-practice/overview))

## Why This Matters

In early agent implementations, the same LLM call that decides what to do also executes the action. This fusion is simple but wasteful — you pay reasoning-level token costs for execution-level work, and you cannot tune the two phases independently.

Production agent systems separate reasoning and execution for two reasons: cost and quality. Reasoning requires a powerful model with strong planning capabilities — expensive per token. Execution requires speed and precision — a smaller, cheaper model can format a tool call. By separating the two, you can use a frontier model to plan and a faster model to act, cutting costs by 40-60% without sacrificing quality.

## The Problem

You build an agent that needs to look up customer data, format it, and send an email. The agent uses Claude Opus for everything. Each reasoning step — "I need to query the customer database, then format the data, then send the email" — costs the same per token as the actual tool calls.

Over 10,000 conversations, the reasoning tokens outnumber the execution tokens 3:1. You are spending 75% of your API budget on the model planning what a much cheaper model could plan nearly as well. Your latency is also higher than necessary because the frontier model is slower per token.

The fix is to separate reasoning (which needs a strong model) from execution (which needs a fast model). But splitting them introduces complexity: the reasoning output must be structured enough that the execution system can follow it reliably.

## The Concept

Reasoning is the phase where the agent determines what to do next. It considers the current state, evaluates options, makes decisions, and produces a plan. This phase benefits from larger models, chain-of-thought techniques, and higher token budgets.

Execution is the phase where the agent carries out a decided action. It calls a tool, runs a query, sends a message, or produces structured output. This phase benefits from lower latency, deterministic behavior, and lower cost.

The separation can be implemented in several ways:

**Same model, separate turns**. The reasoning turn produces a plan. The execution turn carries out one step of the plan. This lets you count and budget each phase independently.

**Different models**. A large model plans. A small model executes tool calls. The small model receives the plan as context and follows instructions without creative deviation.

**Specialized reasoning modules**. A dedicated planner module handles decomposition and produces a structured plan. A separate executor module iterates through the plan's steps.
