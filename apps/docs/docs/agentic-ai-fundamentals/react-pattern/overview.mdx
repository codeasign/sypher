# ReAct Pattern

## Why This Matters

The ReAct pattern — first described by Yao et al. in 2022 — is one of the most widely used agent architectures in production. It combines reasoning traces with action steps in a single interleaved stream. Each turn of the loop produces reasoning (the "why") followed by an action (the "what"). This interleaving gives you auditability: you can see why the agent chose each action.

ReAct is the default pattern behind many popular agent frameworks including LangChain's AgentExecutor and the Vercel AI SDK's agent implementation. Understanding ReAct means you understand the architecture that powers thousands of production agents.

## The Problem

You build an agent that answers customer questions about their accounts. The agent works, but sometimes it calls the wrong tool. When you review the logs, you see the tool call but not the reasoning that led to it. You cannot tell whether the model misunderstood the question, misinterpreted the tool description, or had a context conflict.

Without the reasoning trace, debugging is guesswork. You need the agent to externalize its thinking alongside each action so you can audit, debug, and improve its behavior.

## The Concept

ReAct interleaves reasoning and action at every turn. Instead of a separate planning phase followed by execution, each cycle produces:

1. **A reasoning step** — the model explains what it is thinking, what it knows, and what it needs to find out
2. **An action** — a tool call or a final answer

The reasoning step is not hidden — it is part of the agent's output. You can read it, log it, and use it to debug. The model uses its own reasoning tokens as scratch space, which also improves the quality of the action.

ReAct is particularly effective for:

- Tasks requiring both knowledge retrieval and multi-step reasoning
- Situations where the agent must combine information from multiple tool calls
- Applications where auditability is important (finance, healthcare, legal)
