# System Prompts — Build It

*Reading time: 10 minutes*

## Breaking It Down

### 1. With and Without a System Prompt

See the difference a system prompt makes for the same user message:

```python
import anthropic

client = anthropic.Anthropic()

user_message = "Explain what an API is."

# Without system prompt
response_no_system = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=200,
    temperature=0,
    messages=[{"role": "user", "content": user_message}]
)

# With system prompt
response_with_system = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=200,
    temperature=0,
    system="You are a 5th-grade teacher who explains concepts simply. Use analogies and avoid jargon.",
    messages=[{"role": "user", "content": user_message}]
)

print("=== WITHOUT SYSTEM PROMPT ===")
print(response_no_system.content[0].text[:200])

print("\n=== WITH SYSTEM PROMPT ===")
print(response_with_system.content[0].text[:200])
```

Without a system prompt, the model defaults to its training distribution — typically neutral, generic, and at an adult reading level. With a system prompt, the output adapts to the specified persona.

### 2. Role-Setting System Prompts

The role you assign in the system prompt changes every aspect of the response:

```python
def ask_with_system_prompt(system_prompt: str, question: str) -> str:
    """Send a question with a specific system prompt."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=200,
        temperature=0,
        system=system_prompt,
        messages=[{"role": "user", "content": question}]
    )
    return response.content[0].text


question = "Is investing in cryptocurrency a good idea?"

roles = [
    (
        "You are a conservative financial advisor. "
        "Focus on risk management and diversification. "
        "Recommend against speculative investments."
    ),
    (
        "You are a tech-optimist investor. "
        "Focus on the transformative potential of blockchain. "
        "Discuss long-term growth possibilities."
    ),
    (
        "You are a neutral educator. "
        "Present both sides objectively. "
        "State facts without giving personal advice."
    ),
]

for role in roles:
    print(f"\n=== System: {role[:40]}... ===")
    print(ask_with_system_prompt(role, question))
```

Each system prompt produces a fundamentally different answer — not through prompt tricks but through the role's influence on the model's output distribution.

### 3. System Prompts with Guardrails

System prompts are the right place to set behavioral boundaries:

```python
def ask_with_guardrails(question: str) -> str:
    """Ask a question with a system prompt that enforces boundaries."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=200,
        temperature=0,
        system=(
            "You are a customer support agent for Acme Corp."
            "\n\nRULES:"
            "\n- Only answer questions about Acme Corp products"
            "\n- If asked about competitors, say: 'I can only speak about Acme Corp products'"
            "\n- Never make up specifications — if you do not know, say so"
            "\n- Always offer to transfer to a human if the customer asks"
            "\n- Keep responses under 100 words"
        ),
        messages=[{"role": "user", "content": question}]
    )
    return response.content[0].text


# Test with in-scope and out-of-scope questions
for q in [
    "What are the specs on the Acme X100 laptop?",
    "How does the Acme X100 compare to the Dell Latitude?",
    "Tell me about the weather in Tokyo."
]:
    print(f"\nQ: {q}")
    print(f"A: {ask_with_guardrails(q)}")
```

The guardrails handle each case correctly: product questions get answered, competitor comparisons are declined, and out-of-domain questions are redirected.

## How It Works Internally

The system prompt is prepended to the conversation at the API level, but its positioning in the context is distinct from user messages. In the Anthropic API, the system message is passed as a separate parameter and is processed as a persistent instruction rather than as conversational history.

The model treats the system prompt as having higher priority than user messages. This is built into the training and alignment process — instruction-tuned models learn to prioritize system-level instructions. This is why you can put rules in the system prompt and expect them to persist across the conversation, even if a user message later contradicts them.

However, system prompts are not immune to context length effects. In very long conversations, the influence of the system prompt can decay as more tokens are added. This is why some applications periodically re-send the system prompt (or key rules from it) within the conversation history.

## Build Something Real

Build a multi-role support system that uses different system prompts for different scenarios:

```python
import anthropic
from enum import Enum

client = anthropic.Anthropic()


class SupportTier(Enum):
    TIER_1 = "tier_1"
    TIER_2 = "tier_2"
    BILLING = "billing"


SYSTEM_PROMPTS = {
    SupportTier.TIER_1: (
        "You are a Tier 1 support agent for ShopFast."
        "\n\nResponsibilities: Answer common questions about orders, shipping, returns."
        "\n\nRules:"
        "\n- If you can answer from the FAQ, do so directly"
        "\n- If the issue involves account changes or refunds, escalate to Tier 2"
        "\n- Always be polite and professional"
        "\n- Keep responses under 150 words"
    ),
    SupportTier.TIER_2: (
        "You are a Tier 2 support agent for ShopFast."
        "\n\nResponsibilities: Handle account issues, complex problems, and escalations."
        "\n\nRules:"
        "\n- You can process refunds up to $500"
        "\n- You can reset accounts and change email addresses"
        "\n- For anything over $500, escalate to management"
        "\n- Document every action you take"
    ),
    SupportTier.BILLING: (
        "You are a billing specialist for ShopFast."
        "\n\nResponsibilities: Invoices, payment issues, subscription changes."
        "\n\nRules:"
        "\n- Provide exact amounts and dates when discussing charges"
        "\n- Never modify subscriptions without explicit customer confirmation"
        "\n- Offer discounts only when authorized in the policy"
        "\n- Keep responses precise and data-focused"
    ),
}


def handle_support_request(
    tier: SupportTier,
    message: str
) -> str:
    """Route a support request to the correct tier."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=300,
        temperature=0,
        system=SYSTEM_PROMPTS[tier],
        messages=[{"role": "user", "content": message}]
    )
    return response.content[0].text


# Simulate different requests
requests = [
    (SupportTier.TIER_1, "Where is my order #54321?"),
    (SupportTier.TIER_1, "I need a refund for my last purchase."),
    (SupportTier.TIER_2, "Someone hacked my account and changed the email."),
    (SupportTier.BILLING, "I was charged twice for my subscription this month."),
]

for tier, message in requests:
    print(f"\n{'='*60}")
    print(f"[{tier.value.upper()}] Customer: {message}")
    print(f"{'='*60}")
    print(handle_support_request(tier, message))
```

Each tier uses a distinct system prompt that constrains the model to the appropriate scope, escalation rules, and tone. This pattern is used in production support systems to route and handle requests at scale.
