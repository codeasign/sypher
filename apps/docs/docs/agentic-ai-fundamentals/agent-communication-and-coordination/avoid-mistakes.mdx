# Agent Communication and Coordination — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Structured agent communication adds infrastructure overhead. Skip it when:

- **Two agents, one pipeline**: If your system is two agents in a fixed sequence, direct messaging through an orchestrator is sufficient — you do not need a message bus.
- **Single-node prototype**: For local experiments, a shared dictionary or database is simpler than a message protocol.
- **Real-time audio/video agents**: Streaming media agents need specialized protocols (WebRTC, gRPC streaming), not structured JSON messages.
- **All agents on one codebase**: If every agent lives in the same process, function calls are simpler than message passing.

## The Pattern to Remember

**Pattern**: Agents exchange structured, typed messages via direct addressing or a pub/sub bus. Messages have known schemas. Conflicts are resolved by a defined strategy.

When to use:
- Three or more collaborating agents
- Dynamic topologies where agents join and leave
- Systems needing audit trails of inter-agent communication

When NOT to use:
- Fixed two-agent pipelines
- Single-process prototypes
- Streaming media applications

**Mental model**: Treat inter-agent messages like API contracts — define the schema before you build the agents, not after.

## Common Mistakes

### Mistake 1: Free-Text Messages Between Agents

**Why it happens**: Sending a plain string is faster to implement than defining a schema.

**Symptoms**: Parsing errors cascade. One agent says "revenue is up" and another interprets it as "revenue data is available." The meaning depends on who reads it.

**Fix**: Enforce a structured message format with typed fields. Every message has a sender, recipient, type, and payload. No raw strings.

### Mistake 2: Synchronous Blocking Calls

**Why it happens**: It feels natural to call agent A, wait for the result, then call agent B.

**Symptoms**: System latency equals the sum of all agent runtimes. A slow agent blocks the entire pipeline.

**Fix**: Use asynchronous messaging. Agents publish results and move on. Other agents pick up messages when they are ready.

### Mistake 3: No Message Timeout

**Why it happens**: You assume messages always arrive and agents always respond.

**Symptoms**: An agent crashes silently. The system waits forever for a response. The pipeline stalls and no one detects the stall.

**Fix**: Every message has a TTL (time-to-live). If no response arrives within the TTL, the system logs a warning and retries or escalates.

### Mistake 4: Agents That Talk Past Each Other

**Why it happens**: Agent A sends data in format X, agent B expects format Y. Neither schema is documented.

**Symptoms**: The message arrives successfully but the payload is empty or has unexpected keys. Agents run with missing data and produce wrong output.

**Fix**: Add schema validation on both the publishing and consuming sides. Reject messages that do not match the expected schema.

### Mistake 5: Ignoring Message Ordering

**Why it happens**: Messages are handled in arrival order and the system works in simple tests.

**Symptoms**: In production, an "update" message arrives before the "create" message. The update fails because the target does not exist yet.

**Fix**: Include sequence numbers or causal dependencies in messages. The message bus should deliver messages in order within each channel, or agents should handle out-of-order delivery gracefully.
