# Agent vs. Chatbot vs. Workflow

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Distinguish between chatbots, deterministic workflows, and autonomous agents
- Choose the right architecture based on task complexity, variability, and safety requirements
- Identify hybrid patterns that mix two or more architectures
- Recognize when an agent adds unnecessary complexity

## Prerequisites

You should already understand:

- What defines an agent (covered in [What Is an Agent](../what-is-an-agent/overview))
- How LLMs handle conversational turns (covered in [LLM Fundamentals](../llm-fundamentals/overview))
- How system prompts control behavior (covered in [System Prompts](../system-prompts/overview))

## Why This Matters

Every team building with LLMs eventually faces the same question: should this be a simple chat completion, a predefined pipeline, or an autonomous agent? The answer determines the cost, latency, reliability, and debuggability of your system. Choosing wrong means either building a complex agent where a chatbot would have worked, or building a chatbot that cannot handle tasks an agent should handle.

Product teams at major API companies spend significant effort on this architectural decision. The wrong choice leads to brittle pipelines that break on edge cases, or overly complex agent loops that hallucinate actions a simpler system would never attempt.

## The Problem

Your team ships a customer support bot. The first version uses a single LLM call with a system prompt: "You are a support agent. Answer questions about our return policy." It works for basic questions. Then a user asks "What is the status of my refund order #12345?"

The chatbot generates a plausible-looking answer about a refund status. But it did not actually query the database. The answer may be completely fabricated. The system has no way to take action — it can only generate text.

You have three options for what to build next:

- **Add tools to the chatbot** so it can query databases when needed — this makes it an agent
- **Build a deterministic workflow** that routes to a database query, formats the result, and returns it — controlled but rigid
- **Keep the chatbot** but accept the limitation and add a disclaimer

Each choice has trade-offs. Understanding the trade-offs is what this lesson is about.

## The Concept

The three architectures differ along three axes: control, autonomy, and complexity.

**Chatbot**. A chatbot takes a user message and returns a model response. It may use system prompts and conversation history, but it does not autonomously call tools or follow branching logic. The model is the full decision maker within the bounds of its prompt. Chatbots are the simplest architecture to build and debug.

**Workflow**. A workflow is a predefined sequence of steps. Each step may involve an LLM call, a database lookup, a conditional branch, or any other operation. The sequence is fixed at design time — the system does not decide what to do next; the developer decided at build time. Workflows are predictable, testable, and safe.

**Agent**. An agent combines an LLM with tools and memory in a loop where the LLM decides the next action at each step. The exact sequence of operations is not predetermined — the agent chooses based on the current state. This gives agents flexibility that workflows lack, at the cost of predictability.
