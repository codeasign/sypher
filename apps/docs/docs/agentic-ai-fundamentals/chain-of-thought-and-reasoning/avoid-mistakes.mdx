# Chain of Thought and Reasoning — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Chain-of-thought is not always beneficial:

- **Simple factual lookups**: "What is the capital of France?" does not need step-by-step reasoning. Direct answering is faster and cheaper.
- **Classification tasks with clear categories**: Adding CoT to a label-constraint prompt adds 200+ tokens of reasoning to a 10-token answer. The classification does not improve, but costs do.
- **Tasks where the reasoning is obvious**: If the model already answers correctly without CoT, adding it just adds tokens and latency.
- **Latency-sensitive applications**: CoT responses take longer to generate because the model produces more tokens. If you need sub-second responses, direct answering may be the only option.
- **Tasks requiring creativity**: CoT forces linear, logical reasoning. For creative writing or brainstorming, direct instruction with appropriate temperature is better.

## The Pattern to Remember

**Pattern**: Use CoT when the task requires multiple reasoning steps or involves implicit assumptions that need to be externalized.

When to use:
- Math word problems, logic puzzles, multi-hop questions
- Tasks where you need to audit the model's reasoning
- Problems that commonly trick the model into shortcut answers

When NOT to use:
- Simple fact lookups, classifications with clear labels
- Latency-critical applications
- Creative or open-ended tasks

**Mental model**: CoT is a scratch pad. If you would solve the problem with pen and paper, CoT will help. If you would answer from memory instantly, CoT is unnecessary.

## Common Mistakes

### Mistake 1: Using CoT When Direct Answering Works

**Why it happens**: CoT is a powerful technique, so developers apply it everywhere.

**Symptoms**: The API bill is unnecessarily high. Response latency is 3-5x longer than needed. Output quality is not measurably better.

**Fix**: Test both approaches on your task. If direct answering meets your accuracy requirements, use it. CoT is a tool for hard problems, not a default.

### Mistake 2: Not Structuring the Output

**Why it happens**: You tell the model to "think step by step" but do not specify how to format the final answer.

**Symptoms**: The reasoning and answer are mixed together. Your downstream code cannot extract the final answer programmatically.

**Fix**: Use structured tags (e.g., `<reasoning>` and `<answer>`) to separate the reasoning trace from the final output.

### Mistake 3: Allowing the Reasoning to Go Off-Track

**Why it happens**: CoT gives the model freedom to reason, which sometimes means it reasons in circles or goes down wrong paths.

**Symptoms**: The model produces long reasoning chains that end with wrong answers. The reasoning is internally consistent but starts from a wrong premise.

**Fix**: Use few-shot CoT examples that show correct reasoning patterns. The examples keep the model on track. For critical tasks, validate intermediate steps.

### Mistake 4: Setting Temperature Too High

**Why it happens**: Default temperature settings (0.7-1.0) from creative tasks are carried over to reasoning tasks.

**Symptoms**: The reasoning chain is creative but unreliable. The model takes interesting but wrong paths.

**Fix**: Use `temperature=0` for reasoning tasks. You want deterministic, focused reasoning, not creative exploration.

### Mistake 5: Ignoring Token Costs

**Why it happens**: The CoT output looks valuable, so you do not measure how many extra tokens it consumes.

**Symptoms**: A 200-token direct answer becomes a 800-token CoT response. Across thousands of requests, the cost adds up significantly.

**Fix**: Measure the token ratio (CoT tokens / direct tokens) for your task. If it is consistently above 3x with no accuracy improvement, reconsider whether CoT is needed.

```python
def cost_ratio(direct_tokens: int, cot_tokens: int) -> float:
    """Calculate how many times more expensive CoT is."""
    return round(cot_tokens / direct_tokens, 2)


# Example: direct = 50 tokens, CoT = 400 tokens
print(f"Cost multiple: {cost_ratio(50, 400)}x")
```
