# Shared Memory and State — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Shared state is not always the right architecture:

- **Fully sequential pipelines**: If agents never run concurrently and the output of one feeds directly into the next, pass data through function arguments — no shared state needed.
- **Stateless agents**: If each agent is a pure function that does not need external context, shared state adds complexity with no benefit.
- **Security-critical silos**: If agents operate on sensitive data and must never leak information between domains, shared state is a liability, not a feature.
- **Ephemeral agents**: If agents are created per-request and destroyed after, pass state through the request context instead of a persistent store.

## The Pattern to Remember

**Pattern**: Agents share state through either a centralized store (strong consistency), an event log (append-only, replayable), or per-agent snapshots (isolation with write-back).

When to use:
- Multiple agents collaborating on related tasks
- Systems where agents need to see each other's intermediate results
- Long-running multi-agent workflows

When NOT to use:
- Purely sequential pipelines
- Stateless agents
- Security-isolated domains

**Mental model**: Shared state is a coordination primitive, not a database. Design for the consistency model you need, not the one that is easiest to implement.

## Common Mistakes

### Mistake 1: No Write Conflict Detection

**Why it happens**: Two agents write to the same key independently, and the last writer wins silently.

**Symptoms**: Data disappears or reverts to an older version. The agent that wrote first never knows its data was overwritten.

**Fix**: Use compare-and-swap or versioned writes. Each write includes the expected version. If the version does not match, the write fails and the agent must re-read and retry.

### Mistake 2: Event Log That Grows Forever

**Why it happens**: Every event is appended and never compacted.

**Symptoms**: The event log grows to millions of entries. Replay takes minutes. Memory usage grows without bound.

**Fix**: Periodically snapshot the current state and truncate the log up to that snapshot. Store the snapshot as a checkpoint so replay starts from the most recent snapshot, not from the beginning.

### Mistake 3: Stale Snapshot Assumptions

**Why it happens**: An agent takes a snapshot at the start of its task and never refreshes it.

**Symptoms**: The agent makes decisions based on stale data. It overwrites changes that another agent made after the snapshot was taken.

**Fix**: On merge, check whether the agent's snapshot version matches the current store version. If it does not, flag the merge for review or require the agent to re-read and re-work.

### Mistake 4: Shared State as an Implicit Message Bus

**Why it happens**: Agents communicate by writing to and reading from the same state keys, expecting each other to notice.

**Symptoms**: Updates are missed because the target agent is not polling that key. Race conditions abound because read and write timing is unpredictable.

**Fix**: Use an explicit message bus for communication (see [Agent Communication and Coordination](../agent-communication-and-coordination/overview)). Shared state is for data that multiple agents need — not for signaling or task assignment.

### Mistake 5: No Locking on the Read-Then-Write Pattern

**Why it happens**: An agent reads a value, modifies it in code, and writes it back — but the read and write are separate operations.

**Symptoms**: Two agents both read the same counter value of 5, increment it to 6 in memory, and write back 6. The counter should be 7, but it remains 6.

**Fix**: Use compare-and-swap or a transactional increment operation. Never implement read-then-write as separate operations without a lock or CAS guard.
