# Review: Monitoring Production Agents

⏱️ 5 minutes

## Key Takeaways

1. Agent monitoring extends infrastructure monitoring with agent-specific metrics — task success rate, tool error rate, loop detection, session latency, and cost per session — that capture the health of agent behavior, not just server health
2. Define SLIs (what you measure) and SLOs (target thresholds) before building dashboards or alerts; without SLOs, monitoring is just graphs looking for a problem
3. Every alert must have a runbook attached; if you cannot write step-by-step instructions for what to do when an alert fires, the alert is not ready for production
4. Monitoring instruments your agent with Prometheus counters, gauges, and histograms; Prometheus scrapes and stores the metrics; Grafana visualizes them; Alertmanager handles notification routing to Slack and PagerDuty
5. Avoid alert fatigue by basing thresholds on two weeks of production data, using a `for` duration to prevent flapping, and conducting weekly alert reviews to prune noise

## Recap in 60 Seconds

If you only remember three things:

1. **Instrument agent-specific metrics** — success rate, latency, tool errors, loop detection, and cost per session. Infrastructure metrics alone will not catch agent degradation.
2. **SLOs before alerts** — define what healthy looks like first. An alert without a target and a runbook is just noise.
3. **Dashboards should drive action** — top row: SLO health; middle row: SLI breakdowns; bottom row: drill-down details. If a panel does not answer "should I act now?" it belongs on a different dashboard.
