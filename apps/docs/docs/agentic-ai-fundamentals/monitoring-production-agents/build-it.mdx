# Build It: Setting Up Agent Monitoring

⏱️ 25 minutes

## Breaking It Down

### 1. Defining Agent SLIs and SLOs

Start by identifying the four key Service Level Indicators for your agent system. Each SLI measures a specific aspect of agent health:

```python
# sli_definitions.py
from dataclasses import dataclass
from datetime import timedelta

@dataclass
class AgentSLI:
    name: str
    description: str
    measurement_window: timedelta
    warning_threshold: float
    critical_threshold: float

# Define SLIs specific to agent systems
SLIS = {
    "task_success_rate": AgentSLI(
        name="task_success_rate",
        description="Percentage of agent sessions that complete successfully",
        measurement_window=timedelta(hours=1),
        warning_threshold=0.93,  # Below 93% triggers warning
        critical_threshold=0.90,  # Below 90% triggers critical
    ),
    "p95_session_latency": AgentSLI(
        name="p95_session_latency",
        description="95th percentile of session duration in seconds",
        measurement_window=timedelta(hours=1),
        warning_threshold=30.0,   # Over 30 seconds triggers warning
        critical_threshold=60.0,  # Over 60 seconds triggers critical
    ),
    "tool_error_rate": AgentSLI(
        name="tool_error_rate",
        description="Percentage of tool calls that return errors",
        measurement_window=timedelta(hours=1),
        warning_threshold=0.05,   # Above 5% triggers warning
        critical_threshold=0.10,  # Above 10% triggers critical
    ),
    "loop_session_rate": AgentSLI(
        name="loop_session_rate",
        description="Percentage of sessions exceeding max tool call threshold",
        measurement_window=timedelta(hours=1),
        warning_threshold=0.01,   # Above 1% triggers warning
        critical_threshold=0.05,  # Above 5% triggers critical
    ),
}
```

Define SLOs as targets over a 30-day rolling window:

```python
# slo_definitions.py
@dataclass
class AgentSLO:
    sli_name: str
    target: float
    window_days: int
    description: str

SLOS = [
    AgentSLO(
        sli_name="task_success_rate",
        target=0.95,
        window_days=30,
        description="95% of agent sessions succeed over 30 days",
    ),
    AgentSLO(
        sli_name="p95_session_latency",
        target=30.0,
        window_days=30,
        description="p95 latency stays under 30 seconds over 30 days",
    ),
    AgentSLO(
        sli_name="tool_error_rate",
        target=0.02,
        window_days=30,
        description="Tool error rate remains below 2% over 30 days",
    ),
]
```

### 2. Instrumenting the Agent with Prometheus Metrics

Use the Prometheus Python client to expose agent-specific metrics on a `/metrics` endpoint:

```python
# agent_metrics.py
from prometheus_client import Counter, Gauge, Histogram, Summary, start_http_server
import time

# Counters — cumulative totals that only increase
AGENT_SESSIONS_TOTAL = Counter(
    "agent_sessions_total",
    "Total number of agent sessions",
    ["status"],  # success, failure, loop_terminated
)

TOOL_CALLS_TOTAL = Counter(
    "tool_calls_total",
    "Total number of tool calls",
    ["tool_name", "status"],  # success, error
)

TOOL_ERRORS_TOTAL = Counter(
    "tool_errors_total",
    "Total number of failed tool calls",
    ["tool_name", "error_type"],
)

LOOP_DETECTED_TOTAL = Counter(
    "loop_detected_total",
    "Total number of sessions terminated due to loop detection",
    ["reason"],  # max_calls, max_duration, repetitive_pattern
)

# Gauges — values that go up and down
AGENT_SESSIONS_IN_PROGRESS = Gauge(
    "agent_sessions_in_progress",
    "Number of currently active agent sessions",
)

LLM_TOKEN_USAGE = Gauge(
    "llm_token_usage",
    "Current token usage per model call",
    ["model", "token_type"],  # model name, input/output
)

# Histograms — distributions of observed values
SESSION_DURATION_SECONDS = Histogram(
    "session_duration_seconds",
    "Duration of agent sessions in seconds",
    buckets=[1, 5, 10, 20, 30, 60, 120, 300],
)

TOOL_CALL_DURATION_SECONDS = Histogram(
    "tool_call_duration_seconds",
    "Duration of individual tool calls in seconds",
    ["tool_name"],
    buckets=[0.1, 0.5, 1.0, 2.0, 5.0, 10.0, 30.0],
)

TOOL_CALLS_PER_SESSION = Histogram(
    "tool_calls_per_session",
    "Number of tool calls per session",
    buckets=[1, 3, 5, 10, 15, 20, 30, 50],
)

# Summaries — quantile-based measurements
SESSION_COST_USD = Summary(
    "session_cost_usd",
    "Cost of agent session in USD",
)

# Start the metrics HTTP server
def start_metrics_server(port: int = 8000):
    """Start Prometheus metrics endpoint on the given port."""
    start_http_server(port)
    print(f"Prometheus metrics exposed on port {port}")
```

### 3. Integrating Metrics into the Agent Loop

Wrap your agent's observe-think-act loop with metric recording:

```python
# monitored_agent.py
import time
from agent_metrics import (
    AGENT_SESSIONS_TOTAL, AGENT_SESSIONS_IN_PROGRESS,
    SESSION_DURATION_SECONDS, TOOL_CALLS_TOTAL,
    TOOL_ERRORS_TOTAL, LOOP_DETECTED_TOTAL,
    TOOL_CALL_DURATION_SECONDS, TOOL_CALLS_PER_SESSION,
    SESSION_COST_USD, LLM_TOKEN_USAGE,
)

MAX_TOOL_CALLS = 20

async def run_monitored_agent(user_input: str):
    """Run the agent with full Prometheus instrumentation."""
    AGENT_SESSIONS_IN_PROGRESS.inc()
    start_time = time.time()
    tool_call_count = 0
    success = True
    loop_detected = False

    try:
        # Run the agent's observe-think-act loop
        async for event in agent_loop(user_input):
            if event.type == "tool_call":
                tool_call_count += 1
                tool_start = time.time()
                try:
                    result = await execute_tool(event.tool_name, event.arguments)
                    TOOL_CALLS_TOTAL.labels(
                        tool_name=event.tool_name, status="success"
                    ).inc()
                except Exception as e:
                    TOOL_CALLS_TOTAL.labels(
                        tool_name=event.tool_name, status="error"
                    ).inc()
                    TOOL_ERRORS_TOTAL.labels(
                        tool_name=event.tool_name,
                        error_type=type(e).__name__,
                    ).inc()
                    success = False
                    raise
                finally:
                    duration = time.time() - tool_start
                    TOOL_CALL_DURATION_SECONDS.labels(
                        tool_name=event.tool_name
                    ).observe(duration)

            # Loop detection — terminate if too many calls
            if tool_call_count >= MAX_TOOL_CALLS:
                LOOP_DETECTED_TOTAL.labels(reason="max_calls").inc()
                loop_detected = True
                break

        duration = time.time() - start_time
        SESSION_DURATION_SECONDS.observe(duration)
        TOOL_CALLS_PER_SESSION.observe(tool_call_count)

        if loop_detected:
            AGENT_SESSIONS_TOTAL.labels(status="loop_terminated").inc()
        elif success:
            AGENT_SESSIONS_TOTAL.labels(status="success").inc()
        else:
            AGENT_SESSIONS_TOTAL.labels(status="failure").inc()

    except Exception:
        AGENT_SESSIONS_TOTAL.labels(status="failure").inc()
        raise
    finally:
        AGENT_SESSIONS_IN_PROGRESS.dec()
```

### 5. Setting Up Alerting with Alertmanager

Define Prometheus alerting rules that fire when SLIs breach thresholds:

```yaml
# prometheus-alerts.yml
groups:
  - name: agent_alerts
    rules:
      - alert: HighTaskFailureRate
        expr: |
          rate(agent_sessions_total{status="failure"}[15m])
          / rate(agent_sessions_total[15m]) > 0.10
        for: 5m
        labels:
          severity: critical
          team: agent-ops
        annotations:
          summary: "Agent task failure rate exceeds 10%"
          description: >
            The agent failure rate is {{ $value | humanizePercentage }}
            over the last 15 minutes. Investigate tool availability,
            model degradation, or recent prompt changes.

      - alert: HighLatency
        expr: |
          histogram_quantile(0.95,
            rate(session_duration_seconds_bucket[15m])
          ) > 30
        for: 10m
        labels:
          severity: warning
          team: agent-ops
        annotations:
          summary: "p95 session latency exceeds 30 seconds"
          description: >
            p95 session latency is {{ $value | humanizeDuration }}.
            Check LLM provider status and tool execution times.

      - alert: ToolErrorSpike
        expr: |
          rate(tool_errors_total[5m]) > 0.5
        for: 2m
        labels:
          severity: critical
          team: agent-ops
        annotations:
          summary: "Tool errors spiking on {{ $labels.tool_name }}"
          description: >
            The tool {{ $labels.tool_name }} is experiencing
            {{ $value | humanize }} errors per second.
            Verify API availability and response format.

      - alert: LoopDetectionActive
        expr: |
          rate(loop_detected_total[5m]) > 0.1
        for: 2m
        labels:
          severity: warning
          team: agent-ops
        annotations:
          summary: "Multiple sessions terminated for looping"
          description: >
            Loop detection has triggered
            {{ $value | humanize }} times per second.
            Investigate recent prompt or tool definition changes.

      - alert: NoSessionsWarning
        expr: |
          rate(agent_sessions_total[30m]) == 0
        for: 10m
        labels:
          severity: warning
          team: agent-ops
        annotations:
          summary: "No agent sessions in the last 30 minutes"
          description: >
            The agent has received zero sessions in 30 minutes.
            This could indicate a routing or upstream issue.
```

Configure Alertmanager to route alerts to Slack and PagerDuty:

```yaml
# alertmanager.yml
route:
  receiver: slack-notifications
  routes:
    - match:
        severity: critical
      receiver: pagerduty-critical
      repeat_interval: 15m

receivers:
  - name: slack-notifications
    slack_configs:
      - api_url: "https://hooks.slack.com/services/YOUR/WEBHOOK/URL"
        channel: "#agent-alerts"
        title: '{{ template "slack.title" . }}'
        text: '{{ template "slack.text" . }}'
        send_resolved: true

  - name: pagerduty-critical
    pagerduty_configs:
      - routing_key: "YOUR_PAGERDUTY_KEY"
        severity: critical
        description: '{{ .GroupLabels.alertname }}'

templates:
  - "/etc/alertmanager/templates/*.tmpl"
```

### 6. Creating Incident Runbooks

Write runbooks for the five most common agent failure modes. Each runbook follows a standard template:

<details>
<summary>Runbook: High Task Failure Rate</summary>

**Symptoms**
- Alert: HighTaskFailureRate fires
- Users report incorrect answers or timeouts
- Success rate drops below 90%

**Possible Causes**
1. LLM provider outage or degraded service
2. Tool API change (response format, removed endpoint)
3. Recent prompt or system prompt change
4. Model version change (if using a different model)
5. Upstream data source unavailable

**Investigation Steps**
1. Check provider status page for ongoing incidents.
2. Review the last 3 deploys — did a prompt or tool definition change?
3. Query recent failed sessions for common error patterns:
   ```
   agent_sessions_total{status="failure"} offset 1h
   ```
4. Check tool error rate per tool to isolate the failing component.
5. Retrieve a sample of failed session traces from the observability backend.

**Mitigation**
- Rollback to the previous prompt or model version.
- Disable the failing tool temporarily.
- Redirect traffic to a fallback agent if available.

**Resolution**
- Document the root cause.
- Add a regression test that catches this failure mode.
- Update the monitoring threshold if the failure was acceptable.

**Postmortem**
- File an incident report within 24 hours.
- Update this runbook with new findings.
</details>

## How It Works Internally

The agent monitoring data pipeline has four stages:

1. **Instrumentation** — Your agent code emits Prometheus metrics using the Python client library. Each metric is an HTTP endpoint at `/metrics` served on a dedicated port. The metrics are lightweight numeric snapshots — counters that only increase, gauges that reflect current state, and histograms that bucket observed values.

2. **Scraping** — The Prometheus server polls each agent instance's `/metrics` endpoint on a scrape interval (typically 15 seconds). It stores the raw time series data in its local TSDB (time-series database). Prometheus handles downsampling, retention policies, and query evaluation.

3. **Visualization** — Grafana connects to the Prometheus data source and runs PromQL queries to build dashboards. The dashboard panels refresh automatically (every 30 seconds by default) and apply thresholds to color panels green, yellow, or red based on SLO boundaries.

4. **Alerting** — Prometheus evaluates alerting rules on every scrape cycle. When a rule's expression evaluates to true for the configured `for` duration (e.g., 5 minutes), Prometheus fires an alert to Alertmanager. Alertmanager handles deduplication, silencing, and routing — it sends notifications to Slack for warnings and PagerDuty for critical alerts. On-call engineers receive the alert with a direct link to the relevant dashboard and runbook.

## Performance Considerations

Monitoring overhead is typically small but requires attention. Each Prometheus metric push is a single HTTP request with a few hundred bytes of payload. For an agent handling 10,000 sessions per day, the metrics endpoint generates roughly 1-2 MB of data per hour — negligible compared to the LLM token payload.

The main concern is **metrics cardinality**. Labels like `tool_name`, `model`, and `error_type` create unique time series for each combination. If your agent has 50 tools, 3 models, and 10 error types, a single histogram can create 1500 time series (50 x 3 x 10). At scale, this adds up. Keep labels to a minimum — only include dimensions you will query on. Use the `_total` suffix convention for counters and keep histogram buckets reasonable (under 20 buckets per histogram).

**Alert fatigue** is the second concern. Every alert you add costs engineer attention. If an alert fires but nobody acts on it for more than two weeks, the alert is noise — remove it or raise the threshold. Follow the rule: no alert without a runbook. If you cannot write a runbook for what to do when an alert fires, you are not ready to add that alert.
