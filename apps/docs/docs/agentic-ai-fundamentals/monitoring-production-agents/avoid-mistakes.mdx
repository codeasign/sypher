## When NOT to Use This

A full production monitoring stack with Prometheus, Grafana, and PagerDuty integration is not the right tool for every situation. Skip it when:

- **You are in pre-production or prototyping** â€” setting up alerting before you know what normal looks like produces noise. Run evaluation benchmarks manually and watch metrics casually until the agent behavior stabilizes.

- **Your agent handles fewer than 100 sessions per day** â€” at low volumes, metrics are too sparse to produce meaningful percentiles. A single slow session can double your p95. Wait until you have enough traffic that statistical noise does not drown out signal.

- **You are running experimental agents** â€” one-off agents for internal experiments or A/B tests do not need runbooks and PagerDuty escalation. Log the results and tear down the infrastructure when the experiment ends.

- **You have not defined what success looks like** â€” monitoring without SLOs is just graphs. If you cannot articulate what "healthy" means for your agent, you will not know what to alert on. Define your SLIs and SLOs first, then instrument.

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | SLIs define what you measure; SLOs define what good looks like; alerting tells you when you breach it |
| **When to use** | Production agents with 100+ sessions/day, multiple deployment targets, or customer-facing reliability requirements |
| **When NOT to use** | Pre-production, low-traffic agents, experimental agents, or before SLOs are defined |
| **Mental model** | A car dashboard â€” gauges show current state (speed, fuel, RPM), warning lights alert when things go wrong, and the owner's manual (runbook) tells you what to do |

## Common Mistakes

### Mistake 1: Measuring Only the Happy Path

**Mistake:** Tracking only successful sessions and ignoring errors, loops, and edge cases in your metrics.

**Why it happens:** Success metrics look good in dashboards and it feels productive to report high success rates. Error metrics are uncomfortable to present.

**Symptoms:** The dashboard shows 98% success rate while users quietly abandon sessions after the third timeout. You are measuring the wrong denominator â€” sessions that completed are counted; sessions that timed out and disconnected are not.

**Fix:** Track all sessions that enter the system, not just those that complete. A session that starts but never finishes is a failure. Use `AGENT_SESSIONS_TOTAL` with a `status` label covering `success`, `failure`, `loop_terminated`, and `timeout`.

### Mistake 2: Alert Fatigue from Overly Sensitive Thresholds

**Mistake:** Setting alert thresholds so tight that they fire on normal variance, or setting them so loose that they never fire.

**Why it happens:** Engineers set thresholds based on a few hours of data. A single slow LLM call at 2 AM causes a false positive. The next day the threshold is widened. Repeat until alerts are meaningless.

**Symptoms:** Alerts fire every night at the same time (scheduled maintenance, cron jobs). On-call engineers ignore the alert channel. The alert dashboard shows 200 "firing" alerts that nobody looks at.

**Fix:** Base thresholds on at least two weeks of production data. Add a `for` duration to alerts (fire only if the condition holds for 5+ minutes). Conduct a weekly alert review: any alert that fired without action for two weeks gets removed or adjusted.

### Mistake 3: Ignoring Agent-Specific Metrics

**Mistake:** Monitoring only infrastructure metrics (CPU, memory, HTTP 200/500) and missing agent-specific signals like loop count, tool retries, and token usage.

**Why it happens:** Infrastructure monitoring is well understood with mature tooling. Agent metrics are new and require custom instrumentation.

**Symptoms:** The server is up and healthy (CPU 30%, memory fine, HTTP 200), but the agent is stuck in a loop consuming tokens at $0.10 per second. The infrastructure dashboard shows green while the agent burns money.

**Fix:** Agent-specific metrics are non-negotiable. Track `loop_detected_total`, `tool_calls_per_session`, `llm_token_usage`, and `session_cost_usd` alongside infrastructure metrics. If you have to choose, instrument the agent behavior first and infrastructure second.

### Mistake 4: No Runbooks for On-Call

**Mistake:** Paging an engineer when an alert fires without providing any guidance on what to do.

**Why it happens:** Alerting is set up by the infrastructure team. Runbooks are seen as documentation that can be written later. Later never comes.

**Symptoms:** The on-call engineer receives a PagerDuty notification at 3 AM for "HighTaskFailureRate." They check the dashboard, see the failure rate is elevated, and have no idea what to do next. They restart the server. The alert fires again 10 minutes later.

**Fix:** Every alert must have an associated runbook before it is deployed. The runbook covers symptoms, possible causes, investigation steps, and mitigation actions. Store runbooks in a shared wiki or alongside the alert configuration in version control. Include a direct link to the runbook in the alert notification.

### Mistake 5: Dashboards Without Clear Action Items

**Mistake:** Building dashboards with 30 panels of every possible metric, but no guidance on what to look at first or what constitutes a problem.

**Why it happens:** More data feels better. Engineers add every metric they can think of because they do not want to miss something.

**Symptoms:** The dashboard is overwhelming. Each panel shows different data at different time scales. There is no logical flow from left to right or top to bottom. New team members cannot tell whether the agent is healthy or not by glancing at the dashboard.

**Fix:** Design dashboards with a clear hierarchy:
- **Top row:** SLO health indicators (green/yellow/red for each SLO)
- **Middle row:** Per-SLI breakdowns (task success rate, latency, tool errors)
- **Bottom row:** Drill-down details (per-tool metrics, per-model metrics, recent anomalies)

If a panel cannot answer the question "should I take action right now?" it belongs in a separate deep-dive dashboard, not the main health dashboard.
