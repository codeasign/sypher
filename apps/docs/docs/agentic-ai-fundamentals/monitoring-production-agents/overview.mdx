# Monitoring Production Agents

- Create incident response runbooks tailored to common agent failure modes

## Why This Matters

Agents are non-deterministic systems. Unlike a traditional web service where a 500 error is immediate and obvious, agents degrade silently over hours or days. The model's output drifts. A tool's API changes and the agent starts getting empty results. The agent enters an infinite loop on a particular query pattern. These failures do not crash the system — they produce subtly wrong answers, slower responses, or escalating costs. Without monitoring, you discover them when users complain, and by then the damage has already compounded.

Production agent monitoring is not optional. At scale, an untracked tool failure that raises error rates from 2% to 10% affects thousands of sessions before a human notices. A model upgrade that increases latency by 3 seconds per session adds minutes of wait time across your user base. A single agent stuck in a loop can burn hundreds of dollars in token costs in an hour. Monitoring catches these regressions at the metric level, before they become customer-facing problems. Teams that invest in agent-specific monitoring from day one detect and resolve incidents 4-6 times faster than teams that rely on ad-hoc debugging.

## The Problem

Your agent passes all your evaluation tests in staging. You deploy it to production. In the first week, everything looks fine — success rate is 95%, latency is acceptable, costs are within budget. By week three, success rate has quietly dropped to 82%. Why? A third-party search API changed its response format, and the agent is now misparsing results. The tool itself returns 200 OK — no error — so your existing infrastructure monitoring shows nothing wrong. The agent's output is just slightly worse, and users are slowly losing trust.

Without monitoring, you have no way to detect this. You are relying on users to tell you something is wrong, but they rarely do — they just stop using the agent. You need agent-specific metrics that measure not just whether the system is up, but whether the agent is producing good outcomes. You need alerts that fire when the agent starts looping, when tool error rate crosses a threshold, when success rate trends downward, and when session latency drifts outside its expected range.

## The Concept

Agent monitoring extends traditional infrastructure monitoring with metrics that capture agent-specific behavior patterns. The key components are:

**Service Level Indicators (SLIs)** for agents measure what matters to users:
- **Task success rate** — percentage of sessions that reach a successful conclusion
- **Session latency** — time from user input to final response (p50, p95, p99)
- **Tool error rate** — percentage of tool calls that fail with an error
- **Loop detection** — sessions exceeding a maximum tool call count
- **Cost per session** — total token and API cost for a single session

**Service Level Objectives (SLOs)** define the targets:
- Task success rate greater than 95% over a 30-day rolling window
- p95 session latency under 30 seconds
- Tool error rate below 2%
- Fewer than 0.1% of sessions exceeding the loop limit

**Alerting rules** evaluate metrics against thresholds and trigger notifications:
- Warning alerts fire when metrics approach the SLO boundary
- Critical alerts fire when metrics exceed the SLO
- Anomaly detection catches gradual degradations before they cross thresholds

**Incident runbooks** provide step-by-step guidance for common failures:
- Tool API changes causing silent failures
- Model drift causing quality degradation
- Sudden latency spikes from LLM or tool bottlenecks
- Agent loop detection and graceful termination
