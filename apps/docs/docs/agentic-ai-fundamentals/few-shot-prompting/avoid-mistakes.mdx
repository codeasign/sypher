# Few-Shot Prompting — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Few-shot prompting is effective but not always the best approach:

- **Tasks with clear, unambiguous definitions**: If you can describe the task precisely in one sentence and the model executes it correctly, examples add unnecessary token cost
- **Highly structured output via tool definitions**: When using the Anthropic tool use API with a defined schema, the schema itself constrains the output — few-shot examples on top are redundant
- **Tasks with very long inputs**: Each example adds tokens. If the task input itself is long (multiple pages of text), adding examples may push you past your budget
- **Rapidly changing labels or categories**: If your categories change weekly, maintaining examples for each category is expensive. Zero-shot with clear definitions is more maintainable

## The Pattern to Remember

**Pattern**: Show, do not just tell. Use 3-5 diverse, high-quality examples that cover the range of expected inputs.

When to use:
- Subtle category boundaries that are hard to describe in words
- Specific output formats that must match exactly
- Tasks where the model consistently gets 80% right but needs guidance on the remaining 20%

When NOT to use:
- Simple tasks where instructions alone work perfectly
- Token-constrained contexts (every example costs tokens)

**Mental model**: Examples are training data for in-context learning. Each example is a labeled data point. Five good examples are worth a hundred words of instructions.

## Common Mistakes

### Mistake 1: Using Low-Quality or Incorrect Examples

**Why it happens**: You write examples quickly without verifying them. You assume a correct-looking example is actually correct.

**Symptoms**: The model reproduces the mistakes in your examples. A mislabeled example teaches the wrong classification pattern.

**Fix**: Validate every example before using it. Run each example through the model and confirm the label is correct. One bad example can corrupt the entire pattern.

### Mistake 2: Using Examples That Are Too Similar

**Why it happens**: You pick the first examples that come to mind, which are usually the most common patterns.

**Symptoms**: The model handles common cases well but fails on edge cases. It has never seen an example of the boundary cases.

**Fix**: Include examples that cover the full range of inputs. Specifically include edge cases you have observed failing in zero-shot mode.

### Mistake 3: Using Too Many Examples

**Why it happens**: More data is better in traditional ML, so the instinct is to include as many examples as possible.

**Symptoms**: The prompt is 2000+ tokens of examples. The model starts paying less attention to later examples. Costs are unnecessarily high.

**Fix**: Start with 3 examples. Add more only if the model consistently fails on specific patterns. Stop at the point where additional examples stop improving results.

### Mistake 4: Ignoring Example Order

**Why it happens**: You add examples as you think of them, not strategically.

**Symptoms**: The model is biased toward the last example's category. If the last example is BUG, the model over-classifies ambiguous inputs as BUG.

**Fix**: Put your most representative example last (the one that best defines the category boundary). Alternate categories in the ordering. Test both orderings and pick the one that performs better.

### Mistake 5: Not Matching Example Format to the Target Format

**Why it happens**: You write examples as natural text, but the production system expects structured output.

**Symptoms**: The model produces examples in the format you showed, but the format is not cleanly parseable by your downstream code.

**Fix**: Write examples in the exact format you want the model to produce. If you want JSON output, write the examples as JSON. If you want a single label, write examples as single labels.
