# Transports — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Each transport has scenarios where it is the wrong choice:

- **Stdio for remote servers**: If the server runs on a different machine, stdio cannot work — it requires a child process. Use SSE.
- **SSE for local-only servers**: If every client runs on the same machine, SSE adds unnecessary HTTP overhead. Use stdio.
- **Stdio for long-running servers**: A stdio server process lives only as long as its parent client. If you need a persistent server, use SSE.
- **SSE for high-frequency calls**: If your tool makes hundreds of calls per second, stdio's sub-millisecond overhead is far better than SSE's HTTP round trip.

## The Pattern to Remember

**Pattern**: Stdio for local, single-client, low-latency deployments. SSE for remote, multi-client, network-available deployments. Same server logic, different I/O layer.

When to use stdio:
- Local file system tools
- Same-machine database tools
- Development and testing

When to use SSE:
- Team-shared services
- Cloud-hosted tools
- Multiple concurrent clients

**Mental model**: Stdio is a function call; SSE is an API call. Use the same business logic for both.

## Common Mistakes

### Mistake 1: Server Process Leaks

**Why it happens**: The client spawns a stdio server but crashes without terminating it.

**Symptoms**: Orphaned server processes accumulate on the machine, consuming memory and file handles. Eventually, the OS runs out of resources.

**Fix**: Use context managers or `atexit` handlers to ensure the server process is terminated when the client exits. Monitor for orphaned processes and implement a timeout to kill servers that outlive their clients.

### Mistake 2: Mixing stdout and Logging on Stdio

**Why it happens**: The server adds debug `print()` calls that write to stdout, which is the MCP communication channel.

**Symptoms**: The client receives malformed JSON because log messages are interleaved with JSON-RPC responses. The client's parser fails.

**Fix**: Always direct log output to stderr, never stdout. If you use a logging library, configure it to write to stderr. For stdio transport, stdout is a reserved communication channel.

### Mistake 3: SSE Without Keepalive

**Why it happens**: The SSE connection is established and messages are sent, but no keepalive mechanism is in place.

**Symptoms**: Idle connections are dropped by proxies or load balancers after 30-60 seconds. The client thinks the server is still connected but receives no events.

**Fix**: Send SSE keepalive comments (`: keepalive\n\n`) at regular intervals (every 15-30 seconds) to keep the connection alive through proxies and load balancers.

### Mistake 4: Blocking the SSE Stream

**Why it happens**: The server processes a long-running tool call synchronously and blocks writing to the SSE stream during that time.

**Symptoms**: All clients stop receiving events while one tool call is executing. The SSE stream becomes a bottleneck.

**Fix**: Process tool calls asynchronously. When a tool call arrives via POST, dispatch it on a separate thread and push the response to the SSE stream only when it is ready.

### Mistake 5: Hard-Coded Transport in the Client

**Why it happens**: The client is written to work with one transport and never abstracted.

**Symptoms**: Switching from stdio (development) to SSE (production) requires rewriting the client's communication layer.

**Fix**: Abstract the transport behind a common interface. The client should call `transport.send_message()` regardless of whether the transport is stdio or SSE.
