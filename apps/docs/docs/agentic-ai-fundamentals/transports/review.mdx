# Transports — Review

*Reading time: 3 minutes*

## Key Takeaways

- MCP defines two transports: stdio (child process stdin/stdout) and SSE (HTTP Server-Sent Events)
- stdio is for local, single-client, low-latency deployments; SSE is for remote, multi-client, network-available deployments
- The same server business logic works with either transport — only the I/O layer changes
- On stdio transport, always write log output to stderr, never stdout — stdout is the MCP communication channel
- SSE connections require keepalive messages to prevent proxies and load balancers from dropping idle connections

## Recap in 60 Seconds

1. Stdio = local, low-latency, one-to-one (child process pipes); SSE = remote, higher-latency, multi-client (HTTP events) — choose based on deployment topology
2. Abstract the transport behind a common interface so you can switch between stdio and SSE without changing server logic
3. On stdio, stdout is JSON-RPC only — all logging goes to stderr; on SSE, use keepalive pings every 15-30 seconds
