# Avoid Mistakes: Tool Evaluation Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Tool-level evaluation is detailed and time-consuming. Do not invest in it when:

- **Your agent has fewer than 5 tools** — with only a few tools, you can evaluate quality manually
- **You have not established baseline trajectory metrics** — fix overall agent evaluation before diving into per-tool analysis
- **Tool definitions change every week** — wait until your tool set stabilizes before measuring
- **You have no invocation logs** — tool evaluation requires data; install logging first

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Per-tool invocation, error, and parameter accuracy tracking |
| **When to use** | 5+ tools, stable tool set, established logging infrastructure |
| **When NOT to use** | Few tools, changing definitions, no invocation data |
| **Mental model** | A monitoring dashboard for each tool's usage and error rate |

## Common Mistakes

### Mistake 1: Survivorship Bias in Error Rates

**Mistake:** Computing error rates only from successful tool invocations, ignoring calls the agent never made because it chose the wrong tool.

**Why it happens:** Error rates are easy to compute from logs. Missed opportunities (the agent should have called this tool but did not) are invisible in logs.

**Symptoms:** A tool has a 0% error rate, but it is also never called. The evaluation says "no issues" while the tool is completely unused.

**Fix:** Include a "missed opportunity" metric. For tasks where the expected tool sequence includes a tool, check whether the agent actually called it:

```python
def missed_opportunities(invocations, expected_tool_map):
    """Find tools the agent should have called but did not."""
    missed = Counter()
    for task_id, expected_tools in expected_tool_map.items():
        actual_tools = {inv.tool_name for inv in invocations if inv.task_id == task_id}
        for tool in expected_tools:
            if tool not in actual_tools:
                missed[tool] += 1
    return missed
```

### Mistake 2: Confusing Agent Error with Tool Error

**Mistake:** Attributing every tool error to a tool design problem when the agent is passing nonsensical parameters.

**Why it happens:** The tool returns an error, so the evaluation blames the tool.

**Symptoms:** You improve tool descriptions, but the error rate stays the same because the agent is still generating bad arguments from a different root cause.

**Fix:** Distinguish between "tool validation error" (schema mismatch, missing required field) and "tool execution error" (API failure, timeout, data not found). Validation errors suggest agent-side issues; execution errors suggest tool-side issues.

### Mistake 3: Only Measuring Successful Calls

**Mistake:** Computing average duration and parameter accuracy only from calls that did not error.

**Why it happens:** Error calls often have missing or malformed data that is harder to parse.

**Symptoms:** Your "average duration" metric looks great because it excludes the multi-minute timeouts that happen before errors.

**Fix:** Always include error calls in aggregate metrics. Report `p50` and `p95` latency instead of average to capture the tail.

### Mistake 4: Treating All Error Types the Same

**Mistake:** A `400 Bad Request` and a `500 Internal Server Error` are both counted as 1 error.

**Why it happens:** Logging captures the error flag but not the error category.

**Symptoms:** You cannot tell whether tools are failing because the agent passes wrong parameters (400) or because the backend is down (500).

**Fix:** Categorize errors into validation errors (agent-side) and execution errors (server-side) in your logging:

```python
def categorize_error(error_str: str) -> str:
    if any(code in error_str for code in ["400", "422", "404"]):
        return "validation_error"
    elif any(code in error_str for code in ["500", "502", "503"]):
        return "server_error"
    elif "timeout" in error_str.lower():
        return "timeout"
    else:
        return "unknown"
```

### Mistake 5: Optimizing for Invocation Share

**Mistake:** Setting a target invocation share for each tool and optimizing to meet it.

**Why it happens:** It feels good to see every tool getting "balanced" usage.

**Symptoms:** Some tools are genuinely needed less often than others. Forcing their usage leads to unnecessary tool calls that waste tokens and time.

**Fix:** Measure invocation rates against expected rates from your eval tasks, not against each other. A tool with 2% invocation share is fine if it should only be called in rare scenarios.
