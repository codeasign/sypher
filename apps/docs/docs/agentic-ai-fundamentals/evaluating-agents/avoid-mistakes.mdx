# Avoid Mistakes: Agent Evaluation Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A formal evaluation framework is not always the right tool. Do not use it when:

- **You are in early prototyping** — measuring success rate on 5 tasks tells you less than manual exploration of 5 different scenarios
- **You have no production data** — evaluation datasets should reflect real user behavior; synthetic tasks that do not match actual usage give misleading signal
- **Your agent changes every day** — evaluation is most useful for regression detection; if the agent changes daily, you are measuring churn, not progress
- **You evaluate less than 20 tasks** — small sample sizes produce noisy metrics; a 5-task eval passing 4 (80%) is meaningless with a confidence interval of +/- 35 percentage points

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Multi-dimensional evaluation across step, trajectory, and outcome levels |
| **When to use** | Testing agent quality, regression detection, release gating |
| **When NOT to use** | Early prototyping, unstable agents, tiny task sets |
| **Mental model** | An automated test suite with gradable rubrics, not a binary pass/fail |

## Common Mistakes

### Mistake 1: Only Measuring Final Outcome

**Mistake:** Scoring the task as pass/fail based only on whether the final answer is correct.

**Why it happens:** It is the simplest metric to compute. The agent "succeeded" — good enough, right?

**Symptoms:** The agent succeeds on outcomes but wastes 15 tool calls, hallucinates data in intermediate steps, or violates safety policies that do not affect the final answer.

**Fix:** Add step-level and trajectory-level metrics. An agent that gives the right answer but takes a dangerous path is still a failure.

### Mistake 2: Overfitting to the Evaluation Set

**Mistake:** Tuning prompts, tool descriptions, and system prompts until the eval set passes, then shipping.

**Why it happens:** The eval set becomes the defacto specification. Every change is measured against it.

**Symptoms:** The agent performs well on eval tasks but fails on real user requests that differ in subtle ways.

**Fix:** Rotate 20% of your eval tasks every month to prevent overfitting. Use a separate held-out test set that you never tune against.

### Mistake 3: Ignoring Edge Cases in the Dataset

**Mistake:** Writing eval tasks that only cover happy-path scenarios.

**Why it happens:** It is natural to write tasks for "typical" usage — refund a valid order, search for an existing user.

**Symptoms:** The agent works perfectly in testing but crashes on the first out-of-stock item, missing user, or ambiguous query it encounters in production.

**Fix:** Include at least 30% edge case and failure scenario tasks in your eval set — missing data, ambiguous queries, expired sessions, invalid inputs.

### Mistake 4: Using the Same Judge as the Agent

**Mistake:** Using the same model for both the agent and the LLM judge.

**Why it happens:** You already have access to one model. Why pay for another?

**Symptoms:** The judge shares the agent's blind spots and biases, rating incorrect answers as correct because both models make the same reasoning error.

**Fix:** Use a different model for the judge. If the agent uses Claude Sonnet, use Claude Opus or GPT-4 for evaluation. The slight cost increase is worth the independent perspective.

### Mistake 5: Treating Metrics as Independent

**Mistake:** Optimizing task success rate without tracking latency, cost, or safety.

**Why it happens:** Engineering teams are incentivized to improve the headline metric.

**Symptoms:** Task success rate goes up but latency doubles, cost triples, and safety violations increase — the agent learned to brute-force success with excessive tool calls.

**Fix:** Define a composite score that balances multiple metrics, and gate releases on the composite, not any single metric.
