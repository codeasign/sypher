# Specialist Agents

*Reading time: 6 minutes*

**Difficulty:** Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Design a specialist agent optimized for a single domain or task type
- Configure agent prompts, tools, and model parameters per speciality
- Compose specialists into a multi-agent system without role overlap
- Evaluate when a generalist agent should be split into specialists

## Prerequisites

You should already understand:

- How single agents use tools and memory (covered in [Anatomy of an Agent](../anatomy-of-an-agent/overview))
- How orchestrators delegate work to sub-agents (covered in [Orchestrator and Supervisor Patterns](../orchestrator-and-supervisor-patterns/overview))
- The concept of agent tool definitions and dispatch (covered in [Agent Tools and Tool Calling](../agent-tools-and-tool-calling/overview))

## Why This Matters

A generalist agent that handles every task — research, coding, data analysis, customer support — must carry every tool, every system prompt instruction, and every memory pattern simultaneously. This creates context-window pressure, increases hallucination risk, and makes the agent worse at every task because it is optimized for none.

In production, every successful multi-agent deployment uses specialist agents. A coding agent has access to the codebase and build tools but does not handle customer billing. A research agent has web search and document retrieval but cannot modify files. Each agent's prompt, tool set, and model configuration is tuned for exactly one job.

This separation is not just about organization — it is about safety. A specialist agent with a narrow tool scope cannot accidentally perform actions outside its domain. If your data-analysis agent only has read access to the database, it cannot delete tables.

## The Problem

You build a single agent that can do everything: search the web, write code, query the database, send emails, and update the CRM. The agent works in demos but fails in production. It calls the email tool when you asked it to query the database. It hallucinates CRM fields because the tool descriptions are competing for attention in the prompt. The agent's system prompt is 4000 tokens covering all the rules for every domain.

A single agent with N tools must reason about N tool descriptions on every turn. As N grows, the model's accuracy on individual tool choices decreases. Studies and empirical evidence from production deployments show that tool-call accuracy degrades noticeably beyond 30-40 tools, and the model spends more time deciding which tool to use than actually solving the problem.

## The Concept

A specialist agent is an agent whose system prompt, tool set, model configuration, and memory are scoped to a single domain or task type. The specialist does not know about tools outside its domain — they simply do not exist in its context.

Specialist agents are designed around the principle of bounded competence: each agent is an expert in one area and knows nothing about others. This is not a limitation — it is the feature. By removing irrelevant tools and knowledge, you reduce the decision space the model must consider on each turn, which improves accuracy, reduces latency, and simplifies debugging.

When a task spans multiple domains, an orchestrator routes sub-tasks to the appropriate specialists. Each specialist returns its result to the orchestrator, who assembles the final answer. The specialists never need to know about each other's existence.
