# Specialist Agents — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Specialist agents are not the default architecture. Consider alternatives when:

- **The domain boundaries are unclear**: If you cannot crisply define what one agent does versus another, the routing logic will be fragile and tasks will fall through the cracks.
- **The total tool count is small (under 15)**: A single generalist agent with 15 well-described tools can handle most domains without needing multiple agents.
- **Tasks require deep cross-domain context**: Some tasks genuinely span domains and benefit from a single agent seeing the full picture rather than having context chopped up by routing.
- **Development velocity matters more than safety**: Building and maintaining N specialist agents costs more than one generalist. Start with a generalist and split only when you have evidence of failure.

## The Pattern to Remember

**Pattern**: Each specialist agent has a scoped prompt, a narrow tool set, and no awareness of other domains. An orchestrator routes tasks by domain.

When to use:
- Domains with distinct tool sets (e.g., data analysis vs. file management)
- Systems where safety requires limiting each agent's blast radius
- Large tool catalogs (more than 30 total) that would overwhelm a single agent

When NOT to use:
- Small tool catalogs where one agent suffices
- Tightly coupled tasks requiring shared state
- Early prototypes — split later, not sooner

**Mental model**: A specialist agent is a microservice for AI. It has one job, one tool set, one prompt, and no knowledge of other services.

## Common Mistakes

### Mistake 1: Overlapping Tool Definitions

**Why it happens**: You add the same tool to multiple specialists because "they both might need it."

**Symptoms**: Two specialists produce different results for the same question because each has a slightly different version of the tool. The orchestrator routes inconsistently.

**Fix**: Every tool belongs to exactly one specialist. If two specialists need similar functionality, create two tools with distinct names and scopes, or introduce a shared service that both specialists call through a single tool.

### Mistake 2: Router Too Simple

**Why it happens**: A keyword-based router seems sufficient and the first few tests pass.

**Symptoms**: As the vocabulary grows, the router misroutes more tasks. A query about "login errors" goes to the account agent instead of the tech agent because "login" matches the wrong domain.

**Fix**: Use an LLM call for routing in production. The router itself can be a small, fast model that classifies the task domain before delegating to the specialist.

### Mistake 3: Specialist with No Tools

**Why it happens**: You create a specialist that only needs to reason or summarize, so you give it zero tools, expecting it to use its training data.

**Symptoms**: The specialist produces confident-sounding but inaccurate answers. Without tools to ground its reasoning, it relies entirely on parametric knowledge.

**Fix**: Every specialist should have at least one grounding tool — even a simple data lookup or retrieval-augmented generation (RAG) tool. This prevents hallucination.

### Mistake 4: Context Starvation at Domain Boundaries

**Why it happens**: You split a multi-step workflow across specialists, but the downstream specialist does not receive enough context from the upstream one.

**Symptoms**: The report writer produces generic output because it received only summaries from the data analyst, not the underlying data it needs.

**Fix**: Ensure that when the orchestrator routes between specialists, it passes sufficient context — either the full output of the first specialist or a structured data package. Do not truncate context to save tokens at the expense of quality.

### Mistake 5: Manually Testing Every Domain Combination

**Why it happens**: You write unit tests for each specialist in isolation but never test the routing pipeline end-to-end.

**Symptoms**: Specialists work perfectly in isolation but the system fails in production because the router misclassifies tasks or drops context between steps.

**Fix**: Add integration tests that run full multi-step pipelines spanning multiple specialists. Test the routing logic with edge-case inputs — typos, ambiguous queries, multi-intent requests.
