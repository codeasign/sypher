# Tokens and Context Windows — Build It

*Reading time: 10 minutes*

## Breaking It Down

### 1. Counting Tokens with the Anthropic Tokenizer

Anthropic provides a tokenizer library that matches the model's exact tokenization rules. Install it separately:

```bash
pip install anthropic anthropic-tokenizer
```

Count tokens in any text:

```python
from anthropic import Anthropic, HUMAN_PROMPT, AI_PROMPT

client = Anthropic()

text = "Large language models process text as tokens, not as characters or words."

# Use the API's usage field to get exact token counts
response = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=10,
    messages=[{"role": "user", "content": text}]
)

print(f"Input tokens: {response.usage.input_tokens}")
```

For approximate counting without an API call, use the rule of thumb: multiply word count by 1.3.

```python
def approximate_tokens(text: str) -> int:
    """Estimate token count using the 1.3 multiplier rule."""
    word_count = len(text.split())
    return int(word_count * 1.3)

sample = "This is a sample sentence with several words in it."
print(f"Words: {len(sample.split())}")
print(f"Approx tokens: {approximate_tokens(sample)}")
```

### 2. Calculating API Costs

Models charge per token, with different rates for input and output. For Claude 3.5 Sonnet, the rates are `$3` per million input tokens and `$15` per million output tokens.

```python
INPUT_COST_PER_MILLION = 3.0   # USD
OUTPUT_COST_PER_MILLION = 15.0  # USD


def estimate_cost(input_tokens: int, output_tokens: int) -> dict:
    """Calculate the cost of an API call in USD."""
    input_cost = (input_tokens / 1_000_000) * INPUT_COST_PER_MILLION
    output_cost = (output_tokens / 1_000_000) * OUTPUT_COST_PER_MILLION
    total = round(input_cost + output_cost, 6)

    return {
        "input_cost": round(input_cost, 6),
        "output_cost": round(output_cost, 6),
        "total": total
    }


# Example: Processing a 2000-token document and generating a 500-token response
cost = estimate_cost(2000, 500)
print(f"Input cost:  ${cost['input_cost']}")
print(f"Output cost: ${cost['output_cost']}")
print(f"Total cost:  ${cost['total']}")

# Scale up: 100,000 requests per day
daily = cost['total'] * 100_000
print(f"Cost for 100K requests: ${daily:,.2f}/day")
print(f"Cost for 30 days:       ${daily * 30:,.2f}/month")
```

At scale, the math becomes significant. A system doing 100K requests per day with medium-sized prompts costs thousands per month in API calls alone.

### 3. Context Window Budgeting

A long customer support conversation might look like this:

```python
def check_context_budget(
    system_prompt: str,
    conversation: list[dict],
    response_budget: int,
    max_window: int
) -> dict:
    """Check if a request fits within the context window."""
    # Approximate token count for system prompt and messages
    system_tokens = approximate_tokens(system_prompt)
    messages_text = " ".join(
        m["content"] for m in conversation
    )
    messages_tokens = approximate_tokens(messages_text)

    total_prompt = system_tokens + messages_tokens
    total_needed = total_prompt + response_budget

    return {
        "system_tokens": system_tokens,
        "messages_tokens": messages_tokens,
        "prompt_total": total_prompt,
        "response_budget": response_budget,
        "total_needed": total_needed,
        "window_limit": max_window,
        "fits": total_needed <= max_window,
        "overflow": max(0, total_needed - max_window)
    }


system = "You are a helpful customer support agent."
conversation = [
    {"role": "user", "content": "I need help with my order #12345."},
    {"role": "assistant", "content": "Sure, I can help with order #12345. What seems to be the issue?"},
    {"role": "user", "content": "The package arrived damaged. The box was crushed and the product inside is broken."},
]

budget = check_context_budget(system, conversation, response_budget=500, max_window=8000)
print(f"Prompt tokens: {budget['prompt_total']}")
print(f"Response budget: {budget['response_budget']}")
print(f"Total needed: {budget['total_needed']}")
print(f"Window limit: {budget['window_limit']}")
print(f"Fits: {budget['fits']}")
```

If the conversation grows to 30 messages, the prompt tokens might exceed the window. You need strategies to handle this.

## How It Works Internally

Byte-Pair Encoding (BPE) works by iteratively merging the most frequent character pairs in the training corpus into new tokens. The algorithm starts with individual characters and builds up a vocabulary of subword units. Words that appear frequently in training become single tokens ("the", "is", "and"). Rare words get split into multiple tokens ("tokenization" might become "token" + "ization").

This is why the token count for a word depends on how common it is. The word "hello" is likely one token, while "hippopotamus" might be two or three. The vocabulary is fixed after training — the model never sees new token types, only new sequences of existing tokens.

When you send a prompt, the tokenizer runs on the client side (or server side, depending on the SDK) before the model processes anything. The model operates entirely in token-ID space — it never sees your raw text. The output token IDs are decoded back to text by the same tokenizer after generation.

## Build Something Real

Build a context window manager that simulates a long conversation and truncates intelligently:

```python
import anthropic

client = anthropic.Anthropic()


# Simulate a long conversation by generating messages
def simulate_conversation(turns: int) -> list[dict]:
    """Create a simulated conversation of N user-assistant turns."""
    conversation = []
    for i in range(turns):
        conversation.append({
            "role": "user",
            "content": f"This is customer message number {i + 1} about a technical issue."
        })
        conversation.append({
            "role": "assistant",
            "content": f"Thank you for your message number {i + 1}. Let me help you with that technical issue."
        })
    return conversation


def trim_conversation(
    system_prompt: str,
    conversation: list[dict],
    response_budget: int,
    max_window: int
) -> list[dict]:
    """Keep the system prompt and as much recent conversation as fits."""
    system_tokens = approximate_tokens(system_prompt)
    available = max_window - system_tokens - response_budget

    if available <= 0:
        # Return just the most recent user message
        last_message = conversation[-1] if conversation else {"role": "user", "content": ""}
        return [last_message]

    # Work backwards: keep the newest messages that fit
    trimmed = []
    token_count = 0

    for message in reversed(conversation):
        msg_tokens = approximate_tokens(message["content"])
        if token_count + msg_tokens <= available:
            trimmed.insert(0, message)
            token_count += msg_tokens
        else:
            break

    return trimmed


# Simulate a 10-turn conversation and see how much fits
system = "You are a customer support agent for an e-commerce platform."
conversation = simulate_conversation(10)

print(f"Full conversation: {len(conversation)} messages")
print(f"Full conv tokens:  {approximate_tokens(str(conversation))}")

trimmed = trim_conversation(system, conversation, 300, 4000)
print(f"After trimming:    {len(trimmed)} messages")
print(f"Trimmed tokens:    {approximate_tokens(str(trimmed))}")
```

This sliding-window approach keeps the most relevant context (the most recent exchanges) while dropping older context that may no longer be relevant. More sophisticated systems use summarization — condensing old conversation turns into a short summary that fits within the window.
