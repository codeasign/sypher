# Tokens and Context Windows

*Reading time: 5 minutes*

**Difficulty:** Beginner

## What You Will Learn

By the end of this lesson you will be able to:

- Explain what a token is and why tokenization matters
- Calculate approximate token counts for any text
- Describe how context windows constrain what an LLM can process
- Implement strategies for working within context window limits

## Prerequisites

You should already understand:

- How LLMs process text token by token (covered in [LLM Fundamentals](../llm-fundamentals/overview))
- The difference between input and output tokens returned in API responses
- Basic cost awareness — bills are based on token counts

## Why This Matters

Tokens are the currency of the LLM world. Every API call costs tokens, every model has a maximum context window, and every production system must manage token budgets carefully. Engineers who do not understand tokenization make expensive mistakes: prompts that exceed the context window and fail silently, bills that run far higher than expected, and conversations that lose context because earlier messages were not managed properly.

Cost optimization at scale is token optimization. A system handling 10 million requests per month that reduces its average prompt by 20% saves thousands of dollars without changing model quality. Understanding tokens is not theoretical — it directly affects your bottom line and your system's reliability.

## The Problem

You build a customer support chatbot that answers questions based on your product documentation. It works well for single questions. Then users start asking follow-ups. The conversation grows, and suddenly the API returns an error: `prompt is too long: ...`

You try to reduce the conversation history, but now the model forgets context from earlier in the conversation. You try to split the task across multiple calls, but the model loses coherence. The fundamental constraint — a finite context window — limits everything you can build. You cannot fix this with better prompts alone. You need to understand token budgets.

## The Concept

A token is the smallest unit a language model processes. Tokens are not characters or words — they are subword units produced by a tokenizer algorithm (typically Byte-Pair Encoding). The tokenizer converts text into a sequence of integer token IDs, and the model processes those IDs.

Tokenization rules vary by model family, but some general heuristics apply:

- One English word averages about 1.3 tokens
- 100 tokens is approximately 75 English words
- Code tokens are denser — a single character can be its own token
- Non-English text and special characters use more tokens per character
- Whitespace and formatting (indentation, newlines) consume tokens

The context window is the maximum number of tokens the model can process in a single request, including both the prompt (input) and the generated response (output). If your prompt uses 8,000 tokens and the model has a 4,096-token window, the call fails before it starts. If the window is 100,000 tokens but your prompt uses 95,000, the model can still respond but might lose focus in the middle.

Different models have different context windows: entry-level models might offer 8K-16K tokens, mid-range models offer 32K-64K, and high-end models offer 100K-200K tokens. A larger window costs more per call but allows longer documents and conversations.
