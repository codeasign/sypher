# What Is MCP? — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

MCP is not the right choice for every integration:

- **Single, hard-coded tool**: If your agent only needs one tool and you control both sides of the integration, a direct function call is simpler and faster than running an MCP server.
- **High-frequency, low-latency calls**: Each MCP call involves JSON serialization, transport overhead, and a round trip. For sub-millisecond tool calls, an in-process function is faster.
- **Embedded environments**: MCP requires a process or HTTP connection. On microcontrollers or tight embedded runtimes, direct function calls are the only option.
- **Prototype phase**: When experimenting with a new tool, call it directly first. Wrap it in an MCP server only after the tool design is stable.

## The Pattern to Remember

**Pattern**: MCP standardizes how agents discover and call tools, read resources, and use prompts. JSON-RPC 2.0 messages flow over stdio or SSE between a client and a server.

When to use:
- Multiple tools from multiple sources
- Tools that change or evolve independently from the agent
- Systems where tools are developed by different teams

When NOT to use:
- One hard-coded tool in a prototype
- Sub-millisecond latency requirements
- Embedded systems with no process or HTTP support

**Mental model**: MCP is USB-C for AI — one standard connector for any tool. It adds overhead but eliminates bespoke integration code.

## Common Mistakes

### Mistake 1: Forgetting the `notifications/initialized` Message

**Why it happens**: The spec requires sending `initialized` after receiving the `initialize` response, but it is a notification with no response, so it is easy to skip.

**Symptoms**: The server never activates its tools. Tool calls fail with "server not initialized" errors. The client keeps retrying initialization.

**Fix**: Always send `notifications/initialized` immediately after receiving the `initialize` response. This is required by the protocol — servers must not accept tool calls before receiving it.

### Mistake 2: Mismatched Protocol Version

**Why it happens**: You hard-code a protocol version string and it falls out of sync with the server's version.

**Symptoms**: The `initialize` response includes an error about unsupported protocol version. The connection fails at the first step.

**Fix**: The client should propose the latest version it supports. The server responds with the version it supports. If you control both sides, keep the version in a shared constant. The current MCP protocol version at time of writing is `2024-11-05`.

### Mistake 3: Sending Non-Serializable Arguments

**Why it happens**: You pass a Python object or a complex nested structure to a tool without serializing it to JSON first.

**Symptoms**: The `json.dumps()` call in the transport layer raises a `TypeError`. The message never reaches the server.

**Fix**: Ensure all tool arguments are JSON-serializable. If you need to pass binary data, encode it as base64 and include a MIME type field.

### Mistake 4: Not Checking Server Capabilities

**Why it happens**: You assume every server supports tools, resources, and prompts.

**Symptoms**: You call `resources/list` and get an error. The server only supports tools, not resources.

**Fix**: After initialization, check the `capabilities` field in the server's response. Only call methods the server advertises support for.

### Mistake 5: Ignoring Transport-Specific Behavior

**Why it happens**: You write the client for stdio transport and assume it works identically over SSE.

**Symptoms**: Over SSE, messages arrive asynchronously and the request-response correlation breaks. The client reads responses out of order.

**Fix**: When using SSE transport, match responses to requests using the `id` field. Each transport has different message-delivery guarantees — document which transport your client supports.
