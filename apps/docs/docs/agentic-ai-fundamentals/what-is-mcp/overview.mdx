# What Is MCP?

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Explain what the Model Context Protocol is and what problem it solves
- Distinguish MCP from other agent-tool integration approaches
- Identify the three core primitives MCP defines
- Recognize when MCP is the right integration pattern

## Prerequisites

You should already understand:

- How agents use tools and tool calling (covered in [Agent Tools and Tool Calling](../agent-tools-and-tool-calling/overview))
- Basic understanding of client-server architecture
- Familiarity with JSON as a data interchange format

## Why This Matters

Every agent system needs to connect to external tools and data. Today, that means writing custom integration code for every tool — a weather API, a database connector, a file system reader, a Slack integration. Each integration has its own authentication, its own parameter format, its own error handling, and its own deployment model. If you build ten agents that each need the same three tools, you write the same integration code thirty times.

The Model Context Protocol (MCP) solves this by providing a standard protocol for agents to discover and call tools, read resources, and use prompt templates. Think of MCP as USB-C for AI agents — a single standard connector that any MCP-compliant agent can use to connect to any MCP-compliant server.

MCP was developed by Anthropic and released as an open standard. It has gained adoption across the AI ecosystem, with support in Claude Desktop, Claude Code, and growing support in other agent frameworks. Understanding MCP is becoming a core skill for AI engineers because it defines how agents will connect to the world going forward.

## The Problem

You build an agent that needs to read files, query a database, and send Slack messages. You write three integration modules — one for the filesystem, one for PostgreSQL, one for Slack's API. Each module handles authentication, error codes, rate limiting, and response parsing differently. The agent code is tightly coupled to each integration.

Then you build a second agent — a different role, different team — that needs the same three tools. You either duplicate the integration code or extract a shared library. Either way, every tool integration is a bespoke piece of engineering that does not transfer to other agent systems or other development teams.

There is no standard way for an agent to discover what tools a server offers, understand how to call them, or know what data formats to expect. Every integration is hand-rolled. MCP standardizes this.

## The Concept

MCP defines a client-server protocol where an MCP client (an agent or AI application) connects to an MCP server (a service that provides tools, data, or prompts). The connection uses JSON-RPC 2.0 as the message format, ensuring structured communication.

The server exposes three types of primitives:

**Tools** — functions the agent can call. These are equivalent to the tool definitions you write for an LLM API, but they are defined on the server side and discovered dynamically rather than hard-coded in the agent.

**Resources** — data that the agent can read. These are like read-only endpoints that return structured data — files, database query results, API responses.

**Prompts** — reusable prompt templates stored on the server. These allow servers to provide pre-built prompt patterns that the client can use.

When an agent connects to an MCP server, it first negotiates capabilities through an initialization handshake. Then it can discover available tools, read resources, and call them — all through a standardized protocol that any MCP-compliant agent can use.
