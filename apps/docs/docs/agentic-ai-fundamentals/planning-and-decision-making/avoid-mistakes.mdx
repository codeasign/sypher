# Planning and Decision Making — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Explicit planning architectures are not always needed:

- **Reflex agents**: If every input maps to a single deterministic action, no planning is required.
- **Single-step tasks**: If the task completes in one action, planning adds overhead without benefit.
- **Tasks with a fixed recipe**: If the sequence is always the same, hardcode it as a workflow.
- **Real-time systems**: Planning adds latency. For sub-second responses, use reactive decision-making.

## The Pattern to Remember

**Pattern**: Planning decomposes tasks, decides the order, and adapts when steps fail.

When to use:
- Complex multi-step tasks
- Tasks where failure recovery is important
- Problems requiring task decomposition

When NOT to use:
- Simple single-step lookups
- Fixed-sequence operations
- Latency-critical applications

**Mental model**: Planning is like writing a recipe before cooking. For a simple sandwich you do not need one. For a multi-course meal, the recipe prevents mistakes.

## Common Mistakes

### Mistake 1: Planning Too Far Ahead

**Why it happens**: The planner generates a detailed 20-step plan for a task that changes as it executes.

**Symptoms**: The agent executes 5 steps successfully, but the remaining 15 steps are invalid because the situation has changed. The agent wastes time replanning.

**Fix**: Plan only 2-3 steps ahead. Generate the next steps based on the results of the completed steps. This is called "receding horizon planning."

### Mistake 2: No Fallback for Planner Failure

**Why it happens**: The planner always produces a plan, so you assume it always works.

**Symptoms**: The planner outputs a plan with impossible steps ("fly to the moon"). The agent tries to execute impossible steps and fails.

**Fix**: Validate the plan before executing. Check that each step corresponds to a tool the agent has. If the plan contains impossible steps, request a revised plan with constraints.

### Mistake 3: The Agent Does Not Know When to Stop Planning

**Why it happens**: No termination condition is set. The agent keeps planning and replanning.

**Symptoms**: The agent enters a planning loop — making and remaking the same plan without making progress.

**Fix**: Set a maximum number of planning rounds. Use a timer: if the agent has been planning longer than N seconds without progress, stop and return what it has.

### Mistake 4: Ignoring the Cost of Replanning

**Why it happens**: Each replan is an API call. Several replans add up.

**Symptoms**: A task that should take 3 tool calls takes 10 because the agent replanned 7 times after minor failures.

**Fix**: Distinguish between critical failures (requiring replan) and minor failures (can be handled inline). Only trigger a full replan for critical failures.

### Mistake 5: Not Distinguishing Planning from Execution State

**Why it happens**: The plan and the execution results are stored in the same message list without separation.

**Symptoms**: The model confuses the plan with what actually happened. It thinks a step was completed when it was only planned.

**Fix**: Keep the current plan in a separate data structure. The message list contains only what actually happened (execution results). The plan is a mutable document that evolves.

```python
# Do this separation:
state = {
    "plan": ["Step 1", "Step 2", "Step 3"],  # What we intend to do
    "completed": ["Step 1"],                    # What we actually did
    "failed": [],                               # What went wrong
}
# The message list only contains actual tool calls and results, not planned ones.
```
