# Plan and Execute — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Plan-and-Execute adds overhead compared to ReAct:

- **Short tasks (1-3 steps)**: ReAct or even a direct loop handles short tasks more efficiently.
- **Highly dynamic tasks**: If the task changes every turn and upfront planning is impossible, skip the plan phase.
- **Tasks that are already step-by-step**: If the user provides numbered instructions, the plan is already done.

## The Pattern to Remember

**Pattern**: Generate a plan upfront, execute each step with tool calls, update the plan on failure, synthesize results.

When to use:
- Tasks with 5+ steps
- Tasks where step ordering matters
- Tasks requiring structured progress tracking

When NOT to use:
- 1-3 step tasks
- Rapidly changing environments
- Tasks where the user provides the plan

**Mental model**: Plan-and-Execute is like a project manager writing a Gantt chart, then the team executing it. The chart is independent of the daily standup notes.

## Common Mistakes

### Mistake 1: The Plan Is Too Generic

**Why it happens**: The LLM generates high-level steps that are not specific enough to guide execution.

**Symptoms**: "Research topic" as a step — the executor has no idea what specific search queries to run or what aspects to cover.

**Fix**: After the plan phase, validate that each step is concrete and searchable. Prompt the model to rephrase vague steps as specific actions.

### Mistake 2: The Plan Is Never Updated

**Why it happens**: You generate the plan once and execute blindly, ignoring what happens during execution.

**Symptoms**: The agent completes steps that are no longer relevant. The plan and the reality diverge completely.

**Fix**: After each step, pass the updated plan status to the model. Let the model suggest plan revisions when it encounters unexpected results.

### Mistake 3: Steps Are Too Coupled

**Why it happens**: Step 5 depends on the exact output of Step 3, but the plan does not specify this dependency.

**Symptoms**: The executor completes Step 3, but the output does not include information that Step 5 needs. Step 5 fails or repeats work.

**Fix**: Make dependencies explicit in the plan: "Step 5 requires the output of Step 3." Pass accumulated results to each step.

### Mistake 4: Plan Execution Is Too Rigid

**Why it happens**: The executor follows the plan order strictly, even when a different order would be more efficient.

**Symptoms**: The agent completes steps A, B, C in order when B and C could run in parallel. Wall-clock time is higher than necessary.

**Fix**: Mark independent steps and execute them in parallel. The plan document can include a "parallel group" annotation for steps that do not depend on each other.

### Mistake 5: Plan Retention After Completion

**Why it happens**: The plan document stays in the agent's context even after all steps are complete.

**Symptoms**: The plan steps clutter the context window in subsequent tasks. The agent hallucinates that it still has pending steps.

**Fix**: Archive the plan once all steps are complete. Replace it with a "completed" marker and the final summary.
