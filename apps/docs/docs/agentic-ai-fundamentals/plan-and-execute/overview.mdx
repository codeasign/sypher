# Plan and Execute

*Reading time: 5 minutes*

**Difficulty:** Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Implement a Plan-and-Execute agent that separates planning from execution at the task level
- Design plans that are specific enough to guide execution but flexible enough to handle changes
- Handle plan validation and re-execution when steps fail
- Choose between ReAct and Plan-and-Execute based on task characteristics

## Prerequisites

You should already understand:

- The observe-think-act loop (covered in [Observe-Think-Act Loop](../observe-think-act-loop/overview))
- The ReAct pattern (covered in [ReAct Pattern](../react-pattern/overview))
- Planning strategies for agents (covered in [Planning and Decision Making](../planning-and-decision-making/overview))

## Why This Matters

ReAct works well for tasks requiring 3-5 steps. Beyond that, the agent loses track of where it is in the overall task. It starts chasing side quests, repeating completed steps, or forgetting the original goal. Plan-and-Execute solves this by maintaining an explicit plan document that persists across turns.

Companies like Google (Vertex AI Agent Builder) and Microsoft (AutoGen) support Plan-and-Execute patterns. When you need agents that reliably complete tasks with 10 or more steps, the explicit plan becomes essential for maintaining coherence.

## The Problem

Your research agent has been running for 15 turns. It was asked to "compile a competitive analysis report on three cloud AI platforms." Five turns ago, it was comparing pricing. Now it is searching for the founder biography of one platform. The original goal — a structured comparison — has degraded into random fact-finding.

The agent lost the plan. It has no document that says "I need to cover: pricing, features, limitations, and ecosystem for each platform." Without the plan, the model drifts toward whatever seems interesting at each step.

## The Concept

Plan-and-Execute separates the agent's work into two phases:

**Plan phase**. Before any tool is called, the agent generates a plan. The plan is a structured document — a numbered list of steps, a JSON list of milestones, or a hierarchical task breakdown. The plan is stored outside the message list and persists across all execution turns.

**Execute phase**. The agent works through the plan one step at a time. After each step, it marks the step as complete, failed, or blocked. The plan is updated. If a step fails, the agent can revise the plan from the current state.

The key difference from ReAct is that the plan is an explicit, mutable document. In ReAct, the plan exists only in the model's context and degrades as the conversation grows. In Plan-and-Execute, the plan is preserved and updated independently of the conversation history.
