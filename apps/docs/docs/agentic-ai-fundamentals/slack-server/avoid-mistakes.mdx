# Avoid Mistakes: Slack MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A Slack MCP server gives an agent access to your team's messages. Do not use it when:

- **Messages contain sensitive or confidential data** — the agent reads all messages it can access; consider restricting the bot to a single channel
- **You need ephemeral or real-time responses** — the agent sends messages via the API, not as a live user typing; use Slack's Events API for instant responses
- **The workspace has strict data residency requirements** — messages sent through the agent pass through the MCP host, which may not comply with your data policies
- **You only need outgoing notifications** — a simple webhook integration is lighter than a full MCP server

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Slack Web API wrapped as typed MCP tools for messaging and search |
| **When to use** | Agent needs to read, search, or send messages in Slack |
| **When NOT to use** | Sensitive channels, ephemeral responses, simple notifications |
| **Mental model** | A programmable Slack bot with read, write, and search capabilities |

## Common Mistakes

### Mistake 1: Using a Token with Excess Channel Access

**Mistake:** Installing the bot with default scopes that grant access to all public channels.

**Why it happens:** Slack's default OAuth installation grants broad access unless you explicitly restrict it.

**Symptoms:** The agent reads messages from channels it should not access, like #legal or #hr.

**Fix:** In the Slack app settings under **OAuth & Permissions**, use the **Granular Bot Token** option and limit the bot to specific channels. Alternatively, navigate to the channel and use `/invite @ClaudeBot` only for channels the bot should join.

### Mistake 2: Sending Messages Without Proper Formatting

**Mistake:** Sending raw text that does not render well in Slack's Markdown-like format.

**Why it happens:** The agent sends plain text without using Slack's `mrkdwn` formatting syntax.

**Symptoms:** Links appear as raw URLs, code blocks do not render, and lists are just text with dashes.

**Fix:** Use Slack's formatting in the message text: `*bold*`, `~strikethrough~`, `` `code` ``, and `>quote`. For code blocks, wrap in `` ``` ``. The official server supports `mrkdwn` by default.

### Mistake 3: Searching with Too Broad a Query

**Mistake:** The agent calls `slack_search_messages` with a generic one-word query like "bug".

**Why it happens:** The agent does not know how deeply to qualify its search.

**Symptoms:** Hundreds of results, most irrelevant, exhausting the search quota and the agent's context window.

**Fix:** Encourage the agent to use specific, multi-word queries with channel scoping. A query like `"login timeout" in:#engineering` returns far more relevant results.

### Mistake 4: Forgetting That the Bot Only Sees Messages After It Joins

**Mistake:** Expecting the bot to find messages from before it was installed or joined a channel.

**Why it happens:** Slack API history access for bots depends on the `channels:history` scope, but only covers messages from the point the bot joins.

**Symptoms:** The agent reports "no messages found" for a channel that has years of history.

**Fix:** Use the `search:read` scope for full workspace search across all history, not just `channels:history`. The `search:read` scope is broader and may require additional approval.

### Mistake 5: Rate Limiting — Too Many Messages Too Fast

**Mistake:** The agent calls `slack_send_message` multiple times rapidly.

**Why it happens:** The agent tries to send a multi-part message as separate calls instead of one combined message.

**Symptoms:** Slack returns a `429 Too Many Requests` error, and the agent's messages are dropped.

**Fix:** Combine related content into a single message. The official server implements retry logic, but the agent should still prefer one well-formatted message over many small ones.
