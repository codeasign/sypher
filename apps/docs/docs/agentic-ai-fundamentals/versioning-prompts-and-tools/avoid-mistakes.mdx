# Avoid Mistakes: Versioning Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Full prompt and tool versioning with a registry, semantic tags, and rollback infrastructure is not always the right investment. You can skip it when:

- **You are prototyping** — in the first week or two of a new agent project, prompts change every few minutes. The overhead of registering each version far exceeds the benefit. Keep prompts in a text file and track changes with git.
- **You are the sole developer** — if only one person iterates on prompts and tools, you can reason about changes verbally and track them informally. The audit trail matters most when multiple people make changes independently.
- **Your agent has fewer than 100 daily sessions** — at low volumes, you can debug issues by re-running with the current prompt. You rarely need to reconstruct exactly what prompt ran three weeks ago.
- **Your agent uses a single, fixed prompt** — if the system prompt never changes between deployments and tools are never added or removed, versioning adds complexity with no corresponding benefit. A simple git commit history is sufficient.
- **Your compliance requirements are minimal** — if you do not need to prove which prompt version ran on a specific date for audit or regulatory reasons, the investment in a formal versioning system may not pay off.

In all these cases, git-based versioning alone — storing prompt files and tool definitions in the repository and deploying through your existing CI/CD pipeline — provides enough structure without the operational overhead of a registry service.

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Content-hash registry for prompts + semantic versioning for tool definitions + immutable session pinning |
| **When to use** | Multi-developer teams, production deployments, compliance requirements, A/B testing |
| **When NOT to use** | Prototyping, solo developers, low-volume agents, static prompts |
| **Mental model** | Every prompt and tool definition is a software dependency — version it, tag it, and pin it the same way you pin library versions in a requirements file |

## Common Mistakes

### Mistake 1: Versioning Prompt Text but Not Tool Descriptions

**Mistake:** You store every version of the system prompt in the registry, but tool descriptions (the text that tells the model how to use each tool) live directly in the tool definition files. When you update a tool description to improve call accuracy, there is no record of what the model saw before the change.

**Why it happens:** Tool descriptions feel like implementation details, not user-facing content. You think of them as code comments rather than prompts.

**Symptoms:** A tool starts being called incorrectly after a description update. You cannot tell whether the model was seeing the old or new description when the calls went wrong. You have to grep through git history and guess.

**Fix:** Treat tool descriptions as first-class prompt content. Version them alongside your system prompt in the same registry. Include the full tool schema — name, description, parameter names, parameter descriptions — in the registered prompt version. The agent should reconstruct the complete prompt from the registry, not from separate files.

### Mistake 2: Breaking Backward Compatibility Silently

**Mistake:** You change a tool parameter's name or type and deploy the new server without updating active sessions. Sessions pinned to an older configuration try to call the tool with the old parameter names and fail.

**Why it happens:** In a monorepo or monolith, you assume every component updates at the same time. With agent systems, sessions can run for minutes or hours, and distributed deployments mean version skew is the norm.

**Symptoms:** Intermittent tool call failures that correlate with deployment windows. The error message says "unexpected keyword argument" but gives no indication that the issue is a version mismatch.

**Fix:** Always add new parameters as optional with defaults before removing old ones. Use a deprecation window: mark the old parameter as deprecated in version N, remove it in version N+2. Your runtime should warn when a deprecated parameter is used, giving consumers time to migrate.

```python
# migration_example.py
TOOL_MIGRATIONS = {
    "search_db": [
        # Version 1.0 -> 1.1: added max_results (backward compatible)
        {"from": "1.0.0", "to": "1.1.0",
         "changes": [{"action": "add", "param": "max_results",
                      "type": "integer", "required": False, "default": 10}]},
        # Version 1.1 -> 2.0: renamed limit to max_results (breaking)
        {"from": "1.1.0", "to": "2.0.0",
         "changes": [{"action": "rename", "old": "limit", "new": "max_results"}]},
    ]
}
```

### Mistake 3: No Migration Path for Old Prompts

**Mistake:** You register a new prompt version, move the production tag to it, and forget about the old version. Months later, when debugging a regression, you discover that the old prompt was stored in the database but the tag history is gone — you cannot reconstruct what the production prompt looked like three versions ago.

**Why it happens:** The registry only stores the current tag-to-hash mapping. When you move a tag, the old mapping is overwritten. The prompt content is still in the database by hash, but you lose the chronological record of which hash was active when.

**Symptoms:** You need to answer "what prompt was live on June 15?" but your tag history only shows the current mapping. You have to cross-reference git commits and deployment logs to approximate the answer.

**Fix:** Keep a tag history table that records every tag movement with timestamps. When you move a tag, insert a row in the version history:

```python
# tag_history.py
class TagHistory:
    def record_move(self, tag: str, from_hash: str, to_hash: str, reason: str):
        self.conn.execute(
            "INSERT INTO tag_history (tag, from_hash, to_hash, reason, moved_at) "
            "VALUES (?, ?, ?, ?, ?)",
            (tag, from_hash, to_hash, reason, datetime.now().isoformat())
        )
        self.conn.commit()

    def get_tag_state_at(self, tag: str, timestamp: str) -> Optional[str]:
        """Return the hash that was active for a tag at a given time."""
        row = self.conn.execute(
            """SELECT to_hash FROM tag_history
               WHERE tag = ? AND moved_at <= ?
               ORDER BY moved_at DESC LIMIT 1""",
            (tag, timestamp)
        ).fetchone()
        return row[0] if row else None
```

### Mistake 4: Versioning Prompts but Not Tool Schemas

**Mistake:** You have a beautiful prompt registry with content hashes and tags, but your tool definitions are unversioned Python dataclasses with no version field. When you add a parameter, you have no way to know which version of the schema the agent was using when it made a tool call three weeks ago.

**Why it happens:** Tool schemas feel like code, and code is versioned by git. But git tells you what was in the file at a commit, not what the deployed server was running at a point in time.

**Symptoms:** You can trace a bad output to a specific prompt version, but you cannot tell whether the tool schema the model saw matches the tool schema the server expects. You spend hours reconstructing deployment timelines from deployment logs.

**Fix:** Every tool definition should carry a version string. Record the tool version alongside the prompt hash in the agent configuration. Both are part of what the model sees, and both must be versioned together.

### Mistake 5: No Audit Trail for Prompt Changes

**Mistake:** Multiple developers on your team can register new prompts and move tags. Anyone with database access can change what prompt runs in production. There is no record of who changed what, when, or why.

**Why it happens:** The registry is treated as data, not code. It does not go through code review, and it does not have an access control layer.

**Symptoms:** A production prompt gets changed without anyone knowing who did it or why. The team spends a meeting trying to figure out whether a change was intentional. Trust in the versioning system erodes.

**Fix:** Every registry write operation should record the actor, the change, and the reason. Require an authenticated token or API key for write operations. Keep an immutable audit log separate from the tag history:

```python
# audit_log.py
class AuditLog:
    def __init__(self, log_path: str = "audit.log"):
        self.log_path = log_path

    def record(self, actor: str, action: str, resource: str,
               details: dict) -> None:
        entry = json.dumps({
            "actor": actor,
            "action": action,
            "resource": resource,
            "details": details,
            "timestamp": datetime.now().isoformat(),
        })
        with open(self.log_path, "a") as f:
            f.write(entry + "\n")

    def query(self, actor: Optional[str] = None,
              action: Optional[str] = None,
              since: Optional[str] = None) -> list[dict]:
        """Query the audit log with optional filters."""
        results = []
        with open(self.log_path, "r") as f:
            for line in f:
                entry = json.loads(line)
                if actor and entry["actor"] != actor:
                    continue
                if action and entry["action"] != action:
                    continue
                if since and entry["timestamp"] < since:
                    continue
                results.append(entry)
        return results
```
