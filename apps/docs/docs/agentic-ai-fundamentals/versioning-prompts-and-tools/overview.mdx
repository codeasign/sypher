# Versioning Prompts and Tools

⏱️ 15 minutes · Difficulty: Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Version prompts alongside your application code using content hashing and semantic tags
- Manage tool definition changes across agent versions without breaking existing sessions
- Implement rollback strategies that restore a known-good prompt or tool configuration in seconds
- Apply semantic versioning to agent configurations so you can reason about compatibility at a glance

## Prerequisites

You should already understand:

- How git works for source code versioning — commits, branches, tags, and rollbacks (see [Atlassian Git Tutorial](https://www.atlassian.com/git/tutorials))
- The anatomy of agent tools and how they are defined (see [Anatomy of an Agent](../anatomy-of-an-agent/overview))
- Prompt engineering fundamentals — system prompts, tool descriptions, and how they influence model behavior (see [Prompt Engineering Basics](../prompt-engineering-basics/overview))
- How agents call tools and pass parameters at runtime (see [Agent Tools and Tool Calling](../agent-tools-and-tool-calling/overview))

## Why This Matters

Prompts and tool definitions are code. They change over time through the same natural lifecycle as any software artifact: you iterate, you experiment, you ship improvements. Unlike backend service code, however, a prompt change has no compiler to catch regressions. A single word change in a system prompt can silently degrade your agent's accuracy from 95 percent to 60 percent on a specific task category, while improving it on another. Without versioning, you cannot answer the most basic question in production debugging: "What prompt was this agent running when it generated that output?"

Tool definitions carry similar risk. When you add a parameter, rename a field, or change a tool description, every agent session using the old definition is at risk of a runtime mismatch. A customer's long-running workflow may execute a tool call crafted against version 1.2 of your schema, but your MCP server now expects version 1.3. The result is a silent failure, a cryptic error, or worse — data corruption from a parameter that changed meaning between versions.

Versioning provides the safety net. It lets you roll back a bad prompt deployment the same way you revert a bad code commit. It gives you an audit trail so you can trace a production incident to the exact prompt version that triggered it. And it enables controlled experimentation — deploying a new prompt to five percent of traffic, measuring the impact, and rolling forward or back with confidence.

## The Problem

A prompt tweak that improves one type of query breaks another. For example, you add "be concise" to your system prompt to reduce token usage in simple lookups, and it works — your average response length drops by 40 percent. But your agents that generate detailed technical reports now produce shallow, incomplete answers because the same instruction suppresses their depth. Users start complaining about missing analysis.

Meanwhile, your tool signatures are evolving. The `search_database` tool started with two parameters: `query` and `limit`. In version 1.1 you added `filters`. In version 1.2 you renamed `limit` to `max_results`. Any agent session that was started with the old tool definition and tries to call the renamed parameter will fail at runtime. Worse, if you deployed the server-side change first, old clients see a mysterious "unexpected keyword argument" error that gives no hint about the real cause.

You need to know which prompt version ran in a given session when debugging a production issue. You need to know that a tool call was crafted against schema 1.1 and the server is now on schema 1.2, so the mismatch is expected. You need a system that makes this information explicit and actionable — not buried in a git log you have to reconstruct manually.

## The Concept

Prompt versioning and tool versioning share the same core principle: every change must be uniquely identifiable, reversible, and traceable to a specific runtime instance. Three main approaches exist, and production systems often combine them:

**Git-based versioning** treats prompts as files in your repository. Every commit creates a new version. The prompt file path or git hash becomes the version identifier. This approach is simple — you already use git — but it requires a deployment pipeline to sync prompt updates from the repository to the runtime. You also need a mapping from "which git hash was deployed where" to make rollbacks practical.

**Registry-based versioning** stores prompts in a database or key-value store. Each prompt version gets a content hash and an optional semantic tag. The agent fetches the prompt by tag (for example, "v1.2.3" or "staging") at startup. This decouples prompt deployment from code deployment — you can update a prompt without rebuilding and redeploying the agent. The trade-off is operational complexity: you need a registry service, migration scripts, and a way to audit who changed what.

**Bundled versioning** embeds prompts directly in the agent application binary or container image. The prompt is part of the artifact. This guarantees that a given agent version always runs the exact prompt you tested. Rollback means redeploying the old artifact. The trade-off is slower iteration — every prompt tweak requires a full build and deploy cycle.

Tool contract versioning follows a similar pattern. Each tool definition carries a semantic version (for example, `search_database@1.2.0`). The agent records which version of each tool's schema it used when constructing a tool call. The MCP server advertises its supported version range. At runtime, the host negotiates a compatible version or fails with a clear message. This is the same principle as API versioning in web services, applied at the tool level.

Semantic versioning for agent configurations ties everything together. Your complete agent configuration — system prompt, tool definitions, temperature, max tokens, and so on — gets a single version string. When you change any component, you bump the configuration version. This gives you a one-number answer to "what was this agent running?" and a clear upgrade path for consumers.
