# Orchestrator and Supervisor Patterns — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Orchestrator and supervisor patterns add architectural complexity. They are not always the right choice:

- **Single-agent tasks**: If one agent can handle the job, adding coordination layers adds latency and failure modes for no benefit.
- **Sequential pipelines**: If the work is a fixed sequence of steps where each depends on the previous one, a simple pipeline or workflow is simpler than an orchestrator.
- **Real-time systems**: The round-trip overhead of an orchestrator delegating and collecting results can exceed latency budgets.
- **Tightly coupled work**: If sub-tasks share state so heavily that splitting them is artificial, a single agent is more coherent.
- **Small-scale experiments**: Prototype with a single agent first. Add multi-agent coordination only when you have evidence that a single agent fails.

## The Pattern to Remember

**Pattern**: Orchestrator routes and aggregates; supervisor evaluates and intervenes.

When to use:
- Tasks that decompose cleanly into independent sub-tasks
- Pipelines requiring quality gates between stages
- Systems where different domains need specialized agent configurations

When NOT to use:
- Tasks a single agent handles well
- Real-time systems with strict latency requirements
- Prototypes and experiments

**Mental model**: An orchestrator is a manager who delegates and assembles. A supervisor is a quality inspector who accepts or rejects. Combine them when both routing and quality control matter.

## Common Mistakes

### Mistake 1: Workers That Talk to Each Other Directly

**Why it happens**: It seems efficient to let worker agents communicate directly instead of routing through the orchestrator.

**Symptoms**: The message flow becomes a tangled web. Workers overwrite each other's state. Debugging is impossible because you cannot trace which agent produced which output.

**Fix**: Enforce a rule that workers communicate only through the orchestrator. The orchestrator is the single source of truth for task assignments and results.

### Mistake 2: Supervisor That Is Too Strict

**Why it happens**: You set high quality standards to ensure good output, but you do not calibrate the pass threshold.

**Symptoms**: Every output gets flagged for revision. The system spends most of its time in revision loops. Most tasks eventually escalate even for simple requests.

**Fix**: Start with a lenient supervisor and tighten thresholds based on real data. Log all verdicts for the first week and analyze where false rejections occur.

### Mistake 3: No Escalation Path

**Why it happens**: You assume the orchestrator or supervisor will always resolve issues within the agent team.

**Symptoms**: The system loops indefinitely between revision and evaluation. Bad output gets accepted because all revision attempts are exhausted and the system has no fallback.

**Fix**: Always define an escalation path — to a human operator, a fallback agent, or a safe default response. Every coordination pattern needs an escape hatch.

### Mistake 4: Orchestrator Becomes a Bottleneck

**Why it happens**: You route every interaction through the orchestrator, including trivial status checks that workers could handle themselves.

**Symptoms**: The orchestrator's message log grows linearly with the number of workers. Wall-clock latency increases because the orchestrator processes every message sequentially.

**Fix**: The orchestrator should delegate whole tasks, not individual steps within a task. Once a worker receives a task, let it run to completion before reporting back.

### Mistake 5: Ignoring Worker Failures

**Why it happens**: You write the happy path first and assume workers will always succeed.

**Symptoms**: When a worker throws an exception, the orchestrator crashes or produces a partial result that looks complete but is missing data.

**Fix**: Treat every worker call as a fallible operation. Wrap each delegation in try/except. The orchestrator should handle worker failures gracefully — retry, skip, or escalate based on the failure type.
