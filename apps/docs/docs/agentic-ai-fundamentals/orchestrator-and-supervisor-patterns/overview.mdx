# Orchestrator and Supervisor Patterns

*Reading time: 6 minutes*

**Difficulty:** Advanced

- Choose between centralized and hierarchical coordination for your use case

## Why This Matters

As soon as you deploy an agent in production, you hit the same wall: a single agent cannot do everything well. It might be great at writing code but terrible at navigating the filesystem. It might handle customer inquiries well but hallucinate product IDs. The natural next step is to divide labor across multiple agents, each specialized for one domain.

But dividing labor introduces a new problem — coordination. Who decides which agent handles what? What happens when two agents disagree? How do you aggregate results without losing context? Without a coordination pattern, you end up with agents talking past each other, duplicating work, or stepping on each other's results.

Production multi-agent systems at companies like Anthropic, Google, and Microsoft use two fundamental coordination patterns: orchestrator and supervisor. These patterns determine how control flows between agents and how conflicts are resolved. Getting this right is what separates a team of agents that collaborates effectively from a group of agents that creates chaos.

## The Problem

You have three specialized agents — a research agent, a writing agent, and a review agent. You ask the system to produce a market analysis report. Without coordination:

The research agent writes a raw report. The writing agent tries to write its own version but has no data. The review agent rejects both because neither fits the format. Each agent overwrites the others' work. The message log becomes a mess of conflicting instructions and partial outputs.

You need a coordination layer that decides: who works on what, in what order, and how final decisions are made when agents disagree.

## The Concept

The orchestrator pattern uses a central coordinator agent that breaks down a task, delegates sub-tasks to worker agents, collects their results, and assembles the final output. The workers communicate only with the orchestrator, not with each other. This is the simplest multi-agent pattern and works well when the task decomposition is clear.

The supervisor pattern uses one agent in a monitoring role over one or more worker agents. The supervisor does not do the work itself — it evaluates the workers' output, decides if it is good enough, and intervenes when it is not. The supervisor can reject work, request revisions, or escalate to a human. This pattern works well when quality control is the primary concern.

These two patterns combine naturally. An orchestrator can supervise its workers. A supervisor can orchestrate its subordinates. The distinction is about which role dominates in your architecture.
