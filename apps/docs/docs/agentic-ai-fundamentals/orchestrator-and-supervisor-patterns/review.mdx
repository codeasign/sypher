# Orchestrator and Supervisor Patterns — Review

*Reading time: 3 minutes*

## Key Takeaways

- The orchestrator pattern uses a central coordinator to decompose tasks, delegate to workers, and aggregate results
- The supervisor pattern uses an evaluator that monitors worker output and decides to accept, request revision, or escalate
- Workers should communicate only through the orchestrator — direct worker-to-worker communication creates untraceable state
- Always define an escalation path: every coordination pattern needs a fallback when agents cannot resolve an issue
- Combine orchestrator and supervisor for production pipelines where both work routing and quality control are necessary

## Recap in 60 Seconds

1. An orchestrator splits a task into sub-tasks, assigns each to a specialized worker, and assembles the results — workers never talk to each other directly
2. A supervisor evaluates worker output and can accept, request revision, or escalate — it does not do the work itself
3. Combine both patterns by routing work through an orchestrator and running quality gates through a supervisor before accepting results
