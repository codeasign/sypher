# MCP Architecture — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

The MCP client-server architecture with full lifecycle management is not always required:

- **Single-process tools**: If your tool is a Python function in the same process as your agent, calling it directly is simpler than spawning a subprocess and managing the MCP lifecycle.
- **Read-only, static data**: For resources that never change (embedded configuration, fixed lookup tables), a simple import is better than an MCP server.
- **Ephemeral connections**: If you connect to a server for one tool call and disconnect, the initialization overhead (two round trips) may exceed the tool call itself.

## The Pattern to Remember

**Pattern**: MCP follows a strict lifecycle — transport connect, initialize handshake, initialized notification, operation, transport close. Servers must reject operations received before initialization completes.

When to use:
- Multi-tool servers accessed from multiple clients
- Tools that need setup, teardown, or authentication
- Production deployments with separate client and server processes

When NOT to use:
- Single-process, single-client tools
- Static read-only data
- One-off tool calls where initialization overhead dominates

**Mental model**: The MCP lifecycle is like TLS handshake — you cannot send application data until the handshake completes, and both sides must agree on capabilities.

## Common Mistakes

### Mistake 1: Sending Tool Calls Before Initialization

**Why it happens**: You batch all messages — initialize and tool call — in the same output stream without waiting for the initialize response.

**Symptoms**: The server receives the tool call before it finishes initialization. Depending on implementation, the call is ignored, rejected, or processed with incomplete state.

**Fix**: Wait for the initialize response before sending any tool calls or resource reads. The lifecycle is strict: initialize, then `notifications/initialized`, then operations.

### Mistake 2: Half-Initialized Server State

**Why it happens**: The server returns its capabilities in the initialize response but does not wait for the `notifications/initialized` before making tools available.

**Symptoms**: Race conditions where the server exposes tools before it has fully loaded them. Early tool calls get inconsistent results.

**Fix**: The server should set a flag only after receiving `notifications/initialized`. All operations should check this flag before executing.

### Mistake 3: Ignoring the `id` Field for SSE Transport

**Why it happens**: Over stdio, messages are request-response paired in order, so you can ignore the `id` field. Over SSE, this breaks.

**Symptoms**: The client sends three requests and reads three responses. Over SSE, responses may arrive interleaved. The client matches them in order and gets wrong data for each call.

**Fix**: Always match responses to requests using the `id` field, regardless of transport. This works correctly for both stdio and SSE.

### Mistake 4: Not Handling Server Disconnect

**Why it happens**: You assume the server process stays alive for the duration of the client session.

**Symptoms**: The server crashes. The client's next tool call hangs forever waiting on stdin/stdout. No error is raised.

**Fix**: Set a timeout on transport reads. If the server does not respond within the timeout, assume disconnection and attempt re-initialization.

### Mistake 5: Running the Server in a Different Working Directory

**Why it happens**: The client spawns the server process but does not set the working directory.

**Symptoms**: The server cannot find its configuration files, tool scripts, or data directories. Tools that use relative paths behave differently in development and production.

**Fix**: Always specify the working directory when spawning the server process, or use absolute paths for all server-side resources.
