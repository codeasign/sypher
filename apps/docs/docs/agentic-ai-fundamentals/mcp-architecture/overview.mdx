# MCP Architecture

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Describe the MCP client-server architecture and its components
- Explain the capability negotiation process between client and server
- Understand the lifecycle of an MCP connection
- Distinguish between the two transport layers: stdio and SSE

## Prerequisites

You should already understand:

- The basic concept of MCP and JSON-RPC 2.0 (covered in [What Is MCP?](../what-is-mcp/overview))
- Client-server architecture fundamentals
- The difference between synchronous and asynchronous communication

## Why This Matters

MCP is not just a protocol — it is an architecture for connecting agents to the world. Understanding the architecture is essential because it determines how you design, deploy, and debug MCP-based systems. If you do not understand the architecture, you will make poor decisions about which transport to use, how to handle failures, and how to scale your MCP deployment.

The architecture has three layers: the **client** (the agent or AI application that initiates connections), the **server** (the service that provides tools, resources, and prompts), and the **transport** (the communication channel between them). Each layer has specific responsibilities and failure modes.

When you deploy an MCP server in production, you need to decide: does it run as a child process of the client (stdio), or as a standalone HTTP server (SSE)? Does it maintain persistent state, or is it stateless? Does the client reconnect if the server crashes? These are architectural decisions that flow from understanding how MCP is designed.

## The Problem

You have built an MCP server and a client. In development, they work perfectly — the client connects, initializes, and calls tools without issue. In production, the client gets a "handle" to the server at startup and holds it for the lifetime of the application. When the server crashes or the network drops, the handle is broken. The client does not know whether to reconnect, reinitialize, or fail entirely.

Without understanding the MCP lifecycle, you cannot answer these questions. When does the protocol require re-initialization? What state is preserved across a reconnect? How does capability negotiation differ between the first connection and a reconnection? The answers live in the architecture.

## The Concept

MCP architecture follows a strict client-initiated lifecycle:

**1. Connection** — the client establishes a transport channel to the server. For stdio transport, this means spawning the server process and connecting to its stdin/stdout. For SSE transport, this means making an HTTP request to establish an SSE stream.

**2. Initialization** — the client sends an `initialize` request with its protocol version and capabilities. The server responds with its own version and capabilities. The client then sends an `initialized` notification.

**3. Operation** — once initialized, the client can list tools, call tools, list resources, read resources, list prompts, and use prompts. Both sides send JSON-RPC messages over the transport.

**4. Shutdown** — the client (or server) closes the transport. No explicit "shutdown" message is required — dropping the transport is sufficient.

The architecture enforces that initialization must complete before any operation begins. Servers should reject tool calls and resource reads received before initialization completes. This guarantees that both sides agree on capabilities before any work is done.
