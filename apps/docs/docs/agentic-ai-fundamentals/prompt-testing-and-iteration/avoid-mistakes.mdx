# Prompt Testing and Iteration — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Systematic prompt testing is valuable but not always necessary:

- **One-shot prompts**: If a prompt is used exactly once, building a test suite is overkill
- **Prototypes**: In early exploration, rapid experimentation with manual testing is more productive
- **Trivial tasks**: "Translate to French" does not need a 30-case test suite. A manual spot-check is sufficient
- **Tasks with no ground truth**: For open-ended creative tasks, you cannot define "correct" output. Use human evaluation instead of automated metrics

## The Pattern to Remember

**Pattern**: Test dataset → evaluation → metrics → refine → repeat. Treat prompt changes like code changes — test them before deploying.

When to use:
- Prompts used in production
- Prompts that other developers maintain
- Changes affecting multiple test cases

When NOT to use:
- One-off prompts
- Early prototypes
- Tasks without determinable ground truth

**Mental model**: Prompt evaluation is model testing, not unit testing. The prompt is a probabilistic instruction, and the test suite measures its statistical reliability. A test suite with 30+ cases gives you meaningful signal.

## Common Mistakes

### Mistake 1: Overfitting to the Test Dataset

**Why it happens**: You keep refining the prompt until it passes every test case, optimizing for the specific examples in your dataset.

**Symptoms**: Accuracy on the test set is 100%, but performance on new, unseen inputs is worse than the original prompt. The prompt has memorized your test cases rather than learning the general rule.

**Fix**: Reserve 20% of your dataset as a held-out validation set that you never optimize against. Only evaluate on the validation set after your iteration loop is complete.

### Mistake 2: Using a Test Dataset That Is Too Small

**Why it happens**: You write 5 test cases because they cover the main categories.

**Symptoms**: A change that passes 5 cases might still fail on the 6th category you did not test. Small datasets give unreliable accuracy estimates.

**Fix**: Aim for 20-100 test cases for a production prompt. Include edge cases, empty inputs, very long inputs, and ambiguous inputs.

### Mistake 3: Testing Only Happy Paths

**Why it happens**: You write test cases for the most common inputs, which are also the ones the model handles best.

**Symptoms**: Your prompt accuracy looks great in tests but fails in production on edge cases you never tested.

**Fix**: Specifically add test cases for edge cases: empty input, very long input, mixed languages, unexpected formats, and inputs near category boundaries.

### Mistake 4: Not Versioning Evaluated Prompts

**Why it happens**: You tweak prompts and run evaluations, but you do not track which version produced which result.

**Symptoms**: You cannot tell if a change improved or regressed accuracy because you do not have the old evaluation results.

**Fix**: Store evaluation reports alongside prompt versions. Tag each evaluation with the prompt hash, timestamp, and dataset version.

### Mistake 5: Treating Accuracy as the Only Metric

**Why it happens**: Accuracy is easy to compute and understand.

**Symptoms**: A prompt that achieves 95% accuracy by always predicting "GENERAL" looks great, but it never catches BUG or FEATURE_REQUEST. Accuracy hides the failure on minority classes.

**Fix**: Track per-category accuracy, precision, and recall in addition to overall accuracy. A prompt that fails on a critical category is not safe for production, even if overall accuracy is high.

```python
def per_category_metrics(results: list[dict]) -> dict:
    """Compute per-category accuracy from evaluation results."""
    categories = {}

    for r in results:
        cat = r["expected"]
        if cat not in categories:
            categories[cat] = {"total": 0, "correct": 0}
        categories[cat]["total"] += 1
        if r["correct"]:
            categories[cat]["correct"] += 1

    for cat, data in categories.items():
        data["accuracy"] = data["correct"] / data["total"]

    return categories
```
