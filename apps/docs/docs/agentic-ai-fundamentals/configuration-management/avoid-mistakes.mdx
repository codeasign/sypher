# Avoid Mistakes: Configuration Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A full layered configuration management system adds complexity. Do not implement it when:

- **You are building a single-purpose agent with one deployment** — a prototype, demo, or internal tool that only runs in one environment does not need environment-specific YAML files, a layered loader, or a reload daemon. Hardcoded defaults and a few environment variables are sufficient.

- **Your agent has fewer than 5 configurable parameters** — if you only configure the model name, API key, and log level, a simple `os.environ.get()` at startup is cleaner than a Pydantic config hierarchy.

- **You are the only developer** — the primary benefit of structured config management is team coordination: making config differences visible, reviewable, and auditable across environments. A solo developer on a small project can carry the config in their head.

- **You deploy via a Platform-as-a-Service that has its own config UI** — services like Vercel, Railway, or Heroku provide their own per-environment configuration management. Using their built-in tooling is simpler than replicating it in your codebase.

When you grow beyond these constraints, adopt the layered system incrementally. Start with a single Pydantic model and default YAML file, then add per-environment files and reload only when you need them.

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Layered config resolution: defaults -> environment file -> secrets -> runtime overrides |
| **When to use** | Multi-environment deployments, team-managed agent systems, production agent fleets |
| **When NOT to use** | Single-environment prototypes, solo projects, agents with fewer than 5 config parameters |
| **Mental model** | A hierarchical config file for each environment, with secrets injected from outside, validated at startup |

## Common Mistakes

### Mistake 1: Committing Secrets to Git

**Mistake:** Storing API keys, database passwords, or authentication tokens in configuration files that are checked into version control.

**Why it happens:** It is the path of least resistance. The developer adds the API key to a config file to make the agent work locally, then commits everything without noticing the sensitive value. Even if you remove it in a later commit, the secret lives forever in the git history.

**Symptoms:** A security scan alerts you that API keys are found in your repository. An attacker clones the repo and uses your credentials.

**Fix:** Never write secrets into files that are committed to version control. Use a `.env` file (added to `.gitignore`) for local development. Use environment variables or a secrets vault for production. Add a pre-commit hook that scans for patterns matching API keys and tokens:

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: detect-private-key
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks
```

### Mistake 2: Config Drift Across Environments

**Mistake:** Letting configuration values diverge between development, staging, and production without tracking the differences.

**Why it happens:** Someone tweaks a value directly on the production server to fix an urgent issue. They forget to backport the change to the config files in the repository. The next deployment overwrites the production change with the old config file, and the fix disappears.

**Symptoms:** A bug that exists in production but cannot be reproduced in staging. The exact same code behaves differently in each environment.

**Fix:** Treat configuration files as code. Every change must go through version control and be deployed through the same pipeline as code changes. Never edit configuration files directly on servers. Use a diff tool to audit config files across environments:

```bash
# Compare default and production configs
diff <(yq eval config/default.yaml) <(yq eval config/production.yaml)

# Or use a Python script to show environment differences
python -c "
import yaml
with open('config/default.yaml') as f: default = yaml.safe_load(f)
with open('config/production.yaml') as f: prod = yaml.safe_load(f)

def show_diffs(base, override, path=''):
    for k, v in override.items():
        cur_path = f'{path}.{k}' if path else k
        if k not in base:
            print(f'  ADDED  {cur_path}={v}')
        elif isinstance(v, dict) and isinstance(base.get(k), dict):
            show_diffs(base[k], v, cur_path)
        elif base.get(k) != v:
            print(f'  CHANGED {cur_path}: {base[k]} -> {v}')

show_diffs(default, prod)
"
```

### Mistake 3: No Config Validation at Startup

**Mistake:** Loading configuration without validating it, allowing invalid values to reach agent logic at runtime.

**Why it happens:** Configuration is loaded as a plain dictionary or with a simple YAML parser that accepts any structure. Validation is assumed to happen when the values are used.

**Symptoms:** The agent crashes 30 minutes into a session because `max_tokens` was set to zero, or the model name was misspelled. The error message says something cryptic like "API error: model not found" with no indication that the config file has a typo.

**Fix:** Validate all configuration at startup using Pydantic or a similar schema validator. Fail fast — if configuration is invalid, the agent should refuse to start. This prevents silent failures that waste token budget:

```python
def validate_and_exit(config_manager):
    """Validate config at startup. Exit with clear message on failure."""
    try:
        config = config_manager.load()
        return config
    except Exception as e:
        print(f"Configuration Error: {e}")
        print("Fix the configuration and restart the agent.")
        import sys
        sys.exit(1)
```

### Mistake 4: Mutable Config Causing Race Conditions

**Mistake:** Allowing configuration objects to be mutated at runtime, so that one part of the agent sees different values than another part.

**Why it happens:** The config object is a plain dictionary or a mutable object that gets passed by reference. One component modifies a value as a side effect — for example, caching a computed property by overwriting the config field. Other components reading the same config field get the cached value instead of the original.

**Symptoms:** Intermittent, non-reproducible bugs. The agent works differently depending on the order in which components access configuration. Debugging is extremely difficult because the config values depend on execution history.

**Fix:** Make configuration objects immutable after creation. Pydantic supports this with `frozen=True`:

```python
class AgentConfig(BaseModel):
    model_config = {"frozen": True}
    environment: str = "development"
    model: ModelConfig = ModelConfig()
    tools: ToolConfig = ToolConfig()
```

When config needs to be updated (for example, after a hot-reload), create a new instance rather than modifying the existing one:

```python
# Correct: create new config instance
def reload_config():
    new_config = config_manager.load(environment)
    agent.reinitialize(new_config)  # pass new instance

# Wrong: mutating existing config
def reload_config_bad():
    config.current_config.model.name = "new-model"  # raises error if frozen
```

### Mistake 5: Overloading Environment Variables

**Mistake:** Using environment variables for every configuration value, resulting in dozens or hundreds of environment variables that are undocumented, unvalidated, and impossible to audit.

**Why it happens:** Environment variables are the first config mechanism developers learn. They are easy to set, easy to read, and work everywhere. As the agent grows, every new parameter becomes another environment variable.

**Symptoms:** A single agent deployment requires 40 environment variables. New developers spend hours tracking down which variables are needed. A missing variable causes a silent fallback to defaults. There is no single file that shows the complete configuration for an environment.

**Fix:** Use environment variables only for deployment-specific values and secrets. Use YAML config files for everything else. Aim for fewer than 10 environment variables per agent:

| Process | Variables (max) |
|---|---|
| Secrets and credentials | 3-5 (API keys, tokens) |
| Deployment identity | 1 (AGENT_ENV or equivalent) |
| Feature flags at deploy time | 0-2 (use YAML config or a feature flag service instead) |

Validate that no unexpected environment variables are being used, which signals that something was added ad-hoc without updating the config files. A startup check can help:

```python
EXPECTED_ENV_VARS = {"AGENT_ENV", "ANTHROPIC_API_KEY", "LOG_LEVEL"}

def check_unexpected_env_vars():
    agent_vars = {k for k in os.environ if k.startswith("AGENT_") or "API_KEY" in k}
    unexpected = agent_vars - EXPECTED_ENV_VARS
    if unexpected:
        import logging
        logging.warning(f"Unexpected env vars found (consider adding to config files): {unexpected}")
```
