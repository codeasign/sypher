# Avoid Mistakes: Caching and Retry Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Caching and retry systems are valuable for most production agents, but they are not always the right choice:

- **Non-repetitive tasks.** If your agent handles one-off queries that are never repeated — for example, a code generation tool that produces unique output for each prompt — caching adds overhead with zero benefit. The cache check costs time and memory and will always miss.

- **Very low-volume agents.** If your agent runs a few dozen sessions per day, the cost of LLM calls is negligible and the operational complexity of running Redis and maintaining cache invalidation logic outweighs the savings. Skip caching until you have a measurable cost problem.

- **Write-heavy operations.** Agents that create records, send messages, or mutate data should not cache their results — every invocation is a new action. Retries are still valuable for reliability, but caching responses that represent side effects (for example, "Email sent successfully") would return a stale result to a retried call.

- **Real-time or highly dynamic data.** Agents answering questions about live stock prices, sports scores, or system health metrics should not serve stale cached results. Use very short TTLs (5-15 seconds) or skip caching entirely for these domains.

## The Pattern to Remember

**Pattern**: Cache LLM responses to avoid redundant API calls, and retry tool calls with exponential backoff to survive transient failures.

| Scenario | Caching | Retries |
|----------|---------|---------|
| Repeated user queries (FAQ, policy) | Use exact-match + semantic cache | Optional — failures are rare for cached reads |
| Tool calls to external APIs | Not typically cached | Required — network failures are common |
| Write/mutate operations | Never cache the result | Required — confirm the write succeeded |
| Long-running agent sessions | Cache intermediate LLM responses | Required — each tool call is a failure point |
| Cross-session shared knowledge | Use semantic cache with longer TTL | Not applicable — shared cache handles this |

**Mental model**: Think of caching as a performance optimization that reduces cost and latency, and retries as a reliability mechanism that prevents transient failures from ending agent sessions. They complement each other: caching reduces the number of calls that could fail, and retries handle the failures that still occur.

## Common Mistakes

### Mistake 1: Caching Without TTL

**Why it happens**: You set up a cache, it works perfectly, and you never configure expiration. The cache grows unbounded, consuming memory, and responses become increasingly stale until users notice outdated information.

**Symptoms**: The agent serves the exact same answer about "current pricing" for weeks. Users complain the information is wrong. The cache contains millions of entries and Redis runs out of memory.

**Fix**: Always set a TTL. Start with 3600 seconds (1 hour) for general knowledge and 30-300 seconds for dynamic data. Audit cache entries regularly and remove anything older than your maximum TTL.

```python
# Always set a TTL when caching
await cache.set(key, response, ttl_s=3600)

# Without TTL — memory leak and stale data
# await cache.set(key, response)  # BAD
```

### Mistake 2: Retrying Without Backoff (DoS-ing the API)

**Why it happens**: You add a simple `try-except` retry loop that retries immediately. The downstream service is already under stress, and your immediate retries arrive at the worst possible time, making the problem worse.

**Symptoms**: A database outage causes every agent instance to retry every tool call instantly. The database gets hammered with requests while it is already failing. Recovery takes longer because the retry storm prevents the database from catching up.

**Fix**: Always add exponential backoff with jitter. The delay should grow: 1s, 2s, 4s. Jitter randomizes the delay so multiple agents do not retry in lockstep.

```python
import time
import random

# BAD — immediate retry with no backoff
for attempt in range(3):
    try:
        result = call_api()
        break
    except Exception:
        time.sleep(0)  # Immediate retry — creates retry storm
        continue

# GOOD — exponential backoff with jitter
for attempt in range(3):
    try:
        result = call_api()
        break
    except Exception:
        delay = 1.0 * (2 ** attempt) * (0.5 + random.random() * 0.5)
        time.sleep(delay)
        continue
```

### Mistake 3: Caching Sensitive Data

**Why it happens**: The cache stores the LLM response without checking whether the response contains personally identifiable information (PII), API keys, or other sensitive data. A subsequent user query matches the cache key and receives another user's private information.

**Symptoms**: User A asks "what is my account balance?" and the LLM generates a response containing User A's balance. User B asks the same question, hits the same cache key, and receives User A's balance. This is a data breach.

**Fix**: Include user identity in the cache key when responses contain user-specific data. Or disable caching for queries that return personal data. Use a cache allowlist that only caches responses from approved, non-sensitive query types.

```python
def _cache_key(self, messages, user_id, tools, model):
    """Include user_id in the key for user-specific queries."""
    payload = {
        "user_id": user_id,  # Prevents cross-user cache hits for personal data
        "messages": messages,
        "tools": tools,
        "model": model,
    }
    # ... hash and return key
```

### Mistake 4: Infinite Retry Loops

**Why it happens**: You set `max_retries` to a very high number or forget to set a maximum at all. The agent retries a permanent failure — a 404 endpoint, an invalid API key, a malformed request — indefinitely, consuming time and tokens for a request that will never succeed.

**Symptoms**: An agent session runs for 15 minutes retrying a tool call that returns HTTP 404 every time. The user receives no response. The agent burns through tokens and frustrates the user.

**Fix**: Set a low maximum retry count (3-5 attempts). Differentiate between retryable errors (5xx, timeout, rate limit) and non-retryable errors (4xx, auth failure, validation error). Only retry the retryable ones.

```python
RETRYABLE_STATUSES = {429, 500, 502, 503, 504}
NON_RETRYABLE_STATUSES = {400, 401, 403, 404, 422}


def is_retryable(response_status: int) -> bool:
    return response_status in RETRYABLE_STATUSES


# In your retry loop
for attempt in range(max_retries):
    response = make_request()
    if response.status == 200:
        return response.body
    if not is_retryable(response.status):
        raise PermanentError(f"Non-retryable error: {response.status}")
    # Otherwise, wait and retry
```

### Mistake 5: Cache Poisoning with Bad Responses

**Why it happens**: The LLM generates a bad response — hallucinated facts, incorrect tool calls, offensive content — and the cache stores it. Every subsequent query that hits the same cache key serves the bad response. A single bad LLM output becomes a persistent problem that outlives the generation context.

**Symptoms**: A user asks a factual question, gets a hallucinated answer, and reports it. You investigate but cannot reproduce the issue because the cache serves the same response to every user. The bad response persists until the TTL expires or someone manually clears the cache.

**Fix**: Validate responses before caching. Check for known error patterns, empty responses, or low-confidence outputs. Add a "do not cache" signal that the agent can set when it detects a problematic response. Implement a cache-busting mechanism for manual invalidation of poisoned entries.

```python
def should_cache(response: str) -> bool:
    """Return False if the response should not be cached."""
    if len(response.strip()) == 0:
        return False  # Empty response
    if "I cannot answer" in response and "unsafe" in response:
        return False  # Safety refusal — don't cache refusal patterns
    if len(response) < 20:
        return False  # Too short to be useful
    return True


# Before caching
if should_cache(response):
    await cache.set(key, response, ttl_s=3600)
else:
    print("Skipping cache for low-quality response")
```
