# When to Use Multi-Agent — Review

*Reading time: 3 minutes*

## Key Takeaways

- Start with a single agent — multi-agent is an optimization, not a default
- The decision framework evaluates three dimensions: domain separation, failure mode analysis, and cost tolerance
- Tool counts above 30 and distinct domain boundaries are the strongest signals to split into specialist agents
- Each additional agent multiplies your per-request token cost — compute the cost before committing
- Keep the single-agent implementation as a rollback path when deploying multi-agent systems

## Recap in 60 Seconds

1. Build a single-agent prototype first and measure its failures — only add agents when you have evidence that a single agent cannot handle the load
2. Split by domain boundaries (different tools, data, and rules), not by features — if two agents share tools and data, merge them
3. Always compute the cost multiplier before deploying multi-agent — the accuracy gain may not justify 3x or 4x the token spend
