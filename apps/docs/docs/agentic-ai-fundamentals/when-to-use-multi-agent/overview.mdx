# When to Use Multi-Agent

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Evaluate whether a task benefits from a multi-agent architecture
- Calculate the complexity cost of adding agents to a system
- Recognize the warning signs that a single agent needs to be split
- Choose the right multi-agent pattern for your specific use case

## Prerequisites

You should already understand:

- How single agents work (covered in [What Is an Agent?](../what-is-an-agent/overview))
- The orchestrator, supervisor, and specialist patterns from this section
- Basic understanding of system design trade-offs

## Why This Matters

The most common mistake in agent engineering is over-engineering. A developer reads about multi-agent systems and immediately wants to build one — with a supervisor, three specialists, and a message bus — for a task that a single LLM call could handle. The result is a system that is slower, more expensive, harder to debug, and less reliable than a simple alternative.

Adding agents introduces real costs: increased latency (each agent adds at least one LLM round trip), higher token consumption (more system prompts, more context), more failure modes (agents can fail independently), and harder debugging (you cannot single-step through multiple agents' reasoning).

On the other hand, under-engineering is also a trap. A single agent with 50 tools and a 6000-token system prompt will be worse at every task than three focused agents. Knowing when one agent is not enough and when three agents are too many is the skill that separates production-grade designs from academic exercises.

## The Problem

You need to build a system that answers customer support questions. The naive approach: one agent with access to the knowledge base, the billing system, the CRM, and the product catalog. You build it in a day, and it works in demos. But in production, it hallucinates billing information because the tool descriptions for billing and product queries are too similar. It calls the CRM update tool when it should just read. It says "I've updated your account" when it only read from it.

You need a different architecture. But should you split into three specialist agents? Add a supervisor? Build a pipeline? The answer depends on your specific failure patterns, not on a general preference for one architecture over another.

## The Concept

The decision to use a multi-agent system comes down to three factors: **domain separation**, **failure mode analysis**, and **cost tolerance**.

**Domain separation** asks: does the task involve distinct domains with different tools, different rules, and different success criteria? If yes, those domains should probably be separate agents. If the entire task uses the same tools and the same knowledge, a single agent is better.

**Failure mode analysis** asks: what goes wrong with a single-agent approach? If the failures are caused by tool overload (too many tools, the model picks wrong ones), context pressure (system prompt too long), or instruction conflicts (rules for one domain contradict rules for another), those failures signal that a multi-agent split would help.

**Cost tolerance** asks: how much latency and token overhead can you afford? Each additional agent multiplies your per-request cost. A multi-agent system with three agents costs roughly three times as much per task as a single-agent system (assuming similar model sizes).
