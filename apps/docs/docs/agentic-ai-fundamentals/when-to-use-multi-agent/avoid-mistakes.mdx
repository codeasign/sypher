# When to Use Multi-Agent — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

The decision framework itself has limitations:

- **Unclear requirements**: If you do not yet know the full tool list or domain boundaries, a single-agent prototype is better than a multi-agent design based on guesses.
- **Startup velocity phase**: When speed of iteration matters more than correctness, a single agent is faster to build and modify. Split later.
- **One-person team**: Managing multiple agents, their prompts, their tools, and their interactions is a coordination burden. One agent, one mental model.
- **Task with high ambiguity**: If the task itself is poorly defined, you do not know what tools or domains you need. Build an interactive chatbot first, then formalize.

## The Pattern to Remember

**Pattern**: Start with a single agent. Measure failures. Split only when you have evidence that a single agent cannot handle the load.

When to use:
- 30+ total tools across distinct domains
- Measured tool-confusion errors in a single-agent system
- Context-window pressure on the system prompt

When NOT to use:
- During prototyping and requirements gathering
- When a single agent meets all accuracy and latency requirements
- When you cannot afford the Nx cost multiplier

**Mental model**: Multi-agent is an optimization, not a default. Start with one. Add agents surgically, not architecturally.

## Common Mistakes

### Mistake 1: Designing for Multi-Agent Before Writing Code

**Why it happens**: Multi-agent architecture diagrams look impressive, and it feels like you are "doing AI right" by planning three agents from day one.

**Symptoms**: You spend two weeks designing a multi-agent system. After building it, you discover a single-agent prototype would have handled 90% of the tasks adequately.

**Fix**: Build a single-agent prototype in one day. Test it against real data. If you find specific failure modes that multi-agent would fix, then refactor. Never design a multi-agent system before running a single-agent baseline.

### Mistake 2: Splitting by Feature, Not by Domain

**Why it happens**: Each feature in the codebase gets its own agent. The "password reset" agent, the "email notification" agent, the "profile picture" agent.

**Symptoms**: You have many agents, each with 2-3 tools, all in closely related domains. The orchestrator is overwhelmed routing between them. Most tasks touch multiple agents anyway.

**Fix**: Split by domain boundaries that have different tools, different data access, and different rules, not by individual features. If two agents share the same tools and data sources, they should be one agent.

### Mistake 3: Ignoring the Cost Multiplier

**Why it happens**: You benchmark single-agent accuracy at 85% and multi-agent accuracy at 93%. You choose multi-agent without looking at the cost.

**Symptoms**: The deployment costs 4x more than expected. The 8% accuracy gain might not be worth 4x the token spend.

**Fix**: Always compute the cost-per-task for both architectures before committing. Sometimes a single agent with better prompts or a fine-tuned model is more cost-effective than a multi-agent system.

### Mistake 4: Assuming More Agents = Better Quality

**Why it happens**: It is intuitive that more specialized agents should produce better results.

**Symptoms**: Adding agents increases latency, adds error modes, and can actually lower quality because context is lost in handoffs.

**Fix**: Measure, do not assume. Run A/B tests comparing single-agent and multi-agent output quality. If multi-agent does not measurably improve quality, do not deploy it.

### Mistake 5: No Rollback Plan

**Why it happens**: You go all-in on a multi-agent architecture and delete the single-agent code.

**Symptoms**: When the multi-agent system has a bug in production, you cannot quickly revert to the simpler system that worked.

**Fix**: Keep the single-agent implementation as a fallback. Use a feature flag to switch between architectures. When the multi-agent system fails, fall back to the single-agent path.
