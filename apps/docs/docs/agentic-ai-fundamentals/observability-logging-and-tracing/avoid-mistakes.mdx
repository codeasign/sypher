# Avoid Mistakes: Observability Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Full observability instrumentation is expensive in both engineering time and infrastructure cost. Do not invest heavily when:

- **Your agent runs fewer than 100 sessions per day** — manual log inspection works at this scale
- **You are in the first week of development** — focus on getting the agent working, not measuring it
- **You have no monitoring infrastructure** — starting with simple file-based logs is better than setting up Grafana + Loki + Tempo before you have users
- **Your agent only handles one type of task** — a single-purpose agent has fewer failure modes and needs less observability

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Trace IDs + structured JSON logs + OpenTelemetry spans + time-series metrics |
| **When to use** | 100+ sessions/day, multi-tool agents, production deployments |
| **When NOT to use** | Early prototyping, low volume, single-purpose agents |
| **Mental model** | Flight recorder for every agent decision — capture everything, query when needed |

## Common Mistakes

### Mistake 1: Logging Everything at the Same Detail Level

**Mistake:** Logging every token of every LLM response with the same verbosity as every tool call.

**Why it happens:** It is simpler to log everything with one logging level.

**Symptoms:** A single agent session produces megabytes of logs (full LLM responses, full tool outputs), making storage expensive and querying slow.

**Fix:** Use three log levels:

| Level | Contents | Retention |
|---|---|---|
| DEBUG | Full LLM responses, full tool outputs | 24 hours |
| INFO | Tool calls (name + params), latencies, errors | 30 days |
| WARN | Task failures, safety violations, loops | 90 days |

### Mistake 2: Logging PII and API Keys

**Mistake:** Logging the full user input, which may contain names, emails, credit card numbers, or API keys.

**Why it happens:** The natural thing to log is "what the user typed." That is often PII.

**Symptoms:** Your log aggregator contains sensitive user data, creating a compliance risk and a security liability.

**Fix:** Redact sensitive patterns before logging:

```python
import re

def redact_pii(text: str) -> str:
    text = re.sub(r"\b[\w\.-]+@[\w\.-]+\.\w+\b", "[EMAIL]", text)
    text = re.sub(r"\b\d{16}\b", "[CREDIT_CARD]", text)
    text = re.sub(r"\b(?:ghp_|sk-|xoxb-|BSA_)[A-Za-z0-9_-]+\b", "[API_KEY]", text)
    return text
```

### Mistake 3: Not Correlating Logs Across Servers

**Mistake:** Each MCP server logs independently without a shared trace ID.

**Why it happens:** Each server is a separate process, and without explicit trace propagation, they generate their own log IDs.

**Symptoms:** A failure that spans the filesystem server and the database server produces two sets of logs that cannot be linked.

**Fix:** Pass the trace ID as a header or parameter from the MCP host to every server. Each server includes the trace ID in every log entry. All logs with the same trace ID belong to the same session.

### Mistake 4: Storing Logs Without Aggregation

**Mistake:** Writing logs to local files without shipping them to a central aggregator.

**Why it happens:** File-based logging is the simplest thing that works during development.

**Symptoms:** When the server restarts or crashes, local logs are lost. Debugging a production issue requires SSH-ing into the instance and grepping files.

**Fix:** Ship logs to a central aggregator (Grafana Loki, Datadog, Elasticsearch) from the first day of production use. Use a lightweight log shipper like Promtail or Fluentd.

### Mistake 5: Measuring Everything, Acting on Nothing

**Mistake:** Building a beautiful dashboard with 50 charts but no alerting or action triggers.

**Why it happens:** Dashboards are visible and satisfying to build. Alert rules require defining thresholds.

**Symptoms:** The dashboard shows that error rates spiked at 3 AM, but nobody noticed until the morning standup.

**Fix:** For every dashboard metric, define an alert threshold: error rate above 10%, latency p95 above 30 seconds, tool call count per session above 20. Route alerts to Slack or PagerDuty.
