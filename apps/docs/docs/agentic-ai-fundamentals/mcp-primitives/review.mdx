# MCP Primitives — Review

*Reading time: 3 minutes*

## Key Takeaways

- MCP defines three primitives: Tools (callable functions with arguments and side effects), Resources (read-only data with URIs), and Prompts (reusable templates producing structured messages)
- Choose a Tool for operations that have side effects or need model-mediated selection; choose a Resource for static or computed read-only data; choose a Prompt for reusable prompt engineering expertise
- Resources must be read-only — never add side effects to a resource read
- Use resource templates (`uriTemplate`) for dynamic data that cannot be enumerated at startup
- Tool descriptions should explicitly differentiate between similar tools to prevent model confusion

## Recap in 60 Seconds

1. A Tool does something (action), a Resource is something (data), and a Prompt frames something (template) — choose based on which semantics fit your capability
2. Resources go through direct client reads; Tools go through model selection — this distinction is why choosing the wrong primitive adds unnecessary latency
3. A good Prompt template encodes domain expertise that the client does not have — if it is just "answer the question," it is not adding value
