# Building a Web Search MCP Server

⏱️ 15 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Configure the Brave Search MCP server to give your agent web search capabilities
- Control search scope — web, news, local, or image results
- Fetch and summarize webpage content through the agent
- Implement rate limiting and result filtering to manage API costs

## Prerequisites

You should already understand:

- How MCP tools are defined and called (see [MCP Primitives](../mcp-primitives/overview))
- What a search API is and how API keys work
- How to set environment variables for MCP server configuration (see [GitHub Server](../github-server/overview))
- Basic awareness of web crawling ethics and `robots.txt`

## Why This Matters

An agent that only knows what is in its training data is frozen in time. It cannot answer "what is the current weather in Tokyo," "what were the latest npm package versions released this week," or "what did the CEO say in yesterday's earnings call." A web search MCP server bridges the gap between the agent's static knowledge and the live internet.

Real applications of web search MCP servers include: research assistants that gather and synthesize information from multiple sources, coding agents that search Stack Overflow or MDN for the latest API changes, support agents that look up documentation for the product version a customer is using, and competitive analysis agents that monitor news sources for mentions of competitors.

The search server is often paired with the filesystem server: the agent searches the web for a solution, then writes the solution to a local file. Combined with the GitHub server, an agent can search for a bug fix, implement it, and open a PR — all in a single session.

## The Problem

An LLM's training data has a cutoff date. Ask about anything after that date and the model either says "I don't know" or, worse, hallucinates an answer that sounds plausible but is completely wrong. You need a way to give the model access to current information without retraining it.

Giving the agent a generic HTTP tool and asking it to call a search API directly is fragile. The agent needs to know the API endpoint, the query parameter format, the authentication header schema, and how to parse the response — all from its training data, which may be outdated. A web search MCP server wraps the entire search workflow into a single tool call: `web_search(query, count)` and optionally `fetch_page(url)`. The agent does not need to know which search engine runs behind the scenes.

## The Concept

A web search MCP server uses a search API — Brave Search, Tavily, SerpAPI, or Bing — to retrieve live web results and return them to the agent as structured tool responses. The server handles API authentication, pagination, result formatting, and rate limiting.

The primary tool, `web_search`, accepts a query string and optional parameters like result count, freshness (past day, week, month), and search type (web, news, images). It returns a list of results, each with a title, URL, and snippet. A secondary tool, `fetch_page`, retrieves and renders the text content of a specific URL so the agent can read the full article or documentation page.
