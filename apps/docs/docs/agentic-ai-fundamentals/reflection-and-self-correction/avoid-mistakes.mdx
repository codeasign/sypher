# Reflection and Self-Correction — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Reflection is not always worth the cost:

- **Tasks where errors are not costly**: If a wrong answer has low impact, the extra API call for reflection is wasted.
- **Real-time systems**: Each reflection doubles or triples the response time.
- **Tasks the model handles well**: If the model always produces correct output for a given task, do not add reflection overhead.
- **Creative tasks**: Reflection can make creative output overly cautious and generic.

## The Pattern to Remember

**Pattern**: Produce → Reflect → Revise → Repeat until passing or max revisions reached.

When to use:
- High-stakes tasks (financial calculations, legal analysis, code generation)
- Tasks where errors are expensive
- Situations where the user cannot verify the output

When NOT to use:
- Low-stakes tasks
- Real-time applications
- Creative or open-ended tasks

**Mental model**: Reflection is a code review before deployment. Not every line of code needs a review, but critical code always gets one.

## Common Mistakes

### Mistake 1: Reflection Fatigue — Changing Correct Output

**Why it happens**: The reflection pass is too strict. It flags minor style issues and "corrects" them, degrading the output.

**Symptoms**: The revised output is worse than the original. The reflection keeps finding trivial issues even on correct answers.

**Fix**: Use a scoring rubric with clear pass/fail thresholds. Only revise if the score is below the threshold. Ignore style preferences.

### Mistake 2: False Positives in Reflection

**Why it happens**: The reflection model hallucinates errors — it sees problems that do not exist.

**Symptoms**: The reflection says "missing information" for data that is clearly present. The agent wastes a revision on a non-issue.

**Fix**: Ask the reflection to provide concrete evidence for each claimed issue: "Quote the specific text that is wrong." If the reflection cannot quote evidence, ignore the claim.

### Mistake 3: Runaway Revision Loops

**Why it happens**: Each revision introduces new issues that the next reflection catches, leading to infinite revision cycles.

**Symptoms**: The agent keeps revising forever, alternating between two flawed versions. The revisions never converge on a passing state.

**Fix**: Set a hard max_revisions limit (2-3). After the limit, use the best version (highest score) regardless of whether it passed.

### Mistake 4: Reflection Cost Outweighing Benefit

**Why it happens**: You reflect on every output, including outputs that are almost always correct.

**Symptoms**: The API cost doubles for a 5% improvement in accuracy. The ROI of reflection is negative.

**Fix**: Only reflect when the model's confidence is below a threshold, or only reflect on high-stakes outputs. Measure the accuracy improvement per reflection dollar.

### Mistake 5: The Reflection Model Is Too Weak

**Why it happens**: You use a small, cheap model for reflection to save costs, but it cannot reliably identify errors.

**Symptoms**: The reflection passes outputs that have obvious errors. The self-correction loop provides no quality improvement.

**Fix**: Reflection needs a model at least as capable as the generator. If the reflection model cannot find errors, it is not adding value.
