# Prompt Engineering Basics — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Prompt engineering is essential for most LLM interactions, but there are cases where it is overkill:

- **Tool/function calling**: When using structured tool definitions, the model's output is constrained by the tool schema — heavy prompt engineering is redundant
- **Short, factual queries**: "What is the capital of France?" does not need role assignment or format constraints
- **Guided generation / structured output APIs**: If the API enforces a JSON schema, you do not need to prompt for JSON format
- **Fine-tuned models**: If a model is fine-tuned for a specific task, the prompt can be much simpler — the fine-tuning handles the behavior

## The Pattern to Remember

**Pattern**: Role + Task + Format + Constraints = Reliable Output.

When to use:
- Any prompt where the output must meet specific requirements
- Multi-step instructions
- Prompts that other developers will maintain
- Production systems with quality requirements

When NOT to use:
- Trivial lookups with a single correct answer
- APIs that already constrain output via tool schemas

**Mental model**: A prompt is a function specification. The more precisely you specify inputs, outputs, and constraints, the more consistently the model executes.

## Common Mistakes

### Mistake 1: Contradictory Instructions

**Why it happens**: You layer multiple instructions without checking for conflicts — "be creative" and "output only the category name" contradict each other.

**Symptoms**: The model sometimes follows one instruction, sometimes the other. Output seems random.

**Fix**: Read your prompt as a whole and check for conflicts. If you want structured output, set temperature to 0 and do not ask for creativity.

### Mistake 2: Burying the Task in Background

**Why it happens**: You provide extensive context before stating the actual task, and the model focuses on the context instead.

**Symptoms**: The model summarizes your background information rather than performing the task. It writes about the problem domain instead of solving the problem.

**Fix**: State the task first, then provide context. Or use the system prompt for context and the user message for the task.

### Mistake 3: Asking the Model to Ignore Something

**Why it happens**: You say "Ignore all previous instructions" or "Do not think about elephants." The model must think about elephants to know not to mention them.

**Symptoms**: The model does the thing you told it to ignore. Negative instructions draw attention to the behavior you want to suppress.

**Fix**: Tell the model what TO do, not what NOT to do. Instead of "Do not mention pricing," say "Focus only on features and benefits."

### Mistake 4: Over-Prompting

**Why it happens**: You keep adding instructions to fix every edge case, making the prompt extremely long.

**Symptoms**: The model follows the first instruction and ignores later ones. Response quality degrades as the prompt length increases beyond a certain point.

**Fix**: Remove redundant instructions. Each element should serve a unique purpose. If a 200-token prompt works as well as a 2000-token prompt, use the shorter one.

### Mistake 5: Assuming the Model Interprets Ambiguity the Same Way You Do

**Why it happens**: "Write a brief summary" seems clear to you. The model writes a 500-word summary because "brief" is relative.

**Symptoms**: Output lengths vary wildly. The model misses details you assumed were obvious.

**Fix**: Quantify everything. Instead of "brief," say "max 100 words." Instead of "recent," specify a date range. Instead of "well-formatted," provide an example of the format.
