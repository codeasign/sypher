# Avoid Mistakes: GitHub MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A GitHub MCP server gives an agent write access to your repositories. Do not use it when:

- **You need read-only access** — use a read-only token or a separate server configuration with a restricted token to prevent accidental writes
- **The agent handles untrusted prompts** — an attacker could craft a prompt like "create an issue with the title `rm -rf /` on 100 repos" and the agent would comply
- **You need webhook event processing** — MCP is an agent-invoked protocol, not an event-driven one; use GitHub Actions or a webhook receiver for that
- **You only query public repos sporadically** — the unauthenticated GitHub API has a generous rate limit for simple lookups; avoid the auth overhead

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | GitHub API operations wrapped as typed MCP tools |
| **When to use** | Agent needs to read repos, manage issues, or review PRs |
| **When NOT to use** | Read-only queries, untrusted user prompts, webhook processing |
| **Mental model** | A programmable GitHub CLI that never makes syntax errors |

## Common Mistakes

### Mistake 1: Using a Token with Excess Scopes

**Mistake:** Generating a classic token with the full `repo` scope when only public-repo reads are needed.

**Why it happens:** The setup instructions say "create a token" and most tutorials default to the broadest scope.

**Symptoms:** The agent can delete repositories, modify branch protections, and push to protected branches — all through a single token.

**Fix:** Use fine-grained tokens scoped to specific repositories with the minimum permissions needed. For read-only use cases, select only `Contents: Read` and `Issues: Read`.

### Mistake 2: Hitting the GitHub API Rate Limit

**Mistake:** Making 60+ unauthenticated requests per hour or 5000+ authenticated requests per hour in a burst.

**Why it happens:** The agent calls `search_code`, `list_issues`, and `get_contents` in rapid succession without rate-limit awareness.

**Symptoms:** Tools start returning `403` or `429` status codes, and the agent's behavior degrades.

**Fix:** The official server handles retries on `429` responses. For custom servers, implement exponential backoff:

```python
import time
import random

async def github_request(client, url, headers, max_retries=3):
    for attempt in range(max_retries):
        resp = await client.get(url, headers=headers)
        if resp.status_code == 429:
            wait = 2 ** attempt + random.uniform(0, 1)
            time.sleep(wait)
            continue
        resp.raise_for_status()
        return resp
```

### Mistake 3: Not Handling Pagination

**Mistake:** Only reading the first page of results from `list_issues` or `search_code`.

**Why it happens:** The default GitHub API returns 30 results per page. The agent reads those 30, thinks it has all the data, and draws conclusions.

**Symptoms:** The agent reports "no open issues" when there are 50, or "no matches found" when results exist on page 3.

**Fix:** The official server handles pagination. For custom servers, follow the `Link` header or use `per_page=100` and iterate through pages.

### Mistake 4: Committing Malformed Git Objects

**Mistake:** Using `create_or_update_file` with incorrect base64 encoding or wrong SHA values.

**Why it happens:** The agent constructs the commit without knowing the current file SHA, so the write fails with `422`.

**Symptoms:** "Error: sha does not match" when trying to update a file.

**Fix:** Always call `get_contents` first to retrieve the current SHA, then pass it to `create_or_update_file`. The official server enforces this pattern for you.

### Mistake 5: Exposing the Token in Tool Output

**Mistake:** The token appears in logs, responses, or error messages returned to the agent.

**Why it happens:** A tool response may include a verbose error object from the GitHub API that echoes the request headers, including `Authorization`.

**Symptoms:** The agent prints the token in a response visible to the user.

**Fix:** Filter `Authorization` headers from error responses before returning them to the agent.
