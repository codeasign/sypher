# Build It: Configuring and Using the GitHub MCP Server

⏱️ 20 minutes

## Breaking It Down

### 1. Getting a GitHub Personal Access Token

The GitHub MCP server authenticates using a [GitHub personal access token (classic)](https://github.com/settings/tokens). Create one with these scopes:

- `repo` — Full control of private repositories (needed for private repos)
- `read:org` — Read org membership (optional, for org-scoped queries)

Navigate to **Settings > Developer settings > Personal access tokens > Tokens (classic)**, click **Generate new token (classic)**, select the scopes above, and copy the generated token. It starts with `ghp_`.

For public-repo-only access, you can use the more limited `public_repo` scope instead of `repo`.

### 2. Installing and Configuring the Server

Install the official package:

```bash
npm install -g @modelcontextprotocol/server-github
```

Configure it in your Claude Code `settings.json`. Set the `GITHUB_TOKEN` environment variable:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-github"
      ],
      "env": {
        "GITHUB_TOKEN": "ghp_your_token_here"
      }
    }
  }
}
```

For security, store the token in a `.env` file or your shell profile and reference it through the environment instead of hardcoding:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

Then set `GITHUB_TOKEN` in your shell profile (`~/.bashrc`, `~/.zshrc`, or Windows environment variables):

```bash
export GITHUB_TOKEN="ghp_your_token_here"
```

### 3. Available Tools Overview

Once connected, the GitHub MCP server exposes a comprehensive set of tools:

| Category | Tools |
|---|---|
| **Issues** | `create_issue`, `list_issues`, `get_issue`, `add_issue_comment`, `update_issue` |
| **Pull Requests** | `create_pull_request`, `list_pull_requests`, `get_pull_request`, `merge_pull_request`, `add_pull_request_comment`, `create_pull_request_review` |
| **Code** | `search_code`, `search_repositories`, `get_contents`, `create_or_update_file`, `push_files` |
| **Repos** | `list_repositories`, `get_repository`, `list_branches`, `create_branch`, `list_commits` |
| **Users** | `get_user`, `search_users`, `list_organization_members` |

Each tool has typed parameters defined in the MCP schema. When your agent calls `create_issue`, it knows it needs `owner`, `repo`, `title`, and optionally `body` and `labels`.

### 4. Real-World Usage Examples

**Search code across a repository:**

When you ask "find all TODO comments in the auth module," the agent calls:

```
search_code(q: "TODO repo:myorg/myapp path:auth/")
```

It receives matching file paths and line numbers, then uses the filesystem server to read the surrounding context.

**Create an issue from a bug report:**

Ask "file an issue for the login timeout bug in myorg/myapp":

```python
# The agent constructs this tool call:
create_issue(
    owner="myorg",
    repo="myapp",
    title="Login timeout does not trigger after 30 seconds",
    body="## Steps to reproduce\n1. Go to /login\n2. Enter credentials\n3. Wait 35 seconds\n\nExpected: timeout error after 30s\nActual: page hangs indefinitely",
    labels=["bug", "auth"]
)
```

**Review a pull request:**

Ask "review PR #142 in myorg/myapp for security issues":

The agent calls `get_pull_request` to get the diff, then `search_code` or `get_contents` to read the changed files, and finally `create_pull_request_review` to post a review with inline comments.

### 5. Building a Custom GitHub Server with the MCP SDK

If you need custom logic — comment templates, approval workflows, repository-specific permissions — build your own using the MCP Python SDK:

```python
# github_server.py
import os
from mcp import Server
from mcp.types import TextContent
import httpx

GITHUB_TOKEN = os.environ["GITHUB_TOKEN"]
API_BASE = "https://api.github.com"

server = Server("custom-github")

@server.tool("create_issue")
async def create_issue(owner: str, repo: str, title: str, body: str = "") -> TextContent:
    """Create a GitHub issue in the specified repository."""
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            f"{API_BASE}/repos/{owner}/{repo}/issues",
            json={"title": title, "body": body},
            headers={
                "Authorization": f"token {GITHUB_TOKEN}",
                "Accept": "application/vnd.github.v3+json",
            },
        )
    resp.raise_for_status()
    data = resp.json()
    return TextContent(
        type="text",
        text=f"Issue created: #{data['number']} — {data['html_url']}"
    )

@server.tool("search_code")
async def search_code(query: str) -> TextContent:
    """Search GitHub code with a query string."""
    async with httpx.AsyncClient() as client:
        resp = await client.get(
            f"{API_BASE}/search/code",
            params={"q": query, "per_page": 10},
            headers={
                "Authorization": f"token {GITHUB_TOKEN}",
                "Accept": "application/vnd.github.v3+json",
            },
        )
    resp.raise_for_status()
    items = resp.json().get("items", [])
    results = [
        f"{item['path']} — {item['html_url']}"
        for item in items
    ]
    return TextContent(
        type="text",
        text="\n".join(results) if results else "No results found"
    )
```

Run it:

```bash
GITHUB_TOKEN=ghp_xxx python github_server.py --transport stdio
```

## Build Something Real

Build an automated issue triage agent. Configure both the filesystem and GitHub servers, then ask:

> Look at the README.md in this project, find any outdated installation instructions, create a GitHub issue for each one with the "documentation" label, and link to the relevant section.

The agent reads the README through the filesystem server, identifies stale instructions, and calls `create_issue` through the GitHub server for each finding — all in one interaction.

## Security Considerations

Your GitHub token determines what the agent can do. A token with `repo` scope grants full write access to all repositories the token owner can access. Mitigate this by:

- Using a token scoped to the minimum set of repos via a GitHub App installation token or fine-grained token
- Setting the token's expiration to 30 or 90 days
- Never sharing the token value through MCP tool responses — the agent could include it in an issue body
- Reviewing token usage through the GitHub Security log
