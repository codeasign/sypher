# Building a GitHub MCP Server

⏱️ 15 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Configure and run the official GitHub MCP server
- Authenticate using a GitHub personal access token
- Give your agent tools to read repos, manage issues, and review PRs
- Scope the server's permissions to specific repositories

## Prerequisites

You should already understand:

- How MCP servers are configured in `settings.json` (see [Filesystem Server](../filesystem-server/overview))
- Git and GitHub basics — repositories, branches, pull requests, issues
- How personal access tokens work for API authentication
- MCP tools and how agents invoke them (see [Agent Tools and Tool Calling](../../agent-tools-and-tool-calling/overview))

## Why This Matters

A coding agent that can read and write files is useful. A coding agent that can open issues, review pull requests, merge branches, and query the GitHub API is transformative. The GitHub MCP server turns your agent into a full participant in your development workflow — it can triage bugs, suggest code changes as PR comments, run automated code reviews, and even create releases.

Engineering teams at companies like GitHub, GitLab, and Microsoft use AI-powered bots that operate directly on pull requests and issues. The GitHub MCP server makes this capability available to any agent you build, without writing custom API wrappers. You configure it once and your agent understands your entire GitHub workflow.

The real power comes from combining the GitHub server with the filesystem server. Your agent reads the codebase through the filesystem server to understand the context, then opens a PR or files an issue through the GitHub server with the appropriate changes. Two servers working together enable workflows that would require multiple custom scripts.

## The Problem

An agent that can only read and write files cannot interact with your team's collaboration tools. If a developer asks their agent "what open issues mention the authentication module?", the agent needs to query the GitHub Issues API. If they ask "review the changes in PR #42", the agent needs to read the PR diff, inspect the changed files, and post comments.

Without a GitHub MCP server, you would need to give the agent a generic HTTP tool and teach it the GitHub REST API schema, endpoint URLs, and authentication headers. This is fragile, verbose, and error-prone. Every API change requires prompt updates, and the agent often gets the endpoint wrong.

## The Concept

A GitHub MCP server wraps the GitHub REST API and GraphQL API into discrete MCP tools. Each GitHub operation — creating an issue, listing pull requests, searching code, merging a branch — becomes a single tool call with typed parameters. The server handles authentication, pagination, rate limiting, and error handling so the agent only needs to specify what it wants to do.

Think of it as a distilled GitHub API: instead of teaching your agent the shape of a GitHub API response and hoping it constructs the right HTTP request, you give it tools like `create_issue(title, body, labels)` and `get_pull_request(owner, repo, number)`. The agent gets the schema from the tool definition and calls it correctly every time.
