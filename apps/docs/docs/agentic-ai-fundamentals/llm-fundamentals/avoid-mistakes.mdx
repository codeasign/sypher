# LLM Fundamentals — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Understanding LLM internals is necessary for most AI engineering work, but there are cases where the mental model matters less:

- If you are using a high-level orchestration framework (LangChain, Vercel AI SDK) and the framework handles parameters, you still benefit from understanding them but do not need to tune every one
- If you are using a managed API with preset defaults and the defaults work for your use case, do not over-engineer parameters you do not need to touch
- If your task uses embedding models rather than generative models, parameters like temperature and top-p do not apply

## The Pattern to Remember

**Pattern**: Start deterministic, add creative only where needed. Default to `temperature=0` for any task where output must be consistent.

When to use:
- Classification, extraction, structured output: temperature 0
- Content generation, brainstorming, creative writing: temperature 0.5 – 1.0
- Code generation: temperature 0 – 0.2

When NOT to use:
- High temperature on tasks requiring precision (parsing, math, structured data)
- Low temperature on tasks that genuinely need variety (naming, ideation)
- Top-p below 0.9 with temperature above 0.7 (overly restrictive)

**Mental model**: Temperature selects how flat the probability distribution is. Lower temperature = more conservative, safer, repetitive. Higher temperature = more surprising, riskier, varied.

## Common Mistakes

### Mistake 1: Setting Temperature Too High for Structured Tasks

**Why it happens**: You read that higher temperature produces more creative output and assume creativity helps everything.

**Symptoms**: The model occasionally changes its output format, adds extra text, or invents new categories in a classification task.

**Fix**: Use `temperature=0` for any task where format or categories must be predictable.

### Mistake 2: Confusing Max Tokens with Context Window Length

**Why it happens**: Both parameters involve token counts, and the API error messages about token limits can be confusing.

**Symptoms**: You set `max_tokens=4096` but the model has only a 4096-token context window, leaving no room for the prompt.

**Fix**: Remember that `max_tokens` controls output length only. The context window covers input plus output together. Leave room for your response when building long prompts.

### Mistake 3: Not Checking the Stop Reason

**Why it happens**: The response looks complete, so you do not inspect the metadata.

**Symptoms**: Outputs are silently truncated in production. The model stopped because it hit `max_tokens`, but your code treats the response as complete.

**Fix**: Always check `response.stop_reason` in production code. If it is `"max_tokens"`, the response was cut off — either increase the limit or split the work.

### Mistake 4: Expecting the Same Output Across Different Models

**Why it happens**: Switching from a small model to a large one (or between providers) while keeping the same prompt and parameters.

**Symptoms**: The outputs differ significantly in style, verbosity, and accuracy despite identical settings.

**Fix**: Treat every model as a unique inference engine. Re-tune temperature, system prompt, and examples when changing models, even within the same family.

### Mistake 5: Thinking Higher Max Tokens Means Better Quality

**Why it happens**: It seems intuitive that longer responses are more thorough.

**Symptoms**: The model rambles, repeats itself, or includes irrelevant information to fill the token budget.

**Fix**: Set `max_tokens` to the minimum you need for the task. For a classification task, 10 tokens is enough. For an explanation, 200-500 is usually sufficient. Shorter output budgets force the model to be concise.
