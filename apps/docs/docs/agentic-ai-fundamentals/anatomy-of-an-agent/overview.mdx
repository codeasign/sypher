# Anatomy of an Agent

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Identify the five core components of an agent system
- Explain how the orchestrator manages the agent loop
- Describe how tools, memory, and the model interact
- Design the separation of concerns between components

## Prerequisites

You should already understand:

- What an agent is (covered in [What Is an Agent](../what-is-an-agent/overview))
- How agents differ from chatbots and workflows (covered in [Agent vs. Chatbot vs. Workflow](../agent-vs-chatbot-vs-workflow/overview))
- How tool calling works in the Anthropic or OpenAI API (covered in [LLM APIs in Practice](../llm-apis-in-practice/overview))

## Why This Matters

Building an agent without understanding its anatomy is like building a car without knowing what a transmission does. You might get something that moves, but you will not know why it breaks, how to fix it, or how to make it more efficient.

Production agent systems at companies like Anthropic, Microsoft, and Google share the same fundamental components. Understanding these components lets you read their architectures, borrow their patterns, and debug your own systems when something goes wrong. When your agent starts acting erratically, knowing which component is failing вҖ” the model, the tools, or the orchestrator вҖ” cuts debugging time from hours to minutes.

## The Problem

You build a research assistant agent. You give it a search tool and a web-scraping tool. You tell it to "research the latest climate science papers and summarize key findings."

The agent calls the search tool, gets results, scrapes two pages, produces a summary. It works. Then you add a second tool вҖ” a PDF parser вҖ” and the agent stops working. It calls the PDF parser on web pages, ignores search results entirely, and produces a summary based on one PDF.

You do not know why the behavior changed. You cannot tell if the problem is the model, the tool definitions, the orchestrator loop, or the memory system. You need a clear model of the agent's components to diagnose and fix this.

## The Concept

Every agent system has five core components:

**Model**. The LLM that drives the agent. It receives prompts, tool definitions, and conversation history, and produces responses that may include tool call requests. The model is the reasoning engine.

**Tools**. Functions the agent can call. Each tool has a definition вҖ” name, description, input schema вҖ” and an implementation that executes when the agent requests it. Tools are how the agent interacts with the outside world.

**Memory**. The storage layer that preserves context across steps. At minimum, this is the conversation history in the model's context window. For longer-running agents, memory extends to external stores like vector databases or key-value caches.

**Orchestrator**. The loop that manages the agent's execution. It sends the current state to the model, processes the model's response (which may contain zero or more tool calls), executes tool calls, updates memory, and decides whether to continue or stop.

**Safety layer**. Guards that prevent the agent from doing harm: rate limits, timeouts, human approval gates, output validators, and circuit breakers.
