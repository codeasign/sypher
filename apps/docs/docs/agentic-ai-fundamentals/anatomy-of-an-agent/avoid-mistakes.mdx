# Anatomy of an Agent — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Breaking an agent into its five components (model, tools, memory, orchestrator, safety) is an architectural pattern. It is overkill for:

- **Prototypes and experiments**: A single-file agent script is fine for learning. Component separation adds complexity before you need it.
- **Single-tool agents**: If your agent only calls one tool, the full component architecture is unnecessary overhead.
- **Read-only agents**: An agent that only retrieves data and never writes or acts does not need the full safety layer.

## The Pattern to Remember

**Pattern**: Five components — model, tools, memory, orchestrator, safety — each with a clear responsibility.

When to use full component separation:
- Agents with 3+ tools
- Production deployments requiring testing and monitoring
- Systems where different team members own different components

When NOT to use:
- Quick prototypes
- Single-tool experiments
- Learning exercises

**Mental model**: The orchestrator is the CPU. The model is the decision-maker. Tools are the peripherals. Memory is the RAM. Safety is the fuse box.

## Common Mistakes

### Mistake 1: Tightly Coupling the Orchestrator and Model

**Why it happens**: You build the orchestrator and model call in the same function with no abstraction layer.

**Symptoms**: Swapping the model (e.g., from Haiku to Sonnet) requires rewriting the orchestrator. Testing the loop without calling the API is impossible.

**Fix**: Abstract the model call behind an interface. The orchestrator should not know which model it is calling.

### Mistake 2: Putting Business Logic in Tool Descriptions

**Why it happens**: You write long tool descriptions that try to encode business rules instead of putting those rules in the implementation.

**Symptoms**: The tool descriptions are paragraphs long. The model still misuses the tool. Changing business logic requires updating prompts.

**Fix**: Tool descriptions should describe what the tool does, not how to use it correctly. Enforce rules in the implementation code.

### Mistake 3: Memory as an Afterthought

**Why it happens**: You build the agent loop first, then realize the context is overflowing and add memory as a patch.

**Symptoms**: The agent forgets information from earlier turns. The context window fills with irrelevant data. Adding memory later requires restructuring the message format.

**Fix**: Design the memory strategy before writing the orchestrator. The memory interface determines how messages flow through the loop.

### Mistake 4: Safety Layer That Can Be Bypassed

**Why it happens**: The safety check is in the tool handler itself rather than in a separate layer that intercepts all calls.

**Symptoms**: A new tool added later forgets to include safety checks. The safety logic is duplicated across handlers.

**Fix**: Safety must be a single layer that intercepts every tool call before it reaches any handler. One function, one responsibility.

### Mistake 5: No Monitoring on the Orchestrator

**Why it happens**: The orchestrator is the core loop, so you focus on making it work rather than making it observable.

**Symptoms**: When the agent fails, you have no idea which turn it was on, what the model said, or what tool it called.

**Fix**: Add logging to every orchestrator step: turn number, tool called, result status, token count. This data is essential for debugging.
