# What Is an Agent? — Build It

*Reading time: 12 minutes*

## Breaking It Down

### 1. A Plain LLM Call — Text Only

Start with the baseline: a standard LLM call that generates text but takes no action.

```python
import anthropic

client = anthropic.Anthropic()


def plain_llm_call(user_input: str) -> str:
    """A standard LLM call. It generates text but cannot act."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=1000,
        temperature=0,
        system="You are a helpful assistant.",
        messages=[{"role": "user", "content": user_input}]
    )
    return response.content[0].text


# The model can describe what it would do, but it cannot actually do anything
result = plain_llm_call("What is the weather in Tokyo?")
print(result)
# Output: "I don't have access to real-time weather data..."
```

The model knows it cannot look up weather. It tells you so. It has no tools, no ability to act — only text generation.

### 2. Adding a Single Tool

Now add a tool definition. The model can request a tool call, and your code executes it.

```python
def get_weather(city: str) -> str:
    """Simulated weather lookup tool."""
    weather_data = {
        "Tokyo": "18°C, partly cloudy",
        "London": "12°C, light rain",
        "New York": "22°C, sunny",
    }
    return weather_data.get(city, "Weather data not available for this city.")


def agent_with_tool(user_input: str) -> str:
    """An agent that can call a weather tool."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=1000,
        temperature=0,
        system="You are a helpful assistant with access to weather data.",
        messages=[{"role": "user", "content": user_input}],
        tools=[{
            "name": "get_weather",
            "description": "Get the current weather for a city",
            "input_schema": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name"}
                },
                "required": ["city"]
            }
        }]
    )
    return response.content[0].text


result = agent_with_tool("What is the weather in Tokyo?")
print(result)
# Output: "The current weather in Tokyo is 18°C, partly cloudy."
```

The model called the `get_weather` tool. Your code received the tool request, executed the function, and returned the result. The model then formatted the result into a natural language response. That is the essence of an agent.

### 3. A Complete Tool Loop

The single-call version above only handles one tool call. A real agent loops — it can call multiple tools, use results from one call as input to another, and decide when to stop.

```python
def agent_loop(user_input: str, max_turns: int = 5) -> list[dict]:
    """A minimal agent loop that handles multiple tool calls."""
    messages = [{"role": "user", "content": user_input}]
    turn_count = 0

    while turn_count < max_turns:
        response = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1000,
            temperature=0,
            system="You are a helpful assistant. Call tools when needed.",
            messages=messages,
            tools=[{
                "name": "get_weather",
                "description": "Get the current weather for a city",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "city": {"type": "string", "description": "City name"}
                    },
                    "required": ["city"]
                }
            }]
        )

        response_content = response.content[0]

        # Check if the model wants to call a tool
        if response_content.type == "tool_use":
            tool_name = response_content.name
            tool_input = response_content.input

            # Execute the tool
            if tool_name == "get_weather":
                result = get_weather(tool_input["city"])
            else:
                result = f"Unknown tool: {tool_name}"

            # Append the tool result to the conversation
            messages.append({"role": "assistant", "content": response_content})
            messages.append({
                "role": "user",
                "content": [
                    {
                        "type": "tool_result",
                        "tool_use_id": response_content.id,
                        "content": result
                    }
                ]
            })
            turn_count += 1
        else:
            # Model is responding with text — we are done
            return response_content.text

    return "Max turns reached without final answer."


result = agent_loop("Compare the weather in Tokyo and London.")
print(result)
```

The loop keeps going as long as the model requests tools. When the model produces text instead of a tool call, the loop exits with the final answer.

## How It Works Internally

When you pass `tools` in the API request, the model's token generation is modified. The model can output special `tool_use` content blocks interleaved with text blocks. Each `tool_use` block contains a `name`, an `id`, and `input` matching the tool's schema.

Your orchestrator reads these blocks, executes the corresponding functions, and returns the results as `tool_result` blocks. The model then incorporates the results into its next prediction.

The key insight is that the model does not actually run any code. It generates a structured request to run code. Your application code executes the request. This separation is critical for security — you control which functions are available and what arguments are valid.

## Build Something Real

Build a multi-tool agent that handles weather lookups and currency conversion.

```python
import anthropic
from typing import Any


client = anthropic.Anthropic()


def get_weather(city: str) -> str:
    """Look up current weather for a city."""
    data = {
        "Tokyo": ("18°C", "partly cloudy"),
        "London": ("12°C", "light rain"),
        "New York": ("22°C", "sunny"),
        "Sydney": ("25°C", "clear"),
        "Paris": ("15°C", "cloudy"),
    }
    entry = data.get(city)
    if entry:
        temp, condition = entry
        return f"{city}: {temp}, {condition}"
    return f"Weather data not available for {city}."


def convert_currency(amount: float, from_currency: str, to_currency: str) -> str:
    """Convert an amount from one currency to another."""
    rates = {
        "USD": 1.0, "EUR": 0.92, "GBP": 0.79,
        "JPY": 149.50, "AUD": 1.53, "CAD": 1.36,
    }
    if from_currency not in rates or to_currency not in rates:
        return f"Currency not supported. Available: {', '.join(rates.keys())}"
    usd_amount = amount / rates[from_currency]
    converted = round(usd_amount * rates[to_currency], 2)
    return f"{amount} {from_currency} = {converted} {to_currency}"


TOOLS = [
    {
        "name": "get_weather",
        "description": "Get the current weather for a city",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name"}
            },
            "required": ["city"]
        }
    },
    {
        "name": "convert_currency",
        "description": "Convert an amount from one currency to another",
        "input_schema": {
            "type": "object",
            "properties": {
                "amount": {"type": "number", "description": "Amount to convert"},
                "from_currency": {"type": "string", "description": "Currency code (e.g., USD)"},
                "to_currency": {"type": "string", "description": "Target currency code"}
            },
            "required": ["amount", "from_currency", "to_currency"]
        }
    }
]


def execute_tool(name: str, input_data: dict[str, Any]) -> str:
    """Dispatch tool calls to the correct function."""
    tool_map = {
        "get_weather": lambda: get_weather(input_data["city"]),
        "convert_currency": lambda: convert_currency(
            input_data["amount"],
            input_data["from_currency"],
            input_data["to_currency"]
        ),
    }
    func = tool_map.get(name)
    if not func:
        return f"Error: unknown tool '{name}'"
    try:
        return func()
    except Exception as e:
        return f"Error executing {name}: {e}"


def run_agent(user_input: str, max_turns: int = 10) -> str:
    """Run the agent loop with tool execution."""
    messages = [{"role": "user", "content": user_input}]

    for turn in range(max_turns):
        response = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1000,
            temperature=0,
            system="You are a helpful assistant with access to weather and currency tools.",
            messages=messages,
            tools=TOOLS,
        )

        content = response.content[0]

        if content.type == "tool_use":
            tool_result = execute_tool(content.name, content.input)
            messages.append({"role": "assistant", "content": content})
            messages.append({
                "role": "user",
                "content": [
                    {"type": "tool_result", "tool_use_id": content.id, "content": tool_result}
                ]
            })
        else:
            return content.text

    return "Agent did not produce a final answer within the turn limit."


# Test with a multi-step request
print(run_agent(
    "What is the weather in Sydney? Also convert 100 USD to JPY."
))
```

This is a working agent. It perceives input, decides which tools to call, executes them, and produces a synthesis. The orchestrator loop and tool dispatch form the foundation of every agent system you will build.

## Performance Considerations

Each tool call adds a round trip to the API. If your agent calls three tools sequentially, that is three API calls plus the final response. For latency-sensitive applications, design tools that batch multiple operations. A single `get_weather_batch` tool that accepts a list of cities is faster than three separate `get_weather` calls.

## Security Considerations

Tools execute arbitrary code on your system. Never expose tools that allow the model to execute shell commands, read arbitrary files, or make network calls without validation. Validate all tool arguments before passing them to the implementation function.
