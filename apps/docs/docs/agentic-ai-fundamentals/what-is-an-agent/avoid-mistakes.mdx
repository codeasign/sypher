# What Is an Agent? — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

An agent architecture is not always the right choice:

- **Single-response tasks**: If the task is a one-shot generation with no follow-up, a plain LLM call is simpler and cheaper.
- **Strictly deterministic operations**: If every request follows the same exact path, a workflow is more reliable.
- **Resource-constrained environments**: Agents require multiple API calls, which add cost and latency.
- **Tasks requiring zero hallucination**: Agents introduce more degrees of freedom, increasing the chance of unexpected behavior.
- **When you cannot afford failures**: Agents fail differently than direct calls — they can get stuck in loops, call wrong tools, or go off-track.

## The Pattern to Remember

**Pattern**: An agent perceives, reasons, acts, and remembers in a loop. The LLM decides the sequence of actions within the boundaries you set.

When to use:
- Multi-step tasks requiring tool calls
- Tasks where the exact sequence is not known in advance
- Applications that benefit from the model choosing the next action

When NOT to use:
- Simple Q&A, single API call tasks
- Tasks with a fixed, known sequence
- Safety-critical systems where unpredictability is unacceptable

**Mental model**: An agent is an LLM driving a loop. Without the loop, it is just an API call. Without tools, it cannot act. Without memory, it cannot learn.

## Common Mistakes

### Mistake 1: Calling a Chatbot an Agent

**Why it happens**: Any conversational LLM interface gets labeled an "agent" in marketing and demos.

**Symptoms**: The system has no tools, no loop, and no memory beyond the conversation history — but it is called an agent.

**Fix**: Use precise terminology. An agent must have autonomy, tool use, and memory. Without all three, it is a chatbot or a workflow.

### Mistake 2: No Turn Limit

**Why it happens**: You assume the model will eventually produce a final answer and stop.

**Symptoms**: The agent calls tools indefinitely, consuming tokens and never producing output. The API bill grows without bound.

**Fix**: Always set a `max_turns` parameter. When the limit is reached, return a graceful error message instead of continuing.

### Mistake 3: Forgetting to Return Tool Results

**Why it happens**: You execute the tool but do not append the result to the message list.

**Symptoms**: The agent calls the same tool repeatedly with the same arguments. Each call gets the same response because the result was never fed back in.

**Fix**: After executing a tool, always append a `tool_result` message with the correct `tool_use_id` before the next API call.

### Mistake 4: Putting Everything in One Tool

**Why it happens**: Creating many small tools feels like more work, so you create one `do_everything` tool with many optional parameters.

**Symptoms**: The model struggles to fill in the parameters correctly. It leaves most fields empty or fills them with wrong values.

**Fix**: Each tool should do one thing and do it well. If a tool has more than 5 required parameters, split it.

### Mistake 5: No Error Handling in Tool Execution

**Why it happens**: You assume the tool will always succeed, or you let exceptions propagate.

**Symptoms**: A tool call fails with an unhandled exception, the agent loop crashes, and the user gets no response.

**Fix**: Wrap every tool execution in try/except. Return a descriptive error message as the tool result so the model can decide how to recover.
