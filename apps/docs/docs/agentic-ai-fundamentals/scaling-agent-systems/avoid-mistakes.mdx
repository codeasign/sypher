# Avoid Mistakes: Scaling Pitfalls

*Reading time: 10 minutes*

## When NOT to Use This

Scaling patterns introduce infrastructure complexity that is not always justified. Avoid building a distributed agent architecture when:

- **Low-traffic internal agents.** If your agent serves a single team of fewer than ten people and does not need 24/7 availability, a single-instance deployment behind a basic reverse proxy is sufficient. The Redis cluster, Celery workers, and auto-scaling automation will add maintenance overhead with no practical benefit.

- **Prototyping phases.** During early development, you are iterating on prompts, tools, and agent behavior. Adding scaling infrastructure at this stage slows iteration. Run a single agent process locally, confirm the logic is correct, then add scaling when you are ready for user testing.

- **Fixed-capacity batch jobs.** If your agent processes a known number of tasks on a schedule (nightly report generation, daily data extraction), provision exactly as many workers as needed and skip the auto-scaling logic. Scaling up and down adds unnecessary complexity when workload is predictable.

- **Systems with dedicated hardware.** If your agent runs on a single powerful machine (GPU workstation, edge device), horizontal scaling is not applicable. Optimize vertical scaling instead — increase memory, use async I/O, and rely on multiprocessing to handle concurrent requests within the single machine.

## The Pattern to Remember

| Aspect | Stateless (Scalable) | Stateful (Not Scalable) |
|---|---|---|
| Session storage | External (Redis, database) | In-process memory |
| Instance affinity | None required | Required (sticky sessions) |
| Horizontal scaling | Add instances freely | Not possible |
| Failure recovery | Any instance picks up | Session lost on failure |
| Deployment complexity | Higher | Lower |

**When to use the stateless pattern:** Any multi-user agent system expected to serve more than a handful of concurrent users.

**When NOT to use the stateless pattern:** The scenarios listed above — low-traffic internal agents, prototypes, fixed batch jobs, and dedicated hardware.

**Mental model:** Think of your agent instances as stateless workers in a factory. The factory floor changes workers between shifts without losing progress because the assembly instructions and parts inventory are stored in a central system. Your agent instances change between sessions without losing context because memory and conversation history live in the shared store.

## Common Mistakes

### Mistake 1: Ignoring LLM Rate Limits

**Why it happens:** You test with a single user, never hit the rate limit, and assume it will not be an issue at scale.

**Symptoms:** Under load, the LLM API starts returning HTTP 429 errors. Some requests fail silently. Others retry in a tight loop, amplifying the problem. The agent appears to be working but returns empty or truncated responses under load.

**Fix:** Implement a distributed rate limiter (token bucket in Redis, as shown in Build It) that all instances share. Monitor API response codes and track rate limit headroom. Configure exponential backoff with jitter for all API retries.

### Mistake 2: Assuming Agents Are Naturally Stateless

**Why it happens:** Your single-instance agent stores session data in a module-level dictionary or class instance variable. It works fine when one user tests it, so you assume it will work with many users.

**Symptoms:** User A's message gets answered with User B's conversation context. Tool results get mixed between sessions. The agent seems confused, answering questions the current user never asked.

**Fix:** Never store per-session data in global or instance variables. Every turn must load its context from an external store. Validate that two concurrent sessions produce correct, isolated results before deploying to production.

### Mistake 3: Shared State Race Conditions

**Why it happens:** Two agent instances process turns for the same session simultaneously. Instance A loads the conversation history, adds its turn, and saves. Instance B does the same at the same time, but it loaded the history before A saved. Instance B's save overwrites A's turn, and the conversation loses a message.

**Symptoms:** Intermittent message loss. A user sends a message, gets a response, sends another, and the first response disappears. The conversation history has gaps that are hard to reproduce.

**Fix:** Use optimistic locking with version counters. Include a version field in the session state. Before saving, check that the version has not changed since you loaded it. If it has, reload and reapply the changes. Alternatively, use append-only log structures where each turn is a new entry rather than overwriting the entire history.

```python
def save_session_with_lock(session_id: str, messages: list, expected_version: int) -> bool:
    """Save session state with optimistic locking."""
    key = f"session:{session_id}"
    current = redis_client.hget(key, "version")
    if current is not None and int(current) != expected_version:
        return False  # Conflict — reload and retry
    redis_client.hset(key, mapping={
        "messages": json.dumps(messages),
        "version": expected_version + 1,
    })
    return True
```

### Mistake 4: Not Planning for Cold Starts

**Why it happens:** You configure auto-scaling to spin up new instances when queue depth increases, but you do not account for the time it takes for a new instance to become ready.

**Symptoms:** Under a traffic spike, the auto-scaler detects the queue growing and launches new instances, but those instances take 30-60 seconds to initialize (download dependencies, connect to Redis, warm the cache). During that window, response times degrade further and users experience timeouts.

**Fix:** Keep a buffer of idle instances above your baseline traffic. Pre-warm new instances before they are needed by monitoring leading indicators (request rate, not just queue depth). Use container orchestrators that support readiness probes so the load balancer does not route traffic to instances that are still initializing.

### Mistake 5: Scaling Without Monitoring Latency

**Why it happens:** You add more instances, throughput increases, and you assume everything is fine. You do not measure the end-to-end latency that users actually experience.

**Symptoms:** Average response time drifts upward as more instances compete for the same Redis connection pool and database connections. LLM API calls that took 3 seconds now take 5 seconds because of contention in the shared session store. The system handles more users, but each individual user gets a worse experience.

**Fix:** Monitor p50, p95, and p99 latency for every component: Redis read/write, LLM API call, serialization, and total request time. Set alerts on p95 latency. When latency increases, investigate before adding more instances — the bottleneck may be the shared store, not the agent workers. Scale the infrastructure layer (Redis cluster, database read replicas) alongside the agent instances.
