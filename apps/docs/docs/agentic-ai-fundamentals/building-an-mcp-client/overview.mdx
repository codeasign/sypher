# Building an MCP Client

*Reading time: 6 minutes*

**Difficulty:** Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Build an MCP client that connects to any MCP server over stdio transport
- Discover tools and resources from a server after initialization
- Call tools and read resources through the MCP protocol
- Integrate MCP tools into an agent's tool-calling loop
- Handle multiple MCP servers from a single client

## Prerequisites

You should already understand:

- How MCP servers work (covered in [Building an MCP Server](../building-an-mcp-server/overview))
- The MCP lifecycle and transport options (covered in [MCP Architecture](../mcp-architecture/overview))
- How agents call tools through an LLM API (covered in [Agent Tools and Tool Calling](../agent-tools-and-tool-calling/overview))
- Python async programming

## Why This Matters

A server without a client is just a process. The MCP client is what bridges the gap between the MCP protocol and the agent's tool-calling loop. Building an MCP client means you can connect any agent to any MCP server, regardless of who built the server or what tools it provides.

This is the skill that unlocks the entire MCP ecosystem. Once you have a client, you can connect to filesystem servers, database servers, API gateways, code analysis tools, and any other MCP server that exists in the ecosystem. You are no longer limited to the tools you built yourself — you can use any MCP server published by the community.

In production, MCP clients are typically embedded in agent frameworks. Claude Code, for example, has a built-in MCP client that connects to configured servers at startup. When you build a custom agent, you add an MCP client so your agent can use the same ecosystem of tools.

## The Problem

You have built an MCP server that provides file search and text search tools. Now you need your custom agent to use these tools. The agent's tool-calling loop expects tool definitions in a specific format (JSON Schema in the LLM API format). The MCP server sends tools in MCP format. The formats are different.

Your client must:

1. Connect to the server and initialize
2. Fetch tool definitions and translate them to your agent's format
3. Intercept tool calls from the agent, translate them to MCP format, and send them to the server
4. Translate MCP responses back to the agent's expected format

This translation layer is the core of the MCP client. Without it, your agent cannot use MCP servers.

## The Concept

An MCP client is a program that connects to an MCP server, negotiates capabilities, discovers tools and resources, and translates between the MCP protocol and the agent's tool-calling interface.

The client has three responsibilities:

1. **Connection management** — spawn the server process (stdio) or connect via HTTP (SSE), run the initialization handshake, handle reconnection
2. **Tool discovery and translation** — call `tools/list`, convert MCP tool definitions to the agent's tool format (e.g., Anthropic SDK `tools` parameter format)
3. **Request routing** — when the agent decides to call a tool, translate the call to MCP's `tools/call`, send it, and translate the response back

A well-designed MCP client also handles resources and prompts, translating them into formats the agent can use directly (resources as context, prompts as system messages).
