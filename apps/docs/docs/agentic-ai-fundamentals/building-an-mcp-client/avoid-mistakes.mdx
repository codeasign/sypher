# Building an MCP Client — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Building a custom MCP client is not always the best approach:

- **Using Claude Code**: Claude Code has a built-in MCP client — you configure servers in `mcp_servers` and it handles discovery automatically. No custom client needed.
- **Using Claude Desktop**: Claude Desktop also has a built-in MCP client. You configure servers in the app settings.
- **One-off tool calls**: If you only need to call a single tool once, using the MCP Inspector or a raw `subprocess` call is simpler than building a full client.
- **Synchronous scripts**: The MCP client SDK is async. If you cannot use async Python, consider using a synchronous wrapper or direct JSON-RPC over subprocess pipes.

## The Pattern to Remember

**Pattern**: The MCP client translates between MCP protocol (tools/list, tools/call) and the agent's tool-calling interface. It manages connection, discovery, and routing.

When to use:
- Custom agents that need MCP server access
- Multi-server tool aggregation
- Agents running in custom environments (web apps, services)

When NOT to use:
- Claude Code or Claude Desktop (built-in MCP client)
- One-off tool calls
- Synchronous-only environments

**Mental model**: The MCP client is a translation layer — MCP protocol on one side, agent API format on the other. Both sides speak JSON, but the schemas differ.

## Common Mistakes

### Mistake 1: Not Closing Sessions Cleanly

**Why it happens**: The client spawns server processes but does not terminate them when the client exits.

**Symptoms**: Orphaned server processes accumulate. Each one holds file handles and memory. Eventually the system runs out of resources.

**Fix**: Always use context managers (`async with`) or explicit cleanup in `finally` blocks. The `MultiServerMcpClient` example above tracks all contexts for orderly teardown.

### Mistake 2: Tool Name Conflicts Between Servers

**Why it happens**: Two servers each provide a tool named `search`. The client merges their tool lists without deduplication.

**Symptoms**: The agent has two tools named `search` with different schemas. The LLM is confused about which to call. The routing map sends the call to the wrong server.

**Fix**: Prefix tool names with the server name during translation (e.g., `fileserver_search` and `db_search`). Maintain a reverse mapping for routing.

### Mistake 3: Blocking the Event Loop

**Why it happens**: The client uses `time.sleep()`, synchronous HTTP calls, or CPU-bound operations in async handlers.

**Symptoms**: The client becomes unresponsive. Session timeouts fire. Tool calls fail with unexpected timeouts.

**Fix**: Use async equivalents for everything. The MCP SDK is fully async — match that. Use `asyncio.to_thread()` for blocking operations, or use the `httpx.AsyncClient` for HTTP calls.

### Mistake 4: Assuming All Servers Support All Capabilities

**Why it happens**: After implementing tool support, you try `session.list_resources()` and get an error because the server does not declare resource capabilities.

**Symptoms**: Method-not-found errors for capabilities the server never advertised.

**Fix**: Check the server's declared capabilities from the initialize response. Only call methods for capabilities the server supports. This is especially important when building a generic client that connects to unknown servers.

### Mistake 5: Passing Non-JSON-Serializable Arguments

**Why it happens**: The agent generates tool arguments that include non-serializable types (datetime objects, nested classes, bytes).

**Symptoms**: The MCP session serialization raises a `TypeError`. The tool call fails before reaching the server.

**Fix**: Validate that all tool arguments are JSON-serializable before passing them to `session.call_tool()`. Implement a recursive serializer for edge cases like datetimes (convert to ISO strings).
