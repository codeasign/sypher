# Building an MCP Client — Review

*Reading time: 3 minutes*

## Key Takeaways

- An MCP client connects to servers, discovers tools via `tools/list`, and routes agent tool calls via `tools/call`
- The client translates MCP tool definitions to the LLM API format — primarily renaming `inputSchema` to `input_schema`
- Use the MCP Python SDK's `ClientSession` with `stdio_client` for connection management
- For multi-server setups, prefix tool names with the server name to avoid conflicts (e.g., `fileserver_search`)
- Always clean up sessions with context managers — orphaned server processes accumulate and consume resources

## Recap in 60 Seconds

1. The MCP client does three things: connect and initialize, discover and translate tools, and route tool calls — the MCP SDK handles the protocol, you handle the translation
2. When connecting to multiple servers, watch for tool name collisions — prefix tool names with the server identifier
3. The entire client stack is async — use `asyncio` throughout; call blocking code through `asyncio.to_thread()`
