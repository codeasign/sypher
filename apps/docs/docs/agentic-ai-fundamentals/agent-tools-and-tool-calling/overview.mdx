# Agent Tools and Tool Calling

*Reading time: 6 minutes*

**Difficulty:** Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Define tools with accurate JSON schemas that LLMs can reliably invoke
- Implement tool dispatch with validation, error handling, and retries
- Design tool descriptions that help the model choose the right tool
- Handle parallel and sequential tool calls correctly

## Prerequisites

You should already understand:

- How tool calling works in the Anthropic or OpenAI API (covered in [LLM APIs in Practice](../llm-apis-in-practice/overview))
- How JSON Schema describes function parameters
- The role of tools in agent systems (covered in [Anatomy of an Agent](../anatomy-of-an-agent/overview))

## Why This Matters

Tools are how an agent touches the world. Every action your agent takes — searching a database, sending an email, creating a file — goes through a tool. The quality of your tool definitions directly determines the quality of your agent's actions. Poorly defined tools produce wrong actions. Well-defined tools make your agent reliable.

Production agents at companies like Klarna, Replit, and Notion use carefully designed tool schemas. They invest in tool descriptions that disambiguate similar tools, parameter validation that catches errors before they reach external systems, and error responses that help the model recover.

## The Problem

You define a tool called `search_database` with parameters `query` and `limit`. The agent calls it with `query="SELECT * FROM users WHERE name = 'Smith'"` and `limit=5000`. Your database returns 5000 rows. The agent's context window fills with data. The next call fails because there is no room in the context.

The problem is not the tool — it is the tool definition. You did not set a reasonable default for `limit`, you did not document that `query` should be a natural language description (not raw SQL), and you did not paginate the results. The tool definition left too much to the model's interpretation.

Good tool design constrains the model's choices so it picks the right tool with the right parameters every time.

## The Concept

Every tool has three parts:

- **Definition**: The name, description, and JSON Schema of the tool's input parameters. The model uses this to decide when to call the tool and with what arguments.
- **Implementation**: The actual function that runs when the tool is called. This is regular code — database queries, API calls, file operations.
- **Result**: The output returned to the model. The result becomes part of the agent's context and influences the next decision.

Tool calling in LLM APIs follows a structured flow:

1. The model receives the conversation history plus tool definitions
2. The model responds with either text or a tool use request (or both)
3. Your code executes the tool with the provided arguments
4. The tool result is returned to the model as a new message
5. The model continues with the result in context
