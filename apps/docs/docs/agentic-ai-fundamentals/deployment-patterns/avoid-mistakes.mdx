# Avoid Mistakes: Deployment Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Automated deployment pipelines bring speed and consistency, but they are not always the right choice. Consider manual deployment or a simplified setup when:

- **You are prototyping or in the ideation phase.** A full CI/CD pipeline with blue-green deployment adds overhead that slows down rapid experimentation. Until you have paying users or production traffic, a single Docker host with manual restarts is faster to iterate on.
- **Your agent runs on a single machine with no redundancy.** If downtime is acceptable and there is no load-balancing, a simple `git pull && docker compose up -d` suffices. The complexity of canary releases and health-check orchestration buys you nothing.
- **You are the only developer and the only operator.** Pipeline tooling and IaC make sense for teams; solo projects can skip them until the cost of manual deployment exceeds the setup cost.
- **Your agent has no external dependencies** (no LLM API, no databases, no tool binaries). The main deployment risks come from dependency mismatches and secret handling — without those, a basic Docker run is adequate.

## The Pattern to Remember

| Pattern | When to use | When NOT to use | Mental model |
|---|---|---|---|
| Blue-green | You need zero-downtime deploys and can run two copies side by side | Your infrastructure budget cannot double container costs during deploy | Swap the sign, not the store |
| Canary | You want to validate a new version on real traffic before full rollout | Your traffic volume is too low for statistical confidence (less than ~100 req/day) | Dip your toe before jumping |
| Rolling update | You want incremental rollout without extra capacity | Your agent holds long-running sessions that break on restart | Change tires while the car moves |
| Manual deploy | You are prototyping or alone on the project | You have users depending on uptime or a team that needs reproducible releases | Hand-crank the engine |

## Common Mistakes

### 1. Deploying Without Health Checks

You push a new image, the container starts, but the agent crashes during initialization because a tool dependency is missing or the API key is invalid. Without a health check, the orchestrator thinks the agent is healthy and routes traffic to a dead process.

**Fix:** Always define both `healthcheck` and `restart: unless-stopped` in your Docker Compose file, and implement a readiness endpoint that verifies LLM connectivity and tool availability, not just process liveness.

### 2. Ignoring Rollback Strategy

You deploy a breaking change and realize it five minutes later. Without a rollback plan, every second of downtime compounds. Teams without a strategy often scramble to revert code, rebuild, and redeploy — a process that takes twenty minutes when it should take twenty seconds.

**Fix:** Keep the last two known-good Docker image tags on your registry. Your pipeline should support a one-command rollback. In blue-green, rollback is just switching traffic back to blue.

### 3. Storing Secrets in Environment Files

The `.env` file is convenient for local development, but committing it to version control or baking it into a Docker image leaks your API keys permanently. Even a private repository is a risk — credentials in the git history never truly disappear.

**Fix:** Add `.env` to `.gitignore` immediately. Use GitHub Actions secrets, your cloud provider's secret manager, or a vault service. Inject secrets at runtime, never at build time.

### 4. Mismatching Tool Versions Across Environments

Your agent depends on a specific version of a local tool (a binary, a Python library, a model file). You upgrade the tool in your development Docker image but forget to update the production image. The agent starts, fails on the first tool call, and users see an opaque error.

**Fix:** Pin every dependency version in `requirements.txt` and your `Dockerfile`. Use the same base image tag (`python:3.11-slim`) across all environments. Your CI/CD pipeline should build one image and promote it unchanged through every stage.

### 5. No Session Persistence Strategy

You deploy a new version and every active user session resets. The user was three turns into a conversation with the agent — now they must start over. For stateful agent interactions, this destroys the user experience.

**Fix:** Externalize session state to Redis, PostgreSQL, or your cloud provider's key-value store. The agent loads session history on startup and saves it after each turn. A rolling update then migrates sessions seamlessly to the new version.
