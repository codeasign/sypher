# Build It: Deploying an Agent System

⏱️ 25 minutes

## Breaking It Down

### 1. Dockerizing an Agent

You start by packaging your agent into a Docker image. A well-structured `Dockerfile` installs only what the agent needs at runtime, keeps the image small, and runs the agent as a non-root user for security.

```dockerfile
# Dockerfile
# Multi-stage build to keep the production image lean

# ---- Build stage ----
FROM python:3.11-slim AS builder

WORKDIR /build

# Install build dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Copy dependency file and install into a local directory
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# ---- Runtime stage ----
FROM python:3.11-slim

# Create a non-root user for safety
RUN groupadd --gid 1000 agent && \
    useradd --uid 1000 --gid agent --create-home agent

WORKDIR /app

# Copy only the installed packages from the builder stage
COPY --from=builder /root/.local /home/agent/.local
ENV PATH=/home/agent/.local/bin:$PATH

# Copy the agent source code
COPY --chown=agent:agent src/ ./src/
COPY --chown=agent:agent agent.py .

# Switch to the non-root user
USER agent

# Health check endpoint — the agent exposes /health on this port
EXPOSE 8080

# Default command: start the agent server
CMD ["python", "agent.py"]
```

### 2. Environment Configuration

Agents need secrets (API keys), model configuration, and environment-specific settings. Use environment variables with a validated loading pattern so a missing variable fails early rather than silently at runtime.

```python
# src/config.py
"""Environment configuration loader for the agent system.

Uses pydantic-settings to validate all required environment variables
at startup. A missing variable raises a clear error immediately.
"""

from pydantic_settings import BaseSettings


class AgentConfig(BaseSettings):
    """All environment-derived configuration for the agent."""

    # LLM provider settings
    anthropic_api_key: str   # Required — agent will not start without it
    model_name: str = "claude-sonnet-4-20250514"

    # Deployment context
    environment: str = "development"       # development | staging | production
    log_level: str = "INFO"

    # Session management
    session_timeout_seconds: int = 300
    max_tool_calls_per_turn: int = 25

    # Health check
    health_check_port: int = 8080

    class Config:
        env_file = ".env"
        env_file_encoding = "utf-8"


# Singleton config loaded once at import time
config = AgentConfig()
```

```python
# agent.py
"""Entry point for the agent server.

Exposes a lightweight HTTP server for health checks and provides
the main agent loop. Run directly or through Docker.
"""

import os
import logging
from http.server import HTTPServer, BaseHTTPRequestHandler

from src.config import config
from src.agent_core import AgentCore

# Configure logging based on environment setting
logging.basicConfig(
    level=getattr(logging, config.log_level.upper(), logging.INFO),
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
)
logger = logging.getLogger(__name__)


class HealthCheckHandler(BaseHTTPRequestHandler):
    """Minimal HTTP handler that responds to /health and /ready."""

    def do_GET(self):
        """Route GET requests to health-check endpoints."""
        if self.path == "/health":
            # Liveness probe — agent process is alive
            self.send_response(200)
            self.end_headers()
            self.wfile.write(b'{"status": "alive"}')
        elif self.path == "/ready":
            # Readiness probe — agent is initialized and ready for sessions
            self.send_response(200)
            self.end_headers()
            self.wfile.write(b'{"status": "ready"}')
        else:
            self.send_response(404)
            self.end_headers()


def main():
    """Initialize the agent core and start the health-check server."""
    logger.info(
        "Starting agent in %s environment",
        config.environment,
    )

    # Validate that the API key is present (fail fast)
    if not config.anthropic_api_key:
        raise RuntimeError(
            "ANTHROPIC_API_KEY is not set. "
            "Create a .env file or set the environment variable."
        )

    # Initialize the agent's core logic
    agent = AgentCore(config)
    logger.info("Agent core initialized successfully")

    # Start the health check HTTP server on the configured port
    server = HTTPServer(("0.0.0.0", config.health_check_port), HealthCheckHandler)
    logger.info("Health check server listening on port %d", config.health_check_port)

    try:
        server.serve_forever()
    except KeyboardInterrupt:
        logger.info("Shutting down gracefully")
        server.shutdown()


if __name__ == "__main__":
    main()
```

### 3. Docker Compose for Local Development

Use `docker-compose.yml` to run your agent locally with the same settings it will use in production. This ensures parity between environments.

```yaml
# docker-compose.yml
# Runs the agent alongside any supporting services (database, cache, etc.)

version: "3.9"

services:
  agent:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    env_file:
      - .env
    environment:
      - ENVIRONMENT=development
      - LOG_LEVEL=DEBUG
    # Health check that the orchestrator uses to determine liveness
    healthcheck:
      test: ["CMD", "curl", "--fail", "http://localhost:8080/health"]
      interval: 15s
      timeout: 5s
      retries: 3
      start_period: 10s
    volumes:
      # Mount source code for hot-reloading during development
      - ./src:/app/src
      - ./agent.py:/app/agent.py
    restart: unless-stopped

  # Optional: local Redis for session storage
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```

### 4. CI/CD Pipeline with GitHub Actions

The pipeline below builds, tests, and deploys your agent. It implements a blue-green deployment strategy: it spins up the new version (green) alongside the current version (blue), runs smoke tests against green, then switches traffic.

```yaml
# .github/workflows/deploy-agent.yml
# CI/CD pipeline: lint → test → build → deploy (blue-green)

name: Deploy Agent

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Check out code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest ruff

      - name: Lint with Ruff
        run: ruff check src/

      - name: Run unit tests
        run: pytest tests/ -v

  build-and-push:
    needs: test
    # Only push images on merge to main, not on every PR
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - name: Check out code
        uses: actions/checkout@v4

      - name: Log in to container registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata for Docker
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=
            type=ref,event=branch

      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - name: Deploy via SSH to production host
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            # Pull the new image
            docker compose pull agent

            # Start the green (new) service on an alternate port
            docker compose -f docker-compose.yml \
              -f docker-compose.green.yml up -d agent-green

            # Wait for the green service to pass its health check
            sleep 15
            curl --fail http://localhost:8081/health || exit 1

            # Run smoke tests against the green service
            curl --fail http://localhost:8081/ready || exit 1

            # Swap traffic: reload the reverse proxy to point at green
            docker compose exec proxy nginx -s reload

            # Stop the blue (current) service
            docker compose stop agent-blue
```

### 5. Health Checks for Agents

Standard HTTP health checks are not enough for agents. Your readiness probe should also verify that the agent can reach its LLM provider and that all tool dependencies are available.

```python
# src/health.py
"""Advanced health checks for agent readiness verification.

Extends the basic liveness probe with dependency checks so that
the orchestrator knows when the agent is truly ready for traffic.
"""

import logging

import httpx
from src.config import config

logger = logging.getLogger(__name__)


async def check_llm_connectivity() -> bool:
    """Verify that the agent can reach the LLM API endpoint.

    Sends a lightweight request (token count, not a full generation)
    to confirm network connectivity and API key validity.
    """
    try:
        async with httpx.AsyncClient(timeout=10.0) as client:
            response = await client.get(
                "https://api.anthropic.com/v1/messages",
                headers={
                    "x-api-key": config.anthropic_api_key,
                    "anthropic-version": "2023-06-01",
                },
            )
            # A 401 means the key is invalid; a 200 or 400 means the API is reachable
            return response.status_code != 401
    except httpx.RequestError as exc:
        logger.warning("LLM connectivity check failed: %s", exc)
        return False


async def check_tool_dependencies() -> bool:
    """Verify that all tool dependencies are available on the filesystem."""
    # Example: check that a local tool binary exists
    import shutil

    required_tools = ["curl", "python3"]
    for tool in required_tools:
        if shutil.which(tool) is None:
            logger.warning("Missing tool dependency: %s", tool)
            return False
    return True
```

## Performance Considerations

- **Cold start**: Docker images for agent systems typically have warmup time for model downloads, tool initialization, and connection pooling. Your health check `start_period` should account for this (30-60 seconds).
- **Session migration**: When you roll out a new version, active sessions on the old version are lost unless you externalize session state (Redis, PostgreSQL). Consider whether your agent can tolerate mid-session restarts.
- **Container startup time**: Python-based agents start in 1-3 seconds inside a slim container. If you need sub-second startup, explore pre-forking workers or keeping a warm pool of agent instances.

## Security Considerations

- **Secrets management**: Never store API keys in Docker images or source code. Use GitHub Actions secrets, your cloud provider's secret manager (AWS Secrets Manager, GCP Secret Manager), or a vault like HashiCorp Vault at runtime.
- **API key rotation**: Set up a process to rotate LLM API keys regularly. The agent should reload keys from the secret store without a full restart when possible.
- **Network security**: Run the agent in a private subnet. The health check and agent-API endpoints should not be publicly accessible — place them behind a reverse proxy (NGINX, Cloudflare) with rate limiting and IP allowlisting.
