# Review: Deployment Patterns

⏱️ 5 minutes

## Key Takeaways

1. **Docker containers eliminate environment drift** by packaging your agent, its dependencies, and its tool runtimes into a single immutable image that is built once and promoted unchanged through every environment.
2. **Blue-green, canary, and rolling updates each serve a different reliability goal.** Blue-green gives you instant rollback at double resource cost; canary lets you validate on real traffic; rolling updates minimize extra capacity at the cost of mid-session disruption.
3. **Health checks for agents must go beyond process liveness.** Your readiness probe should verify LLM API connectivity, tool dependency availability, and secret validity — not just that the Python process is running.
4. **Secrets never belong in images, source code, or committed `.env` files.** Use runtime injection from a secrets manager or CI/CD secrets store, and set up automated key rotation.
5. **Session persistence is the difference between a graceful deploy and a user-facing reset.** Externalize session state to Redis or a database so rolling updates transfer active conversations to the new agent version.

## Recap in 60 Seconds

1. Package your agent in a Docker image with a multi-stage build, pin all dependencies, and run as a non-root user.
2. Set up a CI/CD pipeline that builds the image, runs tests, promotes through staging, and deploys via blue-green or canary strategy with health-gated rollouts.
3. Always keep a rollback plan — the last two known-good image tags and a one-command revert — and externalize session state so deploys do not reset user conversations.
