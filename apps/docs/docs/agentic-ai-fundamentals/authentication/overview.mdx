# Authentication

⏱️ 15 minutes · Difficulty: Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Authenticate agents to access external tools and APIs using OAuth 2.0, API keys, and service accounts
- Implement user impersonation patterns so agents act on behalf of a specific user with scoped permissions
- Design secure OAuth 2.0 flows that work with autonomous, unattended agent execution
- Manage API keys, tokens, and credentials using secure storage and automated rotation

## Prerequisites

You should already understand:

- Basic HTTP authentication concepts (headers, bearer tokens, basic auth)
- OAuth 2.0 grant types and the difference between authorization code and client credentials flows
- How agents select and call tools through the observe-think-act loop (see [Observe-Think-Act Loop](../../observe-think-act-loop/overview))
- The tool-calling patterns covered in [Agent Tools and Tool Calling](../../agent-tools-and-tool-calling/overview)

## Why This Matters

Agents that only process local data or work entirely within a sandbox do not need authentication. The moment an agent calls an external API — sending an email, querying a CRM, reading a spreadsheet from cloud storage, or posting to Slack — it needs credentials. Unlike a human user who can type a password, scan a QR code, or approve a push notification, an agent runs autonomously. It cannot pause mid-task to ask for a password. Every credential the agent needs must be available before the task begins, stored securely, and refreshed automatically when it expires.

Poorly managed authentication is both a security risk and the top reason production agent deployments fail audits. Credentials hardcoded in source code, tokens stored in plain text configuration files, and API keys exposed in logs are the most common findings in security reviews of agent systems. A single leaked credential can grant an attacker access to the same tools and data the agent uses. As agent systems grow from experimental scripts to production deployments handling sensitive data, authentication becomes the foundation that all other security controls depend on.

## The Problem

A user gives your agent a single task: "Pull the latest sales report from Google Drive, check the inventory levels in our CRM, and summarize everything in a new database record." Your agent now needs to call three different services, each with a different authentication mechanism:

- **Google Drive** requires OAuth 2.0 with user delegation — the agent must act as the specific user who owns the report
- **CRM** uses an API key passed in a custom header
- **Database** requires a service account with client certificate authentication

The agent runs unattended. It cannot ask the user "please paste your Google Drive refresh token" or "please regenerate that CRM API key." All credentials must already be in place, securely stored, and capable of being refreshed without human intervention. And if any credential expires or is revoked mid-task, the agent must handle the failure gracefully rather than crashing or leaking the error in a log.

## The Concept

Agent authentication builds on standard web authentication patterns but adapts them for autonomous machine-to-machine operation.

### API Key Authentication

The simplest approach. A static string is sent with every request, typically in an HTTP header. API keys work well for agents that authenticate as themselves (not as a user) to internal services. The challenge is key management: storing the key securely, rotating it periodically, and ensuring it is never logged or exposed in error messages.

### OAuth 2.0 with Refresh Tokens

OAuth 2.0 is the standard for delegated access. An agent obtains an access token (short-lived, typically one hour) and a refresh token (long-lived, typically days or months). The agent uses the access token for API calls and exchanges the refresh token for a new access token when the old one expires. For autonomous agents, the refresh token flow is critical because it removes the need for human re-authentication.

The authorization code flow with PKCE works well for agents that need to act on behalf of a user. The user authorizes once (perhaps during agent setup), and the agent stores the resulting refresh token for ongoing use.

### Service Accounts

A service account is a non-human identity with its own permissions. Instead of impersonating a user, the agent authenticates as itself. Cloud platforms (AWS IAM roles, GCP service accounts, Azure managed identities) provide this pattern natively. Service accounts are ideal for background agents that run scheduled tasks, process data pipelines, or operate internal tools where user-level delegation is unnecessary.

### User Impersonation

When an agent must act as a specific user, it needs that user's delegated authority. OAuth 2.0 on-behalf-of (OBO) flows allow an agent to exchange a token for another token scoped to a different downstream service. This is common in multi-hop scenarios where the agent calls service A, which must then call service B on the same user's behalf.

### Token Exchange Patterns

Token exchange is the general pattern of converting one credential into another. An agent might:

- Exchange an API key for a session token
- Exchange a refresh token for a new access token
- Exchange a user token for a service-scoped token via OBO
- Exchange a short-lived cloud credential (like an AWS STS token) for access to a specific resource

Each exchange is an opportunity to inject security controls: check that the token is still valid, verify the scope is appropriate, and audit the access.
