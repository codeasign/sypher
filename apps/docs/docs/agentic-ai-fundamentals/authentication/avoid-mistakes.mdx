# Avoid Mistakes: Authentication Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Complex OAuth 2.0 flows and service account setup are unnecessary overhead for certain agent systems. Skip the heavy authentication infrastructure when:

- **Your agent only works with local data** — reading files from disk, running local databases, processing in-memory data. No external API calls means no credentials to manage.
- **Your agent calls a single internal API with a static API key** — a single encrypted API key stored in environment variables is sufficient. You do not need a full OAuth flow with refresh tokens.
- **You are prototyping in a sandbox environment** — during early development, use temporary credentials with short expiry. Invest in production-grade auth only when the agent moves to staging.
- **Your platform handles auth for you** — if you are building agents inside a platform that manages credentials (like an MCP host with built-in auth delegation), do not re-implement authentication at the agent level.

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Secrets manager for storage + OAuth 2.0 refresh flow + token caching + credential rotation |
| **When to use** | Agents calling external APIs, multi-service workflows, production deployments |
| **When NOT to use** | Local-only agents, single-key prototypes, platform-managed auth |
| **Mental model** | Your agent is a headless OAuth client — it must manage its own token lifecycle without human help |

## Common Mistakes

### Mistake 1: Hardcoding Credentials in Source Code

**Mistake:** Writing API keys, client secrets, or service account passwords directly into Python files or configuration files committed to version control.

**Why it happens:** It is the fastest way to get started. Paste the key, import it, move on.

**Symptoms:** Credentials appear in git history forever. If the repository is ever made public or shared with an external contractor, every hardcoded secret is compromised. Automated secret scanners will flag the repository.

**Fix:** Use a secrets manager (HashiCorp Vault, AWS Secrets Manager) or at minimum environment variables loaded at runtime. Add a `.gitignore` entry for any files that contain secrets. Run a secret scanner like `truffleHog` or `git secrets` in CI.

### Mistake 2: Not Handling Token Expiry in Agent Logic

**Mistake:** Assuming that a token obtained at the start of an agent session will remain valid for the entire session, and crashing when it expires mid-task.

**Why it happens:** During development, tokens are fresh and tests run quickly. The developer never waits long enough for a token to expire.

**Symptoms:** Agents that work perfectly in development fail in production on tasks lasting longer than the token lifetime. The error surfaces as a mysterious 401 response with no clear cause.

**Fix:** Implement automatic token refresh in the authentication layer, not in the agent logic. The agent should never think about token expiry. Use a client class like `OAuthClient` that refreshes transparently, and always retry once on a 401 response before failing.

### Mistake 3: Storing Secrets in Logs

**Mistake:** Logging API request headers, full HTTP responses, or error messages that contain the credential itself. This includes logging the full request URL when the API key is passed as a query parameter.

**Why it happens:** Debugging authentication failures is hard. The natural instinct is to log everything to see what is happening.

**Symptoms:** A log aggregator search reveals plaintext API keys and tokens. Any developer with access to logs now has credential access. If logs are shipped to a third-party service, that service also has the secrets.

**Fix:** Implement a log sanitizer that redacts known credential patterns (bearer tokens, API key headers, `client_secret` fields) before writing any log entry. Test the sanitizer by intentionally including fake credentials in a test run and checking that they are redacted.

```python
import re

LOG_REDACT_PATTERNS = [
    (r'Bearer\s+[A-Za-z0-9\-._~+/]+=*', 'Bearer [REDACTED]'),
    (r'(?i)(api[_-]?key|secret|password)\s*[:=]\s*["\']?\S+["\']?', r'\1=[REDACTED]'),
    (r'(?i)(x-api-key|authorization)\s*:\s*\S+', r'\1: [REDACTED]'),
]

def sanitize_log(line: str) -> str:
    for pattern, replacement in LOG_REDACT_PATTERNS:
        line = re.sub(pattern, replacement, line)
    return line
```

### Mistake 4: Over-Provisioning Permissions

**Mistake:** Giving an agent the broadest possible scope ("admin" or "full_access") because it is easier than figuring out exactly which permissions are needed.

**Why it happens:** Developers do not know which API endpoints the agent will call, so they request all scopes. It works, so there is no immediate pressure to narrow them.

**Symptoms:** A single compromised credential gives an attacker full access to the service. An agent that only needs read access to a spreadsheet can also delete all files. Security auditors flag this as a critical finding.

**Fix:** Start with read-only scopes and add write permissions only when a specific tool requires them. Document exactly which scope each tool needs. Review the scope list regularly as new tools are added. Use separate credentials for read and write operations.

### Mistake 5: Ignoring Token Refresh Race Conditions

**Mistake:** Multiple concurrent tool calls trigger simultaneous token refreshes, causing redundant requests to the authorization server and potential race conditions where one refresh invalidates another.

**Why it happens:** An agent often makes parallel tool calls. Each call checks the token, finds it expired, and initiates a refresh — all at the same time.

**Symptoms:** The authorization server returns errors from duplicate refresh requests. Token state becomes inconsistent. Some parallel calls get the old token while others get the new one, causing intermittent 401 failures.

**Fix:** Use a lock or a flag to ensure only one refresh happens at a time. Concurrent callers should wait for the in-progress refresh to complete rather than starting their own.

```python
import asyncio

class ThreadSafeOAuthClient(OAuthClient):
    """OAuth client that prevents concurrent token refreshes."""

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._refresh_lock = asyncio.Lock()
        self._refresh_in_progress = False

    @property
    async def access_token(self) -> str:
        if self._access_token and time.time() < self._expires_at:
            return self._access_token
        async with self._refresh_lock:
            # Double-check after acquiring the lock
            if self._access_token and time.time() < self._expires_at:
                return self._access_token
            self._exchange_refresh_token()
            return self._access_token
```
