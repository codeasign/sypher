# Building an MCP Server — Review

*Reading time: 3 minutes*

## Key Takeaways

- An MCP server has three layers: transport (stdio/SSE), protocol (JSON-RPC lifecycle), and business logic (tool/resource/prompt handlers)
- Use the MCP Python SDK (`mcp` package) — it handles lifecycle management, JSON-RPC parsing, and transport abstraction
- Register tools with `@server.list_tools()` and `@server.call_tool()` decorators; resources with `@server.list_resources()` and `@server.read_resource()`
- Always resolve and validate file paths in resource handlers to prevent path traversal attacks
- Set `isError: true` on tool error responses so the client recognizes the call failed

## Recap in 60 Seconds

1. The MCP SDK does the heavy lifting — you write decorator-annotated handlers for `list_tools`, `call_tool`, `list_resources`, and `read_resource`
2. All handlers must be async — use `asyncio` and async libraries throughout; use `asyncio.to_thread()` for synchronous code
3. Test your server with `npx @modelcontextprotocol/inspector` to verify the full lifecycle before connecting clients
