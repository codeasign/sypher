# Cost and Latency Tracking

⏱️ 15 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Estimate and track the per-session and per-task cost of your agent
- Measure latency breakdowns across LLM inference, tool execution, and transport
- Set budgets and alert thresholds for cost and latency regressions
- Optimize the biggest cost and latency drivers in your agent system

## Prerequisites

You should already understand:

- How LLM pricing works — input vs output tokens, model tiers (see [LLM APIs in Practice](../../llm-apis-in-practice/overview))
- The observability framework from the previous lesson (see [Observability, Logging, and Tracing](../observability-logging-and-tracing/overview))
- How agents call tools and process results through the observe-think-act loop
- Basic understanding of percentile distributions (p50, p95, p99)

## Why This Matters

Every agent call costs real money. A single complex session can consume hundreds of thousands of tokens across multiple LLM calls and cost more than a dollar. At scale — thousands of sessions per day — agent operating costs can reach tens of thousands of dollars per month. These costs are variable and grow with usage, unlike fixed infrastructure costs.

Latency is equally critical. An agent that takes 30 seconds to answer a simple question frustrates users and reduces adoption. Each additional tool call adds 500ms to 5 seconds of latency. Each LLM inference adds 2-10 seconds. Without tracking, you cannot know which part of your agent pipeline is the bottleneck.

Engineering teams that track cost and latency from day one build agents that are 10-100x more efficient than teams that optimize later. The reason is architectural: early decisions about tool granularity, prompt length, and model choice have outsized impacts on operating costs. If you do not measure these from the start, you build an expensive system and have to re-architect later.

## The Problem

Your agent takes 45 seconds to respond to a user query. Why? Is it the LLM inference time? The three sequential web search calls? The file read that takes 2 seconds? The transport overhead? You have no data, so you guess. You try a different model — latency drops to 40 seconds. Marginal improvement. You optimize tool descriptions — still 40 seconds. You are cargo-culting optimizations without knowing the actual bottleneck.

Cost is similarly opaque. Your monthly API bill increased by $2,000 after you deployed the agent. Is that from 100k sessions at $0.02 each, or 10k sessions at $0.20 each? Different root causes require different fixes, but without cost-per-session tracking, you cannot tell.

## The Concept

Cost and latency tracking breaks the agent pipeline into measurable components:

- **LLM cost** — input tokens × price per token + output tokens × price per token. Different models have different rates. Claude Sonnet costs $3/M input tokens and $15/M output tokens; Claude Opus costs $15/M and $75/M respectively.

- **Tool execution cost** — API call costs for external services. A web search costs ~$0.003 per query through the Brave API. A GitHub API call costs nothing (within rate limits) but has a latency cost.

- **Latency breakdown** — total response time = system prompt construction + first LLM inference + (tool execution + result processing + next LLM inference) × N tool calls + final response generation.

The key insight is that **tool call count is the single biggest lever** for both cost and latency. Each tool call requires a round-trip through the LLM (2-10 seconds) and adds the tool's execution time. Reducing tool calls from 8 to 4 approximately halves both cost and latency.
