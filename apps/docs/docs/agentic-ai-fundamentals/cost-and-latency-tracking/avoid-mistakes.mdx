# Avoid Mistakes: Cost and Latency Tracking Pitfalls

⏱️ 10 minutes

## When NOT to Use This

Detailed cost and latency tracking adds instrumentation overhead. Do not implement it when:

- **You are in prototype or demo stage** — optimize for functionality first; cost optimization comes after product-market fit
- **Your agent runs fewer than 500 sessions/month** — the infrastructure cost of tracking may exceed the savings from optimization
- **You use a single model with no tool calls** — a simple chatbot's cost is just token count × model price; you can estimate it manually
- **Cost is not a constraint** — internal tools with low usage do not need per-session cost tracking

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Per-session cost + latency + tool call tracking with budget alerts |
| **When to use** | Production deployments, user-facing agents, cost-sensitive applications |
| **When NOT to use** | Prototypes, low-volume internal tools, simple chatbots |
| **Mental model** | A financial dashboard for your agent's API and infrastructure spending |

## Common Mistakes

### Mistake 1: Only Tracking Input Token Costs

**Mistake:** Tracking prompt tokens but ignoring output tokens, which are 3-5x more expensive per token.

**Why it happens:** Input tokens outnumber output tokens 5:1, so they seem like the bigger cost. But output tokens cost 5x more per token, making them roughly equal in total cost.

**Symptoms:** Your cost estimate is 40-50% lower than your actual API bill.

**Fix:** Always separate input and output token costs with their respective per-token prices. Never use a single "average cost per token."

### Mistake 2: Ignoring Context Window Growth

**Mistake:** Assuming each LLM call in a session costs the same as the first one.

**Why it happens:** The first cost estimate is easy to compute. You multiply by the number of calls.

**Symptoms:** Actual cost is 2-3x higher than estimated because each subsequent LLM call includes the growing conversation history in its input.

**Fix:** Model the cumulative context: each tool call's result is appended to the conversation. The 10th LLM call has significantly more input tokens than the 1st:

```python
def cumulative_cost_estimate(initial_prompt: int, tool_calls: int, output_per_call: int = 500):
    history_growth = tool_calls * (output_per_call + 200)  # each tool adds output + result
    total_input = initial_prompt * (tool_calls + 1) + sum(
        history_growth * i for i in range(tool_calls + 1)
    )
    total_output = output_per_call * (tool_calls + 1)
    return {"input_tokens": total_input, "output_tokens": total_output}
```

### Mistake 3: Optimizing Latency at the Cost of Correctness

**Mistake:** Switching to a faster, cheaper model and halving response time, but the answer quality drops by 20%.

**Why it happens:** Latency is easy to measure and optimize. Correctness is harder.

**Symptoms:** Users complain about "dumber" answers, but the dashboard shows great latency numbers.

**Fix:** Always track cost and latency alongside task success rate. A fast wrong answer is worse than a slow correct one. Set a minimum success rate threshold before latency optimizations are considered.

### Mistake 4: Benchmarking with Empty Caches

**Mistake:** Measuring latency on the first session of the day, when nothing is cached.

**Why it happens:** Cold-start measurements are the easiest to collect.

**Symptoms:** Reported latency is 2-3x worse than what users actually experience after the first session.

**Fix:** Report both cold-start latency (first session) and warm latency (subsequent sessions). Users experience warm latency most of the time, but cold latency matters for sporadic usage.

### Mistake 5: Neglecting Tool-Level Cost Attribution

**Mistake:** Tracking total session cost but not which tool calls cost the most.

**Why it happens:** Total cost is one number. Per-tool cost requires instrumenting each tool with its own cost tracking.

**Symptoms:** You know the agent is expensive, but you cannot tell whether that is from too many `search_web` calls (each costs $0.003 in API fees) or too many `read_file` calls (free but trigger LLM round-trips).

**Fix:** Attribute cost in two dimensions: direct cost (API fees for the tool) and indirect cost (LLM round-trip cost triggered by the tool's result). The indirect cost is typically 5-20x larger than the direct cost.
