The Strategy pattern is how JavaScript developers implement the Open/Closed Principle. Instead of a monolithic function that branches on a type parameter, you define a common interface (duck-typed or via class inheritance) and let each strategy implement its own behavior. New strategies arrive as new files without touching the core logic.

## The Domain: E-Commerce Discount Calculator

An online store needs to calculate discounts for different customer types and promotional campaigns. New discount types are added regularly as the business runs seasonal promotions and loyalty programs.

### The Violation

The following example shows a monolithic `calculateDiscount` function that branches on a customer type string. Every new discount type requires editing this function.

```javascript
// VIOLATION: calculateDiscount uses a big switch on customer type.
// Adding a new discount type requires editing this function, which
// risks breaking existing discount calculations.

class Order {
  constructor(customerType, amount) {
    this.customerType = customerType;
    this.amount = amount;
  }
}

function calculateDiscount(order) {
  // VIOLATION: switch on customerType -- every new type requires a new case
  // and full regression testing of all existing cases.
  switch (order.customerType) {
    case "regular":
      // Regular customers: no discount
      return 0;

    case "premium":
      // Premium customers: 20% off
      return order.amount * 0.2;

    case "vip":
      // VIP customers: 40% off, capped at $200
      const discount = order.amount * 0.4;
      return Math.min(discount, 200);

    case "seasonal":
      // Seasonal: 15% off during promotion
      return order.amount * 0.15;

    default:
      return 0;
  }
}

// Testing all paths requires exercising the same function
console.log(`Regular (100): $${calculateDiscount(new Order("regular", 100))}`);
console.log(`Premium (100): $${calculateDiscount(new Order("premium", 100))}`);
console.log(`VIP (1000):    $${calculateDiscount(new Order("vip", 1000))}`);
console.log(`Seasonal (100): $${calculateDiscount(new Order("seasonal", 100))}`);

// What if we need to add a "loyalty" discount? We must edit calculateDiscount.
```

Expected output:

```bash
Regular (100): $0
Premium (100): $20
VIP (1000):    $200
Seasonal (100): $15
```

### The Correct Implementation: Strategy Pattern

The solution is to define a `DiscountStrategy` base class and implement each discount type separately. The `DiscountCalculator` class depends on the abstraction, never on concrete discount types.

```javascript
// CORRECT: Discount strategies implement a common interface. The calculator
// delegates to the strategy. New discount types are added as new classes
// without touching any existing code.

// --- Strategy interface (abstract base class) ---
class DiscountStrategy {
  /**
   * Calculate discount for a given order amount.
   * @param {number} amount - The order total before discount
   * @returns {number} Discount amount
   */
  calculate(amount) {
    throw new Error("Subclasses must implement calculate()");
  }

  /**
   * @returns {string} Unique key for this discount strategy
   */
  getType() {
    throw new Error("Subclasses must implement getType()");
  }
}

// --- Concrete strategies ---

class NoDiscount extends DiscountStrategy {
  getType() { return "regular"; }

  calculate(amount) {
    return 0;
  }
}

class PercentageDiscount extends DiscountStrategy {
  constructor(percentage, cap = Infinity) {
    super();
    this.percentage = percentage;
    this.cap = cap;
  }

  getType() { return "percentage"; }

  calculate(amount) {
    const discount = amount * this.percentage;
    return Math.min(discount, this.cap);
  }
}

class PremiumDiscount extends DiscountStrategy {
  getType() { return "premium"; }

  calculate(amount) {
    return amount * 0.2; // 20% off
  }
}

class VipDiscount extends DiscountStrategy {
  getType() { return "vip"; }

  calculate(amount) {
    const discount = amount * 0.4; // 40% off
    return Math.min(discount, 200);
  }
}

class SeasonalDiscount extends DiscountStrategy {
  constructor(rate = 0.15) {
    super();
    this.rate = rate;
  }

  getType() { return "seasonal"; }

  calculate(amount) {
    return amount * this.rate;
  }
}

class LoyaltyDiscount extends DiscountStrategy {
  constructor(yearsAsMember) {
    super();
    // Higher discount for longer membership
    this.rate = Math.min(0.05 + yearsAsMember * 0.02, 0.5);
  }

  getType() { return "loyalty"; }

  calculate(amount) {
    return amount * this.rate;
  }
}

// --- Registry (maps type keys to strategy instances) ---
class DiscountRegistry {
  constructor() {
    this.strategies = new Map();
  }

  register(strategy) {
    this.strategies.set(strategy.getType(), strategy);
  }

  get(type) {
    const strategy = this.strategies.get(type);
    if (!strategy) {
      throw new Error(`Unknown discount type: ${type}`);
    }
    return strategy;
  }
}

// --- Core calculator -- closed for modification ---
class DiscountCalculator {
  constructor(registry) {
    this.registry = registry;
  }

  calculateDiscount(customerType, amount) {
    const strategy = this.registry.get(customerType);
    return strategy.calculate(amount);
  }
}

// --- Setup ---
const registry = new DiscountRegistry();
registry.register(new NoDiscount());
registry.register(new PremiumDiscount());
registry.register(new VipDiscount());
registry.register(new SeasonalDiscount());
registry.register(new PercentageDiscount(0.25, 150)); // 25% off, capped at $150
registry.register(new LoyaltyDiscount(3));             // 3-year member

const calculator = new DiscountCalculator(registry);

// --- Usage ---
console.log(`Regular (100):      $${calculator.calculateDiscount("regular", 100)}`);
console.log(`Premium (100):      $${calculator.calculateDiscount("premium", 100)}`);
console.log(`VIP (1000):         $${calculator.calculateDiscount("vip", 1000)}`);
console.log(`Seasonal (100):     $${calculator.calculateDiscount("seasonal", 100)}`);
console.log(`Percentage (200):   $${calculator.calculateDiscount("percentage", 200)}`);
console.log(`Loyalty 3yr (100):  $${calculator.calculateDiscount("loyalty", 100)}`);

// --- Adding a new discount type without editing existing code ---
//
// New requirement: "Black Friday" flash sale -- 50% off, no cap.
// We create a new class and register it. No existing files are changed.

class BlackFridayDiscount extends DiscountStrategy {
  getType() { return "black-friday"; }

  calculate(amount) {
    return amount * 0.5;
  }
}

registry.register(new BlackFridayDiscount());
console.log(`Black Friday (100): $${calculator.calculateDiscount("black-friday", 100)}`);
```

Expected output:

```bash
Regular (100):      $0
Premium (100):      $20
VIP (1000):         $200
Seasonal (100):     $15
Percentage (200):   $50
Loyalty 3yr (100):  $11
Black Friday (100): $50
```

## Key Takeaways

- **Class inheritance** provides the interface contract. JavaScript's `extends` with a base class that throws in unimplemented methods simulates an abstract class.
- **Dynamic dispatch** through the registry Map means the calculator never needs to know about concrete strategy types.
- **Adding strategies** is a matter of writing a new class and registering it. The `DiscountCalculator` and all existing strategies remain unchanged.
- **Composable strategies** like `PercentageDiscount` accept constructor parameters, allowing the same class to be reused with different configurations (e.g., seasonal rate of 10% vs 15%).
- The registry pattern is a simple `Map<string, DiscountStrategy>`. In a real application, this could be replaced with dependency injection, a plugin system, or dynamic module loading.
- The `default` case in the original switch is now handled by the registry: attempting to use an unregistered type throws a clear error, turning a silent fallthrough into an explicit failure.
