## The Principle

"Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it." -- Barbara Liskov, 1987. The Liskov Substitution Principle (LSP) states that subtypes must be substitutable for their base types. If you have code that works with a base class, you should be able to pass in any derived class and the code should still behave correctly, without surprises.

## Why It Exists

Before LSP was formally defined, inheritance was commonly treated as a code reuse mechanism rather than a subtyping relationship. Developers would extend a base class to inherit its methods, then override them freely without preserving the base class contract. The classic example is the Rectangle-Square problem: a `Square` extending `Rectangle` makes intuitive real-world sense, but mathematically a square is not a rectangle if you can set width and height independently. Code that sets `rectangle.setWidth(5); rectangle.setHeight(3)` and expects `area() == 15` breaks silently when given a `Square` that constrains width == height. This revealed that inheritance without behavioral substitutability creates subtle, hard-to-find bugs.

## The Violation

Consider a `Bird` base class with a `fly()` method. A `Sparrow` subclass works fine, but a `Penguin` subclass cannot fly. The naive approach is to have `Penguin.fly()` throw an exception.

```typescript
// VIOLATION: Penguin is not substitutable for Bird
class Bird {
  fly(): string {
    return "I am flying";
  }
}

class Sparrow extends Bird {
  // Inherits fly() -- fine
}

class Penguin extends Bird {
  fly(): string {
    throw new Error("Penguins cannot fly");
  }
}

function makeBirdFly(bird: Bird): void {
  console.log(bird.fly());
}

const sparrow = new Sparrow();
makeBirdFly(sparrow); // "I am flying"

const penguin = new Penguin();
makeBirdFly(penguin); // Error: Penguins cannot fly
```

Any code that expects a `Bird` and calls `fly()` will break for `Penguin`. The caller cannot know which birds fly and which do not without checking the type at runtime, which defeats polymorphism entirely.

## The Correct Implementation

Split the hierarchy: define `Bird` as the base with only universal behaviors, and introduce a `Flyable` interface for birds that can fly.

```typescript
// CORRECT: Separate Bird and Flyable
class Bird {
  breathe(): string {
    return "Breathing";
  }
}

interface Flyable {
  fly(): string;
}

class Sparrow extends Bird implements Flyable {
  fly(): string {
    return "Sparrow is flying";
  }
}

class Penguin extends Bird {
  swim(): string {
    return "Penguin is swimming";
  }
  // No fly() method -- Penguin never pretends to fly
}

function makeFly(creature: Flyable): void {
  console.log(creature.fly());
}

const sparrow = new Sparrow();
makeFly(sparrow); // "Sparrow is flying"

// Penguin cannot be passed to makeFly -- compiler enforces this
```

Now the type system enforces the contract at compile time. If a function needs flying behavior, it accepts `Flyable`, not `Bird`. A `Penguin` simply does not implement `Flyable`, so it cannot be misused. The base class `Bird` contains only the behaviors all birds share.

## How to Spot Violations in Code Review

Watch for these signals in a pull request:

- Overridden methods that throw `UnsupportedOperationException`, `NotImplementedError`, or any runtime exception with a comment like "not supported in this subclass"
- Empty method bodies in overrides -- overriding just to do nothing is usually a violation
- `if (obj instanceof SubType)` or `if (typeof obj.method === "function")` checks in code that should be polymorphic
- Methods in a base class that are only meaningful for some subclasses (e.g., a `Part-timeEmployee` with a `vestStockOptions()` that throws)
- Subclasses that weaken preconditions (e.g., a base class accepts any string, a subclass silently rejects empty strings)
- Subclasses that strengthen postconditions (e.g., a method that should return any positive integer but a subclass restricts it to even numbers)
- Base classes with a large number of abstract methods that few subclasses fully implement

## Real-World Occurrences

- **Java Collections Framework**: `List` is an interface; `ArrayList`, `LinkedList`, and `CopyOnWriteArrayList` all implement it without violating LSP. You can substitute any `List` implementation anywhere a `List` is expected.
- **Java Stream API**: `Stream` operations work identically regardless of whether the source is a collection, an array, or an I/O channel. Each source produces a `Stream` that respects the same contract.
- **C# IEnumerable**: Any collection, array, or query result implements `IEnumerable<T>`, and all can be substituted into `foreach` loops or LINQ methods.
- **Rust Traits with Blanket Implementations**: When you implement `Display` or `Debug` on a type, it automatically gets `.to_string()`. Any type that implements the trait is fully substitutable in any function expecting it.
- **React Components**: A well-designed component interface (props contract) allows any component satisfying the contract to be substituted without breaking the parent's logic.

## When Strict Adherence Hurts

Strict LSP adherence is not always the right choice:

- **Prototyping and MVPs**: When you are rapidly iterating and the subtype hierarchy is not yet stable, enforcing clean separation from the start adds overhead. A quick throw in an override can be acceptable as a stub.
- **Informal Behavioral Contracts**: In duck-typed languages like Python and JavaScript, LSP is enforced by convention rather than the type system. Applying formal interface hierarchies can feel like writing Java in Python.
- **Small, Self-Contained Hierarchies**: If a base class has only two subclasses and is never used polymorphically outside the module, the risk of substitution violations is minimal. The extra abstraction may not pay for itself.
- **Performance-Critical Code**: Interface indirection and trait objects have runtime costs. In hot paths, you may accept LSP trade-offs for performance, carefully documenting the behavioral differences.

The key is knowing when violated LSP will cause bugs versus when it is an acceptable shortcut. Ship the shortcut with a `TODO` and refactor before the code grows callers that rely on substitutability.
