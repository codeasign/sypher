## Implementation

The violation shows a `PaymentProcessor` base class with `CreditCardProcessor` and `PayPalProcessor` working correctly, but `RefundProcessor` throws because it needs different parameters. The correct version separates `Payable` and `Refundable` interfaces.

```javascript
// LSP -- JavaScript implementation
// Domain: payment processing

// ============================================
// VIOLATION: PaymentProcessor requires refund
// capability that not all processors support
// ============================================

class PaymentProcessor {
  processPayment(amount) {
    throw new Error("Subclasses must implement processPayment");
  }

  processRefund(amount) {
    throw new Error("Subclasses must implement processRefund");
  }
}

class CreditCardProcessor extends PaymentProcessor {
  processPayment(amount) {
    return `Processed credit card payment of $${amount}`;
  }

  processRefund(amount) {
    return `Processed credit card refund of $${amount}`;
  }
}

class PayPalProcessor extends PaymentProcessor {
  processPayment(amount) {
    return `Processed PayPal payment of $${amount}`;
  }

  processRefund(amount) {
    return `Processed PayPal refund of $${amount}`;
  }
}

class RefundOnlyProcessor extends PaymentProcessor {
  // VIOLATION: This processor cannot make payments,
  // but the base class contract promises it can
  processPayment(amount) {
    throw new Error("This processor only handles refunds");
  }

  processRefund(amount) {
    return `Processed standalone refund of $${amount}`;
  }
}

function executePayment(processor, amount) {
  // VIOLATION: Breaks when given RefundOnlyProcessor
  console.log(processor.processPayment(amount));
}

const cc = new CreditCardProcessor();
executePayment(cc, 100); // OK

const refundOnly = new RefundOnlyProcessor();
// executePayment(refundOnly, 50); // Throws!

// ============================================
// CORRECT: Separate Payable and Refundable
// ============================================

class Payable {
  processPayment(amount) {
    throw new Error("Subclasses must implement processPayment");
  }
}

class Refundable {
  processRefund(amount) {
    throw new Error("Subclasses must implement processRefund");
  }
}

class FullPaymentProcessor {
  processPayment(amount) {
    return `Processed payment of $${amount}`;
  }

  processRefund(amount) {
    return `Processed refund of $${amount}`;
  }
}

class CreditCardHandler extends FullPaymentProcessor {
  processPayment(amount) {
    return `Credit card payment: $${amount}`;
  }

  processRefund(amount) {
    return `Credit card refund: $${amount}`;
  }
}

class PayPalHandler extends FullPaymentProcessor {
  processPayment(amount) {
    return `PayPal payment: $${amount}`;
  }

  processRefund(amount) {
    return `PayPal refund: $${amount}`;
  }
}

// Separate concerns: a processor that only takes payments
class PaymentOnlyHandler extends Payable {
  processPayment(amount) {
    return `Payment-only handler processed $${amount}`;
  }
}

// A processor that only handles refunds
class RefundOnlyHandler extends Refundable {
  processRefund(amount) {
    return `Refund-only handler processed $${amount}`;
  }
}

function handlePayment(payable, amount) {
  console.log(payable.processPayment(amount));
}

function handleRefund(refundable, amount) {
  console.log(refundable.processRefund(amount));
}

// Main driver
console.log("=== CORRECT IMPLEMENTATION ===");

const ccHandler = new CreditCardHandler();
handlePayment(ccHandler, 200);
handleRefund(ccHandler, 50);

const payOnly = new PaymentOnlyHandler();
handlePayment(payOnly, 75);

const refundOnly = new RefundOnlyHandler();
handleRefund(refundOnly, 30);

// RefundOnlyHandler cannot be passed to handlePayment -- no processPayment method
// The interface contract is enforced structurally
```

```bash
Credit card payment: $200
Credit card refund: $50
Payment-only handler processed $75
Refund-only handler processed $30
```

### Key Language-Specific Notes

- JavaScript uses **duck typing** -- substitutability is structural, not nominal. If an object has the right methods, it is treated as the type. This means LSP violations often surface as runtime errors rather than compile-time errors.
- ES2022 classes with `extends` share method contracts by convention only. There is no abstract method enforcement at the language level; use `throw new Error(...)` as a pattern, or use the `abstract` base class pattern with constructor checks.
- Prefer **composition over inheritance** in JavaScript. Use mixins or factory functions to compose behaviors rather than deep class hierarchies. This naturally avoids LSP violations because there is no base class contract to violate.
- The structural type system means you can create lightweight interfaces by convention: if a function expects an object with a `processPayment` method, any object satisfying that shape works, regardless of its class hierarchy.
