## Implementation

A Python 3.11+ implementation of ISP using an order management system. The violation shows an `OrderProcessor` ABC with `process_payment()`, `ship_order()`, `send_invoice()`, and `apply_discount()` — an `OnlineOrder` must implement `ship_order()` even though downloads don't ship.

```python
# ISP — Python 3.11+ implementation
# Domain: Order management

from abc import ABC, abstractmethod
from dataclasses import dataclass

# ============================================================
# VIOLATION: Fat OrderProcessor interface forces OnlineOrder
# to implement shipping logic it doesn't need
# ============================================================
#
# class OrderProcessor(ABC):
#     @abstractmethod
#     def process_payment(self) -> str: ...
#     @abstractmethod
#     def ship_order(self) -> str: ...
#     @abstractmethod
#     def send_invoice(self) -> str: ...
#     @abstractmethod
#     def apply_discount(self, percent: float) -> str: ...
#
# class PhysicalOrder(OrderProcessor):
#     def process_payment(self) -> str: return "Payment processed"
#     def ship_order(self) -> str: return "Shipped via courier"
#     def send_invoice(self) -> str: return "Invoice sent"
#     def apply_discount(self, pct: float) -> str: return f"{pct}% discount applied"
#
# class OnlineOrder(OrderProcessor):
#     def process_payment(self) -> str: return "Payment processed"
#     def ship_order(self) -> str: raise NotImplementedError("No physical shipping")
#     def send_invoice(self) -> str: return "Invoice emailed"
#     def apply_discount(self, pct: float) -> str: return f"{pct}% discount applied"

# ============================================================
# CORRECT IMPLEMENTATION — Segregated ABCs
# ============================================================

class PaymentProcessor(ABC):
    @abstractmethod
    def process_payment(self) -> str:
        ...

class Shippable(ABC):
    @abstractmethod
    def ship_order(self) -> str:
        ...

class InvoiceSender(ABC):
    @abstractmethod
    def send_invoice(self) -> str:
        ...

class Discountable(ABC):
    @abstractmethod
    def apply_discount(self, percent: float) -> str:
        ...

@dataclass
class PhysicalOrder(PaymentProcessor, Shippable, InvoiceSender, Discountable):
    order_id: str
    amount: float
    address: str

    def process_payment(self) -> str:
        return f"Payment of ${self.amount:.2f} for order {self.order_id} processed"

    def ship_order(self) -> str:
        return f"Order {self.order_id} shipped to {self.address}"

    def send_invoice(self) -> str:
        return f"Invoice for {self.order_id} sent to billing address"

    def apply_discount(self, percent: float) -> str:
        discount = self.amount * (percent / 100)
        self.amount -= discount
        return f"{percent}% discount applied, new total: ${self.amount:.2f}"

@dataclass
class OnlineOrder(PaymentProcessor, InvoiceSender, Discountable):
    order_id: str
    amount: float
    email: str

    def process_payment(self) -> str:
        return f"Payment of ${self.amount:.2f} for order {self.order_id} processed"

    def send_invoice(self) -> str:
        return f"PDF invoice for {self.order_id} sent to {self.email}"

    def apply_discount(self, percent: float) -> str:
        discount = self.amount * (percent / 100)
        self.amount -= discount
        return f"{percent}% discount applied, new total: ${self.amount:.2f}"

# --- Main driver ---
def main() -> None:
    physical = PhysicalOrder(order_id="ORD-001", amount=299.99, address="123 Main St")
    online = OnlineOrder(order_id="ORD-002", amount=49.99, email="buyer@example.com")

    print("=== Physical Order ===")
    print(physical.process_payment())
    print(physical.ship_order())
    print(physical.apply_discount(10))
    print(physical.send_invoice())

    print("\n=== Online Order ===")
    print(online.process_payment())
    # online.ship_order()  # Not available — correct!
    print(online.apply_discount(5))
    print(online.send_invoice())

if __name__ == "__main__":
    main()
```

## Key Language-Specific Notes

- **ABCs (Abstract Base Classes)** from `abc` module formalize ISP boundaries — each ABC has exactly one abstract method, enforcing the segregation at the type level.
- **Multiple inheritance** — Python's support for multiple inheritance lets classes implement exactly the set of ABCs they need (`PhysicalOrder implements PaymentProcessor, Shippable, InvoiceSender, Discountable`).
- **`@dataclass`** reduces boilerplate for data-oriented order classes, letting the ISP structure shine without constructor clutter.
- **Duck typing** means you could use `Protocol` instead of ABCs — any object with a `process_payment()` method satisfies the payment contract without explicit inheritance.
