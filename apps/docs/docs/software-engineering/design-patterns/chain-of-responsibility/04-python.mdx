## Implementation

Below is a complete Python implementation of the Chain of Responsibility pattern using a stock market domain. A `TradeOrder` flows through an approval chain consisting of `JuniorTrader`, `SeniorTrader`, `PortfolioManager`, and `RiskDirector`. Each handler decides whether to approve the order based on its total value, or passes it to the next handler in the chain. The implementation uses Python 3.11+ features including `abc.ABC`, `@abstractmethod`, and `@dataclass`, and demonstrates OOP concepts (abstraction, encapsulation, inheritance, polymorphism) along with SOLID principles (SRP, OCP, LSP, ISP, DIP).

### How the Implementation Works

#### Class Diagram (Pseudocode)

#### OOP Concepts Applied

1. **Abstraction** — `OrderApprover` is an abstract base class that defines the contract (`set_next` and `approve_order`) while hiding the chain traversal logic. Concrete handlers only implement `_can_approve()`.
2. **Encapsulation** — Each handler encapsulates its approval limit and the reference to the next handler. `TradeOrder` encapsulates its order data and computes `total_value` internally.
3. **Inheritance** — All four concrete handlers inherit from `OrderApprover`. `RiskDirector` overrides `_can_approve` to always return `True` as the final handler.
4. **Polymorphism** — `approve_order` calls `_can_approve()` polymorphically on each handler. The correct limit check runs based on the handler's runtime type, without `isinstance` checks.

#### SOLID Principles Applied

5. **SRP** — Each handler has one reason to change: its approval threshold. `TradeOrder` is solely responsible for representing order data. The chain traversal logic lives in the abstract base class, not in any concrete handler.
6. **OCP** — Adding a new handler (e.g., `VicePresident` with a $50,000 limit) requires creating a new class and inserting it into the chain. No existing handler code is modified.
7. **LSP** — Any concrete handler can replace any other in the chain. `RiskDirector` (which always approves) can substitute for any handler without breaking the chain behavior.
8. **ISP** — The `OrderApprover` interface is minimal: `set_next` and `approve_order`. No handler is forced to implement methods it does not need.
9. **DIP** — Handlers depend on the abstract `OrderApprover` type when calling `set_next` and when delegating to the next handler. The chain is built by injecting concrete handlers into the abstract interface.

### TradeOrder (Request)
```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class TradeOrder:
    """Represents a stock trade order awaiting approval."""
    ticker: str
    quantity: int
    price_per_share: float
    order_type: str  # "buy" or "sell"

    @property
    def total_value(self) -> float:
        """Compute the total monetary value of the order."""
        return self.quantity * self.price_per_share
```

### OrderApprover (Abstract Handler)
```python
class OrderApprover(ABC):
    """Abstract handler in the Chain of Responsibility pattern."""

    def __init__(self, name: str, approval_limit: float) -> None:
        self.name = name
        self._approval_limit = approval_limit
        self._next_approver: OrderApprover | None = None

    def set_next(self, approver: "OrderApprover") -> "OrderApprover":
        """Set the next handler in the chain. Returns the next handler for chaining."""
        self._next_approver = approver
        return approver

    def approve_order(self, order: TradeOrder) -> str:
        """Attempt to approve the order or pass it to the next handler."""
        if self._can_approve(order):
            message = (
                f"{self.__class__.__name__} {self.name} approved: "
                f"{order.order_type.upper()} {order.quantity} {order.ticker} "
                f"at ${order.price_per_share:.2f} (total: ${order.total_value:,.2f})"
            )
            print(message)
            return message

        if self._next_approver is not None:
            print(f"{self.__class__.__name__} {self.name}: "
                  f"Order ${order.total_value:,.2f} exceeds my limit of "
                  f"${self._approval_limit:,.2f}. Escalating...")
            return self._next_approver.approve_order(order)

        message = (
            f"REJECTED: No approver can handle order for "
            f"{order.ticker} (${order.total_value:,.2f})"
        )
        print(message)
        return message

    @abstractmethod
    def _can_approve(self, order: TradeOrder) -> bool:
        """Determine whether this handler can approve the given order."""
        ...
```

### JuniorTrader (Concrete Handler)
```python
class JuniorTrader(OrderApprover):
    """Approves orders with total value <= $1,000."""

    def __init__(self, name: str = "Alice") -> None:
        super().__init__(name=name, approval_limit=1_000.0)

    def _can_approve(self, order: TradeOrder) -> bool:
        return order.total_value <= self._approval_limit
```

### RiskDirector (Concrete Handler)
```python
class RiskDirector(OrderApprover):
    """Final handler — can approve any order."""

    def __init__(self, name: str = "Dave") -> None:
        super().__init__(name=name, approval_limit=float("inf"))

    def _can_approve(self, order: TradeOrder) -> bool:
        return True
```

### Main / Driver
```python
if __name__ == "__main__":
    # Build the chain: JuniorTrader -> SeniorTrader -> PortfolioManager -> RiskDirector
    junior = JuniorTrader("Alice")
    senior = SeniorTrader("Bob")
    portfolio_mgr = PortfolioManager("Carol")
    risk_director = RiskDirector("Dave")

    junior.set_next(senior).set_next(portfolio_mgr).set_next(risk_director)

    # Define test orders
    orders = [
        TradeOrder(ticker="AAPL", quantity=10, price_per_share=150.00, order_type="buy"),
        TradeOrder(ticker="GOOGL", quantity=50, price_per_share=175.00, order_type="buy"),
        TradeOrder(ticker="MSFT", quantity=1000, price_per_share=200.00, order_type="sell"),
    ]

    print("=" * 72)
    print("TRADE ORDER APPROVAL SYSTEM")
    print("=" * 72)
    print()

    for order in orders:
        print(f"Processing: {order.order_type.upper()} {order.quantity} {order.ticker} "
              f"@ ${order.price_per_share:.2f} (total: ${order.total_value:,.2f})")
        print("-" * 72)
        junior.approve_order(order)
        print()
```

### Language-Specific Notes

- **Abstract base class**: Python's `abc.ABC` and `@abstractmethod` enforce the handler contract at runtime. Any attempt to instantiate `OrderApprover` directly raises `TypeError`.
- **`| None` type hint**: The `_next_approver` field uses the `OrderApprover | None` union syntax introduced in Python 3.10, making the optional chaining explicit.
- **`@dataclass`**: `TradeOrder` uses the `@dataclass` decorator which auto-generates `__init__`, `__repr__`, `__eq__`, and `__hash__`, reducing boilerplate for data-only classes.
- **Property for computed fields**: `total_value` is a `@property` rather than a stored field, ensuring it always reflects the current `quantity` and `price_per_share` without manual recalculation.
- **Method chaining**: `set_next()` returns the next handler, enabling fluent chain construction: `junior.set_next(senior).set_next(portfolio_mgr).set_next(risk_director)`.
- **`float("inf")`**: `RiskDirector` uses infinity as its limit, which naturally makes all comparisons return `True` via the `_can_approve` override — a cleaner approach than a sentinel value.
