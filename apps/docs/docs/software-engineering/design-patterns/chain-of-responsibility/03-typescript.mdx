The **Chain of Responsibility** pattern allows a request to travel through a sequence of handlers until one of them processes it. In the stock market domain, trade orders must be approved based on their total dollar value. Different trading personnel have distinct approval limits, forming a natural chain of responsibility.

## How the Implementation Works

### Class Diagram (Pseudocode)

```text
Participants:
  - OrderApprover (Interface)
    - setNext(approver: OrderApprover): OrderApprover
    - approveOrder(order: TradeOrder): string
  - TradeOrder (Class)
    - readonly ticker: string
    - readonly quantity: number
    - readonly pricePerShare: number
    - readonly orderType: string
    - get totalValue(): number
  - JuniorTrader implements OrderApprover  (limit = $1,000)
  - SeniorTrader implements OrderApprover  (limit = $10,000)
  - PortfolioManager implements OrderApprover  (limit = $100,000)
  - RiskDirector implements OrderApprover  (no limit, final handler)

OOP Roles:
  Handler Interface       → OrderApprover
  Concrete Handlers       → JuniorTrader, SeniorTrader, PortfolioManager, RiskDirector
  Request Object          → TradeOrder
  Client / Chain Builder  → main driver

SOLID Principles:
  Single Responsibility  — Each handler knows only its own approval limit
  Open/Closed            — New approval roles can be added without modifying existing handlers
  Liskov Substitution    — Any handler class can replace OrderApprover without breaking the chain
  Interface Segregation  — The handler interface is minimal (setNext + approveOrder)
  Dependency Inversion   — Handlers depend on the abstract OrderApprover interface, not concrete implementations
```

### OOP Concepts Applied

1. **Inheritance vs Interface Implementation** — Instead of extending an abstract class (as in the JS version), concrete handlers implement the `OrderApprover` interface, allowing them to extend a different base class if needed.
2. **Polymorphism** — The client calls `approveOrder()` on the `OrderApprover` interface reference; the actual handler that processes the request is determined at runtime.
3. **Encapsulation** — Each handler's approval limit and next-handler reference are `private readonly` fields, invisible to external code.
4. **Abstraction** — The `OrderApprover` interface defines a clean contract without dictating implementation details; each handler provides its own approval logic.

### SOLID Principles Applied

5. **Single Responsibility Principle** — Each handler class has exactly one job: deciding whether an order falls within its approval limit.
6. **Open/Closed Principle** — Adding a new approval tier (e.g., `ComplianceOfficer`) requires only a new handler class and a chain link — no existing handler code changes.
7. **Liskov Substitution Principle** — Any handler can replace another in the chain as long as it implements `OrderApprover`; the chain logic remains correct.
8. **Interface Segregation Principle** — The handler interface consists of just two methods: `setNext()` to build the chain and `approveOrder()` to handle a request.
9. **Dependency Inversion Principle** — High-level chain logic depends on the abstract `OrderApprover` interface, not on concrete trader or manager implementations.

### Complete TypeScript Implementation

### TradeOrder (Request)
```typescript
class TradeOrder {
  constructor(
    public readonly ticker: string,
    public readonly quantity: number,
    public readonly pricePerShare: number,
    public readonly orderType: 'buy' | 'sell'
  ) {}

  get totalValue(): number {
    return this.quantity * this.pricePerShare;
  }
}
```

### OrderApprover (Handler Interface)
```typescript
interface OrderApprover {
  setNext(approver: OrderApprover): OrderApprover;
  approveOrder(order: TradeOrder): string;
}
```

### BaseOrderApprover (Abstract Base)
```typescript
abstract class BaseOrderApprover implements OrderApprover {
  private nextHandler: OrderApprover | null = null;

  setNext(approver: OrderApprover): OrderApprover {
    this.nextHandler = approver;
    return approver;
  }

  abstract approveOrder(order: TradeOrder): string;

  protected next(order: TradeOrder): string {
    if (this.nextHandler) {
      return this.nextHandler.approveOrder(order);
    }
    return `REJECTED: ${order.ticker} — no approver available for $${order.totalValue}`;
  }

  protected getNextHandler(): OrderApprover | null {
    return this.nextHandler;
  }
}
```

### JuniorTrader (Concrete Handler)
```typescript
class JuniorTrader extends BaseOrderApprover {
  private readonly limit = 1000;
  private readonly role = 'JuniorTrader';

  approveOrder(order: TradeOrder): string {
    const value = order.totalValue;
    if (value <= this.limit) {
      return `APPROVED by ${this.role}: ${order.orderType.toUpperCase()} ${order.quantity} shares of ${order.ticker} at $${order.pricePerShare} (total: $${value})`;
    }
    return this.next(order);
  }
}
```


### Main / Driver
```typescript
function main(): void {
  const junior: OrderApprover = new JuniorTrader();
  const senior: OrderApprover = new SeniorTrader();
  const pm: OrderApprover = new PortfolioManager();
  const director: OrderApprover = new RiskDirector();

  // Build the chain: JuniorTrader -> SeniorTrader -> PortfolioManager -> RiskDirector
  junior.setNext(senior).setNext(pm).setNext(director);

  const orders: TradeOrder[] = [
    new TradeOrder('AAPL', 10, 150, 'buy'),
    new TradeOrder('GOOGL', 50, 175, 'buy'),
    new TradeOrder('MSFT', 1000, 200, 'sell'),
  ];

  for (const order of orders) {
    console.log(`--- Processing ${order.ticker} ---`);
    const result: string = junior.approveOrder(order);
    console.log(result);
    console.log('');
  }
}

main();
```

### Expected Output

```bash
--- Processing AAPL ---
APPROVED by SeniorTrader: BUY 10 shares of AAPL at $150 (total: $1500)

--- Processing GOOGL ---
APPROVED by SeniorTrader: BUY 50 shares of GOOGL at $175 (total: $8750)

--- Processing MSFT ---
APPROVED by RiskDirector: SELL 1000 shares of MSFT at $200 (total: $200000)
```

Note: The AAPL order has a total value of $1,500 (10 x $150), which exceeds the JuniorTrader limit of $1,000. It is approved by SeniorTrader instead. The GOOGL order ($8,750) exceeds the JuniorTrader limit but is within the SeniorTrader limit of $10,000, so SeniorTrader approves it. The MSFT order ($200,000) exceeds all limits up to RiskDirector, the final authority.

### Language-Specific Notes

- **Interfaces over abstract classes** — TypeScript's `OrderApprover` interface defines the contract cleanly, while the optional `BaseOrderApprover` abstract class provides shared chain logic (the `next()` delegation method).
- **Readonly constructor shorthand** — `public readonly ticker: string` in the constructor compiles to idiomatic immutable fields without manual field declarations or `Object.freeze()`.
- **Union type for orderType** — The `orderType: 'buy' | 'sell'` union type catches invalid order types at compile time rather than at runtime.
- **Type annotations on variables** — Declaring `const junior: OrderApprover = new JuniorTrader()` makes the polymorphic intent explicit, though TypeScript's type inference would also work.
- **Compile-time safety** — Unlike the JavaScript version (where a missing `approveOrder()` implementation throws at runtime), TypeScript catches the error during compilation because the interface contract is enforced statically.
- **Fluent interface typing** — `setNext()` returns `OrderApprover` rather than `void`, enabling method chaining with full type preservation through the chain.
