## Implementation

Below is a complete Java 17+ implementation of the Abstract Factory pattern using a banking domain. The `BankFactory` interface defines three creation methods, and two concrete factories (`PersonalBankingFactory` and `BusinessBankingFactory`) produce families of related account objects. Java's type system enforces the contract at compile time: any class that implements `BankFactory` must provide concrete implementations for all three creation methods, or the compiler rejects the code. The client code depends only on the `BankFactory` and `Account` interfaces.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌─────────────────────────────────────────────────────────────┐
//   │   <<interface>> BankFactory                                 │
//   ├─────────────────────────────────────────────────────────────┤
//   │ + createSavingsAccount(String, double): Account              │  ← OOP: Abstraction
//   │ + createCheckingAccount(String, double): Account             │     ISP: focused
//   │ + createLoanAccount(String, double): Account                 │     creation methods
//   └─────────────┬───────────────────────────┬───────────────────┘
//                 │ implements                │ implements
//   ┌─────────────┴───────────────┐  ┌────────┴───────────────────┐
//   │ PersonalBankingFactory      │  │ BusinessBankingFactory     │
//   │ ── creates Personal         │  │ ── creates Business        │  ← OOP: Polymorphism
//   │    product family           │  │    product family           │     LSP: substitutable
//   └─────────────┬───────────────┘  └────────┬───────────────────┘
//                 │                           │
//                 ▼                           ▼
//   ┌──────────────────────────┐  ┌───────────────────────────┐
//   │ PersonalSavingsAccount   │  │ BusinessSavingsAccount    │
//   │ PersonalCheckingAccount  │  │ BusinessCheckingAccount   │  ← OOP: Inheritance
//   │ PersonalLoanAccount      │  │ BusinessLoanAccount       │     (implements Account)
//   └─────────────┬────────────┘  └────────┬──────────────────┘
//                 │                        │
//                 └──────┬─────────────────┘
//                        │ implements
//                  ┌─────┴───────────┐
//                  │ <<interface>>   │
//                  │ Account         │  ← OOP: Abstraction
//                  ├─────────────────┤     ISP: minimal interface
//                  │ getDetails()    │
//                  │ getBalance()    │
//                  │ getMonthlyFee() │
//                  └─────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `BankFactory` and `Account` are Java interfaces that define the contracts. Client code depends on these interfaces, never on the concrete classes.
2. **Encapsulation** — Each account class uses `private final` fields to hide internal state. Access is provided exclusively through public getter methods defined in the `Account` interface.
3. **Inheritance** — All concrete account classes implement the `Account` interface. Both concrete factory classes implement the `BankFactory` interface.
4. **Polymorphism** — The driver stores accounts in a `List<Account>` and calls methods through the `Account` interface. Java's virtual method dispatch ensures the correct implementation runs per concrete type at runtime.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` creates only personal banking products. `BusinessBankingFactory` creates only business banking products. Each account class represents exactly one product type.
2. **OCP** — Adding a `PremiumBankingFactory` class requires no changes to the `BankFactory` interface, existing factories, or existing account classes.
3. **LSP** — Every concrete account type can substitute `Account` without breaking the client. The driver treats all accounts uniformly through the `Account` interface.
4. **ISP** — The `Account` interface is minimal: three methods. The `BankFactory` interface has three creation methods. No class implements methods it does not need.
5. **DIP** — Client code depends on the `BankFactory` and `Account` interfaces, never on concrete types like `PersonalSavingsAccount`.

### Account

```java
// Abstract Factory Pattern — Java 17+ — Banking example
// ======================================================
// Participants: BankFactory (AbstractFactory), PersonalBankingFactory
//   and BusinessBankingFactory (ConcreteFactory), Account (AbstractProduct),
//   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
//   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
//   (ConcreteProducts)
// OOP: abstraction, encapsulation, inheritance, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ======================================================

import java.text.DecimalFormat;
import java.util.List;
// ── Abstract Product Interface ──
// OOP role: Abstraction + ISP (defines the contract for all account types)
// SOLID: ISP (minimal interface), DIP (clients depend on this)
interface Account {
    String getDetails();
    double getBalance();
    double getMonthlyFee();
}
```

### PersonalSavingsAccount

```java
// ── Concrete Product Family 1: Personal Banking Accounts ──
// OOP role: Inheritance (implements Account) + Polymorphism
// SOLID: LSP (substitutable for Account), OCP (extensible), SRP (one responsibility)

final class PersonalSavingsAccount implements Account {
    private final String holder;
    private final double balance;
    private final double interestRate = 3.5;
    private final double monthlyFee = 5.0;

    PersonalSavingsAccount(String holder, double initialDeposit) {
        this.holder = holder;
        this.balance = initialDeposit;
    }

    @Override
    public String getDetails() {
        DecimalFormat df = new DecimalFormat("#,##0.00");
        return "Personal Savings[" + holder + "]: $" + df.format(balance)
            + " @ " + interestRate + "% APY";
    }

    @Override
    public double getBalance() {
        return balance;
    }

    @Override
    public double getMonthlyFee() {
        return monthlyFee;
    }
}
```

### BankFactory

```java
// ── Abstract Factory Interface ──
// OOP role: Abstraction + Encapsulation (encapsulates product family creation)
// SOLID: SRP (creates accounts only), DIP (clients depend on this abstraction)
interface BankFactory {
    Account createSavingsAccount(String holder, double amount);
    Account createCheckingAccount(String holder, double amount);
    Account createLoanAccount(String holder, double amount);
}
```

### PersonalBankingFactory

```java
// ── Concrete Factory 1: PersonalBankingFactory ──
// OOP role: Polymorphism (implements BankFactory interface)
// SOLID: OCP (new product types don't change this), SRP (creates personal accounts only)
final class PersonalBankingFactory implements BankFactory {
    @Override
    public Account createSavingsAccount(String holder, double amount) {
        return new PersonalSavingsAccount(holder, amount);
```

### Main / Driver

```java
// ── Main / Driver ──
public class AbstractFactoryDemo {
    static void displayAccounts(String label, BankFactory factory) {
        System.out.println("=== " + label + " ===");
        List<Account> accounts = List.of(
            factory.createSavingsAccount("Alice", 1000.0),
            factory.createCheckingAccount("Bob", 2500.0),
            factory.createLoanAccount("Carol", 50000.0)
        );

        for (Account account : accounts) {
            System.out.println("  " + account.getDetails());
            System.out.printf("    Balance: $%.2f, Monthly Fee: $%.2f%n",
                account.getBalance(), account.getMonthlyFee());
        }
        System.out.println();
    }

    public static void main(String[] args) {
        BankFactory personalFactory = new PersonalBankingFactory();
        BankFactory businessFactory = new BusinessBankingFactory();

        displayAccounts("Personal Banking Accounts", personalFactory);
        displayAccounts("Business Banking Accounts", businessFactory);

        // Demonstrate family consistency
        System.out.println("=== Cross-Family Comparison ===");
        Account personalSavings = personalFactory.createSavingsAccount("Alice", 1000.0);
        Account businessSavings = businessFactory.createSavingsAccount("Acme Corp", 50000.0);

        System.out.printf("Personal Savings monthly fee: $%.2f%n",
            personalSavings.getMonthlyFee());
        System.out.printf("Business Savings monthly fee: $%.2f%n",
            businessSavings.getMonthlyFee());

        Account personalLoan = personalFactory.createLoanAccount("Carol", 50000.0);
        Account businessLoan = businessFactory.createLoanAccount("Initech", 500000.0);

        System.out.println("Personal Loan details: " + personalLoan.getDetails());
        System.out.println("Business Loan details: " + businessLoan.getDetails());
    }
}
```

### Language-Specific Notes

- **Interface as first-class construct** — Java's `interface` keyword defines a pure abstract type that classes can implement. An interface can contain abstract method signatures (implicitly `public abstract`), `default` method implementations, `static` methods, and `private` helper methods. The `BankFactory` and `Account` interfaces use abstract methods to define the contract.
- **`final` classes** — The `final` keyword on a Java class prevents subclassing. This is used for concrete product and factory classes to enforce that no further inheritance is allowed. This supports the OCP by preventing modification through inheritance while allowing extension through composition.
- **`List.of()` factory method** — Java 9+ introduced convenient factory methods on the `List` interface. `List.of(...)` creates an immutable list from the provided elements. This is used in the driver to create the account list without boilerplate.
- **`DecimalFormat` for formatted output** — Java's `DecimalFormat` class provides locale-sensitive number formatting. The pattern `#,##0.00` formats numbers with commas as thousands separators and exactly two decimal places. This produces the comma-formatted output shown in the expected results (e.g., `$1,000.00`), which differs from the other languages that use `toFixed(2)`.
- **Package structure** — In a real project, each participant would typically live in its own file within a package structure (e.g., `com.bank.factory`, `com.bank.products`). The example demonstrates everything in a single file for copy-paste convenience. The public class `AbstractFactoryDemo` must match the filename `AbstractFactoryDemo.java`.
