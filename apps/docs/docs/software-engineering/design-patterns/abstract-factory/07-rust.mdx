## Implementation

Below is a complete Rust 2021 edition implementation of the Abstract Factory pattern using a banking domain. The `BankFactory` trait defines three creation methods, and two concrete factory structs (`PersonalBankingFactory` and `BusinessBankingFactory`) produce families of related account objects. Rust's trait system enforces the contract at compile time: any struct that implements `BankFactory` must provide concrete implementations for all three creation methods. The client code depends only on the `BankFactory` trait and `Account` trait for polymorphism via `Box<dyn Account>`.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌────────────────────────────────────────────────────────────────┐
//   │   trait BankFactory                                            │
//   ├────────────────────────────────────────────────────────────────┤
//   │ + fn create_savings_account(&self, &str, f64) -> Box<dyn Act>  │  ← OOP: Abstraction
//   │ + fn create_checking_account(&self, &str, f64) -> Box<dyn Act> │     ISP: focused
//   │ + fn create_loan_account(&self, &str, f64) -> Box<dyn Account> │     trait methods
//   └───────────────┬──────────────────────────┬─────────────────────┘
//                   │ impl for                 │ impl for
//   ┌───────────────┴──────────────┐  ┌────────┴─────────────────────┐
//   │ PersonalBankingFactory       │  │ BusinessBankingFactory       │
//   │ ── creates Personal          │  │ ── creates Business          │  ← OOP: Polymorphism
//   │    product family            │  │    product family             │     LSP: substitutable
//   └───────────────┬──────────────┘  └────────┬─────────────────────┘
//                   │                          │
//                   ▼                          ▼
//   ┌───────────────────────────┐  ┌────────────────────────────┐
//   │ PersonalSavingsAccount    │  │ BusinessSavingsAccount     │
//   │ PersonalCheckingAccount   │  │ BusinessCheckingAccount    │  ← OOP: Trait impl
//   │ PersonalLoanAccount       │  │ BusinessLoanAccount        │     (impl Account)
//   └───────────────┬───────────┘  └────────┬───────────────────┘
//                   │                       │
//                   └──────┬────────────────┘
//                          │ impl for
//                   ┌──────┴──────────┐
//                   │ trait Account   │  ← OOP: Abstraction
//                   ├─────────────────┤     ISP: minimal interface
//                   │ fn get_details()│
//                   │ fn get_balance()│
//                   │ fn get_monthly()│
//                   └─────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `BankFactory` and `Account` are Rust traits that define the contracts. Client code depends on these traits via `&dyn BankFactory` and `Box<dyn Account>`, never on concrete struct types.
2. **Encapsulation** — Each account struct uses private fields (by default in Rust, all fields are private to the module). Access is provided only through the trait methods.
3. **Trait implementation** — Instead of classical inheritance, Rust uses trait implementation. Each concrete account struct implements the `Account` trait. Each concrete factory struct implements the `BankFactory` trait.
4. **Polymorphism** — The driver stores accounts in a `Vec<Box<dyn Account>>` and calls methods through trait object dispatch. Rust's vtable mechanism ensures the correct implementation runs per concrete type at runtime.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` creates only personal banking products. `BusinessBankingFactory` creates only business banking products. Each account struct represents exactly one product type.
2. **OCP** — Adding a `PremiumBankingFactory` struct requires no changes to the `BankFactory` trait, existing factory structs, or existing account structs.
3. **LSP** — Every concrete account type can substitute `Box<dyn Account>` without breaking the client. The driver treats all accounts uniformly through the `Account` trait.
4. **ISP** — The `Account` trait is minimal: three methods. The `BankFactory` trait has three methods. No struct implements methods it does not need.
5. **DIP** — Client code depends on the `BankFactory` and `Account` traits, never on concrete types like `PersonalSavingsAccount`.

### Account

```rust
// Abstract Factory Pattern — Rust 2021 — Banking example
// ======================================================
// Participants: BankFactory (AbstractFactory), PersonalBankingFactory
//   and BusinessBankingFactory (ConcreteFactory), Account (AbstractProduct),
//   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
//   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
//   (ConcreteProducts)
// OOP: abstraction, encapsulation, trait implementation, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ======================================================
// ── Abstract Product Trait ──
// OOP role: Abstraction + ISP (defines the contract for all account types)
// SOLID: ISP (minimal interface), DIP (clients depend on this)
trait Account {
    fn get_details(&self) -> String;
    fn get_balance(&self) -> f64;
    fn get_monthly_fee(&self) -> f64;
}
```

### PersonalSavingsAccount

```rust
// ── Concrete Product Family 1: Personal Banking Accounts ──
// OOP role: Trait implementation (impl Account) + Polymorphism
// SOLID: LSP (substitutable for dyn Account), OCP (extensible), SRP (one responsibility)
struct PersonalSavingsAccount {
    holder: String,
    balance: f64,
    interest_rate: f64,
    monthly_fee: f64,
}

impl PersonalSavingsAccount {
    fn new(holder: &str, initial_deposit: f64) -> Self {
        Self {
            holder: holder.to_string(),
            balance: initial_deposit,
            interest_rate: 3.5,
            monthly_fee: 5.0,
        }
    }
}

impl Account for PersonalSavingsAccount {
    fn get_details(&self) -> String {
        format!(
            "Personal Savings[{}]: ${:.2} @ {}% APY",
            self.holder, self.balance, self.interest_rate
        )
    }
    fn get_balance(&self) -> f64 {
        self.balance
    }
    fn get_monthly_fee(&self) -> f64 {
        self.monthly_fee
    }
}
```

### BankFactory

```rust
// ── Abstract Factory Trait ──
// OOP role: Abstraction + Encapsulation (encapsulates product family creation)
// SOLID: SRP (creates accounts only), DIP (clients depend on this abstraction)
trait BankFactory {
    fn create_savings_account(&self, holder: &str, amount: f64) -> Box<dyn Account>;
    fn create_checking_account(&self, holder: &str, amount: f64) -> Box<dyn Account>;
    fn create_loan_account(&self, holder: &str, amount: f64) -> Box<dyn Account>;
}
```

### PersonalBankingFactory

```rust
// ── Concrete Factory 1: PersonalBankingFactory ──
// OOP role: Polymorphism (impl BankFactory trait)
```

### displayAccounts

```rust
// ── Client / Composition Root ──
// OOP role: Composition over inheritance
// SOLID: DIP (depends on BankFactory and Account trait abstractions), SRP (orchestrates only)
fn display_accounts(label: &str, factory: &dyn BankFactory) {
    println!("=== {} ===", label);
    let accounts: Vec<Box<dyn Account>> = vec![
        factory.create_savings_account("Alice", 1000.0),
        factory.create_checking_account("Bob", 2500.0),
        factory.create_loan_account("Carol", 50000.0),
    ];

    for account in &accounts {
        println!("  {}", account.get_details());
        println!(
            "    Balance: ${:.2}, Monthly Fee: ${:.2}",
            account.get_balance(),
            account.get_monthly_fee()
        );
    }
    println!();
}
```

### Main / Driver

```rust
// ── Main / Driver ──
fn main() {
    let personal_factory = PersonalBankingFactory;
    let business_factory = BusinessBankingFactory;

    display_accounts("Personal Banking Accounts", &personal_factory);
    display_accounts("Business Banking Accounts", &business_factory);

    // Demonstrate family consistency — all accounts from the same factory
    // share the same fee structure and product family characteristics
    println!("=== Cross-Family Comparison ===");
    let personal_savings: Box<dyn Account> =
        personal_factory.create_savings_account("Alice", 1000.0);
    let business_savings: Box<dyn Account> =
        business_factory.create_savings_account("Acme Corp", 50000.0);

    println!("Personal Savings monthly fee: ${:.2}", personal_savings.get_monthly_fee());
    println!("Business Savings monthly fee: ${:.2}", business_savings.get_monthly_fee());

    let personal_loan: Box<dyn Account> =
        personal_factory.create_loan_account("Carol", 50000.0);
    let business_loan: Box<dyn Account> =
        business_factory.create_loan_account("Initech", 500000.0);

    println!("Personal Loan details: {}", personal_loan.get_details());
    println!("Business Loan details: {}", business_loan.get_details());
}
```

### Language-Specific Notes

- **Traits instead of interfaces** — Rust uses traits (not interfaces or abstract classes) to define shared behavior. A trait is a collection of method signatures that types can implement. Unlike Java interfaces, Rust traits can also provide default implementations. The `Account` and `BankFactory` traits define the contracts for the Abstract Factory pattern.
- **`Box<dyn Trait>` for polymorphism** — Rust achieves runtime polymorphism through trait objects (`Box<dyn Account>` or `&dyn Account`). The `dyn` keyword indicates dynamic dispatch via a vtable. This is the Rust equivalent of Java's interface reference or C#'s interface variable. Static dispatch via generics (`impl Account`) is also possible and is more performant, but `Box<dyn Account>` provides the runtime polymorphism that the Abstract Factory pattern requires.
- **Zero-sized structs for factories** — The factory structs (`PersonalBankingFactory`, `BusinessBankingFactory`) are zero-sized types (no fields). They exist only to implement the `BankFactory` trait. Rust allows instantiating them without allocating heap memory, making them free abstractions. This is idiomatic in Rust for stateless types that serve as trait implementors.
- **Ownership and borrowing** — The factory methods return `Box<dyn Account>`, which is a heap-allocated trait object. This transfers ownership of the account to the caller. The `display_accounts` function takes `&dyn BankFactory` (a borrowed reference to a trait object) to avoid taking ownership of the factory. This is Rust's way of expressing the Abstract Factory's client-factory relationship without garbage collection.
- **No `new` keyword** — Rust uses `fn new()` as a convention, not a language keyword. The `new` associated function is a regular function that returns `Self`. It is called with `Type::new(...)` syntax, not `new Type(...)`. This is idiomatic Rust and follows the standard library convention.
