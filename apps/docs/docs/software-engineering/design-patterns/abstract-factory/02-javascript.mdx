## Implementation

Below is a complete JavaScript (ES2022+) implementation of the Abstract Factory pattern using a banking domain. The `BankFactory` abstract factory creates families of related account objects: `PersonalBankingFactory` produces accounts with personal fee structures (low fees, low minimum balances), while `BusinessBankingFactory` produces accounts with business fee structures (higher fees, higher minimum balances). The client code works solely through the abstract `Account` interface and `BankFactory` interface, never depending on concrete product types.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌─────────────────────────────────────────────────┐
//   │   BankFactory (Abstract class — interface role) │
//   ├─────────────────────────────────────────────────┤
//   │ + createSavingsAccount(holder, amount): Account │  ← OOP: Abstraction
//   │ + createCheckingAccount(holder, amount): Account│     ISP: focused
//   │ + createLoanAccount(holder, amount): Account    │     creation methods
//   └─────────────┬────────────────────┬──────────────┘
//                 │ extends            │ extends
//   ┌─────────────┴─────────────┐ ┌───┴──────────────────┐
//   │ PersonalBankingFactory    │ │ BusinessBankingFactory│
//   │ ── creates Personal       │ │ ── creates Business   │ ← OOP: Polymorphism
//   │    product family         │ │    product family     │    LSP: both
//   └─────────────┬─────────────┘ └───┬──────────────────┘    substitutable
//                 │                   │
//                 ▼                   ▼
//   ┌────────────────────────┐  ┌────────────────────────┐
//   │ PersonalSavingsAccount │  │ BusinessSavingsAccount │
//   │ PersonalCheckingAccount│  │ BusinessCheckingAccount│ ← OOP: Inheritance
//   │ PersonalLoanAccount    │  │ BusinessLoanAccount    │    (extends Account)
//   └─────────────┬──────────┘  └───┬────────────────────┘
//                 │                  │
//                 └──────┬───────────┘
//                        │ implements
//                 ┌──────┴──────────┐
//                 │ Account         │ ← OOP: Abstraction
//                 ├─────────────────┤    ISP: minimal interface
//                 │ getDetails()    │
//                 │ getBalance()    │
//                 └─────────────────┘
//
//   ┌─────────────────────────────────────────────────┐
//   │ Client (main driver)                            │ ← SRP: orchestration only
//   │ ── depends on BankFactory and Account           │    DIP: depends on
//   │    never on concrete types                      │    abstractions
//   └─────────────────────────────────────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `BankFactory` and `Account` are base classes that define the contract. The client code depends on these abstractions, not on the concrete factory or product classes.
2. **Encapsulation** — Each account class hides its internal state (holder name, balance, fee structure) behind `getDetails()`, `getBalance()`, and `getMonthlyFee()` methods. Private fields (`#`) enforce true encapsulation in ES2022+.
3. **Inheritance** — `PersonalBankingFactory` and `BusinessBankingFactory` extend `BankFactory`. All six concrete account classes extend the `Account` base class and share the common interface.
4. **Polymorphism** — The driver stores all accounts in a single array typed as `Account` and calls the same methods on each. JavaScript's dynamic dispatch ensures the correct implementation runs per type.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` only creates personal banking products. `BusinessBankingFactory` only creates business banking products. Each account class represents only its own type. No class does more than one thing.
2. **OCP** — Adding a new `PremiumBankingFactory` class requires zero changes to the existing factory or account classes. The system is open for extension, closed for modification.
3. **LSP** — Every concrete account type can substitute `Account` without breaking the client. The driver treats all accounts uniformly through the `Account` interface.
4. **ISP** — The `Account` interface is minimal: `getDetails()`, `getBalance()`, `getMonthlyFee()`. No class is forced to implement methods it does not need.
5. **DIP** — Client code depends on the abstract `BankFactory` and `Account` types. The factory receives no concrete product references, and the driver iterates over `Account` objects without checking concrete types.

### Account

```javascript
// Abstract Factory Pattern — JavaScript (ES2022+) — Banking example
// ===================================================================
// Participants: BankFactory (AbstractFactory), PersonalBankingFactory
//   and BusinessBankingFactory (ConcreteFactory), Account (AbstractProduct),
//   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
//   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
//   (ConcreteProducts)
// OOP: abstraction, encapsulation, inheritance, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ===================================================================
// ── Abstract Product: Account ──
// OOP role: Abstraction + ISP (defines the contract for all account types)
// SOLID: ISP (minimal interface), DIP (clients depend on this)
class Account {
  constructor() {
    if (new.target === Account) {
      throw new Error('Account is abstract — cannot instantiate directly');
    }
  }

  getDetails() {
    throw new Error('Subclass must implement getDetails()');
  }

  getBalance() {
    throw new Error('Subclass must implement getBalance()');
  }

  getMonthlyFee() {
    throw new Error('Subclass must implement getMonthlyFee()');
  }
}
```

### PersonalSavingsAccount

```javascript
// ── Concrete Product Family 1: Personal Banking Accounts ──
// OOP role: Inheritance + Polymorphism
// SOLID: LSP (substitutable for Account), OCP (extensible), SRP (one responsibility)
class PersonalSavingsAccount extends Account {
  #holder;
  #balance;
  #interestRate;
  #monthlyFee;

  constructor(holder, initialDeposit) {
    super();
    this.#holder = holder;
    this.#balance = initialDeposit;
    this.#interestRate = 3.5;
    this.#monthlyFee = 5.0;
  }

  getDetails() {
    return `Personal Savings[${this.#holder}]: $${this.#balance.toFixed(2)} @ ${this.#interestRate}% APY`;
  }

  getBalance() {
    return this.#balance;
  }

  getMonthlyFee() {
    return this.#monthlyFee;
  }
}
```

### displayAccounts

```javascript
// ── Client / Composition Root ──
// OOP role: Composition over inheritance
// SOLID: DIP (depends on BankFactory and Account abstractions), SRP (orchestrates only)
function displayAccounts(label, factory) {
  console.log(`=== ${label} ===`);
  const accounts = [
    factory.createSavingsAccount('Alice', 1000.0),
    factory.createCheckingAccount('Bob', 2500.0),
    factory.createLoanAccount('Carol', 50000.0),
  ];

  for (const account of accounts) {
    console.log(`  ${account.getDetails()}`);
    console.log(`    Balance: $${account.getBalance().toFixed(2)}, Monthly Fee: $${account.getMonthlyFee().toFixed(2)}`);
  }
  console.log('');
}
```

### Main / Driver

```javascript
// ── Main / Driver ──
function main() {
  const personalFactory = new PersonalBankingFactory();
  const businessFactory = new BusinessBankingFactory();

  displayAccounts('Personal Banking Accounts', personalFactory);
  displayAccounts('Business Banking Accounts', businessFactory);

  // Demonstrate family consistency — all accounts from the same factory
  // share the same fee structure and product family characteristics
  console.log('=== Cross-Family Comparison ===');
  const personalSavings = personalFactory.createSavingsAccount('Alice', 1000.0);
  const businessSavings = businessFactory.createSavingsAccount('Acme Corp', 50000.0);

  console.log(`Personal Savings monthly fee: $${personalSavings.getMonthlyFee().toFixed(2)}`);
  console.log(`Business Savings monthly fee: $${businessSavings.getMonthlyFee().toFixed(2)}`);

  const personalLoan = personalFactory.createLoanAccount('Carol', 50000.0);
  const businessLoan = businessFactory.createLoanAccount('Initech', 500000.0);

  console.log(`Personal Loan APR: ${personalLoan.getDetails().match(/[\d.]+% APR/)[0]}`);
  console.log(`Business Loan APR: ${businessLoan.getDetails().match(/[\d.]+% APR/)[0]}`);
}

main();
```

### Language-Specific Notes

- **OOP via prototype chain** — JavaScript uses prototypal inheritance under the hood, but ES2022 `class` syntax provides a clean interface familiar to OOP developers. The `extends` keyword connects the prototype chain automatically. The `new.target` check in constructors simulates abstract class behavior that is not natively supported.
- **Private fields** — The `#` prefix syntax (`#holder`, `#balance`) provides true hard-private fields in ES2022+, enforced at the engine level. This is a significant improvement over the older `_` convention which was only a naming convention and did not prevent access.
- **No interface keyword** — JavaScript does not have an `interface` keyword. The pattern uses abstract base classes with methods that throw `Error` if not overridden. This is the idiomatic way to enforce an interface contract in JavaScript. TypeScript (see next page) provides true interface support.
- **Dynamic dispatch** — JavaScript's runtime polymorphism works via the prototype chain. When `account.getDetails()` is called, the engine walks the prototype chain from the instance up to `Account.prototype` and calls the first matching `getDetails`. This is how LSP is achieved without static type checking.
- **Factory as object** — In JavaScript, the factory does not need to be a class. A factory object literal implementing the same interface is equally valid and often more concise. The class-based approach is used here for consistency with OOP terminology and cross-language comparison.
