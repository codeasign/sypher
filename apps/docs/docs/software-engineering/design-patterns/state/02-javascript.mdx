The State pattern allows an object to alter its behavior when its internal state changes. This JavaScript implementation models a stock market order lifecycle, where an Order transitions through states such as New, Submitted, PartiallyFilled, Filled, Cancelled, and Rejected. Each concrete state encapsulates its own validation rules and transition logic, keeping the Order context clean and maintainable while enforcing the legal state graph of a real trading system.

## How the Implementation Works

### Class Diagram in Pseudocode

```text
class OrderState {
  handleSubmit(order)       — return error string by default
  handleFill(order,qty,price) — return error string by default
  handleCancel(order)       — return error string by default
  handleReject(order,reason) — return error string by default
  getStatusLabel()          — "Unknown"
  getAllowedOperations()    — []
}

class NewState extends OrderState {
  handleSubmit()       — transition to SubmittedState
  handleCancel()       — transition to CancelledState
  getStatusLabel()     — "New"
  getAllowedOps()      — ["submit", "cancel"]
}

class SubmittedState extends OrderState {
  handleFill(qty,prc)  — transition to PartiallyFilled or Filled
  handleCancel()       — transition to CancelledState
  handleReject(reason) — transition to RejectedState
  getStatusLabel()     — "Submitted"
  getAllowedOps()      — ["fill", "cancel"]
}

class PartiallyFilledState extends OrderState {
  handleFill(qty,prc)  — transition to Filled if fully filled
  handleCancel()       — transition to CancelledState
  getStatusLabel()     — "PartiallyFilled"
  getAllowedOps()      — ["fill", "cancel"]
}

class FilledState extends OrderState {
  getStatusLabel()     — "Filled"
  getAllowedOps()      — []
  // terminal state: all operations inherit error behavior
}

class CancelledState extends OrderState {
  getStatusLabel()     — "Cancelled"
  getAllowedOps()      — []
  // terminal state: all operations inherit error behavior
}

class RejectedState extends OrderState {
  getStatusLabel()     — "Rejected"
  getAllowedOps()      — []
  // terminal state: all operations inherit error behavior
}

class Order {
  #symbol, #quantity, #filledQuantity, #price, #state
  submit()             delegates to state
  fill(qty, prc)       delegates to state
  cancel()             delegates to state
  reject(reason)       delegates to state
  getStatus()          state.getStatusLabel()
  setState(s)          update internal state
  addFill(qty, price)  increment filled quantity
  isFullyFilled()      filledQuantity >= quantity
}
```

### OOP Concepts Applied

- **Abstraction** — The `OrderState` base class defines the contract; callers interact through the method interface without ever knowing the concrete state class name.
- **Encapsulation** — Each concrete state owns its own transition rules and validation logic. The `Order` class uses private fields (`#symbol`, `#quantity`, `#filledQuantity`, `#price`) to protect internal data from external mutation.
- **Inheritance** — Concrete states inherit default error-returning behavior from `OrderState` and override only the operations they support. Terminal states override nothing and rely on the base class for all method implementations.
- **Polymorphism** — The `Order` delegates every public method call to `this.#state` without knowing which concrete state is active; each state provides its own behavior for the same set of method signatures.

### SOLID Principles Applied

- **Single Responsibility** — Each state class handles exactly the operations valid for its lifecycle phase. `NewState` only manages submission and pre-submission cancellation; `SubmittedState` manages fills, cancellations, and rejections.
- **Open/Closed** — New states (e.g., `ExpiredState`, `StoppedState`) can be added by creating a new subclass of `OrderState` without modifying any existing state class or the `Order` context.
- **Liskov Substitution** — Any `OrderState` subclass can replace the base class in the `Order` context without breaking correctness, because each subclass respects the same method signatures and return types.
- **Interface Segregation** — The `OrderState` base class provides focused, state-specific methods (`handleSubmit`, `handleFill`, `handleCancel`, `handleReject`) rather than a single monolithic method that checks state internally.
- **Dependency Inversion** — `Order` depends on the abstract `OrderState` contract, never on concrete state classes. States are discovered at runtime and swapped transparently.

```javascript
// ── OrderState (abstract base class) ──────────────────────────
class OrderState {
  handleSubmit(order) {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot submit — order is already ${status}`;
  }
  handleFill(order, quantity, price) {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot fill — order is already ${status}`;
  }
  handleCancel(order) {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot cancel — order is already ${status}`;
  }
  handleReject(order, reason) {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot reject — order is already ${status}`;
  }
  getStatusLabel() {
    return 'Unknown';
  }
  getAllowedOperations() {
    return [];
  }
}
```

### NewState

```javascript
class NewState extends OrderState {
  handleSubmit(order) {
    const symbol = order.symbol;
    order.setState(new SubmittedState());
    return `${symbol}: New -> ${order.getStatus()}`;
  }
  handleCancel(order) {
    const symbol = order.symbol;
    order.setState(new CancelledState());
    return `${symbol}: New -> ${order.getStatus()}`;
  }
  getStatusLabel() {
    return 'New';
  }
  getAllowedOperations() {
    return ['submit', 'cancel'];
  }
}
```

### SubmittedState

```javascript
class SubmittedState extends OrderState {
  handleFill(order, quantity, price) {
    const symbol = order.symbol;
    const remaining = order.remaining;
    if (quantity >= remaining) {
      order.addFill(remaining, price);
      order.setState(new FilledState());
      return `${symbol}: Filled ${order.filledQuantity} shares at $${price.toFixed(2)}`;
    }
    order.addFill(quantity, price);
    order.setState(new PartiallyFilledState());
    return `${symbol}: Partially filled ${order.filledQuantity}/${order.quantity} shares at $${price.toFixed(2)}`;
  }
  handleCancel(order) {
    const symbol = order.symbol;
    order.setState(new CancelledState());
    return `${symbol}: Submitted -> ${order.getStatus()}`;
  }
  handleReject(order, reason) {
    const symbol = order.symbol;
    order.setState(new RejectedState());
    return `${symbol}: Rejected — ${reason}`;
  }
  getStatusLabel() {
    return 'Submitted';
  }
  getAllowedOperations() {
    return ['fill', 'cancel'];
  }
}
```

### Order (Context)

```javascript
// ── Order (Context) ───────────────────────────────────────────

class Order {
  #symbol;
  #quantity;
  #filledQuantity;
  #price;
  #state;

  constructor(symbol, quantity, price) {
    this.#symbol = symbol;
    this.#quantity = quantity;
    this.#filledQuantity = 0;
    this.#price = price;
    this.#state = new NewState();
  }

  get symbol() {
    return this.#symbol;
  }
  get quantity() {
    return this.#quantity;
  }
  get filledQuantity() {
    return this.#filledQuantity;
  }
  get price() {
    return this.#price;
  }
  get remaining() {
    return this.#quantity - this.#filledQuantity;
  }

  setState(state) {
    this.#state = state;
  }
  addFill(qty, price) {
    this.#filledQuantity += qty;
    this.#price = price;
  }
  getStatus() {
    return this.#state.getStatusLabel();
  }
  getAllowedOperations() {
    return this.#state.getAllowedOperations();
  }
  isFullyFilled() {
    return this.#filledQuantity >= this.#quantity;
  }

  submit() {
    return this.#state.handleSubmit(this);
  }
  fill(quantity, price) {
    return this.#state.handleFill(this, quantity, price);
  }
  cancel() {
    return this.#state.handleCancel(this);
  }
  reject(reason) {
    return this.#state.handleReject(this, reason);
  }
}
```

### Main / Driver

```javascript
// ── Driver ────────────────────────────────────────────────────

console.log('--- Order 1: AAPL 100 @ $175.50 ---');
const order1 = new Order('AAPL', 100, 175.5);
console.log(order1.submit());
console.log(order1.fill(100, 175.5));
console.log(order1.cancel());

console.log('');
console.log('--- Order 2: GOOGL 50 @ $140.25 ---');
const order2 = new Order('GOOGL', 50, 140.25);
console.log(order2.submit());
console.log(order2.fill(20, 140.25));
console.log(order2.cancel());

console.log('');
console.log('--- Order 3: MSFT 75 @ $378.80 ---');
const order3 = new Order('MSFT', 75, 378.8);
console.log(order3.submit());
console.log(order3.reject('Insufficient funds'));

console.log('');
console.log('=== Final States ===');
console.log(`AAPL: ${order1.getStatus()}`);
console.log(`GOOGL: ${order2.getStatus()}`);
console.log(`MSFT: ${order3.getStatus()}`);
```

## Language-Specific Notes

- JavaScript uses prototypal inheritance under the hood; the `class` and `extends` keywords provide syntactic sugar over prototype chains while making the state hierarchy explicit and familiar to developers from class-based languages.
- Private fields (`#field`) enforce true encapsulation at the language level — they are not accessible outside the class, even through reflection, protecting the Order's internal data (filled quantity, price) from accidental external mutation.
- The base `OrderState` class serves as a virtual interface by providing default error-returning method bodies that concrete states override. JavaScript has no built-in `abstract` or `interface` keyword, making this a common idiomatic pattern.
- Returned error strings (rather than thrown exceptions) keep the driver flow linear and avoid noisy try/catch blocks around every state transition in the example.
- Dynamic typing simplifies state comparison and transition logic but requires discipline to maintain correctness; the explicit class hierarchy compensates for the absence of a compile-time type checker.
- Template literals with embedded expressions (`${...}`) and the `.toFixed(2)` method produce clean, formatted price output without manual string concatenation.
