The Iterator pattern provides a way to access elements of a collection sequentially without exposing the underlying representation. In the stock market domain, traders and analysts regularly traverse price histories, filter trades by volume or date range, and apply custom traversal logic. The Iterator pattern decouples the traversal algorithm from the collection itself, making it easy to swap iteration strategies and add new filtering capabilities without modifying the core data structures.

### OOP Concepts Applied

- **Encapsulation**: The `PriceHistoryIterator` hides its internal position counter and price list. The `PriceHistory` aggregate encapsulates its data behind `add_price()` and `create_iterator()` methods.
- **Abstraction**: The `StockIterator` and `StockPriceCollection` abstract base classes define clean interfaces without exposing implementation details.
- **Polymorphism**: Both `PriceHistoryIterator` and the decorator filters (`HighVolumeFilter`, `DateRangeFilter`) implement the same `StockIterator` interface, allowing them to be used interchangeably.
- **Inheritance**: `PriceHistoryIterator` inherits from `StockIterator`, and `PriceHistory` inherits from `StockPriceCollection`.

### SOLID Principles Applied

- **SRP (Single Responsibility)**: `PriceHistory` manages data storage; `PriceHistoryIterator` handles traversal; each filter has a single filtering responsibility.
- **OCP (Open/Closed)**: The system is open for extension -- new filter iterators (e.g., `PriceRangeFilter`, `MovingAverageFilter`) can be added without modifying existing code.
- **LSP (Liskov Substitution)**: Any `StockIterator` implementation can replace another without breaking the client code. The decorator filters are transparent to the client.
- **ISP (Interface Segregation)**: The `StockIterator` interface defines only two methods (`has_next`, `next`), keeping it focused and minimal.
- **DIP (Dependency Inversion)**: High-level code depends on the abstract `StockIterator` and `StockPriceCollection` interfaces, not concrete implementations.

### StockIterator (Iterator Protocol (ABC))

```python
# в”Җв”Җ Iterator Protocol (ABC) в”Җв”Җ
class StockIterator(ABC):
    """Abstract Iterator -- defines traversal interface"""
    @abstractmethod
    def has_next(self) -> bool:
        pass

    @abstractmethod
    def next(self) -> StockPrice:
        pass
```

### PriceHistoryIterator (Concrete Iterator)

```python
# в”Җв”Җ Concrete Iterator в”Җв”Җ
class PriceHistoryIterator(StockIterator):
    """Iterates through stock prices in order"""

    def __init__(self, prices: List[StockPrice]):
        self._prices = prices
        self._position = 0

    def has_next(self) -> bool:
        return self._position < len(self._prices)

    def next(self) -> StockPrice:
        if not self.has_next():
            raise StopIteration("No more stock prices")
        price = self._prices[self._position]
        self._position += 1
        return price

    def reset(self) -> None:
        self._position = 0
```

### PriceHistory (Concrete Aggregate)

```python
# в”Җв”Җ Concrete Aggregate в”Җв”Җ
class PriceHistory(StockPriceCollection):
    """Manages a collection of stock prices"""

    def __init__(self, symbol: str, prices: Optional[List[StockPrice]] = None):
        self.symbol = symbol
        self._prices = prices if prices is not None else []

    def add_price(self, price: StockPrice) -> None:
        self._prices.append(price)

    def price_count(self) -> int:
        return len(self._prices)

    def create_iterator(self) -> StockIterator:
        return PriceHistoryIterator(self._prices)

    # Pythonic iteration: __iter__ returns a generator
    def __iter__(self) -> Iterator[StockPrice]:
        for price in self._prices:
            yield price
```

### HighVolumeFilter (Decorator: Filtered Iterator (OCP))

```python
# в”Җв”Җ Decorator: Filtered Iterator (OCP) в”Җв”Җ
class HighVolumeFilter(StockIterator):
    """Filters stock prices by minimum volume"""

    def __init__(self, iterator: StockIterator, min_volume: int):
        self._iterator = iterator
        self._min_volume = min_volume
        self._buffer: Optional[StockPrice] = None

    def has_next(self) -> bool:
        while self._buffer is None and self._iterator.has_next():
            candidate = self._iterator.next()
            if candidate.volume >= self._min_volume:
                self._buffer = candidate
        return self._buffer is not None

    def next(self) -> StockPrice:
        if not self.has_next():
            raise StopIteration("No more matching prices")
        result = self._buffer
        self._buffer = None
        return result
```

### DateRangeFilter (Decorator: Date Range Filter (OCP))

```python
# в”Җв”Җ Decorator: Date Range Filter (OCP) в”Җв”Җ
class DateRangeFilter(StockIterator):
    """Filters stock prices within a date range"""

    def __init__(self, iterator: StockIterator, start_date: str, end_date: str):
        self._iterator = iterator
        self._start_date = start_date
        self._end_date = end_date
        self._buffer: Optional[StockPrice] = None

    def has_next(self) -> bool:
        while self._buffer is None and self._iterator.has_next():
            candidate = self._iterator.next()
            if self._start_date <= candidate.date <= self._end_date:
                self._buffer = candidate
        return self._buffer is not None

    def next(self) -> StockPrice:
        if not self.has_next():
            raise StopIteration("No more prices in date range")
        result = self._buffer
        self._buffer = None
        return result
```

### Language-Specific Notes

- Python's `__iter__` and `__next__` dunder methods provide native iterator protocol support, making any custom iterator automatically compatible with `for` loops and list comprehensions.
- The `abc.ABC` and `@abstractmethod` decorator enable clean interface definitions, enforcing that subclasses implement the required traversal methods.
- Python's generator functions (`yield`) provide a lightweight way to implement iterators without writing a separate iterator class, as demonstrated in `PriceHistory.__iter__`.
- The `@dataclass` decorator simplifies data class creation by auto-generating `__init__`, `__repr__`, and `__eq__` methods, making `StockPrice` concise and readable.
- The decorator-style iterator pattern (wrapping one iterator with another) is particularly idiomatic in Python, where generator expressions and `itertools` module provide similar filtering capabilities.
- Python's duck typing means the `StockIterator` ABC is not strictly required -- any object with `has_next()` and `next()` methods would work, but the ABC improves documentation and type safety.
