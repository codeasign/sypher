## Intent

Define the skeleton of an algorithm in a base class method, deferring the definition of some steps to subclasses. The Template Method pattern lets subclasses redefine certain steps of an algorithm without changing the algorithm's structure. It is one of the most fundamental behavioral patterns and embodies the Hollywood Principle: "Don't call us, we'll call you."

## The Problem Without This Pattern

Imagine you are building a stock trading system that supports multiple strategies. A moving-average crossover strategy, a mean-reversion strategy, and a momentum strategy all follow the same high-level workflow: fetch current market data, run an analysis, compute position size, validate risk, place the order, and log the result. Without the Template Method pattern, each strategy class duplicates the algorithm skeleton, and the ordering logic is scattered across implementations. If a new regulatory requirement forces you to add a compliance check after position sizing, you must touch every strategy class independently — a textbook violation of the Open/Closed Principle.

```python
# BROKEN: algorithm skeleton duplicated in every strategy
class MovingAverageCrossover:
    def execute(self, ticker):
        data = self.fetch_data(ticker)
        signal = self.compute_crossover(data)
        shares = self.calculate_size(signal)
        self.validate(shares)
        self.place_order(ticker, shares)
        self.log(ticker, signal, shares)

class MeanReversion:
    def execute(self, ticker):
        data = self.fetch_data(ticker)
        signal = self.compute_reversion(data)
        shares = self.calculate_size(signal)
        self.validate(shares)
        self.place_order(ticker, shares)
        self.log(ticker, signal, shares)
        # The entire execute method is duplicated
```

Every new strategy means copying the same six-step sequence. Any change to the workflow (add a step, reorder checks, insert auditing) requires editing every class. The Template Method pattern solves this by pulling the invariant skeleton into a single base class and letting subclasses implement only the variant steps.

### Participants

- **AbstractClass** — Declares the `templateMethod()` which calls primitive operations and hooks. The template method is typically declared `final` (or sealed) to prevent subclasses from changing the algorithm structure. Primitive operations are declared abstract; hooks are declared with a default (often empty) body.
- **ConcreteClass** — Implements the primitive operations required by the algorithm. May also override one or more hooks to inject additional behavior at specific points in the skeleton.

## Pseudocode — Stock Market Trading Strategies

```
// PSEUDOCODE — Template Method in Stock Market context
// Participants labeled with [ROLE]

[ABSTRACTCLASS] TradingStrategy
  // Template Method — defines the skeleton of a trade execution
  method execute_trade(ticker, price_data):
    // Step 1: Analyze raw market data
    signal, confidence = analyze_market(price_data)

    // Step 2: Hook — optional pre-trade validation
    if not pre_validate(signal, confidence):
      log("Trade rejected by pre-validation")
      return

    // Step 3: Calculate how many shares to trade
    shares = calculate_position_size(confidence, MAX_POSITION)

    // Step 4: Hook — apply risk management filters
    if not apply_risk_controls(ticker, shares):
      log("Trade rejected by risk controls")
      return

    // Step 5: Place the order on the exchange
    order = place_order(ticker, shares, signal)

    // Step 6: Log the completed trade
    log_trade(ticker, shares, signal, confidence, order)

  // Primitive operations — must be overridden
  abstract method analyze_market(price_data) -> (signal, confidence)
  abstract method calculate_position_size(confidence, max_pos) -> integer

  // Hooks — optional overrides with default behavior
  method pre_validate(signal, confidence) -> boolean:
    return confidence > MIN_CONFIDENCE  // default threshold

  method apply_risk_controls(ticker, shares) -> boolean:
    portfolio = get_portfolio()
    return portfolio.can_allocate(shares * ticker.current_price)

  // Concrete operations — shared by all strategies
  method place_order(ticker, shares, signal) -> Order:
    return Order(ticker, shares, signal)

  method log_trade(ticker, shares, signal, confidence, order):
    print("[{timestamp}] {ticker}: {signal} {shares} shares "
          "(confidence: {confidence:.2f}) — Order#{order.id}")

[CONCRETECLASS] MovingAverageCrossover extends TradingStrategy
  method analyze_market(price_data):
    short_ma = average(price_data.close_prices[-SHORT_WINDOW:])
    long_ma = average(price_data.close_prices[-LONG_WINDOW:])
    if short_ma > long_ma:
      return BUY, abs(short_ma - long_ma) / long_ma
    else:
      return SELL, abs(short_ma - long_ma) / long_ma

  method calculate_position_size(confidence, max_pos):
    return floor(max_pos * confidence)

[CONCRETECLASS] MeanReversion extends TradingStrategy
  method analyze_market(price_data):
    current_price = price_data.latest_close
    mean_price = average(price_data.close_prices[-LOOKBACK:])
    deviation = (mean_price - current_price) / mean_price
    if deviation > THRESHOLD:
      return BUY, deviation
    elif deviation < -THRESHOLD:
      return SELL, abs(deviation)
    else:
      return HOLD, 0

  method calculate_position_size(confidence, max_pos):
    return floor(max_pos * confidence * 0.8)  // more conservative

  // Override hook — mean reversion has tighter risk
  method apply_risk_controls(ticker, shares) -> boolean:
    return super.apply_risk_controls(ticker, shares)
       and shares <= MAX_MEAN_REVERSION_SIZE

// ── Client code ──
strategies = [MovingAverageCrossover(), MeanReversion()]
data = MarketDataProvider.fetch("AAPL")

for strategy in strategies:
  strategy.execute_trade("AAPL", data)
```

## How It Works — Step by Step

1. **Client selects a concrete strategy**, such as `MovingAverageCrossover`, and calls its `execute_trade(ticker, data)` method.
2. **The template method begins execution** in the `TradingStrategy` base class. It starts by calling `analyze_market(data)` — a primitive operation that is dispatched polymorphically to the concrete subclass.
3. **The concrete strategy computes a trading signal.** `MovingAverageCrossover` calculates a short-term and long-term moving average on the price data, compares them, and returns a `(signal, confidence)` pair. The base class has no idea how this computation works — it only knows the return type.
4. **The template method invokes the `pre_validate` hook.** The base class provides a default implementation that checks whether confidence exceeds a minimum threshold. `MovingAverageCrossover` may accept this default, or `MeanReversion` may override it to apply stricter filters.
5. **The template method calls `calculate_position_size`**, another primitive operation. Each concrete strategy computes the number of shares differently based on its risk profile. The base class supplies `MAX_POSITION` as a parameter but the subclass makes the decision.
6. **The template method calls the `apply_risk_controls` hook.** This gives strategies a chance to veto the trade based on portfolio-level constraints, sector exposure limits, or strategy-specific caps.
7. **The template method calls `place_order` and `log_trade`** — concrete methods shared by all strategies. The order is submitted to the exchange and the trade is logged with a consistent format. Control returns to the client.

## When to Use

- Multiple classes follow the same algorithm structure but differ in the implementation of specific steps.
- You want to avoid code duplication when the algorithm skeleton would otherwise be copied across several classes (like the trading strategies above).
- You need to enforce a consistent execution order that subclasses must not alter, while still allowing them to customize individual steps.
- You want to provide extension points (hooks) where subclasses can inject optional behavior without being forced to implement every step.
- The steps of the algorithm are likely to change over time, and you want all changes to flow through a single place (the template method).

## When NOT to Use

- Every implementation of the algorithm would override every abstract method with different logic — consider the Strategy pattern instead, which delegates the entire algorithm rather than individual steps.
- The algorithm has only one or two steps that vary — a simple function parameter or callback may be simpler than a class hierarchy.
- You need to vary the algorithm at runtime — Template Method fixes the structure at compile time through inheritance; use Strategy if you need to swap entire algorithms dynamically.
- The shared algorithm is trivial (fewer than three steps) — the overhead of the abstract class and subclass hierarchy is not justified.

## Real-World Occurrences

- **Java `AbstractList`** — The `iterator()` method is a template method that calls `get()`, `size()`, and optionally `remove()` (a hook). Subclasses only override `get()` and `size()` to get a working iterator.
- **Spring Framework `AbstractApplicationContext`** — The `refresh()` method defines the application context initialization skeleton: `prepareRefresh()`, `obtainFreshBeanFactory()`, `prepareBeanFactory()`, `postProcessBeanFactory()` (a hook), `invokeBeanFactoryPostProcessors()`, `registerBeanPostProcessors()`, `initMessageSource()`, `initApplicationEventMulticaster()`, `onRefresh()` (a hook), `registerListeners()`, `finishBeanFactoryInitialization()`, `finishRefresh()`. Subclasses override hooks to customize behavior.
- **.NET `Stream` class** — The `Read(byte[], int, int)` and `Write(byte[], int, int)` methods are template methods in the abstract `Stream` class. They call the primitive `ReadByte()` and `WriteByte()` methods. Subclasses override the primitive byte-level operations while the higher-level buffer operations remain unchanged.
- **Python `concurrent.futures.Executor`** — The `submit()` method in the abstract base class is a template method that calls `_submit()` (a private abstract method). Concrete implementations like `ThreadPoolExecutor` and `ProcessPoolExecutor` implement `_submit()` while inheriting the wrapping logic for futures, callbacks, and result handling.
- **Rust's `std::io::Read` trait** — The `read_to_string()` and `read_to_end()` methods act as template methods built on top of the primitive `read()` method. Implementors only provide `read()`, and the default implementations handle buffering and string conversion.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---------|---------------|----------------------|
| **Strategy** | Strategy delegates the entire algorithm to a separate object; Template Method uses inheritance to vary steps within a fixed skeleton | Choose Strategy when you need to swap complete algorithms at runtime rather than override individual steps |
| **Factory Method** | Factory Method is a specific kind of template method where the primitive operation creates an object | Choose Factory Method when the single variation point is object creation; choose Template Method when multiple steps vary |
| **Bridge** | Bridge separates abstraction from implementation via composition; Template Method varies steps via inheritance | Choose Bridge when you need to vary both the abstraction hierarchy and the implementation hierarchy independently |
| **Hook Method** | Hook is a single optional step in a template method, not a full pattern | Use hooks within a Template Method for optional extension points that have a sensible default (usually empty) |
