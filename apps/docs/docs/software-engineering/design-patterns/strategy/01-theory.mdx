## Intent

Define a family of algorithms, encapsulate each one, and make them interchangeable. The Strategy pattern lets the algorithm vary independently from the clients that use it. In the Stock Market domain, this means a trading bot can switch between different trading strategies (momentum, mean reversion, moving average crossover) at runtime without changing the bot's core logic, keeping each strategy encapsulated in its own class and open for extension.

## The Problem Without This Pattern

Imagine you are building an automated trading system that executes buy and sell decisions based on market data. Without the Strategy pattern, every trading algorithm is crammed into a single monolithic class with a giant conditional chain. Adding a new strategy requires modifying that class, risking regressions in existing strategies. The trading bot cannot swap strategies at runtime without restarting or recompiling.

Consider a typical trading bot that processes market data and produces trading signals. Initially it supports only a simple moving average crossover strategy. As the system grows, you add momentum-based trading, mean reversion, and pairs trading. Without Strategy, all these algorithms live inside the same class, tangled together.

Here is what the broken approach looks like in pseudocode:

```python
# Problem: All trading strategies crammed into one monolithic class
# Every new strategy requires modifying this class

class TradingBot:
    def __init__(self, strategy_type: str):
        self.strategy_type = strategy_type

    def analyze(self, prices: list[float]) -> str:
        # [TANGLED] All strategy logic in one method
        if self.strategy_type == "moving_average_crossover":
            short_ma = sum(prices[-20:]) / min(20, len(prices))
            long_ma = sum(prices[-50:]) / min(50, len(prices))
            if short_ma > long_ma:
                return "BUY"
            elif short_ma < long_ma:
                return "SELL"
            else:
                return "HOLD"

        elif self.strategy_type == "momentum":
            if len(prices) < 10:
                return "HOLD"
            returns = (prices[-1] - prices[-10]) / prices[-10]
            if returns > 0.05:
                return "BUY"
            elif returns < -0.05:
                return "SELL"
            else:
                return "HOLD"

        elif self.strategy_type == "mean_reversion":
            if len(prices) < 20:
                return "HOLD"
            mean = sum(prices[-20:]) / 20
            current = prices[-1]
            deviation = (current - mean) / mean
            if deviation < -0.03:
                return "BUY"
            elif deviation > 0.03:
                return "SELL"
            else:
                return "HOLD"

        else:
            raise ValueError(f"Unknown strategy: {self.strategy_type}")

    def execute(self, prices: list[float]):
        signal = self.analyze(prices)
        # execute trade based on signal
        print(f"Signal: {signal}")


# Adding pairs trading requires a new elif branch — risk of breaking
# existing strategies. No runtime swapping without new TradingBot().
```

This approach violates the Open/Closed Principle (a class should be open for extension but closed for modification). Every new strategy forces a change to the `TradingBot` class, and strategies cannot be composed, tested in isolation, or swapped at runtime.

## Pseudocode — Stock Market Example

The following pseudocode demonstrates the Strategy pattern in the Stock Market domain. A `TradingBot` (Context) delegates trade signal generation to a pluggable `TradingStrategy` (Strategy interface). Three concrete strategies implement different trading algorithms: `MovingAverageCrossover`, `MomentumStrategy`, and `MeanReversionStrategy`. The bot can switch strategies at runtime based on market conditions.

```
// PSEUDOCODE — Strategy Pattern in Stock Market
// Participants labeled with [ROLE]

[STRATEGY INTERFACE]
  interface TradingStrategy:
    method analyze(prices: List[float]) -> Signal

[SIGNAL]
  enum Signal { BUY, SELL, HOLD }

[CONCRETE STRATEGY 1]
  class MovingAverageCrossover implements TradingStrategy:
    field short_window: int = 20
    field long_window: int = 50

    method analyze(prices: List[float]) -> Signal:
      if length(prices) < long_window:
        return HOLD
      short_ma = average(last N = short_window prices)
      long_ma = average(last N = long_window prices)
      if short_ma > long_ma:
        return BUY
      else if short_ma < long_ma:
        return SELL
      else:
        return HOLD

[CONCRETE STRATEGY 2]
  class MomentumStrategy implements TradingStrategy:
    field lookback_period: int = 10
    field threshold: float = 0.05

    method analyze(prices: List[float]) -> Signal:
      if length(prices) < lookback_period:
        return HOLD
      momentum = (last_price - price_N_periods_ago) / price_N_periods_ago
      if momentum > threshold:
        return BUY
      else if momentum < -threshold:
        return SELL
      else:
        return HOLD

[CONCRETE STRATEGY 3]
  class MeanReversionStrategy implements TradingStrategy:
    field window: int = 20
    field deviation_threshold: float = 0.03

    method analyze(prices: List[float]) -> Signal:
      if length(prices) < window:
        return HOLD
      mean_price = average(last N = window prices)
      current = last_price
      deviation = (current - mean_price) / mean_price
      if deviation < -deviation_threshold:
        return BUY    // price below mean — expect reversion up
      else if deviation > deviation_threshold:
        return SELL   // price above mean — expect reversion down
      else:
        return HOLD

[CONTEXT]
  class TradingBot:
    field strategy: TradingStrategy
    field portfolio: dict
    field current_prices: dict

    method set_strategy(strategy: TradingStrategy):
      this.strategy = strategy

    method run_analysis(ticker: string, prices: List[float]) -> Signal:
      return this.strategy.analyze(prices)

[CLIENT / MAIN]
  function main():
    bot = new TradingBot()
    aapl_prices = [daily prices for AAPL over 60 days]

    // Run with moving average crossover
    bot.set_strategy(new MovingAverageCrossover())
    signal = bot.run_analysis("AAPL", aapl_prices)
    print("Crossover signal for AAPL:", signal)

    // Switch to momentum at runtime
    bot.set_strategy(new MomentumStrategy())
    signal = bot.run_analysis("AAPL", aapl_prices)
    print("Momentum signal for AAPL:", signal)

    // Switch to mean reversion without stopping the bot
    bot.set_strategy(new MeanReversionStrategy())
    signal = bot.run_analysis("AAPL", aapl_prices)
    print("Reversion signal for AAPL:", signal)
```

## How It Works — Step by Step

1. **The client creates a TradingBot** — the Context object that holds a reference to a TradingStrategy. The bot is initialized without any strategy or with a default.

2. **The client instantiates a concrete strategy** — for example, `MovingAverageCrossover` with a short window of 20 days and a long window of 50 days. The strategy object is self-contained and can be tested independently.

3. **The client injects the strategy into the bot** via `set_strategy()`. The TradingBot stores the reference in its `strategy` field. It does not care which concrete strategy it received — it only knows the `TradingStrategy` interface.

4. **The client calls `run_analysis()` with market data** — for example, a list of 60 daily closing prices for AAPL. The TradingBot delegates to `strategy.analyze(prices)`.

5. **The concrete strategy executes its algorithm** — `MovingAverageCrossover.analyze()` computes the short and long moving averages, compares them, and returns a `Signal` enum value: `BUY`, `SELL`, or `HOLD`.

6. **The client swaps the strategy at runtime** — based on changing market conditions (e.g., switching from trending to range-bound markets), it calls `set_strategy(new MomentumStrategy())`. The TradingBot accepts the new strategy and uses it for the next `run_analysis()` call. No code paths were modified, no classes were reopened.

7. **Each strategy remains a drop-in replacement** for any other because all implement the same `TradingStrategy` interface. The bot's core logic — fetching prices, logging signals, executing trades — is completely decoupled from the analysis algorithm.

## When to Use

- You have multiple algorithms for the same task and want to select one at runtime without conditional logic scattered through your code.
- You anticipate adding new variants of an algorithm over time and want to follow the Open/Closed Principle.
- You have a class with many conditional branches (if-else or switch) that select different behaviors — each branch can become its own strategy.
- You want to unit test each algorithm in isolation without instantiating the entire application.
- In a Stock Market context: when your trading system needs to support multiple signal-generation strategies that can be swapped based on market regime, asset class, or user preference.
- You need to compose strategies — for example, running a momentum strategy on some assets and a mean reversion strategy on others, using the same TradingBot.

## When NOT to Use

- You have only one algorithm that rarely changes. Adding the strategy infrastructure would be over-engineering.
- The algorithms are trivial single-line expressions. A function reference or lambda is simpler than a full class hierarchy.
- The client code must know the concrete strategy type to configure it properly — if the strategies have wildly different interfaces, the pattern adds indirection without benefit.
- You need to maintain state across strategy switches that is specific to the previous strategy.
- The algorithms share significant internal logic. In that case, consider the Template Method pattern instead, which lets you vary parts of an algorithm while keeping the structure fixed.
- Your trading system has only one strategy and no roadmap for additional ones. Premature abstraction adds complexity without payoff.

## Real-World Occurrences

- **Java standard library** — `java.util.Comparator` is a Strategy interface. Sorting algorithms in `Collections.sort()` and `Arrays.sort()` accept any `Comparator`, letting the caller define the ordering strategy without modifying the sort implementation.
- **Python's `functools.lru_cache`** — the caching strategy (LRU eviction, max size, TTL) is configurable. Different caching strategies can be injected into functions that need memoization.
- **.NET's LINQ** — `IEnumerable` extension methods like `OrderBy()`, `Where()`, and `Select()` accept delegate parameters (strategies) that define the query behavior. The LINQ pipeline itself remains unchanged while the strategy transforms the data.
- **JavaScript's `Array.prototype.sort()`** — accepts a comparator function as a strategy. The sort implementation is fixed (typically TimSort), but the comparison strategy can be customized for ascending, descending, or object-field-based ordering.
- **Spring Framework** — `ResourceLoader` uses a Strategy pattern. Different `Resource` implementations (FileSystemResource, ClassPathResource, UrlResource) are selected based on the resource path prefix (`file:`, `classpath:`, `http:`).
- **Trading platforms** — systems like QuantConnect (LEAN Engine) allow users to write `IAlgorithm` implementations that plug into the engine. The engine is the Context, and each user-written algorithm is a Strategy.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Strategy** | Encapsulates interchangeable algorithms; the client selects which to use | When you need to swap algorithms at runtime and each algorithm is self-contained |
| **Template Method** | Defines the skeleton of an algorithm; subclasses override specific steps | When the algorithm structure is fixed and only some steps vary |
| **State** | Changes behavior based on internal state; the Context's state transitions drive behavior changes | When the algorithm selection depends on the object's internal state, not an external choice |
| **Command** | Encapsulates a request as an object for queuing, logging, or undo | When you need to parameterize, queue, or log operations, not swap algorithms |
| **Visitor** | Separates an algorithm from the object structure it operates on | When you need to add operations to a stable object hierarchy without modifying it |
| **Decorator** | Adds responsibilities to objects dynamically via composition | When you need to layer behaviors on an object, not swap entire algorithms |
