Provide a surrogate or placeholder for another object to control access to it. The Proxy pattern introduces a level of indirection that can serve many purposes: lazy loading, access control, logging, caching, or remote communication. In the Cars domain, a proxy stands between the driver and the real car, intercepting requests and adding behavior before or after forwarding them to the real vehicle.

## The Problem Without This Pattern

Consider a luxury car dealership that displays its vehicle inventory on showroom tablets. Each car listing shows a thumbnail, basic specs, and a "View Full 3D Model" button. Behind the scenes, the full 3D model of each car contains hundreds of thousands of polygons, high-resolution textures, and detailed interior data — consuming 200 MB or more per vehicle.

Without the Proxy pattern, the dealership application has two damaging problems.

**Problem 1 — Wasted Resources (Virtual Proxy).** Every car object loads its full 3D model and complete specification data at initialization, even if the user only browses thumbnails. Loading 50 cars at once translates to 10 GB of data in memory, crushing the tablet's performance and bandwidth. The user waits minutes for the showroom to load, and most of the data is never used because customers only examine a handful of models in detail.

Here is what the naive approach looks like:

```python
# BAD: Every car loads its full 3D model at construction time
class Showroom:
    def __init__(self):
        self.cars = []
        for car_data in self.fetch_car_inventory():
            # EXPENSIVE: 3D model loaded for every car, every time
            car = SportsCar(
                car_data["vin"],
                car_data["make"],
                car_data["model"],
                car_data["year"],
                # Triggers 200 MB 3D model load + texture unpacking
                load_full_3d_model=True,
            )
            self.cars.append(car)

    def display_thumbnails(self):
        # Only needs basic info, but we paid for 3D models
        for car in self.cars:
            print(f"{car.make} {car.model} ({car.year})")
```

**Problem 2 — Uncontrolled Access (Protection Proxy).** A valet parks the customer's Sypher GT supercar. The valet receives the key fob, which gives full access to the car's capabilities: 200 mph top speed, launch control, trunk and glovebox contents, and valet mode settings. There is no way to restrict what the valet can do without modifying the car itself. The owner cannot hand over a limited key that permits only parking maneuvers and denies access to the storage compartments.

```python
# BAD: Valet gets full access, no way to restrict
class ValetService:
    def park_car(self, car, spot):
        # The valet can drive at 150 mph, access trunk, etc.
        car.drive(150)  # No restrictions!
        car.access_trunk()  # Should not be allowed
        car.access_glovebox()  # Should not be allowed
```

The Proxy pattern solves both problems by introducing a controlled intermediary. A VirtualProxy delays expensive operations until they are actually needed, and a ProtectionProxy enforces access rules without modifying the real subject.

The participants in this structure are:

- **Subject (Car):** The interface that defines the common contract for both the real object and the proxy. It declares operations such as `display_details()`, `drive()`, `get_max_speed()`, and `access_trunk()`. Both the real car and the proxy implement this interface, making them interchangeable from the client's perspective.

- **RealSubject (SportsCar):** The real object that the proxy represents. SportsCar contains the actual 3D model data, full engine specifications, and all vehicle capabilities. Its `drive()` method can accelerate to 200 mph, and its `access_trunk()` method opens the physical trunk. Creating a SportsCar is expensive because it loads high-resolution model data from disk.

- **Proxy (ValetCarProxy, LazyCarModelProxy):** The surrogate that controls access to the RealSubject. ValetCarProxy is a protection proxy that restricts speed to 30 mph and denies access to the trunk and glovebox. LazyCarModelProxy is a virtual proxy that defers the creation of the SportsCar until `display_details()` is called, avoiding the 3D model loading cost if the user only browses basic information.

## Pseudocode — Cars Example

The following pseudocode demonstrates both variants of the Proxy pattern in the Cars domain. The LazyCarModelProxy delays loading of the expensive 3D model, and the ValetCarProxy restricts access for valet parking scenarios.

```python
# [SUBJECT] The interface all car representations must implement
from abc import ABC, abstractmethod
from typing import Optional


class Car(ABC):
    """Subject interface for all car types."""

    @abstractmethod
    def display_details(self) -> str:
        """Show full car specifications and 3D model."""
        pass

    @abstractmethod
    def drive(self, speed: float) -> str:
        """Drive the car at the given speed."""
        pass

    @abstractmethod
    def get_max_speed(self) -> float:
        """Return the maximum speed of the car in mph."""
        pass

    @abstractmethod
    def access_trunk(self) -> str:
        """Open and access the trunk compartment."""
        pass

    @abstractmethod
    def get_vin(self) -> str:
        """Return the vehicle identification number."""
        pass


# [REAL SUBJECT] The actual car with full capabilities
class SportsCar(Car):
    """RealSubject: The real sports car with expensive 3D model and full power."""

    def __init__(self, vin: str, make: str, model: str, year: int):
        self._vin = vin
        self._make = make
        self._model = model
        self._year = year
        self._max_speed = 200.0  # mph
        self._horsepower = 750
        self._engine = "5.2L V10 Twin-Turbo"
        self._transmission = "7-Speed Dual-Clutch"
        self._acceleration_0_60 = 2.8  # seconds
        self._full_3d_model = None
        self._trunk_contents = ["First aid kit", "Tool set", "Owners manual"]
        self._glovebox_contents = ["Registration", "Insurance card", "Service record"]
        self._is_engine_running = False
        self._current_speed = 0.0

        # Simulate expensive 3D model loading
        self._load_3d_model()

    def _load_3d_model(self) -> None:
        """Simulate loading a high-resolution 3D model (expensive operation)."""
        print(f"  [SportsCar] Loading 3D model for {self._vin}..."
              f" 200 MB of polygon data, textures, and materials loaded.")
        self._full_3d_model = {
            "polygons": 450_000,
            "textures": ["carbon_fiber", "leather_black", "aluminum_brushed"],
            "materials": ["forged_carbon", "alcantara", "titanium"],
        }

    def display_details(self) -> str:
        """Show full car specifications including 3D model data."""
        return (
            f"--- {self._year} {self._make} {self._model} ---\n"
            f"  VIN: {self._vin}\n"
            f"  Engine: {self._engine}\n"
            f"  Horsepower: {self._horsepower} hp\n"
            f"  Transmission: {self._transmission}\n"
            f"  0-60 mph: {self._acceleration_0_60}s\n"
            f"  Top Speed: {self._max_speed} mph\n"
            f"  3D Model: {self._full_3d_model['polygons']} polygons, "
            f"{len(self._full_3d_model['textures'])} textures\n"
            f"  Trunk Contents: {', '.join(self._trunk_contents)}\n"
            f"  Glovebox: {', '.join(self._glovebox_contents)}"
        )

    def drive(self, speed: float) -> str:
        """Drive the car at the requested speed (full capabilities)."""
        if speed > self._max_speed:
            speed = self._max_speed
        self._is_engine_running = True
        self._current_speed = speed
        engine_note = "ROAR! V10 screams as it climbs through the rev range."
        return (
            f"  [SportsCar] Engine: {engine_note}\n"
            f"  Speed: {speed:.1f} mph | Gear: 7th | RPM: 8200"
        )

    def get_max_speed(self) -> float:
        return self._max_speed

    def access_trunk(self) -> str:
        """Open the trunk and reveal contents."""
        contents = ", ".join(self._trunk_contents)
        return f"  [SportsCar] Trunk opened. Contents: {contents}"

    def get_vin(self) -> str:
        return self._vin


# [PROXY - Virtual Proxy] Lazy-loading car model
class LazyCarModelProxy(Car):
    """Virtual Proxy: Defers creation of the expensive SportsCar until details are needed."""

    def __init__(self, vin: str, make: str, model: str, year: int):
        self._vin = vin
        self._make = make
        self._model = model
        self._year = year
        self._real_car: Optional[SportsCar] = None

    def _initialize_real_car(self) -> None:
        """Create the real car only when necessary (lazy initialization)."""
        if self._real_car is None:
            print(f"  [LazyCarModelProxy] Creating real SportsCar for {self._vin}...")
            self._real_car = SportsCar(self._vin, self._make, self._model, self._year)
        else:
            print(f"  [LazyCarModelProxy] Reusing existing SportsCar instance.")

    def display_details(self) -> str:
        """Loading real car only when details are requested."""
        self._initialize_real_car()
        return self._real_car.display_details()

    def drive(self, speed: float) -> str:
        """Delegates to real car, initializing it if needed."""
        self._initialize_real_car()
        return self._real_car.drive(speed)

    def get_max_speed(self) -> float:
        """Return max speed from real car, initializing if needed."""
        self._initialize_real_car()
        return self._real_car.get_max_speed()

    def access_trunk(self) -> str:
        """Delegates to real car, initializing if needed."""
        self._initialize_real_car()
        return self._real_car.access_trunk()

    def get_vin(self) -> str:
        return self._vin


# [PROXY - Protection Proxy] Valet-limited car access
class ValetCarProxy(Car):
    """Protection Proxy: Restricts what a valet can do with the car."""

    def __init__(self, real_car: Car):
        self._real_car = real_car
        self._max_valet_speed = 30.0  # mph, valet cannot exceed this
        self._trunk_denied_message = (
            "  [ValetCarProxy] ACCESS DENIED: Valet key does not permit trunk access."
        )

    def display_details(self) -> str:
        """Show limited details — valet does not need full specs."""
        return (
            f"--- Valet Card ---\n"
            f"  VIN: {self._real_car.get_vin()}\n"
            f"  Max Permitted Speed: {self._max_valet_speed} mph\n"
            f"  Trunk Access: DENIED\n"
            f"  Glovebox Access: DENIED"
        )

    def drive(self, speed: float) -> str:
        """Enforce the valet speed limit."""
        limited_speed = min(speed, self._max_valet_speed)
        if limited_speed < speed:
            print(f"  [ValetCarProxy] Speed limited to {self._max_valet_speed} mph.")
        return self._real_car.drive(limited_speed)

    def get_max_speed(self) -> float:
        """Return the valet-limited maximum speed, not the car's actual max."""
        return self._max_valet_speed

    def access_trunk(self) -> str:
        """Valet is denied trunk access."""
        return self._trunk_denied_message

    def get_vin(self) -> str:
        return self._real_car.get_vin()


# --- CLIENT CODE ---
def main():
    print("=" * 60)
    print("  PROXY PATTERN — Cars Domain Demonstration")
    print("=" * 60)
    print()

    # Scenario 1: Virtual Proxy — Lazy Loading
    print("--- SCENARIO 1: Virtual Proxy (Lazy Loading) ---")
    print("  Creating a LazyCarModelProxy for the showroom...")
    lazy_car = LazyCarModelProxy("VIN-SPY-001", "Sypher", "GT", 2026)
    print()

    print("  Showroom: Displaying car thumbnail (no 3D model loaded yet)...")
    print(f"  {lazy_car.get_vin()} — {lazy_car._make} {lazy_car._model}")
    print()

    print("  Customer clicks 'View Full Details' — proxy loads the real car now:")
    details = lazy_car.display_details()
    print(details)
    print()

    # Scenario 2: Protection Proxy — Valet
    print("--- SCENARIO 2: Protection Proxy (Valet Key) ---")
    real_car = SportsCar("VIN-SPY-002", "Sypher", "GT-S", 2026)
    valet_key = ValetCarProxy(real_car)
    print()

    print("  Car owner hands over the valet key to the parking attendant.")
    print()

    print("  Valet: Displays car info card:")
    print(valet_key.display_details())
    print()

    print("  Valet: Attempts to drive at 80 mph:")
    print(valet_key.drive(80.0))
    print()

    print("  Valet: Attempts to access the trunk:")
    print(valet_key.access_trunk())
    print()

    # Scenario 3: Owner returns — full access
    print("--- SCENARIO 3: Owner Returns — Full Access ---")
    print("  Owner uses the real car key directly:")
    print(real_car.drive(120.0))
    print(real_car.access_trunk())


if __name__ == "__main__":
    main()
```

## How It Works — Step by Step

The Proxy pattern controls access to a real subject through an intermediary that implements the same interface. Here is the complete flow, traced through the Cars domain examples.

**Step 1 — The client interacts with a proxy object as if it were the real subject.**

The showroom application holds a collection of `Car` references. Each reference may point to either a real `SportsCar` or a `LazyCarModelProxy`. The client code does not know which one it holds — it only calls methods declared on the `Car` interface. This substitutability is the core of the pattern.

**Step 2 — The proxy checks whether it needs to forward the request.**

For a virtual proxy, the `LazyCarModelProxy` checks whether the real `SportsCar` has been created. If the real car is null, the proxy creates it first. This check happens transparently inside each method before the real operation proceeds. For a protection proxy, the `ValetCarProxy` checks whether the operation is permitted — for example, whether the requested speed exceeds the valet limit.

**Step 3 — The proxy enforces any additional rules before delegation.**

The `ValetCarProxy.drive()` method clamps the requested speed to 30 mph before forwarding the call. The `ValetCarProxy.access_trunk()` method returns a denial message without ever calling the real car's trunk access. These rules are enforced before the real subject is involved, so the real car never sees unauthorized requests at all.

**Step 4 — The proxy delegates the approved request to the real subject.**

If the request passes all checks, the proxy forwards it to the real `SportsCar` instance. The real car executes the operation with its full capabilities — the V10 engine, the 7-speed transmission, the actual trunk latch mechanism. The proxy does not modify the result of the operation; it only controls access to it.

**Step 5 — The proxy may post-process the result before returning it.**

The proxy can transform, log, or augment the return value from the real subject. In the valet scenario, the `ValetCarProxy.display_details()` method returns a restricted information card instead of the full specifications. The proxy could also log the request, cache the result, or add latency measurements.

**Step 6 — The client receives the result as if it called the real subject directly.**

The client receives a return value that conforms to the `Car` interface contract. The caller does not know — and does not need to know — whether the result came from the real subject, the proxy, or a combination of both. This transparency allows the proxy to be introduced or removed without modifying any client code.

**Step 7 — The proxy manages the lifecycle of the real subject (virtual proxy specific).**

In the `LazyCarModelProxy`, the real `SportsCar` is created on first use and retained for subsequent calls. The proxy can also destroy the real subject when memory pressure is high, re-creating it on the next access. This lifecycle management is invisible to the client.

**Step 8 — Additional proxy layers can be composed.**

Multiple proxies can wrap the same real subject. A `LazyCarModelProxy` can wrap a `SportsCar` to defer loading, and then a `ValetCarProxy` can wrap the `LazyCarModelProxy` to add access control. Each proxy adds one concern, and the composition is transparent to the client.

## When to Use

Use the Proxy pattern when the following conditions apply to your Cars domain system.

- **You need to defer the creation of an expensive object until it is actually used.** A virtual proxy is ideal when your car showroom displays dozens of vehicles but the user examines only a few in detail. The `LazyCarModelProxy` delays the 200 MB 3D model load until the user clicks "View Full Details."

- **You need to control access to an object based on the caller's permissions.** A protection proxy is the right solution when different actors — owners, valets, service technicians, test drivers — need different levels of access to the same car. The `ValetCarProxy` restricts speed and denies trunk access without modifying the `SportsCar` class.

- **You need to add logging, auditing, or metrics to every method call on an object.** A logging proxy can record every `drive()`, `access_trunk()`, or `display_details()` call with timestamps, caller identity, and success status. This is useful for fleet management, insurance tracking, or security auditing.

- **You need to cache the results of expensive operations.** A caching proxy can store the results of `display_details()` or full specifications so that repeated calls by different clients do not reload the 3D model or recompute complex data.

- **You need to provide a local representative for a remote object.** A remote proxy (stub) sits on the client machine and communicates with the real car over a network. For example, a mobile app that controls a connected car's door locks, climate control, and engine start uses a remote proxy to forward commands to the vehicle's onboard computer.

- **You need to synchronize access to a shared object from multiple threads or processes.** A synchronization proxy ensures that concurrent calls to `drive()` or `access_trunk()` from different threads do not produce race conditions. This is relevant in automotive software where multiple ECUs may request actions simultaneously.

## When NOT to Use

The Proxy pattern can introduce unnecessary complexity when applied to situations that do not require it. Avoid the pattern in these scenarios.

- **The object creation is cheap and always needed.** If the `SportsCar` constructor is lightweight and every client that creates a proxy will immediately access the real subject, the proxy adds indirection with no benefit. A simple constructor call is sufficient.

- **You only have one access level and no plans to add more.** If every user of the car has the same permissions, a protection proxy adds abstraction without value. Introduce the proxy when the second access level — such as a valet, a service technician, or a rental customer — emerges.

- **The proxy would introduce a performance bottleneck in a hot path.** If `drive()` is called hundreds of times per second in a simulation, an additional method call through the proxy, plus the null check in the virtual proxy, may add measurable overhead. In such cases, inline the access control logic or use a compile-time aspect.

- **The pattern is applied prematurely for speculative future requirements.** Do not introduce a proxy based on a guess that lazy loading or access control might be needed someday. Apply the pattern when the actual requirement materializes and the existing code shows the pain of the missing indirection.

- **The real subject is already behind a service layer or facade.** If the system already uses a service layer, API gateway, or facade that abstracts access to the car, adding another proxy layer can create unnecessary indirection. Evaluate whether the existing abstraction can be extended to handle the new concern.

- **You need to add behavior to only one or two methods.** If the only requirement is to log `drive()` calls, a simple decorator function or wrapper method is simpler than a full proxy implementing the entire `Car` interface. Use the Proxy pattern when multiple methods need the same cross-cutting concern.

## Real-World Occurrences

The Proxy pattern appears throughout standard libraries, frameworks, and enterprise applications. Here are notable examples from different ecosystems.

**Java — java.lang.reflect.Proxy**

The Java Reflection API provides `java.lang.reflect.Proxy` for creating dynamic proxy instances at runtime. A `Proxy` instance implements a specified list of interfaces and dispatches method invocations to an `InvocationHandler`. This is the foundation of many Java frameworks — Spring AOP uses it for declarative transaction management, security, and caching. In the Cars domain, a dynamic proxy could intercept all `Car` method calls to add logging, authorization, or lazy loading without writing a separate proxy class for each car type.

**Python — weakref.proxy and Django lazy objects**

Python's `weakref.proxy` creates a weak reference proxy that behaves like the original object but does not prevent garbage collection. Django uses lazy proxy objects for querysets — `QuerySet` objects are virtual proxies that delay the database query until the results are actually iterated. In the Cars domain, SQLAlchemy's lazy loading of related objects (e.g., loading a car's service history only when accessed) is a virtual proxy at the ORM level.

**.NET — System.Runtime.Remoting.RealProxy and DispatchProxy**

The .NET framework provides `RealProxy` for creating transparent proxies that forward method calls to remote objects across application domains or network boundaries. The modern `DispatchProxy` class in `System.Reflection.DispatchProxy` enables dynamic proxy creation for interface types. WCF (Windows Communication Foundation) generates proxy classes that let client code call remote service methods as if they were local. In the Cars domain, a .NET DispatchProxy could wrap a `Car` interface to add retry logic for unreliable telematics connections.

**Rust — lazy_static and once_cell for lazy initialization**

Rust's `lazy_static` crate and the standard library's `std::sync::OnceLock` (stabilized in Rust 1.70) provide virtual proxy semantics for lazy initialization. A global `OnceLock<SportsCar>` only initializes the car on the first access, similar to the `LazyCarModelProxy`. The `once_cell::sync::Lazy` type is used extensively in Rust applications for lazy configuration loading, connection pooling, and expensive resource initialization.

**JavaScript — ES6 Proxy object**

The JavaScript `Proxy` object is a built-in language feature that creates a proxy for any target object. The `Proxy` constructor takes a target and a handler object with traps for `get`, `set`, `has`, `apply`, and other operations. This is the most flexible proxy implementation in any mainstream language — you can intercept property access, method calls, property enumeration, and even constructor calls. In the Cars domain, a JavaScript `Proxy` can wrap a car object to validate speed values before setting them, log all property accesses, or provide default values for missing properties.

**Spring Framework — AOP Proxies**

The Spring Framework uses AOP proxies extensively. When you annotate a method with `@Transactional`, `@Cacheable`, or `@Secured`, Spring creates a proxy that wraps your bean and adds the cross-cutting behavior before or after the method invocation. The proxy implements the same interface as the target bean, so client code is unaware of the indirection. In the Cars domain, a Spring AOP proxy could add transaction management to a `CarRepository` without modifying the repository class.

## Comparison with Related Patterns

The following table shows how the Proxy pattern relates to other structural patterns and when to choose each one for your Cars application.

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Decorator** | Decorator adds behavior to an object dynamically through composition. Proxy controls access to an object. Both have the same interface as the subject, but Decorator is additive while Proxy is restrictive. | Choose Decorator when you want to add features to a car — such as GPS tracking, dashcam recording, or heated seats — by wrapping the car object with additional capabilities. Choose Proxy when you want to control how the car is accessed — lazy loading, speed limits, or trunk access restrictions. |
| **Adapter** | Adapter makes two incompatible interfaces work together. | **Facade** | Facade provides a simplified interface to a complex subsystem. Proxy provides the same interface as the subject, not a simplified one. | Choose Facade when you need to simplify starting a car — the facade could handle the sequence of unlocking doors, disarming the alarm, pressing the brake, and pressing the start button as a single `startCar()` operation. |
| **Bridge** | Bridge separates abstraction from implementation so both can vary independently. | **Composite** | Composite composes objects into tree structures to represent part-whole hierarchies.
