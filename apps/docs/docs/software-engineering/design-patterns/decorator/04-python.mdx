## Implementation

Below is a complete Python implementation of the Decorator pattern using a car dealership domain. Python uses abstract base classes (ABCs) from the `abc` module to define the `Car` interface. `BaseCar` is the concrete component, `CarFeature` is the abstract decorator, and `LeatherSeats`, `Sunroof`, `SportPackage`, and `NavigationSystem` are concrete decorators. Python's duck-typing means the pattern works even without formal interfaces, but using ABCs makes the intent explicit and enforces method implementation.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```

//   │            <<abstract>>                       │     ISP: 2 methods
//   ├──────────────────────────────────────────────┤
```

#### OOP Concepts Applied

1. **Abstraction** — The `Car` ABC defines the abstract interface with `@abstractmethod` decorators. Python enforces that concrete subclasses implement both methods. Client code depends on `Car`, not on specific implementations.

2. **Polymorphism** — `BaseCar` and all decorators are treated as `Car` instances. The driver's iteration over a list of `Car` objects works uniformly because every object implements `getDescription()` and `getCost()`.

3. **Composition over Inheritance** — `CarFeature` holds a reference to a `Car` object and delegates to it. Feature combinations are composed at runtime, avoiding the combinatorial explosion of subclassing.

#### SOLID Principles Applied

4. **SRP** — `BaseCar` tracks only the base model identity. Each decorator is responsible for exactly one feature. The driver is responsible for configuration and output.

5. **OCP** — New features are added as new decorator classes. Existing code (`BaseCar`, `CarFeature`, other decorators) is never modified.

6. **LSP** — Decorators preserve the `Car` interface. Any decorated car can replace a `BaseCar` in client code without behavioral changes.

7. **ISP** — The `Car` interface has exactly two methods. No client is forced to depend on methods it does not use.

8. **DIP** — High-level code depends on the `Car` ABC. `BaseCar` and decorators implement `Car`. The driver constructs concrete objects but interacts with them through the abstract interface.

```python
# Decorator Pattern — Python — Cars example
# ==================================================
# Participants: Car (Component ABC),
# BaseCar (ConcreteComponent),
# CarFeature (Decorator ABC),
# LeatherSeats, Sunroof, SportPackage, NavigationSystem (ConcreteDecorators)
# OOP: abstraction, polymorphism, composition over inheritance
# SOLID: SRP, OCP, LSP, ISP, DIP
# ==================================================

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import List, Tuple
```

### Component (abstract base class)

```python
class Car(ABC):
    @abstractmethod
    def get_description(self) -> str:
        pass

    @abstractmethod
    def get_cost(self) -> float:
        pass
```

### ConcreteComponent

```python
class BaseCar(Car):
    def __init__(self, model: str, base_cost: float) -> None:
        self._model = model
        self._base_cost = base_cost

    def get_description(self) -> str:
        return self._model

    def get_cost(self) -> float:
        return self._base_cost
```

### Decorator (Abstract)

```python
class CarFeature(Car):
    def __init__(self, car: Car) -> None:
        self._car = car

    def get_description(self) -> str:
        return self._car.get_description()

    def get_cost(self) -> float:
        return self._car.get_cost()
```

### ConcreteDecorator 1: LeatherSeats

```python
class LeatherSeats(CarFeature):
    _FEATURE_COST = 1500.00
    _FEATURE_NAME = "Leather Seats"

    def __init__(self, car: Car) -> None:
        super().__init__(car)

    def get_description(self) -> str:
        return f"{super().get_description()} + {self._FEATURE_NAME}"

    def get_cost(self) -> float:
        return super().get_cost() + self._FEATURE_COST
```

### Helper: format a configuration line

```python
def format_config(car: Car) -> str:
    desc = car.get_description()
    cost = f"${car.get_cost():.2f}"
    return f"  {desc:<60} {cost:>8}"
```

### Main / Driver

```python
def main() -> None:
    print("=== Python Car Dealership Configuration System ===")
    print()

    # Build configurations
    base_camry: Car = BaseCar("Toyota Camry", 28000.00)
    with_leather: Car = LeatherSeats(base_camry)
    with_sunroof: Car = Sunroof(base_camry)
    with_leather_sunroof: Car = Sunroof(LeatherSeats(base_camry))
    eco_luxury: Car = NavigationSystem(LeatherSeats(base_camry))
    sport_only: Car = SportPackage(base_camry)
    fully_loaded: Car = SportPackage(
        NavigationSystem(
            Sunroof(
                LeatherSeats(base_camry)
            )
        )
    )

    configurations: List[Tuple[str, Car]] = [
        ("Base Model", base_camry),
        ("With Leather", with_leather),
        ("With Sunroof", with_sunroof),
        ("Leather + Sunroof", with_leather_sunroof),
        ("Eco Luxury (Leather + Nav)", eco_luxury),
        ("Sport Only", sport_only),
        ("Fully Loaded", fully_loaded),
    ]

    # Print header
    print(f"{'Configuration':<35} {'Description':<60} {'Price':>8}")
    print("-" * 105)

    for label, car in configurations:
        desc = car.get_description()
        cost = f"${car.get_cost():.2f}"
        print(f"  {label:<33} {desc:<60} {cost:>8}")

    # Show price breakdown for fully loaded
    print()
    print("--- Price Breakdown: Fully Loaded Camry ---")
    print(f"  Base car:                          ${base_camry.get_cost():>7.2f}")
    print(f"  + Leather Seats:                   ${LeatherSeats._FEATURE_COST:>7.2f}")
    print(f"  + Sunroof:                         ${Sunroof._FEATURE_COST:>7.2f}")
    print(f"  + Navigation System:               ${NavigationSystem._FEATURE_COST:>7.2f}")
    print(f"  + Sport Package:                   ${SportPackage._FEATURE_COST:>7.2f}")
    print(f"  {'─' * 40}")
    print(f"  Total:                             ${fully_loaded.get_cost():>7.2f}")

    # Verify interface compliance
    print()
    print("--- Interface Compliance ---")
    for i, (_, car) in enumerate(configurations, 1):
        is_car = isinstance(car, Car)
        print(f"  Config {i}: isinstance(Car) = {is_car}")

    # Demonstrate reuse with a different car model
    print()
    print("--- Honda Accord ---")
    base_accord: Car = BaseCar("Honda Accord", 27000.00)
    accord_luxury: Car = NavigationSystem(LeatherSeats(base_accord))
    print(f"  {accord_luxury.get_description()}")
    print(f"  Total: ${accord_luxury.get_cost():.2f}")

    # Demonstrate that decorators are stackable in any order
    print()
    print("--- Different Ordering (same features) ---")
    order_a: Car = Sunroof(LeatherSeats(base_camry))
    order_b: Car = LeatherSeats(Sunroof(base_camry))
    print(f"  Order A (Sunroof(Leather)): {order_a.get_description()} — ${order_a.get_cost():.2f}")
    print(f"  Order B (Leather(Sunroof)): {order_b.get_description()} — ${order_b.get_cost():.2f}")
    print(f"  Same cost: {order_a.get_cost() == order_b.get_cost()}")


if __name__ == "__main__":
    main()
```

### Language-Specific Notes

- **ABCs vs duck typing** — Python's `ABC` (abstract base class) from the `abc` module provides formal interface enforcement. Attempting to instantiate `Car` or `CarFeature` directly raises `TypeError`. However, Python's duck typing means the decorator pattern would work even without ABCs — any object with `get_description()` and `get_cost()` methods would satisfy the interface. The ABCs make the design intent explicit and catch errors early.

- **`@abstractmethod` decorator** — The `@abstractmethod` decorator marks methods that must be overridden by concrete subclasses. Combined with `metaclass=ABCMeta` (or inheriting from `ABC`), Python prevents instantiation of any class that has not implemented all abstract methods.

- **Name mangling for private attributes** — Python does not have true private fields. The underscore prefix (`_car`, `_model`, `_base_cost`) is a convention indicating protected/private access. Name mangling (`__car`) would make subclass access more difficult, so a single underscore is used for the decorator's wrapped reference.

- **`super()` delegation** — Python's `super()` returns a proxy object that delegates method calls to the parent class. In `CarFeature.get_description()`, `super().get_description()` calls the parent's implementation, which delegates to the wrapped `_car`. This is the standard Python mechanism for cooperative multiple inheritance and works cleanly for the decorator chain.

- **No compile-time type safety** — Python's type hints (`Car`, `float`, `str`) are not enforced at runtime. The `Car` ABC provides runtime enforcement of method implementation, but the type annotations are for developer guidance and static analysis tools like mypy only.

- **Decorator order symmetry** — The driver demonstrates that wrapping order for additive features produces the same cost and description regardless of order (Leather then Sunroof vs Sunroof then Leather). This holds because the features are orthogonal and additive. For features with dependencies (e.g., "Sport Package requires Sunroof"), order constraints would need explicit validation.
