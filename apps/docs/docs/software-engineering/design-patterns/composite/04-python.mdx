Python implements the Composite pattern using abstract base classes from the `abc` module and duck typing. The `CarPart` ABC defines the interface with `@abstractmethod` decorators. Leaf classes (`Wheel`, `BrakePad`, `Piston`, `SparkPlug`) inherit from `CarPart` and implement the methods directly. The `CompositePart` class holds a `children` list and delegates `get_weight()`, `get_price()`, and `get_part_count()` to each child. Concrete composites (`Engine`, `Suspension`, `Car`) extend `CompositePart` and inherit the aggregation logic. Python's duck typing means any object with the right methods can be treated as a `CarPart`, but using ABCs makes the intent explicit.

## How the Implementation Works

Python's `ABC` (Abstract Base Class) from the `abc` module enforces that all concrete subclasses implement `get_weight()`, `get_price()`, and `get_part_count()`. The `CompositePart` class provides the aggregation logic: `get_weight()` calls `sum(child.get_weight() for child in self._children)`. A `print_parts_tree()` function recursively walks the tree, using `isinstance()` checks to distinguish leaf from composite for display formatting.

### OOP Principles Applied

| Principle | How It Is Applied |
|---|---|
| **Encapsulation** | Leading-underscore attributes (`_weight`, `_price`, `_children`) signal private-by-convention; callers use accessor methods |
| **Inheritance** | All leaf and composite classes inherit from `CarPart` ABC, receiving the common interface and the `name` attribute |
| **Polymorphism** | The `print_parts_tree()` function accepts any `CarPart` — leaf and composite objects respond to the same methods with different behavior |
| **Open/Closed** | New part types (e.g., `Turbocharger` as leaf, `ExhaustSystem` as composite) extend the hierarchy without modifying existing classes |
| **Single Responsibility** | Leaf classes own a single part's data; `CompositePart` manages tree aggregation; driver code builds the tree |
| **Liskov Substitution** | Any `CarPart` subclass can replace a `CarPart` reference in client code without breaking correctness |

### Component

```python
# Composite Pattern — Car Parts Hierarchy (Python)
from abc import ABC, abstractmethod
from typing import List, Optional

class CarPart(ABC):
    """Abstract base class for all car parts — both leaf and composite."""

    def __init__(self, name: str):
        self._name = name

    def get_name(self) -> str:
        return self._name

    @abstractmethod
    def get_weight(self) -> float:
        """Return weight in kilograms."""
        pass

    @abstractmethod
    def get_price(self) -> float:
        """Return price in dollars."""
        pass

    @abstractmethod
    def get_part_count(self) -> int:
        """Return number of individual parts (1 for leaf, sum for composite)."""
        pass

    def add_child(self, part: "CarPart") -> None:
        raise NotImplementedError(
            f"add_child() not supported on {type(self).__name__}"
        )

    def remove_child(self, part: "CarPart") -> None:
        raise NotImplementedError(
            f"remove_child() not supported on {type(self).__name__}"
        )

    def get_child(self, index: int) -> "CarPart":
        raise NotImplementedError(
            f"get_child() not supported on {type(self).__name__}"
        )
```

### Composite

```python
class CompositePart(CarPart):
    """Base class for assemblies that contain child parts."""

    def __init__(self, name: str):
        super().__init__(name)
        self._children: List[CarPart] = []

    def get_weight(self) -> float:
        return sum(child.get_weight() for child in self._children)

    def get_price(self) -> float:
        return sum(child.get_price() for child in self._children)

    def get_part_count(self) -> int:
        return sum(child.get_part_count() for child in self._children)

    def add_child(self, part: CarPart) -> None:
        if not isinstance(part, CarPart):
            raise TypeError("Child must be a CarPart instance")
        self._children.append(part)

    def remove_child(self, part: CarPart) -> None:
        try:
            self._children.remove(part)
        except ValueError:
            pass  # Part not found — silently ignore

    def get_child(self, index: int) -> CarPart:
        if index < 0 or index >= len(self._children):
            raise IndexError(f"Child index {index} out of bounds")
        return self._children[index]

    def get_children(self) -> List[CarPart]:
        return list(self._children)
```

### Print Helper

```python
def print_parts_tree(part: CarPart, indent: str = "") -> None:
    weight = f"{part.get_weight():.2f}"
    price = f"{part.get_price():.2f}"
    count = part.get_part_count()
    kind = "Assembly" if isinstance(part, CompositePart) else "Part"
    print(
        f"{indent}{kind}: {part.get_name()} | "
        f"Weight: {weight} kg | Price: ${price} | Parts: {count}"
    )
    if isinstance(part, CompositePart):
        for child in part.get_children():
            print_parts_tree(child, indent + "  ")
```

## Language-Specific Notes

- Python's `ABC` and `@abstractmethod` from the `abc` module provide compile-like enforcement of method implementation; any concrete subclass that fails to implement an abstract method raises `TypeError` at instantiation time.
- Leading underscores (`_name`, `_weight`, `_children`) signal private attributes by convention; Python does not enforce access control, but the `@property` decorator could be used for stricter getter/setter patterns.
- The `sum()` built-in with a generator expression (`sum(child.get_weight() for child in self._children)`) is the idiomatic Python way to aggregate across children without explicit loops.
- `isinstance(part, CompositePart)` in the print helper demonstrates the one place where a type check is acceptable — tree display formatting — while the core pattern avoids it entirely.
- The `remove_child()` method catches `ValueError` silently, which is acceptable for a demonstration; a production BOM system would use ID-based lookup for reliable removal.
- Run with `python composite.py` — no third-party dependencies required.
