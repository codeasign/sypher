## Intent

The Memento pattern provides the ability to capture and externalize an object's internal state so that the object can be restored to that state later, all without violating encapsulation. It solves the problem of how to implement undo, rollback, or history functionality without exposing the internal data structures of the object whose state is being saved.

## The Problem Without This Pattern

Consider a stock trading portfolio that tracks holdings across multiple tickers. A portfolio manager needs to experiment with different allocation strategies — buying and selling shares, rebalancing positions — and occasionally undo a series of trades when a strategy underperforms. Without the Memento pattern, you face a dilemma:

If the portfolio exposes its internal state (a map of ticker symbols to share counts, average cost bases, and cash balances), any client could directly manipulate these fields, breaking invariants like "total shares cannot exceed the cash on hand" or "average cost basis must be recalculated after each buy." If the portfolio hides its internals behind a strict API, there is no way to capture a snapshot for later restoration.

The naive solution is to store a deep copy of the entire portfolio object before each trade:

```java
// BROKEN — violates encapsulation, couples client to internals
class PortfolioManager {
    private Map<String, int> heldShares;  // must be public or package-private
    private Map<String, double> avgCost;  // for snapshot to work
    private double cashBalance;

    public PortfolioSnapshot save() {
        // Client must know to copy these specific maps
        return new PortfolioSnapshot(heldShares, avgCost, cashBalance);
    }

    public void restore(PortfolioSnapshot snapshot) {
        // Restoring requires directly setting internal fields
        this.heldShares = snapshot.getHeldShares();
        this.avgCost = snapshot.getAvgCost();
        this.cashBalance = snapshot.getCashBalance();
    }
}
```

This approach couples every client to the portfolio's internal representation. Changing the portfolio's internal data structures (switching from `HashMap` to `TreeMap`, adding a new field for dividend tracking) breaks every save and restore call site. The Memento pattern solves this by letting the originator (the portfolio) own both the serialization and deserialization of its state, while exposing only a narrow memento interface to the caretaker.

### Participants

1. **Originator** (`Portfolio`) — The object whose state needs to be saved and restored. Creates a memento containing a snapshot of its current internal state and uses a memento to restore its state later.

2. **Memento** (`PortfolioSnapshot`) — An opaque object that stores the internal state of the originator. It is constructed by the originator and is only readable by the originator. The caretaker never inspects or modifies the memento's contents.

3. **Caretaker** (`PortfolioHistory`) — Manages the lifecycle of mementos. It requests snapshots from the originator before state-changing operations and provides mementos back to the originator when undo is needed. The caretaker never examines the memento's internals.

## Pseudocode — Stock Market Example

The following pseudocode implements a stock portfolio with undo support. The `Portfolio` (originator) holds a collection of stock positions and a cash balance. Before each trade, the caller requests a snapshot from the portfolio and pushes it to the `PortfolioHistory` (caretaker). If a trade needs to be undone, the caretaker pops the most recent snapshot and hands it back to the portfolio for restoration.

```
// PSEUDOCODE — Memento Pattern in Stock Market context
// Participants labeled with [ROLE]

[ORIGINATOR]
class Portfolio:
    // Internal state — fully encapsulated
    field holdings: Map<String, Holding>
    field cashBalance: double

    // Public business methods
    method buyTicker(symbol: String, shares: int, price: double):
        cost = shares * price
        if cost > cashBalance:
            throw InsufficientFundsError
        cashBalance = cashBalance - cost
        if holdings.contains(symbol):
            existing = holdings.get(symbol)
            totalShares = existing.shares + shares
            totalCost = existing.totalCost + cost
            holdings.put(symbol, Holding(totalShares, totalCost))
        else:
            holdings.put(symbol, Holding(shares, cost))

    method sellTicker(symbol: String, shares: int):
        if not holdings.contains(symbol):
            throw PositionNotFoundError
        existing = holdings.get(symbol)
        if shares > existing.shares:
            throw InsufficientSharesError
        avgPrice = existing.totalCost / existing.shares
        proceeds = shares * avgPrice  // simplified
        cashBalance = cashBalance + proceeds
        remaining = existing.shares - shares
        if remaining == 0:
            holdings.remove(symbol)
        else:
            remainingCost = existing.totalCost * (remaining / existing.shares)
            holdings.put(symbol, Holding(remaining, remainingCost))

    // Memento methods
    method save():
        return PortfolioSnapshot(
            deepCopy(holdings),
            copy(cashBalance)
        )

    method restore(snapshot: PortfolioSnapshot):
        // Only Portfolio can read memento internals
        holdings = snapshot.getHoldings()
        cashBalance = snapshot.getCashBalance()

[MEMENTO]
class PortfolioSnapshot:
    field holdings: Map<String, Holding>  (readonly after construction)
    field cashBalance: double              (readonly after construction)

    // Package-private or friend access — only Portfolio constructs/reads
    method PortfolioSnapshot(holdings, cashBalance):
        this.holdings = deepCopy(holdings)
        this.cashBalance = cashBalance

    method getHoldings(): Map<String, Holding>  (accessible only by Portfolio)
    method getCashBalance(): double              (accessible only by Portfolio)

[CARETAKER]
class PortfolioHistory:
    field snapshots: Stack<PortfolioSnapshot>

    method push(snapshot: PortfolioSnapshot):
        snapshots.push(snapshot)

    method pop():
        if snapshots.isEmpty():
            throw NoSnapshotsError
        return snapshots.pop()

    method canUndo():
        return not snapshots.isEmpty()

[SUPPORTING]
struct Holding:
    shares: int
    totalCost: double

    method avgCostPerShare():
        return totalCost / shares

// ── Client / Driver ──
function main():
    portfolio = new Portfolio(initialCash=100000.0)
    history = new PortfolioHistory()

    // Save state before buying
    history.push(portfolio.save())
    portfolio.buyTicker("AAPL", 100, 150.0)

    // Save state before next buy
    history.push(portfolio.save())
    portfolio.buyTicker("GOOGL", 50, 2800.0)

    // Save state before sell
    history.push(portfolio.save())
    portfolio.sellTicker("AAPL", 20)

    // Undo the sell
    snapshot = history.pop()
    portfolio.restore(snapshot)
    // Portfolio is now back to state after both buys, before sell
```

## How It Works — Step by Step

1. **Initialization** — A `Portfolio` is created with an initial cash balance (e.g., $100,000). No holdings exist yet. The caller also creates an empty `PortfolioHistory` that will act as the caretaker.

2. **First save-and-buy** — Before executing a buy order for 100 shares of AAPL at $150, the caller calls `portfolio.save()` to capture the current (empty) state. The portfolio creates a `PortfolioSnapshot` containing a deep copy of its holdings map and a copy of its cash balance. This snapshot is opaque to the caller — it is simply pushed onto the `PortfolioHistory` stack.

3. **First trade executes** — The portfolio processes the buy: it deducts $15,000 from the cash balance, and creates a `Holding(100, 15000.0)` in its holdings map. The portfolio's internal state has changed, but no snapshot of this new state has been saved yet.

4. **Second save-and-buy** — The caller wants to buy 50 shares of GOOGL at $2,800. Before doing so, it calls `portfolio.save()` again. The portfolio captures its current state (100 AAPL shares, $85,000 cash) into a new snapshot and pushes it. Then it executes the buy: deducts $140,000, adds `Holding(50, 140000.0)` for GOOGL.

5. **Third save-and-sell** — The caller saves again, then sells 20 shares of AAPL. The sell calculates the average cost per share ($150), adds $3,000 proceeds to cash, and reduces the AAPL holding to 80 shares.

6. **Undo** — The caller decides the sell was a mistake. It calls `history.pop()` to retrieve the most recent snapshot (the state before the sell, with 100 AAPL, 50 GOOGL, and cash-balance after the GOOGL buy). It passes this snapshot to `portfolio.restore(snapshot)`, which reconstructs the portfolio's internal state exactly as it was before the sell. The portfolio is now back to the state it had after step 4.

7. **Multiple undos** — Because each snapshot is stored on a stack, the caller can undo multiple operations in reverse order. Each `pop()` retrieves progressively older states, enabling a full undo chain.

## When to Use

- You need to implement undo or rollback functionality, and the object's state is complex enough that manual reversal of each operation would be error-prone.
- The object whose state needs saving has internal data structures that must not be exposed to external code.
- You need a history mechanism that allows restoring an object to any previous checkpoint, not just the most recent one.
- Your system has long-lived objects whose state evolves through discrete, reversible operations — such as a trading portfolio that processes buy and sell orders.
- You want to separate the snapshot mechanism from the business logic, so that adding new state fields does not break snapshot code elsewhere.

## When NOT to Use

- The object's state is small and simple enough that client-side copy-and-restore is manageable (e.g., a single integer counter).
- The object has a very large state and snapshots would consume prohibitive amounts of memory — consider delta-based or command-pattern undo instead.
- The object changes state at a very high frequency, making snapshotting on every change impractical (e.g., real-time stock price ticks updating a graph).
- The system already uses the Command pattern, which naturally supports undo through its own `unexecute()` mechanism — adding Memento on top would be redundant.
- Encapsulation is not a concern and the object's state can be safely exposed through getters and setters.

## Real-World Occurrences

1. **Java `java.io.Serializable`** — While not a pure Memento, the serialization mechanism allows an object's state to be captured to a byte stream and later restored. The object controls its serialized form through `writeObject` and `readObject` methods, similar to how an originator creates and consumes a memento.

2. **.NET `System.Windows.Input.UndoCompleted` / ICommand undo** — WPF applications commonly implement undo by capturing UI state into memento-like objects before mutations, managed by an `UndoManager` caretaker.

3. **JavaScript Redux DevTools** — Each dispatched action produces a snapshot of the entire Redux store state. The DevTools store these snapshots as a timeline, allowing developers to "time travel" by restoring any previous store state. The reducer (originator) creates new state; the DevTools (caretaker) manage the history.

4. **Python Undo/Redo in IPython and Jupyter** — The IPython kernel maintains a history of execution results. When a user requests undo, the kernel restores variable state from a previously saved snapshot. The `_oh` dictionary acts as a memento of output results.

5. **Rust `tracing` crate with `span` snapshots** — Tracing spans capture and restore context during async task execution. The span's internal state (fields, metadata) is encapsulated in a `Span` struct, and the tracing subscriber acts as the caretaker that manages span lifecycles.

6. **PostgreSQL MVCC (Multi-Version Concurrency Control)** — Each transaction sees a snapshot of the database state at the time the transaction started. If a transaction rolls back, the system restores the snapshot, discarding all changes made during the transaction. The database engine (originator) owns the snapshot mechanism; the transaction manager (caretaker) tracks which snapshot each transaction uses.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Command** | Command encapsulates an operation as an object with an `execute()` method; undo is handled by a symmetric `unexecute()` method on the command itself. | When each operation knows how to reverse itself and you don't need a full state snapshot. Command undo is cheaper than Memento for simple operations. |
| **State** | State represents an object's entire current configuration as a separate object and allows switching between states. | When the object's behavior depends on its current state and you want to model state transitions explicitly, not save/restore arbitrary checkpoints. |
| **Prototype** | Prototype creates new objects by cloning an existing instance. | When you need copies of an object for creation purposes, not for undo/rollback. Memento focuses on restoration, not duplication. |
| **Snapshot / Checkpoint (idiom)** | A general pattern for capturing system state for fault tolerance. | In distributed systems where you need crash recovery across multiple nodes. Memento is for in-process undo within a single object's lifetime. |
| **Visitor** | Visitor adds operations to a structure without modifying it. | When you need to traverse and operate on a complex object graph without saving intermediate states. |

## Language-Specific Implementation Notes

The Memento pattern is implemented across all six target languages with consistent structure but language-appropriate idioms:

- **JavaScript (ES2022+)** — Uses private class fields (`#`) to enforce memento encapsulation. The caretaker stores mementos in an array used as a stack. Polymorphism through duck typing.

- **TypeScript** — Adds strict type safety with interfaces for the memento (narrow public `Snapshot` interface vs. full internal state) and generics for the caretaker. Private fields via `#` for true encapsulation.

- **Python 3.11+** — Uses `@dataclass` for the internal `Holding` structure and underscores (`_holdings`, `_cash_balance`) for encapsulation by convention. The memento class stores state as a frozen dataclass or as plain attributes accessed only within the same module.

- **Java 17+** — The memento is implemented as a nested class (static inner class) within the originator, so it has access to private fields. The caretaker only sees the memento as an opaque `Object` or narrow interface.

- **C# 12 / .NET 8** — Uses nested class for the memento with `internal` access modifier. The originator exposes a `Snapshot` record as the public type while keeping the full memento class hidden.

- **Rust 2021** — The memento is an opaque struct defined in the same module as the originator. The caretaker stores `Box<dyn Any>` or a concrete memento type with no public field accessors. Encapsulation is enforced by Rust's module system.

Each implementation follows the same Stock Market domain: a `Portfolio` originator with buy/sell operations, a `PortfolioSnapshot` memento for state capture, and a `PortfolioHistory` caretaker for undo management. Real ticker data (AAPL, GOOGL, MSFT) is used in every driver.
