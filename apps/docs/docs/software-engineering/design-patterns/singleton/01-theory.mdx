## Intent

Ensure a class has exactly one instance and provide a single global point of access to it. The Singleton pattern controls object creation, guaranteeing that no matter how many times the system requests the instance, the same object is always returned. In a banking context, this means a single `BankConfiguration` object holds all global settings — interest rates, fee structures, regulatory limits, API endpoints — and every department, service, and module reads from the identical configuration, preventing inconsistencies and redundant resource usage.

## The Problem Without This Pattern

Consider a large banking application with multiple departments: Loan Servicing, Transaction Processing, Compliance, and Reporting. Each department needs to read configuration data — the current base interest rate, wire transfer fee, maximum daily transaction limit, and the regulatory API endpoint for KYC verification.

Without a Singleton, each department creates its own configuration instance by reading the configuration file independently:

```python
# PROBLEMATIC: Each module loads its own copy of configuration
class LoanService:
    def __init__(self):
        # Loads config from file — every time
        self.config = self._load_config("bank_config.yaml")
        self.base_rate = self.config["interest_rates"]["base"]

class TransactionService:
    def __init__(self):
        # Loads the same config again — duplicated effort
        self.config = self._load_config("bank_config.yaml")
        self.wire_fee = self.config["fees"]["wire_transfer"]

class ComplianceService:
    def __init__(self):
        # Loads the same config a third time
        self.config = self._load_config("bank_config.yaml")
        self.kyc_endpoint = self.config["api_endpoints"]["kyc"]
```

This approach has several critical problems:

1. **Resource waste** — The configuration file is parsed repeatedly, loading the same data into memory multiple times. For a large configuration file with hundreds of settings, this wastes CPU cycles and memory.

2. **Inconsistent state** — If the configuration is updated at runtime (e.g., the central bank changes the base interest rate), only the configuration instances created after the update will reflect the change. Older instances hold stale values, leading to inconsistent calculations across departments.

3. **Race conditions on reload** — Without a single lock on the configuration, two threads could simultaneously update their local copies of the configuration with different values, causing data corruption.

4. **No central change notification** — When the compliance officer updates the KYC endpoint URL, there is no mechanism to notify all services that the configuration has changed. Each service discovers the change only when it next reloads its file.

5. **Testing difficulty** — Every service that reads configuration must be tested with the actual file system or mocked individually, increasing test complexity.

## Pseudocode — Banking Example

The following pseudocode demonstrates the Singleton pattern in the context of a banking system. The `BankConfiguration` class ensures all banking services access the same configuration data.

```text
// PSEUDOCODE — Singleton Pattern in Banking Domain
// Participants labeled with [ROLE]

// ── Singleton ──
class BankConfiguration
    // Private static variable holding the single instance
    private static instance: BankConfiguration = null

    // Private constructor prevents external instantiation
    private BankConfiguration()
        // Load all configuration values from file or environment
        this.baseInterestRate = 4.5       // Base rate in percent
        this.savingsInterestRate = 3.75   // Savings account rate
        this.wireTransferFee = 25.00      // USD per wire transfer
        this.maxTransactionLimit = 50000.00  // USD daily limit
        this.minBalanceRequirement = 500.00  // USD minimum balance
        this.kycApiEndpoint = "https://api.bank.com/kyc/v2"
        this.bankName = "Global Trust Bank"
        this.swiftCode = "GTBKUS44"
        this.reserveRequirement = 0.10     // 10% reserve ratio

    // Public static method — the global access point
    public static getInstance(): BankConfiguration
        if instance is null
            instance = new BankConfiguration()
        return instance

    // Getter methods — provide read access to configuration
    public getBaseInterestRate(): float
        return this.baseInterestRate

    public getSavingsInterestRate(): float
        return this.savingsInterestRate

    public getWireTransferFee(): float
        return this.wireTransferFee

    public getMaxTransactionLimit(): float
        return this.maxTransactionLimit

    public getMinBalanceRequirement(): float
        return this.minBalanceRequirement

    public getKycApiEndpoint(): string
        return this.kycApiEndpoint

    public getBankName(): string
        return this.bankName

    public getSwiftCode(): string
        return this.swiftCode

    public getReserveRequirement(): float
        return this.reserveRequirement

// ── Client 1: LoanDepartment ──
class LoanDepartment
    public processLoanApplication(amount: float, creditScore: int): void
        config = BankConfiguration.getInstance()
        rate = config.getBaseInterestRate()
        if creditScore > 700
            rate = rate - 1.5  // Discount for good credit
        print("Loan offered at " + rate + "% interest")
        print("Bank: " + config.getBankName())

// ── Client 2: TransactionService ──
class TransactionService
    public processWireTransfer(amount: float): void
        config = BankConfiguration.getInstance()
        if amount > config.getMaxTransactionLimit()
            print("Amount exceeds limit of " + config.getMaxTransactionLimit())
            return
        fee = config.getWireTransferFee()
        print("Wire fee: $" + fee)

// ── Client 3: ComplianceSystem ──
class ComplianceSystem
    public verifyCustomer(): void
        config = BankConfiguration.getInstance()
        endpoint = config.getKycApiEndpoint()
        print("Sending KYC request to: " + endpoint)

// ── Driver ──
function main()
    dept = new LoanDepartment()
    service = new TransactionService()
    compliance = new ComplianceSystem()

    // All three clients use the same BankConfiguration instance
    dept.processLoanApplication(250000.00, 720)
    service.processWireTransfer(15000.00)
    compliance.verifyCustomer()

    // Verify they all share the same instance
    config1 = BankConfiguration.getInstance()
    config2 = BankConfiguration.getInstance()
    print("Same instance: " + (config1 == config2))  // true
```

Expected output:

```bash
Loan offered at 3.0% interest
Bank: Global Trust Bank
Wire fee: $25.00
Sending KYC request to: https://api.bank.com/kyc/v2
Same instance: true
```

## How It Works — Step by Step

1. **Application startup** — The banking application initializes. No `BankConfiguration` instance exists yet. The static `instance` field is `null`. Configuration has not been loaded.

2. **First request** — `LoanDepartment.processLoanApplication()` calls `BankConfiguration.getInstance()` to obtain the interest rate. This is the first call to `getInstance()`.

3. **Instance creation** — Inside `getInstance()`, the code checks `if instance is null`. Since this is the first call, the condition is true. The constructor `BankConfiguration()` executes, loading all configuration values from the configuration source (file, environment variables, or database).

4. **Instance stored** — The newly created `BankConfiguration` object is assigned to the static `instance` field. The fully populated configuration object is now ready for use.

5. **First client uses config** — `LoanDepartment` receives the `BankConfiguration` instance and calls `getBaseInterestRate()` to get the base rate of 4.5%. It calculates a discounted rate of 3.0% for the customer with good credit.

6. **Second client request** — `TransactionService.processWireTransfer()` calls `BankConfiguration.getInstance()`. This time, the `instance` field is not null, so the condition `if instance is null` is false. The existing instance is returned immediately — no file is reloaded, and no constructor runs.

7. **Third client request** — `ComplianceSystem.verifyCustomer()` calls `BankConfiguration.getInstance()`. Again, the same existing instance is returned. The KYC endpoint URL is read from the same configuration object that `LoanDepartment` and `TransactionService` used.

8. **Identity verification** — The driver explicitly compares the references from two separate calls to `getInstance()`. The comparison `config1 == config2` confirms that both variables point to the exact same object in memory. The singleton invariant holds.

## When to Use

- **Global configuration** — A single set of configuration parameters (interest rates, fees, API endpoints) must be shared across the entire banking application. Every department and service must see the same values.

- **Resource management** — Creating multiple instances of an expensive resource (database connection pool, configuration parser, thread pool) would waste system resources. A single instance is sufficient and efficient.

- **Coordinated state** — The state of the object must be coordinated across the system. In a banking application, if the central bank raises interest rates, the single `BankConfiguration` instance is updated once, and all services immediately see the new rates.

- **Logging and auditing** — A centralized logging or audit trail object ensures all log entries go to the same destination with consistent formatting. Every banking transaction event is recorded through the same channel.

- **Hardware or resource access** — A single point of access to a limited resource such as a printer, a license key, or a hardware security module (HSM) for cryptographic operations.

- **Registry or service locator** — A single registry that maps service names to implementations, allowing the banking application to locate services without hardcoding dependencies.

## When NOT to Use

- **Unit testing concerns** — Singleton introduces global state, which makes unit tests interdependent. If one test modifies the singleton's state, it affects other tests. Use dependency injection to pass configuration objects instead.

- **Hidden dependencies** — Singleton obscures the dependencies of a class. A method that calls `BankConfiguration.getInstance()` internally has a hidden dependency on the configuration, making the code harder to understand and refactor.

- **Concurrent initialization** — In a multithreaded banking application, two threads can both enter `getInstance()` before either completes creation, resulting in two instances. This requires thread-safe initialization (double-checked locking, static initializer, or `Lazy<T>`).

- **Single responsibility violation** — The Singleton class manages both its business logic (holding configuration) and its instantiation control (the `getInstance()` method). This mixes two responsibilities.

- **Scalability bottleneck** — A singleton that holds mutable state can become a contention point in a highly concurrent system. If the banking application processes thousands of transactions per second, threads queue up waiting to read or write the singleton.

- **Configuration changes at runtime** — If the banking application needs to support multiple configurations simultaneously (e.g., different configurations for different customers or regions), a single singleton is insufficient. Consider a configuration registry or factory instead.

## Real-World Occurrences

1. **Java `Runtime.getRuntime()`** — The Java Virtual Machine provides a single `Runtime` object per application via the `Runtime.getRuntime()` singleton. This object gives access to memory information, processor count, and system properties. Every part of the application uses the same `Runtime` instance to query JVM state.

2. **Spring Framework `BeanFactory` / `ApplicationContext`** — While Spring manages beans as singletons by default, the `ApplicationContext` itself is a singleton within a Spring application. It provides a single point of access to the entire dependency-injected object graph. The context is created once at startup and shared across the application.

3. **Python Module-Level Singletons** — Modules in Python are singletons. When you `import config` from two different modules, both receive the same module object. This is why module-level variables (like `config.settings`) are a natural singleton mechanism in Python. Libraries like Django use this pattern for their settings objects.

4. **.NET `HttpClient` and `Lazy<T>`** — Microsoft recommends using a single `HttpClient` instance per application to avoid socket exhaustion. The `Lazy<T>` class in .NET provides a thread-safe, lazy-initialized singleton pattern. Frameworks like ASP.NET Core use this internally for service instances.

5. **Rust `OnceLock` / `OnceCell`** — The Rust standard library provides `std::sync::OnceLock` for thread-safe one-time initialization of static data. Libraries like `lazy_static` and `once_cell` provide ergonomic singleton patterns. The popular `log` crate uses this approach for its global logger registration.

6. **Node.js Module Caching** — In Node.js, modules are cached after the first `require()`. This means that a module exporting a configuration object acts as a singleton. Express.js applications commonly use this pattern for application-level configuration, database connections, and logger instances.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Factory Method** | Factory Method creates multiple instances of different types; Singleton restricts to exactly one instance of a single type. | Choose Factory Method when you need to create families of related objects, not restrict to a single instance. |
| **Monostate** | Monostate achieves shared state through static members while allowing multiple instances of the class. Singleton restricts both the instance count and the access point. | Choose Monostate when you want the transparency of regular object creation but shared state, such as logging configuration where each module creates its own logger but all share the same settings. |
| **Dependency Injection** | DI passes dependencies explicitly rather than having objects fetch them globally. Singleton is a creational constraint; DI is a wiring strategy. | Choose DI when you need testability and loose coupling. A singleton can be injected as a scoped instance, but the DI container manages the lifecycle rather than the class itself. |
| **Object Pool** | Object Pool manages a finite set of reusable instances, returning them to the pool after use. Singleton maintains exactly one instance for the application lifetime. | Choose Object Pool when the cost of creating instances is high but you need more than one, such as database connections in a connection pool. |
| **Lazy Initialization** | Lazy Initialization is a technique for deferring object creation; Singleton often uses lazy initialization internally but adds the uniqueness constraint. | Choose plain Lazy Initialization when you only need deferred creation, not uniqueness. |
