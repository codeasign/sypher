The Factory Method pattern defines an interface or abstract class for creating an object, but lets subclasses decide which concrete class to instantiate. In this TypeScript implementation, we model a banking domain where different account types (SavingsAccount, CheckingAccount, LoanAccount) are created through a common `AccountFactory` abstract class. Each concrete factory encapsulates the construction logic for its specific account type, keeping the client code decoupled from the concrete product classes. TypeScript's static type system — interfaces, readonly modifiers, access modifiers, and strict null checks — makes the pattern safer and more self-documenting than in plain JavaScript.

## How the Implementation Works

### Class Diagram (Pseudocode)

```
abstract class AccountFactory {
  openAccount(owner, initialDeposit): Account
    // common validation and logging
    // delegates to abstract createAccount
  abstract createAccount(owner, initialDeposit): Account
    // subclasses implement this
}

class SavingsAccountFactory extends AccountFactory {
  createAccount(owner, initialDeposit): SavingsAccount
    // constructs SavingsAccount with specific interest rate
}

class CheckingAccountFactory extends AccountFactory {
  createAccount(owner, initialDeposit): CheckingAccount
    // constructs CheckingAccount with overdraft limit
}

class LoanAccountFactory extends AccountFactory {
  createAccount(owner, initialDeposit): LoanAccount
    // constructs LoanAccount with interest rate and term
}

interface Account {
  readonly accountNumber: string
  readonly owner: string
  readonly balance: number
  deposit(amount: number): void
  withdraw(amount: number): void
  getAccountInfo(): string
}
```

### OOP Concepts Applied

**Abstraction** — The `Account` interface exposes only the essential operations (`deposit`, `withdraw`, `getAccountInfo`) without revealing internal details such as field storage, interest calculation, or overdraft logic. The `AccountFactory` abstract class provides a template method `openAccount` that hides the creation workflow.

**Encapsulation** — Each concrete product class keeps its fields private (or protected with readonly exposure through getters). The factory classes encapsulate the construction logic, including account number generation, interest rate assignment, and initial validation.

**Inheritance** — `SavingsAccountFactory`, `CheckingAccountFactory`, and `LoanAccountFactory` inherit from `AccountFactory` and override the `createAccount` method. Each concrete product implements the `Account` interface.

**Polymorphism** — Client code works with the `Account` interface and `AccountFactory` abstract class. The same `openAccount` call produces different account types depending on which factory subclass is used, and `getAccountInfo` behaves differently per product.

### SOLID Principles Applied

**Single Responsibility Principle (SRP)** — Each factory class has one responsibility: creating a specific type of account. Each product class handles only its own account behavior. `SavingsAccount` does not know about checking overdrafts, and `LoanAccountFactory` does not know about savings interest rates.

**Open/Closed Principle (OCP)** — New account types can be added without modifying existing code. To introduce a `MoneyMarketAccount`, we create `MoneyMarketAccount` (implementing `Account`) and `MoneyMarketAccountFactory` (extending `AccountFactory`). No existing factory or product class needs to change.

**Liskov Substitution Principle (LSP)** — Every concrete product can be substituted wherever `Account` is expected. Every concrete factory can be substituted wherever `AccountFactory` is expected. `LoanAccount` adheres to the same contract as `SavingsAccount` — deposit adds funds, withdraw removes them.

**Interface Segregation Principle (ISP)** — The `Account` interface is narrow and focused on account operations. If a loan-specific method like `makePayment` is needed, it belongs on `LoanAccount` directly, not on the shared `Account` interface. Clients depending on `Account` see only what they need.

**Dependency Inversion Principle (DIP)** — High-level client code depends on the `Account` abstraction and the `AccountFactory` abstraction, not on concrete classes. The concrete factories and products are injected or selected at runtime, keeping the client decoupled from implementation details.

### Account (Product Interface)
```typescript
// OOP: Abstraction — defines the contract for all account types.
// ISP: Narrow interface — only the operations every account must support.
interface Account {
  readonly accountNumber: string;
  readonly owner: string;
  readonly balance: number;

  deposit(amount: number): void;
  withdraw(amount: number): void;
  getAccountInfo(): string;
}
```

### SavingsAccount (Concrete Product)
```typescript
// OOP: Encapsulation — fields are private, exposed only through methods.
// LSP: Fully substitutable for Account — no preconditions stronger than
//       the interface contract.
class SavingsAccount implements Account {
  readonly accountNumber: string;
  readonly owner: string;
  private _balance: number;
  private readonly interestRate: number;

  constructor(
    accountNumber: string,
    owner: string,
    initialDeposit: number,
    interestRate: number,
  ) {
    this.accountNumber = accountNumber;
    this.owner = owner;
    this._balance = initialDeposit;
    this.interestRate = interestRate;
  }

  get balance(): number {
    return this._balance;
  }

  deposit(amount: number): void {
    if (amount <= 0) {
      throw new Error('SavingsAccount: deposit amount must be positive');
    }
    this._balance += amount;
  }

  withdraw(amount: number): void {
    if (amount <= 0) {
      throw new Error('SavingsAccount: withdrawal amount must be positive');
    }
    if (amount > this._balance) {
      throw new Error('SavingsAccount: insufficient funds');
    }
    this._balance -= amount;
  }

  applyInterest(): void {
    const interest = this._balance * this.interestRate;
    this._balance += interest;
  }

  getAccountInfo(): string {
    return `Savings Account #${this.accountNumber} | Owner: ${this.owner} | Balance: $${this._balance.toFixed(2)} | Rate: ${(this.interestRate * 100).toFixed(1)}%`;
  }
}
```

### CheckingAccount (Concrete Product)
```typescript
// OOP: Encapsulation — hides overdraft limit and balance.
// LSP: Substitutable for Account — withdraw allows negative balance
//       up to the overdraft limit, which is a valid extension.
class CheckingAccount implements Account {
  readonly accountNumber: string;
  readonly owner: string;
  private _balance: number;
  private readonly overdraftLimit: number;

  constructor(
    accountNumber: string,
    owner: string,
    initialDeposit: number,
    overdraftLimit: number,
  ) {
    this.accountNumber = accountNumber;
    this.owner = owner;
    this._balance = initialDeposit;
    this.overdraftLimit = overdraftLimit;
  }

  get balance(): number {
    return this._balance;
  }

  deposit(amount: number): void {
    if (amount <= 0) {
      throw new Error('CheckingAccount: deposit amount must be positive');
    }
    this._balance += amount;
  }

  withdraw(amount: number): void {
    if (amount <= 0) {
      throw new Error('CheckingAccount: withdrawal amount must be positive');
    }
    if (this._balance - amount < -this.overdraftLimit) {
      throw new Error('CheckingAccount: overdraft limit exceeded');
    }
    this._balance -= amount;
  }

  getAccountInfo(): string {
    return `Checking Account #${this.accountNumber} | Owner: ${this.owner} | Balance: $${this._balance.toFixed(2)} | Overdraft: $${this.overdraftLimit.toFixed(2)}`;
  }
}
```

### LoanAccount (Concrete Product)
```typescript
// OOP: Encapsulation — hides loan-specific fields.
// LSP: Substitutable for Account — deposit acts as payment, withdraw
//       acts as loan disbursement.
class LoanAccount implements Account {
  readonly accountNumber: string;
  readonly owner: string;
  private _balance: number;
  private readonly interestRate: number;
  private readonly loanAmount: number;

  constructor(
    accountNumber: string,
    owner: string,
    loanAmount: number,
    interestRate: number,
  ) {
    this.accountNumber = accountNumber;
    this.owner = owner;
    this.loanAmount = loanAmount;
    this.interestRate = interestRate;
    // Balance starts negative (what the borrower owes)
    this._balance = -loanAmount;
  }

  get balance(): number {
    return this._balance;
  }

  deposit(amount: number): void {
    if (amount <= 0) {
      throw new Error('LoanAccount: payment amount must be positive');
    }
    this._balance += amount;
  }

  withdraw(amount: number): void {
    if (amount <= 0) {
      throw new Error('LoanAccount: disbursement amount must be positive');
    }
    // Allow additional disbursement up to the original loan amount
    const currentOutstanding = -this._balance;
    if (currentOutstanding + amount > this.loanAmount) {
      throw new Error('LoanAccount: disbursement exceeds loan limit');
    }
    this._balance -= amount;
  }

  makePayment(amount: number): void {
    this.deposit(amount);
  }

  getAccountInfo(): string {
    const outstanding = Math.abs(this._balance);
    return `Loan Account #${this.accountNumber} | Borrower: ${this.owner} | Outstanding: $${outstanding.toFixed(2)} | Rate: ${(this.interestRate * 100).toFixed(1)}% | Original: $${this.loanAmount.toFixed(2)}`;
  }
}
```

### AccountFactory (Abstract Creator)
```typescript
// OOP: Abstraction — defines the factory method contract.
// DIP: High-level openAccount depends on the abstract createAccount,
//       not on concrete product constructors.
abstract class AccountFactory {
  private static nextAccountNumber = 1000;

  protected generateAccountNumber(prefix: string): string {
    const num = ++AccountFactory.nextAccountNumber;
    return `${prefix}-${num}`;
  }

  /**
   * Factory Method — subclasses override this to create specific products.
   */
  abstract createAccount(owner: string, initialDeposit: number): Account;

  /**
   * Template Method — calls the factory method and adds common logic.
   * This is the primary entry point for clients.
   */
  openAccount(owner: string, initialDeposit: number): Account {
    if (initialDeposit < 0) {
      throw new Error('Initial deposit cannot be negative');
    }
    const account = this.createAccount(owner, initialDeposit);
    console.log(`[OPEN] ${account.getAccountInfo()}`);
    return account;
  }
}
```

### SavingsAccountFactory (Concrete Creator)
```typescript
// SRP: Only responsible for creating SavingsAccount instances.
// OCP: Open for extension via new factory, closed for modification.
class SavingsAccountFactory extends AccountFactory {
  private static readonly INTEREST_RATE = 0.035;

  createAccount(owner: string, initialDeposit: number): SavingsAccount {
    const accountNumber = this.generateAccountNumber('SAV');
    if (initialDeposit < 100) {
      throw new Error('SavingsAccount: minimum initial deposit is $100');
    }
    return new SavingsAccount(
      accountNumber,
      owner,
      initialDeposit,
      SavingsAccountFactory.INTEREST_RATE,
    );
  }
}
```

### LoanAccountFactory (Concrete Creator)
```typescript
// SRP: Only responsible for creating LoanAccount instances.
// OCP: Open for extension via new factory, closed for modification.
class LoanAccountFactory extends AccountFactory {
  private static readonly INTEREST_RATE = 0.067;
  private static readonly MIN_LOAN = 1000;
  private static readonly MAX_LOAN = 100000;

  createAccount(owner: string, loanAmount: number): LoanAccount {
    const accountNumber = this.generateAccountNumber('LOAN');
    if (loanAmount < LoanAccountFactory.MIN_LOAN) {
      throw new Error(`LoanAccount: minimum loan amount is $${LoanAccountFactory.MIN_LOAN}`);
    }
    if (loanAmount > LoanAccountFactory.MAX_LOAN) {
      throw new Error(`LoanAccount: maximum loan amount is $${LoanAccountFactory.MAX_LOAN}`);
    }
    return new LoanAccount(
      accountNumber,
      owner,
      loanAmount,
      LoanAccountFactory.INTEREST_RATE,
    );
  }
}
```

### Main / Driver
```typescript
// Polymorphism — treats all accounts as Account, all factories as AccountFactory.
// DIP: Client depends on AccountFactory and Account abstractions.
function formatSeparator(label: string): void {
  console.log(`\n${'═'.repeat(60)}`);
  console.log(`  ${label}`);
  console.log(`${'═'.repeat(60)}`);
}

function main(): void {
  // Alice opens a SavingsAccount with a $5,000 initial deposit
  formatSeparator('Alice opens a SavingsAccount');
  const savingsFactory: AccountFactory = new SavingsAccountFactory();
  const aliceAccount: Account = savingsFactory.openAccount('Alice Johnson', 5000);
  console.log(`  Balance: $${aliceAccount.balance.toFixed(2)}`);
  aliceAccount.deposit(1500);
  console.log(`  After $1,500 deposit: $${aliceAccount.balance.toFixed(2)}`);
  aliceAccount.withdraw(200);
  console.log(`  After $200 withdrawal: $${aliceAccount.balance.toFixed(2)}`);

  // Apply interest (cast to SavingsAccount for type-specific method)
  const aliceSavings = aliceAccount as SavingsAccount;
  aliceSavings.applyInterest();
  console.log(`  After interest applied: $${aliceSavings.balance.toFixed(2)}`);

  // Bob opens a CheckingAccount with a $2,000 initial deposit
  formatSeparator('Bob opens a CheckingAccount');
  const checkingFactory: AccountFactory = new CheckingAccountFactory();
  const bobAccount: Account = checkingFactory.openAccount('Bob Smith', 2000);
  console.log(`  Balance: $${bobAccount.balance.toFixed(2)}`);
  bobAccount.deposit(1000);
  console.log(`  After $1,000 deposit: $${bobAccount.balance.toFixed(2)}`);
  bobAccount.withdraw(500);
  console.log(`  After $500 withdrawal: $${bobAccount.balance.toFixed(2)}`);
  // Bob uses the overdraft
  bobAccount.withdraw(2800);
  console.log(`  After $2,800 withdrawal (overdraft): $${bobAccount.balance.toFixed(2)}`);

  // Carol opens a LoanAccount with a $25,000 loan
  formatSeparator('Carol opens a LoanAccount');
  const loanFactory: AccountFactory = new LoanAccountFactory();
  const carolAccount: Account = loanFactory.openAccount('Carol Williams', 25000);
  console.log(`  Balance: $${carolAccount.balance.toFixed(2)}`);
  console.log(`  Info: ${carolAccount.getAccountInfo()}`);

  // Make a payment (cast to LoanAccount for type-specific method)
  const carolLoan = carolAccount as LoanAccount;
  carolLoan.makePayment(500);
  console.log(`  After $500 payment: $${carolLoan.balance.toFixed(2)}`);

  // Summary
  formatSeparator('Account Summary');
  console.log(`  ${aliceAccount.getAccountInfo()}`);
  console.log(`  ${bobAccount.getAccountInfo()}`);
  console.log(`  ${carolAccount.getAccountInfo()}`);
}

main();
```

## Language-Specific Notes

- **Interfaces over abstract classes for pure contracts** — In TypeScript, the `Account` product is defined as an `interface` rather than an abstract class. This is idiomatic: interfaces provide a lightweight, compile-time-only contract with no runtime cost. The `AccountFactory` is an abstract class because it needs to hold shared state (`nextAccountNumber`) and provide a concrete method (`openAccount`) alongside the abstract factory method.

- **Readonly modifiers enforce immutability** — The `readonly` modifier on `accountNumber`, `owner`, and `balance` in the `Account` interface signals that these properties should not be reassigned after construction. TypeScript enforces this at compile time, preventing accidental mutation. The `balance` property uses a getter to expose a computed value while keeping the backing field private.

- **Access modifiers provide encapsulation guarantees** — TypeScript's `private`, `protected`, and `public` keywords make encapsulation explicit at the language level. Fields like `_balance` and `interestRate` are marked `private` to prevent external access, while `protected` on the factory base class allows subclasses to call `generateAccountNumber` without exposing it to clients.

- **Type annotations on factory return types** — Each concrete factory overrides `createAccount` with a precise return type (`SavingsAccount`, `CheckingAccount`, `LoanAccount`) rather than the general `Account`. This is a covariant return type — a TypeScript feature that allows the factory method to return a more specific type than the abstract base declares, enabling the client to access type-specific methods (e.g., `applyInterest`) through a cast.

- **Strict null checks prevent runtime errors** — With `strict: true` in `tsconfig.json`, TypeScript enforces that all fields are initialized in the constructor and that nullable values are handled explicitly. Every constructor parameter in this implementation is non-nullable, and methods validate inputs at runtime, creating a defense-in-depth approach to reliability.

- **No runtime overhead for interfaces** — TypeScript interfaces are erased at compile time and have zero runtime footprint. This contrasts with abstract classes, which generate constructor code in the JavaScript output. Using an interface for `Account` means the pattern introduces no unnecessary runtime cost while still providing full type safety during development.
