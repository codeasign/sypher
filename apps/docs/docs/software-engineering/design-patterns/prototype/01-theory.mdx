The Prototype pattern creates new objects by cloning an existing object, known as the prototype, rather than calling a constructor directly. This avoids the cost of creating objects from scratch when the initial setup is expensive or complex, and it allows you to compose and configure objects at runtime before cloning them. The pattern defines a `clone()` method on the prototype interface, and each concrete prototype implements cloning by copying its own internal state to a new instance. The client never needs to know the concrete class of the object it is cloning; it depends only on the abstract prototype interface.

What makes the Prototype pattern unique among creational patterns is that it decouples the object creation process from the class hierarchy. Instead of a factory deciding which class to instantiate, the object itself decides how to duplicate its own state. This shifts the focus from "how to construct" to "what to copy," and it enables patterns like prototype registries where a catalog of pre-configured objects can be dynamically populated at runtime from external configuration.

## The Problem Without This Pattern

Consider a bank that offers multiple financial product templates. Each template has pre-configured defaults: interest rates, fee structures, minimum balance requirements, overdraft limits, and more. A bank associate needs to create customized accounts for customers based on these templates.

Without the Prototype pattern, every new account must be constructed from scratch with all default values passed explicitly:

```python
# BROKEN — every account constructed from scratch
def create_savings_account(holder, initial_deposit):
    return SavingsAccount(
        account_holder=holder,
        balance=initial_deposit,
        interest_rate=0.035,      # hardcoded default
        monthly_fee=0.0,          # hardcoded default
        min_balance=100.0,        # hardcoded default
        withdrawal_limit=6,        # hardcoded default
        overdraft_protection=False # hardcoded default
    )

def create_premium_checking(holder, initial_deposit):
    return CheckingAccount(
        account_holder=holder,
        balance=initial_deposit,
        interest_rate=0.001,      # hardcoded default
        monthly_fee=25.0,         # hardcoded default
        min_balance=1500.0,       # hardcoded default
        overdraft_limit=2000.0,   # hardcoded default
        atm_fee_reimbursement=True # hardcoded default
    )
```

What breaks:

- **Duplication** — Every factory method repeats the same default values. If the bank changes the standard savings interest rate from 3.5% to 4.0%, every factory method and constructor call site must be updated. In a large banking application with dozens of account types and hundreds of creation points, this becomes a maintenance nightmare. A single missed update introduces inconsistent account configurations across the system.

- **Rigidity** — Adding a new field to a product template (e.g., `early_withdrawal_penalty`) requires updating every creation point. The bank cannot introduce new product features without touching potentially hundreds of lines of factory code. This makes the system resistant to change and slows down product innovation.

- **No runtime customization** — The template is hardcoded into the factory. There is no way to pre-configure a template, clone it, then tweak a few fields for a specific customer. What if a bank manager wants to offer a promotional 4.0% rate to a high-value customer while keeping the standard 3.5% for everyone else? Without cloning, every such variation requires a separate factory method or a massive constructor parameter list.

- **Constructor explosion** — As product templates grow in complexity, constructors become unwieldy with dozens of parameters. A standard savings account might have 15+ configurable parameters: interest rate, monthly fee, minimum balance, withdrawal limit, overdraft protection flag, ATM fee reimbursement flag, paperless statement flag, joint account flag, beneficiary count, early withdrawal penalty percentage, CD term in months, automatic renewal flag, grace period in days, and promotional rate flag. Clients must remember the order and meaning of every parameter, and default values are scattered across calling code.

- **No mechanism for runtime configuration** — Financial product terms change frequently based on market conditions. Interest rates are adjusted quarterly, fee structures are updated annually, and promotional offers are introduced monthly. Each change requires a code deployment. The system cannot load updated product templates from a database or configuration file and begin using them immediately.

The Prototype pattern solves all of these problems by letting you clone a pre-configured template object and then customize the clone. The templates are configured once, stored in a registry, and cloned as needed. When terms change, you update the prototype in one place, and every subsequent clone automatically reflects the new values.

### Participants

- **Prototype (`AccountPrototype`)** — Declares the `clone()` interface. In some languages this is a `Protocol` (Python), an `interface` (Java, TypeScript, C#), a `trait` (Rust), or a prototype object (JavaScript). The interface typically includes only `clone()` and possibly a display or accessor method. The interface is intentionally small to satisfy the Interface Segregation Principle.

- **ConcretePrototype (`StandardSavingsPrototype`, `PremiumCheckingPrototype`, `BusinessLoanPrototype`)** — Implements the `clone()` method. Each carries a pre-configured set of default values for a specific financial product: interest rate, monthly fee, minimum balance, withdrawal limit, overdraft limit, ATM fee reimbursement, and so on. The `clone()` method creates a new instance of the same class and copies all field values to the new instance. Some languages support shallow copy natively (e.g., `Object.clone()` in Java, `MemberwiseClone()` in C#, `#[derive(Clone)]` in Rust), while others require explicit field-by-field copying.

- **Client (`BankBranch`)** — Clones a prototype and customizes the copy for specific customer needs. The client looks up the desired prototype from a registry (typically a `Map` or `Dictionary` keyed by product name), calls `clone()` to obtain a fresh copy, and then sets customer-specific fields like `account_holder` and `balance`. The client never calls constructors directly and never needs to know the concrete class of the prototype it is cloning.

### How the Participants Interact

The interaction follows a consistent sequence. The client maintains a registry of prototype objects, each configured with its product defaults. When a new object is needed, the client retrieves the appropriate prototype from the registry and calls `clone()`. The prototype returns a new object with all the same default values. The client then customizes the clone with instance-specific data. The original prototype remains in the registry unchanged, ready for the next cloning operation. This decouples the client from the concrete classes and centralizes default configuration in the prototype objects themselves.

## Pseudocode — Banking Example

The following pseudocode shows how a bank uses the Prototype pattern to create customized accounts from pre-configured financial product templates:

```
// PSEUDOCODE — Prototype Pattern in Banking Domain
// Participants labeled with [ROLE]

// [PROTOTYPE] — Interface for all clonable account templates
interface AccountPrototype:
    method clone(): AccountPrototype
    method display_config(): string

// [CONCRETE PROTOTYPE] — Standard Savings product template
class StandardSavingsPrototype implements AccountPrototype:
    fields:
        account_type: string = "Standard Savings"
        interest_rate: float = 0.035
        monthly_fee: float = 0.0
        min_balance: float = 100.0
        withdrawal_limit: int = 6
        overdraft_protection: bool = false
        account_holder: string = ""
        balance: float = 0.0

    method clone() -> AccountPrototype:
        return deep copy of this

    method display_config() -> string:
        return format all fields

// [CONCRETE PROTOTYPE] — Premium Checking product template
class PremiumCheckingPrototype implements AccountPrototype:
    fields:
        account_type: string = "Premium Checking"
        interest_rate: float = 0.001
        monthly_fee: float = 25.0
        min_balance: float = 1500.0
        overdraft_limit: float = 2000.0
        atm_fee_reimbursement: bool = true
        account_holder: string = ""
        balance: float = 0.0

    method clone() -> AccountPrototype:
        return deep copy of this

    method display_config() -> string:
        return format all fields

// [CONCRETE PROTOTYPE] — Business Loan product template
class BusinessLoanPrototype implements AccountPrototype:
    fields:
        account_type: string = "Business Loan"
        apr: float = 0.0675
        origination_fee: float = 500.0
        max_principal: float = 50000.0
        term_months: int = 60
        prepayment_penalty: bool = true
        account_holder: string = ""
        principal: float = 0.0

    method clone() -> AccountPrototype:
        return deep copy of this

    method display_config() -> string:
        return format all fields

// [CLIENT] — Bank associate creates accounts
class BankBranch:
    method __init__():
        this.prototypes = {
            "savings": StandardSavingsPrototype(),
            "checking": PremiumCheckingPrototype(),
            "loan": BusinessLoanPrototype()
        }

    method open_account(template_type, holder, initial_amount):
        account = this.prototypes[template_type].clone()
        account.account_holder = holder
        account.balance = initial_amount
        if template_type == "loan":
            account.principal = initial_amount
        return account

// Driver
branch = BankBranch()
alice_savings = branch.open_account("savings", "Alice", 1000.0)
print(alice_savings.display_config())
```

The Prototype pattern's cloning workflow unfolds across seven distinct steps, each of which plays a specific role in decoupling object creation from client code:

1. **Client submits a request** — A bank associate initiates the account opening process by specifying three pieces of information: the product template type (e.g., "savings"), the customer name (e.g., "Alice"), and the initial deposit amount (e.g., $1,000.00). The associate does not know which concrete class implements the savings product — they only know the template's name. This abstraction is the first level of decoupling: the client works with template names, not class names.

2. **Branch looks up the prototype** — The `BankBranch` registry performs a lookup by template type. The registry is typically a hash map or dictionary that stores key-value pairs where the key is a string like `"savings"` and the value is a prototype object. If the template type does not exist, the registry raises an error or returns a fallback default. In this case, the lookup finds the `StandardSavingsPrototype` instance that was registered at startup. The registry lookup is O(1) on average, making it suitable for high-throughput banking systems.

3. **Prototype is returned from registry** — The registry returns the pre-configured prototype object to the client. The prototype holds all the default values for that product: 3.5% annual percentage yield, $0 monthly maintenance fee, $100 minimum balance requirement, 6 withdrawals per month limit, and no overdraft protection. The prototype object is a fully initialized, ready-to-clone template. It was configured once when the application started and has been sitting in the registry waiting to be cloned.

4. **Client calls `clone()` on the prototype** — The bank branch invokes the `clone()` method on the prototype. This is the critical step: the branch calls `clone()` on the abstract `AccountPrototype` interface, not on a concrete class. The prototype's `clone()` method creates a new object of the same concrete type and copies all field values from the original to the copy. The cloning mechanism varies by language: Java uses covariant return types with explicit field copying, C# uses `MemberwiseClone()` for shallow copy, Rust uses `#[derive(Clone)]`, and Python constructs a fresh instance and copies fields manually. The key invariant is that the clone is a fully independent object with the same state as the original.

5. **Cloned copy is returned** — The `clone()` method returns a new, independent object with all of the prototype's default values. The clone is structurally identical to the prototype but is a separate object in memory. Any subsequent modifications to the clone do not affect the prototype, and any modifications to the prototype (e.g., rate changes) do not retroactively affect clones that have already been created. However, future clones from the updated prototype will reflect the new values. This independence is crucial for correctness: each customer's account is a separate object that can be modified independently.

6. **Client customizes the clone** — The bank branch sets customer-specific fields on the clone. The account holder name is set to "Alice", and the balance is set to $1,000.00. For loan products, the principal is set instead of the balance. The clone's default values (interest rate, fees, limits) are preserved from the prototype and do not need to be touched. This is the key benefit: the bank associate only provides the customer-specific information, and the product defaults are carried over automatically. The customization step is typically small (2-3 field assignments) compared to the full configuration (10+ fields).

7. **Configured account is returned** — The fully configured account object is returned to the caller. The account is ready for use: it has the correct product type, interest rate, fee structure, customer name, and balance. The original prototype remains unchanged in the registry, ready for the next cloning operation. The registry's prototype can be used indefinitely to create any number of customized clones, each with different customer-specific data but the same product defaults. A single prototype can be cloned millions of times over the lifetime of a banking application.

The prototype registry also enables a powerful maintenance benefit. When the bank changes product terms — for example, raising the savings interest rate from 3.5% to 4.0% — they update the single prototype object in the registry. Every subsequent clone automatically inherits the new rate. Previously created clones are unaffected (they keep the old rate), which is correct behavior for existing accounts. This separation between current and historical configurations is a natural consequence of the clone-on-demand approach.

## When to Use

- **Object creation is expensive** — Constructing an account template involves loading default values from a configuration file, database query, or network service call. Cloning avoids this overhead entirely. Once the prototype is configured, cloning it is a simple memory operation that does not hit the database, read files, or make network requests. This is particularly valuable in high-throughput systems like loan origination platforms that create thousands of account objects per hour.

- **Objects have many configuration fields** — Financial product templates typically have 10 or more fields: interest rate, monthly fee, minimum balance, withdrawal limit, overdraft protection flag, ATM reimbursement flag, paperless statement flag, joint account eligibility, beneficiary slots, early withdrawal penalty, term length, and promotional rate flags. Cloning avoids repeating these defaults at every creation site and eliminates the risk of inconsistent default values across different parts of the application.

- **You need to create objects that differ only slightly from a template** — A bank associate clones the standard savings template and only changes the customer name and initial deposit. All other values — interest rate, fees, limits — stay as the product defaults. This is the common case in banking: the vast majority of accounts are standard products with minor customer-specific customizations.

- **You want to avoid subclassing for every variation** — A naive object-oriented approach creates a subclass for every product variation: `AliceSavingsAccount`, `BobSavingsAccount`, `PromotionalSavingsAccount`, `StudentSavingsAccount`. This leads to a class explosion that is impossible to maintain. The Prototype pattern eliminates subclasses by treating each variation as a clone with modified fields.

- **Runtime configuration of object templates** — The set of available product templates can be loaded from a database at application startup. The bank's product management system defines templates with specific terms, and the application loads them into the prototype registry. The `clone()` method works regardless of how the prototype was configured, so templates can be changed without code modifications.

- **You need to hide concrete product classes from clients** — The `BankBranch` only interacts with `AccountPrototype` interface. It never imports or references `StandardSavingsPrototype`, `PremiumCheckingPrototype`, or `BusinessLoanPrototype` directly. This means these classes can be in different packages, modules, or even in dynamically loaded plugins.

- **You need to support undo/redo operations** — By combining the Prototype pattern with the Memento pattern, you can save a clone of an object's state before modification and restore it later by cloning the saved prototype. This is useful in banking configuration tools where product managers need to experiment with different rate scenarios and roll back changes.

- **Prototype composition** — You can combine prototypes by cloning one and embedding it inside another. For example, a `JointAccountPrototype` might clone a `StandardSavingsPrototype` and add a second account holder field. This creates a form of prototype composition that does not require changing the original prototype.

## When NOT to Use

- **Simple objects with few fields** — If an account has only 2-3 fields (e.g., just a name and balance), a constructor or factory function is simpler and more readable than setting up a prototype registry. The overhead of defining a prototype interface, implementing `clone()` methods, and managing a registry is not justified for trivial objects.

- **Objects that require deep cloning of complex graphs** — If the prototype contains references to shared resources (database connections, file handles, network sockets), deep cloning may unintentionally duplicate these resources, leading to connection leaks or file handle exhaustion. Shallow copy may be more appropriate in these cases, but shallow copy shares references between the prototype and the clone, which can cause unexpected mutations. Carefully consider whether your object graph is clone-safe.

- **Circular references in object graphs** — Deep cloning object graphs with circular references requires special handling such as memoization or a visited-objects set. Without this, the cloning algorithm enters an infinite loop. The Prototype pattern becomes more complex to implement than it is worth in these scenarios, and a Builder or Factory Method may be a better fit.

- **Frequent prototype changes at runtime** — If the prototype objects themselves are mutated frequently, every clone starts with a moving baseline. This makes it difficult to reason about the state of cloned objects. In this case, consider using immutable prototypes and creating new prototypes when terms change, rather than mutating existing ones.

- **When object identity matters** — If the system relies on `===` (JavaScript), `is` (Python), `==` (Java reference equality), `ReferenceEquals` (C#), or `std::ptr::eq` (Rust) to check object identity, cloning breaks these assumptions. Each clone is a distinct object with a different memory address. The system must use structural equality (field-by-field comparison) rather than reference equality.

- **When constructors perform essential initialization** — If the constructor sets up complex invariants (e.g., registering the account with a central banking system, allocating unique account numbers, setting up audit logs), cloning bypasses the constructor. The clone must replicate or trigger these side effects manually, which is error-prone and easy to forget.

## Real-World Occurrences

1. **Java `Object.clone()` and `Cloneable`** — Every Java object inherits `clone()` from `Object`. The `Cloneable` marker interface signals that `clone()` performs a valid field-by-field copy. Java's standard library uses this extensively: `ArrayList.clone()`, `HashMap.clone()`, `HashSet.clone()`, and `TreeMap.clone()` all return shallow copies of the collection. The Java language specification requires that `clone()` returns `null` and throws `CloneNotSupportedException` for non-`Cloneable` objects. This design has been controversial since Java 1.0 because the marker interface pattern feels like the Prototype pattern but with the clone mechanism inherited rather than defined by the interface.

2. **JavaScript `Object.create()` and `structuredClone()`** — JavaScript's prototypal inheritance is itself a form of the Prototype pattern. The `Object.create(proto)` method creates a new object with `proto` as its internal `[[Prototype]]` link, enabling prototypal delegation. The `structuredClone()` global function (standardized in ES2023) provides deep cloning for any serializable object, handling circular references, `Date`, `Map`, `Set`, `RegExp`, `ArrayBuffer`, and `Blob` objects. This is the recommended approach for deep cloning in modern JavaScript applications.

3. **Python `copy` module** — Python's `copy.copy()` and `copy.deepcopy()` functions provide generic shallow and deep cloning. Classes can customize cloning by implementing `__copy__()` (for shallow copy) and `__deepcopy__(memo)` (for deep copy with memoization for circular references). Python dataclasses support `__copy__` and `__deepcopy__` out of the box, and ORM frameworks like SQLAlchemy use these methods to clone model instances. The `copy` module is used extensively by the standard library, including `collections` and `functools`.

4. **C# `ICloneable` and `MemberwiseClone()`** — The .NET `System.ICloneable` interface defines a single `Clone()` method. Many .NET collection types implement `ICloneable`: `List<T>`, `Dictionary<K,V>`, `SortedList<K,V>`, `Queue<T>`, `Stack<T>`. The protected `MemberwiseClone()` method in `System.Object` provides shallow copy, which is the simplest way to implement `ICloneable`. However, the `ICloneable` interface has been criticized because it does not specify whether `Clone()` performs shallow or deep copy, leading to inconsistent implementations across the .NET ecosystem. The .NET documentation recommends that you consider whether you need `ICloneable` or can use a copy constructor instead.

5. **Rust `Clone` trait** — Rust's `Clone` trait is a core part of the language, appearing in the prelude. Types that implement `Clone` can be duplicated via the `.clone()` method. The `#[derive(Clone)]` macro generates the implementation automatically for structs and enums where all fields implement `Clone`. The standard library uses this pervasively: `String::clone()`, `Vec::clone()`, `HashMap::clone()`, `Box::clone()`, and `Rc::clone()` are all used in virtually every Rust program. Rust's ownership model makes cloning explicit and visible at every call site, which aligns with the language's philosophy of making copying and moving costs transparent.

6. **Spring Framework `Prototype` scope** — The Spring Framework supports a prototype bean scope where a new bean instance is created every time the bean is requested from the application context. While Spring's prototype scope is technically a factory pattern, the name reflects the Prototype pattern's influence. Spring's `@Scope("prototype")` annotation is used when stateful beans must not be shared between callers.

7. **Prototypal inheritance in Self and JavaScript** — The Self programming language pioneered prototype-based programming, where objects are created by cloning existing objects (called prototypes) rather than by instantiating classes. JavaScript's `prototype` property on constructor functions, `Object.create()`, and class inheritance via `extends` all trace their lineage to Self's prototype-based model. The Prototype design pattern from GoF and prototype-based OOP are different concepts, but they share the core idea: creating new objects by copying existing ones.

8. **Apache BeanUtils and Spring BeanUtils** — The Apache Commons BeanUtils library provides `cloneBean()` which performs a shallow property-by-property clone of a JavaBean. Spring Framework's `BeanUtils.copyProperties()` copies properties from one object to another, which is a form of cloning. These utilities are used extensively in enterprise Java applications for DTO (Data Transfer Object) cloning, entity-to-DTO mapping, and configuration object copying.

9. **Protobuf `Clone()` method** — Google's Protocol Buffers generated Java classes include a `Clone()` method that creates a deep copy of the message object. This is used extensively in gRPC services where request and response messages must be cloned before modification to avoid mutating shared references in concurrent request handlers. ## Implementation Considerations

Implementing the Prototype pattern requires making several technical decisions that affect correctness, performance, and maintainability. Each decision depends on the language, the object graph, and the specific requirements of the application.

### Shallow Copy vs Deep Copy

The most critical implementation decision is whether `clone()` performs a shallow copy or a deep copy. A shallow copy duplicates the object's immediate fields but shares references to any nested objects. A deep copy recursively duplicates every object in the object graph.

**Shallow copy** is appropriate when:
- All fields are value types or primitives (numbers, booleans, enums). In the Banking domain, fields like `interest_rate`, `monthly_fee`, `min_balance`, and `overdraft_protection` are all value types, so shallow copy is sufficient.
- Shared references between the prototype and clone are intentional. For example, if the prototype holds a reference to an immutable `BankConfiguration` object, sharing that reference between clones is safe and efficient.

**Deep copy** is appropriate when:
- Fields include mutable objects that should not be shared. If a prototype holds a `List<String>` of beneficiary names, sharing that list between clones means modifying one clone's beneficiaries would affect all other clones.
- The clone must be fully independent from the prototype. In banking applications, account objects must be completely independent to prevent accidental data corruption.
- The object graph contains nested value objects such as `Address`, `ContactInfo`, or `AccountPreferences`.

Some languages provide built-in shallow copy mechanisms: Java's `Object.clone()` performs shallow copy by default; C#'s `MemberwiseClone()` also performs shallow copy. Deep copy must be implemented manually or through serialization-based techniques. Python's `copy.deepcopy()` handles deep copy generically but may be slow for complex graphs. JavaScript's `structuredClone()` provides deep copy for serializable objects. Rust's `Clone` trait can implement shallow or deep copy as needed since it is defined by the implementor.

### Cloning Mechanism per Language

Different languages offer different cloning mechanisms, and choosing the right one affects both performance and correctness:

| Language | Shallow Copy | Deep Copy | Custom Clone |
|---|---|---|---|
| Java | `Object.clone()` (protected) | Serialization or manual | Override `clone()` |
| JavaScript | `Object.assign()` / spread | `structuredClone()` | Manual in `clone()` method |
| TypeScript | `Object.assign()` / spread | Custom recursive | Manual in class |
| Python | `copy.copy()` | `copy.deepcopy()` | `__copy__()` / `__deepcopy__()` |
| C# | `MemberwiseClone()` (protected) | Serialization or manual | Implement `ICloneable` |
| Rust | `#[derive(Clone)]` | Custom cloning logic | Manual `Clone` trait impl |

### Prototype Registry

A prototype registry is a key architectural element in many Prototype pattern implementations. The registry is a lookup table (typically a `Map<String, AccountPrototype>`) that stores named prototypes for retrieval by clients.

The registry provides several benefits:

- **Centralized template management** — All product templates are registered in one place. The `BankBranch` constructor initializes the registry with the available templates. Adding a new product (e.g., `StudentLoanPrototype`) means adding one line to the registry.

- **Dynamic configuration** — The registry can be populated from external sources at startup. The bank's product management database can define templates with specific interest rates, fees, and limits. The application loads these into the registry without any hardcoded defaults in the source code.

- **Runtime updates** — In systems that support hot reloading, the registry can be updated at runtime. When the bank adjusts its prime rate, the prototype registry is updated, and all new clones automatically use the new rate. Existing clones are unaffected.

- **Lazy initialization** — Prototypes can be created on demand and cached in the registry. If a product template is expensive to construct (e.g., requires loading data from multiple database tables), the first access creates it and subsequent accesses clone from the cached instance.

### Shallow vs Deep Copy in the Banking Context

For the Banking domain prototype implementation, all fields are primitives (`f64`, `int`, `boolean`) or immutable strings (`account_holder`, `account_type`). There are no nested mutable objects. This means **shallow copy is sufficient** for all three concrete prototypes. Each clone is fully independent because there are no shared references to mutable objects.

However, if the system were extended to include nested objects such as a `FeeSchedule` with multiple fee tiers, a `BeneficiaryList` with multiple beneficiaries, or an `InterestRateSchedule` with tiered rates, deep copy would become necessary. In that case, each nested object would need its own `clone()` method, and the parent's `clone()` would call `clone()` on each child.

### Implementing clone() Correctly

A correct `clone()` implementation must satisfy three invariants:

1. **Structural equality** — `clone()` returns an object that is structurally equal to the original. All fields have the same values (or copies of the same values for mutable fields).

2. **Reference independence** — Modifying the clone does not modify the original (for deep copy) or affects only shared mutable objects (for shallow copy with documented intentional sharing).

3. **Type identity** — `clone()` returns an object of the same concrete type as the original. In Java, this is achieved through covariant return types. In Rust, `clone_box()` returns `Box<dyn AccountPrototype>` which preserves the type through dynamic dispatch.

A common mistake is implementing `clone()` by calling the constructor and passing all default values. This defeats the purpose of the Prototype pattern because the defaults are now hardcoded in two places (the prototype instance and the constructor). A correct implementation copies field values from `this` to the new instance.

## Variations of the Prototype Pattern

### Registry-Based Prototype

This is the variation implemented in the Banking example. A central registry holds named prototypes, and clients look up prototypes by name before cloning. This variation is useful when:

- There are multiple product templates that clients select by name
- Templates are loaded dynamically from external configuration
- The set of available products changes over time

### Prototype Manager (Parameterized Factory)

In this variation, the registry can register and remove prototypes dynamically at runtime. Product managers can add new promotional templates or retire old ones without restarting the application. The registry becomes a live catalog of available products that changes with market conditions.

### Self-Cloning Prototype

Some implementations embed the clone logic in a base class or mixin so that concrete prototypes inherit the cloning mechanism automatically. For example, Java's `Cloneable` interface combined with `Object.clone()` provides this for all subclasses. This reduces boilerplate but limits control over the cloning process.

### Copy Constructor

Some languages favor copy constructors over the Prototype pattern. A copy constructor takes an instance of the same class and constructs a new instance by copying its fields:

```java
public StandardSavingsPrototype(StandardSavingsPrototype other) {
    this.interestRate = other.interestRate;
    this.monthlyFee = other.monthlyFee;
    this.minBalance = other.minBalance;
    // etc.
}
```

A copy constructor achieves the same result as `clone()` but through a constructor rather than a method. The Prototype pattern's advantage over copy constructors is that you can clone through an interface without knowing the concrete type. Copy constructors require knowing the concrete class at compile time.

## Prototype Pattern vs Copy-on-Write

Copy-on-Write (COW) is a related technique that defers cloning until a write operation occurs. In a COW system, multiple clients share a reference to the same object. When a client wants to modify the object, the system clones it automatically, and the client modifies its private copy. Subsequent reads from other clients still see the original object.

The difference from the Prototype pattern is subtle but important:

- **Prototype** clones eagerly on request. The client explicitly calls `clone()` and then customizes the copy. Each clone is an independent object from the moment it is created.
- **COW** clones lazily on first modification. Clients share the same object until one of them tries to modify it, at which point the system clones it automatically.

In the Banking domain, the Prototype pattern's eager cloning is more appropriate because account objects are always personalized for a specific customer. There is no scenario where two customers share the same account object and only diverge when one makes a transaction. Each account belongs to a specific customer from the moment it is opened.

## Implementing the Prototype Pattern with Deep Copy

When your prototypes contain nested mutable objects, a shallow copy is insufficient. Consider a banking prototype that includes a `FeeSchedule` object with multiple fee tiers:

```
class FeeSchedule:
    fields:
        tiers: List of FeeTier objects
        late_payment_fee: float
        annual_maintenance_fee: float

    method clone() -> FeeSchedule:
        result = new FeeSchedule()
        result.late_payment_fee = this.late_payment_fee
        result.annual_maintenance_fee = this.annual_maintenance_fee
        result.tiers = []
        for tier in this.tiers:
            result.tiers.append(tier.clone())  // Deep copy each FeeTier
        return result
```

With deep copy, each nested object in the clone is an independent copy. Modifying the fee tiers on a cloned prototype does not affect the original or any other clone. This is essential when each clone represents an independent financial product that may be customized independently.

Deep copy can be implemented in several ways:

- **Manual recursive cloning** — Each class implements `clone()` which calls `clone()` on all its mutable fields. This is verbose but gives the developer full control over what gets copied and how.
- **Serialization-based cloning** — Serialize the object to a byte stream and deserialize it. This provides a generic deep copy mechanism but is slower and requires all objects in the graph to be serializable. Java's `ObjectOutputStream`, .NET's `BinaryFormatter` (now deprecated), and Rust's `serde` crate support this approach.
- **Generic deep copy utilities** — Python's `copy.deepcopy()` and JavaScript's `structuredClone()` provide generic deep copy for objects that meet their constraints. These are convenient but may not handle all types correctly (e.g., custom objects with non-serializable fields).

For the Banking domain's account prototypes, all fields are primitives or immutable strings, so shallow copy is sufficient. However, if the system were extended with complex nested fee structures, tiered interest rates, or beneficiary lists, deep copy would become necessary. The `clone()` method would need to be updated to recursively clone these nested objects.

## Benefits and Trade-offs

### Benefits

- **Reduced duplication** — Product defaults are defined once in the prototype objects, not repeated across every factory method and constructor call site. Changing a product's interest rate means updating one prototype object, not dozens of creation points.

- **Runtime configuration** — Product templates can be loaded from a database at startup. The bank's product management team can update interest rates, fee structures, and promotional offers without requiring a code deployment.

- **Encapsulation of creation** — Clients depend only on the abstract `AccountPrototype` interface and call `clone()`. They never need to know the concrete class of the object they are creating. This decouples product definitions from product usage.

- **Reduced subclassing** — Instead of creating a subclass for every product variation, you create one prototype per product and customize clones by setting fields. This avoids the class explosion problem that plagues naive OO designs.

- **Natural fit for prototype-based languages** — In JavaScript and Self, the Prototype pattern aligns with the language's native object model. In Rust, the `Clone` trait is a fundamental language feature that most types implement.

### Trade-offs

- **Clone correctness** — Every `clone()` implementation must be manually maintained to include all fields. Adding a new field to a prototype class requires updating its `clone()` method. This is error-prone and easily forgotten.

- **Object identity confusion** — Each clone is a distinct object. Systems that rely on reference equality or object identity checks will behave differently with clones. Testing frameworks that use reference equality for mock verification can produce false negatives.

- **Constructor bypass** — Cloning bypasses the constructor. If the constructor initializes external resources (database connections, file handles, network sockets) or performs validation, the clone misses these steps. The clone's state may be inconsistent with what the constructor guarantees.

- **Deep copy performance** — Cloning complex object graphs is expensive. In high-throughput systems where thousands of accounts are created per minute, deep copy overhead can become a bottleneck.

- **Memory overhead** — Each clone occupies its own memory. If clones are created but not garbage-collected promptly, memory usage grows linearly with the number of clones. Prototype-based systems must ensure that clones are short-lived or carefully managed.

## When Prototype Fails in Practice

Despite its elegance, the Prototype pattern can create subtle bugs in several scenarios:

- **Constructor bypass** — If the constructor performs essential validation or registration (e.g., assigning a unique account number, logging the creation event, checking regulatory limits), cloning bypasses all of that. The clone is created without running the constructor. Any initialization logic must be replicated in the `clone()` method.

- **Class hierarchy changes** — Adding a new field to a base class requires updating the `clone()` method in every subclass. If you forget to update a subclass, its `clone()` will not copy the new field. This is the same problem as serialization and is caught only through careful code review or automated testing.

- **Fragile base class** — If the base class holds private fields that the subclass cannot access, the subclass's `clone()` cannot copy those fields. This forces the base class to either expose protected accessors for its private fields or provide its own cloning logic that subclasses can invoke.

- **Performance pitfalls** — Deep cloning a large object graph is expensive. If the prototype contains a cache of reference data (e.g., a `HashMap<String, BranchInfo>` with hundreds of entries), every clone duplicates that cache. This can cause memory pressure and poor performance. Use shallow copy for read-only shared data and deep copy only for mutable data.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Factory Method** | Factory Method creates objects via a dedicated method that calls constructors. Prototype creates objects by cloning an existing instance. | Choose Factory Method when the creation logic is simple and constructors are sufficient. Choose Prototype when pre-configured templates with many default fields reduce complexity and duplication. |
| **Abstract Factory** | Abstract Factory provides an interface for creating families of related objects. Prototype clones individual objects. | Choose Abstract Factory when you need to create families of related product objects (e.g., a set of Winter Promotional products). Choose Prototype when you need to clone and customize individual templates from a shared registry. |
| **Builder** | Builder separates the construction of a complex object from its representation, using a step-by-step process. Prototype clones a fully constructed object in one step. | Choose Builder when you need fine-grained control over the construction process with optional steps. Choose Prototype when you have a pre-built template and only need to customize a few fields after cloning. |
| **Singleton** | Singleton ensures a class has exactly one instance. Prototype creates multiple independent copies of a template. | Choose Singleton when you need exactly one instance of a configuration object (e.g., a single global `BankConfiguration`). Choose Prototype when you need to create many similar objects from a shared template. |
| **Object Pool** | Object Pool reuses existing objects by returning them to a pool after use. Prototype creates new objects by cloning templates. | Choose Object Pool when object creation is expensive and objects are reusable (e.g., database connections). Choose Prototype when each object needs unique customization (e.g., customer accounts). |

The key distinction between Prototype and other creational patterns lies in when the default state is established. Factory Method and Abstract Factory encode defaults inside the factory class. Builder encodes defaults inside the director's construction recipe. Singleton encodes defaults in a single global instance. Prototype encodes defaults in a template object that can be cloned, meaning the defaults can be loaded from external configuration, composed at runtime, and changed without modifying code. This makes Prototype the most flexible creational pattern for systems where product templates change frequently and are configured by domain experts rather than developers.

## Summary

The Prototype pattern is a creational pattern that creates new objects by cloning an existing object rather than calling a constructor. It is particularly valuable in domains like banking where there are many product templates with pre-configured defaults, and each new account is a slight customization of an existing template.

The pattern's core mechanism is the `clone()` method, which duplicates a prototype object's state to a new instance. The client retrieves a prototype from a registry, clones it, and customizes the clone. The original prototype remains unchanged and ready for future cloning operations.

Key design decisions when implementing the Prototype pattern include:

- **Shallow vs deep copy** — Choose based on whether the object graph contains mutable nested objects. The Banking domain's primitive-heavy prototypes work well with shallow copy.
- **Registry management** — Decide whether the registry is hardcoded, loaded from configuration, or dynamically populated at runtime.
- **Clone method design** — Decide whether to use language-native cloning mechanisms (Java's `Cloneable`, C#'s `MemberwiseClone`, Rust's `Clone` derive) or custom implementations.
- **Constructor bypass handling** — Ensure that any initialization logic in the constructor is replicated in the `clone()` method, or redesign the system so that constructors do not perform essential side effects.

The Prototype pattern is most effective when object creation is expensive, objects have many configuration fields, and the system needs to support runtime configuration of product templates. It is less effective for simple objects, objects with circular references, or systems where object identity is critical.

In the Banking domain, the Prototype pattern provides a clean, maintainable way to manage financial product templates. Each product type is a clonable prototype with its default configuration. Bank associates clone templates and customize them for specific customers. When product terms change, the bank updates the prototype in one place, and all future clones automatically reflect the new terms. This separation of template configuration from customer-specific customization is the defining characteristic of the Prototype pattern and the reason it remains a fundamental tool in the software architect's toolkit.
