The Bridge pattern in Java uses the `interface` keyword to define the Engine contract and an `abstract class` for the Car abstraction. The `Engine` interface declares `start()`, `sound()`, and `fuelType()` methods. The `Car` abstract class holds a protected `Engine` reference and declares `drive()` as abstract. Java's compile-time type checking ensures that any object passed to a `Car` constructor satisfies the `Engine` interface. Three concrete engine classes implement `Engine`, and three car subclasses extend `Car`. The `HybridEngine` demonstrates the Decorator-like composition of implementations. Java's strong typing and interface system make the Bridge pattern's separation of hierarchies explicit and compiler-verified.

## How the Implementation Works

Java's `Engine` interface defines the three methods that every engine must implement. The `Car` abstract class accepts an `Engine` via constructor injection and stores it as a `protected final` field — the `final` keyword prevents reassignment after construction, making the engine reference immutable. Each concrete engine class (`GasEngine`, `ElectricEngine`, `HybridEngine`) implements `Engine` with its own internal logic and private fields. The `HybridEngine` takes two `Engine` parameters in its constructor and delegates conditionally based on battery level. Java's `@Override` annotations ensure that each implementation method correctly matches the interface declaration. The driver demonstrates five distinct car-engine configurations proving that both hierarchies vary independently.

### OOP Concepts Applied

- **Abstraction** — `Engine` interface declares the engine contract. `Car` abstract class declares the car contract. Neither side depends on concrete types.
- **Encapsulation** — Each engine class uses `private` fields (cylinders, batteryCapacity, motorPower) with no public getters or setters.
- **Polymorphism** — `Car.drive()` calls `engine.start()` and `engine.sound()` polymorphically. Java's virtual method dispatch routes calls to the correct concrete implementation.
- **Composition** — `Car` holds an `Engine` reference passed through the constructor. `HybridEngine` composes two `Engine` references.

### SOLID Principles Applied

- **Single Responsibility Principle** — `GasEngine` handles only combustion mechanics. `Sedan` handles only sedan driving. Each class has a single, focused reason to change.
- **Open/Closed Principle** — New `Engine` implementations (`HydrogenEngine`, `DieselEngine`) and new `Car` subclasses (`Convertible`, `Minivan`) can be added without modifying existing code.
- **Liskov Substitution Principle** — Any `Engine` implementation can be substituted into any `Car` subclass. The `Car` base class interacts only through the `Engine` interface.
- **Interface Segregation Principle** — The `Engine` interface is minimal: three methods. No engine implementation is forced to implement unused operations.
- **Dependency Inversion Principle** — `Car` depends on the `Engine` interface (abstraction), not on `GasEngine` or `ElectricEngine` (concretions).

```java
// Bridge Pattern — Java — Cars Domain
// =====================================
// Participants: Car (Abstraction), Sedan/SUV/Truck (RefinedAbstractions),
// Engine (Implementor interface), GasEngine/ElectricEngine/HybridEngine
// (ConcreteImplementors)
// OOP: abstraction, encapsulation, polymorphism, composition
// SOLID: SRP, OCP, LSP, ISP, DIP
// =====================================

// ── Participant 1: Engine (Implementor Interface) ──
// OOP: Abstraction — defines the contract all engines must satisfy
// SOLID: ISP — minimal three-method interface; DIP — Car depends on this
interface Engine {
    void start();
    String sound();
    String fuelType();
}
```

### Participant 2: GasEngine (ConcreteImplementor)

```java
// OOP: Encapsulation — internal parameters are private
// SOLID: SRP — implements only gas combustion logic
class GasEngine implements Engine {
    private final int cylinders;
    private final double displacement;

    public GasEngine(int cylinders, double displacement) {
        this.cylinders = cylinders;
        this.displacement = displacement;
    }

    @Override
    public void start() {
        System.out.println("Gas engine fires: *vroom vroom*");
    }

    @Override
    public String sound() {
        StringBuilder sb = new StringBuilder("VROOOOM");
        for (int i = 0; i < cylinders; i++) {
            sb.append('.');
        }
        return sb.toString();
    }

    @Override
    public String fuelType() {
        return String.format("Gasoline (%.1fL %d-cyl)", displacement, cylinders);
    }
}
```

### Participant 3: ElectricEngine (ConcreteImplementor)

```java
// OOP: Encapsulation — battery and motor specs are private
// SOLID: SRP — implements only electric motor logic
class ElectricEngine implements Engine {
    private final double batteryCapacity;
    private final int motorPower;

    public ElectricEngine(double batteryCapacity, int motorPower) {
        this.batteryCapacity = batteryCapacity;
        this.motorPower = motorPower;
    }

    @Override
    public void start() {
        System.out.println("Electric motor engages: *whirrr*");
    }

    @Override
    public String sound() {
        return "WHIIIIRRRR";
    }

    @Override
    public String fuelType() {
        return String.format("Electric (%.1fkWh, %dkW)", batteryCapacity, motorPower);
    }
}
```

### Participant 4: HybridEngine (ConcreteImplementor)

```java
// OOP: Composition — wraps both a gas and electric engine
// SOLID: SRP — coordinates delegation without duplicating engine logic
class HybridEngine implements Engine {
    private final Engine gasEngine;
    private final Engine electricEngine;
    private final int batteryLevel;

    public HybridEngine(Engine gasEngine, Engine electricEngine, int batteryLevel) {
        this.gasEngine = gasEngine;
        this.electricEngine = electricEngine;
        this.batteryLevel = batteryLevel;
    }

    @Override
    public void start() {
        electricEngine.start();
        if (batteryLevel < 20) {
            System.out.println("Battery low — gas engine kicks in.");
            gasEngine.start();
        } else {
            System.out.println("Running on electric power — gas engine idle.");
        }
    }

    @Override
    public String sound() {
        if (batteryLevel >= 20) {
            return electricEngine.sound();
        }
        return gasEngine.sound() + " then " + electricEngine.sound();
    }

    @Override
    public String fuelType() {
        return String.format("Hybrid (Gas + Electric, battery at %d%%)", batteryLevel);
    }
}
```

### Participant 5: Car (Abstraction)

```java
// OOP: Abstraction — defines the car interface and holds Engine reference
// SOLID: DIP — depends on Engine interface, not on concrete implementations
abstract class Car {
    protected final Engine engine;

    public Car(Engine engine) {
        this.engine = engine;
    }

    public abstract String drive();

    public String getSpecs() {
        return "Car with " + engine.fuelType();
    }
}
```

### Participant 6: Sedan (RefinedAbstraction)

```java
// OOP: Polymorphism — overrides drive() with sedan-specific behavior
// SOLID: SRP — handles only sedan driving mechanics
class Sedan extends Car {
    private final double trunkCapacity;

    public Sedan(Engine engine, double trunkCapacity) {
        super(engine);
        this.trunkCapacity = trunkCapacity;
    }

    @Override
    public String drive() {
        engine.start();
        return "Sedan cruising smoothly on the highway. " + engine.sound();
    }

    public String openTrunk() {
        return String.format("Trunk opens (%.1f cu ft).", trunkCapacity);
    }

    @Override
    public String getSpecs() {
        return String.format("Sedan with %s | Trunk: %.1f cu ft",
                engine.fuelType(), trunkCapacity);
    }
}
```

### Participant 7: SUV (RefinedAbstraction)

```java
// OOP: Polymorphism — overrides drive() with SUV-specific behavior
// SOLID: SRP — handles only SUV driving mechanics
class SUV extends Car {
    private final double groundClearance;
    private final boolean awd;

    public SUV(Engine engine, double groundClearance, boolean awd) {
        super(engine);
        this.groundClearance = groundClearance;
        this.awd = awd;
    }

    @Override
    public String drive() {
        engine.start();
        return "SUV tackling off-road terrain. " + engine.sound();
    }

    public String engage4WD() {
        if (awd) {
            return "All-wheel drive engaged. Traction optimized.";
        }
        return "This SUV does not have AWD.";
    }

    @Override
    public String getSpecs() {
        String drivetrain = awd ? "AWD" : "2WD";
        return String.format("SUV with %s | Ground: %.1f\" %s",
                engine.fuelType(), groundClearance, drivetrain);
    }
}
```

### Language-Specific Notes

- Java's `interface` keyword provides a formal contract for the `Engine` type. Any class that `implements Engine` must provide all three methods (`start`, `sound`, `fuelType`), enforced at compile time.
- The `protected final Engine engine` field in `Car` prevents reassignment after construction. The `final` keyword guarantees the engine reference never changes throughout the Car's lifetime.
- Java's `@Override` annotation is optional but recommended — it causes a compiler error if the method does not actually override an interface or superclass method, catching signature mismatches early.
- The `StringBuilder` in `GasEngine.sound()` builds the variable-length "VROOOOM...." string efficiently without string concatenation in a loop, following Java best practices.
- Java's `String.format()` provides type-safe string interpolation with format specifiers (`%.1f` for one-decimal floats, `%d` for integers, `%%` for a literal percent sign). This is the idiomatic Java approach.
- The `Main` class with `public static void main(String[] args)` is the standard Java entry point. All classes are defined in a single file for demonstration clarity; in production, each class would be in its own `.java` file.
