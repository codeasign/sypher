# Document Databases — Trade-offs & Failure Modes

Every database architecture involves trade-offs. Document databases exchange
relational rigor for developer velocity, but that exchange is not free. You
gain schema flexibility and intuitive data modeling at the cost of query
power, storage efficiency, and consistency guarantees.

This page examines when that trade-off pays off — and when it burns you.

---

## Core Tension: Schema Flexibility vs Query Power

The defining characteristic of a document database is schema flexibility. Every
document in a collection can have a different set of fields. You can add a
field to one document without altering 50 million others. This eliminates
schema migrations as a deployment blocker and accelerates feature development.

But schema flexibility has a direct cost: the database cannot optimize queries
it knows nothing about. A relational database can reason about types,
constraints, and relationships through its schema. It knows that `orders.total`
is a decimal and that `orders.customer_id` references `customers.id`. The
query planner uses this knowledge to choose optimal join strategies, index
paths, and materialization plans.

A document database sees opaque BSON blobs. Without schema information, the
query planner relies entirely on explicit indexes and the aggregation
pipeline. The database cannot infer relationships. It cannot enforce
referential integrity. It cannot validate that a field named `price` contains
a number and not a string.

The tension manifests in three concrete dimensions:

**Dimension 1: Query Expressiveness**

Relational databases support declarative queries (SQL) that express complex
relationships, aggregations, and transformations in a single statement.
Document databases use a programmatic pipeline (MongoDB Aggregation Pipeline)
that requires you to think in stages rather than sets. Simple queries are
easier in document databases. Complex analytical queries are harder.

**Dimension 2: Data Integrity**

A relational schema enforces types, nullability, uniqueness, and referential
integrity at the database level. Document databases enforce nothing by
default. Your application code is the schema. When a bug in one service
writes a string where a number is expected, every downstream query that
expects a number either crashes or silently returns wrong results.

**Dimension 3: Performance Predictability**

Relational databases give predictable performance because the optimizer can
choose the best execution plan from known statistics. Document databases
are predictable only when queries are covered by well-designed indexes. A
missing index on a collection query triggers a full collection scan (COLLSCAN)
that can take seconds or minutes on large collections.

## Decision Matrix: When to Choose Document Databases

The following matrix maps common application requirements to the appropriate
database choice. Each row compares how document (MongoDB), relational
(PostgreSQL), and specialized databases handle the requirement.

| Requirement | Document DB (MongoDB) | Relational (PostgreSQL) | Specialized |
|---|---|---|---|
| Rapid schema evolution | Excellent: no migrations needed | Moderate: ALTER TABLE with downtime risk | N/A |
| Complex JOINs across 5+ entities | Poor: $lookup is slow, max 1-2 stages | Excellent: query planner optimizes | N/A |
| Geospatial queries | Good: 2dsphere index | Good: PostGIS | Excellent: Elasticsearch |
| Full-text search | Moderate: text index | Moderate: tsvector | Excellent: Elasticsearch, Meilisearch |
| ACID across entities | Poor: multi-doc txns are 2-5x slower | Excellent: native ACID | N/A |
| Time-series data | Poor: document bloat | Moderate: partitioning | Excellent: InfluxDB, TimescaleDB |
| Graph traversals | Very poor: app-level recursion | Poor: recursive CTEs | Excellent: Neo4j, Dgraph |
| Key-value lookups | Excellent: indexed _id queries | Excellent: primary key lookup | Excellent: Redis, DynamoDB |
| Document size > 16 MB | Impossible: hard limit | Possible: TOAST, LOB | N/A |
| Unstructured/semi-structured data | Excellent: no schema required | Moderate: JSONB column | Excellent: Elasticsearch |
| Multi-region writes | Moderate: sharding + cross-region | Poor: complex replication | Excellent: DynamoDB Global Tables |
| Ad-hoc analytical queries | Poor: aggregation pipeline limited | Excellent: SQL + window functions | Excellent: Snowflake, BigQuery |
| Strong consistency required | Configurable: w:majority | Yes: SERIALIZABLE | Varies |
| Eventual consistency acceptable | Configurable: w:1 | N/A | Excellent: Cassandra, DynamoDB |

The decision matrix tells a clear story: document databases excel in the
upper-left quadrant (rapid development, simple queries, semi-structured
data). They struggle in the lower-right quadrant (complex queries, strict
consistency, analytical workloads).

## CAP Theorem Positioning

Document databases (MongoDB) are **CP** by default: they prefer consistency
over availability during a network partition. When a partition splits a
replica set, the majority side elects a primary and continues serving. The
minority side's primary detects it has lost the majority, steps down, and
stops accepting writes. The minority side becomes unavailable for writes
until the partition heals.

This is the correct choice for most applications. A system that accepts
writes on both sides of a partition would produce conflicting data that must
be merged later — a complexity most teams cannot afford.

### Configurable Consistency

MongoDB allows you to adjust consistency and durability per operation:

| Write Concern | Guarantee | Performance Impact |
|---|---|---|
| `w: 1` | Acknowledged by primary only | Fastest writes |
| `w: majority` | Acknowledged by majority of replica set | ~2x slower than w:1 |
| `w: "majority"` + `j: true` | Majority ack + journal commit | ~3x slower than w:1 |
| `w: 0` | Fire-and-forget, no ack | Fastest, data loss risk |

| Read Preference | Consistency | Use Case |
|---|---|---|
| `primary` | Strong consistency (linearizable) | Default, safest |
| `primaryPreferred` | Strong (usually) | Failover tolerance |
| `secondary` | Eventual consistency | Read-heavy dashboards |
| `secondaryPreferred` | Eventual (usually) | Lower latency reads |
| `nearest` | Eventual | Geo-distributed low latency |

The key insight: **consistency is not binary**. You can choose different
consistency levels for different operations in the same application. Payment
writes use `w: majority` with `primary` read preference. Analytics reads
use `w: 1` with `secondary` read preference.

## Failure Modes

Document databases fail in characteristic ways. Understanding these failure
modes is essential to designing robust systems. Each mode has a specific
signature, cause, and remedy.

### Failure 1: Hot Spot on Shard Key

You shard your collection by `{userId: 1}`. Most reads and writes use a
single user at a time. This seems reasonable. But your application has a
"celebrity user" with 1 million followers. Every follower posts a comment on
that user's profile. All those writes go to the same shard because the shard
key is `userId` and the target user's ID maps to shard 2. Shard 2 handles
100x the throughput of shards 1 and 3. It saturates its disk I/O. Writes
queue up. Latency for every operation on shard 2 spikes to seconds.

**The failure signature**: Overall cluster throughput appears sufficient, but
one shard's CPU and disk IO are pinned at 100% while others sit at 15%. Latency
is bimodal — most requests are fast, but requests hitting the hot shard take
10-30 seconds. The problem worsens as the celebrity user gains followers.

**Root cause**: The shard key has low cardinality or high frequency for a
subset of values. Range-based sharding on a key with a skewed distribution
guarantees hot spots.

**Remedies**:
- Use a hashed shard key to distribute writes uniformly.
- Use a compound shard key where the high-cardinality field comes first
  (e.g., `{userId: 1, timestamp: 1}`).
- Implement a fan-out pattern in the application: shard by a computed field
  like `{userId_mod_shardCount: 1}`.
- For known celebrity users, add a synthetic suffix to the shard key value
  (e.g., `userId + "_" + randomSuffix`) and aggregate reads in the application.

### Failure 2: Growing Document Bloat

You embed user comments inside the post document. For the first week, each
post has 5-10 comments. Reads are fast because the post document is under
10KB. Your application takes off. Popular posts accumulate 1,000 comments.
The post document grows to 500KB. Every read fetches 500KB even if the user
only wants the post title. Cache efficiency collapses — the cache now holds
one-fifth the number of documents. Writes to any field in the document
require rewriting the entire 500KB B-tree page. The `$push` to add one comment
causes a full page rewrite.

**The failure signature**: Reads slow down gradually over weeks. You add
more cache nodes, but the cache hit ratio keeps dropping. At 1MB per document,
a 100K-post collection needs 100GB of cache for the hot set alone.

**Root cause**: Embedding unbounded arrays in documents. The document size
grows linearly with the array size, but read/write cost grows super-linearly
because the entire document is loaded and rewritten for every operation.

**Remedies**:
- Move unbounded arrays (comments, notifications, logs) to separate
  collections with references.
- Use a TTL index to auto-purge old data.
- Paginate embedded arrays using `$slice` in projection to avoid fetching
  the entire array.
- Set a maximum array size in application logic and archive older entries.

### Failure 3: Missing Index on Read-Heavy Query

Your application has a frequently-used dashboard: "show me all orders placed
in the last 30 days sorted by total descending." There is no index on `date`
or `total`. MongoDB scans the entire orders collection (COLLSCAN). At 10M
documents, each scan reads 10GB of data. The query takes 30 seconds. Your
connection pool fills with blocked queries. New queries wait for available
connections. The application becomes unresponsive.

**The failure signature**: Sporadic timeouts that correlate with dashboard
usage. Queries that were fast in development (5K documents) are catastrophic
in production (10M documents). CPU spikes during collection scans.

**Root cause**: A query that touches every document in the collection. In
development with small datasets, the scan completes in milliseconds. In
production, it takes seconds or minutes.

**Remedies**:
- Add a compound index covering the query filter and sort:
  `{date: -1, total: -1}`.
- Use `explain("executionStats")` to verify index usage before deploying.
- Monitor `db.currentOp()` for COLLSCAN operations in production.
- Set up the MongoDB Profiler to log all queries slower than 100ms and
  review them weekly.

### Failure 4: Oplog Too Small for Large Migration

You run a schema migration that updates every document in a 500GB collection.
The update changes one field on each of 50M documents. Each update creates an
oplog entry. The oplog is the default 16GB. After updating 10M documents, the
oplog wraps — old entries are overwritten. The secondaries, which are 5
minutes behind, cannot find their next oplog entry. They enter RECOVERING
state. They require a full resync (500GB of data over the network).

**The failure signature**: Secondaries suddenly enter RECOVERING state during
a maintenance window. The cluster loses redundancy. If the primary fails
during the resync, data is unavailable.

**Root cause**: The oplog is a capped collection that retains operations until
it reaches its maximum size. When the oplog is too small, secondaries fall
out of sync during large write operations because the oplog entries they need
have been overwritten.

**Remedies**:
- Set `oplogSizeMB` to at least 5% of total data size, or larger
  for collections that undergo bulk updates. For a 500GB collection, set
  oplog to 50GB.
- Monitor the oplog window daily with `rs.printReplicationInfo()`.
- Run large migrations in batches with `db.collection.bulkWrite()` and
  a delay between batches.
- Schedule migrations during low-traffic periods.

### Failure 5: Write Concern Misconfiguration

Your application is configured with `writeConcern: {w: 1}`. This is the
fastest write concern — the primary acknowledges the write after journaling
it. The primary suffers a catastrophic hardware failure 2ms after acknowledging
a write. The write was journaled on the primary, but the oplog entry was not
yet replicated to any secondary. The replica set elects a new primary. The
acknowledged write is lost. To your application, the write succeeded. To the
database, it never happened.

This is not a bug. It is the semantics of `w: 1`. You traded absolute
durability for write speed. For logs or analytics data, this is acceptable.
For payment transactions, it is not.

**The failure signature**: A primary failure causes data loss that appears as
"missing records" after failover. Application logs show successful writes
that no longer exist in the database. The problem is intermittent and
uncorrelated with any application errors.

**Root cause**: The application chose durability over speed at the wrong
level of granularity. Critical writes used the same write concern as
non-critical writes.

**Remedies**:
- Use `w: majority` for critical data (payments, user accounts, orders).
- Use `w: 1` for non-critical data (logs, analytics events, cache).
- Set a default write concern at the cluster level and override per-operation.
- Test failover scenarios in a staging environment to verify durability.

### Failure 6: Connection Pool Exhaustion Under Load

Your application serves traffic across 100 server instances. Each instance
opens a MongoDB connection pool of 10 connections. That is 1,000 connections
total. MongoDB's default max incoming connections is roughly 60-80% of the
ulimit (`ulimit -n`). On a typical server with a ulimit of 65,536, MongoDB
accepts about 50,000 connections. But each connection consumes 1MB of RAM.
At peak traffic, 50,000 connections consume 50GB of RAM just for connection
overhead. The WiredTiger cache (default 50% of RAM minus 1GB) shrinks. Cache
hit ratio drops. Read latency spikes.

**The failure signature**: Under high traffic, the database reports high
connection counts and memory pressure. Cache hit ratio drops from 95% to 50%.
Read latency doubles every 10 minutes. Connection storms during restarts
compound the problem.

**Root cause**: Too many concurrent connections overwhelm the database server's
memory. Each connection consumes RAM for stack, buffers, and I/O contexts,
reducing the memory available for the working set.

**Remedies**:
- Limit the connection pool size per application instance (10-50 connections
  per instance).
- Use a proxy or connection pooler (like MongoDB Atlas' connection pooling)
  to multiplex connections.
- Implement connection pooling with exponential backoff retry.
- Monitor connection counts with alerts at 80% of the maximum.

## Anti-Patterns Hall of Shame

These anti-patterns appear in production systems with depressing regularity.
Each one looks reasonable at first glance and causes measurable pain at scale.

### Anti-Pattern 1: One Document to Rule Them All

Storing an entire user's lifecycle in a single document, including every
order, every click, every email sent. The document grows to megabytes. Every
read is slow. Every write rewrites large portions. The solution is to use
references for unbounded or loosely related data.

**Why it looks appealing**: "With document databases, everything related to
a user should be in one place." This misunderstands the document model.
Documents are optimal for data that is accessed together and has a 1:1 or
1:few relationship. They are suboptimal for 1:many or many:many relationships.

**Fix**: Split into separate collections with references. Embed only when
the data is accessed together and stays small.

### Anti-Pattern 2: Schema on Read Without Documentation

You take advantage of schema flexibility. Every developer adds fields as
needed, without documentation or validation. After two years, the same field
name (`status`) exists in 6 different spellings in different documents
(`"active"`, `"Active"`, `"ACTIVE"`, `1`, `true`). Queries that filter on
status miss 80% of documents. Fixing this requires a migration across 50M
documents.

**Why it looks appealing**: "Schema flexibility means I never have to think
about schemas." Actually, every application has an implicit schema. Making it
explicit prevents data rot.

**Fix**: Use MongoDB schema validation (`validator` in `createCollection`)
to enforce field types and required fields. Document the schema in a shared
spec. Add integration tests that verify documents match the expected schema.

### Anti-Pattern 3: The Super-Relational Document Schema

You model every relationship as a reference, exactly as you would in a
relational database. A user document has `addressId`, `companyId`, `teamIds`,
`projectIds`. Every page load triggers 5-10 `$lookup` stages or application-
side JOINs. The document database becomes a slow relational database.

**Why it looks appealing**: "References are best practice because they avoid
data duplication." This is true for relational databases. For document
databases, duplication is a feature, not a bug.

**Fix**: If you need this many JOINs, use a relational database. The document
model shines when related data is embedded and accessed together. For the
10-20% of relationships that truly need references, accept the read cost and
optimize with indexes on the referenced fields.

### Anti-Pattern 4: Sharding Without a Good Shard Key

You shard on `{_id: 1}` because it is the default. New documents have
monotonically increasing `_id` values (ObjectIds). All new writes go to a
single shard — the one holding the highest key range. Your write throughput
is capped at one shard's capacity. This is a **jumbo chunk** problem.

**Why it looks appealing**: "_id is always unique, it has perfect cardinality."
Cardinality is only one criterion. A good shard key also needs random
distribution across shards.

**Fix**: Use a hashed shard key (`{_id: "hashed"}`) or a compound shard key
where the first field has high cardinality and random distribution (e.g.,
`{userId: 1, createdAt: 1}` with hashing).

### Anti-Pattern 5: No Index for Sort Operations

You query `db.posts.find({author: "Al"}).sort({createdAt: -1})`. There is an
index on `author` but not a compound index on `{author: 1, createdAt: -1}`.
MongoDB uses the `author` index to find matching documents, then performs an
in-memory sort. If the result set has 50,000 documents, the sort requires
sorting all of them in memory. The operation exceeds the 32MB sort memory
limit. MongoDB either spills to disk (slow) or throws an error.

The compound index `{author: 1, createdAt: -1}` gives MongoDB the sort order
for free — documents come out of the index already sorted.

**Why it looks appealing**: "I have an index on the filter field. That should
be enough." MongoDB can use the index for filtering but not for sorting
unless the index includes the sort field in the correct order.

**Fix**: Always create compound indexes that cover both filter and sort.
Use the ESR (Equality-Sort-Range) rule for compound index field order:
equality fields first, then sort fields, then range fields.

### Anti-Pattern 6: Ignoring the 16MB Document Limit

You store binary image data (1MB profile photos) directly in user documents.
After 15 photos, the document approaches 16MB. A write to any field requires
reading and rewriting the full document. Backups take longer. The solution is
GridFS for large binary data.

**Why it looks appealing**: "I can access everything in one query." But a
single query that returns a 15MB document blocks the connection for the
duration of the transfer, reducing throughput for all other operations.

**Fix**: Use GridFS for files over 1MB. GridFS splits binary data into 255KB
chunks stored in separate collection. Access the file metadata with a
reference in the user document.

### Anti-Pattern 7: Nested Arrays in Documents

You store product categories as nested arrays: `{category: ["Electronics", ["Computers", ["Laptops"]]]}`. Querying for all products in "Laptops" requires
recursive array scanning. MongoDB's `$elemMatch` and `$` positional operator
only work one level deep. Updating a deeply nested element requires positional
operators with complex arrayFilters.

**Fix**: Flatten hierarchical data. Store categories as a flat array of
path strings: `{categories: ["Electronics", "Electronics/Computers", "Electronics/Computers/Laptops"]}`. Use regex queries (`{categories: /^Electronics/}`)
for hierarchical matching.

### Anti-Pattern 8: Using MongoDB as a Message Queue

You insert documents into an `events` collection and have workers poll for
new events using `find()` with `sort({createdAt: 1})`. Workers process the
event and delete it. This pattern has multiple problems: polling creates read
load even when no events exist; workers compete for the same events; deleted
documents create fragmentation in the collection; and the pattern has no
guarantees about at-least-once delivery.

**Fix**: Use a dedicated message queue (RabbitMQ, Kafka, Amazon SQS). If you
must use MongoDB, use a tailable cursor on a capped collection, which avoids
polling overhead. But even this lacks the delivery guarantees of a proper
queue.

## When NOT to Use This

Document databases are not the right choice when:

### Complex Multi-Entity Transactions

If your application requires atomic updates across 5 different entities
(banking ledger, inventory, user balance, audit log, notification), the
relational model with ACID transactions is simpler and more reliable. MongoDB
supports multi-document transactions, but they come with a 2x-5x performance
penalty compared to single-document operations. The transaction coordinator
adds latency and complexity.

**Better choice**: PostgreSQL, MySQL (with InnoDB), or Oracle for strict
ACID compliance across many tables.

### Ad-Hoc Analytical Workloads

If analysts run arbitrary SQL queries with multiple JOINs, GROUP BYs, and
subqueries daily, a document database's aggregation pipeline will feel
restrictive. You cannot `JOIN` two collections in an ad-hoc way. The
`$lookup` stage is slow and has a 16MB memory limit by default (you can
increase it with `allowDiskUse: true`, but this is slow).

**Better choice**: Columnar databases (Snowflake, Redshift, BigQuery) for
analytics. PostgreSQL for moderate analytics needs.

### Strict Referential Integrity

Document databases do not enforce foreign key constraints. If you delete a
document, its references in other documents become dangling pointers. Your
application must handle this. For financial systems, EHR (electronic health
records), or any system where referential integrity is legally mandated, the
database should enforce it.

**Better choice**: PostgreSQL or Oracle with declarative foreign key
constraints and CASCADE/NO ACTION behavior.

### Deeply Interconnected Data

Social networks, recommendation engines, and supply chain systems that
traverse relationships across many hops are better served by graph databases.
In a document database, a friend-of-friend query requires multiple round-trips
or a recursive `$lookup`. This is slow, complex, and does not scale.

**Better choice**: Neo4j (property graph), Dgraph (RDF graph), or Amazon
Neptune.

### No Tolerance for Data Duplication

The document model intentionally duplicates data for performance. A user's
address is copied into every order document so that order reads are fast.
If your storage budget is tight and every byte counts, a normalized relational
database will use less disk space. More importantly, data duplication
increases the risk of inconsistency — when a user updates their address,
every document that embeds the old address becomes stale.

**Better choice**: PostgreSQL with normalized schemas. Use materialized views
if denormalization is needed for performance.

### Time-Series Data at Scale

Storing IoT sensor readings, server metrics, or financial tick data in a
document database is possible but inefficient. Each document has overhead
(BSON keys, `_id` field, metadata). For billions of data points, the storage
overhead is significant. Time-based queries (range scans over time) are not
optimized in document databases.

**Better choice**: InfluxDB, TimescaleDB, or ClickHouse for time-series
workloads. MongoDB's Time Series Collection (MongoDB 5.0+) reduces some of
these issues but still lags dedicated solutions.

### High-Volume Append-Only Logs

Writing application logs to a collection generates enormous write load and
causes indexing overhead on the `_id` field. The collection grows unbounded.
Write throughput is limited by the primary node's capacity.

**Better choice**: Elasticsearch (for searchable logs), Kafka (for streaming
logs), or dedicated log aggregation systems (Datadog, Splunk).

### Multi-Row Joins as a Core Operation

If your application cannot function without joining 3-5 tables on every page
load, a document database will fight you every step of the way. The
`$lookup` pipeline stage is a left outer join only. There is no `INNER JOIN`,
no `RIGHT JOIN`, no `FULL OUTER JOIN`. You must restructure your data model
to avoid joins entirely.

**Better choice**: PostgreSQL, MySQL, or any relational database where JOINs
are a first-class operation.

## Summary: The Balanced View

Document databases are not strictly better or worse than relational databases.
They make different trade-offs:

| Trade | You Get |
|---|---|
| No enforced schema | Faster development, no migrations |
| No referential integrity | Lower complexity, no CASCADE overhead |
| Data duplication | Fast reads, fewer JOINs |
| Limited query expressiveness | Simple API, no SQL learning curve |
| Eventual consistency (configurable) | Higher throughput, lower latency |

The best teams use both. A typical modern stack might store user profiles
and product catalogs in MongoDB (semi-structured, read-heavy, evolving schema)
and store financial transactions in PostgreSQL (strict ACID, referential
integrity, complex queries).

The key to success with document databases is knowing when to say no. If your
data has more relationships than aggregates, if every query touches 5+
collections, or if you need to run analytical queries across the entire
dataset — stop and reach for a relational database instead.

Document databases win when you model your data as **self-contained aggregates**
that are written and read as atomic units. Every operation that violates this
pattern — joins across collections, deep nesting, unbounded arrays — is a
signal that you are fighting the document model rather than using it.
