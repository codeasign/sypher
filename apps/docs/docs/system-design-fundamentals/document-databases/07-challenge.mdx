# Document Databases -- Challenges

## Tier 1 Foundation

### Challenge 1: The Social Media Post Schema (5 min)

A social media app stores posts. Each post has content (text, up to 500
characters), an author (user ID), a timestamp, and a list of likes (user IDs
who liked the post). Posts can have up to 100 comments. Each comment has the
commenter's user ID, text, and timestamp.

Design a document schema for this system. Decide what to embed and what to
reference. Justify each decision.

**Model Answer:**
I would design two collections:

```
posts collection:
{
  _id: ObjectId,
  authorId: ObjectId,
  content: string,
  createdAt: Date,
  likes: [ObjectId],         // embedded array of user IDs who liked
  likeCount: number,          // denormalized for fast display
  comments: [
    {
      commentId: ObjectId,
      userId: ObjectId,
      text: string,
      createdAt: Date
    }
  ],
  commentCount: number
}

users collection (referenced by authorId):
{
  _id: ObjectId,
  username: string,
  avatarUrl: string,
  displayName: string
}
```

**Rationale**: Comments are embedded because they are always read with the
post and bounded at 100 per post -- well under the 16MB limit. Likes are an
array of user IDs (not full user documents) to keep the document small. User
details are referenced because the same user appears in many posts and
storing the full user profile in each post would duplicate data across
millions of documents. `likeCount` and `commentCount` are denormalized to
avoid counting every time the feed is loaded.

**Indexes**: `{authorId: 1, createdAt: -1}` for the user's profile page,
`{createdAt: -1}` for the global feed.

### Challenge 2: Debug the Slow Query (5 min)

Your application has this query that is taking 15 seconds on a 5M document
collection:

```
db.orders.find({status: "pending", total: {$gt: 100}})
  .sort({createdAt: -1})
  .limit(50)
```

There is an index on `{status: 1}`. What is wrong and how do you fix it?

**Model Answer:**
The query uses the `status` index to find pending orders (which may be 1M
documents -- 20% of the collection). For each matching document, it fetches
the full document from disk, checks the `total > 100` filter, and collects
results. After collecting all matching documents, it sorts them all in
memory because the sort field `createdAt` is not in the index.

The fix is a compound index that covers all three query fields in ESR order:
`{status: 1, createdAt: -1, total: 1}`. The index first filters by
`status: "pending"` (equality), then traverses in `createdAt` descending
order (sort), and the `total` field satisfies the range condition. MongoDB
can stop scanning after finding 50 matching documents because the sort order
is already satisfied by the index.

This reduces the query from a 1M-document scan with an in-memory sort to an
index scan of approximately 50-100 documents. Expected time: under 5ms.

### Challenge 3: Choose a Shard Key (5 min)

You are designing a sharded MongoDB cluster for a global user database with
500M user documents. Each document is 2KB. The primary access pattern is
lookup by `email` (login) and occasional range scans by `createdAt` (admin
reports). What shard key do you choose?

**Model Answer:**
I would use a hashed shard key on `email`. Email addresses are unique (high
cardinality) and randomly distributed across users -- no hot spots. Hashing
ensures even distribution across all shards regardless of email domain.

The trade-off: range queries on `email` are not supported. For the admin
reports that need range scans by `createdAt`, I would create a separate
collection (not sharded or sharded differently) that stores user summaries
for analytics. Or I would use the `$sample` aggregation stage for statistical
queries that do not need exact ranges.

I would start with 8 shards, each a 3-node replica set. Total raw storage
with replication: 500M x 2KB x 3 replicas = 3TB, comfortably distributed
across 8 shards (375GB per shard).

## Tier 2 Design

### Challenge 1: E-Commerce Product Catalog (15 min)

An e-commerce platform needs a product catalog with these requirements:

- 10 million products across 5,000 categories
- Each product has: name, description, price, inventory count, 3-10 variant
  options (size, color, material), 10-50 seller offers
- Primary queries: browse by category (sorted by price or popularity),
  search by name, view product detail page (includes all variants + top 5
  reviews)
- Moderate write volume: 100 product updates/second, 1,000 new products/hour
- Reviews: 50M reviews across all products, growing by 1M/week

Design the document schema and the access strategy.

**Model Answer:**

```
products collection (sharded by {categoryId: 1, hashed vendorId: 1}):
{
  _id: ObjectId,
  name: string,
  description: string,
  categoryId: ObjectId,
  vendorId: ObjectId,
  price: number,
  createdAt: Date,
  sku: string,
  variants: [
    { variantId, name: "Size", value: "M", priceDelta: 0, inventory: 50 }
  ],
  topReviews: [                // embedded, top 5 by helpful votes
    { reviewId, userId, rating, text, helpfulCount }
  ],
  reviewCount: number,
  averageRating: number,
  popularityScore: number
}

offers collection (referenced, separate collection):
{
  _id: ObjectId,
  productId: ObjectId (indexed),
  sellerId: ObjectId,
  price: number,
  condition: "new"|"used",
  shippingTime: number,
  inventory: number
}

reviews collection (referenced, separate collection):
{
  _id: ObjectId,
  productId: ObjectId (indexed, shard key),
  userId: ObjectId,
  rating: number,
  text: string,
  helpfulVotes: number,
  createdAt: Date
}
```

**Design rationale**: Variants are embedded because they are part of the
product definition and bounded. Offers are in a separate collection because
a product can have 10-50 offers which change independently and would bloat
the product document beyond reasonable size. The top 5 reviews are embedded
to avoid a separate query on the product detail page. Full reviews are
paginated from the reviews collection.

**Shard key**: Compound `{categoryId: 1, hashed vendorId: 1}`. Category
enables range scans for browse-by-category queries. Vendor hash distributes
writes within a category evenly. Products in popular categories spread
across shards.

**Indexes**: `{name: "text", description: "text"}` for search,
`{categoryId: 1, popularityScore: -1}` for browse, `{categoryId: 1, price: 1}`
for price sorting.

### Challenge 2: Real-Time Chat Application (15 min)

Design a document database schema for a WhatsApp-like chat application with:

- 1 billion users, 1 trillion messages stored
- Each conversation has 2-500 participants
- Messages are written once, read for 30 days, then archived
- You need: send message, read conversation history (last 200 messages),
  search within a conversation, see unread count per conversation
- Messages must be delivered globally with under 100ms p99 for writes

**Model Answer:**

```
conversations collection (referenced by messages, stores metadata):
{
  _id: ObjectId,
  type: "dm" | "group",
  participantIds: [ObjectId],  // indexed
  lastMessage: {
    content: string,
    senderId: ObjectId,
    createdAt: Date
  },
  messageCount: number,
  createdAt: Date
}

messages collection (sharded by {conversationId: 1, hashed messageId: 1}):
{
  _id: ObjectId,
  conversationId: ObjectId,
  senderId: ObjectId,
  content: string,
  contentType: "text" | "image" | "video",
  mediaUrl: string | null,
  createdAt: Date
}

userConversations collection:
{
  _id: ObjectId (userId + conversationId compound),
  userId: ObjectId,
  conversationId: ObjectId,
  lastReadMessageId: ObjectId,
  unreadCount: number,
  isMuted: boolean
}
```

**Sharding strategy**: Messages are sharded by `conversationId` (hashed).
All messages in a conversation land on the same shard, enabling the "last 200
messages" query to hit one shard. The hashed shard key distributes
conversations evenly.

**Read path for conversation history**: Query `messages` collection with
`{conversationId: X, createdAt: {$gte: 30DaysAgo}}`, sorted by `createdAt: -1`,
limited to 200. Uses compound index `{conversationId: 1, createdAt: -1}`.

**Write path**: Insert message document. Update `conversations.lastMessage`
with atomic `$set`. Update `userConversations.unreadCount` for all
participants except the sender. Each operation is a single-document atomic
write -- no multi-document transactions needed.

**Archival**: After 30 days, messages are moved to cold storage (S3 with
Parquet files). The message collection is partitioned by date for efficient
archival using TTL indexes on `createdAt` field.

### Challenge 3: Time-Series IoT Data (15 min)

A fleet of 100,000 IoT sensors publishes a reading every 5 seconds. Each
reading includes: sensorId, timestamp, temperature, humidity, pressure, and
battery level. Queries are: latest reading per sensor, hourly aggregates
(average temperature, max pressure), and history for the last 7 days.

Design a document database schema that handles this efficiently.

**Model Answer:**

```
sensors collection (metadata, rarely updated):
{
  _id: ObjectId,
  sensorId: string (unique),
  location: { lat, lng },
  model: string,
  installedAt: Date,
  status: "active" | "offline"
}

readings collection (time-series data, sharded strategy):
{
  _id: ObjectId,
  sensorId: string,               // part of shard key
  hour: ISODate,                   // truncated to hour boundary
  readings: [
    { ts: ISODate, temp: 22.5, humidity: 60, pressure: 1013, battery: 4.2 }
  ],
  aggregate: {
    avgTemp: number,
    maxTemp: number,
    minTemp: number,
    avgHumidity: number,
    avgPressure: number,
    readingCount: number
  }
}
```

**Design rationale**: Instead of one document per reading (which would create
17 billion documents per year), readings are bucketed by
`sensorId + hour`. Each document holds up to 720 readings (5-second
intervals for 1 hour = 12 per minute x 60 minutes). This reduces document
count from 17 billion to 24 million (100K sensors x 365 days x 24 hours).

**Shard key**: `{sensorId: 1}` (hashed or ranged). Each sensor's data lands
on one shard, making "latest reading per sensor" a single-shard query.

**Write**:
```javascript
db.readings.updateOne(
  {sensorId: X, hour: currentHour},
  {$push: {readings: {ts, temp, humidity, pressure, battery}},
   $inc: {"aggregate.readingCount": 1}},
  {upsert: true}
)
```
Use `upsert: true` to create the hour bucket if it does not exist.

**Aggregation pipeline for latest reading**:
```
db.readings.aggregate([
  {$match: {sensorId: X}},
  {$sort: {hour: -1}},
  {$limit: 1},
  {$project: {latestReading: {$last: "$readings"}}}
])
```

**Aggregation for hourly average**:
Use the pre-computed `aggregate` field from the bucket document -- no need
to scan individual readings for average queries. The aggregate is maintained
incrementally.

**Retention**: A TTL index on `hour` with `expireAfterSeconds: 604800` (7
days) automatically deletes old buckets.

## Tier 3 Expert

### Challenge 1: Design a Multi-Tenant Document Database Platform (30 min)

Design a multi-tenant document database service that runs on Kubernetes and
serves 10,000 tenants. Each tenant stores 10GB-1TB of document data. Tenants
have varying query patterns: some are read-heavy (90% reads), some are write-
heavy (70% writes). The platform must guarantee that one tenant's traffic
spike does not affect other tenants. Design the isolation model, sharding
strategy, resource allocation, and data placement.

**Discussion:**

**Isolation model**: Each tenant gets a dedicated MongoDB database within a
shared cluster, or a dedicated shard for large tenants. The platform uses
three tiers:

- **Small tenants (under 50GB)**: Share a single replica set. 50-100 tenants
  per replica set. Each tenant has a dedicated database. Resource limits are
  enforced by MongoDB's `enableFlowControl` and WiredTiger's cache pressure
  per database -- tenants that exceed their allocated IOPS are throttled.

- **Medium tenants (50-500GB)**: Dedicated replica set. 3-5 medium tenants
  per replica set, each with dedicated database. Kubernetes CPU/memory limits
  on the mongod container prevent one tenant from starving another.

- **Large tenants (over 500GB)**: Dedicated sharded cluster. The tenant gets
  their own mongos, config servers, and 3-16 shards. Full hardware isolation.

**Data placement strategy**: A placement scheduler assigns tenants to replica
sets based on resource usage. The scheduler considers: current CPU/memory
utilization of each replica set, IOPS capacity, storage remaining, and tenant
tenancy tier. When a replica set reaches 80% utilization, new tenants are
placed on other replica sets. The balancer can move tenants between replica
sets during maintenance windows.

**Sharding for large tenants**: The platform recommends a shard key to each
tenant based on their access patterns. For write-heavy tenants, a hashed
shard key on the most selective field. For read-heavy tenants, a ranged shard
key aligned with common query patterns. The platform monitors scatter-gather
ratio and alerts tenants whose shard key causes cross-shard queries.

**Operational challenges**: Backup is per-tenant (mongodump with database
filter). Restore is self-service via an API. The platform runs continuous
backup with oplog tailing for point-in-time recovery. Tenant data is
encrypted at rest with per-tenant encryption keys.

### Challenge 2: Migrate from SQL to Documents at Google Scale (30 min)

Google is migrating a legacy relational system (Spanner) to a document-based
storage layer for a new product. The system stores user profiles -- 2 billion
users, each with 50+ fields (name, email, preferences, address history,
payment methods, device information, security settings). The relational schema
has 12 tables joined by user_id. The document version should reduce read
latency by 10x by storing each user as a single document. However, security
settings and payment methods need field-level access control -- different
services can read different fields.

Design the document schema, the migration strategy, the access control model,
and the query path for a system that serves 500,000 reads/second and 50,000
writes/second.

**Discussion:**

**Document schema**: Each user is a single document with all 12 tables'
data merged. The document is organized into field groups that correspond to
access control scopes:

```
{
  _id: userId,
  schemaVersion: 2,
  public: { displayName, avatarUrl, bio },
  account: { email, phone, emailVerified, phoneVerified },
  preferences: { language, theme, notifications, privacy },
  security: { passwordHash, mfaEnabled, recoveryCodes, lastLoginIp },
  payment: { savedMethods: [{type, last4, expiry, billingAddress}], defaultMethod },
  devices: [{deviceId, platform, lastUsed, pushToken}],
  addressHistory: [{id, label, street, city, zip, country, isDefault}],
  metadata: { createdAt, updatedAt, lastActiveAt, status }
}
```

**Access control**: A middleware layer intercepts each query and rewrites it
based on the caller's service identity. The profile read service sees the
full document. The search indexing service sees only `public` fields. The
payment service sees `payment` and `account` fields. The rewrite uses
MongoDB's aggregation `$project` stage to include only allowed fields.

```
// Payment service sees only payment-related fields:
db.users.aggregate([
  {$match: {_id: userId}},
  {$project: {"account.email": 1, "payment": 1, "preferences.currency": 1}}
])
```

**Migration strategy (zero-downtime, 6-month timeline)**:
1. Phase 1 (months 1-2): Build dual-write. Every write to Spanner is
   mirrored to MongoDB. Read from Spanner.
2. Phase 2 (months 3-4): Backfill 2 billion users from Spanner to MongoDB
   using a batch process (100K users per batch, parallelized across 1,000
   workers). Estimated time: 20,000 batches x 2 seconds/batch = 11 hours.
   Run at low traffic hours.
3. Phase 3 (months 5): Shadow reads. Route 1% of read traffic to MongoDB.
   Compare results with Spanner. Fix discrepancies. Ramp to 10%, 50%, 100%
   over 4 weeks.
4. Phase 4 (month 6): Cut over writes. Make MongoDB the primary. Write to
   Spanner as a fallback for 30 days. After 30 days with zero issues, remove
   Spanner fallback.

**Read path**: A read-through cache layer sits in front of MongoDB. Hot user
profiles are cached in memory (Redis, 100GB cluster, LRU eviction, 5-minute
TTL). Cache hit rate target: 95%. On cache miss: read MongoDB document,
project allowed fields, populate cache, return. Average read latency target:
under 5ms at p99.

**Write path**: Write to MongoDB primary with `writeConcern: majority` for
critical fields (security, payment) and `w: 1` for non-critical fields
(devices, preferences). Invalidate cache entry on write.
