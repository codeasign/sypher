# Document Databases -- Interview Playbook

## How This Topic Appears

Document database questions show up in system design interviews in these
forms:

**Schema Design Questions**
1. "Design a document schema for Twitter's tweet feed. Each user sees the
   last 200 tweets from people they follow."
2. "How would you model an e-commerce product catalog in MongoDB? Include
   categories, variants, and inventory."
3. "A messaging app stores conversations. Each conversation has 2-500
   participants. Design the document model."

**Performance and Scaling Questions**
4. "Your MongoDB cluster is slowing down. Writes are taking 500ms. Walk me
   through your debugging process."
5. "Design a shard key for a global user database with 100M users. Traffic
   is evenly distributed across all users."
6. "How do you handle a hot shard in production? One shard is receiving 10x
   the traffic of others."

**Trade-off and Comparison Questions**
7. "When would you choose MongoDB over PostgreSQL?"
8. "Compare Firestore and MongoDB for a real-time collaboration app."
9. "What is wrong with storing a user's 10,000 transactions as an embedded
   array in the user document?"

**Failure and Recovery Questions**
10. "Your secondary is in RECOVERING state. What happened and how do you fix
    it?"

## What the Interviewer Is Actually Testing

When you hear a document database question, the interviewer is evaluating:

**Schema design judgment**: Can you decide when to embed and when to
reference? Do you understand the access patterns well enough to design a
document that serves them efficiently? This is the primary signal. Most
candidates default to relational thinking and over-use references. The
candidates who understand the document model lean into embedding and can
articulate the trade-offs.

**Distributed systems fundamentals**: Do you understand sharding,
replication, and consistency models? Can you reason about what happens when
a shard key is poorly chosen? The interviewer wants to see that you know
CAP theorem applies to document databases too.

**Operational maturity**: Can you debug a slow cluster? Do you know what
COLLSCAN means? Can you read an explain plan? Junior candidates talk about
features. Senior candidates talk about failure modes and monitoring.

**Level of abstraction**: Do you talk about documents at the application
level (JSON, field names, nesting) or at the storage level (B-tree depth,
cache eviction, page size)? Both are valuable, but more senior roles expect
the storage-level understanding.

## Level-by-Level Expectations

### L3/L4 (Early Career)

**What you need to show**: You understand the document model. You can design
a simple collection with proper indexes. You know the difference between
embedding and referencing and can give examples of when to use each.

**Sample answer (schema design for tweet feed)**:
"I would store tweets in a collection with each tweet as a document. The
document would have fields for tweet ID, author ID, content, timestamp, and
an array of like IDs. I would embed up to the first 2 comments directly in
the tweet document because they are almost always read with the tweet. For
the user feed, I would use a separate user_feed collection where each
document is a user's pre-computed feed: an array of tweet IDs from followed
users, generated asynchronously when a tweet is created."

### L5 (Senior)

**What you need to show**: You design for scale and failure. You understand
shard key selection, query patterns across the full data lifecycle, and
operational implications of schema decisions. You can compare document
databases with other storage systems.

**Sample answer (shard key for 100M users)**:
"I would use a hashed shard key on `userId`. Hashing ensures even
distribution across shards -- no hot spots from celebrity users or
monotonically increasing IDs. The trade-off is that range queries on userId
are not supported. For queries that need user ranges, I would maintain a
secondary collection with a different shard key. I would choose a high
number of shards upfront (16-32) to avoid re-sharding later. Each shard
would be a 3-node replica set for HA."

### L6 (Staff)

**What you need to show**: You design multi-region, multi-team systems. You
consider consistency guarantees, operational cost, data migration strategies,
and the full lifecycle from prototype to 100M users. You can articulate when
the document database is the wrong tool.

**Sample answer (multi-region MongoDB)**:
"I would deploy a primary cluster in us-east-1 with read replicas in
eu-west-1 and ap-southeast-1. Application instances connect to the nearest
replica with readPreference: nearest for sub-5ms local reads. For writes, all
writes go to us-east-1 primary with writeConcern: majority. Cross-region
replication is async, so reads from non-primary regions may be up to 200ms
stale. For consistency-critical operations -- payment confirmation, password
change -- I force reads to the primary region. I monitor the replication lag
between regions and alert if it exceeds 5 seconds. For write-heavy regions,
I consider adding a local shard that writes to the primary region
asynchronously."

### L7/L8 (Principal / Distinguished)

**What you need to show**: You design the system that other teams build on.
You understand the storage engine internals, the cost model at petabyte
scale, and the organizational challenges of operating a multi-tenant document
database platform. You can design a new document database or extend an
existing one.

**Sample answer (building a document database platform for 1000+ teams)**:
"I would build a multi-tenant document database platform where each tenant
gets an isolated document namespace. The storage layer uses a log-structured
merge-tree for write-optimized ingestion, with a separate B-tree index for
point lookups. Each tenant configures their own index set, shard key, and
consistency requirements. The platform auto-scales shards based on CPU and
storage utilization. I would build a query analyzer that detects cross-shard
scatter-gather queries and suggests schema changes to the tenant. For
operational efficiency, I would implement automatic read-replica placement
in regions where the tenant has the most traffic."

## FAANG Framework for Answering

Use this six-step structure for any document database design question:

**Step 1: Clarify scope and constraints** (30 seconds)
"Before designing, I need to know: read-to-write ratio, data size per entity,
latency SLA, consistency requirements, and whether this is single-region or
multi-region."

**Step 2: Define the data model** (1 minute)
"Here is my proposed document schema. I am embedding [X] because [access
pattern reason]. I am referencing [Y] because [growth/reuse reason]. Each
document will be approximately [size] and the collection will have [count]
documents."

**Step 3: Choose the query pattern** (30 seconds)
"The primary queries are: find by [field], aggregate by [field] over [time],
and sort by [field]. To serve these, I need indexes on [list]."

**Step 4: Address scale** (1 minute)
"At [target users], I need sharding. My shard key is [key] with [hash/range].
Each shard will hold approximately [data size]. I choose [number] shards
initially. Read replicas in [regions] for global traffic."

**Step 5: Discuss trade-offs** (30 seconds)
"The main trade-off is [embedding vs referencing / consistency vs latency /
shard key flexibility]. I chose [option] because [reason]. The downside is
[downside], and I mitigate it by [mitigation]."

**Step 6: Describe failure handling** (30 seconds)
"If the primary fails, [replica set election]. If a shard becomes hot,
[rebalance or split chunk]. If secondary lags, [increase oplog or add
resources]. Monitoring: [key metrics]."

## Phrase Bank

Use these phrases in your answers to signal expertise:

**Schema design**:
- "I would embed the relationship because it is always accessed together with
  the parent and has a bounded cardinality of at most 10-20 items."
- "For unbounded arrays, I use a referencing pattern with a separate
  collection and a compound index on the foreign key plus timestamp."
- "I always include a schema_version field in every document to enable safe
  lazy migrations."

**Indexing**:
- "Covering indexes that include all fields in the query projection eliminate
  document fetch entirely."
- "The ESR (Equality, Sort, Range) rule governs compound index design:
  equality fields first, sort field next, range field last."
- "I watch for IXSCAN vs COLLSCAN in the explain plan. COLLSCAN on a
  collection over 1M documents is always a bug."

**Sharding**:
- "A hashed shard key provides uniform distribution but destroys range query
  capability. I accept this trade-off for write-heavy workloads."
- "Jumbo chunks occur when the shard key has low cardinality or monotonically
  increasing values. Use a high-cardinality key with random distribution."
- "Scatter-gather queries that hit every shard are a red flag. If over 10%
  of queries are scatter-gather, the shard key needs redesigning."

**Operations**:
- "The oplog window should stay above 2 hours. A shrinking window means the
  oplog is too small or a secondary cannot keep up."
- "Rolling upgrades let me update each node in a replica set without downtime:
  step down the primary, upgrade, repeat."
- "WiredTiger cache pressure is visible in the eviction rate. Sustained
  evictions above 1,000 pages/second mean the working set exceeds cache."

**Consistency**:
- "Write concern majority gives you journaled durability across the replica
  set at the cost of 2x-3x write latency."
- "Read preference nearest reduces latency for global users but may return
  stale data bounded by maxStalenessSeconds."
- "Causal consistency ensures your reads reflect your own writes: you see
  your own updates even when reading from a secondary."

## Red Flags

Avoid these mistakes in your answers:

1. **Designing a relational schema in MongoDB.** If every relationship is a
   reference and every query needs 5 $lookup stages, you should use a
   relational database. The interviewer will notice this immediately.

2. **Ignoring the 16MB document limit.** Proposing to store a user's entire
   photo library inside their profile document tells the interviewer you
   have not run MongoDB at scale.

3. **No shard key strategy.** Saying "we will add sharding when we need it"
   without specifying the shard key shows you do not understand that shard
   keys are a permanent decision.

4. **Claiming MongoDB is ACID.** MongoDB has multi-document transactions
   since 4.0, but they have performance implications. Do not claim MongoDB
   is a drop-in replacement for PostgreSQL on transactional workloads.

5. **Ignoring index management.** Saying "we add indexes as needed" without
   discussing the write-performance cost of indexes suggests you have not
   experienced an index-bloated cluster.

6. **No failure mode discussion.** An answer that only covers happy-path
   performance but does not discuss what happens during a network partition
   or primary failure reads as junior.

7. **Using "sharding" when you mean "replication."** Sharding scales writes.
   Replication provides HA and read scale. Confusing them is a common
   mistake.

8. **Over-engineering for scale.** Suggesting a 16-shard cluster for an
   application with 10,000 users signals inexperience with operational cost.
   Start with a replica set. Add sharding when you hit a bottleneck.

## Practice Drills

Drill 1 (5 min): Design a document schema for a blog engine with posts,
comments, tags, and author profiles. Identify what you embed, what you
reference, and why.

Drill 2 (5 min): Your MongoDB primary is at 95% CPU. Writes are timing out
at 10 seconds. Walk through your debugging checklist in order of likelihood.

Drill 3 (10 min): Design a sharding strategy for a ride-sharing application
with 50 million users, 10 million rides per day, and global coverage
(US, Europe, Asia). Specify shard key, number of shards, replication factor,
and read/write routing.

Drill 4 (10 min): Your sharded cluster has one shard that stores 80% of the
data. Chunks are unbalanced. Describe the cause, the immediate mitigation,
and the permanent fix.

Drill 5 (15 min): Design a migration from PostgreSQL to MongoDB for a
multiplayer game backend. Data includes: player profiles, match history,
friendship graph, in-game currency ledger. Identify which collections map
well to documents and which do not. Propose the migration strategy with
zero downtime.
