## Tier 1: Foundation

### Challenge 1.1: Implement a Thread-Safe Token Bucket in Python

**Prompt**: Implement a thread-safe token bucket rate limiter in Python. The bucket should support configurable capacity and refill rate. It must handle concurrent access from multiple threads. Also implement a `consume` method that returns True or False. The bucket should use `time.monotonic()` instead of `time.time()` to avoid issues with system clock changes. Include a test that verifies the bucket never allows more requests than the capacity in any burst.

**Model Answer**:

```python
import threading
import time

class TokenBucket:
    def __init__(self, capacity: int, refill_rate: float):
        self.capacity = capacity
        self.refill_rate = refill_rate  # tokens per second
        self.tokens = float(capacity)
        self.last_refill = time.monotonic()
        self.lock = threading.Lock()

    def _refill(self):
        now = time.monotonic()
        elapsed = now - self.last_refill
        if elapsed > 0:
            new_tokens = elapsed * self.refill_rate
            self.tokens = min(self.capacity, self.tokens + new_tokens)
            self.last_refill = now

    def consume(self, count: int = 1) -> bool:
        with self.lock:
            self._refill()
            if self.tokens >= count:
                self.tokens -= count
                return True
            return False

    def get_available_tokens(self) -> float:
        with self.lock:
            self._refill()
            return self.tokens
```

**Key points to verify**:
- Thread safety is achieved through a lock (or threading.RLock for reentrancy)
- Refill calculation uses elapsed time, not a timer callback (no background threads needed)
- Tokens are capped at capacity using `min()`
- `time.monotonic()` is used instead of `time.time()` to avoid issues with system clock changes (NTP adjustments, daylight savings)
- The `consume` method returns a boolean, allowing the caller to decide how to handle rejection
- The token count is stored as a float to handle fractional refills (e.g., 0.5 tokens per second)

**Test case**:
```python
bucket = TokenBucket(capacity=10, refill_rate=1)

# Consume 10 tokens immediately (burst)
for i in range(10):
    assert bucket.consume(), f"Request {i} should be allowed"

# 11th request should be rejected
assert not bucket.consume(), "11th request should be rejected"

# Wait 5 seconds for refill (5 tokens)
time.sleep(5)
assert bucket.consume(), "Should have 5 tokens after 5 seconds"
assert bucket.consume(), "Should have 4 tokens after 6 seconds"
assert bucket.consume(), "Should have 3 tokens after 7 seconds"
assert bucket.consume(), "Should have 2 tokens after 8 seconds"
assert bucket.consume(), "Should have 1 token after 9 seconds"
assert not bucket.consume(), "Should have 0 tokens after 10 seconds"
```

### Challenge 1.2: Implement a Fixed Window Counter with Redis

**Prompt**: Implement a fixed window rate limiter that tracks requests per minute. Use a sliding reset mechanism where the window starts at the first request, not at the top of the clock hour. The implementation should work with Redis commands. Explain the boundary burst problem and how it affects this implementation.

**Model Answer**:

```python
import redis
import time

class FixedWindowRateLimiter:
    def __init__(self, redis_client: redis.Redis, limit: int, window: int = 60):
        self.redis = redis_client
        self.limit = limit
        self.window = window

    def allow_request(self, client_id: str) -> bool:
        key = f"fw:{client_id}"
        now = int(time.time())
        window_key = now // self.window  # floor division for window alignment

        full_key = f"{key}:{window_key}"

        current = self.redis.get(full_key)
        if current is None:
            # First request in this window
            self.redis.setex(full_key, self.window * 2, 1)
            return True

        current = int(current)
        if current >= self.limit:
            return False

        self.redis.incr(full_key)
        return True
```

**Key points to verify**:
- Window is aligned to natural time boundaries (wall clock divisions)
- TTL is set to `window * 2` to handle edge cases with clock drift
- The implementation uses `INCR` for the counter increment
- Race condition: two concurrent requests could both read `current < limit` and both increment. Fix with a Lua script for atomicity.
- The boundary burst problem: a client can send 100 requests at 12:59:59 and another 100 at 1:00:00, effectively getting 200 requests in 2 seconds. This is the main weakness of fixed window.

**Boundary burst explanation**: With a limit of 100 requests per minute, the window resets at each minute boundary. A client sends 100 requests at 12:59:59 and another 100 at 1:00:00. The rate limiter sees 100 requests in the 12:59:00-13:00:00 window and 100 requests in the 13:00:00-13:01:00 window. But from the backend's perspective, it received 200 requests in 2 seconds. The sliding window algorithm solves this.

### Challenge 1.3: Implement Rate Limit Headers Middleware

**Prompt**: Write a Flask middleware function that adds `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers to every response. Use a token bucket rate limiter behind the scenes. The middleware should extract the client identity from the X-API-Key header, falling back to the remote IP. Rejected requests should return a 429 status code with a JSON error body.

**Model Answer**:

```python
from flask import request, jsonify
import time

def rate_limit_middleware(bucket_func):
    """
    Decorator that applies rate limiting and adds headers to responses.
    bucket_func(client_id) returns a TokenBucket instance.
    """
    def decorator(f):
        def wrapper(*args, **kwargs):
            client_id = request.headers.get("X-API-Key",
                           request.remote_addr)
            bucket = bucket_func(client_id)

            allowed = bucket.consume()
            remaining = int(bucket.get_available_tokens())
            reset_time = int(time.time()) + 60  # approximate reset

            if not allowed:
                response = jsonify({
                    "error": "rate_limit_exceeded",
                    "message": "API rate limit exceeded. Please wait before retrying.",
                    "retry_after": 60
                })
                response.status_code = 429
                response.headers["X-RateLimit-Limit"] = bucket.capacity
                response.headers["X-RateLimit-Remaining"] = 0
                response.headers["X-RateLimit-Reset"] = reset_time
                response.headers["Retry-After"] = "60"
                return response

            response = f(*args, **kwargs)
            if isinstance(response, tuple):
                body, status, headers = response
                headers["X-RateLimit-Limit"] = bucket.capacity
                headers["X-RateLimit-Remaining"] = remaining
                headers["X-RateLimit-Reset"] = reset_time
                return (body, status, headers)
            else:
                response.headers["X-RateLimit-Limit"] = bucket.capacity
                response.headers["X-RateLimit-Remaining"] = remaining
                response.headers["X-RateLimit-Reset"] = reset_time
                return response
        return wrapper
    return decorator
```

**Key points to verify**:
- Headers are added to both allowed and rejected responses, not just rejected ones
- The `Retry-After` header is included on 429 responses
- Client identity is extracted from the API key header, falling back to IP
- Headers are set before the response is returned
- The remaining count is accurate at the point of the check (captured before forwarding)
- The middleware handles both direct response objects and tuple responses

## Tier 2: Design

### Challenge 2.1: Design a Multi-Tier Rate Limiter for SaaS Platform

**Prompt**: Design a rate limiter for a SaaS platform with three tiers: Free (100 req/min), Pro (1,000 req/min), and Enterprise (10,000 req/min). Enterprise customers can burst to 50,000 req/min for up to 60 seconds. The rate limiter must prevent a noisy Free tenant from degrading Pro and Enterprise tenants. Include the Redis key structure, the Lua script for each level, and the hierarchical check order.

**Rubric (scored out of 10)**:
- Hierarchical architecture description (2 points)
- Token bucket algorithm with tier-specific parameters (2 points)
- Global pool for system-wide protection (2 points)
- Burst handling for Enterprise tier (2 points)
- Failure mode analysis (2 points)

**Model Answer**:

The design uses a three-level hierarchical rate limiter. Level 1 is a global system-wide limit of 100,000 req/min that protects the entire infrastructure from total overload. Level 2 is a per-tier aggregate limit: Free aggregate limit of 10,000 req/min, Pro aggregate limit of 50,000 req/min, and Enterprise aggregate limit of unlimited (only bounded by the global limit). Level 3 is per-tenant limits within each tier.

Each tenant gets a token bucket with the following parameters:
- Free: capacity=100, refill_rate=100/60 = 1.67 tokens/sec
- Pro: capacity=1,000, refill_rate=1,000/60 = 16.67 tokens/sec
- Enterprise: capacity=50,000, refill_rate=10,000/60 = 166.67 tokens/sec (burst drops to sustained after 60 seconds)

The hierarchical check evaluates level 3 first (per-tenant, fastest because it can be cached locally), then level 2 (per-tier aggregate, requires Redis), then level 1 (global, single Redis key). A request must pass all three levels. This prevents a noisy Free tenant from consuming the Free tier's aggregate limit and affecting other Free tenants, and also prevents the Free tier from consuming global capacity needed by Pro and Enterprise.

Redis keys are structured as `rl:global`, `rl:tier:{tier_name}`, and `rl:tenant:{tenant_id}`. Each uses a Lua script with the token bucket algorithm. The global and tier-level limits use a sliding window counter for memory efficiency (only 24 bytes each), while per-tenant limits use the token bucket for burst handling.

For failure modes: if Redis is unavailable, the circuit breaker degrades to local per-tenant token buckets with conservative limits (50 percent of normal). A background sync job reconciles local state with Redis when it recovers. If one tier's aggregate Redis key fails, only that tier is affected; other tiers continue operating normally.

### Challenge 2.2: Design a Rate Limiter for Real-Time Chat API

**Prompt**: Design a rate limiter for a real-time chat API that supports WebSocket connections. Each user can send 30 messages per minute, but conversations can have up to 200 participants. The system must handle 1 million concurrent WebSocket connections. Include the per-connection and per-conversation limits, the Redis sharding strategy, and the backpressure mechanism.

**Rubric (scored out of 10)**:
- WebSocket rate limiting approach (3 points)
- Connection-level and message-level limits (2 points)
- Concurrency handling for 1 million connections (2 points)
- Per-conversation rate limiting (2 points)
- Backpressure mechanism (1 point)

**Model Answer**:

The design separates connection-level rate limiting from message-level rate limiting. Connection-level: each WebSocket connection is allowed 10 connection attempts per minute from a single IP. Exceeding this results in a temporary IP ban for 5 minutes. This prevents connection-flood attacks. Message-level: each user is allowed 30 messages per rolling minute, tracked via a sliding window counter in Redis.

Per-conversation rate limiting prevents a single conversation from flooding the system. Each conversation has a limit of 200 messages per second, regardless of how many participants are sending. This uses a token bucket with capacity=200, refill_rate=200/sec. If a conversation hits this limit, the rate limiter throttles all participants equally, spreading the backpressure across the group.

For 1 million concurrent connections, the rate limiter must be embedded in the connection handler rather than a separate service to avoid a network hop for every message. Each handler maintains a local in-process token bucket for the connected user, syncing to Redis every 30 seconds. The local bucket is conservative: it uses 70 percent of the user's actual limit to leave room for the sync delay. This means the local bucket allows 21 messages per minute (70 percent of 30), and the remaining 9 messages per minute are allocated from the global pool via Redis sync.

Backpressure is essential for WebSocket rate limiting. When a user exceeds the message limit, the server does not disconnect the WebSocket. Instead, it sends a `rate_limit` frame with the remaining time until the user can send again. The connection stays open for receiving messages. This prevents the reconnect storm that would result from disconnecting rate-limited users. The rate_limit frame includes the retry-after duration in milliseconds.

Redis is sharded by user ID hash across 20 nodes. Each shard handles rate limit state for roughly 50,000 users. The Lua script for message rate limiting runs in under 2ms per evaluation. Total Redis load: 1 million users * 30 messages/min / 60 = 500,000 Redis calls per second, distributed across 20 shards = 25,000 calls per second per shard, which is exactly the capacity of a single Redis core.

### Challenge 2.3: Design a Rate Limiter for a Data Pipeline

**Prompt**: Design a rate limiter for a data pipeline that ingests events from 10,000 sources. Each source can send up to 1,000 events per second. The pipeline has three stages with different throughput capacities: ingestion (50,000 events/sec), processing (20,000 events/sec), and storage (10,000 events/sec). The rate limiter must protect the slowest stage. Include the per-source limits, the adaptive backpressure between stages, and the dead-letter queue strategy.

**Rubric (scored out of 10)**:
- Multi-stage pipeline awareness (3 points)
- Backpressure propagation between stages (2 points)
- Adaptive limiting based on bottleneck (2 points)
- Buffering and queuing strategy (2 points)
- Monitoring and alerting (1 point)

**Model Answer**:

The rate limiter must protect the bottleneck stage (storage at 10,000 events/sec) while allowing higher throughput at earlier stages. The design uses a leaky bucket at each stage boundary, with the bucket size at each stage matching the downstream stage's capacity.

Stage 1 (ingestion): Per-source token bucket with capacity=1,000, refill_rate=1,000/sec. The aggregate across 10,000 sources is capped at 10,000 events/sec to match the storage stage's capacity. This is a global counter: when the storage stage is near capacity, per-source limits are dynamically reduced proportionally.

Stage 2 (processing): A leaky bucket with a queue buffer of 50,000 events. The bucket processes at 20,000 events/sec. If the queue exceeds 40,000 (80 percent full), it sends backpressure to stage 1, reducing per-source limits by 50 percent. The queue is an in-memory buffer on the processing server with spillover to disk for durability.

Stage 3 (storage): A leaky bucket with queue buffer of 20,000 events. Processes at 10,000 events/sec. The queue length is the primary signal for the adaptive rate limiter. When the queue exceeds 10,000, it sends backpressure to stage 2, which propagates to stage 1.

The adaptive controller monitors queue depths at each stage. If stage 3's queue exceeds 10,000, it reduces the global limit at stage 1 from 10,000 to 7,000 events/sec (30 percent reduction). If the queue exceeds 15,000, it reduces to 5,000 (50 percent reduction). This prevents the buffer from filling up and causing data loss. The controller uses a proportional algorithm: reduction = queue_depth / max_queue * 0.5, ensuring gradual adjustments.

A dead-letter queue captures events that are rejected after three retries. Each event that exceeds the rate limit is retried with exponential backoff (1s, 2s, 4s). After three retries, the event is moved to the dead-letter queue in S3. The dead-letter queue is monitored and triggers a PagerDuty alert if it grows beyond 10,000 events. Events in the dead-letter queue can be replayed manually after the bottleneck is resolved.

## Tier 3: Expert

### Challenge 3.1: Design a Global Rate Limiter with Cross-Region Coordination

**Prompt**: Design a rate limiter that operates across three AWS regions (us-east-1, eu-west-1, ap-southeast-1). Each region handles its own traffic, but a client can route to any region depending on their geographic location and DNS resolution. The per-client rate limit must be enforced globally across all regions. The system must tolerate the failure of one region and the complete loss of the coordination layer. Target latency for the rate limit check is under 5ms per request. Discuss three design trade-offs and justify your choices.

**Trade-offs to consider**:

**Trade-off 1: Synchronous vs Asynchronous Coordination**

Synchronous coordination (all regions write to a single global Redis) provides exact consistency but adds 50-100ms of cross-region latency for every request. Asynchronous coordination (each region maintains local state, syncs periodically) keeps latency under 5ms but allows up to 10 seconds of overage during the sync window.

**Preferred approach**: Asynchronous coordination with local-first writes. Each region maintains its own Redis cluster. Every 5 seconds, each region publishes its top-offending clients to a global topic. Other regions consume this topic and apply temporary blocks. The overage is bounded by the sync interval (5 seconds) times the number of regions (3) times the per-client limit. For a limit of 100 req/min, the maximum overage is 5 * 3 * 100 = 1,500 requests, which occurs only if a client simultaneously exceeds limits in all three regions during the sync window. This is acceptable for standard API limits.

**Trade-off 2: Failover Behavior**

If one region fails, the remaining two regions must handle the traffic. The rate limiter should allocate the failed region's capacity to the remaining regions. This means per-client limits are not fixed numbers but percentages of the current active region count.

**Preferred approach**: A dynamic allocation system where each region's limits are proportional to the number of active regions. With 3 regions, each gets 33 percent of the global limit. With 2 regions, each gets 50 percent. The region count is pre-configured rather than computed in real-time to avoid cascading failures during the computation. The region health is determined by a leader-elected coordinator that pings each region every 5 seconds. If a region does not respond for 15 seconds, it is considered failed and the remaining regions adjust their limits.

**Trade-off 3: Consistency Model**

Strong consistency (CP) means a client cannot exceed the limit even by one request, but a single region failure can cause global rate limiting to fail. Eventual consistency (AP) means the system stays available during failures but allows bounded overage.

**Preferred approach**: AP for normal traffic, CP for critical traffic. The critical traffic path (login, payment, write operations) uses synchronous coordination with a global Redis ring that spans all regions. The normal traffic path (read operations, search, list) uses asynchronous coordination. This hybrid approach ensures the most sensitive operations are tightly controlled while keeping latency low for the majority of traffic. The critical path processes roughly 5 percent of total traffic, so the latency penalty of 50-100ms only affects 5 percent of requests.

### Challenge 3.2: Design an Adaptive Rate Limiter Using Reinforcement Learning

**Prompt**: Design a rate limiter that learns optimal limits from traffic patterns without manual configuration. The system should automatically adjust per-client limits based on historical behavior, backend health, and time-of-day patterns. The rate limiter must never allow the system to exceed 80 percent of backend capacity. Discuss the exploration vs exploitation trade-off, the cold start problem, and the feedback loop stability.

**Trade-offs to consider**:

**Trade-off 1: Exploration vs Exploitation**

The rate limiter must explore higher limits to find the optimal throughput, but exploring too aggressively risks overwhelming the backend. The system needs a controlled exploration strategy that backs off immediately when the backend shows signs of stress.

**Preferred approach**: A multi-armed bandit strategy where each client's limit is an arm. The system explores by increasing a random client's limit by 10 percent each minute. If the backend's p99 latency stays under 300ms, the exploration is deemed safe and the new limit is committed. If latency exceeds 500ms, the limit is reduced by 50 percent and the exploration is rolled back. The exploration rate (epsilon) starts at 0.2 and decays by 0.99 per day, so after 30 days the exploration rate is 0.2 * 0.99^30 = 0.148. This means the system explores less over time but never stops entirely, allowing it to adapt to changing traffic patterns.

**Trade-off 2: Cold Start for New Clients**

New clients have no historical data. The rate limiter must assign an initial limit without knowing the client's behavior pattern. Assigning too high a limit risks abuse. Assigning too low a limit frustrates the client.

**Preferred approach**: A default limit based on the client's tier (Free: 100 req/min, Pro: 1,000 req/min). The rate limiter monitors the first 100 requests to classify the client's pattern: bursty, steady, or periodic. Bursty clients have a coefficient of variation greater than 2.0 in their inter-arrival times. Steady clients have a coefficient of variation under 0.5. Periodic clients show a statistically significant autocorrelation at a fixed lag. Bursty clients get a higher capacity bucket with a lower sustained rate (capacity=200, rate=50). Steady clients get a higher sustained rate with a lower capacity (capacity=50, rate=150). Classification is complete within 2 minutes of the client's first request at 50 req/min.

**Trade-off 3: Feedback Loop Stability**

An adaptive rate limiter creates a feedback loop: the rate limiter adjusts limits, which changes traffic patterns, which triggers another adjustment. Unstable feedback loops cause oscillations where limits swing between too high and too low.

**Preferred approach**: A proportional-integral (PI) controller with a damping factor. The controller adjusts limits by a fraction of the error signal (difference between current throughput and target throughput). The proportional term responds to immediate changes with a gain of 0.5. The integral term accumulates persistent errors with a gain of 0.1 and slowly adjusts the baseline. The damping factor of 0.9 prevents overshoot by limiting the maximum adjustment to 10 percent per evaluation cycle. The evaluation cycle runs every 30 seconds, preventing high-frequency oscillations. The PI controller is:

adjustment = kp * error + ki * integral_sum
where kp = 0.5, ki = 0.1, and the integral_sum decays by damping = 0.9 each cycle.

The maximum adjustment is clamped to plus or minus 10 percent. The adjustment is applied to all rate limits as a multiplier. Individual client adjustments are computed separately using the multi-armed bandit strategy. The two systems operate at different time scales: the PI controller at 30-second intervals, the bandit at 1-minute intervals. This time-scale separation prevents interference between the two feedback loops.

```python
class AdaptiveRateController:
    def __init__(self, target_utilization=0.80, kp=0.5, ki=0.1, damping=0.9):
        self.target = target_utilization
        self.kp = kp
        self.ki = ki
        self.damping = damping
        self.integral = 0.0
        self.last_adjustment = 0.0

    def compute_adjustment(self, current_utilization):
        error = self.target - current_utilization
        self.integral = (self.integral * self.damping) + error
        adjustment = self.kp * error + self.ki * self.integral
        # Clamp to prevent oscillations
        adjustment = max(-0.10, min(0.10, adjustment))
        self.last_adjustment = adjustment
        return 1.0 + adjustment  # multiplier for all rate limits
```

## Bonus Challenge: Rate Limiter Evaluation Framework

**Prompt**: Design an evaluation framework that measures the effectiveness of a rate limiter in production. The framework should quantify accuracy, fairness, latency impact, and operational overhead. Include specific metrics, measurement methodology, and actionable thresholds.

**Model Answer**:

The evaluation framework measures four dimensions:

**Accuracy**: How close is the actual behavior to the configured limit?
- Metric: overshoot ratio = (actual_max_rate / configured_limit) - 1
- Measurement: Compare the rate limiter's logged decisions against actual backend traffic
- Target: overshoot under 5 percent for sliding window counter, under 1 percent for sliding window log
- Methodology: Deploy a shadow rate limiter that independently counts requests and compares its counts to the production rate limiter's counts

**Fairness**: Are all clients treated equally within their tier?
- Metric: Jain's fairness index = (sum(allocated_i))^2 / (n * sum(allocated_i^2))
- Range: 0 (completely unfair) to 1 (perfectly fair)
- Target: 0.95 or higher for clients within the same tier
- Methodology: Sample rate limit decisions for 1,000 random clients over 1 hour and compute the fairness index

**Latency Impact**: How much latency does the rate limiter add?
- Metric: p50, p95, p99 latency of the rate limit check
- Measurement: Instrument the rate limiter with histograms tracking time from request arrival to decision
- Target: p50 under 1ms, p99 under 10ms for Redis-backed limiters
- Methodology: Use Prometheus histograms with buckets at 0.1ms, 0.5ms, 1ms, 2ms, 5ms, 10ms, 50ms, 100ms

**Operational Overhead**: What is the resource cost of running the rate limiter?
- Metric: Redis CPU utilization, Redis memory usage, network bandwidth
- Target: Redis CPU under 50 percent at peak, memory under 70 percent of maxmemory
- Methodology: Monitor Redis INFO metrics and alert when utilization exceeds thresholds

```python
class RateLimiterEvaluator:
    def __init__(self, redis_client, limiter):
        self.redis = redis_client
        self.limiter = limiter
        self.shadow_limiter = TokenBucket(100, 1.667)  # independent limiter

    def evaluate_accuracy(self, client_id, num_requests=10000):
        """Compare rate limiter decisions against shadow limiter."""
        limiter_allowed = 0
        shadow_allowed = 0
        for i in range(num_requests):
            limiter_result = self.limiter.allow(client_id)
            shadow_result = self.shadow_limiter.consume()
            if limiter_result:
                limiter_allowed += 1
            if shadow_result:
                shadow_allowed += 1
        # The difference shows the rate limiter's accuracy
        diff = abs(limiter_allowed - shadow_allowed)
        return {
            "limiter_allowed": limiter_allowed,
            "shadow_allowed": shadow_allowed,
            "diff": diff,
            "error_rate": diff / num_requests
        }

    def evaluate_latency(self, num_checks=1000):
        """Measure p50, p95, p99 latency of rate limit checks."""
        latencies = []
        for i in range(num_checks):
            start = time.monotonic()
            self.limiter.allow("benchmark_user")
            elapsed = time.monotonic() - start
            latencies.append(elapsed * 1000)  # convert to ms
        latencies.sort()
        return {
            "p50": latencies[len(latencies) // 2],
            "p95": latencies[int(len(latencies) * 0.95)],
            "p99": latencies[int(len(latencies) * 0.99)]
        }
```

## Key Takeaways

The three-tier challenge structure tests progressively deeper understanding of rate limiter design. Foundation challenges verify implementation skills: building a thread-safe token bucket, a Redis-backed fixed window counter with boundary burst analysis, and rate limit header middleware with proper error handling.

Design challenges require system-level thinking: multi-tier rate limiting for SaaS platforms with three levels of hierarchy, WebSocket-based rate limiting for real-time chat at 1 million connections with per-conversation limits, and pipeline-aware rate limiting for data processing systems with adaptive backpressure between three stages.

Expert challenges push into the hardest problems: cross-region coordination with three trade-off dimensions (synchronous vs asynchronous coordination, failover behavior, consistency model), and adaptive rate limiting using multi-armed bandits for exploration, PI controllers for stability, and cold start classification. Each challenge requires you to articulate the trade-offs explicitly and justify your choices with concrete reasoning and numerical analysis.

The bonus challenge introduces an evaluation framework for measuring rate limiter effectiveness across four dimensions: accuracy, fairness, latency impact, and operational overhead. The self-assessment questions at three levels help you identify gaps in your understanding and focus your study on the areas that need the most improvement.
