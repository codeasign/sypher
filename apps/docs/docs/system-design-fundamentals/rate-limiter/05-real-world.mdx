## Company Case Studies

### Twitter: Application-Layer Rate Limiting

Twitter's API rate limiting is one of the earliest and most influential implementations. Each authenticated user gets 15 requests per 15-minute window for most endpoints, with higher limits for certain authenticated calls. Twitter uses a sliding window approach with per-endpoint, per-user accounting. This means a user consuming their limit on search does not affect their ability to post tweets.

Twitter's approach influenced the industry standard for rate limit headers. The `x-rate-limit-*` headers are now used by GitHub, Stripe, and most major API providers. Twitter's key innovation was making the limits transparent to developers through clear headers and error messages. Before Twitter standardized these headers, each API had its own convention, and developers had to learn each one separately.

**Lessons from Twitter**:
- Per-endpoint limits prevent a single bursty endpoint from exhausting a client's global quota
- 15-minute windows balance accuracy with low memory overhead
- Clear header communication reduces developer frustration and support tickets
- Authentication-based limits (OAuth tokens) are more reliable than IP-based limits
- The industry standard headers originated from Twitter's developer experience team

### GitHub: Conditional Rate Limiting with Secondary Rate Limits

GitHub implements a two-tier rate limiting system. Primary rate limits cap requests per authenticated user (5,000 requests per hour). Secondary rate limits cap the rate of requests to prevent abuse, even within the primary limit. The secondary limit is triggered when a client makes requests faster than expected, even if they are within their hourly quota.

- Secondary rate limits catch a class of abuse that primary limits miss entirely

- The secondary limit should be invisible to well-behaved clients

### Stripe: Idempotency-Key-Based Rate Limiting

Stripe's API rate limiting is unique because it accounts for idempotency keys. If a client sends the same request twice with the same idempotency key, Stripe processes it once and does not count the duplicate against the rate limit. This protects clients doing safe retries without penalizing them for the retry itself.

```python
# Stripe-inspired rate limiter with idempotency awareness
class IdempotentRateLimiter:
    def __init__(self, redis_client):
        self.redis = redis_client
        self.limit = 100  # requests per window
        self.window = 60  # 1 minute window

    def check_and_record(self, api_key, idempotency_key=None):
        key = f"rate:stripe:{api_key}"

        if idempotency_key:
            # Check if we already processed this request
            idem_key = f"idem:{api_key}:{idempotency_key}"
            if self.redis.get(idem_key):
                return True  # Duplicate, already counted, skip rate limit

        # Standard rate limit check
        current = self.redis.get(key)
        if current and int(current) >= self.limit:
            return False  # Rate limit exceeded

        pipe = self.redis.pipeline()
        pipe.incr(key)
        pipe.expire(key, self.window)
        result = pipe.execute()

        if idempotency_key:
            # Record idempotency but don't double-count
            self.redis.setex(idem_key, 86400, time.time())

        return int(result[0]) <= self.limit
```

**Lessons from Stripe**:
- Idempotency-aware rate limiting prevents double-counting during safe retries
- Developers appreciate that safe retries consume fewer rate limit tokens
- The idempotency approach requires careful state management in Redis
- This pattern is especially valuable for payment APIs where retries are common
- The idempotency key must be hashed to avoid storing raw keys in Redis

### AWS API Gateway: Burst vs Sustained Rate Limits

AWS API Gateway implements rate limiting with two distinct parameters: burst limit and rate limit. The burst limit controls the maximum number of requests in a short burst. The rate limit controls the sustained request rate per second. This separation is critical because most web traffic is bursty, not steady.

## Open Source Implementations

### Envoy Rate Limiting

Envoy Proxy includes a global rate limiting service that can be deployed as a standalone gRPC service. It supports token bucket and sliding window algorithms, configurable descriptors (hierarchical limit keys), and shadow mode for testing limits without enforcement. The shadow mode is particularly useful: it logs rate limit decisions without enforcing them, allowing operators to see the impact of a new policy before enabling it.

```yaml
# Envoy rate limit configuration
rate_limits:
  - stage: 1
    actions:
      - remote_address: {}
    limit:
      requests_per_unit: 100
      unit: second
  - stage: 2
    actions:
      - request_headers:
          header_name: ":path"
          descriptor_key: "path"
    limit:
      requests_per_unit: 10000
      unit: minute
```

Envoy's approach is notable for its hierarchical limit descriptors. A limit is the sum of all matching descriptors, allowing complex policies like "100 requests per second per IP, with a maximum of 10,000 requests per minute per path." Envoy handles millions of requests per second in production at Lyft, Google, and Stripe.

### Redis-Module-Based Limiters

Several open source Redis modules implement rate limiting as native C extensions running inside Redis. `redis-cell` implements the generic cell rate algorithm (GCRA) -- the same algorithm used by AWS and Kong. It provides under 1 microsecond evaluation time per check, roughly 20x faster than Lua scripts.

```bash
# redis-cell example
> CL.THROTTLE user:8472 100 50 60 1
1) (integer) 0        # 0 = allowed, 1 = rejected
2) (integer) 100      # total limit
3) (integer) 98       # remaining
4) (integer) -1       # retry after seconds (-1 if allowed)
5) (integer) 12       # time until full reset
```

The GCRA algorithm is a variant of the token bucket that uses a single counter instead of a full bucket state. It is more memory-efficient and faster to compute. Redis modules extend the Redis server with native functions that run in the Redis process, bypassing the Lua interpreter overhead.

### Kong Rate Limiting Plugin

Kong API Gateway's rate limiting plugin supports five policies: local (in-memory), cluster (database-backed), Redis, Redis with sentinel, and Redis cluster. It supports limiting by consumer, credential, IP, and path. The plugin architecture means rate limiting is one of many features Kong provides, reducing the operational burden of running a separate rate limiter.

### Cloudflare Rate Limiting

Cloudflare provides rate limiting at the edge, before traffic reaches your origin server. It supports per-IP, per-country, per-path, and custom filters. Burst detection with automatic mitigation. Managed rulesets for common attack patterns. Analytics showing blocked vs allowed traffic. Cloudflare's edge deployment means rate limiting adds zero latency to your server because the decision happens at Cloudflare's 330+ data centers before requests even reach your infrastructure.

## Cloud Provider Implementations

**AWS WAF Rate Limiting**: Supports rule-based rate limiting with IP sets, geographic matching, and SQL injection detection. Rate limits are applied per 5-minute sliding window. Integrates with CloudFront, ALB, and API Gateway. AWS WAF can handle millions of requests per second at the CloudFront edge. Pricing is based on the number of web ACLs and rules, not on request volume.

**GCP Cloud Armor**: Supports rate limiting at the Google Cloud edge. Provides per-client-IP rate limiting with configurable thresholds and actions (allow, deny, redirect). Integrates with Cloud Load Balancing and Cloud CDN. GCP's Bot Management adds behavioral rate limiting that distinguishes between human users and automated scripts.

**Azure Front Door WAF**: Supports custom rate limit rules per client IP, per geographic region, and per query string. Rate limits are expressed in requests per minute. ## Lessons Learned

**Lesson 1: Start Simple, Add Complexity Gradually**

Every team that implemented a complex rate limiter from day one regretted it. Start with a single token bucket per client. Add per-endpoint limits when you see a need. Add hierarchical limits when you have multi-tier pricing. Premature complexity in rate limiting creates bugs that are hard to debug. The most successful rate limiter deployments started as a simple 20-line Lua script and evolved over months.

**Lesson 2: Headers Are Your Best Documentation**

Clear rate limit headers eliminate 80 percent of support tickets about API throttling. Every response must include the limit, the remaining count, and the reset time. GitHub and Stripe both report that investing in clear headers reduced developer support requests by over 50 percent. The headers should be documented in the API reference with examples.

**Lesson 3: Monitor the Rate Limiter Itself**

Rate limiters fail silently. A Redis connection pool exhaustion in the rate limiter causes every request to be denied or allowed (depending on fail-open/fail-closed configuration) without obvious error signals. Monitor rate limiter health independently from application health. The rate limiter's `/health` endpoint should check Redis connectivity, cache size, and Lua script availability.

**Lesson 4: Plan for Retry Backoff on Both Sides**

A rate limiter is only as good as the retry behavior of its clients. Server-side, enforce Retry-After compliance by tracking retry frequency per client. Client-side, implement exponential backoff with jitter. The most common production incidents involving rate limiters are traced to uncontrolled retry loops. A well-designed retry strategy is as important as the rate limiter itself.

**Lesson 5: Rate Limiting Is Not a Security Tool**

Rate limiting is a traffic management tool, not a security tool. It cannot distinguish between a legitimate burst from a popular new feature and a DDoS attack using 100,000 unique IPs. For security, use DDoS protection services (AWS Shield, Cloudflare DDoS), WAF rules, and authentication. Rate limiting complements security tools but does not replace them. It manages traffic shape and volume; security tools manage threat detection and mitigation.

## Key Takeaways

Four major companies demonstrate different approaches to rate limiting: Twitter's per-endpoint sliding window, GitHub's two-tier primary+secondary system, Stripe's idempotency-key-aware limiting, and AWS's burst vs sustained separation. Each approach solves a different problem: Twitter focuses on endpoint isolation, GitHub on burst protection, Stripe on retry fairness, and AWS on traffic pattern management.

Open source implementations offer battle-tested starting points: Envoy for service mesh deployments with shadow mode, redis-cell for high-performance Redis-native limiting with GCRA, and Kong for gateway-based enforcement. Cloud providers offer edge-based rate limiting that adds zero origin latency: AWS WAF, GCP Cloud Armor, and Azure Front Door WAF.

Five lessons from production deployments emphasize starting simple, communicating clearly through headers, monitoring the rate limiter independently, planning retry behavior on both server and client sides, and understanding that rate limiting is a traffic management tool, not a security solution.

## Extended Case Study: Netflix Rate Limiting

Netflix operates one of the largest API ecosystems in the world, handling billions of requests per day across hundreds of microservices. Their rate limiting strategy evolved through three phases.

### Phase 1: Centralized Gateway Limiting

Netflix initially used Zuul, their API gateway, for centralized rate limiting. All traffic passed through Zuul, which maintained per-client counters in EVCache (Netflix's distributed caching layer, similar to Memcached). This worked well for the first 10 million requests per day but broke down as traffic grew to 100 million+.

Problems faced:
- Zuul became a bottleneck, handling all rate limit decisions for all services
- EVCache latency (5-15ms) was too high for latency-sensitive endpoints
- A single misconfigured rate limit rule could take down the entire gateway

### Phase 2: Per-Service Rate Limiting

Netflix moved rate limiting into each microservice. Each service maintained its own rate limit rules and state. EVCache remained the shared state store, but each service had its own cluster. This eliminated the Zuul bottleneck but introduced new problems:

- Rate limit configuration was duplicated across 100+ services
- Consistent rate limits across services required manual coordination
- New services often forgot to implement rate limiting
- Audit and compliance were difficult because each service had its own implementation

### Phase 3: Sidecar-Based Rate Limiting

Netflix's current approach uses a sidecar proxy (based on Envoy) that runs alongside each service container. The sidecar handles rate limiting, circuit breaking, and other cross-cutting concerns. Rate limit rules are defined centrally in a configuration service and pushed to all sidecars.

```yaml
# Netflix-inspired sidecar rate limit configuration
rate_limits:
  - match:
      source: "service-a"
      destination: "service-b"
    limits:
      - requests_per_unit: 1000
        unit: second
        burst: 5000
  - match:
      source: "service-a" 
      destination: "service-c"  # expensive service
    limits:
      - requests_per_unit: 100
        unit: second
        burst: 200
```

The sidecar approach solved the key problems:
- Centralized configuration, distributed enforcement
- No code changes needed in individual services
- Consistent behavior across all services
- Independent scaling of rate limiter and service

## Extended Case Study: Google's Global Rate Limiting

Google operates one of the most sophisticated rate limiting systems in the world, handling traffic from Search, Gmail, YouTube, Google Cloud, and hundreds of other products. Their system is notable for its global coordination and adaptive behavior.

### Google's Global Rate Limit Architecture

Google's rate limiter operates at multiple levels:

**Level 1: Frontend Server Limiting**: Each frontend server maintains local rate limit counters. If a client exceeds an aggressive local limit (100 req/s), the frontend rejects the request immediately without consulting any central service. This catches obvious abuse at zero additional latency.

**Level 2: Regional Coordinator**: If the request passes the frontend limit, the frontend checks with the regional rate limit coordinator. The coordinator maintains a sliding window counter for each client in the region. If the client exceeds the regional limit (1,000 req/s), the coordinator rejects the request.

**Level 3: Global Coordinator**: If the request passes the regional limit, it is checked against the global coordinator. The global coordinator aggregates counters from all regions every 5 seconds. If the client exceeds the global limit (10,000 req/s across all regions), the global coordinator issues a throttling signal to all regions.

```python
# Simplified Google-style multi-level rate limiter
class GoogleStyleRateLimiter:
    def __init__(self, local_limit=100, regional_limit=1000, global_limit=10000):
        self.local = TokenBucket(local_limit, local_limit)  # capacity=limit
        self.regional_allowed = 0
        self.global_allowed = 0
        self.sync_interval = 5  # seconds

    def allow(self, client_id, region):
        # Level 1: Local check (fast, no network)
        if not self.local.consume():
            return False

        # Level 2: Regional check (async, local network)
        self.regional_allowed += 1
        if self.regional_allowed > self.regional_limit:
            return False

        # Level 3: Global check (sync, cross-region, every 5s)
        if self.regional_allowed % (self.regional_limit // self.sync_interval) == 0:
            global_allowed = self.sync_global(client_id)
            if global_allowed > self.global_limit:
                return False

        return True
```

### Key Design Principles from Google

**Defense in depth**: Multiple layers of rate limiting ensure that a failure in any single layer does not compromise the entire system. The local layer catches obvious abuse. The regional layer handles normal traffic. The global layer catches cross-region attacks.

**Fail open at lower levels, fail closed at upper levels**: The frontend servers fail open (allow all) if the regional coordinator is unreachable. The regional coordinator fails closed (reject all) if the global coordinator is unreachable. This ensures the system stays available during partial failures while preventing global abuse during coordinator failures.

**Automatic adjustment based on backend health**: Google's rate limiters monitor the backend's request queue depth. If the queue depth exceeds a threshold, the rate limiters automatically reduce their limits. This creates a feedback loop where the rate limiters protect the backend from overload without manual intervention.

## Open Source Implementation Deep Dive

### Envoy Rate Limit Service

Envoy's rate limiting service is a standalone gRPC server that Envoy proxies call for rate limit decisions. It supports token bucket and sliding window algorithms. The configuration uses descriptors that define the hierarchy of rate limit keys.

```yaml
# Envoy rate limit service configuration
descriptors:
  - key: tier
    value: premium
    descriptors:
      - key: route
        value: search_api
        rate_limit:
          requests_per_unit: 10000
          unit: minute
          unlimited: false
      - key: route
        value: write_api
        rate_limit:
          requests_per_unit: 1000
          unit: minute
  - key: tier
    value: free
    rate_limit:
      requests_per_unit: 100
      unit: minute
```

Envoy's rate limiter supports shadow mode, where decisions are logged but not enforced. This allows operators to test new rate limit policies in production before enabling them. The shadow mode logs the decision, the current state, and what action would have been taken.

### redis-cell

redis-cell is a Redis module that implements the Generic Cell Rate Algorithm (GCRA). It provides a single command, `CL.THROTTLE`, that handles both token bucket and leaky bucket variants. The module is written in C and executes in under 1 microsecond per call, roughly 20x faster than the equivalent Lua script.

```bash
# redis-cell GCRA: 100 requests per 60 seconds, burst to 200
CL.THROTTLE user:8472 100 200 60 1
```

The GCRA algorithm is notable because it uses a single counter value instead of separate tokens and last_refill fields. The counter encodes both the current capacity and the refill time in a single 64-bit integer. This reduces memory from 48 bytes (two float64 values) to 8 bytes (one int64 value).

### Kong Rate Limiting Plugin

Kong's rate limiting plugin supports five policy backends: local, cluster, Redis, Redis Sentinel, and Redis Cluster. The local policy stores counters in Kong's shared memory (lmdb). The cluster policy stores counters in the database. The Redis policies store counters in Redis.

Kong's plugin architecture allows multiple rate limiters to be chained. You can have one rate limiter for IP-based limiting and another for API key-based limiting on the same route. Both must pass for the request to proceed.

## Real-World Performance Benchmarks

### Redis-Based Rate Limiter (Lua Token Bucket)

- 1 Redis shard: 25,000 req/s, p50 0.5ms, p99 2.5ms
- 6 Redis shards: 150,000 req/s, p50 0.8ms, p99 4.0ms
- 20 Redis shards: 500,000 req/s, p50 1.2ms, p99 6.0ms

### Local In-Process Rate Limiter

- 1 CPU core: 500,000 req/s, p50 0.001ms, p99 0.005ms
- 8 CPU cores: 4,000,000 req/s, p50 0.001ms, p99 0.005ms
- Bottleneck: CPU, not memory or network

### Hybrid Local + Global Rate Limiter

- 5-second sync interval: 50,000 req/s per instance, p50 0.5ms, p99 3.0ms
- 10-second sync interval: 100,000 req/s per instance, p50 0.3ms, p99 2.5ms
- Trade-off: Longer sync interval = lower latency but higher overage risk

## Key Takeaways

The extended case studies from Netflix and Google demonstrate that rate limiting architecture evolves with scale. Netflix progressed from centralized gateway limiting to per-service limiting to sidecar-based limiting. Google's multi-level architecture with local, regional, and global coordinators provides defense in depth with automatic backend health adjustment.

Open source implementations provide production-ready starting points: Envoy for service mesh deployments, redis-cell for high-performance Redis-native limiting, and Kong for gateway-based enforcement. Each has different strengths: Envoy excels at service mesh integration, redis-cell at raw performance, and Kong at ease of use.

Cloud provider implementations offer edge-based rate limiting at global scale: AWS WAF for simple IP-based limits, GCP Cloud Armor for high-throughput token bucket limits, and Azure Front Door WAF for cost-effective integration. Real-world benchmarks confirm that Redis-based rate limiters handle 25,000 req/s per shard, local limiters handle 500,000 req/s per core, and hybrid approaches provide the best balance of latency and consistency.
