# JWT -- Challenges

This page contains eight challenges organized across three tiers. Each challenge
simulates a real interview scenario. You should attempt each challenge under the
stated time limit before reading the model answer.

---

## Tier 1 -- Foundation

These challenges test your understanding of core JWT concepts. Each should take
no more than five minutes to complete.

---

### Challenge 1: JWT Structure

**Time limit:** 5 minutes

**Prompt:**

A teammate who is new to JWTs asks you to explain what a JWT actually is. They
have seen tokens that look like
`eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c`
and want to understand each part, what format each part uses, and why the parts
are separated by dots.

**Evaluation criteria:**

- You identify the three parts: header, payload, signature
- You explain that each part is Base64URL-encoded (not Base64) and padding
  characters are stripped
- You describe the header as containing the algorithm (`alg`) and token type
  (`typ`)
- You describe the payload as containing claims (registered, public, private)
- You explain that the signature is computed over `header.payload` using the
  algorithm specified in the header
- You note that the payload is not encrypted -- anyone with the token can
  decode and read the claims
- Bonus: you explain why Base64URL is necessary (URL safety, no + or /)

**Common mistakes:**

- Claiming that JWTs are encrypted by default
- Confusing Base64URL with standard Base64
- Omitting that the payload is human-readable when decoded
- Stating that the signature covers only the payload

**Model answer:**

A JWT consists of three Base64URL-encoded segments separated by dots:
`Header.Payload.Signature`.

**Header:** A JSON object specifying the signing algorithm and token type, e.g.,
`{"alg": "HS256", "typ": "JWT"}`. The `alg` field tells the verifier which
algorithm was used to produce the signature.

**Payload:** A JSON object containing claims. Registered claims are predefined
standard claims (`iss`, `sub`, `aud`, `exp`, `nbf`, `iat`, `jti`). Public
claims are registered in the IANA registry or use collision-resistant names.
Private claims are custom claims agreed upon between issuer and audience. The
payload is NOT encrypted -- anyone holding the token can Base64URL-decode it.

**Signature:** Computed by applying the algorithm named in the header to the
string `base64URL(header) + "." + base64URL(payload)`. For HMAC-SHA256, this
is `HMACSHA256(base64URL(header) + "." + base64URL(payload), secret)`. The
signature authenticates that the header and payload have not been tampered with.

**Base64URL encoding:** Standard Base64 uses `+`, `/`, and `=` characters that
are not URL-safe. Base64URL replaces `+` with `-`, `/` with `_`, and strips
`=` padding, ensuring the token can appear in URLs and HTTP headers without
escaping.

### Challenge 2: Signature Verification

**Time limit:** 5 minutes

**Prompt:**

You are implementing a JWT verification library for a microservice. Walk
through exactly what steps your code must perform to verify a JWT. Cover both
HMAC (symmetric) and RSA/ECDSA (asymmetric) signatures. Explain the pitfalls
your code must guard against.

**Evaluation criteria:**

- You enumerate the verification steps: decode header, check algorithm, fetch
  key, recompute signature, compare
- You explain that HMAC uses a shared secret, RSA/ECDSA uses the public key
- You describe timing-safe comparison -- do not use string equality
- You explain that the `alg` header must be validated against an expected list
- You mention the `none` algorithm attack and algorithm confusion attack
- Bonus: you discuss `kid` header and path traversal risks

**Common mistakes:**

- Using `==` for signature comparison (timing attack vulnerability)
- Accepting the `alg` field without server-side whitelist validation
- Not checking that the algorithm matches the key type
- Forgetting to validate claims after signature verification
- Hardcoding key resolution without a proper key management layer

**Model answer:**

JWT verification proceeds in five phases:

**Phase 1 -- Decode the header.** Base64URL-decode the first segment and
extract the `alg` field.

**Phase 2 -- Validate the algorithm.** Compare `alg` against a server-side
whitelist. Reject `none` (empty signature -- an attacker can craft unsigned
tokens). Reject symmetric algorithms (HS256) if your service uses asymmetric
keys -- an attacker who obtains your public key can sign a token with HS256
using the public key as the HMAC secret (algorithm confusion attack).

**Phase 3 -- Resolve the key.** If the header contains a `kid` (key ID), use
it to look up the correct key. Guard against path traversal in `kid` (e.g.,
`../../etc/passwd`) and SQL injection. For HMAC, use the shared secret. For
RSA/ECDSA, use the issuer's public key.

**Phase 4 -- Recompute and compare.** Recompute the signature and compare
using a timing-safe function (`hmac.compare_digest` in Python). A timing-safe
comparison checks every byte regardless of where the first mismatch occurs,
preventing attackers from learning how many bytes matched.

**Phase 5 -- Validate claims.** After signature verification: check `exp`
(current time before expiry), `nbf` (current time after not-before), `iss`
(matches expected issuer), `aud` (includes your service), and optionally `jti`
(denylist check).

### Challenge 3: Refresh Tokens

**Time limit:** 5 minutes

**Prompt:**

Your team uses short-lived access tokens (15 minutes) and longer-lived refresh
tokens (7 days). A product manager asks why you cannot just use a single token
that lasts 7 days. Explain the security benefits of the two-token approach, how
refresh tokens are typically stored and validated, and what happens during the
refresh flow.

**Evaluation criteria:**

- You explain that short-lived access tokens limit the blast radius of a leak
- You describe the refresh token as a revocable, server-stored credential
- You explain that refresh tokens are typically opaque (not JWTs) and stored
  server-side
- You describe rotation: the old refresh token is invalidated on refresh
- You mention that refresh tokens are sent only to the authorization server
- Bonus: you discuss refresh token reuse detection as a theft signal

**Common mistakes:**

- Claiming that refresh tokens are encrypted (they are not necessarily)
- Stating that refresh tokens never expire
- Confusing refresh tokens with access tokens (refresh tokens are opaque)
- Omitting the rotation mechanism
- Not addressing the storage requirement (server-side state)

**Model answer:**

The two-token model provides layered security:

**Access token (short-lived, 15 minutes):** A JWT sent with every API request.
If leaked, the attacker can act as the user for at most 15 minutes. Access
tokens are stateless -- resource servers validate them without a database
lookup.

**Refresh token (long-lived, 7 days):** An opaque string stored server-side in
a database or Redis, associated with the user ID, device fingerprint, and
expiration. Because it is opaque and server-stored, it can be individually
revoked. It is sent only to the authorization server, reducing exposure.

**Refresh flow:** The client sends the refresh token to the authorization
server, which validates it against the store, then issues a new access token
and a NEW refresh token (rotation). The old refresh token is invalidated. If
an attacker steals and uses a refresh token, the legitimate user's next refresh
attempt fails because the old token is already consumed. The server detects
this as a token reuse event and can revoke the entire token family.

**Blast radius comparison:** A single 7-day access token grants 7 days of
access on leak and cannot be revoked without a denylist. The two-token model
limits the leak window to 15 minutes and provides a revocable fallback.

### Challenge 4: Token-Based Auth for Microservices

**Time limit:** 15 minutes

**Prompt:**

Design a JWT-based authentication system for a microservices architecture with
15 services. The system must support service-to-service authentication in
addition to user authentication. You have 500,000 users, 2,000 requests per
second through the gateway, and 5,000 internal service-to-service calls per
second. Token validation must add less than 5ms of latency.

**Key decisions you must make:**

- User authentication: JWT access token from the auth service after credential
  verification, containing user ID, roles, and session ID
- Service authentication: separate mechanism for service-to-service calls
  (client credentials grant or mTLS)
- Token propagation: gateway validates the user token and forwards claims
- Token format: RS256 (RSA) so services verify with public key only
- Key distribution: JWKS endpoint, cached by each service

**Model answer:**

The system uses two token types. **User tokens** are RS256-signed JWTs
containing `sub`, `roles`, `sid`, `iat`, `exp`, `iss`, and `aud`. **Service
tokens** are issued via the client credentials grant -- each service has a
client ID and secret and receives a service JWT scoped to its responsibilities.

**Validation strategy -- tiered trust:**

- **API gateway:** validates every incoming user JWT against the cached JWKS.
  Checks `exp`, `iss`, `aud`. Passes user claims as trusted headers to
  downstream services (X-User-Id, X-User-Roles).
- **Critical services (payment, order):** re-validate the user JWT on every
  request. They do not trust the gateway's forwarded headers alone.
- **Non-critical services (notification, analytics):** trust the gateway's
  forwarded headers, reducing validation overhead.

**JWKS endpoint:** The auth service exposes `GET /.well-known/jwks.json`
returning a JSON Web Key Set with the current public keys. Each key includes a
`kid` matching the `kid` in the JWT header. Services cache the JWKS and
refresh every hour, or on demand when encountering an unknown `kid`.

**Why RS256 over HS256:** With HS256, every service that validates tokens must
possess the shared secret. If any service is compromised, all tokens can be
forged. With RS256, only the auth service holds the private key. Services hold
only the public key, which cannot sign tokens.

### Challenge 5: Key Rotation

**Time limit:** 15 minutes

**Prompt:**

Your JWT signing key was compromised. You need to rotate the key immediately
while keeping all currently signed sessions valid. Design a key rotation
strategy that supports both planned (scheduled) and emergency (compromise)
rotation. You have 500K active sessions, 2,000 tokens issued per minute, and
10 microservices that validate tokens. Maximum acceptable downtime: 0 seconds.

**Key decisions you must make:**

- Key storage: HSM or secrets manager (AWS KMS, HashiCorp Vault)
- Key identifier: each key pair has a unique `kid` in the JWT header
- JWKS management: JWKS endpoint returns multiple keys; retired keys remain
  for the maximum token lifetime so old tokens can still be verified
- Rotation schedule: planned every 90 days; emergency on demand
- Grace period: old keys remain valid for verification for the remaining
  lifetime of issued tokens

**Model answer:**

Each key pair transitions through three states: **Active** (signing and
verification), **Grace** (verification only, no signing), and **Retired**
(removed from JWKS).

**Planned rotation:** Generate a new key pair and add its public key to the
JWKS alongside the current key. Switch the signing key reference to the new
key. New tokens are signed with the new `kid`. Old tokens with the old `kid`
are still verifiable. After 7 days (maximum token lifetime), remove the old
key from the JWKS.

**Emergency rotation (compromise):** Generate a new key and set it as the
active signing key. Add both keys to the JWKS. Continue verifying tokens
signed with the compromised key for a grace period of 15 minutes (access token
lifetime). After 15 minutes, remove the compromised key. Invalidate all
existing refresh tokens and force re-authentication. For severe compromise
where the private key was exfiltrated, skip the grace period -- remove the
compromised key immediately and invalidate all sessions.

**Client-side strategy:** Services should not cache the JWKS indefinitely.
On receiving a JWT with an unknown `kid`, fetch the JWKS synchronously, cache
the new key, and retry verification. This ensures seamless rotation.

### Challenge 6: Refresh Token Rotation

**Time limit:** 15 minutes

**Prompt:**

Design a refresh token rotation system that detects and responds to token
theft. When a stolen refresh token is used by an attacker, and later the
legitimate user attempts to refresh, the system should detect the theft and
take action. You have 500K active refresh tokens and ~50,000 refresh requests
per hour.

**Key decisions you must make:**

- Rotation model: issue a new refresh token with every refresh, invalidate the
  previous one atomically
- Reuse detection: if a consumed token is presented again, detect the theft
- Response: revoke the entire token family, force re-authentication
- Token family: all tokens in the same refresh chain share a family ID
- Storage: database with columns for token hash, family ID, user ID, status

**Model answer:**

**Token family model:** Each refresh token belongs to a family created on
initial authentication. Every rotation creates a new token with the same family
ID but a different token value.

```
Initial login:
  Family ID: fam_a1b2c3
  Token 1:   hash(token_1) -> status=active, family=fam_a1b2c3

First refresh:
  Old token: status=consumed
  New token: status=active, family=fam_a1b2c3

Theft scenario:
  Attacker presents token_2 -> creates token_3, consumes token_2
  User presents token_2 (consumed) -> reuse detected!
  Family fam_a1b2c3 -> status=revoked (both parties locked out)
```

**Database schema:**

```sql
CREATE TABLE refresh_tokens (
    token_hash    VARCHAR(64) PRIMARY KEY,
    family_id     VARCHAR(32) NOT NULL,
    user_id       VARCHAR(32) NOT NULL,
    device_fp     VARCHAR(64),
    created_at    TIMESTAMP NOT NULL,
    expires_at    TIMESTAMP NOT NULL,
    status        ENUM('active', 'consumed', 'revoked') NOT NULL,
    INDEX idx_family (family_id)
);
```

**Reuse detection:** When a consumed token is presented, the server transitions
all tokens in the same family to `revoked` and returns HTTP 401 with a
`token_reuse` error. The legitimate user is forced to re-authenticate. The
server may send a notification to the user.

**Race condition handling:** Concurrent refresh requests from the same client
can both present the same token, causing a false positive. Mitigations include
a short reuse window (only trigger revocation if the token was consumed > N
seconds ago), a distributed lock on the family ID, or device fingerprint
matching (replay from the same device is a race; from a different device it is
theft).

### Challenge 7: Fintech Token System with Instant Revocation

**Time limit:** 30 minutes

**Prompt:**

Design a JWT-based token system for a fintech application processing
high-value transactions (up to $10 million per transaction). The system must
support instant token revocation -- when a session is terminated, all tokens
must be invalidated within 100 milliseconds. The system handles 10,000 users
and 500 transactions per second.

**Dimensions to explore:**

- **Stateless vs. stateful JWTs:** Pure stateless JWTs cannot be revoked
  instantly. You must introduce a stateful component like a denylist.
- **Denylist architecture:** Design a denylist checked on every request that
  stays under 1ms per check. Redis with an in-memory cache layer is a
  candidate, but handle cache staleness.
- **Transaction binding:** The token must be bound to specific transaction
  details (amount, payee, account) to prevent a stolen token from authorizing
  a different transaction.
- **Short-lived tokens with frequent refresh:** Issue 1-minute access tokens
  and refresh every minute. The refresh validates the session is still active,
  approximating instant revocation.
- **Audit trail:** Every token issuance, validation, and revocation must be
  logged with sub-millisecond precision.

**Anti-patterns to avoid:**

- Relying solely on stateless JWTs with no revocation mechanism
- Single shared Redis instance without replication
- Storing transaction details in the JWT payload (visible to anyone)
- Asynchronous revocation without a synchronous fallback

**Model answer (one valid approach):**

**Hybrid token design:** Short-lived access tokens (1 minute, RS256) combined
with a high-performance denylist.

The access token contains `sub`, `role`, a token hash (`th`), `iat`, `exp`,
`iss`. The token hash is SHA-256 of the JTI, used as the denylist lookup key.

**Denylist architecture (Redis Cluster + local cache):**

When revocation is triggered, the auth server writes the token hash to a Redis
cluster (3 masters, 2 replicas each, across 3 AZs). It then publishes the
revocation event via Redis pub/sub. Each resource server runs a local in-memory
denylist cache (100 microsecond TTL) populated from the pub/sub channel.

```
Revocation flow:
  1. Auth server receives revocation signal
  2. Auth server writes token hash -> Redis SET with TTL = 60 seconds
  3. Auth server publishes -> Redis PUBLISH "revoke:token" "{hash}"
  4. Resource servers receive event via pub/sub subscriber
  5. Each adds hash to local LRU cache (100ms TTL)
  6. Next request with this token -> check local cache -> reject
```

**Validation path:** Decode JWT, verify signature (1.5ms). Check local
denylist cache (0.01ms) -- reject if found. Optionally check Redis as
fallback (1ms). Validate `exp`, `iss`, `aud`. For high-value transactions,
perform a transaction binding check.

**Transaction binding:** For transactions over a threshold, require a
transaction-specific authorization code. The client requests a binding nonce
from the auth server, which is included in the transaction request header. The
gateway validates the nonce against the transaction details. The binding nonce
is single-use and expires in 30 seconds. A stolen access token alone cannot
authorize a transaction.

**Failover:** If a resource server loses connection to Redis, it continues
using its local cache. Cache entries expire after 1 minute, so a revoked token
would be accepted for at most 1 minute. The local cache is also refreshed by
polling Redis every 100ms as a backup to pub/sub.

### Challenge 8: Cross-Domain SSO with JWT

**Time limit:** 30 minutes

**Prompt:**

Design a cross-domain SSO system using JWTs that allows a user to authenticate
once and access applications on three different domains:
`app.sypher.io`, `dashboard.sypher.com`, and `admin.sypher.net`. Cookies
cannot be shared across these top-level domains. The system supports 1 million
users and integrates with 50 internal applications.

**Dimensions to explore:**

- **Session management:** Cookies cannot cross domains, so the SSO session
  must be managed at a central IdP on its own domain.
- **Token transport:** Use the OAuth 2.0 authorization code flow with PKCE.
  The IdP redirects to the application with an authorization code, which is
  exchanged for tokens via a backchannel.
- **Silent authentication:** For already-authenticated users, use an invisible
  iframe on the IdP domain with `prompt=none` to avoid full page redirects.
- **Cross-domain logout:** Each application registers a logout URL with the
  IdP. On logout, the IdP redirects through each application's logout URL.
- **Token scoping:** Each application receives a JWT scoped to its own
  permissions using the `aud` claim.

**Anti-patterns to avoid:**

- Trying to set cross-domain cookies (browsers block this)
- Passing the JWT in the URL query string (exposed in server logs)
- Using a single JWT for all applications
- Ignoring that iframes may be blocked by SameSite cookie settings
- Omitting PKCE in the authorization code flow

**Model answer (one valid approach):**

**Primary authentication (redirect-based OIDC flow):**

1. User visits `app.sypher.io` and is not authenticated.
2. The application generates a PKCE code_verifier and code_challenge, then
   redirects the user to `https://idp.sypher.io/auth` with `response_type=code`,
   `client_id`, `redirect_uri`, `code_challenge`, and `state`.
3. The user authenticates at the IdP. The IdP sets a session cookie on
   `idp.sypher.io` and redirects back to the application with the auth code.
4. The application validates `state`, then exchanges the code + code_verifier
   for tokens via a server-to-server POST to the IdP's token endpoint.
5. The IdP returns an access token (JWT scoped to `app.sypher.io`), an ID
   token, and a refresh token.

**Silent authentication (iframe-based):**

For users already authenticated at the IdP, the application loads a hidden
iframe pointing to `https://idp.sypher.io/silent-auth?prompt=none`. The IdP
checks its session cookie -- if valid, it redirects the iframe with an auth
code; if not, it returns an error and the application falls back to the full
redirect flow. The IdP session cookie must use `SameSite=None; Secure` to be
sent in the iframe context.

**Cross-domain logout:** Each application registers a logout endpoint. On
logout, the application clears its local session and redirects to the IdP's
logout endpoint. The IdP clears its session cookie and redirects through each
registered application's logout URL in sequence, clearing all sessions.

**Per-application scoping:**

```
Application A token: aud=app.sypher.io, scope="read:profile write:orders"
Application C token: aud=admin.sypher.net, scope="read:users write:users delete:users"
```

The IdP issues a separate access token per application. Each resource server
validates that its own `aud` value is present. A dashboard token cannot be
used against the admin API.

**Session hierarchy:** The IdP session (cookie-based, `idp.sypher.io`) outlives
individual application sessions. Each application maintains its own session
with an httpOnly cookie scoped to its own domain. Application sessions can
have independent timeouts -- the admin app may expire sooner than the standard
app.

**Security considerations:** All redirect URIs must be pre-registered and
validated exactly. PKCE is mandatory even for confidential clients. The
authorization code is single-use and expires in 60 seconds. The `state`
parameter prevents CSRF on the redirect callback. Refresh tokens should have
a limited lifetime and require re-authentication for high-value operations.
