## The Problem Story

A social media startup launches with a simple authentication system. When a user logs in, the server generates a random 128-character string (an opaque session token), stores it in a database table along with the user ID and expiry timestamp, and sends the token to the client. Every subsequent request includes this token. The server looks up the token in the database on every request to find the user. For the first 1,000 users, this works fine. The database handles 20 token lookups per second with 5ms latency.

The startup grows. Three months later, 500,000 users are active. The authentication database handles 5,000 lookups per second. The database CPU sits at 60%. At peak hours, 10,000 lookups per second push CPU to 95%. Response time for auth lookups climbs from 5ms to 120ms. The team adds read replicas. Three replicas buy them six more weeks. Then the write volume from new sign-ups and token creation saturates the primary database's write capacity. The authentication service now accounts for 40% of total database queries across the entire infrastructure. Adding features becomes harder because every microservice needs to call the auth database to verify the user.

A fintech company hits the same wall at a different scale. Their compliance requirements mandate that every API request carry a verifiable user identity. With 50 million monthly active users and 200 internal microservices, each request travels through 3 to 5 services on average. Every hop calls the auth database to verify the token. The auth database handles 2.5 billion lookups per day. The company runs 24 large database instances just for token verification. The monthly infrastructure cost for authentication exceeds $180,000. The engineering team spends two quarters designing a replacement.

An IoT platform faces a different version of the same problem. Their devices send telemetry data every 30 seconds. Each device authenticates with an API key stored in a database. When a firmware update triggers 200,000 devices to reconnect simultaneously, the auth database receives 200,000 lookup requests in under 3 seconds. The database connection pool (configured for 500 connections) exhausts immediately. Devices that cannot authenticate cannot send telemetry. The platform loses 12 hours of sensor data from a critical industrial installation.

Every one of these stories shares the same root cause: the token requires a database lookup to be useful. The token itself carries no information. It is a pointer, not a statement. The difference between a system that authenticates 100 users and a system that authenticates 100 million users is often not a bigger database. It is a token that does not need a database at all.

## The Insight

JWT (JSON Web Token, pronounced "jot") is a compact, URL-safe token format that encodes claims as a JSON object and cryptographically signs the entire payload so that any service can verify the token's authenticity without contacting a central database. The token IS the data.

Think of JWT as a stamped passport. When you travel internationally, your passport contains your name, photo, nationality, and expiry date directly on the page. The immigration officer at the border checks that the passport has a valid stamp from your home country. They do not call your home country to ask who you are. The stamp (the signature) proves the passport is genuine. The data on the passport page tells the officer everything they need to know. JWT works the same way: the token carries the user's identity and permissions as signed claims, and any service with the verification key can trust those claims without calling a central database. The passport does not contain a pointer to a database row. The passport contains the identity itself.

## Core Vocabulary

**JWT (JSON Web Token):** A compact, URL-safe token format consisting of three base64url-encoded segments separated by dots: header.payload.signature. The token encodes claims as JSON and signs them cryptographically. JWTs are used for authentication, authorization, and secure information exchange between parties.

**Header:** The first segment of a JWT. A JSON object that specifies the signing algorithm (alg) and the token type (typ). Example: `{"alg": "HS256", "typ": "JWT"}`. The header tells the verifier how the signature was computed so the verifier can choose the correct verification method.

**Payload:** The second segment of a JWT. A JSON object containing claims — statements about the user and additional metadata. Claims are divided into registered claims (standardized keys like sub, iss, exp, iat), public claims (custom keys defined by the issuer), and private claims (agreed upon between issuer and consumer). The payload is base64url-encoded, not encrypted — anyone who captures the token can read the claims.

**Signature:** The third segment of a JWT. A cryptographic value computed by applying the algorithm specified in the header to the base64url-encoded header and payload concatenated with a dot, using a secret key (for HMAC) or a private key (for RSA/ECDSA). The signature proves the token has not been tampered with and, for asymmetric algorithms, proves the identity of the issuer.

**Claims:** The individual key-value pairs in the JWT payload that make assertions about the user. Standard registered claims include sub (subject — the user identifier), iss (issuer — who created the token), exp (expiration — when the token expires), iat (issued at — when the token was created), aud (audience — who the token is intended for), and jti (JWT ID — a unique identifier for this specific token). Custom claims might include role, permissions, tenantId, or plan.

**HS256:** HMAC with SHA-256, a symmetric signing algorithm. The same secret key is used to sign and verify the token. Fast and simple, but the signing key must be shared between the issuer and all verifiers. Anyone who knows the key can create valid tokens. HS256 is suitable for single-service architectures where the issuer and verifier are the same party.

**RS256:** RSA with SHA-256, an asymmetric signing algorithm. A private key signs the token, and a separate public key verifies it. The private key stays with the issuer only. Any service can verify tokens using the public key without being able to create new tokens. RS256 is the standard choice for multi-service architectures and third-party integrations.

**Token Expiry:** The lifetime of a JWT before it ceases to be valid, encoded in the exp (expiration) claim. Short-lived access tokens (5-15 minutes) limit the damage if a token is stolen. Long-lived refresh tokens (days or months) allow the client to obtain new access tokens without re-entering credentials. Token expiry is a critical security property that requires careful tuning: too short and users re-authenticate constantly; too long and stolen tokens remain valid for an extended window.

**Refresh Token:** A separate, long-lived token that the client uses to obtain new access tokens without requiring the user to log in again. The refresh token is typically stored in a database (making it opaque, not a JWT), and it can be revoked server-side. The access token is short-lived (stateless, no revocation). The refresh token is long-lived (stateful, revocable). This pairing balances security and user experience.

**JWS (JSON Web Signature):** The standard that defines how JWT signatures are represented. JWT is technically a subset of JWS. When people say "JWT," they almost always mean a JWS — a signed token. A JWS represents the signature as a base64url-encoded binary value. The JWS specification also defines serialization formats (compact and JSON) and algorithms for signing.

**JWK (JSON Web Key):** A JSON data structure that represents a cryptographic key. JWK sets (JWKS) are used by identity providers to publish their public keys at a well-known URL endpoint (typically `/.well-known/jwks.json`). Services fetch the JWKS, find the key matching the kid (key ID) in the JWT header, and use it to verify the signature. JWKS enables key rotation without disrupting running services.

### Terminology in Practice

These terms appear in every architecture review, security audit, and authentication design discussion. Here is how they show up in real conversations:

"JWTs are stateless. When a request arrives at the API gateway, the gateway decodes the JWT, extracts the sub claim, and forwards the user context as an internal header. No database call, no cache lookup, no network hop to an auth service. The token itself is the source of truth."

"We use RS256 because we have 15 microservices that all need to verify tokens. Only the auth service holds the private key. The other 14 services fetch the public key once from the JWKS endpoint and cache it for 24 hours. If we used HS256, every service would need the same shared secret, and rotating the secret would require updating 15 services simultaneously."

"The access token expires in 15 minutes. If a token is stolen, the attacker can impersonate the user for 15 minutes maximum. The refresh token lives in the database and can be revoked immediately. When we detect a breach, we invalidate all refresh tokens for the affected users, and their next refresh attempt fails."

## The Big Picture

A JWT is not a single opaque string. It is three distinct components, each with a specific role. The structure is always: `base64url(header).base64url(payload).base64url(signature)`. Three segments separated by two dots.

The issuer constructs the JWT by serializing the header and payload as JSON, base64url-encoding each, concatenating them with a dot, then signing the resulting string. The signature is itself base64url-encoded and appended as the third segment. The final token is a printable, URL-safe string that any service can parse without a network call.

The verification flow reverses this process. The verifier splits the token by the dots, base64url-decodes the header to read the algorithm, recomputes the signature using the same algorithm and the expected key, and compares the computed signature to the received signature. If they match, the payload is authenticated. If they differ, the token is rejected.

The difference in cost is dramatic. An opaque token requires a database query on every request. At 10,000 requests per second with 5ms per query, that is 50,000ms of database time per second — the database needs 50 threads just for auth. A JWT verification at 0.1ms per request costs 1,000ms of CPU time total — one core handles all 10,000 verifications. The database load drops to zero for authentication. The savings compound as the system grows: no read replicas for auth, no cache warming, no connection pool tuning for token lookups.

## How It Actually Works — Step by Step

Understanding JWT means understanding the full lifecycle: how a token is created, how it travels from client to server, how it is verified, how it expires, and how it is replaced. Each step reveals a design decision with security and performance implications.

### Step 1: User Submits Credentials

The client collects the user's credentials — typically an email and password, but possibly an API key, OAuth token, or biometric signature. The client sends a POST request to an authentication endpoint with the credentials in the request body. The transport must be HTTPS — sending credentials over HTTP means anyone on the network can read them in plaintext.

```
POST https://auth.example.com/login
Content-Type: application/json

{
  "email": "alice@example.com",
  "password": "s3cret!"
}
```

The client never constructs or sees a JWT at this stage. The client's only job is to deliver credentials to the authentication server and receive the token in response.

### Step 2: Server Validates Credentials

The authentication server receives the login request. It looks up the user by email in the user database, retrieves the stored password hash (salted and hashed with bcrypt, argon2, or PBKDF2 — never stored as plaintext), and compares the hash of the provided password against the stored hash.

If the credentials match, the server proceeds to token creation. If they do not match, the server returns HTTP 401 Unauthorized with an error response. The server must not reveal whether the email exists or the password was wrong — a generic "invalid credentials" message prevents attackers from enumerating valid user accounts.

### Step 3: Server Constructs the JWT Header

With the user authenticated, the server builds the first segment of the JWT: the header. The header is a JSON object that specifies the signing algorithm and optionally a key ID (kid) for key rotation.

```json
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-2026-07"
}
```

The alg field tells any verifier how to check the signature. The typ field identifies the token as a JWT. The kid field enables the verifier to select the correct public key from a set of rotating keys. The server serializes this JSON and base64url-encodes it to produce the first segment.

### Step 4: Server Constructs the JWT Payload

The server builds the second segment: the payload. The payload contains registered claims (standardized keys with defined meanings) and custom claims (application-specific data).

```json
{
  "sub": "usr_a1b2c3d4",
  "iss": "https://auth.example.com",
  "aud": "https://api.example.com",
  "iat": 1720000000,
  "exp": 1720000900,
  "jti": "tok_f6e7d8c9",
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "role": "admin",
  "permissions": ["read:users", "write:users", "delete:users"],
  "tenant_id": "tnt_xyz789"
}
```

Each claim serves a specific purpose:
- **sub (subject):** The unique identifier of the authenticated user. This is the primary claim that downstream services use to identify the user. It should be an immutable, internal ID — never an email or username that could change.
- **iss (issuer):** The identity of the token issuer. In a multi-tenant system or when accepting tokens from multiple providers, this claim tells verifiers which issuer's keys to use for verification.
- **aud (audience):** The intended recipient of the token. A token issued for the payments API should be rejected by the user management API. The verifier checks that its own identifier matches the aud claim.
- **iat (issued at):** The Unix timestamp when the token was created. Used for token age tracking and to reject tokens that were issued before a key rotation.
- **exp (expiration):** The Unix timestamp when the token expires. Verifiers MUST reject tokens with an exp time in the past. This is the single most important security claim in a JWT.
- **jti (JWT ID):** A unique identifier for this specific token. Used for token replay detection and to support token revocation lists on the server side.

Custom claims carry application context. The user's display name, email, role, permissions, and tenant information are common custom claims. The server serializes this JSON and base64url-encodes it to produce the second segment.

**Critical security note:** The payload is base64url-encoded, not encrypted. Anyone who intercepts the token can decode the payload and read every claim. Never put sensitive data in the payload — no passwords, no credit card numbers, no personal identifiable information that you would not want exposed. The signature proves the payload has not been tampered with, but it does not hide the contents.

### Step 5: Server Creates the Signature

The server takes the base64url-encoded header, appends a dot, appends the base64url-encoded payload, and signs the resulting string using the algorithm specified in the header.

The signature is the heart of JWT security. Without a valid signature, an attacker could modify the payload — change the sub from "user_123" to "user_admin" — and the verifier would have no way to detect the tampering. The signature binds the header and payload together cryptographically: any change to the header or payload invalidates the signature.

### Step 6: Server Returns the JWT to the Client

The server constructs the final JWT by concatenating the three base64url-encoded segments with dots, then returns it in the HTTP response. The response also includes a refresh token and, optionally, the token expiry time for the client to manage its own token lifecycle.

```
HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyXzEyMyJ9.Mko0MlBp...",
  "refresh_token": "rft_a1b2c3d4e5f6...",
  "token_type": "Bearer",
  "expires_in": 900
}
```

The client receives the response and extracts the access_token. The client never needs to decode or inspect the token content — the token is opaque to the client during transport. The client simply stores it and sends it on subsequent requests.

### Step 7: Client Stores the Token

The client stores the JWT in a location that balances security and usability. Three storage options exist, each with trade-offs:

**Browser localStorage:** The simplest option. The JavaScript application reads the token from localStorage and attaches it to every request. Vulnerable to cross-site scripting (XSS) attacks — if an attacker injects JavaScript into the page, they can read localStorage and steal the token. Do not use localStorage for tokens in a browser context unless XSS is provably impossible.

**Secure, HttpOnly cookie:** The server sets the token as a cookie with the Secure flag (only sent over HTTPS), the HttpOnly flag (cannot be read by JavaScript), and SameSite=Strict or Lax (prevents CSRF). The browser automatically sends the cookie with every request to the same domain. Immune to XSS (JavaScript cannot read the cookie) but vulnerable to cross-site request forgery (CSRF) if SameSite is not properly configured. This is the recommended storage for browser-based applications.

**In-memory storage:** The token lives in a JavaScript variable or closure. On page reload, the token is lost and must be re-acquired (either from a refresh token or by redirecting to the login page). Immune to XSS and CSRF because the token is never persisted. The user must re-authenticate on every page reload, which degrades the user experience.

For server-to-server communication, the JWT is typically stored in the client service's memory or a shared cache, and the storage decision is simpler because there is no browser security model to worry about.

### Step 8: Client Sends the JWT on Every Request

On every subsequent API request, the client attaches the JWT in the Authorization header using the Bearer scheme. The header value is always `Bearer <token>`.

```
GET https://api.example.com/v2/users
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyXzEyMyJ9.Mko0MlBp...
```

The Authorization header is the standard convention for carrying credentials in HTTP. The Bearer scheme means "the bearer of this token is authorized to access the resource." The token is sent on every request, including POST, PUT, PATCH, and DELETE operations. For GET requests that include query parameters, the token goes in the header, not in the URL — URLs are logged by servers, proxies, and analytics tools, and a token in a URL would be captured in those logs.

The client never sends the refresh token with API requests. The refresh token is used only in the dedicated refresh flow (step 11).

### Step 9: Server Verifies the JWT

The receiving service (API gateway, microservice, or backend) verifies the JWT before processing the request. Verification follows this sequence:

1. Split the token by dots. Assert exactly three segments. Reject tokens with two or four segments.
2. Base64url-decode the header segment. Parse the JSON. Extract the alg value.
3. Base64url-decode the payload segment. Parse the JSON. Extract the exp value.
4. Check that the current Unix timestamp is before the exp timestamp. If expired, reject with HTTP 401.
5. Check the nbf (not before) claim if present. If the current timestamp is before nbf, reject.
6. Check the iss claim against the expected issuer. If the issuer is unknown, reject.
7. Check the aud claim against the service's own identifier. If the audience does not match, reject.
8. Fetch the verification key. For RS256: fetch the public key from the JWKS endpoint (cached locally). For HS256: use the shared secret from the environment configuration.
9. Recompute the signature: take the first two segments (header.payload), apply the algorithm from the header using the fetched key, and compare the computed signature to the received signature.
10. If the signatures match, extract the claims (sub, role, permissions, custom claims) and pass them to the request context. The downstream handler reads the claims from the context without parsing the token again.
11. If the signatures do not match, reject with HTTP 401.

Verification is a pure CPU operation. No network calls, no database queries, no cache lookups for the auth data. The CPU cost is approximately 0.1ms to 0.5ms per verification, depending on the algorithm. A single core verifies 2,000 to 10,000 tokens per second.

### Step 10: Service Handles the Request with User Context

After verification, the service has access to the user's identity and permissions from the JWT claims. The service puts the claims into a request context object that every downstream handler can access without parsing the token again.

```python
def handle_get_users(request):
    user_id = request.context["sub"]
    user_role = request.context["role"]
    permissions = request.context["permissions"]

    if "read:users" not in permissions:
        return HTTP 403 Forbidden

    users = database.query("SELECT * FROM users WHERE tenant_id = ?",
                          request.context["tenant_id"])
    return HTTP 200, {"users": users}
```

The service never calls the auth service, never queries the user database to identify the user, and never checks a session store. The token itself provides all the information needed to authorize the request. This is the core benefit of JWT: authentication and authorization data travel with the request, eliminating the network and database overhead of retrieving them separately.

### Step 11: Token Expiry and Refresh

The JWT expires after the duration specified in the `exp` claim — typically 15 to 60 minutes for access tokens. When the client receives a 401 response indicating token expiry, or preferably before expiry by checking the token's `exp` claim proactively, the client uses the refresh token to obtain a new access token.

```
POST https://auth.example.com/refresh
Content-Type: application/json

{
  "refresh_token": "rft_a1b2c3d4e5f6..."
}
```

The auth server looks up the refresh token in its database. If the refresh token is valid and not revoked, the server issues a new access token (and optionally a new refresh token). The old refresh token is invalidated (rotation pattern) to prevent replay attacks.

The access token is a JWT: stateless, short-lived, not revocable before expiry. The refresh token is typically opaque and stored in a database: stateful, long-lived, revocable. ### Step 12: Token Revocation

JWTs cannot be revoked before their natural expiry because the server does not track issued tokens. If a user logs out or an admin revokes a user's access, the existing JWTs remain valid until their `exp` claim time. This is the fundamental trade-off of stateless tokens.

Four strategies exist for handling JWT revocation:

**Short expiry:** Set the access token TTL to 5-15 minutes. Even if a token is compromised, the damage window is small. The user re-authenticates or refreshes frequently. This is the most common strategy and works for 90% of use cases.

**Token blacklist:** Maintain a small, in-memory cache (Redis, Memcached) of revoked JWT IDs (jti). On every request, the verifier checks if the jti is in the blacklist. This adds a cache lookup per request, partially negating the stateless benefit of JWT. Use this only when immediate revocation is required, and keep the blacklist TTL short (matching the token's remaining lifetime).

**Refresh token revocation:** Revoke the refresh token. The user cannot obtain new access tokens, so they are locked out within the access token's remaining lifetime (5-15 minutes). This is the recommended pattern: the refresh token is the revocation point, and the access token expiry is the maximum lockout delay.

**Key rotation:** Rotate the signing keys. Any JWT signed with the old key becomes invalid after the new key is deployed. This is a drastic measure — it invalidates every token for every user — and is used only in emergency scenarios like a key compromise.

## Variants and Flavors

The JWT standard supports multiple signing algorithms, each with different security properties, performance characteristics, and operational trade-offs. The three most common algorithms are HS256, RS256, and ES256.

### HS256 — HMAC with SHA-256 (Symmetric)

HS256 uses a single shared secret key for both signing and verification. The signing algorithm computes an HMAC (Hash-Based Message Authentication Code) by hashing the signing input with the secret key using SHA-256. Verification computes the same HMAC and compares it to the received signature.

**Use HS256 when:** Your application has a single service or a small number of services controlled by the same team. The issuer and verifier are the same party. You need maximum verification speed (HS256 is faster than RS256 by approximately 10x). You are operating within a trusted network boundary where the shared secret can be securely distributed to all verifiers.

**Do NOT use HS256 when:** You have many services from different teams that need to verify tokens. You want third parties to verify tokens without sharing a secret. You operate in a zero-trust network where every service boundary is a security boundary. Any service with the HS256 secret can forge tokens, so the blast radius of a secret leak is the entire system.

### RS256 — RSA with SHA-256 (Asymmetric)

RS256 uses a public-private key pair. The private key signs tokens and stays exclusively with the issuer. The public key is distributed to all services that need to verify tokens. The private key cannot be derived from the public key, so even if a verifying service is compromised, the attacker cannot create new tokens.

**Use RS256 when:** You have multiple services that need to verify tokens independently. Your architecture includes third-party integrations where the external service needs to verify tokens from your identity provider. You want to distribute the public key through a standard mechanism (JWKS endpoint) without sharing secrets. You need to support key rotation without updating every service's configuration.

**Do NOT use RS256 when:** Your performance requirements demand sub-0.1ms verification times (RS256 is significantly slower than HS256 or ES256 due to the RSA modular exponentiation). You operate in a resource-constrained environment like IoT devices with limited CPU. RS256 verification with a 2048-bit key takes approximately 0.5ms to 1ms on modern hardware, compared to 0.01ms for HS256.

## Mental Model Check

Test your understanding of JWT concepts before moving to the deep dive.

**Question 1:** Your API service verifies a JWT and the signature matches. The exp claim says the token expires in 2 hours. The sub claim contains "user_456". Do you trust that this request is from user_456?

**Answer:** Yes, but only if you also verify the iss (issuer) and aud (audience) claims. A valid signature proves the token was signed by someone holding the private key. It does not prove that the token was intended for your service. If an attacker stole a token meant for the payments API and sent it to the user management API, the signature would still verify. Your service must check that the aud claim matches your service's identifier and that the iss claim matches a trusted issuer. Without these checks, you are vulnerable to token reuse across services.

**Question 2:** A developer stores the user's role ("admin", "user") in the JWT payload. The token is signed with RS256. Is the role trustworthy?

**Answer:** Yes, the role is trustworthy because the RS256 signature proves the payload has not been tampered with since it was signed by the private key holder (the auth server). However, the role cannot be revoked until the token expires. If an admin user is demoted to a regular user, their existing JWT with the "admin" role remains valid for the token's remaining lifetime. This is a fundamental property of stateless tokens: claims are immutable for the token's lifetime. If you need real-time role changes, use a shorter token expiry (5 minutes) or check a cache for revoked permissions on sensitive operations.

**Question 3:** Your team uses HS256 with 15 microservices. Each service has the shared secret in its environment configuration. A security audit reveals that Service C logged the secret in plaintext during a startup error. What is the impact?

**Answer:** The impact is catastrophic. Anyone with access to the shared secret can create valid JWTs for any user, with any role, for any audience. They can impersonate any user, escalate privileges to admin, and sign tokens that all 15 services will accept. The fix is immediate: rotate the secret, which requires updating the configuration in all 15 services simultaneously. During the rotation window, tokens signed with the old secret are rejected. This is the primary reason HS256 is not recommended for multi-service architectures. RS256 or ES256 would limit the blast radius to token verification capability only — the attacker could verify tokens but could not create new ones.

**Question 4:** A mobile app stores the access token in localStorage. An attacker finds an XSS vulnerability in the app. What can the attacker do with the token?

**Answer:** The attacker can read the access token from localStorage using JavaScript and send it to their own server. They can then impersonate the legitimate user for the remaining lifetime of the token (typically 15-60 minutes). They can call any API that accepts that token, including reading the user's private data, performing actions as the user, and potentially changing the user's credentials if the API allows it without the current password. This is why localStorage is not recommended for tokens in browser-based applications. The fix is to use Secure, HttpOnly cookies that cannot be read by JavaScript. If the XSS vulnerability allows the attacker to make API calls directly (not just read storage), even HttpOnly cookies are vulnerable because the attacker's injected script can make fetch requests that include the cookies automatically. In that case, the token storage mechanism does not matter — the attacker can perform actions as the user regardless.

**Question 5:** An identity provider publishes its public keys at `https://auth.example.com/.well-known/jwks.json`. Your service fetches this endpoint once and caches it indefinitely. The identity provider rotates its keys. What happens?

**Answer:** Your service continues to verify tokens using the old, cached public key. New tokens signed with the new key will fail verification because your service is using the wrong key. All users are locked out until your service refreshes its JWKS cache. The fix: cache the JWKS response for a reasonable TTL (typically 24 hours or the cache-control header from the JWKS response). Before the TTL expires, your service fetches the latest keys. If a token's kid (key ID) from the header does not match any key in your cache, fetch the JWKS immediately and retry verification. This handles both regular key rotation (cache refresh) and emergency key rotation (fetch on kid mismatch). Most JWT libraries handle this pattern automatically when configured with a JWKS endpoint and a caching policy.
