# JWT — Interview Playbook

A targeted interview guide for JWT (JSON Web Tokens) in distributed systems
contexts. Covers the question landscape, level-appropriate expectations across
L3 through L8, a structured FAANG response framework, a curated phrase bank
for articulate answers, common red flags that sink candidates, and five
drills to build fluency.

---

## Question Types

JWT questions appear in system design interviews in five distinct forms. Each
type tests a different dimension of your knowledge.

### Type 1: Conceptual Foundation

These probe whether you understand what JWT is, why it exists, and when to
choose it.

**Examples:**
- What is a JWT and what problem does it solve?
- How does JWT differ from a session cookie?
- What is the difference between a JWT and a random opaque token?
- When would you choose JWT over server-side sessions?

**What interviewers look for:** Clear, jargon-minimal explanation of the
three-part structure (header, payload, signature), the self-contained nature
of JWT, and the stateless authentication trade-off.

### Type 2: Security and Cryptography

These test your understanding of signing algorithms, key management, and
common attack vectors.

**Examples:**
- Explain the difference between HS256 and RS256. When would you use each?
- What is the "none" algorithm attack and how do you prevent it?
- How do you handle key rotation for JWTs?
- What is JWT injection and how does it work?
- How would you detect a forged JWT?

**What interviewers look for:** Deep understanding of symmetric vs asymmetric
signing, awareness of common CVEs, and practical mitigation strategies.

### Type 3: Architecture and Design

These present a system and ask you to design the authentication or
authorization layer using JWT.

**Examples:**
- Design an authentication system for a microservices platform using JWT.
- How would you handle cross-service authorization with JWTs?
- Design a token refresh flow for a mobile application.
- How would you implement JWT-based service-to-service auth?

**What interviewers look for:** End-to-end thinking from issuance through
validation, refresh, and revocation. Trade-off awareness.

### Type 4: Trade-offs and Pitfalls

These test your ability to evaluate JWT against alternatives and identify
situations where JWT is the wrong choice.

**Examples:**
- What are the downsides of JWT? When would you NOT use it?
- How do you handle JWT revocation when a user logs out or is banned?
- How does JWT size affect system performance?
- What happens when a JWT expires mid-request in a long operation?

**What interviewers look for:** Honest assessment of limitations, not just
textbook advantages. Real-world operational experience.

### Type 5: Debugging and Operations

These test your ability to troubleshoot JWT-related issues in production.

**Examples:**
- A user reports they are being logged out randomly. How do you debug?
- Your JWT validation is failing in one service but not another. Why?
- How do you monitor JWT issuance and validation in production?
- What metrics would you track for a JWT-based auth system?

**What interviewers look for:** Systematic debugging approach, logging and
observability mindset, practical tooling knowledge.

## Level Expectations

Interviewers calibrate their expectations by level. The same question about
JWT key rotation receives very different follow-ups depending on whether you
are interviewing for L3 or L7.

### L3 — Individual Contributor (Entry)

**Focus:** Correct understanding of fundamentals. Ability to implement a
standard JWT flow.

**Sample question:** "Explain how a JWT is structured and how it is verified."

**Expected answer (L3 pass):**
"A JWT has three base64url-encoded segments separated by dots: header,
payload, and signature. The header contains the algorithm and token type. The
payload contains claims like `sub`, `iat`, `exp`, and custom claims. The
signature is computed by taking the header and payload, concatenating them
with a dot, and signing with a secret key using the algorithm specified in the
header. Verification repeats the same signing process and compares the
resulting signature. If they match, the token is authentic."

**What separates a strong L3:** Mentioning that the payload is only
base64-encoded, not encrypted, so sensitive data must never be stored in it.
Showing awareness that `exp` must be checked server-side.

**What fails L3:** Confusing encoding with encryption. Thinking the payload
is secure because it is in a JWT. Not knowing how signature verification
works.

### L4 — Mid-Level

**Focus:** Understanding of trade-offs. Ability to design a simple JWT-based
system. Awareness of security concerns.

**Sample question:** "Design a JWT-based authentication flow for a web
application with a React frontend and a Go backend."

**Expected answer (L4 pass):**
"I would use RS256 signing. A trusted identity service issues JWTs after
credential validation. The JWT contains the user ID, roles, and an expiration
of 15 minutes. The React app stores the token in an HTTP-only secure cookie
or in memory — never localStorage. Every API request includes the token in
the Authorization header as Bearer. The Go backend validates the token using
the public key from the identity service, extracts claims, and uses them for
authorization. For longer sessions, I would add a refresh token with a 7-day
lifetime stored server-side or in a secure cookie."

**What separates a strong L4:** Explaining the choice of RS256 over HS256
(no shared secret across services, each service validates independently).
Discussing token storage security (XSS protection). Mentioning refresh token
rotation.

**What fails L4:** Proposing localStorage without acknowledging XSS risk.
Using HS256 in a multi-service architecture without explaining how the
secret is distributed. Omitting token expiry.

### L5 — Senior

**Focus:** Multi-service architecture. Revocation strategies. Performance
considerations. Security hardening.

**Sample question:** "Design a JWT-based authorization system for a platform
with 50 microservices. How do you handle token revocation?"

**Expected answer (L5 pass):**
"I would use RS256 with a centralized identity service that issues tokens.
Each service validates tokens locally using the public key — no network call
for validation, which keeps latency low. For revocation, I would use a
combination of short-lived access tokens (5-15 minutes) and a deny-list
checked by a thin middleware. The deny-list lives in Redis with the same TTL
as the token, so entries auto-clean. For high-priority revocation (compromised
admin account), I add the token's `jti` to the deny-list immediately. I would
also support a token blacklist that is replicated across regions. For
scenarios that need instant revocation, I would evaluate using a token
serialization pattern where the token encodes a session version number that
is checked against a database — incrementing the version invalidates all
previous tokens for that user."

**What separates a strong L5:** Discussing the trade-off between short TTLs
(UX friction) and long TTLs (revocation latency). Mentioning that a deny-list
partially defeats the stateless nature of JWT but is necessary. Quantifying
the latency impact of different approaches.

**What fails L5:** Claiming JWT revocation is "not a problem." Not
considering that a deny-list requires a network call, which adds latency.
Ignoring key rotation for signing keys.

### L6 — Staff

**Focus:** System-wide architecture. Cross-team concerns. Security
architecture. Operational excellence.

**Sample question:** "Our platform has grown to 200+ services across three
data centers. JWT validation is slowing down our request pipeline. How do you
diagnose and fix this?"

**Expected answer (L6 pass):**
"I would start by measuring. I would instrument JWT validation to track P50,
P95, and P99 latency across all services. Common causes: public key fetching
without caching, excessive base64 decoding, large tokens with too many claims,
and RS256 verification being computationally expensive. Mitigations: cache
the public key with a reasonable TTL (e.g., 1 hour) and use a background
refresh. If the token size is an issue, minimize the claims to only what each
service needs — consider a claim-mediation layer where an edge gateway
validates the token and injects only the necessary claims as a header for
downstream services. For extremely high throughput paths, I would evaluate
switching to EdDSA (Ed25519), which has faster verification than RSA. I would
also consider using a token introspection protocol at the edge instead of
every service validating individually, though this introduces a network hop.
Finally, I would implement circuit breakers around key-fetching and set up
alerts on validation failure rates."

**What separates a strong L6:** Specific performance numbers. Knowledge of
EdDSA. Understanding the claim-mediation pattern. Mentioning circuit breakers
and observability.

**What fails L6:** Suggesting removal of JWT validation to improve
performance. Not having a data-driven approach. Ignoring operational
concerns.

### L7 — Principal / L8 — Distinguished

**Focus:** Long-term strategy. Organization-wide standards. Industry trends.
Cross-cutting security architecture.

**Sample question:** "We are building a new platform-wide authentication and
authorization standard. Should we use JWT, and if so, how should we design
it?"

**Expected answer (L7 pass):**
"JWT is a good choice for our platform, but it is not a universal solution. I
would define a standard that uses JWT for service-to-service and
service-to-user auth with EdDSA signatures using a well-known JWKS endpoint.
I would mandate a minimum set of standard claims (`iss`, `sub`, `aud`, `exp`,
`iat`, `jti`) and define a review process for custom claims to prevent
token bloat. For revocation, I would standardize on a session-version pattern:
each user has a session version stored in our identity store, and tokens
include this version. On logout or privilege change, we increment the version.
This avoids a distributed deny-list while supporting instant revocation. I
would also define a migration path from legacy HMAC-signed tokens, establish
a key rotation policy with a 30-day overlap window, and create a shared
library that every service uses for validation — eliminating the variation
that causes security bugs. Finally, I would deprecate the `none` algorithm
entirely at the library level."

**What separates a strong L7:** Thinking about standardization across teams.
Establishing governance for claims. Planning migration paths. Library-level
controls.

**What fails L7:** Proposing a solution without considering organizational
adoption. Not addressing the migration from existing systems. Ignoring the
need for shared libraries and standards.

## FAANG Response Framework

Use this structured approach for any JWT system design question. It mirrors
the evaluation criteria used at FAANG companies.

### Step 1: Scope and Clarify (1-2 minutes)

Before diving into design, ask clarifying questions that demonstrate breadth
of thinking.

**Good clarifying questions:**
- "Are we designing for human users, service-to-service auth, or both?"
- "What is the expected scale — number of users, requests per second?"
- "Do we need instant revocation, or is eventual revocation acceptable?"
- "What is the security classification of the data being protected?"
- "Are there existing identity providers we need to integrate with?"
- "What compliance requirements apply (SOC2, HIPAA, PCI)?"

**Why this matters:** JWT is the wrong answer for some problems. A candidate
who asks clarifying questions before proposing JWT shows they evaluate
trade-offs rather than applying a canned solution.

### Step 2: Define Requirements (2-3 minutes)

State the requirements explicitly so the interviewer can correct any
misassumptions.

**Functional requirements:**
- User authentication and session management
- Authorization claims propagation across services
- Token issuance, refresh, and revocation
- Support for multiple client types (web, mobile, API)

**Non-functional requirements:**
- Stateless validation for low latency
- High availability (99.99%+ for auth service)
- Security: resistance to forgery, replay, and injection
- Operational: observability, key rotation, audit trail

### Step 3: High-Level Design (3-5 minutes)

Present the architecture as a flow. Interviewers want to see you trace the
complete lifecycle of a JWT from issuance to validation to expiry.

**Components to include:**
- **Identity Service (IdP):** Issues JWTs, manages keys, handles refresh
- **Client:** Stores and sends JWTs with requests
- **API Gateway / Edge:** First point of validation (optional but recommended)
- **Services:** Validate JWTs, extract claims for authorization
- **JWKS Endpoint:** Serves public keys for RS256/EdDSA verification
- **Session Store (optional):** For deny-list or session version tracking

**Flow:**
1. User authenticates with credentials → IdP validates → returns JWT + refresh
2. Client sends JWT in `Authorization: Bearer <token>` header
3. API Gateway validates signature, checks expiry, extracts claims
4. Gateway forwards request with validated claims to downstream service
5. Service uses claims for authorization decisions

### Step 4: Deep Dive (5-10 minutes)

The interviewer will choose one area to explore in depth. Be ready for each
of these deep dives.

**Deep Dive A — Signing Algorithm Selection:**
- HS256: Fast, single secret, suitable for monoliths or single-service auth
- RS256: Slower verification, public/private key, suitable for multi-service
- ES256: ECDSA, smaller signatures, faster than RSA at same security level
- EdDSA (Ed25519): Fastest verification, modern, recommended for new systems

**Deep Dive B — Revocation Strategy:**
- Short TTLs (5-15 min): Simple, no revocation mechanism needed, but UX impact
- Deny-list in Redis: Works for any TTL, but adds a network hop for validation
- Session version in token: Instant revocation, requires session store lookup
- Token serialization: Same as session version, different terminology

**Deep Dive C — Token Storage:**
- HTTP-only secure cookie: Best for web, XSS-resistant, but CSRF needs mitigations
- In-memory (JS variable): Safe from XSS, but lost on page refresh
- localStorage: Persistent, but vulnerable to XSS — avoid for sensitive apps
- Secure enclave / Keychain: Mobile best practice

**Deep Dive D — Performance Optimization:**
- Cache public keys (JWKS) with background refresh
- Minimize payload claims (keep under 1KB, ideally under 512 bytes)
- Use EdDSA for fastest verification
- Validate at edge, propagate claims as headers
- Consider token compression for bandwidth-constrained environments

### Step 5: Trade-offs and Alternatives (2-3 minutes)

Explicitly state what you are trading off and what alternatives exist.

**JWT trade-offs:**
- Stateless validation vs. inability to revoke instantly
- Self-contained vs. token size bloat
- No server-side session storage vs. larger network payloads
- Standard format vs. complexity of crypto configuration

**Alternatives to mention:**
- **Opaque tokens + introspection:** Easy revocation, but requires network call
- **Paseto:** Safer by design, fewer foot-guns, but less ecosystem support
- **Session cookies:** Simple, revocation is native, but not stateless
- **Mutual TLS (mTLS):** Strongest for service-to-service, no token needed

### Step 6: Summary and Operations (1-2 minutes)

Close with operational considerations that show production readiness.

**Operational checklist:**
- Key rotation policy (rotate signing keys every 90 days, overlap window)
- Monitoring: JWT issuance rate, validation failure rate, latency percentiles
- Alerting: Sudden spike in validation failures (potential attack)
- Audit logging: Every token issuance and validation failure
- Library standardization: One shared validation library per language
- Penetration testing: Specifically test for algorithm confusion, injection

## Phrase Bank

Use these phrases to express ideas clearly and with authority. Organized by
concept area.

### Explaining JWT Structure

1. "JWT is a self-contained token format — the token carries everything the
   verifier needs to validate it, eliminating a database lookup on every
   request."

2. "The three segments — header, payload, signature — are each base64url
   encoded and dot-separated. Only the signature provides integrity. The
   header and payload are readable by anyone who intercepts the token."

3. "The token is signed, not encrypted. Anyone can decode the payload. Always
   treat JWT payloads as public information."

### Signing and Algorithms

4. "For symmetric signing (HS256), every verifier needs the same secret. This
   works fine for a single service but becomes a key-distribution problem in
   a microservices architecture."

5. "With asymmetric signing (RS256, ES256, EdDSA), the issuer holds the
   private key and any service can validate using the public key. This is the
   standard choice for distributed systems."

6. "EdDSA with Ed25519 offers the best verification performance and security
   margin. It should be the default for new systems."

### Architecture and Design

7. "Short-lived access tokens with long-lived refresh tokens give us the best
   balance: stateless validation for the common case and the ability to
   revoke sessions at the refresh layer."

8. "The API Gateway acts as a validation boundary. We validate the JWT at the
   edge and propagate only the necessary claims as a header to downstream
   services. This keeps token validation centralized."

9. "Token size matters at scale. A 2KB token on every request adds up
   quickly. We carefully curate the claims to include only what is strictly
   necessary."

### Revocation

10. "Instant revocation is fundamentally at odds with stateless JWT. You must
    introduce some form of state — a deny-list, a session version, or a
    token serialization — and accept the trade-off."

11. "Session versioning is my preferred approach for revocation. The token
    carries a version claim, and we check the current version on each
    request. Incrementing the version instantly invalidates all tokens for
    that user."

### Security

12. "The algorithm confusion attack exploits tokens signed with a public key
    algorithm when the verifier accepts the `alg` header from the token
    itself. Always pin the expected algorithm in the validation code."

13. "JWT injection occurs when user-controlled data is embedded in the
    payload without proper validation. Always validate and sanitize claims
    before using them in authorization decisions."

14. "Never store sensitive data in a JWT payload. The payload is base64
    encoded, not encrypted. For sensitive claims, use a JWE (JSON Web
    Encryption) encrypted token."

15. "The `exp` claim must be validated against a trusted clock. NTP
    synchronization across all services is non-negotiable for JWT security."

### Trade-offs and Honest Assessment

16. "JWT trades revocation simplicity for validation simplicity. That is the
    core trade-off of the entire approach."

17. "If you need instant revocation, fine-grained access control, or are
    dealing with highly sensitive data, consider whether JWT is the right
    choice. Opaque tokens with introspection may serve you better."

18. "JWT is a standard, not a protocol. The standard leaves many decisions
    to the implementer — algorithm choice, claim validation, key management.
    Most security vulnerabilities come from implementation mistakes, not the
    spec itself."

## Red Flags

These are mistakes and warning signs that interviewers flag. Avoid them in
your answers and watch for them in system designs you review.

### Red Flag 1: "JWT is encrypted"

Saying the payload is secure because it is in a JWT. The payload is
base64-encoded, not encrypted. Anyone with the token can decode and read the
claims.

**Correct framing:** "JWT provides integrity through the signature, but the
payload is not confidential. Treat all claims as readable by the token holder
and anyone who intercepts the token."

### Red Flag 2: "No need for revocation — just use short TTLs"

Short TTLs reduce the revocation window but do not eliminate the need for
revocation. A 15-minute window is still 15 minutes during which a
compromised token is valid.

**Correct framing:** "Short TTLs minimize the revocation window. For our
use case, we pair a 15-minute access token with a revocation mechanism at
the refresh layer. For high-priority revocation, we use a session-version
pattern."

### Red Flag 3: Using HS256 in a multi-service architecture

HS256 requires every service to share the same secret. Any service that is
compromised exposes the signing key for all services.

**Correct framing:** "HS256 is appropriate for single-service auth or
internal tools. For a multi-service architecture, I would use RS256 or EdDSA
so that each service validates with a public key and the private key stays
with the issuer."

### Red Flag 4: "Store the JWT in localStorage"

localStorage is accessible to any JavaScript running on the same origin. An
XSS vulnerability leaks the token to an attacker.

**Correct framing:** "For web applications, I prefer HTTP-only secure cookies
or in-memory storage. Both are resistant to XSS-based token theft. If
localStorage is necessary, I would pair it with a Content Security Policy and
subresource integrity checks."

### Red Flag 5: Omitting the `aud` claim

Without audience validation, a token issued for one service can be used
against another. This is a common authorization bypass.

**Correct framing:** "I always include and validate the `aud` (audience)
claim. Each service checks that the token was intended for it. This prevents
token reuse across services."

### Red Flag 6: Accepting the `alg` header from the token

A token with `alg: none` or `alg: HS256` when the verifier expects RS256 can
bypass signature verification entirely.

**Correct framing:** "The verification library must pin the expected
algorithm. Never derive the algorithm from the token header. This prevents
algorithm confusion attacks."

### Red Flag 7: Ignoring clock skew

JWT validation requires comparing `exp` and `nbf` against the current time.
Services with unsynchronized clocks reject valid tokens or accept expired
ones.

**Correct framing:** "All services that validate JWTs must run NTP with
monitored synchronization. Additionally, I allow a small clock skew
tolerance (typically 30-60 seconds) in the validation logic."

### Red Flag 8: No key rotation plan

Using the same signing key for years increases the impact of a key leak.
Without a rotation plan, rotating the key breaks all existing tokens.

**Correct framing:** "I design for key rotation from day one. The JWKS
endpoint supports multiple keys identified by `kid`. The old key remains
available for the maximum token lifetime, and new tokens are signed with the
new key. Rotation happens every 90 days with a 30-day overlap window."

### Red Flag 9: JWT as a session replacement without understanding the differences

Treating JWT exactly like a server-side session, ignoring the stateless
nature and the implications for revocation and token size.

**Correct framing:** "JWT is not a session with a different format. It is a
fundamentally different approach. Sessions are stateful on the server; JWT
is stateless on the client. Each has different trade-offs for revocation,
storage, and validation."

### Red Flag 10: Overloading the JWT with too many claims

Including every possible user attribute, role, permission, and metadata in
the token. This bloats every request and increases the attack surface.

**Correct framing:** "I include only the claims that every service needs:
user ID, roles, token ID, and expiration. Service-specific authorization
data is fetched from the service's own data store or a central authorization
service."

## Drills

Five drills to build interview fluency. Each should be practiced aloud with
a timer.

### Drill 1: The Elevator Pitch (2 minutes)

**Task:** Explain JWT to a non-technical product manager in 2 minutes. No
jargon. Focus on what problem it solves and the trade-off.

**Practice script:**
"JWT is a way to prove identity across services without requiring each
service to look up your information in a database. Think of it like a
stamped, sealed envelope. The stamp shows it came from the right place
(signature). The letter inside says who you are (claims). Anyone who knows
the official stamp can verify the envelope is authentic. But the letter
inside is not hidden — anyone can read it. So we only put non-sensitive
information in there. The trade-off is that once we give you the envelope,
we cannot take it back until it expires. So we make them short-lived, and
we give you a way to get a new one when the old one expires."

**Check:** Can you explain it without using the words "base64," "header,"
"payload," or "asymmetric"?

### Drill 2: Technology Choice Justification (3 minutes)

**Task:** A startup founder asks you whether they should use JWT for their
new SaaS product. They have 3 services and 10,000 users. Give your
recommendation in 3 minutes.

**Structure:**
1. Ask clarifying questions (1 min): "Is this human users or API? Do you
   need instant revocation? What is your team's experience with security?"
2. State your recommendation (30 sec): "JWT with RS256 is a good fit for
   your scale and number of services."
3. Explain why (1 min): "Stateless validation keeps latency low, RS256 lets
   each service validate independently, and the standard tooling is mature."
4. State the trade-offs (30 sec): "You will need a key rotation plan and a
   revocation strategy. JWT does not give you instant revocation for free."

**Check:** Did you mention alternatives? Did you clearly state the trade-off?

### Drill 3: Revocation Deep Dive (5 minutes)

**Task:** Your interviewer says "Tell me everything you know about JWT
revocation." You have 5 minutes.

**Structure:**
1. Frame the problem (30 sec): "Revocation is the hardest problem with JWT.
   Stateless validation means nothing to check against. We need to introduce
   some state."
2. List approaches (3 min):
   - Short TTLs: Simplest, but only reduces the window
   - Deny-list in Redis: Works for any TTL, adds a network hop
   - Session version: Token includes a version number checked against a store
   - Token serialization: Each token has a unique ID tracked in a database
   - Blacklist at the gateway: Centralized revocation check
3. Compare and recommend (1 min):
   "For most systems, I recommend short-lived tokens (15 min) paired with
   session versioning. This gives stateless validation for the common case
   and instant revocation for critical events. The session version check is
   a single Redis lookup."
4. Edge cases (30 sec): "What about a leaked admin token? We escalate to
   immediate deny-list entry regardless of the primary strategy."

**Check:** Did you cover at least 3 approaches? Did you give a clear
recommendation? Did you mention the trade-off of each approach?

### Drill 4: Security Audit Review (5 minutes)

**Task:** You are reviewing a team's JWT implementation. Identify all
security issues. Present your findings.

**Scenario:**
The team uses RS256 with a 4096-bit key. The token payload includes the
user's email, full name, role, department, permissions array, and last login
timestamp. The token TTL is 24 hours. The verification library accepts the
`alg` header from the token. The public key is fetched from JWKS on every
validation. There is no `aud` claim validation. Tokens are stored in
localStorage.

**Issues to identify:**
1. Token accepted `alg` from header — algorithm confusion vulnerability
2. 24-hour TTL is too long for an access token
3. Token contains sensitive user data (email, name, permissions) in plaintext
4. localStorage is XSS-vulnerable
5. Public key fetched on every validation — major performance issue
6. No `aud` claim — token can be reused across services
7. 4096-bit RSA key may impact performance unnecessarily; 2048-bit is standard

**Practice running through all 7 issues in 5 minutes.**

**Check:** Did you catch all 7 issues? Did you prioritize them (which is
most critical, which is a best-practice concern)?

### Drill 5: JWT vs. The World (5 minutes)

**Task:** Compare JWT to three alternatives. For each, state one scenario
where you would choose the alternative over JWT.

**Structure:**
1. **JWT vs. Opaque Tokens (90 sec):**
   "Opaque tokens are random strings that require a server-side lookup to
   resolve. I would choose opaque tokens when I need instant revocation with
   no compromise — for example, a banking API where every request must check
   the current auth state. The trade-off is latency from the introspection
   call."

2. **JWT vs. Session Cookies (90 sec):**
   "Session cookies are the traditional server-side session approach. I would
   choose session cookies for a monolithic application where all requests go
   to the same server. The trade-off is server-side storage and the lack of
   a standard format for cross-service auth."

3. **JWT vs. Paseto (90 sec):**
   "Paseto is a newer standard that fixes many JWT foot-guns. It eliminates
   algorithm confusion by design, requires authenticated encryption, and has
   a simpler specification. I would choose Paseto for a greenfield project
   where ecosystem support is not a concern. The trade-off is smaller
   community and fewer library options."

4. **Bonus — JWT vs. mTLS (60 sec):**
   "mTLS is for service-to-service auth using TLS certificates. I would
   choose mTLS for internal service mesh communication where you want
   cryptographic identity without token management. The trade-off is
   certificate lifecycle management and unsuitability for user-facing auth."

**Check:** Did you state a clear scenario for each alternative? Did you
explain the trade-off? Did you avoid bias toward JWT?

## Common JWT Interview Question Bank

Below are ten frequently asked JWT interview questions with concise,
level-appropriate answers. Use these as a reference for the style and depth
expected at each level.

### Question 1: "How does JWT handle expiry?"

**L3 answer:** "JWT includes an `exp` claim in the payload. The verifier
checks the current time against `exp` and rejects the token if it has
expired. The token becomes permanently invalid after expiry — there is no
way to extend it without re-authentication or a refresh token."

**L5+ extension:** "I also consider clock skew. Services without synchronized
clocks may reject valid tokens or accept expired ones. I implement a clock
skew tolerance of 30-60 seconds and ensure all services run NTP with
monitoring. For long-lived operations, I evaluate whether the token should
be validated at the start or periodically during the operation."

### Question 2: "Can you revoke a JWT?"

**L3 answer:** "Not directly. JWT is stateless — once issued, the server
does not track it. The token remains valid until it expires. The standard
way to handle revocation is to use short-lived tokens (15 minutes) and
revoke the refresh token."

**L5+ extension:** "For instant revocation, I introduce a session-version
pattern. Each user has a version number stored in a session store. The JWT
includes this version. On logout or privilege change, I increment the
version. Every request checks that the token's version matches the current
version. This keeps validation fast — a single Redis lookup — while
supporting immediate revocation."

### Question 3: "What is the difference between JWT and OAuth2?"

**L3 answer:** "JWT is a token format. OAuth2 is an authorization framework
that defines how tokens are obtained and used. JWT is often used as the
token format within OAuth2, but OAuth2 can also use opaque tokens."

**L5+ extension:** "OAuth2 defines four grant types for obtaining tokens,
token endpoint conventions, and scope-based authorization. JWT is a
bearer token format that can carry structured claims. In practice, they
are complementary: an OAuth2 authorization server issues JWTs, and resource
servers validate them. The distinction matters when designing a system —
you need both the protocol (OAuth2) and the token format (JWT)."

### Question 4: "How do you handle token refresh?"

**L3 answer:** "The client receives an access token (short-lived) and a
refresh token (long-lived). When the access token expires, the client sends
the refresh token to a token endpoint to get a new access token. The refresh
token can be revoked server-side."

**L5+ extension:** "I implement refresh token rotation — each refresh
request returns a new refresh token and invalidates the old one. This
limits the damage if a refresh token is stolen. I also detect refresh token
reuse: if a revoked refresh token is presented, it may indicate token theft,
and I invalidate all tokens for that user. The refresh token is stored in an
HTTP-only secure cookie or a secure mobile keychain, never in localStorage."

### Question 5: "What is the `jti` claim used for?"

**L3 answer:** "The `jti` (JWT ID) claim is a unique identifier for the
token. It is used to prevent replay attacks by tracking which token IDs
have already been used."

**L5+ extension:** "The `jti` claim is essential for any revocation strategy.
I use it as the key in the deny-list. When a token is revoked, I add its
`jti` to the deny-list with the same TTL as the token's `exp`, so the entry
auto-cleans. I also log `jti` values for audit trails — every validation
failure is correlated with a specific token ID. This makes debugging
revocation issues straightforward."

### Question 6: "How does JWT handle authorization vs. authentication?"

**L3 answer:** "JWT handles authentication by proving the user's identity
through the `sub` claim. It handles authorization through custom claims
like `roles` or `permissions` that are embedded in the payload."

**L5+ extension:** "I separate authentication and authorization concerns.
The JWT proves who the user is (authentication) via the signature and `sub`
claim. Authorization is derived from claims, but I am careful about claim
granularity. Fine-grained permissions (e.g., 'can edit document 42') do not
belong in a JWT — they change frequently and bloat the token. Instead, I
include coarse-grained claims (roles, scopes) and let each service
resolve fine-grained permissions from its own data store or a central
authorization service like OPA (Open Policy Agent) or Casbin."

### Question 7: "What happens if a JWT signing key is leaked?"

**L3 answer:** "If the signing key is leaked, anyone can forge valid tokens.
The immediate step is to rotate the signing key and reissue all tokens. All
tokens signed with the old key become invalid."

**L5+ extension:** "Key leak response is a security incident. I have a
runbook: (1) revoke all current tokens via session version increment; (2)
rotate the signing key immediately and update the JWKS endpoint; (3) keep
the old key available for the maximum token lifetime to avoid breaking
valid in-flight tokens; (4) audit all tokens issued near the leak time to
detect forgeries; (5) rotate all service credentials that may have been
exposed. The key rotation should be automated — manual key rotation is
error-prone and slow. I use a key management service (KMS) or a secrets
store like Vault to manage signing keys."

### Question 8: "How does JWT work in a mobile application?"

**L3 answer:** "The mobile app authenticates with the backend and receives an
access token and refresh token. The access token is stored in memory or a
secure keychain. Every API request includes the access token in the
Authorization header. When it expires, the app uses the refresh token."

**L5+ extension:** "Mobile introduces unique challenges: (1) Token storage
must use the OS keychain (iOS Keychain, Android Keystore) — not shared
preferences or local storage; (2) Biometric authentication can gate access
to the stored token; (3) Token refresh must handle concurrent requests — a
token refresh race condition can cause multiple simultaneous refresh calls;
(4) Offline support requires caching the token and handling refresh when
connectivity returns; (5) Push notifications for forced logout require a
token invalidation channel. I also consider certificate pinning for the
token endpoint to prevent interception on compromised networks."

### Question 9: "What is JWKS and why is it used?"

**L3 answer:** "JWKS (JSON Web Key Set) is a JSON structure that contains
public keys. The JWT issuer exposes a JWKS endpoint, and verifiers fetch
the public key from it to validate the token's signature."

**L5+ extension:** "The JWKS endpoint is a critical piece of the
infrastructure. I design it with caching headers (Cache-Control, ETag) so
verifiers cache the response and avoid fetching the key on every validation.
The JWKS response includes a `kid` (key ID) for each key, which maps to the
`kid` in the JWT header. This supports key rotation — multiple keys are
active simultaneously, and the token's `kid` tells the verifier which key to
use. I also add a short `max-age` for the JWKS response (e.g., 1 hour) and
a `stale-while-revalidate` directive so verifiers can use a stale key while
fetching a fresh one in the background."

### Question 10: "How do you handle JWT in a serverless architecture?"

**L3 answer:** "In serverless, each function invocation is stateless. JWT
validation happens inside the function using the public key. The token is
passed in the Authorization header from the API Gateway."

**L5+ extension:** "Serverless adds cold-start concerns. RSA signature
verification is CPU-intensive and can add 50-100ms to cold starts. I
mitigate this by: (1) using EdDSA (Ed25519) which is faster to verify; (2)
caching the public key outside the function handler (in global scope); (3)
pre-warming functions with a scheduler. The API Gateway can also validate
the JWT before the function is invoked using Lambda authorizers or custom
authorizers, which offloads the verification. For high-throughput scenarios,
I validate at the API Gateway layer and pass the decoded claims as a
header to the Lambda function, eliminating duplication of validation
logic."

## Quick Reference Card

Keep these numbers and facts in mind for quick recall during interviews.

**JWT structure:** `header.payload.signature` — three base64url-encoded
segments separated by dots.

**Standard claims:** `iss` (issuer), `sub` (subject), `aud` (audience),
`exp` (expiration), `nbf` (not before), `iat` (issued at), `jti` (JWT ID).

**Recommended TTLs:** Access token 5-15 minutes. Refresh token 7-30 days.

**Signing algorithms by use case:**
- HS256: Single service, internal tools
- RS256: Multi-service, general purpose
- ES256: Performance-sensitive, mobile
- EdDSA (Ed25519): New systems, best performance

**Key sizes:** RSA 2048-bit (standard), 4096-bit (higher security, slower).
Ed25519 uses 32-byte keys.

**Token size budget:** Aim for under 1KB. Monitor and alert on tokens over
2KB.

**Critical validation checks:**
1. Signature valid and matches pinned algorithm
2. `exp` not in the past (with clock skew tolerance)
3. `nbf` not in the future (if present)
4. `aud` matches the expected audience
5. `iss` matches the expected issuer
6. Token not in deny-list (if revocation is enabled)

**Common ports and protocols:**
- JWT in `Authorization: Bearer` header
- JWKS endpoint at `/.well-known/jwks.json`
- Token introspection at `POST /introspect` (RFC 7662)
- Token revocation at `POST /revoke` (RFC 7009)
