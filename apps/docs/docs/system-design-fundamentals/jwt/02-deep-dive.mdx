# JWT --- Under the Hood

## Why JWT Exists

A JSON Web Token (JWT) is a compact, URL-safe token format that carries claims between two parties. Unlike opaque session tokens that require a server-side lookup, a JWT is **self-contained** --- the server can verify the token and extract identity information without hitting a database.

Before diving into the mechanics, understand the core problem JWT solves: you need to prove identity across stateless HTTP requests without storing session state server-side. The token itself carries the proof.

## JWT Structure: The Three Parts

A JWT is three base64url-encoded segments separated by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
          .
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsZXgiLCJpYXQiOjE1MTYyMzkwMjJ9
          .
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

Each part is base64url-encoded (not base64). The difference: base64url replaces `+` with `-` and `/` with `_`, and strips trailing `=` padding. This makes the token safe to include in URLs and HTTP headers without additional escaping.

### Part 1: The Header

The header is a JSON object that describes the token. It tells the verifier *how* the token was constructed.

| Field | Required | Purpose |
|-------|----------|---------|
| `alg` | Yes | Signing algorithm (e.g., HS256, RS256, ES256) |
| `typ` | Convention | Token type --- typically `"JWT"` |
| `kid` | Optional | Key ID for key rotation lookups |
| `cty` | Rare | Content type for nested JWTs |

The `alg` field is the most critical. It determines the cryptographic operation used to produce and verify the signature.

```json
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "2025-q3-signing-key"
}
```

The `kid` field lets the server rotate signing keys without invalidating active tokens. More on this in the consistency section.

### Part 2: The Payload (Claims)

The payload contains **claims** --- statements about the subject and metadata about the token itself. Claims fall into three categories.

**Registered claims** (standardized by RFC 7519):

| Claim | Full Name | Purpose |
|-------|-----------|---------|
| `iss` | Issuer | Who created the token |
| `sub` | Subject | Who the token is about (usually a user ID) |
| `aud` | Audience | Who the token is intended for |
| `exp` | Expiration | Token expiry timestamp (NumericDate) |
| `nbf` | Not Before | Token is not valid before this time |
| `iat` | Issued At | When the token was issued |
| `jti` | JWT ID | Unique identifier for the token (prevents replay) |

**Public claims** --- defined by you, but should be collision-resistant. Use a namespaced URI or prefix.

**Private claims** --- agreed upon between issuer and consumer. No collision protection.

```json
{
  "sub": "user_42",
  "iss": "auth.sypher.io",
  "aud": "api.sypher.io",
  "iat": 1719876543,
  "exp": 1719880143,
  "nbf": 1719876543,
  "jti": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "roles": ["admin", "billing"],
  "tenant_id": "acme-corp"
}
```

### Part 3: The Signature

The signature is the cryptographic output that protects the token's integrity. It is computed over the header and payload:

```
signature = sign( base64url(header) + "." + base64url(payload) , key )
```

The verifier recomputes this value using the same algorithm and key. If the recomputed signature matches the received signature, the token is authentic and unmodified.

### HMAC (Symmetric): HS256, HS384, HS512

HMAC uses a single shared secret. The same key signs and verifies. This is fast (~1-5 microseconds) but creates a security problem: any service that can verify a token can also forge one.

Use HMAC only when:
- The issuer and verifier are the same service (single-process monolith)
- The secret is stored in a secrets manager and never shared broadly
- You accept that a compromised verifier can mint arbitrary tokens

Best practice: use a 256-bit (32-byte) random secret for HS256. Never use a password or low-entropy string directly.

### RSA (Asymmetric): RS256, RS384, RS512

RSA uses a public/private key pair. The issuer holds the private key (signs). Verifiers hold only the public key (verifies). A compromised verifier cannot forge tokens.

Use RSA when:
- Multiple downstream services need to verify tokens independently
- The issuer is a centralized auth service (e.g., Okta, Auth0, your own)
- Signing frequency is low relative to verification frequency

RSA verification is slow (~100-500 microseconds for 2048-bit keys) compared to HMAC, but this is acceptable because verification is still orders of magnitude faster than a database lookup.

Key sizes: 2048-bit is the minimum today. 4096-bit doubles verification time. The signature output is always 256 bytes for RS256 regardless of key size (the key size determines security, not signature length).

### ECDSA (Asymmetric): ES256, ES384, ES512

ECDSA uses elliptic curve cryptography. It provides equivalent security to RSA with smaller keys and faster operations.

Use ECDSA when:
- Token size matters (mobile, HTTP headers)
- You want asymmetric security with lower computational cost
- You need smaller key sizes

ES256 (using P-256 curve) produces a 64-byte signature --- one quarter of RS256. Verification is ~50-200 microseconds. The trade-off: ECDSA implementations are more complex and subtle implementation bugs (like nonce reuse) can leak the private key.

### Phase 1: Token Creation (Steps 1-4)

**Step 1 --- Create the header.** Build a JSON object with the algorithm and optional metadata.

```python
header = {
    "alg": "RS256",
    "typ": "JWT",
    "kid": "key-2025-q3"
}
```

**Step 2 --- Create the payload.** Build a JSON object with registered and custom claims.

```python
payload = {
    "sub": "user_42",
    "iss": "auth.sypher.io",
    "aud": "api.sypher.io",
    "iat": int(time.time()),
    "exp": int(time.time()) + 3600,  # 1 hour
    "jti": str(uuid.uuid4()),
    "roles": ["admin"],
    "tenant_id": "acme-corp"
}
```

**Step 3 --- Encode header and payload.** Convert each to JSON bytes, then base64url-encode them.

```python
import base64, json

def base64url_encode(data: bytes) -> str:
    return base64.urlsafe_b64encode(data).rstrip(b"=").decode("ascii")

encoded_header = base64url_encode(json.dumps(header).encode("utf-8"))
encoded_payload = base64url_encode(json.dumps(payload).encode("utf-8"))

signing_input = encoded_header + "." + encoded_payload
```

**Step 4 --- Sign.** Feed the signing input into the cryptographic algorithm with the private key.

```python
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import padding, rsa

# Using RS256
signature = private_key.sign(
    signing_input.encode("ascii"),
    padding.PKCS1v15(),
    hashes.SHA256()
)

encoded_signature = base64url_encode(signature)
jwt_token = signing_input + "." + encoded_signature
```

The token is complete: `header.payload.signature`.

### Phase 2: Transmission (Steps 5-7)

**Step 5 --- Attach to request.** The client attaches the token to an HTTP request, typically in the `Authorization` header.

```
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6ImtleS0yMDI1LXEzIn0
  .eyJzdWIiOiJ1c2VyXzQyIiwiaXNzIjoiYXV0aC5zeXBoZXIuaW8iLCJhdWQiOiJhcGkuc3lwaGVyLmlvIiwiaWF0IjoxNz
  E5ODc2NTQzLCJleHAiOjE3MTk4ODAxNDMsImp0aSI6ImExYjJjM2Q0LWU1ZjYtNzg5MC1hYmNkLWVmMTIzNDU2Nzg5MCIs
  InJvbGVzIjpbImFkbWluIl0sInRlbmFudF9pZCI6ImFjbWUtY29ycCJ9
  .SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

**Step 6 --- Transmit over network.** The HTTP request travels across the network. HTTPS encryption protects the token in transit. Without HTTPS, the token is visible in plaintext to anyone on the network path.

**Step 7 --- Receive at server.** The server extracts the token from the `Authorization` header. The server splits the token on dots to recover the three parts:

```python
parts = jwt_token.split(".")
assert len(parts) == 3  # Must have exactly 3 parts
encoded_header, encoded_payload, encoded_signature = parts
```

**Step 8 --- Parse structure.** Split the token into three parts. Reject tokens with fewer or more than three segments. Decode the base64url-encoded header and payload.

```python
raw_header = json.loads(base64url_decode(encoded_header))
raw_payload = json.loads(base64url_decode(encoded_payload))
```

**Step 9 --- Validate the header.** Check the algorithm. This is the most common attack surface.

```python
alg = raw_header.get("alg")
if alg == "none":
    raise InvalidToken("Algorithm 'none' is not allowed")
if alg not in ALLOWED_ALGORITHMS:  # e.g., {"RS256", "ES256"}
    raise InvalidToken(f"Algorithm {alg} is not permitted")
```

**Step 10 --- Verify the signature.** Look up the verification key (from the `kid` if present), reconstruct the signing input, and run the verify operation.

```python
signing_input = encoded_header + "." + encoded_payload
signature_bytes = base64url_decode(encoded_signature)

# Look up the public key
kid = raw_header.get("kid")
public_key = key_store.get_public_key(kid)

# Verify
try:
    public_key.verify(
        signature_bytes,
        signing_input.encode("ascii"),
        padding.PKCS1v15(),
        hashes.SHA256()
    )
except InvalidSignature:
    raise InvalidToken("Signature verification failed")
```

The verification operation uses the *public* key. The signature covers the header and payload --- if either one is modified, the signature becomes invalid. This provides **integrity** (the token has not been tampered with) and **authenticity** (the token was created by the holder of the private key).

**Step 11 --- Validate claims.** Check each registered claim that your application requires.

```python
now = time.time()
LEEWAY_SECONDS = 30  # Clock skew tolerance

# Expiration: token must not be expired
if raw_payload.get("exp") and (raw_payload["exp"] + LEEWAY_SECONDS) < now:
    raise InvalidToken("Token is expired")

# Not Before: token must be active
if raw_payload.get("nbf") and (raw_payload["nbf"] - LEEWAY_SECONDS) > now:
    raise InvalidToken("Token is not yet active")

# Issuer: must match expected
if raw_payload.get("iss") != EXPECTED_ISSUER:
    raise InvalidToken(f"Issuer mismatch: {raw_payload.get('iss')}")

# Audience: must include expected audience
if raw_payload.get("aud") and EXPECTED_AUDIENCE not in (
    raw_payload["aud"] if isinstance(raw_payload["aud"], list)
    else [raw_payload["aud"]]
):
    raise InvalidToken(f"Audience mismatch: {raw_payload.get('aud')}")
```

### Phase 4: Claims Extraction (Step 12)

**Step 12 --- Extract and use claims.** Once the token passes all validation, the payload is trusted. Extract the claims and make them available to your application logic.

```python
# These claims are now trusted
current_user = {
    "user_id": raw_payload["sub"],
    "roles": raw_payload.get("roles", []),
    "tenant_id": raw_payload.get("tenant_id")
}
```

The critical principle: **never use any claim before validation**. An unverified token is attacker-controlled data. Only after steps 8-11 pass can you trust the payload contents.

## Consistency Concerns

Several practical issues arise when using JWTs at scale. You must handle each one deliberately.

### Clock Skew

Server clocks are never perfectly synchronized. NTP keeps them close, but drift of a few seconds is normal. The `exp` and `nbf` checks compare token timestamps against the current server time. If the issuer's clock is 5 seconds ahead of the verifier's clock, a just-issued token might appear expired.

**Solution: apply a leeway window.** Most JWT libraries allow a `leeway` parameter, typically 30-60 seconds.

```python
# Python PyJWT library
decoded = jwt.decode(
    token,
    public_key,
    algorithms=["RS256"],
    leeway=30,           # Allow 30 seconds of clock skew
    issuer="auth.sypher.io",
    audience="api.sypher.io"
)
```

The leeway window is a trade-off. Too large (minutes) and expired tokens remain valid too long. Too small (zero) and legitimate requests fail due to clock drift. **30 seconds** is the industry standard.

### Key Rotation

Signing keys must rotate periodically. A compromised key can forge tokens for any user. Key rotation is the process of retiring old keys and introducing new ones without invalidating active tokens.

Implementation pattern for key rotation:

1. Assign each key a **unique ID** (`kid`) and a **creation timestamp**
2. Store the current signing key and all recent verification keys in a key store (e.g., a secrets manager or a JWKS endpoint)
3. When a token arrives, read `kid` from the header and look up the corresponding verification key
4. Periodically (e.g., every 90 days) generate a new signing key
5. Start signing new tokens with the new key
6. Keep old verification keys until the maximum token lifetime has passed (e.g., 7 days of token expiry + 30 seconds leeway)
7. Remove old keys once no active token references them

```python
# Key store interface
class KeyStore:
    def get_signing_key(self) -> tuple[str, SigningKey]:
        """Return the current (kid, private_key)."""
        ...

    def get_verification_key(self, kid: str) -> VerificationKey:
        """Return the public key for verification, or raise KeyNotFound."""
        ...

    def rotate(self) -> None:
        """Generate a new key pair and update the signing key reference."""
        ...
```

### The "none" Algorithm Attack

A well-known JWT vulnerability: the attacker sets `"alg": "none"` in the header and removes the signature entirely. If the verification library trusts the header's `alg` field without enforcing a whitelist, it accepts the unsigned token.

```python
# Vulnerable code --- DO NOT USE
header = {"alg": "none", "typ": "JWT"}
payload = {"sub": "admin", "exp": 9999999999}
# No signature!
token = base64url(header) + "." + base64url(payload) + "."
```

The fix: **never derive your security policy from the token header**. Always enforce an explicit algorithm whitelist.

```python
# Safe validation
options = {
    "verify_signature": True,  # Never allow unsigned tokens
    "require": ["exp", "iss", "aud"],
    "algorithms": ["RS256", "ES256"],  # Explicit whitelist, never "none"
}
```

### Algorithm Confusion Attack

A more sophisticated variant: the attacker changes `alg` from `RS256` (asymmetric) to `HS256` (symmetric). If the verifier uses the public key as the HMAC secret (it's a string, after all), the attacker can sign tokens using the public key, which is freely available.

```python
# Attacker:
# 1. Obtain the public key (it's public)
# 2. Create a token with alg=HS256
# 3. Sign it using the public key as the HMAC secret
# 4. Send it to the server

# Vulnerable server:
jwt.decode(token, public_key, algorithms=["HS256", "RS256"])
# The server tries HS256 first, uses public_key as the secret -> passes!
```

The fix: **never accept both asymmetric and symmetric algorithms from the same verifier**. If you use RS256, only allow RS256 (and ES256, PS256). Never include HS256 in the whitelist when using asymmetric keys.

## Performance Characteristics

Understanding JWT performance is critical for capacity planning.

### Verification Latency

| Algorithm | Sign (us) | Verify (us) | Key Size |
|-----------|-----------|-------------|----------|
| HS256     | 1-3       | 1-3         | 32 bytes (secret) |
| RS256 (2048-bit) | 200-400 | 100-200 | 256 bytes (signature) |
| RS256 (4096-bit) | 800-1500 | 400-800 | 256 bytes (signature) |
| ES256 (P-256) | 50-100 | 50-100 | 64 bytes (signature) |
| ES384 (P-384) | 80-150 | 80-150 | 96 bytes (signature) |

Numbers are approximate, measured on modern (2024-era) server CPUs. Cryptographic operations are CPU-bound, so performance depends on hardware. AES-NI and similar instruction-set extensions accelerate HMAC significantly.

The key insight: **verification is not the bottleneck**. Even at 200 microseconds per RS256 verification, a single core can handle 5,000 verifications per second. The bottleneck in most systems is the database lookup that the JWT eliminates.

### Token Size Overhead

Token size matters because it travels in every HTTP request header.

```
Token component                  Size (bytes)
─────────────────────────────────────────────
Header (RS256, typ, kid)          ~100
Payload (sub, iss, aud, iat,      ~250
  exp, jti, roles, tenant_id)
Signature (RS256)                  256
─────────────────────────────────────────────
Base64 overhead (~33%)             ~200 (3 parts × 33%)
─────────────────────────────────────────────
Total JWT (RS256)                ~800 bytes
Total JWT (ES256)                ~550 bytes
Total JWT (HS256)                ~450 bytes
```

An 800-byte Authorization header is negligible for most APIs (typical HTTP request headers are 2-10 KB). However, for:

- **High-throughput APIs** (10,000+ req/s) --- the extra bytes add up. 800 bytes per request at 10K req/s = 8 MB/s of token data on the wire.
- **Mobile APIs** --- bandwidth is limited and latency-sensitive. Use ES256 to save ~250 bytes per token.
- **Cookie-based JWTs** --- browser cookies have a 4 KB limit per domain. An 800-byte JWT consumes 20% of your cookie budget. Consider using a session reference (opaque token) instead.

### Impact on HTTP Headers

The `Authorization: Bearer <jwt>` header affects HTTP infrastructure:

- **Load balancers** must parse the header for routing decisions (if you use token-based routing). Some load balancers have header size limits (e.g., AWS ALB: 16 KB for headers).
- **CDNs** can cache responses based on the Authorization header, but most CDNs treat `Authorization` as a cache-busting header by default. Configure cache keys carefully.
- **Logging** --- never log the full JWT in plaintext. The token carries authentication data that can be replayed. Log only the `jti` (token ID) and `sub` (subject) for auditing.

```python
# Safe logging
import logging
logger.info("Token validated: sub=%s, jti=%s", payload["sub"], payload["jti"])
# Never: logger.info("Token: %s", raw_token)
```

## Common Pitfalls

### Pitfall 1: Storing Secrets in the Payload

JWTs are signed, not encrypted. Anyone with the token can base64url-decode the payload and read the contents. Never store secrets, passwords, or sensitive PII in the payload.

```json
// BAD: PII in the clear
{ "sub": "user_42", "ssn": "123-45-6789", "credit_card": "4111-1111-1111-1111" }

// GOOD: Only identifiers
{ "sub": "user_42", "roles": ["admin"] }
```

If you must protect payload contents, use **JWE** (JSON Web Encryption) --- a separate specification that encrypts the entire payload. JWE is rarely used in practice because:
- HTTPS already protects the token in transit
- JWE adds significant complexity and size overhead
- Most JWT libraries do not support JWE well

### Pitfall 2: Stateless Token Revocation

JWTs are designed to be stateless. This means you cannot revoke a JWT before it expires without maintaining a blocklist. If a user logs out, their token remains valid until `exp`.

Solutions:
- **Short expiry** (15 minutes) paired with refresh tokens --- the access token window is small
- **Token blocklist** (stored in Redis) --- check against the blocklist on every request, sacrificing the stateless benefit
- **JTI tracking** --- store the `jti` (token ID) in a database with a TTL matching the token expiry

### Pitfall 3: Token Size in Cookies

If you set a JWT as an HTTP cookie, remember the browser cookie limit: **4,096 bytes per domain** (total across all cookies). A full JWT with RS256 can consume 800+ bytes, leaving limited room for other cookies.

```http
Set-Cookie: session=eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...;
           HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=3600
```

Consider using a session reference (a random opaque string) as the cookie value and storing the JWT server-side.

### Architecture Summary

| Component | Role | Key Material |
|-----------|------|-------------|
| Auth Service | Issues JWTs | Private key (signing) |
| Client | Stores and transmits JWT | None (token is opaque data) |
| API Gateway | Verifies JWTs, extracts claims | Public keys (verification) |
| Backend Services | Receives verified claims via proxy headers | None (trusts gateway) |
| JWKS Endpoint | Distributes public keys for verification | Public keys (read-only) |

This architecture gives you:
- **Stateless verification** --- the gateway does not contact the auth service on each request
- **Decentralized trust** --- any service with the public key can verify independently
- **Key rotation** --- the JWKS endpoint provides a single place to update keys
- **Reduced surface area** --- backend services never handle raw tokens, only verified claims
