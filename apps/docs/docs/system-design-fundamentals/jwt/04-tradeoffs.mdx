# JWT — Trade-offs & Failure Modes

Every engineering decision is a stack of trade-offs. JWT is no exception. On the surface, JSON Web Tokens give you stateless authentication, portable claims, and a clean protocol boundary. But those same features introduce failure modes that can brick your entire system if you don't understand where the trade-offs bite.

This page walks you through the core tension, the decision matrix you need to evaluate, the failure modes you must guard against, and the anti-patterns that will wake you up at 3 AM.

---

## The Core Tension

You can verify a JWT without touching a database. That is the killer feature. But it also means you cannot un-verify a JWT once it is issued. The token is valid until its `exp` claim says otherwise, and no amount of wishful thinking will change that.

The consequence is simple: you design your expiry strategy before you need a revocation strategy, not after. If you need instant revocation for every token (password reset, account suspension, permission change), you either accept a compromise — a token-blacklist table that defeats the stateless promise — or you do not use JWT for that session.

## Decision Matrix

Every architectural choice around JWT involves a trade-off. Here is a structured decision matrix for the six most important dimensions.

### 1. JWT vs. Opaque Token

| Dimension | JWT | Opaque Token |
|---|---|---|
| Verification | Decode + verify signature locally | Look up in database or cache |
| Latency | Sub-millisecond | 1–10 ms (DB/cache round trip) |
| Revocation | Delayed (expiry only) | Immediate (delete from store) |
| Payload | Visible claims (not secret) | Opaque reference only |
| Bandwidth | Larger token (500–2000+ bytes) | Small (16–32 bytes) |
| Debugging | Decode to see user info | Must query store |

**When you pick JWT:** you need low-latency verification at scale, you can tolerate delayed revocation, and you want to avoid a shared session store.

**When you pick opaque:** you need instant revocation, you are on a single server or small deployment, or you cannot afford to leak any information in the token body.

### 2. HMAC (Symmetric) vs. RSA/ECDSA (Asymmetric)

| Dimension | HMAC (HS256) | RSA/ECDSA (RS256/ES256) |
|---|---|---|
| Key management | Single shared secret | Private + public key pair |
| Distribution | Every service holds the secret | Only signing service holds private key |
| Verification | Fast, simple | Slightly slower (asymmetric math) |
| Audience scope | All services must trust the secret | Any service with public key can verify |
| Security risk | Compromise of one = compromise of all | Private key is the only secret |
| Rotation | Rotate on all services simultaneously | Rotate private key, publish new public key |

**When you pick HMAC:** you have a single service or a tightly controlled set of internal services. You accept that a leak anywhere means a leak everywhere.

**When you pick RSA/ECDSA:** you have multiple services that need to verify tokens without sharing a secret. You are willing to pay the small CPU cost for the security boundary.

### 3. Short Expiry vs. Long Expiry

| Dimension | Short (5–15 min) | Long (hours to days) |
|---|---|---|
| Security window | Small if stolen | Large if stolen |
| User experience | Frequent re-auth | Infrequent re-auth |
| Refresh token needed | Yes | Optional |
| Server load | More token refreshes | Fewer refreshes |

**Short expiry is the default for any production system.** You pair it with a longer-lived refresh token that can be rotated and revoked. Long expiry is appropriate only for low-security contexts (internal tools, development environments) or when you accept the risk of a stolen token being usable for hours.

### 4. Refresh Token Rotation

| Dimension | No Rotation | Rotation on Every Use |
|---|---|---|
| Security | Stolen refresh token works forever | Stolen refresh token works once, then invalidated |
| Complexity | Simple | Requires tracking previous refresh token hash |
| User impact | Re-auth only on expiry | Transparent — new pair issued each time |
| Failure mode | Refresh token leak = permanent access | Race condition on concurrent refresh |

**You should rotate refresh tokens in any security-sensitive system.** The trade-off is handling concurrent requests that both try to refresh — you need a locking mechanism or a family-of-tokens approach to avoid logging the user out when two tabs refresh simultaneously.

### 5. Include Claims vs. Reference Token

| Dimension | Include Claims | Reference Token |
|---|---|---|
| Size | Large (all user data in token) | Small (just user ID) |
| Freshness | Stale until token expires | Always fresh (queried on each use) |
| DB load | None for verification | One lookup per request |
| Flexibility | Claims are whatever you signed | Any info, any time |
| Cache control | None needed | Cache the user payload |

**Include claims** when the data rarely changes (roles, permissions that update hourly) and you want zero-latency verification. **Reference token** when the data changes frequently (balance, status, tier) or when the token would otherwise be too large.

### 6. Self-Encoded vs. Reference Token

| Dimension | Self-Encoded (JWT) | Reference (Session ID) |
|---|---|---|
| Storage | Nothing on server | Server-side session store |
| Revocation | By expiry or blacklist | Delete from store |
| Size | 500–2000+ bytes | 16–32 bytes |
| Debugging | Decode to inspect | Must query store |
| Portability | Works across domains | Bound to session store location |

**Self-encoded** is the right choice when you need to verify across services without a shared session store. **Reference** is the right choice when you control the entire stack and want instant revocation.

JWT is AP: **Available** (any node can verify a token without contacting another node) and **Partition Tolerant** (a network split does not prevent verification). The sacrifice is **Consistency** — after you revoke a token, there is a window where it is still accepted. The length of that window is the remaining TTL of the token.

If you need consistency (revoke now, every node agrees immediately), you must add a shared blacklist or switch to opaque tokens backed by a strongly consistent store.

## Failure Modes

You must know how JWT fails. These are not theoretical — every one of these has taken down production systems.

### 1. Algorithm Confusion Attack

This is the most famous JWT vulnerability. Your server signs tokens with RS256 (asymmetric). The attacker takes the public key, encodes it as a symmetric key, and crafts a new token with the `alg` header set to `HS256`. If your verification library does not check that the algorithm matches the expected value, it uses the attacker-supplied public key as the HMAC secret and accepts the forged token.

**Mitigation:** Always pin the expected algorithm. In your verification code, never read the `alg` from the token header — set it from configuration. Use a library like `jose` that enforces algorithm whitelisting by default.

### 2. Expired Token During Long Operation

Your user starts a 10-minute file upload at minute 8 of a 15-minute token. The upload completes at minute 16. The token expired at minute 15. The upload API rejects the final chunk because the token is now invalid.

The user is frustrated. The file is half-uploaded. The system is in an inconsistent state.

**Mitigation:** Use a separate upload token with a longer expiry or a resumable upload protocol that does not require authentication on every chunk. Alternatively, issue a short-lived operation-specific token at the start of the long operation.

### 3. Clock Skew Rejection

Your server and your authentication server have clocks that differ by 60 seconds. The auth server issues a token with `iat` (issued at) set to the current time on its clock. Your server checks that the token is not from the future — a standard `nbf` (not before) check. Because the server clock is 60 seconds behind, it sees the token as issued 60 seconds in the future and rejects it.

**Mitigation:** Most JWT libraries support a `clockTolerance` or `leeway` parameter. Set it to 30–60 seconds to absorb normal clock drift. Better yet, synchronize all your servers with NTP and monitor clock drift as a standard alert.

### 4. JWKS Endpoint Down

Your microservice architecture uses RS256. The verification service fetches the public key from the auth service's JWKS (JSON Web Key Set) endpoint. The auth service is under load and the JWKS endpoint returns 503. Your service cannot verify any new tokens. Every request fails. Cascading failure.

### 5. Stolen Refresh Token

A refresh token is stolen from the user's browser (XSS, malicious extension, leaked device). The attacker now has a long-lived credential that can generate new access tokens indefinitely. The user does not even know because their existing access token still works.

**Mitigation:** Implement refresh token rotation — every time a refresh token is used, issue a new one and invalidate the old one. If the attacker uses the stolen token and the legitimate user tries to refresh, one of them will fail. Detect this as a "token reuse" event and revoke all tokens for the user. Additionally, bind refresh tokens to device fingerprints or IP ranges so that a stolen token is useless from a different context.

### 6. Token Injection via URL

A JWT passed in a URL query parameter is logged by every proxy, load balancer, and CDN along the path. The token is now in plaintext in dozens of log files, any of which could be compromised.

**Mitigation:** Never pass JWTs in URL query parameters. Use the `Authorization: Bearer <token>` header. If you must use a URL (e.g., for email links), use a one-time code that is exchanged for a token server-side.

### 7. Large Token Overhead

A JWT carrying 20 claims, a full user profile, and permission list can exceed 4 KB. Every HTTP request carries this overhead. At 10,000 requests per second, that is 40 MB/s of token data on the wire, plus the CPU cost of parsing and verifying it.

**Mitigation:** Keep claims minimal. Include only the `sub` (user identifier), `iat` (issued at), `exp` (expiry), and maybe a single role or scope claim. Look up everything else. Reference tokens are smaller and cheaper when token size becomes a bottleneck.

## Anti-Patterns

These are patterns you will see in production code written by developers who learned JWT from a single blog post. Do not replicate them.

### 1. Storing Secrets in the Payload

The JWT payload is base64url-encoded, not encrypted. Anybody who intercepts the token can decode and read the payload. Developers sometimes store API keys, database passwords, or internal service URLs in the payload because "the token is signed, so it is secure."

**The truth:** Signed means tamper-proof, not private. The payload is visible to any client, any intermediate proxy, and anyone who reads your logs. If you need private data in the token, use JWE (JSON Web Encryption) — but at that point, ask yourself whether you should be carrying that data in the token at all.

### 2. No Expiry

A token without an `exp` claim is valid forever. Once issued, it can never be revoked. If the token is leaked, the attacker has permanent access. If a user leaves the organization, their token still works.

**Always set an `exp` claim.** Even for internal tools, set it to at least 24 hours. For production systems, 5–15 minutes for access tokens with a refresh token for renewal.

### 3. Using the "none" Algorithm

The JWT specification allows an `alg: "none"` header for unsecured tokens. Some libraries default to accepting this for backward compatibility. An attacker can set `alg: "none"`, remove the signature, and the server will accept the token as valid.

### 4. Symmetric Key Across All Services

Every microservice shares the same HMAC secret. If any one service is compromised, the attacker can forge tokens for any other service. This is common in early-stage architectures where "it works" is the only criterion.

**Mitigation:** Use asymmetric keys (RS256 or ES256). Only the authentication service holds the private key. Every other service keeps only the public key. A compromised microservice can verify tokens (which it needs to do anyway) but cannot forge them.

### 5. Token in URL

Passing a JWT in a URL query parameter exposes it to:
- Browser history (stored in plaintext)
- Server logs
- Referrer headers (leaked to third parties)
- Proxy and CDN logs
- Bookmarking (if someone bookmarks the URL)

**Mitigation:** Always use the `Authorization` header. If you must use a URL (e.g., for an OAuth redirect), use a short-lived authorization code that is exchanged for a token server-side, never expose the token itself in the URL.

### 6. No Audience or Issuer Validation

You sign a token for the `users` service. The `payments` service receives it and accepts it because the signature is valid. But the token was meant for `users`, not `payments`. The `payments` service trusts the `role` claim in the token and allows an admin action.

**Mitigation:** Always validate the `aud` (audience) and `iss` (issuer) claims. Each service should check that the token was issued for it. If you are using a single unified auth service, use a consistent `iss` value and verify it.

### 7. Overloaded Claims

You store a full user object, including permissions, preferences, and metadata, in the JWT. The token grows to 3–5 KB. Every request carries this overhead. When a user's permissions change, they are still using the old token. The user sees stale data or unexpected access denials.

**Mitigation:** Store only the minimum claims needed for verification. Store the user ID (`sub`), issuance time (`iat`), expiry (`exp`), and a single scope or role identifier. Look up dynamic data from a service or cache when needed.

## When NOT to Use JWT

JWT is not the right tool for every job. Here are the situations where you should avoid it.

### Instant Revocation Required

If you need to revoke a user's access immediately — when they change their password, when their account is suspended, when their permissions change — JWT works against you. The existing token remains valid until it expires. You can add a blacklist, but that blacklist is a database lookup that defeats the stateless advantage.

**Better alternatives:** Opaque session tokens with a server-side session store. Or a hybrid approach: short-lived JWTs (1–5 minutes) with a revocable refresh token. The access token window is small enough that the revocation delay is acceptable.

### Small Services

If you have one monolithic server or two services running on the same machine, JWT adds complexity without benefit. You do not need stateless verification across services because there are no cross-service boundaries. Session cookies with a server-side store are simpler, more secure, and easier to debug.

### Server-to-Server with mTLS

If your services communicate over mutual TLS (mTLS), you already have strong identity verification at the transport layer. Adding JWT on top is redundant. The mTLS certificate identifies the service, and the connection is encrypted and authenticated.

**Better approach:** Use mTLS for service-to-service authentication. Reserve JWT for user-to-service or client-to-service authentication where mTLS is not practical (browser clients, mobile apps, third-party integrations).

### Short-Lived, Single-Use Operations

If you need a one-time credential (password reset link, email verification, magic link), a full JWT is overkill. A random 32-byte token stored in the database with a 10-minute TTL is simpler and more secure.

**Better approach:** Generate a random token, store a hash in the database, set an expiry, and delete it after use. No JWT library needed.

### Systems Without a Centralized Auth Service

If every service manages its own authentication, each service would need its own signing key, its own token format, and its own verification logic. You end up with N different auth systems, none of which interoperate.

**Better approach:** Centralize authentication in a single service. That service issues JWTs that all other services verify. If you cannot centralize, use a shared public-key infrastructure and a common token format — but you are still better off with a dedicated auth service.

## Summary: The Trade-off Table

**The rule of thumb:** If you need portable, stateless verification across services and you can tolerate delayed revocation, JWT is a strong choice. If you need instant revocation, if you have a single service, or if you already have mTLS, skip the JWT and use a simpler approach.
