## How to Use These Challenges

Each challenge has a time limit, evaluation criteria, and a model answer. Attempt the challenge within the time limit before reading the model answer. The challenges progress from foundation (10 minutes) through design (20 minutes) to expert (30 minutes).

Passing all challenges at a tier means you have mastered that level. Move to the next tier only after you consistently pass.

---

## Tier 1: Foundation

### Challenge 1.1: Fan-Out Strategy Decision (10 minutes)

**Prompt.** You are designing a feed system for a new social platform. The platform has two types of users: regular users (5 million, averaging 150 followers each) and verified celebrity accounts (500, each with 500,000 to 5 million followers). You need to choose a fan-out strategy.

Calculate the write amplification for a pure fan-out on write approach when every user posts once per day. Compare this to a hybrid approach with a 10,000-follower threshold. Show your math for daily total writes to the feed storage cluster.

**Time limit.** 10 minutes.

**Evaluation criteria.** You correctly calculate the daily write counts. You identify that pure fan-out on write generates 750 million writes per day from regular users plus up to 2.5 billion writes per day from celebrity accounts. The hybrid approach eliminates the celebrity writes entirely, reducing the total to 750 million writes per day. You explain that the 95 percent reduction in write amplification is worth the complexity of the hybrid approach.

**Common mistakes.** Forgetting to account for celebrity posts in the write amplification calculation. Assuming all users post at the same rate. Not considering that many users are inactive on any given day.

**Model answer.** Pure fan-out on write: 5 million regular users times 150 followers equals 750 million writes. 500 celebrities times 500,000 followers averages 250 million writes. Total: 1 billion writes per day minimum. With the hybrid approach, regular users under 10,000 followers get push (750 million writes). Celebrities with over 10,000 followers get pull (0 writes for the fan-out). Total: 750 million writes per day. The hybrid approach reduces write amplification by 25 percent for the low-end estimate and over 95 percent for the celebrity worst case.

### Challenge 1.2: Feed List Storage Sizing (10 minutes)

**Prompt.** Your feed system uses Redis sorted sets with 800 items per user feed list. Each feed list entry is 40 bytes. You have 50 million active users. Calculate the Redis cluster size requirements. Include replication (primary plus one replica) and 30 percent overhead. How many 64 GB Redis nodes do you need?

**Time limit.** 10 minutes.

**Evaluation criteria.** You calculate the raw feed list data: 50 million times 800 times 40 equals 1.6 TB. With replication: 3.2 TB. With 30 percent overhead: 4.16 TB. Divide by 64 GB per node: 4,160 GB divided by 64 GB equals 65 nodes. With a 70 percent memory utilization target (to leave headroom for other Redis operations), you need 93 nodes.

**Common mistakes.** Forgetting replication overhead. Not accounting for Redis memory overhead. Using the raw data size without considering other Redis data structures and connection buffers.

**Model answer.** Raw feed list data: 50 million users times 800 items times 40 bytes equals 1.6 TB. With 2x replication: 3.2 TB. With 30 percent overhead: 4.16 TB. With a 70 percent utilization target: 4.16 divided by 0.7 equals 5.94 TB effective capacity needed. Each 64 GB node provides 44.8 GB usable at 70 percent: 5,940 GB divided by 44.8 GB equals 133 nodes. With 128 shards (128 primaries plus 128 replicas equals 256 nodes), you have more than enough capacity. Each shard stores 4.16 TB divided by 128 equals 32.5 GB, well within the 44.8 GB usable limit.

### Challenge 1.3: Read Path Latency Budget (10 minutes)

**Prompt.** You have a target P99 read latency of 300 milliseconds. Budget the latency across the read path subsystems. List each subsystem, allocate a budget in milliseconds, and justify the allocation. Ensure the total does not exceed 300 milliseconds at P99.

**Time limit.** 10 minutes.

**Evaluation criteria.** You identify 5 to 7 subsystems in the read path. You allocate a realistic budget for each. The total is at or under 300 milliseconds. The largest allocation is for hydration (the most expensive step).

**Common mistakes.** Not accounting for network overhead. Allocating too little time for hydration. Forgetting the cache check step. Not considering that P99 is higher than average latency.

**Model answer.** Cache check: 10 ms (includes network round trip to Memcached). Feed list read (cache miss path): 20 ms (Redis ZREVRANGE, includes network). Ranking score computation: 30 ms (in-memory scoring of 50 items). Deduplication and filtering: 10 ms (set membership checks). Hydration batch fetch: 100 ms (batch get from content service, P99 includes cache misses). Media URL assembly: 15 ms (CDN URL construction, token signing). Response serialization: 15 ms (JSON serialization of 50 items). Network overhead to client: 50 ms (varies by region, allocate generously). Total: 250 ms. Headroom: 50 ms. This leaves room for unexpected spikes without exceeding the 300 ms target.

### Challenge 1.4: Write Amplification Calculation (10 minutes)

**Prompt.** Your platform has 10 million daily active users. The average user posts 2 times per day. The average user has 250 followers. You are using fan-out on write. Calculate the total number of feed list writes per day. Then calculate the writes per second during peak hour, assuming 20 percent of daily writes happen in the busiest hour. How many Redis nodes do you need if each node handles 100,000 writes per second?

**Time limit.** 10 minutes.

**Evaluation criteria.** You calculate the total daily writes. You convert to writes per second during peak hour. You calculate the minimum Redis nodes needed. You include replication in your node count.

**Common mistakes.** Forgetting to multiply posts per day by followers per user. Not accounting for peak hour traffic concentration. Forgetting replication overhead in node count.

**Model answer.** Total daily writes: 10 million users times 2 posts per day times 250 followers equals 5 billion writes per day. Peak hour writes: 5 billion times 20 percent equals 1 billion writes per hour. Writes per second during peak hour: 1 billion divided by 3,600 equals 277,778 writes per second. Redis nodes needed: 277,778 divided by 100,000 writes per second per node equals 3 nodes minimum. With replication (primary plus one replica): 6 nodes. With headroom for failover and spikes (2x): 12 nodes. You should plan for a 12-node Redis cluster (6 primaries, 6 replicas) to handle the peak write throughput with headroom.

### Challenge 1.5: Feed List Eviction Policy (10 minutes)

**Prompt.** Your feed list stores 800 items per user. A user follows 500 accounts. Each account posts 3 times per day on average. The user reads their feed twice per day. When a new item arrives and the feed list is full, the lowest-scored item is evicted. Calculate how many items are evicted per day for this user. What percentage of items does the user miss because of eviction?

**Time limit.** 10 minutes.

**Evaluation criteria.** You calculate the daily incoming volume. You calculate the eviction rate. You calculate the percentage of items the user misses before seeing them. You discuss whether this is acceptable.

**Common mistakes.** Not considering that the user reads their feed twice per day, which means they see some items before eviction. Treating all items as equally likely to be evicted (the lowest-scored items are evicted first).

**Model answer.** Daily incoming items: 500 accounts times 3 posts equals 1,500 items per day. Feed list capacity: 800 items. The user reads twice per day. Between reads, approximately 750 items arrive (1,500 divided by 2). The feed list starts at 800 items at the first read. After the read, some items age and new items arrive. By the time of the next read (12 hours later), approximately 750 new items have arrived. The feed list was full at 800, so 750 new items push out 750 old items. Of those 750 evicted items, some were seen in the previous read (approximately 50 items), and some arrived after the previous read and were evicted before the current read (approximately 700 items). The user misses 700 out of 1,500 daily items, or 47 percent of content. This is acceptable because the feed list retains the highest-scored items. The evicted items are the lowest-scored items, which are the least relevant to the user. If the user wants to see all content, increase the feed list size to 2,000 items.

## Tier 2: Design

### Challenge 2.1: Design a Real-Time Activity Feed (20 minutes)

**Prompt.** Design a real-time activity feed for a collaborative document editing platform (like Google Docs). The feed shows real-time updates: teammate opened a document, teammate edited paragraph 3, teammate added a comment, teammate resolved a comment. The feed must update within 500 milliseconds of the activity occurring. The platform has 10 million users organized into teams of 5 to 50 members.

Your design should include: the data model for activity events, the storage strategy, the fan-out approach, the real-time delivery mechanism, and the failure modes.

**Time limit.** 20 minutes.

**Evaluation criteria.** You propose a real-time feed design that differs from the standard social media feed. You handle the unique requirements: small audience per activity (team of 5-50, not millions), real-time delivery (WebSocket or SSE), ordering by event time. You address failure modes like dropped WebSocket connections and event ordering.

**Common mistakes.** Applying the social media feed design directly without adjusting for the different scale and latency requirements. Not considering that the audience per activity is small (team members only). Forgetting the real-time delivery mechanism.

**Model answer.** The activity feed uses fan-out on write because the audience per activity is small (5 to 50 team members). The fan-out writes an activity event to each team member's feed list in Redis. The feed list has a shorter retention (200 items) because activity feeds show recent events only. Events are delivered in real time via WebSocket connections. When an event is written to the feed list, the fan-out engine publishes the event to a Redis Pub/Sub channel per team. The read path service subscribes to the team channels and forwards events to connected clients. For clients not connected at the time of the event, the feed list provides catch-up on next connection. Failure mode: WebSocket disconnection. The client reconnects and uses cursor-based pagination to fetch missed events from the last seen cursor.

### Challenge 2.2: Design a Multi-Tenant Feed System (20 minutes)

**Prompt.** Design a feed system that serves 100 enterprise tenants. Each tenant has 10,000 to 500,000 users. Each tenant requires data isolation: no tenant can see another tenant's data. Each tenant has its own ranking preferences and content moderation policies. You must minimize infrastructure cost while meeting tenant SLAs (P99 read latency under 300 milliseconds, 99.9 percent uptime per tenant).

**Time limit.** 20 minutes.

**Evaluation criteria.** You propose a multi-tenant architecture with data isolation. You address per-tenant ranking customization. You balance cost and isolation. You estimate the infrastructure cost.

**Common mistakes.** Using database-level isolation (separate databases per tenant) without considering the cost. Ignoring per-tenant ranking differences. Not addressing tenant-level capacity planning.

**Model answer.** The multi-tenant feed system uses a hybrid isolation model. Small tenants (under 50,000 users) share a feed storage shard with logical isolation (key prefix per tenant). Large tenants (over 50,000 users) get dedicated shards. The fan-out engine is tenant-aware: it routes writes to the correct shard based on the tenant ID. Each tenant has a configuration record that specifies the unique ranking weights, content moderation rules, and retention policy. The ranking engine loads the tenant-specific configuration when re-scoring feed lists for that tenant's users. The timeline cache is per-tenant to prevent cache poisoning between tenants. Cost estimate: 20 shared shards for small tenants plus 40 dedicated shards for large tenants equals 60 shards (120 nodes with replication). At 64 GB per node, the total Redis cost is approximately 7,680 GB. At an estimated cost of $0.50 per GB per month, the Redis cost is $3,840 per month. Combined with compute and networking, the total infrastructure cost is approximately $15,000 per month for 10 million users across 100 tenants.

### Challenge 2.3: Design a Feed Ranking Experiment (20 minutes)

**Prompt.** Your feed system currently uses chronological ordering. You want to test whether a ranked feed improves user engagement. Design an A/B testing framework for feed ranking. Your design must: randomly assign 1 percent of users to the ranked feed, 1 percent to a reverse-chronological feed (control for novelty effects), and keep 98 percent on the current chronological feed. The ranking model must be fast enough for real-time use (under 20 milliseconds per inference). You need to measure engagement metrics (likes, comments, shares, time spent) at a 95 percent confidence level.

**Time limit.** 20 minutes.

**Evaluation criteria.** You design a statistically sound A/B test. You handle the real-time ranking requirement. You specify the engagement metrics and the statistical significance criteria. You address the novelty effect (users initially engage more with any change).

**Common mistakes.** Not addressing the novelty effect. Using too small a sample size for statistical significance. Not considering that the ranking model must run in real time.

**Model answer.** The A/B test uses user-level assignment with a hash-based bucketing function. The hash of the user ID modulo 100 determines the bucket. Users 1-99: chronological. User 100: ranked. User 101: reverse-chronological (novelty control). The ranked feed uses a lightweight scoring function with 5 signals: recency weight 0.4, like count weight 0.2, comment count weight 0.15, author affinity weight 0.15, content type preference weight 0.1. The scoring function runs inline in the read path, taking 5 microseconds per item (250 microseconds for 50 items). The A/B test runs for 14 days to account for day-of-week effects and novelty decay. Primary metric: daily active minutes per user. Secondary metrics: likes and comments per feed session, feed scroll depth (are users scrolling more?), content diversity (are users seeing content from more unique authors?). Statistical significance threshold: p-value below 0.05 with a minimum detectable effect of 5 percent on daily active minutes. The experiment is monitored daily. If the ranked group shows a 10 percent or more decrease in engagement (negative impact), the experiment is stopped early.

### Challenge 2.4: Design a Celebrity Onboarding Strategy (20 minutes)

**Prompt.** A celebrity with 20 million followers is joining your platform next week. Your current feed system uses fan-out on write for all users. You have 2 million existing users and your fan-out engine handles 200 writes per second. The celebrity is expected to post 5 times on the first day. Design the onboarding strategy to prevent the celebrity's posts from overwhelming the system.

**Time limit.** 20 minutes.

**Evaluation criteria.** You propose a strategy that protects the existing system from the celebrity's write amplification. You discuss the changes needed before the celebrity joins. You address both the first-post shock and the steady-state operation.

**Common mistakes.** Proposing to upgrade the infrastructure for the celebrity (too expensive for a single account). Not considering that the celebrity's followers are not all on the platform yet (only 2 million existing users, but 20 million followers means many new users will join).

**Model answer.** Before the celebrity joins, implement three changes. First, add a fan-out threshold of 10,000 followers. The celebrity has 20 million followers, so they use pull-based fan-out. Second, pre-create the "new content available" Redis key for the celebrity's account. The fan-out engine writes a flag to this key when the celebrity posts, not 20 million feed list entries. Third, prepare the read path for the merge: when a user reads their feed, the read path checks the celebrity flag and merges the celebrity's recent posts into the feed. On the first day, the celebrity posts 5 times. Each post generates 1 write (the flag) instead of 20 million writes. The fan-out engine handles this with zero additional load. The read path handles the merge by querying the celebrity's recent posts and merging them into each user's feed. The merge adds 50 milliseconds to the read path latency, which is acceptable. After the first day, monitor the fan-out lag and read path latency. If the merge latency is too high, add a short-term cache for the celebrity's recent posts so that the read path does not query the content service for every feed read.

### Challenge 2.5: Design a Feed for Offline Users (20 minutes)

**Prompt.** Design a feed system for a platform where 40 percent of users are offline (no internet connection) for 8 to 12 hours per day. These users expect to see fresh content when they come online. The online users (60 percent) expect real-time feed updates. The platform has 50 million total users. The feed must be chronological. Your design must handle both online and offline users efficiently without wasting resources on offline users while they are disconnected.

**Time limit.** 20 minutes.

**Evaluation criteria.** You differentiate the feed delivery for online vs offline users. You avoid writing feed items to offline users' feed lists while they are disconnected. You provide a catch-up mechanism for offline users when they reconnect.

**Common mistakes.** Treating offline users the same as online users (wasting fan-out writes on disconnected users). Not having a catch-up mechanism. Forgetting that offline users can create content too.

**Model answer.** The system uses device-level online/offline status. The mobile app reports its connection status to a presence service via a lightweight heartbeat. The presence service updates a Redis key: `user:{user_id}:status` with values "online" or "offline" and a timestamp. The fan-out engine checks the presence status before writing. For online users, the fan-out engine writes to the feed list normally. For offline users, the fan-out engine skips the write and records the user ID in a "pending catch-up" set. When an offline user comes online, the app sends a `POST /feed/catchup` request with the last seen cursor. The catch-up handler reads the "pending catch-up" set for that user, queries the content service for all content created by followed users since the last seen cursor, and writes the missing items to the user's feed list in Redis. The catch-up handler then returns the feed to the user. The catch-up operation for a user who was offline for 10 hours processes approximately 500 items (500 followed accounts times 1 post per hour). The catch-up takes 500 milliseconds, which is acceptable because the user expects a delay when coming online. This approach saves 40 percent of fan-out writes (for the 40 percent of users who are offline at any time), reducing the write throughput from 10 million writes per second to 6 million writes per second.

## Tier 3: Expert

### Challenge 3.1: Design a Feed System from 0 to 1 Billion Users (30 minutes)

**Prompt.** Design a feed system that evolves across three growth stages: startup (0 to 1 million users), growth (1 million to 100 million users), and hyper-scale (100 million to 1 billion users). At each stage, describe the architecture, the storage system, the fan-out strategy, and the team structure. At each transition point, describe what changes and why. Provide capacity estimates at each stage.

**Time limit.** 30 minutes.

**Evaluation criteria.** You design an evolutionary architecture that changes at each stage. The startup stage is simple and cheap. The growth stage adds critical optimizations. The hyper-scale stage adds complex distributed systems patterns. You describe the team structure at each stage. You provide realistic capacity numbers.

**Common mistakes.** Designing the hyper-scale architecture for the startup stage (overengineering). Not changing the architecture between stages (underengineering the hyper-scale stage). Not considering the team structure changes.

**Model answer.**

**Stage 1: Startup (0 to 1 million users).** Architecture: monolithic application with PostgreSQL. Feed is generated by a SQL query with a JOIN on the follows table and an index on created_at. One server, no caching. Fan-out strategy: none (fan-out on read by default). Team: 3 backend engineers. Capacity: 1 server with 16 GB RAM, 4 CPU cores. Storage: 500 GB SSD. Read latency: 200 to 500 ms. Write latency: 50 ms. Cost: $100 per month.

**Stage 2: Growth (1 million to 100 million users).** Transition at 1 million users: the SQL JOIN becomes too slow. Add Redis for feed storage. Implement fan-out on write with a 10,000-follower threshold. Fan-out engine runs as a Kafka consumer. Timeline cache with Memcached. Team: 15 engineers (feed team of 5, content team of 4, infrastructure team of 6). Capacity: 50 Redis nodes at 64 GB each, 10 Kafka brokers, 20 application servers. Read latency: 50 to 150 ms. Write latency: 200 ms (async fan-out). Cost: $50,000 per month.

**Stage 3: Hyper-scale (100 million to 1 billion users).** Transition at 100 million users: the Redis cluster hits 500 nodes and becomes expensive. Migrate to Cassandra for feed storage (lower cost per GB). Add ML-based ranking. Multi-region deployment (US, EU, Asia). Dynamic fan-out threshold. Team: 80 engineers (feed platform of 15, ranking of 10, data infrastructure of 12, content of 8, SRE of 15, ML platform of 10, plus supporting roles). Capacity: 1,000 Cassandra nodes, 500 Redis nodes (for timeline cache only), 200 Kafka brokers, 100 ranking servers with GPUs. Read latency: 100 to 250 ms (includes cross-region latency). Write latency: 500 ms (fan-out lag target). Cost: $2 million per month.

### Challenge 3.2: Redesign a Broken Production Feed (30 minutes)

**Prompt.** You join a company where the feed system has been in production for 3 years. The current system has these symptoms:

- Feed reads fail with 503 errors between 2:00 PM and 4:00 PM daily.
- Users see duplicate items and miss items when scrolling.
- The feed list write latency spikes from 5 ms to 2 seconds during viral events.
- The timeline cache hit ratio dropped from 80 percent to 30 percent over the last month.
- The team added 50 percent more Redis nodes last month, but the latency did not improve.

Diagnose each symptom, propose the root cause, and design the fix. Prioritize the fixes by impact.

**Time limit.** 30 minutes.

**Evaluation criteria.** You diagnose each symptom correctly. You prioritize fixes by impact on users. You propose concrete, actionable fixes. You identify the root causes, not just the surface symptoms.

**Common mistakes.** Treating each symptom as an isolated problem without finding the root cause. Proposing complex fixes for simple problems. Not prioritizing the fixes.

**Model answer.**

**Symptom 1: 503 errors between 2:00 PM and 4:00 PM daily.** Root cause: the hydration backend is overwhelmed during peak usage hours. The read path service holds threads waiting for hydration responses, leading to thread pool exhaustion. Fix: add hydration caching (dedicated Memcached cluster for content objects), increase the hydration backend's capacity (double the instances), and implement circuit breaking (if hydration latency exceeds 500 ms, return placeholder content). Priority: critical. This directly affects user-facing availability.

**Symptom 2: Duplicate items and missed items when scrolling.** Root cause: offset-based pagination. When new items are inserted at the top between pages, the offset shifts. Fix: switch to cursor-based pagination. The cursor is the content ID or the timestamp of the last item. Priority: high. This directly affects the user experience.

**Symptom 3: Write latency spikes during viral events.** Root cause: the fan-out engine does not distinguish between regular users and celebrities. A celebrity post with 5 million followers triggers 5 million sequential writes. Fix: implement the fan-out threshold (pull-based for accounts with over 10,000 followers). Also add batch writes (100 writes per batch) and pipeline mode for Redis. Priority: high. This affects feed freshness during viral events.

**Symptom 4: Cache hit ratio dropped from 80 percent to 30 percent.** Root cause: the cache invalidation strategy invalidates the entire timeline cache when any new content arrives. As the user base grew, the invalidation rate increased. Fix: use the "new content flag" approach. Cache entries are not invalidated. Instead, a lightweight flag indicates new content is available. The read path merges new content with cached content. Priority: medium. The drop in cache hit ratio increases read latency but does not break the feature.

**Symptom 5: Adding more Redis nodes did not improve latency.** Root cause: the bottleneck is not Redis node capacity. The bottleneck is the fan-out engine's write throughput. Adding Redis nodes increases parallelism but does not increase the fan-out engine's ability to generate writes. Fix: scale the fan-out engine horizontally (add more worker threads or instances). The bottleneck detection was wrong. Priority: medium. The latency has a ceiling regardless of Redis nodes.

**Priority order.** Fix symptom 1 (availability) today. Fix symptom 2 (pagination) this week. Fix symptom 3 (viral events) this sprint. Fix symptoms 4 and 5 next sprint.

### Challenge 3.3: Design a Feed System with Global Compliance Requirements (30 minutes)

**Prompt.** Your platform is expanding into the European Union, where GDPR requires that users can delete all their data (right to erasure). Design a feed system that supports user data deletion within 30 days of a deletion request. The feed system must remove all references to the deleted user's content from all feed lists. The platform has 200 million users. The fan-out engine writes 20 million feed list entries per second during peak hours.

Your design must include: the data model changes needed for deletion, the deletion workflow (how deleted content is removed from feed lists), the timeline for deletion (how long it takes), and the impact on read path latency during deletion.

**Time limit.** 30 minutes.

**Evaluation criteria.** You design a deletion mechanism that works at scale. You handle the complexity of removing content references from millions of feed lists. You estimate the deletion timeline. You discuss the trade-off between deletion speed and system performance.

**Common mistakes.** Proposing a naive scan-all-feed-lists approach (impractical at 200 million users). Ignoring the read path impact during deletion. Not considering that the fan-out engine continues writing new entries during the deletion process.

**Model answer.** The data model stores content creator ID in each feed list entry. This allows efficient deletion: the system needs to find all feed lists that contain the deleted user's content. For the deletion workflow, use an offline sweep approach rather than an online scan. The sweep works as follows:

Step one: mark the deleted user's content IDs in a Redis set called `deleted_content:{user_id}`. This set is checked during the read path hydration step. If a content ID is in the deleted set, the hydration layer returns a "content unavailable" placeholder instead of the full content. This step is immediate (milliseconds) and makes the content inaccessible to users within seconds.

Step two: a background sweep job iterates over all feed lists and removes entries matching the deleted user's content IDs. The sweep job processes 10,000 feed lists per second using a Redis SCAN cursor. For each feed list, the job uses ZREMRANGEBYSCORE to remove entries with the deleted user's content IDs. The sweep takes approximately 20,000 seconds (5.6 hours) for 200 million feed lists at 10,000 per second.

Step three: after the sweep completes, the Redis set `deleted_content:{user_id}` is cleared. The content references have been physically removed from all feed lists.

The read path impact during deletion is minimal. Step one makes the content invisible immediately via the hydrate-time check. The sweep in step two is purely a cleanup operation that does not affect read latency. The sweep runs with low priority and yields to read operations. If the Redis cluster is under heavy read load, the sweep pauses and resumes later.

The deletion timeline is: content becomes invisible within 1 second (step one). Content references are physically removed within 6 hours (step two). The full deletion cycle meets the GDPR 30-day requirement with a large safety margin.

### Challenge 3.4: Design a Cross-Platform Feed Aggregation System (30 minutes)

**Prompt.** Design a feed system that aggregates content from three different platforms: a microblogging platform (Twitter-like), a video platform (YouTube-like), and a news platform (Medium-like). Users can follow accounts on any of the three platforms. The aggregated feed shows content from all three platforms in a single ranked feed. Each platform has its own content format, its own ranking signals, and its own rate of content creation.

The system has 100 million users. Users follow an average of 50 accounts on the microblogging platform, 20 channels on the video platform, and 10 publications on the news platform. The microblogging platform generates 10,000 posts per second, the video platform generates 500 videos per second, and the news platform generates 200 articles per second.

**Time limit.** 30 minutes.

**Evaluation criteria.** You design a system that handles heterogeneous content types. You address the cross-platform ranking challenge (how do you compare a tweet's relevance to a video's relevance?). You handle different content creation rates across platforms. You discuss the feed storage and fan-out strategy.

**Common mistakes.** Treating all content types the same (a tweet and a video have different engagement patterns). Not normalizing scores across platforms. Ignoring the different content creation rates (10,000 posts per second vs 500 videos per second).

**Model answer.** The aggregated feed system uses a unified feed list with normalized scores. Each content item, regardless of platform, gets a normalized relevance score between 0 and 1. The normalization ensures that a high-engagement tweet can rank above a low-engagement video, even though the raw engagement numbers are different.

The fan-out strategy is hybrid: users with fewer than 10,000 followers on any platform get fan-out on write for that platform's content. Users with more than 10,000 followers get fan-out on read. The fan-out engine has three platform-specific adapters that convert platform-specific content events into a unified format: `{content_id, creator_id, platform, timestamp, normalized_score}`. The unified format is stored in the feed list as a sorted set keyed by normalized_score.

The normalized_score computation uses a platform-specific scoring function and a cross-platform calibration layer. The calibration layer ensures that a score of 0.8 on the microblogging platform means the same relevance as a score of 0.8 on the video platform. Calibration is done by analyzing user engagement patterns across platforms and computing a mapping function.

The feed storage uses Redis sorted sets with the normalized_score as the score. Each feed list entry stores the unified content reference. The feed list size is 1,000 items (higher than the 800-item default) because content from three platforms creates more candidates. During read, the hydration layer determines the platform type from the content reference and calls the appropriate platform's content service for hydration.

The read path applies a per-platform diversity rule: no more than 60 percent of the feed should come from a single platform. This prevents the high-volume microblogging platform from dominating the feed. The diversity rule is applied after ranking but before returning the response.

The capacity estimate: total content creation is 10,700 items per second (10,000 plus 500 plus 200). Total writes to feed lists: 10,700 times 80 average followers equals 856,000 writes per second. The Redis cluster needs approximately 9 nodes (856,000 divided by 100,000) with replication (18 nodes) and headroom (36 nodes). The feed list storage for 100 million users at 1,000 items per user and 50 bytes per item is 5 TB raw (10 TB with replication, 13 TB with overhead). With 128 GB per node, you need 102 nodes for storage. The combined cluster is 128 nodes (sufficient for both throughput and storage).

### Challenge 3.5: Design a Feed System with Ad Insertion (30 minutes)

**Prompt.** Design a feed system that inserts advertisements between organic content items. Ads must be relevant to the user's interests and must not appear more frequently than 1 ad per 5 organic items. The ad insertion must happen in real time during the feed read path. The ad inventory has 10 million active ads. The ad targeting model must select the best ad for each user within 50 milliseconds.

The platform has 50 million daily active users. Each feed request returns 20 items. The ad insertion must not increase the P99 read latency beyond 350 milliseconds (current P99 is 250 milliseconds without ads).

**Time limit.** 30 minutes.

**Evaluation criteria.** You design an ad insertion mechanism that meets the latency budget. You handle the ad targeting model (selecting the best ad for each user in 50 ms). You address the frequency cap (1 ad per 5 organic items). You discuss the failure mode when ad selection fails.

**Common mistakes.** Proposing a separate ad request that adds too much latency. Not considering the frequency cap. Ignoring the case where no relevant ad is available for a user. Treating ad insertion as a batch process rather than a real-time operation.

**Model answer.** The ad insertion is integrated into the read path as an additional step between ranking and response assembly. The read path flow becomes: cache check, feed list read, ranking, ad insertion, deduplication, hydration, response assembly.

The ad insertion step takes the top-ranked 20 organic items and determines ad placement slots. The frequency cap of 1 ad per 5 organic items means that for 20 items, you can insert up to 4 ads. The placement positions are fixed: after items 3, 7, 11, and 15 (every 4th position). If fewer than 4 ads are available, the later slots are left unfilled.

The ad selection model uses a two-stage approach. Stage one (candidate generation): reduce the 10 million active ads to approximately 100 candidates using lightweight filters: user demographics, user browsing history topic clusters, ad targeting keywords, and ad budget remaining. This runs in 10 milliseconds. Stage two (ranking): score the 100 candidates using a lightweight model with 10 signals: predicted click-through rate, ad relevance to the feed content, ad quality score, and ad bid price. The model runs in 5 milliseconds. Total ad selection time: 15 milliseconds.

The ad cache stores the top 100 ad candidates per user topic cluster. The cache is updated every 5 minutes by a background job. Cache hit: the ad selection reads from the cache in 2 milliseconds. Cache miss: the ad selection runs the two-stage model in 15 milliseconds. The cache hit rate is above 90 percent.

The ad content is hydrated separately from organic content. Ad content is smaller (no comments, no likes, no shares) and is cached aggressively. Ad hydration takes 10 milliseconds.

The new P99 read latency budget: cache check 10 ms, feed list read 20 ms, ranking 30 ms, ad selection 15 ms, ad hydration 10 ms, organic hydration 100 ms, deduplication 10 ms, response assembly 15 ms, network overhead 50 ms. Total: 260 ms. The ad insertion adds only 25 milliseconds to the read path, keeping the total well under the 350 ms target.

Failure mode: if the ad selection model fails or times out, the feed returns without ads. The organic items are returned as-is. The ad insertion step has a 50 ms timeout. If the timeout fires, the ad step is skipped. This ensures that ad insertion failures do not break the feed read path. The ad failure rate is monitored and alerted if it exceeds 0.1 percent of feed requests.

### Challenge 2.6: Design a Feed System with Topic-Based Subscriptions (20 minutes)

**Prompt.** Design a feed system where users subscribe to topics (technology, sports, politics, etc.) in addition to following individual creators. A user following a topic receives content from any creator who posts about that topic, even if the user does not follow the creator directly. The platform has 10,000 topics and 50 million users. Each user subscribes to an average of 8 topics. Each topic has 10,000 to 1 million subscribers.

Your design must handle: topic-based content routing (how does content tagged with a topic reach the topic's subscribers?), topic subscription management (users can subscribe and unsubscribe), and the intersection of topic-based and creator-based feeds (how do you merge the two sources without duplication?).

**Time limit.** 20 minutes.

**Evaluation criteria.** You handle the topic subscription as a separate fan-out path. You address the deduplication between topic-based and creator-based content. You estimate the additional write amplification from topic-based subscriptions.

**Common mistakes.** Treating topics like individual creators (fan-out on write to all subscribers is too expensive for topics with 1 million subscribers). Not handling the deduplication between topic-based and creator-based content.

**Model answer.** The system uses two fan-out paths: creator-based and topic-based. The creator-based path uses the standard hybrid fan-out (push for accounts under 10,000 followers, pull for accounts above). The topic-based path uses fan-out on read exclusively because topics can have millions of subscribers. When a creator tags a post with a topic, the fan-out engine writes the post ID to a "topic feed" Redis sorted set: `topic:feed:{topic_id}`. The topic feed is a chronological sorted set of post IDs tagged with that topic. The topic feed has a larger capacity (2,000 items) because topics aggregate content from many creators.

When a user reads their feed, the read path performs a two-way merge. Source one: the user's creator-based feed list (from followed creators). Source two: the topic feed lists for each subscribed topic. The merge interleaves items from both sources sorted by score. Deduplication removes items that appear in both sources (a post from a followed creator that is also tagged with a subscribed topic appears only once).

The read path latency impact: the merge operation adds 20 to 50 milliseconds depending on the number of subscribed topics. For 8 topics, the read path queries 8 topic feed lists and merges them with the creator feed list. The total read latency is 150 to 250 milliseconds, within the acceptable range.

The additional write amplification: each post tagged with a topic generates 1 write to the topic feed list (fan-out on read means no writes to individual subscriber feed lists). For 10,000 topics and 500,000 topic-tagged posts per day, the additional writes are 500,000 per day, which is negligible compared to the creator-based writes.

### Challenge 2.7: Design a Feed Archival and Retrieval System (20 minutes)

**Prompt.** Design a system that archives user feeds for historical analysis. The archive must store every feed item that was shown to each user, in the order it was shown, for the last 90 days. The archive is used for: debugging feed quality issues, auditing content moderation decisions, and training ranking models. The system has 100 million users. Each user views 50 feed items per day on average.

Your design must include: the storage format for archived feeds, the write throughput for archiving, the query pattern for retrieval, and the data retention policy.

**Time limit.** 20 minutes.

**Evaluation criteria.** You design an archival system that can keep up with the feed read throughput. You choose an appropriate storage system for the archive (not Redis, because archive data is different from feed storage). You address the query patterns (retrieving a specific user's feed history for a specific day).

**Common mistakes.** Storing archives in Redis (too expensive for 90 days of data). Not considering the write throughput (100 million users times 50 items equals 5 billion archive entries per day). Designing a retrieval system that is too slow for debugging purposes.

**Model answer.** The archive uses a separate storage system from the feed storage. The archive storage is Apache Cassandra (or Amazon S3 with a partition scheme). The archive data model: `CREATE TABLE feed_archive (user_id text, date text, hour int, item_position int, content_id text, score float, ranking_model_version text, PRIMARY KEY ((user_id, date), hour, item_position))`. The partition key is (user_id, date), which allows efficient retrieval of a user's feed for a specific day.

The archive write happens asynchronously after the feed response is returned to the user. The feed read service publishes an archive event to a Kafka topic: `{user_id, content_id, score, timestamp, model_version}`. A background archive consumer reads the Kafka topic and writes to Cassandra in batches of 100. The archive consumer can handle 5 billion events per day with 100 workers (each worker processes 50,000 events per hour).

The retrieval query: `SELECT * FROM feed_archive WHERE user_id = ? AND date = ? ORDER BY hour, item_position`. This query returns all items shown to a user on a specific day, in the order they were shown. The query takes 5 to 50 milliseconds depending on the partition size.

The retention policy: data is retained for 90 days. A TTL (time-to-live) is set on each Cassandra row at write time. Cassandra automatically deletes expired rows. The TTL approach avoids the need for a separate cleanup job. The storage cost for 90 days: 5 billion events per day times 90 days times 200 bytes per event equals 90 TB. Cassandra with 3x replication requires 270 TB. Using 500 GB nodes, the archive cluster needs 540 nodes. This is a significant cost, but the archive is essential for debugging and model training.

### Challenge 2.8: Design a Feed Migration Strategy (20 minutes)

**Prompt.** Your platform currently uses a monolithic feed system with a PostgreSQL database. You are migrating to a distributed feed system with Redis storage and a fan-out engine. The migration must happen without downtime. You have 5 million users and 100 million feed items. The old system uses a SQL query with a JOIN. The new system uses Redis sorted sets.

Design the migration strategy. Include: the data migration plan (how do you move feed data from PostgreSQL to Redis?), the cutover plan (how do you switch from the old system to the new system?), the rollback plan (what happens if the new system has issues?), and the validation plan (how do you verify that the new system produces correct feeds?).

**Time limit.** 20 minutes.

**Evaluation criteria.** You design a zero-downtime migration. You handle the data migration (transforming PostgreSQL data into Redis sorted sets). You have a rollback plan. You validate the new system's correctness.

**Common mistakes.** Proposing a big-bang cutover (all users switch at once, high risk). Not having a rollback plan. Forgetting to migrate historical feed data. Not validating that the new system produces the same feeds as the old system.

**Model answer.** The migration uses a phased, shadow-read approach:

Phase one (preparation, 1 week): deploy the new feed system in parallel with the old system. The new system writes to Redis but does not serve reads yet. The old system continues to serve all feed reads. A background migration job reads the PostgreSQL follows table and posts table, and populates the Redis sorted sets for all 5 million users. The migration job runs at 50,000 users per hour, completing in 4 days. During this phase, the team validates that the Redis feed lists match the PostgreSQL query results for a sample of 10,000 users.

Phase two (shadow reads, 1 week): the feed read path sends each request to both the old system (PostgreSQL) and the new system (Redis). The old system's response is returned to the user. The new system's response is logged and compared offline. The comparison checks: item count, item order, and item content. Any discrepancies are investigated and fixed. The comparison runs daily and produces a report of correctness percentage.

Phase three (canary cutover, 1 week): 1 percent of users are switched to the new system. Their feed reads go to Redis instead of PostgreSQL. The team monitors error rates, latency, and user feedback. If the new system performs well for 24 hours, the canary is expanded to 5 percent, then 10 percent, then 25 percent, then 50 percent, then 100 percent over 7 days. Each expansion happens only after the previous stage shows no issues.

Phase four (cleanup, 1 week): after all users are on the new system, the old PostgreSQL system is kept running for 7 days as a rollback target. On day 7, if no rollback is needed, the old system is decommissioned.

Rollback plan: if the new system has issues at any point during the canary cutover, the affected users are switched back to the old system. The rollback is instantaneous because the old system is still running and serving reads. The rollback affects only the canary users, not the entire user base.

### Challenge 3.6: Design a Feed System with Reverse-Chronological Recovery (30 minutes)

**Prompt.** Your feed system uses a ranked feed (ML-based scoring). The ranking model has a bug that incorrectly scored all content from the last 2 hours as zero. Users saw a feed that was missing all recent content. The bug was active for 2 hours before it was detected and fixed. You need to recover the feeds for all affected users (50 million users) without causing a cache stampede or overwhelming the system.

Your design must include: the detection mechanism (how do you detect the ranking bug?), the recovery strategy (how do you regenerate the correct feeds?), the capacity planning for the recovery (how many resources do you need?), and the prevention mechanism (how do you prevent this from happening again?).

**Time limit.** 30 minutes.

**Evaluation criteria.** You design a detection mechanism that catches ranking bugs quickly. You design a recovery strategy that does not overwhelm the system. You estimate the recovery time and resources needed. You propose a prevention mechanism.

**Common mistakes.** Proposing a naive approach of regenerating all 50 million feeds at once (causes cache stampede). Not having a detection mechanism (the bug was active for 2 hours before detection). Not considering that users may have already seen the incorrect feed and will not re-read it.

**Model answer.**

Detection mechanism: the ranking model's score distribution is monitored in real time. A dashboard shows the average score, the score distribution (histogram), and the percentage of items with zero scores. An alert triggers if the percentage of zero-score items exceeds 1 percent. Additionally, a canary user (internal employee) reads their feed every 5 minutes and the system compares the feed content to an expected baseline. If the canary's feed deviates from the baseline by more than 20 percent, an alert fires. These two mechanisms would detect the bug within 5 minutes, not 2 hours.

Recovery strategy: the recovery uses a reverse-chronological feed regeneration. Instead of regenerating the ranked feed, the system temporarily switches to a chronological feed for the affected users. The chronological feed is simple to generate: read the feed list from Redis sorted by timestamp (not by the buggy score). The chronological feed shows all items in the correct order. The ranked feed is regenerated in the background using a staggered approach:

- Batch 1 (first 5 minutes): regenerate the top 1 million most active users (users who read their feed most frequently). These users are most likely to notice the bug.
- Batch 2 (next 10 minutes): regenerate the next 5 million users.
- Batch 3 (next 30 minutes): regenerate the remaining 44 million users.

Each batch writes the correct scores to the Redis sorted sets and invalidates the timeline cache for those users. The staggered approach prevents a cache stampede. The total recovery time is 45 minutes.

Capacity planning: the recovery regeneration requires the same capacity as a normal ranking re-score job. The ranking engine has 16 workers that re-score 10,000 feed lists per second each. At this rate, 50 million feed lists take 312 seconds (5.2 minutes) to regenerate. With the staggered approach and cache invalidation overhead, the recovery takes 45 minutes, which is acceptable.

Prevention mechanism: the ranking model deployment uses a staged rollout. A new model version is deployed to 1 percent of users first. The score distribution is monitored for 1 hour. If the score distribution is within expected bounds, the model is rolled out to 10 percent, then 50 percent, then 100 percent. Each stage requires manual approval. The bug would have been caught at the 1 percent stage, affecting only 500,000 users instead of 50 million.

### Challenge 3.7: Design a Feed System with Per-User Retention Policies (30 minutes)

**Prompt.** Your platform has three tiers of users: free users (70 percent, feed retention of 200 items), premium users (20 percent, feed retention of 800 items), and enterprise users (10 percent, feed retention of 5,000 items). Each tier has different SLA requirements: free users tolerate 500 ms P99 latency, premium users require 300 ms P99, and enterprise users require 100 ms P99.

Design a feed system that serves all three tiers efficiently. The total user base is 200 million. Your design must handle the different retention policies, the different latency requirements, and the cost implications of serving enterprise users with 5,000-item feed lists.

**Time limit.** 30 minutes.

**Evaluation criteria.** You design a tiered system that allocates resources proportionally to the SLA requirements. You handle the different feed list sizes (200, 800, 5,000 items). You estimate the cost difference between tiers. You discuss how the fan-out strategy differs between tiers.

**Common mistakes.** Treating all users the same (ignoring the different retention policies). Using the same Redis cluster for all tiers (enterprise users with 5,000 items would dominate the storage cost). Not adjusting the fan-out strategy per tier.

**Model answer.** The tiered feed system uses separate Redis clusters per tier:

Free tier cluster (140 million users, 200 items per feed list, 40 bytes per item): raw storage is 140 million times 200 times 40 equals 1.12 TB. With replication and overhead: 2.91 TB. Cluster size: 64 shards (128 nodes). Each node stores 45 GB. The fan-out strategy is fan-out on read because free users have lower SLA requirements and the cost of pre-computed feed lists is not justified. The read path for free users is pull-based: query recent content from followed users, merge, sort, and return the top 200 items. The P99 read latency is 300 to 500 ms.

Premium tier cluster (40 million users, 800 items per feed list, 40 bytes per item): raw storage is 40 million times 800 times 40 equals 1.28 TB. With replication and overhead: 3.33 TB. Cluster size: 64 shards (128 nodes). Each node stores 52 GB. The fan-out strategy is hybrid (push for accounts under 10,000 followers, pull for accounts above). The read path uses a timeline cache with 30-second TTL. The P99 read latency is 150 to 300 ms.

Enterprise tier cluster (20 million users, 5,000 items per feed list, 40 bytes per item): raw storage is 20 million times 5,000 times 40 equals 4 TB. With replication and overhead: 10.4 TB. Cluster size: 256 shards (512 nodes). Each node stores 40 GB. The fan-out strategy is fan-out on write for all users (enterprise users expect instant feed updates). The read path uses a dedicated timeline cache with 10-second TTL. The enterprise tier has dedicated fan-out engine workers (32 workers) to ensure low fan-out lag. The P99 read latency is 50 to 100 ms.

Cost analysis: free tier cost is $0.0005 per user per month (shared infrastructure, pull-based fan-out). Premium tier cost is $0.002 per user per month (dedicated Redis cluster, hybrid fan-out). Enterprise tier cost is $0.01 per user per month (dedicated Redis cluster, dedicated fan-out workers, dedicated timeline cache, lowest latency). The 10x cost difference between free and enterprise tiers justifies the premium pricing.

The fan-out strategy differs per tier because the cost-benefit trade-off is different. For free users, the cost of fan-out on write is not justified by the lower SLA requirement. For enterprise users, the cost of fan-out on write is justified by the strict SLA requirement. The system routes fan-out writes based on the user's tier: the fan-out engine checks the user's tier and writes to the appropriate Redis cluster.

### Challenge 3.8: Design a Feed System with Multi-Format Content Support (30 minutes)

**Prompt.** Your platform supports four content formats: text posts (up to 4,000 characters), images (up to 10 per post), videos (up to 10 minutes), and polls (up to 4 options). Each format has different engagement patterns, different rendering requirements, and different storage and CDN costs. The feed must mix all four formats.

Design a feed system that handles multi-format content. Your design must include: the content storage strategy (how do you store different formats?), the feed list data model (how do you represent different formats in the feed list?), the ranking model (how do you compare engagement across formats?), and the rendering pipeline (how do you ensure fast loading of images and videos in the feed?).

**Time limit.** 30 minutes.

**Evaluation criteria.** You design a unified content model that handles four formats. You address the cross-format ranking challenge (a video view is not the same engagement as a text post like). You handle the rendering pipeline (images and videos need to load fast).

**Common mistakes.** Treating all formats the same in the ranking model (a video view is a stronger signal than a text post like). Not considering the rendering pipeline (images and videos need thumbnail generation and CDN optimization). Storing media in the feed list (should be references only).

**Model answer.** The content storage strategy uses a unified metadata table with format-specific extension tables. The unified table stores: `content_id, creator_id, format (text|image|video|poll), created_at, engagement_counts (JSON)`. The format-specific tables store: `text_posts(content_id, text, read_time)`, `image_posts(content_id, image_urls[], aspect_ratios[])`, `video_posts(content_id, video_url, thumbnail_url, duration, resolution)`, `poll_posts(content_id, options[], vote_counts[])`. The feed list stores only the content_id and format type (1 byte). The format type allows the hydration layer to call the correct format-specific hydration service.

The ranking model uses format-specific engagement signals normalized to a common scale. A video view is weighted 3x a text post like because a video view requires more user attention. The normalization formula: `normalized_engagement = sum(signal_value * signal_weight * format_multiplier)`. The format multiplier for each signal is calibrated by analyzing user behavior: how much time does a user spend on a video vs a text post? how likely is a user to share a video vs a poll? The calibration runs weekly and adjusts the multipliers based on the latest data.

The rendering pipeline uses a progressive loading strategy. The feed response includes a thumbnail URL for image and video content, not the full-resolution media. The thumbnail is pre-generated at 320x320 pixels and cached on the CDN. The thumbnail loads in 50 to 100 milliseconds. The full-resolution image or video loads on demand when the user taps or scrolls to the item. The CDN serves the thumbnail from edge locations, reducing the load on the origin server.

The feed list data model uses a unified sorted set. Each entry is a content_id with the format type encoded in the ID prefix: `txt_12345`, `img_12345`, `vid_12345`, `pol_12345`. The score is the normalized relevance score. The feed list size is 1,000 items (higher than the default 800) because the diversity of formats requires more candidates for the ranking model to select from.

The read path applies a format diversity rule: no more than 50 percent of feed items should be of the same format. This prevents a viral video format from dominating the feed. The diversity rule is applied after ranking but before the response is returned. If the top 20 items contain 12 videos, the lowest-ranked video is replaced with the highest-ranked non-video item. The diversity rule ensures that users see a mix of content formats.
