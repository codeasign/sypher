## Naive Implementation

### Single-Server Monolith

The naive implementation is a single-server web application with a relational database. Content is stored in a `posts` table. The feed is generated by a SQL query:

```sql
SELECT p.* FROM posts p
JOIN follows f ON p.user_id = f.followee_id
WHERE f.follower_id = ?
ORDER BY p.created_at DESC
LIMIT 20 OFFSET ?;
```

This architecture works for up to 10,000 users with reasonable latency. At 100,000 users, the JOIN becomes a performance disaster. The `follows` table has millions of rows, and the `posts` table has tens of millions. The query planner chooses a nested loop join, which takes 2 to 10 seconds depending on the number of followed accounts.

At 1 million users, the single-server database cannot handle the query volume. The CPU is at 100 percent. The connection pool is exhausted. The application returns 503 errors.

### Naive Fan-Out on Write

The first attempt at scaling uses fan-out on write with a single Redis instance. A background worker reads new posts from a queue and writes the post ID to each follower's feed list in Redis. The feed list is a Redis sorted set keyed by `user:feed:{user_id}`.

This architecture handles 50,000 users with 200 posts per second. The Redis instance processes 40,000 writes per second (200 posts times 200 average followers). At 500,000 users, the Redis instance hits 400,000 writes per second. Redis maxes out at approximately 100,000 writes per second on a single node. The worker falls behind. The queue grows unbounded.

The naive fan-out on write fails at scale because a single Redis node cannot handle the write throughput. The system lacks sharding, replication, and backpressure mechanisms.

## Production-Grade Architecture

### High-Level Design

The production-grade architecture distributes each responsibility across a dedicated service cluster. The system uses a microservices architecture with 10 to 15 services, each independently scalable.

**Content Service.** Handles content creation, storage, and retrieval. Stores content metadata in a distributed key-value store (Cassandra). Stores media in an object store (S3). Publishes content events to a stream (Kafka).

**Follower Graph Service.** Manages the follow relationship graph. Stores follower sets in a partitioned key-value store. Provides gRPC APIs for follower list lookup and membership checks.

**Fan-out Service.** Reads content events from Kafka, queries the Follower Graph Service, and writes feed list entries to the Feed Storage cluster. Runs as a horizontally scalable worker pool.

**Feed Storage Cluster.** A sharded Redis cluster with 100 to 200 nodes. Each node handles a subset of user feed lists. The cluster uses consistent hashing for shard assignment and supports automatic failover.

**Feed Read Service.** A stateless HTTP service that handles feed read requests. Reads from the Feed Storage cluster, applies ranking and filtering, and calls the Content Service for hydration.

**Ranking Service.** A background job that re-computes the ranking of feed items. Runs periodically and writes updated scores back to the Feed Storage cluster.

**Timeline Cache.** A distributed Memcached or Redis cache that stores hydrated feed responses for active users. Reduces the read path latency by 60 to 80 percent.

### Component Deep Dive: Feed Storage Cluster

The feed storage cluster is a Redis Cluster with 128 shards (256 nodes with replication). Each shard is a Redis primary with one replica. The primary handles writes and reads. The replica provides failover and can serve read traffic under heavy load.

Shard assignment uses a hash ring. The feed list key `user:feed:{user_id}` is hashed and assigned to a shard. This ensures all feed list operations for a single user go to the same shard. The consistent hash ring supports adding and removing shards with minimal data movement.

Each shard stores feed lists for approximately 781,250 users (100 million users divided by 128 shards). Each feed list has 800 items at 40 bytes per item, totaling 32 KB per feed list. Each shard stores approximately 25 GB of feed list data (781,250 times 32 KB).

The Redis cluster uses slot-based sharding with 16,384 hash slots. Each shard is responsible for 128 slots (16,384 divided by 128). The cluster's hash ring maps each user ID to a slot: `slot = CRC16(user_id) mod 16384`. Redis Cluster's `MOVED` redirection ensures that clients are always directed to the correct shard.

### Component Deep Dive: Write Path Pipeline

The write path is a multi-stage pipeline that transforms a content creation event into feed list entries. Each stage runs as a separate service or worker pool.

**Stage 1: Event Ingestion.** The content service publishes the event to a Kafka topic. The topic has 64 partitions. The partition key is the content ID, ensuring that events for the same content go to the same partition. This enables ordered processing: the fan-out engine processes events for each content item sequentially.

**Stage 2: Follower List Lookup.** The fan-out worker reads the event from the Kafka partition. It calls the Follower Graph Service with the creator ID. The service returns the follower list as a paginated response: 10,000 followers per page. For a celebrity with 10 million followers, the fan-out worker retrieves 1,000 pages. Each page request takes 5 to 10 milliseconds.

**Stage 3: Fan-Out Decision.** The fan-out worker checks the follower count against the fan-out threshold. If the count is below the threshold, the worker proceeds to write to all follower feed lists. If the count is above the threshold, the worker writes a flag to a "celebrity content available" key and stops. The pull-based read path handles the rest.

**Stage 4: Batch Write to Redis.** The fan-out worker batches follower IDs into groups of 100. For each batch, the worker constructs a Redis pipeline with 100 ZADD commands. The pipeline is sent to the appropriate Redis shard. The shard is determined by the follower's user ID hash. The worker sends pipelines in parallel to all affected shards.

**Stage 5: Write Confirmation and Retry.** The Redis cluster acknowledges the pipeline writes. If a write fails (network error, cluster failover), the worker retries the batch. The retry count is limited to 3. After 3 retries, the failed batch is logged and queued for manual recovery.

**Stage 6: Commit Offset.** After all batches are completed, the worker commits the Kafka offset. This marks the event as processed. If the worker crashes before committing, the event is reprocessed by another worker in the same consumer group. The at-least-once semantics ensure no events are lost.

### Component Deep Dive: Fan-out Service

The fan-out service is a Kafka consumer group with 64 workers. Each worker processes content events from a Kafka partition. The worker reads the event, calls the Follower Graph Service to get the follower list, and writes feed list entries to the Feed Storage Cluster.

The fan-out service uses a two-phase write strategy. Phase one: write the content reference to the first 500 followers. Phase two: write to the remaining followers in batches of 100. This ensures that followers with the strongest affinity (earliest followers, most engaged followers) receive the content first.

For celebrity accounts with over 10,000 followers, the fan-out service switches to pull-based mode. The worker writes a "new content available" flag to a dedicated Redis key. When a follower reads their feed, the read path checks this flag and pulls the celebrity's recent posts individually.

### Read Path Deep Dive

The read path is the second most critical component after the fan-out engine. It handles 100,000 requests per second at peak, with a P99 latency target of 300 milliseconds.

The read path service is a stateless HTTP service deployed behind a load balancer. It has 32 instances, each running on 8 CPU cores with 32 GB RAM. Each instance handles 3,125 requests per second.

The read path request flow has three levels of caching:

**Level 1: CDN Cache.** Feed responses are cached at the CDN edge for anonymous users (not logged in). The CDN cache has a TTL of 60 seconds. For logged-in users, the CDN cache is bypassed because the feed is personalized.

**Level 2: Timeline Cache.** The timeline cache stores the assembled and hydrated feed for logged-in users. The cache is distributed Memcached with 50 nodes. Each cache entry has a TTL of 30 minutes. The cache key is `timeline:{user_id}`. The cache value is the JSON-encoded feed response (approximately 4 KB for 20 items).

**Level 3: Feed List Cache.** The feed list cache stores the raw sorted set data from Redis. This cache is co-located on the read path instance (in-process memory). The cache stores feed list data for the most recently accessed 100,000 users. The cache has a TTL of 60 seconds. If the feed list changes (new content arrives), the cache is invalidated.

## Integration Patterns

### Pattern 1: Fan-Out with Event Sourcing

In this pattern, the content creation event is the source of truth. The fan-out engine is an event processor that transforms the content event into feed list writes. The event stream (Kafka) provides durability and replayability.

If the fan-out engine crashes mid-operation, events remain in the Kafka topic. The engine resumes from the last committed offset, reprocessing any events that were in flight. This guarantees at-least-once delivery to the feed lists.

The event sourcing pattern also enables point-in-time recovery. To recreate the feed state as of a specific timestamp, the system replays all content events up to that timestamp through the fan-out engine.

### Pattern 2: Fan-Out with Change Data Capture

In this pattern, the content service does not explicitly publish events. Instead, a CDC (Change Data Capture) system monitors the content database's write-ahead log and publishes changes to Kafka. The fan-out engine consumes these change events.

This pattern reduces the coupling between the content service and the fan-out engine. The content service does not need to know about the feed system. It only writes to its own database. The CDC system transparently captures all content changes.

The trade-off is increased latency. CDC systems typically have a capture delay of 100 to 500 milliseconds. For feeds where freshness is critical, explicit event publishing is preferred.

### Pattern 3: Multi-Tenant Feed Isolation

In this pattern, each tenant (enterprise customer) has an isolated feed storage shard. Tenant A's feed lists are on shards 0 to 15. Tenant B's feed lists are on shards 16 to 31. The fan-out engine routes writes to the appropriate shard based on the tenant ID.

This pattern prevents noisy-neighbor problems. A viral post from one tenant cannot overwhelm the feed storage for another tenant. Each tenant's capacity can be independently scaled.

The trade-off is increased operational complexity. The system must manage per-tenant capacity planning, per-tenant failover, and per-tenant monitoring. The shard assignment is fixed per tenant, which can lead to uneven resource utilization.

### Pattern 4: Serverless Fan-Out

In this pattern, the fan-out engine is implemented as serverless functions (AWS Lambda). Each content event triggers a Lambda function that fans out to followers. The Lambda function calls the Follower Graph Service and issues parallel Redis writes.

This pattern scales automatically. Lambda handles 10,000 concurrent invocations without any cluster management. The fan-out capacity grows with the event volume.

The trade-off is cold start latency and execution timeout limits. Lambda functions have a maximum execution time of 15 minutes. A user with 10 million followers may require more than 15 minutes to complete the fan-out. The solution is to break the follower list into batches and process each batch as a separate Lambda invocation.

## Multi-Region

### Primary-Region Fan-Out

In a multi-region deployment, the fan-out engine runs in the primary region. This region has the most up-to-date follower graph and the lowest latency to the content database. Content events are published in the primary region. The fan-out engine writes to feed storage clusters in all regions.

### Read Replicas for Feed Storage

Feed storage is replicated across regions. Each region has its own Redis cluster with feed list data replicated from the primary region. Replication uses Redis replication streams or application-level replication.

Cross-region replication latency depends on the geographic distance. US-East to US-West is approximately 60 to 80 milliseconds. US-East to EU-West is approximately 80 to 120 milliseconds. This means a user in Europe sees a new feed item 120 milliseconds after a user in the US East Coast.

### Local Read Path

Each region has its own Feed Read Service instances that read from the local feed storage cluster. The read path is entirely regional: the feed list read, ranking, hydration, and response assembly all happen within the same region. This keeps the read latency low (under 50 milliseconds for the regional portion).

### Regional Tailoring

The ranking engine applies region-specific signals. A post about a local event in Tokyo scores higher for users in Japan. The regional ranking is computed by a local ranking service that adjusts scores based on geographic affinity, language, and local trending topics.

## Capacity Planning

### Storage Capacity

For 100 million active users:
- Feed list items per user: 800
- Bytes per item: 40
- Feed list storage: 100 million x 800 x 40 = 3.2 TB
- Redis replication factor: 2 (primary + replica)
- Redis overhead (RDB, AOF, connections): 30 percent
- Total Redis storage: 3.2 TB x 2 x 1.3 = 8.32 TB

With 128 shards, each shard stores approximately 65 GB of Redis data. Each shard runs on a server with 128 GB RAM (including overhead for other Redis data structures, client buffers, and operating system).

### Compute Capacity

Fan-out engine: 64 workers, each handling 1,000 fan-out operations per second. Total capacity: 64,000 fan-outs per second. Peak load: 50,000 fan-outs per second. Headroom: 28 percent.

Feed Read Service: 32 instances, each handling 5,000 requests per second. Total capacity: 160,000 reads per second. Peak load: 100,000 reads per second. Headroom: 60 percent.

Ranking Service: 16 workers, each re-scoring 10,000 feed lists per second. Total capacity: 160,000 re-scores per second. Full re-score of all active users every 5 minutes.

### Network Bandwidth

Fan-out write traffic: 10 million writes per second x 80 bytes per write = 800 MB/s. This requires 10 Gbps network links on the fan-out workers and the feed storage nodes.

Read traffic: 100,000 reads per second x 4 KB per response = 400 MB/s. This requires 5 Gbps network links on the Feed Read Service nodes.

## Operations

### Monitoring and Alerting

The feed generation system requires monitoring at three levels:

**Pipeline Health.** Fan-out lag (seconds between content creation and feed list write). Queue depth (number of pending content events). Failure rate (percentage of fan-out operations that fail). Alert if fan-out lag exceeds 10 seconds or failure rate exceeds 0.1 percent.

**Storage Health.** Feed storage node CPU and memory utilization. Key count and memory usage per shard. Write and read latency per shard. Alert if any shard exceeds 80 percent memory utilization or P99 latency above 50 milliseconds.

**Read Path Health.** Feed read latency (P50, P99, P999). Cache hit ratio (target: above 75 percent). Error rate (target: below 0.01 percent). Hydration latency (target: P99 below 120 milliseconds).

### Scaling Operations

**Shard Rebalancing.** When a shard approaches capacity, the operations team adds new shards to the Redis cluster. The consistent hash ring is updated, and a portion of the keys is migrated to the new shards. Migration happens without downtime, using Redis cluster's live re-sharding capability.

**Fan-Out Worker Scaling.** The number of fan-out workers is adjusted based on the Kafka consumer lag. If the lag exceeds 10,000 events, an additional 16 workers are added. Scaling is automated through a controller that monitors the lag metric.

**Feed List Size Adjustment.** If feed reads show high cache miss rates due to users scrolling deep into their feed, the feed list max size is increased from 800 to 1,200. This is a dynamic configuration change that takes effect within 60 seconds.
