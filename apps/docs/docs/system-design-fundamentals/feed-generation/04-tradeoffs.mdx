## Core Tension

The fundamental tension in feed generation is between write efficiency and read efficiency. Every decision in the feed system is a negotiation between how much work happens when content is created and how much work happens when a feed is viewed.

Write efficiency means the system handles content creation with minimal cost. A single post should require minimal compute, storage, and network resources. Read efficiency means the system delivers a feed with minimal latency. A feed request should return in under 200 milliseconds regardless of how many accounts the user follows.

These two goals are in direct opposition. Fan-out on write optimizes for reads at the cost of writes. Fan-out on read optimizes for writes at the cost of reads. The hybrid approach attempts to find a middle ground but introduces complexity.

This tension is not a bug. It is the design problem that the feed generation pattern exists to solve. The decision is not which approach is universally better, but which approach is better for your specific workload characteristics.

## Decision Matrix

### 1. Fan-Out on Write vs Fan-Out on Read

| Dimension | Fan-Out on Write | Fan-Out on Read |
|-----------|-----------------|-----------------|
| Read latency | 10-30 ms | 200-2,000 ms |
| Write amplification | 100x-10,000x | 1x |
| Storage cost | High (pre-computed feed lists) | Low (no feed lists) |
| Cold start | No issue | Slow for new users |
| Celebrity handling | Breaks at scale | Handles naturally |
| Consistency | Eventually consistent | Strongly consistent |
| Implementation complexity | High | Low |

### 2. Chronological vs Ranked Feed

| Dimension | Chronological | Ranked |
|-----------|--------------|--------|
| Algorithm complexity | O(1) per item | O(n) per item |
| Freshness | Highest first | May bury fresh content |
| Engagement impact | Low | High (30-50% improvement) |
| Computational cost | Minimal | High (background jobs) |
| Predictability | High | Low (users may not understand ranking) |
| A/B testing required | No | Yes |

### 3. Redis vs Cassandra for Feed Storage

| Dimension | Redis | Cassandra |
|-----------|-------|-----------|
| Read latency | 1-5 ms | 5-20 ms |
| Write throughput | 100K/node | 1M/node |
| Data model | Sorted set (native) | Requires custom sorted merge |
| Persistence | Optional (RDB/AOF) | Native (SSTables) |
| Scaling | Cluster mode (complex) | Linear (native) |
| Memory cost | All data in RAM | Disk + cache |

### 4. Single-Region vs Multi-Region

| Dimension | Single-Region | Multi-Region |
|-----------|--------------|-------------|
| Read latency | 10-30 ms (global avg 150 ms) | 10-30 ms (regional) |
| Write complexity | Low | High (cross-region replication) |
| Disaster recovery | 1-6 hours RTO | Minutes RTO |
| Cost | 1x | 3x-5x |
| Consistency | Strong | Eventual (cross-region) |

### 5. Push vs Pull for Celebrity Accounts

| Dimension | Push (Fan-Out) | Pull (On-Demand) |
|-----------|---------------|-----------------|
| Fan-out cost | 1M writes per post | 0 writes |
| Read cost per follower | 0 extra queries | 1 extra query per celebrity followed |
| Follower experience | Instant | 100-500 ms delay |
| Storage cost | 1M entries per post | 0 entries |
| Best for | Followers < 10K | Followers > 10K |

### 6. Synchronous vs Asynchronous Hydration

| Dimension | Synchronous Hydration | Asynchronous Hydration |
|-----------|----------------------|----------------------|
| Feed response time | 100-200 ms | 10-50 ms (initial), 200 ms (full) |
| User experience | Complete feed immediately | Placeholder then content loads |
| Implementation | Simple (single request) | Complex (streaming response) |
| Cache efficiency | Batch fetch is efficient | May fetch same content multiple times |
| Error handling | Fail the entire feed | Partial feed with retry |

## CAP Theorem Applied

The feed generation system is AP (Available and Partition Tolerant) in the CAP theorem framework. Consistency is sacrificed for availability and partition tolerance.

During a network partition, the feed system continues to serve reads and writes. Different regions may have different versions of the feed. A user in the US may see a post 120 milliseconds before a user in Europe. This is acceptable because feeds are eventually consistent. The system accepts that some users see content slightly later than others.

The system is not CP because it cannot afford to reject reads during a partition. If the feed went down during a network partition, users would see an empty feed or an error message. This would be worse than serving a feed that is a few seconds stale.

The system is not CA because it must tolerate network partitions. A feed system that cannot handle a network failure would be unreliable. The system would be down every time a network switch fails or a fiber optic cable is cut.

## Failure Modes

### Failure Mode 1: Fan-Out Engine Crash

The fan-out engine crashes mid-operation. Events in the Kafka topic are not processed. The consumer lag increases. Content created after the crash is not written to follower feed lists. Users see stale feeds.

**Impact.** Content creation is still possible, but followers do not see the content. The feed becomes increasingly stale as the lag grows. After 10 minutes of lag, users see content that is 10 minutes old at the top of their feed.

**Mitigation.** The fan-out engine is a Kafka consumer group. On restart, it resumes from the last committed offset. Events processed before the crash are not re-processed (at-least-once semantics). The consumer lag is monitored and alerted. If the lag exceeds 10 seconds, the read path falls back to pull-based assembly.

### Failure Mode 2: Feed Storage Node Failure

A Redis node in the feed storage cluster fails. The node's primary is unreachable. Its replica is promoted to primary by the Redis cluster's failover mechanism. However, the replica may have stale data if replication lag was present at the time of failure.

**Impact.** Users whose feed lists are on the failed node see truncated or stale feeds. The promoted replica may be missing the last 100 to 500 milliseconds of feed list writes. These writes are lost.

**Mitigation.** Redis replication with synchronous replication minimizes data loss. The fan-out engine retries failed writes. If the write to the primary fails, the engine retries the write against the new primary after failover. The maximum window of data loss is equal to the replication lag, typically under 5 milliseconds.

### Failure Mode 3: Celebrity Viral Post

A celebrity with 50 million followers posts a viral photo. The fan-out engine attempts to write the post to 50 million feed lists. The write amplification causes a spike in Redis write throughput. The fan-out engine's worker pool is overwhelmed. Other content events queue up behind the celebrity's post.

**Impact.** All content created during the celebrity's fan-out is delayed. The entire feed pipeline slows down. The fan-out lag spikes from 500 milliseconds to 5 minutes.

**Mitigation.** The celebrity post triggers the pull-based threshold. The fan-out engine detects that the creator has more than 10,000 followers and switches to pull-based mode. The worker writes a "new content available" flag instead of 50 million feed list entries. Followers see the celebrity's post when they next read their feed, with a 100 to 500 millisecond delay for the pull-based query.

### Failure Mode 4: Hydration Backend Slowdown

The content service backend experiences a slowdown due to a database issue. Hydration requests take 5 seconds instead of 50 milliseconds. The Feed Read Service threads are all blocked waiting for hydration responses. New requests queue up and eventually time out.

**Impact.** Feed reads fail with 503 errors. Users see error messages or empty feeds. The read path is completely unavailable.

**Mitigation.** The Feed Read Service implements a timeout on hydration requests. If the hydration backend does not respond within 200 milliseconds, the service returns a partial feed with placeholder content. The missing content items are loaded asynchronously and pushed to the client via a WebSocket stream.

### Failure Mode 5: Cache Stampede

A popular event triggers a wave of feed reads from millions of users who have been inactive. None of these users have valid timeline cache entries. Their feed lists are stale or empty. The feed service must assemble fresh feeds for all of them simultaneously.

**Impact.** The Feed Read Service is overwhelmed by the assembly workload. The content service is flooded with hydration requests. The response time degrades from 200 milliseconds to 10 seconds.

**Mitigation.** The timeline cache is pre-warmed for the top 10 percent of users by activity. When a cache miss occurs, the request is rate-limited. Only 1,000 concurrent cache-miss requests are allowed. Additional requests are queued or served a stale cache entry with a background refresh.

### Failure Mode 6: Follower Graph Read Failure

The Follower Graph Service is unavailable. When a user posts content, the fan-out engine cannot retrieve the follower list. The content event sits in the Kafka queue. The fan-out lag increases.

**Impact.** The new content does not appear in any follower's feed. The creator sees their own post, but no one else does. If the outage lasts more than 5 minutes, the fan-out lag becomes noticeable.

**Mitigation.** The fan-out engine caches follower lists locally. The cache stores the follower list for the most recently active 100,000 creators. If the Follower Graph Service is unavailable, the fan-out engine uses the cached follower list. The cache is stale by up to 5 minutes, but a slightly stale follower list is better than no fan-out at all.

### Failure Mode 7: Spam or Viral Manipulation

A coordinated attack generates thousands of fake posts in a short period. The posts contain spam links or manipulative content. The fan-out engine processes these posts like legitimate content, writing them to follower feed lists. The attack overwhelms the fan-out pipeline and pollutes user feeds.

**Impact.** 30 percent of feed items in the next hour are spam. Users see junk content. The engagement metrics show a spike in clicks but a drop in user satisfaction. The fan-out engine is overloaded, delaying legitimate content.

**Mitigation.** The ingestion layer applies spam detection before the content event reaches the fan-out engine. The spam detection model scores each post at upload time. Posts above the spam threshold are blocked and never enter the fan-out pipeline. Additionally, the fan-out engine has a per-creator rate limit: no more than 10 posts per hour from the same creator. If a creator exceeds the limit, their subsequent posts are queued for manual review.

### Failure Mode 8: Partial Shard Outage

A single Redis shard in the feed storage cluster becomes slow (high latency but not completely down). The shard is experiencing a hotspot: a highly active user's feed list is being read and written at an unusually high rate. The shard's CPU is at 100 percent. Responses take 500 milliseconds instead of 5 milliseconds.

**Impact.** Users whose feed lists are on the affected shard experience slow feed reads. The impact is limited to approximately 1 percent of users (1 shard out of 128). Other users are unaffected.

**Mitigation.** The Redis cluster detects the hotspot and migrates the hot key to a new shard. The migration is automated: the cluster monitors per-key access frequency and automatically moves hot keys to a shard with more capacity. Read requests for the hot key are served from the replica during the migration.

## Anti-Patterns

### Anti-Pattern 1: Fan-Out on Write for All Users

The naive approach is to apply fan-out on write to every user, including celebrities with millions of followers. This creates a write amplification disaster. A single post from a celebrity with 10 million followers generates 10 million feed list writes. If the celebrity posts 5 times per day, that is 50 million writes per day for one account.

The system becomes write-bound. The fan-out engine cannot keep up. The Redis cluster is overwhelmed. Other users' content is delayed because the fan-out engine is busy processing the celebrity's followers.

**Fix.** Implement a fan-out threshold. Users with more than 10,000 followers use pull-based feed delivery. The threshold is configurable and should be tuned based on the fan-out engine's capacity.

### Anti-Pattern 2: Storing Hydrated Content in Feed Lists

Some systems store the full content object in the feed list instead of a lightweight reference. This means the feed list contains the post text, media URLs, engagement counts, and author information directly in the sorted set.

This doubles or triples the feed list storage. A 40-byte reference becomes a 2 KB content object. For 100 million users with 800 feed items each, storage grows from 3.2 TB to 160 TB. The Redis cluster becomes prohibitively expensive.

**Fix.** Store only lightweight references in the feed list. Hydrate content on read. The hydration layer caches content objects separately, keeping the fast path efficient.

### Anti-Pattern 3: Synchronous Fan-Out

The fan-out engine blocks the content creation response until all feed list writes are complete. The user who posted the content waits for the fan-out to finish before receiving a response. For a user with 10,000 followers, this adds 500 milliseconds to 5 seconds of latency.

**Impact.** Content creation feels slow. The user sees a spinner after clicking "Post." The application's write API has high tail latency.

**Fix.** Fan-out is asynchronous. The content creation response is returned immediately after the content is stored. The fan-out engine runs in the background. The content creator's own feed is updated immediately, but followers see the content after a short delay.

### Anti-Pattern 4: No Pagination Safety

The system implements offset-based pagination on the feed. A user requests page 1 (items 0-19), then page 2 (items 20-39). Between the two requests, new items are inserted at the top of the feed. The user sees item 19 on page 1 and item 20 on page 2. But item 20 is now the 21st item because 2 new items were inserted. The user misses those 2 items, and item 19 appears on both pages.

**Impact.** Users see duplicate items and miss items. The feed experience is broken.

**Fix.** Use cursor-based pagination. The cursor is the content ID of the last item on the current page. The next page returns items created before that cursor. Cursor-based pagination is stable even when new items are inserted.

### Anti-Pattern 5: Overly Complex Ranking

The ranking model uses 50 signals, each with a learned weight from a deep neural network. The scoring function requires 500 milliseconds of computation per feed item. The ranking service falls behind. Users see a chronological feed with a 5-minute stale ranking.

**Impact.** The ranking is never fresh. The ranking service is always catching up. The chronological fallback is used most of the time, defeating the purpose of ranking.

**Fix.** Use a lightweight ranking model with 5 to 10 signals. The scoring function should complete in under 1 millisecond per item. Complex models can be used for offline ranking, but the online ranking must be fast enough to keep up with the feed volume.

### Anti-Pattern 6: Ignoring Inactive Users

The system fan-outs on write for all followers, including users who have not opened the app in 6 months. The feed lists for these inactive users are written to but never read. The storage is wasted.

**Impact.** 30 to 50 percent of feed list storage is consumed by inactive users. The write amplification is 30 to 50 percent higher than necessary.

**Fix.** Implement the hybrid strategy. Only fan-out to active users (opened the app in the last 7 days). Inactive users get their feed assembled on demand when they next open the app.

## When NOT to Use

The feed generation pattern is not appropriate for every system. Consider these scenarios:

**Real-time collaborative systems.** If users need to see each other's changes instantly with strong consistency, the feed pattern is not suitable. Google Docs, Figma, and collaborative whiteboards use operational transformation or CRDTs, not fan-out feeds.

**Transactional systems.** Systems that process orders, payments, or inventory updates require strong consistency and ACID transactions. The feed pattern's eventual consistency is not acceptable for financial transactions. Use a relational database with transactions instead.

**Search engines.** Search results are not a feed. Search relevance is determined by the query, not by a pre-computed ranked list. Use an inverted index (Elasticsearch, Solr) instead of a feed generation system.

**Notification systems.** Push notifications require low latency delivery to specific devices, not sorted feed lists. Use a notification service (Firebase Cloud Messaging, Apple Push Notification Service) with dedicated infrastructure.

**Small-scale applications.** If your application has fewer than 10,000 users, the feed generation pattern is overengineering. A simple SQL query with a JOIN and an index on `created_at` is sufficient. The overhead of maintaining a Redis cluster and a fan-out engine is not justified.

**Identity and access management systems.** Systems that manage user permissions, roles, and access control lists require strict consistency and audit trails. The feed pattern's eventual consistency cannot guarantee that a permission change is immediately visible. Use a dedicated IAM system with strong consistency (ZooKeeper, etcd) instead.

**Chat and messaging systems.** Chat feeds (a user's list of conversations) are different from content feeds. Chat conversations are ordered by the most recent message, not by relevance. The fan-out pattern is overkill because conversations are point-to-point, not broadcast. Use a dedicated chat system (matrix protocol, IRC, or a custom in-memory store) with a simpler data model.

**Time-series data.** Systems that display metrics or events over time (monitoring dashboards, analytics charts) should not use the feed generation pattern. Time-series data is aggregated and queried by time range, not by personalized relevance. Use a time-series database (InfluxDB, TimescaleDB, Prometheus) with aggregation queries, not a fan-out engine.

**Content moderation queues.** Systems that display a queue of pending content for moderator review are not feeds. The queue is ordered by a business priority (oldest items first, highest-risk items first), not by user preference. The fan-out pattern introduces unnecessary complexity. A simple database query with a priority-ordered index is sufficient.

**Email digests.** Email digests (daily summary emails) are batch feeds, not real-time feeds. The digest is assembled once per day and sent to all users. The fan-out pattern is not needed because the digest is assembled in a batch job, not in real time. Use a batch processing system (MapReduce, Spark, scheduled cron job) to assemble digests.
