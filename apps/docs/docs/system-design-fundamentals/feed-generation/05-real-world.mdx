## Company Case Studies

### Case Study 1: Twitter — Fan-Out Service (Early 2010s)

Twitter's early feed system used fan-out on write for all users. When a user posted a tweet, the system wrote the tweet ID to the timeline of every follower. This worked well until Twitter hit 100 million active users and celebrity accounts started dominating the platform.

The problem manifested in 2012 when Justin Bieber joined Twitter. His first tweet triggered a fan-out to 12 million followers. The fan-out engine fell behind by 30 minutes. The entire timeline pipeline was blocked. Other users' tweets were delayed because the fan-out engine was busy processing Bieber's followers.

Twitter's solution was the Fan-Out Service (FOS), a hybrid approach. For users with fewer than 10,000 followers, the system uses fan-out on write. The tweet is written to each follower's timeline in Redis. For users with more than 10,000 followers, the system uses fan-out on read. The tweet is not written to individual timelines. Instead, the timeline read path checks a separate list of celebrity tweets and merges them into the user's timeline.

This hybrid approach reduced the write amplification by 95 percent. The 99th percentile timeline read latency stayed under 200 milliseconds. The system handled 500 million tweets per day without the fan-out engine falling behind.

**Key lessons.** The fan-out threshold is critical. A single celebrity account can overwhelm the system. The hybrid approach is a proven solution. The threshold must be tuned based on the fan-out engine's capacity.

### Case Study 2: Instagram — Redis Feed Storage (Mid 2010s)

Instagram's feed system processes 95 million photos and videos per day. The feed is the primary consumption surface. Instagram uses fan-out on write with Redis feed storage.

Instagram stores feed lists for each user in Redis. Each feed list contains up to 800 post IDs. The feed list is a sorted set with the post creation timestamp as the score. When a user opens the app, Instagram reads the top 20 post IDs from their feed list, then hydrates them with full metadata.

Instagram's approach has several optimizations. The fan-out engine batches writes in groups of 100. The feed list size is limited to 800 items per user. Old items are evicted from the bottom of the feed list when new items are pushed to the top. The feed list is read using a single `ZREVRANGE` command, which takes 1 to 5 milliseconds.

Instagram's Redis cluster has hundreds of nodes. Each node handles approximately 10,000 feed lists. The cluster is sharded by user ID using consistent hashing. Writes are distributed across all nodes, and reads are served from the closest node.

**Key lessons.** Redis is a battle-tested feed storage backend. The 800-item feed list limit is a good default. Batch writes reduce fan-out latency. Consistent hashing enables linear scaling.

### Case Study 3: Pinterest — Pin-Based Feed with HBase (Late 2010s)

Pinterest's feed system is built on HBase, not Redis. Each user's home feed is a row in HBase with columns for each pin. The feed is assembled using a time-ordered scan of the user's feed row.

Pinterest uses a hybrid fan-out strategy with a twist: the feed is not pre-computed for all users. Instead, Pinterest pre-computes the feed for the top 5 percent of active users (by engagement). The remaining 95 percent of users get their feed assembled on demand.

The pre-computed feed is stored in a Memcached tier. The cache stores the first 50 feed items for each cached user. The cache hit rate is 85 percent, which means the pre-computation effort is well-utilized.

Pinterest's approach is notable because it chose HBase over Redis. HBase provides better durability and linear scalability. The trade-off is higher read latency (10 to 20 milliseconds vs 1 to 5 milliseconds for Redis). Pinterest accepted this trade-off because the feed assembly cost is dominated by the ranking model, not the storage read.

**Key lessons.** HBase is a viable alternative to Redis for feed storage. Pre-computing feeds for only the top 5 percent of users saves 95 percent of the storage cost. The cache hit rate validates the pre-computation strategy.

### Case Study 4: LinkedIn — Feed Ranking with Machine Learning (2020s)

LinkedIn's feed system uses a machine learning model to rank feed items. The ranking model is a deep neural network with 50 features. The features include: creator-viewer connection strength, content type, engagement history, time of day, device type, and topic affinity.

The ranking model runs on a GPU cluster. Each feed request triggers a model inference. The inference takes 5 to 10 milliseconds, which is fast enough for real-time feed delivery.

LinkedIn uses a multi-stage feed assembly pipeline. Stage one: retrieve candidate items from the feed storage (10,000 items). Stage two: apply lightweight filters (blocked users, muted topics, already-seen items) to reduce the candidate set to 500 items. Stage three: run the ML model on the 500 candidates to compute a relevance score. Stage four: sort by score and return the top 50 items.

This multi-stage approach is critical for performance. Running the ML model on 10,000 items would take 50 to 100 milliseconds. Running it on 500 items takes 5 to 10 milliseconds. The lightweight filters eliminate 95 percent of the candidates before the expensive model inference.

**Key lessons.** Multi-stage filtering reduces the ML model workload by 95 percent. The ranking model must be fast enough for real-time inference. GPU inference is viable for production feed systems.

### Case Study 5: TikTok — Single-Column Algorithmic Feed (2020s)

TikTok's feed system is the most algorithmically sophisticated among major platforms. The For You feed uses a deep neural network that predicts the likelihood of a user watching a video to completion. The model is trained on billions of user interactions.

TikTok's feed architecture differs from traditional social feeds in three key ways. First, the feed is not based on a follow graph. TikTok shows content from creators the user does not follow. The follow graph is a signal, not a filter. Second, the feed is a single column. Each item takes the full screen. The user swipes vertically to advance. This means the feed has only one item visible at a time, and the ranking model must predict engagement for each item independently. Third, the feed is generated on the client side, not the server side. The server sends a batch of 30 to 50 candidate videos. The client-side ranking model re-orders them based on real-time user behavior (did the user pause? did they skip?). The client-side re-ranking happens in under 10 milliseconds.

TikTok's feed generation does not use fan-out on write. There is no feed list per user. Instead, the server maintains a large pool of candidate videos (the entire video catalog filtered by basic quality and safety criteria). When a user opens the app, the server runs a retrieval model that selects 500 candidates from the pool, then a ranking model that scores and orders the candidates. The top 30 to 50 are sent to the client.

The retrieval and ranking models run in real time for every feed request. The retrieval model uses collaborative filtering (users similar to the current user watched these videos), content-based filtering (videos similar to videos the user watched), and trending signals (videos that are currently popular in the user's region). The ranking model uses a 200-feature neural network that runs in 20 milliseconds per video on GPU.

**Key lessons.** The follow graph is optional for feed generation. Client-side ranking enables real-time personalization. Real-time retrieval and ranking are viable when the model is fast enough and the candidate pool is efficiently indexed.

## Open Source Solutions

### Apache Kafka

Kafka is the event backbone for most feed generation systems. It provides the durable, ordered event stream between content creation and fan-out. Kafka's partitioning model maps naturally to feed generation: each content creation event is published to a partition, and the fan-out engine consumes from all partitions.

Key Kafka configuration for feed generation: `log.retention.hours=72` allows replay of events for up to 3 days. `num.partitions=64` provides enough parallelism for the fan-out engine. `replication.factor=3` ensures durability.

### Apache Pulsar

Pulsar is an alternative to Kafka that provides native support for topic hierarchies and geo-replication. Pulsar's topic structure maps feed generation concepts: a namespace per content type, a topic per creator, and subscriptions per fan-out worker.

Pulsar's geo-replication feature is particularly useful for multi-region feed systems. Content events are automatically replicated to all regions. Each region's fan-out engine consumes from the local replica, avoiding cross-region read latency.

### Redis

Redis is the dominant feed storage backend. The sorted set data structure is a perfect fit for feed lists. The `ZADD`, `ZREVRANGE`, and `ZREMRANGEBYSCORE` commands cover the entire feed list API.

Redis Cluster provides automatic sharding. Each shard is a Redis primary with one or more replicas. The cluster supports adding and removing shards without downtime. The `redis-cli --cluster` tool handles resharding.

### Apache Cassandra

Cassandra is an alternative to Redis for feed storage. It provides linear scalability, native durability, and multi-region replication. The trade-off is that Cassandra does not have a sorted set data structure. Feed lists must be implemented as a custom data model using a composite partition key (user ID, bucket ID) and a clustering key (timestamp).

The Cassandra data model for feed lists: `CREATE TABLE feed_lists (user_id text, bucket int, post_id text, created_at timestamp, PRIMARY KEY ((user_id, bucket), post_id))`. The bucket is a time-based partition key that limits the partition size. Each bucket covers 1 hour of feed items.

## Cloud Providers

### AWS Feed Generation Reference Architecture

AWS provides a reference architecture for feed generation using managed services. The architecture uses:

- **API Gateway + Lambda** for content ingestion.
- **Kinesis Data Streams** as the event backbone (Kafka alternative).
- **DynamoDB** for the follower graph (follower set per user).
- **ElastiCache for Redis** as the feed storage cluster.
- **Lambda fan-out workers** (serverless fan-out engine).
- **CloudFront** for CDN delivery of media content.
- **DAX** (DynamoDB Accelerator) for the timeline cache.

The reference architecture scales to 10 million active users without custom infrastructure. For larger deployments, Kinesis is replaced with MSK (Managed Kafka), and ElastiCache is replaced with a self-managed Redis cluster.

### GCP Feed Generation Reference Architecture

GCP's reference architecture uses:

- **Cloud Run** for content ingestion.
- **Pub/Sub** as the event backbone.
- **Firestore** for the follower graph.
- **Memorystore for Redis** as the feed storage cluster.
- **Cloud Functions** for fan-out workers.
- **Cloud CDN** for media delivery.
- **Cloud Memorystore** for the timeline cache.

GCP's Pub/Sub has a unique advantage: it supports exactly-once delivery, which simplifies the fan-out engine's error handling. Each event is delivered exactly once, eliminating the need for deduplication in the fan-out pipeline.

### Azure Feed Generation Reference Architecture

Azure's reference architecture uses:

- **Azure Functions** for content ingestion.
- **Event Hubs** as the event backbone.
- **Cosmos DB** for the follower graph.
- **Azure Cache for Redis** as the feed storage cluster.
- **Azure Functions** for fan-out workers.
- **Azure CDN** for media delivery.
- **Azure Cache for Redis** for the timeline cache.

Azure's Cosmos DB is a strong choice for the follower graph because it provides multi-region writes and 99.999 percent availability. The follower graph is active-active across regions, allowing follow/unfollow operations to be processed in any region.

## Lessons Learned

### Lesson 1: Start Simple, Add Complexity Later

The most common mistake is overengineering the feed system. Teams start with a hybrid fan-out strategy, a complex ranking model, and a multi-region deployment before they have 10,000 users. The system is complex, expensive, and harder to debug.

The right approach: start with fan-out on read. A simple SQL query with a JOIN and an index on `created_at` handles 10,000 users. At 100,000 users, add fan-out on write with Redis. At 1 million users, add the hybrid strategy. At 10 million users, add the ranking engine. At 100 million users, add multi-region replication.

### Lesson 2: Monitor Fan-Out Lag Religiously

Fan-out lag is the single most important metric in the feed system. It measures the time between content creation and the content appearing in follower feeds. If the lag exceeds 10 seconds, the feed is stale. If the lag exceeds 60 seconds, users notice. If the lag exceeds 5 minutes, users leave.

Every team that has run a production feed system has a story about fan-out lag. The lag spiked because of a celebrity post, a Redis node failure, or a Kafka consumer crash. The team that monitors lag aggressively catches the problem before users notice.

### Lesson 3: The Fan-Out Threshold Must Be Dynamic

The fan-out threshold (number of followers that triggers push-based fan-out) should not be a static configuration. It should be dynamic based on the current system load. During peak hours, the threshold decreases. During off-peak hours, the threshold increases.

A dynamic threshold requires a control loop that monitors the fan-out engine's capacity and adjusts the threshold accordingly. The control loop checks the current write throughput, the queue depth, and the consumer lag. If any metric exceeds a threshold, the fan-out threshold is reduced by 10 percent.

### Lesson 4: Hydration Is the Hidden Bottleneck

Teams often focus on the feed list storage and the fan-out engine, but the hydration layer is the most common bottleneck. A feed list read takes 5 milliseconds. The hydration of 20 items takes 50 to 120 milliseconds. The hydration layer is the dominant contributor to the read path latency.

The hydration layer must be optimized with batch fetching, aggressive caching, and connection pooling. The hydration cache should be separate from the feed list cache. The hydration cache stores content objects keyed by content ID. The cache hit rate should be above 90 percent for the top 10 percent of content by views.

### Lesson 5: Plan for Celebrity Events

Every platform eventually gets a celebrity user. The celebrity posts at 3:00 PM on a Tuesday, and the system falls over. The fan-out engine crashes. The Redis cluster is overloaded. The feed is down for 5 minutes.

The solution is to plan for celebrity events before they happen. Identify users with rapid follower growth. Automatically switch them to pull-based fan-out when their follower count exceeds the threshold. Monitor the fan-out engine's capacity and scale it proactively before a celebrity event.

### Lesson 6: Cache Invalidation Is Harder Than Caching

The timeline cache reduces read latency, but cache invalidation is the hard part. When a user's feed list changes (new content arrives, old content is removed), the timeline cache must be invalidated. If the cache is not invalidated, the user sees stale content.

The standard approach is to invalidate the timeline cache entry when a new feed item is written. However, this means that every fan-out write triggers a cache invalidation. For a user with 10,000 followers, the cache invalidation is 10,000 operations. This adds to the write amplification.

The solution is to use a "new content available" flag instead of full cache invalidation. The flag is a lightweight Redis key. When the user reads their feed, the read path checks the flag. If the flag is set, the read path merges the new content with the cached content.

### Lesson 7: A/B Test Every Ranking Change

Feed ranking changes have unpredictable effects on user behavior. A change that increases likes may decrease time spent. A change that increases time spent may decrease content diversity. Every ranking change must be A/B tested with at least 100,000 users per variant and a 14-day minimum test duration.

The A/B test infrastructure must be built into the feed system from day one. The test framework assigns users to variants using a consistent hash of the user ID. The variant determines which ranking model, scoring weights, and feed configuration the user receives. Metrics are collected at the user level and aggregated by variant.

### Lesson 8: The Client Side Matters as Much as the Server Side

Many feed systems focus exclusively on the server side and neglect the client. The client-side feed logic is equally important. The client cache, the feed prefetch behavior, the refresh interval, and the scroll position restoration all affect the user experience.

A common mistake is delivering more items than the client can handle. The server sends 20 items per page, but the client requests 50 to reduce network round trips. The server should honor the client's requested page size, up to a maximum of 100 items. Larger page sizes reduce network latency but increase rendering time on low-end devices.
