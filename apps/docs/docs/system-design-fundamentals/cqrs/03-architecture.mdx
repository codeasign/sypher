## Architecture Tiers

CQRS architectures fall into tiers based on their maturity and the demands they handle. Start with the simplest tier that meets your requirements. Move to a higher tier only when you have evidence that the current tier is failing.

### Tier 1: Naive CQRS — Same Database, Separate Models

This is the entry level. You use a single database but separate the read and write models in your application code. The write model uses a normalized schema. The read model uses database views, materialized views, or denormalized tables in the same database.

**Characteristics.** One database, one connection pool, one deployment unit. The read models are implemented as SQL views or materialized views. The command and query are in the same application process. No synchronization mechanism needed.

**When to use this.** You are introducing CQRS to an existing codebase incrementally. Your query volume is moderate (under 10,000 queries per second). Your write volume is also moderate. You want the code organization benefits of CQRS without the operational complexity of separate databases.

**Limitations.** The single database remains a contention point. Under heavy read load, the read views still compete with write operations for database connections, memory, and I/O. Scaling reads requires scaling the entire database, which is expensive. The read views are constrained by the database engine — you cannot use a different storage technology for reads.

### Tier 2: Production CQRS — Separate Databases, Event Bus Sync

**Characteristics.** Two independent databases. The command service and query service are separate deployment units. The event bus provides durability. Each database can be independently scaled, backed up, and maintained.

**Database technology choice.** The write database is optimized for ACID transactions. PostgreSQL, MySQL, or SQL Server are typical choices. The read database is optimized for query speed. Options include: MongoDB for document-shaped views, Elasticsearch for full-text search, Redis for in-memory speed, or a dedicated PostgreSQL read replica with different indexing.

**Event bus choice.** RabbitMQ for reliable delivery with consumer acknowledgments. Apache Kafka for high-throughput event streaming with replay capability. Amazon SQS for fully managed serverless queuing. The choice depends on throughput requirements, durability needs, and operational capabilities.

**When to use this.** Your query volume exceeds 10,000 queries per second. Your read model is significantly different from your write model. You need different storage technologies for reads and writes. You have the operational maturity to manage two databases and an event bus.

Event sourcing stores state changes as an append-only event log. The read models are projections built from the event stream. This combination removes the write database entirely — the event store is the write side.

**Characteristics.** No write database — the event store is the write side. Current state is derived by replaying events or using snapshots. Each read model is a projection built from the event stream. Projections can be rebuilt from scratch at any time.

**When to use this.** You need a complete audit trail of every state change. You need to support temporal queries. You need multiple read models from the same data. You are willing to invest in the higher complexity of event sourcing.

**Regional command routing.** Commands are routed to the nearest region for low-latency writes. Each region has its own command service and writes to its own database. The global event bus replicates events to all other regions. A command submitted in Asia is written to the Asia database and replicated to US and EU within seconds.

**Regional query serving.** Queries are served from the local read database in each region. This gives sub-10ms read latency regardless of where the user is located. The staleness window across regions is higher — a command written in Asia may take seconds to reach the US read database.

**Conflict resolution.** When commands in different regions modify the same aggregate concurrently, conflicts must be resolved. Strategies include: last-writer-wins (simplest, risk of data loss), conflict-free replicated data types (CRDTs, automatic merging), or a single write leader per aggregate (avoids conflicts but requires cross-region coordination).

## Capacity Planning

Capacity planning for a CQRS system must account for three independent paths: command throughput, query throughput, and sync throughput.

### Command Capacity

The write side is limited by your database write throughput. Factors include transaction rate, lock contention, and indexing overhead. Estimate your peak write volume and provision for 2x headroom.

Formula: Required command throughput = (peak concurrent users x commands per user per hour) / 3600. Add 30% for retries and system commands.

### Query Capacity

The read side is limited by your read database query throughput and your cache hit ratio. With a good cache, you can serve 10 to 50 times more reads than your raw database capacity would suggest.

Formula: Required query throughput = (peak concurrent users x queries per user session) / (session duration in seconds). For cache planning: a 90% cache hit ratio reduces database load by 10 times.

### Sync Capacity

The synchronization mechanism must handle peak event throughput, which equals peak command throughput plus any background events. The sync must sustain this rate while maintaining acceptable staleness.

Formula: Required sync throughput = peak command throughput x average events per command. The sync latency target determines batch size and processing interval.

**Command service scaling.** The command service scales based on command queue depth and CPU utilization. Each node runs the same handlers. The event bus partitions ensure ordered delivery per aggregate.

**Query service scaling.** The query service scales based on request rate and latency. Each node is stateless — any node can handle any query. A read-through cache (Redis, Memcached) sits in front of the read database. Cache warming strategies pre-load popular queries.

**Database scaling.** The write database uses a primary-replica setup for high availability. The read database uses a cluster with read replicas. The read model can be sharded by customer ID or region for horizontal scaling.

## Monitoring and Observability

A CQRS system has more moving parts than a monolith, so monitoring is essential.

**Sync lag.** The most important CQRS-specific metric. Measure the time between event publication and read model update. Alert if it exceeds a threshold, for example 10 seconds for near-real-time systems.

**Command success rate.** Percentage of commands that complete successfully. A sudden drop indicates a business rule, infrastructure, or data integrity issue.

**Query cache hit ratio.** The percentage of queries served from cache. A low hit ratio suggests the cache TTL is too short or the read models do not match query patterns.

**Database connection pool utilization.** Both databases should have adequate connection pools. Monitor for connection exhaustion, which indicates insufficient capacity.

**Event bus lag.** The number of unconsumed events in the queue or topic. Persistent backlog indicates the sync consumers cannot keep up with event production.

## Resilience Patterns

**Outbox pattern.** Events are written to a database table within the command transaction. A background process reads from the outbox table and publishes to the event bus. This guarantees at-least-once delivery even if the event bus is unavailable at the time of the command.

**Poison message handling.** If a read model updater repeatedly fails on a specific event, that event is moved to a dead letter queue after N retries. The sync continues processing other events. The dead letter is inspected manually.

**Read model rebuild.** If the read model is corrupted, it is rebuilt from the event store or write database. In event-sourced CQRS, this is a replay. In standard CQRS, this is a full refresh from the write database.

**Circuit breaker.** If the read database is unresponsive, the query service returns cached data if available, or returns an error. The circuit breaker prevents cascading failures from repeated timeouts.
