## The Problem: One Model for Two Different Jobs

Imagine building an e-commerce platform. Your application serves two very different kinds of traffic.

Shoppers browse product listings, filter by category, search by name, sort by price, paginate through thousands of results. Each page load fires multiple read queries against your database. A single product listing page might join across inventory, pricing, reviews, and ratings tables to build the view.

Meanwhile, orders are being placed. Each order triggers writes: decrementing inventory, creating a payment record, updating order status, logging the transaction. These writes lock rows, acquire table-level locks in some engines, and compete for the same connection pool and I/O bandwidth.

When you use one model and one database for both jobs, a tension emerges. The read queries get slower as the write volume grows. The write operations stall when heavy read queries hold locks. The ORM mappings try to be everything at once — eager-loading some relations, lazy-loading others — and neither path performs well.

A product listing page that takes 2 seconds to load costs you customers. An order transaction that takes 500ms costs you revenue. You need each path optimized independently.

This is the problem CQRS solves.

## The Insight: Separate Command from Query

CQRS stands for Command Query Responsibility Segregation. The core idea is that a method should either perform an action (a command) or return data (a query), but not both.

This principle was articulated by Bertrand Meyer in his design of the Eiffel programming language, where he defined the Command-Query Separation (CQS) principle at the method level. CQRS extends this to the architectural level: separate the entire write stack from the entire read stack.

Think of a library. One desk handles checkouts and returns — these are commands that change the state of the system. A different desk handles catalog searches — these are queries that return information without changing anything. Each desk is staffed, laid out, and optimized for its specific job. The checkout desk has a terminal, a scanner, and space for stacking books. The catalog desk has terminals, reference materials, and a quiet atmosphere for browsing.

CQRS applies the same logic to software architecture. The command side processes writes with strong consistency guarantees. The query side serves reads optimized for speed, using denormalized views that mirror exactly what the UI needs.

## Core Vocabulary

These are the terms you need to understand before going deeper.

**Command.** A request to change the state of the system. A command is imperative — `PlaceOrder`, `UpdateInventory`, `CancelReservation`. It expresses intent. By convention, commands are named in the imperative mood. A command either succeeds or fails; it does not return data.

**Query.** A request to read data without changing state. A query is declarative — `GetProductById`, `SearchOrders`, `ListAvailableInventory`. It returns a result set. Queries never mutate the system.

**Command Handler.** A component that receives a command, validates it, performs business logic, and writes to the write data store. Each command type has its own handler. The handler is where business rules live: "Is the item in stock?", "Has this user exceeded their credit limit?"

**Query Handler.** A component that receives a query, retrieves data from the read data store, and returns a result. Query handlers are typically stateless and fast. They may apply filters, projections, and pagination.

**Write Model.** The code and data structures used on the command side. The write model is optimized for consistency — it enforces invariants, manages transactions, and ensures the database stays in a valid state. Usually normalized, often using domain-driven design aggregates.

**Read Model.** The code and data structures used on the query side. The read model is optimized for retrieval speed — it may be denormalized, pre-joined, aggregated, or projected into shapes that match the UI exactly.

**Write Database.** The data store used by the command side. This database enforces constraints, maintains referential integrity, and is the source of truth. It is optimized for write throughput and transactional consistency.

**Read Database.** The data store used by the query side. This database may be a copy, a projection, or a materialized view of the write data. It is optimized for read queries and may use different storage engines or indexing strategies.

**Command Bus.** A messaging infrastructure that routes commands to their handlers. The command bus can provide middleware for logging, validation, authorization, and retry logic. It decouples the command issuer from the command executor.

**Query Bus.** A messaging infrastructure that routes queries to their handlers. Similar to the command bus but optimized for read operations. The query bus may support caching, query optimization, and result pagination.

**Synchronization Mechanism.** The process that keeps the read database in sync with the write database. This could be an event bus, a change data capture pipeline, a message queue, or a batch job. The synchronization may be immediate or eventual.

**Materialized View.** A pre-computed data set stored in the read database that reflects a specific query shape. Materialized views are rebuilt or refreshed when the source data changes. They eliminate expensive joins at query time.

**Eventual Consistency.** The guarantee that, given enough time without new updates, all read replicas will converge to the same state as the write source. In a CQRS system, the read side is eventually consistent with the write side — there is a window where stale data may be served.

## Step by Step: Placing an Order with CQRS

Walk through a concrete scenario. A customer places an order on the e-commerce platform. The system uses CQRS with separate databases.

### Step 1: Client Issues a Command

The browser sends a POST request to the API. The request body contains the `PlaceOrder` command: customer ID, line items, shipping address, payment method. The API endpoint deserializes the request into a command object and sends it to the command bus.

### Step 3: Handler Writes to Write Database

The handler executes business logic. It validates payment, checks inventory for each line item, calculates totals including tax and shipping, applies any promotional discounts, creates the order record, and decrements inventory quantities. All of this happens inside a transaction on the write database.

The write database uses a normalized schema optimized for write integrity. Order headers, line items, payments, and shipments are in separate tables with foreign key constraints.

### Step 4: Handler Emits an Event

After the transaction commits, the handler emits an `OrderPlaced` event. This event contains all the data needed to build the read model: order ID, customer ID, item list, total amount, timestamp.

### Step 8: Query Handler Returns Result

The query handler executes a simple SELECT against the read database. The read database returns the pre-joined order summary row. No joins, no complex ORM loading — just a single index lookup returning one row.

### Step 9: Client Sees the Result

The API serializes the query result to JSON and returns it in the HTTP response. The client renders the order confirmation page. The entire query completes in under 10 milliseconds because the read model is purpose-built for this exact shape.

## CQRS Variants

CQRS can be implemented at several levels of sophistication. Choose the variant that matches your system's requirements.

### Single-Database CQRS

The simplest form. You use one database but separate the read and write models in code. The read operations use different tables or views than the write operations. This variant introduces no synchronization complexity.

<AsciiDiagram
  id="cqrs/concepts/variant-single-db"

│  │ orders        │          │ order_summary      │       │

  }
  alt="One database with separate read tables and write tables, same connection pool"
  caption="Simplest CQRS — separate models, same database."
/>

**Pros:** No eventual consistency issues, no extra infrastructure, easy to implement.

**Cons:** Single database is still a contention point. Read-heavy workloads still compete with write traffic for I/O and connections. Scaling reads requires scaling the entire database.

## When CQRS Helps Most

CQRS is not for every system. It provides the most value when these conditions are true.

Your read workload differs significantly from your write workload. If reads and writes access the same data in the same shape, CQRS adds complexity without benefit.

Your read volume is much higher than write volume. A reporting dashboard that serves thousands of users but receives hourly updates is a strong candidate.

Your write operations require complex business logic and validation. Domain rules, invariants, and multi-step workflows benefit from a dedicated command stack.

Your queries need different data shapes than your commands store. If the UI needs denormalized views that join across multiple aggregates, a dedicated read model simplifies queries dramatically.

Your team has the discipline to maintain two models. CQRS adds surface area. A small team maintaining a simple CRUD app should not use CQRS.

## Mental Model Check

Answer these five questions to confirm you understand the core concept.

**Question 1.** You are designing a system where customers book appointments. The read side shows available slots. The write side creates bookings. Should you use CQRS?

**Answer.** Yes. The read side needs a fast query that shows pre-computed availability. The write side needs transactional integrity to prevent double-booking. Separate models allow you to optimize each — the read side can use a denormalized availability view, the write side enforces constraints.

**Question 2.** Your application has a simple CRUD interface for managing a contact list. Reads and writes operate on the same data shape (name, phone, email). Should you use CQRS?

**Answer.** No. There is no benefit because the read model and write model are identical. CQRS would add infrastructure and code duplication with zero query performance gain. A standard REST API with a single database is the correct choice.

**Question 3.** In a CQRS system, the client sends a command to update user profile. The command is validated and written to the write database. The client then immediately queries the profile. What might happen?

**Answer.** The query reads from the read database, which may not yet reflect the update. The client could see stale data. This is the cost of eventual consistency. Solutions include waiting for synchronization acknowledgement or routing the immediate read to the write database.

**Question 4.** A command handler throws an exception after writing to the database but before emitting the event. What is the consequence?

**Answer.** The write database has been modified but the event was never published. The read database does not reflect the change. The system now has a consistency gap. This is why command handlers should emit events within the same transaction as the write, or use an outbox pattern to guarantee at-least-once event delivery.

**Question 5.** Your system uses CQRS with separate databases. The synchronization pipeline is down for 5 minutes. What happens to new commands and queries?

**Answer.** Commands continue to work — writes succeed on the write database. Queries continue to work — reads return data from the read database. But the read data becomes increasingly stale. When synchronization resumes, a backlog of events must be processed to bring the read database up to date. This is acceptable if your application tolerates minutes of staleness.
