## Question Types by Level

CQRS questions appear at every seniority level. The depth and breadth of the discussion determine your score.

### Entry-Level (0-2 years)

At this level, interviewers want to confirm you understand the basic concept and can explain it clearly.

**Typical questions.**

"What is CQRS?"

"Why would you use CQRS?"

"What is the difference between CQS and CQRS?"

"Can you draw a simple CQRS architecture?"

**What they evaluate.** Can you define commands and queries? Do you understand the separation principle? Can you explain the difference between CQS (method level) and CQRS (architectural level)? Can you sketch a diagram with a command side and a query side connected by a sync mechanism?

**Sample strong answer.**

"CQRS stands for Command Query Responsibility Segregation. It is an architectural pattern that separates read operations from write operations. Commands change state and do not return data. Queries return data and do not change state. The separation applies at the database level — the command side writes to a write-optimized database, and the query side reads from a read-optimized database. A synchronization mechanism, typically an event bus or change data capture pipeline, keeps the two in sync."

### Mid-Level (3-5 years)

At this level, interviewers want to see practical implementation knowledge and awareness of tradeoffs.

**Typical questions.**

"Walk me through a CQRS implementation for an e-commerce order system."

"How do you handle eventual consistency in CQRS?"

"What happens if the sync mechanism fails?"

"How do you version events in a CQRS system?"

"Compare CQRS with event sourcing. When would you combine them?"

**What they evaluate.** Can you design a concrete implementation with specific technologies? Do you understand and can you handle eventual consistency? Do you know failure modes and recovery strategies? Can you make technology choices and justify them?

**Sample strong answer.**

"For an e-commerce order system, I would use CQRS with separate databases. The command side uses PostgreSQL for transactional integrity. The write model is normalized — orders, line items, payments in separate tables with foreign keys and constraints. The command bus uses RabbitMQ for reliable delivery. The query side uses MongoDB with denormalized order summaries, one document per order containing all the data the UI needs. Synchronization is via an event bus: the command handler emits an OrderPlaced event after committing the write transaction. A subscriber picks up the event and upserts the read document. For eventual consistency, I would add a version token to the write response so the client can poll the read side until the version matches."

### Senior Level (6-9 years)

At this level, interviewers want to see architectural judgment, experience with failure modes, and the ability to evaluate CQRS against alternatives.

**Typical questions.**

"Our system has 1 million writes and 50 million reads per day. Should we use CQRS?"

"Design a CQRS system that spans three AWS regions."

"How do you handle schema evolution when you have hundreds of event types?"

"Compare CQRS with a read replica architecture. When would you choose each?"

"Our query latency increased by 10x after adopting CQRS. What could be wrong and how would you fix it?"

**What they evaluate.** Can you evaluate CQRS in the context of concrete scale and business requirements? Can you design a distributed multi-region CQRS system? Do you have operational experience with CQRS in production? Can you debug CQRS performance problems?

**Sample strong answer.**

"One million writes and 50 million reads per day is a 50:1 read-to-write ratio. CQRS would likely help, but the decision depends on more than the ratio. If the read model is the same shape as the write model, CQRS adds complexity without benefit. If the reads require denormalized views, aggregations, or different storage technology, CQRS becomes valuable. I would start by profiling the existing system to identify the actual bottleneck. If the bottleneck is query performance and queries need different data shapes, I would incrementally introduce CQRS, starting with the most expensive query and adding a dedicated read model for it."

### Staff+ Level (10+ years)

At this level, interviewers want to see organizational impact, ability to influence architecture across teams, and deep understanding of CQRS as an organizational pattern.

**Typical questions.**

"Your organization has 15 microservices. How would you introduce CQRS across the platform?"

"Design a CQRS governance model that allows teams to evolve their schemas independently."

"How do you decide when a new team should use CQRS vs. a simpler approach?"

"Your system has been using CQRS for 5 years. The event store has 2 billion events. How do you manage this?"

**What they evaluate.** Can you design organization-wide standards? Do you understand how CQRS affects team boundaries, ownership, and velocity? Can you balance architectural purity with pragmatic delivery? Do you have experience with CQRS at extreme scale?

## Key Phrases to Use

These phrases demonstrate depth and experience.

"Commands are imperative, queries are declarative."

"Each side can be independently scaled, deployed, and optimized."

"Eventual consistency is a feature, not a bug — it is the tradeoff for independent scalability."

"The outbox pattern guarantees at-least-once delivery without distributed transactions."

"Read models are projections, not copies. They transform the data into the shape queries need."

"Sync lag is the most important operational metric for a CQRS system."

"CQRS does not require event sourcing, but they complement each other."

"Start with CQRS where it provides the most value — the query that requires the most complex join."

"Schema evolution in CQRS requires coordination between command producers and read model consumers."

"CQRS is an organizational pattern as much as a technical pattern — it enables different teams to own the read and write sides independently."

## Red Flags to Avoid

These mistakes signal inexperience and lower your score.

**Claiming CQRS solves all problems.** No architecture pattern is a silver bullet. Acknowledge that CQRS adds complexity and is not the right choice for every system.

**Ignoring eventual consistency.** Pretending that the read side is always immediately consistent shows you have not run CQRS in production. Acknowledge the staleness window and discuss strategies to handle it.

**Proposing CQRS for a simple CRUD app.** Suggesting CQRS for a contact list application shows poor architectural judgment. Recognize when CQRS is overkill.

**Using CQRS and event sourcing interchangeably.** CQRS and event sourcing are independent patterns that can be used together or separately. Do not confuse them.

**Forgetting the sync mechanism in your architecture diagram.** Many candidates draw the command side and query side but omit how data flows between them. Always include the sync mechanism.

**Saying queries never cause side effects.** In practice, some queries may trigger logging, caching behavior, or analytics tracking. It is more accurate to say queries should not cause semantically meaningful side effects.

## System Design Drills

### Drill 1: Design a CQRS-Based Analytics Dashboard

**Scenario.** Your startup needs a real-time analytics dashboard showing 50 metrics: active users, revenue, conversion rate, page views, and more. The data is generated by user interactions across web and mobile. Write volume is 5,000 events per second. Read volume is 200 dashboard views per second, each returning 50 metrics.

**What to cover.**

- Write side: ingest events via a message queue (Kafka). Process events through a command handler that updates aggregate state.
- Event store: append-only log of events. Each event is a user interaction.
- Read side: pre-computed materialized views for each metric. Use a columnar store (ClickHouse, Druid) for fast aggregations.
- Sync: real-time event processing with sub-second latency. Projection handlers update metric aggregates on each event.
- Cache: Redis cache for dashboard queries. Invalidate on each metric update.

### Drill 2: Migrate a Monolith to CQRS

**Scenario.** A legacy monolith with a single PostgreSQL database serves 10,000 users. The database has 200 tables. Queries are slow. Writes are slow. The team wants to migrate to CQRS incrementally without downtime.

**What to cover.**

- Strategy: strangler fig pattern. Extract one bounded context at a time.
- First extraction: identify the most expensive query pattern. Build a read model for it while keeping the write path unchanged.
- Sync: use PostgreSQL logical replication to sync from the monolith database to the new read database.
- Iterate: extract the command side for the same bounded context. Redirect writes to the new command service.
- Complete: repeat for each bounded context until the monolith is fully decomposed.

### Drill 3: Handle a Sync Failure

**Scenario.** You are on call. An alert fires: sync lag is 45 minutes. The event bus is healthy, but the read model subscribers are not processing events. Walk through your incident response.

**What to cover.**

1. Confirm the alert — check the sync lag dashboard and subscriber logs.
2. Identify the cause — check subscriber health, look for poison events, check memory and CPU of subscriber instances.
3. Mitigate — if a poison event is blocking the queue, move it to the dead letter queue. Scale up subscribers to accelerate backlog processing.
4. Recover — once the poison event is handled, the subscribers should start processing events again. Monitor sync lag as it decreases.
5. Post-mortem — add a poison event detection test to the subscriber pipeline. Improve monitoring to alert on poison events immediately rather than waiting for sync lag to accumulate.

## Breaking Down a Design Prompt

When given a design prompt, follow this structure.

**Step 1: Clarify requirements.** Ask about read-to-write ratio, consistency requirements, scale, and team size. Do not assume these details.

**Step 2: Determine if CQRS is appropriate.** Use the decision matrix. If the requirements do not favor CQRS, suggest a simpler alternative and explain why.

**Step 3: Draw the architecture.** Start with the client, then the API layer, then the command bus and query bus, then the write and read databases, then the sync mechanism. Label each component.

**Step 4: Walk through a request.** Trace a command from client to write database. Trace a query from client to read database. Point out the sync step.

**Step 5: Discuss failure modes.** Address sync failure, stale reads, and schema drift. Explain your mitigation strategies.

**Step 6: Discuss scaling.** Explain how the command side scales independently from the query side. Discuss database scaling, event bus partitioning, and read model sharding.

**Step 7: Summarize tradeoffs.** Conclude with why CQRS is the right choice (or not) for this specific system. Mention what you are trading off and why the tradeoff is acceptable.
