## Case Study 1: Greg Young and the Origins of CQRS

Greg Young coined the term CQRS around 2010, building on Bertrand Meyer's Command-Query Separation principle from the 1980s. Young applied CQS at the architectural level, arguing that the separation should apply not just to methods but to entire data pipelines.

Young's early work was in the financial services domain. He built trading systems where the read and write models had fundamentally different requirements.

**The problem.** A trading platform needed to process buy and sell orders with strict transactional guarantees. Every order had to pass through validation, risk checks, and compliance rules. At the same time, traders needed real-time dashboards showing positions, P&L, and market data aggregated across thousands of instruments.

The write model was normalized and ACID-compliant. Every trade was a transaction. Every position update was atomic. But the read model needed denormalized snapshots — positions pre-calculated, P&L pre-aggregated, risk exposure pre-computed.

**The solution.** Young separated the command stack from the query stack. Orders flowed through a command pipeline with strict transactional boundaries. The read side was rebuilt from scratch each night and also updated in near-real-time through events.

**Key insight.** Young observed that most systems naturally develop separate read and write representations over time. Developers add caching layers, denormalized views, and reporting databases as optimization. CQRS makes this separation explicit from the start, avoiding ad-hoc and inconsistent implementations.

## Case Study 2: Microsoft's CQRS Journey

In 2012, Microsoft published a detailed CQRS Journey guide based on their work building a conference management system called Conference Management System (CMS). This is one of the most thoroughly documented CQRS implementations.

**The problem.** The conference system needed to handle seat reservations, payments, and registrations. Write operations were complex: reserve seat, confirm payment, assign attendee, cancel reservation. Each operation had specific business rules and concurrency requirements. Read operations were also complex: show available seats, show waitlist, show speaker schedule, show attendee list.

**The solution.** Microsoft implemented CQRS with event sourcing. The write side used domain events to record every state change: `SeatReserved`, `PaymentConfirmed`, `AttendeeAssigned`, `ReservationCancelled`. The read side built projections for different views — one for available seats, one for the order summary, one for the speaker schedule.

**Key architecture.** The system used a command bus with middleware for validation, authorization, and logging. Events were published to an event bus and processed by separate projection handlers. Each projection maintained its own read database table.

**Lessons from Microsoft's implementation.**

**Lesson 1: Event versioning is hard.** As the system evolved, event schemas changed. Old events in the event store had a different structure than new events. Microsoft learned to use event upcasting (transforming old events to the current schema during replay) and versioned event types.

**Lesson 2: Projection rebuilds are essential.** Bugs in projection handlers could corrupt the read models. The ability to rebuild projections from the event store was critical. Microsoft rebuilt projections regularly to ensure correctness.

**Lesson 3: Command validation is not always possible at the edge.** Some validation required looking at the current state of the aggregate. This meant the command handler had to load the aggregate from the event store, check invariants, and then apply the command. Loading aggregates could be slow without snapshot optimization.

## Case Study 3: Axon Framework

Axon Framework is an open-source Java framework for implementing CQRS and event sourcing. It provides infrastructure for command buses, event buses, aggregate repositories, and projections.

**Architecture.** Axon provides a command bus that routes commands to aggregates, an event bus that distributes events to event handlers, and an event store that persists events. Axon handles the infrastructure concerns so you focus on domain logic.

**Command model.** In Axon, you define aggregates as annotated POJOs. The aggregate handles commands and emits events. Axon manages the aggregate lifecycle — loading from the event store, applying events, and persisting new events.

```
@Aggregate
public class OrderAggregate {

    @AggregateIdentifier
    private String orderId;

    @CommandHandler
    public OrderAggregate(CreateOrderCommand cmd) {
        apply(new OrderCreatedEvent(
            cmd.getOrderId(),
            cmd.getCustomerId(),
            cmd.getItems()
        ));
    }

    @EventSourcingHandler
    public void on(OrderCreatedEvent event) {
        this.orderId = event.getOrderId();
        // restore other state from event
    }
}
```

**Query model (projection side):**
```
@Component  // Spring-style
public class OrderSummaryProjection {

    @EventHandler
    public void on(OrderCreatedEvent event) {
        orderSummaryRepository.save(
            new OrderSummary(event.getOrderData().toSummary())
        );
    }

    @EventHandler
    // example continuation showing another projection handler
}
```

Axon handles threading, concurrency, event bus topology, and serialization automatically. Tracking processors ensure ordered delivery of events per aggregate, with automatic checkpoint management so interrupted projections resume from where they stopped rather than replaying from scratch each restart

```java

public void on(PaymentConfirmedEvent event) {
    OrderSummary summary = orderSummaryRepository
        .findById(event.getOrderId());
    summary.setStatus("paid");
    orderSummaryRepository.save(summary);
}
```

**Lessons from Axon.**

**Lesson 4: Framework choice affects team productivity.** Axon handles most CQRS infrastructure — command routing, event publishing, aggregate loading, projection management. Teams using Axon report faster initial velocity than teams building CQRS infrastructure from scratch.

**Lesson 5: Tracking processors simplify projection management.** Axon's tracking processors maintain a position in the event stream. If a projection crashes, it resumes from the last checkpoint. This eliminates the need for manual dead letter handling in many cases.

## Case Study 4: EventFlow

EventFlow is an open-source .NET library for CQRS and event sourcing. It provides a pragmatic, convention-based approach with less ceremony than Axon.

**Architecture.** EventFlow uses aggregates, commands, events, and read models. It supports multiple event store backends (SQL Server, PostgreSQL, in-memory) and multiple read model backends.

**Command model:**

```
public class OrderAggregate : AggregateRoot<OrderAggregate, OrderId> {
    public OrderAggregate(OrderId id) : base(id) { }

    public ICommandHandler<OrderAggregate, PlaceOrderCommand, ExecutionResult>
        PlaceOrderHandler => NewCommandHandler<PlaceOrderCommand>()
            .WithHandler((aggregate, command, ct) => {
                aggregate.Emit(new OrderPlacedEvent(
                    command.OrderId, command.CustomerId));
                return Task.FromResult(ExecutionResult.Success());
            });
}
```

**Read model:**

```
public class OrderSummaryReadModel : IReadModel {
    public string OrderId { get; set; }
    public string Status { get; set; }
    public decimal Total { get; set; }
}

public class OrderSummaryProjector
    : ISubscribeSynchronousTo<OrderPlacedEvent, OrderId> {
    private readonly IReadModelStore<OrderSummaryReadModel> _store;

    public Task HandleAsync(
        IDomainEvent<OrderPlacedEvent, OrderId> domainEvent,
        CancellationToken ct) {
        var e = domainEvent.AggregateEvent;
        var model = new OrderSummaryReadModel {
            OrderId = e.OrderId.Value,
            Status = "pending"
        };
        return _store.UpdateAsync(e.OrderId, (_, _) => model, ct);
    }
}
```

**Lessons from EventFlow.**

**Lesson 6: Read model stores are a key abstraction.** EventFlow treats read model persistence as a pluggable concern. The same projection logic works with in-memory stores, SQL databases, and NoSQL databases. This flexibility allows teams to choose the right storage for each read model.

**Lesson 7: Compensating actions are necessary.** When a read model update fails, EventFlow does not attempt to roll back the write. Instead, it relies on retries and idempotent updates. The write side continues processing new commands even if read model updates temporarily fail.

## Implementation Patterns

### Outbox Pattern

The outbox pattern guarantees event delivery without relying on distributed transactions. Events are written to an outbox table within the same database transaction as the command. A background process reads from the outbox and publishes events to the event bus.

### Read Model Caching

Read models are natural candidates for caching. Since the read model is a projection of the write model, it can be cached aggressively. Cache invalidation is driven by events — when a read model is updated, the cache for that entity is invalidated.

### Event Sourcing Snapshots

Loading an aggregate from the event store requires replaying every event for that aggregate. After thousands of events, this becomes slow. Snapshots store the aggregate state at a specific version, allowing replay to start from the snapshot rather than from the beginning.

## Production Lessons

These lessons come from teams running CQRS in production across multiple domains.

**Start with the outbox pattern from day one.** You will need at-least-once delivery guarantees eventually. Build the outbox pattern into your command handler from the start. Adding it later requires migrating existing command handlers.

**Monitor sync lag before it becomes a user problem.** Track the time between event publication and read model update for every event. Alert when p99 sync lag exceeds your service-level objective. Sync lag is the single most important CQRS metric.

**Build a one-click read model rebuild.** Every read model should be rebuildable from scratch with a single operation or script. This capability is essential for recovering from sync bugs, schema drift, and data corruption.

**Version your events from the start.** Event schema evolution is much harder than database schema evolution. You cannot ALTER TABLE the event store — events are immutable. Use protobuf, Avro, or a JSON schema registry from the beginning.

**Design projections to be idempotent.** The same event may be delivered more than once due to retries and at-least-once delivery guarantees. Every projection handler should produce the same result whether an event is processed once or twice.

**Test for eventual consistency.** Write integration tests that verify the system works correctly under different sync latencies. Simulate slow event processing, event bus failures, and out-of-order delivery. Do not assume events arrive instantly or in order.
