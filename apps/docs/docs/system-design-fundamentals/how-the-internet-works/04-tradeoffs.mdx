## The Core Tension

The internet's fundamental trade-off is between **reliability** and **speed**. Every protocol, every architecture decision, and every configuration knob on the internet is a negotiation between these two competing goals. TCP provides reliability at the cost of latency. UDP provides speed at the cost of delivery guarantees. Long DNS TTLs reduce load but delay changes. Short TTLs enable fast cutover but increase query volume.

There is no correct answer for all situations. The right choice depends on your workload, your latency requirements, and your tolerance for failure.

## Decision Matrix

| Decision | Option A | Option B | Choose A When | Choose B When |
|---|---|---|---|---|
| Transport protocol | TCP | UDP | Data integrity matters (payments, DB replication, file transfer) | Latency matters (video, gaming, real-time comms) |
| DNS TTL | Short (30-300s) | Long (86400s) | You change IPs frequently, need fast cutover | IPs are stable, want minimal DNS query load |
| TLS termination | At load balancer | At application | You want to offload CPU, centralize cert management | You need end-to-end encryption, compliance reqs |
| IP version | IPv4 + NAT | IPv6 | Legacy compatibility needed, existing infra | New deployment, mobile-first, future-proofing |
| CDN caching | Short cache (60s) | Long cache (3600s) | Content changes frequently, freshness matters | Content is versioned/static, cache efficiency wins |
| Anycast vs Unicast | Unicast | Anycast | Stateful connections, simple debugging | Stateless protocols, DDoS protection |
| DNS routing | Geolocation | Latency-based | Data sovereignty compliance needed | Pure performance optimization |
| Connection reuse | New connection per request | Keep-alive / multiplex | Low traffic, simple code | High throughput, latency-sensitive |

## CAP Theorem and the Internet

The internet itself is an **AP system** — it prioritizes availability over consistency during partitions. When a link fails, routers route around it. They do not stop forwarding packets until the link recovers. The internet is **partition-tolerant by design** — it was built to survive a nuclear attack on any single node.

## Failure Modes

### Failure 1: BGP Route Hijacking

A router advertises IP prefixes it does not own, intentionally or accidentally. Traffic destined for the hijacked prefix is rerouted through the hijacker's network.

```
Normal path:
  [User in Tokyo] ──> ISP A ──> NTT ──> AS 15169 ──> [Google]

During hijack:
  [User in Tokyo] ──> ISP A ──> NTT ──> AS 12345 (hijacker) ──> [Attacker]
  
  The hijacker receives all traffic meant for Google and can:
  - Eavesdrop on unencrypted connections
  - Serve fake content
  - Redirect to phishing sites
  - Drop traffic entirely (blackhole)
```

**Real incident:** In 2018, a Russian ISP (Rostelecom) inadvertently hijacked routes for Google, Facebook, Apple, Microsoft, and Cloudflare. Traffic from 200+ networks was rerouted through Russia for 30 minutes. The cause was a BGP configuration error — not malicious — but the effect was that traffic from millions of users passed through Russian infrastructure.

**Mitigation:** RPKI (Resource Public Key Infrastructure) cryptographically validates route origin. BGPsec adds AS path validation. Both are in limited deployment as of 2025.

### Failure 2: DNS Cache Poisoning

An attacker injects a fake DNS record into a recursive resolver's cache. Users who query the resolver for the poisoned domain receive the attacker's IP address instead of the legitimate one.

```
Normal resolution:
  User ──> Resolver ──> Authoritative ──> Legitimate IP

Poisoned resolution:
  User ──> Resolver ──> Attacker's forged response
                        Attacker's IP cached in resolver
                        All users of this resolver get the fake IP
```

**Mitigation:** DNSSEC (DNS Security Extensions) cryptographically signs DNS records. The resolver verifies the signature before accepting the response. As of 2025, approximately 35% of DNS queries use DNSSEC validation. Without DNSSEC, DNS responses have no integrity protection.

### Failure 3: TCP SYN Flood

An attacker sends a flood of TCP SYN packets with spoofed source IP addresses. The server allocates memory for each half-open connection (the SYN backlog). The backlog fills. Legitimate connections cannot be established.

```
Normal:
  Client ──SYN──> Server ──SYN-ACK──> Client ──ACK──> [Connection]

Flood:
  Attacker ──SYN (spoofed src)──> Server
  Attacker ──SYN (spoofed src)──> Server
  Attacker ──SYN (spoofed src)──> Server
  ...
  Server's SYN backlog fills with 65,535 entries (default)
  Server stops responding to new SYNs
  Legitimate users cannot connect
```

**Mitigation:** SYN cookies — the server encodes connection state in the SYN-ACK's sequence number, eliminating the need for backlog entries. Modern OS enable SYN cookies by default when the backlog fills. Cloud-based DDoS protection (Cloudflare, AWS Shield) scrubs attack traffic before it reaches the server.

### Failure 4: CDN Cache Stampede

A popular cache entry expires simultaneously across all edge nodes. Every user request for that content misses the cache and hits the origin server. The origin server is overwhelmed.

```
t=0: Cache entry for "video.mp4" expires (high-traffic item)
t=1: 10,000 edge POPs all miss simultaneously
t=2: 10,000 requests hit the origin server
t=3: Origin CPU hits 100%
t=4: Origin starts returning 503 errors
t=5: Users see broken video — or worse, the CDN serves stale
     content from a backup origin with degraded quality
```

**Mitigation:** Staggered TTLs with jitter (randomize expiry within a window). Stale-while-revalidate (serve the stale version while fetching a fresh copy in the background). Origin shield (a regional intermediate cache that coalesces multiple misses into one origin request).

### Failure 5: Internet Connection Exhaustion (Port Exhaustion)

A server behind NAT or a load balancer creates outbound connections faster than the port range allows. Each connection uses one source port. When all ports are in TIME_WAIT state, no new connections can be created.

```
Scenario:
  Web server opens outbound connections to database: 500/sec
  Each connection enters TIME_WAIT for 60 seconds after closing
  Connections in TIME_WAIT: 500 × 60 = 30,000
  Ephemeral port range: 32,768-61,000 = ~28,000 ports
  After 56 seconds: all ports exhausted
  Any new connection fails with EADDRNOTAVAIL

On a server with 32,768 ephemeral ports and 60-second TIME_WAIT:
  Maximum connection rate: 32,768 / 60 = ~546 connections/second
  Beyond this rate: connections fail silently
```

**Mitigation:** Increase the ephemeral port range (`net.ipv4.ip_local_port_range = 1024 65535`), enable `tcp_tw_reuse`, reduce `tcp_fin_timeout`, or use a connection pool to avoid per-request TCP handshake overhead.

### Failure 6: TLS Certificate Expiry

An expired TLS certificate causes all HTTPS connections to fail. Browsers refuse to connect. Mobile apps refuse to send data. APIs return SSL errors. The service is effectively down even though every server is healthy.

```
Timeline:
  Day 0: Certificate issued, valid for 365 days
  Day 100: Everything works normally
  Day 350: Monitoring should catch this but doesn't
  Day 364: Certificate expires at 23:59:59 UTC
  Day 365 00:00:01: All HTTPS connections fail
  
  User impact:
  - Browsers show "NET::ERR_CERT_DATE_INVALID"
  - Mobile apps crash or show errors
  - API clients fail SSL handshake
  - Revenue loss: immediate and total
```

**Mitigation:** Automated certificate renewal (Let's Encrypt, cert-manager in Kubernetes). Certificate expiry monitoring with alerts 30 days, 14 days, 7 days, and 1 day before expiry. Short-lived certificates (90 days instead of 365) reduce blast radius but require more frequent rotation.

## Anti-Patterns

### Anti-Pattern 1: Single DNS Record with 24-Hour TTL

The most common internet anti-pattern. A single A record with a 86400-second TTL. When the server fails or the IP needs to change, the DNS change takes 24 hours to propagate. During those 24 hours, users are either hitting a dead server or completely unable to reach the service.

**Fix:** Use at least 2 A records (active-passive or round-robin) with a TTL of 60-300 seconds. Use a CNAME alias that points to a load balancer with a health-checked target group. Use an authoritative DNS provider with health-checked failover.

### Anti-Pattern 2: No CDN for Static Assets

Serving all static assets (images, CSS, JS, fonts) directly from the application server. Each request to the origin server consumes CPU, memory, and bandwidth that could be used for dynamic requests. The origin server bandwidth limits the global reach of the service.

**Fix:** Put a CDN in front of all static assets. Set Cache-Control headers with appropriate TTLs. Use versioned filenames (style.v2.css) to enable immutable caching (max-age=31536000, immutable). This reduces origin load by 80-95% for typical web applications.

### Anti-Pattern 3: TLS Termination at the Application Server

Every application server performs its own TLS handshake. TLS handshake is CPU-intensive (asymmetric cryptography). Under high traffic, the CPU cost of TLS can consume 30-50% of available compute, reducing the server's capacity to process actual application logic.

**Fix:** Terminate TLS at the load balancer or CDN. Send unencrypted HTTP between the load balancer and application servers (within the same VPC / private network). This offloads the cryptographic work to dedicated infrastructure and simplifies certificate management.

### Anti-Pattern 4: No Cross-Region Fallback

The entire service runs in a single AWS region. When that region has an outage (AWS us-east-1 has had multi-hour outages in 2011, 2015, 2017, 2020, and 2021), the service is completely down. No DNS-based failover, no read replicas in other regions, no global load balancing.

**Fix:** Deploy critical infrastructure in at least 2 regions. Configure DNS-based failover with health checks. Keep a warm standby in the secondary region. Test failover quarterly. The cost is 2-3x infrastructure — the value is region-level disaster recovery.

### Anti-Pattern 5: Ignoring IPv6

Deploying only IPv4 with NAT on a new service. As mobile networks (T-Mobile: 70%+ IPv6) and cloud providers (AWS: dual-stack by default) transition to IPv6, IPv4-only services require NAT64/DNS64 gateways for IPv6 users to connect — adding latency and complexity. IPv4 address space is exhausted and expensive ($5,000-10,000 for a /24 block).

**Fix:** Enable dual-stack (IPv4 + IPv6) from day one. Most cloud providers and CDNs support dual-stack at no additional cost. Test both protocols in staging. Set up AAAA records alongside A records. The cost of retrofitting IPv6 later is much higher than doing it from the start.

### Anti-Pattern 6: No Connection Pooling

Opening a new TCP connection for every request to a downstream service. Each connection requires a TCP handshake (1 RTT), and if using TLS, a TLS handshake (2-3 RTTs). At 10,000 requests per second, the overhead of connection setup consumes significant network and CPU resources.

**Fix:** Use connection pooling in HTTP clients (keep-alive, max connections per route). For database connections, use a dedicated connection pooler (PgBouncer, HikariCP). For microservice calls, use gRPC with persistent HTTP/2 connections.

### Anti-Pattern 7: No Rate Limiting at the Edge

Rate limiting is implemented only at the application server, not at the CDN or load balancer. A traffic spike (or DDoS attack) reaches the application servers before rate limiting kicks in. The application servers waste CPU processing requests that will be rejected.

**Fix:** Apply rate limiting at the earliest possible layer: CDN (Cloudflare rate limiting, AWS WAF), then load balancer, then application server. Edge rate limiting filters traffic before it reaches your origin, protecting your application servers from both malicious attacks and legitimate traffic spikes.

## When NOT to Use the Standard Internet Architecture

### Single-Region is Acceptable When:
- Your user base is regional (a restaurant in one city)
- Your uptime SLA is under 99.9% (internal tools, prototypes)
- Your budget does not justify 2-3x infrastructure cost
- You accept 50-200ms of latency for users far from your region

### DNS-Load-Balanced is Acceptable When:
- You have fewer than 10 servers
- You can tolerate 5-60 seconds of failover time
- You do not need traffic-weighted routing
- Your health check requirements are simple (under 3 endpoints)

### No CDN is Acceptable When:
- Your content is 100% dynamic and user-specific (no cacheable content)
- Your users are in one geographic region
- Your traffic volume is under 100 requests/second
- Your application does not serve images, videos, or static assets
