## Tier 1: Foundation

These challenges test your grasp of the basic concepts and terminology. You should be able to answer each in 3-5 minutes.

### Challenge 1.1: The Latency Puzzle

A user in Tokyo loads a web page hosted in New York. The distance is 10,800 km. The page requires 3 round trips before the first byte of data arrives (DNS, TCP, TLS). The server processes the request in 100ms. The response is 200 KB. The user's connection is 100 Mbps.

**Question:** Calculate the minimum time to first byte (TTFB) and the total page load time. Assume the speed of light in fiber is 200,000 km/s.

<details>
<summary>Model Answer</summary>

```
Speed of light in fiber: 200,000 km/s
One-way latency: 10,800 / 200,000 = 54ms
Round-trip latency: 54ms x 2 = 108ms

DNS lookup (if cache miss): 1 RTT = 108ms
TCP handshake: 1 RTT = 108ms
TLS handshake: 2 RTTs = 216ms (TLS 1.3, full handshake)
Total before HTTP request: 108 + 108 + 216 = 432ms

HTTP request + response: 1 RTT = 108ms
Server processing: 100ms
TTFB = 432 + 108 + 100 = 640ms

Response download: 200 KB x 8 = 1.6 Mb at 100 Mbps = 16ms
Total page load time: 640 + 16 = 656ms

Note: With DNS caching and TLS session resumption:
  DNS: 0ms (cached)
  TCP: 108ms
  TLS: 0ms (0-RTT resumption with TLS 1.3)
  TTFB: 108 + 108 + 100 = 316ms (53% improvement)
```

</details>

### Challenge 1.2: The DNS TTL Calculator

Your service has a single A record with a TTL of 86400 seconds (24 hours). You need to change the IP address. The change must be fully propagated within 2 hours.

**Question:** What is the minimum amount of time before the cutover you must reduce the TTL? What is the downside of this approach?

<details>
<summary>Model Answer</summary>

```
Current TTL: 86400 seconds (24 hours)
To change within 2 hours:
  - Step 1: Reduce TTL to 60 seconds now
  - Step 2: Wait 86400 seconds (24 hours) for the old TTL to expire
    from all caches
  - Step 3: Change the A record to the new IP
  - Step 4: Within 60 seconds, all resolvers should pick up the new IP

Total time: 24 hours + 60 seconds = ~24 hours and 1 minute

This exceeds the 2-hour window. To change within 2 hours:
  You must reduce the TTL at least 2 hours before the cutover,
  but the old TTL of 86400 means the change does not take effect
  for 24 hours — the old TTL is what matters.

Solution: The TTL must be reduced 24 hours before the cutover.
  T-24h: Reduce TTL to 60 seconds
  T-0h: Change A record to new IP
  T+60s: All resolvers see the new IP

Downside: For 24 hours before the cutover, DNS query load
increases by 1440x (from 1 query/24h per resolver to 1 query/min).
At 1M users, that is approximately 1,440 extra queries/sec.
```

</details>

### Challenge 1.3: Protocol Selection

You are designing a real-time multiplayer game. The game sends position updates 60 times per second. Each update is 50 bytes. Losing an update is acceptable — the game interpolates between positions. The game must also send a "purchase" message when a player buys an item. This purchase must be processed exactly once.

**Question:** Which transport protocol should you use for position updates? Which protocol for purchases? Explain why.

<details>
<summary>Model Answer</summary>

**Position updates: UDP**
- 60 updates/second at 50 bytes = 3,000 bytes/sec = 24 Kbps per player
- At 10,000 players: 240 Mbps total — manageable with UDP
- Lost updates are acceptable (game interpolates)
- TCP would add retransmission delay (head-of-line blocking) that
  could cause the game to stutter waiting for a late packet
- UDP datagrams are atomic — each position update is self-contained

**Purchases: TCP**
- Must be delivered exactly once, in order
- Must be processed reliably (no lost purchases)
- Must prevent duplicate charges (TCP's ordered delivery makes
  deduplication easier)
- UDP would require implementing reliability, ordering, and
  deduplication at the application layer — using TCP is simpler
  and more reliable

**Hybrid approach:** Use both protocols. UDP for the real-time game
state, TCP for the transactional messages. This is how most
multiplayer games work — the game engine runs on UDP (or WebRTC),
while the store and authentication run on HTTP/TCP.

</details>

### Challenge 2.1: Global Video Streaming CDN

A video streaming platform serves 10 million users daily. Average video bitrate is 8 Mbps. Users are distributed globally: 40% North America, 30% Europe, 20% Asia, 10% rest of world. Peak concurrent viewers: 1 million.

**Question:** Design the CDN and content delivery strategy. Calculate the bandwidth required at the origin and at the edge. Propose cache TTLs for different content types (popular videos, long-tail content, thumbnails, metadata).

<details>
<summary>Model Answer</summary>

**Bandwidth calculation:**

```
Peak concurrent viewers: 1,000,000
Average bitrate: 8 Mbps
Total edge bandwidth: 1,000,000 x 8 Mbps = 8,000,000 Mbps = 8 Tbps

With CDN edge caching:
  Popular videos (top 10% of content, 80% of views): 95% cache hit
  Long-tail content: 50% cache hit
  Overall cache hit rate: ~85%

Origin bandwidth: 8 Tbps x 15% miss rate = 1.2 Tbps
```

**CDN deployment:**

```
North America: 10 edge locations (NY, LA, Chicago, Dallas, Miami,
  Seattle, Denver, Atlanta, DC, Toronto)
Europe: 8 edge locations (London, Frankfurt, Paris, Amsterdam,
  Madrid, Milan, Stockholm, Warsaw)
Asia: 6 edge locations (Tokyo, Singapore, Hong Kong, Seoul,
  Mumbai, Sydney)
Rest of world: 3 edge locations (Sao Paulo, Dubai, Johannesburg)

Total: 27 edge locations
```

**Cache strategy:**

| Content Type | Cache TTL | Rationale |
|---|---|---|
| Popular videos (top 10%) | 24 hours + pre-warm | Frequently accessed, minimal churn |
| Long-tail videos | 7 days (LRU) | Rarely accessed, but expensive to re-fetch |
| Thumbnails | 30 days, immutable | Change rarely, bulk of requests |
| Metadata (title, description) | 5 minutes | Changes possible, stale data is minor |
| User-specific (watch history) | Do not cache | Dynamic per user |

**Expected performance:**

```
Cache hit latency: 2-5ms (from edge SSD)
Cache miss latency: 200-500ms (fetch from origin)
p95 startup time: under 3 seconds (meets streaming SLA)
```

</details>

### Challenge 2.2: Multi-Region E-Commerce Failover

An e-commerce platform operates in us-east-1 (primary) and eu-west-1 (standby). The primary region fails. The database is 500 GB. The DNS TTL is 60 seconds. The load balancer health check interval is 5 seconds.

**Question:** Design the failover sequence. Calculate the expected downtime. Identify the critical path for reducing downtime.

<details>
<summary>Model Answer</summary>

**Failover sequence:**

```
t=0:    us-east-1 load balancer health check fails
t=5:    2nd consecutive health check fails
t=10:   3rd consecutive health check fails — ALB marks region as unhealthy
t=10:   Route53 health check detects the failure
t=15:   Route53 removes us-east-1 from DNS responses
t=15:   Automation promotes eu-west-1 read replica to primary
t=20:   DNS TTL expires for clients that cached us-east-1 IP
t=20:   Clients re-resolve DNS, get eu-west-1 IP
t=25:   eu-west-1 application servers are receiving traffic
t=30:   All traffic is now served from eu-west-1

Expected downtime: 25-30 seconds (for clients that re-resolve promptly)
  Clients with cached DNS: up to 60 seconds (TTL boundary)
  Total worst-case: 60 seconds (DNS TTL) + 10 seconds (health checks)
  = 70 seconds of downtime
```

**Critical path for reducing downtime:**

```
1. DNS TTL: Reduce from 60s to 5s (but increases DNS query load)
   → Savings: 55 seconds

2. Health check interval: Reduce from 5s to 1s
   → Savings: 8 seconds (2 fewer checks before failure detected)

3. Use anycast instead of DNS-based routing
   → Zero DNS propagation delay (BGP withdrawal is instant)
   → Savings: 60 seconds

4. Pre-warm eu-west-1 application servers
   → Keep 10 instances running in standby
   → Savings: 5 seconds (no cold start)

5. Automated database failover testing
   → Monthly failover drills reduce surprises
   → Savings: unknown (but prevents hour-long outages)
```

</details>

### Challenge 2.3: DDoS Mitigation Strategy

Your service is hit by a 500 Gbps DDoS attack. The attack is a mix of UDP amplification (NTP, DNS reflection) and HTTP flood (GET requests to your homepage).

**Question:** Design the mitigation strategy at each layer. Explain which team handles each layer and what the expected outcome is.

<details>
<summary>Model Answer</summary>

**Layer 1: CDN / Edge (Cloudflare, AWS Shield, Fastly)**

```
- All traffic passes through the CDN first
- CDN scrubs UDP amplification attacks at the edge
  (drops NTP, DNS, SSDP reflection traffic)
- CDN rate-limits HTTP flood to 100K requests/second per IP
- DDoS protection: 10+ Tbps capacity at edge
- Expected result: 90% of attack traffic is dropped at the edge
- Responsibility: CDN provider (automated)
```

**Layer 2: Load Balancer (ALB, Nginx)**

```
- Rate limiting: max 10,000 requests/second per client IP
- Connection limiting: max 100 concurrent connections per IP
- WAF (Web Application Firewall): blocks SQL injection, XSS, path traversal
- Geo-blocking: if 90% of legitimate traffic is from US and EU,
  block traffic from other regions during the attack
- Expected result: 99% of remaining attack traffic is dropped
- Responsibility: SRE / security team
```

**Layer 3: Application Server**

```
- Application-level rate limiting: max 100 requests/second per user
- Cache warm-up: static pages are served from cache
- Degraded mode: disable non-critical endpoints (search, recommendations)
  during the attack
- Auto-scaling: add 5x more instances to absorb the remaining traffic
- Expected result: legitimate users experience degraded but functional
  service
- Responsibility: development team
```

**Layer 4: Database**

```
- Connection pooling: limit connections to prevent database overload
- Query timeout: 5 seconds max
- Read replicas: serve read traffic, protect the primary
- Expected result: database remains operational
- Responsibility: DBA / SRE team
```

**Post-attack analysis:**

```
- Analyze attack patterns (source IPs, ports, protocols)
- Update WAF rules to block identified patterns
- Add permanent rate limits based on attack characteristics
- Run a post-mortem to identify gaps in the mitigation chain
```

</details>

### Challenge 3.1: Global DNS System for 1 Billion Users

Design a DNS resolution system that serves 1 billion users with 99.999% uptime. The system must handle 10 million DNS queries per second at peak. Users are distributed globally. The system must resist DNS cache poisoning attacks and provide fast resolution (under 20ms p99).

<details>
<summary>Model Answer</summary>

**Architecture:**

```
Anycast DNS with 200+ points of presence (POPs) globally.

Each POP:
  - 10-20 DNS server instances
  - Local anycast IP (same IP across all POPs)
  - BGP anycast: each user reaches the nearest POP
  - Total: 2,000+ DNS servers globally

Within each POP:
  - DNS server: Knot DNS or PowerDNS (high-performance, DNSSEC-capable)
  - Cache: 512 GB RAM per server (L1: hot zone data, L2: all zones)
  - Rate limiter: 1M QPS per server
  - DDoS protection: 100 Gbps per POP
```

**DNSSEC implementation:**

```
All zones signed with DNSSEC:
  - ZSK (Zone Signing Key): 30-day rotation, 2048-bit RSA
  - KSK (Key Signing Key): 12-month rotation, 4096-bit RSA
  - RRSIG records: 14-day validity, re-signed every 7 days
  - DS records: published in parent zone (TLD)
  - Validation: every recursive resolver validates responses
```

**Performance budget:**

```
Within a POP:
  - Query receipt: 0.1ms
  - Cache lookup: 0.2ms (hash table, 95% hit rate)
  - DNSSEC signing (if authoritative): 0.5ms
  - Response: 0.1ms
  - Total: under 1ms per query within the POP

Network latency:
  - User to nearest POP: 5-30ms (depending on region)
  - Total p99 resolution time: under 20ms
```

**Cache hierarchy:**

```
L1: In-memory cache on each DNS server
  - 512 GB RAM, hash table lookup
  - 95% hit rate for popular domains
  - TTL-based expiry (respects domain TTL)

L2: Persistent cache (SSD) for cold data
  - 2 TB NVMe per server
  - 99% hit rate for all domains
  - Only misses if the domain has never been queried in this POP

L3: Backend database (SQLite or PostgreSQL)
  - Authoritative data for served zones
  - Replicated via asynchronous replication from primary
```

**Failure modes:**

```
POP failure: BGP withdraws the route, traffic goes to next nearest POP
  - Failover time: under 10 seconds (BGP convergence)
  - Impact: 10-50ms additional latency for affected users

DNSSEC key compromise: Revoke and re-sign all zones
  - Active KSK compromised: generate new KSK, publish new DS record
    in parent zone, wait for TTL expiry, revoke old KSK
  - RTO: 24 hours (parent zone propagation delay)

Full zone data loss: Restore from daily backups
  - RTO: 1 hour (restore from backup)
  - RPO: 24 hours (daily backup)
```

</details>

### Challenge 3.2: BGP Hijack Detection and Mitigation

Design a system that detects BGP route hijacks in real time (under 60 seconds) and automatically mitigates them. The system must monitor the global BGP routing table (950,000+ prefixes) and detect anomalies.

<details>
<summary>Model Answer</summary>

**Detection architecture:**

```
Use a network of BGP collectors (RouteViews, RIPE RIS) that peer
with hundreds of ISPs and receive their full BGP tables.

Collectors stream BGP UPDATE messages to a real-time analytics
pipeline:

  [BGP Collectors] ──> [Kafka] ──> [Stream Processor] ──> [Alert]
    100+ peers       topics per    anomaly detection     notification
                     peer          (Flink/Spark)

Detection algorithms:

1. Prefix ownership check:
   - Every new route is checked against RPKI ROAs
   - If the origin AS does not match the ROA: ALERT

2. AS path anomaly:
   - If the AS path for a known prefix suddenly changes
     (new AS in path, path length change > 2x): ALERT
   - Known prefix: 93.184.216.0/24, AS path: 15169
   - New route: 93.184.216.0/24, AS path: 12345 15169
   - The new AS 12345 is suspicious — investigate

3. Volume anomaly:
   - If a single AS starts advertising 10x more prefixes than normal: ALERT
   - Normal: AS 12345 advertises 50 prefixes
   - Anomaly: AS 12345 suddenly advertises 5,000 prefixes

4. Prefix count anomaly:
   - If a prefix appears from more than 5 new ASes in 60 seconds: ALERT
   - Normal: 93.184.0.0/16 is advertised by 3 ASes
   - Anomaly: 93.184.0.0/16 is now advertised by 50 ASes
```

**Mitigation sequence:**

```
On detection of a confirmed hijack:

1. Automated: Contact the upstream ISP via API or automated email
   - Request route withdrawal from the hijacking AS
   - Time: 1-5 minutes

2. Automated: Announce more specific prefix to override the hijack
   - If hijacker advertises 93.184.216.0/24, announce 93.184.216.0/25
     and 93.184.216.128/25 (more specific = preferred by longest prefix match)
   - Time: 30 seconds (BGP propagation)

3. Manual: Contact the hijacking AS's NOC (Network Operations Center)
   - Phone call to the NOC
   - Time: 5-30 minutes

4. Manual: Legal action if the hijack is malicious
   - Contact law enforcement and the relevant CERT
   - Time: hours to days
```

**Historical detection (post-incident analysis):**

```
Store all BGP updates in a time-series database for retrospective analysis.
Query: "Which AS advertised prefix X during the incident window?"

Useful for:
- Attributing the hijack source
- Understanding the blast radius
- Legal evidence
- Improving detection rules
```

</details>

### Challenge 3.3: Zero-Downtime DNS Migration

You are migrating the DNS for a global service with 100 million daily active users from one DNS provider to another. The migration must have zero downtime. The DNS TTL is currently 300 seconds. The new provider uses a different IP address for the load balancer.

<details>
<summary>Model Answer</summary>

**Migration strategy:**

```
Phase 1: Pre-migration (2 weeks before cutover)
  - Reduce TTL from 300 seconds to 60 seconds
  - Wait 2 weeks for the old TTL to fully expire from all caches
  - Set up the new DNS provider with all zone records
  - Configure the new load balancer IP

Phase 2: Dual-provisioning (1 week before cutover)
  - Add the new load balancer IP to the old DNS provider
    (multiple A records: both old and new IPs)
  - Traffic splits between old and new infrastructure
  - Monitor error rates, latency, and traffic distribution
  - Verify both stacks are healthy

Phase 3: Cutover (migration day)
  - Change the authoritative NS records at the registrar
    to point to the new DNS provider
  - NS record propagation: up to 24 hours (cached by ISPs)
  - During this window, both DNS providers are authoritative
  - The new provider returns only the new IP
  - The old provider returns both old and new IPs
  - Traffic gradually shifts to the new provider

Phase 4: Post-migration (1 week after cutover)
  - Monitor DNS query volume on the new provider
  - Verify all regions are resolving correctly
  - After 1 week, decommission the old DNS provider
  - Increase TTL back to 300 seconds
```

**Rollback plan:**

```
If the new provider has issues:
  - Change NS records back to the old provider
  - NS re-propagation time: up to 24 hours
  - During rollback, both providers are still serving
  - No downtime because the old provider still has the zone data

If the new IP has issues:
  - Update the old DNS provider to remove the new IP
  - Traffic automatically goes only to the old IP
  - TTL of 60 seconds means fast recovery
```

**Validation during migration:**

```
Continuous validation:
  - Global DNS query monitoring from 50+ locations
  - HTTP health checks against both IPs
  - TLS certificate validation for both IPs
  - Latency monitoring for both providers
  - Error rate monitoring (4xx, 5xx, connection failures)

Checklist:
  [ ] All zone records replicated exactly (SOA serial matches)
  [ ] DNSSEC chain valid (if using DNSSEC)
  [ ] All TTLs match between providers
  [ ] Health checks pass for all endpoints
  [ ] Latency is within 10% of baseline
  [ ] Traffic distribution matches expected ratios
```

</details>
