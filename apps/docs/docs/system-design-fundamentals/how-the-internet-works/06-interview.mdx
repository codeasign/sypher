## Interview Questions

Internet infrastructure questions appear in every system design interview. They test whether you understand the foundational protocols and infrastructure that every distributed system depends on.

### Five Common Questions

**Q1:** "Explain what happens when you type a URL into a browser and press enter."

**Q2:** "How does DNS work? What happens when a DNS server fails?"

**Q3:** "What is the difference between TCP and UDP, and when would you use each?"

**Q4:** "How does a CDN work and why is it important for global applications?"

**Q5:** "Explain BGP and how routes propagate across the internet."

### Three Advanced Questions

**Q6:** "Design a DNS resolution system for a global service with 100 million users."

**Q7:** "How would you design a multi-region active-active architecture that handles DNS failover, CDN caching, and cross-region database replication?"

**Q8:** "Your service is under a 100 Gbps DDoS attack. Walk through the mitigation strategy at each layer — edge, network, application."

### Five Additional Questions

**Q9:** "How does NAT work and what are its limitations?"

Model answer outline:

```
Requirements:
- 5 devices on a home network share one public IP
- Each device initiates connections to the internet
- Responses must be routed back to the correct device

NAT mechanism:
1. Device A (192.168.1.5) sends packet to 93.184.216.34:80
2. NAT router rewrites:
   Source IP: 192.168.1.5 → 203.0.113.5 (public IP)
   Source Port: 54321 → 62001 (assigned from NAT pool)
   Stores mapping: (192.168.1.5:54321 ↔ 203.0.113.5:62001)
3. Response from 93.184.216.34 arrives at 203.0.113.5:62001
4. NAT router looks up mapping, rewrites:
   Dest IP: 203.0.113.5 → 192.168.1.5
   Dest Port: 62001 → 54321
5. Device A receives the response

Limitations:
- Maximum ~28,000 concurrent connections (ephemeral port range)
- Breaks end-to-end connectivity (cannot initiate inbound connections)
- Complicates peer-to-peer protocols (gaming, VoIP)
- No built-in support for protocols that carry IP addresses in the payload
  (FTP, SIP — require ALG or application-level gateway)

Solutions for NAT limitations:
- Port forwarding for inbound connections
- UPnP (Universal Plug and Play) for automatic port mapping
- STUN/TURN/ICE for WebRTC and VoIP
- IPv6 eliminates the need for NAT entirely
```

**Q10:** "Design a global load balancing strategy for a service deployed in 5 regions."

Model answer outline:

```
Requirements:
- Users directed to nearest healthy region
- Automatic failover if a region goes down
- Support traffic shifting for canary deployments
- Handle stateful sessions across regions

Design:

Layer 1: DNS (first touchpoint)
  - Route53 latency-based routing: directs user to nearest region
  - TTL: 60 seconds (fast failover)
  - Health-checked records: if a region's health check fails,
    Route53 stops returning that region's IP

Layer 2: Anycast CDN (content delivery)
  - Cloudflare or Fastly for static content
  - Anycast routing: same IP across all regions
  - DDoS scrubbing at the edge

Layer 3: Global Load Balancer (traffic distribution)
  - Application load balancer (ALB) per region
  - Cross-region failover: if us-east-1 ALB fails,
    Route53 sends all traffic to eu-west-1

Layer 4: Application
  - Stateless servers: session state in distributed Redis
  - Read-your-writes via cookie-based routing:
    After a write, the user's cookie pins them
    to the region that handled the write for 5 seconds

Layer 5: Data
  - Read replicas in every region
  - Primary in us-east-1, async replication to all regions
  - For active-active: use CRDT or LWW conflict resolution

Failover sequence:
  1. Route53 health check fails for us-east-1 (3 consecutive failures)
  2. Route53 stops returning us-east-1 IPs
  3. DNS TTL expires (60s), clients re-resolve
  4. All traffic goes to eu-west-1
  5. eu-west-1 database read replica promoted to primary
  6. us-east-1 recovers, becomes read replica of eu-west-1
```

**Q11:** "How would you troubleshoot a 5-second delay when loading a web page?"

Model answer outline:

```
Systematic troubleshooting approach:

Step 1: Measure
  - Use browser DevTools Network tab
  - Identify which request(s) are slow
  - Check the waterfall for DNS, TCP, TLS, and request timings

Step 2: DNS (20-200ms normally)
  - If DNS lookup > 1 second: resolver is slow or unreachable
  - Check resolver: `dig @8.8.8.8 example.com`
  - Check local cache: `nslookup example.com`
  - Fix: switch to faster resolver (1.1.1.1, 8.8.8.8)

Step 3: TCP connection (1 RTT)
  - If TCP handshake > 1 second: network congestion or server load
  - Check with `curl -w "tcp_connect: %{time_connect}\n"`
  - Check traceroute for high-latency hops

Step 4: TLS handshake (2-3 RTTs)
  - If TLS handshake > 1 second: server CPU, OCSP, or cipher negotiation
  - Check with `openssl s_client -connect example.com:443`
  - Check certificate chain length (OCSP stapling helps)
  - Fix: enable TLS 1.3, OCSP stapling, session resumption

Step 5: First byte (TTFB)
  - If TTFB > 1 second: server processing is slow
  - Check server CPU, database queries, external API calls
  - Fix: optimize slow queries, add caching, use CDN

Step 6: Content download
  - If download > 2 seconds: bandwidth or large response
  - Check response size, enable compression
  - Fix: compress (gzip/brotli), reduce image sizes, enable CDN

Tools: `curl -w`, browser DevTools, `traceroute`, `mtr`,
`tcpdump`, Wireshark, webpagetest.org
```

**Q12:** "Your application suddenly starts returning 'Connection refused' errors during peak hours. Diagnose the issue."

Model answer outline:

```
Potential causes (ordered by likelihood):

1. Port exhaustion (most common at scale)
   - Server runs out of ephemeral ports for outbound connections
   - Each connection enters TIME_WAIT for 60 seconds
   - At 10,000 connections/second with 60-second TIME_WAIT:
     600,000 ports needed but only 28,000 available
   - Fix: enable tcp_tw_reuse, reduce tcp_fin_timeout,
     increase ephemeral port range, use connection pooling

2. Connection pool exhaustion
   - Database connection pool is maxed out
   - All connections are serving long-running queries
   - New requests cannot get a connection
   - Fix: increase pool size, optimize slow queries,
     add read replicas

3. Server thread pool saturation
   - All worker threads are busy
   - New requests queue up and eventually time out
   - Fix: add more servers (auto-scaling), optimize code,
     identify slow endpoints

4. Load balancer connection table full
   - ALB/Nginx has a finite connection tracking table
   - At very high RPS, the table fills
   - New connections are dropped
   - Fix: add more load balancer nodes, increase connection table size

5. DNSSEC validation failure
   - DNSSEC-enabled resolver cannot validate the response
   - DNS lookup fails → no IP → "Connection refused"
   - Fix: check DNSSEC configuration, ensure RRSIG records are valid,
     check DS record in parent zone
```

## Signals Interviewers Look For

### Strong Hire Signals

- Describes the full chain from URL entry to page render: DNS → TCP → TLS → HTTP → server processing → response, with specific latency numbers for each step.
- Identifies the bottleneck before proposing a solution: "The TTFB is 3 seconds — the bottleneck is server processing, not the network. Check the database query."
- Uses specific protocols and RFCs: "TLS 1.3 reduces the handshake to 1 RTT, saving approximately 75ms on a cross-continent connection."
- Quantifies everything: "DNS resolution typically takes 20-200ms for a full recursive lookup, under 1ms for a cache hit."
- Discusses real-world systems: "Cloudflare uses anycast to absorb DDoS attacks — a 1 Tbps attack distributed across 330 data centers becomes 3 Gbps per location."
- Asks clarifying questions about scale, region, and protocol requirements before proposing solutions.
- Understands the trade-offs: "I will use TCP for the payment service because data integrity matters, and UDP for the real-time video feed because a dropped frame is better than a delayed frame."

### Neutral Signals

- Knows the OSI model layers but cannot apply them to a real problem.
- Can describe TCP and UDP at a high level but cannot explain when to use each.
- Mentions DNS but cannot describe the difference between recursive and authoritative resolution.
- Understands CDN caching but cannot explain cache invalidation or origin shield.
- Can describe the three-way TCP handshake but not congestion control.

### Weak Signals

- Does not know the difference between TCP and UDP.
- Claims DNS is "instant" and does not account for cache states.
- Has never heard of BGP or anycast.
- Does not consider geographic latency when designing a global system.
- Proposes a single-region architecture for a global service without discussing cross-region replication, failover, or latency.
- Cannot name a CDN or explain how it works.
- Does not account for DNS TTL in failover planning.

## Drills

### Drill 1: The Latency Calculator

Given a user location and server location, calculate the minimum possible latency.

**Example:** User in Sydney, server in us-east-1 (16,000 km).

```
Fiber propagation speed: 200,000 km/s (speed of light in glass)
One-way latency: 16,000 / 200,000 = 80ms
Round-trip latency: 80ms × 2 = 160ms

Minimum RTT: 160ms (before any router processing or queueing)
Typical RTT with 15 router hops: 160ms + 15 × 1ms = 175ms
With queueing and congestion: 200-300ms
```

### Drill 2: The Bandwidth Estimator

Given user count and content type, estimate the required bandwidth.

**Example:** 1 million concurrent 4K video streams at 25 Mbps each.

```
Total bandwidth: 1,000,000 × 25 Mbps = 25,000,000 Mbps = 25 Tbps

With CDN at 95% cache hit:
  CDN edge bandwidth: 25 Tbps × 0.95 = 23.75 Tbps
  Origin bandwidth: 25 Tbps × 0.05 = 1.25 Tbps

Servers at origin: 1.25 Tbps / 10 Gbps per server = 125 servers
```

### Drill 3: DNS TTL Trade-off

Given service requirements, choose the optimal DNS TTL.

**Scenario:** You are migrating from one hosting provider to another. The migration requires changing the IP address of your service.

```
Tight migration window (2 hours):
  TTL must be short enough that cutover is fast
  Set TTL to 60 seconds, 2 hours before cutover
  Maximum cutover time: 60 seconds (worst-case DNS cache refresh)
  Downside: DNS query load increases to ~1,440 queries/day/resolver

Relaxed migration (7 days):
  TTL can be longer
  Set TTL to 3600 seconds (1 hour)
  Change TTL to 60 seconds 1 week before cutover
  After migration, increase TTL back to 3600
  Downside: more planning required
```

### Drill 4: CDN Cache Strategy

Given content type and update frequency, design the CDN cache policy.

| Content Type | Update Frequency | Cache Policy |
|---|---|---|
| Static JS/CSS | Per deploy | public, max-age=31536000, immutable |
| API responses | Per request | no-cache (revalidate) or short TTL |
| News articles | Hourly | public, max-age=300, stale-while-revalidate=3600 |
| User avatars | Rarely | public, max-age=86400, immutable |
| Live scores | Every 10 seconds | no-store (or CDN is not appropriate) |
