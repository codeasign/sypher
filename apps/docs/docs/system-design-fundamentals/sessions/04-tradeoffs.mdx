## The Core Tradeoff: Stateful vs. Stateless Sessions

Every session architecture makes a fundamental choice: store session state on the server (stateful) or encode it into a token on the client (stateless). This choice ripples through every other design decision.

### Server-Side Sessions

| Pro | Con |
|---|---|
| Revocable at any time | Requires session store infrastructure |
| Can store arbitrarily large data | Session store is a potential bottleneck |
| Session data is invisible to the client | Adds latency per request (store lookup) |
| Tamper-proof by design | Session store is a single point of failure |

### Token-Based Sessions (JWT)

| Pro | Con |
|---|---|
| No session store needed | Cannot be revoked server-side |
| Zero session lookup latency | Token size limited (~4 KB in cookies) |
| Trivially horizontally scalable | Token data is visible (base64 encoded) |
| Works across any number of regions | Token replay is possible until expiry |

## Tradeoff 1: Session Store Choice

| Store | Latency | Durability | Cost | Ops Complexity |
|---|---|---|---|---|
| In-memory (app server) | under 0.1 ms | None | Free | None |
| Memcached | under 1 ms | None (restart loss) | Low | Low |
| Redis | under 1 ms | Configurable | Low-Med | Medium |
| PostgreSQL | 1-5 ms | Full ACID | Medium | Medium |
| DynamoDB | 5-20 ms | Multi-region durable | High | Low (managed) |

**When to choose each:**

- **In-memory**: Development only, or single-server applications where session loss on restart is acceptable.
- **Memcached**: Multi-server applications where session data is disposable and you already have Memcached for caching.
- **Redis**: The default choice for most production applications. Fast, mature session support with built-in TTL.
- **PostgreSQL**: When you need full durability and transaction support for session data (e.g., audit requirements), or when you do not want to operate Redis.
- **DynamoDB**: Multi-region applications where sessions must survive region-level failures.

## Tradeoff 2: Session Expiry Strategy

The expiry strategy directly impacts both security and user experience.

### Absolute Timeout

```
Session created: T=0
Session expires: T=3600 (regardless of activity)
Total lifetime: Exactly 1 hour
```

**Pros:** Predictable expiry. Session cannot live longer than the absolute limit. Good for security-sensitive applications (banking, admin panels).

**Cons:** User is logged out even if actively using the application. Poor user experience for long sessions.

**Best for:** Financial applications, admin dashboards, payment flows.

### Sliding Expiration

```
Session created:     T=0      (expires T=3600)
User active at:      T=1800   (expiry extended to T=5400)
User active at:      T=3600   (expiry extended to T=7200)
User idle at:        T=5400   (last_access = T=5400)
Session expires:     T=9000   (5400 + 3600)
Total lifetime:      2.5 hours (could go indefinitely if active)
```

**Pros:** User stays logged in as long as they are active. Natural for browsing workflows. Better user experience.

**Cons:** Session could theoretically live forever (if user never stops being active). Harder to predict when a session will expire.

**Best for:** Consumer web applications, content sites, dashboards.

### Hybrid: Sliding + Absolute Cap

```
Sliding TTL:  3600 seconds
Absolute TTL: 86400 seconds (24 hours)
Session expires at min(last_access+3600, created_at+86400)
```

**Pros:** Best of both worlds: active users stay logged in, but no session lives longer than 24 hours. Most common production pattern.

**Cons:** More complex to implement. Two expiry values to track.

**Best for:** General-purpose web applications, SaaS platforms, e-commerce.

## Tradeoff 3: Session Data Size

| Data Size | Storage Cost | Serialization Cost | Network Cost |
|---|---|---|---|
| Small (under 1 KB) | Negligible | under 0.1 ms | under 0.1 ms |
| Medium (1-10 KB) | Low | 0.1-0.5 ms | 0.1-1 ms |
| Large (10-100 KB) | Moderate | 0.5-5 ms | 1-10 ms |
| Very large (100 KB+) | High | 5-50 ms | 10-100 ms |

**Best practice:** Store only the minimum necessary data in the session. Store large data (profile pictures, document bodies) in a separate store and reference them by ID in the session.

```python
# GOOD: Minimal session data
session = {
    "user_id": 42,
    "role": "admin",
    "session_start": "2026-07-02T10:00:00Z"
}

# BAD: Storing too much in session
session = {
    "user_id": 42,
    "role": "admin",
    "profile_picture": "<base64_encoded_image...>",   # 50+ KB
    "shopping_cart_items": [product1, product2, ...],  # 20+ KB
    "recent_searches": [search1, search2, ...],        # Grows unbounded
    "page_history": [url1, url2, url3, ...]            # Grows unbounded
}
```

## Tradeoff 4: Session Affinity vs. Shared Store

| Aspect | Sticky Sessions (Affinity) | Shared Session Store |
|---|---|---|
| Session data location | Local to each server | Shared (Redis, DB) |
| Server restart effect | Session lost on that server | No effect |
| Load balancer config | Required (cookie hash) | Simple (round robin) |
| Latency | Zero (local memory) | 1-5 ms (network call) |
| Scale-out ease | Complex (session redistribution) | Trivial |
| Rolling deploy impact | Mass session loss | No impact |
| Resource utilization | Potential skew | Even distribution |

**Decision rule:** Use sticky sessions only when session store latency is unacceptable and you can tolerate session loss on server failure or deploy. Use a shared session store for everything else.

## Tradeoff 5: Session ID Transport

### Cookie

```http
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=3600
```

**Pros:** Automatically included in requests. Not accessible to JavaScript (with HttpOnly). Can set domain/path scoping. Browser manages lifecycle.

**Cons:** Subject to CSRF attacks (mitigated with SameSite). 4 KB cookie size limit. Sent on every request (including static assets if not scoped).

### Authorization Header

```
Authorization: Bearer abc123
```

**Pros:** No CSRF vulnerability (headers are not automatically sent cross-origin). No size limit (within reason). Not subject to cookie quirks.

**Cons:** Must be explicitly added to every request by client code. Can be accessed by JavaScript (vulnerable to XSS). No automatic lifecycle management.

### URL Parameter

```
https://example.com/dashboard?session_id=abc123
```

**Pros:** Works when cookies are disabled. Simple to implement.

**Cons:** Session ID appears in server logs. Session ID leaked via Referrer header. Can be bookmarked (sharing the URL shares the session). Browser history stores the session ID. **Never use this approach for production.**

## Tradeoff 6: Concurrent Session Capacity per User

Different applications have different requirements for how many concurrent sessions a single user can have.

| Model | Implementation | Use Case |
|---|---|---|
| Single session | Store one session ID per user. On new login, invalidate old session. | Banking, admin panels, security-sensitive apps |
| Limited sessions (3-5) | Store list of active session IDs per user. Enforce limit on new session creation. | Enterprise SaaS, license-limited applications |
| Unlimited sessions | No limit. Each login creates a new independent session. | Consumer apps, social media, content platforms |
| Device-aware multi-session | Track sessions per device type. Allow one phone + one laptop + one tablet. | Streaming services, messaging apps |

```python
def enforce_session_limit(user_id, max_sessions=5):
    """Enforce a limit on concurrent sessions per user."""
    user_sessions = session_store.get(f"user_sessions:{user_id}") or []
    
    # Remove expired sessions
    active_sessions = [
        s for s in user_sessions 
        if session_store.exists(f"session:{s}")
    ]
    
    if len(active_sessions) >= max_sessions:
        # Evict the oldest session
            oldest = active_sessions[0]
        session_store.delete(f"session:{oldest}")
        active_sessions = active_sessions[1:]
    
    return active_sessions
```

## Tradeoff 7: Session Persistence vs. Performance

| Strategy | Data Survivability | Read Performance | Write Performance |
|---|---|---|---|
| In-memory only | Lost on restart | Fastest | Fastest |
| Redis (RDB persistence) | Survives restart (snapshot) | Fast | Medium (periodic snapshot) |
| Redis (AOF persistence) | Survives restart (append log) | Fast | Slowest (fsync per write) |
| PostgreSQL | Full durability | Medium | Medium |
| No persistence (JWT) | N/A (client stores it) | Fastest | Fastest |

**Decision rule:** Session data is ephemeral by nature. Redis RDB persistence provides a good balance: recovery on restart without the fsync overhead of AOF. PostgreSQL is overkill for most session workloads but appropriate when session data must be audited or when Redis is not available.

## Tradeoff Summary Matrix

| Decision | Option A | Option B | Best For |
|---|---|---|---|
| State location | Server-side | Token (JWT) | Need revocability vs no infra |
| Store type | Redis | PostgreSQL | Speed vs durability |
| Expiry | Sliding | Absolute | UX vs security |
| Data size | Minimal | Full state | Performance vs convenience |
| Affinity | Sticky sessions | Shared store | Latency vs reliability |
| Transport | Cookie | Auth header | Browser apps vs mobile/API |
| Concurrency | Limited | Unlimited | Security vs UX |
| Persistence | None (Redis) | Full (DB) | Speed vs durability |

Every tradeoff exists on a spectrum. The right choice depends on your application's requirements for security, user experience, operational complexity, and scalability. Document your session architecture decisions and revisit them as your application grows.
