## Session Design Interview Questions

Session management is a frequent topic in system design interviews. These questions test your understanding of stateful vs stateless design, distributed systems, and security.

---

## Question 1: Design a Session Management System for an E-Commerce Platform

**The problem:** Design a session management system for an e-commerce platform with 10 million daily active users. The platform needs to track user sessions, shopping carts, and browsing history.

### 1. Requirements Clarification

| Aspect | Question | Answer |
|---|---|---|
| Scale | Daily active users | 10 million |
| Scale | Concurrent sessions | 100,000 peak |
| Latency | Session read p99 | under 50 ms |
| Availability | Session store uptime | 99.99% |
| Security | Session lifetime | 24 hours max, 1 hour idle |
| Data | Session data size | under 1 KB per session |
| Regions | Deployment | Single region (US-East) |
| State | Shopping cart | Stored in session? |

**Clarify the cart question:** The cart should be stored separately from the session. Cart data persists beyond the session lifetime (e.g., user adds items, closes browser, comes back next day). The session stores only a cart reference ID.

### 2. Core Design Decisions

**Decision 1: Server-side sessions with Redis.** The platform needs server-side revocation (for password resets, account suspension). JWT-based sessions would not provide this.

**Decision 2: Minimal session data.** Store only user_id, role, and cart_id in the session. Cart contents and browsing history are stored separately.

**Decision 3: Redis Cluster for high availability.** A single Redis instance is a single point of failure. Redis Cluster with 3 master nodes and 3 replicas provides the required throughput and availability.

**Decision 4: Sliding + absolute expiry.** 1-hour sliding TTL for user experience, 24-hour absolute TTL for security.

### 4. Session Data Model

```python
# Redis session data (~400 bytes)
session_key = "session:abc123def456"
session_value = {
    "user_id": 42,
    "role": "customer",
    "cart_id": "cart_abc123",
    "created_at": 1720000000,
    "last_access": 1720000050,
    "expires_at": 1720003650  # created_at + 3600
}
# Redis TTL: 3600 (sliding, reset on each access)
# Absolute expiry enforced by examining expires_at
```

### 5. API Design

```
POST   /api/session          # Create session (login)
GET    /api/session          # Get current session
DELETE /api/session          # Delete session (logout)
POST   /api/session/refresh  # Extend session TTL
```

### 6. Failure Scenarios

| Failure | Impact | Mitigation |
|---|---|---|
| Redis master fails | Read-only until replica promoted | Redis Sentinel or Cluster |
| Redis cluster partition | Minority partition unavailable | Majority keeps operating |
| App server fails | No impact (sessions in Redis) | Auto-scaling replaces server |
| DynamoDB unavailable | Cart unavailable, session intact | Show empty cart, block checkout |

### 7. Scalability Limits

| Component | Limit | Next Step |
|---|---|---|
| Redis single node | ~25,000 ops/sec | Add more shards (Redis Cluster) |
| Redis Cluster (3 nodes) | ~75,000 ops/sec | Add more shards |
| App server | Limited by CPU/memory | Auto-scale horizontally |
| DynamoDB cart | Unlimited (auto-scaling) | No upgrade needed |

## Question 2: How Would You Handle Session Revocation with JWT?

**The problem:** Your team is using JWT for session management, but the security team requires the ability to force-logout specific users (e.g., when a password is changed or an account is suspended).

### The Tradeoff

JWT is stateless by design. The server cannot revoke a validly signed token. The token remains valid until its `exp` claim is reached.

### Solution 1: Short-Lived Tokens + Refresh Tokens

Access tokens live for 15 minutes. Refresh tokens live for 30 days and are stored server-side. Revocation means deleting the refresh token.

```
Access token (15 min):  Cannot be revoked. Risk window: 15 minutes.
Refresh token (30 days): Stored in DB. Can be revoked immediately.
```

**Pros:** Minimal infrastructure. Most tokens are not revocable, but the window is short.
**Cons:** 15-minute window of vulnerability. If all tokens must be revoked instantly, this does not work.

### Solution 2: Token Blocklist

Maintain a Redis set of revoked JWT IDs (jti). Check the blocklist on every request.

```python
def check_blocklist(jti):
    """Check if a JWT has been revoked."""
    return redis.sismember("jwt_blocklist", jti)

def revoke_token(jti):
    """Add a JWT to the blocklist until its original expiry."""
    payload = jwt.decode(jti, ...)
    ttl = payload["exp"] - int(time.time())
    if ttl > 0:
        redis.sadd("jwt_blocklist", jti)
        redis.expire("jwt_blocklist", ttl)  # Auto-remove after expiry
```

**Pros:** Immediate revocation. Works for any number of tokens.
**Cons:** Adds a Redis lookup on every request (losing the stateless benefit). Blocklist size grows with revoked tokens.

### Solution 3: Token Version

Include a version number in the JWT. Store the current version per user in a database. On revocation, increment the version.

```python
# JWT payload includes version
token = {
    "user_id": 42,
    "token_version": 5,
    "exp": ...
}

# Database stores current version per user
redis.set(f"token_version:user_42", 5)

def validate_token(token):
    payload = jwt.decode(token, ...)
    current_version = redis.get(f"token_version:user_{payload['user_id']}")
    if payload["token_version"] != current_version:
        return None  # Token is stale
    return payload
```

**Pros:** Single lookup per user (not per token). Blocklist size is bounded by user count.
**Cons:** Requires a database lookup. Version collisions possible if not handled carefully.

### Recommendation

Use short-lived access tokens (15 minutes) with refresh tokens stored in a database. For most applications, the 15-minute revocation window is acceptable. If immediate revocation is required, add a token version check.

## Question 3: What Happens to Sessions When You Deploy New Code?

**The problem:** Every deployment restarts the application servers. How do you prevent 100,000 users from being logged out?

### The Naive Approach (Fails)

In-memory sessions: all sessions are lost on server restart. Users are forced to log in again.

### The Working Approaches

**Approach 1: External session store (Redis).** Sessions live in Redis, not in the application server. Rolling deployment restarts servers one at a time. Sessions are unaffected.

```yaml
# Deployment: rolling update, 1 server at a time
# Session store: Redis (external, not affected by deploy)
# Result: Zero session loss
```

**Approach 2: Session persistence.** If you must use in-memory sessions, serialize the session store to disk and reload on restart. This adds startup time and complexity.

**Approach 3: Graceful degradation.** If sessions are lost, redirect users to the login page with a message. This is acceptable for low-traffic internal tools but not for consumer applications.

### Recommendation

Use an external session store. This is the single most important architectural decision for session reliability.

## Question 4: How Would You Debug a Session Store Performance Issue?

**The problem:** The session store (Redis) is responding slowly. Session reads are taking 50-100 ms instead of the usual 1-2 ms. Users are experiencing slow page loads.

### Debugging Steps

**Step 1: Check Redis latency.**

```bash
redis-cli --latency -h redis.example.com -p 6379
# Shows min/avg/max latency in milliseconds
```

**Step 2: Check slow logs.**

```bash
redis-cli SLOWLOG GET 10
# Shows the 10 slowest commands
```

**Step 3: Check memory usage.**

```bash
redis-cli INFO memory
# Check used_memory, maxmemory, evicted_keys
```

**Step 4: Check command statistics.**

```bash
redis-cli INFO commandstats
# Shows which commands are consuming the most time
```

### Common Causes

| Cause | Symptom | Fix |
|---|---|---|
| High memory usage | Evictions increasing | Add more RAM, reduce TTL, add shards |
| Large session values | SET/GET commands slow | Reduce session data size |
| Network latency | All commands slow | Move Redis closer to app servers |
| Fork of Redis (save) | Pause during BGSAVE | Disable save, use AOF |
| Hot key | One session key accessed very frequently | Shard or cache locally |

## Question 5: Design a Multi-Tenant Session System

**The problem:** You are building a SaaS platform where each customer (tenant) has their own users. Sessions must be isolated per tenant.

### Approach

Use a composite key: `session:<tenant_id>:<session_id>`.

```python
def get_session_key(tenant_id, session_id):
    return f"session:{tenant_id}:{session_id}"

# Redis keys
# session:acme_corp:abc123  ──> {user_id: 42, ...}
# session:megacorp:def456   ──> {user_id: 99, ...}
```

### Tenant Isolation

```python
def validate_session(tenant_id, session_id):
    key = f"session:{tenant_id}:{session_id}"
    data = redis.get(key)
    if not data:
        return None
    session = json.loads(data)
    # Verify tenant matches (double-check)
    if session["tenant_id"] != tenant_id:
        return None
    return session
```

### Key Design Decisions

- **Tenant prefix**: Prevents session ID collisions between tenants.
- **Tenant validation**: Double-check that the tenant_id in the session data matches the tenant_id in the key. This prevents a session from tenant A being used to access tenant B's data.
- **Per-tenant Redis instance**: For high-security tenants, use a dedicated Redis instance. For most tenants, a shared Redis cluster with tenant-prefixed keys is sufficient.

## Question 6: Compare Cookie-Based Sessions, Token-Based Sessions, and Server-Side Sessions

| Aspect | Cookie-Based | Token-Based (JWT) | Server-Side (Redis) |
|---|---|---|---|
| Where state lives | Cookie (client) | Token (client) | Server (Redis) |
| Revocable | No | No (without blocklist) | Yes |
| Scalability | Trivial | Trivial | Requires session store |
| Latency | None | None | 1-5 ms per lookup |
| Security | CSRF risk | XSS risk (if in localStorage) | Most secure |
| Data size | 4 KB max | 4 KB max (cookie) | Unlimited |
| Infrastructure | None | None | Redis cluster |
| Example | PHP sessions | Auth0 | Django sessions with Redis |

**Key insight:** Cookie-based sessions and token-based sessions are both client-side state. The difference is that JWT is signed and structured, while a cookie-based session ID is an opaque reference to server-side state. The two are often confused: a session ID cookie is not a cookie-based session — it is a server-side session with a cookie transport.

## Question 7: How Would You Handle 1 Million Concurrent Sessions?

**The problem:** The application is growing rapidly. You need to handle 1 million concurrent sessions.

### Step 1: Estimate Storage

Assuming 1 KB per session:
- 1 million sessions = 1 GB of session data
- Redis instance with 4 GB RAM can handle this comfortably
- With replication (1 master + 2 replicas): 3 GB total

### Step 2: Estimate Throughput

Assuming 100 requests per second per session:
- 1 million sessions x 100 requests = 100 million requests/second
- This is unrealistic. More realistic: 10 requests per session per minute = 166,000 req/s
- Redis single node: ~50,000-100,000 ops/sec
- Redis Cluster with 10 nodes: ~500,000-1,000,000 ops/sec

### Step 3: Architecture

1. **Redis Cluster with 10 shards** (10 master nodes, 10 replicas)
2. **Session data minimized** to 200 bytes per session
3. **TTL of 1 hour** (sliding) to automatically clean up stale sessions
4. **Circuit breaker**: If Redis is slow, serve stale session data from a local cache
5. **Monitoring**: Track session store latency, memory usage, and eviction rate

### Step 4: Bottleneck Analysis

| Bottleneck | Limit | Mitigation |
|---|---|---|
| Redis CPU | ~100K ops/sec per node | 10 shards = 1M ops/sec |
| Redis memory | 4 GB per node | 1M x 200 bytes = 200 MB, well within limits |
| Network bandwidth | 1 Gbps per node | 200 bytes x 100K ops = 20 MB/s, well within limits |
| App server connections | 1000 per server | 100 app servers x 1000 = 100K connections |
