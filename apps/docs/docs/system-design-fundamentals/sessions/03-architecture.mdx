## Architecture Evolution

Session architecture evolves through stages as applications grow. Each stage introduces new capabilities and tradeoffs.

---

## Stage 1: Naive Architecture — Single Server with In-Memory Sessions

The simplest approach: a single application server stores sessions in memory. The session data is a hash map or dictionary in the application process.

  │  │  │  session:ghi ──> {}  │   │  │

**Pros:** Simplest possible architecture. Sub-millisecond session lookups. No network calls for session data. No additional infrastructure.

**Cons:** Sessions lost on server restart. Cannot scale to multiple servers. Memory limited to one machine's RAM. Not suitable for production beyond single-server prototypes.

**Breaks at:** ~10,000 concurrent sessions (memory), a server restart (all sessions lost), or when you add a second server.

## Stage 2: Load-Balanced Servers with Sticky Sessions

When you add a second server, sessions must be routed to the server that created them. The load balancer uses a cookie or IP hash to route all requests from the same client to the same server.

<AsciiDiagram
  id="sessions/arch-sticky-sessions"
  title="Load-balanced servers with sticky sessions"
  alt="Load balancer routing clients to servers based on session affinity, each server maintaining its own local session store"
  caption="Sticky sessions route each client to the same server, where their session data lives locally"
  content={`Sticky sessions with load balancer

                     ┌──────────────────┐
                     │   Load Balancer  │
                     │                  │
                     │  ┌────────────┐  │
                     │  │  Sticky    │  │
                     │  │  Session   │  │
                     │  │  Table     │  │
                     │  │  A ──> S1  │  │
                     │  │  B ──> S2  │  │
                     │  │  C ──> S3  │  │
                     │  └────────────┘  │
                     └──────────────────┘
                          │     │     │
                  ┌───────┘     │     └───────┐
                  ▼             ▼             ▼
          ┌────────────┐ ┌────────────┐ ┌────────────┐
          │  Server 1  │ │  Server 2  │ │  Server 3  │
          │            │ │            │ │            │
          │ In-Memory  │ │ In-Memory  │ │ In-Memory  │
          │ Sessions   │ │ Sessions   │ │ Sessions   │
          │            │ │            │ │            │
          │ A, D, G    │ │ B, E, H    │ │ C, F, I    │
          └────────────┘ └────────────┘ └────────────┘
`}
/>

**Pros:** Simple upgrade from single server. No external session store needed. Fast local memory access. No new infrastructure.

**Cons:** Server failure loses all sessions on that server. Uneven load if session distribution is skewed. Rolling deployments cause mass session loss. Scaling up/down requires session redistribution.

**Breaks at:** Any server failure (loss of sessions), ~20%+ traffic spikes (uneven distribution), or frequent deployments.

### Sticky Session Implementation

```nginx
# nginx configuration for sticky sessions
upstream app_servers {
    hash $cookie_session_id consistent;
    server app1.example.com;
    server app2.example.com;
    server app3.example.com;
}
```

The `consistent` parameter enables consistent hashing: when a server is added or removed, only 1/N of sessions need to be redistributed rather than all sessions.

## Stage 3: External Session Store (Redis)

The session store moves out of the application server into a dedicated Redis instance. Any server can handle any request.

**Pros:** Any server can serve any request. Sessions survive server restarts. Servers can be added and removed freely. No session affinity needed in the load balancer.

**Cons:** Redis is a dependency and a potential single point of failure. Network round trip adds 1-3 ms to each request. Redis throughput limits total session operations.

**Breaks at:** Redis instance failure (mitigated with replication), or ~100,000+ session operations per second (mitigated with Redis Cluster).

### Implementation with Flask

```python
from flask import Flask, session
from redis import Redis
from flask_session import Session

app = Flask(__name__)
app.config['SESSION_TYPE'] = 'redis'
app.config['SESSION_REDIS'] = Redis(host='redis.example.com', port=6379)
app.config['SESSION_PERMANENT'] = True
app.config['PERMANENT_SESSION_LIFETIME'] = 3600  # 1 hour

Session(app)

@app.route('/login', methods=['POST'])
def login():
    session['user_id'] = 42
    session['role'] = 'admin'
    return 'Logged in'
```

### Implementation with Django

```python
# settings.py
SESSION_ENGINE = 'django.contrib.sessions.backends.cache'
SESSION_CACHE_ALIAS = 'session'

CACHES = {
    'session': {
        'BACKEND': 'django.core.cache.backends.redis.RedisCache',
        'LOCATION': 'redis://redis.example.com:6379/1',
        'OPTIONS': {
            'CLIENT_CLASS': 'django_redis.client.DefaultClient',
        }
    }
}
```

## Stage 4: Redis Cluster with Replication

A single Redis instance is a single point of failure. Redis Sentinel provides automatic failover: if the master fails, a replica is promoted to master.

For larger scale (100,000+ concurrent sessions), Redis Cluster provides automatic sharding across multiple nodes. Each node owns a subset of keys (hash slots), and the cluster handles routing transparently.

```redis
# Redis Cluster: 6 nodes, 3 masters, 3 replicas
# Hash slots automatically distributed across masters
# Each master has one replica for failover
redis-cli --cluster create \
    10.0.0.1:6379 10.0.0.2:6379 10.0.0.3:6379 \
    10.0.0.4:6379 10.0.0.5:6379 10.0.0.6:6379 \
    --cluster-replicas 1
```

## Stage 5: Multi-Region Session Store

For global applications, sessions must be available across geographic regions. A single-region session store adds 100-300 ms of latency for users in distant regions.

Three approaches for multi-region sessions:

### Approach A: Token-Based (No Cross-Region Store)

Use JWTs that self-contain all session data. No session store needed across regions. Each region independently validates the token.

**Good for:** Stateless session data. Short-lived sessions. Applications where server-side revocability is not required.

**Tradeoff:** Cannot revoke tokens server-side. Token size grows with session data.

### Approach B: Global Session Store (DynamoDB Global Tables)

A single multi-region database handles session storage. DynamoDB Global Tables replicate writes across regions with sub-second latency.

```python
import boto3
from boto3.dynamodb.conditions import Key

dynamodb = boto3.resource('dynamodb', region_name='us-east-1')
table = dynamodb.Table('Sessions')

def get_session(session_id):
    response = table.get_item(
        Key={'session_id': session_id},
        ConsistentRead=False  # Eventual consistency for lower latency
    )
    return response.get('Item')

def create_session(session_id, data, ttl_seconds=3600):
    table.put_item(
        Item={
            'session_id': session_id,
            'data': data,
            'ttl': int(time.time()) + ttl_seconds,
            'created_at': int(time.time())
        }
    )
```

DynamoDB's cross-region replication adds 1-3 seconds of latency in the worst case. This means a session created in US-East may not be available in AP-East for a few seconds. Applications must tolerate this window.

### Approach C: Active-Passive with DNS Routing

One region is primary for sessions. All writes go to the primary region. Other regions cache session data locally and fall back to the primary if the cache misses.

```python
def get_session_multiregion(session_id):
    # Try local Redis first (1-5ms)
    local_data = local_redis.get(f"session:{session_id}")
    if local_data:
        return json.loads(local_data)

    # Fall back to primary region Redis (100-300ms)
    primary_data = primary_redis.get(f"session:{session_id}")
    if primary_data:
        # Cache locally for future requests
        local_redis.setex(f"session:{session_id}", 300, primary_data)  # 5-min local cache
        return json.loads(primary_data)

    return None  # Session not found
```

**Good for:** Read-heavy session workloads. Applications that can tolerate 300 ms failover latency. Multi-region deployments with a home region for each user.

## Stage 6: Token-Based Stateless Sessions (JWT)

Eliminate the session store entirely by encoding session data into a signed token.

<AsciiDiagram
  id="sessions/arch-jwt"
  title="JWT-based stateless session architecture"
  alt="Client stores a JWT token. Server validates the token signature without any session store lookup. No shared session store needed between servers."
  caption="JWT sessions eliminate the session store: the server validates the token signature and reads session data directly from the token"
  content={`JWT-based stateless sessions

  ┌──────────────┐     ┌─────────────────────────────────────┐
  │   Client     │     │         Application Server           │
  │              │     │                                      │
  │  ┌────────┐  │     │  ┌──────────────┐                    │
  │  │ JWT    │  │────>│  │  Validate    │                    │
  │  │ Token  │  │     │  │  Signature   │                    │
  │  │        │  │     │  │  └─────────┘ │                    │
  │  │ Header │  │     │  │              │                    │
  │  │ Payload│  │     │  │  Read        │                    │
  │  │ Sig    │  │     │  │  user_id: 42 │                    │
  │  └────────┘  │     │  │  role: admin │                    │
  │              │     │  └──────────────┘                    │
  │  Stored in   │     │                                      │
  │  cookie or   │     │  NO session store lookup needed!     │
  │  localStorage│     │  NO shared infrastructure required!  │
  └──────────────┘     └─────────────────────────────────────┘
`}
/>

```python
import jwt
from datetime import datetime, timedelta

SECRET_KEY = "your-256-bit-secret"

def create_session_token(user_id, role):
    payload = {
        "user_id": user_id,
        "role": role,
        "iat": datetime.utcnow(),
        "exp": datetime.utcnow() + timedelta(hours=1)
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

def validate_session_token(token):
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
        return payload  # Contains user_id, role, etc.
    except jwt.ExpiredSignatureError:
        return None  # Token expired
    except jwt.InvalidTokenError:
        return None  # Token tampered or invalid
```

**Pros:** No session store infrastructure. Zero session store latency. Trivially horizontally scalable (any server can validate any token). No session affinity needed. Works across regions without cross-region replication.

**Cons:** Cannot revoke tokens server-side (until they expire). Token size limits (4 KB in cookies). Token data is visible (base64 decoded) — never store secrets in the token. Token replay is possible until expiry (mitigated with short TTL + refresh tokens).

## Architecture Decision Guide

| Scenario | Architecture | Why |
|---|---|---|
| Single server, < 10K users | In-memory sessions | Simplest. No infra needed. |
| Multiple servers, same DC | Redis session store | Any server, any request. Survives server restarts. |
| Multiple servers, HA required | Redis + Sentinel | Automatic failover. No SPOF. |
| > 100K concurrent sessions | Redis Cluster | Horizontal scaling. Automatic sharding. |
| Multi-region, read-heavy | Active-passive Redis | Local reads. Primary region writes. |
| Multi-region, write-heavy | Token-based (JWT) | No cross-region latency. No shared store. |
| Serverless (AWS Lambda) | Token-based (JWT) | Stateless by nature. No persistent store needed. |
| Microservices, internal | Token-based (JWT) | Pass session context between services without shared store lookup. |

Each architecture represents a tradeoff between simplicity, scalability, availability, and operational complexity. Choose the simplest architecture that meets your current requirements, but design the session interface so you can upgrade to a more sophisticated architecture without changing the application code.
