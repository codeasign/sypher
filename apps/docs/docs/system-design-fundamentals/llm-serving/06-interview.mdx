## How This Topic Appears

LLM serving questions appear in FAANG interviews in four forms. Here are the most common phrasings:

### Direct Questions
- "How does an LLM serving system work?"
- "Explain the difference between prefill and decode in LLM inference."
- "What is the KV cache and why does it matter?"
- "How does continuous batching work?"

### Design Questions
- "Design a system to serve a large language model at scale."
- "Design a low-latency LLM inference API for a chat application."
- "Design a serving infrastructure for a custom fine-tuned model."
- "Design a multi-model serving platform that supports 10 different models."
- "How would you serve Llama 3 70B to 1 million users?"

### Debugging Questions
- "Our LLM API has high latency. Walk me through debugging."
- "The GPU keeps running out of memory during inference. What do you check?"
- "The model generates good responses but takes 30 seconds to start. How do you fix it?"

### Trade-off Questions
- "Why would you choose vLLM over TensorRT-LLM?"
- "What are the trade-offs between GPU and TPU for inference?"
- "How do you decide between self-hosting and using a managed API?"

## What the Interviewer Is Actually Testing

### For Direct Questions
**What they are testing:** Do you understand the fundamentals? Can you explain the inference pipeline clearly? The interviewer wants to see that you know the basic components: tokenization, prefill, KV cache, decode, streaming. A 3/4 answer lists these components. A 4/4 answer explains why each component exists and what breaks without it.

### For Design Questions
**What they are testing:** Can you design a system under real hardware constraints? The interviewer evaluates your ability to account for GPU memory limits, latency requirements, and cost. The signals they look for: you mention KV cache memory before being asked, you discuss batching strategies, you account for cold-start latency, you provide concrete GPU requirements with math.

### For Debugging Questions
**What they are testing:** Can you debug a GPU-bound system systematically? The interviewer wants to see a structured approach: start with the symptom (high latency, OOM), identify the layer (scheduler, GPU, memory), check the metrics, find the root cause, and propose a fix. A weak answer jumps to "buy more GPUs." A strong answer checks queue depth, KV cache utilization, and GPU compute utilization.

### For Trade-off Questions
**What they are testing:** Can you think in trade-offs under real resource constraints? The interviewer wants to see that you understand that every decision has a hardware cost. The signal is: you mention both the benefits and the drawbacks of your choice, and you explain the conditions under which the other choice would be better.

## Level-by-Level Expectations

### L3/L4 — Junior / New Grad

**Minimum expected knowledge:**
- Know what an LLM is (trained model that generates text)
- Know that inference requires a GPU
- Can explain the basic flow: prompt → tokenize → model → generate → detokenize
- Understands that larger models need more GPU memory

**Vocabulary they must know:** Token, prompt, inference, GPU, model weights, temperature, tokenizer.

**What a passing answer looks like:**

Interviewer: "How does an LLM serve a response?"

Candidate: "The user sends a prompt. The prompt is tokenized — split into tokens that the model understands. The tokens go into the GPU, where the model weights are loaded. The model processes the tokens and generates new ones one at a time. Each generated token is appended to the input and fed back into the model. This continues until the model outputs a special end-of-sequence token or reaches the maximum length. The tokens are converted back to text and sent to the user."

**What distinguishes a 3/4 from a 4/4:** The 4/4 candidate mentions the KV cache (the model does not need to recompute attention for previously generated tokens) and streaming (tokens can be sent one at a time to reduce perceived latency).

### L5 — Mid-level

**Expected depth of understanding:**
- Can design an LLM serving system for moderate scale (100K users/day)
- Understands GPU memory constraints — knows how to estimate memory for a model
- Can explain the KV cache, continuous batching, and model parallelism
- Knows the difference between tensor and pipeline parallelism
- Can discuss response caching and streaming

**Example question + model answer:**

Interviewer: "Design a serving system for a 70B parameter model that handles 100K requests per day."

Candidate: "I need to clarify the requirements. How many tokens per request? What is the latency SLA? Let me assume 1,000 prompt tokens and 500 response tokens per request, with under 2 seconds for time to first token and under 10 seconds total.

A 70B model in FP16 requires 140 GB of memory. The largest single GPU (H100) has 80 GB, so I need at least 2 GPUs with tensor parallelism. Each GPU holds 70 GB of weights, leaving 10 GB for KV cache and activations. That is tight.

I will use 4 GPUs with tensor parallelism. Each GPU holds 35 GB of weights, leaving 45 GB for KV cache. With each sequence consuming roughly 12 GB of KV cache for 2,048 tokens, I can handle about 3 concurrent requests per GPU group.

At 100K requests per day, peak is roughly 10 requests per second. Average response time is 10 seconds, so I have 100 concurrent requests in flight. I need 100 / 3 = 34 GPU groups = 136 GPUs. That is roughly 17 instances with 8 GPUs each.

I will use vLLM with PagedAttention for efficient KV cache management. Continuous batching maximizes GPU utilization. A load balancer distributes requests across instances. I will cache frequently asked questions in Redis — a 10% cache hit rate saves 10% of GPU compute."

### L6 — Senior

**Expected depth of understanding:**
- Proactively discusses memory management, scheduling, and cost optimization
- Quantifies everything — GPU memory, tokens per second, cost per token
- Discusses operational complexity: model deployment, monitoring, upgrades
- Considers multi-model serving and model routing
- Addresses the business impact of serving decisions

**Example question + model answer:**

Interviewer: "Design an LLM serving platform for a company with 10 million users that offers both free and paid tiers."

Candidate: "Let me start by estimating scale. With 10 million users, roughly 20% are daily active — 2 million DAU. Each user sends 20 messages per day on average. That is 40 million requests per day. At peak (5x average), 2,315 requests per second. Average 500 prompt tokens and 300 response tokens. Peak tokens per second: 2,315 × 800 = 1.85 million.

I will use a three-tier model architecture. Free tier users get a 7B model (fits on a single H100, 14 GB weights, 50 concurrent sequences). Paid tier users get a 70B model (4 GPUs with tensor parallelism). Premium tier users get a dedicated 70B instance with priority scheduling.

The serving system uses model routing. A classifier model (under 1B parameters, runs on CPU) determines whether the request is simple. Simple requests (80% of traffic) go to the 7B model at $0.0001 per request. Complex requests (20%) go to the 70B model at $0.001 per request. This reduces average cost per request by 60%.

For the 7B tier: 70B model needs ~2,000 tokens/sec/GPU. Peak decode: 2,315 × 0.8 × 300 = 555,600 tok/s. GPUs needed: 555,600 / 2,000 = 278. With 30% headroom: 362 GPUs = 46 instances (8×H100).

For the 70B tier: Peak decode: 2,315 × 0.2 × 300 = 138,900 tok/s. GPUs needed: 138,900 / 500 (70B throughput) = 278. With tensor parallelism (4 GPUs per group): 278 / (500/4) = 70 GPU groups = 280 GPUs = 35 instances.

Total GPU instances: 46 + 35 = 81. Monthly cost: 81 × $30,000 = $2.43 million. Revenue from paid tier ($20/user/month × 1 million paid users = $20 million/month). GPU cost is 12% of revenue — acceptable.

For monitoring, I track time to first token (alert if p99 exceeds 2 seconds), tokens per second per GPU (alert if drops below 50% of baseline), GPU memory utilization (alert if exceeds 95%), queue depth (alert if exceeds 5,000), and cost per request by tier.

The biggest operational risk is model quality regression. I deploy a quality monitoring system that samples 1% of responses and scores them using an LLM-as-judge. If quality drops below baseline, the deployment is automatically rolled back."

### L7/L8 — Staff / Principal

**Expected depth of understanding:**
- Discusses org-level impact — how does this serving platform affect the engineering organization
- Connects technical decisions to business outcomes (cost per token, user retention)
- Addresses operational complexity: who manages GPUs, how to onboard new models
- Discusses 3-5 year evolution: model size trends, hardware trends, cost trends
- Considers multi-region, multi-cloud, and data residency

**Example question + model answer:**

Interviewer: "Design a global LLM serving platform that serves multiple product teams across the company. The platform must support 50+ models, handle 1 billion tokens per day, and reduce costs by 30% year over year."

Candidate: "This is not just a technical system — it is an organizational platform. The design must account for how different product teams will use it, how we manage model versions, and how we optimize costs across the fleet.

The platform provides a unified API that all product teams call. The API is OpenAI-compatible (a REST endpoint with messages array). This ensures that any tool that works with OpenAI's API works with our platform. The platform handles model routing, load balancing, and monitoring.

Each product team gets a namespace. The namespace configures: which models they can access, their rate limits (requests per minute, tokens per day), their routing rules (which requests go to which model), and their cost allocation (chargeback to the team's budget). The namespace configuration is self-service through a web portal.

The platform manages a model registry. Each model has a version, a hardware requirement (GPU count and type), a status (active, deprecated, retired), and a deployment strategy (blue-green, canary). Models are deployed through a CI/CD pipeline that validates performance and quality before promotion to production.

For cost optimization, the platform uses three strategies. First, model routing: a classifier routes simple requests to small models. Second, batch scheduling: batch workloads (offline processing) run during off-peak hours when GPU demand is lower. Third, spot instances: non-critical models run on spot GPU instances that can be preempted.

For the 5-year evolution, I anticipate three trends. Model sizes will grow (from 70B to 1T+), but quantization will keep them deployable (4-bit quantization reduces memory by 4x). Specialized hardware (TPU, LPU, Cerebras) will compete with GPUs, reducing cost per token by 10x over 5 years. Open source serving frameworks will commoditize — the differentiation will be in cost optimization and reliability, not in serving software.

The platform team tracks cost per token by model, by team, and by use case. The target is 30% cost reduction per year through hardware improvements, quantization, and model optimization. The cost per token for our platform is $0.00005 for the 7B model and $0.0005 for the 70B model — 50% cheaper than managed APIs."

## The FAANG Framework for Answering

### Step 1: Clarify Requirements

**What to ask:**
- How many users? What is the DAU/MAU ratio?
- What model sizes? (7B, 13B, 70B, 405B)
- What is the latency SLA? (TTFT under 500ms, total under 10s)
- What is the throughput requirement? (queries per second, tokens per day)
- Is multi-region required? (global user base, data residency)
- What is the budget? (GPU cost per month)

**What NOT to ask:**
- "What programming language should I use?" (irrelevant at the architecture level)
- "Should I use PyTorch or TensorFlow?" (the serving framework abstracts this)
- "Can you tell me the exact requirements?" (you should define reasonable assumptions)

### Step 2: Estimate Scale

**Back-of-envelope calculations for LLM serving:**
- DAU to requests: 1 DAU = 10-50 requests per day
- Requests per second: DAU × requests_per_day / 86400 × peak_factor (5x)
- Tokens per second: req/s × (avg_prompt_tokens + avg_response_tokens)
- GPU memory per request: KV cache = 2 bytes × layers × hidden_dim × seq_len / GPUs
- GPUs needed: total_tokens_per_second / tokens_per_second_per_gpu
- Monthly cost: GPUs × $30,000 (8×H100 instance) + networking + storage

### Step 3: High-Level Design

**What to draw first:**
1. Clients (left side) sending requests
2. Load balancer / API gateway
3. Inference cluster (GPU instances with vLLM)
4. Model weight storage (S3 for loading)
5. KV cache (in GPU memory, managed by vLLM)
6. Response streaming (back to clients)
7. Optional: Redis cache, monitoring stack

### Step 4: Deep Dive

**What to expand on:**
- KV cache: paged memory management, why it is the bottleneck
- Continuous batching: how the scheduler manages waiting/running/done queues
- Model parallelism: tensor vs pipeline, when to use each
- Quantization: FP16 vs FP8 vs INT4, accuracy vs memory trade-off
- Response caching: exact match and semantic similarity

### Step 5: Trade-offs

**What to volunteer:**
- "The trade-off between latency and throughput is the central tension..."
- "I chose continuous batching over static batching because..."
- "The KV cache trades memory for compute — without it, each token would recompute attention over the entire history..."
- "Self-hosting is cheaper at high volume, but managed APIs are simpler for low volume..."

### Step 6: Wrap Up

**How to close confidently:**
- "This design handles 100K requests per day with a 70B model. To handle 10M, I would add multi-region deployment and model routing..."
- "The most important monitoring metric is GPU memory utilization — when it approaches 95%, the system is at risk of OOM."
- "If I had more time, I would explore speculative decoding for 2-3x throughput improvement..."
- "I believe this design meets the requirements because it addresses the primary bottleneck (GPU memory) with paged KV cache..."

## Phrase Bank

### Framing the Trade-off
- "The trade-off here is between latency and throughput. For a chat application, I optimize for latency even if it means lower GPU utilization — users notice the first token delay."
- "One thing interviewers often want to see is whether you account for GPU memory. The KV cache grows with sequence length and batch size — this is almost always the bottleneck before compute."
- "The failure mode I would watch for here is GPU OOM. When memory fills up, the inference process crashes and all in-flight requests are lost."

### Demonstrating Depth
- "A 70B model in FP16 requires 140 GB of memory. The largest single GPU is 80 GB, so I need at least 2 GPUs with tensor parallelism. Each GPU holds 70 GB of weights."
- "The KV cache for a single sequence with 2,048 tokens and 80 layers requires roughly 2 KB per token per layer × 80 layers × 2,048 tokens = 320 MB per sequence."
- "Continuous batching improves GPU utilization from 30% to 90% by dynamically adding new requests to the batch as existing requests finish."
- "PagedAttention reduces KV cache fragmentation from 50% to under 5% by allocating fixed-size blocks instead of contiguous memory."

### Handling Follow-Up Questions
- "If the interviewer asks about cost, I would calculate: at 1 million requests per day with 500 tokens each, the GPU compute cost is roughly $0.0002 per request for a 7B model and $0.002 per request for a 70B model."
- "If the interviewer asks about scaling, I would explain that multi-region deployment with regional GPU clusters reduces latency by 80% for global users."
- "If the interviewer asks about alternative architectures, I would compare GPU serving with TPU serving: TPUs provide better performance per watt but require model adaptation to JAX or TensorFlow."

### Closing Strong
- "The key insight is that LLM serving is a memory-bound problem, not a compute-bound problem. Every optimization — paged KV cache, continuous batching, quantization — is ultimately about using GPU memory more efficiently."
- "The most important lesson from production experience is that GPU memory utilization determines everything: batch size, throughput, latency, and cost. Optimize memory first, compute second."

## Red Flags

### Red Flag 1: No Mention of KV Cache
**What the candidate says:** "The model processes the prompt and generates tokens one at a time."
**Why it signals a gap:** The candidate does not understand the fundamental memory bottleneck in LLM inference. Without the KV cache, every token would require O(n²) recomputation.

### Red Flag 2: No Understanding of GPU Memory Constraints
**What the candidate says:** "We will just deploy the model on a GPU."
**Why it signals a gap:** The candidate does not check whether the model fits on the GPU. A 70B model in FP16 requires 140 GB — no single GPU has that much memory. The candidate did not consider model parallelism.

### Red Flag 3: No Batching Strategy
**What the candidate says:** "Each request goes straight to the GPU."
**Why it signals a gap:** The candidate does not understand that GPUs need batch processing for efficiency. Without batching, GPU utilization is under 10%.

### Red Flag 4: Ignoring Cold-Start Latency
**What the candidate says:** "We will auto-scale based on load."
**Why it signals a gap:** The candidate does not account for model loading time (45 seconds). Auto-scaling with a 45-second cold start is useless for traffic spikes. Pre-warming or a hot pool is essential.

### Red Flag 5: No Streaming
**What the candidate says:** "The response is returned when generation is complete."
**Why it signals a gap:** The candidate does not understand perceived latency. A 10-second wait with no feedback feels like an eternity. Streaming the first token in 200ms makes the system feel fast.

### Red Flag 6: Single Model for All Requests
**What the candidate says:** "One model serves all requests."
**Why it signals a gap:** The candidate does not consider that 80% of requests are simple enough for a small model. Running all requests through a 70B model wastes 5x the cost.

### Red Flag 7: No Cost Awareness
**What the candidate says:** "We will use as many GPUs as needed."
**Why it signals a gap:** The candidate does not understand that GPUs cost $30,000/month each. A system for 1M users might need 80 GPUs — $2.4M/month. Cost optimization is a fundamental requirement.

### Red Flag 8: No Quality Monitoring
**What the candidate says:** "We monitor latency and error rates."
**Why it signals a gap:** The candidate does not consider that the model can degrade silently. Output quality must be monitored — and it is harder to measure than latency.

### Red Flag 9: Assuming Perfect Hardware
**What the candidate says:** "The GPU never fails."
**Why it signals a gap:** GPUs fail. ECC memory errors, thermal throttling, and CUDA crashes are common. The system must handle GPU failures gracefully.

### Red Flag 10: No Token Limit
**What the candidate says:** "The model supports 32K tokens, so we accept any input."
**Why it signals a gap:** The candidate does not consider that a 32K token input consumes 16x more KV cache than a 2K token input. Without per-request token limits, a single request can exhaust GPU memory.

## Practice Drills

### Drill 1: The 2-Minute Explain
**Prompt:** "Explain how a large language model serves a response as if I have never heard of it."

**What to cover:**
1. The prompt is converted to tokens (one sentence)
2. The model processes all tokens at once (the prefill phase) (one sentence)
3. The model generates one token at a time, using the KV cache to speed things up (one sentence)
4. Each token is streamed to the user as it is generated (one sentence)
5. The system checks when to stop — end-of-sequence token or max length (one sentence)
6. The tokens are converted back to text (one sentence)

### Drill 2: The 5-Minute Design
**Prompt:** "You are building an LLM-powered code completion tool. The tool needs to suggest completions within 200ms. The code context is typically 500 tokens. The completions are 20-50 tokens. Design the LLM serving architecture."

**What to cover:**
1. Model selection: 7B model is sufficient for code completion (fits on one GPU, fast)
2. Latency budget: 200ms total → 50ms prefill + 150ms decode (7 tokens at 20ms each)
3. Batching: minimal batching (latency priority), but continuous batching for multiple users
4. GPU: single H100 or A100 (7B model in FP16 = 14 GB, fits easily)
5. Caching: cache completions for common code patterns (10-20% hit rate)
6. Deployment: single region, near the user base

### Drill 3: The 10-Minute Deep Dive
**Prompt:** "Design a global LLM serving platform for a company with 100 million users. The platform must support 10+ models, handle 500 million requests per day, and provide sub-second latency. The infrastructure budget is $10 million per month."

**What to cover:**
1. Requirements clarification (model sizes, latency SLAs, regions, data residency)
2. Scale estimation (500M req/day = 5,787 req/s peak, 1.7M tokens/s)
3. GPU compute sizing (GPUs by model tier, memory per GPU, total fleet size)
4. Multi-region architecture (3 regions, regional GPU clusters, global load balancer)
5. Model routing (classifier for model selection, small model for simple requests)
6. Cost optimization (model routing saves 40%, quantization saves 50%, spot instances save 30%)
7. Monitoring and quality (TTFT, TPOT, GPU memory, output quality scores)
8. Failure scenarios (GPU OOM, network partition, model quality regression)
