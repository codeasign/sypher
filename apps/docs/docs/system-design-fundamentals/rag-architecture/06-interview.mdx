## How This Topic Appears

RAG architecture questions have become a staple of system design interviews at
FAANG and top AI companies. The questions typically fall into these categories:

**Category 1: Pure RAG Design**
1. "Design a RAG system for a company's internal knowledge base."
2. "Design a system that lets employees ask questions about company documents."
3. "Design a chatbot that answers questions about your product documentation."

**Category 2: RAG Variants**
4. "How would you build a system that answers questions with citations?"
5. "Design a system that combines real-time web search with LLM generation."
6. "Design a multi-modal RAG system that handles images and text."

**Category 3: RAG as a Component**
7. "Design a customer support system that uses RAG to answer common questions."
8. "Design a code review assistant that retrieves relevant code context."
9. "Design an enterprise search system that returns AI-generated answers."
10. "How would you add retrieval augmentation to an existing chatbot?"

**Category 4: RAG Debugging and Improvement**
11. "Your RAG system is returning wrong answers. How do you debug it?"
12. "How would you improve the retrieval quality of an existing RAG system?"
13. "How would you handle a user uploading a new document mid-conversation?"

## What the Interviewer Is Actually Testing

### For "Design a RAG system" questions

**Depth of understanding:**
- Do you understand the full pipeline (ingestion, retrieval, augmentation,
  generation)? Or do you only describe the high-level flow?
- Can you articulate the trade-offs between chunking strategies?
- Do you understand vector database internals (HNSW, IVF, ANN)?

**System design skill:**
- Can you separate offline (ingestion) and online (serving) pipelines?
- Do you consider scale in your design? (QPS, storage, latency requirements)
- Do you address failure modes and edge cases?
- Do you discuss caching, monitoring, and operations?

**Communication clarity:**
- Do you start with a high-level overview before diving into details?
- Do you use diagrams to communicate your architecture?
- Do you ask clarifying questions about requirements?

### For "Debug a RAG system" questions

**Systematic thinking:**
- Do you have a structured debugging framework?
- Can you isolate the problem to a specific pipeline stage?
- Do you know what metrics to check for each stage?

**Experience level:**
- Have you actually dealt with RAG failures in production?
- Do you know the common failure modes without being prompted?
- Can you propose concrete fixes, not just identify problems?

### For "Improve a RAG system" questions

**Optimization mindset:**
- Do you know which parameters are most impactful?
- Can you describe a measurable improvement methodology (A/B testing, offline
  evaluation, online metrics)?
- Do you understand the interaction between parameters (e.g., chunk size and
  top-K)?

## Level-by-Level Expectations

### L3 / L4 (Entry Level / Junior)

**What is expected:**
- Basic understanding of the RAG concept: retrieve documents, add to prompt,
  generate answer.
- Can describe the high-level pipeline with 3-5 steps.
- Awareness of at least one embedding model and one vector database.

**Model answer for L4:**
"RAG stands for Retrieval-Augmented Generation. It works by first taking the
user's question, converting it into a vector using an embedding model, then
searching a vector database for similar document chunks, and finally passing
those chunks along with the question to an LLM to generate a grounded answer.
The key benefit is that the LLM can answer questions about private or recent
information that was not in its training data."

**What they are looking for:**
- Can you explain RAG in plain language?
- Do you understand the basic components?
- Can you articulate the value proposition?

**Red flags:**
- Cannot explain why retrieval is separate from generation
- Thinks the LLM can handle everything without retrieval
- Confuses embedding with generation

### L5 (Senior Engineer)

**What is expected:**
- Deep understanding of the full pipeline with all stages.
- Can discuss chunking strategies, embedding model selection, and ANN algorithms.
- Considers trade-offs: chunk size, top-K, similarity threshold, re-ranking.
- Addresses failure modes: empty retrieval, irrelevant documents, hallucination.
- Discusses caching, monitoring, and operations.

**Model answer for L5:**
"For a production RAG system serving 1000 QPS with 10 million documents, I would
design two pipelines — an offline ingestion pipeline and an online serving
pipeline. The ingestion pipeline parses documents, chunks them at 512 tokens
with 50 token overlap using recursive splitting, embeds them with
text-embedding-3-small (1536 dimensions), and indexes them in a Pinecone vector
DB with HNSW (M=32, ef_construction=200). The serving pipeline encodes the query,
searches the vector DB with ef_search=200 for top-20 results, re-ranks with a
cross-encoder to select the top-5, assembles the prompt, and sends it to
GPT-4-turbo for generation. I would add a three-tier Redis cache (query,
embedding, LLM response) and monitor hit rate, relevance scores, and per-stage
latency."

**What they are looking for:**
- Concrete numbers: chunk sizes, dimensions, model names, latency expectations
- Pipeline separation: offline vs. online
- Trade-off awareness: why 512 tokens and not 256 or 1024?
- Operational awareness: what metrics matter in production?

**Red flags:**
- Vague answers: "I would use a vector database" — which one and why?
- No mention of failure modes
- No monitoring or operations discussion
- One-size-fits-all approach without trade-off consideration

### L6 (Staff Engineer)

**What is expected:**
- All L5 expectations plus:
- Multi-region, multi-model, multi-modal considerations
- Deep understanding of consistency, replication, and data residency
- Cost modeling and capacity planning
- Integration patterns with existing infrastructure
- Experience with multiple RAG implementations and their trade-offs

**Model answer for L6:**
"I would start by clarifying whether this is a greenfield or brownfield project.
If it replaces an existing search system, I would build the RAG pipeline
alongside the existing system and A/B test before migrating. For the vector
database, I would evaluate Pinecone for its managed operations versus Weaviate
for its hybrid search capabilities, depending on whether the documents contain
precise terminology requiring keyword matching. The chunking strategy would
depend on document structure — recursive splitting for general documents,
semantic chunking for legal or medical content. For a global deployment, I would
use active-passive replication with a primary in us-east-1 and read replicas in
eu-west-1 and ap-southeast-1. The embedding model would be text-embedding-3-large
for English documents and Cohere embed-multilingual-v3 for multilingual content,
with model-version tagging to handle embedding drift. Cost-wise, at 10M DAU with
5 queries per user per day, the monthly cost would be approximately $80,000 for
infrastructure and $50,000 for LLM API calls. I would target p50 latency under
500ms and p99 under 2 seconds after caching."

**What they are looking for:**
- Strategic thinking: trade-offs between build vs. buy, managed vs. self-hosted
- Cross-system integration: how does RAG fit into the existing architecture?
- Cost optimization: can you estimate and optimize costs?
- Organizational impact: how does this decision affect other teams?

**Red flags:**
- No mention of cost or business impact
- Ignores migration strategy from existing systems
- Proposes a single-region deployment for a global use case
- Does not address data residency or compliance

### L7 / L8 (Principal / Distinguished Engineer)

**What is expected:**
- All L6 expectations plus:
- Industry trends and future directions
- Novel architectures beyond standard RAG
- Influence on product and business strategy
- Mentoring and organizational change management

**Model answer for L7:**
"RAG in its current form is a transitional architecture. The long-term
trajectory is toward agents that decide when and how to retrieve information,
including tool calling, code execution, and multi-hop reasoning. However, for
today's systems, I would build a modular RAG platform where the retrieval,
augmentation, and generation components are independently deployable and
swappable. This allows us to A/B test different embedding models, chunking
strategies, and LLMs without rebuilding the entire system. The key architectural
insight is that RAG is not a single pattern but a family of patterns — naive
RAG for simple Q&A, agentic RAG for complex multi-step reasoning, and
multi-modal RAG for documents with images and tables. I would invest heavily in
evaluation infrastructure — offline metrics (hit rate, MRR, NDCG) and online
metrics (answer acceptance, user satisfaction) — because the RAG field is
evolving rapidly and we need to make data-driven decisions about upgrades."

**What they are looking for:**
- Thought leadership: do you have a point of view on where the field is going?
- Platform thinking: can you build something that serves multiple use cases?
- Organizational leverage: can you enable multiple teams to use RAG effectively?

**Red flags:**
- Treats RAG as a solved problem
- No awareness of agentic RAG, multi-modal RAG, or GraphRAG
- Focuses only on technical details without business context

## Phrase Bank

Use these phrases in your interview responses to communicate depth and
experience.

**On retrieval quality:**
- "The retrieval pipeline determines the upper bound of system quality."
- "The ANN index is an approximation. The re-ranker catches false positives."
- "Hybrid search with RRF fusion gives us both semantic and exact-match recall."

**On chunking:**
- "Chunk size is the single most impactful parameter after embedding model
  choice."
- "I prefer recursive splitting over fixed-size splitting because it respects
  document structure."
- "The chunk overlap prevents information loss at semantic boundaries."

**On embedding:**
- "The query encoder and document encoder must be the same model for the vectors
  to be in the same space."
- "I would start with text-embedding-3-small for its cost-quality ratio and
  only move to large if evaluation shows a meaningful improvement."

**On the vector database:**
- "I choose HNSW over IVF when query throughput is more important than index
  build time."
- "For managed operations, Pinecone reduces operational burden. For maximum
  control, Milvus or Weaviate self-hosted."

**On caching:**
- "The three-tier cache — query, embedding, LLM response — reduces effective
  latency by 60-70%."
- "Cache invalidation is driven by TTL, not document change events, because
  document changes are infrequent enough that TTL-based staleness is acceptable."

**On monitoring:**
- "Hit rate is the most important metric. If it drops below 90%, something is
  wrong with the retrieval pipeline."
- "Per-stage latency tracking lets us identify bottlenecks without guessing."

**On failure modes:**
- "If the answer is wrong, the first question is always: was the right context
  retrieved?"
- "A re-ranker with a relevance threshold prevents the LLM from seeing
  irrelevant documents."

**On scaling:**
- "The LLM API call is the dominant cost at scale. Caching and batching reduce
  it significantly."
- "For multi-region deployment, I use active-passive replication because RAG
  systems are read-heavy and eventual consistency is acceptable."

## Red Flags

Interviewers watch for these specific red flags when evaluating RAG system
design responses.

**Red Flag 1: No mention of chunking.** You describe a RAG system without
discussing how documents are split. This suggests you have never implemented
RAG end-to-end.

**Red Flag 2: One embedding model to rule them all.** You assume one embedding
model works for all document types. Different domains (code, medical, legal,
conversational) benefit from different embedding models.

**Red Flag 3: RAG solves everything.** You propose RAG for every problem without
considering simpler alternatives (fine-tuning, prompt engineering, traditional
search). Interviewers want to see discrimination.

**Red Flag 4: No latency numbers.** You describe components without any estimate
of their latency contribution. This suggests you have not operated a RAG system
in production.

**Red Flag 5: No failure mode discussion.** You present a RAG system that works
perfectly. Interviewers know nothing works perfectly. Discussing failure modes
shows experience.

**Red Flag 6: Vector database is magical.** You treat the vector DB as a black
box without understanding ANN algorithms (HNSW, IVF) or their parameters. You
should be able to explain how the index works.

**Red Flag 7: No cache discussion.** You propose a high-QPS RAG system without
caching. This indicates you have not considered the cost implications of LLM
API calls at scale.

**Red Flag 8: Context window as infinite.** You assume the LLM can handle any
amount of context. You must discuss token budgets, truncation, and prioritization
strategies.

**Red Flag 9: Monolithic ingestion design.** You propose a single-process
ingestion pipeline without considering message queues, batch processing, or
parallelism. This shows a lack of distributed systems thinking.

**Red Flag 10: No access control.** You design a RAG system for an enterprise
without discussing document-level access control. This is a critical requirement
for any real-world deployment.

**What to cover:**

**What to cover:**
- Structured debugging approach (2 minutes)

### Drill 6: Mock Interview Script

Practice the following script in a timed setting. This is a 45-minute system
design interview on RAG.

**0:00-3:00 — Clarify Requirements**

Ask the interviewer:
- What is the knowledge base size and type? (e.g., 100K PDF documents)
- What is the expected QPS? (e.g., 100 queries per second at peak)
- What is the latency SLO? (e.g., p50 under 1 second, p99 under 3 seconds)
- Is multi-lingual support needed? (e.g., English only)

- What is the budget? (e.g., $50K/month for infrastructure)

**3:00-8:00 — High-Level Design**

Draw the two-pipeline architecture:
- Ingestion: document parser, chunker, embedder, vector DB writer
- Serving: query encoder, retriever, re-ranker, context builder, LLM

**8:00-18:00 — Deep Dive: Ingestion**

Cover:
- Document parsing strategy (PDF extraction, OCR if needed)
- Chunking strategy (512 tokens, 50 token overlap, recursive splitting)
- Embedding model selection (text-embedding-3-small, 1536 dimensions)
- Vector DB choice (Pinecone for managed, HNSW with M=32, ef=200)
- Indexing throughput: 200 chunks/sec on GPU, 5000 inserts/sec to vector DB

**18:00-28:00 — Deep Dive: Serving**

Cover:
- Query encoding (embedding cache, batch encoding)
- Hybrid retrieval (vector search for semantics, BM25 for exact terms)
- RRF fusion (k=60, parameter-free)
- Cross-encoder re-ranking (top-20 to top-5, relevance threshold 0.5)
- Context assembly (token budget, priority ordering, citation tagging)
- LLM generation (temperature 0.1, max tokens 500, grounding prompt)

**28:00-35:00 — Scale and Optimize**

Cover:
- Three-tier cache (query, embedding, LLM response)
- Horizontal scaling for each component (embedder: 6 GPU instances)
- Multi-region considerations (active-passive, data residency)
- Cost estimation ($50K/month for infrastructure, $30K/month for LLM API)
- Capacity planning for 10M DAU

**35:00-42:00 — Failure Modes and Monitoring**

Cover:
- Empty retrieval fallback (lower threshold, keyword fallback, "I don't know")
- Irrelevant retrieval (re-ranker, metadata filtering, query rewriting)
- Hallucination (low temperature, strong system prompt, citation verification)
- Monitoring (hit rate, relevance score distribution, per-stage latency)
- Alerting (hit rate drop below 90%, p99 latency above 3 seconds)

**42:00-45:00 — Summary and Trade-offs**

Recap your design and highlight the key trade-offs:
- Re-ranking improves quality but adds 100-300ms latency
- Hybrid search improves recall but adds complexity
- Caching reduces cost but introduces staleness risk
- Per-tenant indexing simplifies access control but increases storage

## Common Mistakes in RAG Interviews

**Mistake 1: Jumping to details too early.** Start with the high-level
architecture before diving into chunk sizes and embedding dimensions. The
interviewer needs to see the big picture first.

**Mistake 2: Forgetting the ingestion pipeline.** Many candidates describe only
the query flow and forget to explain how documents get into the system. The
ingestion pipeline is half the architecture.

**Mistake 3: No concrete numbers.** Saying "I would use a small chunk size" is
vague. Saying "I would use 512 tokens with 50 tokens of overlap" shows you
have thought about the trade-offs.

**Mistake 4: Ignoring cost.** RAG systems can be expensive at scale. A candidate
who does not consider cost shows a lack of production experience. Mention
caching, tiered LLM usage, and batch processing as cost-saving measures.

**Mistake 5: Treating the vector DB as a black box.** You should be able to
explain what HNSW is, how it works, and why you chose it over IVF. Interviewers
at FAANG expect depth, not just breadth.

**Mistake 6: Not discussing failure modes.** A system design that only describes
the happy path is incomplete. Every RAG system fails occasionally. Show the
interviewer you know how to handle failures.

**Mistake 7: One-size-fits-all design.** Different use cases require different
RAG architectures. If you propose the same design for a medical Q&A system and
a code assistant, the interviewer will notice. Tailor your design to the
specific requirements.

## Follow-Up Questions to Expect

After your initial design, the interviewer will probe deeper. Prepare for these
follow-ups:

**Q: "How would you handle a user uploading a document that contradicts what the
LLM already knows?"**

A: The RAG system should prioritize the retrieved context over the LLM's
training data. The system prompt should explicitly instruct the LLM to use only
the provided context. If the document contradicts the LLM's knowledge, the LLM
should follow the document. This is a feature, not a bug — the document
represents the most current information.

**Q: "How would you measure whether the RAG system is actually better than the
LLM alone?"**

A: Run an A/B test with two groups. Group A gets the RAG-enhanced answer. Group
B gets the base LLM answer without retrieval. Measure: answer acceptance rate
(thumbs up/down), answer completeness (rated by human evaluators), and answer
accuracy (factual correctness verified against ground truth). The RAG system
should show statistically significant improvement on all three metrics.

**Q: "How would you handle a query that requires information from multiple
documents, none of which individually contain the full answer?"**

A: This is a multi-hop reasoning problem. The system needs to retrieve the first
set of documents, extract entities or concepts, formulate a second query based
on those entities, retrieve a second set of documents, and then synthesize the
information. This can be implemented with an agentic loop or a multi-query
retrieval strategy.

**Q: "What happens when the vector database goes down?"**

A: The system should have a fallback strategy. If the vector DB is unavailable,
the system can fall back to keyword search (BM25) against the metadata store.
If both are unavailable, the system should return a degraded response: "The
knowledge base is currently unavailable. Please try again later." The LLM should
not be allowed to answer without context, as this defeats the purpose of RAG.

**Q: "How do you handle versioning of the vector index when the embedding model
changes?"**

A: Each vector is tagged with a model_version field. The query pipeline includes
the model version in the search request. During a model migration, both old and
new vectors coexist in the index. Queries are routed to the appropriate vectors
based on the query's model version. Once all documents have been re-indexed with
the new model, the old vectors are purged. This allows zero-downtime model
upgrades.

**Q: "How would you design a RAG system for a regulated industry like healthcare
where all answers must be auditable?"**

A: Every query, retrieved document set, generated response, and user feedback
event is logged with a unique trace ID. The system stores the exact prompt sent
to the LLM, including the full context. The LLM is configured to cite specific
source documents. A post-generation validation step checks that every citation
maps to a retrieval document. The audit trail is immutable and stored in a
write-once database. Compliance officers can replay any query to verify the
response was grounded in the retrieved documents.

## Summary: The FAANG RAG Interview Checklist

Before your interview, review this checklist to ensure you are prepared:

1. Can you explain RAG in one sentence? "Retrieve relevant documents, augment
   the prompt, generate a grounded answer."
2. Can you draw the two-pipeline architecture (ingestion + serving)?
3. Can you explain chunking strategy and justify your choices?
4. Can you name three embedding models and compare their trade-offs?
5. Can you explain HNSW and IVF and when to use each?
6. Can you describe the three-tier cache and its hit rates?
7. Can you discuss five failure modes and their mitigations?
8. Can you estimate cost for a 10M DAU system?
9. Can you compare RAG with fine-tuning and prompt engineering?
10. Can you discuss the emerging trends (GraphRAG, agentic RAG, streaming RAG)?

The final section presents hands-on challenges that test your ability to design,
analyze, and critique RAG architectures across multiple difficulty tiers.
