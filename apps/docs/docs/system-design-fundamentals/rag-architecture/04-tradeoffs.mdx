## The Core Tension

RAG introduces a fundamental three-way tension between retrieval quality,
generation latency, and context utilization. Optimizing one of these dimensions
typically degrades the other two.

**Retrieval Quality** is the measure of how relevant the retrieved documents are
to the query. Higher quality requires more sophisticated retrieval strategies
(multiple rounds of retrieval, re-ranking, query expansion), each of which adds
latency.

**Generation Latency** is the time from query submission to answer delivery. The
LLM generation step is already the dominant cost (500-2000ms). Adding more
context from retrieval widens the prompt, increasing both token processing time
and generation time.

**Context Utilization** measures how much of the available context the LLM
actually uses to generate the answer. Retrieved documents that are irrelevant,
redundant, or poorly formatted waste context window space and confuse the LLM.

## Decision Matrix

Every design decision in a RAG system involves a trade-off. The following table
covers the 8 most critical decision points.

### Decision 1: Chunk Size

| Option | Pros | Cons | Best For |
|--------|------|------|----------|
| Small (128-256 tokens) | High precision retrieval, low storage | May lack context, more vectors | Code snippets, FAQ |
| Medium (512 tokens) | Good balance | Moderate storage | General purpose |
| Large (1024+ tokens) | Rich context per chunk, fewer vectors | Lower precision, dilutes relevance | Long-form reports |

### Decision 2: Top-K Retrieval Count

| K Value | Pros | Cons |
|---------|------|------|
| 3-5 | Low latency, focused context | May miss relevant info |
| 10-20 | High recall, comprehensive | Context window pressure, more noise |
| 20+ | Maximum recall | High latency, LLM overwhelmed by noise |

### Decision 3: Embedding Model

| Model | Quality | Cost | Latency | When to Choose |
|-------|---------|------|---------|----------------|
| text-embedding-3-small | High | Low | Fast | General purpose, cost-sensitive |
| text-embedding-3-large | Highest | High | Slow | Accuracy-critical applications |
| BGE-small | Medium | Free | Fast | Self-hosted, resource-constrained |
| Cohere embed-multilingual | High | Medium | Medium | Multi-lingual corpora |

### Decision 4: Keyword vs. Vector vs. Hybrid

| Approach | Recall (Semantic) | Recall (Exact) | Complexity |
|----------|-------------------|----------------|------------|
| Vector only | High | Low | Low |
| Keyword only | Low | High | Low |
| Hybrid (RRF) | High | High | Medium |

### Decision 5: Re-ranking

| Strategy | Quality Gain | Latency Cost | When to Use |
|----------|-------------|--------------|-------------|
| No re-ranking | — | 0ms | Low-traffic, quality-tolerant |
| Cross-encoder re-rank | +15-25% | +100-300ms | High-quality requirements |
| LLM-as-judge re-rank | +20-30% | +500-2000ms | Maximum quality, small doc sets |

### Decision 6: Cache Strategy

| Cache Type | Hit Rate | Latency Savings | Consistency Risk |
|------------|----------|-----------------|------------------|
| No cache | 0% | 0ms | None |
| Query cache only | 10-20% | 600ms avg | Stale answers for document updates |
| Full three-tier | 50-70% | 400ms avg | Low (TTLs handle staleness) |

### Decision 7: Self-Hosted LLM vs. API

| Factor | Self-Hosted | API-Based |
|--------|-------------|-----------|
| Latency p50 | 100-300ms | 500-2000ms |
| Cost at 100M queries/month | $80,000+ | $200,000+ |
| Data privacy | Complete | Depends on provider |
| Maintenance | High | None |
| Model choice | Limited | Broad |

### Decision 8: Vector DB Type

| DB | Scalability | Consistency | Query Latency | Complexity |
|----|-------------|-------------|---------------|------------|
| Pinecone (managed) | Excellent | Strong | 5-10ms | Low |
| Weaviate (self-hosted) | Good | Strong | 5-20ms | Medium |
| Milvus | Excellent | Configurable | 5-50ms | High |
| FAISS (in-memory) | Limited | N/A | 1-5ms | Low |
| pgvector | Good | Strong | 10-100ms | Low (uses existing PG) |

**Implications for RAG:**
- During a vector DB partition, a CP system returns stale results or errors on
  write-heavy ingestion pipelines. This is acceptable because most RAG queries
  are read-only and can tolerate slightly stale indices.
- An AP system may return different results to different users during a partition.
  This is problematic for regulated industries where auditability requires
  deterministic results.
- For most RAG use cases, CP is the right choice. The cost of inconsistency
  (wrong answers due to stale indices) is lower than the cost of unavailability
  (no answers at all). But for real-time document collaboration scenarios where
  users expect to see their just-uploaded document reflected immediately, AP
  with read-your-writes consistency is required.

## Failure Modes

### Failure 1: Empty Retrieval

The vector database returns no documents above the similarity threshold. The LLM
receives no context and must answer from its training data alone, defeating the
purpose of RAG.

**Causes:**
- The embedding model does not align with the query domain (e.g., using a general
  English embedding model for medical queries)
- The similarity threshold is set too high
- The vector index is empty or corrupted
- The query is in a language or domain the embedding model was not trained on

**Detection:** Monitor the hit rate metric. A sustained hit rate below 90%
indicates a problem.

**Mitigation:**
- Lower the similarity threshold as a fallback
- Add a keyword search fallback that does not depend on embeddings
- Return an "I don't have enough information" message instead of guessing

### Failure 2: Irrelevant Retrieval

Documents are returned but they are not relevant to the query. The LLM is fed
misleading context and generates a wrong answer.

**Causes:**
- ANN search returns false positives (vectors that are numerically close but
  semantically unrelated)
- The chunking strategy splits related content across chunks
- The embedding model has poor performance for the specific domain

**Detection:** Track relevance score distribution. A significant fraction of
retrievals with scores below 0.5 indicates a problem.

**Mitigation:**
- Add a re-ranking step
- Fine-tune the embedding model on domain data
- Increase top-K and rely on the re-ranker to filter
- Use hybrid search (keyword + vector) to anchor retrieval in exact terms

### Failure 3: Hallucination from Retrieved Context

The LLM receives the correct context but ignores it and generates an answer from
its own knowledge, which may be wrong.

**Causes:**
- The system prompt does not sufficiently constrain the LLM to use the context
- The LLM has strong priors on the topic and overrides the context
- The temperature is set too high (above 0.3), encouraging creative generation
- The context contains contradictory information and the LLM resolves it by
  falling back to its own knowledge

**Detection:** Compare generated answers against the retrieved context. If the
answer contains information not present in any retrieved document, it is a
hallucination.

**Mitigation:**
- A lower temperature (0.0-0.1)
- A stronger system prompt: "Answer ONLY using the provided context. If the
  context does not contain the answer, say you do not know."
- Citation enforcement: require the LLM to cite specific source documents for
  each claim
- Post-generation validation: check each claim against the source documents

### Failure 4: Index Corruption

The vector index becomes corrupted due to a software bug, hardware failure, or
partial write during a crash. Queries return garbage results or error out.

**Causes:**
- Partial write to the index during an unexpected shutdown
- Memory corruption in the ANN algorithm
- Version mismatch between index build and query code

**Mitigation:**
- Daily full index rebuilds from the metadata store
- Checksum verification on every index load
- Redundant index replicas with automatic failover
- Backup of the raw chunks (not just the vectors) for rebuild

### Failure 5: Embedding Drift

The embedding model is updated to a new version. Old document vectors are
incompatible with new query vectors. Retrieval quality drops sharply.

**Causes:**
- The embedding model provider releases a new version
- You switch from one embedding model to another
- You fine-tune the embedding model on new data

**Mitigation:**
- Re-index all documents with the new model before switching
- Use a transition period where both models are active
- Tag each vector with its model version and route queries accordingly

### Failure 6: Latency Spike from Large Context

The retrieved context is much larger than expected, causing the LLM generation
time to spike. The query may time out or cost more than planned.

**Causes:**
- The chunk size is too large and too many chunks are retrieved
- The documents are unusually long and the chunker does not respect boundaries
- The top-K is set too high for the document type

**Mitigation:**
- Set a hard limit on total context tokens
- Implement context truncation with priority ordering
- Use a sliding window over large documents
- Monitor prompt size as a metric and alert on spikes

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Garbage In, Worship Out

The team spends months optimizing the LLM prompt and generation parameters while
ignoring retrieval quality. The system retrieves irrelevant documents but the
team blames the LLM for bad answers.

**Fix:** Measure retrieval quality first. The retrieval pipeline determines the
upper bound of system quality. An optimized LLM cannot compensate for bad
retrieval.

### Anti-Pattern 2: Context Dumping

The system retrieves 50 documents and passes all of them to the LLM, reasoning
that more context is always better. The LLM's attention is diluted across noise,
and generation quality degrades.

**Fix:** Retrieve a focused top-5 to top-10 documents. Use re-ranking to select
the best ones. The LLM works better with 3 high-quality documents than 20 mixed
ones.

### Anti-Pattern 3: One-Shot Indexing

Documents are indexed once and never re-indexed. New documents are added but
old, stale documents are never removed or updated. The index becomes a museum
of outdated information.

**Fix:** Implement a scheduled re-indexing job. Track document freshness and
rotate stale content. Use incremental indexing for real-time updates.

### Anti-Pattern 4: Impersonating Facts

The system prompt tells the LLM to be a "helpful assistant" without grounding
instructions. The LLM treats the retrieved context as optional reading and
generates answers from its training data, effectively becoming a traditional
chatbot.

**Fix:** The system prompt must explicitly instruct the LLM to use only the
provided context. Test that the LLM actually follows this instruction with
adversarial examples.

### Anti-Pattern 5: The Kitchen Sink Retrieval

The retrieval pipeline searches all documents regardless of access control,
domain, or relevance to the user. A finance employee asks about accounting
policies and gets HR documents about vacation policy.

**Fix:** Implement metadata filtering in the vector DB search. Tag each chunk
with domain, access level, and document type. Include these filters in every
query.

### Anti-Pattern 6: Zero Monitoring

The system is deployed to production with no monitoring of retrieval quality,
latency, or answer accuracy. Problems are discovered through user complaints.

**Fix:** Instrument every stage of the pipeline. Monitor hit rate, relevance
scores, latency per stage, cache hit rates, and user feedback. Set alerts for
degradation.

## When NOT to Use RAG

RAG is a powerful pattern, but it is not the right solution for every problem.
Here are the scenarios where you should avoid RAG.

**Small or Fixed Knowledge Base:** If your knowledge base fits in the LLM's
context window (under 10K tokens for GPT-4, under 100K for Gemini), you can
simply include it in the system prompt. This is simpler, faster, and avoids the
complexity of a vector database. RAG adds value when the knowledge base exceeds
the context window or changes frequently.

**Real-Time Requirements:** If you need answer latency under 100ms, RAG is
not the right choice. The retrieval pipeline alone (encoding + ANN search +
re-ranking) takes at least 50-100ms. The LLM generation adds 200-2000ms. For
real-time use cases, consider a lookup table, search index, or a smaller model
fine-tuned on your data.

**Highly Structured Query Patterns:** If all queries follow a known pattern
("What is the price of product X?", "Show me order Y"), a traditional database
query or rule-based system will be faster and more reliable. RAG excels at
open-ended, natural language questions where the information need is not known
in advance.

**Low Tolerance for Wrong Answers:** In high-stakes domains like medical
diagnosis or aircraft maintenance, RAG cannot guarantee correctness. Even with
perfect retrieval, the LLM may misinterpret the context. For these use cases,
combine RAG with deterministic validation rules, human-in-the-loop review, or
use a traditional information retrieval system instead.

**Privacy or Regulatory Constraints:** If your data cannot leave your on-premise
network and you cannot deploy an LLM on-premise, RAG with an API-based LLM is
not feasible. You would need to self-host both the retrieval and generation
components, significantly increasing operational complexity.

### Capacity Trade-Offs at Different Scales

The right RAG architecture depends heavily on the scale of your deployment.

**Startup scale (100 documents, 10 QPS):**
A naive implementation with FAISS in-memory and an API-based LLM is sufficient.
No caching needed. No re-ranking. The cost is dominated by the LLM API. Total
monthly cost: under $500.

**Mid-market scale (100K documents, 100 QPS):**
A production-grade pipeline with a managed vector database (Pinecone or
Weaviate), a re-ranker, and a basic Redis cache. The ingestion pipeline uses
a simple queue (Redis streams) for parallelism. Total monthly cost: $2,000-$5,000.

**Enterprise scale (10M documents, 1000 QPS):**
A full distributed architecture with multi-region deployment, dedicated GPU
instances for embedding and LLM serving, a distributed cache, and comprehensive
monitoring. Total monthly cost: $50,000-$120,000.

**Hyperscale (100M documents, 10,000 QPS):**
Custom vector database deployment with tiered storage, custom ANN algorithms
optimized for the specific data distribution, predictive caching, and a
specialized hardware accelerator for embedding. Total monthly cost:
$500,000-$2,000,000.

### Decision Framework for Trade-Offs

When faced with a RAG design decision, use this decision tree:

**Step 1:** Determine the quality requirement. Does the application need high
precision (medical, legal) or is approximate quality acceptable (internal FAQ)?

**Step 2:** If high precision, add re-ranking, use a larger embedding model,
increase chunk overlap, and use hybrid search. Accept the latency and cost
increase.

**Step 3:** Determine the latency requirement. Does the application need
sub-second responses or is 2-3 seconds acceptable?

**Step 4:** If sub-second required, optimize for cache hit rate, use a smaller
embedding model, skip re-ranking for simple queries, and consider a smaller
LLM. Accept the quality trade-off.

**Step 5:** Determine the throughput requirement. How many queries per second
at peak?

**Step 6:** If high throughput (1000+ QPS), invest in aggressive caching,
horizontal scaling for all stateless components, connection pooling for the
vector DB, and batching for the embedding and LLM calls.

**Step 7:** Determine the freshness requirement. How quickly must document
updates be reflected in search results?

**Step 8:** If real-time freshness needed, use incremental indexing with a
write-optimized vector index and near-real-time embedding pipeline. Accept the
complexity trade-off.

### Organizational Considerations

RAG architecture decisions are not purely technical. They also affect team
structure, development velocity, and operational burden.

**Build vs. buy trade-off:** A custom RAG pipeline gives full control and
customization but requires a team with ML infrastructure, data engineering, and
LLM operations expertise. A managed service (AWS Bedrock, Vertex AI RAG) reduces
development time from months to weeks but limits customization and creates
vendor lock-in. The right choice depends on whether RAG is your product's core
differentiator or a supporting feature.

**Team topology:** A RAG system touches document processing (data engineering),
embedding models (ML engineering), vector databases (infrastructure), and LLM
prompting (product engineering). A successful RAG deployment requires
collaboration across these teams. Dedicated RAG platform teams (one team owns
the full pipeline) work better than distributed ownership (each team owns their
piece) for most organizations.

**Operational maturity:** Running a production RAG system requires on-call
rotation, runbooks for each failure mode, and gradual rollout processes.
Organizations without existing ML infrastructure should budget 2-3 months for
operational maturity before going to production.
