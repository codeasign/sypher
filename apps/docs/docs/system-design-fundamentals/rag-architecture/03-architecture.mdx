## Naive Implementation

A single-process RAG system is the simplest possible implementation. It works
for prototyping, personal projects, or low-traffic internal tools. Understanding
it is important because it reveals the fundamental components without the
complexity of distributed systems.

The naive implementation uses FAISS (Facebook AI Similarity Search) as an
in-memory vector index. The entire pipeline runs in a single Python process.
Documents are loaded, chunked, and embedded on startup. Queries are answered
sequentially.

**When it is acceptable:**
- Prototyping a proof-of-concept
- Internal tool with fewer than 10 users
- Fewer than 1000 documents
- No uptime or reliability requirements

**When it fails:**
- The process crashes and the index is lost
- Memory runs out with more than 100,000 chunks
- Multiple concurrent queries cause queueing
- The LLM API call blocks the entire process

### Component Breakdown

**API Gateway (Kong / Envoy):** Handles authentication, rate limiting (100
requests/second per user), request routing, and TLS termination. Routes queries
to the embedder service and ingestion requests to the document pipeline.

**Embedder Service:** A stateless GPU-backed service running the embedding model.
Scales horizontally behind a load balancer. Each instance batches 32 queries
into a single GPU inference call for throughput. Typical latency: 10ms per query
at batch size 32.

**Retriever Service:** Wraps the vector database client. Handles ANN search,
keyword search, and RRF fusion. Stateless, CPU-bound. Scales based on QPS. Each
instance maintains a connection pool to the vector DB cluster.

**Re-ranker Service:** Runs the cross-encoder model on CPU (or GPU for high
throughput). The re-ranker is the most CPU-intensive component. A single instance
handles ~100 queries/second with 10 documents each.

**LLM Service:** Manages the LLM inference. For API-based models (GPT-4, Claude),
this is a thin proxy with retry logic and connection pooling. For self-hosted
models, this is a GPU cluster running vLLM or TensorRT-LLM.

**Cache Layer (Redis Cluster):** Three separate caches with different eviction
policies:
- Query cache: exact match on normalized query, TTL 1 hour, 10GB max
- Embedding cache: semantically similar queries share embeddings, TTL 5 minutes
- LLM response cache: cached by (query_hash, top_k_doc_ids), TTL 1 hour

**Vector DB Cluster:** A distributed vector database (Weaviate, Pinecone, or
Milvus) with 3-6 replicas. Supports hundreds of millions of vectors. Handles
both read (query) and write (indexing) workloads.

**Pattern:** The vector DB stores document chunks and tags. The retriever returns
chunk IDs and metadata. The relational DB enriches the context with structured
data (user tier, access permissions, subscription level). The enriched context
is passed to the LLM.

### Integration with Cache Layer (Redis)

Caching is critical for reducing latency and cost. The three-tier cache strategy
is:

1. **Query cache:** If the exact same query (normalized) was asked before, return
   the cached response. Cache hit rate: 10-20% for typical workloads.
2. **Embedding cache:** If a similar query was recently encoded, reuse the
   embedding vector. Cache hit rate: 30-40% for chat-like interactions.
3. **LLM response cache:** If the same query with the same retrieved documents
   was generated before, reuse the response. Cache hit rate: 15-25%.

### Replication Strategies

**Active-Active:** Each region accepts writes. Documents are replicated
asynchronously. Vector indexes are snapshotted and shipped periodically.
Conflict resolution uses last-writer-wins. Best for write-heavy workloads with
eventual consistency tolerance.

**Active-Passive:** One primary region accepts writes. All other regions serve
read-only replicas. Writes are replicated via change data capture (CDC). Best
for read-heavy workloads with strong consistency requirements.

**Consistency trade-offs:**
- Strong consistency: All reads return the latest write. Requires synchronous
  replication across regions. Adds 50-200ms of write latency.
- Eventual consistency: Reads may return stale data. Writes are fast. Acceptable
  for most RAG use cases because document content changes slowly.
- Read-your-writes: The user who wrote a document always sees their own writes.
  Requires sticky routing or version vectors.

### Data Residency

European data regulations require that personal data stays within the EU. The
RAG system must support data residency constraints.

**Implementation:**
- Document metadata includes a `region` tag
- The vector DB is partitioned by region
- Query routing respects the user's region
- Cross-region replication excludes documents with residency constraints
- The embedding model is deployed in the same region as the data it processes

## Deployment and Operations

### Zero-Downtime Ingestion Pipeline

The ingestion pipeline runs continuously, processing new documents and updating
the vector index. Zero-downtime deployment requires:

1. **Blue-green deployment** of the embedder and chunker services. New versions
   are deployed to a staging environment, validated, then traffic is switched.
2. **Dual-write during transition.** For 5 minutes after a deployment, both old
   and new indices receive writes. This ensures no documents are lost.
3. **Gradual rollback.** If the new version causes quality degradation, traffic
   is routed back to the old version within 30 seconds.

### Monitoring Retrieval Quality

Critical metrics to monitor:

**Hit Rate:** The percentage of queries where at least one document was retrieved
above the relevance threshold. Target: 95%+. A drop below 90% indicates a problem
with the vector index or embedding model.

**Relevance Score Distribution:** The distribution of top-1 relevance scores.
A shift toward lower scores indicates index staleness or embedding drift.

**Answer Acceptance Rate:** The percentage of answers rated positively by users.
A drop often correlates with retrieval quality degradation.

**Latency SLO:** p99 response time under 2 seconds for the complete pipeline
(including LLM generation). p50 under 500ms.

**Cache Hit Rate:** Monitor each cache tier. A drop in cache hit rate may indicate
a change in query patterns or an eviction policy issue.

### Debugging Wrong Answers

When a RAG system produces a wrong answer, the debugging process follows a
systematic chain:

1. **Check the retrieved context.** Were the right documents retrieved? If not,
   the problem is in the retrieval pipeline (embedding, index, chunking).
2. **Check the prompt.** Was the context formatted correctly? Was the system
   prompt appropriate? Token truncation may have removed critical context.
3. **Check the LLM response.** Did the LLM follow the instruction to use only
   the provided context? Did it hallucinate outside the context?
4. **Check the citations.** Are the citations correct? Do they match the source
   documents?

### Security Architecture

RAG systems introduce unique security concerns because they combine document
access control, LLM API calls, and user-facing responses.

**Access control propagation:** The RAG system must respect the access control
policies of the source document system. If a user cannot access a document in
Google Drive, the RAG system must not retrieve that document's chunks. This is
implemented by tagging each vector with an access control list (ACL) and
filtering queries by the user's permissions at retrieval time.

**Prompt injection defense:** Malicious users may craft queries designed to
extract information from documents they should not see, or to manipulate the
LLM into ignoring its instructions. The system prompt should include explicit
instructions about ignoring prompt injection attempts. The retrieval pipeline
should sanitize user input before encoding.

**Data exfiltration prevention:** The LLM's response must not contain the full
text of retrieved documents unless the user has permission to view them. For
users with limited access, the system should summarize rather than quote.

**Audit logging:** Every query, retrieved document, and generated response should
be logged with the user identity and timestamp. This is required for compliance
in regulated industries like healthcare and finance.

### Cost Optimization Strategies

At scale, the dominant cost components are LLM API calls, vector database
hosting, and GPU compute for embedding.

**LLM cost reduction:**
- Cache aggressively. A 40% cache hit rate reduces LLM costs by 40%.
- Use a smaller, cheaper model for simple queries and a larger model for complex
  ones. Route based on query difficulty (detected by query length, ambiguity
  score, or a classifier model).
- Implement tiered service: free tier uses a smaller model with a smaller top-K,
  paid tier uses the full pipeline.
- Batch embedding requests. The OpenAI embedding API allows batches of 2048
  inputs at the same cost as individual requests.

**Vector database cost reduction:**
- Use vector quantization (scalar quantization, product quantization) to reduce
  storage by 4x with minimal quality loss. A 1536-dimension vector stored with
  SQ8 quantization uses 1.5 KB instead of 6 KB.
- Tiered storage: hot vectors in RAM, warm vectors on SSD, cold vectors in
  object storage. Route queries to the appropriate tier based on document
  freshness or access frequency.
- Clean up stale vectors. Documents that are no longer referenced should be
  removed from the index.

**GPU compute cost reduction:**
- Batch embedding generation. A GPU can process 32-64 chunks per inference call.
  Queue chunks and process in batches rather than one at a time.
- Use spot instances for the ingestion pipeline (which is fault-tolerant) and
  reserved instances for the query pipeline (which needs consistent performance).

### SLA and Reliability

A production RAG system should target the following service level objectives:

**Availability:** 99.9% uptime (8.76 hours of downtime per year). This requires
redundant instances of every component and automated failover.

**Latency:**
- p50: under 500ms (including LLM generation)
- p95: under 1500ms
- p99: under 3000ms
- p99.9: under 5000ms

**Freshness:**
- p99 of documents indexed within 60 seconds of upload
- p99 of document updates reflected in search results within 300 seconds

**Quality:**
- Retrieval hit rate: 95%+
- Answer acceptance rate: 85%+ (measured by user feedback)
- Citation accuracy: 99%+ (citations point to correct sources)

### Disaster Recovery

A comprehensive disaster recovery plan for RAG includes:

**Data backup:** Daily backups of the metadata store (PostgreSQL) and raw
documents (object storage). Vector indices can be rebuilt from these backups
but the rebuild takes hours. Store backup snapshots of the vector index as well.

**Replication:** Multi-AZ deployment within a region for high availability.
Cross-region replication for disaster recovery with RPO of 5 minutes and RTO of
30 minutes.

**Recovery procedures:**
1. If the vector DB fails: failover to a secondary replica (RTO under 2 minutes).
2. If the metadata store fails: restore from the latest backup (RTO under 15
   minutes).
3. If the LLM API fails: fall back to a secondary LLM provider with a degraded
   experience (longer latency, lower quality).
4. If the entire region fails: route traffic to a standby region (RTO under 30
   minutes, RPO of 5 minutes).

### Testing Strategy

Testing a RAG system requires both offline evaluation (retrieval quality metrics)
and online evaluation (user-facing quality metrics).

**Offline evaluation:**
- Create a test set of 1000 query-document pairs with known correct answers.
- Measure hit rate, mean reciprocal rank (MRR), and normalized discounted
  cumulative gain (NDCG) for each retrieval strategy.
- Compare chunk sizes, overlap values, and embedding models against the test
  set.
- Automate these tests as part of the CI/CD pipeline so that any change to the
  chunker, embedder, or retriever is validated.

**Online evaluation:**
- A/B test every change. Run the old and new pipelines in parallel and compare
  answer acceptance rates.
- Monitor per-stage latency. A regression in any stage triggers an alert.
- Track user engagement: are users asking more questions over time? Are they
  rephrasing queries (indicating dissatisfaction with the first answer)?

**Regression testing:**
- Maintain a set of 100 edge case queries (ambiguous questions, multi-part
  questions, questions about non-existent topics, questions in different
  languages). Run these against every pipeline change.
- Monitor answer consistency: the same query should produce the same answer
  (or an equivalent answer) across multiple calls with the same retrieval
  context.

### Scaling the Ingestion Pipeline

The ingestion pipeline must handle bursts of document uploads without
overwhelming the system. For example, during a migration, a customer might
upload 100,000 documents in a single batch.

**Backpressure mechanism:** The message queue (Kafka) provides natural
backpressure. If the embedder workers cannot keep up, messages accumulate in
the queue. The system monitors queue depth and alerts if it exceeds a threshold
(100,000 pending messages). During sustained high load, additional embedder
workers are provisioned automatically.

**Idempotent processing:** Each document is processed with an idempotency key
(document hash + upload timestamp). If the same document is uploaded twice, the
second upload is ignored. This prevents duplicate vector entries when the
ingestion pipeline retries after a failure.

**Throttling:** The ingestion API enforces per-tenant rate limits. A single
tenant cannot upload more than 1000 documents per minute. This prevents one
tenant from starving the ingestion resources of other tenants.
