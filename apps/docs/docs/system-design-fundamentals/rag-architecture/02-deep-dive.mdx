## Data Structures and Algorithms

### Embedding Vectors

An embedding vector is a dense array of floating-point numbers produced by a
neural network encoder. The dimension is a critical design parameter.

**Dimension trade-offs:**
- 256-512 dimensions: faster search, less storage, suitable for narrow domains
- 768-1024 dimensions: good balance, used by BGE and E5 models
- 1536-3072 dimensions: higher accuracy, used by OpenAI text-embedding-3-large,
  but 4-6x more storage and slower search

**Similarity metrics:**
- Cosine similarity: measures the angle between vectors. Range [-1, 1]. Most
  common for RAG. Equivalent to normalized dot product.
- Dot product: faster than cosine but sensitive to vector magnitude. Use when
  embeddings are L2-normalized.
- Euclidean distance: measures straight-line distance. Less common for semantic
  search because it is sensitive to vector magnitude.

### ANN Index Structures

#### IVF (Inverted File Index)

IVF partitions the vector space into clusters (Voronoi cells). At query time,
only the closest clusters are searched rather than the entire space.

**Parameters:**
- `nlist`: number of clusters. Higher = finer granularity but more computation
  during index build. Typical range: 100-10000.
- `nprobe`: number of clusters to search at query time. Higher = better recall
  but slower. Typical range: 10-100.
- **Recall vs speed**: IVF with nprobe=10 achieves ~90% recall at 10x speedup
  over brute force. nprobe=50 achieves ~98% recall at 2x speedup.

#### HNSW (Hierarchical Navigable Small World)

HNSW builds a multi-layer graph where each layer is a progressively sparser
approximation of the full dataset. Search starts at the top layer (coarsest) and
descends to finer layers.

**Parameters:**
- `M`: number of connections per node. Typical range: 16-64. Higher = better
  recall but more memory.
- `ef_construction`: index build time search breadth. Typical range: 100-500.
  Higher = better index quality but slower build.
- `ef_search`: query time search breadth. Typical range: 50-500. Higher = better
  recall but slower.
- **Performance**: HNSW typically achieves 99% recall at 50x speedup over brute
  force. It is the dominant algorithm for production vector databases.

### Inverted Index for Keyword Retrieval

The inverted index maps each unique term to a list of document IDs containing
that term. BM25 is the standard ranking function that scores documents based on
term frequency and inverse document frequency.

### Maximum Marginal Relevance (MMR)

MMR is a diversity-enhancing algorithm used after retrieval to ensure the returned
documents are not all redundant. It selects documents that are both relevant to
the query and diverse from each other.

```
MMR = argmax( lambda * Sim(q, d_i) - (1 - lambda) * max Sim(d_i, d_j) )
              d_i                                 d_j in selected
```

A lambda of 1.0 gives pure relevance ranking. A lambda of 0.5 trades off 50%
relevance for 50% diversity. Typical values range from 0.3 to 0.7.

### Sliding Window Chunking

Documents are split into overlapping chunks of a fixed token size. The overlap
ensures that sentences spanning chunk boundaries are not lost.

## The Query Lifecycle

A single user query triggers a chain of operations across multiple subsystems.
Here is the complete lifecycle, step by step.

### Phase 1: Query Reception

**Step 1:** The API gateway receives the HTTP request and validates the user
session, rate limits, and request format.

**Step 2:** The query router checks if an exact match exists in the query cache.
If yes, the cached response is returned immediately (p99 latency under 5ms).

**Step 3:** The query is pre-processed: spelling correction, stop word removal,
and query expansion using synonyms from a domain-specific thesaurus.

### Phase 2: Query Encoding

**Step 4:** The pre-processed query is sent to the embedding model. The embedding
model is co-located on the same GPU node or accessed via a low-latency API.

**Step 5:** The embedding cache is checked. If the same query (or a semantically
similar query) was recently encoded, the cached vector is reused.

**Step 6:** The raw query is also sent to the keyword search index (BM25) for
the hybrid retrieval path.

### Phase 3: Hybrid Retrieval

**Step 7:** The query vector is sent to the vector database. The ANN index
(HNSW) returns the top-K nearest neighbors with their similarity scores.

**Step 8:** The raw query text is sent to the inverted index (BM25). The keyword
search returns the top-K matching documents with their BM25 scores.

**Step 9:** The results from both paths are merged using Reciprocal Rank Fusion
(RRF) or a weighted sum. RRF is parameter-free and robust:
```
RRF_score(d) = 1 / (k + rank_vector(d)) + 1 / (k + rank_keyword(d))
```
where k is typically 60.

### Phase 4: Re-ranking and Filtering

**Step 10:** The top-10 fused results are sent to a cross-encoder re-ranker. The
re-ranker evaluates each query-document pair and produces a fine-grained relevance
score. This is expensive (50-100ms per pair) but runs on only 10-20 documents.

**Step 11:** Documents below a relevance threshold are filtered out. The remaining
documents are sorted by their re-ranker score.

**Step 12:** MMR (Maximum Marginal Relevance) is applied to ensure diversity
among the final top-5 documents.

### Phase 5: Context Assembly

**Step 13:** The final documents are assembled into a structured prompt. The
context builder handles:
- Token counting: ensuring total prompt fits within the LLM context window
- Priority ordering: highest relevance documents first
- Source tracking: each chunk is tagged with its source document ID for citation
- Truncation: if the context exceeds the window, the lowest-ranked chunks are
  shortened or dropped

### Phase 6: LLM Generation

**Step 14:** The augmented prompt is sent to the LLM. The generation parameters
(temperature, top_p, max_tokens) are tuned for the specific use case.

**Step 15:** The LLM generates a response. The response is streamed back to the
user for better perceived latency.

**Step 16:** The response is parsed. Citations are extracted and matched to
source documents. The final answer is formatted with inline citations.

**Step 17:** The response is cached in the LLM response cache for future identical
or similar queries.

## Consistency and State

### Document Update During Active Conversation

When a document is updated mid-conversation, the system must handle inconsistency
between the vector index and the conversation context.

**Approach 1: Optimistic.** The old chunk remains in the vector index until the
background re-indexing job processes the update. The LLM may retrieve stale
information. Acceptable for non-critical use cases.

**Approach 2: Pessimistic.** When a document is updated, its embedding is deleted
from the index immediately. A background job re-embeds the new version. Queries
during the re-indexing window may miss the document entirely.

**Approach 3: Versioned.** Each chunk has a version number. The retrieval query
includes a version filter. If the conversation context references a specific
version, that version is used. This is the most correct but most complex approach.

### Stale Vector Index

Over time, the embedding model may be updated, or the underlying documents may
change. A stale index returns suboptimal results.

**Detection:** Monitor the average retrieval relevance score over time. A
sustained drop indicates index staleness.

**Remediation:** Schedule a full re-indexing job every 24-72 hours. Use
incremental indexing for documents that change frequently.

### Full Context Window

When the combined context exceeds the LLM's context window, the system must make
a decision about what to drop.

**Strategy 1: Truncation.** Keep the highest-relevance documents and drop the
rest. Simple but may lose useful context.

**Strategy 2: Summarization.** Use a separate LLM call to summarize the excess
context. More accurate but adds latency and cost.

**Strategy 3: Sliding Window.** Process the context in overlapping windows and
aggregate the results. Most accurate but expensive.

### Embedding Drift

When the embedding model is updated (e.g., from text-embedding-3-small to
text-embedding-3-large), the new query vectors live in a different space than
the old document vectors. The system must re-index all documents with the new
model.

**Mitigation:** Index documents with both the old and new model during a
transition period. Route queries based on the model version tag.

| Operation | Throughput | Notes |
|-----------|------------|-------|

### Chunk Size Impact

| Chunk Size | Recall@5 | Latency | Storage | Best For |
|-----------|----------|---------|---------|----------|
| 128 tokens | 72% | 1x | 3x | Short queries, code snippets |
| 256 tokens | 85% | 1.1x | 1.5x | FAQ-style Q&A |
| 512 tokens | 91% | 1.2x | 1x | Document Q&A, general purpose |
| 1024 tokens | 88% | 1.5x | 0.5x | Long-form content, reports |
| 2048 tokens | 82% | 2x | 0.25x | Full sections, legal documents |

The sweet spot is 512 tokens for most use cases. Smaller chunks improve recall
for precise queries but increase storage. Larger chunks reduce the number of
vectors but may contain irrelevant content that dilutes relevance.

## Configuration and Tuning

The following parameters are the most impactful knobs in a RAG system.

### Chunking Parameters

**chunk_size** — The target number of tokens per chunk. Default: 512. Trade-off:
smaller chunks improve retrieval precision; larger chunks provide more context
per retrieval.

**chunk_overlap** — The number of tokens overlapping between adjacent chunks.
Default: 50 (10% of chunk_size). Higher overlap reduces information loss at
chunk boundaries but increases storage and introduces duplicate content.

**chunking_strategy** — How chunks are created:
- `fixed`: Split at exactly chunk_size tokens, regardless of content boundaries.
  Fastest but may split sentences.
- `recursive`: Split at natural boundaries (paragraphs, then sentences, then
  words). Better quality but slower.
- `semantic`: Use a separate model to detect topic boundaries. Best quality but
  requires additional compute.

### Retrieval Parameters

**top_k** — The number of documents to retrieve. Default: 5. Range: 3-20.
Higher values increase recall but also increase context size and may introduce
noise. Tune based on the expected information density of your documents.

**similarity_threshold** — The minimum similarity score for a document to be
considered relevant. Default: 0.5. Range: 0.0-1.0. Below this threshold, the
document is excluded even if it is in the top-K. Prevents irrelevant documents
from polluting the context.

**ef_search** (HNSW) — The search breadth. Default: 200. Higher values improve
recall at the cost of latency. At 10M vectors, ef=200 gives ~99% recall. ef=500
gives ~99.5% recall but is 2x slower.

### LLM Parameters

**temperature** — Controls randomness. For RAG, use low temperature (0.0-0.3)
to maximize faithfulness to the retrieved context. Higher temperatures cause the
model to over-rely on its own knowledge.

**max_tokens** — The maximum number of tokens in the generated response. Should
be set based on the expected answer length. For medical Q&A, 500 tokens is
typically sufficient. For document summaries, 1000-2000 tokens.

**system_prompt** — The instructions given to the LLM about how to use the
retrieved context. Critical for grounding. Example:
```
You are a helpful assistant. Answer the question based only on the provided
context. If the context does not contain the answer, say "I don't have enough
information." Cite your sources using [1], [2], etc.
```

### Index Maintenance and Lifecycle

Vector indexes require ongoing maintenance to maintain performance as documents
are added, updated, and deleted.

**Index compaction:** Over time, deleted vectors leave gaps in the index. A
compaction process rebuilds the index segments to reclaim space and maintain
search speed. Compaction should run during low-traffic periods.

**Index snapshots:** Periodic snapshots allow rollback if a corruption event
occurs. Store snapshots in durable object storage (S3, GCS). A typical schedule
is one snapshot every 6 hours with a 48-hour retention window.

**Index versioning:** When the embedding model or chunking strategy changes, the
index must be rebuilt. Version-tagged indices allow gradual migration. Old
queries can be routed to the old index while the new index is being built.

### Memory and Storage Architecture

**In-memory vs. disk-based indexes:**

HNSW indexes are typically memory-resident for performance. At 1536 dimensions,
10 million vectors consume approximately 61 GB of RAM. Disk-based alternatives
(IVF with PQ, DiskANN) trade query speed for lower memory requirements.

**Storage hierarchy:**
- L1 cache (CPU): embedding vectors in L1 cache of the ANN search node. This is
  the fastest path but limited to small indexes (under 1M vectors).
- RAM: the full HNSW graph. For 10M vectors at 1536 dims, this is 60-80 GB of
  RAM. Add 20% overhead for the graph edges.
- SSD: raw chunk text, metadata, and backups. Chunk text for 10M vectors at 512
  tokens each is approximately 2.5 GB of compressed text. Metadata adds another
  0.5 GB.
- Object store (S3): Document originals, index snapshots, and cold storage for
  infrequently accessed documents.

### Query Optimization Techniques

**Query rewriting:** Before encoding the query, apply transformations that
improve retrieval quality:
- Spelling correction for typos in the query
- Synonym expansion (using a domain-specific thesaurus)
- Query decomposition for complex multi-part questions
- Adding implicit context from the conversation history

**Multi-query retrieval:** Generate multiple variations of the query and search
each one independently. Merge the results with RRF. This improves recall for
ambiguous queries at the cost of increased latency (one query becomes 3-5).

**HyDE (Hypothetical Document Embedding):** Ask the LLM to generate a
hypothetical document that would answer the query. Use this generated document's
embedding for retrieval instead of the query embedding. This works because the
generated document is in the same distribution as the indexed documents, making
the vector search more effective. HyDE is particularly useful for queries that
are poorly phrased or very short.

### Advanced Retrieval Strategies

**Iterative retrieval:** For complex questions, perform multiple rounds of
retrieval. The first round retrieves broad context. The second round uses the
retrieved context to formulate more specific sub-queries. This is similar to how
a human researcher might start with a broad search and narrow down.

**Self-RAG:** The LLM itself decides when to retrieve. The model generates
reflection tokens that indicate whether it needs more information. If the
reflection token says "need retrieval," the system retrieves additional context
and continues generation. This reduces unnecessary retrieval for questions the
LLM can answer from its own knowledge.

**Corrective RAG:** After retrieval, evaluate each retrieved document for
relevance. If the top documents are below a relevance threshold, rewrite the
query and retry retrieval before giving up. This adds one retry cycle before
falling back to a "no information available" response.

### Evaluation Metrics for Retrieval Quality

Offline evaluation of retrieval quality is essential for tuning the system
before deployment. The standard metrics are:

**Hit Rate (Recall@K):** The percentage of queries where at least one relevant
document is in the top-K retrieved set. A hit rate of 95% at K=10 means that
for 95% of queries, at least one relevant document appears in the top 10
results. This is the most commonly used metric for RAG evaluation.

**Mean Reciprocal Rank (MRR):** The average of the reciprocal rank of the first
relevant document. If the first relevant document is at position 1, the
reciprocal rank is 1.0. If it is at position 3, the reciprocal rank is 0.33.
MRR measures how quickly the user sees a relevant result.

**Normalized Discounted Cumulative Gain (NDCG):** Accounts for multiple levels
of relevance (not just binary relevant/not relevant). A document that perfectly
answers the query is weighted more heavily than a document that is partially
relevant. NDCG is more informative than hit rate when relevance is graded.

**Precision@K:** The percentage of retrieved documents that are relevant. If 5
out of 10 retrieved documents are relevant, Precision@10 is 0.5. This metric
matters when the LLM is sensitive to noise in the context.
