## Challenge Format

Each challenge asks you to design, analyze, or debug a pub-sub system. Challenges are organized into three tiers. Tier 1 tests basic understanding. Tier 2 requires applying design patterns. Tier 3 tackles advanced production scenarios.

Work through each challenge without looking at the model answer first. Estimate time per challenge: Tier 1 takes 10-15 minutes, Tier 2 takes 20-30 minutes, Tier 3 takes 30-45 minutes.

---

## Tier 1: Foundations

### Challenge 1.1: Identify the Pattern

For each system, determine whether pub-sub is the right pattern. Justify your answer.

(A) A photo upload service. Each photo must be resized to 3 dimensions (thumbnail, medium, full). Each resize runs in a separate worker.

(B) A chat application. When User A sends a message in a group chat with 200 members, all members should receive the message.

(C) A credit card payment system. A payment must be authorized, captured, settled, and reconciled. Each step depends on the previous step succeeding.

(D) An IoT sensor network. 500,000 temperature sensors send readings every 30 seconds. A dashboard displays live readings. An alerting system checks for thresholds. A data lake stores all readings for analysis.

**Model Answer**:

(A) **Yes, pub-sub**. The photo upload event fans out to three resize workers. Each resize is independent. If the thumbnail resize fails, the full-size resize should still proceed.

(B) **Yes, pub-sub**. Each message fans out to all group members. The group topic is the right abstraction: a publisher sends one message, the broker delivers to 200 subscribers.

(C) **No, not pub-sub**. The steps are sequential and dependent. If authorization fails, capture should not proceed. This is a saga or workflow pattern, not a pub-sub broadcast. Each step happens once, not to all subscribers.

(D) **Yes, pub-sub**. The sensor readings event fans out to three independent subscribers (dashboard, alerting, data lake). Each subscriber processes at its own pace. The alerting system needs low latency. The data lake benefits from batch consumption. Pub-sub handles all three from a single topic.

### Challenge 1.3: Explain Delivery Guarantees

You build a food delivery system. A `OrderPlaced` event goes to the restaurant (display on screen), to the customer (push notification), and to the accounting system (billing record). For each subscriber, specify the required delivery guarantee and justify.

**Model Answer**:

| Subscriber | Guarantee | Justification |
|---|---|---|
| Restaurant display | At-least-once | Must not miss the order. Duplicate display is acceptable (restaurant staff can dismiss). Loss is unacceptable — the order never gets cooked. |
| Customer notification | At-most-once | The notification is informational. A duplicate is annoying (push notification saying "order placed" twice). Loss is acceptable — the customer can check order status manually. |
| Accounting system | Exactly-once | Duplicate billing records overcharge the customer. Lost records undercount revenue. Either is a financial error. Use Kafka EOS or idempotent processing with unique order ID as deduplication key. |

## Tier 2: Design

### Challenge 2.1: Design a Real-Time Leaderboard

Design a pub-sub system for a gaming platform with 1 million concurrent players. Player scores must update a live leaderboard within 2 seconds. 50,000 score updates per second peak. The leaderboard is consumed by 10,000 web clients viewing the top 100 scores.

**Requirements**:
- Leaderboard updates must reach all viewing clients within 2 seconds
- The system must remain available during peak hours
- Score updates cannot be lost (a player's score must eventually be reflected)

**Model Answer**:

Architecture: three-tier pub-sub pipeline.

**Tier 1 — Score Ingestion**: each game server publishes `score.update` events to a Kafka topic with 24 partitions. Message key is `game_id` (all scores for the same game go to the same partition, preserving per-game ordering).

**Tier 2 — Leaderboard Aggregator**: a consumer group with 24 members reads from the 24 partitions. Each aggregator maintains an in-memory sorted set of scores (Redis Sorted Set). It applies score updates to the sorted set. The sorted set is sharded by game.

**Tier 3 — Leaderboard Distribution**: each aggregator publishes the updated leaderboard snapshot to a `leaderboard.update` topic. A WebSocket server subscribes to this topic and pushes updates to connected clients.

```text
Game Server  ──>  Kafka topic    ──>  Aggregator    ──>  WebSocket
(publisher)       (24 partitions)     (consumer group)    (subscriber push)
  score.update     50K msg/s         sorted set updater    client push
                                      100ms per update     under 2s to clients
```

**Failure handling**: if an aggregator crashes, Kafka rebalances the partitions to the remaining aggregators. The new aggregator rebuilds its sorted set by replaying the last 60 seconds of score updates from Kafka (retention set to 60s). WebSocket clients reconnect to any available WebSocket server, which reads the latest leaderboard from a shared Redis cache.

### Challenge 2.2: Design a Multi-Tenant Event Bus

Design a pub-sub system where 500 different teams (tenants) publish events. Each team has its own topics. The system must prevent one team's traffic spike from degrading another team's performance (noisy neighbor problem). Target: 10,000 messages/second per tenant peak, 200 tenants active simultaneously.

**Model Answer**:

**Isolation strategy: per-tenant partitions with rate limiting.**

Create a topic structure where each tenant gets dedicated partition capacity. Use a 2-level topic naming: `tenant-{id}.{domain}.{event}`.

**Broker configuration**:
- 16 broker nodes, each with 32 GB RAM, 4 TB NVMe, 25 Gbps network
- 100 partitions per high-traffic tenant, 10 partitions per low-traffic tenant
- Rate limit per tenant: 15,000 msg/s (10% buffer above peak)
- Rate limit per partition: 500 msg/s

**Quota enforcement**:

```yaml
# Per-tenant quotas enforced at broker level
quotas:
  tenant-acme-corp:
    producer_byte_rate: 10000000        # 10 MB/s
    consumer_byte_rate: 50000000        # 50 MB/s
    request_percentage: 10              # max 10% of broker CPU
  tenant-beta-inc:
    producer_byte_rate: 5000000
    consumer_byte_rate: 20000000
    request_percentage: 5
```

A monitoring dashboard tracks per-tenant throughput, latency, and error rates. If a tenant exceeds quota for 5 consecutive minutes, an alert fires. After 15 minutes, the tenant's rate limit is reduced to their contractual limit.

**Tenant onboarding**:
1. Tenant submits YAML declaring topics, partitions, expected throughput
2. CI/CD pipeline creates the topics with the specified partition count
3. Tenant receives credentials scoped to their topics (SASL/SCRAM with ACLs)
4. Monitoring dashboard shows tenant-specific metrics

### Challenge 2.3: Migration Strategy

A monolithic e-commerce application uses direct HTTP calls between modules. The team wants to migrate to an event-driven architecture using pub-sub. The migration must happen without downtime: the system must remain operational during the transition.

Design the migration strategy.

**Model Answer**:

**Phase 1 — Outbox Table (dual writes, no pub-sub yet)**
Add an outbox table in the monolith's database. Every write operation also writes to the outbox table in the same transaction. No subscribers yet. Validate that outbox writes do not impact database performance.

```sql
BEGIN;
  INSERT INTO orders ...;
  INSERT INTO outbox (event_type, payload, created_at)
    VALUES ('OrderPlaced', '{...}', NOW());
COMMIT;
```

**Phase 2 — Publish to Broker (shadow reads)**
Deploy an outbox publisher service that reads from the outbox table and publishes to a broker. The monolith still uses direct HTTP calls for processing. Metrics compare outbox records vs HTTP calls. Validate event schema and format. No subscriber acts on broker events yet.

**Phase 3 — New Services Subscribe (dual processing)**
Deploy new event-driven services that subscribe to broker topics. They process events in parallel with the monolith. This is dual processing. Validate that the new services produce identical results. Run for 2-4 weeks to build confidence.

**Phase 4 — Traffic Switch (feature flag)**
Use feature flags per module to switch from HTTP to event-driven processing. Start with read-only modules (search indexing, analytics). Move to write modules (notification, inventory) last. Each module has a rollback plan.

```text
Phase 1: [Monolith DB] ──> [Outbox table]
Phase 2: [Monolith DB] ──> [Outbox table] ──> [Broker]
Phase 3: [Monolith DB] ──> [Outbox table] ──> [Broker] ──> [New Services] (parallel)
Phase 4: [Monolith DB] ──> [Outbox table] ──> [Broker] ──> [New Services] (primary)
         [HTTP calls removed via feature flag]
```

## Tier 3: Advanced

### Challenge 3.1: Debug a Production Incident

**Scenario**: Your team runs a Kafka cluster handling order processing. At 2:14 PM, customer support receives reports that orders placed after 2:00 PM have not been processed. The analytics pipeline shows no `OrderPlaced` events after 2:00 PM. The inventory service did not receive any orders since 2:00 PM.

The Kafka cluster has 6 nodes, replication factor 3, `min.insync.replicas=2`, `acks=all`. The `orders` topic has 12 partitions.

**Available data**:
- Broker 3 went offline at 2:03 PM (hardware failure)
- Partition leaders at 2:00 PM:
  - Broker 1: partitions 0, 1, 2
  - Broker 2: partitions 3, 4, 5
  - Broker 3: partitions 6, 7, 8  (offline)
  - Broker 4: partitions 9, 10, 11
  - Broker 5: replicas for partitions 0, 3, 6, 9
  - Broker 6: replicas for partitions 1, 4, 7, 10
- Consumer group `order-processor`: 6 members, each handling 2 partitions
- Metrics show consumer lag on partition 6 growing from 0 to 500,000 and holding

What is the most likely cause? What is the fix? How do you prevent recurrence?

**Model Answer**:

**Diagnosis**: Broker 3 hosted partitions 6, 7, and 8 as leader. When Broker 3 went offline, the controller needed to elect new leaders. However, the replicas for partitions 6, 7, and 8 are on Brokers 5 and 6. With `min.insync.replicas=2`, if either Broker 5 or Broker 6 had fallen out of the ISR before the crash (perhaps due to GC pauses or network hiccups), the controller may have been unable to elect a new leader without enabling unclean leader election (`unclean.leader.election.enable=false` by default). Without a leader, these partitions are unavailable — producers cannot write, consumers cannot read.

The growing consumer lag on partition 6 confirms: the partition is stuck. Producers that need to write to partitions 6, 7, or 8 receive `NotLeaderForPartitionException` and retry. If the retries exhaust, the publisher fails silently.

**Immediate fix**:
1. Check if Brokers 5 and 6 are in the ISR for partitions 6, 7, 8:
   `kafka-topics --describe --topic orders --bootstrap-server broker1:9092`
2. If replicas exist but are out of sync, bring them back into ISR (restarting them may help).
3. If replicas are in sync but election failed, trigger preferred leader election:
   `kafka-leader-election --bootstrap-server broker1:9092 --topic orders --election-type preferred`
4. If all replicas are down, enable unclean leader election temporarily:
   `kafka-configs --bootstrap-server broker1:9092 --entity-type topics --entity-name orders --alter --add-config unclean.leader.election.enable=true`
   Then trigger leader election. After recovery, revert the config.

**Prevention**:
- Increase `min.insync.replicas` from 2 to 3 for critical topics (requires 4+ replicas)
- Monitor ISR health: alert when any partition's ISR size falls below 2
- Tune broker JVM GC to reduce pause times (use G1GC, target max pause < 200ms)
- Add Chaos Engineering: kill a broker node weekly in staging to validate automatic recovery

### Challenge 3.2: Design a Multi-Region Active-Active System

Design a pub-sub system spanning 3 AWS regions (us-east-1, eu-west-1, ap-southeast-1). Publishers in each region publish events. Subscribers in each region consume events from their local region with the option to consume events from other regions.

Requirements:
- A publisher in us-east-1 must be acknowledged within 50ms
- A subscriber in eu-west-1 must receive events from us-east-1 within 5 seconds
- No single region failure can cause data loss (RPO = 0)
- Ordering must be preserved per-key across regions

**Model Answer**:

Architecture: active-active with local-first writes and cross-region replication via a combination of Kafka MirrorMaker and shared log.

```text
Region A (us-east-1)        Region B (eu-west-1)      Region C (ap-southeast-1)
┌────────────────────┐    ┌────────────────────┐     ┌────────────────────┐
│  Kafka Cluster A   │    │  Kafka Cluster B   │     │  Kafka Cluster C   │
│                    │    │                    │     │                    │
│  orders.us-east    │    │  orders.eu-west    │     │  orders.ap-southeast
│  Topic             │    │  Topic             │     │  Topic             │
│                    │    │                    │     │                    │
│  MirrorMaker ──────────► MirrorMaker ◄───────────── MirrorMaker        │
│  (push & pull)    │    │  (push & pull)      │     │  (push & pull)     │
└────────────────────┘    └────────────────────┘     └────────────────────┘
```

**Key design choices**:

1. **Local-first writes**: each region has its own Kafka cluster. Publishers write to their local cluster (50ms latency). The local cluster acknowledges immediately.

2. **Per-region topic namespacing**: topics are prefixed with region: `us-east-1.orders.created`, `eu-west-1.orders.created`. This avoids conflict and enables per-region routing.

3. **Cross-region replication**: MirrorMaker 2 runs in each region, pulling events from the other two regions and pushing local events to the other regions. Replication is asynchronous.

4. **Ordering per-key across regions**: use a deterministic region priority scheme (CRC32(key) % N_regions determines which region is authoritative for that key). Events for key "user-123" always route to us-east-1's partition, regardless of where they are published. Mirrors preserve this routing.

5. **RPO=0**: the publisher acknowledgment in us-east-1 means the message is committed to the local cluster. It will be replicated asynchronously. If us-east-1 suffers a total failure before replication, events not yet mirrored are lost (RPO is actually seconds, not 0). To achieve true RPO=0, use synchronous cross-region replication (Kafka with followers in other regions), but that adds 100ms+ to publish latency. Acceptable trade-off here: the 5-second replication SLA provides RPO of 5 seconds.

**Reader configuration**:
- Subscribers in us-east-1 read from both `us-east-1.*` topics (local events) and `eu-west-1.*`, `ap-southeast-1.*` topics (replicated events from other regions)
- A subscriber sees all events across all regions with a delay of 2-5 seconds for cross-region events

### Challenge 3.3: Schema Evolution Disaster

**Scenario**: A team publishes events with schema version 1 (fields: `orderId`, `customerId`, `total`). They add a required field `currency` in version 2. The schema registry accepts the change. All subscribers crash within 5 minutes of the deploy.

What went wrong? How should the schema have been evolved? Design a safe schema evolution process.

**Model Answer**:

**Root cause**: adding a required field is a breaking schema change. Subscribers running version 1 code cannot deserialize version 2 events that include the new required field. If the subscriber uses strict deserialization (fail on unknown/required fields), it throws an exception and crashes.

**Safe schema evolution**:

1. New fields must be optional (nullable or have a default value). In Avro: `{"name": "currency", "type": ["null", "string"], "default": null}`. In Protobuf: `string currency = 4 [json_name = "currency"];` — new fields are optional by default.

2. Schema compatibility must be checked at CI/CD time, not at runtime. Use the schema registry's compatibility checker:
   ```
   Schema Registry: orders-created value
   Compatibility: BACKWARD (default for Confluent)
   Version 1 -> Version 2: allowed only if v2 can read v1 data
     (adding optional field is backward-compatible)
   Version 1 -> Version 2: NOT allowed if v2 adds required field
   ```

3. Double-publish during migration: publish both the old and new schema versions simultaneously for a transition period.

**Safe process**:

```text
Step 1: Add field as optional + default
   Version 1: {orderId, customerId, total}
   Version 2: {orderId, customerId, total, currency?}

Step 2: All subscribers upgrade to handle v2 (gracefully handle missing currency)

Step 3: Make field required and publish version 3
   Version 3: {orderId, customerId, total, currency}
   Schema registry: FORWARD compatibility
     (v3 is forward-compatible with v2 — v2 readers can read v3 because
      they already handle the field, even if they expect it optional)

Step 4: Old subscribers upgraded. Remove v1 schema.
```

This process adds 2-4 weeks to a simple field addition, but it prevents production outages. There is no shortcut to safe schema evolution.
