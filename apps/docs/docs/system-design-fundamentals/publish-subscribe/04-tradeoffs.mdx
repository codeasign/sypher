## The Core Tension

Every pub-sub system navigates a fundamental tension between three competing goals:

**Throughput**: messages processed per second. Maximized by batching, weak ordering, and non-persistent delivery.

**Durability**: guarantee that messages survive failures. Maximized by disk writes, replication, and synchronous acknowledgments.

**Latency**: time from publish to delivery. Minimized by in-memory buffering, push delivery, and minimal replication.

You cannot maximize all three simultaneously. Every pub-sub decision is a triangulation among these axes.

## Decision Matrix

When to choose which trade-off.

| Requirement | Choose | Reason |
|---|---|---|
| 100,000 msg/s, can lose a few | Kafka, acks=1, non-persistent | Maximum throughput, minimal durability |
| 1,000 msg/s, cannot lose any | RabbitMQ, confirm mode, persistent queues | Strong guarantees, moderate throughput |
| Under 10ms latency, loss OK | Redis Pub/Sub, MQTT QoS 0 | In-memory, push-based, minimal overhead |
| Global ordering across topics | Single Kafka partition | All messages through one consumer, limited scale |
| Exactly once, no duplicates | Kafka with EOS + idempotent producer + transactional consumer | Transactional semantics, moderate throughput |
| IoT devices, low bandwidth | MQTT over TLS | Compact wire protocol, QoS levels, small footprint |

## CAP Theorem in Pub-Sub

Pub-sub systems are distributed systems. They face the CAP theorem: Consistency, Availability, Partition Tolerance — choose two.

Pub-sub systems typically prioritize availability and partition tolerance over strong consistency. A broker cluster that splits into two partitions continues to accept publishes and deliveries on both sides. When the partition heals, the two sides reconcile.

### Consistency Models in Practice

Some publish-subscribe implementations expose consistency as configurable parameters rather than a single model.

**Kafka**: configurable per-topic. Default: eventual consistency with unclean leader election disabled (consistency preferred). Option: enable unclean leader election (availability preferred, may lose data).

**RabbitMQ**: configurable per-queue. Quorum queues: strong consistency within the quorum (majority of nodes). Classic mirrored queues: eventual consistency.

**Google Pub/Sub**: strong consistency for published messages (once a publish ack is returned, the message is durable). Eventual consistency for subscription changes (new subscribers may not see messages published milliseconds ago).

```python
# Kafka: consistency vs availability is a config knob
topic_config = {
    # "all" = wait for all in-sync replicas to ack
    # "1" = wait for leader only (faster, may lose on leader crash)
    "min.insync.replicas": 2,
    "acks": "all",
    # Prevent consumers from reading from a partition leader
    # that has lost its replicas (consistency)
    "unclean.leader.election.enable": False
}
```

## Five Failure Modes

Every pub-sub system experiences failures. The question is how it behaves under failure.

### Failure Mode 1: Slow Subscriber (Head-of-Line Blocking)

A single subscriber takes 30 seconds to process each message. Its delivery buffer fills. In some broker implementations, this slow subscriber blocks delivery to other subscribers of the same topic because the broker uses a single delivery thread per topic.

**Impact**: latency for all subscribers of that topic degrades. Messages pile up in the buffer. Memory usage grows.

**Mitigation**: use separate delivery threads per subscriber. Set per-subscriber prefetch limits. Configure timeouts to drop slow subscribers to a dead letter queue after N retries.

```yaml
# Prefetch limit prevents a slow subscriber from hoarding messages
subscriber:
  prefetch_count: 100  # at most 100 unacked messages at once
  consumer_timeout_ms: 300000  # disconnect consumer if idle 5 min
```

### Failure Mode 2: Publisher Overwhelms Broker (Backpressure Failure)

A publisher spikes to 500,000 messages/second, far exceeding the broker's capacity of 100,000 messages/second. The broker's inbound buffer grows until it exhausts memory. The broker crashes or becomes unresponsive.

**Impact**: all topics and subscribers on that broker are affected. Message loss on crash. Recovery time: minutes to hours (rebuilding from replicas).

**Mitigation**: rate limiting at the network layer (per-client token bucket). Quotas per topic. Backpressure signaling (HTTP 429, Kafka produce throttling). Circuit breakers at the publisher.

```python
# Per-client rate limiter
class RateLimiter:
    def __init__(self, rate: int, burst: int):
        self.tokens = burst
        self.max_tokens = burst
        self.rate = rate  # tokens per second
        self.last_refill = time.monotonic()

    def allow(self) -> bool:
        now = time.monotonic()
        elapsed = now - self.last_refill
        self.tokens = min(self.max_tokens,
                          self.tokens + elapsed * self.rate)
        self.last_refill = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False
```

### Failure Mode 3: Broker Node Failure (Loss of Partition Leader)

A broker node crashes. All partition leaders on that node become unavailable. Producers and consumers connected to that node lose connectivity.

**Impact**: publish and subscribe operations for the affected partitions pause. Pause duration: 5-30 seconds (failure detection + leader election + metadata propagation). During the election window, messages sent to the old leader are lost.

**Mitigation**: run at least 3 broker nodes. Set `unclean.leader.election.enable=false` to avoid electing a leader that is missing messages. Use client-side retry with exponential backoff. Implement circuit breakers in publishers.

```python
# Client-side retry with backoff on leader failure
def publish_with_retry(producer, topic, message, max_retries=5):
    for attempt in range(max_retries):
        try:
            future = producer.send(topic, message)
            future.get(timeout=10)
            return
        except LeaderNotAvailableError:
            wait = min(2 ** attempt * 0.1, 10)  # 0.1, 0.2, 0.4, ...
            time.sleep(wait)
            producer = reconnect_producer()
    raise MaxRetriesExceeded(f"Could not publish after {max_retries} attempts")
```

### Failure Mode 4: Network Partition (Split Brain)

A network failure splits the broker cluster into two groups. Each group continues operating independently. Both groups accept publishes. When the network heals, two divergent sets of messages exist.

**Impact**: subscribers that were connected to group A saw messages published to group A. Subscribers connected to group B saw messages published to group B. After healing, some messages may be lost or in different order.

**Mitigation**: use a consensus-based failure detector (ZooKeeper, Raft). A partition that loses quorum stops accepting writes. Configure `min.insync.replicas` to require at least 2 replicas — a single node partition cannot accept writes alone. Accept that some messages may be lost during the partition window.

### Failure Mode 5: Message Ordering Violation

A subscriber expects messages in publish order. A broker crash or rebalancing causes messages to be delivered out of order.

Example: `OrderCreated(id=1)`, `OrderShipped(id=1)`, `OrderCreated(id=2)`, `OrderShipped(id=2)` is the expected order. Subscriber receives `OrderShipped(id=1)` before `OrderCreated(id=1)` after a rebalance.

**Impact**: application logic that assumes ordering breaks. Inventory decrements before it was reserved. Emails about shipped orders sent before the order was placed.

**Mitigation**: enforce ordering within a partition (Kafka). Use sequence numbers and detect gaps. Design idempotent event handlers that can handle out-of-order events. Use version vectors or timestamps to reorder.

## Five Anti-Patterns

Common mistakes teams make with pub-sub.

### Anti-Pattern 1: Pub-Sub as RPC Replacement

Developers model request-response interactions as pub-sub events. Service A publishes a `GetUserProfile` event. Service B processes it and publishes a `UserProfileResponse` event. Service A blocks waiting for the response.

**Why it fails**: pub-sub is asynchronous. Service A does not know when or if the response arrives. Correlation requires complex request ID matching. Timeouts are hard to implement. The one-to-one nature of RPC is replaced by a broadcast mechanism that sends the response to ALL subscribers.

**Better approach**: use RPC (gRPC, HTTP) for request-response. Use pub-sub for fire-and-forget events. If async request-response is needed, use a dedicated pattern like callback queues or websockets.

### Anti-Pattern 2: Over-Wildcarded Subscriptions

A subscriber subscribes to `#` (MQTT) or `*` (Kafka pattern) to receive all topics. The subscriber processes all events in a single handler — order events, payment events, sensor readings, audit logs.

**Why it fails**: the subscriber becomes a bottleneck for all topics. A spike in sensor readings delays order processing. The handler logic becomes a ball of mud with type-switching. Schema changes in any topic risk breaking the subscriber. The broker must deliver every message to this subscriber, consuming bandwidth even for topics the subscriber does not process.

**Better approach**: subscribe only to the topics you need. Use separate subscriber instances for distinct event categories. Use content-based filtering when topic granularity is insufficient.

### Anti-Pattern 3: Infinite Retention

Retention is set to "forever" or "unlimited" to avoid losing data. The broker accumulates messages indefinitely.

**Why it fails**: disk fills up. Log cleanup (compaction or deletion) kicks in and causes I/O spikes that impact live traffic. Recovery from a crash takes hours because the broker replays years of messages. Storage costs grow unboundedly.

**Better approach**: set retention based on business requirements. Order events: 7 days. Audit logs: 30 days. Sensor readings: 24 hours. Use compacted topics for keyed data where only the latest value matters (e.g., last-known state). Archive old data to cheap object storage (S3, GCS) for long-term access.

```yaml
# Reasonable retention defaults
topics:
  orders.*:
    retention.ms: 604800000    # 7 days
    cleanup.policy: delete
  inventory.current:
    retention.ms: 86400000     # 1 day
    cleanup.policy: compact    # keep only latest per key
  audit.log:
    retention.ms: 2592000000   # 30 days
    cleanup.policy: delete
```

### Anti-Pattern 4: Giant Monolithic Event

A single event payload contains all possible data for all subscribers. The `OrderPlaced` event includes the full customer profile, inventory snapshots, shipping labels, payment details, and marketing preferences.

**Why it fails**: the event payload grows to 500 KB. Network bandwidth is wasted because no single subscriber needs all fields. Schema changes affect all subscribers — adding a field requires approval from every team. Serialization/deserialization overhead increases. Storage costs rise.

**Better approach**: publish focused events with minimal payloads. Include only the data that is relevant to the event. Subscribers that need additional data fetch it via API. Use event enrichment at the subscriber side.

### Anti-Pattern 5: Ignoring Dead Letter Queues

Failed messages are retried indefinitely or silently dropped. No DLQ is configured. The engineering team does not monitor delivery failures.

**Why it fails**: a message with invalid data retries forever, consuming resources. A downstream service bug causes all messages to fail silently. The team discovers the data gap weeks later during reconciliation. Reprocessing requires manual scripts or replay from backups.

**Better approach**: configure DLQ for every subscription. Monitor DLQ depth and alert when it exceeds zero. Set up a process to inspect, fix, and replay DLQ messages. Treat DLQ > 0 as a production incident.

```yaml
# Kafka: dead letter topic configuration
dead_letter:
  enabled: true
  topic: "orders.created.dlq"
  max_retries: 5
  alert_on_dlq: true
  alert_threshold: 10  # alert if DLQ has >10 messages
```

## When NOT to Use Pub-Sub

Pub-sub is not a universal solution. These situations call for other patterns.

### Low-Latency Request-Response

You need a response within 5 milliseconds. An HTTP call to a well-tuned service takes 2-10ms. A pub-sub round trip (publish, store, route, deliver, process, respond) takes 20-100ms minimum.

**Use instead**: direct HTTP, gRPC unary call, WebSocket request-response.

### Strongly Consistent State

You need every read to reflect the latest write with no delay. Pub-sub delivers events asynchronously — there is always a window where the write has occurred but the subscriber has not yet processed it.

**Use instead**: database transactions, distributed consensus (Raft, Paxos), or a strongly consistent store like Spanner or etcd.

### Small-Scale Workload

You have 2-3 microservices and handle 100 requests/second. A full pub-sub deployment with brokers, ZooKeeper, and replication adds operational complexity with no benefit.

**Use instead**: HTTP API calls, a shared database with change data capture, or a lightweight message queue like Redis list operations.

### One-to-One Job Distribution

Every task must be processed by exactly one worker. You need to track completion and handle failures. Pub-sub fan-out sends each job to all subscribers. You would need to layer a competing-consumers pattern on top.

**Use instead**: a dedicated job queue (RabbitMQ work queues, Amazon SQS, Redis streams with consumer groups).

### Data Streams Requiring Strict Order Across All Keys

Your application processes events in strict global order. Every event must be processed in the exact order it was generated. Pub-sub ordering guarantees are per-partition or per-key, not global.

**Use instead**: a single-threaded event processor reading from a log (Apache Kafka with 1 partition, or Kinesis with 1 shard). Accept the throughput limit.
