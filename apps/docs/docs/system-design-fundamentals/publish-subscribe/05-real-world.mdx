## Case Study 1: Netflix вҖ” Event-Driven Microservices at 2 Billion Events/Day

Netflix runs one of the largest pub-sub deployments in the world. Their event bus, based on Apache Kafka, processes over 2 billion events per day across hundreds of microservices.

### Architecture

Netflix uses a multi-layered event architecture. At the foundation is Kafka, with over 100 clusters spanning multiple AWS regions. Each cluster handles a specific domain: user activity, content ingestion, playback events, billing, recommendation training.

### Key Decisions

**Topic per domain**: each domain team owns its Kafka cluster. Topic names follow `domain.service.event` convention. This prevents the blast radius of a misconfigured topic from affecting unrelated services.

**Schema registry**: all events use Avro schemas stored in a central schema registry. Schema evolution follows backward-compatible rules: adding optional fields is allowed, removing or renaming fields triggers a breaking change review.

**SLA tiers**: critical events (billing, signup) use synchronous replication with acks=all. High-volume but non-critical events (playback heartbeats, UI interactions) use acks=1. Batch analytics events use acks=0 and are delivered to a separate analytics cluster.

**Lessons learned**: without per-domain clusters, a noisy neighbor in one team's topic (e.g., a bug causing 10x event volume) degraded latency for all teams. Moving to domain-isolated clusters eliminated this cross-team impact. Each domain team operates independently.

## Case Study 2: Uber вҖ” Reliable Event Delivery at 100 Million Messages/Hour

Uber uses Apache Kafka as the backbone for their trip lifecycle, pricing, dispatch, and payment systems. Their pub-sub infrastructure handles over 100 million messages per hour during peak times.

### Architecture

Uber runs one of the largest single Kafka deployments, with clusters spanning thousands of broker nodes. They developed Ringpop, a consistent hashing layer, to distribute topics across brokers with automatic rebalancing.

### Problem: Message Ordering in Multi-Stage Pipelines

Uber's trip lifecycle involves 15+ sequential stages: request, dispatch, driver-accept, passenger-board, trip-start, route-update, trip-end, payment, rating. Each stage publishes events that trigger the next stage.

The challenge: if events arrive out of order (e.g., `trip-end` arrives before `trip-start` due to a network delay or broker failover), downstream systems corrupt state.

**Solution**: per-trip ordering using Kafka message keys. Each event for a specific trip uses `trip_id` as the Kafka message key. All events for that trip go to the same partition, preserving publish order. Each service uses a deterministic processing model: it ignores events for trips it has not yet seen (idempotent rejection).

```python
# Per-trip ordering using message keys
producer.send("trip-events", key=trip_id.encode(), value=event_data)
# All events for trip "abc-123" go to the same partition, in order
```

### Problem: Exactly-Once Semantics for Payment Events

Duplicate payment events cause financial loss (charging a rider twice). At-least-once delivery (the default in most pub-sub systems) could produce duplicates after a broker failover or consumer crash.

**Solution**: Kafka's exactly-once semantics (EOS) with idempotent producers and transactional consumers. The producer assigns a unique transaction ID. The broker deduplicates messages with the same transaction ID within a window. The consumer reads transactions atomically.

This is one of the few real-world deployments of true exactly-once semantics in pub-sub. The cost: approximately 20% throughput reduction compared to at-least-once.

### Operational Practices

Uber invested heavily in operational tooling:

- **Automated topic creation**: developers declare topics in a YAML config. CI/CD creates or updates the topic configuration.
- **Quota enforcement**: per-service publish rate limits prevent one team from overwhelming the cluster.
- **Automated rebalancing**: Ringpop redistributes partitions when nodes are added or removed, with careful throttling to avoid overload.
- **Simian Army testing**: Chaos Monkey randomly kills broker nodes to validate failover behavior.

## Case Study 3: Cloudflare вҖ” Real-Time Event Processing at 25 Million Events/Second

Cloudflare processes over 25 million events per second across their global network. Their pub-sub system handles analytics data, DDoS attack signals, cache purges, and DNS updates.

### Architecture

Cloudflare uses a custom pub-sub system built on their own infrastructure rather than an off-the-shelf broker. Each of their 300+ edge data centers runs a local broker instance. Events are first published locally, then asynchronously replicated to regional aggregators.

### Problem: Global Fan-Out with Minimal Latency

When a customer purges a cache, the purge event must reach all 300+ data centers within 500 milliseconds. A naive approach вҖ” publishing to a central broker that fans out to all edges вҖ” adds 100-300ms just for the first hop + centralized bottleneck.

**Solution**: hierarchical pub-sub with local-first delivery. Each edge data center runs a local broker. Cache purge events are gossiped through a tiered overlay network.

```text
Step 1: Customer sends purge request to nearest edge data center
Step 2: Local broker publishes purge event locally (applied in under 10ms)
Step 3: Local broker sends purge event to regional aggregator (1 per continent)
Step 4: Regional broker fans out to other regional brokers
Step 5: Each regional broker forwards to all edges in its region
Step 6: Edge brokers apply purge event locally

Total: 95% of edges receive the purge within 400ms
```

### Key Insight: Local First

The critical design decision: the purge is applied locally immediately, then propagated asynchronously. This means the customer sees the purge take effect at the edge they hit (low latency), while other edges catch up within milliseconds. The system is eventually consistent globally, but immediately consistent for the requesting customer.

## Case Study 4: Discord вҖ” Scaling from 0 to 5 Million Concurrent Users

Discord's early architecture used a single RabbitMQ node for all event routing. As the platform grew, the single node became a bottleneck and a single point of failure.

### The RabbitMQ Meltdown

At 500,000 concurrent users, the RabbitMQ broker began experiencing cascading failures:

1. A user-created server with 100,000 members went viral
2. Messages in that server generated high event volume
3. RabbitMQ's single queue became saturated
4. Delivery latency spiked from 10ms to 30 seconds
5. Consumers were slow to ack, causing queues to grow
6. Growing queues consumed memory
7. RabbitMQ hit its memory high-water mark and blocked publishers
8. All Discord events stalled вҖ” not just the busy server
9. The engineering team had to restart RabbitMQ to recover

### The Fix: Sharded Pub-Sub with Partitioning

Discord replaced their single RabbitMQ with a sharded architecture. Each guild (server) gets its own event channel. A gateway service maps guilds to shards. Messages within a guild never cross shards.

```text
Before (monolithic):
  All guilds -> Single RabbitMQ -> Single consumer pool

After (sharded):
  Guild 1-1000 -> Shard 1 (Kafka partition 0)
  Guild 1001-2000 -> Shard 2 (Kafka partition 1)
  Guild 2001-3000 -> Shard 3 (Kafka partition 2)
  ...
```

Each shard is independent. A traffic spike in one guild does not affect other guilds. Each shard can be scaled independently by adding consumer instances.

### Lesson: Isolate by Tenant

For multi-tenant pub-sub, the isolation boundary matters. Discord's approach of one event stream per guild (rather than one event stream per event type) means a noisy neighbor affects only its own guild's users, not the entire platform.

## Open Source Projects

The open-source ecosystem offers several production-grade pub-sub systems.

### Apache Kafka

The dominant pub-sub system for event streaming. Strengths: high throughput (2M+ msg/s per cluster), durable log-based storage, strong ordering within partitions, ecosystem (Kafka Connect, Kafka Streams, ksqlDB). Weaknesses: complex operations (ZooKeeper dependency, or KRaft in newer versions), high memory footprint, not ideal for small deployments.

### RabbitMQ

The most popular general-purpose message broker. Strengths: mature, extensive protocol support (AMQP, MQTT, STOMP), flexible routing (direct, topic, headers, fan-out exchanges), easy to operate. Weaknesses: throughput ceiling around 50K msg/s per node, persistence adds latency, no native log-based storage.

### NATS

A lightweight, high-performance pub-sub system. Strengths: 10M+ msg/s, very low latency (sub-millisecond), small footprint (20 MB binary), built-in clustering. Weaknesses: limited persistence, smaller ecosystem, fewer integration options.

### MQTT Brokers (Mosquitto, EMQX, HiveMQ)

Purpose-built for IoT scenarios. Strengths: ultra-compact wire protocol (2-byte minimum header), QoS levels (0, 1, 2), retained messages for new subscribers, MQTT 5.0 features. Weaknesses: topic-based only (no content-based filtering), limited to IoT use patterns.

### Apache Pulsar

A newer entrant with a unique architecture. Separates storage (BookKeeper) from serving (Brokers). Strengths: multi-tenancy, geo-replication built-in, tiered storage (hot + cold), low latency. Weaknesses: smaller ecosystem than Kafka, more complex operation.

## Cloud Managed Services

Most cloud providers offer managed pub-sub services. These abstract cluster management and provide auto-scaling.

### Amazon Simple Notification Service (SNS) + Simple Queue Service (SQS)

SNS provides pub-sub topics. SQS provides durable queues. Together, they form a pub-sub-with-queue pattern.

```python
# AWS: SNS publishes, SQS subscribes
import boto3

sns = boto3.client("sns")
sqs = boto3.client("sqs")

# Create topic and queue
topic = sns.create_topic(Name="orders-created")
queue = sqs.create_queue(QueueName="email-service-queue")

# Subscribe queue to topic
queue_arn = queue["QueueArn"]
topic_arn = topic["TopicArn"]
sns.subscribe(TopicArn=topic_arn,
              Protocol="sqs",
              Endpoint=queue_arn)

# Publisher sends to SNS
sns.publish(TopicArn=topic_arn, Message=json.dumps(order))

# Subscriber receives from SQS
messages = sqs.receive_message(QueueUrl=queue["QueueUrl"])
```

### Google Cloud Pub/Sub

Fully managed, global. Handles 1 GB/s per region. Provides push (HTTP/HTTPS) and pull delivery. Supports exactly-once delivery per message within a region.

### Azure Event Grid + Event Hubs

Event Grid is a pub-sub event router for lightweight events (up to 1 MB, 10M events/second). Event Hubs is a Kafka-compatible event streaming platform for high-throughput scenarios (up to 20 MB/s per throughput unit, scalable to GB/s).

## Cross-Cutting Lessons from Production Deployments

### Lesson 1: Monitoring the Lag Metric Is Non-Negotiable

Every production failure in these case studies had a precursor: growing consumer lag. Netflix monitors consumer lag at per-partition granularity with a 1-minute resolution. Uber alerts when any partition's lag exceeds 10,000 messages. Discord's meltdown was visible as growing queue depth for 20 minutes before the crash.

If you monitor only one metric, monitor consumer lag. Alert before it reaches a critical level.

### Lesson 2: Partitioning Strategy Determines Scaling Ceiling

The number of partitions in a topic is the single most important scaling decision. Too few partitions limits parallelism (maximum consumers that can process in parallel). Too many partitions increases overhead (metadata size, leader elections).

Rule of thumb: start with 3-5 partitions per broker node, tune based on observed throughput. For a 10-node cluster, 30-50 partitions per high-traffic topic.

### Lesson 3: Schema Evolution Requires Process, Not Just Technology

Schema registries catch incompatible changes at publish time, but they do not prevent an analytics subscriber from breaking when a new optional field appears with unexpected semantics. All four companies use a schema review process: breaking changes require a new major version with dual publishing during migration.

### Lesson 4: Test Failure Modes Before They Happen

Netflix runs Chaos Kong (simulates AWS region failure) quarterly. Uber's Simian Army kills Kafka brokers randomly in staging. Discord runs load tests at 2x peak traffic. Cloudflare has a "production readiness review" for every new topic that includes a failure mode analysis.

Without failure testing, you discover your recovery strategy during an outage вҖ” the worst possible time.
