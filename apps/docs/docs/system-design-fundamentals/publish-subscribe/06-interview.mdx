## Question Types

Pub-sub appears in system design interviews in three forms.

### Form 1: Design a Pub-Sub System

The interviewer asks you to design a pub-sub system from scratch. This is common at infrastructure companies (Confluent, Redpanda, Redis) and at general tech companies for senior+ roles.

**Examples**:
- "Design a message queue like Kafka or RabbitMQ"
- "Design a pub-sub system that can handle 1 million messages per second"
- "Design a distributed event bus for a microservices architecture"

### Form 2: Use Pub-Sub in a Broader Design

The interviewer asks you to design a larger system. Pub-sub naturally appears as the integration pattern between components.

**Examples**:
- "Design YouTube. How do you notify the encoding service when a new video is uploaded?"
- "Design Uber. How do you route trip events between dispatch, pricing, and payment?"
- "Design an e-commerce platform. How do you handle order fulfillment workflows?"

### Form 3: Compare and Contrast

The interviewer asks you to compare pub-sub with alternatives.

**Examples**:
- "Pub-sub versus message queue — when do you use each?"
- "Kafka versus RabbitMQ — when is each appropriate?"
- "Push versus pull delivery — what are the trade-offs?"

## Expectations by Level

Interviewers assess different depths of understanding depending on the role level.

### Entry Level (0-2 years)

You should be able to:
- Define publisher, subscriber, topic, broker in one sentence each
- Draw a basic pub-sub diagram with boxes and arrows
- Explain the difference between pub-sub and point-to-point
- Describe at-least-once delivery
- Give a real-world example (YouTube uploads -> encoding notification)

**Sample explanation**:
"Pub-sub is an asynchronous messaging pattern where a publisher sends messages to a named topic without knowing which subscribers receive them. A broker manages the topic and fans out each message to all subscribers. This decouples the publisher from subscribers — the publisher does not wait for subscriber processing, and subscribers can be added or removed without changing the publisher."

### Mid Level (3-5 years)

You should be able to:
- Explain the internal architecture of a broker (topic tree, subscription index, routing table)
- Discuss delivery guarantees (at-most-once, at-least-once, exactly-once) and implementation trade-offs
- Describe partitioning and ordering (Kafka partitions, consumer groups)
- Compare push vs pull delivery with specific use cases
- Sketch a clustered broker deployment with replication
- Discuss failure modes and mitigations

**Sample explanation**:
"For a topic with high throughput, I would partition it — say 12 partitions. Each partition is an ordered log. I distribute partitions across 3 broker nodes, each node handling 4 partitions with 3x replication. Producers hash message keys to partitions to preserve per-key ordering. A consumer group with up to 12 members can process each partition in parallel. If a broker fails, the leader for its partitions fails over to a replica, causing a 5-15 second pause while the cluster re-elects."

### Senior Level (6+ years)

You should be able to:
- Design a multi-region pub-sub deployment with cross-region replication
- Discuss CAP trade-offs specific to pub-sub (partition healing, split-brain)
- Design for exactly-once delivery with idempotent producers and transactional consumers
- Compare Kafka, Pulsar, NATS, and RabbitMQ architectures in depth
- Discuss operational concerns: quota enforcement, auto-scaling, rebalancing, schema evolution
- Provide concrete numbers: throughput, latency percentiles, resource requirements
- Explain the transaction outbox pattern and dual-write problem

### Staff+ Level (10+ years)

You should be able to:
- Design a global pub-sub system spanning 10+ regions with active-active replication
- Discuss CRDTs and conflict resolution for concurrent events
- Design custom storage engines optimized for pub-sub workloads (log-structured, tiered)
- Articulate the economic model: cost per message, storage cost per GB, bandwidth cost
- Discuss when NOT to use pub-sub for non-obvious reasons (backpressure, ordering guarantees, operational complexity)
- Design a migration strategy: moving from a monolith to event-driven pub-sub without downtime

## Phrase Bank for Interview Answers

Use these phrases to demonstrate depth.

### On Decoupling

"Pub-sub breaks the temporal coupling between producers and consumers. The publisher does not wait for subscribers to process the message. It also breaks the location coupling — publishers and subscribers discover each other through the broker, not through hardcoded addresses."

### On the Fan-Out Cost

"Fan-out is both the power and the cost of pub-sub. A single published message generates N deliveries where N is the number of subscribers. At 1,000 subscribers and 50,000 messages/second, the broker handles 50 million deliveries/second. The broker's network bandwidth must be sized for egress, not just ingress."

### On Ordering

"Pub-sub systems guarantee ordering within a partition, not across partitions. If you need global ordering, you use a single partition — which limits throughput to what a single consumer can process. For most systems, per-key ordering is sufficient: all events for a given user or order go to the same partition."

### On Delivery Guarantees

"At-least-once is the practical default for most pub-sub systems. The subscriber must handle duplicates through idempotent processing. Exactly-once is available in Kafka through transactions, but it reduces throughput by roughly 20% and adds complexity. For most use cases, at-least-once with idempotent consumers is the right trade-off."

### On Durability vs Performance

"Every durability level has a cost. In-memory only gives you sub-millisecond publish latency but loses messages on crash. Writing to disk on the leader adds 2-5ms. Writing to the leader plus 2 followers adds 5-15ms. We tune this per topic: critical events use acks=all with 3 replicas, high-volume metrics use acks=1 or acks=0."

### On Backpressure

"The broker must protect itself from publishers that produce faster than subscribers can consume. Without backpressure, the broker's memory grows unbounded. Solutions include per-client rate limits, topic-level quotas, and flow control signals. In Kafka, the `max.in.flight.requests.per.connection` parameter limits how many unacknowledged requests a producer can have in flight."

## Red Flags

These statements indicate a superficial understanding. Avoid them.

**"We just use pub-sub everywhere"**: pub-sub is not a universal solution. Overusing it adds latency, operational complexity, and eventual consistency challenges. Know when to use simpler patterns like direct HTTP calls or queues.

**"Messages are always delivered in order"**: false. Ordering is per-partition, and only if no failures cause leader re-election. Failures can cause reordering within a partition. A consumer that crashes and reconnects may receive duplicate messages out of order.

**"Exactly-once delivery is the default"**: false. Most pub-sub systems default to at-least-once. Exactly-once is a special mode with significant trade-offs. Claiming it is the default shows inexperience.

**"The broker never loses messages"**: false under failure conditions. A broker can lose messages if the leader crashes before replicating, if disk is full, or if unclean leader election is enabled during a split-brain scenario.

**"We do not need a dead letter queue because we never fail"**: every production system has failures. Messages with invalid payloads, downstream service errors, and transient timeouts all cause delivery failures. Without a DLQ, these messages are silently dropped or retried forever.

## Interview Drills

Practice these drills to prepare for the interview.

### Drill 1: Whiteboard a Pub-Sub System (10 minutes)

Draw a broker cluster with 3 nodes. Show:
1. Two publishers sending to partition 0
2. The partition leader on Brokers 1 and its followers on Brokers 2, 3
3. Three subscribers in a consumer group, each reading different partitions
4. The ZooKeeper node storing cluster metadata

Explain how a message flows from publisher to subscriber, highlighting the replication step.

### Drill 2: Handle a Failure Scenario (5 minutes)

**Scenario**: A partition leader on Broker 1 crashes. Describe exactly what happens.

**Model answer**:
"The ZooKeeper session for Broker 1 times out after 10 seconds. ZooKeeper notifies Broker 2 and Broker 3 that the leader for partition 0 is unavailable. The controller (one of the brokers elected to manage cluster state) initiates leader election. It selects Broker 2 as the new leader because Broker 2 is in the ISR (in-sync replica set) and has the highest offset. Broker 2 becomes the new leader for partition 0. Producers connected to the old leader receive a `LeaderNotAvailableException` and retry. They discover the new leader through a metadata refresh. Total downtime: 10-20 seconds, during which producers cannot write to partition 0."

### Drill 3: Design for Scale (10 minutes)

**Scenario**: You have 100 publishers sending to a single topic. Each publishes 1,000 messages/second. Each message is 5 KB. You have 50 subscribers. Design the infrastructure.

**Model answer**:
"Incoming: 100 x 1,000 = 100,000 msg/s = 500 MB/s ingress. Outgoing: 100,000 x 50 subscribers = 5,000,000 deliverable msg/s (but actually each subscriber receives their own copy, so egress is 100,000 msg/s at 5 KB = 500 MB/s per subscriber, 25 GB/s total — wait, that's wrong. Each subscriber receives 100,000 msg/s, each 5 KB, so each egress stream is 500 MB/s. 50 subscribers = 25 GB/s total egress. That requires 25 Gbps per subscriber connection, aggregated, or 25 Gbps x 50 = 1.25 Tbps total network bandwidth.

To reduce this, I would use batching: batch messages into 1 MB chunks at the broker, delivering 500 batches/second instead of 100,000 individual messages. This reduces per-message overhead by a factor of 200.

For the broker cluster, with replication factor 3: total storage writes = 500 MB/s x 3 = 1.5 GB/s. With 10 broker nodes, each handles 150 MB/s write throughput — within a single NVMe drive's sequential write capacity (~500 MB/s). Each node needs 200 Gbps network connectivity (25 Gbps for ingress shares + replication traffic)."

### Drill 4: Compare and Contrast (3 minutes)

**Scenario**: Compare Kafka and RabbitMQ for an order processing system.

**Model answer**:
"Kafka offers higher throughput (1M+ msg/s per cluster vs 50K per RabbitMQ node) and stronger ordering guarantees (per-partition ordering). Kafka's log-based storage enables replay and batch consumption, which is useful for analytics. RabbitMQ offers more flexible routing (topic exchanges with bindings, header exchanges), easier setup for small deployments, and better support for transient messages with low latency.

For an order processing system processing 50,000 orders/second, I would use Kafka for the core event stream (orders, payments, inventory) because of throughput requirements. I would use RabbitMQ for internal notification delivery (email queue, SMS queue) where per-message routing and lower operational complexity matter."

## Practice Questions

### Question 1: Design a Notification Service

Design a system that sends email, SMS, and push notifications when a user performs an action. The system must handle 10,000 events/second. Each notification type has different delivery characteristics: email is best-effort, SMS must be at-least-once, push notifications must arrive within 100ms.

How would you structure this using pub-sub?

### Question 2: Design a Log Aggregation Pipeline

Design a system where 10,000 servers send logs to a central pub-sub system. Each server sends 1 MB of logs per minute. The logs must be searchable within 5 minutes of generation. Downstream consumers include a real-time alerting system, an analytics dashboard, and a cold storage archive.

How many topics? Partitions per topic? Retention? Storage sizing?

### Question 3: Design a Real-Time Ride Matching System

Design the pub-sub infrastructure for a ride-hailing app. Riders publish ride requests. Nearby drivers subscribe to requests. When a driver accepts, a series of events follows: driver location updates (every 2 seconds, 100,000 drivers), ride status changes, payments.

How do you handle the geography-based filtering (drivers only receive requests near them)? What ordering guarantees do you need? How do you handle a surge in requests during peak hours?
