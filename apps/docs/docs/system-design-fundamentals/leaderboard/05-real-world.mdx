## Case Study 1: Supercell -- Clash Royale Trophy Leaderboard

Supercell operates Clash Royale, a real-time multiplayer game with over 100 million registered players and 500,000 concurrent players during peak hours. The game uses a trophy-based ranking system where players gain or lose trophies based on match outcomes. The leaderboard must handle 500,000 concurrent score updates during peak hours while maintaining sub-20ms read latency.

### The Challenge

The original leaderboard used a MySQL database with a `players` table containing a `trophies` column. The top-200 query used `ORDER BY trophies DESC LIMIT 200`. As the player base grew to 10 million active players, the query took 4.2 seconds. The rank query for a player at the 50th percentile required scanning 5 million rows. The database CPU hit 95 percent during evening peak hours, and the connection pool of 500 connections was exhausted within 30 minutes of peak play.

Supercell's engineering team evaluated three approaches: vertical scaling of the database to a 64-core server, migration to a sharded MySQL setup with 8 shards, and adoption of Redis sorted sets. They chose Redis sorted sets because the complexity was lower than sharding MySQL, the performance was 200x better for rank queries, and the operational overhead was significantly less.

### Key Decisions

**Season-based leaderboard reset.** Every two months, the season resets. Players above 4,000 trophies reset to 4,000. Players below 4,000 stay at their current trophy count. This prevents the leaderboard from becoming static and gives new players a chance to compete for the top positions each season.

**Trophy inflation control.** The system uses a zero-sum trophy model. When a player wins, they gain trophies from the opponent they defeat. This keeps the total trophy count in the system constant and prevents inflation over time. Without this control, the average trophy count would increase over time as players accumulate trophies, making it harder for new players to compete.

**Top-200 caching with adaptive refresh.** The top-200 leaderboard is cached in a dedicated Redis cache instance and refreshed every 10 seconds. During the last 24 hours of a season, when competition for top positions intensifies, the cache refresh interval drops to 1 second. This adaptive approach balances accuracy with cost -- 10-second cache during normal play is sufficient, and 1-second cache during the competitive final hours provides the accuracy that top players expect.

### Results

Supercell reduced leaderboard read latency from 4.2 seconds to under 15 milliseconds. The database CPU dropped from 95 percent to 25 percent because the database was no longer serving leaderboard queries. The system scaled from supporting 1 million to 100 million registered players without a major redesign of the leaderboard tier.

## Case Study 2: Duolingo -- Streak Leaderboard and League System

Duolingo, the language learning platform with 80 million monthly active users, uses a leaderboard to drive daily engagement. The leaderboard tracks weekly experience points (XP) and groups learners into leagues of 30 players each. The leaderboard is a core engagement driver -- Duolingo reports that users who participate in leagues have 30 percent higher weekly active usage than those who do not.

### The Challenge

Duolingo's leaderboard had a unique product requirement: it needed to be motivational rather than discouraging. A global leaderboard with 80 million users would demotivate beginners who could never reach the top 100. The solution was league-based grouping -- each league contains exactly 30 players of similar skill level, and players compete for the top 3 positions to advance to the next league tier.

At the system level, the challenge was managing 2.6 million league instances per week. Each league needed its own sorted set, and the system had to create, assign, update, and archive leagues efficiently.

### Implementation Details

Duolingo's leaderboard system uses one Redis sorted set per league per week. When a player earns XP, the system updates the player's league sorted set. The league assignment is computed at the start of each week based on the player's XP from the previous week.

```python
# Duolingo league assignment and score update
class DuolingoLeagueService:
    def __init__(self, redis_client, db_client):
        self.redis = redis_client
        self.db = db_client

    async def assign_league(self, player_id: str, previous_week_xp: int, current_tier: str):
        # Determine target tier based on XP
        tier = self._determine_tier(previous_week_xp, current_tier)

        # Find a league with an open slot in this tier
        league_id = await self._find_or_create_league(tier)

        # Add player to the league
        league_key = f"league:{tier}:{league_id}:week:{current_week()}"
        await self.redis.zadd(league_key, {player_id: 0})

        return {"tier": tier, "league_id": league_id}

    async def update_xp(self, player_id: str, xp_delta: int):
        # Get current league info from player metadata
        league_info = await self.db.fetchrow(
            "SELECT tier, league_id FROM player_leagues WHERE player_id = $1",
            player_id
        )
        if not league_info:
            return  # Player not assigned to a league yet

        league_key = f"league:{league_info['tier']}:{league_info['league_id']}:week:{current_week()}"
        await self.redis.zincrby(league_key, xp_delta, player_id)

    async def get_league_standings(self, player_id: str):
        league_info = await self.db.fetchrow(
            "SELECT tier, league_id FROM player_leagues WHERE player_id = $1",
            player_id
        )
        if not league_info:
            return None

        league_key = f"league:{league_info['tier']}:{league_info['league_id']}:week:{current_week()}"
        rank = await self.redis.zrevrank(league_key, player_id)
        total = await self.redis.zcard(league_key)
        top_3 = await self.redis.zrevrange(league_key, 0, 2, withscores=True)

        return {
            "rank": rank + 1 if rank is not None else None,
            "total": total,
            "top_3": [{"player_id": p.decode(), "score": int(s)} for p, s in top_3]
        }

    def _determine_tier(self, weekly_xp: int, current_tier: str) -> str:
        if weekly_xp >= 5000:
            return "diamond"
        elif weekly_xp >= 3000:
            return "pearl"
        elif weekly_xp >= 2000:
            return "ruby"
        elif weekly_xp >= 1000:
            return "sapphire"
        else:
            return "bronze"
```

### Lessons from Duolingo

**Grouping prevents discouragement.** New players compete against peers at their level, not against veteran players with years of XP. This dramatically increases retention -- Duolingo reports 30 percent higher weekly active usage among users who participate in leagues.

**Weekly reset drives habit formation.** The Monday reset creates a natural cycle. Players return each week to compete, and the clean slate prevents the leaderboard from becoming stale. The predictable schedule also simplifies infrastructure -- the reset happens at a known time, and the team can prepare for the load spike.

**League assignment must be fair.** Duolingo had to iterate on the league assignment algorithm to prevent sandbagging -- players who deliberately stay in lower leagues to win easily. The fix was to base league assignment on the previous week's XP, not the current week's XP. This means that a player who deliberately scores low in Bronze league gets assigned to Bronze again next week, rather than being promoted for winning Bronze.

**Small sorted sets are efficient.** Each league has exactly 30 entries. Rank queries are O(log 30), which is essentially O(1) in practice. The memory overhead per league is approximately 2 KB. For 2.6 million leagues, the total memory is approximately 5 GB, which fits comfortably in a single Redis instance.

## Case Study 3: Strava -- Segment Leaderboard

Strava, the fitness tracking platform with 100 million users, maintains leaderboards for 2 million running and cycling segments worldwide. Each segment is a stretch of road or trail, and athletes compete for the fastest time. The leaderboard must support filtering by gender, age group, weight class, and date range. The write volume is 35 million activity uploads per week, and each activity may match multiple segments.

### The Challenge

Strava's segment leaderboard is massively multi-dimensional. A single segment may have 50,000 to 500,000 attempts. The leaderboard must support filtering by gender (2 values), age group (5 values), weight class (3 values), and date range. Storing a separate sorted set for each filter combination would require 2 x 5 x 3 = 30 sorted sets per segment, which is 60 million sorted sets for 2 million segments -- completely impractical.

The platform also faces a hot segment problem. The top 1,000 segments (famous climbs like Alpe d'Huez and local legends in major cities) account for 40 percent of all leaderboard views. These segments need special treatment for performance.

### The Filtering Solution

The naive approach of storing a separate sorted set for each filter combination would explode the key space. Strava's solution is to store a single sorted set per segment with the raw time, and apply filters in memory at read time.

```python
# Strava's in-memory filtering approach
class SegmentLeaderboardService:
    def __init__(self, redis_client, athlete_cache):
        self.redis = redis_client
        self.athlete_cache = athlete_cache

    async def get_filtered_leaderboard(self, segment_id: int, filters: dict):
        cache_key = f"segment_lb:{segment_id}:{self._hash_filters(filters)}"

        # Check pre-computed cache
        cached = await self.redis.get(cache_key)
        if cached:
            return json.loads(cached)

        # Read all entries from segment sorted set
        segment_key = f"segment:{segment_id}"
        all_entries = await self.redis.zrevrange(
            segment_key, 0, -1, withscores=True
        )

        # Apply filters in memory
        filtered = []
        for athlete_id_bytes, time_seconds in all_entries:
            athlete_id = athlete_id_bytes.decode()
            athlete = await self.athlete_cache.get(athlete_id)

            if not athlete:
                continue

            if self._matches_filters(athlete, filters):
                filtered.append((athlete_id, int(time_seconds)))

        # Sort by time ascending (lower is better for racing)
        filtered.sort(key=lambda x: x[1])

        # Format with ranks
        result = [
            {"rank": i + 1, "athlete_id": aid, "time_seconds": t}
            for i, (aid, t) in enumerate(filtered)
        ]

        # Cache for 5 minutes
        await self.redis.setex(cache_key, 300, json.dumps(result))
        return result

    def _matches_filters(self, athlete: dict, filters: dict) -> bool:
        for key, value in filters.items():
            if key == "gender" and athlete.get("gender") != value:
                return False
            if key == "age_group" and athlete.get("age_group") != value:
                return False
            if key == "weight_class" and athlete.get("weight_class") != value:
                return False
        return True

    def _hash_filters(self, filters: dict) -> str:
        return hashlib.md5(json.dumps(filters, sort_keys=True).encode()).hexdigest()[:8]
```

### Lessons from Strava

**Filtered leaderboards are a storage optimization challenge.** Storing a sorted set per filter combination is not feasible at scale. In-memory filtering on a small sorted set is the pragmatic solution for most segments.

**Hot segments need special treatment.** The top 1,000 segments account for 40 percent of all leaderboard views. These segments get pre-computed filtered leaderboards with aggressive caching. The long tail of segments is served from the single sorted set with in-memory filtering.

**The athlete metadata cache is critical.** For each entry in the sorted set, the system must fetch the athlete's metadata to apply filters. Without caching athlete profiles, each leaderboard read would require 500 to 150,000 database queries. Strava caches athlete profiles in a separate Redis instance with a 24-hour TTL.

**Leaderboard accuracy for racing segments is paramount.** A cyclist who rides a segment 0.1 seconds faster than another athlete deserves the higher rank. Strava uses the exact time from GPS data as a double-precision float, not a rounded integer, to ensure precise ordering.

## Case Study 4: GitHub -- Trending Repositories Leaderboard

GitHub's trending repositories page shows the most popular repositories over a rolling time window. While not a traditional game leaderboard, the trending algorithm uses a leaderboard pattern to rank 200 million repositories by a composite score derived from stars, forks, and other engagement signals.

### The Challenge

GitHub must compute a trending score for 200 million repositories across multiple time windows (daily, weekly, monthly). The score is a composite metric that must resist spam and manipulation. The leaderboard must be updated every few hours, not in real time, because the trending score changes slowly. The system serves 50 million trending page views per day.

### The Trending Score Formula

GitHub's trending score combines multiple signals with a time decay factor. The formula is designed to surface repositories that are gaining traction right now, not repositories that were popular years ago. Each signal has a weight that reflects its importance: stars indicate interest, forks indicate active use, issues indicate community engagement, and pull requests indicate development activity.

```python
# GitHub trending score computation with anti-spam
def compute_trending_score(repo_id: str, events: list, current_time: float) -> float:
    score = 0.0
    signal_counts = {"star": 0, "fork": 0, "issue": 0, "pr": 0}

    for event in events:
        days_ago = (current_time - event["timestamp"]) / 86400.0
        decay = math.exp(-days_ago / 7.0)  # Half-life of 7 days

        if event["type"] == "star":
            score += 1.0 * decay
            signal_counts["star"] += 1
        elif event["type"] == "fork":
            score += 0.5 * decay
            signal_counts["fork"] += 1
        elif event["type"] == "issue_created":
            score += 0.3 * decay
            signal_counts["issue"] += 1
        elif event["type"] == "pull_request":
            score += 0.2 * decay
            signal_counts["pr"] += 1

    # Anti-spam: penalize repositories with unusual patterns
    if is_spam_pattern(repo_id, signal_counts):
        score *= 0.1

    return round(score, 1)


def is_spam_pattern(repo_id: str, counts: dict) -> bool:
    # Burst detection: if 90% of stars arrived in 1 hour
    if counts["star"] > 100 and is_burst_activity(repo_id):
        return True
    # Bot account cluster: if stars come from recently created accounts
    if has_bot_cluster(repo_id):
        return True
    return False
```

### Lessons from GitHub

**Batch processing is sufficient when the leaderboard updates slowly.** GitHub's trending page changes every few hours, not every second. Real-time infrastructure would be wasted. A Spark job that runs every 4 hours is simpler and more cost-effective than a real-time streaming pipeline.

**Score decay is essential for relevance.** Without the exponential decay factor with a 7-day half-life, trending repositories would be dominated by all-time popular projects like React or TensorFlow. The decay ensures that only recent activity matters for the trending score.

**Anti-spam is a leaderboard concern.** GitHub's trending page is a target for spam and manipulation. The system must detect unusual activity patterns -- such as a repository receiving 1,000 stars in 1 hour from newly created accounts -- and penalize the score accordingly.

**The CDN is the most cost-effective cache.** With 99.7 percent cache hit rate on the CDN, only 0.3 percent of requests reach the origin server. This makes the infrastructure cost for the trending page remarkably low -- under $1,000 per month for 50 million daily views.

## Case Study 5: Akamai -- Real-Time Bidding Leaderboard

Akamai operates a real-time bidding (RTB) platform for programmatic advertising. The platform processes 10 million bid requests per second. Each bid request must be evaluated and responded to within 100 milliseconds. The leaderboard tracks which bidders are winning the most auctions, their win rates, and average response times.

### The Extreme Scale Challenge

The RTB leaderboard operates at extreme scale -- 10 million writes per second across 1,000 edge servers. Traditional Redis sorted sets cannot handle 10 million writes per second on a single instance. The leaderboard is used for operational monitoring rather than end-user display, so the accuracy requirement is lower. A 1 percent error in bid counts is acceptable.

Akamai's solution uses a two-tier approach: a lightweight in-memory counter per server that aggregates locally, and a periodic flush to a central Redis cluster. This reduces the write rate to the central Redis from 10 million per second to 200 per second.

```python
# Akamai two-tier aggregation for RTB leaderboard
class LocalBidCounter:
    """Per-server local counter that flushes to central Redis every 5 seconds."""

    def __init__(self, server_id: str):
        self.server_id = server_id
        self.wins = {}
        self.bids = {}
        self.total_time = {}
        self.last_flush = 0.0
        self.flush_interval = 5.0  # seconds

    def record_bid(self, bidder_id: str, won: bool, response_time_ms: int):
        """Called on every bid request -- runs in microseconds."""
        self.bids[bidder_id] = self.bids.get(bidder_id, 0) + 1
        if won:
            self.wins[bidder_id] = self.wins.get(bidder_id, 0) + 1
        self.total_time[bidder_id] = self.total_time.get(bidder_id, 0) + response_time_ms

    async def maybe_flush(self):
        """Flush accumulated counters to central Redis every 5 seconds."""
        now = time.time()
        if now - self.last_flush < self.flush_interval:
            return
        self.last_flush = now

        if not self.bids:
            return

        # Bulk flush using Redis pipeline
        pipeline = central_redis.pipeline()
        for bidder_id, bid_count in self.bids.items():
            key = f"rtb:{bidder_id}"
            pipeline.hincrby(key, "bids", bid_count)
            if bidder_id in self.wins:
                pipeline.hincrby(key, "wins", self.wins[bidder_id])
            if bidder_id in self.total_time:
                pipeline.hincrby(key, "total_time_ms", self.total_time[bidder_id])
        await pipeline.execute()

        # Clear local counters
        self.bids.clear()
        self.wins.clear()
        self.total_time.clear()
```

A separate aggregation job runs every 60 seconds on the central Redis data to compute win rates and update a sorted set with the top 100 bidders by win rate.

### Lessons from Akamai

**At extreme scale, local aggregation before central aggregation is necessary.** The local counter reduces the write load on the central Redis cluster by a factor equal to the number of servers. With 1,000 servers flushing every 5 seconds, the central Redis receives 200 writes per second instead of 10 million.

**Operational leaderboards have different requirements than user-facing ones.** The accuracy requirement is lower -- a 1 percent error in bid counts is acceptable. The system optimizes for throughput over precision.

**Percentile-based leaderboards are more useful than rank-based ones.** Bidders care about their win rate percentile, not their exact rank. Akamai uses a t-digest for percentile estimation across all bidders and a separate sorted set for the top 100 by win rate.

## Case Study 6: Discord -- Server Member Leaderboard

Discord, the communication platform with 150 million monthly active users, uses a leaderboard pattern to rank servers by member count, activity level, and growth rate. The platform has 19 million active servers, and each server has its own member count that changes in real time as users join and leave.

### The Challenge

Discord's leaderboard challenge is the sheer number of sorted sets required. With 19 million servers, each needing a sorted set for member count ranking, the total number of sorted sets can reach millions. However, each sorted set is tiny -- a server typically has between 2 and 1,000 members, so the sorted set size is small.

The platform uses Redis sorted sets for the global server ranking (by member count) and for per-server member rankings (by activity level). The global server ranking is used for the discovery page, which shows the top servers by member count.

### Key Decisions

**Separate Redis instances for global and per-server leaderboards.** The global server ranking is stored in a dedicated Redis instance because it is accessed frequently by the discovery page. Per-server member rankings are stored in a separate cluster because they are accessed only when a user opens the member list for a specific server.

**Leaderboard updates are batched.** Member join and leave events are accumulated in a message queue and processed in batches of 1,000 every 5 seconds. This reduces the write rate to the sorted set by 99.8 percent compared to processing each event individually.

**The discovery page uses a cached top-100 list.** The top 100 servers by member count change slowly -- the list typically shifts by only 1-2 positions per day. The cached list is refreshed every 15 minutes, which is sufficient for the discovery page.

### Lessons from Discord

**Batch processing of leaderboard updates is essential for high-event-volume systems.** Discord processes millions of join and leave events per hour. Processing each event individually would overwhelm the Redis sorted sets. Batching reduces the write rate by orders of magnitude.

**Different leaderboard use cases have different latency requirements.** The discovery page can tolerate 15-minute staleness. The server member list needs near-real-time accuracy. Using separate infrastructure for each use case optimizes cost and performance.

Redis provides the most widely used open source leaderboard primitive. The sorted set data structure supports all four leaderboard operations natively. You do not need any additional software beyond Redis itself.

```bash

ZCARD leaderboard:daily:2026-07-03

# Get the score of a specific player
ZSCORE leaderboard:daily:2026-07-03 player_42

# Remove a player from the leaderboard
ZREM leaderboard:daily:2026-07-03 player_42

# Count players with score in a range
```

Redis sorted sets are the most battle-tested open source leaderboard implementation. Companies including Supercell, Duolingo, Strava, Discord, and Stack Overflow use them in production.

## Lessons Learned

### Lesson 1: Always Set TTL on Leaderboard Keys

The most common production incident is an unbounded sorted set that consumes all Redis memory. Always set a TTL on every leaderboard key, even for all-time leaderboards. For all-time leaderboards, implement a background job that archives inactive players to cold storage.

```python
# Background archival job for all-time leaderboard
async def archive_inactive_players(cutoff_days: int = 90):
    key = "leaderboard:all_time"
    all_players = await redis.zrange(key, 0, -1, withscores=True)
    cutoff_time = time.time() - (cutoff_days * 86400)

    to_archive = []
    for player_id, score in all_players:
        last_active = await redis.hget(f"player:{player_id}", "last_active")
        if last_active and float(last_active) < cutoff_time:
            to_archive.append(player_id)

    if to_archive:
        await redis.zrem(key, *to_archive)
        logger.info(f"Archived {len(to_archive)} inactive players from leaderboard")
```

### Lesson 2: Conditional Cache Invalidation Is Critical

Without conditional cache invalidation, the top-N cache invalidates on every score write. For a game with 10 million players submitting 1 score per minute, that is 166,000 cache invalidations per second. With conditional invalidation, only scores in the top percentile trigger cache rebuilds -- typically 100-200 per second.

### Lesson 3: Shard Before You Need To

Sharding a leaderboard after it has grown to 10 million players is a painful migration that requires data rebalancing, application changes, and extensive testing. Shard from the start, even if you only have 100,000 players. Use 16 logical shards from the beginning. The overhead of 16 Redis sorted sets is negligible, and the migration pain later is avoided.

### Lesson 4: Monitor the Gap Between Cache and Sorted Set

The cache hit rate tells you whether the cache is working, but it does not tell you whether the cache is correct. Monitor the gap -- compare the cached top-100 with the live top-100. A growing gap indicates that the cache is not being invalidated when it should be.

```python
# Monitor cache accuracy
async def monitor_cache_gap(window_key: str):
    cache_key = f"cache:top100:{window_key}:0"
    cached = await cache.get(cache_key)
    if not cached:
        return

    live_top = await redis.zrevrange(
        f"leaderboard:daily:{window_key}", 0, 99, withscores=True
    )

    cached_ids = {e["player_id"] for e in cached["entries"]}
    live_ids = {p.decode() for p, _ in live_top}

    overlap = len(cached_ids & live_ids)
    gap = 100 - overlap

    if gap > 5:
        await alert(
            f"Cache gap: {gap} percent of top-100 differ. "
            f"Cache invalidation may be malfunctioning."
        )
```

### Lesson 5: Design for the Reset

Leaderboard resets are the most fragile operation in the system. When a weekly leaderboard resets, the system must atomically switch from the old key to the new key. A staggered reset -- where the new key is created 5 minutes before the old key expires -- prevents a thundering herd of reads hitting an empty sorted set.

### Lesson 6: Tie-Breaking Must Be Deterministic

When two players have the same score, the tie-breaker must be deterministic and transparent. The most common approach is first-to-reach-the-score wins, which requires storing the achievement timestamp alongside the score. Redis supports this by using a composite score that encodes both the raw score and a timestamp.

### Lesson 7: Test with Realistic Data Volumes

A leaderboard that works perfectly with 10,000 players in development may fail catastrophically with 1 million players in production. The failure modes -- cache thrashing, hot shards, memory exhaustion -- only appear at scale. Load test with at least 10x your expected production volume. Use tools like `redis-benchmark` and `locust` to simulate realistic traffic patterns.

### Lesson 8: The Leaderboard Is a Product, Not Just a Feature

The leaderboard design affects player behavior. A leaderboard that resets weekly drives different engagement than one that resets monthly. A league-based leaderboard drives different behavior than a global leaderboard. The engineering decisions -- reset frequency, decay function, tie-breaking strategy, caching approach -- are product decisions that require collaboration between engineers, product managers, and data scientists.
