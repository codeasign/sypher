## Naive Implementation

The simplest observability implementation is logging every AI interaction to a file. Here is what that looks like:

```python
import json
import logging

logger = logging.getLogger("ai_observability")

def log_llm_call(prompt, response, duration_ms, tokens):
    log_entry = {
        "timestamp": datetime.now().isoformat(),
        "event": "llm_call",
        "prompt": prompt,
        "response": response,
        "duration_ms": duration_ms,
        "tokens": tokens
    }
    logger.info(json.dumps(log_entry))
```

This approach works for a demo with 100 requests per day. It fails catastrophically at scale. Here is what breaks:

**Disk full after 4 hours.** A single LLM call produces approximately 4 KB of log data (prompt text, response text, metadata). At 1,000 requests per minute, that is 4 MB per minute, 240 MB per hour, 5.8 GB per day. A 100 GB disk fills in 17 days. At 10,000 requests per minute, the disk fills in 2 days.

**No structured querying.** Log files contain JSON strings. To find all requests where the response included a specific phrase, you run `grep 'return policy' agent.log` across 5 GB of files. This takes 45 seconds on a fast SSD. You cannot aggregate, filter by model, or compute percentiles.

**No tracing.** Each log entry is independent. You cannot connect the LLM call to the retrieval call to the guardrail call. You cannot see the full request flow. When a user complains about a bad response, you cannot reconstruct what the system did.

**No alerting.** The logging approach is passive. Data is written to disk and never examined until someone manually searches for it. A quality drop at 3 PM stays undetected until someone reads the logs at 5 PM — or the next day.

**No cost tracking.** You cannot answer the question "How much did we spend on AI yesterday?" without writing a custom script that parses all log files and sums token counts.

This is why purpose-built observability platforms exist. They replace ad-hoc logging with structured pipelines that handle scale, provide querying, and enable alerting.

### Component Details

**OpenTelemetry Collectors.** Deployed as a DaemonSet (one collector per Kubernetes node). Each collector handles spans from all pods on its node. The collector validates, enriches, and forwards spans to Kafka. Configuration is managed via a ConfigMap that is version-controlled.

**Kafka.** Acts as a durable buffer between collectors and stream processors. Kafka absorbs traffic spikes without data loss. If the stream processors are down for 30 minutes (for a deployment), Kafka holds all spans and replays them when the processors return. The default retention of 72 hours means you can reprocess data if a bug is found in the evaluation pipeline.

**Stream Processors (Flink).** Three parallel pipelines consume from different Kafka topics:

1. The metrics pipeline computes aggregations (latency percentiles, token counts, cost) and writes time-series data to Mimir.
2. The trace pipeline reconstructs trace trees, applies sampling decisions, and writes traces to ClickHouse.
3. The evaluation pipeline runs 12 evaluators on each trace: factuality, relevance, toxicity, PII detection, sentiment, tone, length compliance, format compliance, instruction adherence, safety, bias, and hallucination. Each evaluator produces a score from 0 to 1.

**ClickHouse.** A columnar OLAP database optimized for analytics queries. Traces are stored in a MergeTree table partitioned by day and ordered by (service_name, trace_id, start_time). This ordering enables efficient queries by service and trace ID. ClickHouse achieves 1-5 second query times on 30-day trace data.

**Grafana.** Provides dashboards for real-time monitoring and a TraceQL-based explore interface for ad-hoc trace analysis. Each team gets a dedicated dashboard folder with pre-built panels.

### With LLM API Calls

The instrumentation intercepts calls to the LLM provider SDK. This is achieved via OpenTelemetry automatic instrumentation:

```python
from opentelemetry.instrumentation.openai import OpenAIInstrumentor

OpenAIInstrumentor().instrument()

# All OpenAI calls are now automatically traced
response = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What is your return policy?"}]
)
# The instrumentor captures: model, tokens, latency, response
```

The instrumentor wraps the SDK's `create` method. It creates a span before the API call, captures all request parameters, executes the call, captures response data, and closes the span. It also captures streaming responses: for streaming, it records time-to-first-token and total streaming duration.

### With Vector Databases

Integrating with vector databases requires explicit instrumentation of retrieval calls:

```python
from app.observability import tracer

class VectorDBClient:
    def query(self, embedding, top_k=5):
        with tracer.start_as_current_span("retrieval.query") as span:
            span.set_attribute("db.system", "pinecone")
            span.set_attribute("db.index", "product-docs-v2")
            span.set_attribute("db.top_k", top_k)

            start = time.monotonic()
            results = self._pinecone_client.query(embedding, top_k=top_k)
            duration_ms = (time.monotonic() - start) * 1000

            span.set_attribute("db.duration_ms", duration_ms)
            span.set_attribute("db.results_count", len(results))
            span.set_attribute("db.results_ids", [r.id for r in results])
            span.set_attribute("db.results_scores", [r.score for r in results])

            return results
```

The key metrics captured: query duration, number of results, result IDs, and similarity scores. These enable you to detect slow retrieval, empty results, or low-relevance results that might cause the LLM to hallucinate.

### With Guardrails

Guardrails evaluate model inputs and outputs. Each guardrail check becomes a span attached to the trace:

```python
def evaluate_response(trace_id, prompt, response):
    with tracer.start_as_current_span("guardrail.evaluate") as span:
        span.set_attribute("guardrail.name", "content_safety_v2")
        span.set_attribute("guardrail.input.text_length", len(prompt))
        span.set_attribute("guardrail.output.text_length", len(response))

        start = time.monotonic()
        toxicity_score = content_safety_classifier(response)
        factuality_score = factuality_evaluator(response, context_docs)
        duration_ms = (time.monotonic() - start) * 1000

        span.set_attribute("guardrail.toxicity_score", toxicity_score)
        span.set_attribute("guardrail.factuality_score", factuality_score)
        span.set_attribute("guardrail.duration_ms", duration_ms)

        if toxicity_score > 0.5:
            span.set_attribute("guardrail.action", "blocked")
            span.set_attribute("guardrail.reason", "toxicity threshold exceeded")
        else:
            span.set_attribute("guardrail.action", "allowed")

        return toxicity_score, factuality_score
```

### With CI/CD Pipelines

Observability data feeds into CI/CD through regression testing. Before deploying a new prompt template, the CI pipeline runs the template against an evaluation dataset of 500 test cases. The results are compared against the previous template's performance:

```yaml
# .github/workflows/prompt-regression.yml
jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Evaluate new prompt
        run: |
          python scripts/evaluate_prompt.py \
            --dataset ./tests/eval_dataset.json \
            --template ./prompts/v2.yaml \
            --output results.json
      - name: Compare with baseline
        run: |
          python scripts/compare_results.py \
            --baseline ./baselines/current.json \
            --candidate results.json \
            --threshold 0.03  # Max 3% quality regression
      - name: Publish results to observability
        run: |
          curl -X POST https://observability.company.com/api/v1/evaluations \
            -H "Authorization: Bearer ${{ secrets.OBSERVABILITY_TOKEN }}" \
            -d @results.json
```

If the new prompt degrades quality by more than 3 percent, the CI pipeline fails and blocks deployment. The evaluation results are published to the observability platform for historical tracking.

### With Cost Tracking

Each LLM call span includes a cost attribute. The observability pipeline aggregates costs by model, service, department, and user. This enables real-time cost dashboards:

```sql
-- ClickHouse query: daily cost by model
SELECT
    toDate(start_time) AS day,
    attributes['llm.model'] AS model,
    sum(attributes['llm.cost_usd']) AS total_cost,
    count() AS request_count
FROM traces
WHERE service_name = 'ai-agent'
  AND start_time >= now() - INTERVAL 30 DAY
GROUP BY day, model
ORDER BY day DESC, total_cost DESC
```

In a multi-region deployment, each region runs its own observability stack. This avoids cross-region latency for instrumentation and ensures that a regional outage does not affect other regions.

**Regional collectors.** Each region has its own OpenTelemetry collector DaemonSet, Kafka cluster, and ClickHouse cluster. Applications within the region send spans to the local collector. The collector adds a region attribute to each span.

**Centralized aggregation.** Metrics from all regions are forwarded to a global metrics store. This enables global dashboards showing total cost, total request volume, and global quality scores. Aggregation is one-way; the global store does not serve individual trace queries.

**Cross-region trace stitching.** When a trace spans multiple regions (for example, a request from US-West that uses a model deployed in US-East), the trace is split across regional ClickHouse clusters. The global query engine merges trace trees by fetching spans from both regions using trace_id. This merge takes 50 to 200 milliseconds and adds the "stitched" attribute to the trace.

**Failure isolation.** If the EU-West observability stack fails, US-East and US-West continue unaffected. Applications in EU-West buffer spans locally for up to 5 minutes and replay when the stack recovers. If the local buffer fills, 500 oldest spans per minute are dropped as a graceful degradation.

## Deployment and Operations

### Zero-Downtime Collector Deployment

OpenTelemetry collectors are deployed as a DaemonSet with a rolling update strategy:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: otel-collector
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  template:
    spec:
      containers:
      - name: otel-collector
        image: otel/opentelemetry-collector-contrib:0.95.0
        ports:
        - containerPort: 4317  # gRPC
        - containerPort: 4318  # HTTP
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
          limits:
            memory: "2Gi"
            cpu: "2"
        volumeMounts:
        - name: buffer
          mountPath: /data/buffer
      volumes:
      - name: buffer
        emptyDir:
          sizeLimit: "10Gi"
```

The rolling update ensures no more than one collector is unavailable at any time. Agents retry failed exports with exponential backoff (100ms, 200ms, 400ms, 800ms, 1600ms, max 30 seconds). If the local collector is unavailable for more than 30 seconds, agents write spans to a secondary collector in a different region.

### Backpressure Handling

Backpressure occurs when the pipeline cannot keep up with the ingestion rate. The system handles this at three levels:

**Level 1: Collector buffer.** The collector buffers spans in memory (1 GB default). When memory exceeds 80 percent, the collector reduces the batch export interval from 5 seconds to 1 second, and increases export batch size from 512 to 2,048 spans.

**Level 2: Kafka backpressure.** If Kafka cannot write fast enough, the collector receives a backpressure signal from the Kafka producer (max.block.ms exceeded). The collector reduces its ingestion rate by sampling more aggressively: the sampling rate drops from 10 percent to 5 percent.

**Level 3: Application-level.** If the collector is completely unavailable, agents buffer spans to disk (max 512 MB). When the disk buffer reaches 90 percent capacity, the agent starts dropping the oldest spans. The agent logs a warning: "Observability backpressure active — dropping spans."

### Retention Management

Data retention is managed via ClickHouse's TTL mechanism and S3 lifecycle policies:

```sql
-- ClickHouse TTL: delete raw traces after 30 days
ALTER TABLE traces
MODIFY TTL toDate(start_time) + INTERVAL 30 DAY DELETE;

-- ClickHouse TTL: move to object storage after 7 days
ALTER TABLE traces
MODIFY TTL toDate(start_time) + INTERVAL 7 DAY TO VOLUME 'slow_ssd',
     toDate(start_time) + INTERVAL 30 DAY TO VOLUME 's3';

-- S3 lifecycle policy: delete sampled traces after 90 days
{
  "Rules": [
    {
      "Prefix": "sampled-traces/",
      "Status": "Enabled",
      "Expiration": { "Days": 90 }
    }
  ]
}
```

The observability system monitors itself. Metrics pipelines track:

- **Collector health:** ingestion rate (spans/sec), error rate, memory usage, CPU usage, gRPC connection count.
- **Kafka health:** messages per second, consumer lag, partition count, under-replicated partitions.
- **ClickHouse health:** query latency (p50, p95, p99), merge queue size, disk usage, insert rate.
- **Evaluation pipeline health:** evaluations per second, evaluator error rate, evaluator latency.

Alerts on the observability system itself:

| Alert | Condition | Action |
|-------|-----------|--------|
| Collector ingestion drop | Spans/sec drops below 50% of baseline for 5 minutes | Page on-call engineer |
| Kafka consumer lag | Lag exceeds 100,000 messages for 5 minutes | Increase stream processor workers |
| ClickHouse query latency | P99 query latency exceeds 5 seconds for 5 minutes | Scale ClickHouse read replicas |
| Evaluation error rate | Evaluator error rate exceeds 1% for 5 minutes | Rollback evaluator deployment |
| Storage utilization | Disk usage exceeds 80% for 10 minutes | Extend retention or add nodes |
