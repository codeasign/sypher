An AI observability platform is a data pipeline disguised as a monitoring tool. At its core, it ingests structured events (spans), processes them into metrics and traces, and makes them queryable in sub-second time. The architecture has five major subsystems.

The architecture separates concerns. Instrumentation agents are lightweight and embedded in application processes. They capture spans and export them to collectors. Collectors act as a gateway, providing authentication, rate-limiting, and initial processing. Stream processors transform raw spans into metrics and searchable traces. Storage systems specialize: time-series databases for metrics, object stores or columnar databases for traces, and inverted-index stores for logs.

### Instrumentation Agents

Each application that makes AI calls runs an instrumentation agent. These agents are typically OpenTelemetry-compatible SDKs for Python, Node.js, Go, Java, or .NET. The agent intercepts function calls to the LLM provider SDK, the vector database client, and the guardrail library.

When the application calls `openai.chat.completions.create()`, the agent:

1. Creates a new span with a unique span ID (64-bit random integer).
2. Attaches the parent trace ID and parent span ID from the current context.
3. Records start timestamp in Unix nanoseconds.
4. Executes the actual API call.
5. On response, records end timestamp, duration, and response attributes.
6. Sets the span status (OK or ERROR).
7. Exports the span to the collector via gRPC or HTTP.

The agent batches spans before export. By default, it sends batches every 5 seconds or when the batch reaches 512 spans, whichever comes first. This batching reduces network overhead by approximately 90 percent compared to sending each span individually.

### Collector Layer

The collector receives span data from multiple agents. It validates the OpenTelemetry protocol (OTLP) schema, checks for required fields (trace ID, span ID, start time, end time), and rejects malformed spans with a 400 response.

After validation, the collector enriches spans with infrastructure metadata:

- Host name and IP address
- Kubernetes pod name and namespace
- Container ID
- Service name and version
- Deployment environment (production, staging, development)
- Cloud region and availability zone

The collector also handles backpressure. If the stream processor cannot keep up, the collector uses an in-memory buffer (default 1 GB) and applies a circuit breaker after 30 seconds of sustained backpressure. When the circuit breaker trips, the collector returns HTTP 503 to agents, which then buffer spans locally (up to 512 MB on disk) and retry.

### Stream Processor

The stream processor is the brain of the pipeline. It receives validated spans and performs three operations:

**Sampling.** Decides whether to keep or discard each span. Head-based sampling makes this decision at the agent, before the span reaches the collector. Tail-based sampling makes the decision after the stream processor has seen all spans for a trace.

**Routing.** Determines which storage system receives each span. Metric-like data (latency, token count, cost) routes to the metrics pipeline. Trace data (span trees, attributes) routes to the trace storage. Log data (raw text, errors) routes to the log store.

**Enrichment.** Computes derived values. Token cost from token counts and model pricing. Quality scores from evaluation results. Session-level aggregates from individual spans.

## Data Structures and Algorithms

### Trace Trees

A trace is a tree of spans. Each span has a parent span ID (except the root span, which has no parent). The root span represents the entire request. Child spans represent sub-operations.

Span data structure (JSON representation):

```json
{
  "trace_id": "a1b2c3d4e5f67890",
  "span_id": "1234567890abcdef",
  "parent_span_id": null,
  "name": "llm.call",
  "kind": "CLIENT",
  "start_time": 1700000000000000000,
  "end_time": 1700000000001400000,
  "duration_ns": 1400000000,
  "status": {"code": "OK"},
  "attributes": {
    "llm.model": "gpt-4o",
    "llm.temperature": 0.3,
    "llm.max_tokens": 1024,
    "llm.input_tokens": 1847,
    "llm.output_tokens": 342,
    "llm.cost_usd": 0.0023,
    "llm.response": "Our return policy for electronics is..."
  },
  "resource": {
    "service.name": "ai-agent",
    "deployment.environment": "production",
    "cloud.region": "us-east-1"
  }
}
```

The trace tree is reconstructed by following parent_span_id pointers. The query engine performs a recursive walk from the root span, building a tree structure. For a trace with N spans, the reconstruction takes O(N) time.

### Span IDs for Distributed Tracing

Span IDs are 64-bit unsigned integers, typically generated using a random number generator or a distributed ID scheme like Snowflake (41-bit timestamp + 10-bit worker ID + 12-bit sequence number).

Trace IDs are 128-bit (16 bytes) for globally unique identification across services and regions. The probability of collision with random 128-bit IDs is approximately 1 in 2^128, making collisions negligible for any practical scale.

### Sampling Algorithms

**Head-based sampling.** The decision to keep a trace is made at the root span. If the root span is sampled, all child spans are also kept. Simple to implement. The default rate is typically 5 to 10 percent. Problem: you might miss rare errors if they occur outside the sampled fraction.

Implementation in pseudocode:

```python
def head_based_sample(root_span, rate=0.05):
    trace_id = root_span.trace_id
    # Use trace ID hash for deterministic sampling
    hash_val = hash(trace_id) % 10000
    return hash_val < (rate * 10000)
```

**Tail-based sampling.** The decision is deferred until all spans for a trace have arrived. The stream processor buff ers spans by trace ID. When the trace completes (or after a timeout of 60 seconds), the processor evaluates sampling criteria:

- Keep all traces with errors (sampling priority HIGH).
- Keep all traces with latency above P99 (sampling priority HIGH).
- Keep traces with unusual attributes (sampling priority MEDIUM).
- Randomly sample remaining traces (sampling priority LOW).

Tail-based sampling preserves rare events significantly better. A study of production LLM systems showed tail-based sampling captured 99.7 percent of error traces compared to 62 percent for head-based sampling at a 10 percent overall rate.

Implementation in pseudocode:

```python
def tail_based_sample(trace_spans, error_rate_high=1.0, latency_p99_threshold=5000):
    has_error = any(s.status.code == "ERROR" for s in trace_spans)
    if has_error:
        return sample_with_probability(error_rate_high)

    max_latency = max(s.duration_ms for s in trace_spans)
    if max_latency > latency_p99_threshold:
        return True

    if has_unusual_attribute(trace_spans):
        return sample_with_probability(0.5)

    return sample_with_probability(0.05)
```

**Importance sampling.** Weights traces by their information value. A trace with a rapidly slowing response time gets higher weight. A trace with tokens-per-second far from the baseline gets higher weight. The sampling probability is proportional to the trace's deviation from the norm.

### Time-Series Data Structures

Metrics are stored as time-series data. Each time series is defined by a metric name and a set of label key-value pairs. For example:

```
llm_latency_ms{model="gpt-4o",environment="production",region="us-east-1"}
```

Time-series databases use specialized data structures for efficient storage and query:

**TSID (Time Series ID).** A 64-bit hash of the metric name and label set. Enables O(1) lookup of a time series. TSID = FNV-1a hash(metric_name + sorted_labels).

**LSM Tree (Log-Structured Merge Tree).** Used for write-heavy workloads. New data is written to an in-memory buffer (memtable), then flushed to disk as sorted SSTable files. Background compaction merges SSTables. Provides 10x to 100x better write throughput compared to B-tree.

**Gorilla compression.** Each time series stores (timestamp, value) pairs. Gorilla compression encodes timestamps as delta-of-delta (saving 12 bytes per point to 1-2 bytes) and values using XOR of the previous value (saving 8 bytes per point to 1-4 bytes). Achieves 10x to 14x compression ratio for time-series data.

Let us walk through each step with concrete examples:

**Step 1: Request Generated.** A user submits a query. The application creates a context with a new trace ID. In OpenTelemetry, this is done via the Tracer SDK:

```python
from opentelemetry import trace
tracer = trace.get_tracer("ai-agent", "1.0.0")
with tracer.start_as_current_span("user.request") as span:
    user_message = "What is the return policy for laptops?"
    span.set_attribute("app.user_message", user_message)
    span.set_attribute("app.user_id", "user_48291")
```

**Step 2: Instrument.** Each sub-operation creates a child span. The LLM call creates an LLM span. The vector database query creates a retrieval span. The guardrail check creates an evaluation span.

**Step 3: Emit.** When the span ends, the agent serializes it to protobuf format and adds it to the export batch. The batch is stored in a thread-local queue. The default batch size is 512 spans or 5 seconds.

**Step 4: Collect.** The OTLP receiver on the collector accepts the batch via gRPC. It deserializes the protobuf, validates the schema, and acknowledges receipt with an empty response. The span is now in the collector's memory.

**Step 5: Batch.** The collector groups spans by trace ID in an in-memory hash map. This grouping enables the collector to apply tail-based sampling decisions before forwarding to downstream processors.

**Step 6: Process.** The stream processor applies sampling rules. If the trace includes an error span (status code ERROR), it is always kept. Otherwise, an importance score is computed: latency deviation, token count deviation, and model version change all contribute to the score. Spans with a score above 0.5 are kept. Others are sampled at 5 percent.

**Step 7: Index.** The processor writes spans to the trace store's inverted index. The index maps attribute values to span IDs. When you search for `model="gpt-4o" AND latency > 5000`, the index returns matching span IDs in under 10 milliseconds.

**Step 8: Store.** Spans are written to persistent storage. Trace stores typically use a columnar storage format (Parquet) on object storage (S3, GCS, Azure Blob). This enables cost-effective storage for large volumes. Storage cost is approximately 0.023 USD per GB per month on S3.

**Step 9: Query.** The query engine receives dashboard requests. A Grafana dashboard showing P50 latency over the last 6 hours queries the metrics store. The PromQL query `histogram_quantile(0.50, sum(rate(llm_latency_ms_bucket[5m])) by (le, model))` executes in under 200 milliseconds.

**Step 10: Alert.** The alert manager evaluates alerting rules every 30 seconds. If P50 latency exceeds 5 seconds for 5 consecutive evaluations, an alert fires and pages the on-call engineer via PagerDuty.

**Step 11: Retain.** Data retention policies are applied. Raw traces are retained for 30 days. Aggregated metrics are retained for 12 months. Sampled traces are retained for 90 days for debugging and model improvement.

**Step 12: Delete.** After the retention period, data is deleted. The storage system uses TTL-based deletion: partitions older than the retention period are dropped. This is a metadata operation that takes under 1 second per partition.

## Consistency and State

Distributed observability systems face five consistency challenges:

### Partial Traces

A trace might arrive with missing spans. The retrieval span succeeded, but the LLM call span timed out on export. The trace appears incomplete. The system must handle this gracefully.

**Solution.** The trace store marks traces as incomplete if they are missing expected span types. The query engine displays a warning: "This trace is incomplete — missing LLM call span." Users can still view the available spans and investigate.

### Dropped Spans

When an application crashes, spans in the export buffer are lost. The same happens when a collector runs out of memory during a traffic spike.

**Solution.** Instrumentation agents write spans to a local file buffer as a fallback. If the network is unavailable, spans are written to disk and replayed when connectivity returns. The file buffer is limited to 512 MB to avoid filling the disk. When the buffer is full, the oldest spans are dropped.

### High-Cardinality Dimensions

A dimension is high-cardinality when it has many distinct values. User IDs, session IDs, trace IDs are all high-cardinality. Prometheus struggles with high-cardinality dimensions because it stores one time series per label combination.

```json
{
  "attributes": {
    "user_id": "user_48291",        // Bad: 10M+ distinct values
    "session_id": "sess_1772901",   // Bad: 100M+ distinct values
    "trace_id": "a1b2c3d4e5f6"      // Bad: unique per request
  }
}
```

**Solution.** Store high-cardinality dimensions in the trace store, not the metrics store. Only low-cardinality dimensions (model name, environment, service name) are promoted to metric labels. High-cardinality values remain as span attributes that can be queried via the trace store's search interface.

### Clock Skew in Distributed Tracing

When the application runs on one server and the LLM API call runs on another, the clocks might be out of sync. A difference of 100 milliseconds can create impossible span trees where a child span appears to start before its parent.

**Solution.** The collector adjusts span timestamps using the parent-offset heuristic. For each child span, the offset is calculated as: child.start_time - parent.start_time. If the offset is negative by more than 1 second, the collector normalizes the child's timestamps using the parent's clock. This heuristic works for 99.5 percent of cases. For the remaining cases, spans are marked with a "clock skew detected" flag.

## Performance Characteristics

### Instrumentation Overhead

The instrumentation agent adds minimal overhead to each request. Benchmarking across 10,000 requests shows:

- Span creation and attribute setting: 0.3 ms (p50) / 1.2 ms (p99)
- Span serialization to protobuf: 0.5 ms (p50) / 2.1 ms (p99)
- Batch export over gRPC: 2.0 ms (p50) / 8.0 ms (p99) — amortized across 512 spans
- Total overhead per request: under 5 ms (p99) — less than 0.5 percent of typical LLM response time (1.5 to 5 seconds).

### Ingestion Throughput

A single collector node processes 100,000 spans per second with the following configuration:

- CPU: 4 cores (AWS c6i.xlarge)
- Memory: 8 GB RAM
- Network: 10 Gbps
- Batch size: 1,024 spans

Throughput scales linearly with collector nodes. A deployment of 10 collector nodes handles 1 million spans per second.

### Storage Requirements

Storage is the dominant cost for observability platforms. For an application serving 10 million LLM requests per day:

- Each trace contains an average of 4 spans.
- Each span with attributes averages 2 KB of data.
- Total daily volume: 80 GB (10 million traces x 4 spans x 2 KB).
- After compression (3x ratio): 27 GB per day.
- Monthly storage: 810 GB (30 days x 27 GB/day).
- Storage cost on S3 (0.023 USD/GB/month): 18.63 USD per month.

With 30-day retention for raw data and 12-month retention for aggregated metrics:

- Raw traces (30 days): 810 GB at 18.63 USD.
- Aggregated metrics (12 months): 50 GB at 1.15 USD.
- Total monthly storage cost: approximately 20 USD.

For a large-scale deployment serving 100 million requests per day, multiply by 10: 200 USD per month in storage.

### Query Latency

Query performance varies by query type:

**Metrics queries (PromQL).** Returns within 50 to 200 milliseconds for queries over 30 days of data. Aggregation over 6 hours completes in under 50 milliseconds.

**Trace queries by ID.** Single-trace lookup by trace ID returns in under 20 milliseconds. The trace store uses a hash index on trace_id for O(1) lookup.

**Trace search by attribute.** Attribute search over 30 days of data: returns results in 200 milliseconds to 5 seconds depending on the selectivity of the attribute filter. Filtering by model name (high selectivity) is faster than filtering by response length (low selectivity).

**Aggregate queries.** Computing average latency across all traces for the last week: completes in 2 to 10 seconds depending on the aggregation granularity.

## Configuration and Tuning

### Sampling Rates

Configure sampling rates at the collector level. The rate can vary by time of day, environment, or span attribute:

```yaml
sampling:
  default_rate: 0.05  # Sample 5% of normal traces
  rules:
    - match: {attribute: "llm.model", value: "gpt-4o"}
      rate: 0.1       # Sample 10% of GPT-4o traces (more expensive)
    - match: {attribute: "deployment.environment", value: "production"}
      rate: 0.1       # Sample 10% of production traces
    - match: {attribute: "deployment.environment", value: "staging"}
      rate: 1.0       # Sample 100% of staging traces
    - if_error: true
      rate: 1.0       # Always keep error traces
```

### Span Attributes

Control which attributes are captured and stored. Sensitive attributes can be redacted:

```yaml
attributes:
  allowed: [llm.model, llm.temperature, llm.input_tokens, llm.output_tokens,
            llm.cost_usd, retrieval.docs_count, retrieval.duration_ms,
            guardrail.toxicity_score, guardrail.factuality_score]
  redacted: [app.user_message, app.user_id, llm.prompt, llm.response]
  redaction_strategy: replace_with_hash
```

### Aggregation Intervals

Metrics are aggregated at multiple granularities to balance query performance and storage:

| Granularity | Interval | Retention | Storage |
|-------------|----------|-----------|---------|
| Raw metrics | 30 seconds | 7 days | 10 GB/day |
| Downsampled | 5 minutes | 30 days | 2 GB/day |
| Hourly | 1 hour | 12 months | 0.5 GB/day |
| Daily | 1 day | 5 years | 0.1 GB/day |

### Retention Policies

Set retention policies based on data value and storage cost:

```yaml
retention:
  raw_traces: 30 days        # For debugging recent incidents
  sampled_traces: 90 days    # For model improvement and A/B analysis
  aggregated_metrics: 365 days  # For trend analysis and capacity planning
  evaluation_results: 180 days  # For model performance tracking
  audit_logs: 2 years         # For compliance requirements
```

### Alert Thresholds

Define alert thresholds with sensitivity tuning to avoid false positives:

```yaml
alerts:
  - name: high_latency
    query: 'histogram_quantile(0.95, rate(llm_latency_ms_bucket[5m]))'
    threshold: 5000
    for: 5m
    severity: warning
  - name: quality_drop
    query: 'avg(quality_factuality_score[15m])'
    threshold: 0.7
    for: 10m
    severity: critical
  - name: budget_exceeded
    query: 'sum(rate(llm_cost_usd_total[1h])) * 24'
    threshold: 500  # Daily budget exceeded $500
    for: 5m
    severity: critical
  - name: hallucination_spike
    query: 'rate(hallucination_detected_total[1h])'
    threshold: 0.01  # More than 1% hallucination rate
    for: 15m
    severity: critical
  - name: error_rate
    query: 'sum(rate(span_errors_total[5m])) / sum(rate(span_total[5m]))'
    threshold: 0.05  # Error rate above 5%
    for: 5m
    severity: warning
```

The key tuning insight: tail-based sampling at 10 percent captures 78 percent of anomalies. Head-based sampling at the same 10 percent rate captures only 35 percent. The additional complexity of tail-based sampling pays significant dividends for anomaly detection.

For most production systems, a combined approach works best:

- 100 percent sampling for staging and development environments.
- 10 percent tail-based sampling for production, with 100 percent for error traces.
- 5 percent head-based sampling during peak traffic hours (10 AM to 4 PM) to reduce ingestion spikes.
- Attribute-based sampling to prioritize expensive model calls (GPT-4o at 20 percent, GPT-4o-mini at 5 percent).
