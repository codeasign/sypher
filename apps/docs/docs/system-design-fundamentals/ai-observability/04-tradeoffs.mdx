## The Core Tension

Every AI observability system faces a fundamental tension: completeness versus cost. You want to capture every detail of every LLM call so you can debug any issue. But storing every detail for every call costs money — often more than the LLM API calls themselves.

The cost of completeness is real. For an application serving 10 million requests per day:

- Full trace storage: 76 GB/day before compression, 25 GB/day after. At 0.30 USD/GB for SSD, that is 7.50 USD per day just for storage.
- Evaluation compute: 12 evaluators per trace x 50 ms each = 600 ms of compute per trace. At 10 million traces per day, that is 6 million seconds of compute. At 0.10 USD per CPU-hour, that is 167 USD per day.
- Network bandwidth: 76 GB/day outbound from the application cluster. At 0.09 USD/GB for inter-region transfer, that is 6.84 USD per day.

Total: approximately 181 USD per day, or 5,430 USD per month for a single application.

The solution is not to choose one extreme. It is to apply a combination of techniques: sampling to reduce volume, tiered storage to reduce cost, and privacy filters to reduce risk.

Use this decision matrix to choose the right approach for your use case:

| Scenario | Sampling Strategy | Retention | Evaluation | Estimated Monthly Cost |
|----------|------------------|-----------|------------|----------------------|
| MVP prototype (100 req/day) | 100% (no sampling needed) | 7 days | 3 evaluators | 5 USD |
| Internal tool (10K req/day) | 100% head-based | 14 days | 6 evaluators | 15 USD |
| Customer-facing app (1M req/day) | 10% tail-based, 100% errors | 30 days raw, 90 days sampled | 8 evaluators | 150 USD |
| High-traffic app (10M req/day) | 5% tail-based, 100% errors | 30 days raw, 90 days sampled | 12 evaluators | 500 USD |
| Regulated app (1M req/day) | 100% metadata-only, 0% text | 90 days | 6 evaluators (no text) | 80 USD |
| Enterprise platform (50M req/day) | 2% tail-based, 100% errors, importance sampling | 7 days raw, 30 days sampled | 12 evaluators | 2,000 USD |

The key insight: sampling is the most powerful lever. Dropping from 10 percent to 5 percent sampling cuts storage and compute costs in half while still capturing 65 percent of anomalies with tail-based sampling.

## CAP Theorem Positioning

AI observability systems are distributed databases. The CAP theorem applies: you cannot have Consistency, Availability, and Partition Tolerance simultaneously. Different observability systems make different trade-offs:

**ClickHouse (CP).** Prioritizes consistency over availability in partition scenarios. If a ClickHouse node fails, queries on that node's data return errors until the node recovers or a replica takes over. This is acceptable because trace data is not write-critical; a few seconds of unavailability is tolerable.

**Kafka (AP with configurable acks).** With `acks=all`, Kafka is CP. With `acks=1`, Kafka is AP. Most observability deployments use `acks=1` for the span ingestion topic (accepting potential data loss for higher throughput) and `acks=all` for the evaluation results topic (preventing loss of evaluation data).

**Dashboards (AP).** Grafana dashboards are eventually consistent. If a user refreshes a dashboard, they might see slightly different data depending on which replica answers. This is acceptable because dashboards show trends, not individual data points. A 5-second delay in metric visibility is not a problem.

**The practical decision for observability systems:** sacrifice consistency for availability. A dashboard showing slightly stale data is better than a dashboard that refuses to load. Trace queries that return 99 percent of spans are better than queries that fail because one span is missing.

## Failure Modes

### Failure Mode 1: Pipeline Backpressure

**The scenario.** A marketing campaign goes viral. Your application traffic spikes 50x in 10 minutes. The observability pipeline receives 50 million spans per minute instead of the usual 1 million. The collector cluster reaches 100 percent CPU. Kafka consumer lag grows from 100 messages to 5 million messages. The evaluation pipeline falls behind by 2 hours.

**The impact.** Dashboards show stale data. Alerts fire for the observability system itself. Engineers are paged for a failing observability system while the actual application is working fine.

**The root cause.** The pipeline was provisioned for 2x peak traffic, not 50x. Auto-scaling was configured with a 5-minute cooldown, too slow to handle the spike.

**The fix.** Implement adaptive sampling: when the collector detects more than 80 percent CPU utilization, it reduces the sampling rate dynamically. Configure the collector to drop spans rather than queue them during extreme spikes. Pre-provision the pipeline for 10x peak traffic instead of 2x.

### Failure Mode 2: Data Loss

**The scenario.** A Kubernetes node running the collector crashes. The collector's in-memory buffer of 500,000 spans is lost. The agent's disk buffer is also corrupted because the node's disk was full. You lose 500,000 spans covering 2 minutes of traffic.

**The impact.** A 2-minute gap in trace data. If an incident occurred during those 2 minutes, you have no data to investigate. The quality score dashboard shows a dip because the missing data skews the average.

**The root cause.** The collector persisted spans to an `emptyDir` volume, which is ephemeral. The agent's disk buffer had no size limit and filled the disk, causing the corruption.

**The fix.** Use a persistent volume for the collector's buffer. Set a hard limit on the agent's disk buffer (512 MB). Configure the agent to use a secondary collector in a different availability zone as a fallback. Implement a health check that alerts when the agent's buffer exceeds 80 percent capacity.

### Failure Mode 3: Sampling Bias

**The scenario.** Your team uses 5 percent head-based sampling. The sampling decision is based on the trace ID hash. A user reports that the AI agent refuses to answer questions about international shipping. You search the trace store for "international shipping" and find zero results. The sampling randomly excluded all traces containing that phrase.

**The impact.** You cannot investigate the user's report. The bug goes undiagnosed for 3 weeks until a second user reports the same issue and you increase sampling to 100 percent for that user's session.

**The root cause.** Head-based sampling does not guarantee that rare events are captured. With 5 percent sampling, there is a 95 percent chance that any single trace is discarded. If the "international shipping" bug affects only 1 percent of users, the probability of capturing at least one affected trace is 1 - (0.95)^(0.01 x 10,000,000), which is approximately 100 percent. But the probability of capturing a specific user's trace at a specific time is only 5 percent.

**The fix.** Switch to tail-based sampling with attribute-based rules. Configure the sampler to always keep traces where the input contains "international shipping" or other critical keywords. Increase sampling for specific user segments during investigation.

### Failure Mode 4: High-Cardinality Explosion

**The scenario.** Your team adds `user_id` as a Prometheus label to track LLM cost per user. Your application has 1 million daily active users. Prometheus creates one time series per user per metric. With 20 metrics, that is 20 million time series. Prometheus's time-series database (TSDB) cannot handle this cardinality. The server runs out of memory and crashes.

**The impact.** The metrics store is unavailable for 30 minutes. All dashboards show gaps. All alerting rules fail to evaluate.

**The root cause.** High-cardinality dimensions were used as metric labels. Prometheus is optimized for low-cardinality dimensions (dozens to low thousands), not millions.

**The fix.** Move high-cardinality dimensions from metrics labels to trace attributes. Query cost per user via the trace store, not the metrics store. The trace store uses an inverted index that handles high cardinality efficiently. A ClickHouse query for cost per user completes in 2 seconds across 30 days of data.

### Failure Mode 5: PII Leakage

**The scenario.** Your observability platform logs full prompt and response text. A compliance audit reveals that customer PII (email addresses, phone numbers, credit card numbers) is stored in the trace store. The platform is not HIPAA compliant. The audit triggers a security incident.

**The impact.** The company must notify affected customers. The observability platform is taken offline for 2 weeks while a PII redaction pipeline is built. The team loses all observability during this period.

**The root cause.** The instrumentation captured all span attributes without filtering sensitive data. The team assumed that the LLM would not receive PII, but customers often paste their email addresses and phone numbers into chat messages.

**The fix.** Implement PII redaction at the instrumentation layer. Before emitting a span, scan the text for PII patterns (email regex, phone regex, credit card Luhn check) and replace matches with `[REDACTED]`. Configure the collector to drop any span attributes that contain the `[REDACTED]` prefix. Set up a second pass at the storage layer that detects any remaining PII and quarantines the trace.

### Failure Mode 6: Alert Fatigue Leading to Ignored Critical Alerts

**The scenario.** The team configured 50 alert rules covering every possible failure mode. The on-call engineer receives 200 alerts per day. Most are false positives triggered by normal traffic patterns. After 2 weeks of this noise, the engineer silently acknowledges all alerts without investigating. A real quality regression (hallucination rate spike from 2 percent to 15 percent) fires a critical alert. The engineer acknowledges it automatically, assuming it is another false positive. The regression goes uninvestigated for 4 hours.

**The impact.** Users receive hallucinated responses for 4 hours before a user complaint escalates to the VP level. The company's reputation is damaged. The team could have caught it in 5 minutes with proper alert tuning.

**The root cause.** Alert rules were configured without tuning thresholds. No noise budget was allocated. No P1/P2/P3 classification was applied. Every alert was treated as equally important.

**The fix.** Implement a three-tier alert classification: P1 (critical, pages engineer immediately, maximum 3 rules), P2 (warning, sends Slack notification within 30 minutes, maximum 10 rules), P3 (info, logged to dashboard, reviewed weekly, no limit). Each alert rule must have a measured false-positive rate below 10 percent. Rules with higher rates are automatically disabled until retuned.

### Failure Mode 7: Cost Tracking Blindness

**The scenario.** The observability platform tracks latency and quality but does not track cost per request. The company's monthly LLM bill grows from 5,000 USD to 45,000 USD over 6 months. The VP of Engineering notices the spike in the cloud billing dashboard and demands answers. The team has no data on which model, which feature, or which team is driving costs. They cannot explain the 9x increase.

**The impact.** A 40,000 USD cost overrun goes undetected for months. The company misses its quarterly budget targets. The AI team is put on a hiring freeze pending cost investigation.

**The root cause.** Cost per request was not an instrumented metric. The observability platform tracked technical metrics (latency, error rate) but not business metrics (cost per user, cost per feature, cost per conversation).

**The fix.** Add cost tracking to every LLM call span. Use model-specific pricing tables: GPT-4o at 2.50 USD / 1M input tokens and 10.00 USD / 1M output tokens; GPT-4o-mini at 0.15 USD / 1M input and 0.60 USD / 1M output. Aggregate cost by model, team, feature, and user segment. Configure a budget alert: if projected daily cost exceeds the daily budget (500 USD), fire a warning. If it exceeds 120 percent of the daily budget, fire a critical alert.

### Failure Mode 8: Evaluation Pipeline Bias

**The scenario.** The evaluation pipeline uses GPT-4o as the judge LLM to evaluate responses from another language model. The evaluation pipeline shows consistently high quality scores (above 0.9). However, user satisfaction surveys show declining satisfaction. The evaluation scores and user satisfaction are negatively correlated.

**The impact.** The team is optimizing for evaluation scores, not user satisfaction. They make changes that improve evaluation scores but degrade the user experience. The disconnect goes undetected for 3 months.

**The root cause.** Using the same LLM family for both generation and evaluation introduced a systemic bias. GPT-4o judged GPT-4o responses favorably (self-enhancement bias). The evaluation pipeline was optimized for correlation with human judgments but the correlation degraded as the model was fine-tuned.

**The fix.** Use a different model family for evaluation than for generation. For example, if the application uses GPT-4o, use Claude or Llama 3 as the judge. Regularly validate the evaluation pipeline against human judgments (every 2 weeks). Track the correlation between evaluation scores and user satisfaction metrics. If the correlation drops below 0.7, retrain or recalibrate the evaluators.

## Anti-Patterns

### Anti-Pattern 1: Collect Everything, Decide Later

You collect every attribute, every span, every trace at 100 percent sampling. The storage bill grows to 20,000 USD per month. You have so much data that queries take 30 seconds. Nobody uses the system because it is too slow. You end up dropping 90 percent of the data after 3 months.

**Better approach.** Start with a clear data collection policy. Define which attributes are needed, at what sampling rate, and for how long. Collect aggressively in staging (100 percent), conservatively in production (5 to 10 percent). Add collection as needed, not preemptively.

### Anti-Pattern 2: Monitoring LLMs Like Servers

You monitor only infrastructure metrics: CPU, memory, disk, network. Your dashboards show server health but give no insight into model behavior. The LLM is hallucinating, but your CPU graphs show 30 percent utilization. You have no way to detect the problem.

**Better approach.** Monitor AI-specific metrics: token usage, latency per model, quality scores, hallucination rate, cost per request. Infrastructure metrics are a complement, not a replacement.

### Anti-Pattern 3: No Feedback Loop

You collect observability data and store it in dashboards. Nobody acts on the data. The quality score drops. The dashboard shows it. Nobody notices because the dashboard is on a monitor that nobody looks at. The quality score stays low for 3 months.

**Better approach.** Close the loop: observability data must trigger actions. Configure alerts for critical thresholds. Route evaluation results to a Slack channel that the team monitors. Schedule weekly reviews of quality trends. Tie observability metrics to the team's OKRs.

### Anti-Pattern 4: Perfect Trace Collection

You spend 3 months building a perfect observability system with 100 percent trace coverage, 12 evaluators, 90-day retention, and sub-100ms query latency. The system costs 15,000 USD per month to run. The team has 2 engineers maintaining it. The ML team still uses print statements to debug their prompts.

**Better approach.** Start with a minimal viable observability system: 5 percent sampling, 3 evaluators, 7-day retention. Get it working in 1 week. Add features incrementally based on team needs. The perfect system is the enemy of the useful system.

### Anti-Pattern 5: Alerts on Everything

You configure 50 alert rules. The on-call engineer receives 200 alerts per day. The alerts are noisy and mostly false positives. The engineer silences alerts and ignores them. The one critical alert that matters (hallucination rate above 5 percent) gets lost in the noise.

**Better approach.** Start with 5 carefully tuned alert rules. Each rule must have a clear action: page the on-call engineer, send a Slack notification, or log for weekly review. Measure the alert-to-action ratio. If more than 10 percent of alerts do not result in action, tune or remove the rule.

## When NOT to Use AI Observability

AI Observability is not always the right solution. Here are scenarios where it adds unnecessary complexity:

**You are prototyping.** If you are experimenting with a prompt in a Jupyter notebook, a print statement is sufficient. Adding OpenTelemetry instrumentation and a collector is overkill. Use print statements or simple logging until you deploy to a shared environment.

**You have fewer than 100 requests per day.** The observability infrastructure costs more than the LLM API calls. Log to a file and inspect manually. Add observability when you reach 1,000 requests per day.

**You are using a fully managed AI service.** If you use a managed service like ChatGPT Enterprise or Google's Vertex AI Agent Builder, the provider handles observability. You can use their built-in dashboards and logs. Adding your own instrumentation duplicates what the provider already provides.

**Your compliance requirements forbid external monitoring.** Some highly regulated environments do not allow any external monitoring of AI systems. In this case, you must build observability entirely on-premises, which is significantly more expensive. Consider whether cheaper alternatives (manual sampling, audit logs) meet your needs.

**Your team has no capacity to act on observability data.** If you collect observability data but nobody has time to review dashboards, respond to alerts, or investigate traces, the system is a cost center with no value. Start with the simplest possible approach and scale when you have the team capacity to use the data.
