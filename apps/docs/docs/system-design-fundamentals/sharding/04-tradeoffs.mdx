## Core Tension

**Sharding improves write throughput and data scalability but sacrifices query flexibility and operational simplicity.**

A system with 1,024 shards can handle 15 million writes per second. Each shard stores 1/1,024 of the data, so single-shard queries complete in 2-5ms. But the same system cannot efficiently run a query that needs data from all 1,024 shards. A simple `SELECT COUNT(*) FROM users` that takes 500ms on a single-node database becomes a 5-10 second scatter-gather across 1,024 shards. A JOIN between users and orders that worked in 3ms on a monolithic database requires a materialized view or fan-out if the tables are on different shards.

The tension manifests in every design decision:

- **Single-shard transactions** are fast and ACID. Cross-shard transactions require two-phase commit, adding 30-80ms of latency and making the coordinator a single point of failure.
- **Local indexes** are small and efficient. Global indexes require maintaining coordination across all shards and are often impractical.
- **Per-shard backups** complete in minutes. Rebuilding an entire 1,024-shard cluster from backups requires verifying 1,024 distinct snapshots.
- **Adding shards** increases write capacity. Adding shards also increases the router's memory footprint, the metadata service's load, and the chance that any single shard has issues.

Every sharding decision is an answer to the question: "How much operational complexity and query inflexibility am I willing to accept for write scalability?"

### Decision 1: Hash Key vs Range Key

**Hash key advantage:** Uniform distribution. No hot spots from key ordering. Every shard receives roughly equal write load. The default choice for OLTP workloads.

**Range key advantage:** Natural partition pruning for time-range queries. Efficient data retention — drop old range shards. Co-located data within a range means queries for "last 30 days" hit exactly one shard.

**Choose hash when:** Write distribution must be uniform, the shard key has no natural ordering, or the dominant queries are point lookups (`WHERE key = ?`).

**Choose range when:** The dominant queries are time-range scans, data has a natural retention period, or you need ordered scan optimization.

### Decision 2: Few Large Shards vs Many Small Shards

**Few large shards (4-16):** Simpler operations. Fewer connections to manage. Lower scatter-gather latency (fewer participants). But each shard is a larger failure domain — a 4-shard cluster loses 25 percent of capacity per failure. Backups of 1 TB shards take longer.

**Many small shards (64-1024):** Better load distribution. Smaller failure domains (losing one out of 256 shards loses 0.4 percent of capacity). Faster per-shard maintenance. But more connections to manage, larger routing tables, and higher coordination overhead.

**Rule of thumb:** Start with enough shards so each holds 100-300 GB at peak projected size. For 30 TB: 100-300 shards. For 100 TB: 333-1,000 shards.

### Decision 3: Proxy-Based vs Application-Level Routing

**Proxy-based routing** (Vitess VTGate, ProxySQL, Citus): The router handles connection pooling, routing table management, and cross-shard queries. Applications use standard database drivers and do not know about shards. Adds 2-5ms of latency per query.

**Application-level routing** (embedded library, SDK): The application computes the shard and connects directly. Zero proxy latency. But every service must implement the same routing logic. Schema changes must be coordinated across all services.

**Choose proxy when:** You have multiple application services, teams, or languages. Operational consistency and centralized routing are priorities.

**Choose app-level when:** You have a single service or monolith. Every microsecond of latency matters. You can enforce routing consistency through a shared library.

### Decision 4: Consistent Hashing vs Hash Modulo

**Consistent hashing:** Adding a shard moves only 1/N of data (where N is the number of shards). Used by Cassandra, DynamoDB, Riak.

**Hash modulo (`hash(key) % N`):** Adding a shard moves approximately (N-1)/N of data. Simpler to implement and reason about. Used by Vitess (with controlled rebalancing), MySQL Fabric.

**Choose consistent hashing when:** The shard count changes frequently (auto-scaling cloud deployments), or you want to minimize data movement during scaling events.

**Choose hash modulo when:** The shard count is fixed or changes rarely (yearly capacity planning), or simplicity is more important than rebalancing efficiency.

**CP configuration (synchronous replication):** Writes wait for acknowledgment from a quorum of replicas before returning success. If a partition separates the primary from its replicas, the primary cannot form a quorum and stops accepting writes. This ensures consistency but sacrifices availability during the partition. Use this for financial systems and inventory management where inconsistency is worse than downtime.

**AP configuration (asynchronous replication):** Writes return success after the local WAL fsync. Replicas receive updates asynchronously. During a partition, the primary continues accepting writes even if replicas cannot be reached. If the primary itself is lost, the most recent writes may be lost too. Use this for social media feeds and analytics where availability is more important than up-to-the-second consistency.

**Typical sharded system behavior during a partition:** The shard's routing layer detects the partition and marks affected shards as degraded. Writes to degraded shards either fail (CP) or continue on the primary (AP). The operator restores connectivity or promotes a replica.

## Failure Modes

### Failure 1: Hot Spot on a Single Shard

**What:** A celebrity user or viral event causes one shard to receive 100x normal traffic. The shard's CPU hits 100 percent, query latency spikes from 10ms to 5 seconds, connection pool saturates, and queries queue.

**Why:** The shard key value for the hot content maps to a single shard. Hash-based sharding distributes data uniformly but cannot protect against access frequency skew — one key can generate 100x more traffic than another.

**Blast radius:** Limited to the hot shard (1/N of users). Other shards operate normally. But the application may experience partial failures if it queries the hot shard in the request path.

**Detection:** Per-shard monitoring shows one shard with 10x the QPS and 50x the p99 latency of the cluster average.

**Recovery:** (1) Identify the hot key from query logs. (2) If the hot key is a single value, split it out to a dedicated shard or replicate it to a dedicated cache layer. (3) If the hot key is a range, split the range into smaller ranges. (4) Implement application-level caching for the hot key (Redis, CDN). (5) Consider using a compound shard key to scatter hot data.

**Prevention:** Choose a shard key with high cardinality and uniform access frequency. Use consistent hashing with virtual nodes to enable fine-grained rebalancing. Implement application-level caching for frequently accessed keys. Consider read replicas for the hot shard.

### Failure 2: Uneven Data Distribution

**What:** Over time, some shards accumulate significantly more data than others. The largest shard is 3x the size of the smallest shard.

**Why:** The shard key does not distribute data uniformly. Common causes: range-based sharding with a monotonically increasing key, hash-based sharding with a low-cardinality shard key, or data that grows unevenly across shard key values.

**Blast radius:** The large shards have longer backup times, slower maintenance, and higher memory pressure. The small shards are underutilized, wasting capacity.

**Detection:** Disk usage monitoring shows the largest shard at 80 percent of capacity while the smallest is at 25 percent.

**Recovery:** Rebalance the cluster. For hash-based sharding, double the shard count and rehash. For range-based sharding, split the large range and merge the small ranges.

**Prevention:** Choose a high-cardinality shard key with uniform value distribution. Monitor per-shard data size weekly. Set an alert when the ratio of largest to smallest shard exceeds 1.5.

### Failure 3: Cross-Shard Query Performance Collapse

**What:** A developer adds a new feature that queries without the shard key. The feature goes viral, generating 1,000 scatter-gather queries per second. Each scatter-gather hits all 64 shards, generating 64,000 database connections and consuming 80 percent of the cluster's CPU.

**Why:** The developer did not realize the new query did not include the shard key. Code review missed it. There was no staging environment with the full shard topology to catch the performance impact.

**Blast radius:** The scatter-gather queries saturate the router's connection pool and starve single-shard queries. All users experience latency spikes, not just those on the affected shard.

**Detection:** The router's scatter-gather metrics show a sudden increase in multi-shard queries. Per-shard CPU and connection counts spike simultaneously on all shards.

**Recovery:** (1) Identify the offending query from the router's slow query log. (2) Hotfix the application to include the shard key in the WHERE clause. (3) If the query cannot include the shard key, route it to a dedicated read replica cluster to isolate it from OLTP traffic.

**Prevention:** Add a scatter-gather detector to the router that alerts when cross-shard queries exceed a configurable percentage of total queries. Make scatter-gather queries visible in the development environment. Add a code review check: "Does this query include the shard key?"

### Failure 4: Rebalancing Data Loss

**What:** During a rebalancing operation, 15 minutes of data is lost. The on-call engineer initiated a rebalance from 16 to 32 shards at 2:00 AM. At 2:15 AM, the engineer cut over to the new routing table. Some writes during the dual-write phase were not properly synchronized.

**Why:** The dual-write phase of rebalancing has a race condition. Writes to a key that moved from shard 0 to shard 16 during the migration window were written to shard 0 but not yet replicated to shard 16 when the routing table flip occurred. After the flip, reads and new writes go to shard 16, which does not have the last 15 minutes of updates for that key.

**Blast radius:** All keys that moved during the rebalancing window (approximately 50 percent of keys for hash modulo rebalancing from N=16 to N=32). The affected users see stale data or missing writes.

**Detection:** A consistency check script runs after rebalancing and finds 52,000 keys with checksum mismatches between old and new locations.

**Recovery:** (1) Reverse the routing table flip to the old topology (all writes go back to old shards). (2) Re-run the rebalancing with a longer dual-write window and a cutoff timestamp. (3) Verify consistency before flipping. (4) For the 52,000 affected keys, replay writes from the old shard's WAL during the window.

**Prevention:** Add a cutoff timestamp to the dual-write phase. After the cutoff, verify that the destination shard has all writes from the window before flipping. Use a checksum comparison as a pre-flight check, not as an after-the-fact audit.

### Failure 5: Cascading Shard Failure

**What:** One shard's primary fails. Patroni promotes a replica. The promoted replica is on the same physical host as another shard's replica. The additional load from the promoted replica causes the host to run out of memory. The OOM killer kills the other shard's replica. That shard's Patroni tries to promote its remaining replica on the same host, restarting the cycle.

**Why:** The deployment topology did not account for the resource overhead of replica promotion during failover. Replicas were colocated on hosts to save costs, and the resource isolation was insufficient.

**Blast radius:** Two shards go down instead of one. The remaining 14 shards absorb the displaced traffic but cannot handle it all. Partial system outage.

**Detection:** PagerDuty receives alerts for shard 3 (initial failure). Two minutes later, alerts for shard 4 (cascading failure). The ops team sees a correlated failure pattern.

**Recovery:** (1) Stop the failover loop by marking both shards' primaries as failed and routing traffic away. (2) Restore from backup or provision new instances on separate hosts. (3) Redesign the deployment topology to ensure no single host carries more than one shard primary.

**Prevention:** Enforce per-host shard separation: at most one primary per physical host. Reserve capacity headroom on each host (at least 30 percent CPU and memory) for failover scenarios. Test failover scenarios in a staging environment.

## Anti-Patterns Hall of Shame

Seven sharding anti-patterns observed in production systems.

**1. The Auto-Increment Shard Key.** Using an auto-increment `user_id` as a range-based shard key. All new registrations go to the last shard. That shard becomes a write hot spot. The first shard (users 1-100M) is cold. This defeats the purpose of sharding. The correct approach: hash the auto-increment value or use a UUID as the shard key.

**2. The Cross-Shard JOIN in the Request Path.** Joining data from two tables on different shards in a user-facing API endpoint. Each request fans out to all shards, merges results, and adds 500ms of latency. The fix: co-locate the tables on the same shard, denormalize the data, or maintain a materialized view.

**3. The Over-Subscribed Shard Count.** Creating 4,096 shards for a 50 GB database. Each shard holds 12 MB of data. The overhead of 4,096 connections, 4,096 routing table entries, and 4,096 monitoring targets exceeds any benefit. The correct approach: 4-8 shards are sufficient for 50 GB.

**4. The Shard Key That Leaks.** Using a shard key that reveals internal database topology. Example: `customer_48192_shard_3`. When users see `shard_3` in error messages or URLs, they can infer the cluster size and distribution. This is a security concern. Never expose shard identity outside the internal network.

**5. The One-Size-Fits-All Shard Key.** Using the same shard key for every table in the database. The `user_id` shard key works for the `users` table but makes the `audit_log` table's queries all scatter-gather (audit logs query by timestamp, not by user_id). The correct approach: each table should have its own shard key based on its access pattern, or use a shard-neutral storage for tables that do not have a natural shard key.

**6. The No-Failover Shard.** Running each shard as a single database instance with no replica. When the shard's server dies, that shard's data is unavailable until a backup is restored. The blast radius is 1/N of data for N shards. The fix: at least one replica per shard, with automatic failover.

**7. The Manual Rebalance.** Triggering rebalancing by manually moving data files between shards with rsync and then updating a configuration file. This always results in data loss or inconsistency. The correct approach: use a proven rebalancing tool (Vitess `reshard`, Cassandra `nodetool rebuild`, custom tool with dual-writes and checksum verification).

## When NOT to Use Sharding

Six scenarios where sharding is the wrong tool.

**1. Total data under 500 GB.** A single PostgreSQL or MySQL instance with proper indexing and 64 GB of RAM handles 500 GB comfortably. Sharding adds complexity without benefit. Try indexing, connection pooling, read replicas, and vertical scaling first.

**2. The workload is read-heavy.** If the workload is 95 percent reads and 5 percent writes, read replicas are a simpler solution. Deploy 3-5 read replicas behind a load balancer. This provides read throughput scaling without the operational complexity of sharding. Sharding only helps when the write throughput of a single node is the bottleneck.

**3. Every query is cross-shard.** If 80 percent of queries cannot include a shard key (because the access pattern is analytical — aggregate across all data, group by date, join across entities), sharding makes every query slower. Consider columnar databases (ClickHouse, Redshift) or distributed analytical databases (Snowflake, BigQuery) instead.

**4. Strong global consistency is required.** If the application requires strict serializable transactions across all data, sharding with cross-shard transactions adds unacceptable latency and complexity. Consider Spanner (globally consistent, auto-sharded) or CockroachDB (distributed SQL with serializable isolation).

**5. The team has no DBA or SRE.** Sharding requires dedicated operational expertise. Someone must monitor shard health, manage rebalancing, handle failover, and tune per-shard configuration. If the team has no database operations experience, postpone sharding until the operational capability exists.

**6. The schema changes frequently.** A table with columns changing weekly is painful to maintain across 64 shards. Each schema change must be rolled out to all shards in a coordinated fashion. If the data model is unstable, consider a schemaless database (MongoDB, Cassandra) that handles schema changes natively.
