## How This Topic Appears in Interviews

Sharding questions appear in three forms: direct "design a sharding strategy" questions, implicit sharding decisions in larger system design problems, and debugging questions about hot spots, data skew, and rebalancing.

### Direct Phrasings

1. "Design a sharding strategy for a social network with 500 million users. How would you distribute the user data across shards?"

2. "How would you shard a ride-hailing trip database that stores 5 billion trips per year? Walk through the shard key selection."

3. "Compare hash-based sharding, range-based sharding, and directory-based sharding. When would you use each?"

4. "Design a shard key for a time-series metrics system that receives 10 million data points per second."

5. "How does consistent hashing work? Draw the hash ring and walk through adding and removing a node."

6. "Our database has a hot spot — one shard receives 10x more traffic than the others. Diagnose the root cause and propose a fix."

7. "Design a multi-tenant SaaS database sharded by tenant. How do you handle a tenant with 100x more data than the average tenant?"

8. "Design a resharding strategy that migrates from 8 to 64 shards without downtime."

9. "How would you handle cross-shard transactions in a sharded database? What are the trade-offs?"

10. "Our sharded system has a scatter-gather query that takes 10 seconds. How do you optimize it?"

### Design Problem Phrasings (Implicit Sharding)

11. "Design Twitter." — The tweets table must be sharded, and the shard key is a critical design decision. Interviewers expect you to discuss sharding without being prompted.

12. "Design YouTube." — Video metadata, view counts, and comments each need different sharding strategies.

13. "Design Uber." — Trip data must be sharded. The city-based sharding approach is a central architectural decision.

14. "Design a global chat application like Discord." — Message storage sharding by guild or conversation is the core scaling challenge.

15. "Design a distributed rate limiter." — Partitioning counters by user or IP affects accuracy, cost, and failure modes.

### Debugging Phrasings

16. "One of our shards is using 80 percent of disk while the others are at 30 percent. What could cause this, and how do you fix it?"

17. "We added a new shard to our cluster, but write throughput did not improve. What's wrong?"

18. "Our scatter-gather queries used to take 2 seconds. Now they take 15 seconds. Nothing changed in the application code. Diagnose."

## What the Interviewer Is Actually Testing

Interviewers ask sharding questions to evaluate five specific competencies:

**1. Data modeling judgment.** Can you look at a dataset and identify the column that appears in 80 percent of WHERE clauses? Can you distinguish between a good shard key (high cardinality, uniform access) and a bad one (low cardinality, hot spot prone)?

**2. Systems thinking.** Do you understand the full architecture — routing, connection pooling, failover, rebalancing, monitoring? Or do you only know the high-level concept?

**3. Trade-off articulation.** When you propose a shard key, can you immediately state what queries become slower as a result? Can you explain why hash-based sharding makes range queries expensive?

**4. Operational awareness.** Do you know what happens when a shard fails? How do you add a shard without downtime? What metrics do you monitor per-shard?

**5. Depth of knowledge.** Can you draw a consistent hash ring and explain virtual nodes? Can you walk through two-phase commit across shards? Can you estimate the number of shards needed for 50 million users?

## Level-by-Level Expectations

### L3 / L4 (Early Career)

**Expectation:** Understand the basic concept of sharding. Can explain why a single database cannot handle 50 million users. Can identify a good shard key from a bad one.

**Sample answer (L3):** "Sharding splits a database into multiple independent databases. Each shard holds a subset of the data. The shard key determines which shard a row goes to. For a user table, I would shard by user_id using hash-based sharding. This distributes users evenly across shards."

**What's missing:** No discussion of routing, rebalancing, consistency, or operational complexity. Acceptable for L3. Not sufficient for L5+.

### L5 (Senior)

**Expectation:** Can design a complete sharded system. Selects the shard key based on query patterns, not just data distribution. Discusses routing strategies, failure modes, and monitoring.

**Sample answer (L5):** "For a social network with 500 million users, I would shard the user table by user_id using hash-based sharding across 256 shards. I chose hash-based sharding because user IDs are opaque and we need uniform write distribution. The shard key is user_id because 95 percent of queries include a user_id filter. I would use a proxy-based router like Vitess VTGate to manage routing and connection pooling. Each shard would have one primary and two replicas for read scaling. I would monitor per-shard latency, disk usage, and connection pool utilization. The trade-off is that queries without user_id require scatter-gather, which I would route to a separate read replica cluster."

**Key differentiators:** Mentions query pattern analysis, proxy routing, per-shard replicas, monitoring, and scatter-gather handling.

### L6 (Staff)

**Expectation:** Can design a global sharded system with multi-region replication, automated failover, and online rebalancing. Discusses the operational runbook, recovery time objectives, and cost optimization.

**Sample answer (L6):** (L5 answer plus): "I would deploy shards across three AWS regions (us-east-1, eu-west-1, ap-southeast-1) with each region serving as the primary for its local shards. Cross-region replication would be asynchronous with a 35-200ms replication lag depending on geographic distance. I would use Patroni for automated per-shard failover with a 30-second recovery time objective. The rebalancing procedure would use dual-writes with a cutoff timestamp and checksum verification before routing table flip. I would implement a circuit breaker at the router layer with a threshold of 5 consecutive failures before marking a shard as unhealthy. For cost optimization, I would use smaller shards (16 vCPU, 64 GB RAM) rather than fewer large shards to minimize the blast radius of any single failure."

**Key differentiators:** Multi-region design, quantified recovery objectives, automated failover, cost awareness.

### L7 / L8 (Principal / Distinguished)

**Expectation:** Can design the sharding infrastructure itself — the routing protocol, the metadata store, the rebalancing algorithm. Discusses trade-offs at the systems research level. Can explain why Cassandra chose consistent hashing while Vitess chose hash modulo, and the implications of each choice.

**Sample answer (L7):** (L6 answer plus): "I would implement a custom shard routing protocol based on Raft consensus for the metadata store. The routing table would be replicated across 5 metadata nodes with a 99.999 percent availability target. The rebalancing algorithm would use a variant of the CRUSH algorithm (used by Ceph) to minimize data movement during topology changes, with a theoretical bound of O(log N) data movement per node addition. The consistency model would be configurable per-query: strong consistency for financial transactions (requiring quorum reads and writes) and eventual consistency for social features. I would implement a custom scatter-gather engine that uses speculative execution — sending the query to the least-loaded replica of each shard rather than the primary — to reduce p99 latency by 30-40 percent."

**Key differentiators:** Mentions Raft, CRUSH, speculative execution, configurable consistency levels. Demonstrates deep systems knowledge beyond what is in standard system design books.

## The FAANG Framework for Answering Sharding Questions

When asked a sharding design question, follow this six-step framework.

### Step 1: Clarify the Requirements

Ask three clarifying questions before proposing a solution:

- "What is the total data volume and growth rate?" (This determines the shard count.)
- "What is the dominant query pattern? What column appears in 80 percent of WHERE clauses?" (This determines the shard key.)
- "What is the consistency requirement? Can the system tolerate eventual consistency?" (This determines the replication strategy.)

### Step 2: Estimate the Scale

Calculate the shard count from the data volume:

- "50 million users, 500 KB per user profile = 25 TB of data."
- "Target 250 GB per shard. 25 TB / 250 GB = 100 shards."
- "Round to 128 shards (next power of 2) for hash alignment."

### Step 3: Choose the Shard Key

State the shard key and justify it with the query pattern analysis:

- "I choose user_id as the shard key because 95 percent of queries filter by user_id."
- "I use hash-based sharding for uniform distribution."
- "The trade-off is that queries for 'all users who signed up last week' require scatter-gather."

### Step 4: Design the Routing Layer

Describe the routing architecture:

- "I use a proxy-based router (Vitess VTGate) for centralized routing."
- "The routing table is stored in etcd and pushed to all router instances."
- "Each router maintains a connection pool of 50 connections per shard."
- "The router is stateless and can be scaled horizontally behind a load balancer."

### Step 5: Address Consistency and Failover

Discuss the replication strategy and failure handling:

- "Each shard has one primary and two replicas."
- "I use synchronous replication for write-heavy shards and asynchronous for read-heavy."
- "Patroni manages automatic failover with a 30-second RTO."
- "The circuit breaker pattern prevents cascading failures."

### Step 6: Discuss Monitoring and Operations

Describe how you would operate the system:

- "Monitor per-shard latency, disk usage, and connection pool utilization."
- "Alert when any shard's p99 latency exceeds 100ms or disk usage exceeds 80 percent."
- "Run a weekly rebalancing check to detect data skew."
- "Test the failover procedure quarterly by killing a random shard's primary."

## Phrase Bank

Fifteen phrases that demonstrate expertise in a sharding interview. Use these naturally in your answers.

**"The shard key must match the dominant query pattern."** Signals that you understand the most important principle of sharding design.

**"Hash-based sharding for uniform distribution, but range queries become scatter-gather."** Shows you understand the fundamental trade-off.

**"The shard key should be in the WHERE clause of 80 percent of queries."** Provides a concrete heuristic for shard key selection.

**"Each shard should be 100-300 GB at peak projected size."** Shows capacity planning awareness.

**"Consistent hashing with virtual nodes limits data movement to 1/N during scaling."** Demonstrates knowledge of the rebalancing algorithm.

**"I would use a proxy-based router for centralized routing and connection pooling."** Shows understanding of the routing architecture.

**"Cross-shard queries should be routed to a dedicated read replica cluster."** Demonstrates operational sophistication.

**"The scatter-gather latency is the maximum of all shard latencies, not the average."** Shows understanding of the distributed query execution model.

**"I would implement a circuit breaker per shard with a threshold of N consecutive failures."** Demonstrates resilience engineering.

**"Monitor per-shard, not per-cluster. Cluster averages hide hot spots."** Shows operational awareness.

**"Dual-writes with a cutoff timestamp and checksum verification before the routing table flip."** Demonstrates knowledge of safe rebalancing procedures.

**"The CAP trade-off in sharded systems: synchronous replication for consistency, asynchronous for availability."** Shows understanding of distributed systems theory.

**"Two-phase commit adds 30-80ms of latency and should be avoided in the request path."** Demonstrates practical knowledge of cross-shard transactions.

**"Pre-allocate shards in powers of 2 to simplify hash modulo rebalancing."** Shows capacity planning foresight.

**"The last-resort nature of sharding: try indexing, read replicas, and vertical scaling first."** Demonstrates mature judgment about when to shard.

## Red Flags

Eight specific statements or behaviors that indicate a candidate does not understand sharding.

**Red flag 1: "I would shard by auto-increment ID."** An auto-increment ID as a range-based shard key creates a hot spot on the last shard. All new writes go to one shard. The correct answer: hash the ID or use a UUID.

**Red flag 2: "More shards are always better."** Each shard adds overhead: connections, routing table entries, monitoring targets, and a chance of failure. The optimal shard count is the smallest number that keeps each shard within 100-300 GB.

**Red flag 3: "I would use the same shard key for all tables."** Different tables have different access patterns. The `users` table queries by `user_id`. The `audit_log` table queries by `timestamp`. They should have different shard keys.

**Red flag 4: "Sharding solves all database problems."** Sharding solves write throughput and data volume problems. It does not solve query optimization, indexing, or connection pooling problems. It makes cross-shard queries slower.

**Red flag 5: "I would rebalance by copying data files with rsync."** Manual file-level rebalancing always causes data loss or inconsistency. The correct approach: dual-writes with checksum verification and atomic routing table updates.

**Red flag 6: "I do not need to monitor individual shards."** Cluster-wide averages hide hot spots. A cluster can show 50ms average latency while one shard is at 500ms and the rest are at 10ms.

**Red flag 7: "I would shard every table, even lookup tables."** Lookup tables (country codes, status labels, configuration) are small and rarely change. They should be replicated to each shard or stored in a shared cache, not sharded.

**Red flag 8: "I can change the shard key later."** Changing the shard key requires moving every row to a new shard based on the new key. This is a full data migration. Choose the shard key carefully the first time.

## Practice Drills

Five timed prompts to practice before the interview. Set a timer for each and answer out loud or in writing.

### Drill 1: Shard Key Selection (5 minutes)

A video streaming platform stores a `views` table with columns: `view_id`, `video_id`, `user_id`, `timestamp`, `duration_watched`, `country`. The table receives 10 million new rows per day. The most common queries are: "show total views for video X in the last 24 hours" (70 percent), "show my watch history" (20 percent), "aggregate views by country for monthly report" (10 percent). Choose a shard key and justify.

### Drill 2: Capacity Planning (5 minutes)

You need to design a sharded database for 200 million users. Each user profile is 5 KB. User data grows by 20 percent per year. The system must support 15,000 writes per second peak. Each shard can handle 5,000 writes per second and 300 GB of storage. How many shards do you need? Show the math.

### Drill 3: Rebalancing Without Downtime (10 minutes)

Your system has 16 shards using hash modulo sharding. You need to scale to 32 shards. Walk through the rebalancing process step by step. What can go wrong? How do you ensure consistency?

### Drill 4: Fix the Hot Spot (10 minutes)

Your 64-shard system has one shard with 500ms p99 latency while the other 63 shards have 15ms p99 latency. The hot shard has 2x the data of the average shard and receives 5x the writes. Diagnose the root cause. Propose three solutions, ranked by feasibility.

### Drill 5: Design a Sharded Chat System (15 minutes)

Design a sharding strategy for a global chat application like Discord. The system stores 500 million messages per day across 1 million guilds (servers). The dominant query is "show the last 50 messages in channel X." Choose the shard key, estimate the number of shards, describe the routing architecture, and explain how you handle large guilds.
