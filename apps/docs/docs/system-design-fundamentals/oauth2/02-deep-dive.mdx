# OAuth2 — Under the Hood

You already understand what OAuth2 is at a high level. Now you need to know how it works under the hood — the wire protocols, the data structures, the state machines, and the failure modes that every production system must handle. This page walks you through the internal machinery of the Authorization Code flow with PKCE, the anatomy of tokens, the request lifecycle end to end, and the configuration knobs you will tune in a real deployment.

## Authorization Code Flow with PKCE

The Authorization Code flow is the most secure OAuth2 grant type for public clients (single-page apps, mobile apps) and confidential clients (server-side web apps). PKCE (Proof Key for Code Exchange, pronounced "pixie") protects against authorization code interception even when the client cannot keep a client secret.

At its core, the flow involves four parties and a small set of cryptographic values. Here is the full picture.

### Step-by-Step Walkthrough

**Step 1 — Client generates PKCE parameters.** Before redirecting the user, the client creates a cryptographically random `code_verifier` (a high-entropy string, 43-128 characters from the unreserved character set). It then computes `code_challenge = BASE64URL-ENCODE(SHA256(code_verifier))`. The client stores `code_verifier` in session state.

**Step 2 — Authorization request.** The client constructs a URL to the Authorization Server's `/authorize` endpoint with these query parameters:

```
GET /authorize?
  response_type=code
  &client_id=sypher-web-app
  &redirect_uri=https://app.sypher.io/callback
  &scope=openid%20profile%20email
  &state=3f8a2b1c-7d4e-4f6a-9b2c-1d3e5f7a8b0c
  &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
  &code_challenge_method=S256
```

The `state` parameter is a nonce that binds the request to the session, preventing CSRF attacks on the redirect flow.

**Step 3 — User authentication.** The Authorization Server authenticates the Resource Owner. How this happens is outside OAuth2 — it might be a login form, SSO, or existing session cookie. The server MUST authenticate the user before issuing any code.

**Step 4 — User consent / scope grant.** The server presents the requested scopes to the user and obtains explicit consent. The user can reduce the scope (e.g., grant only `openid` and `profile` but not `email`). The server records this consent decision.

**Step 5 — Authorization code issued.** The server generates a short-lived, one-time-use authorization code and issues a 302 redirect:

```
HTTP/1.1 302 Found
Location: https://app.sypher.io/callback?
  code=4f6a9b2c-1d3e5f7a-8b0c3f8a-2b1c7d4e
  &state=3f8a2b1c-7d4e-4f6a-9b2c-1d3e5f7a8b0c
```

The client MUST verify that the returned `state` matches the value it sent.

**Step 6 — Client receives the code.** The browser follows the redirect to the client's callback endpoint. The client now has the authorization code.

**Step 7 — Token request with code_verifier.** The client exchanges the authorization code for tokens by calling the Token Endpoint:

```
POST /token HTTP/1.1
Host: auth.sypher.io
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=4f6a9b2c-1d3e5f7a-8b0c3f8a-2b1c7d4e
&redirect_uri=https%3A%2F%2Fapp.sypher.io%2Fcallback
&client_id=sypher-web-app
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

**Step 8 — Server verifies PKCE.** The Authorization Server recomputes `SHA256(code_verifier)` and compares it to the `code_challenge` it received in Step 2. If they match, the client is proven to be the same party that initiated the flow. This prevents an attacker who intercepted the authorization code from redeeming it.

**Step 9 — Tokens issued.** The server responds with a JSON body containing the access token, refresh token (if requested), token type, and expiration:

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "scope": "openid profile email"
}
```

**Steps 10-12** — The client uses the access token to call the Resource Server and renders the response for the user.

## Data Structures

Every OAuth2 deployment relies on four core data structures. You need to understand their shapes and constraints to design a secure system.

### Authorization Code

The authorization code is the simplest and most ephemeral data structure in OAuth2. It is a short-lived, one-time-use credential that proves the Resource Owner granted consent.

  ┌─ expires_at - created_at <= 600 seconds (10 min)
  └─ consumed_at IS NULL on first use, set on exchange

**Key properties:**
- **TTL**: MUST be short-lived. The OAuth2 spec recommends 10 minutes or less. In practice, 60-300 seconds is standard.
- **One-time use**: Once exchanged for a token, the code is marked as consumed. Any subsequent attempt to redeem it returns `invalid_grant`.
- **Binding**: The code is bound to a specific `client_id`, `redirect_uri`, and `code_challenge`. The server MUST validate all three during exchange.
- **No client secret for PKCE**: With PKCE, the code is bound to the `code_challenge` instead of a client secret, making it safe for public clients.

### Refresh Token

Refresh tokens allow the client to obtain new access tokens without involving the Resource Owner. They are long-lived credentials that must be stored securely.

<AsciiDiagram
  id="oauth2-deep-dive/refresh-token-structure"
  alt="Refresh token structure showing opaque token string, client_id, user_id, scope, expires_at, and rotation counter."
  caption="Refresh token data structure"
  title="Figure 4"

content={`
┌─────────────────────────────────────────────────────┐
│                  Refresh Token                        │
├─────────────────────────────────────────────────────┤
│  token_hash       string   (SHA-256 of opaque token) │
│  client_id        string   (FK to client_reg)        │
│  user_id          uuid     (FK to user)              │
│  scope            string[]                           │
│  expires_at       timestamp                          │
│  created_at       timestamp                          │
│  rotation_count   integer  (starts at 0)             │
│  family_id        uuid     (groups rotated tokens)   │
│  revoked_at       timestamp  (nullable)              │
└─────────────────────────────────────────────────────┘

                        ── is a ── opaque string ──> Client stores it
                        ── is a ── DB hash ─────────> Server looks it up
`} />

**Key properties:**
- **Opaque by design**: Unlike access tokens, refresh tokens are opaque strings, not JWTs. The client cannot decode them. The server stores a hash for lookup.
- **Rotation**: Each time you use a refresh token, the server issues a new one and invalidates the old one. This limits the damage window if a token is stolen.
- **Family grouping**: Rotated tokens belong to the same `family_id`. If a compromised token from a previous rotation is used, the server detects the replay and revokes the entire family.
- **Scope narrowing**: A refresh token is bound to the scope of the original grant. You cannot escalate scope through a refresh — you can only request a subset of the original scope.

## Consistency

Consistency in OAuth2 is about ensuring that tokens, codes, and grants remain in a valid state across the system. You need to handle expiration, revocation, and scope changes correctly to avoid security holes or broken user experiences.

### Authorization Code Expiration

Authorization codes are short-lived by design. After the code expires, any attempt to exchange it must be rejected. The server should enforce this both at the database level and the application level.

```sql
-- Database-level enforcement: delete expired codes
DELETE FROM authorization_codes
WHERE expires_at < NOW() - INTERVAL '5 minutes';

-- Application-level check (pseudocode)
function validateCode(code):
    authCode = db.query(
        "SELECT * FROM authorization_codes WHERE code = :code",
        { code: code }
    )
    if authCode is None:
        return error("invalid_grant", "Code not found")

    if authCode.consumed_at is not None:
        return error("invalid_grant", "Code already used")

    if authCode.expires_at < now():
        return error("invalid_grant", "Code expired")

    # All checks passed — proceed to PKCE verification
    return authCode
```

### Token Revocation

You need a mechanism to revoke tokens when a user logs out, changes their password, or suspects a breach. The OAuth2 revocation endpoint (`POST /revoke`) allows clients to invalidate tokens proactively.

```json
// POST /revoke
// Content-Type: application/x-www-form-urlencoded
//
// token=eyJhbGciOiJSUzI1NiIs...
// &token_hint=access_token
// &client_id=sypher-web-app
// &client_secret=sypher-web-app-secret

// Response: HTTP 200 OK (always, even if token was invalid)
// This prevents attackers from probing which tokens are valid.
```

The revocation algorithm works as follows:

1. The client sends the token to the revocation endpoint with a `token_hint` (optional, tells the server which type of token was sent).
2. The server authenticates the client (same as token endpoint).
3. The server checks if the token is an access token or refresh token.
4. If it is an access token (JWT), the server adds the `jti` to a revocation blacklist with a TTL equal to the token's remaining lifetime.
5. If it is a refresh token, the server sets `revoked_at` in the database and rotates the family.
6. The server ALWAYS returns `200 OK` — even if the token was already revoked, expired, or unknown. This prevents token enumeration attacks.

### Scope Changes

What happens when you change the scopes granted to a client? The answer depends on the direction of the change:

**Scope expanded (added new permissions):**
- Existing access tokens retain their original scope — they are not upgraded.
- The next refresh token exchange returns tokens with the old scope, unless the user reauthorizes.
- The user must go through the authorization flow again to grant the new scopes.

**Scope reduced (removed permissions):**
- Existing access tokens continue to work until they expire.
- The next refresh token exchange returns tokens with the reduced scope.
- The server silently narrows the scope — no user interaction needed.
- However, the Resource Server should check the `scope` claim in the JWT and deny access to endpoints that require the removed scope.

## Performance

OAuth2 token endpoints are latency-sensitive because they sit on the critical path of every API request (when using short-lived tokens that require frequent refreshes). Here are the performance characteristics you need to plan for.

### Token Endpoint Latency

The token endpoint typically involves several database operations and a cryptographic signing operation. Here is a breakdown of the expected latency for each stage:

```
Token endpoint latency budget (p95, milliseconds):

  ┌─────────────────────────────────────────────────────┐
  │  Stage                    Time     Cumulative        │
  ├─────────────────────────────────────────────────────┤
  │  Network (TLS handshake)    15ms       15ms          │
  │  Request parsing             2ms       17ms          │
  │  Client authentication      10ms       27ms          │
  │  Code lookup (DB read)       5ms       32ms          │
  │  PKCE verification           1ms       33ms          │
  │  JWT signing (RS256)        12ms       45ms          │
  │  Refresh token hash (DB)     5ms       50ms          │
  │  Response serialization      2ms       52ms          │
  └─────────────────────────────────────────────────────┘

  Total: ~50ms p95 for a well-tuned server.

  Bottlenecks:
  ┌─ Database read latency (code lookup + refresh token)
  ├─ JWT signing (RSA is ~10x slower than HMAC)
  └─ Client authentication (bcrypt comparison for secret)
```

**Optimization strategies:**

- **Database read replicas** — Authorization code lookups are read-once, so they can use replicas. The consumed_at write goes to the primary.
- **JWT signing algorithm** — Consider using EdDSA (Ed25519) instead of RS256. EdDSA is faster and produces smaller signatures. If you must use RSA, use a hardware security module (HSM) or a dedicated signing service.
- **Connection pooling** — The token endpoint opens a database connection on every request. Pooling reduces connection overhead from ~5ms to under 1ms.
- **Caching client registrations** — Client metadata (secrets, allowed grant types) changes infrequently. Cache it in memory with a 60-second TTL to avoid a database read on every token request.

### Refresh Token Rotation Cost

Refresh token rotation adds a write to the database for every refresh. This is cheap but not free. Here is how the cost scales:

```
Refresh token rotation cost:

  Without rotation:
    ┌─ Read refresh token hash from DB (5ms)
    ├─ Issue new access token (12ms signing)
    └─ Return response (2ms)
    Total: ~19ms

  With rotation:
    ┌─ Read refresh token hash from DB (5ms)
    ├─ Verify rotation counter (1ms)
    ├─ Mark old token as stale (5ms write)
    ├─ Issue new access token (12ms signing)
    ├─ Insert new refresh token hash (5ms write)
    └─ Return response (2ms)
    Total: ~30ms  (+58% overhead)

  The overhead is acceptable because:
  ┌─ Refresh is infrequent (every 3600s for 1-hour tokens)
  ├─ Security gain outweighs latency cost
  └─ Writes can be batched or async-committed
```

### Token Size and Network Impact

JWT access tokens can grow large if you include many claims. This affects network latency, especially for mobile clients and high-throughput APIs.

```
Access token size considerations:

  ┌─────────────────────────────────────────────────────┐
  │  Claims included          Size     Example           │
  ├─────────────────────────────────────────────────────┤
  │  Minimal (iss, sub,       350B     Standard JWT      │
  │    aud, exp, iat, jti)                               │
  │  With scope               400B     +50B              │
  │  With custom claims       500B     +100B             │
  │  With groups/roles        800B     +300B             │
  │  With permissions list   1.2KB     +400B             │
  └─────────────────────────────────────────────────────┘

  At 10,000 requests/second:
  ┌─ 350B token → 3.5 MB/s bandwidth
  ├─ 800B token → 8.0 MB/s bandwidth
  └─ 1.2KB token → 12.0 MB/s bandwidth

  Keep tokens under 1KB for optimal performance.
  Use a separate userinfo endpoint for large payloads.
```

## Configuration

Every OAuth2 deployment has a set of configuration parameters that you must tune for your security and performance requirements. Here are the critical ones.

### Grant Types

The `grant_types` configuration controls which OAuth2 flows a client is allowed to use. You should restrict this to the minimum set required.

```json
{
  "grant_types": {
    "authorization_code": {
      "enabled": true,
      "pkce_required": true,
      "pkce_challenge_method": "S256"
    },
    "refresh_token": {
      "enabled": true,
      "rotation_enabled": true,
      "reuse_detection": "revoke_family"
    },
    "client_credentials": {
      "enabled": true,
      "allowed_for": ["machine-to-machine", "service-accounts"]
    },
    "implicit": {
      "enabled": false,
      "reason": "Deprecated — use authorization_code with PKCE"
    },
    "password": {
      "enabled": false,
      "reason": "Deprecated — use authorization_code with PKCE"
    },
    "device_code": {
      "enabled": true,
      "interval": 5,
      "expires_in": 600
    }
  }
}
```

**Key decisions:**
- **Authorization code + PKCE** is the default for all interactive clients. Require PKCE even for confidential clients — it adds defense in depth.
- **Client credentials** is for machine-to-machine communication. The client authenticates with its secret and receives tokens directly, without user involvement.
- **Implicit and password grants** are deprecated. Do not enable them in new deployments.
- **Device code** is for input-constrained devices (smart TVs, CLI tools). The user completes authentication on a separate device.

### Token TTL

Token lifetimes are a tradeoff between security and performance. Shorter lifetimes reduce the damage window for stolen tokens but increase the load on the token endpoint.

```json
{
  "token_ttl": {
    "authorization_code": {
      "value": 300,
      "unit": "seconds",
      "note": "5 minutes — enough for user to complete flow"
    },
    "access_token": {
      "value": 3600,
      "unit": "seconds",
      "note": "1 hour — standard for most web apps"
    },
    "refresh_token": {
      "value": 2592000,
      "unit": "seconds",
      "note": "30 days — long-lived but rotatable"
    },
    "id_token": {
      "value": 3600,
      "unit": "seconds",
      "note": "1 hour — same as access token"
    }
  }
}
```

**Tuning guidelines:**

- **Access token TTL**: Start with 1 hour. For high-security apps, reduce to 15 minutes. For mobile apps with unreliable connectivity, extend to 24 hours and rely on refresh token rotation.
- **Refresh token TTL**: Start with 30 days. For high-security apps, reduce to 7 days. Users will need to reauthenticate when the refresh token expires, which is a friction point.
- **Authorization code TTL**: Keep this at 5 minutes or less. The code is only in transit during the redirect — it should be exchanged immediately.
- **Sliding sessions**: Some implementations extend the refresh token TTL on each use. This creates a "rolling" session that never expires as long as the user is active. Implement this carefully — it can lead to sessions that never expire.

### Refresh Token Rotation

Rotation is a critical security feature. Here is the configuration surface:

```json
{
  "refresh_token_rotation": {
    "enabled": true,
    "rotation_on_every_refresh": true,
    "reuse_detection": {
      "strategy": "revoke_family",
      "grace_period_seconds": 60
    },
    "max_rotations_per_family": 1000,
    "absolute_lifetime_seconds": 7776000
  }
}
```

**Parameters explained:**

- **rotation_on_every_refresh**: When true, every refresh token exchange produces a new refresh token and invalidates the old one. This is the recommended setting.
- **reuse_detection.strategy**: When the server detects a reused (stale) refresh token, it can either reject the request (`reject`) or revoke the entire token family (`revoke_family`). The `revoke_family` strategy is more aggressive and prevents token replay attacks.
- **grace_period_seconds**: A window during which the old refresh token is still accepted. This handles race conditions where the client sends the old token after receiving a new one but before the server processes the rotation.
- **max_rotations_per_family**: Limits the number of times a token can be rotated. After this limit, the client must reauthenticate. This prevents infinite token accumulation.
- **absolute_lifetime_seconds**: The maximum lifetime of a token family regardless of rotation. After this time, all tokens in the family expire.

### CORS Configuration

For browser-based clients that call the token endpoint directly (not recommended, but sometimes necessary), you need to configure CORS.

```json
{
  "cors": {
    "allowed_origins": [
      "https://app.sypher.io",
      "https://admin.sypher.io",
      "https://staging.sypher.io"
    ],
    "allowed_methods": ["POST", "OPTIONS"],
    "allowed_headers": [
      "Content-Type",
      "Authorization",
      "X-Requested-With"
    ],
    "allow_credentials": true,
    "max_age_seconds": 3600
  }
}
```

**Security rules:**
- Never use wildcard origins (`*`) for the token endpoint — this would allow any website to make token requests.
- The `allowed_origins` list should match the `redirect_uris` registered for the client.
- `allow_credentials` must be true if the client sends cookies or HTTP Basic auth.
- Preflight requests (OPTIONS) should be handled quickly — cache the results with `max_age` to reduce latency.

### Client Authentication Methods

Confidential clients need to authenticate when calling the token endpoint. The `token_endpoint_auth_method` configuration controls how this authentication works.

```json
{
  "client_authentication": {
    "client_secret_basic": {
      "enabled": true,
      "description": "HTTP Basic auth — client_id:client_secret as base64"
    },
    "client_secret_post": {
      "enabled": true,
      "description": "client_id and client_secret in POST body"
    },
    "client_secret_jwt": {
      "enabled": false,
      "description": "JWT assertion signed with client secret"
    },
    "private_key_jwt": {
      "enabled": true,
      "description": "JWT assertion signed with client's private key"
    },
    "none": {
      "enabled": true,
      "description": "For public clients with PKCE"
    }
  }
}
```

**Recommendations:**
- Use `client_secret_basic` for server-side web apps — it is the simplest and most widely supported.
- Use `private_key_jwt` for high-security environments — the client proves possession of a private key, and no shared secret travels over the wire.
- Never use `none` for confidential clients. It bypasses authentication entirely.
- The `client_secret_jwt` method is a middle ground: it uses a shared secret but avoids sending it in the clear. Enable it if you need JWT-based client auth but the client cannot manage a private key.
