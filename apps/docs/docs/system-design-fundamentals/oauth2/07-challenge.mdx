**Model Answer:**

The Authorization Code grant is the most secure OAuth2 flow for server-side applications because the access token is never exposed to the user agent.

**Step-by-step:**

1. **User initiates login** — The user clicks "Sign in with Provider" on the client application. The client constructs an authorization request URL with query parameters: `client_id`, `redirect_uri`, `response_type=code`, `scope`, and a cryptographically random `state` value to prevent CSRF attacks.

2. **Authorization request** — The browser is redirected to the authorization server's `/authorize` endpoint. The user authenticates (username/password, SSO, MFA, etc.) and consents to the requested scopes.

3. **Authorization code returned** — The authorization server redirects the browser back to the client's `redirect_uri` with a temporary `code` (authorization code) and the original `state` parameter. The client MUST verify that `state` matches the value sent in step 1 to prevent CSRF.

4. **Code exchanged for token** — The client makes a server-to-server POST request to the authorization server's `/token` endpoint. This request includes the `authorization_code`, `client_id`, `client_secret`, and `redirect_uri`. Because this exchange happens over a secure backchannel, the authorization code and client secret are not exposed to the browser.

5. **Access token issued** — The authorization server validates the code (which is short-lived, typically 1-10 minutes), verifies the client secret, and returns an `access_token`, optionally a `refresh_token`, and `expires_in`.

6. **API access** — The client uses the `access_token` in the `Authorization: Bearer <token>` header to call the resource server's APIs.

**Security properties:** The authorization code is single-use and short-lived. The access token travels only over the server-to-server backchannel, never through the browser. The `state` parameter prevents CSRF. The `client_secret` authenticates the client. This flow is resistant to token interception by malicious scripts in the browser.

### Challenge 1.2 — Explain PKCE

**Prompt:** What problem does PKCE (Proof Key for Code Exchange) solve? Describe the mechanism step by step, including the roles of `code_verifier` and `code_challenge`. Explain how PKCE protects mobile and single-page applications that cannot safely store a client secret.

**Model Answer:**

PKCE (pronounced "pixy") solves the problem of authorization code interception on public clients — mobile apps and single-page applications that cannot securely store a `client_secret`. Without PKCE, a malicious app registered on the same device could intercept the authorization code returned via the system browser or custom URL scheme and exchange it for an access token.

**The Mechanism:**

1. **Code Verifier Generation** — The client generates a cryptographically random string of 43-128 characters from the unreserved character set (`[A-Za-z0-9-._~]`). This is the `code_verifier`. It is stored locally on the client.

2. **Code Challenge Derivation** — The client computes a `code_challenge` using the `S256` method (SHA-256 hash, then base64url-encoded without padding). Alternatively, `plain` is supported but discouraged. `code_challenge = base64url(sha256(code_verifier))`.

3. **Authorization Request** — The client includes `code_challenge` and `code_challenge_method=S256` in the authorization request to the `/authorize` endpoint. The authorization server stores these values associated with the authorization code.

4. **Authorization Code Returned** — The server redirects back with the authorization code (same as standard Authorization Code flow).

5. **Token Exchange with Proof** — When exchanging the authorization code at the `/token` endpoint, the client includes the **original plaintext** `code_verifier`. The server hashes it with SHA-256 and compares the result to the stored `code_challenge`. Only if they match does the server issue tokens.

**Why this works:** Even if an attacker intercepts the authorization code, they do not possess the `code_verifier` (which was never transmitted and exists only on the client). Without the correct verifier, the token exchange fails. The verifier is single-use, high-entropy, and never sent over the redirect URL — it goes directly to the token endpoint over TLS.

PKCE is now recommended for ALL OAuth2 clients, even confidential ones, as a defense-in-depth measure (RFC 7636).

### Challenge 1.3 — Explain the Client Credentials Grant

**Prompt:** Describe the Client Credentials grant. When should it be used? What are its security trade-offs compared to the Authorization Code flow? Provide a concrete example of a production use case.

**Difficulty:** Easy | **Time:** 4 minutes

---

**Model Answer:**

The Client Credentials grant is an OAuth2 flow where the client authenticates itself — not a user — to obtain an access token. It is designed for machine-to-machine communication.

**Flow:**

1. The client sends a POST request to the `/token` endpoint with `grant_type=client_credentials`, its `client_id`, and `client_secret` (or uses a private key JWT assertion for client authentication).
2. The authorization server validates the client credentials and returns an access token scoped to the client's permissions.
3. The client uses the access token to call the resource server's APIs.

**When to use it:**
- Backend services calling other backend services (e.g., a payment processor calling a fraud detection service).
- Cron jobs or batch processors that need to access APIs on behalf of the application itself.
- Daemon processes, microservice-to-microservice communication, and administrative tasks.

**Security trade-offs compared to Authorization Code:**
- **No user context** — The token represents the client application, not an end user. This is fine for system-level operations but unsuitable for user-specific data access.
- **Long-lived tokens** — Without a refresh or rotation mechanism, clients often request long token lifetimes, increasing the window of exposure if the client secret is compromised.
- **Secret management burden** — The client secret must be stored securely (e.g., a secrets vault, not configuration files). Rotating secrets requires coordination.
- **No user consent** — There is no authorization step where a user approves the scope. The authorization server must enforce scope limits administratively.

**Example:** A background job service that generates nightly analytics reports. The service authenticates with its client credentials to obtain a token, then calls the analytics API to fetch aggregated data. No user is involved because the reports are system-generated and the data is aggregated across all tenants.

### Challenge 2.1 — Design OAuth2 for a Third-Party API

**Prompt:** Design the OAuth2 integration for a third-party API that allows external developers to access user data. The API supports reading user profiles, posting on behalf of users, and reading analytics. Cover the authorization flow, token management, scope design, and developer onboarding.

**Difficulty:** Medium | **Time:** 15 minutes

---

**Model Answer:**

**1. Scope Design**

Define granular, least-privilege scopes:

```
user:profile:read       — Read user's profile (name, email, avatar)
user:profile:write      — Update user's profile
content:post            — Create posts on behalf of the user
content:read            — Read user's posts and drafts
analytics:read          — Read aggregated analytics for the user's account
offline_access          — Obtain a refresh token for long-lived access
```

Scopes are additive — a client can request `user:profile:read` alone without requesting `content:post`. The authorization server presents the requested scopes to the user during the consent screen.

**2. Authorization Flow**

Use the Authorization Code flow with PKCE (mandatory for all clients, even confidential ones):

```
GET /authorize?
  response_type=code&
  client_id={client_id}&
  redirect_uri={registered_uri}&
  scope=user:profile:read+content:post&
  state={random_csrf_token}&
  code_challenge={s256_hash}&
  code_challenge_method=S256
```

**3. Token Management**

- **Access tokens:** Short-lived (15-60 minutes), opaque or JWT format. If JWT, include `iss`, `sub`, `aud`, `exp`, `iat`, `scope`, `client_id`.
- **Refresh tokens:** Longer-lived (days to months), rotation-enabled. Each refresh invalidates the previous refresh token and issues a new one (refresh token rotation prevents replay if a refresh token is leaked).
- **Token revocation:** Provide a `/revoke` endpoint (RFC 7009). Clients call this on logout. The authorization server invalidates the token and any derived tokens.

**4. Developer Onboarding**

- **Developer portal** — Register a new application, obtain `client_id` and `client_secret`, configure redirect URIs.
- **Redirect URI validation** — Strict matching, no wildcards, require HTTPS for production.
- **Rate limiting** — Per-client rate limits on the `/token` endpoint to prevent brute force.
- **Sandbox environment** — A separate authorization server with test data for development.

**5. Security Considerations**

- **state parameter** — Mandatory to prevent CSRF on the redirect.
- **Client authentication** — Confidential clients use `client_secret` or `private_key_jwt`. Public clients use PKCE.
- **Audience validation** — The resource server validates that the token's `aud` claim matches its own identifier.
- **Token binding** — Optionally, bind tokens to a TLS session (OAuth Token Binding) or use DPoP (Demonstration of Proof-of-Possession) to prevent token theft.

**Model Answer:**

**Design Overview:**

A hybrid approach combining short-lived tokens for fast validation, a distributed revocation cache for immediate revocation, and a persistent log for audit and recovery.

**1. Token Design**

- **Access tokens:** Short-lived (15 minutes). For most scenarios, simply waiting for expiration is sufficient. Use JWT format so resource servers can validate locally without calling the authorization server.
- **Refresh tokens:** Longer-lived (7 days). Include a `jti` (JWT ID) stored in the revocation database. Support refresh token rotation so each refresh invalidates the previous token.

**2. Revocation Triggers**

- User-initiated logout (revoke current session).
- Admin-initiated revocation (user account disabled, security incident).
- Token rotation (old refresh token automatically revoked when new one issued).
- Scope change (all tokens with outdated scopes revoked).
- Device revocation (user removes a device from their trusted devices list).

**3. Revocation Store Architecture**

**Layer 1 — Bloom Filter in Redis (fast check, may have false positives):**
- When a token is revoked, add its `jti` to a Redis Bloom filter.
- Resource servers check the Bloom filter on every request. If the filter says "not revoked", the token is valid. If "possibly revoked", the resource server falls through to a definitive check.
- O(1) memory: a Bloom filter with 0.1% false positive rate for 10 million tokens uses approximately 170 MB.

**Layer 2 — Positive Set in Redis (definitive check):**
- A Redis hash set mapping `jti -> revocation_timestamp` for confirmed revoked tokens.
- TTL-based auto-expiry: keys expire after the token's original lifetime, so the set doesn't grow unboundedly.
- On a Bloom filter hit, the resource server does a definitive check against this set.

**Layer 3 — PostgreSQL (persistent audit log):**
- Append-only table: `revoked_tokens(jti, client_id, user_id, revoked_at, reason, token_type)`.
- Used for audit, compliance, and replaying the revocation event stream after a cache failure.
- Retention policy: 90 days for active queries, then archival.

**4. Event-Driven Replication**

- Each authorization server instance publishes revocation events to a Kafka topic (`token.revocation`).
- All resource servers and introspection services consume this topic and update their local Bloom filters.
- This ensures eventual consistency within seconds, even across data centers.

**5. Token Introspection (RFC 7662) as Fallback**

- For high-security operations (e.g., financial transactions), the resource server can call the introspection endpoint instead of local validation.
- The introspection endpoint checks the revocation store and returns the token's active status, scopes, and metadata.
- Rate-limit introspection to avoid overwhelming the authorization server.

**6. Monitoring**

- Alert if the Bloom filter false positive rate exceeds 1% (indicates capacity is near limit).
- Track `revocations_per_second`, `introspection_latency_p99`, and `bloom_filter_fill_ratio`.
- If the revocation event bus falls behind, switch to synchronous introspection mode until the backlog clears.

### Challenge 2.3 — Design Multi-Tenancy OAuth2

**Prompt:** Design an OAuth2 system that supports multiple tenants (organizations). Each tenant has its own users, roles, and API resources. A single authorization server must route authentication requests to the correct tenant, enforce tenant-scoped permissions, and prevent cross-tenant data access. Consider tenant isolation, user discovery, and admin APIs.

**Difficulty:** Hard | **Time:** 20 minutes

---

**Model Answer:**

**1. Tenant Identification**

Three approaches, from simplest to most scalable:

**Option A — Subdomain-based (recommended):**
```
acme.auth.example.com/authorize
megacorp.auth.example.com/authorize
```
Each tenant has a unique subdomain. The authorization server resolves the tenant from the subdomain via DNS or a host header lookup. This provides natural tenant isolation and supports wildcard SSL certificates.

**Option B — Path-based:**
```
auth.example.com/acme/authorize
auth.example.com/megacorp/authorize
```
Simpler infrastructure but no natural TLS isolation. The tenant ID is extracted from the URL path.

**Option C — Custom domain:**
```
auth.acme.com/authorize
auth.megacorp.com/authorize
```
Maximum brand customization but requires per-tenant SSL certificate management and DNS configuration.

**2. Tenant Context Propagation**

Each tenant's data is stored in a tenant-scoped database schema or using a `tenant_id` column:

```
tenant_metadata(tenant_id, tenant_name, auth_domain, logo_url, allowed_identity_providers)
tenant_users(tenant_id, user_id, email, hashed_password, mfa_enabled, roles[])
tenant_clients(tenant_id, client_id, client_secret_hash, redirect_uris[], allowed_scopes)
```

**3. Authentication Flow**

```
1. User navigates to acme.auth.example.com/authorize
2. Load balancer terminates TLS, passes Host header to auth server
3. Auth server looks up tenant_id from the Host header
4. User is presented with Acme Corp's branded login page
5. On authentication, the auth server creates a session tied to
   tenant_id + user_id (not just user_id)
6. Access token claims include tenant_id:
   {
     "iss": "https://acme.auth.example.com",
     "sub": "user_12345",
     "tenant_id": "acme",
     "aud": ["api.example.com"],
     "scope": "read write"
   }
```

**4. Tenant Isolation**

- **Database isolation:** Use schema-per-tenant or shared-table-with-tenant_id. For high-security tenants, offer dedicated database instances.
- **Token scope enforcement:** The resource server MUST validate that the `tenant_id` in the token matches the tenant of the requested resource. This prevents tenant A's token from accessing tenant B's data.
- **Client registration isolation:** An admin of tenant A cannot register a client in tenant B. Client IDs are unique within a tenant, not globally (or use a compound key: `tenant_id:client_id`).
- **Rate limiting:** Per-tenant rate limits on the `/token` and `/authorize` endpoints to prevent one tenant from starving others.

**5. User Discovery and Federation**

- **Tenant-scoped user lookup:** `POST /acme/.well-known/openid-configuration` returns tenant-specific OIDC metadata.
- **Identity federation:** Each tenant can configure its own identity providers (SAML, Azure AD, Google Workspace). The authorization server routes to the correct IdP based on the tenant's configuration.
- **SCIM provisioning:** Tenant admins can use SCIM to provision and deprovision users programmatically.

**6. Admin APIs (Tenant-level)**

```
GET  /admin/tenants/{tenant_id}/clients        — List registered apps
POST /admin/tenants/{tenant_id}/clients        — Register a new app
GET  /admin/tenants/{tenant_id}/users          — List users
POST /admin/tenants/{tenant_id}/users/{id}/revoke  — Revoke all tokens
```

**7. Cross-Tenant Concerns**

- **Shared infrastructure:** The authorization server, token signing keys, and event bus are shared. Tenant isolation is enforced at the application layer.
- **Token signing:** Use a single signing key (or a key per tenant for maximum isolation). If using per-tenant keys, include the `kid` (key ID) in the JWT header, and the resource server resolves the tenant's public key.
- **Audit logging:** All events include `tenant_id` for cross-tenant audit and compliance reporting.

**Model Answer:**

**1. PSD2 Requirements Mapping**

PSD2 mandates that banks (ASPSPs) expose APIs to authorized third-party providers (TPPs). The Berlin Group NextGenPSD2 standard defines the OAuth2 profile:

| PSD2 Requirement | OAuth2 Implementation |
|---|---|
| Strong Customer Authentication (SCA) | Multi-factor authentication during the `/authorize` step. Redirect the user to the bank's SCA interface. |
| Transaction-specific consent | The authorization code is bound to a specific payment or account access request. The `scope` parameter specifies the transaction. |
| TPP authentication | The TPP uses its eIDAS certificate (QWAC or QSEAL) for client authentication via `tls_client_auth` or `private_key_jwt`. |
| Non-repudiation | The TPP signs the payment request with its QSEAL certificate. The bank logs the signature. |
| Decoupled SCA | The customer can authorize on a separate device (e.g., banking app on phone) while the TPP flow runs on desktop. |

**2. Authorization Flow**

**Step 1 — TPP initiates payment:**
```
POST /v1/payments
Headers:
  Authorization: Bearer <tpp_token>
  Content-Type: application/json
  Signature: <QSEAL-signed-payload>
Body:
  {
    "paymentId": "UUID",
    "debtorAccount": {"iban": "DE123..."},
    "creditorAccount": {"iban": "FR456..."},
    "instructedAmount": {"currency": "EUR", "amount": "150.00"},
    "remittanceInformation": "Invoice 2024-001"
  }
```

The bank responds with a `202 Accepted` and a `_links.scaRedirect` URL pointing to the authorization endpoint.

**Step 2 — Redirect for SCA:**
```
GET /authorize?
  response_type=code&
  client_id={tpp_client_id}&
  scope=payment:initiate&
  payment_id=UUID&
  redirect_uri={tpp_redirect_uri}&
  state={random}&
  code_challenge={...}&
  code_challenge_method=S256
```

**Step 3 — Strong Customer Authentication:**
The bank presents the payment details to the customer (amount, payee, date). The customer authenticates with at least two factors from:
- **Knowledge:** Something they know (PIN, password)
- **Possession:** Something they have (phone for OTP, hardware token)
- **Inherence:** Something they are (fingerprint, face ID)

The SCA is dynamic — the authentication challenge is generated specifically for this transaction, binding the authorization code to the payment details.

**Step 4 — Consent and Token Issuance:**
After successful SCA, the bank issues an authorization code that is bound to the specific `payment_id`. The TPP exchanges this code for an access token. The token is:
- **Single-use** for payment execution (one API call to confirm the payment).
- **Scoped** to the specific payment (`payment_id` in the token claims).
- **Short-lived** (5 minutes) — the TPP must execute the payment immediately.

**3. Certificate-Based Authentication**

PSD2 requires TPPs to use eIDAS certificates:
- **QWAC** (Qualified Website Authentication Certificate) — authenticates the TPP's website/API.
- **QSEAL** (Qualified Electronic Seal) — signs the payment request for non-repudiation.

The authorization server authenticates the TPP via `tls_client_auth` (mutual TLS using the QWAC certificate) or `private_key_jwt` (signed JWT assertion using the QSEAL certificate).

**4. Regulatory Reporting**

- **Log retention:** All authorization decisions, SCA events, and token issuances are logged with a 5-year retention period (per GDPR/PSD2).
- **Registry check:** The authorization server validates the TPP's certificate against the national competent authority's registry (e.g., BaFin for Germany, FCA for UK) during client registration and periodically thereafter.
- **Audit trail:** Each payment has an immutable audit trail including TPP certificate serial number, SCA method used, timestamp, IP address, and transaction hash.

**5. Security Considerations**

- **Token binding:** Use mTLS or DPoP to bind the access token to the TPP's TLS session, preventing token theft and replay.
- **Consent freshness:** The consent expires after 90 days for AIS (account information). For PIS (payment initiation), consent is single-transaction and expires after the SCA timeout.
- **Rate limiting:** Strict per-TPP rate limits enforced by the PSD2 regulatory framework (e.g., maximum 5 payment attempts per minute per TPP).
- **Fraud monitoring:** Real-time anomaly detection on SCA events — if a user has 5 SCA challenges in 10 minutes, flag as suspicious.

**Model Answer:**

**1. Architecture Overview**

The system has four layers:

**Layer 1 — Identity Provider (IdP):** The source of truth for user identities, groups, and authentication policies. Examples: Azure AD, Okta, or a custom SAML IdP. The IdP handles authentication (including MFA) and feeds identity data to the OAuth2 authorization server via SCIM and a real-time event stream.

**Layer 2 — OAuth2 Authorization Server:** Issues tokens based on the IdP's authentication assertions. Implements token exchange, delegation, and consent management. Supports both internal and external (SaaS) clients.

**Layer 3 — Application Gateway (PEP):** A reverse proxy that enforces token validation, scope checks, and RBAC before forwarding requests to internal applications. This provides a centralized enforcement point without modifying each application.

**Layer 4 — Provisioning Engine:** SCIM-based provisioning that creates, updates, and deactivates user accounts in downstream applications based on events from the IdP.

**2. Authentication Flow**

```
1. User navigates to app.internal.example.com
2. App gateway redirects to auth.example.com/authorize
3. User selects their IdP (e.g., "Login with Azure AD")
4. Browser redirects to Azure AD for authentication
5. Azure AD authenticates the user (SAML or OIDC)
6. Azure AD posts the assertion back to auth.example.com
7. Auth server validates the assertion, maps the user's
   Azure AD groups to local roles, and issues an OAuth2
   token with RBAC claims:
   {
     "sub": "user_12345",
     "email": "user@company.com",
     "groups": ["engineering", "infra-oncall"],
     "roles": ["app:read", "app:write", "admin:deploy"],
     "aud": "app.internal.example.com",
     "iss": "https://auth.example.com",
     "tenant_id": "acme-corp"
   }
8. User is redirected back to the app with the token
9. App gateway validates the token and enforces RBAC
```

**3. Just-in-Time (JIT) Provisioning**

When a user accesses an application for the first time:

1. The app gateway detects the user has no account in the downstream application.
2. The provisioning engine calls the downstream app's SCIM endpoint to create a user account with the appropriate attributes.
3. The user is granted access in real-time.
4. JIT provisioning is disabled for sensitive applications (e.g., finance, HR) where pre-provisioning with manager approval is required.

**4. Role-Based Access Control (RBAC)**

**Group-to-Role Mapping (admin-configured):**
```
Group: "engineering"         → Role: "app:read", "app:write"
Group: "infra-oncall"        → Role: "app:read", "admin:deploy"
Group: "finance"             → Role: "app:read", "finance:approve"
```

**Role evaluation:**
The token includes the user's groups (from the IdP) and the mapped roles (from the authorization server). The application gateway evaluates:
- Is the request's HTTP method allowed by the user's roles? (GET → read, POST → write, DELETE → admin)
- Is the requested resource path allowed by the user's roles?
- Is the user's session still valid (not revoked, not expired)?

**5. Automated Offboarding**

When an employee leaves the company:

1. **IdP event:** The HR system updates the user's status in the IdP (account disabled).
2. **Token revocation:** The IdP publishes a `user.deactivated` event. The authorization server immediately revokes all active tokens for that user by adding their `jti` values to the revocation store.
3. **SCIM deprovisioning:** The provisioning engine sends SCIM `PATCH` requests to all downstream applications to deactivate the user's account. The provisioning engine retries with exponential backoff if a downstream app is unavailable.
4. **Session cleanup:** The user's active sessions in the authorization server are invalidated. Any `refresh_token` issued to the user is added to the revocation list.
5. **Audit trail:** A complete offboarding record is created, including which tokens were revoked, which applications were deprovisioned, and timestamps for each step.

**6. Cross-Domain Federation**

For third-party SaaS applications (e.g., Salesforce, Slack, Workday):

- The authorization server acts as a **SAML/OIDC proxy** between the corporate IdP and the SaaS provider.
- The user authenticates with the corporate IdP once. When accessing a SaaS app, the authorization server generates a SAML assertion or OIDC token for that specific service provider.
- **Attribute mapping:** Corporate groups are mapped to SaaS-specific roles. Example: `group: "engineering"` → `Salesforce: "Developer License"`, `Slack: "#engineering" channel auto-join`.

**7. Security Considerations**

- **Token lifetime:** Internal app tokens: 1 hour (refreshable). SaaS tokens: 15 minutes (no refresh — re-authenticate via the proxy).
- **Device trust:** Integrate with device management (MDM) to check device compliance before issuing tokens. Non-compliant devices are denied or restricted to read-only access.
- **Session management:** Users can view active sessions in a self-service portal and revoke individual sessions. Admins can revoke all sessions for a user or group.
- **Phishing resistance:** Use FIDO2/WebAuthn for MFA where possible. The authorization server validates the `origin` of WebAuthn assertions against the application's registered origin.
