## Interview Questions

OAuth2 questions appear in system design interviews at every FAANG company. They test whether you understand the full flow: authorization server, resource server, client, token types, grant types, and security posture.

### Five Common Questions

**Q1:** "Design an OAuth2-based authorization system for a third-party API platform."

**Q2:** "How does the Authorization Code flow work? Walk through every redirect and token exchange."

**Q3:** "Our mobile app sends user credentials to the backend to authenticate. How do we migrate to OAuth2 without breaking existing clients?"

**Q4:** "Design a token refresh mechanism for a service that handles 100K concurrent users."

**Q5:** "How does OAuth2 handle scoped access? Walk through a design where a calendar app requests read-only access to a user's Google Calendar."

### Three Advanced Questions

**Q6:** "Design a multi-tenant OAuth2 authorization server that supports 1000+ client applications."

**Q7:** "Design a token revocation system that invalidates tokens across a distributed microservice architecture within 5 seconds."

**Q8:** "How would you implement OAuth2 with Proof Key for Code Exchange (PKCE) for a native mobile banking app?"

### Five Additional Questions

**Q9:** "Design an OAuth2 token exchange service for a microservice mesh."

Model answer outline:

```
Requirements:
- Service A receives a user token from the client
- Service A needs to call Service B on behalf of the user
- Service B should not see the original client token
- Token delegation must be auditable and time-bound
- Throughput target: 10K token exchanges per second
- Token lifetime: 5 minutes for exchanged tokens

Design:
- Use the OAuth2 Token Exchange extension (RFC 8693).
- Service A presents the current access token to the
  authorization server (AS) and requests a new token
  scoped to Service B's API.
- The AS validates the original token, checks that
  Service A is authorized to delegate, and issues a
  new token with reduced scope and a shorter TTL.
- Service B validates the exchanged token against
  the AS's public key (JWT) or via introspection.
- All exchanges are logged with correlation IDs for
  audit trail.

Token exchange flow:
  [Client] --(token A)--> [Service A]
                            |
                            +--(token A + scope=B)--> [Auth Server]
                            |                             |
                            |  <--(token B, 5 min TTL)---+
                            |
                            +--(token B)--> [Service B]
                            |
                            +--(response)--> [Client]

Security considerations:
- Token A has scope "calendar:read" and "delegate"
- Token B has scope "calendar:read:user123" — scoped
  down to a specific user and resource
- Exchanged tokens are not refreshable; they must be
  re-exchanged if expired
- Service A's client credentials are also validated
  during exchange to prevent token theft
- Rate limit: 100 exchanges per client per minute

Failure modes:
- Auth server unavailable: Service A caches token B
  until its TTL expires (5 minutes), then fails open
  to the original token A for 30 seconds as fallback
- Revoked token A: exchange returns 401, Service A
  propagates the error to the client
- Clock skew: tokens are checked with 60-second
  leeway on the `nbf` and `exp` claims
```

**Q10:** "How does OAuth2 work with OpenID Connect?"

Model answer outline:

```
OAuth2 vs. OpenID Connect (OIDC):
- OAuth2 is an authorization framework: it issues
  access tokens that grant access to resources.
- OIDC is an authentication layer built on OAuth2:
  it issues an ID token (a JWT) that contains
  identity claims about the user.

Key differences:
┌────────────────────┬────────────────────┬────────────────────┐
│ Aspect             │ OAuth2             │ OIDC               │
├────────────────────┼────────────────────┼────────────────────┤
│ Primary output     │ Access token       │ ID token + access  │
│                    │ (opaque or JWT)    │ token              │
│ Token format       │ Opaque or JWT      │ JWT (required)     │
│ Contains           │ Authorization      │ Identity claims    │
│                    │ (scopes)           │ (sub, name, email) │
│ Standard           │ RFC 6749           │ OIDC spec          │
│ User info endpoint │ Optional           │ Required           │
│ Client types       │ confidential/      │ same + RP-initiated│
│                    │ public             │ logout             │
└────────────────────┴────────────────────┴────────────────────┘

OIDC flow (Authorization Code + OIDC):
1. Client redirects user to the authorization endpoint
   with scope=openid+profile+email (and response_type=code).
2. User authenticates and consents.
3. Auth server redirects back with an authorization code.
4. Client exchanges the code at the token endpoint.
5. Auth server returns:
   - access_token (access to resources)
   - id_token (JWT with identity claims)
   - refresh_token (optional)
6. Client validates the id_token using the auth server's
   JWKS endpoint (JSON Web Key Set).
7. Client extracts user info from the id_token claims
   (sub, name, email, picture).

Key JWT claims in id_token:
{
  "iss": "https://accounts.example.com",
  "sub": "1234567890",
  "aud": "client-id",
  "exp": 1719000000,
  "iat": 1718913600,
  "auth_time": 1718913590,
  "nonce": "n-0S6_WzA2Mj",
  "name": "Jane Doe",
  "email": "jane@example.com",
  "picture": "https://example.com/avatar.jpg"
}

Why OIDC matters:
- Without OIDC, the resource server must call the
  userinfo endpoint to get identity, adding latency
  and coupling.
- The id_token is self-contained: the client can
  authenticate the user without a back-channel call.
- The `nonce` claim prevents replay attacks.
- The `auth_time` claim tells the client when the
  user last authenticated, enabling step-up auth.
```

**Q11:** "Design a token introspection service for a high-throughput microservice architecture."

Model answer outline:

```
Requirements:
- 50 microservices each need to validate access tokens
- 100K token validation requests per second
- Token lifetime: 15 minutes for access tokens
- Token format: opaque (not JWT) for legacy compatibility
- Max validation latency: 5ms p99

Design:

Option A: JWT-based (self-contained validation)
- Auth server issues JWTs signed with RS256.
- Each microservice caches the auth server's public
  key (JWKS) and validates tokens locally.
- No introspection call needed.
- Challenge: token revocation is impossible until
  the JWT expires (15 minutes).
- Mitigation: maintain a short-lived blocklist in
  Redis (TTL = remaining token lifetime) for
  revoked tokens.

Option B: Redis-backed introspection cache
- Auth server maintains a Redis cluster with token
  state (valid, revoked, expired).
- Each microservice checks Redis before accepting
  a token. On cache miss, calls the auth server's
  introspection endpoint.
- Redis replication: 3-node cluster per region.
- Token TTL in Redis: 15 minutes (same as token
  lifetime) or 0 if revoked.

Comparison:
┌────────────────────┬────────────────────┬────────────────────┐
│ Aspect             │ JWT (Option A)     │ Redis (Option B)  │
├────────────────────┼────────────────────┼────────────────────┤
│ Validation latency │ under 1ms (local))       │ 1-2ms (Redis)     │
│ Revocation speed   │ Up to 15 min       │ under 5 seconds        │
│ Infrastructure     │ None (stateless)   │ Redis cluster     │
│ Complexity         │ Low                │ Medium            │
│ Token size         │ Large (~2KB)       │ Small (~20 bytes) │
│ Rate limiting      │ No central limit   │ Limited by Redis  │
└────────────────────┴────────────────────┴────────────────────┘

Recommended: Hybrid approach
- Use JWTs for the common case (local validation,
  under 1ms latency).
- Maintain a small Redis blocklist for revoked
  tokens (checked on every request).
- Blocklist entries are set with TTL = remaining
  token lifetime.
- Fall back to introspection for tokens that are
  not in the blocklist and fail JWT validation.
```

**Q12:** "Design a secure OAuth2 client credential flow for server-to-server communication."

Model answer outline:

```
Requirements:
- Machine-to-machine API calls between services
- No user interaction (no redirect, no consent)
- TTL for tokens: 1 hour
- Throughput: 10K token requests per minute
- Security: client_secret rotation, IP allowlisting

Design:
- Each service registers as a confidential client
  with a client_id and client_secret.
- The client secret is stored in a secrets manager
  (Vault, AWS Secrets Manager, GCP Secret Manager).
- Token request: POST /oauth/token with
  grant_type=client_credentials,
  client_id=service-a, client_secret=...
- Auth server returns: { access_token, expires_in, token_type }

Token lifecycle:
1. Service A starts up, retrieves client_secret from
   secrets manager.
2. Service A requests a token from the auth server.
3. Auth server validates client_id + client_secret,
   checks IP allowlist, verifies scope.
4. Auth server issues a JWT access token with:
   - iss: auth server
   - sub: service-a (client_id)
   - aud: service-b-api
   - scope: "write:orders"
   - exp: now + 3600 seconds
5. Service A caches the token locally and uses it
   for all requests to Service B.
6. Service A refreshes the token 5 minutes before
   expiry (or on 401 response).

Security hardening:
- Client_secret rotation: every 90 days, with a
  24-hour overlap window for both old and new secrets.
- mTLS: require mutual TLS for the token endpoint.
- IP allowlisting: restrict token issuance to known
  IP ranges of Service A.
- Scope limitation: each client gets the minimum
  scope needed (principle of least privilege).
- Rate limiting: 100 tokens per client per hour.
- Token binding: bind the token to the TLS session
  (token binding RFC) to prevent token theft.

Failure modes:
- Secrets manager down: service uses cached secret
  and retries with exponential backoff.
- Token endpoint unavailable: service uses cached
  token until it expires, then fails with a clear
  error message and retry.
- Secret rotation failure: overlap window ensures
  both old and new secrets work during the transition.
```

**Q13:** "How would you handle OAuth2 token refresh in a distributed system?"

Model answer outline:

```
Problem:
A user's access token expires after 15 minutes.
The client must refresh it transparently without
the user noticing. If the refresh token expires
or is revoked, the user must re-authenticate.

Design considerations:
- Refresh tokens have a longer TTL (30 days default,
  configurable).
- Refresh tokens can be rotated (each refresh
  invalidates the previous refresh token and
  issues a new one).
- Refresh tokens must be stored securely (server-side
  or in secure storage on the client).

Token refresh flow:
1. Client detects access token is expired (401 from
   resource server, or checks expiry locally for JWT).
2. Client calls POST /oauth/token with
   grant_type=refresh_token,
   refresh_token=..., client_id=...
3. Auth server validates the refresh token:
   - Check signature and expiry
   - Check if it has been revoked
   - Check if it has been rotated (if rotated, the
     previous refresh token is invalidated)
4. Auth server returns a new access token and a new
   refresh token (if rotation is enabled).
5. Client updates its stored tokens.
6. If the refresh itself fails with 401, the client
   redirects the user to re-authenticate.

Concurrent refresh handling:
- Multiple client instances may attempt to refresh
  simultaneously when the token is near expiry.
- Use a distributed lock (Redis) keyed on the user
  ID + client ID to serialize refresh requests.
- The first request obtains the lock, refreshes,
  and updates the shared token store.
- Subsequent requests wait for the lock, then use
  the already-refreshed token.

Reference architecture:
  [Client A] ----+----> [Token Service]
  [Client B] ----+----> [Token Service]
  [Client C] ----+----> [Token Service]
                   |
                   v
            { Redis distributed lock }
                   |
                   v
            [ Auth Server (AS) ]
                   |
                   v
            [ PostgreSQL ]
            (refresh token store)

Error handling:
- 400: Invalid grant -> refresh token is bad, force
  re-authentication.
- 429: Rate limited -> exponential backoff, max 10
  retries with 30-second cap.
- 5xx: Server error -> retry with backoff, keep the
  old access token for 5 additional seconds (grace
  period).
- Network failure: retry with exponential backoff,
  jitter = 10% of the backoff interval.
```

## Signals Interviewers Look For

### Strong Hire Signals

- Walks through the full Authorization Code flow with redirects, state parameter, authorization code, token exchange, and scope validation without skipping a step.
- Explains the difference between opaque tokens and JWTs, and can justify when to use each.
- Brings up PKCE unprompted when discussing mobile or single-page applications.
- Discusses token revocation mechanisms (token introspection, blocklist, short TTLs) and their tradeoffs.
- Mentions the security implications of the Implicit flow and why it was deprecated in OAuth2.1.
- Can compare OAuth2 grant types (Authorization Code, Client Credentials, Device Code, Refresh Token) and pick the right one for a scenario.
- Articulates the difference between authentication (OIDC) and authorization (OAuth2) and uses both appropriately.
- Quantifies token lifetimes, refresh windows, and revocation latencies in concrete numbers.
- Draws the full token exchange sequence including error paths (expired, revoked, malformed tokens).
- Mentions the threat model: token leakage via referrer headers, CSRF on the redirect URI, authorization code interception, and the mitigations (state parameter, PKCE, redirect URI validation, CORS).

### Strong Hire Behaviors

What a strong hire does in the first 90 seconds of an OAuth2 question:

```
"Before I design the authorization system, I need to
clarify a few things. What kind of clients are we
supporting — web apps, mobile apps, server-to-server?
What is the threat model? Do we need user authentication
or just authorization?"

"Let me walk through the Authorization Code flow step
by step. First, the client redirects the user to the
authorization server with response_type=code, client_id,
redirect_uri, scope, and state. The user authenticates
and consents. The authorization server redirects back
to the client with the code and state. The client
exchanges the code for tokens at the token endpoint.
The state parameter must match to prevent CSRF."

"For mobile apps, I would use PKCE. The client generates
a code_verifier and sends the code_challenge (SHA-256
hash) during the authorization request. When exchanging
the code, it sends the original code_verifier. The auth
server verifies the hash matches. This prevents the
interception of the authorization code."

"I would use JWTs for the access tokens because they are
self-contained. Each microservice can validate the token
locally using the auth server's public key, avoiding a
centralized introspection endpoint. The JWT would contain
the user ID, scopes, and expiration. For revocation, I
would use a short TTL (15 minutes) with a Redis blocklist
for high-value tokens that need immediate revocation."
```

### Neutral Signals

- Knows the basic OAuth2 roles (client, auth server, resource server) but cannot explain the full token exchange flow.
- Mentions "Bearer token" but cannot explain the security implications of bearer tokens or how to mitigate token theft.
- Can describe the Authorization Code flow at a high level but skips the state parameter or PKCE.
- Knows the difference between OAuth2 and OIDC but cannot list the claims in an ID token.
- Mentions scopes but cannot describe how scope checking works in a resource server.
- Knows that tokens expire but cannot explain the refresh token rotation mechanism.
- Can list the grant types but cannot pick the right one for a given scenario.

### Weak Signals

- Confuses authentication with authorization: "OAuth2 is how you log in."
- Proposes storing user passwords in the client and sending them to the auth server.
- Says "I will use the Implicit flow" without mentioning it was deprecated in OAuth2.1.
- Cannot explain the difference between a bearer token and a proof-of-possession token.
- Does not mention the state parameter when describing the Authorization Code flow.
- Uses the same grant type for every scenario without considering the client type.
- Says "access tokens are encrypted" — access tokens are signed, not encrypted. The resource server verifies the signature; it does not need to decrypt the token.
- Cannot name a real-world OAuth2 implementation (Auth0, Okta, Keycloak, AWS Cognito) or describe their relevant features.

## Knowledge Levels

### Level 1: Awareness

You know that OAuth2 is an authorization framework. You can name the four roles: resource owner, client, authorization server, resource server. You have heard of access tokens and refresh tokens. You understand that OAuth2 is not the same as OIDC.

### Level 2: Application

You can implement the Authorization Code flow for a web application. You know when to use PKCE (mobile apps, SPA). You understand the Client Credentials flow for server-to-server communication. You can set up scopes to limit access to specific resources. You know how to refresh tokens transparently. You understand the difference between confidential and public clients.

### Level 3: Mechanism

You understand the full token lifecycle: issuance, validation, refresh, revocation. You can explain how JWT access tokens are signed (RS256, ES256) and how the resource server validates them. You understand the security properties of the Authorization Code with PKCE: how code_challenge and code_verifier prevent interception attacks. You can describe the token exchange protocol (RFC 8693) for delegation scenarios. You understand the security considerations of CORS, redirect URI validation, and referrer header leakage.

### Level 4: Production

You have deployed and operated an OAuth2 authorization server in production. You have debugged token validation failures, clock skew issues, and JWKS rotation problems. You understand the performance characteristics of token introspection vs. local JWT validation under load. You have implemented token revocation with a distributed blocklist. You have configured OAuth2 clients for different application types (web, mobile, SPA, server). You understand the tradeoffs between centralized and decentralized token validation. You have configured OAuth2 audit logging for compliance requirements (SOC 2, GDPR, HIPAA).

### Level 5: Design

You can design a custom OAuth2 authorization server from scratch. You understand the OAuth2 threat model (RFC 6819) and can design mitigations for every attack vector: authorization code interception, CSRF, token theft, redirect URI manipulation, and scope elevation. You can design a multi-region token validation system with sub-5ms latency. You understand the implications of OAuth2.1 changes and can plan a migration path. You could contribute to the OAuth2 working group specifications.

## 30 Phrases to Use

Use these phrases naturally in your answer. They signal depth.

**Client type identification:**
1. "Is this a confidential client that can securely store a client secret, or a public client like a mobile app or SPA?"
2. "For a public client, I would use the Authorization Code flow with PKCE to mitigate the authorization code interception attack."
3. "The Implicit flow is deprecated in OAuth2.1 because the access token is exposed in the URL fragment and cannot be securely bound to a specific client."
4. "For server-to-server communication, the Client Credentials grant is the right choice — no user interaction needed and the client authenticates with its own credentials."

**Grant type selection:**
5. "I choose the Authorization Code grant because it provides a one-time authorization code that is exchanged for tokens on the back channel, keeping the tokens out of the user agent."
6. "PKCE uses a cryptographically random code_verifier and a code_challenge (S256 hash). The auth server verifies the verifier matches the challenge during token exchange."
7. "The Device Code grant is ideal for devices without a browser (smart TVs, CLI tools, IoT). The device polls the token endpoint until the user completes authorization on another device."
8. "I would use the Client Credentials grant with mTLS for stronger authentication than a shared secret."

**Token design:**
9. "I will use JWT access tokens signed with RS256 (asymmetric key). The auth server signs with its private key, and each resource server validates with the public key from the JWKS endpoint."
10. "The JWT contains claims: iss (authorization server), sub (user ID), aud (resource server), exp (expiration), iat (issued at), and scope (permissions)."
11. "I set the access token TTL to 15 minutes. Short enough to limit the damage of a leaked token, long enough to avoid excessive refresh traffic."
12. "Refresh tokens have a 30-day TTL with rotation: each refresh invalidates the previous refresh token and issues a new one. This limits the window for a leaked refresh token."
13. "I use opaque tokens when the resource server cannot validate JWTs (legacy systems) or when the token needs to carry more state than a JWT can practically hold."
14. "The token introspection endpoint (RFC 7662) returns the token's active state, scope, and associated user. It requires the resource server to call the auth server on every request."

**Security:**
15. "The state parameter is a CSRF token that must be validated on the redirect back to the client. Without it, an attacker can inject their own authorization code."
16. "Redirect URI validation prevents open redirector attacks. The auth server must match the exact redirect URI registered for the client, not just the domain."
17. "Bearer tokens are like cash: whoever holds them can use them. I mitigate this with short TTLs, TLS everywhere, token binding, and certificate pinning."
18. "Token binding (RFC 8471) cryptographically binds the token to the TLS session. A stolen token cannot be used from a different TLS connection."
19. "For scope elevation attacks, the auth server must validate that the requested scope is within the client's allowed scope set and the user's granted scope set."
20. "The authorization code is a one-time credential. If the auth server receives the same code twice, it must reject both requests and revoke all tokens issued with that code."

**Token lifecycle:**
21. "Token refresh is transparent to the user. The client detects a 401 or checks the token expiry, refreshes in the background, and retries the original request."
22. "Refresh token rotation means that each refresh request returns a new access token and a new refresh token. The old refresh token is invalidated. This prevents a leaked refresh token from being used repeatedly."
23. "For token revocation, I maintain a blocklist in Redis with the token's jti (JWT ID) as the key and the revocation timestamp as the value. The TTL is set to the remaining token lifetime."
24. "The JWKS endpoint should support key rotation. The auth server publishes a new key pair before the old key expires, and both keys are valid during the rotation window."

**OIDC and identity:**
25. "OpenID Connect adds an id_token (JWT) that contains identity claims. The client validates the id_token and extracts the user's sub, name, email, and picture."
26. "The nonce claim in the id_token prevents replay attacks. The client generates a nonce, sends it in the authentication request, and validates it in the id_token."
27. "The userinfo endpoint returns additional claims about the user. It is called with the access token, not the id_token, and is protected by OAuth2 scopes."
28. "For logout, OIDC provides RP-initiated logout (session management) and the logout endpoint that terminates the session at the OP (OpenID Provider)."

**Production concerns:**
29. "Clock skew between the auth server and resource servers can cause valid tokens to be rejected. I configure a 60-second leeway on the exp and nbf claims."
30. "JWKS caching is essential for performance. The resource server caches the JWKS for 1 hour by default, but the auth server should include a Cache-Control header with max-age and must-revalidate directives."

## 15 Red Flags

These statements reveal gaps in your understanding. Avoid them.

**Red Flag 1:** "OAuth2 is an authentication protocol."

Wrong. OAuth2 is an authorization framework, not authentication. It grants access to resources, it does not authenticate the user. OpenID Connect is the authentication layer built on OAuth2. Say "OAuth2 authorizes access to resources. For authentication, I layer OpenID Connect on top to get identity claims via the id_token."

**Red Flag 2:** "I will use the Implicit flow because it is simpler."

The Implicit flow was deprecated in OAuth2.1. It exposes the access token in the URL fragment and cannot securely bind the token to the client. The Authorization Code flow with PKCE is the recommended replacement for all client types. Say "I use the Authorization Code flow with PKCE for all client types, including SPAs and mobile apps."

**Red Flag 3:** "I will store the client secret in the mobile app."

A client secret in a mobile app is not a secret — it can be extracted by reverse engineering. Mobile apps are public clients. They cannot securely store secrets. Say "For mobile apps, I use PKCE instead of a client secret. The code_verifier is generated at runtime and never stored persistently."

**Red Flag 4:** "I will put the access token in the URL query string."

Access tokens in the URL are exposed to browser history, server logs, referrer headers, and bookmarks. This is a security violation. Say "Access tokens are sent in the Authorization header using the Bearer scheme (Authorization: Bearer &lt;token>)."

**Red Flag 5:** "Access tokens are encrypted."

Access tokens are signed, not encrypted. The resource server verifies the signature to ensure the token was issued by the trusted authorization server and has not been tampered with. Encryption is not needed because the token is sent over TLS. Say "Access tokens are signed with RS256. The resource server validates the signature using the auth server's public key from the JWKS endpoint."

**Red Flag 6:** "I will use the same access token for all services."

Access tokens should be scoped to a specific resource server (audience claim). Using the same token for all services violates the principle of least privilege. Say "Each access token is scoped to a specific audience. If Service A needs to call Service B, it uses the token exchange flow to get a token scoped to Service B."

**Red Flag 7:** "Refresh tokens never expire."

Refresh tokens always expire. They have a TTL (typically 30 days) and can be revoked. If the user changes their password, all refresh tokens should be revoked. Say "Refresh tokens have a configurable TTL. I set it to 30 days with rotation for security."

**Red Flag 8:** "I will validate the token by calling the database on every request."

Calling the database on every request creates a bottleneck and adds latency. Token validation should be stateless and fast. Say "I validate JWTs locally using the cached JWKS public key. For opaque tokens, I use a Redis cache with a TTL to avoid hitting the database on every request."

**Red Flag 9:** "I will build my own OAuth2 server."

Building your own OAuth2 server is a significant undertaking with serious security implications. OAuth2 has many subtle security considerations (RFC 6819). Use a well-tested implementation unless you have a specific reason not to. Say "I would use Keycloak, Auth0, Okta, or AWS Cognito. These are battle-tested and handle the edge cases that are easy to get wrong in a custom implementation."

**Red Flag 10:** "The state parameter is optional."

The state parameter is required for CSRF protection. Without it, an attacker can inject their own authorization code into the client's redirect flow. Say "I always include a cryptographically random state parameter in the authorization request and validate it on the redirect callback."

**Red Flag 11:** "I will send the access token in the response body for GET requests."

Access tokens must never be sent in the URL or response body for GET requests. They are always sent in the Authorization header. Say "The client sends the access token in the Authorization header: Authorization: Bearer &lt;token>."

**Red Flag 12:** "OAuth2 solves all authorization problems."

OAuth2 is a framework for delegated authorization. It does not address fine-grained access control (who can access which resource within a service). For that, you need attribute-based access control (ABAC) or relationship-based access control (ReBAC) on top of OAuth2. Say "OAuth2 handles the delegation and token issuance layer. I use ABAC or ReBAC for fine-grained access control within the resource server."

**Red Flag 13:** "I will use the same client_id for all environments."

Client IDs and secrets must be unique per environment (development, staging, production). Using the same credentials across environments increases the blast radius of a credential leak. Say "Each environment has its own client registration with separate client IDs, secrets, and redirect URIs."

**Red Flag 14:** "I will validate the token at the API gateway and trust it in all downstream services."

This creates a single point of trust and violates the principle of defense in depth. Each service should validate the token independently. The API gateway can pre-validate, but downstream services must not trust the gateway blindly. Say "The API gateway validates the token and passes it downstream. Each service independently validates the token signature and checks the audience and scope."

**Red Flag 15:** "I do not need to worry about token revocation because the TTL is short."

Short TTLs reduce the window of vulnerability but do not eliminate the need for revocation. If a token is compromised, you need to revoke it immediately, not wait for it to expire. Say "I use a 15-minute TTL for access tokens, but I also maintain a revocation blocklist in Redis for immediate revocation when needed."

## 5 Drills

Practice these drills in mock interviews to internalize OAuth2 reasoning.

### Drill 1: The Grant Type Selector

Given a client type and use case, pick the correct OAuth2 grant type and justify your choice.

**Example:** A smart TV app needs to access a user's Netflix profile. The TV has a remote control (no keyboard) and no browser.

```
Question: What grant type do you use?
Answer:    Device Code grant (RFC 8628).

Flow:
1. TV requests a device code from the auth server.
2. Auth server returns: device_code, user_code, and
   verification_uri (e.g., netflix.com/activate).
3. TV displays the user_code and verification_uri.
4. User opens the verification URI on their phone
   or laptop, enters the user_code, and authenticates.
5. TV polls the token endpoint with the device_code.
6. When the user completes authorization, the TV
   receives the access token.

Why not other grant types:
- Authorization Code: TV has no browser for the redirect.
- Client Credentials: There is a user involved.
- Implicit: Deprecated and exposes token in the URL.
```

**Practice with:** A CLI tool, a server-side web app, a mobile banking app, an IoT sensor pushing data to a cloud API, a third-party integration with Slack.

### Drill 2: The Token Walkthrough

Given a scenario, walk through the entire token lifecycle from issuance to revocation. Include every step, every redirect, and every security check.

**Example:** A user logs into a third-party travel app using their Google account.

```
Step 1: Travel app redirects the user to Google's
authorization endpoint:
  GET /authorize?response_type=code
  &client_id=travel-app-123
  &redirect_uri=https://travelapp.com/callback
  &scope=openid+profile+email+calendar:read
  &state=random-csrf-token-abc123
  &code_challenge_method=S256
  &code_challenge=base64url(sha256(verifier))
  &nonce=random-nonce-xyz789

Step 2: Google authenticates the user and shows
consent screen: "Travel App wants to read your
Google Calendar."

Step 3: User consents. Google redirects back:
  HTTP 302 Location: https://travelapp.com/callback
  ?code=one-time-auth-code
  &state=random-csrf-token-abc123

Step 4: Travel app validates the state parameter
matches the one it sent.

Step 5: Travel app exchanges the code:
  POST /token
  grant_type=authorization_code
  code=one-time-auth-code
  redirect_uri=https://travelapp.com/callback
  client_id=travel-app-123
  client_secret=...
  code_verifier=original-verifier-value

Step 6: Google validates the code_verifier against
the code_challenge.

Step 7: Google returns:
  {
    "access_token": "jwt-access-token...",
    "token_type": "Bearer",
    "expires_in": 3600,
    "refresh_token": "refresh-token...",
    "id_token": "jwt-id-token..."
  }

Step 8: Travel app validates the id_token (checks
signature, nonce, issuer, audience).

Step 9: Travel app extracts user info from id_token:
  { sub: "google-123", name: "Jane Doe",
    email: "jane@gmail.com" }

Step 10: Travel app calls Google Calendar API:
  GET /calendar/v3/calendars/primary/events
  Authorization: Bearer jwt-access-token...

Step 11: After 55 minutes, the access token is
about to expire. Travel app refreshes:
  POST /token
  grant_type=refresh_token
  refresh_token=refresh-token...
  client_id=travel-app-123
  client_secret=...

Step 12: Google returns a new access token and
a new refresh token (rotation).

Step 13: User revokes the travel app's access
from their Google Account settings. Google
invalidates all outstanding tokens.
```

**Practice with:** A mobile app using PKCE, a server-to-server Client Credentials flow, a Device Code flow for a CLI tool, an OIDC login with RP-initiated logout.

### Drill 3: The Security Audit

Given a flawed OAuth2 implementation, identify the security vulnerabilities and propose fixes.

**Flawed implementation:**

```
1. Web app uses the Implicit flow (response_type=token).
2. No state parameter in the authorization request.
3. Redirect URI is validated only by domain prefix
   (https://*.myapp.com/*).
4. Access token TTL is 24 hours.
5. No refresh token rotation.
6. Token is sent in the URL query string for GET requests.
7. Client secret is hardcoded in the JavaScript bundle.
8. JWT is signed with "none" algorithm (no signature).
9. Scope is not validated on the resource server.
10. No PKCE for mobile apps.
```

**Fixes:**

```
Vulnerability 1: Implicit flow
→ Fix: Use Authorization Code with PKCE.

Vulnerability 2: No state parameter
→ Fix: Generate and validate a cryptographically
  random state parameter on every authorization request.

Vulnerability 3: Weak redirect URI validation
→ Fix: Exact match on the registered redirect URI.
  https://myapp.com/callback only, not a wildcard.

Vulnerability 4: 24-hour access token TTL
→ Fix: Reduce to 15 minutes. Use refresh tokens for
  longer sessions.

Vulnerability 5: No refresh token rotation
→ Fix: Invalidate the old refresh token on each
  refresh request. Issue a new refresh token.

Vulnerability 6: Token in URL query string
→ Fix: Send tokens in the Authorization header only.

Vulnerability 7: Client secret in JavaScript
→ Fix: SPA is a public client. Use PKCE instead of
  a client secret. If a secret is needed, proxy
  through a backend service.

Vulnerability 8: "none" algorithm
→ Fix: Reject tokens with alg=none. Require RS256
  or ES256. Validate the JWT library's algorithm
  whitelist.

Vulnerability 9: Missing scope validation
→ Fix: Resource server checks that the token's
  scope includes the required permission for
  each API endpoint.

Vulnerability 10: No PKCE
→ Fix: Add PKCE with S256 code challenge method
  for all public clients.
```

**Practice with:** Identify 5 more vulnerabilities not listed above. Consider: token binding, mTLS, CORS, referrer header leakage, token replay, clock skew handling, JWKS endpoint exposure, refresh token theft, authorization code replay, and logout semantics.

### Drill 4: The Token Exchange Design

Given a microservice architecture, design the token exchange and delegation mechanism.

**Architecture:**

```
[Client] --> [API Gateway] --> [Orders Service]
                                  |
                                  +--> [Payments Service]
                                  +--> [Inventory Service]
                                  +--> [Notifications Service]
```

**Requirements:**

```
- User authenticates with OIDC and receives an access
  token scoped to the API Gateway.
- Orders Service needs to call Payments Service,
  Inventory Service, and Notifications Service on
  behalf of the user.
- Each downstream service has its own API and scope.
- Tokens must be auditable and time-bound.
- The user should not need to grant permissions to
  each individual service.
```

**Design:**

```
Token exchange flow:
1. Client obtains an access token from the auth server
   with scope: "orders:write payments:read inventory:read
   notifications:send".

2. Client sends the token to the API Gateway.

3. API Gateway validates the token and routes to
   Orders Service.

4. Orders Service calls the auth server's token
   exchange endpoint (RFC 8693):
   POST /token
   grant_type=urn:ietf:params:oauth:grant-type:token-exchange
   subject_token=<original-token>
   subject_token_type=urn:ietf:params:oauth:token-type:access_token
   requested_token_type=urn:ietf:params:oauth:token-type:access_token
   resource=https://payments.example.com
   scope="payments:read"
   client_id=orders-service
   client_secret=...

5. Auth server validates the original token, checks
   that the original token includes the requested
   scope, and issues a new token scoped to Payments
   Service.

6. Orders Service calls Payments Service with the
   exchanged token.

7. Repeat steps 4-6 for Inventory and Notifications
   services, each with reduced scope.

Security considerations:
- Each exchanged token has a 5-minute TTL (shorter
  than the original token).
- The original token's scope is reduced for each
  downstream service (principle of least privilege).
- Exchanged tokens are not refreshable.
- All exchanges are logged with correlation IDs.
- The auth server rate-limits token exchanges per
  client (100 per minute).

Performance considerations:
- Cache exchanged tokens locally with a TTL of 4
  minutes (1 minute before expiry).
- Use a connection pool for the auth server calls.
- If the auth server is unavailable, fail open with
  the original token for 30 seconds (grace period).
```

**Practice with:** Design the token exchange for a chain of 5 services where Service A calls B, B calls C, C calls D, and D calls E. Identify the token lifetime implications and design a solution. Then design the same system with a service mesh (Istio, Linkerd) handling the token exchange transparently.

### Drill 5: The Migration Planner

Given a legacy system that uses session-based authentication, plan the migration to OAuth2.

**Legacy system:**

```
- Monolithic web app with session cookies
- Session stored in server-side memory (single node)
- User credentials sent to the backend on every login
- No token-based API access
- All services share the same session cookie
```

**Migration plan:**

```
Phase 1: Auth server setup (Week 1-2)
- Deploy Keycloak or Auth0 as the authorization server.
- Configure the auth server with the existing user
  database (LDAP, Active Directory, or custom DB).
- Define scopes for each API endpoint.
- Register the monolithic app as a confidential client.

Phase 2: Token-based login (Week 3-4)
- Add an OAuth2 login option alongside the existing
  session-based login.
- Users can choose to log in with OAuth2 or continue
  using the legacy session.
- The monolithic app acts as both the OAuth2 client
  and the resource server.
- Validate access tokens in the API layer.
- Measure: 5% of users migrated.

Phase 3: API tokenization (Week 5-6)
- Add token validation to API endpoints.
- Support both session cookies and Bearer tokens.
- API clients can use either authentication method.
- Deprecate session-based API access.
- Measure: 50% of users migrated.

Phase 4: Service decomposition (Week 7-8)
- Split the monolithic app into microservices.
- Each microservice validates tokens independently.
- Add refresh token support for long-lived sessions.
- Implement token exchange for service-to-service calls.
- Remove session-based authentication entirely.

Phase 5: Security hardening (Week 9-10)
- Enable PKCE for all public clients.
- Enable refresh token rotation.
- Add token revocation with Redis blocklist.
- Add token introspection for legacy services.
- Hardening: mTLS, token binding, CORS, redirect URI
  validation audit.

Rollback plan:
- Phase 2-3: Toggle the OAuth2 login flag off.
  Users fall back to session-based login.
- Phase 4: Keep the monolithic app running as a
  fallback. Route 10% of traffic to the new
  microservices and monitor.
- Phase 5: Token revocation is reversible by
  removing the blocklist entry.

Key metrics:
- Token validation latency: under 5ms p99
- Token refresh success rate: >99.9%
- Token revocation latency: under 5 seconds
- Auth server uptime: >99.99%
- Login success rate: >99.95%
- Token-related errors: under 0.1% of all API requests
```

**Practice with:** Plan the migration for a mobile app that currently sends username and password to the backend. Consider the challenges of updating existing mobile clients, handling offline users, and rolling back if the migration fails. Then design the migration for a third-party API platform where external developers have already integrated with your API using API keys.

## Key Takeaways

- Every system design interview that involves authentication or authorization will include OAuth2 questions. Prepare with the full Authorization Code flow, PKCE, and OIDC.
- The four-step framework (client type, grant type, token lifecycle, security validation) structures clean answers that cover what the interviewer needs to hear.
- Use specific phrases that signal depth: PKCE, code_challenge, code_verifier, state parameter, JWT bearer token, JWKS, token introspection, refresh token rotation, token binding, audience claim, scope validation.
- Avoid 15 common red flags that reveal shallow understanding: calling OAuth2 authentication, proposing the Implicit flow, storing secrets in mobile apps, putting tokens in URLs, claiming tokens are encrypted, ignoring the state parameter, building custom auth servers, using the same token for all services, and failing to validate redirect URIs precisely.
- Run 5 drills until they become automatic: grant type selector, token walkthrough, security audit, token exchange design, migration planner.
- Know at least three real-world OAuth2 implementations: Auth0 (SaaS), Keycloak (self-hosted), AWS Cognito (cloud-native) — and their relevant features.
- A strong hire quantifies everything — token lifetimes in minutes, revocation latencies in seconds, TTLs with concrete numbers, and scope names with specific permissions.
