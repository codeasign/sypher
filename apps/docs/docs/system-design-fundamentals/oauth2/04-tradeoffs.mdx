# OAuth2 -- Trade-offs and Failure Modes

## The Core Tension

Security versus complexity. OAuth2's flexibility is its greatest strength and its most dangerous weakness. The specification defines four grant types, multiple token formats, optional extensions (PKCE, token exchange, token revocation), and leaves critical security decisions to the implementer. Every knob you can turn is a decision that can compromise your system if you get it wrong.

The tension operates on four axes:

**Flexibility versus security**: OAuth2 supports browser apps, mobile apps, server-side apps, IoT devices, and machine-to-machine communication. Each use case needs a different grant type. Choosing the wrong one -- or using the right one without the required security extension -- opens a vulnerability.

**Developer experience versus strictness**: A simpler implementation (long-lived tokens, no PKCE, implicit flow) gets you to market faster. A strict implementation (short-lived tokens with refresh rotation, PKCE for every public client, audience validation, sender-constrained tokens) takes longer to build but survives a penetration test.

**Delegation granularity versus usability**: Fine-grained scopes (read:email, write:posts, delete:posts) give users precise control. But asking a user to approve 15 separate scopes in a consent screen causes fatigue -- they click "Accept All" without reading, defeating the purpose.

**Self-contained tokens versus introspected tokens**: JWT access tokens carry claims you can validate locally (fast, no network call per request). Opaque tokens require a call to the introspection endpoint (slow, adds latency). But a leaked JWT cannot be revoked until it expires. An opaque token can be revoked instantly.

The fundamental question you must answer before writing a single line of authorization code is: *what damage does a leaked access token cause, and how quickly can an attacker exploit it?* The answer determines where you land on every axis above.

## Decision Matrix

Every OAuth2 deployment forces a chain of interlocking decisions. The table below maps the most critical choices, their trade-offs, and the recommended default for modern systems.

### Grant Type Selection

| Grant Type | Best For | Security Posture | Key Risk | Recommendation |
|---|---|---|---|---|
| Authorization Code + PKCE | Browser SPA, mobile app, native app | High | CSRF on auth code (mitigated by state + PKCE) | Default for all public clients |
| Authorization Code (confidential) | Server-side web app with backend | High | Client secret leak | Default for confidential clients |
| Client Credentials | Machine-to-machine, cron jobs, services | Medium | No user context, shared secret | Use with mTLS or client assertion |
| Device Code | Smart TVs, game consoles, CLI tools | Medium | Phishing (user enters URL from device) | Require device verification |
| Implicit (deprecated) | Legacy SPAs | Low | Token in URL fragment, no refresh token | Do not use. Migrate to Auth Code + PKCE. |
| Resource Owner Password (deprecated) | Legacy first-party apps | Low | Credentials exposed to client | Do not use. Migrate to Auth Code + PKCE. |

The biggest migration in modern OAuth2 is the deprecation of the implicit flow. The OAuth Security BCP (RFC 9700) explicitly recommends that all clients use the authorization code grant with PKCE. Browser SPAs that redirect to the authorization endpoint, receive a code in the URL, and exchange it for tokens via the back channel -- this is the only pattern you should implement today.

### PKCE Decision

| Factor | No PKCE | PKCE with S256 |
|---|---|---|
| Auth code interception risk | Vulnerable | Mitigated |
| Extra round trip | None | One (token request carries code_verifier) |
| Implementation complexity | Zero | Minimal (generate code_verifier, hash, send) |
| Client type requirement | Confidential only | All clients |
| BCP 9700 compliance | Non-compliant | Compliant |

### Opaque vs JWT Access Tokens

| Property | Opaque Token | JWT Access Token |
|---|---|---|
| Self-contained claims | No | Yes |
| Validation method | Introspection endpoint (network call) | Local (crypto signature check) |
| Validation latency | 5-20ms per call | 0.01-0.1ms per call |
| Revocable | Yes (check introspect endpoint) | No (must wait for expiry) |
| Token size | ~20-40 bytes | ~500-2000 bytes |
| Information leakage | Minimal | Claims visible if decoded |
| Clock skew sensitivity | None | Requires accurate clocks (skew tolerance) |

### Token Expiry Strategy

| Strategy | Access Token TTL | Refresh Token TTL | Use Case | Risk |
|---|---|---|---|---|
| Ultra-short | 1-5 minutes | None | High-security financial ops | Frequent re-auth, poor UX |
| Standard | 15-30 minutes | 7-30 days | General purpose | Refresh token theft window |
| Long-lived | 1-24 hours | 90+ days | Internal services | Stolen token lives for hours |
| Refresh rotation | 15 minutes | Single-use, rotated | Modern best practice | Requires rotation logic |

Refresh token rotation -- issuing a new refresh token with each refresh and invalidating the old one -- is the strongest mitigation against refresh token theft. If an attacker steals a refresh token, the next legitimate refresh invalidates the attacker's copy, and vice versa. You detect the theft when one of the two parties presents a revoked token.

## Five Failure Modes

OAuth2 has been implemented at scale by thousands of companies. The failure modes below are drawn from real-world security advisories, CVEs, and penetration test reports. Learn them -- you will encounter every single one.

### 1. Client Secret Leak

The most common OAuth2 vulnerability. A client secret embedded in a mobile app binary, a JavaScript SPA, a public GitHub repo, or a container image layer is compromised the moment anyone inspects the artifact.

```
Attack Timeline:

T+0:  Developer commits client_secret to public GitHub repo
T+1h: Attacker's automated scanner finds the commit
T+2h: Attacker uses secret to obtain tokens via client credentials
T+2h: Attacker calls API as if they are the legitimate app
T+48h: Developer notices unusual API usage in billing report
T+72h: Secret rotated, attacker blocked

Total damage window: 70 hours of unauthorized API access

How it happens:
  - Hardcoded in mobile app: decompile APK, find string
  - Committed to GitHub: forgot .gitignore
  - Stored in container image: docker history reveals layer
  - Sent to client in JavaScript: visible in browser DevTools

Mitigation:
  - Public clients: do not use a client secret. Use PKCE.
  - Confidential clients: rotate secrets every 90 days.
  - Use client assertions (JWT signed with private key) instead of secrets.
  - Scan repos for secrets (git-secrets, truffleHog, GitHub secret scanning).
```

For mobile apps specifically, never store a client secret. The app binary is trivially decompilable. Use the authorization code flow with PKCE where the `code_verifier` is derived from a cryptographically random string stored in the device's secure enclave (Android Keystore, iOS Keychain). The client is authenticated by possession of the code verifier, not a shared secret.

### 2. Redirect URI Manipulation

The authorization server returns the authorization code to the `redirect_uri` registered by the client. If the server accepts a redirect URI that does not exactly match the registered URI, an attacker can intercept the authorization code.

```
Attack Scenario:

Legitimate client:
  Registered redirect URI: https://myapp.com/callback

Attacker tricks user:
  https://authserver.com/auth?
    client_id=myapp&
    redirect_uri=https://myapp.com.evil.com/callback&
    response_type=code

The problem:
  - User sees "myapp.com" in the URL
  - Auth server checks that redirect_uri *contains* myapp.com
  - Redirects code to the attacker's subdomain

Exact match would prevent:
  https://myapp.com.evil.com/callback  ✗ does not match
  https://myapp.com/callback           ✓ matches exactly

Common registration mistakes:
  - Wildcard in redirect URI: https://*.myapp.com/*
  - Using contains() instead of exact match
  - Accepting HTTP redirect URIs (no TLS -> code intercepted)
  - Open redirector on the client side
```

The fix is unambiguous: the authorization server must enforce an exact string match against the registered redirect URI. No suffix matching. No wildcards. No prefix checks. If you register `https://myapp.com/callback`, then only that exact URL is valid.

The OAuth2 spec allows multiple redirect URIs per client. Register exactly as many as you need -- no more. Each one must be a complete, TLS-protected URL.

### 3. CSRF on Authorization Code

Without the `state` parameter, an attacker can initiate an authorization request, intercept the redirect, and inject their authorization code into a victim's session.

```
CSRF Attack on Authorization Code

Attacker's browser:                      Victim's browser:
┌───────────────────┐                    ┌───────────────────┐
│ 1. Initiate auth  │                    │                   │
│    request to     │                    │                   │
│    Google         │                    │                   │
└────────┬──────────┘                    └────────┬──────────┘
         │                                        │
         ▼                                        │
Google sends auth code to                        │
attacker's redirect URI                          │
         │                                        │
         ▼                                        │
Attacker captures code: "abc123"                 │
         │                                        │
         │    Attacker sends link to victim:     │
         │    https://victimapp.com/callback     │
         │    ?code=abc123&state=xyz             │
         │                                        │
         ▼                                        ▼
         ├────────────────────────────────────────┤
         │                                        │
         ▼                                        │
    Victim's browser navigates to                 │
    /callback?code=abc123&state=xyz              │
         │                                        │
         ▼                                        │
    Victim's app exchanges code for token         │
         │                                        │
         ▼                                        │
    Attacker gains access to                      │
    victim's account                              │
         │                                        │
         ▼                                        │
    Result: Attacker's session linked             │
    to victim's account                           │

Without state parameter, the victim's app
cannot detect that this code was issued
for the attacker's session, not the victim's.
```

The `state` parameter is a cryptographically random value generated by the client before initiating the authorization request. The authorization server echoes it back unchanged in the redirect. The client verifies that the returned state matches the one it generated. This ensures the response corresponds to a request the client initiated.

Implementation requirements:

- State must be cryptographically random (at least 128 bits)
- State must be bound to the user's session (stored server-side or in an HttpOnly cookie)
- State must be checked before exchanging the code for tokens
- State must be single-use (consumed after one validation)

PKCE also mitigates this attack by binding the authorization code to the code verifier, but state is the primary CSRF defense and should always be present regardless of PKCE.

### 4. Refresh Token Theft

A stolen refresh token gives an attacker the ability to generate new access tokens indefinitely -- until the refresh token expires or is revoked. Unlike leaked access tokens (which live for minutes), leaked refresh tokens often live for days or months.

```
Refresh Token Theft

Attacker gains access to refresh token via:

  1. Compromised client database
  2. Man-in-the-middle on TLS termination
  3. Malware on end-user device
  4. Logging in client-side code (XSS)

Consequences:
  Token Type      |  Damage Window  |  Actions Possible
  ──────────      |  ─────────────  |  ─────────────────
  Access token    |  15 minutes     |  API calls until expiry
  Refresh token   |  30 days        |  Unlimited new tokens for 30 days

If both tokens are stolen:
  Attacker can refresh every 15 minutes
  No user interaction required
  Original user's session still works (no logout)
  Impossible to detect without rotation

Mitigation -- Refresh Token Rotation:

  Normal flow:
  Step 1: Client has (access_token_1, refresh_token_1)
  Step 2: access_token_1 expires
  Step 3: Client presents refresh_token_1
  Step 4: Auth server returns (access_token_2, refresh_token_2)
  Step 5: refresh_token_1 is invalidated

  Theft detection:
  Step 1: Attacker has (access_token_1, refresh_token_1)
  Step 2: Victim's legitimate app refreshes first
  Step 3: Auth server issues (access_token_2, refresh_token_2)
  Step 4: Auth server invalidates refresh_token_1
  Step 5: Attacker tries to use refresh_token_1
  Step 6: Auth server rejects -- token already used
  Step 7: Auth server alerts: possible token theft

  Or:
  Step 1: Attacker refreshes first
  Step 2: Auth server issues tokens to attacker
  Step 3: Victim app refreshes with stale refresh_token_1
  Step 4: Auth server detects reuse
  Step 5: Auth server revokes ALL tokens for this session
  Step 6: Both attacker and victim lose access
  Step 7: Victim re-authenticates, attacker blocked
```

Refresh token rotation is the single most impactful security improvement you can make after enabling PKCE. Implement it as standard practice for all refreshable token flows.

### 5. Scope Escalation

A client requests a scope it is not authorized for, and the authorization server grants it. This happens when the server does not validate that the client is actually permitted to request a given scope before issuing the token.

```
Scope Escalation Attack

Client registers with scopes: [read:profile]
Client requests token with scopes: [read:profile, admin, delete:users]

What SHOULD happen:
  Auth server checks client's allowed scope list
  Returns token with scopes: [read:profile] only
  Extra scopes are silently dropped

What sometimes happens:
  Auth server trusts the client's scope request
  Returns token with scopes: [read:profile, admin, delete:users]
  Client can now perform admin operations

Real-world example:
  Google's OAuth2 implementation had a scope escalation vulnerability
  in the Google+ API (2018). A client requesting basic profile scope
  could also receive access to Google+ social graph data that the
  user had not consented to.

Mitigation:
  Auth server:
  - Maintain a registry of allowed scopes per client_id
  - Strip scopes the client is not authorized for
  - Log and alert on scope escalation attempts
  - Never trust the client to self-validate scope requests

  Resource server:
  - Validate token scopes on every request
  - Do not trust claims that exceed known allowed scopes
  - Use scope as a minimum, not a maximum: deny if scope is insufficient
```

## Anti-Patterns

These patterns are common in production because they "work" in development. Testing with a single client, a single redirect URI, and a local authorization server hides every one of them.

### 1. Implicit Flow for SPAs

You build a single-page application in 2026. You use the implicit flow because the authorization code flow "needs a backend" and your SPA is fully client-side.

```
Why this is wrong:

Implicit flow returns the access token in the URL fragment:
  https://myapp.com/#access_token=abc123&token_type=Bearer&expires_in=3600

Problems:
  1. Token in URL fragment: browser history stores it, referrer header leaks it
  2. No refresh token: user re-authenticates every time token expires
  3. Cannot use PKCE: implicit flow has no code exchange step
  4. Token exposed to any JavaScript loaded on the page (XSS -> instant compromise)

What you should do:
  Use authorization code flow with PKCE.
  Code goes to your backend API (or to a BFF -- backend for frontend).
  Backend exchanges code for tokens, stores them in HttpOnly cookies.
  SPA never sees a token.

Modern SPA auth stack:
  [SPA] ──redirect──> [Auth Server]
  [Auth Server] ──code──> [SPA's redirect URI]
  [SPA] ──code──> [BFF (backend for frontend)]
  [BFF] ──code + verifier──> [Auth Server]
  [Auth Server] ──tokens──> [BFF] (sets HttpOnly cookie)
  [BFF] ──session cookie──> [SPA]
```

The implicit flow was officially deprecated in the OAuth 2.1 draft and is explicitly discouraged by the OAuth Security BCP. Any SPA you build today should use the authorization code flow with PKCE, with tokens stored server-side in a BFF.

### 2. No PKCE for Public Clients

You add the authorization code flow but skip PKCE because "the authorization server does not support it" or "the client is running on a secure internal network."

```
Without PKCE, the authorization code is a single-use password.
Whoever presents it to the token endpoint gets the tokens.

Attack scenario:
  1. Attacker intercepts authorization code (mobile app inter-process
     communication, browser extension, compromised router)
  2. Attacker presents code to token endpoint
  3. Attacker receives access token and refresh token
  4. Legitimate user presents same code -- rejected (already used)
  5. User sees error. Attacker has access. User thinks it was a glitch.

With PKCE:
  Step 1: Client generates code_verifier (random string)
  Step 2: Client computes code_challenge = SHA256(code_verifier)
  Step 3: Client sends code_challenge with auth request
  Step 4: Auth server stores code_challenge with the issued code
  Step 5: Client presents code + code_verifier to token endpoint
  Step 6: Auth server computes SHA256(code_verifier) and compares
          to stored code_challenge
  Step 7: Only the client that generated the verifier can exchange the code
  Step 8: Intercepted code is useless without the matching verifier

The math:
  code_verifier = crypto.randomBytes(32).toString('base64url')
  code_challenge = crypto.createHash('sha256')
    .update(code_verifier).digest('base64url')

  Attacker intercepts: code + code_challenge
  but code_challenge = SHA256(code_verifier), which is
  computationally infeasible to reverse.

  Attacker needs: code_verifier (which never left the client)
  Without it, the code is worthless.
```

PKCE support is near-universal in modern authorization servers (Auth0, Okta, Keycloak, AWS Cognito, Azure AD, Google, GitHub). There is no legitimate reason to skip it for any public client.

### 3. Storing Client Secret in Mobile Apps

You embed a client secret in your iOS or Android app to authenticate the client. You believe the app binary is "obfuscated enough."

```
Why every mobile app secret is compromised:

Android:
  - APK is a ZIP file. Unzip it.
  - Run `strings` on the DEX file.
  - Decompile with jadx or apktool.
  - Search for client_secret, CLIENT_SECRET, clientSecret.

iOS:
  - IPA is a ZIP file. Unzip it.
  - Run `strings` on the Mach-O binary.
  - Use class-dump or Hopper to extract strings.
  - Search the Info.plist and embedded configuration files.

Worse: the app communicates with your API. An attacker run their own
copy of your app with a proxy (mitmproxy, Burp Suite) and sees every
API call, including the token exchange. The secret is visible in
plaintext in the HTTP request body.

Real example:
  In 2019, a popular VPN app stored their OAuth2 client secret
  in a configuration file within the APK. Researchers decompiled
  the APK, extracted the secret, and used it to make API calls
  on behalf of the VPN provider's infrastructure. The secret
  had not been rotated in 3 years.

The rule: public clients cannot hold secrets. A public client is
any client that runs on a device the user controls. Mobile apps,
desktop apps, browser SPAs -- all public clients. Use PKCE for
these clients. No shared secret required.
```

The distinction between confidential and public clients is the single most misunderstood concept in OAuth2 deployments. If the client binary is distributed to users, it is a public client. If the client runs on a server you control with no user access, it is a confidential client. Treat every client as public until proven otherwise.

### 4. Unlimited Scope Requests

You request every available scope in every authorization request because "future features might need it" and "it avoids another consent screen later."

```
The problem with overbroad scopes:

Scope requested on day 1: openid profile email read:posts write:posts
                          read:friends delete:posts admin

User approves. User has no idea what "admin" means.
Six months later, a bug in your app allows an attacker
to call the admin API. The token already has the admin scope.
No additional consent needed. Full access granted.

Least privilege principle for OAuth2 scopes:

  When registering your client:
  - Request only the scopes your current feature needs
  - Add scopes incrementally as new features ship
  - Remove scopes when features are deprecated

  When requesting a token:
  - Request only the scopes needed for this specific operation
  - A background sync job does not need write scopes
  - A read-only dashboard does not need delete scopes

  Example progression:
  MVP (v1):     [read:profile]
  v2:           [read:profile, write:posts]
  v3:           [read:profile, write:posts, read:analytics]
  v4:           [read:profile, write:posts, read:analytics, admin]
                                  ^
                                  |--- each scope added deliberately
                                       with a documented reason

  Anti-pattern:
  v1:           [read:profile, write:posts, read:analytics, admin,
                 delete:posts, manage:users, read:billing]
                                  ^
                                  |--- requested "just in case"
                                       never reviewed
                                       admin scope never used in code
                                       but present in every token
```

Periodically audit your registered scopes. Remove any scope that has no corresponding code path. Each unused scope is an attack surface expansion that provides zero business value.

### 5. No Token Validation on the Resource Server

Your resource server receives a JWT access token. It decodes the Base64 payload, reads the `sub` claim, and grants access. It never verifies the signature.

```
What Decode-Only Validation Looks Like:

  function getUserFromToken(token) {
    const payload = token.split('.')[1];   // take the middle part
    const decoded = atob(payload);          // Base64 decode
    const claims = JSON.parse(decoded);
    return { userId: claims.sub };
  }
  // No signature verification. No expiry check. No audience check.
  // Any JWT from any issuer is accepted.

What An Attacker Does:

  Step 1: Create a JWT with any claims:
    header:  {"alg": "HS256"}
    payload: {"sub": "admin-user-id", "exp": 9999999999, "scope": "admin"}
    signature: [anything]

  Step 2: Send it to the resource server:
    Authorization: Bearer <forged-jwt>

  Step 3: Resource server decodes payload, trusts claims.
  Step 4: Attacker has admin access.

  The "alg: none" attack:
    header:  {"alg": "none", "typ": "JWT"}
    payload: {"sub": "admin", "scope": "admin"}
    signature: ""

  If the library parses "none" as valid, no signature is checked.
  Many JWT libraries had this vulnerability (CVE-2015-9235, CVE-2016-5431).

Proper Token Validation Checklist:

  [ ] Verify JWT signature using the authorization server's public key
  [ ] Check the "alg" header is expected (reject "none")
  [ ] Validate "iss" (issuer) matches your authorization server
  [ ] Validate "aud" (audience) matches your resource server identifier
  [ ] Check "exp" (expiry) is in the future
  [ ] Check "nbf" (not before) is in the past (if present)
  [ ] Validate "scope" contains the required permissions
  [ ] Reject tokens with unexpected claims

  Example validation in Python:

  ```python
  import jwt
  from jwt import PyJWKClient

  def validate_token(token):
      jwks_client = PyJWKClient("https://auth.example.com/.well-known/jwks.json")
      signing_key = jwks_client.get_signing_key_from_jwt(token)
      try:
          claims = jwt.decode(
              token,
              signing_key.key,
              algorithms=["RS256"],
              audience="https://api.example.com",
              issuer="https://auth.example.com",
              options={
                  "require": ["exp", "iss", "aud"],
                  "verify_exp": True
              }
          )
          return claims
      except jwt.PyJWTError as e:
          raise PermissionError(f"Token validation failed: {e}")
  ```

  The same applies to opaque tokens: always call the introspection
  endpoint. Never trust a token because "it looks valid."
```

| Anti-Pattern | Severity | Fix | Effort |
|---|---|---|---|
| Implicit flow for SPAs | Critical | Migrate to Auth Code + PKCE with BFF | Medium |
| No PKCE for public clients | Critical | Add PKCE S256 to all auth flows | Low |
| Client secret in mobile apps | Critical | Remove secret, use PKCE | Low |

## When NOT to Use OAuth2

OAuth2 is a delegation protocol. It solves one specific problem: a client application acting on behalf of a resource owner (user) to access a protected resource. Not every authorization scenario needs OAuth2. In these cases, a simpler mechanism is better.

### First-Party-Only Applications

Your application is a monolith or tightly coupled set of services, all owned by the same organization. Users authenticate to your system directly. No third-party app needs access to user data.

In this scenario, OAuth2 adds complexity with no benefit:

```python
# What you do NOT need:
# 1. Register a client_id for your own frontend
# 2. Configure redirect URIs
# 3. Implement authorization code flow
# 4. Issue access tokens that your own backend validates
# 5. Manage refresh token rotation

# What you should do instead:
# Use session-based authentication with HttpOnly cookies.
# Runtime: server-side session, CSRF token, SameSite cookie.

from flask import session, request, abort

@app.route('/api/profile')
def get_profile():
    if 'user_id' not in session:
        abort(401)
    user = db.get_user(session['user_id'])
    return {"name": user.name, "email": user.email}

# Simpler. Fewer moving parts. No token lifecycle to manage.
# No scope negotiation. No client registration.
```

OAuth2 makes sense for first-party apps only when:

- You have multiple independent frontends (web, mobile, desktop) sharing the same backend
- You expect to open the API to third parties in the future
- Your architecture requires separate auth service (auth-as-a-service)
- You need standardized token formats across service boundaries

If none of these apply, use session-based auth. You save months of development time and eliminate an entire class of vulnerabilities.

### Internal Microservices with mTLS

Your internal services communicate over a service mesh (Istio, Linkerd, Consul Connect) with mutual TLS. Each service has a valid certificate. The service mesh enforces authentication at the transport layer.

```
Internal mTLS Architecture:

  [Service A] ──mTLS──> [Service Mesh Proxy]
                              │
                              │  Validates:
                              │  - Service A's certificate
                              │  - SPIFFE ID in SAN
                              │  - Not expired
                              │
                              ▼
                      [Service Mesh Proxy] ──mTLS──> [Service B]

  No OAuth2 needed because:
  - Certificate is the credential
  - mTLS provides mutual authentication
  - Service mesh validates on every request
  - SPIFFE IDs are the identity claim
  - Transport-layer security avoids token logic

What you would add with OAuth2:
  - An authorization server (another service to operate)
  - Token issuance endpoints
  - Token validation in every service
  - Token refresh logic
  - Clock synchronization for JWT validation
  - Additional latency for token introspection

  The result: more complexity, more latency, no additional security
  beyond what mTLS already provides.
```

Add OAuth2 to internal microservices only when you need fine-grained delegation: Service A needs to act on behalf of a specific user, not just itself. For service-to-service auth without user context, mTLS with SPIFFE is the simpler, more secure default.

### Other Cases Where OAuth2 Is Overkill

**Single-user APIs**: Your personal automation script calls a single API. A static API key in an environment variable is appropriate. OAuth2's multi-step token acquisition is unnecessary overhead.

**Read-only public data**: Your application fetches weather data from a public API. No authentication needed. Adding OAuth2 would require every visitor to log in, creating a barrier with zero security benefit.

**Admin-only dashboards**: An internal admin panel used by 5 team members. Use a VPN + basic auth + IP allowlist. OAuth2 with its redirect flows, consent screens, and token management adds complexity without meaningful security improvement for this threat model.

## The Cost of Complexity

Every OAuth2 feature you add has a measurable operational cost:

**Authorization server overhead**: Running your own authorization server (Keycloak, Ory Hydra, custom) requires infrastructure, database, TLS certificate management, secret rotation, and monitoring. A managed service (Auth0, Okta, Cognito) reduces this but costs $1,000 to $10,000+ per month at scale.

**Token validation latency**: Every protected API call either makes an introspection request (10-40ms) or validates a JWT signature (0.1-1ms). At 10,000 requests per second, introspection adds 100-400 seconds of cumulative latency per second -- requiring additional API server capacity.

**Token storage**: Access tokens and refresh tokens need secure storage on the client side. On mobile, this means Keychain/Keystore integration with biometric protection. On the web, this means HttpOnly cookies with Secure and SameSite attributes. Each platform has different APIs and edge cases.

**Scope administration**: Each new API endpoint needs a scope decision. Each scope needs documentation, UI text for consent screens, database migration for user consent records, and periodic audit. At 100+ scopes, this is a full-time administrative role.

**Rotation and revocation**: Refresh token rotation requires tracking token lineage, detecting theft, and notifying users. Token revocation requires maintaining an allowlist or checking a revocation endpoint. Both add database writes to the token path and complicate the data model.

## Bridge to Real World

The trade-offs you make in OAuth2 propagate through your entire system architecture. Choosing opaque tokens over JWT tokens affects every API gateway, every microservice, and every client SDK you build. Choosing PKCE over no PKCE determines whether a leaked authorization code is a security incident or a non-event. Choosing refresh token rotation determines whether a leaked refresh token is a catastrophic breach or a detected-and-blocked anomaly.

In production systems, the most successful OAuth2 deployments follow a consistent pattern:

- Authorization code flow with PKCE (S256) for all user-facing clients
- Client credentials grant with JWT client assertions (private_key_jwt) for machine-to-machine
- Short-lived access tokens (15 minutes) with refresh token rotation
- Opaque tokens for high-security operations (with introspection caching)
- JWT tokens for high-throughput operations (with short TTL and signature validation)
- Exact-match redirect URIs, no wildcards, no suffixes
- State parameter on every authorization request
- Scope registry with least-privilege enforcement on the authorization server
- mTLS for internal service-to-service communication (no OAuth2 delegation needed)
- Managed authorization server (Auth0, Okta, Cognito) rather than self-hosted

These decisions are not made once. They are revisited during every architecture review, every penetration test, and every security incident. The trade-offs shift as your system grows, your threat model evolves, and the OAuth2 specification (now OAuth 2.1) stabilizes around a smaller set of recommended patterns.
