A major e-commerce platform runs a routine database migration during a quiet Sunday morning. The team deploys the new code directly to all production servers at once. Seven seconds after the deploy completes, every checkout attempt returns a 500 error. Payment processing halts. The entire site is down for 47 minutes while engineers scramble to roll back. Revenue loss exceeds 2 million dollars. The root cause? A single subtle incompatibility between the new code and the legacy session format that only manifests when real traffic hits.

This story repeats across the industry with alarming frequency. A social media network pushes a bad configuration change to all servers simultaneously, knocking the entire feed service offline for 90 minutes. A streaming platform deploys a new recommendation algorithm to every node at once and watches user engagement metrics plummet before anyone notices. A financial services firm replaces all instances of their transaction processing service in one shot, triggering a race condition that double-processes thousands of trades. Each of these incidents shares the same fundamental mistake: deploying the change to every user at the same time, with no ability to limit blast radius, observe impact, or roll back incrementally.

The problem is not bad code. The problem is how the code arrives in production. Software changes carry risk: new bugs, performance regressions, configuration mismatches, unexpected interactions with live data. Deploying all at once converts that risk into guaranteed downtime. The deployment process itself becomes the most dangerous operation your team performs. Teams that treat deployment as a fire-and-forget step inevitably learn this lesson the hard way, often during their highest-traffic periods when the cost of failure is greatest.

A deployment strategy is the set of decisions and mechanisms that control how, when, and to whom new software reaches production. Choosing the right strategy means trading off speed, safety, cost, and complexity. The wrong choice means accepting downtime, degraded user experience, or prolonged rollback times as inevitable costs of shipping software.

Think of deployment strategies like traffic management for a multi-lane highway. A **recreate deployment** closes all lanes, repaves the entire road, then reopens traffic — simple but completely blocks travel. A **rolling deployment** closes one lane at a time, repaves it, reopens it, then moves to the next lane — traffic keeps flowing but the process takes longer. A **blue-green deployment** builds a completely new parallel highway, diverts all traffic to the new road, then demolishes the old one — zero downtime but double the infrastructure cost. A **canary deployment** opens a single new lane alongside existing traffic, lets a few cars try it, and only widens the new road if no crashes happen — the safest approach but requires sophisticated traffic routing. Each strategy uses the same basic idea — change how you introduce new road surface — but makes fundamentally different tradeoffs in speed, cost, risk, and operational complexity.

## Core Vocabulary

**Deployment Strategy** — The overall plan that governs how a new version of software is introduced into production. It defines the order, rate, and mechanism for replacing old instances with new ones. The strategy determines downtime, rollback speed, and blast radius during failures.

**Rolling Deployment** — A strategy that replaces instances one at a time or in small batches. Old instances are gradually taken out of service, updated, and brought back online while remaining instances continue serving traffic. This minimizes capacity impact but requires careful health checking at each step.

**Blue-Green Deployment** — A strategy that maintains two identical production environments (blue and green). At any time, only one environment serves live traffic. The new version is deployed to the idle environment, fully tested, then traffic is switched instantly. Rollback means switching traffic back to the original environment.

**Canary Deployment** — A strategy that routes a small percentage of traffic to the new version while the majority continues using the old version. If the canary shows no errors or degradation, traffic is gradually increased to 100 percent. This limits blast radius to a small user subset.

**Feature Flag / Feature Toggle** — A configuration mechanism that wraps new functionality behind a conditional check evaluated at runtime. Code is deployed with features disabled, then enabled progressively without a new deployment. This decouples deployment from release, allowing fine-grained control over who sees what.

**A/B Testing Deployment** — A strategy that routes different user segments to different versions of the application based on user attributes, geography, or random assignment. Unlike a canary which aims for full rollout, A/B deployments often run multiple versions concurrently for extended periods to measure behavioral differences.

**Shadow Deployment** — A strategy that deploys the new version alongside the old version, duplicating production traffic to the new version without serving user responses from it. The shadow version processes real requests but its output is discarded. This validates behavior under real traffic without any user-facing risk.

**Rollback** — The process of reverting to a previous known-good version of the software. Rollback speed is a critical metric for any deployment strategy. Some strategies (blue-green) offer instant rollback via traffic switch; others (rolling) require reversing the update process instance by instance.

**Blast Radius** — The set of users, services, or data that are affected when a deployment goes wrong. Smaller blast radius is the primary goal of progressive deployment strategies. A canary deployment limits blast radius to the canary group; a recreate deployment has a blast radius equal to the entire user base.

**Health Check** — A automated probe that determines whether a deployed instance is functioning correctly. Health checks can test basic process liveness, application-level readiness, or deep semantic correctness. Deployment strategies rely on health checks to decide whether to proceed with the next batch or abort the rollout.

**Deployment Pipeline** — The automated sequence of stages that code passes through from commit to production deployment. A mature pipeline includes build, test, artifact creation, environment promotion, deployment, and verification stages. The pipeline enforces quality gates and can automatically pause or roll back on failure.

Most production systems use a combination of strategies: blue-green for major version upgrades, canary for feature releases, rolling for patch updates, and feature flags for day-to-day feature toggling.

## How It Actually Works — Step by Step

### Step 1: Artifact Creation and Registry

Every deployment begins with a deployable artifact. The build process compiles source code, bundles dependencies, and produces a versioned artifact — a Docker image, a JAR file, an AMI, or a compiled binary. This artifact is pushed to a registry with a unique immutable tag (usually a Git commit hash or semantic version). The artifact must never be modified after creation; any change requires a new build with a new tag. Immutability guarantees that what you test in staging is exactly what runs in production.

### Step 2: Environment Selection and Target Group

The deployment orchestrator identifies the target environment and the set of instances to update. In a Kubernetes cluster, this means selecting the deployment or statefulset and its pod template. In a VM-based infrastructure, it means identifying the auto-scaling group or instance group. The orchestrator records the current state — how many instances are running, their current version, and their health status — before making any changes.

### Step 3: Pre-Deployment Health Verification

Before any instance is modified, the orchestrator runs a pre-deployment health check against the current running instances. This confirms the system is in a known-good state. If existing instances show elevated error rates, degraded performance, or partial failures, the deployment is aborted immediately. Deploying into a degraded system compounds problems and makes root cause analysis nearly impossible.

### Step 4: Instance Drain and Traffic Shift

For the first batch of instances (or the idle environment in blue-green), the orchestrator signals the load balancer to stop sending new traffic to those instances. This process is called draining. For a rolling deployment, the load balancer waits for in-flight requests to complete (up to a configurable timeout) before considering the instance drained. For blue-green, the entire active environment is drained in one operation by switching the load balancer target group.

### Step 5: Update and Replacement

The orchestrator replaces the old version with the new version on the drained instance. In containerized environments, this means pulling the new image and restarting the pod. In VM environments, it means provisioning a new instance from the updated AMI or running a configuration management tool. The new instance starts with the new code, connects to shared infrastructure (databases, caches, queues), and registers itself with the service discovery system.

### Step 6: Post-Deployment Health Check

The orchestrator runs a health check against the newly started instance before returning traffic to it. This check goes beyond a simple TCP port test. A proper health check verifies that the application responds correctly to a test request, can connect to its dependencies (database, cache, upstream services), and reports itself as ready. The health check must succeed consistently for a stabilization period — typically 30 seconds to 5 minutes — before the instance is trusted.

### Step 7: Traffic Restoration and Verification

Once the new instance passes health checks, the load balancer restores traffic to it. The orchestrator monitors key metrics for the new instance: error rate, latency percentiles, CPU usage, memory consumption, and business-specific metrics like conversion rate or request throughput. If any metric exceeds defined thresholds, the orchestrator can automatically abort the deployment and trigger a rollback.

### Step 8: Batch Progression (Rolling and Canary)

For rolling and canary deployments, the orchestrator repeats steps 4 through 7 for each subsequent batch. The batch size and progression speed are configurable. A conservative rolling deployment might update 10 percent of instances at a time with a 5-minute stabilization window between batches. An aggressive canary might start at 1 percent, ramp to 5 percent after 2 minutes, then jump to 100 percent if no issues are detected. The progression logic can be automated based on metrics or require manual approval at each gate.

### Step 9: Blue-Green Traffic Switch

In a blue-green deployment, the traffic switch is the pivotal moment. The new version is fully deployed and tested in the idle environment before any user traffic reaches it. When the orchestrator is satisfied with the new environment's health, it updates the load balancer or DNS record to point all traffic to the new environment. This switch is atomic from the load balancer's perspective — all traffic moves at once. The old environment remains running, ready to receive traffic immediately if a rollback is needed.

### Step 10: Canary Gradual Ramp

In a canary deployment, the orchestrator monitors the canary group's performance against the baseline. If error rates, latency, or business metrics remain within acceptable thresholds, traffic to the new version is incrementally increased. A typical canary ramp might progress through 1 percent, 5 percent, 10 percent, 25 percent, 50 percent, and 100 percent. At each step, the system compares metrics between the canary and baseline populations. Statistical significance matters — a 1 percent canary on a 1,000 request-per-second service might need several minutes to collect enough data to detect a 0.1 percent error rate increase.

### Step 11: Feature Flag Gradual Activation

When feature flags are used, the deployment itself is simple — the new code is deployed to all instances with the feature disabled. The release happens separately through the feature flag system. The orchestrator enables the flag for internal users first, then for a small percentage of external users, then gradually expands. Each expansion step is a configuration change, not a deployment. If problems emerge, the flag is toggled off instantly, reverting behavior without any code rollback. This decoupling of deployment from release is the key advantage of feature flags.

### Step 12: Rollback and Abort

Every deployment strategy must define its rollback procedure before deployment begins. For blue-green, rollback means switching the load balancer back to the old environment — a single operation taking seconds. For rolling, rollback means reversing the update process, updating each instance back to the previous version, which can take as long as the original deployment. For canary, rollback means setting the canary traffic to zero percent, which is instant. For feature flags, rollback means toggling the flag off, which is also instant. The rollback procedure must be tested regularly, just like the deployment itself.

## Variants and Flavors

### Variant 1: The Ramped Rolling Deployment

A ramped rolling deployment updates instances in controlled batches with a progressive increase in batch size. The first batch is the smallest — typically 1 to 2 instances — to validate the deployment works. Subsequent batches grow larger as confidence increases. This variant combines the simplicity of rolling updates with the safety of progressive exposure.

Ramped rolling is best suited for stateless services where instances are interchangeable and there is no session affinity requirement. The primary risk is that a problem affecting only a small percentage of traffic (such as a data corruption bug that requires specific user state) might not be detected in the early small batches.

### Variant 2: Blue-Green with Traffic Mirroring

Traffic mirroring extends blue-green deployment by sending a copy of live traffic to the standby environment while the active environment continues serving. The standby environment processes the mirrored traffic but discards the responses. This validates the new version under real production traffic patterns without any user-facing impact. If the standby environment shows elevated errors or latency, the deployment is aborted before the traffic switch.

### Variant 3: Canary with Automatic Rollback

An automated canary deployment integrates directly with the observability and monitoring stack. The deployment pipeline defines a set of metric thresholds: error rate must not exceed 0.1 percent above baseline, p99 latency must not exceed 200ms, and CPU usage must not exceed 80 percent. The canary runs for a minimum observation period before each ramp step. If any threshold is breached, the pipeline automatically rolls back the canary to zero percent and sends an alert. This variant requires no human intervention, enabling safe deployment during off-hours.

### Variant 4: A/B Testing Deployment

A/B testing deployment differs from canary deployment in a critical way: the goal is not to safely roll out a new version but to measure the behavioral difference between two versions. In an A/B deployment, both versions run concurrently for an extended period (hours to weeks), and traffic is split based on user attributes rather than a simple percentage. The split is typically deterministic: users in group A always see version A, users in group B always see version B. This enables clean metric comparison between the two groups.

A/B deployment requires a robust experimentation platform. The platform must:
- Assign users to treatment groups consistently (same user always sees the same version)
- Track user-level metrics (conversion rate, time-on-page, revenue) across sessions
- Perform statistical analysis to determine whether observed differences are significant
- Handle interaction effects between multiple concurrent A/B tests

The infrastructure for A/B deployment is significantly more complex than canary because it requires user-level tracking and statistical analysis, not just aggregate metric comparison. Many organizations use a dedicated A/B testing platform (such as Google Optimize, Optimizely, or a custom solution) that operates independently of the deployment pipeline.

### Variant 5: Recreate Deployment

The recreate deployment is the simplest strategy. The orchestrator terminates all running instances of the old version, waits for them to fully shut down, and then starts new instances with the new version. This strategy is the fastest to execute (no batching, no traffic routing, no health check orchestration) but causes complete downtime during the transition.

Recreate is appropriate for:
- Development and staging environments where downtime is acceptable
- Batch processing systems that run on a schedule (the new version is deployed between batches)
- Systems with no active user traffic (internal tools, scheduled jobs)
- Database migrations that require exclusive access (the application is taken offline, the migration runs, and the new version starts)

The recreate strategy is also the simplest to roll back: stop all instances, restart with the old artifact. The rollback time is identical to the deployment time.

### Variant 6: Immutable Infrastructure Deployment

Immutable infrastructure deployment takes the recreate concept further. Instead of updating existing instances, entirely new instances are provisioned from a fresh machine image or container image. The old instances are terminated only after the new instances have passed health checks. This is the deployment strategy used by auto-scaling groups with instance refresh, and it is the recommended approach for cloud-native applications.

The key advantage of immutable infrastructure is that it eliminates configuration drift. Every instance starts from a known-good base image with the application code baked in. There is no opportunity for manual configuration changes, package updates, or file modifications to create differences between instances. If an instance must be updated, it is replaced with a fresh instance, not modified.

Immutable infrastructure deployment is supported natively by AWS Auto Scaling Groups (instance refresh), Google Cloud Managed Instance Groups (rolling update with replace), and Kubernetes (DaemonSet rolling update). The strategy is most effective when combined with infrastructure-as-code (Terraform, CloudFormation, Pulumi) to manage the instance template or AMI.

### Variant 7: Traffic Shadowing for Risk-Free Validation

Traffic shadowing, also called dark launch or mirroring, is the most conservative deployment variant. The new version is deployed alongside the old version, and a copy of every production request is sent to both versions. The old version's response is returned to the user. The new version's response is recorded but discarded. This gives the team confidence that the new version handles real traffic correctly without any user-facing risk.

Shadowing is most valuable for:
- High-risk changes that could cause data corruption or financial loss
- Services with complex business logic that is difficult to test with synthetic traffic
- Large-scale infrastructure changes where the failure mode is not well understood
- Changes to core infrastructure that affect all other services (such as the service mesh or API gateway)

The cost of shadowing is the highest of any strategy: the new version must be fully provisioned to handle the entire production traffic load, and the infrastructure must support traffic duplication. In a service mesh, traffic shadowing can be configured at the sidecar proxy level, but in traditional infrastructure, it requires a custom load balancer or proxy configuration.

### Choosing Between Variants

The choice between deployment variants depends on the specific risk profile of the change being deployed:

| Risk Level | Change Type | Recommended Variant |
|------------|-------------|-------------------|
| Low | Bug fix, patch update | Ramped rolling |
| Medium | New feature, stateless | Canary with automatic rollback |
| Medium | New feature, stateful | Blue-green with traffic mirroring |
| High | Database migration | Recreate with maintenance window |
| High | Core infrastructure change | Traffic shadowing |
| Very high | Payment/security changes | Blue-green + canary + shadow combined |
| Experimental | UI/UX changes | A/B testing |

### The Interaction Between Deployment Strategies and CI/CD

A deployment strategy does not exist in isolation. It is the final stage of a continuous integration and continuous delivery pipeline. The pipeline stages that precede the deployment determine how safe the deployment can be:

**Stage 1 — Commit and Build:** Every commit triggers an automated build. The build produces an immutable artifact with a unique tag. The artifact is pushed to a registry. If the build fails, the deployment never starts. This is the first safety gate.

**Stage 2 — Unit and Integration Tests:** Automated tests run against the build artifact. Tests must pass before the artifact is considered deployable. The test suite should include: unit tests for individual functions, integration tests for service interactions, contract tests for API compatibility, and security scans for known vulnerabilities.

**Stage 3 — Staging Deployment:** The artifact is deployed to a staging environment that mirrors production configuration. The staging deployment uses the same strategy as production but with relaxed parameters (faster ramp, fewer observation windows). The staging environment runs smoke tests and performance tests to verify the artifact works at scale.

**Stage 4 — Production Canary:** The artifact is deployed to a production canary group. The canary group is a subset of the production fleet, typically 1-5 percent of instances. The canary runs for a minimum observation period during which metrics are compared to the baseline.

**Stage 5 — Production Rollout:** The artifact is deployed to the full production fleet using the chosen strategy (rolling, blue-green, or expanded canary). The rollout is automated but can be paused for manual approval at predefined gates.

**Stage 6 — Post-Deployment Verification:** After the rollout completes, the pipeline runs post-deployment verification. This includes: monitoring metrics for a stabilization period, running synthetic transactions, and checking for any alert activations.

Each stage in this pipeline reduces the risk of the final deployment. A mature pipeline catches 90-95 percent of deployment problems before they reach the production rollout stage. The deployment strategy is the last line of defense, not the only one.

### Real-World Deployment Strategy Anti-Patterns

Beyond the technical anti-patterns discussed in the tradeoffs section, there are organizational anti-patterns that undermine deployment safety:

**Anti-Pattern: Deployment as a Separate Team Responsibility.** When a separate DevOps or infrastructure team owns all deployments, the development team loses the feedback loop that teaches them to deploy safely. Developers who never deploy do not understand the impact of their code changes on the deployment process. They make changes that are difficult to deploy safely — such as non-backward-compatible schema changes or tightly coupled service dependencies.

**Anti-Pattern: Manual Deployment Approval from a Manager.** When a manager must approve every deployment, the approval becomes a rubber stamp. Managers do not have the context to evaluate whether a deployment is safe. The approval step adds delay without adding safety. Automated metric-based gating provides more meaningful safety checks than any manager approval.

**Anti-Pattern: Deploying on a Schedule Rather Than on Demand.** Deploying on a fixed schedule (every Tuesday at 10 AM) encourages code to accumulate between deployments. The larger the change set, the higher the deployment risk. Deploying on demand — as soon as a change is ready — keeps change sets small and deployment risk low. This is the principle behind continuous deployment: deploy every passing commit, not every Tuesday.

**Anti-Pattern: Treating Rollback as a Rare Event.** When rollback is treated as a rare failure case, the rollback mechanism is not maintained or tested. It becomes a snowflake procedure that requires manual steps and tribal knowledge. Rollback should be treated as a normal, routine operation. Test it weekly. Automate it fully. Make it as boring as a deployment.

## Mental Model Check

**Question 1:** Your team deploys a new version of a payment processing service. The deployment strategy uses rolling updates with a batch size of 50 percent. After the first batch deploys, the error rate jumps from 0.1 percent to 12 percent. What happens next?

The deployment should pause or abort. The health check on the first batch has failed. In a properly configured rolling deployment, the orchestrator detects the elevated error rate, stops the deployment, and the remaining 50 percent of instances continue serving the old version. The blast radius is limited to the first batch. The team investigates the root cause while the majority of traffic continues to flow through the old version.

**Question 2:** How does blue-green deployment achieve instant rollback, and what is the hidden cost?

Blue-green achieves instant rollback by keeping the old environment fully running with the old version. Rolling back requires only a load balancer target group update or DNS change to redirect traffic back to the old environment. The hidden cost is double infrastructure: both environments must be fully provisioned and sized to handle full production traffic. For a service running on 100 instances, blue-green requires 200 instances, doubling your compute cost.

**Question 3:** When should you choose a canary deployment over a rolling deployment?

Choose canary over rolling when the cost of failure is high and the blast radius must be minimal. Canary is better for user-facing services where a bad deployment could affect revenue or user trust. Rolling is better for internal services, batch processing jobs, or infrastructure components where the deployment is fast and the impact of failure is contained. Canary requires more sophisticated traffic routing and monitoring infrastructure than rolling.

**Question 4:** How do feature flags change the relationship between deployment and release?

Feature flags decouple deployment from release. Deployment becomes a low-risk technical operation: new code is shipped to servers but remains inactive. Release becomes a business decision: a configuration change enables the feature for specific users, groups, or percentages. This means you can deploy multiple times per day but release features on a schedule that matches business needs. Rollbacks become instant configuration toggles instead of time-consuming code reversions.

**Question 5:** What is the minimum set of metrics you need to monitor during a deployment?

The minimum set includes: error rate (HTTP 5xx and application errors), request latency (p50, p95, p99), request throughput (requests per second), CPU utilization, memory utilization, and at least one business-specific metric (such as conversion rate, signup completion rate, or checkout success rate). Without business metrics, you might detect that the application is running fine technically but still delivering a worse user experience that causes users to abandon their session.

### Deployment Strategy Decision Tree

Use this decision tree to select the right deployment strategy for your specific scenario:

```
Start: What is the service's downtime tolerance?
├── Zero downtime required
│   ├── Is instant rollback required?
│   │   ├── Yes → Do you have budget for 2x infrastructure?
│   │   │   ├── Yes → Blue-Green
│   │   │   └── No → Feature Flags (deploy with flags off, rollback by toggling off)
│   │   └── No → Canary (small blast radius, fast rollback by setting weight to 0%)
│   └── No → Is the service stateless?
│       ├── Yes → Ramped Rolling (fast, simple, low cost)
│       └── No → Rolling with connection drain (stateful service needs drain timeout)
│
├── Brief downtime acceptable (under 60 seconds)
│   ├── Single server or small fleet? → Recreate (fastest, simplest)
│   └── Multiple servers? → Rolling (no capacity overhead)
│
└── Longer downtime acceptable
    └── Recreate (lowest cost, highest risk)
```

This decision tree simplifies the tradeoff analysis but covers the most common scenarios. The key decision points are: downtime tolerance, rollback speed requirement, infrastructure budget, and state management requirements.

### Summary: The Five Laws of Deployment Safety

1. **Deploy to a subset first.** Never deploy to all instances, regions, or users at once. The subset can be a single instance (rolling), a single environment (blue-green), or a percentage of users (canary). The size of the subset determines the blast radius.

2. **Verify before promoting.** Every deployment step must be verified before the next step proceeds. Verification includes health checks, metric comparison, and synthetic transaction testing. The verification must be automated and have clear pass/fail criteria.

3. **Rollback must be faster than deploy.** If the rollback takes longer than the deployment, the deployment is not safe. The rollback procedure must be tested before every deployment, not just documented in a wiki.

4. **Separate deployment from release.** Use feature flags to decouple when code is deployed from when it is activated. This allows you to deploy code continuously and release features when they are ready. The deployment becomes a low-risk technical operation; the release becomes a business decision.

5. **Monitor business metrics, not just technical metrics.** A deployment that passes all technical health checks can still degrade the user experience. Monitor business metrics (conversion rate, signup rate, revenue per request) during and after the deployment. If the business metrics deviate, roll back.

### Deployment Strategy Anti-Patterns in Practice

**Antipattern: The Big Bang Deploy.** The team combines 2 weeks of changes into a single Friday afternoon deployment. The deployment includes 47 commits, 12 changed files, 3 database schema changes, and a configuration change. When something goes wrong, it is impossible to determine which of the 47 commits caused the problem. The rollback requires reverting all 47 commits, including the ones that were working correctly.

Prevention: Deploy every passing commit. If a deployment fails, only the single commit that caused the failure needs to be rolled back. Continuous deployment is the antidote to the Big Bang Deploy.

**Antipattern: The Configuration Leak.** The team uses environment variables for configuration. The deployment pipeline sets different values for the staging environment versus the production environment. A new team member accidentally transposes the staging and production configuration files. The staging deployment works fine (it uses production configuration), but the production deployment fails because it tries to use staging database credentials.

Prevention: Use a single configuration source of truth that is environment-aware. Infrastructure-as-code tools (Terraform, CloudFormation) enforce environment separation at the configuration level. Configuration values should be stored in a parameter store or secrets manager, not in environment variable files that can be accidentally swapped.

**Antipattern: The Zero-Configuration Pipeline.** The team sets up a deployment pipeline with default settings: maxSurge=25%, maxUnavailable=25%, readiness check at port 8080 with a 5-second interval. They never adjust these defaults for their specific service. The default readiness check interval is too fast for their slow-starting service, causing false health check failures. Every deployment requires manual override of the health check.

Prevention: Configure the deployment pipeline for your specific service. Test the deployment configuration in staging before running it in production. Adjust the health check interval, stabilization period, and batch size based on your service's startup time, connection pooling behavior, and request duration characteristics.

### Pre-Deployment Safety Checklist

Before any production deployment, run this safety checklist:

1. Has the artifact been built from a known-good commit? Verify the commit hash matches the expected version.
2. Has the artifact passed all automated tests? Unit tests, integration tests, security scans.
3. Has the deployment been tested in a staging environment that mirrors production?
4. Is the rollback procedure documented and tested? The rollback must be tested within the last 7 days.
5. Is the on-call engineer available and aware of the deployment? They must be reachable within 5 minutes.
6. Are the monitoring dashboards configured to show deployment-specific metrics? Canary vs. baseline comparison must be visible.
7. Are the automated rollback thresholds configured? Error rate delta, latency delta, business metric delta.
8. Has the database schema been verified to be backward-compatible with the old application version?
9. Is the deployment scheduled during a low-traffic period? Avoid peak hours and major events.
10. Has the deployment been approved by the appropriate stakeholders? For critical services, this means at least 2 engineers.

A deployment that passes all 10 checks has a 95+ percent chance of completing without incident. A deployment that skips any check should be treated as high-risk.

### Beyond Basic Deployment Strategies: Emerging Patterns

**Progressive Delivery:** Deployment strategies are evolving toward progressive delivery, a superset of deployment that combines canary deployments, feature flags, and A/B testing into a unified framework. In progressive delivery, you control not just when a new version reaches which users, but exactly which capabilities are exposed to which users at which intensity. This gives product teams fine-grained control over the user experience while giving operations teams fine-grained control over deployment safety.

**Edge-Native Deployments:** As more computation moves to the edge (CDNs, edge computing, IoT devices), deployment strategies must adapt to environments with limited bandwidth, intermittent connectivity, and hardware diversity. The strategy shifts from orchestrator-driven (a central system decides when to deploy) to agent-driven (each edge device decides when to pull and apply the new version). Health checking is decentralized: each edge node independently monitors its own health and rolls back if the new version causes problems.

**AI-Assisted Deployments:** Machine learning models are being applied to deployment strategy selection and deployment monitoring. A model can predict the optimal deployment strategy for a given service based on its historical deployment data, traffic patterns, and dependencies. Another model can detect deployment anomalies earlier than fixed thresholds by learning the normal range of metrics for each service and flagging deviations that are unusual for that specific service, even if they are within the fixed threshold range.
