## Tier 1: Foundation

### Challenge 1: Strategy Selection for a Video Streaming Service

A video streaming service uses a microservice architecture with 30 services. The most critical service is the video transcoding pipeline, which converts uploaded videos into multiple resolutions. Transcoding is a CPU-intensive batch job that runs on dedicated worker instances. The service is stateful — each worker instance processes a batch of videos and maintains temporary files locally. Deployments happen once per week.

**Question:** Which deployment strategy would you recommend for the transcoding service? Justify your choice by comparing at least three strategies.

**Model solution:** The transcoding service is a batch processing system with local state. The key constraint is that in-flight transcoding jobs cannot be interrupted without losing work. A rolling deployment is the best choice for this scenario.

A recreate deployment would terminate all current transcoding jobs, losing work. A blue-green deployment would require double the transcoding capacity, which is expensive for a CPU-intensive workload. A canary deployment would be complex to implement because it requires traffic routing for batch jobs rather than HTTP requests.

A rolling deployment allows the system to let current batches complete before updating each worker. The deployment process is: drain the worker instance (do not assign new jobs, let current jobs finish), update the worker, health-check the worker (verify it can accept new transcoding jobs), restore the worker to the available pool. The maxUnavailable should be set to 1 to minimize lost capacity. The drain timeout should be set to the maximum transcoding job duration (e.g., 30 minutes) to prevent jobs from being forcibly terminated.

**Key insight:** Batch processing services need different deployment semantics than real-time services. The drain mechanism must wait for current jobs to complete, not just for in-flight HTTP requests. The deployment strategy is driven by the job duration, not by the request latency.

### Challenge 2: Rollback Strategy Audit

A team deploys a new version of their user authentication service every two weeks using a rolling deployment. Their rollback procedure is documented as: "Reverse the deployment by re-running the pipeline with the previous version tag." The rollback has never been tested.

The authentication service handles login, registration, and session management. It communicates with a user database, a session store (Redis), and a third-party identity provider.

**Question:** Identify at least three things that could go wrong during an untested rollback. Propose a concrete rollback testing procedure.

**Model solution:**

Three potential rollback failures:
1. The database schema has changed since the previous version was deployed. Rolling back the application code would make it incompatible with the current database schema. For example, the new version added a column to the users table, and the old version crashes when it encounters this column.
2. The artifact for the previous version has been garbage-collected from the registry. The artifact registry's retention policy may have removed older versions, so the rollback pipeline cannot pull the artifact to deploy.
3. The Redis session store contains sessions created by the new version in a format the old version cannot parse. When the rollback completes, all users with active sessions are logged out and must re-authenticate.

**Rollback testing procedure:**
1. Before every deployment, run a rollback test in the staging environment. Deploy the current version, then the new version, then execute the rollback procedure. Verify the system returns to the exact state it was in before the new deployment.
2. Verify the previous artifact exists in the registry with its immutable tag. Set the artifact retention policy to keep at least 5 previous versions regardless of age.
3. Verify that the database schema is backward-compatible. Run the rollback against a copy of the production database schema to confirm the old code works with it.
4. Test session compatibility. Verify that sessions created by the new version can be gracefully handled (e.g., invalidated with a clear error message, not a crash) by the old version.
5. Document the rollback duration. If the rollback takes longer than 30 minutes, it is not fast enough to be useful during an incident. Investigate ways to speed it up.

### Challenge 3: Feature Flag Deployment

A team is introducing feature flags to their deployment process for the first time. Their current process is: build the artifact, deploy to staging, run integration tests, deploy to production using a rolling update. They want to add a feature flag for a new checkout flow.

**Question:** Design the feature flag system architecture and deployment process. Include how the flag is evaluated, how it is configured, and how the team knows whether the flag is working correctly.

**Model solution:**

**Flag evaluation:** The feature flag is evaluated at the application level. In the checkout flow code, a configuration check determines whether the user sees the new flow or the old flow. The flag evaluation uses deterministic hashing based on the user ID to ensure consistent experience within a session. The flag configuration is loaded at application startup and refreshed every 60 seconds from a configuration service.

**Flag configuration:** The flag is configured through a separate configuration service (such as LaunchDarkly, Split, or a simple in-house service backed by etcd). The configuration defines targeting rules: internal users always see the new flow, beta users see it 50 percent of the time, and all other users see it 1 percent of the time. Each targeting rule has a unique key so the system can evaluate the flag for each user consistently.

**Deployment process with feature flags:**
1. The new checkout flow code is deployed with the flag set to off for all users. The deployment uses the existing rolling update process.
2. After deployment, the team enables the flag for internal users only. They verify the new flow works correctly with real production data and infrastructure.
3. If internal testing passes, the flag is enabled for 1 percent of external users. The team monitors error rate, checkout completion rate, and revenue per user for 24 hours.
4. If metrics are stable, the flag percentage is increased to 25 percent, then 50 percent, then 100 percent, with at least 4 hours between each increase.
5. After the flag reaches 100 percent and has been stable for 48 hours, the old code path is removed from the application, and the flag configuration is cleaned up.

**Verification:** The team monitors the difference in checkout completion rate between users in the treatment group (new flow) and the control group (old flow). A statistically significant decrease in the treatment group triggers an automatic flag rollback. The monitoring dashboard shows both the flag percentage and the metric comparison in real time.

**Key insight:** Feature flags require a cultural shift, not just a technical one. The team must commit to removing the old code path after the flag has been fully rolled out. Otherwise, the codebase accumulates dead code paths behind permanent flags, increasing maintenance burden and testing complexity.

### Challenge 4: Multi-Service Deployment Coordination

A team manages three interdependent services that are deployed together as a release train: Service A (API gateway), Service B (order processing), and Service C (inventory management). Each service runs on 20 instances. The services communicate via synchronous HTTP calls. A deployment to any service can affect the others. The team currently deploys each service independently using rolling updates, but this has caused issues when a deployment to Service A (which calls B and C) happens before or after the deployment of B or C.

**Question:** Design a coordinated deployment strategy for all three services. The strategy must ensure that at no point during the deployment is a service calling a version of another service that it is incompatible with. Include the deployment order, coordination mechanism, and rollback plan.

**Rubric:**

- **Exemplary (5/5):** Proposes a feature-flag-based strategy where backward-compatible API changes are used. Services are deployed in dependency order (leaf-first: C, then B, then A). Each service's deployment includes a stabilization period where the new version runs alongside the old version. API contracts are versioned and both old and new endpoints are served during the transition period. Rollback is orchestrated in reverse order (A first, then B, then C) with the same stabilization periods.
- **Strong (4/5):** Proposes a deployment order with clear reasoning. Identifies the compatibility issue but does not fully address the rollback order.
- **Adequate (3/5):** Proposes a deployment order but does not consider the transition period when old and new versions of different services are running simultaneously.
- **Weak (1-2/5):** Treats the services as independent deployments without considering cross-service compatibility.

**Model answer:** The key insight is that synchronous HTTP calls between services mean you cannot deploy a new version of a calling service before the called service is ready, and you cannot deploy a new version of a called service that breaks the contract expected by the calling service.

The deployment order is: C (leaf service, no downstream dependencies), then B, then A. Each service deploys using a canary strategy with backward-compatible API contracts. The new version of each service must continue to support the old API contract for a transition period. This means: new API endpoints are additive, request parameters are backward-compatible, and response fields are additive (clients ignore unknown fields).

The coordination mechanism is a deployment orchestration pipeline that manages all three services as a single release. The pipeline is:
1. Deploy C v2 with support for both v1 and v2 API contracts. Canary at 10 percent for 5 minutes. If healthy, promote to 100 percent.
2. Wait 5 minutes. Verify C v2 is stable under full traffic.
3. Deploy B v2 with support for both v1 and v2 API contracts. Canary at 10 percent. If healthy, promote to 100 percent.
4. Wait 5 minutes. Verify B v2 is stable.
5. Deploy A v2. Canary at 10 percent. If healthy, promote to 100 percent.
6. After all three services are running v2 and stable for 30 minutes, the old API contracts can be deprecated.

Rollback is the reverse order: roll back A to v1, then B to v1, then C to v1. Each rollback follows the same canary process. The old API contracts must be kept alive until the rollback completes.

### Challenge 5: Database Migration During a Rolling Deployment

A service uses PostgreSQL with a table that stores user preferences. The team needs to add a new preference field to the table. The table has 10 million rows. The service runs on 50 instances behind a load balancer. The deployment uses a rolling update with a batch size of 10 percent.

**Question:** Design the database migration strategy that works with the rolling deployment. The migration must be zero-downtime. Consider the phases of the rolling deployment and how the database state interacts with instances running different versions.

**Rubric:**

- **Exemplary (5/5):** Uses the expand-migrate-contract pattern with three separate phases. Each phase is a separate deployment. Phase 1 adds the column as nullable, Phase 2 backfills data while the application runs normally, Phase 3 makes the column non-nullable and deploys the application code that depends on it. Includes a rollback strategy for each phase.
- **Strong (4/5):** Uses the expand-migrate-contract pattern but does not separate the phases into independent deployments. May propose running the migration as a single step during the application deployment.
- **Adequate (3/5):** Proposes adding the column in one step. May cause downtime if the migration takes too long or if old-instance queries fail.
- **Weak (1-2/5):** Proposes modifying the schema at the same time as deploying the new version without considering backward compatibility.

**Model answer:** The database migration uses the expand-migrate-contract pattern split into three independent deployments:

Phase 1 (Expand): Deploy a database migration that adds the column as nullable. This is purely a database change — no application code is deployed. The migration runs `ALTER TABLE user_preferences ADD COLUMN new_feature_enabled BOOLEAN DEFAULT NULL`. This operation takes under 1 second on a table with 10 million rows (metadata-only change in PostgreSQL with no default value). The existing application code ignores the column because it does not reference it.

Phase 2 (Backfill): Deploy a batch backfill job that populates the new column for existing rows. The job runs in batches of 10,000 rows with 1-second pauses to avoid replication lag. For a 10 million row table, the backfill takes approximately 20 minutes. The job runs in the background and does not affect the application. If the job fails, it can be restarted from the last processed row. The application continues running normally during the backfill.

Phase 3 (Contract): Deploy the new application code that depends on the column. The application code reads and writes the new column. This deployment uses the same rolling update strategy. The database migration in this phase adds a NOT NULL constraint and default value: `ALTER TABLE user_preferences ALTER COLUMN new_feature_enabled SET NOT NULL; ALTER TABLE user_preferences ALTER COLUMN new_feature_enabled SET DEFAULT false;`. This migration runs after the backfill is complete and all instances are running the new code. A NOT NULL constraint on a fully backfilled table is a metadata-only operation.

Rollback for Phase 1: Run `ALTER TABLE user_preferences DROP COLUMN new_feature_enabled`. This is fast because the column is nullable and has no dependent objects. Rollback for Phase 2: No action needed; the old application code runs without the column. Rollback for Phase 3: Deploy the old application code that does not reference the column. The column remains in the database but is unused until the rollback is resolved. After the rollback stabilizes, run Phase 3's rollback database migration to drop the column.

### Challenge 6: Blue-Green with Connection Draining

A blue-green deployment of a WebSocket-based real-time analytics service causes a brief connection drop for all active users during the traffic switch. The service maintains WebSocket connections for live data streaming. Users lose 2-3 seconds of data during the switch.

**Question:** Redesign the deployment to eliminate the data loss. What changes to the deployment strategy, the application code, or the infrastructure are required?

**Rubric:**

- **Exemplary (5/5):** Proposes session migration where active WebSocket connections are seamlessly transferred from the old environment to the new environment. Uses a gateway or proxy layer that manages connection migration. The new environment pre-warms connections to backend services before the migration. The old environment is kept running until all connections have naturally terminated or been migrated.
- **Strong (4/5):** Proposes keeping the old environment running until all active connections terminate naturally. New connections go to the new environment. This eliminates data loss for new connections but existing connections eventually terminate and reconnect.
- **Adequate (3/5):** Proposes a longer drain timeout (60 seconds) to allow in-flight data to be sent before the connection is cut. This reduces data loss but does not eliminate it.
- **Weak (1-2/5):** Does not address the WebSocket connection issue or proposes a solution that introduces significant complexity without solving the problem.

**Model answer:** The solution is a session migration approach using a WebSocket gateway. The gateway is a proxy layer that sits between clients and the application servers. Each WebSocket connection is routed through the gateway, which maintains a mapping of connection IDs to backend server instances.

When the blue-green traffic switch begins:
1. The new environment (green) starts and pre-warms connections to shared infrastructure (database, message queue, cache).
2. The gateway receives a migration signal for all connections currently on the blue environment.
3. For each connection, the gateway establishes a new WebSocket connection to the green environment, transparently to the client. The gateway sends a handoff message to the green server containing the connection's state (subscription topics, last message timestamp).
4. Once the green server acknowledges the handoff, the gateway switches the client-facing connection from the blue server to the green server. This is atomic at the gateway level — the client sees no interruption because it remains connected to the same gateway endpoint.
5. The blue server is notified that the connection has been migrated. It stops sending data and cleans up the connection state.
6. After all connections have been migrated (or the migration timeout of 5 minutes is reached), the remaining blue connections are gracefully terminated.

This approach requires:
- A WebSocket gateway that supports connection migration (most commercial API gateways do not support this natively; you may need a custom gateway using Envoy or HAProxy with a Lua filter).
- Application servers that support session handoff: they must be able to serialize their per-connection state and send it to a new server.
- The migration must be idempotent: if a migration handoff fails, the gateway should retry with a different green server instance.

The cost of this approach is significant implementation complexity. A simpler alternative is to use the gradual session-shifting approach: new connections go to the green environment, and the blue environment is kept running until all existing connections have naturally terminated. This eliminates data loss for all connections eventually but extends the deployment window to the maximum session duration (which could be hours for real-time analytics services).

### Challenge 7: Global Edge Deployment with Traffic Profile Diversity

A global CDN service runs on 500 edge locations across 100 countries. Each location has different hardware configurations, traffic profiles, and third-party integrations. The team deploys software updates twice per month. A recent deployment caused 502 errors in locations serving high volumes of API traffic, but the canary location (which primarily serves static content) showed no problems.

**Question:** Design a canary deployment strategy that accounts for traffic profile diversity across edge locations. The strategy must be able to catch problems that only manifest under specific traffic conditions.

**Model answer:** The fundamental problem is that a single canary location cannot represent all traffic profiles. The solution is a multi-profile canary strategy:

**Profile categorization:** Each edge location is categorized by its primary traffic profile:
- Static profile: predominantly serves cached static assets (images, CSS, JavaScript). 200 locations.
- API profile: predominantly serves dynamic API requests. 150 locations.
- Streaming profile: predominantly serves video/audio streaming. 100 locations.
- Mixed profile: balanced mix of all traffic types. 50 locations.

**Multi-profile canary:** The deployment starts with four simultaneous canary deployments, one per profile category. Each canary deploys to a single location representative of its category:
- Static canary: a location serving 95 percent static content.
- API canary: a location serving 80 percent API traffic.
- Streaming canary: a location serving 70 percent streaming traffic.
- Mixed canary: a location serving balanced traffic across all types.

Each canary observes for 30 minutes with profile-specific metric thresholds:
- Static canary: cache hit ratio, origin fetch latency, bytes served.
- API canary: error rate, p50/p99 latency, request throughput.
- Streaming canary: startup time, rebuffering ratio, bitrate stability.
- Mixed canary: all of the above plus CPU utilization and memory pressure.

**Progression logic:**
1. If all four profile canaries pass, the deployment proceeds to 10 locations per profile category (40 total) for 1 hour.
2. If the 40-location rollout passes, proceed to 50 percent of locations (250) for 2 hours.
3. If the 50 percent rollout passes, proceed to 100 percent of locations.
4. At any step, if any profile category shows degraded metrics, the deployment is paused only for that profile category. Other profile categories continue their rollout. This prevents a problem that only affects API locations from blocking the static content locations.

**Rollback at the edge:** Each edge location independently monitors health metrics after receiving the new version. If error rate exceeds 1 percent above baseline or p99 latency exceeds 500ms, the location automatically reverts to the previous version and alerts the operations team. This distributed rollback prevents a bad deployment from propagating through all locations.

**Key insight:** The standard single-canary approach is insufficient for heterogeneous environments. You must test each traffic profile independently because problems are often profile-specific. The cost is increased deployment time and monitoring complexity, but the benefit is catching problems that a uniform canary would miss.

### Challenge 8: Intent-Based Deployment System Design

As a platform engineer, you are tasked with designing an intent-based deployment system for a platform serving 200 microservices across 10 teams. Each team has different deployment requirements, risk tolerances, and operational maturity levels.

The system must accept an intent statement from a developer ("I want to deploy service X to production with maximum safety and zero downtime") and automatically select the strategy, execute it, monitor the results, and report the outcome.

**Question:** Design the intent model and the decision engine that maps intents to deployment strategies. The system must handle:
- Services with different criticality levels (Tier 1: payment processing, Tier 2: user-facing features, Tier 3: internal APIs)
- Services with different state requirements (stateless APIs, stateful databases, batch jobs)
- Services with different infrastructure (Kubernetes, VMs, serverless)
- The system must learn from past deployment outcomes to improve strategy selection over time

**Model answer:**

**Intent model:** The developer specifies three dimensions in the intent:
1. Safety level: "maximum" (no user impact acceptable), "standard" (brief degradation acceptable for up to 1 percent of users), "minimum" (brief downtime acceptable)
2. Speed requirement: "as-fast-as-possible," "standard," "no-rush"
3. Rollback requirement: "instant" (under 10 seconds), "fast" (under 5 minutes), "standard" (under 30 minutes)

The system also reads the service's metadata from a service registry: criticality tier, state type, infrastructure type, typical traffic volume, and historical deployment performance.

**Decision engine:** The engine maps the intent to a deployment strategy using a decision tree:

```
Intent: maximum safety, no-rush speed, instant rollback
Service: Tier 1 (payment), stateful, Kubernetes
Decision: Blue-green deployment with canary validation

Intent: standard safety, as-fast-as-possible, fast rollback
Service: Tier 2 (user-facing), stateless, Kubernetes
Decision: Ramped rolling deployment with feature flags

Intent: minimum safety, as-fast-as-possible, standard rollback
Service: Tier 3 (internal API), stateless, VMs
Decision: Rolling deployment with 50% batch size
```

**Execution and monitoring:** The engine selects the strategy, generates a deployment plan (including specific percentages, observation windows, and thresholds), and executes the plan through the deployment pipeline. During execution, the engine monitors metrics and compares them to historical baselines. If the current deployment's metrics deviate from the historical pattern beyond a threshold, the engine adjusts the strategy.

For example, if a canary deployment of Service X shows a 0.5 percent error rate increase on the first step (compared to a historical baseline of 0.1 percent), the engine extends the observation window from 5 minutes to 15 minutes to collect more data. If the deviation persists, the engine rolls back and recommends a blue-green deployment for this service's next attempt.

**Learning system:** The engine records every deployment's outcome (success, failure, duration, metrics deviation) in a deployment history database. After each deployment, the engine updates its decision model: if a particular strategy consistently performs poorly for a service category, the engine deprioritizes that strategy for similar services. Over time, the engine learns that stateful services on VMs perform best with blue-green deployments, while stateless services on Kubernetes perform best with ramped rolling deployments.

The learning system requires a minimum of 30 deployments per service to build a reliable model. For new services with no deployment history, the engine defaults to the most conservative strategy (blue-green with extended observation windows) and relaxes the strategy as the history accumulates.

**Key insight:** An intent-based system is only as good as its observability. Without accurate, real-time metrics and complete deployment history, the engine cannot learn or adapt. The hardest part of building this system is not the decision engine — it is the metrics infrastructure that feeds it.

### Supplementary Challenge: Evaluating Deployment Strategy Tradeoffs

**Scenario:** A streaming video platform deploys a new video transcoding pipeline. The pipeline runs on a Kubernetes cluster with 200 GPU-enabled worker nodes. Each transcoding job takes 15-30 minutes. The pipeline is stateful — each worker maintains a local cache of video segments and has a 45-minute warmup period before reaching peak throughput.

**Question:** Which deployment strategy would you recommend, and what specific modifications would you make to account for the long warmup period and stateful nature of the workers?

**Discussion points:**

The long warmup period (45 minutes) eliminates most standard deployment strategies. A rolling deployment with a 10-minute stabilization window would never complete because each newly started worker would take 45 minutes to reach full throughput. The deployment would bottleneck on the warmup time.

A blue-green deployment would require 400 GPU instances (200 active + 200 standby), which is prohibitively expensive for GPU hardware. However, if the hardware is already provisioned for peak capacity, the standby environment might already exist.

The recommended approach is a **dual-speed rolling deployment**:

1. **Pre-warming phase:** Start the new version worker instances in the background while the old version continues serving. The new workers warm up their caches over 45 minutes without receiving traffic. This phase runs in parallel with normal operations.

2. **Gradual traffic migration:** Once the new workers are warmed up, traffic is gradually migrated from old to new workers. The migration happens at a rate of 10 percent of all jobs per batch, with 5-minute observation windows between batches. This is a canary approach applied to the migration, not the deployment.

3. **Completion phase:** Once all traffic is on the new workers, the old workers are kept running for 60 minutes to handle any long-running jobs that started before the migration. After the drain period, old workers are terminated.

This dual-speed approach adds 45 minutes to the deployment (the pre-warming phase) but avoids the cost of blue-green's 2x infrastructure requirement. The total deployment time is approximately 60 minutes (45 minutes pre-warm + 10 minutes migration + 5 minutes observation).

### Supplementary Challenge: Compliance-Driven Deployment Strategy

**Scenario:** A healthcare application must comply with HIPAA regulations. Any deployment that changes how patient data is processed requires a documented audit trail, a 72-hour pre-deployment notice period, and the ability to prove that no data was exposed during the deployment.

**Question:** Design a deployment strategy that satisfies these compliance requirements while still enabling frequent deployments.

**Discussion points:**

The compliance requirements fundamentally change the deployment strategy. The technical strategy (canary, blue-green, rolling) is secondary to the compliance workflow:

1. **Pre-deployment compliance gate:** Before any production deployment, the deployment pipeline generates a compliance report. The report includes: the artifact hash and build logs, the list of changed files since the last deployment, a diff of any database schema changes, a security scan result (SAST/DAST), and a data flow diagram showing how patient data is affected by the changes.

2. **Compliance review period:** The compliance report is submitted to the compliance team. The team has 72 hours to review and approve or reject the deployment. The deployment pipeline is locked during this period. Automated approvals are not allowed for HIPAA changes.

3. **Deployment with audit trail:** The deployment uses blue-green to minimize the risk of data exposure. The traffic switch is recorded in the deployment audit log: timestamp, operator identity, artifact version, and the health check results before and after the switch. The audit log is written to an immutable audit store (write-once, read-many storage) that cannot be modified after the fact.

4. **Post-deployment compliance verification:** After the deployment, the system runs a data integrity check: verify that patient data that existed before the deployment still exists after the deployment, verify that no unauthorized access occurred during the deployment by analyzing access logs, and generate a post-deployment compliance report.

5. **Rollback compliance:** If the deployment is rolled back, the rollback generates its own compliance report. The rollback is treated as a separate deployment event for compliance purposes, with its own audit trail and data integrity verification.

The compliance-driven approach adds significant overhead (72-hour pre-deployment notice, compliance report generation, audit log management). To enable frequent deployments, the compliance review is automated for changes that do not affect patient data processing. Changes that touch the data processing pipeline always require manual compliance review. The system automatically classifies the change type based on the files changed and the database schema diff, and routes the deployment to the appropriate compliance workflow.

### Challenge 9: Self-Healing Deployment System

**Scenario:** A large e-commerce platform deploys 200 microservices across 5 regions. Each service deploys independently using its own strategy. The platform averages 15 deployments per hour. The operations team is overwhelmed by the number of deployments and wants a self-healing deployment system that can automatically detect and correct deployment problems without human intervention.

**Question:** Design a self-healing deployment system that can detect deployment problems, diagnose the root cause, and take corrective action — all without human intervention. The system must handle the following scenarios:
- A canary deployment shows elevated error rates for the new version
- A rolling deployment stalls because a batch of instances fails health checks
- A blue-green deployment's traffic switch causes a brief latency spike
- A feature flag deployment causes a business metric (conversion rate) to drop
- A deployment causes a dependency service to become overloaded

**Model answer:**

**Architecture overview:** The self-healing system consists of four components: a deployment health monitor, a root cause analyzer, a corrective action selector, and a deployment state manager. These components work together in a closed-loop control system.

**Component 1: Deployment Health Monitor**

The health monitor continuously observes deployment metrics and compares them to expected baselines. It uses a combination of fixed thresholds and statistical anomaly detection:

```python
class DeploymentHealthMonitor:
    def __init__(self, baseline_db):
        self.baseline_db = baseline_db
        self.anomaly_detector = StatisticalAnomalyDetector()

    def check_deployment_health(self, deployment_id, metrics_stream):
        # Get the pre-deployment baseline for this service
        baseline = self.baseline_db.get_baseline(deployment_id.service_id)

        # Compare current metrics to baseline
        for (metric_name, current_value) in metrics_stream:
            expected_mean = baseline[metric_name]["mean"]
            expected_std = baseline[metric_name]["std"]

            # Fixed threshold check
            if metric_name == "error_rate" and current_value > 0.01:
                return HealthStatus.UNHEALTHY, f"error_rate {current_value} > 0.01"

            # Statistical anomaly check
            z_score = (current_value - expected_mean) / expected_std
            if abs(z_score) > 3.0:  # 3 standard deviations from mean
                return HealthStatus.UNHEALTHY, f"{metric_name} z-score {z_score}"

            # Business metric check
            if metric_name == "conversion_rate" and current_value < baseline["conversion_rate"]["mean"] * 0.95:
                return HealthStatus.UNHEALTHY, f"conversion_rate dropped 5% below baseline"

        return HealthStatus.HEALTHY, "all_metrics_normal"
```

**Component 2: Root Cause Analyzer**

When the health monitor detects an anomaly, the root cause analyzer determines the most likely cause. It uses a decision tree:

1. Is the error rate elevated only on instances running the new version? If yes, the likely cause is a code bug in the new version. Action: roll back the deployment.

2. Is the error rate elevated on both old and new version instances? If yes, the likely cause is a shared infrastructure issue (database, cache, network). Action: pause the deployment, alert the infrastructure team, do not roll back (the old version is also affected).

3. Is the latency elevated only during the traffic switch? If yes, the likely cause is a connection pool exhaustion or cache warming issue. Action: extend the pre-switch warmup period, retry the switch.

4. Is the business metric (conversion rate) dropping but the technical metrics are normal? If yes, the likely cause is a user experience regression in the new version. Action: roll back the feature flag or the deployment.

5. Is a dependency service showing elevated error rate but the deployed service is normal? If yes, the likely cause is that the deployment increased load on the dependency. Action: scale the dependency service, do not roll back the deployment.

**Component 3: Corrective Action Selector**

The corrective action selector maps the diagnosis to an action:

```python
class CorrectiveActionSelector:
    def select_action(self, diagnosis):
        actions = {
            "code_bug_in_new_version": "rollback_deployment",
            "shared_infrastructure_issue": "pause_deployment",
            "traffic_switch_transient": "retry_switch_with_warmup",
            "business_metric_regression": "rollback_feature_flag",
            "dependency_overloaded": "scale_dependency",
            "configuration_error": "rollback_configuration",
            "health_check_misconfiguration": "pause_and_alert",
        }
        return actions.get(diagnosis, "alert_human_operator")
```

**Component 4: Deployment State Manager**

The deployment state manager executes the corrective action and updates the deployment state. It ensures that the corrective action does not conflict with other ongoing deployments:

```python
class DeploymentStateManager:
    def __init__(self):
        self.active_deployments = {}

    def execute_action(self, deployment_id, action):
        if action == "rollback_deployment":
            # Set the deployment to rollback mode
            # The deployment orchestrator handles the actual rollback
            self.active_deployments[deployment_id].status = "ROLLING_BACK"
            # Notify the deployment orchestrator
            orchestrator.rollback(deployment_id)
            # Log the action for audit
            self.log_action(deployment_id, "rollback", "code_bug_in_new_version")

        elif action == "pause_deployment":
            self.active_deployments[deployment_id].status = "PAUSED"
            orchestrator.pause(deployment_id)
            self.log_action(deployment_id, "pause", "shared_infrastructure_issue")

        elif action == "rollback_feature_flag":
            flag_service.disable_flag(deployment_id.service_id, deployment_id.feature_flag)
            self.active_deployments[deployment_id].status = "FLAG_ROLLED_BACK"
            self.log_action(deployment_id, "flag_rollback", "business_metric_regression")
```

**Self-healing loop:** The four components operate in a continuous loop:

1. Health monitor detects anomaly (within 30 seconds of occurrence)
2. Root cause analyzer diagnoses the cause (within 10 seconds)
3. Corrective action selector chooses the action (within 1 second)
4. Deployment state manager executes the action (within 10 seconds for rollback, 30 seconds for scaling)
5. Health monitor verifies the corrective action resolved the anomaly (within 60 seconds)
6. If the anomaly persists, escalate to a human operator

The total time from anomaly detection to corrective action is under 60 seconds for rollback scenarios and under 120 seconds for scaling scenarios. This is significantly faster than the typical 5-15 minute human response time.

**Key insight:** A self-healing deployment system requires deep integration between the deployment orchestrator, the monitoring system, and the infrastructure management system. The deployment orchestrator must be able to execute rollback, pause, scale, and flag-toggle commands programmatically. The monitoring system must support real-time anomaly detection with statistical methods, not just fixed thresholds. The infrastructure management system must support automated scaling of dependent services.

The self-healing system is not a replacement for human operators. It handles the 80 percent of deployment problems that have well-known causes and well-defined corrective actions. The remaining 20 percent of problems — those with novel root causes or complex interactions — are escalated to human operators. The self-healing system's primary value is reducing the human operator's cognitive load: they only need to intervene for the most complex problems, not for every deployment hiccup.
