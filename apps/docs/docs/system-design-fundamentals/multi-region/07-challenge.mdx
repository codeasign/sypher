## Tier 1: Foundation

### Challenge 1: The TTL Trade-Off

A system uses Route53 geo-DNS with a TTL of 300 seconds (5 minutes) for the production DNS record. A regional failure occurs at 14:00 UTC. The failover process completes at 14:01 UTC, updating the DNS record to point to the standby region. However, a significant portion of users continue hitting the failed region for several minutes. Identify the problem, calculate the impact, and propose a fix.

**Model Answer:**

The problem is the DNS TTL value. With a 300-second TTL, DNS resolvers and client caches will keep the old DNS record for up to 300 seconds after the update. The failover completed at 14:01, but users whose DNS resolver last refreshed at 13:59 will not query DNS again until 14:04. This means users experience errors from 14:00 (the failure time) until their DNS cache expires — up to 14:04 at worst.

The impact window extends from 14:00 to 14:04, a 4-minute outage for some users beyond the failover completion time. The actual outage duration for individual users equals: time until failover (1 minute) plus remaining TTL on their cached record (up to 5 minutes from last refresh). A user whose DNS was refreshed at 13:55 experiences 6 minutes of errors total.

The fix: Reduce the TTL to 60 seconds or less. This limits the maximum DNS propagation delay to 60 seconds after the DNS update. The failover RTO becomes: health check detection (15 seconds) plus failover execution (15 seconds) plus DNS propagation (60 seconds) = 90 seconds total. The trade-off is increased DNS query volume — from 12 queries per hour per resolver to 60 queries per hour — but this is negligible for modern DNS infrastructure. For production systems, many teams use a 60-second TTL as standard.

A secondary fix: Use anycast IP addresses that do not change on failover. The same IP is advertised from the new region via BGP. BGP convergence takes 30 to 60 seconds and does not depend on DNS TTL. This eliminates the DNS propagation delay entirely.

### Challenge 2: The Replication Lag Budget

A company has three regions: us-east-1 (primary), eu-west-1 (read replica), and ap-southeast-1 (read replica). The application serves a read-heavy news feed. The SLO states that 99 percent of read requests must return data that is no more than 2 seconds stale. The average write rate is 1,000 writes per second. The average write size is 2 KB. Calculate the minimum cross-region bandwidth needed to meet the SLO, accounting for the fact that replication can use up to 70 percent of the available bandwidth.

**Model Answer:**

Total replication throughput: 1,000 writes per second times 2 KB per write = 2,000 KB/s. This is approximately 16 Mbps.

If replication can use up to 70 percent of the available bandwidth, the actual bandwidth provisioned must be at least 16 Mbps divided by 0.70 = 22.9 Mbps. Rounding up, you need a minimum 25 Mbps link between the primary region and each replica region.

However, this calculation uses average throughput. Replication lag depends on peak throughput, not average. If the write rate spikes to 10,000 writes per second during a flash sale or marketing campaign, the required bandwidth is 20,000 KB/s divided by 0.70 = 228 Mbps. The system must be provisioned for the peak, not the average, or the replication lag will exceed the SLO during spikes.

A better approach: Target the peak write rate of 10,000 writes per second. At 2 KB per write, this is 160 Mbps. With the 70 percent utilization cap, provision 230 Mbps of bandwidth. During normal operation (1,000 writes per second), replication completes in under 200 milliseconds. During peak (10,000 writes per second), replication completes in under 2 seconds — just meeting the SLO.

If bandwidth is constrained, implement write prioritization: user-facing writes (posting a status update) get priority over non-critical writes (analytics, logging). The priority queue allocates 80 percent of bandwidth to priority writes. For the peak scenario, priority writes (say 2,000 per second) use 184 Mbps and complete in under 300 milliseconds. Non-critical writes (8,000 per second) use the remaining 46 Mbps and complete in 2.8 seconds, which is acceptable for analytics data.

### Challenge 3: The Read-Your-Write Guarantee

A user in Singapore updates their profile picture through the application, which has its primary database in us-east-1 and a local read replica in ap-southeast-1. After the update, the user immediately refreshes the page and sees the old profile picture. Explain why this happens and design a solution.

**Model Answer:**

The root cause is asynchronous replication. The user's write went to the primary in us-east-1 and the application confirmed the update. But the read from the refresh goes to the local read replica in ap-southeast-1, which has not yet received the replication update. The replication lag between us-east-1 and ap-southeast-1 is typically 150 to 200 milliseconds, but if the user refreshes within that window, they see the old data.

Solution: Read-after-write consistency. After a user performs a write, the response includes a timestamp or version number for the written record. When the user performs a subsequent read, the application checks whether the local replica's replication watermark has passed that timestamp.

Implementation options:

1. **Timestamp-based routing**: The write response includes a `last_write_at` timestamp. The read request includes this timestamp. The application checks the local replica's replication lag. If `current_time - replication_lag` is greater than `last_write_at`, the replica is fresh enough and the read uses the local replica. Otherwise, the read is redirected to the primary region.

2. **Session sticky routing**: After a write, all subsequent reads from that user session are routed to the primary region for a configurable duration (typically 2 to 5 seconds). After that window expires, reads return to the local replica. This is simpler to implement but sends more traffic to the primary.

3. **Version watermark**: Each user record has a version number. The read replica tracks the highest version it has received from replication. If the user's version is higher than the replica's watermark, the read goes to the primary. This avoids timestamp-based calculations but requires version tracking per record.

For profile picture updates, option 2 is the simplest and most common approach. The session cookie stores a `write_region` flag set to "primary" for 2 seconds after a write. During those 2 seconds, reads bypass the local replica. The trade-off is that 2 seconds of slightly higher latency for reads after a write is imperceptible to the user.

## Tier 2: Design

### Challenge 3: Global Inventory Management System

Design a multi-region inventory management system for a global e-commerce platform with the following requirements:

- Warehouses in the US, Germany, Japan, and Brazil
- Each warehouse has its own local inventory database
- Customers worldwide browse the product catalog and place orders
- Inventory counts must be accurate within 1 percent of the real count
- Order fulfillment (reserving inventory) must succeed within 2 seconds for 99.9 percent of requests
- The system must handle 50,000 orders per hour at peak

**Rubric for Evaluation:**

| Criteria | Excellent (5 pts) | Good (3 pts) | Poor (1 pt) |
|----------|-------------------|--------------|-------------|
| Regional architecture | Clearly defines per-warehouse sharding with local primary databases; global catalog with multi-region replication | Identifies need for regional sharding but lacks detail on global catalog | Proposes a single global database or ignores regional concerns |
| Inventory consistency | Uses per-warehouse strong consistency for inventory deductions; optimistic concurrency control with version numbers | Proposes strong consistency but without specific mechanism | Relies on eventual consistency for inventory |
| Cross-region order fulfillment | Uses saga pattern with compensating transactions; reserves inventory in the warehouse region; provides timeout and rollback logic | Handles cross-region orders but without compensating transactions | Proposes distributed transactions across regions |
| Global catalog | Multi-region read replicas for catalog; eventual consistency with 5-second staleness acceptable; per-region caching | Replicates catalog but without staleness consideration | Single-region catalog serving all global traffic |
| Latency optimization | Localized reads and writes for within-region operations; cross-region operations use optimized network paths; CDN for static catalog assets | Optimizes reads but not writes | No mention of latency optimization |
| Failure handling | Region failure isolates to that warehouse; remaining regions continue; inventory reconciliation after recovery | Handles region failure but without reconciliation plan | No failure handling described |

**Model Answer for Excellent Rating:**

The architecture uses regional sharding with four independent warehouse databases, each in its own cloud region. The warehouse in Germany uses eu-central-1. The warehouse in Japan uses ap-northeast-1.

Each warehouse database is a local PostgreSQL instance with a synchronous replica in a different availability zone within the same region. Inventory writes are strongly consistent within the warehouse region. Each inventory record has a version number. When an order attempts to reserve inventory, it issues an UPDATE with a WHERE clause checking the version: `UPDATE inventory SET quantity = quantity - 1, version = version + 1 WHERE product_id = X AND warehouse_id = Y AND version = expected_version`. If the version does not match (concurrent update), the reservation fails with a conflict. The application retries up to 3 times.

The global product catalog is a separate system. It stores product descriptions, images, and pricing. Each region hosts a read-only copy of the catalog, served from a CDN for static assets. Catalog changes are propagated via S3 cross-region replication for images and a change-data-capture pipeline for pricing. Catalog staleness is under 60 seconds and is acceptable because a 1-minute-old price is close enough for browsing; the final price is confirmed at checkout.

For cross-region orders: a customer in Argentina orders a product that is only available in the US warehouse. The order is placed in the nearest region (us-east-1). The application in us-east-1 processes the payment locally and then sends a ReserveInventory message to the US warehouse's SQS queue in us-east-1 (which is collocated because the warehouse is in the same region). The inventory reservation is local: under 10 milliseconds. If the customer orders a product available in Germany, the sa-east-1 region application sends a ReserveInventory message to the eu-central-1 SQS queue. The eu-central-1 worker processes the reservation locally. If the reservation succeeds, the order continues. If it fails or times out after 5 seconds, the sa-east-1 application sends a RollbackOrder message to a compensation queue. The entire saga is tracked with a correlation ID.

For failure handling: if the German warehouse region becomes unavailable, orders that require German inventory are queued for later processing or redirected to alternative warehouses if available (with a price or delivery time adjustment). The US, Japan, and Brazil warehouses continue operating independently. After the German region recovers, a reconciliation process compares the pending orders queue against actual inventory levels and resolves any discrepancies.

### Challenge 4: Real-Time Chat Across Three Regions

Design a multi-region chat system where users can send messages and receive them with less than 200 milliseconds latency regardless of their region. The system must support group chats with members in different regions. The maximum group size is 500 members.

**Rubric for Evaluation:**

| Criteria | Excellent (5 pts) | Good (3 pts) | Poor (1 pt) |
|----------|-------------------|--------------|-------------|
| Message propagation | Uses a fan-out pattern with per-region delivery queues; messages stored in CRDT-based global store | Proposes global pub-sub without regional delivery optimization | All messages through one region |
| Membership and presence | Global user directory with multi-region read replicas; presence information via localized WebSocket connections with cross-region sync | Global user directory but no presence consideration | Single-region user directory |
| Group chat | Messages for groups with members in multiple regions are replicated to all relevant regions; uses vector clocks for ordering | Handles cross-region groups with latency penalty | All group messages through one region |
| Offline support | Messages persisted per-region with cross-region replication; causal ordering preserved | Messages persisted but ordering may break across regions | No offline support or order guarantee |
| Latency | In-region delivery under 10ms; cross-region delivery under 200ms | Cross-region delivery over 200ms | Cross-region delivery over 500ms |

**Model Answer for Excellent Rating:**

Three regions: us-east-1, eu-west-1, ap-southeast-1. Each region runs a chat service with WebSocket connections to local users. Messages are stored in Cassandra with a per-region local table for fast writes.

When a user sends a message, it is written to the local Cassandra table (under 5 milliseconds). The local chat service then broadcasts the message to all recipients in the same region through the local WebSocket connections (under 10 milliseconds total). For recipients in other regions, the local service publishes the message to a cross-region SNS topic. Each region subscribes to the cross-region topic and receives messages from other regions via a subscription endpoint. When a region receives a cross-region message, it looks up which local users are in the target group and delivers over their local WebSocket connections.

For group chats with members across all three regions: a user in Singapore sends a message. The ap-southeast-1 region writes the message locally and broadcasts to local group members. The message is published to the cross-region SNS topic. Within 150 milliseconds, us-east-1 and eu-west-1 receive the message and deliver to their local group members. Total time from send to delivery in the farthest region: under 200 milliseconds.

For ordering, each message carries a vector clock. The vector clock tracks the causal relationship between messages: message M1 from user A in Singapore has vector [A:1, B:0, C:0]; message M2 from user B in London replying to M1 has vector [A:1, B:1, C:0] because B knows about M1. When M1 arrives in London after M2 (due to network latency), the chat client uses the vector clock to display M1 before M2, preserving causal order. Concurrent messages from different users that do not reference each other can be displayed in any order.

### Challenge 5: Multi-Region Authentication Service

Design an authentication service that operates across four regions and must maintain strong consistency for authentication decisions. A user should never be able to authenticate with revoked credentials. The system handles 100,000 authentication requests per second globally.

**Model Answer:**

Authentication cannot tolerate eventual consistency. If a user's access is revoked, every subsequent authentication attempt must be rejected. This means the authentication decision must be strongly consistent.

Architecture: A single primary region for authentication writes (credential creation, revocation, password changes) with synchronous replication to a local standby within the same region. Read replicas in other regions are NOT used for authentication decisions. Every authentication request, regardless of the user's location, is forwarded to the primary region.

This means authentication latency equals the network round trip from the user's region to the primary region. For a user in Singapore with a primary in us-east-1, this is approximately 200 milliseconds. For a user in London, it is 80 milliseconds. 100,000 requests per second times approximately 10 KB per request (token, credentials) equals 1 GB/s of authentication traffic to the primary region. The primary region must be provisioned to handle this load.

To reduce the load, use token caching on the client side. After authentication, the user receives a signed JWT with a short expiration (5 minutes). The client presents this JWT for subsequent requests. The application verifies the JWT signature locally without contacting the authentication service. Only when the JWT expires (every 5 minutes) does the client re-authenticate. This reduces the load on the authentication service from 100,000 requests per second to approximately 100,000 divided by (5 minutes times 60 seconds) = 333 requests per second.

For revocation: when a user's access is revoked, the authentication service increments a global revocation counter stored in the primary region's database. Each JWT includes the current revocation counter for the user. If the JWT's counter is lower than the global counter, the JWT is rejected. The application verifies this by checking the revocation counter on each request — either by reading from a local cache of revocation counters (updated every 60 seconds) or by forwarding the request to the primary region for sensitive operations.

For failover: if the primary authentication region fails, a standby region takes over. During the failover window (approximately 60 seconds), authentication requests are queued or returned with a "try again" response. Clients retry with exponential backoff. The failover must ensure that the standby region has the latest revocation data — this requires a consensus-based approach where revocation data is replicated synchronously to at least one other region.

## Tier 3: Expert

### Challenge 6: Conflict Resolution at Scale

Your company deploys an active-active Cassandra cluster across five regions. Each region handles writes independently. After a major sports event, two regions processed conflicting ticket reservations for the same seat. 5,000 users were affected. The current conflict resolution uses last-writer-wins based on Cassandra's built-in timestamp mechanism. Because of clock skew between regions (maximum measured skew: 2.3 seconds), some valid reservations were silently discarded.

Design a conflict resolution strategy that handles this class of problem. Discuss the trade-offs between correctness, latency, and complexity.

**Discussion Points:**

The fundamental issue is that last-writer-wins is too simplistic for resources that should have unique allocation — a seat can only be sold once. The clock skew makes LWW unreliable because a write from a region with a faster clock always wins, regardless of the actual order of operations.

**Option 1: Application-level conditional writes**. Before reserving a seat, the application checks if the seat is available by reading at QUORUM consistency level across all regions. If available, it writes with a conditional expression: `UPDATE seats SET status = 'reserved', user_id = ? WHERE seat_id = ? AND status = 'available' IF status = 'available'`. This is a lightweight transaction supported by Cassandra. If two regions attempt to reserve the same seat simultaneously, one succeeds and the other fails. The loser returns an error to the user. This eliminates conflicts entirely but adds latency because the write requires QUORUM acknowledgment across regions (approximately 150 to 250 milliseconds for a 3-region QUORUM).

**Option 2: Deterministic seat assignment**. Instead of allowing any region to reserve any seat, partition seats by region. Seats 1 through 20,000 are reserved in us-east-1 only. Seats 20,001 through 40,000 are reserved in eu-west-1 only. Each region has exclusive ownership of its seat range. Writes are local and fast (under 10 milliseconds). No cross-region coordination needed. The trade-off is reduced flexibility: a user in Europe cannot buy a US-assigned seat unless the application redirects them. This is acceptable if seat assignments are logically tied to geographic sections of a venue.

**Option 3: CRDT-based seat counting with a quorum for finalization**. Use a CRDT counter per seat that tracks pending reservations. Each region can increment the counter locally (fast, under 5 milliseconds). The actual seat assignment is fnalized by a separate service that reads the CRDT counters at QUORUM, compares them, and confirms the winner. The user gets a "pending" status immediately and a "confirmed" or "rejected" status within 2 seconds. This gives fast local writes with eventual correctness, but adds complexity with the two-phase approach.

For the ticket reservation use case, Option 1 (conditional writes at QUORUM) is the best balance. The 200-millisecond latency for writes is acceptable for ticket purchasing. The failure mode is clear: the user knows immediately whether they got the seat. No hidden conflicts, no silent discards.

### Challenge 7: DNS-Free Global Routing

Design a global routing mechanism that does not depend on DNS for directing traffic to the correct region. DNS has known issues: caching, TTL propagation delays, and resolver misbehavior. Your design should provide sub-second failover and per-request routing granularity. Discuss the trade-offs compared to DNS-based routing.

**Discussion Points:**

DNS-free routing can be achieved through several mechanisms:

**Anycast**: The simplest DNS-free approach. The same IP address is advertised from multiple regions via BGP. When a user connects to this IP, BGP routes them to the nearest data center that advertises the IP. If a data center fails, BGP automatically converges to the next closest data center. Failover is sub-second — limited by BGP convergence time, which is typically under 30 seconds and often under 5 seconds for well-configured networks.

The trade-off: Anycast is not per-request granular. BGP makes routing decisions at the network layer based on AS-path length and other metrics, not on application-level concerns like load, user identity, or compliance requirements. You cannot route a specific user to a specific region; you route all traffic from a network to the same data center. Compliance-based routing (EU users to EU regions) requires IP-based geolocation on the application side to redirect after the connection is established.

**Client-side routing with load balancer SDK**: AWS Global Accelerator and Google Cloud Load Balancing provide anycast front-ends with regional backends. The client connects to a global anycast IP, which terminates at the nearest edge location. The edge location forwards the request to the designated backend region based on a routing configuration. In AWS Global Accelerator, you can set traffic dials per region (0 to 100 percent) and update them in seconds. This provides DNS-free routing with sub-second failover when you change the traffic dial. The trade-off is vendor lock-in and cost (AWS Global Accelerator costs approximately $0.025 per GB processed).

**Application-layer redirect**: The client connects to any region's load balancer. The load balancer checks a routing service that determines which region should handle this request based on user identity, compliance rules, and current health. If the initial region is not the correct one, the load balancer returns an HTTP 307 redirect to the correct region's endpoint. The client follows the redirect. This adds one round trip for misrouted requests but requires no DNS changes for failover.

The trade-off: Every request initially lands on a potentially wrong region, incurring latency for the redirect. For a well-configured system where most users are already in the correct region, this overhead is minimal. For failover scenarios, the redirect ensures that traffic moves instantly without DNS propagation delays.

**Hybrid approach (recommended)**: Use anycast for the first connection (via Global Accelerator or CloudFront), use HTTP redirects for compliance-based routing, and use DNS TTL of 60 seconds as a backup. The anycast layer provides sub-second failover for the network path. The redirect layer provides application-level routing precision. DNS serves as a fallback for clients that do not support your preferred routing protocol. This hybrid approach combines the speed of anycast with the flexibility of application-level routing while maintaining backward compatibility with standard DNS-based clients.
