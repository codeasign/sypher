## The Problem Story

Imagine you are building a social media platform. A user updates their profile picture. They refresh the page and see the old picture. They refresh again. Still the old picture. Their friends scroll through the feed and also see the old picture. The user is frustrated. The friends are confused. Someone files a bug report: "Profile pictures never update."

You investigate. The database has the new picture. The API returns the new picture when called directly. But the web server cache is serving a copy of the old picture from ten minutes ago. The cache did what caches do: it kept data around so subsequent requests would be fast. Nobody told it the data changed.

This is the cache invalidation problem. It shows up everywhere.

- An e-commerce site shows "in stock" for an item that sold out three hours ago. Customers order it. Support gets flooded with cancellation requests.
- A news site publishes a correction. Readers who visited the article in the last hour still see the incorrect version. The correction becomes a second correction.
- A ride-sharing app shows a driver's estimated arrival as three minutes away. The driver passed the pickup point thirty seconds ago. The passenger waits, watching a stale dot on a frozen map.
- A CI/CD dashboard shows a build as "running" for four hours. The build failed two minutes after it started. The dashboard cache never checked again.

Each of these is the same underlying problem: something in the system cached a value, the source of truth changed, and no mechanism existed to tell the cache the old value was no longer valid.

The cache is not malicious. It is doing exactly what it was built to do: serve fast responses by keeping data close at hand. The problem is the missing link between the database writer and the cache. When the source changes, something must tell the cache to let go of its copy.

## The Insight

Cache invalidation means removing or updating cached data when the source data changes, so that subsequent reads see fresh data instead of stale data.

Think of a library reference shelf. The librarian keeps popular books on a display shelf so patrons can grab them without searching the stacks. When a new edition of a book arrives, the librarian removes the old edition from the display shelf and puts the new one in its place. If the librarian only added new editions without removing old ones, patrons would keep grabbing the outdated edition from the shelf. The display shelf would be worse than useless: it would actively mislead.

The cache is the display shelf. Invalidation is the librarian removing the old edition.

The challenge is that in a distributed system, the "librarian" has to know about every single update across potentially hundreds of database nodes, propagate that knowledge to every cache node in the cluster, and do it fast enough that users do not notice the gap. And it has to do this without slowing down the system so much that the cache stops being useful.

This insight leads to a design principle: every cache must have a defined invalidation strategy. If you cannot articulate how a cache entry becomes invalid, you have not designed your caching layer. You have merely added a time bomb.

## Core Vocabulary

These are the terms you need to discuss cache invalidation precisely.

**Invalidation.** The act of marking a cached entry as no longer fresh. The entry may be removed immediately, or it may be kept but flagged so the next read skips it. Invalidation is not the same as eviction: eviction frees memory, invalidation protects correctness.

**TTL (Time-To-Live).** A pre-configured duration after which a cache entry is automatically considered stale. TTL is the simplest invalidation mechanism: set it when the entry is written, and the cache does the rest. Every serious cache system supports TTL. It matters because it provides a safety net even when explicit invalidation fails.

**Stale data.** A cached value that no longer matches the source of truth. Users see stale data when the cache serves an entry that should have been invalidated but was not. The time between source update and cache invalidation is the staleness window.

**Cache coherency.** A property of a cache system: every read returns the most recently written value. Strong coherency means invalidation is immediate and global. Weak coherency means stale reads are possible within some bounded window. Most distributed cache systems choose weak coherency for performance.

**Write-invalidate.** A strategy where an update to the source data sends an invalidation message to the cache, removing the stale entry. The next read misses the cache and fetches fresh data. This is the most common invalidation strategy in production systems.

**Write-update.** A strategy where an update to the source data sends the new value directly to the cache, overwriting the entry. The next read hits the cache and gets fresh data immediately. Write-update uses more bandwidth but eliminates the miss penalty.

**Purge.** A bulk invalidation operation that removes multiple cache entries at once, usually matching a pattern or tag. Purges are common when a large dataset changes (for example, regenerating a product catalog). Purge must be used carefully: too broad, and you empty the cache and cause a thundering herd.

**Cache tag.** A metadata label attached to a cache entry that groups it with related entries. Tags allow selective invalidation: when a source object changes, you invalidate all entries with the matching tag. Tags are the foundation of fine-grained invalidation.

**Selective invalidation.** Invalidating only the specific cache entries affected by a data change, rather than purging everything. Selective invalidation minimizes cache misses but requires a mapping from source changes to cache keys. This mapping is the core complexity of invalidation systems.

**Invalidation storm.** A cascade of invalidations that overloads the cache or the backing database. This happens when a single sources changes triggers invalidation of many cache entries, all of which miss and hit the database simultaneously. Invalidation storms are a primary operational risk of any invalidation system.

**Cross-instance invalidation.** Invalidation that propagates across multiple cache nodes in a distributed deployment. A cache hit on one node must be invalidated when the data changes on another. Cross-instance invalidation requires a coordination protocol or a shared event bus.

**Eventual consistency.** A consistency model where the cache does not immediately reflect source updates but converges over time. Eventual consistency is the default for most distributed caches. The staleness window depends on invalidation propagation delay.

**Staleness window.** The time between a source data change and the moment the cache reflects that change. TTL-based systems have a staleness window of 0 to TTL. Event-based systems have a staleness window equal to the invalidation propagation delay.

The flow breaks down into two phases:

**The invalidation phase** (steps 1–4). A writer updates the database. The database confirms. Then invalidation is triggered: either by the writer itself, by a database trigger, or by a change data capture process. The cache receives the invalidation event and marks the affected entries as stale or removes them entirely.

**The recovery phase** (steps 5–9). A client makes a read request. The cache checks its entry for the requested key. The entry is marked stale, so the cache skips it — a cache miss. The request goes to the database, which returns the fresh value. The cache stores the fresh value and returns it to the client. Subsequent reads hit the fresh cache entry until the next invalidation.

The critical detail: nothing in this flow is synchronous by default. The writer does not wait for the invalidation to complete before returning. The cache invalidation happens in the background. This is why stale reads are still possible: the window between step 2 and step 4 is the staleness window.

## How It Actually Works

Cache invalidation, in practice, involves a sequence of coordinated operations across multiple system components. Here is the detailed walkthrough.

### Step 1: Data Changes

The source of truth (database, file system, external API) receives an update. This could be a SQL UPDATE, a NoSQL document write, an S3 object put, or any other mutation.

### Step 2: Invalidation Triggered

The invalidation must be triggered somehow. There are three common trigger mechanisms.

### Step 3: Invalidation Manager Receives Event

In an event-based system, an invalidation manager component receives the raw event (for example, "user_123 updated") and translates it into specific cache keys or tags. This translation step is where the mapping between source data and cache entries lives.

### Step 4: Key Resolution

The manager resolves which cache entries are affected. This may involve:

- A direct key mapping: source ID `user_123` maps to cache key `user:123`
- A tag-based mapping: source ID `user_123` maps to tags `user:123`, `org:456`, `feed:789`
- A pattern-based mapping: all keys matching `user_123:*`
- A query-based mapping: entries whose content references the changed data

### Step 5: Invalidation Action

The manager performs the invalidation action on each resolved entry. The action can be:

- **Delete**: remove the entry immediately. Simplest, most common.
- **Mark stale**: set a flag so the entry is not served but can be refreshed on next read.
- **Overwrite**: replace the value with fresh data (write-update).

### Step 6: Cache Removes Entry

The cache performs the operation on its internal data structure. In a single-node cache, this is a hash map delete. In a distributed cache, the operation must be sent to the specific node holding the entry (or broadcast to all nodes if using replication).

### Step 7: Next Read Finds Invalid Entry

A client makes a read request. The cache looks up the key. If the entry was deleted, the key is not found at all. If the entry was marked stale, the cache sees the stale flag and treats it as a miss.

### Step 8: Cache Miss Triggers DB Query

The cache miss causes the request to fall through to the database. The database returns the current value.

### Step 9: Cache Populated with Fresh Data

The cache stores the fresh value, optionally with a new TTL.

### Step 10: Response Returned

The fresh data is returned to the client.

### The Role of TTL in the Lifecycle

TTL is both a safety net and a primary invalidation mechanism. As a safety net, TTL ensures that even if explicit invalidation fails, the cache entry will eventually become stale and be refreshed. This bounds the maximum staleness window.

As a primary mechanism, TTL is the simplest form of invalidation: just set a time duration, and the cache handles everything. Many systems use TTL as the sole invalidation strategy for data that changes at known intervals, such as a configuration file that is regenerated hourly.

The trade-off is that TTL creates a hard upper bound on staleness but no lower bound. A value invalidated by TTL could have been stale for the entire TTL duration. For data that changes unpredictably, TTL alone is insufficient.

## Variants

There are several approaches to cache invalidation, each with different characteristics.

### TTL-Based Invalidation

TTL-based invalidation is the simplest approach. Each cache entry has a time-to-live duration. When the TTL expires, the entry is automatically invalidated. No external signals, no coordination, no infrastructure.

When to use: data that changes at predictable intervals, data where bounded staleness is acceptable, or as a safety net alongside another invalidation strategy.

### Explicit Invalidation (Write-Invalidate)

Write-invalidate is the most common approach in production systems. When the source data changes, an explicit invalidation message is sent to the cache. The cache removes or marks the entry, and the next read fetches fresh data.

The invalidation message can originate from:

- The application layer: the service that writes the data also purges the cache.
- A middleware layer: a database trigger or CDC pipeline detects changes and sends invalidation events.
- A dedicated invalidation service: a standalone component that monitors data changes and manages cache coherence.

### Write-Update

Write-update takes the invalidation concept one step further. Instead of just marking the entry as stale, the invalidation message includes the new value. The cache updates in place, and subsequent reads hit the cache immediately.

Write-update reduces the cache miss penalty but increases bandwidth consumption and places more burden on the cache to accept large payloads.

- The new value is small (a few hundred bytes at most).
- The read-to-write ratio is high — you want every read to hit.
- The invalidation bus is fast and reliable.
- The cache nodes have sufficient write throughput.

### Purge by Pattern or Tag

When a single data change affects many cache entries, you need bulk invalidation. This is achieved through patterns or tags.

**Pattern-based purge**: invalidate all keys matching a pattern, such as `user:123:*`. This approach is simple but risky: patterns can accidentally match unintended keys, and scanning for keys by pattern is expensive in some cache systems (Redis SCAN, for example).

**Tag-based purge**: attach one or more tags to each cache entry. When data changes, invalidate all entries with a specific tag. This approach is more controlled but requires a tag index: a secondary data structure that maps tags to key sets.

### Time-Based vs Event-Based

These two approaches sit on different sides of the invalidation spectrum.

**Time-based (TTL)** is simple, reliable, and requires no external coordination. It is appropriate when:

- Data changes at predictable intervals.
- Bounded staleness is acceptable (the user can tolerate seeing data up to N seconds old).
- You want a safety net alongside event-based invalidation.
- The operational cost of maintaining an invalidation pipeline is not justified.

**Event-based** is more complex but provides tighter staleness bounds. It is appropriate when:

- Data changes unpredictably.
- Staleness must be minimized (seconds or milliseconds).
- You are already building a CDC or event bus for other purposes.
- The cost of stale data (revenue loss, user trust) justifies the infrastructure.

Many production systems use both: event-based invalidation for primary freshness, and TTL as a safety net for when events are lost or delayed.

## Mental Model Check

Test your understanding with these questions.

**Question 1.** A cache entry has a TTL of 60 seconds. The source data changes 5 seconds after the entry is cached. What is the maximum possible staleness window?

**Answer.** 55 seconds. The entry could remain stale from t=5 until the TTL expires at t=60. The next read after t=60 will miss and get fresh data. The minimum staleness window is 0 seconds (if the data changes at t=60 exactly), but the maximum is TTL minus the time between caching and change.

**Question 2.** You are designing an e-commerce product page cache. A single product detail page includes the product name, price, description, and inventory count. The price changes daily, but inventory changes every few seconds. What invalidation strategy do you recommend?

**Answer.** Use two separate caches or cache entries with different TTLs. Cache the product name, price, and description with a 1-hour TTL (they change rarely). Cache the inventory count with a 10-second TTL (it changes frequently). This avoids invalidating the entire product page every time inventory decrements. Alternatively, use tag-based invalidation where inventory updates only invalidate the inventory portion.

**Question 3.** Write-update seems strictly better than write-invalidate because there is no cache miss. Is it always better? Why or why not?

**Answer.** No. Write-update requires sending the new value to the cache, which increases bandwidth and may not be practical if the value is large (an image, a serialized object, a rendered HTML page). Write-update also increases the write load on the cache, which could affect read performance. Write-invalidate sends just a key (a few bytes) and lets the cache lazily fetch the value on the next read, which distributes the reload load over time.

**Question 4.** You send an invalidation message for a cache key, but the message is lost. What happens? How do you recover?

**Answer.** The cache entry remains fresh from the cache's perspective but corresponds to stale source data. Without a safety net, this stale entry could be served indefinitely. The recovery mechanisms are: (1) TTL safety net — the entry will eventually expire and be refreshed; (2) retry mechanism — the invalidation publisher retries on failure; (3) checksum or version check — the cache compares a version number with the database on each read, lazily detecting staleness.

**Question 5.** You have 10 cache nodes in a cluster. An invalidation message is sent but only reaches 8 of them. The remaining 2 nodes continue serving stale data. Does this matter, and how do you fix it?

**Answer.** Yes, this matters. Any request routed to the 2 stale nodes gets stale data. The fix depends on the routing mechanism: (1) if requests are routed by consistent hashing, ensure invalidation messages also use consistent hashing to reach the correct node; (2) if the cache uses replication, broadcast invalidation to all replicas; (3) implement a gossip protocol where nodes share invalidation state; (4) add a short TTL as safety net so even uninvalidated entries eventually expire; (5) use a shared invalidation log that all nodes consume from.
