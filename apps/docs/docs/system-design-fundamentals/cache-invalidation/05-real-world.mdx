## Company Case Studies

Every large-scale internet company has faced the cache invalidation problem. Their solutions reveal patterns that generalize across architectures.

### Twitter: Manhattan DB and Cache Invalidation

Twitter's caching layer evolved through several generations. In the early days, Twitter used Redis with TTL-based invalidation. As the platform grew to hundreds of millions of users, this approach broke down.

Twitter built Manhattan, a distributed key-value store, to replace Redis for their core caching needs. Manhattan provides explicit invalidation through its write path: every write to Manhattan triggers an invalidation event that is propagated to all read replicas.

Key lessons from Twitter:

- **Integrate invalidation into the storage layer.** When the database itself manages invalidation, there is no risk of the writer forgetting to invalidate.
- **Use read replicas for cache population.** Invalidations propagate to replicas, which re-populate from their local copy of the data.
- **Fanout service for timeline construction.** Timeline writes fan out to multiple caches, and each cache entry is individually invalidated when the source tweet changes.

### Facebook: TAO Cache Invalidation

Facebook's TAO (The Association Object) cache is a distributed cache for the social graph. It was built explicitly to solve the invalidation problems Facebook encountered with memcached.

TAO uses leader-follower replication with write-through semantics. Every write goes through a TAO leader, which updates both the database and the cache. The leader then propagates the write to TAO followers.

Key lessons from Facebook:

- **Write-through invalidation.** The leader updates the cache synchronously with the database. The writer always sees fresh data.
- **Followers are eventually consistent.** Followers receive asynchronous updates, which means stale reads are possible for non-writers. This is acceptable for the social graph use case.
- **Cold-start mitigation.** TAO pre-populates follower caches with popular data to avoid cold-start misses.

### GitHub: Repository Cache Purge

GitHub caches repository metadata, commit histories, file contents, and pull request data. A single repository update (a push, a PR merge, an issue comment) can affect dozens of cache entries across multiple services.

GitHub uses a combination of Redis and memcached with an event-driven invalidation system. When a repository event occurs, GitHub's internal event bus (based on Kafka) broadcasts an invalidation event to all services that cache repository data.

Key lessons from GitHub:

- **Service-specific cache invalidation.** Each service maintains its own cache and subscribes to invalidation events relevant to it. No central cache coordinator is needed.
- **Event-driven with domain-specific events.** Events are not raw database changes but domain events ("push", "PR merged") that carry semantic meaning.
- **Short TTL safety net.** A 60-second TTL ensures that even if the event bus fails, stale data persists for at most one minute.

### Netflix: Zuul Event Bus Invalidation

Netflix uses a microservice architecture where each service manages its own data and caches. Invalidation is coordinated through the Zuul event bus, which provides a centralized publish-subscribe mechanism.

When a service updates data, it publishes an invalidation event to Zuul. All services that cache that data subscribe to the relevant events. If a service subscribes but does not process the event within a configurable timeout, the subscriber is marked as degraded and the event is retried.

Key lessons from Netflix:

- **Service-owned caches.** Each microservice owns its cache and knows exactly which events invalidate which entries.
- **Subscriber timeout.** If a subscriber does not process an invalidation event within a timeout, it is degraded. This prevents slow subscribers from causing stale data across the system.
- **EVCache (Ephemeral Volatile Cache).** Netflix's in-house cache solution supports TTL-based expiry as a fallback, with event-driven invalidation as the primary mechanism.

## Open Source Solutions

Open source caching systems offer built-in invalidation mechanisms that can be used out of the box or customized.

### Redis Keyspace Notifications

Redis supports keyspace notifications: pub/sub messages emitted when keys are modified or expire. These can be used for invalidation monitoring.

```bash
# Enable keyspace events for expired and evicted keys
CONFIG SET notify-keyspace-events Eg

# Subscribe to expired events on database 0
PSUBSCRIBE __keyevent@0__:expired

# Example output when key "user:123:profile" expires
# 1) "psubscribe"
# 2) "__keyevent@0__:expired"
# 3) "user:123:profile"
```

Limitations: Redis pub/sub is fire-and-forget. If no consumer is subscribed when a key expires, the event is lost. For reliable invalidation, use Redis streams with consumer groups instead of pub/sub.

### Apache Ignite Continuous Queries

Apache Ignite is a distributed database and caching platform. It supports continuous queries: SQL queries that stay active and emit events when the underlying data changes.

```java
// Register a continuous query that monitors product price changes
ContinuousQuery<String, Product> query = new ContinuousQuery<>();
query.setInitialQuery(new SqlQuery<>(Product.class, "price > 100.0"));

query.setLocalListener(events -> {
    for (CacheEntryEvent<? extends String, ? extends Product> event : events) {
        String cacheKey = event.getKey();
        Product newValue = event.getValue();
        // Invalidate dependent cache entries
        cacheService.invalidateByTag("product:" + newValue.getCategoryId());
    }
});

cache.query(query);
```

Continuous queries are useful for cross-cache coherence: when data in one cache changes, other caches are notified and can invalidate dependent entries.

### Hazelcast Invalidation Events

Hazelcast is a distributed in-memory data grid. It supports invalidation events through its near-cache feature. When a remote node updates an entry, the near-cache on other nodes is automatically invalidated.

```java
// Configure near-cache with invalidation
NearCacheConfig nearCacheConfig = new NearCacheConfig()
    .setInvalidateOnChange(true)
    .setTimeToLiveSeconds(300)
    .setEvictionConfig(new EvictionConfig()
        .setEvictionPolicy(EvictionPolicy.LRU)
        .setMaxSizePolicy(EvictionConfig.MaxSizePolicy.ENTRY_COUNT)
        .setSize(10000));

mapConfig.setNearCacheConfig(nearCacheConfig);
```

Hazelcast's invalidation is built on a cluster-wide event bus. When a map entry is updated, the updating node broadcasts an invalidation event to all cluster members. The near-cache on each member checks the event against its cached keys and invalidates matching entries.

### Caffeine Invalidation API

Caffeine is a high-performance local caching library for Java. It provides a rich invalidation API with support for individual keys, multiple keys, and predicate-based invalidation.

```java
Cache<String, Product> cache = Caffeine.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .maximumSize(100_000)
    .build();

// Individual invalidation
cache.invalidate("product:42");

// Bulk invalidation by predicate
cache.invalidateAll(key -> key.startsWith("product:"));

// Refresh (reload asynchronously)
cache.refresh("product:42");

// Schedule periodic invalidation
cache.schedule().expireAfter("product:42", 30, TimeUnit.SECONDS);
```

Caffeine's invalidation is local only. For distributed invalidation, combine Caffeine with a remote event bus.

## Cloud Provider Solutions

### AWS ElastiCache with SNS-Based Invalidation

AWS ElastiCache (Redis or Memcached) does not provide built-in invalidation coordination across clusters. You build your own invalidation pipeline using AWS services.

Common pattern:

1. Application writes to DynamoDB or RDS.
2. DynamoDB Streams or RDS change events are captured via AWS Lambda.
3. Lambda publishes an invalidation message to SNS.
4. All cache nodes subscribe to the SNS topic and invalidate affected keys.
5. Lambda also updates ElastiCache directly (write-update pattern).

```json
// DynamoDB Streams event → Lambda → SNS invalidation
{
  "Records": [
    {
      "eventID": "1",
      "eventName": "MODIFY",
      "dynamodb": {
        "Keys": { "id": { "S": "product:42" } },
        "NewImage": { "price": { "N": "9.99" } }
      }
    }
  ]
}
```

### GCP Memorystore Invalidation Patterns

GCP Memorystore (managed Redis) invalidation typically uses Pub/Sub as the coordination layer.

Pattern:

1. Write to Cloud SQL or Firestore.
2. Cloud SQL change events through Pub/Sub.
3. Cloud Run or GKE service consumes events and invalidates Memorystore.

For high-throughput scenarios, use a Dataflow pipeline to batch invalidation events before sending to Memorystore.

### Azure Redis with Event Grid

Azure Redis Cache integrates with Azure Event Grid for event-driven invalidation.

1. Write to Azure Cosmos DB or SQL Database.
2. Change Feed (Cosmos DB) or Change Tracking (SQL) generates events.
3. Events flow through Event Grid.
4. Azure Function consumes events and invalidates Redis cache keys.

Azure also provides a "Redis Cache Invalidation" Event Grid system topic that directly monitors Redis key expiry and eviction events.

## Lessons Learned

### Lesson 1: Event-Based Beats TTL

Across all case studies, event-based invalidation is the primary mechanism, and TTL is the safety net. Event-based invalidation provides sub-second staleness; TTL provides a maximum bound for when events are lost.

Do not rely on TTL alone. The operational complexity of an event pipeline is justified even at moderate scale.

### Lesson 2: Make Invalidation Idempotent

Invalidation events should be safe to process multiple times. Processing the same invalidation twice should have the same effect as processing it once: the cache entry is removed.

Idempotency allows you to retry failed invalidations, replay events for recovery, and tolerate at-least-once delivery from the event bus without side effects.

### Lesson 3: Do Not Trust Invalidation for Correctness

Cache invalidation operates in a distributed, concurrent system. Events can be lost, delayed, or reordered. The cache is an optimization, not a source of truth.

Design your application so that stale cache data causes degraded performance, not incorrect behavior. For example, a stale product price might cause a customer to see the old price momentarily, but the order processing system must always read the fresh price from the database.

### Lesson 4: Measure Your Staleness

You cannot fix what you do not measure. Instrument your invalidation system with end-to-end staleness tracking. For each cache read, record:

- The time the source data was last modified (from the database).
- The time the cache entry was last refreshed.
- The staleness = last_modified - last_refreshed.

Track the p50, p95, p99, and p100 of staleness values over a sliding window. Alert when p99 staleness exceeds your target.

### Lesson 5: Test Partition Scenarios

Network partitions are rare in production but catastrophic when they occur. Test your invalidation system under partition scenarios:

- Simulate a partition between the invalidation manager and a subset of cache nodes.
- Verify that the system continues serving data (eventually consistent is acceptable).
- When the partition heals, verify that the consistency checker detects and repairs stale entries.
- Measure how long it takes for the system to reach consistency after the partition heals.

Use chaos engineering tools (Chaos Monkey, Gremlin) to inject partition failures in a controlled environment.
