## Naive Implementation

Most teams start with a spreadsheet. The AI lead exports token usage from the provider dashboard once a week, pastes it into Google Sheets, and manually calculates estimated costs. The spreadsheet has columns for date, model, input tokens, output tokens, and estimated cost. A simple formula multiplies token counts by pricing.

This works for the first month. Then the team grows. Multiple models are used. The provider dashboard shows aggregate usage but not per-feature or per-team breakdowns. The weekly spreadsheet becomes a guessing game. The CTO asks how much the new chatbot feature costs, and the AI lead cannot answer.

What breaks in the naive approach:

- **No real-time visibility**: The spreadsheet is updated weekly. By the time you see a cost spike, it has been running for days.
- **No per-feature attribution**: All costs are aggregated. You cannot tell which feature is burning money.
- **No budget enforcement**: The spreadsheet is read-only. There is no way to stop a runaway process.
- **No anomaly detection**: A misconfigured retry loop can burn $80,000 before the next spreadsheet update.
- **No optimization feedback**: Without granular cost data, you cannot measure the impact of optimizations. You do not know if caching saved 20% or 40%.

### Cost Tracking Service

The cost tracking service is a lightweight event processor that receives cost events from every inference call. It runs as a sidecar in the application pod or as a standalone service. Key design decisions:

- **Event format**: Structured JSON with fields for provider, model, input tokens, output tokens, calculated cost, latency, request ID, user ID, feature ID, and team ID.
- **Ingestion**: Events are sent via UDP to minimize overhead (no retry, no acknowledgement). The service listens on a local UDP socket and batches events before writing to the database.
- **Storage**: Time-series data goes to Prometheus (7-day retention). Aggregated data goes to ClickHouse or BigQuery (indefinite retention). Raw events go to S3/GCS for audit trail (90-day retention).
- **Scaling**: The service is stateless and horizontally scalable. Each instance handles 5,000 events per second. A consistent hash ring routes events by user ID for locality.

### Budget Enforcer

The budget enforcer is a critical service that must be fast, accurate, and always available. It runs on every request path and must complete in under 5ms.

The budget enforcer maintains budgets in Redis for fast access. Each budget is a sorted set with timestamps as scores and cumulative costs as values. The service checks budget consumption by summing values in the current time window.

```python
class BudgetEnforcer:
    def check_budget(self, team_id, estimated_cost):
        # Get current window consumption
        current = self.redis.zscore(
            f"budget:{team_id}:{self.window_key()}",
            "cumulative"
        ) or 0.0

        # Check against budget
        budget = self.budgets[team_id]
        consumption_pct = (current + estimated_cost) / budget.monthly_limit

        if consumption_pct >= budget.hard_cap:
            return BudgetDirective.DENY
        elif consumption_pct >= budget.soft_cap:
            return BudgetDirective.DOWNGRADE
        else:
            return BudgetDirective.ALLOW
```

The budget enforcer uses a two-phase approach: provisional deduction before inference, then actual deduction after. This prevents race conditions at budget boundaries.

### Billing Aggregator

The billing aggregator runs as a batch job every hour. It reads raw cost events from the event store, aggregates them by time period (hourly, daily, monthly), and writes the aggregated data to the billing database.

The aggregator handles three types of aggregation:

1. **Direct costs**: Per-request costs attributed to a specific team, feature, and user.
2. **Shared costs**: Infrastructure costs (cache servers, monitoring, the cost optimization system itself) that are allocated across teams proportionally.
3. **Provider adjustments**: Discounts, credits, and taxes that are applied after the fact.

### Cost Allocation Engine

The cost allocation engine is a reporting service that generates chargeback data. It queries the aggregated billing data and produces reports by team, feature, model, user, and time period.

The allocation engine supports three allocation models:

- **Direct attribution**: Costs are attributed to the team that owns the feature making the call.
- **Proportional allocation**: Shared costs are split by request volume or token usage.
- **Tiered allocation**: Premium teams pay a higher share of shared costs because they use more expensive infrastructure.

### Model Pricing Registry

The model pricing registry is a configuration service that stores current and historical prices for all models and providers. It is the source of truth for cost calculations.

The registry stores pricing data with effective dates, so the system can accurately calculate historical costs even if prices have changed. It also supports regional pricing variants and volume discount tiers.

### Optimizer Feedback Loop

The optimizer feedback loop is the intelligence layer. It analyzes cost trends, identifies optimization opportunities, and suggests routing changes. The loop runs as a weekly batch job that produces a report with:

- Cost trends by feature, team, and model
- Optimization opportunities ranked by estimated impact
- A/B test results for proposed routing changes
- Quality metrics for each model tier

## Integration Patterns

### Integration with LLM Gateways

The cost optimization system integrates with LLM gateways (Kong, Envoy, custom gateways) at the middleware layer. The gateway intercepts every API call and runs the classification and routing logic before forwarding to the model provider.

Integration points:
- **Request interceptor**: The gateway calls the model selection engine before forwarding the request.
- **Response interceptor**: After the response, the gateway emits a cost event to the cost tracking service.
- **Rate limiter**: The gateway enforces rate limits from the budget enforcer.

### Integration with Observability

Cost events are emitted as OpenTelemetry spans. The span attributes include cost, tokens, model, and provider. This allows cost data to be correlated with latency, error rates, and other observability signals in the same dashboard.

The integration is straightforward:
```python
with tracer.start_as_current_span("llm_inference") as span:
    response = provider.invoke(prompt)
    span.set_attribute("cost", calculate_cost(response))
    span.set_attribute("input_tokens", response.usage.input_tokens)
    span.set_attribute("output_tokens", response.usage.output_tokens)
    span.set_attribute("model", response.model)
```

### Integration with Billing Systems

The cost allocation engine exports chargeback data to corporate billing systems (NetSuite, Workday, or custom ERP). The export format is a CSV file with columns for team, feature, cost, and time period, generated daily and uploaded to an S3 bucket.

### Integration with CI/CD Cost Gates

Cost gates prevent expensive code from reaching production without review. The CI/CD pipeline triggers a cost estimation step that simulates the cost impact of the proposed changes. If the estimated cost exceeds a threshold, the pipeline fails and requires a cost review.

```yaml
# .github/workflows/cost-gate.yml
steps:
  - name: Estimate Cost Impact
    run: |
      python scripts/estimate_cost_impact.py \
        --feature my-new-feature \
        --estimated-requests 100000 \
        --estimated-tokens 1500 \
        --model gpt-4
  - name: Cost Gate Check
    run: |
      python scripts/cost_gate.py --threshold 50000
```

## Multi-Region and Global Scale

### Regional Pricing Differences

Model providers charge different prices in different regions. Regional pricing differences can be 10-30%:

| Region | GPT-4 Input (per M tokens) | GPT-4 Output (per M tokens) |
|---|---|---|
| US East (N. Virginia) | $30.00 | $60.00 |
| EU West (Ireland) | $33.00 | $66.00 |
| Asia Pacific (Tokyo) | $34.50 | $69.00 |
| South America (Sao Paulo) | $45.00 | $90.00 |

A global cost optimization system routes requests to the cheapest region that meets latency requirements. For a user in Europe, routing to US East is 10% cheaper but adds 80ms of latency. The routing decision considers the latency budget: if the user has 500ms of budget, the 80ms latency cost is worth the 10% price savings.

### Data Transfer Costs

Moving data between regions incurs egress costs. For a request that routes from Europe to US East, the input prompt (2,000 tokens = ~3KB) costs $0.00006 in data transfer. The output (500 tokens = ~750B) costs $0.000015. These are negligible compared to the inference cost. But for training data or large batch operations (GBs of data), data transfer costs become significant.

### Multi-Cloud Cost Management

Using multiple cloud providers requires managing different pricing models, discount structures, and minimum commitments. The cost optimization system maintains a unified view of costs across all providers and handles the complexity of:

- Converted currency: Prices in USD, EUR, and local currencies.
- Discount structures: Percentage discounts, tiered pricing, and committed use discounts.
- Minimum commitments: Monthly minimum spend that must be met regardless of usage.
- Surcharges: Regional surcharges, data transfer surcharges, and API surcharges.

## Capacity Planning

### Cost Per User at Scale

Understanding cost per user is essential for pricing and capacity planning. The cost per user depends on usage patterns, not just user count. Here is a typical breakdown:

| User Tier | Avg Requests/Day | Avg Tokens/Request | Daily Cost | Monthly Cost |
|---|---|---|---|---|
| Free | 5 | 500 input, 100 output | $0.01 | $0.30 |
| Basic | 20 | 1,000 input, 200 output | $0.08 | $2.40 |
| Pro | 50 | 2,000 input, 500 output | $0.55 | $16.50 |
| Enterprise | 100 | 4,000 input, 1,000 output | $3.00 | $90.00 |

With optimization (caching, routing, compression), these costs drop by 40-60%:
- Free: $0.18/month
- Basic: $1.20/month
- Pro: $8.25/month
- Enterprise: $45.00/month

### Break-Even Analysis for Caching

The break-even point for a cache is where the cache infrastructure cost equals the savings from reduced API calls. For a semantic cache costing $5,000/month:

- Monthly API cost without cache: $500,000
- Monthly API cost with 40% cache hit rate: $300,000
- Savings: $200,000 - $5,000 = $195,000 net benefit
- Break-even hit rate: $5,000 / $500,000 = 1%

Even a 1% hit rate saves enough to cover the cache infrastructure. The ROI of caching is overwhelmingly positive for any application with any degree of request repetition.

### ROI of Batching

Batching requires accepting additional latency. The ROI calculation must account for the value of the latency:

- Batch discount: 50%
- Fraction of requests that can be batched: 60%
- Additional latency for batched requests: 3 seconds
- Value of latency: If 20% of users churn due to 3 seconds of additional latency, and each user is worth $10/month, the cost of batching is 20% of 60% of user base x $10.

For a system with 100,000 users:
- Batch savings: 60% x 50% x $500,000 = $150,000/month
- User churn cost: 20% x 60% x 100,000 x $10 = $120,000/month
- Net benefit: $30,000/month

The net benefit is positive but requires careful monitoring of user experience metrics.

### Reserved Capacity

Reserved capacity requires a commitment of monthly spend in exchange for lower per-token pricing. The break-even analysis compares the commitment amount to the projected usage:

- Current monthly spend: $500,000
- Reserved capacity discount: 30%
- Commitment: $400,000/month minimum
- Effective cost with reservation: $400,000 + (actual spend - $400,000) x 0.70
- If actual spend stays at $500,000: $400,000 + $100,000 x 0.70 = $470,000
- Savings: $30,000/month

The risk is that usage drops below the commitment. If usage drops to $300,000, the cost becomes $400,000 (you pay the commitment regardless), which is $100,000 more than the pay-as-you-go cost.

## Deployment and Operations

### Cost Anomaly Detection

Cost anomalies are detected using a statistical method based on historical usage patterns. The system maintains a baseline of cost per minute for each model, feature, and team. When current cost deviates from the baseline by more than 3 standard deviations, an anomaly is flagged.

The anomaly detection system has three tiers:

1. **Spike detection**: Current cost exceeds baseline by 3x for 5 minutes.
2. **Trend detection**: Cost is increasing at more than 200% of the historical rate for 30 minutes.
3. **Pattern break detection**: The distribution of costs across models changes significantly (e.g., 80% of traffic suddenly shifts from GPT-3.5 to GPT-4).

### Budget Alerting

Budget alerts are configured with multiple thresholds, each with a different notification channel:

| Threshold | Alert Type | Channel | Response Time |
|---|---|---|---|
| 50% of budget | Summary | Slack | 24 hours |
| 75% of budget | Warning | Slack + Email | 4 hours |
| 90% of budget | Critical | PagerDuty | 15 minutes |
| 100% of budget | Emergency | PagerDuty + Phone | 5 minutes |

### Chargeback Reports

Chargeback reports are generated weekly and distributed to each team. The report shows:

- Total cost for the team
- Cost by feature
- Cost by model
- Cost trend (week over week, month over month)
- Top 10 most expensive users
- Optimization recommendations with estimated savings

The reports are generated as PDFs and published to an internal dashboard. Each team can drill down into their own data to understand cost drivers.

## Error Handling and Resilience

Every component must handle failures gracefully. If the cost tracking service is down, the system still serves requests. If the budget enforcer fails, requests proceed with conservative defaults. The system is designed to favor availability over strict cost control during failures.

### Failure: Cost Tracking Service Down

When the cost tracking service is unavailable, the application falls back to local tracking. Each instance maintains a local buffer of cost events in memory (up to 10,000 events). When the service recovers, the buffer is flushed to the central database. If the buffer overflows (unlikely under normal conditions), older events are dropped with a warning log. This ensures no critical cost data is lost while preventing memory pressure during extended outages.

### Failure: Budget Enforcer Unavailable

If the budget enforcer cannot be reached (Redis is down, network partition), the system allows requests through but logs every decision for audit. A cached copy of budget data (updated every 60 seconds from the primary) is used for routing decisions. If the cache is also stale beyond 5 minutes, the system allows all requests with a warning log.

This is a deliberate design choice. It is better to spend slightly over budget than to block legitimate requests due to an infrastructure failure. The expected overspend during a budget enforcer outage is bounded by the recovery time multiplied by the average spend rate.

### Failure: Pricing Registry Stale

If the pricing registry has stale data, the system uses the most recent prices available and logs a warning. The operations team receives an alert. Stale pricing data causes inaccurate cost tracking, but routing decisions remain valid because they rely on relative model pricing which changes slowly.

## Testing the Cost Optimization System

Testing requires both functional tests (does it work correctly?) and cost tests (does it save money?).

### Functional Tests

Unit tests verify each component in isolation:

```python
def test_classifier_routes_simple_query_to_cheap_model():
    query = "What is the return policy?"
    result = classifier.classify(query)
    assert result.complexity_score < 0.3
    assert result.recommended_tier == "budget"

def test_cache_returns_cached_response():
    cache.set("What is the return policy?", "Returns accepted within 30 days.")
    result = cache.get("What is the return policy?")
    assert result == "Returns accepted within 30 days."

def test_budget_enforcer_blocks_over_budget():
    enforcer = BudgetEnforcer(monthly_budget=1000)
    enforcer.record_spend(1000)
    assert enforcer.check_budget(estimated_cost=10) == BudgetDirective.DENY
```

Integration tests verify the end-to-end flow: a request flows through classification, routing, caching, and inference. Cost events are emitted and aggregated correctly. Chargeback reports contain accurate data.

### Cost Tests

Cost tests measure the system's effectiveness:

- **Replay test**: Historical traffic is replayed through the production system offline. The test measures cost savings compared to a baseline with no optimization.
- **Regression test**: Synthetic traffic with known characteristics verifies that cost per request, cache hit rate, and compression ratios remain within expected ranges after each deployment.
- **Load test**: Traffic spikes are simulated to verify that the budget enforcer handles the spike without blocking legitimate requests.

### Chaos Engineering for Cost

Introduce controlled failures to test resilience:

- Simulate a provider price increase of 50% and verify that the routing system shifts traffic to cheaper alternatives.
- Simulate a budget constraint that reduces available budget by 50% mid-month and verify that non-critical requests are downgraded.
- Simulate a cache service outage and verify that all requests fall through to the API without exceeding latency budgets.

## Capacity Planning Examples

### Example 1: Estimating Cost for a New Feature

Your product team proposes adding an AI-powered search feature. They estimate 50,000 queries per day with 3,000 input tokens and 200 output tokens each, using GPT-4:

- Input cost per day: 50,000 x 3,000 x $30 / 1,000,000 = $4,500
- Output cost per day: 50,000 x 200 x $60 / 1,000,000 = $600
- Total cost per day: $5,100
- Total cost per month: $153,000

With optimization (60% savings from caching, routing, and compression): estimated cost per month of $61,200. The cost gate in CI/CD would flag this as exceeding the $50,000 threshold but approve it with the optimization plan attached.

### Example 2: ROI of Self-Hosting

Your application processes 5 million requests per day. Current GPT-4 cost is $500,000 per month. Evaluate self-hosting Llama 3 70B on 8 x A100 GPUs:

- GPU cost: 8 x $3/hour x 24 hours x 30 days = $17,280
- Engineering maintenance: $20,000/month (0.5 FTE)
- Amortized setup: $100,000 / 24 months = $4,167/month
- Total self-hosting cost: $41,447/month

Savings: $500,000 - $41,447 = $458,553/month. Payback period: $100,000 setup / $458,553 = 0.22 months.

The risk is quality: the self-hosted model may not match GPT-4 for all tasks. The system routes only suitable requests to the self-hosted model and falls back to GPT-4 for complex queries.

### Example 3: Multi-Region Cost Optimization

Your application serves users in North America (60%), Europe (25%), and Asia-Pacific (15%). Current architecture routes all traffic to US East. Monthly cost is $500,000.

Route European traffic to EU West: 10% cheaper regional pricing. Savings: $500,000 x 25% x 10% = $12,500/month. Route APAC traffic to Tokyo: 15% more expensive but 200ms less latency. Decision: accept higher cost for better user experience.

Net savings: $12,500/month, plus better latency for European users and improved resilience from distributed traffic.

## Cost Governance

Technical optimization is only half the battle. The other half is building an organizational culture where cost is a first-class concern.

### Cost Ownership

Every feature should have a cost owner. The cost owner is responsible for understanding the feature's cost structure, implementing optimizations, and responding to cost anomalies. Cost ownership is assigned when the feature is designed, not after it is already burning money.

### Cost Reviews

Schedule monthly cost reviews with each product team. The agenda: review cost trends, identify optimization opportunities, discuss upcoming features that impact costs, and adjust budget allocations. The tone should be collaborative. The goal is to help teams optimize, not to blame them.

### Cost Training

Every engineer working on AI features should understand the basics of cost optimization. Provide training on: how token pricing works, common optimization techniques, how to estimate the cost of a new feature, and how to use the cost tracking tools.
