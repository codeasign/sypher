## Tier 1: Foundation Exercises

These exercises test your grasp of the core formulas and concepts. You should be able to complete each one in under 5 minutes.

### Exercise 1.1: Queueing Delay Calculation

A database server processes 200 queries per second on average (mu = 200). Each query takes 5ms of actual processing time. The arrival rate is 180 queries per second (lambda = 180).

Calculate:
1. The utilization (rho) of the database server.
2. The average queueing delay (Wq) using the M/M/1 formula.
3. The average total latency (processing + queueing).
4. The average number of queries in the system (L).

**Model Answer:**

```
rho = lambda / mu = 180 / 200 = 0.9 (90% utilization)

Wq = rho / (mu - lambda)
   = 0.9 / (200 - 180)
   = 0.9 / 20
   = 0.045 seconds = 45ms queueing delay

Total latency = processing + queueing = 5ms + 45ms = 50ms

L = lambda × W = 180 × 0.050 = 9 queries in the system

At 90% utilization, queueing delay is 9x processing time.
If we added another server (180 queries across 2 servers):
  rho per server = 90/200 = 45%
  Wq = 0.45 / (200 - 90) = 0.45 / 110 = 4.1ms
  Total latency: 5ms + 4.1ms = 9.1ms (5.5x improvement)
```

### Exercise 1.2: Cache Hit Rate Impact

An API serves 50,000 requests per second. A cache lookup takes 1ms. A database query takes 15ms. Currently, the cache hit rate is 80 percent.

Calculate:
1. The average latency at 80 percent hit rate.
2. The average latency if you improve the hit rate to 95 percent.
3. The database queries per second saved by the 15 percent hit rate improvement.
4. The new database utilization (assume the database can handle 15,000 QPS).

**Model Answer:**

```
1. Average latency at 80% hit rate:
   = 0.80 × 1ms + 0.20 × (1ms + 15ms)
   = 0.8ms + 3.2ms = 4.0ms

2. Average latency at 95% hit rate:
   = 0.95 × 1ms + 0.05 × (1ms + 15ms)
   = 0.95ms + 0.8ms = 1.75ms

3. Database QPS saved:
   Before: 50,000 × 0.20 = 10,000 QPS
   After: 50,000 × 0.05 = 2,500 QPS
   Saved: 7,500 QPS (75% reduction)

4. Database utilization:
   Before: 10,000 / 15,000 = 67% (in the efficient zone)
   After: 2,500 / 15,000 = 17% (vastly underutilized)
   The database now has headroom for other workloads
   or can be scaled down to reduce cost.

The 15% hit rate improvement cut latency by 56% and
reduced database load by 75%. This is why cache optimization
is the highest-ROI performance investment.
```

### Exercise 1.3: Connection Pool Sizing

A web application handles 4,000 requests per second. Each request requires a database query that takes an average of 25ms. The application uses a connection pool.

Calculate the minimum connection pool size using Little's Law. Then add 30 percent headroom. If each server can handle 200 concurrent connections, how many servers are needed?

**Model Answer:**

```
Using Little's Law: L = λ × W
L = 4,000 × 0.025 = 100 connections (minimum)

With 30% headroom: 100 × 1.3 = 130 connections

Servers needed at 200 connections per server:
130 / 200 = 0.65 → 1 server

But wait — we also need to consider thread pool capacity:
Each request holds a thread while waiting for the database.
At 4,000 RPS with 25ms latency, L = 100 in-flight requests.
With 200 threads per server, the thread pool is fine.

However, if the database query latency includes queueing at
the database (which it always does under load), the effective
latency is higher. At 70% database utilization and 25ms
processing, total latency ≈ 25 + (0.7/0.3 × 25) = 83ms.
L = 4,000 × 0.083 = 332 connections → need 2 servers.
```

### Exercise 2.1: Social Media Feed Architecture

Design a timeline feed system that shows the latest 20 posts from a user's followed accounts. The system must support 10 million daily active users, with 100,000 feed loads per second at peak. Each feed load needs to query the 200 most recent posts from each of the user's 300 followed accounts, merge them by timestamp, and return the top 20.

**Your task:** Identify the latency bottleneck and throughput bottleneck. Design a caching strategy that reduces the P99 latency from a naive 15 seconds to under 200ms.

**Model Answer:**

The naive approach queries 300 accounts x 200 recent posts each = 60,000 posts fetched per feed load. At 100,000 feed loads per second, that is 6 billion post fetches per second — impossible.

The solution is a feed pre-computation (fanout-on-write):

```
Write path (when a user posts):
1. User creates a post → insert into posts table
2. Get all followers of this user (fanout to followers' feed caches)
3. For each follower: prepend the new post to their cached feed
4. Trim each follower's cached feed to 500 posts
5. If follower count > 10,000: switch to fanout-on-read
   (store post in a "recent posts by follows" list, merge on read)

Read path (when a user loads feed):
1. Check cache for pre-computed feed (key: "feed:user_id")
2. If cached: return top 20 posts (under 5ms)
3. If not cached (cold start): compute from follows' recent posts
   (expected: under 100ms for users with under 500 follows)

Latency breakdown after optimization:
Cache hit (99% of requests): 2-5ms
Cache miss (1%): 50-150ms (only for users with many follows)
P50: 3ms  |  P99: 50ms  |  P999: 200ms

Throughput:
Redis cache: 100,000 ops/s per node, can scale with sharding
Write fanout: each post creates N writes (one per follower)
     10M posts/day × 300 avg followers = 3B fanout writes/day
     = 35,000 writes/s
     Redis handles this with 3-5 shards.
```

### Exercise 2.2: E-commerce Checkout Optimization

An e-commerce site has a checkout flow that takes 3 seconds on average. The flow includes: validate cart (50ms), calculate shipping (200ms with a third-party API), apply promotions (500ms with a rules engine), process payment (1,500ms with a payment gateway), create order (50ms), send confirmation email (asynchronous). The site processes 200 orders per second.

**Your task:** Identify which steps can be parallelized, which can be cached, and which can be made asynchronous. Calculate the new median latency after your optimizations.

**Model Answer:**

```
Original flow (sequential):
Cart → Shipping → Promotions → Payment → Order → Email
50ms    200ms       500ms       1,500ms  50ms     async
Total: 2,300ms synchronous wait for user (£2.3 seconds)

Optimized flow:
1. Parallelize independent steps:
   Cart check (50ms) — must be first (validates items exist)
   Then parallel:
     ├─ Apply promotions (500ms) — depends on cart items
     ├─ Calculate shipping (200ms) — depends on cart items
     └─ Begin payment (1,500ms) — can start early with amount

   Parallel path latency = max(500ms, 200ms, 1,500ms) = 1,500ms
   But we can pipeline: start payment before promotions complete
   by using estimated amount, then adjust if promotions change it.

2. Cache what you can:
   Shipping rates by zip code + weight: cache for 1 hour
   (200ms → 2ms, save 198ms)
   Promotion results by cart hash: cache for 5 minutes
   (500ms → 5ms, save 495ms)

3. Make async what doesn't block the response:
   Confirmation email was already async (0ms on critical path)

New optimized flow:
Cart (50ms) → Parallel[Promotions(cache: 5ms), Shipping(cache: 2ms),
              Payment(1,500ms)] → Order (50ms) = 1,600ms

New median latency: 1,600ms (30% improvement)
With payment optimization (tokenization reduces payment to 500ms):
New median: 607ms (74% improvement from 2,300ms)
```

### Exercise 2.3: Database Replication Latency Budget

You are designing a database layer for a global social network. Writes originate from anywhere in the world. Users must see their own writes immediately (read-after-write consistency). The P99 latency target for writes is 100ms. The P99 latency target for reads is 50ms.

**Your task:** Design a replication strategy that meets these targets. Consider: geo-local writes, synchronous vs async replication, read-after-write routing, and the latency budgets for each hop.

**Model Answer:**

```
Design: Active-passive with geo-local writes and read-after-write routing

Architecture:
- 3 regions: US-East (primary), EU-West (replica), AP-Southeast (replica)
- Each region has a local cluster that accepts writes
- Writes are forwarded to the primary asynchronously
- To guarantee read-after-write: client includes a write timestamp
  in the read request. If the local replica's data is older than the
  timestamp, the read is forwarded to the primary.

Latency budget for write:
Client to local cluster: 5-20ms (network, depends on user location)
Local write acknowledgment: 5ms (write-ahead log commit)
Async replication to primary: accepted lag of 50-500ms
Total user-facing latency: 10-25ms (P99 = 50ms ✓ under 100ms target)

Read-after-write latency (worst case: user in Tokyo, primary in US-East):
Client in Tokyo → local replica: 15ms
Check: is replica caught up to write timestamp?
  - Compare write timestamp with replica's max timestamp
If replica is behind (lag > 0):
  Forward read to primary (US-East): 80ms one-way = 160ms RTT
  Primary responds: 10ms
  Total: 15ms + 160ms + 10ms = 185ms (EXCEEDS 100ms target!)

Solution: Route writes to the closest active region
If many users are in Tokyo, deploy a write-primary in Tokyo instead of
routing all writes through US-East. Make each region its own primary
for local users, and use conflict resolution for cross-region data.

Revised design: Multi-primary with CRDT-based conflict resolution
Tokyo primary for APAC users (write latency: 15ms + 5ms = 20ms)
US-East primary for Americas users (write latency: 10ms + 5ms = 15ms)
EU-West primary for European users (write latency: 5ms + 5ms = 10ms)
Cross-region replication: async, conflict resolution via CRDTs or LWW
Read-after-write: read from the same primary that handled the write
  (guaranteed by routing: same user always goes to same region)
```

### Exercise 3.1: Global Chat System Latency

A real-time chat application has 100 million users sending 1 billion messages per day. Messages must be delivered to the recipient in under 200ms end-to-end. Users are globally distributed.

**Your task:** Design the message delivery pipeline with explicit latency budgets for each component. Identify the bottleneck and explain how you handle the following scenarios:
1. A user in Japan sends a message to a user in Brazil.
2. A user sends a message to a group with 100,000 members.
3. A user sends a message while on a slow 3G connection in a tunnel.

**Model Answer:**

```
Latency budget (target: 200ms end-to-end):

Sender app → closest edge server: 10-40ms (varies by network)
Edge server processes message: 2ms (validate, persist, route)
Edge server → recipient's edge server (if different region):
  50-150ms (cross-region propagation)
Recipient's edge → recipient app: 10-40ms (push notification)
Processing and rendering on recipient: 10-20ms

Total: 82-252ms (under 200ms target for same-region messages,
may exceed for cross-continent messages)

Optimization for cross-continent delivery (Japan → Brazil):
1. Cannot reduce propagation delay below 160ms RTT
2. Solution: use persistent TCP connection from edge to edge,
   eliminates connection setup overhead
3. Accept that 200ms target may be 250ms for extreme distances
4. Alternative: route through intermediate edge (US-West) that
   has better peering, reducing effective delay by 20-30ms

Scenario 2: Group chat with 100,000 members
Naive: fan-out 100,000 messages from sender → bottleneck
Optimized: fan-out at edge (push once to each edge region,
then each edge region fans out to local recipients)
  Sender → edge server → 10 regional edge servers
    Each regional edge → 10,000 local recipients
  Total fan-out: 10 regional pushes, not 100,000 individual pushes
  Edge to recipient: WebSocket push (under 50ms)
  Total: 100ms (sender → edge) + 20ms (edge → regional edges) +
         50ms (regional edge → recipients) = 170ms ✓

Scenario 3: Slow 3G connection in a tunnel
1. Client buffers message locally until connectivity resumes
2. Client sends with timestamp, server accepts out-of-order
3. Server delivers immediately if recipient is online
4. Recipient gets message when network allows
5. Latency target does not apply — the network is the bottleneck
6. The system should not retry excessively (exponential backoff)
   and should provide delivery receipts when the message is sent
```

### Exercise 3.2: Capacity Planning for a Video Platform

A video platform serves 500,000 video views per hour. Each view averages 10 minutes of streaming at 5 Mbps. The platform needs to serve content globally with P99 startup latency under 5 seconds. Content is stored in a single primary region (US-East) and encoded in 4 bitrate levels.

**Your task:** Calculate the total bandwidth required. Design the caching and CDN strategy. Calculate how many edge servers are needed. Then calculate the cache hit rate needed to achieve the 5-second startup latency target.

**Model Answer:**

```
Step 1: Bandwidth calculation
500,000 views/hour = 139 views/second
Each view at 5 Mbps = 5 Mbps × 139 = 695 Mbps total bandwidth
With peak factor 3x: 695 × 3 = 2,085 Mbps = 2.1 Gbps

This is manageable for a single origin. The challenge is latency,
not bandwidth.

Step 2: Startup latency budget (5 seconds)
For a user in Europe connecting to US-East origin:
  DNS: 30ms
  TCP + TLS: 100ms
  Request video manifest: 50ms
  Request first video segment: 50ms
  Download first 2 seconds of video (5 Mbps × 2s = 1.25 MB):
    Serialization on 1 Gbps link: 10ms
    Propagation (US-East → Europe, 80ms one-way): 80ms
    Total segment download: 90ms
  Buffer 2 seconds of video for smooth start: 2,000ms (zero because
  streaming starts playing as data arrives)

Total startup: 30 + 100 + 50 + 50 + 90 = 320ms from origin
This is under 5 seconds — the origin can serve globally if
the network is uncongested.

But real-world congestion means cross-region latency degrades
during peak hours. The solution is edge caching.

Step 3: Edge caching strategy
Deploy edge servers in EU-West, AP-Southeast, AP-Northeast, US-West

Cache hit rate needed: if cache miss goes to US-East origin:
  Cache miss latency = 320ms (as calculated above)
  Cache hit latency (from local edge):
    DNS: 5ms (local resolver)
    TCP + TLS: 30ms (optimized, persistent connections)
    Video manifest: 5ms (local cache read)
    First segment: 10ms (local SSD read)
    Total: 50ms

  To achieve 5-second startup at P99:
    P99 cache hit rate must be high enough that the 320ms cache miss
    latency does not push P99 past 5 seconds.
    If 99% of requests are cache hits:
      P99 = mostly 50ms ✓ (well under 5s)
    If 80% are hits and 20% are misses (mixed in distribution):
      P99 falls in the cache miss range (320ms) → still under 5s ✓

    The 5-second target is easily achievable with edge caching.
    The real concern is throughput at the edge (serving 5 Mbps streams
    to thousands of concurrent viewers from each edge server).

Step 4: Edge server throughput calculation
Each edge server: 40 Gbps network capacity
Concurrent viewers per server: 40 Gbps / 5 Mbps = 8,000 viewers
At 139 viewers/second average with 10-minute average watch time:
  Concurrent viewers: 139 × 600 = 83,400
  Edge servers needed: 83,400 / 8,000 = 11 servers
With 3x peak factor: 33 servers across 4 regions = 9 per region

Conclusion: 33 edge servers globally, with 95%+ cache hit rate
for popular content, achieves under 500ms P99 startup latency.
The cache miss rate (under 5%) goes to origin at 320ms, still
well under the 5-second target.
```

### Exercise 3.3: Tail Latency Budget Allocation

You are designing a microservice for a financial trading platform. A single trade request must call 8 downstream services: authentication (50ms P99), risk check (100ms), market data lookup (200ms), order validation (50ms), fee calculation (50ms), ledger write (100ms), notification (50ms), and audit log (50ms).

**Your task:** The total P99 latency target for the trade is 500ms. Currently, the sum of all P99 latencies is 650ms. You must bring this under 500ms. You cannot change the individual service latencies — they are owned by other teams.

Describe at least three strategies to meet the 500ms target without changing individual service latencies.

**Model Answer:**

```
Strategy 1: Parallelize independent calls
Currently the services may be called sequentially:
  Auth → Risk → Market → Order → Fee → Ledger → Notify → Audit
  50ms   100ms   200ms   50ms    50ms   100ms    50ms     50ms  = 650ms

Identify which calls are independent:
  Group A (must be sequential): Auth → Risk (risk needs auth)
  Group B (parallelizable): Market data, Order validation, Fee calc
  Group C (depends on A + B): Risk result
  Group D (post-trade): Ledger write, Notification, Audit log

Parallel execution:
  Path 1: Auth (50ms) → Risk (100ms) = 150ms sequential
  Path 2: Market (200ms) + Order (50ms) + Fee (50ms) = 200ms parallel
  Path B result → Risk (already running, adds 0ms)
  Wait for both paths: max(150ms, 200ms) = 200ms
  Post-trade: Ledger + Notify + Audit = max(100ms, 50ms, 50ms) = 100ms
  Total: 200ms + 100ms = 300ms ✓ (under 500ms target)

Strategy 2: Speculative execution with a shorter deadline
For services with wide P99/P50 gaps:
  Market data: P50 = 50ms, P99 = 200ms (4x gap)
  Send the market data request to 2 replicas in parallel
  Use whichever responds first
  Effective P99 for market data with 2 hedged requests:
    P99_hedged = P99^2 = 0.2^2 = 0.04 = 4%
    P99 market data drops from 200ms to approximately 80ms
  Save: 120ms

Strategy 3: Degrade non-critical services in the hot path
Notification and audit log do not need to block the trade response.
Move them to async processing:
  Trade response returns to user after Ledger write (100ms)
  Notification and audit log fire asynchronously via queue
  Save: 50ms + 50ms = 100ms removed from critical path

Combined results:
  Original sum: 650ms sequential
  After Strategy 1 (parallelization): 300ms
  After Strategy 2 (hedged market data): 260ms
  After Strategy 3 (async notif + audit): 160ms
  Final: 160ms P99 ✓ (well under 500ms target)

Note: None of these strategies changed the individual service latencies.
They changed the ARCHITECTURE of how the trade request coordinates
the downstream calls. This is the difference between optimizing
code and optimizing system design.
```
