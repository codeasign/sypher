## Questions You Will Face

Latency and throughput questions appear in system design interviews in three forms: direct knowledge questions, bottleneck analysis in design questions, and troubleshooting scenarios.

### Direct Questions

These test your grasp of the fundamentals — definitions, relationships, and formulas.

**Question: Explain the relationship between latency and throughput in a queueing system.**

Your answer should reference queueing theory explicitly. Start with the utilization formula: as throughput approaches capacity, latency increases non-linearly because queueing delay grows as (utilization) / (1 - utilization). Give concrete numbers: at 50 percent utilization, queueing delay equals processing time. At 80 percent, queueing delay is 4x processing time. At 95 percent, queueing delay is 19x. Mention Little's Law (L = lambda * W) as the universal relationship.

**Question: How would you measure and optimize P99 latency?**

Start with measurement: timestamp requests at client and server, record in percentiles, use a histogram (HDR Histogram, Prometheus summary) with a sliding window of 5 minutes. Optimization strategies in order of impact: (1) add caching to eliminate slowest queries, (2) reduce queueing by scaling horizontally, (3) implement hedged requests, (4) add timeouts and circuit breakers, (5) optimize query performance. Always measure before optimizing — find the component contributing the most to the tail.

**Question: What is the difference between bandwidth and throughput?**

Bandwidth is the maximum theoretical data transfer rate of a network path. Throughput is the actual measured rate. Bandwidth is a capacity — a 10 Gbps link. Throughput is a usage — 3 Gbps actual transfer. Throughput is always under bandwidth due to protocol overhead, contention, packet loss, and retransmission. Goodput is an even narrower measure: useful data only, excluding headers and retransmissions.

**Question: How does caching affect both latency and throughput?**

Caching improves both simultaneously, which makes it unique among optimization techniques. Latency improves because a cache hit (under 1ms) replaces a database query (5-50ms). Throughput improves because the database receives fewer queries (only cache misses pass through), freeing capacity for more requests. A 95 percent cache hit rate reduces database load by 95 percent, allowing the same database to handle 20x more total requests.

**Question: Explain Amdahl's Law and its implications for a system where 60 percent of the workload is parallelizable.**

Amdahl's Law states that the maximum speedup from parallelization is limited by the serial portion of the workload. With 60 percent parallelizable and 40 percent serial, the maximum speedup with infinite processors is 1 / 0.4 = 2.5x. With 4 processors: 1 / (0.4 + 0.6/4) = 1 / 0.55 = 1.8x. With 16 processors: 1 / (0.4 + 0.6/16) = 1 / 0.4375 = 2.28x. The first 4 processors give 80 percent of the possible speedup. Beyond that, diminishing returns set in hard. This is why optimizing the serial portion of your system is more valuable than adding more parallel resources once you have enough.

**Question: What is the difference between serialization delay and propagation delay?**

Serialization delay is the time to convert data into bits on the wire. It depends on packet size and link speed: a 1 MB packet on a 1 Gbps link takes 8ms to serialize. Propagation delay is the time for those bits to travel from source to destination. It depends on distance and the speed of light in the medium (approximately 200,000 km/s in fiber). A 5,570 km fiber from New York to London has approximately 28ms one-way propagation. Serialization delay improves with faster links. Propagation delay improves only by moving the endpoints closer together.

**Question: A system shows P50 of 10ms and P999 of 5 seconds. What is the most likely cause, and what metric would you monitor to confirm?**

The 500x gap between median and tail strongly suggests garbage collection pauses in a managed runtime (Java, Go). Monitor GC pause duration, GC frequency, and heap utilization. If GC pauses exceed 1 second and occur during high throughput, the young generation is too small or the allocation rate exceeds the GC throughput. Alternative causes: lock contention (monitor thread dumps for blocked threads), kernel page faults (monitor major page faults), or disk I/O contention (monitor iowait). The fix is usually tuning GC parameters, reducing allocation rate, or moving to a concurrent GC algorithm.

**Question: How do you calculate the optimal batch size for a throughput-sensitive pipeline?**

Optimal batch size balances two costs: the per-batch overhead (fixed cost regardless of batch size) and the per-event latency budget (how long an event can wait before processing). The formula: optimal_batch_size = min(throughput * max_latency_budget / 2, max_batch_capacity). The division by 2 accounts for the average wait time being half the batch interval. For 1,000 events per second with a 5-second latency budget: 1,000 * 5 / 2 = 2,500 events per batch. If the database handles 10,000 rows per INSERT, the max batch capacity is 10,000, so the optimal batch size is 2,500.

**Question: Explain how TCP congestion control affects latency at high throughput.**

TCP congestion control algorithms (Cubic, BBR, Reno) detect congestion through packet loss or increasing RTT. When a packet is lost, the sender reduces the window size, which reduces throughput but also increases latency (the lost packet must be retransmitted and the window must recover). BBR (Bottleneck Bandwidth and Round-trip propagation time) is designed specifically to avoid bufferbloat — it estimates the available bandwidth and keeps the sending rate below the point where queues build up. BBR achieves lower latency at high throughput compared to Cubic, which fills buffers until they overflow.

### Bottleneck Analysis Questions

These appear within larger system design problems. You are asked to design a system, and the interviewer probes your understanding of where latency and throughput constraints will appear.

**Question: You are designing a URL shortener. How many servers do you need for 10,000 writes per second and 100,000 reads per second?**

Walk through the calculation using the numbers you know. A single PostgreSQL instance handles 5,000-10,000 QPS. At 10,000 write QPS, you need at least 2 database nodes for writes (with sharding). Redis handles 100,000 ops/s, so a single Redis cluster node can handle the read volume. Each application server handles 500-1,000 concurrent requests. Little's Law: at 100ms average latency, L = 100,000 * 0.1 = 10,000 concurrent requests. At 500 per server, you need 20 app server instances. Always include the headroom calculation (add 30 percent for spike tolerance).

**Question: A search service takes 500ms on average. Users complain. What do you check?**

The interviewer wants to see your diagnostic process, not a single answer. Walk through: (1) Is it all requests or a subset? Check P50 vs P99. If P99 is 2 seconds but P50 is 100ms, then tail latency is the problem. (2) Is the system CPU-bound or I/O-bound? Check CPU utilization, disk I/O, network latency. (3) Is queueing the issue? Check number of in-flight requests vs thread pool size. (4) Is a downstream dependency slow? Use distributed tracing to find the slowest component. (5) Is the cache working? Check cache hit rate. Present solutions in priority order: fix the biggest contributor first.

**Question: How do you handle latency spikes during traffic surges?**

Three-layer defense: (1) Autoscaling based on request queue depth or P99 latency — add servers before latency degrades past the threshold. (2) Rate limiting at the load balancer — accept only the maximum throughput the system can handle, return 429 for the rest. (3) Graceful degradation — disable non-critical features (recommendations, personalization) to free resources for core functionality. Always design for the spike, not the average.

### Troubleshooting Scenarios

These present a system that is behaving unexpectedly and ask you to diagnose the root cause.

**Scenario: A team deploys a new caching layer expecting latency to drop from 200ms to 50ms. After deployment, P50 is 180ms and P99 is 5 seconds. What went wrong?**

Possible causes: (1) The cache is cold — new deployment means empty cache, so every request is a miss, and the overhead of checking cache before going to the database adds latency. (2) Cache stampede — when the cache expires a popular key, all requests simultaneously hit the database, causing a throughput spike that overwhelms it. (3) Serialization overhead — the cached objects are large, and serialization/deserialization time negates the cache speed benefit. (4) Network latency — the cache is in a different availability zone than the application, adding network hop latency. Diagnosis: check cache hit rate, cache key expiry pattern, payload sizes, and network round trip time to the cache.

**Scenario: A batch processing pipeline that processes 1 million events per hour works fine during the day but at night, when traffic drops to 100,000 events per hour, latency per event increases 10x. Why?**

This is the classic throughput-latency inversion for batch systems. The batch pipeline has a minimum batch size threshold (e.g., 10,000 events) and a maximum batch timeout (e.g., 5 minutes). At 1 million events per hour (278 events/second), the batch fills in 36 seconds. At 100,000 events per hour (28 events/second), the batch takes 357 seconds to fill — and the timeout fires, processing a partial batch. But the per-event latency includes the waiting time. Solution: reduce the batch timeout or the minimum batch size for low-traffic periods. Or switch to a streaming architecture at low throughput and batch at high throughput.

**Scenario: A service exposes an API that returns data in under 50ms P99 during load testing. In production with 5,000 RPS, the P99 jumps to 1,200ms. Load testing showed the same throughput with stable latency. What changed?**

The most likely cause is the load testing traffic pattern. Load tests typically send requests at a constant rate from a few client machines. Production traffic has a Poisson arrival pattern with bursts and natural variability. At 5,000 RPS average, the peak may hit 8,000 RPS for 100ms bursts, exceeding the server's capacity during those bursts. The queueing delay spikes during the micro-bursts and takes time to drain. The fix: add a small buffer (queue depth of 100-200 requests) in front of the processing pipeline to absorb micro-bursts without dropping requests. Also ensure load testing uses realistic traffic patterns with Poisson arrivals, not constant-rate generators.

**Scenario: A team deploys their application in a second region (US-West) to reduce latency for West Coast users. After deployment, East Coast users see 20ms higher latency. P50 went from 40ms to 60ms for East Coast. What happened?**

The most likely cause is that the load balancer is now using geographic-based routing (GeoDNS), or the database primary was moved to US-West, or the application now has to synchronize state across regions. If the database primary moved to US-West, every write from East Coast incurs a 60ms round trip for synchronous replication. If a cross-region dependency was added for cache coherency or session sharing, every request now pays the cross-region latency cost. Solution: keep the database primary in US-East, use US-West for read-only traffic, and ensure session data is either replicated asynchronously or stored client-side (JWT tokens) to avoid cross-region lookups.

**Scenario: A real-time dashboard refreshes every 10 seconds with aggregated data. The dashboard queries run directly against the production database. After a feature launch that increased traffic 10x, the dashboard queries take 45 seconds instead of 2 seconds, and the production database CPU is at 95 percent. What is the root cause and the fix?**

Root cause: the dashboard queries are competing with production traffic for database resources. The dashboard's aggregation queries (full table scans, GROUP BY over millions of rows) consume CPU and I/O that production queries need, causing both to slow down. Fix: move dashboard queries to a read replica or a dedicated analytics database. If a read replica is not available, schedule dashboard refreshes during low-traffic periods or throttle the refresh frequency to every 60 seconds instead of 10 until a read replica is provisioned.

## Signals You Must Send

Interviewers evaluate your latency and throughput knowledge based on specific signals. These are the phrases and concepts that demonstrate depth.

### Core Signals

**"P50, P95, P99, P999"** — You do not talk about average latency. You talk about percentiles. You demonstrate that you know the difference between median and tail latency and when each matters.

**"Tail at scale"** — You mention Google's finding that fan-out multiplies tail probability. You can calculate: with 100 downstream calls each at 99 percent reliability, 36 percent of requests will experience a slow call.

**"Little's Law: L = lambda * W"** — You use Little's Law naturally in capacity planning. You can calculate server count: L = throughput * latency, then divide by connections per server.

**"Queueing delay grows as rho / (1 - rho)"** — You show that you know the M/M/1 queueing formula and can explain that latency is stable up to 60 percent utilization, then spikes exponentially after 80 percent.

**"Propagation delay is bounded by the speed of light"** — You demonstrate awareness that some latency is non-negotiable. You suggest deploying servers closer to users as the only solution for propagation delay.

### Advanced Signals

**"Head-of-line blocking at TCP level"** — You understand that HTTP/2 solves application-layer HOL blocking but TCP-level HOL blocking remains. You mention QUIC/HTTP/3 as the solution.

**"Hedged requests and request cancellation"** — You know Google's tail-at-scale mitigation techniques. You can explain how hedged requests convert tail probability from P_slow to P_slow^2.

**"Incast congestion"** — You understand that simultaneous responses from many servers can overwhelm a network buffer. You can describe solutions: staggering, reduced fan-out, pull-based consumption.

**"Connection pool sizing from Little's Law"** — You calculate connection pool size as throughput * latency, not guessing or using defaults.

**"Batch size optimization for the latency-throughput curve"** — You know that optimal batch size balances per-event latency constraints against throughput efficiency.

## Framework for Answering Latency/Throughput Questions

Use this four-step framework for any latency-throughput question in a system design interview:

### Step 1: Define the Metrics

Establish what you are optimizing. "What is the target latency — P50, P99, or P999? What is the target throughput in requests per second? Can we tolerate higher latency for higher throughput, or is latency the hard constraint?" The interviewer wants to see that you optimise based on requirements, not preferences.

### Step 2: Estimate the Numbers

Use Little's Law to estimate concurrent connections, queue depth, and server count. Sound comfortable with the math: "At 10,000 RPS with a 200ms latency target, Little's Law says we need 10,000 * 0.2 = 2,000 connections. Each server handles 200 concurrent connections, so we need at least 10 servers."

### Step 3: Identify the Bottleneck

Walk through each component and state its expected throughput and latency limit. "The database handles 5,000 QPS. At 10,000 RPS with a 10:1 read ratio, that is 9,090 reads/s to the database after caching — we need read replicas. Redis handles 100,000 ops/s — not a bottleneck. The app servers handle 500 concurrent connections each — at 2,000 needed, we need 4 app servers minimum."

### Step 4: Design for the Trade-off

State the trade-off explicitly. "I am choosing async replication for this service. This gives us 5ms write latency instead of 60ms with synchronous replication, which is critical for our 200ms P99 target. The trade-off is that we will lose the last few milliseconds of writes if the primary fails — this is acceptable because this data is cached anyway and can be regenerated."

## Red Flags

These responses signal shallow understanding to an interviewer:

**"We should use caching to fix the latency."** — Too vague. Which cache? What cache pattern (cache-aside, write-through, write-behind)? What size? What TTL? What consistency guarantee? An interviewer hearing "use caching" without specifics knows you do not understand caching trade-offs.

**"We need to optimize the database queries."** — Without first confirming that the database is the bottleneck, this is premature optimization. If the bottleneck is network bandwidth or CPU contention, query optimization does nothing. Always diagnose before prescribing.

**"The average latency is 50ms."** — Anyone who reports average latency without percentiles reveals that they do not measure correctly. The average hides the tail. A P50 of 5ms and a P99 of 2 seconds both contribute to the same average of 50ms at the right mix.

**"We can just add more servers."** — Horizontal scaling increases throughput but does not fix latency problems caused by slow code, poor database queries, or propagation delay. Adding servers to a latency problem wastes money and achieves nothing.

**"We need low latency and high throughput."** — Without qualification, this says you do not understand the trade-off. You must specify which metric is the priority and what you are willing to sacrifice. "We need P99 latency under 200ms for the user-facing search endpoint, and for the batch analytics pipeline, we prioritize throughput over latency."

**"Let's use microservices for better performance."** — Microservices increase latency (network hops between services) and reduce throughput (serialization/deserialization overhead). They are chosen for organizational scaling and independent deployability, not for latency or throughput.

**"The speed of light is the limit."** — Correct but overused. This is a thought-stopper. If an interviewer asks how to reduce cross-region latency and you say "the speed of light is the limit," you forgot data locality (move computation to the data), compression (reduce the data that travels), and prediction (pre-fetch data before the user requests it).

**"We have a 10ms latency target."** — Without specifying whether this is P50, P99, or P999, the statement is meaningless. A 10ms P50 with 500ms P99 is achievable on a single machine. A 10ms P99 at 10,000 RPS with multi-region users is a multi-million-dollar infrastructure problem.

**"The database is slow, so we need a faster database."** — Without first analyzing whether the database is actually the bottleneck, this wastes money. Throwing a faster database (or more expensive hardware) at a query that needs an index, or a workload that needs caching, or a volume that needs sharding, does not fix the problem. Diagnosis first, prescription second.

**"We should batch all our API calls."** — Batching everything reduces throughput overhead but increases latency for every call. User-facing calls must not be batched. Background pipeline calls should be batched aggressively. The distinction is critical.

**"Our system is fast because the CPU is only at 20 percent."** — Low CPU does not mean fast. An I/O-bound system can have 20 percent CPU and 5-second latency because every thread is waiting on a database query or network call. CPU utilization measures processor busyness, not system responsiveness.

**"Let's add a message queue to make it faster."** — Adding a message queue does not make anything faster. It decouples producers from consumers, which can improve throughput by letting each operate at its own speed, but it adds latency (the message must be written to the queue and read by the consumer). Queues improve throughput and resilience, not latency.

**"We need to optimize the P50."** — In most production systems, optimizing P50 does not improve user experience. A P50 of 10ms with a P99 of 5 seconds still gives 1 in 100 users a terrible experience. Optimize the tail. The median is vanity, the tail is sanity.

**"The system scales linearly."** — No system scales linearly past a certain point. Amdahl's Law guarantees diminishing returns. Coordination overhead, network contention, lock contention, and resource limits all introduce non-linearity. If an interviewer says their system scales linearly, they have not pushed it past 10 servers.

**"We don't need to measure latency; we can just estimate."** — You cannot estimate queueing delay, GC pauses, network jitter, or contention. You must measure. An unmeasured system is an uncontrolled system.

## Common Interview Mistakes

Beyond red flags, there are specific behavioral mistakes candidates make in latency-throughput questions.

**Mistake 1: Answering before understanding the scale.** An interviewer asks about latency optimization. The candidate immediately dives into query tuning before asking how many requests per second the system handles, what the current percentiles are, or what the hardware looks like. Always clarify scale before prescribing solutions.

**Mistake 2: Confusing latency with throughput in the answer.** A candidate says "we need to reduce latency, so let's add more servers." Adding servers does not reduce per-request latency unless queueing is the dominant component. The candidate should say: "Is the latency caused by queueing or by processing time? If queueing, more servers help. If processing time, we need to optimize the code or add caching."

**Mistake 3: Forgetting the cost dimension.** Every latency optimization has a cost. Caching costs memory. Hedged requests cost network bandwidth. Multi-region deployment costs infrastructure. A senior candidate demonstrates awareness of cost: "This optimization reduces P99 from 200ms to 100ms but doubles our infrastructure cost. Is the business willing to pay that for the latency improvement?"

**Mistake 4: Proposing the most complex solution first.** The candidate jumps straight to "let's implement hedged requests with request cancellation and speculative execution" when the problem is simply a missing database index. Start with the simplest fix that could work. Complex solutions have complex failure modes.

## Mock Interview Transcript

The following mock transcript shows a strong answer to a latency-throughput question. Study the structure and language.

```
Interviewer: "Design the news feed for a social network with 100 million
daily active users. The feed must load in under 500ms."

Candidate: "Let me start by understanding the requirements. What is the
latency target — P50, P99, or P999? And what is the expected throughput
in feed loads per second at peak?"

Interviewer: "P99 under 500ms. Peak throughput is 100,000 feed loads
per second."

Candidate: "Good. My first calculation: at 100,000 RPS with a 500ms P99
target, Little's Law says we have 100,000 * 0.5 = 50,000 concurrent feed
loads in flight. Each feed load requires fetching posts from followed
accounts. With an average of 300 follows per user, a naive approach of
querying each followed account's recent posts would cost 300 database
queries per feed load — 30 million queries per second. That is not
feasible.

The bottleneck is clearly database query throughput. Even with caching,
300 queries per feed load at 100,000 loads per second means the fan-out
to storage is too large.

My approach is fanout-on-write with pre-computed feeds:

When a user creates a post:
1. Write the post to a posts table (single write, 5ms)
2. Fetch the user's followers from a graph database or cache
3. For each follower: prepend the post ID to their feed cache in Redis
4. Trim each follower's cache to 500 post IDs
5. If a user has more than 10,000 followers: switch to fanout-on-read
   (store the post in a 'recent posts by follows' list, merge on load)

When a user loads their feed:
1. Read the pre-computed feed list from Redis (key: feed:{user_id})
2. Fetch the actual post data from a content cache (mget the post IDs)
3. Return the top 20 posts

Now let me estimate the numbers:
- Write path: each post creates N writes (one per follower).
  At 10 million posts per day with 300 average followers:
  10M * 300 = 3 billion fanout writes/day = 35,000 writes/second
  Redis handles 100,000 ops/s per node. With 3-5 shards, this is fine.
- Read path: cache hit serves feed in under 5ms (Redis mget).
  The P99 here is almost entirely determined by cache hit rate.
  At 99% hit rate: P99 ~ 10ms. At 95% hit rate: P99 ~ 50ms.
  The remaining 1-5% of users who are cache misses (new users, users
  who haven't loaded feed in days) must compute their feed from scratch.
  For these users, latency is 100-300ms (query follows' recent posts).

The trade-off: fanout-on-write increases write cost (35K writes/s instead
of 10K writes/s) but reduces read latency from potentially seconds to
under 10ms. For a social network with a 10:1 read-to-write ratio,
paying more on writes to save on reads is the correct trade-off.

To meet the 500ms P99 target:
- Main read path: 10ms (Redis cache hit)
- Cache miss path: 200ms (fallback computation)
- P99 at 99% hit rate: definitely under 100ms
- This architecture comfortably meets the 500ms P99 target."

Interviewer: "Good. What happens if Redis fails?"

Candidate: "On Redis failure, the fallback is a direct read from the
database. We compute the feed from scratch by querying the user's
followed accounts' recent posts from PostgreSQL. This is slow (200-500ms)
but functional. We also need: (1) circuit breaker on Redis so the app
doesn't wait for Redis timeouts, (2) gradual reconnection with jitter
to avoid thundering herd when Redis recovers, (3) a warm-up period after
Redis recovery where cache miss rates are temporarily high."
```

## Level Expectations

Your answer depth should match your experience level.

### Junior (0-2 years)

Define latency and throughput correctly. Know the difference between P50 and P99. Use Little's Law for basic capacity calculation. Identify obvious bottlenecks (database, CPU, memory). Propose caching as a solution. Mention connection pooling.

Expected answer quality: "Latency is the time for one request to complete. Throughput is how many requests per second. We measure both in percentiles, not averages. We can use caching to improve both — Redis can handle 100,000 ops/s. For capacity, Little's Law tells us how many concurrent connections we need."

### Mid-Level (2-5 years)

Explain queueing theory and the utilization curve. Use percentiles naturally in reasoning. Identify non-obvious bottlenecks (connection pool exhaustion, TLS handshake overhead, head-of-line blocking). Propose specific cache architectures (cache-aside, write-through). Use Little's Law for precise capacity planning. Discuss trade-offs between synchronous and async replication.

Expected answer quality: "For this read-heavy API with a 10:1 read-to-write ratio, I would use cache-aside with Redis. Expected hit rate is 95 percent, which cuts database QPS from 10,000 to 500. This is important because queueing theory tells us that at 95 percent utilization, the database would have 19x queueing delay. With caching, the database runs at 30 percent utilization, where queueing delay is negligible."

### Senior (5+ years)

Discuss tail latency amplification and hedged requests. Design multi-region latency strategies considering the CAP theorem. Use advanced queueing models (M/M/c, Erlang-C). Design around head-of-line blocking at multiple protocol levels. Propose adaptive concurrency limiting based on latency feedback. Discuss batching strategies that optimize the latency-throughput trade-off mathematically.

Expected answer quality: "The biggest challenge here is tail latency at scale. With 500 downstream calls per request, the chance of hitting a slow server is essentially 100 percent. I would implement hedged requests with a 5ms delay before the hedge fires. I would also add request cancellation so that when one hedged response arrives, we cancel the other. This doubles our outbound request throughput but cuts P99 latency by 4x. The throughput overhead is acceptable because our bottleneck is the database, not the network or app server."

### Staff+ (8+ years)

Design custom latency optimization infrastructure. Propose novel trade-off strategies specific to the business domain. Discuss the organizational implications of latency budgets (team ownership, SLA enforcement, cost of latency vs cost of infrastructure). Reference specific academic papers (The Tail at Scale, Taming Latency with Jitter). Design the monitoring and alerting system that catches latency regressions.

Expected answer quality covers not just what but why and how the organization would implement it: "I would implement a latency budget system where each service publishes its P99 SLA and each consuming service tracks whether the SLA is met. When a service exceeds its budget, we page the owning team within 5 minutes. This is the only way to maintain sub-200ms P99 at 100,000+ RPS — without organizational accountability, latency creeps up as every team optimizes for their own throughput."

## Practice Drills

### Drill 1: The 5-Second Estimation

Given a throughput number and a latency target, estimate the number of servers needed in under 5 seconds. Practice: 50,000 RPS at 150ms P99. Little's Law: L = 50,000 * 0.15 = 7,500 concurrent connections. At 500 connections per server: 7,500 / 500 = 15 servers. Add 30 percent headroom: 20 servers. The skill is doing this without a calculator.

### Drill 2: The Bottleneck Walkthrough

For a given system diagram, identify the bottleneck in 30 seconds. Practice on any architecture you encounter: web server → application → cache → database → queue. Each component has known limits. The bottleneck is the component with the lowest capacity relative to the required throughput.

### Drill 3: The Trade-off Statement

For any latency-throughput decision, state the trade-off in one sentence: "We are choosing X over Y because Z matters more." Examples: "We are choosing async replication over synchronous because write latency matters more than strict consistency for this user preference data." "We are choosing batching over streaming because throughput efficiency matters more than per-event latency for this analytics pipeline."

### Drill 5: The Formula Recall

Quiz yourself on these formulas until they are automatic:

- M/M/1 utilization: rho = lambda / mu
- M/M/1 queueing delay: Wq = rho / (mu - lambda)
- Little's Law: L = lambda * W
- Batch size optimal: buffer_time * throughput / 2
- Connection pool: throughput * latency (Little's Law)
- P99 with hedged requests: P99_hedged = P99_single^2
- Amdahl's Law speedup: S = 1 / ((1 - P) + P/N)

### Drill 6: The Design Defense

Take any system you have worked on and prepare a 2-minute explanation of its latency and throughput characteristics. Structure: (1) What is the throughput in RPS at peak? (2) What is the P50, P99, and P999 latency? (3) What is the bottleneck? (4) What would you optimize next? (5) What cutout did you make? Practice this until the numbers come naturally.

### Drill 7: The Cost Calculation

For any latency optimization, calculate the cost in dollars: A cache cluster with 9 Redis nodes at $500/month each costs $4,500/month. Does it reduce latency enough to justify the cost? At Amazon's estimate of 1 percent revenue per 100ms latency, a site doing $10M/month in revenue would gain $100,000/month by reducing latency by 100ms. The cache cluster at $4,500/month pays for itself 22x over. This is how you justify infrastructure investments to product managers.

## Sample Interview Walkthroughs

### Walkthrough 1: Design a Rate Limiter

This is a common system design question where the latency-throughput trade-off is central.

```
Interviewer: "Design a rate limiter that handles 1 million requests per
second with P99 latency under 5ms."

Candidate: "Let me define the metrics. The key metrics here are latency
(P99 under 5ms) and throughput (1M RPS). The rate limiter is in the
critical path of every request, so it cannot add significant latency.

My approach is a token bucket algorithm using Redis as the backing store.
Redis handles 100,000-200,000 ops/s per node. At 1M RPS, we need
10 Redis nodes sharded by client ID (consistent hashing).

Latency budget:
- Network to Redis: 0.5ms (same datacenter)
- Redis command execution: 0.1ms (EVALSHA for token bucket Lua script)
- Network response: 0.5ms
- Total: 1.1ms per check
- Headroom: 3.9ms under the 5ms P99 target

The trade-off: using Redis adds a network hop (1ms) but keeps the
algorithm simple and the data consistent. An alternative is a local
in-memory counter (0ms network, under 0.01ms latency) but then
rate limits are per-server, not global. If a user spreads requests
across 10 servers, each server allows 100 RPS for a 1,000 RPS
global limit — the user can send 10,000 RPS total.

For a rate limiter, global consistency matters more than 1ms of
latency. Redis-backed token bucket is the right choice."

What the candidate demonstrated:
1. Specified metrics upfront (P99 5ms, 1M RPS)
2. Calculated capacity (10 Redis shards)
3. Quantified latency budget (1.1ms vs 5ms target)
4. Stated the trade-off explicitly (Redis vs local)
5. Defended the choice with reasoning
```

### Walkthrough 2: Diagnose a Production Incident

This tests troubleshooting ability under pressure.

```
Interviewer: "At 3 PM, our checkout service P99 latency jumped from
200ms to 3 seconds. It stayed there for 15 minutes, then recovered.
What do you check first?"

Candidate: "I check three things immediately:

1. The latency breakdown by component. Is it the checkout service
   itself, or a downstream dependency? If the database P99 also
   spiked, the root cause is in the database. If only checkout
   spiked, it is in the application logic or an upstream caller.

2. The error rate during the spike. Did errors increase? If errors
   increased, the service may be spending CPU on exception handling
   and retries instead of normal processing. If errors decreased,
   the service may be queueing requests without failing fast.

3. The deployment and config change timeline. Was anything deployed
   at 2:55 PM? Any config change? Any database migration? Any
   traffic reroute? The most common cause of latency spikes is a
   recent change.

Given that it recovered in 15 minutes, I suspect one of:
a) A downstream service (payment gateway, shipping API) had a
    transient slowdown that caused queue buildup in checkout.
b) A garbage collection cycle in the JVM that took 15 minutes
    for the heap to return to normal (unlikely — GC usually
    recovers faster).
c) A cron job or batch process that ran at 3 PM and consumed
    database CPU, starving checkout queries.
d) A traffic surge (marketing campaign, email blast) that
    saturated the connection pool, causing queueing.

The 15-minute recovery time suggests (c) or (d) — a recurring
batch job or a burst that took that long to drain from the queue."

What the candidate demonstrated:
1. Systematic diagnosis (three checks in order)
2. Specific hypotheses with reasoning
3. Connection between symptoms and root cause
4. Considers the recovery pattern as a diagnostic clue
```

### Walkthrough 3: Choose Between Two Architectures

This tests your ability to compare approaches quantitatively.

```
Interviewer: "We need to process 10,000 financial transactions per
second. Option A uses synchronous processing with a single database:
each transaction is written immediately, P99 write latency is 50ms.
Option B uses a queue with async processing: transactions are queued
and written in batches of 100 every 200ms. P99 write latency for the
user is 5ms (just the queue write), but the database write happens
200ms later. Which do you choose?"

Candidate: "The core distinction is what the user waits for.

Option A (50ms latency) is better if the user needs to see the
transaction result immediately — a stock trade confirmation, a
bank transfer receipt, a payment success page. The 50ms P99 is
excellent for user-facing transactions.

Option B (5ms user latency + 200ms async write) is better if the
user only needs an acknowledgment and the actual processing can
happen offline — a donation receipt emailed later, a report
generation request, a non-critical settings update.

For financial transactions, there is a third dimension: consistency
and durability. Option A guarantees the write is committed before
the user sees a response. Option B risks queue loss (if the queue
fails before the batch writes) or duplicate writes (if the queue
replays after a failure).

My recommendation: Option A for the 10,000 TPS core transactions
with a sharded database. At 10,000 TPS with 50ms latency, Little's
Law says L = 10,000 * 0.05 = 500 in-flight transactions. With 4
database shards handling 2,500 TPS each, each shard has L = 125
transactions. A single PostgreSQL instance handles 5,000+ QPS,
so 2,500 TPS per shard is well within capacity. This gives us
50ms P99 with full durability.

Use Option B only for non-critical side effects (notification,
audit logging, analytics aggregation) that can tolerate 200ms
delay and potential loss."
```

## Preparation Checklist

Use this checklist before your system design interview to ensure you have covered all latency-throughput topics.

### Concepts to Review

- [ ] Latency vs throughput definition (can explain in 30 seconds)
- [ ] P50, P95, P99, P999 — what each means and when to use each
- [ ] Queueing theory fundamentals (M/M/1 formula, utilization)
- [ ] Little's Law (L = lambda * W) — can calculate server count from RPS and latency
- [ ] Tail latency and why it matters at scale
- [ ] Head-of-line blocking (HTTP/1.1 vs HTTP/2 vs QUIC)
- [ ] Amdahl's Law and parallel speedup limits
- [ ] I/O latency hierarchy (L1 cache to WAN RTT)

### Numbers to Memorize

- [ ] Redis throughput: 100,000-200,000 ops/s per node
- [ ] PostgreSQL QPS: 5,000-10,000 simple queries, 1,000-3,000 complex
- [ ] Intra-datacenter latency: 0.5ms
- [ ] Inter-region latency (US cross-country): 60-80ms RTT
- [ ] Cross-continent latency (US-East to EU): 80-120ms RTT
- [ ] TLS handshake: 30-100ms
- [ ] Cache hit (Redis): 0.5-2ms
- [ ] Database query (indexed): 2-15ms
- [ ] Disk sequential read (NVMe): 3-7 GB/s
- [ ] Course of a context switch: 1-10 microseconds

### Phrases to Practice

- [ ] "P99 latency under Xms at Y RPS..."
- [ ] "Little's Law tells us we need Z concurrent connections..."
- [ ] "The bottleneck is the component at 100% utilization..."
- [ ] "Queueing delay grows as rho divided by (1 minus rho)..."
- [ ] "We are choosing X over Y because Z matters more for this workload."
- [ ] "The speed of light bounds propagation delay; we must deploy closer to users."
- [ ] "Tail latency amplification: with N downstream calls, P(slow) = 1 - (1 - P)^N."
- [ ] "This system is I/O-bound, so parallelism increases throughput without affecting latency."

### Common Traps to Avoid

- [ ] Do not say "average latency" — always use percentiles
- [ ] Do not propose "use caching" without specifying which cache and what pattern
- [ ] Do not say "add more servers" to fix latency (only fixes queueing delay)
- [ ] Do not confuse bandwidth with throughput
- [ ] Do not forget the cost dimension of every optimization
- [ ] Do not ignore the physical limits (speed of light, I/O latency hierarchy)
- [ ] Do not propose batch processing for user-facing operations
- [ ] Do not claim linear scalability (Amdahl's Law applies everywhere)
