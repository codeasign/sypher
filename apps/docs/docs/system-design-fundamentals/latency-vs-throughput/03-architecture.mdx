## From Naive to Production

A naive system treats latency and throughput as independent concerns. A production system understands that every component in the request path affects both, and the architecture must be designed to manage the tension between them.

### Naive Architecture — Single Server, Everything Coupled

The simplest possible deployment: one server running the application, database, and cache on the same machine. This is the architecture of every prototype and MVP.

The naive architecture has excellent latency (no network hops between components) but severely limited throughput (everything competes for the same CPU, memory, and disk). At this stage, latency is low but throughput capacity is the constraint. The transition to production begins when throughput demand exceeds a single machine's capacity.

### Production Architecture — Horizontally Scaled, Decoupled

Once throughput exceeds a single machine, you must scale horizontally. The key principle: separate components onto independent machines so each can scale independently based on its own bottleneck.

The transition from naive to production architecture introduces a deliberate latency increase. Each network hop between components adds 0.5-5ms. A request that hits cache, then database, then writes to a queue traverses 3-5 network hops, adding 5-25ms compared to the localhost calls in the naive architecture. This is the latency cost of scaling — and it is almost always worth paying.

## Integration Patterns — DB Latency

Databases are the most common latency bottleneck. Every production system must design around database latency.

### Read Replicas for Query Throughput

Read replicas increase read throughput without increasing write latency. The primary handles writes and propagates changes to replicas asynchronously.

```
# Pattern: Read replicas for read-heavy workloads
# 10:1 read-to-write ratio
#
# Without read replicas:
#   Primary handles 1,000 writes/s + 10,000 reads/s = 11,000 QPS
#   Primary at 90% CPU → both read and write latency spike to 200ms+
#
# With 3 read replicas:
#   Primary handles 1,000 writes/s + cache misses = 1,500 QPS (15% CPU)
#   Replica 1: 3,333 reads/s (30% CPU)
#   Replica 2: 3,333 reads/s (30% CPU)
#   Replica 3: 3,333 reads/s (30% CPU)
#
# Read latency: P50 2ms, P99 10ms (no queueing)
# Write latency: P50 5ms, P99 20ms (no queueing)
# Total throughput: 11,000 QPS successfully, room to grow
```

The critical detail: read replicas work only for eventually consistent reads. If your application requires read-after-write consistency (the user just created a resource and immediately reads it), you must route that read to the primary or accept that the replica may not have the latest data.

### Connection Pooling to Avoid Connection Storm

Each new database connection consumes 5-30ms and 2-10MB of memory. Under high throughput, creating a new connection for every request causes connection storms that crash the database.

```
# Pattern: Centralized connection pool (PgBouncer)
#
# Without connection pooling:
#   Each request creates a new DB connection → 10ms overhead × 1,000 req/s
#   = 10 seconds of connection overhead per second (impossible)
#   Database runs out of connections at 100 concurrent requests
#
# With PgBouncer pool:
#   50 persistent connections to database
#   Requests borrow and return connections in under 1ms
#   0 connection overhead per request
#   Database sees a steady 50 connections, not bursty spikes
#
# PgBouncer configuration:
# pool_mode = transaction
#   (connection stays open only for one transaction, not the whole session)
# default_pool_size = 50
# reserve_pool_size = 5
# reserve_pool_timeout = 3.0
# max_client_conn = 500

# Connection pool sizing example (Node.js with pg-pool)
const { Pool } = require('pg');
const pool = new Pool({
    host: 'db.example.com',
    port: 5432,
    database: 'appdb',
    user: 'appuser',
    password: process.env.DB_PASSWORD,
    max: 50,                    // Max connections in pool
    idleTimeoutMillis: 30000,   // Close idle connections after 30s
    connectionTimeoutMillis: 5000,  // Fail if can't connect in 5s
    maxUses: 7500               // Recycle connection after 7,500 uses
});
```

### Query Optimization for Latency

The most effective latency optimization for databases is query optimization — specifically, index usage.

```
# Bad query: full table scan (latency proportional to table size)
SELECT * FROM orders WHERE customer_id = 12345;
# Without index: scans all 10M rows → 3-30 seconds

# Good query: index scan (latency independent of table size)
# With CREATE INDEX idx_orders_customer ON orders(customer_id);
# Index scan: log₂(10M) ≈ 23 lookups → 2-5ms
# 1,000x latency improvement from a single index

# Covering index: eliminates data page lookups entirely
CREATE INDEX idx_orders_customer_covering
ON orders(customer_id) INCLUDE (order_date, total, status);
# All needed data is in the index → no data page access → 0.5-1ms
# Another 5x improvement over non-covering index scan

# Tip: monitor slow queries with auto_explain
# PostgreSQL: LOAD 'auto_explain';
# SET auto_explain.log_min_duration = '100ms';
# SET auto_explain.log_analyze = true;
```

## Integration Patterns — Cache Throughput

Caches improve both latency and throughput — simultaneously — which makes them the most powerful performance tool in system design. A cache hit replaces a 5-50ms database query with a 0.5-2ms memory lookup.

### Cache Cluster Sizing

```
# Cache cluster size calculation
# Data volume: 500 GB active dataset
# Cache ratio: 80% of data accessed regularly
# Cache size needed: 500 GB × 0.8 = 400 GB
# Redis node size: 64 GB per node
# Nodes needed (with 25% headroom): 400 GB / (64 GB × 0.75) = 9 nodes
#
# Throughput calculation
# Request rate: 100,000 reads/s
# Redis per-node throughput: 150,000 ops/s
# Nodes needed for throughput: 100,000 / 150,000 = 1 node
#
# Winner: 9 nodes (storage-bound, not throughput-bound)
# Final design: Redis cluster with 9 shards, 3 replicas each
# Memory per node: 64 GB
# Total memory: 9 × 64 GB = 576 GB (with 27% headroom)
```

## Integration Patterns — Queue Throughput

Message queues decouple producers from consumers, allowing each to operate at its own speed. A queue acts as a throughput buffer — when producers spike, the queue absorbs the excess, and consumers process at their own pace.

```
# Queue throughput patterns
#
# Without queue: synchronous processing
# Request → Write to DB → Process image → Send email
# Total latency: 3 seconds (image processing + email)
# User waits 3 seconds for response
#
# With queue: asynchronous processing
# Request → Write to DB → Emit to queue → Response in 100ms
# Worker picks up message from queue → Process image → Send email
# User gets response in 100ms (1/30th the latency)
# Worker processes at own pace (may take 5 seconds)
#
# Throughput benefit:
# Synchronous: max 20 requests/minute (limited by image processing)
# Asynchronous: max 10,000 requests/minute (limited by request handling)
# Workers scale independently: add more workers to consume faster
```

### Multi-Region Write Latency

Writes in a multi-region architecture face a fundamental choice: synchronous cross-region replication (strong consistency, high write latency) or asynchronous replication (eventual consistency, low write latency).

```
# Multi-region write latency options
#
# Option A: Synchronous cross-region replication
# User in Tokyo writes to Tokyo region
# Tokyo primary must replicate to Virginia primary before acknowledging
# RTT Tokyo to Virginia: 200ms
# Write latency: 5ms (local write) + 200ms (ack from Virginia) = 205ms
#
# Option B: Asynchronous replication (active-passive)
# User in Tokyo writes to US-East primary (same location as primary DB)
# US-East primary replicates asynchronously to Tokyo replica
# Write latency: 60ms (Tokyo to US-East) + 5ms (write) = 65ms
# Risk: if US-East fails before replication, data loss
#
# Option C: Multi-primary with conflict resolution
# Each region has a primary that accepts writes
# Changes replicated asynchronously between regions
# Conflicts resolved by last-writer-wins or CRDTs
# Write latency: 5ms (local write, no cross-region wait)
# Risk: conflicts during concurrent edits from different regions
#
# Recommendation:
# Most applications use Option B (async active-passive) for critical data
# and Option C (multi-primary) for data that tolerates conflicts
# (shopping cart items, user preferences, session data)
```

## Capacity Planning with Little's Law

Little's Law is the foundation of capacity planning. Given target throughput and acceptable latency, it tells you exactly how many concurrent resources you need.

```
# Capacity planning example: Ride-sharing dispatch service
#
# Requirements:
#   Peak throughput: 10,000 ride requests per second
#   Acceptable latency: 200ms P99 (including queueing)
#   Service time per request: 50ms (pure processing, no queueing)
#
# Step 1: Calculate average requests in system
# L = λ × W = 10,000 × 0.2 = 2,000 requests in flight at any time
#
# Step 2: Calculate concurrent processing capacity needed
# Each request takes 50ms processing time
# One server can handle 1/0.05 = 20 concurrent requests at a time
# Wait: 50ms processing for 200ms target means 150ms queueing is okay
#
# Step 3: Utilization calculation
# Target utilization: 70% (leaves headroom for spikes)
# Each server handles 20 × 0.7 = 14 concurrent requests (effective)
# Servers needed: 2,000 / 14 = 143 servers
#
# Step 4: Validate with queueing math
# λ = 10,000 req/s distributed across 143 servers
# λ per server = 10,000 / 143 = 70 req/s
# Service rate per server = 1/0.05 = 20 req/s
# Wait — 70 req/s per server exceeds 20 req/s capacity!
# This means utilization > 100% — impossible
#
# Correction: each server processes requests in PARALLEL
# Server with 32 threads can process 32 requests simultaneously
# Each server throughput at 50ms = 32 / 0.05 = 640 req/s
# At 70% utilization: 640 × 0.7 = 448 effective req/s per server
# Servers needed: 10,000 / 448 = 23 servers
#
# Step 5: Queueing check
# M/M/c model with c = 23 servers, ρ = 0.7
# P(no queueing) = 0.3 (30% chance request is served immediately)
# Average queueing delay: approximately 15ms
# Total latency: 50ms + 15ms = 65ms P50
# P99 total latency (with queueing): approximately 150ms
# Well within 200ms target
```

The most common capacity planning mistake: assuming linear scaling. Doubling servers does not double throughput because of coordination overhead, network contention, and increased queueing probability. Always add 20-30 percent headroom to your server count estimate.

## Deployment — Latency Optimization Strategies

### Blue-Green Deployment for Zero-Downtime

Blue-green deployment eliminates deployment-related latency spikes by running two identical environments and switching traffic atomically.

```
# Blue-green deployment strategy
#
# Phase 1: Steady state
#   Blue environment: production traffic, 100%
#   Green environment: idle, running same version
#
# Phase 2: Deploy new version to green
#   Blue continues serving traffic
#   Green deploys new code, runs migrations, warms caches
#   No user-facing latency impact
#
# Phase 3: Warm the green environment
#   Send 1% of traffic to green for 5 minutes
#   Monitor latency, error rate, CPU utilization
#   If metrics match blue → proceed
#   If metrics degrade → rollback (shift remaining 99% back to blue)
#
# Phase 4: Switch traffic
#   Load balancer switches 100% traffic to green
#   Cutover is instantaneous (DNS change or LB config update)
#   No downtime, no connection drops (connections drain naturally)
#
# Phase 5: Monitor and clean up
#   Blue remains as rollback target for 24 hours
#   After 24 hours, blue becomes next deployment target
```

The critical latency consideration: cold caches. When the green environment starts fresh, its caches are empty. The first requests to green will be cache misses, causing 5-10x higher latency until caches warm up. The fix: pre-warm caches by sending 1-5 percent of traffic for 5-15 minutes before the full switch.

### Connection Draining for Graceful Shutdowns

Connection draining prevents latency spikes during deployments by allowing in-flight requests to complete before terminating a server.

```
# Connection draining configuration (AWS ALB)
#
# Deregistration delay: 300 seconds
#   After a target instance is deregistered (marked unhealthy),
#   the load balancer waits this long to allow in-flight requests
#   to complete before closing connections.
#
# Health check interval: 30 seconds
#   Load balancer checks health every 30 seconds.
#   Unhealthy threshold: 2 consecutive failures
#   Time to detect unhealthy instance: 60 seconds (2 × 30s)
#
# Connection draining sequence:
# 1. Instance enters "draining" state (no new connections)
# 2. Existing connections continue for up to 300 seconds
# 3. After all connections complete → instance is terminated
# 4. New connections go to remaining healthy instances
#
# Without draining:
#   Instance terminates mid-request → 5% of requests fail
#   Failed requests retry → 5% throughput spike on remaining servers
#   Retry latency adds 200-500ms to affected user requests
```

### Autoscaling Based on Latency Metrics

Autoscaling based on CPU utilization reacts too late for latency-sensitive systems. CPU spikes when the system is already overloaded. The better approach is to scale based on request queue depth or P99 latency.

```
# Autoscaling policies comparison
#
# CPU-based scaling (slow, reactive):
#   Scale up when: CPU > 70% for 5 minutes
#   Scale down when: CPU < 40% for 15 minutes
#   Problem: by the time CPU hits 70%, latency may have tripled
#   Problem: takes 5 minutes to add a new server = 5 minutes of bad latency
#
# Request-based scaling (fast, predictive):
#   Scale up when: requests per second > 500 per server for 2 minutes
#   Scale down when: requests per second under 200 per server for 10 minutes
#   Advantage: RPS spikes before CPU does — more lead time
#   Advantage: directly tied to throughput demand
#
# Latency-based scaling (responsive, user-facing):
#   Scale up when: P99 latency > 200ms for 1 minute
#   Scale down when: P99 latency under 100ms for 15 minutes
#   Advantage: directly measures user experience
#   Disadvantage: scales in response to problems, not before them
#
# Best practice: combine request-based (predictive) with
# latency-based (reactive) scaling policies
#
# AWS Auto Scaling configuration (example)
# Target tracking policy:
#   Metric: ALBRequestCountPerTarget
#   Target value: 500 requests per minute per target
#   Scale-out cooldown: 60 seconds
#   Scale-in cooldown: 300 seconds
```

### Pre-warming Connections and Pools

Cold starts — when a new server joins the pool with empty caches and no established connections — cause latency spikes for the first requests it handles.

```
# Pre-warming strategies for new server instances
#
# 1. Connection pool pre-warming
#    Before accepting traffic, open database connections
#    Connect to Redis, establish connection pool entries
#    Time: 5-10 seconds (30 connections × 200ms per connection)
#
# 2. Cache pre-warming
#    Load most-frequently-accessed keys into local cache
#    Or: DNS redirect to shift traffic gradually (warm via real traffic)
#    Time: 30-300 seconds (depends on cache size)
#
# 3. JVM warm-up (Java applications)
#    JIT compilation happens during first execution of each code path
#    Send synthetic traffic to trigger JIT before real traffic arrives
#    Time: 30-60 seconds (JIT warmup completes)
#
# 4. TLS session cache pre-warming
#    New server has no cached TLS sessions
#    First 100+ clients experience full handshake latency (30-100ms)
#    Solution: pre-generate session tickets or use shared session cache
#
# Implementation: health check endpoint
# GET /health
# Response: { "status": "ready", "warmed": true }
# Load balancer only sends traffic when health check returns "ready"
# Health check endpoint blocks until pre-warming completes
```

## Service Level Objectives for Latency

Production systems need explicit SLOs for latency that are tracked, monitored, and used to drive engineering decisions.

### Defining Latency SLOs

A latency SLO specifies the acceptable latency for a given percentile over a measurement window. It is the contract between the service provider and the service consumer.

```
# Latency SLO specification
#
# Endpoint: GET /api/search
# SLO: P99 latency under 200ms over any 5-minute rolling window
# Measurement: server-side request duration (excluding client network)
# Exclusions: scheduled maintenance windows, known slow queries
# 
# Error budget: 99.9% reliability → 0.1% of 5-minute windows can exceed SLO
# Monthly error budget: 0.1% × 43,200 windows = 43 windows can exceed
# If exceeded in 44+ windows, the team freezes feature releases
# until latency is fixed

# SLO waterfall — each downstream service contributes to the total
# User-facing API (200ms P99)
#   ├─ Auth Service (50ms P99)
#   │   └─ Redis (5ms P99)
#   ├─ Search Service (100ms P99)
#   │   ├─ Cache (5ms P99)
#   │   └─ Index (80ms P99)
#   └─ Recommendation Service (80ms P99) [called in parallel]
#       └─ ML model (70ms P99)
# 
# Total budget: 200ms (parallel paths use max, not sum)
# Auth + max(Search, Rec) = 50ms + max(100ms, 80ms) = 150ms
# Remaining budget: 50ms for serialization, queueing, network overhead
```

### Error Budgets in Practice

The error budget concept (from Google SRE) converts latency targets into a concrete operational limit. The team can release features as long as the error budget has not been exhausted. When the budget runs out, development stops and reliability work begins.

```
# Monthly error budget calculation for a latency SLO
#
# SLO: P99 latency under 200ms, 99.9% of the time
# Total 5-minute windows per month: 43,200
# Allowed failures: 43,200 × 0.001 = 43 windows
#
# If P99 latency exceeds 200ms in 44+ windows:
#   → Feature releases frozen
#   → Team works on latency improvements
#   → Releases resume when budget recovers (next month)
#
# Real-world example: the team notices that every Tuesday at 2 PM,
# P99 spikes to 500ms for 3 minutes. That is 0.6 windows per week.
# Over a month: 2.4 windows consumed. 43 - 2.4 = 40.6 remaining.
# This is fine — the error budget has room for occasional spikes.
#
# But if a new feature adds 50ms to every request, and the original
# P99 was 180ms, the new P99 is 230ms. This exceeds the 200ms SLO
# continuously — 100% of windows burn budget. The feature would be
# blocked until the team reduces latency elsewhere.
```

### Latency SLOs in Multi-Service Architectures

In a microservice architecture, each service has its own latency SLO, and the total user-facing latency is the combination of all services on the critical path.

```
# Multi-service latency SLO budget
#
# User-facing checkout flow: target P99 under 500ms
#
# Service budget allocation:
#   API Gateway:     50ms P99 (routing, auth, rate limiting)
#   Cart Service:    60ms P99 (validate cart contents)
#   Pricing Service: 100ms P99 (calculate totals with discounts)
#   Inventory:       80ms P99 (check stock availability)
#   Order Service:   100ms P99 (create order record)
#   Payment:         250ms P99 (process with payment gateway)
#   Notification:    20ms P99 (send confirmation, async, main path)
#
# Critical path (sequential): API Gateway + Cart + Pricing + Order = 310ms
# Parallel: Inventory (runs with Pricing) + Payment (runs after Pricing)
# Payment max: 250ms → runs after Pricing (100ms) → 350ms from start
# Order runs after Payment: 100ms → total 450ms from start
# Notification: async, not on critical path → 0ms
# Total: 450ms ✓ (under 500ms target with 50ms headroom)
#
# If any service exceeds its budget by 10ms, the total exceeds 500ms.
# This is why latency budgets must be enforced per-service, not just
# measured at the user-facing endpoint.
```
