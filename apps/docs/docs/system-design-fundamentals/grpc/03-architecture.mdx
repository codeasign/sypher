# gRPC at Scale

## Naive Implementation

The simplest gRPC deployment looks like this:

```python
# server.py
import grpc
from concurrent import futures
import user_pb2_grpc, user_pb2

class UserService(user_pb2_grpc.UserServiceServicer):
  def GetUser(self, request, context):
  user = query_db(request.user_id)
  return user_pb2.GetUserResponse(**user)

server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
user_pb2_grpc.add_UserServiceServicer_to_server(UserService(), server)
server.add_insecure_port("[::]:50051")
server.start()
server.wait_for_termination()
```

```python
# client.py
import grpc
import user_pb2_grpc

channel = grpc.insecure_channel("localhost:50051")
stub = user_pb2_grpc.UserServiceStub(channel)
response = stub.GetUser(user_pb2.GetUserRequest(user_id="123"))
```

**At 1,000 users**: This works perfectly. The single server handles 50 RPCs/second with 5ms p50 latency.

**At 100,000 users**: The server processes 5,000 RPCs/second. The 10-thread thread pool becomes saturated. Requests queue in the gRPC server's internal work queue. p50 latency climbs to 200ms. p99 hits 2 seconds. Some requests time out.

**At 10,000,000 users**: The naive approach fails catastrophically:
- Single server cannot handle 500,000 RPCs/second — f--?s--, CPU is 100% on protobuf serialization alone.
- No load balancing: all traffic hits one process.
- No health checking: if the server crashes, clients get UNAVAILABLE errors for every call.
- No service discovery: IP addresses are hardcoded. Adding a server instance requires a redeploy.
- No TLS: `insecure_channel` sends unencrypted data over the wire.
- No deadlines: hung requests consume server threads indefinitely, eventually freezing the server.
- No retry policy: a transient network blip causes 5,000 failed requests.

### Key Components

**Service Discovery**: gRPC clients must find available server instances. Use a service registry like etcd, Consul, or Kubernetes DNS. The gRPC resolver periodically queries the registry and updates the channel's endpoint list.

```yaml
# Kubernetes headless service — f--?s--, DNS-based service discovery
apiVersion: v1
kind: Service
metadata:
  name: user-service-grpc
spec:
  clusterIP: None  # Headless — f--?s--, DNS returns all pod IPs
  ports:
  - port: 50051
  protocol: TCP
  selector:
  app: user-service
```

**Internal Load Balancing (L7 Proxy)**: Use Envoy Proxy or NGINX as a sidecar or front-proxy. These proxies understand HTTP/2 and can route gRPC calls by `:path` (e.g., route `/UserService/GetUser` to the user service pool).

**Client-Side Load Balancing**: Modern gRPC (1.40+) supports client-side load balancing with pick_first (default) or round_robin. Enable with:

```python
channel = grpc.insecure_channel(
  "dns:///user-service:50051",
  options=[("grpc.lb_policy_name", "round_robin")]
)
```

### Pattern 3: gRPC Gateway (REST-to-gRPC Translation)

Many clients (browsers, mobile apps) cannot use gRPC directly. The gRPC Gateway pattern provides a RESTful JSON API that translates to gRPC calls:

```yaml
# Using grpc-gateway: annotations in .proto generate reverse proxy
service UserService {
  rpc GetUser (GetUserRequest) returns (GetUserResponse) {
  option (google.api.http) = {
  get: "/v1/users/{user_id}"
  };
  }
} />
```

This generates a reverse proxy that accepts HTTP/1.1 JSON requests, deserializes them, makes a gRPC unary call, and translates the response back to JSON. The browser gets REST; the backend gets gRPC speed.

### Pattern 4: gRPC Interceptor Chain (Middleware)

gRPC supports interceptors (unary and stream) for cross-cutting concerns:

```go
// Server-side unary interceptor for logging + metrics
func loggingInterceptor(ctx context.Context, req interface{},
  info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {

  start := time.Now()
  resp, err := handler(ctx, req)
  latency := time.Since(start)

  log.Printf("method=%s latency=%s err=%v", info.FullMethod, latency, err)
  metrics.RecordLatency(info.FullMethod, latency)

  return resp, err
} />

// Chain multiple interceptors
server := grpc.NewServer(
  grpc.ChainUnaryInterceptor(
  loggingInterceptor,
  authInterceptor,
  rateLimitInterceptor,
  ),
)
```

Always use interceptors for: authentication, logging, metrics, rate limiting, deadline propagation, and error handling. Never embed these concerns in business logic.

### Routing Strategy

**Regional affinity**: The global load balancer routes traffic to the nearest region based on latency or geographic DNS. Clients in Europe reach eu-west servers. Clients in Asia reach ap-southeast servers.

**Cross-region fallback**: If a region is degraded, traffic is routed to the next-closest region. The gRPC client library handles this transparently through the channel's resolver — f--?s--, when endpoints become unhealthy, the resolver removes them and the load balancer picks remaining healthy endpoints.

**Read/write splitting**: Write operations go to the primary region (us-east). Read operations go to the nearest replica. The gRPC service can implement this by checking the request type:

```go
func (s *UserService) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.GetUserResponse, error) {
  // Read — f--?s--, use local replica (5ms latency)
  user, err := s.dbReplica.GetUser(ctx, req.UserId)
  if err != nil {
  // Fallback to primary if replica is stale or unavailable
  user, err = s.dbPrimary.GetUser(ctx, req.UserId)
  }
  return user, err
} />
```

## Capacity Planning

Let us walk through capacity planning for a gRPC user service at 10 million DAU.

### Assumptions

| Metric | Value |
|--------|-------|
| DAU | 10 million |
| Requests per user per day | 50 (profile views, searches, updates) |
| Daily requests | 500 million |
| Peak requests per second (10x daily average) | ~58,000 |
| Average message size | 2 KB (request + response) |
| Acceptable p99 latency | 100ms |
| Server spec | 8 vCPU, 32 GB RAM |

### CPU Capacity

Each gRPC call consumes approximately 0.03ms of CPU (serialization + deserialization + gRPC overhead) on an 8-core machine. At 58,000 RPS:

- CPU seconds per second for gRPC overhead: 58,000 — f'-- 0.03ms = 1.74 CPU seconds
- CPU seconds for business logic (5ms average per call): 58,000 — f'-- 5ms = 290 CPU seconds
- Total CPU demand per second: 291.74 CPU seconds
- Cores needed at 70% utilization: 291.74 / 0.7 / 1.0 = ~417 cores
- 8-core instances needed: 417 / 8 = 53 instances

### Memory

- gRPC channel overhead: ~500 KB per connection
- Connections per instance (from internal load balancer): ~100
- Channel memory per instance: 100 — f'-- 500 KB = 50 MB
- Request buffer (concurrent requests — f'-- payload): 500 concurrent — f'-- 2 KB = 1 MB
- Application heap: 2 GB
- Total per instance: ~2.1 GB (well within 32 GB)

### Network Bandwidth

- Peak RPS: 58,000
- Average bytes per call (round trip): 4 KB (2 KB request + 2 KB response)
- Peak throughput: 58,000 x 4 KB/s = 232 MB/s = 1.86 Gbps
- Per-instance: 1.86 Gbps / 53 = 35 Mbps (well within 1 Gbps NIC)

### Final Sizing

| Region | % Traffic | RPS | Instances |
|--------|-----------|-----|-----------|
| us-east (primary) | 40% | 23,200 | 22 |
| eu-west (read replica) | 35% | 20,300 | 18 |
| ap-southeast (read replica) | 25% | 14,500 | 13 |
| **Total** | **100%** | **58,000** | **53** |

Add 30% headroom for traffic spikes: 69 instances.

## Deployment and Operations

### Zero-Downtime Deployment

gRPC services require care during rolling updates because HTTP/2 connections are long-lived. A client connected to an old instance will not automatically reconnect to new instances.

**Strategy**: Graceful shutdown with a drain period.

```go

  server := grpc.NewServer()
  pb.RegisterUserServiceServer(server, &UserService{})

  // Listen for SIGTERM (Kubernetes pod termination signal)
  sigCh := make(chan os.Signal, 1)
  signal.Notify(sigCh, syscall.SIGTERM)

  go func() {
  <-sigCh
  log.Println("Shutting down gracefully...")

  // 1. Stop accepting new RPCs (health check returns NOT_SERVING)
  server.GracefulStop()

  // 2. Wait for in-flight RPCs to complete (up to 30 seconds)
  // gRPC's GracefulStop handles this internally
  }()

  server.Serve(listener)
} />
```
