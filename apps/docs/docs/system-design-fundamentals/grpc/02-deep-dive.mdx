## Data Structures and Algorithms

### Protocol Buffer Wire Format (Varint Encoding)

The fundamental data structure in protobuf is the **varint** — f--?s--, a variable-length integer encoding that uses 1 byte for small values and up to 10 bytes for large values.

How varint works: each byte uses the most significant bit (MSB) as a continuation flag. If MSB = 1, more bytes follow. If MSB = 0, this is the last byte. The remaining 7 bits per byte carry the value, in little-endian order.

**Example: encoding the number 300**

Binary of 300: `100101100`

- Split into 7-bit groups: `0000010 0101100`
- Reverse groups: `0101100 0000010`
- Add continuation bits: `10101100 00000010` (MSB=1 means more bytes)
- Result: `0xAC 0x02` (2 bytes)

Space complexity: O(1) for most values. A typical field tag (1-15) with a small value encodes in 1-2 bytes. Compare to JSON which would emit `"age": 150` — f--?s--, 9 bytes of text. Protobuf encodes it in 2 bytes.

### Wire Type Encoding

Each protobuf field on the wire starts with a tag byte that encodes both the field number and the wire type:

```
Tag = (field_number <under 3) | wire_type
```

Wire types:
| Type | Meaning | Used For |
|------|---------|----------|
| 0 | Varint | int32, int64, uint32, uint64, bool, enum |
| 1 | 64-bit | fixed64, sfixed64, double |
| 2 | Length-delimited | string, bytes, embedded messages, repeated |
| 5 | 32-bit | fixed32, sfixed32, float |

**Why this matters**: The wire type tells the parser how many bytes to read for this field. The field number tells the parser which struct field to populate. Together, they eliminate the need for delimiters or field terminators — f--?s--, the parser always knows exactly how many bytes each field occupies.

### HTTP/2 Frame Structure

Every byte on the wire is accounted for by the HTTP/2 frame format:

```
+-----------------------------------------------+
|  Length (24 bits)  |
+---------------+---------------+---------------+
|  Type (8)  |  Flags (8)  |
+-+-------------+---------------+-------------------+
|R|  Stream Identifier (31 bits)  |
+=+=================================================+
|  Frame Payload (0+)  ...
+---------------------------------------------------+
```

9 bytes of fixed overhead per frame, regardless of payload size. For a 40-byte protobuf message, the total wire cost is 49 bytes — f--?s--, 5 bytes for the gRPC framing layer, 9 bytes for the HTTP/2 frame header, and 35 bytes of protobuf payload.

### HPACK Compression Table

HPACK uses two tables to avoid retransmitting HTTP/2 headers:

**Static Table** (61 predefined entries): Contains common headers like `:method`, `:path`, `content-type`, `grpc-status`. These are referenced by an index (1 byte) instead of being sent as text.

**Dynamic Table**: Populated with headers seen during the connection. On first request, `grpc-timeout: 5000m` must be sent as literal text (~18 bytes). On the second request, it is referenced by its table index (1 byte).

**Why this matters**: A typical gRPC call carries `:method POST`, `:path /UserService/GetUser`, `content-type application/grpc`, `te trailers`, and `grpc-timeout` — f--?s--, approximately 70 bytes of headers uncompressed. After HPACK, these compress to under 30 bytes. Over 10,000 RPCs/second, that saves 400,000 bytes/second in header overhead alone.

## Consistency and State

### Server Crash Mid-Request

**Scenario**: A gRPC server processes a `CreateOrder` RPC. It debits inventory, creates the order record, then crashes before returning the response.

**What happens**: The TCP connection drops. The client receives a `UNAVAILABLE` status with error details "TCP connection closed by remote host." The client has no way to know whether the order was created.

**Mitigation**: Use idempotency keys. The client generates a unique `idempotency_key` and attaches it as gRPC metadata. The server checks for duplicate keys before processing. If the server crashed after persisting the order, the retry with the same key returns the existing result rather than creating a duplicate.

### Network Partition

**Scenario**: A network partition separates the client from the server for 10 seconds.

**What happens**: The client's gRPC library continues sending data, but the TCP connection eventually times out (default TCP keepalive is 2 hours — f--?s--, way too long). The gRPC layer's deadline mechanism is the primary defense: if the client set a 5-second deadline, the RPC is cancelled after 5 seconds without waiting for TCP timeout. Without a deadline, the call hangs for minutes.

**Mitigation**: Always set deadlines on every gRPC call. Use client-side timeouts as a safety net below the deadline.

### Concurrent Writes (Race Condition Detection)

**Scenario**: Two clients simultaneously call `UpdateInventory(item_id, quantity=-5)`. Both read the same current value of 100, compute 95, and write. The second write overwrites the first — f--?s--, both see 95 when the correct result should be 90.

**gRPC approach**: This is an application-level concern, not a protocol concern. Use optimistic concurrency control with version numbers in your protobuf messages:

```protobuf
message UpdateInventoryRequest {
  string item_id = 1;
  int32 delta = 2;
  int64 expected_version = 3;  // read before write
} />

message UpdateInventoryResponse {
  int64 new_version = 1;
} />
```

The server rejects the request if `expected_version` does not match the current version (returns `ABORTED` status code). The client retries by re-reading the current value.

## Configuration and Tuning

### Channel Configuration

| Setting | Default | What It Controls | When to Change | What Breaks |
|---------|---------|-----------------|----------------|-------------|
| `grpc.max_send_message_length` | 4 MB | Maximum outgoing message size | Increase for file uploads, large dataset streaming | Memory exhaustion if set too high |
| `grpc.max_receive_message_length` | 4 MB | Maximum incoming message size | Increase for large responses | Memory exhaustion, OOM kills |
| `grpc.keepalive_time` | 2 hours | How often to send HTTP/2 PING frames on idle connections | Reduce to 10-30 seconds to detect dead peers faster | Increased bandwidth (tiny, but non-zero) |
| `grpc.keepalive_timeout` | 20 seconds | How long to wait for PING ACK before closing | Reduce to 5 seconds for faster failure detection | False positives on slow networks |
| `grpc.client_idle_timeout` | Disabled | Close idle channels after duration | Set to 5 minutes on server to clean up zombie connections | Disconnecting long-lived but quiet streaming clients |
| `grpc.max_concurrent_streams` | 100 | Max concurrent RPCs per connection | Increase to 1000 for high-throughput services | Increased head-of-line blocking under high concurrency |

### Deadline and Timeout

| Setting | Default | What It Controls | When to Change | What Breaks |
|---------|---------|-----------------|----------------|-------------|
| Client deadline | None (caller must set) | Max wall-clock time for RPC | **Always set this** to 5-30 seconds per call | Without this, crashed servers cause 2-hour TCP timeouts |
| Server deadline check | None | Server checks if deadline has expired before processing | Enable in health check middleware | Slight CPU overhead from deadline comparisons |

### Retry Configuration

| Setting | Default | What It Controls | When to Change |
|---------|---------|-----------------|----------------|
| `grpc.enable_retries` | true | Enable automatic retry on UNAVAILABLE | Disable for non-idempotent mutations (or use idempotency keys) |
| `grpc.max_retry_attempts` | 5 | Maximum retry count | Reduce to 2-3 for latency-sensitive calls |
| `grpc.initial_backoff` | 1 second | First retry delay | Increase to 5 seconds to reduce server load during cascading failures |
| `grpc.max_backoff` | 120 seconds | Maximum retry delay | Keep default — f--?s--, prevents retry storm |
| `grpc.backoff_multiplier` | 2.0 | Exponential backoff factor | Increase to 3.0 for more aggressive backoff |

### Flow Control

| Setting | Default | What It Controls | When to Change |
|---------|---------|-----------------|----------------|
| HTTP/2 initial window size | 65,535 bytes | Per-stream flow control window | Increase to 1 MB for streaming large responses (reduces ACK overhead) |
| HTTP/2 connection window | 65,535 bytes | Per-connection flow control window | Increase to 5-10 MB for high-throughput connections |
| `grpc.http2.min_time_between_pings` | 5 minutes | Server-side: min interval between client pings | Reduce to 1 minute for real-time streaming apps |


The most impactful tuning for most services: **set deadlines** on every call, set **keepalive to 30 seconds** for fast failure detection, and **increase max_receive_message_length** if you handle payloads over 4 MB. Everything else is situational.
