## Google: gRPC Retry Design

Google's gRPC framework includes a built-in retry mechanism that is one of the most well-designed in the industry. The retry system is part of the gRPC specification and is implemented consistently across all supported languages.

### gRPC Retry Policy Configuration

The gRPC retry policy is configured in the service configuration as a JSON object. The policy specifies the maximum number of retry attempts, the initial backoff, the maximum backoff, the backoff multiplier, and the set of status codes that should be retried.

```json
{
  "methodConfig": [
    {
      "name": [
        { "service": "payment.v1.PaymentService", "method": "Charge" }
      ],
      "retryPolicy": {
        "maxAttempts": 5,
        "initialBackoff": "0.1s",
        "maxBackoff": "30s",
        "backoffMultiplier": 2,
        "retryableStatusCodes": ["UNAVAILABLE", "DEADLINE_EXCEEDED"]
      }
    }
  ]
}
```

The `retryableStatusCodes` list is critical. gRPC defines a set of status codes, and only specific codes are retryable. `UNAVAILABLE` (the server is not reachable) is retryable. `DEADLINE_EXCEEDED` (the operation timed out) is retryable only if the operation is idempotent. `INVALID_ARGUMENT` is never retryable.

### Per-Status-Code Retry Behavior

gRPC uses the status code to determine whether the retry is safe. For `UNAVAILABLE`, gRPC does not require idempotency because the server never received the request. For `DEADLINE_EXCEEDED`, gRPC requires idempotency because the server may have received and processed the request before the deadline expired.

This distinction is important. A `DEADLINE_EXCEEDED` retry of a non-idempotent mutation could cause duplicate processing. gRPC enforces the idempotency check at the client level, refusing to retry non-idempotent operations on `DEADLINE_EXCEEDED`.

### gRPC Retry Budget

gRPC also includes a retry budget. The budget limits the total number of retries across all operations to a configurable percentage of the total request volume. The default is 10%. If 10% of requests are retries, no further retries are allowed. This prevents retries from overwhelming the network.

```json
{
  "retryThrottling": {
    "maxTokens": 10,
    "tokenRatio": 0.1
  }
}
```

The token ratio of 0.1 means that 10% of the request volume is allocated to retries. If the retry volume exceeds this threshold, retries are throttled. This is a global retry budget that applies across all operations, preventing any single client from consuming the entire retry budget.

## AWS: SDK Retry Architecture

The AWS SDK implements one of the most sophisticated retry systems in the industry. The retry system has evolved through three major versions.

### AWS SDK v1: DefaultRetryPolicy

The original AWS SDK had a `DefaultRetryPolicy` with a fixed set of parameters: maxRetries = 3, initialBackoff = 100ms, backoffMultiplier = 2, maxBackoff = 20s. The policy retried on all 5xx errors and certain 4xx errors (throttling errors).

### AWS SDK v2: Adaptive Retry Policy

The AWS SDK v2 introduced the `AdaptiveRetryPolicy`. This policy dynamically adjusts the retry behavior based on the observed failure rate. The adaptive policy uses a token bucket for rate limiting and a variable delay that increases as the failure rate increases.

The adaptive policy has two modes: `legacy` and `adaptive`. In legacy mode, the behavior is the same as v1. In adaptive mode, the retry delay is computed from the measured client-side rate of retryable errors. When the error rate is low, the delay is short. When the error rate is high, the delay increases automatically.

The adaptive algorithm works as follows: the client maintains a moving average of the retryable error rate. Each error increases the moving average. Each success decreases it. The retry delay is proportional to the moving average. If 10% of requests fail, the delay is 10% of the maximum delay. If 50% fail, the delay is 50% of the maximum delay.

### AWS SDK v3: Standard Retry Mode

The AWS SDK v3 introduced the `StandardRetryMode`. This mode combines the adaptive rate limiting with a fixed exponential backoff curve. The standard mode is the default.

The standard mode configuration has three presets: `legacy` (3 retries, no rate limiting), `standard` (3 retries, adaptive rate limiting), and `adaptive` (dynamically computed retry count, aggressive rate limiting).

```python
# AWS SDK v3 retry configuration (conceptual)
retry_config = {
    "mode": "adaptive",
    "maxAttempts": 5,
    "rateLimiter": {
        "capacity": 500,
        "fillRate": 0.5,  # tokens per second
    }
}
```

The `adaptive` mode is the most conservative. It limits retries aggressively when the error rate is high. It allows more retries when the error rate is low. This is the recommended mode for production systems.

## Stripe: Idempotency Key System

Stripe's idempotency key system is considered the gold standard for payment API reliability. Stripe requires that all mutations include an idempotency key. The key is generated by the client and sent as the `Idempotency-Key` header.

### Stripe Idempotency Key Design

The idempotency key is an arbitrary string chosen by the client. A UUID is recommended. The key is scoped to a specific Stripe account. The same key cannot be used for different accounts.

Stripe stores the idempotency key with the response for 24 hours. Within that window, any request with the same key returns the same response. The request is not processed again. This guarantees that retries do not cause duplicate charges.

### Stripe Retry Recommendations

Stripe recommends the following retry strategy for their API:

- Retry on 5xx errors and network errors
- Max retries: 3
- Initial delay: 500ms
- Backoff multiplier: 2
- Max delay: 30s
- Always use idempotency keys

Stripe also recommends using exponential backoff with jitter. They recommend full jitter as the default. This prevents retry storms when their API experiences a transient issue.

### Stripe Infrastructure for Idempotency

Stripe stores idempotency keys in a globally replicated, strongly consistent key-value store. The store uses DynamoDB with DAX (DynamoDB Accelerator) for low-latency reads. The write path uses conditional writes to ensure that only the first request for a given key is processed.

The system is designed to handle the full key volume: millions of keys per second during peak traffic. The 24-hour TTL ensures that the storage cost is bounded. The strong consistency ensures that retries are correctly deduplicated across all regions.

## Netflix: Fault Tolerance with Hystrix

Netflix's Hystrix library was one of the first production-grade retry and circuit breaker systems. Although Hystrix is now in maintenance mode, its design principles are widely adopted.

### Hystrix Retry Model

Hystrix treated retry as part of a broader fault tolerance framework. The framework included bulkheads (thread pool isolation), circuit breakers, timeouts, and retries. Retries were the last resort, applied only after the circuit breaker closed and the timeout expired.

Hystrix used a simple retry model: maxRetries = 1 (retry once). The single retry used an immediate retry with no backoff. The reasoning was that if the operation failed once, the circuit breaker would open and prevent further retries.

### The Hystrix Lesson

The key lesson from Hystrix is that retries are most effective when combined with other fault tolerance patterns. Retries without circuit breakers are dangerous. Retries without bulkheads are dangerous. Retries without timeouts are dangerous.

Hystrix showed that a single retry with immediate execution is often sufficient. The first attempt and the retry happen within the same timeout window. If both fail, the circuit opens. This is a simple model that is easy to reason about and debug.

### Modern Netflix: Resilience4j

Netflix now uses Resilience4j, which is a more modern retry library. Resilience4j supports exponential backoff, jitter, and a configurable retry budget. It integrates with circuit breakers, rate limiters, and bulkheads.

Resilience4j's retry configuration is typically set per operation. A read operation may have 3 retries with 100ms backoff. A write operation may have 5 retries with 500ms backoff and idempotency key support.

## Open Source Retry Libraries

### Resilience4j

Resilience4j is a lightweight fault tolerance library for Java. It supports retry with exponential backoff, jitter, and a retry budget. The retry configuration is specified as a `RetryConfig` object.

```java
RetryConfig config = RetryConfig.custom()
    .maxAttempts(5)
    .waitDuration(Duration.ofMillis(100))
    .intervalFunction(IntervalFunction.ofExponentialBackoff())
    .retryExceptions(TransientException.class)
    .build();

Retry retry = Retry.of("paymentService", config);
```

Resilience4j integrates with Spring Boot, Micrometer, and the circuit breaker. The retry metrics are automatically exported to Prometheus. The circuit breaker and retry are composed in a chain: the circuit breaker opens when the retry count exceeds the failure threshold.

### Polly

Polly is the .NET equivalent of Resilience4j. It supports retry, circuit breaker, bulkhead, timeout, and rate limiting. Polly's retry policy supports exponential backoff with jitter.

```csharp
var retryPolicy = Policy
    .Handle<HttpRequestException>()
    .OrResult<HttpResponseMessage>(r => r.StatusCode == HttpStatusCode.ServiceUnavailable)
    .WaitAndRetryAsync(
        3,
        retryAttempt => TimeSpan.FromMilliseconds(
            Random.Shared.Next(0, (int)Math.Pow(2, retryAttempt) * 100)
        )
    );
```

Polly's jitter is applied per-attempt using full jitter. The backoff multiplies by 2 for each attempt. The jitter is computed using `Random.Shared.Next(0, baseDelay)`. This is the standard full jitter formula.

### Envoy Retry Policies

Envoy, the service proxy, supports configurable retry policies at the proxy level. Envoy can retry failed requests before they reach the application. The retry policy is configured in the route configuration.

```yaml
retry_policy:
  retry_on: "5xx,gateway-error,connect-failure"
  num_retries: 3
  retry_host_predicate:
    - name: envoy.retry_host_predicates.previous_hosts
  host_selection_retry_max_attempts: 3
```

Envoy's retry policies are powerful because they work at the network level. They do not require application changes. The retry is transparent to the service. However, Envoy cannot understand application-level semantics like idempotency. The application must still handle idempotency on the server side.

## Cloud Provider Retry SDKs

### Google Cloud Client Libraries

The Google Cloud client libraries use the gRPC retry policy. Each API method has a configured retry policy in the service definition. The policy specifies which status codes are retryable and the default backoff parameters.

For example, the Pub/Sub client retries on `UNAVAILABLE` and `ABORTED` with a default of 3 retries and 100ms initial backoff. The client library handles the retry transparently. Application code does not need to add retry logic.

### AWS SDK Retry in Practice

The AWS SDK's `standard` retry mode is the default for all modern AWS clients. The retry mode is configured through the environment variable `AWS_RETRY_MODE`. Valid values are `legacy`, `standard`, and `adaptive`.

The `adaptive` retry mode is recommended for production workloads. It dynamically adjusts the retry rate based on the observed error rate. When errors are rare, retries are fast. When errors are common, retries slow down.

The `adaptive` mode is particularly effective for services like DynamoDB and Lambda, where throttling errors are common during traffic spikes. The adaptive retry automatically slows down the retry rate, giving the service time to scale.

### Best Practices from Cloud Providers

Cloud providers converge on a common set of best practices:

1. Always use idempotency keys for mutations
2. Set max retries to 3 for most operations
3. Use exponential backoff with full jitter
4. Cap the maximum delay at 30 seconds
5. Integrate retries with circuit breakers
6. Use a retry budget to limit amplification
7. Monitor retry metrics and alarm on exhaustion
8. Use per-operation retry policies
9. Test retry behavior with fault injection
