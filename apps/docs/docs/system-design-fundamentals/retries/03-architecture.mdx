## Naive Architecture: Infinite Immediate Retry

The naive retry architecture is what most developers write in their first attempt at error handling. It is a loop: send request, if error, send again immediately. No delay. No counter. No budget.

This architecture has three fatal flaws. First, there is no maximum retry count. The loop runs forever if the server never recovers. Second, there is no delay. The server is hammered continuously with no chance to recover. Third, there is no budget. If 1000 clients all use this pattern, the server sees infinite amplification.

The naive architecture causes exactly the retry storm problem described in the concepts section. It is the most common cause of cascading failures in production systems.

## Production Architecture: Layered Retry System

A production retry architecture consists of multiple layers, each adding a specific safeguard. The layers are: the client-side retry interceptor, the retry queue with backoff, the circuit breaker, the jitter layer, the idempotency layer, and the monitoring layer.

### Layer 1: Retry Interceptor

The retry interceptor is a middleware component in the HTTP or gRPC client stack. It intercepts every outbound request before it reaches the transport layer. The interceptor wraps the request with a unique idempotency key, increments the attempt counter, and passes the request to the next layer.

The interceptor is typically implemented as a decorator or interceptor function. In a gRPC client, it is a client interceptor. In an HTTP client, it is middleware in the request pipeline. The interceptor is configured once at startup and applies to all requests matching a set of rules.

The interceptor classifies errors. A 503 response is retryable. A 429 Too Many Requests is retryable but should respect the Retry-After header. A 400 Bad Request is not retryable. Timeouts are retryable for idempotent operations.

### Layer 2: Retry Queue and Backoff

The retry queue holds pending retry attempts sorted by wake time. Each entry contains the request context, the attempt number, and the wake timestamp. A worker goroutine or thread continuously checks the queue, dispatching entries whose wake time has arrived.

The backoff calculator uses the precomputed curve described in the deep dive. The curve is indexed by attempt number. The resulting delay is the base delay before jitter is applied.

The max retries check happens before the entry is inserted into the queue. If the attempt counter has reached max retries, the entry is discarded and the error is returned to the caller.

### Layer 3: Jitter Generator

The jitter generator applies randomization to the base delay. Full jitter is the default. The jitter function uses a per-client random seed seeded from the client's hostname and process ID. This ensures that different clients produce different jitter values even if they start at the same time.

### Layer 4: Retry Budget

The retry budget is the rate limiter. It uses a token bucket with a small capacity and a slow refill rate. The budget is per-client-instance, not per-request. This prevents a single misconfigured client from exhausting the server's resources.

The budget check happens before the delay queue insertion. If the budget is empty, the retry is skipped immediately. This saves the work of computing backoff, applying jitter, and queueing.

### Layer 5: Transport Layer

The transport layer handles the actual network communication. It maintains a connection pool, sets timeouts, and handles DNS resolution. The transport layer must be configured with appropriate timeouts that account for the retry backoff.

A common mistake is to set a very short transport timeout that expires before the backoff delay finishes. The transport timeout should be large enough to cover the entire retry cycle, or the timeouts should be layered (transport timeout per attempt, overall deadline for all attempts).

### Layer 6: Server-Side Idempotency

The server maintains an idempotency key store. The store maps keys to responses. When a request with an idempotency key arrives, the server checks the store. If the key exists, the server returns the cached response without processing the request. If the key does not exist, the server processes the request and stores the response.

The idempotency key store must be durable and consistent. Redis with persistence or DynamoDB with strong consistency are common choices. The keys have a TTL — typically 24 hours — after which they are automatically deleted.

## Integration Patterns

### Pattern 1: Client-Only Retry

The simplest integration pattern. The client has a retry interceptor, but the server has no retry logic. The client retries on 5xx errors and timeouts. The server is stateless and does not track idempotency keys.

Use this pattern for read-only APIs or idempotent write APIs. The advantage is simplicity. The disadvantage is that the server may process the same request multiple times without knowing it.

### Pattern 2: Client Retry + Server Idempotency

The client has a retry interceptor with idempotency keys. The server has an idempotency key store. The client generates a unique key per request. The server deduplicates requests using the key.

Use this pattern for payment APIs, order creation APIs, and any API where duplicate processing is unacceptable. The advantage is exactly-once semantics. The disadvantage is the operational overhead of the idempotency key store.

### Pattern 3: Mesh Retry

In a service mesh, the sidecar proxy handles retries. The application code has no retry logic. The proxy intercepts all traffic, applies retry policies, and manages circuit breakers. Envoy and Linkerd provide this capability.

Use this pattern in a service mesh environment where you want to centralize retry policy. The advantage is that application code does not need to be modified. The disadvantage is that the proxy cannot understand application-level semantics like idempotency.

### Pattern 4: Async Retry with Dead Letter Queue

Some retries should not block the caller. Async retry dispatches the failed request to a queue, processes it asynchronously, and retries according to a schedule. If all retries are exhausted, the request moves to a dead letter queue for manual inspection.

Use this pattern for background jobs, notifications, and webhook deliveries. The advantage is that the caller is not blocked by retries. The disadvantage is that the result is not immediately available.

## Multi-Region Retry

In a multi-region deployment, retries must account for regional failure. A request to the primary region fails. The retry system should consider routing the retry to a secondary region.

### Regional Retry Strategy

The regional retry strategy has three tiers. Tier 1: retry within the same region. Tier 2: retry in a different availability zone within the same region. Tier 3: retry in a different region entirely.

Each tier has a separate retry configuration. Within-region retries use a short delay (initialDelay: 50ms, maxRetries: 2). Cross-zone retries use a medium delay (initialDelay: 200ms, maxRetries: 2). Cross-region retries use a long delay (initialDelay: 1000ms, maxRetries: 1).

The total cross-region retry time must not exceed the client's timeout. A typical timeout for a cross-region operation is 10 seconds. The retry budget for three tiers is: 50 + 100 + 200 + 400 + 1000 = 1750ms of delay, plus five round trips at 100ms each = 500ms, for a total of 2250ms. This is well within the 10-second budget.

## Capacity Planning for Retries

Capacity planning must account for the retry amplification factor. The total request rate that the server sees is: baseRate * (1 + failureRate * maxRetries). If the base rate is 1000 requests per second, the failure rate is 5%, and maxRetries is 3, the server sees: 1000 * (1 + 0.05 * 3) = 1150 requests per second on average.

During a failure event, the failure rate rises. At a 50% failure rate with 3 retries, the server sees: 1000 * (1 + 0.5 * 3) = 2500 requests per second. The server must be provisioned for the peak amplification, not the average.

### Provisioning Formula

serverCapacity = baseRate * (1 + worstCaseFailureRate * maxRetries) * safetyMargin

If baseRate = 1000 rps, worstCaseFailureRate = 0.3, maxRetries = 3, safetyMargin = 1.5: serverCapacity = 1000 * (1 + 0.9) * 1.5 = 2850 rps. Without accounting for retries, the provisioning would be 1000 * 1.5 = 1500 rps, leaving the server vulnerable during failures.

## Operations

### Monitoring Retries

A retry system must expose the following metrics:

**retry.attempts.count** — Total number of retry attempts. Tracked by attempt number (first retry, second retry, third retry). A spike in second and third retry attempts indicates a chronic failure.

**retry.success.count** — Number of retries that resulted in success. A high success rate for retries indicates that the retry system is effective. A low success rate indicates that retries are wasted.

**retry.exhausted.count** — Number of operations that exhausted all retries. This is the most important metric. Each exhausted retry is a user-visible failure.

**retry.budget.exhausted.count** — Number of times the retry budget was empty, causing a retry to be skipped. A high budget exhaustion rate indicates that the retry budget is too small or the failure rate is too high.

**retry.latency.ms** — The additional latency contributed by retries. Tracked as a histogram. A high p99 retry latency indicates that the backoff is too aggressive or the server recovery time is too long.

### Alarming

Alarm on retry.exhausted.count exceeding a threshold. Each exhausted retry represents a failure that the retry system could not recover from. A significant number of exhausted retries indicates a systemic problem that requires immediate attention.

Alarm on retry.budget.exhausted.count exceeding a threshold. Budget exhaustion means the retry system is suppressing retries. This is a safety mechanism, but it also means that failures are not being retried. It may indicate that the budget is too small or that the failure rate is too high.

Alarm on retry.latency.ms p99 exceeding a threshold. High retry latency means that users are experiencing slow responses due to retries. This may indicate that the backoff configuration is too aggressive.

### Circuit Breaker Integration

The retry system reports each retry failure to the circuit breaker. The circuit breaker tracks the failure rate over a sliding window. When the failure rate exceeds the threshold, the circuit opens. The retry system checks the circuit before each attempt and skips retries when the circuit is open.

The circuit breaker recovery time must account for retry backoff. If the circuit opens and the retry backoff is still running, the retries are wasted. The retry system should cancel pending retries when the circuit opens.

### Gradual Rollout of Retry Configuration

Retry configuration changes should be rolled out gradually. A change from 3 retries to 5 retries doubles the amplification factor during failures. Roll out to 1% of traffic, monitor the retry metrics, then increase to 10%, then 50%, then 100%.

A retry configuration change that looks safe in isolation can cause cascading failures when combined with other services' retry configurations. A load test with realistic failure scenarios is essential before rolling out retry changes to production.
