## Foundation Challenges

These challenges test your understanding of retry fundamentals. You should be able to complete each one in 15-20 minutes.

### Challenge F1: Configure a Retry System

**Scenario:** You are building a weather API client that fetches current temperature data. The API has a 95th percentile response time of 150ms. During storms, the API returns 503 errors approximately 2% of the time. Each 503 lasts about 200ms.

**Task:** Design a retry configuration for this API. Specify the initialDelay, multiplier, maxDelay, maxRetries, and jitter type. Calculate the total additional latency that retries add to the p95 response time.

**Guiding questions:**
- What is the trade-off between catching short failures and adding latency?
- How many retries are needed to cover a 200ms failure window?
- What happens to the p95 latency with your configuration?

**Expected solution:** initialDelay = 100ms, multiplier = 2, maxDelay = 5000ms, maxRetries = 2, jitter = full. The base delay is 100ms + 200ms = 300ms for two retries. With jitter, the delays are random within [0, 100] and [0, 200]. The average additional latency is 150ms. The p95 response time increases from 150ms to approximately 300ms.

**Check your work:** Calculate the amplification factor. At a 2% failure rate with 2 retries: 1 + (0.02 * 2) = 1.04. The API sees 4% more traffic due to retries. This is acceptable.

### Challenge F2: Identify the Retry Bug

**Scenario:** A developer wrote the following retry code for an order service. The service calls a payment gateway. Find all the bugs and explain how to fix each one.

```python
def charge_customer(amount, token):
    max_retries = 10
    for attempt in range(max_retries):
        try:
            response = payment_gateway.charge(amount, token)
            return response
        except Exception:
            if attempt < max_retries - 1:
                continue
            else:
                raise
```

**Bugs to identify:**

1. **Infinite retry in practice:** maxRetries = 10. The total time with no backoff is 10 * (timeout + processing time). If the timeout is 5 seconds and the processing time is 500ms, the total time is 55 seconds. The caller times out long before the retries exhaust.

2. **No backoff:** The retry fires immediately. No delay between attempts. This creates a retry storm against the payment gateway.

3. **No error classification:** The code catches all exceptions. It retries non-retryable errors (invalid token, insufficient funds). These errors will never succeed on retry.

4. **No idempotency:** The charge is not idempotent. If the first request succeeded but the response was lost, the retry charges the customer again.

5. **No circuit breaker check:** The code does not check if the payment gateway is healthy. If the gateway is down, all 10 retries are wasted.

**Fix:**

```python
def charge_customer(amount, token, idempotency_key):
    config = RetryConfig(
        initial_delay_ms=500,
        multiplier=2,
        max_delay_ms=30000,
        max_retries=3,
        jitter_type="full",
    )
    executor = RetryExecutor(config)

    def is_retryable(error):
        # Only retry on 503 and timeout
        if isinstance(error, ServiceUnavailable):
            return True
        if isinstance(error, TimeoutError):
            return True
        return False

    def circuit_open():
        return circuit_breaker.is_open()

    result = executor.execute(
        request_id=idempotency_key,
        operation=lambda: payment_gateway.charge(amount, token, idempotency_key),
        is_retryable=is_retryable,
        circuit_open=circuit_open,
    )
    return result
```

### Challenge F3: Calculate Amplification

**Scenario:** A system has a base request rate of 2,000 requests per second. During a traffic spike, 15% of requests fail. The retry configuration is maxRetries = 3. Calculate the following:

1. The total request rate (including retries) that the server sees during the spike
2. The total request rate if maxRetries is increased to 5
3. The maxRetries value that keeps the total request rate under 2,800 rps
4. The capacity that the server should be provisioned for, with a 20% safety margin

**Answers:**

1. totalRate = 2000 * (1 + 0.15 * 3) = 2000 * 1.45 = 2900 rps
2. totalRate = 2000 * (1 + 0.15 * 5) = 2000 * 1.75 = 3500 rps
3. 2800 = 2000 * (1 + 0.15 * maxRetries). 2800/2000 = 1 + 0.15 * maxRetries. 1.4 = 1 + 0.15 * maxRetries. 0.4 = 0.15 * maxRetries. maxRetries = 2.67. So maxRetries = 2 keeps the rate under 2800 rps.
4. provisionedCapacity = 2900 * 1.2 = 3480 rps (using the 3-retry configuration)

## Design Challenges

These challenges require designing a retry system from scratch. Allocate 30-45 minutes per challenge.

### Challenge D1: Design Retries for a File Upload Service

**Scenario:** You are building a file upload service that accepts files up to 100MB. The service stores files in S3 and returns a URL. Uploads sometimes fail due to network interruptions. The service handles 500 uploads per minute during peak hours.

**Requirements:**
- Uploads must eventually succeed (at-least-once semantics)
- The client should not block for more than 30 seconds total
- The server must not be overwhelmed by retry traffic
- Partial uploads should not require re-uploading the entire file

**Design an approach that covers:**
1. The retry strategy and configuration parameters
2. How partial upload failures are handled
3. The idempotency mechanism (if any)
4. The capacity planning for the retry load
5. How the system degrades during extended failures

**Solution outline:**

1. **Retry strategy:** Use exponential backoff with full jitter. initialDelay = 500ms, multiplier = 2, maxDelay = 10000ms, maxRetries = 4. The total delay is 500 + 1000 + 2000 + 4000 = 7500ms for four retries. Plus the upload time. The 30-second client timeout is respected.

2. **Partial upload handling:** Use S3 multipart uploads. Each part is independently uploaded and can be retried. If a part fails, only that part is retried, not the entire file. The upload ID serves as the idempotency key for the overall upload. Each part has a part number and an ETag for retry detection.

3. **Idempotency mechanism:** The client generates an upload ID (UUID) before starting the upload. The server associates the ID with the upload state. If the server receives a request with an existing upload ID, it returns the current state (in-progress or complete). This prevents duplicate uploads.

4. **Capacity planning:** 500 uploads per minute = 8.3 uploads per second. With a 10% failure rate and 4 retries: 8.3 * (1 + 0.1 * 4) = 11.6 upload attempts per second. Each attempt may involve multiple part uploads. Provision for 15 upload attempts per second.

5. **Degradation:** If S3 is unavailable, the retry system detects the failure and stops retrying after 4 attempts. The circuit breaker opens. The upload service returns a 503 to the client. The client can retry later with the same upload ID. The partial upload is preserved in S3 for 24 hours.

### Challenge D2: Design Retries for a Real-Time Chat Notification Service

**Scenario:** You are building a chat notification service that delivers push notifications to mobile devices. The service must deliver 100,000 notifications per minute. Each notification must be delivered within 5 seconds or discarded. The downstream push notification service (APNS/FCM) occasionally returns transient errors.

**Requirements:**
- Notifications must be delivered as quickly as possible
- Notifications that cannot be delivered within 5 seconds are discarded
- The downstream push service must not be overloaded
- Duplicate notifications are acceptable (at-least-once delivery)

**Design an approach that covers:**
1. The retry strategy that respects the 5-second deadline
2. How the retry budget protects the downstream service
3. The priority of retries vs. new notifications
4. How to handle a sustained failure of the downstream service

**Solution outline:**

1. **Retry strategy:** Use immediate retry with a short exponential backoff. initialDelay = 100ms, multiplier = 1.5, maxRetries = 3. The total delay is 100 + 150 + 225 = 475ms. Plus the round-trip time of approximately 500ms. Total time for three retries: < 1 second. This fits within the 5-second deadline with 4 seconds of buffer.

2. **Retry budget:** Use a token bucket with capacity 1000 tokens and refill rate 500 tokens per second. At 100,000 notifications per minute, the average rate is 1,667 notifications per second. The budget allows 500 retries per second, which is 30% of the base rate. This protects the downstream service from retry storms.

3. **Priority:** New notifications have higher priority than retries. The retry queue uses a priority system. New notifications are processed immediately. Retries are queued and processed when capacity allows. If the queue exceeds 10,000 entries, retries are dropped in favor of new notifications.

4. **Sustained failure:** If the downstream service is down for more than 30 seconds, the circuit breaker opens. Retries are stopped. New notifications are stored in a buffer with a 5-second TTL. When the service recovers, buffered notifications are sent immediately. Notifications that exceed the 5-second TTL are discarded.

### Challenge D3: Design Retries for a Distributed Database Write

**Scenario:** You are building a distributed database that uses quorum-based writes. A write requires acknowledgment from at least 2 out of 3 replicas. Replicas occasionally fail or become slow. The write path must be reliable.

**Requirements:**
- Writes must complete within 200ms
- The write must be durable (survive replica failures)
- Conflicting writes must be detected and resolved
- The retry system must handle replica failures without blocking

**Design an approach that covers:**
1. How retries are scoped (retry the write to a specific replica vs. retry the entire quorum write)
2. The retry strategy for individual replica requests
3. How the retry interacts with the quorum requirement
4. How conflicting writes are detected after retries

**Solution outline:**

1. **Scoped retry:** Retries are scoped to individual replica requests, not the entire quorum write. If two replicas acknowledge and one fails, the retry targets the failed replica only. This avoids the overhead of re-sending to replicas that already acknowledged.

2. **Retry strategy:** Use immediate retry with a short timeout. initialDelay = 10ms, multiplier = 2, maxRetries = 3. The timeout per attempt is 50ms. The total retry time is 10 + 20 + 40 = 70ms. Plus the original attempt at 50ms. Total: 120ms. The write completes within the 200ms deadline.

3. **Quorum interaction:** The quorum is established after the initial attempt. If the initial attempt achieves a quorum, retries are optional. If the initial attempt does not achieve a quorum, retries target the missing replicas. The retry system stops as soon as the quorum is achieved. If the quorum is not achieved after all retries, the write fails.

4. **Conflict detection:** Each write carries a timestamp and a client ID. Replicas store the latest write per key. If a retry arrives with an older timestamp than the current value, the replica returns the current value (not an error). The client detects the conflict and can retry with the merged value.

## Expert Challenges

These challenges require deep distributed systems knowledge. Allocate 60 minutes per challenge.

### Challenge E1: Global Retry Budget Across a Microservice Mesh

**Scenario:** You are designing a retry system for a microservice mesh with 50 services. Each service calls 3-5 other services. The total request volume is 500,000 requests per second. During a failure event, retries can cascade across the mesh, causing a complete collapse.

**Requirements:**
- A global retry budget that limits total retry volume across all services
- Per-service retry budgets that prevent any single service from consuming the global budget
- Dynamic retry rate adjustment based on mesh-wide error rates
- No single point of failure in the retry budget system

**Design an approach that covers:**
1. How the global retry budget is computed and enforced
2. How per-service budgets are allocated and adjusted
3. How the budget system handles partial failures (some services healthy, some not)
4. How the budget state is shared across services without a centralized coordinator

**Solution outline:**

1. **Global budget computation:** The global retry budget is computed as a percentage of the total request volume. Default: 10% of total requests can be retries. The budget is enforced at each service's sidecar proxy. Each proxy maintains a local view of the global budget using a gossip protocol.

2. **Per-service allocation:** Each service gets a retry budget proportional to its request volume. Service A, which handles 10% of requests, gets 10% of the retry budget. Service B, which handles 2% of requests, gets 2% of the retry budget. Budgets are adjusted every 60 seconds based on the observed traffic patterns.

3. **Partial failure handling:** When a service is failing, its retry budget is prioritized. Healthy services reduce their retry rate to free budget for failing services. The adjustment is automatic: each service monitors its own error rate and reduces its retry budget proportionally.

4. **Decentralized state sharing:** Each proxy shares its retry budget consumption with other proxies using a gossip protocol. Every 10 seconds, each proxy sends its budget state to 3 random peers. The state converges within 30 seconds. The total budget is the sum of all proxy budgets. Each proxy independently computes its fair share.

### Challenge E2: Retry System with Temporal Awareness

**Scenario:** You are building a retry system for a time-series database that stores sensor data. The data has temporal locality: if a write fails, it is likely to succeed if retried within a specific window correlated with the sensor's reporting interval. The retry system must learn the optimal retry timing from past success patterns.

**Requirements:**
- The retry system must adapt its backoff strategy to the observed failure patterns
- The system must learn the optimal retry timing for each sensor or data source
- The learning must be online (no batch processing of historical data)
- The system must handle concept drift (failure patterns change over time)

**Design an approach that covers:**
1. How the system learns the optimal retry timing
2. How the learned timing is used to configure the backoff
3. How the system detects and adapts to changing failure patterns
4. How the system handles cold start (no historical data)

**Solution outline:**

1. **Learning mechanism:** The retry system maintains a histogram of successful retry latencies per data source. Each time a retry succeeds, the system records the elapsed time since the original request. The histogram is maintained as an array of exponential moving averages (EMA). The bin size is 50ms. The EMA alpha is 0.1, giving more weight to recent observations.

2. **Backoff configuration:** The system computes the optimal backoff delay as the 75th percentile of the successful retry latency histogram. If 75% of successful retries happen within 500ms of the original request, the initial backoff is set to 500ms. The backoff curve is adjusted every 100 successful retries.

3. **Concept drift detection:** The system monitors the divergence between the current histogram and the baseline histogram using the Kullback-Leibler divergence. If the divergence exceeds a threshold of 0.5, the system detects concept drift and resets the histogram. The new histogram starts fresh, and the backoff falls back to the default configuration until enough new data is collected.

4. **Cold start:** Without historical data, the system uses a default configuration: initialDelay = 100ms, multiplier = 2, maxRetries = 3. After 10 successful retries, the system switches to the learned configuration. The cold start period is approximately 100 seconds at a 10% failure rate.
