## How It Appears

Distributed lock questions appear in system design interviews in multiple forms. The interviewer rarely asks "design a distributed lock" directly. Instead, the distributed lock requirement emerges as one component of a larger system design.

Here are the 10 most common ways the topic surfaces:

1. **Design a distributed job scheduler**: You need to ensure that a scheduled job runs on exactly one worker. A distributed lock is the mechanism that provides this guarantee.

2. **Design a leader election system**: You need to ensure that only one replica acts as the leader at any time. A distributed lock with a TTL provides this.

3. **Design a distributed cache**: You need to ensure that only one replica rebuilds a cache entry when a cache miss occurs. A distributed lock prevents the thundering herd.

4. **Design a distributed rate limiter**: You need to coordinate rate limit counters across multiple application servers. A distributed lock with a short TTL provides the coordination.

5. **Design a distributed counter**: You need a counter that is consistent across multiple processes. The interviewer asks whether you need a distributed lock or can use an atomic operation.

6. **Design a distributed file system**: You need to ensure that only one writer modifies a file at a time. A distributed lock on the file path provides this.

7. **Design a distributed transaction coordinator**: You need to coordinate a two-phase commit across multiple databases. A distributed lock prevents concurrent transactions from accessing the same resources.

8. **Design a database migration system**: You need to ensure that only one instance runs a schema migration. A distributed lock on the migration version provides this.

9. **Design a distributed mutex for resource allocation**: You need to allocate a limited pool of resources such as connections, slots, or licenses across multiple processes. A distributed lock per resource provides this.

10. **Design a distributed configuration system**: You need to ensure that configuration updates are applied atomically across multiple servers. A distributed lock on the configuration key provides this.

### What the Interviewer Tests in Each Question

| Question | What Is Tested | Key Follow-Up |
|---|---|---|
| Job scheduler | Lock as coordination primitive | What happens when a worker crashes mid-job? |
| Leader election | Leader election as distributed lock | How do you prevent split-brain? |
| Cache rebuild | Thundering herd prevention | What is the double-check pattern? |
| Rate limiter | Distributed counting with coordination | What is the fallback when lock fails? |
| Transaction coordinator | Lock for two-phase commit | How do you detect deadlock? |
| File system | Readers-writer lock pattern | How do you prevent writer starvation? |
| Migration system | Coarse-grained lock | What if migration takes longer than TTL? |
| Resource allocation | Lock granularity | How do you prevent resource starvation? |
| Configuration mgmt | Optimistic vs. pessimistic locking | How do you handle conflicting updates? |

## Level-by-Level Expectations

### L3 / Entry-Level

**What you need to know**: You should understand the basic concept of a distributed lock: what problem it solves and when to use it. You should be able to describe the simplest implementation using Redis SETNX with TTL.

**Sample question**: "Our application runs on 10 servers. Each server runs a cron job that processes a shared queue. How do we ensure that only one server processes each queue item?"

**Sample answer**: "I would use a distributed lock. Each server would try to acquire a lock on the queue item before processing it. The simplest implementation is Redis SETNX with a TTL. The server calls `SET queue_item:{id} server_id NX PX 30000`. If the command returns OK, the server holds the lock and processes the item. If it returns nil, another server is already processing it. The TTL of 30 seconds ensures that if the processing server crashes, the lock is automatically released. The release operation uses a Lua script to verify that the releasing server is the lock owner: `if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end`."

**Why this works**: The answer demonstrates understanding of the core concept, a specific implementation, and the safety mechanism (TTL). It does not go into advanced topics like fencing tokens or consensus protocols, which are not expected at this level.

**What to avoid**: Do not propose a database-based lock using SELECT FOR UPDATE without mentioning connection pool exhaustion. Do not claim that Redis SETNX provides perfect safety. Do not forget the TTL.

### L4 / Intermediate

**What you need to know**: You should understand the difference between safety and liveness, and the trade-off between them. You should be able to compare Redis SETNX, ZooKeeper, and etcd lock implementations across latency, throughput, and consistency.

**Sample question**: "Design a distributed lock service. Compare your design with using Redis SETNX versus ZooKeeper ephemeral znodes."

**Sample answer**: "I would design a distributed lock service with four components: a lock registry, a lease manager, a token generator, and a wait queue.

The lock registry maps resource identifiers to lock records. Each record contains the owner identifier, fencing token, TTL, and lease metadata. The registry must support atomic compare-and-swap operations.

The lease manager maintains a timer wheel for all active leases. It checks for expired leases every 500 milliseconds. When a lease expires, it triggers a lock release and notifies waiters.

The token generator produces monotonically increasing fencing tokens. The generator persists the last-issued token to disk so that a crash does not cause token reuse.

The wait queue maintains the ordered list of processes waiting for each lock. When a lock is released, the queue pops the next waiter and notifies it.

Comparing Redis SETNX with ZooKeeper: Redis SETNX is faster, with 1-3 milliseconds acquisition latency and 50,000 operations per second throughput. But Redis does not provide fencing tokens natively, and lock safety is probabilistic -- during leader failover, locks can be lost. ZooKeeper is slower, with 5-15 milliseconds acquisition latency and 8,000 operations per second throughput. But ZooKeeper provides ephemeral sequential znodes that serve as fencing tokens, and the Zab consensus protocol guarantees linearizable consistency.

The choice depends on the use case. For non-critical operations like cache warming, Redis SETNX is adequate. For critical operations like leader election or financial transactions, ZooKeeper or etcd is required."

**Why this works**: The answer demonstrates knowledge of the internal architecture, the key trade-offs, and the decision framework. It shows that you understand both the implementation details and the higher-level constraints.

**What to avoid**: Do not present a single solution as universally correct. Do not ignore the consistency implications. Do not claim that any lock implementation provides perfect safety.

### L5 / Senior

**What you need to know**: You should understand the failure modes of distributed locks in detail -- delayed writer, network partition, clock drift, GC pauses. You should be able to explain fencing tokens and why they are necessary. You should be able to design a lock service with strong consistency guarantees.

**Sample question**: "Our payment processing system uses Redlock for distributed locking. We are seeing occasional duplicate payments. What is the likely cause, and how would you fix it?"

**Sample answer**: "The most likely cause is the delayed writer problem. Here is the sequence of events:

1. The payment service acquires a Redlock for transaction T-847 from a majority of 5 Redis nodes. The lock has a 10-second TTL.
2. The service starts processing the payment. It deducts from account A and prepares to credit account B.
3. A stop-the-world GC pause freezes the service for 15 seconds.
4. After 10 seconds, the Redlock expires. The 5 Redis nodes release the lock.
5. A second instance of the payment service acquires a new Redlock for the same transaction.
6. The second instance processes the entire payment, including crediting account B.
7. The first instance resumes from the GC pause. It does not know its lock expired. It completes the payment, also crediting account B.
8. Account B receives double the expected amount.

Redlock does not eliminate the delayed writer problem. The lock paper claims safety only if bounded clock drift and network delays are guaranteed, but GC pauses violate these assumptions.

The fix has two parts. First, implement fencing tokens. The Redis nodes would assign a monotonically increasing fencing token with each lock acquisition. The payment service sends the fencing token with each write to the accounting database. The database rejects any write with a token lower than the last-seen token. In our scenario, the second instance would receive a higher token than the first instance. The first instance's post-GC write would be rejected because its token is stale.

Second, the payment service should detect that it missed its renewal window. Each renewal attempt increments a client-side counter. If the counter exceeds the expected number of renewals, the service should assume the lock is lost and abort the critical section.

The deeper issue is that Redlock depends on synchronized clocks across 5 Redis nodes. In a cloud environment, clock drift is unbounded during GC pauses or CPU throttling. I would recommend migrating to a consensus-based lock service such as etcd or ZooKeeper, which uses session-based leases rather than wall-clock TTLs."

**Why this works**: The answer demonstrates deep understanding of a specific failure mode, proposes concrete mitigations, and shows awareness of the limitations of the proposed solution.

**What to avoid**: Do not defend Redlock as a perfect solution. The interviewer is presenting a known failure case. Do not propose solutions that do not address the root cause.

### L6 / Staff

**What you need to know**: You should understand the formal guarantees of different consensus protocols (Paxos, Raft, Zab) and their implications for lock safety. You should be able to design a lock service that spans multiple data centers.

**Sample question**: "Design a global distributed lock service for a system that runs across 5 geographic regions. The system requires strong consistency for critical resources and high throughput for non-critical resources."

**Sample answer**: "I would design a two-tier lock architecture with a local lock service in each region and a global lock service spanning all regions.

The local lock services use a 3-node etcd cluster per region. They handle high-throughput locks for region-local resources such as trip assignments and cache rebuilds. The local lock acquisition latency is 3-8 milliseconds. Each local cluster can handle 10,000 operations per second.

The global lock service uses a 7-node etcd cluster with nodes distributed across 3 of the 5 regions. It handles locks for global resources such as configuration keys, feature flags, and cross-region account transfers. The global lock acquisition latency is 100-250 milliseconds due to cross-region Raft consensus.

The resource partitioning is deterministic. Each resource identifier includes a region prefix. Resources starting with `us-east-1:` are handled by the us-east-1 local lock service. Resources starting with `global:` are handled by the global lock service.

For the hierarchical approach, when a local operation needs a global lock:

1. The process acquires the local lock first (3-8 ms).
2. If needed, it declares intent to the global lock service (100-250 ms).
3. If the global lock is acquired, the process proceeds with the cross-region operation.
4. The global lock has a longer TTL of 60 seconds to account for cross-region latency.
5. The local lock is released after the global operation completes.

The fencing token mechanism requires coordination between local and global lock services. The global token generator produces tokens with a region prefix and a sequence number, for example `us-east-1:1042`. The protected resource checks both the prefix and the sequence number. This ensures that a token from one region cannot be confused with a token from another region.

The key insight is that most resources are region-local. Only a small fraction of resources, less than 1 percent, require global coordination. The two-tier architecture provides high throughput for the common case while maintaining strong consistency for the critical case."

**Why this works**: The answer demonstrates architectural thinking, practical experience with multi-region systems, and a nuanced understanding of the trade-offs. It shows judgment in partitioning resources.

**What to avoid**: Do not propose a single global lock service for all resources. Do not ignore the cross-region latency impact.

### L7 / Principal

**What you need to know**: You should be able to design a lock service from scratch, including its consensus protocol, state machine, and client library. You should understand the formal properties of distributed locking.

**Sample question**: "Design a new distributed lock protocol that provides better safety guarantees than Redlock while maintaining similar performance characteristics."

**Sample answer**: "I would design a protocol that combines the performance of Redis with the safety guarantees of a consensus protocol. The key insight is that the lock service does not need consensus for every operation -- it only needs consensus for the fencing token assignment and lease management.

The protocol works as follows. Each lock service node maintains an in-memory lock registry and a Raft-based token generator. The token generator is a small state machine that produces monotonically increasing fencing tokens. The token generator runs on a 3-node Raft cluster.

Lock acquisition proceeds in two phases. Phase 1 is the fast path: the client sends the lock request to any node. The node checks its local lock registry. If the resource is free and the node's clock has not advanced beyond the lease expiry, the node tentatively grants the lock with a provisional token. The node returns the grant immediately in 1-3 milliseconds.

Phase 2 is the confirmation path: the node asynchronously commits the lock acquisition to the Raft token generator. The token generator assigns a confirmed fencing token. If the confirmed token matches the provisional token, the lock is permanent. If the confirmed token is different, the node revokes the tentative grant and the client retries.

The protocol provides two modes. Optimistic mode: the client proceeds with the provisional token. If the confirmation fails, the client must roll back. This mode provides 1-3 millisecond latency with a 99.9 percent success rate under normal operation. Pessimistic mode: the client waits for the Raft confirmation before proceeding. This mode provides 5-10 millisecond latency with 100 percent safety.

The key innovation is that the fast path succeeds unless there is contention or a failure. Under normal operation with low contention, the client gets Redis-like performance with ZooKeeper-like safety guarantees."

**Why this works**: The answer demonstrates deep technical expertise, innovative thinking, and practical experience with consensus protocols.

**What to avoid**: Do not propose a solution that already exists. The interviewer is asking for something new.

### L8 / Distinguished

**What you need to know**: You should be able to discuss the theoretical limits of distributed locking and the implications of the FLP impossibility result. You should be able to design systems that work without distributed locks.

**Sample question**: "Is it possible to build a perfectly safe distributed lock? If not, what is the fundamental limitation?"

**Sample answer**: "No, it is not possible to build a perfectly safe distributed lock in an asynchronous system with the possibility of process crashes. This is a direct consequence of the FLP impossibility result, which states that in an asynchronous distributed system where at least one process may crash, no consensus protocol can guarantee both safety and liveness.

A distributed lock requires consensus: all participants must agree on who holds the lock. The FLP result implies that any distributed lock implementation can be delayed indefinitely by a combination of message delays and process pauses. In practice, this means there is always a window where two processes can believe they hold the same lock.

The window manifests differently in different implementations. In TTL-based locks, the window is the TTL duration. A delayed writer can resume within the TTL and overwrite data. In session-based locks, the window is the session timeout. A process whose session expires but continues running can overwrite data. In consensus-based locks, the window is the leader election timeout. During leader election, no decisions are made, and the new leader may have incomplete state.

The fundamental limitation is that we cannot distinguish between a crashed process and a slow process. A process that is paused by a GC pause looks the same as a crashed process to the lock service. The lock service must either wait indefinitely (safety) or grant the lock to a new holder (liveness). It cannot do both.

Given this limitation, the practical approach is to design systems that tolerate the residual risk. Idempotent operations eliminate the impact of duplicate execution. Fencing tokens reduce the risk to a vanishingly small probability. Conflict-free replicated data types (CRDTs) provide eventual consistency without locks."

**Why this works**: The answer demonstrates deep theoretical understanding, awareness of the literature, and practical wisdom.

**What to avoid**: Do not claim perfect safety is achievable. Do not dismiss the question as theoretical.

## FAANG Framework

The FAANG interview framework for distributed lock questions follows a four-part structure: Frame, Analyze, Architect, Navigate.

### Frame

Start by framing the problem. Ask clarifying questions to understand the scope:

- "What is the resource being protected?"
- "What is the consistency requirement?"
- "What is the throughput requirement?"
- "What is the latency budget?"
- "How many processes will contend for the lock?"
- "What is the failure tolerance?"

Example framing for a distributed job scheduler: "We need to ensure that 100 worker processes running on Kubernetes do not execute the same scheduled job simultaneously. There are 1,000 distinct job types. Jobs run for 30 seconds to 5 minutes. We can tolerate occasional duplicate executions for non-critical jobs but not for critical jobs like payment processing."

### Analyze

Analyze the requirements against the trade-off space:

- **Consistency requirement**: Critical jobs require strong consistency. Non-critical jobs can tolerate eventual consistency.
- **Scalability requirement**: 100 workers times 1,000 job types equals 100,000 potential lock acquisitions per minute.
- **Latency requirement**: Lock acquisition should complete within 50 milliseconds. The job execution is orders of magnitude longer.
- **Availability requirement**: The system must handle lock service failures gracefully. Jobs should still run if the lock service is unavailable.

### Architect

Present your architecture:

- Use a two-tier approach: etcd for critical jobs (strong consistency), Redis for non-critical jobs (high throughput).
- Implement the lock client library with lease renewal, fencing tokens, and graceful degradation.
- The client acquires the lock from the appropriate service based on the job's criticality tag.
- On lock service failure, critical jobs fail closed (stop processing). Non-critical jobs fail open (process without lock).

### Navigate

Navigate the trade-offs and failure modes:

- Discuss the delayed writer failure mode and how fencing tokens prevent it.
- Discuss the network partition scenario and how the quorum-based lock service behaves.
- Discuss the clock drift scenario and why etcd's session-based leases are more resilient.
- Discuss the performance characteristics and how the two-tier architecture provides the best of both worlds.

## Phrase Bank

Use these phrases to communicate your ideas clearly and confidently.

### Explaining the Problem

- "The fundamental challenge is that we cannot distinguish between a crashed process and a slow process in a distributed system."
- "Without a distributed lock, concurrent processes can read and modify the same shared state, leading to race conditions and data corruption."
- "The key insight is that a distributed lock is not a mutex -- it operates across a network with millisecond latency, not across shared memory with nanosecond latency."
- "The CAP theorem constrains distributed lock services: during a network partition, the lock service must choose between consistency and availability."

### Explaining the Solution

- "The lock service acts as an arbiter, ensuring that only one process holds the lock at any time. The other processes wait in a contention queue."
- "The TTL provides a liveness guarantee -- a crashed holder will not block the system indefinitely because the lock automatically expires."
- "Fencing tokens provide a safety guarantee -- even if two processes hold the lock simultaneously due to a delayed writer, the protected resource rejects stale writes."
- "The lease renewal mechanism provides a heartbeat that extends the lock while the holder is actively processing."

### Discussing Trade-offs

- "There is a fundamental tension between safety and liveness. We can optimize for one at the expense of the other, but we cannot maximize both."
- "Redis provides the best performance but the weakest safety guarantees. ZooKeeper provides the strongest safety but higher latency. etcd sits in the middle with a good balance."
- "The TTL is a tuning parameter that slides along the safety-liveness spectrum. A longer TTL favors safety. A shorter TTL favors liveness."
- "Choosing a lock implementation is a risk management decision. The cost of a duplicate lock grant must be weighed against the cost of lock service unavailability."

### Discussing Failure Modes

- "The delayed writer is the most insidious failure mode. A process pauses, the lock expires, and the process resumes writing with stale authority."
- "Network partitions are unavoidable. The lock service must handle the partition by refusing lock acquisitions on the minority side."
- "Clock drift can cause unbounded safety violations in TTL-based lock services. Consensus-based lock services avoid this by using session-based leases."
- "The thundering herd problem occurs when many waiters all detect the lock release simultaneously and race to acquire it. Watch-based notification with sequential waiting prevents this."

### Discussing Implementation Details

- "Redis SETNX with a Lua script for release provides atomic check-and-set with safe release. The script ensures that only the current holder can release the lock."
- "ZooKeeper ephemeral sequential znodes provide automatic fencing tokens through the sequence number. The process creates a znode, lists children, and watches the preceding znode."
- "etcd leases with the concurrency package provide a production-grade lock primitive. The lease is attached to the lock key, and the key is auto-deleted on lease expiry."
- "Database SELECT FOR UPDATE provides row-level locking with transactional semantics. The trade-off is connection pool consumption and limited throughput."

### Handling Follow-Up Questions

- "If the lock service leader crashes during a lock acquisition, the client's request may timeout. The client should retry with exponential backoff. The new leader will replay the Raft log and recover the lock state."
- "If two processes both believe they hold the same lock, the protected resource must detect the conflict. Fencing tokens on the resource side provide this detection."
- "The maximum throughput of a lock service is limited by the consensus protocol. For a single resource, the throughput is bounded by the inverse of the acquisition latency. For multiple resources, throughput scales with the number of resources."
- "The trade-off between a 3-node and 5-node cluster is fault tolerance versus latency. A 5-node cluster tolerates 2 failures but requires agreement from 3 nodes, which adds latency."

### Defending Your Design

- "I recommend a consensus-based lock service such as etcd for critical operations because it provides linearizable consistency and automatic fencing tokens."
- "The two-tier architecture provides high throughput for the common case while maintaining strong consistency for critical operations."
- "The client library handles the complexity of lease renewal and failure detection behind a simple acquire and release API."
- "I chose Redis SETNX for non-critical operations because the 1 in 1,000 risk of duplicate execution is acceptable for cache warming."

## Red Flags

Interviewers watch for these red flags when evaluating distributed lock knowledge:

1. **Claiming perfect safety**: Saying that your lock implementation guarantees that two processes cannot hold the lock simultaneously. No distributed lock provides perfect safety. Acknowledge the residual risk.

2. **Ignoring the TTL**: Designing a lock without a TTL or lease mechanism. This creates a permanent block if the holder crashes. Always include a TTL or session-based lease.

3. **Confusing local and distributed locks**: Saying that a mutex or synchronized keyword solves the problem. The processes run on different machines and do not share memory.

4. **Not mentioning fencing tokens**: Proposing a distributed lock solution without addressing the delayed writer problem. Fencing tokens are the only mechanism that guarantees safety in the presence of delays.

5. **Single implementation bias**: Presenting only one solution such as Redis or ZooKeeper without discussing trade-offs. The right solution depends on the requirements.

6. **Ignoring failure modes**: Designing a lock service without discussing what happens during network partitions, process crashes, or clock drift. The interviewer expects you to acknowledge and address these scenarios.

7. **Over-engineering**: Proposing a consensus-based lock service for a simple cache warming use case. The solution should match the requirements.

8. **Under-engineering**: Proposing a Redis SETNX lock for a financial transaction system. The solution should match the safety requirements.

9. **Ignoring the client side**: Designing the lock service without considering how the client handles lock acquisition, lease renewal, and failure detection. The client library is a critical component.

10. **Not considering alternatives**: Reaching for a distributed lock without considering idempotent operations, optimistic concurrency, or atomic operations. The best solution may not involve a lock at all.

### Drill 7: Code Review a Lock Implementation

Review this code and identify the bugs:

```python
def acquire_lock(redis_client, resource_id, owner_id, ttl_ms=30000):
    result = redis_client.set(resource_id, owner_id, nx=True, px=ttl_ms)
    return result == "OK"

def release_lock(redis_client, resource_id, owner_id):
    redis_client.delete(resource_id)

def execute_with_lock(redis_client, resource_id, fn):
    if acquire_lock(redis_client, resource_id, owner_id):
        try:
            fn()
        finally:
            release_lock(redis_client, resource_id)
```

Time limit: 5 minutes.

**Bugs to find**: The `release_lock` function does not verify that the deleting client is the lock owner. Another client could delete the lock. The `execute_with_lock` function uses an undefined `owner_id` variable. There is no lease renewal or TTL monitoring. There is no fencing token mechanism.

## Key Takeaways for Interview Success

1. **Start with the problem, not the solution**: Frame the requirements before proposing a specific lock implementation. The interviewer wants to see that you understand when a distributed lock is needed and when it is not.

2. **Show depth, not breadth**: Pick one or two implementations and explain them in depth rather than listing many implementations superficially. Demonstrate that you understand the internal architecture of a lock service.

3. **Acknowledge failure modes**: Every distributed lock implementation has failure modes. Acknowledging them and proposing mitigations shows maturity. Claiming perfect safety shows inexperience.

4. **Know the alternatives**: A distributed lock is often the wrong tool. Show you know when to use an atomic operation, optimistic concurrency, or a queue instead.

5. **Use concrete numbers**: "Redis SETNX has 1-3 millisecond latency" is better than "Redis is fast." "The TTL should be 1.5 times the maximum critical section duration" is better than "The TTL should be long enough."

6. **Structure your answer**: Use the FAANG framework (Frame, Analyze, Architect, Navigate) to structure your response. This demonstrates clear thinking and covers all dimensions the interviewer is evaluating.

7. **Practice the sequence diagram**: Many interview questions ask you to draw the interaction flow. Practice drawing the lock acquisition sequence until you can do it from memory.

8. **Know the Redlock debate**: The Redlock controversy is frequently discussed in system design interviews. Having an informed opinion on both sides shows deep knowledge of the distributed systems literature.

9. **Learn the phrase bank**: The specific phrases in the phrase bank section are designed to communicate technical concepts precisely. Practice them until they feel natural.

## Extended Practice Scenarios

Use these additional scenarios for deeper practice. Each scenario targets a specific interview skill.

### Scenario 1: The Thundering Herd Problem

You are designing a cache layer for a social media platform. The platform has 500 cache nodes. When a popular post goes viral, many cache nodes miss simultaneously. All 500 nodes query the database to rebuild the cache. The database is overwhelmed.

**Your task**: Design a solution using distributed locks that prevents the thundering herd while minimizing lock overhead.

**Key points for your answer**:
- Use a per-key lock with a short TTL (2-5 seconds).
- Implement the double-check pattern: check the cache, acquire the lock, check the cache again, rebuild the cache, release the lock.
- If the lock acquisition fails, wait for the cache to be rebuilt rather than querying the database.
- Use a probabilistic early expiry approach to prevent all nodes from detecting the miss simultaneously.

### Scenario 2: The Split-Brain Problem

Your distributed lock service uses a 3-node etcd cluster. A network partition splits the cluster into two groups: node-1 in group A and nodes 2 and 3 in group B. Node-1 was the leader when the partition occurred.

**Your task**: Analyze what happens during the partition and explain how the system recovers.

**Key points for your answer**:
- Group B has a majority (2 of 3 nodes). Node-2 becomes the new leader through Raft election (election timeout of 300ms).
- Group A (node-1) cannot form a quorum. Node-1 steps down as leader. It does not accept writes.
- Lock operations are accepted only by the majority group (group B).
- When the partition heals, node-1 re-joins the cluster as a follower. It catches up on the Raft log entries it missed.
- No split-brain occurs because Raft's single-leader invariant ensures that only one leader exists at any time.

### Scenario 3: The Cascading Failure

Your application holds 5,000 distributed locks simultaneously. The lock service experiences a 2-second latency spike. The clients' lease renewal requests arrive late. 2,000 leases expire. The application loses 2,000 locks. 2,000 critical section operations abort simultaneously. The applications retry, generating 10,000 lock acquisition requests per second, overwhelming the lock service.

**Your task**: Design a defense against this cascading failure.

**Key points for your answer**:
- Implement client-side rate limiting: cap the lock acquisition rate per client.
- Use exponential backoff with jitter for retry attempts.
- Stagger the lease renewal intervals so that not all renewals happen at the same time.
- Implement a circuit breaker: if the lock service latency exceeds a threshold, stop acquiring new locks and gracefully degrade.
- Monitor the lock service's queue depth and apply backpressure to clients before the latency spike occurs.

### Scenario 4: The Global Clock Problem

Your system spans 5 data centers across 3 continents. Each data center has its own lock service instance. You need a global lock for coordinating database schema migrations across all data centers.

**Your task**: Design a global lock service that works across data centers within a 200-millisecond acquisition latency budget.

**Key points for your answer**:
- Use a hierarchical approach: per-data-center locks for local operations and a global lock for cross-data-center operations.
- The global lock service uses a 5-node consensus cluster with nodes in 3 continents (2 in US, 2 in EU, 1 in Asia).
- Write latency is determined by the slowest majority link. A client in the US needs agreement from 2 US nodes and 1 EU node (60ms round trip).
- Use lease-based locks with the lease TTL set to 30 seconds to tolerate the increased latency.
- For the schema migration use case, the infrequent lock acquisition makes the latency acceptable.

### Scenario 5: The Non-Technical Stakeholder

Your product manager asks why the team is spending two sprints building a distributed lock service. "Can't we just use a database row with a flag column?" they ask.

**Your task**: Explain the need for a distributed lock service in non-technical terms. Use analogies and concrete examples of failure scenarios.

**Key points for your answer**:
- "Imagine a single-occupancy restroom with a key on a hook. If someone takes the key and walks away, the restroom is unusable forever. A distributed lock service is like having an automatic timer that returns the key after 5 minutes."
- "A database flag is like a piece of paper that says 'occupied.' If the person holding the paper checks out of the hotel (server crash), the paper remains and no one can use the restroom."
- "The lock service handles failure scenarios that a simple flag cannot: network partitions, process crashes, and concurrent requests."
- "The cost of not having a proper lock service is data corruption and duplicate payments. The two-sprint investment prevents multi-million-dollar incidents."

## Handling the Curveball Question

Interviewers often throw curveball questions to test your composure and depth of understanding. Here are common curveball questions and how to handle them.

**Curveball 1: "What if I told you that distributed locks are inherently unsafe and should never be used?"**

This is a provocative question designed to test your conviction. Do not agree with the premise. Instead, acknowledge the limitations and show that you understand the appropriate use cases.

**Response**: "Distributed locks are not inherently unsafe. They provide probabilistic safety, which is sufficient for many use cases. The key is to understand the failure modes and mitigate them. Fencing tokens make distributed locks safe for critical operations. Idempotent operations provide a safety net if the lock fails. The question is not whether locks are safe, but whether the specific implementation is appropriate for the specific use case."

**Curveball 2: "Why not just use a database row with a flag column instead of a distributed lock?"**

This question tests whether you understand the differences between a simple flag, a database row lock, and a proper distributed lock.

**Response**: "A database flag column can work for simple scenarios with low concurrency. But it has several failure modes. First, there is no atomic check-and-set -- two processes can both read the flag as false and both set it to true. Second, there is no TTL or lease mechanism -- if the process crashes, the flag remains set forever. Third, there is no fencing token -- a delayed writer can overwrite data after its lock expired. A distributed lock service addresses all three failure modes with atomic acquisition, TTL-based liveness, and token-based safety."

**Curveball 3: "How would you implement a distributed lock without using any existing lock service or consensus protocol?"**

This is a deep systems design question that tests your understanding of fundamental distributed systems concepts.

**Response**: "I would implement a distributed lock using a shared file system with atomic file creation. The algorithm is: each process tries to create a unique file in a shared directory. The file name includes a unique identifier and a sequence number. The process that creates the file with the lowest sequence number holds the lock. Other processes watch the directory and wait for the file to be deleted. The lock is released by deleting the file. The crash case is handled by a heartbeat mechanism -- if the holder stops updating the heartbeat file, another process can reclaim the lock after a timeout."

**Curveball 4: "What happens if the network latency between the client and lock service is 500 milliseconds?"**

This question tests whether you understand the practical implications of latency on lock behavior.

**Response**: "Five hundred milliseconds of network latency has several implications. First, lock acquisition takes at least one round trip or 500 milliseconds, which may be too slow for time-sensitive operations. Second, the lease renewal interval must be adjusted. With a 30-second TTL and 500-millisecond latency, the renewal interval should be 10 seconds, with at least 2 seconds of buffer. Third, the lock holder's operations must tolerate the increased latency. If the holder needs to renew the lease while holding the lock, it must start the renewal early enough to account for the round trip time. Finally, cross-region latency of 500 milliseconds suggests that the lock service should be deployed in the same region as the clients, or resources should be partitioned by region."

**Curveball 5: "Your lock service is experiencing 100 percent packet loss for 10 seconds. What happens to your locks?"**

This question tests your understanding of failure handling under extreme conditions.

**Response**: "If the lock service experiences 100 percent packet loss for 10 seconds, several things happen:

1. All pending lock acquisition requests fail with timeout errors.
2. Existing lock holders cannot renew their leases. If the TTL is shorter than 10 seconds, the leases expire and the locks are released.
3. When the packet loss ends, the waiters see that the locks are free and acquire them. The original holders may still believe they hold the locks if they were not notified of the lease expiry.
4. The original holders may write to the protected resource with stale fencing tokens. The resource must reject these stale writes.

The critical mitigations are:
- Set the TTL to longer than the expected maximum packet loss duration. If packet loss can last 10 seconds, set the TTL to at least 15 seconds.
- Implement fencing tokens on the resource side to reject stale writes.
- Implement client-side detection of lease expiry. The client should monitor the heartbeat response and abort the critical section if the lock service is unreachable."

## Common Mistakes Candidates Make

Interviewers consistently observe these mistakes across all levels. Avoiding them will set you apart.

**Mistake 1: Jumping to a solution without clarifying requirements.** You say "I would use Redis SETNX" before asking whether the system needs strong consistency. The interviewer interprets this as premature optimization. Fix: Always clarify requirements first.

**Mistake 2: Ignoring the failure case.** You describe the happy path (acquire, process, release) but do not mention what happens when the lock holder crashes. The interviewer notes that you have not considered liveness. Fix: Always include TTL or lease-based auto-release in your design.

**Mistake 3: Hand-waving the lock service implementation.** You say "we use a lock service" without discussing how it works internally. The interviewer probes deeper, and you cannot describe the lock registry, lease manager, or token generator. Fix: Study the internal architecture of distributed lock services before the interview.

**Mistake 4: Not quantifying.** You say "the lock is fast" instead of "the lock acquires in 1-3 milliseconds with Redis SETNX." The interviewer sees vague thinking. Fix: Use concrete numbers for latency, throughput, and TTL values.

**Mistake 5: Fencing token oversight.** You propose a distributed lock but do not mention how the protected resource validates that the writer still holds the lock. The interviewer knows that delayed writers are the most common failure mode. Fix: Always include fencing tokens or an equivalent mechanism.

**Mistake 6: Single-answer rigidity.** You insist on a single approach (e.g., "ZooKeeper is always the right choice") without acknowledging trade-offs. The interviewer sees dogmatic thinking. Fix: Present multiple options with a decision framework.

**Mistake 7: Ignoring operational complexity.** You propose a 7-node ZooKeeper cluster for a startup with 2 engineers. The interviewer sees a lack of operational awareness. Fix: Match the operational requirements to the team's capacity.

**Mistake 8: Not having a fallback strategy.** Your design assumes the lock service is always available. The interviewer asks "what if the lock service is down?" and you have no answer. Fix: Always include a degradation strategy such as fail-open, fail-closed, or degraded mode. Explain the trade-offs of each option.

**Mistake 9: Confusing locks with transactions.** You propose using a distributed lock to achieve ACID properties across multiple resources. The interviewer notes that locks are not a substitute for distributed transactions. Fix: Use locks for mutual exclusion and distributed transactions for atomicity. Acknowledge that combining locks and transactions adds complexity.

**Mistake 10: Not knowing the foundational literature.** You are unfamiliar with the Redlock debate, the FLP impossibility result, or the Chubby paper. The interviewer sees a gap in your distributed systems knowledge. Fix: Read the canonical papers on distributed locking before the interview. The Chubby paper, the ZooKeeper paper, and the Redlock critique are essential reading.

**Mistake 11: Premature optimization.** You propose a complex sharded lock service before understanding the scale requirements. The interviewer sees over-engineering. Fix: Start with the simplest solution that meets the requirements and iterate. Complexity should be justified by concrete metrics, not theoretical future needs.

**Mistake 12: Ignoring the resource side.** You design the lock service in detail but do not discuss how the protected resource validates lock authority. The interviewer notes that the resource side is equally important. Fix: Always include the resource-side fencing token validation in your design.

**Prepare for the curveball**: The interviewer may challenge your design with an unexpected failure scenario. Stay calm, think through the failure mode step by step, and propose a mitigation. Acknowledging a weakness in your design and proposing a fix is better than defending an indefensible position.

**Final tip**: Practice the full interview flow with a timer. The distributed lock question typically takes 30 to 45 minutes. Spend the first 5 minutes clarifying requirements, 10 minutes designing the solution, 10 minutes analyzing failure modes, 5 minutes discussing trade-offs, and 5 minutes summarizing. Staying within this time budget shows that you can communicate complex technical ideas efficiently.
