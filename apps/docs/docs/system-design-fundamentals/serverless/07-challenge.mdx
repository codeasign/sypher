#### Challenge 1: Identify the Cold Start Problem

A team deploys a Python-based API to AWS Lambda. The function has 128MB memory, a 3-second timeout, and uses the `requests` and `pandas` libraries. The API endpoint is called once every 10 minutes by a monitoring system. Response times vary between 200ms and 3,500ms.

**Question**: Why does the response time vary so dramatically? What specific changes would you recommend to stabilize it?

**Model Answer**:

The response time variation is caused by cold starts. The function is invoked once every 10 minutes, and the platform's idle sandbox reclamation policy (typically 5-15 minutes of inactivity) means most invocations hit a cold start. The 200ms responses are warm starts — the sandbox was reused from a recent invocation. The 3,500ms responses are cold starts — the platform must provision a new container, download the function code (including the large `pandas` library), initialize the Python runtime, and import all dependencies.

Two changes would fix this:

1. **Increase memory to 512MB or 1,024MB**. Higher memory allocates proportionally more CPU, which reduces initialization time. The `pandas` import alone takes 1-2 seconds on 128MB but 200-400ms on 1,024MB. The cost stays roughly the same because reduced duration compensates for the higher per-GB-second rate.

2. **Use provisioned concurrency with 1 unit**. One provisioned concurrency unit keeps a single sandbox always warm. At $0.00000417 per GB-second for 512MB, provisioned concurrency costs approximately $5.50/month for this function. The cost is negligible for production workloads.

#### Challenge 2: Design a Retry Strategy

A serverless function processes orders from an SQS queue. The function calls an external payment gateway API. Occasionally, the payment gateway returns a 503 Service Unavailable error. When this happens, the function throws an error, and SQS retries the message. After 3 retries, the message goes to the dead-letter queue. The team manually inspects the DLQ daily and replays messages.

**Question**: The team reports that some orders are lost even though the payment eventually succeeds. What is the scenario where this happens? How would you redesign the retry strategy?

**Model Answer**:

The lost order scenario occurs when the payment gateway is temporarily unavailable but recovers within the retry window. The sequence:

1. First attempt: payment gateway returns 503. Message is retried after 1 second.
2. Second attempt: payment gateway still returns 503. Retried after 5 seconds.
3. Third attempt: payment gateway returns 503 again. Message goes to DLQ.
4. Payment gateway recovers 30 seconds later.
5. The message is in the DLQ, and the team checks it once per day.
6. The customer has already been charged (the payment gateway accepted the charge on a retry in a different session), but the order is not created because the Lambda function failed.

The fix: Make the Lambda function idempotent. Before calling the payment gateway, generate a unique idempotency key (based on order ID + timestamp). Store the key in DynamoDB with a conditional write. If the Lambda is retried, it checks the idempotency key first. If the key exists with a "processing" status, return the cached result. This ensures that even if the payment gateway is called multiple times, the charge is only applied once.

Additionally, increase the retry count to 5 with exponential backoff (1s, 5s, 30s, 2min, 5min). This gives the payment gateway more time to recover while the order is still fresh. Set a shorter DLQ inspection interval — 1 hour instead of 24 hours.

#### Challenge 3: Calculate Serverless Capacity

An e-commerce site expects 50,000 requests per second during a flash sale. Each request is processed by a Lambda function with an average duration of 150ms. The function uses 512MB memory. The account concurrency limit is 1,000.

**Question**: Can the current setup handle the flash sale traffic? If not, what changes are needed? Calculate the concurrency requirement and estimate the cost.

**Model Answer**:

**Concurrency calculation**:

```
Concurrency = Requests per second × Average duration (seconds)
Concurrency = 50,000 × 0.15 = 7,500 concurrent executions
```

The current limit of 1,000 concurrent executions cannot handle this. The function would be throttled immediately, and approximately 87% of requests would fail with HTTP 429 errors.

**Changes needed**:

1. **Request a concurrency limit increase** from AWS support. Request a limit of 10,000 for the account and 8,000 for this specific function. This must be done at least 2 weeks before the flash sale.

2. **Use provisioned concurrency** on the critical path. Pre-warm 2,000 concurrent sandboxes before the sale starts. This prevents the initial cold start burst that would occur when 50,000 requests arrive simultaneously.

3. **Consider burst limits**. AWS Lambda's initial burst is 500-3,000 per region. After the burst, it adds 500 per minute. With 2,000 provisioned concurrency, the burst handles the first 2,000 requests. The remaining 5,500 need to scale up at 500/minute = 11 minutes to reach full capacity. This is acceptable for a flash sale if the ramp-up is gradual, but if all 50,000 requests arrive in the first second, a message queue buffer is needed.

4. **Add SQS buffer**. Route API Gateway requests to SQS instead of Lambda directly. The queue absorbs the initial spike. Lambda reads from the queue at a controlled rate, limited by the concurrency settings.

**Cost estimate**:

```
Cost per invocation = Duration × Memory × GB-second rate
= 0.15 × 0.5 × $0.0000166667
= $0.00000125 per invocation

Cost for 50,000 req/s × 3,600 seconds (1 hour)
= 180,000,000 invocations × $0.00000125
= $225.00 per hour

Provisioned concurrency (2,000 units × 1 hour):
= 2,000 × 0.5 × $0.0000041667 × 3,600
= $15.00 per hour

Total estimated cost: $240/hour for the flash sale
```

#### Challenge 4: Design a Serverless Video Processing Pipeline

**Scenario**: A social media platform allows users to upload videos. Each video must be transcoded to multiple resolutions (360p, 720p, 1080p), generate a thumbnail, scan for inappropriate content, and store metadata. Videos range from 10 seconds to 30 minutes. Users upload 100,000 videos per day.

**Requirements**:
- Uploads must be acknowledged within 5 seconds
- Processing must complete within 15 minutes per video
- Duplicate uploads must be detected and skipped
- Failed videos must be retried up to 3 times
- Cost must be optimized for variable traffic (5x daily spikes)

**Design a serverless architecture. Include event flow, storage, processing functions, error handling, and cost considerations.**

**Rubric for Evaluation**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Needs Work (1 pt) |
|----------|-------------------|--------------|-------------------|
| Event flow | Clear end-to-end flow from upload to completion, with every step explained | Flow is mostly complete but has gaps | Missing key steps or unclear dependencies |
| Error handling | DLQ with alarms, retry with exponential backoff, idempotency key | Basic retry with DLQ | No error handling mentioned |
| Scalability | Addresses concurrency limits, burst handling, and parallel processing | Mentions scaling but lacks specifics | No scaling discussion |
| Cost optimization | Breaks even analysis, compares Lambda vs. containers | Mentions cost but no analysis | No cost discussion |
| Complete solution | All requirements addressed with concrete technologies | Most requirements addressed | Key requirements missed |

**Model Design**:

The pipeline uses S3 for uploads, SQS for buffering, and multiple Lambda functions for processing stages. Step Functions orchestrates the workflow.

```
Upload Flow:
1. Client uploads video to S3 pre-signed URL (5-second acknowledgment)
2. S3 "object-created" event triggers validation Lambda
3. Validation Lambda checks file size, format, and computes SHA256 hash
4. Hash is checked against DynamoDB deduplication table (TTL: 7 days)
5. If duplicate, return existing video ID. If new, publish to SQS

Processing Flow (Step Functions):
6. Step Functions reads from SQS and starts execution
   - Parallel: Thumbnail generation (Lambda, 1GB, 30s)
   - Parallel: Transcoding to 3 resolutions (Lambda, 3GB, 5min each)
   - Sequential: Content moderation (Lambda, 512MB, 30s)
   - Sequential: Metadata update (Lambda, 256MB, 5s)
7. Each step handles errors independently
8. On completion, update DynamoDB with video metadata and CDN URLs

Error Handling:
- Each Lambda has a DLQ for unhandled errors
- Step Functions retries on transient errors (exponential backoff: 5s, 30s, 5min)
- Max retries: 3 per step
- After 3 retries, video goes to manual review queue

Cost Optimization:
- Thumbnails and transcoding use 3GB memory for faster processing
- Step Functions adds $0.025 per 1,000 state transitions
- Estimated cost: $0.005 per video (transcoding) + $0.001 per video (Step Functions) + $0.002 per video (thumbnail)
- Total: $0.008 per video × 100,000 videos/day = $800/day
- Alternative: Dedicated transcoding instances = $1,500/day
- Serverless saves 47% and scales automatically
```

#### Challenge 5: Design a Serverless Real-Time Chat API

**Scenario**: A messaging app needs a serverless backend for real-time chat. The system supports 1:1 messaging and group chats (up to 500 users per group). The app has 10 million daily active users sending 1 billion messages per day.

**Requirements**:
- Messages must be delivered within 200ms (p99)
- Message history must be queryable for 30 days
- Group messages must be fanned out to all members
- Online/offline presence must be tracked in real-time
- The system must handle 10x traffic spikes during major events

**Design a serverless architecture. Consider message ingestion, storage, fan-out, and real-time delivery.**

**Rubric for Evaluation**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Needs Work (1 pt) |
|----------|-------------------|--------------|-------------------|
| Real-time delivery | WebSocket API + DynamoDB Streams + Lambda push | Polling-based approach | No real-time mechanism |
| Fan-out | EventBridge + SQS per group member | Single SQS queue | Missing fan-out |
| Presence system | DynamoDB TTL + DynamoDB Streams + Lambda | Cache-based | Not addressed |
| Message storage | Time-series table with TTL, partition key design | Single table | No schema design |
| Performance | Addresses cold starts, provisioned concurrency, DAX | Mentions performance | No performance discussion |

**Model Design**:

The architecture uses API Gateway WebSocket API for real-time messaging, DynamoDB for storage, and EventBridge for fan-out.

```
Message Ingestion:
- Client connects via API Gateway WebSocket API
- Connection ID stored in DynamoDB (presence table, TTL: 5 minutes)
- Message sent as WebSocket frame → API Gateway → Lambda

Message Processing:
1. Lambda receives message, validates, stores in Messages table
2. Message table: PK = conversation_id, SK = timestamp
   TTL = 30 days (auto-expiry)
3. Lambda publishes "message.created" event to EventBridge
4. EventBridge fans out to:
   - Recipient's WebSocket connection (push via ApiGatewayManagementApi)
   - Offline notification queue (SQS, consumed by notification service)
   - Analytics pipeline (Kinesis Firehose → S3 → Redshift)

Group Chat Fan-out:
- EventBridge rule matches group chat pattern
- For each group member, SQS queue receives copy of message
- Lambda reads from queue, pushes to member's WebSocket connection
- Max 500 members per group → 500 SQS messages per group message

Presence Tracking:
- Client sends heartbeat every 30 seconds via WebSocket
- Lambda updates presence table: PK = user_id, TTL = 60 seconds
- When TTL expires, user marked as offline
- DynamoDB Streams trigger Lambda on presence changes
- Presence change published to EventBridge for UI updates

Performance:
- API Gateway WebSocket API: managed, scales to millions of connections
- DynamoDB on-demand: handles 10x spikes without provisioning
- DAX (DynamoDB Accelerator) for message read path: 10x faster reads
- Provisioned concurrency (50 units) for message processing Lambda
- p99 message delivery: 150ms (warm), 2,500ms (cold — mitigated by provisioned concurrency)
```

#### Challenge 6: Design a Multi-Region Serverless Architecture with Strong Consistency

**Scenario**: A financial services company is building a serverless system for international money transfers. The system must handle 10,000 transactions per second, with strong consistency guarantees for account balances. The system operates in 3 regions (US, EU, APAC) with active-active traffic distribution.

**Constraints**:
- Account balances must be strongly consistent within a region
- Cross-region transactions must be eventually consistent with a maximum 5-second delay
- Transactions must be idempotent — duplicate processing must be prevented
- The system must survive a full region failure
- Regulatory compliance requires audit trails for all transactions
- Cold start latency must stay under 100ms for the transaction processing path

**Complexity factors**:
- DynamoDB Global Tables offer eventual consistency across regions
- Strong consistency requires read-after-write within a single region
- Cross-region transactions need a coordination protocol
- Failover must maintain data integrity

**Discussion**:

**Regional Architecture**:

Each region operates independently with its own DynamoDB table for account balances. Within a region, balance reads use strongly consistent reads (DynamoDB's `ConsistentRead: true`). Transactions use DynamoDB transactions for atomicity — a single transfer is a `TransactWriteItems` call that debits one account and credits another atomically.

**Cross-Region Transactions**:

For transfers between regions (e.g., US account to EU account), use a two-phase saga pattern:

1. **Phase 1 (Prepare)**: Debit the source account in the source region. Write a "pending transfer" record with a unique transaction ID to DynamoDB. Set a TTL of 30 seconds on the pending record.

2. **Phase 2 (Complete)**: Publish the transaction ID to EventBridge cross-region event bus. The target region's event bus triggers a Lambda that credits the destination account. On success, mark the pending record as "completed". On failure, the TTL expiry triggers a compensation Lambda that credits the source account back.

This is a saga pattern — each step has a compensating action. The system is eventually consistent with a target of under 5 seconds.

**Cold Start Mitigation**:

Transaction processing is the most latency-sensitive path. Use provisioned concurrency with 200 units per region for the transaction Lambda. This adds approximately $200/month per region but eliminates cold starts. Use SnapStart for Java or Python Lambda to reduce any remaining cold start to under 100ms.

**Failover Strategy**:

Each region operates independently. If a region fails, Route53 health checks redirect traffic to the nearest healthy region. The primary challenge is data — the failed region's DynamoDB table is unavailable. Global Tables handle this automatically: the remaining regions continue operating with the last-replicated data. Pending cross-region transfers from the failed region are retried from the remaining regions using the EventBridge replay feature.

**Audit Trail**:

Every transaction, regardless of success or failure, writes an audit record to a separate DynamoDB table. The audit table uses a write-sharding pattern to distribute writes across partitions. Audit records are also streamed to S3 via Kinesis Firehose for long-term retention (7-year regulatory requirement). The S3 path is partitioned by `region/year/month/day/` for efficient querying.

**Key trade-off**: The strong consistency requirement within a region limits the system to a single DynamoDB table per region. The system cannot horizontally partition account data across tables within a region because DynamoDB transactions are limited to a single table. This means each region's table must handle all transactions for that region's accounts. At 10,000 transactions per second, DynamoDB's partition throughput limits apply — provisioned capacity must be carefully calculated and monitored.

**Capacity planning for this scale**:

Each DynamoDB partition supports 3,000 read units or 1,000 write units. At 10,000 transactions per second (each requiring 2 writes: debit and credit, plus 1 read for balance verification), the write requirement is 20,000 write units per second. DynamoDB auto-scaling adds partitions as needed. The cost at this scale: approximately $5,000-8,000/month per region for DynamoDB.

The Lambda cost: 10,000 transactions/second × 0.5 GB × 0.1 seconds = 500 GB-seconds/second. Per month: 500 × 2,628,000 seconds = 1,314,000,000 GB-seconds × $0.0000166667 = $21,900/month. This is the dominant cost. Provisioned concurrency (200 units × 3 regions): $600/month.

Total monthly cost for the multi-region system: approximately $70,000-85,000/month. This is competitive with a traditional EC2-based deployment when you factor in the operational savings from auto-scaling and zero capacity planning.

**Security and compliance considerations**:

Financial data requires encryption at rest and in transit. DynamoDB encryption at rest uses AWS KMS. Lambda functions use IAM roles with least-privilege permissions — each function only has access to the tables and operations it needs. Audit logs are written to a separate AWS account (audit account) with immutable S3 bucket policy.

The system must pass PCI-DSS and SOC 2 audits. The serverless platform (AWS Lambda, DynamoDB) is already PCI-DSS compliant. The application layer must implement additional controls: input validation, output encoding, access control logging, and session management. Use API Gateway's WAF integration for additional protection against OWASP Top 10 vulnerabilities.

#### Challenge 7: Design a Serverless Event-Sourcing System for an E-Commerce Platform

**Scenario**: An e-commerce platform needs to migrate their order management system to event sourcing. Every order state change (created, paid, shipped, delivered, returned, refunded) must be recorded as an immutable event. The event store must support replay, point-in-time queries, and audit.

**Requirements**:
- 1 million orders per day, each with 5-10 state changes
- Events must be immutable and append-only
- Point-in-time queries (state of order X at time Y)
- Full event replay for bug fixes and analytics
- Event retention for 7 years (regulatory)
- Integration with existing services via event streams

**Discussion**:

The event store is the core of the system. DynamoDB is a natural choice for the event store — each order is a partition key, and events are items sorted by timestamp (SK). However, DynamoDB has a 400KB item size limit, and an order with 50 events over 7 years could exceed this. Use a sharded approach: partition events into chunks of 25 events per DynamoDB item.

**Event store schema**:

Event table:
PK = order_id
SK = event_sequence_number (monotonic increasing integer)
Attributes:
  event_type: string (ORDER_CREATED, PAYMENT_RECEIVED, etc.)
  event_data: map (the event payload)
  timestamp: number (epoch milliseconds)
  version: number (aggregate version)
  metadata: map (correlation ID, user ID, source IP)

Event chunk table (for large event streams):
PK = order_id
SK = chunk_id (e.g., "events_001", "events_002")
Attributes:
  events: list (up to 25 events per chunk)
  start_sequence: number
  end_sequence: number
  event_count: number

**Event sourcing Lambda**:

Append Event Function:
1. Receive command (e.g., "MarkOrderPaid")
2. Load current events for order (strongly consistent read)
3. Rebuild aggregate state from events (fold left)
4. Validate command against current state
5. Generate new event with incrementing sequence number
6. Write event to DynamoDB (conditional write on sequence number)
7. Publish event to EventBridge for downstream consumers
8. Return confirmation with new state

Read Model Function:
1. Load events for order
2. Rebuild state from events
3. Cache state in DAX with TTL of 5 minutes
4. Return current state

Event Replay Function:
1. Scan events by partition (order_id range)
2. Filter by event_type and timestamp range
3. Fold events to rebuild aggregate state
4. State can be compared with current read model for consistency check

**Downstream consumers**:

Each consumer subscribes to EventBridge events independently:
- Order read model updater (maintains current state in DynamoDB for fast reads)
- Notification sender (email, SMS, push updates)
- Analytics pipeline (Kinesis Firehose → S3 → Redshift)
- Inventory updater (adjusts stock levels)
- Fraud detection service (evaluates order patterns)
- Search indexer (updates Elasticsearch for order search)

**Event retention**:

DynamoDB events are retained for 7 years. After 7 years, events are archived to S3 Glacier for long-term retention. The archive process runs as a scheduled Lambda that scans for events older than 7 years, serializes them to Parquet format, and uploads to S3. Archived events are deleted from DynamoDB.

**Testing event sourcing**:

Event sourcing introduces unique testing challenges. Each test scenario should verify:
1. Given a sequence of events, the correct state is rebuilt
2. Given a state and a command, the correct event is generated
3. Invalid commands produce appropriate errors
4. Concurrent event appends are handled correctly (optimistic concurrency)
5. Event replay produces the same state as the original processing
6. Schema evolution is handled (new event types added without breaking existing events)

#### Challenge 8: Serverless Migration Strategy for a 50-Service Microservices Architecture

**Scenario**: A mid-stage startup has 50 microservices running on Kubernetes with 100 nodes. The services handle 500 million API requests per month with 200ms p99 latency. The infrastructure team spends 60% of their time managing Kubernetes. The CTO wants to migrate to serverless to reduce operational overhead.

**Challenges**:
- 50 services, each with different traffic patterns and resource requirements
- Some services are stateful (WebSocket connections, stream consumers)
- Current p99 latency is 200ms — cannot increase
- The migration must be incremental (no big-bang rewrite)
- The team has no serverless experience
- Existing monitoring and alerting is built around Kubernetes

**Discussion**:

This is a complex migration that requires careful planning. The core insight: serverless migration should start with the services that benefit most and have the lowest migration risk.

**Phase 1: Assessment and classification (2 weeks)**

Classify each service into one of three categories:
- **Serverless-ready**: Stateless, event-driven, short-duration, variable traffic. Approximately 25-30 services (notification sending, report generation, file processing, webhook handling, email sending, image resizing, data transformation, authentication token validation).
- **Serverless-compatible with changes**: Services that need minor refactoring — splitting a stateful component, extracting long-running work into a queue. Approximately 10-15 services (order processing after adding SQS buffer, API endpoints after adding provisioned concurrency).
- **Stay on Kubernetes**: Services that fundamentally cannot work on serverless — WebSocket servers for real-time collaboration (unless migrating to API Gateway WebSocket), stream consumers that maintain position in Kafka, services requiring more than 15 minutes of execution time, services with specific hardware requirements. Approximately 5-10 services.

**Phase 2: Quick wins (4 weeks)**

Migrate the first 5 serverless-ready services:
1. Notification sending: The simplest service. Subscribe to EventBridge events, send emails/SMS/push. No state, no latency concerns. Expected migration time: 2 days.
2. Report generation: Scheduled Lambda functions. Generate PDF/CSV reports, upload to S3, send notification. No latency concerns. Expected migration time: 3 days.
3. Image processing: S3-triggered Lambda. Resize, optimize, upload to CDN. Serverless is ideal for this workload. Expected migration time: 4 days.
4. Webhook handling: Receive external webhooks via API Gateway, validate, publish to SQS. Expected migration time: 3 days.
5. Data transformation: ETL jobs triggered by S3 events or EventBridge schedules. Expected migration time: 4 days.

**Phase 3: Core service migration (8 weeks)**

Migrate the serverless-compatible services:
1. Order processing: Add SQS buffer between API Gateway and Lambda. Implement idempotency keys. Set up DLQ monitoring. Expected migration time: 3 weeks.
2. User management API: Split into individual Lambda functions (create user, update user, get user, delete user). Each function has its own memory and timeout settings. Expected migration time: 3 weeks.
3. Payment processing: Extract from monolith. Use Step Functions for the payment workflow. Add provisioned concurrency for the latency-critical path. Expected migration time: 3 weeks.

**Phase 4: Optimization and consolidation (ongoing)**

After migration:
- Right-size memory for all functions (profile at multiple memory levels)
- Set up cross-service tracing with X-Ray
- Implement cost allocation tags per service
- Create serverless deployment pipeline with canary deployments
- Roll back any functions that do not meet the latency SLO

**Expected outcomes**:
- Kubernetes cluster reduces from 100 nodes to 20 nodes (keeping stateful services)
- Infrastructure team time on Kubernetes reduces from 60% to 20%
- Serverless cost increases by 10-20% compared to Kubernetes baseline
- Operational cost decreases by 30-40% (infrastructure team reduction)
- Net savings: 15-25% of total infrastructure + operations spend
