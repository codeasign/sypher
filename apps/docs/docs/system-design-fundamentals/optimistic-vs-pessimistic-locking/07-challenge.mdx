## How to Approach These Challenges

Each challenge lists a scenario, a set of requirements, and specific deliverables. Write your solution as a short design document (4-6 paragraphs per challenge for Tier 1-2, up to 10 paragraphs for Tier 3). Include:

1. Your chosen locking strategy and justification.
2. The relevant code or SQL for the core mechanism.
3. Failure mode analysis: what can go wrong and how you handle it.
4. A simple ASCII diagram of the lock flow (draw it in text using box-drawing characters).

Solutions are available — but attempt each challenge fully before checking. The goal is to build an intuition that survives production incidents.

## Tier 1 — Foundation

### Challenge 1.1: Explain the Lost Update

**Scenario:** A project management app has a task description field. Two team members open the same task at the same time. Both edit the description. Both save. One person's changes vanish.

**Deliver:**

Part A: Write a step-by-step timeline (like the table in the concepts page) showing exactly what happens at each step when no locking is used.

Part B: Show the same timeline with optimistic locking (version column). Include the version check in the UPDATE statement.

Part C: Show the same timeline with pessimistic locking. Include the SELECT FOR UPDATE statement.

Part D: In one paragraph, state which strategy you recommend for this scenario and why.

**Check your work:**

- In Part A, the second save should overwrite the first without any error — that is the lost update.
- In Part B, the version column UPDATE should affect zero rows for the second writer, triggering a retry.
- In Part C, the second SELECT FOR UPDATE should block until the first transaction commits.
- Part D should justify the choice based on collision probability (low for a project management app with hundreds of tasks) and conflict cost (lost edits are frustrating but not financially catastrophic).

### Challenge 1.2: Identify the Deadlock

**Scenario:** Examine this SQL execution sequence. Two transactions run concurrently.

```
Time  | Transaction 1                     | Transaction 2
------+-----------------------------------+--------------------------------
T1    | BEGIN;                            | BEGIN;
T2    | UPDATE accounts SET balance =      | UPDATE accounts SET balance =
      |   balance - 100 WHERE id = 1001;   |   balance - 200 WHERE id = 1002;
T3    |                                    | UPDATE orders SET status =
      |                                    |   'shipped' WHERE id = 5001;
T4    | UPDATE orders SET status =         |
      |   'paid' WHERE id = 5001;          |
T5    | UPDATE accounts SET balance =      | UPDATE accounts SET balance =
      |   balance + 100 WHERE id = 1002;   |   balance + 200 WHERE id = 1001;
T6    | (deadlock detected)                | (deadlock detected)
```

**Deliver:**

Part A: Draw the wait-for graph at time T5. Each node is a transaction. Each edge is "waits for lock held by."

Part B: Explain why the deadlock occurs and which transaction the database will abort.

Part C: Rewrite the transactions to prevent the deadlock by enforcing a consistent lock ordering.

Part D: What happens if the database aborts Transaction 1? What should Transaction 1's application code do?

**Check your work:**

- Part A: T1 holds lock on account 1001 and order 5001, waits for account 1002 (held by T2). T2 holds lock on account 1002, waits for account 1001 (held by T1). Cycle detected.
- Part B: The youngest transaction (or the one with least work, depending on database) is aborted.
- Part C: Lock resources in a fixed global order — for example, always lock lower account ID first, then higher account ID, then the order row. This prevents the cross-lock cycle.
- Part D: The aborted transaction must retry from the beginning, re-acquiring all locks in the fixed order.

### Challenge 1.3: Choose the Strategy

**Scenario:** For each of the following operations, state whether you would use optimistic locking, pessimistic locking, or no locking at all. Provide a one-sentence justification for each.

| Operation | Rows | Transaction Rate | Conflict Cost |
|---|---|---|---|
| A. User profile photo update | 10M users | 100/sec | Low (wrong photo shown briefly) |
| B. Airline seat selection | 200 seats per flight | 5,000/sec during booking | High (two passengers assigned same seat) |
| C. Blog post view counter increment | 10M posts | 10,000/sec | Low (one view lost is negligible) |
| D. Inventory decrement for a flash sale | 50 items | 50,000/sec | High (oversell = angry customers) |
| E. Newsfeed content moderation flag | 100M posts | 200/sec | Medium (wrong post flagged, can revert) |
| F. Payment capture | 1M transactions/day | 50/sec | Extreme (double-charge = legal risk) |
| G. Analytics event append (log) | 1B events/day | 100,000/sec | Zero (append-only, no conflict) |
| H. Multi-player game score update | 10K players | 20,000/sec | Low (leaderboard position, eventual OK) |

**Check your work:**

- A: Optimistic (low contention, low conflict cost).
- B: Pessimistic — SELECT FOR UPDATE on seat row.
- C: Atomic increment: `UPDATE posts SET views = views + 1 WHERE id = X` — no read-modify-write, no lock needed.
- D: Pessimistic with queue-based processing — the collision rate is near 100%.
- E: Optimistic (low collision rate across 100M posts).
- F: Pessimistic with serializable isolation.
- G: No locking (append-only, no update conflict).
- H: Optimistic with retry, or atomic increment if using Redis.

## Tier 2 — Design

### Challenge 2.1: Ticket Booking System

**Scenario:** Design the locking strategy for a ticket booking system for a 500-seat theater. The system handles 10,000 concurrent users during a ticket launch. Each booking holds up to 4 seats.

**Requirements:**

- No seat can be double-booked.
- Users have 10 minutes to complete payment after selecting seats.
- Seats should be released if payment does not complete in 10 minutes.
- The system should handle 3x peak traffic (30,000 concurrent users).
- Bookings are atomic: either all selected seats are booked or none are.

**Deliver:**

Part A: Write the SQL schema (simplified) including any locking-related columns.

Part B: Write the pseudocode for the booking transaction. Include lock acquisition, seat reservation, payment processing, and the timeout mechanism.

Part C: Describe how you handle the 10-minute payment window without holding database locks for the entire duration.

Part D: Design the retry logic for deadlocks and lock timeouts.

Part E: Identify one failure mode specific to this design and how you mitigate it.

**Hints:**

- Use a two-phase approach: temporary reservation (optimistic, short-lived) and final booking (pessimistic).
- The 10-minute window should not involve holding a SELECT FOR UPDATE lock. Consider a reservation table with a TTL.
- The `SKIP LOCKED` clause in PostgreSQL 9.5+ is useful: `SELECT ... FROM seats WHERE ... FOR UPDATE SKIP LOCKED` returns only unlocked rows.

### Challenge 2.2: Inventory Management

**Scenario:** Design the locking strategy for an e-commerce inventory system. The platform has 50,000 SKUs. During normal operation, 500 orders per minute are placed. During Black Friday, traffic spikes to 10,000 orders per minute on 2,000 hot SKUs.

**Requirements:**

- Inventory counts must never go negative (no oversell).
- The system should handle the 20x Black Friday spike without collapsing.
- Low-stock alerts should fire when a SKU drops below 5 units.
- Orders may contain multiple items; all items must be in stock or the entire order is rejected.

**Deliver:**

Part A: State your locking strategy for normal operation vs Black Friday. Justify the difference.

Part B: Write the core SQL for the inventory decrement operation under both normal and peak load.

Part C: Design a contention monitoring system that detects when the collision rate on a SKU exceeds 5% and switches the strategy dynamically.

Part D: How do you handle multi-item orders atomically? Write the transaction pseudocode.

Part E: Describe the retry storm mitigation for Black Friday.

**Hints:**

- For Black Friday, consider queue-based processing: each order is a job in a work queue, and a single worker processes one SKU at a time. This serializes access to hot SKUs without requiring database locks.
- Use a circuit breaker: if the retry rate exceeds 20%, switch the hot SKUs to queue-based mode.
- Multi-item orders should use lock ordering by SKU ID to prevent deadlocks.

### Challenge 2.3: Bank Transfer System

**Scenario:** Design a bank transfer system that handles concurrent transfers between accounts. The system processes 1,000 transfers per second across 10 million accounts. Each transfer reads the source account balance, deducts the amount, and credits the destination account.

**Requirements:**

- Account balances must never go negative.
- Transfers must be atomic: debit and credit both succeed or both fail.
- The system must handle concurrent transfers involving the same accounts.
- A single account may have multiple concurrent incoming and outgoing transfers.
- Daily reconciliation must be possible (all transfers logged permanently).

**Deliver:**

Part A: State your locking strategy and justify it in terms of collision probability and conflict cost.

Part B: Write the transfer transaction SQL, being careful about lock ordering.

Part C: Explain how the lock ordering prevents deadlocks. Show the order rule.

Part D: What is the maximum throughput per account? Derive the formula.

Part E: Design a retry strategy for deadlock errors with exponential backoff.

**Hints:**

- Lock ordering by account ID (always lower ID first) prevents deadlocks.
- For a single account, throughput is limited by lock contention: if each transfer takes 10ms of lock-holding time, a single account can handle at most 100 transfers per second.
- Consider batching: if 100 people each send $1 to the same account, batch them into a single credit transaction.

## Tier 3 — Expert

### Challenge 3.1: Distributed Locking for Global Multi-Region Inventory

**Scenario:** A global e-commerce company has data centers in US-East, EU-West, and Asia-Pacific. Inventory is replicated asynchronously across all regions. Users browse inventory from their local region. When a user adds an item to their cart and checks out, the system must reserve the inventory.

**Requirements:**

- Read latency for inventory queries must be under 50ms (serve from local region).
- Inventory reservations must be globally consistent: the same item cannot be sold in two regions.
- If the network link between regions is down, each region should still be able to sell its locally-allocated inventory.
- Cross-region latency is 70ms (US-EU), 180ms (US-APAC), 250ms (EU-APAC).
- The system handles 5,000 orders per minute globally.

**Deliver:**

Part A: Design the inventory allocation scheme. Propose a primary-region model or a distributed consensus model. Justify your choice with latency and availability numbers.

Part B: Describe the reservation protocol. What happens when a user in Asia-Pacific tries to buy an item whose inventory primary is in US-East?

Part C: What happens during a network partition? How does each region behave? What happens when the partition heals?

Part D: Write the pseudocode for the cross-region inventory reservation. Include the lock acquisition, the timeout handling, and the fallback strategy.

Part E: Identify the worst-case scenario for your design and describe how you would detect and recover from it.

**Hints:**

- Consider sharding inventory by SKU to a primary region, with the primary holding the authoritative lock.
- Use a two-phase reservation with a short-lived lease that expires if the second phase is not completed.
- For partition tolerance, consider an offline allocation model: each region has a pre-allocated pool per SKU that it can sell locally without cross-region coordination.
- AWS DynamoDB global tables with optimistic locking (conditional updates) is one real-world approach.

### Challenge 3.2: Hybrid Optimistic-Pessimistic for a Marketplace

**Scenario:** A peer-to-peer marketplace connects buyers and sellers. Each listing has a price, quantity, and description. The system must handle:

- Sellers updating listing details (price, description, photos).
- Buyers purchasing items (decrement quantity).
- Sellers pausing/reactivating listings.
- A bidding system for auction-style listings.
- A review system where buyers and sellers leave feedback.

**Requirements:**

- 200,000 active listings.
- 10,000 purchases per hour during peak.
- 500 listing edits per hour.
- 50,000 bids per hour during auction events.
- Bidding in the last minute of an auction is time-sensitive: a bid must commit in under 500ms.
- Purchase quantity must never oversell.

**Deliver:**

Part A: Classify each operation by collision probability and conflict cost. Create a table showing which strategy you assign to each operation.

Part B: Design the schema for the listings table including locking-related columns.

Part C: For the auction bidding system, design a lock-free approach that can commit a bid in under 50ms (not 500ms — under 50ms). How do you handle the last-minute bid rush?

Part D: For the purchase operation, design a retry strategy that degrades gracefully: if optimistic retries fail 5 times, the system falls back to pessimistic locking with a queue.

Part E: For the review system, explain why locking is or is not necessary. If not necessary, explain how you prevent conflicting reviews.

## Deliverables Checklist

For each challenge, confirm you have completed:

**Foundation (Tier 1):**
- [ ] Step-by-step timeline with and without locking.
- [ ] Wait-for graph drawn correctly.
- [ ] Strategy justified with collision probability and conflict cost.
- [ ] SQL statements shown for the core locking mechanism.
- [ ] Retry logic described.

**Design (Tier 2):**
- [ ] Schema with locking columns.
- [ ] Transaction pseudocode with lock acquisition and release.
- [ ] Lock ordering rule to prevent deadlocks.
- [ ] Failure mode analysis with specific mitigations.
- [ ] Throughput or capacity calculation.

**Expert (Tier 3):**
- [ ] Primary-region or distributed consensus justification with latency numbers.
- [ ] Partition behavior and recovery plan.
- [ ] Lock-free or queue-based alternative for extreme contention.
- [ ] Fallback strategy from optimistic to pessimistic.
- [ ] Worst-case scenario and detection/recovery mechanism.

## Next Steps

After completing these challenges, you should be able to:

- Identify which locking strategy fits any given workload.
- Write correct SQL for both optimistic and pessimistic locking.
- Design retry logic with proper backoff, jitter, and circuit breaking.
- Analyze failure modes and design mitigations before they cause incidents.
- Design distributed locking for multi-region systems.
- Build hybrid strategies that adapt to changing contention levels.

Review the trade-offs and real-world pages for additional context on any challenge you found difficult.
