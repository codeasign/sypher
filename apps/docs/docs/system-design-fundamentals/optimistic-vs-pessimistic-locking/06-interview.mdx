## How This Topic Appears

Locking and concurrency control questions appear in system design and coding interviews across FAANG and tier-2 companies. The topic is most common in design rounds but surfaces in specialized coding rounds (database internals, distributed systems roles).

### Common Question Phrasings

1. "Design a ticket booking system for a popular concert venue. How do you prevent overselling?"
2. "A social media app has a like counter that sometimes shows incorrect counts. Diagnose and fix."
3. "Design an inventory management system for an e-commerce platform with 10,000 orders per minute."
4. "What happens when two users book the same hotel room at the same time? How do you prevent this?"
5. "Design a bank transfer system that handles concurrent transfers between accounts."
6. "How would you implement optimistic locking in a distributed database with eventual consistency?"
7. "Our checkout system occasionally double-charges customers. Walk through the root cause and fix."
8. "Compare optimistic and pessimistic concurrency control. When would you use each?"
9. "Design a distributed lock service that works across three geographic regions."
10. "A leaderboard updates in real-time during an esports tournament. Scores sometimes disappear. Investigate."

## What the Interviewer Is Testing

Interviewers evaluate four dimensions when they ask about locking:

| Dimension | What They Look For |
|---|---|
| **Understanding of contention** | Do you recognize that not all data needs the same strategy? Can you characterize an operation's collision probability? |
| **Strategy selection** | Given a workload description, can you pick the right locking approach and justify the choice with numbers? |
| **Failure mode awareness** | Do you know what can go wrong? Deadlocks, retry storms, pool exhaustion — can you design mitigations proactively? |
| **System-level thinking** | Can you design a complete solution that integrates locking with the rest of the system (timeouts, monitoring, retry logic, circuit breakers)? |

The interviewer is not testing whether you know the textbook definitions. They want to see whether you can apply concurrency control to an ambiguous, real-world scenario and navigate the trade-offs.

## Level Expectations

### L3 / Entry Level (0-2 years)

**Expectation:** Define pessimistic and optimistic locking. State the basic trade-off. Implement a simple retry loop.

**Sample answer:** "Pessimistic locking means acquiring a lock before accessing the resource so no one else can modify it. Optimistic locking means reading a version number and checking it on write — if it changed, someone else modified it first, so I retry. Pessimistic is safer but slower. Optimistic is faster but requires retries."

**Interviewer assesses:** Basic understanding. Does the candidate know the concepts exist? Can they describe a retry loop in code?

### L4 / Mid-Level (2-5 years)

**Expectation:** Compare strategies with specific criteria. Identify when each is appropriate. Write retry logic with exponential backoff.

**Sample answer:** "For a ticket booking system with 500 seats and 5,000 concurrent users, the collision probability is above 80%, so I would use pessimistic locking with SELECT FOR UPDATE. I would set a lock timeout of 5 seconds and monitor lock wait times. If the wait time exceeds the timeout, the transaction aborts and the user sees a 'sold out' message rather than an infinite spinner."

**Interviewer assesses:** Can the candidate produce a concrete, reasoned recommendation? Do they mention configuration parameters, monitoring, and error handling?

### L5 / Senior (5-8 years)

**Expectation:** Design a hybrid system that selects strategy per operation. Design distributed locking across services. Handle failure modes proactively.

**Sample answer:** "I would classify each write operation by collision rate and conflict cost. Profile updates are low-collision and low-cost — optimistic locking with retry. Inventory decrements are high-collision and high-cost — pessimistic with FOR UPDATE. I would build a contention analyzer that reads collision metrics from the database and can move a hot operation from optimistic to pessimistic automatically. For distributed locking across services, I would use Redis RedLock with a fencing token: each lock grant includes a monotonically increasing token that the resource checks to prevent stale lock holders from writing."

**Interviewer assesses:** Can the candidate design a system that adapts? Do they know about fencing tokens, contention monitoring, and automatic strategy switching?

### L6 / Staff (8-12 years)

**Expectation:** Multi-region distributed locking design. Consensus trade-offs. CAP theorem analysis. Operational runbooks.

**Sample answer:** "For a global multi-region inventory system, I would use a primary-region approach: the US region holds the authoritative lock for North American inventory, the EU region for European inventory. Cross-region inventory transfers use a two-phase commit with a transaction coordinator reading from the region holding the lock. During a network partition, the secondary region cannot modify primary-region inventory — we accept reduced availability over inconsistent inventory counts. I would design a dead man's switch: if the primary region's lock lease expires without a heartbeat, a secondary region can acquire the lock after a grace period."

**Interviewer assesses:** Can the candidate handle global scale? Do they understand the operational implications of their design choices?

### L7 / Principal (12+ years)

**Expectation:** Novel approaches to concurrency control. Influence on database engine design. Trade-off analysis that considers organizational and business constraints.

**Sample answer:** "At this scale, database-level locking is insufficient. I would design an application-level lock service using etcd for linearizable consensus, with a custom sharding layer that assigns lock partitions to regions based on data locality. The system uses speculative execution: when collision probability is low, it optimistically executes both paths and checks for conflicts at commit time. When contention spikes, the lock manager detects the load shift and pre-allocates lock slots. The key insight is that lock granularity should match the hotness gradient of the data — a single hot row should not block an entire partition."

**Interviewer assesses:** Can the candidate invent? Do they understand the limits of existing approaches and have ideas for pushing past them?

## The FAANG Locking Framework

Use this four-step framework in any locking-related system design interview:

### Step 1: Understand Contention Level

Given the workload description, estimate the collision probability:

```
collision_rate ≈ 1 - (1 - 1/rows)^(concurrent_txns × txn_duration)
```

Ask the interviewer for numbers:
- How many rows are in the table?
- How many concurrent users/transactions?
- What is the average transaction duration?
- Are reads or writes more common?

Example: "For a concert with 500 seats and 10,000 people trying to buy tickets simultaneously, with each booking transaction taking 200ms, the collision probability is approximately 1 - (1 - 1/500)^(10,000 × 0.2) which is effectively 100%. Every booking will conflict with others."

### Step 2: Choose Optimistic vs Pessimistic

Apply the decision criteria:

| Contention | Conflict Cost | Recommendation |
|---|---|---|
| Low (< 1%) | Low | Optimistic (version column) |
| Low (< 1%) | High | Optimistic with retry + monitoring |
| Medium (1-5%) | Low | Optimistic with exponential backoff |
| Medium (1-5%) | High | Pessimistic (SELECT FOR UPDATE) |
| High (> 5%) | Any | Pessimistic. At extreme contention, use queue-based processing instead. |

### Step 3: Design Retry Logic

For optimistic locking, design the retry path:

1. Detect conflict (0 rows updated).
2. Wait with exponential backoff + jitter.
3. Re-read current state.
4. Re-compute write.
5. Retry update.
6. After max retries, fail gracefully (queue for async, return error, or use fallback).

```python
def retry_optimistic(max_retries=3, base_delay_ms=50):
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return fn(*args, **kwargs)
                except OptimisticLockException:
                    if attempt == max_retries - 1:
                        raise
                    delay = base_delay_ms * (2 ** attempt)
                    delay += random.randint(0, delay // 2)  # jitter
                    time.sleep(delay / 1000)
            raise Exception("Max retries exceeded")
        return wrapper
    return decorator
```

For pessimistic locking, design the deadlock handling path:

1. Set `lock_timeout` to bound wait time.
2. Catch deadlock errors (error code 40P01 in PostgreSQL, error 1213 in MySQL).
3. Retry the entire transaction after a brief delay.

```python
def retry_on_deadlock(fn, max_retries=3):
    for attempt in range(max_retries):
        try:
            with db.transaction():
                return fn()
        except DeadlockError:
            if attempt == max_retries - 1:
                raise
            time.sleep(random.uniform(0.05, 0.2))  # jittered delay
```

### Step 4: Handle Failure Modes

Proactively address the failure modes from the trade-offs page:

| Failure Mode | Mitigation |
|---|---|
| Deadlock | Automatic detection + retry. Consistent lock ordering. |
| Lock wait timeout storm | Randomized timeouts. NOWAIT for deferrable operations. |
| Connection pool exhaustion | Separate pools for read/write. Short timeouts. |
| Retry storm (optimistic) | Exponential backoff + jitter. Circuit breaker. |
| Lock escalation | Keep transactions small. Monitor escalation events. |
| Distributed lock expiration | Fencing tokens. Lease renewal. Graceful degradation. |

## Phrase Bank

Use these phrases to demonstrate depth during the interview. Each phrase signals a specific area of expertise.

### Contention Analysis

- "The collision probability follows a binomial distribution where P(collision) = 1 - (1 - 1/N)^(C × T) for N rows, C concurrent transactions, and T transaction time."
- "At 80% lock utilization, wait time equals 4x service time due to queueing effects."
- "The deadlock rate grows quadratically with concurrent transactions sharing overlapping lock sets."
- "I would instrument the application to track retry_rate as retry_count divided by total_write_attempts."

### Strategy Selection

- "For this workload, the cost of a conflict outweighs the cost of waiting, so I would prefer pessimistic locking."
- "Optimistic locking optimizes for the common case (no collision) and pays the cost on the rare case."
- "I would classify operations by contention level and conflict cost, then assign a locking strategy per class."
- "A hybrid approach uses optimistic as the default and escalates to pessimistic when the contention analyzer detects a hot row."
- "The switching point is typically around 5% collision rate — below that, optimistic wins on throughput."

### Failure Mode Awareness

- "I would set a lock timeout to prevent connection pool starvation from long-held locks."
- "Exponential backoff with jitter prevents retry storms when multiple clients detect a conflict simultaneously."
- "Fencing tokens prevent the stale lock holder problem in distributed locking — each lock grant carries a monotonically increasing token."
- "Deadlock detection via wait-for graph cycle detection with DFS — the youngest transaction is aborted."
- "A circuit breaker prevents the retry feedback loop from cascading into system-wide failure."

### Implementation Details

- "SELECT FOR UPDATE acquires an exclusive row lock that blocks both writes and other FOR UPDATE reads."
- "MVCC provides snapshot isolation: readers see a consistent snapshot without blocking writers."
- "The version column approach uses `UPDATE ... WHERE id = X AND version = current_version` — zero rows affected means conflict."
- "pg_advisory_lock provides application-defined locks without MVCC overhead — ideal for coordinating non-row resources."
- "In PostgreSQL, deadlock_timeout defaults to 1 second and deadlock detection runs when a lock request enters the wait queue."
- "The innodb_lock_wait_timeout defaults to 50 seconds in MySQL — I would lower it to 5-10 seconds in production."
- "Two-phase locking (2PL) guarantees serializability: acquire all locks in the growing phase, release in the shrinking phase."

### Distributed Systems

- "RedLock provides probabilistic safety — it works well in practice when clock drift is bounded and GC pauses are short."
- "ZooKeeper ephemeral sequential nodes provide linearizable distributed locks with lease-based failure detection."
- "etcd's concurrency package implements the same recipe on top of Raft consensus."
- "Google Spanner uses TrueTime to provide external consistency without distributed locking — timestamps eliminate the coordinator."
- "During a network partition, pessimistic locks favor consistency over availability (CP), while optimistic locks favor availability (AP)."

## Red Flags

The following answers signal to the interviewer that the candidate lacks depth in concurrency control.

1. **"I would just use a database lock."** — No specific strategy named, no analysis of which lock type or isolation level. The interviewer has no evidence the candidate understands trade-offs.

2. **"Optimistic locking is always better because it's faster."** — Ignores retry overhead at high contention. The interviewer will counter with a flash sale scenario and watch the candidate backtrack.

3. **"I would lock the whole table."** — Table-level locks serialize all access. Unless the use case is DDL or bulk operations, this destroys throughput. The candidate has not thought about granularity.

4. **No mention of timeouts.** — Transactions without timeouts can hold locks indefinitely. This suggests the candidate has never debugged a production lock contention incident.

5. **"I would just retry until it works."** — No backoff strategy, no max retry count, no jitter. This causes retry storms. The interviewer knows the candidate will cause production incidents.

6. **"Pessimistic locking prevents all concurrency problems."** — Ignores deadlocks, lock escalation, and pool exhaustion. No system design is free of failure modes.

7. **"I would use SELECT FOR UPDATE for everything."** — Applies the heaviest strategy universally. Ignores the throughput cost for low-contention operations. Shows no ability to measure and adapt.

8. **"Optimistic locking is just adding a version column."** — Misses the entire retry orchestration: backoff, circuit breaking, monitoring, and fallback. The version column is the smallest part of the solution.

9. **No discussion of monitoring.** — "I would add the column and we are done" — No plan to measure collision rates, retry rates, lock wait times, or deadlock frequency. The candidate treats locking as a one-time configuration, not an ongoing operational concern.

10. **"We don't need locking because we use a NoSQL database."** — NoSQL databases still have concurrency control (DynamoDB conditional updates, MongoDB atomic operators). This answer suggests the candidate thinks NoSQL eliminates distributed systems problems entirely.

## Practice Questions by Level

### L3 Practice

**Question:** You are building a simple blog platform. Two authors try to edit the same blog post simultaneously. What goes wrong and how do you fix it?

**Approach:**
1. Identify the lost update problem: both authors read version A, both edit, the second save overwrites the first.
2. Propose optimistic locking: add a `version` column to the `posts` table.
3. Write the retry loop: detect version conflict, re-read, re-apply changes, retry up to 3 times.
4. Mention that if the platform has very high traffic on specific posts, consider pessimistic locking.

### L4 Practice

**Question:** Design an inventory system for a flash sale with 10,000 items and 100,000 concurrent users.

**Approach:**
1. Calculate collision probability: 100,000 concurrent users, 200ms transaction time, 10,000 items. P(collision) is near 100% for hot items.
2. Propose pessimistic locking with FOR UPDATE on inventory rows.
3. Set lock_timeout to 2 seconds to prevent pool starvation.
4. Design a queue-based approach: instead of all 100,000 users hitting the database, push requests through a queue that processes at max throughput of the database (e.g., 500 writes/sec).
5. For multi-item carts, acquire locks in sorted item ID order to prevent deadlocks.

### L5 Practice

**Question:** Design a global multi-region ticket booking system for a world tour.

**Approach:**
1. Primary-region locking: each show's ticket inventory has a single primary region that holds the authoritative lock.
2. Read from local replicas for availability queries (may show stale inventory).
3. Write path for non-primary regions: forward to primary region for lock acquisition.
4. Optimistic fallback for short-lived races: if lock acquisition takes > 500ms, fall back to optimistic with retry (acceptable for < 1% of operations).
5. Fencing tokens: each lock grant includes a sequence number that the inventory row checks on write.
6. Monitoring: track cross-region lock latency, collision rate, and fallback rate.

## Summary

- Locking questions appear in both system design and coding interviews. Common scenarios: ticket booking, inventory, financial transactions, social media counters.
- Interviewers test four dimensions: contention understanding, strategy selection, failure mode awareness, and system-level thinking.
- L3 expectations: definitions and basic retry. L7 expectations: novel approaches and organizational impact.
- The FAANG framework guides your answer: Understand Contention → Choose Strategy → Design Retry → Handle Failure Modes.
- Use the phrase bank to signal depth: measurable terms (collision probability, lock utilization), specific implementations (SELECT FOR UPDATE, pg_advisory_lock), and failure mode vocabulary (retry storms, fencing tokens, pool starvation).
- Avoid the red flags: no blanket statements, no missing timeouts, no one-size-fits-all strategies, no ignoring monitoring.
- Practice by level: start with the lost update explanation (L3), progress to flash sale design (L4), hybrid multi-region system (L5), and global consensus architecture (L6+).

The challenges page provides tiered exercises to test your understanding — from identifying deadlock scenarios to designing a multi-region distributed locking system for a global marketplace.
