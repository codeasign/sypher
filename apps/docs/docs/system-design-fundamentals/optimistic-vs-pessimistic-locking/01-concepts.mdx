## The Problem Story

A customer support team of twelve agents handles incoming cases for a SaaS company. Two agents, Priya and Marcus, both receive a ticket from a high-value enterprise customer reporting a billing discrepancy.

**Priya loads the record at 09:31:14.** She sees the customer's account status is "active" and the last invoice was $4,200. She edits the status to "disputed" and opens the billing system to verify the charges.

**Marcus loads the same record at 09:32:07.** He sees "active" and $4,200 — the same state Priya saw one minute earlier. Marcus notes that the customer called about a duplicate charge. He edits the notes field to add "Investigating duplicate charge — escalate to billing team."

**Priya hits Save at 09:34:22.** The database writes: `status = disputed, notes = "Customer claims overcharge on invoice #4021"`.

**Marcus hits Save at 09:35:01.** The database writes: `notes = "Investigating duplicate charge — escalate to billing team"` — but Priya's `status = disputed` is still in the database because Marcus never loaded the disputed status. Marcus's save does not touch the status field, but his version of the record came from a snapshot before Priya's edit.

The result: the account status shows "disputed," but the customer's detailed notes about the overcharge are gone — overwritten by Marcus's update. Two days later, a different agent picks up the case, sees incomplete notes, and closes it without resolution. The customer escalates to the CEO. The company loses the client.

This is a **lost update** — the classic failure that locking prevents.

---

## The Insight

Concurrency control prevents data corruption when multiple processes access the same data simultaneously. Without it, `read-update-write` cycles collide: Process A reads state S, Process B reads the same state S, both compute a new state from S, and the second write silently destroys the first.

The everyday analogy is collaborative editing:

| System | Behavior | Locking Analogy |
|---|---|---|
| Library checkout | Only one person can borrow a physical book at a time | Pessimistic locking |
| Google Docs | Everyone edits simultaneously; conflicts merge or flag | Optimistic locking |
| Git push | Multiple branches diverge; merge conflict only on push | Optimistic with manual resolution |
| Whiteboard markers | Only one person draws at a time (physical limit) | Pessimistic (hard lock) |

**Pessimistic locking** says: "Assume a conflict will happen. Guard the resource before touching it." You acquire a lock at the start of the operation and release it when done. Every other operation must wait.

**Optimistic locking** says: "Assume no conflict will happen. Record a version marker when you read. When you write, check that the marker hasn't changed. If it has, retry or abort."

The choice depends on the probability of collision, the cost of waiting, and the cost of retrying.

## Core Vocabulary

| Term | Definition |
|---|---|
| Lock | A mechanism that restricts access to a resource to prevent concurrent modification |
| Pessimistic Locking | Lock the resource before accessing it; hold the lock until the transaction completes |
| Optimistic Locking | Read the resource with a version marker; verify the marker on write; retry on mismatch |
| Shared Lock (Read Lock) | Multiple transactions can read simultaneously; no transaction can write |
| Exclusive Lock (Write Lock) | Only one transaction can read or write; all others block |
| Version Number | An integer column incremented on every write; used for optimistic conflict detection |
| Timestamp | A timestamp column checked on write; alternative to version number for optimistic locking |
| Lost Update | Two concurrent writes: the second overwrites the first without incorporating its changes |
| Dirty Read | Reading data written by an uncommitted transaction |
| Phantom Read | A query returns different rows at two points within the same transaction because another transaction inserted or deleted rows |
| Serializable | The strictest isolation level: transactions execute as if they ran one after another |
| Isolation Level | A configurable guarantee about which concurrency anomalies a transaction allows |
| Deadlock | Two transactions each hold a lock the other needs — neither can proceed |

## Step by Step: Two Concurrent Transactions

### Pessimistic Locking Walkthrough

Assume two transactions T1 and T2 operate on inventory row product_id=42 with stock=10.

| Step | T1 | T2 |
|---|---|---|
| 1 | BEGIN | BEGIN |
| 2 | `SELECT stock FROM inventory WHERE id = 42 FOR UPDATE;` — acquires exclusive lock | — |
| 3 | T1 reads stock = 10 | `SELECT stock FROM inventory WHERE id = 42 FOR UPDATE;` — BLOCKS (waits for T1's lock) |
| 4 | T1 computes new stock = 9 | Still waiting |
| 5 | `UPDATE inventory SET stock = 9 WHERE id = 42;` | Still waiting |
| 6 | COMMIT — releases lock | Lock acquired; reads stock = 9 |
| 7 | — | T2 computes new stock = 8 |
| 8 | — | `UPDATE inventory SET stock = 8 WHERE id = 42;` |
| 9 | — | COMMIT |
| 10 | Final stock = 8 | Correct. |

Total wall-clock time: T1 time + T2 time, because T2 waited.

### Optimistic Locking Walkthrough

Same scenario with a version column.

| Step | T1 | T2 |
|---|---|---|
| 1 | BEGIN | BEGIN |
| 2 | `SELECT stock, version FROM inventory WHERE id = 42;` — reads (stock=10, v=1) | `SELECT stock, version FROM inventory WHERE id = 42;` — reads (stock=10, v=1) |
| 3 | T1 computes new stock = 9 | T2 computes new stock = 8 |
| 4 | `UPDATE inventory SET stock = 9, version = 2 WHERE id = 42 AND version = 1;` — succeeds (1 row updated) | `UPDATE inventory SET stock = 8, version = 2 WHERE id = 42 AND version = 1;` — fails (0 rows updated, version mismatch) |
| 5 | COMMIT | T2 detects zero rows affected. Retries: re-reads stock=9, v=2 |
| 6 | — | T2 computes new stock = 8 from latest snapshot |
| 7 | — | `UPDATE inventory SET stock = 8, version = 3 WHERE id = 42 AND version = 2;` — succeeds |
| 8 | — | COMMIT |
| 9 | Final stock = 8 | Correct, but T2 did extra work (retry). |

Total wall-clock time: max(T1, T2) — no waiting, but T2 re-executed its work.

## Variants

### Database-Level Pessimistic Locking

The most common form uses `SELECT ... FOR UPDATE` in SQL:

```sql
BEGIN;
SELECT * FROM accounts WHERE account_id = 1001 FOR UPDATE;
-- Now compute new balance
UPDATE accounts SET balance = balance - 500 WHERE account_id = 1001;
COMMIT;
```

The `FOR UPDATE` clause acquires an exclusive row lock that blocks other `FOR UPDATE` reads and all writes. The lock releases on COMMIT or ROLLBACK.

PostgreSQL and MySQL InnoDB also support `FOR SHARE` (shared/read lock) — multiple transactions can hold `FOR SHARE` on the same row, but none can write until all shared locks release.

### Application-Level Pessimistic Locking (Distributed Lock)

When the resource spans multiple database nodes or services, use an external lock service:

```python
import redis

r = redis.Redis(host="lock-cluster.example.com")
lock_key = "lock:account:1001"

# Acquire lock with 30-second TTL
lock_acquired = r.set(lock_key, "session-abc", nx=True, ex=30)
if lock_acquired:
    try:
        account = db.query("SELECT * FROM accounts WHERE id = 1001")
        account.balance -= 500
        db.save(account)
    finally:
        r.delete(lock_key)  # Release
else:
    raise Exception("Could not acquire lock, try again later")
```

Redis, etcd, and ZooKeeper provide distributed lock primitives. The key challenge is handling lock expiration: if the lock TTL expires before the transaction finishes, a second process grabs the lock and a race condition re-emerges.

### Multi-Version Concurrency Control (MVCC)

MVCC is not a locking strategy per se but an isolation mechanism that underpins both optimistic and pessimistic approaches. Each write creates a new version of the row; readers see a snapshot of committed versions at the time their transaction began.

```sql
-- PostgreSQL MVCC: each row has hidden xmin/xmax system columns
-- xmin = transaction that created this version
-- xmax = transaction that deleted/updated this version (NULL if current)

SELECT *, xmin, xmax FROM inventory WHERE id = 42;
-- id=42, stock=10, xmin=12345, xmax=<NULL>
```

MVCC allows readers never to block writers and writers never to block readers — a critical property for high-concurrency OLTP systems.

### Application-Level Optimistic Locking (Version Column)

Add an integer `version` column to every table that needs concurrency control:

```sql
-- Schema
CREATE TABLE inventory (
    id INTEGER PRIMARY KEY,
    stock INTEGER NOT NULL,
    version INTEGER NOT NULL DEFAULT 1
);

-- Read
SELECT id, stock, version FROM inventory WHERE id = 42;
-- Returns: id=42, stock=10, version=3

-- Write (check version)
UPDATE inventory
SET stock = 9, version = 4
WHERE id = 42 AND version = 3;
-- If 0 rows updated: conflict detected, retry needed
```

The application checks the return value: if zero rows matched the `WHERE version = X` clause, another transaction modified the row between read and write.

### Conditional Update (Timestamp)

Instead of an integer version, use a `last_updated_at` timestamp:

```sql
UPDATE inventory
SET stock = 9, last_updated_at = NOW()
WHERE id = 42 AND last_updated_at = '2025-06-15 14:31:22.123';
```

Timestamps have a subtle flaw: clock skew between servers can cause false conflicts or missed conflicts. Version integers are immune to clock issues and are preferred in distributed systems.

## Mental Model Check

### Question 1

A social media feed loads 50 posts. Each post has a like count displayed on screen. Users click the like button, which increments the count. What locking strategy should the like counter use?

**Answer:** Optimistic locking with a version column, or even a simpler atomic increment. Likes are high-frequency, low-value operations where retries are cheap (`UPDATE posts SET likes = likes + 1 WHERE id = X` — SQL atomic increment avoids the read-modify-write pattern entirely). Pessimistic locking would throttle throughput unnecessarily.

### Question 2

A hospital scheduling system books surgery slots. Two schedulers must never book the same operating room at the same time. What strategy?

**Answer:** Pessimistic locking. The cost of a conflict (double-booked surgery) is catastrophic — patient safety, legal liability. Use `SELECT FOR UPDATE` on the schedule row for the specific OR and time slot. Hold the lock for the duration of the booking transaction. If a conflict is detected at write time (optimistic), the scheduler would have already told the patient a slot was available — unacceptable.

### Question 3

An inventory system manages 100,000 SKUs in a warehouse. Most transactions touch different SKUs. Collisions occur on approximately 0.1% of writes. What strategy?

**Answer:** Optimistic locking. With a 0.1% collision rate, retries are rare. Pessimistic locking would add lock overhead (acquire, hold, release) to 99.9% of transactions that would never conflict. The cost of occasional retries (re-reading and re-computing) is far lower than the latency of always acquiring a lock.

### Question 4

A banking system processes transfers between accounts. Transaction A transfers from Account 1 to Account 2. Transaction B transfers from Account 2 to Account 1 simultaneously. What anomaly can occur?

**Answer:** A deadlock: Transaction A holds a lock on Account 1 and waits for Account 2. Transaction B holds a lock on Account 2 and waits for Account 1. Neither can proceed. The database detects the deadlock (via wait-for graph cycle detection) and aborts one transaction. The application must retry the aborted transaction. The fix: always acquire locks in a consistent order (e.g., lower account ID first).

### Question 5

A configuration management system stores feature flags. Thousands of servers read the configuration every minute. An admin updates a flag once per day. What strategy?

**Answer:** Optimistic locking. The read-to-write ratio is millions to one. Conflicts are practically zero. Pessimistic locking would force every read to check a lock, adding unnecessary latency. A simple version column on the config row handles the rare update collision.

## Key Takeaways

- Pessimistic locking prevents conflicts by serializing access; it trades throughput for safety.
- Optimistic locking detects conflicts at write time; it trades retry overhead for higher throughput under low contention.
- Choose pessimistic when the cost of a conflict exceeds the cost of waiting.
- Choose optimistic when conflicts are rare and retries are cheap.
- MVCC is not a locking strategy but an isolation mechanism that powers both approaches.
- Always have a retry strategy — optimistic locking guarantees conflicts will occur eventually.
- Measure your actual collision rate in production rather than guessing.
