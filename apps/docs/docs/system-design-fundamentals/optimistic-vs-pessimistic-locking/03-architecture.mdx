## Naive Implementation

### No Locking at All

The simplest approach — no locking — works until it does not. A startup with 100 users and a single database instance might never notice lost updates. As the user base grows, the probability of concurrent writes on the same row crosses the threshold from "extremely rare" to "daily incident."

**100 users:** With 10 concurrent transactions per second across 10,000 rows, the probability that two transactions touch the same row in the same millisecond window is approximately 0.001%. Lost updates happen once every few weeks. Nobody notices.

**1,000 users:** 100 transactions per second. Collision probability rises to 0.1%. A lost update occurs every few days. Customer support gets occasional reports of "my edit disappeared." Engineers add optimistic locking to the most critical tables.

**10,000 users:** 1,000 transactions per second. Collision probability reaches 1-2% on high-traffic rows. Multiple lost updates per hour. Data corruption becomes visible: double-charged customers, inventory oversells, overwritten support notes. The system needs locking now.

**100,000 users:** 10,000 transactions per second. Without locking, the system faces continuous data corruption. The `lost update` is no longer a rare anomaly — it is the normal operation pattern. Every hot row is updated by multiple concurrent transactions every second. This is chaos.

## Production-Grade Architecture

A production system uses a **hybrid approach**: optimistic locking for read-heavy, low-contention operations, and pessimistic locking for high-value, high-contention operations. The architecture integrates both within a single application, selecting the strategy per operation based on data criticality and expected contention.

### Operation Classification

Each write operation is classified by two dimensions:

| Dimension | Values | Examples |
|---|---|---|
| Contention level | Low (&lt;1%), Medium (1-5%), High (&gt;5%) | Likes: low. Inventory: high. |
| Conflict cost | Low (retry), Medium (user sees error), High (data corruption, money loss) | Profile update: low. Payment: high. |
| Operation duration | Short (&lt;10ms), Medium (10-100ms), Long (&gt;100ms) | Like: short. Order fulfillment: long. |
| Consistency requirement | Weak (eventual), Strong (serializable), Strict (linearizable) | Feed: weak. Balance: strong. |

The strategy selector:

```
if contention_level == LOW:
    use_optimistic_locking()
elif conflict_cost == HIGH and contention_level == HIGH:
    use_pessimistic_locking(database_level)
elif operation_duration == LONG:
    use_optimistic_locking()  # long locks cause pool starvation
else:
    use_optimistic_locking_with_retry()
```

## Integration Patterns

### ORM Optimistic Locking

Most ORMs support optimistic locking with a version column out of the box.

**Hibernate (Java):**

```java
@Entity
public class Inventory {
    @Id
    private Long id;

    @Version  // Hibernate auto-increments this on every update
    private Integer version;

    private Integer stock;
}
```

Hibernate generates:

```sql
UPDATE inventory
SET stock = ?, version = ?
WHERE id = ? AND version = ?
```

If the update affects zero rows, Hibernate throws `OptimisticLockException`. The application catches it and retries.

**Active Record (Rails):**

```ruby
class Inventory < ApplicationRecord
  # Rails automatically uses the `lock_version` column
  # Add migration: add_column :inventories, :lock_version, :integer, default: 0
end

# Usage
item = Inventory.find(42)
item.stock -= 1
item.save!  # Raises ActiveRecord::StaleObjectError on version conflict
```

Rails increments `lock_version` on every update and includes it in the WHERE clause. The default column name is `lock_version`.

### Redis RedLock for Distributed Pessimistic Locking

When multiple services (not just multiple threads) need to synchronize on a shared resource, a distributed lock is required. Redis RedLock is a well-known (though debated) algorithm for this:

```python
import redis
import time
import uuid

class RedLock:
    def __init__(self, redis_nodes, quorum=None):
        self.nodes = redis_nodes
        self.quorum = quorum or (len(redis_nodes) // 2 + 1)

    def acquire(self, resource, ttl_ms=30000):
        """Try to acquire a distributed lock."""
        lock_value = str(uuid.uuid4())
        start_time = int(time.time() * 1000)
        acquired = 0

        for node in self.nodes:
            try:
                if node.set(resource, lock_value,
                            nx=True, px=ttl_ms):
                    acquired += 1
            except redis.RedisError:
                pass

        # Check if we acquired enough locks and didn't exceed TTL
        elapsed = int(time.time() * 1000) - start_time
        if acquired >= self.quorum and elapsed < ttl_ms:
            self.lock_value = lock_value
            self.resource = resource
            return True

        # Rollback partial acquisitions
        self.release(resource, lock_value)
        return False

    def release(self, resource=None, lock_value=None):
        """Release the lock using a Lua script for atomicity."""
        resource = resource or self.resource
        lock_value = lock_value or self.lock_value

        # Lua script ensures we only delete our own lock
        script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """
        for node in self.nodes:
            try:
                node.eval(script, 1, resource, lock_value)
            except redis.RedisError:
                pass
```

**RedLock controversy:** Martin Kleppmann and Redis author Salvatore Sanfilippo debated whether RedLock provides true mutual exclusion under all failure scenarios (clock jumps, GC pauses). The consensus: RedLock works well enough in practice when:

- Clock synchronization is bounded (NTP with monitoring)
- Lock TTL is generous (5-10x expected operation time)
- A fencing token mechanism is used (monotonically increasing token checked by the resource)

### ZooKeeper / etcd for Leader Election Locks

For leader election (one process acts as coordinator among replicas), ZooKeeper and etcd provide ephemeral, sequential locks:

```
ZooKeeper lock recipe:
1. Create an ephemeral SEQUENTIAL znode under /locks/resource
2. List all children of /locks/resource
3. If your znode has the lowest sequence number → lock acquired
4. Otherwise, watch the znode with the next-lower sequence number
5. When that znode disappears (session ends), re-check
```

etcd provides the `concurrency` package:

```go
import (
    "go.etcd.io/etcd/client/v3/concurrency"
)

func acquireLock(client *etcd.Client, key string) (*concurrency.Mutex, error) {
    session, err := concurrency.NewSession(client)
    if err != nil {
        return nil, err
    }
    mutex := concurrency.NewMutex(session, key)
    if err := mutex.Lock(context.TODO()); err != nil {
        return nil, err
    }
    return mutex, nil
}
```

ZooKeeper and etcd provide stronger safety guarantees than Redis (linearizable writes via Raft consensus), but at the cost of higher latency (5-50ms per lock operation vs 1-5ms for Redis).

### Database Advisory Locks

PostgreSQL provides `pg_advisory_lock()` — application-defined locks that do not depend on database rows:

```sql
-- Acquire an advisory lock (session level)
SELECT pg_advisory_lock(42);  -- 42 is an arbitrary application key

-- Critical section: use the lock to protect application-level resources

-- Release
SELECT pg_advisory_unlock(42);

-- Or use transaction-level advisory locks that auto-release
SELECT pg_advisory_xact_lock(42);
```

Advisory locks are lighter than row locks (no MVCC overhead) and are ideal for coordinating access to application-level resources like cache rebuild operations, schema migrations, or periodic job scheduling.

```python
def rebuild_cache_safely():
    """Only one process rebuilds the cache at a time."""
    with db.cursor() as cur:
        cur.execute("SELECT pg_try_advisory_lock(12345)")
        acquired = cur.fetchone()[0]
        if not acquired:
            return  # Another process already rebuilding

        try:
            rebuild_cache()
        finally:
            cur.execute("SELECT pg_advisory_unlock(12345)")
```

`pg_try_advisory_lock` is the non-blocking variant — it returns `false` immediately if the lock is held, rather than waiting.

## Multi-Region Distributed Locking

When a system spans multiple geographic regions, locking becomes significantly harder. Network latency between regions is 50-200ms (NY to London: ~70ms, NY to Sydney: ~180ms). Every lock operation must cross this latency.

### Distributed Locking Challenges

**Network Latency:** A pessimistic lock acquire across regions takes at least one round trip (RTT). With a three-phase lock protocol (request → grant → confirm), a multi-region lock takes 3x RTT = 150-600ms. This adds unacceptable latency to most user-facing operations.

**Clock Skew:** Timestamp-based optimistic locking assumes clocks are synchronized. Across regions, clock skew can reach 50-100ms (NTP synchronization error). A write from Region A at clock time T1 might arrive at Region B after a write from Region B at clock time T1-20ms, causing a phantom conflict or a missed conflict.

**Consensus Overhead:** Strongly consistent locks require consensus across regions. Paxos and Raft require majority agreement, which means at least one cross-region round trip for every lock operation. This is why Google Spanner (which provides true serializable isolation globally) uses TrueTime — GPS + atomic clocks — to bound clock uncertainty and avoid locking in the happy path.

### Multi-Region Locking Strategies

**Primary-region locking:** Route all writes for a given resource to a single region that holds the authoritative lock. Reads can be served from local replicas. This is the simplest approach: no cross-region coordination for locks, but writes are always remote for non-primary clients.

```
Write path (non-primary region):
  Client → Local API → Forwarded to primary region → DB lock → Write → Acknowledge
  Total latency: 2x cross-region RTT + DB write time (~150-400ms)
```

**Optimistic with conflict resolution:** Accept that conflicts will happen across regions. Use CRDTs (Conflict-Free Replicated Data Types) or application-level merge logic to resolve conflicts after the fact. This is the approach used by DynamoDB global tables.

**Google Spanner's approach:** TrueTime provides bounded clock uncertainty (epsilon = 1-7ms). Transactions use wound-wait or timestamp ordering based on TrueTime, avoiding distributed locks in most cases.

## Capacity Planning

### Contention Math

The collision probability for optimistic locking follows a binomial distribution. For a system with N concurrent transactions, average transaction time T, and R total rows:

```
P(no collision for a given row) = (1 - 1/R)^(N × T)

P(at least one collision) = 1 - (1 - 1/R)^(N × T)

When P(collision) > 5%, switch from optimistic to pessimistic.
```

**Example calculations:**

| Scenario | Rows (R) | Concurrent (N) | Txn Time (T) | Collision Probability |
|---|---|---|---|---|
| Profile updates on 1M users | 1,000,000 | 500 | 100ms | ~0.005% |
| Ticket sales for 500 seats | 500 | 5,000 | 200ms | ~86% |
| Inventory for 50K SKUs | 50,000 | 1,000 | 50ms | ~0.1% |
| Flash sale: 200 items | 200 | 10,000 | 300ms | ~99.999% |

The ticket sales case (86% collision probability) is clearly unsuitable for optimistic locking. The application would waste enormous effort on retries. Pessimistic locking with `SELECT FOR UPDATE` serializes access and provides predictable throughput.

The flash sale (99.999%) is a degenerate case: almost every transaction collides. Even pessimistic locking struggles because the queue length becomes enormous. At this extreme, consider alternative architectures: queue-based processing (one order at a time), batch allocation, or pre-allocation of tickets to time slots.

### Connection Pool Sizing

Connection pool sizing interacts directly with locking behavior. Each connection in the pool can hold at most one lock at a time (assuming a single-threaded connection usage pattern). The pool size limits the maximum number of concurrent lock holders.

**Rule of thumb:**

```
pool_size = num_cores × 2 + effective_spindle_count
           ≈ 10-30 for most applications
```

When pessimistic locking causes transactions to wait for locks, each waiting transaction occupies a connection from the pool. If all connections are waiting for locks and no connection is making progress, the system deadlocks at the connection level. This is **connection pool starvation**.

Mitigation strategies:

1. **Short timeouts:** `lock_timeout` of 5-10 seconds prevents infinite waits.
2. **Queue at the application level:** Use a work queue to limit concurrent lock-holding transactions to fewer than the pool size.
3. **Monitor pool utilization:** Alert when pool usage exceeds 80%.
4. **Separate pools:** Use a small pool for pessimistic operations and a larger pool for lock-free reads.

### Deadlock Monitoring Dashboards

A deadlock monitoring dashboard should track:

| Metric | Source | Alert Threshold |
|---|---|---|
| Deadlocks per second | PostgreSQL `pg_stat_database` → `deadlocks` | > 1/s sustained |
| Average lock wait time | `pg_stat_activity` → `wait_event = 'Lock'` | > 1s |
| Max lock wait time | Query on `pg_locks` with `pg_stat_activity` | > 10s |
| Lock timeout errors | Application error log | > 0.1% of transactions |
| Idle-in-transaction count | `pg_stat_activity` → `state = 'idle in transaction'` | > 5 for > 30s |
| Optimistic retry rate | Application metric: retry_count / total_writes | > 5% |

### Retry Circuit Breaker Pattern

When the optimistic retry rate exceeds a threshold, a circuit breaker prevents cascading failures:

```python
import time
from functools import wraps

class LockRetryCircuitBreaker:
    def __init__(self, failure_threshold=10, reset_timeout_s=30):
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.reset_timeout = reset_timeout_s
        self.last_failure_time = 0
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN

    def __call__(self, fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            if self.state == "OPEN":
                if time.time() - self.last_failure_time > self.reset_timeout:
                    self.state = "HALF_OPEN"
                else:
                    raise CircuitBreakerOpen(
                        "Retry circuit breaker is open"
                    )

            try:
                result = fn(*args, **kwargs)
                if self.state == "HALF_OPEN":
                    self.state = "CLOSED"
                    self.failure_count = 0
                return result
            except OptimisticLockException:
                self.failure_count += 1
                self.last_failure_time = time.time()
                if self.failure_count >= self.failure_threshold:
                    self.state = "OPEN"
                raise
        return wrapper


# Usage
@LockRetryCircuitBreaker(failure_threshold=10, reset_timeout_s=30)
def update_inventory(product_id, delta):
    for attempt in range(3):
        item = db.query("SELECT * FROM inventory WHERE id = %s",
                        product_id)
        new_stock = item.stock - delta
        rows = db.execute(
            "UPDATE inventory SET stock = %s, version = version + 1 "
            "WHERE id = %s AND version = %s",
            new_stock, product_id, item.version
        )
        if rows > 0:
            return
        time.sleep(0.05 * (2 ** attempt))
    raise OptimisticLockException(f"Failed to update {product_id}")
```

## Deployment Considerations

### Blue-Green Deployment with Locking

During a deployment, old and new application versions run simultaneously. Both versions must use compatible locking strategies:

- **Schema changes:** Adding a version column to an existing table is safe (new apps use it, old apps ignore it). Removing a version column is unsafe (old apps still check it).
- **Lock key changes:** If the new version acquires locks on different resources, deadlock risks change. Validate by running both versions against a staging environment with production traffic patterns.

### Gradual Migration

When migrating from no locking to optimistic locking:

1. Add the version column with default value 1.
2. Deploy application code that sets version = version + 1 on updates (but does not check it on read).
3. After one deployment cycle (all writes are maintaining the version), enable the WHERE version = X check.
4. Monitor retry rate for the first 24 hours.

```sql
-- Phase 1: Add column, populate
ALTER TABLE inventory ADD COLUMN version INTEGER NOT NULL DEFAULT 1;

-- Phase 2: Update code to increment version but don't check yet
UPDATE inventory SET stock = 9, version = 2 WHERE id = 42;
-- (No version check in WHERE)

-- Phase 3: Enable optimistic check
UPDATE inventory SET stock = 9, version = 3 WHERE id = 42 AND version = 2;
```

## Summary

- Naive no-locking architectures fail between 1,000 and 10,000 concurrent users as lost updates become frequent.
- Production systems use a hybrid: optimistic for high-throughput/low-contention, pessimistic for critical/high-contention operations.
- ORMs like Hibernate and Rails provide built-in optimistic locking via version columns.
- Distributed locks (Redis RedLock, ZooKeeper, etcd) extend pessimistic locking across services but add complexity, latency, and safety trade-offs.
- Multi-region locking faces fundamental challenges: latency, clock skew, and consensus overhead. Primary-region routing is the simplest solution.
- Collision probability formulas guide strategy selection. At greater than 5% collision rate, switch to pessimistic locking.
- Circuit breakers prevent retry storms from cascading into system-wide failures.
- Deployment of locking changes must be phased to avoid conflicts between old and new application versions.
