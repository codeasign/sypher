## The Core Tension

The fundamental tension in distributed tracing is between completeness and cost. A fully instrumented, no-sampling tracing system provides perfect visibility into every request. But the storage cost for 100% trace fidelity at scale is prohibitive — measured in terabytes per day for a mid-size platform. Every dimension of tracing involves this trade-off.

- **Sampling**: More traces means higher storage cost and slower queries. Fewer traces means missed anomalies.
- **Granularity**: More span attributes means richer debugging context but slower span creation and larger storage.
- **Cardinality**: Indexing high-cardinality attributes enables precise filtering but bloats the index.
- **Freshness**: Real-time indexing means lower throughput and higher cost. Batch indexing means delayed visibility.
- **Retention**: Longer retention supports trend analysis and compliance but multiplies storage cost.

## CAP Theorem Applied to Tracing

Tracing systems face a CAP-like trade-off between trace completeness and system availability. The standard CAP theorem does not apply directly because tracing is not a distributed data store, but the principles of trade-off between consistency, availability, and partition tolerance inform tracing architecture.

**Consistency in tracing** means every span for a trace is present and correctly assembled. An inconsistent trace has missing spans, orphan spans, or incorrect parent-child relationships.

**Availability in tracing** means the tracing system accepts and stores spans without rejecting them. An unavailable tracing system drops spans.

**Partition tolerance** means the tracing system continues to function when network partitions occur between services and collectors, or between collectors and storage.

In practice, tracing systems prioritize availability and partition tolerance over consistency. When a network partition occurs, the span processors in the services continue accepting spans and buffering them. If the buffer fills up, spans are dropped (loss of availability) rather than blocking the application. When the partition heals, spans in the buffer are exported and the trace is assembled from whatever spans exist. The trace might be incomplete, but the system remains operational.

This trade-off means you must accept that some traces will be incomplete. During a partition event, the tracing system will lose spans. The p50 trace completeness (percentage of expected spans that are present) should be above 99% during normal operation. During a partition event, it can drop to 80% or lower. Plan your debugging workflows around the understanding that traces are best-effort, not guaranteed-consistent.

## Failure Modes

### Failure Mode 1: Span Explosion

A single request generates thousands of spans because of excessive instrumentation or deep call chains. A service that loops over 100 database queries and instruments each call creates 100 database spans plus the parent span. If the call chain is 5 services deep and each service makes 100 database calls, a single trace produces 500 spans.

The impact is twofold. First, the trace becomes too large to visualize in the UI, loading slowly or timing out. Second, the span export queue fills up faster, increasing memory pressure on the service.

**Mitigation**: Set maximum span depth limits in the SDK. The OpenTelemetry SDK allows configuring a maximum span limit per trace. Configure it to 100 spans for typical services. Services with known deep call chains can override the limit to 200 spans.

### Failure Mode 2: Context Propagation Loss

A service fails to propagate trace context to downstream services, breaking the trace into disconnected fragments. This happens when a service uses an HTTP client that is not instrumented, or when the developer manually constructs HTTP requests without copying the trace headers.

The symptom is an unusually high number of root spans in the tracing system. If the ratio of root spans to total traces exceeds 20% in a service that is not an entry point, context propagation is broken.

**Mitigation**: Add automated testing that verifies trace context propagation. A simple test starts a trace, makes an HTTP call to a downstream service, and verifies that the downstream service's spans appear as children of the caller's span. Use the OpenTelemetry test SDK to assert parent-child relationships.

### Failure Mode 3: Sampling Mismatch

Head-based sampling and tail-based sampling disagree on which traces to keep. A head sampler at 1% drops a trace, but a downstream service's tail sampler decides the trace is interesting because it errored. The tail sampler's buffer contains the downstream spans but the root span was dropped by head sampling, so the trace is incomplete.

The result is a trace showing only the downstream error spans without the root context. The trace lacks the request URL, the caller service identity, and the original authentication context.

**Mitigation**: Use consistent sampling across the entire pipeline. If you use head-based sampling, configure all services with the same sampling probability and algorithm. If you use tail-based sampling, configure the collector to buffer all spans and make the decision at the collector level.

### Failure Mode 4: Storage Backpressure

The storage backend (Elasticsearch) cannot keep up with the write rate. Write requests are rejected. The collector's batch processor retries with exponential backoff, but the retries accumulate in the queue. When the queue is full, older spans are dropped.

The symptom is a growing gap between the current time and the newest available traces. The indexing delay metric shows 60 seconds, then 120 seconds, then 300 seconds. Traces ingested during this period are permanently lost.

**Mitigation**: Implement a circuit breaker in the collector. When the storage backend's error rate exceeds 10%, the collector stops writing and switches to file-based storage. It writes spans to local files in compressed format. When the storage backend recovers, the collector replays the files. This preserves traces at the cost of temporary unavailability.

### Failure Mode 5: High Cardinality Index Bloat

An attribute with millions of unique values, such as `user_id` or `session_id`, is indexed in Elasticsearch. The inverted index for this field grows to tens of gigabytes. Each new unique value adds to the index. Query performance degrades because the index scan time increases.

The hidden impact is on cluster stability. The large index consumes heap memory in Elasticsearch. As the heap fills, garbage collection pauses increase. Long GC pauses cause the node to drop out of the cluster. Other nodes take over the workload, which increases their memory pressure, creating a cascading failure.

**Mitigation**: Set a cardinality threshold on indexed span attributes. Elasticsearch's `ignore_above` mapping parameter rejects field values longer than a configured length. Use the OpenTelemetry attribute value length limiter to truncate values to 256 characters. Move high-cardinality attributes to span events instead of span attributes; events are stored but not indexed.

## Anti-Patterns

### Anti-Pattern 1: Instrumenting Everything at Maximum Fidelity

The temptation is to instrument every library, every HTTP call, every database query, and every function with detailed attributes. The result is a trace with 500 spans where 95% provide no actionable information. The storage cost is 10 times higher than necessary. The query performance degrades because the index contains noise.

**Instead**: Start with instrumentation for your critical paths only — the request flow, the payment pipeline, the authentication flow. Add instrumentation to secondary paths only when you need to debug them. Use the 80-20 rule: 80% of debugging value comes from 20% of the instrumentation.

### Anti-Pattern 2: Using Trace IDs as Correlation IDs

Some teams use the trace ID as a general-purpose correlation ID across logging, metrics, and billing systems. The trace ID leaks into customer-facing error messages, API responses, and database rows. When the tracing system changes the ID format (for example, switching from 64-bit to 128-bit IDs), all dependent systems break.

**Instead**: Use a separate correlation ID for cross-system coordination. The correlation ID is a stable, application-defined identifier. The trace ID is an internal tracing system identifier. Map between them in the tracing backend if needed.

### Anti-Pattern 3: Synchronous Span Export

The span exporter is configured to send spans synchronously before the response is returned to the client. This adds 2 to 5 milliseconds of latency to every request. At 10,000 requests per second, this wastes 20 to 50 CPU seconds per second on export work.

**Instead**: Always use the batch span processor with asynchronous export. The background thread handles serialization and network I/O. The request thread creates spans and returns immediately. The only cost is the span creation overhead, which is under 1 microsecond.

### Anti-Pattern 4: Unlimited Sampling During Incidents

When an incident occurs, someone increases the sampling rate to 100% on all services to capture every detail of the failure. The tracing system, designed for 1% sampling, receives 100 times the normal throughput. The collectors run out of memory. The Elasticsearch cluster falls behind on indexing. The tracing system itself becomes a casualty of the incident it was supposed to help debug.

**Instead**: Design your tracing system for a maximum sampling rate of 10% to 20%. For high-detail debugging, use on-demand span recording that captures a fixed number of traces on demand, not all traces. The on-demand recording saves the next 100 traces that pass through a service, then stops.

### Anti-Pattern 5: Ignoring Drop Rates

The team deploys tracing and notices that some traces are incomplete but assumes this is normal. The span drop rate is 5% but goes unmonitored because nobody set up alerts for the tracing infrastructure. A slow memory leak in the collector causes the drop rate to climb to 30% over 2 weeks. The team has been operating with a broken tracing system and does not know it.

**Instead**: Monitor the tracing infrastructure as carefully as you monitor your application infrastructure. Alerts for span drop rate, export latency, and indexing delay are as important as alerts for application error rates. The tracing system must be observable.

## When NOT to Use Tracing

Tracing is a powerful tool but not the right tool for every situation.

**Do not use tracing for real-time alerting**. The indexing delay of 10 to 30 seconds means tracing alerts fire noticeably after the actual incident start time. Use metrics for real-time alerting. Metrics are aggregated at the service level and reported every 10 to 60 seconds with sub-second freshness.

**Do not use tracing to replace structured logging**. Logs provide detailed, human-readable context for debugging (stack traces, error messages, variable values). Tracing provides structural context (call paths, latency breakdown, service dependencies). Both are necessary. Use logs for what happened and traces for where it happened in the call chain.

**Do not use tracing for billing or auditing**. The best-effort nature of tracing means spans can be lost. If you need guaranteed, complete records for billing or compliance, use a purpose-built audit logging system that writes to durable storage before responding to the request.

**Do not use tracing for high-frequency, low-value operations**. Health check endpoints, poll requests, and keep-alive pings generate noise in the tracing system without providing debugging value. Configure your SDK to drop spans for these operations using a filter processor.

**Do not use tracing in constrained environments**. Serverless functions with 128 MB of memory, IoT devices with limited CPU, or high-frequency trading systems with microsecond latency requirements cannot afford the overhead of tracing. In these environments, use log-based correlation with manually injected request IDs instead of full tracing.

## Sampling Decision Framework

### When to Choose Head-Based Sampling

Head-based sampling is the right choice when you need predictable resource usage, simple configuration, and statistically representative trace data. Choose head-based sampling when:

- Your system handles more than 10,000 requests per second and the storage budget is limited.
- You need all spans within a trace to be consistently sampled or dropped.
- Your debugging workflows rely on aggregate statistics (percentiles, error rates) rather than individual traces.
- You do not have the infrastructure to buffer all spans for tail-based sampling.

Head-based sampling is also preferred when you are deploying tracing for the first time. The configuration is simpler, and the predictable resource usage reduces the risk of the tracing infrastructure impacting production.

### When to Choose Tail-Based Sampling

Tail-based sampling is the right choice when you need to retain 100% of error traces and slow traces. Choose tail-based sampling when:

- Your system has a low error rate (under 1%) and you need every error trace for debugging.
- You cannot afford to miss rare, high-value traces that head-based sampling would drop.
- You have the infrastructure budget to buffer traces in memory (Kafka, large RAM for collectors).
- Your debugging workflows rely on examining individual traces rather than aggregate statistics.

Tail-based sampling is preferred for critical systems where every error is a potential revenue loss. The additional infrastructure cost is justified by the guarantee of capturing every error trace.

### When to Choose Hybrid Sampling

Hybrid sampling combines head-based and tail-based sampling. Choose hybrid sampling when:

- You need a base sampling rate for normal traffic (head-based) but also want to capture errors at higher rates.
- Your system has a mix of critical and non-critical services.
- You want to start with head-based sampling and later add tail-based sampling for specific services.

Hybrid sampling is the most flexible approach but also the most complex to configure. It requires careful planning of the sampling pipeline and the interaction between head-based and tail-based decisions.

## Storage Tier Trade-offs

### Hot Storage (SSD)

Hot storage uses SSD-backed Elasticsearch nodes for the most recent trace data. The advantages are fast query performance (trace ID lookup under 50 milliseconds) and real-time indexing. The disadvantages are high cost per GB and limited capacity.

Hot storage is appropriate for the most recent 1 to 7 days of trace data. The query latency for hot storage is under 100 milliseconds at p99. The cost is approximately $0.20 per GB per month for managed Elasticsearch.

### Warm Storage (HDD)

Warm storage uses HDD-backed Elasticsearch nodes for older trace data. The advantages are lower cost per GB and higher capacity. The disadvantages are slower query performance (trace ID lookup under 500 milliseconds) and slower indexing.

Warm storage is appropriate for trace data from 7 to 30 days ago. The query latency for warm storage is 200 to 500 milliseconds at p99. The cost is approximately $0.05 per GB per month.

### Cold Storage (Object Store)

Cold storage uses object store (S3, GCS) for archived trace data. The advantages are the lowest cost per GB and unlimited capacity. The disadvantages are very slow query performance (trace ID lookup under 60 seconds) and the need to recall data before querying.

Cold storage is appropriate for trace data older than 30 days. Traces in cold storage are not directly searchable. To query a cold trace, you must know the exact trace ID and request a recall. The recall takes 30 to 60 seconds. The cost is approximately $0.01 per GB per month.

## Operational Complexity Trade-offs

### Self-Hosted vs Managed

Self-hosted tracing (Elasticsearch + Jaeger/OpenTelemetry Collector) provides full control over the infrastructure, data retention, and cost. The advantages are lower cost at scale and complete data ownership. The disadvantages are operational overhead for maintaining the Elasticsearch cluster, the collector fleet, and the Kafka buffer.

Managed tracing (Datadog APM, New Relic, AWS X-Ray) provides a zero-ops experience. The advantages are no operational overhead, automatic scaling, and built-in integrations. The disadvantages are higher cost at scale and vendor lock-in.

The break-even point depends on the span volume. At 100 million spans per day, self-hosted costs approximately $50,000 to $80,000 per year, while managed costs approximately $150,000 to $300,000 per year. At 1 billion spans per day, self-hosted costs approximately $300,000 to $500,000 per year, while managed costs approximately $1 million to $2 million per year.

### Sampling Configuration Complexity

Head-based sampling is simple to configure: set a single percentage value, and the system handles the rest. Tail-based sampling is more complex: configure the buffer size, the policy rules, the retention criteria, and the storage backend. The operational complexity of tail-based sampling is 3 to 5 times higher than head-based sampling.

The complexity trade-off must be weighed against the value of retained traces. For a system with 0.1% error rate, head-based sampling at 1% captures only 10% of error traces. Tail-based sampling captures 100% of error traces. The additional complexity is justified by the 10x improvement in error trace coverage.

## Organizational Impact of Tracing

### Developer Workflow Changes

Introducing tracing changes how developers debug issues. Instead of SSH-ing into a server and reading logs, developers open the tracing UI, filter for the affected operation, and examine the waterfall view. The debugging time for a production issue decreases from hours to minutes.

The organizational challenge is training developers to use tracing effectively. A developer who is used to log-based debugging needs to learn the tracing UI, understand span structure, and practice tracing-based debugging. The learning curve is approximately 2 weeks for most developers.

### On-Call Impact

Tracing transforms the on-call experience. The on-call engineer no longer needs to guess which service is causing a problem. The trace shows the exact service, the exact operation, and the exact error. The mean time to resolution decreases by 40% to 60% for teams that adopt tracing.

The reduction in resolution time is most pronounced for latency issues. A latency issue that previously took 2 hours to debug (checking each service's metrics, guessing the bottleneck, checking logs) takes 15 minutes with tracing (opening the trace, finding the slow span, examining its attributes).

### Cost Allocation

Tracing costs must be allocated to teams. The simplest approach is to allocate cost based on the number of spans per team. Each team's spans are tagged with the team name. The monthly cost per team is proportional to the team's span count.

The challenge is that a team's spans are created by other teams. If the payment service calls the fraud service, the fraud service's spans are created by the fraud team but are necessary for debugging payment issues. The cost allocation should account for both the creating team and the consuming team.

## Precision-Recall Trade-off in Sampling

### Precision vs Recall

Sampling involves a fundamental trade-off between precision and recall. Precision measures the percentage of stored traces that are useful for debugging. Recall measures the percentage of useful traces that are successfully stored.

Head-based sampling at 5% has high precision (every stored trace is a randomly selected representative sample) but low recall (95% of useful traces are dropped). Tail-based sampling with error retention has high recall (100% of error traces are stored) but lower precision (some non-error traces are also stored).

### Measuring Sampling Effectiveness

Track two metrics to measure sampling effectiveness. The useful trace retention rate measures the percentage of traces with an error or latency anomaly that are stored. The storage utilization rate measures the percentage of stored traces that contain an error or latency anomaly.

A well-configured tail-based sampling system achieves a useful trace retention rate of 99% or higher and a storage utilization rate of 80% or higher. A head-based system at 1% achieves a useful trace retention rate of 1% (for normal traces) and a storage utilization rate that mirrors the natural error rate.

### Adaptive Sampling Tuning

The sampling rate should be tuned continuously based on the current error rate. When the error rate is low (under 0.1%), tail-based sampling can retain 100% of error traces with minimal storage overhead. When the error rate spikes to 10% during a database outage, the system should automatically reduce the retention rate for error traces to prevent storage overload.

Implement an adaptive threshold: if the error trace retention causes the storage write rate to exceed 80% of capacity, reduce the error trace retention rate to 50%. When the error rate returns to normal, restore the retention rate to 100%.

## Technology Choice Trade-offs

### OpenTelemetry vs Vendor SDK

OpenTelemetry is the standard for tracing instrumentation. It provides a vendor-neutral API and wide language support. The trade-off is that OpenTelemetry has a larger SDK footprint and more configuration options than vendor-specific SDKs.

Vendor SDKs (Datadog, New Relic, AWS X-Ray) provide simpler setup but tighter coupling to the vendor. Switching vendors requires replacing the SDK and instrumentation. OpenTelemetry decouples instrumentation from backend, enabling vendor switching without code changes.

The recommended approach is to use OpenTelemetry for instrumentation and export to your chosen backend. This gives you the simplicity of a managed backend with the portability of OpenTelemetry.

### Elasticsearch vs Cassandra vs Columnar Storage

Elasticsearch is the most common storage backend for tracing. It provides full-text search on span attributes and fast trace ID lookups. The trade-off is that Elasticsearch has high memory overhead and requires careful index tuning at scale.

Cassandra provides higher write throughput and linear scalability. The trade-off is that Cassandra lacks full-text search capabilities and requires more complex query patterns. Cassandra is better suited for write-heavy tracing workloads at extreme scale.

Columnar storage (ClickHouse, Druid) provides the best compression and query performance for analytical queries on trace data. The trade-off is that columnar stores have higher query latency for single-trace lookups. Columnar storage is better suited for trace analytics than single-trace debugging.

## Technology Maturity Trade-offs

### Open Source vs Managed

Open-source tracing (Jaeger, Zipkin, OpenTelemetry) provides full control over the infrastructure and data. The trade-off is operational effort: you must deploy, configure, monitor, and upgrade the infrastructure. Open-source tracing is best for organizations with dedicated platform engineering teams.

Managed tracing (Datadog APM, New Relic, AWS X-Ray, Google Cloud Trace) provides a zero-ops experience. The trade-off is higher cost at scale and vendor lock-in. Managed tracing is best for smaller teams that cannot justify dedicated platform engineering resources.

The decision depends on the organization's scale and expertise. At 100 million spans per day, a startup should use managed tracing. At 1 billion spans per day, an enterprise with a platform team should use open-source tracing.

### Language Support

OpenTelemetry provides SDKs in all major languages. The maturity of each SDK varies. Java and Go SDKs are the most mature, with the lowest overhead and the widest automatic instrumentation support. Python and Node.js SDKs are mature but have higher overhead. PHP, Ruby, and C++ SDKs are less mature and have limited automatic instrumentation.

The trade-off is that using a less mature SDK requires more manual instrumentation work. If your stack includes PHP or Ruby, you will need to add manual spans for operations that are automatically instrumented in Java or Go. Plan for 2 to 3 weeks of manual instrumentation work for each less-mature SDK.

### API Stability

The OpenTelemetry API is stable for traces since version 1.0. The SDK implementation is stable but continues to evolve. The OpenTelemetry Collector is stable but has frequent minor releases with new processors and extensions.

The trade-off is between using the latest features (requires more frequent upgrades) and using a stable, well-tested version (fewer features). The recommended approach is to use the latest stable release of the SDK and Collector and upgrade every 3 months.

## Failure Mode Recovery Strategies

### Span Explosion Recovery

When a span explosion occurs, the system must recover without losing unrelated traces. The recovery strategy is to reset the SDK's per-trace span counter. The SDK tracks the number of spans per trace. When the limit is reached, the SDK stops creating child spans. The limit is configurable: set it to 100 spans per trace for most services, and 200 for services with deep call chains.

After the span explosion is resolved (the request completes or times out), the SDK resets the counter. The next request starts with a clean slate. The dropped spans are noted in the trace as a span event: "maximum span count reached, 47 child spans suppressed."

### Context Propagation Loss Recovery

When context propagation is broken, the traces are incomplete. The recovery strategy is to identify the service that is not propagating context and add the missing instrumentation. Use the tracing system's data to identify the service: filter for orphan spans and group by service name. The service with the most orphan spans is the one missing context propagation.

The fix is to add the HTTP client instrumentation to that service and deploy. After the fix, the tracing system automatically detects that the orphan span rate has decreased. The target orphan span rate is under 5% after the fix.

### Storage Backpressure Recovery

When Elasticsearch falls behind on indexing, the recovery strategy is to reduce the write load. The collector's circuit breaker detects the backpressure and reduces the sampling rate. The sampling rate is reduced by 50% every 5 minutes until the indexing delay drops below 30 seconds.

After the indexing delay is resolved, the sampling rate is gradually increased. The rate increases by 10% per hour until it returns to the original level. This gradual recovery prevents the storage backend from being overwhelmed again.

### High Cardinality Index Bloat Recovery

When an attribute with high cardinality is indexed, the recovery strategy is to stop indexing that attribute. Elasticsearch supports dynamic mapping updates that change a field from indexed to not-indexed. The mapping update does not require reindexing existing data. New spans after the mapping update are stored but not indexed.

The high-cardinality attribute is identified by querying the Elasticsearch index stats. The field with the highest cardinality is the likely culprit. The mapping update is applied through the Elasticsearch API. After the mapping update, the index growth rate drops to normal levels. The existing data remains searchable through the stored field but is no longer added to the inverted index.

### Collector Failure Recovery

When a collector instance fails, the load balancer routes traffic to healthy instances. The failed collector's in-memory spans are lost. The services connected to the failed collector re-establish connections to healthy collectors within 5 seconds.

The recovery strategy is to run enough collector instances (minimum 2, recommended 3) to absorb the traffic of one failed instance. If one of 3 instances fails, the remaining 2 instances handle 150% of their normal load. The remaining capacity must be sufficient to prevent span drops.

## Vendor Lock-In Risk Assessment

### OpenTelemetry as a Lock-In Prevention

OpenTelemetry is explicitly designed to prevent vendor lock-in. The API is vendor-neutral: the same instrumentation code can export to any backend by changing the exporter configuration. If you use OpenTelemetry, you can switch from Jaeger to Datadog with a configuration change, not a code change.

The risk of lock-in is highest when you use vendor-specific SDK features like Datadog's custom metrics or New Relic's distributed tracing extensions. These features are not available through OpenTelemetry. The solution is to use only OpenTelemetry-standard features and accept the limitation that vendor-specific features are not available.

### Storage Backend Lock-In

The storage backend is harder to change than the instrumentation because the data format is specific to the backend. Elasticsearch, Cassandra, and columnar stores all have different data models. Switching storage backends requires re-indexing all trace data.

Mitigation: Use a storage backend that supports the OpenTelemetry Protocol (OTLP) natively. Elasticsearch with the OTLP plugin, Grafana Tempo, and Jaeger all support OTLP. This ensures that the data format is standard and re-indexing is not required.

### Migration Costs

The cost of migrating from one tracing backend to another includes:
- Instrumentation changes: low cost if using OpenTelemetry (configuration change only)
- Storage migration: high cost (re-indexing all historical traces)
- Query tool changes: medium cost (training on new UI, updating dashboards)
- Alerting changes: medium cost (recreating alert rules)

The total migration cost is typically 2 to 4 weeks of engineering time for a mid-size deployment. The risk of lock-in is worth considering but should not prevent you from choosing a managed tracing service that meets your needs today.

## Latency vs Cost in Sampling Decisions

### High-Volume Service Trade-off

A high-volume service handling 50,000 requests per second faces a stark trade-off. At 5% sampling, the service generates 25,000 spans per second. The monthly storage cost for 30-day retention is approximately $30,000. At 0.5% sampling, the cost drops to $3,000 per month but only 1 in 200 traces is captured.

The decision depends on the service's criticality. A payment service at this volume should use 5% sampling because a single undetected latency regression could cause revenue loss that exceeds the $30,000 storage cost. A recommendation service should use 0.5% sampling because a latency regression affects only user experience, not revenue.

### Low-Volume Service Trade-off

A low-volume service handling 100 requests per second can trace 100% of requests for a monthly storage cost of approximately $100. The trade-off is clear: the cost of 100% tracing is negligible, and the benefit of complete visibility is significant. Every low-volume service should trace 100% of requests.

The threshold for "low volume" depends on the budget. As a rule of thumb: if the monthly storage cost for 100% tracing is under $500, trace 100%. If it is between $500 and $5,000, use 10% sampling. If it exceeds $5,000, use 1% to 5% sampling with error priority.

## Trace Completeness Guarantees

### Best-Effort vs Guaranteed Delivery

Tracing systems are inherently best-effort. Spans can be lost at any stage: the SDK might drop spans when the export queue is full, the collector might drop spans under memory pressure, the network might drop span batches, and the storage backend might reject spans during indexing.

The alternative is guaranteed delivery, which requires a write-ahead log (WAL) at the SDK level. The WAL writes each span to disk before acknowledging it to the application. If the application crashes, the WAL is replayed on restart. The trade-off is significantly higher latency: each span creation requires a disk write, adding 1 to 5 milliseconds per span.

The decision between best-effort and guaranteed delivery depends on the use case. For debugging and performance analysis, best-effort is sufficient. For compliance and auditing, guaranteed delivery is required. Most organizations use best-effort tracing and accept that a small percentage of spans will be lost.

### SLA for Trace Completeness

Define an SLA for trace completeness. A reasonable SLA is: 99% of traces have at least 90% of expected spans present. This accounts for normal span loss due to network issues, queue overflow, and storage backpressure. If the completeness drops below this threshold, the tracing system is considered degraded and an alert fires.

### Monitoring Trace Completeness

Trace completeness is measured by comparing the actual span count per trace to the expected span count. The expected span count is derived from historical data: for a specific operation, the expected span count is the median span count over the past 7 days. If a trace has significantly fewer spans than expected, the span completion rate is reduced.

## Organizational Maturity Model for Tracing

### Level 1: Ad Hoc

At this level, a few services have tracing instrumentation. The instrumentation is inconsistent across services. Some services use OpenTelemetry, others use vendor SDKs, and others have no tracing at all. Traces are incomplete because context propagation is broken between services that use different instrumentation. The organization knows tracing is valuable but has not invested in standardization.

### Level 2: Consistent

At this level, all services use the same tracing SDK (OpenTelemetry) with consistent configuration. Context propagation is verified through integration tests. The tracing pipeline (collector, buffer, storage) is standardized and monitored. The organization has a defined sampling strategy and retention policy. The tracing system is reliable and provides value for debugging.

### Level 3: Proactive

At this level, tracing is integrated into the development workflow. New services must include tracing instrumentation as part of the service template. The CI/CD pipeline verifies trace completeness before deployment. Trace data is used for capacity planning, performance optimization, and proactive anomaly detection. The organization uses tracing to prevent incidents, not just debug them.

### Level 4: Automated

At this level, trace data drives automated actions. The auto-scaling system uses trace data to detect bottlenecks and scale services. The deployment pipeline uses trace data to detect regressions and trigger rollbacks. The alerting system uses trace data to identify the root cause of incidents automatically. The organization has fully integrated tracing into its operational fabric.
