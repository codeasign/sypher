## The Problem Story

You are on-call for a service that processes payments. At 2:47 AM, your pager fires. Customers are reporting that their orders take over 30 seconds to confirm. Some are timing out entirely. You open your monitoring dashboard and see that every individual service reports healthy CPU, memory, and response times. The payment service shows p50 latency of 45 milliseconds. The inventory service shows p50 of 12 milliseconds. The notification service shows p50 of 30 milliseconds. Everything looks normal.

But orders are failing. Why?

You have no way to trace a single request across all the services it touches. You cannot see that the payment service calls the fraud detection service, which calls the customer scoring service, which queries a database that has a contended row lock. You cannot see that the notification service is retrying three times because the email provider is throttling, and each retry blocks the response. You cannot see that the authentication token is being re-validated at every hop instead of being passed through.

This is the problem that distributed tracing solves. Without tracing, a distributed system is a black box. You can measure each component in isolation, but you cannot connect the dots when a single request weaves through dozens of microservices.

Consider a more concrete example. A ride-sharing app has 15 microservices: the mobile app calls the API gateway, which calls the ride-matching service, which calls the pricing service, which calls the surge-pricing service, which calls the demand-forecasting service, which queries a Redis cluster for real-time demand data. The ride-matching service also calls the driver-location service, which calls the geospatial-index service, which runs on a separate database. A single "request ride" operation touches 8 services, 3 databases, and 2 caches. If the end-to-end latency is 4 seconds, which service caused it? Without tracing, you can only guess. With tracing, you see that the geospatial-index service's database query took 2.8 seconds because a new index was missing.

## The Insight

Tracing gives each request a unique identifier and tracks its journey through every service, every database call, every queue, and every cache lookup. Think of it as a GPS tracker for a single user request across your entire infrastructure. Each service records its work as a span, and all spans belonging to the same request are stitched together into a trace.

The key insight is that distributed systems fail at the boundaries, not inside the services themselves. A service responds in 45 milliseconds in isolation, but when it waits for a downstream dependency that is also waiting for another dependency, the chain of waiting compounds. Tracing reveals the real path and timing of every request.

Another important insight is that average-case metrics hide the worst-case behavior. A service might report p50 latency of 50 milliseconds and p99 latency of 200 milliseconds, which looks healthy. But the tracing data shows that 0.5% of requests experience 10-second cascading timeouts because a specific database partition is overloaded. The average hides the tail. Tracing reveals the tail.

## Core Vocabulary

**Trace** — The complete end-to-end record of a single request as it travels through a distributed system. A trace is a tree of spans that share a root trace ID. For example, a trace for "checkout" might include spans for authentication, inventory lookup, payment processing, and notification delivery. A trace captures both the happy path and error paths, including retries and fallbacks.

**Span** — A single unit of work within a trace. Each span records a start time, duration, optional parent span ID, and metadata such as service name, operation name, and tags. A span for "query database" might record the SQL statement, the number of rows returned, and the database host. Spans can be nested (parent-child) or sequential (sibling).

**Trace ID** — A globally unique identifier assigned to a request when it enters the system. Every span in the trace carries this ID so the tracing backend can assemble them into a coherent tree. Trace IDs are typically 128-bit or 64-bit random values encoded as hex strings. The probability of two traces having the same ID is essentially zero at practical scale.

**Span ID** — A unique identifier for a single span. The parent span ID field links a span to its caller, forming the parent-child relationships that define the trace tree structure. Span IDs are 8 bytes (64 bits) and generated randomly by each service.

**Context Propagation** — The mechanism by which trace context (trace ID, span ID, and other metadata) is passed from one service to the next. In HTTP services, propagation happens through headers like `traceparent` and `tracestate` (W3C Trace Context standard). For message queues, context is injected into message headers. For gRPC, context is carried in the metadata.

**Sampling** — The practice of recording only a fraction of traces to control storage costs and processing overhead. Head-based sampling decides at the root whether to trace a request. Tail-based sampling decides after the trace completes, which allows keeping only the most interesting traces (slow, errored, or high-value).

**Span Attributes** — Key-value pairs attached to a span that provide additional context. Common attributes include HTTP method, URL, status code, database statement, cache key, and error details. Attributes are searchable and filterable in the tracing backend. Each attribute has a type: string, integer, double, boolean, or array of these types.

**Span Events** — Timestamped, named annotations within a span that record significant occurrences. For example, a span might emit an event when a cache miss occurs, when a retry is attempted, or when an exception is thrown. Events carry their own set of attributes. Unlike attributes, events are not indexed for search but are stored within the span record.

**Trace Context (W3C Trace Context)** — The industry standard for propagating trace metadata across service boundaries. The `traceparent` header carries the trace ID, parent span ID, and trace flags. The `tracestate` header carries vendor-specific data. Together, they enable interoperability between different tracing systems.

**Service Graph** — A directed graph showing how services connect to each other, derived from trace data. Each node is a service, and each edge represents a traced call from one service to another. The service graph reveals dependencies, critical paths, and potential cascading failure points.

**Root Span** — The first span in a trace, representing the entry point of the request. For a web application, the root span typically covers the full HTTP request handling. The root span has no parent span ID.

**Latency Breakdown** — The distribution of time within a trace across different spans. A latency breakdown shows which service or operation consumed the most time. For example, a 2-second checkout trace might show 1.5 seconds spent in payment processing, 300 milliseconds in inventory lookup, and 200 milliseconds in response serialization.

**Span Kind** — A classification of the span's role in the distributed call. The five span kinds are: INTERNAL (within a single service), SERVER (handling an incoming request), CLIENT (making an outgoing request), PRODUCER (sending a message to a queue), and CONSUMER (receiving a message from a queue). The span kind helps the backend understand the direction of the call.

**Orphan Span** — A span whose parent span ID does not match any known span in the trace. Orphan spans occur when the parent span was dropped by sampling, lost during a network partition, or never created due to a context propagation bug. The tracing backend reports orphan spans as a quality metric.

**Trace Flags** — Bits in the trace context header that control tracing behavior. The most important flag is the sampled flag (bit 0 of the trace flags byte). When set to 1, the trace should be sampled. When set to 0, the trace is not sampled. The sampled flag is propagated through all services, ensuring consistent sampling decisions.

## Step by Step

### Step 1: Request Arrives

A user sends an HTTP POST request to `https://api.example.com/orders`. The load balancer terminates the TLS connection and forwards the request to an API gateway instance. The tracing instrumentation in the load balancer checks for an incoming trace context. Finding none, it generates a new trace ID: `abc123def456`. It creates a root span with ID `span001` and starts the timer. The root span's kind is set to SERVER because it is handling an incoming request.

### Step 2: Context Injection

The load balancer serializes the trace context into HTTP headers. Following the W3C Trace Context standard, it sets the `traceparent` header to the format `00-abc123def456-0000000000000span001-01`. The version prefix `00` indicates the standard format. The `01` flag signals that this trace should be sampled. The `tracestate` header carries vendor-specific metadata. The context is now ready to be passed to downstream services.

### Step 3: API Gateway Processing

The API gateway receives the request. Its tracing middleware extracts the `traceparent` header and retrieves the trace ID and parent span ID. It creates a child span for authentication, setting its parent to `span001`. The gateway validates the JWT token, taking 45 milliseconds. It adds attributes: `http.method: POST`, `http.url: /orders`, `auth.user_id: 42`. The gateway also checks a rate limit, creating a second child span for the rate limit check which takes 3 milliseconds.

### Step 4: Service Boundary Crossing

The gateway calls the payment service over gRPC. Before sending the RPC, it serializes the current trace context into the gRPC metadata. The trace ID `abc123def456` now carries across the network boundary. The payment service extracts the context and creates a new child span for the payment processing operation. The span ID is `span003`, parent is `span002` (the gateway's span). The span kind is set to SERVER because this service is receiving the request, while the gateway's call span is kind CLIENT.

### Step 5: Database Call

Inside the payment service, the code queries the customer database to retrieve account details. The database driver is instrumented with tracing, so it automatically creates a child span of the payment span. This span records the database host, the query text (sanitized to remove PII), and the number of rows returned. The query takes 250 milliseconds. The database span has kind CLIENT because the service is making an outbound call to the database.

### Step 6: Parallel Execution

The payment service makes two calls in parallel: one to the fraud detection service and one to the notification service. Both calls are children of the payment span. Because they run in parallel, their spans overlap in time. The fraud call takes 300 milliseconds and the notification call takes 150 milliseconds. The payment span's duration is determined by the slower of the two parallel calls, which is the fraud call at 300 milliseconds.

### Step 7: Downstream Dependency

The fraud service then calls the customer scoring service, which is unresponsive. The fraud service waits 5 seconds for a response before timing out. This 5-second wait is captured in the fraud span as a span event: `error: timeout`. The fraud service's span also captures the retry attempt: it tries the customer scoring service twice, with each attempt taking 2.5 seconds before timeout. The two retry attempts appear as sequential child spans within the fraud span.

### Step 8: Error Handling

The fraud service returns an error to the payment service. The payment service's span records the error status and attaches a span event: `received_error_from_fraud`. The payment service then executes its fallback logic: it uses a cached risk score from 5 minutes ago. The cache lookup creates a separate child span with kind CLIENT. The cache hit takes 2 milliseconds. The fallback is successful, but the total payment processing time is now 5.5 seconds instead of the normal 300 milliseconds.

### Step 9: Span Finalization

Each service finalizes its span when the operation completes. Finalization records the duration, the status (OK or ERROR), and any final attributes. The span is then sent to the tracing exporter, which batches spans and sends them to the tracing backend over HTTP or gRPC. The exporter typically sends spans in batches every 1 to 5 seconds. The batch processor groups spans by trace ID where possible, to help the backend assemble traces faster.

### Step 10: Trace Assembly

The tracing backend receives spans from all services. An asynchronous pipeline processes incoming spans, indexes them by trace ID, and assembles them into a complete trace tree. The backend must handle out-of-order arrival — spans from the fraud service might arrive before spans from the API gateway because the fraud service finished earlier. The backend buffers spans until all children of a parent have arrived. If a parent span does not arrive within 30 seconds, the backend assumes the trace is complete and assembles the available spans.

### Step 11: Visualization

The completed trace appears in the tracing UI. You see the waterfall view: a horizontal bar chart where each span is a row, sorted by start time, with parent-child relationships shown through indentation. The width of each bar represents the duration. You immediately spot that the fraud service span is 5 seconds long, accounting for nearly all of the 5.5-second total request time. You click on the fraud span to see its attributes: the target service, the timeout duration, and the retry count.

### Step 12: Analysis and Alerting

The tracing system aggregates latency data across all traces. It calculates p50, p95, and p99 latency for each span name. It tags traces that exceed thresholds as anomalies. An alert fires when the p99 latency of the fraud service exceeds 3 seconds for 5 consecutive minutes. You drill into the trace to see the exact request that triggered the timeout. The trace shows that the customer scoring service has a database connection pool exhaustion issue. You route the incident to the database team with the trace ID as evidence.

### Variant 1: Head-Based Sampling

Head-based sampling is the most common approach. The decision to trace a request is made at the entry point before any work begins. The sampling rate is typically a percentage of all requests, such as 5% or 10%. The key advantage is simplicity: the sampling decision is consistent across all services because it is based on the trace ID. A hash of the trace ID modulo 100 determines whether the trace is sampled.

Head-based sampling works well for high-volume systems where you need a statistically representative sample of all traffic. The downside is that you cannot selectively keep rare, high-value traces. If a critical bug surfaces in 0.01% of requests, head-based sampling at 5% might miss it entirely. The probability of capturing a trace that occurs in 0.01% of traffic with a 5% head-based sampler is only 5%. You would need 100% sampling to guarantee capture.

Tail-based sampling collects all spans for all traces in a temporary buffer, then decides after the trace completes whether to keep it. The decision can be based on criteria such as error status, latency exceeding a threshold, or specific user IDs. This allows you to keep 100% of error traces and slow traces while discarding healthy traces.

The trade-off is infrastructure cost. You need enough buffer capacity to hold all traces while they are in flight. For a system processing 100,000 requests per second, the buffer must handle 100,000 concurrent traces. Tail-based sampling also introduces latency: traces become visible only after the trace completes and the sampling decision is made. The typical delay is 1 to 5 seconds, compared to near-real-time for head-based sampling.

### Variant 3: Probability Sampling with Priority

A hybrid approach that combines head-based and tail-based sampling. The head sampler applies a base rate, such as 1% of all requests. But it also allows services to override the sampling decision by setting a priority flag. If a service detects an error, it can mark the span as "keep" and the trace is retained regardless of the head sampling decision.

This approach gives you the cost efficiency of head-based sampling for normal traffic while preserving the ability to capture important traces. The priority flag is propagated through the `tracestate` header, so all services in the trace path respect the override. The priority flag uses a numeric value from 0 to 2: 0 means "use default sampling," 1 means "sampled but no priority," and 2 means "must sample." A span with priority 2 is always retained.

### Variant 4: Edge-Sampled Tracing

Edge-sampled tracing moves the sampling decision to edge proxies or API gateways rather than individual services. The edge proxy applies a consistent sampling policy and injects the decision into the trace context. Downstream services respect the edge decision and do not make their own sampling choices.

This variant simplifies configuration because you define sampling policy in one place. It also reduces overhead because downstream services do not need to evaluate sampling logic. The downside is that the edge proxy becomes a single point of failure for tracing decisions. If the edge proxy is misconfigured, the entire tracing system produces incorrect data.

### Variant 5: Always-On Critical Path Tracing

Some organizations trace 100% of requests on their critical revenue path. The checkout flow, the payment pipeline, and the authentication flow are traced at 100% because the cost of a missed trace during an incident exceeds the storage cost. Non-critical paths like recommendation engines, analytics, and background jobs use lower sampling rates.

This tiered approach acknowledges that not all requests are equally important. A payment failure affecting 1,000 users costs real money. A recommendation engine latency spike affects only user experience, not revenue. The tracing budget is allocated proportionally to business impact.

## Trace Quality Metrics

### Span Completion Rate

The span completion rate measures the percentage of traces that have all expected spans present. A trace with 10 expected spans that only has 9 has a 90% span completion rate. The industry target is above 99% during normal operation. Below 95% indicates a systematic problem with context propagation or span export.

### Trace Assembly Rate

The trace assembly rate measures the percentage of spans that are successfully assembled into complete traces. Orphan spans that cannot be matched to any parent are excluded from the assembly rate. A healthy assembly rate is above 98%. Below 95% indicates context propagation issues.

### Indexing Freshness

Indexing freshness measures the delay between a span being received by the backend and becoming visible in search results. The target is under 30 seconds for p99. Freshness above 60 seconds indicates the storage backend is overwhelmed.

### Span Drop Rate

The span drop rate measures the percentage of spans that are dropped by the SDK or collector before being stored. The target is 0% during normal operation. A drop rate above 0.1% indicates inadequate capacity or configuration issues.

## Mental Model Check

**Question 1**: A trace contains 47 spans. The root span has a duration of 3.2 seconds. The sum of all child span durations is 8.7 seconds. How is this possible?

**Answer**: Spans can overlap in time. Parent spans cover the entire duration of their children, including the time spent waiting for children to complete. The root span starts when the request enters the system and ends when the response is sent. During that time, multiple child spans can execute in parallel. The sum of durations exceeds the root span duration because parallel execution is counted in multiple spans.

**Question 2**: You are designing a tracing system for a platform that processes 500,000 requests per second. Each trace produces an average of 8 spans, each consuming 512 bytes of storage. You want to sample at 1%. How much storage per second does head-based sampling consume?

**Answer**: 500,000 requests per second times 1% sampling equals 5,000 traces per second. Each trace has 8 spans, totaling 40,000 spans per second. At 512 bytes per span, that is 20,480,000 bytes per second, or approximately 20 MB per second. Over 24 hours, that is roughly 1.7 TB of raw span data, not including indexing overhead.

**Question 3**: A trace shows that a database query span has a duration of 2 seconds, but the database itself reports p50 query latency of 50 milliseconds. What might explain this discrepancy?

**Answer**: The database span duration includes network round-trip time, connection pool acquisition time, and any retries. If the connection pool is exhausted, the span might include 1.5 seconds of waiting for a connection. Additionally, if the query is part of a transaction with row locks, the span includes time spent waiting for locks to be acquired. The database-reported p50 latency measures only the query execution time on the server.

**Question 4**: You enable tracing on a production service and notice that the p99 response time increases by 15%. Should you be concerned?

**Answer**: This indicates that the tracing instrumentation itself is adding overhead, likely at the tail end. The p99 increase suggests that tracing is causing contention or allocation pressure in the hot path. You should profile the instrumentation overhead, check for synchronous exporting (which blocks the request), and consider using an asynchronous span processor that batches exports. The tracing overhead should be below 5% at p99 for production use.

**Question 5**: Two spans in the same trace have the same parent span ID but different trace IDs. Is this possible?

**Answer**: No. All spans in a trace must share the same trace ID. If two spans have different trace IDs, they belong to different traces. This situation typically indicates a context propagation bug. A service likely failed to extract the incoming trace context and created a new trace instead of continuing the existing one. This is called a "broken trace" and is a common symptom of missing or misconfigured instrumentation.

**Question 6**: You see that 30% of traces in your system have exactly one root span and zero child spans. What does this indicate?

**Answer**: This indicates that the tracing system is receiving root spans from services that are not entry points. These services are not extracting trace context from incoming requests, so they create new traces instead of continuing existing ones. The single-span traces are broken traces. The most common cause is an uninstrumented HTTP client or missing context propagation configuration.

**Question 7**: Your tracing system stores 10 million spans per day. After enabling tail-based sampling that retains only error traces and slow traces (over 2 seconds), the storage drops to 800,000 spans per day. What percentage of traces were healthy?

**Answer**: The tail sampler retained 800,000 out of 10 million spans, meaning 8% of spans were from error or slow traces. The remaining 92% of spans were from healthy traces that were discarded. This is a typical ratio: the vast majority of traces are normal and provide no debugging value.

## Common Debugging Patterns

### Pattern 1: The Slow Downstream Dependency

A trace shows that Service A calls Service B, and Service B's span accounts for 80% of Service A's total duration. Service B is the bottleneck. You drill into Service B's span and see that it calls Service C, which also accounts for 80% of Service B's duration. The root cause is three levels deep: Service C's database query is slow because of a missing index. The trace reveals the cascading latency that is invisible at the top level.

### Pattern 2: The Retry Loop

A trace shows a service calling a downstream dependency, getting a timeout, retrying, timing out again, and retrying a third time before succeeding. The total duration is 6 seconds, but the actual work took 200 milliseconds. The retry loop is invisible to metrics because the final response is successful. Only the trace reveals the three retries. The fix is to reduce the timeout or implement a circuit breaker.

### Pattern 3: The Connection Pool Exhaustion

A trace shows a database query span with high duration, but the database itself reports low latency. The span attributes show a high "connection acquisition time" of 1.5 seconds. The connection pool is exhausted because the number of concurrent requests exceeds the pool size. The trace reveals that the bottleneck is in the application, not the database.

### Pattern 4: The Cache Stampede

A trace shows a cache miss followed by a slow database query. You look at 100 traces and see that 80 of them have cache misses simultaneously. A popular item expired from the cache, and all concurrent requests hit the database at the same time. The trace reveals the stampede pattern. The fix is to use a mutex or a background refresh strategy.

### Pattern 5: The Cross-Region Latency

A trace shows a request that starts in US-East, calls a service in EU-West, and calls a database in AP-Southeast. The cross-region network latency accounts for 60% of the total duration. The trace reveals the routing inefficiency. The fix is to route the request to the region closest to the user or to replicate data across regions.

## Cost of Tracing

### Storage Cost Breakdown

Tracing storage cost is dominated by span storage and index storage. A single span of 512 bytes uncompressed grows to approximately 1 KB in Elasticsearch because of the inverted index overhead. At 1 billion spans per day, the raw storage requirement is 1 TB per day, and the Elasticsearch storage requirement (with index overhead and replication) is approximately 3 TB per day.

The annual cost for 1 billion spans per day with 30-day retention on Elasticsearch (using AWS i3.2xlarge instances at $0.832 per hour) is approximately $120,000. Reducing the sampling rate from 5% to 1% reduces the annual cost to $24,000.

### Compute Cost for Collectors

Each collector instance handles approximately 2,000 spans per second. A c5.xlarge instance (4 vCPU, 8 GB RAM) at $0.17 per hour can handle 10,000 spans per second if properly configured. The annual cost per collector is approximately $1,500. For a system processing 100,000 spans per second, you need 10 collectors at $15,000 per year.

### Network Egress Cost

Exporting spans from services to collectors and from collectors to the storage backend consumes network bandwidth. A span batch of 512 spans compressed to 200 bytes each is approximately 100 KB per batch. At 100,000 spans per second, that is 200 batches per second, consuming 20 MB per second of network bandwidth. The annual network cost at $0.05 per GB is approximately $30,000.

### Total Cost of Ownership

For a mid-size platform (10 million DAU, 5% sampling, 30-day retention), the annual cost of a self-hosted tracing system is approximately $150,000 to $200,000 for infrastructure. A managed service like Datadog APM or New Relic would cost $300,000 to $500,000 per year at the same scale. The self-hosted option is cheaper but requires operational expertise.

## Adoption Patterns

### Pattern 1: Greenfield Adoption

Starting tracing on a new service is straightforward. Add the OpenTelemetry SDK dependency, configure the exporter endpoint, and enable automatic instrumentation for HTTP, gRPC, and database drivers. The instrumentation works out of the box for standard frameworks. The cost is a single dependency and a few lines of configuration.

### Pattern 2: Incremental Adoption

For existing systems, add tracing incrementally. Start with the entry point services (API gateway, load balancer) and the most critical services (payment, authentication). Add tracing to non-critical services over time. The incremental approach reduces risk and allows you to validate the tracing infrastructure before scaling it.

### Pattern 3: Library-First Adoption

Instead of adding tracing to every service, create a shared library that includes the OpenTelemetry SDK configuration. Every service imports the shared library and gets tracing automatically. The shared library enforces consistent configuration, sampling policies, and exporter settings across all services.

### Pattern 4: Sidecar-Based Adoption

Use a sidecar proxy (Envoy, Linkerd) to add tracing without modifying application code. The sidecar intercepts inbound and outbound HTTP traffic, creates spans, and propagates context. This approach requires no application changes but provides less context (no application-level attributes like database query text).

## Trace Quality Metrics

### Span Completion Rate

The span completion rate measures the percentage of traces that have all expected spans present. A trace with 10 expected spans that only has 9 has a 90% span completion rate. The industry target is above 99% during normal operation. Below 95% indicates a systematic problem with context propagation or span export.

### Trace Assembly Rate

The trace assembly rate measures the percentage of spans that are successfully assembled into complete traces. Orphan spans that cannot be matched to any parent are excluded from the assembly rate. A healthy assembly rate is above 98%. Below 95% indicates context propagation issues.

### Indexing Freshness

Indexing freshness measures the delay between a span being received by the backend and becoming visible in search results. The target is under 30 seconds for p99. Freshness above 60 seconds indicates the storage backend is overwhelmed.

### Span Drop Rate

The span drop rate measures the percentage of spans that are dropped by the SDK or collector before being stored. The target is 0% during normal operation. A drop rate above 0.1% indicates inadequate capacity or configuration issues.

## Tracing vs Logging vs Metrics

These three pillars of observability serve different purposes. Understanding the distinction is critical for designing an effective observability strategy.

### What Each Provides

**Metrics** answer the question "what is happening?" Metrics are numerical aggregations measured over time intervals: request count, error rate, p50/p99 latency, CPU utilization, memory usage. Metrics are lightweight, low-cardinality, and designed for dashboards and alerting. A metric like "p99 latency of payment service" tells you that something is wrong, but not which specific request or which downstream service caused it.

**Logs** answer the question "why did it happen?" Logs are detailed, human-readable records of individual events: error messages, stack traces, request parameters, state transitions. Logs are high-cardinality, verbose, and designed for debugging. A log entry like "ERROR: connection timeout connecting to database-primary-1 after 5000ms" tells you the specific error but not the full request context.

**Traces** answer the question "where did it happen?" Traces are structured records of a request's path through the distributed system. A trace shows the sequence of services, the latency at each hop, and the error at each step. Traces are medium-cardinality, structured, and designed for finding bottlenecks across service boundaries.

### How They Complement Each Other

When debugging a production incident, you use all three in sequence. First, metrics tell you that the p99 latency of the checkout flow increased from 500ms to 5 seconds. This is the alert that wakes you up. Second, traces tell you that the payment service's call to the fraud detection service is responsible for 4 seconds of the 5-second total. You open the trace and see the specific fraud service span that is slow. Third, logs tell you that the fraud service's database connection failed with a specific error message, and the service retried three times before timing out.

### When to Use Each

Use metrics for alerting, dashboards, and capacity planning. A metric alert fires within 10 seconds of a threshold breach. Use traces for debugging specific requests, finding latency bottlenecks, and understanding service dependencies. A trace is available 10 to 30 seconds after the request completes. Use logs for detailed error investigation, audit trails, and debugging complex logic errors. Logs are available within 1 to 5 seconds.

### Storage Costs

Metrics are the cheapest to store. A single metric time series with 1-minute resolution consumes approximately 12 bytes per data point. For 10,000 time series, the storage cost is approximately 10 GB per year.

Logs are more expensive. A single log entry consumes approximately 1 KB. For 10 billion log entries per day, the storage cost is approximately 10 TB per day before compression, and 3 TB per day after compression.

Traces are the most expensive. A single span consumes approximately 512 bytes uncompressed, and each trace has 10 to 20 spans. For a system with 10,000 requests per second at 5% sampling, the storage cost is approximately 40 GB per day after compression.

## Real-World Debugging Example

### Scenario: Payment Failures at a Food Delivery Platform

A food delivery platform processes 50,000 orders per day. At 6:00 PM, the order failure rate jumps from 0.5% to 8%. The on-call engineer is paged.

**Step 1: Check metrics**. The engineer opens the metrics dashboard. The overall order service error rate is 8%. The payment service error rate is 7%. The notification service error rate is 1%. The database CPU is at 60%, which is normal. The engineer narrows the problem to the payment service.

**Step 2: Open traces**. The engineer filters for traces with the order service's checkout operation that have an ERROR status in the last 15 minutes. The tracing system returns 500 traces. The engineer opens the first trace.

**Step 3: Examine the trace waterfall**. The trace shows:
- Root span: order-service.checkout (2.3 seconds, ERROR)
  - Child span: order-service.validate_cart (50ms, OK)
  - Child span: order-service.call_payment_service (2.1 seconds, ERROR)
    - Child span: payment-service.charge (2.0 seconds, ERROR)
      - Child span: payment-service.authorize (1.9 seconds, ERROR)
        - Child span: payment-service.call_bank_api (1.8 seconds, ERROR)
          - Span event: "timeout after 1500ms" at 1500ms
          - Span event: "retry attempt 1" at 1600ms
          - Span event: "timeout after 1500ms" at 3100ms
          - Span event: "retry attempt 2" at 3200ms

**Step 4: Identify the root cause**. The bank API is timing out. The payment service retries twice, but the retries also timeout. The total time is 2.3 seconds, of which 1.8 seconds is waiting for the bank API. The engineer looks at the bank API span attributes: the bank API endpoint is `https://bank-api.prod/v2/authorize`, and the timeout is set to 1500ms.

**Step 5: Check for recent changes**. The engineer checks the deployment timeline. The payment service was deployed 30 minutes ago with a new version that updated the bank API integration. The release notes mention switching from v1 to v2 of the bank API. The tracing system's span attributes show that the new deployment uses the v2 endpoint, while the old deployment used v1.

**Step 6: Roll back**. The engineer rolls back the payment service to the previous version. The error rate drops from 8% to 0.5% within 2 minutes. The engineer creates a ticket to investigate why the bank API v2 integration is failing.

**Step 7: Follow up with trace data**. The engineer attaches the trace ID of a failing request to the ticket. The bank API team uses the trace data to reproduce the issue: the v2 API requires a new authentication header that the payment service is not sending. The trace's span attributes show the request headers, confirming the missing header.

## Common Pitfalls in Tracing

### Pitfall 1: Over-Instrumentation

Instrumenting every function, every loop iteration, and every variable access creates traces with thousands of spans. The trace becomes too large to visualize, and the SDK's memory usage grows with the span count. Over-instrumentation also increases the risk of span explosion, where a single trace consumes more than 1 MB of memory.

**Fix**: Instrument at the service boundary level only. HTTP calls, database queries, and message queue operations are sufficient. Add function-level instrumentation only for specific hot paths that you are actively debugging.

### Pitfall 2: Under-Instrumentation

Instrumenting only the entry point service without instrumenting downstream services produces traces with a single root span and no child spans. The trace shows the total duration but not the breakdown. This is only marginally better than not having tracing at all.

**Fix**: Instrument every service in the request path. If a service is called by other services, it must have tracing instrumentation. Use automatic instrumentation to reduce the effort.

### Pitfall 3: Ignoring Context Propagation

The most common tracing bug is missing context propagation. A service receives a request with trace headers but does not forward them to downstream services. The downstream service creates a new trace, breaking the chain. The result is two disconnected traces instead of one connected trace.

**Fix**: Test context propagation as part of your integration tests. Create a trace, call a downstream service, and verify that the downstream service's spans appear as children of the caller's span.

### Pitfall 4: Storing Too Many Attributes

Adding 50 attributes to every span increases storage by 5x compared to adding 10 attributes. Most attributes are never queried. The storage cost of unnecessary attributes accumulates over time, especially at high span volumes.

**Fix**: Define a standard set of attributes for all spans: HTTP method, URL, status code, service name, and deployment version. Add application-specific attributes only for spans that need them. Use a span processor to strip unnecessary attributes before export.

### Pitfall 5: Not Planning for Scale

A tracing system designed for 100 spans per second will fail at 100,000 spans per second. The collector runs out of memory, the storage backend falls behind on indexing, and spans are dropped. The tracing system becomes unreliable exactly when you need it most.

**Fix**: Design the tracing system for 10x the expected peak throughput. Use capacity planning formulas to calculate collector count, storage capacity, and network bandwidth. Load test the tracing system before production deployment.

## OpenTelemetry SDK Configuration

### Basic Setup

The OpenTelemetry SDK is the standard way to enable tracing in an application. Here is the basic Python setup:

```python
from opentelemetry import trace
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.sdk.resources import Resource

resource = Resource.create({
    "service.name": "payment-service",
    "service.version": "2.1.4",
    "deployment.environment": "production"
})

provider = TracerProvider(resource=resource)
processor = BatchSpanProcessor(
    OTLPSpanExporter(endpoint="http://collector:4317"),
    max_queue_size=2048,
    schedule_delay_millis=5000,
    max_export_batch_size=512,
)
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)
```

The `BatchSpanProcessor` is the most important configuration object. The `max_queue_size` parameter controls how many spans can be buffered in memory during a collector outage. The `schedule_delay_millis` parameter controls how often spans are exported. The `max_export_batch_size` controls how many spans are sent in a single batch.

### Sampling Configuration

The sampler is configured on the tracer provider. The `ParentBased` sampler ensures that the sampling decision is consistent across parent-child span relationships:

```python
from opentelemetry.sdk.trace.sampling import ParentBased, TraceIdRatioBased

sampler = ParentBased(root=TraceIdRatioBased(0.05))
provider = TracerProvider(sampler=sampler)
```

The `ParentBased` sampler delegates to the root sampler for root spans and automatically inherits the sampling decision for child spans. The `TraceIdRatioBased` sampler samples 5% of root spans. Child spans use the same sampling decision as their parent.

### Automatic Instrumentation

Automatic instrumentation hooks into popular frameworks without code changes:

```python
from opentelemetry.instrumentation.flask import FlaskInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.instrumentation.psycopg2 import Psycopg2Instrumentor

FlaskInstrumentor().instrument()
RequestsInstrumentor().instrument()
Psycopg2Instrumentor().instrument()
```

Each instrumentation hooks into the framework's lifecycle. FlaskInstrumentor intercepts incoming HTTP requests, creates server spans, and extracts trace context from headers. RequestsInstrumentor intercepts outgoing HTTP requests, creates client spans, and injects trace context into headers. Psycopg2Instrumentor intercepts database queries and creates database spans.

### Manual Instrumentation

For operations not covered by automatic instrumentation, add manual spans:

```python
tracer = trace.get_tracer(__name__)

with tracer.start_as_current_span("process_refund") as span:
    span.set_attribute("refund_id", refund_id)
    span.set_attribute("refund_amount", amount)
    span.add_event("refund.validation.started")
    # Business logic here
    span.add_event("refund.validation.completed")
    if error:
        span.set_status(trace.Status(trace.StatusCode.ERROR, str(error)))
```

Manual spans capture application-specific operations that automatic instrumentation cannot detect. The `start_as_current_span` context manager automatically sets the span as active, so any child spans created within the block use this span as their parent.

## Tracing in Different Architectures

### Monolith Architecture

In a monolith, tracing is simpler because all operations happen within a single process. The trace spans are created within the same process, context propagation is managed through thread-local storage, and there are no service boundaries. However, tracing is still valuable for understanding the internal call path: which modules call which other modules, and how long each operation takes.

The key difference is that monolith tracing does not require context propagation across network boundaries. The trace ID and span ID are passed through thread-local or async-local context. The trace is assembled within a single process, making it simpler and more reliable.

### Serverless Architecture

Serverless functions (AWS Lambda, Google Cloud Functions) present unique challenges for tracing. A function instance handles a single request, so there is no persistent process to buffer and batch spans. The function must export spans before it terminates, or the spans are lost.

The solution is to use a synchronous exporter that sends spans before the function returns. AWS Lambda supports sending traces to AWS X-Ray through the Lambda runtime API. The OpenTelemetry Lambda layer wraps the function handler and automatically exports spans at the end of each invocation.

Serverless functions also have limited memory and CPU. The tracing SDK must be lightweight. OpenTelemetry provides a simplified SDK for serverless environments that uses less memory and has a smaller initialization time.

### Event-Driven Architecture

Event-driven architectures use message queues and event buses. Tracing across events is challenging because the producer and consumer are decoupled in time. A message might be produced and consumed seconds or hours later.

The trace context is embedded in the message headers. The producer creates a span with kind PRODUCER and injects the context into the message. The consumer creates a span with kind CONSUMER and extracts the context from the message. The resulting trace shows the full path from the original request through the queue.

The time gap between production and consumption appears as a gap in the trace waterfall. The gap is expected and does not indicate a problem. The tracing backend handles time gaps by connecting the producer and consumer spans through the parent-child relationship, regardless of the time difference.

## Advanced Span Concepts

### Span Depth

The depth of a span is the number of ancestors in the trace tree. A root span has depth 0. A span created by a service called by the root service has depth 1. The maximum recommended span depth is 100. Beyond this depth, the trace becomes too large and the SDK's memory usage grows.

Span depth limits are enforced by the SDK. When a span reaches the maximum depth, the SDK does not create additional child spans. Instead, it adds a span event to the deepest span, noting that further spans were suppressed.

### Span Links

Span links connect spans that are related but not in a direct parent-child relationship. For example, a background job that processes a batch of requests might link to each request's trace. The link provides a way to navigate from the job trace to the individual request traces.

Links are different from parent-child relationships. A linked span is not in the direct ancestry of the current span. The link is a pointer to another trace, providing cross-trace correlation.

### Span Events

Span events are timestamped annotations within a span. Unlike attributes, events are not indexed for search. Events are stored in the span record and are visible in the span detail view. Common uses of events include:

- Recording when a retry occurs
- Logging an exception with stack trace
- Noting a cache miss or cache hit
- Recording a state transition (e.g., "order status changed from PENDING to CONFIRMED")

Events are more expensive than attributes because they include a timestamp and a name. Use events sparingly, for information that is not queried in aggregate but is valuable during deep debugging.
