## Internal Architecture

A tracing system consists of five major components: instrumentation libraries, span processors, exporters, the backend pipeline, and the storage layer. Each component has a specific role in the data path from request to visualization.

### Instrumentation Libraries

Instrumentation libraries wrap existing frameworks and protocols to automatically create spans. The OpenTelemetry project provides instrumentation for over 50 libraries and frameworks, including HTTP servers and clients, gRPC, database drivers, message queue clients, and cloud SDKs.

Each instrumentation library hooks into the library's lifecycle. For HTTP servers, the instrumentation intercepts incoming requests, extracts trace context from headers, and creates a server span. For HTTP clients, it injects trace context into outgoing requests and creates a client span. The pair of server and client spans for a single HTTP call are linked through context propagation and form a parent-child relationship where the client span is the parent and the server span is the child.

### Span Processors

Span processors run in the application process. They receive completed spans and can batch, filter, enrich, or drop spans before they are exported. The most common processor is the batch span processor, which accumulates spans into memory and exports them in batches on a configurable schedule, such as every 5 seconds or when 512 spans are accumulated.

A filter processor can drop spans that match certain criteria, such as health check endpoints or spans from non-critical services. An enrichment processor adds environment tags, deployment version, or host information to every span. Processors form a pipeline and run in sequence.

### Exporters

Exporters convert spans from the internal representation to the wire format expected by the backend. The OpenTelemetry Protocol (OTLP) is the standard wire format, supporting both gRPC and HTTP transports. Legacy backends like Jaeger and Zipkin use their own formats: Jaeger uses Thrift over UDP or gRPC, while Zipkin uses JSON over HTTP.

Exporters typically use batching and compression. A batch of 100 spans compressed with gzip might shrink from 50 KB to 8 KB. The exporter sends batches asynchronously on a background thread to avoid blocking the application.

### Backend Pipeline

The backend pipeline receives spans, validates them, indexes them by trace ID, and stores them. The receiver is the entry point, handling HTTP or gRPC connections from exporters. The indexer creates lookup tables mapping trace IDs to the list of spans and timestamps. The tail sampler runs after all spans for a trace have arrived, deciding which traces to retain based on configurable rules.

### Storage Layer

Tracing backends use multi-tier storage. Hot storage holds recent traces for fast queries. Warm storage holds traces for the past 30 days with slower query performance. Cold storage archives traces beyond 30 days, typically in an object store like S3, with recall times measured in minutes.

Elasticsearch is the most common hot storage backend for open-source tracing systems. It indexes spans by trace ID, service name, operation name, and tags, enabling fast filtered queries. Some commercial systems use dedicated columnar storage engines optimized for trace data, achieving query latencies under 100 milliseconds for the 99th percentile.

## Data Structures and Algorithms

### Span Data Structure

Every span is a structured record with these fields:

- **trace_id**: 16-byte binary identifier
- **span_id**: 8-byte binary identifier
- **parent_span_id**: 8-byte binary identifier (null for root spans)
- **name**: operation name like "HTTP POST /orders"
- **kind**: SPAN_KIND_INTERNAL, SERVER, CLIENT, PRODUCER, CONSUMER
- **start_time**: nanosecond-precision Unix timestamp
- **end_time**: nanosecond-precision Unix timestamp
- **attributes**: key-value pairs with typed values (string, int, double, boolean, array)
- **events**: list of timestamped named events with attributes
- **status**: OK, ERROR, or UNSET with optional description
- **resource**: metadata about the source, like service name and version
- **scope**: identifies the instrumentation library that created the span

### Trace Tree Assembly

The backend assembles spans into a trace tree using a hash map keyed by span ID. The algorithm processes spans in batches:

```python
def assemble_trace(spans):
    span_map = {}
    root_spans = []

    # First pass: index all spans by span_id
    for span in spans:
        span.children = []
        span_map[span.span_id] = span

    # Second pass: build parent-child relationships
    for span in spans:
        if span.parent_span_id is None:
            root_spans.append(span)
        elif span.parent_span_id in span_map:
            parent = span_map[span.parent_span_id]
            parent.children.append(span)
        else:
            # Orphan span: parent not yet arrived
            span_map[span.parent_span_id] = PendingSpan()
            pending_children[span.parent_span_id].append(span)

    return root_spans
```

The assembly must handle out-of-order arrival. A child span might arrive before its parent, or a span might arrive while its parent is still in flight. The algorithm buffers orphan spans until their parent arrives, using a pending-children hash map. If a parent does not arrive within a configurable timeout, the orphan spans are reported as a broken trace fragment.

### Span ID Generation

Span IDs are generated randomly using a cryptographically secure pseudo-random number generator. The probability of collision for 8-byte span IDs is approximately `N^2 / 2^65`, where N is the number of active spans. For a system with 1 million concurrent spans, the collision probability is roughly 1 in 36 billion.

Trace IDs are 16 bytes when generated by applications that follow the W3C Trace Context standard. The extra length reduces collision probability further and enables coordination with other systems that might generate IDs independently.

### Trace Query Index

The backend builds a secondary index for fast trace queries. The index maps service names, operation names, and attribute values to the set of matching trace IDs. Queries like "find all traces where payment-service.process returned ERROR in the last hour" use this index to narrow the search space.

The index is implemented as an inverted index similar to a search engine. Each unique attribute value points to a sorted list of trace IDs with timestamps. Range queries on time are supported through the timestamp ordering. The index is updated asynchronously with a typical freshness latency of 10 to 30 seconds.

## Request Lifecycle

### Step 1: Initialization

The application starts and initializes the OpenTelemetry SDK. The SDK creates a tracer provider, which manages tracer instances. The tracer provider is configured with a span processor pipeline and an exporter. The initialization registers instrumentation libraries for HTTP, gRPC, and the database driver.

```python
from opentelemetry import trace
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor

provider = TracerProvider()
processor = BatchSpanProcessor(
    OTLPSpanExporter(endpoint="http://collector:4317"),
    max_queue_size=2048,
    schedule_delay_millis=5000,
)
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)
```

### Step 2: Context Propagation Setup

The SDK initializes the context propagation infrastructure. The W3C Trace Context propagator is registered to handle both injection (serializing context into outgoing requests) and extraction (deserializing context from incoming requests).

### Step 3: Incoming Request Handler

A POST request arrives at Service A. The HTTP server instrumentation intercepts the incoming request. It calls the registered propagator's extract method on the request headers. Finding no existing trace context, the propagator returns a new root context. The instrumentation creates a root span with a new trace ID and span ID. The span starts with the current timestamp.

### Step 4: Business Logic Execution

Service A executes its business logic. The code makes two outgoing calls in parallel: one to Service B over HTTP and one to the database. The tracer provider's active tracer creates child spans for each operation. The SDK's context management ensures that the active span is available through thread-local storage or async context variables.

### Step 5: Outgoing HTTP Call

Service A constructs the HTTP request to Service B. The HTTP client instrumentation calls the propagator's inject method, which sets the `traceparent` and `tracestate` headers on the outgoing request. The client span is created with parent set to the currently active span. The span records the HTTP method, URL, and request headers (with sensitive headers redacted).

### Step 6: Database Query

The database driver instrumentation intercepts the SQL query execution. It creates a database span as a child of the active span. The span records the database connection string (with credentials stripped), the SQL statement (parametrized to remove literal values), and the number of rows affected. The driver's instrumentation hooks into the connection pool as well, creating separate spans for connection acquisition and query execution.

### Step 7: Response Handler at Service B

Service B receives the HTTP request. Its instrumentation extracts the trace context from the headers and creates a server span as a child of Service A's client span. The parent span ID in Service B's span matches the span ID of Service A's client span. Service B processes the request, calling the inventory service and the cache.

### Step 8: Cache Lookup

Service B checks the cache before querying its database. The cache client instrumentation creates a span for the cache get operation. If the cache returns a hit, the span records the cache key and hit status. If the cache returns a miss, the span records the miss and the subsequent database query latency. The cache span is a sibling of the database span.

### Step 9: Response Construction and Span Finalization

Service B constructs its response. The spans in Service B are finalized in reverse order of creation. The innermost spans (cache and database) are finalized first, followed by the server span. Finalization records the end timestamp and calculates the duration. The span status is set to OK if no errors occurred.

### Step 10: Batch Export

The batch span processor in Service B collects the completed spans. Every 5 seconds, it serializes the accumulated spans into protobuf format, compresses the payload with gzip, and sends it to the collector over gRPC. The export is non-blocking: the application continues processing requests while the export happens on a background thread. If the collector is unreachable, the processor retries with exponential backoff: 1 second, 2 seconds, 4 seconds, up to a maximum of 30 seconds.

## Consistency and State

### Crash During Span Creation

If a service crashes after creating a span but before exporting it, that span is lost entirely. The parent service sees its client span with a duration that covers the connection until timeout. The client span's status is set to ERROR, and a span event records the connection error. The trace is incomplete but the failure is clearly visible.

The recovery strategy is to use a local file exporter that writes spans to disk as a fallback. When the service restarts, the file exporter replays unexported spans. This adds durability at the cost of disk I/O overhead.

### Partition Between Service and Collector

When the network partitions between a service and the trace collector, the span processor's export queue fills up. The default queue capacity is 2048 spans. When the queue is full, the processor drops the oldest spans to make room for new ones. A counter metric tracks the number of dropped spans.

The service continues to function normally. Request processing is not blocked by the export queue being full. The only impact is lost trace data. When the partition heals, the processor drains the remaining queue and resumes normal export.

### Concurrent Writes to the Trace Store

Multiple collector instances can receive spans for the same trace simultaneously. The storage backend must handle concurrent writes to the same trace. The standard approach is to store each span as an independent document with the trace ID as the shard key. The indexer merges spans from concurrent writes into the final trace tree during query time, not write time.

Elasticsearch handles this naturally because each span is a separate document. The trace assembly happens at query time by fetching all spans with the matching trace ID and building the tree in memory. This trades query latency (typically 50 to 200 milliseconds) for simpler write semantics.

### Stale Reads After Indexing Delay

Trace data is not immediately visible after ingestion. The indexing pipeline introduces a freshness delay of 10 to 30 seconds. If you query for traces from the last minute, you might miss traces that are still being indexed. This is a deliberate trade-off: batching and indexing improve throughput at the cost of visibility latency.

Query UIs compensate by showing a "live" mode that polls for new traces every few seconds. The live mode shows traces as they become available. A progress indicator shows the indexing lag: "Showing results within 25 seconds of real time."

## Performance Characteristics

### Span Creation Overhead

Creating a span involves allocating memory for the span object, generating a span ID, and capturing the current timestamp. The overhead per span creation is approximately 200 to 500 nanoseconds in a single-threaded benchmark. However, the timestamp capture using `clock_gettime(CLOCK_REALTIME)` takes approximately 50 to 100 nanoseconds on modern hardware.

The total overhead depends on the number of spans per request. A request that creates 20 spans incurs roughly 4 to 10 microseconds of CPU time for span creation alone, which is negligible compared to request processing times.

### Span Export Overhead

Exporting spans is the most significant overhead source. The batch processor serializes spans into protobuf, compresses the payload, and sends it over gRPC. A batch of 512 spans takes approximately 10 to 30 milliseconds to serialize and compress. The gRPC call adds network round-trip time.

The export runs on a background thread, so it does not directly affect request latency. However, it consumes CPU and memory bandwidth. On a 16-core machine handling 10,000 requests per second, the tracing overhead is typically 1% to 3% of total CPU.

### Sampling Impact on Storage

With head-based sampling at 1%, a system processing 50,000 requests per second generates 500 traces per second. Each trace with 10 spans of 512 bytes each consumes 5 KB. The total storage rate is 2.5 MB per second, or 216 GB per day before compression. After gzip compression, the storage requirement drops to approximately 40 GB per day.

Increasing the sampling rate to 5% multiplies the storage by 5, reaching 200 GB per day after compression. The cost of storage must be balanced against the value of having more traces for debugging.

### Query Latency

Trace queries fall into two categories: single-trace lookups and filtered searches. A single-trace lookup by trace ID completes in 10 to 50 milliseconds because it is a direct key lookup. A filtered search across 24 hours of data takes 200 milliseconds to 2 seconds, depending on the index coverage.

The p99 query latency for filtered searches is typically 5 seconds in systems with more than 10 billion spans. Indexing optimizations like pre-aggregated rollups can reduce p99 latency to under 500 milliseconds.

## Configuration and Tuning

### Sampling Rate Configuration

The sampling rate is the most impactful tracing configuration. Start at 1% for production systems. Monitor the storage growth rate and the query performance. Increase to 5% only if you have budget for 5 times the storage and your debugging workflows require more complete data.

For high-throughput services handling over 100,000 requests per second, use a rate limit instead of a percentage. A rate limit of 500 traces per second ensures predictable resource usage regardless of traffic spikes. Set a minimum rate to ensure low-traffic periods still produce enough traces for analysis.

### Span Attribute Cardinality

Attribute values with high cardinality, such as user IDs, session IDs, or request parameters, cause index bloat. The inverted index grows proportionally to the number of unique attribute values. A query indexing a user ID field could cause the index to grow by terabytes per day in a system with millions of users.

The solution is to set a cardinality limit on index attributes. Elasticsearch has a default limit of 10,000 unique values per field per shard. Attributes exceeding the limit are dropped from the index but remain in the stored span document. You can still view them in individual span details, but you cannot search or filter by them.

### Export Queue Size

The batch span processor queue size determines how many spans can be buffered in memory during a collector outage. The default of 2048 spans provides approximately 20 seconds of buffer at 100 spans per second. For a service handling 1,000 spans per second, increase the queue size to 10,240 to provide 10 seconds of buffer.

Monitor the queue fill rate and export latency. If the queue consistently stays above 50% capacity, either increase the queue size or reduce the export interval. If the queue reaches 100% capacity and drops spans, reduce the span creation rate through sampling or increase the queue size significantly.

### Collector Configuration

The trace collector is a critical infrastructure component. Run at least two collector instances behind a load balancer for high availability. Each collector instance should have enough memory to buffer spans during storage backend slowdowns. A rule of thumb is 1 GB of memory per 10,000 spans per second of throughput.

Configure the collector's batch processor to match your storage backend's write throughput. Elasticsearch can handle approximately 5,000 to 10,000 documents per second per shard. Tune the collector's batch size to send batches of 1,000 to 5,000 spans at a time, balanced against the 10-second maximum delay for traces to become visible.

## Advanced Span Attributes

### Attribute Types and Performance

OpenTelemetry span attributes support several types: string, boolean, int64, float64, and arrays of these types. The performance cost of setting an attribute varies by type. Setting a string attribute involves a memory allocation for the string copy. Setting an integer or boolean attribute involves a single register write. Setting an array attribute is the most expensive because the array must be copied.

For high-throughput services, limit attributes to scalar types (string, int, boolean) and avoid array attributes. The cost of a string attribute is approximately 150 nanoseconds, while the cost of an int attribute is approximately 15 nanoseconds.

### Attribute Value Truncation

Span attribute values should be truncated to prevent excessively long values from consuming storage. The OpenTelemetry SDK supports configurable attribute value length limits. The default maximum attribute value length is 4,096 characters, but most setups reduce this to 256 characters to save storage.

Long attribute values occur in several scenarios:
- HTTP request bodies that are logged as attributes
- Database query strings that include long WHERE clauses
- Error messages that include full stack traces
- Authentication tokens (which should not be logged at all)

Truncation is applied at the SDK level before the span is exported. The truncation preserves the beginning of the value, since that typically contains the most useful information. A truncated value is marked with an ellipsis suffix: "SELECT * FROM orders WHERE user_id = 42 AND status = ..."

### Resource Attributes

Resource attributes describe the environment that produced the span. Unlike span attributes, resource attributes are attached to all spans from the same service instance. Common resource attributes include:

- `service.name`: the logical name of the service
- `service.namespace`: the team or product group
- `service.version`: the deployed version
- `service.instance.id`: the unique instance identifier
- `deployment.environment`: production, staging, development
- `host.name`: the hostname of the machine
- `cloud.provider`: aws, gcp, azure
- `cloud.region`: us-east-1, eu-west-2

Resource attributes are stored once per batch, not per span, so the storage overhead is minimal. They are essential for filtering and grouping traces by environment or deployment version.

## Error Handling in Tracing

### Capturing Errors

When a span encounters an error, the span status should be set to ERROR. The status object includes a description field for the error message. Additional error context can be added as span events:

```python
try:
    result = process_payment(order_id)
except PaymentError as e:
    span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
    span.record_exception(e)
    span.set_attribute("error.type", "PaymentError")
    span.set_attribute("error.code", e.error_code)
```

The `record_exception` method creates a span event with the exception type, message, and stack trace. The exception event type is `exception`. The attributes on the exception event include `exception.type`, `exception.message`, and `exception.stacktrace`.

### Error Propagation

When a service receives an error from a downstream dependency, the error should be propagated in the span chain. The client span in the calling service records the error status. The server span in the called service also records the error status. Both spans capture the error, providing visibility at both ends of the call chain.

```python
# Client side: calling service
with tracer.start_as_current_span("call_fraud_service") as span:
    try:
        response = fraud_service.check(order)
        span.set_attribute("fraud.score", response.score)
    except FraudServiceError as e:
        span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
        span.set_attribute("fraud.error", e.error_code)
        raise  # Re-raise for the caller to handle
```

### Sampling Errors at Higher Rates

Errors are the most valuable traces for debugging. Configure your sampling strategy to always retain error traces. With head-based sampling, set the error sampling to 100% by checking the final span status before deciding whether to export. With tail-based sampling, the error retention policy is configurable in the sampler.

The trade-off is that error rates can spike during incidents. A service that normally has 0.1% error rate might spike to 50% during a database outage. At 50% error rate, the tracing system must handle 50 times the normal error trace volume. The collector and storage capacity must account for error spikes.

### Error Trace Analysis

After deploying a fix, use the tracing system to verify that the error traces have stopped. Filter for the error type and the service name, and check that the count of error traces drops to zero. The tracing system provides immediate feedback on whether the fix was effective.

## Multi-Language Tracing Considerations

### Python

The Python SDK has the highest overhead of all OpenTelemetry SDKs because of the Global Interpreter Lock (GIL). Span creation in Python takes approximately 1 to 2 microseconds, compared to 200 to 500 nanoseconds in Go or Java. The GIL prevents parallel span processing on a single thread.

The Python SDK supports async frameworks through context variables. The `contextvars` module provides asynchronous context management for the active span. FastAPI and aiohttp are fully supported with automatic instrumentation.

### Java

The Java SDK uses bytecode manipulation for automatic instrumentation. The Java agent attaches to the JVM at startup and instruments library classes at load time. No code changes are required. The Java agent supports Spring Boot, JAX-RS, JDBC, Kafka, gRPC, and 40+ other libraries.

The Java SDK has the lowest overhead of all SDKs. Span creation takes approximately 200 nanoseconds. The Java agent's instrumentation is applied at class loading time, so there is zero runtime overhead for uninstrumented code paths.

### Go

The Go SDK uses explicit instrumentation at compile time. There is no runtime agent or bytecode manipulation. Each library must be instrumented with a wrapper or by using the library's plugin interface. Go's compiler optimizations make span creation very fast, approximately 150 nanoseconds.

The Go SDK's context propagation uses Go's `context.Context` interface. Every function that accepts a context automatically propagates the trace context. This makes context propagation in Go natural and less error-prone than in languages where context is managed through thread-local storage.

### Node.js

The Node.js SDK uses monkey-patching for automatic instrumentation. The instrumentation replaces library methods with wrapped versions that create spans. This approach works for popular libraries like Express, Axios, and MongoDB.

The Node.js SDK runs on a single thread, so span processing competes with request processing for CPU time. The batch span processor runs on the event loop, so heavy span processing can delay other operations. The `schedule_delay_millis` parameter should be set higher (10,000 milliseconds instead of 5,000) to reduce the frequency of export work.

### Performance Regression Testing

Tracing instrumentation should not cause performance regressions. Set up a performance test that measures the p50, p99, and p999 latency of a service with and without tracing:

## Span Link Types and Their Performance

### Span Kind Classification

The OpenTelemetry specification defines five span kinds. Each kind affects how the backend interprets the span's role in the trace.

**INTERNAL**: The span represents an operation within a single service. No network boundary is crossed. Internal spans are the most common kind for function calls, computations, and local operations. The overhead of an internal span is the lowest because there is no context propagation involved.

**SERVER**: The span represents an operation handling an incoming request from another service. The server span is created when a service receives a request and extracts trace context from the request headers. Server spans pair with CLIENT spans from the calling service.

**CLIENT**: The span represents an operation making an outgoing request to another service. The client span is created when a service sends a request and injects trace context into the request headers. The CLIENT span is the parent of the corresponding SERVER span in the downstream service.

**PRODUCER**: The span represents sending a message to a queue or event bus. The producer span records the message being sent. The PRODUCER span may not have a matching CONSUMER span if the message is consumed at a later time.

**CONSUMER**: The span represents receiving a message from a queue or event bus. The consumer span records the message being processed. The CONSUMER span uses a LINK to the PRODUCER span rather than a parent-child relationship, because the producer and consumer may execute at different times and in different trace contexts.

### Span Link Performance

Span links are more expensive than parent-child relationships because the link includes the trace ID and span ID of the linked span. A link requires a 16-byte trace ID and an 8-byte span ID and an 8-byte attribute, totaling approximately 32 bytes per link. A parent-child relationship only requires an 8-byte parent span ID.

The performance cost of creating a link is approximately 100 nanoseconds, compared to 20 nanoseconds for setting a parent span ID. Links should be used sparingly, only when the parent-child relationship does not apply (such as across message queue boundaries).

## Trace Context Formats

### W3C Trace Context Format

The W3C Trace Context format is the industry standard. The `traceparent` header uses the format `00-traceid-spanid-flags`. The version `00` is the current version. The trace ID is 16 bytes (32 hex characters). The span ID is 8 bytes (16 hex characters). The flags byte uses bit 0 for the sampled flag.

Example: `00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01`

The `tracestate` header carries vendor-specific data. It is a comma-separated list of key-value pairs. Each vendor uses its own key. The tracestate is copied verbatim to downstream services. The total tracestate header size is limited to 512 bytes.

### B3 Format (Zipkin)

The B3 format is used by Zipkin. It uses three headers: `X-B3-TraceId`, `X-B3-SpanId`, `X-B3-ParentSpanId`, and `X-B3-Sampled`. The B3 format is simpler than W3C Trace Context but lacks the extensibility of the tracestate header.

The OpenTelemetry SDK supports both formats through the `Propagator` abstraction. The `W3CTraceContextPropagator` and `B3Propagator` can be used together or separately. The propagator is configured at SDK initialization:

```python
from opentelemetry.propagate import set_global_textmap
from opentelemetry.propagators.composite import CompositePropagator
from opentelemetry.propagators.otlp import OTLPPropagator

set_global_textmap(
    CompositePropagator([
        W3CTraceContextPropagator(),
        B3Propagator(),
    ])
)
```

The composite propagator extracts from both W3C and B3 formats and injects into both formats. This ensures compatibility with services that use either format.

## SDK Performance Optimization

### Reducing Allocation Pressure

Span creation involves memory allocation for the span object and attributes. In high-throughput services, this allocation pressure causes garbage collection pauses. Use object pooling to reuse span objects:

```python
from opentelemetry.sdk.trace import Span as SpanBase

class PooledSpan(SpanBase):
    _pool = []

    @classmethod
    def create(cls, name, context):
        if cls._pool:
            span = cls._pool.pop()
            span._name = name
            span._context = context
            return span
        return cls(name, context)

    def end(self):
        super().end()
        self.__class__._pool.append(self)
```

Object pooling reduces GC pressure by 30% in high-throughput services. The pool size should be equal to the maximum number of concurrent spans, which is typically the number of concurrent requests multiplied by the average span depth.

### Batching Strategies

The batch span processor sends spans in batches. The batch size and interval affect both the export latency and the network efficiency. A smaller batch interval reduces the time spans spend in the queue but increases the number of HTTP requests. A larger batch size improves compression efficiency but increases memory usage.

Optimal configuration for most services: batch size of 512 spans, schedule interval of 5 seconds, and max queue size of 2,048 spans. This configuration provides a good balance between export latency (5 seconds max) and network efficiency (512 spans per HTTP request).
