## Tier 1: Foundation

### Challenge 1: Span Assembly

You are given a list of spans from a single trace. The spans arrived out of order. Here is the data:

```json
[
  {"span_id": "C", "parent_span_id": "B", "op": "query_db", "start": 100, "end": 350},
  {"span_id": "A", "parent_span_id": null, "op": "handle_request", "start": 0, "end": 500},
  {"span_id": "D", "parent_span_id": "B", "op": "cache_lookup", "start": 100, "end": 150},
  {"span_id": "B", "parent_span_id": "A", "op": "process_payment", "start": 50, "end": 400}
]
```

**Task**: Sort the spans into a correct trace tree. Calculate the total duration of the trace. Identify which span is on the critical path.

**Model answer**:

The trace tree structure is:

```
Span A (handle_request):     0 to 500  (500ms)  — root span
  Span B (process_payment):  50 to 400 (350ms)  — child of A
    Span C (query_db):       100 to 350 (250ms) — child of B
    Span D (cache_lookup):   100 to 150 (50ms)  — child of B
```

Total duration: 500 milliseconds (from A's start to A's end).

The critical path is A (0) to B (50) to C (100) to C (350). The critical path length is 350 milliseconds. Span D is not on the critical path because it runs in parallel with C and finishes earlier.

The query_db span (C) is the bottleneck. At 250 milliseconds, it accounts for 71% of the process_payment span's duration and 50% of the total trace duration.

### Challenge 2: Sampling Rate Calculation

Your platform processes 500,000 requests per second during peak hours. Each request generates an average of 12 spans. Each span consumes 700 bytes of storage uncompressed, and compression reduces this to 200 bytes. You have a budget of 500 GB per day for tracing storage.

**Task**: Calculate the maximum sampling rate that fits within the budget. Then calculate the daily span count at that sampling rate. Finally, determine how many days of retention you can afford if you add a 3-node Elasticsearch cluster with 3 TB of SSD per node, assuming 2x replication.

**Model answer**:

Total uncompressed span storage per trace: 12 spans * 700 bytes = 8,400 bytes per trace.

Compressed storage per trace: 12 spans * 200 bytes = 2,400 bytes per trace.

500,000 requests per second * 86,400 seconds = 43.2 billion requests per day.

At 100% sampling, daily compressed storage: 43.2 billion * 2,400 bytes = 103.68 TB.

To fit in 500 GB per day: 500 GB / 103,680 GB = 0.00482, or approximately 0.48% sampling.

Maximum sampling rate: 0.48%, or approximately 0.5% for practical purposes.

Daily span count at 0.5%: 43.2 billion * 0.005 * 12 = 2.592 billion spans per day.

Elasticsearch cluster capacity: 3 nodes * 3 TB = 9 TB raw. With 2x replication, usable capacity = 9 TB / 2 = 4.5 TB.

Daily compressed storage at 0.5%: 500 GB.

Days of retention: 4.5 TB / 500 GB = 9 days.

To achieve 30-day retention, you need (30 * 500 GB) * 2 (replication) = 30 TB of raw storage, or 10 nodes with 3 TB each.

### Challenge 3: Context Propagation Debug

A team reports that their tracing system shows many broken traces. Each day, the tracing backend receives 10,000 traces, but 4,000 of them are root spans from a service called "order-processor." The order-processor service is not an entry point service — it is called by the "api-gateway" service.

**Task**: Identify the likely cause of the broken traces. Describe how you would confirm the diagnosis. Propose a fix.

**Model answer**:

The likely cause is that the order-processor service is not extracting trace context from incoming requests. Instead of continuing the existing trace, it starts a new trace for every request it receives. This produces 4,000 new root spans per day that should have been child spans of the api-gateway's spans.

To confirm the diagnosis:
1. Check the order-processor's HTTP request handling code for trace context extraction.
2. Verify that the HTTP client instrumentation is enabled in the api-gateway service.
3. Temporarily enable debug logging on the tracing SDK in the order-processor to see what context it receives.
4. Send a test request with a known traceparent header and verify whether the order-processor uses it.

The fix depends on the root cause:
- If the order-processor is using an uninstrumented HTTP library, add the correct instrumentation dependency.
- If the instrumentation is present but not configured, enable it in the SDK initialization.
- If the code manually handles HTTP requests, add context extraction using the propagator API.

After the fix, the broken trace rate should drop from 40% to under 1% (the remaining 1% accounts for network partitions and other transient issues).

## Tier 2: Design

### Challenge 4: Design a Tracing System for a Payment Platform

You are designing a distributed tracing system for a payment platform that processes $10 million in transactions per day. The platform has 50 microservices. The payment flow touches 8 services on average. The platform handles 5,000 requests per second at peak.

**Requirements**:
- Every payment transaction must be traced at 100% sampling.
- Non-payment traffic (account lookups, balance checks) can be sampled at 1%.
- Traces must be visible within 30 seconds of the request completing.
- Storage retention must be 90 days for compliance.
- The budget for tracing infrastructure is $50,000 per month.

**Provide a design that meets these requirements. Include the architecture, sampling strategy, storage plan, and cost estimate.**

**Model answer**:

**Architecture**:
- Each service runs the OpenTelemetry SDK with a batch span processor exporting to a local collector.
- A Kubernetes Deployment of 5 collector instances behind a gRPC load balancer receives spans from all services.
- Collectors publish spans to a Kafka topic with 8 partitions. Kafka provides decoupling and buffering.
- A stream processor (Kafka Streams or Flink) consumes from Kafka. It applies a dual sampling policy: all spans with a `payment_txn=true` attribute are retained, and non-payment spans are sampled at 1%.
- Retained spans are written to an Elasticsearch cluster with daily indices.
- A query service provides the API for trace retrieval and visualization.

**Sampling strategy**:
- Use head-based sampling at the SDK level. The payment service is configured with a custom sampler that always samples. Non-payment services use a rate-based sampler at 1%.
- This avoids the need for tail-based sampling, which would require buffering all spans and increase infrastructure cost.

**Storage plan**:
- Payment traffic: 5,000 req/s * 8 services * 10 spans per service = 400,000 spans per second. At 200 bytes compressed, that is 80 MB per second, or 6.9 TB per day.
- Non-payment traffic: Assume 20,000 req/s * 3 services * 5 spans = 300,000 spans per second. At 1% sampling: 3,000 spans per second. At 200 bytes compressed: 0.6 MB per second, or 52 GB per day.
- Total: approximately 7 TB per day before replication.
- Elasticsearch cluster: 12 data nodes, each with 4 TB SSD. With 1x replication, usable capacity = 48 TB. Retention = 48 TB / 7 TB per day = 6.8 days. This does not meet the 90-day requirement.
- Solution: Use a hot-warm-cold storage tier. Hot tier (7 days, SSD, 12 nodes): 49 TB. Warm tier (30 days, HDD, 8 nodes): 210 TB. Cold tier (90 days, S3-compatible object store): unlimited. Queries for recent traces hit the hot tier. Historical traces are recalled from cold storage on demand, taking 30 to 60 seconds.

**Cost estimate**:
- 5 collector instances: $500/month (AWS c5.large reserved instances)
- 3 Kafka brokers: $600/month (m5.large instances)
- 12 hot-tier Elasticsearch nodes: $7,200/month (i3.2xlarge instances)
- 8 warm-tier Elasticsearch nodes: $3,200/month (d2.2xlarge instances)
- Cold storage (S3): $1,500/month for 630 TB at S3 Glacier Deep Archive rates
- Stream processor: $400/month (2 c5.xlarge instances)
- Total: approximately $13,400/month, well within the $50,000 budget.

### Challenge 5: Debug a Latency Anomaly

Your team's e-commerce platform suddenly experiences a 3x increase in checkout latency. The p95 checkout time went from 800 milliseconds to 2.4 seconds. You have a fully instrumented tracing system.

**Task**: Describe your step-by-step debugging process. What would you look for in the trace data? How would you narrow down the root cause?

**Model answer**:

**Step 1: Check the aggregate trace data**. Open the trace dashboard and look at the latency breakdown for the checkout operation. Compare the current latency distribution to the baseline from the past 7 days. Identify which span in the checkout trace shows the largest absolute increase.

**Step 2: Examine the slowest traces**. Filter for checkout traces with duration over 2 seconds. Open the 10 slowest traces. Look at the waterfall view. The slowest span within each trace is the likely culprit. If all slow traces show the same slow span, you have found the source.

**Step 3: Compare against the baseline**. If the slow span is a database query, compare the query duration to the baseline. A database query that normally takes 50 milliseconds but now takes 800 milliseconds indicates a database performance issue. Check the database span attributes for the query text, the number of rows returned, and the database host.

**Step 4: Check for changes**. Look at the span attributes for the slow span. The database host attribute might show that queries are being routed to a different host than before. The query text might show a different query pattern. The number of rows returned might be 10 times higher than normal.

**Step 5: Correlate with metrics and logs**. The trace identifies the slow span. Cross-reference with database metrics: CPU, connection count, lock contention, and query throughput. Check the database logs for slow query logs. The combination of trace data, metrics, and logs tells the full story.

**Step 6: Determine the root cause**. Common scenarios:
- A database query that was previously using an index is now doing a full table scan because a new deployment changed the query shape.
- A connection pool exhaustion is causing queries to wait for available connections. The connection pool span shows high wait time.
- A new deployment added a new service call in the checkout path. The trace shows a new span that was not present before.
- A downstream service is throttling. The client span shows a 503 status code and a retry loop.

**Step 7: Fix and verify**. Apply the fix and monitor the trace data to confirm the latency returns to baseline. Use the tracing system to compare the pre-fix and post-fix traces side by side.

### Challenge 6: Multi-Region Trace Assembly

Your company operates in 3 regions: US-East, EU-West, and AP-Southeast. A user in London sends a request that hits the EU-West load balancer, calls a service in US-East for fraud detection, and queries a database in AP-Southeast for historical data. The trace must be visible as a single, complete trace.

**Task**: Design the cross-region trace assembly mechanism. Describe how spans from different regions are collected, correlated, and assembled into a single trace. Address the challenges of network latency, partial failures, and time synchronization.

**Model answer**:

**Collection**: Each region runs its own local collector fleet and Elasticsearch cluster. Spans are stored in the local cluster with the region name as a tag. The trace ID is the same across all regions because the root span's trace ID propagates through the trace context headers.

**Cross-region trace discovery**: When a trace query arrives, the query service must locate which regions have spans for the requested trace ID. The query service maintains a lightweight index mapping trace ID to the set of regions that have stored spans for that trace. This index is populated asynchronously by each region's stream processor, which sends a "trace found" message to a global Kafka topic whenever it stores a new trace.

**Trace assembly**: The query service queries all regions in parallel for the spans of a given trace ID. Each region returns its local spans. The query service merges the spans, sorts by start time, and builds the parent-child tree. The assembly happens in memory on the query service.

**Time synchronization**: Network latency between regions is 50 to 200 milliseconds. Span timestamps use the local system clock of each service, which may drift by up to 10 milliseconds across regions. The assembly algorithm uses the parent-child relationship to resolve ambiguity: a child span must start after its parent span starts. If a child span's start time is before its parent's start time (due to clock skew), the query service adjusts the timeline by aligning child start times to parent start times.

**Partial failure handling**: If one region is unreachable, the query service returns the assembled trace from the other regions with a warning: "Spans from AP-Southeast are unavailable. Trace is incomplete." The missing region's spans are fetched when the region recovers.

**Sampling consistency**: All regions use the same hash-based sampling algorithm. The sampling decision is based on the trace ID, not the region. This ensures that a trace is sampled or dropped consistently across all regions. If the US-East classifier decides to sample a trace, the EU-West and AP-Southeast collectors will also receive and store the spans for that trace.

## Tier 3: Expert

### Challenge 7: Zero-Overhead Tracing

You are designing a tracing system for a high-frequency trading platform that processes 1 million orders per second with a strict end-to-end latency budget of 10 microseconds. Any instrumentation overhead that adds more than 100 nanoseconds to the request path is unacceptable.

**Task**: Design a tracing approach that meets the latency budget while providing useful debugging information. Discuss the trade-offs you are making and how you validate that the tracing system does not affect production latency.

**Discussion**:

**Approach**: Zero-overhead tracing cannot use standard OpenTelemetry instrumentation because even the span creation overhead of 200 to 500 nanoseconds exceeds the 100 nanosecond budget. The solution is to use sampling-based context propagation with out-of-band span recording.

**Pre-sample the trace context**: The entry point randomly decides whether to trace a request using a lightweight hash of the request ID. The decision is encoded in a single bit in the request header. The sampling rate is set to 0.01% (1 in 10,000 requests). This means 99.99% of requests incur zero tracing overhead — just a single bit check.

**Minimal context propagation**: For the 0.01% of traced requests, the trace context is propagated as a single 64-bit trace ID in a custom header. No span ID, no parent span ID, no tracestate. The trace ID is generated by the entry point and passed through. The 64-bit ID is chosen because it fits in a CPU register and can be compared in a single cycle.

**Out-of-band span recording**: Each service runs a secondary thread that periodically (every 100 milliseconds) reads the current trace context from a lock-free ring buffer. The ring buffer stores the trace ID, the current timestamp (read from the TSC register for nanosecond precision), and the operation name. The secondary thread batches these records and writes them to a local log file. The log file is shipped to the collector asynchronously.

**No in-band span creation**: The hot path never allocates memory, acquires locks, or makes system calls for tracing. The only cost is storing the trace ID in a thread-local variable and writing it to the ring buffer. The ring buffer write is a single atomic store, costing approximately 10 to 20 nanoseconds.

**Post-processing assembly**: The collector receives the out-of-band span records. Because there is no explicit parent-child relationship, the collector must infer the trace structure from the timing and ordering of records. A service that receives a request and then sends a request to another service within the same microsecond window is inferred to be the parent. This heuristic works for 95% of traces but may produce incorrect parent-child relationships in rare cases of concurrent requests.

**Validation**: Run a shadow deployment where the same traffic is processed with and without tracing. Compare the p50, p99, and p999 latency of the two deployments. The difference must be statistically indistinguishable (within 10 nanoseconds at p99). Use a paired t-test on 1 million samples to confirm.

**Trade-offs**:
- Trace completeness is reduced. The out-of-band recording may miss spans if the ring buffer overflows.
- Parent-child relationships are inferred, not explicit. Rare traces may have incorrect structure.
- The 0.01% sampling rate means most requests are completely invisible. Debugging relies on the 1 in 10,000 sampled traces being representative.
- The latency budget is met, but at the cost of significantly reduced tracing fidelity.

### Challenge 8: Trace-Aware Auto-Scaling

Design a system that uses tracing data to drive automated scaling decisions. The system should detect when a service is becoming a bottleneck and automatically scale it up before latency degrades for end users. Address how you detect the bottleneck, how you differentiate between a scaling need and a transient spike, and how you prevent the tracing system itself from becoming a scaling bottleneck.

**Discussion**:

**Bottleneck detection**: The tracing system continuously computes the critical path for every trace. The critical path is the sequence of spans that determines the total trace duration. For each span on the critical path, the system tracks the span's duration relative to the parent span. A span that consistently accounts for more than 60% of its parent's duration is a candidate bottleneck.

The system aggregates this data across all traces for each service and operation pair. A service is flagged as a bottleneck if more than 30% of its traces show the service on the critical path AND the service's critical path contribution has increased by more than 20% over the past 5 minutes.

**Differentiating scaling needs from transient spikes**: The tracing system uses a moving window of 5 minutes with 1-minute granularity. A bottleneck is considered a scaling need if the critical path contribution is elevated for 3 consecutive 1-minute buckets. A single bucket of elevation is treated as a transient spike and ignored.

The system also correlates with deployment events. If a bottleneck appears immediately after a new deployment, the system triggers a rollback instead of a scale-up. The deployment event is recorded in the tracing system's span attributes (deployment version tag on each service's spans).

**Auto-scaling action**: When the system determines a scaling need, it triggers the auto-scaler for the affected service. The auto-scaler increases the replica count by 50% (minimum 2 additional replicas) and sets a cooldown of 5 minutes. During the cooldown, the system monitors the critical path contribution. If it drops below 40% of the parent span duration, the scaling is considered successful. If it remains above 60%, the system triggers another scale-up.

**Preventing the tracing system from being a scaling bottleneck**: The tracing system runs on a dedicated infrastructure pool that is independently scaled. The tracing system's own resource usage (collector CPU, storage I/O) is monitored and scaled separately from the application services. If the tracing system's resource usage exceeds 70%, the system automatically reduces the sampling rate rather than requesting more resources. This ensures the tracing system never competes with application services for resources.

**Cross-service scaling dependencies**: A bottleneck in one service might be caused by a downstream dependency. If Service A calls Service B, and Service B is slow, Service A appears on the critical path. The scaling system must differentiate between a service that is slow because it is under-resourced and a service that is slow because its downstream dependency is slow. The solution is to examine the span's child spans. If the slow span's children account for more than 80% of its duration, the bottleneck is downstream. If the children account for less than 50% of the duration, the bottleneck is in the service itself.

**Feedback loop**: The system records each scaling decision as a span event on the service's root span. The event includes the old replica count, the new replica count, and the reason for the decision. This creates a feedback loop: you can trace the auto-scaling system's decisions alongside the application traces, enabling debugging of the scaling system itself.

### Challenge 9: Trace Quality Monitoring

Design a system that continuously monitors the quality of trace data and alerts when quality degrades. Define quality metrics, set thresholds, and describe the alerting and remediation process.

**Discussion**:

**Quality metrics**: The system monitors four quality metrics continuously.

The span completion rate measures the percentage of expected spans that are present in each trace. For each service, the system knows the expected number of child spans based on historical patterns. If the payment service normally creates 5 child spans per trace but a trace only has 3, the completion rate is 60%. The alert threshold is 90% for critical services and 80% for standard services.

The orphan span rate measures the percentage of spans whose parent span ID does not match any known span. A high orphan rate indicates broken context propagation. The alert threshold is 5% of all spans.

The indexing delay measures the time between a span being exported from the SDK and becoming visible in the query UI. The system samples this by marking a subset of spans with a synthetic timestamp and measuring the delay. The alert threshold is 60 seconds at p99.

The span drop rate measures the percentage of spans dropped by the SDK or collector before storage. The system tracks the number of spans dropped at each stage. The alert threshold is 0.1% of all spans.

**Alerting**: When any quality metric exceeds its threshold for 5 consecutive minutes, an alert fires. The alert includes the metric name, the current value, the threshold, the affected services, and a link to a dashboard showing the metric over time. The alert severity is P2 for span completion rate and orphan span rate (systematic issues), and P3 for indexing delay and span drop rate (infrastructure issues).

**Remediation**: Each quality degradation has a specific remediation playbook.

Span completion rate degradation: Check for recent deployments of the affected services. If a deployment occurred within the last 30 minutes, roll it back. Check the SDK configuration for recent changes. Verify that the batch span processor is not dropping spans due to queue overflow.

Orphan span rate degradation: Check which services have the highest orphan span count. Those services are likely missing context propagation. Verify that the HTTP client instrumentation is enabled and that the propagator is configured correctly. Check for custom HTTP clients that bypass the instrumentation library.

Indexing delay degradation: Check the collector CPU and memory usage. Check the Elasticsearch cluster health. Verify that the cluster is not overloaded. Check for blocking Elasticsearch queries that are consuming resources.

Span drop rate degradation: Check the collector's memory usage. If the collector is running out of memory, it drops spans under memory pressure. Increase the collector's memory allocation. Check the Kafka buffer for backpressure. If the buffer is full, the collector cannot export spans.

**Automated remediation**: For simple quality degradations, the system can auto-remediate. If the span drop rate exceeds 1% due to collector memory pressure, the system automatically increases the collector's memory allocation by 50% and reduces the sampling rate by 20% to reduce memory pressure. The auto-remediation is logged as a trace event in the meta-tracing system.

### Challenge 10: Tracing Migration Strategy

A company has 200 microservices running on a legacy tracing system based on Zipkin with Cassandra storage. The system handles 500,000 spans per second. The Zipkin SDK is deprecated and the Cassandra cluster is at capacity. You need to migrate to OpenTelemetry with Elasticsearch storage without losing trace continuity.

**Task**: Design the migration strategy. Describe the phases, the risks, and the rollback plan. Address how you maintain trace continuity during the migration.

**Model answer**:

**Phase 1: Dual instrumentation** (2 weeks). Add the OpenTelemetry SDK alongside the existing Zipkin SDK to all 200 services. Both SDKs create spans for the same requests. The Zipkin SDK continues exporting to the existing Cassandra cluster. The OpenTelemetry SDK exports to a new Elasticsearch cluster. Traces appear in both systems simultaneously.

The dual instrumentation phase validates that the OpenTelemetry instrumentation produces correct data. Compare traces from both systems for the same trace ID. The span names, attributes, and timestamps should match. The risk is doubled resource usage: two SDKs creating spans for every request.

**Phase 2: Parallel run** (4 weeks). Route 10% of production traffic to the OpenTelemetry pipeline using a feature flag. 90% of traffic continues using Zipkin. Monitor the OpenTelemetry pipeline for correctness and performance. If issues arise, disable the feature flag and revert to Zipkin-only tracing.

During parallel run, compare the trace quality metrics between the two systems. The span completion rate, orphan span rate, and indexing delay should match within 5%. The OpenTelemetry SQL should not add more than 2% overhead at p99 compared to the Zipkin SDK.

**Phase 3: Gradual cutover** (4 weeks). Increase the OpenTelemetry traffic share by 25% per week. Week 1: 25% OpenTelemetry, 75% Zipkin. Week 2: 50% each. Week 3: 75% OpenTelemetry, 25% Zipkin. Week 4: 100% OpenTelemetry.

At each step, monitor the tracing system health metrics and the application performance metrics. If any metric degrades by more than 5%, pause the cutover and investigate. The rollback plan at each step is to set the feature flag back to previous week's value.

**Phase 4: Decommission Zipkin** (1 week). Remove the Zipkin SDK from all services. Remove the Cassandra cluster. Archive the Cassandra data to S3 for compliance retention.

**Risks and mitigations**:

Risk: OpenTelemetry SDK has higher overhead than the Zipkin SDK. Mitigation: Apply more aggressive sampling in the OpenTelemetry SDK during the migration. Reduce the sampling rate from 5% to 2% to compensate for any overhead increase.

Risk: Elasticsearch cluster cannot handle the write throughput. Mitigation: Pre-provision the Elasticsearch cluster with 50% more capacity than needed. Monitor the write rejection rate and add nodes if needed.

Risk: Context propagation breaks during migration when some services use Zipkin and others use OpenTelemetry. Mitigation: Configure the Zipkin SDK to accept OpenTelemetry trace context headers. Configure the OpenTelemetry SDK to propagate Zipkin-compatible headers (b3 format). Both SDKs read and write both trace context formats.

**Rollback plan**: If the OpenTelemetry pipeline has a critical failure, set all feature flags to 0% and fall back to the Zipkin pipeline. The rollback takes 10 minutes. Restore the Zipkin SDK configuration from backup. The Cassandra cluster must be kept running during the entire migration for rollback readiness.

**Success criteria**: The migration is successful when:
- All 200 services export spans through OpenTelemetry.
- The span completion rate is above 95% (matching the Zipkin baseline).
- The p99 query latency is under 500 milliseconds (matching the Zipkin baseline).
- The storage cost is reduced by at least 20% compared to the Cassandra cluster.
- The Zipkin SDK and Cassandra cluster are decommissioned.

## Additional Practice: Trace Analysis Exercises

### Exercise 1: Trace Waterfall Interpretation

You are shown the following trace:

```
Root span (handle_request): 0 to 3500ms
  Span A (authenticate): 0 to 45ms
  Span B (check_inventory): 45 to 120ms
  Span C (process_payment): 45 to 3200ms
    Span D (fraud_check): 45 to 3000ms
      Span E (external_api_call): 45 to 2900ms
    Span F (charge_card): 3000 to 3150ms
  Span G (send_notification): 3200 to 3350ms
```

**Questions**:
1. What is the bottleneck span?
2. Why does Span F start at 3000ms when Span D ends at 3000ms?
3. What would you investigate first?
4. How would you fix this?

**Answers**:
1. The bottleneck is Span E (external_api_call) at 2900ms. It accounts for 83% of the total trace duration (2900 / 3500).
2. Span F (charge_card) starts immediately after Span D (fraud_check) ends because Span D is a synchronous call. The payment service waits for the fraud check to complete before charging the card.
3. Investigate Span E first. The external API call took 2900ms, which is excessive. Check the span attributes for the external API endpoint URL, the timeout setting, and any error codes. The external API might be down, throttling, or the timeout might be set too high.
4. Reduce the timeout on the external API call from 3000ms to 500ms. Add a circuit breaker that trips after 3 consecutive timeouts. Implement a fallback to a cached fraud score if the external API times out. After the fix, the trace duration for the payment service should drop from 3200ms to under 1000ms.

### Exercise 2: Capacity Planning Calculation

Design a tracing system for a platform with 50 million daily active users. Each user makes 50 requests per day. Each request generates 10 spans. You have a monthly infrastructure budget of $200,000. Storage costs $0.20 per GB per month for hot storage and $0.05 per GB per month for warm storage.

**Questions**:
1. What is the peak span rate in spans per second?
2. What sampling rate fits within the storage budget?
3. How many collectors do you need?
4. What is your recommended architecture?

**Answers**:
1. Peak spans per second: 50 million users * 50 requests / 86400 seconds per day * 10 spans per request = approximately 289,000 spans per second at average. At 3x peak, approximate 867,000 spans per second peak.
2. At 0.5% sampling: 289,000 * 0.005 = 1,445 sampled spans per second. Each span is 200 bytes compressed: 1,445 * 200 = 289 KB per second. Daily storage: 289 KB * 86400 = 25 GB per day. Monthly storage: 750 GB. At $0.20 per GB, monthly hot storage cost: $150. Warm storage for 30 days: $750 * 0.05 = $37.50. Total monthly storage: approximately $187.50. This fits within the $200,000 budget with room to increase sampling to 5% if needed.
3. At 0.5% sampling with 1,445 sampled spans per second: 1 collector instance. At 867,000 peak raw spans per second: 434 collector instances (at 2,000 spans per second each). However, if sampling is done at the SDK level, only 1,445 sampled spans per second reach the collector, requiring only 1 collector. This is the recommended approach.
4. Recommended architecture: SDK-level head-based sampling at 0.5% (or higher, since the budget allows up to 5%). Use OpenTelemetry SDK in all services. A single collector instance (2 for high availability) behind a gRPC load balancer. The collector writes to a Kafka topic with 1 partition. A stream processor consumes from Kafka and writes to a 3-node Elasticsearch cluster with 2 TB of SSD each. Hot storage for 7 days, warm storage for 30 days. This architecture costs approximately $5,000 per month, well within the $200,000 budget.

## Bonus Challenge: Distributed Tracing in a Serverless Architecture

### Scenario

You are building a tracing system for a serverless architecture that uses AWS Lambda for all backend processing. The platform processes 10,000 requests per second at peak. Each request triggers 3 Lambda functions in sequence: API Gateway validates the request, a payment function processes the payment, and a notification function sends a confirmation.

Lambda functions are short-lived and cannot buffer spans for later export. Each function must export spans before it terminates. Lambda has a maximum execution duration of 15 minutes, but most functions complete in under 1 second.

**Task**: Design the tracing architecture for this serverless platform. Address the challenges of short-lived functions, cold starts, and limited memory. Describe how context propagation works across Lambda invocations.

**Model answer**:

**Context propagation**: The API Gateway receives the HTTP request and creates the root span. The trace context is serialized into the JSON event payload that triggers the first Lambda function. The Lambda function's OpenTelemetry wrapper extracts the trace context from the event payload and creates a child span. When the payment function calls the notification function (asynchronously via SNS or EventBridge), the trace context is injected into the SNS message attributes.

**Span export**: Each Lambda function exports spans synchronously before returning. The OpenTelemetry Lambda wrapper intercepts the function's return and calls the span exporter synchronously. The exporter sends spans to AWS X-Ray through the Lambda runtime API. AWS X-Ray stores the spans and makes them available for query within 10 seconds.

**Cold start handling**: Lambda cold starts add latency to the first request. The OpenTelemetry SDK initialization takes 200 to 500 milliseconds during a cold start. To minimize the cold start impact, the SDK initialization is done lazily: the first span is created immediately with a minimal context, and the full SDK initialization completes in the background.

**Memory constraints**: Lambda functions have a minimum of 128 MB of memory. The OpenTelemetry SDK adds approximately 20 MB to the function's memory footprint. The batch span processor queue is set to a maximum of 256 spans to stay within the memory budget. If the queue is full, spans are dropped rather than blocking the function's exit.

**Trace visualization**: Traces are visualized in AWS X-Ray's service map and trace list views. Each Lambda invocation appears as a single span. The complete trace shows the API Gateway span, the payment function span, and the notification function span. The trace waterfall reveals the total end-to-end latency and each function's contribution.

## Bonus Challenge: Zero-Cost Tracing for Internal Tooling

### Scenario

Your company runs a set of internal tools (CI/CD pipeline, deployment automation, infrastructure provisioning) that process 100 requests per hour. Each request involves 5 to 10 services. The internal tools team has no budget for tracing infrastructure. They need tracing for debugging, but they cannot spend any money.

**Task**: Design a zero-cost tracing solution for the internal tools. Describe the architecture, the sampling strategy, and the storage approach. Address how you would provide value without spending money on infrastructure.

**Model answer**:

**Architecture**: Use the OpenTelemetry SDK in each service with an in-memory span exporter. The in-memory exporter stores spans in the application's memory. When a request completes, the spans are serialized to a JSON file and stored on the local filesystem. A simple Python script runs as a cron job every hour, reads the JSON files from all services, and assembles traces.

**Sampling**: Trace every request at 100% because the volume is negligible (100 requests per hour, 5 to 10 services, 50 to 1,000 spans per hour). The total storage is under 1 MB per day.

**Storage**: Store traces as JSON files in the existing shared filesystem that the team already uses for CI/CD artifacts. Each file is named with the trace ID and contains all spans for that trace. The storage is effectively free because the shared filesystem already exists.

**Visualization**: Use a simple HTML page that reads the JSON files and renders a waterfall view using JavaScript. The HTML page is served from the shared filesystem. No server-side infrastructure is needed. The waterfall view is rendered entirely in the browser.

**Value**: Even without dedicated infrastructure, the tracing system provides significant value. When a CI/CD pipeline fails, the team can open the trace and see which service failed and why. The JSON files can be searched with grep for specific trace IDs. The team can debug issues in minutes instead of hours, all at zero cost.

**Limitations**: The zero-cost solution has no query capability beyond grep. There is no filtering, no aggregation, and no dashboard. The team can only debug individual traces by knowing the trace ID or the approximate time. For 100 requests per hour, this limitation is acceptable.
