## The Problem Story

Imagine you are building the next big e-commerce platform. Your catalog has 50 million products. A customer types "wireless noise-canceling headphones under $200 with long battery life" into the search bar and hits Enter.

You have three seconds before that customer loses patience and moves to a competitor. In those three seconds, you need to:

- Parse a messy natural language query
- Correct any spelling mistakes (they typed "noice cancelling")
- Understand intent (they want headphones, not earbuds, not speakers)
- Apply 17 different filters (price range, brand, rating, availability)
- Rank 12,847 matching products by relevance
- Account for stock levels, seller reputation, and promotional boosts
- Return the top 20 results with enough metadata for the UI to render

If you scan every product linearly, you need 50 million comparisons per query. Each comparison involves string matching, numeric comparisons, and scoring functions. A naive loop takes 200 to 500 milliseconds just for the scan. Then you need ranking, filtering, and aggregation on top. The total exceeds your latency budget by 10x.

This is the search problem. Every large-scale system faces it: how do you find the most relevant items from a massive collection in milliseconds?

## The Insight

Search looks like a lookup problem, but it is actually three separate problems stacked together:

1. **Discovery** -- find everything that might match the query
2. **Filtering** -- narrow results by explicit constraints
3. **Ranking** -- order results by relevance

The key insight is that you cannot solve all three with one data structure. Discovery needs an index that supports fast fuzzy matching. Filtering needs a different index that handles range scans and boolean predicates. Ranking needs yet another structure that supports real-time scoring.

The **Search System Pattern** separates these three concerns into distinct layers, each with its own data structures and scaling strategy. You build an **inverted index** for discovery, a **filter index** for constraints, and a **scoring layer** for ranking.

### The Library Analogy

Imagine a library with 5 million books. A patron asks for "books about space exploration for children published after 2015."

If you walk through every aisle reading titles, you never finish. Instead, the library has three systems:

- **The card catalog** -- an alphabetized index by subject, author, and title. You look up "space exploration" and get 400 card entries. This is the inverted index.
- **The metadata columns** -- each card shows publication year, target age, section. You flip through and keep only cards where year is after 2015 and age is children. This is the filter index.
- **The recommendation list** -- the librarian has a small notebook of frequently requested books, recent arrivals, and patron favorites. They sort the remaining 47 cards by this score. This is the ranking layer.

The library does not reshelve every book for each query. The catalogs and indexes are built once, updated as books arrive, and queried millions of times. A search system works the same way.

## Core Vocabulary

These terms appear throughout every search system design. Internalize them.

**Inverted Index.** A map from each unique term to the list of document IDs containing that term. For the query "wireless headphones," you look up both terms and intersect their posting lists. The inverted index is the heart of any text search system.

**Posting List.** The list of document IDs associated with a single term in the inverted index. For a common word like "headphones," this list might contain millions of IDs. Posting lists are stored compressed and ordered to enable fast intersection.

**Tokenization.** The process of breaking raw text into individual tokens or terms. "Wireless noise-canceling headphones" becomes `["wireless", "noise", "canceling", "headphones"]`. Tokenization handles whitespace, punctuation, hyphenation, and Unicode normalization.

**Normalization.** Transforming tokens into a canonical form so different variants match the same index entry. Lowercasing ("Headphones" becomes "headphones"), stemming ("running" becomes "run"), and lemmatization ("better" becomes "good") are all normalization techniques.

**TF-IDF (Term Frequency-Inverse Document Frequency).** A scoring formula that measures term importance. TF counts how often a term appears in a document. IDF measures how rare the term is across all documents. Rare terms that appear frequently in a document get high scores. Stop words like "the" get near-zero IDF and contribute nothing.

**BM25 (Best Matching 25).** An improved version of TF-IDF used by modern search engines including Elasticsearch and Lucene. BM25 adds saturation (diminishing returns for very high term frequency) and field-length normalization. It consistently outperforms TF-IDF in practice.

**Index Segmentation.** Dividing the inverted index into segments, where each segment is an immutable sorted collection of posting lists. New documents write to a small in-memory segment. When memory fills, the segment is flushed to disk. Background merging consolidates segments over time. This enables high write throughput without blocking reads.

**Query Rewriting.** The process of transforming the raw query string before index lookup. Rewriting includes spell correction ("noice" becomes "noise"), synonym expansion ("headphones" becomes `["headphones", "earphones", "cans"]`), and query relaxation (removing restrictive terms if too few results match).

**Scoring Function.** The formula that computes a relevance score for each matching document. BM25 is the most common base scorer, but production systems add dozens of signals: recency boost, popularity boost, personalization score, click-through rate, and business rules.

**Filter Bitmap.** A specialized data structure (often a roaring bitmap) representing which documents satisfy a categorical or range constraint. Filter bitmaps can be combined with boolean logic (AND, OR, NOT) in microseconds using bitwise operations.

## How It Works

A search query moves through the system in a well-defined sequence. Each step transforms the data and passes it to the next stage.

### Step 1: Query Intake

The raw query string arrives from the application layer. The search service parses it to extract structured fields: the text portion, any prefixed field filters (like `brand:Sony`), faceted navigation selections, sorting preferences, and pagination parameters. This parsing happens in under one millisecond.

Modern search systems support rich query syntax. Elasticsearch uses JSON query DSL. Google uses special operators like `site:`, `filetype:`, and quotes for exact phrases. Your system defines its own query syntax based on the domain.

### Step 2: Tokenization and Normalization

The query text is broken into tokens. Each token is normalized to match how documents were indexed. Standard normalization includes:

- **Lowercasing**: "Headphones" becomes "headphones"
- **Unicode normalization**: "café" (composed) and "café" (decomposed) both normalize to the same form
- **Stop word removal**: common words like "the", "a", "for" are removed (though modern systems often keep them with low weight)
- **Stemming**: "running", "runner", "ran" all stem to "run"
- **Synonym expansion**: "laptop" expands to `["laptop", "notebook", "ultrabook"]`

Tokenization rules depend on the language. CJK (Chinese, Japanese, Korean) languages require special tokenizers because words are not separated by spaces. English uses whitespace and punctuation splitting.

### Step 3: Query Rewriting

Before hitting the index, the system rewrites the query to improve recall or precision. Two common strategies are:

- **Spell correction**: The system computes edit distance between query tokens and indexed terms. If "noice" has low document frequency but "noise" has high frequency, the system replaces "noice" with "noise" and adds a "did you mean" suggestion.
- **Query relaxation**: If the original query returns fewer than N results (say, fewer than 10), the system drops the least selective term and retries. This continues until enough results are found or all relaxed variants are exhausted.

Query rewriting is implemented as a pipeline of plugins. Each plugin can modify the query, add alternative queries, or suggest corrections. The rewritten query is sent to the index lookup stage.

### Step 4: Inverted Index Lookup

Each normalized token is looked up in the inverted index. The index returns the posting list: a sorted array of document IDs that contain the term. For a query with multiple terms, the system computes the intersection of their posting lists.

Posting list intersection uses a skiplist algorithm. Both lists are sorted ascending. The algorithm walks both lists simultaneously, advancing the list with the smaller current ID. When IDs match, the document is added to the candidate set. For common terms with long posting lists, this takes 1 to 5 milliseconds.

The intersection produces an intermediate result set of candidate document IDs. This set is delivered to the filter engine.

### Step 5: Filter Application

The candidate document set is intersected with filter constraints. Filters are precomputed as bitmaps or range trees. Each filter type uses a specific data structure:

- **Categorical filters** (brand, color, category): stored as roaring bitmaps. AND-ing two filters is a bitwise operation on compressed integer arrays.
- **Range filters** (price, date, rating): stored as B-tree or segment tree. Range queries return a bitmap of matching documents.
- **Geo filters** (within 10 miles of a point): stored in an R-tree or geohash index. The system computes which documents fall within the geographic boundary.

Filters are combined using boolean logic. The system builds a filter expression tree. For example, `(brand:Sony OR brand:Bose) AND price < 200 AND rating >= 4.0` becomes a tree of bitmap operations. The entire filter evaluation completes in under 2 milliseconds for most queries.

### Step 6: Scoring and Ranking

Each document in the filtered candidate set receives a relevance score. The base score comes from BM25, which considers:

- How many times each query term appears in the document (term frequency)
- How rare each term is across all documents (inverse document frequency)
- The length of the document relative to the average document length

BM25 score for a single term is:

```
score = IDF * (TF * (k1 + 1)) / (TF + k1 * (1 - b + b * docLen / avgDocLen))
```

Where `k1` (typically 1.2) controls term frequency saturation and `b` (typically 0.75) controls length normalization.

On top of BM25, production systems add boost factors:

- **Recency boost**: newer documents get a multiplier. A document published today scores 10 percent higher than one from a year ago.
- **Popularity boost**: documents with more views, purchases, or clicks get a multiplier. The top 1 percent of products by sales get a 1.5x boost.
- **Personalization**: user-specific signals like purchase history, browsing behavior, and location adjust scores. A user in Tokyo sees local results ranked higher.
- **Business rules**: promoted content, sponsored results, and editorial picks are injected with fixed scores or multipliers.

The final score is a weighted sum of all signals:

```
finalScore = w1 * BM25 + w2 * recency + w3 * popularity + w4 * personalization
```

Weights are tuned through A/B testing and machine learning.

### Step 7: Aggregation and Faceting

After ranking, the system computes aggregations for faceted navigation. For the query "headphones," the UI needs to show:

- How many results are in each price bracket (0-50, 50-100, 100-200, 200+)
- How many results per brand
- How many results per rating level

Aggregations are computed from the candidate set before pagination. This ensures the facet counts are accurate for the current query. Some systems use approximation (like HyperLogLog) to compute cardinalities for large result sets without scanning all documents.

### Step 8: Result Building and Pagination

The final step assembles the response. The ranked document IDs are mapped to full document metadata by fetching from a document store or cache. The response includes:

- The top N results (default 20, configurable up to 10,000)
- Total result count (exact or approximate)
- Facet counts for each category
- Did-you-mean suggestions if spell correction was applied
- Pagination tokens for deep paging

Deep pagination is expensive. Requesting page 100 (results 1,981 to 2,000) requires the system to compute scores for all matching documents, sort them, then skip the first 1,980. Elasticsearch limits `from + size` to 10,000 by default. For deep paging, systems use search-after cursors that avoid re-scanning the full result set.

## Variants

The Search System Pattern has several variants. Each variant optimizes for a different workload.

### Variant 1: Full-Text Search Engine

This is the classic variant used by Elasticsearch, Apache Solr, and Amazon CloudSearch. It indexes unstructured text documents and supports rich query syntax, fuzzy matching, phrase queries, and relevance scoring.

**Key characteristics:**

- Document-oriented: each indexed item is a JSON document with multiple fields
- Schema-flexible: fields can be added without reindexing
- Real-time indexing: documents are searchable within one second of ingestion
- Distributed: shards are spread across multiple nodes for horizontal scaling

The full-text search engine variant is best for log analytics, content management systems, and knowledge bases.

### Variant 2: E-Commerce Search

E-commerce search adds filtering, faceting, and merchandising on top of full-text search. The query "blue Nike running shoes under $100" combines text search with structured filters and ranking boosts.

**Key differences from full-text search:**

- **Faceted navigation**: the UI shows category counts, price ranges, and brand filters that update with each query
- **Merchandising**: certain products are boosted or pinned based on business rules (promotions, seasonal items, sponsored listings)
- **Inventory-aware**: out-of-stock items are demoted or hidden
- **Personalized ranking**: search results are influenced by the user's purchase history, browsing behavior, and preferences

E-commerce search is the most demanding variant because it combines the complexity of text search with the strict latency requirements of a web application (under 200 milliseconds).

### Variant 3: Geospatial Search

Geospatial search finds documents near a location. Think "find all coffee shops within 1 mile of my current location." This variant replaces text-based ranking with distance-based scoring.

**Key data structures:**

- **Geohash**: divides the earth into a grid of cells. Each cell is encoded as a string. Nearby locations share a prefix. A query for "within 1 mile" translates to a prefix search on the geohash index.
- **R-tree**: a tree data structure that groups nearby geometries into bounding boxes. Used by PostGIS and MongoDB for geospatial queries.
- **S2 Geometry**: Google's library for spherical geometry. Used by Google Maps and Foursquare. S2 decomposes the sphere into cells at multiple levels of detail.

Geospatial search is critical for ride-sharing apps (find nearby drivers), restaurant discovery (find nearby restaurants), and logistics (find warehouses near a delivery address).

### Variant 4: Hybrid Search (Vector + Keyword)

Modern search systems combine keyword search with vector (semantic) search. A vector embedding model converts both documents and queries into high-dimensional vectors (typically 768 or 1,536 dimensions). The system finds documents whose vectors are closest to the query vector.

**How hybrid search works:**

1. The query is embedded using a model like text-embedding-3-small from OpenAI or a BERT-based model
2. The vector index (built with HNSW or IVF algorithms) finds the top K nearest neighbors
3. The keyword index finds matching documents using BM25
4. A fusion algorithm (like Reciprocal Rank Fusion) combines both result sets into a single ranked list

Hybrid search provides better results than either approach alone. Keyword search handles exact matches and rare terms. Vector search handles synonyms, paraphrases, and conceptual matches. The combination catches both.

## Mental Model Check

Test your understanding of the Search System Pattern with these questions.

### Question 1

You are designing search for a legal document repository with 10 million documents. Lawyers need to find exact phrases like "fourth amendment protection." What indexing strategy do you use?

**Answer.** You need a phrase query index that stores both term and position information. The inverted index must store, for each term occurrence, the document ID and the position within the document. To query "fourth amendment protection," you look up all three terms, get their positions, and verify that the positions are consecutive. This is called a phrase query, and Lucene supports it through the `match_phrase` query type.

### Question 2

Your search system returns results in 50 milliseconds with 95 percent accuracy. But the product team wants 99 percent accuracy. What do you trade off?

**Answer.** Higher accuracy typically requires more complex query processing, additional index lookups, and more sophisticated ranking. You might need to add synonym expansion, improve spell correction, implement personalization, or use a neural reranker. Each of these adds latency. The trade-off is between accuracy and speed. You can mitigate this with a two-phase approach: a fast first pass (50ms) selects 1,000 candidates, then a slower second pass (200ms) reranks the top 100. The user sees the first pass results immediately while the reranker refines the order.

### Question 3

You are building search for a social media platform. Users post 100,000 messages per minute. How do you index new content fast enough?

**Answer.** You use index segmentation with a small in-memory buffer. New posts are written to an in-memory segment (refresh interval: 1 second). The in-memory segment is searchable immediately. When it reaches 50 MB, it is flushed to disk as a read-only segment. Background threads merge small segments into larger ones (up to 5 GB per segment). This approach supports 100,000 writes per minute because each write is a simple append to the in-memory buffer, not a full index rebuild.

### Question 4

Your e-commerce search shows 10,000 results for "shoes." Users never scroll past page 3. How do you optimize the system?

**Answer.** You introduce a two-phase ranking strategy. The first phase scores all matching documents using a lightweight scoring function (BM25 only, no boosts). This completes in under 10 milliseconds. The second phase applies merchandising boosts, personalization, and business rules to only the top 500 documents (25 pages worth). The remaining 9,500 documents are never fully scored. This saves 80 percent of ranking computation for queries with many matches.

### Question 5

A competitor claims their search is 10x faster than yours. What questions do you ask to evaluate their claim?

**Answer.** You ask four questions: (1) What is the index size relative to yours? They might have a smaller index. (2) What is the recall rate? They might be returning fewer relevant results. (3) What query types are they testing? They might be optimizing for simple queries and ignoring complex ones. (4) What is their indexing freshness? They might be using stale indexes that are not updated in real time. Speed without context is meaningless. The relevant metric is precision at N within a latency budget.

## Summary

The Search System Pattern is a layered architecture that separates discovery, filtering, and ranking. The inverted index enables fast term lookup. Filter bitmaps enable efficient constraint application. The ranking layer combines relevance scoring with business signals.

Key takeaways:

- Search is three problems -- discovery, filtering, ranking -- each requiring different data structures
- The inverted index with posting lists is the core data structure for text search
- BM25 is the standard scoring function, but production systems add many boost signals
- Index segmentation enables high write throughput without blocking reads
- Variants include full-text search, e-commerce search, geospatial search, and hybrid vector search
- Query rewriting (spell correction, synonym expansion, query relaxation) improves recall
- Faceted navigation requires aggregation computation on the candidate set
- Deep pagination is expensive; use search-after cursors for deep pages

## Deep Dive Examples

### Example 1: Processing a Multi-Term Query

Consider the query "blue suede shoes." The query processor tokenizes this into three terms: "blue," "suede," and "shoes." Each term is looked up in the inverted index.

Assume the posting lists are:

- "blue": [1, 4, 7, 12, 15, 22, 31, 42, 55, 67, 78, 90, 101, ...] -- 150,000 documents
- "suede": [2, 7, 12, 18, 31, 42, 55, 67, 90, ...] -- 12,000 documents
- "shoes": [1, 3, 7, 12, 15, 22, 31, 42, 55, 67, 90, ...] -- 200,000 documents

The intersection planner orders lookups by rarest term first: "suede" (12,000) then "blue" (150,000) then "shoes" (200,000). The intersect of "suede" and "blue" produces 8,000 matching documents. Intersecting with "shoes" reduces that to 6,000. Only 6,000 documents out of 500,000 contain all three terms.

Each of those 6,000 documents receives a BM25 score. A document where "suede" appears in the title and "blue" and "shoes" appear in the description gets a higher score than a document where all three terms appear only in the description.

After scoring, the top 20 documents are returned. The remaining 5,980 are available for pagination.

### Example 2: Handling a Misspelled Query

The query "blew suede shooz" enters the system. The tokenizer produces "blew," "suede," and "shooz."

The inverted index lookup for "blew" finds no matches (zero documents contain this term). "suede" finds 12,000 documents. "shooz" finds zero matches.

The spell correction plugin runs. It computes the Levenshtein distance between "blew" and all indexed terms. "blew" is closest to "blue" (distance 1: w-to-u transposition) and "blew" (distance 0: it is a valid English word, so the system also keeps it as-is). The system adds "blue" as an alternative term with a suggestion.

Similarly, "shooz" is corrected to "shoes" (distance 1: z-to-s substitution).

The corrected query becomes: `("blue" OR "blew") AND "suede" AND ("shoes" OR "shooz")`

The system runs the corrected query and also returns a "Did you mean: blue suede shoes?" suggestion.

### Example 3: Personalized Ranking in Action

Two users search for "coffee" on the same e-commerce platform.

User A has a purchase history that includes espresso machines, French presses, and Starbucks gift cards. User B has a purchase history that includes instant coffee, tea bags, and mugs.

The base BM25 score for each product is identical for both users. But the personalization layer adds boost scores:

- For User A: espresso beans get +2.0 boost, Starbucks products get +1.5 boost, premium coffee makers get +1.3 boost
- For User B: instant coffee gets +2.0 boost, affordable options get +1.3 boost, bundle deals get +1.2 boost

User A sees espresso beans, whole bean coffee, and premium brewers at the top. User B sees instant coffee, value packs, and coffee-mug bundles at the top. Both users find relevant results, but the relevance definition is personalized.

## Common Questions

### Why not use a database full-text index?

Databases like PostgreSQL support full-text search through tsvector indexes. For many applications, this is sufficient. The database approach breaks down at scale for three reasons:

1. **Relevance scoring**: Database full-text search lacks advanced scoring models. PostgreSQL uses a simple TF-IDF variant. It does not support BM25, field boosting, or custom scoring signals.

2. **Query features**: Databases do not support fuzzy matching, synonym expansion, phrase queries with slop, or regular expression search. These features require the inverted index with positional information.

3. **Query throughput**: A database full-text index consumes CPU and I/O resources that the database needs for transactional workloads. Running 1,000 full-text queries per second on a database that also handles OLTP transactions is not practical.

The threshold for moving from database full-text to a dedicated search engine is approximately 100,000 documents or 100 QPS.

### How does search handle multilingual content?

Multilingual search requires language-specific analyzers. Each language has different tokenization rules, stop words, and stemming algorithms. Elasticsearch provides analyzers for over 30 languages.

For documents in multiple languages, the standard approach is to index each language field separately:

```
document {
  title_en: text (English analyzer)
  title_es: text (Spanish analyzer)
  title_fr: text (French analyzer)
  content_en: text (English analyzer)
  content_es: text (Spanish analyzer)
  content_fr: text (French analyzer)
}
```

The query is analyzed with each language analyzer and searched across all language fields. Language detection (either from the user's profile or from the document content) determines which fields to boost. A user searching in English gets higher scores from English fields.

### What is the relationship between search and recommendations?

Search and recommendations are complementary. Search finds what the user explicitly asks for. Recommendations find what the user might want based on their behavior.

Modern systems blur the line. Search results are personalized based on recommendation signals. Recommendations use search indexes to find candidate items. The Netflix and Airbnb case studies from the real-world section demonstrate this convergence.

## Performance Characteristics

### Query Latency Breakdown

The total query latency is the sum of each stage's processing time. Here is a typical breakdown for a production query:

| Stage | Time (ms) | Cumulative (ms) | % of Total |
|-------|-----------|-----------------|------------|
| Network and auth | 2 | 2 | 2% |
| Query parsing | 1 | 3 | 1% |
| Query rewriting | 2 | 5 | 2% |
| Term lookup | 5 | 10 | 5% |
| Posting list intersection | 8 | 18 | 8% |
| Filter application | 3 | 21 | 3% |
| BM25 scoring | 35 | 56 | 35% |
| Recency/popularity boost | 10 | 66 | 10% |
| Aggregation | 20 | 86 | 20% |
| Result merging | 5 | 91 | 5% |
| Response building | 9 | 100 | 9% |
| **Total** | **100** | **100** | **100%** |

Scoring is the dominant cost at 35 percent. Aggregations are second at 20 percent. Term lookup and posting list intersection together account for 13 percent. This breakdown guides optimization: to reduce latency, focus on scoring and aggregation first.

### Throughput Scaling

Query throughput scales with the number of replicas. Each additional replica doubles the query capacity of the shard. The throughput formula is:

```
Total QPS = QPS_per_shard * num_shards * (1 + num_replicas)
```

If a single shard handles 100 QPS, and you have 5 shards with 1 replica each, the total throughput is:

```
100 * 5 * (1 + 1) = 1,000 QPS
```

This assumes the coordinator nodes and network are not bottlenecks. In practice, a well-designed cluster achieves 80 to 90 percent of the theoretical maximum.

### Memory Footprint

The memory footprint of a search index is dominated by:

- **Term dictionary FST**: 10 to 20 percent of the term text size. For 10 million unique terms with average length 10 characters: 200 to 400 MB.
- **Posting lists (cached)**: variable. The OS page cache caches the most frequently accessed posting list blocks. For a 1 TB index, a 64 GB page cache caches approximately 6 percent of the index.
- **Filter cache**: 2 to 4 KB per unique filter value. For 10,000 brands and 500 categories: 50 MB.
- **Query cache**: 10 to 100 KB per cached query. For 1,000 cached queries: 100 MB.
- **Field data cache**: 2 to 4 MB per field for aggregations. For 50 aggregation fields: 150 MB.

Total cache memory for a typical production node: 10 to 20 GB of RAM beyond the JVM heap.

The key architectural insight: both search and recommendations benefit from the same inverted index infrastructure. The difference is in the ranking layer. Search uses query-dependent scoring (BM25). Recommendations use query-independent scoring (collaborative filtering, content-based filtering).
