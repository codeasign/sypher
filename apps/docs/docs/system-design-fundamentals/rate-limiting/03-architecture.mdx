# Rate Limiting at Scale

## From Naive to Production

Rate limiting is a progression. You start with a simple counter in memory, discover it breaks the moment you add a second server, and iterate toward a distributed architecture that coordinates rate decisions across thousands of nodes, multiple data centers, and global user populations.

### Naive: In-Memory Counters on a Single Server

The simplest rate limiter lives in the application process memory. A dictionary of counters, a timer that resets them every minute, and a check before each request.

```
┌──────────────────────────────────────────────────┐
│              Single Application Server            │
│                                                    │
│  ┌──────────────────────────────────────────────┐ │
│  │          In-Memory Rate Limiter              │ │
│  │                                              │ │
│  │  process: {                                  │ │
│  │    "user:42:api:minute": 85,                 │ │
│  │    "user:42:api:hour": 1200,                 │ │
│  │    "ip:10.0.0.1:api:minute": 32,             │ │
│  │    "user:99:api:minute": 100                 │ │
│  │  }                                           │ │
│  │                                              │ │
│  │  Reset timer: every 60 seconds               │ │
│  │  Memory: ~200 bytes per counter entry        │ │
│  │  Max entries: unlimited (until OOM)          │ │
│  └──────────────────────────────────────────────┘
│                    │
│                    ▼
│  ┌──────────────────────────────────────────────┐ │
│  │           Request Handler                     │ │
│  │  if counter[user][minute] < limit:            │ │
│  │    counter[user][minute]++                    │ │
│  │    process_request()                          │ │
│  │  else:                                        │ │
│  │    return 429 Too Many Requests               │ │
│  └──────────────────────────────────────────────┘ │
│                                                    │
│  Throughput: ~500K checks/sec (single-threaded)   │
│  Latency: <1 microsecond per check                 │
│  Memory: grows unbounded (no eviction)             │
└──────────────────────────────────────────────────────┘
```

**What works at this level:**
- Sub-microsecond latency per check (pure dictionary lookup)
- Zero network calls (everything is local)
- Simple to implement: 50 lines of Python or Java
- Atomic increment with `synchronized` or `Lock` in single-threaded context

**What breaks immediately:**
- **Server restart loses all counters** — every user gets a fresh window. An attacker can restart the server and bypass the limit.
- **No memory limit** — a malicious user can create millions of unique keys (e.g., spoofed IPs) and cause an OOM crash.
- **Single server only** — the moment you scale to two servers, each server has its own counter. A user can send 100 requests to server A and 100 to server B, hitting 200 total while each server thinks they only sent 100.

### Step 1: Sticky Sessions (Wrong Direction)

The naive fix is to pin each user to a specific server using sticky sessions. This keeps the in-memory counter working — but it creates a worse problem.

```
┌────────────────────────────────────────────────┐
│              Load Balancer                       │
│         sticky session: hash(user_id) % N       │
└────────────────────┬───────────────────────────┘
                      │
         ┌────────────┼────────────┐
         │            │            │
         ▼            ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │ Server A │ │ Server B │ │ Server C │
   │          │ │          │ │          │
   │ user:42  │ │ user:99  │ │ user:155 │
   │ count: 85│ │ count: 92│ │ count: 78│
   │          │ │          │ │          │
   └──────────┘ └──────────┘ └──────────┘

Problems:
- Uneven load: some servers get more users than others
- Server failure = all state lost for that user partition
- Scaling up (adding servers) remaps hash → limits reset for moved users
- Deployment (rolling update) drains servers → state lost
```

Sticky sessions for rate limiting are worse than the original problem. You trade a counting problem for a distribution and availability problem, and you still lose all counters on deploy.

### Step 2: Centralized Redis Counter

The first real improvement: move the counter out of the application process and into a shared Redis instance.

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  Server A    │  │  Server B    │  │  Server C    │
│              │  │              │  │              │
│  INCR key    │  │  INCR key    │  │  INCR key    │
│  TTL 60s     │  │  TTL 60s     │  │  TTL 60s     │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                 │                 │
       └─────────────────┼─────────────────┘
                         │
                  ┌──────▼───────┐
                  │   {Redis}    │
                  │  Single node │
                  │              │
                  │  INCR "user: │
                  │   42:minute" │
                  │  → returns 86│
                  │  TTL: 60s   │
                  └──────────────┘
```

**Advantages:**
- All servers see the same counter — accurate rate limiting
- Atomic operations (INCR, EXPIRE) prevent race conditions
- Counters survive server restarts
- Adding or removing servers does not affect counters

**Disadvantages:**
- Every request incurs a network round-trip to Redis (~1ms)
- Redis is a single point of failure — if Redis dies, rate limiting stops
- Redis can become a bottleneck at high throughput (single-threaded event loop)
- Network latency adds ~0.5-2ms to every request

## Production Architecture: The Hybrid Global Rate Limiter

A production-grade rate limiter combines three layers: a centralized Redis cluster for global accuracy, per-node local rate limiters for latency-sensitive enforcement, and a hybrid approach that blends both. This design gives you the accuracy of global counters and the speed of local counters, with graceful degradation when either layer fails.

### How the Hybrid Decision Logic Works

The hybrid approach gives you the best of both worlds. Here is the exact algorithm:

**Local token bucket** (per-node, in-memory):
- Each node maintains a token bucket for every rate limit key.
- A background process refills tokens at the configured rate (e.g., 10 tokens per second).
- If the local bucket has tokens, the request is allowed immediately — zero network calls.
- If the local bucket is empty, the node checks the global counter before denying.

**Global Redis counter** (cluster-wide, shared state):
- The Redis cluster maintains the authoritative counter for each rate limit key.
- On every request, the node asynchronously sends an INCR command to Redis (fire-and-forget).
- Redis responds with the current count. If it exceeds the limit, the node returns 429.
- Redis counters have a TTL matching the rate limit window (e.g., 60 seconds for per-minute limits).

**Graceful degradation:**
- If Redis is unreachable (network partition, failover, overload), the node falls back to local-only rate limiting.
- In degraded mode, each node allows up to its local token bucket limit. Total throughput may exceed the configured limit by (N * local_bucket_size) during the degradation window.
- The system logs the Redis failure and alerts the operations team.
- Once Redis recovers, the system returns to normal operation — counters are recalculated from scratch.

### Pseudocode: Hybrid Decision Logic

```
allow_request(key, user_id):
    # Step 1: Fast path — local token bucket (0 microseconds)
    if local_bucket.try_consume():
        redis.incr_async("ratelimit:" + key + ":" + user_id + ":" + window)
        return ALLOW

    # Step 2: Slow path — check global counter
    count = redis.get("ratelimit:" + key + ":" + user_id + ":" + window)
    if count >= max_requests:
        return DENY  # 429 Too Many Requests

    redis.incr(key)
    return ALLOW

    # Step 3: Degraded mode
    on redis.ConnectionError:
        log("Redis unreachable, falling back to local-only")
        return ALLOW
```

## Four Integration Patterns

Rate limiting touches every layer of your system. You must integrate it with the API gateway, the CDN, microservices, and the authentication service. Each integration has different requirements for latency, accuracy, and failure modes.

**API gateway rate limit configuration (Kong example):**

```yaml
# Kong declarative config for rate limiting
_format_version: "3.0"
services:
  - name: orders-service
    host: orders.internal
    port: 8080
    routes:
      - name: orders-route
        paths:
          - /api/v1/orders
    plugins:
      - name: rate-limiting
        config:
          second: null
          minute: 1000
          hour: 50000
          policy: redis
          redis_host: redis-cluster.rate-limiting.svc
          redis_port: 6379
          redis_database: 0
          redis_timeout: 2000
          fault_tolerant: true
          hide_client_headers: false
          limit_by: consumer  # consumer, credential, ip, service, path, header
```

**Key configuration parameters:**
- **policy**: `redis` for production (shared state), `local` for development (per-node, inaccurate)
- **fault_tolerant**: If true, allow the request when Redis is unreachable (degraded mode)
- **limit_by**: How to identify the client — by consumer (API key), credential, IP, or a custom header
- **hide_client_headers**: Controls whether `X-RateLimit-*` headers are sent to the client

**Response headers the API gateway should return:**

```
HTTP/1.1 200 OK (or 429 Too Many Requests)
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 842
X-RateLimit-Reset: 1625097600
Retry-After: 37
```

These headers let the client understand their current rate limit state and back off appropriately. The `Retry-After` header on 429 responses tells the client how many seconds to wait before retrying.

**CDN edge rate limiting strategies:**

- **IP-based rate limiting**: The simplest edge strategy. Block IPs that exceed a threshold (e.g., 100 requests per minute). This is effective against DDoS attacks from a small number of IPs but ineffective against distributed attacks (thousands of unique IPs each sending one request).
- **Geo-based rate limiting**: Block or throttle traffic from regions where you do not have users. If your application is US-only, blocking traffic from outside North America at the edge eliminates 30-50% of automated abuse.
- **Path-based rate limiting**: Protect expensive endpoints (search, checkout, data export) with stricter limits at the edge. A search endpoint might have a 10 req/min limit per IP, while a static page has no limit.
- **User-Agent rate limiting**: Block or throttle requests with suspicious User-Agent headers (e.g., curl, python-requests, or missing User-Agent). This catches scripted attacks that do not bother to spoof a browser User-Agent.

**Cloudflare rate limiting example (abbreviated):**

```json
{
  "rules": [
    {
      "expression": "(http.request.uri.path eq \"/api/v1/login\")",
      "actions": [{
        "mode": "rate_limit",
        "rate_limit": {
          "characteristics": ["ip.src"],
          "requests_per_period": 10,
          "period": "60s",
          "mitigation_timeout": 300
        }
      }]
    }
  ]
}
```

**Microservice rate limiting strategies:**

- **Per-caller limits**: Each service tracks how many requests each upstream service sends. This prevents one noisy neighbor from starving others. The order service tracks requests from the user service separately from requests from the admin service.
- **Per-endpoint limits**: Expensive endpoints (e.g., report generation, data export) get stricter limits than cheap endpoints (e.g., list items, get by ID). A report generation endpoint might have a 10 req/min limit while a list endpoint has 1000 req/min.
- **Circuit breaker integration**: Rate limiting and circuit breakers work together. The rate limiter prevents the caller from sending too many requests. The circuit breaker prevents the callee from being overwhelmed when it is already degraded. Configure the circuit breaker to open when the rate limiter is returning 429 for more than 50% of requests.
- **Propagate rate limit state**: Include rate limit headers in inter-service HTTP calls. The caller sees `X-RateLimit-Remaining: 12` and can slow down before hitting the limit. This prevents cascading 429 responses.

The service mesh approach (Envoy + Istio) lets you enforce rate limits at the network level without modifying application code. Every sidecar proxy enforces limits before the request reaches the application process.

**Auth rate limiting best practices:**

- **Never reveal whether the username exists**: Return a generic "Invalid credentials" message regardless of whether the username is valid. If you return "User not found" for invalid usernames and "Invalid password" for valid usernames, you are helping attackers enumerate valid accounts.
- **Account lockout with exponential backoff**: After 5 failed attempts, lock for 15 minutes. After 10, lock for 1 hour. After 20, lock until manually unlocked. This makes brute-force attacks impractical.
- **CAPTCHA escalation**: Instead of locking accounts outright, introduce CAPTCHA after the first rate limit violation. Humans can solve CAPTCHAs. Automated scripts cannot. Apply CAPTCHA to all requests from the suspected IP or account for the next hour.
- **Separate Redis instance for auth**: Auth rate limiting has different latency and durability requirements than general API rate limiting. Use a dedicated Redis instance with `maxmemory-policy: allkeys-lru` and ensure it has enough capacity to never evict active rate limit keys.

### Eventually Consistent Counters

The key insight for multi-region rate limiting is that you cannot have strong consistency across regions without adding significant latency. Every request to a global counter would need to wait for responses from all regions — adding 100-300ms of latency. Instead, you accept eventual consistency and account for the staleness.

**CRDT (Conflict-Free Replicated Data Type) counters:**
- Each region maintains a local counter for each rate limit key.
- Every 5 seconds, regions exchange counter values using a gossip protocol.
- When two regions report different values for the same key, the merge rule is MAX (the highest value wins).
- Each region keeps a local replica of the merged counter from other regions.

**Staleness budget:**
- If the maximum staleness is 10 seconds and users can send 10 requests per second from each region, the worst-case overshoot is 3 regions x 10 seconds x 10 req/s = 300 extra requests.
- To stay within a hard limit of 100 req/min, configure the limit to 70 on each region. This gives you a 30% safety margin for staleness.
- For stricter limits, reduce the staleness window (more frequent syncs) or increase the safety margin.

**Handling user migration:**
- A user starts in US-East, sends 50 requests, then switches to EU-West.
- The EU-West counter may not have the latest US-East count for up to 10 seconds.
- During this window, the user can exceed the limit by sending requests in EU-West.
- After the sync, the EU-West counter merges with the US-East count, and the user is correctly limited.

### Global Coordination Strategies

You have three strategies for coordinating rate limits across regions, each with different trade-offs.

**Strategy 1: Strict Global (Strong Consistency)**

Use a single Redis cluster in one region for all rate limit checks. Every request to every region must read from this cluster.

```
Pros:  Accurate, no overshoot
Cons:  +200ms latency for non-local regions
       Single point of failure
       Bandwidth costs (all regions send all requests to one Redis)
       Not viable for global applications
```

**Strategy 2: Per-Region with Safety Margin (Recommended)**

Each region has its own Redis cluster. Rate limits are configured lower than the true limit to account for staleness.

```
Pros:  Low latency (local Redis)
       No single point of failure
       Scales to any number of regions
Cons:  Overshoot possible during staleness window
       Requires careful safety margin calculation
       Users who switch regions can briefly exceed limits
```

**Strategy 3: Centralized with Local Cache**

A single authoritative Redis cluster in one region. Each region has a local Redis cache that serves rate limit checks with a configurable TTL. The local cache is refreshed asynchronously from the central cluster.

```
Pros:  Eventual accuracy (no permanent overshoot)
       Lower latency than Strategy 1
       Central cluster is the source of truth
Cons:  Central cluster is still a SPOF
       Local cache returns stale data during refresh interval
       Higher bandwidth costs than Strategy 2
```

**Recommendation:** Start with Strategy 2 (per-region with safety margin). It is the simplest, most resilient, and most scalable. The safety margin is a small downside (you leave some capacity unused) but the operational simplicity is worth it. Move to Strategy 3 only if you need tighter accuracy and can tolerate the central cluster dependency.

## Capacity Planning for Rate Limiting

Rate limiting infrastructure must be provisioned for peak traffic plus a safety margin. If the rate limiter itself becomes a bottleneck, the system is unprotected at the moment it needs protection most.

### Redis Memory for Rate Counters

Each rate limit counter in Redis consumes memory. The exact amount depends on the key structure, the value size, and the TTL.

```
Memory per counter entry:

  Key:    "ratelimit:user:42:api:1740960000"  → ~45 bytes
  Value:  "85"                                → ~3 bytes (as string)
  Redis overhead:                             → ~50 bytes (dict entry, etc.)
  TTL metadata:                               → ~20 bytes
  ──────────────────────────────────────────────
  Total per counter:                          → ~120 bytes

With 1 million active users and 3 rate limit keys per user:
  1,000,000 * 3 * 120 bytes = 360 MB

With 10 million active users:
  10,000,000 * 3 * 120 bytes = 3.6 GB

Additional keys:
  IP-based limits: 100,000 active IPs * 2 keys * 120 bytes = 24 MB
  Global limits: 100 endpoints * 1 key * 120 bytes = 12 KB
  Auth limits: 1,000,000 users * 4 keys * 120 bytes = 480 MB

Total estimated memory for 10M users:
  User limits:    3.6 GB
  IP limits:      24 MB
  Auth limits:    480 MB
  Global limits:  12 KB
  ─────────────────────
  Total:          ~4.1 GB
  With 50% buffer: ~6.2 GB
```

**Memory sizing rules:**
- Estimate 120-150 bytes per active counter entry.
- Count only active users (users who made a request in the last window), not total registered users.
- Add 50% buffer for unexpected spikes and key expansion.
- Add 30% for replication overhead (Redis replica nodes maintain their own copy).
- Set `maxmemory` on each Redis node to prevent OOM crashes.
- Configure `maxmemory-policy: allkeys-lru` so Redis evicts least-recently-used keys when memory is full.

### QPS Limits for the Rate Limiter Itself

The rate limiter must handle the full request throughput of your application. Each request triggers at least one Redis command (INCR or GET + INCR). If the rate limiter cannot keep up, requests are allowed through without checking — defeating the purpose.

```
Redis single-node throughput: ~100,000 INCR operations/sec
Redis cluster (3 shards): ~300,000 INCR operations/sec
Redis cluster (12 shards): ~1,200,000 INCR operations/sec

If your application handles 500,000 requests per second:
  - Each request triggers 1 INCR (rate limit check)
  - Total: 500,000 INCR operations/sec
  - Redis cluster needed: 5-6 shards (with 50% headroom)

If your application handles 2,000,000 requests per second:
  - Each request triggers 1 INCR
  - Total: 2,000,000 INCR operations/sec
  - Redis cluster needed: 20-25 shards
  - Consider: local rate limiter to reduce Redis load by 90%
```

**Reducing Redis load with local rate limiting:**
- If the local token bucket handles 90% of requests (fast path), only 10% reach Redis.
- 2,000,000 RPS application → 200,000 Redis INCR/sec → 2-3 shards instead of 20-25.
- This is the primary benefit of the hybrid approach described earlier.

- **Redis is the cache**: Cache active quotas in Redis with a 60-second TTL. This gives you sub-millisecond access during rate limit checks without hitting the database.
- **Local cache on each node**: Each application node maintains an in-memory cache of the quotas it accesses most frequently. This eliminates Redis lookups for the hot path. Refresh every 60 seconds.
- **Background sync**: A background job queries the database for changed quotas and updates Redis. This is more efficient than TTL-based expiry because it pushes changes immediately rather than waiting for the next cache miss.

## Operational Considerations

Rate limiting infrastructure must be monitored as carefully as the application it protects. Key metrics and alerting thresholds:

```
Rate Limiter Metrics:
  p50 latency: <1ms (local) / <2ms (Redis)        p99 latency: <5ms / <10ms
  Local cache hit ratio: target >90%               Quota cache hit ratio: target >99%
  Redis connection errors: 0/min                   Degraded mode activations: 0/min
  429 rate by endpoint/tier/region                 False positive rate: <0.1%

Alerting Thresholds:
  Redis latency > 50ms for 5 min             → Check cluster health and network
  Local cache hit ratio < 80%                → Increase local bucket size
  Degraded mode active > 1 min               → Redis is unreachable, investigate
  429 rate > 10% of total requests           → Spike or misconfigured limit
  Quota sync failure > 5 min                 → Users may have stale quotas
```
