# Rate Limiting — Trade-offs & Failure Modes

Every rate-limiting decision you make is a bet. You are betting that the
algorithm you choose, the storage backend you pick, and the parameters you
configure will protect your system without annoying legitimate users. Get it
wrong and you either let through a flood of requests or you block paying
customers on a Friday afternoon.

This page covers the uncomfortable part of rate limiting that interviewers and
system design books gloss over: the trade-offs, the failure modes, the
anti-patterns, and the cases where you should not rate-limit at all.

---

## The Core Tension: Accuracy vs. Memory Cost

The fundamental trade-off in rate limiting is between how precisely you can
measure a user's request rate and how much memory you are willing to spend to
do it.

A token bucket with a single counter per user uses very little memory, but it
cannot distinguish between 100 requests arriving in one second and 100 requests
arriving evenly over a minute. A sliding window log that stores every single
request timestamp gives you perfect accuracy, but it costs O(n) memory per user
where n is the number of requests in the window.

**The rule of thumb:** If you need to enforce a limit of 10 requests per minute
per user, a sliding window log is overkill. A token bucket or a sliding window
counter costs a fraction of the memory and produces behavior that is close
enough for most use cases. Reserve the expensive algorithms for cases where
boundary conditions matter — for example, billing-tier enforcement where a
burst at the window boundary could cost you revenue.

the best accuracy. You evaluate six dimensions: accuracy, memory, CPU cost,
burst tolerance, distributed friendliness, and implementation complexity.

Here is the detailed breakdown of when to choose each algorithm:

- **Token Bucket** — Use this when you want to allow bursts up to a configured
  capacity while enforcing a long-term average rate. It is the most widely used
  algorithm in production because it is simple, memory-efficient, and maps well
  to API pricing tiers. AWS and Stripe both use variants of token bucket.

- **Leaky Bucket (queue-based)** — Use this when you need to smooth out traffic
  and prevent any burst from reaching the downstream service. The leaky bucket
  enforces a constant output rate regardless of input pattern. This is useful
  when your backend is a database connection pool or a legacy system that cannot
  handle spikes.

- **Fixed Window** — Use this only as a first pass when you are prototyping and
  need the simplest possible implementation. The traffic spike at window
  boundaries (the "stampede" problem) makes it unsuitable for production systems
  with strict limits. Do not use it for billing-tier enforcement.

- **Sliding Window Counter** — Use this when you need reasonable accuracy
  without the memory cost of a full sliding window log. It stores only two
  counters per user (current window and previous window) and interpolates the
  count. This is a good default choice for most API rate limiters.

- **Sliding Window Log** — Use this only when you need exact per-request
  accounting and cannot tolerate even a 5% error margin. Financial systems and
  certain compliance workloads may require this. Be prepared to pay the memory
  cost and handle the cleanup of expired timestamps.

- **GCRA (Generic Cell Rate Algorithm)** — Use this if you want the burst
  behavior of token bucket with the constant-rate enforcement of leaky bucket,
  using a single integer per user. GCRA is the algorithm behind the popular
  `rate` library in Go and is used in telecommunications and high-performance
  API gateways. It is a strong candidate for distributed systems because the
  state update is a single atomic compare-and-swap on one key.

## Failure Modes: What Breaks and How Bad Is It

Rate limiters are infrastructure components. When they fail, the failure is
systemic — every request that goes through the affected node is impacted. You
need to understand the failure modes before you deploy, not after the pager
goes off at 3 AM.

### Redis Down

If you use Redis as your centralized rate-limit storage and Redis goes down,
your rate limiter has three options:

1. **Fail open** — Allow all requests through. Your downstream services see a
   traffic surge and may collapse. This is a cascading failure: the rate
   limiter failure causes an overload failure in the service it was meant to
   protect.

2. **Fail closed** — Reject all requests. Your API returns 429 Too Many Requests
   for every user, even legitimate ones. The entire service is effectively
   down. This is a self-inflicted denial of service.

3. **Fallback to local mode** — Each node uses a local in-memory counter with
   a reduced limit (for example, 50% of the configured limit). This is the
   safest option, but it requires you to build the fallback logic ahead of
   time and test it.

Most production systems choose option 3 with a circuit breaker that detects
Redis connectivity loss and switches to local mode automatically. The key
insight: a degraded rate limiter is better than a dead one.

### Counter Reset on Failover

When your rate limiter stores counters in Redis, and the Redis master node
fails, the replica that is promoted to master may have stale data. If the
replica was behind by even a few milliseconds, you lose the most recent counter
updates.

This means that after a failover, some users' counters reset to lower values
than they should be. Those users can send more requests than their limit allows
until the counters catch up. In a worst-case scenario, a user who was just
rate-limited before the failover suddenly has a clean slate and can send another
full batch of requests.

**Mitigation:** Write-ahead logging or using Redis with strong consistency
settings (WAIT command) reduces the window of data loss but cannot eliminate it
entirely. The practical approach is to accept the small window of over-limit
traffic and design your downstream services to handle brief surges.

### Distributed Race Condition

When you have multiple API gateway instances behind a load balancer, and each
instance checks and updates a shared Redis counter, you run into a classic
read-check-write race condition.

Two requests from the same user arrive at two different gateway instances at
nearly the same time. Both instances read the counter from Redis and see a
value of 9, with a limit of 10. Both instances increment the counter and write
back. The user gets 11 requests through, exceeding the limit by 1.

**The fix is always the same: make the check-and-increment atomic.**

- **Redis Lua scripting** — Write a Lua script that reads the counter, checks
  the limit, and increments in a single atomic operation on the Redis server.
  This is the most common approach.
- **Redis transactions with WATCH** — Use WATCH and MULTI/EXEC to implement
  optimistic locking. Retry if the transaction fails.
- **Redis atomic operations** — Use INCR and EXPIRE with a single command, and
  check the return value against the limit. This works for simple counters but
  cannot handle the more complex logic of token bucket or GCRA.

### Clock Skew

Clock skew is a problem for rate limiters that use timestamps — which is almost
all of them. If your Redis server's clock is 5 seconds ahead of your API
gateway's clock, a request that the gateway timestamps at T may be recorded at
T+5 in Redis. The rate limiter sees the request in a different time window than
the one you intended.

In a distributed system, clock skew between nodes causes these problems:

- **Inconsistent window boundaries** — A request near the edge of a window may
  be counted in the wrong window depending on which node processes it.
- **Negative time-to-live** — A counter may expire too early or too late if the
  clock on the Redis node that sets the TTL is different from the clock on the
  node that checks it.
- **Garbled sliding window log** — Timestamps in the log may not be in the
  order you expect, causing the pruning logic to remove the wrong entries.

**Mitigation:** Use NTP everywhere. Accept that millisecond-level skew is
inevitable and design your rate limiter to tolerate it. Use server-side
timestamps (Redis TIME command) rather than client-side timestamps for all
rate-limit decisions.

### Hot Key Sharding

When you shard your rate limit storage by user ID, you create hot keys. A
single high-traffic user (say, a popular API consumer) generates requests that
all hit the same shard. That shard becomes a bottleneck while other shards sit
idle.

**Mitigation:** Use Redis Cluster with automatic sharding, and accept that hot
keys are a fact of life. Alternatively, use a write-behind cache layer where
the rate limit counter is updated locally and asynchronously written to the
central store. This trades a small amount of accuracy for better shard balance.

## Anti-Patterns: What Not to Do

These anti-patterns appear in every system design interview and in many
production systems. Recognize them and avoid them.

### No Rate Limiting on Authentication Endpoints

Your login endpoint, password reset endpoint, and registration endpoint must be
rate-limited at the most restrictive tier. Attackers target these endpoints for
credential stuffing, brute-force attacks, and account enumeration.

Yet many systems leave authentication endpoints unthrottled because the
developers assume the rate limiter is for the "real" API endpoints. This is a
critical security gap. Authentication endpoints should be rate-limited by IP
address, by username, and by global total — three separate rate limiters
applied simultaneously.

### Per-IP Rate Limiting Only

Rate limiting by IP address alone is insufficient for three reasons:

1. **NAT and shared IPs** — A single corporate NAT gateway can serve thousands
   of employees. Rate limiting that IP to 100 requests per minute blocks an
   entire office building.

2. **Botnet distribution** — Attackers rotate through thousands of IP addresses
   to stay under per-IP limits. A credential stuffing attack from 10,000 IPs
   each making 1 request per second is invisible to an IP-based rate limiter.

3. **IPv6 allocation** — An IPv6 user can have a /64 subnet with 2^64
   addresses. Per-IP rate limiting is meaningless when one user can trivially
   rotate through millions of addresses.

**The fix:** Combine IP-based rate limiting with user-ID-based rate limiting,
API-key-based rate limiting, and behavioral rate limiting (velocity of
distinct actions). Use the most restrictive result across all limiters.

### Infinite Retries Without Backoff

When a client receives a 429 response, it should back off and retry later.
But many clients retry immediately, in a tight loop, or with no exponential
backoff. This makes the rate limiting problem worse: the rate limiter must
process each retry request, consuming CPU and network bandwidth, and the
retry storm may persist for minutes.

**The fix:** Return a `Retry-After` header in your 429 responses with a
specific number of seconds. Clients that respect this header (well-behaved
clients) will back off. For misbehaving clients, implement a secondary rate
limiter that detects retry storms and returns a longer penalty period.

### Fixed Window Boundaries for Billing Tier Enforcement

This is the most expensive anti-pattern in the list. If you use a fixed window
counter to enforce billing-tier API limits, a user who sends 1000 requests at
11:59:59 and another 1000 requests at 12:00:00 gets 2000 requests processed
in a two-second span, even though the limit is 1000 per hour.

The user sees no problem because the rate limiter did not block any requests.
But your billing system counts 2000 requests against the hourly limit — and
the user exceeds their tier. You either overcharge them or you eat the cost.

**The fix:** Use a sliding window algorithm for any rate limit that is tied to
billing or usage metering. The extra memory cost is negligible compared to the
revenue accuracy you gain.

### Global Rate Limiter as a Synchronization Point

If you make every request go through a centralized rate limiter before reaching
the application logic, you have created a global synchronization point. The
rate limiter becomes a single point of failure and a performance bottleneck.
Every request must wait for the rate limiter to respond before it can proceed,
which adds latency to every API call.

**The fix:** Use a two-tier rate limiter. A local in-memory rate limiter on
each API gateway instance handles the fast path for the vast majority of
requests. Only when the local counter approaches the limit (say, 80% of the
configured limit) does the gateway check the centralized Redis-based rate
limiter. This reduces the latency overhead and distributes the load.

## When NOT to Use Rate Limiting

Rate limiting is not always the right tool. Here are the cases where you should
reconsider applying rate limiting.

### Internal Service-to-Service Communication

If you have two services that communicate within the same data center or VPC,
and you control both services, rate limiting is often a sign of a deeper
problem. Why is the downstream service unable to handle the upstream service's
request rate? The correct fix is usually one of these:

- **Increase capacity** — Scale the downstream service horizontally or
  vertically to handle the expected load.
- **Add a queue** — If the downstream service can handle the load over time
  but not all at once, put a message queue between the services. This is
  better than rate limiting because it does not drop requests.
- **Fix the caller** — If the upstream service is sending requests in a tight
  loop, fix the polling loop, add caching, or implement a push-based
  notification pattern instead.

Rate limiting between internal services adds latency, operational complexity,
and a failure mode that can cause cascading outages. Reserve it for cases where
you genuinely cannot control the caller's behavior.

### Read-Heavy Internal Dashboards

Your internal monitoring dashboard queries your production database every 30
seconds. If you rate-limit the dashboard, you hide the very data you need to
diagnose problems. The dashboard should have a dedicated read replica or a
caching layer, not a rate limiter.

### When You Need Every Request

If you are processing payments, handling webhook deliveries, or processing
critical audit logs, rate limiting may cause you to drop the wrong request.
The last payment of the month should not be rejected because the user hit an
arbitrary rate limit. Use a queue with backpressure instead of a rate limiter
for these workloads.

## Summary of Key Trade-offs

Before you write any rate limiting code, answer these three questions:

1. **What is the cost of a false positive?** — If you block a legitimate user,
   how much revenue does that cost? If the answer is "a lot," prioritize
   accuracy over memory. Use a sliding window or GCRA.

2. **What is the cost of a false negative?** — If you let a rogue client
   through, how much damage can it do? If the answer is "a lot," prioritize
   defense in depth. Use multiple rate limiters in parallel (IP, user, global).

3. **What happens when the rate limiter itself fails?** — If Redis goes down at
   3 AM, does your system survive? Test this with a chaos engineering exercise
   before you go to production.

The table below summarizes the trade-offs you need to make for each decision
point in your rate limiter design.

| Decision | Trade-off | Recommendation |
|---|---|---|
| Storage backend | Latency vs. consistency | Redis with local fallback |
| Algorithm | Memory vs. accuracy | Sliding window counter as default |
| Distribution | Complexity vs. precision | Hybrid local + central |
| Failure handling | Availability vs. correctness | Degraded mode with circuit breaker |
| Scope | IP vs. user vs. global | All three, most restrictive wins |
| Retry policy | User experience vs. protection | Exponential backoff with jitter |
