# Understanding Rate Limiting

## The Launch Day Meltdown

Your startup launches at 9:00 AM. You have been coding for six months. The fridge is stocked with energy drinks. Your co-founder is live-tweeting the launch. The first thousand users sign up without issue. You high-five.

At 9:14 AM, the dashboard turns red. CPU is at 98%. Memory is maxed out. The database connection pool is exhausted. By 9:17 AM, the API returns nothing вҖ” just connection timeouts and empty responses. Your monitoring dashboard shows 10,000 requests per second hitting a single API server that was load-tested for 500.

The postmortem reveals the truth: a single competitor scraped your entire product catalog using 50 concurrent threads. The scrape looked like legitimate traffic вҖ” valid API keys, reasonable user-agent headers, well-formed HTTP requests. There was no malicious payload, no SQL injection, no zero-day exploit. Just an enormous volume of legitimate-looking requests that your infrastructure could not handle.

Your system had no protective layer. Any client, anywhere, could send unlimited requests and consume unlimited resources. The API had no **rate limiting**.

A competitor with 50 lines of Python brought down six months of work. You did not need a distributed denial-of-service attack. You needed a single rate limiter.

## The Insight

**Rate Limiting** is the practice of controlling how many requests a client can make to an API within a specified time window.

Think of the on-ramp metering light on a crowded freeway. Without the light, hundreds of cars merge simultaneously, the highway gridlocks, and nobody moves. The metering light lets cars enter one at a time вҖ” three per green вҖ” keeping the freeway flowing at its designed capacity. Rate limiting does the same for your API. It spaces out incoming requests so your servers operate at sustainable throughput. The metering light does not reject cars permanently. It slows the entry rate so the system stays healthy for everyone.

## Core Vocabulary

### Rate Limiting

**Rate Limiting** is the mechanism that restricts the number of API requests a client can make within a given time interval. It is the policy layer that decides who gets through and who must wait. Rate limiting policies are defined per client, per endpoint, or per global system. A typical policy looks like "100 requests per minute per API key."

### Throttling

**Throttling** is the enforcement action taken when a client exceeds their rate limit. The server slows the client down by delaying responses, reducing bandwidth, or temporarily blocking requests. Throttling is the active response to rate-limit violations. Some systems throttle gradually вҖ” first delaying, then rejecting вҖ” while others reject immediately on the first over-limit request.

### Quota

A **Quota** is a hard limit on resource consumption over a longer time period, often daily or monthly. Unlike rate limits that reset every second or minute, quotas reset over longer calendars. Example: "Your free tier plan allows 10,000 API calls per month." Quotas protect against sustained abuse and provide a billing boundary for API monetization.

### Token Bucket

The **Token Bucket** algorithm maintains a bucket that fills with tokens at a steady rate. Each request consumes one token. If the bucket has tokens, the request passes. If the bucket is empty, the request is rejected. The bucket has a maximum capacity, allowing short bursts of traffic up to that capacity. This is the most widely deployed rate-limiting algorithm because it balances steady-state throughput with burst tolerance.

### Leaky Bucket

The **Leaky Bucket** algorithm processes requests at a fixed rate regardless of arrival speed. Requests enter a queue (the bucket) and leak out at a constant rate. If the queue is full, new requests are rejected. This algorithm smooths out traffic bursts entirely вҖ” no burst passes through. It is ideal for protecting downstream systems that cannot handle variable throughput, like legacy databases.

### Fixed Window

The **Fixed Window** algorithm divides time into discrete buckets of a fixed duration (1 second, 1 minute, 1 hour). Each bucket has a counter. When a request arrives, the system checks the counter for the current window. If the counter is below the limit, the request passes and the counter increments. If the counter is at the limit, the request is rejected. The counter resets at the start of the next window. This algorithm is simple but suffers from boundary spikes at window edges.

### Sliding Window

The **Sliding Window** algorithm improves on Fixed Window by tracking the request count over a continuously moving time window. Instead of resetting at hard boundaries, the window advances request-by-request. This eliminates the boundary spike problem. The trade-off is higher memory and computation cost because the system must track individual request timestamps or use approximation techniques.

### Sliding Log

The **Sliding Log** algorithm maintains a sorted log of timestamps for each client. When a new request arrives, the system removes timestamps older than the window duration from the log, counts the remaining timestamps, and compares the count against the limit. This provides exact sliding-window semantics but requires O(n) storage per client where n is the maximum requests per window.

### Distributed Rate Limiting

**Distributed Rate Limiting** enforces rate limits across multiple servers or nodes in a distributed system. A counter stored in local memory on one server is invisible to another server. Distributed rate limiting requires a shared counter вҖ” typically in Redis, Memcached, or a distributed data store вҖ” with atomic increment operations and careful handling of network partitions and clock skew.

### Backpressure

**Backpressure** is a feedback signal from a downstream system to an upstream system indicating that the downstream cannot keep up. When a rate limiter starts rejecting requests, the 429 responses act as backpressure to the client. The client is expected to slow down. In event-driven systems, backpressure can propagate through the entire pipeline, causing upstream services to buffer or shed load.

### Burst

A **Burst** is a short period during which traffic exceeds the average rate. Bursts happen naturally вҖ” a flash crowd, a social media post going viral, a cron job firing simultaneously across thousands of machines. Rate-limiting algorithms differ in how they handle bursts. Token Bucket allows bursts up to bucket capacity. Leaky Bucket eliminates bursts entirely. A good rate limiter absorbs legitimate bursts while rejecting abusive ones.

### 429 Too Many Requests

The HTTP status code **429 Too Many Requests** is the standard response sent by a rate limiter when a client exceeds its limit. The response should include a `Retry-After` header indicating how long the client must wait before retrying. Well-behaved clients read this header and back off. Ignoring 429 responses and retrying immediately worsens the problem вҖ” a pattern called a retry storm.

### The Rate-Limiting Trade-Off Landscape

Every rate-limiting algorithm balances three competing concerns:

| Concern | Question | Example |
|---|---|---|
| Fairness | Does every client get equal access? | Sliding Window prevents a client from hogging window boundaries |
| Burst Tolerance | Can a spike pass through? | Token Bucket allows short bursts up to capacity |
| Resource Cost | How much CPU and memory does it cost? | Fixed Window uses one counter per client; Sliding Log uses an array of timestamps |

Your choice depends on traffic patterns. A payment API should never burst вҖ” customers accept slight delays. A social media feed API should allow bursts during peak news cycles. No single algorithm is optimal for every use case.

## How Rate Limiting Works Step by Step

Rate limiting looks simple from the outside вҖ” accept or reject вҖ” but the implementation requires careful thought. Walk through the full lifecycle of a rate-limited request.

### Step 1: Request Arrives at the Gateway

A client sends an HTTP request to your API. The request includes identifying information: an API key in the header, an IP address, or an authenticated user ID. The rate limiter must extract this identity to determine which policy applies.

The identity extraction happens at the API gateway or middleware layer, before the request reaches any application logic. Common identity sources:

- **API Key**: `X-API-Key` header for third-party developers
- **User ID**: Session token or JWT for authenticated users
- **IP Address**: Fallback when no authentication is present
- **Combination**: `user-id + endpoint-path` for per-endpoint limits

### Step 2: Identity Lookup in the Policy Store

The rate limiter looks up the applicable policy for the extracted identity. The policy defines:

- The algorithm (Token Bucket, Fixed Window, Sliding Window)
- The limit (100 requests per minute)
- The scope (per API key, per user, per IP, or global)
- The action on violation (reject with 429, delay, or degrade)

Policies are stored in a configuration database or a fast local cache. Production systems use a hierarchical approach: global defaults applied first, then endpoint-specific overrides, then client-specific overrides.

```
Policy Resolution Order:
  1. Client-specific override (if exists)
  2. Endpoint-specific override (if exists)
  3. Plan default (free tier vs. pro tier)
  4. Global default fallback
```

A free-tier client may have a policy of 10 requests per second. A pro-tier client with the same API key prefix may have 100 requests per second. The policy store resolves the most specific rule first.

### Step 3: Counter Check Against the Algorithm

With the identity and policy resolved, the rate limiter checks the current counter for the time window. This is the core algorithmic step.

For a **Fixed Window** algorithm:

```
window_id = floor(current_timestamp / window_size)
counter_key = "rate_limit:{identity}:{window_id}"
current_count = redis.get(counter_key)  // or memory if single-node

if current_count is null:
    current_count = 0

if current_count >= limit:
    return REJECT
else:
    redis.incr(counter_key)
    return ALLOW
```

For a **Token Bucket** algorithm:

```
bucket_key = "token_bucket:{identity}"
bucket = redis.hgetall(bucket_key)

if bucket is empty:
    // Initialize bucket
    tokens = capacity
    last_refill = now
else:
    tokens = bucket.tokens
    last_refill = bucket.last_refill

// Refill tokens based on elapsed time
elapsed = now - last_refill
tokens = min(capacity, tokens + elapsed * refill_rate)
tokens = tokens - 1  // consume one token

if tokens < 0:
    return REJECT
else:
    redis.hset(bucket_key, "tokens", tokens, "last_refill", now)
    return ALLOW
```

### Step 4: Atomic Counter Update with Race Condition Protection

The read-then-increment pattern in Step 3 has a classic race condition. Two concurrent requests both read `current_count = 99`, both see it below the limit of 100, and both increment. The result is 101 requests allowed through for a limit of 100.

Race conditions are solved with **atomic operations**:

- **Redis INCR**: Atomic increment that returns the new value in one operation
- **Lua Scripting**: Run the read-and-check in a Redis Lua script for guaranteed atomicity
- **Redis Transactions**: Use WATCH/MULTI/EXEC for optimistic locking

The safest approach is a Redis Lua script that performs the entire check-and-increment as one atomic unit:

```lua
-- Rate Limit Check Script (Redis Lua)
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window_seconds = tonumber(ARGV[2])
local current = redis.call('INCR', key)

if current == 1 then
    -- First request in this window
    redis.call('EXPIRE', key, window_seconds)
    return {1, limit - 1}  -- allowed, remaining tokens
end

if current <= limit then
    return {1, limit - current}  -- allowed, remaining tokens
else
    return {0, 0}  -- rejected, zero remaining
end
```

This script handles the race condition because Redis executes Lua scripts atomically вҖ” no other command runs between the INCR and the comparison.

### Step 5: Rejected Request Handling

When a request is rejected, the rate limiter must communicate the rejection clearly to the client. The HTTP response uses status code **429 Too Many Requests** with standard headers:

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 45
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1625097600

{
  "error": "rate_limit_exceeded",
  "message": "You have exceeded the rate limit of 100 requests per minute. Retry after 45 seconds.",
  "retry_after_seconds": 45
}
```

The response headers serve two purposes. The `Retry-After` header tells automated clients when to retry. The `X-RateLimit-*` headers help developers debug their integration by showing the policy and current state.

### Step 6: Client Response to Backpressure

A well-behaved client reads the `Retry-After` header and waits before sending the next request. The wait should be slightly longer than `Retry-After` to avoid contending with other clients that also retry at exactly the same moment вҖ” a phenomenon called **thundering herd** or **retry storm**.

Client-side backpressure strategies:

- **Exponential Backoff**: Start with a small delay and double it on each consecutive 429
- **Jitter**: Add random variation to the delay to spread retries across time
- **Circuit Breaker**: Stop sending requests entirely after N consecutive 429s, then probe periodically

```python
import time
import random

def call_with_backoff(api_client, request_fn, max_retries=5):
    for attempt in range(max_retries):
        response = request_fn()
        if response.status_code != 429:
            return response
        retry_after = int(response.headers.get("Retry-After", 1))
        delay = retry_after + random.uniform(0, retry_after * 0.5)
        time.sleep(delay)
    raise Exception("Max retries exceeded")
```

### Step 7: Counter Cleanup and Window Transition

Rate limit counters must be cleaned up after the window expires to prevent memory leaks. The cleanup strategy depends on the algorithm:

- **Fixed Window**: The key includes the window ID. The old window expires naturally when no new requests reference it. Use Redis EXPIRE with a TTL of `window_duration * 2` to handle edge cases.
- **Sliding Log**: Old timestamps are removed during each request check. A periodic cleanup job sweeps stale entries for clients that stopped sending requests mid-window.
- **Token Bucket**: The bucket does not expire вҖ” tokens refill over time. But idle clients accumulate tokens up to capacity. This is by design.

```python
# Pseudo-code for Fixed Window cleanup
# Each counter key includes the window ID
# The key auto-expires, no manual cleanup needed

def get_window_key(identity, window_size_seconds):
    window_id = int(time.time() / window_size_seconds)
    return f"ratelimit:{identity}:{window_id}"

def check_rate_limit(identity, limit, window_size_seconds):
    key = get_window_key(identity, window_size_seconds)
    count = redis.incr(key)
    if count == 1:
        redis.expire(key, window_size_seconds * 2)
    return count <= limit
```

### Step 8: Monitoring and Observability

Rate limiting is invisible when it works correctly. You must monitor it to detect problems. Key metrics to track:

- **Allow Rate**: Requests / second that pass the rate limiter
- **Block Rate**: Requests / second that receive 429 responses
- **Block Rate by Client**: Which clients are hitting limits most frequently
- **Rate Limit Headroom**: Average remaining capacity across all clients
- **Latency Impact**: P99 latency added by the rate limiter itself

A sudden spike in Block Rate usually indicates one of three scenarios:

1. A misconfigured client sending too many requests
2. A DDoS attack overwhelming the system
3. An overly aggressive rate-limit policy that is blocking legitimate users

Alert when the Block Rate exceeds a threshold вҖ” but also alert when it drops to zero. Zero blocks could mean the rate limiter crashed silently and all requests are passing through unfiltered.

### Step 9: Distributed Coordination (Multi-Node)

A single-node rate limiter is simple вҖ” counters live in local memory. A multi-node rate limiter requires distributed coordination. Each application instance runs a rate-limiting middleware, and all instances share a central counter store (typically Redis).

Distributed coordination introduces new challenges:

**Challenge 1: Atomicity across nodes.** Two nodes cannot see each other's local counters. The solution is a shared Redis instance with atomic INCR commands. Every node reads and writes to the same keys in Redis.

**Challenge 2: Network latency.** Every rate-limit check requires a Redis round trip. This adds 1-5 ms per request. Mitigation: batch requests in a pipeline, or use local caching with periodic sync for a small window of drift.

**Challenge 3: Redis failure.** If Redis goes down, all rate limiting fails open (all requests pass) or fails closed (all requests are rejected). The common approach is **fail open** with a fail-safe local rate limiter that kicks in when Redis is unreachable:

```python
class DistributedRateLimiter:
    def __init__(self, limit, window):
        self.limit = limit
        self.window = window
        self.redis_client = RedisClient()
        self.local_limiter = LocalTokenBucket(limit * 2, self.window)

    def check(self, identity):
        try:
            return self._redis_check(identity)
        except RedisConnectionError:
            # Fall back to local limiter when Redis is down
            # Local limit is doubled to absorb the lack of global coordination
            return self.local_limiter.check(identity)
```

## Variants of Rate Limiting

Different rate-limiting algorithms suit different traffic patterns and system constraints. Understand each variant to choose the right one.

### Token Bucket

The Token Bucket is the most popular rate-limiting algorithm because it balances steady throughput with burst tolerance.

**How it works:** A bucket holds tokens. Tokens are added at a fixed rate (the refill rate) up to a maximum capacity. Each request removes one token. If tokens remain, the request passes. If the bucket is empty, the request is rejected.

**Burst behavior:** A client that goes idle accumulates tokens up to the bucket capacity. When traffic spikes, the client can consume all accumulated tokens at once, allowing a burst of up to `capacity` requests instantly. After the burst, the client is limited to the refill rate.

```
  Bucket Capacity: 10 tokens
  Refill Rate:     1 token per second

  Time 0:  Bucket = 10 (full)    Client sends 8 requests
  Time 1:  Bucket = 2            Client sends 2 requests
  Time 2:  Bucket = 0            Client sends 1 request -> REJECT
           Refill +1 вҶ’ Bucket = 1
  Time 3:  Bucket = 1            Client sends 1 request -> ALLOW
           Refill +1 вҶ’ Bucket = 1
```

**Best for:** APIs where clients should be allowed short bursts (social media feeds, news APIs, real-time dashboards).

**Configuration parameters:**

- **Capacity (bucket size):** Determines the maximum burst size. A capacity of 100 allows 100 consecutive requests at full speed.
- **Refill Rate:** Determines the sustained throughput. A refill rate of 10 tokens/second allows a sustained rate of 10 requests per second after the burst is exhausted.
- **Formula:** The steady-state rate equals the refill rate. The burst size equals the capacity. These are independent knobs вҖ” tune them separately.

**Redis implementation:**

```python
import time
import redis

class TokenBucket:
    def __init__(self, redis_client, capacity, refill_rate):
        self.redis = redis_client
        self.capacity = capacity
        self.refill_rate = refill_rate  # tokens per second

    def allow_request(self, key):
        lua_script = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])

        local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
        local tokens = tonumber(bucket[1])
        local last_refill = tonumber(bucket[2])

        if tokens == nil then
            tokens = capacity
            last_refill = now
        end

        local elapsed = now - last_refill
        tokens = math.min(capacity, tokens + elapsed * refill_rate)

        if tokens >= 1 then
            tokens = tokens - 1
            redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
            redis.call('EXPIRE', key, math.ceil(capacity / refill_rate) + 1)
            return 1
        else
            return 0
        end
        """
        result = self.redis.eval(lua_script, 1, key, self.capacity, self.refill_rate, int(time.time()))
        return result == 1
```

### Leaky Bucket

The Leaky Bucket algorithm enforces a strict constant output rate regardless of input spikes. It is the opposite of Token Bucket in burst behavior.

**How it works:** Requests enter a FIFO queue (the bucket). A worker processes requests from the queue at a fixed rate (the leak rate). If the queue is full, incoming requests are rejected.

**Burst behavior:** Bursts are absorbed by the queue up to its capacity, but the output rate never exceeds the leak rate. Unlike Token Bucket, a burst of 100 requests still exits at the same rate as a single request вҖ” the burst just fills the queue.

```lua
-- Leaky Bucket in Redis
-- KEYS[1] = queue key
-- ARGV[1] = queue capacity
-- ARGV[2] = leak rate (requests per second)

local queue_length = redis.call('LLEN', KEYS[1])
local capacity = tonumber(ARGV[1])

if queue_length >= capacity then
    return 0  -- Queue full, reject
else
    redis.call('RPUSH', KEYS[1], ARGV[3])  -- Add timestamp
    redis.call('EXPIRE', KEYS[1], 10)
    return 1  -- Accepted into queue
end
```

**Best for:** Downstream systems with strict throughput limits, legacy databases, batch processing pipelines, payment gateways.

### Fixed Window

The Fixed Window algorithm divides time into discrete buckets. Each bucket has a counter that resets at the window boundary.

**How it works:** Time is sliced into windows of duration `W` (1 second, 1 minute, 1 hour). Each window has an integer counter. The counter starts at zero at the beginning of each window and increments with each allowed request. When the counter reaches the limit, subsequent requests are rejected until the next window.

**The boundary spike problem:** At the exact moment a window expires, the counter resets. A client that was blocked at the end of window N can immediately send requests at the start of window N+1. This creates a traffic pattern where requests can exceed the limit by up to 2x at the boundary:

```
Window 1 (00:00 - 01:00):   100 requests вҶ’ limit reached, blocking
Window 2 (01:00 - 02:00):   100 requests вҶ’ all arrive at 01:00:01
Actual rate over 1 second at boundary: 200 requests / 2 windows вүҲ 100 req/min
But the instantaneous rate at 01:00:01 is 200 requests in 1 second = 12000 req/min
```

The boundary spike allows up to `2 * limit - 1` requests in a narrow boundary window. For strict rate limits, this can be dangerous.

**Best for:** Simple use cases where a small amount of boundary drift is acceptable. Monitoring dashboards, non-critical endpoints, internal tools.

### Sliding Window Log

The Sliding Window Log maintains a timestamped log of every request for each client. The window slides with each new request.

**How it works:** For each client, the system stores a sorted list of timestamps representing recent requests. When a new request arrives, the system removes all timestamps older than `now - window_duration`. It counts the remaining timestamps. If the count is below the limit, the request passes and its timestamp is appended. If the count is at or above the limit, the request is rejected.

**Performance characteristics:**

- **Time complexity:** O(n) per request in the worst case, where n is the maximum number of requests per window per client. With a limit of 1000 req/min, each check scans up to 1000 timestamps.
- **Space complexity:** O(n) per client. Each timestamp requires 4-8 bytes. A client with 1000 req/min uses ~8 KB of memory. With 10,000 clients, that is 80 MB.
- **Optimization:** Use a balanced tree (Redis Sorted Set) for O(log n) insertion and deletion. Use ZREMRANGEBYSCORE to remove old timestamps and ZCOUNT to count remaining entries.

**Best for:** Scenarios requiring exact sliding-window accuracy. Financial APIs, compliance-regulated systems, rate limits that must be provably precise.

**Redis implementation:**

```python
import time
import redis

class SlidingWindowLog:
    def __init__(self, redis_client, limit, window_seconds):
        self.redis = redis_client
        self.limit = limit
        self.window = window_seconds

    def allow_request(self, identity):
        key = f"sliding_log:{identity}"
        now = time.time()
        window_start = now - self.window

        lua_script = """
        local key = KEYS[1]
        local now = tonumber(ARGV[1])
        local window_start = tonumber(ARGV[2])
        local limit = tonumber(ARGV[3])

        -- Remove old entries
        redis.call('ZREMRANGEBYSCORE', key, '-inf', window_start)

        -- Count remaining entries
        local count = redis.call('ZCOUNT', key, window_start, '+inf')

        if count < limit then
            -- Add current timestamp
            redis.call('ZADD', key, now, tostring(now))
            redis.call('EXPIRE', key, 10)
            return 1
        else
            return 0
        end
        """

        result = self.redis.eval(lua_script, 1, key, now, window_start, self.limit)
        return result == 1
```

### Sliding Window Counter

The Sliding Window Counter is a memory-efficient approximation of the sliding window log. Instead of storing every timestamp, it stores only the count for the current window and a partial count for the previous window.

**How it works:** The algorithm divides the overall window into two sub-windows:

1. The current (partial) window вҖ” a counter that increments with each request
2. The previous (complete) window вҖ” a counter from the last full window

The sliding window count is calculated as:

```
sliding_count = previous_window_count * (1 - weight) + current_window_count

where weight = (current_time - current_window_start) / window_duration
```

This is an approximation. The actual count depends on how evenly requests were distributed in the previous window. In practice, the approximation error is under 5% for most traffic patterns and the memory saving is significant.

**Memory comparison:**

| Algorithm | Memory per client per window | Memory for 100K clients at 100 req/min |
|---|---|---|
| Sliding Log | O(n) | ~800 MB |
| Sliding Counter | O(1) | ~2 MB |

**Best for:** High-scale systems where memory efficiency matters and slight inaccuracy is acceptable. CDNs, API gateways, social media platforms.

### Variant Selection Guide

| Algorithm | Burst Handling | Accuracy | Memory | Use Case |
|---|---|---|---|---|
| Token Bucket | Allows bursts up to capacity | Exact per-token | Low per key | General purpose, most common |
| Leaky Bucket | Smooths all bursts | Exact per-queue | Medium (queue) | Downstream protection |
| Fixed Window | Boundary spikes | Approximate at boundaries | Very low | Simple internal tools |
| Sliding Window Log | No bursts (exact window) | Exact | High (O(n)) | Compliance, finance |
| Sliding Window Counter | Minor drift (< 5%) | Approximate | Low (O(1)) | High-scale APIs |

## Mental Model Check

Test your understanding of rate limiting with these questions. Work through each one before reading the answer.

### Question 1

Your API uses a Fixed Window rate limiter with a limit of 100 requests per minute. A client sends 95 requests in the last 10 seconds of minute 1 and 95 requests in the first 10 seconds of minute 2. How many requests did the client successfully send within a 20-second span?

**Answer:** 190 requests. The Fixed Window reset at the boundary between minute 1 and minute 2. The client was under the limit in both windows (95 < 100 in each), so both batches passed. The effective rate over that 20-second span was 570 requests per minute вҖ” nearly 6x the intended limit. This is the boundary spike problem. A Sliding Window algorithm would have prevented this by tracking the rolling 1-minute window.

### Question 2

Your rate limiter uses a Token Bucket with capacity 20 and refill rate 5 tokens/second. A client sends requests at 10 requests/second for 10 seconds, then stops. How many requests are rejected, and how many tokens remain in the bucket at the end of the 10-second period?

**Answer:** The bucket starts with 20 tokens. In the first 2 seconds, the client sends 20 requests and consumes 20 tokens. The refill adds 5 tokens/second, so after 2 seconds the bucket has `20 - 20 + 10 = 10` tokens. In the remaining 8 seconds, the client sends 80 requests. The bucket refills `8 * 5 = 40` tokens during this time, giving a total of `10 + 40 = 50` tokens available. The client consumes 50 of 80 requests successfully. The remaining 30 requests are rejected. At the end of 10 seconds, the bucket is empty (0 tokens). The sustained rate after the initial burst is exactly the refill rate of 5 requests/second.

### Question 3

Your distributed rate limiter uses a shared Redis counter. A network partition occurs, splitting your application servers into two groups that cannot reach Redis. Both groups fail open вҖ” they let all requests through. What happens, and how would you prevent it?

**Answer:** During the partition, rate limiting is completely disabled. Requests pass through at full speed, potentially overwhelming downstream systems. This is the fail-open vs. fail-closed trade-off. To prevent this, implement a hybrid approach:

1. Configure a local (in-memory) rate limiter as a fallback on each server
2. Give the local limiter a more conservative limit (e.g., 2x the normal limit as a safety buffer)
3. Monitor Redis connectivity and alert immediately when the connection drops

The local fallback is not perfect вҖ” without global coordination, two servers could each allow 100 requests and the combined total would be 200. But 200 is better than unlimited (thousands or tens of thousands).

### Question 4

You must design a rate limiter for a payment processing API. Which algorithm would you choose and why?

**Answer:** Choose **Leaky Bucket** or **Sliding Window Log** for a payment API.

Leaky Bucket is preferred if the downstream payment processor has a hard capacity limit (e.g., 50 transactions/second). The queue absorbs short spikes while the leak rate guarantees the downstream never exceeds its limit. No burst passes through.

Sliding Window Log is preferred if you need exact per-second counts for auditing and compliance. Payment APIs often have regulatory requirements that demand precise rate-limit enforcement.

Reject Fixed Window вҖ” boundary spikes could cause a double batch of payment requests to hit the processor simultaneously, triggering duplicate detection failures or settlement errors. Reject pure Token Bucket вҖ” bursts up to capacity could overwhelm the downstream processor even if the sustained rate is within limits.

### Question 5

Your API responds with `Retry-After: 60` on every 429 response. All blocked clients retry after exactly 60 seconds. Describe the thundering herd problem and how you would fix it.

**Answer:** When all blocked clients retry at exactly the same moment, they create a synchronized wave of traffic that hits the rate limiter simultaneously. The rate limiter allows a subset through (up to the limit) and rejects the rest with another `Retry-After: 60`. The rejected clients all retry again in exactly 60 seconds. The pattern repeats indefinitely вҖ” a standing wave of synchronized traffic that wastes capacity without ever clearing the backlog.

Fix with **jittered exponential backoff**:

1. The rate limiter should return a `Retry-After` value that includes random jitter: `Retry-After: uniform(45, 75)`
2. Clients should add their own jitter: `delay = retry_after * (1 + random(0, 0.5))`
3. Use exponential backoff with jitter on the client side:

```python
def retry_delay(attempt, base_ms=1000, max_ms=60000):
    exponential = min(base_ms * (2 ** attempt), max_ms)
    jitter = random.uniform(0, exponential * 0.5)
    return exponential + jitter
```

Spread the retries across time, and the standing wave collapses into a smooth recovery pattern.
