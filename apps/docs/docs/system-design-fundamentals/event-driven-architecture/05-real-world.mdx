Netflix runs one of the largest event-driven systems in the world. Their event infrastructure processes over 700 billion events per day, powering everything from content recommendations to infrastructure monitoring to A/B testing.

### Concrete Numbers

- **700 billion events/day**: Approximately 8 million events per second average, peaking at over 15 million events per second during prime hours.
- **1.5 PB of event data ingested daily**: Each event is compact but the volume is staggering.
- **6 Kafka clusters in production**: Organized by event priority and processing latency requirements.
- **4,000+ consumer applications**: Each team runs its own consumers subscribing to relevant event streams.
- **P99 latency of 5 seconds end-to-end**: From a user action on a device to the event being available for consumption.

### Key Lessons from Netflix

**Lesson 1: Lose events gracefully, not catastrophically.** Netflix designs for "graceful degradation of data fidelity." If the event pipeline is overloaded, they drop lower-priority events (UI interactions) rather than critical events (playback starts). Each event has a priority level, and the system has configurable drop policies per level.

**Lesson 2: Buffer before the broker.** Netflix buffers events in S3 before publishing to Kafka. This protects the pipeline against Kafka cluster failures. If Kafka is unavailable for 30 minutes, events are written to S3 and replayed when Kafka recovers. The buffer decouples client event emission from broker availability.

**Lesson 3: Schemas are non-negotiable.** Every event type has a schema registered in their schema registry. The registry enforces compatibility rules. A producer cannot change an event schema without going through a review process. This prevents consumer crashes from unexpected field changes.

**Lesson 4: Drop events to stay healthy.** Netflix uses load shedding aggressively. When consumers fall behind, the system drops lower-priority events rather than allowing the entire pipeline to back up. The philosophy: it is better to lose some analytics data than to lose the ability to stream videos.

## Uber: Trip Lifecycle Events

Uber processes the lifecycle of every ride through an event-driven system. From the moment a rider requests a trip to the moment payment is processed, events flow through Kafka topics connecting dozens of services.

### The Trip Event Model

```
RiderRequests ──> DriverAssigned ──> DriverEnRoute ──> DriverArrived
    │                                                    │
    ▼                                                    ▼
RiderPickedUp ──> TripStarted ──> TripEnded ──> PaymentProcessed
                                                      │
                                                      ▼
                                                RatingSubmitted
```

Each event carries the full trip context: rider ID, driver ID, location coordinates, timestamp, and pricing data. Services subscribe to the events they need:

- **Dispatch service**: Subscribes to `RiderRequests`, emits `DriverAssigned`.
- **ETA service**: Subscribes to `DriverAssigned`, `DriverEnRoute`, continuously updates estimated arrival times.
- **Pricing service**: Subscribes to `TripEnded`, calculates fare, emits `PaymentProcessed`.
- **Safety service**: Subscribes to all trip events, monitors for anomalies (route deviation, prolonged stops).

### Key Lessons from Uber

**Lesson 1: Partition by entity ID.** Uber partitions events by trip UUID. All events for a trip go to the same partition, guaranteeing ordering. This prevents the race condition where `TripEnded` arrives before `TripStarted`.

**Lesson 2: Handle geolocation scale.** Uber ingests 100+ million GPS location updates per hour from active drivers. Each location is an event. The geolocation stream has its own dedicated Kafka cluster with high partition count (128 partitions) and aggressive retention (24 hours). After 24 hours, raw locations are moved to HDFS for batch analytics.

**Lesson 3: Use event-carried state transfer heavily.** Trip events carry the full trip state. A consumer subscribing to `TripEnded` has everything it needs to calculate fares, send receipts, or update rider profiles. No synchronous API calls needed.

## Amazon: Order Processing Pipeline

Amazon order processing is a textbook event-driven system. When a customer clicks "Buy Now," the action triggers a cascade of events across hundreds of services.

### The Order Flow

1. Customer clicks "Buy Now" on amazon.com.
2. The frontend calls the order service, which validates the order and emits `OrderPlaced`.
3. `OrderPlaced` fans out to multiple subscribers:
   - **Inventory service**: Reserves items in the fulfillment center closest to the customer.
   - **Payment service**: Authorizes the payment.
   - **Fraud detection service**: Scores the order for fraud risk.
   - **Recommendation service**: Updates purchase history for recommendations.
4. After inventory reservation succeeds, inventory service emits `InventoryReserved`.
5. `InventoryReserved` triggers:
   - **Fulfillment service**: Creates a pick list for warehouse workers.
   - **Shipping service**: Calculates shipping options and costs.
6. After payment authorization, payment service emits `PaymentAuthorized`.
7. Both `InventoryReserved` and `PaymentAuthorized` must complete before the order can proceed to fulfillment. An orchestrator service waits for both events.

### Key Lessons from Amazon

**Lesson 1: Orchestration with event correlation.** Amazon uses a saga pattern with an orchestrator that correlates events from multiple services. The orchestrator subscribes to both `InventoryReserved` and `PaymentAuthorized`. It only proceeds to fulfillment when both events arrive for the same order_id. This ensures the order moves forward only when all prerequisites are met.

**Lesson 2: Compensation events.** When a step fails, Amazon emits compensation events that reverse previous steps. If payment fails after inventory has been reserved, the system emits `InventoryRelease` to unreserve the stock. Compensation events are crucial for maintaining consistency in long-running workflows.

```json
{
  "event_type": "InventoryRelease",
  "compensation_for": "InventoryReserved",
  "payload": {
    "order_id": "ord_42",
    "sku": "WIDGET-A",
    "quantity": 2,
    "reason": "PAYMENT_FAILED"
  }
}
```

**Lesson 3: Event sourcing for audit.** Amazon stores every order event in an event store. This provides a complete audit trail for every order: who placed it, what was ordered, when payment was processed, which fulfillment center picked it, which carrier shipped it, and when it was delivered. The event store is used for customer service, fraud investigations, and compliance audits.

## Slack: Event Delivery at Scale

Slack delivers events to integrations and bots through its Events API. When something happens in a Slack workspace — a message posted, a channel created, a user joined — Slack emits an event to registered subscribers.

### The Challenge

Slack faces a unique event delivery challenge: it delivers events to external systems that Slack does not control. External services can be slow, unreliable, or malicious. A slow subscriber receiving events should not impact other subscribers.

### Architecture

- **Event buffer**: Events are buffered in a Kafka cluster before delivery.
- **Per-subscriber queue**: Each subscriber has its own delivery queue. A slow subscriber only blocks its own queue.
- **Retry with exponential backoff**: Failed deliveries are retried with 1s, 2s, 4s, 8s backoff up to 6 hours.
- **Subscriber rate limiting**: Slack enforces per-subscriber rate limits. Subscribers that exceed their limit have their events queued and delivered at a controlled rate.
- **Event deduplication**: Events carry a unique event_id. Subscribers must use it for idempotency. Slack delivers at-least-once, and subscribers may receive duplicates.

### Key Lessons from Slack

**Lesson 1: Isolate slow consumers.** Each subscriber has its own queue and its own consumer. A subscriber that takes 30 seconds to process an event does not impact the delivery latency for other subscribers. This is critical when you do not control the consumer.

**Lesson 2: Enforce rate limits.** Without rate limits, a subscriber that processes events slowly builds up an unbounded backlog. Slack enforces a configurable rate limit per subscriber. Events that exceed the limit are queued and delivered at the allowed rate.

**Lesson 3: Provide a retry mechanism with eventual failure.** Slack retries failed deliveries for up to 6 hours. Subscribers that are temporarily down (deploying new code, suffering an outage) have a window to recover without losing events. After 6 hours, undelivered events are discarded and reported in a failure log.

## Open Source Tools Comparison

### Apache Kafka

Kafka is the dominant event streaming platform. Designed for high-throughput, durable, replayable event streams.

- **Strengths**: Proven at massive scale, strong durability guarantees, excellent ecosystem (Kafka Connect, Kafka Streams, KSQL), schema registry integration.
- **Weaknesses**: Complex to operate, higher latency than competitors, no native scheduled delivery, no native priority queuing.
- **Best for**: High-throughput event streaming, event sourcing, log aggregation, data pipelines.

### RabbitMQ

RabbitMQ is a traditional message broker with sophisticated routing capabilities.

- **Strengths**: Easy to set up, flexible routing (topic, direct, fanout, headers exchanges), low latency, management UI, per-message acknowledgements.
- **Weaknesses**: Lower throughput than Kafka, messages are removed after consumption (no replay), limited long-term retention, less suited for event streaming.
- **Best for**: Task queues, RPC, work distribution, systems needing complex routing.

### Apache Pulsar

Pulsar combines Kafka streaming with RabbitMQ queuing.

- **Strengths**: Native multi-region replication, compute-storage separation (scalable independently), lower latency than Kafka, supports both streaming and queuing, built-in DLQ support.
- **Weaknesses**: Smaller ecosystem than Kafka, fewer managed services, newer and less battle-tested.
- **Best for**: Multi-region deployments, systems needing both streaming and queuing, geo-replicated event pipelines.

### NATS

NATS is a lightweight, high-performance messaging system.

- **Strengths**: Extremely low latency (sub-millisecond), simple to operate, supports at-most-once and at-least-once delivery, small footprint.
- **Weaknesses**: Limited durability, no long-term retention, smaller ecosystem, fewer features.
- **Best for**: Low-latency messaging, IoT, internal service communication.

## Cloud Managed Services

### AWS EventBridge

AWS EventBridge is a serverless event bus that connects AWS services, custom applications, and SaaS partners.

- **Strengths**: Serverless (no infrastructure to manage), schema registry built-in, integrates with 140+ AWS services and 20+ SaaS partners, content-based filtering, archive and replay.
- **Weaknesses**: Higher latency (50-200ms P50), lower throughput than Kafka, limited retention (14 days max), vendor lock-in.
- **Pricing**: $1 per million events published + $1 per million events delivered.

### Google Cloud Pub/Sub

Google Cloud Pub/Sub is a serverless messaging service.

- **Strengths**: Auto-scaling from zero to millions of messages per second, exactly-once delivery (with idempotent processing), global topic support, low operational overhead.
- **Weaknesses**: No schema registry built-in (must use Apicurio or custom), no native event sourcing, no replay of acknowledged messages (pull subscriber).
- **Pricing**: $40 per TiB of data published + $10 per TiB delivered.

### Confluent Cloud

Confluent Cloud is a managed Kafka service.

- **Strengths**: Full Kafka API compatibility, schema registry included, KSQL, Kafka Connect, multi-region clusters, 99.95% SLA.
- **Weaknesses**: Higher cost than self-managed, data egress fees, shared tenant in basic tier.
- **Pricing**: Starting at $0.10/GB/hour of provisioned throughput.
