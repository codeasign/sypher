## Naive Architecture: Single Event Bus

The simplest event-driven architecture uses a single event bus connecting all producers and consumers. All events flow through one broker cluster.

This architecture works for small to medium deployments — up to a few hundred thousand events per day with 5-10 microservices. It has serious limitations:

- **Single point of failure**: The broker cluster is a critical dependency. If it goes down, the entire system stops.
- **Shared infrastructure**: One noisy consumer can impact others. A consumer that processes slowly causes broker disk usage to grow, triggering retention policies that delete events before other consumers have read them.
- **No isolation**: A schema change that breaks one consumer can corrupt events for all consumers sharing that topic.
- **Limited scaling**: One cluster has finite capacity. Adding more partitions requires rebalancing, which causes brief unavailability.

When to use this architecture: internal tools, prototypes, low-volume systems, or systems where 30 minutes of downtime is acceptable.

### Schema Registry

The schema registry sits between producers and the broker. Every event is validated against its registered schema before being accepted. This prevents malformed events from entering the system.

The registry also manages schema evolution. When a producer wants to update a schema, the registry checks compatibility with existing consumers. If the change breaks backward or forward compatibility, the registry rejects it.

Implementation: Confluent Schema Registry (open source), AWS Glue Schema Registry, or Apicurio Registry.

### Event Store

An event store persists events beyond the broker retention period. Events in the broker are stored for the operational window (7-30 days). After that, they are moved to long-term storage (S3, GCS, Azure Blob) for compliance, analytics, and replay needs.

The event store is not a replacement for the broker. It is a cold storage layer. Events are written to the event store asynchronously by a dedicated consumer that subscribes to all topics.

Query patterns for the event store:
- Replay events for a specific entity (order ID) to debug a production issue
- Run analytics queries on historical event data
- Rebuild a consumer's state after a data loss event
- Generate compliance reports showing all events for a time range

### Stream Processor

A stream processor reads from topics, applies transformations, and writes to output topics. It enables real-time transformations without writing custom consumers for every use case.

Use cases:
- **Filtering**: Drop events that do not match criteria. Drop `PageView` events from bot traffic.
- **Enrichment**: Join event streams with database tables. Enrich `OrderPlaced` with customer tier information.
- **Aggregation**: Compute windowed aggregates. Count orders in the last 5 minutes.
- **Pattern detection**: Detect sequences of events. Alert when `LoginFailed` occurs 5 times for the same user in 1 minute.

### Monitoring and Observability

Monitoring an event-driven system requires metrics at each pipeline stage:

**Broker metrics**:
- Bytes in/out per second
- Request rate and latency
- Under-replicated partitions
- Leader election rate

**Consumer metrics**:
- Consumer lag (most important metric)
- Processed events per second
- Processing latency (P50, P99, P999)
- Error rate
- Retry rate
- DLQ event count

**Producer metrics**:
- Published events per second
- Publish latency
- Publish error rate (timeouts, rejected schemas)

### Dead Letter Queue

The DLQ is a separate topic that stores events that could not be processed after all retries were exhausted. It is the safety net of the event-driven system. Without it, a single poisoned event can stall a consumer permanently.

### Distributed Tracing

Event-driven systems scatter processing across multiple services. Traditional logging (each service logs independently) cannot reconstruct the full flow. Distributed tracing solves this.

Each event carries a trace ID and span ID. When a consumer processes an event and emits a new event, it propagates the trace context to the new event. Tracing tools (Jaeger, Zipkin, OpenTelemetry) reconstruct the full event flow across services.

## Integration Patterns

### Database Integration

Event-driven systems often emit events based on database changes. When a row is inserted, updated, or deleted, an event should reflect that change. Two approaches:

**Change Data Capture (CDC)**: Read the database transaction log and emit events for each change. Tools: Debezium, AWS DMS, Kafka Connect.

CDC produces events like:
```json
{
  "op": "c",  // c=create, u=update, d=delete
  "before": null,
  "after": {
    "order_id": "42",
    "status": "placed",
    "total_cents": 8997
  },
  "source": {
    "db": "orders_db",
    "table": "orders",
    "lsn": "123456789"
  }
}
```

**Dual Write**: The application writes to the database and publishes an event in the same transaction. This is simpler but has a failure mode: the database write succeeds but the publish fails, or vice versa. The transactional outbox pattern solves this.

### Cache Integration

Event-driven systems can update caches by subscribing to events. When an `InventoryUpdated` event arrives, the cache invalidation service removes the stale product from the cache.

```typescript
// Cache invalidation consumer
eventBus.subscribe("inventory", "cache-invalidator", async (event) => {
  const cacheKey = `product:${event.payload.sku}`;
  await cache.del(cacheKey);
  console.log(`Invalidated cache key: ${cacheKey}`);
});
```

This pattern keeps caches eventually consistent with the source of truth without synchronous coupling.

### Microservice Integration

Services communicate through events rather than HTTP calls. Each service subscribes to the events it cares about and publishes events when its state changes.

```typescript
// Order service publishes OrderPlaced
// Inventory service subscribes to OrderPlaced and publishes InventoryReserved
// Billing service subscribes to InventoryReserved and publishes PaymentCaptured
// Shipping service subscribes to PaymentCaptured and publishes Shipped
```

This is the choreography pattern. Each service is both a consumer and a producer. The result is a chain of events that propagates state changes through the system.

### Replication Strategies

**Active-Passive**: One region handles all events. Events are replicated asynchronously to a passive region. If the primary fails, the passive region takes over. Consumer lag during failover can be minutes.

**Active-Active**: Both regions handle events simultaneously. Events are replicated bidirectionally. Consumers in each region process local events. This requires conflict resolution for events that are produced in both regions for the same entity.

**Sticky Producer**: Each producer is assigned a primary region. If the primary is unavailable, the producer switches to a secondary. This prevents conflicts because each producer is active in only one region at a time.

### Cross-Region Consistency

Multi-region event replication introduces conflict potential. Two regions could process events for the same entity concurrently. Strategies:

- **Last-writer-wins**: The event with the latest timestamp wins. Simple but can lose data.
- **CRDTs**: Conflict-free Replicated Data Types ensure automatic conflict resolution. Complex but correct.
- **Regional affinity**: Each entity is assigned a home region. All events for that entity are produced from that region. No conflicts.

## Capacity Planning

### Event Volume Estimation

Estimate your event volume to size the broker cluster:

```
Events per day = Daily active users × Events per user per day
               = 1,000,000 × 50 = 50,000,000 events/day

Events per second = 50,000,000 / 86,400 = 579 events/second (average)
                  = 579 × 5 = 2,895 events/second (peak, 5x multiplier)

Bandwidth = 2,895 events/sec × 2KB (average event size) = 5.8 MB/sec
          = 5.8 × 8 = 46.4 Mbps

Storage per day = 50,000,000 events × 2KB = 100 GB/day
Storage for 30 days = 100 GB × 30 = 3 TB
Storage with replication factor 3 = 3 TB × 3 = 9 TB
```

### Partition Sizing

Partitions determine parallelism. Each partition can be consumed by one consumer at a time. General guidelines:

- Minimum partitions = number of consumer instances you plan to scale to
- Maximum partitions = broker throughput / throughput per partition
- A partition should not exceed 10GB on a single broker (rebalancing cost)
- Each partition adds memory overhead (~10MB per partition per broker)

For the e-commerce example with 2,895 events/second:
- Each consumer instance can process ~500 events/second
- You need 2,895 / 500 = 6 consumer instances
- Minimum 6 partitions (one per consumer)
- With 20 percent growth buffer: 8 partitions

### Broker Sizing

| Traffic | Small | Medium | Large | X-Large |
|---|---|---|---|---|
| Events/sec | 1,000 | 10,000 | 100,000 | 1,000,000 |
| Nodes | 3 | 6 | 12 | 24 |
| CPU | 8 cores | 16 cores | 32 cores | 64 cores |
| RAM | 32 GB | 64 GB | 128 GB | 256 GB |
| Disk | 4 × 1TB SSD | 8 × 2TB SSD | 12 × 4TB SSD | 24 × 8TB SSD |
| Network | 10 Gbps | 25 Gbps | 50 Gbps | 100 Gbps |
### Deployment

Event-driven systems are typically deployed on Kubernetes for scalability and self-healing.

**Producer deployment**: Deploy as stateless microservices. Scale based on CPU or request rate. Each producer instance uses a connection pool to the broker.

**Consumer deployment**: Deploy as Kubernetes deployments with a consumer group ID. The number of pods should match the number of partitions for optimal throughput. Too few pods: some partitions are unassigned, events pile up. Too many pods: some pods are idle.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inventory-consumer
spec:
  replicas: 4  # Match partition count for optimal throughput
  selector:
    matchLabels:
      app: inventory-consumer
  template:
    metadata:
      labels:
        app: inventory-consumer
    spec:
      containers:
      - name: consumer
        image: ecommerce/inventory-consumer:latest
        env:
        - name: KAFKA_BOOTSTRAP_SERVERS
          value: "kafka-cluster:9092"
        - name: KAFKA_CONSUMER_GROUP
          value: "inventory-group"
        - name: KAFKA_TOPIC
          value: "orders"
        resources:
          requests:
            cpu: "1"
            memory: "2Gi"
          limits:
            cpu: "2"
            memory: "4Gi"
```

**Broker deployment**: Deploy as a StatefulSet (Kafka on Kubernetes) or use a managed service (Confluent Cloud, AWS MSK, Redpanda). StatefulSets ensure stable network identities, critical for Kafka brokers.

**Stream processor deployment**: Deploy as a Flink or Kafka Streams application. Flink uses checkpointing for fault tolerance. Kafka Streams uses the consumer group protocol for parallelism and fault tolerance.

### Deployment Topologies

**Sidecar pattern**: Each microservice has a sidecar process that handles event production and consumption. The sidecar manages connections, serialization, retries, and offset commits. The microservice communicates with the sidecar via localhost. This isolates event infrastructure from business logic.

**Gateway pattern**: An event gateway accepts events from producers, validates them against schemas, and writes them to the broker. Producers send events to the gateway via HTTP/2 or gRPC. The gateway offloads schema management, authentication, and rate limiting from the producer.

**Hub-and-spoke**: Multiple broker clusters connected by a central hub. Each team owns its cluster (spoke). The hub routes events between spokes. This provides team isolation while enabling cross-team event sharing.
