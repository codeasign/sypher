## Question Types

Event-driven architecture questions appear in system design interviews at three levels:

### Level 1: Conceptual (Junior to Mid)

These questions test your understanding of basic EDA concepts and terminology.

**Sample questions:**
- What is event-driven architecture and how does it differ from request-driven?
- Explain the difference between choreography and orchestration.
- What is the difference between a queue and a stream?
- What is event sourcing? When would you use it?
- Explain consumer lag and how to handle it.
- What is a dead letter queue and why is it important?

**Pass criteria**: Correct definitions, clear examples, understanding of when to use each concept.

### Level 2: Design (Mid to Senior)

These questions ask you to design an event-driven system for a specific use case.

**Sample questions:**
- Design an order processing system using event-driven architecture.
- Design a real-time notification system for a social media platform.
- Design a ride-sharing dispatch system.
- How would you build a fraud detection pipeline using events?
- Design a multi-region event-driven architecture for a global payment system.

**Pass criteria**: Clear architecture diagram, component reasoning, trade-off discussion, handling failure modes, capacity estimation.

### Level 3: Operational (Senior to Staff)

These questions focus on operating event-driven systems at scale.

**Sample questions:**
- Your consumer lag is growing. How do you diagnose and fix it?
- A schema change broke your event pipeline. How do you recover?
- How do you ensure exactly-once processing in a Kafka consumer?
- Your event broker cluster is running out of disk space. What do you do?
- How do you test an event-driven system end-to-end?
- Design a strategy for migrating from a monolith to event-driven microservices.

**Pass criteria**: Systematic debugging approach, operational experience demonstrated, trade-off awareness, monitoring and alerting knowledge.

## Interview Phrases That Signal You Know EDA

Use these phrases to demonstrate depth:

### Explaining the Core Concept

"Event-driven architecture decouples producers and consumers in time and space. The producer emits a fact about what happened — an event — without knowing which consumers will process it. Consumers subscribe to events they care about and process them independently. This gives us temporal decoupling, independent scalability, and resilience to individual service failures."

### Explaining Why Choreography vs Orchestration

"For this workflow, I would use choreography because the steps are independent and we want maximum decoupling. Each service emits events and reacts to events from other services. For a loan approval workflow with branching logic and human decision points, I would use orchestration — a central workflow engine that coordinates the steps. The orchestrator simplifies error handling and gives visibility into the workflow state."

### Explaining Exactly-Once Processing

"Exactly-once delivery is achieved through idempotent consumers and transactional output, not through the broker alone. The broker provides at-least-once delivery. The consumer must handle duplicates by checking a deduplication key or using idempotent operations. For Kafka, exactly-once semantics combine idempotent producer, transactional consumer, and atomic offset commits with output writes."

### Explaining Consumer Lag

"Consumer lag measures how far behind a consumer is from the latest event. It is the most important metric in an event-driven system. If lag grows, the consumer is not keeping up. The fix is either to optimize the consumer processing logic — reduce database query time, batch writes, offload synchronous calls — or to scale horizontally by adding more consumer instances, which requires more partitions."

### Explaining Schema Evolution

"Event schemas are contracts between producers and consumers. A schema registry enforces compatibility rules. I design for backward compatibility: new consumers can read old events. This means adding optional fields, never removing fields, and never changing field types. Deployment order matters: upgrade consumers first, then upgrade producers, giving consumers time to adapt to the new schema."

## Red Flags Interviewers Look For

### Red Flag 1: Confusing Events and Commands

**Candidate says**: "The order service sends a 'ReserveInventory' event to the inventory service."

**Issue**: `ReserveInventory` is a command, not an event. Commands are instructions to do something. Events are facts about what happened. An event would be `InventoryReserved` or `InventoryReservationFailed`.

**Better**: "The order service publishes an `OrderPlaced` event. The inventory service subscribes to `OrderPlaced`, processes it, and publishes `InventoryReserved`."

### Red Flag 2: Ignoring Failure Modes

**Candidate says**: "The event goes from the producer to the broker to the consumer."

**Issue**: No discussion of what happens when each component fails. The interviewer is waiting for you to discuss retries, DLQs, idempotency, and monitoring.

**Better**: "The producer publishes with acks=all for durability. If the broker fails, the event is replicated across multiple brokers. The consumer processes with at-least-once semantics and uses idempotency keys to handle duplicates. Failed events are retried with exponential backoff and moved to a DLQ after 5 attempts."

### Red Flag 3: Underestimating Operational Complexity

**Candidate says**: "We just add Kafka and everything is decoupled."

**Issue**: Kafka adds significant operational overhead — cluster management, partition rebalancing, consumer group management, monitoring, schema registry.

**Better**: "Using Kafka introduces operational complexity. We need to manage the cluster, monitor consumer lag, handle rebalancing, and manage schemas. For this use case, the decoupling benefits outweigh the operational costs. I would recommend using a managed service like Confluent Cloud or AWS MSK to reduce operational burden."

### Red Flag 4: No Schema Consideration

**Candidate says**: "Producers emit JSON events. Consumers parse whatever they get."

**Issue**: No schema management leads to silent data corruption when schemas evolve.

**Better**: "Every event type has a schema registered in a schema registry. The registry validates every event before accepting it. Schema evolution follows compatibility rules — backward compatible by default."

### Red Flag 5: Ignoring Idempotency

**Candidate says**: "The consumer processes each event once."

**Issue**: At-least-once delivery means events can be processed multiple times.

**Better**: "The consumer is idempotent. It stores the event_id in a deduplication table before processing. If it receives the same event again, it checks the deduplication table and skips processing."

## Drills

### Drill 1: Design an E-Commerce Order Processing System

**Scenario**: An e-commerce platform processing 50,000 orders per day wants to migrate from a monolith to event-driven microservices.

**Task**: Design the event-driven architecture. Cover events, producers, consumers, broker choice, schema management, error handling, monitoring, and testing.

**Answer structure**:

1. **Identify events**: Start with OrderPlaced, PaymentAuthorized, PaymentCaptured, InventoryReserved, Shipped, Delivered.
2. **Choose broker**: Kafka. The platform needs high throughput, event replay for debugging, and long retention.
3. **Define schemas**: Each event type has an Avro schema in the schema registry. Schema evolution is backward compatible.
4. **Design producers**: Order service emits OrderPlaced. Payment service emits PaymentAuthorized and PaymentCaptured.
5. **Design consumers**: Inventory service subscribes to OrderPlaced, emits InventoryReserved. Shipping subscribes to PaymentCaptured.
6. **Handle errors**: Retry with exponential backoff. DLQ for poisoned events. Idempotency keys for duplicate handling.
7. **Monitor**: Consumer lag, processing latency, error rate. Alerts at each stage.
8. **Test**: Unit test handlers. Integration test with Testcontainers. Chaos test with injected failures.

### Drill 2: Diagnose Consumer Lag

**Scenario**: Your notification consumer is 15 minutes behind. Orders are being placed but customers are not receiving confirmation emails.

**Task**: Diagnose and fix the lag.

**Answer structure**:

1. **Check the metric**: Confirm the lag. is it growing or stable? A growing lag means the consumer cannot keep up.
2. **Check processing rate**: How many events per second is the consumer processing? How many are being produced?
3. **Identify bottleneck**:
   - Is the consumer blocked on an external API? Check downstream latency.
   - Is the consumer CPU-bound? Check CPU utilization.
   - Is the consumer blocked on database writes? Check database query latency and connection pool.
4. **Immediate fix**:
   - If blocked on an external API: optimize or remove the API call. Use async processing.
   - If CPU-bound: optimize serialization. Move to a faster serialization format.
   - If database-bound: batch writes. Use bulk insert instead of one insert per event.
5. **Long-term fix**:
   - Add more partitions and consumer instances.
   - Implement auto-scaling based on consumer lag.
   - Review processing logic for unnecessary work.

### Drill 3: Handle a Schema Change

**Scenario**: The order service added a "discount_code" field to the OrderPlaced event. The notification consumer crashes when it receives the new event because it uses strict schema validation.

**Task**: Recover the system and prevent recurrence.

**Answer structure**:

1. **Immediate recovery**:
   - Identify the crashed consumer and its error log.
   - Check if the DLQ captured the poisoned events.
   - Deploy a fix to the notification consumer: make the discount_code field optional.
   - Replay the DLQ events through the fixed consumer.
2. **Root cause analysis**:
   - The schema change was not coordinated with consumer teams.
   - The schema registry did not catch the incompatibility because compatibility checking was not enforced.
3. **Preventive measures**:
   - Configure the schema registry to enforce backward compatibility.
   - Add contract tests that verify each consumer against the latest schema.
   - Implement a deployment pipeline that checks schema compatibility before deploying producers.
   - Establish a schema change review process: notify all consumer teams before changing a schema.

### Drill 4: Design Multi-Region Event Replication

**Scenario**: Your SaaS platform serves customers in the US and Europe. You need low-latency event delivery in both regions and disaster recovery if one region fails.

**Task**: Design the multi-region event architecture.

**Answer structure**:

1. **Replication strategy**: Active-active with sticky producers. US customers write to us-east-1. EU customers write to eu-west-1. Each region has its own Kafka cluster.
2. **Cross-region replication**: Use MirrorMaker 2.0 or Pulsar geo-replication to replicate events between regions asynchronously. US events are available in EU within 100ms.
3. **Conflict resolution**: Use sticky producers to prevent conflicts. Each customer has a home region. All events for a customer are produced from that region. No cross-region conflicts.
4. **Disaster recovery**: Each region can handle full load if the other fails. DNS-based failover routes customers to the active region.
5. **Event store in each region**: Both regions store events in their own event store (S3). In a disaster, replay events from the surviving region.
6. **Monitoring**: Track cross-region replication lag. Alert if replication delay exceeds 1 minute. Track consumer lag per region.
