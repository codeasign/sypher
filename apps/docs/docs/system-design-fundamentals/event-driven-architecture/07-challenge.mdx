## Tier 1: Core Concepts (Foundation)

### Challenge 1.1 — Event or Command?

For each of the following, determine whether it is an event (a fact that happened) or a command (an instruction to do something):

A. "ReserveInventory"
B. "OrderCancelled"
C. "SendEmail"
D. "InventoryReserved"
E. "ChargeCustomer"
F. "PaymentFailed"
G. "UpdateUserProfile"
H. "UserProfileUpdated"
I. "ValidateAddress"
J. "AddressValidationFailed"

**Model Answer:**
Events: B (OrderCancelled), D (InventoryReserved), F (PaymentFailed), H (UserProfileUpdated), J (AddressValidationFailed).
Commands: A (ReserveInventory), C (SendEmail), E (ChargeCustomer), G (UpdateUserProfile), I (ValidateAddress).

The pattern: events are named in the past tense. They describe something that has already happened. Commands are in the imperative tense. They describe something that should happen.

### Challenge 1.2 — The Idempotency Check

Your inventory consumer processes OrderPlaced events. Each event reduces the stock count by the ordered quantity. Your broker delivers at-least-once. A network glitch causes the same OrderPlaced event to be delivered three times to the consumer.

**Questions:**
A. What happens to the inventory count if the consumer is not idempotent?
B. Design an idempotency mechanism for this consumer.
C. Write pseudocode showing the idempotent consumer logic.

**Model Answer:**

A. The inventory count is decremented three times instead of once. If the event ordered 2 units of SKU WIDGET-A, the stock drops by 6 instead of 2. The inventory goes negative, and future orders for that SKU may incorrectly show as out of stock.

B. Use a deduplication table. Before processing an event, check if the event_id exists in the deduplication table. If it does, skip processing. If it does not, process the event and insert the event_id into the deduplication table in the same database transaction.

C.
```sql
-- Idempotent inventory consumer logic
BEGIN TRANSACTION;

  -- Check if already processed
  SELECT 1 FROM processed_events WHERE event_id = 'evt_42';

  -- If not found:
  UPDATE inventory
  SET quantity = quantity - 2
  WHERE sku = 'WIDGET-A';

  INSERT INTO processed_events (event_id, processed_at)
  VALUES ('evt_42', NOW());

COMMIT;
```

The transaction ensures that both the inventory update and the deduplication record are committed together. If the consumer crashes between processing and committing, the transaction rolls back, and the event is processed again on restart. The deduplication check prevents double processing.

### Challenge 1.3 — Consumer Group Math

Your order processing system has a Kafka topic with 12 partitions. You deploy a consumer group with 4 consumer instances.

**Questions:**
A. How many partitions does each consumer instance handle?
B. What happens if you scale to 16 consumer instances?
C. What happens if one consumer instance crashes?
D. What happens during partition rebalancing?

**Model Answer:**

A. With 12 partitions and 4 consumers, each consumer handles 3 partitions. Kafka assigns partitions as evenly as possible: 12 / 4 = 3 partitions per consumer.

B. With 16 consumer instances and 12 partitions, 12 consumers get 1 partition each and 4 consumers get 0 partitions. The 4 idle consumers sit unused. This wastes resources. Rule: never deploy more consumers than partitions in a consumer group.

C. If one consumer crashes, the remaining 3 consumers rebalance. Kafka assigns the crashed consumer's 3 partitions to the remaining consumers. Each surviving consumer now handles 4 partitions. They pick up processing from the last committed offset of the crashed consumer. No events are lost, but the surviving consumers carry more load.

D. During rebalancing, all consumers in the group stop processing and wait for the new assignment. This is a "stop-the-world" event. Processing resumes after rebalancing completes (typically 100ms to a few seconds for large groups). Static group membership reduces rebalancing impact by allowing consumers to retain their partition assignments during restarts.

## Tier 2: Design Challenges (Intermediate)

### Challenge 2.1 — The Shipping Notification Delay

Your e-commerce platform uses event-driven architecture for order processing. Orders are placed, payment is processed, inventory is reserved, and shipping is notified. The shipping service is slow — it takes 5 seconds per event because it calls a third-party shipping API.

**Problem**: The shipping service is causing a backup. Events pile up in the shipping topic. Consumer lag grows to 10 minutes and keeps growing. Other consumers on the same topic (analytics, recommendations) are unaffected, but the shipping delay means customers do not receive tracking numbers for hours.

**Task**: Redesign the shipping consumer to resolve the lag. Consider the trade-offs of each approach.

**Model Answer:**

Approach 1: Increase parallelism (horizontal scaling)
- Increase the number of partitions in the shipping topic from 3 to 12.
- Deploy 12 shipping consumer instances (one per partition).
- Result: throughput increases 4x. Each consumer handles fewer events.
- Trade-off: Requires repartitioning the topic. Repartitioning is not trivial in Kafka — you must create a new topic with more partitions and migrate consumers.

Approach 2: Async offloading
- The shipping consumer does not call the shipping API synchronously. Instead, it writes the shipping request to a queue and returns immediately.
- A separate worker pool reads from the queue and calls the API.
- Result: The consumer processes events in milliseconds. Lag drops to near zero. The worker pool scales independently based on API latency.
- Trade-off: Adds complexity. Two failure modes instead of one. The worker pool needs its own monitoring and retry logic.

Approach 3: Batching
- The consumer accumulates events for 5 seconds (or 100 events, whichever comes first) and sends them in a batch to the shipping API.
- Result: Reduces API call overhead. Throughput increases because one API call handles multiple events.
- Trade-off: Introduces 5 seconds of latency before the first event in each batch is sent. The API must support batch operations.

**Recommended approach**: Async offloading with a worker pool. It provides the best decoupling and allows independent scaling of the event consumer and the API caller.

### Challenge 2.2 — The Orphan Order

An order is placed. The order service emits `OrderPlaced`. The inventory service reserves stock and emits `InventoryReserved`. Then the payment service crashes mid-processing. The order is stuck in "inventory reserved, payment pending" state indefinitely.

**Problem**: There is no timeout or recovery mechanism. The payment service never recovers on its own. The inventory remains reserved for an order that will never be paid.

**Task**: Design a recovery mechanism for this scenario.

**Model Answer:**

Approach 1: Saga with compensating events
- Implement a saga orchestrator that monitors the event flow.
- The orchestrator sets a timeout when it sees `InventoryReserved` without a corresponding `PaymentCompleted` within 5 minutes.
- On timeout, the orchestrator emits `InventoryRelease` (a compensating event) to unreserve the stock.
- The inventory consumer processes `InventoryRelease` and releases the reserved stock.

Approach 2: Event-driven timeout service
- A dedicated timeout service subscribes to intermediate events.
- When it receives `InventoryReserved`, it schedules a timeout for 5 minutes.
- If it receives `PaymentCompleted` before the timeout, it cancels the timer.
- If the timeout fires, it emits a compensation event.
- The timeout service uses a persistent timer store (database) so it survives crashes.

Approach 3: Payment service health check + retry
- The payment service, on restart, checks for incomplete payment records.
- It queries the database for orders with "payment_pending" status older than 1 minute.
- It retries payment processing for those orders.
- This works if payment failures are rare and manual recovery is acceptable.

**Recommended approach**: Saga with compensating events. It is the most robust and handles all failure scenarios systematically. Approach 3 is a good fallback for edge cases that slip through.

### Challenge 2.3 — The Event Ordering Problem

Your event-driven system processes financial transactions. Each transaction goes through: Authorized -> Captured -> Settled. Events for the same transaction must be processed in this exact order.

**Problem**: A network partition causes `Settled` to arrive at the consumer before `Captured`. The consumer rejects `Settled` because it has no record of `Captured`. The transaction is stuck.

**Task**: Design a solution that preserves event ordering and handles out-of-order delivery.

**Model Answer:**

Approach 1: Partition by transaction ID
- Use the transaction ID as the partition key. All events for the same transaction go to the same partition.
- Kafka guarantees ordering within a partition. Events are processed in the order they were produced.
- This prevents out-of-order delivery entirely (assuming the producer sends events in the right order).

Approach 2: State machine with idempotency
- The consumer maintains a state machine for each transaction: Authorized -> Captured -> Settled.
- Each event carries a sequence number. The consumer tracks the last sequence number processed.
- If an event arrives out of order (sequence number gap), the consumer buffers it and waits for the missing event.
- After a timeout, the consumer requests a replay from the producer.

Approach 3: Event versioning
- Each event type has a version number. The consumer knows the expected processing order.
- If `Settled` arrives before `Captured`, the consumer stores `Settled` in a pending queue.
- When `Captured` arrives, the consumer processes `Captured` and then processes `Settled` from the pending queue.
- This handles desequenced events without data loss.

**Recommended approach**: Partition by transaction ID (Approach 1). It is the simplest and most reliable. Kafka guarantees ordering within a partition as long as the producer sends events in order. Use Approach 3 as a safety net for edge cases.

## Tier 3: Mastery Challenges (Advanced)

### Challenge 3.1 — Exactly-Once Semantics at Scale

Your fintech platform processes payment events through Kafka. You need to guarantee that each payment is processed exactly once — no duplicates, no losses. Your broker provides at-least-once delivery. Your consumers must handle the rest.

**Task**: Design an end-to-end exactly-once processing pipeline. Consider consumer crashes, broker failures, and network partitions. Include the database write and the offset commit in your design.

**Model Answer:**

The solution combines four mechanisms:

1. **Idempotent producer**: The producer assigns a unique ID to each event and sends with `enable.idempotence=true`. Kafka deduplicates duplicate sends within a session.

2. **Transactional consumer**: The consumer uses Kafka transactions to atomically commit the offset and write the output. This is Kafka's exactly-once semantics (EOS).

```typescript
const consumer = new KafkaConsumer({
  groupId: "payment-processor",
  enableAutoCommit: false,
  isolationLevel: "read_committed"
});

const producer = new KafkaProducer({
  transactionalId: "payment-processor-tx",
  enableIdempotence: true
});

await producer.initTransactions();

while (true) {
  const events = await consumer.poll(100);
  await producer.beginTransaction();

  for (const event of events) {
    // 1. Check deduplication (in case of re-delivery)
    const processed = await db.query(
      "SELECT 1 FROM processed_events WHERE event_id = ?",
      [event.event_id]
    );
    if (processed.length > 0) continue;

    // 2. Process the event
    await db.execute(
      "UPDATE accounts SET balance = balance - ? WHERE account_id = ?",
      [event.payload.amount, event.payload.account_id]
    );

    // 3. Record as processed
    await db.execute(
      "INSERT INTO processed_events (event_id, processed_at) VALUES (?, NOW())",
      [event.event_id]
    );

    // 4. Send result event
    await producer.send({
      topic: "payment-results",
      messages: [{ key: event.event_id, value: resultPayload }]
    });
  }

  // 5. Commit offset and output transaction atomically
  await producer.sendOffsetsToTransaction(
    consumer.assignment(),
    consumer.groupMetadata()
  );
  await producer.commitTransaction();
}
```

3. **Database transaction with deduplication**: Steps 1-3 execute in a database transaction. If the consumer crashes after processing but before committing the Kafka transaction, the database transaction rolls back. On restart, the event is re-processed, and the deduplication check prevents double application.

4. **Monitoring**: Track `processed_events` table size. If it grows too large, partition by date. Monitor transaction abort rate — a high abort rate indicates processing issues.

### Challenge 3.2 — Event Schema Migration at Scale

Your event-driven system has 50 services, 200 event types, 500 consumer applications, and runs on Kafka with Avro schemas in Confluent Schema Registry. You need to rename a field in a widely-used event type. The field `customer_email` should become `contact_email`.

**Task**: Design a zero-downtime migration strategy that does not break any existing consumers.

**Model Answer:**

**Phase 1: Prepare (2 weeks)**

1. Add the new field `contact_email` to the event schema as an optional field. Set backward and forward compatibility.
2. Notify all consumer teams about the upcoming change. Provide migration documentation and a timeline.
3. Update the schema registry with the new schema version. Verify compatibility checks pass.

New schema:
```json
{
  "type": "record",
  "name": "OrderPlaced",
  "fields": [
    {"name": "customer_email", "type": "string"},      // existing, will be deprecated
    {"name": "contact_email", "type": ["null", "string"], "default": null}  // new field
  ]
}
```

**Phase 2: Dual Write (4 weeks)**

1. Producers start populating both fields. `customer_email` and `contact_email` contain the same value.
2. Consumers start migrating to read from `contact_email` instead of `customer_email`.
3. Each consumer team has 1 week to migrate. Track migration progress in a dashboard.
4. Monitor for any consumers that still use `customer_email`.

Producer publishing logic:
```typescript
const event = {
  event_id: generateId(),
  event_type: "OrderPlaced",
  customer_email: customer.email,
  contact_email: customer.email  // dual write
};
```

**Phase 3: Migrate Consumers (2 weeks)**

1. Consumers that still read `customer_email` are migrated. The order: less critical consumers first, most critical last.
2. After migration, all consumers read from `contact_email`.
3. Run end-to-end tests to verify all downstream systems work correctly.

**Phase 4: Deprecate (2 weeks)**

1. Mark `customer_email` as deprecated in the schema registry.
2. Producers stop writing to `customer_email`. They only write `contact_email`.
3. Old consumers fail if they still try to read `customer_email`. The deprecation window gives them time to notice and fix.
4. After confirming no consumers read `customer_email`, remove the field from the schema.

**Total duration**: 10 weeks. This is deliberately slow. A schema change affecting 50 services and 500 consumers cannot be rushed. The phased approach ensures zero downtime and zero data loss.

### Challenge 3.3 — The Global Event Bus

Design an event-driven architecture for a global SaaS platform that operates in 6 regions: us-east, us-west, eu-west, eu-central, ap-southeast, ap-northeast. The platform processes 5 billion events per day. Events must be available in all regions within 5 seconds. Each region must operate independently during network partitions.

**Task**: Design the global event bus architecture. Consider replication, consistency, conflict resolution, disaster recovery, and capacity planning.

**Model Answer:**

**Architecture Overview**:

Use Apache Pulsar for its geo-replication capabilities. Pulsar supports multi-region replication natively with configurable replication policies per topic.

**Component Design**:

Each region has a full Pulsar cluster:
- 12 broker nodes per region
- 24 bookies (storage nodes) per region
- All regions are connected via dedicated inter-region links (each 10 Gbps)

Replication topology: Full mesh. Each region replicates to all other regions. Every event produced in any region is available in all regions within 5 seconds.

**Replication configuration**:
- Topics are configured with `replicationClusters: [us-east, us-west, eu-west, eu-central, ap-southeast, ap-northeast]`
- Replication is asynchronous. Producers are not blocked waiting for cross-region replication.
- Pulsar guarantees at-least-once cross-region delivery.

**Conflict resolution**:
- Sticky producers per entity. Each tenant has a home region. All events for a tenant are produced from that region.
- This prevents cross-region write conflicts because only one region produces events for a given entity.
- During a regional failure, the tenant's home region fails over to a secondary region.

**Disaster recovery**:
- Each region has a secondary that can take over. us-east backs up us-west. eu-west backs up eu-central.
- During a full region failure, DNS routes traffic to the secondary region.
- The secondary region starts producing events for the failed region's tenants.
- When the primary region recovers, events are replicated back and reconciled.

**Capacity planning**:

```
Global events: 5 billion/day = 57,870 events/second average
Peak: 5x = 289,350 events/second
Per region (assuming 20/20/20/15/15/10 split):
  us-east: 57,870 events/s
  eu-west: 34,722 events/s
  ap-southeast: 28,935 events/s

Storage: 5B × 2KB = 10 TB/day
Retention: 7 days hot = 70 TB
            90 days cold (S3) = 900 TB
Replication factor: 3 = 210 TB total storage
```

**Monitoring**:
- Cross-region replication lag. Alert if > 5 seconds.
- Per-region consumer lag. Alert if > 1 minute.
- Inter-region bandwidth utilization. Alert if > 80%.
- Regional health. Alert if any region reports errors.

**Key trade-offs**:
- Cross-region latency: 60-200ms for replication. The 5-second SLA is achievable with Pulsar geo-replication.
- Eventual consistency across regions: An event produced in us-east is available in ap-southeast within ~200ms.
- Higher cost: 6 Pulsar clusters with dedicated inter-region links. Estimated $200K/month in infrastructure costs.
- Complexity: Multi-region event architecture is complex to operate. Requires a dedicated platform team.
