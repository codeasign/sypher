## The Problem Story

Your e-commerce platform has grown from a weekend project to a business doing $2 million per day. The original monolith handled everything: user authentication, product catalog, shopping cart, order processing, inventory management, billing, shipping, and notifications.

It worked fine at 100 orders per day. At 10,000 orders per day, it started creaking. At 100,000 orders per day, it cracked.

The breaking point came during a flash sale. Traffic spiked to 50,000 concurrent users. The order service processed a purchase for a popular item that was already sold out. Inventory was decremented four times for the same unit. Payment was charged. The customer received a confirmation email. Then the shipping service failed because the pick-list showed a negative inventory count. The entire pipeline deadlocked.

The postmortem revealed the root cause: **tight coupling**. Every service directly called every other service's API. The order service called inventory, billing, shipping, and notifications via synchronous HTTP requests. When one service slowed down, it cascaded to all others. A database lock in inventory blocked the order endpoint, which blocked the frontend, which blocked users from adding items to their cart.

The team split the monolith into microservices but repeated the same mistake at a finer granularity: each microservice still called the others via synchronous REST. The architecture looked like a spiderweb of point-to-point dependencies:

Every arrow in that diagram is a synchronous HTTP call. The order service cannot complete a request until inventory, billing, shipping, and notifications all respond. If any of them is slow, down, or throwing errors, the order fails. This is **temporal coupling** вҖ” the producer and consumer must be available at the same time.
## The Insight

Event-Driven Architecture (EDA) is like a smart home. When a motion sensor detects movement, it does not call the lights, the AC, the cameras, and the alarm system one by one. It broadcasts a single event: "motion detected." The lights, AC, cameras, and alarm each subscribe to that event type and react independently. The sensor does not know what will react to its signal. It does not care. It just emits the event.

Apply this to your e-commerce platform. When a customer places an order, the order service does not call inventory, billing, shipping, and notifications. It emits a single event: `OrderPlaced`. The inventory service picks it up and reserves stock. The billing service charges the customer. The shipping service creates a label. The notification service sends a confirmation email. Each service runs independently, at its own pace, without waiting for the others.

The key insight: **the producer and consumer are decoupled in time and space**. The producer emits the event and moves on. The consumer may process it immediately, in 10 seconds, or in 10 hours. The producer does not need to know who the consumers are, how many there are, or whether they are even running.

This decoupling gives you four superpowers:

1. **Resilience**: A consumer failure does not cascade to the producer or other consumers. If the shipping service is down, orders are still placed and inventory is still reserved. The shipping service catches up when it recovers.

2. **Scalability**: Each consumer scales independently. If order volume spikes, you scale the order service. If notification sending is slow, you scale the notification consumer without touching anything else.

3. **Extensibility**: Adding a new consumer requires zero changes to the producer. Want to add a fraud detection service? Subscribe to `OrderPlaced` events. The order service does not even know it exists.

4. **Asynchrony**: The producer does not block. It emits the event and returns to serving the next request. The consumers process at their own pace, potentially in parallel.
## Core Vocabulary

### Event

An **event** is a fact that has occurred in the system. It is immutable вҖ” once it happened, it cannot be undone. Events are named in the past tense: `OrderPlaced`, `PaymentProcessed`, `InventoryUpdated`, `ShipmentDelivered`.

An event has three parts:

- **Header**: Metadata вҖ” event ID, type, timestamp, producer, version, correlation ID.
- **Payload**: The data describing what happened.
- **Context**: Partition key, tracing headers, authentication context.

```json
{
  "event_id": "evt_9a8b7c6d5e",
  "event_type": "OrderPlaced",
  "event_version": 2,
  "timestamp": "2026-07-02T14:30:00.123Z",
  "producer": "order-service",
  "correlation_id": "corr_abc123",
  "payload": {
    "order_id": "ord_42",
    "customer_id": "cus_789",
    "items": [
      {"sku": "WIDGET-A", "quantity": 2, "unit_price_cents": 1999},
      {"sku": "GADGET-B", "quantity": 1, "unit_price_cents": 4999}
    ],
    "total_cents": 8997,
    "currency": "USD"
  },
  "partition_key": "cus_789"
}
```

### Event Producer

The **event producer** detects an occurrence and publishes an event. A producer emits events without knowing which consumers will process them. The order service is a producer of `OrderPlaced` events. The inventory service is a producer of `InventoryUpdated` events.

A producer follows a fire-and-forget pattern: it publishes the event and immediately continues its work. This is the fundamental difference from request-driven communication.

### Event Consumer

The **event consumer** receives events and processes them. Multiple consumers can subscribe to the same event type, each processing independently.

Types of consumers:

- **Stateless**: Transform and forward. Example: reading `OrderPlaced` and writing to a search index.
- **Stateful**: Update local state. Example: reading `OrderPlaced` and updating inventory count.
- **Side-effecting**: Perform an action. Example: reading `OrderPlaced` and sending an email.

### Event Channel

The **event channel** is the communication medium through which events flow. The producer writes to the channel, and the consumer reads from it. They never interact directly. Kafka topics, RabbitMQ exchanges, and SQS queues are event channels.

### Event Processing

Three main processing styles:

- **Simple Event Processing**: Each event processed independently.
- **Event Stream Processing**: Continuous stream with windowed operations.
- **Complex Event Processing (CEP)**: Correlating multiple events to detect patterns.

### Event Schema

An **event schema** defines the structure and types of an event. It is the contract between producer and consumer. Without a schema, producers and consumers can disagree on event structure, leading to silent data corruption. Common formats: Avro, Protocol Buffers, JSON Schema.

### Event Bus

An **event bus** is the central nervous system connecting producers and consumers. It handles routing, delivery, buffering, and persistence. Implemented by a message broker.

### Stream

A **stream** is an ordered, append-only sequence of events. Unlike a queue where events are consumed and removed, a stream preserves events for replay. Multiple consumers read the same stream at different positions, each maintaining its own offset.

### Event Handler

An **event handler** is a function that processes a specific event type:

```typescript
function handleOrderPlaced(event: OrderPlacedEvent): void {
  const inventory = getInventory(event.items);
  inventory.reserveStock(event.items);
  inventory.save();
}
```

Handlers should be idempotent вҖ” processing the same event twice produces the same result. This is essential because delivery is often at-least-once.

### Event-Driven vs Request-Driven

The fundamental difference is **who controls the flow**:

- **Request-driven**: The consumer initiates. The producer waits and returns a response.
- **Event-driven**: The producer initiates. The consumer reacts.

| Aspect | Request-Driven | Event-Driven |
|---|---|---|
| Communication | Synchronous, point-to-point | Asynchronous, broadcast |
| Coupling | Temporal (both must be online) | Temporally decoupled |
| Flow control | Consumer controls | Producer controls |
| Error handling | Immediate (return error) | Deferred (retry, DLQ) |
| Visibility | Easy to trace | Needs distributed tracing |
| Backpressure | HTTP 429, connection pool | Consumer lag, partitions |

### Choreography

**Choreography** has each service react to events and emit its own events without a central controller. Each service is both consumer and producer. The sequence emerges from the event chain.

In choreography: OrderService emits `OrderPlaced`. InventoryService receives it, reserves stock, emits `InventoryReserved`. BillingService receives it, charges, emits `PaymentCompleted`. ShippingService receives it, creates shipment, emits `Shipped`.

Choreography is decentralized and scalable but harder to debug.

### Orchestration

**Orchestration** uses a central coordinator that tells each service what to do. The orchestrator sends commands and waits for responses. Easier to understand but the orchestrator can become a bottleneck.

### Event Broker

The **event broker** receives events from producers and delivers to consumers. Responsibilities: authentication, routing, buffering, persistence, replication, partitioning.

### Event Store

An **event store** is a database optimized for storing events as the source of truth. Current state is derived by replaying the event sequence. Supports append-only writes and temporal queries.

## Step-by-Step: From Monolith to Event-Driven

### Step 1: Identify Events

Walk through your business processes and list every significant occurrence. For e-commerce:

- UserRegistered, ProductListed, ProductAddedToCart
- OrderPlaced, OrderConfirmed
- PaymentAuthorized, PaymentCaptured, PaymentFailed
- InventoryReserved, InventoryExhausted
- ShipmentCreated, ShipmentInTransit, ShipmentDelivered
- ReturnRequested, RefundIssued

Each event is a fact in the past tense. "ReserveInventory" is not an event. "InventoryReserved" or "InventoryReservationFailed" are events.

### Step 2: Choose a Broker

The broker is the backbone of your system:

- **Apache Kafka**: High-throughput streaming, long-term retention, replay. Used by 80 percent of Fortune 500. Millions of events/second.
- **RabbitMQ**: Complex routing, low-latency, task acknowledgements. Tens of thousands/second.
- **Apache Pulsar**: Combines Kafka streaming with RabbitMQ queuing. Native multi-region replication. Millions/second with lower latency.
- **AWS EventBridge**: Serverless, SaaS integration, rule-based routing.
- **Google Pub/Sub**: Serverless, auto-scaling. Best for Google Cloud.

### Step 3: Define Schemas

Define a schema for each event type. The schema is the contract between producers and consumers. Use a schema registry.

```json
{
  "namespace": "com.ecommerce.order",
  "name": "OrderPlaced",
  "type": "record",
  "fields": [
    {"name": "event_id", "type": "string"},
    {"name": "order_id", "type": "string"},
    {"name": "customer_id", "type": "string"},
    {"name": "items", "type": {"type": "array", "items": {
      "type": "record", "name": "OrderItem",
      "fields": [
        {"name": "sku", "type": "string"},
        {"name": "quantity", "type": "int"},
        {"name": "unit_price_cents", "type": "long"}
      ]
    }}},
    {"name": "total_cents", "type": "long"},
    {"name": "currency", "type": "string"},
    {"name": "timestamp", "type": "string"}
  ]
}
```

### Step 4: Implement Producers

Change from "call and wait" to "fire and forget":

```typescript
// Before: synchronous calls
async function placeOrder(order: Order): Promise<void> {
  await inventoryService.reserveStock(order.items);
  await billingService.chargeCustomer(order.customerId, order.total);
  await shippingService.createShipment(order);
  await notificationService.sendConfirmation(order);
}

// After: emit event
async function placeOrder(order: Order): Promise<void> {
  const event = {
    event_id: generateId(),
    event_type: "OrderPlaced",
    timestamp: new Date().toISOString(),
    producer: "order-service",
    payload: order
  };
  await eventBus.publish("orders", event);
  // Consumers handle the rest
}
```

### Step 5: Implement Consumers

Each consumer handles one concern independently:

```typescript
eventBus.subscribe("orders", "inventory-service", async (event) => {
  if (event.event_type === "OrderPlaced") {
    const inventory = await getInventory(event.payload.items);
    const reserved = inventory.reserveStock(event.payload.items);
    if (!reserved) {
      await eventBus.publish("inventory", {
        event_type: "InventoryShortage",
        payload: { order_id: event.payload.order_id }
      });
    }
  }
});
```

### Step 6: Handle Failures

1. **Retry**: Exponential backoff. Start at 1 second, double each attempt, cap at 5 minutes.
2. **Dead Letter Queue**: Events exceeding retry limit move to DLQ for manual inspection.
3. **Idempotency**: Use a deduplication key. Processing the same event twice produces the same result.

### Step 7: Monitor and Observe

Use distributed tracing. Each event carries a correlation ID linking it to the original request. This allows reconstructing the full lifecycle of an order across all services.

### Step 8: Scale Consumers

Each partition is a unit of parallelism. A Kafka topic with 10 partitions supports up to 10 consumer instances in a group. The broker assigns each consumer a subset of partitions.

### Step 9: Evolve Schemas

- **Backward compatible**: New consumers read old events. Add optional fields, never remove.
- **Forward compatible**: Old consumers read new events. Set defaults for new fields.
- **Full compatible**: Both directions. The safest approach.

### Step 10: Test the Event Flow

- **Unit test**: Each handler in isolation.
- **Integration test**: End-to-end with a test broker (Kafka in Docker via Testcontainers).
- **Contract test**: Producers and consumers agree on schemas.
- **Chaos test**: Inject broker failures, consumer crashes, network partitions.

## Mental Model Check

### 1. What is the fundamental difference between event-driven and request-driven communication?

In request-driven, the consumer initiates and waits for a response. Both must be online simultaneously. In event-driven, the producer initiates by emitting an event. The consumer processes later, or never. Producer and consumer are decoupled in time and space.

### 2. A team designs an event-driven system but every handler calls a REST API to get more data. What problem does this create?

This is the event notification anti-pattern taken to extremes. Consumers still depend on the producer API. If the producer is down, consumers cannot function. The system has EDA operational complexity without the resilience benefits. The fix: use event-carried state transfer.

### 3. Why is idempotency important in event-driven systems?

Brokers typically guarantee at-least-once delivery. The same event may arrive multiple times. Without idempotency, processing the same event twice produces incorrect results: double-charging, double-inventory reservation, duplicate emails. Idempotent handlers make multiple deliveries equivalent to one.

### 4. What happens when a consumer cannot keep up with the event rate?

The consumer accumulates lag вҖ” unprocessed events. The broker stores unconsumed events. If lag grows too large, the broker may run out of storage, or the consumer must process weeks of backlog. Solutions: scale consumers (more instances), increase partitions, or optimize processing logic.

### 5. When would you choose orchestration over choreography?

Choose orchestration for workflows with many steps, branching logic, and human decision points вҖ” loan processing, insurance claims, multi-step approvals. Choose choreography for maximum decoupling, scalability, and straightforward event chains вҖ” order processing, notifications, data pipelines.
