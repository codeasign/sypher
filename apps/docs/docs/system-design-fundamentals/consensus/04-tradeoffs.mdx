## Core Tension: Safety vs Liveness

The fundamental tension in consensus algorithms is between safety and liveness. Safety means the system never makes a wrong decision (no two nodes ever commit conflicting values). Liveness means the system eventually makes progress (some proposal is eventually accepted).

The FLP impossibility result, proved by Fischer, Lynch, and Patterson in 1985, states that in an asynchronous distributed system (where message delays are unbounded), no consensus algorithm can guarantee both safety and liveness when even one node may fail. Every consensus algorithm must choose: guarantee safety always, with liveness only under favorable network conditions, or weaken safety to maintain liveness.

All practical consensus algorithms (Paxos, Raft, Zab) choose to guarantee safety at all times. They sacrifice liveness when the network behaves badly. This means:
- Under perfect network conditions, the system makes progress on every proposal.
- Under network partitions or node failures, the system may stop making progress (no decisions committed), but no two nodes ever commit different values.

## Failure Modes

### Failure 1: Leader Election Storm

A leader election storm occurs when a cluster cycles through leader elections rapidly. Each election resets the term, and each new leader must commit a no-op entry to establish its leadership. During the storm, no writes are processed. Clients receive timeouts.

Cause: Flapping network connectivity, or election timeouts that are too short and not randomized sufficiently.

Mitigation: Use randomized election timeouts (150-300ms range). Configure heartbeat intervals appropriate for the network latency. Monitor leaderChanges metric and alert if it exceeds 1 per hour.

### Failure 2: Log Divergence and Recovery

When a leader crashes after replicating to some followers but not others, the logs diverge. The new leader must reconcile the differences. In Raft, the new leader forces all followers to copy its log, overwriting any conflicting entries. This is safe because the conflicting entries were never committed (they were not replicated to a majority).

Cause: Leader crash after partial replication.

Mitigation: None needed -- the algorithm handles this correctly. But the recovery process can be slow if the logs are far apart. The leader must search backward to find the first matching entry, which can take multiple RTTs for large log gaps.

### Failure 3: Quorum Loss

If the cluster loses enough nodes to fall below quorum, no writes can be committed. Reads from surviving nodes return the last committed state but cannot be guaranteed fresh.

Cause: Simultaneous failure of multiple nodes (e.g., a power outage affecting two of three data centers in a 3-node cluster).

Mitigation: Use an odd number of nodes. For two-data-center deployments, use 3 nodes in the primary data center and 2 in the secondary (total 5), ensuring the primary can operate in isolation if the secondary goes down.

### Failure 4: Disk Corruption on a Quorum Node

If a node that holds a quorum-forming position experiences disk corruption, the cluster may lose that node's vote. If the remaining nodes still form a quorum, the cluster continues. If the corrupted node was the only node holding a critical piece of committed state (which should not happen if a majority committed it), the cluster may need to restore from backup.

Cause: Hardware failure, bit rot, filesystem bugs.

Mitigation: Use redundant storage (RAID), regular fsck, and monitor disk health metrics. Always test backup restoration procedures.

### Failure 5: Split-Brain with an Even-Numbered Cluster

A 2-node or 4-node cluster can experience split-brain during a network partition. Each side may elect a leader (since each has a quorum of its own count). With 2 nodes, each side has 1 node, and both may consider themselves the leader with a quorum of 1 (which is not a real majority). Even with odd numbers, if the cluster membership is not properly managed, a misconfiguration that adds a node but does not remove the old one can create an even count.

Cause: Using an even number of consensus nodes, or cluster membership misconfiguration.

Mitigation: Always use an odd number of consensus nodes. Use cluster membership that guarantees a unique majority.

## Anti-Patterns

### Anti-Pattern 1: Consensus for Everything

Using a consensus cluster for every piece of configuration, including data that updates frequently or has low consistency requirements. Consensus writes are expensive (multiple RTTs, disk fsyncs). Storing ephemeral or high-churn data in the consensus cluster creates unnecessary load and can trigger compaction issues.

Fix: Use the consensus cluster only for data that requires strong consistency. Use a key-value store or cache for data that can tolerate eventual consistency.

### Anti-Pattern 2: Single-Node Consensus

Running a consensus cluster with a single node. A single node provides no fault tolerance -- if it fails, the entire cluster is unavailable. The consensus overhead (RPCs, fsync, log management) adds latency without any benefit.

Fix: Use at least 3 nodes for any deployment that requires fault tolerance. For single-node setups, use a standard database without consensus overhead.

### Anti-Pattern 3: Ignoring Network Latency

Deploying consensus nodes across data centers with 200+ ms RTT and expecting sub-10ms write latency. Every consensus write must travel to the farthest node in the quorum and back. The write latency is bounded by the slowest link in the quorum.

Fix: Measure inter-data-center latency before deployment. Use leaders in regions with the most nodes to minimize cross-region round trips. Consider using a single-region deployment with secondary read-only replicas.

### Anti-Pattern 4: Disabling Fsync

Disabling fsync for performance in the mistaken belief that the data will not be lost. Without fsync, a power failure can lose committed writes, violating the consensus durability guarantee. The system may tell clients that a write is committed when in fact it is only in memory.

Fix: Always enable fsync in production. Tune fsync behavior by using a dedicated disk for the write-ahead log, using SSDs, and batching fsyncs. The performance impact of fsync is typically 1-5ms per write on modern SSDs.

### Anti-Pattern 5: Running Out of Disk

Configuring the consensus cluster with insufficient disk space for the log. The log grows with every write. Without compaction, the disk fills up. When the disk is full, the node becomes unavailable, potentially causing quorum loss.

Fix: Configure automatic compaction. Monitor disk usage and alert when it exceeds 70%. Use a separate disk or partition for the consensus log. Plan for 2x the expected log size in capacity.

## When NOT to Use Consensus

Consensus is not the answer for every distributed system problem. Consider alternatives in these cases:

**High write throughput**: Consensus algorithms cap throughput at 10,000-30,000 writes per second. If the system requires 100,000 writes per second, consider using CRDTs (conflict-free replicated data types) or last-writer-wins strategies with eventual consistency.

**Single-node deployments**: A single node does not need consensus. Use a traditional database with transaction logs and periodic backups.

**Read-heavy workloads with weak consistency**: If the application can tolerate stale reads, a leader-follower replication setup with async replication provides much higher read throughput and lower latency than a consensus cluster.

**Geographically distributed with high latency**: Consensus across continents adds 100-300ms to every write. Consider using a single-region consensus cluster with asynchronous replication to other regions for disaster recovery.

**Temporary coordination**: For short-lived operations like distributed locks that only need to prevent concurrent access, a simpler lease-based approach or a lightweight lock service (Redis Redlock) may suffice. Consensus is needed when the lock decision must be durable and survive node failures.

## Detailed Failure Mode Analysis

### Partial Failure: Slow Node

A single node with high disk latency can severely impact consensus performance. If a follower takes 100ms to fsync each entry (due to disk contention or a failing disk), the leader's write latency may increase. In Raft, the leader tracks matchIndex per follower. A slow follower falls behind. The leader continues committing entries with the fast followers. The slow follower catches up when its disk recovers.

The impact depends on the cluster size. In a 3-node cluster, a slow follower forces the leader to wait for 1 acknowledgment (the leader + 1 fast follower = quorum of 2). The slow follower does not affect quorum. In a 3-node cluster with a slow leader, every write is delayed by the slow disk. This is why leader election should prefer nodes with fast disks.

### Failure During Leader Election

A leader election that takes longer than expected can cause cascading failures. If the election timeout is 150ms and the election fails to produce a leader (split vote), a new election starts after another 150ms. Each failed election resets the term. With 3 nodes and randomized timeouts, the probability of a split vote in a single election is approximately 33% (the probability that two nodes time out simultaneously). The expected number of elections is 1.5.

If the election succeeds but the new leader's log is far behind the old leader's log, the new leader may need to install a snapshot and replicate many entries before it can accept new client requests. During this catch-up period, the cluster is unavailable for writes. The catch-up time depends on the snapshot size and the number of entries to replicate. For a 1 GB snapshot on a 1 Gbps link, the catch-up takes approximately 10 seconds.

### Disk Full on All Nodes

If all nodes' disks fill up simultaneously (e.g., due to a compaction failure that the operator did not notice), the cluster cannot write any new entries. The cluster becomes completely unavailable. Recovery requires freeing disk space on at least a majority of nodes, then restarting the consensus process.

Prevention: monitor disk usage on all consensus nodes. Alert when usage exceeds 70%. Configure automatic compaction with a maximum log size. Use separate disks for the consensus log to avoid interference from other processes.

### Network Congestion

Network congestion increases the RTT between nodes. If the congestion is symmetric (all links affected equally), the quorum latency increases proportionally. If the congestion is asymmetric (one link is congested, others are not), the leader may avoid the congested link by choosing a different follower for the quorum.

Severe congestion can cause missed heartbeats, triggering unnecessary leader elections. If the congestion lasts long enough, the cluster may cycle through leaders, each of which fails to replicate to the congested nodes. The solution is to prioritize consensus traffic using QoS (Quality of Service) settings on the network switches.

### Software Bugs

Consensus implementations are complex and contain subtle bugs. Common bugs include: incorrect handling of term comparisons during leader election, log inconsistency during snapshot installation, race conditions between the election timer and incoming RPCs, and incorrect compaction of entries that are still needed for recovery.

The best defense is rigorous testing. All consensus implementations should be tested with the Jepsen framework or similar fault injection tools. Production deployments should have a staging environment where failure scenarios are tested before they occur in production.

## When to Accept the Latency Cost

Consensus adds measurable latency to every write. The decision to use consensus depends on whether the system can tolerate the cost.

Accept the cost when: the system needs strong consistency across replicas, the system uses leader election that must prevent split-brain, the system stores configuration that must not be lost, the system provides distributed locks where correctness is critical, or the system needs atomic broadcast where all nodes must see the same messages in the same order.

Do not accept the cost when: the system can tolerate eventual consistency, the write throughput exceeds 30,000 ops/sec on a single consensus group, the write latency must be under 1ms, or the data is ephemeral and can be regenerated.

## Design Alternatives Comparison

| Approach | Consistency | Latency | Throughput | Fault Tolerance |
|---|---|---|---|---|
| No consensus (single node) | Strong | 0.1ms | 100K+ | None |
| Async replication | Weak | 0.1ms write + async | 100K+ | Read replicas survive |
| Quorum (N=3, W=2, R=2) | Strong | 1-5ms | 10-30K | 1 failure |
| Raft/Paxos | Strong | 1-10ms | 5-30K | floor(N/2) failures |
| CRDT | Eventual | 0.1ms | 100K+ | All nodes survive |
| Gossip | Eventual | 0.1ms | 100K+ | Most nodes survive |

**Peer-to-peer systems with no central authority**: Systems like Bitcoin and Ethereum use Byzantine fault tolerance (BFT) consensus or proof-of-work, not classical consensus. Classical consensus assumes benign failures (crashes), not malicious behavior.
