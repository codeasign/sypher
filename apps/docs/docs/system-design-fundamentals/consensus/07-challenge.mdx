## How the Challenges Work

Each challenge is tiered by difficulty. Foundation challenges test core knowledge. Design challenges test the ability to apply consensus concepts to real systems. Expert challenges test deep understanding of algorithm internals and production deployment.

Attempt each challenge without looking at the answer. Write or sketch your solution. Then compare with the model answer. The goal is not to produce an identical answer but to verify that your reasoning covers the same key points.

## Foundation Challenges

### Challenge F1: Quorum Math

A company deploys a consensus cluster with 4 nodes. A network partition splits the cluster into two groups: Group A has 3 nodes, Group B has 1 node. Explain why this deployment is problematic. Calculate the quorum size. Determine whether each group can elect a leader. Propose a better cluster size.

**Answer:**

A 4-node cluster is even-numbered, which is problematic. The quorum for 4 nodes is floor(4/2) + 1 = 3. If a partition splits the cluster into Group A (3 nodes) and Group B (1 node), Group A has 3 nodes which equals the quorum. Group A can elect a leader and continue operating. Group B has 1 node which is below the quorum of 3, so Group B cannot elect a leader or commit writes. So far this is acceptable: only one group operates, which is the correct behavior.

The problem occurs if the cluster splits into two groups of 2 nodes each. Each group has 2 nodes, and 2 does not meet the quorum of 3. Neither group can operate. The cluster is completely unavailable even though all 4 nodes are alive and connected within their groups.

The better approach is to use an odd number of nodes. With 3 nodes, the quorum is 2. A 2-1 split leaves the group of 2 operational and the group of 1 unable to operate. With 5 nodes, the quorum is 3. A 3-2 split leaves the group of 3 operational and the group of 2 unable to operate. Odd cluster sizes prevent ties and guarantee that at most one group can form a quorum.

### Challenge F2: Log Matching

In a Raft cluster with 5 nodes, the leader has the following log: entries at indices 1-5 with terms [term=1, term=2, term=2, term=3, term=4]. A follower has entries at indices 1-4 with terms [term=1, term=2, term=3, term=3]. The leader sends AppendEntries with prevLogIndex=4 and prevLogTerm=3. Will the follower accept this request? Explain why or why not. Then describe what happens next.

**Answer:**

The follower checks if its log has an entry at index 4 with term 3. The follower's log at index 4 has term 3. The check passes: prevLogIndex=4, prevLogTerm=3 matches the follower's entry. The follower appends the new entry (index 5, term=4) to its log and responds with success.

After this, the follower's log has entries [term=1, term=2, term=3, term=3, term=4]. The follower's log now matches the leader's log at indices 1, 2, 4, and 5 but not at index 3 (follower has term=3, leader has term=2). Wait -- let me re-examine.

Actually, the leader's log is:
- index 1: term=1
- index 2: term=2
- index 3: term=2
- index 4: term=3
- index 5: term=4

The follower's log is:
- index 1: term=1
- index 2: term=2
- index 3: term=3
- index 4: term=3

The leader sends AppendEntries with prevLogIndex=4, prevLogTerm=3. The follower checks index 4: it has term=3. The check passes because the terms match. The follower appends the new entry at index 5 with term=4.

After appending, the follower's log is:
- index 1: term=1
- index 2: term=2
- index 3: term=3
- index 4: term=3
- index 5: term=4

The logs now match at indices 4 and 5. But they still diverge at index 3: the leader has term=2, the follower has term=3. This divergence exists because the follower previously accepted an entry from a different leader in term 3 at index 3. The leader's AppendEntries did not check index 3 because prevLogIndex was 4. However, the leader will eventually detect this divergence when it tries to commit. The leader can only commit entries that are replicated to a majority. More importantly, the leader uses the log matching property: if two logs have an entry at the same index with the same term, they must be identical from that index backward. Since the entry at index 4 (term=3) matches, entries 1-4 should be identical. But in this case, index 3 differs. This means the AppendEntries should have failed. The prevLogIndex check should have been at index 3 (the previous index before the new entry), not index 4.

I need to correct this. In Raft, when the leader appends entry at index 5, it sets prevLogIndex=4, prevLogTerm=3. The follower checks index 4: term=3 matches. The follower appends the new entry at index 5. But the logs diverge at index 3. However, this is a problem because the log matching property is violated if the leader later needs to reference index 3. In practice, the leader will eventually send an AppendEntries that covers index 3 itself, which will fail, triggering the log reconciliation process. The leader decrements nextIndex for this follower and retries with earlier entries until the mismatch is resolved.

The key insight: the prevLogIndex check guarantees log matching only for the immediate predecessor. Deep log mismatches are detected and resolved iteratively.

### Challenge F3: Committed Entry Survival

A 5-node Raft cluster has committed entry at index 10 with term=5. Then the leader crashes. A new leader is elected in term=6. Prove that the new leader must have index 10, term=5 in its log.

**Answer:**

The entry at index 10 was committed in term=5. This means it was replicated to a majority of nodes (3 of 5) before the leader responded to the client. Call this majority set M_commit.

The new leader was elected in term=6. Election requires a majority of nodes to vote for the candidate. Call this majority set M_election.

In a 5-node system, any two majorities must intersect in at least one node. This is because |M_commit| + |M_election| >= 3 + 3 = 6, which exceeds the total of 5 nodes. So |M_commit intersect M_election| >= 1.

The intersection node has the committed entry (index 10, term=5) in its log because it was in the commit majority. When the new leader sends RequestVote, each voter checks that the candidate's log is at least as up-to-date as its own. The intersection node has the committed entry. The candidate must have a log as up-to-date as this node to win its vote. This means the candidate must have either a higher term in its last entry or the same term with a higher index. Since no entry beyond index 10 is committed, the candidate must have at least index 10, term=5 in its log. Therefore, the new leader always has the committed entry in its log.

## Design Challenges

### Challenge D1: Design a Distributed Lock Service

Design a distributed lock service using a consensus algorithm. The service must support lock acquisition, lock release, and lock timeout. Clients may crash while holding a lock. The service must be fault-tolerant (survive up to 2 node failures). Target 10,000 lock acquisitions per second with p99 latency under 10ms.

**Answer:**

**Architecture**: A 5-node Raft cluster. Each node runs a lock service process that reads from the Raft state machine. The lock state is stored as key-value pairs in the Raft log. Key = lock name, Value = lock holder (client ID) + lease expiration timestamp.

**Lock acquisition**: The client sends an Acquire(lock_name, client_id, lease_duration) request to any node. The request is forwarded to the Raft leader. The leader appends a log entry `{op: acquire, lock: lock_name, holder: client_id, expires_at: current_time + lease_duration}`. When committed, the leader checks the state machine: if the lock is free or expired, the request succeeds and the lock is assigned to the client. If the lock is held by another client whose lease has not expired, the request fails. The leader responds with success or failure.

**Lock release**: The client sends a Release(lock_name, client_id) request to the leader. The leader appends a log entry to clear the lock state. Only the current lock holder can release the lock.

**Lock timeout**: The lease duration is recorded in the state machine. After a crash, the lock remains held for the lease duration, then expires. Other clients can acquire the lock after the lease expires. This handles client crashes automatically -- no explicit cleanup is needed.

**Performance**: Raft with 5 nodes handles 10-30K writes/sec. p99 latency depends on the far majority node: for a leader in us-east-1 with followers in us-east-1, us-west-1, us-west-2, the farthest node for quorum is the second us-west node at ~40ms RTT. The leader needs 2 of 4 followers to acknowledge, so it can avoid the farthest follower. Expected p99 latency: 5-8ms in a single region.

**Failure handling**: If the Raft leader fails, a new leader is elected in 150-300ms. In-flight lock requests fail and should be retried by the client with exponential backoff. If a client crashes, its locks expire automatically after the lease duration. The lease duration should be long enough to tolerate GC pauses (5-10 seconds is typical).

### Challenge D2: Design a Strongly Consistent Key-Value Store

Design a key-value store that provides linearizable reads and writes. The store must handle 20,000 operations per second across 3 data centers. Each write must be durably stored and immediately visible to all reads.

**Answer:**

**Architecture**: Use a 5-node Raft cluster distributed across 3 data centers: 2 nodes in the primary data center, 2 in the secondary, and 1 in the third. This ensures that the primary data center can operate independently if the secondary data centers are unreachable (the primary's 2 nodes plus the leader itself form a quorum of 3 out of 5).

**Write path**: All writes go to the Raft leader. The leader appends the key-value pair to its log, replicates to followers, waits for a quorum of acknowledgments, commits, and responds to the client. The write is linearizable: once the client receives a success, all subsequent reads (including from other data centers) will see the new value, provided reads are also served through the Raft consensus.

**Read path for strong consistency**: Read requests must also go through the Raft leader or use a quorum read. A quorum read reads the value from the leader and a quorum of followers, choosing the value with the highest term/index. This guarantees linearizability. The simplest approach: route all reads to the leader and have the leader serve the value from its committed state machine. This avoids additional Raft round trips for reads but adds load on the leader.

**Read path optimization**: To reduce leader load, use read leases (also called leader leases). The leader grants a lease to followers: for a window of time (typically 1-2 seconds), the followers can serve linearizable reads directly without consulting the leader. The leader ensures no other leader can be elected during the lease period. This technique is used in etcd and Consul.

**Performance**: Writes: 15-20K/sec with 5 nodes across data centers, p99 latency ~20ms (includes cross-region RTT). Reads with leader lease: 100K+/sec, p99 latency under 5ms (local read).

### Challenge D3: Design a Multi-Region Consensus System

Design a consensus system that provides strongly consistent operations across 4 geographic regions. Each region has application servers. The system must tolerate the loss of any 1 region. Writes must be acknowledged within 200ms p99.

**Answer:**

The requirement for strongly consistent operations across 4 regions with single-region failure tolerance is challenging. A single Raft group spanning 4 regions would require 3 regions in the quorum at all times. If any region is lost, the remaining 3 regions can still form a quorum. However, write latency is bounded by the farthest region in the quorum.

**Architecture option 1: Single 7-node Raft group**: Deploy 2 nodes in the primary region and 1 node in each of the other 3 regions (total 5 nodes). Wait -- with 4 regions, use 2 nodes in the primary region and 1 node in each of the other 3 regions = 5 nodes total. Quorum is 3. If the primary region is lost, the remaining 3 regions have 3 nodes (one each). They form a quorum. Write latency: the leader in the primary region needs 2 acknowledgments from followers. It can get one from the other primary region node and one from the closest secondary region. This keeps p99 write latency under 100ms if regions are well-connected.

**Architecture option 2: Hierarchical consensus**: Use a local consensus group per region (3 nodes each, with local leader) and a global consensus group (5 nodes, one per region). Local writes go to the local group first, then propagate to the global group. This approach increases complexity but reduces write latency because local writes complete in 1 RTT.

**Recommended approach for 4 regions with under 200ms p99**: Use a 5-node Raft group with 2 nodes in the primary region and 1 node in each of the other 3 regions. The leader resides in the primary region. The quorum requires 3 nodes. The leader selects the fastest 2 followers to acknowledge, which are usually the second node in the primary region and the node in the nearest secondary region. This configuration provides write latency of 30-80ms p50 and under 200ms p99 if regions are within 50ms RTT of each other.

## Expert Challenges

### Challenge E1: Implement a Minimal Raft Protocol

Describe the minimum set of RPCs, state, and timers needed to implement a correct Raft consensus protocol. Do not write full code -- describe the data structures, the state machine transitions, and the message handling logic at a level of detail sufficient for implementation.

**Answer:**

**Core state on each node**: currentTerm (int), votedFor (nodeId or null), log (array of LogEntry `{term, command}`), commitIndex (int), lastApplied (int), state (Follower | Candidate | Leader).

**Leader-only state**: nextIndex[] (array of int, one per follower), matchIndex[] (array of int, one per follower).

**Persistent state** (must survive crashes): currentTerm, votedFor, log. Recover from disk on restart.

**RPC 1: RequestVote(term, candidateId, lastLogIndex, lastLogTerm) -> (term, voteGranted)**

Receiver logic:
- If term < currentTerm: respond with currentTerm, voteGranted=false.
- If votedFor is null or equals candidateId, and candidate's log is at least as up-to-date as receiver's log (lastLogTerm > receiver's lastLogTerm, or lastLogTerm == receiver's lastLogTerm and lastLogIndex >= receiver's lastLogIndex): set votedFor=candidateId, respond voteGranted=true.
- Otherwise: respond voteGranted=false.
- If term > currentTerm: set currentTerm=term, step down to Follower, reset votedFor, and then re-evaluate the vote.

**RPC 2: AppendEntries(term, leaderId, prevLogIndex, prevLogTerm, entries[], leaderCommit) -> (term, success)**

Receiver logic:
- If term < currentTerm: respond with currentTerm, success=false.
- If term > currentTerm: set currentTerm=term, step down to Follower.
- If log does not contain an entry at prevLogIndex with term == prevLogTerm: respond success=false.
- If log at prevLogIndex matches but any existing entry conflicts with a new entry (same index, different term): delete the existing entry and all that follow.
- Append any new entries not already in the log.
- If leaderCommit > commitIndex: set commitIndex = min(leaderCommit, index of last new entry).
- Respond success=true.

**Timers**:
- Election timeout: 150-300ms, randomized per node. Reset on receiving AppendEntries from current leader. On timeout: increment term, become Candidate, vote for self, send RequestVote to all nodes, reset election timeout.
- Heartbeat interval: 50-100ms. Only active on Leader. Send empty AppendEntries to maintain authority.

**Leader election**:
1. On election timeout: increment currentTerm. Set votedFor=self. Transition to Candidate. Send RequestVote to all nodes. Reset election timeout.
2. If receive votes from majority: become Leader. Set nextIndex[i] = lastLogIndex+1 for each follower. Set matchIndex[i] = 0 for each follower. Start sending heartbeats.
3. If receive AppendEntries from another leader with term >= currentTerm: step down to Follower.
4. If election times out (split vote): restart the election with a new randomized timeout.

**Log replication**:
1. On receiving a client request: leader appends entry to its log. Send AppendEntries to all followers with entries from nextIndex[target] onward.
2. On successful response: update matchIndex[i] and nextIndex[i] for that follower. Find N such that there exists a majority of matchIndex[i] >= N and log[N].term == currentTerm. If N > commitIndex, set commitIndex = N.
3. On failed response (term mismatch or log inconsistency): if term in response > currentTerm, step down. Otherwise decrement nextIndex[i] and retry AppendEntries with earlier entries.

**Safety**: An entry is committed only if it is replicated to a majority and it is from the leader's current term. Entries from previous terms are committed only when a later entry in the current term is committed (which implicitly confirms all earlier entries).

### Challenge E2: Consensus with Latency Asymmetry

A 5-node Raft cluster is deployed across 3 data centers with asymmetric network latencies: same-data-center RTT = 1ms, cross-data-center RTT = 50ms (A to B), 100ms (A to C), 50ms (B to C). The cluster has 2 nodes in Data Center A, 1 node in Data Center B, and 2 nodes in Data Center C. Design a placement strategy and leader selection policy that minimizes write latency. Calculate the expected p50 write latency.

**Answer:**

**Placement analysis**: The latency matrix is:
- A-A: 1ms (2 nodes in same DC)
- A-B: 50ms
- A-C: 100ms
- B-B: 1ms (1 node only, no intra-DC)
- B-C: 50ms
- C-C: 1ms (2 nodes in same DC)

**Quorum requirement**: 3 of 5 nodes must acknowledge.

**Leader placement options**:

Option 1: Leader in Data Center A (2 nodes).
- When leader is in A, the quorum can be: Leader A + other A node (1ms) + B node (50ms). Total latency: max(1, 50) = 50ms. Alternatively: Leader A + other A node + C node (100ms) = 100ms. The leader selects the fastest 2 followers, so p50 = 50ms.

Option 2: Leader in Data Center B (1 node).
- When leader is in B, the quorum must include nodes from at least one other DC. Best case: B + A node (50ms) + C node (50ms) = max(50, 50) = 50ms.
- Or B + A node + A's other node (50ms, all of A's within 1ms of each other) = 50ms.
- The B leader has only a single node in its DC. Each needed follower is at least 50ms away. p50 = 50ms.

Option 3: Leader in Data Center C (2 nodes).
- Similar to A: quorum of C leader + other C node (1ms) + B node (50ms) = 50ms.
- Or C + C node + A node (100ms) = 100ms. The leader picks the faster option.

**Optimal strategy**: Place the leader in either Data Center A or Data Center C (the DCs with 2 nodes). When the leader is in a DC with 2 nodes, it can form a quorum entirely within its own DC plus the single node in DC B (50ms). The p50 write latency is 50ms.

**Expected p50 latency**: 50ms. P99 may be higher: if the follower in DC B is slow or the network has jitter, the leader may need the follower in the far DC (100ms). Expected p99: 100-120ms.

**Leader selection policy**: Prefer a leader in the data center with the most nodes. If multiple data centers have equal node counts, prefer the data center with the lowest latency to the other nodes. Implement this by configuring the Raft cluster with lower election timeouts for nodes in the preferred data center, or by using a pre-vote phase that prevents non-preferred nodes from starting elections.

## Testing Your Knowledge: Full Design Blueprint

As a final challenge, sketch the complete design for a consensus-based coordination service. This blueprint brings together all the concepts from the previous sections.

### Requirements

- 5-node consensus cluster using Raft
- Distributed lock service with leases
- Configuration storage (key-value)
- Service discovery (registration and health checking)
- Must tolerate 2 node failures
- Must handle 5,000 writes/sec and 50,000 reads/sec
- Cross-region deployment (2 regions)
- Write latency under 50ms p99
- Read latency under 5ms p99

### Blueprint

**Cluster topology**: 5 nodes, 3 in primary region (us-east-1), 2 in secondary region (us-west-2). The leader is in the primary region. Quorum size: 3. During normal operation, the quorum consists of the leader plus 2 nodes in the primary region (1ms latency). Write latency: 5ms (1ms intra-region + 1ms fsync + 3ms overhead). If the primary region fails, the secondary region has 2 nodes. A leader election cannot complete in the secondary region alone because 2 nodes is below quorum of 3.

**Read optimization**: Use leader leases (read leases). The leader grants a 1-second lease to followers. Followers serve reads during the lease without contacting the leader. Read throughput: 50,000+ ops/sec.

**Lock service**: Locks stored as keys with values `{client_id, lease_expiry}`. Lock acquisition: conditional write (success only if key does not exist or lease is expired). Lock release: delete key. Lease renewal: periodic background heartbeat from client.

**Configuration storage**: Key-value pairs stored as Raft state machine entries. Watches implemented as long-polling HTTP connections. Each watch request registers a callback. When the key changes, the callback is triggered and the response is sent.

**Service discovery**: Services register themselves by creating an entry in the key-value store with their health check endpoint. A background health checker pings each registered endpoint. If the health check fails 3 times, the entry is removed.

**Failure handling**: If a follower fails, the leader continues with 4 nodes. If a second follower fails, the leader has 3 nodes, which meets quorum. If 3 nodes fail, quorum is lost, and the system becomes unavailable for writes. Reads from the 2 surviving nodes return the last committed state (not guaranteed fresh).

**Disaster recovery**: Daily snapshots to object storage (S3). In a total cluster loss, restore the snapshot to a new cluster in the primary region, then add nodes in the secondary region.
