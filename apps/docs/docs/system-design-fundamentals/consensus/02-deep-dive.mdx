## Internal Architecture

A consensus node consists of three main components: the consensus state machine, the replicated log, and the messaging layer. The state machine processes committed commands and maintains application state. The log stores commands in order. The messaging layer handles RPCs between nodes.

The log stores every command received by the system in order. Each entry has an index (sequential position), a term (the leader's term when the entry was created), and a command (the state machine operation). The commit index marks the highest log entry that is known to be safely replicated on a majority of nodes. The last applied index marks the highest log entry that has been executed against the state machine. Entries between lastApplied and commitIndex are committed but not yet applied.

## Algorithms

### Paxos in Detail

Paxos operates in three phases across two rounds of communication. The protocol involves three roles: proposers (who suggest values), acceptors (who vote on values), and learners (who learn the chosen value).

**Phase 1: Prepare**

The proposer selects a proposal number N (must be unique and monotonically increasing, typically using the pattern term*10000 + nodeId). The proposer sends a Prepare(N) request to all acceptors. The acceptors respond with a Promise if they have not seen a higher proposal number.

When an acceptor receives a Prepare(N) request, it compares N with its promisedProposalNumber. If N is greater, the acceptor promises not to accept any proposal less than N and responds with a Promise containing: the proposal number N it is promising for, the highest proposal number it has previously accepted (lastAcceptedProposal), and the value of that highest previously accepted proposal (lastAcceptedValue). If N is less than or equal to the promised number, the acceptor ignores the request (or responds with a rejection).

**Phase 2: Accept**

If the proposer receives Promises from a majority of acceptors, it may proceed to Phase 2. The proposer sets the value to propose: if any Promise included a previously accepted value, the proposer MUST use the value from the highest-numbered previously accepted proposal. If no Promises included a previous value, the proposer may choose any value.

When an acceptor receives an Accept(N, value) request, it checks that N is greater than or equal to its promisedProposalNumber. If true, the acceptor accepts the proposal: it records lastAcceptedProposal=N and lastAcceptedValue=value, and responds with Accepted(N). If N is less than promisedProposalNumber, the acceptor rejects the request.

**Phase 3: Learn**

After reaching quorum in Phase 2, the proposer sends a Learn(value) message to all learners so they can execute the chosen value. In practice, the acceptors can also broadcast the accepted value to learners to avoid a single point of failure at the proposer.

**Complexity Analysis**

Paxos requires two round trips (2 RTTs) for a single decision in the best case (Prepare + Accept). Each round trip involves sending to all nodes and receiving from a majority. Message complexity is O(2N) messages per decision, where N is the number of acceptors. Latency is dominated by the network RTT between the proposer and the farthest acceptor in the quorum.

**Multi-Paxos**

Multi-Paxos optimizes for the common case of many sequential decisions by using a long-lived leader. The leader performs Phase 1 once (Prepare/Promise) to establish leadership. Subsequent proposals skip Phase 1 and proceed directly to Phase 2 (Accept). This reduces the cost to 1 RTT per decision.

### Raft in Detail

Raft divides consensus into three subproblems: leader election, log replication, and safety. Each subproblem has a well-defined protocol with clear invariants.

**Leader Election**

Each node is in one of three states: Leader, Follower, or Candidate. Nodes start as Followers. Followers expect periodic heartbeats from the leader. If a Follower receives no heartbeat within the election timeout (typically 150 to 300 milliseconds), it increments its term, transitions to Candidate, and starts an election.

The election proceeds as follows. The Candidate sends RequestVote RPCs to all other nodes. A node grants its vote to the Candidate if: (1) the candidate's term is at least as large as the node's current term, (2) the node has not already voted in this term, and (3) the candidate's log is at least as up-to-date as the node's log (compared by last log term, then last log index). If the Candidate receives votes from a majority of nodes, it becomes the Leader. If it receives a RequestVote from another Candidate with a higher term, it steps down to Follower. If the election times out without a winner (split vote), the Candidate starts a new election with an incremented term and a new randomized timeout.

**Log Replication**

Once a Leader is elected, it handles all client requests. Each request is appended to the leader's log as a new entry. The leader then sends AppendEntries RPCs to all Followers in parallel.

The AppendEntries RPC includes the leader's term, the index and term of the log entry immediately preceding the new ones (prevLogIndex, prevLogTerm), the new entries, and the leader's commitIndex. Each follower checks that its log contains an entry at prevLogIndex with term prevLogTerm. If the check passes, the follower appends the new entries to its log. If the check fails (log inconsistency), the follower rejects the RPC. The leader then decrements nextIndex for that follower and retries with an earlier entry, eventually finding the point where the logs match. This mechanism is called log matching and is the core of Raft's consistency guarantee.

**Safety Properties**

Raft guarantees five safety properties:

Election safety: At most one leader can be elected in any given term.

Leader append-only: A leader never overwrites or deletes entries in its log; it only appends new entries.

Log matching: If two logs contain an entry with the same index and term, the logs are identical from that index down to index 1. This follows from the AppendEntries consistency check.

Leader completeness: If a log entry is committed in a given term, that entry will be present in the logs of all future leaders. This follows from the RequestVote consistency check: a candidate must have at least as much committed state as a majority of nodes.

State machine safety: If a server has applied a log entry at a given index to its state machine, no other server will ever apply a different log entry at the same index.

**Complexity Analysis**

Raft requires 1 RTT for a decision in the steady state (leader sends AppendEntries, receives majority response). Leader election takes between 1 and 2 election timeouts (150 to 600 ms) in the worst case. Message complexity is O(N) per AppendEntries broadcast. Raft achieves throughput proportional to the leader's bandwidth for outbound RPCs divided by the average entry size, typically 10,000 to 30,000 operations per second on modern hardware with a 3-node cluster.

**Stage 1: Client Request** -- The client sends a PUT or POST request to any node in the cluster. Many consensus implementations use a proxy layer that routes to the leader.

**Stage 2: Forward to Leader** -- If the request arrives at a follower, the follower forwards the request to the current leader. The redirect typically includes the leader's address in the response headers.

**Stage 3: Leader Appends** -- The leader validates the request, serializes it as a log entry, appends it to its own log, and assigns a term and index.

**Stage 4: Broadcast** -- The leader sends AppendEntries RPCs to all followers in parallel. The RPC includes the new entry plus any previously sent but unacknowledged entries (batching).

**Stage 5: Follower Validation** -- Each follower checks the log consistency condition: does the entry at prevLogIndex match prevLogTerm? If yes, the follower appends the new entries.

**Stage 6: Follower Acknowledgment** -- The follower sends a success response back to the leader. In failure cases, the follower sends a negative response with its current term, allowing the leader to detect its own staleness.

**Stage 7: Leader Counts Acknowledgments** -- The leader maintains an array matchIndex[] tracking the highest log index known to be replicated on each follower. When the leader receives an acknowledgment, it updates matchIndex for that follower.

**Stage 8: Quorum Detection** -- The leader finds the highest index N such that a majority of matchIndex[] entries are >= N. This index N becomes the new commitIndex. The leader must also ensure that N is from its own current term (Raft does not commit entries from previous terms by counting replicas alone -- it waits for a new entry in the current term to be committed first, which implicitly commits all previous entries).

**Stage 9: Commit** -- The leader sets commitIndex = N. All entries up to commitIndex are considered committed.

**Stage 10: Apply to State Machine** -- The leader applies all entries from lastApplied+1 to commitIndex to the state machine. The state machine produces an output (the result of the operation).

**Stage 11: Client Response** -- The leader sends the result back to the client. The client knows the write is durable on a majority of nodes.

**Stage 12: Commit Broadcast** -- The next AppendEntries RPC (or a dedicated heartbeat) includes the updated leaderCommit index. Followers see the new commitIndex and apply entries to their own state machines.

## Consistency and State

### Node Crash Mid-Round

Consider a 3-node cluster where the leader crashes after appending an entry to its log but before sending any AppendEntries. The entry exists only in the leader's memory and is lost. The client receives a timeout. The new leader (elected by the remaining 2 nodes) has no knowledge of the lost entry. The client must retry -- and the retried operation must be idempotent.

### Network Partition

A network partition splits a 5-node cluster into two groups: Group A (3 nodes) and Group B (2 nodes). Group A has a quorum (3 nodes), so it can elect a leader and continue processing. Group B does not have a quorum (2 nodes), so it cannot elect a leader. Requests to Group B fail or time out. When the partition heals, Group A's leader steps down (sees higher term from Group B's potential leader) or continues as leader. Group B's nodes catch up by receiving the log entries they missed during the partition.

### Concurrent Proposals

Two clients issue writes at nearly the same time. Client 1 sends x=42, Client 2 sends x=99. Both reach the leader. The leader serializes: it appends x=42 to its log at index 8, broadcasts AppendEntries for index 8. Before receiving quorum, it appends x=99 at index 9. The leader commits index 8 (x=42) after receiving majority ack. It then commits index 9 (x=99). Both writes are applied to the state machine in order: first x=42, then x=99. The final value of x is 99. The ordering is deterministic because only the leader appends to its log.

### Stale Leader Detection

A node that was leader in a previous term but was partitioned away may still believe it is the leader. When it reconnects, it may attempt to send AppendEntries to nodes that have already elected a new leader in a higher term. The followers detect the stale leader because the AppendEntries contains a term lower than their current term. They respond with their current term. The stale leader sees a higher term and immediately steps down to follower. This detection happens within one RTT.

## Performance

### Latency by Cluster Size

The latency of a consensus write depends on the cluster size and the network RTT between nodes. In the steady state (no leader election), each write requires one round trip from the leader to the followers. The leader must receive acknowledgments from a majority of nodes.

### Configuration Parameters

**Cluster size**: 3 nodes for minimum fault tolerance (tolerates 1 failure). 5 nodes for standard production (tolerates 2 failures). 7 nodes for high resilience (tolerates 3 failures, used in ZooKeeper metadata clusters). Clusters larger than 9 nodes are rare because throughput decreases significantly.

**Election timeout**: 150 to 300 milliseconds is standard. Shorter timeouts detect failures faster but increase false positive elections. Longer timeouts reduce election frequency but delay failover. The timeout should be randomized across nodes to prevent split votes.

**Heartbeat interval**: 50 to 100 milliseconds is typical. Heartbeats carry no log entries and consume minimal bandwidth. A common ratio is heartbeat interval = election timeout / 3 to allow 3 missed heartbeats before election.

**Batch size**: The leader can batch multiple client requests into a single AppendEntries RPC. Batch sizes of 64 to 256 entries balance latency and throughput. Larger batches improve throughput at the cost of per-request latency for the trailing entries.

**Pipeline**: Many consensus implementations support pipeline replication where the leader sends the next entry before the previous one is acknowledged. Pipeline depth is typically 64 to 256 entries for maximum throughput.

## Advanced Performance Characteristics

### Impact of Entry Size on Throughput

The size of each log entry directly affects throughput. Entry size includes the command payload plus the Raft/Paxos metadata (term, index, previous log index/term). For small entries (under 256 bytes), the metadata overhead dominates. For large entries (over 10 KB), the network bandwidth is the limiting factor.

Typical throughput for different entry sizes on a 3-node cluster with 1 Gbps links:
- 100-byte entries: 30,000 ops/sec (bandwidth: 3 MB/sec, CPU-bound)
- 1 KB entries: 15,000 ops/sec (bandwidth: 15 MB/sec)
- 10 KB entries: 3,000 ops/sec (bandwidth: 30 MB/sec, network-bound)
- 100 KB entries: 300 ops/sec (bandwidth: 30 MB/sec, network-bound)

### Fsync Strategies

The disk fsync is the most expensive single operation in a consensus write. Modern SSDs complete an fsync in 0.5 to 2 milliseconds. Hard disk drives take 5 to 15 milliseconds. The fsync strategy determines the durability-latency trade-off.

**Per-entry fsync**: The leader calls fsync after appending each entry. This provides the strongest durability guarantee: a committed entry is guaranteed to be on disk. The throughput is bounded by the fsync latency: at 1ms per fsync, the maximum throughput is 1,000 entries per second.

**Batch fsync**: The leader accumulates multiple entries before calling fsync. A batch of 64 entries with 1ms fsync latency achieves 64,000 entries per second. The trade-off: the last entry in the batch is not visible until the batch is flushed. The leader may lose the entire batch if it crashes before the fsync completes.

**Group commit**: Multiple concurrent write requests are collected into a single batch before fsync. This is the standard approach in databases and consensus implementations. The leader waits up to a configurable timeout for additional requests, then fsyncs the batch. Group commit with a 1ms timeout achieves approximately 1,000 batches per second, with each batch containing the entries that arrived during that 1ms window.

### The Cost of the Third Acknowledgement

In a 5-node cluster, the leader needs 3 acknowledgments (including its own). The third acknowledgment is the latency bottleneck. If the leader and one follower are in the same data center (1ms), and the second follower is in another data center (40ms), the write latency is 40ms. The leader could choose to accept only 3 acknowledgments from the fast followers and ignore the slow ones, but the slow followers may be needed for fault tolerance.

In practice, the leader should be placed in the data center with the most nodes. With 2 nodes in DC A, 2 nodes in DC B, and 1 node in DC C, the leader in DC A can get quorum from the other DC A node (1ms) and one node from DC B (40ms). The write latency is 40ms. The DC C node is not needed for quorum.

### Prioritizing Leadership

Not all nodes are equally qualified to be leader. Nodes with faster disks, lower network latency, and more CPU resources should be preferred. Consensus implementations should support a leadership preference mechanism.

In Raft, this can be achieved through pre-vote. A node that is not in the preferred leader set does not start elections. Instead, it waits for a node in the preferred set to become leader. This prevents elections by nodes that would degrade performance.

In practice, many deployments use a manual approach: configure the preferred leader with a shorter election timeout, making it more likely to win elections. This is not a perfect solution but works well in practice.

## Log Compaction and Snapshots

The consensus log grows with every write. Without intervention, the log would grow without bound. Two mechanisms control log growth: compaction and snapshots.

**Snapshotting**: The state machine periodically serializes its entire state to a snapshot file. The snapshot is stored on disk and replaces all log entries up to the snapshot index. After the snapshot, the log starts fresh from the snapshot index. This is the primary mechanism for controlling log size.

**Log compaction**: Older committed entries that are no longer needed for crash recovery are deleted. The deletion is safe because the current state is captured in the snapshot. The log only needs to contain entries after the last snapshot.

**InstallSnapshot RPC**: When a new or slow follower joins the cluster, it may be behind the last snapshot. The leader sends the snapshot directly to the follower via the InstallSnapshot RPC. The follower applies the snapshot to its state machine and then receives subsequent log entries normally.
