## The Problem Story

A distributed database cluster running in production across three data centers handles 50,000 requests per second. All nodes replicate data using asynchronous replication. One day, a network switch fails between Data Center A and Data Center B. The partition lasts 47 seconds. During those 47 seconds, nodes in Data Center A and Data Center B each elect their own leader. Both leaders accept writes. Both leaders assign the same version number to different values for the same key. When the partition heals, the two leaders attempt to reconcile: Node A has value "X" with version 3, Node B has value "Y" with version 3. The system has no way to determine which value is correct. Data corruption propagates to all downstream services — caches, search indexes, reporting pipelines. The incident takes 8 hours to fully resolve, affects 1.2 million users, and costs the company $340,000 in SLA penalties.

A second team encounters a different consensus failure. A configuration service stores the mapping of which server is the primary for each shard. Two administrators issue conflicting updates within 200 milliseconds of each other. The configuration service accepts both updates because it uses last-writer-wins with clock timestamps. The problem: the two administrators' clocks are skewed by 400 milliseconds due to unsynchronized NTP. The wrong update wins. Half the shards map to servers that are no longer primary. The system serves stale data for 14 minutes before the error is detected.

A third incident involves a distributed lock service. A microservice acquires a lock on a shared resource, then performs a 30-second operation. Before the operation completes, a garbage collection pause of 8 seconds causes the lock's heartbeat to expire. A second microservice acquires the same lock. Both microservices now believe they hold the exclusive lock. Both write to the shared resource. The result is a corrupted state that takes three days to reconstruct from backups.

All three incidents share the same root cause: **the nodes in the system could not agree on a single truth**. The system lacked a consensus mechanism that guarantees all nodes reach the same decision even when failures occur.

## The Insight

Consensus is the problem of getting multiple distributed nodes to agree on a single value, even when some nodes fail. It is the foundation of leader election, atomic broadcast, and strongly consistent replication.

A jury in a courtroom must reach a unanimous verdict. Twelve jurors hear evidence, deliberate, and vote. If all twelve agree, the verdict is delivered. If the jury cannot reach consensus, the judge declares a mistrial. The process requires that every juror votes, that the votes are counted correctly, and that the final verdict is the same for everyone. Now imagine that some jurors cannot hear the evidence because of noise (network delay), some jurors leave the room temporarily (node failure), and some jurors change their vote based on incomplete information (partial updates). The legal system has procedures for each of these cases — sequestering the jury, rereading testimony, requiring unanimous consent. Consensus algorithms have the same structure: a set of participants, a proposal mechanism, a voting protocol, and a commitment rule.

The core difference between a single-node system and a distributed consensus system is that a single node can make a decision instantly by looking at its own state. A distributed system must coordinate across multiple nodes, each of which may fail, lie, or disagree. Consensus algorithms provide the guarantees that make distributed coordination possible.

## Core Vocabulary

**Consensus** — A distributed algorithm that ensures multiple nodes agree on a single value. The value can be a configuration entry, a leader identity, a log entry, or a database row. Consensus guarantees that if any node commits a value, all correct nodes eventually commit the same value.

**Quorum** — The minimum number of nodes that must participate in a decision for the decision to be valid. A simple majority quorum for a cluster of N nodes is floor(N/2) + 1. For a 5-node cluster, the quorum is 3. For a 7-node cluster, the quorum is 4.

**Leader** — A distinguished node that coordinates the consensus process. The leader receives client requests, proposes values to followers, and tracks acknowledgments. Only one leader exists at any time in a correct consensus run.

**Follower** — A node that accepts proposals from the leader, acknowledges them, and commits values when instructed. Followers do not initiate proposals but can initiate a leader election if the leader becomes unresponsive.

**Proposal** — A value submitted for consensus. The leader assigns a unique proposal number or term to each proposal to distinguish it from previous and concurrent proposals.

**Accept** — The phase where a node promises to commit a proposed value. In Paxos, this is the accept phase. In Raft, this is the log replication phase where followers append entries to their logs.

**Commit** — The point at which a value is considered final. Once a value is committed, no correct node can decide a different value. The committed value is applied to the state machine.

**Term or epoch** — A monotonically increasing number that identifies a period of leadership. Each term starts with a leader election. Terms prevent stale leaders from making decisions and ensure that newer decisions override older ones.

**Fault tolerance** — The maximum number of node failures a consensus system can survive while remaining operational. With N nodes, a consensus system tolerates floor((N-1)/2) failures. A 3-node cluster tolerates 1 failure. A 5-node cluster tolerates 2 failures.

**Majority** — More than half of the nodes in the cluster. A majority is the minimum quorum for most consensus algorithms. If a majority of nodes agree on a value, that value is committed.

**Liveness** — The property that the system eventually makes progress. A consensus algorithm is live if, given enough time, a proposal is eventually accepted. The FLP impossibility result proves that no consensus algorithm can guarantee liveness in an asynchronous network with even one faulty node.

**Safety** — The property that the system never makes a wrong decision. A consensus algorithm is safe if no two correct nodes ever decide different values and no node decides a value that was not proposed. Safety is preserved even during failures, as long as the algorithm is followed correctly.

**Split-brain** — A failure scenario where a network partition causes two groups of nodes to each elect a leader and operate independently. When the partition heals, the two leaders may have committed conflicting values. Consensus algorithms prevent split-brain by requiring a majority quorum for any decision.

**Log replication** — The process of copying state machine commands from the leader to followers. Each command is appended to a replicated log, and when a command is committed, it is applied to the state machine. Log replication ensures that all nodes execute the same commands in the same order.

### Step 3: Leader broadcasts to followers

The leader sends an AppendEntries RPC (in Raft) or an Accept message (in Paxos) to all followers. The message includes the new log entry, the leader's term, and the index of the preceding log entry.

### Step 4: Followers validate and acknowledge

Each follower checks that the leader's term is at least as recent as its own term. If valid, the follower appends the entry to its own log and sends an acknowledgment back to the leader.

### Step 5: Leader detects quorum

The leader counts acknowledgments. For a 5-node cluster, the quorum is 3 (the leader's own vote plus 2 followers, or 3 followers). Once the leader has received acknowledgments from a majority of nodes, the entry is committed.

### Step 6: Leader commits and broadcasts commit

The leader applies the entry to its state machine (x = 42) and sends a commit notification to all followers. The commit updates the commit index on each follower.

### Step 7: Followers apply to state machine

Each follower, upon receiving the commit notification, applies the entry to its own state machine. All nodes now have x = 42 in their state machine.

### Step 8: Leader responds to client

The leader sends a success response to the client. The client knows that the value is durably stored on a majority of nodes and will never be lost or overwritten.

### Step 9: Leader election (failure scenario)

If the leader fails, followers detect the failure through heartbeat timeout. A follower increments its term number and starts a leader election. It votes for itself and requests votes from other followers. A candidate wins the election if it receives votes from a majority of nodes.

### Step 10: New leader handles stale entries

After election, the new leader checks its log for entries that were committed in the previous term but not yet replicated to all followers. It replicates any missing entries to ensure all nodes are synchronized.

## Variants

### Paxos (Original)

Paxos was the first practical consensus algorithm, published by Leslie Lamport in 1998. It uses three phases: Prepare, Promise, Accept, and Learn. Paxos is proven correct under asynchronous network conditions with crash failures. The algorithm is notoriously difficult to implement correctly — even experienced engineers struggle with the subtle interactions between phases.

Raft was designed specifically for understandability. It decomposes consensus into three subproblems: leader election, log replication, and safety. Raft uses a strong leader approach: the leader handles all client requests, coordinates log replication, and manages commit decisions. The algorithm is used by etcd, Consul, and many production systems.

## Mental Model Check

Five questions to verify understanding before moving to the deep dive.

**Question 1: A 3-node consensus cluster has one node fail. The remaining two nodes disagree on a proposed value. What happens?**

Answer: The two remaining nodes cannot reach consensus because they need a majority quorum of floor(3/2) + 1 = 2 nodes. With 2 nodes alive, they can still form a quorum if both agree. But if they disagree, neither can commit. The cluster is stuck until the failed node recovers or a new leader election resolves the disagreement. This is why production clusters use 5 nodes — with 2 failures, the remaining 3 nodes can still form a quorum.

**Question 2: What is the minimum number of nodes needed to tolerate 2 failures?**

Answer: 5 nodes. A consensus cluster with N nodes tolerates floor((N-1)/2) failures. For 2 tolerated failures: floor((N-1)/2) >= 2, so N-1 >= 4, N >= 5. A 5-node cluster tolerates 2 failures. A 3-node cluster tolerates only 1 failure.

**Question 3: A client sends a write to a consensus cluster. The leader receives the request but crashes before replicating to any follower. Is the write committed?**

Answer: No. The write exists only in the leader's log. When the leader crashes, the new leader may or may not have this entry. If the entry was not replicated to any follower, it is lost. The client receives a timeout or error and must retry. This is the difference between a leader that has accepted a request and a cluster that has committed it.

**Question 4: Two leaders exist in the same consensus cluster at the same time. What happened?**

Answer: Two leaders existing simultaneously is a split-brain condition, which should not happen in a correct consensus implementation. The most likely causes are: a network partition split the cluster into two groups, each with a majority; or a bug in the leader election protocol. In Raft, a leader must receive a majority of votes to be elected. If the cluster is partitioned into two groups of 3 and 2 nodes, only the group of 3 can elect a leader (majority of 5). The group of 2 cannot elect a leader because it cannot reach majority. This is why odd-sized clusters are preferred.

**Question 5: How does a consensus algorithm guarantee that a committed value is never lost?**

Answer: A value is committed only when it has been replicated to a majority of nodes. Any future leader election requires a majority of nodes to participate. Since the majority set that committed the value and the majority set that elects the new leader must overlap by at least one node, the new leader always has at least one node that knows the committed value. The new leader replicates this value to all followers. This overlap guarantee is the core safety property of consensus algorithms.

## The Three Node States

Every node in a consensus system operates in one of three states. Understanding these states is essential for reasoning about consensus behavior.

**Leader**: The node that coordinates all consensus activity. The leader receives all client write requests, appends them to its log, and broadcasts them to followers. The leader determines the order of entries in the log. Only one leader may exist at any time in a correct consensus run. The leader sends periodic heartbeats to maintain its authority.

**Follower**: A passive node that accepts entries from the leader. Followers do not initiate proposals or respond to client writes directly. They maintain their own copy of the log and apply committed entries to their state machine. If a follower receives no communication from the leader within the election timeout, it transitions to Candidate.

**Candidate**: A transitional state during leader election. A follower becomes a Candidate when its election timeout fires. The Candidate increments its term, votes for itself, and sends RequestVote messages to all other nodes. If it receives votes from a majority, it becomes the Leader. If it receives a message from a valid leader, it returns to Follower. If the election times out without a winner, it starts a new election.

## Consensus and the CAP Theorem

Consensus systems are CP systems under the CAP theorem: they provide Consistency and Partition tolerance, but sacrifice Availability during partitions. When a network partition splits the cluster, the minority side must stop accepting writes to maintain consensus safety. This is the availability sacrifice.

The trade-off is explicit: during a partition, the system is available on the majority side and unavailable on the minority side. This is the correct behavior. A system that remained available on both sides during a partition would risk split-brain, which violates the consensus safety guarantee.

The partition tolerance of a consensus system is determined by its cluster size. A 3-node cluster tolerates 1 failure but loses availability if 2 nodes become unreachable (the remaining 1 node cannot form a quorum). A 5-node cluster tolerates 2 failures and remains available as long as 3 nodes can communicate.

## The Role of Timeouts

Consensus algorithms use timeouts to detect failures and trigger leader elections. The timeout values directly affect system behavior.

**Election timeout**: The duration a follower waits without hearing from the leader before starting an election. Typical values: 150 to 300 milliseconds. These values are randomized per node to prevent split votes. If all nodes used the same timeout, they would all become candidates simultaneously, producing a split vote in every election cycle.

**Heartbeat interval**: The interval at which the leader sends heartbeats to followers. Typical values: 50 to 100 milliseconds. A shorter interval detects leader failures faster but generates more network traffic. A common rule: heartbeat interval * 3 = minimum election timeout, allowing 3 missed heartbeats before election.

**RPC timeout**: The maximum time a leader waits for a response from a follower before considering the follower unresponsive. Typical values: 50 to 500 milliseconds. If a follower does not respond within this timeout, the leader stops trying to replicate to that follower and continues with the remaining nodes.

## Consensus vs Other Coordination Mechanisms

Consensus is one approach to distributed coordination, but it is not the only one. Understanding the alternatives helps determine when consensus is the right tool.

**Leader election without consensus**: A simple approach where a node announces itself as leader and other nodes accept the announcement. This works if only one node ever attempts to become leader. If two nodes announce simultaneously, a tie-breaking mechanism is needed. This approach is used by systems that tolerate a brief period of uncertainty during leader handoff.

**Distributed locks with leases**: A lock service (Redis, ZooKeeper) provides lock acquisition and release with time-based leases. The lock holder renews the lease periodically. If the holder crashes, the lease expires and another node can acquire the lock. Distributed locks do not provide the same guarantees as consensus: a lock can be acquired by two nodes if the lease expires prematurely due to clock skew or GC pauses.

**Gossip-based membership**: Systems like Cassandra use gossip protocols to maintain cluster membership and detect failures. Gossip is eventually consistent: nodes converge on the membership view over time, but there is a window where different nodes may have different views. Gossip is not suitable for decisions that require immediate agreement.

**CRDTs (Conflict-free Replicated Data Types)**: CRDTs allow concurrent updates without coordination. Each node can update its local copy independently, and the updates merge deterministically. CRDTs provide eventual consistency with no coordination overhead. They are suitable for data that can tolerate a merge-based reconciliation model.

## Practical Consensus in Three Sentences

Consensus guarantees that a majority of nodes agree on a value before the value is considered final. This requires a leader to coordinate, a log to record entries, and a quorum to commit. The cost is one network round trip plus a disk write per decision, and the system stops making progress if a majority of nodes cannot communicate.
