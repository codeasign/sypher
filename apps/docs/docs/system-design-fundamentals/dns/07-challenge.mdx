## Tier 1 — Foundation

These challenges are designed to be completed in 5 minutes each. They test basic understanding of DNS concepts and terminology. Say your answer aloud or write it down, then compare with the model answer.

### Challenge 1: Explain the Recursive Resolution Path

**Prompt**: Draw the full path of a DNS recursive resolution for `api.example.com` starting from the browser and ending at the authoritative nameserver. For each step, state what information is exchanged. Identify the three caching layers along the path.

**Evaluation criteria (pass / fail)**:
- Lists 3 caching layers (browser, OS, resolver) — pass
- Names root, TLD, and authoritative servers in order — pass
- Mentions referrals (not final answers) from root and TLD — pass
- Mentions TTL is returned with the answer — pass
- Fails on any of the above — fail

**Model answer**: The path has 5 hops through 4 caching layers:

1. Browser cache check: the browser looks up `api.example.com` in its in-memory DNS cache. On hit (within TTL): returns IP immediately. On miss: proceeds to OS.
2. OS cache check: the operating system's stub resolver checks its own cache (or /etc/hosts file on Linux). On miss: sends query to configured recursive resolver (e.g., 8.8.8.8).
3. Root server: the recursive resolver queries a root server (e.g., 198.41.0.4). The root does not have the answer. It returns a referral to the .com TLD servers (a.gtld-servers.net) with glue records for their IPs.
4. TLD server: the resolver queries a .com TLD server. The TLD server does not have the answer either. It returns a referral to the authoritative nameservers for example.com (ns1.example.com) with glue records. 
5. Authoritative server: the resolver queries the authoritative nameserver for example.com. The authoritative server holds the A record for `api.example.com` and returns the IP address (e.g., 93.184.216.34) with a TTL.

Three caching layers along the path: (1) browser cache, (2) OS cache, (3) recursive resolver cache. Each caches the result for the duration of the TTL. The total time for a complete miss is 20-200ms.

### Challenge 2: Diagnose a TTL Problem

**Prompt**: Your team changed the IP address for `api.example.com` from `203.0.113.10` to `198.51.100.20` exactly 12 hours ago. You set the TTL to 3600 seconds on the new record. 40% of users are still hitting the old IP. What is happening? What three things would you check to confirm your diagnosis?

**Evaluation criteria (pass / fail)**:
- Identifies TTL as the root cause — pass
- Mentions that users who resolved the old record within the last 3600 seconds still have cached result — pass
- Mentions at least two of: check resolver cache via dig, check actual TTL on the record, check if old server still accepts connections — pass
- Mentions that TTL was set on the new record but the old record may have had a longer TTL — pass
- Fails on any of the above — fail

**Model answer**: The TTL of 3600 seconds means any resolver that cached the OLD IP (203.0.113.10) before the change will continue to return the old IP for up to 3600 seconds from when it last resolved. If a resolver resolved the old record 5 minutes before the change, that resolver serves the old IP for another 3550 seconds. Since resolvers around the world queried at different times, they have different cache expiry times. The 40% figure suggests that resolvers representing 40% of your traffic cached the record less than 3600 seconds ago.

Three things to check:
1. Query the actual A record with `dig @ns1.example.com api.example.com A` to verify the authoritative server returns the new IP with the correct TTL.
2. Query from a resolver that users report as broken: `dig @8.8.8.8 api.example.com A` to see what the resolver returns. Include `+ttlid` to see remaining TTL.
3. Check the old server's logs: is it still receiving traffic? If 40% of users still hit it, the server should show 40% of normal traffic.

To fix: wait for the remaining TTL to expire. Alternatively, reduce TTL on the new record to 60 seconds so all resolvers refresh within 1 minute, then maintain the reduced TTL for 48 hours to ensure all resolvers have refreshed at least once with the shorter window.

### Challenge 3: Explain Anycast DNS

**Prompt**: You are deploying DNS for a global application. Explain why anycast is the standard approach for global DNS. What would happen if you used unicast instead? Name two cloud providers that offer anycast DNS.

**Evaluation criteria (pass / fail)**:
- Defines anycast: multiple servers share the same IP, BGP routes to the nearest — pass
- Explains unicast alternative: single server per IP, needs GeoDNS to route users — pass
- Identifies anycast advantage: automatic failover (BGP withdraws bad route), no client configuration — pass
- Names two providers: Route53 and CloudDNS (or Azure DNS) — pass
- Fails on any of the above — fail

**Model answer**: Anycast DNS announces the same IP address from multiple data centers connected via BGP. When a client's resolver sends a query to that IP, BGP routers direct it to the nearest data center. This provides two critical benefits: (1) low latency — the query goes to the nearest server (Tokyo client resolves in Japan, not Virginia); (2) automatic failover — if the Tokyo server fails, BGP withdraws the route and queries automatically flow to the next closest server (Seoul or Singapore) with zero configuration change.

With unicast DNS, each server has a unique IP. You would need GeoDNS (which returns different IPs based on client location) to route users to the nearest server, and you would need health monitoring to update the GeoDNS configuration when a server fails. Unicast DNS requires explicit traffic management; anycast handles it at the network layer.

Two cloud providers with anycast DNS: AWS Route53 and Google Cloud DNS (also Azure DNS with Azure Traffic Manager).

## Tier 2 — Design

These challenges are 15-minute mini system design exercises. Design a solution, draw the architecture, and explain your decisions. Each challenge includes a rubric with scoring dimensions.

### Challenge 4: Design DNS for a Multi-Region API

**Prompt**: Design the DNS architecture for a REST API that serves 5 million daily active users across 3 regions: us-east-1, eu-west-1, and ap-southeast-1. The API must route each user to the nearest region. If a region fails, traffic must shift to the next-healthy region within 2 minutes. Users access the API at `api.example.com`.

**Constraints**:
- Cloud provider: AWS
- Budget: under USD 500/month for DNS infrastructure
- The API instances in each region run behind an internal load balancer

**Rubric**:

| Dimension | Poor (0 pt) | Acceptable (1 pt) | Strong (2 pts) |
|---|---|---|---|
| DNS provider choice | No mention | Route53 | Route53 with health checks |
| Routing strategy | Single A record | Multiple A records | Latency-based routing + health checks |
| TTL strategy | Fixed TTL | 60s default TTL | 60s for active, 300s for stable, explains why |
| Failover design | No failover | Manual DNS change | Automated (health check removes failed region) |
| Cost awareness | No mention | Rough estimate | Itemized breakdown |

**Model answer**:

Architecture:
1. Use AWS Route53 with latency-based routing. Create a single hosted zone for `example.com`.
2. Create three latency records for `api.example.com`, one per region:
   - `api.example.com` — latency — us-east-1 — points to us-east-1 ALB DNS name
   - `api.example.com` — latency — eu-west-1 — points to eu-west-1 ALB DNS name
   - `api.example.com` — latency — ap-southeast-1 — points to ap-southeast-1 ALB DNS name
3. Create Route53 health checks for each ALB endpoint. Each health check hits the ALB's health check endpoint every 10 seconds (3 consecutive failures = unhealthy).
4. Associate health checks with the latency records. If ap-southeast-1 goes unhealthy, Route53 stops returning its record. Traffic distributes between the remaining two healthy regions.
5. TTL: 60 seconds. This ensures that within 60 seconds of a health check failure, all resolvers have stopped returning the failed region's IP.

Failover timing: 3 consecutive health checks failing * 10 seconds = 30 seconds detection + 60 seconds TTL = 90 seconds total. Within the 2-minute requirement.

Cost: USD 0.50/month for the hosted zone. 5M DAU * 50 page views * 2 DNS queries = 500M queries/month * USD 0.40/million = USD 200/month. 3 health checks * USD 0.50/check/month = USD 1.50/month. Total: approximately USD 202/month. Well under budget.

### Challenge 5: Design DNS for Internal Microservice Discovery

**Prompt**: You have 200 microservices running on 500 Kubernetes pods across 3 AWS VPCs (us-east-1, us-west-2, eu-west-1). Services need to discover each other by name (e.g., `user-service.prod.svc.internal`). The discovery must work across VPCs. Design the DNS architecture.

**Constraints**:
- Must support SRV records for port-based discovery
- Caching should be minimal (services restart frequently, IPs change)
- Must not expose internal service names to the public internet

**Rubric**:

| Dimension | Poor (0 pt) | Acceptable (1 pt) | Strong (2 pts) |
|---|---|---|---|
| Service registry | No registry | Mentions Consul | Consul or etcd with automatic registration |
| Cross-VPC resolution | No cross-VPC | VPC peering + forwarding | VPC peering + Route53 Resolver forwarding rules |
| DNS interface | Kubernetes DNS | CoreDNS | CoreDNS with etcd/Consul plugin |
| SRV record support | Not mentioned | Supports SRV | Explains SRV format with example |
| Security | No security | Restricts resolver access | Private zones, TSIG authentication |

**Model answer**:

Architecture:
1. Deploy Consul as the service registry in each VPC. Services register themselves on startup — Consul agent on each pod monitors health and deregisters on failure.
2. Deploy CoreDNS in each VPC configured with the Consul plugin. CoreDNS resolves queries against the Consul registry in real-time. TTL: 0 seconds (no caching — always fetch fresh).
3. Connect VPCs with VPC peering. Configure Route53 Resolver forwarding rules so that queries for `*.svc.internal` from any VPC forward to the CoreDNS instances in the VPC that owns that service.
4. SRV records: CoreDNS automatically generates SRV records for Consul services. Example: `user-service.prod.svc.internal SRV 0 50 8080 10.0.1.50` — service runs on port 8080, weight 50, priority 0.
5. Security: Route53 Resolver forwarding rules restrict inbound queries to the peering VPCs. No public exposure. Optionally add TSIG authentication between CoreDNS and Consul.

Cross-VPC resolution flow:
```
Pod in us-west-2 queries: user-service.prod.svc.internal
-> CoreDNS in us-west-2 (check local Consul: miss for us-east-1 service)
-> Route53 Resolver in us-west-2 forwards to us-east-1 resolver endpoint
-> CoreDNS in us-east-1 (check local Consul: hit)
-> Returns SRV: user-service.prod.svc.internal 10.0.1.50:8080
-> Response propagates back
Total latency: under 20ms
```

### Challenge 6: Migrate DNS Providers with Zero Downtime

**Prompt**: You are migrating `example.com` from a legacy DNS provider to Route53. The domain has 50 records, 10,000 QPS of traffic, and an MX record for email that must never experience downtime. Design the migration plan with specific steps, timings, and rollback criteria.

**Rubric**:

| Dimension | Poor (0 pt) | Acceptable (1 pt) | Strong (2 pts) |
|---|---|---|---|
| Pre-migration preparation | No preparation | Backs up zone file | Backs up zone + reduces TTL + tests secondary transfer |
| Cutover strategy | Change NS at registrar | Gradually shift with overlapping NS | Dual-provider with monitoring and phased shift |
| Email risk | Not mentioned | Mentions MX | Has explicit email validation check |
| Rollback plan | No rollback | "Undo the change" | Specific rollback steps with criteria |
| Monitoring | No monitoring | Pings the domain | DNS-level synthetic checks + resolver-specific probes |

**Model answer**:

Phase 1 — Preparation (Day 1):
1. Export zone file from legacy provider. Validate all 50 records.
2. Import zone file into Route53. Verify all records match: `dig @ns-route53.example.com ANY` vs `dig @ns-legacy.example.com ANY`.
3. Reduce TTL on ALL records to 60 seconds. This is critical — the old TTL may be 86400 seconds. Without this reduction, the cutover takes 24 hours.
4. Allow Route53 to act as a secondary: configure AXFR from legacy to Route53. Monitor zone transfers for 24 hours to ensure they stay in sync.
5. Set up Route53 health checks for the web and MX endpoints.

Phase 2 — Validation (Day 3, after 48 hours of low TTL):
6. Add Route53 nameservers to the domain at the registrar alongside the legacy nameservers. The domain now has 4+ NS records.
7. Wait 48 hours for NS record propagation (NS records may still have 172800s TTL from the TLD).
8. Monitor: resolve `example.com` from 5 different resolvers (Google, Cloudflare, OpenDNS, two ISP resolvers). Verify all return identical records.
9. Verify email: send test emails to and from the domain. Check SPF, DKIM, DMARC pass. Confirm MX records resolve to the same mail server via both provider paths.

Phase 3 — Cutover (Day 5):
10. Remove legacy nameservers from the registrar. The domain now points only to Route53.
11. Monitor: SERVFAIL rate must stay under 0.01%. Query error rate must not increase. Email must continue flowing.
12. Rollback criteria: if SERVFAIL rate exceeds 0.1% or email fails within 2 hours, re-add legacy NS records at the registrar.

Phase 4 — Post-migration (Day 7):
13. Increase TTL back to production values (300-3600 seconds depending on record type).
14. Decommission the legacy provider account.
15. Document the new DNS setup in the operations runbook.

Email-specific protections:
- During Phase 2, both provider NS records serve the same MX records. If one provider fails, email flows through the other.
- Monitor: `dig @8.8.8.8 example.com MX` must return the same MX targets from both providers.
- Test: send email from gmail to an address on the domain and verify delivery within 30 seconds (for Phase 3 cutover validation).

## Tier 3 — Expert

These challenges are 30-minute open-ended design problems. There is no single correct answer. The evaluation is based on the depth of your analysis, how many trade-offs you identify, and whether you anticipate failure modes.

### Challenge 7: Design a DNS System That Survives a TLD Registry Outage

**Prompt**: In January 2021, a .com zone file validation issue caused intermittent failures for millions of domains. The root cause was a corrupted zone file at the registry level. Design a DNS architecture for a top-100 global website (google.com/amazon.com level) that can survive a .com TLD registry outage lasting 6 hours.

During a TLD outage, recursive resolvers cannot refresh their cached .com NS records. This means:
- New domains registered under .com stop resolving
- Domains whose .com NS records have expired also stop resolving
- Everything else continues working until individual domain TTLs expire

Your architecture must ensure that your domain continues resolving even when the .com TLD servers are unreachable.

**Analysis depth indicators**:
- Identifies that the .com NS delegation is cached by recursive resolvers for 172800 seconds (48 hours) — any resolver that resolved within 48 hours will still have the delegation. Only resolvers that start cold during the outage (new instance, cache flush) will fail.
- Proposes a strategy to minimize cold-starts: maintain resolver cache persistence across restarts (Unbound allows dump_cache/load_cache).
- Considers DNS-over-HTTPS (DoH) to an alternate provider as a fallback resolution path that bypasses standard recursive resolution.
- Considers operating a domain on multiple TLDs (example.com and example.net) with a client-side redirect or a shared authoritative infrastructure.
- Observes that the real vulnerability is not the .com TLD itself but the resolvers' inability to refresh cached delegations. The solution is to keep resolver caches warm.

**Scoring guide**:
- Pass (L5 level): Identifies the root cache persistence strategy and proposes at least two mitigation layers.
- Strong pass (L6+): Discusses multi-TLD strategy, DoH fallback, and identifies the specific SOA/NS record TTLs that matter. Mentions that actual outage data from the 2021 .com incident shows most resolvers were unaffected because of the 48-hour TTL.

**Model approach**: The primary mitigation is to ensure your recursive resolver fleet never goes cold. Configure Unbound with `dump_cache` to persist cache across restarts. Supplement with secondary resolution paths: (1) DoH queries to Cloudflare's 1.1.1.1 or Quad9 bypass the standard recursive path and may use alternative upstream sources; (2) direct hints for your authoritative nameserver IPs in your resolver configuration as `stub-zone` — this bypasses the .com delegation entirely. Finally, as an insurance policy, operate the same service on a second TLD (example.net) and configure DNS-based failover. If example.com stops resolving, the client-side SDK retries via example.net.

### Challenge 8: DNS-Free Service Discovery

**Prompt**: DNS was designed in 1983 for a network where domain names changed rarely. Modern microservice architectures change IPs every few minutes. Design an alternative to DNS for service discovery in a Kubernetes cluster running 10,000 services.

Your design must:
- Support sub-second service discovery changes (when a pod restarts, its new IP must be discoverable within 500ms)
- Not rely on TTL-based caching (which introduces staleness)
- Support 50,000 QPS of discovery requests
- Support SRV-record-style queries (return port number with IP)
- Work across multiple Kubernetes clusters in different regions

Identify the trade-offs compared to DNS-based service discovery. When would you still use DNS despite its limitations?

**Analysis depth indicators**:
- Identifies that the core problem is polling-based discovery (resolvers poll on TTL intervals, introducing latency and staleness).
- Proposes a push-based alternative: subscribers register interest in a service name and receive real-time updates when the IP changes (gRPC stream, WebSocket, or custom protocol).
- Discusses consistent hashing or rendezvous hashing as a way to distribute the discovery state without a centralized registry.
- Identifies that etcd is the most common solution in Kubernetes for this (not DNS — DNS is the query interface, etcd is the state backend).
- Mentions that gRPC provides built-in load balancing and service discovery via its xDS protocol (control plane), which is already push-based.
- Notes the trade-off: DNS is simpler and battle-tested. Push-based discovery requires maintaining persistent connections and managing state consistency.

**Scoring guide**:
- Pass (L5 level): Identifies that etcd + watch API replaces polling. Proposes using gRPC xDS for internal discovery.
- Strong pass (L6+): Designs a multi-region control plane with push-based discovery using gRPC streaming. Discusses consistency model (etcd is CP, DNS is AP — for service discovery, CP may be acceptable since the cluster control plane already uses etcd). Identifies the hybrid approach: DNS for external traffic (clients, browsers), push-based for internal traffic (microservice-to-microservice).

**Model approach**: Use etcd as the authoritative state store for service locations. Each pod writes its IP and port to etcd on startup with a lease (time-based expiration). Other pods watch for changes to the key prefix corresponding to the service they depend on. When a pod registers or deregisters, the etcd watch fires in under 10ms, and the watcher updates its local connection pool immediately. This eliminates TTL-based staleness entirely.

For the query interface, expose a gRPC API instead of DNS: `LookupService(serviceName, namespace) returns (repeated Endpoint)`. Each endpoint includes IP, port, region, and health status. The API returns the current state within the latest raft commit — consistent within the etcd cluster.

For cross-region discovery, configure a multi-cluster etcd topology with a federation layer. Regional etcd clusters replicate service state to a global etcd cluster (or use a service mesh like Istio for cross-cluster routing).

When to still use DNS: for internet-facing traffic (clients, browsers) that cannot run a custom discovery client. DNS is the universal interface. The hybrid approach is standard at companies like Google, Netflix, and Uber: DNS for external traffic, custom service discovery for internal microservices.
