## Company Case Studies

### Netflix: DNS-Driven Global Traffic Engineering

**Scale**: 230 million paid subscribers, 3+ billion hours streamed per month, 15% of global downstream traffic.

**Problem**: Netflix serves video from a global CDN of thousands of Open Connect Appliances (OCAs) deployed inside ISP networks. Each ISP has one or more OCAs. When a user presses play, Netflix must route them to the OCA that is nearest, healthiest, and has the requested content pre-cached. The decision must be made in under 100 milliseconds to avoid buffering delays.

**Solution**: Netflix built a DNS-based traffic steering system called the "Traffic Director" that integrates three data sources:

1. **Cloud-based control plane** (AWS Route53): Manages the authoritative DNS zones and routing policies.
2. **OCA health and capacity** (internal monitoring): Each OCA reports its CPU load, disk cache hit rate, upstream bandwidth, and active connection count every 10 seconds.
3. **ISP topology mapping** (internal database): Maps each client's resolver IP (via EDNS0 client subnet) to the nearest OCA within their ISP.

The Traffic Director generates per-resolver DNS responses that change dynamically. A user in Austin, Texas, querying from a Spectrum resolver, gets routed to the Spectrum OCA in Dallas. A user in the same city but using a Comcast resolver gets the Comcast OCA in Houston.

### Cloudflare: DNS as a Security Product

**Scale**: 1.1.1.1 resolver handles over 10 million DNS queries per second at peak. Authoritative DNS serves 4 million domains.

**Problem**: DNS is the most effective attack vector for taking services offline. DNS amplification attacks, NXDOMAIN attacks, and random subdomain attacks can generate 100+ Gbps of traffic targeting the DNS infrastructure. Cloudflare needed a DNS architecture that absorbs attacks at scale while providing low-latency resolution for legitimate traffic.

**Solution**: Cloudflare runs its own DNS server software from scratch (not BIND or Unbound). Key design decisions:

1. **Kernel bypass**: Direct NIC access via XDP (eXpress Data Path) processes DNS packets in the kernel before they reach userspace. This cuts per-packet processing from 1000+ CPU cycles to under 200.

2. **JIT compilation of zone data**: Zone records are compiled into a native x86-64 shared library at deploy time. Query resolution becomes a function call instead of a database lookup. Resolution time: under 10 microseconds.

3. **Anycast at 330+ locations**: Every Cloudflare data center serves every domain. No DNS query travels more than 50ms from any internet-connected client.

4. **DPDK-based packet processing**: Data Plane Development Kit enables 100 Gbps throughput per server.

### Discord: DNS for Real-Time Service Discovery

**Scale**: 150 million monthly active users, 19 million daily voice users in 8 million concurrent voice channels.

**Problem**: Discord's microservice architecture requires low-latency service discovery. When a user joins a voice channel, the voice gateway service must discover the nearest regional voice server and connect within 200ms. Traditional DNS polling (every 30-60 seconds) would introduce unacceptable latency for discovery.

**Solution**: Discord uses a combination of DNS for initial discovery and a direct peer-to-peer connection for real-time communication.

1. **Initial DNS resolution**: Voice gateway DNS returns a list of 3-5 regional voice servers sorted by latency to the user's resolver subnet. TTL: 300 seconds.
2. **Health-based DNS updates**: When a voice server region suffers an outage (happens a few times per year), Discord updates the DNS records to remove the affected region. The 300-second TTL means a 5-minute propagation window.
3. **WebSocket fallback**: The client also maintains a persistent WebSocket to the API gateway. If the voice connection fails, the client requests an updated server assignment through the WebSocket — bypassing DNS entirely.

**Result**: 99.97% voice connection success rate. DNS handles 95% of initial discovery requests. The WebSocket fallback handles the remaining 5% during DNS propagation windows.

### Dropbox: DNS and Infrastructure Migration

**Scale**: 700 million registered users, 500 PB of stored data, migrating from AWS to self-built infrastructure.

**Problem**: Dropbox migrated from AWS to their own data centers between 2016 and 2020. The migration required routing traffic to two completely different infrastructure stacks simultaneously. DNS had to split traffic by percentage (10% prod to new infra, 90% to old infra) and gradually shift over 18 months.

**Solution**: Weighted DNS routing with very short TTLs (60 seconds) during the migration window.

```
Phase 1: 10% → new infrastructure, 90% → AWS
  api.dropbox.com A 10.0.1.10 [weight 10]
  api.dropbox.com A 203.0.113.50 [weight 90]
  TTL: 60 seconds

Phase 6 (18 months later): 100% → new infrastructure
  api.dropbox.com A 10.0.1.10 [weight 100]
  TTL: 300 seconds (back to normal)
```

Each weighted DNS response gave them fine-grained traffic shift without any traffic loss. If a region of the new infra had issues, they reverted the weight in under 60 seconds.

### Google: Internal DNS at Planetary Scale

**Scale**: Exabyte-scale data processing across 20+ global data centers, millions of servers, thousands of internal services.

**Problem**: Google's internal services need DNS-level service discovery across a global network that spans continents. A MapReduce job in Oregon might read data from the European datacenter, and the task scheduler needs to discover the data node's IP via DNS.

**Solution**: Google operates an internal DNS system (separate from public DNS) that uses Borg (their cluster manager) as the source of truth for service locations. Key properties:

- **Short TTLs** (5-30 seconds) for internal services whose IPs change with every job rescheduling
- **Global domain hierarchy**: `service.datacenter.cluster.google.internal`
- **Automatic service registration**: When Borg schedules a task, it automatically creates DNS records for the service name pointing to the task's IP
- **Inter-datacenter resolution**: A service in us-central1 can discover a service in europe-west1 by resolving `service.europe-west1.cluster.google.internal`

## Open Source Implementations

### BIND (Berkeley Internet Name Domain)

The original DNS server software. Still the most widely deployed authoritative nameserver on the internet.

- **Role**: Authoritative and recursive
- **Strengths**: Most feature-complete. Supports every RFC extension. Battle-tested since 1984.
- **Weaknesses**: Performance lags behind modern implementations. Configuration complexity (named.conf is notoriously error-prone). Single-threaded core for many operations.
- **Configuration example**: Zone files in standard RFC format. TSIG for dynamic updates. Views for split-horizon DNS.

### Unbound

A modern, high-performance recursive resolver developed by NLnet Labs.

- **Role**: Recursive resolver (can also serve as authoritative for small deployments)
- **Strengths**: Single-threaded event loop with epoll/kqueue handles 200,000+ QPS. DNSSEC verification built-in. QNAME minimization by default. Aggressive caching with prefetch.
- **Weaknesses**: Limited authoritative features. No built-in zone transfer client (needs separate tooling).
- **Memory**: ~200 MB for typical workload. Scales to millions of cached records.

### CoreDNS

A cloud-native DNS server written in Go, originally developed for Kubernetes.

- **Role**: Service discovery DNS for container environments
- **Strengths**: Plugin architecture (50+ plugins for etcd, Kubernetes, Prometheus, Redis). Hot-reloadable configuration. Designed for Kubernetes pod lifecycle awareness.
- **Weaknesses**: Lower performance than Unbound for recursive resolution (~50,000 QPS). Less mature DNSSEC support.
- **Use case**: Replace kube-dns in Kubernetes clusters. Serve as cluster-internal DNS with etcd backend.

### PowerDNS

A modular DNS server with multiple backends (MySQL, PostgreSQL, SQLite, LDAP, generic API).

- **Role**: Authoritative server with database backend
- **Strengths**: SQL-backed zone storage makes DNS programmable. Pipe backend allows custom logic in any language. DNSSEC signing integrated with database.
- **Weaknesses**: Database dependency adds latency and operational complexity. Less suitable for high-QPS recursive resolution.

## Cloud Provider Implementations

### AWS Route53

- **Product**: Authoritative DNS + health checks + traffic flow + resolver
- **SLA**: 100% uptime (with credit if below)
- **Pricing**: USD 0.50/zone/month + USD 0.40/million queries
- **Key features**: Latency-based routing, geolocation routing, weighted routing, failover routing, health checks, DNSSEC, alias records (CNAME-like at zone apex), Route53 Resolver for VPC DNS.
- **Cost vs operations**: Higher cost than self-managed (USD 400-2000/month at 100M queries) but zero operational overhead. No servers to patch. No anycast to configure.

### Google Cloud DNS

- **Product**: Authoritative DNS + Cloud DNS for VPC
- **SLA**: 100% uptime
- **Pricing**: USD 0.20/zone/month + USD 0.20/million queries (50% cheaper than Route53)
- **Key features**: Managed DNSSEC, split-horizon via VPC peering, REST API for zone management, Cloud Console integration.
- **Differentiator**: Lower cost than Route53. Tighter integration with GCP networking. Terraform module is more mature.

### Azure DNS

- **Product**: Authoritative DNS + Private DNS zones
- **SLA**: 100% uptime
- **Pricing**: USD 0.50/zone/month + USD 0.40/million queries (comparable to Route53)
- **Key features**: Azure Private DNS (resolution within VNet), alias records, DNSSEC, role-based access control.
- **Differentiator**: Tighter integration with Azure Active Directory for access management. Best option if you are all-in on Azure.

## Lessons Learned

**Lesson 1: Short TTLs are a trap — never go under 60 seconds without a specific reason.**

Every production outage reviewed in this chapter was made worse by TTLs that were too short for the query volume. Short TTLs amplify traffic spikes into cascading failures. The minimum safe TTL for any record that generates more than 100 QPS is 60 seconds. For records with 1000+ QPS, 300 seconds is safer.

**Lesson 2: DNS-based failover is measured in minutes, not seconds.**

If you need sub-second failover, DNS is the wrong tool. A load balancer with active health checking can failover in under 1 second. DNS-based failover, even with 60-second TTLs and aggressive health checks, takes 90-180 seconds end-to-end. Design your architecture accordingly: use DNS for geographic routing, not for backend health.

**Lesson 3: Caching is the most powerful lever in DNS — and the most dangerous.**

Caching makes DNS fast (cache hit: under 1ms; cache miss: 50-200ms). It also makes DNS slow to react to change. Every DNS-related production incident involves someone discovering that their cache did not refresh as quickly as they expected. Always assume the cache is working against you.

**Lesson 4: Multi-provider DNS requires active monitoring, not just configuration.**

Companies that run DNS from two providers often discover during an outage that one provider's zone data is stale. The zone transfer failed silently days ago. Active monitoring of zone serial numbers across providers — alerting if they diverge by more than 1 — is essential.

**Lesson 5: DNSSEC is essential for security but expensive in operations.**

DNSSEC prevents the cache poisoning attacks described in the trade-offs chapter. But key rotation is a manual process at most companies, and a missed rotation causes your domain to fail DNSSEC validation — making your domain unresolvable for validation-enabled resolvers. Automate key rotation before enabling DNSSEC.
