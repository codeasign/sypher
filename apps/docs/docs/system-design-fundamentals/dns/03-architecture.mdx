## Naive Implementation

The simplest possible DNS setup has three components: a domain registered with a registrar, an authoritative nameserver (often the registrar's own), and a handful of A records.

```
Phase 1 — Single Server, Single Record:

  Registrar: Namecheap
  Nameserver: dns1.registrar.com
  Records:
    example.com.  A  203.0.113.10
    www           A  203.0.113.10
    api           A  203.0.113.10
```

This works for a small blog or landing page. A single web server handles all traffic. The registrar's nameserver handles DNS queries as a free add-on.

### What Breaks at 1,000 Users

At 1,000 daily visitors, nothing visible breaks. The registrar's nameserver easily handles the 50-100 queries per minute this generates. The single web server handles the traffic. You might not even notice DNS is part of the stack.

### What Breaks at 100,000 Users

At 100,000 daily visitors (roughly 30-50 queries per second on DNS), three cracks appear:

1. **Registrar DNS reliability**: Shared nameservers from the registrar see request-rate limits. During traffic spikes, you may see intermittent SERVFAIL responses. The registrar's DNS console may take 5-15 minutes to propagate record changes.

2. **No health-based routing**: When the single web server at `203.0.113.10` goes down, DNS still returns that IP. Users see connection timeouts. DNS has no way to detect the server is down.

3. **Single point of failure**: If the registrar's nameserver suffers an outage, your entire domain goes offline. There is no redundancy built into the free-tier service.

### What Breaks at 10 Million Users

At 10 million daily visitors (thousands of queries per second), the naive setup catastrophically fails:

1. **Query overload**: The registrar's shared nameserver rate-limits your domain. Resolvers get SERVFAIL or truncated responses. Users see DNS resolution failures.

2. **No geographic distribution**: All traffic routes to a single region. A user in Sydney experiences 200ms DNS resolution latency because the registrar's nameserver is in Virginia.

3. **No traffic management**: You cannot route users to the nearest region, split traffic for canary deployments, or failover between regions. Every user gets the same IP regardless of their location or your server health.

4. **No operational tooling**: There is no API for automated DNS updates, no audit log of changes, no DNSSEC support, no metrics on query volume.

## Production-Grade Architecture

A production DNS system at scale requires four layers working together: a managed DNS provider (for authoritative serving), a recursive resolver fleet (for low-latency resolution), health monitoring and traffic management, and automation tooling.

### Key Architectural Decisions

**Decision 1: Anycast for all layers**. Every authoritative nameserver listens on the same IP address at multiple data centers worldwide. BGP anycast routing ensures that each client reaches the nearest operational server. A client in Tokyo connects to a nameserver in Tokyo, not Virginia. Anycast also provides instant failover: if the Tokyo server goes down, BGP withdraws the route and traffic flows to the next-closest server (Seoul or Singapore).

**Decision 2: Separate authoritative from recursive**. The authoritative servers that hold your zone records must never be the same infrastructure that recursive resolvers use. If your recursive resolver fleet is misconfigured or attacked, the authoritative data remains safe. This separation also lets you use managed DNS providers (Route53, CloudDNS, NS1) for authoritative serving while running your own recursive resolvers in the VPC.

**Decision 3: API-driven record management**. No operator should SSH into a server to edit a zone file. Every record change goes through a version-controlled API backed by infrastructure-as-code (Terraform, Pulumi). Changes require a pull request, review, and deployment pipeline approval.

### Pattern 2: DNS + CDN Integration

Content Delivery Networks (CDNs) use DNS as the first routing decision point. The CDN's authoritative DNS uses EDNS0 client subnet information to return the optimal edge server IP.

### Pattern 3: DNS + Service Discovery Integration

Internal microservice architectures use DNS for service discovery. Services register themselves with a service registry (Consul, etcd, Eureka). A DNS interface translates registry data into DNS records.

```
Workflow:
  1. Service "user-service" starts, registers IP 10.0.1.50:8080
  2. Service registry stores: user-service.service.consul A 10.0.1.50
  3. Service "api-gateway" queries: user-service.service.consul A
  4. DNS returns: 10.0.1.50
  5. api-gateway connects to 10.0.1.50:8080

Resolver used: 127.0.0.1:8600 (Consul DNS interface)
Cache TTL: 0 seconds (always pull fresh from registry)
```

### Pattern 4: DNS + Rate Limiting Gateway

Place a rate-limiting gateway between recursive resolvers and authoritative nameservers to protect against DDoS and misconfigured clients.

```
Integration point: Query classification and throttling
  1. Classify: known resolver (allow), unknown resolver (throttle), known abuser (block)
  2. Count: per-resolver QPS, per-zone QPS, per-client-subnet QPS
  3. Enforce: token bucket at gateway, not at nameserver process
  4. Return: SERVFAIL with 2-second TTL for throttled queries (client retries later)
```

### Replication Strategy

Zone data replicates from the primary authoritative nameserver to secondary servers via zone transfers:

- **Full zone transfer (AXFR)**: Transmits the entire zone file. Used when a secondary server is first configured or after a prolonged outage where IXFR history was lost. Size: comparable to zone file (typically 1-100 KB for most domains).

- **Incremental zone transfer (IXFR)**: Transmits only the changes since the last version. Uses the SOA serial number to identify the starting point. Size: typically 10-1000 bytes per update.

- **DNS NOTIFY**: The primary sends a NOTIFY message to all configured secondaries when the zone changes, triggering an immediate IXFR rather than waiting for the refresh interval.

### Cross-Region Latency Budget

| Path | Distance | RTT | DNS queries/day (10M DAU) |

| Recursive resolver to anycast NS | Regional | 1-5ms | 500M (cached miss rate 20%) |
| NS to primary for zone transfer | Inter-region | 50-200ms | 144 (IXFR every 10 min) |

## Capacity Planning

### Scaling Math for 10 Million Daily Active Users

**Assumptions:**
- Each DAU generates 50 page views
- Each page view triggers 4 DNS queries (page + API + CDN + analytics)
- Recursive resolver cache hit rate: 85%
- Peak-to-average ratio: 3x

**Request volume:**

```
Daily queries = 10M DAU * 50 pages * 4 queries = 2 billion queries/day
Peak queries per second (QPS) = 2B / 86400 * 3 = 69,444 QPS
Auth NS queries (15% uncached) = 69,444 * 0.15 = 10,417 QPS
```

**Authoritative nameserver capacity:**

```
Per-node capacity: 50,000 QPS (Unbound on c5.4xlarge)
Nodes needed: 10,417 / 50,000 = 0.21 → 2 nodes (minimum for redundancy)
Anycast sites: 3 regions * 2 nodes = 6 total nodes
```

**Recursive resolver fleet capacity:**

```
Per-node capacity: 200,000 QPS (cached, Unbound on c5.4xlarge)
Nodes needed: 69,444 / 200,000 = 0.35 → 2 nodes per region
But for fault tolerance and headroom: 5 nodes per region
Total: 15 nodes across 3 regions
```

**Memory for recursive resolvers:**

```
Unique domains per day: ~500,000
Records per domain: 3.5 (A, AAAA, MX, TXT, etc. average)
Cache entries: 500,000 * 3.5 = 1.75 million
Memory per entry: 200 bytes (includes hash table overhead)
Total cache memory: 1.75M * 200 = 350 MB per resolver
Plus working buffer: 512 MB
Total: ~1 GB per resolver node
```

**Bandwidth:**

```
Average response size: 150 bytes
Peak bandwidth = 69,444 QPS * 150 bytes * 8 bits = 83 Mbps
This is negligible. DNS is query-heavy but bandwidth-light.
```

## Deployment and Operations

### Zero-Downtime DNS Migration

Migrating DNS from one provider to another without downtime requires a carefully sequenced process:

```
Phase 1 — Setup (Day 1)
  1. Configure new DNS provider as secondary for all zones
  2. Allow AXFR from new provider's IPs on current primary
  3. Verify zone transfer completes: new provider returns identical data
  4. Reduce TTL to 60 seconds on all records

Phase 2 — Cutover (Day 3, after TTL propagation)
  5. Add new provider's nameservers to domain at registrar
  6. Wait 24-48 hours for NS record propagation
  7. Monitor query volume: split 50/50 between old and new NS
  8. Verify all resolvers can reach both nameserver sets

Phase 3 — Decommission (Day 5+)
  9. Remove old provider's nameservers from registrar
  10. Increase TTL back to operational values
  11. Deactivate old provider account
```

### Debugging DNS Issues

**Troubleshooting toolkit:**

```bash
# Check resolution path with explicit nameserver
dig @8.8.8.8 example.com A +trace

# Query specific record type with timing
dig example.com A +stats

# Check DNSSEC chain
delv example.com A

# Verify zone transfer works
dig @ns1.example.com example.com AXFR

# Query from specific resolver IP (emulate client location)
dig @resolver-ip -b client-ip example.com A

# Check caching behavior
dig example.com +norecurse

# Monitor DNS traffic in real-time
tcpdump -i eth0 port 53 -n -c 1000

# Bulk resolve from file of domains (performance testing)
while read domain; do
  dig +short $domain A
done < domains.txt | sort | uniq -c | sort -rn
```

### Upgrade Patterns

**Rolling upgrade of recursive resolver fleet:**

```
1. Mark node as draining (stop accepting new queries)
2. Wait for in-flight queries to complete (5 seconds)
3. Stop resolver process
4. Deploy new binary
5. Start resolver process (warm cache lost; expect 5-minute cache warmup)
6. Re-enable node in load balancer
7. Repeat for next node
8. Monitor cache hit rate during warmup: should return to 85% within 5 minutes
```

**Cache warming strategy:**

Before a rolling upgrade, warm the new resolver's cache by feeding it a replay of recent query traffic:

```bash
# Capture recent query log on old node
tcpdump -i eth0 port 53 -n -c 100000 -w query_dump.pcap

# Replay queries to new node (before production traffic arrives)
tcpreplay --mbps=1000 --loop=3 query_dump.pcap --intf1=eth1
```

This pre-populates the cache so the new node starts at 80% hit rate instead of 0%.
