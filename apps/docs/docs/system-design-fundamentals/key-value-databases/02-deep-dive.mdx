# Key-Value Databases — Under the Hood

## Internal Architecture

A key-value store looks like a black box from the outside: put key, get value, delete key. Inside, there are several subsystems working together to handle memory management, persistence, replication, eviction, and network I/O. The architecture differs significantly between in-memory stores (Redis, Memcached) and persistent stores (DynamoDB, RocksDB). Here is what lives inside the box.

## Data Structures and Algorithms

The hash table is the heart of nearly every key-value store, but the supporting data structures determine performance characteristics under different workloads.

### Hash Tables — The Primary Structure

A hash table maps keys to array indices using a hash function. Redis uses a dictionary (dict) with incremental rehashing. When the table grows beyond its load factor (default 1:1 ratio of entries to buckets), Redis does not stop the world to resize — it rehashes incrementally, moving a few buckets per millisecond of server time.

<AsciiDiagram id="key-value-databases/deepdive-hashtable" title="Hash Table Internals with Chaining" caption="Keys are hashed to buckets; collisions are resolved with a linked list of entries in each bucket."

### Skip Lists — Ordered Data

Redis uses skip lists (not balanced trees) for sorted sets. A skip list is a layered linked list that provides O(log n) search, insert, and delete with simpler implementation than a red-black tree.

### LSM-Trees — Persistent Storage

Persistent key-value stores like RocksDB and LevelDB use Log-Structured Merge Trees (LSM-trees). Writes go to an in-memory memtable first (fast), then flush to immutable SSTables on disk. Reads check the memtable, then search SSTables from newest to oldest. Bloom filters eliminate unnecessary SSTable reads.

## Consistency and State Handling

Distributed key-value stores face three hard problems: crashes, partitions, and concurrent writes. Here is how each is handled.

### Crash Recovery with AOF

<AsciiDiagram id="key-value-databases/deepdive-crash" title="Crash Recovery with AOF Log" caption="After a crash, Redis replays the AOF log to rebuild the in-memory state."
  content={`┌──────────────────────────────────────────────────────────┐
│  Timeline:                                                │
│                                                          │
│  t0: Server running, 50K keys in memory                  │
│  t1: SET "a" 1 ──> hash table + AOF buffer              │
│  t2: SET "b" 2 ──> hash table + AOF buffer              │
│  t3: SET "c" 3 ──> hash table + AOF buffer              │
│  t4: POWER FAILURE ──> Server dies                       │
│                                                          │
│  t5: Server restarts                                     │
│      Step 1: Detect AOF file exists                      │
│      Step 2: Read AOF: SET a 1 / SET b 2 / SET c 3     │

│                                                          │
│  What if fsync didn't happen before crash?              │

**Legend:** AOF persistence offers a configurable trade-off between durability and write throughput.`}

Without transactions, two clients writing to the same key can race. The last write wins by default, which may not be correct for your application.

### Memory Overhead

Key-value stores have significant memory overhead beyond the raw data size. Redis stores each key with its type, encoding, LRU timestamp, reference count, and pointer metadata. A 20-byte key with a 100-byte value consumes approximately 200 bytes of RSS — roughly 60% overhead.
