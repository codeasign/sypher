# Key-Value Databases in Production

## Company Case Studies

Real companies use key-value stores in ways that reveal the technology's strengths, limits, and surprising failure modes. Each case study below includes the architecture, the problem, and the solution that shipped to production.

### Twitter — The Cache Stack That Survives 500M Tweets Per Day

Twitter runs one of the largest Memcached deployments in the world. In 2021, they operated over 100 clusters with thousands of Memcached nodes, serving over 12 billion cache requests per day. Their architecture reveals how a key-value cache operates at planetary scale.

**What went wrong at Twitter:** In 2018, a bad deployment of a new caching library caused a cascading cache failure. The client library's connection pooling had a bug where it would not properly close connections. Over 8 hours, half the Memcached connections were "zombie" connections that consumed resources but never sent requests. The cluster hit its connection limit, started rejecting requests, and the fallback to MySQL caused a multi-hour partial outage.

**Lesson:** Client library configuration matters as much as server configuration. Connection pooling, retry logic, and timeouts must be tested under failure conditions.

**What went wrong at Discord:** In 2020, Discord suffered a Redis incident where a single large guild's state data (a community server with 500,000 members) caused a Redis cluster shard to run out of memory. The shard's eviction policy was incorrectly set to `noeviction`, which caused all writes to that shard to fail. Since Discord used Redis for session state, 50,000 users were disconnected and could not reconnect until the shard was resized.

**Lesson:** Set maxmemory-policy to `allkeys-lru` for any cache-like workload. Monitor memory usage by shard, not just cluster-wide. A single "elephant key" can fill a shard.

### Amazon DynamoDB — The Database That Scales Itself

DynamoDB is Amazon's fully managed key-value and document database. It powers Amazon.com's shopping cart, Alexa's session management, and thousands of AWS customer workloads. DynamoDB's design was published in the famous 2007 "Dynamo" paper, which influenced an entire generation of distributed databases.

**What went wrong at DynamoDB:** In 2021, Amazon experienced a DynamoDB throttling incident that affected many AWS customers. A single large customer's traffic pattern changed dramatically — from mostly reads to mostly writes — which exceeded the write capacity on several partitions. DynamoDB's auto-scaling could not react fast enough, and write requests were throttled for 45 minutes. The customer's application did not handle throttling with exponential backoff, so retries compounded the problem.

**Lesson:** Monitor your request pattern mix, not just total volume. Configure your SDK for exponential backoff with jitter. Do not assume auto-scaling will react within seconds — it takes minutes.

**What went wrong at Cloudflare:** An edge data center's Memcached instance ran out of memory because a user uploaded a 100 MB JSON blob as a KV value. The single large value caused eviction of thousands of smaller, frequently accessed keys. **Lesson:** Enforce value size limits at the application level. A single large value can evict your entire hot set.

## Open Source Implementations

**Redis** (BSD license) is the most popular open-source key-value store. It supports strings, hashes, lists, sets, sorted sets, bitmaps, HyperLogLog, streams, and geospatial indexes. Redis Cluster provides automatic sharding and replication. Redis Sentinel provides high availability. Redis is written in C and single-threaded for data operations, with background threads for I/O and persistence.

**Memcached** (BSD license) is the simplest and fastest key-value store. It supports only strings, uses LRU eviction, has no persistence, and no replication. Its simplicity makes it extremely fast (30-60 microseconds per operation) and operationally simple. It uses a multi-threaded architecture with a hash table per thread.

**RocksDB** (GPL v2) is an embedded persistent key-value store from Facebook. It uses LSM-trees for efficient writes on SSD storage. RocksDB is not a server — it is a library that applications embed. It powers many higher-level systems including MySQL (via MyRocks), Kafka Streams, and Apache Flink.

**LevelDB** (BSD license) is Google's original LSM-tree implementation. It is simpler and slower than RocksDB but serves as the reference implementation for the LSM-tree design.

**FoundationDB** (Apache 2.0) from Apple is a distributed, ordered key-value store with ACID transactions across multiple keys. It uses a layered architecture: the core is a pure key-value store, and higher-level data models (document, relational) are built as layers on top.

**TiKV** (Apache 2.0) is a distributed, transactional key-value store from PingCAP. It uses the Raft consensus protocol for replication and the Percolator model for distributed transactions. TiKV is the storage layer for TiDB, a distributed SQL database.

## Cloud Provider Implementations

**AWS:** Amazon ElastiCache for Redis (fully managed Redis/Memcached), Amazon DynamoDB (managed key-value and document), Amazon MemoryDB for Redis (Redis-compatible with durable write-ahead logging across multiple AZs).

**GCP:** Google Cloud Memorystore for Redis (managed Redis, up to 300 GB per instance), Google Cloud Firestore (document/key-value store with real-time synchronization).

**Azure:** Azure Cache for Redis (managed Redis, up to 1.2 TB in Premium tier), Azure Cosmos DB (global-scale multi-model database with key-value API).

## Lessons Learned

### Pattern 1: Cache Everything, But Not Forever

Every case study shows that caching improves performance dramatically, but the TTL must match your data's freshness requirements. Twitter caches timelines for 5 seconds. Discord caches guild state for 60 seconds. Cloudflare caches IP reputation for 300 seconds. None cache for 24 hours. Short TTLs prevent serving stale data but require a hot working set that fits in memory.

### Pattern 2: Right-Size Your Value

Small values (under 10 KB) deliver the best performance. Large values (over 1 MB) cause problems everywhere: network buffering, memory fragmentation, slow serialization, and event-loop blocking. Every team that broke this rule regretted it. The fix is almost always the same: store large objects in blob storage and put only the reference in the key-value store.

### Pattern 3: Monitor Eviction, Not Just Memory

Memory utilization alone tells you nothing about whether your cache is effective. Eviction rate and cache hit rate are the real signals. A 90% memory-utilized Redis instance with a 99% hit rate is fine. A 50% memory-utilized instance with a 70% hit rate means your working set does not fit. The eviction rate should be near-zero during steady-state — a sudden spike means your working set grew or a single large key is displacing the hot set.

### Pattern 4: Plan for the Single Large Key

Twitter's 2018 outage, Discord's 2020 failure, and Cloudflare's memory crash all trace back to the same root cause: one unexpectedly large or popular key caused disproportionate impact. Plan for this. Implement it at the application layer, not the database layer. Detect it. Protect against it with per-key rate limiting, value size enforcement, and per-shard monitoring.

### Pattern 5: Prefer Managed Services

Every company that runs self-managed Redis at scale (Twitter, Discord, Pinterest, GitHub) has dedicated SRE teams for it. The operational burden is real: kernel tuning for THP (transparent huge pages), swap configuration, network latency tuning, backup scheduling, version upgrades, and failure recovery. Unless your organization has Redis experts on staff, use a managed service (ElastiCache, Memorystore, Azure Cache). The cost premium is far less than the salary for a dedicated Redis SRE.
