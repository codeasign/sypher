# Understanding Key-Value Databases

## The Startup That Nearly Died From Speed

In 2016, a five-person social-gaming startup called NudgeLaunch hit 50,000 daily active users. Their app let people play quick trivia rounds against friends, storing scores, session state, and friend lists in a PostgreSQL database on a single server. Everything worked fine at 1,000 users. By 10,000 users, queries started taking 200ms. By 50,000 users, the main scoreboard query took 3.8 seconds. Every page load triggered four or five JOIN queries across three tables. The database server's CPU sat at 94% utilization with 400 connections open. Users saw spinning spinners for 10 seconds and then closed the tab.

The founders spent two weeks adding read replicas, connection pooling with PgBouncer, and query optimization. Each fix bought them another week of runway. Then they hit 80,000 users and the primary database crashed during a weekend tournament. The CTO, at a wedding, had to SSH in from his phone to restart the server. They were losing $8,000 per day in ads because the app was down.

On Monday morning, they sat down and asked a brutal question: what if they stopped storing everything relationally? The user session data, the high-score cache, the active-tournament state, the friend-presence indicators — none of these needed JOINs. None of these needed transactions. They all needed one thing: look up a value by a key, instantly. Within 48 hours they added Redis as a cache layer in front of PostgreSQL. Scoreboard queries dropped from 3.8 seconds to 4 milliseconds. Database CPU fell to 12%. The app stopped crashing. That week, their retention rate climbed from 38% to 71% because the app no longer felt broken.

This is the moment every engineer encounters key-value databases. You watch a system that was drowning in relational complexity transform into something fast, simple, and reliable. The key-value store is not a replacement for SQL databases. It is a different tool entirely — one that trades query flexibility for raw speed and horizontal scalability.

## The Insight

Think of a key-value database like a coat-check counter at a busy theater. When you arrive, you hand your coat to the attendant, who gives you a numbered ticket. You do not care which rack your coat hangs on, which shelf, or whether it sits next to someone else's coat. You care about one thing: when you hand back that ticket, you get your coat back immediately. The attendant does not search every rack — they look at your ticket number, walk directly to the right hook, and hand you your coat.

Relational databases are like a vast library with a card catalog. You can ask "find every book published after 2018 by a French author that has been checked out more than 20 times." That is incredibly powerful, but it requires the librarian to consult multiple indexes, cross-reference tables, and compute the answer. Key-value databases cannot answer that question at all. They answer exactly one question: "give me the value for this key." And they answer it in microseconds by going to exactly one place in memory or on disk.

This simplicity is the entire point. By removing everything else — schemas, indexes, relationships, query planners, joins — the system can focus on one operation: get key, return value. The result is predictable, astonishingly fast, and horizontally scalable to hundreds of terabytes across thousands of nodes.

## Core Vocabulary

**Key**: A unique identifier that maps to exactly one value. Keys are typically strings (like "user:42:session" or "tournament:live-scores"). A key is the only way to retrieve data — there is no search-by-value.

**Value**: An opaque blob of data stored under a key. Values can be strings, integers, JSON documents, binary data, or structured types depending on the implementation. The database does not interpret the value — it stores and returns it as-is.

**Key-Value Store**: A database designed around the key-value model, exposing simple Get(key), Put(key, value), and Delete(key) operations. Examples include Redis, Memcached, DynamoDB, Riak, and FoundationDB.

**Hash Table**: The primary data structure powering most key-value stores. A hash function transforms a key into an array index, giving O(1) average lookup time. Collisions are handled via chaining or open addressing.

**In-Memory Store**: A key-value database that stores all data in RAM for maximum speed. Redis and Memcached are in-memory stores. Reads take under 100 microseconds. The trade-off is data is lost on power failure unless persistence is configured.

**Persistent Store**: A key-value database that writes to disk. DynamoDB, LevelDB, and RocksDB are persistent. They survive restarts but trade some latency for durability.

**Time-To-Live (TTL)**: An expiration time set on a key. When the TTL elapses, the key is automatically deleted. This is essential for cache use cases so stale data does not accumulate indefinitely.

**Eviction Policy**: A strategy for removing data when the store reaches its memory limit. Common policies include Least Recently Used (LRU), Least Frequently Used (LFU), First-In-First-Out (FIFO), and random eviction.

**Consistent Hashing**: A distributed hashing scheme that maps keys to nodes in a way that minimizes remapping when nodes are added or removed. When a node fails, only the keys assigned to that node are reassigned, not all keys in the cluster.

**Sharding**: Splitting data across multiple nodes based on key ranges or hash buckets. Each node is responsible for a subset of the key space. This is how key-value stores scale beyond a single machine's memory.

**Replication**: Copying data across multiple nodes for durability and read capacity. Leader-follower replication is common: the leader handles writes and propagates changes to followers.

**Ordered Key-Value Store**: A variant that maintains keys in sorted order, enabling range scans. RocksDB and FoundationDB are ordered stores. This adds complexity but enables prefix scans and ordered iteration.

**Value Size Limit**: The maximum size of a single value. Memcached caps at 1MB, Redis at 512MB, DynamoDB at 400KB. Exceeding the limit requires splitting data across multiple keys.

## Mental Model Check

**Question 1:** Why can a key-value database serve a lookup in under 100 microseconds when a relational database often takes 1-10 milliseconds?

**Answer:** A key-value database uses a hash table for O(1) lookups — it computes the hash of the key and goes directly to that position in memory. A relational database must parse the SQL, consult the query planner, navigate a B-tree index (O(log n)), and often touch multiple pages. The hash table skips all of that complexity. The answer is one hash computation and one memory dereference.

**Question 2:** If a key-value store is so fast, why not use it for everything?

**Answer:** Because key-value stores cannot answer questions about the data itself. You cannot ask "find me all users who signed up last week" or "compute the average order value by region." The only way to retrieve data is by knowing its key. If you need ad-hoc queries, joins, aggregations, or referential integrity, you need a relational database. Key-value stores excel at lookups by known key — nothing more.

**Question 3:** What happens when the hash table grows larger than available RAM?

**Answer:** In a pure in-memory store like Redis, that cannot happen — the system returns errors when memory is full and eviction is disabled. In persistent stores like DynamoDB or RocksDB, data lives on disk and the hash table (or an LSM-tree index) is paged. An in-memory cache like Memcached will evict older keys to make room. For very large datasets, you must either shard across more nodes or switch to a persistent store.

**Question 4:** Your cache hit rate dropped from 95% to 60% after adding a new node. Why?

**Answer:** You likely used a simple mod-N sharding strategy. When you added the fourth node, the modulus changed from mod-3 to mod-4, so every key remapped to a different node. All cached data became invalid. Consistent hashing solves this: adding a node only remaps the fraction of keys that fall within the new node's hash range. This is why consistent hashing is standard in production key-value clusters.

**Question 5:** You set a TTL of 86400 seconds on a cache entry, but the key was evicted after 2 hours. What happened?

**Answer:** The cache hit its memory limit. TTL and eviction are separate mechanisms. TTL removes keys after a time duration. Eviction removes keys when the store is full, regardless of TTL, based on the eviction policy (typically LRU). If your cache is too small for your working set, keys will be evicted long before their TTL expires. Solutions: increase memory allocation, reduce the working set, or switch to a larger tier.
