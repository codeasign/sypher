# Key-Value Databases — Trade-offs & Failure Modes

## The Core Tension

The one fundamental trade-off in key-value databases is **simplicity versus capability**. By stripping away everything except key-based lookups, you gain speed and predictability but lose the ability to ask questions about your data. Every decision in key-value design flows from this tension.

A relational database with 20 tables, 12 indexes, and a query planner can answer 10,000 different questions about your data. A key-value store can answer exactly one type of question: "what is the value for this key?" In exchange, it answers that one question in microseconds instead of milliseconds, scales horizontally with near-linear throughput, and has a operational complexity that is a fraction of a relational system.

The practical implication: you must know the keys to access the data. There is no search. There is no join. There is no aggregation. If your application needs to find data without knowing the key, you need a secondary index — which is no longer a pure key-value system. Every layer you add on top of a key-value store (secondary indexes, materialized views, event streams) is a capitulation to this core tension.

## CAP Theorem Positioning

Every distributed key-value store must choose two out of three CAP guarantees: Consistency, Availability, and Partition Tolerance.

## Failure Modes

Key-value stores fail in specific, predictable ways. Here are the five most common failure scenarios.

## Anti-Patterns Hall of Shame

These are real anti-patterns that teams have shipped to production.

**Anti-Pattern 1: Using Redis as the Primary Database for Financial Data.** Redis persistence modes (RDB snapshots, AOF) are not designed for ACID guarantees. RDB snapshots can lose up to the last save interval of data. AOF with "everysec" fsync loses up to one second of writes. If your server crashes during a write, partially written AOF records can corrupt the database. Redis replication is asynchronous — a leader crash before replication means the write is lost forever. If you need a primary database with durability guarantees, use DynamoDB, FoundationDB, or a relational database.

**Anti-Pattern 2: Storing Large Values (over 10 MB).** Key-value stores are optimized for small, fast access. Storing a 50 MB video file as a Redis value blocks the event loop for 5+ milliseconds during read and write. Every other request queued during that time is delayed. The 512 MB value size limit in Redis is a trap — hitting it does not crash the server but it devastates latency for all other clients. Solution: store large objects in blob storage (S3, GCS) and put only the URL in the key-value store.

**Anti-Pattern 3: No Key Naming Convention.** Production key-value stores accumulate thousands of keys. Without a naming convention, you cannot reason about what is stored, what can be deleted, or which service owns each key. A common pattern is `{service}:{entity}:{id}:{field}` — for example, `user:42:profile:name`. This has immediate benefits: you can scan keys by prefix, you know which service created each key, and you can set up namespace-level TTLs and eviction policies.

**Anti-Pattern 4: Ignoring Eviction Policy Configuration.** The default eviction policy in Redis is `noeviction`. When memory is full, every write returns an error. Teams that do not change this default discover it during their first traffic spike. The cache becomes a write-only data graveyard. The correct default for a cache workload is `allkeys-lru`, which evicts the least recently used keys when memory is full. For a persistent workload, `volatile-lru` evicts only keys with TTL set.

**Anti-Pattern 5: Using the Same Cluster for Cache and Persistent Data.** Cache data and persistent data have opposite requirements. Cache data should use LRU eviction and tolerate data loss. Persistent data must survive restarts and must never be evicted. If you put both in the same Redis instance, you cannot configure eviction correctly — either your persistent data gets evicted or your cache data never gets evicted and fills memory. Use separate instances or Redis databases (logical databases 0-15) with different `maxmemory-policy` settings.

**Anti-Pattern 6: Over-relying on KEYS Command in Production.** The `KEYS *` command in Redis scans the entire keyspace, blocking the event loop for seconds or minutes on a large dataset. Every request queued during that scan is delayed. Use `SCAN` with a cursor instead, which returns results incrementally without blocking.

## When NOT to Use This

Key-value databases are the wrong choice for these scenarios:

**Complex queries and reporting.** If you need to answer questions like "what was the average order value last month" or "find all customers in California who spent over $500," you need a relational database or a data warehouse. Key-value stores cannot aggregate, filter by value, or join across keys.

**ACID transactions across multiple entities.** Key-value stores offer at best single-key atomicity. If you need to transfer money from account A to account B atomically, a key-value store cannot guarantee that both operations succeed or fail together without application-level coordination (which is error-prone).

**Schema enforcement.** Key-value stores do not validate the structure of values. A misbehaving application can store malformed JSON, wrong data types, or corrupt data, and the store will happily accept it. If you need schema enforcement, use a relational database or a document store with schema validation.

**Data with inherent relationships.** If your data model has complex relationships (user has orders, orders have items, items have reviews), storing this in a key-value store means you must manually manage the relationships in your application code. Every "find orders for this user" query becomes an N+1 nightmare. Use a relational database for relational data.

**Workloads requiring range scans over many keys.** Ordered key-value stores (RocksDB, FoundationDB) support range scans, but they are slower and more complex than point lookups. If the majority of your queries are range scans (pagination, analytics, time-series queries), consider a time-series database or a relational database with appropriate indexes.
