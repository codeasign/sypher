# Key-Value Databases — Challenges

## Tier 1 — Foundation (5 minutes each)

### Challenge 1: The Read-Through Cache

**Scenario:** Your application reads a configuration object from a key-value store. The object is 10 KB and is read 5,000 times per second. The value changes once per hour via a manual update. The current implementation reads from Redis on every request.

**Task:** Design a better caching strategy. Assume the application runs on 20 servers.

**Things to consider:** How many Redis reads per second can you eliminate? What happens to correctness when the value changes? What is the worst-case latency for seeing the updated value?

**Model answer:** Add an in-process cache (local to each application server) with a TTL of 60 seconds. Each server stores the configuration in a local variable, refreshing it from Redis every 60 seconds or on demand. This reduces Redis reads from 100,000 per second (5,000 * 20 servers) to 20 per second (one refresh every 60 seconds per server). The worst-case latency for an update is 60 seconds (until the next TTL refresh). To reduce this, expose a webhook or admin endpoint that invalidates the local cache on demand, reducing update latency to under 1 second.

**Why this matters:** This is the most common optimization in production systems. A local cache is faster than any network call, and most configuration data changes infrequently enough that a short TTL is acceptable.

### Challenge 2: The Broken Leaderboard

**Scenario:** You are building a real-time game leaderboard. Players earn points and the leaderboard shows the top 100 players by score. Your teammate implemented it with a SQL query: `SELECT * FROM players ORDER BY score DESC LIMIT 100`. At 1,000 concurrent players, this query takes 3 seconds. At 10,000 players, it takes 45 seconds.

**Task:** Redesign using a key-value store. Show the data structure, write path, and read path. Estimate the latency at 10 million players.

**Model answer:** Use a Redis sorted set. The key is `leaderboard:{game_id}`. On each score update, call `ZADD leaderboard:game1 {score} {player_id}`. This is O(log n) per insert. To read the top 100, call `ZREVRANGE leaderboard:game1 0 99 WITHSCORES`. This is O(log n + m) where m is 100. At 10 million players, each ZADD takes approximately 60 microseconds. Each ZREVRANGE takes approximately 20 microseconds. The sorted set handles 50,000 writes per second on a single Redis instance. If you need more write throughput, shard by game ID (each game gets its own sorted set).

**Why this matters:** This is a textbook example of picking the right data structure. A sorted set is purpose-built for leaderboards. The SQL query is O(n log n) scanned over the entire table. The Redis approach is O(log n) per operation.

### Challenge 3: The Expiring Session

**Scenario:** A web application stores user sessions in Redis as JSON strings. Each session key is `session:{session_id}` with a value containing user ID, roles, and preferences. The session should expire 24 hours after the last activity. Users complain they are being logged out after 2 hours.

**Task:** Diagnose and fix the problem.

**Model answer:** The diagnosis is straightforward: you are setting the TTL only when the session is created, not refreshing it on each request. After 24 hours from creation, the session expires regardless of user activity. The fix is to call `EXPIRE session:{session_id} 86400` on every request that uses the session, resetting the 24-hour countdown. Alternatively, use `SET session:{session_id} {value} EX 86400` on every update. Common mistake: teams assume the TTL is refreshed automatically on read access, but that is not how Redis works. TTL is set once and counts down. You must explicitly refresh it.

**Why this matters:** Session expiration is one of the most common production bugs. Users attribute session timeouts to "the app is broken" and stop using the product.

## Tier 2 — Design (15 minutes each)

### Challenge 4: The Job Queue

**Scenario:** You need a lightweight job queue for processing background tasks: sending emails, resizing images, generating thumbnails. You process approximately 100,000 jobs per hour. Each job takes 2-10 seconds to process. Jobs must be processed at least once (at-least-once delivery). You have 5 worker processes consuming from the queue.

**Task:** Design the queue using a key-value store. Include the data model, the push operation, the pop operation, the failure handling, and the dead-letter queue.

**Model answer:** Use Redis lists with BRPOPLPUSH (or LMOVE in Redis 6.2+). The active queue is a list: `queue:pending`. The in-flight queue is a list: `queue:processing`. Workers call `BRPOPLPUSH queue:pending queue:processing 30` to atomically pop a job and push it to the processing queue. After completing a job, the worker calls `LREM queue:processing 1 {job_id}` to remove it. If a worker crashes, the job stays in `queue:processing`. A separate watchdog process scans `queue:processing` for jobs older than 5 minutes and pushes them back to `queue:pending`. After 3 reprocessings, move the job to `queue:dead-letter` for manual inspection.

Data model: each job is a JSON string containing `{job_id, type, payload, created_at, retry_count}`.

**Why this matters:** BRPOPLPUSH provides atomic pop-and-move semantics. The processing queue acts as a "checkpoint" — even if the worker crashes, the job is not lost. This is a production pattern used in thousands of applications.

### Challenge 5: The API Rate Limiter

**Scenario:** Design a rate limiter for a public API. Each API key has a limit of 100 requests per minute. You must enforce this limit across multiple application servers (10 servers). The rate limiter must handle 500,000 API requests per second at peak. False negatives (blocking a legitimate request) are unacceptable. False positives (allowing a blocked request) are acceptable if rare.

**Task:** Design the rate limiter using a key-value store. Show the algorithm, the data model, the read path, and the write path. Estimate the Redis throughput required.

**Model answer:** Use a sliding window counter with Redis sorted sets. For each API key, maintain a sorted set: `ratelimit:{api_key}`. The score is the timestamp of each request. The member is a unique request ID (UUID or hash of request).

Write path for each request: `ZREMRANGEBYSCORE ratelimit:{key} -inf {now-60}` to remove expired entries (O(log n)). Then `ZCARD ratelimit:{key}` to count remaining (O(1)). If count is under 100, `ZADD ratelimit:{key} {now} {request_id}` and `EXPIRE ratelimit:{key} 60`, then allow. Otherwise, deny with 429.

Optimization: for high-volume API keys (100 req/min is the max), the sorted set has at most 100 entries. ZREMRANGEBYSCORE is O(log 100) which is negligible.

Throughput estimation: at 500,000 req/s, assuming 100 unique API keys, each sorted set operation takes approximately 20 microseconds. On a single Redis instance, you can handle approximately 50,000 req/s. You need 10 Redis shards. Use a hash ring to distribute API keys across shards.

Failure mode: if Redis is unavailable, fall back to a local in-memory counter with best-effort rate limiting. The local counter may allow slightly more requests than the limit during Redis downtime, which is acceptable per the requirements.

**Why this matters:** Rate limiting is a common system design interview question. The sliding window sorted set approach is the industry standard. The fallback to local state shows operational maturity.

### Challenge 6: The Session Store

**Scenario:** Design a distributed session store for a web application with 50 million monthly active users. Each user has 1-3 active sessions (browser, mobile, tablet). Sessions store user ID, roles, preferences, and CSRF token — approximately 2 KB per session. Sessions expire 7 days after creation or 24 hours after last activity (whichever comes first). The system must handle a Black Friday spike of 4x normal traffic.

**Task:** Design the session store. Include key schema, TTL strategy, cluster topology, failover, capacity planning, and monitoring.

**Model answer:** Key schema: `session:{session_token_hash}` where the value is a Redis hash with fields `user_id`, `roles`, `preferences`, `csrf_token`, `created_at`. Using a hash enables updating individual fields without rewriting the entire 2 KB value.

TTL strategy: set TTL to 86400 seconds (24 hours) on every session read. Additionally, track the `created_at` field and delete any session where `created_at + 604800 (7 days)` has passed. This gives the "max 7 days or 24 hours idle" requirement.

Capacity: 50M users * 2 sessions average = 100M sessions. 100M * 2 KB * 1.6x overhead = 320 GB. With replication factor 2, that is 640 GB across the cluster. Using r6g.4xlarge nodes (128 GB RAM each, 100 GB usable), you need 7 nodes for data, rounding to 10 nodes (6 data nodes + 1 spare per 3 data nodes). Shard into 10 partitions, each with 2 replicas = 30 nodes total.

Black Friday: 4x traffic means 4x QPS. Redis handles this easily (QPS scales linearly with nodes, and your normal QPS is well under capacity). The concern is memory: 4x traffic may mean more logged-in users. Monitor memory and have auto-scaling configured to add nodes if utilization exceeds 80%.

Failover: Each shard has a leader and 2 replicas. Redis Sentinel (or ElastiCache auto-failover) promotes a replica if the leader fails. Client-side retry with jitter prevents thundering herd on failover.

Monitoring: Track cache hit rate (should be 99%+ for sessions), eviction rate (should be near-zero), memory utilization per node, and TTL expiry rate.

**Why this matters:** Session storage is a universal infrastructure component. The capacity planning and failover design show you can operate at scale.

## Tier 3 — Expert (30 minutes each)

### Challenge 7: The Distributed Counter

**Scenario:** Your social media platform has a feature where users can "like" posts. The like count is displayed on every post and must be eventually consistent across the globe. At peak, users generate 1 million likes per second globally. The count must not decrease (no undoing likes). The system must survive a regional outage without losing likes. You operate data centers in 5 regions.

**Task:** Design a distributed like counter system using key-value stores. Address conflict resolution, durability, cross-region replication, and the read-path optimization. Estimate cost.

**Discussion:** This is a CRDT (Conflict-free Replicated Data Type) problem. The like counter is a G-Counter (grow-only counter), which is a CRDT that supports only increment operations. Each region maintains its own counter shard, and the total count is the sum across all regions.

Architecture: In each region, a Redis cluster stores counters as strings. For each post `post:{post_id}:likes:{region_id}`, the value is a simple integer. When a user likes a post in us-east-1, the application increments `post:123:likes:us-east-1` using `INCR`. This is atomic and costs approximately 50 microseconds. The global like count is the sum of all 5 regional counters: read all 5 keys and add them. This is eventually consistent and monotonic (counts never decrease).

Cross-region replication is not needed because each region manages its own counter. The drawback: read latency for the global count requires 5 key reads (one per region). For a post viewed primarily in one region, you can cache the global count with a 60-second TTL in the viewing region's Redis.

Durability: Each region's Redis has AOF persistence to survive crashes. A regional outage pauses likes for that region but does not lose data. When the region recovers, it catches up and its counter resumes incrementing.

Cost: 5 regions * 10 Redis nodes per region = 50 nodes at approximately $500/month each = $25,000/month. For 1 billion daily active users, that is approximately $0.000025 per user per month.

Alternative: DynamoDB with atomic increment can replace Redis in each region. The auto-scaling handles traffic spikes without manual intervention.

**Why this matters:** CRDTs are an advanced distributed systems topic. The G-Counter is the simplest CRDT and directly applicable to real problems. This challenge tests whether you understand monotonic consistency, conflict-free data types, and partitioned counting.

### Challenge 8: The Key-Value Store From Scratch

**Scenario:** You are building a new key-value store for a specific internal workload: 1 KB values, 10-byte keys, 99% reads, read latency must be under 200 microseconds p99, write latency under 1 ms p99, data fits in 256 GB RAM, no persistence needed (rebuilt on restart), must handle 2 million QPS.

**Task:** Design the data structure, the threading model, the network layer, and the memory allocator. This is a whiteboard design — you are not expected to actually build it.

**Discussion:** For this specific workload, a general-purpose key-value store like Redis has too much overhead. Redis uses approximately 60 bytes of overhead per key, stores values with type metadata, and has a single-threaded event loop that can bottleneck at 2M QPS.

Data structure: Use a lock-free hash table with open addressing and Robin Hood hashing. Keys are 10 bytes — store them inline in the hash table entries (no separate memory allocation for keys). Values are 1 KB pointers to a slab allocator. The hash table size is a power of 2 with a load factor of 0.75. For 100M items (estimated from 256 GB / 1 KB), you need 133M buckets at approximately 24 bytes each = 3.2 GB for the hash table itself.

Threading model: Use multiple threads with shard-level locking. Modern servers have 64+ cores. Shard the hash table into 64 shards, each protected by a spin lock. Each thread handles its own network connections and hash table shard. This removes the single-thread bottleneck.

Network layer: Use io_uring (Linux) for asynchronous I/O or DPDK for kernel bypass. Avoid the epoll + read/write cycle — copy user-space data directly from NIC buffers to the hash table. Target 1 microsecond network overhead per request.

Memory allocator: Use a slab allocator for values (1 KB slots pre-allocated in contiguous memory). No per-allocation overhead. Slabs are recycled when values are deleted. The slab allocator eliminates fragmentation and malloc overhead.

Performance estimation: Each GET is: hash computation (50 ns), array lookup (10 ns), key compare (10 ns), memcpy value to response buffer (200 ns). Total: approximately 270 ns per GET. With 64 threads, theoretical throughput is approximately 64 / 270 ns = 237 million QPS. Real-world (accounting for contention, cache misses, network): approximately 10-20 million QPS on a single 64-core server.

**Why this matters:** This challenge tests whether you truly understand the internals. Designing a specialized key-value store requires knowledge of hash tables, memory allocators, threading models, and network I/O that goes far beyond knowing Redis commands.
