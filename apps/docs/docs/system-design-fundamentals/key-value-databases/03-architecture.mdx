# Key-Value Databases at Scale

## Naive Implementation

The simplest key-value store is a single server with a hash table in memory. It works perfectly at small scale. Here is what breaks as you grow.

### What Breaks at Each Milestone

**1,000 users (MVP stage):** Everything works. Latency is under 100 microseconds. CPU is under 5%. 512 MB of RAM is enough. You have zero operational complexity. This is the right architecture for this stage.

**100,000 users (growth stage):** Latency spikes to 500-2000 microseconds during traffic bursts. CPU sits at 60-80%. Memory is at 80% of 32 GB. A single large key (a cached HTML page of 500 KB) causes latency spikes for all other requests because Redis is single-threaded. Connection count hits 4,000, and the event loop shows measurable slowdown. You start seeing "max number of clients reached" errors during traffic peaks. A single server failure takes down your entire cache. Recovery takes 30 minutes because RDB loading from disk is slow for a 25 GB dataset.

**10 million users (scale stage):** The single server cannot serve this many requests. Memory cannot hold the working set. CPU is pegged at 100% with request queuing. The system is down more than it is up. You need a distributed cluster.

### How This Architecture Scales

**Adding capacity:** To add a fourth shard, you run a resharding operation. Redis Cluster moves hash slots from existing nodes to the new node. Consistent hashing ensures only 25% of keys move (the fraction assigned to the new node's hash range). The cluster stays online during the migration.

**Read scaling:** Add more replicas per shard. The leader handles writes; replicas serve read-only traffic. With one leader and three replicas per shard, you can serve 4x the read throughput.

**Write scaling:** Add more shards. Each shard handles its own write volume independently. A 10-shard cluster can handle roughly 10x the writes of a single node (accounting for replication overhead).

**Node failure:** When a leader fails, the cluster promotes its replica to leader. Redis Cluster detects failure via gossip protocol. Promotion takes under 10 seconds with default settings.

## Multi-Region and Global Scale

When your users are distributed across the globe, a single-cluster deployment gives slow responses to distant users. Multi-region key-value stores solve this with replication and routing.

### Consistency Models in Multi-Region

**Eventual consistency** is the default for cross-region replication. A write in us-east-1 may take 50-200 ms to reach eu-west-1. Users in Europe see stale data during that window. For many use cases (cached content, session data, leaderboards), this is acceptable.

**Strong consistency** requires writing to all regions synchronously, which adds cross-continent round-trip time. A write from Singapore to a primary in us-east-1 takes 200+ ms. This is unacceptable for most applications. If you need strong consistency, co-locate your primary cluster with the majority of your write traffic.
