# Key-Value Databases — Interview Playbook

## How This Topic Appears

Key-value database questions show up in system design, coding, and behavioral interviews. Here are the most common phrasings by category.

### System Design Questions (Most Common)

1. "Design a distributed key-value store." (The classic — expect a Dynamo-style discussion)
2. "Design a cache for a social media feed." (Cache-aside, eviction, TTL, consistent hashing)
3. "Design a rate limiter for an API." (Sliding window with Redis sorted sets or atomic counters)
4. "Design a distributed session store for a web application." (TTL, replication, multi-region)
5. "Design a real-time leaderboard for a game with 10 million players." (Sorted sets, sharding by score range)
6. "Design a URL shortener." (Key generation, hash collisions, key-value storage)

### Coding Questions

7. "Implement a Least Recently Used (LRU) cache with O(1) get and put." (Doubly linked list + hash map)
8. "Implement consistent hashing with virtual nodes." (Ring, hash function, node management)
9. "Implement a thread-safe key-value store with expiration." (Hash map + priority queue for TTL)

### Behavioral Questions

10. "Tell me about a time your team had a caching incident. What happened and how did you fix it?"
11. "How would you decide between Redis and Memcached for a new project?"
12. "Your production cache hit rate just dropped from 95% to 60%. Walk me through your debugging process."

## What the Interviewer Is Actually Testing

When an interviewer asks about key-value databases, they are not testing whether you know Redis commands. They are testing five deeper competencies:

**Trade-off awareness.** Can you articulate the explicit trade-offs between consistency and availability, between speed and durability, between simplicity and capability? The best candidates name the trade-off unprompted and give concrete examples of when to choose each side.

**Distributed systems thinking.** Key-value databases expose fundamental distributed systems problems: partitioning, replication, failure detection, conflict resolution. Interviewers want to see that you can reason about what happens when a node fails, a network partition occurs, or a client sends a conflicting write.

**Operational intuition.** What breaks in production? How do you debug a slow cache? What metrics do you monitor? Interviewers want to know if you have real operational experience or just theoretical knowledge.

**Capacity planning and sizing.** Can you estimate how many nodes you need for 10 million daily active users? Can you calculate whether your working set fits in memory? These back-of-envelope calculations reveal years of production experience.

**Pragmatism and trade-off selection.** The best architects do not propose perfect solutions. They propose good-enough solutions that match the constraints. An interviewer wants to see you choose between Redis and Memcached, between eventual and strong consistency, between in-memory and persistent storage, and articulate your reasoning clearly.

## Level-by-Level Expectations

### L3 / L4 (Early Career)

**Expected knowledge:** You know what a key-value store is, can name Redis and Memcached, understand the basic get/set operations, and can explain the difference between a cache and a primary database.

**Sample question:** "Explain the difference between a key-value database and a relational database."

**Strong answer:** "A key-value database stores data as opaque values indexed by a unique key. It supports three operations: Get(key), Put(key, value), and Delete(key). There is no schema, no query language, and no indexing beyond the primary key. A relational database stores data in tables with rows and columns, supports complex queries through SQL including JOINs and aggregations, enforces schemas, and maintains indexes on multiple columns. The trade-off is that key-value stores are much faster for lookups by known key (microseconds vs milliseconds) but cannot answer questions about the data itself. You use a key-value store when you know the key and need the value fast — for example, caching database query results. You use a relational database when you need to ask complex questions about your data — for example, 'find all orders from last month worth over $100.'"

### L5 (Senior Engineer)

**Expected knowledge:** You can design a caching layer for a distributed system, explain consistent hashing, describe eviction policies in detail, and discuss replication strategies.

**Sample question:** "Design a caching layer for a social media feed that serves 1 million requests per second."

**Strong answer:** "I would use a two-tier cache architecture. Tier 1 is a local in-process cache (like a Guava cache or Caffeine) on each application server. This cache stores the most frequently accessed feed data with a TTL of 5 seconds. Tier 1 absorbs approximately 40% of reads and has single-digit microsecond latency. Tier 2 is a distributed Redis cluster with 20 shards and 2 replicas per shard, totaling 60 nodes. I would use consistent hashing to distribute keys across shards and client-side smart routing with a hash ring. The eviction policy is `allkeys-lru` with a maxmemory of 80% of RAM on each node.

For writes, we use a write-through pattern: when a user posts new content, we invalidate the affected feed caches by key pattern. To avoid a cache stampede, we use stale-while-revalidate: serve the stale cached feed for up to 2 seconds while the new feed is computed in the background. Each node monitors cache hit rate, eviction rate, and memory utilization. If hit rate drops below 85%, we trigger a scale-up event to add more nodes.

The bottleneck is likely the hot key problem: a celebrity user's feed getting 100x more reads than average. We handle this by replicating the hot key to multiple shards and letting the client randomly pick one, spreading the load."

### L6 (Staff Engineer)

**Expected knowledge:** You can design a distributed key-value store from scratch (Dynamo-style), understand consistency models deeply (vector clocks, quorums, CRDTs), and have production troubleshooting experience.

**Sample question:** "Design a distributed key-value store with strong consistency guarantees."

**Strong answer:** "I would design a system based on Raft consensus protocol with a leader-per-partition architecture. The system has three layers: a routing layer, a consensus layer, and a storage layer. The routing layer uses consistent hashing to map keys to partitions. Each partition runs the Raft protocol with 3 or 5 nodes. All writes go through the Raft leader, which appends to a write-ahead log and replicates to a majority of followers before acknowledging the write.

For reads, the client can choose between strong consistency (read from the leader, which guarantees the latest data) and eventual consistency (read from any follower, which may return stale data but at lower latency). Strong consistency costs an extra RTT to verify the leader is still the leader.

Write throughput is limited by the leader's single-node capacity. To scale writes, we partition the key space into 256 or more partitions, each with its own Raft group. Throughput scales linearly with the number of partitions.

The hard part is handling cluster membership changes. When adding or removing nodes, we must safely move partition leadership without violating consistency. We use joint consensus (as described in the Raft paper) to transition membership in two phases.

For conflict resolution during network partitions (in the eventual consistency mode), we use version vectors. Each value carries a list of `{node, counter}` pairs. When returning a value, we return all conflicting versions. The application resolves conflicts using last-writer-wins, CRDT merge, or custom logic."

### L7 / L8 (Principal / Distinguished)

**Expected knowledge:** You can design a global-scale key-value infrastructure with multi-region replication, disaster recovery, and capacity planning across thousands of nodes. You have opinions on the trade-offs between Dynamo-style and Raft-based designs. You have led post-mortems for major outages.

**Sample question:** "Design the key-value infrastructure for a global social network with 500 million daily active users."

**Strong answer:** "I would use a multi-region, multi-cluster architecture with three tiers. The foundation is a platform layer that abstracts the specific key-value implementation behind a common API. This lets us use different engines for different workloads without coupling applications to a specific vendor.

For the storage plane, we use a combination of approaches. Session data and ephemeral state use a Redis Cluster with 500 nodes, deployed in each region. For durable data (user profiles, preferences), we use an internal DynamoDB-compatible store built on Raft, with 1000 partitions across 200 nodes per region. For the hot read path, we have a Memcached tier with local in-process caching on every application server.

Cross-region replication uses an event-driven approach. Each region has a change-data-capture stream (similar to DynamoDB Streams) that publishes all mutations to a central event bus. Other regions consume these events and update their local stores. We accept that cross-region consistency is eventual with a typical lag of 1-5 seconds.

The critical failure mode is cross-region write conflicts. A user might update their profile in us-east-1 and, 2 seconds later, update it again in eu-west-1. We use CRDTs (Conflict-free Replicated Data Types) for data that can be merged automatically — counters, sets, maps. For data that requires manual conflict resolution, we use last-writer-wins with a logical timestamp and surface conflicts to the user.

Capacity planning: each region serves approximately 100 million users. With a working set of 50 items per user at 2 KB each, we need approximately 10 TB of hot data plus replication overhead. With 128 GB per node and 70% usable capacity, we need approximately 140 nodes per region. With 3 replicas for durability, that is 420 nodes per region. Across 5 regions, we operate approximately 2,100 total storage nodes.

The operational burden is significant. We have a dedicated team of 15 SREs per region, on-call rotations, and a runbook for the 12 most common failure modes."

## FAANG Framework for Answering

When you get a key-value system design question, use this six-step framework.

**Step 1 — Clarify scope.** "What is the primary workload? Is this a cache or persistent storage? What are the consistency requirements? What is the scale in QPS and data size?" Do not start designing until you have the answers.

**Step 2 — Define the API.** "The system exposes three operations: Get(key) returns value or nil. Put(key, value, ttl) returns OK. Delete(key) returns OK." Write these down explicitly. A clear API prevents scope creep.

**Step 3 — Choose the data model.** "Values are opaque byte arrays up to 1 MB. Keys are strings with a maximum length of 256 bytes." State your assumptions about the data.

**Step 4 — Design the single-node architecture.** "Each node has a hash table in memory with incremental rehashing, a TTL expiry engine with lazy and active expiration, and an AOF persistence log fsynced every second." Prove you understand the internals before scaling.

**Step 5 — Scale to multiple nodes.** "We use consistent hashing with 256 virtual nodes per physical node. Each key maps to a hash ring position, and the nearest N replicas store it. We replicate to 3 nodes for fault tolerance. The replication factor determines read and write quorum sizes."

**Step 6 — Discuss trade-offs and failure modes.** "With quorum-based read/write (R + W greater than N), we can tune for consistency or availability. Setting R=2, W=2, N=3 gives strong consistency at the cost of tolerating only 1 node failure. Setting R=1, W=1, N=3 gives maximum availability but risks stale reads and write conflicts. The hot key problem is the most common failure mode, which we address with client-side caching and automatic key replication."

## Phrase Bank

Use these expert phrases to communicate depth during your interview.

1. "The fundamental trade-off in any key-value store is between consistency and availability during a partition."

2. "Consistent hashing minimizes the number of keys that must be remapped when the cluster topology changes."

3. "The working set size determines memory requirements. If the working set exceeds available memory, eviction rate spikes and hit rate collapses."

4. "Last-writer-wins conflict resolution uses wall-clock timestamps, which requires synchronized clocks across all nodes."

5. "Vector clocks provide causal consistency without requiring synchronized clocks, but the metadata grows with the number of writers."

6. "A read-repair mechanism detects stale replicas during a read and updates them asynchronously to maintain eventual consistency."

7. "The gossip protocol disseminates membership information: each node periodically exchanges state with a few random peers."

8. "Hinted handoff allows a node to temporarily accept writes for a downed replica and forward them when it recovers."

9. "The eviction policy must match the access pattern. `allkeys-lru` works for most caching workloads; `volatile-ttl` is better when you manually set TTLs with intent."

10. "Stale-while-revalidate serves cached data past its TTL while asynchronously refreshing it, avoiding cache stampedes."

11. "Bloom filters probabilistically test whether a key exists in an SSTable, eliminating unnecessary disk reads for non-existent keys."

12. "Write amplification in LSM-trees means each write causes multiple disk I/Os during compaction, which wears SSDs faster."

13. "A cache stampede occurs when a popular key expires and N concurrent requests all trigger a cache miss, overwhelming the backend."

14. "The thundering herd problem appears during failover when all clients reconnect to the new leader simultaneously, overwhelming it."

15. "Partition-level auto-scaling adjusts the number of read capacity units per partition independently, enabling fine-grained throughput management."

16. "Read-after-write consistency guarantees that a client sees its own writes immediately, even if other clients may see stale data."

17. "The single-threaded event loop in Redis is simultaneously its greatest strength (atomicity, no race conditions) and its greatest limitation (a slow command blocks all other requests)."

## Red Flags

Avoid these specific mistakes in your interview answers.

**Red Flag 1: Thinking all key-value stores are the same.** Saying "I would just use Redis" without discussing the trade-offs between Redis, Memcached, and DynamoDB shows inexperience. Acknowledge the differences and explain why you chose each one.

**Red Flag 2: Not mentioning consistent hashing.** If you design a distributed key-value store without discussing consistent hashing, an interviewer notices immediately. It is the foundational technique for scaling key-value stores.

**Red Flag 3: Ignoring the hot key problem.** Any distributed cache design that does not address hot keys is incomplete. Mention client-side caching, key replication, or local caches as mitigation strategies.

**Red Flag 4: Proposing strong consistency without cost.** If you say "all reads return the latest write" without mentioning the latency cost, quorum requirements, or availability trade-offs, you sound like you have never operated a distributed system at scale.

**Red Flag 5: No mention of monitoring.** An experienced engineer always discusses what to monitor: cache hit rate, eviction rate, memory fragmentation ratio, replication lag, and connection counts.

**Red Flag 6: Over-engineering.** Proposing a Dynamo-style multi-master system for a problem that needs a simple cache-aside Redis setup signals inexperience with pragmatic trade-offs. Match the solution complexity to the problem.

**Red Flag 7: No backup plan for failure.** "The cache always works" is not a design. Discuss what happens when the cache nodes fail, the network partitions, or the hot key overloads a shard.

**Red Flag 8: Confusing cache and database.** Using a cache as a primary database without discussing durability, backup, and recovery mechanisms shows a fundamental misunderstanding. If data must survive a restart, you need persistence or use a persistent store.

## Practice Drills

Set a timer for each drill and practice speaking your answer aloud.

**Drill 1 (5 minutes):** "A teammate wants to replace all PostgreSQL queries with Redis because 'it is faster.' How do you respond?" Structure your answer around the core tension: speed versus query capability.

**Drill 2 (5 minutes):** "Your cache hit rate dropped from 92% to 45% overnight. Walk through your debugging process." Start with monitoring tools, then check eviction rate, memory usage, recent deployments, and traffic pattern changes.

**Drill 3 (10 minutes):** "Design a session store for a web application with 5 million daily active users. Users log in from multiple devices and sessions must expire after 24 hours." Cover key schema (`"session:{token}"`), TTL, replication for durability, and emergency capacity for Black Friday traffic.

**Drill 4 (10 minutes):** "A production Redis node crashed and all data on that node was lost because persistence was not configured. What do you tell the team, and what do you change?" Cover the technical issue (enable AOF, set appendfsync everysec) and the process issue (change review, runbook gaps, monitoring).

**Drill 5 (15 minutes):** "Design a rate limiter for a public API with 100,000 registered developers, each with different rate limits (free tier: 100 req/min, pro tier: 10000 req/min)." Cover the sliding window algorithm, Redis sorted set implementation, per-developer counters, and graceful degradation when Redis is unavailable.
