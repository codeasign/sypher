The theory of logging is clean. The practice is messy. Every company that operates at scale has war stories about logging systems that failed, cost too much, or saved the day. This section examines how four major companies — Netflix, Uber, Discord, and Cloudflare — handle logging in production, then distills the lessons into actionable patterns you can apply to your own system.

## Netflix: The Logging-Free Zone

Netflix operates one of the largest streaming platforms on the planet, serving over 260 million subscribers across 190 countries. Their logging philosophy is surprising: they deliberately minimize logging in the hot path and rely on a sophisticated metrics and tracing system instead.

### The Problem

Netflix's streaming service processes over 1 billion events per second during peak hours. Each event is a video playback session, a UI interaction, a recommendation request, a content encoding job, or a CDN health check. If they logged every event at INFO level, the log volume would exceed 500 PB per day — economically infeasible and operationally useless.

### The Solution

Netflix made a deliberate architectural decision: logs are for debugging, not for operations. Their observability stack uses three tiers:

1. **Atlas (metrics)**: every code path emits metrics — counters, timers, gauges, distributions. Metrics are aggregated in-memory and reported every 60 seconds. This gives them operational visibility without the storage cost of individual log events.

2. **Distributed tracing (Zipkin/OpenTelemetry)**: every request carries a trace ID. Spans capture the service boundaries and latency. Traces are sampled at 1:100 by default, with tail-based sampling to capture all events for failed requests.

3. **Structured logging (debug mode)**: logs are written at DEBUG level to local files with rotation. Each service has a "debug mode" flag that can be toggled via a configuration push. When an engineer needs to investigate a specific issue, they enable debug mode for a single service instance, reproduce the issue, and inspect the local logs.

### Key Takeaway

Logging is not the default observability tool. For most operational questions, metrics and traces provide faster answers with lower cost. Reserve logs for the specific scenarios where you need the raw event-level detail: debugging a new feature, investigating a multi-service failure, or performing a security audit. Netflix's approach saves them approximately $100 million per year in logging infrastructure costs compared to a naive "log everything" approach.

## Uber: The Centralized Logging Giant

Uber operates at a similar scale to Netflix but took the opposite approach: they built a massive centralized logging system called CLS (Continuous Logging Service) that processes over 10 PB of log data per day.

### The Problem

Uber's microservices architecture includes over 4,000 services. When an incident occurs, engineers need to correlate logs across dozens of services, and each service team was building its own logging solution. The result was fragmentation: 15 different logging systems, no cross-service search, and 30-minute delays between log generation and availability.

### The Solution

Uber built CLS as a unified logging platform. The architecture:

1. **Ingestion**: each service writes structured JSON logs to stdout. A sidecar process (uLogger) reads stdout, adds metadata, and publishes to Kafka.
2. **Buffering**: a dedicated Kafka cluster with 100 partitions per topic handles 10 million events per second. Data is retained in Kafka for 48 hours to allow replay.
3. **Indexing**: CLS uses a custom indexing engine built on top of HDFS (not Elasticsearch). The custom engine supports 5x higher throughput per node than Elasticsearch at the time of building.
4. **Storage**: logs are stored in Parquet format on HDFS. This allows them to run SQL queries (via Hive/Presto) on historical logs.
5. **Query**: a web UI with a search interface similar to Kibana, plus a REST API for programmatic access.

### Key Takeaway

At Uber's scale, building a custom logging system was cheaper than any commercial product. The custom system saved 60 percent compared to the cost of Splunk or Datadog at the same volume. However, the investment was enormous — a team of 20 engineers spent 18 months building CLS. For most organizations, a managed service (Datadog, Splunk, Grafana Cloud) is the right choice.

## Discord: From MySQL to ScyllaDB

Discord serves over 150 million monthly active users, generating 100 million log events per second at peak. Their logging journey illustrates how storage backend choices affect the entire system.

### The Problem

Discord's original logging system stored events in MySQL. Each event was a row in a large table. As the user base grew, MySQL could not keep up with the write throughput. Queries on the log table would block other application queries. The team tried sharding manually, but the operational burden was unsustainable.

### The Solution

Discord migrated to ScyllaDB (a Cassandra-compatible NoSQL database) as the log storage backend. The key properties:

1. **Write throughput**: ScyllaDB handles 1 million writes per second on a 10-node cluster. Each write is a single partition key (timestamp bucket + trace ID) with a single row.
2. **Time-series optimized**: logs are written with a time-based partition key. Queries within a time range hit a single partition, making them fast.
3. **No single point of failure**: ScyllaDB is fully distributed with no master. Any node can accept writes.
4. **Compression**: ScyllaDB compresses data at 4:1 ratio, reducing storage costs.

```python
# Discord's ScyllaDB schema for logs
CREATE TABLE logs (
    bucket_hour timestamp,    # Partition key: hour bucket
    trace_id text,            # Clustering key: sorted within bucket
    event_time timestamp,     #
    service text,
    level text,
    message text,
    metadata map<text, text>,
    PRIMARY KEY (bucket_hour, trace_id, event_time)
) WITH compaction = { 'class': 'TimeWindowCompactionStrategy' };
```

The migration reduced query latency from 2 seconds to 50 milliseconds and cut the storage cost by 60 percent.

### Key Takeaway

The storage backend for logs must be optimized for time-series workloads. Traditional relational databases (MySQL, PostgreSQL) are poorly suited for log ingestion because they optimize for transactional consistency, not write throughput. Use a time-series database (ScyllaDB, Cassandra, TimescaleDB, InfluxDB) or a specialized log store (Elasticsearch, Loki).

## Cloudflare: Logging at the Edge

Cloudflare operates a global network spanning 310 cities. Every HTTP request that passes through their network — over 50 million per second — generates a log entry. Their logging challenge is unique: they need to collect logs from 310 edge locations and centralize them for analysis.

### The Problem

Cloudflare's edge locations have limited compute and storage. Each edge server processes millions of requests per second and cannot store logs locally — the disk would fill in minutes. Sending every log entry to the central system over the public internet would cost a fortune in bandwidth: 50 million requests per second at 500 bytes each is 25 GB/s, or approximately $15 million per year in egress charges.

### The Solution

Cloudflare designed a tiered logging system:

1. **Edge sampling**: each edge server samples log entries at a variable rate based on current load. During normal traffic, 1 in 1,000 requests are logged. During an attack, the sampling rate increases to 1 in 100.
2. **Local aggregation**: sampled logs are aggregated in 60-second batches, compressed with zstd, and uploaded to R2 (Cloudflare's object storage) in the nearest region.
3. **Central indexing**: a Spark cluster reads compressed batches from R2, decompresses, indexes into Elasticsearch, and stores raw data in S3 for long-term retention.
4. **Customer-facing analytics**: customers can query their own logs through a dashboard.

At extreme scale, you cannot centralize all logs. Sampling is not optional — it is a survival mechanism. The key is to make sampling intelligent: capture more data during anomalies and less during normal operation.

### CloudWatch Logs Deep Dive

AWS CloudWatch Logs is the most widely used managed logging service. It integrates natively with every AWS service. Key features:

**Automatic ingestion**: EC2 instances, Lambda functions, ECS containers, and ELB load balancers automatically send logs to CloudWatch when configured. The CloudWatch Agent runs on EC2 and collects system logs, application logs, and custom metrics.

**Logs Insights**: a SQL-like query language for searching and analyzing logs. Queries run against data stored in S3, so they are subject to S3 query latency (5-10 seconds for the first result).

**Subscription filters**: forward log events to Lambda, Kinesis, or OpenSearch in real-time. This enables custom processing pipelines.

**Metric filters**: extract metrics from log events. For example, count the number of ERROR log lines per minute and create a CloudWatch metric.

Cost: $0.50 per GB ingested, plus $0.03 per GB stored. At 1 TB/day, CloudWatch costs $15,000/month for ingestion alone. This is expensive compared to self-hosted solutions but includes the operational expertise.

### Grafana Cloud / Loki Deep Dive

Grafana Cloud with Loki is a popular alternative to Elasticsearch-based log systems. Loki does not index the content of log lines — it only indexes labels (service name, level, environment). This makes it significantly cheaper for high-volume logging.

**How Loki works**: each log stream is identified by a set of labels (e.g., `{service="orders", env="production"}`). Log lines are appended to the stream without indexing. Queries filter by labels and then search the filtered log lines. This is much faster than Elasticsearch for simple queries and much slower for complex text searches.

**When to use Loki**: Loki is ideal for teams that already use Grafana for metrics and dashboards. It integrates natively with Grafana, so you can view logs and metrics on the same dashboard. Loki is not suitable for full-text search, regex queries, or complex aggregations.

**Cost comparison**: Loki at $0.50/GB ingested is 3-5x cheaper than Elasticsearch-based services. The trade-off is query flexibility: Loki supports label-based queries and simple text search, but not the rich query language of Elasticsearch.

## Lessons Learned

From these case studies, seven lessons emerge.

**Lesson 1: Logging strategy must match scale.** Netflix's approach (minimize logging, rely on metrics) works at their scale. For a startup with 10 services, the same approach is over-engineering. Use structured logging + a managed service until you hit 100 TB per day, then consider custom infrastructure.

**Lesson 2: Cost grows linearly with volume, but value grows logarithmically.** The first 10 GB of logs per day provides 90 percent of the debugging value. The next 90 GB provides 9 percent. The next 900 GB provides 1 percent. Sample aggressively and index selectively.

**Lesson 3: The storage backend determines the system's ceiling.** MySQL cannot handle log ingestion at scale. Elasticsearch works well up to 100 TB. Beyond that, consider ScyllaDB, ClickHouse, or a custom solution on HDFS/Parquet.

**Lesson 4: Separate pipelines for different purposes.** Application logs, audit logs, access logs, and system logs have different retention, indexing, and query requirements. Mixing them degrades all of them.

**Lesson 5: Sampling is not optional.** Every system that processes more than 1 TB of logs per day must sample. The question is not whether to sample, but how to sample intelligently.

**Lesson 6: Tune the refresh interval.** The default Elasticsearch refresh interval (1 second) is designed for search, not ingestion. For log ingestion, increasing it to 10-30 seconds improves throughput by 3-5x.

**Lesson 7: Monitor the logging system separately.** The logging system is critical infrastructure. If it goes down, your entire operations team is blind. Monitor its health independently of the applications it monitors.

### Building Your Own Roadmap

Based on these lessons, here is a phased roadmap for implementing logging at your organization:

**Phase 1: Foundation (week 1-2)**
- Add structured JSON logging to all services using your language's structured logging library (structlog, zap, tracing).
- Add correlation IDs to all requests via middleware.
- Configure log rotation with 100 MB max file size and 10 backup files.
- Set up log level configuration via environment variables.

**Phase 2: Centralization (week 3-4)**
- Deploy Filebeat as a DaemonSet on Kubernetes, or install it on each VM.
- Deploy a single Elasticsearch node (or use Elastic Cloud free tier) for centralized search.
- Configure Kibana dashboards for basic log search and filtering.
- Set up a 30-day retention policy.

**Phase 3: Scale (month 2-3)**
- Add Kafka between the shipper and indexer for buffering and backpressure.
- Add a second Elasticsearch node for failover.
- Set up ILM for tiered storage: hot (SSD, 7 days), warm (HDD, 30 days), cold (S3, 365 days).
- Configure log-based alerts for common error patterns.

**Phase 4: Optimize (month 3-6)**
- Implement tail-based sampling for error detection.
- Set up per-service log budgets to control cost.
- Add log obfuscation filters for sensitive data.
- Integrate logs with the existing metrics and tracing system.

**Phase 5: Advanced (month 6-12)**
- Evaluate managed vs. self-hosted for the next growth phase.
- Implement multi-region query for global deployments.
- Build custom dashboards for common debugging workflows.
- Automate log schema governance through CI/CD.

### Common Mistakes in Production Logging

Even experienced teams make these mistakes. Watch for them in your own system.

**Mistake 1: Not testing the logging pipeline under load.** A team deploys their logging system, tests it with 10 requests per second, and it works. On launch day, traffic is 10,000 requests per second, and the logging system collapses. Always test the logging pipeline at 2x the expected peak load.

**Mistake 2: Not having a log retention policy from day one.** Logs accumulate without any deletion policy. After 6 months, the disk is full, and the application crashes. The team frantically deletes old indices, losing data they needed. Set up retention policies before the first log event is written.

**Mistake 3: Using the same logging configuration for development and production.** Development logs are at DEBUG level and generate 10x more data than production logs. A developer accidentally deploys a DEBUG configuration to production, and the logging cost triples overnight. Use separate configuration files or environment variables for each environment.

**Mistake 4: Not handling log file rotation in containerized environments.** The application inside the container writes to a log file, but the container runtime only captures stdout. When the file rotates, the old file is no longer visible to the container runtime. The rotated logs are lost. In containerized environments, always write logs to stdout, not to a file.

**Mistake 5: Using a single Elasticsearch index for all logs.** A single index grows to 10 TB. Queries become slow, and the index cannot be managed by ILM. Every time a field is added to the mapping, the entire index must be reindexed. Use time-based indices with daily rollover.

### Real-World Incident: How Logging Saved the Day

**Incident**: A major e-commerce platform experienced a 15-minute outage during Black Friday. The checkout service was returning 503 errors. The on-call engineer was woken up at 3 AM.

**Response**: The engineer opened the logging dashboard and searched for `level:error AND service:checkout` in the last 15 minutes. The search returned 12,000 error events. The engineer filtered by `error_type` and found that 95 percent of the errors were `redis_connection_timeout`. The Redis cluster was overloaded.

**Root cause**: A new deployment had introduced a feature that cached product recommendations in Redis. The cache key was not scoped to the user, so every user request for the same product overwrote the same key. The Redis write throughput increased by 10x, saturating the cluster.

**Resolution**: The engineer rolled back the deployment. The error rate dropped to zero within 2 minutes. The total outage was 15 minutes.

**Follow-up**: The team added `user_id` to the cache key, reducing Redis write throughput by 90 percent. They also added an alert on `redis_connection_timeout` errors in the logging system, so the next incident would be detected before users were affected.

**Key logging lessons from this incident**:
- The error log contained the error type (`redis_connection_timeout`), which told the engineer exactly where to look.
- The error rate metric (12,000 events in 15 minutes) told the engineer the severity.
- The structured error fields (`error_type`, `redis_host`, `duration_ms`) provided enough context to diagnose the root cause without additional debugging.
- The correlation ID in each log event allowed the engineer to trace the failing requests from the checkout service to the Redis cluster.

### Real-World Incident: How Logging Hid the Problem

**Incident**: A SaaS platform's logging system had been silently dropping events for 3 weeks. The operations team did not notice because they only used logs during incidents.

**Discovery**: An engineer was investigating a user-reported bug and searched for log events from the user's session. The search returned no results. The engineer assumed the user's session was not logged. The bug remained unfixed for 2 weeks. Eventually, another engineer noticed that the logging system was not returning any results for any query.

**Root cause**: The Elasticsearch cluster had run out of disk space. The shipper was receiving HTTP 429 responses and dropping events. The shipper's disk queue was also full, so events were dropped at the shipper level.

**Why it was not detected**: The team had not set up monitoring on the logging system. The Elasticsearch cluster health was not checked. The shipper's disk queue was not monitored. The Kafka consumer lag was not monitored.

**Resolution**: The team added 10 more nodes to the Elasticsearch cluster, deleted old indices, and restored the logging system. The incident investigation took 4 hours.

**Key logging lessons from this incident**:
- The logging system must be monitored independently of the applications it monitors.
- Set up alerts on shipper uptime, disk usage, and Elasticsearch cluster health.
- The monitoring system for the logging system must use a separate channel (email, SMS, or a separate monitoring stack) so that it works when the logging system is down.

### Real-World Incident: How Logging Prevented a Security Breach

**Incident**: A financial services company's security team detected unusual query patterns in their database audit logs. The logs showed that a single user account was querying the database at 3 AM, reading thousands of customer records.

**Response**: The security team searched the application logs for the same user's activity. They found that the user had logged in through the VPN from an IP address in a different country, which was unusual for this user. The login was marked as successful, but the user's session was created 2 minutes after the login event, which was suspicious.

**Root cause**: The user's credentials had been compromised. The attacker used a stolen password to log in, created a session, and exfiltrated customer data.

**Resolution**: The compromised account was disabled, the password was reset, and all customer records accessed by the attacker were flagged for review.

**Key logging lessons from this incident**:
- Audit logs are critical for security incident response. Without them, the breach would not have been detected.
- The combination of application logs (login events, session creation) and database audit logs (query patterns) provided the full picture.
- Logging the IP address and geo-location of each login event enabled the detection of the unusual login pattern.
- The correlation between the login event (2 minutes before the session) and the data access (3 AM) was the key pattern that triggered the investigation.

### Comparison of Logging Costs Across Providers

Understanding the true cost of logging requires comparing not just the per-GB price, but the operational overhead and query performance.

**Elastic Cloud**: $0.10-0.20 per GB ingested for hot tier, $0.02-0.05 for warm tier. Includes managed Elasticsearch and Kibana. Best for teams that need full-text search, complex aggregations, and custom dashboards.

**Datadog Logs**: $0.10-1.50 per GB ingested depending on retention and indexing. Includes log management, metrics, and traces in a single platform. Most expensive but most integrated. Best for teams that use Datadog for monitoring.

**Grafana Cloud (Loki)**: $0.50 per GB ingested for 30-day retention. Includes Grafana dashboards and alerting. Cheapest for simple log storage. Best for teams already using Grafana.

**Splunk**: $1.50-2.50 per GB ingested. Expensive but feature-rich. Best for enterprises with compliance requirements who need the SPL query language.

**Self-hosted ELK**: infrastructure cost only (approximately $0.03-0.08 per GB). Requires operational expertise. Best for teams with dedicated DevOps resources and over 1 TB/day of logs.

### How Teams Transition Between Providers

Teams rarely stay with one logging provider forever. Here are common transition patterns:

**Startup → Growth**: start with Grafana Cloud Loki ($500/month for 1 TB/day). As the team grows, migrate to Elastic Cloud for better search capabilities ($3,000/month for 1 TB/day).

**Growth → Scale**: as the team hits 10 TB/day, the cost of Elastic Cloud exceeds $30,000/month. Migrate to self-hosted Elasticsearch on AWS ($15,000/month for 10 TB/day).

**Scale → Enterprise**: at 100 TB/day, self-hosted Elasticsearch requires 200+ nodes and a dedicated operations team. Migrate to Datadog or Splunk for the integrated experience ($150,000-250,000/month for 100 TB/day).

### Real-World Incident: How Logging Caused the Outage

**Incident**: A social media platform experienced a cascading failure when their logging system saturated the application's CPU. The application was using a synchronous logging library, and every log event acquired a lock, serialized the message, and wrote to disk. As traffic increased, the logging system consumed 80 percent of the CPU, leaving only 20 percent for the application logic.

**Response**: The operations team could not SSH into the servers because the load was too high. The servers had to be hard-rebooted. The logging system was disabled, and the application was restored. The team then migrated to an asynchronous logging library.

**Root cause**: The application used `logging.basicConfig()` which writes synchronously to stderr. At 50,000 requests per second, the logging library consumed 80 percent of the CPU. The application's throughput dropped from 50,000 to 10,000 requests per second, causing a cascade of timeouts across the microservices.

**Key logging lessons from this incident**:
- Never use synchronous logging in high-throughput services.
- Always use an asynchronous logger with a ring buffer.
- Test the logging library's CPU overhead under load.
- The logging system should never consume more than 5 percent of the application's CPU.

### Real-World Incident: The Cost Explosion

**Incident**: A startup's monthly AWS bill jumped from $10,000 to $50,000 overnight. The engineering team was asked to investigate.

**Root cause**: A junior developer had added a new feature that logged every database query result, including the full response body. The average log event size increased from 500 bytes to 5,000 bytes. The logging volume went from 10 GB/day to 100 GB/day. The Elasticsearch cluster needed 3x more nodes to handle the increased volume.

**Resolution**: The verbose logging was removed. The developer was coached on appropriate log levels. The team added a CI check that warns when log event size exceeds 1,000 bytes.

**Key logging lessons from this incident**:
- Log event size directly impacts cost. A 10x increase in event size leads to a 10x increase in storage cost.
- Review log schema changes in code review, just like API schema changes.
- Add automated checks for log event size and volume.
- Use log budgets per service to prevent one team's logging from affecting another team's budget.

### Real-World Incident: The Debugging Time Sink

**Incident**: A mid-size company's engineering team spent 30 percent of their on-call time searching for logs. Engineers reported that debugging a single production issue took 45 minutes on average, with 30 of those minutes spent finding the relevant log events.

**Root cause**: The team had no correlation IDs. Each service generated logs independently. When an error occurred in the payment service, the engineer had to search the payment service logs by timestamp, then cross-reference with the API gateway logs by approximate time, then check the orders service logs by customer ID. The process was entirely manual.

**Resolution**: The team added correlation IDs to all services. After the change, debugging a single production issue took 10 minutes on average — a 75 percent reduction in debugging time.

**Key logging lessons from this incident**:
- The cost of not having correlation IDs is not just storage — it is engineering time.
- A 75 percent reduction in debugging time translates to a 30 percent reduction in on-call burden.
- Correlation IDs are the single highest-impact logging improvement you can make.

### Real-World Incident: The Multi-Region Query Puzzle

**Incident**: A global SaaS company deployed servers in 3 regions. The logging system was per-region, with no cross-region query. When an engineer needed to debug a request that spanned regions, they had to query each region's logging system separately.

**Resolution**: The team built a global query layer that fans out queries to all regions and merges results. The implementation took 2 months.

**Key logging lessons from this incident**:
- Design multi-region logging from the start, even if you are only deploying in one region initially.
- A global query layer is essential for debugging cross-region requests.
- The global query layer must handle region failures gracefully.
