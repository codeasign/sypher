Every logging decision involves a trade-off. Write every event with fsync? Your data is safe but your application slows to a crawl. Batch events for throughput? You risk losing the last few seconds of data on a crash. Store everything in the hot tier? Your costs explode. Sample aggressively? You miss the one error that matters. This section maps the decision landscape so you can make intentional, informed choices for your system.

## The Core Tension

The fundamental tension in logging is between completeness and performance. You want to record every event to have perfect observability, but recording every event consumes CPU, memory, disk, network bandwidth, and money.

Every team must answer four questions:
1. How much data can we afford to store?
2. How much latency can we tolerate for log writes?
3. How quickly do we need to find logs after they are written?
4. What events must we never lose?

The answers differ by organization. A financial trading platform must never lose audit logs and can afford expensive storage. A social media startup prioritizes low latency and samples aggressively. A healthcare company prioritizes compliance and retains logs for 7 years.

### The Four Dimensions of Trade-offs

Every logging decision involves four dimensions: cost, latency, durability, and queryability. Improving one dimension typically worsens at least one other.

**Cost vs. Durability**: storing logs on SSD with 3x replication costs 10x more than storing on S3 with no replication. The trade-off is query speed vs. cost.

**Latency vs. Completeness**: synchronous writes with fsync guarantee every event is persisted but add 1-10ms per write. Asynchronous writes batch events and add only 1-10µs but risk losing buffered events on crash.

**Queryability vs. Storage**: indexing every field makes queries faster but increases storage by 50-200 percent. Selective indexing reduces storage at the cost of slower queries on unindexed fields.

**Retention vs. Cost**: keeping logs for 90 days costs 3x more than keeping them for 30 days. The trade-off is the ability to investigate long-term trends vs. monthly storage costs.

## Decision Matrix

The following matrix maps common logging decisions to their trade-offs.

| Decision | Option A | Option B | When to Choose A | When to Choose B |
|----------|----------|----------|------------------|------------------|
| Log format | Structured JSON | Plaintext | You need programmatic queries, dashboards, alerting | You are using a simple `tail -f` workflow with no tools |
| Write mode | Synchronous (fsync) | Asynchronous (buffered) | Audit logs, financial transactions, regulatory compliance | Application logs, debug logs, high-throughput services |
| Sampling | Head-based (1:N) | Tail-based (per-request outcome) | Uniform coverage is acceptable, simple implementation | You must capture all events of failed requests |
| Retention | Short (7 days) | Long (1 year+) | Cost-sensitive, fast-moving environment, no compliance requirements | Compliance requirements (HIPAA, SOC 2), long debugging cycles |
| Indexing | Index all fields | Index only filtered fields | You need to query on any field, small cluster | High throughput, large cluster, cost optimization |
| Aggregation | Pull model (shipper pulls logs) | Push model (app pushes logs) | Existing processes, legacy systems, file-based logging | Containerized environments, structured logging, real-time needs |
| Storage tier | Hot only (SSD) | Hot + cold (SSD + S3) | Small volume (under 100 GB/day), fast queries required | Large volume (over 1 TB/day), cost-sensitive, historical queries acceptable at higher latency |
| Alerting | Log-based (pattern matching) | Metric-based (rate counting) | Complex patterns that cannot be expressed as counters | Simple rate-based alerts, lower cost, faster |
| Log shipper | Filebeat | Vector | Elastic-native stack, simpler configuration | Higher throughput, WASM transforms, multi-output |
| Compressor | gzip | zstd | Ubiquitous support, good compression | Better compression ratio, faster decompression |
| Search engine | Elasticsearch | Loki | Rich query language, aggregations, dashboards | Lower cost, simpler operations, label-based queries |
| Buffer | Kafka | Redpanda | Mature ecosystem, wide integration | Lower latency, simpler operations, Kafka-compatible |

### How to Use the Matrix

For each decision, identify your primary constraint (cost, latency, durability, query speed) and choose the option that aligns with it. Revisit these decisions quarterly as your data volume and business requirements evolve.

## CAP Theorem Applied to Logging

The CAP theorem states that a distributed data system can guarantee at most two of consistency, availability, and partition tolerance. Logging systems make the same trade-off.

**CP (Consistency + Partition Tolerance)**: Elasticsearch with quorum-based writes. A write is acknowledged only after a majority of replicas confirm. If the cluster partitions, writes to the minority side are rejected. This guarantees that all replicas see the same data but sacrifices availability during a network partition.

**AP (Availability + Partition Tolerance)**: Kafka with asynchronous replication. A write is acknowledged as soon as the leader receives it; replication to followers happens in the background. If the partition splits, both sides accept writes. When the partition heals, the two sides merge, which may produce conflicts or duplicates.

**CA (Consistency + Availability)**: a single-node syslog server. No partition tolerance — if the node fails, the system is unavailable. This is acceptable for small deployments but not for production systems at scale.

Most production logging systems use AP: they accept the possibility of losing a small number of events during a partition in exchange for always being able to accept new writes.

## Failure Modes

A logging system must survive its own failures without impacting the applications it monitors. Here are eight common failure modes and how to mitigate them.

### Failure Mode 1: Disk Full

The most common logging failure. A runaway process generates logs at 100 MB/s, fills the disk, and crashes the application (because the application cannot write to its own data directory on the same disk).

**Mitigation**: always put log files on a separate partition or volume. Configure disk usage alerts at 70 percent, 80 percent, and 90 percent. Use log rotation with aggressive max size limits. Use a separate mount point for the shipper's disk queue.

### Failure Mode 2: Indexer Saturation

The Elasticsearch cluster cannot keep up with the ingestion rate. Write queues fill up, the cluster starts rejecting requests, and the Kafka consumer lag grows without bound.

**Mitigation**: monitor Kafka consumer lag. Set up auto-scaling for the indexer (or provision for 2x peak). Use separate indices per service with different replica counts — critical services get more replicas, debug services get fewer. If saturation is sustained, increase sampling or reduce the number of indexed fields.

### Failure Mode 3: Shipper Crash

The log shipper process crashes and does not restart. Applications continue running and writing logs to files, but logs are no longer shipped to the central system. The gap is discovered hours later when an engineer tries to search for recent events and finds nothing.

**Mitigation**: run the shipper as a Kubernetes DaemonSet or systemd service with `Restart=always`. Monitor shipper uptime with a health check endpoint. Set up an alert if logs from a node are not appearing in the central system within 5 minutes.

### Failure Mode 4: Clock Skew

Log timestamps are generated by the application's local clock. If the clock on different servers is not synchronized, events from different services appear in the wrong order. A database query log may appear 2 seconds before the request that triggered it.

**Mitigation**: run NTP on every node with frequent synchronization (every 60 seconds, not the default 5 minutes). Log the time source and offset in each event to detect skew:

```json
{"timestamp": "2026-07-02T14:30:00Z", "ntp_offset_ms": 150, "event": "db_query"}
```

If NTP detects an offset greater than 100ms, emit a warning log and correct the clock. For sub-millisecond accuracy, use a PTP (Precision Time Protocol) appliance.

### Failure Mode 5: Cost Explosion

An engineering team adds a new field to all log events without realizing the impact on storage costs. The average event size doubles, storage costs double, and the monthly bill increases by $50,000.

**Mitigation**: enforce a log budget per service. Each service has a quota of log volume (GB/day). When the quota is exceeded, the shipper drops events from that service. Review log schema changes in code review. Use dynamic field mapping in Elasticsearch to reject unknown fields.

### Failure Mode 6: Kafka Broker Failure

A Kafka broker fails. The partitions hosted on that broker become unavailable. Producers that are configured with `acks=1` continue writing to the remaining brokers. Producers configured with `acks=all` block until the failed broker is replaced.

**Mitigation**: configure Kafka with `min.insync.replicas=2` and `replication.factor=3`. This ensures that even if one broker fails, the other two replicas are available. The producer should use `acks=all` for critical log streams and `acks=1` for non-critical streams.

### Failure Mode 7: Elasticsearch Master Node Failure

The Elasticsearch master node fails. The cluster elects a new master from the remaining master-eligible nodes. During the election (30-60 seconds), the cluster is in read-only mode. No writes are accepted.

**Mitigation**: run 3 dedicated master nodes. Use `discovery.zen.minimum_master_nodes: 2` to prevent split-brain. Monitor cluster health and alert on `status: yellow` (one replica missing) and `status: red` (one primary missing).

### Failure Mode 8: Network Partition Between Shipper and Aggregator

The shipper cannot reach the aggregator. The shipper's disk queue fills up. When the queue is full, the shipper must drop events.

**Mitigation**: size the disk queue for 10-30 minutes of peak traffic. Monitor the queue size and alert when it exceeds 50 percent. Use a redundant network path or a second aggregator endpoint.

## Anti-Patterns

These practices look reasonable in isolation but cause systemic problems in practice.

### Anti-Pattern 1: Logging Everything at INFO

```python
logger.info("entering loop iteration")
logger.info("value of x is", x=42)
logger.info("exiting loop iteration")
```

Every line of code generates an INFO log. The result is 10 million events per second, storage costs explode, and the important signals are buried in noise.

**Fix**: use INFO for business-relevant events (request lifecycle, external calls, auth decisions, state changes). Use DEBUG for detailed code-path tracing. Configure production to emit only INFO and above by default.

### Anti-Pattern 2: Logging Sensitive Data

```python
logger.info("user_login", password=request.password, credit_card=request.credit_card)
```

Passwords, credit card numbers, session tokens, and PII are logged in plaintext. This is a compliance violation (GDPR, PCI-DSS, HIPAA) and a security incident waiting to happen.

**Fix**: never log sensitive fields. Use a log obfuscation filter at the shipper level as a safety net. Implement a pre-commit hook that scans new code for common patterns like `password` in log arguments. Use a CI pipeline that checks log statements for sensitive parameter names.

### Anti-Pattern 3: No Log Rotation

A log file grows to 500 GB. When the disk fills, the application crashes. The 500 GB file takes 30 minutes to compress and archive. During that downtime, the service is unavailable.

**Fix**: configure log rotation in the application's logging library: max file size 100 MB, max 10 backup files. Test rotation by sending a SIGUSR1 signal to the process and verifying a new file is created.

```python
import logging.handlers
handler = logging.handlers.RotatingFileHandler(
    "app.log", maxBytes=100*1024*1024, backupCount=10
)
```

### Anti-Pattern 4: Single Pipeline for All Logs

A startup logs everything — application logs, system logs, audit logs, access logs, infrastructure logs — into a single Elasticsearch index. When an operator needs to find an application error, the search is slow because the index is 10x larger than needed.

**Fix**: separate logs by purpose into different pipelines. Application logs go to a hot index with 7-day retention. Audit logs go to a separate immutable index with 7-year retention. System logs go to a low-priority index with 3-day retention. This way, each pipeline is tuned for its workload and retention requirements.

### Anti-Pattern 5: No Correlation ID

Each service generates log events independently. When a request fails, you see an error in the payment service but have no way to find the corresponding request in the API gateway logs.

**Fix**: generate a unique correlation (trace) ID at the edge of the system and propagate it through every service via HTTP headers. Log the correlation ID in every event. Use a middleware library (OpenTelemetry, Jaeger) that handles propagation automatically.

```python
import uuid
from flask import g

@app.before_request
def set_correlation_id():
    g.trace_id = request.headers.get("X-Trace-ID", str(uuid.uuid4()))

@app.after_request
def log_request(response):
    logger.info("request_complete", trace_id=g.trace_id, status=response.status_code)
    return response
```

### Anti-Pattern 6: Over-Indexing

Every field in a JSON log event is automatically mapped as `keyword` in Elasticsearch. Over time, the index mapping grows to hundreds of fields, each adding to the index size. A single extra indexed field can increase storage by 10-20 percent.

**Fix**: use explicit mappings with only the fields you query on. Disable dynamic mapping. For fields that are rarely queried, use `"index": false` to store them without indexing.

```json
{
  "mappings": {
    "dynamic": false,
    "properties": {
      "level": { "type": "keyword" },
      "service": { "type": "keyword" },
      "trace_id": { "type": "keyword" },
      "event": { "type": "keyword" },
      "@timestamp": { "type": "date" },
      "duration_ms": { "type": "integer" }
    }
  }
}
```

### Anti-Pattern 7: Blocking on Log Writes

```python
import logging
logging.basicConfig(level=logging.INFO)
logging.info("request started")
```

In this configuration, every `logging.info()` call acquires a global lock and writes to stdout synchronously. At high throughput, the application thread spends more time logging than doing actual work.

**Fix**: use an asynchronous logger. In Python, use `QueueHandler` with a dedicated thread. In Go, use `zap` with the standard config (non-blocking by default). In Java, use `Logback` with `AsyncAppender`.

```python
import logging.handlers
import queue

log_queue = queue.Queue(-1)
queue_handler = logging.handlers.QueueHandler(log_queue)
logger.addHandler(queue_handler)
```

### Anti-Pattern 8: No Monitoring of the Logging System

The logging system fails silently. Engineers search for logs and find nothing, but they do not know whether the search returned no results or the system is down. Hours pass before someone realizes the logging system stopped working.

**Fix**: treat the logging system as critical infrastructure. Monitor its health independently: shipper uptime, Kafka consumer lag, Elasticsearch cluster health, and indexing rate. The monitoring system for the logging system must use a separate channel from the logging system itself.

### Anti-Pattern 9: Using the Same Index for All Environments

Development, staging, and production logs are written to the same index. Developers accidentally query production logs during development, and operators cannot tell which environment a log came from.

**Fix**: use separate indices per environment: `logs-production-*`, `logs-staging-*`, `logs-development-*`. Apply different retention policies: production logs retain for 30 days, staging for 7 days, development for 1 day. Use different Elasticsearch clusters for production and non-production to prevent noisy neighbors.

### Anti-Pattern 10: Storing Everything in Hot Tier

All logs are stored on SSDs with 2x replication regardless of age. The cluster grows to 500 nodes, and the monthly cost exceeds $500,000.

**Fix**: implement tiered storage with ILM. Hot tier (SSD) for 7 days. Warm tier (HDD) for 30 days. Cold tier (S3) for 365 days. The hot tier should hold only the data that needs sub-second query latency. Older data can tolerate 5-10 second query latency from S3.

### Anti-Pattern 11: No Structured Error Handling

Errors are logged as plaintext strings with no structure:
```python
logger.error(f"Failed to process order {order_id}: {e}")
```

This makes it impossible to search for all errors of a specific type or to count error rates by category.

**Fix**: use structured error fields:
```python
logger.error("order_processing_failed", order_id=order_id, error_type=type(e).__name__, error_message=str(e))
```

### Anti-Pattern 12: Logging in the Hot Path

A database query is logged inside the database driver, which is called 10,000 times per second. The logging library acquires a lock for each call, serializes the message, and writes to disk. The application's throughput drops by 50 percent.

**Fix**: use an asynchronous logger with a ring buffer. The hot path writes to the ring buffer in under 100 nanoseconds. A background thread flushes the buffer. This decouples logging latency from application latency.

## Trade-off Deep Dives

### Synchronous vs. Async Writes

Synchronous writes with `fsync()` guarantee that the log event is on disk before the logger returns. The cost is 1-10ms per write. At 10,000 events per second, synchronous writes consume 10-100 seconds of wall time per second — the application cannot keep up.

Asynchronous writes batch events in memory and flush every 100ms or 1000 events. The application sees sub-microsecond latency. The cost is potential data loss of up to 100ms of events on crash.

The right choice depends on the audit requirements:
- Audit logs: synchronous, fsync, separate file per day
- Application logs: asynchronous, non-fsync, stdout
- Error logs: asynchronous, non-fsync, but with higher priority queue

### Head-Based vs. Tail-Based Sampling

Head-based sampling decides at the beginning of a request whether to log it. It is simple and cheap but misses rare failures that happen to fall in the unsampled window.

Tail-based sampling logs all events for every request, then decides after the request completes whether to store them. It captures all events for failed requests but requires buffering until the request completes, which consumes memory proportional to concurrent request count.

The right choice depends on the request failure rate:
- Failure rate under 1 percent: tail-based sampling captures all failures at low cost (99 percent of requests are discarded after completion)
- Failure rate over 10 percent: head-based sampling is cheaper because tail-based sampling would store most requests anyway
- Unknown failure rate: start with tail-based sampling, measure the cost, then adjust

### gzip vs. zstd for Log Compression

gzip is the default compression algorithm for most logging systems. It compresses JSON logs at 5-10x and decompresses quickly. zstd is a newer algorithm that compresses 6-12x and decompresses 4x faster than gzip.

The trade-off is CPU usage during compression: zstd uses more CPU than gzip for the same compression ratio. On modern hardware, zstd at level 3 compresses faster than gzip at level 6 while achieving better compression.

Recommendation: use zstd at level 3 for log shipping. The 30 percent better compression ratio reduces storage costs by 30 percent, and the faster decompression reduces query latency.

## When NOT to Use a Centralized Logging System

Centralized logging is not always the right answer. There are scenarios where the cost and complexity exceed the benefits.

**Tiny deployments**: a single server running a single application does not need Elasticsearch + Kafka + Filebeat. Use `journalctl` on Linux or the application's built-in log file with rotation. Add centralized logging only when you have more than 5 servers or multiple engineers who need access to logs.

**Extreme throughput, low-value logs**: a CDN edge node generates billions of log lines per day for every HTTP request. The signal-to-noise ratio is extremely low — most log lines are 200 OK responses. Instead of indexing everything, use statistical sampling (1:100,000) or metrics-based monitoring and keep raw logs in compressed object storage for forensics only.

**Real-time systems with strict latency SLAs**: a high-frequency trading platform cannot tolerate the latency overhead of structured JSON serialization and fsync. These systems use in-memory ring buffers that are occasionally dumped to disk for post-mortem analysis. Live debugging is done through metrics and traces, not logs.

**Disconnected environments**: offline or air-gapped systems (military, industrial control) cannot send logs to a centralized service. Logs are stored locally and analyzed on-site. In these environments, log rotation and local indexing are the primary tools.

**Compliance-constrained environments**: some regulations prohibit sending logs outside the region or country. In these environments, each region must run its own independent logging stack. Cross-region query is not allowed.

**Extremely low-budget teams**: a team of 2 engineers running 3 services on a $100/month budget should not spend $300/month on logging. Use `journald` and `grep` until the team grows to 5+ engineers.

### Decision Tree for Choosing a Logging Strategy

Use this decision tree to determine the right logging strategy for your system:

1. How many servers do you have?
   - 1-5 servers: use local logging with rotation. No centralized system needed.
   - 5-50 servers: deploy a lightweight centralized system (Elasticsearch + Filebeat, or Loki + Promtail).
   - 50+ servers: deploy a full pipeline (Kafka + Elasticsearch + S3) or use a managed service.

2. How much log data do you generate per day?
   - Under 10 GB/day: any solution works. Use managed service for simplicity.
   - 10-100 GB/day: self-hosted Elasticsearch or managed service.
   - 100 GB - 1 TB/day: self-hosted with Kafka buffering or managed service.
   - Over 1 TB/day: custom pipeline with Kafka + S3 + Presto, or managed service if cost allows.

3. What is your query latency requirement?
   - Sub-second: Elasticsearch hot tier on SSDs.
   - 1-10 seconds: Elasticsearch warm tier on HDDs, or Loki.
   - 10+ seconds: SQL queries on Parquet files in S3.

4. What is your retention requirement?
   - 7 days: hot tier only.
   - 30 days: hot + warm tier.
   - 1 year+: hot + warm + cold (S3) tier.
   - 7 years: hot + warm + cold + frozen (S3 Glacier) tier.

5. What compliance requirements do you have?
   - None: standard logging with 30-day retention.
   - SOC 2: audit logging with 1-year retention and access control.
   - HIPAA: audit logging with 6-year retention, encryption at rest and in transit, access controls.
   - PCI-DSS: audit logging with 1-year retention, no cardholder data in logs, quarterly log reviews.
