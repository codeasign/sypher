The trade-offs page laid out the theory. This page examines how four major technology companies apply stateless and stateful design in their production systems, what broke, and what they learned.

## Netflix -- Stateless Compute, Stateful Data Layer

Netflix operates one of the largest stateless compute fleets in the world. Every customer request can be handled by any server instance, in any region, at any time.

### The Stateless API Layer

Netflix's API gateway processes 2 billion requests per day across 200+ microservices. Every service instance is stateless. Sessions are externalized to EVCache (Netflix's in-house Memcached fork, built on Redis). User identity, viewing history, and playback state all live in EVCache or Cassandra.

**Why stateless:** Netflix deploys multiple times per day across 1,000+ microservices. Stateless instances allow instant rollback and canary deployments. A bad deploy is rolled back in under 60 seconds by flipping the load balancer to the previous version. No session state is ever pinned to a specific instance.

**How it scales:** During peak hours (Friday evening in the US), Netflix auto-scales from 5,000 to 20,000 application instances across AWS regions. Each new instance registers with the load balancer and starts handling requests immediately. The EVCache layer is partitioned across hundreds of nodes, replicated within and across regions.

**Key learning:** Netflix's Chaos Monkey deliberately kills stateless instances to verify that the system survives. Because instances hold no state, killing any instance causes zero data loss. The team has internalized that instance death is normal, not exceptional.

### The Stateful Recommendation Engine

Netflix's recommendation pipeline is stateful at the model level. The ML models that generate personalized recommendations are computed in batch jobs and loaded into memory on dedicated servers. These servers are stateful: they hold model parameters and feature embeddings in memory.

**Trade-off:** The recommendation servers are harder to deploy and scale than the stateless API layer. Model updates require careful shadow-deployment and A/B testing. But the latency gain is enormous: a memory lookup of the model takes under 1ms versus 50-100ms for a remote model inference call.

## Uber -- Stateless Rides, Stateful Payments

Uber's ride-matching system is a carefully balanced mix of stateless and stateful design.

### The Stateless Dispatch Layer

The ride request API is stateless. Every request contains all the information needed: pickup location, destination, rider ID. The dispatch service validates the request, finds nearby drivers, and sends the request to the appropriate stateful driver-matching system. If a dispatch instance crashes mid-request, the rider retries. The retry lands on any healthy instance.

**Why stateless:** Ride requests come in unpredictable bursts (rain, events, rush hour). Uber auto-scales dispatch instances aggressively. Stateful dispatch would mean session pinning during the critical few seconds between request and driver assignment.

### The Stateful Driver GPS Service

Uber's driver location service is stateful. Each driver's GPS socket is pinned to a specific server that maintains the real-time location in memory. The server broadcasts the driver's location to nearby riders every 2-3 seconds.

**Why stateful:** Relaying a driver's GPS position through a database would add 20-50ms of latency per position update. At 2-second intervals, the database would handle 5,000 writes per second per driver (for a city with 10,000 active drivers, that is 5 million writes per second). In-memory state is the only way to achieve the throughput and latency required.

**How failover works:** Each driver connection is replicated to a backup server. If the primary server fails, the backup promotes itself and the driver reconnects via WebSocket. The failover takes 1-3 seconds, during which the driver's last known position is shown with a "reconnecting" indicator.

## Discord -- Stateful for Real-Time, Stateless for REST

Discord runs one of the largest real-time communication systems in the world, handling 10 million+ concurrent voice and text users.

### The Stateful Voice Server

Discord's voice infrastructure is heavily stateful. Each voice channel runs as a dedicated process that holds the audio mixing state, connected user list, and encryption keys in memory. Voice packets must be mixed and forwarded in under 20ms to maintain call quality. A database round trip per packet would make this impossible.

**How state is managed:** Each voice server handles a fixed number of channels (typically 200-500). When a user connects to a voice channel, the gateway service assigns them to a specific voice server based on the server's current load. The voice server keeps all state in shared memory within the process. Voice servers are not replicated -- if a server crashes, all users in those channels must reconnect to a new server.

**Why no replication:** Voice state is ephemeral. When a voice call ends, the state is meaningless. If a voice server crashes mid-call, the users reconnect and a new mixing context is created. The cost of replicating tens of thousands of audio streams per second outweighs the benefit of slightly faster reconnection.

### The Stateless REST API

Discord's REST API for sending messages, managing guilds, and fetching history is stateless. Every request goes through a load balancer to any available API server. The API server fetches data from Cassandra or ScyllaDB.

**Why stateless:** Discord's API traffic is highly spiky. A large server (like the 2022 Ukraine solidarity server) can grow from 0 to 200,000 members in hours. Stateless API servers can be provisioned and registered in minutes.

## Cloudflare -- Stateless Edge, Stateful Workers

Cloudflare operates a global edge network where every request is stateless at the proxy layer but Workers (serverless functions) can be stateful.

### The Stateless CDN

Cloudflare's core CDN is the definition of stateless. A request for `image.png` arrives at any of Cloudflare's 330+ data centers. The edge server checks its local cache. If present, it serves the cached copy. If not, it fetches from the origin, caches it locally, and serves it. The edge server retains nothing about the request after the response is sent.

### Stateful Workers with Durable Objects

Cloudflare Workers can opt into stateful behavior via Durable Objects, which provide strongly consistent key-value storage co-located with the worker. A Durable Object is a single-instance stateful entity that processes requests in sequence.

**Use case:** Real-time collaboration features (WebSocket connections, shared counters, multiplayer game state). The Durable Object holds the authoritative state in memory on a specific edge location. All operations on that state go through the single Durable Object instance, ensuring strong consistency.

**Trade-off:** A Durable Object is pinned to one location. If that location goes down, the state is lost unless it was persisted to Durable Object storage, which adds latency.

## Open Source Implementations

### Apache Kafka -- Stateful Message Broker

Kafka is a stateful system. Each broker holds a subset of partitions in local log files. The partition leader handles all writes and reads for that partition. State (the log) is persisted to disk and replicated to followers.

**Trade-off:** Kafka achieves 5-10x the throughput of a stateless message queue because it avoids network hops for state storage. However, scaling Kafka requires careful partition management, rebalancing, and broker coordination.

### Redis -- Stateful Cache

Redis is a stateful key-value store. All data lives in memory on a single node (or partitioned across a cluster). Writes are acknowledged when the data is in memory; persistence is asynchronous.

**Why stateful:** Every Redis operation is a memory access, achieving sub-millisecond latency. A stateless cache would need to read from disk or a remote store for every miss, adding 10-100ms of latency.

### ZooKeeper -- Stateful Coordination Service

ZooKeeper is a strongly consistent stateful service for cluster coordination. It maintains the cluster state tree in memory on every node, synchronized via ZAB (ZooKeeper Atomic Broadcast) consensus.

**Why stateful:** Coordination requires strongly consistent reads and writes. A stateless coordination service would need to fetch state from a remote database for every read, adding latency and failure risk exactly when a cluster is under stress.

## Cloud Provider Implementations

### AWS

**Stateless options:** AWS Lambda (stateless function execution), Fargate (stateless container orchestration), ELB with target groups (stateless load balancing).

**Stateful options:** ElastiCache (managed Redis/Memcached), DynamoDB Accelerator (DAX, in-memory cache), ElastiCache for Redis with replication across AZs.

**Cost trade-off:** Lambda is billed per invocation and is inherently stateless. RDS Proxy maintains a stateful connection pool so that Lambda functions can share database connections without overwhelming the database. The Proxy adds $15-20/month but reduces database connection overhead by 100x.

### GCP

**Stateless options:** Cloud Run (stateless container execution), HTTP(S) Load Balancer (stateless request distribution), Cloud Functions.

**Stateful options:** Memorystore (managed Redis), Firestore (document DB with strong consistency), Bigtable (stateful wide-column store).

### Azure

**Stateless options:** Azure Functions, Azure Container Apps, Azure Front Door (stateless CDN).

**Stateful options:** Azure Cache for Redis, Cosmos DB with consistency levels ranging from strong to eventual.

## Lessons Learned

1. **Stateless is simpler at scale, not at startup.** For the first 1,000 users, local session state on a single server is easier and faster than setting up Redis. Migrate to stateless when you add a second server or when deployment frequency increases.

2. **The best architecture is a hybrid.** Every major company uses both stateless and stateful services. The trick is knowing which part of your system needs which. Latency-sensitive and coordination-heavy components are stateful. Everything else is stateless.

3. **Stateful failover is the hardest distributed systems problem.** Netflix can kill instances randomly because they are stateless. Discord accepts that voice server crashes lose active calls. If your stateful service must survive failures without data loss, you need consensus, replication, and careful testing.

4. **Externalizing state does not eliminate state management.** Moving session state from server memory to Redis does not remove the state; it moves it. You now operate a Redis cluster with its own failure modes, capacity planning, and operational needs.

5. **Idempotency is the superpower of stateless systems.** The ability to retry any request without side effects makes stateless architectures resilient. Invest in idempotency keys and retry logic -- they pay for themselves in reduced incident severity.
