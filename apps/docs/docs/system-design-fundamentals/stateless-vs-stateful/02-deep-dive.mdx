## Internal Architecture -- Stateless Service

A stateless service is an exercise in deliberate non-ownership. Every internal component is designed to either avoid holding state or to push state to the boundary of the process.

### The Core Components

**Request Router:** Maps incoming requests to handler functions based on path, method, and headers. No reference to previous requests is stored. The router operates from a static routing table loaded at startup.

**Handler Pool:** A fixed-size thread pool or event loop that processes requests. Handlers are pure functions: given a request and a set of dependencies (database clients, cache clients), they produce a response. Handlers are interchangeable -- any handler can process any request.

**Connection Pool:** A set of pre-established database or cache connections. The pool itself is stateful (it tracks which connections are in use), but this state is internal to the service and invisible to clients. The pool is shared across all handlers.

**Middleware Chain:** A sequence of processing steps (authentication, rate limiting, logging, compression). Each middleware is stateless or pushes state to request-scoped context objects that are discarded after the response.

**Circuit Breaker:** A component that tracks failure rates to downstream services. This is deliberately stateful memory within the stateless paradigm. The circuit breaker state (open, closed, half-open) is local to each instance and is lost on restart -- which is acceptable because the state is a performance optimization, not business data.

## Internal Architecture -- Stateful Service

A stateful service is built around ownership. It holds authoritative state in local memory and manages the lifecycle of that state across connections, crashes, and scale events.

### The Core Components

**Session Map:** A concurrent hash map keyed by session ID. Each entry holds the user's state object. The map is the primary data structure and the source of truth for active sessions. Typical size: tens of thousands to millions of entries.

**State Machine:** For workflows, the state machine tracks which step each session is in. The state machine transitions on events and enforces ordering constraints. It prevents invalid transitions (e.g., charging a credit card twice for the same order).

**Replication Log:** An append-only log of state mutations. The log is shipped to follower nodes for durability. On leader failure, a follower replays the log to reconstruct state. The log is truncated periodically via snapshotting.

**Lease Manager:** A component that assigns ownership of partitions or sessions to specific nodes. Leases have time-to-live values and must be renewed. The lease manager prevents split-brain scenarios where two nodes both believe they own the same state.

**Consensus Module:** For replicated stateful services, the consensus module implements Raft or Paxos to agree on state changes across nodes. This is the most complex component in any stateful system.

## Data Structures and Algorithms

### The Session Hash Map

**What it is:** A concurrent hash map where the key is a session identifier (UUID or token hash) and the value is a session state object. For stateless architectures, this map lives in the external store (Redis hash). For stateful architectures, it lives in the server's heap.

**Why this structure:** Hash maps provide O(1) average lookup and insert, which is critical when every request requires a session lookup. The concurrent variant (e.g., Go's `sync.Map` or Java's `ConcurrentHashMap`) allows reads without blocking writes, which is essential for high-throughput services.

**Time complexity:** Lookup O(1) average, O(n) worst case. Insert O(1) average. Delete O(1) average. Memory: O(sessions * state_size) plus overhead for hash table buckets.

**Space complexity:** For 1 million sessions with 1 KB each, the map consumes approximately 1 GB of heap for data plus 32 MB of hash table overhead.

### The Write-Ahead Log (WAL)

**What it is:** An append-only sequence of state mutations. Every state change is recorded in the WAL before it is applied to the in-memory state. On crash recovery, the WAL is replayed to reconstruct the state.

**Why this structure:** Sequential writes to a file are orders of magnitude faster than random writes on spinning disks and significantly faster on SSDs. Append-only also simplifies concurrency: there is no locking contention on the log.

**Time complexity:** Append O(1). Read (for recovery) O(n) where n is the number of entries since the last snapshot. Snapshotting compacts the log to O(1) for recovery.

### Consistent Hashing Ring

**What it is:** A hash ring that maps keys to nodes in a distributed stateful system. Each node owns a range of the hash space. When nodes join or leave, only the neighboring nodes on the ring need to reassign state.

**Why this structure:** Consistent hashing minimizes state movement during cluster membership changes. A naive hash mod N approach would reshuffle N-1 out of N partitions when a node is added or removed. Consistent hashing moves only K/N partitions, where K is the number of virtual nodes.

### The Lease Mechanism

**What it is:** A time-bound grant of ownership. A node that holds a lease for partition X is the only node that can modify partition X. Leases expire after a configurable TTL and must be renewed.

**Why this algorithm:** Leases prevent split-brain without requiring a consensus round for every operation. A node can act autonomously as long as its lease is valid. The trade-off is the TTL: a short TTL (1 second) means frequent renewal overhead. A long TTL (30 seconds) means slow failover when a node dies.

## The Request/Write/Read Lifecycle

### A Stateless Read Request in 12 Steps

1. Client sends HTTP GET /api/users/42 with an Authorization header containing a JWT.
2. Load balancer receives the request and selects backend instance B3 using least-connections routing.
3. B3 receives the request. The middleware chain decodes the JWT, extracting user ID 42 and role "admin".
4. B3 checks a local in-memory circuit breaker for the downstream user-service. Status: closed (healthy).
5. B3's handler constructs a SQL query: SELECT name, email FROM users WHERE id = 42.
6. B3 acquires a connection from the database connection pool.
7. B3 sends the query to PostgreSQL read replica R2.
8. R2 returns the row.
9. B3 serializes the response as JSON.
10. B3 returns the response to the load balancer.
11. The load balancer forwards the response to the client.
12. B3 releases the database connection back to the pool and forgets everything about this request.

### A Stateful Collaborative Editing Session in 12 Steps

1. Client connects via WebSocket to the editing service. The load balancer hashes the document ID and routes to server S7.
2. S7 creates a session entry in its session map: session_id=abc123, document_id=doc456, user_id=user42.
3. S7 loads the current document state from PostgreSQL into memory: `{version: 104, content: "..."}`.
4. S7 registers the client connection as a subscriber for document doc456. All clients editing doc456 are tracked in a subscriber list.
5. Client sends an operation: `insert("hello", position=42)`.
6. S7 applies the operation to the in-memory document state. The version increments to 105.
7. S7 appends the operation to the WAL: `{op: insert, pos: 42, text: "hello", version: 105, timestamp: ...}`.
8. S7 broadcasts the operation to all other subscribers (clients editing doc456).
9. S7 sends an acknowledgment to the originating client.
10. A separate snapshot thread writes the document state to PostgreSQL every 30 seconds.
11. If S7 crashes, a new server loads the latest snapshot from PostgreSQL and replays the WAL.
12. Clients reconnect via WebSocket, the load balancer routes them to the new server, and editing resumes.

## Consistency and State

### Crash Scenario: Stateless Service

A stateless service crashes in the middle of processing a database write. The database write either completes (if the service sent the query and the database committed it) or does not (if the service crashed before sending). Because the service holds no state, recovery is trivial: the client retries. If the write was idempotent, the retry is safe. If it was not idempotent, the application needs idempotency keys.

### Crash Scenario: Stateful Service

A stateful service crashes with 50,000 active sessions in memory. A new instance must be started and the state must be reconstructed. This involves loading the latest snapshot from the database and replaying the WAL, which can take seconds to minutes depending on the data volume. During recovery, all 50,000 sessions experience downtime.

### Partition Scenario: Network Split

During a network partition, a stateless service behaves differently depending on whether its downstream stores are reachable. If the database is on the same side of the partition, the service continues to work. If the database is on the other side, the service returns errors. No state inconsistency arises because the service holds no state.

A stateful service with active-active partitioning experiences divergence. Nodes on both sides of the partition accept writes for their partitions. When the partition heals, the state must be merged or reconciled. This is the source of the "eventual consistency" guarantee: the system is consistent only after reconciliation.

### Concurrent Writes: Stale Reads in Stateless Systems

In a stateless architecture, reads always go to the database. This means reads always see the latest committed state -- there is no staleness from a server cache. However, if read replicas lag behind the write primary, stale reads are possible. The staleness window equals the replication lag, which can be milliseconds or seconds depending on load.

In stateful architectures, every node holds a copy of the state. Reads from a follower that has not yet received the latest write produce stale values. The staleness window depends on the replication strategy: synchronous replication eliminates staleness but adds latency; asynchronous replication accepts staleness for lower latency.

## Performance Characteristics

### Stateless Service Performance

| Metric | Value | Condition |
|--------|-------|-----------|
| Request latency (p50) | 2-5 ms | In-memory processing, no external calls |
| Request latency (p99) | 10-50 ms | With database call, cache hit |
| Request latency (p999) | 100-500 ms | With database call, cache miss, retry |
| Throughput (single instance) | 10,000-50,000 req/s | 8-core instance, 16 GB RAM |
| Throughput (scaled) | Linear with instances | Limited by downstream DB capacity |
| Memory per instance | 50-500 MB | No session state stored locally |
| Startup time | 1-5 seconds | No state to load |

### Stateful Service Performance (with Replication)

| Metric | Value | Condition |
|--------|-------|-----------|
| Request latency (p50) | 100-500 micros | In-memory state hit |
| Request latency (p99) | 1-5 ms | With WAL append |
| Request latency (p999) | 10-50 ms | With synchronous replication (Raft) |
| Throughput (single instance) | 100,000-500,000 op/s | Pure in-memory, no persistence |
| Throughput (replicated) | 50,000-200,000 op/s | Limited by Raft consensus |
| Memory per instance | 1-100 GB | Proportional to active sessions |
| Startup time | 10-300 seconds | Snapshot load + WAL replay |

## Configuration and Tuning

### Stateless Service Tuning

**Max idle connections in pool:** Controls how many database connections each instance keeps open. Safe default: 10-20. Too low: requests queue waiting for connections. Too high: database runs out of connections.

**Keep-alive timeout:** How long an idle HTTP connection stays open. Safe default: 60 seconds. Too short: frequent reconnections increase latency. Too long: memory leaks from stale connections.

**Request timeout:** Maximum time a request can take before the service returns a 504. Safe default: 30 seconds. Too short: legitimate slow requests fail. Too long: resource exhaustion from stuck handlers.

**Circuit breaker threshold:** Number of consecutive failures before the circuit opens. Safe default: 5 failures in a 10-second window. Too sensitive: false positives under transient load. Too lenient: cascading failures propagate.

### Stateful Service Tuning

**WAL fsync interval:** How often the WAL is flushed to disk. Safe default: 1 flush per write (maximum durability). Tuning: set to batch 100 writes per flush for 10x throughput at the cost of losing the last batch on crash.

**Replication factor:** Number of copies of each partition. Safe default: 3. Too low (1): no durability on node failure. Too high (5): write latency increases with each additional replica.

**Snapshot interval:** How often the in-memory state is snapshotted to persistent storage. Safe default: every 30 seconds or every 10,000 mutations, whichever comes first. Too frequent: disk I/O overhead. Too rare: slow recovery (more WAL entries to replay).

**Lease TTL:** How long a node claims ownership of a partition without renewing. Safe default: 5 seconds. Too short: high renewal overhead. Too long: slow failover when a node dies.

The architecture page connects these internals into production-grade deployments with load balancers, caches, and multi-region topologies.
