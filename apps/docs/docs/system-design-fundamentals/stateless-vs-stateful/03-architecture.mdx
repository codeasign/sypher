## Naive Implementation

### Phase 1: 1,000 Users -- One Server with Local State

A startup launches a SaaS product with two servers behind a round-robin load balancer. Each server stores user sessions in local memory. The code is simple: `Map<String, Session> sessions = new HashMap<>()`. It works. Deployments happen at midnight. No one notices brief downtime because the user count is low.

**What breaks at 1,000 users:** A user might be logged in on Server A, then a page refresh lands on Server B, which has no session for that user. The user sees a 401 error and must log in again. This happens to roughly 50% of navigation actions under round-robin routing.

### Phase 2: 100,000 Users -- Sticky Sessions Introduce Ops Pain

The engineering team adds sticky sessions to the load balancer. Now user 42 always hits Server A. The 401 problem is solved. But new problems emerge. Rolling deployments require session drain: each server must be told to stop accepting new sessions, then wait for existing sessions to expire, which takes minutes. If the server crashes mid-deployment, all its sessions are lost. Server A gets a traffic spike and runs out of memory because it carries 40% of all sessions (hash distribution is uneven).

**What breaks at 100,000 users:** Uneven hash distribution causes memory hot spots. One server runs at 90% memory while another is at 40%. Rolling deployments require session drain times of 5-15 minutes. A server crash loses 30,000-40,000 active sessions simultaneously.

### Phase 3: 10 Million Users -- Externalized Session Store

The team moves session state to a dedicated Redis cluster. Application servers now hold zero state. The load balancer goes back to round-robin. Any server can handle any request. Deployments are instant: kill a server, start a new one, and sessions survive because they live in Redis. Memory utilization across servers is perfectly balanced. The system scales horizontally.

## Integration Patterns

### Pattern 1: Stateless API with Relational Database

The most common integration. The API service is stateless. Every request loads user context from Redis and queries PostgreSQL for data. Writes go through an ORM or query builder. Read replicas handle the bulk of read traffic.

/>

### Pattern 2: Stateless with Distributed Cache

Cache-aside (lazy loading) pattern. The service checks a distributed cache (Redis/Memcached) before hitting the database. Misses populate the cache. The cache improves read latency from 10-50ms (database) to 1-3ms (cache hit). The service remains stateless because the cache is external.

**Key design decision:** Cache TTL determines staleness. A 60-second TTL means stale data for up to 60 seconds after a write. For data that must be read-after-write consistent, bypass the cache or invalidate on write.

### Pattern 3: Stateful Service with Message Queue Backlog

A stateful order processing service maintains in-memory order queues partitioned by region. When a new order arrives, the service appends it to the correct region queue, processes it, and writes the result to the database. If the service crashes, the in-memory queues are lost. The message queue (Kafka) serves as the durable backlog: the service can replay unprocessed messages from Kafka after recovery.

### Pattern 4: Stateless with External API Integration

A stateless service that calls an external API (Stripe, Twilio, OpenAI) on behalf of the user. The service holds no state about the external API call beyond its own database records. Retries are safe because the external API provides idempotency keys. The service scales horizontally because any instance can make the call.

**Critical detail:** The external API key is stored in a secrets manager, not in the application code or a local config file. Every instance fetches the key from the secrets manager at startup and caches it locally for the duration of the key's validity period.

## Multi-Region and Global Scale

For global deployments, stateless architectures shine because state is externalized. Application servers are deployed in every region. Each region's servers talk to a local Redis cluster and local database read replicas. The write primary lives in one region. Cross-region replication keeps other regions eventually consistent.

## Capacity Planning

### Example: Stateless API at 10 Million Daily Active Users

**Traffic assumptions:**
- 10 million DAU
- Each user makes 10 requests per day on average
- Peak hour handles 20% of daily traffic: 2 million requests per hour = 555 requests per second
- Peak minute within peak hour handles 2x the average: 1,110 requests per second

**Application server capacity:**
- One stateless instance handles 2,000 requests/second (8-core, 16 GB)
- Instances needed: 1,110 / 2,000 = 0.55, round up to 2 for redundancy
- With N+2 redundancy (handle one failure, one maintenance): 4 instances
- Each instance: 50-500 MB memory (session data is in Redis, not local)

**Redis cluster capacity:**
- 10 million sessions, each session 2 KB
- Total session data: 20 GB
- Redis replication factor: 2 (1 primary, 1 replica per shard)
- Total Redis memory needed: 40 GB
- Shard count: 6 shards of 8 GB each (allowing headroom)

**Database capacity:**
- Write throughput: 555 writes/second average, 1,110 peak
- A single PostgreSQL primary handles 10,000 writes/second comfortably
- Read throughput: 555 reads/second average (assuming 50/50 read/write)
- Three read replicas each handle 1,000 reads/second
- Storage: estimate 1 KB per user record = 10 GB. With indexes: 20 GB

### Example: Stateful Game Server at 100,000 Concurrent Players

**Traffic assumptions:**
- 100,000 concurrent players
- Each player sends 20 position updates per second
- Total writes per second: 2,000,000
- Each write: 100 bytes (player ID, x, y, z, timestamp)

**Server capacity:**
- One stateful instance handles 50,000 writes/second with WAL append
- Instances needed: 2,000,000 / 50,000 = 40 instances
- Each instance manages 2,500 players
- Memory per instance: 2,500 players x 5 KB state = 12.5 MB, plus 500 MB overhead = ~512 MB

**Replication:**
- WAL is shipped to 2 follower nodes per partition
- Network bandwidth per instance: 2,000,000 writes x 100 bytes = 200 MB/s aggregate
- Per instance: 200 MB/s / 40 = 5 MB/s write out, 10 MB/s replication out (to 2 followers)

## Deployment and Operations

### Zero-Downtime Deployment for Stateless Services

1. Start new instances with the updated code alongside existing instances.
2. Wait for health checks to pass.
3. Register new instances in the load balancer.
4. Deregister old instances from the load balancer.
5. Wait for in-flight requests to drain (typically 30-60 seconds).
6. Terminate old instances.

The entire process takes 2-5 minutes with zero user impact because sessions live in the shared Redis store, not on the application servers.

### Zero-Downtime Deployment for Stateful Services

1. Update follower nodes one at a time (no leader election needed).
2. Step down the leader (triggers election of an updated follower).
3. Update the old leader.
4. The cluster has one node out of sync momentarily.

This process requires careful orchestration. If the leader steps down but no updated follower is ready, the cluster selects an outdated leader and must roll forward.

### Monitoring

For stateless services, monitor:
- Request latency (p50, p99, p999)
- Error rate (5xx responses)
- Redis connection pool utilization
- Database connection pool utilization
- Instance count

For stateful services, monitor everything above plus:
- Session count per instance
- Memory utilization (session map size)
- WAL write latency
- Replication lag
- Leader election frequency
- Lease renewal success rate
