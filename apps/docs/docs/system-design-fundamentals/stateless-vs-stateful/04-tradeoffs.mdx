The architecture page showed how stateless and stateful systems are deployed. This page maps the hard choices: when to use each approach, what breaks under stress, and how to avoid common mistakes.

## The Core Tension

**Stateless services trade per-request latency for operational simplicity and horizontal scalability. Stateful services trade operational complexity for the lowest possible latency and the ability to own authoritative state in memory.**

This is not a ranking. One is not better than the other. They are tools for different jobs. The wrong choice for your workload burns money, time, and engineering morale.

## Decision Matrix

| Decision Point | Stateless | Stateful | Choose Stateless When | Choose Stateful When |
|---|---|---|---|---|
| Session data size | Stored externally (Redis) | Stored in server heap | Sessions exceed 10 GB total | Sessions fit in server memory (under 2 GB per instance) |
| Session access latency | 2-5 ms (network round trip) | 0.1-0.5 ms (local memory) | 10 ms latency is acceptable | Under 1 ms latency is required |
| Deployment frequency | Multiple times per day | Once per day or less | You deploy frequently (CI/CD) | You deploy infrequently or use blue-green |
| Traffic pattern | Predictable or spiky | Predictable with slow growth | Auto-scaling is needed | Fixed capacity is sufficient |
| State durability requirement | Low (state can be reconstructed) | High (state is authoritative) | Session data is ephemeral | Session data is the source of truth |
| Consistency requirement | Strong consistency (external DB) | Eventual consistency (local state) | Reads must be strongly consistent | Stale reads are acceptable |
| Cross-instance coordination | Not needed (share nothing) | Required (consensus protocol) | Service is read-heavy or CRUD | Service requires stateful coordination |
| Operatonal maturity | Low (simple operations) | High (requires SRE expertise) | Small team or limited ops | Dedicated platform/SRE team |

## CAP Theorem Positioning

Stateless services sit on the CA side of the CAP theorem: they prioritize Consistency and Availability, and they tolerate Partitions by failing when the database is unreachable. A stateless service that cannot reach its database returns errors. It does not serve inconsistent data.

Stateful services sit on the AP side: they prioritize Availability and Partition tolerance, and they accept Eventual Consistency. A stateful service that is partitioned continues to serve requests from local memory, even if that memory diverges from other nodes.

## Failure Modes

### Failure 1: Redis Cluster Failure Brings Down All Stateless Instances

**What happens:** The shared Redis cluster that stores all session data goes down. Every application server cannot authenticate users. Every request returns 401 or 503. The entire application is unavailable.

**Blast radius:** 100% of users. The application is completely down.

**Why it happens:** The Redis cluster was provisioned with a single primary and no failover replicas. A memory spike due to a traffic surge triggered the OOM killer on the primary node.

**Detection:** Monitoring alerts for Redis connection failure rate. The alert fires within 30 seconds of the first connection timeout.

**Recovery:** Restart the Redis primary and load the RDB/AOF backup. Recovery time: 2-5 minutes for a 20 GB dataset. After recovery, all sessions are restored from the last checkpoint.

**Prevention:** Deploy Redis with at least one replica and automatic failover (Redis Sentinel or Redis Cluster). Use Redis persistence (AOF with fsync every second). Provision memory with 50% headroom above peak usage.

### Failure 2: Stateful Node Crash Causes Mass Session Loss

**What happens:** A stateful server hosting 40,000 sessions crashes. All 40,000 users are logged out and must re-authenticate. If the session contained shopping cart data, those carts are lost.

**Blast radius:** 40,000 users (one server's worth of sessions).

**Why it happens:** The server ran out of memory because the session map grew beyond the heap size. The JVM OOM killer terminated the process. Since session state was only in local memory with no replication, all sessions vanished.

**Detection:** Load balancer health check fails. Number of active sessions on the server drops to zero. Support tickets spike from users reporting "lost my cart."

**Recovery:** Start a new instance. The new instance has no sessions. Users must log in again. Lost carts must be handled by customer support.

**Prevention:** Set memory limits per instance and implement session replication to backup nodes. Use a shared session store (hybrid pattern) instead of purely local state.

### Failure 3: Partition Causes State Divergence in Stateful Cluster

**What happens:** A network partition splits a 5-node stateful cluster into two groups: a 3-node majority and a 2-node minority. The majority continues to operate and elects a leader. The minority cannot reach the leader and starts accepting writes independently. When the partition heals, the 2-node minority has data that conflicts with the majority.

**Blast radius:** Depends on the conflict resolution strategy. Best case: last-writer-wins discards minority data. Worst case: manual reconciliation is required.

**Detection:** Vector clock comparisons show concurrent updates. Replication lag metrics show a gap. Data integrity checks surface inconsistencies.

**Recovery:** Apply conflict resolution strategy (last-writer-wins, CRDT merge, or manual reconciliation). For LWW, the minority's writes are silently discarded.

**Prevention:** Use a consensus protocol (Raft, Paxos) that requires a majority for writes. The minority nodes will reject writes during the partition, preventing divergence but reducing availability.

### Failure 4: Sticky Session Imbalance Causes Cascading Overload

**What happens:** A load balancer with sticky sessions hashes user IDs to 10 backend servers. A flash sale event causes 80% of traffic to hash to 3 servers. Those 3 servers hit 100% CPU and start dropping connections. The load balancer retries failed requests, which land on the same servers (sticky), making the overload worse.

**Blast radius:** The 3 overloaded servers affect 80% of users. The remaining 7 servers are underutilized.

**Detection:** CPU alert on the 3 servers. Error rate increases. Load balancer shows uneven connection distribution.

**Recovery:** Remove sticky session configuration. Move to a shared session store. In the short term, add more servers and manually redistribute sessions.

**Prevention:** Avoid sticky sessions for production workloads. Use a shared session store instead. If sticky sessions are unavoidable, use a consistent hash ring with virtual nodes to improve distribution.

### Failure 5: Stateless Instance Startup Thundering Herd

**What happens:** An auto-scaling event adds 20 new stateless instances simultaneously. All 20 instances connect to the Redis cluster and database at the same time. The Redis cluster sees a 20x spike in connection requests. The database connection pool is exhausted. Both Redis and the database experience latency spikes.

**Blast radius:** The new instances cannot establish connections and fail health checks. Existing instances are not affected. The auto-scaling event fails to improve capacity.

**Detection:** Auto-scaling triggers, but error rate does not decrease. New instances show as unhealthy. Redis and database connection metrics show spikes.

**Recovery:** Implement connection pool warm-up in the instance startup sequence. Stagger instance startup with jitter (100ms to 5 seconds random delay).

**Prevention:** Configure connection pool minimum sizes to handle expected scaling events. Use connection pooling proxies (PgBouncer, Redis Sentinel) to buffer connection spikes.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Stateful with No Replication

A server stores session state in a local HashMap. No replication. No backup. When the server crashes, all sessions are lost. The team accepts this as "acceptable data loss" until a crash during Black Friday costs $200,000 in abandoned carts.

**Fix:** Use a shared session store, or at minimum replicate sessions to a secondary node.

### Anti-Pattern 2: Sticky Sessions as a Scaling Strategy

A team deploys sticky sessions thinking it is a "free" way to handle state. They discover that sticky sessions prevent horizontal scaling, complicate deployments, and cause uneven load distribution. The load balancer is now a bottleneck and a single point of failure.

**Fix:** Remove sticky sessions. Externalize session state to a shared store.

### Anti-Pattern 3: Stateless for Everything

A team dogmatically insists on stateless design for every service. They build a real-time collaboration tool with stateless services. Every keystroke requires a round trip to the database. The latency makes the tool unusable.

**Fix:** Use stateful design for the real-time editing path. Use stateless for the document listing and metadata APIs.

### Anti-Pattern 4: Shared State as a Single Point of Failure

A team externalizes all session state to a single Redis instance. The Redis instance goes down, and the entire application goes down. The team had assumed that "externalizing state" automatically makes the system resilient.

**Fix:** Deploy Redis with replication and automatic failover. Use a multi-AZ deployment. Consider a managed service (ElastiCache, Memorystore) with built-in HA.

### Anti-Pattern 5: Session State in the Database

A team stores session state in a relational database because "it is already there." Every request requires a SELECT query against a large sessions table with a B-tree index. At 10,000 requests/second, the database spends 40% of its CPU on session lookups.

**Fix:** Use Redis or Memcached for session state. Sessions are ephemeral key-value data, not relational data. Databases are optimized for relational queries, not high-throughput key lookups.

### Anti-Pattern 6: Ignoring Session Replication Lag

A team deploys a stateful service with three replicas and synchronous replication. They assume synchronous replication means zero data loss. During a partition, the quorum is lost, and writes are rejected. The team blames the stateful design.

**Fix:** Understand the trade-off between replication latency and availability. Synchronous replication provides strong consistency but reduces availability during partitions. Asynchronous replication improves availability but accepts data loss on failure.

## When NOT to Use Stateless Design

- When per-request latency must be under 1 millisecond and the network round trip to an external store is 2-5ms. Real-time gaming, audio processing, and high-frequency trading need stateful design.
- When the data is inherently session-scoped and cannot be shared across instances. Some compliance requirements mandate that user data stays within a specific jurisdiction, which may preclude externalized state stores.
- When the cost of externalizing state exceeds the cost of managing stateful servers. At very small scales (under 1,000 concurrent users), the operational overhead of running Redis might not be justified.

## When NOT to Use Stateful Design

- When the system must scale horizontally to handle unpredictable traffic spikes. Stateless instances can be added in seconds. Stateful partitions require careful rebalancing.
- When the team has limited operational experience. Stateful systems require expertise in consensus protocols, replication, failover, and data recovery.
- When the deployment frequency is high (multiple times per day). Stateful deployments require careful orchestration to avoid data loss.
- When the business cannot tolerate data loss during a failure. Stateless systems with an external database lose zero data on application server failure. Stateful systems with local memory lose data on crash unless replicated.
