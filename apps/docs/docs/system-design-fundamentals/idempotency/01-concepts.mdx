# Core Concepts

Idempotency is the property that a single operation or a sequence of identical operations produces the same result. When you press the button on a vending machine twice, you expect one soda and one charge, not two. The button press operation is idempotent: the second press has no additional effect beyond the first.

---

## Why Idempotency Matters

A customer browsing an e-commerce site finds a product, adds it to the cart, and clicks "Place Order." The page hangs for five seconds. The customer clicks again. Without idempotency, two orders ship and two charges appear on the card. With idempotency, the second click is recognized as a duplicate and the system returns the existing order confirmation without processing a second payment.

This scenario happens every day in production systems. Network timeouts, browser retries, and API client retry logic all generate duplicate requests. Idempotency is the mechanism that makes these duplicates harmless.

The core problem is that distributed systems cannot distinguish between "the request was lost" and "the request was processed but the response was lost." Without idempotency, every retry risks double processing. With idempotency, only the first request is processed; subsequent requests with the same identity return the cached result.

---

## Core Vocabulary

| Term | Definition |
| --- | --- |
| Idempotency | Property where applying an operation multiple times has the same effect as applying it once |
| Idempotency Key | A unique identifier attached to a request that allows the server to detect duplicates |
| Idempotent Operation | An operation that produces the same side effects regardless of how many times it is executed |
| Safe Retry | A retry that is guaranteed not to cause duplicate processing or side effects |
| Deduplication | The process of detecting and discarding duplicate requests |
| Idempotency Window | The time period during which the system retains idempotency keys for duplicate detection |
| Lock | A synchronization primitive that prevents concurrent processing of the same idempotency key |
| Idempotency Store | The persistent storage where idempotency keys and their associated responses are kept |
| Concurrency Control | Mechanisms that ensure correct behavior when multiple requests with the same key arrive simultaneously |
| Optimistic Locking | A concurrency strategy that assumes conflicts are rare and checks for them at write time |
| Version Token | A value attached to a resource that changes on each update, enabling conditional writes |
| PUT Semantics | An HTTP method that replaces a resource entirely, naturally idempotent because the same PUT body always produces the same state |
| POST Semantics | An HTTP method that creates a new resource, naturally non-idempotent because each POST creates a new entity |

## Step by Step: Idempotent Request Processing

The following sequence shows how an idempotent request flows through the system from the moment the client creates the idempotency key to the moment the client receives the final response.

### Step 1: Client Generates an Idempotency Key

The client creates a unique identifier before sending the request. This is typically a UUID version 4 string. The client attaches this key to the request as an HTTP header, typically named `Idempotency-Key` or `Idempotency-Key`.

```python
import uuid

def place_order(order_data):
    idempotency_key = str(uuid.uuid4())
    response = http_client.post(
        "https://api.example.com/orders",
        json=order_data,
        headers={"Idempotency-Key": idempotency_key}
    )
    return response
```

The client stores the idempotency key locally so it can be reused on retry. If the client does not receive a response within the timeout, it retries with the same key.

### Step 2: Server Receives the Request

The server extracts the idempotency key from the request headers. If the key is missing, the server may reject the request outright or process it as a non-idempotent request, depending on the API design. For operations that modify state, a missing key should cause a rejection.

```python
def handle_order_request(request):
    idempotency_key = request.headers.get("Idempotency-Key")
    if not idempotency_key:
        return HTTPResponse(
            status=400,
            body={"error": "Idempotency-Key header is required for this operation"}
        )
    # Continue processing...
```

### Step 3: Server Validates the Key

The server validates the format of the idempotency key. A valid key is typically a UUID string, a string of a specific length, or a string matching a specific pattern. Invalid keys are rejected with a 400 Bad Request response.

```python
import re

def validate_idempotency_key(key):
    uuid_pattern = r"^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$"
    if not re.match(uuid_pattern, key, re.IGNORECASE):
        return False
    return True
```

### Step 4: Server Checks the Idempotency Store

The server queries the idempotency store for the key. The idempotency store is a distributed key-value store that maps idempotency keys to responses. The query must be strongly consistent: if a previous request stored a response, this query must find it.

```python
def check_idempotency_store(key):
    # Strongly consistent read from the idempotency store
    record = idempotency_store.get(key, consistent_read=True)
    return record
```

### Step 5: Key Found — Return Cached Response

If the key exists in the store, the server returns the cached response. The response includes the original status code, headers, and body. The server does not process the request again. This is the core of idempotency: the second and subsequent requests produce the same response as the first.

```python
def handle_duplicate_request(record):
    return HTTPResponse(
        status=record.status_code,
        headers=record.headers,
        body=record.body
    )
```

### Step 6: Key Not Found — Acquire a Lock

If the key is not found, the server must prevent concurrent requests with the same key from processing simultaneously. The server acquires a distributed lock on the key. This lock prevents race conditions where two requests arrive at the same time and both see the key as missing.

```python
def acquire_lock(key):
    # Try to acquire a distributed lock with a TTL
    lock = distributed_lock.try_acquire(
        resource=f"idempotency:{key}",
        ttl_seconds=30,
        owner=server_instance_id
    )
    return lock
```

### Step 7: Process the Request

The server processes the request and generates a response. This is the actual business logic: charging the credit card, creating the order, sending the email, and so on. The server must ensure that the processing is itself idempotent or is protected by the idempotency key.

```python
def process_request(request_data):
    # Business logic that must produce consistent results
    payment = payment_service.charge(
        amount=request_data.amount,
        currency=request_data.currency,
        source=request_data.payment_source
    )
    order = order_service.create(
        items=request_data.items,
        payment_id=payment.id
    )
    return order
```

### Step 8: Store the Result and Release the Lock

The server stores the response in the idempotency store with the key. The store entry includes the status code, response headers, response body, and a timestamp. The server then releases the lock. The idempotency store entry is retained for the duration of the idempotency window.

```python
def store_result_and_release(key, response, lock):
    idempotency_store.put(
        key=key,
        value={
            "status_code": response.status_code,
            "headers": dict(response.headers),
            "body": response.body,
            "created_at": current_timestamp()
        },
        ttl_seconds=IDEMPOTENCY_WINDOW_SECONDS
    )
    lock.release()
```

## Variants of Idempotency

Idempotency is not a single technique. Different systems use different approaches depending on their requirements for consistency, latency, and complexity.

### Natural Idempotency

Some operations are idempotent by their nature, regardless of any idempotency key mechanism. The defining characteristic is that the operation's effect is defined in terms of a final state, not an incremental change.

HTTP PUT is the canonical example of natural idempotency. PUT replaces the entire resource at a given URL. Sending the same PUT request twice leaves the resource in the same state as sending it once. This is because PUT is defined as a complete replacement, not an incremental update.

HTTP DELETE is also naturally idempotent. Deleting a resource that has already been deleted returns a success response (typically 200 or 204). The second delete has no additional effect because the resource is already gone.

HTTP GET is naturally idempotent and additionally safe (no side effects at all). GET requests should never modify state.

For operations that are not naturally idempotent, such as creating a resource or processing a payment, you introduce an idempotency key to make them idempotent.

The idempotency key approach works for any operation, including POST and PATCH, which are not naturally idempotent. The key provides a way for the server to link a duplicate request to the original response.

The key must be:
- Unique across all requests for the duration of the idempotency window
- Generated by the client to ensure uniqueness across retries
- Stable across retries (the same key is used for all retries of the same request)

### Version-Based Idempotency

Version-based idempotency uses a version token or sequence number attached to the resource to ensure that updates are applied in order and only once.

Version-based idempotency is commonly used in optimistic locking scenarios. The version token is typically an integer, a UUID, or a timestamp. The client reads the current version, applies the update, and specifies the version it read. The server only applies the update if the version matches. This ensures that:
- Updates are applied exactly once
- Stale updates from concurrent writers are rejected
- Duplicate requests are rejected because the version has already changed

### Conditional Writes

Conditional writes use a database-level condition to ensure that a write only happens if the resource is in a specific state. This is a more general form of version-based idempotency.

Conditional writes are powerful because they delegate the idempotency guarantee to the database, which has strong consistency guarantees. The database ensures that the condition is checked atomically with the write.

## Mental Model: The Vending Machine

Think of an idempotent API as a vending machine. When you insert a dollar and press the button for a soda, the machine starts the dispensing process. If you press the button again while the soda is dispensing, the machine ignores the second press. If the soda is already dispensed and you press the button again, the machine either ignores the press or returns the same result (soda already dispensed, no additional charge).

The vending machine has an internal state machine that tracks which buttons have been pressed and which sodas have been dispensed. This is the idempotency store. The button press is the idempotency key. The soda is the response.

## Summary

Idempotency is a fundamental concept in distributed systems that ensures safety during retries. The key ideas are:

- Idempotent operations produce the same result regardless of how many times they are executed.
- Natural idempotency comes from the operation itself (PUT, DELETE, GET).
- Idempotency keys make any operation idempotent by providing a unique identifier for duplicate detection.
- Version-based idempotency and conditional writes provide idempotency at the database level.
- The idempotency store must be strongly consistent, durable, and have a configurable TTL.
