# Real-World Implementations

This section examines how major technology companies implement idempotency in their production systems. You will learn from the design decisions made by Stripe, PayPal, AWS, and Google Cloud, and see how open-source projects implement idempotency stores.

---

## Stripe: Idempotency Keys for Payment Processing

Stripe was one of the first major payment platforms to standardize idempotency keys. Stripe's approach is the de facto standard for idempotent API design.

Stripe's implementation has several notable design decisions:

### Server-Side Key Hashing

Stripe hashes the idempotency key before storing it. This means Stripe does not store the raw key in their database. The hash is stored instead. This protects the client's privacy: if the database is compromised, an attacker cannot correlate idempotency keys across different Stripe accounts.

### Request Body Validation

If a duplicate request arrives with a different request body than the original, Stripe returns a 409 Conflict response. This protects against the case where a client accidentally reuses an idempotency key for a different request.

```python
def check_duplicate(request):
    key_hash = sha256(request.headers["Idempotency-Key"])
    record = idempotency_store.get(key_hash)

    if record is not None:
        # Duplicate detected
        if record.request_body != request.body:
            # Different body → return 409 Conflict
            return HTTPResponse(
                status=409,
                body={
                    "error": {
                        "type": "idempotency_error",
                        "message": "This idempotency key was already used with a different request body."
                    }
                }
            )
        # Same body → return cached response
        return HTTPResponse(
            status=record.status_code,
            headers=record.headers,
            body=record.body
        )

    return None  # No duplicate
```

### 24-Hour TTL

Stripe uses a 24-hour TTL for all idempotency keys. This covers the typical retry window for payment processing. If a merchant retries a charge within 24 hours, the duplicate is detected. After 24 hours, the merchant must generate a new key.

The 24-hour TTL is a good default for payment systems because:
- Payment retries typically happen within minutes or hours
- 24 hours covers a full business day
- Storage requirements are manageable (5 GB per 10M requests)

### Internal Infrastructure

Stripe uses an internal distributed key-value store for idempotency records. The store is replicated across multiple data centers for high availability. Reads are strongly consistent within a data center and eventually consistent across data centers. This means a duplicate request that arrives in a different data center within a few seconds may not be detected.

PayPal's approach differs from Stripe in several important ways:

### Business Key as Idempotency Key

PayPal uses the invoice_number as the idempotency key. This is a business-level ID that the merchant generates. The invoice number must be unique within the merchant's account. PayPal enforces this with a database unique constraint on the (merchant_id, invoice_number) pair.

The advantage of this approach is that the idempotency key has business meaning. If a merchant needs to check whether a payment was processed, they can look up the payment by invoice number.

The disadvantage is that the merchant must manage invoice number uniqueness. If the merchant accidentally reuses an invoice number, the second payment is rejected.

### Database-Level Deduplication

PayPal's deduplication is implemented at the database level using unique constraints. This provides strong consistency guarantees. The database ensures that duplicate invoice numbers are rejected, even under concurrent access.

```sql
CREATE TABLE payments (
    payment_id UUID PRIMARY KEY,
    merchant_id VARCHAR(64) NOT NULL,
    invoice_number VARCHAR(128) NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    status VARCHAR(32) NOT NULL,
    created_at TIMESTAMP NOT NULL,
    UNIQUE (merchant_id, invoice_number)
);

-- Attempt to insert a payment
INSERT INTO payments (merchant_id, invoice_number, amount, status)
VALUES ('merchant_123', 'INV-12345', 20.00, 'created')
ON CONFLICT (merchant_id, invoice_number) DO NOTHING;
```

### Stateless Idempotency

PayPal's approach does not require a separate idempotency store. The payment data itself serves as the idempotency record. If a merchant retries with the same invoice number, the database rejects the duplicate, and the merchant can query the existing payment.

This approach is stateless from the idempotency perspective: no additional infrastructure is needed beyond the payment database.

## AWS S3 PUT Object Idempotency

AWS S3 uses natural idempotency for PUT Object operations. The PUT request replaces the object at a given key. Sending the same PUT request twice results in the same state: the object at the given key contains the same data.

### Conditional Idempotency with ETags

S3 also supports conditional writes that provide idempotency at the version level:

- **If-Match**: Only update the object if the current ETag matches the specified value. This prevents the lost-update problem. If two clients read the object, both get the same ETag. Only the first client to write succeeds. The second client's write is rejected.

- **If-None-Match**: Only create the object if no object exists at the key. This prevents accidentally overwriting an existing object. This is useful for idempotent creation: the first PUT creates the object, and subsequent PUTs with If-None-Match are rejected.

- **If-Unmodified-Since**: Only update the object if it has not been modified since the specified date. This is a time-based version check.

### S3 Consistency Model

S3 provides read-after-write consistency for PUT requests to new objects. For overwrite PUTs (replacing an existing object), S3 provides eventual consistency. This means that a duplicate PUT that arrives shortly after the first PUT may see the old version of the object.

## Google Cloud: Idempotency in Cloud Pub/Sub

Google Cloud Pub/Sub provides idempotent message publishing using a unique message ID. The client generates a unique message ID, and Pub/Sub deduplicates messages with the same ID within a configurable time window.

### Design Decisions

Google Cloud Pub/Sub's idempotency design reflects the messaging use case:

1. **Short TTL**: The default 10-minute deduplication window is appropriate for message publishing. If a publisher retries within 10 minutes, the duplicate is detected. After 10 minutes, the publisher can safely resend.

2. **Region-scoped**: Deduplication is per region. Messages published in different regions are not deduplicated against each other. This is acceptable for Pub/Sub because publishers typically publish to a single region.

3. **Server-generated IDs**: If the client does not provide a message ID, Pub/Sub generates one. However, server-generated IDs do not provide idempotency because the ID is generated after the message is received.

## Open Source Implementations

Several open-source projects provide idempotency implementations that you can study and use.

### Redis-Based Idempotency

Redis is the most common backend for idempotency stores. The implementation is straightforward:

```python
import redis
import json
import hashlib

class RedisIdempotencyStore:
    def __init__(self, redis_client, ttl=86400, key_prefix="idemp:"):
        self.redis = redis_client
        self.ttl = ttl
        self.key_prefix = key_prefix

    def _make_key(self, raw_key):
        key_hash = hashlib.sha256(raw_key.encode()).hexdigest()
        return f"{self.key_prefix}{key_hash}"

    def check_and_set(self, raw_key, request_body, response):
        key = self._make_key(raw_key)
        value = json.dumps({
            "request_body": request_body,
            "response": response
        })

        # Atomic check-and-set
        result = self.redis.set(key, "processing", ex=self.ttl, nx=True)

        if not result:
            # Key exists → return cached response
            cached = self.redis.get(key)
            if cached and cached != "processing":
                return json.loads(cached)
            return None  # Still processing

        return True  # First request

    def store_result(self, raw_key, response):
        key = self._make_key(raw_key)
        self.redis.set(key, json.dumps(response), ex=self.ttl)
```

### Django REST Framework Idempotency

The django-rest-framework-idempotency package provides a reusable idempotency middleware for Django. It stores idempotency keys in a database table and automatically handles deduplication.

```python
# django-rest-framework-idempotency usage
from rest_framework_idempotency.middleware import IdempotencyMiddleware

# In settings.py
MIDDLEWARE = [
    'rest_framework_idempotency.middleware.IdempotencyMiddleware',
    # ...
]

IDEMPOTENCY_SETTINGS = {
    'ENABLED': True,
    'TTL': 86400,  # 24 hours
    'LOCK_TTL': 30,  # 30 seconds
    'CACHE_BACKEND': 'default',
}
```

### FastAPI Idempotency

FastAPI does not have a built-in idempotency middleware, but you can implement one using a dependency:

```python
from fastapi import FastAPI, Header, HTTPException
import redis.asyncio as aioredis
import hashlib

app = FastAPI()
redis = aioredis.from_url("redis://localhost")

async def check_idempotency(
    Idempotency_Key: str = Header(None),
):
    if Idempotency_Key is None:
        raise HTTPException(status_code=400, detail="Missing Idempotency-Key")

    key_hash = hashlib.sha256(Idempotency_Key.encode()).hexdigest()
    result = await redis.set(f"idemp:{key_hash}", "processing", nx=True, ex=86400)

    if not result:
        raise HTTPException(status_code=409, detail="Duplicate request")

    return Idempotency_Key

@app.post("/orders")
async def create_order(key: str = Depends(check_idempotency)):
    # Process order
    order = {"id": "ord_123", "status": "confirmed"}
    await redis.set(f"idemp:{hashlib.sha256(key.encode()).hexdigest()}", json.dumps(order))
    return order
```

## Summary

Real-world idempotency implementations vary based on the specific use case. Stripe uses explicit idempotency keys with request body validation and a 24-hour TTL. PayPal uses business-level keys (invoice numbers) with database-level unique constraints. AWS S3 relies on the natural idempotency of PUT operations and offers conditional writes with ETags. Google Cloud Pub/Sub uses message-level idempotency with a configurable deduplication window. Open-source implementations in Redis, Django, and FastAPI provide reusable patterns.
