# Trade-offs and Analysis

Every idempotency design involves trade-offs. This section examines the core tension between consistency and availability, provides a decision matrix for choosing between approaches, analyzes failure modes, and identifies anti-patterns.

---

## Core Tension: Consistency vs. Availability

The fundamental trade-off in idempotency systems is between consistency and availability. A strongly consistent idempotency store guarantees that every duplicate request is detected. An eventually consistent store may miss duplicates but remains available during network partitions.

The choice between consistency and availability depends on the cost of a duplicate. For payment processing, a duplicate charge costs real money and damages customer trust. For notification delivery, a duplicate email is a minor annoyance. For analytics, a duplicate event has minimal impact on aggregated metrics.

### When Consistency Wins

Choose strong consistency when:
- A duplicate operation has a direct financial cost
- A duplicate operation creates a legal or compliance issue
- A duplicate operation modifies state in a way that cannot be automatically reversed
- The system processes transactions that involve external parties (banks, partners)

### When Availability Wins

Choose eventual consistency when:
- A duplicate operation is a minor inconvenience
- The system can detect and correct duplicates asynchronously
- The business can tolerate occasional duplicates
- The cost of rejecting a legitimate request during a partition is higher than the cost of a duplicate

## CAP Theorem Positioning

In CAP theorem terms, an idempotency system that uses strong consistency is a CP system (consistent and partition-tolerant). An idempotency system that uses eventual consistency is an AP system (available and partition-tolerant).

The key insight is that during a network partition, no idempotency system can be both perfectly consistent and perfectly available. You must choose which property to sacrifice.

For a CP system during a partition:
- The idempotency store cannot be reached.
- The application server cannot check for duplicates.
- The server must reject the request or accept the risk of duplicates.
- The correct choice depends on the business: reject the payment (CP) or accept the risk of a duplicate (AP).

For an AP system during a partition:
- The idempotency store is available but may return stale data.
- The server may miss a duplicate and process the request again.
- The duplicate is detected after the partition heals.
- The system must have compensating logic for the duplicate.

## Failure Modes

Designing for failure is essential in distributed idempotency systems. Here are the failure modes you must plan for.

### Failure Mode 1: Idempotency Store Unavailable

The idempotency store (Redis, database) is down. The application server cannot check for duplicates. Every request is processed as a new request, even if the same idempotency key was already processed.

**Impact**: All requests are processed, possibly multiple times. No deduplication occurs.

**Mitigations**:
1. Circuit breaker: If the store is unavailable, fail open (process all requests) or fail closed (reject all requests). The choice depends on the business.
2. Local cache: Maintain a small local cache of recently seen keys. This handles short store outages.
3. Fallback store: Use a secondary store (for example, a local database) while the primary store is down.

### Failure Mode 2: Lock Holder Crash

The server that holds the lock for an idempotency key crashes before releasing the lock. Other servers cannot process requests with the same key until the lock expires.

**Impact**: Requests with the affected key are blocked for the duration of the lock TTL.

**Mitigations**:
1. Lock TTL: Set a reasonable lock TTL (for example, 10 seconds). If the holder crashes, the lock expires and other servers can proceed.
2. Heartbeat: The lock holder sends periodic heartbeats to extend the lock. If the heartbeat stops, the lock is released.

### Failure Mode 3: Key Collision

Two different clients generate the same idempotency key. The second request is treated as a duplicate of the first. The second request's operation is never performed.

**Impact**: Data loss. The second request's operation is silently dropped.

**Mitigations**:
1. Use UUID v4 with 122 bits of entropy. The collision probability is negligible.
2. Include a namespace in the key (for example, "order:uuid").
3. Monitor for collision rates. A single collision in a year is cause for investigation.

### Failure Mode 4: TTL Too Short

The idempotency key TTL is shorter than the maximum retry interval. A client retries after the TTL expires. The server treats the retry as a new request and processes it again.

**Impact**: Duplicate processing for late retries.

**Mitigations**:
1. Set the TTL to at least 2x the maximum expected retry interval.
2. For payments, use a 24-hour TTL to cover same-day retries.
3. Log the retry timestamp and adjust the TTL based on the observed retry distribution.

### Failure Mode 5: Clock Skew

The application server and the idempotency store have different clocks. The TTL is calculated based on the server's clock or the store's clock. If the clocks differ, keys may expire early or persist longer than expected.

**Impact**: Early TTL expiry causes duplicates. Late TTL expiry causes storage bloat.

**Mitigations**:
1. Use NTP to synchronize clocks across all servers.
2. Use the store's clock for TTL calculations (Redis TTL is calculated on the Redis server).
3. Add a clock skew tolerance buffer to the TTL (for example, add 60 seconds to the configured TTL).

### Failure Mode 6: Response Cache Poisoning

A malicious client sends a request with a valid idempotency key but with a different request body. The server returns the cached response from the first request, which contains data from the original request. The client sees the original data, not the data they sent.

**Impact**: Information disclosure. The client can infer the contents of the original request.

**Mitigations**:
1. Validate that the request body matches the stored request body for the key.
2. Include the request hash in the idempotency store record. On duplicate, compare the hash.
3. Return 409 Conflict if the request body does not match the stored body.

## Anti-Patterns

These anti-patterns are common mistakes in idempotency system design.

### Anti-Pattern 1: Using the Request Body as the Idempotency Key

Some systems hash the request body and use the hash as the idempotency key. This breaks idempotency because:
- Two different requests with the same body are treated as duplicates (false positive).
- A single request with a slightly different body (for example, a timestamp field) is treated as a new request (false negative).

**Correct approach**: Use a client-generated UUID as the idempotency key. The request body is stored alongside the key for verification.

### Anti-Pattern 2: Using a Timestamp as the Idempotency Key

Timestamps are not unique. Two clients can generate the same timestamp. Two retries from the same client within the same millisecond produce the same timestamp.

**Correct approach**: Use a UUID or a server-generated monotonic sequence number.

### Anti-Pattern 3: Storing Only the Status, Not the Full Response

If the store only records that a key was processed (status: completed) but not the full response, the server cannot reconstruct the response for a duplicate request. The server must either reprocess the request or return an incomplete response.

**Correct approach**: Store the full response including status code, headers, body, and content type.

### Anti-Pattern 4: Not Handling Concurrent Requests for the Same Key

Two requests with the same key arrive simultaneously. The server checks the store, finds nothing for both requests, and processes both. This is a race condition that produces duplicates.

**Correct approach**: Use atomic check-and-set (SET NX) or a distributed lock with a double-check pattern.

### Anti-Pattern 5: Using a Single Global TTL for All Operations

Payment operations need a 24-hour TTL. Notification operations need a 1-hour TTL. Analytics operations need a 5-minute TTL. Using a single TTL wastes storage for short-lived operations and risks duplicates for long-lived operations.

**Correct approach**: Allow the client or the operation type to specify the TTL. Use a default TTL that is appropriate for the operation.

### Anti-Pattern 6: Relying on Database Auto-Increment for Idempotency

Database auto-increment columns do not provide idempotency. An INSERT with auto-increment always creates a new row, even if the same data was inserted before. The auto-increment ID is new each time.

**Correct approach**: Use a unique constraint on a business key or an idempotency key column. Use INSERT ON CONFLICT DO NOTHING or INSERT ... WHERE NOT EXISTS.

### Anti-Pattern 7: Not Cleaning Up Expired Keys

Without a cleanup mechanism, the idempotency store grows indefinitely. Storage costs increase. Read performance degrades. Eventually, the store runs out of memory or disk space.

**Correct approach**: Implement a cleanup scheduler that runs every hour and deletes expired keys. Use TTL-based expiry in Redis or a background job in a database.

## When Not to Use Idempotency

Idempotency is not always the right choice. Consider these scenarios:

### Read-Only Operations

GET requests are naturally safe and idempotent. No additional idempotency mechanism is needed for read-only operations. Adding idempotency keys to GET requests adds overhead without benefit.

### Idempotent-by-Design Operations

Operations that are already idempotent do not need an idempotency key mechanism. For example, setting a user's email address to "alice@example.com" is idempotent. The second and third writes produce the same state as the first.

### Operations with Natural Uniqueness

If an operation has a natural unique identifier that is generated server-side, the server can use that identifier for deduplication without an explicit idempotency key. For example, a database primary key with a unique constraint on the natural business key.

### Low-Cost Duplicates

If the cost of a duplicate is trivial (for example, incrementing a view counter), the complexity of implementing idempotency outweighs the benefit. The duplicate is harmless.

### Prototypes and MVPs

In early-stage products, the complexity of idempotency may not be justified. You can add idempotency when the product proves its value and the risk of duplicates becomes significant.

## Summary

The core tension in idempotency design is between consistency and availability. The decision matrix helps you choose the right approach based on your requirements for consistency, latency, throughput, complexity, and cost. CAP theorem positioning determines how your system behaves during network partitions. Six failure modes (store unavailable, lock holder crash, key collision, short TTL, clock skew, cache poisoning) require specific mitigations. Seven anti-patterns (request body as key, timestamp as key, partial response storage, no concurrency handling, single global TTL, auto-increment reliance, no cleanup) should be avoided. Finally, idempotency is not always needed: read-only operations, naturally idempotent operations, and low-cost duplicates may not warrant the complexity.
