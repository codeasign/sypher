# Challenges

These challenges test your ability to apply idempotency concepts to real-world design problems. The challenges are organized into three tiers: foundation, design, and expert. Each challenge describes a scenario and asks specific questions about the design, implementation, and trade-offs.

## Foundation Challenges

### Challenge 1: Payment Gateway Duplicate Detection

**Scenario**: You are building a payment gateway that processes credit card charges. Each charge request has an idempotency key provided by the merchant. The gateway must ensure that no duplicate charges are processed, even under concurrent requests. The gateway processes 1,000 requests per second.

**Requirements**:
- Detect duplicate requests with the same idempotency key.
- Handle concurrent requests with the same key arriving simultaneously.
- Return the cached response for duplicate requests.
- Store responses for 24 hours.
- Support 1,000 requests per second.

**Tasks**:
1. Design the idempotency store data structure. Specify the key format, value format, and storage backend.
2. Write the pseudocode for the idempotency check-and-set operation, including the concurrency handling.
3. Calculate the storage required for 24 hours of idempotency records. Assume each record is 1 KB.
4. Describe how you would handle a crash of the payment gateway between processing the charge and storing the response.

**Expected answer structure**:

1. Data structure: Redis with key `idemp:payment:{hash_of_key}` and value `{status, response_code, response_body, created_at}`. TTL of 86,400 seconds.

2. Pseudocode using Redis SET NX for atomic check-and-set:

```python
import hashlib
import json

def process_charge(request):
    key_hash = hashlib.sha256(request.idempotency_key.encode()).hexdigest()
    redis_key = f"idemp:payment:{key_hash}"

    # Step 1: Atomic check-and-set
    result = redis.set(
        redis_key,
        json.dumps({"status": "processing"}),
        nx=True,
        ex=86400
    )

    if not result:
        # Step 2: Duplicate detected
        cached = redis.get(redis_key)
        cached_data = json.loads(cached)
        if cached_data["status"] == "processing":
            # Still processing, wait and retry or return 409
            return HTTPResponse(409, "Request is still being processed")
        return HTTPResponse(
            cached_data["response_code"],
            cached_data["response_body"]
        )

    try:
        # Step 3: Process the charge
        response = charge_credit_card(request.amount, request.card_info)

        # Step 4: Store the result
        redis.set(
            redis_key,
            json.dumps({
                "status": "completed",
                "response_code": 200,
                "response_body": response.to_dict(),
                "created_at": time.now()
            }),
            ex=86400
        )
        return response
    except Exception as e:
        # Step 5: Store the error and re-raise
        redis.set(
            redis_key,
            json.dumps({
                "status": "failed",
                "response_code": 500,
                "response_body": {"error": str(e)},
                "created_at": time.now()
            }),
            ex=86400
        )
        raise
```

3. Storage calculation:
- 1,000 requests/sec x 86,400 sec = 86.4M requests per day
- 86.4M x 1 KB = 86.4 GB per day
- With replication factor of 3: 259.2 GB
- Redis cluster with 10 shards: ~26 GB per shard

4. Crash handling: Use a processing state. If the gateway crashes between processing and storing, the next retry sees the "processing" status and must determine whether the charge was processed. Use a write-ahead log: write "processing started" to durable storage before charging. On retry, check the write-ahead log to determine the true status.

### Challenge 2: User Registration Idempotency

**Scenario**: You are building a user registration system. The registration endpoint receives email and password. The system must ensure that duplicate registration attempts do not create multiple accounts. Each registration attempt may come from the user clicking the "Register" button multiple times.

**Requirements**:
- Prevent duplicate account creation.
- Return the existing account details for duplicate registration attempts.
- Handle concurrent registration attempts with the same email.
- Do not require the client to generate an idempotency key.

**Tasks**:
1. Design an idempotency mechanism that does not require client-generated keys.
2. Write the SQL schema that enforces idempotency at the database level.
3. Write the pseudocode for the registration flow, including the idempotency check.
4. Explain how your design handles concurrent registration attempts.

**Expected answer structure**:

1. Use the email as the natural idempotency key. The database has a unique constraint on the email column. This provides idempotency without requiring client-generated keys.

2. SQL schema:

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(256) NOT NULL,
    password_hash VARCHAR(256) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    -- Unique constraint on email provides idempotency
    UNIQUE (email)
);

-- Index for efficient email lookup
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

3. Registration pseudocode:

```python
def register_user(email, password):
    password_hash = hash_password(password)

    try:
        # INSERT with ON CONFLICT DO NOTHING
        # This is atomic and prevents race conditions
        result = db.execute("""
            INSERT INTO users (email, password_hash)
            VALUES (?, ?)
            ON CONFLICT (email) DO NOTHING
            RETURNING id, email, created_at
        """, email, password_hash)

        if result:
            # New user created
            return HTTPResponse(201, {
                "id": result["id"],
                "email": result["email"],
                "is_new": True
            })
        else:
            # User already exists
            existing = db.execute(
                "SELECT id, email, created_at FROM users WHERE email = ?",
                email
            )
            return HTTPResponse(200, {
                "id": existing["id"],
                "email": existing["email"],
                "is_new": False
            })
    except IntegrityError:
        # Concurrent registration: user was created between our check
        # and our insert. Return the existing user.
        existing = db.execute(
            "SELECT id, email, created_at FROM users WHERE email = ?",
            email
        )
        return HTTPResponse(200, {
            "id": existing["id"],
            "email": existing["email"],
            "is_new": False
        })
```

4. Concurrency handling: The database unique constraint ensures that only one INSERT succeeds. The ON CONFLICT DO NOTHING clause handles concurrent inserts: if two registration attempts arrive at the same time, one succeeds and the other returns no rows. Both receive the correct response (201 for the first, 200 for the second).

### Challenge 3: File Upload Idempotency

**Scenario**: A file upload service allows users to upload files with a client-generated file ID. The service must ensure that uploading the same file twice does not create duplicate files. The upload may be interrupted and retried.

**Requirements**:
- Use the client-generated file ID as the idempotency key.
- If the same file ID is used for a different file, return 409 Conflict.
- Support resumable uploads: if the first upload was interrupted, the retry continues from where it left off.
- Store files in S3 and metadata in PostgreSQL.

**Tasks**:
1. Design the database schema for tracking file uploads with idempotency.
2. Write the pseudocode for the upload flow, including idempotency check and resumability.
3. Describe how you detect that a retry has a different file than the original.
4. Explain the storage cleanup strategy for failed uploads.

**Expected answer structure**:

1. Database schema:

```sql
CREATE TABLE file_uploads (
    file_id VARCHAR(128) PRIMARY KEY,
    original_filename VARCHAR(512) NOT NULL,
    file_hash VARCHAR(64) NOT NULL,
    total_size BIGINT NOT NULL,
    uploaded_size BIGINT NOT NULL DEFAULT 0,
    status VARCHAR(32) NOT NULL DEFAULT 'pending',
    s3_key VARCHAR(1024),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMP NOT NULL
);
```

2. Upload flow:

```python
def upload_file(file_id, filename, chunk_data, chunk_offset):
    # Check if file ID already exists
    record = db.execute(
        "SELECT * FROM file_uploads WHERE file_id = ?", file_id
    )

    if record and record.status == "completed":
        # Duplicate upload
        return HTTPResponse(200, {
            "file_id": file_id,
            "status": "completed",
            "s3_url": generate_s3_url(record.s3_key)
        })

    if record and record.file_hash != hash_file(filename, chunk_data):
        # Same file ID, different file - conflict
        return HTTPResponse(409, {
            "error": "File ID already used for a different file"
        })

    if not record:
        # First chunk - create record
        db.execute("""
            INSERT INTO file_uploads (file_id, original_filename, file_hash,
                                      total_size, status, expires_at)
            VALUES (?, ?, ?, ?, 'uploading', ?)
            ON CONFLICT (file_id) DO NOTHING
        """, file_id, filename, hash_file(filename),
            get_total_size(), now() + timedelta(hours=24))

    # Upload chunk to S3
    s3_key = f"uploads/{file_id}"
    upload_chunk_to_s3(s3_key, chunk_data, chunk_offset)

    # Update uploaded size
    db.execute("""
        UPDATE file_uploads
        SET uploaded_size = uploaded_size + ?, updated_at = NOW()
        WHERE file_id = ?
    """, len(chunk_data), file_id)

    # Check if upload is complete
    record = db.execute(
        "SELECT * FROM file_uploads WHERE file_id = ?", file_id
    )
    if record.uploaded_size >= record.total_size:
        db.execute("""
            UPDATE file_uploads SET status = 'completed', updated_at = NOW()
            WHERE file_id = ?
        """, file_id)

    return HTTPResponse(200, {
        "file_id": file_id,
        "uploaded_size": record.uploaded_size,
        "total_size": record.total_size,
        "status": record.status
    })
```

3. File comparison: Compute a hash of the filename and the first chunk's content. Store this hash in the database. On retry, recompute the hash and compare. If the hash differs, return 409 Conflict.

4. Cleanup: Run a background job that deletes records with status 'uploading' or 'pending' where expires_at has passed. Delete the corresponding S3 objects. Run this job every hour.

## Design Challenges

### Challenge 4: Distributed Order Processing with Idempotency

**Scenario**: An e-commerce platform processes orders through a chain of services: Order Service, Payment Service, Inventory Service, and Notification Service. Each service must handle idempotency independently, and the overall order must be processed exactly once.

**Requirements**:
- The client sends one request to create an order.
- The order goes through payment, inventory reservation, and notification.
- If any step fails, the entire order must be rolled back or retried.
- The client must be able to retry without creating duplicate orders.
- The system processes 10,000 orders per day.

**Tasks**:
1. Design the idempotency flow across all four services.
2. Specify the idempotency key management strategy.
3. Describe the rollback strategy for partial failures.
4. Calculate the storage requirements for the idempotency stores across all services.

**Guidance**: Each service should have its own idempotency store with its own TTL. The top-level idempotency key (from the client) is propagated to all downstream services as a derived key. For example, if the client sends key "K", the Order Service uses "K", the Payment Service uses "K:payment", the Inventory Service uses "K:inventory", and the Notification Service uses "K:notification". This ensures that each service's idempotency keys are unique within that service.

The rollback strategy uses a saga pattern. If the Payment Service succeeds but the Inventory Service fails, the system fires a compensating transaction: refund the payment and mark the order as failed. The client can retry with the same top-level key. The system detects that the order processing is in a failed state and retries from the point of failure.

Storage requirements:
- Order Service: 10,000 keys/day x 2 KB per record = 20 MB/day
- Payment Service: 10,000 keys/day x 1 KB per record = 10 MB/day
- Inventory Service: 10,000 keys/day x 512 bytes per record = 5 MB/day
- Notification Service: 10,000 keys/day x 256 bytes per record = 2.5 MB/day
- Total: 37.5 MB/day, well within the capacity of a single Redis instance.

### Challenge 5: Exactly-Once Message Processing

**Scenario**: You are building a stream processing system that consumes events from Apache Kafka and writes results to a database. Kafka provides at-least-once delivery, meaning the same event may be delivered multiple times. Your system must provide exactly-once processing semantics.

**Requirements**:
- Each event has a unique event ID.
- Events may be replayed from Kafka (for example, after a consumer crash).
- The system must process each event exactly once.
- The system must handle out-of-order events (events with the same ID arriving after a different event has already been processed).
- The system processes 100,000 events per second.

**Tasks**:
1. Design the deduplication mechanism for Kafka events.
2. Specify the data structure for tracking processed events.
3. Write the pseudocode for the event processing loop with deduplication.
4. Describe how you handle event replay without duplicate processing.
5. Calculate the storage and throughput requirements.

**Guidance**: Use a Redis-backed deduplication table with the event ID as the key. Use Redis SET NX for atomic deduplication. Set a TTL of 7 days to cover the maximum replay window.

```python
def process_event(event):
    # Atomic deduplication
    result = redis.set(
        f"dedup:{event.id}",
        json.dumps({"status": "processing"}),
        nx=True,
        ex=604800  # 7 days
    )

    if not result:
        # Event already processed or being processed
        cached = redis.get(f"dedup:{event.id}")
        status = json.loads(cached)["status"]
        if status == "completed":
            return  # Already processed
        elif status == "processing":
            # Being processed by another consumer
            # Wait and check again
            time.sleep(1)
            return process_event(event)
        elif status == "failed":
            # Previous attempt failed, retry
            pass

    try:
        # Process the event
        result = apply_event_to_database(event)

        # Mark as completed
        redis.set(
            f"dedup:{event.id}",
            json.dumps({"status": "completed", "result": result}),
            ex=604800
        )
    except Exception as e:
        # Mark as failed
        redis.set(
            f"dedup:{event.id}",
            json.dumps({"status": "failed", "error": str(e)}),
            ex=604800
        )
        raise
```

Storage: 100,000 events/sec x 86,400 sec = 8.64B events per day. A 7-day window would require 60.5B entries, which is too large for Redis. Use a compacted Kafka topic instead: the topic stores only the latest state for each event ID. Consume the compacted topic to rebuild the deduplication set on consumer restart.

### Challenge 6: API Gateway Idempotency Layer

**Scenario**: Your company has 50 microservices, each with its own API. The operations team wants to implement idempotency at the API gateway level so that all services benefit from idempotency without modifying each service individually.

**Requirements**:
- The API gateway intercepts all POST, PUT, PATCH requests.
- If a request has an idempotency key, the gateway handles deduplication.
- If the key is new, the gateway forwards the request to the appropriate service.
- The gateway caches the response and returns it for duplicate requests.
- The gateway must handle 50,000 requests per second.

**Tasks**:
1. Design the API gateway idempotency middleware.
2. Specify the response caching strategy.
3. Describe how the gateway handles partial failures (gateway caches the response but the service has not finished processing).
4. Explain how the gateway invalidates cached responses when a service is rolled back.
5. Discuss the trade-offs of gateway-level vs service-level idempotency.

**Guidance**: The gateway stores the idempotency key and the response in Redis. The response is cached for the duration of the idempotency window (24 hours). If the request is a duplicate, the gateway returns the cached response without forwarding to the service.

For partial failures: the gateway should not cache a response until the service has confirmed completion. Use a processing state: when the request is forwarded, set the key to "processing". When the response arrives, update it to "completed" with the response data. If a duplicate request arrives while the status is "processing", return 409 Retry-After.

For service rollback: the gateway cannot easily detect service rollbacks. The service must explicitly call a gateway endpoint to invalidate the cached response. This is a best-effort mechanism.

Gateway-level vs service-level: gateway-level provides centralized management and is transparent to services, but the gateway must understand response formats and cannot provide operation-specific TTLs. Service-level provides maximum flexibility but requires each service to implement idempotency.

### Challenge 7: Multi-Region, Multi-Datacenter Idempotency at Global Scale

**Scenario**: You work at a global payment company that processes 1M transactions per second across 10 regions. Each region has 3 data centers. The system must provide exactly-once processing for payment transactions. A duplicate payment is a serious incident.

**Requirements**:
- 1M transactions per second globally.
- 10 regions, 3 data centers per region (30 data centers total).
- Exactly-once processing for payments.
- P99 latency for idempotency check under 5 milliseconds within a region.
- Cross-region latency: 100-200 ms.
- Tolerance for regional outages.

**Tasks**:
1. Design the idempotency key management strategy for global scale.
2. Specify the data replication strategy across regions and data centers.
3. Describe the consistency model and how you handle network partitions.
4. Design the failover mechanism when a region becomes unavailable.
5. Calculate the infrastructure requirements (storage, compute, network).
6. Describe how you test this system for correctness.

**Guidance**: Use a hierarchical idempotency architecture. Each transaction is assigned to a home region based on the idempotency key hash. The home region provides strong consistency for that key. Other regions route the request to the home region.

Within a region, use a three-data-center setup with Raft consensus. Writes require acknowledgment from at least 2 of 3 data centers. This provides strong consistency within the region and tolerance for a single data center failure.

For cross-region coordination, use a request router that maps each idempotency key to its home region. The router is a consistent hash ring that is replicated across all regions. If the home region is unavailable, the router assigns a secondary region. The secondary region operates in a mode that accepts the risk of duplicates (AP mode) and reconciles after the primary region is restored.

This architecture requires approximately 10 TB of storage (1M keys/sec x 86,400 sec x 512 bytes x 7 days TTL), distributed across 30 data centers. Each data center handles 33K keys/sec and stores approximately 333 GB.

### Challenge 8: Idempotency with External Third-Party Dependencies

**Scenario**: Your service depends on a third-party payment processor that does not support idempotency keys. Each call to the third-party processor may result in a charge, and the processor does not detect duplicates. Your service must provide exactly-once semantics to your clients while using this non-idempotent third party.

**Requirements**:
- Clients send idempotent requests to your service.
- Your service calls a non-idempotent third-party API.
- You must prevent duplicate charges even if the third party does not.
- The third-party API has a response time of 500-2000 ms.
- Your service must handle 5,000 requests per second.

**Tasks**:
1. Design a mechanism to make the third-party API effectively idempotent.
2. Specify how you detect and handle duplicate calls to the third party.
3. Describe the reconciliation process for undetected duplicates.
4. Explain the trade-offs of your approach.

**Guidance**: Use an outbox pattern with a transactional outbox table. When your service receives a client request, it writes the payment intent to an outbox table in the same database transaction as the idempotency check. A separate process reads the outbox and sends the payment to the third party. If the process crashes, it re-reads the outbox and checks whether the payment was already sent by looking at the third-party transaction log.

For duplicate detection at the third party, use idempotency fields in the third-party API if available. Many third-party APIs allow you to specify a unique transaction ID or invoice number. If the third-party does not support any form of idempotency, you must rely on reconciliation: periodically compare your transaction log with the third-party transaction log to find duplicates, and issue refunds for duplicates.

The trade-off is between immediate consistency and eventual consistency. The outbox pattern provides immediate consistency for your service but eventual consistency with the third party. This means there is a window (typically seconds to minutes) where a duplicate charge could occur. The reconciliation process closes this window by detecting and refunding duplicates.
