This section covers interview questions about idempotency that you may encounter in system design interviews at FAANG companies. The questions range from fundamental concepts to advanced distributed systems challenges.

---

## Question 1: What is idempotency and why is it important in distributed systems?

**Answer**: Idempotency is the property that an operation can be applied multiple times without changing the result beyond the first application. In distributed systems, idempotency is important because network failures, timeouts, and retries are inevitable. Without idempotency, a retry of a request that was already processed but whose response was lost can cause duplicate side effects, such as charging a customer twice or creating duplicate orders. Idempotency provides safety guarantees for retry logic, which is fundamental to building reliable distributed systems.

The key scenarios where idempotency matters are:
- **Network timeouts**: The client sends a request, the server processes it, but the response is lost. The client retries. Without idempotency, the operation runs twice.
- **Client-side retry logic**: The client automatically retries failed requests. Each retry must be safe.
- **At-least-once delivery**: Message queues often deliver messages at least once. The consumer must handle duplicates.
- **User impatience**: Users click buttons multiple times when the UI is slow. Each click sends a request.

---

## Question 2: How does HTTP PUT differ from HTTP POST in terms of idempotency?

**Answer**: HTTP PUT is naturally idempotent because it replaces the entire resource at a specific URL. Sending the same PUT request with the same body twice produces the same state: the resource at the URL is set to the provided value. The first PUT may create the resource, and the second PUT replaces it with the same data, resulting in no change.

HTTP POST is not naturally idempotent because it creates a new resource. Each POST request creates a new resource with a new identifier. Sending the same POST request twice creates two resources. POST is designed for actions that have side effects, such as creating an order or processing a payment.

To make POST idempotent, you add an idempotency key. The server uses the key to detect duplicates and return the cached response from the first request. This does not change the nature of POST; it adds a layer of duplicate detection on top of the non-idempotent operation.

## Question 3: What is the idempotency window and how do you choose its duration?

**Answer**: The idempotency window is the time period during which the system retains idempotency keys for duplicate detection. If a duplicate request arrives within this window, the system detects the duplicate and returns the cached response. If a duplicate request arrives after the window expires, the key is gone, and the system processes the request as if it were new.

To choose the duration, consider these factors:

1. **Maximum retry interval**: How long does the client retry? If the client retries for up to 5 seconds, a 10-second window is sufficient. If the client retries for up to 1 hour, the window must be at least 1 hour.

2. **Business requirements**: For payment processing, a 24-hour window is standard because customers may retry a failed payment throughout the day. For analytics events, a 5-minute window may be sufficient.

3. **Storage cost**: Longer windows require more storage. A 24-hour window at 10M requests/day with 512 bytes per record requires 5.12 GB of storage. A 7-day window requires 35.84 GB.

4. **Regulatory requirements**: Some regulations require that transactions be recorded for a minimum period. The idempotency window should align with these requirements.

A good default is 24 hours for financial operations and 1 hour for non-financial operations.

## Question 4: How do you handle race conditions when two requests with the same idempotency key arrive simultaneously?

**Answer**: Race conditions occur when two requests with the same idempotency key arrive at the server before either one has been processed. Both requests check the idempotency store, both see the key is missing, and both proceed to process the request. This results in duplicate processing.

There are three approaches to handle this:

**Approach 1: Atomic check-and-set (CAS)**
Use the database's atomic CAS operation. In Redis, this is SET NX: set the key only if it does not exist. The first request's SET NX succeeds. The second request's SET NX fails because the key now exists. The second request then returns the cached response.

```python
# Atomic check-and-set
result = redis.set(key, value, nx=True, ex=ttl)
if result:
    # First request - process
    process_request()
else:
    # Duplicate - return cached response
    return get_cached_response(key)
```

**Approach 2: Distributed lock with double-check**
Acquire a distributed lock on the key, then check the store again before processing. This ensures that even if two requests arrive simultaneously, only one processes.

```python
def handle_request(request):
    key = request.idempotency_key
    if check_store(key):
        return get_cached(key)

    with distributed_lock(f"lock:{key}", ttl=10):
        # Double-check under lock
        if check_store(key):
            return get_cached(key)
        # Process and store
        result = process(request)
        store_result(key, result)
        return result
```

**Approach 3: Database unique constraint**
Use a database unique constraint on the idempotency key column. The first INSERT succeeds. The second INSERT fails with a unique constraint violation. The application catches the violation and returns the cached response.

```python
try:
    db.execute("INSERT INTO idempotency (key, result) VALUES (?, ?)", key, result)
    # First request - process
    process_request()
except UniqueConstraintViolation:
    # Duplicate - return cached response
    return get_cached_response(key)
```

## Question 5: How would you design idempotency for a multi-region payment system?

**Answer**: Designing idempotency for a multi-region payment system requires balancing consistency, availability, and latency. Payment systems require strong consistency: duplicate charges are unacceptable. The challenge is that cross-region strongly consistent operations have high latency (50-200 ms).

Here is a design approach:

**Tier 1: Strongly consistent within region**
Each region has its own idempotency store that provides strong consistency within the region. The store uses a distributed consensus protocol (Raft or Paxos) to ensure that all nodes in the region agree on the state of each idempotency key.

**Tier 2: Routing-based key grouping**
Assign each idempotency key to a home region based on the key's hash. All requests with that key must be routed to the home region. This provides strong consistency for the key without requiring cross-region coordination.

```
key_hash = hash(idempotency_key) % num_regions
home_region = regions[key_hash]  # 0 = US, 1 = EU, 2 = ASIA
```

**Tier 3: Global failover**
If the home region is unavailable, the request is routed to a failover region. The failover region uses a replicated copy of the idempotency store. The failover store may have stale data, so the system must accept the risk of duplicates during failover.

**Tier 4: Compensation**
If a duplicate charge is detected after the fact (for example, through reconciliation), the system initiates a refund. This is a compensating transaction that undoes the duplicate. The refund is logged and tracked.

The key design decision is that for a payment system, the cost of rejecting a legitimate request during a partition is lower than the cost of processing a duplicate charge. The system should err on the side of rejecting requests when the idempotency store is unavailable.

## Question 6: How do you implement idempotency for a message queue consumer?

**Answer**: Implementing idempotency for a message queue consumer has different requirements than implementing it for an API. The consumer processes messages asynchronously, often at high throughput. The consumer cannot return a response to the producer; it must handle duplicates internally.

Here is a design approach:

**Approach: Deduplication table in the consumer database**

The consumer maintains a deduplication table that records which message IDs have been processed. Each message has a unique ID generated by the producer. When the consumer receives a message, it checks the deduplication table.

```sql
CREATE TABLE processed_messages (
    message_id VARCHAR(128) PRIMARY KEY,
    consumer_id VARCHAR(64) NOT NULL,
    processed_at TIMESTAMP NOT NULL DEFAULT NOW(),
    ttl_expires_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_processed_messages_ttl
ON processed_messages(ttl_expires_at);
```

```python
def process_message(message):
    # Check if message was already processed
    existing = db.execute(
        "SELECT 1 FROM processed_messages WHERE message_id = ?",
        message.id
    )
    if existing:
        # Message already processed, skip
        return

    # Mark as processing (prevents duplicate processing)
    # on consumer crash or restart
    db.execute(
        "INSERT INTO processed_messages (message_id, consumer_id, ttl_expires_at) "
        "VALUES (?, ?, ?) "
        "ON CONFLICT (message_id) DO NOTHING",
        message.id, consumer.id, now() + ttl
    )

    # Process the message
    result = handle_message(message)

    # Update the record with the result
    db.execute(
        "UPDATE processed_messages SET result = ? WHERE message_id = ?",
        result, message.id
    )
```

**Key considerations for message queue idempotency**:

1. **Message ID uniqueness**: The producer must generate unique message IDs. UUID v4 is the standard choice.

2. **TTL and cleanup**: The deduplication table must be cleaned up periodically. A 7-day TTL is typical for message queues.

3. **Consumer crashes**: If the consumer crashes after inserting the deduplication record but before processing the message, the record is a tombstone. The consumer must handle this case by checking the processing status.

4. **At-least-once delivery**: Most message queues provide at-least-once delivery. The consumer must handle duplicates, which is exactly what this approach does.

5. **Transactional processing**: The deduplication check and the business logic should be in the same database transaction to ensure atomicity.

## Question 7: How would you design an idempotency system for a financial API that handles 100,000 requests per second?

**Answer**: Designing at this scale requires careful consideration of throughput, latency, and consistency. Here is a design:

**Data store**: Use a Redis cluster with 10 shards, each shard running on dedicated hardware. Each shard handles 10,000 requests per second, well within Redis's capacity of 100,000 ops/second per shard.

**Key design**: Use a two-level key structure. The first level is the shard key (hash of the idempotency key modulo 10). The second level is the idempotency key hash. This ensures uniform distribution across shards.

```python
def get_shard(idempotency_key):
    key_hash = hashlib.sha256(idempotency_key.encode()).hexdigest()
    shard_id = int(key_hash[:8], 16) % 10
    return shard_id, key_hash
```

**Atomic operations**: Use Redis SET NX for atomic check-and-set. The SET NX command is single-threaded in Redis, so it is safe under concurrent access.

```python
def process_request(request):
    shard_id, key_hash = get_shard(request.idempotency_key)
    redis_client = get_redis_client(shard_id)

    result = redis_client.set(
        f"idemp:{key_hash}",
        json.dumps({"status": "processing"}),
        nx=True,
        ex=86400  # 24 hours
    )

    if not result:
        # Duplicate detected
        cached = redis_client.get(f"idemp:{key_hash}")
        return json.loads(cached)

    # Process the request
    response = process_payment(request)

    # Store the result
    redis_client.set(
        f"idemp:{key_hash}",
        json.dumps({"status": "completed", "response": response}),
        ex=86400
    )

    return response
```

**Replication and failover**: Each Redis shard has a master and two replicas. Writes go to the master. Reads during duplicate detection can go to replicas if eventual consistency is acceptable. For financial operations, reads should go to the master to ensure strong consistency.

**Batching**: For operations that can be batched (for example, refunds), use a batch idempotency key. The batch key covers all operations in the batch. If the batch is retried, the entire batch is skipped.

**Rate limiting**: The idempotency system itself must be rate-limited. If the idempotency store is overwhelmed, requests should be rejected with a 429 Too Many Requests response rather than allowing duplicates.

**Monitoring**: Monitor the duplicate rate, store latency, and store size. Alert on any anomaly. Track the P50, P99, and P999 latency for idempotency operations.

## Question 8: What happens when the idempotency key TTL expires and a duplicate request arrives?

**Answer**: When the TTL expires, the idempotency key is removed from the store. If a duplicate request arrives after expiry, the server treats it as a new request. The server checks the store, finds no key, and processes the request. This results in duplicate processing.

This is a correctness issue, not a performance issue. The system must ensure that the TTL is long enough to cover the maximum expected retry interval.

**Mitigations**:

1. **Set TTL appropriately**: The TTL should be at least 2x the maximum expected retry interval. If the client retries for up to 1 hour, the TTL should be at least 2 hours. For payment systems, 24 hours is standard.

2. **Client-side key tracking**: The client should track which idempotency keys have been used and stop retrying after the TTL expires. This is a best-effort mitigation.

3. **Business-level deduplication**: Even if the idempotency key expires, business-level deduplication can catch duplicates. For example, if the payment has a unique transaction ID, the database unique constraint prevents duplicate charges.

4. **Reconciliation**: Run periodic reconciliation to detect and correct duplicates. For payment systems, reconcile with the bank statement to find duplicate charges and issue refunds.

5. **Extended TTL for specific operations**: Some operations may need a longer TTL. Allow the client to specify a TTL or use a default TTL that is appropriate for the operation type.

## Question 9: How do you test an idempotency system?

**Answer**: Testing an idempotency system requires testing at multiple levels:

**Unit tests**:
- Test that the idempotency key validator rejects invalid keys.
- Test that the idempotency store correctly stores and retrieves records.
- Test that the lock manager acquires and releases locks correctly.
- Test that the cleanup scheduler removes expired keys.

**Integration tests**:
- Send a request with an idempotency key and verify the response. Send the same request again and verify the response is the same.
- Send two concurrent requests with the same idempotency key and verify that only one request is processed.
- Send a request with an idempotency key, let the TTL expire, and verify that the next request is treated as new.
- Send a request with an idempotency key and a different body on the second request. Verify that the system returns 409 Conflict.

**Chaos engineering tests**:
- Crash the server between processing and storing the result. Verify that the retry is handled correctly.
- Partition the network between the application server and the idempotency store. Verify that the system degrades gracefully.
- Overload the idempotency store with high traffic. Verify that the system throttles correctly.

**Load tests**:
- Send requests at peak throughput with unique idempotency keys. Verify that the system maintains latency SLAs.
- Send requests with a 1% duplicate rate. Verify that the duplicate detection works correctly.
- Fill the idempotency store to capacity. Verify that the eviction policy works correctly.

**Key test scenarios**:

```
Scenario 1: Happy path
  Send: POST /orders with Idempotency-Key: K1
  Expect: 201 Created
  Send: POST /orders with Idempotency-Key: K1
  Expect: 201 Created (same response)

Scenario 2: Concurrent requests
  Send: POST /orders with Idempotency-Key: K2 (concurrent)
  Send: POST /orders with Idempotency-Key: K2 (concurrent)
  Expect: Both return 201 Created with the same response

Scenario 3: TTL expiry
  Send: POST /orders with Idempotency-Key: K3
  Expect: 201 Created
  Wait: TTL + 1 second
  Send: POST /orders with Idempotency-Key: K3
  Expect: 201 Created (new order, different response)

Scenario 4: Different body, same key
  Send: POST /orders with Idempotency-Key: K4, body: {item: "A"}
  Expect: 201 Created
  Send: POST /orders with Idempotency-Key: K4, body: {item: "B"}
  Expect: 409 Conflict
```

## Question 10: Compare and contrast idempotency keys with version-based idempotency.

**Answer**: Both approaches provide idempotency, but they work at different levels and have different characteristics.

**Idempotency keys**:
- Work at the request level: each request is assigned a unique key.
- The key is generated by the client and attached to the request.
- The server stores the key and the response for the duration of the idempotency window.
- Duplicate detection is based on the key: if the key exists, the request is a duplicate.
- The key is independent of the resource state.
- The server must store the response for each key.
- The TTL determines how long the key is retained.
- The client must generate and manage unique keys.

**Version-based idempotency**:
- Works at the resource level: each resource has a version token.
- The version token is stored on the resource and returned in read responses.
- The client reads the current version, applies the update, and specifies the version.
- The server only applies the update if the version matches.
- Duplicate detection is based on the version: if the version has changed, the request is a duplicate.
- The version token is part of the resource state.
- No separate response storage is needed.
- The version is retained indefinitely (as part of the resource).
- The client must read the version before writing.

**Comparison table**:

| Aspect | Idempotency Keys | Version-Based |
| --- | --- | --- |
| Scope | Request-level | Resource-level |
| Client responsibility | Generate unique key | Read version before write |
| Server storage | Separate store for keys | Version on the resource |
| TTL | Configurable (24h typical) | Indefinite (part of resource) |
| Concurrency | Lock per key | Optimistic locking |
| Best for | Create operations | Update operations |
| Overhead | 1 read + 1 write per request | 1 read + 1 conditional write |
| Consistency | Strong consistency required | Strong consistency built-in |

**When to use each**:

Use idempotency keys for create operations (POST) where the resource ID is not known in advance. The client generates a key, sends the request, and the server creates the resource. If the response is lost, the client retries with the same key.

Use version-based idempotency for update operations (PUT, PATCH) where the client knows the resource ID. The client reads the current version, applies the update, and specifies the version. If the update is applied twice, the second attempt sees a different version and is rejected.
