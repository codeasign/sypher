# Deep Dive

This section examines the internal architecture of an idempotency system. You will learn about the components that make idempotency work, the data structures that store idempotency keys, the lifecycle of an idempotency key, and the performance characteristics of idempotency stores at scale.

---

## Internal Architecture

A production idempotency system consists of several interacting components. Each component has a specific responsibility and must meet specific reliability and performance requirements.

### Key Validator

The key validator is the first line of defense. It checks that the idempotency key meets the system's requirements:

- **Format**: The key must match the expected pattern (UUID, base64, or custom format).
- **Length**: The key must be within the minimum and maximum length bounds.
- **Charset**: The key must contain only allowed characters.
- **Uniqueness**: The key must not collide with recently used keys.

The key validator runs on every request. It must be fast: validation takes under 1 millisecond. The validator uses a precompiled regular expression and a length check. No IO is involved.

```python
class IdempotencyKeyValidator:
    KEY_PATTERN = re.compile(
        r"^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$",
        re.IGNORECASE
    )

    def validate(self, key: str) -> ValidationResult:
        if key is None:
            return ValidationResult(False, "Key is required")
        if len(key) > 128:
            return ValidationResult(False, "Key exceeds maximum length")
        if not self.KEY_PATTERN.match(key):
            return ValidationResult(False, "Key format is invalid")
        return ValidationResult(True, None)
```

### Deduplication Store

The deduplication store is the core data store that maps idempotency keys to their responses. It must support three operations:

- **Get(key)**: Retrieve the stored response for a key. Must be strongly consistent.
- **Put(key, value, ttl)**: Store a response for a key with a time-to-live.
- **Delete(key)**: Remove a key and its response (used by the cleanup scheduler).

The store must handle millions of keys concurrently. Each lookup must complete in under 10 milliseconds at the 99th percentile. The store must be highly available and durable.

### Lock Manager

The lock manager prevents concurrent processing of the same idempotency key. Without locks, two requests with the same key could both check the store, both see the key is missing, and both proceed to process the request. This would result in duplicate processing.

The lock manager provides a distributed lock that is:
- **Key-scoped**: One lock per idempotency key.
- **TTL-based**: Locks expire after a configurable timeout to prevent deadlocks.
- **Fair**: Locks should be acquired in a first-come-first-served manner when possible.

### Response Cache

The response cache stores the full response for each idempotency key. When a duplicate request arrives, the system returns the cached response. The cache must store:

- HTTP status code
- Response headers
- Response body
- Content type
- Creation timestamp

### Cleanup Scheduler

The cleanup scheduler removes expired idempotency keys from the store. It runs periodically and deletes keys that have exceeded the idempotency window. The cleanup must be efficient and not impact the performance of the main request path.

## Data Structures

The choice of data structure for the idempotency store has a significant impact on performance, durability, and operational complexity.

### Key-Value Store with TTL

The most common data structure for an idempotency store is a distributed key-value store with TTL support. This is the simplest and most reliable approach.

### Redis-Based Store

Redis is the most common choice for idempotency stores in production systems. It provides:
- In-memory performance with sub-millisecond latency
- Built-in TTL expiry (EXPIRE, PEXPIRE)
- Atomic operations for lock and check-and-set
- Replication for durability

The Redis data structure for idempotency is a simple string key with a JSON value:

```
SET idempotency:abc-123-def-456 '{"status":201,"body":"..."}' EX 86400
```

The `NX` flag on SET provides atomic check-and-set behavior:

```
SET idempotency:abc-123-def-456 '{"status":201,"body":"..."}' EX 86400 NX
```

If the key already exists, the SET NX returns nil, which indicates a duplicate request.

The lifecycle has five phases:

### Phase 1: Creation (Steps 1-3)

The client generates a unique idempotency key. This is typically a UUID version 4 string. The client stores this key in memory so it can be reused on retry. The key is attached to the request as an HTTP header.

### Phase 2: Validation (Steps 4-5)

The server validates the key format and length. Invalid keys are rejected with a 400 Bad Request response. This prevents malformed keys from reaching the store.

### Phase 3: Processing (Steps 6-12)

This is the core phase. The server checks the idempotency store. If the key exists, the cached response is returned immediately. If not, the server acquires a distributed lock on the key, double-checks the store (to handle the race condition where two requests arrive simultaneously), and then processes the request. The result is stored with the key, and the lock is released.

### Phase 4: Retention (Steps 13-15)

The key and its response remain in the store for the duration of the idempotency window, typically 24 hours. During this time, any duplicate request with the same key gets the cached response without reprocessing.

### Phase 5: Deletion (Steps 16-18)

After the TTL expires, the cleanup scheduler removes the key from the store. If a request arrives after deletion with the same key, it looks like a new request to the server. This is why the idempotency window must be long enough to cover the maximum expected retry interval.

## Consistency and State Management

Idempotency systems must handle several consistency challenges. These challenges arise from the distributed nature of modern systems.

### Crash During Processing

If the server crashes between step 10 (process request) and step 11 (store result), the request was processed but the result was not stored. When a retry arrives, the server sees no key in the store, processes the request again, and stores the result. This means the operation was executed twice.

To handle this scenario, the business logic itself must be idempotent or the system must use a write-ahead log. With a write-ahead log, the server writes "processing started for key X" to durable storage before processing begins. If a crash occurs, the retry sees the in-progress entry and can check whether the processing actually completed.

### Race Conditions

Two requests with the same idempotency key can arrive at the same time. Without proper synchronization, both requests check the store, both see the key is missing, and both proceed to process.

The solution is a distributed lock with a double-check pattern: acquire the lock, then check the store again before processing. This ensures that even if two requests arrive simultaneously, only one processes.

### Key Collisions

Two different clients might generate the same idempotency key. The probability of a UUID v4 collision is astronomically low (2^122 possible values), but not zero. For systems that generate keys server-side, collisions are impossible if the server uses a monotonic counter or a database sequence.

The impact of a key collision is serious: the second request with the same key is treated as a duplicate and returns the first request's response. The second request's operation is never performed. This is a data loss scenario.

To mitigate collisions:
- Use UUID v4 with a namespace prefix (for example, "order:uuid").
- For server-generated keys, use a monotonically increasing sequence.
- Monitor for collision rates in production.

### Partial Failures

A partial failure occurs when the request processing succeeds but storing the result fails. The client sees a timeout and retries. The server finds no key in the store and processes again. The payment or order creation runs twice.

Partial failures are the most difficult consistency challenge in idempotency systems. The fundamental issue is that the processing and the storage are not atomic. Various approaches address this:

1. **Two-phase commit**: Coordinate the processing and storage in a distributed transaction. This is correct but slow.

2. **Outbox pattern**: Write the processing result to an outbox table in the same database transaction as the business operation. A separate process reads the outbox and stores the idempotency result.

3. **Compensating transactions**: If duplicate processing is detected, run a compensating transaction to undo the duplicate. This is complex but works for many business scenarios.

## Performance Characteristics

Understanding the performance characteristics of idempotency systems helps you design systems that meet latency and throughput requirements.

### Write Overhead

Every mutating request that carries an idempotency key incurs a write to the idempotency store. This write is in addition to the main business logic write.

The idempotency overhead is typically 5-10% of the total request latency. This is a reasonable cost for the safety it provides. The overhead is dominated by the lock acquisition and store write, which involve network round trips to the idempotency store.

### Storage Requirements

The storage required for idempotency keys depends on the key TTL, the number of requests per day, and the size of each stored response.

For a system with 10M requests per day, a 24-hour idempotency window, and 512 bytes per record:

- 10M keys per day x 512 bytes = 5.12 GB per day
- After 24-hour TTL: ~5.12 GB at steady state
- With replication factor of 3: ~15.36 GB total

For a system with 100M requests per day:

- 100M keys per day x 512 bytes = 51.2 GB per day
- Steady state: ~51.2 GB
- With replication factor of 3: ~153.6 GB total

These storage requirements are modest for modern infrastructure. The cost of storage is typically less than the cost of a single duplicate-payment incident.

### Read-to-Write Ratio

The read-to-write ratio for the idempotency store depends on the retry rate. With a 1% retry rate, for every 100 requests:

- 1 request is a duplicate (1 read, returns cached response)
- 99 requests are new (99 reads + 99 writes)

Read-to-write ratio: 100 reads to 99 writes, approximately 1:1.

As the retry rate increases, the read-to-write ratio shifts toward reads. At a 10% retry rate:

- 10 requests are duplicates (10 reads)
- 90 requests are new (90 reads + 90 writes)

Read-to-write ratio: 100 reads to 90 writes, approximately 1.1:1.

The read-to-write ratio is always close to 1:1 because every unique request requires both a read (check) and a write (store), while every duplicate requires only a read.

## Configuration Parameters

A production idempotency system requires several configuration parameters. The values depend on the specific use case and business requirements.

| Parameter | Default | Range | Description |
| --- | --- | --- | --- |
| Key TTL | 86400 (24h) | 3600-604800 | How long idempotency keys are retained |
| Storage Backend | redis | redis, postgresql, dynamodb | The data store for idempotency records |
| Cleanup Interval | 3600 (1h) | 300-86400 | How often expired keys are purged |
| Lock TTL | 30s | 5-120 | Maximum time a lock is held |
| Max Key Length | 128 | 36-1024 | Maximum characters in an idempotency key |
| Response Size Limit | 64KB | 1KB-1MB | Maximum size of stored response |
| Replication Factor | 3 | 1-5 | Number of copies of each key |
| Consistent Reads | true | true/false | Whether reads must be strongly consistent |

## Summary

The internal architecture of an idempotency system consists of five components: the key validator, deduplication store, lock manager, response cache, and cleanup scheduler. The data structures used for the store (KV store with TTL, LSM tree, Redis) determine the performance characteristics. The lifecycle of an idempotency key spans five phases: creation, validation, processing, retention, and deletion. Consistency challenges include crashes, race conditions, key collisions, and partial failures, each requiring specific mitigations.
