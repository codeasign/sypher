## Internal Architecture

A split-brain prevention system comprises five major components that work together to detect partitions, enforce safety, and manage recovery.

**Partition Detector.** Runs on every node. Each node sends heartbeats to every other node at a configured interval, typically 500 milliseconds to 1 second. The detector tracks response times and missed heartbeats. A node is suspected dead after N consecutive missed heartbeats, configurable from 3 to 10. The detector also tracks asymmetric partitions where Node A can reach Node B but Node B cannot reach Node A.

**Quorum Calculation Engine.** Receives partition information from the detector and calculates which nodes are reachable. Determines if the current group has a majority. In a 5-node cluster, the engine checks: current group size >= 3. If yes, the group is quorate and can continue. If no, the group must step down.

**Fencing Mechanism.** Activated when a new leader is elected or when a node is determined to be unreachable. The fencing mechanism ensures that the old leader cannot serve requests. The mechanism can use physical STONITH (IPMI power-off), network fencing (disable switch port), or logical fencing (lease-based fencing tokens).

**Rejoin Coordinator.** Manages the process when a partitioned node reconnects. The coordinator determines the rejoining node's last known state, identifies any writes the node accepted during the partition, synchronizes from the current leader, and discards the stale writes.

**Conflict Resolver.** Handles data conflicts that arise from writes accepted on both sides. Resolution strategies include last-writer-wins (LWW) using timestamps, CRDT-based merge where possible, application-level merge hooks, and manual reconciliation queues.

## Algorithms

### Majority Quorum Algorithm

The most fundamental split brain prevention algorithm. A cluster of N nodes requires at least N/2 + 1 nodes to agree on any decision.

```
Quorum size = floor(N / 2) + 1

N = 3:  quorum = 2  (tolerates 1 failure)
N = 5:  quorum = 3  (tolerates 2 failures)
N = 7:  quorum = 4  (tolerates 3 failures)
```

The quorum size determines both fault tolerance and availability. Larger quorums provide more safety but reduce availability because more nodes must be online.

The algorithm works in three phases. Phase 1: each node maintains a view of cluster membership. Phase 2: when a node suspects another node is dead, it initiates a quorum check by broadcasting to all nodes. Phase 3: nodes respond with their current view. If the initiating node collects responses from a majority, it can proceed with leadership changes. If not, it must wait.

### Fencing Tokens

Fencing tokens are monotonically increasing numbers that accompany every write operation. Each leader election generates a new, higher token. Nodes reject any write that carries a token lower than the highest token they have seen.

  caption="Fencing tokens increase with each election; stale writes with low tokens are rejected"

   │Election│      │Election│      │Election│
   │Node A  │      │Node B  │      │Node C  │
   │wins    │      │wins    │      │wins    │
   └────────┘      └────────┘      └────────┘

   Event Log:
   ───────────────────────────────────────────────────────
   Token 1: Node A elected. Writes accepted with token=1.
   Token 2: Node A partitioned. Node B elected.
            All writes from token=1 now invalid.
            Network fencing: "Node A, token=1 is expired."
   Token 3: Node B partitioned. Node C elected.
            All writes from token=2 now invalid.

   Write Validation:
   ───────────────────────────────────────────────────────
   Write comes with token=1  → Current token=3 → REJECT
   Write comes with token=2  → Current token=3 → REJECT
   Write comes with token=3  → Current token=3 → ACCEPT

   that has seen token=2 or higher.
`} />

Fencing tokens provide lightweight split brain prevention without hardware manipulation. The algorithm requires a consensus service like ZooKeeper or etcd to generate monotonic tokens. Each node stores the highest token it has seen in a hardened fenced register.

### STONITH via Power Switch / IPMI

STONITH is the most aggressive fencing mechanism. The cluster management software connects to the node's baseboard management controller (BMC) via IPMI and issues a power-off command.

```
STONITH Sequence:
1. Cluster detects Node A is unresponsive
2. Cluster elects Node B as new leader
3. Node B connects to Node A's IPMI interface
   (credentials stored in cluster configuration)
4. Node B issues: ipmitool -H mgmt-A -U admin -P pass power off
5. IPMI confirms: "Chassis Power is off"
6. Node B waits 2 seconds for confirmation
7. Node B verifies Node A is no longer reachable
8. Node B starts accepting writes
```

The sequence takes 200 to 800 milliseconds depending on IPMI response time and network latency. Failures in STONITH — such as incorrect IPMI credentials or unreachable management network — must abort the failover entirely.

### Lease-Based Leadership

Leases provide time-bound leadership authority without requiring consensus for every heartbeat.

```
Lease Algorithm:
- Leader obtains a lease with epoch number E and expiry time T
- Leader must renew lease before T by contacting quorum
- Renew increments T by lease_duration
- If lease expires without renewal, leader must step down
- A node cannot accept writes from lease E if it has seen lease E+1

Configuration parameters:
- lease_duration: 10 seconds (default)
- lease_renewal_interval: 2 seconds
- lease_grace_period: 2 seconds (after expiry before forced step-down)
- clock_skew_tolerance: 200 milliseconds
```

Clock skew is the primary threat to lease correctness. If Node A's clock is 5 seconds behind Node B's clock, a lease that expires at T=100 on Node B expires at T=95 on Node A. Node A might think its lease is still valid when it has already expired. Clock synchronization via NTP with tight bounds is essential.

The cluster initiates conflict resolution. The authoritative leader (Node B) has x = 10. Node A had x = 5. Since fencing token 2 is higher than token 1, Node B's version wins. If Node A accepted writes not seen by Node B, those writes are discarded.

## Consistency and State

### Stale Leader Returning

When a stale leader reconnects claiming authority, the cluster must reject that claim. The fencing token mechanism handles this. The stale node's token is lower than the current cluster token. The stale node's claim is rejected.

Potential issue: if the stale leader has uncommitted writes that contain critical data, those writes are lost. Applications must be designed to tolerate this. Logged but not committed writes are the most common casualty.

### Minority-Side Commits

Writes committed only on the minority side of a partition present a special problem. From the minority side's perspective, these writes were committed durably. From the cluster's perspective, they never happened.

```
Example scenario:
- Node A (leader) commits write W1, acknowledges to client
- Partition occurs before W1 replicates to B and C
- Nodes B and C elect new leader with token 2
- Node A continues with token 1, commits W2 and W3
- Partition heals, Node A fenced
- W1, W2, W3 are all discarded

Impact: Client received acknowledgement for W1
        but W1 is lost. Application must retry.
```

The application must implement retry logic with idempotency keys. The cluster provides no guarantee that acknowledged writes survive a split brain.

### Clock Drift and Overlapping Leases

Clock drift can cause overlapping leases, creating a window where two nodes legitimately believe they hold the lease. This is the most subtle split brain vulnerability.

The overlap window equals the clock skew. If clocks are synchronized to within 200 milliseconds, the overlap window is 200 milliseconds. If clocks drift by 5 seconds, the overlap window is 5 seconds. NTP configuration is a critical safety parameter for any lease-based system.

### Fencing Token Gaps

If fencing tokens are generated incorrectly, a stale leader might hold a token that the current cluster has not yet seen. This can happen when the token generation service itself is partitioned.

```
Token Gap Scenario:
- Token 1: Node A elected leader
- Token 2: Node A partitioned, Node B elected
- Token 3: Node B partitioned, Node C elected
- Token 4: Token generation service (etcd) partitioned
           Node A's group generates Token 4 (different sequence)
           Node C's group generates Token 5 (different sequence)
- Both Token 4 and Token 5 claim to be current
- Nodes cannot determine which is authoritative

Solution: Monotonic token generation from a single source.
          Use Raft-based consensus for token generation.
          Reject tokens from non-quorate groups.
```

## Performance

### Quorum Size Impact

The quorum size directly affects cluster performance and availability.

**3-node cluster:** Quorum = 2. Tolerates 1 failure. Write latency includes 2 acknowledgments (leader + 1 follower). 99.9% availability with 99.0% uptime per node. The smallest practical cluster for fault tolerance.

**5-node cluster:** Quorum = 3. Tolerates 2 failures. Write latency includes 3 acknowledgments. 99.999% availability with 99.0% uptime per node. The standard configuration for production deployments.

**7-node cluster:** Quorum = 4. Tolerates 3 failures. Write latency includes 4 acknowledgments. Used for geo-distributed deployments spanning 3 regions.

### Latency Added by Fencing

Fencing adds measurable latency to the failover process. The table below shows typical latencies for each fencing method.

```
Fencing Method        Typical Latency     Worst Case     Notes
─────────────────     ───────────────     ──────────     ─────
Fencing Token          2-5 ms             50 ms         Software only, no hardware
Network (switch)      50-200 ms           2 s            SSH to switch, disable port
IPMI STONITH         200-800 ms           5 s            Power-off via BMC
PDU STONITH          500-3000 ms         10 s           Power Distribution Unit
```

The total failover time is the sum of partition detection time (3-10 heartbeats at 500ms each = 1.5-5s), election time (200-500ms), fencing time, and leader initialization time (200ms-1s). Total: 2 to 12 seconds.

## Leader Election Protocols

Different leader election protocols provide different guarantees for split brain safety. The three most common protocols each have distinct properties.

### Raft

Raft divides time into terms. Each term has at most one leader. A leader is elected when it receives votes from a majority of the cluster. Raft guarantees that only one leader can exist in a given term.

```
Raft Term Safety:
- Each term has a monotonically increasing term number
- A candidate must receive votes from a majority to become leader
- A leader commits entries by replicating to a majority
- A follower that misses a term transition cannot accept old-term entries
- Formal guarantee: at most one leader per term

Split brain prevention:
- A partitioned candidate cannot get majority votes → cannot become leader
- A leader that loses majority connection → steps down
- Fencing token = term number
- No term can have two leaders
```

Raft's safety comes from the requirement that a candidate must receive its own vote plus N/2 additional votes. In a 5-node cluster, a candidate needs 3 votes. A partitioned node can only gather at most 2 votes (itself plus potentially one other). It cannot form a majority and cannot become leader.

### Paxos

Paxos uses numbered proposals with a similar safety property. Proposers must prepare with a quorum before proposing a value.

```
Paxos Safety:
- Each proposal has a unique, increasing proposal number
- A proposer must receive promise from a majority before proposing
- An acceptor promises not to accept any lower-numbered proposal
- Formal guarantee: at most one value can be chosen per round

Split brain prevention:
- Two proposers on opposite sides of a partition
- Each sends prepare(N) to their accessible nodes
- Only the proposer that reaches a majority first gets promises
- The other proposer's prepare is rejected — it has seen higher number
- Only one proposer can proceed with the accept phase
```

Paxos is more complex than Raft but provides the same fundamental safety guarantee. The practical difference is that Paxos is harder to implement correctly, which is why Raft is more commonly used in modern systems.

### Zab (ZooKeeper Atomic Broadcast)

Zab is the protocol used by ZooKeeper. It is similar to Raft but with different epoch management.

```
Zab Safety:
- Uses epoch numbers (similar to Raft terms)
- Leader synchronization phase ensures all new leader's data is authoritative
- Follower synchronization guarantees no divergence

Zab's unique property:
- A newly elected leader must synchronize with a quorum before accepting writes
- This eliminates the "stale leader" window that exists in some Raft implementations
- Slightly slower failover but stronger safety guarantees
```

## Clock Synchronization and Lease Safety

Lease-based systems depend on accurate clocks. The relationship between clock skew and lease safety is mathematical.

### Clock Skew Bound

```
Let:
  L = lease duration in seconds
  S = maximum clock skew between any two nodes (seconds)
  R = maximum network round-trip time (seconds)
  P = processing time for lease renewal (seconds)

Safe lease duration:
  L > S + R + P + margin

For a typical deployment with NTP synchronization:
  S = 200ms (NTP with local time server, tight bound)
  R = 50ms (p99 intra-datacenter latency)
  P = 10ms (lease renewal processing)

  L > 200 + 50 + 10 = 260ms
  With 50% margin: L > 390ms

  Practical minimum: L = 1 second (includes margin and operational tolerance)
  Recommended production: L = 5-10 seconds

If NTP is unavailable (air-gapped deployment):
  S = 5,000ms (5 seconds drift possible without NTP)
  R = 50ms
  P = 10ms

  L > 5,000 + 50 + 10 = 5,060ms
  With 50% margin: L > 7,590ms
  Practical minimum: L = 10 seconds
```

### Google TrueTime

Google's TrueTime API, used in Spanner, eliminates the clock skew problem by explicitly reporting the clock uncertainty interval.

```
TrueTime:
- Uses GPS and atomic clocks in each datacenter
- Returns an interval [earliest, latest] for the current time
- The actual time is guaranteed to be within this interval
- Interval width: typically 1-7ms

Lease safety with TrueTime:
- Leader holds lease until absolute time T
- New leader cannot be elected until time T + interval_width
- Guarantees no overlap between leases
- Eliminates clock skew as a failure mode

TrueTime is not available outside Google. Alternatives include
AWS Time Sync Service and Azure Precision Time Protocol, which
provide similar but weaker guarantees.
```

## Advanced: Fencing Token Algorithms

### Counter-Based Fencing

The simplest fencing token algorithm. A monotonically increasing counter is stored in a consensus service. Each election increments the counter.

```
Algorithm:
  Counter: integer, starts at 1

  On leader election:
    new_token = CAS(counter, current_value, current_value + 1)
    // CAS = compare-and-swap, guaranteed atomic by consensus service
    leader.fencing_token = new_token

  On write:
    if write.token < node.highest_seen_token:
      reject("stale token")
    else:
      accept_write()
      node.highest_seen_token = write.token

  On rejoin:
    node.highest_seen_token = cluster.current_token
    discard all data written with lower tokens
```

### Hybrid Logical Clocks (HLC)

HLC combines physical clocks with a logical counter to provide fencing-like guarantees without a consensus service.

```
HLC Properties:
- Each event gets a timestamp (physical_time, logical_counter)
- Physical time is the node's clock, bounded by NTP
- Logical counter increments when physical time doesn't advance
- HLC timestamps are partially ordered
- Two events can be concurrent (happened-before relationship unknown)

HLC for split brain prevention:
- Every write gets an HLC timestamp
- On rejoin, nodes compare HLC timestamps
- Writes with timestamps that are not causally related require conflict resolution
- HLC cannot guarantee total ordering, so split brain resolution is possible

HLC is used in CockroachDB for distributed transaction ordering.
It provides causal consistency without a centralized token service.
```
