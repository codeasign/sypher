## The Problem Story

A Friday afternoon. Three database nodes in a cluster. Node A in us-east-1a, Node B in us-east-1b, Node C in us-east-1c. All three form a replicated database cluster with automatic failover. Node A is the elected leader. Nodes B and C are followers replicating from A. The system has been running for 14 months without incident.

A network engineer runs a cable replacement in the datacenter. The patch panel for the switch connecting us-east-1a and us-east-1b is accidentally disconnected. The link drops. Node A can no longer reach Node B. Node B can no longer reach Node A. Node C, sitting in a third availability zone, can still reach both.

But the cluster's heartbeat mechanism uses a simple timeout. Node A detects no heartbeat from Node B for 3 seconds. Node A assumes Node B is dead. Node B detects no heartbeat from Node A for 3 seconds. Node B assumes Node A is dead. Both nodes independently start a leader election.

Node A already is the leader, so it continues as leader. Node B, believing Node A is gone, holds an election with Node C. Node C agrees. Node B becomes the new leader. Now you have two leaders in the same cluster. Both accept writes. Both replicate to their followers.

A user updates their profile picture. The write goes to Node A. The same user changes their display name 2 seconds later. That write goes to Node B. When the network cable is plugged back in, the cluster discovers the conflict. Two different versions of the same user's data exist. Neither side can prove their version is authoritative. Data is corrupted.

This is split brain. A single network failure caused a cascade of incorrect assumptions, resulting in two independent clusters operating on the same dataset. Recovery took 8 hours of manual reconciliation. 3,247 records had conflicting updates. 189 records could not be automatically resolved and required human intervention.

## The Insight

Split brain is a failure mode where a distributed system's nodes divide into two or more groups that each believe they are the authoritative cluster, operating independently and creating data conflicts. The condition arises when a network partition causes nodes to lose communication, and the system lacks a mechanism to prevent multiple groups from acting as primary.

Think of a city with a single mayor. A dispute arises, and two factions each claim the mayor's office. Faction A controls city hall on Monday, Wednesday, Friday. Faction B controls it on Tuesday, Thursday, Saturday. Both sign contracts, both issue permits, both hire staff. When the dispute is resolved, the city discovers it has two conflicting sets of laws and obligations. Which contracts are valid? Which permits are legal? The city cannot function until the authority question is resolved.

In distributed systems, the same problem occurs when multiple nodes believe they hold the leadership role. The system fragments into two independent clusters, each operating on the same data, creating irreconcilable differences.

Split brain is not a theoretical edge case. It is a well-documented failure mode that has caused data loss at major companies including GitLab, MongoDB deployments at scale, Elasticsearch clusters, and PostgreSQL replication setups. Every distributed system that supports automatic failover must guard against split brain.

The fundamental cause is a time-of-check to time-of-use problem. At time T1, Node A checks whether Node B is alive. Node B does not respond. At time T2, Node A concludes Node B is dead and starts acting as sole leader. But between T1 and T2, Node B was not dead — it was merely unreachable. At time T3, Node B becomes reachable again, but now both nodes believe they are the leader.

## Core Vocabulary

**Split Brain.** The condition where a distributed system fragments into two or more independent groups, each believing it is the sole authoritative cluster. Each group continues to accept operations independently, leading to data divergence and conflicts. Split brain is the most dangerous failure mode in distributed systems because it produces silent data corruption rather than a visible outage.

**Network Partition.** A failure that splits the network into two or more segments that cannot communicate with each other. A partition can be caused by a hardware failure, a software bug, a configuration error, or a deliberate maintenance action. Partitions can be symmetric (A cannot reach B and B cannot reach A) or asymmetric (A can reach B but B cannot reach A, known as a partial partition).

**Quorum.** The minimum number of nodes that must agree on a decision for the cluster to act. In a 5-node cluster, a majority quorum is 3 nodes. The node group that has a quorum can elect a leader and accept writes. The group without a quorum must stop accepting writes. Quorum is the primary mechanism for preventing split brain.

**Fencing.** The process of isolating a node that may be faulty or partitioned from the rest of the system. Fencing ensures that a node that was previously the leader cannot continue to serve requests after a new leader has been elected. Fencing is the safety mechanism that prevents split brain from causing data corruption.

**STONITH.** Stands for Shoot The Other Node In The Head. A fencing technique that forcibly removes a node from the cluster by cutting its power or resetting it. STONITH is the nuclear option — it guarantees that a fenced node cannot participate in cluster operations. Common STONITH mechanisms include IPMI, iLO, and network-controlled power distribution units.

**Fencing Token.** A monotonically increasing number issued by the cluster's consensus system. Each leader election increments the fencing token. Any operation from a leader with a stale token is rejected by other nodes. Fencing tokens provide a lightweight mechanism for fencing without physical node isolation.

**Lease.** A time-bound grant of authority. A leader holds a lease from the cluster that expires after a fixed duration. The leader must continuously renew the lease by communicating with the cluster. If the lease expires, the leader's authority ends automatically. Leases prevent split brain by ensuring that a leader cannot hold authority indefinitely without cluster communication.

**Consensus.** A distributed algorithm that ensures all nodes in a cluster agree on a single value or state. Consensus algorithms like Paxos and Raft provide strong guarantees against split brain by ensuring that only one leader can be elected at a time. Consensus is the foundation of quorum-based prevention.

**Quorum Loss.** The state where a cluster loses enough nodes that no group can form a majority quorum. In this state, the cluster cannot elect a leader or accept writes. Quorum loss is the price of safety — the cluster becomes unavailable to prevent the greater danger of split brain.

**Rejoin / Resync.** The process by which a node that was partitioned from the cluster reconnects and synchronizes its state with the current leader. The rejoining node must discard any writes it accepted during the partition, as those writes may conflict with the authoritative cluster's state. Rejoin and resync are critical recovery procedures.

**Partial Cluster.** A subset of nodes that can still communicate with each other after a partition. In a 5-node cluster, a partition might create a partial cluster of 3 nodes and another partial cluster of 2 nodes. The 3-node partial cluster has a quorum. The 2-node partial cluster does not.

**Witness / Arbiter.** A non-data node that participates in elections solely to provide a quorum vote. Witnesses have no data and serve only as tiebreakers. A witness enables a 2-node cluster to achieve a majority quorum (2 out of 3) and prevent split brain. Witnesses are commonly used in clusters with an even number of data nodes.

### Step 3: Both Sides Suspect Other Dead

Node A's heartbeat timer expires after 3 seconds of no communication from Node B or Node C. Node A marks both nodes as dead. Node A does not crash — it continues running, fully operational, but isolated.

Node B's heartbeat timer expires for Node A. Node B marks Node A as dead. Node C also marks Node A as dead. Nodes B and C, still connected, form their own group.

Now x = 5 on Node A. x = 10 on Nodes B and C. The data has diverged.

### Step 5: Both Sides Accept Writes

With two leaders, the cluster has entered a split brain state. Both sides accept writes, and the data diverges further with each operation.

## Variants of Prevention

### Quorum-Based Prevention

The most common approach. A cluster requires a majority of nodes (N/2 + 1) to agree on any decision. In a 5-node cluster, at least 3 nodes must agree to elect a leader. Without a majority, a group cannot form an authoritative cluster.

  caption="Quorum-based prevention: the group without a majority cannot act as authoritative"

The disadvantage: you lose availability when you lose enough nodes. A 5-node cluster can survive 2 node failures. On the 3rd failure, the cluster goes read-only. The trade-off is availability for safety.

### Fencing-Based Prevention

Fencing physically or logically removes the partitioned node from the cluster. The most aggressive form is STONITH (Shoot The Other Node In The Head), which cuts power to the node.

### Lease-Based Prevention

Leases provide time-bound authority. A leader's lease expires after a fixed duration. If the leader cannot contact the cluster to renew the lease, it must step down. Leases prevent split brain by ensuring a leader cannot hold authority indefinitely without cluster communication.

A witness node participates in elections but does not store data. Witnesses break ties when the data nodes are evenly split. In a 2-node cluster, a witness provides the third vote needed for a majority.

## Real-World Scenario Walkthrough

A complete walkthrough of a split brain incident from detection to resolution helps cement the mental model.

### Setup

A financial application runs on a 3-node PostgreSQL cluster with Patroni for high availability. The cluster spans 3 availability zones in us-east-1. Node A is in us-east-1a and is the leader. Node B is in us-east-1b. Node C is in us-east-1c. All three use synchronous replication. Writes are acknowledged only after all three nodes confirm.

### Incident Timeline

**T=0s.** The cluster is healthy. The application processes 2,400 transactions per second. Replication lag is 0 milliseconds. Heartbeats fire every 500ms. Lease expires in 10 seconds and is renewed every 2 seconds.

**T=47s.** An AWS network event causes increased packet loss between us-east-1a and us-east-1b. Two consecutive heartbeats from Node A to Node B time out. Node A does not act yet — the suspicion threshold is 3 missed heartbeats. Node B does not act either.

**T=48.5s.** The third heartbeat from Node A to Node B times out. Node A marks Node B as suspect. Node A sends a probe to Node C asking: "Can you reach Node B?" Node C confirms it can reach Node B. Node A waits — the partition is asymmetric. Node B might still be alive.

**T=50s.** Full partition establishes. Node A cannot reach Node B or Node C. Node A marks both as dead. Node A increments its own term and continues as leader with fencing token 6. On the other side, Node B and Node C can communicate. They detect Node A is unreachable. They hold an election. Node B has higher priority and becomes leader with fencing token 7. Node C becomes a follower.

**T=50.2s.** Node B triggers STONITH fencing. Node B's management controller sends an IPMI power-off command to Node A's BMC. The command succeeds. Node A powers off. However, between T=50s and T=50.2s, Node A accepted 480 transactions on token 6. Those transactions are committed on Node A's local disk but have not replicated.

**T=50.5s.** Node B confirms Node A is offline (no heartbeat, no ping, no IPMI power status). Node B starts accepting writes as the sole leader. Node C follows. The cluster is operational again.

**T=120s.** An operator reboots Node A. Node A comes back online with a full transaction log. Node A contains 480 transactions that the cluster has never seen. Node A attempts to rejoin the cluster with fencing token 6. The cluster's current token is 7. Node A's token is stale. The cluster rejects Node A's claim of authority.

**T=125s.** Patroni on Node A detects the token mismatch. Node A transitions to a replica role. Node A begins a full pg_basebackup from Node B, downloading the complete database state. The 480 transactions on Node A are overwritten. They are permanently lost. The application must retry those transactions using idempotency keys.

### Data Loss Analysis

Of the 480 transactions on Node A:
- 312 were committed on Node B before Node A went offline (duplicates, safe)
- 168 were committed on Node A but never reached Node B
- Of those 168, the application had idempotency keys for 142
- The remaining 26 transactions required manual reconciliation
- 4 transactions had to be re-entered manually because the idempotency key store was also affected by the partition

### Key Takeaway

The cluster handled the partition correctly. Quorum and fencing prevented split brain. But 168 transactions were lost because they were acknowledged by Node A before the partition was complete. This is the fundamental cost of choosing consistency over availability in a CP system. The application was designed for this — idempotency keys handled 142 of 168 losses automatically.

## Common Misconceptions

**Misconception: Split brain only happens in leader-based systems.**
Reality: Peer-to-peer systems like Cassandra can produce split-brain-like conditions where two nodes accept writes for overlapping token ranges. The effect is the same — data divergence — even though there is no leader election involved.

**Misconception: More nodes always means safer from split brain.**
Reality: Large clusters introduce more failure modes. In a 9-node cluster, partitions are more likely because there are more network paths. The quorum size is 5, but a partition can create multiple groups with various sizes. Configuration complexity increases. Operational errors become more common.

**Misconception: Cloud providers handle split brain automatically.**
Reality: Managed services like AWS RDS Multi-AZ handle infrastructure-level split brain but do not protect against application-level data conflicts. If your application reads from one AZ and writes to another during a partition, the managed service cannot prevent data divergence.

**Misconception: Split brain is always caused by network failures.**
Reality: Software bugs, CPU overload, memory pressure, and garbage collection pauses can all produce the same symptoms as a network partition. A node that is overwhelmed may appear dead to other nodes, triggering the same failure mode.

## Mental Model Check

### Question 1
You have a 5-node cluster using quorum-based voting. A network partition creates two groups: Group A with 3 nodes, Group B with 2 nodes. What happens?

Group A has a quorum (3 out of 5) and can continue operating. Group B lacks a quorum (2 out of 5) and must stop accepting writes. Group B cannot elect a leader. Any writes sent to Group B are rejected. When the partition heals, Group B resynchronizes from Group A's leader.

### Question 2
What is the difference between fencing and STONITH?

Fencing is the general concept of isolating a potentially faulty node from the system. STONITH is a specific fencing method that forcibly powers off the node. Other fencing methods include disabling the node's network port, revoking its access tokens, or using fencing tokens in a consensus algorithm. STONITH is the most aggressive — it provides the strongest guarantee but also causes hardware-level disruption.

### Question 3
A 3-node cluster loses one node entirely to a hardware failure. Can split brain occur?

No. With 2 remaining nodes, one of two things happens. If the cluster uses quorum-based voting (needs 2 out of 3), it continues with 2 nodes. No split brain because there is only one group. If the network also partitions the remaining 2 nodes from each other, both would lack quorum and stop — preventing split brain but causing complete unavailability.

### Question 4
Why are lease durations a critical configuration parameter?

Short lease durations cause false failovers. A transient network spike of 200 milliseconds can cause a leader's lease to expire, triggering an unnecessary election and failover. Long lease durations mean the cluster takes longer to detect a real leader failure and fail over. A 30-second lease means 30 seconds of downtime during a leader failure. The correct lease duration depends on the network's reliability characteristics and the application's downtime tolerance.

### Question 5
Can a 2-node cluster ever be safe from split brain without a witness?

No. Two nodes can each believe the other is dead and continue independently. Without a third vote, there is no way to break the tie. A witness or arbiter node is mandatory for any cluster with an even number of nodes. Some configurations use a shared disk or external lock service as a tiebreaker, but the principle is the same — a third decision entity is required.
