## Level-by-Level Expectations

### Entry Level

At this level, interviewers expect you to understand what split brain is and why it is dangerous. You should be able to explain the basic scenario and the primary prevention mechanisms.

```
What you must know:
- Definition: two or more nodes acting as leader independently
- Primary cause: network partition
- Main prevention: quorum-based voting
- Key danger: silent data corruption

What you should be able to do:
- Draw a 3-node cluster with a partition
- Explain why both sides elect leaders
- Describe quorum as floor(N/2)+1
- Name two prevention strategies (quorum and fencing)

Example interview question:
"Your database has three nodes. A network partition separates
Node A from Nodes B and C. What happens?"
```

### Mid Level

Mid-level candidates should understand the trade-offs and be able to design a split brain prevention system for a given scenario.

```
What you must know (adds to entry level):
- Quorum calculation and fault tolerance implications
- Fencing types: STONITH, network fencing, token fencing
- Lease-based leadership and clock skew concerns
- Witness/arbiter nodes for even-sized clusters
- MongoDB, Cassandra, or PostgreSQL split brain behavior

What you should be able to do:
- Design a 5-node cluster with quorum-based prevention
- Compare fencing vs. quorum for a specific use case
- Calculate availability impact of different cluster sizes
- Discuss recovery strategies for split brain scenarios
- Recommend cluster sizing for given availability requirements

Example design question:
"Design a highly available database cluster that can tolerate
two simultaneous node failures without split brain."
```

### Senior Level

Senior candidates are expected to have deep expertise and experience with production split brain incidents.

```
What you must know (adds to mid level):
- Detailed failure modes (undetected split brain, wrong fencing)
- Recovery runbooks and data reconciliation strategies
- Cross-region cluster design with quorum zones
- CAP theorem implications for split brain prevention
- Leader election protocol internals (Raft, Paxos, Zab)
- Asymmetric partition handling
- Hinted handoff and read repair in no-leader systems

What you should be able to do:
- Design a multi-region cluster with cross-region quorum
- Analyze a production split brain incident root cause
- Design automated recovery procedures
- Evaluate trade-offs between different consensus protocols
- Design conflict resolution for CRDT-based systems
- Lead a post-mortem for a split brain incident

Example design question:
"Design a global financial transaction system that must
maintain strong consistency across three regions while
achieving 99.995% availability."
```

## Phrase Bank

### Describing the Problem

"Split brain occurs when a network partition causes nodes in a distributed system to divide into two or more groups, each believing they are the authoritative cluster and operating independently."

"The fundamental challenge is the time-of-check to time-of-use problem — by the time a node confirms another node is unreachable, the unreachable node may have already become the leader on its side."

"Silent data corruption is the most dangerous aspect of split brain because the system continues to respond to requests without indicating that an error has occurred."

### Describing Prevention Mechanisms

"Quorum-based prevention requires a majority of nodes to agree on any decision, ensuring that only one group can achieve consensus."

"Fencing provides a guarantee that a stale leader cannot continue operating — either through physical power-off via STONITH or through logical fencing tokens that invalidate stale writes."

"Lease-based leadership uses time-bound authority. A leader must continuously renew its lease by communicating with the cluster. If the lease expires, authority is automatically lost."

"Witness nodes provide tie-breaking votes for clusters with an even number of data nodes, preventing the stalemate that would otherwise lead to split brain."

### Describing Recovery

"Recovery from split brain requires fencing the stale leader, identifying writes committed on the minority side, and determining which data to keep through application-level conflict resolution."

"The key metric during recovery is recovery time objective — how long until the system is operational — and recovery point objective — how much data is lost."

"Idempotency keys allow applications to safely retry writes after split brain recovery without creating duplicate operations."

### Describing Trade-offs

"Stronger split brain prevention reduces availability because the cluster requires more nodes to reach consensus."

"Fencing provides strong safety guarantees at the cost of hardware complexity and potential data loss from the fenced node."

"The optimal cluster size balances fault tolerance against availability probability. Five nodes offers a good trade-off for most production systems."

## Red Flags

### Red Flag 1: "Split brain won't happen to us"

This indicates a lack of understanding about network reliability. All distributed systems experience partitions. It is not a question of if, but when.

```
Interviewer probe: "What happens when the network switch fails?"
Bad answer: "That's very unlikely. Our network is reliable."
Good answer: "We design for partition tolerance. Our cluster uses
quorum-based voting with a 5-node configuration that can tolerate
2 simultaneous failures. Fencing ensures safety during recovery."
```

### Red Flag 2: Two-node cluster without quorum mechanism

Candidates who propose two-node clusters without addressing split brain do not understand the fundamental problem.

```
Bad answer: "We use two nodes for redundancy. If one fails, the
other takes over. This gives us high availability."
Good answer: "A two-node cluster requires a witness or arbiter
to prevent split brain. We would use three nodes, or two nodes
plus a witness, to ensure a majority can be achieved."
```

### Red Flag 3: Ignoring clock skew

Candidates who design lease-based systems without addressing clock synchronization miss a critical vulnerability.

```
Bad answer: "Leases expire after 10 seconds. The leader must
renew before expiry. This prevents split brain."
Good answer: "Leases expire after 10 seconds, but we also
configure a 200-millisecond clock skew tolerance and use NTP
with tight bounds. The effective lease is 10.2 seconds to
account for clock drift between nodes."
```

### Red Flag 4: No recovery plan

Candidates who focus entirely on prevention without addressing recovery are missing the complete picture.

```
Bad answer: "With proper quorum configuration, split brain
is prevented. Recovery is not a concern."
Good answer: "Prevention first — quorum with 5 nodes. But
we also have an automated recovery runbook that fences the
stale leader, reconciles data using last-token-wins, and
resynchronizes the rejoining nodes."
```

### Red Flag 5: Single prevention mechanism

Relying on one safety mechanism creates a single point of failure.

```
Bad answer: "STONITH prevents split brain. If a partition
occurs, we power off the old leader."
Good answer: "We use defense in depth. Quorum prevents dual
leadership during partitions. Fencing tokens prevent stale
writes from being accepted. STONITH provides physical isolation
as a last resort. Leases ensure automatic time-bound authority."
```

### Red Flag 6: Ignoring asymmetric partitions

Candidates who only model symmetric partitions are missing a common real-world scenario.

```
Bad answer: "When a partition occurs, both sides detect it
immediately and the minority side stops accepting writes."
Good answer: "Asymmetric partitions are common. Node A might
reach Node B but B cannot reach A. We use suspicion-based
detection that requires confirmation from multiple nodes. A
single node reporting a partition does not trigger failover."
```

### Red Flag 7: Manual override without safety

Candidates who suggest manual intervention without verifying fencing state reveal operational inexperience.

```
Bad answer: "Our operators can manually restart both sides
to recover from split brain."
Good answer: "We have an automated recovery procedure. Manual
intervention follows a strict runbook that verifies fencing
state, identifies the authoritative side, and brings nodes
back one at a time. Simultaneous restart is prohibited."
```

### Red Flag 8: No monitoring for split brain

Candidates who do not mention monitoring for split brain conditions are missing operational awareness.

```
Bad answer: "The database cluster handles split brain. We
monitor general cluster health."
Good answer: "We monitor fencing token consistency, quorum
health, heartbeat success rates, and election frequency.
Dedicated alerts trigger when any indication of split brain
is detected, before data corruption occurs."
```

## Practice Drills

### Drill 1: Quorum Calculation

Given a cluster of N nodes, calculate the quorum size, fault tolerance, and availability probability.

```
Example: 7-node cluster, 99.5% uptime per node
- Quorum size: floor(7/2) + 1 = 4
- Fault tolerance: 3 nodes can fail
- Availability: P(at least 4 nodes up) = sum of binomial
  P(4) + P(5) + P(6) + P(7) = 99.97%

Try: 5 nodes, 99% uptime
- Quorum: 3, tolerance: 2, availability: 99.9%

Try: 3 nodes, 99.9% uptime
- Quorum: 2, tolerance: 1, availability: 99.9997%
```

### Drill 2: Fencing Method Selection

```
Scenario:
- 3-node PostgreSQL cluster
- Running on bare metal in a single datacenter
- Management network available
- IPMI available on all nodes
- RTO: 30 seconds, RPO: zero (no data loss)

Recommended fencing:
- Primary: STONITH via IPMI (200-800ms, strongest guarantee)
- Secondary: Network fencing via switch (if IPMI fails)
- Tertiary: Fencing tokens (software safety net)

RTO achieved: ~5 seconds (detection) + ~1 second (fencing)
RPO: zero (STONITH ensures no writes after fencing)

Scenario:
- 5-node etcd cluster
- Running on Kubernetes pods
- No IPMI available
- RTO: 10 seconds, RPO: near-zero

Recommended fencing:
- Fencing tokens via term number (Raft mechanism)
- Pod eviction via Kubernetes API
- No STONITH possible (no physical access)

RTO achieved: ~3 seconds (election timeout)
RPO: near-zero (Raft commits to majority before ack)
```

### Drill 3: Recovery Walkthrough

Walk through the recovery process for a 5-node cluster that experienced a split brain.

```
Situation:
- 5-node MongoDB replica set
- Network partition: 3 nodes in AZ1, 2 nodes in AZ2
- Both sides elected primaries
- 147 writes on AZ2 side that did not replicate to AZ1
- Partition lasted 4 minutes and 23 seconds

Recovery steps:
1. Verify the authoritative side (AZ1 has 3 nodes, higher term)
2. Fence the stale primary in AZ2 (STONITH via IPMI)
3. Export the 147 unique writes from AZ2 side
4. Analyze each write for conflicts with AZ1 state
5. Apply writes with no conflicts (98 of 147)
6. Queue conflicting writes (49) for manual review
7. Rejoin the 2 AZ2 nodes as secondaries
8. Trigger full resync from AZ1 primary
9. Verify all nodes have identical data (checksum)
10. Return cluster to normal operations
11. Resolve the 49 conflicting writes with application team

Total recovery time: ~15 minutes
Data loss: 0 writes (all recovered via export)
Application impact: read-only mode during recovery
```

### Drill 4: Architecture Design

Design a split brain-proof architecture for a financial transaction system.

```
Requirements:
- 99.999% availability
- Strong consistency (no conflicting transactions)
- Cross-region deployment (3 regions)
- Sub-100ms write latency within a region
- 50,000 transactions per second

Design:
- 7-node cluster per shard (quorum = 4)
- 3 regions: us-east-1 (3 nodes), us-west-2 (2 nodes),
  eu-west-1 (2 nodes)
- Preference zones: us-east-1 preferred leader
- Raft-based consensus for leader election
- Write latency: local region 50ms, cross-region 200ms
- Read: local region (eventual consistency for local reads,
  strong consistency via quorum reads for critical reads)
- Fencing: fencing tokens + IPMI STONITH
- Recovery: automated with manual approval for conflict
  resolution on financial transactions

Availability: 99.9996% (with 99.99% per-node uptime)
Failure tolerance: 3 simultaneous node failures
Max region failure: 1 region can fail completely
```

### Drill 5: Troubleshooting a Split Brain Incident

```
You are on call. Alert: "cluster split brain detected" on a
3-node PostgreSQL cluster running Patroni on Kubernetes.

Step 1: Gather information
- Connect to each pod: kubectl exec -it db-n-0 -- patronictl list
- Check etcd status: kubectl exec -it etcd-0 -- etcdctl endpoint health
- Compare fencing tokens across nodes

Step 2: Identify the authoritative side
- Which node has the highest fencing token? That node was
  elected most recently.
- Which side has more nodes? The side with 2+ nodes has quorum.
- Which side has the most recent WAL position? The side with
  the highest LSN has the most committed data.

Step 3: Verify fencing state
- Check if STONITH was attempted: grep STONITH /var/log/cluster.log
- Check if the stale leader pod is running: kubectl get pods
- If stale leader is still running, fence it:
  kubectl delete pod db-n-0 --grace-period=0 --force

Step 4: Recover data
- Export any unique WAL segments from the fenced node
- Compare against the authoritative cluster's state
- Identify any writes that need manual reconciliation

Step 5: Restore cluster
- The fenced node will be restarted by StatefulSet controller
- Patroni will detect the node is a replica and sync from leader
- Verify replication lag reaches 0

Expected recovery time: 5-15 minutes
Potential data loss: Any writes acknowledged by the fenced node
  but not replicated to the quorum side
```

## Common Interview Questions

### Question 1: "Explain split brain to a non-technical manager"

"Split brain is like having two managers who each think they are in charge. If the phone line between them goes dead, both start making independent decisions. When the phone line comes back, you discover they made conflicting decisions about the same things. In a database cluster, this means both sides accepted writes, and now we have two versions of the data. We prevent this by requiring a majority of managers to agree before anyone can make decisions, and by having a way to forcibly remove a manager who disconnects."

### Question 2: "Design a database cluster that can survive an AZ failure without split brain"

"Use 3 nodes in 3 different AZs. Configure quorum-based leader election requiring 2 out of 3 votes. Use synchronous replication — the leader does not acknowledge a write until at least one follower has confirmed. Use fencing tokens to prevent stale leaders from writing. When one AZ fails, the remaining 2 nodes have quorum and continue. The failed AZ's data is recovered when the node rejoins by synchronizing from the leader."

### Question 3: "How do you test split brain prevention?"

"Run chaos experiments. Block network traffic between nodes at the switch level. Simulate asymmetric partitions where one node can send but not receive. Inject packet loss and latency spikes. Test STONITH failure by simulating an unresponsive IPMI interface. Measure detection time, fencing time, and recovery time. Verify that no data is corrupted or lost after each test. Run these tests in staging before production."

### Question 4: "What happens if you set quorum too low?"

"Setting quorum below majority increases split brain risk. In a 5-node cluster, setting quorum to 2 means any 2 nodes can form a leadership group. A partition splits the cluster into 2-node and 3-node groups — both have quorum, both elect leaders, and both accept writes. The cluster operates with two leaders and diverging data. This is equivalent to disabling split brain protection entirely."

### Question 5: "Can split brain happen in a single-region deployment?"

"Yes. A misconfigured network switch, a faulty NIC, or even a software bug in the heartbeat mechanism can cause a partition within a single datacenter. Availability zone failures within a region are also common. Single-region deployments are not immune to split brain. The only difference from multi-region is latency and the probability of full-region failures."
