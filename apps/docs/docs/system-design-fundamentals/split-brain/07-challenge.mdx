## Tier 1: Foundational Challenges

### Challenge 1: Quorum Calculation

A cluster has 5 nodes. Each node has 99.9% uptime. Calculate the probability that the cluster loses quorum.

<AsciiDiagram
  id="split-brain/challenge-t1c1"
  title="Tier 1 Challenge 1 — Quorum Failure Probability"
  alt="Five-node cluster with three nodes up and two nodes down, showing the boundary between quorum and quorum loss"
  caption="Calculate the probability that 3 or more nodes fail simultaneously in a 5-node cluster"
 content={`
   5-Node Cluster — Quorum Boundary
   ─────────────────────────────────
   Quorum = 3 nodes needed
   ┌─────────────────────────────────────────────────┐
   │  Up Nodes    │  Down Nodes   │  Quorum Status   │
   ├─────────────────────────────────────────────────┤
   │  5           │  0            │  Quorate         │
   │  4           │  1            │  Quorate         │
   │  3           │  2            │  Quorate         │
   │  2           │  3            │  QUORUM LOST     │
   │  1           │  4            │  QUORUM LOST     │
   │  0           │  5            │  QUORUM LOST     │
   └─────────────────────────────────────────────────┘
   Quorum lost when 3 or more nodes fail.
`} />

**Model answer:**

The probability of quorum loss is the sum of probabilities that exactly 3, 4, or 5 nodes fail.

P(node failure) = 1 - 0.999 = 0.001

Using binomial distribution:
P(3 failures) = C(5,3) * 0.001^3 * 0.999^2 = 10 * 1e-9 * 0.998001 = 9.98e-9
P(4 failures) = C(5,4) * 0.001^4 * 0.999^1 = 5 * 1e-12 * 0.999 = 4.995e-12
P(5 failures) = C(5,5) * 0.001^5 * 0.999^0 = 1 * 1e-15 = 1e-15

P(quorum loss) = 9.98e-9 + 4.995e-12 + 1e-15 approximately 1e-8

The cluster loses quorum with probability 0.000001% or about once every 114,000 years if each attempt is considered independent. In practice, correlated failures (power outage, network switch failure) make quorum loss much more likely.

### Challenge 2: Three-Node Cluster Failure

Node A in a 3-node cluster stops responding to heartbeats. Nodes B and C can communicate with each other but not with Node A. Describe what happens with and without a proper quorum configuration.

**Model answer:**

With proper quorum configuration (quorum = 2):
- Nodes B and C form a group of 2. They have quorum (2 >= 2).
- Node B or C is elected leader. The cluster continues.
- Node A, alone, has 1 node. It has no quorum (1 < 2).
- Node A rejects all writes. It cannot elect a leader.
- When Node A reconnects, it synchronizes from the new leader.

Without proper quorum configuration (quorum = 1 or no quorum enforcement):
- Node A detects heartbeat loss from B and C. Node A assumes both are dead.
- Node A continues as leader, accepting writes.
- Nodes B and C detect heartbeat loss from A. They assume A is dead.
- Nodes B and C elect their own leader.
- Both sides accept writes. Data diverges.
- Split brain occurs. Recovery requires manual reconciliation.

### Challenge 3: Fencing Token Sequence

A cluster uses fencing tokens. The current token is 5. A write arrives with token 3. Should the write be accepted?

**Model answer:**

No. The write carries token 3, which is lower than the current token 5. This indicates the write originated from a stale leader that was elected in a previous term. The write must be rejected. Any data from a stale leader is potentially based on outdated cluster state and could cause conflicts.

The node receiving the write should respond with an error indicating a stale token. The client should retry with the current leader, obtaining a fresh token.

## Tier 2: Intermediate Challenges

### Challenge 4: Multi-Region Quorum Design

Design a split brain prevention system for a global database cluster spanning 3 regions. Each region has 2 data nodes. Determine the quorum size, fault tolerance, and write latency implications.

           │                        │                         │
           │───────────────250ms latency─────────────────────►│
   Total nodes: 6
   Quorum needed: 4 (floor(6/2)+1)
   Regional quorum zones: 4 votes from any combination
   of regions that includes at least 4 nodes.
`} />

**Model answer:**

The cluster has 6 nodes. Quorum = floor(6/2) + 1 = 4. Any group of 4 or more nodes can form a quorum.

Fault tolerance: The cluster can lose 2 nodes and still have quorum (4 nodes remaining). However, if a full region (2 nodes) is lost, 4 nodes remain across 2 regions, which still has quorum. If 2 regions are lost (4 nodes), 2 nodes remain in 1 region, which lacks quorum.

Write latency: A write must be acknowledged by 4 nodes. If the leader is in us-east-1, the fourth acknowledgment could come from eu-west-1 (80ms) or ap-southeast-1 (250ms). The practical write latency is 80ms (ack from 4 nodes: 2 local + 2 from eu-west-1).

Improvement: Use a witness node in a third region instead of a full data node. This reduces the data node count to 5, quorum to 3, and write latency to local region only (2 local nodes + 1 witness acknowledgment).

### Challenge 5: STONITH Failure Recovery

A 2-node cluster with STONITH experiences a partition. The standby attempts to fence the primary but the IPMI command fails. What happens?

**Model answer:**

The standby cannot safely promote itself to primary. Without successful fencing, the standby cannot guarantee that the primary has stopped accepting writes. Promoting without fencing creates a high risk of split brain.

The cluster enters a failed state:
- Both nodes are operational but cannot communicate.
- The primary continues accepting writes (if it believes it has quorum).
- The standby cannot promote itself.
- The cluster is effectively in a degraded state.

Recovery options:
1. Manual intervention: An operator verifies the primary's status via an independent channel, confirms the primary is really offline, and manually promotes the standby.
2. Fallback fencing: If IPMI fails, the cluster attempts secondary fencing methods (network switch port disable, management network kill).
3. Forced promotion: The standby promotes itself after a configurable timeout, accepting the risk of split brain. This should only be used when availability is prioritized over consistency.

The safest approach is to call an operator. Automated fallback to a secondary fencing method is the next best option. Never promote without fencing in a system that requires strong consistency.

### Challenge 6: Lease Duration Configuration

A ZooKeeper cluster has a lease duration of 10 seconds. The network between nodes has a 99.9th percentile latency of 50 milliseconds and a maximum observed latency of 2 seconds during GC pauses. Is the lease duration appropriate?

**Model answer:**

The lease duration must account for:
- Network round-trip time: 50ms at p99, up to 2s at max
- Clock skew: typically 200ms with NTP synchronization
- Processing time: 10-50ms for lease renewal processing
- Safety margin: 50% of base duration

Total required: 2s (max network) + 0.2s (clock skew) + 0.05s (processing) = 2.25s multiplied by 1.5 (safety) = 3.375s

The 10-second lease duration is more than sufficient. It provides a 7-second safety margin beyond the worst-case scenario. The trade-off is that failover takes up to 10 seconds (lease expiry) + election time (~200ms) = 10.2 seconds total.

If the application requires faster failover, the lease could be reduced to 5 seconds. This still provides a 1.75-second safety margin. Below 5 seconds, the risk of false failovers from GC pauses becomes significant.

## Tier 3: Advanced Challenges

### Challenge 7: CRDT-Based Split Brain Resolution

Design a distributed counter system that uses CRDTs (Conflict-free Replicated Data Types) to handle split brain. Explain how the CRDT ensures that merging two divergent states produces a correct result.

**Model answer:**

A CRDT-based counter uses a data structure that is mathematically guaranteed to converge when merging concurrent updates. The most common approach is the G-Counter (Grow-only Counter) or PN-Counter (Positive-Negative Counter).

```
G-Counter design:
- Each node has an array of N counters, one per node
- Node i can only increment its own counter in the array
- A merge operation takes the element-wise maximum of two arrays
- The total value is the sum of all elements in the array

Example:
  Node A's state:  [5, 0, 0]  (Node A has 5 increments)
  Node B's state:  [0, 3, 0]  (Node B has 3 increments)

  Partition occurs:
  Node A increments:  [6, 0, 0]  (total = 6)
  Node B increments:  [0, 4, 0]  (total = 4)

  Partition heals, merge:
  Merge: [max(6,0), max(0,4), max(0,0)] = [6, 4, 0]
  Total after merge: 6 + 4 + 0 = 10

  The correct total is 10 (6 from A's side, 4 from B's side).
  The CRDT guarantees this result regardless of merge order.
```

The PN-Counter extends this with separate arrays for increments and decrements. The value is the sum of increments minus the sum of decrements.

CRDTs eliminate the need for split brain prevention because they are designed to handle concurrent updates safely. However, they are limited to data types that have well-defined merge semantics. Complex data structures like ordered lists or maps with nested fields are harder to implement as CRDTs.

### Challenge 8: Hybrid Quorum and Fencing

Design a system that uses both quorum and fencing, with a fallback hierarchy. When does each mechanism activate? How do they interact?

**Model answer:**

A hybrid system uses quorum as the primary safety mechanism and fencing as the enforcement mechanism. The hierarchy is:

```
Layer 1: Quorum (always active)
- Every write requires quorum acknowledgment
- During partitions, the minority side automatically stops
- This prevents split brain in 99.9% of cases

Layer 2: Fencing tokens (always active)
- Every write carries a fencing token
- Nodes reject stale tokens even if quorum is satisfied
- This prevents stale leaders from reconnecting

Layer 3: STONITH (activated on election)
- When a new leader is elected, STONITH fences the old leader
- This provides physical isolation
- Activated even if quorum is satisfied

Layer 4: Network fencing (activated on STONITH failure)
- If IPMI fails, disable the old leader's switch port
- Requires management network access to the switch
- Provides alternative isolation path

Layer 5: Manual intervention (last resort)
- If all automated fencing fails, call an operator
- Operator verifies status via independent channel
- Operator manually shuts down the stale leader
```

Interaction between layers:
- Quorum and fencing tokens are complementary. Quorum prevents dual leadership during partitions. Fencing tokens prevent stale data from being accepted after recovery.
- STONITH is triggered by the new leader after election. The new leader does not accept writes until STONITH completes successfully.
- If STONITH fails, the cluster enters a reduced state. It can continue serving reads but requires operator approval before accepting writes.
- Layers 1-4 are automated. Layer 5 requires human intervention. The system should never promote a new leader without at least one fencing layer succeeding.

### Challenge 9: ZooKeeper Cluster Disaster Recovery

A 5-node ZooKeeper cluster experiences a cascading failure. Three nodes fail simultaneously in different AZs. The remaining 2 nodes can still communicate. Describe the recovery process and data integrity guarantees.

**Model answer:**

The 5-node ZooKeeper cluster has quorum = 3. With 2 nodes remaining, quorum is lost. The following occurs:

1. The 2 remaining nodes detect quorum loss. They stop accepting writes. Reads may or may not continue depending on configuration (ZooKeeper defaults to stopping all operations).

2. ZooKeeper's Zab protocol ensures that all committed transactions are stored on at least 3 nodes before being acknowledged. The 2 surviving nodes may have different transaction logs if the 3 failed nodes had committed transactions that had not yet propagated to the survivors.

3. Recovery process:
   a. Identify which nodes can be recovered. If 1 of the 3 failed nodes can be restored, the cluster can form a new quorum.
   b. If all 3 failed nodes are unrecoverable, the 2 survivors must be restarted with a manual quorum override. This is done by setting `4lw.commands.whitelist=*` and using the `stat` command to verify data consistency, then using `setquota` to force a new quorum.
   c. Data integrity: The 2 survivors may have different last-committed-ZXIDs. The system must reconcile to the highest common ZXID. Transactions with ZXIDs above the common point on one survivor are lost.

4. Data loss: Any transaction committed on the 3 failed nodes but not yet propagated to the survivors is permanently lost. ZooKeeper's 3-node acknowledgment model ensures that a transaction committed on 3 nodes is guaranteed to survive if at least 2 of those 3 nodes survive. With 3 nodes completely lost, this guarantee cannot be met.

5. Prevention: Cross-AZ deployment with 5 nodes across 3 AZs (2-2-1 distribution). A single AZ failure affects at most 2 nodes, leaving 3 nodes with quorum. The 2-2-1 distribution ensures that no single AZ failure can cause quorum loss.

### Challenge 10: Fencing Token Collision

Two independent ZooKeeper clusters each generate fencing token 5 for their respective leaders. A network reconfiguration merges the clusters. Both leaders claim authority with token 5. How does the system detect and resolve this?

**Model answer:**

Fencing tokens must be globally unique across the merged cluster. Token 5 from Cluster A and token 5 from Cluster B are indistinguishable. The merged cluster cannot determine which leader has the legitimate claim.

Resolution requires additional metadata:
- Append a cluster identifier to each token: "cluster-A/5" vs "cluster-B/5"
- Use a higher-scope token authority that spans both clusters
- Generate tokens from a shared source (etcd, ZooKeeper) that both clusters recognize

After merge, the system must:
1. Detect that two leaders exist (separate cluster IDs in their tokens)
2. Compare additional metadata: term number, commit index, timestamps
3. Select one leader as authoritative based on the highest commit index
4. Force the other leader to step down
5. Reissue tokens from the unified token authority

This scenario is common when two independently managed clusters are merged after an acquisition or during migration. The solution is to plan token namespace unification before the merge.

### Challenge 11: Deadlock from Mutual Fencing

Two nodes fenced each other simultaneously. Each node received a fence command from the other before either could execute. Both nodes are now fencing targets. Neither can serve writes. Recovery?

**Model answer:**

This is a fencing deadlock. Each node is the intended target of a fence command issued by the other node. Both nodes refuse to serve writes because they are in the process of being fenced.

Three recovery paths:

Path A (automated): Each node implements a fencing timeout. If a fence command is received but not executed within T seconds, the node cancels the fence and retries. The node with the higher fencing token wins the retry. The lower-token node accepts the fence.

Path B (operator): An operator manually determines which node should be the leader based on data freshness. The operator cancels the fence on that node and executes the fence on the other node.

Path C (self-sacrifice): Both nodes recognize the deadlock after a timeout and both power off. The cluster detects all nodes are down. The cluster operator manually starts the nodes one at a time, using a configuration override to designate the initial leader.

Prevention: Never fence simultaneously. The fencing mechanism should have a priority-based ordering. The node with the higher fencing token should fence first. The lower-token node should wait 2x the fencing timeout before attempting its own fence.

### Challenge 12: Split Brain in a Microservices Architecture

A microservices architecture uses a shared PostgreSQL cluster. Service A and Service B both write to the same table. During a network partition, Service A connects to Node X (new leader) and Service B connects to Node Y (old leader, accepting writes because fencing failed silently). Both services write to the same rows. Design a protection strategy.

 content={`
                    ┌────────────────────────┐
                    │    Network Partition    │
                    └────────────────────────┘
                              │
   ┌──────────────────────────┼──────────────────────────┐
   │                          │                          │
   ▼                          │                          ▼
   Service A                  │                      Service B
   │                          │                          │
   │                          ▼                          │
   ▼                    ┌──────────┐                     ▼
   ┌──────────┐         │  Node X  │        ┌──────────┐
   │ Node Y   │         │  (New    │        │  Node Y  │
   │ (Stale)  │         │  Leader) │        │  (Stale) │
   │ x=5      │         │  x=10    │        │  x=5     │
   └──────────┘         └──────────┘        └──────────┘
        │                     │                   │
        └─────────────────────┼───────────────────┘
                              │
   Service A writes x=5 to Y  │  Service B writes x=10 to X
   Both services think they   │
   are writing to the leader  │
`} />

**Model answer:**

The core problem is that the services are not partition-aware. They connect to the database without verifying that the target node is the current leader. Three protection layers are needed:

Layer 1 — Connection verification: Before accepting a write, each service must verify the target node is the leader by querying a consensus service (etcd, ZooKeeper) that explicitly tracks leadership. This adds ~5ms per write but guarantees correctness.

Layer 2 — Application-level fencing: Each write must carry a fencing token obtained from the consensus service. The database rejects writes with stale tokens. This prevents the "old leader accepting writes" scenario even if the database's internal fencing fails.

Layer 3 — Service-level retry with idempotency: Both services must implement idempotent writes with unique request IDs. If a write is rejected (stale token), the service retries with a fresh token from the current leader. The database uses the request ID to deduplicate.

Implementation:
```
Service pseudocode:
1. Get current token from etcd: token = etcd.get("/leader/token")
2. Get leader connection string: conn = etcd.get("/leader/connection")
3. Connect to conn
4. Send write: INSERT INTO table VALUES (...) WITH token={token}
5. If rejected (stale token):
   a. Rollback transaction
   b. Sleep 100ms
   c. Go to step 1
6. If accepted: commit and return success

This ensures both services always write to the current leader,
even during and after a partition.
```
