## Case Study 1: GitLab NFS Split Brain (2017)

On January 31, 2017, GitLab experienced one of the most well-documented production incidents in the industry. An NFS-based storage system serving Git repositories suffered a split brain condition that led to 6 hours of downtime and data loss for 3,247 code review discussions.

### Key Lessons

**NFS is not a distributed filesystem.** It is a network-attached storage protocol. It provides no built-in fencing or lease mechanism. Multiple clients can mount the same NFS share as read-write, and the filesystem has no mechanism to prevent concurrent conflicting writes.

**Single shared storage does not prevent split brain.** GitLab used NFS as a single shared storage to avoid database-level split brain. The assumption was: shared storage means no split brain. The reality was: NFS allowed writes from multiple nodes, and the lack of coordination between writers caused the incident.

**The restoration script was the final blow.** The team followed a runbook that deleted the NFS-backed database directory as part of the restoration process. The directory contained data, but the deletion proceeded anyway. The runbook had never been tested against the production data volume.

**Recovery took 6 hours.** The team had to identify which data was lost, restore from a secondary backup, and manually reconstruct 189 records that could not be recovered from any backup. The incident led to a comprehensive redesign of GitLab's storage architecture, moving away from NFS to object storage with proper locking.

## Case Study 2: MongoDB Replica Set Split Brain

MongoDB replica sets can experience split brain when network partitions combine with misconfigured election timeouts. While MongoDB's Raft-based protocol provides strong safety guarantees, three specific scenarios have caused production incidents.

<AsciiDiagram
  id="split-brain/realworld-mongodb"
  title="MongoDB Replica Set Split Brain Scenarios"
  alt="Three MongoDB split brain scenarios: asymmetric partition, priority misconfiguration, and delayed election timeout, each shown as a 3-node replica set"
  caption="Three documented MongoDB split brain scenarios and their causes"
 content={`
   Scenario 1: Asymmetric Partition
   ─────────────────────────────────
   ┌──────────────┐     ┌──────────────────┐
   │  Primary A   │◄───►│  Secondary B     │
   │              │     │  Can reach A     │
   │  Can't reach │     │  Can't reach C   │
   │  B or C      │     └────────┬─────────┘
   └──────────────┘              │
                                 │
                          ┌──────┴──────┐
                          │ Secondary C │
                          │  Can reach B│
                          │  Can't reach│
                          │  A          │
                          └─────────────┘
   B sees A alive so B doesn't trigger election.
   A cannot reach B or C so A thinks it's alone.
   A continues as primary, accepting writes.
   B never takes over. Single node accepts writes.
   Data stops replicating. Silent divergence.
   Scenario 2: Priority Misconfiguration
   ──────────────────────────────────────
   3-node set: A(priority=5), B(priority=5), C(priority=1)
   If A is partitioned from B, both A and B
   attempt to become primary. C votes for
   whichever it can reach. Split brain avoided
   if C can reach only one. Data loss if C
   votes differently than expected.
   Scenario 3: Delayed Election Timeout
   ─────────────────────────────────────
   electionTimeoutMillis = 20,000ms (too high)
   Primary A fails. 20 seconds pass before
   election triggers. During those 20 seconds,
   clients send writes to A. A accepts them
   but cannot replicate. Data is lost when
   A rejoins as secondary.
`} />

### MongoDB Safety Recommendations

```
1. electionTimeoutMillis: 10,000ms minimum
   Lower values cause more elections.
   Higher values increase divergence window.

2. Always use odd number of voting members.
   3 member sets are minimum for production.
   Add an arbiter for even-numbered sets.

3. Priority values must be unique in the set.
   Duplicate priorities cause election storms.
   Set priority=0 for members that should never be primary.

4. Set writeConcern: majority for critical writes.
   This ensures writes are acknowledged by a quorum.
   Writes with w:1 can be lost in failover.

5. Test failover behavior with network chaos.
   Use mongodb's failIndex and test harness.
   Verify that every node pair handles partition correctly.
```

## Case Study 3: Cassandra and Virtual Nodes

Cassandra uses a peer-to-peer architecture that eliminates leader-based split brain. However, it introduces a split-brain-like problem with virtual nodes (vnodes) and hinted handoff.

   │  Node A (DC1) and Node D (DC2)                  │
   │  Both accept writes independently               │
   │  Timestamp-based resolution determines winner    │
   └──────────────────────────────────────────────────┘
   Split brain-like scenario:
   1. DC1 and DC2 partitioned
   2. Node A accepts write to key "user:42" at T=1000
   3. Node D accepts write to same key at T=1001
   4. Partition heals
   5. Hinted handoff delivers both writes
   6. T=1001 wins (last-writer-wins)
   7. Node A's write at T=1000 is overwritten
   8. Clock skew may cause wrong version to win
`} />

### Cassandra Safety Recommendations

```
1. Use consistent clocks (NTP with tight bounds).
   Cassandra relies on timestamps for conflict resolution.
   200ms clock skew can cause incorrect last-writer-wins.

2. Set read_repair_chance to 1.0 for critical data.
   This ensures every read triggers a consistency check.
   Repairs data before it is returned to the client.

3. Use LOCAL_QUORUM for local writes and reads.
   Avoid QUORUM (cross-datacenter) for latency-sensitive ops.
   LOCAL_QUORUM limits voting to the local datacenter.

4. Monitor tombstone_compaction to prevent data resurrection.
   Tombstones that are not compacted can cause deleted data
   to reappear after a network partition heals.

5. Configure hinted_handoff_window to 3 hours maximum.
   Longer windows increase the chance of data conflicts.
   Shorter windows may lose hints during extended partitions.
```

## Case Study 4: Elasticsearch Split Brain

Elasticsearch has historically been vulnerable to split brain scenarios due to its quorum configuration. The minimum_master_nodes setting is critical for prevention.

  alt="Elasticsearch cluster showing the minimum_master_nodes quorum setting and the difference between correct and incorrect configurations during a partition"
  caption="Elasticsearch split brain is prevented by setting minimum_master_nodes to floor(N/2)+1"
 content={`
   Elasticsearch Cluster (5 nodes)
   ─────────────────────────────────
   Correct Configuration:
   minimum_master_nodes: 3
   ┌──────────────┐           ┌──────────────────┐
   │  Group A     │           │  Group B          │
   │  2 nodes     │           │  3 nodes          │
   │  No quorum   │           │  Has quorum (3)   │
   │  Rejects ops │           │  Elects master    │
   └──────────────┘           └──────────────────┘
   No split brain
   Incorrect Configuration:
   minimum_master_nodes: 2
   ┌──────────────┐           ┌──────────────────┐
   │  Group A     │           │  Group B          │
   │  2 nodes     │           │  3 nodes          │
   │  Has quorum! │           │  Has quorum!      │
   │  Elects      │           │  Elects           │
   │  master      │           │  master           │
   └──────────────┘           └──────────────────┘
   SPLIT BRAIN — Two masters!
`} />

### Elasticsearch Recovery

```
Elasticsearch recovery from split brain:
1. Identify which cluster has the most recent data
2. Shut down all nodes in the other cluster
3. Restart the shut-down nodes one at a time
4. Each restarting node joins the surviving cluster
5. Translog replay catches up with committed writes
6. Uncommitted writes on shut-down nodes are lost

Recovery time: 5-30 minutes for a 5-node cluster
Data loss: Any write that was committed on the minority
           side during split brain is permanently lost.
```

## Open Source Tools

### Pacemaker / Corosync

Pacemaker is the standard cluster resource manager for Linux high-availability deployments. It uses Corosync for cluster membership and messaging.

```
Pacemaker split brain prevention:
- Corosync provides cluster membership and quorum
- Pacemaker manages fencing (STONITH) configuration
- Fencing topology: multiple fencing methods in sequence
- Resources are started only on quorate nodes

STONITH configuration example:
  pcs stonith create ipmi-node1 fence_ipmilan \
    pcmk_host_list=node1 \
    ipaddr=192.168.1.101 \
    login=admin \
    passwd=secret \
    op monitor interval=60s

Quorum policy options:
  stop:        Stop all resources if quorum lost (default)
  freeze:      Keep running but don't start new resources
  suicide:     Fence self if quorum lost
  ignore:      Continue without quorum (dangerous)

Recommended policy: stop (safety first)
```

### ZooKeeper and etcd

ZooKeeper and etcd are consensus services that provide the foundation for split brain prevention in many distributed systems.

```
ZooKeeper:
- Uses Zab (ZooKeeper Atomic Broadcast) protocol
- Majority quorum required for leader election
- 3 servers minimum for production
- 5 servers for fault tolerance
- Fencing tokens via zxid (ZooKeeper transaction ID)
- zxid monotonically increases with each operation

etcd:
- Uses Raft consensus protocol
- Majority quorum for leader election
- Lease-based leadership with configurable TTL
- Cluster reconfiguration requires majority consent
- Fencing tokens via raft term number

Both provide:
- Reliable leader election
- Fencing token generation
- Configuration storage
- Cluster membership tracking
```

## Cloud Provider Solutions

### AWS Multi-AZ

AWS provides Multi-AZ deployments for RDS, ElastiCache, and other managed services. These deployments span multiple Availability Zones (AZs) within a region.

```
AWS RDS Multi-AZ:
- Synchronous replication to standby in different AZ
- Automatic failover with DNS update (30-60 seconds)
- No fencing required — AWS handles infrastructure isolation
- Single writer, multiple readers (no leader election)

AWS ElastiCache for Redis (Cluster Mode):
- Automatic failover with replica promotion
- Quorum-based detection (consensus via majority nodes)
- Multi-AZ auto-failover enabled by default
- Node replacement automated by AWS

AWS benefits:
- AZs are physically isolated, reducing overlap risk
- AWS control plane handles fencing
- Separate management network is standard
- Automated recovery reduces operator error
```

### Azure Availability Zones

Azure provides Availability Zones that correspond to physically separate datacenters within a region.

```
Azure SQL Database:
- Built-in high availability with no split brain exposure
- Microsoft manages failover internally
- No customer configuration required
- 99.995% availability SLA

Azure Virtual Machines with Availability Zones:
- Customer-managed clustering
- Azure Load Balancer handles health probes
- Availability Sets protect against rack-level failures
- Managed disks support shared storage fencing

Azure Kubernetes Service (AKS):
- etcd managed by Azure control plane
- Node auto-repair detects and replaces unhealthy nodes
- Pod Disruption Budgets prevent simultaneous node removal
- Zone-redundant storage handles data plane availability
```

## Lessons Learned

### Common Patterns Across Incidents

1. **Configuration is the most common root cause.** Every major split brain incident involves a configuration error: wrong quorum size, incorrect timeout, missing STONITH configuration, or inconsistent settings across nodes.

2. **Testing is essential.** Organizations that test split brain scenarios regularly recover faster and with less data loss. Organizations that consider split brain "too rare to test" are the most affected when it happens.

3. **Monitoring must detect split brain specifically.** Generic "node down" alerts are not sufficient. Dedicated split brain detection with fencing token tracking and quorum health monitoring is necessary.

4. **Manual recovery is slow and error-prone.** Split brain recovery should be automated to the maximum extent possible. Manual steps introduce delay and risk of incorrect action.

5. **Every layer needs protection.** Application-level idempotency, database-level quorum, infrastructure-level fencing. Relying on a single protection layer is insufficient.
