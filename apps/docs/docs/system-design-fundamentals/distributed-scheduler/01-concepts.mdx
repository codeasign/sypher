## Problem Story

Imagine you run a global food delivery platform. Every second, customers place thousands of orders. Each order triggers a chain of timed tasks: send a confirmation email immediately, alert the restaurant after 30 seconds, dispatch a driver in 3 minutes, ping the customer with an ETA update in 10 minutes, and if the order is late by 15 minutes, escalate to a supervisor.

Now add the complexity of scale. Orders fail mid-chain and need retries. A payment timeout must release inventory after exactly 2 minutes. Peak hours see 50,000 concurrent pending tasks. Your single-server cron job cannot keep up. It misses deadlines, double-processes orders after crashes, and dies entirely under traffic spikes.

This is the problem a distributed scheduler solves: reliably executing time-based work across many machines while maintaining correctness, durability, and low latency.

The core challenge is not scheduling. It is coordination. You have multiple workers, each pulling work, each possibly failing. You need exactly-once or at-most-once semantics. You need tasks to survive machine failures. You need millisecond-precision for some jobs and hour-level tolerance for others. You need to backfill missed runs when a node goes down. You need a system that scales horizontally without a single point of failure.

Consider a concrete edge case. A customer places an order at 19:45:23. The dispatch driver task is due in 3 minutes at 19:48:23. At 19:48:22, the scheduler node holding that task crashes. The task is lost from memory. Without persistence, the driver never gets dispatched and the order is forgotten. A distributed scheduler writes the task to durable storage before acknowledging receipt, so the task survives the crash and another node picks it up when it comes back online.

Here is another edge case. Two tasks have the same trigger time of 19:50:00.000. Task A releases inventory for a popular item. Task B charges a late fee. If they both execute, the inventory is released before the fee is charged. If they execute in reverse order, the late fee is charged on already-released inventory, creating an accounting discrepancy. Your scheduler must guarantee consistent ordering within the same time bucket, not just trigger time equality.

Consider the ghost-task problem. A worker starts executing a send-email task. Before it can mark the task as COMPLETED, the worker loses network connectivity. The lease expires. Another worker picks up the same task and sends the email again. Your customer receives two confirmation emails. The scheduler cannot prevent this. Your downstream system must handle duplicates with idempotency checks.

Think about clock skew. Your scheduler runs across 50 servers in 3 data centers. Server A's clock is 200ms ahead of Server B's clock. A task scheduled by Server A at 14:00:00 might appear to Server B as scheduled at 13:59:48. Server B might fire it early. Distributed schedulers handle this by using a monotonic reference clock and applying a clock skew tolerance window during lease acquisition.

Traditional tools like cron, Windows Task Scheduler, or Quartz on a single node work well for small deployments. They break at the FAANG scale where you have millions of delayed tasks, each with its own trigger time, retry policy, and state.

## The Insight

A distributed scheduler is a time-indexed priority queue replicated across nodes, combined with a lease-based execution model.

Think of it as a highly reliable alarm clock with many hands. You set an alarm for each task. The clock ticks down. When an alarm fires, exactly one worker picks it up, locks it with a lease, executes it, and reports the result. If the worker dies before reporting, the lease expires and another worker retries the task.

The fundamental insight is separating the clock from the hands. The time-indexed data structure (the clock) lives in a reliable backing store like a database or consensus system. The workers (the hands) are stateless and interchangeable. This separation lets you scale hands independently, replace failed hands, and even replace the clock without losing scheduled work.

To push the metaphor further, consider a traditional wall clock with hour, minute, and second hands. If one hand breaks, you cannot tell the time accurately until it is fixed. The clock and the hands are coupled. Now think of a digital clock display. The display is separate from the timekeeping mechanism. You can replace the display without stopping the clock, and the clock continues ticking regardless of whether any display is connected.

A distributed scheduler follows the digital clock model. The schedule index is the timekeeping mechanism. It ticks forward independently, tracking what work is due and when. Workers are the displays. They can come and go. When a new worker starts, it reads the current state of the schedule and begins processing. When a worker shuts down, the schedule continues advancing. No single worker failure affects the system's ability to track time.

This separation also enables independent scaling. If you need more throughput, you add more workers. You do not resize the schedule index. If you need finer time precision, you optimize the index structure. You do not reconfigure workers. The two concerns are decoupled, letting each evolve independently.

The clock itself is a data structure that supports four operations: `enqueue(task, triggerTime)`, `dequeue()`, `ack(id)`, and `nack(id)`. `enqueue` adds a task at a specific time. `dequeue` retrieves the next due task. `ack` confirms completion. `nack` signals failure and triggers retry. Everything else — lease management, retry policies, backfill — is built on top of these four primitives.

## Core Vocabulary

**Task** — A unit of work to execute at a specific time. Contains an id, trigger time, payload, and metadata like retry count and timeout.

**Schedule** — A clock or data structure that maintains the ordered list of pending tasks by their trigger time.

**Worker** — A process that picks up due tasks from the schedule and executes them. Workers are stateless and interchangeable.

**Lease** — A time-limited lock on a task. The worker holds the lease while executing. If the lease expires before completion, the task becomes available for another worker.

**Tick** — A heartbeat or polling cycle. Workers tick the schedule to discover tasks that are due.

**Deadline** — The absolute or relative time by which a task must start executing. Distinct from the execution duration or timeout.

**Delay Queue** — A data structure that holds tasks until their trigger time arrives. Implementations include sorted sets, priority queues, or time-wheel buckets.

**Retry Policy** — Rules governing how many times a failed task is retried, with what backoff interval and under what conditions (transient vs. permanent failures).

**Backfill** — The process of scanning for tasks that were missed due to a worker outage or clock drift, and re-queuing them.

**Shard** — A horizontal partition of the schedule space. Each shard owns a range of task IDs or time buckets, allowing parallel scheduling across nodes.

**Watermark** — A timestamp tracking how far the scheduler has progressed through time. Workers use watermarks to detect gaps or stalls.

**Lease Duration** — The maximum time a worker can hold a lease before it must either complete the task or extend the lease. Short durations mean faster failure recovery but more false-positive reclaims.

**Grace Period** — A configurable window added to the poll query so that tasks due slightly in the past (within the grace window) are still picked up. This prevents tasks from being missed due to clock skew or minor delays.

**Dead Letter Queue** — A holding area for tasks that have exhausted all retry attempts. Tasks in the dead letter queue require manual inspection or an alert to operators.

**Idempotency Key** — A unique identifier embedded in the task payload that the downstream system uses to detect and reject duplicate executions. The scheduler generates the key at enqueue time and passes it through unchanged.

**Cohort** — A logical grouping of tasks that share the same trigger time or time bucket. Cohorts enable batch processing and atomic visibility within a time window.

**Heartbeat** — A periodic signal from a worker indicating it is still alive and processing. Workers that stop heartbeating are presumed dead and their leases are reclaimed.

The flow begins when a client service enqueues a task with a delay or trigger time. The scheduler API writes the task to the delay queue, indexed by trigger time. Workers poll the delay queue in short intervals, claiming due tasks via lease mechanism. The claimed task moves to the state store with status RUNNING. After execution, the result updates the state store to COMPLETED or FAILED. Failed tasks return to the delay queue with an adjusted trigger time based on the retry policy.

The schedule index and delay queue are the backbone of the clock. The worker pool is the set of hands. A monitoring layer (not shown) tracks watermarks and triggers alerts if the system stalls.

## How It Works

### Step 1 — Enqueue

A client calls the scheduler API with a task payload and a trigger time. The scheduler assigns a globally unique ID, validates the payload, and writes the task to the backing store.

The enqueue operation must be durable before the API returns a success response. If the scheduler acknowledges the enqueue but crashes before persisting, the task is lost and the client has no way to recover it. Implementations use write-ahead logging or synchronous replication to ensure durability.

The enqueue API typically accepts these parameters:

- `payload` — The data needed to execute the task, serialized as JSON or a binary blob.
- `trigger_time` — An absolute timestamp or a relative delay in milliseconds.
- `retry_policy` — Optional. Defaults to 3 retries with exponential backoff.
- `timeout` — Optional. The maximum execution duration before the task is aborted. Defaults to 30 seconds.
- `idempotency_key` — Optional. A caller-supplied key that the scheduler uses to deduplicate enqueue calls. If the same key is submitted twice, the second call returns the existing task ID without creating a duplicate.

Here is a Python-like pseudocode example of the enqueue handler:

```python
def enqueue(payload, trigger_time, retry_policy=None, timeout=30):
    task_id = generate_uuid()
    task = {
        "id": task_id,
        "payload": payload,
        "trigger_time": normalize_time(trigger_time),
        "status": "PENDING",
        "retry_count": 0,
        "max_retries": retry_policy.max_retries if retry_policy else 3,
        "timeout": timeout,
        "created_at": current_timestamp(),
    }
    with durable_transaction() as tx:
        tx.write_task(task)
        tx.insert_schedule_index(task_id, task["trigger_time"])
    return task_id
```

### Step 2 — Index

The scheduler inserts the task ID and trigger time into a time-indexed data structure. In most implementations this is a sorted set or a B-tree keyed by trigger timestamp. This index is the core of the clock.

The index must support efficient range queries: find all tasks with trigger time between T and current-time-plus-grace. Without an index, every poll cycle would scan the entire task table, which does not scale beyond a few thousand tasks.

The index is also the primary contention point. Multiple workers read from the same index simultaneously. If the index is a B-tree, leaf-page contention becomes the bottleneck under high load. Some implementations shard the index by time range, assigning one B-tree per hour or per minute.

### Step 3 — Tick

Each worker maintains a periodic tick loop with a configurable interval, typically between 100ms and 5 seconds. At every tick, the worker queries the schedule index for tasks whose trigger time is in the past and whose status is PENDING. The query window typically spans from the beginning of time to current time plus a small grace period.

The tick interval sets an upper bound on scheduling latency. If you tick every 500ms, no task waits longer than 500ms beyond its trigger time to be discovered. The trade-off is load. Faster ticks mean more queries against the schedule index. A worker that ticks every 100ms generates 600 queries per minute. Across 100 workers, that is 60,000 queries per minute to the schedule index.

A common optimization is adaptive ticking. Workers tick fast when the queue is non-empty and slow down to a low-frequency heartbeat when the queue is empty. This reduces load during idle periods while maintaining responsiveness during busy periods.

```python
def worker_loop():
    last_queue_empty = False
    while True:
        due_tasks = query_schedule_index(
            max_time=current_time() + GRACE_PERIOD,
            status="PENDING",
            limit=BATCH_SIZE,
        )
        if not due_tasks:
            interval = IDLE_TICK_INTERVAL if last_queue_empty else BUSY_TICK_INTERVAL
            sleep(interval)
            last_queue_empty = True
            continue
        last_queue_empty = False
        for task in due_tasks:
            claim_and_execute(task)
```

### Step 4 — Lease

The worker attempts to lease the due tasks. Leasing atomically updates the task status from PENDING to LEASED, assigning the worker ID and setting a lease expiration timestamp. This MUST be an atomic compare-and-swap operation to prevent double-claiming.

In a SQL database, the lease operation looks like this:

```sql
UPDATE tasks
SET status = 'LEASED',
    worker_id = ?,
    lease_expires_at = NOW() + INTERVAL '30 seconds'
WHERE id = ?
  AND status = 'PENDING';
```

If the `WHERE` clause does not match (because another worker already leased the task), the update affects zero rows and the worker moves on. The worker checks the affected row count to determine if it won the lease.

The lease expiration timestamp is critical. It determines the recovery time objective. If you set it too short, slow workers lose their leases before they finish executing, causing redundant retries. If you set it too long, failed tasks stall for longer before another worker picks them up. A good starting point is 3x the expected task execution duration.

### Step 5 — Execute

With the lease held, the worker executes the task payload. Execution can mean calling an external API, running a local function, publishing a message, or any other side effect. The worker monitors execution against the task timeout. If execution exceeds the timeout, the worker aborts and marks the task as FAILED.

During execution, the worker should periodically extend the lease for long-running tasks. This prevents the lease from expiring while the task is still making progress. Lease extension is a lightweight update that resets the `lease_expires_at` timestamp without changing the task status.

```python
def execute_with_lease_extension(task, execute_fn):
    deadline = current_time() + task["timeout"]
    next_extension = current_time() + LEASE_EXTENSION_INTERVAL

    def extend_lease():
        if current_time() >= next_extension:
            extend_lease_in_db(task["id"], task["worker_id"])
            next_extension = current_time() + LEASE_EXTENSION_INTERVAL

    result = None
    while current_time() < deadline:
        try:
            extend_lease()
            result = execute_fn(task["payload"], timeout=remaining_time(deadline))
            break
        except TimeoutError:
            mark_failed(task["id"], "execution_timeout")
            return
        except TransientError:
            extend_lease()
            continue
    mark_completed(task["id"], result)
```

### Step 6 — Complete

On successful execution, the worker updates the task status to COMPLETED and releases the lease. The worker then removes the task from the pending schedule.

The completion step should be atomic. If the worker marks the task as COMPLETED but fails to remove it from the schedule index, a subsequent tick might re-discover the task and attempt execution. Use a single transaction that updates the status and removes the index entry.

```python
def mark_completed(task_id):
    with transaction() as tx:
        tx.update_task_status(task_id, "COMPLETED")
        tx.remove_from_schedule_index(task_id)
```

### Step 7 — Handle Failure

If execution fails, the worker checks the retry policy. If retries remain, the worker updates the task with a new trigger time computed from the backoff schedule and sets status back to PENDING. If no retries remain, the task moves to DEAD_LETTER with the failure reason recorded.

The backoff computation typically uses exponential backoff with jitter:

```python
def compute_retry_time(retry_count, base_delay=1):
    delay = base_delay * (2 ** retry_count)
    jitter = random.uniform(0, delay * 0.1)
    return current_time() + delay + jitter
```

With a base delay of 1 second, retry 0 fires at 1 second, retry 1 at 2 seconds, retry 2 at 4 seconds, retry 3 at 8 seconds, and retry 4 at 16 seconds. The jitter prevents thundering-herd when many tasks fail simultaneously.

The dead letter queue stores the full task record including the failure reason chain. Operators monitor the dead letter queue depth and investigate tasks that accumulate there. Automated alerts should fire when the dead letter queue exceeds a threshold, such as 100 tasks or a sustained growth rate.

### Step 8 — Lease Expiry

If a worker dies mid-execution, the lease expires after the configured duration. A background reclaim process scans for expired leases and resets those tasks to PENDING, making them available for other workers. This is the mechanism that provides fault tolerance.

The reclaim process runs on a separate cycle from the main tick loop. It queries for tasks where `status = 'LEASED' AND lease_expires_at < NOW()`. Each reclaimed task gets its status reset to PENDING, its retry count incremented, and its worker ID cleared.

```sql
UPDATE tasks
SET status = 'PENDING',
    worker_id = NULL,
    retry_count = retry_count + 1,
    lease_expires_at = NULL
WHERE status = 'LEASED'
  AND lease_expires_at < NOW()
RETURNING id;
```

The reclaim query returns the IDs of reclaimed tasks so the scheduler can log the recovery event. Monitoring the reclaim rate is important. A sudden spike in reclaims indicates workers are crashing or becoming unresponsive.

### Step 9 — Watermark Advancement

Workers periodically update a global watermark — the latest timestamp for which all tasks at or before that point have been processed. If a worker detects the watermark is not advancing, it triggers a backfill scan to find and re-queue missed tasks.

The watermark is stored in a highly available location, such as a distributed consensus store or a database table with a single row. Workers update the watermark with an atomic compare-and-swap: they read the current watermark, compute the new value, and write it back only if it is greater than the stored value.

A stalled watermark can indicate several problems. A shard might be down, leaving a time range unprocessed. A task might be stuck in the RUNNING state with an expired lease that the reclaim process missed. Or clock skew might have caused a task to be scheduled in a time bucket that falls between two workers' responsibility ranges.

```python
def advance_watermark(current_watermark):
    safe_time = current_time() - WATERMARK_SAFETY_MARGIN
    if safe_time > current_watermark:
        compare_and_swap_watermark(current_watermark, safe_time)
```

The safety margin ensures that tasks which were just enqueued or are still being processed are not missed. A typical safety margin is 2x the poll interval.

### Step 10 — Backfill

When the watermark stops advancing, the backfill process scans the time range between the stalled watermark and the current time for any PENDING or LEASED tasks that should have been processed. It re-queues these tasks by updating their status and logging the backfill event.

Backfill is a safety net, not a primary processing path. It should trigger an alert so operators know the system experienced a stall. Frequent backfills indicate a systemic problem like insufficient workers, misconfigured lease durations, or a bug in the lease acquisition logic.

### Step 11 — Rebalance

When a worker joins or leaves the pool, the remaining workers rebalance the shard assignments. This involves reassigning time buckets or task ID ranges so no shard is overloaded and no time range goes unprocessed.

Rebalancing uses a consistent hashing ring or a coordinator-based assignment protocol. When the pool size changes, each worker computes its new set of assigned shards. Workers transfer ownership of shards they no longer own to the newly responsible workers.

The transfer protocol has three phases:

1. **Drain** — The old worker stops polling for tasks in the shard being transferred. It completes any in-flight tasks from that shard.
2. **Handoff** — The old worker marks the shard as available. The new worker begins polling.
3. **Verify** — The new worker runs a backfill scan on the acquired shard to catch any tasks that were enqueued or missed during the handoff window.

### Step 12 — Monitoring and Alerting

A production distributed scheduler requires continuous monitoring. Key metrics include queue depth, drain rate, lease acquisition latency, reclaim rate, watermark age, and dead letter queue depth.

Each metric has a corresponding alert threshold. Queue depth exceeding 100,000 tasks triggers a scale-out alert. Reclaim rate above 1% of total task throughput triggers an investigation into worker health. Dead letter queue growth above 10 tasks per minute triggers a notification to the on-call engineer.

### Step 13 — Task Cancellation

Clients can cancel a previously enqueued task by providing its task ID. Cancellation is straightforward if the task is still PENDING: the scheduler deletes the task from the schedule index and marks it as CANCELLED. If the task is LEASED or RUNNING, the scheduler sends a cancellation signal to the worker and marks the task for cancellation once execution completes.

Graceful cancellation is important for resource cleanup. If a cancellation arrives while the task is executing an API call, the worker should wait for the API call to complete, then discard the result rather than forcibly interrupting the in-flight request.

### Step 14 — Scheduled Maintenance Windows

Large-scale schedulers need maintenance windows for upgrades and migrations. During a maintenance window, the scheduler drains all active leases, prevents new enqueues (or buffers them), and waits for all in-flight tasks to complete or time out. Once the drain completes, the system is safe to shut down for upgrades.

After the upgrade, workers restart and resume polling from the schedule index. The watermark ensures no tasks are missed. If the maintenance window exceeds the longest task timeout, all in-progress tasks eventually time out and enter the retry or dead letter flow.

## Variants

### Variant 1 — Time-Wheel Scheduler

The time-wheel is a circular buffer where each slot represents a fixed time interval. Tasks are placed into the slot corresponding to their trigger time modulo the wheel size. A pointer advances through the wheel at the tick rate, firing all tasks in the current slot.

The time-wheel trades scheduling precision for memory efficiency. It is ideal for high-throughput scenarios with short delays (milliseconds to seconds) where tasks are numerous but deadline precision is moderate.

The time-wheel excels in scenarios like connection timeout management, session expiration, and retry backoff tracking. In these cases, you have thousands of concurrent timers, each with a delay of a few seconds at most. A priority queue storing all these timers would require O(n) memory and O(log n) insertion. A time-wheel stores the same timers in O(wheel_size) slots with O(1) insertion.

The pointer rotates through slots 0 through 7, then wraps around. At each tick (every 100ms), all tasks in the pointed slot fire. A task scheduled 350ms from now goes in slot 3. If the pointer is at slot 3 when it wraps, tasks scheduled beyond one full rotation use a secondary overflow wheel with coarser granularity.

A common multi-tier design uses three wheels: a 100ms wheel with 10 slots (covering 1 second), a 1-second wheel with 60 slots (covering 1 minute), and a 1-minute wheel with 60 slots (covering 1 hour). Tasks cascade from the fine wheel to the coarse wheel when they exceed the current wheel's range. This design handles timers from 100ms to 1 hour using only 130 total slots.

Hashed-Wheel-Timer, popularized by Apache Kafka and Netty, uses this approach. Kafka uses it for delayed operation queues with configurable tick duration and wheel size. A 20-slot wheel at 1ms tick gives 20ms of fine-grained timing, while tasks beyond 20ms cascade into a second wheel tier.

### Variant 2 — Priority Queue Scheduler

A priority queue (min-heap) stores tasks ordered by trigger time. Workers pop the minimum-timestamp task, check if it is due, execute if yes, or sleep until it is due. This gives optimal O(1) peek and O(log n) insertion.

Priority queues excel when the schedule has a mix of near-term and far-term tasks, and you need precise ordering. The downside is contention on the heap root under high concurrency.

In a distributed setting, each shard maintains its own priority queue. The challenge is distributing tasks across shards such that no shard is overloaded and time-ordering is preserved within each shard. A common strategy is hash-based sharding on the task ID, which distributes load evenly but scatters tasks from the same time range across shards. An alternative is time-based sharding, which keeps time-ordering within a shard but creates hot spots during peak hours.

Sidekiq and Celery use priority queues backed by Redis sorted sets. The sorted set key is the trigger timestamp. Workers use ZRANGEBYSCORE to find due tasks and ZREM to atomically claim them. This approach handles millions of tasks but requires Redis persistence for durability. In production, you configure Redis with AOF (append-only file) persistence and replication to avoid data loss on restart.

### Variant 3 — Database-Polling Scheduler

A database-backed scheduler stores tasks in a table with a trigger_time column, an indexed status column, and a worker_id column. Workers run SQL queries to find and claim due tasks:

```sql
BEGIN;
SELECT * FROM tasks
WHERE status = 'PENDING'
  AND trigger_time <= NOW()
ORDER BY trigger_time
LIMIT 100
FOR UPDATE SKIP LOCKED;
UPDATE tasks SET status = 'LEASED', worker_id = ?
WHERE id IN (...);
COMMIT;
```

This variant leverages the database for consistency and durability. It is the simplest to implement and the most reliable. It is also the most resource-intensive because every poll cycle hits the database with a range scan.

The composite index `(status, trigger_time)` is essential for this query. Without it, the database performs a sequential scan of the tasks table on every poll. With the index, the database seeks directly to the first row where status is PENDING and trigger_time is in the past.

The SKIP LOCKED clause is critical. Without it, workers contend on the same rows, causing lock waits and reduced throughput. SKIP LOCKED tells the database to skip rows locked by other transactions and return only readily available rows.

Database-polling schedulers work well in the range of 10,000 to 100,000 tasks per second. Beyond that, the database becomes the bottleneck. At that point, you typically move to a time-wheel or priority queue variant with a separate persistence layer.

Popular implementations include Amazon RDS with pg_cron or the built-in `pg_tle` extension for scheduling, and Google Cloud Tasks which wraps a similar database-backed model behind a managed API.

## Variant Comparison

| Dimension | Time-Wheel | Priority Queue | Database Polling |
|---|---|---|---|
| Precision | Coarse (tick-granular) | Exact | Exact |
| Throughput | Very high (1M+ tasks/s) | High (100K tasks/s) | Moderate (10K-100K tasks/s) |
| Memory | O(wheel_size) | O(task_count) | N/A (disk-based) |
| Durability | Requires external store | Requires external store | Built-in (ACID) |
| Implementation | Complex (multi-tier) | Moderate | Simple |
| Consistency | Atomic per slot | Heap-root contention | SQL ACID transactions |
| Scaling | Fixed wheel size | Shard by ID range | Shard by table |
| Latency | Tick-bound (e.g., 100ms) | O(log n) insert, O(1) peek | Query + network round-trip |
| Recovery speed | Fast (state in memory) | Moderate (rebuild heap) | Fast (database handles it) |
| Failure isolation | Slot-level (partial) | Shard-level | Row-level via SKIP LOCKED |
| Operational cost | Low (in-memory) | Moderate (Redis cluster) | Higher (database resources) |

## Mental Model Check

These questions help you verify your understanding of the distributed scheduler pattern. Each question targets a specific concept from this chapter.

### Question 1

You have a scheduler where workers poll every 500ms. A task becomes due at 14:00:00.000. What is the earliest and latest time it could start executing?

**Answer:** The earliest is 14:00:00.000 if the poll cycle happens to align perfectly. The latest is 14:00:00.500 because the next poll will find it. In practice, add network latency and database query time, so the actual range is 14:00:00.000 to approximately 14:00:00.520. Larger poll intervals increase this window but reduce database load.

### Question 2

Two workers both read the same PENDING task in the same poll cycle. What prevents them from both executing it?

**Answer:** The lease acquisition must be atomic. In a database variant, `SELECT ... FOR UPDATE SKIP LOCKED` ensures only one worker gets the row lock. The second worker's SELECT skips the locked row. In Redis, the atomic `ZREM` (or Lua script) ensures only one worker removes and claims the task. Without an atomic claim, both workers execute the task, causing duplicate side effects.

### Question 3

A worker holds a lease on a task and then crashes. The lease duration is 30 seconds. What happens to the task?

**Answer:** After 30 seconds, the lease expires. A background reclaim process (or the next poll cycle) detects the expired lease by checking if the lease timestamp plus duration is before the current time. It resets the task status to PENDING and clears the worker ID. Another worker will pick it up on the next poll. The task experiences a delay equal to the lease duration plus the poll interval. This is the trade-off: shorter leases mean faster recovery but more false-positive reclaims if workers are merely slow.

### Question 4

You need exactly-once execution semantics for a payment task. Can a distributed scheduler guarantee this?

**Answer:** No distributed scheduler can guarantee exactly-once execution in the presence of network failures and crashes. This is the Two Generals Problem. The best you can achieve is at-most-once or at-least-once. For payments, you design tasks to be idempotent: if the same payment request executes twice, the payment gateway deduplicates by an idempotency key. The scheduler provides at-least-once, and the downstream system provides idempotent deduplication to achieve effectively-once semantics.

### Question 5

Your scheduler runs 10,000 tasks per second, each completing in 50ms on average. How many workers do you need, and what happens during a burst of 50,000 tasks?

**Answer:** Each worker handles approximately 20 tasks per second (50ms execution plus overhead). For 10,000 tasks per second, you need at least 500 workers under ideal conditions. Add a safety margin of 2x for bursts and failures: 1,000 workers. During a 50,000-task burst, tasks that cannot be claimed immediately remain PENDING in the queue. The queue depth grows. Workers process them as fast as they can. The critical metric is queue drain time: 50,000 tasks at 10,000/second drains in 5 seconds plus the backlog processing. You need to ensure your backing store can handle the queue depth without memory pressure or performance degradation.

### Question 6

Your time-wheel scheduler has 10 slots at 100ms granularity, covering 1 second. You schedule a task with a delay of 3.5 seconds. Where does it go?

**Answer:** A single time-wheel cannot hold a 3.5-second task with only a 1-second range. You need a multi-tier design. The task goes into the overflow wheel at the second tier. In a two-tier design with a 1-second fine wheel (10 slots at 100ms) and a 10-second coarse wheel (10 slots at 1 second), the 3.5-second task goes into slot 3 of the coarse wheel. After 3 seconds, it cascades into the fine wheel at slot 5 (500ms remaining at that point). The cascade happens on the tick when the coarse wheel pointer passes slot 3. The task is then visible to the fine wheel and fires 500ms later.

### Question 7

Your database-polling scheduler has 50 workers polling a single table every second. You notice database CPU at 80% and query latency spikes during peak hours. What changes would you investigate?

**Answer:** Three areas to investigate. First, check the query plan for the poll query. A missing or fragmented composite index on `(status, trigger_time)` causes sequential scans that spike CPU. Second, consider increasing the `LIMIT` and reducing the poll frequency. Polling for 200 tasks every 2 seconds generates half the queries of polling for 100 tasks every 1 second, with the same average latency. Third, evaluate whether the `SKIP LOCKED` contention is high. If many tasks are PENDING simultaneously, each poll locks and skips rows, increasing CPU. In that case, shard the tasks table by a hash of the task ID into multiple physical tables, each polled by a subset of workers.

### Question 8

Your scheduler uses a 60-second lease duration. A worker processing a task crashes. A task that normally takes 5 seconds to complete is recovered by another worker after 62 seconds. The task has a retry policy of 3 max retries. How many times has the task been retried after the crash recovery?

**Answer:** The task has been retried once. The crash recovery via lease expiry increments the retry count. The task has 2 remaining retries. The retry count increments during the lease expiry reclaim, not during re-execution. If the task fails during execution on the second worker, the retry count increments again. After the crash, the retry count is 1, the task is PENDING, and it proceeds normally. The key insight is that a crash consumes a retry because the system cannot distinguish between a transient failure and a worker crash. If you want to distinguish them, use a heartbeat mechanism that detects crashes faster than the lease expiry and marks the task for immediate reclaim without incrementing retry count.

## Putting It All Together

A distributed scheduler is one of the foundational building blocks of large-scale systems. It appears everywhere: sending notifications, orchestrating microservice workflows, managing cache invalidation, coordinating batch jobs, and handling timeout-based state transitions.

When you design a scheduler for your system, start with these questions:

1. **What precision do you need?** If milliseconds matter, use a time-wheel. If seconds are fine, database polling is simpler.
2. **How many tasks per second?** Below 10,000 tasks per second, database polling works well. Above 100,000, switch to a time-wheel or priority queue with a separate durable store.
3. **What is your tolerance for duplicate execution?** If duplicates cause monetary loss, invest in idempotency keys and downstream deduplication. If duplicates are merely annoying, at-least-once semantics are acceptable.
4. **How fast must you recover from failures?** Lease durations in the range of seconds give fast recovery. Lease durations in the range of minutes reduce false-positive reclaims but delay recovery.
5. **What operational complexity can you support?** A database-backed scheduler requires only SQL. A Redis-backed priority queue requires a Redis cluster. A time-wheel scheduler requires custom implementation and tuning.

The pattern is mature. Do not build your own unless you have a specific requirement that none of the existing solutions meet. Use AWS Step Functions, Google Cloud Tasks, RabbitMQ delayed queues, Sidekiq, Celery, or Apache Airflow. These systems have solved the hard problems of clock skew, partition recovery, and lease management.

Here is a summary decision framework:

- If you need a simple, reliable scheduler with moderate throughput, use **database polling** with a managed database.
- If you need high throughput with sub-second precision, use a **time-wheel** scheduler backed by a durable store.
- If you need precise ordering and a mix of near-term and far-term tasks, use a **priority queue** backed by Redis or a similar in-memory store.
- If you need a fully managed solution, use a **cloud service** like AWS EventBridge Scheduler or Google Cloud Scheduler.

The clock and hands metaphor carries through all variants. The schedule index is the clock. The workers are the hands. Keep them separate, and your scheduler will scale.
