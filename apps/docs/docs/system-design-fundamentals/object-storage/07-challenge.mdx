# Object Storage --?" Challenges

## Tier 1: Foundation (5 minutes each)

### Challenge 1.1: Write Quorum Calculation

Your object storage system has a replication factor of 5. You want to ensure that concurrent reads always see the most recent write. What is the minimum write quorum (W) and read quorum (R) that guarantees strong consistency?

**Hint**: The condition is W + R greater than N, where N is the replication factor.

**Model Answer**:
For N = 5, any pair (W, R) where W + R > 5 guarantees that at least one replica in the read quorum overlaps with the write quorum.

Minimum practical values: W = 3, R = 3 (3 + 3 = 6 > 5).

This means a write must be acknowledged by 3 out of 5 replicas, and a read must consult 3 out of 5 replicas. Since any set of 3 replicas overlaps with any other set of 3 replicas by at least 1 replica (by the pigeonhole principle), the read always sees the latest write.

Trade-off: W = 3, R = 3 gives strong consistency but higher latency than W = 3, R = 2 (which would be eventually consistent). If you set W = 5, R = 1, you get strong consistency but writes are slow (must wait for all 5). If you set W = 1, R = 5, reads are slow (must query all 5) but writes are fast.

### Challenge 1.2: Durability Estimation

You store 100 million objects in S3 Standard, which advertises 99.999999999% (11 nines) durability per object. How many objects do you statistically expect to lose per year?

**Model Answer**:
Durability of 99.999999999% means the annual failure rate (AFR) is 1 - 0.99999999999 = 1 x 10^(-11) per object.

Expected annual objects lost = 100,000,000 x 1 x 10^(-11) = 0.001 objects per year.

Statistically, you expect to lose one object every 1,000 years.

But this assumes independent failures. In practice, correlated failures (a data center flood, a software bug that corrupts all replicas) dominate the actual risk. When AWS reported the 2017 S3 outage (human error took down us-east-1 for 4 hours), no data was lost --?" but if a similar error had caused data corruption instead of unavailability, the correlated-loss rate would be far higher than the per-object AFR suggests.

### Challenge 1.3: Prefix Throughput

Your application needs 20,000 PUT requests per second to a single S3 bucket. S3's documented limit is 3,500 PUT requests per second per prefix. How do you achieve the required throughput?

**Model Answer**:
The 3,500 limit applies to a single prefix (the portion of the key before the first / character). To achieve 20,000 PUTs/sec, you need to distribute writes across at least 6 prefixes (20,000 / 3,500 = 5.7, round up to 6).

You can use a hash of the object key as the first prefix component. For example, instead of `uploads/photos/{photo_id}.jpg`, use `uploads/{hash(photo_id) % 100:04x}/{photo_id}.jpg`. This creates 100 prefixes, each capped at 3,500 writes/sec, giving you a theoretical 350,000 writes/sec aggregate.

Alternatively, use S3 Transfer Acceleration to reduce latency for global clients, but that does not increase throughput per prefix --?" you still need prefix sharding.

**Common mistake**: Thinking you can use more buckets. The 3,500 writes/sec limit is per prefix, not per bucket. Using more buckets does not help unless you also change the prefixes.

## Tier 2: Design (15 minutes each)

### Challenge 2.1: Design a Video Thumbnail Generation Pipeline

You need to generate 3 thumbnail sizes (150px, 300px, 600px) for every video uploaded to your platform. Users upload 10,000 videos per day. Average video size is 200 MB. Videos are stored in S3. Design the pipeline.

**Requirements**:
- Thumbnails must be generated within 5 minutes of upload
- Cost should be minimized
- Failed thumbnail generations must be retried automatically

**Model Answer**:

**Storage layout**:
- `videos/{hash(video_id):04x}/{video_id}/original.mp4`
- `videos/{hash(video_id):04x}/{video_id}/thumbnails/{size}px.jpg`

**Pipeline**:
1. User uploads video via pre-signed URL directly to S3.
2. S3 event notification (s3:ObjectCreated) triggers an SQS queue (not directly Lambda, to allow retry and back-pressure).
3. A worker fleet (AWS Batch or ECS with spot instances) polls the queue. Each worker:
   a. Downloads the video from S3 to local NVMe (ephemeral storage).
   b. Uses ffmpeg to extract a frame at the 10-second mark.
   c. Resizes the frame to 3 sizes using ImageMagick.
   d. Uploads the 3 thumbnails to S3.
   e. Deletes the local files.
   f. Deletes the SQS message.
4. If the worker crashes or the job takes longer than the SQS visibility timeout (15 minutes), the message reappears in the queue for retry.
5. Dead-letter queue captures messages that failed 3+ times for manual inspection.

**Cost estimate**: 10,000 videos x 3 thumbnails = 30,000 S3 writes. Worker compute: ~3 minutes per video using spot instances = 500 instance-hours/day at ~$0.04/hour = $20/day. S3 storage for thumbnails: 30,000 x 3 sizes x 50 KB average = 4.5 GB/day = $0.10/day. Total: ~$20/day.

**Throughput**: With 10,000 videos/day and a 5-minute SLA, you need enough workers to process 10,000 / (5 x 60) = ~34 videos concurrently. At 3 minutes per video, a worker processes about 80 videos/day. You need 10,000 / 80 = ~125 workers. With spot instance interruptions, provision 140-150.

### Challenge 2.2: Design a Multi-Tier Archival System

Design an object storage system with three tiers: Hot (NVMe), Warm (HDD), Cold (tape). Objects automatically transition between tiers based on access frequency. Access patterns: recently accessed objects should be promoted to Hot; objects not accessed in 30 days move to Warm; objects not accessed in 365 days move to Cold.

**Model Answer**:

**Metadata tracking**: Each object has an `access_counter` and `last_access_time` in the metadata database. Reads increment the counter and update the timestamp.

**Access heat scoring**: A background process runs hourly and computes a heat score for each object:
- Score = (access_count_last_7_days / 7) / object_size_in_MB
- This biases toward small, frequently-accessed objects

**Transition rules**:
- Score >= 10 reads/MB/day: Hot tier (NVMe)
- Score between 1 and 10: Warm tier (HDD)
- Score under 1: Cold tier (tape)
- Force-demote: no reads in 365 days --?' Cold regardless of score

**Promotion on read**: When a Cold or Warm object is read, it is promoted to Hot tier asynchronously. The read is served from the current tier; a background job copies the object to the NVMe tier. Future reads hit the Hot tier.

**Implementation details**:
- Each tier is a separate storage pool with its own placement engine
- The metadata store tracks which pool holds each chunk
- Tier migration is a background copy: copy chunk to new pool, update metadata, delete old chunk
- Migration priority queue: hot objects get promoted first; cold demotion is low priority

**Cold tier specifics**: Tape storage has a retrieval latency of 30-300 seconds. To mitigate this, maintain a tape cache (recently-read tapes stay loaded for 24 hours). Use a robotic tape library with 10+ drives for concurrent reads.

### Challenge 2.3: Design an S3-Compatible API Gateway

You are building an S3-compatible API gateway that sits in front of your custom object storage (MinIO or Ceph). The gateway must support the S3 REST API including authentication (AWS Signature V4), multi-part uploads, and pre-signed URLs. Design the request flow.

**Model Answer**:

**Components**:
1. **Auth service**: Validates AWS Signature V4. Stores no data, only verifies HMAC.
2. **Request router**: Parses the request, extracts bucket/key, and routes to the correct storage backend.
3. **Translation layer**: Maps S3 API calls to the backend's native API.
4. **Response formatter**: Constructs S3-compatible XML responses.

**Request flow for a PUT**:
1. Gateway receives PUT /bucket/key. Headers include Authorization, Content-Length, and Content-Type.
2. Auth service extracts the access key from the Authorization header, looks up the corresponding secret key (from a local database that mirrors IAM policy), and recomputes the signature. If it does not match, return 403 SignatureDoesNotMatch.
3. Request router parses "bucket" and "key" from the path. Checks that the bucket exists in the metadata database. If not, return 404 NoSuchBucket.
4. Gateway forwards the PUT to the backend (e.g., MinIO), passing the S3 headers through or translating them.
5. Backend stores the object and returns the ETag.
6. Gateway wraps the backend response in an S3-compatible XML response: `HTTP 200 OK` with headers `x-amz-id-2`, `x-amz-request-id`, and `ETag`.
7. Gateway logs the request with duration, object size, and status code.

**Multi-part upload handling**: The gateway must support InitiateMultipartUpload, UploadPart, and CompleteMultipartUpload. The gateway generates upload IDs (UUIDs), tracks part numbers and ETags, and translates these to the backend's multi-part API. Part uploads can go to the backend directly (no gateway buffering) to avoid the gateway becoming a bottleneck.

**Pre-signed URL validation**: The gateway parses pre-signed URLs by reconstructing the signature from the query parameters and the known secret key. It checks the expiration timestamp --?" if expired, return 403 AccessDenied. No additional IAM lookup is needed because the signature validation proves the URL was generated by an authorized party.

## Tier 3: Expert (30 minutes each)

### Challenge 3.1: Global Shared-Nothing Object Storage with Strong Consistency

Design a globally distributed object storage system that provides strong consistency across 3 continents. Users in any region should be able to read the latest write regardless of where the write originated. Latency target: writes confirm in under 100ms for local users; reads complete in under 50ms for local users.

**Requirements**:
- 3 regions: us-east-1, eu-west-1, ap-southeast-1
- Strong consistency across all regions
- Write latency under 100ms for writes to the local region
- Read latency under 50ms for reads from any region
- 5 PB total data, growing at 1 PB/year

**Discussion**:

**Approach A --?" Global leader (not feasible)**: A single leader handling all writes would provide strong consistency but violates the latency requirement. A write from Singapore to us-east-1 would take 200ms+ round trip before the user sees a confirmation.

**Approach B --?" Consensus across all regions (not feasible)**: Running Paxos across 3 continents means every write requires at least 2 of 3 regions to acknowledge. If us-east-1 and eu-west-1 are the two that ACK, the Singapore write waits for both ACKs over transatlantic links --?" still 150ms+.

**Approach C --?" Local writes, async replication, strong reads via read-repair**: Each region accepts writes independently. Metadata is partitioned with the primary copy in one region but replicated asynchronously. To achieve strong reads, the system reads from the primary region for that partition. This means a read in Singapore for a key whose primary is in us-east-1 must cross the Pacific --?" latency is high.

**Approach D --?" CRDT-based (best fit)**: Use Conflict-Free Replicated Data Types (CRDTs) for metadata. Each region accepts writes independently. Metadata includes version vectors. On read, the system computes the merged state from visible replicas. If two replicas conflict (concurrent writes to the same key), CRDT merge rules produce a deterministic result. S3 does not use this approach because it does not fit the last-writer-wins semantics, but this is the cleanest way to meet the latency requirements.

**Recommended approach**: Accept that strong consistency and low latency across continents are fundamentally in tension. Instead, present three options to the stakeholder:

1. **Strong consistency, global, 200ms+ writes**: Single region for primary metadata, other regions serve replicas. Meets strong consistency but not latency.
2. **Strong consistency within region, eventual across regions**: Standard S3 cross-region replication. Local writes under 10ms, strong locally. Remote reads may see stale data for up to 15 minutes.
3. **Custom solution with CRDTs**: Complex to build but meets both latency and consistency. Requires changes to the client API (applications must handle CRDT semantics).

For most product requirements, option 2 is the right answer. Option 3 is only justified if the business requirement truly demands strong consistency across continents with local write latency.

### Challenge 3.2: Small-File Storage for 500 Billion IoT Sensor Readings

An IoT platform ingests 500 million sensor readings per day. Each reading is a 256-byte payload with a 64-byte metadata header (320 bytes total). Readings must be stored for 90 days before deletion. The system serves 100 million read requests per day, typically requesting a batch of 1,000 readings for a specific sensor and time range. Design the storage system for lowest total cost.

**Discussion**:

**Baseline S3 costs**: 500M objects/day x 320 bytes = 160 GB/day. Storing each reading as an individual S3 object: 500M PUT requests/day at $0.005/1K = $2,500/day. After 90 days: 14.4 TB stored. Total 90-day cost: $2,500/day x 90 = $225,000 in PUT costs alone, plus $12,000 for storage. Total: ~$237,000. Unacceptable.

**Batched approach**: Aggregate readings into 64 MB files. Each batch file contains 64 MB / 320 bytes = ~200,000 readings. Daily batches: 500M / 200K = 2,500 batches. PUT cost: 2,500 x $0.005/1K = $0.0125/day. Storage cost: 160 GB/day x 90 days x 3x replication = 43.2 TB = $1,000/month. Total: ~$1,000/month.

**Metadata index**: Build a metadata database (DynamoDB or custom) mapping `(sensor_id, timestamp) --?' (batch_key, offset, length)`. Each batch is stored with a manifest file inside the batch that maps sensor IDs to byte offsets.

**Read path**: Client requests sensor X data for hour Y. Look up (X, Y) in the metadata database --?' get batch file ID and offsets. Fetch the batch file from S3 (if not cached). Extract the 1,000 readings from the batch file. Cache the batch file locally (in-memory or on SSD) because multiple sensors may share the same batch.

**Total cost comparison**:

| Approach | Monthly Storage | Monthly Requests | Monthly Total |
|----------|----------------|-----------------|---------------|
| Individual objects | $1,000 | $75,000 | $76,000 |
| Batched (64 MB) | $1,000 | $0.38 | ~$1,000 |
| Batched + metadata DB | $1,000 + $500 (DB) | $0.38 | ~$1,500 |

**Key insight**: The metadata database cost ($500/month for DynamoDB) is more than offset by the elimination of per-object request costs. The batch approach also reduces GET costs for 100M daily reads: instead of 100M GET requests at $0.0004/1K = $40/day, you fetch ~16,000 batch files/day = $0.0064/day.

**Operational considerations**: Batch compaction is needed --?" when old batches span the deletion boundary (some readings deleted, others kept), you must rewrite the batch without the deleted entries. Run compaction daily on batches that intersect the 90-day cutoff. Monitor batch file sizes --?" if a batch grows beyond 64 MB due to inserting late-arriving data, split it into two batches.
