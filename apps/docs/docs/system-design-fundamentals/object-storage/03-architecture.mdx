### Breaking Points

**At 10K objects**: The database performs fine. The NFS mount has enough inodes. You feel good about yourself.

**At 100K objects**: Database queries on non-indexed columns start taking hundreds of milliseconds. Listing objects by user (a common operation) requires a full table scan on `WHERE user_id = ?`. You add indexes and it helps temporarily.

**At 10M objects**: The metadata table is 200 GB. Indexes do not fit in RAM anymore. A simple LIST operation takes 2-5 seconds. The NFS server handles 500 concurrent requests but you have 3,000 application threads. NFS file locks cause frequent "stale file handle" errors. The nightly backup window has grown from 30 minutes to 8 hours.

**At 100M objects**: The NFS filesystem has exhausted its 32-bit inode count (about 4 billion on ext4, but practical limits are reached far sooner due to directory size). Directory listing for a single user's prefix takes 60 seconds because the directory entry count is in the millions. A single disk failure requires restoring from backup — taking 24 hours with 20% data loss risk.

### Key Architectural Decisions

**Metadata Sharding**: The key space is divided into 256 or more shards. Each shard is an independent Raft group with its own leader. This allows 256 parallel write operations simultaneously. The partition manager routes requests to the correct shard based on the key hash.

**Separation of Metadata and Data**: Metadata operations (key lookups, permission checks, listing) never touch the data path. Data operations (reading/writing bytes) never block on the metadata path beyond the initial location lookup. This means you can saturate your network with data transfers without affecting metadata response times.

**Placement Isolation**: The placement engine ensures that the three replicas of any chunk are placed on different racks and, in multi-AZ configurations, different availability zones. This prevents correlated failures (a single power strip failure cannot destroy all copies).

## Multi-Region and Global Scale

### Cross-Region Replication

Cross-region replication (CRR) copies objects from a source bucket in one region to a destination bucket in another. It is asynchronous: the source write completes and is confirmed before replication begins. Replication typically completes within 15 minutes (often under 1 minute for small objects). CRR is used for disaster recovery, compliance, and latency reduction for global users.

### Global Bucket Performance

When your users are distributed globally, writing to a single region incurs cross-continent latency. To solve this, you can use a multi-region access point or write to the closest region and let CRR propagate.

| User Location | Write to us-east-1 | Write to closest region |
|---------------|-------------------|------------------------|
| New York      | 5 ms              | 8 ms                   |
| London        | 65 ms             | 12 ms                  |
| Sydney        | 180 ms            | 15 ms                  |
| Tokyo         | 150 ms            | 10 ms                  |

## Capacity Planning

### QPS Calculation

You estimate 10 million users, each uploading 2 photos per day and viewing 50 photos per day. Peak-to-mean ratio is 5:1.

- Daily writes: 10M users x 2 photos = 20M writes
- Daily reads: 10M users x 50 reads = 500M reads
- Peak writes per second: (20M / 86,400) x 5 = ~1,160 writes/sec
- Peak reads per second: (500M / 86,400) x 5 = ~28,900 reads/sec

A single metadata shard handles 5,500 reads/sec. You need 6 shards for reads. Writes are distributed across shards by key hash. With 6 shards, each handles ~200 writes/sec (well under the 3,500 limit).

### Storage Sizing

- Average photo size: 3 MB (JPEG)
- Thumbnail: 50 KB x 3 sizes = 150 KB
- Total per uploaded photo: ~3.15 MB
- Daily storage growth: 20M photos x 3.15 MB = 63 TB/day
- 90-day retention on hot storage: 63 TB x 90 = 5.7 PB
- Replication factor 3x: 5.7 PB x 3 = 17.1 PB raw
- Additional 10% overhead for metadata, logs, and scratch space: ~19 PB
- Total cluster raw capacity needed at any time: ~20 PB

### Bandwidth

- Peak write throughput: 1,160 writes/sec x 3 MB avg = 3.48 GB/s
- Peak read throughput: 28,900 reads/sec x 3 MB avg = 86.7 GB/s (hot content mostly served from CDN)
- With 80% CDN hit rate: reads from origin = 28,900 x 20% = 5,780 reads/sec = 17.3 GB/s
- Network requirement per storage node: 10 Gbps minimum (1.25 GB/s)
- With 20 PB raw at 16 TB per disk and 12 disks per node: ~104 storage nodes
- Each node handles 17.3 GB/s / 104 = ~166 MB/s — well within 10 Gbps

### Compute Sizing for Metadata

- 6 metadata shards, each a 3-node Raft group: 18 total metadata servers
- Each metadata server: 64 GB RAM, 16 vCPU, 4 TB NVMe for WAL
- Memory per shard: 200M objects across 6 shards = ~33M objects per shard
- Each object metadata record ~256 bytes: 33M x 256 = 8.4 GB (fits in 64 GB with room for indexes)

## Deployment and Operations

### Zero-Downtime Deploy

| Step | Action | Expected Impact |
|------|--------|----------------|
| 1 | Drain connections from Router-4 | 0.1% capacity reduction |
| 2 | Deploy new binary to Router-4 | 0% (other routers handle load) |
| 3 | Re-enable Router-4 | 0% |
| 4 | Repeat for Routers 1-3 | 0% |
| 5 | Transfer metadata shard 0 leadership | 200ms write pause for shard 0 keys |
| 6 | Deploy new metadata binary to old leader | 0% |
| 7 | Transfer leadership back | 200ms pause |
| 8 | Repeat for all 6 shards | 200ms pauses, non-overlapping (2 min window) |
| 9 | Rolling restart storage nodes (10% at a time) | 10% capacity reduction, no data loss |

### Monitoring and Debugging

| Signal | What to Watch | Action |
|--------|---------------|--------|
| Request error rate (4xx/5xx) | Above 0.1% is unusual | Check auth service, rate limits |
| Latency p99 spike | Above 500ms for writes | Look for metadata leader issues |
| Disk latency | Above 50ms average | Replace failing disk |
| Replication queue depth | Above 10,000 pending | Add storage nodes or investigate network |
| GC backlog | Deleting slower than adding | Increase GC workers |
| Metadata DB WAL rate | Above 50 MB/second | Scale metadata shards |

### Capacity Upgrades

When your cluster approaches 80% disk utilization, you add storage capacity. Adding storage nodes is as simple as racking new servers and configuring the placement engine to recognize them. Consistent hashing ensures that only a fraction of objects (proportional to new capacity / total capacity) are relocated. No downtime is needed.

## Key Takeaways

- Metadata and data planes are decoupled — each scales independently
- Consistent hashing distributes load evenly and minimizes rebalancing
- Replication across failure domains (rack, AZ, region) provides durability
- Write quorum = 2 out of 3 balances durability and latency
- Cross-region replication is asynchronous; plan for replication lag under 15 minutes
- Capacity planning must account for 3x replication overhead
