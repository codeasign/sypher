# Object Storage — Interview Playbook

## How This Topic Appears

Object storage questions show up in system design interviews in several forms. The interviewer rarely asks "design S3" directly — they embed object storage within a larger problem or focus on a specific aspect.

### Category 1: Full System Design (45-60 min)

1. "Design YouTube. How would you store and serve the videos?"
2. "Design a photo-sharing application like Instagram. How do you store 2 billion photos?"
3. "Design Dropbox. How do you handle file storage, sync, and versioning?"
4. "Design Netflix's content delivery pipeline. How are movies stored and transcoded?"
5. "Design a backup service like Backblaze. How do you store petabytes of backup data?"

### Category 2: Feature Deep-Dive (20-30 min)

6. "How does S3 achieve 99.999999999% durability?"
7. "How would you design a multi-region object storage system?"
8. "Compare object storage, block storage, and file storage. When would you use each?"
9. "How does object storage handle concurrent writes to the same key?"
10. "Design a system that stores 10 billion small (1 KB) files efficiently."

### Category 3: Open-Ended Discussion (10-15 min)

11. "What happens when you PUT a 5 GB object to S3? Walk through the steps."
12. "How would you reduce storage costs for a system storing 500 TB of infrequently accessed data?"

## What the Interviewer Is Actually Testing

When an interviewer asks about object storage, they are evaluating these capabilities:

**System decomposition** — Can you break a distributed storage system into its fundamental components (metadata plane, data plane, replication, consistency)? Are you describing monolithic or modular architecture?

**Consistency understanding** — Do you understand strong vs. eventual consistency? Can you explain read-after-write consistency, quorum, and how partitions affect correctness?

**Durability mechanisms** — Can you explain replication, erasure coding, checksums, and scrubbers? Do you know why 11 nines requires more than just 3x replication?

**Performance intuition** — Do you know when object storage is the right choice versus a database or filesystem? Can you estimate throughput, latency, and capacity needs?

**Cost awareness** — Do you consider operational cost in your design? Can you compare standard vs. IA vs. Glacier tiers and recommend based on access patterns?

## Level-by-Level Expectations

### L3/L4 (Entry-level / Junior)

**What they test**: Basic understanding of what object storage is and a simple architecture.

**Sample question**: "How would you store user-uploaded profile pictures?"

**Good answer**: "I would use S3 with a bucket per environment. The application generates a pre-signed URL for uploads and stores the S3 key in a database. Images are served directly from S3 or through a CDN. For thumbnails, I would use S3 Event Notifications to trigger a Lambda that resizes the image."

**What to include**: Buckets, keys, pre-signed URLs, S3 events. Estimate storage: 10M users x 2 profile photos x 200 KB = 4 TB. This fits in standard tier with 3x replication.

**What to avoid**: Designing custom object storage from scratch. Discussing erasure coding or Paxos.

### L5 (Senior)

**What they test**: Deep understanding of trade-offs and the ability to design for scale with realistic constraints.

**Sample question**: "Design a photo storage system for 100 million users."

**Good answer**: "I would use S3 as the primary store but with careful key design. Photos are stored with key pattern `{user_id_hash:04x}/{year}/{month}/{day}/{photo_id}.jpg` to distribute load across prefixes and enable efficient listing by date. I use a CDN for delivery with 80%+ cache hit rate. Metadata (user ID to photo list, captions, tags) is in a separate database — DynamoDB for the photo-to-user map and a relational database for social features. Cost estimate: 100M users x 50 photos x 3 MB = 15 PB. At $0.023/GB, that is $345,000/month for storage plus CDN costs. I would implement lifecycle rules to move old photos to Glacier and consider compression."

**What to include**: Key prefix design, CDN, metadata separation, cost estimation, lifecycle management. Mention throughput limits (3,500 writes/sec per prefix) and how you scale beyond them.

**What to avoid**: Over-engineering. Do not design a custom Paxos-based metadata store for a problem that S3 handles well.

### L6 (Staff / Principal)

**What they test**: Ability to design custom infrastructure when S3 does not fit, with deep knowledge of consensus protocols, erasure coding, and operational concerns.

**Sample question**: "We need to store 100 billion small documents (4 KB each) with strict read-after-write consistency. S3 request costs are too high. Design a custom solution."

**Good answer**: "I would not store each document as an individual S3 object. The request cost would dominate. Instead, I would batch documents into 64 MB archives and store them in S3. I would build a metadata service using a sharded key-value store (FoundationDB or custom Raft-based store) that maps document IDs to (archive key, offset, length). For reads, the metadata store returns the archive location and the client fetches the 64 MB archive and extracts the document. To reduce latency, I would add a caching layer (Redis or memcached) for hot documents. To handle concurrent writes to the same batch, I would use a write-ahead log and merge batches periodically. Cost comparison: direct S3 for 100B objects = ~$5M/month in PUT requests. Batched: 100B x 4 KB = 400 TB, divided into 64 MB archives = 6.25 million PUTs = ~$31,000/month. Plus metadata database costs."

**What to include**: Request cost calculations, batching strategies, metadata store architecture, caching, consistency guarantees. Discuss the trade-offs of batching (increased read latency, write amplification on deletes).

**What to avoid**: Ignoring operational complexity. Acknowledge that custom storage infrastructure requires a dedicated team to operate.

### L7/L8 (Distinguished / Fellow / VP)

**What they test**: Vision, multi-year strategy, organizational impact, and ability to define company-wide storage architecture.

**Sample question**: "Our company stores 10 exabytes across 5 cloud providers. Storage costs are growing 50% year-over-year. What is your strategy?"

**What they look for**: Can you identify cost drivers (egress, API calls, storage class)? Can you design a multi-cloud storage abstraction layer? Can you influence organizational structure (dedicated storage team, unified SLAs)?

## FAANG Framework for Answering

Use this 6-step framework for any object storage interview question.

### Step 1: Scope and Requirements (2 minutes)

Ask clarifying questions:
- "How many users? Expected growth over 3 years?"
- "Average object size? Range? (thumbnail: 50 KB, video: 500 MB)"
- "Read-to-write ratio? Access patterns? (recent vs. old data)"
- "Consistency requirements? (read-after-write or eventual?)"
- "Any compliance or data residency requirements?"
- "Is this a new system or replacing an existing one?"

### Step 2: High-Level Design (5 minutes)

Draw the two-plane architecture:
- Metadata plane: request router, metadata service, location database
- Data plane: storage nodes, placement engine, replication
- Call out the separation and why it matters (independent scaling)

### Step 3: Data Flow (5 minutes)

Walk through one write and one read:
- Write: client PUT — auth — metadata registration — chunk placement — parallel writes — quorum — response
- Read: client GET — metadata lookup — chunk fetch — assembly — response
- Label each step with the latency it adds

### Step 4: Deep Dive (10 minutes)

Pick 2-3 areas the interviewer signals interest in:

**Consistency**: "How do you ensure read-after-write consistency?" Talk about quorum (W+R > N) and the metadata leader serializing writes.

**Durability**: "How do you achieve 11 nines?" Talk about 3x replication across AZs, checksums, scrubbers, repair daemons.

**Scale**: "How do you handle 100K writes/sec?" Talk about sharding the metadata store by key hash, multipart upload parallelism, CDN offload.

**Cost**: "How do you keep costs under control?" Talk about storage tiers, lifecycle policies, compression, deduplication.

### Step 5: Edge Cases and Failure Modes (3 minutes)

- "What happens during a network partition?" — CP vs. AP trade-off
- "What if a disk fails during repair of another disk?" — correlated failure
- "What if all three replicas of a chunk are corrupted?" — scrubber detection limits
- "What if the metadata leader crashes?" — Raft election + WAL replay

### Step 6: Bottlenecks and Trade-offs (5 minutes)

- "What is the bottleneck at 10x current scale?" — metadata shard saturation
- "What would you change if write latency is too high?" — lower W from 3 to 2
- "What would you change if storage costs are too high?" — erasure coding instead of replication
- "What would you change if read latency is too high?" — add CDN, caching layer, or SSD tier

## Phrase Bank — 15 Expert Phrases

Use these phrases to demonstrate depth. Do not force them — weave them naturally into your explanation.

1. "Metadata and data planes are decoupled, allowing independent scaling."
2. "Consistent hashing with virtual nodes minimizes data movement during cluster changes."
3. "Write quorum of 2 out of 3 balances durability against tail latency."
4. "Background scrubbers with Merkle tree verification detect and repair silent corruption."
5. "The last-writer-wins semantics means concurrent PUTs to the same key overwrite without locking."
6. "Cross-region replication is asynchronous; you should plan for replication lag under 15 minutes."
7. "Erasure coding (e.g., 12+4 scheme) provides equivalent durability to 3x replication at 33% lower storage cost."
8. "The 3,500 writes/second per prefix limit is a metadata partition constraint, not a total system limit."
9. "Multi-Part Upload converts a serial transfer into a parallel one, improving throughput by up to 10x for large objects."
10. "Pre-signed URLs shift the bandwidth burden from your servers to the object storage edge."
11. "Lifecycle transitions are the single most impactful cost optimization for object storage."
12. "The CAP trade-off here is CP: during a partition, we reject writes that cannot achieve quorum."
13. "Object immutability is a feature, not a limitation — it simplifies caching, replication, and auditing."
14. "Data at rest encryption should be server-side with customer-managed keys for compliance."
15. "Bloom filters at the metadata layer reject nonexistent key lookups without querying the database."

## Red Flags — 8 Things to Avoid

1. **Saying "I would store files on the filesystem"** for a distributed system. This shows you do not understand that POSIX filesystems do not scale horizontally across machines.

2. **Ignoring cost**. An interviewer who asks about storing 100 billion objects expects you to calculate request costs. Saying "just use S3" without mentioning the $500K/month request bill shows lack of production experience.

3. **Over-engineering**. Do not describe a custom Raft-based metadata store when S3 would work. The interviewer wants to see judgment about when to build vs. buy.

4. **Mixing up block storage, file storage, and object storage**. If you describe mounting an EBS volume for multi-server access, the interviewer will correct you — and you lose points.

5. **Forgetting about failure modes**. If your design handles 10 million writes/sec but crashes when any single disk fails, you missed the point.

6. **Not mentioning CDN for read-heavy workloads**. Serving 100 MB video files directly from object storage to every user is expensive and slow. CDN is table stakes for media delivery.

7. **Ignoring the object size distribution**. Designing for 1 KB objects the same way as 1 GB objects ignores the metadata amplification problem. Small objects need batching.

8. **Claiming unlimited throughput**. Every system has limits. Stating "S3 has no performance limits" is incorrect — there are well-documented per-prefix limits.

## Practice Drills

### Drill 1: Estimation (5 minutes)

Estimate the storage cost for a Twitter-like system where 500 million tweets are posted daily, each containing up to 4 images (300 KB each after compression). Images are accessible for 7 days on hot storage, then moved to cold storage for 90 days, then deleted.

- Daily image storage: 500M x 4 x 300 KB = 600 TB/day
- Hot storage (7 days): 4.2 PB x $0.023/GB = $96,600/month
- Cold storage (90 days): 54 PB x $0.004/GB = $216,000/month
- Request costs: 500M writes/day x $0.005/1K = $2,500/day = $75,000/month
- Image CDN: 4 images x 300 KB x 500M users x 50 views/day = 30 PB/month egress = ~$1.5M/month
- **Total: ~$1.9M/month for image storage and delivery**

### Drill 2: Key Design (5 minutes)

Design the key prefix structure for a file storage system where users store files organized in folders, and the most common operations are "list all files in a folder" and "find a file by name."

```
{user_id_hash:04x}/{parent_folder_id}/{file_name}
```

Challenge: Folder renames are expensive because every file key contains the parent folder. Solution: Store folder membership in a database; use opaque file IDs as keys.

### Drill 3: Consistency Scenario (5 minutes)

A user uploads their profile photo (PUT), then immediately opens their profile page (GET). In a system with eventual consistency, they might see the old photo. How would you fix this?

Answer: Use strong read-after-write consistency (S3 default since 2020). Alternatively, use session consistency: after a write, the metadata server's response includes a write token. Subsequent reads from the same client include the token, and the metadata server waits until the token's version is visible before responding.

### Drill 4: Failure Scenario (5 minutes)

Three replicas of a popular 500 MB file are stored on disks A, B, and C. Disks A and B are on the same rack. The rack loses power. What happens?

Answer: Two of three replicas become temporarily unavailable. The third replica on disk C (different rack) serves reads. Write operations for that file can continue if the write quorum threshold is 1 (dangerous) but cannot continue if W=2 (since only one replica is available). After power is restored, the placement engine verifies no data was corrupted during the outage. If either A or B developed bad sectors on restart, the repair daemon copies a fresh replica from C.

### Drill 5: Cost Optimization (5 minutes)

Your monthly S3 bill is: storage $50,000, PUT requests $15,000, GET requests $8,000, data transfer $25,000. Total: $98,000. Identify three cost reduction actions.

1. **PUT requests**: Use Multi-Part Upload with larger part sizes. If average object is 1 MB and you are making individual PUTs, switch to batching or increase part threshold.
2. **Storage**: Set lifecycle policy to transition objects older than 30 days to S3 Standard-IA ($0.0125/GB vs $0.023/GB). If 70% of data is older than 30 days, storage drops from $50K to $26K.
3. **Data transfer**: Add CloudFront CDN. If 80% of GET requests hit the CDN cache, egress costs drop from $25K to $5K.
