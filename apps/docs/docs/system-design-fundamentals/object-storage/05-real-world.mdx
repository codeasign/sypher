# Object Storage in Production

## Company Case Studies

### Netflix — Media Storage at Petabyte Scale

Netflix stores its entire content library in AWS S3. Every movie and show is stored as a set of master files (video, audio, subtitles) and transcoded renditions. As of 2025, Netflix has over 240 PB of video content in S3, distributed across 5+ AWS regions.

**The challenge**: Netflix must deliver content to 260 million subscribers across 190 countries. Each title is transcoded into 50+ rendition combinations (resolution + codec + bitrate). When a new movie is ingested, it generates hundreds of output files. The total object count in S3 exceeds 50 billion.

**The solution**: Netflix uses S3 as the origin store behind its CDN (Open Connect). Master files are stored once; transcoded renditions are stored with a key structure that encodes resolution, codec, and language.

### Dropbox — Migrating from S3 to Custom Object Storage

Dropbox originally stored all user files on AWS S3. By 2014, they were storing 500 PB and paying AWS tens of millions annually. They built their own object storage system called **Magic Pocket** and migrated 500 PB off S3.

**The challenge**: Dropbox needed to match S3's durability (11 nines) while reducing storage costs by 50%. They also needed to support block-level sync (splitting files into 4 MB blocks) and Delta Sync (uploading only changed portions of files, not entire files).

**The solution**: Magic Pocket uses a custom metadata store called Edel (based on Paxos), a block store that distributes 4 MB blocks across commodity servers, and a multi-tier caching system for frequently accessed blocks.

### Cloudflare R2 — Egress-Free Object Storage

Cloudflare launched R2 in 2022 with a disruptive pricing model: zero egress fees. Competitors charge $0.05-$0.09/GB for data transfer out. R2 charges zero, making it attractive for data-heavy applications.

**The challenge**: Supporting S3-compatible APIs while operating on Cloudflare's edge network (330+ locations) with no egress charges. The business model relies on storage revenue and compute (Workers) rather than bandwidth.

**The solution**: R2 stores data across Cloudflare's data centers using a global metadata plane. Objects are stored in a primary region and cached at edge locations on access. Egress is free because it primarily serves data from regions close to the user, minimizing cross-region transfer.

**Key lesson**: Egress pricing is a significant differentiator. For data-intensive workloads (media distribution, ML datasets, backups), egress costs can exceed storage costs. R2's zero-egress model challenges the traditional S3 pricing structure.

### Amazon S3 — The Original

Amazon S3 launched in March 2006 and was Amazon's first truly global cloud service. Before S3, developers had to manage their own storage infrastructure — buy servers, configure RAID, set up backups, and handle scaling. S3 abstracted all of that behind a simple API.

**The architecture**: S3's design was inspired by Dynamo (Amazon's internal key-value store) and Google's GFS. The key insight was the separation of metadata and data planes. The metadata plane, initially based on a modified Dynamo, was later replaced by a custom system called ShardStore. The data plane uses a cluster of storage nodes called "cells," each containing 10-100 PB of storage.

## Open Source Implementations

### MinIO

MinIO is the most widely deployed open-source S3-compatible object store. It is written in Go, designed to run on commodity hardware, and claims compatibility with the majority of the S3 API.

- Single binary, under 100 MB
- Can run on a single server or scale to hundreds of nodes
- Supports erasure coding (default Reed-Solomon with data+parity shards)
- Bitrot protection with SHA-256 checksums
- Bucket replication, versioning, lifecycle management
- No separate metadata service — metadata is stored alongside data on each node
- Widely used in on-premises, edge, and AI/ML pipelines

**When to choose MinIO**: You need S3 compatibility on your own hardware. You have consistent, moderate throughput demands. You want a simple deployment (single binary, no external dependencies).

### Ceph RGW (RADOS Gateway)

Ceph is a distributed storage system that provides object, block, and file storage through a single cluster. The object gateway (RGW) exposes an S3-compatible API.

- Based on RADOS (Reliable Autonomic Distributed Object Store)
- Uses CRUSH algorithm (no central index) for data placement
- Supports S3 and Swift APIs
- Multi-site replication for geo-redundancy
- Built-in lifecycle management, versioning, encryption
- More complex to deploy and tune than MinIO

**When to choose Ceph**: You already run Ceph for block or file storage and want to add object storage. You need multi-protocol access (object + block + file) on the same cluster. You have a dedicated operations team.

### OpenStack Swift

Swift is the original open-source object store, predating S3's API dominance. It is used primarily in OpenStack deployments as the default object store for VM images and tenant data.

- Uses a ring-based consistent hashing algorithm
- Supports S3 compatibility via middleware
- Multi-region replication with eventual consistency
- Strong focus on multi-tenancy and quota management

## Lessons Learned — 5 Patterns

### Pattern 1: Plan for Cost Growth

The fastest-growing cost in object storage is rarely storage itself — it is requests, data transfer, and API calls. A team at a streaming startup stored 50,000 small subtitle files per movie. Their S3 bill was $2,000/month for storage and $14,000/month for GET requests. They merged subtitles into larger archives and cut the request cost by 90%.

**Take action**: Use S3 analytics to understand request patterns. Monitor Cost Explorer for per-API cost breakdowns. Set up budgets and alerts for request costs.

### Pattern 2: Use Multi-Part Upload from the Start

A developer uploaded 10 GB database backups using single PUT requests. A network glitch at 9.5 GB caused the upload to fail. After 3 retries, they retransferred 28.5 GB total. Switching to Multi-Part Upload (100 parts of 100 MB) meant each failed part retransfers only 100 MB.

**Take action**: Always use Multi-Part Upload for files over 10 MB. Most SDKs handle this automatically with a threshold setting. Set the part size based on average file size and expected network reliability.

### Pattern 3: Design Key Prefixes for Access Patterns

A gaming company stored player replays with keys like `replay_{player_id}_{timestamp}`. The most common query was "get all replays for today." This required scanning all objects, regardless of date. They restructured to `replays/{date}/{player_id}/{timestamp}`. Now "list today's replays" is a single prefix scan.

**Take action**: Design key prefixes to match your 3 most common query patterns. Include the highest-cardinality filter first. Use dates when time-range queries are common. Use hash prefixes for random distribution.

### Pattern 4: Configure Lifecycle Policies Early

A startup stored application logs in S3 standard and forgot about them. Two years later, they had 200 TB of logs costing $4,600/month. They implemented lifecycle rules: transition to IA after 30 days ($2,500/month), to Glacier after 90 days ($800/month), and expire after 1 year ($0/month for expired data). Monthly cost dropped 83%.

**Take action**: Apply lifecycle policies at bucket creation. Start with aggressive rules (move to IA after 30 days, expire after 1 year) and relax them only if data access patterns justify it.

### Pattern 5: Test Disaster Recovery

A mid-sized e-commerce company configured cross-region replication to us-west-2 from us-east-1. They never tested failover. When us-east-1 experienced an outage, they discovered the replication destination bucket lacked the correct IAM policies. Restoring access took 6 hours instead of 30 minutes.

**Take action**: Automate disaster recovery drills. Test failover quarterly. Verify that cross-region replication destinations have correct permissions. Document the failover process end-to-end.
