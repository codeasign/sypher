# Object Storage — Trade-offs and Failure Modes

## Core Tension

The one fundamental trade-off in object storage is **durability versus cost versus performance**. You cannot maximize all three simultaneously.

If you maximize durability (replication factor 5, write quorum 5, cross-region replication to three continents), your storage cost is 5x the raw data size, your write latency includes cross-continent round trips, and your system becomes expensive to operate.

If you maximize performance (single copy on NVMe, no replication, write quorum 1, read from local disk), your latency drops to under 2 milliseconds, but a single disk failure causes permanent data loss.

If you minimize cost (write to tape, no replication), you pay under $0.001 per GB per month, but retrieval takes hours and durability is limited to what a single medium provides.

Every object storage design picks a point on this triangle. AWS S3 Standard picks high durability (11 nines) at moderate cost ($0.023/GB) with moderate latency (5-50ms). Glacier picks low cost ($0.004/GB) at the expense of retrieval time (minutes to hours). Local NVMe storage picks low latency (under 0.1ms) with lower durability (RAID 10 on one host).

## Decision Matrix

| Decision Point | Option A | Option B | When to Choose |
|----------------|----------|----------|----------------|
| Replication factor | 3 (default) | 5+ (extreme) | RF=3 for standard workloads. RF=5 for regulatory compliance or single-region with extreme durability requirements |
| Replication mode | Synchronous (within region) | Asynchronous (cross-region) | Sync for read-after-write consistency. Async for disaster recovery where seconds of lag are acceptable |
| Storage media | HDD | SSD / NVMe | HDD for bulk, cold, or large-file workloads. SSD for hot data, small objects, and latency-sensitive apps |
| Storage tier | Standard | IA / Glacier | Standard for data accessed more than once per 30 days. IA or Glacier for backups, logs, archival data |
| Consistency model | Strong (S3) | Eventual (older systems) | Strong for most applications. Eventual only if you can tolerate stale reads (analytics, content distribution) |
| Object immutability | Mutable (overwrite) | Immutable (WORM) | Mutable for general use. Immutable for compliance (SEC 17a-4, FINRA) or ransomware protection |
| Access control | IAM policies + bucket policies | Pre-signed URLs only | IAM for internal services and automated workflows. Pre-signed for user-facing direct uploads |
| Encryption | Server-side (SSE-S3) | Client-side | SSE-S3 for simplicity. Client-side for compliance where the provider must not see plaintext |

## CAP Theorem Positioning

Object storage is a **CP system** in CAP theorem terms: it prioritizes Consistency and Partition Tolerance over Availability during a partition. When a network partition isolates one Availability Zone, the system will reject writes that cannot achieve quorum rather than accepting a write that might lose data.

## Failure Modes

### Failure 1: Disk Failure

A single disk fails in a storage node. The repair daemon detects that chunks on that disk have fewer replicas than the target (RF drops from 3 to 2 for affected chunks). It begins copying the remaining replicas to spare disks.

### Failure 2: Correlated Rack Failure

A power supply unit fails in a rack, taking down all 12 storage nodes in that rack simultaneously. The repair daemon sees 192 TB of data lose one replica. Because the remaining two replicas are in other racks (guaranteed by the placement engine), no data is lost. Recovery time is limited by how quickly you can provision replacement capacity.

### Failure 3: Silent Data Corruption

A memory bit flip on a storage node causes a chunk to be written incorrectly. The background scrubber reads the chunk on the next cycle, computes its checksum, and detects the mismatch. It deletes the corrupt copy and copies a good replica from another node. The corrupt data was never served to a user because the good replica was prioritized during reads. This is why checksum verification is critical.

### Failure 4: Metadata Server Crash

The metadata leader for shard 3 crashes. The remaining two followers detect the leader's heartbeat timeout (5 seconds). They hold a Raft election. Follower B has a slightly more up-to-date WAL and wins the election. It replays any entries from the WAL that were not yet reflected in the metadata store (typically zero or one entry). It begins accepting writes for shard 3. Total downtime for shard 3: 8-12 seconds.

### Failure 5: Full AZ Outage

An entire Availability Zone becomes unreachable due to a regional power event. In a three-AZ deployment, the remaining two AZs still have every chunk (at least two replicas each). Write quorum (2/3) is still achievable. Read operations use the remaining replicas. System operates at reduced capacity but with no data loss. When the third AZ recovers, the repair daemon re-replicates data to restore full replication factor.

### Failure 6: Object Corruption Goes Undetected

All three replicas of a chunk are corrupted by the same root cause (e.g., a metadata bug that stored a wrong checksum). The scrubber reads each replica, computes checksums that match the stored (wrong) checksums, and declares everything healthy. This is an "undetectable corruption" scenario mitigated by periodic full-object integrity checks that compare a hash of the entire reassembled object against the ETag stored at upload time.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Using Object Storage as a Database

You store individual user sessions as separate objects and update them every time the user makes a request. Each write replaces the entire object because object storage does not support partial updates. With 50,000 users active simultaneously, you are sending 50,000 PUT requests per minute for 256-byte session records.

**Why it fails**: Object storage is designed for files (KB to TB), not for small records. Metadata overhead (the database transaction for each key) costs far more per byte than a database write. Write throughput is capped at 3,500 writes/sec per prefix. A database handles 50,000 writes/sec per node for sub-1KB records.

**Better approach**: Use a database (DynamoDB, Redis, PostgreSQL) for session data. Use object storage for the session's associated files (profile images, uploaded documents).

### Anti-Pattern 2: Frequent Listing on a Prefix with Millions of Objects

You store all objects in a single bucket with no prefix hierarchy, then list all objects every minute to check for new uploads. Listing 20 million objects takes 30 seconds and costs $0.05 per 1,000 list requests.

**Why it fails**: LIST operations are expensive because the metadata store must scan the keyspace. With 20 million objects, a LIST requires scanning 20 million index entries. Even paginated, you pay for every 1,000 results.

**Better approach**: Use S3 Event Notifications to trigger Lambda on each upload. Maintain a database table of recent uploads with an indexed timestamp. Query the database for new uploads instead of listing the bucket.

### Anti-Pattern 3: Frequent Small Writes to a Single Prefix

Your IoT devices send 1 KB telemetry readings every 5 seconds. All 100,000 devices write to the same prefix (`telemetry/2026/07/01/`). You hit the 3,500 writes/sec per prefix limit after 17,500 devices.

**Why it fails**: A single prefix maps to one metadata partition. That partition's write capacity is fixed. Scaling requires more partitions.

**Better approach**: Shard by device ID or timestamp at the prefix level. Use `telemetry/{device_id_hash:02x}/{year}/{month}/{day}/{device_id}.json`. This distributes writes across 256 possible prefix hash values, raising the effective throughput limit to 256 x 3,500 = 896,000 writes/sec.

### Anti-Pattern 4: Storing Many Tiny Files Individually

You store 100 million avatar images that are 4 KB each. You pay for 100 million PUT requests, 100 million GET requests, and 100 million metadata entries. The storage cost for the data: 100M x 4 KB = 400 GB = ~$10/month. The cost for requests (PUT $0.005/1k + GET $0.0004/1k): ~$540/month.

**Why it fails**: Request costs dominate for tiny objects. The metadata overhead (256+ bytes per object) is a significant fraction of the object size.

**Better approach**: Batch tiny files into larger archives (tar, Zip). Store a single 400 MB archive containing all 100,000 small files. Use a database index to track individual file locations within the archive. Alternatively, use a dedicated small-file storage system like Facebook's Haystack.

### Anti-Pattern 5: No Lifecycle Policy on Log Data

You store application logs in object storage and never delete them. After 3 years, you have 500 TB of logs in standard storage costing $11,500/month. Your team only needs the last 90 days for debugging; older logs have never been accessed.

**Why it fails**: You are paying standard-tier prices for data that is never read.

**Better approach**: Implement lifecycle rules: transition to IA after 30 days ($0.0125/GB), to Glacier after 90 days ($0.004/GB), and expire (delete) after 365 days. Cost for 500 TB over 3 years: standard tier for 30 days, IA for 60 days, Glacier for 275 days = ~$2,100/month instead of $11,500/month — an 82% savings.

## When NOT to Use Object Storage

**You need sub-millisecond latency.** Object storage adds network hops, metadata lookups, and data assembly overhead. Even the fastest object storage (S3 Express One Zone) has p50 latency of about 2 ms. If you need under 1 ms, use local NVMe or a RAM-based store.

**You need partial updates to files.** Object storage requires rewriting the entire object for any change. If your workflow involves appending to a log file, editing a JSON document in place, or modifying a few bytes of a large binary file, you need a filesystem or a database with page-level updates.

**You need POSIX filesystem semantics.** Object storage does not support file locking, hard links, symbolic links, in-place renames of directories, or concurrent append operations. If you are running a database on top of shared storage, you need a block store (EBS) or a distributed filesystem (EFS, Lustre).

**Your objects are all under 1 KB.** The minimum chargeable object size in S3 is 128 KB (for PUT requests, you pay per request, not per byte — and each request has a fixed overhead). A database or key-value store is orders of magnitude cheaper and faster for tiny records.

**You need strict transactional guarantees across multiple objects.** Object storage does not support transactions, rollbacks, or atomic multi-object operations. If you need to atomically move 1,000 objects from one bucket to another, you must implement your own two-phase commit. Use a database and reference the objects by key instead.

**Your throughput exceeds a single partition's capacity and you cannot shard by prefix.** Some workloads (e.g., a single extremely popular file) cannot be sharded. A single object can only be served from a single partition. At extreme scale, consider a CDN to offload reads or a P2P distribution mechanism.
