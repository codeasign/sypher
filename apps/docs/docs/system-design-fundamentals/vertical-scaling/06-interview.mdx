# Vertical Scaling — Interview Playbook

## How This Topic Appears

Interviewers ask about vertical scaling in four distinct categories. The phrasing reveals what they are actually testing.

### Direct Questions

- "What is vertical scaling and when would you use it?"
- "Compare vertical scaling and horizontal scaling. When do you choose each?"
- "What are the limits of vertical scaling?"
- "How does vertical scaling relate to the CAP theorem?"

### Design Questions

- "Design a system that handles 10,000 writes per second. Start with a single database node and explain how you would scale."
- "You have a PostgreSQL database with a 200 GB working set. The server has 256 GB RAM. Projected growth is 30 GB per month. What happens in 3 months? Walk through your options."
- "Design a caching layer for a read-heavy application. How do you decide between vertical scaling and sharding the cache?"
- "A customer runs their application on a single server and is outgrowing it. Walk them through their scaling options."

### Debugging Questions

- "Your database queries are slow. CPU is at 20%, memory at 85%, iowait at 30%. What is the bottleneck and how do you fix it?"
- "After upgrading from 16 GB to 64 GB of RAM, query latency improved by only 10%. Why?"
- "A single-node application starts returning connection timeouts after a traffic spike. The server has plenty of CPU and RAM. What is wrong?"
- "You migrate from a c5.4xlarge to c5.9xlarge. Query throughput increases by 20% instead of the expected 50%. What would you check?"

### Trade-off Questions

- "You need to store 500 GB of user-generated content with strong consistency. Would you use one large database instance or shard across multiple?"
- "Your team is debating between a single large Redis instance (256 GB) and a Redis Cluster with 4 nodes (64 GB each). What factors affect the decision?"
- "A startup grows 10x per quarter for three quarters. Should they vertically scale or horizontally scale from the start?"
- "Licensing costs for your database are per-core. You need to scale up. What is your strategy?"

## What the Interviewer Is Actually Testing

When an interviewer asks about vertical scaling, they are evaluating:

**Bottleneck diagnosis**: Can you identify which resource is saturated and match the right upgrade to the right bottleneck? Half the answers at this level say "add more RAM" when the problem is CPU-bound, or "add more CPU" when the problem is I/O-bound.

**Instance family knowledge**: Do you know that different cloud instance families optimize for different resources — compute, memory, storage, network — and that picking the wrong family wastes money?

**Cost awareness**: Do you factor in the cost per GB of RAM, the super-linear cost curve of large instances, and the licensing implications for commercial databases?

**The hard ceiling**: Do you know that vertical scaling has a physical limit and can project when that limit will be reached? Can you articulate the migration path to horizontal scaling before the ceiling hits?

**Operational maturity**: Can you describe the downtime impact of instance upgrades, the monitoring signals that precede a bottleneck, and the configuration changes required after scaling?

## Level-by-Level Expectations

### L3/L4 (Entry-Level / Junior)

**What to answer**: Define vertical scaling as adding resources to a single machine. Give a simple example: upgrading from t3.medium to t3.large when CPU is saturated. Mention that horizontal scaling adds more machines instead.

**Sample answer**: "Vertical scaling means making your server bigger — more CPU, more RAM, faster disk. For example, if your database is running at 80% CPU on a 4-core instance, you could move to an 8-core instance. The application code does not need to change. The trade-off is that there is a maximum instance size, and the largest instances are expensive."

**Evaluation criteria**: Accurate definition. Basic trade-off awareness. No significant gaps.

### L5 (Senior Engineer)

**What to answer**: In addition to the L3/L4 content, describe the NUMA implications for large instances, the cache hit ratio and working set concept, the cost per GB analysis across instance families, and the connection pool tuning required after scaling.

**Sample answer**: "Before scaling, I would identify the bottleneck. If the database cache hit ratio is under 95%, adding RAM directly addresses the issue. If CPU is saturated but it is wait time (iowait), I would upgrade storage first, not CPU. On instances larger than 16 vCPU, NUMA topology matters — if the database is on a 32-core dual-socket machine and not NUMA-pinned, you can lose 30% throughput to cross-socket memory access. After upgrading, I would tune shared_buffers for PostgreSQL and increase the connection pool size to match the new thread capacity. I would also monitor the new bottleneck that emerges — often, after scaling CPU, the connection pool or storage becomes the next constraint."

**Evaluation criteria**: Multi-dimensional analysis. Bottleneck-first approach. Understanding of NUMA, cache ratio, and post-upgrade tuning. Awareness that vertical scaling has a ceiling and can project time-to-ceiling.

### L6 (Staff Engineer)

**What to answer**: In addition to L5 content, describe the capacity planning math — projecting working set growth, instance saturation timeline, and total cost of ownership across vertical and horizontal options. Discuss the migration strategy from vertical to horizontal before the ceiling is reached.

**Sample answer**: "I would approach this with a capacity plan. Given current working set of 200 GB on a 256 GB instance with 30 GB/month data growth, you have approximately 2 months before cache misses start degrading performance. In 4 months, the working set exceeds RAM entirely. The vertical option is an r7g.8xlarge (512 GB) at $2,400/month, which buys another 8 months. The horizontal option is a sharded architecture with 4 nodes — 4x r7g.xlarge at $800/month total — plus application changes for query routing. I would recommend immediately adding a Redis cache layer to reduce the working set pressure, buying 3-4 months for the horizontal migration. Start the sharding design in month 2, implement in months 3-5, and dual-write in month 6, switching over before month 8 when the vertical ceiling is reached. The migration cost is approximately 2 engineering-months, compared to licensing cost escalation if we stay on a single commercial database instance."

**Evaluation criteria**: Quantitative capacity planning. Migration roadmap with timeline. Cost comparison across options. Trade-off articulation taking into account engineering effort, licensing, and operational risk.

### L7/L8 (Principal / Distinguished Engineer)

**What to answer**: In addition to L6 content, describe the organizational impact of choosing vertical versus horizontal scaling — team structure, operational maturity, on-call rotation, deployment frequency, and risk posture. Discuss how the choice aligns with business constraints like compliance, regulatory requirements, and auditability.

**Sample answer**: "The decision between vertical and horizontal scaling is not just technical — it is organizational. A vertically scaled system requires a team that can manage a single critical machine: deep expertise in one database, one OS, one storage stack. If the team is 3 people, vertical scaling is likely the correct choice because it minimizes operational surface area. The trade-off is that the machine is a single point of failure, and if the one person who understands NUMA tuning leaves, the knowledge gap is critical.

A horizontally scaled system requires 5+ engineers with distributed systems experience: the team must handle network partitions, eventual consistency, retry storms, and distributed tracing. The blast radius is smaller (one shard out of 20), but the cognitive load is higher. For a regulated industry like payments or healthcare, vertical scaling simplifies audit: all data is in one jurisdiction, one database. For a consumer social app with unbounded data growth, horizontal scaling is inevitable regardless of team size.

My recommendation: start vertical, plan the horizontal migration before the instance ceiling, and invest in the team's distributed systems expertise during the vertical scaling phase. By the time the vertical ceiling approaches, the team should have the skills needed for the horizontal architecture."

**Evaluation criteria**: Organizational awareness of scaling decisions. Risk posture alignment. Business constraint integration. Long-term strategic thinking. Team capability planning.

## The FAANG Framework for Answering

Use this 6-step framework for answering any vertical scaling question in a system design interview.

**Step 1: Identify the bottleneck**. Start by asking: is this CPU-bound, memory-bound, I/O-bound, or network-bound? The correct answer depends entirely on which resource is saturated. Do not suggest a solution until you have identified the bottleneck.

**Step 2: Determine if vertical scaling fits**. Ask three questions:
- Does the bottleneck fit within the largest available instance?
- What is the growth rate — will it exceed that instance within 12 months?
- Are there licensing constraints that make per-core scaling expensive?

**Step 3: Decide the variant**. Which resource needs upgrading? Compute (more CPU), memory (more RAM), or I/O (faster storage)? Each variant has a different instance family.

**Step 4: Plan the cost impact**. Calculate the cost per GB of RAM or per vCPU at the target instance size. Compare with the current cost. Determine whether the upgrade is economically viable.

**Step 5: Plan the operational execution**. Describe the upgrade path: blue-green deployment or stop-start. List the configuration changes needed after the upgrade (shared_buffers, connection pool, worker count). Define the monitoring that will validate the upgrade succeeded.

**Step 6: Plan the ceiling**. Determine when the next upgrade will be needed. If the time horizon is under 12 months, describe the horizontal migration plan that must start now.

## Phrase Bank

Use these phrases in answers to demonstrate expertise:

"Cache hit ratio is the leading indicator of memory saturation. If it drops below 90%, adding RAM is usually the fix."
Use when discussing memory-bound bottlenecks. Shows you monitor the right metric.

"The working set is the subset of data accessed frequently enough that it should remain in memory. Once it exceeds RAM, performance collapses."
Use to explain why vertical scaling for memory is a cliff, not a slope.

"NUMA topology means that on a dual-socket machine, accessing the remote socket's memory costs 50-100% more latency than local memory."
Use in L5+ answers to demonstrate hardware-level understanding.

"The cost per GB of RAM is consistent within an instance family, but you pay a premium for the largest instances in certain families."
Use to show cost awareness — you know vertical scaling is not always linear in price.

"Post-upgrade configuration is as important as the upgrade itself. The application's connection pool, worker count, and database buffer sizes all need adjustment."
Use to show you know vertical scaling is not just a button press.

"Vertical scaling buys time, but you should know how much time it buys."
Use to demonstrate forward-looking capacity planning.

"The OOM killer targets the process using the most memory, which on a database server is almost always the database itself."
Use in debugging questions about unexplained crashes.

"EBS bandwidth is per-instance, not per-volume. A large instance can drive more IOPS to the same EBS volume than a small one."
Use when discussing I/O bottlenecks in AWS environments.

"For commercial databases, the per-core license cost can exceed the instance cost by 10x at scale."
Use in trade-off discussions that involve Oracle, SQL Server Enterprise, or similar.

"The migration from vertical to horizontal takes 12-18 months. Start before you hit the ceiling, not after."
Use in design questions to show migration planning maturity.

"Predictable growth favors vertical scaling. Unpredictable growth favors horizontal scaling."
Use in cost and strategy discussions.

"Reducing the working set by adding an application-level cache often delays the need for vertical scaling by 6-12 months at a fraction of the cost."
Use to show you consider caching as an alternative to scaling.

"Every vertical scaling event shifts the bottleneck. The old problem is solved, but a new one appears."
Use to demonstrate systems thinking — you understand that scaling is an ongoing process.

"Connection pool exhaustion after vertical scaling is the most common post-upgrade failure mode."
Use in debugging questions about post-upgrade issues.

"A vertically scaled system provides strong consistency by default because there is no network partition within a single machine."
Use in CAP theorem discussions to frame vertical scaling's natural advantage.

## Red Flags

These answers signal gaps in understanding:

**"Vertical scaling is always easier than horizontal scaling."**
It is easier in the short term, but the migration from vertical to horizontal under crisis is harder than building for horizontal from the start. Acknowledging the trade-off timeline shows depth.

**"Just add more RAM."**
Without checking whether RAM is actually the bottleneck. Always confirm the constraint before proposing the solution.

**"There is no limit to vertical scaling."**
Every cloud provider has a maximum instance size. Claiming otherwise shows you have not operated at scale.

**"Vertical scaling means the same thing for databases and stateless applications."**
Databases have persistent state, working sets, WAL architecture, and crash recovery — all of which are affected differently by vertical scaling than stateless application servers.

**"NUMA is a feature you turn on."**
NUMA is a hardware topology, not a feature. You can configure whether the OS balances across NUMA nodes, but the physical topology is fixed.

**"Upgrading the instance and forgetting to change application config will still give most of the benefit."**
Wrong. PostgreSQL will still use the old shared_buffers size. The connection pool will still serialize requests. Post-upgrade tuning is mandatory for realizing the full benefit.

**"You can vertically scale indefinitely with cloud providers."**
Cloud providers have the largest instances of any option, but they still have limits: the largest x86 instance has 24 TB of RAM, every available spot is allocated at peak, and some instance families are in limited supply.

**"Just move to a bigger instance type — no one will notice the downtime."**
For systems with uptime SLAs, the 1-5 minute stop-start window may violate the SLA. Blue-green deployments or replica promotion is needed, which adds operational complexity that was not accounted for.

## Practice Drills

Set a timer for each drill. Aim for complete answers within the time limit.

**Drill 1: Bottleneck Diagnosis (5 minutes)**
A PostgreSQL database runs on a 16-core, 64 GB instance. At peak: CPU at 25%, memory at 85% including file cache, iowait at 35%, disk queue depth at 45 on a gp3 volume with 3,000 IOPS baseline. Query P99 is 2 seconds. What is the bottleneck? What upgrade would you recommend? What would you check after the upgrade?

**Drill 2: Capacity Planning (10 minutes)**
An application serves 20,000 requests per second from a single server with 128 GB RAM. The working set is 100 GB and grows at 8 GB/month. CPU is at 60%. Storage is 1.5 TB of 2 TB. How many months until each resource saturates? Which saturates first? What is the next instance type? What is the alternative to upgrading the instance?

**Drill 3: Migration Design (15 minutes)**
A payment processing system runs on a single SQL Server instance on an AWS m5.24xlarge (96 vCPU, 384 GB RAM). The license costs $47,500 per core per year. Data is 8 TB and growing at 15% per year. The system requires ACID compliance and regulatory audit. Design a 12-month plan. Should you continue vertical scaling, migrate to PostgreSQL, or migrate to a distributed database? Frame the decision in terms of cost, compliance, and operational risk.

**Drill 4: Debugging (5 minutes)**
After upgrading a database from r5.4xlarge (16 vCPU, 128 GB) to r5.8xlarge (32 vCPU, 256 GB), query throughput increased by 12% instead of the expected 40-60%. CPU on the new instance is at 35% at peak. Explain three possible causes and how to verify each.

**Drill 5: Trade-off Analysis (10 minutes)**
A social media startup stores user posts (500 million posts, growing at 10 million/day, 1 KB each). Posts are write-heavy (80% writes, 20% reads) and must be available within 5 seconds of creation. The engineering team has 4 people. Their current single MongoDB instance has 2 TB of storage and is at 60% capacity. Should they vertically scale MongoDB, shard it, or switch to a different database? Justify the recommendation considering team size, operational cost, and growth rate.

Mastering these drills and the framework above will prepare you for any vertical scaling question. The challenges section that follows provides hands-on exercises to test your understanding.
