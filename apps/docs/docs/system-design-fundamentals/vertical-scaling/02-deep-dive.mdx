# Vertical Scaling — Under the Hood

Vertical scaling appears deceptively simple — swap a smaller instance for a larger one — but the internal mechanics involve the operating system scheduler, the memory hierarchy, NUMA topology, kernel I/O stacks, and hypervisor resource allocation. Understanding these internals determines whether your upgrade delivers 2x performance or 1.2x performance.

## Internal Architecture

A vertically scaled system consists of the same logical components as any single-server deployment, but at larger resource scales the physical layout of these components matters.

### Key Internal Components

**Process Scheduler (CFS)**: The Linux Completely Fair Scheduler distributes CPU time among runnable threads. It runs a red-black tree of tasks, always selecting the one with the smallest `vruntime` (virtual runtime). On a 32-core system, the scheduler load-balances threads across cores every 1-10 milliseconds. When a thread migrates from core 0 to core 16 (a different NUMA node), its cached memory in L1/L2 becomes invalid — this costs approximately 50 microseconds of stall time.

**NUMA topology**: Modern servers with more than 16 cores typically use multiple CPU sockets. Each socket has its own memory controller connected to local RAM banks. Accessing local RAM takes approximately 100 nanoseconds. Accessing RAM on another socket takes 150-350 nanoseconds depending on whether the data must traverse one inter-socket link (QPI/UPI) or two. A 50% memory access penalty for remote reads means database workloads can suffer a 20-30% throughput loss on a poorly-configured NUMA system.

**Memory hierarchy**: Data travels from the CPU register through L1 cache (0.5ns, 32 KB per core), L2 cache (3ns, 512 KB per core), L3 cache (10ns, 16-32 MB shared per socket), local memory (100ns), remote memory (200ns+), and finally to disk (5,000,000ns for NVMe, 10,000,000ns for EBS). Vertical scaling often means fitting the working set into a higher level of this hierarchy — ideally entirely in RAM, avoiding disk entirely.

**Kernel I/O stack**: Block device requests go through the virtual file system (VFS), page cache, block layer (with I/O schedulers like BFQ or none), and NVMe driver before reaching the physical device. Each layer adds 1-5 microseconds of overhead per I/O operation. On large instances with high IOPS, the kernel's block layer itself can become a bottleneck — Linux `blk-mq` (multi-queue block layer) was introduced specifically to scale I/O processing across many cores.

## Data Structures and Algorithms

### NUMA-Aware Memory Allocation

**What structure**: The kernel maintains a per-node memory allocator with a buddy allocator for contiguous pages and slab allocators for small kernel objects.

**Why this structure**: On a NUMA system, allocating memory from the local node is faster than allocating from a remote node. The kernel's `numa_membind` API lets processes specify which nodes to allocate from. The `mbind()` system call sets a memory policy: `MPOL_BIND` (allocate only from specified nodes), `MPOL_PREFERRED` (prefer specified nodes, fall back to others), `MPOL_INTERLEAVE` (round-robin across specified nodes).

**Time complexity**: Memory allocation from the buddy allocator is O(log N) where N is the number of free page blocks. Allocation with `GFP_KERNEL` flags typically completes in 0.5-3 microseconds. On a contended NUMA node with memory pressure and reclaim, allocation can take milliseconds.

**Space complexity**: Each memory page (4 KB on x86_64) has a `struct page` entry — 64 bytes per page, or approximately 1.6% overhead for a 256 GB system (approximately 4 GB kernel memory for page tracking).

### Linux Completely Fair Scheduler (CFS)

**What structure**: A red-black tree of runnable tasks, keyed by `vruntime` (virtual runtime — actual runtime normalized by task priority).

**Why this structure**: The red-black tree provides O(log N) insertion and deletion of tasks, with O(1) minimum-value lookup. CFS selects the leftmost node in the tree (the task with the smallest vruntime) as the next task to run. This ensures fairness: tasks that have run less get scheduled sooner.

**Time complexity**: 
- Task enqueue/dequeue: O(log N) where N is the number of runnable tasks
- Task selection: O(1) (leftmost node of the tree)
- Load balancing between cores: O(M) where M is the number of cores in the scheduling domain

**Space complexity**: O(N) for the red-black tree of runnable tasks. On a 32-core system with 500 runnable threads, approximately 50 KB for the scheduler data structures.

### Multi-Queue Block Layer (blk-mq)

**What structure**: Per-core submission queues feeding into a shared dispatch queue. Each CPU core has its own software staging queue, which feeds into hardware dispatch queues (one per NVMe device).

**Why this structure**: Before blk-mq (Linux 3.13, 2014), the block layer had a single request queue protected by a global lock. On 32-core systems, lock contention on this single queue caused I/O throughput to plateau at approximately 500,000 IOPS regardless of the underlying device. blk-mq gives each core its own queue, eliminating the lock contention.

**Time complexity**: 
- I/O submission: O(1) per core (no lock, write to per-CPU queue)
- I/O completion: O(1) per core (softirq on the submitting core)
- Dispatch: O(N) where N is the number of software queues

**Space complexity**: Approximately 4 KB per queue per device. For a 32-core system with 2 NVMe devices: 32 CPU queues x 2 devices x 4 KB = 256 KB overhead.

### Full lifecycle on a vertically scaled instance

For a read query on a 32-core, 256 GB server with the working set in memory:

```
Read query (SELECT * FROM orders WHERE id = 12345):
  Network + parse:       ~20 microseconds
  Plan + lock:           ~15 microseconds
  Cache hit (no disk):   ~100 microseconds (buffer cache read)
  Row processing:        ~10 microseconds
  Response:              ~5 microseconds
  ──────────────────────────────────────
  Total P50:             150 microseconds

  On a 4-core, 16 GB server with working set > RAM:
  Cache miss (disk):     ~10,000 microseconds (10ms)
  Total P50:             ~10,150 microseconds (10.15ms)
  ──> 67x slower due to cache miss

Write query (INSERT INTO orders ...):
  Network + parse:       ~20 microseconds
  Plan + lock:           ~20 microseconds (RowExclusiveLock)
  Buffer cache modify:   ~50 microseconds
  WAL write + fsync:     ~2,000 microseconds (2ms — NVMe)
  Response:              ~5 microseconds
  ──────────────────────────────────────
  Total P50:              ~2,095 microseconds (2.1ms)
```

The difference between a cache hit and a cache miss is the difference between 150 microseconds and 10 milliseconds — a 67x penalty. This is why vertical scaling for memory is the most impactful upgrade for database workloads.

## Consistency and State

Vertical scaling keeps all state on a single machine, which avoids many distributed consistency problems but introduces others specific to large single-node systems.

When a vertically scaled server crashes, the recovery time scales with the data size. For a PostgreSQL database with 2 TB of data, recovery can take 30-60 minutes.

### Concurrent Write Contention

On a single-node database, concurrent writes contend for the same resources — CPU, memory, and disk — in a way that distributed writes do not. The write throughput on a single node is ultimately bounded by:

```
PostgreSQL maximum write throughput (single node, NVMe):
  - WAL flush rate:        ~500 MB/s (NVMe sequential write)
  - TPS (transactions/sec) at 1 KB WAL per transaction: ~500,000 TPS
  - TPS at 100 KB per transaction: ~5,000 TPS
  - TPS at 1 MB per transaction (large inserts): ~500 TPS

  Each transaction requires:
    1. WAL write + fsync()        ~2ms
    2. Data page modification      ~0.1ms (in shared_buffers)
    3. Lock acquire/release        ~0.005ms
    4. Transaction commit log       ~0.01ms

  Total per transaction overhead:  ~2.2ms
  Theoretical max TPS:             ~450 TPS (non-batched)
  With batching (group commit):    ~50,000 TPS
```

Vertical scaling increases the ceiling by adding more cores and more RAM (fewer cache misses, less I/O wait), but the single-threaded WAL write process means write throughput scale
s less than linearly with cores. Going from 4 to 32 cores might improve write throughput by 3x, not 8x.

### Stale Reads (Replica Lag)

On a single-node setup with no replicas, there is no replication lag — all reads see the latest committed data because they read from the same storage. This is a strong consistency guarantee that horizontal scaling sacrifices.

However, if the application uses a local cache (Redis on the same node, or in-process cache), stale reads can occur:

```
Scenario: Local cache on single node
  1. App writes new value to PostgreSQL (committed)
  2. App invalidates local cache entry
  3. Another thread reads the cache key
  4. Cache entry is being rebuilt — thread sees empty cache
  5. Thread falls through to database, reads fresh value
  6. Race: if step 5 starts before step 2, thread reads stale data

Resolution:
  - Use read-through cache with TTL (stale tolerated for TTL duration)
  - Use Redis on the same node with exclusive cache-aside pattern
  - Disable local caching if strong consistency is required
```

## Configuration and Tuning

These settings must be adjusted when vertically scaling the database server.

### PostgreSQL Tuning per Instance Size

| Parameter | 4 core/16 GB | 8 core/32 GB | 16 core/64 GB | 32 core/256 GB | Effect of wrong value |
|---|---|---|---|---|---|
| max_connections | 100 | 200 | 400 | 800 | Too low: connection errors. Too high: memory exhaustion from connection overhead |
| shared_buffers | 4 GB | 8 GB | 16 GB | 64 GB | Too low: poor cache hit ratio. Too high: OS page cache starved |
| effective_cache_size | 12 GB | 24 GB | 48 GB | 192 GB | Too low: planner chooses sequential scans over index scans |
| work_mem | 32 MB | 64 MB | 128 MB | 256 MB | Too low: disk sorts. Too high: memory exhaustion with many concurrent sorts |
| maintenance_work_mem | 512 MB | 1 GB | 2 GB | 4 GB | Too low: slow VACUUM and index creation |
| wal_buffers | 16 MB | 16 MB | 64 MB | 256 MB | Too low: WAL write contention under high write load |
| max_worker_processes | 8 | 16 | 32 | 64 | Too low: parallel query not used. Too high: unnecessary process overhead |
| random_page_cost | 1.1 | 1.1 | 1.1 | 1.1 (SSD) | Too high: planner avoids index scans even when beneficial |

### NUMA Tuning

For multi-socket servers, NUMA configuration can make a 30% difference in throughput:

```
NUMA configuration checklist:

1. Check NUMA topology:
   numactl --hardware

2. Pin database to local NUMA node:
   numactl --cpunodebind=0 --membind=0 postgres -D /data

3. Disable NUMA balancing (can cause page migration overhead):
   echo 0 > /proc/sys/kernel/numa_balancing

4. Configure PostgreSQL to disable NUMA:
   In postgresql.conf: numa_memory = off (default since PG 16)

5. For Java applications on NUMA:
   -XX:+UseNUMA   (enables NUMA-aware allocation in JVM)

6. Monitor cross-socket memory access:
   numastat -p postgres
   If "other_node" accesses > 20% of total, NUMA tuning needed
```

### I/O Scheduler Tuning

```
NVMe devices should use the "none" I/O scheduler:
  echo none > /sys/block/nvme0n1/queue/scheduler

EBS volumes should use "none" or "mq-deadline":
  echo none > /sys/block/xvd*/queue/scheduler

Tuning for latency vs throughput:
  Latency-sensitive:  queue/depth = 32  (lower queue depth = lower latency)
  Throughput-focused:  queue/depth = 128 (higher throughput, higher P99 latency)

  PostgreSQL checkpoint tuning to reduce I/O spikes:
    checkpoint_completion_target = 0.9   # spread writes over checkpoint window
    checkpoint_timeout = 15min           # reduce checkpoint frequency
    max_wal_size = 8 GB                  # absorb write bursts without checkpoint
```

### Linux Kernel Tuning

For high-throughput database workloads on large instances:

```
sysctl settings for 32-core / 256 GB server:

# Network
net.core.somaxconn = 65535              # Connection backlog
net.ipv4.tcp_tw_reuse = 1               # Reuse TIME_WAIT sockets
net.ipv4.tcp_fin_timeout = 15            # Faster connection teardown
net.core.netdev_max_backlog = 50000     # NIC backlog
net.core.rmem_max = 16777216             # Max receive buffer
net.core.wmem_max = 16777216             # Max send buffer

# Memory
vm.swappiness = 1                        # Almost never swap
vm.dirty_ratio = 30                      # Allow 30% dirty pages before blocking writers
vm.dirty_background_ratio = 5            # Start background writeout at 5% dirty
vm.vfs_cache_pressure = 200              # Reclaim dentries/inodes more aggressively
vm.zone_reclaim_mode = 1                 # Reclaim local zone before remote (NUMA)

# Kernel
kernel.sched_migration_cost_ns = 5000000 # Reduce context switch overhead
kernel.numa_balancing = 0                # Disable NUMA page migration
```
