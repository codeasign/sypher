# Vertical Scaling in Production

Atlassian ran Jira on a single PostgreSQL instance for over a decade. Discord scaled their Nitro membership service vertically before hitting limits and migrating to sharding. Cloudflare handles 25 million HTTP requests per second on single-node infrastructure for many of their services, pushing vertical scaling further than most companies attempt. These real-world stories reveal when vertical scaling works, where it breaks, and how to plan the migration.

## Company Case Studies

### Atlassian — Single-Node PostgreSQL for Jira (2010-2022)

Atlassian famously ran Jira's largest customers on a single PostgreSQL instance for years. Their database server for enterprise Jira Cloud was an AWS `db.m5.24xlarge` (96 vCPU, 384 GB RAM) with 16 TB of provisioned IOPS storage.

**What they did**: Atlassian chose to vertically scale their PostgreSQL database rather than distribute across multiple nodes. Their reasoning, detailed in their engineering blog: the ACID guarantees of relational databases made application code simpler. Sharding would require rewriting core transaction logic, which carried risk and development cost. Instead, they invested in:

- PostgreSQL replication (async streaming to a warm standby in another AZ)
- Per-customer connection limits to prevent noisy-neighbor issues
- Regular vacuum and index maintenance to prevent bloat
- Query optimization and targeted indexing to reduce working set size

**Where they hit the ceiling**: By 2021, the largest managed instance type could no longer keep up with write load for Jira's enterprise customers. The `db.m5.24xlarge` was at 80% CPU during peak hours. Write throughput was bottle-necked by the single WAL writer. The working set exceeded 256 GB, meaning cache misses forced disk reads.

**The transition**: Atlassian announced a multi-year migration to a horizontally partitioned architecture (they call it "Jira Data Platform"). The migration, started in 2022, splits customers across multiple PostgreSQL clusters. Each cluster handles approximately 500 customers. The total cost is higher (more instances), but the per-customer throughput ceiling is effectively removed.

**Lesson**: Atlassian extracted over a decade of value from a single-node PostgreSQL instance before hitting the ceiling. Their growth was predictable enough that vertical scaling was the right call for most of their history.

### Discord — Single-Node MongoDB to Sharding (2015-2022)

Discord started with a single MongoDB replica set for their message storage. As the platform grew to 150 million monthly active users, they hit hard limits documented in their 2022 engineering blog post "How Discord Stores Billions of Messages."

**What they did**: Discord ran a 6-node MongoDB replica set (one primary, 5 secondaries) for years. Each node was a large EC2 instance. The primary handled all writes. As message volume grew to billions per day, the primary's CPU reached saturation. Write latency increased. The working set of recent messages exceeded the primary's RAM, causing cache misses on every write.

**The vertical scaling ceiling**: Discord's largest MongoDB instance had 2 TB of RAM. The message working set (messages from the last 30 days) exceeded 2 TB. They could not scale up further because MongoDB's WiredTiger storage engine performs best when the working set fits in RAM. Beyond the largest instance, they had to shard.

**The transition**: Discord migrated from a single MongoDB replica set to a sharded cluster with 100+ shards. Each shard handles a range of guilds (servers). The migration took 18 months and required a custom proxy layer to route queries without application changes.

**Lesson**: Discord hit the vertical scaling ceiling because their data was append-only and unbounded. Old messages accumulate indefinitely. Any system with unbounded storage growth will eventually exceed the largest machine. There is no vertical scaling solution for unbounded data — only data retention limits or horizontal sharding can solve this.

### Cloudflare — Vertical Scaling at Massive Scale

Cloudflare's engineering blog describes how many of their edge services run on single nodes — even at 25 million HTTP requests per second. Their architecture uses vertical scaling per point of presence (PoP), with each PoP running on a single large machine.

**What they do**: Cloudflare's core reverse proxy, the L4 load balancer (Unimog), and many caching services are single-process, single-machine designs within each PoP. Each PoP is independently scaled: they add more PoPs for geographic distribution (horizontal at the PoP level), but within each PoP, they use a single large machine (vertical at the node level).

**Why it works**: Cloudflare controls their hardware. They deploy custom servers with 100+ cores and 512 GB+ RAM in each PoP. They do not use cloud instances — they build their own. This gives them access to hardware configurations that cloud providers may not offer.

**Where it breaks**: Cloudflare cannot vertically scale a single PoP beyond what their custom hardware supports. When a PoP reaches capacity, they add another server to that PoP (horizontal at the node level within a PoP). Their architecture is a hybrid: vertical (single machine per service) + horizontal (multiple PoPs) + horizontal (multiple machines per PoP).

**Lesson**: Cloudflare's approach shows that vertical scaling at the component level is viable even at extreme scale, as long as the deployment topology (PoPs, availability zones, regions) provides horizontal capacity at the higher level.

### Uber — The Vertical-to-Horizontal Migration (2014-2016)

Uber's early architecture was a single Node.js application (called "Mongoose") backed by a single PostgreSQL database. As Uber expanded to 100+ cities, the single database could not keep up.

**What they did**: Uber's single PostgreSQL instance (one database, one machine) handled all trip data, driver data, and payment processing. By 2014, the instance was at capacity. Queries for the dashboard analytics took 30+ minutes because they ran on the same database that processed live trip writes.

**The transition**: Uber migrated from the monolith to a microservices architecture with per-service databases. Each service scaled independently — some vertically (small databases), some horizontally (Cassandra clusters for trip data). The trip history database became a Cassandra cluster. The payments database ran on a single large PostgreSQL instance (vertically scaled). The driver locations database became a Redis cluster with sharding.

**Lesson**: Uber's migration shows that vertical scaling is not binary — you can vertically scale some services while horizontally scaling others. The trip history needed unbounded storage (horizontal). The payments database needed ACID compliance (vertical). Different data characteristics call for different scaling strategies.

## Open Source Implementations

### PostgreSQL

PostgreSQL on a single large node is the most well-documented vertical scaling pattern. Key configurations for large instances:

```
PostgreSQL on 64-core / 512 GB instance:
  shared_buffers:        128 GB (25% of RAM)
  effective_cache_size:  384 GB (75% of RAM)
  max_connections:       500
  max_parallel_workers:  32
  max_parallel_workers_per_gather: 8
  max_worker_processes:  32
  wal_buffers:           256 MB

Trade-offs:
  + Strong ACID guarantees on single node
  + No distributed transaction overhead
  - Single node WAL write bottleneck (~500 MB/s max)
  - Crash recovery proportional to data size
  - Parallel query limited to 32 workers
```

### Redis

Redis on a single large instance provides the best latency at the cost of data volume limits:

```
Redis on 256 GB instance:
  maxmemory:         200 GB (leave 56 GB for OS + overhead)
  maxmemory-policy:  allkeys-lru
  maxclients:        10,000
  save: ""            (no RDB — cache workload)
  appendonly: no     (no AOF — cache workload)

Trade-offs:
  + Sub-millisecond P99 latency
  + Simple operational model
  - Hard memory ceiling (200 GB max)
  - Single-threaded CPU (1 core bottleneck for command processing)
  - No persistence for cache workloads (rebuild from DB on restart)
```

### SQLite

SQLite on a single large instance is an under-appreciated vertical scaling pattern for read-heavy workloads. SQLite handles up to approximately 50,000 read transactions per second on a fast SSD — enough for many applications.

```
SQLite on 48-core / 256 GB server:
  PRAGMA journal_mode = WAL;
  PRAGMA cache_size = -64000000;   (64 MB cache)
  PRAGMA mmap_size = 268435456000; (256 GB mmap)
  PRAGMA synchronous = NORMAL;

Trade-offs:
  + Zero configuration, zero maintenance
  + No network latency (embedded, local file)
  + Up to 50,000 reads/sec on modern NVMe
  - Single-writer at a time
  - Corporate compliance concerns (no access controls)
  - No replication built in
```

### AWS Vertical Scaling Path

AWS offers the broadest range of instance sizes. The typical vertical scaling path:

```
t3.medium (2 vCPU, 4 GB, $30/month)
  ──> m7g.large (2 vCPU, 8 GB, $50/month)
    ──> m7g.xlarge (4 vCPU, 16 GB, $100/month)
      ──> m7g.2xlarge (8 vCPU, 32 GB, $200/month)
        ──> r7g.4xlarge (16 vCPU, 128 GB, $600/month)
          ──> r7g.8xlarge (32 vCPU, 256 GB, $1,200/month)
            ──> r7g.16xlarge (64 vCPU, 512 GB, $2,400/month)
              ──> x2gd.16xlarge (64 vCPU, 1,024 GB, $4,500/month)
                ──> u-6tb1.metal (224 vCPU, 6 TB, $25,000/month)
                  ──> u-24tb1.metal (448 vCPU, 24 TB, $100,000+/month)

Cost per GB of RAM:
  t3.medium:   $7.50/GB/month
  r7g.4xlarge: $4.69/GB/month
  r7g.16xlarge: $4.69/GB/month
  u-6tb1:      $4.17/GB/month

  Memory-optimized families have consistent per-GB cost.
  The value is in: IOPS (larger instances drive more EBS throughput).
```

**RDS managed vertical scaling**:
- Downtime: 2-5 minutes during instance class change (30-60 seconds of actual downtime, rest is DNS propagation)
- Storage scaling: gp3 supports online storage increase (no downtime)
- Multi-AZ: failover to standby during instance upgrade (under 1 minute downtime)

### GCP Vertical Scaling Path

GCP offers live migration during instance upgrades, making vertical scaling less disruptive.

```
GCP n2 family:
  n2-standard-2 (2 vCPU, 8 GB, $48/month)
    ──> n2-standard-4 (4 vCPU, 16 GB, $96/month)
      ──> n2-standard-8 (8 vCPU, 32 GB, $192/month)
        ──> n2-standard-16 (16 vCPU, 64 GB, $384/month)
          ──> n2-standard-32 (32 vCPU, 128 GB, $768/month)
            ──> n2-standard-48 (48 vCPU, 192 GB, $1,152/month)
              ──> n2-standard-64 (64 vCPU, 256 GB, $1,536/month)

Key advantage: Live migration means changing instance type
can be done without stopping the VM in many cases.
```

**Cloud SQL managed vertical scaling**:
- Downtime: approximately 60 seconds during machine type change
- Storage: automatic increase (no downtime, no detach/re-attach)
- High availability: failover replica in different zone

## Lessons Learned

### Lesson 1: Know Your Ceiling

Every cloud provider has a maximum instance size. For AWS compute-optimized instances, the ceiling is approximately 64 vCPU. For memory-optimized instances, it is 24 TB RAM (at $100,000+/month). Estimate how many months until you hit that ceiling given your growth rate. If the answer is under 18 months, start designing for horizontal migration now.

### Lesson 2: Vertical Scaling Is Not Binary

As Uber demonstrated, you can vertically scale some services and horizontally scale others. The choice depends on data characteristics:
- Bounded data with ACID requirements: vertical scaling
- Unbounded data or append-only logs: horizontal scaling
- Read-heavy with predictable working set: vertical scaling with caching
- Write-heavy with variable load: horizontal scaling

### Lesson 3: Cache Before You Scale

Before upgrading an instance, add a cache layer. Redis or Memcached on a separate node (or same node) can reduce database load by 80-95%. This often eliminates the need for the instance upgrade entirely. Atlassian, Discord, and Cloudflare all use aggressive caching to extend the useful life of their vertically scaled databases.

### Lesson 4: Plan the Migration Before You Hit the Ceiling

Discord's sharding migration took 18 months. Atlassian's data platform migration took multiple years. If your growth projections show the vertical ceiling is 12 months away, you should have started the migration 6 months ago. The migration itself requires rewriting data access patterns, changing application code, testing at scale, and running dual-write during the transition.

### Lesson 5: Monitor the Right Signals

The signal that it is time to scale up is not just CPU at 90%. Watch the trend of:
- **Cache hit ratio** (database and Redis): declining means working set is growing faster than RAM
- **Disk queue depth**: increasing means I/O throttling is imminent
- **Connection pool wait times**: increasing means more concurrent users than the pool handles
- **P99 latency trend**: creeping up by 5-10ms per month signals approaching saturation

Each of these gives a 3-6 month warning before the bottleneck becomes critical — long enough to plan either an instance upgrade or a horizontal migration.
