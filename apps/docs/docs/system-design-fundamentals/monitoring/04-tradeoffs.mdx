## The Core Tension

Every monitoring system faces a fundamental three-way tradeoff: you cannot simultaneously maximize accuracy, minimize cost, and maximize speed. Pick two.

If you need high accuracy and low cost, you batch ingest and accept that dashboards lag by 5-15 minutes. If you need high accuracy and fast dashboards, you pay for more storage and compute -- every data point is stored at full resolution and indexed immediately. If you need low cost and fast dashboards, you sample aggressively and accept that your metrics are approximations, not exact counts.

Most production systems sit between the cost-speed and accuracy-speed corners. The cost-accuracy corner (batched, delayed) is used for capacity planning and cost analysis, not real-time alerting.

The choice of monitoring architecture depends on your workload characteristics and operational constraints.

| Decision | Option A | Option B | When A Wins | When B Wins |
|---|---|---|---|---|
| Collection model | Pull | Push | You control the network and want to detect dead agents | Agents behind NAT or firewall; no inbound access |
| Storage format | Prometheus TSDB | Elasticsearch/Loki | Metrics-heavy workload, low cardinality | Logs-heavy workload, text search required |
| Sampling strategy | Head-based (client) | Tail-based (server) | Low data volume, simple implementation | High volume, need to keep slow traces |
| Alert evaluation | Server-side | Agent-side | Centralized control, consistent rules | Edge computing, disconnected operation |
| Retention policy | Full resolution + downsampling | Tiered storage (hot/warm/cold) | Single cluster, simple operations | Multi-region, cost-sensitive at scale |
| Query interface | PromQL | SQL | Metrics-native, simple aggregations | Complex joins, business logic in queries |

### Collection Model: Pull vs Push

Pull-based collection gives you automatic detection of dead agents -- if the scrape fails, the agent is down. Push-based collection works through firewalls but cannot distinguish a dead agent from one with no new data. Push systems use heartbeat metrics and dead-man switches as workarounds.

### Storage Format: TSDB vs Log Store

Time-series databases are optimized for numeric data with low cardinality dimensions. A Prometheus TSDB can ingest 1 million samples per second on a single node if the cardinality stays under 100,000 series. Log stores like Elasticsearch are optimized for text search and high-cardinality fields but have 10x higher storage overhead per data point. Use the right tool for the data type.

### Sampling Strategy: Head vs Tail

Head-based sampling decides what to keep at the client, before any context about the request is available. It is simple and cheap but blind to rare events. Tail-based sampling waits until all spans for a trace have arrived, then decides whether to keep it. It can selectively keep all traces with errors or high latency, which is far more useful for debugging, but requires a centralized buffer that adds complexity and cost.

### Alert Evaluation: Server vs Agent

Server-side alert evaluation gives you a single source of truth for alerting rules and consistent evaluation across all agents. Agent-side evaluation works during network partitions -- the agent can still fire alerts even when disconnected from the central system. The tradeoff is that agent-side rules drift from the central configuration over time.

## CAP Theorem and Monitoring

Monitoring systems are CP or AP, never CA. You must choose between consistency and availability during a partition.

│  because partitions are inevitable in distributed systems.           │

│    - Accept that writes may produce duplicates or gaps (AP)          │
└────────────────────────────────────────────────────────────────────┘

Prometheus with a single node is CP: during a network partition, the agent cannot scrape the Prometheus server, and the server cannot accept writes. The system is unavailable for writes from that agent until the partition heals. The agent continues buffering data locally, so no data is lost, but the data is not available to queries.

Cortex and Thanos are AP: during a partition, writes continue to the partition's primary. When the partition heals, writes are reconciled through asynchronous replication. This can produce duplicate data points during the reconciliation window, but the system never rejects writes.

The right choice depends on your requirements. If exact accuracy matters more than continuous availability -- for billing metrics or SLO compliance tracking -- use a CP system for those specific metrics. If continuous availability matters more -- for real-time alerting and dashboards -- use an AP system.

### Failure Mode 1: Cascading Collapse

The monitoring system itself falls over under load, creating a negative feedback loop. High traffic causes the application to emit more metrics. The additional metrics overload the aggregator. The overloaded aggregator drops data and runs slow queries. The slow queries cause Grafana dashboards to time out. The team, now blind, cannot see that traffic has doubled. They deploy a fix but have no way to verify it worked.

```text
Cascading collapse sequence:
  1. Traffic spikes from 50K to 200K req/s
  2. Metrics emitted per second: 5,000 to 20,000
  3. Aggregator CPU: 40 percent to 95 percent
  4. Query latency: 100ms to 5 seconds
  5. Alert manager evaluation: 50ms to 2 seconds per rule
  6. Grafana dashboards: 2 seconds to 30 seconds load time
  7. Team gets timeout errors on every dashboard
  8. Team deploys a scaling fix but cannot verify impact
  9. Without monitoring feedback, team cannot tell if fix worked
```

Prevention: over-provision the monitoring pipeline to handle 5x the expected peak load. Use auto-scaling for aggregators. Set up a separate, independent health monitoring system that tracks the monitoring system itself (a "monitor of monitors").

### Failure Mode 2: Silent Agent Death

An agent process crashes and never restarts. Metrics from that node stop flowing. The dashboard continues showing the last cached values, which look normal. No alert fires because the alert rules are configured on conditions that assume data is flowing.

```text
Silent agent death scenario:
  1. Agent crashes due to OOM at 3:14 AM
  2. Metrics for node web-42 stop after last batch at 3:14 AM
  3. TSDB still serves queries for web-42, using cached segment data
  4. Dashboard shows "web-42 p99 latency: 45ms" -- stale data
  5. Alert rule: "p99 > 200ms for 5 minutes" never fires
     because the stale value is 45ms, which is under the threshold
  6. Discovery: user reports web-42 is down at 7:45 AM
```

Prevention: add a `up` metric that the agent emits every 10 seconds with value 1. If the `up` metric stops arriving, the system knows the agent is dead. Set an alert: `absent(up{instance="web-42"})`. This is the most important alert in your monitoring system.

### Failure Mode 3: Cardinality Explosion

A deployment adds a new metric dimension that creates millions of new time series. The TSDB runs out of memory for the inverted index. Write throughput drops to zero.

```text
Cardinality explosion timeline:
  - Before: 50,000 time series, memtable size: 256 MB
  - Deployment adds label "user_id" to http_requests_total
  - After: 1,050,000 time series (1M new series from 1M users)
  - Inverted index grows from 10 MB to 200 MB
  - Memtable overflows: new series are created faster than
    segments can be flushed
  - TSDB enters "too many series" mode, rejects all writes
  - All metrics for all services stop flowing
  - Complete monitoring blackout
```

Prevention: set a hard limit on the number of time series per metric in the aggregator configuration. A typical limit is 100,000 series per metric. If a metric exceeds this limit, the aggregator rejects writes for that metric and logs an error with the offending label values.

### Failure Mode 4: Alert Fatigue

Too many alerts fire, and the on-call engineer starts ignoring them. Real incidents are missed because they are indistinguishable from noise.

```text
Alert fatigue cycle:
  1. Team adds 20 alert rules for every new service
  2. Total rules: 500 rules, evaluating every 60 seconds
  3. Average alerts per day: 150 (one every 10 minutes)
  4. On-call investigates each alert: 5 minutes per alert
  5. Time spent on alerts: 12.5 hours per day
  6. On-call becomes exhausted, starts acknowledging without investigating
  7. Real incident fires at 3 AM -- acknowledged but not investigated
  8. System goes down for 2 hours before anyone notices
```

Prevention: enforce a signal-to-noise ratio target. Every alert must have fired at least once in a production incident within the last 30 days to remain in the active rule set. If an alert has not fired in 30 days, it is either noise or a metric that never violates its threshold. In both cases, the alert is useless.

### Failure Mode 5: Time Skew

The agent's clock is skewed relative to the aggregator's clock. Metrics are timestamped with incorrect times, causing them to be rejected as "too old" or "too far in the future."

```text
Time skew scenario:
  1. Agent server clock drifts 5 minutes slow
  2. Current time: 10:00 AM
  3. Agent timestamps metrics as 9:55 AM
  4. Aggregator receives at 10:00 AM
  5. Aggregator rejects: "timestamp too old" (>5-minute window)
  6. All metrics from this agent are dropped
  7. Node goes dark in the monitoring system
  8. Team sees node as "down" even though application is running fine
```

Prevention: run NTP on every node with a local NTP server for redundancy. Include the agent's clock accuracy as a metric. Set an alert if the clock skew exceeds 100 milliseconds.

## Anti-Patterns

### Anti-Pattern 1: Dashboard Overload

You create a single dashboard with every metric collected. The dashboard has 50 panels, each querying different metrics with different time ranges. The dashboard takes 30 seconds to load, and nobody can find the panel they need.

**Why it fails**: A dashboard with 50 panels creates 50 separate queries. Each query fan-outs to all TSDB partitions. With 8 partitions and 50 queries, the system executes 400 queries for one dashboard load. If 10 team members load the dashboard simultaneously, that is 4,000 queries.

**Fix**: Create focused dashboards. One dashboard per service, one dashboard per concern (latency, errors, capacity), and one top-level "overview" dashboard with no more than 6 panels. Use dashboard links instead of cramming everything into one screen.

### Anti-Pattern 2: Alerting on Every Spike

You configure alerts on every metric with a static threshold. CPU exceeds 80 percent? Alert. Memory exceeds 70 percent? Alert. Disk exceeds 85 percent? Alert. The on-call engineer gets paged 20 times per night.

**Why it fails**: Static thresholds do not account for normal operating ranges. A service that typically runs at 30 percent CPU might spike to 80 percent during a legitimate traffic burst and recover in 30 seconds. The alert is noise.

**Fix**: Use dynamic thresholds based on historical baselines. Alert on rate of change rather than absolute values. Use the `for` duration parameter to require sustained violation. Every alert must be actionable -- if the response does not involve a specific action, remove the alert.

### Anti-Pattern 3: Collecting Everything

You collect every metric, log, and trace at full resolution with no sampling. The storage cost doubles every quarter. Query performance degrades as the dataset grows. The team cannot find relevant data in the noise.

**Why it fails**: Collecting everything is the monitoring equivalent of storing every security camera frame for 10 years. The storage and compute costs grow linearly with data volume, while the marginal value of each additional data point approaches zero.

**Fix**: Apply a tiered collection strategy. Collect high-value metrics (error rates, latency percentiles, throughput) at full resolution. Collect medium-value metrics (per-endpoint breakdowns) with 10-second resolution. Collect low-value metrics (per-user histograms) with head-based sampling at 1 percent.

### Anti-Pattern 4: Monitoring as an Afterthought

You build the application first and add monitoring as an afterthought. The team ships without any instrumentation. When the first production incident hits, there is no data to debug with.

**Why it fails**: Monitoring cannot be retrofitted effectively. Without baseline metrics from day one, you cannot distinguish a regression from normal behavior. Without structured logging from the start, you cannot search for error patterns. Without distributed tracing, you cannot debug cross-service latency.

**Fix**: Treat monitoring as a first-class feature with the same priority as functional requirements. Every new service or feature must include a monitoring plan in the design review. The plan must specify: what metrics will be collected, what alerts will fire, and what dashboards will be created.

### Anti-Pattern 5: No Monitoring for the Monitoring System

The monitoring system has no health checks, no redundancy, and no alerting. When it fails, nobody knows until a user reports an outage that should have been caught by monitoring.

**Why it fails**: The monitoring system is a critical dependency. If it goes down, the team is blind. Without health checks on the monitoring pipeline itself, failures go undetected until they cause a customer-impacting incident.

**Fix**: Run a secondary, independent monitoring system for the primary monitoring system. This can be a simple SaaS-based uptime monitor that runs a synthetic check every minute and pages the team if the monitoring dashboard does not respond. This is the "monitor of monitors" pattern.

## When NOT to Use

Monitoring is not always the answer. There are situations where a dedicated monitoring system wastes effort.

**You have a single server running a single application**: A single server does not need a distributed monitoring pipeline. Install a simple process manager (systemd), enable logging to a file, and check the logs when something breaks. The complexity of Prometheus, Grafana, and alerting is not justified.

**You are prototyping**: During the prototyping phase, your application code changes every hour. Instrumentation added today will be outdated tomorrow. Focus on building the core functionality and add monitoring as a separate task before the first production deployment.

**Your SLA allows for reactive response**: If your contract allows 4-hour response time for outages, you do not need real-time monitoring. A daily cron job that checks log files for errors and sends an email report is sufficient. Monitoring tools are designed for sub-minute detection and response.

**The monitoring system costs more than the system it monitors**: If your total cloud bill is $500 per month and a full monitoring stack would cost $300 per month, the monitoring is too expensive. Use a free tier of a SaaS monitoring provider or a lightweight, single-node Prometheus setup.

## Cost Analysis Framework

Understanding the total cost of a monitoring system requires analyzing six cost dimensions. Use this framework to estimate the cost of any monitoring architecture before building it.

### Cost Dimension 1: Compute

Every component in the monitoring pipeline consumes CPU and memory. The cost scales with the number of instances and the throughput per instance.

```text
Compute cost estimation:
  Aggregator instances: 4 x $200/month = $800/month
  TSDB partitions: 8 x $500/month = $4,000/month
  Query frontends: 2 x $300/month = $600/month
  Alert manager: 2 x $200/month = $400/month
  Grafana: 2 x $300/month = $600/month
  Total compute: $6,400/month
```

### Cost Dimension 2: Storage

Storage is typically the largest cost in a monitoring system, especially for logs and traces. The cost depends on retention period, replication factor, and storage tier.

```text
Storage cost estimation (10M DAU, 30-day retention):
  Metrics (full resolution, 7 days): 280 GB x $0.08/GB = $22/month
  Metrics (downsampled, 23 days): 92 GB x $0.08/GB = $7/month
  Logs (30 days, with indexing): 1,848 GB x $0.08/GB = $148/month
  Traces (30 days): 414 GB x $0.08/GB = $33/month
  Total storage: $210/month
  With replication (2x): $420/month
```

Logs dominate the storage cost, accounting for 70 percent of the total. This is why aggressive log sampling -- keeping only ERROR and WARN logs at full resolution, sampling DEBUG and INFO at 1 percent -- is the single most effective cost optimization.

### Cost Dimension 3: Network

Data transfer between agents, aggregators, and storage backends incurs network costs, especially in multi-region deployments. Cross-region data transfer is typically 5 to 10 times more expensive than within-region transfer.

```text
Network cost estimation:
  Within-region agent to aggregator: 40 GB/day x $0.01/GB = $0.40/day
  Cross-region replication: 40 GB/day x $0.09/GB = $3.60/day
  Monthly network: $40 + $108 = $148/month
```

### Cost Dimension 4: SaaS Licensing

If using a SaaS monitoring provider, the licensing cost is typically based on data volume. Datadog charges approximately $5 per host per month for infrastructure monitoring plus $1.50 per million log events per month.

```text
SaaS cost estimation:
  Infrastructure monitoring (2,000 hosts): $10,000/month
  Log management (40 GB/day): $6,000/month
  APM and tracing: $8,000/month
  Total SaaS: $24,000/month
```

### Cost Dimension 5: Operational Overhead

The operational cost of running a monitoring system includes the time spent by the monitoring team on configuration, maintenance, and upgrades. Estimate this at 1 full-time engineer per 500 services for an in-house system and 0.25 full-time engineers per 500 services for a SaaS system.

```text
Operational overhead:
  In-house: 2 FTEs x $15,000/month = $30,000/month
  SaaS: 0.5 FTEs x $15,000/month = $7,500/month
```

### Cost Dimension 6: Build vs Buy Decision

The total cost comparison between in-house and SaaS monitoring depends on scale. For small deployments under 500 instances, SaaS is always cheaper. For large deployments over 10,000 instances, in-house can be cheaper if the team has the expertise.

```text
Total cost comparison (10M DAU, 2,000 instances):
  In-house:
    Compute: $6,400/month
    Storage: $420/month
    Network: $148/month
    Operations: $30,000/month
    Total: $36,968/month

  SaaS:
    Infrastructure + APM + Logs: $24,000/month
    Operations: $7,500/month
    Total: $31,500/month

  At this scale, SaaS is 15 percent cheaper.
  At 10,000 instances, in-house becomes cheaper:
    SaaS would be: 10,000 x $9 per host = $90,000/month
    In-house: $12,800 + $1,200 + $500 + $30,000 = $44,500/month
```

## SLO-Based Alerting Tradeoffs

SLO-based alerting replaces static thresholds with dynamic thresholds based on error budget consumption. It reduces alert fatigue significantly, but it introduces its own tradeoffs.

### Multi-Burn-Rate Alerting

Multi-burn-rate alerting uses three alert levels, each with a different burn rate threshold and evaluation window. The burn rate is the speed at which the error budget is consumed relative to the SLO.

```text
Multi-burn-rate configuration:

  SLO: 99.9 percent latency under 200ms over 30 days
  Error budget: 0.1 percent = 43.2 minutes of violation per month

  Level 1: Critical (10x burn rate)
    - Window: 1 hour
    - Threshold: >10x burn rate sustained for 5 minutes
    - Response: Page on-call engineer immediately
    - Error budget would be exhausted in 3 days at this rate

  Level 2: Warning (4x burn rate)
    - Window: 6 hours
    - Threshold: >4x burn rate sustained for 30 minutes
    - Response: Slack notification, investigate during business hours
    - Error budget would be exhausted in 7.5 days at this rate

  Level 3: Informational (1x burn rate)
    - Window: 3 days
    - Threshold: >1x burn rate sustained for 12 hours
    - Response: Daily email report
    - Error budget would be exhausted exactly at the end of the window
```

The tradeoff with SLO-based alerting is that it requires accurate baseline data. If the SLO is set incorrectly -- too tight or too loose -- the alerts are either too sensitive or not sensitive enough. The SLO must be calibrated against real data before the alerting rules are deployed.

### Alert Reliability vs Alert Timeliness

There is a fundamental tradeoff between alert reliability and alert timeliness. A reliable alert with a low false positive rate requires a longer evaluation window, which delays notification. A timely alert with sub-minute notification requires a short evaluation window, which increases the false positive rate.

```text
Reliability vs Timeliness tradeoff:

  Evaluation window: 1 minute
    False positive rate: 30 percent
    Detection latency: 1 minute
    Use case: Error rate > 5 percent

  Evaluation window: 5 minutes
    False positive rate: 5 percent
    Detection latency: 5 minutes
    Use case: p99 latency > 500ms

  Evaluation window: 15 minutes
    False positive rate: 1 percent
    Detection latency: 15 minutes
    Use case: CPU utilization > 90 percent
```

The correct approach uses different evaluation windows for different alert types. Error-rate alerts use short windows because errors are immediately actionable. Latency alerts use medium windows because latency spikes can be transient. Resource utilization alerts use long windows because resource issues develop over minutes.

## Granularity Tradeoffs

The granularity of monitoring data -- how often metrics are collected, how long they are retained, and at what resolution -- is a continuous tradeoff between visibility and cost.

### Collection Interval Tradeoff

A shorter collection interval provides finer-grained visibility but increases storage and compute costs linearly. A 5-second collection interval costs 12x more than a 60-second interval for the same metric.

```text
Collection interval cost comparison:
  60-second interval: 1,440 data points per day per metric
  15-second interval: 5,760 data points per day per metric
  5-second interval: 17,280 data points per day per metric

  Cost scaling:
    60s ─▶ 15s: 4x cost for 4x finer granularity
    15s ─▶ 5s: 3x cost for 3x finer granularity
    Total: 12x cost for 12x finer granularity
```

The tradeoff is that most monitoring use cases do not need 5-second granularity. Alerting rules evaluate every 60 seconds. Dashboards auto-refresh every 30 seconds. The only use case that needs sub-10-second granularity is real-time anomaly detection for latency-critical systems. For all other use cases, 15-second or 60-second collection intervals provide adequate visibility at 1/12th the cost.

### Retention Period Tradeoff

Longer retention provides better trend analysis and capacity planning but increases storage costs linearly. The marginal value of retaining data decreases with age.

```text
Retention value degradation:
  Last 7 days: Critical for debugging and alerting
  Days 8-30: Important for trend analysis and SLO tracking
  Days 31-90: Useful for capacity planning
  Days 91-365: Occasionally useful for annual comparisons
  Beyond 365 years: Rarely queried, primarily for compliance
```

The standard retention strategy uses three tiers. Tier 1 (full resolution, 7 days) provides the data needed for debugging. Tier 2 (1-minute downsampled, 90 days) provides the data needed for trend analysis and capacity planning. Tier 3 (1-hour downsampled, 1 year) provides the data needed for annual comparisons and compliance.

```text
Three-tier retention costs (relative to full resolution for 365 days):
  Full resolution for 365 days: 100 percent cost
  Three-tier strategy:
    Tier 1: 7 days at full resolution = 7/365 = 2 percent of full cost
    Tier 2: 83 days at 1-minute resolution = 83/365 x 1/15 = 1.5 percent of full cost
    Tier 3: 275 days at 1-hour resolution = 275/365 x 1/60 = 1.3 percent of full cost
    Total: 4.8 percent of full cost
```

The three-tier strategy reduces storage costs by 95 percent while preserving the most valuable data at full resolution for the most recent period.

### Sampling Rate Tradeoff

Sampling trades data completeness for cost reduction. The tradeoff is nonlinear: reducing the sampling rate by 50 percent reduces costs by 50 percent but may reduce debugging effectiveness by much more than 50 percent.

```text
Sampling rate impact on debugging:
  100 percent sampling: Can find any event, debug any issue
  10 percent sampling: Will miss 90 percent of events
    Probability of catching a specific error: 10 percent
    Probability of catching a rare error (0.1% rate): 0.01 percent
  1 percent sampling: Will miss 99 percent of events
    Only useful for aggregate statistics, not debugging
```

The correct sampling strategy uses value-based sampling: keep all events that have high diagnostic value and sample events that have low diagnostic value. Error events, slow requests, and trace events with unique span IDs are high-value. Health check events, debug log lines, and trace events for rapid-success requests are low-value.

```text
Value-based sampling strategy:
  Errors: 100 percent sampling (keep every error)
  p99+ latency: 100 percent sampling (keep every slow request)
  Successful requests: 1 percent sampling (statistics only)
  Debug logs: 0.1 percent sampling (rarely needed)
  Health check logs: 0 percent sampling (not useful for debugging)
```
