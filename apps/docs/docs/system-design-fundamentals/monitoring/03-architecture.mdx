## Naive Implementation

Most monitoring systems start as a single-server setup. The application writes metrics to a local file, a small collector script parses the file and sends data to a central database, and a single Grafana instance renders dashboards. This works until it does not.

### At 1,000 Users: The Single Server

With 1,000 users and a handful of services, a single server running a time-series database and Grafana handles everything. The agent runs on each application server, sends metrics to the central TSDB, and the team views dashboards on a single Grafana instance. This architecture is simple, cheap, and requires no specialized operations knowledge.

```text
1,000 users:
  - 5 application servers
  - 500 metrics/second total
  - 10,000 time series
  - Single TSDB instance: 2 CPU, 8 GB RAM, 100 GB SSD
  - Single Grafana instance
  - Cost: $50/month
  - Reliability: 99 percent (single point of failure)
```

### At 100,000 Users: The First Bottlenecks

At 100,000 users, the single-server architecture starts to creak. The TSDB runs out of memory for the memtable. WAL writes slow down because the disk is saturated with concurrent reads and writes. Queries that touch many time series exceed the 30-second timeout. The team adds more CPU and RAM to the single server, but this only buys time.

```text
100,000 users:
  - 50 application servers
  - 50,000 metrics/second
  - 500,000 time series
  - Single TSDB instance: 16 CPU, 64 GB RAM, 2 TB SSD
  - Problems:
    - Memtable fills every 30 seconds, constant flushing
    - Compaction can never catch up: 50+ segment files
    - Query latency: 5-30 seconds for any non-trivial dashboard
    - Disk IO saturation: 500 MB/sec read + 200 MB/sec write
    - Single point of failure: TSDB crash = 10-minute recovery
```

At 10 million users, the naive architecture hits a wall. The single TSDB server cannot handle 5 million metrics per second. The memtable overflows in seconds. The WAL saturates the disk IO bandwidth. Compaction falls behind, and the number of segment files grows without bound. Query latency degrades to tens of seconds, and dashboards time out. Alerts stop firing because the alert manager cannot read current data.

The lesson is clear: monitoring is a data pipeline, and like any data pipeline, it must be partitioned, replicated, and scaled horizontally.

The production architecture has four layers. The agent layer runs on every node and handles local buffering and batching. The aggregator layer is stateless and horizontally scalable, receiving data from all agents through a load balancer. The storage layer is partitioned into 8 shards, each with a primary and a replica TSDB instance. The query layer fans out queries to all partitions and merges results.

### Horizontal Scaling Rules

The architecture scales each dimension independently:

- **Aggregators**: Scale horizontally by adding instances behind a load balancer. Each aggregator handles a share of the agents. The total ingest capacity scales linearly with the number of aggregator instances.
- **TSDB partitions**: Increase the number of partitions to reduce the write load per instance. Each partition handles 1/Nth of the total metric volume. The partition count is set at deployment time and is difficult to change later.
- **Query capacity**: Scale query frontends horizontally. Each query frontend fans out to all partitions, merges results, and caches them. Query throughput scales with the number of frontends, but query latency is bounded by the slowest partition.
- **Alert evaluation**: Run alert managers in an active-passive pair. The active instance evaluates all rules. The passive instance takes over if the active fails. Alert evaluation is CPU-bound, and a single instance can handle up to 10,000 rules.

Database exporters connect to the database and query system tables -- `pg_stat_activity` for connection counts, `pg_stat_replication` for replication lag, `pg_locks` for lock contention. The exporter exposes these metrics on a `/metrics` endpoint that the monitoring agent scrapes. The key metrics to collect from your database are: active connections as a percentage of max connections, replication lag in bytes and seconds, query latency percentiles from `pg_stat_statements`, and deadlock rate per minute.

### Pattern 2: Cache Integration

Caches like Redis and Memcached provide critical performance data through their built-in metrics interfaces. Cache hit rate, eviction rate, and memory usage directly impact application performance.

```text
Cache monitoring integration:

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ [App]    │────▶│   {Cache}    │────▶│ [Exporter]   │
  │ Service  │     │  Redis/Mem   │     │ redis_exporter
  └──────────┘     └──────────────┘     └──────┬───────┘
                                               │
                                               ▼
                                        ┌──────────────┐
                                        │ [Monitoring] │
                                        │   Pipeline   │
                                        └──────────────┘
```

The critical cache metrics are: hit rate (target above 95 percent), eviction rate (should be zero), memory usage as a percentage of maxmemory, and command latency at p99. A rising eviction rate combined with falling hit rate indicates the cache is too small for the workload.

### Pattern 3: Queue Integration

Message queues and stream processors are common sources of monitoring blind spots. The queue itself is a buffer -- it hides problems by absorbing traffic spikes, but when the buffer fills, the failure is sudden and catastrophic.

```text
Queue monitoring integration:

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ [Producer]│────▶│  <Queue>    │────▶│ [Consumer]   │
  │ Service  │     │ Kafka/Rabbit │     │ Worker Pool  │
  └──────────┘     └──────────────┘     └──────┬───────┘
                       │                       │
                       ▼                       ▼
                 ┌──────────────┐     ┌──────────────┐
                 │ [Exporter]   │     │ [Exporter]   │
                 │ broker       │     │ consumer     │
                 └──────┬───────┘     └──────┬───────┘
                        │                    │
                        └────────┬───────────┘
                                 ▼
                          ┌──────────────┐
                          │ [Monitoring] │
                          │   Pipeline   │
                          └──────────────┘
```

The essential queue metrics are: producer lag (rate of messages produced minus rate of messages consumed), consumer lag (the number of messages not yet processed), queue depth, and consumer group rebalance frequency. Queue depth alone is misleading -- a queue with 100,000 messages could be fine if consumed in 10 seconds, or catastrophic if the consumers are down.

### Pattern 4: CDN and Load Balancer Integration

CDN and load balancer metrics are the closest you get to the user experience without synthetic monitoring. They see every request, including those that never reach your application servers.

```text
CDN / LB monitoring integration:

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐
  │ [User]   │────▶│   [CDN]     │────▶│   [LB]       │
  │ Browser  │     │  CloudFront  │     │  ALB/NGINX   │
  └──────────┘     └──────┬───────┘     └──────┬───────┘
                          │                    │
                          ▼                    ▼
                    ┌──────────────┐     ┌──────────────┐
                    │ [CDN Logs]   │     │ [LB Metrics] │
                    │ S3 → Athena  │     │ request count│
                    └──────────────┘     │ error rate   │
                                         │ byte volume  │
                                         └──────────────┘
```

CDN metrics reveal regional issues that internal monitoring misses. If 99 percent of requests from Europe succeed but only 50 percent from Asia succeed, the problem is in the CDN edge configuration, not your application. Load balancer metrics show the request rate, error rate, and byte volume at the entry point to your infrastructure. A rising 5xx error rate at the load balancer paired with a flat 5xx rate at your application servers points to a load balancer configuration issue, not an application bug.

Each region runs its own complete monitoring pipeline: agents, aggregators, TSDB, alert manager, and dashboards. This ensures that a regional outage does not affect alerting for other regions. A global query layer reads from all regions and provides a unified view. The global alert manager evaluates cross-region rules -- for example, triggering a global incident if two out of three regions show elevated error rates simultaneously.

The cross-region replication uses asynchronous streaming. Data flows from each region's TSDB replica to the global store over a persistent connection. The replication lag target is 60 seconds -- meaning data from one region is available in the global view within 60 seconds of ingestion. This lag is acceptable for SLO monitoring and capacity planning but too slow for real-time alerting. Regional alerts use local data with sub-second latency.

## Capacity Planning

Capacity planning for monitoring starts with a single question: how many data points per second must your pipeline ingest?

### 10 Million DAU Example

```text
Capacity planning calculation for 10M DAU:

Assumptions:
  - 500 microservices
  - 2,000 application instances
  - 100 metrics per instance (CPU, memory, request rate, error rate, latency p50/p95/p99, etc.)
  - 5 log lines per request
  - 1 trace span per service hop per request
  - 1 request per user per 60 seconds on average
  - 20 percent peak-to-average ratio

Metrics:
  - Total metrics per instance: 100
  - Scrape interval: 15 seconds
  - Metrics per second: 2,000 instances x 100 metrics / 15s = 13,333 data points/sec
  - Peak: 13,333 x 1.2 = 16,000 data points/sec

Logs:
  - Requests per second: 10,000,000 / 86,400 = 116 requests/sec
  - Peak: 116 x 1.2 = 139 requests/sec
  - Log lines per second: 139 x 5 = 695 log lines/sec
  - Average log line size: 512 bytes
  - Log data per second: 695 x 512 = 356 KB/sec
  - Log data per day: 356 KB/sec x 86,400 = 30.8 GB/day

Traces:
  - Traces per second: 139 (one per request)
  - Average service hops: 3
  - Spans per second: 139 x 3 = 417 spans/sec
  - Average span size: 256 bytes
  - Trace data per second: 417 x 256 = 107 KB/sec
  - Trace data per day: 107 KB/sec x 86,400 = 9.2 GB/day

Total ingest:
  - Metrics: 16,000 data points/sec (negligible storage per point)
  - Logs: 356 KB/sec, 30.8 GB/day
  - Traces: 107 KB/sec, 9.2 GB/day
  - Total: 40 GB/day raw, ~200 GB/day with indexing overhead

Storage sizing (30-day retention):
  - Metrics (full resolution, 7 days): 40 GB x 7 = 280 GB
  - Metrics (downsampled, 23 days): 40 GB x 0.1 x 23 = 92 GB
  - Logs (30 days): 30.8 GB x 30 = 924 GB
  - Traces (30 days): 9.2 GB x 30 = 276 GB
  - Indexing overhead: 2x on logs, 1.5x on traces
  - Total: 280 + 92 + 924 x 2 + 276 x 1.5 = 280 + 92 + 1,848 + 414 = 2,634 GB
  - With replication (2x): 5,268 GB or approximately 5.3 TB

Compute sizing:
  - Aggregators: 16,000 data points/sec, each aggregator handles 5,000/sec
    - Need 4 aggregator instances
  - TSDB partitions: target 2,000 data points/sec per partition
    - Need 8 partitions
  - Query frontends: 50 concurrent users, each query reads 10 partitions
    - Need 2 query frontends
  - Alert manager: 500 rules, evaluated every 60 seconds
    - Need 1 alert manager (with 1 standby)
```

The numbers show that monitoring is storage-bound, not compute-bound. The compute requirements are modest -- 4 aggregator instances and 8 TSDB partitions fit on medium-sized VMs. The storage requirement of 5.3 TB for 30 days of retention fits on a handful of provisioned IOPS SSDs.

## Deployment and Operations

### Rolling Deployment

Deploying a monitoring system requires careful coordination to avoid losing data during the deployment itself. Use a rolling deployment strategy where each component is updated in sequence.

```text
Deployment order:
  1. Deploy TSDB replicas (new version)
  2. Wait for replicas to catch up
  3. Promote replicas to primary
  4. Deploy TSDB primaries (old version, now replicas)
  5. Deploy aggregators (blue-green, drain old connections)
  6. Deploy agents (rolling, reboot one node at a time)
  7. Deploy query frontends
  8. Deploy alert manager (standby first, then active)
```

The critical rule is: never deploy agents and aggregators simultaneously. If agents send data in a format the new aggregators do not understand, or vice versa, the pipeline breaks. Deploy aggregators first, verify they are healthy, then deploy agents.

### Operational Runbooks

Every monitoring system needs a runbook for its own health. The most common operational issues are:

- **Agent crash**: The agent process stops on a node. The application continues working, but metrics stop flowing. Detection: the agent's health check endpoint returns 503. Action: restart the agent process. Prevention: run the agent as a systemd service with `Restart=always`.

- **TSDB disk full**: The TSDB partition runs out of disk space. The database stops accepting writes. Detection: disk usage alert at 80 percent, 90 percent, and 95 percent. Action: increase retention period or add more disk. Prevention: set up automated retention enforcement and monitor disk usage trends weekly.

- **Compaction storm**: All TSDB partitions start compaction simultaneously, saturating disk IO. Query latency spikes. Detection: disk IO utilization above 90 percent with high compaction queue depth. Action: reduce compaction concurrency or stagger compaction schedules across partitions. Prevention: randomize compaction start times within a window.

- **High cardinality explosion**: A deployed code change adds a user ID or session ID as a metric label. The number of time series jumps from 100,000 to 10 million. The TSDB memory usage spikes and the database starts rejecting writes. Detection: series count alert. Action: roll back the deployment and add cardinality limits to the aggregator. Prevention: set a hard limit on the number of time series per metric in the aggregator configuration.

## Observability-Driven Architecture

Observability-driven architecture (ODA) extends the concept of monitoring to include the system's ability to be explored and debugged without writing new code. A monitoring architecture designed for observability has three additional properties beyond the basic pipeline.

### High-Cardinality Event Store

Traditional TSDBs struggle with high cardinality because they index every unique label combination. An observability-oriented system uses a separate event store that handles high-cardinality data natively. This store ingests raw events -- each with dozens of unique attributes -- and provides a query interface that can filter, group, and aggregate across any combination of attributes without pre-defining indexes.

For example, instead of storing a metric `http_request_duration_ms{user_id="abc123", endpoint="/checkout", country="US"}` in the TSDB (which creates one series per user), you store the raw event in the event store with all attributes preserved. The TSDB stores the low-cardinality aggregate `http_request_duration_ms{endpoint="/checkout"}` for real-time dashboards. The event store stores the high-cardinality raw data for debugging specific user issues.

Honeycomb is the most prominent example of a high-cardinality event store. It uses a columnar storage format with aggressive compression -- each column is stored separately and compressed using dictionary encoding for low-cardinality columns and run-length encoding for high-cardinality columns. This design enables queries over any combination of dimensions without pre-computed indexes.

### Structured Event Taxonomy

Every event in the monitoring pipeline should follow a standard taxonomy. The OpenTelemetry semantic conventions provide a starting point, but most organizations need to extend them with business-specific fields.

```yaml
Standard event fields:
  # Required fields
  event_id:          uuid-string
  timestamp:         ISO 8601 datetime
  service_name:      string (e.g., "checkout-service")
  service_version:   semver (e.g., "1.2.3")
  environment:       string (e.g., "production", "staging")

  # Observability fields
  trace_id:          uuid-string
  span_id:           uuid-string
  parent_span_id:    uuid-string (empty for root spans)

  # Business fields
  user_id:           string (for user-specific debugging)
  customer_tier:     string (free, pro, enterprise)
  region:            string (us-east-1, eu-west-1, ap-southeast-1)
  deployment_id:     string (links to CI/CD pipeline)

  # Performance fields
  duration_ms:       float
  cpu_time_ms:       float
  memory_mb:         float
  error:             boolean
  error_message:     string (empty if no error)
```

The taxonomy ensures that every event can be correlated across services. A trace ID in a log line lets you find all log lines for a specific request. A deployment ID in a metric lets you correlate a latency regression with a specific deployment.

### Debugging-First Instrumentation

The final property of an observability-driven architecture is that instrumentation is designed for debugging, not just dashboards. This means every service emits events that answer questions like: "Show me all requests from user X that took longer than 1 second," "Show me all errors in the payment service in the last hour grouped by error code," and "Show me the p99 latency trend for the checkout service broken down by A/B test cohort."

Debugging-first instrumentation requires more event attributes than dashboard-first instrumentation. Every event includes the user ID, the request path, the error code, the deployment version, and the A/B test cohort. These fields are expensive to include in a TSDB metric but cheap to include in an event store record.

## Real-Time Monitoring Architecture

Some monitoring use cases require sub-second data visibility. Real-time monitoring -- used for fraud detection, trading platforms, and live streaming quality monitoring -- requires a different architecture from the standard 10-second flush pipeline.

### In-Process Monitoring

For sub-millisecond visibility, the monitoring agent must run in-process with the application instead of as a separate sidecar. The in-process agent writes metrics directly to a shared memory region that a dedicated monitoring thread reads every 100 milliseconds.

```text
In-process monitoring architecture:

  ┌─────────────────────────────────────┐
  │ Application Process                 │
  │                                     │
  │ ┌──────────────┐   ┌──────────────┐│
  │ │ Request      │   │ Monitoring   ││
  │ │ Handler      │   │ Thread       ││
  │ │ Thread Pool  │   │ (reads every ││
  │ │              │   │  100ms)      ││
  │ └──────┬───────┘   └──────┬───────┘│
  │        │                  │        │
  │        │   Shared Memory  │        │
  │        └────────┬─────────┘        │
  │                 ▼                  │
  │        ┌──────────────────┐        │
  │        │ Metrics Ring     │        │
  │        │ Buffer           │        │
  │        │ (lock-free,      │        │
  │        │  fixed size)     │        │
  │        └──────────────────┘        │
  │                 │                  │
  │                 ▼                  │
  │        ┌──────────────────┐        │
  │        │ Real-Time        │        │
  │        │ Notifier         │        │
  │        │ (WebSocket push) │        │
  │        └──────────────────┘        │
  └─────────────────────────────────────┘
```

The in-process agent uses a lock-free ring buffer to avoid contention with the request handler threads. Each request handler thread writes to its own cache line (to avoid false sharing) and the monitoring thread reads from all cache lines atomically. This design adds under 100 nanoseconds of overhead per metric write.

### Stream Processing Pipeline

For real-time monitoring, the batch-based aggregator is replaced with a stream processor. Apache Kafka combined with Kafka Streams or Apache Flink processes metrics with sub-second latency.

```text
Real-time stream processing:

  Agents ──▶ Kafka (topic per service, 10 partitions)
               │
               ▼
          ┌────────────┐
          │ Flink Job  │
          │ Window: 10s│
          │ Sliding    │
          └──────┬─────┘
                 │
        ┌────────┼────────┐
        ▼        ▼        ▼
   ┌────────┐┌────────┐┌────────┐
   │ Alert  ││ TSDB   ││ Cache  │
   │ Engine ││ Write  ││ Update │
   └────────┘└────────┘└────────┘
```

The Flink job maintains a sliding window of the last 10 seconds. Every 100 milliseconds, it computes the current request rate, error rate, and latency percentiles. If any metric exceeds a threshold, the alert engine fires within 100 milliseconds. The cache update pushes the new metric values to all connected dashboards through a WebSocket, enabling real-time dashboard updates.

### Real-Time Alert Evaluation

Real-time alerting uses a different evaluation model from the standard Prometheus Alertmanager. Instead of evaluating rules against stored metrics every 60 seconds, the real-time alert engine evaluates rules against the stream processor's output every 100 milliseconds.

```java
// Pseudocode for real-time alert evaluation
public class RealTimeAlertEngine {
    private final Map<String, AlertRule> rules = new HashMap<>();

    public void onMetricUpdate(String metricName, double value, Map<String, String> labels) {
        for (AlertRule rule : rules.values()) {
            if (rule.matches(metricName, labels)) {
                rule.recordValue(value);
                if (rule.isBreaching()) {
                    if (!rule.isFiring()) {
                        rule.startFiring();
                        alertManager.fire(rule.getAlert(labels, value));
                    }
                } else {
                    if (rule.isFiring()) {
                        rule.stopFiring();
                        alertManager.resolve(rule.getAlert(labels, value));
                    }
                }
            }
        }
    }
}
```

The real-time alert engine maintains in-memory state for each rule. It records the last N values and evaluates the breach condition on every update. When the condition transitions from not-breaching to breaching, the alert fires immediately. When the condition transitions back, the alert resolves immediately. There is no `for` duration in real-time alerting -- the latency requirement of sub-second detection means that the checking window is built into the stream window instead.

## Securing the Monitoring Pipeline

The monitoring pipeline itself is a security-sensitive component. It contains detailed information about system behavior, error messages, and potentially sensitive data in log lines.

### Authentication and Authorization

Every component in the monitoring pipeline must authenticate its peers. Agents authenticate to aggregators using mTLS with client certificates. Aggregators authenticate to TSDB instances using short-lived tokens from a secrets manager. Dashboards authenticate users through an OAuth2 proxy.

```yaml
Monitoring security configuration:

  Agent to Aggregator:
    - mTLS with rotating client certificates
    - Certificate issued by internal CA
    - Certificate rotation every 24 hours
    - Revocation list checked every 5 minutes

  Aggregator to TSDB:
    - Service account tokens (30-minute TTL)
    - Tokens issued by Vault or AWS Secrets Manager
    - Token embedded in gRPC metadata headers

  Dashboard to Query Layer:
    - OAuth2 proxy (OAuth2 Proxy or Ory Oathkeeper)
    - Backed by Okta, Google Workspace, or internal IdP
    - Role-based access: admin, editor, viewer
    - API keys for programmatic access
```

### Data Redaction

Log lines can contain sensitive data: PII, credit card numbers, API keys, and passwords. The monitoring pipeline must redact sensitive data before it reaches the storage layer.

```python
# Log redaction example
import re

SENSITIVE_PATTERNS = [
    (r'"credit_card":"\d{16}"', '"credit_card":"[REDACTED]"'),
    (r'"ssn":"\d{3}-\d{2}-\d{4}"', '"ssn":"[REDACTED]"'),
    (r'"password":"[^"]+"', '"password":"[REDACTED]"'),
    (r'"api_key":"[^"]+"', '"api_key":"[REDACTED]"'),
    (r'\b\d{4}[- ]?\d{4}[- ]?\d{4}[- ]?\d{4}\b', '[CREDIT_CARD_REDACTED]'),
]

def redact_log_line(line: str) -> str:
    for pattern, replacement in SENSITIVE_PATTERNS:
        line = re.sub(pattern, replacement, line)
    return line
```

Redaction can happen at two points. Client-side redaction happens in the application before the log line is emitted. This is the most secure approach because sensitive data never leaves the application process. Server-side redaction happens in the aggregator as a fallback for applications that do not implement redaction.

### Audit Logging

The monitoring system must log its own access patterns. Every query to the monitoring system is logged with the user identity, the query text, the timestamp, and the result size.

```yaml
Audit log entry:
  timestamp: 2026-07-02T14:30:00Z
  user_id: alice@company.com
  action: dashboard_query
  resource: service_latency_overview
  query: |
    histogram_quantile(0.99, rate(request_duration_seconds_bucket[5m]))
  result_size: 1200
  duration_ms: 450
  source_ip: 10.0.12.45
```

Audit logs are stored in a separate, append-only store that the monitoring team cannot delete. This ensures compliance with SOC2, HIPAA, and other regulatory requirements.

### Testing Monitoring Configurations

Monitoring configurations must be tested just like application code. The CI/CD pipeline should validate that dashboard queries return valid results, that alert rules are syntactically correct, and that dashboard load times are under 3 seconds.

```yaml
# CI/CD pipeline for monitoring configs
stages:
  - lint:
      - check-yaml-syntax *.yaml
      - validate-promql dashboard-queries/*.yaml
      - validate-alert-rules alerts/*.yaml
  - test:
      - run: query-test --dashboard checkout-service-dashboard.yaml
        assert: response_time < 3s
      - run: alert-test --rule checkout-high-latency.yaml
        assert: evaluation_time < 500ms
  - deploy:
      - apply-dashboard checkout-service-dashboard.yaml
      - apply-alert-rule checkout-high-latency.yaml
```

### Self-Service Monitoring Platform

At scale, the monitoring team should provide a self-service platform that lets service teams configure their own monitoring without needing help from the platform team. The self-service platform includes:

- A dashboard template library with pre-built panels for common patterns (RED metrics, USE metrics, database metrics, queue metrics)
- An alert rule template library with pre-configured thresholds and runbook links
- A monitoring playground where teams can test queries before deploying them
- Automated dashboard generation based on service metadata (service name, team, dependencies, criticality)

The self-service platform reduces the monitoring team's workload from 50 tickets per week to 5 tickets per week, freeing them to work on improving the monitoring pipeline itself rather than configuring individual dashboards and alerts.
