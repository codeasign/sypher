A single Kafka broker works for development and low-throughput use cases. Production systems require multiple brokers, partitions, replication, consumer groups, and supporting infrastructure. This page walks through the architectural evolution from a single broker to a multi-region deployment.

## Naive Architecture: Single Broker

The simplest Kafka deployment is a single broker running on one server. This is suitable for development, testing, and non-critical workloads with throughput under 50 MB/s.

```
Single server: 16 CPU cores, 64 GB RAM, 2 TB NVMe
Topics: 10  |  Partitions per topic: 3  |  Replication factor: 1
```

Single-broker architecture has no fault tolerance. If the broker fails, all topics are unavailable. If the disk fails, all data is lost. There is no replication and no load distribution.

**Risks of single-broker deployments:**
- No data redundancy — if the disk fails, all data is lost.
- No load distribution — all producers and consumers compete for the same resources.
- No rolling maintenance — any broker restart causes downtime.
- No horizontal scaling — the single server's capacity is the system's capacity.

## Production Architecture: Multi-Broker Cluster

A production deployment uses multiple brokers, partitions, and replication to provide fault tolerance and scale.

### Cluster Topology

A typical production cluster has the following characteristics:

```
Brokers: 6 (minimum 3 for quorum-based operations)
Partitions per broker: 50-200
Replication factor: 3
Replica assignment: Balanced across brokers and racks
```

### Partition Distribution

Kafka's partition assignment algorithm distributes replicas across brokers to balance load and maximize fault tolerance. The algorithm for a topic with 6 partitions, replication factor 3, and 6 brokers works as follows:

1. For partition 0, select a leader from any broker (round-robin from the first partition).
2. The leader of partition 0 goes on broker 1.
3. The first replica goes on broker 2 (next broker).
4. The second replica goes on broker 3 (next after that).
5. Repeat for each partition, advancing the starting broker.

The result is that each broker is the leader for 1 partition and a follower for 2 partitions. This balances both the leader load (produce requests) and the follower load (replication traffic).

### Leader Rebalancing

Over time, leadership can become unbalanced due to broker failures and replacements. The `auto.leader.rebalance.enable` setting (default true) triggers periodic rebalancing. The preferred replica election algorithm moves leadership back to the first replica in the replica list, which the original assignment algorithm distributed evenly.

## ZooKeeper and KRaft

### ZooKeeper Mode (Legacy)

ZooKeeper was the original metadata store for Kafka. It runs as a separate cluster of 3 or 5 nodes. ZooKeeper stores:

- Broker IDs and connection information.
- Topic configurations (partitions, replication factor, retention settings).
- Partition leadership assignments.
- Consumer group offsets (in a separate Kafka topic, not ZooKeeper, since Kafka 0.9).
- ACLs and quotas.

ZooKeeper operations are write-on-read-many. The ZooKeeper cluster handles low-throughput metadata operations — 100-500 operations per second. ZooKeeper uses the Zab consensus protocol for consistency.

**ZooKeeper failure modes:**
- If the ZooKeeper ensemble loses its majority (2 out of 3 nodes fail), the entire Kafka cluster becomes unavailable. No metadata changes can be made. Existing produce and consume operations continue briefly until metadata changes are required (for example, a leader reconnection).
- ZooKeeper is sensitive to disk latency. A slow ZooKeeper disk can cause the entire cluster to stall while waiting for metadata operations.

### KRaft Mode (Modern)

KRaft (Kafka Raft) replaces ZooKeeper with a Raft-based controller quorum running within the Kafka cluster. Introduced in Kafka 2.8 as experimental, KRaft became production-ready in Kafka 3.5+.

**Key differences from ZooKeeper mode:**

- No separate ZooKeeper cluster — fewer servers to manage and monitor.
- The controller quorum uses Raft consensus to commit metadata changes.
- Metadata API is more efficient — metadata changes are pushed to brokers via a dedicated metadata topic rather than polled.
- Leadership election is faster — KRaft can elect a new controller in under 1 second compared to 3-10 seconds with ZooKeeper.
- Scalability — KRaft supports larger clusters (100+ brokers) more reliably than ZooKeeper.

**KRaft architecture:**

```
Controller quorum: 3 nodes (minimum for fault tolerance)
  - Active controller handles all metadata writes
  - Standby controllers follow the active via Raft replication
  - All controllers can serve metadata reads

Metadata topic: __cluster_metadata
  - Stores cluster state as an append-only log
  - Replicated by Raft across controller nodes
  - Brokers subscribe to metadata changes
```

## Integration Patterns

Kafka integrates with external systems through several patterns.

### Kafka Connect

Kafka Connect is a framework for streaming data between Kafka and other systems. It runs as a separate cluster of worker nodes. Connectors are plugins that implement the source (read from external system, write to Kafka) or sink (read from Kafka, write to external system) interface.

**Source connectors** read from databases (Debezium for CDC), message queues, file systems, and APIs. **Sink connectors** write to databases (JDBC, MongoDB, Elasticsearch), data warehouses (Snowflake, BigQuery), and search indexes.

Connect provides distribution, fault tolerance, and offset tracking out of the box. A source connector tracks its position in the source system so it can resume from the correct point after a restart.

### Kafka Streams

Kafka Streams is a library for building stream processing applications. It runs in your application's JVM — no separate processing cluster is needed. Kafka Streams provides:

- **Stateless operations**: map, filter, flatMapValues.
- **Stateful operations**: aggregate, join, windowed operations.
- **Exactly-once semantics**: reads from Kafka, processes, writes to Kafka as a single transaction.
- **Interactive queries**: query the state store directly via a REST API.

Kafka Streams uses the same consumer group mechanism for parallel processing. Each stream task processes one or more partitions. Tasks can run on different application instances.

### Kafka Streams in Detail

Kafka Streams is a client library for building stream processing applications. It embeds the Kafka consumer and producer APIs within your application, managing the stream processing logic.

**Core abstractions:**

- **KStream**: A record stream where each record is an independent event. Multiple records with the same key are not merged — each is processed individually.
- **KTable**: A changelog stream where each record for a given key replaces the previous record for that key. This is a table-like view of the stream, showing the latest state per key.
- **GlobalKTable**: Like KTable but the entire dataset is available on all instances, not partitioned.

**Stateful operations:**

Kafka Streams supports stateful operations through local state stores. Each stream task maintains a RocksDB instance on disk for state. The state is backed by a compacted Kafka topic for fault tolerance.

- **Aggregations**: `groupByKey().count()`, `groupByKey().reduce()`, `aggregate()`.
- **Joins**: KStream-KStream join (windowed), KStream-KTable join (stream enrichment), KTable-KTable join (table merge).
- **Windowing**: Tumbling windows (fixed-size, non-overlapping), hopping windows (fixed-size, overlapping), sliding windows (defined by gap), session windows (defined by inactivity gap).

**Exactly-once processing in Kafka Streams:**

Kafka Streams uses Kafka's transactional API to achieve exactly-once semantics. When `processing.guarantee=exactly_once_v2` is set:

1. Records are consumed from the source topic within a transaction.
2. State store updates are included in the same transaction.
3. Produced results are written to the sink topic within the same transaction.
4. Consumer offsets are committed within the same transaction.
5. If any step fails, the entire transaction is aborted and replayed.

This provides end-to-end exactly-once processing without external coordination.

### ksqlDB

ksqlDB is a database purpose-built for stream processing on Kafka. It provides a SQL interface for defining streams, tables, and continuous queries. ksqlDB supports:

- **Streams**: Continuous, unbounded sequences of events.
- **Tables**: Mutable views of the latest value for each key.
- **Push queries**: Continuous queries that emit results as events arrive.
- **Pull queries**: Point-in-time queries against materialized state.

ksqlDB is built on Kafka Streams and provides a familiar SQL syntax. A query to compute the average order value per user over 5-minute windows looks like:

```sql
CREATE TABLE avg_order_value AS
  SELECT user_id, AVG(order_amount) AS avg_amount
  FROM orders
  WINDOW TUMBLING (SIZE 5 MINUTES)
  GROUP BY user_id
  EMIT CHANGES;
```

ksqlDB runs in two modes: interactive (REST API for ad-hoc queries) and headless (all queries specified in configuration). Interactive mode is useful for development and exploration. Headless mode is for production deployments where queries are version-controlled.

**When to use Kafka Streams vs. ksqlDB:**

| Factor | Kafka Streams | ksqlDB |
|--------|--------------|--------|
| Language | Java/Scala library | SQL |
| Flexibility | Full programming model | SQL constraints |
| Deployment | Part of app JAR | Separate server cluster |
| State | Local RocksDB | Managed by ksqlDB |
| Monitoring | Application metrics | Server-level metrics |
| Learning curve | Steep (distributed programming) | Shallow (SQL knowledge) |
| Use case | Complex transformations | Simple aggregations, filtering, joins |

### Schema Registry

The Schema Registry is not a Kafka component in the same way as Connect or Streams — it is a companion service that manages schemas for Kafka topics. It runs as a separate REST service and stores schemas in its own storage backend (typically Kafka itself using the `_schemas` topic).

Producers register schemas before writing. Consumers fetch schemas when reading. The Schema Registry validates schema evolution:

- **Backward-compatible**: New schema can read data written with the old schema (default).
- **Forward-compatible**: Old schema can read data written with the new schema.
- **Full compatible**: Both backward and forward compatible.
- **None compatibility**: No compatibility checks.

## Multi-Region Architecture

Distributed systems that span multiple geographic regions require additional architecture to handle cross-region replication.

### MirrorMaker 2

MirrorMaker 2 (MM2) is Kafka's built-in cross-cluster replication tool. It uses Kafka Connect with a special source connector that reads from one cluster and writes to another.

**MM2 features:**
- **Topic replication**: Reproduces topics, partitions, and configurations across clusters.
- **Consumer group offset replication**: Syncs consumer group offsets so failover consumers resume from the correct position.
- **Topic configuration synchronization**: Partition count, replication factor, and other settings are replicated.
- **Active-passive replication**: One cluster is the primary, others are replicas.

**MM2 limitations:**
- **Replication lag**: Cross-region replication has inherent latency (50-200 ms for US-East to US-West).
- **No global ordering**: Records on the target cluster do not maintain the same ordering as the source.
- **No exactly-once**: MM2 provides at-least-once delivery. Duplicates are possible during failover.

### Confluent Cluster Linking

Confluent's Cluster Linking is an alternative to MirrorMaker 2. It provides:

- **Synchronous replication**: Configurable to provide nearly synchronous replication between clusters.
- **Richer semantics**: Supports read-from-follower and client-controlled failover.
- **Lower overhead**: Runs in the broker rather than in a separate Connect cluster.

Cluster Linking is a commercial Confluent feature. MirrorMaker 2 is open source and works with any Kafka distribution.

### Confluent Multi-Region Clusters

Confluent's Multi-Region Clusters extend Kafka with synchronous replication across data centers. A partition leader in US-East replicates synchronously to followers in US-West and EU-West. This provides:

- **Automatic failover**: Clients connect to any region and are routed to the leader.
- **No data loss**: Synchronous replication means every acknowledged record is in at least two regions.
- **Trade-off**: Write latency equals the round-trip time to the farthest region (60-200 ms).

## Capacity Planning

Capacity planning for Kafka involves estimating throughput, storage, and cluster size.

### Throughput Estimation

Calculate the required throughput:

```
Write throughput (MB/s) = Average message size (bytes) x Messages per second / 1,048,576
Read throughput (MB/s) = Write throughput x Number of consumer groups
Total throughput (MB/s) = Write throughput + Read throughput
```

**Example:** 100,000 messages/second with 2 KB average message size and 3 consumer groups:

```
Write: 2000 x 100000 / 1048576 = 191 MB/s
Read: 191 x 3 = 573 MB/s
Total: 191 + 573 = 764 MB/s
```

Each broker handles a share of the throughput based on the number of partition leaders it hosts.

### Broker Sizing

Factor in replication:

```
Net write per broker (MB/s) = Write throughput x Replication factor / Number of brokers
```

With 6 brokers, 191 MB/s write, and RF=3:

```
Net write per broker: 191 x 3 / 6 = 95.5 MB/s
Net read per broker: 573 / 6 = 95.5 MB/s (assuming even read distribution)
Total I/O per broker: 191 MB/s
```

A modern server with 25 Gbps network (3,125 MB/s) and NVMe storage (3,000 MB/s sequential write) can handle this load comfortably. CPU usage at 191 MB/s with compression enabled is approximately 2-4 cores per broker.

### Storage Estimation

```
Daily storage = Write throughput x 86,400 seconds x Replication factor / Compression ratio
Retention storage = Daily storage x Retention days
```

Continuing the example with snappy compression (2:1 ratio) and 7-day retention:

```
Daily: 191 MB/s x 86400 x 3 / 2 = 24.7 TB/day
Retention: 24.7 TB x 7 = 173 TB
```

With 6 brokers, each broker needs 173 TB / 6 = 29 TB of storage. This is achievable with 4 x 8 TB NVMe drives per broker.

### End-to-End Sizing Example

Design a Kafka cluster for the following workload:

```
Topics: 5
Total produce throughput: 1,200 MB/s (peak)
Average message size: 2 KB
Consumer groups per topic: 3 (average)
Replication factor: 3
Retention: 7 days
Compression: snappy (2:1 ratio)
```

**Step 1: Total ingest throughput**

```
Write throughput: 1,200 MB/s
Read throughput: 1,200 MB/s x 3 consumer groups = 3,600 MB/s
Total cluster throughput: 1,200 + 3,600 = 4,800 MB/s
```

**Step 2: Broker count**

Each broker can handle approximately 500 MB/s of mixed produce+consume traffic (assuming 25 Gbps networking and NVMe storage). The number of brokers is:

```
Minimum brokers: 4,800 / 500 = 10 (rounded up)
But with RF=3, each byte is stored 3 times, so replication traffic adds overhead:
Effective write throughput per broker: 500 MB/s / 3 = 167 MB/s
Brokers needed for writes: 1,200 / 167 = 8
Brokers needed for reads: 3,600 / 500 = 8
So 10 brokers is sufficient for both read and write.
```

**Step 3: Partition count per topic**

Assume each topic has 200 MB/s average throughput with 30 MB/s target per partition:

```
Partitions per topic: 200 / 30 = 7, rounded up to 8
Total partitions: 8 partitions x 5 topics x 3 replicas = 120 replica assignments
Partitions per broker: 120 / 10 = 12 (easily fits within limits)
```

**Step 4: Storage per broker**

```
Daily data: 1,200 MB/s x 86,400 seconds = 103,680 GB/day = 101 TB/day
With RF=3: 101 x 3 = 303 TB/day raw
With snappy (2:1): 303 / 2 = 152 TB/day
7-day retention: 152 x 7 = 1,064 TB total cluster storage
Storage per broker: 1,064 / 10 = 107 TB per broker
```

This is large but achievable with 14 x 8 TB NVMe drives per broker in RAID 0.

**Step 5: Network capacity**

```
Each broker: 4,800 / 10 = 480 MB/s (3.84 Gbps)
With 25 Gbps networking, this is 15% utilization — comfortable headroom.
With 10 Gbps networking, this is 38% utilization — acceptable but plan for growth.
```

**Step 6: Consumer group sizing**

```
Total consumers: 3 groups x 8 partitions = 24 consumers
Consumers per broker: 24 / 10 = 2.4 — well within limits.
```

### Partition Count Sizing

Partition count affects both throughput and operational complexity:

- Throughput scales with partition count (each partition can handle approximately 10-50 MB/s).
- Each partition consumes memory and file descriptors on the leader broker.
- Consumer group parallelism is limited by partition count.
- Rebalance time increases with partition count.

General guidelines:
- Minimum: 10-20 partitions per broker for small clusters.
- Maximum: 2,000-4,000 partitions per broker (Kafka's default is 4,000).
- Total partitions in a cluster: 20,000-200,000 depending on Kafka version and hardware.

To calculate the number of partitions:

```
Partitions = Max(Throughput_per_topic / Target_throughput_per_partition,
                Max_consumers_in_group)
```

With a topic handling 500 MB/s, target throughput of 20 MB/s per partition, and 50 consumers:

```
Partitions = Max(500 / 20, 50) = Max(25, 50) = 50 partitions
```

### Network Topology and Latency Budgeting

Kafka's performance depends on network topology. Consider these factors when designing the cluster:

**Same-rack placement:** Place multiple brokers within the same rack for high throughput (25-100 Gbps intra-rack) and low latency (under 0.5 ms). The trade-off is reduced fault tolerance — a rack failure takes out multiple brokers.

**Cross-rack placement:** Distribute brokers across racks for fault tolerance. Each rack has independent power, cooling, and network switches. Cross-rack replication adds 0.5-2 ms of latency per hop. With RF=3 across 3 racks, every produce request must wait for followers in other racks to acknowledge.

**Latency budgeting for acks=all:**

```
Same rack:     leader 0.1ms + follower 0.1ms + follower 0.1ms = 0.3ms replication delay
Cross rack:    leader 0.1ms + follower 0.5ms + follower 0.5ms = 1.1ms replication delay
Cross region:  leader 0.1ms + follower 60ms + follower 60ms = 120.1ms replication delay
```

Cross-region produce with acks=all adds 120+ ms of latency, which is unacceptable for most applications. Use async replication (acks=1) for cross-region produce, or use MirrorMaker 2 for cross-region replication without affecting produce latency.

### Operating System Tuning

Linux OS tuning is critical for Kafka performance. These kernel parameters should be adjusted on every broker:

```bash
# Increase max open file handles (Kafka opens many files for log segments)
fs.file-max = 1000000

# Increase socket buffer sizes for network throughput
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# Increase max backlog of pending connections
net.core.somaxconn = 4096

# Reduce TIME_WAIT socket accumulation
net.ipv4.tcp_fin_timeout = 15

# Page cache flush tuning
vm.dirty_ratio = 40
vm.dirty_background_ratio = 10

# Virtual memory
vm.swappiness = 1
vm.max_map_count = 262144
```

**File system:** Use XFS with large block sizes for sequential write performance. Avoid ext4 which has slower allocation for small files. Mount with `noatime` to disable access time updates on every read.

**Disk scheduler:** Use `none` (NVMe) or `noop` (SSD) for disk scheduling. Kafka's sequential I/O pattern does not benefit from the CFQ or deadline schedulers.

### Hardware Selection

| Component | Development | Production (High Throughput) | Production (Latency Sensitive) |
|-----------|-------------|------------------------------|--------------------------------|
| CPU | 4-8 cores | 16-32 cores | 16-32 cores with high clock speed |
| RAM | 16 GB | 64-128 GB | 128-256 GB (larger page cache) |
| Disk | 1 TB SSD | 4-8 TB NVMe x 4-6 in RAID 0 | 4-8 TB NVMe x 2-4 |
| Network | 1 Gbps | 25 Gbps | 50-100 Gbps |
| JVM Heap | 4 GB | 8 GB | 4 GB (prefer page cache) |

The most common hardware mistake is under-provisioning disk I/O. Kafka's throughput is limited by disk write speed in produce-heavy workloads and by network bandwidth in consume-heavy workloads. CPU is rarely the bottleneck.

### Rack Awareness

Kafka supports rack-aware replica placement. The rack ID is a configurable property for each broker. When Kafka assigns replicas, it ensures that replicas of the same partition are placed on different racks (assuming enough brokers and racks exist).

Rack awareness protects against rack-level failures — a power outage or network switch failure in one rack does not cause data loss because replicas exist on other racks.

```
Racks: 3 (rack-a, rack-b, rack-c)
Brokers per rack: 2
Replication factor: 3
Each partition has 1 replica in each rack
```

### Broker Replacement

When a broker fails in production:

1. The controller detects the failure (no heartbeat for `session.timeout.ms`).
2. The controller reassigns partition leadership from the failed broker to followers on other brokers.
3. The cluster continues operating with reduced replication factor.
4. A replacement broker is provisioned and added to the cluster.
5. Kafka automatically replicates data to the new broker through partition reassignment.

Automatic partition reassignment can generate significant network traffic. Limit the reassignment rate with `num.replica.fetchers` and `reassignment.rate.per.node`.

### Graceful Shutdown

Kafka brokers support graceful shutdown. When a broker receives a SIGTERM:

1. The broker notifies the controller that it is shutting down.
2. The controller moves leadership away from the shutting-down broker.
3. The broker waits for all ISR followers to catch up for partitions where it was leader.
4. The broker closes all connections and shuts down.

The shutdown can be delayed if a leader is waiting for followers to catch up. The `controlled.shutdown.max.retries` and `controlled.shutdown.retry.backoff.ms` parameters control this behavior.

A forced shutdown (SIGKILL or power loss) causes leader reassignment to happen after the broker is detected as failed. This takes longer and may cause data loss for partitions where the broker was the leader with unacknowledged produces.

Production Kafka deployments use multi-broker clusters with replicated partitions, rack awareness, and ZooKeeper or KRaft for metadata management. Kafka Connect handles integration with external systems. MirrorMaker 2 enables multi-region replication. Capacity planning requires estimating throughput, storage, and partition counts.
