Every architectural choice involves trade-offs. Kafka optimizes for high-throughput, durable, ordered message delivery at the cost of operational complexity, latency under certain configurations, and unsuitability for specific workloads. This page examines the trade-offs systematically.

## The Core Tension: Durability vs. Throughput

Kafka's fundamental design tension is between durability (how many copies of the data exist and how synchronously they are written) and throughput (how fast the system can ingest and serve data).

At one extreme: `acks=0`, `replication.factor=1`. Maximum throughput — 500+ MB/s per broker. Minimum durability — a single broker failure loses all data.

At the other extreme: `acks=all`, `min.insync.replicas=3`, `replication.factor=5`. Maximum durability — a write is only acknowledged after five nodes have it on disk. Throughput drops by 40-60% compared to the acks=0 baseline because the leader must wait for acknowledgment from four followers before responding.

The correct trade-off depends on your workload. A clickstream analytics pipeline can tolerate rare data loss (acks=1 is sufficient). A financial transaction system needs acks=all with min.insync.replicas=2 at minimum.

## Decision Matrix

The following decision matrix maps common requirements to Kafka configurations.

| Requirement | Configuration | Impact |
|---|---|---|
| Zero data loss | `acks=all`, `min.insync.replicas=2`, `replication.factor=3` | 40-60% throughput reduction vs. baseline |
| Maximum throughput | `acks=0`, `replication.factor=1`, `compression=zstd` | Highest risk of data loss |
| Low latency (p99 less than 10 ms) | `acks=1`, `linger.ms=0`, `batch.size=8192` | Lower throughput, more network round trips |
| Exactly-once processing | `enable.idempotence=true`, `transactional.id` enabled | Idempotent overhead, higher latency |
| Long retention (30+ days) | `log.retention.ms=2592000000`, `log.retention.bytes` set | High storage cost, slower consumer startup |
| Large messages (10+ MB) | `max.message.bytes=20971520`, `max.request.size=20971520` | Increased latency, more memory per broker |
| Multi-region replication | MirrorMaker 2 or Cluster Linking | Cross-region bandwidth cost, replication lag |
| Exactly-once stream processing | `processing.guarantee=exactly_once_v2` | 10-20% throughput reduction, transaction coordinator overhead |
| High compaction rate | `cleanup.policy=compact`, `min.cleanable.dirty.ratio=0.2` | CPU and I/O overhead for compaction, may impact produce throughput |
| Low-latency consumption | `fetch.min.bytes=1`, `fetch.max.wait.ms=10` | More fetch requests per second, higher broker CPU |
| Zero data loss on any failure | `acks=all`, `min.insync.replicas=3`, `replication.factor=4` | Only 1 broker can fail without losing ISR quorum |

### Trade-Off in Practice: Choosing acks and Replication

The most common trade-off decision in Kafka is choosing the `acks` setting and replication factor. Here is a decision tree for making this choice:

**Step 1: Determine the data criticality.**

- Is data loss acceptable in any scenario? If yes, consider `acks=0` or `acks=1` with RF=2.
- Is data loss unacceptable? If yes, use `acks=all` with RF=3 and `min.insync.replicas=2`.

**Step 2: Determine the latency requirement.**

- Is p99 latency under 10 ms required? Use `acks=1` with low `linger.ms` and small batches.
- Is latency of 10-50 ms acceptable? Use `acks=all` with moderate batching and compression.

**Step 3: Determine the operational tolerance.**

- Can the cluster survive 2 broker failures? Set RF=4 with `min.insync.replicas=3`.
- Can the cluster survive 1 broker failure? Set RF=3 with `min.insync.replicas=2`.
- Is the cluster small (3 brokers, non-critical)? Set RF=2 with `min.insync.replicas=2`.

**Real-world examples:**

| Workload | acks | RF | min.insync | Rationale |
|----------|------|----|------------|-----------|
| Clickstream analytics | 1 | 2 | 1 | Occasional loss acceptable, throughput priority |
| Application logs | 0 | 1 | 1 | Loss acceptable, max throughput needed |
| User activity events | 1 | 3 | 2 | Some loss OK, need survival of 1 failure |
| Payment transactions | all | 3 | 2 | Zero loss, survival of 1 failure |
| Order fulfillment | all | 3 | 2 | Zero loss, survival of 1 failure |
| Inventory updates | all | 4 | 3 | Zero loss, survival of 2 failures (retail peak) |
| Metrics (monitoring) | 0 | 1 | 1 | Latest value matters, history not critical |

## Common Operational Mistakes

Beyond the five anti-patterns, these operational mistakes appear frequently in production.

**Not monitoring the ZooKeeper/KRaft ensemble:** ZooKeeper latency affects the entire cluster. A ZooKeeper node with high disk latency causes the controller to process metadata changes slowly. Monitor ZooKeeper request latency and ensure it stays under 10 ms. In KRaft mode, monitor controller quorum health and Raft round-trip time.

**Setting retention too low for catch-up recovery:** If consumers can be offline for 2 hours during a deployment, retention must be at least 4 hours to allow catch-up without data loss. Factor in the time to detect the problem, diagnose it, fix it, and catch up.

**Ignoring network bandwidth limits:** A 10 Gbps network connection can handle 1,250 MB/s of raw throughput. With RF=3, the effective write throughput is 1,250 / 3 = 417 MB/s plus consume traffic. If the produce rate is 500 MB/s with 3 consumer groups, the total network traffic is 500 x 4 = 2,000 MB/s, exceeding 10 Gbps. The solution is either 25 Gbps networking or distributing brokers across more network interfaces.

**Running brokers on shared hardware:** Kafka competes for CPU, memory, and disk I/O with other processes. A broker that shares a server with a database or a build server experiences variable latency. Run Kafka brokers on dedicated servers with guaranteed resources.

**Forgetting to test the recovery path:** Leader election, rebalance, and partition reassignment are rarely tested until they are needed in production. Simulate failures regularly. Test what happens when a broker loses power, a consumer crashes, or a disk fills. Run a "game day" exercise where a broker is taken down intentionally and the team practices the recovery procedure.

**Not planning for client version upgrades:** Kafka client libraries have their own versioning and compatibility. An older consumer library may not work with a newer broker (Kafka protocol is backward compatible but not forward compatible). Upgrade clients before brokers. Test client versions against the target broker version in staging. Maintain a compatibility matrix that maps client versions to broker versions.

**Ignoring the impact of message size on throughput:** Kafka's default `max.message.bytes` is 1 MB. Larger messages (10 MB or more) consume significantly more broker resources per message. A single 10 MB message takes the same broker time to process as 10,000 1 KB messages but carries far less throughput. If you have large messages, either split them (send a reference and fetch separately) or increase broker and client buffers accordingly.

**Running consumer groups without static membership:** By default, every consumer restart generates a new member ID and triggers a rebalance. With static group membership (setting `group.instance.id`), a restarting consumer retains its identity and rejoins without triggering a rebalance. This is critical for any system that deploys new consumer versions regularly. Without static membership, a simple deployment can trigger cascading rebalances across the entire consumer fleet.

## Kafka in the CAP Theorem

The CAP theorem states that a distributed data system can guarantee at most two of three properties: Consistency, Availability, and Partition Tolerance. Kafka is a CP system under its default configuration.

**Consistency**: Kafka provides strong consistency for partition leaders. With `acks=all` and `min.insync.replicas=2`, a record is committed only when all ISRs acknowledge. Consumers with `isolation.level=read_committed` see only committed records.

**Partition Tolerance**: Kafka handles network partitions between brokers. If a network partition occurs, the replica on each side of the partition continues operating independently. The side without the ISR majority cannot accept writes with `acks=all`.

**Availability trade-off**: When a partition leader fails and no ISR is available, Kafka chooses consistency over availability by default (`unclean.leader.election.enable=false`). The partition stays unavailable until the old leader recovers. If you enable unclean leader election, Kafka becomes an AP system — it stays available but may lose records.

This trade-off is explicit and configurable. Kafka provides knobs to slide along the CP-AP spectrum:

- **CP (default)**: `unclean.leader.election.enable=false`. Availability suffers during extended failures.
- **AP**: `unclean.leader.election.enable=true`. Consistency suffers — some records may be lost if an out-of-sync replica becomes leader.

## Five Failure Modes

Understanding how Kafka fails is essential for operating it reliably.

### Failure Mode 1: Leader Failure

**What happens**: A broker that is the leader for 50 partitions crashes or becomes unresponsive.

**Impact duration**: 10-30 seconds. During this time, the partitions are unavailable for produce and consume.

**Recovery sequence:**
1. The controller detects the missed heartbeat after `session.timeout.ms` (default 45 seconds in ZooKeeper mode, shorter in KRaft).
2. The controller selects a new leader from the ISR for each partition that lost its leader.
3. The controller sends LeaderAndIsr requests to the affected brokers.
4. Followers start fetching from the new leader.
5. Producers and consumers discover the new leader through metadata refresh.

**Prevention:**
- Set `session.timeout.ms` to a value appropriate for your failure detection requirements (10 seconds for fast detection, 45 seconds to reduce false positives).
- Ensure at least `min.insync.replicas` replicas are operational at all times.
- Use rack awareness to avoid correlated failures.

### Failure Mode 2: Disk Full

**What happens**: One broker's disk reaches 100% capacity. The broker can no longer accept writes, or in some configurations, crashes entirely.

**Impact duration**: Until the disk is freed or the broker is replaced. All partition leaders on this broker cannot accept writes.

**Recovery sequence:**
1. If `log.disk.failure.occurrence.actions=SHUTDOWN`, the broker shuts down, triggering leader reassignment.
2. If the broker does not shut down, produce requests to its leader partitions fail with a disk error.
3. An operator must either delete old data (if retention policies allow) or add disk capacity.
4. The operator restarts the broker. It catches up from the ISR state.

**Prevention:**
- Set `log.retention.bytes` to 80% of total disk capacity.
- Monitor disk usage with alerts at 70% (warning) and 85% (critical).
- Use separate disks for Kafka data and OS logs.
- Enable `log.segment.delete.delay.ms` = 0 to eagerly delete expired segments.

### Failure Mode 3: Consumer Lag

**What happens**: A consumer group falls behind the latest write offset. The gap grows over time as producers write faster than consumers can process.

**Impact**: Increasing delay between event occurrence and event processing. If the retention threshold is crossed, consumers start missing records.

**Detection:** Monitor `kafka.consumer:type=consumer-fetch-manager-metrics,client-id=*` with the `records-lag-max` metric.

**Causes:**
- Consumer processing is CPU-intensive and cannot keep up with the produce rate.
- Consumer group has fewer consumers than partitions, leaving some partitions unprocessed.
- Consumer is blocked on a downstream system (database write, API call).
- Consumer group experiences frequent rebalances, reducing processing time.

**Recovery:**
1. Identify the lagging consumer group and its assigned partitions.
2. Scale up consumers (if partition count allows).
3. Optimize consumer processing (batch database writes, reduce per-record overhead).
4. If lag is severe, temporarily increase `max.poll.records` to consume faster (risk of processing timeout).
5. For extreme cases, reset the consumer offset to skip old data (data loss).

**Prevention:**
- Configure monitoring and alerting for consumer lag.
- Set retention to several days to provide a buffer for recovery.
- Right-size consumer group parallelism to match the partition count.
- Implement backpressure signaling from downstream systems.

### Failure Mode 4: Rebalance Storm

**What happens**: A consumer group enters an unstable state where consumers repeatedly join and leave the group, triggering continuous rebalances. No processing happens during rebalances.

**Impact**: Zero throughput for the consumer group during rebalances. Repeated rebalances can prevent any processing from occurring.

**Causes:**
- `max.poll.interval.ms` is too low for the processing time. The consumer is kicked out while processing a batch.
- Transient network issues cause consumers to miss heartbeats.
- Consumer group has too many consumers for the number of partitions (idle consumers drop out).
- Static group membership is not configured, so every restart changes group membership.

**Recovery:**
1. Stop all consumers in the group.
2. Increase `max.poll.interval.ms` and `session.timeout.ms`.
3. Ensure processing time per poll batch is consistently under `max.poll.interval.ms`.
4. Restart consumers one at a time.

**Prevention:**
- Set `max.poll.interval.ms` high enough to cover worst-case processing time.
- Set `session.timeout.ms` to at least 3x the expected processing time per record.
- Use the cooperative rebalancing protocol (`partition.assignment.strategy` set to `CooperativeStickyAssignor`).
- Configure `group.initial.rebalance.delay.ms` to avoid rebalancing during a batch of restarts.

### Failure Mode 5: Controller Failure

**What happens**: The controller broker fails or becomes unresponsive. The controller manages metadata operations — topic creation, leadership assignments, and partition reassignments. When the controller is unavailable, no metadata changes can be made.

**Impact**: Existing produce and consume operations continue for a limited time. New topic creation, partition reassignment, and broker membership changes are blocked. If a partition leader fails, no new leader can be elected — those partitions become unavailable.

**Recovery sequence (ZooKeeper mode):**
1. A surviving broker detects that the controller has not sent a heartbeat to ZooKeeper.
2. The broker creates an ephemeral node in ZooKeeper to claim controller leadership.
3. The new controller reads the full cluster state from ZooKeeper (broker list, topic configurations, partition assignments).
4. The new controller starts handling metadata requests.
5. Total recovery time: 5-30 seconds depending on cluster size.

**Recovery sequence (KRaft mode):**
1. The standby controllers detect the active controller's failure through Raft heartbeat timeout.
2. A standby initiates a Raft election.
3. The winning standby becomes the new active controller.
4. The new controller continues from the last committed metadata offset.
5. Total recovery time: 1-5 seconds.

**Prevention:**
- Deploy dedicated controller nodes with low-latency, high-availability networking.
- Monitor controller metrics: `kafka.controller:type=KafkaController,name=ActiveControllerCount` (should be exactly 1).
- In ZooKeeper mode, ensure the ZooKeeper ensemble has quorum (at least 3 nodes).
- In KRaft mode, deploy 3 controller nodes minimum (5 for larger clusters).

### Failure Mode 6: Partition Skew

**What happens**: Partition leaders are unevenly distributed across brokers. Some brokers handle 200 leader partitions while others handle 20. The heavily loaded brokers become bottlenecks.

**Impact**: Uneven resource utilization. The busiest broker determines the cluster's maximum throughput. Producers and consumers that write to or read from specific partitions experience degraded performance.

**Causes:**
- Brokers join or leave the cluster, and the partition assignment algorithm does not redistribute existing partitions.
- Topics with different partition counts use the cluster unevenly.
- Certain partitions have disproportionately high throughput (hot partitions).
- Operator error during manual partition reassignment.

**Detection:**
```
# Check partition leadership distribution
./kafka-topics.sh --describe --bootstrap-server localhost:9092
```
Look for brokers with significantly different leader counts.

**Recovery:**
1. Generate a partition reassignment plan using Kafka's built-in reassignment tool.
2. For hot partitions, consider splitting the partition by adding partitions to the topic (requires careful planning — partition count cannot be reduced).
3. For key-based hot spots, re-architect the key schema to distribute load more evenly.

**Prevention:**
- Enable `auto.leader.rebalance.enable=true`.
- Monitor leader distribution and trigger manual rebalance when skew exceeds 20%.
- When resizing the cluster, redistribute partitions immediately.

## Five Anti-Patterns

These patterns appear frequently in production and should be avoided.

### Anti-Pattern 1: Using Kafka as a Database

Kafka is a log, not a database. Teams sometimes treat Kafka topics as primary data stores, relying on compaction to maintain the latest state.

**Why it fails:**
- Kafka does not support indexes beyond offset-by-time and offset-by-position.
- Kafka cannot perform point lookups. To find a record by key, you must scan the entire partition or build an external index.
- Kafka does not support transactions beyond produce-to-multiple-partitions.
- Kafka does not support queries. You cannot ask "how many orders did user 42 place yesterday" without consuming the entire topic.

**What to do instead:** Store state in a database. Use Kafka for event streaming and change data capture. Use Kafka Streams or ksqlDB for stream-table joins if you need to enrich streams with state, but persist the state store in a proper database.

### Anti-Pattern 2: Ignoring Consumer Lag

Teams deploy Kafka, set up consumers, and never monitor consumer lag. By the time they notice a problem, consumers are hours or days behind.

**Why it fails:**
- Consumer lag grows silently until the retention threshold is crossed.
- Records that are deleted before being consumed are lost forever.
- Recovery becomes harder as lag increases — catching up from hours of lag puts more strain on the system.

**What to do instead:** Monitor `records-lag-max` for every consumer group. Alert when lag exceeds a threshold appropriate to your SLAs (1000 records for latency-sensitive systems, 100,000 for batch-oriented systems). Investigate lag increases promptly.

### Anti-Pattern 3: Too Many Partitions

Teams create topics with thousands of partitions to maximize parallelism, without considering the operational cost.

**Why it fails:**
- Each partition adds overhead: file descriptors, memory for replica state, and metadata management.
- Rebalance time increases with partition count. A rebalance for a topic with 2000 partitions can take 30-60 seconds.
- More partitions means more potential for leadership imbalance.
- Consumer lag metrics per partition become overwhelming.

**What to do instead:** Calculate the required partition count from throughput requirements and consumer parallelism. Start with fewer partitions (20-50 per broker) and scale up when needed. Remember that part of the cost of partitions is fixed — going from 50 to 100 partitions doubles the overhead but may not double the throughput.

### Anti-Pattern 4: Rebalance Timeouts Too Aggressive

Teams set `session.timeout.ms` to very low values (3 seconds or less) to fail fast, causing unnecessary rebalances during normal operation.

**Why it fails:**
- JVM garbage collection pauses of 100-500 ms are normal in Kafka consumers.
- Network jitter can delay heartbeat delivery by 50-200 ms.
- Processing time per poll varies with batch size.
- The combination of GC pauses, network jitter, and processing variability triggers false-positive rebalances.

**What to do instead:** Set `session.timeout.ms` to 15-45 seconds (the Kafka default is 45 seconds). Set `max.poll.interval.ms` to at least 2x the expected processing time for a full poll batch. Use `heartbeat.interval.ms` = 1/3 of `session.timeout.ms` for prompt failure detection without false positives.

### Anti-Pattern 5: Not Sizing Retention

Teams use the default 7-day retention without considering storage requirements or consumer behavior.

**Why it fails:**
- 7 days of data at 191 MB/s throughput requires 173 TB of storage with RF=3 and snappy compression.
- Consumers that are down for more than 7 days miss data permanently.
- Recovery from consumer lag requires the consumer to catch up within the retention window.

**What to do instead:** Calculate retention storage requirements explicitly. Set retention to match your consumer recovery SLA — if consumers must handle being offline for 3 days during holidays, retention must be at least 4-5 days. Use size-based retention as a safety net to prevent disks from filling. For topics with long retention, consider tiered storage (Confluent's Tiered Storage moves older data to S3 or similar).

## When NOT to Use Kafka

Kafka is not the right tool for every job. These scenarios suggest a different tool.

### Small-Scale Applications

Kafka's operational overhead — ZooKeeper, broker management, monitoring — is not justified for applications that process under 10,000 messages per day. Use a message queue like RabbitMQ or a managed pub/sub service.

### Real-Time Requirements Under 5 Milliseconds

Kafka's end-to-end latency is 2-15 milliseconds under normal conditions. For sub-millisecond delivery, use a different technology: Redis Streams for in-memory throughput, NATS for ultra-low latency messaging, or Aeron for financial trading systems.

### Database of Record

Kafka is not a database. If you need to query individual records by key, update records in place, or run ad-hoc queries, use a database. Kafka excels at streaming, not at serving.

### Job Queue with Per-Message Semantics

Kafka is built for streaming, not for individual job processing. If each message requires acknowledgment and complex retry logic, use RabbitMQ or Amazon SQS. Kafka consumer groups with offsets provide coarser-grained tracking than per-message acknowledgments.

### Small Messages Under 100 Bytes with High Fan-Out

Kafka's batching and compression overhead is wasted on very small messages. For IoT sensor readings (50 bytes, 1 million per second), NATS or Pulsar may be more efficient.

### Rapidly Changing State

Kafka is not designed for state that changes thousands of times per second for the same entity. Each change is a new record. If a single stock price changes 1,000 times per second and you only need the latest price, Kafka's log will accumulate 999 stale records for every 1 useful record. Compaction helps but runs periodically. Use Redis or a database for rapidly changing state.

### Real-Time Leaderboards and Counters

Kafka's at-least-once delivery and asynchronous replication make it unsuitable for applications that require strongly consistent counters. If two consumers each produce increment events for the same counter, the final value may be incorrect due to duplicate processing. Use Redis with atomic increment operations or a database with transactional counters.

### Complex Event Processing with Tight Latency SLAs

If events must be correlated across multiple streams within 10 milliseconds, Kafka's consumer poll model adds too much latency. Use a stream processing engine like Apache Flink with low-latency checkpointing, or a specialized CEP engine like Esper.

## Failure Mode Recovery Summary

Understanding the expected recovery time for each failure mode helps you set SLAs and design for graceful degradation.

| Failure Mode | Detection Time | Recovery Time | Total Unavailability | Data Loss Risk |
|------------|---------------|---------------|---------------------|----------------|
| Leader failure | `session.timeout.ms` (10-45s) | 2-10s (leader election) | 12-55s | None (ISR elects new leader) |
| Controller failure (ZK) | 10-30s | 5-30s (new controller bootstrap) | 15-60s | None (metadata persisted) |
| Controller failure (KRaft) | 3-10s | 1-5s (Raft election) | 4-15s | None (Raft log persisted) |
| Disk full | Immediate (error on produce) | 1-60 min (operator action) | Until resolved | Partial (unflushed page cache) |
| Consumer lag | None (automatic) | Varies (scale consumers) | Processing delay | If retention exceeded |
| Rebalance storm | After 2+ rebalances | 5-30 min (config change + restart) | Processing halted | None (offsets preserved) |
| Partition skew | Gradual (weeks) | 1-24 hours (reassignment) | Performance degradation | None |
| Network partition | `session.timeout.ms` | 10-60s (ISR re-election) | 10-60s | Rare (if unclean leader elected) |
| ZooKeeper quorum loss | 10-30s | 5-60 min (ZK recover or rebuild) | Until ZK restored | None (data safe, metadata frozen) |

The most dangerous failure modes are those with long recovery times — disk full, ZooKeeper quorum loss, and severe consumer lag. These failures cannot be automated away and require operator intervention. Invest monitoring, runbooks, and automation in these areas first. Proactive monitoring and early detection are the most effective tools for minimizing the impact of any failure mode.

## Summary

Kafka trades operational complexity for throughput, durability, and ordering guarantees. It is a CP system by default, configurable toward AP at the cost of potential data loss. Six failure modes — leader failure, disk full, consumer lag, rebalance storm, controller failure, partition skew — require monitoring and proactive management. Five anti-patterns — using Kafka as a database, ignoring lag, too many partitions, aggressive timeouts, ignoring retention — cause production incidents. Knowing when to use Kafka and when to choose another tool is as important as knowing how to configure it. The most successful Kafka deployments are those that respect its strengths as a distributed commit log and do not force it into roles it was not designed for.
