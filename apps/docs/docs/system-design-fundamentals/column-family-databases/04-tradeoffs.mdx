# Column-Family Databases — Trade-offs & Failure Modes

## Core Tension — Consistency vs Availability in Cassandra

Every distributed system grapples with the fundamental trade-off between consistency and availability. Column-family databases, particularly Apache Cassandra, make a deliberate choice to prioritize availability and partition tolerance over strong consistency. This choice is not accidental --- it is baked into every layer of the architecture, from the gossip protocol to the read-repair mechanism.

### The CAP Constraint

The CAP theorem states that a distributed data store can only guarantee two of the following three properties simultaneously: consistency, availability, and partition tolerance. Column-family databases like Cassandra choose AP --- availability and partition tolerance --- while accepting eventual consistency as the default model.

### Tunable Consistency

Rather than enforcing a fixed consistency model, Cassandra exposes tunable consistency --- the application decides how many replicas must acknowledge a read or write before the operation is considered successful. This is the single most important concept for understanding column-family trade-offs.

#### Write Consistency Levels

| Level | Description | Behavior |
|-------|-------------|----------|
| ANY | Any node (may be a hint) | Fastest, lowest durability |
| ONE | At least one replica | Fast, low durability |
| QUORUM | Majority of replicas (N/2 + 1) | Balance of speed and safety |
| LOCAL_QUORUM | Majority in local DC | Avoids cross-DC latency |
| EACH_QUORUM | Majority in every DC | Highest durability, slowest |
| ALL | All replicas | Strongest guarantee, least available |

#### Read Consistency Levels

| Level | Description | Behavior |
|-------|-------------|----------|
| ONE | Return first response | Fastest, may be stale |
| QUORUM | Compare majority of replicas | Consistent reads at cost |
| ALL | Check all replicas | Strongest, slowest |
| LOCAL_QUORUM | Majority in local DC | Avoids cross-DC reads |
| SERIAL | Linearizable for lightweight transactions | Strong sequencing |

### The Quorum Math

Understanding quorum behavior requires knowing the replication factor (RF) and the consistency level. Cassandra uses a simple formula to determine whether read repairs or writes succeed:

```python
# RF = Number of copies of each row across the cluster
# CL = Consistency level (number of replicas that must respond)

# Write success condition:
# CL_write <= number of live replicas

# Read success condition:
# CL_read <= number of live replicas

# Strong consistency condition (linearizable read):
# CL_read + CL_write > RF

# Example: RF = 3
# CL_write = QUORUM (2), CL_read = QUORUM (2)
# 2 + 2 = 4 > 3  --> Strong consistency achieved

# Example: RF = 3
# CL_write = ONE (1), CL_read = ONE (1)
# 1 + 1 = 2 < 3  --> Eventual consistency only
```

The critical insight: when `CL_read + CL_write > RF`, you get strong consistency. When `CL_read + CL_write <= RF`, you get eventual consistency. This is the lever you pull to trade latency for guarantees.

### Linearizable Consistency via Lightweight Transactions

For the rare cases where you need true linearizability --- such as distributed locks or uniquenes constraints --- Cassandra offers lightweight transactions (LWT) built on the Paxos consensus protocol. The cost is severe: LWTs run at the pace of the slowest replica and incur 4 round trips per operation. They should be used sparingly, typically for operations that execute less than 5-10% of total throughput.

```python
# Lightweight transaction (IF NOT EXISTS) — 4 Paxos round trips
session.execute(
    "INSERT INTO user_profiles (user_id, email, name) "
    "VALUES ('u123', 'alice@example.com', 'Alice') "
    "IF NOT EXISTS"
)

# Regular insert — no consensus, single round trip
session.execute(
    "INSERT INTO user_profiles (user_id, email, name) "
    "VALUES ('u123', 'alice@example.com', 'Alice')"
)
```

### The Hinted Handoff Fallback

When a replica is unavailable during a write, Cassandra does not fail the operation immediately. Instead, the coordinator node stores a hint --- a note that says "when node X comes back, replay this write to it." Hints are stored locally and replayed when the downed node rejoins. This dramatically improves write availability at the cost of potential consistency windows: a hint may be lost if the coordinator itself fails before replay, or it may be replayed long after the write, creating a window of inconsistency.

### Column-Family Selection Criteria

Choose column-family when:

| Criterion | Threshold | Example |
|-----------|-----------|---------|
| Write throughput | > 50,000 writes/sec/node | IoT sensor ingestion |
| Data volume | > 10 TB per cluster | Time-series event logs |
| Availability SLA | 99.99%+ uptime | User session store |
| Geographic distribution | Multi-region active-active | Global user profiles |
| Data model | Wide rows, static schema | Financial tick data |
| Query pattern | Known access paths, PK-based | Event sourcing |

Avoid column-family when:

| Criterion | Threshold | Alternative |
|-----------|-----------|-------------|
| ACID transactions | Required | PostgreSQL, MySQL |
| Complex joins | Frequent | Relational DB |
| Dynamic queries | Unknown at design time | MongoDB |
| Consistency > availability | Strict linearizability | HBase, Spanner |
| Small data (< 100 GB) | No scale needs | Postgres, Redis |

## CAP Theorem — AP Positioning

Column-family databases like Cassandra are architected as AP systems under the CAP theorem. Understanding what this means in practice --- and how Cassandra bends the CAP rules through tunable consistency --- is essential for making correct engineering decisions.

### How Cassandra Achieves AP

Cassandra distributes data across nodes using consistent hashing, with each node responsible for a range of token values. When a partition occurs (network split between data centers, for example), Cassandra's design ensures that both sides of the partition continue to accept reads and writes independently:

```
Network partition between US-East and EU-West:

  US-EAST (continues operating)          EU-WEST (continues operating)
  ┌──────────────────────────┐          ┌──────────────────────────┐
  │   ┌──┐ ┌──┐ ┌──┐ ┌──┐  │          │   ┌──┐ ┌──┐ ┌──┐ ┌──┐  │
  │   │A1│ │A2│ │A3│ │A4│  │          │   │B1│ │B2│ │B3│ │B4│  │
  │   └──┘ └──┘ └──┘ └──┘  │          │   └──┘ └──┘ └──┘ └──┘  │
  │   Writes: OK            │          │   Writes: OK            │
  │   Reads: OK (local)     │  ─ ─ ─ ─│   Reads: OK (local)     │
  │   Hinted handoff: ON    │  PARTITN│   Hinted handoff: ON    │
  └──────────────────────────┘          └──────────────────────────┘

  After partition heals → read repair + hinted handoff reconcile data
```

This AP behavior is prioritized through several mechanisms:

**Gossip protocol**: Nodes communicate via gossip to exchange state information. Gossip is decentralized --- any node can communicate with any other node --- so the cluster does not depend on a single point of coordination. During a partition, gossip continues within each side of the split, and nodes on each side maintain a consistent view of their local partition.

**No single point of failure**: Every node is symmetric. There is no master, no primary replica, and no centralized coordinator. Every node can accept reads and writes for any token range (though it may forward to the correct coordinator). This eliminates the single-node bottleneck that forces CP systems to sacrifice availability during failures.

**Hinted handoff + read repair**: When a write target is unreachable during a partition, the coordinator stores a hint. When the partition heals, the hint is replayed. Separately, read repair ensures that stale replicas are updated during normal read operations when digests diverge. These mechanisms together mean that data eventually converges without requiring a quorum to be available at write time.

### The Cost of AP

The AP choice carries real costs. Applications that read their own writes may see stale data if the read consistency level is ONE and the write is still propagating. Counters can drift during partitions. Secondary indexes can become inconsistent. The application layer must be designed to tolerate eventual consistency --- or must explicitly use QUORUM or SERIAL consistency levels at the cost of reduced availability.

```python
# Eventual consistency pitfall (default CL.ONE for both):
session.default_consistency_level = ConsistencyLevel.ONE

# User writes a profile update:
session.execute(
    "UPDATE user_profiles SET email = 'new@email.com' WHERE user_id = 'u123'"
)

# Then immediately reads it back — may get OLD value!
result = session.execute(
    "SELECT email FROM user_profiles WHERE user_id = 'u123'"
)
print(result.one().email)  # Could print 'old@email.com' !

# Fix: use CL.QUORUM for read-your-writes:
session.execute(
    "UPDATE user_profiles SET email = 'new@email.com' WHERE user_id = 'u123'",
    consistency_level=ConsistencyLevel.QUORUM
)

result = session.execute(
    "SELECT email FROM user_profiles WHERE user_id = 'u123'",
    consistency_level=ConsistencyLevel.QUORUM
)
```

### Cassandra vs HBase — AP vs CP

The most direct comparison is between Cassandra (AP) and HBase (CP). Both are column-family stores, but HBase builds on HDFS and uses ZooKeeper for coordination, making it a CP system under the CAP theorem:

| Property | Cassandra | HBase |
|----------|-----------|-------|
| CAP | AP | CP |
| Coordination | Gossip + hinted handoff | ZooKeeper + HMaster |
| Write availability during partition | Writes accepted on both sides | Writes accepted only on majority side |
| Read during partition | OK from any live node | OK from region servers on majority side |
| Consistency model | Tunable (eventual to strong) | Strong (per-row) |
| Recovery after partition | Read repair + anti-entropy | WAL replay + region reassignment |

## Failure Modes

Column-family databases exhibit specific failure modes that operators must understand. These failures are not bugs --- they are consequences of architectural choices, particularly the AP orientation and the lack of cross-row transactions.

### Read Repair Amplification

Read repair is Cassandra's mechanism for fixing stale data during normal reads. When a read at CL.QUORUM returns digests that disagree, Cassandra fetches the full data from the responding replicas, compares them, and writes the most recent version to any replicas that had stale data.

The problem: read repair causes **amplification** --- a single read operation can trigger multiple internal writes. Under high contention or when many rows are stale simultaneously, read repair can saturate the cluster:

```
Read at CL.QUORUM (RF=3):
1. Coordinator sends digest request to all 3 replicas
2. Responses come back:
   - Replica A: digest=0x7A (latest)
   - Replica B: digest=0x7A (latest)
   - Replica C: digest=0x39 (stale)
3. Coordinator fetches full row from Replica A (digest match)
4. Coordinator sends write with latest data to Replica C  ← AMPLIFICATION
5. Coordinator returns result to client

Amplification factor: 1 read → 2 digest requests + 1 full data sync + 1 write
```

Under heavy read-repair load, 30-50% of cluster throughput can be consumed by repairs rather than actual application reads. The mitigation is to use hinted handoff and repaired data tracking (Cassandra 3.0+) to minimize unnecessary repairs.

### Compaction Storms

Cassandra writes are append-only (LSM-tree). Over time, SSTables accumulate and must be compacted --- merged, sorted, and rewritten. Compaction is I/O and CPU intensive. When multiple nodes trigger compaction simultaneously, or when a single node has many large SSTables to compact, the result is a **compaction storm**:

```
Timeline of a compaction storm:

T+0:  Normal operation — 8 SSTables per node, compaction running smoothly
T+5:  Bulk load of 500 GB historical data — SSTables jump to 50 per node
T+10: Compaction kicks in across all 12 nodes simultaneously
T+15: Disk I/O saturation — 95% iowait on all nodes
T+20: Read latency spikes from 5ms to 5000ms
T+25: Some nodes start rejecting writes due to backlog
T+30: Cluster enters "hinted handoff emergency" mode
T+45: Compaction backpressure kicks in, gradual recovery
```

Mitigation strategies include:

- **Size-tiered compaction** (STCS): fast but can cause large temporary disk usage
- **Leveled compaction** (LCS): more predictable I/O pattern at cost of higher write amplification
- **Time-window compaction** (TWCS): ideal for time-series data where SSTables expire naturally
- **Throttle compaction**: `compaction_throughput_mb_per_sec` limits I/O per node
- **Stagger bulk loads**: distribute large writes across hours, not minutes

### Tombstone Overload

When you delete data in Cassandra, it does not remove the data immediately (to avoid inconsistency with unreachable replicas). Instead, it writes a **tombstone** --- a deletion marker that persists until compaction. Excessive tombstones cause severe read performance degradation:

```python
# Problematic pattern — creates tombstones on every UPDATE:
session.execute(
    "INSERT INTO user_events (user_id, event_time, data) "
    "VALUES ('u123', '2026-01-01', 'old_data')"
)

# This update creates a tombstone for the 'data' column:
session.execute(
    "UPDATE user_events SET data = 'new_data' "
    "WHERE user_id = 'u123' AND event_time = '2026-01-01'"
)

# Even worse — frequent row deletions in a wide partition:
for i in range(100000):
    session.execute(
        "DELETE FROM user_events WHERE user_id = 'u123' AND event_time = ?",
        (some_timestamp,)
    )
    # Each DELETE creates 1+ tombstone
```

Symptoms of tombstone overload:

- **TombstoneOverwhelmed** exception: Cassandra refuses to scan a partition with too many tombstones (default threshold: 100,000 tombstones per query)
- Read latency spikes of 10x or more as Cassandra scans through tombstones
- Compaction can never catch up because tombstones propagate through compactions

Detection and mitigation:

```python
# Detect tombstone-heavy queries via tracing:
session.execute(
    "SELECT * FROM user_events WHERE user_id = 'u123'",
    trace=True
)
# Look for: "Scanned 50000 live rows and 150000 tombstones"

# Per-node tombstone count:
# nodetool tablestats keyspace_name.table_name | findstr "Tombstone"

# Fixes:
# 1. Use TTL-based expiration instead of explicit DELETE where possible
#   CREATE TABLE user_events (...) WITH default_time_to_live = 86400;
#
# 2. Set gc_grace_seconds appropriately (default 864000 = 10 days)
#   ALTER TABLE user_events WITH gc_grace_seconds = 432000;
#
# 3. Use leveled compaction for faster tombstone removal
#   ALTER TABLE user_events WITH compaction = {
#     'class': 'LeveledCompactionStrategy'
#   };
```

### Write Backpressure and Hints Pressure

When a Cassandra node cannot keep up with write volume, writes pile up in the memtable, then in the commit log, then in the messaging queue. The write path has several pressure points:

```python
# Key configuration knobs for write backpressure:

# Memory buffer per table
# memtable_heap_space_in_mb: 2048 (default 1/4 of heap)
# memtable_offheap_space_in_mb: 2048

# Commit log
# commitlog_segment_size_in_mb: 32
# commitlog_total_space_in_mb: 8192

# Maximum write threads
# native_transport_max_threads: 128

# Hint storage
# max_hint_window_in_ms: 10800000 (3 hours)
# hints_compression: LZ4Compressor
```

When backpressure builds, symptoms cascade:

1. Write latency increases
2. Messaging queue fills
3. Some writes start to time out
4. Coordinators begin storing hints
5. Hint pressure increases (if max_hints_per_node is reached, new hints are dropped)
6. Once hints are dropped, data loss occurs permanently

The critical metric to monitor is **pending compaction** and **dropped mutations**. If either trends upward, the cluster is approaching a failure state.

### Coordinator Failure Cascade

In Cassandra, the coordinator node --- the node that receives the client request --- is responsible for managing the full request lifecycle. If a coordinator fails mid-flight, the client must retry. Under high load, coordinator failures can cascade:

```
Normal state:
  Client ──> Coordinator A ──> Replica B, C, D
                                    │
                          All succeed, coordinator returns to client

Coordinator failure:
  Client ──> Coordinator A ──> Replica B, C, D (request in-flight)
                │
              CRASH
                │
  Client ──> TIMEOUT ──> RETRY ──> Coordinator E ──> Replica B, C, D
                                    │
                          Replicas B and C respond with success
                          Replica D: "I see no record of the partial write"
                          (Write may be partially applied or lost)
```

For critical write workloads, use an idempotent retry strategy or specify `RetryPolicy.DOWNGRADING_CONSISTENCY` to gracefully fall back to a lower consistency level rather than failing.

## Anti-Patterns

Column-family databases reward specific access patterns and punish others. These anti-patterns represent the most common mistakes teams make when adopting Cassandra or ScyllaDB.

### Anti-Pattern 1: Using Cassandra Like a Relational Database

The most common anti-pattern is designing schemas as if Cassandra were MySQL. This manifests as:

- **Normalized tables with foreign keys**: Cassandra has no JOIN support. Each query must be served by a single table. Denormalization is required, not optional.
- **Indexing everything**: Secondary indexes in Cassandra are local (per-node), not global. Querying by a secondary index requires fanning out to all nodes. Above 100,000 rows per partition, secondary index performance collapses.
- **Expecting ORDER BY on any column**: Cassandra can only order rows within a partition by the clustering column order. You cannot ORDER BY an arbitrary column without an explicit index or materialized view (which have their own limitations).

```python
# RELATIONAL ANTI-PATTERN — don't do this:
session.execute("""
    CREATE TABLE orders (
        order_id UUID PRIMARY KEY,
        customer_name TEXT,
        customer_email TEXT,
        product_name TEXT,
        product_price DECIMAL,
        order_total DECIMAL
    )
""")

# Then trying to query like a relational DB:
# SELECT * FROM orders WHERE customer_email = 'a@b.com'  -- Requires ALLOW FILTERING (slow)
# SELECT * FROM orders WHERE product_price > 100.00      -- Requires ALLOW FILTERING (disaster)
# SELECT * FROM orders JOIN customers ON ...              -- Not possible

# COLUMN-FAMILY CORRECT PATTERN — query-first design:
session.execute("""
    CREATE TABLE orders_by_customer (
        customer_id UUID,
        order_time TIMESTAMP,
        order_id UUID,
        customer_name TEXT,
        product_name TEXT,
        product_price DECIMAL,
        order_total DECIMAL,
        PRIMARY KEY (customer_id, order_time, order_id)
    ) WITH CLUSTERING ORDER BY (order_time DESC)
""")

# Now the query is fast and single-partition:
# SELECT * FROM orders_by_customer WHERE customer_id = ?
```

### Anti-Pattern 2: Unbounded Partition Growth

Cassandra partitions (all rows sharing the same partition key) are stored contiguously on disk. A partition that grows without bound creates a **hot partition** --- a single point of contention that limits scalability:

```python
# ANTI-PATTERN: Single partition key grows indefinitely
session.execute("""
    CREATE TABLE sensor_readings (
        sensor_id TEXT,           # Only 10 sensors, each creates huge partition
        reading_time TIMESTAMP,
        reading_value DOUBLE,
        PRIMARY KEY (sensor_id, reading_time)
    )
""")

# Over 6 months, each sensor may have 15M+ rows in ONE partition
# Read performance degrades, compaction slows, memory pressure increases

# BETTER: Bucket the partition key
session.execute("""
    CREATE TABLE sensor_readings_by_day (
        sensor_id TEXT,
        reading_date DATE,         # Day-level bucket
        reading_time TIMESTAMP,
        reading_value DOUBLE,
        PRIMARY KEY ((sensor_id, reading_date), reading_time)
    )
""")

# Now each partition holds ~8640 rows (one per 10 seconds × 24 hours)
# Queries for "last hour" hit one partition, not a 15M-row monolith
```

Guidelines for partition size:

| Metric | Healthy | Warning | Critical |
|--------|---------|---------|----------|
| Rows per partition | < 100,000 | 100,000 - 1,000,000 | > 1,000,000 |
| Partition size | < 100 MB | 100 MB - 1 GB | > 1 GB |
| Partitions per node | < 10 million | 10M - 50M | > 50M |

### Anti-Pattern 3: Using ALLOW FILTERING for Production Queries

`ALLOW FILTERING` tells Cassandra to scan every row in the table to satisfy a query. This is equivalent to a full table scan in a relational database and should never appear in production code:

```python
# NEVER DO THIS IN PRODUCTION:
session.execute(
    "SELECT * FROM user_profiles WHERE city = 'New York' ALLOW FILTERING"
)
# This fans out to EVERY node, reads EVERY partition,
# and can saturate cluster I/O in seconds.

# CORRECT: Design a table for the query pattern:
session.execute("""
    CREATE TABLE users_by_city (
        city TEXT,
        user_id UUID,
        name TEXT,
        email TEXT,
        PRIMARY KEY (city, user_id)
    )
""")

# Now the query is a single-partition read:
session.execute(
    "SELECT * FROM users_by_city WHERE city = 'New York'"
)
```

### Anti-Pattern 4: Batch Operations for Bulk Loading

Batches in Cassandra are often misunderstood. A batch is not a performance optimization --- it is a coordination primitive. Using batches for bulk loading creates excessive coordinator overhead:

```python
# ANTI-PATTERN: Massive batch for bulk load
batch = BatchStatement()
for i in range(10000):
    batch.add(
        "INSERT INTO events (id, data) VALUES (?, ?)",
        (i, f"event_data_{i}")
    )
session.execute(batch)
# Coordinator must coordinate all 10,000 operations → OOM risk

# CORRECT: Use asynchronous writes
prepared = session.prepare("INSERT INTO events (id, data) VALUES (?, ?)")
futures = []
for i in range(10000):
    futures.append(session.execute_async(prepared.bind((i, f"event_data_{i}"))))

# Wait for all to complete
results = [f.result() for f in futures]
```

Use batches only when:
- **Atomicity is required** across multiple operations
- **Operations share the same partition key** (batches across partitions are expensive)
- **Batch size is small** (< 10-20 operations)

### Anti-Pattern 5: Read-Heavy Workloads on Uncompressed Data

Cassandra is optimized for write throughput, not read performance. Heavy read workloads expose weaknesses unless you plan for them:

```
Read amplification by configuration:

                    READ THROUGHPUT   DISK USAGE   CPU USAGE
No compression:     100% (baseline)   100%          10%
LZ4 compression:    125% (faster)     60%           20%
Zstd compression:   110%             45%           40%

Key observation: compression reduces disk I/O so much that
LZ4-compressed reads can be FASTER than uncompressed reads,
because more data fits in the OS page cache.
```

For read-heavy workloads, also consider:
- **Caching**: enable row cache for hot partitions
- **Compression**: use LZ4Compressor (fastest decompression)
- **Bloom filters**: ensure they are sized correctly (default `fp_chance=0.01`)
- **Separate read-only replicas**: if consistent reads are tolerable from slightly stale data

## When NOT to Use

Column-family databases are powerful tools for specific use cases. They are counterproductive --- even dangerous --- for others. Recognizing when NOT to use a column-family database is as important as knowing when to use one.

### Strong Consistency Requirements

If your application requires linearizable consistency for every operation, you should not use Cassandra without careful design. The AP orientation means that under normal operation with CL.QUORUM, you get strong consistency --- but only while the cluster is healthy. During a partition, you lose either consistency (by accepting writes) or availability (by using CL.ALL).

Systems that require strict ACID: financial ledgers, inventory systems with oversell prevention, reservation systems, and any system where a single inconsistent read could lead to financial loss or data corruption.

```python
# Cassandra is WRONG for this use case:
# Flight seat reservation — must prevent double-booking
session.execute(
    "UPDATE seats SET passenger_id = ? "
    "WHERE flight_id = ? AND seat_number = ? "
    "IF passenger_id = null",           # LWT required
    (user_id, "FL123", "14A")
)
# LWT uses Paxos — 4 round trips, limited to ~1000 ops/sec/node
# At scale, a relational DB with row-level locking is more appropriate
```

### Complex Query Patterns

If your application requires ad-hoc analytical queries, joins across multiple entities, aggregation queries with GROUP BY, or full-text search, Cassandra is the wrong tool. Each of these patterns requires either:

- ALLOW FILTERING (dangerous)
- Spark or similar analytics engine (operational overhead)
- Materialized views (limited, with known bugs in production)

For these workloads, use a purpose-built tool: PostgreSQL for OLTP with complex queries, Elasticsearch for full-text search, ClickHouse or Druid for real-time analytics, or Trino for query federation.

### Small-Scale Deployments

Cassandra's operational complexity makes it inappropriate for single-node or small-scale deployments. The minimum practical cluster is 3 nodes (for quorum-based replication), and a production cluster typically requires 6-12 nodes across 2-3 data centers.

| Scale | Recommendation |
|-------|---------------|
| < 100 GB data, single region | PostgreSQL, MySQL, or SQLite |
| < 1 TB data, single region | PostgreSQL with read replicas |
| > 1 TB data, multi-region | Cassandra or ScyllaDB |
| > 100 TB data, global | Cassandra or ScyllaDB |

### Transactional Workloads with Cross-Partition Atomicity

If you need atomic operations spanning multiple rows or partitions --- traditional relational transactions --- Cassandra is not the right choice. While Cassandra offers lightweight transactions for single-partition atomic compare-and-set, there are no distributed transactions across partitions:

```python
# Cassandra CANNOT atomically:
# UPDATE account_a SET balance = balance - 100 WHERE id = 'A'
# UPDATE account_b SET balance = balance + 100 WHERE id = 'B'
# These are two independent operations with no transactional guarantees.

# CQL batch across partitions does NOT provide atomicity:
batch = BatchStatement()
batch.add("UPDATE accounts SET balance = balance - 100 WHERE id = 'A'")
batch.add("UPDATE accounts SET balance = balance + 100 WHERE id = 'B'")
session.execute(batch)
# If the batch fails after applying the first update, the second may not apply.
# Use a relational database with 2PC for money-moving transactions.
```

### Low-Latency Reads on Arbitrary Columns

Cassandra's data model is optimized for queries by partition key. If your application needs to query by arbitrary fields with single-digit-millisecond latency, Cassandra (or any column-family store) is the wrong choice:

```
Query pattern        Cassandra  MongoDB  PostgreSQL  Elasticsearch

PK lookup               1-5ms     1-5ms     1-5ms       5-10ms
Secondary index        10-50ms    5-10ms    2-5ms        2-5ms
Full-text search        N/A       N/A       10-50ms      2-10ms
Ad-hoc filter (no idx) 1-30s*    10-100ms  10-200ms     5-50ms
Joins                   N/A       N/A       2-50ms       N/A
Range scan on non-PK   N/A       N/A       2-10ms       N/A
```

*ALLOW FILTERING — do not use in production

### Real-Time Analytics with Frequent Schema Changes

Cassandra's schema is static and schema changes (ALTER TABLE) have operational costs. Adding a column is lightweight (metadata-only). But dropping a column, changing a data type, or modifying the primary key requires a full table rebuild:

| Schema Change | Cassandra Cost | MongoDB Cost |
|--------------|----------------|--------------|
| Add column | Metadata only (ms) | Automatic (0ms) |
| Drop column | Tombstone + compaction | Automatic (0ms) |
| Rename column | Not supported | Simple rename |
| Change type | Recreate table (hours/TB) | Schema validation change |
| Add index | Background (minutes) | Background (minutes) |

For workloads with evolving schemas --- event tracking where new fields appear frequently, A/B test data with dynamic metrics --- a document database (MongoDB, Couchbase) or a schemaless columnar store (BigQuery) is more appropriate.

### When Cassandra IS the right choice

To close the loop, here is a consolidated summary of when column-family databases truly shine:

1. **Time-series data at scale** --- IoT events, metrics, logs, clickstreams. The wide-row model maps naturally to time-ordered data, and partition pruning via time-based bucket keys keeps partitions manageable.

2. **High-volume event ingestion** --- Systems that ingest 100,000+ events per second across distributed sources. Examples include ad-tech impression logging, financial tick data, and social media activity feeds.

3. **Global user profiles / session stores** --- Applications that serve users from multiple geographic regions and need active-active writes in every region. Cassandra's multi-DC replication handles this naturally without a central coordinator.

4. **Shopping cart / recommendation systems** --- Workloads with high write throughput and eventual consistency tolerance. If a user's cart takes a few milliseconds to propagate to all replicas, the business impact is zero.

5. **Message queues (with Cassandra-backed queues)** --- Some queue implementations (Kafka-style, SQS-style) use Cassandra as the durable backing store. The high write throughput and linear scalability make it suitable for this role.

6. **Feature stores for ML** --- Feature stores that serve pre-computed features for ML inference need fast key-value lookups and high write throughput for feature materialization. Cassandra's wide-row model fits streaming feature computation well.

### Consequences of Wrong Choice

Finally, understanding the cost of using Cassandra for the wrong workload:

| Wrong Workload | Pain Points | Observable Symptoms |
|---------------|-------------|---------------------|
| OLTP with joins | Painful denormalization, data duplication | 10x storage amplification |
| Ad-hoc analytics | Full scans, ALLOW FILTERING | Seconds-to-minutes query times |
| Strong consistency | LWT bottlenecks, write amplification | >50% throughput loss, latency spikes |
| Small data (< 100 GB) | Operational overhead dwarfs data size | 6-node cluster for 50 GB data |
| Schema evolution | Migration downtime, data rebuilds | Hours of planned maintenance per schema change |
| Cross-partition transactions | Application-level compensation logic | Complex error recovery, risk of data loss |

## Key Takeaways

- Cassandra sits in the AP corner of CAP --- it chooses availability and partition tolerance over strong consistency by default. Tunable consistency lets you slide between eventual and strong consistency on a per-request basis.

- The quorum condition `CL_read + CL_write > RF` determines whether you get strong or eventual consistency. Mastering this formula is the key to operating Cassandra with predictable guarantees.

- Hot partitions, tombstone overload, compaction storms, and unbounded partition growth are the four most common failure modes. Each has known mitigations, but all require proactive monitoring and schema design discipline.

- The most common anti-pattern is using Cassandra like a relational database. Query-first schema design --- modeling tables around the queries they will serve --- is non-negotiable for performance.

- Cassandra is a specialized tool for write-heavy, multi-region, eventually-consistent workloads at massive scale. For ACID transactions, complex queries, small data, or dynamic schemas, reach for a different database type.
