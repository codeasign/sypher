# Column-Family Databases -- Challenges

The best way to learn column-family databases is to design, build, and debug them. These challenges are organized into three tiers. Tier 1 tests your foundation. Tier 2 tests your design skills. Tier 3 tests your expertise.

## Tier 1: Foundation

These challenges take about 5 minutes each.

### Challenge 1: Schema Design for a Blog Comments System

Design a Cassandra schema for a blog comment system. Each blog post can have thousands of comments. The queries are:

- Get all comments for a blog post, sorted by newest first, paginated 20 at a time.
- Get a single comment by its ID.

**Model Answer.**

```sql
CREATE TABLE blog_comments_by_post (
    post_id uuid,
    comment_id timeuuid,
    author_id uuid,
    author_name text,
    content text,
    created_at timestamp,
    PRIMARY KEY (post_id, comment_id)
) WITH CLUSTERING ORDER BY (comment_id DESC);

CREATE TABLE blog_comments_by_id (
    comment_id uuid PRIMARY KEY,
    post_id uuid,
    author_id uuid,
    author_name text,
    content text,
    created_at timestamp
);
```

The first table serves the "comments by post" query. `post_id` is the partition key, so all comments for a post are co-located on one node. `comment_id` (a timeuuid) is the clustering key sorted in descending order, so the newest comments come first. Pagination uses `LIMIT 20` and the last `comment_id` from the previous page.

The second table serves the "get comment by ID" query. `comment_id` is the primary key, providing a direct lookup. This is a secondary table maintained alongside the first one. In Cassandra, you denormalize to serve different query patterns.

### Challenge 2: Why Does This Query Fail?

Your team writes the following Cassandra query and it is slow:

```sql
SELECT * FROM events WHERE event_type = 'click'
  AND created_at >= '2024-01-01' AND created_at < '2024-02-01';
```

The table schema is:

```sql
CREATE TABLE events (
    event_id uuid PRIMARY KEY,
    event_type text,
    created_at timestamp,
    payload text
);
```

Why is this query slow, and how do you fix it?

**Model Answer.**

The query is slow because `event_type` is not part of the primary key. Cassandra must scan every partition in the cluster to find rows matching `event_type = 'click'`. This is a full cluster scan -- it sends the query to every node, and each node scans every SSTable.

The fix depends on the query pattern:

```sql
CREATE TABLE events_by_type (
    event_type text,
    created_at timestamp,
    event_id uuid,
    payload text,
    PRIMARY KEY (event_type, created_at, event_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

Now `event_type` is the partition key. The query only needs to read one partition (or a few, if `event_type` has low cardinality). The clustering key `created_at` enables the range filter on dates.

If `event_type` has very low cardinality (for example, only 3 event types), you would need a composite partition key to avoid hot partitions:

```sql
CREATE TABLE events_by_type_and_date (
    event_type text,
    date_bucket text,        -- "2024-01-01", "2024-01-15"
    created_at timestamp,
    event_id uuid,
    payload text,
    PRIMARY KEY ((event_type, date_bucket), created_at, event_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

### Challenge 3: Identify the Anti-Pattern

You are reviewing a Cassandra schema designed by a teammate. Spot the problems:

```sql
CREATE TABLE users (
    user_id uuid PRIMARY KEY,
    name text,
    email text,
    created_at timestamp,
    last_login timestamp,
    login_count counter,
    -- 30 more columns...
);

CREATE TABLE orders_by_user (
    id uuid PRIMARY KEY,
    user_id uuid,
    order_total decimal,
    status text,
    created_at timestamp
);
```

List at least three anti-patterns.

**Model Answer.**

**Anti-pattern 1: Counter column in a table with non-counter columns.** In Cassandra, a table containing a counter column can only contain the counter and the primary key columns. No other non-PK columns are allowed. This schema will fail to compile because `login_count` is a counter but `name`, `email`, and other columns exist.

**Anti-pattern 2: `orders_by_user` has no partition key for user_id.** The only way to query this table by user is `SELECT * FROM orders_by_user WHERE user_id = ?`, which is a full cluster scan because `user_id` is not indexed. The table should have `user_id` as part of the primary key, ideally as the partition key.

**Anti-pattern 3: `orders_by_user` uses a random UUID as the sole primary key.** A random UUID partition key distributes data evenly but prevents range queries. If you need to query orders by date for a user, you need a clustering key. The schema should be:

```sql
CREATE TABLE orders_by_user (
    user_id uuid,
    created_at timestamp,
    order_id uuid,
    order_total decimal,
    status text,
    PRIMARY KEY (user_id, created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

## Tier 2: Design

These challenges take about 15 minutes each.

### Challenge 4: Design a Real-Time Chat System

Design a Cassandra schema for a real-time chat application similar to Discord or Slack. Requirements:

- 10 million daily active users.
- 1 million channels.
- Each channel stores up to 1 million messages.
- Queries: show the most recent 50 messages in a channel; load older messages via infinite scroll (pagination); send a message.
- Messages must be retained for 90 days.

**Model Answer.**

```sql
CREATE TABLE messages_by_channel (
    channel_id bigint,
    message_id timeuuid,          -- timeuuid provides chronological sorting
    author_id bigint,
    author_name text,
    content text,
    created_at timestamp,
    PRIMARY KEY (channel_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC)
  AND default_time_to_live = 7776000   -- 90 days in seconds
  AND compaction = {
    'class': 'TimeWindowCompactionStrategy',
    'compaction_window_unit': 'DAYS',
    'compaction_window_size': 1
  };
```

Key decisions:

- `channel_id` as the partition key. All messages for a channel are co-located on one node. For very active channels (Discord's busy channels), consider splitting by month: `(channel_id, yyyymm)`.
- `message_id` as a timeuuid, sorted DESC. This automatically orders messages newest-first and avoids sequential ID hotspotting (timeuuids are unique and distributed across time).
- TTL of 90 days. No manual cleanup needed.
- TWCS with daily windows. Each day's messages are compacted once, minimizing write amplification.

Pagination query:

```sql
-- First page (most recent 50)
SELECT * FROM messages_by_channel WHERE channel_id = ?
  LIMIT 50;

-- Next page (messages before the oldest message on the first page)
SELECT * FROM messages_by_channel WHERE channel_id = ?
  AND message_id < ?   -- last message_id from previous page
  LIMIT 50;
```

Capacity estimate for a busy channel with 100 messages/second:

- Storage: 100 msg/s * 500 bytes * 86,400 s/day = 4.3 GB/day per channel.
- With RF=3: 12.9 GB/day.
- With 90-day TTL: steady state at approximately 388 GB per busy channel.
- For the entire system of 1M channels (assuming 1,000 active): 388 TB.

This fits in a 20-node cluster with 4 TB SSDs per node.

### Challenge 5: Multi-Tenant SaaS Analytics

Design a column-family database schema for a multi-tenant analytics platform. Each tenant sends events. Queries:

- Count events by event_type over the last 24 hours for a tenant.
- Get all events for a tenant in the last hour.
- Get all events for a specific user in a tenant.

The platform has 1,000 tenants. Some tenants produce 100 events/day, others produce 10 million events/day.

**Model Answer.**

```sql
-- Wide events table for raw event storage
CREATE TABLE events_by_tenant (
    tenant_id text,
    bucket timestamp,              -- hourly bucket: '2024-03-15T10:00:00Z'
    event_id timeuuid,
    user_id text,
    event_type text,
    properties map<text, text>,
    created_at timestamp,
    PRIMARY KEY ((tenant_id, bucket), event_id)
) WITH CLUSTERING ORDER BY (event_id DESC)
  AND default_time_to_live = 259200    -- 3 days
  AND compaction = {
    'class': 'TimeWindowCompactionStrategy',
    'compaction_window_unit': 'HOURS',
    'compaction_window_size': 1
  };

-- Secondary table for user-level queries
CREATE TABLE events_by_user (
    tenant_id text,
    user_id text,
    event_id timeuuid,
    event_type text,
    properties map<text, text>,
    created_at timestamp,
    PRIMARY KEY ((tenant_id, user_id), event_id)
) WITH CLUSTERING ORDER BY (event_id DESC)
  AND default_time_to_live = 259200;

-- Counter table for event_type counts (hourly)
CREATE TABLE event_type_counts (
    tenant_id text,
    date_hour text,               -- "2024-03-15-10"
    event_type text,
    count counter,
    PRIMARY KEY ((tenant_id, date_hour), event_type)
);
```

Key decisions for the large-tenant problem:

- The composite partition key `(tenant_id, bucket)` prevents a single tenant with 10 million events/day from creating a hot partition. Each hour is a separate partition, and data is distributed across the ring.
- The hourly bucket also enables efficient deletion: when TTL expires on an hour bucket, the entire SSTable can be dropped (TWCS per hour).
- Counter tables for hourly event_type counts avoid scanning raw events for aggregation queries. The application increments the counter on each event write.
- For small tenants (100 events/day), the hourly bucket creates small partitions. This is acceptable: even 24 small partitions per day is manageable, and they are compacted efficiently.

### Challenge 6: Debug a Production Latency Spike

Your Cassandra cluster has been running smoothly for 6 months. Today, read latency spiked from 5 ms p99 to 5,000 ms p99. The spike started 30 minutes ago and is getting worse. Write latency is also elevated but less severely.

You have access to `nodetool` and the metrics dashboard. Walk through your debugging process.

**Model Answer.**

**Step 1: Check cluster health.** Run `nodetool status` to see if any nodes are down or in an unusual state. Run `nodetool gossipinfo` to verify gossip convergence. If a node is down, that is your first priority.

**Step 2: Check compaction.** Run `nodetool compactionstats`. If pending compactions are high (50+), compaction is likely consuming disk I/O, starving reads. Check `nodetool getcompactionthroughput` to see if throttling is limiting compaction to a very low value (under 16 MB/s).

**Step 3: Check for a hot partition.** Run `nodetool cfstats` (or `tablestats`) for the affected keyspace and table. Look for extremely high read and write counts on a single table. If one table has 90% of the reads, examine its partition key design.

**Step 4: Check SSTable count per read.** Run `nodetool tablestats <table>`. Look at the SSTable count. If it is above 20, read performance degrades because every read must check all SSTables. This suggests compaction is behind.

**Step 5: Check tombstone warnings.** Enable tracing on a sample read: `nodetool settraceprobability 0.01`. Run the problematic query and check the trace. If the trace shows thousands of tombstones scanned, you have a tombstone overload problem.

**Step 6: Check GC (Cassandra only).** Run `nodetool gcstats` and `nodetool tpstats`. If GC pauses exceed 1 second frequently, JVM garbage collection is causing latency spikes. Check the GC logs in `/var/log/cassandra/gc.log` for long STW (stop-the-world) pauses.

Likely root causes in order of probability:
- A traffic spike triggered a compaction storm (check compaction stats).
- A recently created large partition (viral event) is overloading one node (check cfstats per node).
- Tombstone accumulation from high delete volume (check tracing).
- G1GC heap pressure from oversized memtables (check GC logs).
- Disk failure causing I/O latency on one node (check disk metrics for await time).

## Tier 3: Expert

These challenges take about 30 minutes each.

### Challenge 7: Design a Global Session Store

Your company needs a global session store for a social media platform with 500 million monthly active users. Sessions are created on login and updated on every page view. Sessions expire after 24 hours of inactivity. The system must handle 2 million session lookups per second and 500,000 session writes per second at peak.

Requirements:
- Active-active in 3 regions (US-East, EU-West, AP-Southeast).
- 99.99% availability target.
- Read latency under 10 ms for the user's home region.
- Sessions must survive a single-region outage.
- Conflict resolution must handle users logged in from multiple devices.

Design the schema, the replication strategy, the capacity plan, and the failover mechanism.

**Discussion.**

This challenge tests your ability to design a globally distributed, highly available column-family database deployment. There is no single correct answer, but a strong answer addresses these points:

**Schema.**

```sql
CREATE TABLE sessions (
    session_id text,
    user_id bigint,
    device_id text,
    created_at timestamp,
    last_accessed_at timestamp,
    is_active boolean,
    attributes map<text, text>,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400;
```

Each session is a single partition (no clustering columns). TTL auto-expires after 24 hours. The `attributes` map stores arbitrary session data (locale, theme, feature flags).

**Replication strategy.**

NetworkTopologyStrategy with RF=3 in each region. Each region stores 3 copies of every session. Writes use LOCAL_QUORUM (2 of 3 within the writer's region). Asynchronous cross-region replication handles the other two regions.

**Active-active writes.** A user could be logged in from the US and travel to Europe, creating a session in both regions. Conflict resolution uses last-write-wins by timestamp. The session with the later `created_at` or `last_accessed_at` wins. This is acceptable because session data is ephemeral. If two devices are active simultaneously, each has a different `session_id`, so there is no conflict.

**Capacity plan.** 2M reads/second + 500K writes/second = 2.5M ops/second. With RF=3 and LOCAL_QUORUM, each operation touches 2 of 3 replicas. Cluster operations = 2.5M * 2/3 = approximately 1.67M ops/second per region.

Each node handles 15K ops/second. Nodes needed = 1.67M / 15K = 112 nodes per region. Add 30% headroom and compaction overhead: 146 nodes per region. At 64 GB RAM, 8 vCPU, 2 TB NVMe per node, that is approximately 146 nodes * $500/month = $73,000/month per region in cloud costs.

**Failover.** If one region goes down, DNS-based routing (Route53, Traffic Manager) directs traffic to the remaining regions. The surviving regions have the full session data via asynchronous replication (with some seconds of replication lag). Users whose sessions were created in the failed region may experience stale sessions until the last write catches up, but the system remains available.

**Operations.** Weekly repair within each region. Automated backup of SSTable snapshots to S3 (or GCS, Azure Blob) with 7-day retention. Monitoring on compaction pending count, p99 read/write latency per region, hint count, and gossip convergence time.

### Challenge 8: Migrate from Cassandra to ScyllaDB

Your company runs 200 Cassandra nodes across 2 regions. You are experiencing JVM GC pauses of 2-5 seconds during compaction, causing read latency spikes to 10 seconds at the p999 level. You decide to migrate to ScyllaDB.

Design the migration plan. The migration must not cause downtime. The cluster serves 800,000 writes/second and 300,000 reads/second at peak. You cannot lose data.

**Discussion.**

**Strategy: Dual-write with gradual cutover.** Run both clusters in parallel for the migration period. Applications write to both Cassandra and ScyllaDB simultaneously. Reads come from Cassandra initially, then gradually switch to ScyllaDB.

**Phase 1: Provision ScyllaDB cluster (week 1).** Deploy a ScyllaDB cluster sized to handle the production workload: at 10x throughput per node compared to Cassandra (ScyllaDB's claim), you need approximately 20-30 ScyllaDB nodes to replace 200 Cassandra nodes. Use the same instance types (or convert Cassandra's AWS instance type to equivalent bare-metal or cloud-optimized instances for ScyllaDB).

**Phase 2: Seed data (week 2).** Use `sstableloader` (Cassandra's bulk loader tool) to load existing SSTables from Cassandra into ScyllaDB. Since ScyllaDB is API-compatible with Cassandra's SSTable format, the loader can read Cassandra's SSTables directly. Run this during low-traffic hours. Validate row counts and checksums after loading.

**Phase 3: Dual-write (week 3-4).** Modify the application layer to write to both clusters. Use a write-through proxy pattern: one library call writes to both Cassandra and ScyllaDB. If ScyllaDB write fails, the application logs the error but does not block the user (Cassandra write is the source of truth). This allows ScyllaDB to catch up without impacting production.

**Phase 4: Backfill (week 3, concurrent with phase 3).** Run a backfill job that reads data from Cassandra and writes any missing data to ScyllaDB. This covers the window between the initial seed and the start of dual-writes. Use a distributed job (Spark or custom worker) that scans Cassandra partitions in parallel and compares timestamps.

**Phase 5: Read migration (week 5).** Start shifting read traffic from Cassandra to ScyllaDB. Begin with 10% of reads, then 25%, 50%, 75%, 100% over the course of a week. At each step, compare p50/p99/p999 latencies between the two systems. If ScyllaDB metrics are stable, continue the shift.

**Phase 6: Decommission Cassandra (week 6).** After all reads and writes are on ScyllaDB and the system is stable, stop the dual-write. Run a final consistency check (compare checksums of a random sample of partitions). Decommission Cassandra nodes via `nodetool decommission`. Archive a final backup.

**Rollback plan.** If ScyllaDB has stability issues during the migration, switch all reads back to Cassandra immediately. Dual-writes ensure Cassandra has all the data. There is zero data loss risk if the rollback happens within the TTL window (since ScyllaDB is receiving all writes and Cassandra is the authoritative source).

**Risk mitigation.**

- **Schema compatibility.** Verify that all CQL types used by the application are supported in the ScyllaDB version being deployed. ScyllaDB is generally API-compatible, but check for any Cassandra 4.0+ features that ScyllaDB may not yet support.
- **Compaction differences.** ScyllaDB's compaction is configured differently from Cassandra's. Test compaction behavior with actual traffic patterns in a staging environment before the dual-write phase.
- **Monitoring integration.** Ensure ScyllaDB's Prometheus metrics integrate with your existing monitoring stack (Grafana dashboards, alert thresholds) before the cutover.
- **Team training.** Schedule 2-3 days of ScyllaDB training for the operations team. The ScyllaDB monitoring UI, tooling (`nodetool` port), and troubleshooting procedures differ from Cassandra's.
