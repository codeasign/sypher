# Column-Family Databases in Production

The theory is useful. What matters is how real companies use column-family databases to solve real problems at scale. This page covers four case studies, the major open source implementations, cloud provider offerings, and the lessons learned from years of production experience.

## Company Case Studies

### Netflix: Cassandra as the Backbone of Personalization

Netflix runs one of the largest Cassandra deployments in the world. As of 2024, Netflix operates over 500 Cassandra clusters and more than 2,500 nodes across multiple AWS regions. Cassandra is the primary data store for Netflix's personalization engine, serving tailored recommendations to over 260 million subscribers.

Netflix contributed several key tools to the Cassandra ecosystem: **Astyanax** (a robust Java client with connection pooling, automatic failover, and request retry), **Cassandra Reaper** (for automated anti-entropy repair scheduling), and **Priam** (for configuration management and backup/restore). These tools are still widely used or have been adapted into community projects.

Netflix's key lesson: design each cluster for a single workload. A user-history cluster has different compaction, caching, and consistency settings than a video-metadata cluster. Monolithic clusters that serve multiple workloads are harder to tune and harder to operate.

### Discord: Migrating from MongoDB to ScyllaDB

Discord migrated its messaging platform from MongoDB to ScyllaDB in 2020 after hitting performance limits. Discord's message store handles billions of messages, with 100,000+ active concurrent operations on a single channel during peak hours (large gaming communities).

The problem with MongoDB was predictable: under high concurrent write load, MongoDB's document-level locking caused latency spikes. A busy channel's message writes would queue behind each other, causing P99 latency to exceed 1 second during peak events.

Discord's lesson: the single-partition read pattern is the killer use case for column-family databases. Every message query is a partition scan on `channel_id`. There are no cross-partition queries, no joins, no aggregations. This is the ideal workload.

### Apple: iCloud Mail with Cassandra

Apple uses Cassandra for iCloud Mail, storing metadata about billions of emails across hundreds of nodes. The workload is write-heavy and partition-tolerant: users tolerate eventual consistency for their email metadata (flags, folders, read status). Apple's key contribution was the "Leveled Compaction" strategy, which they proposed as a Cassandra Improvement (CASSANDRA-2752) and which later became the default for read-heavy workloads.

Apple's cluster operated at 500,000+ writes per second across multiple datacenters. Their primary concern was read latency: when a user opens their iCloud Mail, all metadata for that mailbox must be fetched within 500 ms. Leveled compaction (LCS) ensures that at most 2-3 SSTables need to be checked per read, keeping read latency under 10 ms per node.

### Uber: Time-Series Data with Schemaless in Cassandra

Uber built its "Schemaless" database on top of Cassandra to store trip data. Each trip has a unique set of attributes that change over time: driver location, route waypoints, surge multiplier, payment status, and customer rating. The wide-column model allows each trip's attributes to evolve independently without schema migrations.

Uber's Cassandra deployment spans multiple regions with automatic failover. During peak hours (rush in multiple cities simultaneously), the system handles 1+ million writes per second. Uber uses Cassandra's TTL feature extensively: trip metadata is retained for 90 days, after which it is automatically deleted. No garbage collection scripts needed.

Uber's lesson: column-family databases are ideal for data with variable schemas and inherent TTL expiry. A 90-day data retention policy maps directly to Cassandra's table-level TTL setting, making the database self-cleaning.

## Open Source Implementations

**Apache Cassandra.** The original open-source wide-column store from Facebook (2008), now an Apache Top-Level Project. Written in Java. The standard for peer-to-peer column-family databases. Mature ecosystem with extensive tooling. Best for teams that want the most battle-tested solution and are comfortable with JVM tuning.

**Apache HBase.** Built on HDFS. Master-slave architecture with strong consistency. Best for use cases that need consistent reads and writes and already run Hadoop. Used by Facebook Messenger, OpenTSDB, and numerous telecom companies.

**ScyllaDB.** C++ rewrite of Cassandra that is API-compatible with Cassandra's CQL and binary protocol. Claims 10x throughput per node compared to Cassandra. Uses SeASTAR framework for shared-nothing async I/O. No GC pauses. Best for performance-critical workloads where Cassandra's JVM overhead is a bottleneck.

**Bigtable Emulator.** Google publishes a local emulator of Bigtable for development and testing. The emulator is not suitable for production but allows you to develop against the Bigtable API without a cloud account.

## Cloud Provider Implementations

**AWS Keyspaces (for Apache Cassandra).** Fully managed Cassandra-compatible service. Serverless: no nodes to manage, no compaction to configure, no repair to schedule. Automatically scales to handle any throughput. Supports CQL. Ideal for teams that want a Cassandra-like data model without operational overhead. Limitations: no custom compaction strategies, no nodetool access, no multi-region write in a single keyspace (must use cross-region replication via AWS Global Tables, which adds latency).

**Google Cloud Bigtable.** The original wide-column store, now available as a managed service. Supports the HBase API (via the Apache HBase client for Cloud Bigtable) and the Bigtable API. Key difference from Cassandra: Bigtable is CP (consistent), not AP. Use Bigtable when you need strong consistency with the wide-column model. Pricing is based on cluster size (compute nodes + storage), not on request volume.

**Azure Cosmos DB for Apache Cassandra.** Microsoft's managed Cassandra-compatible service. Cosmos DB offers multi-region writes with configurable consistency levels. The Cassandra API supports CQL. Key advantage: turnkey global distribution with 99.999% availability SLA. Key limitation: not all CQL features are supported (for example, materialized views and lightweight transactions have limited support).

## Lessons Learned

Five patterns emerge from thousands of production deployments.

**1. Schema design is the critical skill.** In a relational database, schema design is about normalization. In a column-family database, schema design is about query patterns. You cannot "just query it differently later." Every access path must be designed into the schema upfront. The most expensive production incidents come from schemas that were designed for the wrong access patterns.

**2. Monitoring compaction is not optional.** Compaction is not a background maintenance task. It is a core part of the write path that directly affects read performance. Without visibility into compaction pending count and throughput, you will be surprised by latency degradation during peak traffic.

**3. Repair, repair, repair.** Skipping repairs for weeks or months is the most common cause of data inconsistency in production Cassandra clusters. Automate repair scheduling with tools like Cassandra Reaper. Run full repairs at least weekly. Incremental repairs (available in Cassandra 3.0+) reduce the time by only repairing ranges that changed since the last repair.

**4. Test with production-scale data.** Cassandra behaves differently at 10 GB than at 10 TB. Secondary indexes that work at development scale cause timeouts at production scale. Bloom filter performance degrades when SSTable counts climb. Test with realistic data volumes and request patterns in a pre-production environment.

**5. Plan for failure, not just performance.** A cluster that runs smoothly at 50% utilization will behave differently at 80% utilization during a node failure. The remaining nodes must absorb the failed node's load. Design for N+1 redundancy and use load balancing that accounts for node health. Similarly, plan for region-level failover: cross-region replication should be tested quarterly with actual traffic shifts.
