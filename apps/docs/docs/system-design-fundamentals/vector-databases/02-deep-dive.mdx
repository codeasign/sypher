## Internal Architecture

A vector database is not one component — it's a stack of layers, each with its own data structures and algorithms. Understanding what happens inside the black box is what separates passing an interview from acing it.

The architecture is organized into stacked layers: the API layer handles communication, the query planner coordinates search across segments, each segment holds its own HNSW index and metadata index in memory with vectors on disk, and the WAL ensures crash recovery.

## Data Structures and Algorithms

### The HNSW Graph

HNSW is the canonical data structure for vector search. It's a multi-layer graph where each layer is a sub-sampled view of the layer below.

```
Structure: Directed graph with N layers (typically 16-32)
Each node = one vector
Each edge = "this node is a neighbor of that node"
```

**What structure is used:** A hierarchical graph. Each node stores a list of neighbor node IDs per layer. The top layer has approximately 1% of total nodes (exponentially decreasing). The bottom layer has 100% of nodes.

**Why this structure:** A flat graph requires O(n) traversal in the worst case. A hierarchical graph gives O(log n) search by enabling coarse-to-fine navigation. Without the hierarchy, you'd have to check every node's neighbors.

**Time complexity:**
- Insert: O(log n × ef_construction × M) — log n for the top-layer navigation, ef_construction for the candidate expansion, M for neighbor updates
- Search: O(log n × ef) — log n layers, ef candidate checks per layer
- Memory: O(n × M × layers × 8 bytes) — each edge stored as a 64-bit pointer

### The Reservoir Sampling Trick for Layer Assignment

When a new vector is inserted into HNSW, the algorithm must decide which layers it belongs to. This is done using an exponentially decaying random function:

```
layer = floor(-ln(uniform(0,1)) × L)
```

Where L is a normalization factor (typically 1/log(M)). This means:
- ~50% of vectors go to layer 1 only (bottom layer)
- ~25% go to layers 1-2
- ~12.5% go to layers 1-3
- And so on, exponentially fewer as layers increase

If you skip this and assigned all vectors to all layers, the top layer would have N nodes instead of exp(-N), and navigation would require O(n) checks at every layer.

### Metadata Indexing

Vector databases also need fast metadata filtering. Common data structures:

- **Hash index:** O(1) lookups for exact match filters (category = "outerwear"). Used for low-cardinality fields.
- **B-tree:** O(log n) range queries for numeric fields (price between 50 and 100). Used for high-cardinality fields.
- **Bitmap index:** O(1) bitwise AND/OR for multi-value filters. Used when you need to combine many filters quickly.

### Distance Computation

The core operation: given two vectors, how similar are they? The database must compute this millions of times per second.

<AsciiDiagram title="Distance Metrics — How Similarity Is Computed" content={`
  1. Cosine Similarity (most common for text embeddings)
     ─────────────────────────────────────────────────
     cos(θ) = (A · B) / (║A║ × ║B║)
     
     Result: 1.0 = same direction, 0.0 = orthogonal, -1.0 = opposite
     
     A: [0.23, -0.45, 0.67]    B: [0.19, -0.38, 0.71]
     A·B = 0.23×0.19 + (-0.45)×(-0.38) + 0.67×0.71
         = 0.0437 + 0.171 + 0.4757
         = 0.6904
     ║A║ = √(0.23² + 0.45² + 0.67²) = 0.830
     ║B║ = √(0.19² + 0.38² + 0.71²) = 0.817
     cos(θ) = 0.6904 / (0.830 × 0.817) = 0.690

  2. Euclidean Distance (L2) — good for image embeddings
     ───────────────────────────────────────────────────
     d(A,B) = √(Σ(Ai - Bi)²)
     
     Result: 0.0 = identical, larger = more different
     A: [0.23, -0.45, 0.67]    B: [-0.15, 0.72, -0.33]
     diff² = (0.38)² + (-1.17)² + (1.0)²
           = 0.1444 + 1.3689 + 1.0 = 2.5133
     d = √2.5133 = 1.585

  3. Dot Product — for normalized vectors
     ──────────────────────────────────────
     A·B = Σ(Ai × Bi)
     Result: larger = more similar
     Requires vectors to be length-normalized first
`} />

Modern vector databases use SIMD (Single Instruction Multiple Data) instructions to compute distances in parallel. AVX-512 can process 16 float operations per cycle, reducing a 768-dimension dot product from hundreds of cycles to under 50.

## Consistency and State

### Node Crashes Mid-Operation

**What happens:** The WAL contains every write operation that was acknowledged to a client. On restart, the database replays the WAL to reconstruct the in-memory index. Any writes that were in-progress but not acknowledged are lost — clients must retry.

**Resolution strategy:** Write-ahead logging with periodic checkpointing. The database snapshots the in-memory index to disk every N operations (configurable, typically every 10,000-100,000 inserts). On recovery, it loads the latest snapshot then replays WAL entries after that snapshot.

### Network Partitions

**What happens:** In a distributed vector database (Pinecone, Milvus cluster), a partition splits nodes into two groups that can't communicate. Each group continues accepting writes independently.

**Resolution strategy:** Most vector databases are AP (available, partition-tolerant) rather than CP. During a partition, both sides accept writes. When the partition heals, the database uses last-writer-wins (LWW) conflict resolution based on timestamps or vector clocks. This means writes during a partition on the losing side may be lost.

If you skip conflict resolution and kept both versions, you'd need application-level merging — something most teams aren't prepared for.

### Two Writes Simultaneously

**What happens:** Two clients send inserts for the same vector ID at the same time.

**Resolution strategy:** Last-writer-wins. The database picks a winner based on a timestamp (client-side or server-side). The losing write is discarded. Some databases (Qdrant) support optimistic concurrency control — the client provides an expected version number, and the write fails if the version doesn't match.

If you skipped deduplication and kept both copies, the index would have duplicate vectors for the same ID, and searches would return the same item twice with slightly different scores.

### Stale Reads

**What happens:** A vector was inserted 5ms ago but a search query returns results that don't include it.

**Resolution strategy:** This is expected behavior in most vector databases — they offer eventual consistency by default. New vectors are visible only after the current segment finishes its indexing cycle (typically 1-10 seconds). For read-after-write consistency, some databases support "refresh" operations that flush pending writes to the index, at the cost of performance.

If you skipped the consistency trade-off and forced every write to be immediately visible, the index would need to lock on every search — reducing throughput by 10x or more.

## Configuration and Tuning

| Setting | Controls | Safe Default | When to Change | What Breaks If Wrong |
|---------|----------|-------------|----------------|----------------------|
| **M** | Max neighbors per node in HNSW graph | 16 | Increase to 32-48 for higher recall on sparse datasets | Too high (M over 64): memory blows up, inserts slow down. Too low (M under 8): recall collapses. |
| **ef_construction** | Search width during index build | 200 | Increase to 400-800 for higher recall on difficult datasets | Too high: index build takes hours instead of minutes. Too low: poor graph quality, low recall. |
| **ef** | Search width at query time | 200 | Increase for higher recall (500), decrease for lower latency (50) | Too high: queries take 50ms instead of 5ms. Too low: recall drops below 0.85. |
| **nlist** | Number of IVF clusters | 10× sqrt(n) | Increase for larger datasets | Too high: empty clusters waste memory. Too low: poor partition quality. |
| **nprobe** | IVF cells to search | 10 | Increase for higher recall (20), decrease for lower latency (3) | Too high: queries become slow. Too low: miss close vectors in unsearched cells. |
| **Segment size** | Max vectors per segment | 1M-10M | Larger for write-heavy workloads, smaller for read-heavy | Too large: compaction takes too long. Too small: too many segments, slow searches. |
| **Replication factor** | Number of copies per shard | 1 | Set to 3 for production HA | Too low: data loss on node failure. Too high: expensive writes, wasted memory. |
