## How This Topic Appears

Vector databases appear in system design interviews in four distinct forms:

### Direct Questions

1. "Explain how a vector database works at a high level."
2. "What's the difference between HNSW and IVF?"
3. "How does cosine similarity work and when would you use it?"
4. "What is an embedding and how is it generated?"

### Design Questions

5. "Design a semantic search system for an e-commerce platform."
6. "Design a RAG (Retrieval-Augmented Generation) pipeline for a customer support chatbot."
7. "Design a content recommendation system for a video streaming service."
8. "Design a similarity-based fraud detection system for a payment platform."

### Debugging Questions

9. "Your vector search is returning irrelevant results. Walk me through debugging."
10. "The search latency jumped from 20ms to 200ms after adding metadata filters. What's happening?"

### Trade-Off Questions

11. "Why would you choose a vector database over a traditional search engine like Elasticsearch?"
12. "What's the trade-off between HNSW and IVF indexes?"
13. "When would you use PostgreSQL with pgvector instead of a dedicated vector database?"
14. "How would you handle the cold start problem for vector search with new data?"
15. "What happens to your vector search system when 90% of queries use a rare metadata filter?"

## What the Interviewer Is Actually Testing

**For direct questions:** Can you explain a complex technical concept clearly? Do you understand the fundamentals, not just the vocabulary?

**For design questions (L3-L5):** Can you architect a system that actually works at scale? Do you know what components are needed beyond just the vector database?

**For design questions (L6+):** Can you identify the hard problems before they happen? Do you proactively raise trade-offs and failure modes?

**For debugging questions:** Do you have a systematic approach to debugging? Can you reason from symptoms to root causes?

**For trade-off questions (all levels):** Do you understand that engineering is about choosing the least bad option, not finding the perfect one?

**What distinguishes a 3/4 from a 4/4:**
- 3/4: Correct understanding, mentions the right terms, gives a plausible design.
- 4/4: Identifies failure modes before being asked. Quantifies everything. Discusses alternatives unprompted. Shows awareness of real-world constraints (cost, operations, team size).

## Level-by-Level Expectations

### L3/L4 — Junior / New Grad

**Minimum expected knowledge:**
- What a vector is and how embedding works
- The difference between exact search and ANN
- What cosine similarity measures
- Basic awareness of HNSW as an algorithm

**Vocabulary they must know:** Vector, embedding, dimension, similarity score, ANN, nearest neighbor, recall.

**What a passing answer looks like:**
"A vector database stores data as vectors — lists of numbers that represent meaning. When you search, the database converts your query into a vector and finds other vectors that are mathematically close to it. This is different from keyword search because it finds things by meaning, not by exact words. For speed, it uses approximate nearest neighbor algorithms like HNSW, which builds a graph that lets it find close vectors without checking everything."

### L5 — Mid-Level

**Expected depth of understanding:**
- Can explain HNSW and IVF differences in detail
- Understands the recall-speed trade-off
- Knows how metadata filtering interacts with vector search
- Can discuss embedding quality and its impact
- Awareness of hybrid search

**Example question:** "Design a semantic search system for an e-commerce platform."

**Model answer:**
"I'd start by considering the data flow. Products need to be embedded when they're created or updated — I'd use a sentence-transformer model that runs as a sidecar service. The embedding service processes product descriptions, titles, and categories into 768-dimensional vectors.

For storage, I'd use two systems: PostgreSQL for the product catalog (source of truth) and a vector database for the embeddings. The vector database serves search queries while PostgreSQL handles all other CRUD operations.

For the search flow: when a user types a query, the same embedding service converts it to a vector. The vector database searches its HNSW index with ef=200 to get 100 candidate results. Metadata filters (price range, category, in-stock status) are applied. Then the full product records are fetched from PostgreSQL by ID.

Key concerns: embedding latency (I'd cache recent query embeddings with a TTL), the metadata filtering strategy (pre-filter if the filter is very selective, post-filter otherwise), and index maintenance (re-index every 500K new products).

For scale at 10M products and 5K QPS, I'd need about 4 nodes with 64GB RAM each, sharded by product ID hash. Each node holds its own HNSW index. A coordinator node handles routing and merging."

### L6 — Senior

**Expected to drive the conversation proactively:**
- Raises failure modes before being asked
- Discusses multi-region deployment
- Talks about cost optimization (PQ compression, memory sizing)
- Mentions operational concerns (re-indexing, monitoring)
- Compares alternatives (pgvector, FAISS, Pinecone, self-hosted)

**Example question:** "Design a RAG pipeline for a customer support chatbot."

**Model answer:**
"I'd think about this in four layers: ingestion, retrieval, generation, and feedback.

**Ingestion:** Support documents are chunked into 512-token segments with overlap. Each chunk is embedded using a model fine-tuned on support conversation data — this is critical because general embedding models don't understand support-specific language. Chunks are stored in a vector database with metadata: product version, document type, creation date.

**Retrieval:** I'd use hybrid search — vector similarity for semantic matching and BM25 for keyword matching (product names, version numbers, error codes are handled better by keyword). The two scores are fused with a learned weight. Retrieved chunks go through a re-ranking step using a cross-encoder model on the top 30 results.

**Generation:** The LLM generates answers using the top 5 chunks as context. If none of the chunks have a similarity score above 0.8, the system returns a fallback response: 'I couldn't find specific information. I'll connect you to a human agent.'

**Feedback:** Every answer that gets a thumbs-down or escalates to a human agent is logged. We analyze these cases to identify gaps in the knowledge base and trigger re-embedding of new documents.

**Failure modes I'd design for:** What if the embedding model goes down? Fall back to keyword-only search. What if the vector database gets a bad index? Monitor recall rate and trigger re-index automatically. What if no good matches exist? Return the fallback response rather than hallucinating.

**Cost considerations:** At 1M support documents and 10K QPS, we'd need careful capacity planning. The HNSW index for 1M vectors is about 5GB in memory. With 3x replication for HA, that's 15GB across nodes. Embedding costs: in-house model on GPUs is cheaper than an API-based embedding service at this scale."

### L7/L8 — Staff / Principal

**Expected to discuss org-level impact:**
- How the design affects team structure and ownership
- Migration strategy from legacy search systems
- Vendor evaluation and build-vs-buy analysis
- Business impact of recall degradation

**Example question:** "We're migrating our search from Elasticsearch to a vector database. Design the migration."

**Model answer:**
"I'd approach this as a data-consistency problem. The hard part isn't the vector database — it's maintaining search quality during the transition and avoiding a multi-month rewrite.

**Migration phases:**
1. Shadow mode (month 1): Run both Elasticsearch and the vector database. Route traffic to Elasticsearch but log what the vector DB would have returned. Compare results offline. Fix issues with embedding quality and indexing before cutting over.

2. A/B testing (month 2-3): Route 1% of traffic to the vector DB. Measure: click-through rate, zero-result rate, average position of clicked result. Compare to Elasticsearch baseline. Gradually increase to 50%.

3. Cutover (month 4): Route 100% of search traffic to vector DB. Keep Elasticsearch as a fallback with a 30-second timeout — if the vector DB takes longer, fall back to ES.

4. Decommission (month 5): After confirming no rollbacks needed, archive the Elasticsearch cluster.

**Org considerations:** Who owns the embedding model? That's a new role — ML engineer with NLP focus. Who's on call for the vector database? Add it to the existing search-on-call rotation. Runbook needed for: high latency, low recall, index rebuild failures.

**Risk mitigation:** Design a recall monitoring pipeline that runs 1,000 canonical queries against both systems daily. Alert if vector DB recall drops below 0.90. Pin the embedding model version — don't let the model change silently and shift retrieval behavior."

## The FAANG Framework for Answering

### Step 1: Clarify Requirements (2 minutes)
- "What type of data? Text, images, audio?"
- "What's the scale? Number of vectors, queries per second?"
- "What's the latency requirement? Real-time (under 100ms) or batch?"
- "What accuracy is acceptable? 90% recall or 99%?"
- "Do we need metadata filtering? What kind?"

**What NOT to ask:** "What database should I use?" (you should suggest it). "How should I deploy it?" (that's your job to design).

### Step 2: Estimate Scale (3 minutes)
- "1 million documents × 3 chunks each = 3 million vectors"
- "3 million vectors × 768 dims × 4 bytes = 9.2GB raw + HNSW overhead = ~14GB in memory"
- "100,000 daily active users × 20 searches/day = 2M searches/day = 23 QPS average, ~120 QPS peak"
- "23 QPS × 500KB/sec = 11.5 MB/s query traffic"

### Step 3: High-Level Design (5 minutes)
- Draw the 3-tier architecture: application → embedding service → vector database → storage
- Show the query flow: text → embed → search → filter → fetch → return
- Show the write flow: data → embed → index → store

### Step 4: Deep Dive (5 minutes)
- Pick one component and go deep: HNSW graph structure, the index build process, or metadata filtering
- For HNSW: "Each node has M=16 connections per layer. Top layer has 1% of nodes. Search starts at the entry point, navigates greedily at top layer, then descends..."

### Step 5: Trade-offs (3 minutes)
- "The main trade-off is memory vs recall vs latency. HNSW is fastest but uses the most memory. IVF uses less memory but slower queries. PQ compresses aggressively but loses recall."
- "For persistence, we need a WAL for crash recovery. For availability, we need replication. For cost, we might use PQ compression. Pick based on what's most important for the business."

### Step 6: Wrap Up (2 minutes)
- "The design is optimized for [latency / recall / cost] at the expense of [the other dimension]."
- "If requirements change to [new scenario], I'd change [specific component]."
- "The biggest risk is [embedding quality / index drift / filtering performance], and we mitigate it by [specific strategy]."

## Phrase Bank

1. "The trade-off here is between recall and latency. For this use case, I'd optimize for recall because wrong recommendations lose more revenue than slow ones."

2. "One thing interviewers often want to see is whether you account for index drift — vectors degrade over time, and you need scheduled re-indexing."

3. "The failure mode I'd watch for here is the warm cache miss cascade — when one node restarts and the remaining nodes get overloaded."

4. "I'd use hybrid search — vector search for semantic understanding and BM25 for exact term matching — because our queries contain both natural language and specific product SKUs."

5. "The embedding model is the bottleneck for quality. I'd budget 2 months for embedding model selection and evaluation before committing to the vector database."

6. "For filtering at scale, pre-filtering is efficient only when the filter eliminates most candidates. Post-filtering is simpler but wastes work on vectors that get filtered out anyway."

7. "The vector database is AP in the CAP theorem — available during partitions but with eventual consistency. We need the application to tolerate stale reads."

8. "I'd start with pgvector in PostgreSQL for the first year. Only graduate to a dedicated vector database when we hit 10 million vectors or 1,000 QPS."

9. "The cold start problem is real — new vectors won't appear in search results until the segment is indexed. We need a fallback: keyword search for very fresh data."

10. "Memory sizing is the most common cost surprise. A 100M-vector HNSW index needs ~470GB of RAM. At cloud prices, that's $10,000+/month just for memory."

11. "For production, I'd set up a recall monitoring pipeline — daily A/B tests against brute-force search on a sample of 1,000 queries, tracking recall@10 over time."

12. "The hardest problem in production is combining vector search with complex metadata filters. Test your filtering patterns with your specific workload, not just the vendor's benchmarks."

13. "I'd design for graceful degradation: if the vector database is slow or unavailable, fall back to keyword search. The user gets slightly worse results instead of no results."

14. "For multi-region deployment, writes go to the primary region and replicate asynchronously to replicas. Reads are served locally. The trade-off is 1-2 seconds of replication lag."

15. "Don't over-engineer the index upfront. Start with HNSW defaults (M=16, ef_construction=200), measure recall, and tune one parameter at a time."

## Red Flags

1. **Can't explain what a vector is.** If you can't describe what a vector represents, you don't understand the technology at all.

2. **Treats the vector database as a black box.** If you say "the vector database does similarity search" without any explanation of how, the interviewer knows you've only read marketing materials.

3. **Ignores the embedding step.** If you design a semantic search system without mentioning how data gets turned into vectors, you're missing the most critical component.

4. **No mention of index maintenance.** If you don't bring up re-indexing or index drift, the interviewer knows you've never run a vector database in production.

5. **Equates all vector databases.** If you say "we'll use a vector database" without distinguishing between HNSW/IVF/PQ or between FAISS/Milvus/Pinecone, you lack depth.

6. **No failure mode awareness.** If you design a system that assumes the vector database is always fast and correct, you haven't thought about production realities.

7. **Over-engineering for small scale.** If you propose a distributed vector database for 100,000 vectors, you look inexperienced. "A single PostgreSQL with pgvector would handle this" is a better answer.

8. **No filtering strategy.** If you design a search system without discussing metadata filtering, you haven't thought about how real-world search works — every production query has filters.

## Practice Drills

### Drill 1: 2-Minute Explain
"Explain how HNSW works to someone who knows what a vector is but has never heard of ANN indexes. You have 2 minutes. Go."

**Timer starts now.** Key points you must cover in 2 minutes: multi-layer graph, top-down traversal, greedy navigation at each layer, descending to finer layers, trade-off between build complexity and search speed.

### Drill 2: 5-Minute Design
"You work at a startup building an AI-powered recipe search app. Users search for what they want to cook ('something quick with chicken and broccoli'), and you need to return recipes that match the description even if they use different words. Design the search system. You have 5 minutes."

**Timer starts now.** Key decisions: embedding model (sentence-transformer fine-tuned on recipe data), chunking strategy (recipes have ingredients and instructions as separate chunks), metadata filters (dietary restrictions, prep time, difficulty), fallback for no matches (expand search to similar categories).

### Drill 3: 10-Minute Deep Dive
"Design a system that lets users search across 50 million images using natural language descriptions. You need: a pipeline to index all 50 million images, a real-time search API with p99 under 200ms, and support for filtering by image metadata (date, location, camera type). Go."

**We don't show timer here; you time yourself.** Key components: multimodal embedding model (CLIP or similar), embedding batch processing pipeline (Spark or similar for 50M images), vector database with sharded HNSW, metadata index for filters, async indexing for new images, re-ranking stage for top results.

## Practice Drills

### Drill 1: 2-Minute Explain
"Explain how HNSW works to someone who knows what a vector is but has never heard of ANN indexes."

Key points: HNSW builds a multi-layer graph where each layer is a sparser version of the layer below. Search starts at the top layer (fewest nodes), navigates greedily toward the query vector, then descends to finer layers. Each step expands the candidate set by checking neighbors of the current best node. The hierarchy enables logarithmic time search instead of linear brute-force.

### Drill 2: 5-Minute Design
"Design an AI-powered recipe search app that matches natural language queries like 'something quick with chicken and broccoli' to recipes."

Model answer: Use a sentence-transformer model fine-tuned on recipe data. Index each recipe's title, ingredients, and instructions as separate embedding chunks. Support filtering by prep time (under 30min), dietary restrictions, and ingredients you have. For queries like "something quick with chicken and broccoli", embed the query and search for closest recipe vectors. If no recipe scores above 0.75, relax filters and expand search. Return top 10 results with recipe cards. Scale: 1M recipes, 100 QPS — fits on a single node with pgvector.

### Drill 3: 10-Minute Deep Dive
"Design a system to search 50 million images using natural language across a vector database with strict latency and filtering requirements."

Model answer: Use CLIP for multimodal embeddings (image and text in same space). Batch-process 50M images through CLIP on a Spark cluster (4 GPU nodes, ~2 weeks). Store 512-dim vectors in a 50-shard HNSW index. Each shard holds ~1M vectors in 8GB RAM. Query flow: embed text query → broadcast to all shards in parallel → each returns top 100 → coordinator merges, applies metadata filters (date, location, camera), returns top 20. For latency: use ef=500 on shards, merge with heaps. Cache common query embeddings in Redis. Re-index monthly. p99 budget: 150ms search + 50ms embedding + 20ms merge = 220ms (meets 200ms with optimization).
