# Transactions — Interview Playbook

Transaction questions test a candidate's understanding of correctness, consistency, and recovery in data systems. These questions appear in system design interviews, database design interviews, and debugging rounds.

## How This Topic Appears

Transaction questions come in four categories: core concept, system design with transaction emphasis, debugging, and comparative evaluation.

### Core Concept Questions

These questions test whether the candidate understands the fundamentals of transactions and ACID.

1. "Explain the four ACID properties. What does each guarantee, and what is the cost of each?"

2. "What is the difference between Read Committed, Repeatable Read, and Serializable isolation levels? Give a concrete example of an anomaly prevented by each."

3. "How does a Write-Ahead Log work? Walk through what happens during a COMMIT and what happens during crash recovery."

4. "What is a dirty read? A non-repeatable read? A phantom read?"

### System Design Questions with Transaction Emphasis

5. "Design a banking system. How do you ensure that money is never lost or duplicated?"

6. "Design an inventory reservation system for a flash sale. How do you prevent overselling when 1,000,000 users try to buy 100 items simultaneously?"

7. "Design a payment processing system that integrates with an external payment gateway. How do you handle the case where the gateway times out but the charge actually succeeded?"

8. "Design a multi-region e-commerce platform. How do you handle a user buying the last item in stock from two different regions simultaneously?"

### Debugging and Troubleshooting Questions

9. "Your database throughput is dropping. You notice many transactions are aborted with 'could not serialize access.' What is happening and how do you fix it?"

10. "Your application suddenly cannot insert rows into a table. SELECTs work fine. What could be wrong?"

11. "After a database crash, recovery takes 45 minutes. How do you reduce this to under 5 minutes?"

### Comparative Evaluation Questions

12. "When would you choose a Saga over a distributed transaction?"

13. "Compare PostgreSQL's WAL-based recovery with Oracle's Redo Log-based recovery."

14. "Compare DynamoDB's transactions with PostgreSQL's transactions. When would you use each?"

## What the Interviewer Is Testing

| Skill | What the Interviewer Looks For |
|---|---|
| ACID understanding | Precise definitions of each property, not vague descriptions |
| Isolation level knowledge | Concrete examples of anomalies each level prevents |
| Practical trade-off thinking | Recognition that stronger isolation costs throughput |
| Recovery understanding | Deep knowledge of WAL, REDO, UNDO, checkpointing |
| Debugging ability | Systematic approach to transaction failures |
| Design judgment | Choosing the right isolation level for the workload |
| System perspective | Recognizing when transactions are the wrong tool |

## Level Expectations

### L3 / Entry Level

The candidate should explain the four ACID properties with examples. They should describe the difference between Read Committed and Serializable. They should write a correct BEGIN/COMMIT block for a bank transfer.

**Sample answer for "Explain ACID":**

> Atomicity means the transaction either fully completes or fully rolls back. In a bank transfer, if the debit succeeds and the credit fails, the debit is rolled back. Consistency means the transaction preserves database invariants — the sum of balances across all accounts is the same before and after a transfer. Isolation means concurrent transactions do not interfere — one transaction does not see another transaction's uncommitted changes. Durability means committed changes survive a crash — the database flushes the WAL to disk before returning COMMIT OK.

### L4 / Mid-Level

The candidate should describe isolation levels with concrete anomaly scenarios. They should explain WAL recovery phases. They should discuss performance implications of different isolation levels.

**Sample answer for "Walk through what happens during COMMIT":**

> When COMMIT is issued, the database writes a COMMIT record to the WAL buffer, then calls fsync to flush the WAL buffer to disk. When fsync returns, the transaction is durable. The database then releases all locks held by the transaction and marks the transaction as committed in the commit log. The dirty pages in the buffer pool are not flushed to disk at this point — that happens later during a checkpoint. If the server crashes before the fsync, the transaction is lost. If it crashes after the fsync but before the checkpoint, the WAL replay reconstructs the changes.

### L5 / Senior

The candidate should discuss trade-offs between isolation levels with specific performance numbers. They should explain the ARIES recovery algorithm. They should design a transaction system for a specific workload, including failure handling.

**Sample answer for "Design a banking system":**

> For the core ledger, I would use PostgreSQL at Serializable isolation. Every transfer is a single transaction: BEGIN, debit source, credit destination, COMMIT. The account row is small and accessed by primary key, so contention is minimal for most transactions. However, hot accounts could be a problem. For a high-volume merchant account processing thousands of debits per hour, I would split it into sub-accounts and aggregate. I would use optimistic concurrency control: read the current version, compute the new balance, UPDATE WHERE version = old_version. If the UPDATE affects zero rows, retry. This avoids explicit locking and handles contention gracefully.

> For crash recovery, I would measure recovery time: WAL size from last checkpoint divided by replay speed. I would set checkpoint_timeout to keep WAL under 1 GB, ensuring recovery under 10 seconds with NVMe storage. I would test crash scenarios monthly by simulating a power failure.

### L6 / Staff

The candidate should discuss distributed transaction coordination, including 2PC, 3PC, and Sagas. They should explain CAP implications for transactions. They should design a multi-region transaction system.

**Sample answer for "Design a multi-region e-commerce platform":**

> I would use a CP database for inventory within each region — CockroachDB or Spanner. The inventory row for each SKU is geo-partitioned to the region where the item is stored. For last-item-in-stock scenarios across regions, I would use a global inventory counter with a distributed transaction: PREPARE on both region shards, then COMMIT. This adds latency (2x RTT across regions) but prevents overselling.

> For the ordering workflow, I would not put the entire order in a single distributed transaction. The order processing involves multiple steps — inventory check, payment, fulfillment. I would use a Saga: each step has its own local transaction and a compensating action. The Saga coordinator (Temporal or a custom service) tracks state and executes compensations on failure.

### L7 / Principal

The candidate should discuss the theoretical limits of distributed transactions, the implications of the FLP impossibility result, and how to build systems that provide transaction-like guarantees at internet scale.

**Sample answer for "What are the fundamental limits of distributed transactions":**

> The FLP impossibility result shows that in an asynchronous distributed system, consensus cannot be guaranteed in bounded time. This limits distributed transactions: 2PC blocks when the coordinator fails. 3PC avoids blocking but requires bounded network delay, which is unrealistic across regions. Spanner's TrueTime is the most practical approach: it uses clock synchronization to minimize latency while providing Serializable isolation. But TrueTime is expensive — it requires atomic clocks and GPS receivers in every datacenter. CockroachDB's HLC approach is a cheaper alternative but provides weaker guarantees.

> The practical solution is to minimize the need for distributed transactions: geo-partition data so that most transactions are local. For the rare cross-region transaction, accept the latency of 2PC or use a Saga.

## FAANG Framework for Transaction Questions

When a transaction-related question appears, follow this framework:

### Step 1: Identify Transaction Boundaries

Determine which operations must be atomic. A bank transfer needs one transaction wrapping both db
its and credits. An order placement needs one transaction wrapping inventory deduction and order creation.

Ask: "Which operations in this system cannot be partially visible?"

### Step 2: Choose the Isolation Level

Select the minimum isolation level that prevents the relevant anomalies.

| Workload | Isolation Level | Why |
|---|---|---|
| Bank transfer | Serializable | No anomaly is acceptable |
| E-commerce checkout | Repeatable Read | Prevent lost updates on inventory |
| Social media post | Read Committed | Stale reads are acceptable |
| Analytics dashboard | Read Uncommitted | Approximate values are fine |

### Step 3: Design for Failure

Identify every failure mode and design the transaction handling for each:

- **Transaction failure**: Implement retry with exponential backoff for serialization failures.
- **Network failure**: Use idempotency keys to prevent duplicate charges.
- **Server crash**: Ensure WAL flush happens before COMMIT returns.
- **Replica lag**: Route read-your-writes queries to the primary.
- **Distributed failure**: Use Saga with compensating transactions for multi-service operations.

### Step 4: Handle Edge Cases

- What if the transaction is too large? Split it into smaller transactions.
- What if contention is too high? Use optimistic concurrency control or shard hot rows.
- What if the application cannot tolerate serialization failures? Use a weaker isolation level with application-level locking.
- What if recovery time is too long? Reduce checkpoint intervals or increase hardware speed.

## Phrase Bank

Use these phrases in interviews to demonstrate depth.

### On Transaction Boundaries

- "The transaction boundary should be as narrow as possible while still providing atomicity for the complete operation."
- "Operations that require atomicity should be in the same transaction. Operations that don't should be separate."
- "A transaction should not span user interactions — the cost of holding locks while a user decides is too high."

### On Isolation Level Selection

- "Read Committed is the default for a reason: it prevents dirty reads while maintaining high concurrency."
- "Serializable is necessary when even phantom reads could cause incorrect behavior — the sum of account balances must be consistent."
- "Repeatable Read provides a consistent snapshot for reporting without the overhead of predicate locking."

### On Durability

- "The fsync at COMMIT is the single most expensive operation in a transaction."
- "Group commit batches multiple fsyncs into one call, dramatically increasing throughput under high concurrency."
- "Setting synchronous_commit = off sacrifices at most one second of data for a 10x throughput improvement."

### On Crash Recovery

- "The WAL is the source of truth for crash recovery. If the WAL says committed, the data survives."
- "Recovery time is bounded by the WAL size from the last checkpoint. Frequent checkpoints reduce recovery time but increase I/O overhead."
- "ARIES is the standard recovery algorithm: REDO all changes from the last checkpoint, then UNDO uncommitted transactions."

### On Distributed Transactions

- "Two-phase commit guarantees atomicity across multiple databases at the cost of a blocking coordinator."
- "The Saga pattern trades isolation for availability — intermediate states are visible, but the system guarantees eventual consistency."
- "Geo-partitioning reduces the need for distributed transactions by keeping related data in the same region."

## Red Flags

These mistakes signal to the interviewer that the candidate does not understand transactions deeply.

### Red Flag 1: "I would use a transaction for everything."

Transactions are not free. Using transactions for read-only workloads, single-row operations, or cache updates adds unnecessary overhead. Show judgment about when transactions are appropriate.

### Red Flag 2: "Serializable is always the safest choice."

Serializable is the safest for correctness but produces more aborts under contention. The candidate should recognize that higher isolation levels can reduce throughput and increase retry rates, potentially reducing overall correctness.

### Red Flag 3: "COMMIT writes data to disk."

COMMIT writes WAL to disk, not data pages. The data pages are written later by the background writer or checkpoint. This distinction matters for understanding crash recovery and performance.

### Red Flag 4: "Transactions prevent all concurrency problems."

Transactions prevent ACID-related anomalies but do not prevent all concurrency problems. Deadlocks, serialization failures, and lock waits still occur. Transactions must be combined with retry logic, lock timeouts, and deadlock detection.

### Red Flag 5: "Two-phase commit solves distributed transaction problems perfectly."

Two-phase commit has the blocking problem: if the coordinator crashes, participants block indefinitely. The candidate should mention this limitation and discuss alternatives (three-phase commit, Paxos-based commit, Saga).

### Red Flag 6: "The isolation level is just a setting."

Different isolation levels have very different performance characteristics. Serializable can be 2-10x slower than Read Committed under contention. The candidate should discuss the performance implications of each level.

### Red Flag 7: "I set isolation level to Serializable and never worry about anomalies."

Serializable prevents database-level anomalies but does not prevent application-level anomalies. If the application reads data, computes a value outside the database (e.g., checking a balance in application code), and then writes, the operation is not atomic from the database's perspective.

### Red Flag 8: "Crash recovery always works automatically."

Crash recovery usually works automatically, but candidates should understand when it fails: corrupted WAL segments, full disk that prevents WAL writes, or XID wraparound. Monitoring and testing are required to ensure recovery works when needed.

## Sample Walkthrough: "Design a Banking System"

This is a classic interview question that tests transaction knowledge. Here is a complete walkthrough.

**Interviewer:** "Design a system for a bank that handles 10 million accounts and processes 5,000 transfers per second."

**Candidate walkthrough:**

> Let me start by identifying the requirements. The core operation is a transfer of money between accounts. This must be atomic — either both the debit and credit succeed, or neither does. The system must handle concurrent transfers on the same account, and must recover correctly from crashes.
>
> **Transaction boundaries:** Each transfer is a single database transaction. The transaction includes: checking the source account has sufficient funds, debiting the source, crediting the destination, and logging the transfer.
>
> **Isolation level:** I would use Serializable or Repeatable Read. Serializable is safest but produces aborts under contention. For a banking system, correctness is paramount, so I would start with Serializable and add retry logic.
>
> **Atomic operation:** I will combine the balance check with the debit in a single UPDATE statement. This is critical. A separate SELECT followed by UPDATE has a race condition — two concurrent transactions could both read the same balance and both believe there are sufficient funds.
>
> ```sql
> BEGIN ISOLATION LEVEL SERIALIZABLE;
>
> -- Atomically check balance and debit
> UPDATE accounts
> SET balance = balance - 100
> WHERE id = 'source-123'
>   AND balance >= 100;
>
> -- If zero rows affected, insufficient funds
> -- ROLLBACK and return error
>
> UPDATE accounts
> SET balance = balance + 100
> WHERE id = 'dest-456';
>
> INSERT INTO transfer_log (from_id, to_id, amount, status)
> VALUES ('source-123', 'dest-456', 100, 'completed');
>
> COMMIT;
> ```
>
> **Handling contention:** A hot account (e.g., a merchant account receiving thousands of payments per second) would cause serialization failures. I would handle this by:
> 1. Retrying failed transactions with exponential backoff.
> 2. Splitting hot accounts into sub-accounts (100 sub-accounts, each with 1/100th the balance). Transfers are distributed across sub-accounts.
> 3. Using optimistic locking with version numbers instead of Serializable isolation.
>
> **Crash recovery:** After a crash, the database replays the WAL. Any transfer that wrote a COMMIT record to the WAL before the crash is recovered. Any transfer that didn't is rolled back. The client application must check the status of a transfer after a crash timeout and retry if necessary.
>
> **Scaling:** For 5,000 TPS with 1 KB WAL records, I need approximately 50 MB/s of WAL write bandwidth, plus fsync overhead. A single NVMe drive can handle this. For higher throughput, I would shard by account ID across multiple database instances. Sharding eliminates hot spots because each account is in one shard, and transfers between accounts in different shards require a distributed transaction or a Saga.
>
> **Read-your-writes:** After a transfer, the user checks their balance. This read must go to the primary database, not a replica, to guarantee they see the updated balance. I would route reads by context: balance reads after a transfer go to the primary; historical statement reads can go to replicas.

## Common Follow-Up Questions

Interviewers rarely stop at the first answer. These are common follow-ups and how to handle them.

**Follow-up 1: "What if the bank transfer needs to call an external service (fraud check, notification)?"**

The external call should NOT be inside the database transaction. The transaction should handle only the database writes. Call the external service after COMMIT. If the external service fails, the transaction has already committed, so use a compensating transaction.

Better design: use the transactional outbox pattern. Write the transfer data and a "pending notification" record in the same transaction. A background worker reads pending notifications and calls the external service. This decouples the transaction from the external call.

**Follow-up 2: "What if two transactions try to transfer from the same account simultaneously?"**

With Serializable isolation, one transaction will be aborted. The application must retry. The key insight: the UPDATE with a WHERE balance >= amount check is atomic. Even with Read Committed, the UPDATE statement itself is atomic — only one transaction can update the row at a time. However, with Read Committed, a second transaction might read stale data for other computations. Serializable prevents all anomalies.

**Follow-up 3: "How do you handle the case where the client crashes after receiving COMMIT OK but before displaying the result to the user?"**

The transfer is durable (COMMIT OK was returned). When the user logs in again, the application fetches the current balance from the database, which reflects the completed transfer. The transaction log provides an audit trail. No correction is needed because the transfer completed successfully.

**Follow-up 4: "How do you detect and resolve deadlocks?"**

Deadlocks occur when Transaction A holds a lock on row 1 and waits for row 2, while Transaction B holds a lock on row 2 and waits for row 1. The database's deadlock detector runs periodically (every 1 second by default in PostgreSQL). When detected, it aborts one transaction. The application must retry.

To minimize deadlocks, all transactions should acquire locks in the same order. In a bank transfer, always lock the lower account ID first:

```sql
BEGIN;
-- Lock accounts in ID order to prevent deadlocks
UPDATE accounts SET balance = balance - 100
WHERE id = LEAST('source-123', 'dest-456');
UPDATE accounts SET balance = balance + 100
WHERE id = GREATEST('source-123', 'dest-456');
COMMIT;
```

### Advanced Question: "How does MVCC work and why does it matter for transactions?"

This question tests depth. MVCC is the mechanism that allows readers and writers to coexist without blocking each other.

> Each transaction sees a snapshot of the data as of the start of the transaction (or the start of the query in Read Committed). The snapshot is constructed from the database's internal transaction state: the system records which transactions were active at the snapshot time.
>
> When a transaction reads a row, it checks the row's header for the XID that created it and the XID that deleted it. If the creating XID committed before the snapshot, the row is visible. If the deleting XID committed before the snapshot, the row is not visible. Old row versions are kept in the data pages until VACUUM removes them.
>
> MVCC matters for transactions because it enables the isolation property without blocking. In PostgreSQL, Serializable isolation still uses MVCC as the foundation, but adds a serialization dependency graph to detect conflicts. The key trade-off: MVCC requires more storage (old row versions persist) and VACUUM overhead, but it eliminates read-write contention.

### Advanced Question: "How would you handle a hot-spot account with millions of transactions per hour?"

This tests the candidate's ability to handle real-world contention.

> A single row can only be updated serially — the database serializes updates to the same row through locking. With millions of transactions per hour targeting the same account, that row becomes a bottleneck.
>
> There are three approaches:
>
> 1. **Sub-account sharding**: Split the hot account into N sub-accounts, each with its own balance row. Distribute incoming transactions across sub-accounts. Read the total by summing all sub-accounts. This converts a single hot row into N cooler rows.
>
> 2. **Batching**: Accumulate transactions in a buffer table (INSERT-only, no contention) and batch-update the hot account periodically. A background process aggregates the buffer into a single UPDATE per batch window.
>
> 3. **External counter**: Use Redis for the hot counter, with periodic reconciliation to the database. Redis can handle 100,000+ atomic increments per second on a single key. The database stores the authoritative balance, and Redis serves as a fast cache with eventual consistency.
>
> The right choice depends on the accuracy requirements. Sub-account sharding provides the strongest consistency. Batching provides high throughput with delayed visibility. External counter provides the highest throughput with the weakest consistency.

### Key SQL to Memorize

```sql
-- Transaction control
BEGIN ISOLATION LEVEL SERIALIZABLE;
COMMIT;
ROLLBACK;
SAVEPOINT sp1;
ROLLBACK TO sp1;

-- Atomic balance check and debit
UPDATE accounts SET balance = balance - 100 WHERE id = 1 AND balance >= 100;

-- Row-level locking
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
SELECT * FROM accounts WHERE id = 1 FOR SHARE;

-- Skip locked for queue processing
SELECT * FROM job_queue ORDER BY id LIMIT 1 FOR UPDATE SKIP LOCKED;
```
