# Transactions in Production

## Company Case Studies

### Stripe: Transactional Integrity for Payments

Stripe processes billions of dollars in payments every day. Every single payment is a transaction with full ACID guarantees. Stripe's transaction architecture is built on a custom sharded MySQL layer, but the principles apply to any ACID-compliant database.

Each payment goes through a carefully designed transaction boundary:

```sql
-- Stripe's payment transaction (simplified)
BEGIN;
-- 1. Reserve funds from the customer
UPDATE accounts SET balance = balance - 50.00 WHERE id = 'customer-xyz';
-- 2. Record the pending charge
INSERT INTO charges (id, amount, status, created) VALUES ('ch_123', 50.00, 'pending', NOW());
-- 3. Create the balance transaction
INSERT INTO balance_transactions (id, amount, type) VALUES ('bt_456', 50.00, 'charge');
COMMIT;
```

If any step fails, the entire payment is rolled back. The customer's money never leaves their account. The merchant never sees a phantom charge.

Key lessons from Stripe's approach:

- **Small transaction boundaries**: Each payment is its own transaction. No transaction touches more than one customer and one merchant. This minimizes lock contention and maximizes throughput.
- **Idempotency**: Stripe's API supports idempotency keys. If a client retries a payment request, Stripe detects the duplicate and returns the existing result rather than creating a second charge. This is critical because a transaction that commits but whose network response is lost will otherwise appear as a failed payment to the client.
- **Audit trail via WAL**: Stripe uses the WAL as a source of truth for auditing. Every financial transaction is recorded in the WAL before it is visible. Auditors can replay the WAL to verify the integrity of any period.
- **Read-your-writes routing**: After a payment, Stripe routes subsequent balance reads to the primary shard, not a replica. This ensures the merchant always sees their updated balance immediately.

### Uber: Transaction Coordination with Cadence / Temporal

Uber built Cadence (now open-sourced as Temporal) to handle transaction coordination across microservices. A ride from pickup to drop-off involves payment, driver payout, surge pricing, routing, and ratings — all of which must be coordinated.

```python
# Temporal workflow for a ride (conceptual)
@workflow.defn
class RideWorkflow:
    @workflow.run
    async def run(self, ride_id, user_id, driver_id, fare_amount):
        # Step 1: Charge the rider
        charge_result = await workflow.execute_activity(
            charge_rider, user_id, fare_amount,
            start_to_close_timeout="10s"
        )
        # Step 2: Payout the driver (compensation: refund rider)
        payout_result = await workflow.execute_activity(
            pay_driver, driver_id, fare_amount * 0.75,
            start_to_close_timeout="10s"
        )
        # Step 3: Update ride status
        await workflow.execute_activity(
            complete_ride, ride_id,
            start_to_close_timeout="5s"
        )
```

Uber's Cadence/Temporal replaces distributed transactions with a workflow engine that provides:

- **Durable execution**: The workflow state is persisted. If the workflow engine crashes, it resumes from the last completed step.
- **Automatic retries**: Failed activities (e.g., payment service is down) are retried with configurable backoff.
- **Compensation**: If an activity fails, compensating activities undo previous steps.
- **Visibility**: The workflow execution can be queried at any time to see its progress.

This is the Saga pattern implemented at production scale. Uber runs millions of workflow executions per day across hundreds of microservices.

### Netflix: From DB Transactions to Event-Driven Sagas

Netflix's billing system originally used MySQL transactions for everything: charge the customer, update the subscription, send a receipt, update analytics. As the system grew to 200+ million subscribers, the transaction boundaries became bottlenecks.

Netflix migrated to an event-driven Saga architecture:

The migration traded ACID isolation for massive scalability. Netflix processes over 1,000 billing transactions per second during peak periods. An ACID transaction holding locks for even 100 ms would create unacceptable contention. With Sagas, each service independently manages its own data, and eventual consistency is acceptable for billing operations (the downstream reconciliation system catches any inconsistencies within minutes).

### Google Spanner: TrueTime + 2PC for Global Transactions

Google Spanner is the most ambitious implementation of global transactions. It provides Serializable isolation across data centers on a planetary scale using two technologies:

1. **TrueTime**: Google's global clock service using GPS and atomic clocks. TrueTime exposes a time interval [earliest, latest] during which the clock is guaranteed to contain the actual time. This allows Spanner to assign globally meaningful timestamps to transactions.

2. **Two-Phase Commit**: Spanner uses 2PC across Paxos groups (each group is a set of replicas that use the Paxos consensus protocol). The transaction coordinator sends PREPARE to all participating Paxos groups, each group reaches consensus internally, and the coordinator commits.

The critical insight: TrueTime eliminates the need for centralized clock synchronization. Each Spanner server runs a TrueTime daemon that continuously calibrates the clock. Transactions use TrueTime timestamps to determine serialization order without any centralized lock manager.

| Aspect | Conventional 2PC | Spanner 2PC |
|---|---|---|
| Clock sync | NTP (best effort, ~10ms error) | TrueTime (bounded, 1-7ms error) |
| Lock duration until | All participants respond | All Paxos groups commit |
| Failure handling | Coordinator is SPOF | Paxos ensures no SPOF |
| Key innovation | Standard protocol | TrueTime eliminates centralized clock sync |

Spanner achieves 1-10 millisecond write latency across regions and 100,000+ transactions per second globally, with full Serializable isolation. It is the gold standard for global ACID transactions.

## Open Source Technologies

### PostgreSQL (ACID Leader)

PostgreSQL is the gold standard for open-source ACID transactions. It implements:

- **MVCC** (Multiversion Concurrency Control): Each transaction sees a snapshot of data as of the start of its first query. Readers never block writers, and writers never block readers.
- **Four isolation levels**: Read Uncommitted (behaves like Read Committed), Read Committed (default), Repeatable Read (snapshot isolation), and Serializable (SSI — Serializable Snapshot Isolation).
- **SSI**: PostgreSQL's Serializable implementation uses Serializable Snapshot Isolation, which detects conflicts using a dependency graph of read-write conflicts. When a serialization anomaly would occur, PostgreSQL aborts one of the conflicting transactions.
- **WAL**: PostgreSQL's WAL is the foundation of its durability, replication, and point-in-time recovery capabilities.

### MySQL / InnoDB

MySQL with InnoDB provides ACID transactions with:

- **MVCC**: Similar to PostgreSQL, but with some differences in how snapshot isolation works.
- **Isolation levels**: Read Uncommitted, Read Committed (default), Repeatable Read, Serializable.
- **Undo log**: InnoDB uses an undo log (stored in the undo tablespace) rather than directly embedding undo information in WAL records.
- **Group commit**: MySQL's group commit batches fsync calls for higher throughput.

The main difference from PostgreSQL: MySQL's Repeatable Read prevents phantom reads (via gap locking), which PostgreSQL does not guarantee at Repeatable Read unless using SSI.

### SQLite

SQLite provides ACID transactions for embedded use cases:

- **Single-writer**: SQLite allows only one writer at a time. Concurrent reads are allowed.
- **WAL mode**: In WAL mode, SQLite uses a write-ahead log for concurrent reads and writes. This is the recommended mode for most applications.
- **Atomic commit**: SQLite uses a rollback journal (by default) or WAL to implement atomic commit.
- **Performance**: On an embedded device, SQLite handles 50,000+ transactions per second in WAL mode.

### CockroachDB

CockroachDB implements distributed transactions without a central coordinator:

- **Timestamp ordering**: Transactions are ordered by a global timestamp based on HLC (Hybrid Logical Clocks, similar to TrueTime but without atomic clocks).
- **Two-phase commit with parallel commits**: CockroachDB accelerates 2PC by allowing parallel writes during the prepare phase, reducing commit latency.
- **Serializable isolation by default**: CockroachDB's default isolation level is Serializable (actually Serializable Snapshot Isolation, similar to PostgreSQL).
- **Geo-partitioning**: Data is partitioned by geographic region, and transactions that stay within a region avoid cross-region latency.

### Temporal (Saga Orchestration)

Temporal (originally Cadence from Uber) is an open-source workflow engine for the Saga pattern:

- **Durable execution**: Workflow state is persisted to Cassandra, PostgreSQL, or MySQL.
- **Activity retries**: Failed activities are retried automatically.
- **Signal handling**: Workflows can wait for external signals (e.g., a payment confirmation webhook).
- **Multi-language SDK**: Go, Java, Python, TypeScript.

## Cloud Providers

### AWS Aurora

Aurora is AWS's MySQL/PostgreSQL-compatible database:

- **Distributed storage**: The WAL is written to a custom distributed storage system (6 copies across 3 availability zones).
- **Crash recovery**: Aurora does not replay WAL during crash recovery. Instead, the storage layer uses the WAL to reconstruct pages on demand. Recovery is sub-second regardless of WAL size.
- **Throughput**: Aurora handles 100,000+ transactions per second on the largest instances.

### Google Cloud Spanner

As discussed above, Spanner provides global ACID transactions with:

- **Serializable isolation**: The default (and only) isolation level is Serializable.
- **TrueTime**: Global consistent timestamps without centralized coordination.
- **Automatic sharding**: Data is automatically rebalanced across nodes.
- **99.999% availability SLA**: Five nines for multi-region configurations.

### Azure SQL Database

Microsoft Azure's managed SQL Server offering:

- **Snapshot isolation**: Azure SQL uses RCSI (Read Committed Snapshot Isolation) by default.
- **Elastic transactions**: Transactions across multiple Azure SQL databases using distributed transactions.
- **Hyperscale tier**: Separates transaction log from data storage, allowing 100 TB+ databases with fast transaction commit.

## Lessons Learned

### 1. Keep Transactions Short

A transaction should hold locks for the shortest possible time. Every millisecond a transaction holds a lock is a millisecond another transaction might be blocked.

```sql
-- GOOD: Short transaction
BEGIN;
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 42 AND quantity > 0;
INSERT INTO orders (product_id, user_id) VALUES (42, 100);
COMMIT;

-- BAD: Long transaction (sends email inside the transaction)
BEGIN;
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 42;
INSERT INTO orders (product_id, user_id) VALUES (42, 100);
-- Sending email takes 500ms, holding locks the entire time
CALL send_email_in_db_function(100, 'Your order confirmed');
COMMIT;
```

### 2. Measure TPS Before Optimizing

Many teams optimize transaction throughput without measuring whether the bottleneck is actually transactions. Measure TPS with a realistic workload before tuning parameters.

```bash
# Use pgbench to measure baseline TPS
pgbench -c 50 -j 2 -T 60 -P 5 bank
# -c 50: 50 concurrent clients
# -j 2: 2 threads
# -T 60: run for 60 seconds
# -P 5: report progress every 5 seconds
```

### 3. Monitor Long-Running Transactions Religiously

Set up monitoring and alerting for:

- Transactions running longer than 5 seconds (potential lock contention).
- Idle-in-transaction connections (locks held while doing nothing).
- Prepared transactions (orphans from distributed transactions).
- Transaction ID age (approaching wraparound).

### 4. Test Crash Recovery Scenarios

The only way to know if transaction recovery works is to test it.

```bash
# Simulate a crash and verify recovery
pg_ctl -D /var/lib/postgresql/data stop -m immediate
pg_ctl -D /var/lib/postgresql/data start
# Check the logs for recovery messages
tail -100 /var/log/postgresql/postgresql.log | grep -E "LOG:|recovery"
```
