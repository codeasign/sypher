# Transactions at Scale

## Naive Implementation

### Single-Disk, No Transaction Protection

The simplest possible data system: a single application writes directly to a file on disk.

```python
# Naive bank transfer — no transaction protection
def transfer(from_account, to_account, amount):
    data = json.load(open("accounts.json"))
    if data[from_account]["balance"] < amount:
        raise InsufficientFunds()
    data[from_account]["balance"] -= amount
    data[to_account]["balance"] += amount
    json.dump(data, open("accounts.json", "w"))
```

At 100 concurrent users, this fails in predictable ways:

- **Crash during write**: The file is truncated and partially written. Corrupted.
- **Concurrent writes**: Two processes overwrite each other. Race conditions.
- **Read during write**: A reader sees the file mid-write. Half the data is visible.

Adding a primitive lock file helps slightly but introduces its own problems. If the application crashes while holding the lock, the lock file persists and blocks all future operations until manually removed.

### Adding a Database — Still Naive

Even with a database, transactions must be used correctly. A common mistake is wrapping each individual operation in its own transaction, losing atomicity for multi-step business operations.

```python
# Bad: Each operation is its own transaction
def transfer(from_account, to_account, amount):
    db.execute("BEGIN")
    db.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?",
               (amount, from_account))
    db.execute("COMMIT")
    # Crash happens HERE
    db.execute("BEGIN")
    db.execute("UPDATE accounts SET balance = balance + ? WHERE id = ?",
               (amount, to_account))
    db.execute("COMMIT")
```

This is the same problem as the file-based approach. The two operations are not wrapped in a single transaction — they are two separate transactions. A crash between the two commits leaves the accounts in an inconsistent state.

### Connection Pooling

Every database connection consumes resources: memory for the connection state, a backend process in PostgreSQL, and overhead for authentication. At scale, maintaining thousands of idle connections is wasteful.

PgBouncer pools database connections. The application opens a connection to PgBouncer, which maps it to one of a smaller pool of persistent database connections. PgBouncer supports three pooling modes:

| Mode | Connection Reuse | Transaction Reuse | Use Case |
|---|---|---|---|
| Session | Per session | Per session | Long-lived connections |
| Transaction | Per transaction | No | Web apps, short txns |
| Statement | Per statement | No | Reporting, batch jobs |

The transaction pooling mode is the most common for web applications: PgBouncer assigns a database connection at BEGIN and releases it at COMMIT. This allows 1000 application connections to share 50 database connections.

### Read Replicas for Transaction Offloading

Long-running or read-intensive transactions should be routed to read replicas, leaving the primary free to process write transactions.

```python
import psycopg2

# Write transactions go to the primary
primary = psycopg2.connect("host=primary.example.com dbname=bank")
primary.autocommit = False

def transfer(from_account, to_account, amount):
    with primary:
        cur = primary.cursor()
        cur.execute("BEGIN")
        cur.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?", (amount, from_account))
        cur.execute("UPDATE accounts SET balance = balance + ? WHERE id = ?", (amount, to_account))
        cur.execute("COMMIT")

# Read-only queries go to replicas
replica = psycopg2.connect("host=replica.example.com dbname=bank")

def get_balance(account_id):
    cur = replica.cursor()
    cur.execute("SELECT balance FROM accounts WHERE id = ?", (account_id,))
    return cur.fetchone()[0]
```

The caveat: replication lag means the replica may return stale data. For a bank balance, this is unacceptable. Route read-your-writes queries back to the primary.

## Integration Patterns

### Transactional Outbox Pattern

The outbox pattern ensures that database changes and messages (events, queue messages) are sent atomically. Instead of sending a message directly, the application writes the message to an outbox table in the same transaction as the data change. A separate process reads the outbox and sends the messages.

```sql
BEGIN;
-- Insert the order
INSERT INTO orders (user_id, total, status) VALUES (42, 99.99, 'confirmed');
-- Insert the outbox message in the same transaction
INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload)
VALUES ('order', currval('orders_id_seq'), 'order.confirmed', '{"user_id":42,"total":99.99}');
COMMIT;
-- Separate process (e.g., Debezium, pg_outbox) reads the outbox
-- and publishes the event to Kafka or SQS
```

### Transactional Cache Invalidation

Caches must be invalidated or updated when the underlying data changes. Without transactional guarantees, a cache can serve stale data after a partial update.

```python
def update_balance(account_id, new_balance):
    with db.transaction():
        db.execute("UPDATE accounts SET balance = ? WHERE id = ?",
                   (new_balance, account_id))
        cache.delete(f"balance:{account_id}")
```

By deleting the cache key inside the transaction, the cache is guaranteed to be consistent with the database. If the transaction rolls back, the cache delete is also rolled back (or the application handles the cache deletion separately).

### Saga Pattern for Microservices

When a transaction spans multiple services, each with its own database, a distributed transaction is the wrong choice. The Saga pattern provides a scalable alternative.

```python
# Saga orchestration for ride payment
class RideSaga:
    def execute(self, ride_id, user_id, driver_id, amount):
        saga_id = str(uuid.uuid4())
        try:
            self.step("charge_user", saga_id, lambda: payment.charge(user_id, amount))
            self.step("pay_driver", saga_id, lambda: payment.credit(driver_id, amount * 0.8))
            self.step("update_ride_status", saga_id, lambda: ride.complete(ride_id))
        except Exception as e:
            log.error(f"Saga {saga_id} failed: {e}")
            self.compensate(saga_id)
            raise

    def compensate(self, saga_id):
        # Reverse each completed step in reverse order
        for step in reversed(self.completed_steps[saga_id]):
            step.compensate()

    def step(self, name, saga_id, action, compensation=None):
        action()
        self.completed_steps[saga_id].append(
            Step(name, compensation)
        )
```

### Transactional Event Publishing

A common pattern in modern architectures is publishing events (to Kafka, RabbitMQ, or SQS) as part of a database transaction. The naive approach has a dual-write problem:

```python
# BAD: Dual-write problem
def create_order(user_id, product_id):
    with db.transaction():
        db.execute("INSERT INTO orders ...")
    # Crash happens between these two lines
    kafka.produce("order.created", {"user_id": user_id, "product_id": product_id})
```

If the application crashes after the database commit but before the Kafka produce, the event is lost. Downstream consumers never learn about the order.

The correct approach is the transactional outbox pattern (described above) or **transactional event publishing** using PostgreSQL's `NOTIFY`:

```sql
-- Use LISTEN/NOTIFY for transactional event publishing
BEGIN;
INSERT INTO orders (user_id, product_id, status) VALUES (42, 101, 'confirmed');
PERFORM pg_notify('order_events', '{"type":"order.created","order_id":123}');
COMMIT;
-- The notification is only sent if the transaction commits
-- If the transaction rolls back, the notification is never sent
```

The `pg_notify` function enqueues the notification in shared memory. After COMMIT, the notification is sent to all connected listeners. If the transaction rolls back, the notification is discarded. This guarantees that the event is published if and only if the transaction commits.

### Transactional Cache Invalidation with Write-Through

A more sophisticated cache invalidation approach uses a write-through cache pattern:

```python
def update_balance_with_cache(account_id, new_balance):
    # Step 1: Update the database
    with db.transaction():
        db.execute("UPDATE accounts SET balance = ? WHERE id = ?",
                   (new_balance, account_id))
        # Step 2: Mark cache as invalid (inside the transaction)
        db.execute("INSERT INTO cache_invalidations (key) VALUES (?)",
                   (f"balance:{account_id}",))

# Background process reads cache_invalidations and evicts from Redis
def process_cache_invalidations():
    while True:
        rows = db.execute("SELECT key FROM cache_invalidations LIMIT 100")
        for row in rows:
            redis.delete(row.key)
            db.execute("DELETE FROM cache_invalidations WHERE key = ?",
                       (row.key,))
```

This approach guarantees that the cache invalidation is never lost. If the background process crashes, it picks up the unprocessed invalidations on restart. The cache_invalidations table acts as a durable queue.

### Two-Phase Commit (2PC)

The standard protocol for distributed transactions across multiple databases or shards:

1. **Prepare phase**: The coordinator sends a PREPARE request to all participants. Each participant writes the transaction's WAL records (making the transaction durable but not visible) and votes READY or ABORT.

2. **Commit phase**: If all participants voted READY, the coordinator sends COMMIT. If any voted ABORT, the coordinator sends ABORT. Each participant completes the commit or abort.

The critical problem with 2PC: if the coordinator crashes after sending PREPARE but before sending COMMIT, the participants hold locks and WAL space for the prepared transaction indefinitely. This is called the "blocking problem." The prepared transaction must be manually resolved or the coordinator must be recovered.

### Synchronous vs. Asynchronous Replication

| Aspect | Synchronous | Asynchronous |
|---|---|---|
| Durability | Transaction commits only after replica confirms | Transaction commits immediately on primary |
| Latency | COMMIT waits for replica Acknowledgment | COMMIT returns as fast as local fsync |
| Data loss on primary failure | Zero (replica has all data) | Up to seconds or minutes of data |
| Availability | Lower (replica must be available) | Higher (replica can be down) |

## Capacity Planning

### Maximum TPS Calculation

Transaction throughput is fundamentally limited by WAL write bandwidth.

```
max TPS = WAL write throughput / WAL write size per transaction

Given:
  WAL write bandwidth:   100 MB/s  (NVMe SSD sequential write)
  WAL write per txn:     1 KB     (simple UPDATE, small payload)
  Overhead per txn:      0.02 ms  (NVMe fsync latency)

Calculation:
  max TPS = 100 MB/s / 1 KB = ~102,400 TPS

Scenario 2 (larger transactions):
  WAL write bandwidth:   100 MB/s
  WAL write per txn:     10 KB    (multiple rows, larger payloads)
  Overhead per txn:      0.02 ms

  max TPS = 100 MB/s / 10 KB = ~10,240 TPS

Scenario 3 (with replication):
  WAL write bandwidth:   100 MB/s
  WAL write per txn:     1 KB
  Replica acknowledgment: +0.5 ms network latency

  max TPS = 1 / (1/102400 + 0.0005) = ~1,960 TPS
```

Replication synchronous commit reduces throughput by ~50x compared to async commit. The network round-trip time dominates.

### Scaling Strategy

1. **Horizontal scaling**: Shard data across multiple database instances. Each shard handles a subset of transactions. Global transactions (spanning shards) use 2PC or Sagas.

2. **Vertical scaling**: Faster storage (NVMe, Optane), more memory (larger buffer pool), faster CPU (higher clock speed). This is simpler but has a ceiling.

3. **Read scaling**: Add read replicas. This works only if the bottleneck is reads, not writes. Write-heavy workloads need sharding.

4. **Batching**: Combine multiple operations into a single transaction. This reduces the number of WAL flushes per operation but increases transaction size and lock holding time.

## Deployment

### Connection Pooling with PgBouncer

```ini
[databases]
bank = host=127.0.0.1 port=5432 dbname=bank

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432
auth_type = scram-sha-256
pool_mode = transaction
default_pool_size = 50
max_client_conn = 5000
max_db_connections = 50
```

The key setting is `pool_mode = transaction`. Each application connection gets a database connection only for the duration of a transaction. Between transactions, the database connection returns to the pool.

### Transaction Timeout Configuration

PostgreSQL's `idle_in_transaction_session_timeout` (default: 0, infinite) should be set to a reasonable value:

```sql
ALTER SYSTEM SET idle_in_transaction_session_timeout = '30000';  -- 30 seconds
```

Any transaction open for more than 30 seconds without activity is automatically rolled back. This prevents abandoned transactions from holding locks indefinitely.

### Monitoring Long-Running Transactions

```sql
-- Find all transactions running longer than 5 seconds
SELECT
    pid,
    xact_start,
    now() - xact_start AS duration,
    state,
    query
FROM pg_stat_activity
WHERE state = 'active'
  AND xact_start < now() - interval '5 seconds'
ORDER BY xact_start;

-- Find idle-in-transaction connections (holding locks, doing nothing)
SELECT
    pid,
    xact_start,
    now() - xact_start AS duration,
    query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY xact_start;
```

### Orphan Transaction Cleanup

PostgreSQL can reach the transaction ID wraparound limit (about 2 billion transactions) if VACUUM does not keep up. When this happens, the database shuts down to prevent data loss:

```sql
-- Monitor XID usage
SELECT datname, age(datfrozenxid) AS xid_age
FROM pg_database;

-- Alert when xid_age exceeds 1 billion
-- Force vacuum when xid_age approaches 2 billion
VACUUM FREEZE;
```
