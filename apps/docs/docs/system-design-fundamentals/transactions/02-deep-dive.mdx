### Transaction Manager

The transaction manager assigns transaction IDs (XIDs) and tracks the state of every transaction. In PostgreSQL, XIDs are 32-bit integers assigned monotonically. The transaction manager maintains a commit log (pg_xact) that records whether each XID committed or aborted.

Key responsibility: determining visibility. When a transaction reads a row, the transaction manager checks whether the row's XID is committed. If not, the row is invisible to the reader (at Read Committed or higher isolation).

### Lock Manager

The lock manager controls concurrent access to database resources. It maintains a queue of lock requests and grants them when compatible:

- **Shared locks** (read): Multiple transactions can hold shared locks on the same resource simultaneously.
- **Exclusive locks** (write): Only one transaction can hold an exclusive lock. No shared locks are granted while an exclusive lock is held.
- **SIREAD locks** (PostgreSQL Serializable): Predicate locks that prevent phantom reads by locking the conditions of a query, not just the rows it returned.

The lock manager also detects deadlocks. In PostgreSQL, the deadlock detector runs every `deadlock_timeout` (default 1 second). When it detects a cycle — Transaction A waits for B, B waits for A — it aborts one of the transactions.

### Write-Ahead Log Manager

The WAL manager writes change records to the WAL buffer (shared memory) and flushes them to disk on COMMIT. It manages WAL segment files (typically 16 MB each) and handles WAL archiving for point-in-time recovery.

### Buffer Pool

The buffer pool caches data pages in memory. When a transaction modifies a page, the modification happens in the buffer pool copy. The page is marked "dirty" and is eventually written to disk by the background writer or during a checkpoint.

### Recovery Manager

The recovery manager handles crash recovery. On database startup after a crash, it reads the WAL from the last checkpoint and determines which transactions need to be redone and which need to be undone.

### PostgreSQL Transaction Internals

PostgreSQL's transaction subsystem has several PostgreSQL-specific structures that are worth understanding:

**pg_xact (commit log):** A shared memory array mapping XID to transaction status. Each XID maps to a 2-bit value: IN_PROGRESS (00), COMMITTED (01), ABORTED (10), or SUB_COMMITTED (11). This array is stored on disk in the pg_xact directory and is heavily cached because it is checked for every row visibility test.

**pg_xact (subtransaction log):** Tracks the parent-child relationship for subtransactions (savepoints). This is necessary because a subtransaction's visibility depends on its parent's state.

**pg_multixact:** Handles multiple transaction IDs for a single tuple. When multiple transactions hold a shared lock on the same row, PostgreSQL creates a multixact ID that represents the set of locking transactions. This is critical for the "multiple row-level share lock" scenario.

**clog buffer:** A 8 KB buffer (default, configurable via CLOG_BUFFERS) that caches the most recently accessed commit log pages. Since the commit log is checked for every row visibility test, caching is essential for performance.

**Transaction slot:** Each backend process has a transaction slot in shared memory. The slot tracks the transaction's start timestamp, XID, and current state. Visible to other processes for lock conflict detection and snapshot building.

### PostgreSQL Snapshot Building

When a transaction takes its first snapshot (at first query in Read Committed, or at BEGIN in Repeatable Read), PostgreSQL builds a snapshot structure:

```c
typedef struct SnapshotData {
    TransactionId xmin;  // Earliest XID that might be active
    TransactionId xmax;  // First XID that is definitely in the future
    TransactionId *xip;  // Array of active XIDs at snapshot time
    uint32 xcnt;         // Number of active XIDs
    // ... additional fields for subtransactions, cursors, etc.
} SnapshotData;
```

A row is visible to the snapshot if its XID satisfies:
- XID < xmin: definitely committed (or aborted, checked via pg_xact).
- XID >= xmax: definitely from the future, not visible.
- `xmin \<= XID \< xmax`: check if XID is in xip array. If in xip, the transaction was active and the row is not visible. If not in xip, the transaction committed before the snapshot was taken and the row is visible.

This snapshot mechanism is the heart of PostgreSQL's MVCC (Multiversion Concurrency Control). Readers never block writers, and writers never block readers, because each transaction sees a consistent snapshot of committed data from its start time.

### MVCC in Practice: Tuple Header

Every row in PostgreSQL has a tuple header that tracks its creation and deletion XIDs:

```text
┌────────────────────────────────────────────────────────────────┐
│                    POSTGRESQL TUPLE HEADER                     │
├───────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────────┬──────────┬──────────┬──────────┬────────────┐   │
│  │ t_xmin   │ t_xmax   │ t_ctid   │ t_infomask│ t_hoff    │   │
│  │ 4 bytes  │ 4 bytes  │ 6 bytes  │ 2 bytes   │ 1 byte    │   │
│  └──────────┴──────────┴──────────┴──────────┴────────────┘   │
│                                                                │
│  Fields:                                                        │
│  t_xmin: XID that created this tuple version                   │
│  t_xmax: XID that deleted or locked this tuple version         │
│  t_ctid: pointer to next tuple version (for UPDATE chains)    │
│  t_infomask: bit flags (has_oids, has_null, is_updated...)    │
│  t_hoff: offset to tuple data start                           │
│                                                                │
│  Visibility Rule:                                               │
│  A tuple is visible if t_xmin is COMMITTED and (t_xmax is     │
│  IN_PROGRESS/ABORTED, OR t_xmax was the current transaction   │
│  doing the read).                                              │
└────────────────────────────────────────────────────────────────┘
```

When an UPDATE occurs, PostgreSQL does not modify the existing tuple. It creates a new tuple version (new physical row) and updates the old tuple's t_xmax and t_ctid to point to the new version. The old version remains in the page until VACUUM removes it. This is MVCC in action: transactions that started before the UPDATE see the old version; transactions that started after see the new version.

## Data Structures and Algorithms

### Write-Ahead Log Structure

Every WAL record has a specific binary format. In PostgreSQL, a WAL record contains:

| Field | Size | Description |
|---|---|---|
| LSN (Log Sequence Number) | 8 bytes | Position in the WAL stream |
| Prev-LSN | 8 bytes | LSN of previous record for this XID |
| XID | 4 bytes | Transaction that wrote this record |
| Resource Manager ID | 2 bytes | Which subsystem generated this record |
| Record Type | 1 byte | INSERT, UPDATE, DELETE, COMMIT, etc. |
| Payload Length | 4 bytes | Length of the payload data |
| Payload | Variable | Before/after images, page data |
| CRC-32C | 4 bytes | Checksum for corruption detection |
| Total | ~48 + payload | Minimum size for any WAL record |

### ARIES Recovery Algorithm

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is the recovery algorithm used by PostgreSQL, Oracle, DB2, and most commercial databases. It has three components:

1. **WAL Protocol**: Before any data page is modified, the corresponding WAL record must be flushed to stable storage. This is the Write-Ahead Log rule.

2. **Redo Phase**: During recovery, the database replays all changes from the WAL starting from the last checkpoint, regardless of whether the originating transaction committed. This ensures that any change that made it to disk can be reconstructed.

3. **Undo Phase**: After redo, the database rolls back all transactions that were active at the time of the crash. It reads the WAL records for these transactions and applies the inverse operations.

### STEAL/NO-FORCE Buffer Management

PostgreSQL uses a STEAL/NO-FORCE buffer management policy:

- **STEAL**: The buffer manager can evict dirty pages (pages modified by an uncommitted transaction) from the buffer pool and write them to disk before the transaction commits. This is safe because the WAL contains undo information to revert the changes if the transaction aborts.

- **NO-FORCE**: At commit time, the database does NOT force dirty pages to disk. It only forces the WAL to disk. The dirty pages are written later by the background writer or during a checkpoint.

The advantage of NO-FORCE: COMMIT latency depends on WAL write speed (sequential I/O) rather than random page write speed. Sequential WAL writes are typically 10-100x faster than random page writes.

### Shadow Paging

An alternative to WAL-based recovery is shadow paging, used by databases like SQLite in WAL mode (partially) and LMDB.

Instead of writing changes to the WAL and then replaying them, shadow paging maintains two copies of each page: the current page and a shadow page. When a transaction modifies data, it writes to a new page (the shadow) without touching the current page. On commit, the database atomically switches from the current page table to the shadow page table by writing a single pointer in the database header.

| Aspect | WAL | Shadow Paging |
|---|---|---|
| Recovery time | Must replay WAL (can be slow) | Instant (single pointer switch) |
| Write overhead | Sequential append (fast) | Random writes to new pages |
| Checkpoint complexity | Complex (must track dirty pages) | None (snapshot is always consistent) |
| Read performance | Buffer pool reads any page | Shadow pages may be scattered |

PostgreSQL, Oracle, and DB2 use WAL because it provides better write throughput under high concurrency. Shadow paging is simpler but suffers from write amplification.

### Checkpointing Algorithms

A checkpoint is a point in the WAL where all dirty pages from committed transactions have been written to disk. This limits recovery time: during recovery, the database only needs to replay WAL records from the last checkpoint, not from the beginning of time.

PostgreSQL uses a fuzzy checkpoint algorithm:

1. The checkpoint process wakes up at intervals defined by `checkpoint_timeout` (default 5 minutes) or when `max_wal_size` is approached.
2. It writes a CHECKPOINT begin record to the WAL.
3. It scans the buffer pool for dirty pages and writes them to disk.
4. It updates the checkpoint location in the control file.
5. It writes a CHECKPOINT done record to the WAL.

During recovery, the database starts from the last checkpoint location in the control file and replays forward.

### Detailed WAL Write Analysis

At commit time, the following WAL writes happen:

1. **WAL buffer**: The UPDATE WAL record was written to the in-memory WAL buffer during step 10 (the actual UPDATE). The COMMIT WAL record is written during step 14.

2. **WAL flush (fsync)**: The WAL buffer is flushed to disk during step 15. This is the single fsync call that makes the transaction durable.

3. **Return to client**: The client receives COMMIT OK only after step 17 completes.

If the server crashes between step 15 and step 16, the transaction is safe: the WAL records are on disk, and during recovery, the database will replay them. The locks are lost (they only exist in memory), but the data is durable. The database does not need to re-acquire locks — it simply replays the WAL records and everything is consistent.

### How fsync Actually Works

The `fsync` system call tells the operating system to write all buffered data for a file descriptor to physical storage. This is the moment of truth for durability.

```c
// Pseudocode for the COMMIT path
write_wal_record(wal_buffer, commit_record);  // Write to OS page cache
fsync(wal_fd);                                 // Force OS to write to disk
// If fsync returns success, data is on physical storage
// If power fails now, data survives
return_commit_ok();
```

The cost of fsync is significant: a typical HDD has 3-10 ms seek time per fsync. An SSD can do 10,000-50,000 fsyncs per second. A RAM disk or NVMe with Optane can do over 100,000 fsyncs per second.

## Consistency and State

### Recovery from Different Failure Points

**Failure before WAL flush:** The transaction's WAL records are in the WAL buffer (in memory), never fsynced to disk. On crash, none of the transaction's changes are recovered. The transaction is effectively lost. The application must re-issue it.

**Failure after WAL flush but before buffer pool flush:** The transaction's WAL records are on disk. The dirty pages are in the buffer pool (also lost on crash). During recovery, the REDO phase replays the WAL records and reconstructs the changes. The transaction survives.

**Failure after commit record flush:** The COMMIT WAL record is on disk. The transaction is durable. During recovery, REDO replays the commit record and the transaction is visible. Even if the underlying data pages were never written to disk, the WAL contains enough information to reconstruct them.

## Performance Characteristics

### WAL Write Overhead

Each COMMIT requires at least one fsync call. The fsync is the dominant cost of a transaction commit:

| Storage Type | fsync Latency | Max TPS (single-threaded) |
|---|---|---|
| HDD (7200 RPM, single disk) | 3-10 ms | 100-300 |
| SATA SSD | 0.1-0.5 ms | 2,000-10,000 |
| NVMe SSD | 0.02-0.1 ms | 10,000-50,000 |
| NVMe + battery-backed RAID | 0.01-0.05 ms | 20,000-100,000 |
| RAM disk (no fsync) | 0.001-0.005 ms | 200,000+ |

### Group Commit

Databases batch multiple commits into a single fsync call. Instead of calling fsync after every COMMIT, the database accumulates COMMIT records for a few microseconds and calls fsync once for the batch.

```sql
-- Without group commit (3 fsyncs for 3 commits)
COMMIT;  -- fsync 1
COMMIT;  -- fsync 2
COMMIT;  -- fsync 3
-- Total: 3 fsync calls, 3x latency

-- With group commit (1 fsync for 3 commits)
-- Database waits ~100μs, collects all three, flushes once
COMMIT;  -- wait for group flush
COMMIT;  -- wait for group flush
COMMIT;  -- all three returned after single fsync
-- Total: 1 fsync call, 1x latency + 100μs wait
```

Group commit turns 3 ms of fsync latency into 1 ms — a 3x improvement at very high concurrency. PostgreSQL has supported group commit since version 9.2.

### Checkpoint Impact

Checkpoints cause a burst of I/O as the database writes all dirty pages to disk. During a checkpoint, overall system throughput may drop by 10-30 percent because disk bandwidth is consumed by checkpoint writes instead of WAL writes and data reads.

Tuning checkpoint frequency involves a trade-off:

- **Frequent checkpoints**: Recovery is faster (less WAL to replay). But checkpoint I/O overhead occurs more often.
- **Infrequent checkpoints**: Less checkpoint I/O overhead. But recovery is slower and more WAL must be retained (more disk space).

## Configuration and Tuning

PostgreSQL exposes several parameters that control transaction behavior and performance.

### `wal_level`

Controls how much information is written to the WAL.

| Value | Writes | Use Case |
|---|---|---|
| `minimal` | Just enough for crash recovery | Development, no replication |
| `replica` (default) | Full WAL for recovery + replication | Production with streaming replication |
| `logical` | Full WAL + logical decoding info | Logical replication, CDC tools |

### `synchronous_commit`

Controls when COMMIT returns to the client.

| Value | Behavior | fsync | Use Case |
|---|---|---|---|
| `on` (default) | Wait for WAL flush to local disk | Yes | Standard durability |
| `remote_write` | Wait for WAL flush to local disk + write to replica | Yes + network | Synchronous replication |
| `remote_apply` | Wait for WAL flush to local disk + replica applied | Yes + network | Read-your-writes on replicas |
| `off` | Return immediately, WAL flushed eventually | No | Performance-critical, loss-tolerant |

Setting `synchronous_commit = off` increases throughput by 5-10x but risks losing the last 0.5 to 1 second of committed transactions on a crash. This is acceptable for systems like session stores or analytics ingestion.

### `commit_delay`

Microseconds to wait before flushing the WAL, hoping to batch multiple commits. Default is 0 (no delay). Setting this to 100-1000 microseconds can increase throughput under high concurrency by enabling group commit.

### `wal_buffers`

Size of the shared memory WAL buffer. Default is -1 (auto, typically 4% of shared_buffers, minimum 64 KB, maximum WAL segment size). Larger values allow more WAL records to be kept in memory before flushing.

### `checkpoint_timeout`

Maximum time between automatic checkpoints. Default is 300 seconds (5 minutes). Longer intervals mean less checkpoint overhead but more WAL to replay during recovery.

### `max_wal_size`

Soft limit on total WAL size. Default is 1024 MB. When total WAL exceeds this, a checkpoint is triggered. Larger values reduce checkpoint frequency but increase disk usage and recovery time.

### `full_page_writes`

When enabled (default), every page modified after a checkpoint starts must be written as a full page to the WAL the first time it is modified. This protects against partial page writes during crashes. Disabling this increases performance but risks data corruption if the page is partially written during a crash.

## The WAL in Distributed Systems

In distributed databases, the WAL has additional responsibilities:

- **Replication**: The primary streams WAL records to replicas. Replicas replay the WAL to stay synchronized. This is how PostgreSQL streaming replication works: the replica connects to the primary and continuously receives and replays WAL records.
- **Cascading replication**: A replica can forward WAL to another replica, reducing load on the primary.
- **WAL archiving**: Completed WAL segments are archived to long-term storage (S3, NFS) for point-in-time recovery. The database can replay archived WAL segments to restore the database to any point in time.
- **Logical decoding**: WAL records are decoded into logical change events (INSERT, UPDATE, DELETE) for change data capture (CDC) pipelines.

The WAL is not just a transactional primitive. It is the foundation of the database's entire durability, replication, and recovery architecture. Every major relational database and many NoSQL databases (MongoDB, CockroachDB, FoundationDB) use a WAL for the same reason: it is the simplest way to guarantee that committed data survives crashes while keeping COMMIT latency as low as possible.
