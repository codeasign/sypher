# Disaster Recovery -- Challenges

## Tier 1: Foundation

These challenges test your understanding of core disaster recovery concepts. They assume you can define RPO and RTO, distinguish between DR strategies, and identify common failure modes.

### Challenge 1: Calculate the RTO Budget

You are designing a disaster recovery system for a SaaS application. The business requires an RTO of 5 minutes. The system components and their recovery times are:

- DNS propagation (TTL-based): 60 seconds
- Database promotion: 90 seconds
- Application auto-scaling: 120 seconds (pilot light to full capacity)
- Cache warm-up: 45 seconds
- Configuration reload: 30 seconds
- Health check validation: 30 seconds
- Load balancer health check propagation: 30 seconds

**Question**: Does this system meet the 5-minute RTO target? If not, which component would you optimize first? Show your math.

**Model answer**:

The total recovery time is the sum of sequential steps. However, some steps can run in parallel.

Sequential steps (must happen in order):
1. DNS propagation: 60 seconds
2. Database promotion: 90 seconds
3. Configuration reload: 30 seconds
4. Health check validation: 30 seconds

Parallel steps (can run concurrently with sequential steps):
- Application auto-scaling: 120 seconds (starts after DNS propagation)
- Cache warm-up: 45 seconds (starts after database promotion)
- Load balancer health check propagation: 30 seconds (starts with DNS)

Critical path:
- Step 1: DNS propagation (60s)
- Step 2: Database promotion (90s) -- starts after DNS
- Step 3: Auto-scaling (120s) -- starts after DNS, runs in parallel with DB promotion
- Step 4: Configuration reload (30s) -- starts after DB promotion AND auto-scaling complete
- Step 5: Health check validation (30s) -- starts after config reload

Critical path calculation:
```
60s (DNS) + max(90s, 120s) + 30s + 30s = 60s + 120s + 30s + 30s = 240s
```

Total: 240 seconds = 4 minutes. This is under the 5-minute RTO target.

**However**, if the auto-scaling takes 150 seconds instead of 120 seconds (p99 case), the critical path becomes:
```
60s + 150s + 30s + 30s = 270s = 4.5 minutes
```

This still meets the 5-minute target but leaves only 30 seconds of headroom. The optimization priority should be application auto-scaling, as it is on the critical path and has the most variability.

### Challenge 2: Choose the DR Strategy

Your company operates a photo-sharing social network with 50 million monthly active users. The service allows users to upload, browse, and share photos. The database stores user profiles, photo metadata, and social graph relationships. The actual photo files are stored in an object store.

Requirements:
- Users should be able to upload and browse photos at all times
- A 1-hour outage costs $50,000 in lost ad revenue
- The engineering budget for DR infrastructure is 30% of the total infrastructure budget
- Regulatory requirements mandate that user data must not leave the country (the primary region is in the EU)

**Question**: Recommend a DR strategy. Justify your RPO and RTO targets. Explain where the secondary region should be located.

**Model answer**:

This is a read-heavy workload (browsing photos) with writes during uploads. The object store is already durable by default. The critical component is the database.

Recommended strategy: **Warm standby with cross-region replication within the EU**.

**RTO target**: 10 minutes. The cost of a 1-hour outage is $50,000, so a 10-minute recovery saves $41,667 per outage. A 10-minute RTO is achievable with warm standby.

**RPO target**: 5 minutes. Photo uploads are not financial transactions. Losing 5 minutes of photo metadata is annoying but not catastrophic. Users can re-upload. A 5-second RPO would require synchronous replication and would add latency to all uploads.

**Secondary region**: Another EU country (e.g., if primary is in Ireland, secondary is in Frankfurt). This satisfies the data residency requirement while providing geographic diversity. Both regions must be in the EU.

**Architecture**:
- Primary region: Ireland (eu-west-1)
- Secondary region: Frankfurt (eu-central-1)
- Database: Aurora PostgreSQL with cross-region read replica
- Object storage: S3 with Cross-Region Replication
- Compute: Auto-scaling groups in both regions
- DNS: Route53 latency-based routing with health check failover
- Cache: ElastiCache Redis in primary, cold start in secondary (acceptable for 10-minute RTO)

**Cost**: The secondary region runs at 30% of primary capacity. Total DR cost is approximately 1.3x the single-region cost. This fits within the 30% budget increase.

### Challenge 3: Identify the Single Points of Failure

A startup has the following infrastructure for their API service:

- One EC2 instance running the Node.js application
- One RDS PostgreSQL instance (db.t3.medium, single-AZ)
- One ElastiCache Redis instance (cache.t3.small)
- An S3 bucket for static assets
- A Route53 hosted zone with an A record pointing to the EC2 instance

**Question**: List every single point of failure in this setup. For each SPOF, explain what would happen if it fails. Recommend the minimum changes to eliminate each SPOF.

**Model answer**:

| Component | Failure Impact | Minimum Fix |
|---|---|---|
| Single EC2 instance | Application becomes unavailable. All API requests fail. | Add auto-scaling group with minimum 2 instances across 2 availability zones |
| Single RDS instance (single-AZ) | Database becomes unavailable. All data-dependent operations fail. | Enable Multi-AZ deployment (automatic failover to standby in different AZ) |
| Single Redis instance | Cache becomes unavailable. Performance degrades but app survives (cache miss = database query). | Add Redis replica or use ElastiCache with cluster mode (multi-node) |
| Single Availability Zone | If the AZ fails, all resources in that AZ fail. | Deploy resources across at least 2 AZs in the same region |
| Single Region | If the region fails, the entire application goes down. | Implement cross-region backup or warm standby (depending on RTO requirements) |
| No automated backups | If data is corrupted, there is no recovery point. | Enable automated backups with cross-region copy |

The minimum viable fix is to eliminate the single-AZ dependencies by deploying across 2 AZs. This protects against an AZ failure, which is the most common type of infrastructure failure in cloud environments. Cross-region DR can be added later as the system grows.

## Tier 2: Design

These challenges test your ability to design complete disaster recovery systems. They require you to make trade-offs, estimate costs, and produce concrete architectures.

### Challenge 4: Design DR for a Global Payment Platform

**Scenario**: You are designing disaster recovery for a global payment platform that processes $500 million in transactions per day. The platform operates in 20 countries and supports 10 currencies. Transactions must not be lost. The platform has a SLA of 99.995% availability with customers (enterprise merchants).

**Requirements**:
- Zero transaction loss (RPO = 0)
- Failover under 60 seconds (RTO = 60s)
- Regulatory compliance in each country (data must stay within the country for local transactions)
- 99.995% availability (under 26 minutes of downtime per year)

**Design task**: Produce the DR architecture. Include:
1. The failover strategy (active-active, warm standby, etc.)
2. The database replication approach
3. The DNS and traffic routing strategy
4. The failover orchestration
5. The cost estimate as a multiple of single-region infrastructure
6. The regulatory compliance approach

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Needs Work (1 pt) |
|---|---|---|---|
| RPO = 0 strategy | Uses synchronous replication with quorum-based commit across 3 regions | Uses synchronous replication across 2 regions | Proposes async replication (does not achieve RPO = 0) |
| RTO under 60s | Active-active deployment with no failover needed; latency-based routing | Warm standby with pre-warmed capacity and automated failover | Backup and restore with manual failover |
| Regulatory compliance | Regional sharding: transactions processed and stored within the country; cross-region replication only for aggregated data | Single global database with country-level data segregation in the schema | Ignores regulatory requirements |
| Cost estimate | Provides a specific cost multiple (e.g., 2.5x) with breakdown by component | Gives a range (2-3x) without breakdown | Does not estimate cost |
| Failover automation | Distributed orchestrator with active-active deployment, fencing, and split-brain prevention | Single-region orchestrator with automatic failover | Manual failover |

**Model discussion**:

**Strategy**: Active-active across 3 regions per continent. Within each continent, three regions form a quorum. Transactions are routed to the nearest region based on the user's location.

Why active-active: With an RTO of 60 seconds and an RPO of zero, warm standby is not sufficient. The database promotion alone takes 60-90 seconds. Active-active means no promotion is needed — the secondary is already serving writes.

**Database replication**: Synchronous replication with a quorum of 2 out of 3 regions. A write is committed when at least 2 regions acknowledge it. This means the system can tolerate one region failure without losing data or pausing writes.

For regulatory compliance, each continent operates independently. A transaction originating in Germany is processed in the EU cluster (Ireland, Frankfurt, London). A transaction originating in Japan is processed in the Asia cluster (Tokyo, Seoul, Singapore). Cross-continent replication is asynchronous and contains only aggregated, anonymized data.

**DNS and routing**: Global traffic manager (Route53 or Cloudflare) routes traffic to the nearest region based on latency. Each region can handle 100% of the continent's traffic. If one region fails, the remaining two regions in the continent share the load. No DNS update is needed — the traffic manager detects the failed region via health checks and stops routing to it.

**Failover orchestration**: Each region runs an orchestrator in active-passive mode. If the orchestrator in a region stops reporting, the orchestrators in the other regions detect it and reconfigure the quorum. The database writes continue with a reduced quorum of 2 regions.

**Cost estimate**: 2.5x single-region cost. The breakdown is:
- Compute: 3x (three regions)
- Database: 2x (writes require 2 of 3, but reads can be served from any region)
- Cache: 3x
- Networking: 2x (cross-region data transfer costs)
- Orchestration: 1x (marginal additional cost)

**Regulatory compliance**: Data residency is enforced at the application layer. The transaction router inspects the payment origin and routes to the appropriate continental cluster. Each cluster has its own database, its own encryption keys, and its own access controls. Only aggregated metrics (total volume, success rate) are shared across continents.

### Challenge 5: Design a DR Testing Program

**Scenario**: You join a company as the first SRE. The company has 20 microservices running on Kubernetes in a single AWS region. There is no disaster recovery plan. The CTO wants you to build a DR testing program that validates recovery for all 20 services within 3 months.

**Constraints**:
- Budget: $20,000 for tooling and infrastructure
- Team: You and one other SRE
- Cannot cause production outages
- Must produce measurable results (actual RTO and RPO numbers)

**Design task**: Design a phased DR testing program. Include:
1. The testing phases and timeline
2. The testing methodology for each phase
3. The tooling choices
4. How you measure and report results
5. How you handle failing tests

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Needs Work (1 pt) |
|---|---|---|---|
| Phased approach | 4+ phases with clear progression (tabletop, staging, partial prod, full prod) | 3 phases without clear criteria for progression | Single phase (e.g., "test everything at once") |
| Safety measures | Blast radius limits, circuit breakers, rollback plans, and manual abort for every test | Some safety measures but not comprehensive | No safety measures mentioned |
| Measurement | Records RTO, RPO, detection time, execution time for every test | Records RTO and RPO only | Does not measure results |
| Action items | Each test produces tracked action items with owners and deadlines | Action items are noted but not tracked | No action items from tests |

**Model discussion**:

**Phase 1 (Week 1-2): Tabletop exercises**
For each service, the team walks through the DR plan verbally. A facilitator presents a failure scenario and the team describes their response. This costs $0 in tooling and takes 30 minutes per service.

Goal: Identify gaps in the DR plan before spending money on infrastructure. Expected outcome: Each service has a documented DR plan with RPO and RTO targets.

**Phase 2 (Week 3-4): Staging environment tests**
Deploy a staging environment that mirrors production topology at 10% scale. Run chaos experiments using LitmusChaos: terminate pods, fail the database, block network access to a service.

Tooling: LitmusChaos is open source. The staging environment costs approximately $1,000/month in compute.

Goal: Validate that the DR plan works at a smaller scale. Expected outcome: 80% of services pass staging tests.

**Phase 3 (Week 5-8): Read-only production tests**
For services that passed staging, run read-only DR tests in production. This means simulating a failover by verifying that the secondary can serve traffic, but not actually switching traffic away from the primary.

Method: deploy a secondary instance of the service in the secondary region, route a small percentage of read-only traffic to it, and measure correctness.

Tooling: Traffic shadowing (using service mesh like Istio) costs approximately $500/month in additional compute.

Goal: Validate that the secondary infrastructure is configured correctly. Expected outcome: 90% of services pass read-only tests.

**Phase 4 (Week 9-12): Full failover tests**
For services that passed read-only tests, schedule a full failover test during low-traffic hours. The service operates from the secondary region for 1 hour, then fails back.

Safety measures: Manual abort available within 10 seconds of any issue. Blast radius limited to one service at a time. Rollback plan documented and tested.

Goal: Validate end-to-end failover and failback. Expected outcome: All 20 services achieve their RTO and RPO targets.

**Reporting**: After each phase, publish a DR readiness scorecard. Each service gets a green/yellow/red rating based on the last test result. The scorecard is visible to the entire engineering organization.

**Handling failures**: If a service fails a test, the test stops immediately. A post-mortem is scheduled within 48 hours. The action items from the post-mortem must be resolved before the service can retry the test.

### Challenge 6: Design DR for a Real-Time Messaging System

**Scenario**: Design disaster recovery for a real-time messaging system (similar to WhatsApp or Messenger) with 1 billion users. Messages must be delivered reliably. Users expect to see their message history even after a regional outage.

**Requirements**:
- Messages must not be lost (RPO = 0 for sent messages)
- Users should be able to send and receive messages within 5 seconds of a region failure (RTO = 5s)
- Message history must be available after recovery
- The system handles 100 billion messages per day

**Design task**: Design the DR architecture for the message storage and delivery system.

**Model discussion**:

**Key insight**: The messaging system is naturally partitionable. Users communicate within clusters of relationships. If a user in Japan messages a user in Brazil, the message crosses regions anyway. But most messages are local — users in the same city message each other.

**Architecture**: Regional message stores with cross-region replication for cross-region conversations.

**User data partitioning**: Each user's data is assigned to a primary region based on their registered location. The user's messages are stored in that region and replicated to a secondary region in the same continent. If the primary region fails, the user connects to the secondary region and their messages are available.

**Message delivery**: When a user sends a message, it is written to the primary region's message store with synchronous replication to the secondary region within the same continent. The message is acknowledged when both replicas confirm. If the message is for a user in a different continent, the message is forwarded through a cross-continent queue.

**RTO of 5 seconds**: This is achieved through an active-active connection tier. Users maintain a persistent WebSocket connection to the nearest edge server. If the edge server detects that the user's primary region is unreachable, it routes the user to the secondary region. The connection handoff happens transparently — the user does not notice.

**RPO of 0**: Synchronous replication within the continent ensures no messages are lost. If the secondary region is unreachable, the message is queued in a local buffer and sent when the secondary recovers.

**Message history**: Message history is stored in a time-ordered database (Cassandra or Scylla) that is replicated across regions. The history is read from the nearest replica. During a failover, the secondary replica serves the history.

## Tier 3: Expert

These challenges require deep distributed systems knowledge and the ability to design novel solutions.

### Challenge 7: Design a Global DR System That Ensures Strict Serializability

**Scenario**: You are designing a global financial exchange that matches buy and sell orders. The system processes 1 million trades per second. Every trade must be processed in strict serializable order — the order in which transactions appear to execute must match the actual order in which they were submitted. A single trade processed out of order creates an arbitrage opportunity and costs the exchange millions of dollars.

**Challenge**: Design a disaster recovery system for this exchange that maintains strict serializability across regions while surviving a full region failure.

**Discussion**:

This is an extremely hard problem. Strict serializability and disaster recovery are in tension: strict serializability requires a total order of all operations, which typically requires a single leader that sequences all writes. A single leader is a single point of failure. Removing the single point of failure while maintaining strict serializability requires a consensus protocol that operates across regions.

**Approach 1: Spanner-style TrueTime**

Google Spanner achieves strict serializability across regions using TrueTime, Google's synchronized clock technology. TrueTime provides bounded clock uncertainty (typically under 10 ms). Spanner assigns commit timestamps to transactions and waits out the clock uncertainty before making the commit visible. This ensures that transactions are ordered by their real-world time even across regions.

For the exchange, a Spanner-like system deployed across 3 regions with synchronous replication and TrueTime can provide:
- Strict serializable transactions across all regions
- Automatic failover (the remaining 2 regions continue with a reduced quorum)
- No data loss (RPO = 0)
- RTO under 10 seconds (the time needed to detect the failed region and reconfigure the quorum)

The cost is high: TrueTime requires specialized hardware (GPS clocks and atomic clocks in each data center). This approach is only available on Google Cloud (Cloud Spanner).

**Approach 2: Deterministic ordering with a sequencer**

A sequencer service assigns a monotonically increasing sequence number to every incoming trade. The sequencer is replicated across regions using Raft. Each trade carries its sequence number, and the matching engine processes trades in sequence number order.

If the primary sequencer fails, a new sequencer is elected via Raft. The election takes approximately 5 seconds (typical Raft election timeout). During the election, incoming trades are buffered but not processed. Once the new sequencer is elected, it continues from the last committed sequence number.

This approach provides:
- Strict serializable ordering (sequence numbers enforce order)
- RPO = 0 (Raft ensures all committed sequence numbers are durable)
- RTO = 5 seconds (Raft election time)
- Lower cost than TrueTime (no specialized hardware)

**Approach 3: Single-region primary with active standby**

This is the simplest approach that meets the requirements. All trades are processed in a single primary region. A standby region runs a replica with synchronous replication. If the primary fails, the standby is promoted and becomes the new primary. During the promotion, no trades are processed.

The RTO is determined by the database failover time (typically 30-60 seconds for a promoted replica). The 30-60 second gap means no trades are processed during that window. The cost is a backlog of up to 30 million trades (at 1 million per second) that must be processed after recovery.

**The hard truth**: There is no free lunch. Strict serializability requires a single ordering authority, which is inherently a bottleneck and a single point of failure. The approaches above represent different trade-offs between complexity, cost, RTO, and performance. For a real exchange, the choice depends on the exchange's tolerance for trade processing pauses and its willingness to pay for specialized infrastructure.

### Challenge 8: Design a Multi-Cloud DR System That Survives a Cloud Provider Outage

**Scenario**: Your company operates a critical infrastructure service used by hospitals, emergency services, and government agencies. A single cloud provider outage — like the ones that have affected AWS, Azure, and GCP — would be unacceptable. You need disaster recovery across two cloud providers: AWS and GCP.

**Challenges**:
1. Different APIs, different services, different authentication
2. Data must be consistent across providers
3. Failover must be automatic
4. The system must work correctly even if both providers degrade simultaneously (but not completely)

**Discussion**:

Multi-cloud disaster recovery is the hardest DR problem. Most companies should not attempt it — the complexity and cost are enormous. But for systems that truly cannot tolerate any provider-level outage, it is the only option.

**Challenge 1: Heterogeneous APIs**

Each provider has different services: AWS uses RDS, GCP uses Cloud SQL. AWS uses ELB, GCP uses Cloud Load Balancing. AWS uses Route53, GCP uses Cloud DNS. The application must be written to an abstraction layer that hides these differences.

Solution: Use Kubernetes, Terraform, and provider-agnostic tools. Kubernetes provides the same orchestration API regardless of the underlying infrastructure. Terraform modules can be written once and configured for different providers. Database drivers abstract the database differences.

```hcl
# Multi-cloud module (abstracted)
module "compute" {
  source = "./modules/compute"
  provider = var.cloud_provider  # "aws" or "gcp"
  region = var.region
  instance_count = var.instance_count
}

module "database" {
  source = "./modules/database"
  provider = var.cloud_provider
  region = var.region
  db_size = var.db_size
}
```

**Challenge 2: Cross-provider data consistency**

Replicating a database from AWS to GCP is harder than replicating within a single provider. Direct replication (PostgreSQL WAL streaming across providers) requires a VPN connection between the two provider networks and may have high latency.

Solution: Use application-level replication rather than database-level replication. The application writes events to a log that is replicated to both providers. Each provider independently processes the event log and builds its own database state. This is similar to the event sourcing pattern.

```python
def write_event(event):
    # Write to both providers
    aws_result = aws_queue.send(event)
    gcp_result = gcp_queue.send(event)
    return aws_result and gcp_result
```

**Challenge 3: Automatic failover across providers**

Automatic failover across providers requires a third-party monitoring system that lives outside both providers. You cannot use Route53 health checks (AWS) to detect a GCP outage, because the health check itself is part of AWS.

Solution: Use a third-party DNS provider like Cloudflare or DNS Made Easy that is independent of both cloud providers. The third-party DNS runs health checks against both providers and can route traffic to either one.

**Challenge 4: Simultaneous degradation**

If both providers degrade simultaneously (e.g., a shared dependency like a TLS certificate authority goes down), the system must degrade gracefully. Design for partial functionality: read-only mode, stale data serving, or reduced feature set.

**Final architecture**:

```
Third-party DNS (Cloudflare)
        |
    ┌───┴───┐
    |       |
  AWS     GCP
  Region  Region
  
  Each provider runs independently
  Events replicated via cross-cloud queue
  DNS health checks from outside both clouds
  Manual override for simultaneous degradation
```

**Cost**: Multi-cloud DR is 3-4x the single-cloud cost. The complexity and operational overhead are even higher. This is the right solution for hospitals, emergency services, and critical infrastructure. For most other systems, single-cloud multi-region DR is sufficient.

## Summary of Design Principles

Across all three tiers of challenges, several design principles emerge:

**Principle 1: DR is a business decision, not a technical one.** The RPO and RTO targets should be driven by business impact analysis, not by technical convenience. The cost of DR must be justified by the cost of downtime.

**Principle 2: Test everything, especially the things you assume work.** The most common cause of DR failure is an untested assumption — the backup works, the secondary is configured correctly, the orchestration runs. Every assumption must be validated through testing.

**Principle 3: Automation is necessary but not sufficient.** Automated failover is essential for meeting aggressive RTO targets. But every automation must have a manual override, a rollback plan, and a cooldown timer to prevent flapping.

**Principle 4: Plan for the disaster after the disaster.** The primary failure is bad enough. The cascading failures, human errors, and configuration drifts that follow can be worse. A complete DR plan accounts for secondary and tertiary failure modes.

**Principle 5: DR is a practice, not a project.** The DR system must be continuously tested, maintained, and improved. A one-time DR project decays into irrelevance. A quarterly DR practice stays sharp.
