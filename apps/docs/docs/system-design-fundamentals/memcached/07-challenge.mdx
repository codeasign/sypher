## Tier 1 — Core Concepts

### Challenge 1.1: Cache-Aside Implementation

You are building a social media application with a PostgreSQL database. Your team has decided to use Memcached for caching user profiles. User profiles are 2-5 KB each, and your application has 1 million registered users with 100,000 daily active users.

**Task.** Implement the cache-aside pattern for user profile retrieval. Your solution must:
- Check Memcached for the profile.
- On miss, query PostgreSQL and populate the cache.
- Set a TTL of 1 hour.
- Handle the case where Memcached is unreachable (graceful degradation to database).

```python
import json
from pymemcache.client.base import Client
import psycopg2

# Assume these are already configured
cache = Client(('localhost', 11211))
db = psycopg2.connect("dbname=social_app")

def get_user_profile(user_id):
    # Your code here
    pass
```

**Model Answer.**

```python
def get_user_profile(user_id):
    key = f"profile:{user_id}"

    try:
        cached = cache.get(key)
        if cached is not None:
            return json.loads(cached)
    except Exception:
        # Memcached unreachable — fall through to database
        pass

    # Cache miss or cache down — query database
    with db.cursor() as cur:
        cur.execute(
            "SELECT user_id, username, display_name, avatar_url, bio "
            "FROM users WHERE user_id = %s", (user_id,)
        )
        row = cur.fetchone()

    if row is None:
        return None

    profile = {
        "user_id": row[0],
        "username": row[1],
        "display_name": row[2],
        "avatar_url": row[3],
        "bio": row[4],
    }

    # Populate cache (best-effort — do not fail if cache is down)
    try:
        cache.set(key, json.dumps(profile), expire=3600)
    except Exception:
        pass

    return profile
```

**Key points the answer should include:**
- Exception handling around cache calls (cache failures should not break the application).
- JSON serialization/deserialization (Memcached stores bytes, not Python objects).
- TTL of 3600 seconds (1 hour).
- Cache population on miss.
- Graceful degradation: if the cache is unreachable, the application still works.

### Challenge 1.2: Consistent Hashing Explanation

**Task.** Explain consistent hashing to a non-technical product manager. Your explanation must include:
- The problem it solves (what happens with simple modulo hashing when you add a node).
- How the hash ring works (at a conceptual level, not mathematically).
- Why virtual nodes matter.

**Model Answer.**

"Imagine you have 10 servers, each storing a piece of a giant puzzle. When a new puzzle piece arrives, you need to decide which server to store it on. The simplest approach is to count the pieces, divide by 10, and assign each piece to a server based on the remainder.

"This works until you add an 11th server. Now every piece must be re-counted and re-assigned — nearly all pieces have to move. This is expensive: every moved piece must be fetched from the database and re-stored in the new cache server.

"Consistent hashing is a smarter approach. Instead of the remainder method, you arrange the 10 servers in a circle (the ring). Each puzzle piece is assigned to the nearest server clockwise on the ring. When you add an 11th server, only the pieces between the new server and its clockwise neighbor need to move — the rest stay exactly where they are.

"Virtual nodes improve balance. Without them, the ring has only 10 points, and some servers might end up with more ring space than others. By placing 100-200 virtual copies of each server on the ring, the distribution becomes much more even.

"The result: you can add or remove cache servers with minimal disruption. Most cached data stays valid, and only a small fraction of requests hit the database during the transition."

### Challenge 1.3: Slab Allocator Sizing

**Task.** Your Memcached instance has 16 GB of RAM (`-m 16384`). You are storing:
- 10 million session objects at 200 bytes each
- 500,000 API response objects at 8 KB each
- 100,000 page fragments at 50 KB each

The slab growth factor is the default 1.25x. Slab classes are approximately: class 1 (96B), class 2 (120B), class 3 (152B), class 4 (192B), class 5 (240B), class 6 (304B), class 7 (384B), class 8 (480B), class 9 (600B), class 10 (752B), class 11 (944B), class 12 (1.2K), class 13 (1.5K), class 14 (1.9K), class 15 (2.4K), class 16 (3K), ..., class 28 (64K), class 29 (80K), class 30 (100K), etc.

**Question.** Will all three data types fit without eviction? Which slab class will evict first, and why?

**Model Answer.**

Compute the memory needed per slab class:

**Session objects (200 bytes each):**
- Fits in slab class 5 (240B chunks).
- Items per page: 1 MB / 240 B ≈ 4,369 items per page.
- Total pages needed: 10,000,000 / 4,369 ≈ 2,289 pages = 2,289 MB.

**API responses (8 KB each):**
- Fits in slab class approximately class 21 (8KB chunk).
- Items per page: 1 MB / 8 KB = 128 items per page.
- Total pages needed: 500,000 / 128 ≈ 3,907 pages = 3,907 MB.

**Page fragments (50 KB each):**
- Fits in slab class approximately class 26 (50KB chunk).
- Items per page: 1 MB / 50 KB ≈ 20 items per page.
- Total pages needed: 100,000 / 20 = 5,000 pages = 5,000 MB.

**Total: 2,289 + 3,907 + 5,000 = 11,196 MB ≈ 10.9 GB.**

The total fits within 16 GB. However, the problem is not total memory — it is how pages are claimed. If the application starts by storing all 10 million session objects first, slab class 5 claims 2,289 pages. Then when API responses are stored, slab class 21 claims pages from the remaining pool.

If all items are stored simultaneously, the total fits. However, if the application workload shifts over time (e.g., sessions grow to 12 million after the cache is already full with API responses and page fragments), slab class 5 would need more pages but none would be available. It would start evicting session objects even though other slab classes have free chunks.

**Eviction prediction.** Slab class 26 (page fragments) is the biggest consumer. If any slab class runs out, it will be class 26 first because it uses the most memory (5 GB). But with 5 GB of headroom (16 GB total - 10.9 GB used), all items fit without eviction during normal operation. Eviction would only occur if the workload grows beyond the initial estimate.

## Tier 2 — Architecture and Scale

### Challenge 2.1: Multi-Data-Center Cache Design

**Task.** You are designing the caching layer for a global social media platform. Requirements:
- Users are evenly distributed across US-East, US-West, and EU-West.
- Each user's data can be read from any region (they travel).
- The primary database is in US-East with async replication to other regions.
- Cache hit ratio target: above 90%.
- Cache data types: user profiles (2 KB), session data (500 B), feed fragments (10 KB).
- 100 million DAU.

**Question.** Design the Memcached deployment. Should you have one global cache or per-region caches? How do you handle cross-region reads? What happens during a region failover?

**Model Answer.**

**Architecture: Per-region independent Memcached pools.**

Each region has its own Memcached cluster. This is the only correct design because:
- Cross-region latency (60-200 ms round trip) would make a global cache slower than the database.
- Memcached has no cross-region replication — data cannot move between regions.
- Each region should be independently operable.

**Data placement:**
- User profiles: cached in the region where the user is primarily located (determined by the user's home region). When a user travels, their profile is cached in the visited region on first read (cache-aside populates it from the local database replica).
- Session data: cached in the region where the user logged in. If the user appears in another region, the session token must be readable. Solution: store session tokens with a region prefix. The application checks the user's home region first, then the local region.
- Feed fragments: cached per-region. Each region might show different trending content.

**Read path for a traveling user:**
1. User A (home: US-East) travels to EU-West.
2. Request arrives at EU-West app server.
3. App checks EU-West Memcached for User A's profile → miss (cold in this region).
4. App reads User A's profile from the EU-West database replica.
5. App populates EU-West Memcached with User A's profile (TTL: 15 minutes).
6. Future requests from EU-West hit the local cache.

**Region failover:**
If US-East becomes unavailable:
1. Traffic is redirected to US-West or EU-West.
2. Database reads go to the read replica in the active region.
3. Memcached in the active region starts cold for US-East user data.
4. Cache-aside pattern populates the cache over 10-20 minutes.
5. Database must handle the full read load during cache warmup.

### Challenge 2.2: Hot Key Mitigation

**Task.** Your social media app has a "celebrity" feature where verified users have a special profile page. One celebrity (user ID: 42) has 10 million followers. When this celebrity posts anything, their profile page receives 50,000 QPS within seconds. Your Memcached cluster has 10 nodes, each capable of 200,000 QPS. Consistent hashing maps user:42 to node 3.

**Question.** Node 3 is receiving 50,000 QPS for a single key plus its regular load of 10,000 QPS for other keys. Node 3 is at 30% CPU. The other 9 nodes are at 5% CPU each. Why is this a problem even though node 3 is not overloaded? What are three mitigation strategies?

**Model Answer.**

**Why it is a problem at 30% CPU:**
The problem is not CPU saturation — it is the single-flow bottleneck. All 50,000 QPS for user:42 go through one network interface, one hash table bucket (the bucket chain for user:42), and one LRU list position. Under the global cache lock, each request for user:42 must acquire the lock to touch the LRU list. The lock contention from 50,000 QPS on a single key can cause tail latency spikes for all keys on that node, not just the hot key.

Furthermore, if the celebrity posts something and the cache needs to be invalidated, 50,000 concurrent SET operations on the same key will cause CAS conflicts and lock contention.

**Mitigation 1: Local L1 cache.**
Each application server caches user:42's profile locally for 5-10 seconds. With 500 application servers, each server only sends 100 QPS to Memcached instead of 50,000 QPS total. The local cache absorbs the spike.

**Mitigation 2: Key replication.**
Store the celebrity's profile on 3-5 nodes instead of 1. The client randomly selects one replica to read from. This distributes the 50,000 QPS across multiple nodes (e.g., 10,000 QPS per node for 5 replicas).

```python
# Client-side replication for hot key
class HotKeyAwareClient:
    def __init__(self, nodes, hot_keys):
        self.nodes = nodes
        self.hot_keys = hot_keys  # {"user:42": {"replicas": [mc2, mc5, mc7]}}

    def get(self, key):
        if key in self.hot_keys:
            replicas = self.hot_keys[key]["replicas"]
            node = random.choice(replicas)
            return node.get(key)
        # Normal consistent hashing path
        return self.consistent_hash_ring.get(key)
```

**Mitigation 3: Throttle cache invalidations.**
When the celebrity's profile changes, do not SET a new value immediately. Instead, invalidate the key (DELETE) and let the cache-aside pattern repopulate it on the next read. This spreads the write load across the TTL window rather than concentrating it at the moment of the post.

**Question.** Design three strategies to prevent this cache stampede (thundering herd) when the TTL expires.

**Model Answer.**

**Strategy 1: Early recompute with lock.**
Use Memcached's `add` command (which only succeeds if the key does not exist) to elect a single recompute leader.

```python
def get_leaderboard():
    key = "leaderboard:weekly"
    result = cache.get(key)
    if result:
        return result

    # Key is missing — try to become recompute leader
    # add() succeeds only if key does not exist
    lock_key = f"{key}:lock"
    locked = cache.add(lock_key, "1", expire=10)

    if locked:
        # This server is the leader — recompute
        data = compute_leaderboard_from_db()
        cache.set(key, data, expire=300)
        cache.delete(lock_key)
        return data
    else:
        # Another server is computing — wait briefly and retry
        time.sleep(0.1)
        result = cache.get(key)
        if result:
            return result
        # Fallback: compute anyway (avoid infinite loop)
        return compute_leaderboard_from_db()
```

**Strategy 2: Random TTL jitter.**
Instead of all servers seeing the same TTL, add random jitter. Each server independently decides when to recompute based on a probabilistic factor.

```python
# Server-side: each server independently recomputes before
# the TTL expires, with random jitter spread over the TTL window
def get_leaderboard():
    key = "leaderboard:weekly"
    result = cache.gets(key)  # gets returns CAS value too

    if result is None:
        return recompute_and_cache(key)

    value, cas_id, stored_time = result
    age = time.time() - stored_time
    ttl = 300

    # Random early recompute: 10% chance per check when age > 80% of TTL
    if age > ttl * 0.8 and random.random() < 0.1:
        # Try CAS update — only one server succeeds
        new_value = compute_leaderboard_from_db()
        if cache.cas(key, new_value, cas_id, expire=ttl):
            return new_value
        # CAS failed — another server already updated
        return value

    return value
```

**Strategy 3: Stale-while-revalidate.**
Serve stale data while asynchronously refreshing in the background. The cache is never empty — the old value is always available.

```python
def get_leaderboard():
    key = "leaderboard:weekly"
    result = cache.gets(key)

    if result is None:
        return recompute_and_cache(key)

    value, cas_id, absolute_expiry = result
    now = time.time()
    soft_ttl = 240  # 80% of 300s — serve fresh
    hard_ttl = 600  # 200% of 300s — must refresh

    if now > absolute_expiry + hard_ttl:
        # Too old — must recompute synchronously
        return recompute_and_cache(key)

    if now > absolute_expiry + soft_ttl:
        # Stale but servable — trigger async refresh
        # (in production, send to background task queue)
        threading.Thread(target=recompute_and_cache, args=(key,)).start()

    return value  # Return stale value while refresh happens
```

- Routes read requests to the closest data center.

**Question.** Design the proxy's architecture. What data structures does it maintain? How does it handle the read-vs-write split? What consistency model does it provide?

**Model Answer.**

**Architecture.** The proxy is a stateless TCP proxy (similar to mcrouter or Twemproxy) that accepts Memcached protocol commands and forwards them to backend nodes. It runs as a process on each application server or as a standalone service.

**Data structures.**
1. **Server registry.** A map of data center → [node list]. Each node has an address, port, weight, and health status.
2. **Consistent hash ring per data center.** Maps keys to nodes within a data center.
3. **Health tracking.** Each node has a failure count and last-seen time. After 3 consecutive failures, the node is marked dead and removed from the ring. After 30 seconds, the proxy probes the node for reconnection.
4. **Connection pool.** A pool of TCP connections per backend node, managed by a connection pool with configurable min/max connections.

**Read-vs-write split.**
```
GET / GETS key:
  → Route to the closest data center's consistent hash ring
  → Return the value

SET / ADD / REPLACE / DELETE key:
  → Route to ALL data centers' consistent hash rings
  → Wait for all DCs to respond (or a quorum of 2 of 3)
  → Return STORED or error

CAS key value cas_token:
  → Route to ALL data centers
  → If any DC reports EXISTS (CAS conflict), return EXISTS
  → If all report STORED, return STORED
```

**Consistency model.** Read-your-writes consistency. A write goes to all data centers. Any subsequent read (even from a different data center) sees the write because the write was propagated globally. However, there is a propagation window: if the write to US-West takes 50 ms and the read comes 5 ms after the write, the read might hit US-East before the write arrives. This is eventual consistency with a window.

| Operation | Data Centers | Consistency |
|---|---|---|
| GET | Closest only | May miss recent writes (propagation delay) |
| SET | All | Strong (write propagated everywhere before returning) |
| DELETE | All | Strong (delete propagated everywhere before returning) |

**Failure handling.** If a data center's health check fails for all nodes, the proxy marks the entire DC as degraded. Reads route to the next-closest DC. Writes continue to all remaining healthy DCs. When the DC recovers, it is re-added to the routing table. Nodes that return to health within the DC are probed and restored to the ring.

### Challenge 3.2: Stateless Cache Warmup

**Task.** Your auto-scaling group replaces a Memcached node every 2 hours (due to spot instance termination, rolling upgrades, or crash). Each replacement creates a cache miss storm that spikes database CPU to 95%.

**Question.** Design a warmup mechanism that does not require maintaining a persistent database of cached keys. The warmup must complete in under 5 minutes and must not require changes to the application code.

**Model Answer.**

**Approach: request-level warmup via proxy shadowing.**

The proxy (mcrouter/Twemproxy) maintains a lightweight LRU-based bloom filter of recently seen cache keys (approximate most-accessed set). When a new node joins:

1. The proxy redirects GET requests for the new node's key range (determined by consistent hashing) to both the new node AND an existing (healthy) node.
2. If the existing node returns a HIT, the proxy also SETs the value on the new node.
3. The new node is populated passively — no explicit warmup loop needed.

```
Proxy warmup logic:

  1. Client sends: GET key:X
  2. Proxy checks consistent hash → key:X belongs to new_node
  3. Proxy forwards GET to: new_node (primary) AND existing_node (shadow)
  4. new_node returns: MISS (cold)
  5. existing_node returns: VALUE key:X ... {data}
  6. Proxy sends VALUE to client (from existing_node)
  7. Proxy sends SET key:X ... {data} to new_node (background)
  8. new_node now has key:X cached
```

**Without a proxy.** If you cannot use a proxy, the warmup can happen at the application level:

```python
# Application-level warmup for a known new node
def get_with_warmup(client, key):
    result = client.get(key)
    if result is None:
        # The key might be on a cold node — try an alternative read
        # In practice, this means reading from the database
        # and setting on the client (standard cache-aside)
        # The database takes the load, but only for this node's keys
        pass
    return result
```

**Metrics to verify warmup completion:**
- Node eviction rate: should be 0 during warmup (node is not full yet).
- Node miss rate: should drop from 100% to near zero within 5 minutes.
- Database CPU: should return to baseline when warmup completes.
- Node `curr_items`: should reach steady state (approximately match other nodes).

**Expected warmup time.** For a 32 GB node with 2,500 QPS of unique key requests (out of 30,000 total QPS), warmup takes approximately: 32 GB / (2,500 × 4 KB per item) ≈ 32 GB / 10 MB per second ≈ 3,200 seconds. This exceeds the 5-minute target. Optimizations:
- Reduce unique keys: serve hot keys first (proxy shadowing naturally does this because hot keys are requested more frequently).
- Increase request rate during warmup: temporarily redirect additional traffic to the warming node.
- Pre-warm: maintain a list of the top 100,000 keys and explicitly load them after node start.
