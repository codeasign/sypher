### Challenge 1.1: Draw the ER Diagram

A university has departments, professors, courses, and students. Each department has many professors. Each professor belongs to exactly one department. Each professor teaches many courses. Each course is taught by exactly one professor. Each course has many enrolled students. Each student can enroll in many courses.

**Question:** Draw the ER diagram showing entities, attributes (at least 3 per entity), and relationships with cardinality. Identify the primary key for each entity.

<details>
<summary>Model Answer</summary>

The ER diagram has four entities:

**Department**: department_id (PK), name, building, phone
**Professor**: professor_id (PK), first_name, last_name, email, department_id (FK)
**Course**: course_id (PK), title, credits, max_enrollment, professor_id (FK)
**Student**: student_id (PK), first_name, last_name, email, enrollment_date
**Enrollment**: enrollment_id (PK), course_id (FK), student_id (FK), grade, semester (junction table for many-to-many)

Relationships:
- Department 1--N Professor (one department has many professors)
- Professor 1--N Course (one professor teaches many courses)
- Course N--M Student (many-to-many, resolved via Enrollment junction table)
- Enrollment N--1 Course, N--1 Student (the junction table links both sides)

```sql
CREATE TABLE Department (
    department_id SERIAL PRIMARY KEY,
    name          VARCHAR(100) NOT NULL UNIQUE,
    building      VARCHAR(50) NOT NULL,
    phone         VARCHAR(20)
);

CREATE TABLE Professor (
    professor_id  SERIAL PRIMARY KEY,
    first_name    VARCHAR(50) NOT NULL,
    last_name     VARCHAR(50) NOT NULL,
    email         VARCHAR(200) NOT NULL UNIQUE,
    department_id INTEGER NOT NULL REFERENCES Department(department_id)
);

CREATE TABLE Course (
    course_id      SERIAL PRIMARY KEY,
    title          VARCHAR(200) NOT NULL,
    credits        SMALLINT NOT NULL CHECK (credits > 0 AND credits <= 6),
    max_enrollment INTEGER NOT NULL CHECK (max_enrollment > 0),
    professor_id   INTEGER NOT NULL REFERENCES Professor(professor_id)
);

CREATE TABLE Student (
    student_id     SERIAL PRIMARY KEY,
    first_name     VARCHAR(50) NOT NULL,
    last_name      VARCHAR(50) NOT NULL,
    email          VARCHAR(200) NOT NULL UNIQUE,
    enrollment_date DATE NOT NULL DEFAULT CURRENT_DATE
);

CREATE TABLE Enrollment (
    enrollment_id SERIAL PRIMARY KEY,
    course_id     INTEGER NOT NULL REFERENCES Course(course_id),
    student_id    INTEGER NOT NULL REFERENCES Student(student_id),
    grade         CHAR(2),
    semester      VARCHAR(20) NOT NULL,
    UNIQUE (course_id, student_id, semester)
);
```

</details>

### Challenge 1.2: Identify Relationships

You are given these three tables:

```sql
CREATE TABLE authors (
    author_id   SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL
);

CREATE TABLE books (
    book_id     SERIAL PRIMARY KEY,
    title       VARCHAR(200) NOT NULL,
    author_id   INTEGER REFERENCES authors(author_id)
);

CREATE TABLE book_categories (
    book_id     INTEGER REFERENCES books(book_id),
    category_id INTEGER REFERENCES categories(category_id),
    PRIMARY KEY (book_id, category_id)
);
```

**Question:** Identify the relationship type between each pair of tables. What does the `book_categories` table represent? Is the `author_id` in `books` mandatory or optional? How would you enforce that a book has at least one category?

<details>
<summary>Model Answer</summary>

- **authors to books**: One-to-many. One author writes many books. A book has one author. The FK `author_id` in `books` is NULLable, meaning the relationship is optional -- a book can exist without an author assigned.

- **books to categories**: Many-to-many. A book can belong to many categories (fiction, mystery, science). A category can contain many books. The `book_categories` table is the junction table that resolves this many-to-many relationship.

- **categories to book_categories and book_categories to books**: One-to-many. One category appears in many book_categories rows. One book appears in many book_categories rows.

To enforce that a book has at least one category, you have three options:

1. **Application-level enforcement**: Check in the application code that every book INSERT is followed by at least one INSERT into `book_categories`. This can fail in edge cases (application crash after book insert, concurrent operations).

2. **Deferred constraint**: Create a trigger that checks after INSERT or UPDATE that the book has at least one category entry. A deferred constraint runs at transaction commit time.

3. **Stored procedure**: Require all book creation to go through a stored procedure that inserts the book and its initial category in a single transaction.

The most reliable approach is the trigger with a deferred constraint:

```sql
CREATE FUNCTION check_book_category() RETURNS TRIGGER AS $$
BEGIN
    IF NOT EXISTS (SELECT 1 FROM book_categories WHERE book_id = NEW.book_id) THEN
        RAISE EXCEPTION 'Book must have at least one category';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE CONSTRAINT TRIGGER ensure_book_category
    AFTER INSERT OR UPDATE ON books
    DEFERRABLE INITIALLY DEFERRED
    FOR EACH ROW EXECUTE FUNCTION check_book_category();
```

</details>

### Challenge 1.3: Choose Data Types

A table stores product information for an e-commerce site. The columns needed are:

- A unique product identifier (internal, never shown to customers)
- A human-readable product SKU (shown in URLs, 8-16 characters, alphanumeric)
- Product name (up to 200 characters)
- Price in USD (up to $999,999.99, two decimal places)
- Weight in kilograms (up to 500.000 kg, three decimal places)
- In-stock flag (true/false)
- Description (up to 10,000 characters, rarely queried)
- Created timestamp

**Question:** Choose the optimal data types for each column. Justify each choice.

<details>
<summary>Model Answer</summary>

```sql
CREATE TABLE products (
    product_id   SERIAL PRIMARY KEY,
    -- SERIAL (4 bytes): sequential, fast for B-tree insertion.
    -- UUID is not needed for single-node internal IDs.

    sku          VARCHAR(16) NOT NULL UNIQUE,
    -- VARCHAR(16): variable length up to 16 chars.
    -- CHAR(16) would pad shorter values with spaces.
    -- VARCHAR is correct for variable-length strings.

    name         VARCHAR(200) NOT NULL,
    -- VARCHAR(200): up to 200 characters.
    -- TEXT would work but VARCHAR(200) documents the max length.

    price        DECIMAL(10,2) NOT NULL CHECK (price >= 0),
    -- DECIMAL(10,2): exact decimal, no floating-point rounding.
    -- 10 digits total, 2 after decimal: max 99,999,999.99.
    -- CHECK (price >= 0): no negative prices.

    weight       DECIMAL(6,3),
    -- DECIMAL(6,3): 6 digits total, 3 after decimal: max 999.999 kg.
    -- NULLable because weight may be unknown for digital products.

    in_stock     BOOLEAN NOT NULL DEFAULT TRUE,
    -- BOOLEAN (1 byte): exact fit for true/false.

    description  TEXT,
    -- TEXT (up to 1GB): PostgreSQL uses TOAST for large values.
    -- VARCHAR(10000) would have the same storage but TEXT is more
    -- conventional for long free-form text.

    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
    -- TIMESTAMPTZ (8 bytes): stores timezone-aware timestamp.
    -- TIMESTAMP (without TZ) would lose timezone information.
);
```

</details>

### Challenge 2.1: Ride-Sharing App Schema

Design the schema for a ride-sharing application.

**Requirements:**
- Riders can request a ride by specifying pickup and dropoff locations.
- Drivers can set their availability (online/offline) and accept ride requests.
- A ride goes through states: requested, driver_assigned, in_progress, completed, cancelled.
- Pricing is calculated based on distance, time, and surge multiplier.
- Each ride must have a payment record.
- Riders can rate drivers (1-5 stars) after each completed ride.
- Drivers can rate riders (1-5 stars) after each completed ride.

**Design the schema including:**
- All tables with columns, data types, and constraints
- Primary keys and foreign keys
- Indexes for the primary access patterns
- A brief justification for your primary key choice

<details>
<summary>Model Answer</summary>

```sql
-- Primary key choice: UUID v7 for all tables.
-- Ride-sharing is inherently distributed (riders and drivers connect
-- from any region) so globally unique IDs are needed.

CREATE TABLE riders (
    rider_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(100) NOT NULL,
    email       VARCHAR(200) NOT NULL UNIQUE,
    phone       VARCHAR(20) NOT NULL UNIQUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE drivers (
    driver_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    email           VARCHAR(200) NOT NULL UNIQUE,
    phone           VARCHAR(20) NOT NULL UNIQUE,
    license_plate   VARCHAR(20) NOT NULL,
    vehicle_model   VARCHAR(100) NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'offline'
                    CHECK (status IN ('online', 'offline', 'on_trip')),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE rides (
    ride_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rider_id        UUID NOT NULL REFERENCES riders(rider_id),
    driver_id       UUID REFERENCES drivers(driver_id),
    status          VARCHAR(20) NOT NULL DEFAULT 'requested'
                    CHECK (status IN ('requested', 'driver_assigned',
                           'in_progress', 'completed', 'cancelled')),
    pickup_lat      DECIMAL(9,6) NOT NULL,
    pickup_lng      DECIMAL(9,6) NOT NULL,
    dropoff_lat     DECIMAL(9,6),
    dropoff_lng     DECIMAL(9,6),
    distance_km     DECIMAL(8,3),
    duration_min    SMALLINT,
    surge_multiplier DECIMAL(3,2) NOT NULL DEFAULT 1.00,
    base_fare       DECIMAL(10,2) NOT NULL DEFAULT 0,
    total_fare      DECIMAL(10,2),
    requested_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    cancellation_reason VARCHAR(100)
);

CREATE TABLE payments (
    payment_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ride_id         UUID NOT NULL UNIQUE REFERENCES rides(ride_id),
    amount          DECIMAL(10,2) NOT NULL CHECK (amount > 0),
    method          VARCHAR(20) NOT NULL CHECK (method IN ('cash', 'card', 'wallet')),
    status          VARCHAR(20) NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'completed', 'failed', 'refunded')),
    transaction_id  VARCHAR(100),
    paid_at         TIMESTAMPTZ
);

CREATE TABLE ratings (
    rating_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ride_id         UUID NOT NULL UNIQUE REFERENCES rides(ride_id),
    rider_id        UUID NOT NULL REFERENCES riders(rider_id),
    driver_id       UUID NOT NULL REFERENCES drivers(driver_id),
    rider_score     SMALLINT CHECK (rider_score BETWEEN 1 AND 5),
    driver_score    SMALLINT CHECK (driver_score BETWEEN 1 AND 5),
    rider_comment   TEXT,
    driver_comment  TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes for primary access patterns
CREATE INDEX idx_rides_rider ON rides(rider_id, requested_at DESC);
-- Rider history: find all rides for a rider, newest first.

CREATE INDEX idx_rides_driver_status ON rides(driver_id, status)
    WHERE status IN ('requested', 'driver_assigned', 'in_progress');
-- Active rides for a driver: find current ride quickly.

CREATE INDEX idx_rides_status_requested ON rides(status, requested_at)
    WHERE status = 'requested';
-- Available rides: find all unassigned ride requests, oldest first.

CREATE INDEX idx_drivers_online ON drivers(status)
    WHERE status = 'online';
-- Find available drivers for ride assignment.
```

</details>

### Challenge 2.2: Food Delivery Platform Schema

Design the schema for a food delivery platform.

**Requirements:**
- Customers can browse restaurants and menus.
- Restaurants have multiple menu items, each with modifiers (size, extras).
- Customers place orders with items from a single restaurant.
- Each order goes through states: placed, confirmed, preparing, out_for_delivery, delivered.
- Delivery drivers are assigned to orders.
- Customers can track their order status in real time.
- Each order has a payment and can be split across multiple payment methods.

<details>
<summary>Model Answer</summary>

```sql
-- Core entities
CREATE TABLE customers (
    customer_id  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name         VARCHAR(100) NOT NULL,
    email        VARCHAR(200) NOT NULL UNIQUE,
    phone        VARCHAR(20) NOT NULL,
    address      TEXT NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE restaurants (
    restaurant_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name          VARCHAR(200) NOT NULL,
    cuisine_type  VARCHAR(50) NOT NULL,
    address       TEXT NOT NULL,
    phone         VARCHAR(20) NOT NULL,
    status        VARCHAR(20) NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'inactive', 'closed')),
    delivery_fee  DECIMAL(6,2) NOT NULL DEFAULT 0,
    min_order     DECIMAL(6,2) NOT NULL DEFAULT 0,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE menu_items (
    item_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    restaurant_id  UUID NOT NULL REFERENCES restaurants(restaurant_id),
    name           VARCHAR(200) NOT NULL,
    description    TEXT,
    base_price     DECIMAL(8,2) NOT NULL CHECK (base_price > 0),
    category       VARCHAR(50) NOT NULL,
    is_available   BOOLEAN NOT NULL DEFAULT TRUE,
    UNIQUE (restaurant_id, name)
);

CREATE TABLE menu_modifiers (
    modifier_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    item_id        UUID NOT NULL REFERENCES menu_items(item_id),
    name           VARCHAR(100) NOT NULL,
    choices        JSONB NOT NULL,  -- e.g., {"small": 0, "medium": 2, "large": 4}
    max_selections SMALLINT NOT NULL DEFAULT 1
);

-- Order entities
CREATE TABLE orders (
    order_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id       UUID NOT NULL REFERENCES customers(customer_id),
    restaurant_id     UUID NOT NULL REFERENCES restaurants(restaurant_id),
    driver_id         UUID REFERENCES drivers(driver_id),
    status            VARCHAR(20) NOT NULL DEFAULT 'placed'
                      CHECK (status IN ('placed', 'confirmed', 'preparing',
                             'out_for_delivery', 'delivered', 'cancelled')),
    delivery_address  TEXT NOT NULL,
    delivery_notes    TEXT,
    subtotal          DECIMAL(10,2) NOT NULL CHECK (subtotal > 0),
    delivery_fee      DECIMAL(6,2) NOT NULL,
    tax               DECIMAL(6,2) NOT NULL,
    total             DECIMAL(10,2) NOT NULL CHECK (total > 0),
    placed_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    delivered_at      TIMESTAMPTZ,
    estimated_arrival TIMESTAMPTZ
);

CREATE TABLE order_items (
    order_item_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id      UUID NOT NULL REFERENCES orders(order_id),
    item_id       UUID NOT NULL REFERENCES menu_items(item_id),
    quantity      SMALLINT NOT NULL CHECK (quantity > 0),
    unit_price    DECIMAL(8,2) NOT NULL,
    modifiers     JSONB,  -- e.g., {"size": "large", "extras": ["cheese"]}
    line_total    DECIMAL(10,2) NOT NULL
);

CREATE TABLE payments (
    payment_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id       UUID NOT NULL REFERENCES orders(order_id),
    amount         DECIMAL(10,2) NOT NULL CHECK (amount > 0),
    method         VARCHAR(20) NOT NULL CHECK (method IN ('card', 'cash', 'wallet')),
    status         VARCHAR(20) NOT NULL DEFAULT 'pending'
                   CHECK (status IN ('pending', 'completed', 'failed', 'refunded')),
    paid_at        TIMESTAMPTZ
    -- For split payments, multiple payment rows reference the same order_id
);

CREATE TABLE order_status_log (
    log_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id    UUID NOT NULL REFERENCES orders(order_id),
    status      VARCHAR(20) NOT NULL,
    timestamp   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Immutable log of every status change for real-time tracking.

-- Indexes
CREATE INDEX idx_orders_customer ON orders(customer_id, placed_at DESC);
CREATE INDEX idx_orders_restaurant ON orders(restaurant_id, placed_at DESC);
CREATE INDEX idx_orders_driver_active ON orders(driver_id, status)
    WHERE status IN ('confirmed', 'preparing', 'out_for_delivery');
CREATE INDEX idx_menu_items_restaurant ON menu_items(restaurant_id, category);
CREATE INDEX idx_order_status_log ON order_status_log(order_id, timestamp);
```

</details>

### Challenge 2.3: Content Management System Schema

Design the schema for a content management system that supports articles with multiple authors, categories, tags, version history, and scheduled publishing.

**Requirements:**
- Articles have a title, body (HTML), excerpt, featured image, and status (draft, published, archived).
- Articles can have multiple authors.
- Articles can belong to multiple categories.
- Articles can have unlimited tags.
- The system tracks version history for every article edit.
- Articles can be scheduled for future publication.
- The system supports an audit log of all user actions on articles.

<details>
<summary>Model Answer</summary>

```sql
CREATE TABLE users (
    user_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name       VARCHAR(100) NOT NULL,
    email      VARCHAR(200) NOT NULL UNIQUE,
    role       VARCHAR(20) NOT NULL DEFAULT 'editor'
               CHECK (role IN ('admin', 'editor', 'author', 'viewer')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE categories (
    category_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name          VARCHAR(100) NOT NULL UNIQUE,
    slug          VARCHAR(100) NOT NULL UNIQUE,
    description   TEXT,
    parent_id     UUID REFERENCES categories(category_id)
);

CREATE TABLE tags (
    tag_id  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name    VARCHAR(50) NOT NULL UNIQUE,
    slug    VARCHAR(50) NOT NULL UNIQUE
);

CREATE TABLE articles (
    article_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title            VARCHAR(300) NOT NULL,
    slug             VARCHAR(300) NOT NULL UNIQUE,
    excerpt          TEXT,
    body             TEXT NOT NULL,
    featured_image   VARCHAR(500),
    status           VARCHAR(20) NOT NULL DEFAULT 'draft'
                     CHECK (status IN ('draft', 'review', 'published', 'archived')),
    published_at     TIMESTAMPTZ,
    scheduled_at     TIMESTAMPTZ,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CHECK (scheduled_at IS NULL OR scheduled_at > created_at)
);

CREATE TABLE article_authors (
    article_id UUID NOT NULL REFERENCES articles(article_id) ON DELETE CASCADE,
    user_id    UUID NOT NULL REFERENCES users(user_id),
    role       VARCHAR(20) NOT NULL DEFAULT 'author'
               CHECK (role IN ('primary_author', 'co_author', 'contributor')),
    PRIMARY KEY (article_id, user_id)
);

CREATE TABLE article_categories (
    article_id  UUID NOT NULL REFERENCES articles(article_id) ON DELETE CASCADE,
    category_id UUID NOT NULL REFERENCES categories(category_id),
    PRIMARY KEY (article_id, category_id)
);

CREATE TABLE article_tags (
    article_id UUID NOT NULL REFERENCES articles(article_id) ON DELETE CASCADE,
    tag_id     UUID NOT NULL REFERENCES tags(tag_id),
    PRIMARY KEY (article_id, tag_id)
);

CREATE TABLE article_versions (
    version_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    article_id   UUID NOT NULL REFERENCES articles(article_id),
    version_num  INTEGER NOT NULL,
    title        VARCHAR(300) NOT NULL,
    body         TEXT NOT NULL,
    excerpt      TEXT,
    changed_by   UUID NOT NULL REFERENCES users(user_id),
    change_note  VARCHAR(500),
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (article_id, version_num)
);

CREATE TABLE audit_log (
    log_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id    UUID NOT NULL REFERENCES users(user_id),
    action     VARCHAR(50) NOT NULL,
    entity     VARCHAR(50) NOT NULL,
    entity_id  UUID NOT NULL,
    details    JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_articles_status ON articles(status, scheduled_at)
    WHERE status = 'draft' OR status = 'review';
CREATE INDEX idx_articles_published ON articles(published_at DESC)
    WHERE status = 'published';
CREATE INDEX idx_article_versions ON article_versions(article_id, version_num DESC);
CREATE INDEX idx_audit_log_entity ON audit_log(entity, entity_id, created_at DESC);
CREATE INDEX idx_audit_log_user ON audit_log(user_id, created_at DESC);
```

</details>

### Challenge 3.1: Multi-Tenant SaaS with Per-Tenant Customization

Design the schema for a multi-tenant SaaS application where each tenant (organization) can customize their data schema by adding custom fields.

**Requirements:**
- The application serves 1,000+ tenants on a shared database.
- All tenants share a common base schema (contacts, accounts, deals).
- Each tenant can define custom fields on any entity.
- Custom fields can be of types: text, number, date, boolean, dropdown, user.
- Custom field values must be queryable and indexable.
- Tenant A should see only Tenant A's data and custom fields.
- Performance must be acceptable with 100 million+ rows across all tenants.
- Adding a custom field should not require a database migration.

<details>
<summary>Model Answer</summary>

This is the classic schema design challenge for multi-tenant SaaS. Four approaches exist, in order of recommendation:

**Approach 1: JSONB with GIN Index (Recommended)**

Store tenant-specific custom fields as a JSONB column. The base schema handles all standard fields. Custom fields are stored as key-value pairs in JSONB, indexed with a GIN index for queryability.

```sql
CREATE TABLE contacts (
    id          UUID NOT NULL,
    tenant_id   UUID NOT NULL,
    -- Base schema (all tenants share these fields)
    first_name  VARCHAR(100) NOT NULL,
    last_name   VARCHAR(100) NOT NULL,
    email       VARCHAR(200),
    phone       VARCHAR(20),
    -- Custom fields stored as JSONB
    custom_fields JSONB NOT NULL DEFAULT '{}',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (tenant_id, id)
) PARTITION BY LIST (tenant_id);

-- GIN index for querying inside JSONB
CREATE INDEX idx_contacts_custom ON contacts USING GIN (custom_fields jsonb_path_ops);
```

Custom field definitions are stored in a metadata table:

```sql
CREATE TABLE custom_field_definitions (
    field_id    UUID PRIMARY KEY,
    tenant_id   UUID NOT NULL,
    entity_type VARCHAR(50) NOT NULL,  -- 'contact', 'account', 'deal'
    field_name  VARCHAR(100) NOT NULL,
    field_type  VARCHAR(20) NOT NULL CHECK (field_type IN ('text', 'number', 'date', 'boolean', 'dropdown', 'user')),
    is_required BOOLEAN NOT NULL DEFAULT FALSE,
    options     JSONB,  -- For dropdown: ["option1", "option2"]
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (tenant_id, entity_type, field_name)
);
```

Querying a custom field:

```sql
-- Find contacts where custom field "industry" = "technology"
SELECT * FROM contacts
WHERE tenant_id = 'tenant_abc'
  AND custom_fields @> '{"industry": "technology"}';
```

Partitioning by tenant ensures data isolation. Each tenant's data is stored in its own partition, making it efficient to query within a tenant. The GIN index on `custom_fields` supports efficient queries on any custom field.

**Approach 2: Sparse Columns**

Pre-define 10-20 generic columns (custom_text_1 through custom_text_5, custom_number_1 through custom_number_5, etc.) and map them to custom fields via a metadata table. Used by Salesforce and other early SaaS platforms. Simple but limited to the number of pre-defined columns.

**Approach 3: EAV (Entity-Attribute-Value) -- Not Recommended**

A separate table stores custom field values as rows. Extremely flexible but query performance degrades with scale because filtering requires JOINs or aggregation.

**Approach 4: Per-Tenant Dynamic Schema**

Create a separate schema or database per tenant. Provides full isolation and custom schema per tenant. Operation cost is high: managing 1,000+ schemas with migrations on each.

**Recommendation:** JSONB with GIN index for most multi-tenant SaaS applications. It balances flexibility, performance, and operational simplicity. Partition by tenant for data isolation and query efficiency.

</details>

### Challenge 3.2: Schema Evolution for an Event-Sourced System

Design a schema evolution strategy for a banking system that uses event sourcing. The system processes 10 million events per day across 50 event types.

**Requirements:**
- The system stores immutable events (no updates, no deletes).
- Each event has a type and type-specific payload.
- Event types evolve over time (new fields added, fields deprecated).
- Consumers should not break when new event versions are introduced.
- Old events must remain readable with the original schema.
- The system must support replaying events from any point in time.
- Schema evolution should not require downtime.

<details>
<summary>Model Answer</summary>

**Event store schema:**

```sql
CREATE TABLE events (
    event_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_id UUID NOT NULL,       -- ID of the entity this event belongs to
    aggregate_type VARCHAR(50) NOT NULL, -- 'account', 'transaction', 'customer'
    event_type   VARCHAR(100) NOT NULL,  -- 'account.created', 'funds.deposited'
    event_version SMALLINT NOT NULL,     -- Schema version of this event
    event_data   JSONB NOT NULL,         -- Type-specific payload
    metadata     JSONB NOT NULL DEFAULT '{}', -- Event metadata (correlation_id, causation_id)
    recorded_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for aggregate replay
CREATE INDEX idx_events_aggregate ON events(aggregate_type, aggregate_id, recorded_at);
-- Index for event type queries
CREATE INDEX idx_events_type ON events(event_type, recorded_at);
-- Partition by month for efficient time-range queries
CREATE INDEX idx_events_recorded_at ON events(recorded_at DESC);
```

**Schema versioning approach:**

Each event type has a version number. When the schema changes, the version number increments. The `event_version` column records which version of the schema the event conforms to.


**Consumer handling:**

Each consumer registers with the event types and versions it supports. The consumer code processes events with the version it understands.

```python
class FundsDepositedHandler:
    def can_handle(self, event):
        return (event["event_type"] == "funds.deposited"
                and event["event_version"] >= 1)

    def handle(self, event):
        data = event["event_data"]
        amount = data["amount"]
        currency = data.get("currency", "USD")  # default for legacy events
        deposit_method = data.get("deposit_method", "unknown")  # new field, default

        # Process event with safe defaults for missing fields
        process_deposit(amount, currency, deposit_method)
```

**Schema evolution rules:**

- **Add fields**: Always. New fields are optional (consumers provide defaults). Version increments.
- **Rename fields**: Never. Add the new field name alongside the old one. Deprecate the old field over 6 months.
- **Remove fields**: Never. Mark as deprecated in the schema registry. Consumers should ignore deprecated fields.
- **Change field type**: Never. Add a new field with the new type. Deprecate the old field.
- **Break changes**: Always require a new event type, not a version increment. For example, `funds.deposited_v2` instead of `funds.deposited` with version 3 if the change is breaking.

**Schema registry table:**

```sql
CREATE TABLE event_schema_registry (
    schema_id    UUID PRIMARY KEY,
    event_type   VARCHAR(100) NOT NULL,
    version      SMALLINT NOT NULL,
    schema_json  JSONB NOT NULL,  -- JSON Schema definition
    status       VARCHAR(20) NOT NULL DEFAULT 'active'
                 CHECK (status IN ('active', 'deprecated', 'retired')),
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deprecated_at TIMESTAMPTZ,
    UNIQUE (event_type, version)
);
```

**Replay strategy:**

To replay events for a specific aggregate:

```python
def replay_aggregate(aggregate_type, aggregate_id, up_to_timestamp=None):
    """Replay all events for an aggregate to rebuild its current state."""
    query = """
        SELECT * FROM events
        WHERE aggregate_type = %s AND aggregate_id = %s
        ORDER BY recorded_at ASC
    """
    params = [aggregate_type, aggregate_id]

    if up_to_timestamp:
        query += " AND recorded_at <= %s"
        params.append(up_to_timestamp)

    events = db.execute(query, params)

    state = {}
    for event in events:
        handler = get_handler(event["event_type"], event["event_version"])
        state = handler.apply(state, event)

    return state
```

The `handler.apply(state, event)` function is written to handle any version of the event. Each handler knows how to process all versions of its event type, applying the same defaults that the consumer uses in production.
