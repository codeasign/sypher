## How This Topic Appears

Schema design questions appear in almost every system design interview. They test whether you can translate vague requirements into a precise, implementable data model.

### Ten Common Phrasings

**Q1:** "Design the database schema for an e-commerce platform."

**Q2:** "Design a schema for a ride-sharing app like Uber or Lyft."

**Q3:** "How would you model a social media platform's data?"

**Q4:** "Design the schema for a hotel booking system."

**Q5:** "How would you design a schema for a real-time chat application?"

**Q6:** "Design a schema for a video streaming platform like Netflix."

**Q7:** "How would you model a multi-tenant SaaS application's data?"

**Q8:** "Design the schema for a food delivery service."

**Q9:** "How would you design a schema for a task management tool like Jira?"

**Q10:** "Design a schema for a payment processing system."

### Five Variations

**"Draw the ER diagram":** They want to see entities, relationships, and cardinality. Draw on the whiteboard or describe in words.

**"Write the CREATE TABLE statements":** They want to see data types, constraints, and indexes. Be specific about types (INT vs BIGINT, VARCHAR vs TEXT).

**"How would you handle [specific feature]?":** They test your ability to extend a schema for a new requirement. Show that you think about migrations.

**"What indexes would you create?":** They test your understanding of access patterns. List the queries and map each to an index.

**"How would this schema change at scale?":** They test your ability to plan for growth. Discuss partitioning, denormalization, and read replicas.

---

## What the Interviewer Is Testing

Interviewers look for four things in schema design questions:

**Can you translate requirements into a clean schema?** Given a vague description ("build a food delivery app"), can you identify the entities, relationships, and attributes? This tests your ability to extract structure from ambiguity.

**Do you think about access patterns?** The most common mistake is designing a schema that stores data correctly but cannot answer the queries efficiently. Interviewers want to see you ask "what queries will this schema serve?" before you write the CREATE TABLE statements.

**Do you understand constraints?** Using NOT NULL, UNIQUE, CHECK, and FOREIGN KEY constraints shows you understand data integrity. Omitting them suggests you think of databases as storage buckets, not integrity enforcers.

**Can you plan for scale?** Schema design at a startup is different from schema design at Google. Interviewers want to see that you can design a schema that works at 1,000 rows and can evolve to 1 billion rows.

## Level Expectations

### L3 (Entry Level)

At L3, the expectation is to produce a correct, normalized schema for a well-defined problem. You should identify the entities, define the relationships, and add basic constraints.

**Sample answer (L3):** "For a hotel booking system, I would create a `hotels` table with columns for name, address, and star rating. A `rooms` table with room type and price. A `guests` table with name and email. A `bookings` table that connects a guest to a room for specific dates. The bookings table has a foreign key to guests and a foreign key to rooms."

**What is missing:** Indexes, constraints, handling of edge cases (overbooking, partial cancellations), access patterns, partitioning strategy.

### L4 (Mid Level)

At L4, you add constraints, indexes, and discuss access patterns. You show awareness of common pitfalls.

**Sample answer (L4):** "I would add a UNIQUE constraint on guest email and a CHECK constraint on `star_rating BETWEEN 1 AND 5`. I would create a composite index on `bookings(room_id, check_in, check_out)` to quickly find available rooms. To prevent overbooking, I would add a CHECK constraint that ensures `check_out > check_in`. I would store the nightly price in the `rooms` table and the total price in the `bookings` table as a denormalized column for fast queries."

**What is missing:** Migration strategy, partitioning, denormalization trade-offs, multi-region considerations.

### L5 (Senior)

At L5, you discuss trade-offs explicitly. You compare options and justify your choices. You plan for schema evolution.

**Sample answer (L5):** "I would choose UUID v7 for primary keys because this system will eventually shard across databases. The time-ordered nature of UUID v7 avoids the random-insert performance problem of UUID v4. For the `bookings` table, I would use a composite key of `(property_id, room_id, booking_date)` as the clustering key to keep bookings for the same property physically co-located. I would partition the `bookings` table by month to support efficient date-range queries. For the migration path, I would use the expand-migrate-contract pattern: first add new columns alongside existing ones, then backfill, then drop the old columns."

**What is missing:** Real-world operational experience, nuanced failure mode knowledge.

### L6 (Staff)

At L6, you discuss failure modes, operational concerns, and team-level schema governance. You show experience with production incidents.

**Sample answer (L6):** "I would implement a schema-as-contract approach using Avro with a schema registry. The bookings schema is a contract between the booking service and the analytics pipeline. Any schema change goes through a compatibility check. I would enforce a SQL review process: every migration that touches a table with over 10 million rows requires a second reviewer and a runbook for rollback. For the hot `bookings` table, I would use a tool like `pt-online-schema-change` to avoid locking during migrations. I would set up drift detection as a CI check: every deployment compares the schema to the migration definitions."

**What is missing:** Organizational strategy, long-term schema governance for 50+ teams.

### L7 (Principal)

At L7, you discuss organizational schema governance, standardization across teams, and long-term data architecture strategy.

**Sample answer (L7):** "I would establish a data modeling guild that defines naming conventions, data type standards, and migration policies across all 50+ teams. I would mandate that every new service produces a data model document before writing migrations. I would introduce a schema review board with rotating membership from each team. For the hotel booking domain specifically, I would champion migrating from a monolithic payments schema to an event-sourced approach, because the current mutable schema causes a production incident every quarter when a payment status migration goes wrong."

## FAANG Framework

Use this six-step framework for any schema design interview question.

### Step 1: Gather Entities

List the nouns in the requirements. Each noun is a candidate entity.

```
Requirements: "Design a schema for a food delivery platform where
customers can order food from restaurants, drivers deliver orders,
and customers can rate their experience."

Nouns: customer, restaurant, order, driver, rating
Entities: Customer, Restaurant, Order, Driver, Rating
```

Validate each entity: does it have attributes? Does it participate in relationships? If a noun is just a label (e.g., "experience" in the requirements above), it is not a separate entity -- it is an attribute of another entity (Rating).

### Step 2: Identify Relationships

Map how entities connect. Use the Entity-Relationship notation.

```
Customer --places--> Order
Restaurant --receives--> Order
Order --has--> Driver
Customer --writes--> Rating
Order --has--> Rating
```

Convert to cardinality:

```
Customer 1--N Order (one customer places many orders)
Restaurant 1--N Order (one restaurant receives many orders)
Order 1--1 Driver (one order is delivered by one driver)
Customer 1--N Rating (one customer writes many ratings)
Order 1--1 Rating (one order has one rating)
```

### Step 3: Define Attributes

Add columns for each entity. Start with the obvious ones, then add edge-case columns.

```
Customer: id, name, email, phone, address, created_at
Restaurant: id, name, cuisine_type, address, phone, status, created_at
Order: id, customer_id, restaurant_id, driver_id, status, 
       total_amount, delivery_address, placed_at, delivered_at
Driver: id, name, phone, vehicle_type, status, current_location
Rating: id, order_id, customer_id, score, comment, created_at
```

### Step 4: Add Constraints

Attach database-level rules.

```sql
CREATE TABLE customers (
    id          UUID PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    email       VARCHAR(200) NOT NULL UNIQUE,
    phone       VARCHAR(20),
    address     TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE restaurants (
    id          UUID PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    cuisine_type VARCHAR(50) NOT NULL,
    address     TEXT NOT NULL,
    phone       VARCHAR(20) NOT NULL,
    status      VARCHAR(20) NOT NULL DEFAULT 'active'
                CHECK (status IN ('active', 'inactive', 'closed')),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE orders (
    id              UUID PRIMARY KEY,
    customer_id     UUID NOT NULL REFERENCES customers(id),
    restaurant_id   UUID NOT NULL REFERENCES restaurants(id),
    driver_id       UUID REFERENCES drivers(id),
    status          VARCHAR(20) NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'accepted', 'preparing',
                           'picked_up', 'delivered', 'cancelled')),
    total_amount    DECIMAL(10,2) NOT NULL CHECK (total_amount > 0),
    delivery_address TEXT NOT NULL,
    placed_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    delivered_at    TIMESTAMPTZ
);

CREATE TABLE drivers (
    id              UUID PRIMARY KEY,
    name            VARCHAR(100) NOT NULL,
    phone           VARCHAR(20) NOT NULL UNIQUE,
    vehicle_type    VARCHAR(20) NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'offline'
                    CHECK (status IN ('online', 'offline', 'on_delivery')),
    current_location POINT
);

CREATE TABLE ratings (
    id          UUID PRIMARY KEY,
    order_id    UUID NOT NULL UNIQUE REFERENCES orders(id),
    customer_id UUID NOT NULL REFERENCES customers(id),
    score       SMALLINT NOT NULL CHECK (score BETWEEN 1 AND 5),
    comment     TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Step 5: Consider Access Patterns

List the queries and add indexes.

```sql
-- Access Pattern 1: Find orders by customer (order history page)
CREATE INDEX idx_orders_customer ON orders(customer_id, placed_at DESC);

-- Access Pattern 2: Find orders by restaurant (restaurant dashboard)
CREATE INDEX idx_orders_restaurant ON orders(restaurant_id, placed_at DESC);

-- Access Pattern 3: Find available drivers near a location
CREATE INDEX idx_drivers_status_location ON drivers(status) 
    WHERE status = 'online';

-- Access Pattern 4: Find active orders for a driver
CREATE INDEX idx_orders_driver_active ON orders(driver_id)
    WHERE status NOT IN ('delivered', 'cancelled');

-- Access Pattern 5: Prevent duplicate ratings for the same order
-- (Already enforced by UNIQUE on ratings.order_id)
```

### Step 6: Plan Migrations

Describe how the schema will evolve as the product grows.

```
Migration 1: Add restaurant menu (5 new tables: menu_items, modifiers, categories)
Migration 2: Add promo codes (promo_codes table, order_promos link table)
Migration 3: Split restaurants into franchise model (add franchise_id FK)
Migration 4: Partition orders by month (range partition on placed_at)
Migration 5: Add rating categories (food_rating, delivery_rating, packaging_rating)
```

## Phrase Bank

Use these phrases in your interview answers to demonstrate depth.

### On Entities

- "The core entities in this domain are..."
- "I would identify these bounded contexts:..."
- "This entity is a value object -- it does not need its own identity."
- "This entity is a weak entity -- it depends on its parent for existence."

### On Relationships

- "The relationship between X and Y is one-to-many, enforced by a foreign key on the many side."
- "This is a many-to-many relationship, so I need a junction table."
- "The modality of this relationship is optional -- a customer may have zero orders."
- "I would model this as a self-referencing foreign key for the hierarchy."

### On Constraints

- "I would enforce this constraint at the database level, not the application level."
- "A NOT NULL constraint here prevents phantom orders without required fields."
- "A UNIQUE constraint on (team_id, channel_name) prevents duplicate channel names within a team."
- "A CHECK constraint ensures that check_out is always after check_in."
- "A foreign key with ON DELETE RESTRICT prevents accidental deletion of referenced data."

### On Indexes

- "The primary access pattern for this table is..., so I would create a composite index on..."
- "I would use a partial index here because only active records need fast lookups."
- "A covering index on (column1, column2) INCLUDE (column3) would make this query index-only."
- "I would avoid indexing this column because it has low cardinality and the query planner would ignore it."

### On Schema Evolution

- "I would use the expand-migrate-contract pattern for this schema change."
- "This migration is backward-compatible because the new column has a default value."
- "I would add the constraint as NOT VALID first, then validate it in a separate transaction."
- "For zero-downtime deployment, I would split this migration into three phases."

### On Trade-offs

- "The trade-off here is between consistency and flexibility. I would choose..."
- "Normalizing this table would reduce duplication but require a JOIN for every query."
- "Using UUIDs instead of auto-increment IDs gives us global uniqueness at the cost of larger indexes."
- "Denormalizing this field into the fact table avoids a JOIN but introduces data duplication."

## Red Flags

Interviewers watch for these eight mistakes. Avoid them.

**Red Flag 1: No primary key.** Every table must have a primary key. If you forget to specify one, the interviewer will notice. Answer with: "Every table has a primary key. For this system, I would use [UUID v7 / auto-increment INT]."

**Red Flag 2: No foreign keys.** Relationships between tables should be enforced by foreign keys. If you say "the orders table has a customer_id column" without saying "REFERENCES customers(id)", the interviewer will ask about referential integrity.

**Red Flag 3: All columns are VARCHAR(255).** This is the most common amateur mistake. Use appropriate data types: INT for IDs, DECIMAL for prices, DATE for dates, BOOLEAN for flags. If you use VARCHAR for everything, you signal that you do not understand data types.

**Red Flag 4: No constraints.** If your schema has no NOT NULL, UNIQUE, or CHECK constraints, the interviewer will ask: "What prevents a user from registering with a duplicate email? What prevents a negative price?" Have constraints ready for every column.

**Red Flag 5: No consideration of access patterns.** If you design a schema without discussing queries, the interviewer will ask: "How would you find all orders for a customer placed in the last month?" If you have not added an index for that query, you have not designed the schema.

**Red Flag 6: Over-normalization.** A table for every micro-entity with 10 levels of JOINs for a simple query is a red flag. Show that you understand when to normalize and when to denormalize. "I would keep the address in the orders table rather than creating a separate addresses table because addresses are queried with every order and rarely reused."

**Red Flag 7: No discussion of scale.** If the interviewer asks "How does this schema perform at 10 million rows?" and you have not considered partitioning, archiving, or read replicas, you are not thinking about scale. Answer with: "At 10 million rows, I would partition the orders table by month and archive orders older than 12 months."

**Red Flag 8: No migration plan.** If you propose a schema change that alters an existing table with millions of rows, the interviewer will ask: "How do you deploy this without downtime?" If you have not mentioned the expand-migrate-contract pattern, online schema change tools, or rolling deployments, you are missing a key production concern.

## Sample Dialogue

Here is how a strong interview answer flows:

**Interviewer:** "Design a schema for a simple task management tool like Trello or Jira."

**Candidate:** "Let me start by identifying the entities. A task management tool has boards, lists, cards, and users. Let me list them out.

(Board, List, Card, User, Comment)

Now let me think about the relationships. A board has many lists. A list has many cards. A card has many comments. A user can belong to many boards, and a board has many users -- that is a many-to-many relationship, so I need a junction table called board_members.

For the attributes:
- Board: id, name, description, created_at
- List: id, board_id, name, position (for ordering)
- Card: id, list_id, title, description, due_date, assigned_to, position, created_at
- User: id, name, email
- BoardMember: board_id, user_id, role
- Comment: id, card_id, author_id, body, created_at

For constraints, I would use:
- NOT NULL on name, title, email, body
- UNIQUE on user email
- FOREIGN KEY on all relationships
- CHECK on role IN ('admin', 'member', 'viewer')
- UNIQUE on (board_id, name) for list names within a board

For the primary key, I would use UUID v7 because this system will likely grow to multiple databases as teams adopt it.

The primary access pattern is loading a board with all its lists and cards. For that, I need indexes on:
- list(board_id, position) -- to load lists in order
- card(list_id, position) -- to load cards in order
- board_member(user_id) -- to find boards a user belongs to

For labeling, I would use an ENUM for the role column or a small reference table since roles are unlikely to change often.

For future migrations, I would add a labels/tags table, an attachments table, and a card_activity_log table for audit trails. I would partition the activity log by month since it will grow quickly."

This answer covers entities, relationships, constraints, primary key choice, access patterns, indexes, and future planning. It demonstrates the full six-step framework.

## Practice Template

Use this template when practicing schema design questions on a whiteboard or in a code editor.

```
Step 1: Gather Entities
  - List the nouns from the requirements
  - Circle which ones are entities (have attributes, participate in relationships)
  - Cross out which ones are attributes of other entities

Step 2: Identify Relationships
  - Draw lines between entities
  - Label cardinality: 1:1, 1:N, M:N
  - Add modality: mandatory or optional

Step 3: Define Attributes
  - For each entity, list 3-5 attributes
  - Identify the primary key
  - Choose data types

Step 4: Add Constraints
  - NOT NULL: which attributes are required?
  - UNIQUE: which attributes must be unique?
  - CHECK: what range or pattern rules apply?
  - FOREIGN KEY: which attributes reference other tables?

Step 5: Consider Access Patterns
  - List the 3-5 most common queries
  - For each query, specify the WHERE and ORDER BY columns
  - Create indexes for those columns

Step 6: Plan Migrations
  - What new features will be added in the next 6 months?
  - How will the schema evolve to support them?
  - What is the rollback plan for each migration?
```

### Five More Practice Questions

**Practice 1:** Design a schema for a library management system. Books can be checked out by members. Members have a borrowing limit. Late returns incur fines.

**Practice 2:** Design a schema for a movie streaming service. Users can browse movies by genre, create watchlists, and rate movies they have watched.

**Practice 3:** Design a schema for an inventory management system. Warehouses store products. Products can be in multiple warehouses with different quantities. Orders deduct from inventory.

**Practice 4:** Design a schema for a ride-sharing platform with scheduled rides. Riders can book a ride for a future time. Drivers can accept scheduled rides in advance.

**Practice 5:** Design a schema for a survey platform. Surveys have multiple questions. Questions can be multiple-choice, text, or rating scale. Responses are collected and aggregated.

### Common Interview Mistakes and How to Fix Them

**Mistake 1: Forgetting to ask clarifying questions.**

Many candidates start writing CREATE TABLE statements before understanding the requirements. Always ask: "What are the primary use cases? What is the expected scale? What are the most common queries?"

Fix: Spend the first 2 minutes asking questions. Write the answers on the whiteboard. Then start designing.

**Mistake 2: Over-normalizing every entity.**

A separate lookup table for every enum value (order_status, gender, salutation, title) creates a schema with 30 tiny tables and 15 JOINs for a simple query.

Fix: Use ENUM types for small, stable sets (under 20 values). Use reference tables only for values that change frequently or have additional attributes (sort order, display label, description).

**Mistake 3: Ignoring the null problem.**

Many candidates do not specify which columns are NULLable and which are NOT NULL. The interviewer notices.

Fix: After writing each CREATE TABLE, mentally INSERT a row with all important columns. If any INSERT would produce garbage data, add a NOT NULL or CHECK constraint.

**Mistake 4: Designing only for the happy path.**

The schema should handle edge cases: What happens when a user deletes their account? What happens when a product is discontinued? What happens when a booking is cancelled?

Fix: For each entity, ask "what happens when this entity is deleted, updated, or queried in a way that violates the relationship?"

**Mistake 5: Using SERIAL/BIGSERIAL without discussing scale.**

Auto-increment primary keys are fine for single-node systems but break in distributed systems.

Fix: Ask "is this system single-node or distributed?" If distributed, use UUID v7. If single-node, auto-increment INT is better.

**Mistake 6: Not mentioning indexes.**

A schema without indexes is a schema that stores data but cannot retrieve it efficiently.

Fix: After designing the tables, list the 3-5 most common queries. For each query, say "I would create an index on (columns) for this query."

**Mistake 7: Adding too many indexes.**

Every index slows down writes. A table with 15 indexes on a write-heavy workload will have slow INSERT performance.

Fix: Start with indexes only for the critical queries. Add indexes as query profiling reveals needs. A general rule: no more than 5-7 indexes per table for OLTP workloads.

**Mistake 8: Not planning for growth.**

"10 million rows? My schema handles it fine." -- Every candidate who did not think about partitioning.

Fix: Mention partitioning strategy, archival of old data, and read replica scaling for the schema. Even if the answer is "partition by customer_id range," it shows you think about scale.

### How to Handle Follow-up Questions

Interviewers often start with a simple schema question and then add constraints to test your depth.

**Follow-up 1:** "Now this system needs to support multi-language content. How does your schema change?"

Answer: Add a `locale` column to translatable tables. Create a separate translations table for content that needs multiple translations per row.

**Follow-up 2:** "Now we need soft deletes -- users can restore deleted data within 30 days."

Answer: Add a `deleted_at TIMESTAMPTZ` column to each table. Queries that should show only active records add `WHERE deleted_at IS NULL`. A background job permanently removes records where `deleted_at` is older than 30 days.

**Follow-up 3:** "Now the system needs an audit trail -- every change to every record must be tracked."

Answer: Create a generic audit_log table with columns: table_name, record_id, old_values (JSONB), new_values (JSONB), changed_by, changed_at. Use database triggers or application-level callbacks to populate it.

**Follow-up 4:** "Now the schema must support tenant isolation for a multi-tenant SaaS."

Answer: Add a `tenant_id` column to every table. Make it the first column of every primary key and every index. Add a row-level security policy that filters by `tenant_id` automatically.

**Follow-up 5:** "Now we need to handle 1 billion rows in the orders table."

Answer: Partition by month or by tenant. Archive orders older than 12 months to a separate cold storage table. Consider moving to a columnar storage format for analytical queries on the archived data.

### Before the Interview: A Preparation Checklist

In the week before your system design interview, review this checklist:

- Can you draw an ER diagram from a verbal description in under 3 minutes?
- Do you know the storage sizes of INT, BIGINT, UUID, VARCHAR, DECIMAL, TIMESTAMP?
- Can you list the five constraint types (PK, FK, UNIQUE, NOT NULL, CHECK)?
- Do you know when to use ENUM vs a reference table?
- Can you explain the expand-migrate-contract pattern?
- Have you practiced the six-step FAANG framework at least 5 times?
- Can you discuss the trade-off between UUID and auto-increment PKs?
- Do you know how to design for a multi-tenant schema?
- Can you describe how schema design affects query performance at scale?

If you can answer all nine questions confidently, you are ready for schema design questions at any level.

### Sample Answer: E-Commerce Schema

Here is a complete sample answer for the most common schema design question: "Design a schema for an e-commerce platform."

**Candidate:** "Let me apply the six-step framework.

Step 1: Entities. The core entities are Customer, Product, Order, OrderItem, Category, and Cart. Secondary entities are Review, Shipment, and Payment.

Step 2: Relationships. Customer 1--N Order. Order 1--N OrderItem. Product 1--N OrderItem. Product N--1 Category. Customer 1--N Review. Product 1--N Review. Order 1--1 Shipment. Order 1--1 Payment.

Step 3: Attributes. Customer has id, name, email, phone, address, created_at. Product has id, name, description, price, sku, stock_quantity, category_id. Order has id, customer_id, status, total, shipping_address, placed_at, delivered_at.

Step 4: Constraints. NOT NULL on name, email, price, sku. UNIQUE on email and sku. CHECK on price > 0 and status IN list. FOREIGN KEY on customer_id, product_id, category_id. For the Cart, I would use a UNIQUE constraint on (customer_id, product_id) to prevent duplicate cart entries.

Step 5: Access patterns. The most common queries are: find orders by customer (index on customer_id + placed_at), find products by category (index on category_id), search products by name (full-text index), and check stock for a product (index on product_id with stock_quantity).

Step 6: Migrations. I would plan for adding a product_variants table (sizes, colors), a wishlist table, and a promo_codes table. Each migration would use the expand-migrate-contract pattern.

For the primary key, I would use UUID v7 because this platform will likely shard across databases as it grows. The time-ordered nature of UUID v7 avoids the random-insert performance degradation of UUID v4."

This answer covers all six steps, shows awareness of constraints and indexes, and discusses future evolution. It would score well at the L4-L5 level.

### Summary of Key Signals

Interviewers evaluate schema design answers on these signals:

- **Precision**: Exact data types, not generic "string" or "number". DECIMAL(10,2) not just "decimal". VARCHAR(200) not just "varchar".
- **Completeness**: Every table has a PK. Every FK has REFERENCES. Every column has a data type.
- **Constraint awareness**: NOT NULL, UNIQUE, CHECK, and FOREIGN KEY are used intentionally, not as afterthoughts.
- **Access pattern thinking**: Indexes are proposed for specific queries, not all columns.
- **Evolution mindset**: Migration paths are planned before the schema is created.
- **Trade-off articulation**: Normalization vs denormalization, UUID vs auto-increment, NULL vs NOT NULL -- all discussed with reasoning.
- **Scale awareness**: Partitioning, archiving, read replicas are mentioned at the appropriate point.

### Timing Yourself

In a real interview, you have 15-20 minutes for the schema design portion of a system design question. Practice with a timer:

- **Minutes 0-3**: Gather entities and identify relationships. Draw the ER diagram.
- **Minutes 3-8**: Define attributes, data types, and constraints. Write the CREATE TABLE statements.
- **Minutes 8-12**: Consider access patterns and add indexes. Discuss trade-offs.
- **Minutes 12-15**: Discuss migration plan and scalability. Mention partitioning, archiving, and read replicas.
- **Minutes 15-20**: Answer follow-up questions and edge cases.

If you finish the schema in 10 minutes, you have time to discuss the deeper trade-offs. If you are still defining entities at 10 minutes, you are going too slow -- skip the detailed data types and focus on the structure.

## Key Takeaways

- Schema design questions appear in every system design interview. They test data modeling, constraint awareness, and scalability thinking.
- The FAANG framework has six steps: gather entities, identify relationships, define attributes, add constraints, consider access patterns, plan migrations.
- Level expectations increase from L3 (correct schema) to L7 (organizational schema governance).
- Use the phrase bank to demonstrate depth: bounded contexts, composite indexes, partial indexes, expand-migrate-contract, backward compatibility.
- Avoid the eight red flags: no PK, no FKs, all VARCHAR, no constraints, no access patterns, over-normalization, no scale discussion, no migration plan.
- Every constraint you add signals that you understand data integrity. Every index you propose signals that you understand performance.
