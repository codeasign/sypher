## Core Tension

The fundamental tradeoff in schema design is between rigidity and flexibility.

**Rigid schemas** provide safety and consistency but resist change. Every column has a fixed type. Every constraint is enforced. The database guarantees data integrity. Changing the schema requires a migration, which requires a deployment, which requires downtime or the expand-migrate-contract pattern.

**Flexible schemas** adapt quickly but shift correctness responsibility to application code. Tables accept any data. Columns can be added without migrations. But the database does not enforce data quality. A bug in the application produces corrupt data that the database accepts without complaint.

```
Rigid schema (safe, slow to change):
  Column: price DECIMAL(10,2) NOT NULL CHECK (price > 0)
  Database rejects: price = -5, price = 'abc', price = NULL

Flexible schema (adaptable, risk of corruption):
  Column: price VARCHAR(50)
  Database accepts: price = 'abc', price = '-5', price = NULL
  Application must validate every value in every code path
```

The tension is not resolvable in a universal sense. The right choice depends on the stability of the data model, the cost of data corruption, and the velocity of feature development.

## Decision Matrix

### Strict vs Loose Schema

| Factor | Choose Strict | Choose Loose |
|---|---|---|
| Data model stability | Stable, well-understood | Rapidly evolving, experimental |
| Cost of corruption | High (financial, medical) | Low (analytics, logs) |
| Development velocity | Moderate | High (no migrations) |
| Query performance | Fast (optimizer knows types) | Slow (implicit casts, no indexes) |
| Enforcement location | Database | Application code |

**Decision rule**: Start strict for transactional data (customers, orders, payments). Start loose for event data, logs, and user-generated content where the structure is unknown ahead of time.

### Normalized vs Denormalized

| Factor | Normalized | Denormalized |
|---|---|---|
| Storage | Minimal (no redundancy) | Higher (duplicated data) |
| Write speed | Fast (single table) | Slower (multiple tables updated) |
| Read speed (single row) | Fast (JOIN with indexes) | Fastest (one table, one read) |
| Read speed (aggregation) | Slow (JOIN many tables) | Fast (one table) |
| Data integrity | High (single source of truth) | Risk of inconsistency |
| Migration complexity | Low (change one table) | High (change everywhere) |

**Decision rule**: Normalize for transactional systems where data integrity is critical. Denormalize for read-heavy analytical systems and caches.

### UUID vs Auto-Increment Primary Key

| Factor | Auto-Increment (INT/BIGINT) | UUID v4 | UUID v7 |
|---|---|---|---|
| Storage | 4-8 bytes | 16 bytes | 16 bytes |
| Insert performance | Sequential (no page splits) | Random (page splits) | Sequential (time-ordered) |
| Index size | Small | 2-4x larger | 2-4x larger |
| Global uniqueness | No | Yes | Yes |
| Sequentiality visible | Yes (exposes row count) | No | Partial (exposes time) |
| Merge friendly | No (conflicts) | Yes | Yes |

**Decision rule**: Use auto-increment for single-node systems. Use UUID v7 for distributed systems. Use UUID v4 only when the sequential nature of UUID v7 is unacceptable (security concerns about exposing creation order).

### Single-Table Inheritance vs Joined Tables

| Factor | Single-Table | Class-Table | Concrete-Table |
|---|---|---|---|
| Simplicity | Simple (one table) | Complex (N tables) | Moderate (N tables, no base) |
| NULL columns | Many | None | None |
| Query performance | Fast (no JOIN) | Slow (JOIN always) | Fast (per type) |
| Cross-type queries | Easy | Expensive | Impossible |
| Adding a subclass | Add columns | Add table + FK | Add table |
| Constraint enforcement | No per-type control | Strong | Strong |

**Decision rule**: Use single-table for simple hierarchies with few subclasses and few unique attributes. Use class-table for complex hierarchies with many subclass-specific attributes. Avoid concrete-table unless you never query across types.

### Embedded vs Referenced (Document Databases)

| Factor | Embedded | Referenced |
|---|---|---|
| Read performance | Fast (one document read) | Slower (multiple reads or JOINs) |
| Write performance | Single write (update one doc) | Multiple writes (update doc + references) |
| Document size | Larger (nested data included) | Smaller (just IDs) |
| Data duplication | Possible (same data in multiple docs) | Minimal (single source of truth) |
| Consistency | Risk of divergence | Single source of truth |
| Max nesting | 100 levels (MongoDB limit) | No limit |

**Decision rule**: Embed when the child data is always accessed with the parent and rarely changes independently. Reference when the child data is large, shared across parents, or accessed independently.

### NULLable vs NOT NULL

| Factor | NULLable | NOT NULL |
|---|---|---|
| Storage | 1 bit per column in NULL bitmap | No overhead |
| Query complexity | Complex (IS NULL, COALESCE, NULL-safe comparisons) | Simple |
| Integrity | Cannot enforce "this value must exist" | Enforced by database |
| Migration flexibility | High (can add without backfill) | Low (must backfill before adding) |

**Decision rule**: Default to NOT NULL. Only use NULLable for columns where the absence of a value is meaningful (e.g., optional phone number, optional middle name). Never use NULLable as a lazy way to avoid deciding whether a column is required.

### Text vs Enum vs Reference Table

| Factor | TEXT | ENUM | Reference Table |
|---|---|---|---|
| Storage | 1-256 bytes per value | 1-2 bytes (internal integer) | 4 bytes (FK to reference table) |
| Validation | None (application checks) | Database enforces | FK + DB enforces |
| Flexibility | High (any string accepted) | Low (ALTER TYPE to add) | High (INSERT into lookup table) |
| Query performance | Slow (string comparison) | Fast (integer comparison) | Fast (integer FK comparison) |
| Documentation | None | Schema documents values | Table documents values |

**Decision rule**: Use ENUM for small, stable sets (under 20 values) that rarely change: status, gender, currency code. Use reference tables for medium-sized sets that change occasionally: product categories, shipping methods, country codes. Use TEXT only for values that are truly free-form with no predefined set.

## CAP Positioning

Schema enforcement requires consistency. Every constraint check (NOT NULL, UNIQUE, CHECK, FOREIGN KEY) is a consistency operation. The database must read the current state, verify the constraint, and then write the new state.

In a distributed system, this creates tension with the CAP theorem. Schema enforcement sits on the CP side of the CAP axis.

  caption="Schema enforcement aligns with CP in the CAP theorem; weaker enforcement aligns with AP"

  ┌─────────────────────────────────────────────────────────────────────┐
  │  STRONG enforcement (CP)   │  WEAK enforcement (AP)  │ NO enforcement │
  ├────────────────────────────┼─────────────────────────┼───────────────┤
  │  Normal operation:         │  Normal operation:      │ Normal op:    │
  │  All constraints checked   │  Constraints checked    │ No checks    │
  │  before write commit       │  before write commit    │ at all       │
  │                            │                         │               │
  │  During partition:         │  During partition:      │ During part:  │
  │  Writes BLOCKED if         │  Writes ACCEPTED        │ Writes always │

  └────────────────────────────┴─────────────────────────┴───────────────┘
`}
/>

**During a partition**: A CP database blocks writes that cannot be verified against the current state. If the database cannot check a foreign key because the referenced table is on an unreachable node, the write is rejected.

**The AP alternative**: If the database accepts writes during a partition without schema enforcement, the schema constraints are only eventually enforced. This is the approach taken by eventually consistent databases like Cassandra, where schema enforcement is a best-effort operation.

```
CAP positioning for schema enforcement:

  Strong schema enforcement (CP):
  - All constraints checked before write acknowledged
  - Writes blocked during partition if FK references unreachable partition
  - Guarantees: no orphan rows, no duplicate unique values
  - Examples: PostgreSQL, MySQL, Spanner

  Weak schema enforcement (AP):
  - Constraints checked when possible, skipped during partition
  - Writes accepted during partition even if FK references are unreachable
  - Orphan rows possible until partition heals
  - Examples: Cassandra (lightweight transactions), DynamoDB (conditional writes)

  No schema enforcement (AP):
  - No constraints at database level
  - All data quality is application responsibility
  - Writes always accepted
  - Examples: S3, Redis (without CAS), Kafka
```

## Failure Modes

### Schema Drift

Schema drift occurs when the actual database schema differs from the version-controlled schema definition. The mismatch between application and database schemas causes runtime errors.

```
Causes of schema drift:
- Hotfix applied directly to production (bypassing migration system)
- Migration partially applied (some statements succeeded, others failed)
- Two environments drifted apart (staging and production)
- Manual intervention by DBA during an incident
- Replication lag causing schema mismatch between primary and replica

Symptoms:
- Application code references columns that do not exist
- Migration tool refuses to apply (checksum mismatch)
- Queries fail with cryptic column-not-found errors
- Different behavior on different database replicas
```

Detection: Run periodic schema comparison between the database and the migration definitions.

```bash
# Detect drift: compare database schema to migration output
pg_dump --schema-only mydb > /tmp/current_schema.sql
git diff /tmp/current_schema.sql migrations/expected_schema.sql
# Any differences are schema drift -- investigate before deploying
```

### Migration Deadlock

Two services block on each other's migration. Service A needs a column from Service B's database. Service B needs a column from Service A's database. Both cannot deploy until the other deploys first.

```
Service A needs: column X in Service B's database
Service B needs: column Y in Service A's database

Deployment order:
1. Deploy A? A's code references column X -- B does not have it yet. Fails.
2. Deploy B? B's code references column Y -- A does not have it yet. Fails.
3. Deadlock. Neither service can deploy without the other.
```

Prevention: Avoid cross-service schema dependencies. If Service A needs data from Service B, it should access it through Service B's API, not through direct database access. Cross-service schema dependencies are an architecture smell.

### Out-of-Disk During ALTER TABLE

Certain ALTER TABLE operations rewrite the entire table. PostgreSQL requires disk space for the new table copy alongside the old table. If the table is 100GB, the operation requires 100GB of free disk space.

```sql
-- These operations rewrite the table:
ALTER TABLE products ALTER COLUMN price TYPE NUMERIC(12,2);
ALTER TABLE products ADD COLUMN description TEXT NOT NULL DEFAULT '';
ALTER TABLE products ALTER COLUMN name SET DATA TYPE VARCHAR(200);

-- If the table is 200GB, the operation needs 200GB free space.
-- If free space is 50GB, the operation fails mid-way.
-- The table is left in an inconsistent state.
```

Mitigation: Run ALTER TABLE rewrites during maintenance windows with monitoring on disk space. Use `pg_repack` or `gh-ost` (MySQL) for online table rewrites that do not require double the space.

### Long-Running Migrations Locking Tables

Some migrations acquire locks that block reads, writes, or both. The lock duration depends on the migration type and the table size.

```sql
-- Blocks ALL access (ACCESS EXCLUSIVE lock):
ALTER TABLE orders ADD COLUMN discount DECIMAL(5,2);  -- PostgreSQL 11+: fast if no default
ALTER TABLE orders ADD COLUMN priority INTEGER DEFAULT 0;  -- SLOW: rewrites table

-- Blocks writes but allows reads (SHARE ROW EXCLUSIVE lock):
ALTER TABLE orders ADD FOREIGN KEY (customer_id) REFERENCES customers(customer_id);  -- Fast: metadata only
ALTER TABLE orders VALIDATE CONSTRAINT fk_customer;  -- SLOW: scans both tables

-- Blocks writes (SHARE UPDATE EXCLUSIVE lock):
CREATE INDEX CONCURRENTLY idx_orders_date ON orders(order_date);  -- Fast: no table lock
```

A migration that runs for 30 minutes with an ACCESS EXCLUSIVE lock causes 30 minutes of downtime. Mitigation: Use `CONCURRENTLY` options where available. Break large migrations into batches. Run during low-traffic windows.

### NOT NULL Migration on Large Table

Adding a NOT NULL constraint to an existing column requires verifying that every row has a non-NULL value. The database scans the entire table. If the scan finds a NULL, the migration fails.

```sql
-- This scans the entire table to verify no NULLs exist:
ALTER TABLE customers ALTER COLUMN email SET NOT NULL;

-- Mitigation: add with NOT VALID, then validate
-- Phase 1: Add constraint (no validation, no lock)
ALTER TABLE customers ADD CONSTRAINT nn_email
    CHECK (email IS NOT NULL) NOT VALID;

-- Phase 2: Validate (scans table, but only SHARE lock)
ALTER TABLE customers VALIDATE CONSTRAINT nn_email;

-- If validation fails, drop the constraint, fix the NULLs, retry
```

### Dead Code Columns Never Removed

Over years of development, columns accumulate. A column added for a feature that was never launched. A column that was replaced by a new column in a different table. A column that was deprecated but never dropped.

```
Dead columns have costs:
- Storage: each column takes space in every row (1 bit in NULL bitmap, potentially more)
- Mental overhead: developers must understand what each column does
- Query complexity: SELECT * returns columns nobody uses
- Migration risk: a dead column can be dropped safely, but nobody has the confidence to do it
```

Prevention: Document every column with its purpose. Add a `deprecated_since` comment when a column is no longer used. Schedule quarterly cleanup migrations to remove columns that have been deprecated for more than 6 months.

## Anti-Patterns

### The God Table

One table stores everything. Every new feature adds more columns. The table grows to 100+ columns. No single query needs more than 10 columns, but every row stores all columns.

```
Symptoms:
- Table has 80+ columns
- Most columns are NULL for most rows
- Column names are generic (field1, field2, data1, extra)
- No clear entity boundary
- Queries SELECT * because nobody knows which columns exist
- Every query does a full table scan or a poorly selective index scan
```

**Fix**: Split into multiple tables by entity. Each table should have 3-15 columns. Each entity should be a logically cohesive unit.

### Too Many NULL Columns

A table where 50% of columns are NULLable and 80% of rows have NULLs in most of those columns.

```
Symptoms:
- 30+ NULLable columns
- Most rows have fewer than 10 of those columns populated
- NULL columns are used as flags: column_used_for_different_purposes
- NULL bitmap overhead is significant
- Developers are confused about which columns are actually used
```

**Fix**: Split the table vertically. Frequently-NULL columns belong in a separate table with a FK. The separate table only stores rows when the optional data exists.

### Storing JSON for Everything

Every table has a JSON column to store data that does not fit the schema. Over time, all the important data moves to the JSON column, and the structured columns are empty.

```
Symptoms:
- Every table has a `data` or `metadata` JSONB/TEXT column
- Queries extract values from JSON in WHERE clauses
- Indexes on JSON paths are attempted but rarely effective
- JSON validation is in application code, not the database
- Schema documentation is non-existent (the JSON structure is undocumented)
```

**Fix**: Extract frequently-queried JSON fields into structured columns. Keep the JSON column only for truly variable data that is never queried directly. Add JSON schema validation (PostgreSQL JSONB constraints, MongoDB validator).

### EAV (Entity-Attribute-Value) Antipattern

The EAV pattern stores attributes as rows instead of columns. Used when the schema designer wants infinite flexibility at the cost of query performance.

```sql
-- EAV schema: attributes as rows
CREATE TABLE product_attributes (
    product_id   INTEGER NOT NULL REFERENCES products(product_id),
    attribute_name  VARCHAR(50) NOT NULL,
    attribute_value VARCHAR(255) NOT NULL,
    PRIMARY KEY (product_id, attribute_name)
);

-- To query: "Find products with color=red and size=large"
-- Requires: JOIN to itself, or aggregation with HAVING
SELECT p.* FROM products p
WHERE p.product_id IN (
    SELECT pa1.product_id FROM product_attributes pa1
    JOIN product_attributes pa2 ON pa1.product_id = pa2.product_id
    WHERE pa1.attribute_name = 'color' AND pa1.attribute_value = 'red'
    AND pa2.attribute_name = 'size' AND pa2.attribute_value = 'large'
);
```

This query is slow, complex, and does not scale. As the attribute table grows, the JOIN becomes progressively slower.

**Fix**: Use a document column (JSONB) for variable attributes, with a GIN index for queries. Or use separate tables for each product type with concrete columns. EAV is only justified when the number of attributes is truly unbounded and queries are limited to single-attribute lookups, which is almost never the case.

## When NOT to Use Schema Design

### Prototypes and Proofs of Concept

In the early stages of a project, the data model is unknown. Investing time in schema design is premature. Use a schemaless approach initially.

```
Start with: A single JSON/JSONB column or a document database.
Iterate: Add structure as access patterns emerge.
Migrate: Once the data model stabilizes, move to a formal schema.
```

The cost of refactoring a schemaless prototype into a structured schema is acceptable. The cost of designing a schema that turns out to be wrong is higher.

### Event Sourcing

Event sourcing stores a sequence of events rather than the current state. Each event type has its own schema. The schema varies by event type, not by a fixed table structure.

```json
[
  {"type": "OrderPlaced", "order_id": "123", "items": [...], "timestamp": "..."},
  {"type": "OrderShipped", "order_id": "123", "tracking": "1Z...", "timestamp": "..."},
  {"type": "OrderDelivered", "order_id": "123", "signed_by": "Alice", "timestamp": "..."}
]
```

Each event type has a different structure. There is no single schema for the event store. The schema is the union of all event type schemas, enforced by the event schema registry, not by the database schema.

### Full-Text and Document-Centric Data

When the primary access pattern is full-text search (Elasticsearch, Meilisearch) or document storage (MongoDB, S3), a rigid relational schema is counterproductive.

```
Full-text search schema: inverted index, analyzed fields, relevance scoring.
Document storage schema: nested documents, arrays, variable structures.
```

Use the database that matches the access pattern. Trying to fit full-text search into a relational schema produces a system that is bad at both search and relations.

## Key Takeaways

- The core tension is between rigid schemas (safe, consistent, slow to change) and flexible schemas (adaptable, risky, fast to change).
- Eight key decision points each have clear tradeoffs: strict vs loose, normalized vs denormalized, UUID vs auto-increment, inheritance patterns, embedded vs referenced, NULLable vs NOT NULL, text vs enum vs reference table.
- Schema enforcement is a CP operation in distributed systems -- it conflicts with AP during partitions.
- Five failure modes plague schema design: drift, migration deadlock, out-of-disk during ALTER TABLE, long-running locks, and dead code columns.
- Four anti-patterns to avoid: the God table, too many NULL columns, JSON for everything, and the EAV antipattern.
- Schemas are not always the right answer. Prototypes, event sourcing, and full-text search are cases where schema design is not the primary concern.
