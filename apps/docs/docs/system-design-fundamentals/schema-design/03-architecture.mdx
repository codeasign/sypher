## Naive Implementation

The naive approach to schema design is the single table with all columns as VARCHAR. It is also known as the "universal table" or "one table to rule them all."

```sql
CREATE TABLE everything (
    id          VARCHAR(36) PRIMARY KEY,
    type        VARCHAR(50),
    field1      VARCHAR(255),
    field2      VARCHAR(255),
    field3      VARCHAR(255),
    -- ... 50 more VARCHAR columns ...
    field50     VARCHAR(255),
    payload     TEXT,  -- JSON blob for "flexibility"
    created_at  VARCHAR(30),
    updated_at  VARCHAR(30)
);
```

This schema breaks at scale in four ways:

**Wrong types cause overflow and corruption**: `field1` stores prices as VARCHAR. A query that sums prices must convert every value from string to number. If any value is `'abc'`, the query fails with a type error. If any value has trailing spaces, the sum is silently wrong.

**No constraints allow corrupt data**: No foreign keys, no unique constraints, no check constraints. The application is responsible for all data integrity. Every code path must validate every column. A single bug in one code path produces corrupt data that is almost impossible to detect without manual inspection.

**No relationships cause data inconsistency**: Customer information is stored in the same table as order data. If a customer changes their email, every row containing that customer's information must be updated. If one row is missed, the customer has two different emails in the system.

**Everything requires a full table scan**: No meaningful indexes can be created because the columns have generic names and generic types. The database cannot optimize queries. Every query is a sequential scan or requires application-level filtering.

### When Naive Fails

```
Metrics for naive schema at 5 million rows:
- Average query time (single row lookup by ID on TEXT PK): 45ms
- Average query time (filter by type on VARCHAR): 1,200ms
- Storage used (50 VARCHAR(255) columns, mostly empty): 2.8 GB
- Storage used after correct schema design: 380 MB
- Query time after correct schema design: 0.5ms
- Application bugs caused by missing constraints per quarter: 12-18
```

## Production-Grade Architecture

A production-grade schema architecture uses multiple layers, each designed for its access pattern.

### Layer 1: OLTP Schema (Normalized, Transactional)

The Online Transaction Processing schema is the system of record. It is highly normalized (3NF), transactional (ACID), and optimized for writes and point lookups.

```
Characteristics:
- Normalized to 3NF or BCNF
- Single-row inserts and updates
- Narrow tables with focused column sets
- Primary key lookups and short-range scans
- Foreign keys enforced at database level
- Row-based storage engine (InnoDB, PostgreSQL heap)
- Typical row size: 50-500 bytes
- Typical query: "Get order 12345 with all its items"
```

### Layer 2: ETL Pipeline

The Extract-Transform-Load pipeline moves data from the OLTP schema to the OLAP schema. It runs on a schedule (hourly, daily) or streams changes via CDC (Change Data Capture).

```
ETL transforms:
- Joins normalized tables into denormalized fact tables
- Creates surrogate keys for dimensions (type-2 slowly changing dimensions)
- Aggregates daily metrics from raw transaction data
- Cleans and validates data (handles NULLs, deduplicates)
- Removes soft-deleted records
- Recalculates derived columns
```

### Layer 3: OLAP Schema (Star, Analytical)

The Online Analytical Processing schema is optimized for complex queries across many rows. It uses the star schema pattern: a central fact table surrounded by dimension tables.

```
Characteristics:
- Denormalized fact table with numeric measures
- Dimension tables with descriptive attributes
- Columnar storage engine (ClickHouse, Redshift)
- Large batch inserts, no point updates
- Pre-computed aggregations and materialized views
- Typical row size: 100-500 bytes (fact), 500-2000 bytes (dimension)
- Typical query: "Total sales by product category per month for 2024"
```

### Layer 4: Denormalized Cache Schema

The cache layer serves high-traffic API queries at sub-millisecond latency. It is highly denormalized, sometimes combining dozens of tables into a single document.

```
Characteristics:
- Denormalized into the query shape (one cache entry = one API response)
- Key-value or document format (Redis, Memcached, Elasticsearch)
- TTL-based expiration or cache-aside pattern
- Pre-computed from OLTP or OLAP data
- Typical document size: 1-50 KB
- Typical query: "Get user profile with orders and recommendations"
```

## Integration Patterns

### Microservices and Database Per Service

In a microservices architecture, each service owns its database. No service directly accesses another service's database. Service boundaries correspond to bounded contexts in Domain-Driven Design.

```
Service: User Service
Schema: users(id, name, email, password_hash, created_at)

Service: Order Service
Schema: orders(id, user_id, total, status, created_at)
Schema: order_items(id, order_id, product_id, quantity, price)

Service: Inventory Service
Schema: inventory(product_id, quantity_available, warehouse_id)

Service: Payment Service
Schema: payments(id, order_id, amount, status, transaction_id)
```

Each service's schema is independently versioned and migrated. Breaking changes require coordination between services through API versioning, not database migrations.

### Shared Schema (Data Warehouse)

The data warehouse schema is the shared truth for analytics. It is designed once but consumed by multiple teams.

```sql
-- Shared fact table -- owned by the data platform team
CREATE TABLE fact_order (
    order_sk     BIGINT PRIMARY KEY,
    date_sk      INTEGER REFERENCES dim_date(date_sk),
    cust_sk      INTEGER REFERENCES dim_customer(cust_sk),
    prod_sk      INTEGER REFERENCES dim_product(prod_sk),
    quantity     INTEGER NOT NULL,
    revenue      DECIMAL(12,2) NOT NULL,
    discount     DECIMAL(5,2),
    etl_batch_id INTEGER NOT NULL
);

-- Every team reads from this table for order analytics
-- Sales team: total revenue by month
-- Marketing team: top products by customer segment
-- Operations team: order volume by warehouse
```

The shared schema is governed by a data platform team. Changes require a proposal, review, and migration plan.

### Schema-as-Contract

In event-driven architectures, the schema of the event is a contract between producer and consumer services. Tools like Apache Avro, Protocol Buffers, and Apache Thrift define the schema in a language-neutral format.

```protobuf
// Schema-as-contract: OrderCreated event
syntax = "proto3";

message OrderCreated {
    string order_id = 1;
    string customer_id = 2;
    repeated OrderItem items = 3;
    double total_amount = 4;
    string currency = 5;
    int64 created_at_unix = 6;

    message OrderItem {
        string product_id = 1;
        int32 quantity = 2;
        double unit_price = 3;
    }
}
```

The schema registry (Confluent Schema Registry, Azure Schema Registry) stores every version of every schema. Producers write events with a schema version ID. Consumers read events and deserialize using the registered schema.

The registry enforces compatibility rules:

- **Backward compatible**: New schema can read old data. Fields can be added with defaults.
- **Forward compatible**: Old schema can read new data. Fields can be removed.
- **Full transitive**: Every schema version is compatible with every other version.

### Event Schema (Kafka)

Kafka topics have a schema. Each event is a structured record with a key, value, headers, and timestamp.

```json
{
  "schema": {
    "type": "struct",
    "fields": [
      { "field": "order_id", "type": "string", "optional": false },
      { "field": "customer_id", "type": "string", "optional": false },
      { "field": "event_type", "type": "string", "optional": false },
      { "field": "data", "type": "string", "optional": false },
      { "field": "timestamp", "type": "int64", "optional": false }
    ]
  },
  "payload": {
    "order_id": "ord_12345",
    "customer_id": "cus_678",
    "event_type": "order_placed",
    "data": "{\"items\": [...], \"total\": 149.99}",
    "timestamp": 1705000000000
  }
}
```

Event schema design differs from database schema design in four ways:

- **Append-only**: Events are never updated. New event types are appended to the topic.
- **Envelope pattern**: Every event has metadata (type, timestamp, source) wrapped around the payload.
- **Versioned per topic**: Each topic has its own schema version lineage.
- **Tolerant reader**: Consumers should handle unknown fields gracefully.

## Multi-Region

### Schema Versioning Across Regions

In a multi-region deployment, different regions may be on different schema versions during a rolling deployment.

```
Region US-East: Schema V42 (latest)
Region EU-West: Schema V41 (deploying)
Region AP-Southeast: Schema V41 (scheduled)
```

Schema changes must be backward-compatible across the deployment window. A change that adds a column is safe -- the old schema ignores the column. A change that renames a column breaks the old schema -- queries against the old column name fail.

Rules for cross-region schema safety:

- **Add-only**: Never drop or rename columns. Add new columns with default values or allow NULLs.
- **Old code ignores new columns**: Application code in older regions must not reference columns it does not know about.
- **New code handles old data**: Application code in newer regions must handle missing columns gracefully (provide defaults).
- **Feature flags**: Gate new features that depend on new schema columns behind a feature flag. Enable the flag only after the schema change has propagated to all regions.
- **Read-only phase**: After adding a column, deploy code that reads from the old location and writes to both old and new locations. In the next deployment, switch reads to the new location and stop writing to the old location.

### Rolling Schema Migrations

A rolling migration applies a schema change to one database replica at a time to avoid downtime.

```
Step 1: Apply migration to a read replica.
Step 2: Promote the read replica to primary.
Step 3: Apply the migration to the old primary (now a replica).
Step 4: Verify the migration on all nodes.

This technique works for schema changes that are:
- Non-blocking (no ACCESS EXCLUSIVE lock required)
- Backward-compatible (old code can still run)
- Fast to apply (no full table rewrites)
```

For blocking migrations, use the expand-migrate-contract pattern across the entire cluster:

```
Week 1 (Expand): Add new column "status_v2" alongside "status".
Week 2 (Migrate): Application writes to both "status" and "status_v2".
                  Backfill "status_v2" for existing rows.
Week 3 (Contract): Switch reads from "status" to "status_v2".
                   Deprecate "status" column.
Week 4 (Cleanup): Drop "status" column. Remove dead code.
```

### Forward/Backward Compatibility

Schema compatibility ensures that different versions of the application can read the same data.

**Forward compatibility**: A newer version of the schema can read data written by an older version. Achieved by: new fields have defaults, removed fields are ignored.

**Backward compatibility**: An older version of the schema can read data written by a newer version. Achieved by: old code ignores unknown fields, required fields in old schema are still present in new schema.

```sql
-- Original schema (V1)
CREATE TABLE products (
    product_id  SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    price       DECIMAL(10,2) NOT NULL
);

-- V2 adds description column (backward compatible)
ALTER TABLE products ADD COLUMN description TEXT;
-- Old code can still INSERT without description (NULL allowed)
-- Old code can still SELECT name, price (description ignored)

-- V3 adds NOT NULL to description (NOT backward compatible!)
ALTER TABLE products ALTER COLUMN description SET NOT NULL;
-- Old INSERT without description now fails!
-- This breaks backward compatibility -- requires coordinated deploy
```

The cardinal rule: **Never add a NOT NULL column without a default to an existing table in a multi-region deployment.** The deploy window creates a gap where old code runs against the new schema, and every INSERT fails.

## Capacity Planning

### Schema Storage Estimation

Calculate how much storage a table will consume before creating it.

```sql
-- Estimate storage for orders table
-- Columns: order_id (INT, 4B), customer_id (INT, 4B),
--           order_date (DATE, 4B), total (DECIMAL(10,2), 6B),
--           status (VARCHAR(20), avg 8B + 1B length)

Row header overhead (PostgreSQL): 27 bytes per row
Column data: 4 + 4 + 4 + 6 + 9 = 27 bytes
Null bitmap: 1 byte (5 nullable columns, but only 8 bits needed)
Total per row: 27 + 27 + 1 = 55 bytes

With 10 million rows:
Data: 55 * 10,000,000 = 550 MB
Primary key index (B-tree): approximately 30% of data = 165 MB
Secondary indexes (2 indexes): approximately 50% each = 275 MB
Total: 550 + 165 + 275 = 990 MB

For estimation, add 30% for PostgreSQL MVTO overhead:
Total: 990 * 1.3 = 1.29 GB
```

### Future Column Requirements

Anticipate future columns by analyzing the application roadmap.

```
Near-term (3 months):
- Add promo_code to orders (VARCHAR(50), NULLable)
- Add phone to customers (VARCHAR(20), NULLable)

Mid-term (6 months):
- Add loyalty_tier to customers (VARCHAR(20), NOT NULL with default)
- Add warehouse_id to inventory (INT, NOT NULL)

Long-term (12 months):
- Add returns table (separate, not new column)
- Add rating table (separate, not new column)
```

Plan column additions in batches. Each migration has overhead (testing, deployment, validation). Adding 10 columns across 10 migrations is 10x the overhead of adding 10 columns in 1 migration.

### Archive Strategy

Old data grows storage and slows queries. An archive strategy moves cold data out of the primary schema.

```
Hot data (current month): In primary OLTP database. Full indexing.
Warm data (1-6 months): In same database but compressed or in a separate tablespace.
Cold data (6+ months): Archived to a data warehouse or cold storage. Accessible by API but not in the primary query path.
Frozen data (2+ years): Compressed Parquet files in object storage (S3, GCS). Queryable via Athena or Trino if needed.
```

The schema for cold data is often wider and denormalized compared to the hot schema. Joins are acceptable at cold storage because query volume is low.

### Partition Strategy by Time

Time-based partitioning keeps the hot partition small and fast.

```sql
-- PostgreSQL native partitioning
CREATE TABLE orders (
    order_id    SERIAL,
    customer_id INT NOT NULL,
    order_date  DATE NOT NULL,
    total       DECIMAL(10,2) NOT NULL,
    status      VARCHAR(20) NOT NULL
) PARTITION BY RANGE (order_date);

-- Monthly partitions
CREATE TABLE orders_2024_01
    PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
CREATE TABLE orders_2024_02
    PARTITION OF orders FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
CREATE TABLE orders_2024_03
    PARTITION OF orders FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');
-- Remaining months created by cron job each month

-- Old partitions can be detached (not dropped) for archival
ALTER TABLE orders DETACH PARTITION orders_2023_01;
-- Detached table still exists as a standalone table for archival queries
```

Each partition is an independent table under the hood. Queries that filter by `order_date` scan only the relevant partitions (partition pruning). A query for January 2024 orders scans one partition. A query for all of 2024 scans 12 partitions. A query without a date filter scans all partitions.

## Deployment

### Schema Migration Tools

Three major migration tools with different philosophies.

**Flyway** (Java, but CLI works for any language): SQL-first. Migration files are numbered SQL scripts. Simple, fast, and easy to review.

```bash
# Flyway CLI
flyway migrate -url=jdbc:postgresql://localhost/mydb \
               -user=admin \
               -locations=filesystem:./migrations

# Migration files in ./migrations
V001__create_customers.sql
V002__create_orders.sql
V003__add_discount_to_orders.sql
```

**Liquibase** (Java, XML/YAML/JSON/SQL): Defines changesets with rollback instructions. More verbose but more powerful for complex migrations.

```xml
<!-- Liquibase changeset with rollback -->
<changeSet id="001" author="alice">
    <createTable tableName="customers">
        <column name="id" type="int" autoIncrement="true">
            <constraints primaryKey="true"/>
        </column>
        <column name="name" type="varchar(100)">
            <constraints nullable="false"/>
        </column>
    </createTable>
    <rollback>
        DROP TABLE customers;
    </rollback>
</changeSet>
```

**Alembic** (Python/SQLAlchemy): Python-based migrations. Uses auto-generation to detect schema changes from SQLAlchemy models.

```bash
# Generate a migration from model changes
alembic revision --autogenerate -m "add discount to orders"

# Generated migration
def upgrade():
    op.add_column('orders', sa.Column('discount', sa.DECIMAL(5,2), nullable=True))

def downgrade():
    op.drop_column('orders', 'discount')
```

### Zero-Downtime Migrations

The expand-migrate-contract pattern enables schema changes without application downtime.

```
Phase 1: Expand
- Add new columns/tables alongside existing ones.
- Application still reads/writes old columns.
- New columns are NULLable or have defaults.

Phase 2: Migrate
- Application writes to both old and new columns.
- Background job backfills new columns for existing rows.
- Monitor: new column values match old column values.

Phase 3: Contract
- Switch reads to new columns.
- Stop writing to old columns.
- Remove old columns (migration).
- Remove dead code (application deploy).
```

```python
# Application code during Phase 2 (dual-write)
class OrderService:
    def update_status(self, order_id: int, new_status: str):
        # Write to old column
        self.db.execute(
            "UPDATE orders SET status = ? WHERE order_id = ?",
            (new_status, order_id)
        )
        # Write to new column (may not exist yet during deploy)
        try:
            self.db.execute(
                "UPDATE orders SET status_v2 = ? WHERE order_id = ?",
                (new_status, order_id)
            )
        except DatabaseError:
            pass  # Old schema -- column does not exist yet

    def get_status(self, order_id: int) -> str:
        # Read from new column, fallback to old
        row = self.db.fetch_one(
            "SELECT status_v2, status FROM orders WHERE order_id = ?",
            (order_id,)
        )
        return row["status_v2"] if row["status_v2"] else row["status"]
```

### Canary Schema Deployments

Canary deployments apply schema changes to a subset of servers before rolling out globally.

```
Canary phase:
- Apply migration to 1 of 10 database shards.
- Route 5% of traffic to the canary shard.
- Monitor: error rates, latency, data integrity.
- Duration: 30 minutes of observation.

Rollout phase:
- Apply migration to 3 of 10 shards.
- Route 30% of traffic.
- Monitor for 30 minutes.

Full rollout phase:
- Apply to all remaining shards.
- Route 100% of traffic.
- Final monitoring for 60 minutes.
```

Rollback during canary: promote a pre-migration replica to primary for the canary shard. Data written since the migration was applied is lost, but only for 5% of users. Backfill this data from logs after rollback completes.

## Key Takeaways

- The naive schema (single table, all VARCHAR) breaks at scale from wrong types, missing constraints, data inconsistency, and full table scans.
- Production architecture uses four layers: normalized OLTP for transactions, ETL for data movement, star-schema OLAP for analytics, and denormalized cache for API responses.
- Each schema layer is designed for its specific access pattern. OLTP is write-optimized. OLAP is read-optimized. Cache is query-shaped.
- Microservices use database-per-service with bounded contexts. The data warehouse is the shared schema for cross-team analytics.
- Schema-as-contract (Avro, Protobuf, Thrift) formalizes event schemas with compatibility rules enforced by a schema registry.
- Multi-region deployments require backward-compatible schema changes. Add columns; never drop or rename them without a multi-phase process.
- Rolling migrations apply schema changes one replica at a time. The expand-migrate-contract pattern enables zero-downtime schema evolution.
- Storage estimation accounts for row data, indexes, MVCC overhead, and a 30% buffer.
- Partition by time to keep hot partitions small. Archive cold data to separate storage.
- Migration tools (Flyway, Liquibase, Alembic) version-control schema changes with automatic drift detection.
- Canary schema deployments apply changes to a subset of shards first, providing a rollback window.

These architecture patterns set the stage for the tradeoffs section, where we examine the hard choices between rigid and flexible schemas, normalization levels, and primary key strategies.
