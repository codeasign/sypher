## Internal Architecture

A production write-through cache is far more than a simple key-value store that also writes to a database. Its internal architecture contains several layered components that coordinate to ensure correctness, durability, and performance.

### Component Responsibilities

**Network / Listener Layer**: Accepts incoming connections from application clients. Handles protocol parsing (Redis protocol, HTTP, gRPC, or custom). Manages connection lifecycle and backpressure. When the write buffer fills up, this layer applies backpressure to slow down incoming requests.

**Write Buffer**: An in-memory queue that holds writes before they enter the transaction pipeline. The buffer absorbs bursts of writes and smooths out the flow to the database. Configurable size limits prevent unbounded memory growth. When full, the buffer either blocks callers or drops writes with an error.

**Transaction Manager**: The brain of the write operation. It tracks each write as a transaction with begin, commit, and rollback states. It ensures that writes are applied in order and that partial failures result in proper rollback. It also manages concurrent write isolation — ensuring two writes to the same key don't interleave in corrupting ways.

**Serialization Layer**: Converts application-level data structures into the wire format used for database writes. Also handles the reverse direction for reads. Supports multiple serialization formats — JSON for human readability, Protocol Buffers for efficiency, MsgPack for compactness. Schema validation happens here, catching malformed writes before they reach the database.

**DB Connection Pool**: Maintains a pool of persistent connections to the database. Creating a new database connection is expensive (5-50ms handshake overhead), so the pool keeps connections warm and reuses them. The pool handles connection health checks, automatic reconnection, and load balancing across database nodes.

**Consistency Checker**: A background process that periodically compares cache contents against database contents. It detects entries where the cache and database have diverged due to partial failures, bugs, or manual interventions. When drift is detected, it can reconcile by either refreshing from the database or flagging for investigation.

**Retry Queue**: Failed database writes are moved to a retry queue with exponential backoff. The queue limits retries (typically 3-5 attempts) before moving the write to a dead-letter queue for manual inspection. This prevents transient failures (network blips, DB connection drops) from causing data loss.

### Write-Ahead Log (WAL)

The WAL is an append-only log on disk that records every write before it is applied. Before the cache updates its in-memory state or sends the write to the database, it writes a record to the WAL. This ensures durability: if the cache crashes mid-write, it can recover by replaying the WAL.

WAL entries include a sequence number (for ordering and deduplication), a timestamp, the key being written, the serialized value, and a checksum for integrity verification. The WAL is typically stored on fast local storage (NVMe SSD) and is rotated periodically to prevent unbounded growth.

### Transaction Buffer

The transaction buffer holds in-flight writes that have been accepted but not yet completed. Each transaction has a state: `PREPARED` (WAL written, ready to apply), `IN_FLIGHT` (currently being written to cache and database), or `COMMITTED` (completed). The buffer allows the cache to track progress of concurrent writes and provides the data needed for rollback in case of failure.

### Commit Queue

The commit queue tracks writes that have completed the cache update but are pending database confirmation. Writes in this queue are visible to cache reads (the cache has been updated) but are not yet durable in the database. If a failure occurs at this stage, the cache must either complete the database write or roll back the cache update.

### Conflict Resolution Table

This table tracks concurrent writes to the same key. It uses optimistic locking with version numbers. When two writes target the same key simultaneously, the conflict resolution table detects the conflict, determines the correct ordering, and ensures only one write proceeds while the other is queued or rejected.

**Stage 2 — Parse**: The raw bytes are parsed into structured fields: operation type (set, update, delete), key, value, and optional metadata (TTL, tags, version). Invalid formats are rejected immediately.

**Stage 3 — Validate**: The parsed fields are validated against schema constraints. Key format, value size, and data types are checked. Validation rejects malformed data before it enters any critical path.

**Stage 4 — Lock**: The conflict resolution table is consulted. If another write is in progress on the same key, the current write either waits (if lock is held briefly) or is queued (if contention is high). Optimistic locking with version numbers reduces unnecessary waiting.

**Stage 8 — Wait for DB ACK**: The cache blocks on the database response. This is the longest part of the write lifecycle, typically 5-50ms depending on database load and network round-trip time. During this wait, the cache can process other writes to different keys concurrently.

**Stage 9 — Verify Write**: When the database responds, the transaction manager verifies the response. It checks that the database confirmed the write with the correct number of affected rows or a success indicator. A verification failure triggers rollback.

**Stage 10 — Commit**: The transaction is marked as committed. The WAL entry is updated from `PENDING` to `COMMITTED`. The write is now fully durable in both cache and database.

**Stage 10a — Rollback**: If verification failed, the cache update is reverted. The old value is restored in the in-memory cache. The WAL entry is marked as `ROLLED_BACK` with the error code for auditing. The application receives an error.

**Stage 11 — Unlock**: The lock on the key is released. Any queued writes for this key are unblocked and can proceed. The conflict resolution table entry is cleared.

**Stage 12 — Acknowledge Client**: The success (or error) response is sent back to the application over the network. The write lifecycle is complete.

### Scenario 4: Concurrent Writes

Two application instances write to the same key simultaneously. Without proper conflict resolution, one write could silently overwrite the other.

Write-through handles this through the conflict resolution table with version numbers. Write A arrives first, acquires the lock, writes v7. Write B arrives, finds the lock held, detects version v6 is stale, and is rejected with a version conflict. The application must retry with the current version.

### Scenario 5: Read-After-Write Consistency

Read-after-write consistency is the primary guarantee of write-through cache. Once a write is acknowledged, any subsequent read of that key must return the written value. This is guaranteed because the cache is updated before the write is acknowledged, and all reads hit the cache first.

However, there is a subtle edge case: if the database write succeeds but the acknowledgment to the cache is lost (network issue), the cache would roll back while the database has the new value. The next read would hit the cache, get the old value, and the application would not see the write it thought failed. This is a read-your-writes violation at the application level, though the cache itself is internally consistent.

## Performance

Write-through caching has a distinct performance profile: slower writes, faster reads.

### Write Latency

The write latency of a write-through cache is the sum of cache update time plus database write time.

- **Cache-only write**: 1-5ms (in-memory update, no I/O)
- **Database write**: 5-50ms (network round-trip + disk I/O)
- **Total write-through latency**: 6-55ms

Compare this to a write-back cache, where the application only waits for the cache update: 1-5ms total.

### Write Throughput

Throughput is limited by the database connection pool and the database's write capacity. With a pool of 50 connections and an average write latency of 20ms, the maximum throughput is:

50 connections / 0.02 seconds = 2,500 writes per second

Adding more database connections or using connection pooling with multiplexing can increase this, but the database itself becomes the bottleneck.

### Read Performance

Read performance is excellent — typically 1-5ms for in-memory cache hits. The cache hit rate approaches 100% for recently written keys because the cache was explicitly populated during the write.

### When Write-Through Makes Performance Sense

Write-through is beneficial when:

- The workload is read-heavy (high read-to-write ratio). Each write pays a penalty, but many subsequent reads benefit from the fresh cache data.
- Read-your-writes consistency is a hard requirement. No amount of performance optimization justifies returning stale data in these scenarios.
- The database write latency is moderate (under 50ms). If the database is very slow (100ms+ per write), the write penalty becomes prohibitive.

## Configuration

A production write-through cache requires careful configuration of several parameters. Each parameter balances a concern: performance, durability, consistency, or resource usage.

### Write Buffer Size

Controls how many pending writes can be queued in memory before being flushed to the database.

- **Small (1K entries)**: Low memory usage, high risk of backpressure during write bursts.
- **Medium (10K entries)**: Good balance for most workloads. Absorbs typical burst patterns.
- **Large (100K entries)**: High memory usage, but can handle extreme write spikes. Risk of data loss on cache crash if the buffer is not backed by WAL.

### Flush Interval

Controls how often buffered writes are flushed to the database in batch mode.

- **Aggressive (10ms)**: Lower latency, higher DB write frequency, more IOPS.
- **Moderate (50ms)**: Good balance for most workloads.
- **Lazy (500ms)**: Higher latency for DB durability, but better batching efficiency.

### Connection Pool Size

The number of persistent database connections maintained by the cache.

- **Formula**: Pool size = (expected write QPS * average write latency) + buffer for spikes
- **Example**: For 2,000 writes/sec with 20ms latency: pool size >= 40

### Retry Policy

Controls how the cache handles failed database writes.

- **Max retries**: 3 for transient failures, 1 for structural failures (schema violations).
- **Backoff**: Exponential backoff with jitter: 100ms, 500ms, 2,500ms.
- **Dead-letter threshold**: After max retries, write is moved to dead-letter queue for manual inspection.

### Timeout

Maximum time to wait for a database write to complete.

- **Default**: 10 seconds
- **Short (1-2s)**: Use when fast failure detection is critical. Risk of false positives on slow DB.
- **Long (30-60s)**: Use when database occasionally experiences slow queries. Risk of blocked cache threads.

### Commit Strategy

Controls when a write is considered committed.

- **Write-through**: Commit after cache update + DB acknowledgment. Strongest consistency, highest latency.
- **Write-around**: Commit after DB acknowledgment only; cache updated lazily. Weaker consistency, lower write latency.
- **Async write-through**: Commit after cache update; DB write is asynchronous with redo log for recovery. Between write-through and write-back in both consistency and latency.

### Serialization Format

The wire format used to encode data for database writes.

- **JSON**: Human-readable, flexible schema, moderate size overhead (~30% larger than binary). Good for development.
- **Protocol Buffers**: Binary, strongly typed, compact, fast to serialize. Good for production.
- **MsgPack**: Binary, JSON-compatible types, slightly less compact than Protobuf. Good when interoperability with JSON tooling is valuable.`}
