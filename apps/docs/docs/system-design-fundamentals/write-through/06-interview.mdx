## Question Types

Interview questions about write-through caching fall into several categories. Understanding the question type tells you what the interviewer is evaluating.

### Type 1: Definition and Comparison

**Sample questions**:
- "What is write-through caching and how does it differ from write-back?"
- "Explain the write penalty. When is it acceptable?"
- "Compare write-through, write-back, and write-around."

**What is being evaluated**: Foundational knowledge. The interviewer wants to confirm you understand the basic concepts before diving deeper. This is usually a warm-up question.

**How to answer**: Define the term precisely. Give the analogy. Immediately contrast with other caching strategies. Show that you understand both the benefit (cache consistency) and the cost (write latency).

### Type 2: Design a System with Write-Through

**Sample questions**:
- "Design a write-through cache for an e-commerce inventory system."
- "How would you implement write-through caching for a social media comments service?"
- "Design the write path for a ride-sharing app that uses write-through caching."

**What is being evaluated**: System design ability. The interviewer wants to see how you apply write-through in a concrete system. They evaluate your architecture decisions, trade-off analysis, and failure handling.

**How to answer**: Start with requirements. Identify which data needs write-through (consistency-critical) and which does not. Draw the architecture. Walk through the write lifecycle. Discuss failure modes and how you handle them.

### Type 3: Failure Scenarios

**Sample questions**:
- "What happens when the database goes down in a write-through system?"
- "How do you handle a failed write-through? The cache was updated but the DB write failed."
- "What happens to concurrent writes to the same key in a write-through cache?"

**What is being evaluated**: Debugging and resilience thinking. The interviewer wants to see if you understand failure modes beyond the happy path.

**How to answer**: Acknowledge the failure scenario. Explain the state of the system (cache updated, DB not). Describe the recovery mechanism. Discuss how to prevent the failure from recurring.

### Type 4: Performance Optimization

**Sample questions**:
- "What is the performance bottleneck in a write-through cache and how do you address it?"
- "How would you optimize write-through for a high-throughput system?"
- "Your write-through cache is serving stale data. How do you debug it?"

**What is being evaluated**: Deep systems understanding. The interviewer is probing your knowledge of real-world performance characteristics and your ability to identify bottlenecks.

**How to answer**: Identify the bottleneck (likely the database write path). Propose specific solutions (batch commits, connection pooling, write buffer tuning). Discuss the trade-offs of each optimization.

### Type 5: Advanced Topics

**Sample questions**:
- "How would you implement write-through across multiple data centers?"
- "Design a write-through cache that guarantees atomicity across cache and database."
- "How do you apply the write-through pattern for a globally distributed system?"

**What is being evaluated**: High-level architectural thinking. These questions are typically asked at senior and staff levels. The interviewer wants to see your ability to handle complexity and distribute consistent writes across a global system.

**How to answer**: Acknowledge the difficulty. Discuss trade-offs between consistency and latency. Propose a specific architecture (active-passive, global write coordinator, CRDT-based conflict resolution). Explain the limitations of your approach.

## Interviewer Evaluation Criteria

### What Interviewers Look For

| Dimension | Good Answer | Great Answer |
|---|---|---|
| Definition | "Write-through updates both cache and DB in the same operation." | "Write-through ensures the cache is always fresh but adds write latency equal to the DB round-trip. It guarantees read-your-writes consistency." |
| Trade-offs | "It's slower for writes but faster for reads." | "Write-through is best when the read-to-write ratio is high, read-your-writes consistency is required, and the DB write latency is under 50ms. For write-heavy workloads, the write penalty is prohibitive." |
| Failure handling | "If the DB fails, the write fails." | "If the DB fails, the cache must roll back the speculative update and return an error. The retry queue should have bounded retries and a dead-letter queue. A circuit breaker prevents cascading failures." |
| Optimization | "Use connection pooling." | "Use batched writes, adaptive connection pool sizing, WAL-backed durability, and asynchronous replication to the DB. Optimize serialization format. Configure write buffer size based on burst patterns." |
| Scale | "Add more cache nodes." | "At scale, a write coordinator serializes writes to the same key. Consistent hashing routes writes to the correct cache shard. A global consistency checker detects drift. Multi-region requires active-passive or CRDT-based conflict resolution." |

### Level Expectations

**L3 / Entry-level**: Can define write-through and contrast it with write-back. Understands the basic trade-off. Can describe a simple write-through implementation.

**L4 / Mid-level (Google L4, Amazon SDE II)**: Can design a write-through cache for a specific use case. Understands connection pooling, write buffers, and retry logic. Can analyze performance bottlenecks.

**L5 / Senior (Google L5, Amazon SDE III)**: Can design a production-grade write-through system with fault tolerance. Handles concurrent writes, failure recovery, and monitoring. Discusses trade-offs between consistency, latency, and throughput.

**L6 / Staff (Google L6, Amazon Principal)**: Can design globally distributed write-through with strong consistency guarantees. Understands CAP theorem implications deeply. Designs failure modes and recovery strategies. Recommends when NOT to use write-through.

**L7 / Senior Staff**: Defines caching strategy across the organization. Designs system-wide write-through patterns. Mentors teams on trade-off analysis. Contributes to open-source caching or database projects.

## FAANG Interview Framework

### The Write-Through Response Structure

When answering a write-through design question, follow this framework:

1. **Clarify requirements**: "What type of data are we caching? What is the read-to-write ratio? What consistency guarantees do we need?"
2. **Define scope**: "Which operations use write-through? Do we use write-through for all data or only for consistency-critical data?"
3. **Propose architecture**: Draw the high-level architecture. Show the application, cache layer, and database.
4. **Walk through the write path**: Describe each step from application write to acknowledgment.
5. **Walk through the read path**: Show how reads benefit from the fresh cache.
6. **Discuss failure modes**: What happens when the DB is down? When the cache crashes? When there is a network partition?
7. **Analyze trade-offs**: Explicitly state the costs (write latency) and benefits (cache consistency).
8. **Propose optimizations**: Connection pooling, batching, write buffer tuning, serialization optimization.
9. **Scale considerations**: Multi-region, horizontal sharding, global consistency.

## Phrase Bank

Use these phrases in your responses to demonstrate depth and precision.

### For Definitions

- "Write-through cache synchronously updates both the cache and the backing store in a single write operation."
- "The write penalty is the additional latency incurred by writing through to the database — typically 5-50ms added to the write path."
- "Read-your-writes consistency means that once a write is acknowledged, any subsequent read returns that written value."
- "The staleness window is effectively eliminated because the cache is updated before the write is acknowledged to the client."

### For Trade-offs

- "Write-through optimizes for read performance at the cost of write latency."
- "The read-to-write ratio determines whether the write penalty is worth paying."
- "There is an irreducible tension between cache consistency and write throughput."
- "You should consider write-through when consistency is a hard requirement and the database can handle the write load."
- "Write-through is not a free lunch — you are trading write speed for read consistency."

### For Failure Handling

- "If the database write fails, the cache must roll back to its previous state to avoid inconsistency."
- "The circuit breaker pattern prevents cascading failures by failing fast when the database is unhealthy."
- "A write-ahead log provides durability for in-flight writes during cache node failures."
- "The retry queue with exponential backoff and a dead-letter channel handles transient database failures."
- "Connection pool exhaustion is the most common failure mode — it amplifies a small slowdown into a full outage."

### For Optimization

- "Batching multiple cache writes into a single database transaction reduces connection overhead."
- "Adaptive connection pool sizing adjusts to traffic patterns and provides headroom for bursts."
- "The write buffer absorbs traffic spikes and smooths out write flow to the database."
- "Serialization format choice directly impacts write latency — Protocol Buffers can be 2-3x faster than JSON."

### For Architecture

- "A write coordinator serializes writes to the same key across a distributed cache cluster."
- "Consistent hashing ensures that all writes for a given key route to the same cache node."
- "In multi-region setups, active-passive write-through is simpler and provides stronger consistency than active-active."
- "Cross-region replication lag breaks the read-your-writes guarantee if reads route to a different region than writes."

## Red Flags

Avoid these mistakes in your interview responses. Each signals a gap in understanding.

**Red flag 1**: "Write-through is always better than write-back."

Why it is a problem: Write-through is not universally better. For write-heavy workloads, the write penalty is prohibitive. Write-back is the right choice when write throughput is the primary concern and some data loss risk is acceptable.

**Red flag 2**: "The cache and database are always consistent."

Why it is a problem: They are consistent only when every write succeeds. Failures — DB crashes, network partitions, cache node failures — can cause inconsistency. A deep answer describes the failure modes and recovery mechanisms.

**Red flag 3**: "Write-through guarantees atomicity."

Why it is a problem: A simple write-through implementation does not provide atomicity across cache and database — the cache could be updated but the database write could fail. True atomicity requires a two-phase commit or distributed transaction, which adds complexity.

**Red flag 4**: "I would use write-through for all data."

Why it is a problem: Not all data needs strong consistency. Using write-through for non-critical data wastes write throughput. A good answer identifies which data needs write-through and uses other strategies for the rest.

**Red flag 5**: "It is just a cache, so it is fast."

Why it is a problem: Write-through is slower for writes than write-back or write-around. The cache helps reads, not writes. Understanding this distinction is fundamental.

**Red flag 6**: "I would ignore write failures and let the cache serve stale data."

Why it is a problem: This violates the write-through contract. If you ignore failures, the cache silently diverges from the database. The guarantee of fresh data is broken.

**Red flag 7**: "Adding more cache nodes solves all problems."

Why it is a problem: Write-through bottlenecks are typically at the database layer, not the cache layer. Adding cache nodes does not increase database write capacity. The database is the limiting factor.

**Red flag 8**: "The cache handles concurrency automatically."

Why it is a problem: Concurrent writes to the same key require explicit handling — locking, versioning, or a write coordinator. Caches do not automatically serialize concurrent writes without configuration.

## Practice Drills

### Drill 1: The One-Sentence Definition

Practice: Explain write-through caching in one sentence.

**Sample answer**: "Write-through caching is a strategy where every write updates both the cache and the database synchronously, ensuring the cache always contains the latest data at the cost of additional write latency."

### Drill 2: The Three-Minute Design

Practice: Design a write-through cache for an inventory system in three minutes. Cover requirements, architecture, write path, read path, and failure handling.

**Sample answer structure**:
- Requirements: Read-heavy (100:1), strong consistency needed for inventory counts, under 50ms write latency acceptable.
- Architecture: App → Redis cluster → MySQL primary + read replicas.
- Write path: App sets key in Redis → Redis adapter writes to MySQL → If MySQL fails, Redis rolls back → App gets error or success.
- Read path: App reads from Redis → Guaranteed fresh → No DB hit.
- Failures: MySQL down → Redis rejects writes, returns clear error. Redis node down → Another node handles reads, writes go to MySQL and repopulate cache.

### Drill 3: Trade-off Deep Dive

Practice: Given a system with 1,000 writes per second and 10,000 reads per second, should you use write-through? Why or why not?

**Answer**: Yes, with a 10:1 read-to-write ratio, the write penalty (20ms per write = 20 seconds of write latency per second) is offset by the read benefit (saving 20ms per read = 200 seconds saved per second). The net latency benefit is 180 seconds per second. However, this assumes the database can handle 1,000 writes per second. If the database becomes the bottleneck, the system degrades.

### Drill 4: Debug the Failure

Practice: You notice that your write-through cache occasionally returns stale data. The database has the latest value, but the cache does not. What do you investigate?

**Answer**: Check the following in order:
1. Did the database write succeed when the cache wrote through? (Check error logs for silent failures.)
2. Is the cache rolling back correctly on DB failures? (Check for bugs in rollback logic.)
3. Is there a race condition where another write to the same key occurred? (Check version numbers.)
4. Is the cache node serving stale data from a different write path? (Check if there is a secondary path that bypasses the cache.)
5. Are cache TTLs evicting entries prematurely? (Check TTL configuration.)

### Drill 5: Whiteboard the Write Path

Practice: On a whiteboard, draw the complete write path of a write-through cache, including failure scenarios. You should be able to draw this without hesitation.

The diagram should include:
- Application (initiates write)
- Network layer (receives write)
- Write buffer (queues pending writes)
- WAL (durability checkpoint)
- Cache storage engine (updates in-memory state)
- DB connection pool (manages database connections)
- Database (durable storage)
- Retry queue (handles failures)
- Consistency checker (background verification)

Include arrows for the success path (solid) and failure paths (dashed).
