The simplest write-through cache is a single server running an in-process cache that writes through to a single database. This is where most systems start.

### Breaking Points

The naive implementation breaks at three distinct scale thresholds.

#### 1K Requests per Second — Write Contention

At 1K writes per second on a single server, the application and cache compete for CPU time. Every write requires serialization, WAL I/O, and network I/O to the database. The in-process cache adds memory pressure. The server's single thread pool becomes a bottleneck as write operations block waiting for database acknowledgments.

**Symptoms**: Response times degrade from 20ms to 200ms. Database connection pool under the cache exhausts. Application threads pile up waiting for cache writes.

#### 100K Requests per Second — Connection Pool Exhaustion

At 100K writes per second, the database connection pool is the primary bottleneck. With 50 connections and 20ms per write, the maximum throughput is 2,500 writes per second. Beyond that, connections are queued. The queue grows without bound, consuming memory and eventually causing the cache to fail.

**Symptoms**: Connection timeouts dominate error logs. Error rate spikes. The cache enters a death spiral — failing connections trigger retries, which consume more connections.

#### 10 Million Requests per Day — Cascading Failures

At 10M requests per day (roughly 115 writes per second sustained, with peaks much higher), the naive architecture fails catastrophically. The single server cannot scale horizontally because each server has its own in-process cache with no coordination. If the server goes down, the cache disappears with it, and the database is pounded by every client reading from scratch.

**Symptoms**: Server restart triggers a cache-warming storm. The database is overwhelmed by read requests. The cache never fully warms because writes keep failing. The system enters a cascading failure mode.

### Key Components

**Write Coordinator**: A dedicated service that serializes writes to the same key across all cache nodes. It maintains the conflict resolution table and ensures that concurrent writes to the same key are ordered correctly. The write coordinator can be a single node (simpler but a single point of failure) or a distributed consensus group (more complex but fault-tolerant).

**Transaction Manager**: Within the write coordinator, the transaction manager tracks the state of each write through its lifecycle. It ensures atomicity — either the write completes on all nodes or it is rolled back on all nodes.

**Consistency Checker**: A background service that periodically compares cache contents against the database. It detects entries where the cache has diverged from the database due to bugs, partial failures, or manual interventions. When divergence is detected, it can trigger a refresh or alert operations.

**Database Cluster**: A primary database handles all writes. Read replicas handle read traffic, though reads that require strong consistency should hit the primary (or the cache, which is always fresh). Synchronous replication to a standby provides failover capability.

When integrating with SQL, the write-through cache typically:

- Opens a database transaction for each write
- Executes the write against the primary table
- Updates any secondary indexes or materialized views
- Commits the transaction
- Updates the cache with the final state
- Returns success to the application

The database transaction ensures that all related writes are atomic at the database level, even if the cache updates are not transactionally coupled to the database commit.

### Cross-Region Read-Your-Writes Guarantee

In a multi-region setup, read-your-writes consistency becomes harder to guarantee. A user writes to Region A, then reads from Region B. If Region B's cache has not yet been updated by cross-region replication, the user sees stale data.

Solutions include:

- **Read affinity**: Route reads to the region where the write occurred for a configurable window (for example, 5 minutes).
- **Global cache with local reads**: Write to both the local cache and a global cache. Reads check the global cache after local cache miss.
- **Quorum reads**: Require agreement from multiple regions before returning a read result.

## Capacity Planning

For a system targeting 10 million daily active users (DAU), here is the capacity planning approach for write-through cache.

### Write QPS Estimation

Assume each user performs 5 write operations per day on average:

- Daily writes: 10M users x 5 writes = 50M writes per day
- Peak writes (assuming 2x the hourly average during business hours): 50M / 86,400 seconds x 2 = ~1,157 writes per second
- Peak burst (assuming 5x sustained): ~5,800 writes per second

### Database Write Capacity

With write-through, every write must hit the database. At 5,800 writes per second peak, the database must handle this throughput with acceptable latency (under 50ms).

- **Connection pool**: 5,800 writes/sec x 0.02s latency = 116 connections (minimum)
- **With headroom**: 200 connections
- **Storage IOPS**: Assuming 8KB per write: 5,800 x 8KB = 46MB/s sustained write throughput

### Cache Memory

The cache must hold all actively written keys plus frequently read keys.

- **Active write keys**: Assume each user has 10 actively written keys monitored over a 1-hour window
- **Total active keys**: 50M daily writes / (24 hours x 3,600 seconds) x 3,600 seconds x 10 keys = ~20.8 million keys
- **Per key size**: 2KB average
- **Total cache memory**: 20.8M x 2KB = ~42GB
- **With replication factor 2**: 84GB cluster-wide
- **With headroom**: 120GB cluster-wide

### Network Bandwidth

- **App to cache**: 5,800 writes/sec x 2KB = 11.6MB/s (plus read traffic)
- **Cache to DB**: Same 11.6MB/s (plus read traffic from cache miss fill)
- **Total internal traffic**: ~30MB/s per direction during peak
- **Network capacity**: 1Gbps (125MB/s) is sufficient. For larger clusters, 10Gbps is recommended.

## Deployment

### Zero-Downtime Deployment

Write-through cache requires careful deployment to avoid data loss or consistency violations.

**Blue-green deployment**: Two identical cache clusters are maintained. The blue cluster serves traffic while the green cluster is deployed with new configuration. Traffic switches to green after validation. Both clusters must receive the same writes during the transition.

**Rolling deployment**: Nodes are replaced one at a time. Before removing a node, its data is drained to remaining nodes or flushed to the database. New nodes warm up by reading from the database.

**Canary deployment**: A small percentage of traffic is routed to a new cache cluster. Write-through behavior is monitored for errors and latency before full rollout.

### Monitoring

The following metrics must be monitored for a production write-through cache:

| Metric | Warning Threshold | Critical Threshold |
|---|---|---|
| Write latency (p50) | over 30ms | over 100ms |
| Write latency (p99) | over 100ms | over 500ms |
| Consistency lag | over 1 second | over 10 seconds |
| Error rate (writes) | over 0.1% | over 1% |
| DB connection pool utilization | over 70% | over 90% |
| Write buffer fill level | over 50% | over 80% |
| Retry queue size | over 100 | over 1,000 |

### Alerting

Set alerts for the following conditions:

- **Write failures spike**: More than 1% of writes fail in a 5-minute window
- **Consistency check failures**: Background consistency checks detect divergence between cache and DB
- **Connection pool exhaustion**: All connections are in use and new writes are queued
- **WAL recovery**: A cache node restarted and replayed WAL entries (may indicate crashes)
- **Cross-region replication lag**: For multi-region setups, replication delay exceeds 5 seconds

### Rollback

A rollback strategy is essential. Steps for rolling back a write-through cache deployment:

1. **Switch traffic**: Route app traffic away from the problematic cache cluster
2. **Verify DB integrity**: Ensure the database has all committed writes (it should, since write-through guarantees durability)
3. **Restore previous cache**: Deploy the previous cache version
4. **Warm the cache**: The new cache will be cold — reads during warmup may be slow
5. **Resume traffic**: Gradually route traffic back to the restored cache
6. **Monitor**: Watch for elevated error rates and latency during the warmup period.
