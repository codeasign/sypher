# Autoscaling -- Challenges

## Tier 1: Foundation

### Challenge 1: Calculate Desired Capacity

You are configuring an autoscaling group for a web application. The current setup has 8 instances running. The average CPU utilization across all instances is 72%. Your target tracking policy is configured for 50% CPU.

**Question**: What is the desired capacity that the autoscaler will calculate?

**Model Answer**:

Using the target tracking formula:

```
desired_capacity = ceil(current_capacity * (current_metric / target_metric))
desired_capacity = ceil(8 * (72 / 50))
desired_capacity = ceil(8 * 1.44)
desired_capacity = ceil(11.52)
desired_capacity = 12
```

The autoscaler will set desired capacity to 12 instances. Since the current capacity is 8, the autoscaler needs to launch 4 new instances.

**Follow-up**: If the minimum size is 5 and the maximum size is 15, would this action succeed?

**Answer**: Yes. 12 is within the [5, 15] bounds. The autoscaler will launch 4 instances.

**Further follow-up**: If 4 new instances are launched and they have a warm-up time of 180 seconds, what is the expected timeline for the CPU to return to the target?

**Answer**: The launch and warm-up take approximately 180 seconds (instance boot) + 30 seconds (health check) + 60 seconds (slow-start ramp-up) = 270 seconds (4.5 minutes). After warm-up, the 12 instances should bring the average CPU from 72% down to approximately 72% * (8/12) = 48%, which is close to the 50% target.

### Challenge 2: Identify the Scaling Policy Type

You observe the following behavior in an autoscaling group with 6 instances:

- CPU at 45%: No scaling action
- CPU at 55%: No scaling action
- CPU at 68%: Add 2 instances (group becomes 8)
- CPU at 75%: Add 4 instances (group becomes 12)
- CPU at 82%: Add 8 instances (group becomes 20)
- CPU at 35%: Remove 1 instance
- CPU at 25%: Remove 3 instances
- CPU at 15%: Remove 5 instances

**Question**: What type of scaling policy is being used? How is it configured?

**Model Answer**:

This is a **step scaling** policy with the following tiers:

**Scale-out thresholds**:
- 60-69% CPU: Add 2 instances
- 70-79% CPU: Add 4 instances
- 80%+ CPU: Add 8 instances

**Scale-in thresholds**:
- 30-39% CPU: Remove 1 instance
- 20-29% CPU: Remove 3 instances
- Under 20% CPU: Remove 5 instances

Note that there is a dead zone between 40% and 59% CPU where no action is taken. This prevents thrashing around the target. The dead zone is 20 percentage points wide.

**Follow-up**: What would happen if the CPU was at 68% for 2 minutes but then dropped to 55% before the alarm triggered?

**Answer**: If the step scaling policy has a "breach duration" parameter (number of evaluation periods the metric must remain in the tier before acting), the action would not trigger. The CPU dropped below the tier before the duration elapsed. This is the purpose of breach duration -- to filter out transient metric spikes.

### Challenge 3: Cooldown Timing

An autoscaling group has the following configuration:
- Metric collection interval: 60 seconds
- Evaluation period: 2 consecutive periods
- Cooldown period: 300 seconds
- Warm-up time: 180 seconds

At time T=0, traffic surges and CPU hits 75%. At what time T will the autoscaler:
- (a) Detect the breach and trigger a scale-out?
- (b) Have the new instances ready to serve traffic?
- (c) Be ready to evaluate another scaling action?

**Model Answer**:

**(a) Detect breach and trigger**: T = 120 seconds (2 minutes)

The metric at T=0 is collected at the next 60-second interval. The first data point showing 75% CPU is available at approximately T=60 seconds. The alarm must see 2 consecutive evaluation periods above the threshold, so the second data point arrives at T=120 seconds. The alarm transitions to ALARM state at T=120 seconds.

**(b) New instances ready**: T = 420 seconds (7 minutes)

The warm-up time is 180 seconds starting from the launch API call at T=120 seconds. T = 120 + 180 = 300 seconds for boot. Then add 30 seconds for health checks and load balancer registration. Then 60 seconds for slow-start ramp-up. So approximately T = 120 + 180 + 30 + 60 = 390 seconds for full capacity. But the autoscaler's metrics will begin to show the effect at around T=420 seconds (one metric interval after the instances are registered).

**(c) Ready for next evaluation**: T = 420 seconds (7 minutes)

The cooldown period is 300 seconds. It starts at T=120 seconds (when the scaling action was initiated). Cooldown ends at T = 120 + 300 = 420 seconds. Note that the cooldown ends at roughly the same time the new instances begin affecting metrics. This is intentional but risky -- if the cooldown were shorter, the autoscaler might see the still-elevated CPU and trigger another scale-out before the new instances take effect.

## Tier 2: Design

### Challenge 1: Design Autoscaling for a Video Transcoding Pipeline

**Prompt**: Design the autoscaling system for a video transcoding pipeline. Videos are uploaded by users and placed in an SQS queue. Each video is transcoded into 3 formats (1080p, 720p, 480p). Transcoding is CPU-intensive and takes 5-15 minutes per video depending on length and resolution. The transcoding worker is a Docker container running on EC2. The system receives 500-5,000 uploads per hour.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| Metric choice | Uses SQS queue depth as primary metric, with CPU as secondary | Uses CPU as primary metric | Does not specify metric or uses unrelated metric |
| Scaling policy | Target tracking on queue depth with custom metric, step scaling as fallback | Single target tracking policy | Single step scaling or no policy specified |
| Instance lifecycle | Describes warm-up, health check, lifecycle hooks, and graceful shutdown | Describes warm-up and health check | Skips lifecycle details |
| Cost optimization | Mixed spot/on-demand instances, batch processing with right-sizing | Spot instances only or on-demand only | No cost consideration |
| Failure handling | Describes queue backup handling, worker failure, and re-queue strategy | Describes worker failure handling | No failure handling |

**Model Approach**:

**Metric**: The primary metric should be SQS ApproximateNumberOfMessagesVisible, not CPU. CPU on a transcoding worker is always near 100% while it is processing, so CPU-based scaling would keep instances running unnecessarily. Scale-out when queue depth exceeds N * batch_size, where N is a configurable threshold (e.g., 10 batches deep). Scale-in when queue depth is near zero and instances are idle.

**Policy**: Use a custom metric that measures the ratio of "queued video minutes" to "processing capacity in minutes." Target this ratio at 2.0 (meaning you want at most 2x the queue depth relative to processing capacity). If the ratio exceeds 2.0, launch more workers. If the ratio drops below 0.5, terminate workers.

**Instance lifecycle**: Use a lifecycle hook on the terminating transition to upload transcoding logs before shutdown. The worker should handle SIGTERM gracefully -- complete the current job but do not pick up a new one. Use scale-in protection on workers that are actively transcoding.

**Cost optimization**: Use spot instances for 80% of the worker fleet with a fallback to on-demand if spot is unavailable. Use the "lowest-price" allocation strategy across multiple instance types (c5.large, c5a.large, c5d.large).

**Failure handling**: If a worker fails mid-transcoding, the job should appear back in the queue (SQS visibility timeout expires) and be picked up by another worker. Track failed jobs per input file and alert if the failure count exceeds 3.

### Challenge 2: Design Autoscaling for a Multi-Tenant SaaS Platform

**Prompt**: Design the autoscaling strategy for a multi-tenant SaaS platform with 500 customers. Each customer has a service-level agreement (SLA) specifying their guaranteed throughput and latency. Tenants range from small (10 req/s, 99.9% uptime) to enterprise (10,000 req/s, 99.99% uptime, under 50ms p99 latency). The platform runs on Kubernetes.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| Tenant isolation | Per-tenant resource pools or weighted fair queuing | Namespace isolation | No tenant isolation |
| Scaling granularity | Pod-level and node-level autoscaling with coordinated policies | Pod-level autoscaling only | Node-level autoscaling only |
| SLA awareness | Scaling policies defined per SLA tier, priority-based scheduling | Single scaling policy for all tenants | No SLA consideration |
| Burst handling | Pre-computed burst budgets per tenant, per-tenant rate limiting | Global rate limiting | No burst handling |

**Model Approach**:

**Tenant isolation**: Use Kubernetes namespaces per tenant with resource quotas. Enterprise tenants get dedicated node pools (with node autoscaling). Small tenants share node pools with resource limits. Use priority classes -- enterprise tenants have higher priority and can preempt small-tenant pods during resource contention.

**Scaling granularity**: Use HPA at the pod level and Cluster Autoscaler at the node level. The HPA uses request count per pod as the primary metric, with a target of 500 req/s per pod for standard tenants and 1000 req/s per pod for enterprise tenants (enterprise pods run on larger nodes with more capacity).

**SLA awareness**: Define scaling policies per SLA tier. Enterprise tenants get aggressive scaling (shorter cooldown, larger step sizes, higher max pod count). Small tenants get conservative scaling (longer cooldown, smaller step sizes). Use Priority-Based Autoscaling -- when the node pool is full, enterprise tenant pods get the new capacity first.

**Burst handling**: Each tenant has a burst budget (e.g., 2x their committed throughput for 5 minutes). The burst budget is enforced by a rate limiter at the ingress level. If a tenant exceeds their burst budget, requests are queued or returned 429. The autoscaler watches the combined load across all tenants and scales the node pool accordingly. During burst events, the autoscaler first uses existing node pool headroom before launching new nodes.

### Challenge 3: Design Autoscaling for a Real-Time Gaming Backend

**Prompt**: Design the autoscaling system for a real-time multiplayer game backend. The game supports 10,000 concurrent players per match, with 50 matches running simultaneously. Each player maintains a WebSocket connection to a game server. Game servers are stateful (they hold the game state in memory). Players join and leave matches continuously. A new match starts every 2 minutes during peak hours.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| State management | Describes state migration, session handoff, and graceful drain | Describes connection draining | Does not address state |
| Scaling metric | Uses connection count and match scheduling as metrics | Uses CPU or memory only | No metric specified |
| Scale-in handling | Drains matches before termination, moves players to new server | Kills connections with reconnection logic | No drain mechanism |
| Match lifecycle integration | Autoscaler aware of match start/end cycles | Manual match-server binding | No integration |

**Model Approach**:

**State management**: Each game server tracks its current number of players and matches. The autoscaler does not terminate a server that has active matches. Instead, the match scheduler stops assigning new matches to servers that are marked for scale-in. When the server's last match ends (players leave or the match completes), the server is terminated. This is a "drain before delete" strategy.

**Scaling metric**: The primary metric is the number of available player slots across all game servers. Each server has a max capacity of 10,000 players. Scale-out when the total available slots drops below 20% of total capacity (meaning 80% of slots are occupied). Scale-in when available slots exceed 50% for 5 minutes.

**Scale-in handling**: Before terminating a server, mark it as "draining" in the match scheduler. No new matches are assigned to it. Existing players continue playing their current match. When the match ends, players are not reassigned to the draining server for their next match. When all matches on the server have ended, the server is terminated. Add a maximum drain timeout of 30 minutes -- if a match takes longer, the server stays alive until the match ends.

**Match lifecycle integration**: The match scheduler and autoscaler share a control plane. Before starting a new match, the scheduler checks with the autoscaler whether capacity is available. If not, the scheduler waits for capacity. The autoscaler pre-warms servers during known peak hours (evenings, weekends) using scheduled scaling, so match start is not delayed by instance warm-up.

### Challenge 4 (Design): Autoscaling with Noisy Neighbors

**Prompt**: You run a multi-tenant SaaS platform on a shared Kubernetes cluster. 50 tenants share the same node pool. Each tenant's traffic pattern is different: Tenant A has steady traffic, Tenant B has spikey traffic (social media campaigns), and Tenant C has batch processing traffic that runs at night. The noisy neighbor problem occurs when Tenant B's spike consumes all node resources, starving Tenant A and Tenant C.

Design an autoscaling solution that prevents noisy neighbor problems while maintaining cost efficiency.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| Isolation strategy | Per-tenant node pools with cluster autoscaler, plus burst capacity | Namespace resource quotas only | No isolation described |
| Burst handling | Tenant B bursts into dedicated burst pool, then returns to shared pool | Tenant B uses its own dedicated pool | No burst handling |
| Cost efficiency | Burst pool uses spot instances, shared pool uses reserved instances | Mix of spot and on-demand | All on-demand |
| Scaling coordination | HPA per tenant, CA per pool, coordinated to prevent cascade | HPA per tenant only | Single HPA for all tenants |

**Model Approach**:

**Isolation strategy**: Use three node pools:
1. **Shared pool**: Reserved instances (60% cheaper). All tenants share this pool. Resource quotas limit each tenant to 40% of the pool.
2. **Burst pool**: Spot instances (70% cheaper). Tenant B's pods are scheduled here during spikes. The burst pool has its own cluster autoscaler with 60-second warm-up.
3. **Batch pool**: Spot instances. Tenant C's batch jobs run here at night. This pool can scale to zero during the day.

**Burst handling**: Tenant B's HPA uses a custom metric that measures the deviation from the tenant's baseline traffic. When traffic exceeds 2x the baseline, Tenant B's pods include a nodeSelector that targets the burst pool. When traffic returns to baseline, the pods are scheduled back to the shared pool. A lifecycle hook on the burst pool handles the transition.

**Cost efficiency**: The shared pool uses 3-year reserved instances for maximum savings. The burst pool uses spot instances with a fallback to on-demand. The batch pool uses spot instances exclusively. The cluster autoscaler uses the "cost-optimized" allocation strategy across pools.

**Scaling coordination**: Each tenant has its own HPA. Each node pool has its own cluster autoscaler. A global coordinator monitors the total cluster utilization. If the total utilization exceeds 80%, the coordinator scales up the burst pool preemptively, anticipating that Tenant B may need it.

### Challenge 5 (Design): Autoscaling for a Zero-Downtime Database Migration

**Prompt**: You need to perform a zero-downtime migration of a database that serves 50,000 req/s. The migration involves moving from PostgreSQL 13 to PostgreSQL 16 on a new instance type. The migration must complete within 4 hours with zero downtime and zero data loss. The application is behind an autoscaling group that handles variable traffic.

Design the migration strategy that coordinates application autoscaling with the database migration.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| Migration strategy | Blue/green with dual-write, autoscaling coordinated | Read replica promotion | Direct upgrade with downtime |
| Autoscaling coordination | Autoscaler pauses scale-in during migration, adds buffer capacity | Autoscaler continues normally | No coordination |
| Rollback plan | Full rollback procedure with autoscaling adjustments | Basic rollback | No rollback |
| Testing | Load test with production traffic patterns before migration | Basic load test | No testing |

**Model Approach**:

**Migration strategy**: Use a blue/green database migration with dual-writes. The application writes to both the old (blue) and new (green) databases. Reads come from the blue database initially. After the data is validated on the green database, reads are switched to the green database. The blue database is retained for 24 hours as a rollback option.

**Autoscaling coordination**: Two hours before the migration, increase the autoscaling group's minimum by 50% and set the cooldown to 600 seconds. This creates a buffer of extra instances that can absorb any performance degradation during the migration. During the dual-write phase, the application requires 2x the database connections, so the application instances may need more memory. The autoscaler should use memory utilization as a secondary metric during the migration.

**Rollback plan**: If the green database shows elevated error rates or latency, the application switches back to the blue database by toggling a feature flag. The autoscaler continues normally during the rollback -- no special configuration needed. After 24 hours with no issues, the blue database is decommissioned.

### Challenge 6 (Design): Autoscaling for a Real-Time Collaborative Application

**Prompt**: Design autoscaling for a real-time collaborative document editing application (like Google Docs). The application supports 1 million concurrent users. Each user has a WebSocket connection to the server. Documents are replicated across servers using CRDTs (Conflict-Free Replicated Data Types). The system must handle flash crowds (a popular document goes viral) and gradual growth.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|----------|-------------------|--------------|-------------|
| State management | CRDT-based state with dynamic shard rebalancing | Sticky sessions with session migration | No state management |
| Scaling metric | Concurrent connections + document popularity | Concurrent connections only | CPU only |
| Flash crowd handling | Pre-warmed buffer pool, doc-level sharding, load shedding | Buffer pool only | Reactive scaling only |
| Connection draining | Graceful handoff to peer instance, CRDT state sync | Connection close with reconnection | No drain |

**Model Approach**:

**State management**: Each document is a CRDT that is replicated across 3 server instances. When a new server instance is added, the CRDT replicas are rebalanced -- some documents are moved to the new instance. The CRDT ensures that no data is lost during the rebalance. The rebalance is a background process that does not block user edits.

**Scaling metric**: The primary metric is the number of concurrent WebSocket connections per instance. Each instance can handle 10,000 concurrent connections. The target is 8,000 connections per instance (80% utilization). A secondary metric is document popularity -- if a document has more than 500 concurrent editors, it is sharded across multiple instances.

**Flash crowd handling**: When a document goes viral (100,000 concurrent editors on one document), the system must scale rapidly. A pre-warmed buffer pool of 50 instances is maintained. These instances are ready to handle connections within 10 seconds. The document is sharded across the buffer instances. Load shedding is the last resort -- if all buffer instances are exhausted, new editors see a "read-only" mode until capacity is available.

**Connection draining**: When an instance is scaled in, it gracefully hands off its WebSocket connections to peer instances. The CRDT state is synchronized to the peer. The handoff takes 1-2 seconds. Users see a brief "reconnecting" message but do not lose any edits.

## Tier 3: Expert

### Challenge 1: The Global Autoscaling Coordinator

**Prompt**: Design a global autoscaling coordinator for a platform with 10 regions, 200 microservices, and 50,000 instances. The coordinator must handle cross-region traffic shifts (when a region fails, traffic is redirected to another region), coordinated scaling (Service A scaling out should not overwhelm Service B), and cost optimization across regions.

**Discussion Guide**:

This challenge tests your ability to think at the level of a platform engineering team designing a system that spans the entire company.

**Key considerations**:

1. **Hierarchical control**: A centralized coordinator cannot make real-time decisions for 200 services across 10 regions. The latency would be too high, and the failure domain would be too large. Instead, use a hierarchical design: a global coordinator sets high-level policies (budgets, targets, constraints), and each region has its own controller that makes local scaling decisions within the global bounds.

2. **Cross-region traffic shifts**: When us-east-1 fails, Route53 routes traffic to us-west-2. The global coordinator detects the failover and tells the us-west-2 controller to expect 2x traffic. The controller pre-warms capacity by launching 50% more instances before the traffic arrives. Without this coordination, us-west-2 would be overwhelmed by the redirected traffic.

3. **Cascading failure prevention**: When Service A scales out, it calls Service B more frequently. If Service B cannot scale out fast enough, it becomes a bottleneck. The coordinator should implement a "scaling budget" per service per region. Service A is allocated 100 instances total. If it needs more, the request goes to the coordinator, which checks whether the downstream services can handle the additional load. If not, the coordinator queues the request.

4. **Cost optimization**: The coordinator runs a daily optimization job that analyzes traffic patterns and proposes changes to scaling policies. The optimization considers: spot vs on-demand mix, reserved instance coverage, region distribution, and cooldown adjustments. The proposals are reviewed by a human before deployment.

5. **Global predictive scaling**: The coordinator runs a global ML model that predicts traffic by region and service 24 hours ahead. The forecast incorporates: time of day, day of week, upcoming events (product launches, marketing campaigns), and historical trends. The forecast produces a capacity plan that is distributed to each region's controller.

### Challenge 2: Designing for Zero-Warm-Up Autoscaling

**Prompt**: Design an autoscaling system where the warm-up time is zero -- new capacity must be available to serve traffic instantly. This is required for a service that cannot tolerate the 3-5 minute warm-up delay of traditional VM autoscaling.

**Discussion Guide**:

This challenge tests your ability to think outside the traditional autoscaling model and design novel solutions.

**Key approaches**:

1. **Pre-warmed buffer pool**: Maintain a pool of pre-warmed instances that are fully initialized but not serving traffic. The buffer pool is sized at 20% of the peak capacity. When a scale-out is needed, instances are moved from the buffer pool to the serving pool instantly (they just need to register with the load balancer, which takes seconds, not minutes). The buffer pool is replenished by launching new instances in the background. This approach trades compute cost for warm-up time.

2. **Traffic shaping with queuing**: Instead of scaling instantly, buffer excess traffic in a high-throughput queue and process it as capacity becomes available. For a video transcoding service, this is natural (videos wait in a queue). For a real-time API, this is harder but possible with careful load shedding. The queue absorbs the spike, and the autoscaler processes the backlog gradually. This does not reduce warm-up time but eliminates the need for instant capacity.

3. **Container pre-fetching**: Use a sidecar pattern where new containers are started before traffic arrives. The orchestrator (Kubernetes) pre-fetches container images to all nodes in the cluster. When a scale-out is needed, the container starts in 1-2 seconds because the image is already cached. Combined with HPA that actively manages the pod count based on real-time metrics, this reduces warm-up from minutes to seconds.

4. **Serverless offloading**: Route the excess traffic to a serverless platform (AWS Lambda, Cloudflare Workers) during the warm-up gap. The serverless layer handles the surge while traditional capacity is being provisioned. This is a hybrid approach that uses serverless as a temporary capacity buffer. The serverless layer handles the short-term spike, and the autoscaled VM layer handles the sustained load.

5. **Predictive pre-warming with reinforcement learning**: Use a reinforcement learning agent that learns the traffic patterns and pre-warms capacity at the optimal time. The agent is trained on historical traffic data and learns to pre-warm capacity 5-10 minutes before predicted spikes. The reward function penalizes under-provisioning (latency spikes) and over-provisioning (wasted cost). After training, the agent makes pre-warming decisions autonomously.

**Evaluation criteria for this challenge**:

- **Architecture**: Does the solution have a clear architecture diagram? Are the components and their interactions described?
- **Trade-off analysis**: Does the candidate discuss the cost implications of zero-warm-up approaches (pre-warmed buffer pools are expensive)?
- **Feasibility**: Is the solution implementable with current cloud provider capabilities, or does it require custom infrastructure?
- **Generalization**: Does the solution apply to multiple workload types, or is it specific to one scenario?

### Challenge 3 (Expert): Autoscaling with Reinforcement Learning

**Prompt**: Design an autoscaling system that uses reinforcement learning (RL) to optimize scaling decisions. The RL agent should learn the optimal scaling policy over time, balancing the trade-off between cost and performance. The system must handle changing traffic patterns, seasonal variations, and unexpected spikes.

**Discussion Guide**:

This challenge tests your ability to design a learning system that improves over time.

**Key considerations**:

1. **State space**: The RL agent's state should include the current metric values (CPU, request count, latency), the current capacity, the time of day, the day of week, and recent scaling history. The state space is continuous and high-dimensional, so function approximation (neural network) is needed.

2. **Action space**: The agent can take actions: add N instances, remove N instances, or do nothing. The action space is discrete but with multiple possible magnitudes. The agent must learn when to scale and by how much.

3. **Reward function**: The reward function encodes the trade-off:
   - Positive reward for keeping the metric at the target
   - Negative reward for latency degradation (below the SLO)
   - Negative reward for scaling actions (cost of launching and terminating)
   - Large negative reward for errors (5xx responses)

   The reward function is the key design choice. If the negative reward for scaling is too high, the agent will never scale. If it is too low, the agent will scale too aggressively.

4. **Training**: The agent is trained in a simulation environment using historical traffic data. The simulator models the cloud provider's API latency, instance warm-up time, and metric collection intervals. After 1,000 simulated days, the agent is deployed to production in a shadow mode (it makes recommendations but does not execute them). After 1 week of shadow mode validation, the agent is promoted to production with a human-in-the-loop override.

5. **Safety constraints**: The RL agent operates within safety bounds. The minimum and maximum capacity are enforced by the autoscaling group configuration, not by the agent. If the agent recommends an action that would violate the bounds, the action is clamped. If the agent's recommendations cause errors for more than 5 minutes, the system falls back to the standard target tracking policy.

6. **Practical challenges**: RL-based autoscaling is not widely deployed in production because of the difficulty of reward function design, the risk of catastrophic actions during training, and the regulatory requirements for explainability. Most production systems use simpler heuristics (target tracking, step scaling) with manual tuning. The RL approach is more promising for internal tools and infrastructure optimization where the failure cost is lower.

### Challenge 4 (Expert): Designing Autoscaling for a Low-Latency Trading Platform

**Prompt**: Design autoscaling for a high-frequency trading platform that processes 1 million trades per second. The platform runs on bare metal servers in a co-located data center. The latency requirement is under 10 microseconds for the matching engine and under 1 millisecond for the order management system. The traffic pattern is driven by market events (earnings reports, economic data releases, Fed announcements) that cause 100x traffic spikes in milliseconds.

**Discussion Guide**:

This challenge tests your ability to design autoscaling at the extreme edge of performance requirements.

**Key considerations**:

1. **Autoscaling is not the answer**: For a high-frequency trading platform, autoscaling is not appropriate for the real-time trading path. The warm-up time of any autoscaling system (even container-based) is measured in seconds, not microseconds. The market event happens in milliseconds. Reactive autoscaling cannot keep up.

2. **Pre-provisioned capacity**: The correct approach is to pre-provision capacity for the worst-case scenario. The trading platform must have enough capacity to handle the 100x spike at all times. The cost of pre-provisioning is the cost of doing business in this industry.

3. **Predictive scheduling for known events**: For known market events (Fed announcements at 2:00 PM ET, earnings reports after market close), the platform can schedule capacity increases ahead of time. The capacity is provisioned 5 minutes before the event and maintained for 30 minutes after.

4. **Load shedding for unknown events**: For unexpected events (flash crash, geopolitical event), the platform uses load shedding rather than autoscaling. The matching engine prioritizes orders from the top 100 customers and sheds lower-priority orders. The order management system queues non-critical orders and processes them after the event.

5. **Autoscaling for non-critical path**: Autoscaling is appropriate for the non-real-time parts of the platform: the risk management system, the reporting system, the backtesting system, and the data pipeline. These systems can tolerate 5-10 minute scaling delays. They use standard target tracking on CPU and queue depth.

6. **Bare metal considerations**: Bare metal servers cannot be autoscaled in the traditional sense -- you cannot launch a new bare metal server in 5 minutes. The pre-provisioned capacity must be physical servers that are already installed and configured. The autoscaling of bare metal servers is more like capacity planning: order servers 4-6 weeks in advance, install them, and have them ready to activate when needed.

### Challenge 5 (Expert): Self-Tuning Autoscaling with Bayesian Optimization

**Prompt**: Design an autoscaling system that automatically tunes its own parameters (cooldown period, target tracking value, evaluation periods, step sizes) using Bayesian optimization. The system should continuously learn from its performance and adjust without human intervention.

**Discussion Guide**:

This challenge tests your ability to design a closed-loop optimization system.

**Key considerations**:

1. **Parameter space**: The autoscaler has approximately 10 tunable parameters. A full grid search over all parameters would require thousands of evaluations. Bayesian optimization is designed for this problem -- it builds a probabilistic model of the objective function and selects the most promising parameters to evaluate next.

2. **Objective function**: The objective function combines cost and performance. A simple formulation:
   ```
   objective = w1 * (p99_latency / latency_SLO) + w2 * (actual_cost / optimal_cost)
   ```
   Where w1 and w2 are weights that represent the relative importance of performance and cost. The objective is to minimize this value. If latency exceeds the SLO, the objective function penalizes heavily.

3. **Safety constraints**: The tuner should not change parameters that could cause a production incident. Safety constraints limit the parameter search space:
   - Cooldown period: minimum 60 seconds, maximum 600 seconds
   - Target tracking value: minimum 30%, maximum 80%
   - Evaluation periods: minimum 1, maximum 5
   - Scale-out step size: minimum 10%, maximum 200%

4. **Evaluation protocol**: Each parameter configuration is evaluated for 24 hours. The tuner measures the objective function over the evaluation period. After 24 hours, the tuner selects the next parameter configuration to evaluate. The system runs a new configuration every 24 hours, so it takes approximately 10 days to converge to the optimal configuration.

5. **Practical challenges**: Self-tuning autoscaling is an active research area. The main challenges are: (1) the objective function is noisy (traffic patterns change daily), (2) counterfactual evaluation is hard (you cannot run two configurations simultaneously), and (3) the optimal configuration changes as traffic patterns evolve. Despite these challenges, self-tuning autoscaling shows promise for large-scale systems where manual tuning is impractical.

- [ ] Scaling metric choice (why this metric, not others)
- [ ] Scaling policy type (target tracking, step, scheduled, predictive)
- [ ] Minimum and maximum capacity bounds
- [ ] Cooldown period and warm-up time consideration
- [ ] Health check configuration (path, interval, thresholds)
- [ ] Connection draining for scale-in
- [ ] Multi-AZ distribution
- [ ] Mixed instance types for cost optimization
