# Autoscaling -- Trade-offs and Failure Modes

## Core Tension

The fundamental tension in autoscaling is between **responsiveness** and **stability**.

A system that scales too aggressively responds quickly to traffic spikes but wastes money on excess capacity and may thrash (scale out, then immediately scale in). A system that scales too conservatively saves money but leaves the application under-provisioned during traffic spikes, causing latency degradation and errors.

This tension is inherent because the metric used for scaling decisions is always a lagging indicator. CPU rises after the traffic spike begins. The autoscaler responds to the rise, but the response takes time. By the time the response arrives, the traffic pattern may have already changed.

The optimal balance depends on your business priorities:

- **Cost-sensitive**: Accept slower scaling and occasional latency degradation during unexpected spikes. Over-provision by 10-15% instead of 20-30%.
- **Performance-sensitive**: Over-provision aggressively. Accept higher cost to ensure latency stays under the SLO. Use predictive scaling to pre-warm capacity.
- **Availability-sensitive**: Maintain significant buffer capacity (30-50% headroom). Use multiple AZs and regions. Accept higher cost as insurance against failure.

## Decision Matrix

| Factor | Aggressive Scaling | Conservative Scaling | Best For |
|--------|--------------------|---------------------|----------|
| Target tracking value | 30-40% CPU target | 60-70% CPU target | Performance vs cost |
| Cooldown period | 60-120 seconds | 300-600 seconds | Quick recovery vs stability |
| Evaluation periods | 1 period | 3-5 periods | Responsiveness vs noise filtering |
| Scale-out step size | 100% increase | 25-50% increase | Rapid growth vs gradual growth |
| Scale-in step size | 10% decrease | 25-50% decrease | Cost savings vs stability |
| Warm-up parameter | Disabled | Enabled in metric aggregation | Simplicity vs accuracy |
| Predictive scaling | Required for flash crowds | Optional for stable patterns | Flash crowds vs predictable traffic |
| Buffer capacity | 10% over target | 30-50% over target | Cost savings vs safety margin |
| Instance type | Spot + On-Demand mixed | On-Demand only | Cost savings vs reliability |
| Scale-in protection | Disabled | Enabled on all instances | Simplicity vs safety |

### When to Choose Each Column

Choose **Aggressive Scaling** when:
- Traffic spikes are large and unpredictable
- Your application has a high tolerance for cost variability
- Warm-up time is short (containers, not VMs)
- Latency SLO is strict (p99 under 100ms)
- You have a large on-demand budget

Choose **Conservative Scaling** when:
- Traffic patterns are predictable
- Your budget is fixed and tight
- Warm-up time is long (5+ minutes for VMs)
- The cost of over-provisioning is worse than the cost of occasional latency degradation
- You are running batch or async workloads where latency is less critical

## CAP Theorem Positioning

Autoscaling is not a data system, so the CAP theorem does not apply directly. However, the same trade-off between consistency and availability manifests in a different form:

- **Consistency of capacity**: The autoscaler's view of desired capacity should match the actual number of running instances. Inconsistent state can lead to incorrect scaling decisions.
- **Availability of scaling**: The autoscaler should always be able to launch or terminate instances when needed.
- **Partition tolerance**: The autoscaler should continue operating correctly when network partitions occur between components.

In practice, autoscaling systems sacrifice **consistency** for **availability** and **partition tolerance**. When a partition occurs between the metrics pipeline and the autoscaler controller, the controller must make a decision with stale data. The typical choice is to not scale (conservative approach) rather than risk an incorrect scale action.

## Failure Modes

### Failure 1: Thrashing (Scale-Out/Scale-In Oscillation)

Thrashing occurs when the autoscaler repeatedly scales out and scales in within a short time window. Each scale-out reduces the metric, which triggers a scale-in, which raises the metric, which triggers another scale-out.

**Root Causes**: Cooldown period too short. Metric oscillates around the threshold. Multiple conflicting scaling policies. Warm-up time exceeds cooldown period.

**Symptoms**: Alternating scale-out and scale-in events in the activity log. Desired capacity oscillates between two values. Instances are launched and terminated within minutes of each other. Cloud bill spikes from the constant launch/terminate activity.

**Mitigation**: Increase the cooldown period to 2x the warm-up time. Use a single target tracking policy instead of multiple step scaling policies. Widen the "no-action zone" between scale-out and scale-in thresholds. Enable the warm-up parameter to exclude new instances from metric aggregation.

### Failure 2: Runaway Scaling

Runaway scaling occurs when the autoscaler keeps adding instances without stopping, far beyond what is needed. The autoscaler launches instance after instance, each one immediately overwhelmed because the underlying problem is not capacity -- it is a database bottleneck, a slow dependency, or a code defect.

**Root Causes**: Application has a scalability bottleneck that prevents new instances from reducing the metric. Memory leak causes each instance to degrade over time. A downstream dependency is throttling all instances equally. A code deployment introduced an infinite loop or a blocking operation.

**Symptoms**: Instance count grows to the maximum size. New instances show the same high CPU as existing instances. Error rate does not improve with additional instances. Cloud bill increases rapidly.

**Mitigation**: Always set a maximum size on the autoscaling group. Monitor the correlation between instance count and error rate. If error rate stays constant or increases with more instances, something is wrong. Implement a "guardrail" alarm that suspends the autoscaling process if the error rate exceeds a threshold. Investigate application-level bottlenecks before increasing capacity.

### Failure 3: Slow Scale-Out (Under-Provisioning)

The autoscaler launches instances too slowly to handle the traffic spike. By the time new instances are online, the spike has already caused latency degradation, errors, or downtime.

**Root Causes**: Warm-up time is too long. Evaluation period is too long. The scale-out step size is too small. The cloud provider has run out of capacity for the instance type in the AZ.

**Symptoms**: CPU stays above 80% for extended periods despite scaling activity. P99 latency spikes during the warm-up period. 5xx errors appear. The desired capacity increases but InstancesInService does not.

**Mitigation**: Reduce warm-up time by using pre-baked AMIs, lighter container images, or instance refresh. Shorten the evaluation period (use 1-minute metrics instead of 5-minute). Increase the scale-out step size to add instances more aggressively (200% increase instead of 100%). Use predictive scaling to pre-warm capacity. Configure capacity reservations in the cloud provider to guarantee instance availability.

### Failure 4: Scale-In Storm

A scale-in storm occurs when the autoscaler terminates too many instances too quickly. The sudden reduction in capacity causes the remaining instances to become overloaded, which raises the metric again and triggers another scale-out.

**Root Causes**: The scale-in step size removes too many instances at once. The scale-in threshold is set too aggressively. The cooldown period after scale-in is too short. The application has a hard dependency on a specific instance count for load distribution.

**Symptoms**: A cluster of scale-in events followed immediately by scale-out events. Sawtooth pattern on the instance count graph. CPU oscillates between low and high. Cloud bill increases from constant churn.

**Mitigation**: Use gradual scale-in policies. Instead of removing 50% of instances, remove 10-20% per evaluation period. Implement a scale-in stabilization window (Kubernetes HPA has this by default -- it waits 5 minutes before scaling down after any scaling event). Monitor the remaining instances' metrics during scale-in and stop if they exceed a safety threshold.

### Failure 5: Stale Metrics After Deployment

After a deployment, all instances are replaced. The new instances may have different performance characteristics than the old ones (new code is slower or faster). The autoscaler's historical data is stale -- it learned patterns from the old code. The scaling behavior may be incorrect until enough data is collected.

**Root Causes**: A new deployment changes CPU per request (slower code needs more CPU, so the autoscaler sees higher CPU and scales out unnecessarily). A performance improvement reduces CPU per request (the autoscaler sees lower CPU and scales in, potentially leaving the system under-provisioned for the next spike).

**Symptoms**: Unexpected scaling activity immediately after a deployment. Desired capacity changes without corresponding traffic changes. The error rate may increase as the system adjusts.

**Mitigation**: Reset the autoscaler's historical data after a deployment. Perform a controlled rollout (blue/green or canary) with separate autoscaling groups for old and new code. Monitor the new group's scaling behavior before routing full traffic to it. Temporarily increase the cooldown period after a deployment to allow the system to stabilize.

## Anti-Patterns: Hall of Shame

### Anti-Pattern 1: CPU as the Only Metric

Using only CPU utilization for all scaling decisions. CPU is a lagging indicator and may not reflect actual load for I/O-bound or network-bound applications. A web server waiting on database queries may show low CPU while the request queue is backing up.

**Fix**: Use request count per target, queue depth, or custom application metrics alongside CPU. For I/O-bound applications, use latency percentiles or connection count as the primary metric.

### Anti-Pattern 2: Same Threshold for Scale-Out and Scale-In

Setting the same metric threshold for both directions. This creates a binary on/off switch at the threshold boundary. When CPU is at 51% and the threshold is 50%, the system scales out. When CPU drops to 49%, the system scales in. At 50% CPU, the system oscillates indefinitely.

**Fix**: Always use different thresholds for scale-out and scale-in with a dead zone between them. For target tracking, the system automatically has this dead zone through the metric stabilization. For step scaling, use 60% for scale-out and 40% for scale-in, giving a 20-percentage-point dead zone.

### Anti-Pattern 3: Ignoring Warm-Up Time

The autoscaler configuration does not account for the time it takes a new instance to become ready. The system scales out, sees no change in the metric for 3 minutes, and scales out again. It repeats this until the first batch of instances comes online and the metric drops sharply.

**Fix**: Set the cooldown period to at least 2x the warm-up time. Enable the warm-up parameter if available. Use predictive scaling to pre-warm capacity so the reactive component has less work to do.

### Anti-Pattern 4: Overly Aggressive Scale-In

The system saves money by scaling in as fast as possible. When traffic drops, the autoscaler terminates 50% of instances in one action. The remaining instances are overwhelmed by the residual traffic. The metric spikes, and the system scales out again. The cost savings are lost to the churn.

**Fix**: Scale in gradually. Remove 10-20% of capacity per evaluation period. Use a 5-minute stabilization window before scale-in. Consider the cost of churn (launch/terminate overhead) versus the cost of running an extra instance.

### Anti-Pattern 5: Autoscaling Stateless Services Without Connection Draining

Terminating instances without draining connections. Clients that were connected to the terminated instance receive connection reset errors. For WebSocket applications, all active connections are dropped.

**Fix**: Always enable connection draining on the load balancer. Set the drain timeout to at least the maximum expected request duration. For long-lived connections, implement graceful shutdown in the application and use lifecycle hooks to wait for drain completion.

## When NOT to Use Autoscaling

### Workloads That Should NOT Autoscale

**Stateful services with in-memory data**: Stateful services like single-node caches, session stores, and in-memory databases should not autoscale without careful consideration. Adding a new instance does not automatically distribute the existing data. The new instance has an empty cache and must be warmed up, which can take hours for a large cache.

**Stateful services solutions**: Use a distributed cache (Redis Cluster) that handles shard rebalancing. Use sticky sessions with a load balancer that maintains session affinity. For databases, use read replicas with autoscaling on the replica count rather than the primary instance.

**Batch jobs with fixed resource requirements**: A batch processing job that requires exactly N instances for M hours should use a fixed-size cluster. Autoscaling introduces uncertainty in completion time because the number of workers varies. Use scheduled scaling to set the exact cluster size for the batch window.

**Real-time streaming processors**: Stateful stream processors (Flink, Kafka Streams) require careful handling of state redistribution when instances change. Autoscaling a stateful stream processor requires task redistribution, checkpoint synchronization, and state migration. This is possible but is not a simple CPU-based autoscaling setup.

**GPU-intensive workloads**: GPU instances are expensive and have limited availability. Autoscaling GPU instances can be cost-prohibitive if GPU instances are launched speculatively. Use scheduled scaling for known GPU workloads and reserve GPU capacity in advance.

**Workloads with extremely long warm-up times**: If your application takes more than 15 minutes to start and become healthy, autoscaling is ineffective. By the time new capacity is available, the traffic pattern has likely changed. Consider architecture changes to reduce warm-up time before implementing autoscaling.

### Alternatives to Autoscaling

- **Buffer capacity**: Run a fixed set of instances with significant headroom (50% utilization target). This is simpler and more predictable for stable traffic patterns.
- **Scheduled capacity**: Adjust capacity on a fixed schedule based on known patterns. No reactive complexity.
- **Spot instance fallback**: Use a fixed number of on-demand instances plus a dynamic pool of spot instances. The spot pool absorbs traffic variations without formal autoscaling.
- **Queuing and backpressure**: Instead of scaling the serving layer, buffer excess traffic in a queue and process it as capacity allows. This absorbs spikes without adding servers.

## Failure Modes (Continued)

### Failure 6: INSUFFICIENT_DATA State

The autoscaler receives INSUFFICIENT_DATA for the metric -- meaning no data is available. This can happen when the metrics agent is down, the monitoring system is unreachable, or the network between instances and the monitoring system is partitioned.

**Root Causes**: The CloudWatch agent on all instances crashes. A network ACL blocks metrics traffic. The monitoring system (Prometheus, Datadog) is itself down. The instance's IAM role does not have permissions to publish metrics.

**Symptoms**: The autoscaling alarm shows INSUFFICIENT_DATA state. The autoscaler does not take any scaling action. If this persists during a traffic spike, the system remains under-provisioned.

**Mitigation**: Configure the alarm to treat INSUFFICIENT_DATA as "breaching" (alarm) or "not breaching" (OK). A common approach is to treat INSUFFICIENT_DATA as "not breaching" during normal hours (do not scale based on missing data) but "breaching" during known peak hours (assume the worst and scale out). Implement a "heartbeat" metric that is always published, and alarm if the heartbeat is missing for 5 minutes.

### Failure 7: Instance Limit Exceeded

The autoscaler tries to launch an instance but the cloud provider account's instance limit is reached. Each AWS account has a default limit of 20-100 instances per region per instance type.

**Root Causes**: The instance limit is too low for the autoscaling maximum. Another team's deployment consumed the remaining limit. A previous runaway scaling action consumed the limit.

**Symptoms**: The autoscaling activity log shows "Failed to launch instance. You have requested more instances than your current instance limit."

**Mitigation**: Request a limit increase from the cloud provider early (before you need it). Monitor the instance usage against the limit and alarm when usage exceeds 80% of the limit. Use multiple instance types in the mixed instances policy so the autoscaler can fall back to a different type if the limit for one type is reached.

### Failure 8: Health Check Loop

The autoscaler terminates an instance because it fails health checks, launches a replacement, which also fails health checks, and so on indefinitely. This creates a constant churn of instances without adding any serving capacity.

**Root Causes**: The health check endpoint is misconfigured (wrong path, wrong port, wrong protocol). The application is crashing on startup due to a configuration error. The database is down, so the application's health check fails because it depends on the database.

**Symptoms**: The autoscaling activity log shows a constant stream of launch and terminate events. The desired capacity is high, but InstancesInService is zero. CloudWatch costs spike from the constant instance launch/terminate activity.

**Mitigation**: Use the health check grace period to give new instances time to initialize before health checks begin. Implement independent health checks -- the health check endpoint should verify that the application process is running without verifying downstream dependencies. A separate "readiness" check should verify downstream dependencies. The load balancer should use the health check for routing, and the autoscaler should use the readiness check for scaling.

## CAP Theorem Positioning (Extended)

The autoscaling system has its own consistency, availability, and partition tolerance trade-offs that parallel the CAP theorem.

### Consistency of Scaling State

When the autoscaler decides to launch 5 instances, it must ensure that exactly 5 instances are launched. If the autoscaler crashes after launching 3 instances but before updating its internal state, the system has a consistency problem. On recovery, the autoscaler may think it has 10 instances when it actually has 13.

**Approach**: The autoscaler uses the cloud provider's API to reconcile its state. The autoscaler periodically calls `DescribeAutoScalingGroups` to get the actual instance count and compares it against its internal desired capacity. If they differ, the autoscaler launches or terminates instances to match the desired capacity. This is an eventual consistency model.

### Availability of Scaling Decisions

When a network partition occurs between the autoscaler controller and the cloud provider's API, the autoscaler cannot launch or terminate instances. The desired capacity cannot change. The system is "stuck" at the current capacity.

**Approach**: Continue operating at the current capacity. The instances continue serving traffic. The system is not degraded -- it just cannot scale. When the partition heals, the autoscaler reconciles.

### Partition Tolerance

When the metrics pipeline is partitioned from the autoscaler controller, the autoscaler receives INSUFFICIENT_DATA. The autoscaler must decide whether to scale out (assume the worst) or stay at current capacity (assume the best).

**Approach**: Most autoscalers default to "do nothing" when data is missing. This is the safer approach for production systems. A system that does not scale during a partition is better than a system that scales incorrectly during a partition.

## Cost Analysis: Autoscaling vs. Fixed Capacity

The financial justification for autoscaling is straightforward:

**Without autoscaling (fixed for peak)**:
```
peak_instances = 100
hourly_cost = 100 * $0.0832 = $8.32/hour
monthly_cost = $8.32 * 730 = $6,073.60
```

**With autoscaling**:
```
avg_instances = (20 * 12h + 80 * 8h + 40 * 4h) / 24h = 40 (weighted average)
hourly_cost = 40 * $0.0832 = $3.33/hour (average)
monthly_cost = $3.33 * 730 = $2,429.44
```

**Monthly savings**: $6,073.60 - $2,429.44 = $3,644.16 (60% reduction)

However, autoscaling introduces additional costs:

- **Monitoring costs**: Detailed monitoring at 1-minute intervals costs $3.50 per instance per month. For 100 instances, that is $350/month.
- **Churn costs**: Each instance launch and termination costs approximately $0.01 in API calls and metadata storage. With 200 scaling events per day, that is $2/day or $60/month.
- **Engineering costs**: Configuring, testing, and maintaining autoscaling requires approximately 40 hours of engineering time per quarter. At $150/hour, that is $6,000/quarter or $2,000/month.

**Net savings**: $3,644.16 - $350 - $60 - $2,000 = $1,234.16/month (net 20% reduction)

Autoscaling is not free. The engineering cost of maintaining it can exceed the compute savings if the system is overly complex. For small systems (under 10 instances), the engineering cost may outweigh the compute savings.

## Anti-Patterns Hall of Shame (Continued)

### Anti-Pattern 6: Autoscaling Without Load Testing

Deploying autoscaling without any load testing. The first real traffic spike is the first time the autoscaling system is exercised. This inevitably reveals configuration errors, wrong metric choices, and incorrect threshold settings.

**Fix**: Load test every autoscaling configuration before deploying to production. Use tools like k6, Locust, or Artillery to generate traffic at 2x, 5x, and 10x the baseline. Verify that the autoscaler responds correctly at each level.

### Anti-Pattern 7: Using the Same Instance Type for Everything

Using a single instance type for all autoscaling groups. Different workloads have different resource profiles. A web server needs balanced CPU and memory. A video transcoder needs high CPU. A Redis cache needs high memory.

**Fix**: Choose instance types based on the workload profile. Use compute-optimized (c5, c6g) for CPU-bound workloads, memory-optimized (r5, r6g) for memory-bound workloads, and general-purpose (t3, m5) for balanced workloads.

### Anti-Pattern 8: Ignoring Cost of Churn

Focusing only on the instance runtime cost and ignoring the cost of launching and terminating instances. The churn cost includes the warm-up period (instances running but not serving traffic), the API calls, the metadata storage, and the engineering time to debug scaling issues.

**Fix**: Calculate the total cost of autoscaling, including churn. If the churn cost exceeds 20% of the runtime cost, adjust the scaling policies to be less aggressive. Use longer cooldown periods and smaller step sizes.

### Anti-Pattern 9: Autoscaling Based on Aggregate Metrics Only

Using only aggregate metrics (average CPU across all instances) without considering per-instance variation. A system with 10 instances at 50% average CPU could have 5 instances at 90% and 5 at 10%. The aggregate suggests no scaling is needed, but half the instances are saturated.

**Fix**: Monitor the maximum CPU across instances in addition to the average. If the maximum exceeds the target by a significant margin, scale out even if the average is within bounds. Set up a separate alarm on the maximum metric.

### Anti-Pattern 10: Autoscaling Without Graceful Degradation

Assuming that autoscaling will always keep the system healthy. Autoscaling has limits -- it cannot add instances faster than the cloud provider can provision them, and it cannot fix application-level bottlenecks.

**Fix**: Implement graceful degradation alongside autoscaling. When the system is overloaded, reduce service quality (lower video quality, longer cache TTLs, simplified responses) rather than serving errors. Use circuit breakers to protect downstream services. Use queue-based load shedding for non-critical requests.

## Anti-Patterns Hall of Shame (Continued)

### Anti-Pattern 11: Autoscaling Without Health Checks

Configuring an autoscaling group without health checks. The autoscaler launches instances, but if the application crashes on startup, the instance sits in the target group as "healthy" (because the autoscaler does not check) and serves errors to every request routed to it.

**Fix**: Always configure ELB health checks. Set the health check path to a dedicated endpoint that verifies the application can process requests. Set the unhealthy threshold to 3 and the interval to 10 seconds. The autoscaler will terminate instances that fail health checks and launch replacements.

### Anti-Pattern 12: Autoscaling Groups That Span Too Many AZs

Creating an autoscaling group that spans all 6 AZs in a region. The autoscaler tries to balance instances across AZs, but with 6 AZs and 3 instances, each AZ gets 0-1 instances. A single AZ failure could remove 1/3 of the capacity.

**Fix**: Use 3 AZs maximum. This provides redundancy without creating too many thin slices. If you need more AZs for capacity, use multiple autoscaling groups.

### Anti-Pattern 13: Autoscaling Monolithic Applications

Applying autoscaling to a monolithic application that cannot be horizontally scaled. The monolith has a single database, a single cache, and a single file system. Adding more instances overloads the database and cache because they are shared bottlenecks.

**Fix**: Decompose the monolith into services that can be independently scaled. Each service should have its own database or database shard, its own cache, and its own scaling policy. If decomposition is not possible, use vertical scaling (bigger instances) instead of horizontal scaling (more instances).

### Anti-Pattern 14: Autoscaling Without Notifications

Configuring autoscaling without any notification mechanism. The autoscaler silently scales out and in. The operations team has no idea that scaling events are happening until they see the cloud bill.

**Fix**: Configure SNS notifications for all scaling events. Send notifications to the operations team's Slack channel or email. Include the scaling action, the number of instances, the metric that triggered the action, and the current capacity. Create a weekly autoscaling report that summarizes scaling activity, costs, and efficiency.

### Anti-Pattern 15: Using Autoscaling to Cover Up Application Problems

Treating autoscaling as a band-aid for application performance problems. The application has a memory leak, so you add more instances to keep the memory below the threshold. The application has a database query that takes 10 seconds, so you add more instances to handle the increased request time.

**Fix**: Fix the root cause of the performance problem before implementing autoscaling. Autoscaling should handle traffic variations, not application defects. If the application has a memory leak, fix the leak. If the database query is slow, optimize the query or add an index.

## The Decision Matrix in Practice

The decision matrix is not a theoretical exercise. It has real, measurable consequences. Here are two real-world scenarios:

### Scenario A: E-Commerce Platform (Performance-Sensitive)

An e-commerce platform with a strict latency SLO (p99 under 200ms) chooses aggressive scaling:

- **Target tracking**: 35% CPU (leaves headroom for spikes)
- **Cooldown**: 120 seconds
- **Evaluation periods**: 1
- **Scale-out step**: 100% increase
- **Scale-in step**: 10% decrease
- **Predictive scaling**: Enabled
- **Buffer capacity**: 40% above target

**Result**: The platform handles Black Friday traffic (10x normal) with p99 latency under 180ms. The cost is 50% higher than the conservative approach, but the revenue from Black Friday ($2 million) far exceeds the additional compute cost ($12,000).

### Scenario B: Batch Processing Pipeline (Cost-Sensitive)

A batch processing pipeline with no real-time requirements chooses conservative scaling:

- **Target tracking**: 65% CPU (accepts occasional latency)
- **Cooldown**: 600 seconds
- **Evaluation periods**: 3
- **Scale-out step**: 25% increase
- **Scale-in step**: 50% decrease
- **Predictive scaling**: Disabled
- **Buffer capacity**: 10% above target

**Result**: The pipeline processes 500,000 jobs per day with an average processing time of 15 minutes. The cost is 40% lower than the aggressive approach. The processing time occasionally spikes to 22 minutes during traffic surges, but the batch jobs have a 60-minute SLA, so there is no business impact.

### Making the Decision

To decide between aggressive and conservative scaling, ask:

1. What is the cost of under-provisioning (latency, errors, lost revenue)?
2. What is the cost of over-provisioning (excess compute spend)?
3. What is the traffic volatility (how much and how fast does traffic change)?
4. What is the warm-up time (how long before new capacity helps)?
5. What is the business impact of degraded performance?

The answers to these questions determine the optimal position on the cost-performance curve.

## Summary of Trade-offs

The autoscaling decision matrix can be summarized in three key trade-offs:

### Trade-off 1: Responsiveness vs. Stability

An aggressive autoscaler responds quickly to traffic spikes but may over-provision and thrash. A conservative autoscaler is stable but may under-provision during spikes. The optimal balance depends on your traffic volatility and tolerance for latency.

**Rule of thumb**: Start conservative (60-second metric interval, 300-second cooldown, 2 evaluation periods). Tighten the parameters based on observed behavior. Loosen only if the system consistently under-performs during spikes.

### Trade-off 2: Cost vs. Performance

Autoscaling with aggressive settings (low CPU target, large buffer capacity) costs more but provides better performance. Conservative settings save money but accept occasional latency degradation.

**Rule of thumb**: The cost of over-provisioning is the cloud bill. The cost of under-provisioning is lost revenue from degraded user experience. Calculate both and find the equilibrium point. For most systems, the equilibrium is around 50% CPU target with 20% buffer capacity.

### Trade-off 3: Simplicity vs. Sophistication

A simple autoscaling setup (single target tracking policy on CPU) is easy to understand and debug. A sophisticated setup (predictive scaling, multiple metrics, lifecycle hooks, mixed instances) is more efficient but harder to maintain.

**Rule of thumb**: Start with a single target tracking policy on CPU. Add complexity only when the simple setup is not meeting your requirements. Each new feature adds configuration overhead, monitoring overhead, and potential failure modes. The 80/20 rule applies -- 80% of the benefit comes from the first 20% of the configuration.
