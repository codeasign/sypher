# Autoscaling at Scale

## Naive Implementation: 1,000 Users

A single server running a web application with a manual scale-up process. The server is provisioned with ample capacity for the expected load -- 4 vCPUs, 16 GB RAM, a 500 GB SSD. The deployment is manual: SSH into the server, pull the latest code, restart the service. If the server crashes, an engineer is paged and manually restores service from a backup AMI.

## Integration Patterns

### Pattern 1: Load Balancer Integration

The autoscaling group integrates with the load balancer through a target group. When instances launch, they automatically register with the target group. When instances terminate, they deregister after connection draining. The load balancer health check status determines whether the autoscaler considers an instance healthy.

```json
// AWS ALB Target Group Integration
{
  "AutoScalingGroup": {
    "TargetGroupARNs": ["arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-app/abc123"],
    "HealthCheckType": "ELB",
    "HealthCheckGracePeriod": 300
  },
  "TargetGroup": {
    "HealthCheckPath": "/health",
    "HealthCheckIntervalSeconds": 30,
    "HealthCheckTimeoutSeconds": 5,
    "HealthyThresholdCount": 2,
    "UnhealthyThresholdCount": 3,
    "SlowStartDurationSeconds": 60
  }
}
```

When the load balancer detects an unhealthy instance, it stops routing traffic to it and marks it as Unhealthy. The autoscaler checks the ELB health status. If an instance is Unhealthy for longer than the health check grace period, the autoscaler terminates it and launches a replacement.

### Pattern 2: Queue-Based Scaling

For worker-based architectures, scaling is driven by queue depth rather than CPU. The autoscaler monitors the SQS queue depth or Kafka consumer lag and launches workers when the backlog grows.

```json
{
  "ScalingPolicy": {
    "PolicyType": "TargetTrackingScaling",
    "TargetTrackingConfiguration": {
      "PredefinedMetricSpecification": {
        "PredefinedMetricType": "SQSQueueMessagesVisible"
      },
      "TargetValue": 100
    }
  }
}
```

The queue depth metric is a leading indicator of worker demand. Unlike CPU, which is a lagging indicator (it rises only after the worker is already processing), queue depth rises before the worker starts processing. This makes queue-based scaling more responsive for batch workloads.

The relationship between queue depth and worker count is:

```
workers_needed = ceil(queue_depth / (batch_size * processing_rate * target_utilization))
```

For a system with 10,000 messages in the queue, batch size of 10, processing rate of 60 batches per hour per worker, and target utilization of 80%:

```
workers_needed = ceil(10000 / (10 * 60 * 0.8)) = ceil(20.8) = 21 workers
```

### Pattern 3: Predictive + Reactive Hybrid

The most sophisticated integration pattern combines predictive scaling for the baseline forecast and reactive scaling for deviations. The predictive component pre-provisions capacity based on historical patterns. The reactive component adjusts for unexpected traffic variations.

The hybrid pattern works in three phases:

1. **Forecast phase**: The ML model generates a 24-hour capacity forecast every 30 minutes. The forecast is published as a set of scheduled scaling actions for the next 24 hours.

2. **Pre-provision phase**: The scheduled actions execute ahead of the predicted demand. For a forecast that predicts 60% CPU at 2:00 PM, the autoscaler scales to the required capacity at 1:50 PM -- 10 minutes before the demand arrives.

3. **Reactive adjustment phase**: The reactive target tracking policy runs alongside the scheduled actions. If actual demand exceeds the forecast, the reactive policy adds more capacity. If actual demand is below the forecast, the reactive policy removes excess capacity after the cooldown period expires.

The hybrid pattern reduces the scaling gap from 5-7 minutes (reactive only) to near zero for predictable traffic, while maintaining the ability to handle unexpected spikes.

### Pattern 4: Lifecycle Hooks for Custom Actions

Lifecycle hooks pause the instance lifecycle at specific transitions and execute custom logic. The hook puts the instance in a "wait" state until the custom action completes or the timeout expires.

```json
{
  "LifecycleHooks": [
    {
      "LifecycleHookName": "warm-up-cache",
      "LifecycleTransition": "autoscaling:EC2_INSTANCE_LAUNCHING",
      "NotificationTargetARN": "arn:aws:sns:us-east-1:123456789012:warm-up-cache",
      "RoleARN": "arn:aws:iam::123456789012:role/lifecycle-hook-role",
      "HeartbeatTimeout": 600,
      "DefaultResult": "ABANDON"
    },
    {
      "LifecycleHookName": "backup-before-terminate",
      "LifecycleTransition": "autoscaling:EC2_INSTANCE_TERMINATING",
      "NotificationTargetARN": "arn:aws:sns:us-east-1:123456789012:backup-before-terminate",
      "RoleARN": "arn:aws:iam::123456789012:role/lifecycle-hook-role",
      "HeartbeatTimeout": 300,
      "DefaultResult": "CONTINUE"
    }
  ]
}
```

Common lifecycle hook use cases:
- **Pre-warm cache**: Before the instance is added to the load balancer, pre-populate the local cache with frequently accessed data.
- **Drain connections**: Before termination, gracefully close WebSocket connections and notify connected clients.
- **Upload logs**: Before termination, upload local log files to S3 for analysis.
- **Run integration tests**: After launch but before serving traffic, run a smoke test to verify the instance is functioning correctly.

## Multi-Region and Global Scale

At global scale, each region has its own autoscaling group. The global traffic manager (Route53, CloudFront) routes traffic to the closest region. Each region's autoscaler operates independently based on local demand.

### Cross-Region Scaling Strategy

| Region | Baseline | Peak | Scaling Policy | Notes |
|--------|----------|------|----------------|-------|
| us-east-1 | 40 instances | 200 | CPU 50% + Predictive | Primary region, highest traffic |
| eu-west-1 | 20 instances | 100 | CPU 50% + Schedule | Business hours 8-18 UTC |
| ap-southeast-1 | 15 instances | 80 | CPU 50% + Schedule | Business hours 0-12 UTC |
| sa-east-1 | 5 instances | 20 | CPU 50% | Lower traffic, reactive only |

### Capacity Planning Math

**Calculating Baseline Capacity**:

```
average_requests_per_second = 50,000
requests_per_instance_per_second = 500
target_utilization = 0.5 (50% CPU target)

baseline_instances = ceil(50000 / (500 * 0.5)) = ceil(200) = 200 instances
```

**Calculating Peak Capacity**:

```
peak_requests_per_second = 150,000 (3x baseline)
requests_per_instance_per_second = 500
target_utilization = 0.5

peak_instances = ceil(150000 / (500 * 0.5)) = ceil(600) = 600 instances
```

**Calculating Buffer for Failover**:

```
failover_buffer = 0.3 (30% extra capacity for region failover)
total_instances = peak_instances * (1 + failover_buffer)
total_instances = 600 * 1.3 = 780 instances
```

**Cost Calculation**:

```
instance_type = t3.large ($0.0832/hour)
instances = 780 (peak)
hours_per_day = 4 (peak hours)
days_per_month = 30

peak_cost = 780 * 0.0832 * 4 * 30 = $7,787/month
baseline_cost = 200 * 0.0832 * 20 * 30 = $9,984/month (non-peak hours)
total = $7,787 + $9,984 = $17,771/month
```

Without autoscaling, provisioning for peak would cost:

```
always_at_peak = 780 * 0.0832 * 24 * 30 = $46,725/month
```

**Savings**: $46,725 - $17,771 = $28,954/month (62% savings)

## Deployment and Operations

### Blue/Green Deployment with Autoscaling

A blue/green deployment creates a new autoscaling group (green) alongside the existing one (blue). Traffic is shifted from blue to green after the green group passes health checks.

```yaml
# AWS CodeDeploy Blue/Green with Autoscaling
deployment:
  type: BLUE_GREEN
  autoScalingGroups:
    - web-app-asg
  terminationWaitTimeInMinutes: 5
  deploymentConfig:
    type: CANARY
    canary:
      percentage: 10
      interval: 5
  loadBalancerInfo:
    targetGroupPairInfoList:
      - targetGroups:
          - name: web-app-blue-tg
        prodTrafficRoute:
          listenerARNs:
            - arn:aws:elasticloadbalancing:...:listener/app/web-alb/abc123
        testTrafficRoute:
          listenerARNs:
            - arn:aws:elasticloadbalancing:...:listener/app/web-alb/abc456
```

During deployment, the autoscaler for the green group starts with the minimum capacity and scales up based on the Green group's own scaling policies. The Blue group remains at full capacity until traffic is fully shifted. After the shift, the Blue group is terminated.

### Canary Deployments

A canary deployment routes a small percentage of traffic (5-10%) to new instances to validate the release before full rollout. The autoscaler treats the canary instances as part of the same autoscaling group, but the load balancer target group weight is adjusted to control traffic distribution.

```yaml
# ALB canary routing
rules:
  - priority: 1
    actions:
      - type: forward
        targetGroups:
          - name: web-app-stable
            weight: 90
          - name: web-app-canary
            weight: 10
```

The canary group autoscales independently. If the canary shows elevated error rates, traffic is rolled back to the stable group. If the canary is healthy for 15 minutes, the weight is shifted to 50/50, then 100/0, and the old instances are terminated.

### Operational Runbooks

**Runbook: Handling a Scale-Out Failure**

1. Check the autoscaling activity history in the AWS console. Look for failed scaling activities.
2. Check the launch template. Verify the AMI ID exists and the instance type is available in the AZ.
3. Check service quotas. The account may have hit the EC2 instance limit for the region.
4. Check the VPC configuration. Ensure the subnets have available IP addresses.
5. Check the IAM role. The instance may not have permissions to join the target group.
6. If the issue is a quota limit, request a limit increase through the support console.
7. If the issue is a subnet IP exhaustion, add more subnets to the autoscaling group.

**Runbook: Handling a Scale-In Event That Terminates the Wrong Instance**

1. Verify the termination policy. The default is "Default" (OldestInstance), which terminates the oldest instance first.
2. Check if scale-in protection is enabled on instances that should not be terminated.
3. Check the instance's launch template version. The autoscaler may be terminating instances with an older template version.
4. If the termination policy is not optimal, change it to "NewestInstance" or "ClosestToNextInstanceHour" based on your requirements.
5. If instances are terminating mid-request, increase the connection draining timeout on the load balancer.

**Runbook: Diagnosing Autoscaling Thrashing**

1. Check the scaling activity log for alternating scale-out and scale-in events.
2. Check the cooldown period. A short cooldown period allows rapid oscillation.
3. Check the metric. If the metric oscillates around the threshold, widen the tolerance band.
4. Check for multiple conflicting policies. Remove all but one target tracking policy.
5. Check the warm-up time. If instances take too long to warm up, the autoscaler may over-provision while waiting.
6. Increase the cooldown period by 50% and observe for 24 hours.
7. If thrashing persists, switch to a step scaling policy with a wider "no-action zone."

### Autoscaling Across Availability Zones

Always distribute instances across multiple Availability Zones (AZs). The autoscaler balances instances across the specified subnets. If one AZ goes down, the instances in the other AZs continue serving traffic.

```json
{
  "VPCZoneIdentifier": "subnet-us-east-1a,subnet-us-east-1b,subnet-us-east-1c",
  "AvailabilityZoneDistribution": "balanced",
  "CapacityRebalance": true
}
```

The autoscaler maintains balance across AZs. When it launches instances, it distributes them evenly. When it terminates instances, it maintains the balance. If an AZ has fewer instances, the autoscaler launches more in that AZ during the next scale-out event.

### Scaling Limits and Safeguards

Every production autoscaling deployment should have the following safeguards:

- **Minimum size**: Never go below this number. Protects against accidental scale-in that would remove all capacity.
- **Maximum size**: Never go above this number. Protects against runaway scaling that could exhaust the budget.
- **Instance protection**: Instances with critical tasks should have scale-in protection enabled.
- **Suspension processes**: In an emergency, you can suspend specific autoscaling processes (Launch, Terminate, HealthCheck, ReplaceUnhealthy, etc.) without terminating the group.
- **Alarm-based guardrails**: If the error rate exceeds 1% across all instances, send a critical alert and consider suspending the autoscaling process.

## Autoscaling with Kubernetes

Kubernetes provides two levels of autoscaling: Horizontal Pod Autoscaler (HPA) at the pod level and Cluster Autoscaler (CA) at the node level. Understanding the interaction between these two is essential for designing a Kubernetes-based autoscaling system.

### HPA: Pod-Level Autoscaling

HPA manages the replica count of a deployment based on observed metrics. It uses the same proportional control formula as AWS target tracking:

```
desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue))
```

HPA evaluates every 15 seconds (configurable via `--horizontal-pod-autoscaler-sync-period`). The metrics are obtained from the metrics server (CPU/memory) or the custom metrics API (application-specific metrics).

### Cluster Autoscaler: Node-Level Autoscaling

Cluster Autoscaler manages the size of the node pool. It watches for pods in Pending state (unschedulable) and adds nodes when needed. It also watches for nodes with low utilization and removes them.

The key interaction between HPA and CA is:

1. Traffic increases, causing existing pods to use more CPU.
2. HPA detects the increased CPU and scales the deployment to more replicas.
3. The new replicas require more CPU and memory than the existing nodes can provide.
4. The new pods remain in Pending state because no node has enough free resources.
5. Cluster Autoscaler detects the pending pods and launches a new node.
6. The new node joins the cluster, and the pending pods are scheduled.
7. The pods start serving traffic.

The total time from traffic spike to new pods serving traffic is the sum of the HPA reaction time (15-60 seconds) plus the CA reaction time (60-120 seconds) plus the node warm-up time (2-5 minutes) plus the pod startup time (10-30 seconds). Total: 3-8 minutes.

### Vertical Pod Autoscaler (VPA)

VPA adjusts the CPU and memory requests of pods based on historical usage. It does not change the number of replicas -- it changes the resource allocation per replica. VPA is useful for applications that have variable resource requirements but cannot be easily horizontally scaled.

VPA operates in three modes:
- **Off**: VPA generates recommendations but does not apply them.
- **Auto**: VPA applies the recommendations by updating the pod's resource requests and recreating the pod.
- **Initial**: VPA applies the recommendations only at pod creation time.

### Kubernetes Autoscaling Best Practices

1. **Set resource requests and limits**: HPA cannot scale without resource requests. Every pod must have CPU and memory requests defined.
2. **Use multiple metrics**: Combine CPU utilization with custom metrics for better scaling decisions.
3. **Configure the stabilization window**: Set `--horizontal-pod-autoscaler-cpu-initialization-period` to 5 minutes to prevent HPA from scaling based on startup CPU spikes.
4. **Use pod disruption budgets**: Protect critical pods from being terminated during cluster autoscaler scale-down.
5. **Pre-fetch container images**: Use image pre-fetching on all nodes to reduce pod startup time.
6. **Use cluster proportional autoscaler**: For cluster-level components (DNS, metrics server), use the cluster proportional autoscaler that scales based on the cluster size.

## Capacity Planning with Autoscaling

Autoscaling is not a replacement for capacity planning. It is a complement. You still need to plan for the minimum capacity, the maximum capacity, and the budget.

### Minimum Capacity Planning

The minimum capacity should be the number of instances needed to handle the baseline traffic. Do not set the minimum to 1 -- you need at least 2 instances for redundancy (2 AZs minimum). The minimum should account for:

- **Baseline traffic**: The traffic level during the lowest traffic period (typically 3-4 AM).
- **Headroom for failures**: The minimum should be at least 2x the number of instances needed for the baseline. This allows one instance to fail without losing all capacity.
- **AZ distribution**: The minimum should be divisible by the number of AZs. With 3 AZs, use multiples of 3 (3, 6, 9, 12).

### Maximum Capacity Planning

The maximum capacity should be based on the budget, not the theoretical peak. The maximum is a cost control boundary, not a performance boundary. To determine the maximum:

1. Calculate the cost per instance per hour.
2. Calculate the maximum acceptable monthly spend for compute.
3. Divide the monthly budget by the hourly cost per instance and the hours in the month.

```
monthly_budget = $50,000
hourly_cost_per_instance = $0.0832 (t3.large)
hours_per_month = 730

max_instances = floor(50000 / (0.0832 * 730)) = floor(823) = 823
```

Set the maximum to 823. If the autoscaler needs more, it will be constrained by the budget. This is a deliberate design choice -- the system will degrade gracefully under extreme load rather than exceeding the budget.

### Buffer Capacity Planning

Buffer capacity is the extra capacity maintained above the target. The buffer absorbs traffic variations without triggering scaling actions. The buffer size depends on your traffic volatility and scaling latency.

```
buffer = traffic_volatility * scaling_latency / instance_capacity
```

For a system with 10% traffic volatility per minute, 6-minute scaling latency, and 500 req/s per instance:

```
buffer = 0.10 * 6 * 500 = 300 req/s worth of buffer = 1 extra instance
```

In practice, most systems maintain a buffer of 20-30% above the target. This is achieved by setting the target tracking value lower than the actual capacity threshold. For example, if you want to keep CPU at 50%, set the target tracking to 40%. The extra 10% is the buffer.

## Autoscaling with Serverless

Serverless platforms (AWS Lambda, Google Cloud Functions, Azure Functions) handle autoscaling automatically. The platform manages the compute resources, and the function scales based on the number of concurrent invocations.

### Lambda Autoscaling

AWS Lambda can handle up to 1,000 concurrent executions per region (default limit, can be increased). Each function invocation is an isolated execution environment. When traffic increases, Lambda creates more execution environments. When traffic decreases, Lambda terminates idle environments.

Lambda's autoscaling is:
- **Instant**: No warm-up time for the first 1,000 concurrent executions (within the burst limit).
- **Automatic**: No configuration needed.
- **Per-function**: Each function scales independently.

Lambda's limitations:
- **Burst limit**: The first 1,000 concurrent executions per region scale instantly. Beyond that, Lambda scales at 500 additional execution environments per minute.
- **Cold starts**: When a new execution environment is created, there is a cold start penalty (50-500ms depending on the runtime and package size).
- **Provisioned concurrency**: You can pre-warm execution environments to eliminate cold starts. This is similar to a warm pool for Lambda.

### When to Use Serverless vs. Instance-Based Autoscaling

| Factor | Serverless (Lambda) | Instance-Based (EC2) |
|--------|--------------------|---------------------|
| Scale-up speed | Instant (up to burst limit) | 3-5 minutes |
| Warm-up | 50-500ms cold start | 3-5 minutes boot |
| State | Stateless only | Stateful or stateless |
| Max execution time | 15 minutes | Unlimited |
| Cost model | Per-invocation | Per-hour |
| Configuration | None | Extensive |

## Deployment and Operations (Continued)

### Gradual Deployment with Instance Refresh

An instance refresh gradually replaces all instances in the autoscaling group with instances launched from a new launch template. This is the autoscaling-native way to perform rolling deployments.

```json
{
  "InstanceRefresh": {
    "Strategy": "Rolling",
    "Preferences": {
      "MinHealthyPercentage": 90,
      "InstanceWarmup": 300,
      "SkipMatching": true,
      "AutoRollback": true
    }
  }
}
```

The instance refresh replaces instances in batches. The `MinHealthyPercentage` parameter ensures that at least 90% of instances remain healthy during the refresh. The `InstanceWarmup` parameter gives each new instance 5 minutes to stabilize before the next batch is replaced. The `AutoRollback` parameter automatically rolls back the refresh if the new instances fail health checks.

### Autoscaling with Canary Deployments

Canary deployments with autoscaling require careful handling of the autoscaling group. The canary instances are part of the same autoscaling group but use a different launch template version. The load balancer routes a small percentage of traffic to the canary instances.

The autoscaler treats the canary instances the same as the stable instances. If the canary causes higher CPU (slower code), the autoscaler may scale out unexpectedly. To prevent this, monitor the canary instances separately and compare their metrics to the stable instances.

### Autoscaling Rollback

If a deployment causes issues, the autoscaling group can be rolled back to the previous launch template version. The instance refresh command can be used to roll back all instances:

```bash
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name web-app-production \
  --strategy Rolling \
  --preferences MinHealthyPercentage=90,InstanceWarmup=300
```

The rollback restores the previous launch template version and replaces all instances. The autoscaling history retains the previous version for 30 days.

## Autoscaling with Kubernetes (Continued)

### HPA Configuration Patterns

**Pattern 1: Standard HPA with CPU and custom metrics**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  minReplicas: 5
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: 500
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
      - type: Pods
        value: 5
        periodSeconds: 15
      selectPolicy: Max
```

This configuration scales up aggressively (100% increase or 5 pods per 15 seconds, whichever is greater) and scales down conservatively (10% per 60 seconds with 5-minute stabilization window).

**Pattern 2: HPA with external metrics (queue depth)**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: worker-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: worker
  minReplicas: 1
  maxReplicas: 100
  metrics:
  - type: External
    external:
      metric:
        name: sqs_approximate_number_of_messages_visible
        selector:
          matchLabels:
            queue: transcoding-queue
      target:
        type: AverageValue
        averageValue: 100
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 600
```

This HPA scales workers based on SQS queue depth. The scale-down stabilization window is 10 minutes to prevent rapid termination of workers.

**Pattern 3: HPA with multiple metrics and priority**

When multiple metrics are configured, the HPA calculates the desired replicas for each metric independently and uses the largest result. This ensures that the most demanding metric drives the scaling decision.

For example, if CPU suggests 10 replicas and request count suggests 20 replicas, the HPA uses 20. The rationale is that it is better to have too many replicas (cost) than too few (performance).

### Cluster Autoscaler Configuration

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-autoscaler-config
  namespace: kube-system
data:
  config.yaml: |
    nodeGroups:
    - name: worker-pool
      minSize: 3
      maxSize: 50
      instanceType: t3.large
    scaleDown:
      enabled: true
      delayAfterAdd: 10m
      delayAfterDelete: 10m
      delayAfterFailure: 10m
      unneededTime: 10m
      utilizationThreshold: 0.5
    maxNodeProvisionTime: 15m
    balancingSimilarNodeGroups: true
```

Key parameters:
- `delayAfterAdd`: Wait 10 minutes after adding a node before considering scale-down.
- `unneededTime`: A node must be underutilized for 10 minutes before it is considered for scale-down.
- `utilizationThreshold`: Nodes below 50% utilization are candidates for scale-down.
- `maxNodeProvisionTime`: If a node takes more than 15 minutes to provision, the CA cancels the request.

### KEDA: Event-Driven Autoscaling

KEDA extends HPA with 50+ event source integrations. It is particularly useful for workloads that need to scale based on external events.

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: kafka-consumer
spec:
  scaleTargetRef:
    name: kafka-consumer-deployment
  pollingInterval: 15
  cooldownPeriod: 300
  minReplicaCount: 1
  maxReplicaCount: 50
  triggers:
  - type: kafka
    metadata:
      bootstrapServers: kafka:9092
      consumerGroup: my-group
      topic: my-topic
      lagThreshold: "100"
      offsetResetPolicy: latest
  - type: cpu
    metadata:
      type: Utilization
      value: "60"
```

KEDA monitors Kafka consumer lag and CPU utilization. When the lag exceeds 100 messages, it scales out. When the lag drops below 100 for 300 seconds (cooldown), it scales in. This combination provides responsive scaling for event-driven workloads with protection against thrashing.

## Cost Optimization with Mixed Instances

The mixed instances policy is the primary mechanism for cost optimization in autoscaling. It allows you to specify a mix of on-demand and spot instances across multiple instance types.

### On-Demand vs. Spot Pricing

| Instance Type | On-Demand (per hour) | Spot (per hour) | Savings |
|--------------|---------------------|-----------------|---------|
| t3.large | $0.0832 | $0.0250 | 70% |
| t3.xlarge | $0.1664 | $0.0499 | 70% |
| m5.large | $0.0960 | $0.0288 | 70% |
| c5.large | $0.0850 | $0.0255 | 70% |

### Allocation Strategies

**lowest-price**: The autoscaler launches instances from the cheapest instance type first. This maximizes cost savings but may increase reclaimation risk if the cheapest type is also the most popular.

**capacity-optimized**: The autoscaler selects instance types that are most likely to have available capacity. This minimizes reclaimation risk but may use more expensive instance types.

**capacity-optimized-prioritized**: The autoscaler follows your prioritized list but skips instance types that are likely to be reclaimed. This balances cost and availability.

**Strategy recommendation**: For production workloads, use "capacity-optimized-prioritized" for spot instances. This reduces reclaimation events by 60% compared to "lowest-price" while maintaining 50-60% cost savings.
