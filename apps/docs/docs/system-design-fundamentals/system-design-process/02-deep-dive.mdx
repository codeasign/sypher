## Internal Architecture of Each Phase

The six-step process is the surface. Underneath each step lies a set of sub-steps, decisions, and concrete outputs. Understanding the internal architecture of each phase is what separates an engineer who follows a script from an engineer who adapts the process to the problem.

### Phase 1: Requirements Clarification — Internals

The requirements phase is a question-asking machine. You have a prompt (for example, "Design a video streaming platform") and you need to transform it into a set of structured requirements. There are three categories of questions you must ask.

**Category A — Scope questions:** What is in scope and what is out of scope? For a video platform: "Do we support live streaming or only on-demand? Do users upload or only view? Do we need comments and ratings?"

**Category B — Scale questions:** How many users, how much data, how much traffic? "How many daily active users? How many hours of video uploaded per minute? What is the average video duration and file size? What are the peak traffic patterns?"

**Category C — Quality questions:** What are the performance and reliability targets? "What is the p99 latency target for video playback? What availability do we need? What is the target region — US, EU, or global?"

The output of the requirements phase is a structured document: functional requirements (bullet list), non-functional requirements (numbered targets), and scale estimates (order-of-magnitude numbers).

### Phase 2: Capacity Estimation — Internals

Capacity estimation is the phase where most candidates stumble. The math is simple arithmetic. The challenge is knowing which numbers to calculate and which formulas to apply.

**The four capacity buckets:**

| Bucket | Formula | Example (video platform) |
|--------|---------|-------------------------|
| Storage | daily_uploads x avg_size x retention_days | 500 hours/min x 60 min x 500 MB/hr = 15 TB/min storage |
| Bandwidth | peak_viewers x bitrate | 50M viewers x 5 Mbps = 250 Tbps |
| QPS — Read | daily_views / 86400 x peak_factor | 5B views/day / 86400 x 3 = 173,000 QPS |
| QPS — Write | daily_uploads / 86400 | 720,000 uploads/day / 86400 = 8 writes/sec |

**Rules of thumb for single-node capacity:**

| Resource | Single node capacity |
|----------|---------------------|
| PostgreSQL (primary) | 10,000 QPS reads, 5,000 QPS writes |
| PostgreSQL (read replica) | 15,000 QPS reads |
| Redis (single instance) | 100,000 ops/sec |
| Application server | 5,000 concurrent connections, 10,000 RPS |
| Network bandwidth | 10 Gbps per server |
| Disk I/O (SSD) | 100,000 IOPS, 2 GB/s sequential |

**Estimation is not precision.** The goal is order-of-magnitude accuracy. If your estimate says 10 servers and the real number is 12, you win. If your estimate says 10 servers and the real number is 500, you missed something fundamental.

### Phase 3: High-Level Design — Internals

The high-level design phase is a box-drawing exercise — but only after you have done the thinking. The question you answer in this phase is: "Given the requirements and capacity numbers, what are the major components and how do they connect?"

**The minimum viable design process:**

1. Identify the entities (user, video, comment, playlist)
2. Map each entity to a storage system
3. Identify the operations on each entity (CRUD + search)
4. Draw the read path and write path for the primary operation
5. Add supporting infrastructure (load balancer, cache, queue)
6. Add cross-cutting concerns (auth, monitoring, CDN)

**Common component decisions:**

| Decision | Options | When |
|----------|---------|------|
| Load balancer | DNS round-robin, ALB/NGINX, HAProxy | Always needed for multi-server setup |
| Cache | Redis, Memcached, CDN | Read-heavy workloads, under 5ms access needed |
| Queue | Kafka, RabbitMQ, SQS | Async processing, decoupling, buffering |
| Database | PostgreSQL, Cassandra, MongoDB, DynamoDB | See storage decision matrix below |
| File storage | S3, GCS, HDFS, EBS | Blobs, videos, images |
| Search | Elasticsearch, Algolia, Meilisearch | Full-text search, filtering |

**Storage decision matrix:**

| Requirement | PostgreSQL | Cassandra | MongoDB | DynamoDB |
|-------------|-----------|-----------|---------|----------|
| Relational data (joins) | Best | No | Limited | No |
| High writes (100K+/sec) | Needs sharding | Native | Needs sharding | Native |
| Multi-region writes | Complex | Native | Complex | Native |
| Strong consistency | Native | Tunable | Tunable | Optional (costly) |
| Complex queries | Best | Limited | Good | Limited |
| Schema flexibility | Rigid | Flexible | Flexible | Flexible |

### Phase 4: Deep Dive — Internals

The deep dive is where you turn a box-and-arrow diagram into a real design. Each box on your high-level diagram needs its own sub-design. You do not design every box in a single interview — you ask the interviewer which box they want you to open.

**Deep dive checklist per component:**

For a database: schema (CREATE TABLE statements), indexes, partitioning strategy, replication factor, consistency level, connection pool size, backup strategy.

For a cache: data structure (string, hash, sorted set), key naming convention, TTL policy, eviction policy, memory budget, cache invalidation strategy.

For a queue: topic/queue name, partition count, replication factor, message format, retention period, consumer group configuration, dead-letter queue setup.

For an API: endpoints with request/response schemas, authentication method, rate limits, pagination strategy, error codes, versioning strategy.

### Phase 5: Trade-Off Analysis — Internals

Trade-off analysis is where you demonstrate that you understand what you are giving up. Every design decision has a cost. The trade-off analysis makes those costs visible.

**The trade-off matrix structure:**

| Decision | Option A | Option B | Winner | Why |
|----------|----------|----------|--------|-----|
| Database | PostgreSQL | Cassandra | PostgreSQL | Joins needed, write volume moderate |
| Cache | Redis | Memcached | Redis | Data structures + persistence |
| Queue | Kafka | RabbitMQ | Kafka | Throughput + replay needed |
| Storage | S3 | HDFS | S3 | Global access + no ops needed |

**Failure mode analysis during trade-offs:**

For each decision, ask: "What happens when this component fails?" Trace the failure and determine if the system survives.

- What happens when Redis goes down? → Cache miss rate spikes to 100%. Database handles the load (already sized for this). Degraded but working.
- What happens when PostgreSQL primary fails? → Failover to replica. 30-60 seconds of read-only mode. Writes queue in application buffer. After failover, writes resume.
- What happens when Kafka broker goes down? → Producers retry with backoff. Consumers rebalance to remaining brokers. Data is preserved on disk.

### Phase 6: Wrap — Internals

The wrap phase has three parts: summary, scaling discussion, and next steps. The summary restates the design in 30 seconds. The scaling discussion shows you have thought about the future. The next steps show you know what you would do with more time.

**Wrap structure:**

"To summarize, we designed a video streaming platform serving 500 million DAU. The architecture uses a CDN for video delivery, a microservices API layer for metadata, PostgreSQL for user and metadata storage, and Kafka for asynchronous processing of uploads and analytics. We deep-dived into the upload pipeline and the video transcoding queue. The key trade-offs were choosing PostgreSQL over Cassandra for metadata consistency, and using a CDN to offload 95% of bandwidth. For the future, we would add a recommendation service, implement geo-aware content routing, and explore WebRTC for live streaming. With more time, I would deep dive into the recommendation engine and the live streaming architecture."

## Data Structures — The Mental Toolkit

The system design process uses several mental data structures that you construct and manipulate during the interview.

### The Requirements Checklist

A mental checklist you run through for every design:

```
[ ] Functional requirements: 3-5 bullet points
[ ] Non-functional requirements: latency, availability, durability
[ ] Scale: DAU, reads/day, writes/day, storage growth rate
[ ] Out of scope: what are we NOT designing
```

### The Estimation Formulas

Memorize these formulas. They come up in every single system design interview.

```
Writes per second = daily_writes / 86,400
Reads per second = daily_reads / 86,400
Storage per day = daily_writes x avg_record_size
Peak QPS = average_QPS x peak_factor (usually 2-3x)
Cache memory = hot_set_size x avg_entry_size x 1.5 (overhead)
Number of servers = total_QPS / QPS_per_server
```

### The Trade-Off Matrix

A 2-column table you build in your head (or on paper) for each decision:

```
Decision: Database
  Option A: PostgreSQL — mature, ACID, joins, 10K writes/sec
  Option B: Cassandra — linear scaling, no SPOF, multi-region, 1M writes/sec
  When to use A: moderate writes, relational data, team knows SQL
  When to use B: high writes, multi-region, flexible schema
  Selected: PostgreSQL (writes are moderate, schema is relational)
```

## Lifecycle — Applying the Process End-to-End

Here is the full lifecycle of applying the system design process to a chat system. Each stage shows the state of the design at that point.

### Step 1: Clarify

Prompt: "Design a WhatsApp-like messaging system."

Questions asked:
- "One-on-one chat, group chat, or both?" → Both.
- "Do messages persist forever?" → Yes, unlimited history.
- "How many DAU?" → 1 billion.
- "Average messages per user per day?" → 50 sent, 100 received.
- "Do we need read receipts, typing indicators, delivery receipts?" → Yes.
- "Do we support media messages (images, video)?" → Yes, up to 100 MB.
- "Latency target for message delivery?" → Under 100ms p99.
- "Availability target?" → 99.999% (for a messaging app, downtime means missed messages).

Output:
- Functional: send message (text + media), receive message, group chat (up to 256 users), read receipts, typing indicators, message history.
- Non-functional: p99 delivery under 100ms, 99.999% availability, unlimited history, media up to 100 MB.
- Scale: 1B DAU, 50B messages sent/day, 100B received/day.

### Step 2: Estimate

Storage: 50 billion messages per day. Each message: 100 bytes of metadata (sender, receiver, timestamp, ID) + 1,000 bytes average text = ~1.1 KB. Storage per day: 50B x 1.1 KB = 55 TB. Per year: 20 PB. With 3x replication: 60 PB.

Media storage is larger. 10% of messages contain media. Average media size: 2 MB (mix of images, short videos, voice messages). 5 billion media messages per day x 2 MB = 10 PB per day. With compression (re-encode to 500 KB average): 2.5 PB per day. This requires a blob storage system (S3 or equivalent), not a database.

QPS: 50 billion messages sent / 86,400 = 578,000 writes per second. 100 billion received / 86,400 = 1.16 million reads per second. Peak factor of 3x during business hours.

### Step 4: Deep Dive — Message Delivery

The critical deep dive for a chat system is message delivery — how does a message from Alice in New York arrive at Bob in Tokyo in under 100ms?

**Message delivery flow:**

1. Alice's app sends POST /messages to the load balancer. Request body: `{"receiver_id": "Bob123", "text": "Hello!", "message_type": "text"}`.
2. Load balancer forwards to a chat service instance.
3. Chat service writes the message to Kafka (for persistence via Cassandra consumer) and to the in-memory delivery buffer.
4. Chat service checks Redis for Bob's connection info (which chat service instance Bob is connected to via WebSocket).
5. Chat service forwards the message to the chat service instance serving Bob's WebSocket connection.
6. That instance pushes the message to Bob's app over the persistent WebSocket connection.
7. Bob's app sends a delivery receipt. The receipt flows back through the same path.
8. If Bob is offline, the message is stored in Cassandra. When Bob connects, a "fetch missed messages" request pulls the backlog.

**Why Kafka?** Cassandra handles 1M+ writes per second but has 1-10ms latency per write. The chat service cannot wait for Cassandra to confirm the write before delivering — that adds unacceptable latency. Kafka acts as a write buffer: the chat service writes to Kafka in under 5ms, Kafka acknowledges, and the chat service immediately starts the delivery flow. A separate consumer reads from Kafka and writes to Cassandra for persistence and history retrieval.

**Why Redis for session storage?** The chat service is stateless except for WebSocket connections. When Alice connects, she connects to any available chat service instance. That instance records "Alice is connected here" in Redis. When Bob's message arrives, any chat service instance can look up Alice's location in Redis and forward the message. This allows any instance to handle any user — critical for horizontal scaling when instances crash.

### Step 5: Trade-Offs

**Trade-off 1: Cassandra vs PostgreSQL for message storage.**

PostgreSQL struggles with 578,000 writes per second. Even with sharding and connection pooling, PostgreSQL's write throughput tops out around 50,000 writes per second on a single node. You would need 12+ primaries with application-level sharding — complex to operate.

Cassandra handles 1M+ writes per second on a 12-node cluster. It offers no single point of failure, linear scalability, and multi-region replication built in. Cassandra cannot do joins, but message queries are simple point queries: "get all messages for conversation X" or "get all messages for user Y."

Decision: Cassandra. The write volume requires a write-optimized database. The trade-off is giving up joins and ACID transactions — neither of which message storage needs.

**Trade-off 2: WebSocket vs long-polling vs Server-Sent Events.**

WebSocket: persistent bidirectional connection. Lowest latency (under 10ms for server push). Requires sticky sessions on the load balancer or external session store. Most complex to scale (connection limits, reconnection logic).

Long-polling: client sends request, server holds connection until data is available. Works everywhere (no protocol negotiation). Higher latency (average one RTT for each message). Higher server load (request overhead per poll).

Server-Sent Events: unidirectional push only (server to client). Cannot send from client to server. Simpler than WebSocket but only half the solution.

Decision: WebSocket with Redis-backed session store. The latency requirement (under 100ms) and bidirectional nature of messaging rule out the alternatives.

### Step 6: Wrap

Summary: Chat system for 1 billion DAU with 578,000 writes per second and under 100ms p99 delivery latency. Architecture uses 200 stateless chat service instances behind a load balancer, Redis for session state (WebSocket connection mapping), Kafka for write buffering, Cassandra for persistent message storage, and S3 for media files. Key trade-offs: Cassandra over PostgreSQL for write throughput, WebSockets over polling for low-latency delivery, Kafka buffer for decoupling delivery from persistence.

Scaling roadmap: 2x growth in year 2 → add more chat service instances (horizontal), add Cassandra nodes (linear scaling). 10x growth → multi-region deployment with local Kafka clusters and cross-region replication, geo-aware session routing. 100x growth → evaluate custom WebSocket gateway for connection management, consider erasure coding for storage efficiency.

### Getting Stuck — Recovery Strategies

Even with the process, you can get stuck. Here is how to recover from each stuck point.

**"I do not know what scale to assume."** The interviewer is testing your ability to make reasonable assumptions. Say: "I will assume 10 million DAU for this design. If that assumption is wrong, please adjust me." Then proceed. A reasonable assumption is better than no assumption.

**"I cannot estimate the storage because I do not know the record size."** Estimate based on comparable systems. A tweet is ~200 bytes. A Facebook post is ~500 bytes. A message is ~1 KB. If you are wrong by a factor of 2, the design still works (you oversized or undersized by 2x, which does not change the architecture).

**"I am spending too long on one deep dive and running out of time."** The interviewer guided you into a deep dive that is too detailed. Say: "I see we are spending a lot of time on the database schema. Let me summarize the key decisions and move to the next component. We chose VARCHAR(10) for the short key with a primary key index, and partitioned click events monthly. Where would you like me to go next?"

## Configuration — Adapting the Process

The process is not fixed. You configure it for the context.

### Adapting for Company Type

**FAANG (Amazon, Google, Meta, Microsoft, Apple):** Expect deep dives into a single component. Google asks about data models and consistency. Amazon asks about scalability and cost. Meta asks about real-time interaction and state management. Microsoft asks about enterprise constraints (security, compliance). Emphasize trade-off analysis and defend your choices with data.

**High-growth startups (Stripe, Airbnb, Uber, Databricks):** Expect product-driven questions with ambiguous scope. You need to clarify aggressively — the interviewer may not know the requirements themselves. Emphasize pragmatic trade-offs: "We chose PostgreSQL because we do not need Cassandra's throughput yet. We will migrate when we reach 10x our current scale."

**Traditional enterprises (banks, healthcare, government):** Expect questions about compliance, security, and reliability over performance. Emphasize redundancy, disaster recovery, audit trails, and regulatory requirements. The process is the same, but the trade-off analysis weights security and compliance above cost and latency.

### Adapting for Role Level

**L3 / L4 (Entry-level):** The interviewer wants to see that you have a process and can follow it. Do not skip steps. Show you can ask questions and do basic estimation. Keep the deep dive shallow but correct.

**L5 (Senior):** The interviewer wants to see that you can make trade-off decisions independently. Your deep dive should demonstrate depth. Your trade-off analysis should show you considered alternatives. 45-minute interviews expect 2-3 deep dives.

**L6 (Staff):** The interviewer wants to see that your process accounts for ambiguity and organizational constraints. Your design should discuss cross-team dependencies, migration strategies, and operational complexity. Show you can design systems that work within organizational realities.

**L7 (Principal):** The interviewer wants to see that you can define the process for others. Your design should identify which decisions are reversible and which are not. Discuss how you would communicate the design to multiple teams. Show you think about the system over a multi-year horizon.

## Common Mistakes During Estimation

Even experienced engineers make specific, predictable mistakes during capacity estimation. Here are the most common ones and how to avoid them.

**Mistake 1: Forgetting the peak factor.** You calculate average QPS as 10,000 (daily traffic / 86,400) and size your servers for 10,000 QPS. At peak hour, traffic is 3x the average — 30,000 QPS. Your servers handle 12,000 QPS. Everything falls over at 10:00 AM. The fix: always multiply average QPS by a peak factor of 2-3x.

**Mistake 2: Ignoring headroom.** You calculate that 8 servers handle the load. You provision 8 servers. One server fails. You now have 7 servers handling 8 servers worth of traffic. Latency spikes. More servers fail from the overload. The fix: add 20-30% headroom. If you need 8 servers, provision 10.

**Mistake 3: Estimating storage for one year instead of five.** Your system stores 23 GB per day. One year = 8.4 TB. That fits on a single server with 12 TB of disk. You sign a 3-year contract for a single-server setup. In year 3, you have 25 TB of data and the 12 TB disk is full. You must migrate to a distributed database while the system is live — a painful, high-risk operation. The fix: always estimate for 5 years, not 1.

**Mistake 4: Confusing bits and bytes.** Network bandwidth is measured in bits per second (Gbps). Storage is measured in bytes (GB, TB). A 10 Gbps network link transfers 1.25 GB per second. Engineers who confuse these consistently undersize their network by 8x.

**Mistake 5: Not accounting for replication.** You estimate 42 TB of storage and buy 42 TB of disk. You enable 3x replication for fault tolerance. You now have 42 TB of usable storage on 126 TB of raw disk — except you only bought 42 TB. The fix: multiply storage by the replication factor before buying hardware.

**Mistake 6: Forgetting that indexes take space.** Your 10-column table with 100 million rows is 50 GB. You add 5 indexes for query performance. Each index adds 10-30 GB. Your 50 GB table becomes 150 GB with indexes. The fix: estimate index size at 50-100% of table size for a write-optimized schema, 100-200% for a read-optimized schema.

**Mistake 7: Assuming linear scaling.** You estimate that 1 server handles 10,000 QPS, so 10 servers handle 100,000 QPS. In reality, 10 servers handle 70,000-80,000 QPS because of coordination overhead, network contention, and the thundering herd problem. The fix: assume 70-80% linear scaling efficiency for distributed systems.

## The Deep-Dive Workflow — A Practical Sequence

When you sit down to design a system in production (not an interview), the deep dive follows a specific sequence that maximizes learning velocity.

**Day 1-2: Understand the existing system.** If this is a brownfield design, spend the first two days reading the existing codebase, running queries against the production database, and looking at monitoring dashboards. You cannot design a better system until you understand the current one.

**Day 3: Write the requirements document.** One page. Functional requirements as bullet points. Non-functional requirements as numbered targets. Mark which requirements are hard constraints (legal, regulatory) and which are soft (nice to have).

**Day 4-5: Prototype the data model.** Write the CREATE TABLE statements. Run EXPLAIN ANALYZE on the queries. Adjust indexes based on the query plan. If the data model is wrong, the architecture is wrong. Fix the data model before drawing any boxes.

**Day 6-7: Draw the component diagram and get feedback.** Draw it on a whiteboard. Show it to two senior engineers. Listen for the question "what happens when X fails?" — if you cannot answer, you have not designed for failure.

**Day 8-10: Write the design document.** The document should be structured exactly like a Google design doc: context, goals, non-goals, proposed design, alternatives, rollout plan. The act of writing forces you to think through each step of the process.

**Day 11-12: Implementation plan and tickets.** Break the design into implementable chunks. Each chunk should produce value independently. Label each ticket with its ordering dependency — "this must be deployed before that."
