# API Security at Scale

## From Naive to Production

API security is a progression. You start with the simplest possible protection, discover it breaks under pressure, and iterate toward a layered defense that scales across thousands of requests per second without compromising on latency or developer velocity.

### Naive: Basic Auth on Public Endpoints

The simplest approach: every API endpoint is publicly accessible and protected by a single shared secret sent as an HTTP header.

**What breaks at this scale:**

At 10 requests per minute this works. At 10,000 requests per second, every flaw becomes a crisis.

```
Failure mode 1: Shared secret leaked
  - Secret checked into git by accident
  - Secret sent in cleartext over HTTP (no TLS)
  - Every client knows the same secret
  - Revoking the secret means updating every client
  - Impact: total compromise, no way to rotate

Failure mode 2: No rate limiting
  - Attacker sends 1,000,000 requests in 10 seconds
  - Database connection pool exhausts at 100 connections
  - All legitimate users see 503 Service Unavailable
  - Cost: $500/hour in cloud resources for the attacker
  - Cost: $50,000/hour in lost revenue for you

Failure mode 3: No authentication granularity
  - Every client has the same permissions
  - A read-only data pipeline client can DELETE records
  - A mobile app can access admin endpoints
  - No way to revoke a single compromised client
  - No way to audit who did what

Failure mode 4: No TLS
  - Credentials sent in plaintext
  - Anyone on the same network captures the secret
  - Public Wi-Fi is a permanent security risk
  - Regulatory compliance: GDPR, PCI-DSS, HIPAA all violated

Failure mode 5: No input validation
  - SQL injection: `' OR 1=1 --` in the username field
  - Attacker dumps the entire users table
  - No parameterized queries, no query sanitization
  - No request body size limits
  - No content-type validation
```

### Problems at 10,000 Requests Per Second

Let's quantify the failure modes at production scale.

```
Metric                              Naive Value          Production Requirement
─────────────────────────────────────────────────────────────────────────────
Requests per second                 10,000               10,000+
Authentication latency              5ms (secret check)   2ms (token validation)
Rate limit granularity              None                 Per-user, per-IP, per-endpoint
TLS handshake overhead              None                 50ms initial, 0ms resumed
Secret rotation capability          0/min (manual)       10,000+/min (automated)
Audit log volume                    0 entries            10,000+ entries/second
Burst tolerance                     0 (instant crash)    5x sustained for 30 seconds
Compromised credential detection    Manual review        500ms automated
```

A naive auth check on a shared secret is not an architecture. It is a placeholder. You need a system.

### How the Layers Work Together

Each layer handles a specific security concern. The layers are ordered so that the cheapest checks happen first, and the most expensive checks happen last.

```
Request flow and the cost of each check:

Layer 1: CDN Edge (WAF)
  Cost: 0.1ms per check
  Blocks: SQL injection, XSS, known bad IPs, DDoS
  Pass-through rate: 95% of requests pass

Layer 2: API Gateway
  Cost: 1ms per check (TLS + validation + rate limit)
  Blocks: Malformed requests, rate limit violations, CORS violations
  Pass-through rate: 90% of WAF-passed requests pass

Layer 3: Auth Service
  Cost: 5ms per validation (plus 0.01ms if cached)
  Blocks: Invalid tokens, expired tokens, wrong scopes
  Pass-through rate: 85% of gateway-passed requests pass

Layer 4: Backend Service
  Cost: 10-100ms (business logic)
  Blocks: Authorization failures, invalid data
  Pass-through rate: 95% of auth-passed requests pass

Overall pass-through rate: 0.95 * 0.90 * 0.85 * 0.95 = 69%
Overall overhead: 0.1 + 1 + 5 + 10 = 16.1ms per request
```

The key insight: you filter aggressively at the edge. A DDoS attack with 1,000,000 RPS hits the CDN WAF first, which drops 99% of it for 0.1ms each. Only 10,000 RPS ever reaches the API Gateway. Only 1,000 RPS reaches the auth service. Only 150 RPS reaches your backend services.

## Integration with Other Systems

API security does not exist in isolation. You must integrate it with your CDN, your internal service mesh, your authentication provider, and your audit logging infrastructure.

**Key configuration for API security at the CDN level:**

```
WAF Rule Priority (highest priority first):

Priority 1: IP Block List
  - Known attacker IPs from threat intelligence feeds
  - Tor exit nodes (if your API is not anonymous)
  - Known VPN/proxy IPs (if your API is geo-restricted)
  - Action: BLOCK, no further processing

Priority 2: Rate Limiting
  - 100 requests per second per IP
  - 1,000 requests per second per authenticated user
  - 10,000 requests per second per API key
  - Action: 429, include Retry-After header

Priority 3: Signature-Based WAF
  - Managed rulesets (AWS WAF, Cloudflare OWASP, Fastly Signal Sciences)
  - Custom rules for your API's specific attack surface
  - Action: BLOCK with 403, log rule ID

Priority 4: Behavioral Detection
  - Anomaly scoring (0-1000): aggregate of suspicious signals
  - Signals: unusual User-Agent, abnormal request rate, unexpected geo
  - Action: CAPTCHA challenge at score 500+, BLOCK at 800+

Priority 5: Allow List
  - Your own IP ranges (internal tools, CI/CD pipelines)
  - Trusted partner API keys (whitelisted)
  - Action: ALLOW, skip remaining rules
```

**Why mTLS matters for API security:**

```
Without mTLS:
  - Any compromised service can impersonate any other service
  - An attacker who compromises Service A can call Service B as if it were the gateway
  - No cryptographic proof of service identity
  - Network segmentation is the only defense (and it is porous)

With mTLS:
  - Every service has a unique cryptographic identity (SPIFFE ID)
  - Every call requires mutual certificate verification
  - A compromised Service A cannot impersonate Service B
  - Compromised certificates are revoked via the mesh CA
  - Certificate rotation is automated and transparent to applications
  - Zero-trust networking: no implicit trust based on network location
```

### Rate Limit Coordination Across Regions

Without cross-region coordination, a user can bypass per-region rate limits by rotating through regions.

```
Problem:
  User A sends 100 requests/second to us-east-1
  Per-region limit: 100 requests/second
  User A is allowed through

  User A sends 100 requests/second to eu-west-1
  Per-region limit: 100 requests/second (independent counter)
  User A is allowed through

  User A sends 100 requests/second to ap-southeast-1
  Per-region limit: 100 requests/second (independent counter)
  User A is allowed through

  Total: 300 requests/second from one user, bypassing all limits

Solution: CRDT-based global rate limit counters:

  Each region maintains a local counter for each user.
  Counters are synchronized via CRDTs (Conflict-Free Replicated Data Types).
  CRDTs ensure eventual consistency without coordination overhead.

  Region us-east-1:
    counter: user_A = 85 (local)
    counter: global = 85 + 72 + 91 = 248 (from all regions)
    Decision: ALLOW (248 < 300 global limit)

  Region eu-west-1:
    counter: user_A = 72 (local)
    counter: global = 85 + 72 + 91 = 248 (from all regions)
    Decision: ALLOW

  Region ap-southeast-1:
    counter: user_A = 91 (local)
    counter: global = 85 + 72 + 91 = 248 (from all regions)
    Decision: DENY (reached 300, last request exceeds limit)

  Synchronization interval: 1 second (bounded staleness)
  Max burst: 300 + 3 (one per region) = 303 requests (controlled overshoot)
  This is acceptable: a user cannot sustain more than 300 RPS across regions
```

### Token Revocation Propagation

Token revocation must be fast and consistent across all regions.

```
Revocation propagation time budget:

  User reports stolen device via app
  ──> Auth service marks token as revoked (0ms)
  ──> Local cache updated (1ms)
  ──> Pub/sub message published to cross-region topic (50ms)
  ──> Message consumed by all regions (100ms)
  ──> Each region updates local revoked-token cache (1ms)
  ──> Total: ~150ms to revoke a token globally

  Worst case: 1 second (if a region's subscriber is lagging)
  Acceptable: a stolen token is valid for at most 1 second after reporting

  Without cross-region propagation:
    - Token is revoked in us-east-1 immediately
    - Token remains valid in eu-west-1 for 15 minutes (until JWT expiry)
    - Attacker can continue using the token in other regions
    - 15 minutes of potential damage is unacceptable
```

## From Architecture to Implementation

You now have a production-ready API security architecture. The key principles:

1. **Defense in depth**: Every layer independently validates and filters. A flaw in one layer does not compromise the whole system.

2. **Cheap checks first**: The WAF at the edge costs 0.1ms and drops 95% of bad traffic. The backend does expensive business logic only for clean requests.

3. **Stateless validation at the edge**: JWT tokens are self-contained. The CDN and API Gateway can validate them without calling the auth service.

4. **Stateful coordination where needed**: Rate limits and token revocation require cross-region synchronization. Use CRDTs for rate limits and pub/sub for revocation.

5. **Audit everything**: Every request leaves a trace. You can investigate any security incident retroactively and satisfy compliance requirements.

6. **Plan capacity at every layer**: WAF throughput, rate limiter memory, and log volume all grow with traffic. Size each component independently.

This architecture scales from 1,000 RPS to 1,000,000 RPS. The cost grows linearly with traffic, but the security posture does not degrade.

### Bridge to Trade-offs

Every architectural decision in this system involves a trade-off. Managed services reduce operational burden but increase cost per request. Self-hosted solutions give you full control but require operational expertise. The right choice depends on your scale, your team's skills, and your compliance requirements.
