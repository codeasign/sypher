# API Security — Trade-offs & Failure Modes

## The Core Tension

Security and latency are natural antagonists. Every security check you add — authentication, authorization, input sanitization, rate limiting, WAF scanning, payload encryption — sits in the critical path of every request. Each check adds milliseconds. Stack enough of them and your snappy API turns sluggish.

The tension operates on three axes:

**Security vs Latency**: A single JWT verification takes 1-5ms. WAF inspection adds 5-20ms. Rate limiting checks add 0.5-2ms per lookup. Input validation adds 1-10ms depending on payload size. Encryption and decryption for sensitive fields adds 2-15ms. A fully-locked-down API endpoint can carry 30-50ms of pure security overhead before your business logic even starts.

**Security vs Developer Experience**: Every lock you turn makes the API harder to use. Developers need to rotate keys, manage tokens, pass through multi-factor flows, and handle signature errors. The safer you make the system, the more friction developers feel. Too much friction and they will find ways around it — storing tokens in plaintext, disabling certificate validation in dev, or sharing API keys across teams.

**Security vs Cost**: WAF rules, rate limiting state storage, token introspection endpoints, audit log systems, and certificate management all cost money. A cloud WAF (AWS WAF, Cloudflare) costs $5-20 per rule per month plus data transfer fees. Token introspection against a database costs CPU and connection pool capacity. Audit logging at scale requires dedicated storage and retention infrastructure.

### The Performance Stack

Each security layer has a different latency profile. You need to understand where the cost is highest.

```
Layer                  Latency Cost    Mitigation Strategy
─────────────────────────────────────────────────────────────
TLS handshake          5-20ms          Session resumption, TLS 1.3
JWT verification       1-5ms           Cache public keys, use ED25519
Rate limiting lookup   0.5-2ms         In-memory counters, Redis pipeline
WAF inspection         5-20ms          Tune rules, use positive security model
Input validation       1-10ms          Reject early, validate schema once
Payload encryption     2-15ms          Target only sensitive fields
Audit logging          1-3ms           Async writes, never block response
Authorization (RBAC)   2-10ms          Precompute role->permission maps
CSRF token validation  0.5-2ms         Double-submit cookie (stateless)
```

## Decision Matrix

Choosing the right security posture depends on what you are protecting. Here is a decision matrix for common API types.

| API Type | Auth Mechanism | Security Layers | Acceptable Overhead | Critical Risks |
|---|---|---|---|---|
| Public read-only (catalog, docs) | API key (low security) | TLS, rate limit, IP allowlist | 5-15ms | Data scraping, DDoS |
| Auth endpoint (login, register) | Username + password + MFA | TLS, rate limit (strict), WAF, account lockout | 20-50ms | Brute force, credential stuffing, account takeover |
| Payment / financial API | JWT + HMAC signing | TLS, mutual TLS, WAF, payload encryption, audit, rate limit | 50-200ms | Data breach, fraud, PCI compliance |
| Admin / internal dashboard | JWT + RBAC + IP allowlist | TLS, MFA, audit logging, rate limit | 20-80ms | Privilege escalation, insider threat |
| Webhook receiver | HMAC signature verification | TLS, IP allowlist, signature check, deduplication | 10-30ms | Replay attacks, unauthorized callbacks |
| Internal microservice API | mTLS + service account | mTLS, authorization token, rate limit (loose) | 5-15ms | Lateral movement, service impersonation |
| GraphQL endpoint | JWT + field-level auth | TLS, depth limiting, query cost analysis, WAF | 30-100ms | Query complexity attack, data over-fetching |
| File upload API | Pre-signed URL (time-bound) | TLS, file type validation, size limit, AV scan | 50-300ms | Malware upload, storage exhaustion |

The numbers above are per-request overhead introduced by security layers. Your baseline business logic latency is separate. A payment API with 200ms business logic plus 100ms security overhead crosses the 300ms threshold — which may violate your P99 SLA.

## Five Failure Modes

Security mechanisms that protect your system can also break it. Here are five common failure modes you need to account for.

### 1. WAF False Positives Blocking Legitimate Traffic

A WAF rule designed to block SQL injection patterns uses regex like `.*' OR '1'='1.*`. This works well. But a legitimate search query for "O'Brien" contains a single quote, the word "OR", and a comparison pattern — and gets blocked.

The scale of the problem: a typical production WAF with 20+ rules generates 0.5-2% false positive rate. At 10,000 requests per second, that means 50-200 legitimate requests blocked per second. For a black-Friday sale or a search-heavy site, this can translate to thousands of blocked customers.

```
Legitimate request:  GET /search?q=O'Brien%20OR%20Reilly
                         │
                         ▼
                    WAF rule #1042:
                    Detects: ' OR
                    ┌───────────────────────┐
                    │ SQL injection pattern │
                    │ matched → BLOCK       │
                    └───────────────────────┘
                         │
                         ▼
                    HTTP 403 Forbidden
                    User sees: "Search failed"
                    No error details (security through obscurity)
                    User retries = same result
                    User leaves, tells 10 friends

What should have happened:
                    ┌───────────────────────┐
                    │ Parameterized query   │
                    │ in application layer  │
                    │ renders SQL injection │
                    │ impossible            │
                    └───────────────────────┘
```

**Prevention**: Use WAF in log-only mode for 2-4 weeks before enforcing blocks. Whitelist known-good patterns. Use positive security models (allow known-good patterns) instead of negative models (block known-bad patterns). Add per-rule exception lists for common false positives.

### 2. Rate Limit False Positives

Rate limiting protects your API from abuse. But when legitimate users exceed limits due to shared IP addresses (NAT, corporate proxies, mobile carriers), they get blocked and cannot use your service.

The most painful scenario: an office of 500 employees behind a single corporate NAT IP calling your API. The first 100 requests in a minute succeed. The 101st fails — and the employee cannot figure out why, because nothing changes on their end.

```
Corporate NAT (203.0.113.0/24)
    500 employees sharing one public IP
         │
         ▼
    ┌─────────────────────────────┐
    │ Rate limiter:               │
    │ 100 requests/min per IP     │
    └─────────────────────────────┘
         │
         ▼
    9:00:00   — Employee 1 request  — OK (count: 1)
    9:00:01   — Employee 2 request  — OK (count: 2)
    ...
    9:01:30   — Employee 100 req    — OK (count: 100)
    9:01:31   — Employee 101 req    — 429 Too Many Requests
    9:02:00   — Employee 50 req     — 429 (window not reset)
    ...
    Window resets at 9:02:?? (sliding or fixed?)
    
    Result: 400 employees cannot use the API
            for 30-60 seconds per burst
```

**Prevention**: Use token-bucket or sliding-window rate limiting (more graceful than fixed window). Rate-limit by API key, not by IP alone. Use IP-based limits as a secondary layer. Return `Retry-After` headers so clients know when to retry. Rate limit on authenticated endpoints by user ID instead of IP.

### 3. CORS Misconfiguration

Cross-Origin Resource Sharing (CORS) is one of the most misconfigured security mechanisms in web APIs. Two common failure modes:

**Too permissive**: `Access-Control-Allow-Origin: *` on an authenticated API means any website can make requests from a browser. The browser enforces CORS, but a malicious site can still make pre-flighted requests, and cookies with `SameSite=None` will be forwarded.

**Too restrictive**: A frontend at `https://app.example.com` cannot reach your API at `https://api.example.com` because you only allowed `https://example.com` (missing the subdomain). The error message "CORS header `Access-Control-Allow-Origin` missing" gives no hint that the allowed origin list is the problem.

```
Browser                        Your API
  │                               │
  │ GET /users                    │
  │ Origin: https://evil.com      │
  │──────────────────────────────▶│
  │                               │
  │ Too permissive:               │
  │ Access-Control-Allow-Origin:* │
  │ Access-Control-Allow-Credentials: true
  │◀──────────────────────────────│
  │ Browser allows evil.com       │
  │ to read response              │
  │                               │
  ─────────── or ────────────
  
  │ GET /users                    │
  │ Origin: https://app.example.com
  │──────────────────────────────▶│
  │                               │
  │ Too restrictive:              │
  │ Access-Control-Allow-Origin:  │
  │   https://example.com         │
  │ (missing www. and app.        │
  │  subdomains)                  │
  │◀──────────────────────────────│
  │ Browser blocks response       │
  │ CORS error in console         │
```

**Prevention**: Use a dynamic origin validator that checks against a whitelist of allowed origins (including all subdomains). Never echo the `Origin` header back unconditionally. Set `Access-Control-Allow-Credentials: true` only when the origin is explicitly trusted. Test CORS configuration with `curl -H "Origin: https://your-frontend.com" -I`.

### 4. CSRF Token Expiry

CSRF (Cross-Site Request Forgery) tokens protect against malicious form submissions. But they have a time-to-live (typically 15-30 minutes). When a user spends time filling out a long form and the token expires before they submit, the server rejects the request and the user loses their work.

The problem worsens in multi-step workflows. A user fills step 1, navigates to step 2, gets interrupted by a meeting, returns 20 minutes later, submits — and gets a CSRF validation error with no recovery path.

```
User begins check-out flow at t=0
  │
  ├── Server issues CSRF token (TTL: 30 min)
  │
  ├── Step 1: Shipping address (t=2min)
  ├── Step 2: Payment info (t=5min)
  │
  │   ── User attends meeting ──
  │
  ├── Step 3: Review order (t=35min)
  │
  ▼
  Form submit ──▶ POST /checkout
                     │
                     ▼
                CSRF token expired
                (issued at t=0, now t=35min)
                     │
                     ▼
                HTTP 403 Forbidden
                User sees: "Session expired"
                Form data lost
```

**Prevention**: Use double-submit cookie pattern (stateless, no server-side token storage). Extend CSRF token TTL to match your session TTL. Implement token refresh on form load (AJAX endpoint returns a fresh token and updates the form silently). For multi-step workflows, rotate the token between steps so it stays alive. Use `SameSite=Strict` or `SameSite=Lax` cookies as a more modern alternative that eliminates CSRF for most cases.

### 5. JWT Algorithm Confusion

JWT supports multiple signing algorithms. The `alg` header in the token tells the verifier which algorithm to use. This leads to a dangerous attack: an attacker changes the `alg` from `RS256` (asymmetric) to `HS256` (symmetric) and signs the token with the server's public key — which is, by definition, public.

If your JWT library does not explicitly enforce the algorithm, it will happily verify the token using the public key as an HMAC secret. The attacker can forge arbitrary tokens.

```
JWT token structure:
  Header: {"alg": "RS256", "typ": "JWT"}
  Payload: {"sub": "user_42", "admin": false, "iat": 1700000000}
  Signature: <RSA signature using private key>

Attack:
  1. Extract the server's PUBLIC key (it is public)
  2. Create a new JWT:
     Header: {"alg": "HS256", "typ": "JWT"}  ◀── Changed!
     Payload: {"sub": "admin", "admin": true, "iat": 1700000000}
     Signature: HMAC-SHA256(public_key, header.payload)
  
  3. Send to server:
     POST /admin/delete-all
     Authorization: Bearer <forged JWT>
     
  4. Server reads "alg": "HS256"
     Verifies using public key as HMAC secret... SUCCESS
     Server sees "admin": true... grants access

The "alg: none" attack (older libraries):
  Header: {"alg": "none", "typ": "JWT"}
  Payload: {"sub": "admin", "admin": true}
  Signature: (empty)
```

**Prevention**: Always explicitly set the expected algorithm on the server. Reject `alg: none`. Do not derive the algorithm from the token header. Use a library that requires algorithm configuration at the verifier level (e.g., `jwt.verify(token, publicKey, { algorithms: ['RS256'] })`). Validate the algorithm *before* any cryptographic operation. For internal services, consider using opaque tokens (like a session ID stored in Redis) instead of self-contained JWTs.

The algorithm confusion attack is particularly dangerous because it exploits a design flaw in the JWT specification itself — the verifier delegates algorithm selection to the token creator. You must override this behavior at the library level.

```
Secure verification pattern (Node.js):
  jwt.verify(token, publicKey, {
    algorithms: ['RS256'],    // Explicitly set, never read from token
    issuer: 'auth.example.com',
    audience: 'api.example.com'
  });

Vulnerable verification (reads algorithm from token header):
  jwt.verify(token, publicKey);   // Default reads 'alg' from token
  // Token says "alg: HS256" → lib uses public key as HMAC secret
  // Token says "alg: none"   → lib accepts unsigned token
```

Validation checklist for every JWT library you use:
1. Can you lock the accepted algorithms to a specific set? (Yes = safe)
2. Does the library reject `alg: none` by default? (Yes = safe)
3. Does the library fall back to the token header if no algorithm is specified? (Yes = vulnerable — always specify)
4. Does the library warn about algorithm mismatch or silently switch? (Silent = dangerous)

### Additional Failure Modes

**6. TLS Certificate Expiry**: Your TLS certificate expires at midnight. At 12:01 AM, every client that checks certificate validity sees your API as untrusted. All HTTPS connections fail. You have a frantic midnight scramble to renew.

The expiry problem is amplified at scale. A microservice mesh with 200 services using mTLS means each service pair needs valid certificates. A single expired certificate breaks communication between that pair — and the failure cascades as retries accumulate.

```
Failure timeline:
  Day -90:  Certificate issued for api.example.com
  Day -30:  No alert configured
  Day -7:   No alert configured
  Day -1:   No one checks
  Day 0:    12:00 AM — Certificate expires
            12:01 AM — curl https://api.example.com
                      ──▶ SSL certificate expired error
            12:02 AM — Mobile app users see "Connection failed"
            12:05 AM — Monitoring alerts "API unreachable"
            12:10 AM — Engineer discovers expired cert
            12:15 AM — Cert renewed, deployment pushed
            12:20 AM — Service restored
            Total downtime: 20 minutes

For mTLS meshes, the same problem scales linearly:
  200 services × 199 possible pairs = 39,800 certificate pairs to track
```

Prevention: automate certificate renewal (Let's Encrypt + certbot, ACME protocol). Set up alerts for certificates expiring within 30/14/7 days. Monitor certificate expiry in your observability stack. For mTLS meshes, use a service mesh that handles certificate rotation automatically (Istio, Linkerd, Consul Connect).

**7. API Key Rotation Breaking Clients**: You rotate your API keys every 90 days for security hygiene. A microservice uses an old key cached in an environment variable. Nobody remembers it was there. The key rotates, the service starts returning 401 errors, and downstream systems fail silently.

This failure mode is insidious because the consumer of the rotated key is often a service owned by a different team. They do not get your rotation notification. They do not know where the key is configured. The 401 errors appear as "intermittent failures" in their logs and get triaged at low priority.

```
Key rotation without overlap:
  Day 0:   Old key "abc123" in use
  Day 89:  Teams notified: "Key rotates tomorrow"
  Day 90:  Old key "abc123" revoked
           New key "def456" activated
  Day 90:  Service A uses cached "abc123" → 401
  Day 90:  Service A retries → 401 again
  Day 90:  Service A's circuit breaker opens
  Day 91:  Service B (upstream) starts timing out
  Day 92:  Incident raised, root cause found
  Day 93:  Service A deploys with new key
           Total impact: 3 days of degraded service

Key rotation with overlap window:
  Day 0:   Old key "abc123" in use
  Day 89:  New key "def456" activated (both valid)
  Day 90-92: Overlap window — both keys accepted
            Monitoring: "abc123 last used at timestamp X"
  Day 93:  Warning sent to teams still using old key
  Day 95:  Old key revoked
           No downtime. Clean transition.
```

Prevention: implement key rotation windows (old key and new key both valid for 48 hours). Monitor for old key usage after rotation. Send expiry notifications to service owners. Use a key management system that tracks which services use which key version. Automate key distribution so rotation is a config change, not a redeployment.

**8. Missing Replay Protection**: You accept API requests within a time window but do not track which requests have already been processed. An attacker intercepts a valid request (POST /payments with body `{"amount": 100, "to": "attacker"}`) and replays it 10 times. The server processes all 10.

Replay attacks exploit the gap between "the request is valid" and "the request was already processed." Without a nonce or idempotency key, every identical request looks like a fresh, legitimate request to the server.

```
Replay attack flow:
  Victim:     POST /payments
              Body: {"amount": 100, "to": "merchant"}
              Signature: HMAC-SHA256(body + timestamp + nonce)

  Interceptor captures request at network layer.
  
  Attacker:   POST /payments  (same body, same signature)
              ──▶ Server checks: valid HMAC ✓, timestamp within 5min ✓
              ──▶ No nonce tracking → processes duplicate payment
              ──▶ Victim charged $100 again

  Attacker replays 10 times:
              ──▶ Victim charged $1,000 total
              ──▶ No record of duplicate detection in logs

  With idempotency key:
              POST /payments
              Idempotency-Key: uuid-123
              ──▶ Server checks: uuid-123 exists in dedup table?
              ──▶ Yes → return cached response. No processing.
```

Prevention: require an idempotency key (`Idempotency-Key: <UUID>`) on all mutating requests. Store processed keys in a fast KV store (Redis) with a TTL matching your retry window (24 hours is common). Validate timestamps with a max age (5 minutes is standard). Include nonces in signed payloads and reject duplicates.

## Security Budget: A Cost Analysis Framework

Just as you budget latency per request, you should budget security overhead. Not all endpoints need the same protection level. A cost-conscious security design assigns the heaviest checks to the most sensitive operations.

```
Security budget allocation by endpoint:

  Endpoint                  Sensitive   Overhead Budget   Layers Applied
  ─────────────────────────────────────────────────────────────────────
  GET /api/v1/products      Low         0-10ms           TLS + API key
  POST /api/v1/search       Low         0-10ms           TLS + rate limit
  POST /api/v1/login        High        20-40ms          TLS + rate limit + 
                                                          CAPTCHA + lockout
  POST /api/v1/payments     Critical    40-80ms          TLS + mTLS + WAF +
                                                          encrypt + audit
  GET /api/v1/orders/:id    Medium      10-25ms          TLS + JWT + RBAC
  POST /api/v1/webhooks     Medium      10-20ms          TLS + IP allowlist +
                                                          HMAC verify

Budget calculation example:
  P99 latency target: 200ms
  Business logic: 120ms
  Remaining budget: 80ms

  Security layers for payment API:
    TLS handshake (resumed):  3ms
    JWT verify:               5ms
    RBAC check:               3ms
    WAF inspection:          15ms
    Payload decrypt:          8ms
    Audit log (async):        1ms
    Rate limit check:         1ms
    ──────────────────────────────────
    Total security:          36ms ← within 80ms budget ✓

  If business logic grows to 160ms:
    Remaining budget: 40ms
    Options:
    A. Accept higher latency (P99 moves to 196ms)
    B. Skip payload encryption for non-PII fields (save 8ms)
    C. Use edge WAF instead of origin WAF (save 10ms)
    D. Cache RBAC results per session (save 2ms)
```

### When to Invest in a Security Layer

Use this prioritization when building security for a new API:

```
Priority 1 (must-have, zero latency concern):
  ── TLS encryption
  ── Server-side input validation
  ── Authentication (JWT, session, API key)

Priority 2 (should-have, small latency impact):
  ── Rate limiting on public endpoints
  ── Authorization (RBAC, ACL)
  ── Audit logging for sensitive operations

Priority 3 (nice-to-have, moderate latency):
  ── WAF for public endpoints
  ── Payload encryption for PII/PCI fields
  ── Idempotency keys for mutating operations
  ── IP allowlist for admin endpoints

Priority 4 (context-dependent, high latency):
  ── mTLS for every service pair
  ── Full payload encryption on all fields
  ── Deep WAF inspection with 50+ rules
  ── Real-time token introspection on every request
```

Add layers bottom-up. Start with Priority 1. Add Priority 2 when you see abuse. Add Priority 3 when compliance requires it. Add Priority 4 only when your threat model demands it.

## Anti-Patterns

These patterns sound like good ideas at first and cause problems in practice.

### 1. Security Through Obscurity

You rename your API endpoints from `/api/v1/users` to `/api/v1/a1b2c3/users`. You think this hides your API from attackers.

**Why it fails**: Attackers do not discover endpoints by guessing URLs. They read your client-side JavaScript, inspect network traffic, monitor the API responses of your mobile app, or use automated scanners that crawl your documentation. Obscurity adds zero protection against a determined attacker.

```
What you do:                            What attackers see:
  /api/v1/a1b2c3/login                    Browser DevTools Network tab
  /api/v1/x7y8z9/payments                 POST https://api.example.com/api/v1/a1b2c3/login
  /api/v1/m3n4p5/users                    POST https://api.example.com/api/v1/x7y8z9/payments
                                          GET  https://api.example.com/api/v1/m3n4p5/users

What you should do instead:
  /api/v1/login       — Clear, documented, stable
  /api/v1/payments    — Protected by proper auth
  /api/v1/users       — Protected by RBAC
  No security depends on the path being unknown
```

**Instead**: Use proper authentication, rate limiting, and WAF protection. Your endpoints do not need to be secret — they need to be secure.

### 2. Disabling CORS Entirely

You set `Access-Control-Allow-Origin: *` and `Access-Control-Allow-Credentials: true` on every endpoint to "make development easier" and never revisit it.

**Why it fails**: Any website your users visit can make authenticated requests to your API. A malicious site loads a script that posts to your API with `credentials: include`. The browser sends cookies. Your API processes the request. CORS does not block the request from being sent — it only blocks the response from being read client-side. For write operations, the damage is done.

**Instead**: Maintain a whitelist of allowed origins. Use environment-specific CORS configuration (strict in production, relaxed in development). Validate CORS configuration in CI.

### 3. No Input Validation on the Server

You trust your frontend validation and skip validation on the server side. You think "the frontend already validates this field — why duplicate the work?"

**Why it fails**: Attackers do not use your frontend. They use `curl`, Postman, or a custom script to send requests directly to your API. No frontend validation applies. A missing input validation layer means SQL injection, NoSQL injection, command injection, buffer overflow, and XSS attacks all succeed directly.

```
Frontend validates:                     Server validates:
  ┌────────────────────────┐             ┌────────────────────┐
  │ Name: required          │             │ Name: not checked  │
  │ Email: valid format     │             │ Email: not checked │
  │ Age: number 18-120      │             │ Age: not checked   │
  │ Comment: max 1000 chars │             │ Comment: not       │
  │                         │             │   checked          │
  │ Blocked: SQL injection  │             └────────────────────┘
  │ Patterns: ' OR 1=1--    │
  └────────────────────────┘                  ╔══════════════╗
                                              ║   Attacker   ║
  Browser user: ✓                             ║ POST /submit ║
  curl attacker:                              ║ Name: '; DROP║
    ">'><script>alert(1)</script>             ║   TABLE users;║
    ' OR '1'='1'; DROP TABLE users--          ║   --          ║
                                              ╚══════════════╝
                                                  │
                                                  ▼
                                            Database compromised
```

**Instead**: Always validate on the server. Treat frontend validation as a UX feature, not a security control. Use a validation library (Joi, Zod, Pydantic) with a schema definition that runs on every request before business logic.

### 4. No Rate Limiting on Auth Endpoints

You deploy authentication endpoints — `/login`, `/register`, `/forgot-password`, `/reset-password` — without rate limiting. You rely on password complexity and HTTPS to keep things safe.

**Why it fails**: Without rate limiting, an attacker can try 1,000 passwords per second against a single account. With a leaked password list (common from data breaches), credential stuffing attacks test millions of username-password combinations. Your auth endpoint becomes a brute-force oracle.

```
Attack timeline (no rate limit):
  t=0s     — Start: iterate leaked credential pairs
  t=1s     — 1,000 attempts
  t=60s    — 60,000 attempts
  t=5min   — 300,000 attempts
  t=1hr    — 3,600,000 attempts
  
  With 1M leaked credentials and 100K users:
  Probability of finding at least one valid pair: ~90%
  (assuming 1% password reuse rate)

Attack timeline (rate limited: 5/min per IP + 5/min per user):
  t=0s     — Start: 5 attempts from IP A
  t=1min   — 5 more attempts from IP A
  t=2min   — 5 more, blocked at 15 total
  ── Attacker switches to IP B ──
  t=3min   — 5 attempts from IP B (same user)
  t=4min   — User account locked after 10 failed attempts
  
  Total attempts before lockout: 10
```

**Instead**: Rate limit `/login` and `/register` aggressively (5-10 attempts per minute per IP and per user). Implement progressive delays (each failed attempt waits 1s, 2s, 4s, 8s...). Use CAPTCHA after 3 failed attempts. Lock accounts after 10 consecutive failures for that username. Monitor for credential stuffing patterns (same user, different IPs).

### 5. Rolling Your Own Crypto

You implement your own JWT library because "the existing ones are too heavy." You use MD5 for hashing because "it is faster." You design a custom encryption scheme because "AES is overkill for our data."

**Why it fails**: Cryptography is exceptionally difficult to get right. Side-channel attacks, padding oracle attacks, timing attacks, IV reuse, and key derivation mistakes are invisible to standard testing. Your custom scheme will be broken within hours of release.

```
Custom JWT library (found in production):
  ┌─────────────────────────────────────────────┐
  │ function verifyToken(token, secret) {        │
  │   const parts = token.split('.');            │
  │   const payload = JSON.parse(                │
  │     atob(parts[1])                           │
  │   );                                         │
  │   const expectedSig = MD5(parts[0] +         │
  │     '.' + parts[1] + secret);                │
  │   if (parts[2] !== expectedSig) return null; │
  │   return payload;                            │
  │ }                                            │
  └─────────────────────────────────────────────┘

  Problems:
  1. Uses MD5 (collision-prone, trivially broken)
  2. No algorithm validation (accepts any header)
  3. String concatenation instead of proper signing
  4. No expiry validation
  5. No issuer/audience validation
```

**Instead**: Use well-vetted libraries (jsonwebtoken, PyJWT, jose for Node; python-jose for Python). Use standard algorithms (HS256, RS256, ES256). Configure algorithm explicitly at the verifier level. Resist the temptation to "improve" on standard crypto.

### 6. Storing Secrets in Code

You hardcode API keys, database passwords, and JWT secrets in your source code. You tell yourself "nobody can see this because our repo is private."

**Why it fails**: Private repos get leaked. Ex-employees have access. CI logs display environment variables. Docker images are pulled from registries. A single `git push` to a public fork by mistake exposes every credential in your commit history.

**Instead**: Use environment variables loaded at runtime. Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager). Use `.env` files (never committed to git) for local development. Scan your repo for secrets with tools like `truffleHog` or `git-secrets`. Rotate any secret that was ever committed — even if removed, it is in git history.

### 7. Trusting the Client

You make authorization decisions based on headers sent by the client. `X-User-Role: admin` in the request header means the user is an admin. You validate the token, but the role comes from the request, not from the token payload.

**Why it fails**: Anyone can set any header. A user with a valid token but a regular role changes `X-User-Role: user` to `X-User-Role: admin` and gains access to privileged operations.

```
Vulnerable API:
  POST /admin/delete-user
  Authorization: Bearer <valid token>
  X-User-Role: admin  ←── User sets this header
  X-User-ID: 42       ←── User sets this too

  Server checks:
    ✓ Token is valid (signature OK)
    ✓ Token is not expired
    ✓ X-User-Role is "admin"  ←── From request header!
    ✗ Does NOT check token payload for role

Secure API:
  POST /admin/delete-user
  Authorization: Bearer <valid token>

  Server checks:
    ✓ Token is valid
    ✓ Token payload.role === "admin"  ←── From token, not header
    ✓ Token payload.user_id matches target user
    ✗ Ignores all X-User-* headers from request
```

**Instead**: Derive identity and role from the authentication mechanism only (JWT payload, session data, mTLS certificate). Never trust `X-User-*` headers. Validate authorization on every request using server-side data.

## When NOT to Use Everything

Not every API needs every security layer. Over-engineering security has real costs — latency, complexity, developer friction, and maintenance burden.

### Internal-Only Services in a VPC

If your service communicates only within a private VPC (no internet exposure, no cross-account access, no external clients), you can safely skip several security layers:

**What to skip**:
- **WAF**: No external traffic means no external threats. WAF rules add 5-20ms for no benefit.
- **CORS**: No browser-based client. Remove CORS handling entirely.
- **Rate limiting (loose)**: Trusted internal services should not hit rate limits. Use very generous limits or per-service quotas instead of per-IP limits.
- **Payload encryption**: Internal VPC traffic is encrypted at the network level (TLS/mTLS). Encrypting the payload again adds 2-15ms for overlapping protection.
- **Complex auth**: Service-to-service communication can use simple API tokens or mTLS certificates instead of full OAuth2 flows.

**What to keep**:
- **mTLS**: Encrypt and authenticate at the transport layer. Mutual TLS ensures both ends are who they say they are.
- **Input validation**: Always validate inputs, even internally. An internal service can still propagate bad data.
- **Audit logging**: Internal audits matter — especially for compliance (SOC 2, HIPAA, PCI).

```
External API (full security stack):       Internal API (VPC, minimal):

  ┌──────────────────────┐                ┌───────────────────┐
  │ TLS                  │                │ mTLS (mandatory)  │
  │ WAF                  │                │ Service token     │
  │ JWT + RBAC           │                │ Input validation  │
  │ Rate limiting        │                │ Audit logging     │
  │ CORS                 │                │                   │
  │ Input validation     │                │ Latency: 5-15ms   │
  │ Audit logging        │                │                   │
  │ Payload encryption   │                │ Skip:             │
  │                      │                │ WAF, CORS,        │
  │ Latency: 35-80ms     │                │ rate limit,       │
  └──────────────────────┘                │ payload encrypt   │
                                          └───────────────────┘
```

### Other Scenarios to Relax Security

**Prototype / MVP**: A prototype with 10 users does not need WAF, mTLS, or audit logging. Basic TLS + API key is sufficient. Add layers as the product grows. Premature security optimization burns engineering time on protection you will not need for months.

**Public read-only content**: An API serving product catalog data (no auth required, no sensitive data) can skip JWT, RBAC, and payload encryption. Use TLS + loose rate limiting. Add more only if you see abuse patterns.

**Batch processing**: A nightly batch job that reads from internal storage and writes to another internal store does not need per-request auth or WAF. Use a long-lived service account token exchanged at startup. The security model is at the data layer (encrypted storage, access-controlled S3 buckets).

**Cost-sensitive environments**: A startup serving developing markets may choose simpler security to keep infrastructure costs low. TLS-only with API keys and basic rate limiting is better than no security at all. Layer in additional protection as revenue supports it.

### Decision Framework

Ask these questions before adding each security layer:

1. **Is this endpoint exposed to the internet?** No -> skip WAF and CORS.
2. **Does the request carry sensitive data?** No -> skip payload encryption.
3. **Is the caller a known, trusted service?** Yes -> use service token, skip complex OAuth2 and rate limiting.
4. **Is the data read-only and public?** Yes -> skip authentication entirely; use loose rate limiting.
5. **Is this a prototype with under 100 users and no PII?** Yes -> use TLS + API key, nothing else.
6. **Is compliance (PCI, HIPAA, SOC 2) required?** Yes -> add everything in scope, but tune for performance.

Security is not binary. It is a sliding scale. The right level of security for your API depends on exposure, data sensitivity, compliance requirements, and latency budget.
