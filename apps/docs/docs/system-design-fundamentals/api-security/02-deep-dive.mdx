## Internal Architecture

A production API security layer is not a single firewall or a library call. It is a pipeline of distinct subsystems that inspect, transform, validate, and rate-limit every request before it reaches your application logic.

## Data Structures and Algorithms

### CSRF Token Store

The CSRF token store maps session identifiers to their associated tokens, creation timestamps, and usage flags. This store lives in Redis or Memcached, not in the application's local memory, so it survives restarts and works across horizontally scaled instances.

| Property | Value |
|---|---|
| Structure | Redis hash: `HASH csrf:tokens:<session_id>` |
| Lookup | O(1) via `HGET` |
| Token TTL | 30 minutes from creation (configurable) |
| Mem usage | ~120 bytes per token entry |
| Eviction | TTL-based expiry (Redis `EXPIRE`) |

```
Redis key:   csrf:token:session_a1b2c3d4
Fields:
  token       = "7f3a1b8c9d0e2f4a6b5c7d8e9f0a1b2c"
  created_at  = "1712345678"
  expires_at  = "1712347478"
  used        = "false"
  method      = "POST"
  path        = "/api/checkout"
```

The store uses a single-use token pattern. After a token is validated, it is marked `used = true` and the TTL is reduced to 60 seconds for cleanup. This prevents token replay attacks — if an attacker steals a token, it can only be used once.

### Rate Limit Counters

Rate limiting relies on per-key counters that track how many requests a client has made in a given time window. The data structure varies by algorithm.

**Sliding Window Log** (most accurate):

```
Sorted set key: rate_limit:ip:192.168.1.1:endpoint:/api/login
Members:        [timestamp1, timestamp2, timestamp3, ...]
Score:          Unix timestamp in milliseconds
Count:          ZCOUNT rate_limit:... 1712345600000 1712345900000
```

Each request adds a member with `ZADD` and counts members in the window with `ZCOUNT`. This is O(log N) per request and O(N) for memory, but it is exact — no approximation.

**Sliding Window Counter** (approximate, lower memory):

```
Redis key:   rate_limit:ip:192.168.1.1
Window:      60 seconds
Current window counter:  rate_limit:ip:192.168.1.1:1712345670  =  42
Previous window counter: rate_limit:ip:192.168.1.1:1712345610  =  87
Weight:      (elapsed_seconds / window_size)
Weighted count = 42 + (87 * weight)
```

The sliding window counter uses two keys per rate limit key, each with a 60-second TTL. At 10,000 rate limit keys, memory usage is approximately 2 MB.

**Concurrent Request Guard**:

```
Key:      concurrent:ip:192.168.1.1
Value:    12  (current in-flight requests)
INCR on request arrival
DECR on response completion
If value > max_concurrent: reject with 429
```

This guard catches bursts that the rate limiter misses — a client sending 50 requests simultaneously before the rate limiter's window counter increments.

### CORS Configuration

The CORS enforcer is configured at the application or reverse proxy level. The configuration is a read-heavy data structure loaded at startup and cached.

| Field | Type | Example |
|---|---|---|
| `allowed_origins` | `string[]` | `["https://app.example.com", "https://admin.example.com"]` |
| `allowed_methods` | `string[]` | `["GET", "POST", "PUT", "DELETE", "PATCH"]` |
| `allowed_headers` | `string[]` | `["Content-Type", "Authorization", "X-CSRF-Token"]` |
| `exposed_headers` | `string[]` | `["X-Request-Id", "X-RateLimit-Remaining"]` |
| `max_age` | `int` (seconds) | `7200` |
| `allow_credentials` | `bool` | `true` |

The origin matching algorithm supports exact match, wildcard domain match (`*.example.com`), and regex match. A trie-based matcher provides O(K) lookup where K is the origin string length, compared to O(N) for a linear scan across N allowed origins.

```
Origin trie for ["https://app.example.com", "https://admin.example.com"]:

            root
              │
          "https://"
              │
          ┌────┴────┐
          │         │
        "app."   "admin."
          │         │
      "example.com" "example.com"

Match: "https://app.example.com" → O(1) per character → O(K) total
```

### Connection and Security Validation

Every API request passes through 14 distinct security checks before the application sees the payload.

**Step 1: HTTPS termination.** TLS terminates at the edge (load balancer or API gateway). The connection is decrypted once and the request proceeds as plaintext through the internal network. Mutual TLS (mTLS) adds a client certificate verification step here.

**Step 2: WAF inspection.** The Web Application Firewall runs a set of rules against the raw request. Rules include:
- SQL injection patterns: `UNION SELECT`, `OR 1=1`, `' OR '1'='1`
- XSS patterns: `<script>`, `onerror=`, `javascript:`
- Path traversal: `../`, `%2e%2e/`, `..;`
- Known attack signatures: Log4Shell, ShellShock, directory brute force

**Step 3: Input sanitization.** The application framework sanitizes each input parameter. This includes:
- Stripping HTML tags from text fields
- Normalizing Unicode (canonicalization to prevent bypasses)
- Rejecting inputs exceeding maximum length
- Escaping special characters for the target database

**Step 4: Rate limit check.** The rate limiter increments the counter for the request's key (IP, API key, or user ID). If the counter exceeds the configured limit, the request is rejected with HTTP 429 Too Many Requests. The `Retry-After` header tells the client when to retry.

**Step 5: Auth token extraction.** The authentication middleware extracts the token from the `Authorization: Bearer <token>` header, a cookie, or a custom header. If the token is missing, the request is rejected with 401 Unauthorized.

**Step 6: JWT validation.** The token is decoded and validated:
- Signature verification using the public key (RS256, ES256)
- Expiration check (`exp` claim)
- Issuer check (`iss` claim matches the expected issuer)
- Audience check (`aud` claim includes the expected audience)
- Not-before check (`nbf` claim)

**Step 7: CSRF token check.** For state-changing methods (POST, PUT, DELETE, PATCH), the CSRF protector validates that the request includes a valid CSRF token. The token is compared against the stored value in the session store. If the token is missing, invalid, or expired, the request is rejected with 403 Forbidden.

**Step 8: CORS origin check.** For cross-origin requests, the `Origin` header is matched against the allowed origins list. If the origin is not allowed, the request is rejected. For preflight requests (OPTIONS), the CORS headers are returned without forwarding to the backend.

**Step 9: RBAC authorization.** The authorization engine checks whether the authenticated user's role has permission to perform the requested action on the requested resource. This is typically a database query or a policy engine evaluation.

**Step 10: Forward to backend.** The request, now sanitized and validated, is forwarded to the application backend. Internal headers are added (original client IP, user ID, request ID).

**Step 11: Parameterized query execution.** The backend executes the database query using parameterized statements, not string concatenation. This prevents SQL injection even if the WAF missed something.

**Step 12: Response from backend.** The backend returns the response. The response body is trusted because it comes from the application, but it may still be scanned for sensitive data leakage.

**Step 13: HTTPS response.** The response is encrypted with TLS and sent to the client.

**Step 14: Security headers injected.** The response passes through the security header injector, which adds headers like `Content-Security-Policy`, `Strict-Transport-Security`, `X-Content-Type-Options`, and `X-Frame-Options`.

### Parameterized Queries vs String Concatenation

The single most impactful API security control is parameterized query execution. Here is what happens at the database driver level.

<AsciiDiagram
  id="api-security/parameterized-query"
  title="Parameterized Query vs String Concatenation"
  content={`
String concatenation (insecure):
┌─────────────────────────────────────────────────────────────────────┐
│  user_input = "1' OR '1'='1"                                        │
│                                                                      │
│  query = "SELECT * FROM users WHERE id = '" + user_input + "'"      │
│                                                                      │
│  Result: SELECT * FROM users WHERE id = '1' OR '1'='1'              │
│         └──────────────────────────────────────────────────────┘    │
│         This returns ALL rows -- the attacker has bypassed auth     │

Parameterized query (secure):
┌─────────────────────────────────────────────────────────────────────┐
│  user_input = "1' OR '1'='1"                                        │
│                                                                      │
│  cursor.execute("SELECT * FROM users WHERE id = ?", (user_input,))  │
│                                                                      │
│  Driver sends to database:                                           │
│    Query:  SELECT * FROM users WHERE id = ?                         │
│    Params: ["1' OR '1'='1"]                                         │
│                                                                      │

┌─────────────────────────────────────────────────────────────────────┐

## Consistency and Race Conditions

### Rate Limiting Race Condition

The most common consistency bug in API security is a race condition in the rate limiter. When two requests from the same client arrive simultaneously, both can read the same counter value and both pass the limit check, even though together they exceed the limit.

Request A arrives at T=0ms     Request B arrives at T=1ms
         │                              │

   │ Redis    │                  │ Redis    │
   │ GET count│                  │ GET count│
   │ count=99 │                  │ count=99 │
   └──────────┘                  └──────────┘

         │  Both read count=99          │
         │  Both check: 99 < 100        │
         │  Both pass!                  │
         │                              │

   ┌──────────┐                  ┌──────────┐

   │ INCR     │                  │ INCR     │

   └──────────┘                  └──────────┘
         │                              │

         │  Two requests slipped past.  │

Mitigation options:
  1. Redis INCR + check in one atomic operation:
     value = INCR(key)
     if value > limit: reject

       return 0  -- reject
     end
     return 1    -- allow

  3. Distributed lock (too expensive for rate limiting):
     Race eliminated but adds 2-5ms per request.
`}

The race window is tiny — typically less than 1 millisecond — but in a high-traffic system handling 10,000 requests per second, concurrent arrivals are common. Without atomic operations, the effective limit can be exceeded by 10-20% during traffic spikes.

### CORS Preflight Cache Inconsistency

When the CORS configuration changes (a new origin is added or an existing one is removed), browsers that have cached the preflight response continue to use the old configuration until the `Access-Control-Max-Age` expires.

  1. Set Max-Age to a short value (60-300 seconds) during

  2. Use a Vary: Origin header to ensure the cache key

     cache entries.
  3. Invalidate CDN caches when CORS config changes.

### CSRF Token Double-Spend

A less common but important consistency issue is CSRF token double-spend. If the user submits the same form twice rapidly, both requests carry the same CSRF token. Under single-use token semantics, the first request invalidates the token, and the second request fails.

```
Request 1: POST /checkout, X-CSRF-Token: abc123
  ─▶ Redis: HGET csrf:token:session_a → "abc123"
  ─▶ Redis: HSET csrf:token:session_a used=true
  ─▶ 200 OK

Request 2: POST /checkout, X-CSRF-Token: abc123 (same token!)
  ─▶ Redis: HGET csrf:token:session_a → "abc123"
  ─▶ Redis: used flag is true
  ─▶ 403 Forbidden (token already used)
```

This is actually correct behavior — the double-submit is prevented. But the user experience is poor. Solutions:

1. **Token rotation**: Issue a new CSRF token in the response of the first request. The client uses the new token for the second request.
2. **Per-request nonce**: Include a unique nonce in each form submission, derived from the CSRF token.
3. **Idempotency key**: Allow the client to provide an idempotency key that lets the server detect duplicates.

## Performance Characteristics

### WAF Latency

The WAF is the most expensive security subsystem because it performs deep packet inspection against a large rule set.

| Operation | Latency (p50) | Latency (p99) | Throughput Impact |
|---|---|---|---|
| No WAF (passthrough) | 0.02 ms | 0.05 ms | Baseline |
| WAF, 100 rules, small body (1 KB) | 0.15 ms | 0.40 ms | -15% |
| WAF, 500 rules, small body (1 KB) | 0.60 ms | 2.00 ms | -35% |
| WAF, 500 rules, large body (100 KB) | 3.50 ms | 12.00 ms | -60% |
| WAF, 1000 rules, large body (1 MB) | 25.00 ms | 80.00 ms | -85% |

**Optimization strategies:**

- **Positive security model**: Block known-bad patterns (negative model) is a game of whitelist. A positive model (allow only known-good patterns) is faster because the rule set is much smaller.
- **Rule profiling**: Most WAF rules never match. Profile your traffic and disable rules with zero matches over 30 days.
- **Request size limit**: Reject requests larger than 1 MB before the WAF inspects them. This prevents the WAF from spending CPU on oversized payloads.
- **Regex optimization**: Replace backtracking regex with atomic groups or DFA-based matchers. A single catastrophic regex can cause 10-second delays.

### Rate Limiting Overhead

| Algorithm | Latency per check | Redis ops per request | Memory per key | Accuracy |
|---|---|---|---|---|
| Fixed window | 0.3 ms | 1 (INCR) | 8 bytes | Approximate |
| Sliding window (counter) | 0.5 ms | 2 (GET + INCR) | 16 bytes | Good |
| Sliding window (log) | 1.2 ms | 2 (ZADD + ZCOUNT) | 80 bytes per request | Exact |
| Token bucket | 0.4 ms | 1 (EVAL) | 32 bytes | Good |
| Concurrent guard | 0.2 ms | 1 (INCR + DECR) | 8 bytes | Exact |

**Key insight**: Sliding window log is the most accurate but also the most expensive because each request adds a sorted set member that persists for the entire window duration. At 10,000 requests per second with a 60-second window, each key stores 600,000 members — approximately 48 MB per rate limit key.

**Recommended approach**: Use sliding window counter for most endpoints (good accuracy, low memory) and sliding window log only for endpoints where exact counting matters (payment processing, password reset).

### CORS Preflight Cost

CORS preflight requests are OPTIONS requests that add latency to the first cross-origin request.

| Scenario | Preflight latency | Cached? | Effective cost |
|---|---|---|---|
| Same-origin request | 0 ms | N/A | None |
| Cross-origin, simple request | 0 ms | N/A | None (no preflight needed) |
| Cross-origin, first request | 1.5 ms | No | Full preflight + actual request |
| Cross-origin, cached preflight | 0 ms | Yes (Max-Age) | Zero (preflight skipped) |
| Cross-origin, cache expired | 1.5 ms | No | Full preflight again |

**Simple request** (no preflight required):
- Method: GET, HEAD, or POST
- Headers: Accept, Accept-Language, Content-Language, Content-Type (only `application/x-www-form-urlencoded`, `multipart/form-data`, or `text/plain`)
- If your API uses `Content-Type: application/json` or `Authorization` headers, all requests are non-simple and require preflight.

**Preflight amplification**: If your API has 100,000 unique clients, each client sends one preflight request per Max-Age period. With Max-Age=7200, that is 100,000 / 7200 = 14 preflight requests per second. With Max-Age=60, that rises to 1,667 preflight requests per second — a 119x increase.

### Security Header Injection Overhead

Security headers are added by the reverse proxy or API gateway near the end of the response pipeline. The overhead is negligible because the headers are static strings appended to the response buffer.

| Header | Size | Overhead |
|---|---|---|
| `Content-Security-Policy` | ~200 bytes | 0.005 ms |
| `Strict-Transport-Security` | ~50 bytes | 0.001 ms |
| `X-Content-Type-Options` | ~25 bytes | 0.001 ms |
| `X-Frame-Options` | ~25 bytes | 0.001 ms |
| `Referrer-Policy` | ~30 bytes | 0.001 ms |
| **Total** | **~330 bytes** | **0.009 ms** |

The real cost is not latency but bandwidth. At 10,000 requests per second, security headers add 330 bytes * 10,000 = 3.3 MB/s of downstream bandwidth. Over a month, that is approximately 8.5 TB of additional data transfer.

### CPU Breakdown

CPU profile for a production API gateway handling 5,000 requests per second with all security features enabled:

| Component | CPU Share | Notes |
|---|---|---|
| TLS termination | 30% | Hardware acceleration (AES-NI) helps |
| WAF inspection | 25% | Dominated by regex execution |
| Rate limiting | 8% | Redis network calls |
| JWT validation | 12% | Asymmetric crypto (RS256) |
| Input sanitization | 10% | String processing and normalization |
| CSRF validation | 3% | Simple string comparison |
| CORS enforcer | 2% | Origin lookup and header injection |
| Security headers | 1% | String append operations |
| Other (routing, logging) | 9% | General request processing |

**Optimization target**: WAF inspection and TLS termination consume 55% of CPU. Offloading TLS to a hardware security module (HSM) or a dedicated TLS proxy can reclaim 30% CPU. Moving WAF to a separate tier (dedicated WAF appliance) prevents the WAF from starving the rest of the pipeline.

## Configuration and Tuning

### WAF Rules

| Rule ID | Category | Pattern | Default Action | False Positive Rate |
|---|---|---|---|---|
| `SQLI-001` | SQL Injection | `(\bUNION\b.*\bSELECT\b)` | Block | 0.5% |
| `SQLI-002` | SQL Injection | `('.+OR\b.*\d+\s*=\s*\d+)` | Block | 1.2% |
| `SQLI-003` | SQL Injection | `('\s*--\s*.*$)` | Block | 0.1% |
| `XSS-001` | Cross-Site Scripting | `(<script[\s>])` | Block | 0.3% |
| `XSS-002` | Cross-Site Scripting | `(onerror\s*=)` | Block | 1.5% |
| `XSS-003` | Cross-Site Scripting | `(javascript\s*:)` | Block | 0.8% |
| `TRAV-001` | Path Traversal | `(\.\.\/|\.\.\\)` | Block | 0.2% |
| `TRAV-002` | Path Traversal | `(%2e%2e%2f)` | Block | 0.1% |
| `ANOM-001` | Rate Anomaly | `(requests > 100/min)` | Log & Alert | 5.0% |
| `ANOM-002` | Payload Size | `(body > 1 MB)` | Block | 1.0% |

**Tuning guidance for WAF rules:**

- **Start in log-only mode** for all rules. Monitor false positives for 7 days before enabling blocking.
- **SQLI-001** (UNION SELECT) is the most reliable rule — near-zero false positives in production APIs. Enable blocking immediately.
- **XSS-002** (onerror=) has a 1.5% false positive rate because legitimate applications use `onerror` for image error handling. Whitelist known endpoints.
- **ANOM-001** (rate anomaly) should never block automatically — use it for alerting only. The dedicated rate limiter handles blocking.

### CORS Origins

| Origin | Pattern Match | Access | Credentials | Use Case |
|---|---|---|---|---|
| `https://app.example.com` | Exact | Full | Yes | Main web application |
| `https://admin.example.com` | Exact | Full | Yes | Admin dashboard |
| `https://*.preview.example.com` | Wildcard | Full | Yes | Preview deployments |
| `https://partner-*.example.com` | Regex | Read-only | No | Partner integrations |
| `https://docs.example.com` | Exact | Read-only | No | Documentation site |
| `chrome-extension://*` | Regex | Restricted | No | Browser extension |

**Wildcard caution**: `*` matches any origin. If you use `Access-Control-Allow-Origin: *`, you cannot use `Access-Control-Allow-Credentials: true` — the browser rejects it. This is a security feature: credentialed requests require explicit origin matching.

**Production checklist for CORS:**

1. Never use `Access-Control-Allow-Origin: *` in production for APIs that handle user data
2. Prefer exact origin matching over wildcard matching
3. Validate the `Origin` header server-side — do not echo it back blindly
4. Set `Vary: Origin` on all responses with CORS headers to prevent CDN cache poisoning
5. Set `Access-Control-Max-Age` to 7200 seconds (2 hours) for stable configurations
6. Use `Access-Control-Allow-Credentials: true` only when the origin is explicitly validated

### Rate Limit Thresholds

| Tier | Endpoint Group | Window | Limit | Key | Burst |
|---|---|---|---|---|---|
| Critical | Login, Password Reset | 60 seconds | 10 requests | IP | 5 |
| High | Checkout, Payment | 60 seconds | 30 requests | User ID | 10 |
| Medium | Profile Update, Settings | 60 seconds | 100 requests | User ID | 20 |
| Low | Read-only, Search | 60 seconds | 500 requests | API Key | 100 |
| Internal | Health, Metrics | 60 seconds | 1000 requests | Internal IP | 500 |

**Burst calculation**: The burst limit is the number of requests allowed in a single second before the rate limiter activates. With a limit of 100 requests per minute and a burst of 20, the client can send 20 requests in the first second followed by 80 requests spread over the remaining 59 seconds.

**Key selection strategy:**

| Key Type | Use Case | Evasion Risk |
|---|---|---|
| IP address | Unauthenticated endpoints | Shared IPs (NAT, VPN) are penalized together |
| API key | Authenticated endpoints | Attacker can rotate keys, but keys are revocable |
| User ID | User-specific endpoints | Attacker needs to compromise multiple accounts |
| Session ID | Session-based endpoints | Same as User ID, but per-session |
| Combined (IP + API key) | High-security endpoints | Most granular, but more keys to track |

**Thundering herd protection**: When a rate limit resets (e.g., at the start of a new minute), all clients that were blocked may retry simultaneously. Use jittered retry with `Retry-After` header to spread the retry load.

```
Retry-After header strategies:
  Fixed:    Retry-After: 60    → all clients retry at second 60
  Jittered: Retry-After: 45    → clients retry between 45-75 seconds
  Exponential: Retry-After: 60 → double on each retry (60, 120, 240)
```

### CSRF Token TTL

| Setting | Default | Short | Long | Rationale |
|---|---|---|---|---|
| Token TTL | 30 minutes | 5 minutes | 60 minutes | Shorter TTL reduces replay window but forces more token refreshes |
| Token length | 32 bytes | 16 bytes | 64 bytes | Longer tokens are harder to brute-force but increase cookie size |
| Renewal policy | On each POST | On session start | On expiry | Frequent renewal is safer but generates more Redis writes |
| Max uses | 1 (single-use) | 1 | 5 | Single-use is safest; multi-use trades security for UX |

**Token generation algorithm:**

```python
import secrets
import hashlib
import hmac

def generate_csrf_token(session_id: str, secret_key: bytes) -> str:
    """
    Generate a CSRF token using HMAC-based derivation.
    The token is deterministic for a given session_id and secret_key,
    eliminating the need to store tokens in Redis.
    """
    # 32 bytes of random data
    random_part = secrets.token_hex(32)
    # HMAC signature to prevent tampering
    signature = hmac.new(
        secret_key,
        session_id.encode() + random_part.encode(),
        hashlib.sha256
    ).hexdigest()
    return f"{random_part}.{signature}"

def validate_csrf_token(
    token: str, session_id: str, secret_key: bytes
) -> bool:
    """
    Validate a CSRF token by recomputing the HMAC.
    No server-side storage needed.
    """
    try:
        random_part, signature = token.rsplit(".", 1)
        expected = hmac.new(
            secret_key,
            session_id.encode() + random_part.encode(),
            hashlib.sha256
        ).hexdigest()
        return hmac.compare_digest(signature, expected)
    except (ValueError, AttributeError):
        return False
```

This stateless approach avoids Redis entirely for CSRF token storage. The server's secret key is the only thing that needs protection. The trade-off is that tokens cannot be individually revoked — if the secret key is compromised, all tokens are compromised.

### Security Headers

| Header | Recommended Value | Purpose | Browsers Protected |
|---|---|---|---|
| `Content-Security-Policy` | `default-src 'self'; script-src 'self'` | Prevent XSS by controlling resource sources | All modern |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Enforce HTTPS for all subdomains | All modern |
| `X-Content-Type-Options` | `nosniff` | Prevent MIME type sniffing | IE, Chrome, Firefox |
| `X-Frame-Options` | `DENY` | Prevent clickjacking | IE, Chrome, Firefox, Safari |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Control referrer header leakage | All modern |
| `Permissions-Policy` | `geolocation=(), microphone=()` | Restrict browser API access | Chrome, Firefox |
| `Cross-Origin-Embedder-Policy` | `require-corp` | Prevent cross-origin data leakage | Chrome |

**CSP deployment strategy:**

1. Start with `Content-Security-Policy-Report-Only` and collect violation reports
2. Analyze reports for 2-4 weeks to identify all legitimate resource sources
3. Build the policy incrementally, adding sources as needed
4. Switch to enforcement mode with `Content-Security-Policy` header
5. Use `report-uri` or `report-to` directive to monitor for new violations

### Rate Limiter Redis Configuration

```python
import redis
import time

class SlidingWindowRateLimiter:
    def __init__(self, redis_client: redis.Redis):
        self.redis = redis_client

    def is_allowed(
        self, key: str, limit: int, window: int
    ) -> tuple[bool, int]:
        """
        Sliding window counter rate limiter.
        Returns (allowed, remaining) tuple.
        """
        now = int(time.time())
        current_window = now // window
        previous_window = current_window - 1

        # Current window key: rate_limit:<key>:<window_number>
        current_key = f"rate_limit:{key}:{current_window}"
        previous_key = f"rate_limit:{key}:{previous_window}"

        # Atomic Lua script for the check
        script = """
        local current_key = KEYS[1]
        local previous_key = KEYS[2]
        local limit = tonumber(ARGV[1])
        local window = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])

        -- Get current window count
        local current_count = redis.call("GET", current_key)
        if not current_count then
            current_count = 0
        end

        -- Get previous window count
        local previous_count = redis.call("GET", previous_key)
        if not previous_count then
            previous_count = 0
        end

        -- Calculate weighted count
        local elapsed = now % window
        local weight = (window - elapsed) / window
        local weighted_count = current_count + (previous_count * weight)

        if weighted_count >= limit then
            return {0, math.max(0, limit - math.floor(weighted_count))}
        end

        -- Increment current window counter
        redis.call("INCR", current_key)
        redis.call("EXPIRE", current_key, window * 2)
        return {1, math.max(0, limit - math.floor(weighted_count) - 1)}
        """

        result = self.redis.eval(
            script,
            2,
            current_key,
            previous_key,
            limit,
            window,
            now
        )
        allowed = bool(result[0])
        remaining = int(result[1])
        return allowed, remaining
```

The Lua script ensures atomicity. Without it, two concurrent requests could both read the same counter value and both be allowed past the limit.
