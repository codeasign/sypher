## Tier 1: Foundation

### Challenge 1: The Card Game

You are designing the metrics instrumentation for a real-time multiplayer card game. The game has 1 million daily active users, 10,000 concurrent players at peak, and 50 game servers. Each game server handles 200 players in separate game rooms. A player performs roughly 10 actions per second (play card, draw card, pass turn, chat emote, etc.).

**Your task**: Design the complete metrics instrumentation for this system. Specify:
1. The metrics you would emit.
2. The labels on each metric.
3. The cardinality of each metric.
4. The storage requirements for 30 days of retention.

**Stop and think before reading the model answer.**

**Model answer:**

The metrics should be designed to capture health, performance, and player experience without exploding cardinality.

**Core metrics:**

| Metric | Type | Labels | Cardinality | Purpose |
|--------|------|--------|-------------|---------|
| game_actions_total | Counter | server_id, action_type, game_mode | 50 × 30 × 5 = 7,500 | Total actions by type |
| game_action_latency_seconds | Histogram | server_id, action_type | 50 × 30 = 1,500 | Action processing time |
| active_players | Gauge | server_id | 50 | Current concurrent players |
| game_room_count | Gauge | server_id, game_mode | 50 × 5 = 250 | Active rooms per mode |
| game_errors_total | Counter | server_id, error_type | 50 × 20 = 1,000 | Error count by type |
| player_connection_events_total | Counter | server_id, event_type | 50 × 2 = 100 | Connect/disconnect events |
| match_duration_seconds | Histogram | game_mode | 5 | Match length per mode |
| server_cpu_usage | Gauge | server_id | 50 | CPU utilization |
| server_memory_usage | Gauge | server_id | 50 | Memory utilization |
| server_network_io_bytes | Counter | server_id, direction | 50 × 2 = 100 | Network bandwidth |

**Total active series**: 7,500 + 1,500 + 50 + 250 + 1,000 + 100 + 5 + 50 + 50 + 100 = 10,605 series.

**What we deliberately excluded**: player_id, user_id, room_id, and match_id as labels. These would have cardinality in the millions and would crash the metrics system. If you need per-player debugging, use logs. If you need per-match analysis, use a dedicated analytics database.

**Storage calculation**: 10,605 series, each scraped every 15 seconds, producing 10,605 / 15 = 707 samples per second. At 2 bytes per sample: 707 × 86,400 × 30 = 1.8 GB for 30 days. This fits comfortably on a single server.

**Key insight**: the metrics system is designed for operations, not analytics. The question "how many players are playing right now" is a gauge. The question "how many players played Poker last week" is an analytics query that should go to a data warehouse, not the metrics system.

### Challenge 2: The Alerting Philosophy

Your team has 15 microservices, each with 20 alerts. The total is 300 alerts. The on-call engineer receives 40 alerts per shift. Most alerts are ignored because they are not actionable. A real incident happened last week that was missed for 30 minutes because the engineer was dismissing alerts.

**Your task**: Design an alerting philosophy that reduces the alert count to a manageable level while ensuring real incidents are caught. Specify:
1. The criteria for an alert to exist.
2. The categories of alerts.
3. The process for auditing existing alerts.
4. The target number of alerts per shift.

**Stop and think before reading the model answer.**

**Model answer:**

**Criteria for an alert to exist:**
1. Every alert must have a runbook. The runbook describes exactly what to do when the alert fires: where to look, what commands to run, who to contact.
2. Every alert must have a documented threshold and the rationale for that threshold. The threshold must be based on observed normal behavior, not guesswork.
3. Every alert must have a defined severity: P0 (pages immediately), P1 (pages during business hours), P2 (ticket in the backlog), P3 (informational, no action needed).

**Categories of alerts:**

| Category | Examples | Severity | Expected count |
|----------|----------|----------|----------------|
| Service down | 5xx rate over 5 percent for 5 minutes | P0 | 1 per service |
| Latency spike | p99 over 1 second for 5 minutes | P0 | 1 per service |
| Resource exhaustion | Disk over 80 percent, memory over 90 percent | P1 | 3 per service |
| Degraded dependency | Downstream API latency high, queue depth growing | P1 | 2 per service |
| Capacity warning | Connections approaching limit, throughput approaching ceiling | P2 | 2 per service |
| Informational | Deployment complete, certificate expiring in 30 days | P3 | 2 per service |

**Total per service**: 11 alerts. For 15 services: 165 alerts. This is a 45 percent reduction from 300.

**Expected alerts per shift**: P0 alerts should fire less than once per week per service. P1 alerts should fire less than once per day. P2 alerts should fire less than once per shift. The expected alerts per shift is: 0 P0 + 2 P1 + 3 P2 = 5 alerts. This is manageable.

**Audit process:**
1. For each of the 300 existing alerts, ask: "Does this alert have a runbook?" If no, delete the alert.
2. For the remaining alerts, ask: "Has this alert fired in the last 30 days?" If no, delete the alert or reduce its severity.
3. For the remaining alerts, ask: "Did the runbook help resolve the incident?" If no, fix the runbook or delete the alert.
4. Review the alert list quarterly. Add new alerts only through the same process.

**Key insight**: alerting is a product, not a side effect. Every alert is a notification you send to a human. Treat it with the same care as a user-facing notification. If it is not valuable, do not send it.

### Challenge 3: The Capacity Planning Problem

Your company runs a video streaming service. You have 5 million daily active users. Each user streams 2 hours of video per day. The service has 50 microservices with 20 replicas each. Each microservice emits 300 metrics. The average metric has 8 labels.

**Your task**: Calculate the infrastructure requirements for the metrics system. Provide:
1. Active time series count.
2. Ingestion rate (samples per second).
3. Daily storage.
4. 30-day raw retention storage.
5. 12-month downsampled retention storage.
6. Number of servers required.
7. Estimated monthly cost.

**Stop and think before reading the model answer.**

**Model answer:**

**Active time series:**
50 services × 20 replicas × 300 metrics × 8 labels = 2,400,000 active series.

**Ingestion rate:**
2,400,000 series / 15-second scrape interval = 160,000 samples per second.

**Daily storage:**
160,000 samples/sec × 86,400 sec/day × 2 bytes/sample = 27.6 GB per day.

**30-day raw retention:**
27.6 GB × 30 = 828 GB.

**12-month downsampled retention:**
Downsampling to 5-minute resolution reduces data by 20x (300 seconds / 15 seconds).
Downsampled data: 27.6 GB / 20 = 1.38 GB per day.
12 months: 1.38 GB × 365 = 504 GB.
Downsampling to 1-hour resolution: 27.6 GB / 240 = 115 MB per day.
12 months: 115 MB × 365 = 42 GB.
Total cold storage: 504 GB + 42 GB = 546 GB (approximately 0.5 TB).

**Total storage:**
828 GB (raw) + 546 GB (downsampled) = 1.4 TB.

**Number of servers:**
Each server handles 500,000 series. 2,400,000 / 500,000 = 5 servers.
With 2x replication for high availability: 10 servers.

**Server specs:**
CPU: 8 cores, RAM: 16 GB, Disk: 500 GB SSD. Total: 10 servers.

**Estimated monthly cost:**
10 servers × $500/month = $5,000.
Cold storage in S3: 546 GB × $0.023/GB = $13/month.
Total: $5,013/month.

**As percentage of infrastructure**: if the video streaming service costs $250,000/month in infrastructure, the metrics system is 2 percent of the total. This is within the industry standard of 2 to 5 percent.

## Tier 2: Design

### Challenge 4: The Dashboard Design

Your company is launching a new social media platform. The platform has 1 million users in the first month, growing to 10 million by month 6. The platform has 10 microservices: user service, post service, feed service, notification service, search service, media service, analytics service, moderation service, payment service, and the API gateway.

**Your task**: Design a dashboard hierarchy for the platform. Specify:
1. The top-level dashboard (5 to 10 panels).
2. The drill-down dashboards (one per service, 5 to 10 panels each).
3. The key alerts for the top-level dashboard.
4. How the dashboard hierarchy helps the on-call engineer debug an incident.

**Stop and think before reading the model answer.**

**Model answer:**

**Top-level dashboard (Service Health):**

This dashboard is the first thing the on-call engineer checks. It should load in under 2 seconds.

| Panel | Metric | Purpose |
|-------|--------|---------|
| Overall error rate | sum(rate(http_5xx_total[5m])) | Overall health signal |
| p99 latency across all services | histogram_quantile(0.99, sum(rate(http_latency_seconds_bucket[5m])) by (le)) | User experience |
| Requests per second | sum(rate(http_requests_total[5m])) | Traffic volume |
| Active users | sum(active_users) | Business health |
| Service health map | 10 panels showing error rate for each service | Quick identification of failing service |
| Global error rate by service | stacked bar chart of errors per service | Compare service health |
| Infrastructure overview | CPU, memory, disk across all servers | Resource saturation |
| Alert firing count | count of active alerts | Alert noise indicator |

**Drill-down dashboard for each service (example: Feed Service):**

| Panel | Metric | Purpose |
|-------|--------|---------|
| Feed generation latency | histogram_quantile(0.99, feed_generation_seconds_bucket) | Core user experience |
| Feed request rate | rate(feed_requests_total[5m]) | Traffic volume |
| Feed error rate | rate(feed_errors_total[5m]) by (error_type) | Error breakdown |
| Feed cache hit ratio | rate(feed_cache_hits_total[5m]) / rate(feed_cache_requests_total[5m]) | Cache effectiveness |
| Database query latency | histogram_quantile(0.99, db_query_seconds_bucket) | Dependency health |
| Database connection pool | db_connections_used / db_connections_max | Saturation |
| Queue depth | feed_queue_depth | Backlog indicator |
| Consumer lag | feed_consumer_lag | Processing health |

**Key alerts:**
1. Overall error rate over 1 percent for 5 minutes. P0.
2. Any single service error rate over 5 percent for 5 minutes. P0.
3. p99 latency over 2 seconds for 5 minutes. P0.
4. Any service p99 latency over 5 seconds for 5 minutes. P0.
5. Feed service error rate over 1 percent for 5 minutes. P0 (feed is the most critical service).
6. Any service with zero requests for 5 minutes. P0 (service is dead).

**Debugging flow:**
1. On-call engineer sees the alert: "Overall error rate over 1 percent."
2. Opens the top-level dashboard. The Service Health Map shows the Post Service is red.
3. Clicks on the Post Service panel. Opens the Post Service drill-down dashboard.
4. The error rate panel shows most errors are HTTP 503 (upstream unavailable).
5. The dependency panel shows the Media Service is timing out.
6. Opens the Media Service drill-down dashboard. The disk usage panel shows 95 percent full.
7. The runbook for disk full says: "Delete old media files or increase disk. Run: cleanup_media.sh --dry-run."
8. Engineer runs the cleanup script, disk frees up, error rates return to normal.

The dashboard hierarchy guides the engineer from symptom to root cause in 8 clicks. Without it, the engineer would have to SSH into servers and run commands to find the same information.

### Challenge 5: The Migration Plan

Your company has been using a single Prometheus server for 2 years. It is now hitting its limits: 1.2 million active series, 80,000 samples per second, 30-second query latency for the last 7 days. The server has 8 GB of RAM and a 500 GB SSD. The OOM killer has killed the process twice in the last month.

**Your task**: Design a migration plan from the single Prometheus server to a scalable architecture. The plan must be zero-downtime (no gap in metrics). Specify:
1. The target architecture.
2. The migration steps in order.
3. The rollback plan.
4. How you verify the migration succeeded.

**Stop and think before reading the model answer.**

**Model answer:**

**Target architecture:**
- 3 Prometheus servers, each handling 400,000 series (hash ring sharding).
- Thanos querier for global query view.
- Thanos compactor for downsampling.
- S3 for long-term storage.
- Existing single server runs during migration, then retires.

**Migration steps:**

Week 1: Preparation
1. Deploy the 3 new Prometheus servers with empty storage.
2. Deploy the Thanos sidecar on each new server.
3. Deploy the Thanos querier and compactor.
4. Configure S3 bucket for long-term storage.
5. Test the new architecture: route one non-critical service to the new setup. Verify metrics appear correctly.

Week 2: Traffic Migration
6. Hash the existing scrape targets into 3 groups. Use consistent hashing so the hash is stable: hash(target) % 3 = shard_id.
7. Route the first group (1/3 of targets) to shard 1. The old server continues scraping everything, so there is no gap.
8. Verify: shard 1 shows the same data as the old server for the rerouted targets.
9. Wait 24 hours. Verify no data loss.
10. Route the second group to shard 2. Repeat verification.
11. Route the third group to shard 3. Repeat verification.

Week 3: Cutover
12. Configure the Thanos querier as the primary query endpoint. Point Grafana to Thanos instead of the old server.
13. Verify: all dashboards show the same data through Thanos as they did through the old server.
14. Reduce the old server's scrape interval to 60 seconds (keep it running for backup).
15. After 7 days of stable operation, stop the old server.

**Rollback plan:**
- At any point, Grafana can be pointed back to the old server. The old server is kept running and scraping everything until step 14.
- If a shard fails during migration, route its targets back to the old server.
- Keep the old server's data directory untouched for 30 days after migration.

**Verification checklist:**
- Every dashboard metric matches between old server and Thanos (within scrape interval granularity).
- Query latency under 2 seconds for last 7 days (down from 30 seconds).
- No alert fires during migration that is caused by the migration itself.
- Meta-metrics show all scrapers are healthy.
- Thanos querier returns results for cross-service queries.

**Key insight**: the migration is zero-downtime because the old server never stops scraping during the transition. The new servers start receiving traffic gradually, and the old server is a fallback at every step.

### Challenge 6: The Cost Optimization

Your company's metrics bill is $50,000 per month. The CFO wants to reduce it to $30,000 per month. The metrics system has 10 million active time series, 500,000 samples per second, and 90-day raw retention. The system uses 50 servers.

**Your task**: Propose three cost reduction strategies, each with its own savings estimate, risk assessment, and implementation timeline.

**Stop and think before reading the model answer.**

**Model answer:**

**Strategy 1: Reduce Raw Retention from 90 to 30 Days**

| Metric | Before | After | Savings |
|--------|--------|-------|---------|
| Raw retention | 90 days | 30 days | Reduces storage by 66% |
| Storage cost | $20,000/month | $6,667/month | $13,333/month |
| Servers needed | 50 | 40 | $5,000/month |
| Total savings | | | $18,333/month |

Risk: debugging incidents older than 30 days may require downsampled data or logs. Mitigation: keep 5-minute downsampled data for 1 year. The downsampled data is sufficient for most retrospective analysis.

Implementation: change the retention configuration in the time-series database. The data is deleted automatically. Timeline: 1 hour.

**Strategy 2: Optimize Cardinality by Removing Unused Labels**

Audit the 10 million active time series. Identify metrics with labels that are never used in queries, dashboards, or alerts. Common candidates: debug_id, experiment_variant, internal_version.

Assume 20 percent of time series are unused. Removing them saves 2 million series and 100,000 samples per second.

| Metric | Before | After | Savings |
|--------|--------|-------|---------|
| Active series | 10 million | 8 million | 20% reduction |
| Servers needed | 50 | 40 | $5,000/month |
| Storage cost | $20,000/month | $16,000/month | $4,000/month |
| Total savings | | | $9,000/month |

Risk: removing a label that is used in a dashboard will break the dashboard. Mitigation: audit dashboard queries first. Use a label usage analyzer to find which labels are actually queried.

Implementation: deploy a label usage analyzer, run it for 7 days, generate a report of unused labels, remove them from instrumentation code. Timeline: 2 weeks.

**Strategy 3: Increase Scrape Interval from 15 to 30 Seconds for Non-Critical Services**

Classify services into critical (latency-sensitive, user-facing) and non-critical (batch jobs, admin tools, background workers). Assume 40 percent of services are non-critical.

| Metric | Before | After | Savings |
|--------|--------|-------|---------|
| Non-critical scrape interval | 15s | 30s | 50% less data |
| Series reduction | 4 million | 2 million | 2 million fewer |
| Servers needed | 50 | 35 | $7,500/month |
| Total savings | | | $7,500/month |

Risk: non-critical services may have slower alert detection. Mitigation: keep the alerting rules on a 15-second evaluation interval even if the data is 30-second resolution. The alert might be delayed by 15 seconds, which is acceptable for non-critical services.

Implementation: update the scrape configuration for non-critical services. Timeline: 1 day.

**Combined savings**: $18,333 + $9,000 + $7,500 = $34,833/month. This exceeds the $20,000 target. The new cost is $50,000 - $34,833 = $15,167/month, which is under the $30,000 target.

**Recommendation**: implement Strategy 1 first (1 hour, immediate savings of $18,333/month). Then implement Strategy 3 (1 day, additional $7,500/month). Total savings: $25,833/month. This reaches the $20,000 target with minimal risk and effort. Strategy 2 can be implemented later for additional savings.

## Tier 3: Expert

### Challenge 7: The Multi-Region Observability Platform

You are designing the observability platform for a global fintech company. The company operates in 10 regions (US East, US West, EU West, EU Central, AP South, AP Southeast, AP Northeast, South America, Middle East, Africa). Each region has 100 microservices. The company processes 1 million transactions per day across all regions.

Requirements:
- Regulators require 2 years of metric retention for auditing.
- A transaction that starts in one region and completes in another must be traceable end-to-end.
- A regional outage must not affect metrics collection in other regions.
- The platform must support the company's SLO monitoring: 99.99 percent uptime for the payment service.
- The operations team is 5 people.

**Your task**: Design the metrics architecture for this platform. Address:
1. Regional versus global collection.
2. Retention strategy (2 years of compliance).
3. Cross-region transaction tracing.
4. SLO monitoring implementation.
5. Operational model for a 5-person team.

**Discussion:**

This is an expert-level challenge because it combines technical constraints (2-year retention, multi-region, cross-region tracing) with operational constraints (5-person team, regulatory compliance). There is no single right answer, but the best answers address all constraints.

**Regional versus global collection**: Each region runs its own metrics stack. This provides fault isolation: a regional outage does not affect other regions. The regional stack uses 3 Prometheus instances with Thanos sidecars. The Thanos querier in each region provides a local query endpoint. A global Thanos querier reads from all regional stores for cross-region queries.

**2-year retention**: Raw data is kept for 30 days on SSD. 5-minute downsampled data is kept for 6 months on HDD. 1-hour downsampled data is kept for 2 years in S3 Glacier. The 2-year data is used only for compliance audits, not for daily operations. The query path for old data is slower (10-30 seconds) but acceptable for compliance queries.

**Cross-region transaction tracing**: A trace ID is propagated through the transaction. Each service emits a metric with the trace_id label. This is a high-cardinality label, but the volume is manageable: 1 million transactions per day, each with 5 spans, produces 5 million trace ID-labeled metrics per day. With 2 bytes per sample, this is 10 MB per day — negligible. The trace ID label is dropped after 1 hour (stored in a separate high-cardinality time-series database or in logs).

**SLO monitoring**: The payment service's SLO is 99.99 percent uptime, which translates to 52.56 minutes of downtime per year. The SLO is monitored by computing the ratio of successful requests to total requests over a 30-day rolling window. The alert fires when the SLO burn rate indicates the SLO will be breached within the next 7 days. This uses the burn rate alerting method: if the error rate over the last 1 hour is 10x the allowed error rate, the alert fires immediately because the SLO will be breached in 0.1 of the window.

**Operational model for a 5-person team**: The team cannot manually operate 10 regional stacks. Automation is essential:
- Infrastructure as code: the entire metrics stack is deployed via Terraform from a single configuration. A new region is added by running terraform apply with the region name.
- Managed services: use managed Prometheus (Grafana Cloud, AWS Managed Prometheus, or GCP Cloud Monitoring) to reduce operational burden. The premium is worth the 5-person team constraint.
- Centralized configuration: a single Git repository defines all scrape targets, recording rules, and alerting rules. Changes are reviewed, approved, and deployed via CI/CD.
- On-call: one person covers the metrics platform, with a secondary from the broader infrastructure team. The on-call rotation is a 1-week primary, 1-week secondary pattern.
- Compliance reports: automated scripts generate compliance reports from the 2-year S3 data. The reports are generated on demand, not continuously.

### Challenge 8: The Streaming Metrics Pipeline

You are building a metrics pipeline for a real-time data streaming platform (like Apache Kafka or Apache Pulsar). The platform processes 100 million events per second across 1,000 nodes. Each event goes through multiple stages: produce, route, transform, consume.

**Your task**: Design the metrics pipeline for this platform. The metrics system must:
- Collect 10,000 metrics per node (10 million total).
- Support 100 millisecond scrape intervals for latency-critical metrics.
- Handle 1 million samples per second.
- Provide sub-second query latency for the last hour.
- Survive node failures without data loss.

**Discussion:**

This is an expert-level challenge because of the extreme performance requirements: 100ms scrape intervals, 1 million samples per second, and sub-second query latency are beyond the capabilities of standard Prometheus deployments.

**Architecture**: The pipeline uses a custom streaming metrics system, not a generic time-series database. The streaming system is built on top of the platform itself: metrics are published as events to a dedicated metrics topic. This leverages the platform's existing scalability and fault-tolerance.

**Collection**: Each node emits metrics as structured events to a local buffer. The buffer is flushed every 100 milliseconds to the metrics topic. The topic is configured with 100 partitions, each handled by a different broker. This gives 100x parallelism for ingestion.

**Processing**: A streaming processor reads from the metrics topic and computes: per-node aggregates (sum, avg, min, max) and global aggregates. The processor writes the aggregates to an in-memory store for the last hour (Redis or Memcached) and to a time-series database for longer retention.

**Storage**: The in-memory store keeps 1 hour of 100ms resolution data. This is 10,000 metrics per node × 1,000 nodes × 36,000 data points = 360 billion data points. At 16 bytes per point (timestamp + value), this is 5.7 TB of RAM. This is too expensive. The solution is to store only the aggregates, not every data point. Per-node aggregates are 10,000 × 1,000 = 10 million data points per 100ms window. At 16 bytes, this is 160 MB per window, 5.7 GB per hour. This is feasible with 16 GB of RAM.

**Query**: Queries for the last hour are served from the in-memory store. The query latency is under 100 milliseconds because the data is in RAM. Queries for older data go to the time-series database, which has 5-minute resolution.

**Failure tolerance**: If a node fails, the metrics topic still has the last 100ms of data. When the node restarts, it replays the metrics topic from the last committed offset. This provides exactly-once semantics for metrics: every metric observation is captured exactly once, even through node failures. The in-memory store is replicated across 3 nodes for fault tolerance.

**Scaling**: As the platform grows, the metrics topic partitions are increased. The streaming processor auto-scales to match the partition count. The in-memory store is sharded by node ID. This gives linear scalability: 2x the nodes requires 2x the partitions, 2x the processors, and 2x the memory.

### Putting It All Together

The challenges in this chapter span foundation (cardinality estimation, alerting philosophy, capacity planning), design (dashboard hierarchy, migration planning, cost optimization), and expert (multi-region observability, streaming pipelines). Each challenge tests a different skill that you need to design and operate a metrics system at scale.

The common themes across all challenges are:
- **Cardinality is the defining constraint**: every design decision flows from the cardinality budget. If you control cardinality, everything else follows.
- **Trade-offs are everywhere**: granularity versus cost, pull versus push, retention versus resolution. There is no perfect answer, only the right answer for your context.
- **Failure must be assumed**: the metrics system will fail. Plan for it, design for graceful degradation, and monitor the monitors.
- **Automation is essential**: at scale, manual operations do not work. Automate deployment, configuration, alerting, and scaling.
- **Operational maturity matters**: a technically correct system that nobody can operate is a failed system. Runbooks, on-call rotations, and escalation policies are not optional.

Use these challenges as interview practice or as design exercises with your team. The best way to learn metrics is to design a system, find its weaknesses, and redesign it. Repeat until the design holds up under every failure scenario you can imagine.
