# Durability Architecture

Durability mechanisms are rarely deployed in isolation. A production system layers multiple durability strategies — WAL at the database level, replication at the cluster level, backups at the infrastructure level, and cross-region copies at the disaster recovery level. This page shows how these layers compose into real architectures.

---

### The 3-2-1 Backup Rule in Architecture

The 3-2-1 rule is the industry standard for backup strategy:

```
3  copies of your data (primary + 2 backups)
2  different media types (SSD + tape, or SSD + cloud storage)
1  copy offsite (different region, different facility, different provider)
```

Applied to a production database architecture:

```
Copy 1 (primary): Database on enterprise NVMe SSD with RAID-10
  → This is your live, serving copy

Copy 2 (local backup): Daily pg_dump to a separate NAS/NFS volume
  → Different device, different controller, different filesystem
  → Protects against primary storage failure and filesystem corruption

Copy 3 (offsite backup): WAL archive + full backup to S3 in us-west-2
  → Different region, different cloud (or at least different AZ)
  → Protects against region-level disasters and ransomware

Media diversity:
  Copy 1: NVMe SSD (fast random access)
  Copy 2: HDD / NAS (cheap sequential storage)
  Copy 3: S3 / Glacier (cloud object storage)

Cost impact of 3-2-1 for a 1TB database:
  Primary storage (NVMe, 3x replication): ~USD 900/month
  NAS backup (10TB NAS, shared): ~USD 100/month
  S3 backup (compressed ~600GB): ~USD 14/month
  Cross-region S3 replication: ~USD 3/month
  Total additional cost over primary: ~10%
```

## Integration Patterns

### Durability with RDBMS (PostgreSQL Example)

PostgreSQL's architecture layers WAL, replication, and point-in-time recovery.

```
Data flow:
Client → SQL query → PostgreSQL backend → shared_buffers → WAL buffer
  → WAL file (pg_wal) → fsync → commit ACK to client
  → WAL sender process → streaming replication to replica
  → WAL archiver → archive_command → S3/NetApp/cloud storage

Key durability features:
- Full_page_writes = on: prevents torn pages after checkpoints
- Data_checksums = on: CRC32C on every 8KB page
- Synchronous_commit = on: no ACK until WAL fsyncs
- wal_level = replica: enough info for replication and PITR
```

**Point-in-time recovery (PITR) workflow:**

```
1. Base backup taken weekly (pg_basebackup)
2. WAL segments archived continuously (every 64MB or archive_timeout)
3. To restore to a specific moment:
   a. Restore base backup to recovery server
   b. Apply all WAL segments from archive up to target time
   c. Database is now in the exact state at that moment
4. RPO: limited by WAL archive frequency (typically under 1 minute)
5. RTO: time to restore base backup + apply WAL (typically 1-4 hours/TB)
```

### Durability with Message Queues (Kafka)

Kafka's durability model is based on its append-only commit log.

```
Kafka durability guarantees:

Producer acks setting:
  acks = 0     → fire-and-forget, data can be lost
  acks = 1     → leader wrote to its local log (default)
  acks = all   → all in-sync replicas confirmed (strongest)

Each partition has a log stored on disk:
  /data/kafka/topics/my-topic-0/
    ├── 00000000000000000000.log   (segment starting at offset 0)
    ├── 00000000000000000000.index  (offset-to-file-position mapping)
    ├── 00000000000000000000.timeindex
    ├── 00000000000000104523.log   (segment starting at offset 104523)
    ├── 00000000000000104523.index
    └── ...

Segment is flushed to disk:
  - After N messages (log.flush.interval.messages)
  - After M seconds (log.flush.interval.ms)
  - Default: no periodic flush (relies on OS page cache)
  - For production: flush every 5,000 messages OR every 1 second

Min in-sync replicas (min.insync.replicas):
  - Default: 1 (data loss if leader fails before replication)
  - Production: 2 (at least one follower must acknowledge)
  - With replication factor = 3 and min.isr = 2:
    - Losing 1 broker: no data loss
    - Losing 2 brokers: service unavailable, no data loss
```

### Durability with Object Storage (S3)

S3 achieves 99.999999999% (11 nines) durability through automated replication within a region.

```
S3 durability mechanism:

1. Object uploaded to S3 is written to multiple (minimum 3) devices
   across at least 3 availability zones within the region

2. S3 automatically detects and repairs data redundancy
   - Background integrity checks run continuously
   - Checksums verified on every read
   - Redundant copies repaired when discrepancies found

3. Versioning protects against accidental deletion:
   - Each object version is stored separately
   - Delete creates a delete marker; object data persists
   - Previous versions recoverable for any time period

4. Cross-region replication (CRR) for multi-region durability:
   - Asynchronous replication to a bucket in another region
   - RPO: typically under 15 minutes
   - Protects against region-wide outages

Copy cost: 1 copy expense = USD 0.023/GB/month
3 copies (standard) = USD 0.069/GB/month
S3 Standard = USD 0.023/GB/month includes 3 replicas
```

### Durability with Redis (In-Memory Store)

Redis offers multiple persistence options that trade durability for performance.

```
Redis persistence modes:

RDB (Redis Database file):
  - Point-in-time snapshot of all data
  - Trigger: every N seconds if M keys changed
  - Default: save 900 1 (15 min, 1 change)
              save 300 10 (5 min, 10 changes)
              save 60 10000 (1 min, 10000 changes)
  - Performance: forks the process, writes to temp file, renames
  - Loss window: up to the snapshot interval (default: minutes)
  - File size: compact binary format, typically 30-50% of dataset

AOF (Append-Only File):
  - Logs every write operation as a command
  - fsync policies:
    - always: fsync after every write (safe, 100x slower)
    - everysec: fsync once per second (default, ~1s loss)
    - no: OS flushes when it wants (up to 30s loss)
  - File size: larger than RDB, requires AOF rewrite
  - AOF rewrite: BGREWRITEAOF creates a compact AOF

RDB + AOF combined:
  - Both enabled: Redis loads AOF on restart (more complete)
  - RDB for periodic snapshots, AOF for fine-grained durability
  - Recommended for production

Production recommendation:
  - Enable AOF with fsync everysec
  - Enable RDB with save 300 100 (5 min, 100 changes)
  - Replication factor 2 (one replica)
  - For critical data: use Redis in cluster mode with AOF + RDB
```

### Durability with Distributed Databases (Cassandra)

Cassandra offers tunable consistency that directly controls durability.

```
Cassandra write durability:
┌─────────────────────────────────────────────────────────────────────┐
│  Coordinator node                                                   │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  Client writes → coordinator → memtable → commit log (fsync) │ │
│  │                           → send to replicas                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                             │                                        │
│       ┌─────────────────────┼────────────────────┐                  │
│       │                     │                    │                    │
│   ┌───┴────┐          ┌────┴───┐          ┌────┴───┐              │
│   │Replica1│          │Replica2│          │Replica3│              │
│   │Write   │          │Write   │          │Write   │              │
│   │ACK     │          │ACK     │          │ACK     │              │
│   └────────┘          └────────┘          └────────┘              │
└─────────────────────────────────────────────────────────────────────┘

Write consistency levels:
  ONE:       1 replica acknowledges → fastest, lowest durability
  QUORUM:    majority (N/2 + 1) → balanced
  ALL:       every replica → slowest, highest durability
  LOCAL_QUORUM: majority in local datacenter → multi-DC friendly

Commit log (Cassandra's WAL):
  - Written to a dedicated commit log directory
  - fsync interval: 10ms (default)
  - On crash: replay commit log to rebuild memtables
  - Truncated when memtables are flushed to SSTables
```

## Multi-Region Durability

### Active-Passive (Cold Standby)

Primary region handles all traffic. A second region has infrastructure ready but idle. Backups are continuously copied to the passive region.

```
Region 1 (us-east-1):           Region 2 (us-west-2):
┌─────────────────────────┐    ┌──────────────────────────┐
│  [App Servers] Active   │    │  [App Servers] Standing  │
│  ((Primary))            │    │  ((Standby DB))           │
│  WAL archive to S3      │───>│  WAL archive from us-east│
│  S3 bucket with          │    │  S3 bucket               │
│  cross-region replication│    │  Restored from backup   │
│                         │    │  Ready to promote        │
└─────────────────────────┘    └──────────────────────────┘

Failover process:
  1. Detect primary region outage (health check timeout)
  2. Promote standby database to primary
  3. Point DNS to region 2 load balancer
  4. Scale up app servers in region 2
  RTO: 15-30 minutes
  RPO: 0 (if using synchronous replication), 1-5 min (async WAL shipping)
```

### Active-Active (Multi-Region Writes)

Both regions accept writes concurrently. Requires conflict resolution.

```
Region 1 (us-east-1):           Region 2 (us-west-2):
┌─────────────────────────┐    ┌──────────────────────────┐
│  [App Servers]          │    │  [App Servers]           │
│  ((Primary Region 1))   │◄──►│  ((Primary Region 2))    │
│  Writes to local DB     │    │  Writes to local DB      │
│  Replicate to Region 2  │    │  Replicate to Region 1   │
│  Conflict resolution:   │    │  Conflict resolution:    │
│  last-writer-wins       │    │  last-writer-wins        │
└─────────────────────────┘    └──────────────────────────┘

Challenges:
  - Writes can conflict (same record modified in both regions)
  - Resolution strategy must be deterministic
  - CRDTs (Conflict-Free Replicated Data Types) help
  - DynamoDB Global Tables handles this automatically

Durability:
  - Each write is durable in 2 regions
  - Region failure: zero data loss (other region has complete data)
  - Application must tolerate eventual consistency
```

## Storage Sizing and Backup Planning

### Estimating Storage Requirements

```
Storage needed = data_size + index_overhead + replication_factor
               + WAL_reserve + temp_space

Example: 500GB database with 3 replicas
  data_size = 500 GB
  index_overhead = 50% → 250 GB (more if many indexes)
  data + indexes = 750 GB
  replication factor 3 = 2,250 GB (2.25 TB)
  WAL reserve = 10% = 225 GB
  temp space = 20% = 450 GB
  Total = ~2.9 TB

Backup storage:
  Weekly full backup: 750 GB (compressed, ~50% of raw size)
  Daily incremental: 10-50 GB (daily change rate)
  Retention: 4 weeks full + 12 months monthly = ~6 TB cumulative
  Cross-region copy: +100% of backup storage
```

### Backup Window Calculation

```
Database size     Backup type    Time to backup    Time to restore
───────────────────────────────────────────────────────────────────
100 GB            Full           10 min            15 min
500 GB            Full           45 min            60 min
1 TB              Full           90 min            120 min
5 TB              Full           8 hours           10 hours
10 TB             Full           16 hours          20 hours
50 TB             Full           3+ days           5+ days
100 TB            Full           1+ week           2+ weeks

For databases over 5TB, consider:
  - Physical backup (pg_basebackup) over logical (pg_dump)
  - Incremental backup to reduce backup window
  - Storage snapshot (EBS snapshots, SAN snapshots)
  - Parallel backup (multiple WAL archiving processes)
```

## Disaster Recovery Drills

A backup that has never been restored is a wish, not a recovery plan.

### Quarterly Restore Drill Process

```
Phase 1: Restore (Week 1)
  - Provision a recovery environment (same DB version, same OS)
  - Restore the most recent full backup
  - Apply WAL to a specific point in time
  - Validate row counts match production (random sampling, 10,000 rows)
  - Measure restore time

Phase 2: Application Validation (Week 2)
  - Point a staging application environment at the restored database
  - Run the application's health check suite
  - Verify all read paths return expected data
  - Verify write paths (with rollback after validation)

Phase 3: Documentation Update (Week 3)
  - Update runbook with any new steps discovered
  - Fix any discrepancy between documented and actual procedure
  - Review restore time against RTO target
  - If RTO target was missed: identify bottleneck and fix

Phase 4: Report (End of Quarter)
  - Summary of drill results
  - RPO and RTO metrics vs targets
  - Action items for improvement
  - Escalate if targets were missed by more than 50%
```

### Common Restore Problems Found in Drills

```
Problem                    Frequency   Impact
─────────────────────────────────────────────────────────
Backup file corrupt        5-10%      Restore fails, need older backup
Missing WAL segment        2-5%       PITR fails, must use full backup date
Incompatible DB version    3-8%       Restored DB won't accept application
Permission issues          15-20%     Application can't connect to restored DB
Network connectivity       5-10%      Restored DB in wrong VPC/network
Insufficient storage       10-15%     Recovery environment too small
Missing configuration      8-12%      Restored DB uses default settings
```

These numbers are why drills matter. A 5-10% backup corruption rate means you cannot afford to discover corruption during an actual disaster.

## Deployment and Operations

### Automated Backup Script (Conceptual)

```
Backup pipeline (cron-driven, daily at 2:00 AM):

1. Check replication lag → if > 10 seconds, wait
2. Take pg_basebackup to temporary directory
3. Compress with pigz (parallel gzip, 4 threads)
4. Upload to S3 with server-side encryption
5. Verify checksum of uploaded file
6. Register backup metadata in backup catalog table
   (backup_id, timestamp, size, checksum, WAL_end_lsn)
7. Delete local temporary files
8. Alert on any step failure (PagerDuty, OpsGenie)

WAL archiving (continuous, every 60 seconds):
1. Archive any WAL segment older than 5 minutes
2. Copy to S3 path: /backups/wal/{YYYY}/{MM}/{DD}/{segment_name}
3. Delete local copy after confirmed upload
4. Retention: 7 days on S3 Standard, then move to Glacier
5. Glacier retention: 1 year, then delete
```
