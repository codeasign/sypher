# Durability Concepts

## The Photo That Vanished

Maya ran a photo-sharing app called Snapsquare. Users uploaded 50 million photos a month. The team stored every file on a single RAID-5 array attached to a file server. It had worked for three years. Nobody thought about it.

One Tuesday at 2:14 AM, two disks in the RAID array failed simultaneously. RAID-5 only tolerates one. The array went offline. The file server could not mount the volume. Fifty million photos — family vacations, wedding albums, pet memorials, baby pictures — became inaccessible. The restore from tape backup started at 3:00 AM. The tape drive had a read error at 3:47 AM. The backup was corrupt.

Maya spent the next 72 hours on the phone with a data recovery company. The cost: USD 180,000 for a partial recovery. User trust: gone. Snapsquare shut down within six months.

Maya's mistake was not the RAID level. It was believing that storing data once was enough.

## The Committed Transaction That Never Was

A financial trading platform processed 200,000 transactions per day. Each trade was a database transaction: BEGIN, debit account A, credit account B, COMMIT. The application called COMMIT and immediately sent a confirmation email to the user.

One afternoon, the database server's power supply failed during a COMMIT. The application received an error — ambiguous, could mean committed or not. The code assumed failure and retried. Both the original and the retry appeared committed after the server restarted. The same trade executed twice. The firm lost USD 40,000 before someone caught the discrepancy during end-of-day reconciliation.

The root cause: the application treated "COMMIT response received" as the durable write signal. In reality, the write-ahead log had flushed to disk before the crash, but the response never made it back over the network. The write was durable. The acknowledgment was not.

## The SaaS Customer Data Lawsuit

A SaaS company storing medical appointment data for 2,000 clinics experienced a storage controller firmware bug that silently zeroed out 15% of their database pages over six months. Because only the primary database had checksums enabled — replicas were streaming unfiltered — the corruption silently propagated to every copy. By the time the bug was discovered, the company had overwritten all valid copies with corrupted data.

The result: a class-action lawsuit from affected clinics, a USD 4.2 million settlement, and eighteen months of regulatory audits. The root cause was not the firmware bug itself — firmware bugs happen. The root cause was the absence of durability verification. No checksums. No scrub jobs. No restore tests.

## What Durability Actually Means

Durability is the guarantee that once a write is acknowledged to the client, the data will persist permanently and survive any subsequent failure — process crash, power loss, disk failure, network partition, or natural disaster.

### The Tattoo Analogy

A whiteboard marker on a glass board: information lasts until someone erases it. A pencil on paper: information lasts until the paper is lost. A tattoo on skin: information lasts through washing, swimming, and decades of wear.

A durable database is a tattoo. A non-durable cache is a whiteboard marker.

### The Fireproof Safe Analogy

You keep important documents — passport, deed, birth certificate — in a fireproof safe. The safe protects against fire, flood, and theft. You might keep copies in a bank safety deposit box in another city. That is multi-region durability.

A cardboard box in your garage is the equivalent of storing data on a single disk with no backups. It works until it does not.

## Core Vocabulary

**Durability**: The property that committed data survives failures permanently.

**Persistence**: The mechanism of storing data on non-volatile media (disk, SSD, tape) so it survives power loss.

**Write-Ahead Log (WAL)**: A sequential log of all changes written to disk *before* the database applies them. The canonical structure for durable writes in almost every database.

**Commit Log**: The segment of the WAL that records transactions as committed. Kafka, PostgreSQL, MySQL, SQLite, and most databases use a commit log.

**fsync**: The system call that forces the operating system to flush a file's data from the page cache to the physical storage device. Without fsync, a power loss can lose data the OS thinks it has written.

**Durable Write**: A write that has reached non-volatile storage and will survive a power cycle.

**Replication Factor**: The number of copies of each piece of data maintained across different nodes. A replication factor of 3 means three copies.

**Replication**: Copying data from one node to another to protect against node failure.

**Snapshot**: A point-in-time copy of the entire database state, used for recovery and backups.

**Backup**: A copy of data stored separately from the primary system, used for disaster recovery.

**Disaster Recovery (DR)**: The process of restoring system functionality after a catastrophic failure, using backups and replicated data.

**Checksum**: A cryptographic hash of a data block used to detect corruption. Common checksums: CRC32, SHA-256, xxHash.

**Bit Rot**: The gradual physical degradation of storage media that causes individual bits to flip. Also called data decay.

**WAL (Write-Ahead Log)**: An append-only log file that records every mutation before it is applied to the main data structures. Essential for crash recovery.

**Append-Only Log**: A data structure where entries are only added, never modified or deleted in place. The foundation of WALs, Kafka, and event sourcing.

**Torn Page**: A partial write where only some bytes of a disk page make it to storage, leaving the page in an inconsistent state.

**Double-Write Buffer**: A technique used by InnoDB (MySQL) to prevent torn pages by writing a copy of the page before writing in place.

**Quorum Write**: A write that must be acknowledged by a majority of replicas before being considered successful. Used in distributed consensus systems.

The flow has four critical phases:

1. **Application receives write** — the app server receives an HTTP request with data to persist.
2. **WAL write with fsync** — the database appends the change to the WAL and calls fsync() to ensure it reaches physical media.
3. **Acknowledgment to client** — only after the WAL write is confirmed on stable storage does the server acknowledge success.
4. **Replication to other nodes** — the change is sent to replicas asynchronously (or synchronously, depending on configuration).

Each phase is a potential durability failure point.

### Additional Key Terminology

These terms extend the core vocabulary with concepts needed for deeper durability discussions.

**fsync barrier**: A point in the write path where all previously issued writes must be on stable storage before proceeding. Used by databases to enforce ordering guarantees.

**Group commit**: The technique of batching multiple transactions into a single fsync call. Without group commit, each transaction pays the full fsync latency cost. With group commit, N transactions share one fsync.

**WAL archiving**: The process of copying completed WAL segments to a remote, durable location (S3, NFS, tape) before they are recycled. Enables point-in-time recovery to any second within the archive window.

**Replication slot**: A PostgreSQL mechanism that prevents the primary from deleting WAL segments until all consuming replicas have confirmed receipt. Prevents data loss from premature WAL recycling.

**Hinted handoff**: A Cassandra durability mechanism where a coordinator node stores writes intended for an unavailable replica and delivers them when the replica recovers. Provides eventual durability for temporarily unavailable nodes.

**Anti-entropy repair**: A background process in distributed databases (Cassandra, DynamoDB) that compares data across replicas and reconciles differences. Runs periodically to fix divergence from missed writes or failed repairs.

**Snapshot isolation**: A concurrency control mechanism where each transaction sees a consistent snapshot of the database as of its start time. Durability ensures that once committed, the snapshot is permanent.

**Checkpoint**: A database event that ensures all dirty pages (pages modified in memory but not yet written to disk) are flushed to the data files. Reduces crash recovery time by limiting how far back the WAL must be scanned.

**Fuzzy checkpoint**: A checkpoint that writes dirty pages incrementally over time rather than all at once. Reduces I/O spikes. PostgreSQL uses fuzzy checkpoints.

**Redo point**: The oldest LSN that might need to be reapplied during crash recovery. Determined by the last checkpoint's position in the WAL.

**Compensation Log Record (CLR)**: A WAL record that describes an undo operation. If recovery itself crashes during UNDO, the CLR ensures that already-undone operations are not redone.

## Step by Step: The Path of a Durable Write

### Step 1: Client Issues Write

Your application calls `db.execute("INSERT INTO photos VALUES (...), (...)")`. This frames an SQL statement and sends it over a TCP connection.

### Step 2: Database Receives and Parses

The database server receives the TCP packet, parses the SQL, checks permissions, and determines which tables and indexes need updating.

### Step 3: WAL Append

Before modifying the actual data pages, the database appends a record to the write-ahead log. The record contains enough information to redo the change during crash recovery.

### Step 4: fsync() — The Moment of Truth

The database calls `fsync()` on the WAL file descriptor. This is the critical moment. The operating system has been buffering writes in the page cache. If power fails before those cached pages reach disk, the data is lost.

`fsync()` blocks until the storage device confirms the data is on non-volatile media. On an SSD with proper power-loss protection, this takes 1-5 milliseconds. On a spinning disk with a write cache that reports "written" before data actually lands on platters, fsync can report success falsely.

### Step 5: Data Page Modification

With the WAL entry safely on disk, the database modifies the actual data pages in memory. These pages are now "dirty" — they differ from what is on disk. The checkpoint process will eventually write them to the main data files.

### Step 6: Transaction Commit

The database writes a COMMIT record to the WAL and fsyncs again (or relies on the same fsync if the commit is part of the same WAL flush). Only then does it send the success response to the client.

### Step 7: Acknowledgment to Client

The database sends the success response through the network stack. Note a subtle point: the network send itself can fail. The write is durable on the server, but the client may never learn about it. This is the exact scenario from the second story — a committed write with a lost acknowledgment.

Idempotency keys solve this. The client includes a unique request ID. If the acknowledgment is lost, the client retries with the same ID. The database detects the duplicate and returns the stored result without applying the change again.

The WAL entry is shipped to replica databases asynchronously or synchronously.

### Practical Example: PostgreSQL vs MySQL Commit Behavior

The same transaction behaves differently in different databases regarding fsync timing.

```
PostgreSQL 16, synchronous_commit = on:
  BEGIN;
  INSERT INTO orders VALUES (1, 'item', 29.99);
    -- WAL record written to buffer (memory)
  COMMIT;
    -- WAL buffer flushed to WAL file
    -- fsync() called on WAL file
    -- Block until storage confirms (1-5ms)
    -- ACK returned to client
  -- Data is durable. Crash recovery will replay this INSERT.

MySQL 8, innodb_flush_log_at_trx_commit = 1:
  BEGIN;
  INSERT INTO orders VALUES (1, 'item', 29.99);
    -- Redo log entry written to log buffer
  COMMIT;
    -- Log buffer flushed to redo log file
    -- fsync() called on redo log file
    -- Block until storage confirms (1-5ms)
    -- ACK returned to client
  -- Data is durable. Crash recovery will replay this INSERT.

MySQL 8, innodb_flush_log_at_trx_commit = 2:
  COMMIT;
    -- Log buffer written to redo log file (write(), not fsync)
    -- Returns immediately (0.05ms)
    -- OS will flush within its default window (typically 30s)
    -- ACK returned to client immediately
  -- If power fails within 30 seconds, this INSERT may be lost.

MySQL 8, innodb_flush_log_at_trx_commit = 0:
  COMMIT;
    -- Log buffer written to redo log file... eventually
    -- No write() called at COMMIT time
    -- OS page cache flush happens at OS's discretion
    -- ACK returned to client before any I/O occurs
  -- Up to 1 second of data loss (or more if OS is busy)
```

The default settings (PostgreSQL `on`, MySQL `1`) provide the strongest durability. The other settings are for workloads where speed matters more than absolute safety.

### Practical Example: Python Code for a Durable Write

```python
import os
import struct
import hashlib

class DurableWAL:
    """Minimal write-ahead log demonstrating fsync patterns."""

    def __init__(self, path: str):
        self.path = path
        self.fd = os.open(path, os.O_WRONLY | os.O_CREAT | os.O_APPEND)
        self.lsn = os.path.getsize(path)  # Log Sequence Number

    def append(self, record: bytes) -> int:
        """Append a record to the WAL and fsync. Returns LSN."""
        # Record format: [4-byte length][payload][32-byte SHA-256 checksum]
        checksum = hashlib.sha256(record).digest()
        payload = struct.pack("I", len(record)) + record + checksum
        os.write(self.fd, payload)           # Write to OS page cache
        os.fsync(self.fd)                    # Force to disk (blocking)
        self.lsn += len(payload)
        return self.lsn

    def crash_recover(self) -> list[bytes]:
        """Read all valid records from WAL after a crash."""
        recovered = []
        with open(self.path, "rb") as f:
            while True:
                header = f.read(4)
                if not header or len(header) < 4:
                    break
                rec_len = struct.unpack("I", header)[0]
                record = f.read(rec_len)
                stored_checksum = f.read(32)
                if len(record) != rec_len or len(stored_checksum) under 32:
                    break  # Partial write at end of WAL — ignore
                computed = hashlib.sha256(record).digest()
                if computed == stored_checksum:
                    recovered.append(record)
                else:
                    break  # Corrupt record — stop here
        return recovered

    def close(self):
        os.close(self.fd)

# Usage:
wal = DurableWAL("data.wal")
wal.append(b"INSERT INTO users VALUES (1, 'alice')")
wal.append(b"INSERT INTO users VALUES (2, 'bob')")
wal.close()

# After crash:
wal2 = DurableWAL("data.wal")
records = wal2.crash_recover()
print(f"Recovered {len(records)} records")
```

## Durability at Three Scales

### Single-Node Durability

On a single machine, durability comes from:

- **WAL**: Sequential write-ahead logging ensures no transaction is lost on crash.
- **fsync**: Forces OS buffer cache to flush to physical media.
- **Battery-backed write cache**: RAID controllers and some SSDs have capacitors that power the cache during the few milliseconds needed to flush pending writes during a power outage.
- **Double-write buffer**: InnoDB's mechanism to prevent torn pages — write a full 16KB page to a scratch file, then write it in place. If the in-place write is interrupted, the scratch copy saves the day.

**Limitations**: Single-node cannot survive beyond a single hardware failure. Disk failure, motherboard failure, or catastrophic OS corruption all cause data loss.

### Distributed Durability

Multiple nodes coordinate to ensure data survives individual node failures.

- **Replication factor**: Each piece of data lives on N nodes (typically 3).
- **Quorum writes**: A write is acknowledged only when W replicas confirm (e.g., 2 out of 3).
- **Synchronous replication**: The primary waits for at least one replica to confirm the write is durable before acknowledging the client. Adds latency but guarantees multiple copies.
- **Majority commits** (Paxos/Raft): A transaction commits only when a majority of nodes have durably stored it.

**Tradeoff**: Distributed durability adds latency — a synchronous write to 3 nodes across a datacenter takes 5-15ms compared to 1-5ms for a local write.

### Archival Durability

The deepest layer. Cold storage for worst-case scenarios.

- **Full backups**: Complete snapshot of the database, taken daily or weekly.
- **Incremental backups**: Changes since the last full backup, taken hourly.
- **Backup verification**: Automated restore tests that validate backup integrity.
- **Cross-region storage**: Backups stored in a different geographic region.
- **Glacier/Deep Archive**: Very cheap storage (USD 1/TB/month) for data that may never be needed but must be kept for compliance.

## Mental Model Check

1. A database fsyncs every WAL write. The server crashes. When it restarts, will all committed transactions be present, even those that had not been written to the main data files yet? How does the database distinguish committed transactions from rolled-back ones during crash recovery?

2. You configure PostgreSQL with `synchronous_commit = off`. Writes return instantly without waiting for WAL flush. How long after a power loss could you lose data? What does the recovery window look like?

3. A RAID-5 array with six disks experiences one disk failure. During the rebuild, a second disk develops uncorrectable read errors. What happens to the data? How would RAID-6 or RAID-10 change the outcome?

4. An object storage system (S3-compatible) stores 100 million objects with 11 nines durability. How many objects are expected to be lost per year? If an object is lost, what mechanisms detect the loss and recover it?

5. Your database replication has a replication lag of 30 seconds. The primary's disk fails completely. How many transactions are lost? How would you measure this precisely, and what configuration changes would reduce the loss window?

### Common Misconceptions

**"fsync guarantees data safety."** fsync guarantees the OS has flushed data to the storage device. It does not guarantee the storage device has written to non-volatile media. Consumer SSDs with DRAM write caches can report fsync success before data is on NAND. Enterprise SSDs with power-loss protection (capacitors) make fsync trustworthy.

**"RAID is a backup."** RAID protects against disk failure by reconstructing data from parity or mirrors. It does not protect against human error (DROP TABLE), software bugs (corrupt writes), ransomware (encrypted files), or site disasters (fire, flood). RAID is redundancy, not backup.

**"Three replicas means no data loss."** Three replicas with asynchronous replication still have a loss window equal to the replication lag. If the primary fails before replicating the latest writes, those writes are lost even though two other copies exist. Only synchronous replication prevents this.

**"Backup success = backup can be restored."** A backup process can complete successfully and produce a file that is internally corrupt. Without automated restore testing, you will not discover the corruption until you need the backup. Regular restore drills are the only verification.

**"WAL protects against all crashes."** WAL protects against process crash and power loss. It does not protect against disk failure (if the WAL is on the same disk as data), firmware bugs that corrupt writes before they reach the WAL, or operator errors that delete the WAL directory.

### Real-World Durability Incident Timeline

Understanding how durability failures unfold helps you design better defenses.

```
Example timeline of a preventable durability failure:

Day 1: Engineer configures PostgreSQL with synchronous_commit = on
        and one async replica. No backup strategy documented.
        WAL archiving not configured.

Month 6: Database reaches 500GB. Checkpoints take 3 minutes.
         WAL segments fill 1GB/hour.

Month 9: Replica falls behind by 45 seconds during peak hours.
         No alert configured for replication lag.

Month 12: Primary disk starts producing SMART reallocated sector
          errors. No alert configured for SMART metrics.

Month 13: Power supply on primary server fails at 3:14 AM.
          Database process stops. WAL buffer has 200ms of
          unflushed writes. These writes are lost.

          Failover to replica. 45 seconds of committed transactions
          are missing from the replica (async replication lag).

          Total data loss: 45.2 seconds of transactions.
          Affected records: ~2,000 customer orders.
          Recovery: restore from streaming backup... wait,
          no backup strategy was configured.

          Outcome: 2,000 orders permanently lost.
          Company compensates affected customers: USD 160,000.
          Engineering team spends 3 weeks building backup system
          after the incident.

          Root cause: async replication without backup safety net.
          Second cause: no replication lag monitoring.
          Third cause: no hardware health monitoring.
          Fourth cause: no documented backup strategy.

Prevention at each layer:
  Layer 1: Synchronous replication eliminates replication lag gap
  Layer 2: WAL archiving provides point-in-time recovery
  Layer 3: Regular backups provide safety net
  Layer 4: Monitoring alerts on replication lag and disk health
  Layer 5: Regular restore drills verify the safety net works
```

These cases bridge from the concrete mechanisms of durable writes to the architectural decisions that determine how much data loss a system can tolerate. The next page dives into the internal mechanisms — WAL internals, checksum trees, and crash recovery algorithms.

| Term | Applies To | Key Metric |

### Durability vs Availability: Understanding the Difference

Durability and availability are closely related but distinct concepts that are frequently confused.

```
Availability: "Is the data accessible right now?"
Durability:   "Will the data survive a failure?"

A system can have:
  - High availability + low durability:
    Database with 3 replicas but async replication.
    Data is always accessible (high availability).
    If primary fails, the last 2 seconds of writes are lost (low durability).

  - Low availability + high durability:
    Database on a single server with nightly backup to S3.
    If the server fails, data is unavailable until restored (low availability).
    Once restored, all data including the last backup is present (high durability).

  - Both high:
    3 replicas with synchronous replication + daily backups + WAL archiving.
    Data is always accessible. No data loss on any single failure.

  - Both low:
    Single database server, no backups, no replication.
    Data is unavailable during the failure and may be permanently lost.
```

In system design interviews, always clarify which property the requirement is about. A "five nines" requirement usually means availability, not durability.

### How to Explain Durability to Non-Engineers

When discussing durability with product managers or executives, use concrete analogies:

"This database has the durability equivalent of a bank vault — data is written to three separate locations in different buildings before we tell you it is saved."

"Our current setup is like writing in pencil. The data exists, but it can be erased. We need to upgrade to a carving-in-stone approach for customer records."

"Every time someone saves a document, we immediately make two photocopies and store them in different cities. That is synchronous replication."

These analogies help non-technical stakeholders understand why durability investments matter without needing to know about fsync or WAL internals.
