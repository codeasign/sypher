# Durability Deep Dive

Durability seems simple — write data to disk and you are done. The reality involves operating system buffering races, storage controller behavior variances, atomic page write guarantees, and complex crash recovery algorithms. This page covers the internals that determine whether your data survives when things break.

---

## WAL Internals: The Heart of Crash Recovery

Every major database that cares about durability uses a write-ahead log. Before any change touches the data files, the change is appended to a sequential log on stable storage. If the server crashes, the database replays the WAL during recovery to reconstruct any committed transactions that had not yet been flushed to the data files.

### WAL Record Structure

A WAL record typically contains:

```
WAL Record:
┌─────────────────────────────────────────┐
│ LSN (8 bytes)          Log Sequence #   │
│ Prev LSN (8 bytes)     Previous record  │
│ Transaction ID (4 bytes)                │
│ Resource Manager ID (2 bytes)           │
│ Operation Code (2 bytes)                │
│ Payload Length (4 bytes)                │
│ ┌─────────────────────────────────┐     │
│ │ Payload:                        │     │
│ │ - Page ID (table + page number) │     │
│ │ - Redo image (bytes to reapply) │     │
│ │ - Undo image (bytes to revert)  │     │
│ │ - Checksum (CRC32 of payload)   │     │
│ └─────────────────────────────────┘     │
│ Checksum (CRC32 of entire record)       │
└─────────────────────────────────────────┘
Total overhead: ~40 bytes per record
Payload: scales with operation size
```

### PostgreSQL WAL vs MySQL InnoDB Redo Log

Two dominant WAL implementations exist in the open-source world. They solve the same problem differently.

**PostgreSQL WAL:**
- Default segment size: 16MB
- Written in 8KB blocks (matching data page size)
- Each WAL record includes full page images for the first modification of a page after a checkpoint (full-page writes)
- Replication slots prevent WAL deletion until all replicas have consumed the data
- `wal_level = replica` (minimal) or `logical` (for change data capture)

**MySQL InnoDB Redo Log:**
- Circular log with two or more files (default: `ib_logfile0`, `ib_logfile1`)
- Default size: 48MB per file (MySQL 8 defaults to 512MB each)
- Written in 512-byte blocks (matching disk sector size)
- Uses mini-transactions (mtr) within InnoDB for atomic page writes
- `innodb_flush_log_at_trx_commit` controls fsync behavior:
  - 1 (default): fsync every commit — safest, 50-200 writes/sec per connection
  - 2: fsync once per second — slight durability gap, 10x more throughput
  - 0: no fsync — OS flushes when it wants, up to 1 second of data loss

### WAL Sizing Impact

WAL size affects three things: recovery time, write throughput on checkpoints, and disk space usage.

```
WAL size        Recovery time   Max throughput   Disk usage
────────────────────────────────────────────────────────────────
64MB            1-3 seconds     10,000 writes/s  128MB
512MB           5-15 seconds    50,000 writes/s  1GB
4GB             1-3 minutes     200,000 writes/s 8GB
16GB            5-15 minutes    500,000 writes/s 32GB

Rule of thumb: WAL size should be 10-30 minutes of write capacity.
Larger WAL = faster checkpoints, slower crash recovery.
```

## fsync: The System Call Everyone Depends On

fsync is the operating system primitive that guarantees data has reached physical storage. Your database's durability is only as trustworthy as fsync's implementation on your hardware.

### The Buffer Cache Problem

When you call `write()`, the operating system does not immediately write to disk. It copies data into the page cache — a region of RAM managed by the kernel. Writes to the page cache complete in microseconds. The kernel flushes dirty pages to disk asynchronously, usually within 30 seconds by default.

If a power failure occurs before the kernel flushes, those writes vanish. The application called write(), got a success return code, and the data never touched a disk platter or NAND cell.

```python
# Unsafe: write only -- data may live in OS cache
with open("data.db", "ab") as f:
    f.write(record)       # Returns instantly, data in page cache
    # Power loss here loses the write
    f.close()             # Flushes to disk (but may return before completion)
```

```python
# Durable: write + fsync
with open("wal.log", "ab") as f:
    f.write(record)       # Copies to page cache
    os.fsync(f.fileno())  # BLOCKS until storage confirms durable write
    # Power loss here is safe -- data is on physical media
```

### Hardware Lies: Battery-Backed Caches

Many storage devices have write-back caches that report "written" to the OS before data is actually on persistent media. The OS calls fsync, the controller says "done," and the data sits in volatile controller RAM. Power fails. Data vanishes.

The fix: battery-backed write cache (BBWC) or capacitor-backed cache. These components have enough stored energy to flush the cache to persistent storage during a power loss. Enterprise RAID controllers (LSI, Adaptec) and enterprise SSDs (Intel Optane, Samsung PM-series) have this protection. Consumer SSDs may not.

### fsync Performance Impact

fsync is expensive because it forces the storage device to drain its write cache and confirm persistence.

```
Operation           Latency
──────────────────────────────────
write() to page cache    1-5 microseconds (no disk I/O)
fsync() on HDD           5-15 milliseconds
fsync() on SSD           0.5-5 milliseconds
fsync() on NVMe          0.1-1 milliseconds

Ratio: fsync is 100-10,000x slower than write()
```

Databases batch fsync calls to amortize the cost. Instead of fsyncing each transaction individually, they accumulate multiple WAL writes and fsync once per batch.

### Group Commit

Group commit is the optimization where the database collects multiple transactions that arrive between fsync calls and flushes them together in one fsync.

```
Without Group Commit:
  T1: write WAL -> fsync (5ms) -> commit
  T2: write WAL -> fsync (5ms) -> commit
  T3: write WAL -> fsync (5ms) -> commit
  Total: 15ms for 3 transactions (5 tx/sec)

With Group Commit:
  T1 arrives: write WAL -> wait for group
  T2 arrives: write WAL -> wait for group
  T3 arrives: write WAL -> wait for group
                fsync (5ms) for all 3 -> T1,T2,T3 commit together
  Total: 5ms for 3 transactions (600 tx/sec)
```

Group commit increases throughput by 100x at the cost of slightly delaying each transaction (the wait time for the group to form, typically sub-millisecond).

## Crash Recovery Algorithms

When a database restarts after a crash, it must determine which transactions committed and which did not. The WAL makes this possible.

### ARIES Recovery Algorithm

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is the most widely used crash recovery algorithm. Used by PostgreSQL, IBM DB2, Microsoft SQL Server, and many others.

**Three phases:**

**Phase 1 — Analysis:** Scan the WAL from the last checkpoint to build a table of active transactions at the time of crash. Determine the redo point (oldest LSN that might need redo).

**Phase 2 — REDO:** Replay all changes from the redo point forward. Apply every operation that was in a committed or preparing transaction. This brings the database to the exact state it was at the moment of the crash.

**Phase 3 — UNDO:** Roll back all transactions that were active but not committed at the time of the crash. The undo information is stored in the WAL records themselves.

### LSM-Tree Crash Recovery

Log-Structured Merge (LSM) trees, used by LevelDB, RocksDB, Cassandra, and ScyllaDB, have a different recovery model.

The LSM tree is an append-only structure. Writes go into a memtable (in-memory sorted structure), which is periodically flushed to disk as an SSTable (Sorted String Table) file. The write-ahead log protects the memtable.

```
LSM Crash Recovery:
1. On startup, scan WAL directory for log files
2. Read WAL entries from the last known checkpoint
3. Rebuild memtable by replaying WAL entries
4. Discard WAL entries older than the oldest live SSTable

Key difference from ARIES: LSM recovery only replays the
in-memory state. On-disk SSTables are immutable and never
need redo. Recovery is faster: typically under 1 second.
```

### Advanced WAL Features

**WAL Compression**: PostgreSQL 15+ supports WAL compression using LZ4 or ZSTD. Compressed WAL reduces disk I/O by 30-50% on write-heavy workloads. Tradeoff: 5-10% additional CPU per WAL write.

```
Without compression: 100GB WAL in 30 minutes (3.3GB/min write rate)
With ZSTD (level 1): 70GB WAL in 28 minutes (2.5GB/min + CPU overhead)
Recovery time: compressed WAL requires decompression during recovery
  PostgreSQL: decompression adds ~10% recovery time
```

**WAL Encryption**: PostgreSQL supports Transparent Data Encryption (TDE) for WAL in the enterprise edition. Each WAL record is encrypted before being written to disk. Decryption happens during recovery.

**WAL Parallelism**: PostgreSQL 15+ supports parallel WAL writes. Multiple WAL writer processes can write to separate WAL buffers, reducing contention on systems with high write concurrency.

## Consistency During Crashes: Torn Pages and Partial Writes

A disk page is typically 4KB, 8KB, or 16KB. Disk sectors are 512 bytes or 4KB (advanced format). When the database writes a 16KB page and the power fails after only 4KB has been written, the page contains a mix of old and new data. This is a torn page.

### Torn Page Consequences

A torn page can corrupt the database's internal structures:
- An index page might have a half-updated pointer that references a non-existent child page
- A table page might have a partially applied row update, with the row length indicating more data than actually exists
- A page checksum computed against complete data will fail on torn data

### Preventing Torn Pages

**Method 1 — Double-Write Buffer (InnoDB):**

Before writing a data page in place, InnoDB writes a full copy to a double-write buffer on disk. The buffer is a dedicated area of 128 pages (2MB). If the in-place write is interrupted, InnoDB recovers using the intact copy in the double-write buffer.

```
Write flow with double-write buffer:
1. Copy dirty page to double-write buffer (sequential write)
2. fsync double-write buffer
3. Write page to actual location (random write)
4. fsync actual location

On crash during step 3:
- Double-write buffer has the complete page
- Recovery copies it from buffer to the correct location
- Page is restored to its consistent state
```

**Method 2 — Full Page Writes (PostgreSQL):**

After every checkpoint, the first modification to each page writes the entire page image to the WAL. If a page is torn during the checkpoint, the WAL contains the complete page for recovery to reapply.

**Method 3 — Atomic Writes (modern storage):**

Some storage systems support atomic write operations — a write of any size either completely succeeds or completely fails. NVMe with atomic write unit (AWU) support can make 16KB page writes atomic natively, eliminating torn pages.

## Checksum Trees: Detecting Silent Corruption

Writing data is only half the durability equation. The other half is detecting when stored data has silently changed. A disk can return corrupted data without reporting any error. This is called silent data corruption.

### Checksum Strategies

**Page-level checksums (PostgreSQL):** Every 8KB data page ends with a checksum. When the page is read, the checksum is verified. If it does not match, a page corruption is detected.

```
PostgreSQL checksum:
- CRC32C (Castagnoli) of page contents, stored in page header
- Computed on write, verified on read
- Impact: ~3% CPU overhead for reads and writes
- Detection: database logs error, returns error to query
```

**File-level checksums (ZFS):** The filesystem stores checksums separately from data, in a Merkle tree. Each block's checksum is stored in its parent block pointer, all the way up to the root.

**Application-level checksums:** The application computes and stores a checksum alongside each record. When the record is read, the checksum is verified. This catches corruption anywhere in the stack — disk, controller, cable, OS, library.

### Bit Rot and Scrubbing

Bit rot (also called data decay) is the gradual change of stored bits due to physical media degradation. On a hard disk, magnetic domains lose their orientation over years. On NAND flash, electrons leak from floating gates. Each technology has a characteristic bit error rate.

```
Media type       Bit error rate (BER)   Uncorrectable after
────────────────────────────────────────────────────────────────
Consumer HDD     1 in 10^14 bits read   ~12 TB read
Enterprise HDD   1 in 10^15 bits read   ~125 TB read
Consumer SSD     1 in 10^15 bits read   ~125 TB read
Enterprise SSD   1 in 10^16 bits read   ~1.25 PB read
NVDIMM           1 in 10^18 bits read   ~125 PB read

For a 10TB database read once per week:
- Consumer HDD: 1 error every 12 weeks
- Enterprise SSD: 1 error every 125 weeks
- With checksums: errors are detected and corrected
- Without checksums: errors silently produce wrong query results
```

Scrubbing (also called patrol read or data integrity scan) reads all data periodically and verifies checksums. This detects bit rot before the corrupted data is read during normal operation.

```
Scrubbing strategy:
- Read every data page once per N days
- Verify checksum on each page
- If checksum fails: repair from replica or backup
- Log any unrecoverable pages for operator attention

Typical configuration:
- Scrubbing interval: 7-30 days
- I/O throttle: 20% of available disk bandwidth
- Priority: low, does not impact production performance
```

## Performance: Sync vs Async Writes

The fundamental tension in durability is speed vs safety. Every durability mechanism costs latency and throughput.

### Configurable Durability Levels

Most databases let you tune durability at the session or transaction level.

```
PostgreSQL synchronous_commit:
```

### Batch Writes and Throughput

Batching multiple writes into one fsync dramatically improves throughput.

```
Write rate (single connection, synchronous commit):
────────────────────────────────────────────────────
Individual writes:   200 writes/sec
Batch of 10:       1,800 writes/sec  (9x improvement)
Batch of 100:     10,000 writes/sec  (50x improvement)
Batch of 1000:    20,000 writes/sec  (100x improvement)

Diminishing returns beyond batch size ~500 due to
WAL write contention and memory pressure.
```

## Configuration Reference

### PostgreSQL Durability Settings

```
# postgresql.conf — durability-related settings

# Flush WAL to disk before acknowledging commit
synchronous_commit = on          # on, remote_write, remote_apply, off

# WAL segment size (affects recovery time)
wal_segment_size = 16MB          # 1MB, 4MB, 8MB, 16MB, 32MB, 64MB

# Number of WAL buffers (reduce WAL write contention)
wal_buffers = 16MB               # default: -1 (auto, 1/32 of shared_buffers)

# Maximum WAL size (triggers checkpoint)
max_wal_size = 4GB               # default: 1GB
min_wal_size = 1GB               # default: 80MB

# Checkpoint frequency
checkpoint_timeout = 15min       # default: 5min
checkpoint_completion_target = 0.9  # spread checkpoint writes over 90% of interval

# Full page writes (prevent torn pages after checkpoint)
full_page_writes = on            # on by default, critical for crash safety

# Data checksums (detect disk corruption)
data_checksums = on              # requires initdb with --data-checksums
```

### MySQL InnoDB Durability Settings

```
# my.cnf — durability-related settings

# WAL flush behavior
innodb_flush_log_at_trx_commit = 1  # 1=safest, 2=1s loss window, 0=OS flush

# Double-write buffer (prevents torn pages)
innodb_doublewrite = ON             # ON by default

# Log file size (affects recovery time)
innodb_log_file_size = 512M         # default in MySQL 8: 512MB

# Log buffer size
innodb_log_buffer_size = 64M        # default: 16MB

# Flush method (controls how InnoDB writes to disk)
innodb_flush_method = O_DIRECT      # O_DIRECT bypasses OS page cache

# Checksums for redo log pages
innodb_log_checksums = ON           # ON by default
```
