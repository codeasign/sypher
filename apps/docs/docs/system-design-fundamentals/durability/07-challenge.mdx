## Tier 1: Core Mechanisms

These problems verify you understand the fundamental mechanisms of durable writes.

### Challenge 1.1: The Read-After-Write Guarantee

A social media platform uses PostgreSQL with `synchronous_commit = on` and one synchronous replica. A user posts a comment. The application receives an HTTP 200 response. Immediately after, the application reads the comment from a read-only replica and finds it missing.

**Question**: Is this a durability failure? Explain what happened and how the system should handle this case.

<details>
<summary>Model Answer</summary>

This is **not** a durability failure. The write was durably stored:

- The primary received the write, appended to WAL, called fsync, and received confirmation
- The synchronous replica also received and acknowledged the WAL entry
- The client received HTTP 200 — the data is durably stored on both primary and replica

The problem is **consistency**, not durability. The read-only replica is behind. Although it is a "synchronous replica" for replication purposes, PostgreSQL's synchronous replication confirms the write at the WAL level — the replica may not have applied the change to its data pages yet.

**Solution options:**

1. Read from the primary for recently-written data (session stickiness)
2. Use PostgreSQL's `synchronous_commit = remote_apply` — waits until the replica has applied the change before acknowledging. Tradeoff: higher latency (3-15ms vs 2-5ms).
3. Accept stale reads and handle it in the application layer (e.g., show a "comment posted" confirmation while background-refreshing the data)

The key insight: durability does not imply immediate visibility.
</details>

### Challenge 1.2: Estimating Backup Storage

An analytics database is 2TB in size. It grows by 5% per month. Indexes add 40% overhead. You run full backups weekly and incremental backups daily. Full backups compress to 60% of raw size. Incremental backups capture 2% of data each day.

**Question**: Calculate the storage needed for 4 weeks of full backups plus 12 monthly full backups retained. Assume daily incremental backups are deleted after 30 days.

<details>
<summary>Model Answer</summary>

```
Raw data size: 2 TB
With index overhead (40%): 2.8 TB
Compressed full backup size: 2.8 TB × 0.60 = 1.68 TB

Weekly full backups, keep 4 weeks:
  4 × 1.68 TB = 6.72 TB

Monthly full backups, keep 12 months:
  12 × 1.68 TB = 20.16 TB (but we recycle weekly into
  monthly, so only the last monthly is additional)

More precisely:
  Week 1 full: 1.68 TB
  Week 2 full: 1.72 TB (after 1 month growth)
  Week 3 full: 1.76 TB
  Week 4 full: 1.80 TB
  → becomes monthly backup for month 1 (1.80 TB)

  Month 1: 1.80 TB
  Month 2: 1.85 TB (another 5% growth)
  Month 3: 1.90 TB
  ...
  Month 12: ~2.60 TB (at 5% monthly growth)

  Cumulative monthly storage:
  1.80 + 1.85 + 1.90 + 1.95 + 2.00 + 2.06 +
  2.12 + 2.18 + 2.24 + 2.31 + 2.38 + 2.60 = ~25.4 TB

Daily incremental backups:
  Each ~2% of compressed data = 0.02 × 1.68 = 33.6 GB
  30 days total: 30 × 33.6 GB = ~1 TB
  Deleted after 30 days (rolling)

Total backup storage needed: ~25.4 + 1.0 = ~26.4 TB
For cross-region copies: double that = ~53 TB
Estimated cost at S3 Standard (USD 23/TB/month): ~USD 610/month
Estimated cost at S3 Glacier (USD 4/TB/month): ~USD 106/month
```
</details>

## Tier 2: Architecture and Design

These problems require you to compose multiple durability mechanisms into a system architecture.

### Challenge 2.1: Multi-Tier Durability Design

Design the durability architecture for a payment processing system. Requirements:

- Processes 10,000 transactions per second at peak
- Zero data loss (RPO = 0)
- Recovery time under 5 minutes (RTO under 5 min)
- Must survive a single-AZ failure
- Must have cross-region disaster recovery with RPO under 1 minute
- Audit trail of all transactions for 7 years

**Constraint**: The database connection pool can handle at most 500 concurrent connections.

<details>
<summary>Model Answer</summary>

```
Durability architecture for payment processing:

Database layer:
- Primary in us-east-1a with synchronous_commit = on
- Two synchronous replicas in us-east-1b and us-east-1c
- Replication: synchronous (synchronous_standby_names = '*')
- Write quorum: all 3 must acknowledge before client ACK
- RPO = 0 within region: no single-AZ failure loses data

Connection pool management:
- 500 connections = 100 connections per database writer × 5 writer pools
- Group commit batches transactions → 200 writes/s per connection
- 500 connections × 200 writes/s = 100,000 writes/s capacity
- Actual peak: 10,000 writes/s → 5% utilization → safe

Backup strategy:
- Continuous WAL archiving to S3 (in us-east-1) every 5 seconds
- Full backup daily at 2 AM (compressed, 30-minute window)
- Cross-region WAL shipping to us-west-2 S3 bucket
- Weekly restore test in us-west-2 staging environment

Cross-region DR:
- Async WAL shipping to us-west-2 standby database
- RPO = 5 seconds (WAL archive cycle time)
- Failover: promote us-west-2 standby to primary
- RTO under 5 minutes: automated failover script pre-deployed

Audit trail (7 years):
- Transaction log written to append-only table
- Table partitioned by month for efficient archival
- Monthly archive to S3 Glacier (USD 1/TB/month)
- Access only through read-only views (no direct table access)

Total cost estimate (monthly):
  3 × r6g.8xlarge database instances: USD 9,720
  S3 storage for WAL and backups: USD 350
  S3 Glacier for audit trail: USD 80
  Cross-region data transfer: USD 120
  Total: ~USD 10,270/month
```
</details>

### Challenge 2.2: Durability vs Throughput Tuning

A real-time analytics system ingests 50,000 events per second from 10,000 sensors. Each event is a JSON payload averaging 2KB. The system stores raw events for 90 days in a time-series database for replay and debugging.

**Requirements:**
- Acceptable data loss: 0.1% (events can be dropped under extreme load)
- Must survive single-node failure without losing more than 1 second of data
- Write latency per event must stay under 50ms (p99)

**Question**: Design the ingestion pipeline with specific durability configuration choices. Justify each tradeoff.

<details>
<summary>Model Answer</summary>

```
Ingestion pipeline design:

Layer 1: Kafka as ingestion buffer
- 10 partitions (one per 1,000 sensors)
- Replication factor = 3
- acks = 1 (leader confirms write to its log)
- min.insync.replicas = 2
- flush.interval.ms = 100 (fsync every 100ms)

Justification:
- acks=1 provides durability for single-leader-failure scenarios
- min.isr=2 preserves durability during broker maintenance
- 100ms flush allows up to 100ms of data loss on simultaneous
  leader+disk failure — within the 1-second requirement
- Throughput: each partition handles 5,000 events/s × 2KB = 10MB/s
  10 partitions × 10MB/s = 100MB/s → well within Kafka's capacity

Layer 2: Time-series database (ClickHouse or TimescaleDB)
- Asynchronous consumption from Kafka
- Batch inserts every 5 seconds or 10,000 events
- WAL enabled with fsync every 1 second
- Replication factor = 2 (single node failure safe)

Justification:
- Batch inserts amortize fsync cost
- 1-second fsync window keeps data loss under the 0.1% target
  (0.1% of 50,000 = 50 events lost per second worst case)
- 5-second batch cycle in ClickHouse: ~2,500 events in flight
  at any moment — far below the 0.1% threshold

Layer 3: Fallback
- If Kafka disk is full: producer spools to local file
- If database ingest fails: messages remain in Kafka (consumer lag)
- Alert on consumer lag > 30 seconds

Durability math:
  Event loss in worst 1-second window:
    - Kafka unsent data: 0 (leader has it in memory)
    - Kafka unflushed: up to 50,000 events (100ms × 50K/s)
    - DB batch in-flight: up to 10,000 events
    - Total at risk: ~15,000 events (0.03% of 50K/s)
    - Acceptable loss: 0.1% = 50 events/s

  The risk exceeds the acceptable loss only if both Kafka leader
  and its disk fail simultaneously within the 100ms flush window.
  With RF=3 and 3 brokers, the probability of this is negligible
  (< 0.001% annual probability based on AWS EBS failure rates).
```
</details>

## Tier 3: Failure Analysis and Debugging

These problems present real-world durability incidents for you to diagnose and fix.

### Challenge 3.1: The Phantom Data Loss

A team runs a PostgreSQL 14 database with `synchronous_commit = on`, one synchronous replica in a different AZ, and daily `pg_dump` backups to S3. The application reports that data written exactly 48 hours ago is missing from some query results.

Investigation reveals:
- The data exists in the primary database (SELECT COUNT returns correct row count)
- The data does not appear in a query that JOINs across three tables
- A `VACUUM` ran 48 hours ago
- No errors in PostgreSQL logs for the past 7 days

**Question**: What happened? Is this a durability failure? How do you confirm your hypothesis?

<details>
<summary>Model Answer</summary>

This is **not** a durability failure — the data is present. It is a **visibility** issue caused by PostgreSQL's MVCC (Multi-Version Concurrency Control) and transaction wraparound.

Hypothesis: A transaction ID wraparound occurred during the VACUUM 48 hours ago. PostgreSQL's MVCC uses 32-bit transaction IDs (max ~4 billion). When the counter wraps, old row versions can appear as "future" transactions and become invisible to current queries.

How this happens:
1. The application inserted rows 48 hours ago with transaction ID near the wraparound point
2. VACUUM ran and, as part of anti-wraparound processing, froze old row versions (marked them as permanently visible)
3. However, if the database was not configured with `autovacuum_freeze_max_age` set appropriately, or if the VACUUM was interrupted, some rows may have been left in a state where they appear to be in the future

Confirmation:
1. Check `pg_stat_activity` for current transaction ID: `SELECT txid_current();`
2. Check the wraparound horizon: `SELECT age(datfrozenxid) FROM pg_database WHERE datname = current_database();`
3. Check for rows with unexpected xmin: `SELECT xmin, xmax, count(*) FROM your_table GROUP BY xmin, xmax;`
4. Look for xmin values close to the wraparound point

Fix:
1. Set `autovacuum_freeze_max_age = 200000000` (2 billion)
2. Run manual VACUUM FREEZE on the affected tables
3. Monitor `age(datfrozenxid)` to stay below 1.5 billion

Prevention:
- Monitoring alert when `age(datfrozenxid)` exceeds 1 billion
- Configure `vacuum_freeze_min_age = 100000000`
- Use `txid_current()` monitoring to track wraparound progress
</details>

### Challenge 3.2: The Silent Replica Divergence

A database administrator notices that a read replica has 0.5% more rows than the primary for a specific table. The replica is an asynchronous streaming replica. No errors appear in logs. Checksums are enabled and report no errors.

**Question**: How is this possible? What mechanism caused the replica to have data the primary does not have? How do you detect and prevent this?

<details>
<summary>Model Answer</summary>

This is a **replica divergence** — a rare but serious condition where the replica's state differs from the primary's.

Most likely cause: A **write to the replica** bypassed the replication stream. This can happen when:

1. A direct connection was made to the replica with write permission
   - Someone ran a manual query directly on the replica
   - An application configuration pointed to the replica's address for writes

2. A function or trigger on the replica executed a write
   - PostgreSQL's `pg_stat_statements` or an extension with side effects
   - A trigger that writes to a log table when data is read

3. A **partial failover and rollback** scenario:
   - The replica was briefly promoted to primary, accepted writes, then was demoted back
   - The additional writes from the brief promotion period persisted
   - Streaming replication resumed from the original primary's position, leaving the extra rows

Detection:
1. Compare row count per table: `SELECT count(*) FROM table` on both servers
2. Use `pgchecksums` or `pg_verify_checksums` to verify page-level integrity
3. Compare table checksums using `pg_stat_replication` and `pg_wal_replay_pause` for point-in-time comparison
4. Use `pg_rewind` to check timeline divergence

Fix:
1. Stop the replica from accepting writes: set `hot_standby = off` temporarily
2. `pg_rewind` the replica to match the primary's timeline
3. Alternatively: rebuild the replica from a fresh `pg_basebackup`
4. Add monitoring: `pg_stat_replication.sync_state` and `pg_stat_wal_receiver`

Prevention:
1. Configure `recovery_target_timeline = 'latest'` on replicas
2. Set `wal_log_hints = on` (required for pg_rewind)
3. Application code should use separate connection strings for read/write with read-only flag on replica connections
4. Audit logs for direct connections to replica addresses
</details>

### Challenge 3.3: The Backup That Would Not Restore

You attempt to restore a PostgreSQL database from last night's backup. The backup process reported success. The backup file exists in S3. The restore process fails with:

```
FATAL:  could not open file "pg_wal/0000000100000023000000AF": No such file or directory
```

**Question**: What caused this error and how do you prevent it from recurring?

<details>
<summary>Model Answer</summary>

This is a **missing WAL segment** error. The backup captured the data files at a specific point, but the WAL segment needed to bring the database to a consistent state is missing.

Two possible causes:

**Cause 1: Concurrent backup took WAL at the wrong time**
- `pg_basebackup` was taken while the database was active
- The backup includes data files that reflect state at different points in time
- The WAL segment needed to reconcile those states was recycled by a checkpoint
- PostgreSQL cannot achieve a consistent state for recovery

**Cause 2: WAL archiving gap**
- The backup includes data files from full backup
- The WAL archive has a gap — segment 0000000100000023000000AF was deleted
  before it was archived
- This happens when `archive_timeout` is too short and `wal_keep_segments` is too low
- Or when replication_slot lag caused the garbage collector to delete needed segments

Investigation:
1. Check `pg_wal` on the source server for the missing segment
2. Check WAL archive directory for gap in segment sequence
3. Query `pg_stat_archiver` for archival failure counts
4. Check if `wal_keep_segments` was set correctly (default: 0 in older versions)

Fix for this restore:
1. Try a more recent backup (if the gap is only in this one)
2. If WAL archive exists elsewhere: locate the segment in a different archive path
3. If all else fails: restore the backup with `recovery_target_timeline = 'immediate'` to skip WAL replay (data may be inconsistent — use only for emergency)

Prevention:
1. Set `wal_keep_segments` to a value that keeps at least 24 hours of WAL
   - At 10MB per segment and 64MB per minute: 24 hours = 90GB ≈ 9,000 segments
   - Set `wal_keep_segments = 10000` (or use replication slots instead)
2. Enable replication slots on the primary: `CREATE_REPLICATION_SLOT ...`
3. Set `archive_timeout = 60` (archive WAL segments at least every 60 seconds)
4. Monitor `pg_stat_archiver.failed_count` — alert on any non-zero value
5. Test backup restore weekly, not just after incidents
6. Use `pgBackRest` or `barman` for automated WAL management and validation
</details>

---

These challenges demonstrate that durability is not a binary property — it is a continuous spectrum that requires careful configuration, monitoring, and testing. The mechanisms and architectures covered throughout this topic provide the foundation for designing systems where data truly stays written.
