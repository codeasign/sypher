Docker's apparent simplicity — a single `docker run` command starts your application — hides a complex stack of kernel features, filesystem technologies, and network plumbing. Understanding these internals is essential for debugging production issues, optimizing container performance, and designing secure containerized systems.

## Internal Architecture

The Docker engine consists of four major components that work together to build, ship, and run containers. Each component can be replaced or upgraded independently, and understanding their boundaries helps you diagnose where failures occur.

### dockerd — The Docker Daemon

dockerd is a long-running background process that exposes the Docker API on a Unix socket (`/var/run/docker.sock`). It handles all high-level container management including image building, volume management, network creation, and container lifecycle operations. dockerd is written in Go and communicates with containerd over gRPC. When dockerd crashes, running containers continue running because containerd and the shims operate independently. You can restart dockerd without affecting running containers.

### containerd — The Container Supervisor

containerd manages the complete container lifecycle: image transfer, image storage, container creation, and container supervision. It exposes its own gRPC API used by both Docker and Kubernetes. containerd handles image pulling from registries, decompresses image layers, and manages the overlay filesystem snapshots. It does not directly create containers — it delegates that to runc. The separation between dockerd and containerd means that Kubernetes can use containerd directly, bypassing the Docker API entirely.

### containerd-shim — The Per-Container Watcher

Each running container has a dedicated containerd-shim process. The shim acts as an intermediary between containerd and runc. It keeps the container's STDIO streams open so that dockerd can attach and detach without affecting the container. When runc creates the container process and exits, the shim becomes the parent of the container's init process. If containerd restarts, the shim reconnects to containerd and reports the container's status. If dockerd restarts, the shim and container are unaffected. This architecture ensures container uptime is independent of Docker daemon restarts.

### runc — The Low-Level Runtime

runc is the reference implementation of the OCI runtime specification. It creates the actual container by setting up Linux namespaces, applying cgroup resource limits, mounting the root filesystem using overlayfs, and executing the container process. runc is a standalone CLI tool that takes an OCI bundle (a directory containing a config.json and a root filesystem) and creates a container from it. runc does one thing — create and run a container — and exits after the container process starts. The shim keeps the container alive after runc exits.

## Data Structures and Algorithms

### 1. The OCI Image Format — Content-Addressable Storage

Docker images are stored using a content-addressable storage scheme. Each blob (layer) is identified by its SHA-256 digest. If two images share the same base layer, they share the same blob on disk. The image manifest is a JSON document that references these blobs:

```json
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.manifest.v1+json",
  "config": {
    "mediaType": "application/vnd.oci.image.config.v1+json",
    "digest": "sha256:44136fa355b3678a1146ad16f7e8649e94fb4fc21fe77e8310c060f61caaff8a",
    "size": 7023
  },
  "layers": [
    {
      "mediaType": "application/vnd.oci.image.layer.v1.tar+gzip",
      "digest": "sha256:a3ed95caeb02ffe68cdd9fd84406680ae93d633cb16422d00e8a7c22955b46d4",
      "size": 32654
    }
  ]
}
```

The content-addressable scheme provides three benefits. First, deduplication — if 100 containers use `node:18-alpine` as their base, that image is stored once. Second, integrity verification — any corruption changes the digest and becomes immediately detectable. Third, pull optimization — Docker downloads only layers not already present on the host, using the Layer Diff algorithm.

### 2. Copy-on-Write (CoW) with OverlayFS

Docker uses OverlayFS (or aufs on older systems) to implement the layered filesystem. OverlayFS merges multiple directories (layers) into a single view. The lower layer is the read-only image root. The upper layer is the container's writable layer. When a container reads a file:

1. Check upper directory — if the file exists, return it
2. Check lower directory — if the file exists, return it
3. If neither has the file, return ENOENT

When a container writes to a file that exists only in the lower layer:

1. Copy the entire file from the lower layer to the upper layer (copy-up)
2. Open the new copy in the upper layer for writing
3. Subsequent reads and writes use the upper layer copy

The copy-up operation copies the entire file, regardless of how many bytes you change. For large files where you only modify a small portion, this is expensive. The copy-up happens the first time any write occurs to the file — subsequent writes to the same file are direct.

```text
OverlayFS Merge:

        Container View                      On Disk
  ┌─────────────────────┐        ┌────────────────────────┐
  │ /app/package.json   │        │ Upper Layer (rw)       │
  │ /app/node_modules/  │  ──>   │ Modified files only    │
  │ /app/src/server.js  │        │ e.g. /app/node_modules │
  │ /app/logs/app.log   │        └────────────────────────┘
  └─────────────────────┘        ┌────────────────────────┐
                                 │ Lower Layer 3 (ro)     │
                                 │ src/server.js          │
                                 └────────────────────────┘
                                 ┌────────────────────────┐
                                 │ Lower Layer 2 (ro)     │
                                 │ package.json           │
                                 │ node_modules/          │
                                 └────────────────────────┘
                                 ┌────────────────────────┐
                                 │ Lower Layer 1 (ro)     │
                                 │ /lib, /usr, /bin       │
                                 │ (base image: Alpine)   │
                                 └────────────────────────┘
```

### 3. Layer Caching Algorithm

When Docker builds an image, it processes each Dockerfile instruction and creates a cache key. The cache key includes:
- The instruction text (e.g., `RUN npm install`)
- The hash of any files copied by COPY or ADD
- The hash of the parent layer

Docker looks up this cache key in its local cache. If found, Docker skips the instruction and uses the cached layer. This algorithm is why the order of instructions in your Dockerfile matters. Consider:

```dockerfile
# BAD: Changes to source invalidate the npm install cache
COPY . /app
RUN npm install

# GOOD: npm install only reruns when dependencies change
COPY package.json /app
RUN npm install
COPY src/ /app
```

The Layer Caching algorithm is a key insight for Docker performance. A well-organized Dockerfile reduces build times from minutes to seconds.

## Request Lifecycle — docker run

The `docker run` command triggers a chain of operations across all four internal components. Here is the complete lifecycle of starting a container.

### Step 1: CLI Parses Arguments

The Docker CLI parses `docker run -d -p 8080:3000 --name webapp --memory=512m my-app:1.0`. It validates flags, resolves image references, and constructs a REST API request. The CLI appends default values from the Docker configuration file and environment variables.

### Step 2: CLI Sends HTTP Request to dockerd

The CLI connects to dockerd over the Unix socket `/var/run/docker.sock` (or a TCP socket for remote Docker). It sends a POST request to `/containers/create` with a JSON body containing the image reference, port mappings, resource limits, and container configuration.

### Step 3: dockerd Validates and Creates Container Object

dockerd validates the configuration. It checks that:
- The requested image exists in the local image store (or can be pulled)
- Port mappings do not conflict with existing containers
- The container name is unique
- Resource limits are valid (memory under host capacity, CPU shares within range)

dockerd creates a container object in its internal state store with status "created" and a unique container ID.

### Step 4: dockerd Sets Up Networking

dockerd calls the Docker network controller to create the container's network namespace. It creates a veth pair — one end inside the container's namespace, the other attached to the Docker bridge (docker0 or a user-defined bridge). Docker assigns an IP address from the bridge subnet and sets up NAT rules so the container can reach the outside world. If port mappings are specified, dockerd adds iptables rules to forward traffic from the host port to the container port.

```bash
# iptables rules created by Docker for -p 8080:3000
iptables -t nat -A DOCKER ! -i docker0 \
  -p tcp --dport 8080 -j DNAT --to-destination 172.17.0.2:3000
iptables -A DOCKER -d 172.17.0.2/32 ! -i docker0 \
  -o docker0 -p tcp --dport 3000 -j ACCEPT
```

### Step 5: containerd Creates a Snapshot

containerd creates a new snapshot for the container using OverlayFS. It creates an empty upper directory in `/var/lib/docker/overlay2/<container-id>/diff` and a merged view in `/var/lib/docker/overlay2/<container-id>/merged`. The lower directories are the image layers. containerd records the snapshot metadata so it can be garbage-collected when the container is removed.

### Step 6: containerd Creates the Task

containerd constructs an OCI runtime specification (config.json) that describes the container's configuration — namespaces to create, cgroup limits to apply, root filesystem to mount, and the command to execute. The OCI spec is written to `/var/lib/docker/containers/<container-id>/config.json`. containerd then starts a new containerd-shim process for this container.

### Step 7: Shim Invokes runc

The shim reads the OCI spec and invokes runc. runc reads config.json and performs the following in order:

1. Create a new mount namespace
2. Create a new PID namespace
3. Create a new network namespace
4. Create a new UTS namespace (set hostname)
5. Create a new IPC namespace
6. Mount the root filesystem using OverlayFS
7. Create a cgroup and apply CPU, memory, and I/O limits
8. Chdir to the container's working directory
9. Set the user and group IDs
10. Clone the child process using `clone(CLONE_NEWNS|CLONE_NEWPID|...)` syscall
11. The child process pivots root to the container filesystem
12. The child process execs the CMD (or ENTRYPOINT + CMD)

### Step 8: Shim Detaches, Keeps Communication Open

After runc creates the container process and exits, the shim becomes the parent of the container process. The shim keeps the container's STDIO pipes open and forwards them to containerd's log system. The shim monitors the container process for exit. If the container exits, the shim captures the exit code and reports it to containerd, which reports to dockerd, which can trigger restart policies.

### Step 9: Container Runs

The container executes its main process. The Docker CLI returns the container ID to the terminal. You can interact with the running container:

- `docker logs webapp` — dockerd requests logs from containerd, which reads from the shim's STDIO buffer
- `docker exec webapp bash` — dockerd creates a new process in the container's existing namespaces
- `docker stats webapp` — dockerd reads cgroup metrics for the container

### Step 10: Container Exits or Is Stopped

When the main process exits, runc tears down the container. The PID namespace is destroyed, killing all remaining processes. The cgroup is cleaned up. The network namespace is removed, and the veth pair is deleted. The overlay mount is unmounted. The container enters the "exited" state. The writable layer persists until you remove the container with `docker rm`. The image layers remain cached for reuse.

## Consistency and State

Docker containers are ephemeral by design. The writable layer exists only as long as the container exists. This design imposes constraints on how stateful applications run in containers.

### Failure Scenario 1: Container Crash and Data Loss

A database container crashes due to an out-of-memory kill. If the database writes data only to the container's writable layer, all that data is lost. The database's recovery log, which exists in the same writable layer, is also lost. When the container restarts (possibly from the same image), it starts with a fresh writable layer containing no data.

**Mitigation**: Use Docker volumes for all persistent data. Volume data lives outside the container's writable layer, on the host filesystem or on a network filesystem. The container can be destroyed and recreated, and the volume is mounted on the new container:

```bash
docker run -v mydbdata:/var/lib/postgresql/data postgres:15
```

The volume `mydbdata` survives container deletion. When a new container mounts the same volume, it accesses the same data files.

### Failure Scenario 2: Container Restarts and PID 1 Behavior

When a container's PID 1 process exits, the container stops. If restart policy is `--restart always`, Docker starts a new container from the same image with a fresh writable layer. Any logs, temporary files, or intermediate state in the writable layer are lost. If the application depends on initialization state from a previous run, it fails.

**Mitigation**: Design applications to be stateless. Store session state in Redis, file uploads in S3, and database data in a volume-mounted database. For initialization sequences that must run once, use entrypoint scripts that check for pre-existing state:

```bash
#!/bin/sh
if [ ! -f /data/initialized ]; then
  ./run-migrations.sh
  touch /data/initialized
fi
exec "$@"
```

### Failure Scenario 3: Resource Exhaustion Cascading

A memory leak in one container consumes all available host memory. The kernel OOM killer starts terminating processes. Because all containers share the host kernel, the OOM killer can terminate processes in any container, not just the leaking one. Critical services can be killed due to another container's memory leak.

**Mitigation**: Set hard memory limits on every container using `--memory`. Never run a container without a memory limit in production. Configure `--memory-reservation` for soft limits that guide the kernel's OOM killer. Use `--oom-kill-disable` only with extreme caution — it can cause the kernel to hang waiting for memory. Monitor container metrics with `docker stats` and external monitoring tools.

## Performance Characteristics

Docker adds minimal overhead compared to running applications directly on the host. The following numbers assume a modern Linux kernel (5.x+) on bare metal.

### CPU Overhead

Docker adds virtually no CPU overhead for compute-bound workloads. The container process runs directly on the host CPU with no virtualization layer. There is a minor cost from namespace operations during container creation — approximately 10-20 milliseconds per container start. For running workloads, benchmark tests show that containerized CPU performance is 97-99% of native, with the 1-3% difference attributed to cgroup accounting overhead.

### Memory Overhead

Each container incurs negligible memory overhead for the kernel data structures that maintain namespaces and cgroups. This overhead is approximately 5-10 MB per container. The application's memory usage is identical to native execution. Memory limits enforced by cgroups add approximately 1 microsecond per memory allocation for accounting, which is not measurable in real workloads.

### Network Throughput

Docker bridge networking adds a small amount of overhead due to the veth pair and iptables NAT. Benchmark measurements show approximately 95-98% of native network throughput for most workloads. Latency increases by 10-50 microseconds per packet due to the veth and iptables processing. For latency-sensitive workloads, consider using host networking mode (--net=host), which achieves 99-100% of native network performance by sharing the host's network stack.

### Storage Performance

OverlayFS write performance has a significant caveat. The first write to a file that exists only in the read-only lower layer triggers a copy-up operation that copies the entire file. For a 100 MB log file where you only append 1 KB, the first write copies 100 MB to the upper layer. Subsequent writes to the same file are fast. This characteristic means:

- Databases writing to data files can suffer from slow initial writes
- Log file initialization in the container triggers the copy-up
- Reading files that exist in the image layers has no CoW penalty

### Container Start Time

Starting a container from a cached image typically takes 100-500 milliseconds. This includes namespace creation, cgroup setup, overlay mount, and process execution. Pulling a new image depends on layer sizes and network bandwidth — a typical 200 MB image pulls in 5-10 seconds on a 200 Mbps connection. Cold start (pull + start) is dominated by image pull time.

## Configuration and Tuning

### Docker Daemon Configuration

Docker's daemon is configured through `/etc/docker/daemon.json`. Critical settings for production deployments:

```json
{
  "exec-opts": ["native.cgroupdriver=systemd"],
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "storage-driver": "overlay2",
  "storage-opts": [
    "overlay2.override_kernel_check=true"
  ],
  "max-concurrent-downloads": 10,
  "max-concurrent-uploads": 10,
  "live-restore": true,
  "default-ulimits": {
    "nofile": {
      "Name": "nofile",
      "Hard": 65536,
      "Soft": 65536
    }
  }
}
```

`live-restore: true` allows dockerd to restart without stopping running containers. The shim processes keep the containers alive and reconnect when dockerd starts again. This is essential for production hosts where dockerd updates or crashes should not affect running workloads.

### Container Runtime Constraints

Set resource limits on every container in production:

```bash
docker run \
  --memory=512m \
  --memory-reservation=384m \
  --cpus=0.5 \
  --cpuset-cpus=0-1 \
  --blkio-weight=500 \
  --ulimit nofile=65536:65536 \
  my-app:1.0
```

- `--memory`: Hard limit. Container is killed if it exceeds this.
- `--memory-reservation`: Soft limit. Container can burst above this if memory is available, but is throttled when the host is under memory pressure.
- `--cpus`: Fractional CPU limit. 0.5 means half a CPU core.
- `--cpuset-cpus`: Pin to specific CPU cores. Important for NUMA-aware workloads.
- `--blkio-weight`: Relative I/O priority. Range 10-1000.

### Logging Configuration

Docker's default logging driver stores JSON-formatted logs in files on the host. Without log rotation, a chatty container can fill the host disk:

```bash
docker run \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  my-app:1.0
```

For centralized logging, use a different driver:

```bash
docker run \
  --log-driver awslogs \
  --log-opt awslogs-group=my-app \
  --log-opt awslogs-stream=production \
  my-app:1.0
```

### Health Checks

Define health checks in the Dockerfile or at runtime to enable automatic recovery:

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```

Docker runs the health check command inside the container every 30 seconds. Three consecutive failures mark the container as unhealthy. Orchestrators like Docker Swarm or Kubernetes use this health status to replace unhealthy containers.

### Security Configuration

Apply security hardening to every container:

```bash
docker run \
  --security-opt no-new-privileges \
  --security-opt seccomp=/path/to/seccomp-profile.json \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --read-only-rootfs \
  --tmpfs /tmp \
  my-app:1.0
```

`--cap-drop=ALL` removes all Linux capabilities, then `--cap-add=NET_BIND_SERVICE` adds back only the capability to bind to privileged ports (under 1024). `--read-only-rootfs` makes the container's filesystem read-only, preventing malware from writing to the filesystem. `--tmpfs /tmp` provides a writable temporary directory for runtime files that does not persist.

### Resource Limiting Trade-offs

Setting resource limits too low causes throttling and OOM kills. Setting them too high wastes capacity and risks noisy neighbors. The general guideline for Java applications is to set memory to approximately 1.5x the expected heap size to account for JVM overhead, native memory, and thread stacks. For Node.js, set memory to 2x the expected RSS. For Go, memory usage typically matches the RSS closely, so 1.2x headroom is sufficient. CPU limits should account for burst capacity — set the limit 20-50% above the average CPU usage to handle traffic spikes without throttling.
