Running a single container on your laptop is straightforward. Operating thousands of containers across hundreds of servers in production is a different problem entirely. This section covers the architecture patterns, deployment strategies, and operational practices that enable Docker at scale.

## Naive Implementation — The Solo Developer

### 1000 Users: Single Host, Manual Deploy

For a small application serving under 1000 users, a single Docker host with Compose is sufficient. The application runs as a set of containers defined in a `docker-compose.yml` file. The developer SSHes into the server, pulls the latest images, and restarts services.

```yaml
version: "3.8"
services:
  web:
    image: my-app/web:latest
    ports:
      - "80:3000"
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db
  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secret
volumes:
  pgdata:
```

Deployment is manual: you SSH into the server, pull the new image, and restart. The `docker compose up -d` command recreates containers whose images changed. There is a few seconds of downtime as the web container restarts. If the new image has a bug, you SSH back in and redeploy the previous version. This works for internal tools and low-traffic apps but becomes unmanageable as traffic grows.

### 100K Users: Multi-Host with Orchestration

At 100,000 users, a single server cannot handle the load. You need multiple application servers behind a load balancer, a separate database cluster, and a cache layer. Docker Compose on a single host cannot manage this — you need an orchestrator.

You deploy containers onto a cluster of machines. The orchestrator schedules containers onto hosts with available capacity, monitors their health, and replaces failed containers. ### 10M Users: Global Scale with Multi-Region

At 10 million users, you need multiple regions for latency and fault tolerance. Container images are distributed across regions using a geo-replicated registry. Each region runs its own cluster with local services and cross-region replication for stateful data.

## Integration Patterns

### Pattern 1: Sidecar Container

A sidecar container runs alongside the main application container in the same pod or task, sharing the same network namespace and volume mounts. The sidecar extends or enhances the main container without modifying its code.

```yaml
# Docker Compose sidecar for logging
services:
  app:
    image: my-app:1.0
    volumes:
      - logs:/var/log/app
  log-collector:
    image: fluent/fluentd:latest
    volumes:
      - logs:/var/log/app
      - ./fluentd.conf:/fluentd/etc/fluentd.conf
volumes:
  logs:
```

Common sidecar patterns include:
- **Log shipper** — reads application logs and forwards to centralized logging
- **Metrics exporter** — scrapes application metrics and exposes Prometheus endpoint
- **Service mesh proxy** — Envoy or Linkerd proxy handling traffic encryption and routing
- **Config reloader** — watches config changes and signals the main process

### Pattern 2: Ambassador Container

An ambassador container proxies network traffic to and from the main container. This pattern abstracts external service dependencies so the main container connects to "localhost" and the ambassador handles discovery, retries, and circuit breaking.

```yaml
services:
  app:
    image: my-app:1.0
    environment:
      - REDIS_URL=redis://localhost:6379
  redis-ambassador:
    image: haproxy:2.0
    volumes:
      - ./haproxy-redis.cfg:/usr/local/etc/haproxy/haproxy.cfg
    # Haproxy configured to connect to the actual Redis cluster
```

### Pattern 3: Init Container

An init container runs to completion before the main container starts. Init containers are used for setup tasks that require different permissions or tools than the main application:

- Database schema migrations
- File permission setup on shared volumes
- Asset precompilation
- Configuration file generation from templates

```yaml
services:
  app:
    image: my-app:1.0
    depends_on:
      migrations:
        condition: service_completed_successfully
  migrations:
    image: my-app:migration-tools
    command: ["node", "run-migrations.js"]
```

### Pattern 4: Batch Job Container

Containers that run to completion, perform a task, and exit. Batch jobs process data, generate reports, or perform maintenance. They are specified with `restart: "no"` and orchestrated through cron-like schedulers or workflows.

```yaml
services:
  daily-report:
    image: my-app:1.0
    command: ["node", "generate-report.js"]
    restart: "no"
    environment:
      - REPORT_DATE=${REPORT_DATE}
    volumes:
      - reports:/reports
```

## Multi-Region and Global Scale

### Container Image Distribution

At global scale, pulling images from a single registry across continents adds latency. A 500 MB image pulled from us-east-1 to a host in Singapore takes 30-60 seconds, delaying deployment. The solution is geo-replicated registries.

Amazon ECR supports cross-region replication. An image pushed to us-east-1 is automatically replicated to eu-west-1 and ap-southeast-1. Each region's Docker hosts pull from their local registry endpoint. Replication is asynchronous and typically completes within 5-15 minutes. For critical deployments, push to all regions simultaneously.

### Region-Specific Configuration

Containers in different regions need region-specific configuration — database endpoints, queue URLs, feature flags. This configuration is injected through environment variables or config volumes, not baked into the image. The same image runs in every region, with configuration determining behavior:

```bash
# us-east-1
docker run -e DB_HOST=db.us-east-1.example.com -e REGION=us-east-1 my-app:1.0

# eu-west-1
docker run -e DB_HOST=db.eu-west-1.example.com -e REGION=eu-west-1 my-app:1.0
```

### Cross-Region State Management

Stateful containers present a challenge at global scale. A database container in us-east-1 cannot serve writes from eu-west-1 with acceptable latency. The standard approach is to run regional databases with cross-region replication and route read traffic to the local replica. Write traffic goes to the primary region or uses a multi-region database like CockroachDB or Spanner.

## Capacity Planning

### Estimating Container Density

Docker's low overhead means you can run many containers per host. The limiting factor is typically memory, not CPU. Here is a capacity planning calculation for a typical web service:

```text
Service requirements:
- 1 container instance: 512 MB RAM, 0.5 CPU
- Expected peak traffic: 10,000 requests/second
- Per-container throughput: 500 requests/second
- Required containers: 10,000 / 500 = 20 containers

Host capacity (AWS c5.4xlarge: 16 vCPU, 32 GB RAM):
- Memory-limited: 32 GB / 512 MB = 62 containers
- CPU-limited: 16 vCPU / 0.5 CPU = 32 containers
- Actual limit (min): 32 containers per host

Hosts needed for 20 containers:
- 20 / 32 = 1 host (with 37% overhead for headroom)
- Actual: 2 hosts for redundancy (N+1)
```

### Image Size Optimization

Image size directly affects deployment speed. A bloated image increases pull time, consumes disk space, and increases the attack surface.

```dockerfile
# Production-optimized Node.js Dockerfile
FROM node:18-alpine AS builder
WORKDIR /build
COPY package.json package-lock.json ./
RUN npm ci
COPY src/ ./src/
RUN npm run build

FROM node:18-alpine AS production
WORKDIR /app
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup
COPY --from=builder /build/dist ./dist
COPY --from=builder /build/node_modules ./node_modules
COPY package.json ./
USER appuser
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

Multi-stage builds reduce the final image from 1.2 GB (with dev dependencies and build tools) to 150 MB (only runtime code and production dependencies). The first stage installs build tools and compiles code. The second stage starts from a fresh base image and copies only the compiled artifacts.

### Network Bandwidth

Each container consumes network bandwidth for requests, responses, and log shipping. Planning for a 10,000 request/second workload with 50 KB average response size:

```text
Per-container: 500 req/s × 50 KB = 25 MB/s
Total (20 containers): 20 × 25 MB/s = 500 MB/s (4 Gbps)
Per host (5 containers): 5 × 25 MB/s = 125 MB/s (1 Gbps)
```

A c5.4xlarge instance has up to 10 Gbps network bandwidth, so the network is not the bottleneck in this scenario. If response sizes are larger (e.g., 500 KB for API responses), the network becomes the constraint and you need fewer containers per host or larger instance types.

## Deployment and Operations

### Blue-Green Deployments with Docker

Blue-green deployment runs two identical environments (blue and green). At any time, only one environment serves production traffic. When you deploy a new version, you deploy to the idle environment, run smoke tests, and then switch the load balancer.

```bash
# Deploy green (new version)
docker compose -f docker-compose.green.yml up -d

# Run smoke tests against green
curl -f http://green:3000/health

# Switch load balancer to green
# (Update DNS or LB target group)

# Tear down blue (old version)
docker compose -f docker-compose.blue.yml down
```

### Rolling Updates

Rolling updates replace containers one at a time, maintaining service availability throughout the deployment. The orchestrator starts a new container with the new image, waits for it to pass health checks, then stops an old container. This continues until all containers are updated.

```yaml
# Docker Compose deploy configuration
services:
  web:
    image: my-app:1.0
    deploy:
      replicas: 6
      update_config:
        parallelism: 2
        delay: 10s
        order: start-first
        failure_action: rollback
      restart_policy:
        condition: any
        delay: 5s
        max_attempts: 3
```

### Canary Deployments

Canary deployments route a small percentage of traffic to the new version to validate it before full rollout. The canary runs alongside the existing version, receiving 1-5% of traffic. After monitoring for errors and latency, you gradually increase the canary's traffic share.

### Health Checks and Auto-Recovery

Every container should expose a health endpoint. The orchestrator polls this endpoint and replaces containers that fail health checks. Configure health checks with appropriate thresholds:

```yaml
services:
  web:
    image: my-app:1.0
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 15s
```

The `start_period` parameter gives the container time to initialize before health checks begin. A container that fails three consecutive health checks is marked unhealthy and replaced.

### Logging and Monitoring

Centralized logging collects logs from all containers across all hosts. Docker's json-file driver writes logs to the host filesystem, but at scale you need to forward logs to a central system. The standard pattern is a log-shipper sidecar or a daemon-set that runs on every host and forwards all container logs.

```yaml
# Logging configuration for production
docker run \
  --log-driver awslogs \
  --log-opt awslogs-region=us-east-1 \
  --log-opt awslogs-group=my-app-production \
  --log-opt awslogs-stream=web-1 \
  --log-opt awslogs-multiline-pattern='^\d{4}-\d{2}-\d{2}' \
  my-app:1.0
```

Monitoring container metrics at scale requires collecting from each Docker host. Prometheus can scrape the Docker engine metrics endpoint or use cAdvisor to collect per-container metrics. Key metrics to monitor:

- Container CPU usage vs. limit
- Container memory usage vs. limit
- Container restart count
- Image pull latency
- Disk usage of overlay2 directory
- Number of containers per host

### Backup and Disaster Recovery

Container configurations are ephemeral and can be recreated from source control. The critical data to back up is:

1. **Volume data** — database data, file storage, and any stateful content
2. **Registry** — Docker images stored in the registry
3. **Orchestrator state** — Kubernetes manifests, Docker Compose files, configuration

Back up volumes using standard filesystem tools or cloud-native snapshots. Back up the registry by replicating to another region or using a registry backup tool. Store all configuration in version control — the entire Docker infrastructure should be reproducible from a `git clone` and a `docker compose up`.
