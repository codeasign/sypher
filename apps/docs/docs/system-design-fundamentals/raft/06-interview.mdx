## How Raft Questions Appear

Raft questions in system design interviews rarely ask "explain Raft step by step." Instead, Raft surfaces as the answer to deeper design problems. Interviewers probe your understanding of consensus by asking you to design systems that require strong consistency, fault tolerance, and leader election.

**Typical question formats:**

"Design a distributed configuration service like etcd or ZooKeeper. How do you ensure that all clients see the same configuration?"

"Design a distributed locking service. How do you handle leader failures?"

"Our database has a single primary that handles all writes. If the primary fails, how do we elect a new one without losing committed data?"

"Design a fault-tolerant key-value store with strong consistency across three data centers."

In each case, Raft is not the surface-level topic but the underlying mechanism you should reach for.

## Level-by-Level Expectations

### Entry Level (0-2 years experience)

**What is expected:** You understand that Raft is a consensus algorithm for distributed systems. You can name the three subproblems (leader election, log replication, safety) and explain why consensus is hard.

**Sample question:** "What problem does Raft solve?"

<details>
<summary>Model Answer</summary>

Raft solves the consensus problem: multiple nodes in a distributed system must agree on a sequence of state changes, even when some nodes fail or the network experiences delays. Consensus is needed whenever a system requires strong consistency across replicas. Raft solves this by electing a single leader that coordinates all decisions, replicating a log of state changes to followers, and using safety rules that guarantee committed entries are never lost. It is designed to be more understandable than the alternative (Paxos) while providing identical safety guarantees.

</details>

### Mid Level (2-5 years experience)

**What is expected:** You can describe Raft's mechanisms in detail: leader election with timeouts and terms, log replication with AppendEntries RPC, and the log matching property. You understand trade-offs like the impact of timeout tuning on availability.

**Sample question:** "How does Raft handle leader failures and ensure data is not lost?"

<details>
<summary>Model Answer</summary>

When a leader fails, followers detect the failure through missing heartbeats. Each follower maintains an election timer randomized between 150-300ms. The first follower to timeout increments its term, transitions to candidate, and requests votes from all nodes. A candidate becomes leader if it receives votes from a majority.

Data safety is maintained through two mechanisms. First, the election restriction: a node only votes for a candidate whose log is at least as up-to-date as its own, ensuring the new leader has all committed entries. Second, committed entries are replicated to a majority of nodes before acknowledgment. A committed entry exists on at least one node that voted for the new leader, so the new leader's log contains it.

If a client's write was acknowledged before the leader failed, the entry is committed and survives. If the write was not acknowledged, it may be lost — the client must retry. The database never loses acknowledged data.

</details>

### Senior Level (5-8 years experience)

**What is expected:** You can compare Raft with alternatives (Paxos, Zab, EPaxos) and discuss operational concerns: cluster sizing, timeouts, performance optimization, failure recovery, and deployment topologies.

**Sample question:** "Design a fault-tolerant configuration service for 1,000 microservices. Justify your choice of consensus algorithm and cluster topology."

<details>
<summary>Model Answer</summary>

I would use Raft with a 5-node cluster spread across 3 availability zones (2:2:1 distribution). Raft is the right choice here because: it is well-understood and simpler to debug than Paxos, the single-leader model matches the configuration workload (writes are rare, reads dominate), and the safety guarantees protect configuration consistency.

The 5-node cluster tolerates 2 node failures, which covers an entire AZ failure in the 2:2:1 deployment. Election timeout is set to 300-500ms to handle the inter-AZ latency of 1-5ms.

For the 1,000 microservices reading configuration: each service connects to any node in the cluster. Writes go to the leader. Reads can use the read-index optimization: a follower asks the leader for the current commit index, waits until its state machine reaches that index, then serves the read. This distributes read load across all 5 nodes without sacrificing consistency.

For operational safety: enable pre-vote to prevent spurious elections, set snapshot threshold to 10,000 entries, and configure monitoring alerts on term changes and election rates.

</details>

### Staff+ Level (8+ years experience)

**What is expected:** You can design novel extensions to Raft, deeply reason about edge cases, and discuss real-world production incidents involving Raft. You understand when Raft is the wrong choice.

**Sample question:** "Design a globally distributed database that supports multi-region writes with strong consistency. Raft's single-leader model cannot support this directly — how would you extend it?"

<details>
<summary>Model Answer</summary>

This requires a multi-Raft architecture with data partitioned into ranges, each range as its own Raft group. For multi-region writes, I would use CockroachDB's approach: each range has a leaseholder (serves reads and coordinates writes) that can be in a different region than the Raft leader. The Raft leader handles log replication while the leaseholder handles client coordination.

For cross-region transactions spanning multiple ranges, use a two-phase commit protocol where the transaction coordinator commits the prepare phase through Raft, then commits the final phase through Raft. The parallel commits optimization eliminates one Raft round trip by committing optimistically.

To handle region failures: the system automatically rebalances range replicas away from the failed region. The Raft leader lease movement ensures that even if an entire region goes offline, the remaining regions can form majorities for all ranges.

</details>

## Phrase Bank

Use these phrases to demonstrate depth during interviews.

**"Raft decomposes consensus into leader election, log replication, and safety."** — Shows you understand the three subproblems.

**"The log matching property guarantees that if two entries have the same index and term, they are identical."** — Shows you understand the fundamental invariant.

**"Raft uses a strong leader: all writes go through the leader, which simplifies the protocol."** — Shows you understand the architectural choice vs. Paxos.

**"Committed entries survive leader changes because of the intersection of majorities."** — Shows you understand the safety proof.

**"The randomized election timeout prevents split votes statistically."** — Shows you understand the failure mode and mitigation.

**"Pre-vote prevents a partitioned node from disrupting a healthy cluster with a higher term."** — Shows awareness of production Raft extensions.

**"Raft is designed for understandability, not just correctness."** — Shows you know the design philosophy.

**"The commitment rule requires a current-term entry to be committed before previous-term entries are considered committed."** — Shows deep understanding of a subtle point.

**"In a cross-region deployment, I would tune the election timeout to at least 4x the measured inter-region RTT."** — Shows practical operational knowledge.

**"Raft performance is bounded by the leader's fsync rate and network bandwidth to followers."** — Shows awareness of performance characteristics.

**"Multi-Raft splits the key space into ranges, each with its own Raft group and leader."** — Shows understanding of the scaling pattern.

**"A 5-node cluster tolerates 2 node failures, while a 3-node cluster tolerates only 1."** — Shows practical sizing knowledge.

**"Raft uses an AppendEntries RPC for both heartbeat and log replication."** — Shows protocol-level knowledge.

**"The InstallSnapshot RPC handles the case where a follower has fallen behind the leader's log compaction point."** — Shows knowledge of log compaction.

**"Read-index optimization allows followers to serve linearizable reads without forwarding to the leader."** — Shows understanding of read optimization.

## Red Flags

Avoid these mistakes in interviews.

**"Raft guarantees there is always exactly one leader."** — Wrong. During elections there is zero leaders. A brief period with no leader is normal.

**"Raft is faster than Paxos."** — Incorrect. They provide the same guarantees with similar performance. Raft is designed for understandability, not speed.

**"All nodes in Raft are equal."** — Wrong. Raft has asymmetric roles: leader, candidate, follower. The leader has unique responsibilities.

**"Raft stores all data on all nodes."** — Oversimplified. Raft replicates a log of commands, not the data itself. The state machine is derived from the log.

**"Raft handles network partitions by letting both sides keep working."** — Wrong. Only the majority side can make progress. The minority side blocks writes.

**"Once a leader is elected, it stays leader forever."** — Wrong. Leaders can fail, lose connectivity, or discover a higher term and step down.

**"Raft guarantees exactly-once delivery."** — Wrong. Raft provides at-least-once. Exactly-once requires client-side deduplication with request IDs.

**"Cluster membership changes are easy in Raft."** — They require careful handling via joint consensus to prevent split-brain scenarios.

## Practice Drills

### Drill 1: The Whiteboard Walkthrough

Walk through a complete write lifecycle on a whiteboard:
1. Draw 3 nodes in a Raft cluster.
2. Write "PUT x=5" from a client.
3. Walk through the leader receiving it, appending to log, replicating to 2 followers, receiving 2 acks, committing, and responding.
4. Now crash the leader before replication completes. What happens?
5. Now crash the leader after commit. What happens?

### Drill 2: Failure Mode Diagnosis

Given a cluster diagnosis, identify the problem:

- **Scenario:** A 5-node cluster handles 100 ops/sec normally. Suddenly throughput drops to 10 ops/sec and the leader changes every 2-3 seconds.
- **Diagnosis:** Leader flapping caused by network jitter, disk latency spikes, or GC pauses. Check network latency between nodes, disk IOPS, and GC logs.

### Drill 3: Configuration Tuning

You have a 3-node Raft cluster across 3 AWS AZs with 2ms inter-AZ latency. Your workload is 5,000 writes/sec. What election timeout do you choose? What heartbeat interval?

**Answer:** Election timeout = 300-500ms (10x the latency for safety margin). Heartbeat = 50ms (6-10x faster than timeout). Cluster of 3 is appropriate for this throughput and latency profile.

### Drill 4: Raft vs. Alternatives

For each scenario, decide: Raft, Multi-Raft, Paxos, or none (use async replication)?

- **Scenario A:** Configuration store for a 100-node cluster, 100 writes/sec, 10,000 reads/sec.
  **Answer:** Single Raft group (3 or 5 nodes). Write throughput is well within Raft's single-group limits.

- **Scenario B:** Global SQL database with 1M writes/sec across 10 regions.
  **Answer:** Multi-Raft with per-range groups, leaseholder for locality, and geo-partitioned ranges.

- **Scenario C:** Image sharing platform where eventual consistency is acceptable.
  **Answer:** No Raft. Use asynchronous replication or CRDTs. Raft's coordination cost is unnecessary overhead.
