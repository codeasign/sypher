## The Problem Story

Imagine you are on a distributed systems team at a rapidly growing company. Your product uses a configuration store that must remain consistent across all servers: if one server writes a new routing rule, every server must see exactly that rule, in exactly the same order. The team chooses Paxos, the long-standing consensus algorithm proven correct by Leslie Lamport in 1998. What happens next is a cautionary tale.

The first engineer tasked with implementing Paxos spends six weeks reading the original paper. The algorithm is described in terms of legislative bodies, decrees, and ballots вҖ” a metaphor that obscures more than it reveals. The engineer produces an implementation that passes unit tests but fails catastrophically under network partitions. A second engineer, reviewing the code, cannot determine whether the logic is correct because the mapping between the formal protocol and the code is opaque. Three more months pass. The team ships a second implementation that passes an extended test suite. In production, a node failure triggers a rare edge case: two nodes each believe they are the proposer for the same ballot number, and the cluster enters a state where no value can be committed. The team spends two weeks reproducing the bug, then another month patching it.

This story repeats across the industry. Paxos is notoriously difficult to implement correctly. Google's Chubby lock service uses Paxos, but its lead engineer noted that even after years of production experience, subtle bugs remained. Amazon's DynamoDB team chose an alternative path. Apache ZooKeeper uses Zab, a protocol similar to Paxos but designed for practical implementation.

In 2014, Diego Ongaro and John Ousterhout at Stanford published Raft. Their goal was not to invent a fundamentally different consensus algorithm, but to design one that could be understood completely by a typical computer science student in a single session. Raft achieves this by decomposing consensus into three independent subproblems and using a strong leader approach that mirrors how human groups naturally reach agreement.

**Raft is consensus designed for understandability first, correctness second вҖ” and it achieves both.**

## The Insight

A classroom needs to decide on a date for the final exam. The teacher asks for a show of hands for three proposed dates. One date receives the most votes. The teacher announces the result and writes it in the class gradebook. If the teacher is absent, the students elect a new teacher. The new teacher reviews the gradebook to see what was already decided, then continues managing the class.

This everyday scenario captures the essence of Raft.

Raft decomposes the consensus problem into three independent subproblems:

**Leader election:** The cluster must pick a single node to coordinate decisions. If the existing leader fails, a new leader must be elected quickly and correctly. Raft uses randomized timers to ensure elections converge rapidly.

**Log replication:** The leader accepts client requests, records them as log entries, and replicates these entries to the follower nodes. When a majority of nodes have stored the entry, the entry is committed and applied to the state machine.

**Safety:** Raft guarantees that if any node has applied a particular log entry at a particular index, no other node can apply a different entry at that same index. This is enforced through the Leader Completeness Property: a leader must contain all committed entries from previous terms.

The insight is that strong leadership simplifies consensus dramatically. In Paxos, any node can propose values at any time, which forces the protocol to handle a combinatorial explosion of concurrent proposals. Raft restricts proposal authority to a single elected leader, reducing the problem to: keep the leader alive, replicate its log to followers, and ensure the next leader inherits all committed state.

## Core Vocabulary

Master these 20 terms before moving to the deep dive.

**Leader** вҖ” The node that manages the cluster. It receives all client requests, replicates log entries to followers, and decides when entries are committed. There is at most one leader per term.

**Candidate** вҖ” A node that has started an election. It requests votes from other nodes. If it receives votes from a majority, it becomes the leader.

**Follower** вҖ” A passive node that accepts log entries from the leader and votes in elections. Followers do not initiate client interactions.

**Term** вҖ” A logical clock divided into epochs. Each term begins with an election. If the election succeeds, the term continues with the elected leader. If the election results in a split vote, the term ends immediately and a new term begins.

**Log Entry** вҖ” A single record in the replicated log. Each entry contains a command (the operation to apply to the state machine) and the term number when the entry was created.

**Commit** вҖ” A log entry is committed when the leader has replicated it to a majority of nodes. Committed entries are durable and will eventually be applied to the state machine.

**Apply** вҖ” The act of executing a committed log entry against the state machine. The state machine executes commands in the exact order of the log.

**Quorum** вҖ” A majority of nodes in the cluster: floor(N/2) + 1. In a 5-node cluster, quorum is 3. All committed decisions require a quorum.

**Election Timeout** вҖ” The duration a follower waits before starting an election. Randomized between 150ms and 300ms by default. This randomization prevents split votes.

**Heartbeat** вҖ” Periodic messages sent by the leader to followers to maintain authority. If a follower does not receive a heartbeat within the election timeout, it assumes the leader is dead and starts an election.

**Split Vote** вҖ” An election scenario where no candidate receives a majority because votes are distributed across multiple candidates. Each candidate times out and starts a new election with a new randomized timeout.

**Log Compaction** вҖ” The process of removing old log entries after they have been applied to the state machine. Raft uses snapshotting: the entire state machine state is written to a snapshot, and log entries up to the snapshot point are discarded.

**RequestVote RPC** вҖ” The remote procedure call used during leader elections. A candidate sends RequestVote to all other nodes, asking for their vote.

**AppendEntries RPC** вҖ” The remote procedure call used for log replication and heartbeats. The leader sends AppendEntries to followers containing new log entries (or empty for heartbeats).

**Match Index** вҖ” Per-follower tracking variable held by the leader. It records the highest log index known to be replicated on that follower.

**Next Index** вҖ” Per-follower tracking variable held by the leader. It records the next log entry to send to that follower.

**Log Matching Property** вҖ” A fundamental invariant: if two nodes have the same entry at the same log index, they have identical logs up to that index. Raft ensures this by checking term and index consistency during AppendEntries.

**Election Safety** вҖ” At most one leader wins per term. Only a node that receives a majority of votes becomes leader, and a node votes for at most one candidate per term.

**Leader Completeness** вҖ” A committed log entry is present on at least one node that votes for the new leader. Combined with the restriction that a candidate cannot become leader unless its log is at least as up-to-date as a majority of nodes, this guarantees that committed entries are never lost.

**State Machine Safety** вҖ” If a node has applied a log entry at a particular index, no other node can apply a different entry at the same index. This follows from the combination of leader completeness and the log matching property.

## How It Works

Raft proceeds through a continuous cycle of leader election, log replication, and safety enforcement.

### Phase 1: Leader Election

A Raft cluster begins with all nodes as followers. When a follower does not hear from a leader (no heartbeats received within the election timeout), it transitions to candidate state and starts an election.

**Step 1 вҖ” A follower detects leader failure.** Node A's election timer expires. It has not received a heartbeat from the leader for 200ms (the randomized timeout value).

**Step 2 вҖ” The candidate requests votes.** Node A increments its current term from 3 to 4, sets its vote to itself, and sends RequestVote RPCs to all other nodes in parallel. The RPC includes Node A's term (4), its last log index (7), and its last log term (3).

**Step 3 вҖ” Nodes decide whether to vote.** Each receiving node compares its own log with the candidate's log. A node grants its vote only if: (a) the candidate's term is at least as large as the node's current term, and (b) the candidate's log is at least as up-to-date as the node's log (last term is higher, or last term equal and log at least as long).

**Step 4 вҖ” The candidate counts votes.** Node A waits for responses. If it receives votes from a majority of the cluster (3 out of 5), it becomes the leader. If it receives a response from a node with a higher term, it reverts to follower.

**Step 5 вҖ” The leader establishes authority.** The new leader immediately sends heartbeats (empty AppendEntries RPCs) to all followers to prevent new elections.

### Phase 2: Normal Operation with Log Replication

Once a leader is elected, the cluster enters normal operation. The leader handles all client requests.

**Step 6 вҖ” Client sends a command.** A client sends a write request (for example, PUT x=42) to the leader. If a client contacts a follower, the follower rejects the request and returns the leader's identity.

**Step 7 вҖ” Leader appends to its log.** The leader creates a new log entry containing the command and the current term number. It appends this entry to its own log.

**Step 8 вҖ” Leader replicates to followers.** The leader issues parallel AppendEntries RPCs to all followers. The RPC includes the new entry, the leader's commit index, and the index and term of the preceding entry (for consistency checking).

**Step 9 вҖ” Followers validate and acknowledge.** Each follower checks that the preceding log entry matches its own log at that index (log matching property). If it matches, the follower appends the new entry to its log and acknowledges success. If it does not match, the follower rejects the request.

**Step 10 вҖ” Leader commits upon majority.** The leader waits for acknowledgments. When a majority of nodes (including the leader) have stored the entry, the leader marks the entry as committed at that index. It then applies the entry to its state machine.

**Step 11 вҖ” Leader responds to client.** The leader sends the success response back to the client.

**Step 12 вҖ” Leader notifies followers of commit.** On subsequent AppendEntries RPCs (including heartbeats), the leader includes its current commit index. Followers apply any newly committed entries to their own state machines.

### Phase 3: Log Replication with Recovery

When a follower's log diverges from the leader's (because the follower was offline or the leader crashed mid-replication), the leader must bring the follower up to date.

**Step 13 вҖ” Leader detects mismatch.** The leader sends AppendEntries with the next log entry for a follower. The follower responds with a rejection because the preceding entry does not match.

**Step 14 вҖ” Leader decrements nextIndex.** The leader decrements the nextIndex for that follower and retries with the previous entry. This continues until the leader finds a log index where the follower's log matches.

**Step 15 вҖ” Leader overwrites divergent entries.** Once the matching point is found, the leader sends all entries from that point forward. The follower overwrites any conflicting entries and appends the new ones.

**Step 16 вҖ” Consistency restored.** The follower's log now matches the leader's log exactly. Normal replication continues.

### Phase 4: Safety Guarantees

**Step 17 вҖ” Election restriction.** A candidate cannot become leader unless its log contains all committed entries. During election, the voter compares the candidate's last log term and index with its own. The candidate with the more up-to-date log wins.

**Step 18 вҖ” Committed entry preservation.** If a leader has committed an entry (replicated to a majority), that entry is present on at least one node that voted for the new leader. The new leader therefore inherits it.

**Step 19 вҖ” State machine safety.** Since all committed entries are preserved across leadership changes, and the log matching property ensures all nodes that have a given entry agree on its content, the state machine applies entries in identical order on all nodes.

### Phase 5: Log Compaction (Snapshotting)

**Step 20 вҖ” Snapshot trigger.** When the log grows beyond a configurable threshold (for example, 10,000 entries), or on a periodic schedule, the leader (and optionally followers) takes a snapshot of the current state machine state.

**Step 21 вҖ” Snapshot creation.** The node serializes the entire state machine state to a persistent snapshot file. It records the last included index (the index of the last entry applied to the state machine) and the last included term.

**Step 22 вҖ” Log truncation.** All log entries up to the last included index are discarded. The snapshot plus remaining log entries represent the complete system state.

**Step 23 вҖ” InstallSnapshot RPC.** When a new follower joins or a slow follower has fallen behind the leader's snapshot point, the leader sends an InstallSnapshot RPC containing the snapshot data. The follower replaces its state machine with the snapshot and discards its log up to the snapshot point.

## Variants

### Multi-Raft

A single Raft group is limited by the throughput of its leader. Multi-Raft addresses this by partitioning data across multiple independent Raft groups, each with its own leader. CockroachDB uses this approach: each range (a contiguous chunk of data) is its own Raft group, and ranges are automatically rebalanced across nodes.

Multi-Raft introduces the concept of a Raft store per node вҖ” a shared state machine that hosts multiple Raft groups on the same node. Each group maintains its own log, commit index, and election timers. The shared storage layer efficiently persists logs for all groups together.

### Raft with Pre-Vote

A standard Raft election can disrupt a healthy cluster if a partitioned node reconnects with a higher term and forces an unnecessary election. Pre-vote addresses this: a candidate first checks if it can win an election by sending a pre-vote request. Nodes respond based on log freshness without incrementing their term or changing their vote state. Only if the pre-vote succeeds does the candidate start a real election.

### Raft with Joint Consensus

Cluster membership changes (adding or removing nodes) are difficult in consensus protocols because the old and new configurations may disagree on what constitutes a majority. Joint consensus handles this by transitioning through an intermediate configuration where both the old and new quorums must agree. The leader replicates a joint consensus entry (combining old and new member sets), commits it once both old and new majorities acknowledge it, then replicates the new configuration alone.

### Read-Only Leader Optimization

By default, Raft reads go through the leader to guarantee linearizability. But every read still requires a round to the leader, which handles writes. Optimizations include:

**Leader leasing:** The leader assumes no other leader can exist for a specific duration after its election. It can serve reads locally without contacting followers. The lease duration is set to be less than the minimum election timeout.

**Read indices:** The leader records its current commit index, sends a lightweight heartbeat to confirm leadership, and waits until the commit index advances to the recorded value. It then serves the read from its state machine.

**Follower reads with read indices:** Followers can serve reads if they know the leader's current commit index. The follower asks the leader for the current commit index, waits for its own state machine to reach that index, then serves the read.

## Mental Model Check

**Q1:** Why does Raft use randomized election timeouts?

<details>
<summary>Model Answer</summary>

Randomized election timeouts prevent split votes. Without randomization, after a leader fails, all followers would start their elections at nearly the same time. Each would vote for itself and receive roughly the same number of votes, resulting in a split vote where no candidate wins a majority. The cluster would hold repeated elections without making progress. With randomized timeouts (typically 150-300ms), one follower's timer expires first, giving it time to collect votes before others transition to candidate state.

</details>

**Q2:** What happens to uncommitted log entries when a leader crashes?

<details>
<summary>Model Answer</summary>

Uncommitted log entries that existed only on the crashed leader are lost. The new leader will not have these entries because they were never replicated to a majority. If some followers received the entries, the new leader will overwrite them during log matching. The client will receive a timeout or error for the original request and must retry. This is safe because the entries were never committed вҖ” no node applied them to its state machine.

</details>

**Q3:** How does Raft guarantee that committed entries survive leader changes?

<details>
<summary>Model Answer</summary>

Raft guarantees this through the Leader Completeness Property. A committed entry is replicated to a majority of nodes. For a candidate to become leader, it must receive votes from a majority. By the intersection property of majorities, at least one node in the voting majority has the committed entry. The candidate's log must be at least as up-to-date as the voter's log (the voter compares last log term and index). Therefore, the candidate's log must contain the committed entry. Once elected, the new leader makes the entry available to all followers.

</details>

**Q4:** Can a follower serve reads directly?

<details>
<summary>Model Answer</summary>

A follower cannot serve linearizable reads directly from its state machine because it may not know the latest committed index. The follower's commit index lags behind the leader's. However, follower reads are possible with the read-index optimization: the follower asks the leader for the current commit index, then waits for its own state machine to apply up to that index before responding. This preserves linearizability while offloading read traffic from the leader. Without this optimization, all reads must go through the leader.

</details>

**Q5:** What is the difference between commit and apply in Raft?

<details>
<summary>Model Answer</summary>

Commit and apply are two distinct phases. Commit is when the leader has verified that a log entry is stored on a majority of nodes. The entry is durable вҖ” even if the leader fails, the entry will survive and be made available by the next leader. Apply is the subsequent step where the committed entry is executed against the state machine (for example, actually setting x=42 in a key-value store). The leadership tracks a commitIndex (the highest committed log index) and a lastApplied (the highest applied log index). The apply process runs in the background, advancing lastApplied toward commitIndex.

</details>
