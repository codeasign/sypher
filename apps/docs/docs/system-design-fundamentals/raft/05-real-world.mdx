## Case Studies

### etcd — The Kubernetes Backbone

etcd is a distributed key-value store that uses Raft for consensus. It is the primary datastore for Kubernetes, storing all cluster state including pods, services, config maps, and secrets.

**Deployment pattern:** etcd runs as a 3-node or 5-node Raft cluster. Kubernetes writes every state change through etcd, typically at rates of 100-5,000 writes per second and 1,000-50,000 reads per second in production clusters.

**Lessons learned:**
- **Defragmentation is mandatory.** etcd's MVCC backend creates version bloat that can consume gigabytes of storage. Without periodic defragmentation, the database grows linearly with the number of key versions, degrading performance.
- **Read optimization matters.** Kubernetes reads config maps frequently. etcd serves linearizable reads with read-index optimization, bypassing disk writes for read-only operations.
- **Heartbeat tuning for large clusters.** With 5,000+ Kubernetes nodes all watching etcd for changes, the watch stream can overwhelm the leader's network stack. etcd uses a streaming gRPC API and compacted event history to mitigate this.
- **The 8 GB database limit.** etcd has a hard limit of 8 GB for the database by default. Beyond this, snapshot creation times become impractical and compaction cannot keep up. Mitigation: use periodic compaction and defragmentation, or increase the limit with care.

### Consul — Service Discovery at Scale

HashiCorp Consul uses Raft for its consensus layer, providing service discovery, health checking, and key-value storage. Consul's Raft implementation is derived from the Hashicorp Raft library.

**Deployment pattern:** Consul runs a 3-node or 5-node Raft cluster as the server tier. Client agents on each node perform health checks and forward requests to the server cluster. Service discovery reads are served locally by client agents using cached data refreshed from the Raft cluster (eventual consistency for read performance).

**Lessons learned:**
- **Gossip protocol for failure detection.** Consul uses a separate gossip protocol (SWIM) for failure detection, not Raft heartbeats. This decouples membership detection from consensus, allowing Raft to focus on log replication.
- **Stale reads for scale.** Consul allows configuring read consistency: "default" is strongly consistent through the leader, "stale" allows any server to serve reads for lower latency, and "consistent" enforces leader-based linearizable reads.
- **The 5-node bottleneck.** In Consul's early days, 5-node clusters became a bottleneck for large deployments (10,000+ nodes) because all health check information had to pass through the Raft consensus layer. Modern Consul uses a separate streaming subsystem for health information.

### TiDB — Raft-Based Distributed SQL

TiDB is a distributed NewSQL database that uses Raft for consensus within each region (shard). It is inspired by Google Spanner but uses Raft instead of Paxos and a hybrid logical clock instead of TrueTime.

**Architecture:** TiDB uses a multi-Raft approach. The key space is divided into 64 MB regions, each managed by an independent Raft group. The Placement Driver (PD) manages region distribution and load balancing.

**Lessons learned:**
- **Raft log = write-ahead log.** TiDB made the Raft log the same as the storage engine's write-ahead log. This eliminates redundant writes (write amplification drops from 2x to 1x).
- **Parallel raft append.** Follower nodes can write to temporary storage and acknowledge before the data is fully applied, reducing perceived latency.
- **Learner nodes for read scaling.** TiDB introduced learner nodes (non-voting Raft members) that receive the log asynchronously and can serve stale reads without affecting write latency.
- **Dynamic region splitting.** A region exceeding 64 MB is automatically split into two smaller regions. Each new region starts its own Raft group with the same replica set.

### CockroachDB — Global Multi-Raft

CockroachDB is a distributed SQL database designed for global deployments. It uses Raft for per-range consensus, with ranges automatically rebalanced across nodes and regions.

**Key innovations in CockroachDB's Raft usage:**
- **Parallel commits:** Eliminates one network round trip from the write path for non-conflicting transactions. Instead of waiting for Raft commit then applying, the transaction is optimistically applied and committed via Raft simultaneously.
- **Follower reads:** CockroachDB allows reading from any replica as long as the read timestamp is within a safe interval (defined by the closed timestamp mechanism, which tracks the maximum timestamp at which data can be safely served from followers).
- **Range lease:** Separates the range lease (which replica serves reads and coordinates writes) from the Raft leader. This enables geographic read locality: a replica in the nearest data center holds the range lease while a replica in a different region holds the Raft leadership.

## Open Source Raft Implementations

### Hashicorp Raft (Go)

The most widely used standalone Raft library. Powers Consul, Nomad, Vault, and many other HashiCorp products. Features include: automatic leader election, log replication and snapshotting, cluster membership changes, and a pluggable transport and storage layer.

### etcd raft library (Go)

A highly optimized Raft library extracted from etcd. Used by TiDB (via TiKV) and many other projects. Features include: Raft groups in a shared core, unlimited log entry retention with async snapshotting, and checkpoint-based log compaction.

### Apache Ratis (Java)

A Raft implementation in Java used by Apache Ozone and Apache Hadoop. Features include: gRPC and Netty transport, state machine machine interface, and multi-Raft support.

## Cloud Providers and Raft

**AWS MemoryDB for Redis:** Uses Raft for cluster metadata consensus and primary election. Each shard has a primary node (writes) and up to 5 replicas. Raft elects a new primary if the current one fails.

**Google Cloud Spanner:** Uses Paxos, not Raft, but the lessons are similar. Spanner's TrueTime API provides clock synchronization that eliminates the need for Raft-style randomized timeouts. This is a key advantage for global deployments.

**Azure Cosmos DB:** Uses a custom consensus protocol based on Paxos, not Raft. The design is optimized for multi-region writes, which Raft's single-leader model cannot support.

## Lessons Learned from Production Deployments

**Lesson 1 — Tune timeouts to your network, not the defaults.** The Raft paper's 150-300ms defaults work for low-latency datacenters. Cross-region deployments need 500-1000ms, and edge deployments may need 2-5 seconds. Measure actual network latency jitter before setting timeouts.

**Lesson 2 — Monitor term changes.** A healthy Raft cluster has extremely rare term changes (effectively zero after initial leader election). Unexpected term increments indicate leader flapping, network issues, or GC pauses. Alert on any term change not caused by planned maintenance.

**Lesson 3 — Plan for quorum loss.** A 5-node cluster tolerates 2 node failures. When you lose 2 nodes, replace them immediately. Running with 3 of 5 nodes means any additional failure causes complete unavailability. Have automated node replacement procedures.

**Lesson 4 — Log compaction is not optional.** Without snapshots, the Raft log grows unbounded. A 100,000-entry log at 200 bytes each is trivial, but a cluster running for months at 10,000 writes/sec accumulates billions of entries. Compaction (snapshotting) must be configured from day one.

**Lesson 5 — Benchmark with production conditions.** A Raft cluster that handles 10,000 ops/sec in benchmarks may handle only 2,000 ops/sec under real workloads with concurrent reads, watches, and client disconnections. Benchmark with your actual workload pattern, including network latency, disk write amplification, and concurrent access patterns.
