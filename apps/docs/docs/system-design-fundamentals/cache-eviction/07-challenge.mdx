# Cache Eviction — Challenges

The interview page prepared you to answer questions. This page tests your ability to apply cache eviction knowledge to real design problems. Work through each tier sequentially. Solutions are provided after each challenge.

---

## Tier 1 — Foundation

These challenges test whether you understand the basic eviction policies and can select the right one for a given workload.

### Challenge 1.1: Policy Identification

For each workload description, identify the best eviction policy and explain why.

1. A cache that holds user authentication tokens. Each token is valid for 15 minutes and is accessed every 30-60 seconds during an active session.

2. A cache that stores the top 10,000 products on an e-commerce site. The product rankings change weekly. Each product in the top 10,000 receives 100-10,000 requests per day.

3. A cache that stores the results of database queries in a data analytics pipeline. The pipeline runs the same 10 queries repeatedly for 1 hour, then switches to a different set of 10 queries.

4. A cache that stores blog post content. New blog posts get a burst of traffic in the first 24 hours, then are rarely accessed again.

5. A cache that stores configuration values that change once per week. The values are read 10,000 times per second by every request.

<details>
<summary>Solution for Challenge 1.1</summary>

1. **LRU with TTL.** Auth tokens have strong temporal locality: an active session accesses the token every 30-60 seconds. When the session ends, the token is never accessed again. LRU evicts expired sessions naturally. The 15-minute TTL ensures tokens do not live beyond their validity period.

2. **LFU with weekly frequency decay.** Product popularity is relatively stable. The most popular products get 100x more traffic than the least popular. LFU retains the high-traffic products. Weekly frequency decay accounts for ranking changes.

3. **ARC.** The pipeline cycles through different query sets hourly. No single static policy fits because the hot set changes every hour. ARC adapts to each query set's access pattern without manual reconfiguration.

4. **2Q or SLRU.** Blog post traffic has a burst pattern: high traffic initially, then a long tail of rare accesses. The post gets one burst of accesses (promoting it to the protected segment), then it ages out naturally. Scans of new posts do not pollute the protected segment.

5. **LFU.** Configuration values have the most stable access pattern possible: they are read constantly and changed rarely. LFU retains them in the highest frequency buckets indefinitely, ensuring near-zero eviction for these critical entries.
</details>

### Challenge 1.2: Eviction Rate Analysis

Your cache monitoring shows the following metrics:

- Current memory usage: 3.7 GB (limit: 4 GB)
- Eviction rate (last hour): 450 evictions/second
- Write rate (last hour): 500 writes/second
- Hit ratio (last hour): 62%
- Average entry size: 2.5 KB
- Average entry lifetime before eviction: 45 seconds

Is this cache healthy? If not, diagnose the problem and propose a fix.

<details>
<summary>Solution for Challenge 1.2</summary>

**Diagnosis:** The cache is unhealthy. Key findings:

1. **Eviction rate is 90% of write rate (450/500).** This means almost every write triggers an eviction. This is the definition of cache thrashing.

2. **Average entry lifetime before eviction is 45 seconds.** Entries are being evicted shortly after creation, before they have a chance to be re-accessed.

3. **Hit ratio is 62%.** For a cache of this size (4 GB), a healthy hit ratio should be 85-95%.

4. **The working set is larger than the cache.** With 500 writes/second and 2.5 KB per entry, the cache fills in approximately 3,200 seconds (4 GB / 1.25 MB per second). But entries live only 45 seconds, meaning the working set is approximately 500 * 45 * 2.5 KB = 56 MB. Wait -- this calculation contradicts the diagnosis. Let me re-examine.

   Actually, 500 writes/second * 2.5 KB = 1.25 MB/second written. In 45 seconds (average entry lifetime), that is 56 MB of data written. But the cache has 4 GB. Something else is happening.

   The average entry lifetime of 45 seconds despite only 56 MB of data being written in that window suggests the cache is being filled by a process that writes data that is never read. Possibly a batch job or a background task is flooding the cache with one-hit-wonders.

**Root cause:** The high write rate (500 writes/second) combined with low hit ratio (62%) suggests cache pollution. Most entries are evicted before they are read a second time.

**Fix options:**
1. **Check for batch jobs or scans.** If a background process is writing data that is never read, exclude it from the cache or give it a separate cache namespace with a smaller memory quota.
2. **Switch to a scan-resistant policy.** 2Q or SLRU would protect the hot set from one-hit-wonder pollution.
3. **Add TTLs.** If data has a natural expiration (e.g., API responses with Cache-Control headers), setting TTLs would cause entries to self-evict rather than waiting for LRU eviction.
4. **Reduce write rate.** Not all data needs to be cached. Consider which data accesses benefit most from caching and only cache that data.
</details>

### Challenge 1.3: Memory Overhead Calculation

Your cache stores 5 million entries. Each entry's key is 32 bytes and value is 1,000 bytes. The cache uses LRU eviction with a doubly-linked list and hash map. Calculate:

1. The total memory used by data (keys + values).
2. The total memory overhead for the LRU policy (pointers, hash map entries).
3. The total cache memory usage (data + overhead).
4. The overhead percentage.

Assume 64-bit architecture, 8-byte aligned. An LRU node has: prev pointer (8 bytes), next pointer (8 bytes), key pointer (8 bytes), value pointer (8 bytes), size field (4 bytes), last-access timestamp (8 bytes), hash map next pointer (8 bytes).

<details>
<summary>Solution for Challenge 1.3</summary>

**Data memory:**
- Keys: 5,000,000 * 32 bytes = 160 MB
- Values: 5,000,000 * 1,000 bytes = 5,000 MB
- Total data: 5,160 MB

**Overhead per entry:**
- prev pointer: 8 bytes
- next pointer: 8 bytes
- key pointer: 8 bytes
- value pointer: 8 bytes
- size field: 4 bytes
- last-access timestamp: 8 bytes
- hash map next pointer: 8 bytes
- Total overhead per entry: 52 bytes

**Total overhead:** 5,000,000 * 52 = 260 MB

**Total cache memory:** 5,160 MB + 260 MB = 5,420 MB

**Overhead percentage:** 260 / 5,420 = 4.8%

**Analysis:** The overhead is approximately 5% of total memory. This is acceptable for most workloads. If this were a cache with very small values (e.g., 32-byte keys and 32-byte values), the overhead would be 52/(32+32+52) = 44.8%, which is significant. For small-value caches, consider using the clock algorithm instead of exact LRU to reduce overhead from 52 bytes per entry to approximately 16 bytes per entry.
</details>

## Tier 2 — Application

These challenges require designing a caching solution with eviction as a central consideration.

### Challenge 2.1: Design a CDN Edge Cache Eviction System

A CDN edge node has 500 GB of SSD storage. It serves 10,000 unique URLs per hour. Each URL has a Cache-Control header with a max-age of 1 hour to 1 year. The traffic pattern: 80% of requests go to 20% of URLs. The remaining 20% of requests are spread across 8,000 URLs.

Design the eviction system. Specify:
1. The eviction policy (or combination of policies).
2. How you handle TTL expiration versus LRU eviction.
3. How you prevent scan-based cache pollution.
4. How you handle content with no TTL (Cache-Control: no-cache but still cacheable).
5. The per-entry memory overhead you consider acceptable.

<details>
<summary>Solution for Challenge 2.1</summary>

1. **Eviction policy: Two-tier (TTL + LRU + probation).**

   - **TTL expiration (primary).** Every object with a Cache-Control max-age is evicted when the TTL expires. A background thread runs every 60 seconds, scanning for expired objects and freeing their storage.

   - **LRU eviction (secondary, when storage is full).** Maintain a global LRU list. When storage exceeds 90% capacity, evict from the LRU tail until storage drops below 80% (watermark approach).

   - **Probation queue (scan resistance).** New objects go into a FIFO probation queue (10% of storage). Objects accessed twice within the probation period are promoted to the main LRU cache. This prevents one-hit-wonders from polluting the hot set.

2. **TTL expiration vs LRU eviction.** TTL expiration is the primary mechanism. The background scanner removes expired objects before the LRU eviction is triggered. If the cache is large enough (500 GB, only 10K URLs/hour), TTL expiration should handle most evictions. LRU eviction only fires during traffic spikes or DDoS attacks.

3. **Scan prevention.** The probation queue is the primary defense. A scan sends unique URLs that are accessed once, never reaching the promotion threshold. The probation queue naturally cycles these out. Additionally, implement scan detection: if the rate of new unique URL requests exceeds 1,000/minute, throttle or reject requests without Cache-Control headers.

4. **No TTL content.** For Cache-Control: no-cache, set a default TTL of 60 seconds. This prevents indefinite caching while still benefiting from short-term reuse. When the default TTL expires, the object is revalidated with the origin.

5. **Per-entry overhead.** Aim for approximately 64 bytes per entry: URL hash (8), file offset (8), file size (8), insertion time (8), last access time (8), TTL (4), ref count (4), LRU pointers (16). Accept up to 128 bytes for the 20% hot URLs (more metadata for finer-grained LRU ordering).
</details>

### Challenge 2.2: Multi-Tenant Cache Eviction

You are designing a cache that serves 100 tenants (different customers). Each tenant has different cache needs:
- Tenant A: 10 GB working set, high read rate, low write rate, session data.
- Tenant B: 1 GB working set, very high write rate, data analytics results.
- Tenant C: 50 GB working set, moderate read/write rate, bursty traffic.
- Tenants D-J: 100 MB working set each, low traffic.
- Tenants K-100: 10 MB working set each, very low traffic.

The total cache memory is 100 GB. Design an eviction system that:
1. Prevents one tenant from starving others.
2. Respects each tenant's optimal eviction policy.
3. Maintains high overall hit ratio.
4. Minimizes per-tenant configuration overhead.

<details>
<summary>Solution for Challenge 2.2</summary>

**Architecture: Per-tenant memory quotas with shared overflow.**

1. **Hard quotas (fairness guarantee).** Each tenant gets a guaranteed minimum memory allocation:
   - Tenant A: 12 GB (10 GB working set + 20% headroom)
   - Tenant B: 1.5 GB
   - Tenant C: 15 GB (50 GB is too large -- cap at 15 GB)
   - Tenants D-J: 200 MB each (100 MB + 100 MB headroom)
   - Tenants K-100: 20 MB each (10 MB + 10 MB headroom)
   - Total guaranteed: 12 + 1.5 + 15 + 7*0.2 + 90*0.02 = 30.9 GB

2. **Soft quotas (elastic overflow).** The remaining 69.1 GB is shared. Any tenant can use beyond its hard quota if there is free space. When memory pressure occurs, excess over hard quota is evicted first.

3. **Per-tenant eviction policy.**
   - Tenant A (session data): LRU + TTL (30-minute timeout)
   - Tenant B (analytics, high write): 2Q (scan-resistant, high write tolerance)
   - Tenant C (bursty): ARC (adapts to burst pattern)
   - Tenants D-100 (low traffic): LRU (default, simple)

4. **Eviction priority.** When the total cache exceeds 95% capacity:
   - First evict from tenants using more than their hard quota.
   - Within each tenant, use the tenant's configured eviction policy.
   - If all tenants are within quota, evict from the largest over-quota tenant first.

5. **Configuration overhead.** Use tenant-specific configuration files or database entries. A simple approach: expose the eviction policy selection as a cache client option. Each tenant sets its policy when it initializes its cache connection.

**Trade-offs:** Per-tenant policies add implementation complexity. For Tenants D-100 (low traffic), the added complexity is not worth it. A simpler approach would be to use tenant A, B, C with custom policies and a shared pool for everyone else.
</details>

### Challenge 2.3: Real-Time Eviction Policy Tuning

Your cache has 16 shards. Each shard uses LRU eviction with default settings (maxmemory-samples = 5). You notice that shards 1-8 have a 92% hit ratio, but shards 9-16 have a 65% hit ratio. The data distribution is uniform (all shards hold similar data volumes and access patterns).

Diagnose the problem and propose a solution that does not require shard-level configuration changes.

<details>
<summary>Solution for Challenge 2.3</summary>

**Diagnosis:** The problem is likely data access pattern skew, not configuration differences. Even though data volume is uniform, shard 9-16 may be serving a different access pattern. Possible causes:

1. **Hot keys concentrated on specific shards.** If a few hot keys (trending products, viral content) are hashed to shards 9-16, those shards experience higher access frequency and more eviction pressure.

2. **Write-heavy keys on shards 9-16.** If keys that are frequently written (counter updates, session refreshes) reside on shards 9-16, the write rate is higher, leading to more evictions.

3. **Scan operations targeting shard 9-16's key range.** If a batch job scans keys in the hash range assigned to shards 9-16, those shards experience cache pollution while others are unaffected.

**Solutions:**

1. **Increase maxmemory-samples on the cluster level.** From 5 to 10 or 15. This improves LRU accuracy for shards with high eviction rates. The CPU cost is low (5-15 extra comparisons per eviction). This helps all shards, but especially benefits the struggling ones.

2. **Check for hot keys.** Use Redis's `--hotkeys` option (or equivalent) to identify keys with exceptionally high access frequency. Consider splitting hot keys across multiple shards using key sharding (add a shard key suffix to distribute hot keys).

3. **Add cache warming for shards 9-16.** Identify the top 1,000 keys accessed on shards 9-16 and pre-load them after any restart or rebalancing. This reduces the cold-start effect if shards 9-16 were recently rebalanced.

4. **If the pattern persists, consider using consistent hashing with virtual nodes.** Standard hash-based sharding places keys arbitrarily. Virtual nodes provide better key distribution, reducing the chance of hot key concentration.
</details>

## Tier 3 — Advanced

### Challenge 3.1: Design a Learning Eviction Policy

You are tasked with designing a next-generation eviction policy that adapts to workload changes faster than ARC. The policy should:

1. Detect changes in access pattern within 60 seconds.
2. Maintain 95%+ hit ratio across mixed workloads (session, scan, burst).
3. Use under 128 bytes of metadata per entry.
4. Not require manual configuration or tuning.

Propose an algorithm and explain how it handles each of the three workload types.

<details>
<summary>Solution for Challenge 3.1</summary>

**Proposed algorithm: Adaptive Hybrid with Sliding Window (AHSW).**

**Core idea:** Maintain three small sample caches (each 1% of total memory) running different policies in parallel: LRU, LFU, and 2Q. Monitor the hit ratio of each sample cache every 30 seconds. Use the policy with the best recent hit ratio for the main cache (97% of memory).

**Data structures:**
- Three sample caches (LRU, LFU, 2Q), each holding 1% of total capacity.
- Main cache (97% of capacity) using the currently active policy.
- Sliding window of hit ratios: per-policy hit ratio averaged over the last 60 seconds.
- Metadata per entry: key hash (8 bytes), value pointer (8 bytes), flags (4 bytes), access counter (4 bytes), last access timestamp (8 bytes), LRU/2Q pointers (16 bytes). Total: 48 bytes per entry.

**Algorithm:**

Every 30 seconds:
1. Compute hit ratio for each sample cache over the last 60-second window.
2. Select the policy with the highest hit ratio.
3. If the selected policy changed from the previous window, drain the main cache (gradually, over 60 seconds) and rebuild with the new policy.
4. When rebuilding, promote entries from the winning sample cache to the main cache first.

**How it handles each workload:**

1. **Session workload (temporal locality):** The LRU sample cache has a high hit ratio (90%+). The LFU sample is lower (sessions expire, so frequency does not accumulate). The 2Q sample is comparable to LRU. The algorithm selects LRU for the main cache.

2. **Scan attack:** The LRU sample cache's hit ratio crashes (from 90% to 20%) as the scan evicts the sample. The 2Q sample cache is unaffected (its probation queue absorbs the scan). Within 30-60 seconds, the algorithm switches from LRU to 2Q, protecting the main cache from pollution.

3. **Burst traffic (flash crowd):** A flash crowd of requests for the same popular items causes all three sample caches to perform well initially. If the burst traffic introduces new items that are not in the cache, the LRU sample adapts fastest (recency-based), so it is selected. The algorithm handles the burst within one 30-second window.

**Trade-off:** The sample caches consume 3% of memory (3 * 1% = 3%), which is wasted on non-production data. However, this is a small price for adaptive behavior that handles mixed workloads.
</details>

### Challenge 3.2: Eviction in a Write-Back Cache

Your system uses a write-back cache: data is written to the cache first, then asynchronously flushed to the database. The cache has a dirty data queue of entries that have been modified but not yet flushed to the database.

When the cache is full and eviction selects a dirty entry, the system must either:
1. Flush the dirty entry to the database before evicting (adds latency).
2. Evict the dirty entry without flushing (loses data).
3. Skip dirty entries and evict clean ones (may not find a clean victim).

Design the eviction policy for a write-back cache with the following requirements:
- Data must not be lost (durability requirement).
- Eviction should not add more than 10ms of latency to the write path.
- The cache must maintain at least 80% hit ratio.

<details>
<summary>Solution for Challenge 3.2</summary>

**Architecture: Two-stage eviction with background flush.**

1. **Stage 1: Clean eviction (fast path).** Maintain a separate LRU list for clean entries. When eviction is needed, evict from the clean LRU list first. This is fast (no flush needed) and meets the 10ms latency requirement. If the clean list has entries, eviction completes immediately.

2. **Stage 2: Dirty flush (background).** Maintain a background thread that continuously flushes dirty entries from the tail of the dirty LRU list. The thread flushes at a rate matching the write rate (or slightly faster). When a dirty entry is flushed, it moves from the dirty LRU to the clean LRU.

3. **Emergency eviction (rare).** If the clean list is empty (all entries are dirty) and eviction is needed, flush the least recently written dirty entry synchronously. This is slow (~50ms for a database write) but should be rare if the background flush thread keeps up.

**Policy details:**

- **Clean list:** LRU. Percentage of total cache: dynamically allocated based on the clean-to-dirty ratio. Target: at least 10% of entries are clean.
- **Dirty list:** LRU ordered by last modification time. The background thread flushes from the tail (oldest dirty entries first).
- **Background flush rate:** Target: flush as many entries per second as are being written. If write rate is 1,000/second, flush rate should be 1,100/second (10% cushion).
- **Watermark:** When clean entries drop below 5% of total, the background flush rate doubles. When clean entries drop to 1%, synchronous flushes trigger.

**Durability guarantee:** No dirty entry is evicted without being flushed. If the clean list is empty and the dirty list has entries, the system waits for the synchronous flush to complete. This may exceed the 10ms latency target, but it is a rare event (cushion: 99.9% of evictions should be from the clean list).

**Performance analysis:**
- 99.9% of evictions: clean eviction, under 1ms.
- 0.09% of evictions: background flush completes, under 10ms.
- 0.01% of evictions: synchronous flush, under 100ms (rare).

**Hit ratio:** By retaining dirty entries (which are likely to be re-accessed before they are flushed), the policy improves the hit ratio compared to immediately flushing all dirty entries. Expected hit ratio: 85-92%.
</details>

## Final Thoughts

Cache eviction is one of the most practical topics in system design. It is not a theoretical concept -- you will encounter it in every production cache you configure, every caching library you use, and every distributed system you design.

The key takeaways from these challenges:

- **Match the policy to the workload.** There is no universal best policy. Understand your access pattern before choosing.
- **Monitor the eviction rate.** It is the single most important metric for cache health.
- **Always set TTLs.** TTL reduces eviction pressure and prevents stale data accumulation.
- **Plan for failure modes.** Every policy can be attacked (scan, starvation, thrashing). Know the mitigations.
- **Start simple, adapt as needed.** LRU with TTL handles most workloads. Only add complexity (ARC, 2Q, custom policies) when the simple approach fails.
