## Naive Implementation

The simplest eviction implementation is FIFO using a queue. Every entry is pushed onto the tail of the queue when inserted. When eviction is needed, the entry at the front of the queue is popped and removed from the hash table.

### Failure at Each Scale

The naive FIFO implementation breaks at every meaningful scale:

**At 1K entries: Cold start thrashing.**

When the cache starts cold, every request is a miss. By the time you have processed 1,000 unique requests, the cache is full. The next new request evicts the oldest entry. If that oldest entry is part of the hot set (say, a popular product page), the next request for that product is a miss. The cache cycles through entries without establishing a stable hot set.

The hit ratio never exceeds 30-40%. The database sees sustained load equal to 60-70% of request volume.

**At 100K entries: FIFO removes hot data.**

At 100K entries, the cache now holds a meaningful amount of data. But FIFO evicts the oldest entries regardless of access frequency. A product page that is accessed 5,000 times per hour but was inserted 30 minutes ago is evicted in favor of a one-hit-wonder that was inserted 29 minutes ago.

FIFO has no concept of "hot" versus "cold." Every entry is equally likely to be evicted, regardless of how frequently it is accessed. The hit ratio plateaus at 50-60% even when the cache could sustain 90%+ with LRU.

**At 10M entries: Eviction storms.**

At 10M entries, the queue traversal is O(1) in theory, but in practice, memory bandwidth becomes the bottleneck. Evicting an entry involves:

1. Dequeue from the head (pointer update).
2. Remove from the hash table (hash computation, bucket traversal, pointer update).
3. Free the key string memory.
4. Free the value string memory.
5. Update memory counters.

Each eviction at 10M scale involves multiple cache misses (the hash table and queue may not be in L2 cache). Sustained eviction rates above 10,000 evictions per second show measurable latency impact on the application.

FIFO also has no protection against eviction loops: if the write rate exceeds the cache capacity, every write triggers an eviction, and the eviction time dominates the write latency.

### Per-Shard Eviction

In a distributed cache, each shard runs its own eviction engine independently. There is no global eviction coordinator. This is essential for performance: coordinating eviction across shards would require distributed consensus, which is too slow for cache operations.

Each shard has:
- Its own memory limit (typically the shard's portion of total cluster memory).
- Its own eviction policy (can be configured per shard for heterogeneous workloads).
- Its own eviction counters and hit ratio metrics.

### Global vs Local LRU

Global LRU: The eviction policy considers all keys in the cache, regardless of which client or application stored them. This is the default in most systems. The most recently used keys across the entire cache are retained.

Local LRU: The eviction policy operates within groups (e.g., per-database in Redis, per-application in a multi-tenant cache). Each group has its own LRU list. This prevents a write-heavy application from evicting a read-heavy application's data.

### Eviction Event Bus

When an entry is evicted, the shard publishes an event to the eviction event bus. Consumers of these events:
- **Monitoring:** Track eviction rates, identify hot keys being evicted.
- **Cache warming:** If a critical key is evicted, the warm-up process may pre-load it before the next request.
- **Database back-pressure:** The backing database may increase capacity if eviction rates exceed a threshold.
- **CDN purge:** If the evicted entry was a cached asset, the CDN may need to re-fetch the origin.

Redis uses an approximate LRU implementation. Instead of maintaining a globally sorted linked list (which would require O(log n) insertion per access), Redis samples `maxmemory-samples` (default 5) random keys and evicts the least recently used among the sample.

This approximation is within 5-10% of the optimal LRU in terms of hit ratio, while requiring significantly less overhead per access. Increasing `maxmemory-samples` to 10 reduces the gap to 2-3% at the cost of doubling eviction CPU.

### Memcached: LRU with Slab Allocator

Memcached combines LRU eviction with a slab allocator. The slab allocator divides memory into slabs of different sizes (64 bytes, 128 bytes, 256 bytes, etc.). Each slab class has its own LRU list.

When a write arrives:
1. The slab allocator selects the appropriate slab class based on the item size.
2. If the slab class is full, the LRU tail of that slab class is evicted.
3. The new item is inserted at the LRU head of that slab class.

This per-slab-class LRU ensures that evictions are proportional to item size. A write of 1,000 small items evicts 1,000 small items. A write of 1 large item evicts 1 large item. Without per-slab LRU, a write of 1 large item could evict many small items, damaging the hit ratio for small-item workloads.

### CDN Eviction: Purge API + TTL

CDN caches use time-based eviction (TTL from `Cache-Control: max-age`) as the primary mechanism, with LRU as the overflow mechanism.

```
CDN Eviction Hierarchy:
  1. TTL expiration (primary):
     - Content expires per Cache-Control: max-age
     - Can be overridden with Surrogate-Control: max-age
     - Minimum TTL: 0 seconds (no caching)
     - Maximum TTL: 1 year (static assets with versioned URLs)

  2. LRU eviction (secondary, when CDN edge node is full):
     - Evicts the least recently requested content
     - Edge nodes have ~1-10 TB of storage
     - LRU is per-edge-node (no global LRU)

  3. Purge API (manual, immediate):
     - CDN provider API: POST /purge?url=https://...
     - Purges specific URLs, URL patterns, or tags
     - Global: purges from all edge nodes simultaneously
     - Propagation time: 1-60 seconds across global edge network
```

CloudFront and CloudFlare support purge by URL or by tag. Tag-based purging (Purge by Tag) allows purging multiple related assets simultaneously without knowing their URLs.

### Browser Cache: HTTP Cache-Control

Browser caches use a combination of TTL and implicit eviction (browser-managed LRU).

```
HTTP Cache-Control Policies:
  Cache-Control: max-age=3600
    Browser caches for 1 hour. After 1 hour, the browser must revalidate.

  Cache-Control: no-cache
    Browser must revalidate every request (but can store the response).

  Cache-Control: no-store
    Browser must not store the response at all.

  Cache-Control: immutable
    Browser does not revalidate during the max-age lifetime. Used for
    versioned static assets (e.g., app.bundle.a1b2c3.js).

Browser eviction behavior (not configurable by developers):
  - Each browser sets its own cache size limit (Chrome: ~80% of disk space)
  - Eviction is LRU-based: oldest, least-accessed entries are removed
  - Entries with longer TTL may be evicted before expiration if disk is full
  - Service workers can manage cache via Cache API (explicit add/delete)
```

### Coordination-Free Eviction

Each region's cache operates independently. There is no cross-region eviction coordination. This is by design: cache operations must complete in under 1 millisecond. Cross-region RPC adds 50-200 milliseconds of latency, which is unacceptable for cache operations.

The cost of coordination-free eviction is that each region may make different eviction decisions. Region A may evict key X (which is cold in region A) while region B retains key X (which is hot in region B). This is acceptable because cache is eventually consistent -- the backing store is the consistent source of truth.

### Global Working Set

The global working set is the union of all regions' working sets. In most multi-region deployments, the working sets overlap significantly (popular content is popular everywhere), but with region-specific additions.

Capacity planning must account for:
- **Overlapping hot set:** The 1,000 most popular items are hot in all regions. Each region retains them independently.
- **Region-specific tail:** Items popular only in a specific region (local news, region-specific products).
- **Global unique set:** Items requested globally but infrequently per region. Each region may evict them independently, causing each region to fetch from the backing store.

## Capacity Planning

### 10M DAU Example

Assume 10 million daily active users, each making 100 requests per day. Total daily requests: 1 billion.

| Metric | Value | Calculation |
|--------|-------|-------------|
| Requests per second (peak) | 50,000 | 1B / 86,400 * 4 (peak factor) |
| Working set size | 20 GB | 10M users * 2 KB session data |
| Cache memory per node | 4 GB | Typical Redis/Memcached instance |
| Nodes needed (working set) | 5 | 20 GB / 4 GB |
| Nodes needed (headroom) | 10 | 2x for growth and eviction headroom |
| Overhead per entry | 52 bytes | LRU list pointers + hash table entry |
| Total overhead | 520 MB | 10M entries * 52 bytes |
| Evictions per second (steady) | 100 | 10% of entries turnover per day |
| Evictions per second (peak) | 500 | Traffic spikes + cache misses |
| Network bandwidth (eviction) | 10 Mbps | 500 evictions * 2.5 KB per entry |
| Memory bandwidth (eviction) | 50 MB/s | 500 evictions * 100 KB per entry |

### Eviction Rate Estimation

The eviction rate depends on:
- **Write rate:** Higher write rate = more evictions.
- **Cache size relative to working set:** Larger ratio = fewer evictions.
- **TTL distribution:** Data with short TTL self-evicts, reducing eviction pressure.
- **Access pattern skew:** 80/20 pattern = higher cache efficiency = fewer evictions.

Rule of thumb: If eviction rate exceeds 10% of the write rate, consider increasing cache size or switching to a scan-resistant eviction policy.

### Working Set Estimation

Estimate your working set before sizing the cache:

1. **Instrument your application.** Log every cache access with key, timestamp, and response size.
2. **Sample for 7 days.** A full week captures weekly patterns (weekday vs weekend).
3. **Compute unique keys per time window.** 1 hour, 1 day, 1 week. The 1-hour unique key count is the immediate working set. The 7-day unique key count is the total data footprint.
4. **Size the cache to the 1-hour working set.** The 1-hour working set is what the cache must hold to maintain a 90%+ hit ratio. The 7-day set tells you the maximum memory needed to cache everything (which is rarely practical).
5. **Add 30-50% headroom.** The cache needs headroom to absorb traffic spikes without entering emergency eviction mode.

```
Working Set Estimation Example:
  Hourly unique keys:  500,000  (immediate working set)
  Daily unique keys:   2,000,000 (daily footprint)
  Weekly unique keys:  5,000,000 (total data footprint)
  Avg entry size:      2.5 KB
  
  Cache size target:   500,000 * 2.5 KB * 1.5 (headroom) = ~1.9 GB
```

## Deployment

### Zero-Downtime Policy Changes

Changing the eviction policy requires caution. A policy change can cause an immediate drop in hit ratio as the cache repopulates under the new policy.

**Procedure for changing eviction policy in production:**

1. **Deploy the policy change to a single shard.** Use the canary shard (lowest traffic). This limits the blast radius if the new policy performs poorly.

2. **Warm the new policy.** The new policy may have different internal state (e.g., switching from LRU to LFU requires frequency counters to accumulate). The first 5-10 minutes after a policy change show lower hit ratios.

3. **Monitor:** Track eviction rate, hit ratio, p99 latency, and database load. Compare to the baseline.

4. **Gradually roll out.** If the canary shows stable results, apply the change to 25%, then 50%, then 100% of shards.

5. **Have a rollback plan.** If hit ratio drops below threshold, revert the policy change on the affected shards.

### Monitoring

Essential monitoring metrics for cache eviction:

**Eviction rate (evictions/second).** The primary indicator of cache health. A sudden increase signals that the working set has grown or the access pattern has changed. Alert when eviction rate exceeds 2x the baseline.

**Hit ratio.** The fraction of requests served from cache. The eviction policy directly impacts this metric. A sudden drop in hit ratio (5%+) signals that the policy is evicting data that is still being accessed.

**Memory pressure.** Current memory usage as a percentage of `maxmemory`. Consistently above 90% indicates the cache is too small or the working set has grown. Alert when sustained >95%.

**Eviction age.** The average age (time since creation) of evicted entries. A low eviction age (under 60 seconds) indicates cache thrashing: entries are being evicted shortly after creation, suggesting the cache is too small for the write rate.

**Policy effectiveness.** The ratio of evicted entries that are accessed again within their natural TTL. High values (over 10%) indicate the policy is evicting data that should have been retained.
