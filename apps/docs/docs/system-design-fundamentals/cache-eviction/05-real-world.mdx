## Company Case Studies

### Redis: Approximate LRU, LFU, and Volatile-TTL

Redis is the most widely used cache in production systems. Its eviction subsystem is a masterclass in practical engineering: it prioritizes performance over perfect accuracy, and it provides enough policy options to cover most workloads.

**Key design decisions:**

- **Approximate LRU over exact LRU.** Exact LRU requires a sorted linked list, which would require O(log n) insertion per access. Redis's approximate LRU samples 5 keys and picks the oldest. This is within 5% of exact LRU at a fraction of the overhead.

- **LRU and LFU share the same metadata field.** Redis stores a 24-bit timestamp (last access time) for LRU, or a 24-bit frequency counter for LFU, in the same field in the Redis object header. The field is interpreted as LRU or LFU based on the configured policy. This saves 3 bytes per entry.

- **No eviction per database.** Redis supports multiple logical databases (db0, db1, etc.) but eviction is global across all databases. If you need per-database eviction, run separate Redis instances.

- **Keyspace notification on eviction.** Redis can publish eviction events to subscribers via the keyspace notification system. This allows applications to react to evictions (e.g., re-fetch the evicted key from the database).

**Production considerations:**

- Redis eviction runs in the main event loop. A large eviction batch (evicting 10,000+ keys in one cycle) blocks all other operations. Set `maxmemory-eviction-tenacity` (Redis 7.0+) to control how aggressively Redis evicts per cycle.

- Monitor `evicted_keys` metric. A sudden increase signals that the working set has grown beyond the cache size. Investigate immediately.

- Use `CONFIG SET maxmemory-policy` to change policies at runtime. No restart needed.

### Memcached: LRU with Slab Allocator

Memcached's approach to eviction is tightly coupled to its slab allocator. The slab allocator divides memory into slabs of different sizes. Each slab class maintains its own LRU list.

**Key design decisions:**

- **Per-class LRU prevents size-based priority inversion.** Without per-class LRU, one large entry (1 MB) evicts 1,000 small entries (1 KB each), destroying the small-entry hot set. Memcached's per-class LRU ensures that evictions are proportional to item size.

- **Slab rebalancing (Memcached 1.5+).** Fixed slab sizes can cause memory waste: if you have 10 GB of slab class 1 but only 1 GB of small items, 9 GB is wasted. Slab rebalancing moves pages from underutilized classes to overutilized classes.

- **No explicit LFU support.** Memcached only supports LRU. For LFU workloads, you must use a different cache or implement frequency tracking at the application level.

**Production considerations:**

- Monitor `evictions` per slab class. If one class has high evictions while others have free space, slab rebalancing may be needed.
- Memcached does not persist. All data is lost on restart. The eviction policy is reset on restart.
- Memcached's LRU is approximate, similar to Redis, using a sampling approach.

### Varnish: Cache Eviction for HTTP Caching

**Key design decisions:**

- **TTL as primary eviction.** HTTP responses have explicit TTLs (Cache-Control: max-age). Varnish uses TTL expiration as the primary eviction mechanism. LRU only fires when storage is full. This matches the HTTP workload where most objects have a natural expiration time.

- **Grace mode for stale content.** Varnish serves stale content (within grace period) while asynchronously fetching fresh content. This improves hit ratio and reduces the impact of eviction-driven misses.

- **Ban-based invalidation.** Varnish's ban system allows pattern-based invalidation (e.g., ban all objects with a specific URL prefix). This is more efficient than purging individual URLs when a large set of content needs to be invalidated.

### Linux Kernel Page Cache: Clock Algorithm

The Linux kernel page cache uses the clock algorithm (also known as the "second chance" algorithm) for eviction. This is a well-studied, battle-tested implementation that has evolved over decades.

The clock algorithm in the Linux kernel:

1. Pages are stored in a radix tree (for lookup) and a circular linked list (for eviction).
2. Each page has a reference bit (PG_referenced) that is set on access.
3. The kernel's page reclaim daemon (kswapd) sweeps through the circular list.
4. Pages with PG_referenced set: clear the bit, move to the next page (second chance).
5. Pages with PG_referenced cleared: evict the page.
6. Dirty pages must be written back before eviction (added to the writeback queue).

The Linux kernel implementation is notable for its handling of dirty pages: before evicting a dirty page, the kernel must write it back to disk. This adds complexity that user-space caches (Redis, Memcached) do not face.

The kernel also maintains multiple LRU lists for different page types (active, inactive, anonymous, file-backed). This provides better isolation between page types and prevents one type of memory pressure from evicting pages of another type.

### Caffeine: W-TinyLFU

Caffeine is a Java caching library that implements the W-TinyLFU eviction policy. W-TinyLFU combines a window LRU (for recency) with a sketch-based frequency estimator (for frequency).

The "W" in W-TinyLFU stands for "window": a small LRU cache (1% of total size) that captures recent accesses. The main cache uses a frequency-based policy that estimates frequency using a Count-Min Sketch (a probabilistic data structure).

Key innovations:
- **Count-Min Sketch** for frequency estimation. Instead of storing exact frequency counters per entry (which would require O(entries) memory), W-TinyLFU uses a probabilistic sketch that gives approximate frequency with bounded error. This reduces memory overhead from O(entries) to O(1).
- **Adaptive window size.** The window size adjusts based on the workload. If the workload has strong recency, the window grows. If the workload has strong frequency, the window shrinks.
- **Reset on frequency overflow.** The sketch periodically resets (halves all frequency estimates) to adapt to changing access patterns.

Caffeine is the default cache implementation in Spring Boot 3.x and is used by many Java applications.

### Ehcache: LRU, LFU, FIFO

Ehcache is a Java caching library that supports LRU, LFU, and FIFO eviction policies. It is widely used in enterprise Java applications, particularly in conjunction with Hibernate.

Ehcache's key contribution is its tiered storage model: data can be stored in heap, off-heap, and disk tiers, each with its own eviction policy. Eviction at the heap tier moves data to the off-heap tier rather than dropping it entirely. This provides a graceful degradation: hot data stays in fast memory, warm data moves to slower memory, and cold data is evicted entirely.

### Guava Cache: LRU-Based

Guava's Cache is a popular Java caching library with a simple LRU-based eviction policy. It is not intended for high-performance production caching (Caffeine is the successor), but it is widely used in smaller applications and for prototyping.

Guava Cache uses a ConcurrentHashMap with a segment-based LRU. Each segment has its own LRU list. This provides good concurrency but can lead to skewed LRU ordering across segments.

## Cloud Provider

### AWS ElastiCache

AWS ElastiCache provides managed Redis and Memcached clusters. The eviction behavior depends on the engine:

**ElastiCache for Redis:** Supports all Redis eviction policies. Additional features:
- Automatic failover with Multi-AZ. If the primary node fails, a replica is promoted. The new primary starts with an empty cache, causing a cold start. AWS recommends pre-warming with a dummy workload.
- Reserved memory setting. AWS recommends reserving 25% of memory for non-data purposes (replication buffers, Redis Cluster bus, etc.). The remaining 75% is available for data. Eviction is based on the available memory, not the total instance memory.
- CloudWatch metrics: `Evictions` (number of evictions), `CurrItems` (current item count), `Reclaimed` (expired key count).

**ElastiCache for Memcached:** Supports Memcached's LRU eviction. Additional features:
- Auto-discovery: clients automatically discover new nodes when the cluster scales.
- Slab rebalancing is supported in Memcached 1.5+.

### CDN Eviction: CloudFront and CloudFlare

CDN eviction is primarily TTL-based, with LRU as the overflow mechanism at edge nodes.

**CloudFront:** Purge by invalidation request. You can invalidate specific URLs, URL patterns, or use wildcards. The first 1,000 paths per month are free. CloudFront propagates invalidations to all edge nodes within 1-5 minutes.

**CloudFlare:** Purge by URL, by tag, by hostname, or purge everything. CloudFlare propagates purges within 1-30 seconds. Tags are particularly useful for purging groups of related assets (e.g., all assets related to a product update).

**CDN evolution in practice:** CDN eviction is rarely an issue in practice because edge nodes have large storage (terabytes) and TTLs handle most expiration. The primary concern is purge propagation time: when you invalidate a URL, how long does it take for all edge nodes to stop serving the stale version?

## Lessons Learned

### 1. Approximate Is Good Enough

Exact LRU (with a globally sorted linked list) is never necessary in production. Redis's approximate LRU with 5 samples is within 5% of exact LRU. The 5% difference in hit ratio translates to a 5% increase in database load, which is acceptable for most workloads.

The complexity of exact LRU (O(log n) or O(1) with a list and hash map) is not worth the marginal hit ratio improvement. Always start with an approximate policy.

### 2. Monitor Eviction Rate

The eviction rate is the single most important metric for cache health. A sudden increase signals that the working set has grown, the access pattern has changed, or the cache is too small.

Set up alerting on the eviction rate. A sustained increase of 2x over baseline requires investigation. The investigation should check:
- Has the data volume increased? (More users, more products, more content.)
- Has the access pattern changed? (A new feature that scans more data, a batch job that runs during peak hours.)
- Has the cache size decreased? (Memory allocation changed, another process is consuming memory on the same host.)

### 3. Choose Policy Based on Access Pattern

There is no universal best eviction policy. The best policy depends on your access pattern:

- **Strong temporal locality (recent = likely to be accessed again):** LRU. Most web application workloads.
- **Stable popularity (frequently accessed = likely to be accessed again):** LFU. Reference data, configuration.
- **Unknown or changing pattern:** ARC. Production systems that cannot be manually tuned.
- **Scan-heavy workload:** 2Q or SLRU. Proxy caches, CDN edge caches.
- **Minimal complexity:** Random. Medium-traffic caches with unpredictable access.

### 4. Test With Production Traffic

Eviction policies behave differently under production traffic than under synthetic benchmarks. A policy that achieves 95% hit ratio on a benchmark may achieve 60% on production traffic.

The reason: production traffic has complex access patterns that benchmarks do not capture. Session expiry, batch jobs, cache warming, and traffic spikes all affect eviction behavior.

Test policy changes using a canary deployment: apply the new policy to one shard (or 1% of cache nodes) and compare hit ratios, eviction rates, and latency before rolling out.

### 5. TTL Reduces Eviction Pressure

Every entry with a TTL will eventually expire, freeing memory without triggering eviction. This reduces the eviction rate and improves cache performance.

Set TTLs aggressively. A 1-hour TTL means the entry is automatically removed after 1 hour, even if it is never accessed again. This prevents the cache from filling with stale data and reduces the burden on the eviction policy.

The ideal TTL balances freshness (shorter = fresher data) and eviction pressure (longer = less eviction). For most web caches, 5-60 minutes is appropriate. For static assets with versioned URLs, 1 year is safe.
