## The Naive Implementation

Every alerting system starts simple. You have a monitoring tool — Prometheus, Datadog, Grafana — and you configure a few alert rules. The tool evaluates the rules periodically and sends notifications. This works for small teams with a handful of services.

### Breaking at 1,000 Users (Single Service)

Your startup has one service, 10 alert rules, and 5 team members. You use Prometheus with Alertmanager. Rules are defined in a YAML file committed to a repository. The Prometheus server evaluates rules every 60 seconds. When an alert fires, Alertmanager sends a Slack message to the team channel.

This works flawlessly. Your team gets alerts for real problems. The noise is low. Everyone pays attention. The single Prometheus server handles the load easily. The YAML configuration file is 50 lines. Life is good.

### Breaking at 100,000 Users (Five Services)

Five services. 50 alert rules. 15 team members. The naive setup starts showing cracks. The YAML configuration file grows to 500 lines. Rules are duplicated across services because there is no easy way to share common configurations. A deployment happens at 7 PM, and by 10 PM, nobody remembers to unsilence the deployment alerts. The on-call engineer gets paged at 3 AM for a false alarm.

The alert manager cannot keep up with deduplication. A database connection pool exhaustion triggers 12 separate alerts — one from each service that depends on the database. The team Slack channel is flooded. People start muting the channel. The single Prometheus server starts showing query latency spikes. Evaluation cycles that used to take 100ms now take 800ms.

### Breaking at 10 Million Users (50 Services)

Fifty services. 500 alert rules. 40 team members. The YAML configuration is unmanageable. No single person understands all 500 rules. Most rules have never been reviewed. Alert fatigue is severe. The on-call engineer receives 50-100 alerts per shift and has learned to ignore everything except PagerDuty phone calls.

The single Prometheus server hits its query capacity. Each evaluation cycle takes 30 seconds because the server must scan millions of time-series samples. The evaluation interval was 60 seconds, so the server barely keeps up. Some evaluation cycles are skipped because the previous one has not finished. Alerts that should have fired are delayed or missed.

### Breaking at 100 Million Users (500 Services)

Five hundred services. 5,000 alert rules. 200 team members. The naive architecture is completely broken. The single Prometheus server cannot handle the query load. The Alertmanager is overwhelmed by the volume of firing alerts. The Slack channel is permanently flooded. The on-call rotation has burned out three engineers in six months.

The system needs a fundamental redesign. The monolithic architecture must be replaced by a distributed, multi-tier system. This is the point where most organizations realize that alerting is not a feature of the monitoring system — it is a separate system that must be designed and built with the same rigor as the application itself.

### Tier 1: Rule Storage

Alert rules are stored in a configuration database, not flat files. A PostgreSQL or etcd cluster holds the canonical rule definitions. Rules are versioned. Changes are audited. Rules can be grouped into rule groups for easier management. A configuration API allows rules to be created, updated, and deleted without restarting the alerting engine.

**Key design decisions:**
- Rules are stored with a unique ID, name, expression, threshold, duration, severity, labels, annotations, and runbook URL.
- Rule groups share common labels and evaluation intervals. A group for "database health" applies to all database-related rules.
- A webhook notifies the evaluator shards when rules change, so they can reload without polling. The webhook is idempotent — receiving the same notification twice does not cause issues.
- The configuration database is the source of truth. The evaluator shards cache the rules in memory and refresh the cache every 5 minutes. The cache prevents a database outage from taking down the alerting system.

### Tier 2: Evaluation Tier

The evaluator tier consists of N stateless shards. Each shard loads a partition of the rules from the configuration store and evaluates them on a loop. Shards are stateless — evaluation state is written to a shared database. This means any shard can be replaced without losing state.

**Key design decisions:**
- Shards are assigned to rules using consistent hashing on the rule ID. This minimizes rebalancing when a shard is added or removed. When a new shard joins, only a fraction of rules are reassigned.
- Each shard maintains a persistent connection pool to the monitoring system to avoid connection setup overhead. The pool size is tuned to the number of concurrent queries. A pool of 20 connections is sufficient for most workloads.
- Evaluation results are written to a shared state store (PostgreSQL, Cassandra, or Redis). The state store is the single source of truth for alert states.
- The evaluation loop includes a deadline. If the evaluation takes longer than the evaluation interval, the shard skips the next cycle and logs a warning. The deadline prevents cascading delays.
- Each shard exposes a health endpoint that reports the last successful evaluation timestamp. The orchestrator uses this to detect and replace unhealthy shards.

### Tier 3: Alert Manager Cluster

The alert manager cluster receives evaluation results and handles deduplication, grouping, silencing, and escalation. The manager is stateful — it maintains the current set of firing alerts, the grouping hierarchy, and the escalation timers.

**Key design decisions:**
- The manager uses a leader-election protocol (Raft, etcd) to ensure only one manager processes alerts at a time. If the leader fails, a new leader is elected within seconds.
- Deduplication uses a hash of the rule ID and the alert label set. If an alert with the same hash is already firing, the new result is merged into the existing alert rather than creating a new one.
- Silencing rules are evaluated against every incoming alert. Silences can be time-based (mute every night from 2-4 AM for maintenance), event-based (mute while a deployment is in progress), or label-based (mute all alerts with severity=info).
- The alert manager persists its state to the shared state store. If the leader crashes, the new leader reconstructs the state from the store.

### Tier 4: Notification Router

The notification router takes resolved, unsilenced alerts and delivers them through the configured channels. The router handles channel-specific formatting, rate limiting, retries, and escalation.

**Key design decisions:**
- Each notification channel is a separate module with its own configuration, rate limits, and retry logic. Adding a new channel does not require changes to the core router.
- Rate limiting prevents the router from overwhelming the notification provider. Slack has a rate limit of 1 message per second per channel. PagerDuty has a rate limit of 60 events per minute per integration. The router enforces these limits with a token bucket per channel.
- Retries use exponential backoff with jitter: 30 seconds, 1 minute, 2 minutes, 4 minutes, up to 30 minutes maximum. The jitter prevents retry storms when multiple notifications fail simultaneously.
- Escalation is driven by a timer. If the alert is not acknowledged within the policy-defined window, the router sends the notification to the next contact in the escalation chain.

### Tier 5: Monitoring and Observability

The alerting system itself must be monitored. Each tier exposes metrics (evaluation latency, state store throughput, notification delivery rate), logs (evaluation errors, state store failures, notification failures), and traces (end-to-end alert flow). The monitoring system is the same system that the alerting system uses to monitor the rest of the infrastructure.

## Integration Patterns

An alerting system does not exist in isolation. It integrates with every other part of the observability stack.

### Pattern 1: Integration with the Metrics Database

The alerting engine queries the metrics database to evaluate rules. This is the most performance-critical integration.

**Best practices:**
- Use a dedicated query endpoint for alerts, separate from dashboard queries. Dashboard queries can tolerate higher latency. Alert queries need consistent sub-second performance.
- Enable query result caching for rules that use the same query expression. A Prometheus query for cpu_usage evaluated for 50 services can share the same query result for all 50 services.
- Set a query timeout. If the database does not respond within the timeout, the evaluation fails gracefully. A failed evaluation is better than a hung evaluator.
- Use a circuit breaker. If the metrics database is unhealthy, stop querying it and use the last known state for each rule. The circuit breaker prevents cascading failures.

### Pattern 2: Integration with the Service Catalog

The alerting engine enriches alerts with metadata from the service catalog. The service catalog knows which team owns each service, which Slack channel to use, and which on-call schedule to page.

**Best practices:**
- The service catalog is the source of truth for ownership. Alerts use the catalog to determine the notification channel and escalation policy.
- Service metadata is cached in the alerting engine to avoid a catalog lookup for every alert. The cache is refreshed every 5 minutes. If the catalog is unavailable, the cache serves the last known metadata.
- When a service changes owners, the alert routing updates automatically. No manual reconfiguration needed. The catalog pushes a change event to the alerting engine.

### Pattern 3: Integration with Change Management

Change events are the most important context for an alert. If a deployment happened 10 minutes before an alert fired, the deployment is likely the root cause. The alerting engine queries the deployment system and includes recent changes in every alert.

**Best practices:**
- The deployment system pushes change events to a Kafka topic. The alerting engine consumes this topic and stores the last 10 changes per service in memory.
- When an alert fires, the engine attaches the recent change list to the alert notification. The responder sees: "Alert fired at 10:15 AM. Deployments: payment-service v2.3.1 at 10:05 AM, auth-service v1.8.0 at 9:30 AM."
- The runbook link in the alert should point to a dashboard that shows the change timeline alongside the alerting metric.

### Pattern 4: Integration with Incident Management

When an alert fires, it should create an incident in the incident management system. The incident management system tracks the alert status, assigns responders, and manages the postmortem process.

**Best practices:**
- The alert fires as a webhook to the incident management system. The system creates an incident, assigns the on-call engineer, and tracks the incident status.
- When the alert resolves, the incident management system is notified. The incident is marked as resolved.
- If a human modifies the alert state — silencing it, acknowledging it, or adjusting the threshold — the change is propagated back to the alert manager.

### Pattern 5: Integration with Automated Remediation

Some alerts should trigger automated remediation, not human notification. If the alert is common and the fix is well-understood, the alerting system should trigger a remediation workflow automatically.

**Best practices:**
- Automated remediation is configured as a notification channel. When the alert fires, the notification router sends the alert to the remediation system instead of a human.
- The remediation system executes the fix and reports the result. If the remediation succeeds, the alert is resolved automatically. If the remediation fails, the alert is escalated to a human.
- Automated remediation is only used for alerts with a high confidence level. The alert must have a false-positive rate under 1 percent to qualify for automated remediation.

## Multi-Region Architecture

When your system runs in multiple regions, the alerting system must run in every region. Cross-region latency (100-300ms) makes centralized alerting impractical.

**Per-region stack:** Each region has its own alerting evaluator, manager, and router. The evaluator queries the local metrics database. The manager evaluates locally and routes local alerts through local channels. This ensures that alerting continues to work even if the region loses connectivity to other regions.

**Global aggregator:** A global layer receives alert status from every region. The aggregator creates cross-region alerts — "region us-east-1 is unreachable" — and provides a unified view of all firing alerts. The aggregator does not re-evaluate rules. It simply collects and collates.

**Failover:** If a region alerting stack fails, the global aggregator detects the failure and alerts the operations team. Meanwhile, the affected region services continue to run but without local alerting. The incident management system routes alerts through the global aggregator for the affected region.

### Cross-Region Deduplication

When a service runs in multiple regions, the same failure might trigger alerts in every region. The global aggregator deduplicates cross-region alerts. If the payment service is down in us-east-1, eu-west-1, and ap-southeast-1, the aggregator creates a single incident: "Payment service is down in 3 regions." The incident includes the per-region details.

### Cross-Region Escalation

If the on-call engineer in the affected region does not acknowledge the alert within the timeout, the global aggregator escalates to the global on-call rotation. The global rotation is responsible for coordinating cross-region incidents. The global rotation is the highest level of escalation.

## Capacity Planning

You are building an alerting system for a platform with 10 million daily active users. Here is how to plan capacity.

### Inputs

- 50 microservices
- 500 alert rules (10 per service)
- Average evaluation interval: 2 minutes
- Average query execution time: 200ms
- Metrics retention: 30 days
- Average alert notification rate: 50 per hour

### Rule Evaluation Throughput

Each rule evaluation requires one query to the metrics database. With 500 rules and a 2-minute average evaluation interval:
- Evaluations per minute: 500 / 2 = 250
- Queries per second: 250 / 60 = 4.2 QPS

This is low. One evaluator shard can handle it easily. But when you add labels — each rule evaluated per service, per host, per endpoint — the cardinality explodes.

A rule like "alert if p99 latency exceeds 500ms" applied to 50 services, each with 10 endpoints: 500 unique label combinations per rule. If you have 50 such rules, you need 25,000 evaluations. At 250ms per evaluation, that is 6,250 seconds of evaluation time per cycle. You need multiple shards.

### Shard Count

Target: each evaluation cycle completes within 60 seconds.
- Total evaluations: 25,000 per cycle
- Evaluation time per shard: 250ms
- Evaluations per shard per second: 4
- Evaluations per shard per minute: 240
- Shards needed: 25,000 / 240 = 105 shards

Round up to 120 shards for headroom. Each shard needs 1 CPU core and 2 GB of memory. Total: 120 CPU cores and 240 GB of memory.

### Notification Throughput

At 50 alerts per hour, notification throughput is negligible. A single notification router can handle tens of thousands of notifications per hour. The bottleneck is the rate limits imposed by notification providers, not the router capacity.

At 500 alerts per hour (a high-traffic incident), the router must handle 8-9 notifications per minute. This is easily handled by a single router instance. The router is provisioned for 10x the expected peak to handle alert storms.

### Storage

Each alert rule evaluation produces a state record of roughly 1 KB. At 250 evaluations per minute:
- Storage per minute: 250 KB
- Storage per day: 360 MB
- Storage per year: 131 GB

This is modest for the state store. The metrics database, storing the actual time-series data, will require orders of magnitude more storage — tens of terabytes for 30 days of retention.

### State Store Capacity

The state store must handle read and write operations at the evaluation throughput rate. At 250 evaluations per minute, the state store must handle 250 writes per minute. Each write is a small transaction (1 KB). The state store must also handle reads when the alert manager queries the state for deduplication and grouping.

A single PostgreSQL instance can handle this load easily. For higher throughput, use a PostgreSQL cluster with read replicas. The evaluator shards write to the primary. The alert manager reads from the replicas.

## Deployment and Operations

### Deployment Pattern

Deploy the alerting engine as a stateless Kubernetes deployment. Each shard runs as a separate pod. The alert manager runs as a stateful set with persistent storage. The notification router runs as a stateless deployment.

**Health checks:**
- The evaluator shard exposes a /health endpoint that reports the last successful evaluation timestamp. If the last evaluation was more than 3 evaluation intervals ago, the shard is unhealthy.
- The alert manager exposes a /ready endpoint that confirms the state store is reachable. If the state store is down, the manager should not accept traffic.
- The notification router exposes a /health endpoint that reports connectivity to each configured notification provider.

**Resource requests and limits:**
- Evaluator shard: request 1 CPU, 2 GB memory. Limit 2 CPU, 4 GB memory.
- Alert manager: request 2 CPU, 4 GB memory. Limit 4 CPU, 8 GB memory.
- Notification router: request 0.5 CPU, 1 GB memory. Limit 1 CPU, 2 GB memory.

### Monitoring the Monitoring System

The alerting system must monitor itself. If the alerting system fails, you will not know.

**Self-monitoring rules:**
- Alert if any evaluator shard has not evaluated its rules in the last 5 minutes.
- Alert if the alert manager has not processed an evaluation result in the last 5 minutes.
- Alert if the notification router has a delivery failure rate above 5 percent.
- Alert if the state store write latency exceeds 1 second for more than 5 minutes.
- Alert if the configuration database is unreachable for more than 1 minute.
- Alert if the number of skipped evaluation cycles exceeds 1 percent of total cycles.

**Self-monitoring channels:**
- The self-monitoring alerts use a separate notification channel that is independent of the main alerting system. If the main alerting system is down, the self-monitoring channel still works.
- The self-monitoring channel is monitored by a different team. The infrastructure team monitors the alerting system. The application team monitors the application.

### Gradual Rollout

When deploying a new version of the alerting engine, use a gradual rollout. Start with 10 percent of shards. Monitor for 30 minutes. If no issues, roll out to 50 percent. After another 30 minutes, roll out to 100 percent.

**Rollback plan:** If the new version causes issues, the old version is still running on the remaining shards. Roll back by scaling the new version to zero and restoring the old version deployment.

**Canary analysis:** Before rolling out to all shards, compare the canary shard metrics to the baseline shard metrics. Look for differences in evaluation latency, error rate, and state store throughput. If the canary metrics are worse by more than 10 percent, abort the rollout.

### Incident Response for the Alerting System

When the alerting system itself fails, the incident response follows a different process. The first step is to determine whether the alerting system failure is causing a monitoring gap. If the alerting system is down but the monitoring system is still collecting data, the gap is limited. If both are down, the gap is critical.

**Recovery steps:**
1. Restart the alerting engine components in order: configuration database, state store, evaluator shards, alert manager, notification router.
2. Verify that the evaluator shards are evaluating rules and writing state.
3. Verify that the alert manager is processing evaluation results.
4. Verify that the notification router is delivering notifications.
5. Verify that the self-monitoring alerts are working.

### Configuration Management

Alert rules are configuration artifacts. They should be managed with the same rigor as application code: version-controlled, reviewed, tested, and deployed through CI/CD.

**Version control:** All alert rules are stored in a Git repository. Each rule is a YAML file with a unique name, a query expression, a threshold, a duration, a severity, and a runbook URL. The repository is the source of truth. Changes go through the same review process as code changes.

**CI/CD pipeline:** When a rule change is pushed to the repository, the CI/CD pipeline validates the rule. Validation includes: checking that the query expression is syntactically valid, checking that the threshold is within a reasonable range, checking that the runbook URL is accessible, and checking that the duration is at least 3 evaluation intervals. If validation passes, the rule is deployed to the staging environment. After 48 hours of staging observation, the rule is promoted to production.

**Environment isolation:** The staging environment has a separate metrics database, separate evaluator shards, and separate notification channels (notifications go to a test channel, not the real on-call). The staging environment replicates production traffic patterns using synthetic data. This ensures that rules behave the same way in staging and production.

### Load Testing the Alerting System

The alerting system must be load-tested before it is deployed to production. Load testing validates that the system can handle the expected evaluation throughput, state store operations, and notification delivery.

**Load testing methodology:**
1. Generate synthetic metrics at the expected production volume (10,000 data points per second).
2. Configure 500 alert rules on the synthetic metrics.
3. Run the alerting system for 1 hour.
4. Measure: evaluation latency, state store throughput, notification delivery latency, memory usage, CPU usage.
5. Repeat with 2x, 5x, and 10x the expected volume.

**Acceptance criteria:**
- Evaluation latency p99: under 2 seconds.
- State store write throughput: 100 percent of evaluations written successfully.
- Notification delivery latency: under 10 seconds for critical alerts.
- Memory usage: under 80 percent of available memory.
- CPU usage: under 70 percent of available CPU.

### Disaster Recovery

The alerting system must be recoverable from catastrophic failures. A disaster recovery plan documents the steps to restore alerting after a major outage.

**Recovery from configuration store failure:** If the configuration database (PostgreSQL) is lost, the evaluator shards cannot load new rules. They continue evaluating the last loaded rules from memory. The recovery steps: restore the configuration database from backup, reload the rules from the database, verify that the loaded rules match the expected configuration.

**Recovery from state store failure:** If the state store (Redis, Cassandra) is lost, the evaluator shards cannot persist evaluation state. All duration queues are reset. The recovery steps: notify the on-call team that all alerts will be re-evaluated from scratch, restart the state store, allow the evaluator shards to rebuild state from the metrics database. Alerts that were firing before the failure will re-fire after the state store is restored.

**Recovery from complete data center failure:** If an entire region is lost, the global aggregator detects the failure and alerts the operations team. The recovery steps: the alerting system is restored in a new region using the backup configuration and state. The global aggregator is updated to route traffic to the new region. The metrics database is restored from cross-region backup.

### Security Considerations

The alerting system handles sensitive data: metric values, rule definitions, notification configurations, and responder contact information. Security must be designed into the architecture.

**Authentication and authorization:**
- The alerting system API requires authentication. All API calls are authenticated using OAuth2 or mutual TLS.
- Each team can only access their own rules, alerts, and configurations.
- The notification configuration (PagerDuty integration keys, Slack webhook URLs) is encrypted at rest and in transit.

**Audit logging:**
- Every change to a rule configuration is logged with the user, the timestamp, and the change details.
- Every notification delivery is logged with the channel, the recipient, and the delivery status.
- Every acknowledgment and resolution is logged with the responder identity and the timestamp.

**Secrets management:**
- Integration keys and webhook URLs are stored in a secrets manager (HashiCorp Vault, AWS Secrets Manager).
- The alerting engine reads secrets from the secrets manager at startup and caches them in memory.
- Secrets are rotated automatically. The alerting engine picks up the new secret when the cache expires.

### Integrating with Existing Observability Stack

The alerting system does not replace the existing observability stack. It integrates with it. The integration points are:

**Metrics database:** The alerting engine queries the existing metrics database (Thanos, Mimir, VictoriaMetrics) for rule evaluation. The metrics database is the single source of truth for time-series data.

**Log aggregation:** The alerting engine queries the existing log aggregation system (Loki, Elasticsearch, Splunk) for log-based alert rules. The log aggregation system is the single source of truth for log data.

**Trace analysis:** The alerting engine queries the existing trace analysis system (Jaeger, Tempo, Zipkin) for trace-based alert rules. The trace analysis system is the single source of truth for trace data.

**Incident management:** The alerting engine creates incidents in the existing incident management system (PagerDuty, Opsgenie, ServiceNow). The incident management system is the single source of truth for incident tracking.

**Change management:** The alerting engine queries the existing change management system (Spinnaker, ArgoCD, Jenkins) for recent deployment information. The change management system is the single source of truth for deployment history.

### Alert Rule Organization at Scale

At scale, alert rules must be organized into a hierarchy for manageability. The hierarchy has four levels:

**Level 1 — Alert templates.** A template defines a reusable rule pattern. The template specifies the metric type, the query pattern, the threshold formula, and the notification channel. Templates are parameterized: the service name, the endpoint, and the threshold values are parameters.

**Level 2 — Rule groups.** A rule group is a collection of rules that share the same evaluation interval, duration, and severity. Rule groups map to services or components. The payment service has one rule group. The authentication service has another.

**Level 3 — Service-level rules.** Each service has a minimum set of rules: one for latency, one for errors, one for traffic, and one for saturation. These are the four golden signals. Every service must have at least these four rules.

**Level 4 — Custom rules.** Each service can add custom rules for service-specific metrics. A queue processing service might have a rule for queue depth. A database service might have a rule for replication lag. Custom rules are limited to 10 per service.

### Service Ownership Models

The ownership model determines who creates and maintains alert rules. Three models are common:

**Centralized model:** A centralized observability team creates and maintains all alert rules. The team has deep expertise in alerting but limited knowledge of each service. The model ensures consistency but can produce generic rules that miss service-specific failure modes.

**Decentralized model:** Each service team creates and maintains their own alert rules. The team has deep knowledge of the service but limited expertise in alerting. The model produces tailored rules but can lead to inconsistency and alert sprawl.

**Federated model:** A centralized observability team creates alert templates. Service teams customize the templates for their services. The observability team reviews all rules quarterly. The model combines consistency with service-specific knowledge.

### Alert Rule Versioning and Rollback

Alert rules are versioned configuration artifacts. Each version is assigned a unique ID. The version history is stored in the configuration store. The alerting engine can roll back to any previous version.

**Versioning:** Each time a rule is created or updated, a new version is created. The version includes the rule definition, the author, the timestamp, and the change description. The version is immutable — once created, it cannot be modified.

**Rollback:** If a rule change causes unexpected behavior (too many false positives, missed incidents), the alerting engine can roll back to the previous version. The rollback is a configuration change that replaces the current version with the previous version.

**Canary deployment:** Rule changes are deployed to a subset of services first. The canary services use the new rule version. The remaining services use the old version. If the canary shows acceptable behavior after 24 hours, the new version is rolled out to all services.

### The Evaluation Tier in Detail

The evaluation tier is the most performance-critical component. Its design determines the system throughput and latency.

**Shard assignment:** Rules are assigned to shards using consistent hashing on the rule ID. The hash ring has 1,000 virtual nodes per shard. When a shard is added or removed, only the rules assigned to the affected virtual nodes are reassigned. The reassignment is smooth — no rule is evaluated by two shards simultaneously.

**Evaluation loop:** Each shard runs an evaluation loop that processes rules in batches. The shard loads the rules from the configuration store, groups them by evaluation interval, and evaluates each group on its timer. The loop is non-blocking — if one rule evaluation is slow, it does not delay the next rule.

**State persistence:** The shard writes evaluation state to the shared state store after each evaluation. The write is asynchronous — the shard does not wait for the write to complete before starting the next evaluation. If the state store is unavailable, the shard buffers the writes in memory and retries.

**Shard health:** Each shard exposes a health endpoint. The health endpoint reports: the number of rules assigned, the number of rules evaluated in the last minute, the p50 and p99 evaluation latency, and the last evaluation timestamp. The orchestrator (Kubernetes, Nomad) uses the health endpoint to detect and replace unhealthy shards.

### The Alert Manager Cluster in Detail

The alert manager is the stateful component that coordinates alert processing. Its design determines the system reliability and consistency.

**Leader election:** The alert manager uses Raft consensus for leader election. The leader is responsible for processing all evaluation results. Followers are standby nodes that replicate the leader state. If the leader fails, a follower is elected as the new leader within 2 seconds.

**State replication:** The leader replicates evaluation state to followers asynchronously. The replication lag is typically under 100ms. If the leader fails, the new leader may miss the last 100ms of state. The missed state is recovered from the shared state store.

**Deduplication table:** The leader maintains an in-memory deduplication hash table. The table is a hash map from (rule_id, labels_hash) to alert_record. The table is replicated to followers as part of the state replication. If the leader fails, the new leader reconstructs the deduplication table from the state store.

**Grouping tree:** The leader maintains an in-memory grouping tree. The tree is built from the service dependency graph. The tree is updated when the dependency graph changes. The leader processes evaluation results by traversing the grouping tree and assigning alerts to incidents.

### The Notification Router in Detail

The notification router is the outbound component that delivers alerts to external systems. Its design determines the reliability and latency of notification delivery.

**Channel abstraction:** Each notification channel implements a common interface: send(alert) -> DeliveryResult. The interface abstracts the provider-specific details (Slack API, PagerDuty API, SMTP). Adding a new channel requires implementing the interface and registering it with the router.

**Rate limiter:** The router maintains a token bucket per channel. The token bucket enforces the provider rate limits. If a channel is rate-limited, the router queues the notification and retries when tokens are available.

**Retry queue:** The router maintains a retry queue for failed notifications. The retry queue is a priority queue ordered by the next retry time. The router processes the retry queue in the background. If a notification fails after 10 retries, the router escalates the alert.

**Escalation engine:** The escalation engine is a background process that checks the acknowledgement status of every firing alert. If an alert is not acknowledged within the policy timeout, the escalation engine raises the severity and sends a new notification through the next channel in the escalation chain.

### Load Shedding and Graceful Degradation

When the alerting system is under extreme load, it must shed load gracefully rather than crash. Load shedding strategies ensure the system remains operational.

**Evaluation shedding:** If the evaluation queue is backed up, the shard skips evaluations for non-critical rules. Critical rules are always evaluated. The shard logs the skipped evaluations for post-incident review.

**Notification shedding:** If the notification queue is backed up, the router prioritizes critical alerts over warning and info alerts. Warning and info alerts are queued and processed when the load decreases.

**State store shedding:** If the state store is overloaded, the evaluator shards reduce the write frequency. Instead of writing after every evaluation, the shard batches writes and writes every 5 seconds. The batch write reduces the state store throughput by 5x.

**Alert manager shedding:** If the alert manager is overloaded, it processes evaluation results in batches. Instead of processing each result individually, the manager accumulates results for 1 second and processes them as a batch. The batch processing reduces the overhead of the deduplication and grouping logic.

### Alert Rule Distribution Across the Hash Ring

The consistent hash ring distributes alert rules across evaluator shards. The distribution is not uniform — some rules are more expensive to evaluate than others. The hash ring accounts for the cost of each rule.

**Cost-weighted distribution:** Each rule has a cost (the execution time of the rule query). The hash ring distributes rules so that the total cost per shard is balanced. The cost is measured in production and updated periodically. A rule with a 500ms query cost is weighted 5x more than a rule with a 100ms query cost.

**Dynamic rebalancing:** When a shard is overloaded (evaluation latency exceeds the evaluation interval), the hash ring rebalances the rules. A fraction of the rules from the overloaded shard are reassigned to underloaded shards. The rebalancing is gradual — no more than 10 percent of rules are reassigned per minute.

### The Self-Healing Alerting System

A self-healing alerting system detects and fixes common problems automatically. The self-healing reduces the burden on the on-call engineer.

**Self-healing 1 — Restart the service.** If the alerting system detects that a service is unresponsive, it automatically restarts the service. The restart is logged and monitored. If the restart does not fix the problem within 5 minutes, the alert is escalated to a human.

**Self-healing 2 — Roll back the deployment.** If the alerting system detects a degradation immediately after a deployment, it automatically rolls back the deployment. The rollback is logged and monitored. The deployment team is notified.

**Self-healing 3 — Scale up the service.** If the alerting system detects a traffic spike, it automatically scales up the service. The scale-up is logged and monitored. The scaling is gradual to avoid overwhelming the infrastructure.

**Self-healing 4 — Kill the slow query.** If the alerting system detects a database query that is running for more than 30 seconds, it automatically kills the query. The kill is logged and monitored. The developer who wrote the query is notified.

### The Alerting System Cost Breakdown

Understanding the cost of running the alerting system helps with capacity planning and budget allocation.

**Infrastructure cost:** The evaluator shards, alert manager, and notification router run on Kubernetes. Each shard costs approximately $50 per month (1 CPU, 2 GB memory). At 120 shards, the infrastructure cost is $6,000 per month. The state store (PostgreSQL or Redis) costs approximately $500 per month.

**Metrics database cost:** The alerting system queries the metrics database. The database cost is proportional to the query volume. At 250 queries per minute, the database cost is approximately $1,000 per month. The database cost is shared with the dashboard system.

**Notification provider cost:** PagerDuty costs $20 per user per month. Slack costs $8 per user per month. SMS costs $0.01 per message. At 50 alerts per hour, the SMS cost is $0.50 per hour or $360 per month.

**Total cost:** The total cost of running the alerting system is approximately $8,000 per month. The cost is less than 1 percent of the total infrastructure cost for a 10-million-user platform.

### The Evaluation Pipeline: A Worked Example

Follow a single alert through the sharded evaluation pipeline:

**Step 1:** The configuration store has 500 rules. The consistent hash ring assigns rules 1-100 to shard 1, rules 101-200 to shard 2, and so on. Each shard loads its assigned rules at startup.

**Step 2:** Shard 1 starts its evaluation loop. It evaluates 100 rules every 2 minutes. The loop iterates through the rules, executes the queries, and updates the state store. The p50 evaluation latency is 200ms. The p99 evaluation latency is 800ms.

**Step 3:** Rule 47 (database connection pool > 80 percent) evaluates to FIRING. The shard writes the evaluation result to the state store. The state store records: rule_id=47, current_state=FIRING, metric_value=87, timestamp=2026-07-02T14:30:00Z.

**Step 4:** The alert manager polls the state store every 1 second. It finds the new FIRING state for rule 47. The alert manager computes the deduplication hash: hash(47, {}) -> 0x7A3F. The hash table lookup finds no existing entry. The alert manager creates a new alert record.

**Step 5:** The alert manager checks the silence rules. No silence rule matches rule 47. The alert manager checks the grouping tree. The database connection pool alert is a child of the database service incident. If the database service is already in an incident, the alert is grouped under that incident. If not, a new incident is created.

**Step 6:** The alert manager checks the escalation policy. The alert severity is CRITICAL. The escalation policy specifies: Slack immediately, page after 5 minutes if not acknowledged. The alert manager sends the Slack notification and starts the 5-minute timer.

**Step 7:** The notification router formats the Slack message and sends it. The message includes: the alert name, the metric value, the threshold, the runbook link, and the dashboard link. The message is sent to the database team Slack channel.

**Step 8:** The on-call engineer acknowledges the alert at 14:31:30Z. The acknowledgment is recorded in the state store. The escalation timer is stopped. The engineer starts investigating the database connection pool.

### The Architecture Evolution: From Startup to Enterprise

The alerting system architecture evolves as the organization grows. The evolution follows a predictable pattern:

**Stage 1 — Startup (1-10 services).** Use a single Prometheus server with Alertmanager. Rules are defined in a YAML file. Alerts go to a Slack channel. The on-call is a single person. The system is simple and works well for small teams.

**Stage 2 — Growth (10-100 services).** The single Prometheus server is overloaded. Add a Thanos or Mimir layer for horizontal scaling. Use Grafana for dashboards. Implement a proper on-call rotation with PagerDuty. Start writing runbooks.

**Stage 3 — Scale (100-1,000 services).** The monolithic Alertmanager cannot handle the load. Implement a multi-tier architecture with evaluator shards, a dedicated alert manager cluster, and a notification router. Use a configuration database (etcd or PostgreSQL) instead of YAML files. Implement dynamic thresholds and SLO-based alerting.

**Stage 4 — Enterprise (1,000+ services).** Implement a multi-region alerting architecture. Use a global aggregator for cross-region visibility. Implement automated remediation and self-healing. Use machine learning for anomaly detection and predictive alerting. The alerting system is a platform that supports multiple teams and multiple use cases.

### The Alerting System Data Model

The alerting system data model defines the entities and their relationships. The data model is used by the configuration store, the state store, and the alert history database.

**Rule entity:**
- id: UUID (primary key)
- name: string (human-readable name)
- expression: string (query expression)
- threshold: float (threshold value)
- duration: integer (duration in seconds)
- severity: enum (CRITICAL, WARNING, INFO)
- labels: map (key-value pairs for routing)
- annotations: map (key-value pairs for context)
- runbook_url: string (URL of the runbook)
- owner: string (team name)
- created_at: timestamp
- updated_at: timestamp
- version: integer

**Alert entity:**
- id: UUID (primary key)
- rule_id: UUID (foreign key to rule)
- labels: map (key-value pairs from the evaluation)
- value: float (metric value that triggered the alert)
- threshold: float (threshold that was breached)
- state: enum (FIRING, ACKNOWLEDGED, RESOLVED, AUTO_CLOSED)
- fired_at: timestamp
- acknowledged_at: timestamp (nullable)
- acknowledged_by: string (nullable)
- resolved_at: timestamp (nullable)
- resolved_by: string (nullable)

**Incident entity:**
- id: UUID (primary key)
- title: string (human-readable title)
- severity: enum (CRITICAL, WARNING, INFO)
- state: enum (OPEN, ACKNOWLEDGED, RESOLVED, CLOSED)
- alert_ids: list of UUID (related alerts)
- created_at: timestamp
- resolved_at: timestamp (nullable)
