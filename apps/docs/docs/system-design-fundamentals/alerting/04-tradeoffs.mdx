## The Core Tension

Every alerting system faces an inescapable tension: you cannot simultaneously maximize sensitivity, specificity, and timeliness. Improving any one of these dimensions worsens at least one of the others.

**Sensitivity** is the ability to detect real problems. A highly sensitive alert fires for every metric anomaly. The problem with high sensitivity is false positives — alerts that fire when nothing is actually wrong. A sensitive alert ensures you never miss a real incident, but it comes at the cost of trust erosion.

**Specificity** is the ability to avoid false positives. A highly specific alert only fires when the evidence is overwhelming. The problem with high specificity is false negatives — real problems that go undetected because the threshold was set too high or the duration was too long. A specific alert preserves trust, but it may miss subtle degradation.

**Timeliness** is the speed of detection. Faster detection means shorter duration windows and shorter evaluation intervals. The problem with faster detection is that it increases false positives because transient anomalies are more likely to trigger alerts before they self-correct. A timely alert catches problems early, but it comes at the cost of false alarms.

**Example:** A payment service has a p99 latency alert set at 200ms with a 1-minute duration. During a traffic spike, the latency briefly hits 300ms for 45 seconds. The alert fires (sensitive, timely). But the spike was caused by a garbage collection pause that resolved itself. The responder wasted time investigating a non-issue (low specificity).

The solution is not to pick one dimension and ignore the others. The solution is to understand the trade-off and configure your alerts for the right balance given your system tolerance for downtime versus noise.

### Finding the Right Balance

The right balance depends on three factors:

**Business impact.** A payment service that processes $1 million per hour needs more sensitive alerts than a reporting dashboard that is updated daily. The cost of missing a payment outage is high. The cost of false positives is acceptable.

**System volatility.** A recommendation engine that fluctuates constantly needs less sensitive alerts than a database cluster with stable baselines. A volatile system will produce false positives if thresholds are set too aggressively.

**Team capacity.** A team with 5 on-call engineers rotating weekly can tolerate more false positives than a team with 2 on-call engineers rotating daily. The smaller team will burn out faster if the alerting system is noisy.

### Decision Matrix Entries

**Entry 1 — High Impact, High Volatility (Payment Processing):** Use dynamic thresholds that adapt to traffic patterns. Set short durations (60 seconds) because every second of payment failure costs revenue. Use multi-channel notification with immediate paging. The false-positive cost is high — noisy alerts erode trust — but the cost of missing a real incident is catastrophic.

**Entry 2 — High Impact, Low Volatility (Database Cluster):** Use static thresholds with long durations (10 minutes). Database metrics like disk usage and connection count have stable baselines. A static threshold at 80 percent disk usage is appropriate. Do not page immediately; use a 10-minute duration to confirm the trend. False positives here are dangerous because they train responders to ignore database alerts.

**Entry 3 — Medium Impact, High Volatility (Recommendation Engine):** Use static thresholds with medium durations (3-5 minutes). Recommendation quality fluctuates constantly. A brief degradation is acceptable. Set the threshold at p99 of normal operation. Alert to Slack during business hours. Do not page at night.

**Entry 4 — Medium Impact, Low Volatility (Search API):** Use static thresholds with long duration (10 minutes). Search latency tends to be stable. A sustained degradation indicates a real problem. Alert to Slack. Page only if the degradation persists for 30 minutes.

**Entry 5 — Low Impact, High Volatility (Analytics Pipeline):** Do not alert. Log anomalies to a daily digest. The analytics pipeline processes batch jobs that run for hours. A 5-minute latency spike is irrelevant. The cost of alerting here is pure noise.

**Entry 6 — Low Impact, Low Volatility (Reporting Dashboard):** Do not alert. Use a weekly report of dashboard health. If the dashboard is down for 30 minutes, nobody notices. The on-call engineer should not be paged for a dashboard.

## Alerting CAP Theorem

Alerting has its own version of the CAP theorem. In a distributed alerting system, you face a three-way trade-off between completeness, accuracy, and timeliness.

**Completeness:** Every condition that should trigger an alert does trigger an alert. No false negatives. Completeness requires evaluating every rule on every data point without dropping any evaluations.

**Accuracy:** Every alert that fires represents a genuine problem. No false positives. Accuracy requires confirmation through multiple evaluation cycles, multiple metrics, and multiple sources.

**Timeliness:** Alerts fire quickly after the condition is met. Timeliness requires short evaluation intervals and short duration windows.

**CP-alerting system:** Prioritizes completeness and timeliness over accuracy. Evaluate every data point with a short duration. Fire alerts quickly. Accept false positives. Use this for systems where any delay in detection is unacceptable — security intrusions, fraud detection, active incidents.

**AP-alerting system:** Prioritizes accuracy and timeliness over completeness. Use high thresholds and long durations to avoid false positives. Accept that some genuine issues might fire late or be missed. Use this for systems where alert fatigue is the primary concern — low-priority services, infrastructure health.

**CA-alerting system:** Prioritizes completeness and accuracy over timeliness. Evaluate every rule against every data point with long confirmation windows. Accept delays. Use this for systems where false positives are unacceptable — billing systems, compliance monitoring.

### When to Use Each Approach

The CAP approach is not a permanent architecture choice. It is a per-rule strategy. A single alerting system can use CP rules for critical metrics, AP rules for warning metrics, and CA rules for compliance metrics. The key is to classify each rule by the cost of false positives versus false negatives.

## Failure Modes

### Failure Mode 1: Alert Storm

A single root cause triggers dozens or hundreds of alerts. The database goes down, and every service that depends on it fires alerts for latency, errors, and availability. The responder phone blows up. The Slack channel is unusable.

**Root cause:** No deduplication or inadequate grouping. Each service defines its own alerts independently, and nobody considers the blast radius of a single failure.

**Prevention:** Implement deduplication in the alert manager. Group alerts by root cause using dependency mapping. Use hierarchical alerting — if the database is down, suppress alerts from dependent services.

**Detection:** Monitor the number of firing alerts per minute. If the count spikes by 10x or more, an alert storm is happening. Automatically enable grouping and suppression.

### Failure Mode 2: Flapping Alert

An alert fires and resolves repeatedly. The metric crosses the threshold, the alert fires, the metric drops below the threshold, the alert resolves. Then the metric crosses again. The pattern repeats every few minutes.

**Root cause:** Duration set too short. The threshold is close to the metric normal operating range, so small fluctuations cause the alert to oscillate between firing and resolved.

**Prevention:** Increase the duration window. Add hysteresis to the threshold: alert when the metric exceeds X, and resolve when it drops below X - Y (where Y is a margin). Hysteresis prevents flapping by creating a dead zone.

### Failure Mode 3: Silent Failure

A critical component fails, and no alert fires. The team discovers the outage when customers complain.

**Root cause:** Missing alert rule. The component was not covered by monitoring, or the threshold was set too high. The most common silent failure is alerting on the wrong metric. You monitor CPU but not connection count. CPU is low because the server is idle — all connections are hung.

**Prevention:** Map every user-facing flow to a specific alert rule. If a user can experience an error, there should be an alert for it. Use service-level objectives to drive alert creation. If an SLO burn rate exceeds the budget, an alert must fire.

### Failure Mode 4: Alert Sprawl

The number of alert rules grows unbounded. Every team adds alerts without review. After two years, you have 5,000 alert rules. Nobody knows what half of them do. The evaluation latency is 45 seconds per cycle.

**Root cause:** No governance. No review process. No limit on rules per service. No rule retirement policy.

**Prevention:** Enforce a maximum of 20 alert rules per service. Require every alert to have a runbook. Automatically disable alerts that have not fired in 90 days. Quarterly review of all alert rules.

### Failure Mode 5: Escalation Cascade

An alert fires and the primary on-call does not acknowledge within 5 minutes. The alert escalates to the secondary on-call. The secondary is in a meeting and misses it. The alert escalates to the engineering manager. The manager pages the entire team. Everyone gets a phone call at 2 AM for a minor warning alert that could have waited until morning.

**Root cause:** Escalation policies are configured without considering the alert severity. All alerts use the same aggressive escalation chain.

**Prevention:** Different severities get different escalation policies. Critical: 5-minute acknowledgement window, page secondary at 10 minutes, page manager at 20 minutes. Warning: 15-minute window, Slack message to channel. Info: no escalation, no paging.

### Failure Mode 6: Notification Burnout

The on-call engineer receives 20-30 notifications per shift. Most are from the same recurring alerts that fire and resolve automatically. The engineer has learned to acknowledge and dismiss without investigation. The alerts are training the engineer to ignore them.

**Root cause:** No feedback loop. Alerts that fire frequently without requiring action are never reviewed or removed.

**Prevention:** Track the acknowledgment-to-resolution ratio per alert. If an alert is acknowledged but never results in a manual action, flag it for review. Automatically disable alerts that are dismissed without action more than 10 times per month.

### Failure Mode 7: Dead Man Switch Failure

The dead man switch — an alert that fires if a job does not complete on schedule — fails silently. The cron job that was supposed to check the dead man switch fails, and nobody notices because the dead man switch was the only way to detect the cron job failure.

**Root cause:** Single point of failure. The dead man switch depends on the same infrastructure that it is supposed to monitor.

**Prevention:** Use an external monitoring service for the dead man switch. The external service operates independently of the infrastructure being monitored. It checks the dead man switch from outside the system.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Alerting on Symptoms, Not Causes

You alert when latency is high, but you do not alert when a connection pool is exhausted. The result: you know the system is slow, but you have no idea why. By the time you investigate, the connection pool has recovered, and you have no evidence.

**Fix:** Alert on causes, not symptoms. Latency is a symptom. Connection pool utilization is a cause. Queue consumer lag is a cause. Error rate is a symptom. Error types and stack traces are causes. Map each symptom alert to a cause alert. If the cause alert fires first, suppress the symptom alert.

### Anti-Pattern 2: Paging for Everything

Every alert, regardless of severity, goes to PagerDuty and pages the on-call engineer. CPU spikes at 3 AM. Disk usage at 70 percent. A single failed health check. The on-call engineer is paged 15 times per night. They learn to ignore pages.

**Fix:** Classify alerts by severity. Only critical alerts page. Warning alerts go to Slack. Info alerts go to a dashboard. The classification should be based on the expected response time. If the condition can wait until morning, it should not page at night.

### Anti-Pattern 3: No Runbooks Attached

An alert fires at 3 AM. The responder opens the alert and sees: "CPU above 90 percent." No context. No instructions. No dashboard link. The responder spends 20 minutes figuring out what to check.

**Fix:** Every alert must have a runbook URL in its annotations. The runbook is reviewed quarterly. If the runbook is out of date, the alert is disabled until the runbook is updated. The runbook should answer three questions: what to check, what to look for, and what to do.

### Anti-Pattern 4: The Single Static Threshold

You set CPU alert at 90 percent. It fires constantly during peak hours and never fires during off-peak hours. The threshold is either too low (peak) or too high (off-peak). You constantly adjust it.

**Fix:** Use multiple thresholds for different time periods. Or use dynamic thresholds that adapt to the metric baseline. Better yet, use burn rate alerts based on SLOs instead of static thresholds. A single static threshold is almost never correct for a metric with diurnal patterns.

### Anti-Pattern 5: Paging the Same Person

Every alert pages the same person. The on-call rotation is fictional — the same engineer has been on-call for six months because nobody else has been trained. The engineer is burned out and has stopped responding to alerts.

**Fix:** Implement a proper on-call rotation with at least 4 engineers. Each engineer serves 1 week as primary and 4 weeks as secondary. Train every engineer on the runbooks. If a team is too small to support a rotation, they are too small to have 24/7 alerting.

### Anti-Pattern 6: The "Add an Alert for Everything" Approach

Management mandates that every metric must have an alert. Result: thousands of alerts, most of which never fire. The ones that do fire are noise. The team ignores all of them.

**Fix:** Require a justification for every alert. "What action will the responder take?" If nobody can answer that question, do not create the alert. Alerting is a privilege, not a default.

### Anti-Pattern 7: Copying Alerts from Another Team

You see Team X has a set of alerts for their service. You copy them for your service. Result: your service has different traffic patterns, different dependencies, and different failure modes. The copied alerts are inappropriate and noisy.

**Fix:** Design alerts for your specific service. Understand your failure modes first, then design alerts to detect them. Use common templates for similar services, but customize the thresholds, durations, and severities based on your service characteristics.

## The Cost of Over-Alerting

Every alert has a hidden cost. Understanding this cost helps you decide whether an alert is worth creating.

### Direct Infrastructure Cost

Each alert rule requires computational resources for evaluation, storage for state, and network bandwidth for notifications. At 50,000 rules, the infrastructure cost of running the alerting system can exceed $10,000 per month in cloud compute and storage. Each evaluation cycle consumes CPU cycles on the evaluator shard and query capacity on the metrics database.

### Responder Opportunity Cost

Each time an alert pages someone, it interrupts their work. If the average responder takes 30 minutes to investigate and resolve an alert, and the team has 10 responders each handling 20 alerts per week, the total cost is 100 hours per week of engineering time dedicated to alert response. If 80 percent of those alerts are noise, 80 hours per week are wasted.

### Trust Erosion Cost

Every false positive degrades trust in the alerting system. After enough false positives, responders stop responding to alerts entirely. The cost of rebuilding trust after an alerting system has been ignored is higher than the cost of setting up the system in the first place. Trust erosion is the most expensive failure mode because it is invisible — you do not know you have lost trust until a real incident is missed.

### Coverage Gap Cost

Every unnecessary alert consumes budget and attention that could be spent on useful alerts. If a team is limited to 20 alert rules per service, every noise alert prevents a useful alert from being created. The opportunity cost of a noise alert is the useful alert that was not created.

## When NOT to Use Alerting

Alerting is not always the right tool. Here are situations where you should not use alerting.

**When the problem is noise, not detection.** If your system is unstable and metrics are constantly abnormal, alerting will not help. Fix the system first. Alerting a system that is constantly on fire teaches everyone to ignore alerts.

**When nobody can respond.** If the team is too small or does not have 24/7 coverage, alerting at night is pointless. Better to let alerts accumulate for morning review than to page someone who will ignore them.

**When the metric is not actionable.** A metric that cannot be acted on — "number of Kubernetes pods" — should not have an alert. What would the responder do with that information? If there is no action, there should be no alert.

**When the cost of false positives exceeds the cost of missing the alert.** If paging someone costs 30 minutes of sleep disruption and the alert is rarely genuine, the net benefit is negative. The responder will learn to ignore the alert, and eventually a real incident will be missed.

**When the system is not critical enough.** Not every service needs 24/7 alerting. Internal tools, batch jobs that run overnight, and experimental services can use dashboard monitoring instead of active alerting.

**When the metric has high variance with no actionable threshold.** Some metrics naturally fluctuate over a wide range. The number of active users in a social app can vary by 5x between normal and viral-load conditions. No threshold works because the normal range overlaps with the incident range. For these metrics, alert on the rate of change rather than the absolute value, or use anomaly detection.

**When automated remediation already exists.** If a common failure is handled by an automated rollback or auto-scaling policy, creating an alert for it adds noise. The automated system handles it. Alert only if the automated system fails or if human judgment is required. An alert for "auto-scaler triggered" is noise. An alert for "auto-scaler triggered 10 times in 5 minutes and still cannot keep up" is signal.

**When the team is already burned out.** If the team is experiencing alert fatigue, adding more alerts will not help. The solution is to reduce the number of alerts, not increase it. Fix the first three anti-patterns before creating any new alerts.

### The Cost of False Positives vs. False Negatives

Every alerting decision involves a trade-off between false positives (alerting when nothing is wrong) and false negatives (not alerting when something is wrong). The cost of each type of error depends on the system context.

**False positive cost:** A false positive costs the responder 30 minutes of investigation time, disrupts their sleep or focus, and erodes trust in the alerting system. If the false positive rate is high, the entire alerting system becomes unreliable. The cumulative cost of false positives is the number of false positives multiplied by the investigation time plus the trust erosion cost.

**False negative cost:** A false negative costs the business the impact of the undetected incident. A payment service outage that goes undetected for 30 minutes can cost $50,000 in lost revenue. A security breach that goes undetected for 24 hours can cost millions. The cost of a false negative is the incident impact multiplied by the detection delay.

**The break-even analysis:** For each alert rule, calculate the break-even point: the false positive rate at which the cost of false positives equals the cost of false negatives. If the false positive rate is below the break-even point, the alert is worth keeping. If it is above, the alert is doing more harm than good.

### The Alert Budgeting Model

A practical approach to managing the trade-off is the alert budgeting model. Each team has a monthly alert budget: the maximum number of alerts that can fire per month. The budget is allocated across services. If a service exceeds its budget, the team must reduce the number of alerts or increase the budget (by reducing alerts from other services).

The alert budget forces the team to make explicit trade-offs. Every alert that fires consumes budget. The team must decide which alerts are worth the budget consumption. Alerts that fire frequently but rarely result in action are the first candidates for deletion.

### The Severity Classification Matrix

Severity classification is one of the most important trade-off decisions in alerting. If too many alerts are classified as critical, the team experiences alert fatigue. If too few are critical, real problems go unnoticed.

The severity classification matrix maps the metric severity to the appropriate response:

| Severity | Impact | Response Time | Channel | Escalation |
|----------|--------|---------------|---------|------------|
| P0 (Critical) | Customer-facing outage | 5 minutes | PagerDuty + SMS + Phone | 5 min primary, 10 min secondary, 20 min manager |
| P1 (High) | Service degradation | 15 minutes | PagerDuty + SMS | 15 min primary, 30 min secondary |
| P2 (Warning) | Potential issue | 1 hour | Slack | No escalation |
| P3 (Info) | Informational | 24 hours | Dashboard | No escalation |

The severity classification should be based on the metric impact, not the metric value. A 500ms latency increase might be P0 for a payment service and P3 for a reporting dashboard. The severity is determined by the business context, not the technical characteristics.

### The Threshold Selection Paradox

Selecting the right threshold is one of the hardest problems in alerting. The paradox is that you need historical data to set the right threshold, but you need the threshold to collect historical data. The solution is an iterative approach:

**Phase 1 — Initial guess.** Use p99 of the metric from the first week of data. If you do not have a week of data, use the p99 from the monitoring system default. The initial guess is intentionally conservative — it will produce some false positives, but it will not miss real incidents.

**Phase 2 — Adjustment.** After 2 weeks, review the alert performance. If the false positive rate is above 50 percent, raise the threshold by 10 percent. If the alert never fired, lower the threshold by 10 percent. Repeat every 2 weeks.

**Phase 3 — Stabilization.** After 3 months, the threshold should be stable. The false positive rate should be under 10 percent. The alert should have fired at least once during the 3-month period.

### The Duration-Volatility Relationship

The relationship between metric volatility and the appropriate duration is nonlinear. A volatile metric needs a duration that is long enough to avoid false positives but short enough to detect real problems. The relationship is:

```
duration = k * volatility
```

Where k is a constant that depends on the cost of false positives. For a high-cost system (payment service), k is 5. For a low-cost system (reporting dashboard), k is 20.

**Example:** A volatile metric with a standard deviation-to-mean ratio of 0.5. For a payment service (k=5), the duration is 5 * 0.5 = 2.5 evaluation intervals, round up to 3. For a reporting dashboard (k=20), the duration is 20 * 0.5 = 10 evaluation intervals.

### The Multiplicity Trade-off

Each alert rule can be configured with a single threshold or multiple thresholds. Multiple thresholds provide more nuanced alerting but increase complexity.

**Single threshold:** Simple, easy to understand, but limited. A single threshold gives you two states: below threshold (OK) and above threshold (firing). This is sufficient for most metrics.

**Dual threshold:** Two thresholds create three states: below lower threshold (OK), between thresholds (warning), above upper threshold (critical). This provides more granularity. The warning threshold is set lower than the critical threshold. The warning alerts the team before the condition becomes critical.

**Multi-threshold:** Three or more thresholds create multiple states. This is useful for metrics that have different severity levels. For example, CPU usage: 80 percent (warning), 90 percent (high), 95 percent (critical). Each threshold triggers a different notification channel and escalation policy.

## Failure Modes: Advanced Analysis

### The Cascade Failure of Silence Rules

A team silences an alert for a planned deployment. The deployment causes an unexpected failure. The alert is silenced, so nobody is notified. The failure goes undetected for 30 minutes.

**Root cause:** The silence rule was too broad. It matched the alert name, severity, and labels, but it also matched alerts that were not related to the deployment.

**Prevention:** Silence rules should be as specific as possible. Match on the exact alert name, the exact time window, and the exact set of labels. Do not use wildcards in silence rules. The silence rule should be reviewed by a second person before activation.

### The Feedback Loop of Alert Fatigue

Alert fatigue creates a dangerous feedback loop. As false positives increase, responders become less responsive. As responders become less responsive, the team adds more alerts to compensate. The additional alerts increase false positives further. The cycle continues until the alerting system is completely ignored.

**Breaking the cycle:** The only way to break the feedback loop is to reduce the number of alerts. Delete the most frequently firing alerts first. Then reduce the thresholds and increase the durations for the remaining alerts. The goal is to reduce the total number of alerts per day by 50 percent in the first month.

### The Black Swan Alert

A black swan alert is an alert that fires for a condition that was never anticipated. The alert fires for the first time, and the runbook does not cover the scenario. The responder has no guidance on what to do.

**Prevention:** Black swan alerts cannot be eliminated, but they can be mitigated. Every alert should have a generic runbook section that covers "unknown root cause." The generic section should include: how to collect diagnostic data, who to contact for help, and how to escalate if the problem is not understood.

### The Tail Latency Problem

Alerting on p99 latency is standard practice. But p99 latency can be misleading. A single slow request can push the p99 to 5 seconds, even though 99 percent of requests are under 200ms. The alert fires for a condition that affects 1 percent of requests.

**Trade-off:** Alerting on p50 is too sensitive — it fires for normal fluctuations. Alerting on p99 is too specific — it misses problems that affect a minority of users. The solution is to alert on multiple percentiles: p50, p90, p99, and p99.9. Each percentile provides a different perspective on the system health.

### The Prioritization Matrix

When you have limited engineering resources and too many possible alerts to create, use the prioritization matrix to decide which alerts to implement first:

| Criteria | Weight | Alert A | Alert B | Alert C |
|----------|--------|---------|---------|---------|
| Business impact | 5 | 5 (25) | 3 (15) | 2 (10) |
| Detection latency | 4 | 4 (16) | 2 (8) | 3 (12) |
| False positive risk | 3 | 1 (3) | 4 (12) | 4 (12) |
| Implementation effort | 3 | 2 (6) | 3 (9) | 1 (3) |
| Total score | | 50 | 44 | 37 |

The matrix forces you to evaluate each potential alert against the same criteria. The highest-scoring alerts are implemented first. The matrix is reviewed quarterly as the system evolves and new failure modes are discovered.

### The Alert Quality Score

Each alert rule should have a quality score. The quality score is a number from 0 to 100 that rates the alert effectiveness. The score is computed from:

**Actionability (30 points):** Does the alert have a runbook? Does the runbook cover the most common causes? Has the runbook been followed in the last 3 incidents? An alert with no runbook gets 0 points.

**Reliability (25 points):** What is the false-positive rate? Alerts with a false-positive rate under 5 percent get full points. Alerts with a false-positive rate over 50 percent get 0 points. The false-positive rate is calculated from the last 90 days.

**Timeliness (20 points):** What is the detection latency — the time from the onset to the alert firing? Alerts that detect within 1 minute get full points. Alerts that detect after 10 minutes get 0 points. The detection latency is measured from the metric timeline.

**Coverage (15 points):** Does the alert cover the most common failure modes? An alert that covers the top 3 failure modes gets full points. An alert that covers 0 gets 0 points. The failure modes are identified from the incident history.

**Maintenance (10 points):** Is the alert reviewed regularly? Alerts reviewed in the last 90 days get full points. Alerts never reviewed get 0 points. The review timestamp is tracked in the configuration store.

### The Alert Debt Concept

Alert debt is the accumulated cost of poorly configured alerts. Similar to technical debt, alert debt grows over time if left unaddressed. An alert that has never fired, has no runbook, or has a false-positive rate above 50 percent is alert debt.

**Measuring alert debt:** Alert debt is measured as the number of alert rules that fail the quality check. Each rule that has no runbook, has not fired in 90 days, or has a false-positive rate above 50 percent is counted as debt. The debt-to-equity ratio is the ratio of debt alerts to healthy alerts.

**Paying down alert debt:** Each quarter, the team dedicates 20 percent of their alerting time to paying down debt. The team reviews the lowest-quality alerts and either fixes or deletes them. The target debt-to-equity ratio is under 0.2 — no more than 20 percent of alerts should be debt.

### The On-Call Burnout Index

The on-call burnout index measures the strain that the alerting system puts on the on-call engineer. The index is computed from:

**Alert volume:** The number of alerts per shift. If the alert volume exceeds 10 per shift, the index increases by 1. If it exceeds 20 per shift, the index increases by 2.

**False positive rate:** The percentage of alerts that are false positives. If the false positive rate exceeds 50 percent, the index increases by 1. If it exceeds 80 percent, the index increases by 2.

**Off-hours pages:** The number of pages received between 10 PM and 6 AM. If the off-hours page count exceeds 3 per week, the index increases by 1. If it exceeds 5, the index increases by 2.

**Investigation time:** The average time to investigate and resolve an alert. If the investigation time exceeds 30 minutes, the index increases by 1. If it exceeds 60 minutes, the index increases by 2.

An index of 0-2 is healthy. An index of 3-4 requires intervention. An index of 5 or above is critical — the on-call engineer is burned out and the alerting system must be redesigned.

### The Five Whys for Alert Design

Use the five whys technique when designing a new alert rule:

**Why 1:** Why do we need this alert? (To detect when the payment service is slow.)

**Why 2:** Why is the payment service slow a problem? (Because customers cannot complete purchases.)

**Why 3:** Why does the payment service become slow? (Because the database connection pool is exhausted.)

**Why 4:** Why does the database connection pool become exhausted? (Because the connection timeout is too long.)

**Why 5:** Why is the connection timeout too long? (Because the timeout was increased to handle a previous issue and was never reverted.)

The five whys reveal that the root cause is a configuration change, not a resource exhaustion. The alert should detect the configuration drift, not the connection pool exhaustion. The alert is redesigned to detect timeout value changes.

### The Alerting Paradox

The alerting paradox is that the best alerting system is the one that fires the least. If the system is well-designed — services are resilient, failures are handled gracefully, and incidents are rare — the alerting system fires rarely. But a system that fires rarely is hard to maintain because the alert rules are not exercised and the responders are not practiced.

**Resolution:** The paradox is resolved by separating alerting from testing. Use chaos engineering to exercise the alerting system regularly. Chaos experiments simulate failures and verify that the alerting system fires correctly. The experiments keep the alerting system sharp even when the production system is stable. This is the Netflix approach: run chaos experiments during business hours, verify that the alerting system catches every experiment, and fix any gaps.

### The Alerting System Decision Tree

When deciding whether to create an alert, follow this decision tree:

1. Does this metric represent a condition that requires human action?
   - Yes: Continue to step 2.
   - No: Do not create an alert. Log the metric to a dashboard.

2. Is there a runbook for this alert?
   - Yes: Continue to step 3.
   - No: Write the runbook first. If you cannot write a runbook, do not create the alert.

3. Is the threshold based on historical data?
   - Yes: Continue to step 4.
   - No: Collect 30 days of historical data. Set the threshold based on the p99 value.

4. Is the duration set to at least 3 evaluation intervals?
   - Yes: Continue to step 5.
   - No: Set the duration to 3 evaluation intervals. Adjust based on metric volatility.

5. Is the notification channel appropriate for the severity?
   - Yes: Continue to step 6.
   - No: Set the notification channel based on the severity. Critical -> PagerDuty. Warning -> Slack. Info -> Dashboard.

6. Is the escalation policy configured?
   - Yes: Continue to step 7.
   - No: Configure the escalation policy. Critical -> 5-min acknowledgment window. Warning -> 15-min window. Info -> No escalation.

7. Is the alert rule tested in staging?
   - Yes: Deploy to production.
   - No: Test in staging for 48 hours. Review the results. Deploy to production only if the false-positive rate is under 10 percent.

### The Alerting System Anti-Patterns Summary

| Anti-Pattern | Symptom | Root Cause | Fix |
|-------------|---------|------------|-----|
| Paging for everything | 50 pages per night | No severity classification | Classify alerts by severity |
| No runbooks | 30-minute investigation | No runbook requirement | Add runbook to every alert |
| Single static threshold | Flapping at peak hours | No threshold tuning | Use dynamic thresholds |
| Alerting on symptoms | Slow investigation | Wrong metric selection | Alert on causes, not symptoms |
| Copying from other teams | Wrong thresholds | No service-specific tuning | Design alerts for each service |
| No deduplication | Alert storms | No grouping | Implement deduplication |
| No escalation | Missed alerts | No escalation policy | Configure escalation chains |
| No self-monitoring | Alerting system is down | No monitoring of monitoring | Add self-monitoring alerts |
| No pruning | 5,000 alerts | No review process | Quarterly alert review |
| No postmortems | Same incidents repeat | No feedback loop | Postmortem for every incident |
