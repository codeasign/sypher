## The Three Incidents That Define the Problem

Three real-world scenarios illustrate why distributed counters dominate system design discussions — and why getting them wrong costs real money.

**Scenario 1: The Oversold Movie Ticket**

A ticketing platform sells 500 seats for a premiere screening. Two application servers each read the current seat count at precisely the same instant: both see 13 remaining seats. Server A books seat 14 of 13 for Customer X. Server B books seat 15 of 13 for Customer Y 200 milliseconds later. The platform has sold 515 tickets for 500 seats. At showtime, 15 paying customers with valid tickets are turned away. The counter was a single integer column in PostgreSQL with no atomic increment — each server read the value, decremented it in application memory, and wrote the result back, creating a classic read-modify-write race.

**Scenario 2: The Viral Post That Broke the Like Counter**

A social media platform launches a new feature: live like counts on posts. A celebrity posts a photo. Within 90 seconds, 340,000 likes pour in from users across the globe. The MySQL database backing the like counter hits 100% CPU on the primary node. Write queries queue 8 seconds deep. The counter increments at 28 writes per second — the database cannot keep pace with the 3,800 concurrent likes per second arriving at the application layer. The application starts batching writes in memory every 5 seconds, but when one application server crashes, it loses 14,000 buffered likes. The counter stabilizes at 298,000, but the celebrity's agent has screenshots showing 326,000 and demands an explanation. The platform has no way to reconcile the gap.

**Scenario 3: The Ad Impression Auditing Failure**

An ad network counts impressions for a campaign budgeted at $500,000. The counter is stored in a distributed key-value store with eventual consistency. Three ad servers in different regions each serve 12,000 ads for the same campaign within the same second. Each server increments a local replica of the counter. Because the system uses last-writer-wins conflict resolution, two of the three increments are silently discarded during replication. The counter reads 12,000 instead of 36,000. The campaign budget burns through its $500,000 allocation 3 times faster than expected. The advertiser is billed $1.5 million. The ad network eats the difference.

All three incidents share one root cause: **a single integer counter becomes a contention hotspot that breaks under concurrent writes from distributed clients**.

## Insight

A distributed counter is a data structure that maintains a numerical count across multiple machines, handling concurrent increments from any node without data loss, races, or unbounded contention.

The group-chat reaction analogy: imagine a live event with 10,000 people in a chat room all reacting to the same announcement. If everyone sends "1" and you add them up one at a time, the first 50 come through instantly, then messages queue, then the server times out. The total is wrong. Now imagine each person's reaction increments a local counter on their own device, and those local counters merge periodically using a rule that never loses an increment. That is a distributed counter — the group chat where every reaction counts exactly once, no matter how many people tap at the same instant.

The core challenge is not counting — it is coordinating the count across machines that cannot trust their clock, their network, or their peers to stay alive.

## Core Vocabulary

Master these terms before moving to the deep dive.

**Distributed counter** — A data structure that accepts concurrent increment and decrement operations from multiple nodes and converges to a single authoritative total. Why it matters: without this construct, you resort to centralized locks or single-node databases that become contention bottlenecks.

**Conflict-free Replicated Data Type (CRDT)** — A data structure designed so that concurrent updates from different nodes can be merged without conflicts, producing the same state on all nodes. PN-Counter (positive-negative counter) is the CRDT counter: two grow-only counters, one for increments and one for decrements, that merge as element-wise maximum. Why it matters: CRDTs eliminate the need for consensus on every write, enabling sub-millisecond increments on any node.

**G-Counter (Grow-Only Counter)** — A CRDT counter that supports only increment operations. Each node tracks its own contribution. The total is the sum of all node contributions. Merging takes the element-wise maximum of each node's value. Why it matters: for use cases that never decrement (page views, impressions), a G-Counter is the simplest possible distributed counter with no conflict resolution needed.

**PN-Counter (Positive-Negative Counter)** — A CRDT counter that supports both increments and decrements by combining two G-Counters: one for positives, one for negatives. The total is the sum of positives minus the sum of negatives. Why it matters: supports like/dislike, seat inventory, and any count that goes both up and down, without single-node coordination.

**Merge operation** — The function that combines two replicas of a distributed counter into a single consistent state. For G-Counters, merge is element-wise max. Why it matters: the merge operation determines what happens during network partitions and concurrent writes — a well-defined merge means the system converges automatically.

**Sharded counter** — A single logical counter partitioned across multiple physical nodes, each handling a subset of the increment traffic. The total is the sum of all shard values. Why it matters: sharding eliminates the single-node write bottleneck, enabling throughput from hundreds of thousands to millions of increments per second.

**Clock-based staleness** — The condition where two replicas of a counter disagree because updates have propagated from one replica but not yet to the other. Every eventually-consistent counter experiences this. Why it matters: applications that need threshold semantics (stop selling when the counter hits zero) cannot use stale counters without overshoot.

**Read-after-write staleness window** — The time between writing an increment and being able to read it from any replica. Why it matters: the staleness window determines how much overshoot a counter can experience. A 100-millisecond window with 50,000 increments per second means up to 5,000 increments of overshoot.

**Counter tombstone** — A record that a counter was decremented, preserved to prevent re-application of the same decrement during merge. Why it matters: without tombstones, merging replicas can double-apply decrements and undercount. Tombstones make the state grow unboundedly unless garbage-collected.

**Delta state replication** — A technique where replicas exchange only the changes since the last sync, not the full counter state. Why it matters: reduces network bandwidth and merge compute cost by orders of magnitude for large counters, especially those with many shards or nodes.

**Distributed sequence counter** — A counter that generates globally unique, monotonically increasing sequence numbers across nodes (like a distributed auto-increment). Why it matters: different from a summation counter — it does not track a total; it tracks ordering. Snowflake IDs, Twitter's approach, and database sequence generators are examples.

## How It Actually Works

Nine numbered steps trace a distributed counter write from application code to convergent state. The scenario: a social feed with a "likes" counter that must withstand 100,000 concurrent likes per second.

### Step 2: Shard Assignment

The API gateway hashes `post_id` using consistent hashing across 64 counter shards. `hash("abc123") % 64 = 7`. The request is forwarded to shard 7 — a Redis instance dedicated to approximately 1/64th of all post counters.

### Step 3: Local Increment on Shard

Shard 7's Redis executes `INCR likes:abc123`. This is a single-threaded, atomic operation — no two concurrent increments can interleave because Redis processes commands sequentially on a single event loop. The operation completes in 0.08 milliseconds (80 microseconds) on a warm cache.

### Step 4: Async Replication to Backup

Shard 7 writes the increment to its Redis AOF (Append-Only File) and asynchronously replicates to a Redis replica. The replica acknowledges receipt but the primary does not wait for the replica to apply the change before responding to the client.

```

      │
      │ Response < 0.1ms
      ▼
┌──────────┐

└──────────┘
```

### Step 5: Client Gets Acknowledgment

The API gateway responds to the client with HTTP 200 and the new counter value from shard 7. The entire round trip takes 3 to 15 milliseconds depending on network distance.

**Problem**: The response reflects only shard 7's value. If this post has been liked from multiple shard assignments (possible with reshards or migration), the response is an undercount — it represents the value from one shard. This is the staleness trade-off.

### Step 6: Periodic Merge (for CRDT-based counters)

In a CRDT-based approach, each node maintains a vector of (node_id, contribution). Periodically — every 100 milliseconds or every 1,000 increments, whichever comes first — nodes exchange their delta vectors using a gossip protocol, taking the element-wise maximum for each node's contribution.

```
Node A: [A: 342, B: 0, C: 0]
Node B: [A: 0, B: 287, C: 0]
Node C: [A: 0, B: 0, C: 401]

After merge:
All nodes: [A: 342, B: 287, C: 401]
Total: 342 + 287 + 401 = 1,030
```

### Step 7: Read Path — Aggregate Across Shards

When a client reads the counter value, the read coordinator queries all 64 shards in parallel, sums their values, and returns the total.

```python
# Read: aggregate across all shards
shard_values = []
for shard_id in range(64):
    val = redis_client.get(f"post:{post_id}:shard:{shard_id}")
    shard_values.append(int(val or 0))
total = sum(shard_values)
```

This read path is expensive: 64 parallel Redis calls per read. To reduce the cost, systems cache the aggregated total with a TTL of 100-500 milliseconds, serving a read from the cache and updating it asynchronously.

### Step 8: Reconciliation During Partition Healing

When a network partition heals — say, shard 7 was isolated from shard 8 for 2 seconds — the replicas exchange their delta state. Shard 7 received 4,200 increments while isolated; shard 8 received 3,800. Each side applies the missing increments from the other. Because increments are commutative (A+1+2 equals A+2+1), the total converges without conflict.

```
Shard 7 before partition:  142,000
Shard 8 before partition:  138,000
During partition:
  Shard 7 receives +4,200 → 146,200
  Shard 8 receives +3,800 → 141,800
After merge:
  Both shards: 146,200 + (delta from shard 8) ??? No.
  Actually: both shards should show the SAME merged state.
  If they merge by summing shard contributions:
  Shard 7's contribution during partition: +4,200
  Shard 8's contribution during partition: +3,800
  Total after merge: 142,000 + 138,000 + 4,200 + 3,800 = 288,000
```

The merge is lossless because each node's contribution is an independent vector element, and merge = element-wise max (for G-Counter) or element-wise merge of two G-Counters (for PN-Counter) — never a simple overwrite.

### Step 9: Garbage Collection of Tombstones (PN-Counter)

For counters that support decrements, the PN-Counter stores its decrements in a separate grow-only counter. Each decrement is a "tombstone" entry. Over time, the tombstone vector grows without bound. A periodic garbage collection pass identifies node contributions that are older than a retention window (24 hours) and zeros out the corresponding element if the node has not been active, preserving the net total while compacting state.

```
Before GC:
  Positive: [A: 482,000, B: 0, C: 310,000, D: 0]
  Negative: [A: 12,000, B: 0, C: 14,000, D: 0]
  Total: 792,000 - 26,000 = 766,000

After GC (node A last active 30 hours ago, node C active now):
  Positive: [A: 0, B: 0, C: 310,000, D: 0]
  Negative: [A: 0, B: 0, C: 14,000, D: 0]
  Total: 310,000 - 14,000 = 296,000
```

GC changes the total because node A's 470,000 net positive (482,000 - 12,000) was from an expired billing cycle. The system has a separate authoritative total stored in the data warehouse. The counter is an estimate for real-time display, not the source of truth.

## Variants and Flavors

Three architectural variants of distributed counters, each suited to different workloads.

### Variant 1: Single-Writer Sharded Counter (Redis Cluster)

Each logical counter has a designated primary shard (one Redis node). All increments route to that shard. The shard is one of many, but each counter is owned by exactly one node.

**When to use**: You need strong read-after-write semantics — when a user likes a post and immediately sees the incremented count. The single-writer guarantee means the read from the same shard returns the latest value.

**When NOT to use**: You have counters that must survive the loss of a shard without data loss (a single Redis node is a single point of failure for its counters). You also hit throughput ceilings at roughly 500,000 increments per second on a single Redis node, though with 64 shards you reach 32 million per second aggregate.

### Variant 2: Multi-Writer CRDT Counter

Every node accepts increments for any counter. No routing is needed. Nodes merge their state periodically via gossip or direct synchronization.

**When to use**: You need ultra-high write throughput (millions per second) and can tolerate eventual consistency. You are counting something that never decrements (page views, impressions, API calls) — a G-Counter suffices.

**When NOT to use**: You need strong decrement semantics — a PN-Counter requires tombstones and garbage collection, adding complexity. You also need bounded staleness — gossip convergence has no guarantee on how long a merge takes.

**When NOT to use**: You cannot tolerate data loss on application server crash — each local batch buffer is lost if the server dies before flushing. You need writes to be durably committed before the client gets a response.

## Mental Model Check

Test your understanding with these five questions. Cover the answer, attempt a response, then compare.

**Q1: A G-Counter uses element-wise maximum as its merge function. Why maximum and not sum?**

A: Sum would double-count increments during merge. If Node A has value 5 and Node B has value 3, and both increments represent the same logical events (replicated to both nodes), summing gives 8 — overcounting by the overlapping events. Element-wise maximum means "the authoritative value for each node's contribution is the highest value any replica has seen for that node." Since each node owns its own contribution, the max preserves the most up-to-date value without duplication.

**Q2: A PN-Counter uses two G-Counters internally. When you merge two PN-Counter replicas, how do you compute the final value?**

A: Let P1, N1 be the positive and negative G-Counters of replica 1, and P2, N2 be those of replica 2. The merged positive G-Counter is element-wise max(P1, P2). The merged negative G-Counter is element-wise max(N1, N2). The final value is sum(merged_positive) - sum(merged_negative). This preserves every increment and every decrement without double-counting.

**Q3: You are designing a like counter that must display the updated count within 200 milliseconds of a user tapping the like button. Can you use an eventually-consistent CRDT?**

A: Yes, if you also implement a read-repair mechanism. When a client reads the counter, the serving node requests pending delta from N-1 peers before responding, providing strong read-after-write within a bounded window. Alternatively, use a sharded Redis counter with the read going to the same shard as the write — the staleness window is essentially zero for a single shard because Redis INCR is atomic and the read hits the same node.

**Q4: A gossip-based distributed counter has 100 nodes. Each gossip round takes 500 milliseconds and each node contacts 3 random peers per round. After how many rounds does the probability of a given node having received an update exceed 99%?**

A: After 4 rounds (2 seconds), the update has spread with high probability. The "infectious" spread is exponential: each round multiplies the informed set by approximately the fan-out factor. After log_base_fan-out(N) rounds, all nodes are informed. With fan-out 3 and 100 nodes: log_3(100) = 4.2, so approximately 4-5 rounds.

**Q5: Your team implements a two-layer counter where application servers batch increments for 2 seconds before flushing to Redis. During a deployment, 3 out of 20 application servers crash over the span of 10 seconds. Approximately how many increments are lost?**

A: Assuming each server handles 5,000 increments per second, each server has up to 10,000 increments buffered (2 seconds of batch). Three crashing servers lose up to 30,000 increments. The total counter could be undercounted by 0.03% if the system processes 100 million total increments — or by a meaningful amount if the crashed servers handled a popular post. Mitigation: flush the batch to a local write-ahead log before acknowledging the client, or reduce the batch window.
