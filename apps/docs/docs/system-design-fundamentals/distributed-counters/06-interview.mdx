## How This Topic Appears

Distributed counters appear in system design interviews in four categories. Here are the actual phrasings you will encounter.

### Direct Questions (2-3)

1. "Design a distributed counter system that can handle 1 million increments per second."
2. "How would you implement a like counter for a social media platform with 200 million daily active users?"
3. "Design a real-time view counter for a video streaming platform that processes 100 million views per day."

### Design Questions (2-3)

4. "Design a rate limiting system that enforces per-user API limits across 100 application servers."
5. "Design a real-time leaderboard that updates within 100 milliseconds of a new score submission."
6. "Design a system to track available inventory across hundreds of distributed warehouse pick stations."

### Debugging Questions (1-2)

7. "Our like counter shows 1.2 million likes, but the actual number from our event log is 1.35 million. Where did 150,000 likes go?"
8. "A Redis cluster node handling counter increments crashed, and after failover, some counters lost 30 seconds of data. How do you recover?"

### Trade-off Questions (2-3)

9. "Should we use a CRDT counter or a sharded Redis counter for our voting system? Why?"
10. "Our counters are in Cassandra and reads are slow. Should we add a Redis cache layer or fix the Cassandra read path?"
11. "What is the maximum accuracy we can guarantee for a distributed counter, and what determines that bound?"

## What the Interviewer Is Actually Testing

Interviewers ask about distributed counters to evaluate five skills:

**1. Understanding of the CAP theorem in practice.** Do you know that a distributed counter is AP by default and can you explain when and how to shift toward CP? The interviewer wants to see that you do not treat CAP as an abstract triangle but as a concrete tool for making design decisions.

**2. Ability to identify and mitigate the hot key problem.** The most common production failure of distributed counters. Can you spot that a viral post creates a single-shard bottleneck? Can you propose pre-sharding or multi-key distribution before being prompted?

**3. Comfort with CRDT merge semantics.** Can you explain why element-wise max works for G-Counters and why it fails for PN-Counters without tombstones? The interviewer is checking whether you understand the math or just memorized the term "CRDT."

**4. Operational maturity.** Do you know that counters lose increments? Do you have a plan for monitoring staleness, reconciling with an event log, and handling failover loss? The interviewer wants to see the production engineer, not the algorithm designer.

**5. Trade-off articulation under constraints.** Given a specific latency, throughput, and accuracy requirement, can you choose between Redis shards, Cassandra counters, and CRDT gossip? Can you justify the choice with concrete numbers and failure scenarios?

## Level-by-Level Expectations

Your answer must match the seniority level of the role.

### L3/L4 (Entry to Mid-Level)

**What is expected**: Identify the core problem. Propose a sharded Redis counter with consistent hashing. Mention idempotency keys. Calculate basic throughput requirements.

**Sample answer**: "I would shard the counter across multiple Redis instances using consistent hashing. Each increment becomes an INCR command to one Redis node. I would use 64 shards to handle 3.2 million ops/sec, with 2 replicas per shard for fault tolerance. Idempotency keys prevent double-counting on retries. Reads fan out to all shards and sum the results, with a 100ms cache to reduce read load."

**What is NOT expected**: CRDT internals, gossip protocol convergence, vector clock analysis, or multi-region topology. These are bonus points at this level.

### L5 (Senior)

**What is expected**: All of L3/L4 plus: trade-off analysis between sharded Redis and CRDT counters, hot key mitigation strategy, explanation of how consistent hashing handles shard addition/removal, numbers for throughput and latency, and a basic failure mode analysis.

**Sample answer**: (L3/L4 answer plus:) "The trade-off between sharded Redis and CRDT counters comes down to read-after-write consistency. Sharded Redis gives strong consistency per shard because INCR is atomic and the read goes to the same shard. CRDT counters sacrifice read-after-write for unrestricted write throughput — any node can increment any counter. For a like counter where users expect to see their like reflected immediately, sharded Redis is better. For multi-region impression counting where throughput dominates and staleness is acceptable, CRDT is better.

For hot keys, I would pre-shard individual counters into N sub-counters at the application layer. A counter for a viral post becomes 64 sub-counters distributed across 64 shards. The total is the sum of all sub-counters. This spreads the write load. The downside is that reads become more expensive: 64 reads instead of 1.

In production, we would also run batch reconciliation hourly from an event log to correct any lost increments."

### L6 (Staff)

**What is expected**: All of L5 plus: design a multi-region topology with cross-region CRDT merge, quantify the staleness window in each mode, discuss when to use synchronous vs. asynchronous replication for the counter state, present a detailed monitoring and alerting plan, and discuss the cost implications.

**Sample answer**: (L5 answer plus:) "For global scale, each region runs its own counter cluster. Cross-region replication uses Kafka MirrorMaker to stream increment events between regions with a 500ms-2 second lag. The global total is a CRDT merge of each region's shard totals, computed by a global aggregator every 5 seconds.

The staleness window for a cross-region read is: (1) local read cache TTL (100ms) + (2) cross-region replication lag (500ms-2s) + (3) CRDT merge interval (5s) = total staleness of 5.6-7.1 seconds. If this is unacceptable, we can reduce the CRDT merge interval to 1 second, trading compute cost for freshness.

For cost: 64 Redis shards × 8 GB at $240/month each = $15,360/month. Cross-region data transfer for replication adds another $5,000-10,000/month. Total infrastructure cost: approximately $25,000/month for three regions serving 10 million writes/sec."

### L7/L8 (Principal/Distinguished)

**What is expected**: All of L6 plus: challenge the premise of the question and propose alternative approaches, discuss the system at the level of business impact ("this saves us $2M/year in infrastructure"), frame the trade-off in terms of revenue and user experience, and describe how this system evolves over 3-5 years as the product changes.

**Sample answer**: (L6 answer plus:) "Before building this, I would question whether the counter needs to be distributed at all. If the requirement is 10K increments per second and the latency target is 10ms, a single PostgreSQL row with an atomic UPDATE works and costs $500/month instead of $25,000/month. Many teams over-engineer counters because they assume they will reach Twitter scale.

If distributed is indeed required, I would also ask: who reads the counter and how often? If it is displayed to users, does a 10% undercount for the first 5 seconds matter? If not, a Cache-Aside pattern with a 5-second TTL eliminates the need for a read aggregator entirely, simplifying the architecture.

Over 3 years: start with Redis shards (fast, simple), add CRDT merge for multi-region at year 2, and replace Redis with a purpose-built in-memory counter service using something like Dragonfly or a custom Rust service at year 3 when the throughput exceeds what Redis sharding can handle."

## The FAANG Framework for Answering

Use this 6-step framework to structure any distributed counter question.

**Step 1: Clarify requirements (2-3 minutes)**

Ask the interviewer three clarifying questions before proposing any design:

- "What is the target write throughput: 10K, 100K, or 10M increments per second?"
- "What is the acceptable staleness for reads: under 10ms, under 1 second, or eventual?"
- "Can we tolerate lost increments, or must each increment be durably stored before acknowledgment?"

**Step 2: Define the counter type (1 minute)**

Based on the answers, choose a counter architecture:

- Single DB counter: under 10K writes/sec, under 10ms staleness, no loss tolerated.
- Sharded Redis: 10K-1M writes/sec, under 10ms staleness per shard, small loss tolerated.
- CRDT gossip: 1M-10M writes/sec, eventual staleness, loss tolerated within sync window.
- Two-layer (buffer + Redis): write peaks over 1M/sec with cushioning, loss on server crash tolerated.

**Step 3: Calculate shard count (2 minutes)**

Use the throughput requirement to size the cluster:

- "At 50,000 ops/sec per Redis shard, for 500,000 writes/sec I need 10 shards minimum. I will use 16 shards for 37% headroom. Each shard gets 1 replica in a different availability zone."

**Step 4: Describe the write path (3 minutes)**

Walk through the increment lifecycle: client → rate limiter → idempotency key check → consistent hash → shard INCR → replication → response. Use specific numbers for each step's latency.

**Step 5: Describe the read path (2 minutes)**

Explain how reads work: aggregator fan-out with parallel requests to all shards, cache layer, timeout handling, and staleness implications. Include the trade-off: faster reads mean staler data.

**Step 6: Discuss failure modes and mitigations (5 minutes)**

Cover the top three failures: hot key, shard crash with lost increments, partition with divergence. For each: detection, blast radius, recovery steps.

## Phrase Bank

Use these phrases to signal expertise. The italicized context tells you when to use each one.

1. "The fundamental tension is write throughput versus read accuracy — every additional shard increases write capacity but widens the staleness window for reads." (Opening the trade-off discussion)

2. "I will pre-shard hot keys at the application layer into N sub-keys distributed across all shards." (Hot key mitigation)

3. "Idempotency keys on every write with a configurable TTL — 300 seconds as default." (Write reliability)

4. "The CRDT merge function is element-wise maximum for the G-Counter, which means no increment is ever lost." (Explaining CRDT correctness)

5. "For a PN-Counter, decrements are stored as a separate grow-only counter. The final value is the sum of positives minus the sum of negatives. Tombstone GC runs every 24 hours." (PN-Counter explanation)

6. "Read-after-write consistency is guaranteed only when the read targets the same shard (and preferably the same node) as the write." (Explaining per-shard consistency)

7. "The staleness window is bounded by: (cache TTL) + (replication lag) + (gossip convergence time). For our architecture, that is 100ms + 5ms + 200ms = approximately 305ms." (Quantifying staleness)

8. "Under a network partition, both sides continue accepting writes. After the partition heals, the CRDT merge converges both sides to the sum of all increments from both sides." (Partition behavior)

9. "We accept that the distributed counter is a real-time estimate. The authoritative total comes from replaying the event log through a batch pipeline." (Second source of truth)

10. "The hot key problem is the ceiling on write throughput for a single counter. Pre-sharding is the only mitigation — auto-scaling does not help because the bottleneck is serial execution on a single thread." (Hot key root cause)

11. "For LWW (last-writer-wins) we rely on HLC (hybrid logical clocks) to avoid clock skew causing lost increments." (Clock synchronization)

12. "Cassandra counters use PN-CRDT internally, which means reads must contact all replicas to get the latest total." (Cassandra counter behavior)

13. "The p99 staleness is often 10-100x worse than p50 during hot key events. Alert on p999, not p50." (Monitoring wisdom)

14. "Tombstone GC must be scheduled, or the PN-Counter state grows unboundedly. We GC on a 24-hour schedule and archive the removed state to S3." (Operational detail)

15. "Dual-write during shard rebalancing: write to both old and new shards for one full TTL period, then cut over." (Operational procedure)

16. "The CAP positioning is AP with tunable read modes. Default is eventual (fastest), strong reads contact the primary shard (slower but consistent)." (CAP positioning)

## Red Flags

Phrases and approaches that signal inexperience.

1. "I will use a single Redis instance with INCR." (At 1M increments/sec, this works for 20 seconds before the single core is pegged.)

2. "CRDTs always converge immediately." (They converge after O(log N) gossip rounds, which takes hundreds of milliseconds.)

3. "I do not need idempotency because the network is reliable." (Network retries happen at every layer. Without idempotency, you overcount.)

4. "I will use a distributed counter for the billing system." (Distributed counters lose increments. Billing needs a deterministic, auditable count.)

5. "I will just add more Redis shards to fix the hot key." (The hot key goes to one shard. More shards do not help — the bottleneck is single-threaded serial execution on one node.)

6. "I will set the cache TTL to 0 to always get fresh data." (A 0 TTL means every read fans out to all shards. Read load explodes by 100x, and the read aggregator becomes the bottleneck.)

7. "I will use LWW with wall-clock timestamps because it is simpler than CRDTs." (Clock skew will silently drop increments. Use HLC or CRDT merge instead.)

8. "The counter is the source of truth." (Distributed counters are estimates. Event logs are the source of truth.)

9. "I will use a consensus algorithm (Raft) for every increment." (Raft commits add 50-200ms per write. This destroys the throughput advantage of distributed counters.)

10. "I do not need to worry about tombstone GC." (PN-Counter state grows by 1 MB per million decrements. Over 1 billion decrements, that is 1 GB of metadata per counter.)

## Practice Drills

Five timed prompts. Set a timer for each and practice speaking your answer aloud.

**Drill 1 (5 minutes): Design a like counter for a social platform with 1 million writes per second peak. Include the key numbers: shard count, per-shard throughput, cache TTL, read latency.**

**Drill 2 (5 minutes): A read aggregator is showing a counter value that is 5% lower than the event log total. Identify three possible causes and describe how to detect each one.**

**Drill 3 (10 minutes): Your Redis cluster has 64 shards. Shard 7's primary crashes. The failover promotes the replica, but 3 seconds of increments are lost. How do you recover the lost count? How do you prevent this in the future?**

**Drill 4 (10 minutes): A teammate proposes using a PN-Counter to track free seat inventory for a flight booking system. Identify why this is dangerous and propose an alternative architecture.**

**Drill 5 (15 minutes): Design a global view counter for a video platform that must handle 10 million views per minute across 5 AWS regions, with cross-region consistency within 5 seconds.**

Practice each drill until you can answer without pausing for more than 10 seconds.
