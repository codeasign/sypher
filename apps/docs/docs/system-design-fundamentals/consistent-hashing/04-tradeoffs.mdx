## The Core Tension

Consistent hashing trades perfect load distribution for minimal rebalancing cost when the cluster size changes.

Every design decision in a consistent hashing system is a negotiation between these two poles. More virtual nodes means better load distribution but larger ring arrays and slower topology changes. Higher replication factor means better durability but more storage overhead and write latency. Aggressive failure detection means faster recovery but more false positives. Understanding where your system sits on these axes is the difference between a ring that works and a ring that causes incidents.

## Decision Matrix

| Decision Point | Option A | Option B | Choose A when | Choose B when |
|---|---|---|---|---|
| Virtual node count | High (1,000+ per node) | Low (100 per node) | Nodes are fewer than 50, load balance is critical | Nodes are more than 50, topology changes must complete in under 5 seconds |
| Replication factor | 3 | 5 | Cache workloads where data loss means a cache miss (acceptable) | Durable storage where data loss is permanent and costly |
| Hash function | MurmurHash3 (fast) | SHA-256 (uniform) | Throughput above 200k lookups/sec is required | Adversarial key selection is a concern (a malicious tenant crafts keys that all hash to the same node) |
| Ring topology storage | Sorted array in memory | Skip list or balanced tree | Lookup throughput is the primary metric | Topology changes are frequent (every few seconds) |
| Failure detection | Short (3 heartbeats, 1s interval) | Long (15 heartbeats, 3s interval) | The business requires sub-5-second failover | Network jitter is common in the deployment environment |
| Routing location | Sidecar proxy per host | Library embedded in the application | Multiple services share the same ring topology, zero-downtime proxy upgrades are needed | Deployment simplicity is paramount, the team wants one less process to manage |
| Consistency model | Eventual (read replicas may lag) | Strong (quorum reads) | Read latency must stay under 5ms, stale data is acceptable for seconds | The application cannot tolerate stale reads, even for 10ms |
| Data transfer during rebalance | Parallel with rate limiting | Sequential | Rebalancing must complete in under 15 minutes | Network bandwidth is scarce, neighbor nodes cannot handle replication traffic |
| Ring topology discovery | Gossip protocol (decentralized) | Configuration service (centralized) | The system operates in an environment with unreliable network (air-gapped, edge) | Operational simplicity is preferred, configuration service already exists in the stack |
| Key ownership precision | Probabilistic (virtual nodes) | Deterministic (exact weight calculation) | Even small load imbalances (5 percent) cause p99 latency spikes | Slight imbalances are tolerable, simplicity is preferred |

## CAP Theorem Positioning

Consistent hashing is partition-tolerant by design. The ring itself does not mandate whether the system favors consistency or availability — that choice is determined by the replication strategy and read/write quorums configured on top of the ring.

### Partition Behavior

When a network partition splits a consistent hashing cluster, the ring topology on each side of the partition diverges. Nodes on side A remove the nodes on side B from their local ring. Nodes on side B do the same for side A.

- **In a CP configuration** (quorum read + quorum write): The smaller side of the partition cannot achieve quorum and rejects writes. A 5-node cluster split 3/2 means the side with 2 nodes cannot form a 3-node quorum and stops accepting writes. Data on the minority side is frozen until the partition heals and the minority nodes catch up.
- **In an AP configuration** (write one, read one): Both sides accept writes. When the partition heals, conflicting writes must be resolved through last-write-wins or CRDT merging. During the merge window, reads may return divergent values.

## Failure Modes

### Failure Mode 1: Cascading Node Failure

**What happens**: A single cache node fails from memory pressure. Its keys reassign to the next clockwise node. That node was already running at 70 percent capacity. Adding the failed node's 10 percent share pushes it to 80 percent. The 80 percent capacity triggers memory compaction, which spikes CPU to 95 percent. Request latency triples. The application retries on timeout, doubling the request rate. The node fails from the combined overload. Its keys move to the next node, which was already handling 80 percent. The cascade continues.

**Why it happens**: The ring topology has no per-node load shedding. Each node accepts all keys assigned to it, regardless of its current capacity.

**Blast radius**: 3 to 5 nodes over 60 seconds. In a 10-node cluster, a single node failure can trigger a chain reaction that takes down half the cluster.

**Detection**: Monitor the skew ratio metric. A rapid increase in skew ratio (above 1.5) indicates a cascade in progress. Alert when skew ratio exceeds 1.3 for more than 10 seconds.

**Recovery**: Three mechanisms:
1. **Per-node capacity limit**: Each node rejects new keys when above 80 percent capacity. Keys stay on the ring but the node returns a redirect to the next healthy successor.
2. **Load-aware routing**: The sidecar proxy maintains per-node latency percentiles and routes requests away from nodes showing elevated latency.
3. **Rapid provisioning**: Auto-scaling group provisions replacement nodes within 60 seconds. The new nodes register on the ring with a high weight, immediately pulling load from overloaded nodes.

### Failure Mode 2: Hot Spot from a Popular Key

**What happens**: A celebrity posts on a social platform. The celebrity's user profile is stored on one cache node. Millions of users view the profile in 60 seconds. The single cache node handling that key receives 50,000 requests per second while its peers handle 500. The node's CPU saturates at 100 percent. Latency for all keys on that node spikes from 2ms to 500ms.

**Why it happens**: Consistent hashing distributes keys evenly across nodes, but it does not distribute request frequency evenly. A single popular key concentrates all its traffic on one node.

**Blast radius**: One node. All keys on that node suffer elevated latency.

**Detection**: Monitor per-node request rate variance. A single node with request rate more than 5x the cluster average indicates a hot key.

**Recovery**: Three approaches:
1. **Key replication**: Replicate the hot key to multiple nodes using a different hash scheme: `hash(key + ":0")`, `hash(key + ":1")`, etc. Reads pick a random replica.
2. **Local cache**: The application caches the popular key on the client side with a short TTL. A single client caching a popular profile for 10 seconds reduces server-side requests by 90 percent.
3. **Read replicas**: Dedicated read-only replicas of the hot key, updated asynchronously, handle read traffic without affecting the primary node.

### Failure Mode 3: Rebalancing Storm

**What happens**: During a rebalance, the ring topology changes propagate to all sidecar proxies simultaneously. Every proxy recomputes its sorted ring array and starts routing requests according to the new topology. The nodes receiving new keys experience a sudden traffic surge as all proxies redirect requests simultaneously.

**Why it happens**: The configuration service publishes the topology change as a single atomic update. All proxies receive the notification within the same 5-50ms window and switch to the new topology simultaneously.

**Blast radius**: The newly added nodes experience a 5x-10x request surge in under 100ms. If the nodes are not pre-warmed, every request to a new key is a cache miss, cascading to the database.

**Detection**: Monitor the rebalancing rate metric. A rebalancing rate above 500 MB/s or a request rate surge above 5x on the new node is a rebalancing storm.

**Recovery**: Two mechanisms:
1. **Gradual topology propagation**: Proxies are updated in batches (10 percent per 10 seconds) rather than all at once.
2. **Pre-warming**: Before the topology update, the new node's neighbors stream the affected keys to it. The new node is fully loaded before any request routes to it.

### Failure Mode 4: Uneven Key Distribution

**What happens**: The hash function produces non-uniform output. A cluster with 10,000 virtual nodes shows a skew ratio of 1.45 — one node handles 45 percent more traffic than average. CPU utilization on that node averages 75 percent while other nodes average 52 percent. The node is the first to fail during traffic spikes.

**Why it happens**: The hash function has a distribution bias. CRC32, for instance, has known patterns that cause clustering in the hash space. Even MurmurHash3 can produce uneven distributions at certain virtual node counts.

**Blast radius**: One node. The overloaded node fails first during traffic surges, potentially triggering a cascade.

**Detection**: Skew ratio above 1.15 for more than 24 hours indicates a systematic hash function issue.

**Recovery**: Switch to a statistically tested hash function (MurmurHash3, CityHash, xxHash). Increase virtual node count to the recommended range for the cluster size.

### Failure Mode 5: Split-Brain with Inconsistent Ring Topology

**What happens**: Two sidecar proxies have different ring topologies at the same time. Proxy A routes a key to Node_X. Proxy B routes the same key to Node_Y. Both nodes receive writes for the same key. When topology eventually converges, Node_X and Node_Y have diverged — each holds a version of the key that the other does not know about.

**Why it happens**: The configuration service update did not atomically reach all proxies. A proxy crashed during the update and recovered with stale state. A network partition isolated one proxy from the configuration service.

**Blast radius**: Affected keys (the set of keys whose ring position changed in the topology update). In a 10-node cluster adding one node, approximately 9 percent of keys are affected.

**Detection**: Compare ring topology versions across proxies. Any proxy with a version older than 2 seconds should be flagged.

**Recovery**: Version the ring topology. Every topology update increments a monotonic version number. Proxies reject any request for a key whose topology version is older than the version held by the data node. The data node returns a redirect with the current topology version. The proxy refreshes its ring and retries.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Using Node Count as Hash Input

`bucket = hash(key) % len(nodes)` — This is modulo-N hashing, not consistent hashing. Every node count change remaps every key. This has no place in a production distributed system.

### Anti-Pattern 2: No Virtual Nodes

One hash per physical node produces severe load imbalance. In a 5-node cluster, the largest ring arc can be 35 percent of the ring while the smallest is 5 percent. The most-loaded node handles 7x the load of the least-loaded node. Always use virtual nodes.

### Anti-Pattern 3: Client-Side Ring Without Topology Watch

Each application instance builds the ring from a static configuration file. Adding a node requires a deploy of every application instance. Topology stale for minutes or hours after changes. Use a configuration service or gossip protocol for dynamic topology discovery.

### Anti-Pattern 4: Ignoring Node Heterogeneity

Every node gets the same number of virtual nodes regardless of capacity. A 128 GB RAM node and a 16 GB RAM node both get 1,000 virtual nodes. The large node is underutilized; the small node runs at 90 percent capacity. Weight virtual nodes by node capacity.

### Anti-Pattern 5: Synchronous Rebalancing

The application blocks requests during rebalancing. No keys are served until the rebalance completes. For a 182 GB rebalance at 1 Gbps, that is 24 minutes of downtime. Never block traffic during rebalancing.

### Anti-Pattern 6: Unlimited Replication Factor

Setting replication factor to N (the number of nodes) causes every node to store every key. This negates the benefit of partitioning — each node stores the entire dataset. The system becomes a replicated monolith rather than a partitioned database.

## When NOT to Use This

Consistent hashing is not the right solution for every distributed system problem. Consider alternatives in these situations:

- **Single-node deployments**: If your dataset fits on one server, consistent hashing adds complexity with zero benefit. A single PostgreSQL or Redis instance is simpler and faster.
- **Fixed-size clusters**: If the number of nodes never changes (embedded systems, dedicated hardware), modulo-N hashing with a carefully chosen N is simpler and has better load distribution properties.
- **Sequential key access**: If the workload requires scanning keys in order (time-series analytics, log processing), range-based partitioning preserves locality. Consistent hashing scatters sequential keys across nodes.
- **Small clusters (under 3 nodes)**: With 2-3 nodes, the benefit of consistent hashing over modulo is minimal. The overhead of maintaining ring topology outweighs the rebalancing savings.
- **Systems requiring perfect balance**: Consistent hashing is probabilistic. Virtual nodes improve distribution but cannot guarantee perfect balance. Jump consistent hashing or rendezvous hashing provides better distribution guarantees for fixed-size clusters.
