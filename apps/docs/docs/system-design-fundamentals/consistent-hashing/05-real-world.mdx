## Company Case Studies

### Case Study 1: Discord — From MongoDB to Cassandra to ScyllaDB

Discord started with a single MongoDB replica set for message storage. As the platform grew to 100 million monthly active users, MongoDB could not keep up. Discord migrated to Cassandra in 2017, using consistent hashing to distribute message data across nodes.

The original Cassandra ring used 256 virtual nodes per physical node. Each message was assigned a partition key of `(channel_id, message_id)`. Consistent hashing ensured that all messages in a single channel landed on the same node, enabling efficient chronological queries within a channel.

In 2022, Discord migrated from Cassandra to ScyllaDB (a C++ rewrite of Cassandra). ScyllaDB maintains the same consistent hashing ring semantics as Cassandra but achieves 10x throughput per node. Discord's engineering blog reported that their ScyllaDB cluster handles 1 billion messages per day across 177 nodes. Each message write touches exactly one node (plus replicas), with no cross-node coordination.

The key insight from Discord's postmortems: the ring topology must be versioned and monitored. During their Cassandra-to-ScyllaDB migration, Discord ran both systems in parallel for months, using the ring version number to ensure read-after-write consistency across the two systems. Reads went to whichever system had the latest version of a message, determined by comparing ring topology version timestamps.

**Public reference**: Discord Engineering Blog, "How Discord Stores Trillions of Messages" (2022).

### Case Study 2: Netflix — Ephemeral Cache with Consistent Hashing

Netflix's caching layer uses EVCache (Ephemeral Volatile Cache), which employs consistent hashing to distribute cached data across memcached nodes. Each cache operation uses the request URL or a computed cache key as the hash input.

Netflix published that their consistent hashing implementation uses 1,000 virtual nodes per physical node and a replication factor of 2. The ring topology is stored in ZooKeeper. Each EVCache client (an embedded Java library in every Netflix service) watches ZooKeeper for topology changes and updates its local ring within 15 milliseconds.

A single Netflix cache cluster handles 2 million requests per second across 50 nodes. The p99 read latency is under 3 milliseconds, including the consistent hash lookup, network round trip, and data fetch. The key operational insight from Netflix's experience: virtual nodes must be weighted by node capacity. When Netflix introduced larger instance types into existing clusters, unweighted virtual nodes caused the larger instances to be underutilized because they had the same number of vNodes as smaller instances.

Netflix also developed a technique called "fast property loading" for cache warmup. When a new node joins the ring, its neighbors stream the affected keys to it before the topology update propagates. This pre-warming eliminates the cache-miss spike that normally follows a topology change.

**Public reference**: Netflix Tech Blog, "EVCache: A Distributed In-Memory Data Store" (2013); re:Invent 2018 talk "Caching at Netflix."

### Case Study 3: Cloudflare — Consistent Hashing for Load Balancing

Cloudflare uses consistent hashing to distribute traffic across origin servers in their load balancing product. Each DNS resolution request is assigned to an origin server based on the consistent hash of the client IP address and the service name.

Cloudflare's implementation is notable because it operates at an extreme scale: 30 million HTTP requests per second across 300+ data centers worldwide. Their consistent hashing ring uses 512 virtual nodes per physical origin server. The ring topology is distributed via Cloudflare's internal KV store (Quicksilver), which propagates changes to all data centers within 5 seconds.

Cloudflare engineers reported that their consistent hashing design prevents connection churn during origin server scaling. When a new origin server joins the pool, only 1/512 of existing TCP connections need to be re-established, compared to 100 percent of connections under modulo-N hashing. This is critical because re-establishing TCP connections at Cloudflare's scale means millions of SYN packets arriving at the origin simultaneously, which can overwhelm the TCP SYN backlog.

The most valuable lesson from Cloudflare's implementation: use a hash function that works well with the Anycast routing layer. Cloudflare tested Jenkins hash, MurmurHash3, and xxHash before settling on xxHash for its combination of speed (40 ns per 64 bytes) and distribution uniformity.

**Public reference**: Cloudflare Engineering Blog, "Consistent Hashing for Load Balancing" (2020); Cloudflare Architecture blog series.

### Case Study 4: Uber — Ringpop for Sharded Service Communication

Uber built Ringpop, an open-source consistent hashing library, to handle service-to-service communication in their microservice architecture. Ringpop uses the SWIM gossip protocol for failure detection and membership management, combined with a consistent hash ring for request routing.

Ringpop's key innovation is treating the consistent hash ring as a self-organizing overlay network. Each service instance joins a ring, and Ringpop handles both health monitoring and request routing automatically. When a service instance fails, Ringpop detects the failure within 5 seconds using SWIM gossip and updates the ring topology across all nodes within 10 seconds.

Uber published that Ringpop handles 100,000 nodes across their fleet. Each ring uses 100 virtual nodes per physical node, producing a total of 10 million ring entries per cluster. The sorted ring array is stored in shared memory, enabling multiple processes on the same host to share the ring without duplication.

The lesson from Uber's use of Ringpop: consistent hashing is not just for data storage — it works equally well for routing requests between microservices. Any pool of interchangeable service instances can use a consistent hash ring to distribute requests with minimal rebalancing.

**Public reference**: Uber Engineering Blog, "Ringpop: A Scalable, Fault-Tolerant Application Layer" (2015).

### Case Study 5: Amazon DynamoDB

Amazon's DynamoDB uses a variant of consistent hashing as the foundation for its distributed storage. Each table is partitioned across multiple storage nodes using consistent hashing. The partition key determines the hash ring position.

DynamoDB adds two innovations on top of the basic ring: **split** and **merge**. When a partition grows too large (over 10 GB), DynamoDB splits it into two partitions by inserting a new node at the midpoint of the partition's hash range. When partitions shrink, DynamoDB merges adjacent partitions. The ring topology evolves dynamically without any manual intervention.

DynamoDB's consistent hashing ring uses a replication factor of 3 within an availability zone and cross-AZ replication for disaster recovery. Each write is replicated to 3 nodes in the ring before acknowledgement. The NeXT-gen Dynamo paper (2022) reported that this architecture handles 10 million requests per second for a single table.

**Public reference**: Amazon, "Dynamo: Amazon's Highly Available Key-value Store" (SOSP 2007); "Next-Gen Dynamo" (AWS re:Invent 2022).

## Open Source Implementations

### Cassandra

Cassandra uses a consistent hash ring with virtual nodes (vnodes) as its core data distribution mechanism. Each Cassandra node owns a range of the ring. The ring is modified through the gossip protocol, which propagates topology changes across the cluster.

**Trade-off**: Cassandra uses 256 virtual nodes per physical node by default. This provides good distribution for clusters under 100 nodes but causes ring propagation delays in larger clusters. Cassandra 4.0 introduced "transient replication" to reduce the storage overhead of virtual nodes.

**Best for**: Write-heavy workloads requiring linear scalability across commodity hardware.

### Riak

Riak uses a consistent hash ring with a fixed partition count of 64 partitions per physical node. Riak's implementation is notable for its use of hinted handoff — when a write target node is unavailable, another node temporarily accepts the write and forwards it when the target recovers.

**Trade-off**: Riak's fixed partition count avoids the complexity of virtual node redistribution but requires manual resizing when adding nodes. Each node addition triggers a full data redistribution across all partitions.

**Best for**: Systems requiring read-repair and hinted handoff for maximum write availability.

### ScyllaDB

ScyllaDB is a C++ rewrite of Cassandra that maintains API compatibility while redesigning the consistent hashing layer. ScyllaDB uses a shard-aware consistent hash ring: each CPU core on each node acts as an independent shard on the ring.

**Trade-off**: ScyllaDB's shard-aware ring eliminates cross-core coordination within a node, achieving 10x throughput per node compared to Cassandra. However, the shard-level mapping adds complexity to topology management — a 10-node cluster with 32 cores each has 320 ring entries.

**Best for**: High-throughput workloads where Cassandra's JVM garbage collection pauses cause latency spikes.

## Cloud Provider Implementations

### AWS — Amazon ElastiCache

Amazon ElastiCache for Redis supports cluster mode, which uses consistent hashing to distribute keys across shards. The hash ring uses a slot-based scheme (16,384 hash slots, standard Redis cluster) rather than virtual nodes. Each shard owns a range of hash slots.

**Cost**: ElastiCache cluster mode charges per shard. A 10-shard cluster costs 10x the base node price. Consistent hashing is built in — no additional implementation cost.

**Ops**: AWS manages the ring topology. Adding a shard triggers an automatic rebalancing of hash slots. The rebalance runs in the background with configurable slot migration concurrency.

**Trade-off**: No fine-grained control over virtual node count or hash function. AWS uses Redis's built-in CRC16 hash over hash slots — no alternative hash function is available.

### GCP — Memorystore

Google Cloud Memorystore for Redis uses the same hash slot approach as AWS ElastiCache. Memorystore supports automatic rebalancing when nodes are added or removed.

**Cost**: Similar to ElastiCache — per-shard pricing. GCP offers committed-use discounts for 1-year and 3-year terms.

**Trade-off**: GCP does not support weighted nodes in the consistent hash ring. Every node in a Memorystore cluster gets the same number of hash slots, making heterogeneous clusters impractical.

### Azure — Azure Cache for Redis

Azure Cache for Redis Enterprise uses Redis Enterprise's clustering technology, which includes consistent hashing with proxy-based routing. Azure's implementation uses a proxy layer that abstracts the ring topology from clients.

**Cost**: Azure charges a premium for Enterprise tier, which includes the proxy-based consistent hashing. The Premium tier uses Redis Cluster's standard hash slot scheme at a lower price point.

**Trade-off**: The proxy layer adds 0.5-1ms of additional latency per request. For latency-sensitive workloads under 5ms, the proxy overhead is significant.

## Lessons Learned

**Lesson 1**: Test the hash function before production. A non-uniform hash function creates hot spots that no amount of virtual nodes can fix. For any hash function, compute the distribution of 1 million random keys across 1,000 virtual buckets and verify that the standard deviation is under 1 percent.

**Lesson 2**: Pre-warm new nodes before topology propagation. Netflix and Discord both learned that adding a cold node to the ring causes a cache-miss cascade. Stream keys from neighbors before updating the topology.

**Lesson 3**: Version the ring topology. Discord's parallel-run migration and Cloudflare's multi-datacenter deployment both rely on topology versioning to prevent split-brain routing. A version counter in the ring metadata is cheap insurance.

**Lesson 4**: Weight virtual nodes by node capacity. Netflix's heterogeneous clusters could not achieve balanced load until they switched from equal vNode counts to capacity-weighted vNode counts. The rule: `vNodes = base_count * (node_capacity / avg_capacity)`.

**Lesson 5**: Plan for cascading failures. Uber Ringpop's SWIM-based failure detection includes a "suspicion" phase before declaring a node dead — the system waits for confirmation from 3 independent peers. This reduces false positives during network jitter by 90 percent compared to single-source detection.
