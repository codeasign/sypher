## Typical Questions

### Level 1: Concepts and Knowledge (Entry-Level to Mid-Level)

1. "Explain the difference between a local transaction and a distributed transaction."
2. "Describe the Two-Phase Commit protocol. What are its strengths and weaknesses?"
3. "What is the blocking problem in 2PC?"
4. "How does Three-Phase Commit differ from Two-Phase Commit?"
5. "What is a saga pattern? When would you use it instead of 2PC?"
6. "Explain the TCC pattern."

### Level 2: Design and Tradeoffs (Mid-Level to Senior)

1. "Design a distributed transaction system for an e-commerce checkout flow."
2. "How would you handle cross-service consistency for a ride-hailing platform?"
3. "Design a system that ensures exactly-once payment processing across multiple services."
4. "Your team is migrating from a monolith to microservices. How do you handle distributed transactions during the migration?"
5. "How would you design a multi-region booking system with strong consistency requirements?"

### Level 3: Advanced and Open-Ended (Senior to Staff)

1. "Compare and contrast 2PC, 3PC, Saga, and TCC. Under what conditions would each break?"
2. "Design a globally distributed transaction system that can handle 10,000 transactions per second across three continents."
3. "What changes would you make to the Saga pattern to provide stronger consistency guarantees?"
4. "How would you detect and resolve inconsistencies in a system that uses eventual consistency?"
5. "Design a reconciliation system that continuously verifies data consistency across 50 microservices."

## The Interview Framework: ACID-R

Use the ACID-R framework to structure your answers to distributed transaction design questions:

**A -- Analyze the requirements**: Determine the consistency, availability, latency, and throughput requirements. Ask clarifying questions about the business scenario.

**C -- Choose the protocol**: Based on the requirements, select the appropriate protocol (2PC, 3PC, Saga, TCC, or no distributed transaction at all).

**I -- Identify failure modes**: Walk through the failure scenarios for your chosen protocol. Show the interviewer that you understand what can go wrong.

**D -- Design the recovery**: Describe how the system recovers from each failure mode. Discuss the transaction log, recovery manager, and retry mechanisms.

**R -- Review tradeoffs**: Summarize the tradeoffs you made. Acknowledge what the system sacrifices and why the sacrifice is acceptable for this use case.

### Example: Applying ACID-R to an E-Commerce Checkout

**Analyze**: "The checkout requires deducting inventory, charging the payment, and creating a shipment. We need strong consistency for money and inventory -- we must not lose money or oversell. The checkout takes under 30 seconds. The availability requirement is 99.9%."

**Choose**: "I will use TCC. The Try phase reserves inventory and authorizes the payment. The Confirm phase deducts the inventory and captures the payment. The Cancel phase releases the inventory and voids the authorization."

**Identify**: "Failure 1: The payment authorization fails during Try. Cancel is called on inventory. Failure 2: The coordinator crashes after Confirm on inventory but before Confirm on payment. On recovery, the coordinator retries the Confirm on payment. Both operations are idempotent."

**Design**: "The coordinator writes the transaction state to a replicated log. On recovery, it reads the log and retries any pending Confirm or Cancel operations. Each participant exposes idempotent Try, Confirm, and Cancel endpoints."

**Review**: "TCC provides stronger consistency than a pure Saga because resources are reserved during Try. The tradeoff is that reserved resources are unavailable to other transactions until the reservation expires or is cancelled. For our checkout flow with a 30-second reservation window, this is acceptable."

## 15+ Phrases to Use in Interviews

Use these phrases to demonstrate depth in your answers:

1. "The blocking problem occurs when a participant that has voted YES waits indefinitely for the coordinator decision after a coordinator crash."

2. "The transaction log is the source of truth for recovery. The coordinator writes every state transition to the log before sending any protocol message."

3. "Idempotency is critical. Participants must handle duplicate COMMIT, ABORT, or compensation messages safely."

4. "Heuristic resolution is a last resort. An operator manually commits or aborts a stuck transaction, risking inconsistency."

5. "Sagas provide eventual consistency through compensating transactions. There is a window where the intermediate state is visible."

6. "The prepare phase is where the participant makes its changes durable and acquires all necessary locks."

7. "Group commit batches multiple commit decisions into a single log flush, improving throughput at the cost of slightly higher latency."

8. "Optimistic concurrency avoids locks by detecting conflicts at commit time and aborting one of the conflicting transactions."

9. "Presumed abort means the coordinator treats any transaction not found in the log as aborted, reducing log volume."

10. "The coordinator is a point of failure. Deploy it as a clustered service with synchronous log replication."

11. "CAP theorem positions 2PC as CP. It provides consistency but sacrifices availability during partitions."

12. "Compensating transactions are business operations, not technical rollbacks. They are visible to users and must be designed as part of the business logic."

13. "Read-only participants can skip the commit phase because they hold no locks and have no changes to commit."

14. "Transaction timeout prevents indefinitely held locks. Set it to 10-30 seconds for most use cases."

15. "Reconciliation jobs periodically verify that all participants are in the expected state and fix inconsistencies."

16. "Idempotency keys allow participants to detect and ignore duplicate requests."

17. "The recovery manager reads the log on startup and resolves all transactions in the PREPARED state."

## 8+ Red Flags Interviewers Look For

| Red Flag | Why It Is a Problem |
|---|---|
| "I would use 2PC for everything." | Shows no understanding of tradeoffs. 2PC has significant availability and latency costs. |
| "Saga guarantees strong consistency." | Shows misunderstanding of saga fundamentals. Saga provides eventual consistency at best. |
| "We do not need a transaction log." | Shows no understanding of recovery. Without a log, coordinator crashes leave transactions in unknown state. |
| "The coordinator cannot fail." | Shows no understanding of distributed systems fundamentals. Every component can fail. |
| "2PC has no blocking problem." | Shows ignorance of the most cited weakness of 2PC. |
| "Compensating transactions are invisible." | Shows misunderstanding of sagas. Compensations are business operations that users see. |
| "We will coordinate transactions across 50 services with 2PC." | Shows no understanding of practical limits. 2PC with many participants has poor performance and reliability. |
| "Distributed transactions always work." | Shows no understanding of failure modes. All distributed transaction protocols have failure scenarios. |

## Sample Answer: Design a Distributed Transaction System for E-Commerce

### The Question

"Design a checkout system that handles inventory deduction, payment processing, and shipping label creation atomically across three microservices."

### The Answer (ACID-R Framework)

**Analyze**: "The checkout flow involves three services: Inventory, Payment, and Shipping. The key requirement is atomicity -- we must not charge the customer without deducting inventory, and we must not deduct inventory without creating a shipment label. The transaction must complete in under 30 seconds. The system must handle 1,000 checkouts per second at peak. We need 99.9% availability."

**Choose**: "I will use TCC. Here is why:
- 2PC would hold locks across all three databases for the duration of the checkout. If the shipping service is slow, inventory and payment locks block other customers.
- Saga would work, but there is a window between steps where the state is inconsistent. If the payment is charged but inventory deduction fails, the customer is overcharged.
- TCC reserves resources during Try without committing. If Confirm fails, the reservation expires automatically. This provides medium consistency with good availability."

**Design**: "The architecture has three components:

1. **Coordinator Service**: A clustered service that manages the TCC lifecycle. Each coordinator instance is stateless -- transaction state is stored in a replicated transaction log.

2. **Participant Adapters**: Each service exposes Try, Confirm, and Cancel endpoints. The Try endpoint reserves resources. The Confirm endpoint makes the reservation permanent. The Cancel endpoint releases the reservation.

3. **Transaction Log**: A replicated log stored in DynamoDB or Apache Kafka. The log records every state transition: Try sent, Try completed, Confirm sent, Confirm completed.

The flow is:

1. Client calls createOrder on the coordinator.
2. Coordinator calls Try on Inventory to reserve stock.
3. Coordinator calls Try on Payment to authorize the charge.
4. Coordinator calls Try on Shipping to reserve a shipping slot.
5. If all Try calls succeed, coordinator calls Confirm on all three.
6. If any Try fails, coordinator calls Cancel on any that succeeded."

**Failure Modes**: "Consider these failure scenarios:

- Scenario A: Inventory Try succeeds. Payment Try fails. Coordinator calls Cancel on Inventory. The stock reservation is released.
- Scenario B: All Try succeed. Coordinator crashes before sending Confirm. On recovery, the coordinator reads the log and resumes sending Confirm.
- Scenario C: Confirm on Payment succeeds. Confirm on Inventory fails. The coordinator retries Confirm on Inventory. Inventory must handle duplicate Confirm safely."

**Recovery**: "The recovery manager reads the transaction log on startup. For each unresolved transaction:

- If Try was sent but not completed: Send Cancel to all participants that completed Try.
- If Confirm was started but not completed: Retry Confirm on all participants.
- If any participant fails repeatedly: Move the transaction to a dead letter queue for manual investigation.

Each endpoint is idempotent. The Try endpoint uses a reservation ID. If a duplicate Try arrives, the participant returns the existing reservation. The Confirm and Cancel endpoints use the same reservation ID for deduplication."

**Review**: "TCC provides a good balance of consistency and availability for this use case. The tradeoff is that reserved resources are temporarily unavailable to other customers. With a 30-second reservation window, this is acceptable. The system sacrifices the strong consistency of 2PC but gains better availability and avoids long-held locks."

## Sample Answer: Design a Distributed Transaction System for Ride-Hailing

### The Question

"Design a system that ensures atomicity for the ride lifecycle: driver assigned, rider picked up, trip completed, payment charged, driver earnings credited. The system must handle partial failures without losing money or creating inconsistent state."

### The Answer (ACID-R Framework)

**Analyze**: "The ride lifecycle has five steps: driver assignment, pickup confirmation, trip completion, payment processing, and earnings credit. Each step is a separate microservice with its own database. The critical consistency requirement is that a payment must never be charged without the trip being completed, and driver earnings must never be credited without the payment being collected. The ride lifecycle takes 10-30 minutes, so holding locks for the full duration is not feasible. The system processes 50,000 rides per day with a peak of 200 concurrent rides."

**Choose**: "I will use an orchestration saga with compensating transactions. Here is why:

- 2PC is not feasible for a 30-minute transaction. Holding locks for 30 minutes would block all other operations on the affected data.
- TCC would work if resources could be reserved for 30 minutes, but a driver cannot be locked into a ride for 30 minutes without confirmation.
- Saga is the natural fit because it models the ride as a sequence of business steps, each with a local transaction and a compensation.

The orchestrator manages the saga state and executes compensations in reverse order if any step fails."

**Design**: "The architecture has three components:

1. **Saga Orchestrator**: A state machine that tracks each ride through its lifecycle. The orchestrator is a clustered service with state stored in a replicated database.

2. **Saga Log**: A durable log of every saga state transition. The log is stored in a replicated database (DynamoDB or PostgreSQL with streaming replication).

3. **Participant Services**: Each service exposes a forward operation and a compensating operation:

| Service | Forward Op | Compensating Op |
|---|---|---|
| Dispatch | Dispatch driver | Cancel dispatch |
| Rider | Record pickup | Record cancellation |
| Trip | Complete trip | Mark as disputed |
| Payment | Charge rider | Refund rider |
| Earnings | Credit driver | Debit driver |

The saga flow is:

1. Orchestrator sends "Dispatch Driver" to Dispatch service.
2. Dispatch service dispatches the nearest driver and returns driver assignment.
3. Orchestrator records the state change in the saga log.
4. Orchestrator waits for the "Rider Picked Up" event from the rider app.
5. Rider service publishes the pickup event.
6. Orchestrator records the state change in the saga log.
7. Orchestrator waits for the "Trip Completed" event.
8. Trip service publishes the completion event.
9. Orchestrator sends "Charge Rider" to Payment service.
10. Payment service charges the rider. If successful, orchestrator sends "Credit Driver" to Earnings service.

If any step fails, the orchestrator executes compensations in reverse order."

**Failure Modes**: "Consider these scenarios:

- Scenario A: Driver dispatched but rider cancels. The orchestrator sends "Cancel Dispatch" to the Dispatch service.
- Scenario B: Trip completed but payment fails. The orchestrator calls "Mark as Disputed" on the Trip service and does not credit the driver.
- Scenario C: Payment succeeds but earnings credit fails. The orchestrator retries the earnings credit. If it continues to fail, the ride is marked as "Paid, Earnings Pending" and a reconciliation job fixes it within 5 minutes."

**Review**: "The saga pattern provides good availability: a failure in one step does not block other rides. The tradeoff is eventual consistency -- there is a window between "Trip Completed" and "Payment Charged" where the rider has completed the trip but has not been charged. For a ride-hailing platform, this is acceptable because the rider's identity is known and the charge can be retried."

## Additional Sample Answer: Database Sharding and Distributed Transactions

### The Question

"Your database is growing at 50 GB per month. You need to shard it across 10 database instances. Some queries touch multiple shards. How do you handle distributed transactions across shards?"

### The Answer

"The best strategy is to design the shard key so that transactions that need atomicity are confined to a single shard. For an e-commerce platform, this means sharding by customer ID. All data for a single customer -- their orders, payments, and shipping addresses -- are on the same shard.

For queries that must touch multiple shards, I use a two-tier approach:

1. **Single-shard transactions use local database transactions**: Since all data for one customer is on one shard, most operations use fast local transactions.

2. **Cross-shard operations use saga pattern**: Operations that span shards are decomposed into single-shard local transactions with compensating actions. For example, a report that aggregates data across all shards does not need a distributed transaction -- it can tolerate reading slightly stale data from each shard.

For the rare case where a true cross-shard transaction is required (such as transferring funds between customers on different shards), I use TCC:

1. Try: Reserve the funds on shard A, reserve the deposit on shard B.
2. Confirm: Deduct the funds on shard A, add the funds on shard B.
3. Cancel: Release the reservation on both shards.

This approach handles 95% of transactions with local transactions, 4.9% with sagas, and 0.1% with TCC."

## Deep Dive: ACID-R Framework Applied to Three Scenarios

### Scenario 1: Social Media Post Creation

**Analyze**: Create a post, update follower feeds, send notifications. Notifications can tolerate 1-minute delay. Inconsistent feed state for a few seconds is acceptable.

**Choose**: Choreography-based saga. The post creation service publishes a "PostCreated" event. Feed services consume the event and update their local feeds independently. Notification service consumes the event and sends push notifications.

**Tradeoff**: Eventual consistency across feeds and notifications. Some followers may see the post on their feed before the notification arrives, or vice versa. This is acceptable for social media.

### Scenario 2: Medical Records Update

**Analyze**: Update patient diagnosis, update treatment plan, update prescription, notify pharmacy. All updates must be consistent -- a diagnosis must match the treatment plan. Non-negotiable strong consistency.

**Choose**: 2PC with a clustered coordinator. All four databases join the same distributed transaction. If any update fails, all are rolled back.

**Tradeoff**: Lower availability during coordinator failover. The patient record is locked during the transaction. Acceptable for medical systems where consistency is legally required.

### Scenario 3: Cloud Resource Provisioning

**Analyze**: Provision a virtual machine, allocate an IP address, attach a storage volume, configure a firewall. Each resource is managed by a separate service. Provisioning takes 30 seconds.

**Choose**: TCC. Each resource service supports Try (reserve), Confirm (provision), and Cancel (release). The Try phase reserves all resources. The Confirm phase provisions them all. The Cancel phase releases any reserved but unprovisioned resources.

**Tradeoff**: Resources are reserved during the Try phase. If the Try takes 10 seconds, resources are held for 10 seconds. If the reservation expires, the TCC degrades gracefully to a saga with manual reconciliation.

## Glossary of Interview Terms

| Term | Definition |
|---|---|
| Atomic commitment | The property that all participants in a distributed transaction reach the same final state |
| Blocking | The inability of a participant to make progress while waiting for the coordinator decision |
| Commit protocol | A set of rules that participants follow to reach agreement on transaction outcome |
| Compensation | A business operation that semantically undoes a previous operation |
| Coordinator | The entity that manages the distributed transaction protocol |
| Deadlock | A state where two or more transactions are waiting for locks held by each other |
| Heuristic resolution | A manual decision to commit or abort a stuck transaction |
| Idempotent | An operation that produces the same result when executed multiple times |
| Indoubt transaction | A transaction that has prepared but not received a decision |
| Linearizability | The strongest consistency model, where operations appear to execute atomically |
| Local transaction | A transaction that involves a single database |
| Participant | A database or resource manager that participates in a distributed transaction |
| Paxos | A consensus algorithm used for log replication |
| Prepare phase | The first phase of 2PC where participants make changes durable and vote |
| Raft | A consensus algorithm used for log replication |
| Recovery | The process of resolving transactions after a crash |
| Saga | A pattern that decomposes a distributed transaction into local transactions with compensations |
| TCC | Try-Confirm-Cancel pattern for distributed transactions |
| Transaction log | A durable record of all transaction state transitions |
| TrueTime | Google's globally synchronized clock infrastructure |
| Two-phase commit | The foundational protocol for distributed transaction atomicity |
| Write-ahead logging | The practice of writing log records before applying changes |

## Mock Interview: Distributed Transaction for Payment Processing

### Interviewer Prompt

"Your team is building a payment processing system that handles 1,000 transactions per second. The system debits one account and credits another. Both accounts may be on different database shards. Design the solution."

### Candidate Response (Good)

"First, I need to understand the requirements. Can you confirm the consistency requirements -- must both the debit and credit happen atomically, or can we accept eventual consistency?"

Interviewer: "Both must happen atomically. Money cannot be lost."

"Understood. With atomicity required and accounts on different shards, we need a distributed transaction. I see two viable approaches:

Approach 1: Use 2PC with a coordinator. The debit shard and credit shard both prepare. If both prepare successfully, the coordinator commits. If either fails, both abort. This provides strong consistency but has two problems: it holds locks on both accounts during the prepare phase, and the coordinator becomes a single point of failure.

Approach 2: Use TCC. The debit shard reserves the funds (Try). The credit shard reserves the deposit (Try). If both succeed, both confirm. If either fails, both cancel. This reduces lock duration because the reservation can be short-lived.

I recommend Approach 2 for two reasons. First, the lock duration is shorter -- only the Try phase holds locks, not the full transaction. Second, TCC handles the failure case more gracefully -- if the credit shard is slow, the debit reservation can be released automatically after a timeout.

The architecture would include a coordinator cluster with a replicated transaction log. Each shard exposes Try, Confirm, and Cancel endpoints. The Try endpoint checks the account balance and reserves the amount. The Confirm endpoint deducts the reserved amount. The Cancel endpoint releases the reservation.

For failure handling, if the coordinator crashes after Try but before Confirm, the recovery manager reads the log and calls either Confirm or Cancel based on the transaction state. If the Confirm on one shard succeeds but the other fails, the coordinator retries the failed Confirm. After 5 retries, the transaction is moved to a dead letter queue.

The key tradeoff is that TCC provides medium consistency rather than strong consistency. There is a window between Try and Confirm where the funds are reserved but not yet deducted. If the coordinator crashes permanently during this window, the funds remain reserved until the reservation expires. A reconciliation job can detect and resolve this."

### Interviewer Feedback

Strengths:
- Asked clarifying questions about requirements.
- Identified two approaches and compared them.
- Chose a specific approach with clear rationale.
- Described architecture, failure modes, and recovery.
- Acknowledged tradeoffs.

Improvements:
- Could have quantified the availability impact.
- Could have discussed the reconciliation job in more detail.
- Could have mentioned the CAP theorem positioning.

## Mock Interview: Failure Analysis

### Interviewer Prompt

"Your production system uses 2PC for order processing. The operations team reports that orders are getting stuck in 'pending' state. The coordinator logs show that participants voted YES but the transaction was never committed. What do you investigate?"

### Candidate Response (Good)

"I would investigate in this order:

First, check the coordinator health. Is the coordinator process running? If it crashed, the recovery manager should have read the log and resumed the commit. If the recovery manager did not run, we need to restart it.

Second, check the coordinator transaction log for the stuck transactions. The log should show a COMMIT_DECISION record followed by COMMIT_SENT records. If the COMMIT_DECISION exists but COMMIT_SENT does not, the coordinator crashed after deciding but before sending. The recovery manager should have sent the COMMIT messages on restart.

Third, check the participants. Are all three participants that voted YES still healthy? If a participant crashed after voting YES, the coordinator cannot deliver the COMMIT. The coordinator should be retrying the delivery. If the participant is still down after 5 retries, the transaction should be in the dead letter queue for manual resolution.

Fourth, check the network. Is there a network partition between the coordinator and one or more participants? If so, the coordinator cannot reach the participant to deliver the COMMIT. The coordinator retries, but the retries also fail.

Fifth, check the timeout settings. If the commit_timeout is set to a very long value (hours), the coordinator may be waiting for a participant that is not going to respond. We should have a reasonable timeout (30-60 seconds) and a retry limit.

The most common cause is a participant crash. The coordinator is retrying the COMMIT delivery. The participant needs to be restarted. Once it starts, it receives the COMMIT from the coordinator and applies the changes."

### Interviewer Feedback

Strengths:
- Systematic investigation approach.
- Covered coordinator, participants, network, and configuration.
- Identified the most likely cause.

Improvements:
- Could have mentioned checking the transaction log consistency.
- Could have described how to resolve the stuck transactions without restarting the participant.

## Quick Reference: Interview Answer Structure

Use the following template for any distributed transaction design question:

```
Step 1: Requirements (30 seconds)
- "Let me confirm the requirements: we need [consistency/availability/latency]..."
- "The transaction involves [N] services..."
- "The system processes [X] transactions per second..."

Step 2: Protocol Selection (30 seconds)
- "For this scenario, I recommend [2PC/3PC/Saga/TCC] because..."
- "The alternatives [are worse because]..."

Step 3: Architecture (60 seconds)
- "The architecture has three components: a [coordinator], [participants], and a [transaction log]..."
- "The flow is: [describe the protocol steps]..."

Step 4: Failure Modes (60 seconds)
- "Let me walk through the failure scenarios: [coordinator crash], [participant crash], [network partition]..."
- "For each failure, [recovery strategy]..."

Step 5: Tradeoffs (30 seconds)
- "The tradeoffs are: [what we gain] vs [what we sacrifice]..."
- "For this use case, the tradeoff is acceptable because..."
```

Practice this structure until you can deliver it in under 4 minutes without notes. The interviewers are evaluating your ability to structure the answer, not just your knowledge of protocols.

## Common Follow-Up Questions and Responses

After you give your initial answer, interviewers often probe deeper with follow-up questions. Here are common ones and how to respond:

**Q1: "What if a participant is slow to respond?"**

A: "I would set a prepare timeout, typically 10-30 seconds. If the participant does not respond within the timeout, the coordinator marks the participant as TIMEOUT and aborts the transaction. The application can retry the transaction. For the TCC approach, the Try phase has a shorter timeout because it only needs to reserve resources, not make changes persistent."

**Q2: "How do you handle the case where the coordinator is the bottleneck?"**

A: "The coordinator is stateless -- all state is in the transaction log. I can scale the coordinator horizontally by adding more instances behind a load balancer. Each transaction is assigned to one coordinator instance. The transaction log is the shared state, and it must be scalable too. I would use a partitioned log (Kafka) or a scalable database (DynamoDB) for the log."

**Q3: "What monitoring metrics would you track?"**

A: "I would track five key metrics: (1) active transactions count, to detect build-up; (2) prepare latency, to detect slow participants; (3) retry rate, to detect participant failures; (4) unresolved transactions, to detect stuck transactions; and (5) coordinator CPU and memory, to detect capacity issues. Each metric has a warning and critical threshold with corresponding alerts."

**Q4: "How do you test distributed transactions?"**

A: "I use three layers of testing. Unit tests verify each participant endpoint (prepare, commit, abort) is idempotent and produces correct responses. Integration tests verify the coordinator protocol with all participants in a test environment, including failure scenarios. Chaos tests inject failures (network latency, packet loss, process kills) and verify that the system recovers correctly without data loss."

**Q5: "What if you cannot modify the participant services to support 2PC?"**

A: "If the participants are legacy systems that cannot be modified, I cannot use 2PC or TCC. I would use an event-driven saga instead. The coordinator publishes events to a message queue. Each participant listens for relevant events and executes its local transaction. Compensating events are published if a step fails. This approach does not require modifying the participants beyond adding event listeners."
