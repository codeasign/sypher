# Distributed Transactions -- Tradeoffs and Decision Making

## The Core Tension: Consistency vs. Availability

Distributed transactions exist at the center of the fundamental tension in distributed systems: you cannot simultaneously achieve strong consistency and high availability when network partitions occur.

This is the CAP theorem in action. Strong consistency requires all participants to agree on whether to commit or abort before proceeding. Agreement requires communication. When the network is partitioned, communication fails, and the coordinator must choose: block until the partition heals (sacrificing availability) or proceed without full agreement (sacrificing consistency).

The choice is not academic. Every production system navigates this tension through the protocol it selects and the timeouts it configures.

## Decision Matrix

The following matrix helps determine which distributed transaction approach to use for a given scenario.

| Scenario | Recommended Approach | Rationale |
|---|---|---|
| Financial transfer between banks | 2PC with idempotent participants | Must not create or destroy money. Strong consistency required. |
| E-commerce order checkout | TCC or Orchestration Saga | Can reserve inventory (Try), charge card (Confirm), or cancel. Acceptable to lose a sale but not to oversell. |
| Content recommendation update | No distributed transaction | Eventually consistent. Stale recommendations are acceptable. |
| Multi-region user account creation | Saga with regional orchestrators | Avoids cross-region 2PC latency. Acceptable to have briefly inconsistent state. |
| Inventory reservation + payment capture | TCC | Reserves inventory and authorizes payment before committing either. |
| Social media analytics aggregation | No distributed transaction | Each analytics service writes independently. Occasional inconsistency is acceptable. |
| Airline seat booking across alliance partners | Saga (Choreography) | Each airline is independent. Saga compensates if final booking fails. |
| Healthcare patient record update | 2PC with synchronous replication | Must not lose or duplicate patient data. Consistency is legally required. |

## CAP Theorem Positioning

Distributed transactions interact directly with all three properties of the CAP theorem.

**Consistency (C)**: 2PC and 3PC provide strong consistency by ensuring all participants agree on commit or abort. Sagas and TCC provide eventual consistency.

**Availability (A)**: 2PC sacrifices availability during coordinator failures -- participants block until the coordinator recovers. 3PC improves availability through timeouts. Sagas maintain high availability because each step is independent.

**Partition Tolerance (P)**: All distributed transaction protocols must tolerate partitions because they operate across networks. The difference is in how they behave during partitions: 2PC blocks, 3PC uses pre-commit timeouts, and Sagas use compensation.

## Failure Modes

### Failure Mode 1: Coordinator Crash After Prepare

The coordinator sends PREPARE to all participants and receives YES votes. One participant acknowledges. Before the coordinator can write the decision log, the coordinator crashes.

**Effect**: Participants that received PREPARE are in the PREPARED state, holding locks. They cannot commit or abort because the coordinator decision is unknown.

**Mitigation**: Replicated coordinator with synchronous transaction log. If the primary coordinator crashes, the standby coordinator reads the log and continues the protocol.

### Failure Mode 2: Participant Crash During Commit

A participant receives the COMMIT message, applies the changes, and crashes before sending the acknowledgment. The coordinator does not receive the ACK and retries the COMMIT.

**Effect**: The participant misses the first COMMIT. On recovery, the participant queries the coordinator, receives the COMMIT, and applies it. No inconsistency, but the commit is delayed.

**Mitigation**: Participants must be idempotent -- applying the same COMMIT twice must be safe.

### Failure Mode 3: Network Partition During Vote

A partition separates the coordinator from one participant. The coordinator cannot reach the participant to send PREPARE. The coordinator waits for the timeout and then aborts the transaction.

**Effect**: The transaction is aborted, which is the safe behavior. The application may retry the transaction when the partition heals.

**Mitigation**: Set prepare timeouts appropriately (typically 10-30 seconds). Use health checks to detect partitions early.

### Failure Mode 4: Split-Brain in Coordinator Cluster

Two coordinator instances both believe they are the leader and both attempt to manage the same transactions.

**Effect**: Participants receive conflicting instructions. One coordinator sends COMMIT while another sends ABORT for the same transaction.

**Mitigation**: Use a consensus-based leader election (Raft, Paxos) that guarantees a single leader. Use epochs to detect stale leaders.

### Failure Mode 5: Heuristic Hazard

An operator performs heuristic resolution on a stuck participant, committing it manually. Later, the coordinator recovers and sends ABORT to the same participant.

**Effect**: The participant is in COMMITTED state but receives an ABORT. The participant must reject the ABORT, creating inconsistency between participants.

**Mitigation**: Heuristic resolution should be a last resort. Participant must log the heuristic decision and refuse subsequent coordinator messages for that transaction. Run reconciliation jobs to detect and fix inconsistent transactions.

## Anti-Patterns

### Anti-Pattern 1: Distributed Transactions Everywhere

Applying distributed transactions to every multi-service operation, including those that do not need strong consistency.

**Why It Fails**: Distributed transactions add latency, complexity, and reduce availability. Most operations do not need immediate consistency across services. Analytics updates, recommendation refreshes, and logging operations are examples of operations that tolerate eventual consistency.

**Fix**: Categorize operations by consistency requirement. Use distributed transactions only for operations where inconsistency is unacceptable (financial transfers, inventory deduction, seat assignment).

### Anti-Pattern 2: Mixing Long-Running Operations in 2PC

Including a long-running operation (such as a document approval workflow or a shipping process that takes 30 minutes) inside a 2PC transaction.

**Why It Fails**: 2PC holds locks for the duration of the transaction. A 30-minute transaction holds locks on critical resources, blocking all other transactions that access those resources.

**Fix**: Use Sagas or TCC for long-running operations. These patterns release locks between steps. A saga can complete over hours without holding any locks.

### Anti-Pattern 3: Single Coordinator as Single Point of Failure

Running a single coordinator instance without replication.

**Why It Fails**: If the coordinator crashes, all active transactions are stuck in the PREPARED state. Participants hold locks until the coordinator recovers.

**Fix**: Deploy the coordinator as a clustered service with synchronous log replication. Use leader election to ensure continuity.

### Anti-Pattern 4: Ignoring Timeouts

Setting the prepare timeout to infinite or to a very large value (hours) to avoid aborts.

**Why It Fails**: When a participant fails, the coordinator waits indefinitely. The application blocks. Resources are held. Other transactions that need the same resources also block.

**Fix**: Set prepare timeout to 10-30 seconds. Use application-level retry for failed transactions. The transaction will be retried quickly, and the system recovers faster.

### Anti-Pattern 5: Cascading Compensations Without Idempotency

Designing saga compensations that are not idempotent, such as "increment stock by 1" instead of "set stock to snapshot value".

**Why It Fails**: If the compensation is retried (due to a network failure during the first attempt), the compensation executes twice. For "increment stock by 1", this incorrectly adds 2 instead of 1.

**Fix**: Make compensations idempotent. Use "set stock to value X" instead of "increment by 1". Include a compensation ID that the participant deduplicates.

### Anti-Pattern 6: Expecting Sagas to Provide Strong Consistency

Using a saga pattern and expecting the same guarantees as 2PC.

**Why It Fails**: Sagas provide eventual consistency. There is a window between the first step and the compensation where the intermediate state is visible to other transactions.

**Fix**: Accept eventual consistency. Design your application to handle the intermediate states. Display "pending" status to users instead of hiding inconsistency.

## When NOT to Use Distributed Transactions

### Scenario 1: Read-Heavy Systems with Stale Data Tolerance

Systems that primarily read data and can tolerate stale reads should never use distributed transactions. A content delivery network, a product catalog, or a news feed do not need transactional guarantees across services. In these systems, distributed transactions add latency and reduce availability for no benefit.

### Scenario 2: High-Throughput Event Processing

Systems that process millions of events per second (click streams, sensor data, logs) cannot afford the latency and overhead of distributed transactions. These systems use batching, idempotent writes, and eventual consistency.

### Scenario 3: Systems with Autonomous Services

When each service is independently owned and deployed by a different team, introducing a shared transaction coordinator creates a tight coupling between deployment cycles. The coordinator becomes a shared dependency that all teams must coordinate on. In these environments, Sagas or choreography-based patterns are preferred.

### Scenario 4: Cross-Organizational Transactions

When services span organizational boundaries (different companies, different legal entities), a shared coordinator is typically infeasible due to security, compliance, and trust concerns. These systems use message queues with at-least-once delivery and idempotent consumers.

### Scenario 5: Systems with High Availability Requirements (99.99%+)

Systems that require four or more nines of availability cannot use synchronous 2PC because a single coordinator failure can block all transactions. These systems use Sagas with async compensation, typically with a best-effort delivery mechanism.

## Detailed Tradeoff Analysis

### Latency Breakdown by Protocol

**2PC Latency**: Each 2PC transaction requires three network round trips between the coordinator and each participant. These are: PREPARE sent, VOTE received, and COMMIT sent plus ACK received. For a transaction with 3 participants at 50ms RTT each:

```
T_2PC = T_app + 3 * RTT + T_flush = 20ms + 150ms + 10ms = 180ms
```

**3PC Latency**: Each 3PC transaction requires four round trips (PREPARE sent, VOTE received, PRE-COMMIT sent, COMMIT sent plus ACK received):

```
T_3PC = T_app + 4 * RTT + T_flush = 20ms + 200ms + 15ms = 235ms
```

**Saga Latency**: An orchestration saga with N steps requires N sequential calls. Each step is a local transaction with no distributed coordination:

```
T_saga = SUM(T_step_i + RTT_i) = (15ms + 10ms) * 3 = 75ms
```

Saga compensations add latency if a step fails and all previous steps must be compensated. Compensation runs in reverse order:

```
T_saga_failure = T_steps_up_to_failure + T_compensation_steps = 50ms + 40ms = 90ms
```

**TCC Latency**: TCC requires two phases (Try and Confirm/Cancel), each with one round trip:

```
T_TCC = T_try + T_confirm + 2 * RTT = 10ms + 5ms + 100ms = 115ms
```

### Availability Impact

The availability of a 2PC system with N participants is:

```
A_2PC = A_coordinator * A_participant^N
```

If each component has 99.9% availability:

```
A_2PC = 0.999 * 0.999^3 = 0.996 = 99.6% availability
```

This translates to approximately 35 hours of downtime per year for the 2PC system, compared to 8.76 hours per year for a single 99.9% component.

For a saga, the availability is better because the failure of one participant does not block other participants:

```
A_saga = MIN(A_participant_i) = 99.9%
```

The saga availability is bounded by the least available participant, but the failure only affects saga steps that involve that participant. Other sagas that do not use the failed participant continue normally.

### Choosing the Right Protocol: A Quantitative Framework

Use the following weighted scoring system to choose a protocol. Score each protocol from 1 (worst) to 5 (best) for each criterion, multiply by the weight, and sum the results.

| Criterion | Weight | 2PC | 3PC | Saga | TCC |
|---|---|---|---|---|---|
| Consistency | 0.30 | 5 | 5 | 2 | 3 |
| Availability | 0.20 | 2 | 3 | 5 | 4 |
| Latency | 0.15 | 2 | 2 | 4 | 3 |
| Operational Complexity | 0.15 | 3 | 2 | 3 | 3 |
| Developer Effort | 0.10 | 4 | 3 | 2 | 2 |
| Recovery Simplicity | 0.10 | 3 | 2 | 2 | 3 |

**Example: Financial Payment System** (consistency weight = 0.40, availability = 0.15):

```
2PC score: 5*0.40 + 2*0.15 + 2*0.15 + 3*0.10 + 4*0.10 + 3*0.10 = 3.70
Saga score: 2*0.40 + 5*0.15 + 4*0.15 + 3*0.10 + 2*0.10 + 2*0.10 = 2.95
```
