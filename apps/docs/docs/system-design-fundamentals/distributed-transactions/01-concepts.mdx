# Understanding Distributed Transactions

## The Problem Story

At 10:37 AM, a customer places an order on an e-commerce platform for a laptop priced at $1,299. The platform has moved beyond a monolithic database. Three separate services handle the order, each with its own dedicated database:

1. **Inventory Service** -- Deduct one laptop from the warehouse stock count.
2. **Payment Service** -- Charge $1,299 to the customer credit card.
3. **Shipping Service** -- Generate a shipment label for the customer address.

The request arrives. Inventory deducts the laptop -- success. Payment charges the card -- success. The shipping database is under heavy load and the connection times out. The shipment label is never created.

The customer receives a confirmation email. The credit card is charged. The warehouse now shows one fewer laptop. But no shipment is ever dispatched. The customer calls support after three days. Support sees a charged card and no shipment record. The warehouse inventory is inconsistent with real stock. The platform loses $1,299 in goods and spends hours on manual reconciliation.

This is the distributed transaction problem. In a monolithic database, a single `ROLLBACK` would have undone both the inventory deduction and the payment. In a distributed system where each service owns its database, there is no global rollback. Each service committed its change independently, and there is no mechanism to undo them as a group.

The core challenge is this: how do you maintain atomicity across multiple independent databases, each controlled by a different service, when no single database can guarantee consistency across all of them?

### The Airline Check-In Counter Analogy

Walk into any major airport terminal. Twenty check-in counters serve a single flight. Each counter has its own terminal connected to the airline system. The goal is simple: process all 200 passengers for flight AA-101, departing at 14:30.

Imagine what happens without coordination. Counter 7 checks in passenger Smith and assigns seat 14A. Counter 12 simultaneously checks in passenger Jones and assigns seat 14A. Two passengers now hold boarding passes for the same seat. When both try to board, the gate agent must resolve the conflict manually -- delaying departure.

Now add a baggage drop. Counter 3 accepts passenger Brown luggage. The luggage goes to the ramp. But the passenger never completes check-in because the system crashes. Now there is luggage on the ramp for a passenger who is not on the flight. The luggage must be retrieved, delaying departure further.

The airline solves this with a transaction coordinator. When a passenger checks in, the system assigns the seat, accepts bags, prints the boarding pass, and updates the manifest -- all as a single atomic operation. If any step fails, none of the changes persist.

In the distributed version of this problem, each check-in counter is a separate service running its own database. The seat assignment is in one database. The baggage record is in another. The passenger manifest is in a third. No single database coordinates these three. The system needs a distributed transaction protocol to ensure that all three databases either agree to commit or all agree to abort.

## The Insight

A distributed transaction is a set of operations performed against multiple independent databases that must all succeed or all fail together, preserving atomicity across system boundaries.

A distributed transaction works through a coordination protocol that puts all participants through a two-phase decision process:

1. **Prepare Phase**: The coordinator asks each participant whether it can commit the transaction. Each participant votes yes or no.
2. **Commit Phase**: If all participants voted yes, the coordinator tells everyone to commit. If any voted no, the coordinator tells everyone to abort.

This protocol, known as Two-Phase Commit (2PC), is the foundational mechanism for distributed transactions. Variants such as Three-Phase Commit (3PC), Sagas, and TCC (Try-Confirm/Cancel) address specific limitations of 2PC.

## Core Vocabulary

| Term | Meaning |
|---|---|
| Distributed Transaction | A set of operations across multiple databases treated as a single atomic unit |
| Coordinator | The entity that manages the two-phase decision process across all participants |
| Participant | A database or resource manager that takes part in the distributed transaction |
| Two-Phase Commit (2PC) | A protocol that uses prepare and commit phases to achieve atomic commitment |
| Three-Phase Commit (3PC) | A variant of 2PC that adds a pre-commit phase to reduce blocking |
| Prepare Phase | The first phase where the coordinator asks each participant to prepare and vote |
| Commit Phase | The second phase where the coordinator sends commit or abort based on votes |
| Rollback | Undoing a prepared but uncommitted transaction when the coordinator sends abort |
| Saga | A pattern that decomposes a distributed transaction into a sequence of local transactions with compensating actions |
| Compensating Transaction | A local transaction that semantically undoes a previous local transaction in a saga |
| Isolation Level | The degree to which a transaction is isolated from the effects of other concurrent transactions |
| Transaction Log | A durable record of every prepare, commit, and abort decision, used for recovery |
| TCC (Try-Confirm/Cancel) | A pattern where each participant reserves resources in the Try phase, then confirms or cancels |

## Step by Step: The 10 Steps of Two-Phase Commit

Two-Phase Commit is the foundational protocol for distributed transactions. Every participant and the coordinator follow a strict sequence of ten steps.

### Phase 1: Prepare

**Step 1 -- Begin**: The application sends a `BEGIN DISTRIBUTED TRANSACTION` to the coordinator. The coordinator assigns a globally unique transaction ID (XID) and records the transaction state as `ACTIVE` in its transaction log.

**Step 2 -- Execute Local Operations**: The application performs operations on each participant within the scope of the distributed transaction. Each participant records its local changes but does not make them visible to other transactions. The changes are held in a provisional state.

**Step 3 -- Prepare Request**: The coordinator sends a `PREPARE` message to every registered participant, including the transaction ID and a timeout value. Each participant receives this message and begins its internal prepare process.

**Step 4 -- Internal Prepare**: Each participant writes all its dirty pages to disk, acquires all necessary locks, and writes a prepare record to its own transaction log. The prepare record contains the transaction ID, the list of modified pages, and a checksum. Once the prepare record is durable on disk, the participant is ready to commit.

**Step 5 -- Vote**: Each participant responds to the coordinator with either:
- `VOTE YES`: The participant has prepared successfully and is ready to commit.
- `VOTE NO`: The participant cannot prepare. This can be caused by a resource conflict, a constraint violation, or a timeout.

### Phase 2: Commit

**Step 6 -- Decision**: The coordinator collects all votes. If all participants voted YES, the coordinator decides to commit. If any participant voted NO or the coordinator times out waiting for a vote, the coordinator decides to abort.

**Step 7 -- Log Decision**: The coordinator writes the decision (COMMIT or ABORT) to its transaction log on durable storage. This is a critical step -- the decision must survive a coordinator crash.

**Step 8 -- Send Decision**: The coordinator sends the decision to every participant. For a COMMIT decision, it sends a `COMMIT` message. For an ABORT decision, it sends an `ABORT` message.

**Step 9 -- Execute Decision**: Each participant receives the decision. If COMMIT, the participant makes all prepared changes permanent and releases all locks. If ABORT, the participant discards all prepared changes and releases all locks. Each participant writes the outcome to its own transaction log.

**Step 10 -- Acknowledge**: Each participant sends an acknowledgment to the coordinator. Once the coordinator has received acknowledgments from all participants, it writes the final outcome to its transaction log and marks the transaction as `COMPLETED`.

## Variants of Distributed Transactions

### Two-Phase Commit (2PC)

2PC is the standard protocol for distributed transaction coordination. It guarantees atomic commitment across all participants. Its main weakness is blocking: if the coordinator crashes after sending PREPARE but before sending COMMIT, participants hold locks and resources until the coordinator recovers. This blocking period can last hours in production systems.

### Three-Phase Commit (3PC)

3PC extends 2PC by adding a third phase called pre-commit. This phase allows participants to timeout and unilaterally abort if the coordinator does not respond within a timeout period. 3PC eliminates the blocking problem of 2PC but is more complex and requires all participants to support timeouts.

In 3PC, the coordinator:

1. Sends `PREPARE` to all participants and collects votes.
2. If all vote YES, sends `PRE-COMMIT` to all participants. This tells them to expect a COMMIT and to prepare to wait.
3. Sends `COMMIT` only after all participants acknowledge the pre-commit.

The pre-commit phase gives participants a timeout: if a participant has acknowledged pre-commit but does not receive a COMMIT within the timeout, it can query other participants. If no coordinator decision is found, the participant can abort safely.

### Saga Pattern

The Saga pattern takes a fundamentally different approach. Instead of coordinating all participants through a protocol, it decomposes the distributed transaction into a sequence of local transactions, each executed against a single service. Each local transaction has a compensating transaction that semantically undoes its effects.

There are two variants of the Saga pattern:

**Choreography-based Saga**: Each service publishes an event after its local transaction completes. The next service in the saga listens for the event and executes its own transaction. If a service fails, it publishes a compensation event that triggers rollback of the saga.

**Orchestration-based Saga**: A central saga orchestrator manages the sequence. The orchestrator sends commands to each service in order, tracks the state of the saga, and executes compensating transactions in reverse order if any step fails.

### TCC Pattern (Try-Confirm/Cancel)

TCC is a protocol where each participant exposes three operations: Try, Confirm, and Cancel. The Try phase reserves resources without committing them. The Confirm phase makes the reservation permanent. The Cancel phase releases the reservation.

TCC is commonly used in payment systems and resource booking scenarios where resources must be reserved temporarily before final commitment.

**Try Phase**: Each participant reserves the required resources. For a payment service, this means authorizing a hold on the credit card. For an inventory service, this means setting aside the items. The reservation is not yet visible to other transactions.

**Confirm Phase**: If all participants successfully reserved resources, the coordinator calls Confirm on each participant. The reservation becomes permanent. In payment terms, the hold is captured as a charge.

**Cancel Phase**: If any participant fails to reserve resources, or the coordinator decides to abort, the coordinator calls Cancel on each participant that successfully reserved resources. The hold is released.

## When to Use Each Variant

| Variant | Best For | Avoid When |
|---|---|---|
| 2PC | Short-lived transactions, homogeneous databases, reliable network | Long-running operations, high latency, heterogeneous systems |
| 3PC | Environments where coordinator crash is likely | Systems that cannot tolerate protocol overhead |
| Saga (Choreography) | Event-driven microservices, existing message infrastructure | Complex compensation logic, strong consistency requirements |
| Saga (Orchestration) | Multi-step business processes, audit requirements | High-throughput systems, coordinator becomes bottleneck |
| TCC | Resource reservation (payments, bookings) | Operations that cannot reserve resources in advance |

## Mental Model Check

Answer these five questions to verify your understanding:

**Question 1**: An e-commerce platform uses distributed transactions for checkout. The inventory service prepares successfully. The payment service prepares successfully. The shipping service times out during prepare. What should the coordinator do?

**Answer**: The coordinator must abort the transaction because one participant failed to prepare. The coordinator sends ABORT to inventory and payment, which roll back their prepared changes. The customer is not charged and the inventory is restored.

**Question 2**: What is the fundamental difference between a distributed transaction using 2PC and a saga?

**Answer**: 2PC uses a global lock-and-commit protocol where participants hold locks until the coordinator decides. A saga uses local transactions with compensating actions and does not hold locks across participants. 2PC provides stronger consistency guarantees at the cost of availability and lock duration. A saga provides eventual consistency with better availability and throughput.

**Question 3**: In a TCC pattern, what happens if the coordinator crashes after calling Try on two of three participants?

**Answer**: The coordinator must recover and determine the state. If the transaction log shows that Try was sent to the first two participants but not the third, the coordinator should call Cancel on the first two to release their resource reservations. This is why idempotency is critical in TCC implementations.

**Question 4**: Why does 3PC reduce the blocking problem compared to 2PC?

**Answer**: 3PC adds a pre-commit phase that allows participants to set timeouts. If a participant acknowledges pre-commit but does not receive a COMMIT within its timeout, it queries other participants to determine the coordinator decision. If no decision is found, the participant can safely abort. In 2PC, a participant that received PREPARE but not COMMIT blocks indefinitely.

**Question 5**: A payment system uses 2PC to charge a credit card. The card network approves the charge during prepare. The coordinator crashes before sending COMMIT. What happens when the coordinator recovers?

**Answer**: The coordinator reads its transaction log from durable storage. If the log contains a COMMIT decision, it resumes sending COMMIT to all participants. If the log contains no decision (the coordinator crashed before deciding), it sends ABORT to all participants to release the holds. The transaction log is the source of truth for coordinator recovery.

## Additional Mental Model Questions

**Question 6**: In an orchestration saga for hotel booking, the reserve-room step succeeds, the charge-card step succeeds, but the confirm-booking step fails. What should the orchestrator do?

**Answer**: The orchestrator must execute compensating transactions in reverse order. First, it voids the card charge by calling the payment service compensation. Then it releases the room reservation by calling the inventory service compensation. The booking is completely undone, and the customer sees a "Booking failed" message.

**Question 7**: A coordinator uses 2PC with three participants. The coordinator receives YES from participants A and B but times out waiting for C. What is the correct decision?

**Answer**: The coordinator must abort the transaction. Unanimous consent is required for commit. Since participant C did not vote YES (either voted NO or timed out), the coordinator writes ABORT_DECISION to the log and sends ABORT to A and B, which roll back their prepared changes.

**Question 8**: In a TCC pattern, why is idempotency critical for the Confirm operation?

**Answer**: Idempotency ensures that if the coordinator crashes after sending Confirm but before receiving the ACK, retrying the Confirm does not double-apply the change. The participant checks if the reservation was already confirmed using the reservation ID and returns the existing result instead of processing the confirmation again.

**Question 9**: What is the difference between a compensating transaction in a saga and a rollback in 2PC?

**Answer**: A rollback in 2PC is invisible to the outside world -- it restores the database to its state before the transaction began, and no other system ever sees the intermediate state. A compensating transaction in a saga is a visible business operation. Other systems may have observed the intermediate state (such as a room showing as reserved and then becoming available again), and the compensation must handle this visibility.

**Question 10**: A system uses choreography-based saga for order processing. The inventory service publishes an "ItemReserved" event. The payment service listens and charges the card. The payment service publishes a "PaymentProcessed" event. The shipping service listens and creates a label. If the payment service crashes after receiving the event but before publishing "PaymentProcessed", how does the system recover?

**Answer**: The payment service, upon recovery, reads its event log and finds the unprocessed "ItemReserved" event. It reprocesses the event, charges the card (idempotently -- checking if the charge was already made), and publishes "PaymentProcessed". The shipping service receives this event and creates the label. The inventory reservation is still valid (assuming it has not expired), and the saga continues from where it was interrupted.

## Common Misconceptions

**Misconception 1: "2PC guarantees that all participants commit at the same instant."**

No. 2PC guarantees that all participants reach the same final state (all committed or all aborted), but they do not reach this state simultaneously. There is a window where some participants have committed and others have not yet received the COMMIT message. During this window, the system is in an inconsistent intermediate state.

**Misconception 2: "A saga provides the same consistency as 2PC."**

No. A saga provides eventual consistency at best. There is a window between saga steps where the intermediate state is visible to other transactions. For example, after the inventory deduction but before the payment charge, a read-only query might see an item as sold but no revenue recorded.

**Misconception 3: "TCC avoids all locking issues."**

No. TCC reduces locking compared to 2PC because resources are only held during the Try phase, not during the Confirm phase. However, during Try, participants hold resource reservations, which can block other transactions from accessing those resources until the reservation expires or is confirmed.

**Misconception 4: "Distributed transactions solve all multi-service consistency problems."**

No. Distributed transactions address atomicity across participants, but they do not solve all consistency problems. Network delays, clock skew, partial failures, and data conflicts can still cause inconsistencies that require manual reconciliation.

**Misconception 5: "Three-phase commit eliminates all blocking."**

No. 3PC reduces but does not eliminate blocking. Under certain failure scenarios (such as all participants crashing simultaneously), blocking can still occur. 3PC also assumes that all participants support timeouts and can coordinate recovery, which is not always the case.

## Frequently Asked Questions

**Q: Can 2PC be used across different database technologies?**

A: Yes, as long as each database supports the XA protocol or provides a participant adapter. PostgreSQL, MySQL, Oracle, SQL Server, and DB2 all support XA transactions. NoSQL databases like DynamoDB and MongoDB do not support XA but may provide their own transaction APIs.

**Q: What happens if a participant never responds after voting YES?**

A: The coordinator marks the participant as unresponsive and continues retrying the COMMIT message. The retry interval is typically 1-5 seconds with an exponential backoff cap. If the participant recovers, it receives the COMMIT and applies the changes. The transaction log ensures that the coordinator does not forget about the unresponsive participant.

**Q: Is it possible to have a mixed transaction where some participants use 2PC and others use a saga?**

A: Yes, this is called a hybrid transaction. For example, a booking flow might use 2PC for the payment and inventory steps (which need strong consistency) and a saga for the notification and analytics steps (which can tolerate eventual consistency). The coordinator manages both types of participants within the same transaction context.

**Q: How does clock skew affect distributed transactions?**

A: Clock skew affects transaction ordering and timeout calculations. If the coordinator clock and a participant clock differ by 100ms, a prepare timeout of 30s might expire prematurely or late. Production systems use NTP with sub-millisecond accuracy and include clock drift compensation in timeout calculations. Google Spanner solves this with TrueTime hardware clocks.

**Q: What is the minimum number of participants needed for a distributed transaction?**

A: Two. A distributed transaction requires at least two participants because the transaction would be local with a single participant. However, some systems use a single-participant distributed transaction for consistency with the coordinator protocol, especially when the system may later add more participants.
