# Distributed Transactions -- Production Architecture

## Naive Architecture: The Single-Database Assumption

The naive approach to distributed transactions is to embed transaction logic directly into the application code without a dedicated coordinator. In this approach, the application sends operations to each database independently and attempts to roll back manually if something fails.

This architecture breaks down immediately in production for three reasons:

1. **No Atomicity**: There is no coordinator to enforce the all-or-nothing guarantee. If the application crashes mid-transaction, some databases may have committed and others may not.

2. **No Recovery**: The application has no durable log to recover from. If the application restarts, it has no record of which transactions were in progress.

3. **No Healing**: There is no retry mechanism for failed operations. If a database is temporarily unavailable, there is no background process to complete the transaction when it recovers.

## Production Architecture

A production-grade distributed transaction system deploys a dedicated coordinator service alongside the application services. The coordinator is a separate, highly available service with its own replicated transaction log.

### Core Components

**Transaction Coordinator Service**: A horizontally scalable service that manages distributed transactions. The coordinator is stateless in terms of transaction processing -- all state is kept in the transaction log. Multiple coordinator instances can run behind a load balancer, with each transaction assigned to one coordinator instance.

**Transaction Log**: A replicated, durable log stored in a distributed database (Apache Kafka, Amazon DynamoDB, or a dedicated RDBMS). The log is the source of truth for all transaction state. Replication across multiple availability zones ensures the log survives zone failures.

**Participant Adapter**: A library or sidecar process that adapts each participant service to the coordinator protocol. The adapter intercepts database operations within the scope of a distributed transaction and manages the local prepare/commit/abort lifecycle.

**Participant Transaction Manager**: Each participant service runs a local transaction manager that handles the participant side of the 2PC protocol. This manager writes prepare records to the participant local database and communicates with the coordinator.

**Dead Letter Queue (DLQ)**: Transactions that cannot be resolved after the maximum number of retries are moved to a dead letter queue for manual investigation. Operations teams monitor the DLQ and resolve stuck transactions.

**Monitoring and Alerting**: A dashboard that tracks active transactions, prepared transactions, unresolved transactions, coordinator health, participant latency, and retry counts.

## Integration Patterns

### Pattern 1: Coordinator as Library

The coordinator logic is embedded as a library within the application process. This is the simplest integration pattern and works well for systems with a small number of services (2-5) and moderate transaction volumes (under 1,000 TPS).

**Pros**: Low latency (no network hop for coordination), simple deployment.

**Cons**: Coordinator crashes with the application, no independent recovery, limited scalability.

### Pattern 2: Coordinator as Sidecar

The coordinator runs as a sidecar process alongside each application instance. This pattern is common in Kubernetes deployments where each pod includes a coordinator sidecar container. The sidecars communicate with each other to form a distributed coordinator mesh.

**Pros**: Decouples coordinator lifecycle from application, enables independent scaling, supports rolling upgrades.

**Cons**: Additional resource overhead, more complex deployment.

### Pattern 3: Coordinator as Service

The coordinator runs as an independent service with its own cluster. Applications communicate with the coordinator over gRPC or HTTP. This pattern is used by production-grade transaction coordinators such as Apache Seata and Narayana.

**Pros**: Fully decoupled, scales independently, supports cross-service transactions.

**Cons**: Higher latency due to network hops, requires service discovery and load balancing.

## Multi-Region Architecture

Deploying distributed transactions across multiple geographic regions introduces additional complexity. Inter-region latency is typically 50-200ms, which makes synchronous 2PC across regions impractical.

### Strategy 1: Region-Local Transactions with Async Replication

Transactions are confined to a single region. All participants for a transaction must be in the same region. Changes are replicated to other regions asynchronously using database replication or event streaming.

**Pros**: Low latency within each region, simple 2PC within the region.

**Cons**: Eventual consistency across regions, potential for conflicts.

### Strategy 2: Global Coordinator with 3PC

A global coordinator runs in a third region or uses a multi-region consensus protocol (such as Spanner TrueTime). The coordinator uses 3PC instead of 2PC to tolerate the higher latency and increased failure probability of cross-region communication.

**Pros**: Strong consistency across regions.

**Cons**: High latency (minimum 3 RTTs across regions), complex infrastructure.

### Strategy 3: Saga with Regional Orchestrators

Each region has its own saga orchestrator. The orchestrators communicate asynchronously using a global event bus. Cross-region transactions are decomposed into region-local sagas connected by async events.

**Pros**: Low latency within each region, no cross-region locking.

**Cons**: Eventual consistency across regions, complex compensation logic.

## Capacity Planning

### Sizing the Coordinator

The coordinator capacity is determined by three factors:

1. **Transaction Throughput**: The number of transactions per second the coordinator must process. Each transaction requires three message exchanges (prepare, decision, ack) per participant.

2. **Transaction Duration**: The average time a transaction holds resources. Longer transactions consume more coordinator memory because the coordinator must keep their state in memory for the transaction duration.

3. **Participant Count**: The number of participants per transaction. Each participant adds state tracking and message overhead.

**Formula for coordinator memory**:

```
Memory = ActiveTransactions * (ParticipantCount * ParticipantStateSize + TransactionStateSize)
```

For example, with 1,000 active transactions, 3 participants each, 256 bytes per participant state, and 512 bytes per transaction state:

```
Memory = 1000 * (3 * 256 + 512) = 1000 * 1280 = 1.28 MB
```

This is the minimum memory. Production systems typically allocate 4-8 GB to handle spikes and garbage collection.

### Sizing the Transaction Log

The transaction log must be sized to accommodate the peak transaction volume for the log retention period. Each transaction generates approximately N+2 log records (N prepare records, 1 decision record, 1 completion record).

**Formula for log storage**:

```
LogSize = TransactionRate * RecordSize * AvgLifetimeSeconds * RetentionFactor
```

For example, with 5,000 transactions per second, 500 bytes per record, 4 records per transaction, and a 24-hour retention:

```
LogSize = 5000 * 500 * 4 * 86400 = 864 GB per day
```

Production systems typically configure automatic log pruning: records older than 7 days are deleted.

### Network Capacity

Each transaction generates 3N messages (PREPARE, COMMIT/ABORT, ACK). The network bandwidth required is:

```
Bandwidth = TransactionRate * MessageCount * MessageSize
```

For example, with 5,000 TPS, 3 participants, and 1 KB messages:

```
Bandwidth = 5000 * 9 * 1024 = 46 Mbps
```

### Monitoring Metrics

| Metric | Description | Warning Threshold | Critical Threshold |

| log_write_latency | Latency of log writes | > 10ms | > 50ms |

### Runbook: Unresolved Transaction

When a transaction is stuck in the PREPARED state:

1. **Identify**: Query the coordinator dashboard for unresolved transactions. Note the XID and participant list.
2. **Diagnose**: Check participant health. Is the participant database running? Is the network between coordinator and participant operational?
3. **Retry**: Force the coordinator to retry the commit or abort for the stuck transaction.
4. **Heuristic Resolution**: If the participant is down and cannot be recovered within the SLA, perform a heuristic resolution: manually determine whether to commit or abort and instruct the participant accordingly.
5. **Audit**: Log the resolution. If a heuristic resolution was used, schedule a data reconciliation job to verify consistency.

### Runbook: Coordinator Failure

When the coordinator process crashes:

1. **Failover**: If the coordinator is clustered, traffic is routed to another coordinator instance.
2. **Recovery**: The new coordinator instance reads the transaction log and resolves all transactions in PREPARED state.
3. **Verification**: Check that all unresolved transactions were properly resolved. Verify that no locks are held indefinitely.
4. **Root Cause**: Investigate the crash and apply fixes before restarting the failed instance.

## Deployment Topology

A production distributed transaction system is typically deployed across three or more availability zones within a single region. Each zone runs a coordinator instance and a set of participant services. The transaction log is replicated synchronously across zones using consensus (such as Raft or Paxos).

The deployment follows these principles:

1. **Coordinator instances are stateless**: All state is in the replicated log. Coordinator instances can be scaled horizontally without data migration.

2. **Participant services are deployed with their databases**: Each participant runs in the same availability zone as its database to minimize latency.

3. **The transaction log uses synchronous replication**: The log writes are acknowledged only after at least two of three zones have confirmed the write.

4. **Monitoring is deployed in a separate observability cluster**: Monitoring does not share infrastructure with the coordinator to ensure it remains operational during coordinator failures.

## Security Architecture

### Authentication

Every message between the coordinator and participants must be authenticated. The coordinator issues a token at the start of each transaction that participants verify before accepting PREPARE, COMMIT, or ABORT messages. The token contains the XID, the participant identity, and an HMAC signature.

The token format is:

```
Token = Base64(XID + ":" + ParticipantID + ":" + Expiry + ":" + HMAC(key, XID + ParticipantID + Expiry))
```

Participants verify the HMAC using a shared secret key distributed during deployment. Tokens expire after the transaction timeout (typically 30-60 seconds) to prevent replay attacks.

### Authorization

The coordinator enforces authorization rules for each transaction:

1. **Participant Authorization**: Only registered participants can be included in a distributed transaction. The coordinator maintains an allowlist of participant endpoints.

2. **Operation Authorization**: The coordinator validates that the requested operations are permitted for the calling service. For example, a checkout service is authorized to call inventory deduction and payment charge, but a recommendation service is not.

3. **Read-Only Access**: Participants that only read data during a transaction are flagged as read-only and are excluded from the commit phase.

### Encryption

All coordinator-participant communication uses TLS 1.3. The transaction log is encrypted at rest using AES-256. The encryption keys are managed by a key management service (KMS) with automatic rotation every 90 days.

### Audit Trail

Every state transition is logged with:

- XID and participant identity.
- Timestamp with nanosecond precision.
- Calling service identity and IP address.
- Request and response payload hashes.
- The coordinator node that processed the transaction.

Audit logs are append-only and written to a separate immutable storage system (such as AWS S3 with Object Lock or a write-once-read-many appliance). Audit logs are retained for a minimum of 7 years for financial services compliance.

## Testing Distributed Transactions

### Unit Testing

Test each participant endpoint (prepare, commit, abort, try, confirm, cancel) in isolation. Verify that:

1. The endpoint returns the correct response for valid inputs.
2. The endpoint returns the correct error for invalid inputs.
3. The endpoint is idempotent -- calling it twice with the same parameters produces the same result.
4. The endpoint handles concurrent calls correctly.

### Integration Testing

Test the coordinator with all participants in a test environment. Verify:

1. Happy path: All participants vote YES, transaction commits.
2. Unanimous abort: One participant votes NO, transaction aborts.
3. Timeout: One participant does not respond, transaction aborts after timeout.
4. Coordinator crash: Coordinator crashes at each state, recovery resolves correctly.
5. Participant crash: Participant crashes during prepare, recovery queries coordinator.

### Chaos Engineering

Run chaos experiments in a staging environment:

1. Network latency injection: Add 100ms, 500ms, and 1000ms of latency between coordinator and participants.
2. Packet loss: Drop 1%, 5%, and 10% of packets between coordinator and participants.
3. Process kills: Randomly kill coordinator and participant processes.
4. Resource exhaustion: Limit CPU, memory, and disk IO for coordinator and participants.

Each chaos experiment must verify that:

- No committed transaction is rolled back.
- No aborted transaction is committed.
- The system recovers within the configured timeout.
- No data corruption occurs.

### Performance Testing

Run performance tests to determine the throughput ceiling:

1. Baseline: Measure throughput without distributed transactions (local transactions only).
2. 2PC overhead: Measure throughput with 2PC for 2, 3, and 5 participants.
3. Scaling: Measure throughput as the coordinator cluster scales from 1 to 10 nodes.
4. Latency distribution: Measure the P50, P95, P99, and P99.9 commit latency.

## Disaster Recovery and Business Continuity

### Recovery Point Objective (RPO) and Recovery Time Objective (RTO)

For distributed transaction systems, the RPO and RTO are defined as follows:

| Component | RPO | RTO | Strategy |
|---|---|---|---|
| Transaction log | 0 (no data loss) | 30 seconds | Synchronous replication across 3 AZs |
| Coordinator cluster | 0 (no state loss) | 10 seconds | Active-passive with automatic failover |
| Participant services | 0 (replay from log) | 60 seconds | Auto-scaling group with health checks |
| Monitoring | 5 minutes | 5 minutes | Separate observability cluster |

**Scenario 1: Single Availability Zone Failure**

The coordinator is deployed across three AZs. One AZ becomes unavailable. The transaction log is still accessible from the remaining two AZs (quorum). The coordinator continues processing transactions. Participants in the failed AZ are unavailable, so new transactions that require those participants fail. Transactions that were in PREPARE state on the failed AZ participants are rolled back after the coordinator timeout.

**Scenario 2: Entire Region Failure**

The coordinator and participants are replicated to a secondary region. The transaction log is asynchronously replicated to the secondary region with a 5-second lag. When the primary region fails:

1. The secondary region detects the failure through health checks (30-second timeout).
2. The secondary region promotes its coordinator to active.
3. The secondary region replays the transaction log from the last checkpoint.
4. Transactions committed before the 5-second replication lag are preserved.
5. Transactions in the 5-second window are lost (RPO = 5 seconds).
6. The secondary region is ready to serve traffic within 60 seconds (RTO = 60 seconds).

**Scenario 3: Data Corruption**

The transaction log is corrupted by a storage subsystem failure. The coordinator detects the corruption through checksum validation. Recovery steps:

1. The coordinator stops accepting new transactions.
2. The coordinator switches to a read-only replica of the transaction log.
3. The corrupted log is restored from the most recent backup (taken every 5 minutes).
4. Transactions in the 5-minute window between the backup and the corruption are replayed from participant logs.
5. The coordinator resumes accepting transactions.

### Backup Strategy

| Backup Type | Frequency | Retention | Storage |
|---|---|---|---|
| Full transaction log backup | Daily | 30 days | S3/Glacier |
| Incremental log backup | Every 5 minutes | 7 days | S3 Standard |
| Coordinator configuration | Per deployment | Life of deployment | Git repository |
| Participant state snapshot | Hourly | 90 days | S3/Glacier |

### Business Continuity Plan

When a disaster affects the distributed transaction system:

1. **Detection (0-1 minute)**: Monitoring alerts the operations team. The system automatically attempts failover.

2. **Assessment (1-5 minutes)**: The operations team determines the scope of the disaster. Is it a single AZ failure, a region failure, or a data corruption event?

3. **Activation (5-30 minutes)**: If the disaster affects transaction processing, the operations team activates the disaster recovery plan. This may involve switching to a secondary region, restoring from backup, or running in degraded mode.

4. **Degraded Mode (30 minutes - 24 hours)**: While the primary system is being restored, the system operates in degraded mode. In degraded mode, the system accepts read requests but does not accept new transactions that require distributed coordination. Some operations may be queued for processing after recovery.

5. **Recovery (1-24 hours)**: The primary system is restored from backup. The transaction log is replayed. The system verifies consistency across all participants.

6. **Verification (24-48 hours)**: A reconciliation job verifies that all transactions are consistent. Any discrepancies are fixed. The system is returned to normal operation.

### Recovery Drills

The operations team practices the following recovery drills quarterly:

**Drill 1: Coordinator Failover**
- Simulate a coordinator process crash.
- Verify that the standby coordinator detects the failure within 10 seconds.
- Verify that the standby coordinator reads the transaction log and resumes processing.
- Verify that no transactions are lost or duplicated.

**Drill 2: Participant Failure**
- Simulate a participant database crash.
- Verify that the coordinator detects the participant failure and retries.
- Verify that transactions involving the failed participant are aborted.
- Verify that transactions not involving the failed participant continue normally.
- Restart the participant and verify that the coordinator resumes communication.

**Drill 3: Network Partition**
- Introduce a network partition between the coordinator and one participant.
- Verify that the coordinator times out and aborts transactions involving the partitioned participant.
- Heal the partition.
- Verify that the coordinator and participant resume normal communication.
- Verify that the participant recovers correctly from the aborted transactions.

**Drill 4: Full Region Failover**
- Simulate a full region failure (all services in a region become unavailable).
- Verify that the secondary region detects the failure and activates its coordinator.
- Verify that the secondary region replays the transaction log.
- Verify that clients are redirected to the secondary region.
- Verify that the secondary region processes transactions correctly.
- Restore the primary region and verify that it catches up with the secondary region.

Each drill must complete within the RTO targets. Drill results are documented and reviewed by the operations team.
