# Distributed Transactions -- Under the Hood

## Internal Architecture

A distributed transaction coordinator is composed of three core subsystems that work together to guarantee atomic commitment across multiple participants.

### Coordinator

The coordinator is the central decision-making entity. It receives the transaction request, manages the two-phase protocol, and makes the final commit or abort decision. The coordinator maintains the following state for each transaction:

- **Transaction ID**: A globally unique identifier (XID) assigned at transaction start.
- **State**: The current state in the transaction lifecycle (ACTIVE, PREPARED, COMMITTED, ABORTED, COMPLETED).
- **Participant List**: The set of registered participants and their current vote status.
- **Timeout**: A deadline for receiving votes from all participants.
- **Decision**: The commit or abort decision, written to durable storage.

### Transaction Log

The transaction log is a durable append-only record of every transaction state change. The coordinator writes to the log before sending any protocol message. This ensures that the coordinator can recover its state after a crash. The log contains:

- **Begin Record**: Transaction ID, timestamp, participant list.
- **Prepare Record**: The coordinator sent PREPARE to all participants.
- **Decision Record**: The coordinator decided COMMIT or ABORT, along with the list of participants.
- **Ack Record**: A participant acknowledged the decision.
- **Completion Record**: The transaction is fully resolved.

### Recovery Manager

The recovery manager handles crash recovery for both the coordinator and participants. On startup, the recovery manager reads the transaction log and determines the state of every unresolved transaction. For each unresolved transaction, the recovery manager either completes the commit or initiates a rollback.

The recovery manager has three responsibilities:

1. **Replay**: Re-read the log and reconstruct in-memory state for each transaction.
2. **Resolve**: For transactions in PREPARED state, determine the correct outcome.
3. **Cleanup**: Release resources held by orphaned transactions.

## Data Structures

### Transaction Log Record

Each record in the transaction log contains the following fields:

```
TransactionLogRecord {
    LSN: uint64              // Log Sequence Number, monotonically increasing
    XID: string              // Global transaction identifier
    RecordType: enum {       // Type of log record
        BEGIN,
        PREPARE_SENT,
        VOTE_YES,
        VOTE_NO,
        COMMIT_DECISION,
        ABORT_DECISION,
        COMMIT_SENT,
        ABORT_SENT,
        ACK,
        COMPLETED
    }
    Timestamp: uint64        // Unix millisecond timestamp
    ParticipantID: string    // The participant this record refers to
    Payload: bytes           // Optional data (checksums, lock info)
    PrevLSN: uint64          // Previous LSN for this transaction, for rollback
}
```

### Lock Table

The coordinator maintains a lock table to track which resources are held by which transactions. Each entry contains:

```
LockTableEntry {
    ResourceID: string       // The resource being locked
    LockMode: enum {         // Shared, Exclusive, Update
        SHARED,
        EXCLUSIVE,
        UPDATE
    }
    OwnerXID: string         // The transaction holding the lock
    Status: enum {           // HELD, WAITING, TIMEOUT
        HELD,
        WAITING,
        TIMEOUT
    }
    AcquiredAt: uint64       // Timestamp when the lock was acquired
    Timeout: uint64          // Maximum hold time in milliseconds
    Queue: []string          // Queue of transactions waiting for this lock
}
```

### Participant State

The coordinator maintains a state table for each participant in each transaction:

```
ParticipantState {
    ParticipantID: string    // Unique identifier for the participant
    Endpoint: string         // Network address for the participant
    TransactionXID: string   // The transaction this participant is part of
    Phase: enum {            // Current phase of the participant
        INITIAL,
        PREPARED,
        COMMITTED,
        ABORTED,
        UNKNOWN
    }
    Vote: enum {             // The vote cast by this participant
        YES,
        NO,
        NOT_YET_VOTED,
        TIMEOUT
    }
    RetryCount: int          // Number of retries for this participant
    LastMessageTime: uint64  // Timestamp of last communication
}
```

## Lifecycle: Twelve Steps of 2PC with Failure Points

A distributed transaction passes through twelve distinct states during its lifecycle. Each state transition is triggered by a protocol message, a timeout, or a failure condition.

### State 1: INITIAL

The transaction is created by the application. The coordinator assigns an XID and records the BEGIN record in the transaction log. No participants are registered yet. The application begins executing operations on participant databases.

### State 2: ACTIVE

The application registers participants and executes operations on each one. Each participant runs its local operations within the scope of the distributed transaction. The coordinator tracks which participants are registered but has not yet sent PREPARE.

**Failure Point 1**: The application crashes before registering all participants. The coordinator's timeout detects the inactivity and aborts the transaction. The coordinator writes an ABORT record to the log.

### State 3: PREPARE_SENT

The coordinator sends PREPARE messages to all registered participants. This is the most critical phase. The coordinator writes a PREPARE_SENT record to the log before sending each message, ensuring that the act of sending is durable.

**Failure Point 2**: A participant is unreachable. The coordinator retries the PREPARE message up to a configurable number of retries (typically 3). If all retries fail, the coordinator records the participant as TIMEOUT and proceeds to abort.

### State 4: PREPARE_WAITING

The coordinator waits for votes from all participants. Each participant executes its internal prepare logic: flushing dirty pages, acquiring locks, and writing a prepare record to its own transaction log.

### State 5: VOTE_COLLECTED

Each participant sends its vote. The coordinator records each vote in the participant state table. The coordinator must receive votes from all participants within the configured timeout.

**Failure Point 3**: A participant votes NO. This is a definitive abort. The coordinator immediately moves to the ABORT_DECISION state without waiting for remaining votes.

**Failure Point 4**: The coordinator times out waiting for a vote. The coordinator records the participant as TIMEOUT and moves to ABORT_DECISION.

### State 6: DECISION_PENDING

The coordinator has received all votes and is about to make a decision. This is a temporary state. The coordinator evaluates the votes: if all votes are YES, the decision is COMMIT. Otherwise, the decision is ABORT.

### State 7: COMMIT_DECISION or ABORT_DECISION

The coordinator writes the decision to the transaction log. This is the most important write in the entire protocol. Once the decision is durable, the transaction has a definitive outcome. The coordinator writes the list of participants alongside the decision.

**Failure Point 5**: The coordinator crashes after writing the decision but before sending it to participants. On recovery, the recovery manager reads the decision from the log and resumes sending COMMIT or ABORT messages to all participants.

**Failure Point 6**: The coordinator crashes during the decision write, leaving a partial or corrupted log record. Modern transaction logs use write-ahead logging with checksums to detect and recover from partial writes.

### State 8: COMMIT_SENT or ABORT_SENT

The coordinator sends the decision to each participant. The coordinator records in the log that each participant has been sent the decision.

**Failure Point 7**: A participant is unreachable during decision delivery. The coordinator retries in a background loop. This is called the "commit retry" or "indoubt resolution" process. The coordinator continues retrying until the participant acknowledges or until manual intervention.

### State 9: COMMIT_WAITING or ABORT_WAITING

The coordinator waits for acknowledgments from all participants. Each participant applies the decision (commits or aborts its local changes), releases locks, and sends an acknowledgment.

**Failure Point 8**: A participant applies the commit but crashes before sending the acknowledgment. The coordinator retries the COMMIT message. The participant receives a duplicate COMMIT. This is why idempotency is required -- the participant must handle duplicate COMMIT messages safely.

### State 10: ACK_COLLECTED

The coordinator receives acknowledgments from all participants. Each acknowledgment is recorded in the log.

### State 11: COMPLETED

The coordinator writes a COMPLETED record to the log. The transaction is fully resolved. The coordinator can now release all in-memory state for this transaction.

### State 12: FORGET

The coordinator marks the transaction as eligible for garbage collection. The log records for the transaction may be pruned during a future checkpoint.

## Consistency and State

### Global Consistency Guarantee

The 2PC protocol guarantees that all participants reach the same final state: either all committed or all aborted. This guarantee holds even under failures, assuming the coordinator and participants follow the protocol correctly.

The guarantee is based on the following invariants:

1. **Unanimous Commit**: A commit decision is made only if all participants voted YES.
2. **Durable Decision**: Once the coordinator writes a decision to the log, the decision is final and must be honored even after a crash.
3. **Participant Compliance**: A participant that voted YES must be able to commit when asked. A participant that voted NO must be able to abort.
4. **Idempotent Delivery**: Participants must handle duplicate COMMIT or ABORT messages safely.

### State Transitions During Recovery

When the coordinator recovers from a crash, it reads the transaction log and determines the state of each unresolved transaction:

| Last Log Record | Recovery Action |
|---|---|
| BEGIN | Abort the transaction. Participants never received PREPARE, so no resources are held. |
| PREPARE_SENT | Abort the transaction. The coordinator sent PREPARE but may not have received all votes. Send ABORT to all participants. |
| VOTE_COLLECTED | Abort the transaction. The coordinator received votes but did not write a decision. This is safe because the decision was not made durable. |
| COMMIT_DECISION | Complete the commit. Resume sending COMMIT to all participants until all acknowledge. |
| COMMIT_SENT | Complete the commit. Continue sending COMMIT to participants that have not yet acknowledged. |
| ABORT_DECISION | Complete the abort. Resume sending ABORT to all participants until all acknowledge. |
| ABORT_SENT | Complete the abort. Continue sending ABORT to participants that have not yet acknowledged. |
| COMPLETED | No action needed. The transaction is fully resolved. |

### Participant Recovery

When a participant recovers from a crash, it checks its own transaction log for prepared but unresolved transactions. For each unresolved transaction, the participant queries the coordinator to determine the final decision. If the coordinator is unreachable, the participant holds the locks and resources until the coordinator becomes available.

This is the blocking problem in 2PC: a participant that has prepared but not received a decision cannot unilaterally abort or commit. It must wait for the coordinator. In practice, production systems implement a heuristic resolution: if the participant has been waiting for more than a configurable timeout (typically 60 seconds), it may unilaterally abort, risking inconsistency if the coordinator later decides to commit.

## Performance Characteristics

### Latency Breakdown

The total latency of a 2PC distributed transaction is the sum of:

1. **Application execution time**: The time to execute operations on each participant.
2. **Network round trips**: Three round trips for 2PC (PREPARE, decision, acknowledgment).
3. **Log flush time**: Each participant flushes its prepare record to disk. Each coordinator decision flush is synchronous.
4. **Lock contention**: Time spent waiting for locks held by other transactions.

For a transaction with N participants, the total latency is approximately:

```
T_total = T_app + 3 * RTT + T_log_flush + T_lock_contention
```

Where:
- `T_app` is the time to execute operations on each participant.
- `RTT` is the network round-trip time between coordinator and participants.
- `T_log_flush` is the time to flush log records to disk (typically 2-10ms for SSDs).
- `T_lock_contention` is the time spent waiting for locks.

### Throughput Limits

The maximum throughput of a 2PC coordinator is limited by:

1. **Log write throughput**: The coordinator must write a commit decision to its log synchronously. This limits throughput to the sequential write throughput of the coordinator disk (approximately 50,000-100,000 transactions per second for SSD-backed logs).

2. **Participant capacity**: Each participant must process prepare, commit, or abort operations. The overall throughput is bounded by the slowest participant.

3. **Network bandwidth**: Each transaction requires 3N messages (prepare, decision, ack for each participant). At 1,000 transactions per second with 3 participants, this is 9,000 messages per second.

### Optimizations

**Presumed Abort**: If a transaction record is not found in the log during recovery, the coordinator presumes the transaction was aborted. This avoids writing abort records to the log, reducing log volume.

**Read-Only Optimization**: Participants that only read data during the transaction can vote YES without holding locks or writing prepare records. The coordinator skips these participants in the commit phase.

**One-Phase Commit**: If the coordinator knows there is only one participant, it can skip the prepare phase and go directly to commit. This is a common optimization for transactions that are distributed across multiple databases but only modify data in one.

**Group Commit**: Multiple commit decisions are batched and written to the log as a single group, reducing the number of log flushes per transaction.

Production distributed transaction coordinators expose a set of configuration parameters that affect performance, reliability, and behavior under failure.

## Log Management in Depth

### Write-Ahead Logging (WAL)

The transaction log uses write-ahead logging: the coordinator writes a record to the log before sending any protocol message. This ensures that if the coordinator crashes, the recovery manager can reconstruct the exact state of every transaction from the log.

The WAL protocol enforces two rules:

**Rule 1 (Write Ahead)**: Before sending a PREPARE message to any participant, the coordinator must have written the PREPARE_SENT record to the log. Before sending a COMMIT or ABORT, the coordinator must have written the decision record.

**Rule 2 (Redo)**: During recovery, the recovery manager redoes all COMMIT_DECISION records by resending COMMIT messages to participants. This ensures that committed transactions are fully applied even if the coordinator crashed before delivering the COMMIT.

### Log Structure

The log is organized as a sequence of fixed-size log blocks, typically 4KB or 8KB each. Each block contains a header with:

```
LogBlockHeader {
    BlockNumber: uint64      // Sequential block number
    Checksum: uint32         // CRC32 checksum of block contents
    FirstLSN: uint64         // LSN of the first record in this block
    LastLSN: uint64          // LSN of the last record in this block
    RecordCount: uint16      // Number of records in this block
    BlockSize: uint16        // Size of this block in bytes
}
```

### Log Pruning

Transaction logs grow continuously. Production systems implement a checkpoint mechanism to prune old records:

1. **Periodic Checkpoint**: Every 60 seconds, the coordinator writes a checkpoint record that includes the list of all active (unresolved) transactions.

2. **Log Truncation**: After a checkpoint, log records before the checkpoint can be discarded, provided that all transactions that completed before the checkpoint have been fully resolved.

3. **Archive**: Completed transaction records are archived to long-term storage (S3, GCS) for audit purposes before being pruned from the active log.

### Log Replication

The coordinator replicates the log synchronously across multiple nodes to survive node failures. Common replication strategies include:

**Quorum-based replication**: The coordinator writes the log to N replicas and waits for acknowledgment from a majority (N/2 + 1). This ensures that at least one replica survives a minority failure.

**Leader-based replication**: A leader coordinator receives all log writes and replicates them to followers. If the leader fails, a follower is elected as the new leader. This is the approach used by Raft-based log implementations.

### Log Performance

Log writes are typically the bottleneck in distributed transaction systems. Each log write requires:

1. Serialization of the log record to bytes.
2. Write to the log buffer (in-memory).
3. Flush to disk (fsync) or network acknowledgment for replicated logs.

The fsync latency for modern SSDs is 1-3ms. For replicated logs, the network round trip adds 1-5ms per replica. The total log write latency for a quorum-based replicated log is:

```
T_log = max(fsync_latency, max_replica_ack_latency) * quorum_size
```

For a three-node quorum with 2ms fsync and 3ms replica ack:

```
T_log = max(2ms, 3ms) * 2 = 6ms
```

This means a single coordinator can process approximately 160 log writes per second without batching. With group commit (batching 10 writes), this increases to 1,600 writes per second.

## Concurrency Control in Distributed Transactions

### Lock Manager Architecture

The lock manager maintains a hash table mapping resource IDs to lock queues. Each lock queue contains:

```
LockQueue {
    ResourceID: string
    CurrentOwner: XID           // The transaction currently holding the lock
    LockMode: SHARED | EXCLUSIVE
    WaitQueue: PriorityQueue<XID>  // Transactions waiting for this lock
    DeadlockDetectionTimer: uint64  // Timer for periodic deadlock checks
}
```

### Deadlock Detection

Distributed deadlocks occur when two or more transactions are waiting for locks held by each other across different participants. The coordinator detects deadlocks using a wait-for graph:

1. Each participant maintains a local wait-for graph of transactions waiting for locks.
2. The coordinator periodically collects local wait-for graphs from all participants.
3. The coordinator merges the graphs and checks for cycles.
4. If a cycle is found, the coordinator aborts the youngest transaction in the cycle.

### Concurrency Control Strategies

**Pessimistic Concurrency**: Locks are acquired before operations are executed. This prevents conflicts but reduces concurrency. Suitable for high-contention workloads.

**Optimistic Concurrency**: Operations are executed without locks. At commit time, the system checks for conflicts. If a conflict is detected, the transaction is retried. Suitable for low-contention workloads.

**Multiversion Concurrency Control (MVCC)**: Each operation creates a new version of the data. Transactions see a snapshot of the database at their start time. Conflicts are detected at commit time. MVCC is used by Spanner and many modern databases.
