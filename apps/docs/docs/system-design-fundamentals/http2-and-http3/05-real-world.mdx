## Company Case Studies

### Case Study 1: Google -- Where HTTP/3 Was Born

**Scale:** Google serves billions of search queries, YouTube streams, and Gmail messages daily. In 2024, over 80% of Google's outbound traffic uses QUIC (HTTP/3).

**Problem:** In 2012, Google observed that TCP's head-of-line blocking was the dominant source of latency for search users on mobile networks. A search page loads 15-20 resources (HTML, CSS, JS, images). On a 3G network with 2% packet loss, TCP retransmissions caused visible delays. Google's internal measurements showed that 5% of search queries took over 5 seconds on mobile -- and TCP HOL blocking was responsible for 30% of that delay.

**Solution:** Google designed and deployed gQUIC (Google QUIC) in 2013, years before the IETF standardized HTTP/3. They deployed it between Google's frontend servers and Chrome browsers. Chrome attempted QUIC on every HTTPS connection, and Google's servers responded when they could. By 2015, QUIC handled 30% of all outbound Google traffic.

**Result:** Google published a study in 2017 showing that QUIC reduced the probability of search result rebuffering (video) by 20% on desktop and 30% on mobile. For search page loads, QUIC reduced median latency by 8% and tail latency (p99) by 15% on lossy networks. YouTube found that QUIC reduced rebuffer time by 18-20% on WiFi and 36-40% on cellular connections.

### Case Study 2: Cloudflare -- HTTP/3 at the Global Edge

**Scale:** Cloudflare operates 330+ edge data centers across 120+ countries, proxying traffic for 25% of all websites (2024 estimate). Their edge serves 50+ million HTTP requests per second.

**Problem:** Cloudflare's customers expect fast load times for a global audience ranging from users in New York (fiber, 5ms RTT) to users in rural India (4G, 100ms RTT). A single website might have 50% of users on mobile connections. HTTP/2's TCP HOL blocking disproportionately hurt those mobile users. Additionally, Cloudflare's edge servers needed to terminate client connections efficiently, and TCP's per-connection memory overhead made 200,000+ concurrent connections expensive.

**Solution:** Cloudflare deployed HTTP/3 across its entire edge network. Every edge server terminates QUIC connections from clients and proxies to origin servers using HTTP/2 (internal). They adopted a strategy called "happy eyeballs" for HTTP/3: the client tries both QUIC and TCP simultaneously and uses whichever connects first.

```python
# Cloudflare's happy-eyeballs approach (simplified):
def connect_to_origin(host, port):
  # Try both QUIC and TCP simultaneously
  quic_future = async_connect_quic(host, port)  # UDP
  tcp_future = async_connect_tcp(host, port)  # TCP
  
  # Wait for whichever succeeds first, cancel the other
  winner = await first_completed(quic_future, tcp_future)
  if winner == tcp_future:
  # Cache that this path prefers TCP for next time
  cache_set(f"blocked_udp:{host}", True, ttl=3600)
  return winner
```

**Result:** Cloudflare reported that HTTP/3 reduced connection establishment time by 33% (from 3 RTT to 1-2 RTT) for first-time visitors and to 0-1 RTT for returning visitors. Mobile users in emerging markets saw the largest improvements: 2-3 second page loads dropped by 25% to 35%. Edge server CPU usage increased by 15-20% due to QUIC's userspace processing, but this was offset by reduced per-connection memory (QUIC uses less state than TCP for short-lived connections).

### Case Study 3: Netflix -- Streaming at Scale with HTTP/2

**Scale:** Netflix delivers 15-20 million streaming hours per day across 190+ countries. Their CDN (Open Connect) serves 200+ Gbps per appliance.

**Problem:** Netflix's Open Connect appliances deliver video content via HTTP progressive download. Each streaming session downloads video segments (typically 4-10 seconds each) over HTTPS. With HTTP/1.1, each segment request opened a new TCP connection, causing 2-3 RTT overhead per segment. For a 4-second segment, the connection handshake consumed 1-2 seconds of that budget. With millions of concurrent sessions, the connection churn was enormous.

**Solution:** Netflix deployed HTTP/2 across their Open Connect CDN. Clients (Netflix apps on smart TVs, game consoles, set-top boxes) maintained a single HTTP/2 connection to their nearest Open Connect appliance. All video segment requests were multiplexed onto that connection.

```
HTTP/2 Multiplexing for Video Streaming:
  Client тФА,тФА,тФА"┬╢ Open Connect Appliance (single HTTP/2 conn)
  Stream 1:  GET /video/segment_001.ts (4s of 1080p)
  Stream 3:  GET /video/manifest.mpd  (playlist)
  Stream 5:  GET /audio/segment_001.m4a (audio track)
  Stream 7:  PROXY /logs/bandwidth_metric  (telemetry)

  Headers compressed via HPACK:
  Each segment request has similar headers (same host, same auth)
  After the first request, headers are ~15 bytes (index references)
```

**Result:** HTTP/2 reduced Netflix's CDN connection overhead by 70%. The number of TCP connections per client dropped from 6-8 (HTTP/1.1) to 1 (HTTP/2). This saved TCP port allocation, TLS handshake CPU, and kernel connection tracking resources on the Open Connect appliances. The HPACK header compression reduced header overhead from 500 bytes per request to under 100 bytes -- significant for the millions of telemetry and manifest requests that accompany each streaming session.

**Key lesson:** Even a single-stream workload (video) benefited from HTTP/2 because of the parallel telemetry and manifest requests. The multiplexing overhead was minimal, and the connection reuse saved significant handshake cost.

### Case Study 4: Discord -- When HTTP/2 Hurts More Than Helps

**Scale:** Discord handles 25+ million voice and text users daily. Their API handles 10,000+ requests per second during peak hours.

**Problem:** Discord's real-time API uses HTTP long-polling: the client sends a request, and the server holds the response open (up to 30 seconds) until new data arrives or the request times out. With HTTP/2, each long-poll occupied one stream. With 10,000 concurrent users, that is 10,000 open streams. The HTTP/2 connection between the client and Discord's CDN (Cloudflare) carried these long-lived streams alongside normal API requests.

The problem: a single long-poll stream that stalled (due to a slow client network) blocked the connection's flow control window. Other streams on the same connection could not send data because the connection-level window was consumed by the stalled stream. Discord observed periodic latency spikes where API requests waited 500ms to 2 seconds while a stalled stream's flow control window recovered.

**Root cause:** HTTP/2's connection-level flow control. Even though each stream has its own flow control window, the connection-level window limits total bytes in flight across all streams. A stalled stream with a large pending response can consume the connection window, blocking all other streams.

**Solution:** Discord moved to a different pattern: instead of long-polling over a shared HTTP/2 connection, they used WebSocket for real-time events. WebSocket is a single stream, so the multipathing and flow control issues do not apply. API requests remained on HTTP/2 but were separated from the real-time channel.

```
Before (HTTP/2 long-poll):  After (WebSocket + HTTP/2):
  One connection:  Two connections:
  Stream 1: GET /events (wait)  WebSocket: real-time events
  Stream 3: GET /messages  UDP or TCP, dedicated
  Stream 5: GET /users  No flow control conflict
  (Stalled stream blocks  HTTP/2 connection:
  Streams 3 and 5)  Stream 1: GET /messages
  Stream 3: GET /users
  No long-poll streams
```

**Result:** Separating real-time streams from API requests eliminated the flow control contention. API p99 latency dropped from 800ms to 120ms. The WebSocket connection used a separate TCP socket, so its buffering did not affect the API connection.

**Key lesson:** HTTP/2 multiplexing works best when all streams process at roughly the same speed. Mixing fast API calls with slow long-lived streams causes flow control contention.

### Case Study 5: Fastly -- Priority-Aware HTTP/2 Caching

**Scale:** Fastly's edge cloud delivers traffic for major media and e-commerce sites. Their Varnish-based cache handles millions of requests per second across 90+ POPs.

**Problem:** With HTTP/2, a single user connection multiplexes 50 to 100 streams for a page load. The edge cache must prioritize which streams to serve first. A naive round-robin across streams delivers fragments of each resource, delaying the critical HTML and CSS that the browser needs to render the page.

**Solution:** Fastly implemented priority-based HTTP/2 scheduling in their Varnish cache. They parse the HTTP/2 priority tree and allocate bandwidth proportionally:

- HTML (highest dependency, weight 256): 50% of available bandwidth
- CSS (depends on HTML, weight 128): 25% of available bandwidth
- JavaScript (depends on HTML, weight 64): 12.5% of available bandwidth
- Images (depends on HTML, weight 16): 6.25% of available bandwidth
- Fonts, analytics, etc.: remaining 6.25%

**Result:** Fastly observed that priority-aware HTTP/2 scheduling improved perceived page load time (time to first paint) by 40% compared to fair round-robin scheduling. The key insight: completion time of the critical path matters more than total throughput.

## Open Source Implementations

### Nginx

Nginx HTTP/2 support is mature (since version 1.9.5 in 2015). HTTP/3 support is available since Nginx 1.25.0 (experimental, via quictls fork).

```nginx
# Nginx HTTP/2 configuration
server {
  listen 443 ssl http2;
  http2_max_concurrent_streams 128;
  http2_max_field_size 8k;
  http2_max_requests 1000;
  # No native HTTP/3 yet -- use quictls fork or Cloudflare's patch
}

# Nginx HTTP/3 (via quictls fork)
server {
  # QUIC and HTTP/3
  listen 443 quic reuseport;
  # HTTP/2 as fallback
  listen 443 ssl http2;
  
  ssl_protocols TLSv1.3;
  # Advertise HTTP/3
  add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

**Limitations:** Nginx's HTTP/2 priority engine is basic. It does not fully implement the RFC 7540 priority tree; instead, it uses a simplified round-robin weighted by stream weight. For most use cases this is fine, but it means RFC 9218 (Extensible Priorities) adoption is optional.

HAProxy added HTTP/2 support in version 2.0 (2019). It terminates HTTP/2 from clients and can forward to backends using HTTP/1.1, HTTP/2, or FastCGI.

### Envoy Proxy

Envoy has first-class HTTP/2 (since 1.0) and HTTP/3 (since 1.23) support. It is the most feature-rich open-source HTTP/3 proxy.

**Cost considerations:** HTTP/3 termination costs more CPU at the edge (20-40% more than HTTP/2 due to userspace QUIC processing). For low-throughput services (under 1 Gbps), the extra cost is negligible (under $50/month in compute). For high-throughput services (100+ Gbps), the CPU overhead translates to 10-20% more edge nodes. AWS CloudFront charges the same per-request regardless of protocol. Self-hosted QUIC at scale adds server costs.

**Operations complexity:** HTTP/3 adds debugging complexity. Traditional tools (`tcpdump`, `ss`, `netstat`) show only "UDP" traffic without QUIC frame visibility. You need QUIC-aware tools (`qlog`, Wireshark with QUIC dissector, Chromium's `net-internals`). On-call engineers must be trained on these tools before deploying HTTP/3.

## Lessons Learned

**Lesson 1: HTTP/2's real value is connection reuse, not just multiplexing.**

The biggest performance gain from HTTP/2 comes from eliminating multiple TCP+TLS handshakes. A page that previously opened 20 TCP connections over 60 seconds now uses 1 connection for the entire session. This saves 3-6 round trips per page load. Multiplexing is a secondary benefit.

**Lesson 2: HTTP/3's real value is on lossy networks, not fast ones.**

Do not deploy HTTP/3 expecting improvements on your local network (0% loss, 2ms RTT). On perfect networks, HTTP/2 and HTTP/3 perform identically. The gains come from (a) mobile users with 1-3% packet loss, (b) intercontinental links with 50-200ms RTT, and (c) users behind NAT or middleboxes that fragment TCP connections.

**Lesson 3: Always test with real client traffic before committing.**

HTTP/3 behaves differently across networks. VPNs, enterprise proxies, public WiFi, cellular carriers, and satellite all handle UDP differently. Deploy behind Alt-Svc with short max-age (1 hour initially). Monitor HTTP/3 adoption by network. Some networks that claim "UDP open" actually rate-limit UDP aggressively (below 1 Mbps), making HTTP/3 slower than HTTP/2.

**Lesson 4: Separate fast and slow streams onto different transports.**

Discord's experience shows that mixing real-time (slow, long-lived) streams with synchronous (fast, short-lived) streams on the same HTTP/2 connection creates flow control contention. Separate them: use WebSocket for real-time, HTTP/2 for API, or use different HTTP/2 connections with different flow control settings.

**Lesson 5: Prioritization matters more than parallelism.**

Without priority-aware scheduling, HTTP/2 multiplexing can make pages load slower than HTTP/1.1 with 6 parallel connections. The fix is to implement proper resource prioritization based on the critical rendering path. CDN vendors who built priority-aware schedulers (Fastly, Cloudflare) saw 30-40% improvement in perceived page load time versus naive round-robin scheduling.
