## The Problem Story

Imagine a busy restaurant kitchen on a Saturday night. The chef stands at the pass, tickets pinned to a board in a single row. Each ticket lists several items: a steak, a salad, a glass of wine. The chef picks the first ticket, cooks the steak, plates it, hands it to the server, and only then looks at the next ticket. The salad wilts. The wine sits uncorked. Tables that ordered simple dishes wait behind tables with complex ones. Customers complain about slow service even though the kitchen is working nonstop. The bottleneck is not the cooking -- it is the system for accepting and processing orders.

The first version of HTTP, HTTP/1.1, worked exactly like this chef. It opened one TCP connection per request, processed one request at a time per connection, and queued everything behind the first request in line. A browser loading a modern webpage needed 80 to 120 separate requests for CSS files, JavaScript bundles, fonts, and images. Each request waited for the previous one to finish. Even with multiple parallel connections (browsers opened 6 per domain), the overhead of TCP handshakes, slow-start congestion windows, and head-of-line blocking meant pages took 3 to 5 seconds to load on fast networks and 10 to 15 seconds on mobile connections.

The breaking point came around 2010. Web pages had grown from simple documents into full applications. Gmail, Facebook, and Google Maps delivered dynamic interfaces that required dozens of tiny resources. The HTTP/1.1 model could not keep up. Sites resorted to workarounds: concatenating all CSS into one file, inlining images as base64 data, using CSS sprites -- techniques that fought against the protocol rather than working with it. Latency was not the problem; the protocol's serial nature was.

HTTP/2 and HTTP/3 are the rewrites that fixed this. They replaced the single-file line with a system that processes multiple items concurrently, prioritizes urgent requests, and recovers from failures without blocking everything behind them. The chef now reads all tickets at once, delegates side dishes to different stations, and sends completed items to tables as soon as they are ready.

## The Insight

HTTP/2 and HTTP/3 are transport protocols that allow a single connection to carry multiple simultaneous, interleaved request-response streams without head-of-line blocking.

Think of a single-lane road versus a multi-lane highway. HTTP/1.1 was a single lane: one car at a time, every car behind it waits. If the lead car breaks down, the entire road stops. HTTP/2 built a multi-lane highway where cars can pass each other, but the highway still crosses a single bridge (TCP). If the bridge has a problem, all lanes still stop. HTTP/3 replaced the bridge with a set of independent ferry boats (QUIC): each boat carries its own cargo, takes its own route, and if one boat has a problem, the others continue independently.

## Core Vocabulary

**Multiplexing** -- The ability to send multiple request-response pairs on a single TCP connection simultaneously. In HTTP/1.1, each request waited for the previous one to complete. In HTTP/2, requests are broken into frames and interleaved on the same connection. This eliminates most of the latency from concurrent resource loading.

**Stream** -- An independent, bidirectional sequence of frames exchanged between client and server within a single HTTP/2 connection. Each stream has a unique identifier and carries its own set of HTTP headers and data. Multiple streams are multiplexed onto one connection, meaning 100 streams can share the same TCP socket without waiting for each other.

**Frame** -- The smallest unit of communication in HTTP/2 and HTTP/3. A frame is a binary chunk (not text like HTTP/1.1) with a type, length, flags, and stream identifier. Frame types include HEADERS (request or response headers), DATA (body content), SETTINGS (configuration parameters), and PRIORITY (stream importance hints). Frames from different streams are interleaved on the wire.

**HPACK (HTTP/2)** and **QPACK (HTTP/3)** -- Header compression algorithms. HTTP/1.1 sent headers as plain text on every request, often repeating the same cookie, user-agent, and accept headers. A typical request header was 500 to 800 bytes. HPACK uses a static table (predefined common headers), a dynamic table (headers seen during the connection), and Huffman encoding to reduce header overhead to under 100 bytes. QPACK adapts this for HTTP/3's out-of-order delivery by using separate encoder and decoder streams.

**Server Push** -- A capability where the server sends resources to the client before the client requests them. When the server sends an HTML page, it can push the CSS and JavaScript files it knows the page needs. The client can cancel unwanted pushes. Server Push was designed to eliminate the round-trip of discovering dependencies. (Note: Chrome removed Server Push support in 2022 due to low adoption and better alternatives like 103 Early Hints.)

**Head-of-Line (HOL) Blocking** -- A performance bottleneck where one slow or failed operation blocks subsequent operations in a queue. HTTP/1.1 had HOL blocking at the request level (one request blocks all others on that connection). HTTP/2 eliminated request-level HOL blocking through multiplexing, but TCP-level HOL blocking remained because a lost TCP packet blocks all streams until retransmission. HTTP/3 eliminates transport-level HOL blocking by using QUIC over UDP, where each stream recovers independently.

**QUIC** -- A transport protocol built on UDP that provides reliable, ordered delivery within streams but allows streams to progress independently. QUIC integrates TLS 1.3 encryption into the handshake (0-RTT for returning clients), reduces connection establishment from 2-3 round trips (TCP + TLS) to 0-1 round trips, and supports connection migration when a client changes networks (switching from WiFi to cellular without dropping the connection).

**Prioritization** -- The mechanism by which the client tells the server which resources matter most. HTTP/2 originally used a complex tree-based priority system (each stream had a weight and a dependency on another stream), but it was so complicated that most servers implemented it incorrectly. HTTP/3 simplified this to an Extensible Priorities framework where each request gets a single urgency value (0-7) and an incremental flag.

**Connection Coalescing** -- The ability to reuse a single HTTP/2 or HTTP/3 connection for multiple origins that resolve to the same IP address and share a TLS certificate. If example.com and static.example.com both serve from the same IP and have a wildcard certificate, the client can reuse one connection instead of opening two. This reduces connection overhead by 30% to 50% for sites with multiple subdomains.

## How It Actually Works -- Step by Step

### Step 1: Connection Establishment

Before any HTTP request can be sent, the client and server must establish a transport-layer connection and negotiate the protocol version.

The client discovers protocol support via TLS negotiation. During the TLS handshake, the client includes an ALPN (Application-Layer Protocol Negotiation) extension listing supported protocols: h2 for HTTP/2, h3 for HTTP/3, http/1.1 as fallback. The server picks the highest mutually supported version. For HTTP/3, the client also needs a DNS HTTPS record (SVCB or Alt-Svc header) telling it that the server supports QUIC on a specific UDP port.

### Step 2: Settings Exchange

Once the transport connection is established, both sides send a SETTINGS frame to configure the connection parameters.

```
Client ──── SETTINGS(stream=0) ───────────────▶ Server
  - SETTINGS_MAX_CONCURRENT_STREAMS: 100
  - SETTINGS_INITIAL_WINDOW_SIZE: 65535 bytes
  - SETTINGS_MAX_FRAME_SIZE: 16384 bytes
  - SETTINGS_ENABLE_PUSH: 0 (disabled)

Server ──── SETTINGS(stream=0) ──────────────── Client
  - SETTINGS_MAX_CONCURRENT_STREAMS: 256
  - SETTINGS_INITIAL_WINDOW_SIZE: 1048576 bytes  (1 MiB)
  - SETTINGS_MAX_FRAME_SIZE: 16777215 bytes  (16 MiB max)
  - SETTINGS_MAX_HEADER_LIST_SIZE: 8192 bytes
```

Both sides send these frames immediately after the handshake completes. The WINDOW_UPDATE frame controls flow control at both the connection level (all streams combined) and the stream level (individual streams cannot exceed their window).

The browser needs to load an HTML page plus three CSS files, two JavaScript bundles, and four images. Instead of opening 10 separate connections, it opens one HTTP/2 connection and creates 10 streams.

The client sends HEADERS frames for each stream, tagged with a unique stream ID (odd numbers for client-initiated, even for server-initiated). The server processes the requests and sends back HEADERS and DATA frames, mixing frames from different streams on the same connection. The client reassembles frames by their stream ID.

This multiplexing eliminates the need for HTTP/1.1 workarounds like domain sharding (splitting resources across multiple domains to increase parallel connections) and resource inlining.

### Step 4: Header Compression

Before sending HEADERS frames, HTTP/2 compresses them using HPACK. The compression is stateful: both sides maintain a dynamic table of previously seen header fields.

```
First request headers (before compression):  After HPACK:
  :method: GET          ──▶  Static table index 2
  :scheme: https        ──▶  Static table index 6
  :path: /index.html    ──▶  Not in table ──▶ literal (21 bytes)
  :authority: example.com ──▶  Static table index 1
  accept: text/html     ──▶  Not in table ──▶ literal (10 bytes)
  user-agent: Mozilla/5.0... ──▶  Not in table ──▶ literal (40 bytes)

  Raw size: ~350 bytes  ──▶  After HPACK: ~85 bytes
```

On subsequent requests to the same origin, the dynamic table already contains entries for the path, accept, and user-agent headers, reducing transmission to a few index references.

### Step 5: Prioritization

The client assigns a priority to each stream. In HTTP/2's original model, each stream declares a dependency on another stream plus a weight.

```
Dependency tree for an HTML page with CSS and images:
  ┌──────────────────────────┐
  │  Stream 1: GET /index.htm│  (HTML - weight 256)
  │  :path: /index.html     │
  ├───────┬─────────┬───────┤
  │       │         │       │
  │       ▼         ▼       ▼
  │  ┌────┐  ┌──────┐  ┌─────┐
  │  │Str3│  │Str5  │  │Str7 │
  │  │CSS │  │app.js│  │logo │
  │  │wt:1│  │wt:64 │  │wt:16│
  │  └────┘  └──────┘  └─────┘
```

The server uses this tree to allocate bandwidth: it should send Stream 1 first (the HTML), then allocate twice as much bandwidth to Stream 3 (CSS) as Stream 5 (JS), and four times as much to Stream 5 as Stream 7 (image). In practice, most servers implement a simpler round-robin with these weight hints, and HTTP/3 simplified the model to a single urgency value (0-7).

### Step 6: Flow Control

Each stream and the connection as a whole have flow control windows. The initial window is 65,535 bytes per stream. As the server sends data, it consumes window bytes. When the client processes the data, it sends WINDOW_UPDATE frames to replenish the window.

### Step 7: Packet Loss and Recovery

This is where HTTP/2 and HTTP/3 diverge fundamentally. When a TCP packet is lost on an HTTP/2 connection, all streams on that connection stall until the lost packet is retransmitted and acknowledged. This is TCP HOL blocking.

The receiver reassembles streams independently: if packet 3 is lost but packets 4, 5, and 6 arrive, the receiver delivers data for streams 1, 3, and 7 immediately while buffering stream 5 data until the retransmission arrives.

## Variants and Flavors

### Variant 1: h2c -- HTTP/2 Cleartext (No TLS)

Before the standardization of HTTP/2 over TLS, h2c was proposed as a cleartext variant. The client would use HTTP Upgrade headers or a prior-knowledge flag to indicate h2c support.

```
Client ────────────────▶ Server
  GET / HTTP/1.1
  Upgrade: h2c
  HTTP2-Settings: <base64 settings>

Server ─",─,-- - Client
  HTTP/1.1 101 Switching Protocols
  (then switches to HTTP/2 binary framing)
```

**When to use:** Never in production. Major browsers never implemented h2c due to security concerns (no encryption means no privacy and vulnerability to downgrade attacks). All major CDNs and servers require TLS for HTTP/2. The IETF effectively deprecated h2c by making TLS mandatory in the HTTP/2 specification for browser implementations.

**When not to use:** Any environment where security matters (all production environments). Use h2c only for local development and testing where certificate setup adds friction.

### Variant 2: HTTP/2 over Cleartext TCP (as deployed by proxies)

Internal load balancers and reverse proxies often terminate TLS at the edge and forward traffic as HTTP/2 cleartext inside the datacenter.

```
Client (TLS) ────────────▶ Edge Proxy (HTTP/2 cleartext) ────────────▶ Backend

Client sees:  Internal link:  Backend sees:
  h2 (TLS)  h2c (cleartext)  h2 (as received)
```

**When to use:** Inside trusted network boundaries where encryption overhead matters but multiplexing benefits are still desired. Common in microservice architectures with a service mesh.

**When not to use:** Across untrusted networks or the public internet. The data center bypass saves the TLS cost on internal links, but you lose end-to-end encryption guarantees.

### Variant 3: HTTP/3 Early Deployments (Behind Alt-Svc Advertisement)

HTTP/3 requires UDP port support, which many enterprise firewalls block. Servers deploy HTTP/3 alongside HTTP/2 and advertise it via the Alt-Svc response header.

```
HTTP Response:
  Alt-Svc: h3=":443"; ma=86400, h3-29=":443"; ma=86400

Client behavior:
  1. Attempts HTTP/3 over UDP
  2. If UDP blocked or no response: falls back to HTTP/2
  3. Caches the Alt-Svc directive for 86400 seconds (24 hours)
```

**When to use:** For public-facing web services where the majority of users benefit from reduced latency (especially mobile users and users in regions with packet loss). Cloudflare reports that HTTP/3 adoption, as measured by Alt-Svc advertisement, covers over 30% of the top 10 million websites as of 2024.

**When not to use:** Internal enterprise applications behind strict firewalls that drop UDP. The fallback to HTTP/2 works transparently, but you incur the overhead of maintaining both stacks.

### Variant 4: WebTransport -- Beyond HTTP

WebTransport extends HTTP/3's QUIC-based streams to support unidirectional streams, datagrams (unreliable delivery), and server-initiated streams. It is designed for real-time applications (gaming, streaming, collaboration tools) that need lower latency than WebSocket.

```
WebSocket (over TCP):  WebTransport (over QUIC):
  One bidirectional stream per connection  Multiple unidirectional streams
  Ordered delivery  Out-of-order delivery support
  Emulated by HTTP Upgrade  Native QUIC transport
  50-100ms handshake  0-RTT handshake for returning users
```

**When to use:** Building real-time multiplayer games, live collaborative editing, or video conferencing where sub-50ms latency matters and packet loss is expected.

**When not to use:** Standard REST APIs or CRUD applications where HTTP/2 or HTTP/3 with regular request-response patterns suffice. WebTransport is a specialized tool for low-latency bidirectional use cases.

## Mental Model Check

**Q1:** A video streaming service loads 50 small thumbnail images and 1 large video on its homepage. With HTTP/1.1, the thumbnails load visibly one by one. With HTTP/2, they appear almost simultaneously. What property of HTTP/2 causes this improvement - **A1:** Multiplexing. HTTP/2 sends all 50 thumbnail requests as independent streams on a single connection. The server interleaves the response data, so small partial responses from multiple streams arrive concurrently. In HTTP/1.1, each thumbnail had to complete its download before the next one started (on a single connection), or the browser opened 6 parallel connections and load-balanced across them.

**Q2:** A user on a train with unstable WiFi tries to load a website. HTTP/2 takes 8 seconds. HTTP/3 takes 3 seconds on the same network. What explains the difference - **A2:** Transport-level head-of-line blocking. HTTP/2 runs over TCP, where a single lost packet blocks all active streams until retransmission completes. On a lossy network (1% to 5% packet loss is common on mobile connections), this causes visible stalls for all resources. HTTP/3 runs over QUIC, where each stream recovers independently -- lost packets only affect their specific stream, and other streams continue delivering data.

**Q3:** Your server sends an HTML page with 12 referenced CSS and JavaScript files. Should you enable server push for all 12 - **A3:** No. Server push should be used sparingly. Pushing all 12 resources wastes bandwidth if the client has some in its cache. Chrome removed server push support in 2022 because pushed resources often went unused (cache hits). The better alternative is 103 Early Hints: the server sends a preliminary HTTP status (103) with the Link headers for critical resources, giving the client a head start on discovery without committing to send data the client already has.

**Q4:** You run an API gateway that handles 200,000 requests per minute. Should you switch from HTTP/1.1 to HTTP/2 - **A4:** It depends on your traffic pattern. If clients make many concurrent requests (a web app loading many API endpoints in parallel), HTTP/2's multiplexing reduces connection overhead and improves latency by 15% to 40%. If clients make one long-lived request at a time (a mobile app polling a single endpoint), HTTP/2 offers little benefit and adds complexity -- you may even regress performance if your server's HTTP/2 implementation has poor prioritization or flow control handling.

**Q5:** A colleague argues that QUIC is always faster than TCP because it uses UDP. Is this correct - **A5:** No. QUIC's performance advantage does not come from UDP itself -- UDP is simply a carrier. QUIC performs better because it (a) eliminates transport HOL blocking, (b) reduces handshake overhead with 0-RTT, (c) supports connection migration, and (d) brings congestion control into application space where it can be updated without OS kernel changes. On a perfect network (no loss, no latency variation), HTTP/2 over TCP and HTTP/3 over QUIC perform similarly. QUIC's advantage grows as network conditions degrade, especially with packet loss above 0.1%.
