## Internal Architecture

The internal architecture of HTTP/2 and HTTP/3 reveals how each layer of the protocol stack interacts. The critical difference is where multiplexing and reliability live.

### HTTP/2 Binary Framing Layer

The binary framing layer sits between the HTTP application layer and TCP. It converts HTTP messages (headers and body) into frames for transmission and reassembles received frames into HTTP messages.

```
HTTP Message ─,─,─"¶ Framing Layer ─,─,─"¶ TCP Stream
  ─?s
  ─'─,─,─,─,─´─,─,─,─,-- - ─?s  Frame  ─?s
  ─?s  ─'─,─,─,─,─,─,─¤
  ─?s  ─?s Type:  HEADERS (0x1)  ─?s
  ─?s  ─?s  DATA  (0x0)  ─?s
  ─?s  ─?s  PRIORITY (0x2)  ─?s
  ─?s  ─?s  RST_STREAM(0x3)  ─?s
  ─?s  ─?s  SETTINGS (0x4)  ─?s
  ─?s  ─?s  PING  (0x6)  ─?s
  ─?s  ─?s  GOAWAY  (0x7)  ─?s
  ─?s  ─?s  WINDOW_UPDATE(0x8) ─?s
  ─?s  ───,─,─,─,─,─,─¤
  ─?s  Length: 24-bit payload length
  ─?s  Stream ID: 31-bit identifier
  ───,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─o
```

### HTTP/3 QUIC Layer

HTTP/3 moves the transport logic into the application layer via QUIC. The QUIC layer provides reliability, flow control, and multiplexing, replacing TCP entirely.

```

QUIC Packet Wire Format:
─'─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,-- - ─?s Header Byte ─?s Connection ID ─?s Packet Number ─?s Frames─?s
─"─,─,─,─,─,─,─,─,─,─,─,─,─,─¤ (variable)  ─?s (variable)  ─?s ...  ─?s
```

## Data Structures and Algorithms

### HPACK: Header Compression Table

HPACK uses two tables: a static table of 61 predefined header entries and a dynamic table that grows as headers are exchanged.

```
Static Table (first 10 entries):
  Index ─?s Name  ─?s Value
  1  ─?s :authority  ─?s
  2  ─?s :method  ─?s GET
  3  ─?s :method  ─?s POST
  4  ─?s :path  ─?s /
  5  ─?s :path  ─?s /index.html
  6  ─?s :scheme  ─?s http
  7  ─?s :scheme  ─?s https
  8  ─?s :status  ─?s 200
  9  ─?s :status  ─?s 204
  10  ─?s :status  ─?s 206
  ...61 entries total

Dynamic Table (grows during connection):
  Index 62 ─?s accept  ─?s text/html
  Index 63 ─?s user-agent  ─?s Mozilla/5.0...
  Index 64 ─?s cookie  ─?s session=abc123

Encoding: A header can be:
  - Indexed (single integer): just the table index ─?T 1-2 bytes
  - Literal with indexing:  name + value added to table ─?T the next request references by index
  - Literal without indexing: name + value sent inline ─?T not added (for one-off headers)
  - Huffman encoded: string values optionally Huffman-coded (typically saves 20% to 40%)
```

**Time complexity:** Table lookup is O(1) via array indexing. Header encoding is O(n) where n is the number of header fields. Decoding is O(k) where k is the number of referenced entries.

**Space complexity:** The dynamic table defaults to a maximum of 4,096 bytes. The server can adjust this via SETTINGS_HEADER_TABLE_SIZE. Exceeding the limit evicts entries using LRU (least recently used) policy.

### QPACK: Header Compression for HTTP/3

QPACK adapts HPACK for out-of-order delivery. HPACK assumes ordered delivery (TCP guarantees it), but QUIC streams can deliver data out of order. QPACK solves this by splitting table updates and references into separate streams.

```
QPACK Architecture:
─'─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,-- - ─?s  Encoder Stream (unidirectional, server to client)  ─?s
─?s  ─,─,─"¶ Table Insert (index=62, name=accept, val=html)  ─?s
─?s  ─,─,─"¶ Table Insert (index=63, name=cookie,...)  ─?s
─?s  ─?s
─?s  Decoder Stream (unidirectional, client to server)  ─?s
─?s  ─",─,-- - Table Acknowledged (index=62)  ─?s
─?s  ─",─,-- - Stream Cancelled (stream=7)  ─?s
─?s  ─?s
─?s  Request Stream 1 (bidirectional)  ─?s
─?s  ─,─,─"¶ Header: index=2 (GET)  ─?s
─?s  ─,─,─"¶ Header: index=6 (https)  ─?s
─?s  ─,─,─"¶ Header: literal "/index.html"  ─?s
─?s  ─,─,─"¶ Header: index=62 (accept)  ─?s
─?s  ─,─,─"¶ Header: index=63 (cookie)  ─?s
─?s  [Requires index 62 and 63 to be inserted first]  ─?s
─?s  ─?s
─?s  If Stream 1 arrives before Encoder Stream:  ─?s
─?s  ─?T Encoder inserts at index 62 ─?T Blocked stream  ─?s
─?s  ─?T QPACK has "required insert count" in header block  ─?s
─?s  ─?T Receiver waits until that many inserts arrived  ─?s
───,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─o
```

**Why QPACK:** HPACK's dynamic table requires in-order delivery -- an insert at index 62 must arrive before a reference to index 62. Since QUIC delivers packets out of order, QPACK tracks a "required insert count" in each header block. The receiver delays header processing until the encoder stream has delivered enough insertions. If the encoder stream is lost, the decoder blocks all streams referencing its table entries.

**Time complexity:** Same as HPACK -- O(1) lookups, O(n) encoding. But QPACK adds a potential stall when table entries arrive out of order.

### Stream State Machine

Each HTTP/2 stream progresses through a defined state machine.

```
Stream State Machine:
  ─'─,─,─,─,─,─,─,─,─,─,-- - ─?s  Idle  ─?s  Stream ID assigned, no frames sent yet
  ───,─,─,──,─,─,─,─o
  ─?s Open (client sends HEADERS)
  ─"¼
  ─'─,─,─,─,─,─,─,─,─,─,-- - ─?s  Open  ─?s  Both sides can send HEADERS and DATA frames
  ───,─,─,──,─,─,─,─o
  ─?s Half-Closed (one side sends END_STREAM flag)
  ─"¼
  ─'─,─,─,─,─,─,─,─,─,─,─,─,─,─,-- - ─?s Half-Closed  ─?s  One side done, other may still send
  ─?s (Local/Remote)─?s
  ───,─,─,─,─,──,─,─,─,─,─,─o
  ─?s Both sides send END_STREAM
  ─"¼
  ─'─,─,─,─,─,─,─,─,─,─,-- - ─?s  Closed  ─?s  Stream is finished; cannot send frames
  ───,─,─,─,─,─,─,─,─,─,─o
  ─"²
  ─?s
  ─'─,─,─,─,─,─,─,─,─,─,-- - ─?s Reserved ─?s  Server push: stream ID reserved but
  ─?s (Push)  ─?s  not yet active
  ───,─,─,─,─,─,─,─,─,─,─o
```

**Space complexity:** Each open stream consumes approximately 256 bytes of server memory for its state (flow control window, priority, headers). At 100,000 concurrent streams, that is 25 MB plus header data. Most servers limit concurrent streams to 256 or 1000 to bound memory.

### QUIC Packet Numbering

QUIC uses a monotonically increasing packet number per connection (not per stream). This differs from TCP's per-byte sequence number and enables accurate loss detection without ambiguity from retransmissions.

```
QUIC Packet Number Space:
  ─'─,─,─,─,──,─,─,──,─,─,──,─,─,──,─,─,──,─,─,──,─,─,──,─,─,─,-- - ─?s  P1 ─?s  P2 ─?s  P3 ─?s  P4 ─?s  P5 ─?s  P6 ─?s  P7 ─?s  P8 ─?s
  ───,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─´─,─,─,─,─,─o
  ─?s  ─?s  ─?s  ─?s  ─?s  ─?s  ─?s  ─?s
  ─"¼  ─"¼  ─"¼  ─"¼  ─"¼  ─"¼  ─"¼  ─"¼
  ACK  ACK  LOST  ACK  ACK  ACK  ACK  ACK
  (1)  (2)  (3)  (4)  (5)  (6)  (7)  (8)

  Loss detected when:
  Packet 3 not acked within threshold
  AND packets 4,5,6,7,8 were acked

  Retransmission of P3 data:
  ─"─,-- - Data resent as new packet: P9 (not a retransmission of P3)
  ─"─,-- - P3 is "lost" permanently in the number space
  ───,-- - Receiver uses stream offset (not packet number) to deduplicate

Key difference from TCP:
  TCP retransmits byte range [seq=x, seq=x+len] 
  ─?T Receiver sees duplicate bytes and uses SACK to signal
  QUIC retransmits as a new packet with same stream offset
  ─?T Receiver matches by stream offset, ignores packet number
```

## The Request/Write/Read Lifecycle

### Lifecycle Step 1: Client Resolves Protocol

The browser determines which HTTP version to use. It checks the Alt-Svc cache from a previous response, looks for DNS HTTPS records, and finally checks the ALPN result from the TLS handshake.

```
Client Decision Tree:
  Does Alt-Svc exist for this origin - Yes ─?T Attempt HTTP/3 on UDP port X
  Timeout (200ms)─ ─?T Fall back to HTTP/2 via TLS
  No ─?T Does DNS HTTPS record exist - Yes ─?T Attempt HTTP/3 on UDP port from DNS
  No ─?T Use ALPN during TLS (negotiate h2 or http/1.1)
```

### Lifecycle Step 2: Connection Establishment (HTTP/2)

For HTTP/2, the client opens a TCP connection, performs a TLS handshake with ALPN extension, and both sides send the connection preface.

```
Client:  TCP SYN ─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─"¶ Server
Client: ─",─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,-- - TCP SYN-ACK Server
Client:  TCP ACK ─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─"¶ Server
Client:  TLS ClientHello (ALPN: h2) ─,─,─,─,─,─,─"¶ Server
Client: ─",─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,-- - TLS ServerHello Server
Client:  TLS Finished ─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─"¶ Server
Client:  HTTP/2 Preface (PRI * HTTP/2.0) ─,─"¶ Server
Client:  SETTINGS (max_streams=100) ─,─,─,─,─,─,─"¶ Server
Client: ─",─,─,─,─,─,─,─,-- - SETTINGS (max_streams=256) Server
Client: ─",─,─,─,─,─,─,─,-- - WINDOW_UPDATE (window=1MB) Server
```

The connection preface is mandatory: `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n` followed by an empty SETTINGS frame. The server must receive this before processing any other frames.

### Lifecycle Step 3: Client Creates Streams

The browser parses the HTML and discovers resource dependencies. It creates one stream for each resource. Each stream is assigned the next odd-numbered stream ID.

```
Stream Creation:
  Stream 1: GET /index.html  (highest priority)
  Stream 3: GET /style.css  (depends on stream 1)
  Stream 5: GET /app.js  (depends on stream 1)
  Stream 7: GET /font.woff2  (depends on stream 1)
  Stream 9: GET /logo.png  (depends on stream 1, low priority)
  ...
  Stream 101: GET /analytics.js  (100th stream)

  Each stream sends a HEADERS frame with END_STREAM flag
  (for GET requests with no body)
```

### Lifecycle Step 4: Server Receives HEADERS Frames

The server reads frames from the TCP socket, dispatches each frame to its stream handler, assembles the HTTP request from the compressed headers (using HPACK/ QPACK decoding), and delivers the resulting request to the application handler.

```
Server Frame Processing:
  TCP Socket Buffer
  ─?s
  ─"¼
  HTTP/2 Frame Parser
  ─"─,-- - Stream ID 1 ─?s Type: HEADERS ─?s Payload length: 42
  ─"─,-- - Stream ID 3 ─?s Type: HEADERS ─?s Payload length: 38
  ─"─,-- - Stream ID 5 ─?s Type: HEADERS ─?s Payload length: 45
  ─"─,-- - Stream ID 7 ─?s Type: HEADERS ─?s Payload length: 36
  ───,-- - Stream ID 9 ─?s Type: HEADERS ─?s Payload length: 33

  Stream 1 ─,─,─"¶ HPACK Decoder ─,─,─"¶ GET /index.html
  Stream 3 ─,─,─"¶ HPACK Decoder ─,─,─"¶ GET /style.css
  Stream 5 ─,─,─"¶ HPACK Decoder ─,─,─"¶ GET /app.js
  ...
```

### Lifecycle Step 5: Server Multiplexes Responses

The server application processes requests, possibly in parallel (using worker threads or async IO), and sends responses back as HEADERS and DATA frames. The server interleaves frames from different streams.

```
Server sends interleaved frames:
  [Stream 1 HEADERS]  (HTTP 200 OK, content-type: text/html)
  [Stream 1 DATA]  (first 4096 bytes of HTML)
  [Stream 3 HEADERS]  (HTTP 200 OK, content-type: text/css)
  [Stream 1 DATA]  (next 4096 bytes of HTML)
  [Stream 3 DATA]  (first 4096 bytes of CSS)
  [Stream 5 HEADERS]  (HTTP 200 OK, content-type: text/javascript)
  [Stream 1 DATA]  (final 1024 bytes, END_STREAM)
  [Stream 3 DATA]  (next 4096 bytes of CSS)
  [Stream 5 DATA]  (first 4096 bytes of JS)
  ...
```

The application decides the interleaving order. A well-behaved server uses the stream priority information to decide which DATA frames to send next. A naive server round-robins across all active streams.

### Lifecycle Step 6: Flow Control Enforces Pacing

As DATA frames arrive at the client, the client's receive buffer fills. The client sends WINDOW_UPDATE frames to the server to permit more data.

```
Flow Control Sequence:
  Server Window: 1,048,576 bytes (connection level)
  Server sends:  [Stream 1: 300KB HTML]
  Client receives, forwards to application, sends WU:
  WINDOW_UPDATE(stream=1, increment=300KB)
  ─?T Stream 1 window replenished

  Server sends:  [Stream 3: 50KB CSS]
  Client receives: buffered, window drops to 698,576 bytes

  Client sends:
  WINDOW_UPDATE(connection, increment=350KB)
  ─?T Connection window replenished
```

If the application is slow to consume data, the window shrinks. When the window reaches zero, the server must pause sending on that stream. This backpressure prevents the server from overwhelming the client's memory.

### Lifecycle Step 7: Stream Completion

When the server finishes sending a response, it sets the END_STREAM flag on the final DATA frame. The stream transitions to half-closed (remote), and eventually to closed.

```
Stream Closure:
  Server sends:
  [Stream 1: DATA (4096 bytes, END_STREAM)]
  Client receives, processes response, sends:
  RST_STREAM (stream=1, error_code=0)  ─ or implicit close

  Server receives and frees stream state:
  Stream 1 ─?T Closed
  Stream 1 ID can never be reused in this connection
```

### Lifecycle Step 8: Connection Termination

When the client is done (page navigated, tab closed), it sends a GOAWAY frame. The server stops accepting new streams and drains remaining streams before closing the TCP or QUIC connection.

```
Graceful Shutdown:
  Client ─,─,GOAWAY(last_stream_id=51)─,─,─"¶ Server
  "Stop accepting streams with ID > 51"

  Server ─,─,GOAWAY(last_stream_id=51)─,─,─"¶ Client
  "Acknowledged; current streams will drain"

  Server processes streams 1-51
  Server closes connection after all streams complete
  (or after timeout, whichever comes first)
```

## Consistency and State

### Crash Scenario: Server Restart Mid-Request

What happens when the HTTP/2 server crashes while processing stream 5 (a large file upload):

Time 0: Client sends HEADERS (stream 5, POST /upload)
Time 1: Client sends DATA (stream 5, chunk 1 of 10)
Time 2: Client sends DATA (stream 5, chunk 2 of 10)
Time 3: Server CRASHES ──→ TCP connection drops
Time 4: Client detects connection loss
  All streams 1-7 are lost ──→ no way to continue
  Client must reestablish connection
  Client must retransmit all requests
  If upload was 90% complete, the entire upload is lost

HTTP/3 advantage:
  Connection ID persists across restarts? No.
  QUIC cannot survive a server restart either.
  But QUIC can resume: client sends 0-RTT with original stream data
  Server rejects 0-RTT if state not available ──▶ retry

**Impact:** Any crash on the server loses all in-flight streams. There is no session persistence in HTTP/2 or HTTP/3. The client detects the failure via TCP timeout (typically 30-60 seconds for HTTP/2) or QUIC loss detection (faster, typically 3x RTT).

### Partition Scenario: Asymmetric Routing

A network partition causes packets from the server to the client to be delayed, but client-to-server packets flow normally.

```
Partition Scenario ─,-- - Unidirectional Loss
─'────────────────────────────────────────────────────────,-- - ─?s  ─'─,─,─,─,─,─,─,─,─,─,-- - ─?s
─?s  Client ─,─,─,─,─,─,─,─,─,─,─,─,─,─,─"¶─?s Network  ─?s─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─,─"¶ Server
─?s  (sends requests OK)  ─?s  50%  ─?s (receives requests)  ─?s
─?s  ─?s loss on  ─?s  ─?s
─?s  Client ─",─,────"────"────"────"────"────"─,─,─?s reverse  ─?s─",─,─,─,-- - responses  ─?s
─?s  (cannot receive)  ─?s  path  ─?s (sent but lost)  ─?s
─?s  ───,─,─,─,─,─,─,─,─,─,─o  ─?s
─?s  ─?s
─?s HTTP/2 behavior:  ─?s
─?s  Server sends DATA frames  ─?s
─?s  Client TCP buffer fills with lost packets  ─?s
─?s  Client sends duplicate ACKs  ─?s
─?s  After 3 duplicate ACKs: fast retransmit  ─?s
─?s  After RTO (300ms, then 600ms, 1200ms...):  ─?s
─?s  TCP retransmits, but they're also lost  ─?s
─?s  Eventually TCP gives up (typically 12 retries)  ─?s

─?s  Server sends QUIC packets  ─?s
─?s  Client QUIC layer detects loss by packet gap  ─?s
─?s  Client sends ACK for received packets  ─?s
─?s  Server detects ACK gap for lost packets  ─?s
─?s  Server retransmits only the lost streams  ─?s
─?s  Other streams continue to deliver  ─?s
─?s  After timeout: individual streams fail, not all  ─?s
──────────────────────────────────────────────────────────,─o
```

### Concurrent Writes Scenario: HTTP Pipelining vs. Multiplexing

HTTP/1.1 pipelining was an early attempt at multiplexing that failed due to protocol bugs and server complexity. HTTP/2's multiplexing handles concurrent writes correctly.

```
HTTP/1.1 Pipelining (broken):
  Client sends:
  GET /page1  ─,─,─"¶
  GET /page2  ─,─,─"¶  ─ Pipelined: sent before response 1
  GET /page3  ─,─,─"¶
  Server must respond IN ORDER:
  Response 1 ─",─,-- - (slow - takes 500ms)
  Response 2 ─",─,-- - (fast - ready at 1ms, holds for response 1)
  Response 3 ─",─,-- - (fast - ready at 2ms, holds for responses 1+2)
  Head-of-line blocking: fast responses wait for slow ones

HTTP/2 Multiplexing (correct):
  Client sends:
  GET /page1 (stream 1)
  GET /page2 (stream 3)  ─ Independent request
  GET /page3 (stream 5)  ─ Independent request
  Server sends:
  Response 2 ─",─,-- - stream 3 (1ms, no waiting)
  Response 3 ─",─,-- - stream 5 (2ms, no waiting)
  Response 1 ─",─,-- - stream 1 (500ms, slow but not blocking)
  No head-of-line blocking: responses arrive in any order
```

### Stale Reads Scenario: HTTP Caching and Versioning

HTTP/2 and HTTP/3 do not provide read-after-write consistency across requests. If the client writes data via POST to stream 5 and reads via GET on stream 7, the read may return stale data.

```
Stale Read Scenario:
  Client stream 5:  POST /api/user/123
  Body: {"email": "new@example.com"}
  [Request reaches server at T+0ms]

  Client stream 7:  GET /api/user/123
  [Request reaches server at T+2ms]

  Server processing order (depends on thread scheduling):
  Option A: Write processed at T+1ms, Read at T+2ms
  ─?T Consistent (sees new email)

  Option B: Read processed at T+1ms, Write at T+2ms
  ─?T Stale read (sees old email)

  HTTP/2 guarantees NO ordering across streams.
  The client must handle application-level consistency.
```

**Mitigation:** Use conditional requests (If-None-Match, If-Modified-Since) at the HTTP layer. For strong consistency, send dependent requests on the same stream (serialized), or use application-level version vectors.

### Packet Loss Impact

Packet loss has a nonlinear impact on HTTP/2 performance due to TCP's retransmission behavior. At 2% loss, HTTP/2 throughput can drop by 50% or more because TCP interprets every loss as congestion and reduces its congestion window.

```
Throughput vs. Packet Loss Rate (10MB transfer):
  0% loss:  HTTP/2 = 100 Mbps  HTTP/3 = 100 Mbps
  0.1% loss:  HTTP/2 = 85 Mbps  HTTP/3 = 92 Mbps
  0.5% loss:  HTTP/2 = 55 Mbps  HTTP/3 = 78 Mbps
  1% loss:  HTTP/2 = 32 Mbps  HTTP/3 = 61 Mbps
  2% loss:  HTTP/2 = 15 Mbps  HTTP/3 = 42 Mbps
  5% loss:  HTTP/2 = 3 Mbps  HTTP/3 = 18 Mbps
```

QUIC's advantage at higher loss rates comes from more accurate loss detection (packet-number-based instead of sequence-number-based) and faster recovery (no retransmission ambiguity).

## Configuration and Tuning

### HTTP/2 Server Configuration

**Setting: `SETTINGS_MAX_CONCURRENT_STREAMS`**
- What it controls: Maximum number of active streams the client can open simultaneously.
- Safe default: 256 for general web servers, 100 for API servers.
- When to change: Raise to 1000 for servers handling many small concurrent requests (API gateways). Lower to 100 if each request is memory-intensive.
- What breaks if wrong: Too low ──▶ clients see stream creation failures, request queuing. Too high ──▶ memory exhaustion, OOM risk.

**Setting: `SETTINGS_INITIAL_WINDOW_SIZE`**
- What it controls: Per-stream flow control window (bytes of data the server can send before receiving WINDOW_UPDATE).
- Safe default: 1,048,576 (1 MB) -- up from the spec default of 65,535.
- When to change: Raise to 16 MB for servers delivering large payloads (video, file downloads). Lower to 64 KB for chatty API servers where responses are under 10 KB.
- What breaks if wrong: Too small ──▶ frequent WINDOW_UPDATE overhead, reduced throughput. Too large ──▶ memory pressure, risk of slow consumer overwhelming client.

**Setting: `SETTINGS_MAX_FRAME_SIZE`**
- What it controls: Maximum size of a single DATA frame.
- Safe default: 16,384 (16 KB) -- the HTTP/2 minimum.
- When to change: Raise to 65,535 or 1 MB for servers delivering large payloads (reduces frame overhead from 9 bytes per frame to effectively zero).
- What breaks if wrong: Too large ──▶ receiver must buffer entire frame, increasing latency for other streams. The maximum is 16,777,215 bytes (16 MB - 1).

**Setting: `SETTINGS_MAX_HEADER_LIST_SIZE`**
- What it controls: Maximum total header size the server will accept.
- Safe default: 8,192 bytes (8 KB).
- When to change: Raise to 32 KB or 64 KB if authentication tokens or custom headers are large.
- What breaks if wrong: Too low ──▶ requests with large headers are rejected with HTTP 431. Too high ──▶ vulnerable to HPACK bomb attacks (compressed headers that expand to gigabytes).

### OS-Level Tuning for HTTP/3

HTTP/3 uses UDP, so kernel-level TCP tuning does not apply. Key UDP tuning parameters:

```bash
# Linux kernel UDP buffer tuning
# Increase max UDP buffer size (default: 208 KB)
sysctl -w net.core.rmem_max=26214400  # 25 MB receive
sysctl -w net.core.wmem_max=26214400  # 25 MB send

# Set UDP buffer for QUIC (applies per-socket)
sysctl -w net.ipv4.udp_mem="26214400 32768000 52428800"

# GRO (Generic Receive Offload) for UDP
# Improves QUIC throughput by coalescing packets
ethtool -K eth0 rx-udp-gro-forwarding on

# Increase netdev backlog for high UDP packet rates
sysctl -w net.core.netdev_max_backlog=5000
```

The most impactful setting is `rx-udp-gro-forwarding`. Without it, the kernel delivers each QUIC packet individually to the application, increasing CPU overhead by 30% to 50%. With GRO, the kernel coalesces multiple UDP packets into a single larger receive, dramatically reducing context switches.
