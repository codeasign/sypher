**What breaks at 1,000 users (10 concurrent connections):** Nothing. A single server process handles this trivially. Each connection maintains its HPACK table (~4 KB) and stream state (~256 bytes per stream). Memory usage totals under 10 MB.

**What breaks at 100,000 users (1,000 concurrent connections):** The server process becomes a bottleneck. Each connection in HTTP/2 is long-lived (minutes to hours, unlike HTTP/1.1 where connections open and close per request). The event loop processing 1,000 connections with multiplexed streams must handle frame parsing, HPACK encoding/decoding, flow control management, and stream priority scheduling for each. CPU usage climbs to 80% to 90%. TLS termination overhead grows significantly because each connection has its own TLS session.

**What breaks at 10 million users (100,000 concurrent connections):** Everything. A single server cannot handle 100,000 concurrent TLS connections with HTTP/2 multiplexing. The OS runs out of file descriptors (each TCP connection takes one). Memory exceeds 8 GB (100,000 connections вҖ” f?" 80 KB per connection for QUIC, or 40 KB for HTTP/2). The event loop cannot schedule 100,000 concurrent streams fairly -- some connections starve. TLS handshake cost for new connections overwhelms the CPU (up to 5,000 handshakes per second per core). The single server model is not viable beyond a few thousand connections.

### Key Architectural Decisions

**1. Terminate TLS at the edge.** Edge proxies handle HTTPS termination and protocol negotiation. Internal connections use cleartext HTTP/2 (h2c) or gRPC. This reduces TLS overhead on backend servers by 60% to 80% and allows centralized certificate management.

**2. Use HTTP/2 or gRPC between tiers.** Backend connections reuse a small pool of persistent HTTP/2 or gRPC connections (10-50 per server pair). Connection overhead is negligible. The edge proxy maintains long-lived client connections (potentially thousands), but the backend sees only the multiplexed streams routed to it.

**3. Separate connection state from request processing.** The edge server holds connection state (HPACK tables, flow control windows), while request processing happens on stateless application servers. When an edge server fails, connections drop, but in-flight requests can be retried on another edge node.

**4. Graceful shutdown via GOAWAY.** When a server needs to restart, it sends GOAWAY with the last processed stream ID, stops accepting new streams, drains remaining streams, and closes the connection. This enables zero-downtime deployments.

### Replication Strategy

HTTP/2 and HTTP/3 connections are inherently tied to a single server. Connection state is not replicated across regions. The global architecture handles this with:

- **Anycast IP routing:** Clients connect to the nearest edge server. All regions advertise the same IP via BGP. Routing changes happen at the network layer.
- **Request-based routing (not connection-based):** The edge proxy extracts the request and forwards to the appropriate regional backend. The client's connection stays at the edge; the request goes to the data center with the needed data.
- **QUIC connection migration:** If a mobile client moves from the US to Europe, the QUIC connection ID allows packets to follow the client. In practice, the edge proxy reassigns the connection to the nearest edge within seconds.

### Cross-Region Latency

| Path | Round-Trip Time | Bandwidth | Notes |
|---|---|---|---|
| Client вҶ’ Local Edge | 5-30ms | 50-100 Mbps | Fixed wireless, fiber |
| US East вҶ’ US West | 60-70ms | 10-40 Gbps | Cross-country fiber |
| US East вҶ’ EU West | 70-90ms | 10-40 Gbps | Transatlantic fiber |
| US East вҶ’ Asia | 150-200ms | 5-20 Gbps | Submarine cables |
| US East вҶ’ South America | 100-140ms | 5-15 Gbps | Terrestrial + submarine |

### Failover

When a region loses connectivity, the anycast BGP route withdraws, and clients are routed to the next nearest region. QUIC connections are dropped, but the HTTP/3 clients reconnect to the new region with a 0-RTT handshake, losing only in-flight requests.

```
Failover Timeline:
  T+0s:  US-East edge detects datacenter failure
  T+0s:  US-East BGP route withdrawn
  T+5s:  BGP converges (global routing tables update)
  T+5s:  Clients in US-East time out on QUIC
  QUIC client reconnects to EU-West (next nearest)
  0-RTT handshake: 1-2 round trips saved
  T+6s:  Retry in-flight requests in EU-West region
  T+8s:  Service fully restored for all users
```

## Deployment and Operations

### Zero-Downtime Deployment

HTTP/2 and HTTP/3 make zero-downtime deployments more complex because connections are long-lived. A server cannot simply drain all connections immediately.

```
Rolling deployment procedure:
  1. Health check: Verify the new node is healthy
  2. Add to load balancer pool
  3. Old node sends GOAWAY(last_stream_id=current_max)
  "No new streams. Existing streams will drain."
  4. Old node waits for all streams to complete
  (or MAX_DRAIN_TIMEOUT, typically 30-60 seconds)
  5. If timeout expires: RST_STREAM remaining streams
  6. Close old node's connections
  7. Remove old node from pool

  Total drain time: 30-60 seconds per node
  Rolling 8 edge nodes: 4-8 minutes total
```

### Debugging Common Issues

**Issue: Client connections fail with GOAWAY immediately.**
Root cause: The server cannot accept new connections -- either at the process limit (too many file descriptors) or at the connection limit (configured `max_connections`).
Check: `ulimit -n` on Linux, `netstat -an | grep 443 | wc -l`.

**Issue: High latency but low CPU usage.**
Root cause: HPACK dynamic table too small, causing header compression inefficiency. Each request sends full headers instead of table references.
Check: HPACK table miss rate. Increase `SETTINGS_HEADER_TABLE_SIZE` from 4,096 to 16,384 or 32,768.

**Issue: Streams timeout frequently.**
Root cause: Backend services are slow, causing stream queue buildup. The server accepts streams faster than it can process them.
Check: `SETTINGS_MAX_CONCURRENT_STREAMS` is too high. Reduce from 256 to 100.

**Issue: Some streams stall while others complete.**
Root cause: A single slow backend response blocks other streams due to flow control exhaustion (connection-level window exhausted by one stream).
Check: Connection-level WINDOW_UPDATE frequency. Raise connection window from 1 MB to 16 MB.

**Issue: HTTP/3 connections fail on certain networks.**
Root cause: UDP traffic is blocked by firewalls, NAT, or carrier-grade NAT devices. HTTP/3 falls back to HTTP/2, but the fallback detection (200ms timeout) adds latency.
Check: Verify Alt-Svc header is served. Confirm UDP port 443 reaches the server from multiple networks.

### TLS Certificate Management

HTTP/2 (with TLS required) and HTTP/3 (with TLS built into QUIC) depend on valid certificates. Deploy a centralized certificate management system (cert-manager, ACME protocol).

```
Certificate renewal process:
  1. cert-manager detects certificate expires in 30 days
  2. ACME challenge (HTTP-01 or DNS-01)
  3. New certificate issued, stored in secret store
  4. Edge servers reload certificates (zero-downtime)
  5. Old certificate revoked or allowed to expire

  HTTP/3 consideration:
  QUIC uses TLS 1.3 at the transport layer
  Certificate validation is identical to HTTPS
  But: some QUIC implementations require the full
  certificate chain in the transport handshake
  (no separate TLS layer)
  Ensure your CA chain is complete and compact
```
