## The Core Tension

The fundamental trade-off in HTTP/2 and HTTP/3 is **connection reuse versus connection isolation**.

HTTP/2 aggressively reuses a single TCP connection for all streams. This eliminates connection overhead, reduces TLS handshake costs, and enables multiplexing. But it creates a single point of failure: one lost TCP packet blocks every stream on that connection. The same reuse that makes HTTP/2 fast on good networks makes it fragile on lossy ones.

HTTP/3 solves the TCP HOL blocking problem by using QUIC over UDP, where each stream recovers independently. But QUIC introduces its own trade-offs: UDP is often blocked by enterprise firewalls, QUIC implementations are newer and less battle-tested, and the userspace transport stack consumes more CPU than kernel TCP.

The choice between them is not about which is "better"  it is about which failure mode you can tolerate.

## Decision Matrix

| Decision | Option A | Option B | Choose A When | Choose B When |
|---|---|---|---|---|
| Protocol version | HTTP/2 | HTTP/3 | Network has UDP blocking; enterprise VPN users; need mature tooling | Mobile users with packet loss; global audience; latency-sensitive apps |
| TLS termination | Edge (CDN) | Backend server | Centralized cert management; multiple services share domain | Compliance requires end-to-end encryption; internal-only service |
| Connection pool size | Large (1000+) | Small (10-100) | Many concurrent requests per user; chatty APIs | Small payloads; streaming video; long-lived connections |
| HPACK table size | 4 KB (default) | 16-64 KB | Low memory environment; header-heavy APIs | Most requests share similar headers; cookie-heavy sessions |
| Priority scheme | HTTP/2 tree (RFC 7540) | Extensible Priorities (RFC 9218) | Existing HTTP/2 implementation; no upgrade planned | New deployment; need simple, correct prioritization |
| Server push | Enabled | Disabled | Users never cache; first-time visits (deprecated in Chrome) | Most users return; bandwidth matters; cache hit rate over 30% |
| Alt-Svc max-age | Short (3600s) | Long (86400s) | Testing HTTP/3 rollout; canary deployments | Stable HTTP/3 service; want minimal fallback latency |

HTTP/2 load balancers are AP systems: they favor availability during a partition. Each LB node independently decides backend health based on its own connectivity. ## Failure Modes

### Failure Mode 1: TCP HOL Blocking on Lossy Networks

**What happens:** A mobile user on a train experiences 2% to 5% packet loss. Their single HTTP/2 connection carries 30 multiplexed streams for a page load. A single lost TCP packet stalls all 30 streams until TCP retransmits and acknowledges the packet. Each retransmission takes one RTT (80ms on 4G). With 2% loss on 30 packets, the probability of at least one packet being lost is 1 - (0.98)^30 = 45%. The user sees a partially loaded page for 80-240ms while retransmissions complete.

**Root cause:** TCP's in-order delivery guarantee combined with multiplexed streams. TCP delivers bytes in order to the HTTP/2 layer. A gap in the byte stream (from a lost packet) stalls delivery of all subsequent bytes, even if those bytes belong to a different stream.

**Blast radius:** All streams on the affected connection. A mobile app with 100 users on the same tower experiences this simultaneously.

**Detection:** Monitor TCP retransmission rate. When retransmission rate exceeds 1%, measure HTTP/2 stream completion times. If completion times correlate with retransmission spikes, HOL blocking is active.

**Recovery:** Migrate to HTTP/3. If HTTP/3 is not an option, increase the number of TCP connections (undoing the multiplexing benefit) or use techniques like domain sharding to spread streams across multiple connections.

### Failure Mode 2: UDP Blocking by Firewalls

**What happens:** An enterprise user behind a corporate firewall tries to load a website served over HTTP/3. The firewall drops all UDP traffic except DNS (port 53). The client sends QUIC Initial packets to port 443. They are silently dropped. The client waits for the QUIC timeout (typically 200ms), then falls back to HTTP/2 over TCP. The page load takes 200ms longer than necessary, and if the fallback is not cached, every page load suffers this delay.

**Root cause:** Enterprise and hotel networks commonly block UDP to prevent DNS tunneling, QUIC-based VPNs, and other UDP-based circumvention tools. Some firewalls also prohibit UDP because they cannot inspect encrypted QUIC traffic for DPI (deep packet inspection) policies.

**Blast radius:** All users behind the blocking firewall. A company with 10,000 employees on a corporate network all experience the 200ms fallback penalty.

**Detection:** Monitor HTTP/3 connection failure rate by ASN or IP range. A sudden drop in HTTP/3 adoption from a specific network range indicates UDP blocking.

**Recovery:** Lower the QUIC fallback timeout from the default 200ms to 100ms for known-blocking networks. Use DNS HTTPS records (SVCB) to signal HTTP/3 availability rather than relying on Alt-Svc-based discovery. Cache the fallback decision per origin (avoid the 200ms penalty on every page load).

### Failure Mode 3: HPACK State Desynchronization

**What happens:** An HTTP/2 proxy between client and server modifies response headers (adding security headers, rewriting cookies). The proxy modifies headers without updating the HPACK dynamic table state. The client's HPACK decoder has a different table than the proxy's encoder. The next compressed header reference points to a table entry that does not exist or contains different data. The connection fails with a COMPRESSION_ERROR.

**Root cause:** HPACK relies on synchronized dynamic tables between encoder and decoder. Any intermediary that modifies headers must either (a) use literal encoding (no table indexing) for modified headers, or (b) maintain its own HPACK state separate from the upstream. Many proxies and middleware do neither correctly.

**Blast radius:** The specific HTTP/2 connection. All streams on that connection fail simultaneously. The client must open a new connection, losing all in-flight requests.

**Detection:** Monitor HTTP/2 COMPRESSION_ERROR frame counts. These should be zero in normal operation.

**Recovery:** Fix the proxy to use literal header encoding for modified headers. Set `SETTINGS_HEADER_TABLE_SIZE` to 0 on the proxy to disable the dynamic table (forces literal encoding for all headers, at the cost of increased bandwidth).

### Failure Mode 4: Memory Exhaustion from Slow Consumers

**What happens:** A server handles 1000 HTTP/2 connections, each with 100 streams. One client has a very slow network (high latency, small window), so its flow control window stays small. The server has already buffered 2 MB of response data for that client's streams, waiting for WINDOW_UPDATE frames. Across 1000 connections with a few slow consumers each, buffer memory reaches 500 MB. Other clients share the same process, so their streams also slow down as the server throttles memory.

**Root cause:** HTTP/2 flow control permits the server to send data up to the connection-level window without waiting for individual stream windows. If the server prefetches or generates responses eagerly, it can fill buffers for slow consumers while fast consumers are starved for attention.

**Blast radius:** The entire server process. Memory pressure affects garbage collection (if JVM-based) or malloc performance. Other connections experience latency spikes as the server struggles to manage fragmented memory.

**Detection:** Monitor per-connection buffer sizes. Alert when any connection's buffered data exceeds 10 MB, or when total buffered data exceeds 50% of heap/available memory.

**Recovery:** Implement per-stream flow control with tighter limits. Use backpressure in the application layer: do not generate response data until the stream window has available space. Set `SETTINGS_INITIAL_WINDOW_SIZE` to a lower value (256 KB instead of 1 MB) to bound buffer memory.

### Failure Mode 5: Connection Migration Churn

**What happens:** A user walks through a building with multiple WiFi access points. Their phone switches between access points every 60 to 90 seconds. Each switch triggers a QUIC connection migration: the client sends packets from a new IP/port but with the same connection ID. The server revalidates the path (sends PATH_CHALLENGE frames). If the migration completes faster than the old path times out, the connection survives. But if the migrations happen faster than the server can process PATH_RESPONSE frames, the connection drops.

**Root cause:** QUIC connection migration was designed for network changes (WiFi to cellular), not for rapid roaming between APs. The path validation sequence (PATH_CHALLENGE ??T PATH_RESPONSE) takes at least 1 RTT on the new path. Rapid roaming produces a burst of migrations that overwhelm the path validation logic.

**Blast radius:** The individual user's connection. All streams on the connection fail and must be retried.

**Detection:** Monitor QUIC connection migration events per client. Alert on more than 5 migrations per minute.

**Recovery:** Increase the server's path validation timeout to handle back-to-back migrations. Implement server-side connection migration coalescing: if a second migration arrives before the first completes, drop the first and validate only the newest path.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: 100+ Concurrent Streams for a Single Page

**The mistake:** Creating one stream per resource on a page that loads 200 tiny images. The client opens 200 streams simultaneously.

**Why it was reasonable:** HTTP/2 supports multiplexing, so more streams should mean more parallelism, right - **Production impact:** Each stream consumes server memory (state, flow control window, header table reference). 200 streams — f?" 256 bytes = 51 KB per connection. With 10,000 connections, that is 500 MB of stream state. But the real problem is priority scheduling: the server round-robins across 200 streams, delivering tiny fragments of each. TCP segments are small. The client reassembles 200 partial responses simultaneously. Completion time for any single stream increases because the server spreads its bandwidth thinly.

**Correct pattern:** Limit concurrent streams to 6-10 per page load. Use a prioritization scheme that delivers critical resources first (HTML, CSS, visible images). Defer low-priority resources (analytics, below-the-fold images) to fewer streams.

### Anti-Pattern 2: Disabling HPACK Dynamic Table

**The mistake:** Setting `SETTINGS_HEADER_TABLE_SIZE` to 0 to "reduce memory usage."

**Why it was reasonable:** The dynamic table consumes memory per-connection. With 200,000 connections, even 4 KB per connection is 800 MB. Disabling it saves memory.

**Production impact:** Every request sends full uncompressed headers. A 600-byte header set repeats on every request. For a site serving 100 resources per page load, that is 60 KB of uncompressed headers per page. Compared to 8 KB with HPACK, the bandwidth overhead is 52 KB per page load. At 10 million page loads per day, that is 520 GB of additional bandwidth daily.

**Correct pattern:** Keep the dynamic table at 4 KB minimum. For memory-constrained environments, reduce `SETTINGS_MAX_CONCURRENT_STREAMS` instead  it has a larger impact on memory and affects fewer users.

### Anti-Pattern 3: Pushing Everything via Server Push

**The mistake:** Enabling server push for all resources referenced in an HTML page. Every HTML response triggers push of 20 CSS, JS, font, and image files.

**Why it was reasonable:** Server push was designed to eliminate the round-trip for discovering dependencies. If the server knows the page needs style.css, why wait for the browser to ask - **Production impact:** Most resources are already in the browser's cache. Server push sends them anyway, wasting bandwidth. Chrome telemetry showed that only 1% to 3% of server-pushed resources were actually used  the rest were discarded due to cache hits. This wasted 10-30 KB per page load across the Internet. Chrome removed server push in 2022.

**Correct pattern:** Use 103 Early Hints instead of server push. Send a preliminary HTTP 103 status with `Link` headers for critical CSS and JS. The browser pre-connects and starts requesting these resources immediately, but the server does not send data until the browser asks. This provides the same latency benefit without the cache waste.

### Anti-Pattern 4: HTTP/3 Without Fallback Testing

**The mistake:** Deploying HTTP/3 in production without testing on the networks where users actually connect. Assuming "QUIC works everywhere UDP works."

**Why it was reasonable:** QUIC is standardized, Cloudflare and Google have deployed it widely. It should work on most networks.

**Production impact:** Corporate VPNs, hotel WiFi, and some cellular carriers block or rate-limit UDP. Users on these networks see 200ms delays for every QUIC fallback. Some networks even accept QUIC Initial packets but drop subsequent handshake packets, causing connections to appear to hang for 30 seconds before the application-layer timeout triggers.

**Correct pattern:** Deploy HTTP/3 alongside HTTP/2 with Alt-Svc advertisement. Monitor HTTP/3 connection rates by network. Set a conservative Alt-Svc max-age (1 hour) during rollout, increasing to 24 hours once backbone stability is confirmed. Test on at least 10 distinct network types (corporate, edu, public WiFi, 4G, 5G, satellite, VPN) before declaring HTTP/3 ready.

### Anti-Pattern 5: Ignoring the 0-RTT Replay Problem

**The mistake:** Celebrating 0-RTT handshakes without understanding replay risks. Enabling 0-RTT data for all endpoints, including payment and authentication APIs.

**Why it was reasonable:** 0-RTT saves a full round trip on repeat visits. Every millisecond matters.

**Production impact:** A 0-RTT request can be replayed by an attacker who intercepts the encrypted client Initial packet. If the request is a POST to `/api/charge`, the attacker can replay the charge request multiple times. The server sees each replay as a legitimate request because 0-RTT data is not forward-secure  it is encrypted with the previous session's key.

**Correct pattern:** Only enable 0-RTT for idempotent, safe requests (GET, HEAD). Use application-level idempotency keys for POST/PUT/PATCH requests, even in 0-RTT. Set a maximum 0-RTT window of 10 seconds. Never use 0-RTT for payment processing, authentication, or any state-changing operation without idempotency protection.

### Anti-Pattern 6: Assuming QUIC Is Just "TCP but Faster"

**The mistake:** Replacing a TCP-based load balancer with a QUIC-based one without changing any other infrastructure, expecting immediate performance gains.

**Why it was reasonable:** QUIC has better loss recovery and 0-RTT. Upgrading should improve performance.

**Production impact:** QUIC's userspace implementation is 2x to 3x more CPU-intensive than kernel TCP for the same throughput. The load balancer's CPUs spike to 90% utilization. Even worse, the monitoring stack does not understand QUIC  packet captures show only "UDP payload" with no way to decode QUIC frames. Debugging a connection reset becomes impossible without QUIC-aware tools.

**Correct pattern:** When adopting QUIC, also adopt QUIC-aware tooling (wireshark with QUIC dissector, qlog, QUIC load testing tools). Budget 2x the CPU for QUIC processing at the edge. Maintain TCP fallback. Keep the old monitoring stack running alongside the new QUIC-specific metrics.

## When NOT to Use This

### Over-Engineering Signals

You should consider whether HTTP/2 or HTTP/3 is the right choice if you observe any of these signals:

**"Our API has 5 endpoints and serves 1,000 requests per minute."** HTTP/1.1 with keep-alive handles this perfectly. HTTP/2's multiplexing adds complexity (HPACK state, stream management, flow control) for no benefit. The overhead of supporting both HTTP/1.1 and HTTP/2 in your server library adds testing burden.

**"All our clients are mobile apps that make one request at a time."** Mobile apps typically open a single persistent connection and pipeline requests sequentially. HTTP/2's multiplexing offers no advantage  there is only one stream at a time. HTTP/3's 0-RTT helps with reconnection after the app goes to background, but the benefit is marginal (one RTT saved once per session).

**"Our workload is streaming video to 4K TVs."** Video streaming uses a single large response per connection. Multiplexing is irrelevant. HTTP/2's priority scheduling may even interfere with video streaming if other concurrent streams are prioritized higher. Use plain TCP or QUIC for the video transport and HTTP only for manifest files.

**"We are behind a corporate firewall that does deep packet inspection."** Many enterprise security tools cannot inspect encrypted QUIC traffic. They may block UDP entirely or degrade QUIC performance. HTTP/2 over TCP is a safer choice for enterprise-deployed applications.

### Simpler Alternatives

| Scale | Recommended Protocol | Reasoning |
|---|---|---|
| Under 100 req/s | HTTP/1.1 with keep-alive | Minimal complexity, broadest compatibility |
| 100-10,000 req/s, web app | HTTP/2 | Mature implementation, good tooling, vast majority of CDNs support it |
| 10,000+ req/s, global audience | HTTP/3 | Packet loss on long paths becomes significant; connection migration helps mobile users |
| Microservice internal comms | gRPC (HTTP/2) | gRPC's protobuf encoding and bidirectional streaming are stronger motivators than multiplexing |
| Real-time gaming, streaming | WebTransport over QUIC | Direct QUIC access with unidirectional streams and datagrams |

### Scale Thresholds to Watch

- **10 million requests/day (116 req/s):** HTTP/1.1 with 6 parallel connections is sufficient. Upgrade to HTTP/2 for marginal latency improvement (10-15%).
- **100 million requests/day (1,160 req/s):** HTTP/2 becomes worthwhile. Connection overhead of HTTP/1.1 starts to strain server resources. 50-100 persistent connections vs. 1 multiplexed connection.
- **1 billion requests/day (11,600 req/s):** HTTP/3 starts to show measurable advantage, especially if your audience is global. Packet loss on intercontinental links (0.5% to 2%) causes significant HOL blocking on HTTP/2.
- **10 billion requests/day (116,000 req/s):** HTTP/3 is essential. The 0-RTT handshake saves millions of round trips per day. Connection migration reduces mobile reconnection overhead. QUIC's loss recovery improves throughput by 30% to 50% on lossy paths.
