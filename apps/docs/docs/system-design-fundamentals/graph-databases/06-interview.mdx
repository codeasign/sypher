# Graph Databases — Interview Playbook

## How This Topic Appears

Graph database questions show up in system design interviews for roles involving social features, recommendations, fraud detection, or knowledge graphs. Interviewers at FAANG use this topic to test your ability to choose the right data model for connected data.

### Social Network Questions

- "Design a social graph for Facebook. How would you store and query friend-of-friend connections?"
- "Design LinkedIn's People You May Know feature."
- "How would you implement a news feed that shows posts from friends of friends?"

### Recommendation Questions

- "Design a recommendation system for Netflix content using a graph database."
- "How would you build a 'frequently bought together' feature for Amazon?"
- "Design a graph-based content discovery system for TikTok."

### Fraud Detection Questions

- "Design a fraud detection system that uses graph relationships to find fraudulent accounts."
- "How would you detect a coordinated bot network using a graph database?"
- "Design a transaction graph for real-time fraud scoring."

### Knowledge Graph Questions

- "Design Google's Knowledge Graph for entity search."
- "How would you model a music knowledge graph connecting artists, albums, songs, and listeners?"
- "Design a graph database schema for a movie recommendation engine."

### Architecture Questions

- "Compare graph databases to relational databases for storing social relationships."
- "How would you scale a graph database to 1 billion users?"
- "What happens to graph queries when the data does not fit in memory?"

## What the Interviewer Is Actually Testing

Interviewers are not testing whether you know the Cypher syntax. They are testing four things:

**Can you recognize connected data?** — The first step is identifying that the problem involves relationships that matter. Many candidates reach for a relational database by default. The interviewer wants to see that you notice when the data is fundamentally a graph.

**Can you explain index-free adjacency?** — This is the core technical insight. Describe how nodes store physical pointers to their neighbors. Explain why this makes multi-hop traversals fast. Show that you understand the trade-off between write amplification and read efficiency.

**Can you reason about scale?** — How does the graph behave at 100 MB versus 100 GB versus 10 TB? What happens when the working set does not fit in memory? How do you handle supernodes? The interviewer wants to see that you have thought about the failure modes before they happen.

**Can you design a hybrid architecture?** — The strongest candidates do not propose using only a graph database. They describe how a graph database fits alongside relational databases, caches, search engines, and ML models. They understand that production systems are never one database.

## Level-by-Level Expectations

### L3/L4 (Early Career)

**What is expected**: Define nodes, edges, and properties. Compare graph databases to relational databases at a high level. Write a simple Cypher query. Identify that a social network is well-suited to a graph database.

**Sample answer**: "A graph database stores entities as nodes and relationships as edges. Each node can have properties like name and age. Each edge has a type like FOLLOWS or REVIEWS. To find friends of Alice, the database starts at Alice's node and follows her FOLLOWS edges to Bob, Carol, and Diana. This is faster than SQL JOINs because the connections are stored as pointers, not computed at query time."

### L5 (Senior Engineer)

**What is expected**: Explain index-free adjacency in detail. Discuss write amplification and its impact on throughput. Describe supernode problems and mitigation strategies. Compare CAP theorem trade-offs for graph databases. Reference real-world examples (Facebook TAO, LinkedIn Liquid).

**Sample answer**: "In a graph database, each relationship is stored as a physical pointer from one node to another. When Alice follows Bob, the database adds a relationship record that links directly from Alice's node to Bob's. The cost is 6 writes per relationship because both adjacency lists need to be updated. But reads become O(degree) instead of O(log N). Facebook TAO is the best example of this principle applied at internet scale — they cache adjacency lists and serve 2 billion reads per second by treating the social graph as an adjacency lookup service, not a traversal engine."

### L6 (Staff Engineer)

**What is expected**: Design a multi-region graph database deployment. Discuss trade-offs between different graph database models (LPG vs RDF). Design a hybrid architecture combining graph, relational, and cache tiers. Analyze cost-performance trade-offs at billion-node scale. Propose solutions for supernode handling in a distributed context.

**Sample answer**: "For a global social graph with 1 billion users, I would design a three-tier architecture. Tier 1 is a graph-optimized cache layer following Facebook TAO's pattern: cache adjacency lists in memory, serve reads from cache, fall back to persistent storage. Tier 2 is the graph database itself, sharded by node ID range with a hash ring. Tier 3 is the source of truth, which could be a relational database (as in TAO) or a distributed key-value store. Cross-region replication is async — writes go to the local leader and propagate to other regions with 100-500ms lag. For supernodes like celebrities, I would split them into sub-nodes by follower ID range and route queries to the relevant sub-node. The key insight is that at Facebook scale, you cannot traverse the graph at query time. You must precompute, cache, and serve adjacency lists."

### L7/L8 (Principal / Distinguished Engineer)

**What is expected**: Define the graph database strategy for the entire organization. Propose new graph database features or optimizations. Design a graph query language or execution engine. Evaluate build-vs-buy for graph infrastructure. Set technical direction spanning multiple teams.

**Sample answer**: "I would build a unified graph infrastructure that serves both OLTP traversals and OLAP graph analytics. The OLTP path uses index-free adjacency with a TAO-like cache layer for sub-millisecond reads. The OLAP path uses a separate bulk-processing layer that materializes the graph into columnar format for PageRank, community detection, and graph neural network training. Both paths read from the same node and edge stores but use different engines optimized for their access patterns. The infrastructure must support online schema evolution — adding new edge types and node labels without downtime. I would standardize on a single query language across both paths, with a cost-based optimizer that routes queries to the appropriate engine. The key principle is separation of concerns: fast path for point queries, batch path for analytics, with the storage layer as the unifying abstraction."

## FAANG Framework for Answering

Use this 6-step framework when a graph database question comes up in an interview.

**Step 1: Identify the graph.** State clearly: "This problem involves connected data. The entities are X and the relationships between them are Y." This shows you recognize the graph-shaped nature of the problem.

**Step 2: State the graph model.** "I would use a property graph model with labeled nodes and typed, directed edges. For example, users are nodes with labels like `:User`, and friendships are edges with type `:FRIENDS_WITH`."

**Step 3: Show the adjacency advantage.** "To find friends of friends, the database starts at the user's node and follows edge pointers. This is index-free adjacency — the cost is proportional to the number of edges traversed, not the total graph size. A query that takes 2.8 seconds with SQL JOINs at depth 3 takes under 50ms with a graph database."

**Step 4: Address scale.** "At small scale (under 100 million edges), a single graph database instance works. At moderate scale (100 million to 10 billion edges), I would shard by node ID and add read replicas. At extreme scale (10 billion+ edges), I would use a TAO-like approach: cache adjacency lists and precompute popular traversals."

**Step 5: Discuss trade-offs.** "The trade-off is write amplification. Each relationship requires 6 write operations to maintain adjacency pointers. This limits write throughput to about 5,000-10,000 writes per second per node. For read-heavy workloads with deep traversals, this is a great trade. For write-heavy workloads, it is a poor fit."

**Step 6: Propose a complete architecture.** "The full system includes: an application tier, a graph database for traversals, a caching layer (Redis or TAO-style) for hot paths, a relational database for non-graph data, and a monitoring system for traversal depth and page cache hit rate."

## Phrase Bank

Use these phrases to sound confident and knowledgeable during your interview.

| Instead of This | Say This |
|---|---|
| "It stores connections" | "Index-free adjacency gives us O(1) neighbor lookups" |
| "It is fast" | "Traversal cost is O(degree) — proportional to the number of edges from the starting node" |
| "It has tables" | "The node store is a fixed-size direct-addressed array — node ID to file offset is a multiplication" |
| "It uses pointers" | "Each node record stores the file offset of its first relationship record, forming a doubly-linked list" |
| "It is slow for writes" | "Write amplification ratio is 6:1 because we update both the source and target adjacency lists" |
| "A celebrity causes problems" | "Supernodes require threshold pruning, sub-node splitting, or eager subgraph precomputation" |
| "It works with other databases" | "I would use a CQRS pattern with the graph database as a read-optimized projection of the primary datastore" |
| "It can scale" | "Horizontal scaling requires careful sharding because cross-shard traversals lose index-free adjacency" |
| "It recovers from failure" | "The WAL replay reconstructs doubly-linked relationship chains from the transaction log" |
| "It has caching" | "I would add an adjacency list cache following Facebook TAO's design: cache the full edge set per node ID" |
| "It does recommendations" | "I would precompute candidate sets in batch and serve from the graph with real-time personalization scoring" |
| "The graph is big" | "If the working set exceeds available memory, page cache miss rate increases and traversal latency degrades by 10-100x" |
| "It handles formats" | "The property graph model uses labeled nodes with key-value properties; edges also carry properties" |
| "It fails sometimes" | "Cascading replica lag from write amplification creates stale reads on followers that compound over time" |

## Red Flags

### What NOT to Do in a Graph Database Interview

**Red Flag 1: Proposing a graph database for everything.** "I would use Neo4j for all the data." This shows lack of judgment. Graph databases are not general-purpose databases. You should always propose a hybrid architecture.

**Red Flag 2: Ignoring write throughput.** "The graph database handles writes like any other database." This shows you have not thought about write amplification. Acknowledge the 6:1 write amplification and explain how you would batch writes or add write capacity.

**Red Flag 3: Saying "it just scales."** No database scales infinitely without architecture. Explain your sharding strategy, your replication factor, and what happens when you add nodes.

**Red Flag 4: Forgetting about supernodes.** Every large graph has supernodes. If you do not mention them, the interviewer will ask about them and you will look unprepared.

**Red Flag 5: Writing incorrect Cypher.** If you choose to write a Cypher query, get it right. Know the difference between `-[:FOLLOWS]->` (single edge) and `-[:FOLLOWS*2]->` (two hops). A syntax error costs credibility.

**Red Flag 6: Not knowing the CAP trade-off.** Graph databases fall on the CP side of the CAP theorem. If a partition happens, you lose availability on the minority side. Claiming "eventual consistency" without explaining the trade-off looks naive.

**Red Flag 7: Avoiding numbers.** "The graph is big" means nothing. Say: "With 100 million nodes and 1 billion edges, the store is approximately 50 GB for edges and 1.5 GB for nodes. With properties, call it 100 GB total. This fits in memory on a single large instance."

**Red Flag 8: Not knowing the alternatives.** If you propose a graph database, be ready to compare it to relational, document, and key-value databases for the same use case. "Why not PostgreSQL with a join table?" is the most common interviewer follow-up.

## Practice Drills

### Drill 1: Social Graph Design (5 minutes)

Design a graph database for a social network with 500 million users. Focus on: node and edge schema, traversal patterns (friends, friends of friends, followers), supernode handling for celebrities with 50 million followers, and read/write throughput estimates.

### Drill 2: Fraud Detection Graph (5 minutes)

Design a fraud detection system that connects accounts to devices, IP addresses, payment methods, and shipping addresses. How would a traversal detect a fraud ring? What edges would you create? How do you score suspicious patterns?

### Drill 3: Recommendation Graph (10 minutes)

Design a content recommendation graph for a streaming service with 200 million subscribers and 50,000 titles. Model the graph schema. Describe traversal patterns for "because you watched X." How do you combine graph traversal with ML-based scoring?

### Drill 4: Global Graph Scaling (10 minutes)

You have a graph database deployed in US-East. You need to expand to EU-West and AP-Southeast. Design the multi-region architecture. Address: replication lag, consistency model, write conflicts, failover, and cache warming in new regions.

### Drill 5: Graph vs Relational Decision (5 minutes)

An engineer on your team wants to store a user's address history (past 10 addresses with dates) in a graph database. Addresses are never queried by relationship — only by user ID. The engineer argues that "graph databases are more flexible." How do you respond? Design a better approach.
