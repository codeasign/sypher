# Graph Databases — Challenges

## Tier 1: Foundation

Each challenge should take approximately 5 minutes. Attempt the answer before reading the model solution.

### Challenge 1: Write a Cypher Query

**Problem**: You have a graph with `(:Person)` nodes and `(:FOLLOWS)` edges. Write a Cypher query that finds all friends of friends of a person named "Alice" (excluding Alice herself and her direct friends). Limit the results to friends who are older than 25.

**Model Answer**:

```
MATCH (alice:Person {name: "Alice"})-[:FOLLOWS]->(direct:Person)-[:FOLLOWS]->(fof:Person)
WHERE direct <> alice
  AND fof <> alice
  AND NOT (alice)-[:FOLLOWS]->(fof)
  AND fof.age > 25
RETURN DISTINCT fof.name, fof.age
```

The key points: using `<>` to exclude Alice and her direct friends, the `NOT` pattern to filter out already-known connections, and property filtering with `fof.age > 25`. The `DISTINCT` is necessary because there could be multiple paths to the same friend of a friend.

### Challenge 2: Compare Storage Requirements

**Problem**: You have a social network with 10 million users and an average of 200 followers per user. Each user has a profile with an average of 500 bytes of properties. Calculate the total storage required for nodes and edges. Assume node records are 15 bytes and edge records are 50 bytes.

**Model Answer**:

| Component | Calculation | Size |
|---|---|---|
| Nodes | 10,000,000 * 15 bytes | 150 MB |
| Node properties | 10,000,000 * 500 bytes | 5,000 MB (5 GB) |
| Edges | 10,000,000 * 200 edges * 50 bytes | 100,000 MB (100 GB) |
| Edge properties (assume 20% have 100B avg) | 400,000,000 * 100 bytes | 40,000 MB (40 GB) |
| Indexes (30% overhead on node properties) | 5,150 MB * 0.30 | 1,545 MB (1.5 GB) |
| **Total** | | **~147 GB** |

This fits on a single large instance with 256 GB RAM. If the graph grows to 100 million users, the total would be approximately 1.4 TB, requiring either a cluster or a disk-backed configuration with carefully managed page cache.

### Challenge 3: Identify the Supernode

**Problem**: A traversal on your graph suddenly takes 45 seconds instead of its usual 20ms. The query finds "people who might know" candidates by looking at friends of friends of friends (depth 3). One user in particular causes the problem. How do you identify the supernode? What do you check first?

**Model Answer**:

First, check the degree (number of relationships) of the starting nodes that trigger the slow query. The supernode will have a degree far above average (average is 200, supernode has millions). Run:

```
MATCH (p:Person)
WHERE size((p)-[:FOLLOWS]-()) > 10000
RETURN p.name, size((p)-[:FOLLOWS]-()) AS degree
ORDER BY degree DESC LIMIT 10
```

If the highest-degree user has 10+ million followers, that user is your supernode. The fastest fix is to add a threshold to your traversal: stop expanding through nodes with more than 10,000 edges. The longer-term fix involves modifying your data model: split the celebrity into sub-nodes (by follower ID range, by geography, by language) and route queries to the appropriate sub-node.

## Tier 2: Design

Each challenge should take approximately 15 minutes.

### Challenge 4: Design a Fraud Detection Graph

**Problem**: Design a graph database schema for detecting fraudulent account clusters on a payment platform. Accounts exhibit fraud by creating multiple accounts from the same device, using the same payment method across accounts, and sending money in circular patterns.

**Provide**:
1. A complete node and edge schema with labels and properties
2. The Cypher query that detects a circular money transfer pattern
3. A scoring mechanism for suspicious account clusters

**Model Answer**:

**Graph schema:**

Nodes: `(:Account {id, created_at, kyc_level, status})`, `(:Device {id, device_fingerprint, ip_address, user_agent})`, `(:PaymentMethod {id, type, last_four, bin, issuer})`, `(:Transaction {id, amount, timestamp, status})`, `(:IPAddress {ip, isp, country, asn})`.

Edges: `(:Account)-[:USES_DEVICE {first_seen, count}]->(:Device)`, `(:Account)-[:HAS_PAYMENT {added_at, verified}]->(:PaymentMethod)`, `(:Account)-[:SENT {amount, timestamp}]->(:Account)`, `(:Transaction)-[:FROM_ACCOUNT]->(:Account)`, `(:Transaction)-[:TO_ACCOUNT]->(:Account)`.

**Circular transfer detection:**

```
MATCH (a:Account)-[:SENT]->(b:Account)-[:SENT]->(c:Account)-[:SENT]->(a)
WHERE a.id < b.id AND b.id < c.id
RETURN a.id, b.id, c.id,
       SUM(r1.amount + r2.amount + r3.amount) AS total_cycled
ORDER BY total_cycled DESC LIMIT 100
```

**Scoring mechanism**: Each cluster gets a risk score = (shared_device_count * 30) + (shared_payment_count * 25) + (circular_transfer_count * 20) + (average_age_in_days < 7? 15 : 0) + (kyc_level == "none"? 10 : 0). Clusters scoring above 50 are flagged for manual review.

### Challenge 5: Design a Recommendation Graph

**Problem**: Design a graph database for a music streaming service with 100 million users and 50 million songs. The recommendation engine needs to find songs listened to by users who share similar listening history (collaborative filtering via graph traversal).

**Provide**:
1. Schema design with node labels, edge types, and properties
2. The traversal pattern for "users who listened to this song also listened to..."
3. A strategy for handling a 50 million song catalog in a 32 GB memory budget

**Model Answer**:

**Schema**: `(:User {id, region, joined_date, genre_affinity})`, `(:Song {id, title, artist, album, genre, duration, popularity})`, `(:Artist {id, name, genre})`, `(:Album {id, title, year})`.

Edges: `(:User)-[:LISTENED {count, last_listened, skip_rate}]->(:Song)`, `(:Song)-[:BY_ARTIST]->(:Artist)`, `(:Song)-[:ON_ALBUM]->(:Album)`, `(:Artist)-[:PERFORMS]->(:Song)`.

**Traversal pattern**: Find users who listened to the same songs as the target user, then find what they listened to that the target user has not heard.

```
MATCH (me:User {id: "user_42"})-[:LISTENED]->(s:Song)<-[:LISTENED]-(similar:User)
WHERE similar <> me
MATCH (similar)-[:LISTENED]->(rec:Song)
WHERE NOT (me)-[:LISTENED]->(rec)
RETURN rec.id, rec.title, COUNT(DISTINCT similar) AS popularity
ORDER BY popularity DESC LIMIT 20
```

**Memory strategy**: With 50 million songs, you cannot fit the full graph in 32 GB. Solution: partition the graph by genre. The traversal stays within the target user's primary genre. Cache the most popular 1 million songs in memory (approximately 5 GB for properties and edges). Store the rest on disk. Page cache hit rate for traversals stays above 95% because hot songs are in memory.

### Challenge 6: Design a Multi-Region Graph

**Problem**: Your social network has grown to 500 million users globally. You need to deploy graph database clusters in US-East, EU-West, and AP-Southeast. Users in each region must see local data with under 50ms read latency. Cross-region consistency can tolerate up to 5 seconds of staleness.

**Provide**:
1. The replication topology
2. How writes are routed
3. What happens when the US-East region experiences a full outage

**Model Answer**:

**Topology**: Each region has a local graph database cluster with 1 leader and 2 followers. Cross-region async replication connects the leaders in a ring topology: US-East replicates to EU-West, EU-West to AP-Southeast, AP-Southeast to US-East. This avoids a single point of failure in the replication pipeline.

**Write routing**: Users are assigned a home region based on their profile location. Writes always go to the home region's leader. Read queries use the local replica. A user traveling to a different region can read their data from the local replica (up to 5 seconds stale) but writes are forwarded to the home region with acknowledgment to the user (200ms write latency penalty for remote writes).

**US-East outage**: The EU-West leader detects the outage (no heartbeat from US-East for 10 seconds). EU-West and AP-Southeast continue serving their local reads normally. Writes destined for US-East users are queued at the edge (the routing layer holds them). After 30 seconds, a failover is triggered: the EU-West leader takes over as the write target for US-East users. US-East followers are not promoted because they are also unavailable. When US-East recovers, the replication backlog is replayed from EU-West, and writes are migrated back to avoid persistent latency penalties for US-East users.

## Tier 3: Expert

Each challenge should take approximately 30 minutes. These do not have cut-and-dried answers — they require you to reason through trade-offs and justify your design decisions.

### Challenge 7: Design a Graph Query Engine from Scratch

**Problem**: You are building a new graph database from scratch. You need to design the traversal engine that executes pattern-matching queries. The engine must support variable-length path queries like `(a)-[:FOLLOWS*1..5]->(b)` — find all paths between 1 and 5 hops.

**Discussion points**:

What traversal algorithm do you use for variable-length paths? The naive approach is BFS from the start node, but BFS exhaustively visits all nodes at each depth. For a node with degree 200 and depth 5, BFS visits approximately 200^5 = 320 trillion nodes — impossible. Solutions include: bidirectional BFS (start from both ends and meet in the middle), or pruning by node label and property filters at each step.

How do you handle cycles? A graph with cycles means the same node can be reached through multiple paths. The engine must track visited nodes to avoid infinite loops. The standard approach is a visited set per query — if you encounter a node already in the current path, skip it (you are not building Hamiltonian paths, you are finding distinct paths).

How do you order the expansion? The order of expanding edges matters for performance. If the query has filters, you should apply them as early as possible to prune the search space. Filter pushdown means evaluating `WHERE b.age > 25` while traversing, not after collecting all candidates. Cost-based optimization estimates the selectivity of each filter to determine the optimal expansion order.

How do you parallelize? A single traversal can be parallelized by dispatching neighbor expansions to worker threads. Each worker walks a subtree of the path space. The challenge is work-stealing — one worker might have a path that quickly reaches the target while another gets stuck in a dense subgraph. Work-stealing queues balance the load.

### Challenge 8: Migrate 1 Billion Edges from Relational to Graph

**Problem**: Your company uses PostgreSQL for a social network with 500 million users and 2 billion edges stored in a `user_edges` join table. You need to migrate to a graph database (Neo4j) with zero downtime and under 10 minutes of data staleness during migration.

**Discussion points**:

How do you export 2 billion edges from PostgreSQL? A naive `SELECT * FROM user_edges` reads the entire table, which locks the table for hours. Use batched cursor-based reads: `SELECT * FROM user_edges WHERE id > ? ORDER BY id LIMIT 100000`. This allows concurrent writes during export.

How do you import into Neo4j? Use the Neo4j batch importer (`neo4j-admin import`) which bypasses the transaction log and writes store files directly. It can import 2 billion edges in under 4 hours on a machine with fast SSDs. During import, the Neo4j instance is offline.

How do you achieve zero downtime? Run a dual-write phase first: for 1 week, write new edges to both PostgreSQL and Neo4j. Use a change data capture (CDC) pipeline to catch up Neo4j with PostgreSQL. After the batch import is complete and the CDC pipeline has caught up (under 10 minutes lag), switch reads from PostgreSQL to Neo4j. Old PostgreSQL reads continue during the catch-up phase for safety.

How do you verify correctness? Run parallel queries on both databases: sample 100,000 random users and compare their friend lists. Any discrepancy means the migration has a bug. Run this verification hourly during the dual-write phase.

How do you rollback? If Neo4j has issues, switch reads back to PostgreSQL. The dual-write phase means PostgreSQL has all the data. The rollback is instant — just change the read path. The dual-write phase should run for at least 1 week to catch any long-tail issues with the graph database.

This migration pattern has been used successfully by multiple companies. The key principle: never trust a migration you cannot roll back. The dual-write phase is your insurance policy.
