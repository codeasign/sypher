# Graph Databases at Scale

## Naive Implementation

A naive graph database stores each node and edge as a JSON document in a document store, with edges referenced by IDs. When you need to traverse, you fetch the start node, extract edge IDs, and query the edge collection for each one.

### What Breaks at Each Scale

**At 1,000 users**: The naive approach works fine. traversals complete in under 200ms. The document store handles the load. No one notices the N+1 queries because everything fits in memory.

**At 100,000 users**: Exponential blowup becomes visible. A depth-4 traversal generates tens of thousands of individual queries. Response times cross 5 seconds. The document store starts showing high connection counts. Developers add caching, which helps for popular nodes but misses on long-tail data.

**At 10 million users**: The naive approach is unusable. A single depth-4 traversal generates millions of queries — enough to overwhelm the database connection pool and crash the application. Caching helps only for the hottest 1% of nodes. The system effectively cannot answer any multi-hop query. This is the point where engineering teams decide to migrate to a real graph database.

### Components Explained

**Load Balancer** — Distributes queries across query routers. Terminates TLS. Performs health checks. Target: 50,000 QPS per load balancer instance.

**Query Router** — Parses and routes queries to the appropriate graph engine. Read queries go to followers. Write queries go to the leader. Algorithms (PageRank, community detection) are routed to dedicated worker pools. Query routers are stateless and scale horizontally.

**Graph Engine** — The core database process. Each engine holds a portion of the graph in memory and on disk. Leader handles writes and replicates to followers via Raft consensus. Followers serve read queries. Sync replicas acknowledge every write; async replicas lag by up to 200ms.

**Storage Backend** — Stores node records, edge records, and properties. In Neo4j Enterprise, this is a local filesystem with Raft replication. In JanusGraph, this can be Cassandra, HBase, or Bigtable. Amazon Neptune uses a shared storage volume replicated across three Availability Zones.

## Integration Patterns

### Pattern 1: Graph as Read-Projection (CQRS)

Use a graph database as a read-optimized projection of a primary relational or event-sourced database. Write to PostgreSQL, stream changes to a graph database for connected queries.

### Pattern 2: Graph as Dedicated Service

Encapsulate the graph database behind a microservice. Only the service talks to the graph database. Other services call its API.

### Pattern 3: Graph + Cache

Place Redis in front of the graph database for hot paths. Common traversals are cached by key: `friendsof:{user_id}`. Cache hit avoids traversal entirely. Set TTL based on data volatility.

### Pattern 4: Hybrid Relational-Graph

Store tabular data (user profiles, billing info) in PostgreSQL. Store relationship data (social graph, product recommendations) in a graph database. Application code queries both and merges results.

### Consistency Across Regions

Cross-region reads are **eventually consistent**. A write in US-East may take 100-500ms to appear in AP-Southeast-1. For use cases like fraud detection, this is acceptable. For social features like "friends list," users in different regions may briefly see different data. Applications must tolerate this — or route all writes for a user's data shard to a single region.

### Failover Strategy

When a leader fails, the cluster promotes the most up-to-date follower. Promotion takes 10-30 seconds during which writes are rejected. Read traffic continues on remaining followers. Clients must retry on connection failure with exponential backoff.

## Capacity Planning

### QPS Calculation

Estimate query throughput for a social feed feature. Each request requires: one `MATCH` to find the user, one traversal to get friends (depth 1, average 150 edges), and one projection.

| Component | Cost per Query | Queries/Second | Total |
|---|---|---|---|
| Node lookup | 15us | 20,000 | 0.3s CPU |
| Edge traversal (150 edges) | 300us | 20,000 | 6.0s CPU |
| Property reads (150 nodes) | 225us | 20,000 | 4.5s CPU |
| **Total CPU per core** | **540us** | **20,000** | **10.8s (10.8 cores)** |

For 20,000 QPS, you need approximately 11 dedicated cores for the graph engine. With 3x redundancy, provision 33 cores across 4 nodes.

### Storage Sizing

| Component | Per Entity | 100M Users / 1B Edges | 1B Users / 10B Edges |
|---|---|---|---|
| Node records | 15 bytes | 1.5 GB | 15 GB |
| Edge records | 50 bytes | 50 GB | 500 GB |
| Properties (avg 200B/node) | 200 bytes | 20 GB | 200 GB |
| Indexes | 30% overhead | 21 GB | 214 GB |
| **Total** | | **92.5 GB** | **929 GB** |

### Bandwidth Sizing

A traversal that reads 150 nodes at 200 bytes each reads 30 KB per query. At 20,000 QPS, the graph engine reads 600 MB/s from storage. This requires network bandwidth of at least 5 Gbps per node if storage is remote (EBS, network-attached).

## Deployment and Operations

### Zero-Downtime Deployment

Graph databases require careful upgrade procedures because schema changes can affect the entire graph. The pattern: add a new replica running the new version, verify it catches up, redirect traffic, decommission old replicas.

### Monitoring

| Metric | Warning | Critical | Action |
|---|---|---|---|
| Page cache hit rate | below 95% | below 90% | Add memory or reduce graph size |
| Checkpoint duration | over 500ms | over 2s | Reduce write load or increase checkpoint interval |
| Transaction log size | over 1 GB | over 10 GB | Run checkpoint or increase log space |
| Average traversal depth | over 4 | over 6 | Add indexes or restructure queries |
| Connection count | over 200 | over 500 | Add query router instances |

### Debugging Slow Queries

Use `EXPLAIN` and `PROFILE` in Cypher to see the query plan. Common issues: the database starts from a label scan instead of an index seek; the traversal expands edges of the wrong type; a supernode is the bottleneck. Apply schema indexes on label+property combinations used in `MATCH` clause WHERE filters.

### Upgrades

Major graph database upgrades involve re-encoding the store files. This can take hours for large graphs. The safe approach: spin up a new cluster with the new version, take a snapshot, restore to the new cluster, verify data integrity, and cut over traffic.
