# Graph Databases — Trade-offs & Failure Modes

## Core Tension: Write Cost vs Read Speed

The fundamental trade-off in graph databases is between **write amplification** and **read efficiency**. Every relationship you create requires updating multiple pointer chains in the storage engine. Creating a single edge can require 6 disk writes. Compare this to a relational database, where inserting a row into a join table is a single append.

In exchange for this write cost, you get **constant-time adjacency lookups**. A multi-hop traversal that would require 4 JOINs in SQL completes in microseconds in a graph database — because the physical pointers already exist.

This is not a trade-off you can optimize away. It is the defining characteristic of the graph database model. Choose graph databases when:

- Your workload is read-heavy with deep traversals
- Your data is highly connected (avg degree above 10)
- Write throughput is under 10,000 writes per second

Avoid graph databases when:

- Your workload is write-heavy (logging, event ingestion, time-series)
- Your data has few connections per entity
- Most queries are flat aggregations and range scans

## Decision Matrix

| Decision Point | Graph Database | Relational Database | Document Database |
|---|---|---|---|
| Connected queries (depth 3+) | 12ms | 2.8s | 45s |
| Simple lookups by ID | 200us | 100us | 80us |
| Range scan (all users age 30-40) | 800ms | 200ms | 300ms |
| Write throughput (single node) | 5,000 writes/s | 50,000 writes/s | 80,000 writes/s |
| Storage efficiency (bytes/node) | 15 + edges | Minimal | Full document |
| Complex aggregations (SUM, AVG) | Poor | Excellent | Good |
| Schema flexibility | Labels only | Schema required | Schema-optional |
| Horizontal scaling | Complex | Mature | Mature |

## CAP Theorem Positioning

In a distributed graph database, the CAP trade-off is acute. Maintaining index-free adjacency across partitions is fundamentally hard. Most graph databases choose **consistency over availability**: if a partition occurs, some nodes become unavailable rather than serve stale or conflicting data.

## Failure Modes

### Failure 1: The Supernova Supernode

A node with 10 million edges becomes the target of a traversal. Every query that passes through this node tries to visit 10 million neighbors. The traversal engine either times out (30s default) or exhausts memory.

### Failure 2: Unbounded Traversal

A query without a depth limit runs until it exhausts resources. `MATCH (a)-[*]->(b)` matches all paths of any length between all pairs of nodes in the graph — a factorial explosion. Without `MAX_DEPTH` or LIMIT, this query will crash the database.

### Failure 3: Cross-Partition Relationship Torn

In a distributed graph, nodes A and B are on different partitions. A transaction creates an edge between them. If the network fails mid-transaction, Node A has the edge, but Node B does not. The edge becomes a dangling pointer. On recovery, a reconciliation process must detect and remove the half-created edge.

### Failure 4: Cascading Replica Lag

A write-heavy application sends 50,000 edge creations per second to the leader. The leader replicates asynchronously to followers. Followers cannot keep up — each write requires updating pointer chains on both the source and target node stores. Replica lag grows from 10ms to 30 seconds over 5 minutes. Read queries on followers return increasingly stale data. The fix: throttle writes or add more replica capacity.

### Failure 5: Memory Exhaustion from Large Results

A graph algorithm like PageRank materializes intermediate results in memory. A graph with 100 million nodes running PageRank needs to store float scores for every node — 800 MB just for the scores. If the algorithm also stores edge weights and intermediate state, memory usage can exceed the heap limit (default 512 MB), causing OutOfMemoryError and JVM crash.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Graph Database for Log Data

A team stores all application logs as nodes with edges to the server they came from. Logs are high-volume (100,000/second), append-only, and almost never queried by relationship. The graph database struggles with write throughput. The 6 writes per edge create a write bottleneck. The logs have no meaningful edges. The team should have used Elasticsearch or a time-series database.

### Anti-Pattern 2: Everything Is a Node

Modeling every checkbox and dropdown as a node in the graph. The graph becomes a dense web of trivial entities with low information density per edge. Queries that should be simple lookups require traversing through 5 meaningless intermediate nodes. The rule: if an entity has fewer than 3 relationships, it probably should not be its own node.

### Anti-Pattern 3: No Labels, No Indexes

All nodes have the same generic label (`:Entity`). All edges have type `:RELATED_TO`. The query planner cannot estimate cardinality. Every query degenerates into a full graph scan. Indexes are not created for common property filters. A query for `{name: "Alice"}` scans 10 million nodes because there is no label+property index.

### Anti-Pattern 4: Using a Graph Database as a Key-Value Store

Storing session data as nodes with no edges. The graph database provides zero benefit over Redis or DynamoDB, but costs 10x more per GB of storage and has higher read latency. If your data has no relationships, do not use a graph database.

### Anti-Pattern 5: Deep Traversals Without Limits

A dashboard query uses `MATCH (start)-[*]->(end) RETURN path` with no depth limit on a graph with 1 billion edges. The query never completes. It locks resources, starves other queries, and eventually times out after 30 minutes. Every traversal must have a bounded depth.

## When NOT to Use This

Do not use a graph database when:

- **You are ingesting time-series data.** Metrics, logs, and event streams are tabular and append-only. Use InfluxDB, TimescaleDB, or Elasticsearch.

- **Your schema is purely tabular.** If the most complex query is `SELECT * FROM orders WHERE user_id = ?`, you do not need a graph. Use PostgreSQL.

- **You need full-text search.** Graph databases have weak text search. Use Elasticsearch for search and connect it to a graph database for relationship queries. Do not combine them into one.

- **Your team has no graph experience.** Graph databases require different query patterns, different indexing strategies, and different operations procedures. The learning curve is steep. Start with a relational database and add a graph only when the pain is undeniable.

- **Your writes exceed 50,000 per second.** The write amplification of maintaining adjacency lists makes high-throughput ingestion difficult. Consider a specialized ingestion pipeline that batches writes and feeds them to the graph database in periodic bulk loads.
