## Data Structures and Algorithms

### The Node Store

Nodes are stored in fixed-size records. Neo4j uses 15-byte node records. The fixed size means the database can compute the file offset from any node ID: `offset = node_id * record_size`. This is a direct-addressed array, not an index. Finding node 1,048,576 costs one multiplication and one disk read.

  Byte:  0 │  1-4  │  5-8  │  9-12  │  13-14
  ┌───────┬───────┬────────┬────────┬─────────┐
  │ inUse │ label │ first  │ first  │ extra   │
  │ 1 bit │ bits  │ rel ID │ prop ID│ flags   │
  └───────┴───────┴────────┴────────┴─────────┘

  Node Store is a flat file:
  Offset 0:  Node 0     (record bytes 0-14)
  Offset 15: Node 1     (record bytes 15-29)
  Offset 30: Node 2     (record bytes 30-44)
  ...
  Offset N*15: Node N   (contiguous, no gaps)

  Lookup: Node ID 42 → file_read(42 * 15, 15)
  One multiplication, one read. No index.
`}

### The Relationship Store

Relationships are stored as doubly-linked lists. Each relationship record contains the start node ID, end node ID, relationship type, and pointers to the next and previous relationships for both the start and end nodes. This enables traversal in both directions without scanning the entire store.

### Property Store

Properties are stored as dynamic records in a separate store, chained from the node or relationship that owns them. Property records are variable-length because values have different types: integers (4 bytes), strings (variable), floats (8 bytes), booleans (1 byte), and arrays.

### Label Store

Labels are stored as a bitmap in the node record. With 64 bits available, a node can have up to 64 distinct labels. Label-to-node indexes are maintained separately as b-trees for efficient label-based lookups.

### The Index-Free Adjacency Data Structure

The combination of direct-addressed node store, doubly-linked relationship chains, and chained property store creates **index-free adjacency**. To find all neighbors of a node, the database only needs to:

1. Read the node record at offset `id * 15` (one file read)
2. Follow the `first_rel_id` pointer to the first relationship record
3. Walk the relationship chain reading neighbor IDs
4. Read each neighbor node record at offset `neighbor_id * 15`

Every step is a pointer dereference or a direct-address computation. No indexes, no hash lookups, no tree walks.

## Request/Write/Read Lifecycle

### Step 1: Client Sends Cypher Query

```
MATCH (alice:Person {name: "Alice"})-[:FOLLOWS]->(friends)
RETURN friends.name
```

### Step 2: Query Parsing

The Cypher parser tokenizes the query into an Abstract Syntax Tree (AST). The AST represents the query structure: what to match, how to filter, what to return.

### Step 3: Query Planning and Optimization

The planner converts the AST into an execution plan. For this query, the planner must decide: start from a label+property index (`Person.name`), or scan all nodes? Neo4j uses a cost-based optimizer that estimates row counts at each pipeline stage.

### Step 4: Index Seek

The planner picks the schema index plan. The database looks up "Alice" in the b-tree index on `Person.name`. The index entry returns the internal node ID. With the node ID, the storage engine reads Alice's node record at O(1) cost.

### Step 5: Relationship Expansion

From Alice's node record, the database reads `first_rel_id` and walks the relationship chain. Each relationship record of type `FOLLOWS` is included; others (like `REVIEWED`) are skipped. The neighbor node IDs are collected.

### Step 6: Neighbor Node Lookup

Each neighbor's node ID is used to read its node record directly. The `friend.name` property is retrieved from the node's property chain. This is approximately 5 pointer-following operations per friend.

### Step 7: Write Path — Creating a Node

When you create a node, the storage engine finds a free slot in the node store file, marks it as `inUse`, sets the label bits, and initializes the `first_rel_id` and `first_prop_id` to null. Property records are appended to the property store.

### Step 8: Write Path — Creating a Relationship

Creating a relationship is more expensive because both adjacency lists must be updated. The database appends a new relationship record and then updates the `prev` and `next` pointers in the existing chain for both the start and end nodes.

### Step 9: Deleting a Relationship

Deletion is the reverse of creation. The database repairs the linked list by connecting the deleted record's previous and next pointers to each other, then marks the record as free for reuse.

### Step 10: Writing Properties

Property writes update the property chain. Each property is stored as a key-value pair in the property store. Updating a property usually means appending a new record and updating the chain pointer, with the old record eventually being garbage collected.

## Consistency and State Handling

### Crashes During Writes

Graph databases use write-ahead logging (WAL). Before any record is modified, the change is written to a transaction log. If the database crashes mid-write, it replays the log on restart to reach a consistent state. The key challenge is maintaining the linked-list invariants of relationship chains.

Neo4j uses a logical log that records the before-and-after state of each record. If a relationship creation crashes after step 3 (of the 6 steps above), the recovery process will see that R303 is partially written and either complete the operation or roll it back.

### Concurrent Writes

Graph databases use record-level locking. When two transactions try to modify the same node's relationships, one acquires a lock on the node record and the other waits. Deadlock detection runs a cycle-detection algorithm on the lock-wait graph — fittingly, also a graph problem.

### Stale Reads and Read Committed

Most graph databases default to READ COMMITTED isolation. A traversal sees only committed data. If a relationship is being deleted in an uncommitted transaction, the traversal skips it. This prevents dirty reads but allows non-repeatable reads — a second traversal of the same path moments later might see a different set of edges.

### Partition Recovery

When a graph database cluster partitions (some nodes cannot talk to others), each partition continues operating independently. On reconnection, the databases must merge their divergent graphs. This is the most difficult operation in distributed graph databases because relationships that existed in one partition may conflict with deletions in another.

## Performance Characteristics

### Traversal Latency

Neo4j traversals run at approximately 2-5 million relationships per second per core on modern hardware when the working set fits in memory. Each hop adds 200-500 nanoseconds of traversal time. A depth-3 traversal visiting 100 nodes costs approximately 30-75 microseconds.

### Write Performance

Write throughput is constrained by the number of pointer updates per write. A single relationship creation requires 6 write operations. For bulk inserts, batching transactions reduces the per-operation overhead by 10-50 times.

### Memory and Storage

Node records are 15 bytes each. Relationship records are approximately 50 bytes each. Properties and labels add variable overhead. A social graph with 100 million nodes and 1 billion edges requires roughly: nodes (1.5 GB) + edges (50 GB) + properties (variable, typically 100-500 GB). Indexes add 10-30% overhead.

## Configuration and Tuning

### Key Knobs

**pagecache_memory** — Memory allocated for the OS page cache, exclusive to Neo4j. Default: 10% of RAM. Aggressive: 70% of RAM if the graph fits in memory.

**dbms.memory.heap.max_size** — JVM heap for query processing, not for data storage. Default: 512 MB. Aggressive: 16 GB for complex analytical queries.

**dbms.tx_state.memory_allocation** — How transaction state is tracked. Default: on-heap. Aggressive: off-heap for large write transactions.

### Default vs Aggressive Settings

| Parameter | Default | Aggressive | Effect |
|---|---|---|---|
| pagecache_memory | 512 MB | 32 GB | 50x faster traversals for cached graphs |
| heap.max_size | 512 MB | 4 GB | Prevents OOM on large result sets |
| dbms.relationship_grouping_threshold | 1 | 10 | Reduces chain walk overhead for dense nodes |
| dbms.index_sampling_update_percentage | 5 | 25 | Better cardinality estimates for query planning |

### Monitoring

Monitor: page cache hits (target above 99%), checkpoint times (target under 200ms), transaction log size, and average traversal depth. When page cache hit rate drops below 90%, the graph no longer fits in memory and performance degrades by 10-100x.
