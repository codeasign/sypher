# NoSQL Overview — Trade-offs and Failure Modes

## The Core Tension

The single fundamental trade-off in NoSQL is this: **you cannot have consistency, availability, and partition tolerance simultaneously in a distributed system**. This is the CAP theorem, and it is not theoretical — it is the engineering reality you face every time your database splits into two groups that cannot talk to each other.

But CAP is too coarse. The real trade-off is more nuanced: given that you must handle network partitions (the P in CAP is not optional in any distributed system), you choose between **consistency** (all nodes see the same data at the same time) and **availability** (every request gets a response). Most NoSQL databases choose availability: they accept writes from any node at any time, even if some nodes are isolated, and reconcile conflicts later.

This choice creates a cascade of secondary trade-offs:
- **Read freshness vs. read cost**: strong consistency doubles read latency and halves throughput.
- **Write throughput vs. read performance**: write-optimized LSM-trees degrade read performance unless compaction keeps up.
- **Schema flexibility vs. query power**: flexible schemas make it impossible for the database to optimize queries ahead of time.
- **Horizontal scaling vs. transaction support**: you cannot have distributed ACID transactions at high throughput.
- **Developer productivity vs. data integrity**: letting the application define the schema means the application must enforce integrity.

## Decision Matrix

| Decision Point | Option A | Option B | When to Pick A | When to Pick B |
|---------------|----------|----------|----------------|----------------|
| **Consistency model** | Eventual consistency | Strong consistency | User-facing reads where staleness is acceptable (scores, feeds, views) | Financial transactions, inventory counts, any operation requiring immediate visibility |
| **Data model** | Key-Value or Document | Wide-Column or Graph | Simple access by primary key, variable fields | Complex queries, time-series ranges, relationship traversal |
| **Write concern** | HIGH throughput (CL=ONE) | Strong durability (CL=ALL) | Logging, metrics, sensor data where losing some data is acceptable | Payment audit trail, critical business events |
| **Partition key design** | High cardinality key (unique per item) | Low cardinality key (groups related items) | Even distribution is critical, no hot keys | You need range scans within a partition |
| **Replication factor** | RF=3 (standard) | RF=5+ (high durability) | Most workloads; balances durability and cost | Regulatory compliance, extreme durability requirements |
| **Compaction strategy** | STCS (write-optimized) | LCS (read-optimized) | Write-heavy ingest (logs, metrics, events) | Read-heavy serving (catalog, profiles) |
| **Cache strategy** | Cache-aside (lazy population) | Write-through (eager population) | Read-heavy with predictable access patterns | Applications needing consistent cache state |
| **Schema evolution** | Additive only (new fields) | Full migration (field changes) | Rapid development with evolving requirements | Mature system with stable schema |
| **Scaling approach** | Scale up (bigger nodes) | Scale out (more nodes) | Medium scale (under 10 TB, predictable growth) | Large scale (over 10 TB, unpredictable growth) |
| **Conflict resolution** | Last-write-wins (LWW) | CRDT or application-merged | Non-critical data where latest value is always correct | Data where concurrent edits must be merged (collaboration, shopping carts) |

### What CAP Actually Means in Practice

Consider a 6-node Cassandra cluster with RF=3. A network issue splits the cluster into two groups: Group 1 (nodes A, B, C) and Group 2 (nodes D, E, F). The partition is between racks — the network switch for rack 1 has failed.

**In an AP system (Cassandra default)**: Group 1 accepts writes to the keys it owns. Group 2 accepts writes to the keys it owns. Both groups remain available. If a key's primary replica is in Group 1 but its quorum spans both groups, a write may fail (cannot achieve QUORUM). But the system keeps running for the majority of keys.

**In a CP system (HBase)**: The HMaster detects the partition via ZooKeeper and decides that Group 2 (the smaller group) is dead. It stops accepting writes and reads from Group 2's region servers. Group 1 continues serving, but with reduced capacity. Clients connected to Group 2 hang or receive errors.

The practical lesson: CAP tells you what happens at the theoretical limit (a network partition), but most failures are not network partitions. They are slow disks, high GC pauses, overloaded CPUs, or memory pressure that causes a node to stop responding within the timeout window. In these cases, an AP system treats the slow node as down and routes around it; a CP system may stall writes while waiting for the slow node to respond.

## Failure Modes

### Failure Mode 1: Compaction Storm

**Scenario**: A write-heavy application sustains 100,000 writes per second. Compaction is configured with SizeTieredCompactionStrategy. After 24 hours of peak traffic, four similarly-sized SSTables accumulate in Level 0. The compaction trigger fires and all four are compacted simultaneously, consuming 100% of disk I/O for 20 minutes.

**Impact**: Write latency jumps from 2ms to 200ms. Read latency (which must search multiple uncompacted SSTables) jumps from 5ms to 500ms. Application timeouts cascade. The database appears dead.

**Root cause**: The compaction strategy is not matched to the write pattern. STCS creates bursts of compaction when enough SSTables accumulate.

**Remedy**: Switch to LCS (Leveled Compaction) which compacts continuously with smaller, more frequent operations. Or use TWCS if the data is time-series. Set a compaction throughput limit to prevent compaction from saturating disk I/O. Monitor compaction backlog and scale out (add nodes) before compaction pressure becomes critical.

### Failure Mode 2: Hot Partition

**Scenario**: You design a table keyed by `(user_id, timestamp)` to store user activity events. You have a power user with millions of followers who triggers 10,000 events per second. All of these events land on the same partition (the partition for that user_id).

**Impact**: One node handles 10,000 writes per second while the other nodes handle under 100. That node's CPU and I/O max out. P99 latency for the entire cluster degrades because the hot node falls behind and quorum reads must wait for it. When the hot node fails under the load, its replica nodes inherit the same problem.

**Root cause**: A single partition key value ("the influencer") generates disproportionate load. The partition key was designed for query convenience (all of one user's events together), not for load distribution.

**Remedy**: Add a shard key suffix to the partition key. Instead of `(user_id, timestamp)`, use `(user_id + shard_id, timestamp)` where `shard_id` is a random number from 1-100. Distribute the hot user's events across 100 partitions. To read all of that user's events, you must query all 100 partitions and merge the results — this is the trade-off.

### Failure Mode 3: Tombstone Overload

**Scenario**: You store chat messages in Cassandra with a TTL of 7 days. Users send 500,000 messages per day across 10,000 chat rooms. After 7 days, 3.5 million tombstones accumulate per partition. A read that scans a partition (loading the recent 50 messages) must skip 3.5 million tombstones to find the 50 live rows.

**Impact**: Reads that should take 5ms take 30 seconds. The client times out. The node's GC pressure increases dramatically because it deserializes millions of tombstones just to skip them.

**Root cause**: TTL-based deletes create tombstones that are not cleaned up until compaction runs. If compaction does not keep pace with tombstone creation, the tombstone count grows without bound.

**Remedy**: Upgrade to Cassandra 4.0+ which has improved tombstone compaction. Use `gc_grace_seconds` to control how long tombstones survive. Run `nodetool compact` on problem tables after the TTL expiration wave. Consider time-windowed tables: instead of TTL on individual rows, drop entire time partitions (e.g., one table per week, drop the table after 8 weeks).

### Failure Mode 4: Read-Repair Amplification

**Scenario**: A 20-node cluster runs with `read_repair_chance = 1.0` (100%) to ensure consistency. Each read checks all three replicas. At 50,000 reads per second, this means 150,000 internal read operations per second. A single node failure causes read-repair to attempt writes to the dead node, which times out after 10 seconds.

**Impact**: Read-repair writes to dead nodes accumulate. The coordinator holds open connections waiting for timeouts. Connection pools exhaust. New requests queue and timeout. Cascading failure ensues — the database is unavailable.

**Root cause**: Read-repair set too aggressively, combined with no circuit breaker for dead nodes.

**Remedy**: Set `read_repair_chance` to 0.1 (10%) for most workloads. Use `dclocal_read_repair_chance` for cross-datacenter scenarios. Implement circuit breakers at the driver level — if a node fails to respond within 2 seconds, mark it as down and stop sending requests for 30 seconds.

### Failure Mode 5: Clock Skew and Last-Write-Wins Data Loss

**Scenario**: You use DynamoDB's strong consistency (which relies on timestamps for conflict resolution). One application server's clock drifts 3 seconds ahead due to a failed NTP sync. A legitimate write from another server occurs at "real time T" but its timestamp says T-1. The write from the clock-drifted server (timestamp T+3) wins, even though it happened 2 seconds earlier in real time.

**Impact**: Two orders placed by the same customer. The first order has a timestamp of T+3 (wrong server clock), the second has a timestamp of T+1 (real time). The second order (the one the customer actually submitted last) is silently discarded during conflict resolution because it has an earlier timestamp. The customer sees the first order's confirmation but the second order's items are missing from their history.

**Root cause**: Using wall-clock timestamps for conflict resolution without clock synchronization guarantees.

**Remedy**: Use monotonic clocks synchronized via NTP with tight tolerances (Chrony or ntpd with `maxpoll` set to 64 seconds maximum interval). Better: use Amazon DynamoDB transactions (strong consistency with conditional writes) for critical data. Best: avoid last-write-wins entirely for data where losing a write is unacceptable — use DynamoDB's conditional writes or Cassandra's lightweight transactions (Paxos-based compare-and-set).

### Anti-Pattern 1: The SQL Mentality

You model your data in normalized third normal form with separate tables for "users", "orders", "order_items", and "products." Then you try to perform a join-equivalent query across these tables in DynamoDB or Cassandra.

**Why it fails**: NoSQL databases do not support joins. You must either denormalize (duplicate data across tables, accepting the cost of maintaining consistency) or perform the join in your application code (fetch all orders, then fetch each order's items, then fetch each item's product — an N+1 nightmare).

**The fix**: Design your data model around your access patterns, not around normalization. If you always load a user's orders with their items, store the items as a list within the order document. If you need "all users who bought product X," maintain a materialized view (a separate table keyed by product_id containing user IDs).

### Anti-Pattern 2: Building a Queue in a Key-Value Store

You use Redis to implement a task queue by polling: `KEYS *` every second, sort by timestamp, process the oldest. At 10,000 queued tasks, the scan takes 500ms and blocks all other operations.

**Why it fails**: `KEYS *` blocks the Redis event loop for the duration of the scan — no other client can read or write. `SCAN` (the cursor-based alternative) does not block but is O(N) over the entire keyspace. This does not scale past a few thousand keys.

**The fix**: Use Redis Lists with `BLPOP` (blocking pop) for a simple queue. Use Redis Streams for a consumer-group-based queue. Better: use a real message queue (RabbitMQ, Kafka, SQS) for production workloads. Redis is not a queue — it is a cache that can be pressed into queue service at small scale, with sharp limits.

### Anti-Pattern 3: The Magical Scalability Expectation

You assume that because you chose Cassandra, your application will automatically scale horizontally without any data modeling effort. You create a table with a single partition key that every query uses.

**Why it fails**: NoSQL databases distribute data by partition key. Every item with the same partition key lives on one node. If all your queries hit the same partition key, you are running a single-node database in a cluster — you get zero benefit from the other nodes.

**The fix**: Every data model must be validated against the question "does this query distribute load evenly across all nodes?" If not, redesign the partition key. Use compound primary keys. Pre-join and denormalize to avoid queries that scan multiple partitions.

### Anti-Pattern 4: Schema-Less Chaos

Your development team loves the "schema-less" nature of MongoDB and starts storing documents with different field names for the same concept — `customer_name`, `customerName`, `cust_name`, and `name` all appear in different documents. After 6 months, querying by customer name requires a four-way OR condition.

**Why it fails**: NoSQL databases are schema-flexible, not schema-absent. The schema has moved from the database to the application code, but it still exists. Without discipline, the implicit schema degrades into an unqueryable mess.

**The fix**: Enforce schema at the application layer using a library (Mongoose for MongoDB, Pydantic for Python, JsonSchema validation). Run periodic audits to detect fields that deviate from the agreed schema. Document the schema in a shared repository. Treat "no schema" as "the application enforces the schema," not "anything goes."

### Anti-Pattern 5: Over-Provisioning for Peak Load Then Under-Provisioning for Compaction

You provision 100 nodes based on write throughput requirements during Black Friday (500,000 writes/sec). After Black Friday, write traffic drops to 50,000 writes/sec, but you have not removed nodes from the cluster.

During the post-Black Friday lull, compaction runs on the massive dataset accumulated during the peak. With the same 100 nodes but lower traffic, compaction finishes quickly. You declare victory and scale down to 20 nodes. Next week: compaction on the 20-node cluster runs 5x slower than on 100 nodes, and compaction backlog grows.

**The fix**: Always model compaction throughput as a function of your data volume, not your write throughput. A cluster with 10 TB of data needs enough nodes to compact 10 TB within your maintenance window, regardless of current write rate. When scaling down, validate compaction throughput first.

## When NOT To Use NoSQL

NoSQL is powerful, but it is not the right tool for every problem. Here is when you should stay with a relational database:

**You need multi-row ACID transactions**: If your application transfers money between accounts, updates an inventory count while confirming an order, or manages a booking system where double-booking is unacceptable, you need a relational database with ACID transactions. NoSQL's eventual consistency or single-document transactions are not sufficient for correctness.

**Your data is inherently relational**: If you have a complex entity-relationship diagram with 50+ tables, many-to-many relationships, and frequent ad-hoc joins across entities, a relational database with a query optimizer is the right tool. Converting this to a NoSQL data model would require extensive denormalization and application-level join logic.

**You need ad-hoc analytical queries**: If your business analysts need to run exploratory queries across 20 dimensions and aggregate 10 million records, a columnar SQL database (Redshift, BigQuery, ClickHouse) is the right tool. NoSQL databases optimize for known access patterns, not ad-hoc exploration.

**Your scale does not require it**: If you have under 500 GB of data, under 5,000 writes per second, and your application runs in a single region, a well-tuned PostgreSQL or MySQL instance will outperform a NoSQL database on cost, developer productivity, and query capability. Premature adoption of NoSQL adds complexity without benefit.

**Strong consistency is non-negotiable**: If your domain requires strict serializability — every read sees the latest write, no exceptions — and you cannot tolerate conflict resolution at the application level, stick with a relational database. No amount of NoSQL "consistency levels" can guarantee serializability in a distributed, partition-tolerant system.
