# NoSQL Overview — Interview Playbook

## How This Topic Appears

NoSQL questions in system design interviews test your ability to choose the right storage technology for a given workload, reason about consistency trade-offs, and design scalable data models. Interviewers do not ask you to recite CAP theorem definitions — they ask you to solve a problem and watch whether you reach for SQL or NoSQL, and whether you can justify the choice with specifics.

### Question Categories and Typical Phrasings

**Category 1: Database Selection**
- Should we use SQL or NoSQL for a ride-sharing app's driver location service?
- What database would you choose for a real-time leaderboard serving 10 million users?
- Design the data store for an e-commerce product catalog where each product has different attributes.
- Would you use a document store or a key-value store for a user profile service?

**Category 2: Consistency and CAP**
- You have a social media app. A user posts a comment, refreshes the page, and the comment is gone. How do you debug this?
- Your e-commerce site shows wrong inventory counts during a flash sale. Walk through the possible causes.
- How would you design an eventually-consistent system that handles write conflicts for a collaborative document editor?
- Your multi-region database shows different data in different regions. How do you fix it?

**Category 3: Data Modeling**
- Design the schema for a Twitter-like feed system using Cassandra.
- How would you model a multi-tenant SaaS application in DynamoDB?
- You have a table with a hot partition. How do you redesign it?
- Model a time-series sensor data store that handles 1 million devices reporting every 10 seconds.

**Category 4: Performance and Operations**
- Your Cassandra cluster's read latency just doubled. Walk through your debugging process.
- How do you handle a compaction storm in production?
- Your DynamoDB table is throttling writes. What do you check?
- Design a caching strategy that reduces database load by 90% for a read-heavy API.

**Category 5: Multi-Region and Global Scale**
- Design a globally-available user session store.
- How would you replicate a document store across 5 continents with under 100ms write latency?
- Your application needs active-active multi-region deployment with conflict resolution. Design it.

**Category 6: NoSQL Fundamentals**
- Explain how an LSM-tree works and why it is good for writes but not reads.
- What is the difference between eventual consistency and strong consistency, and when would you use each?
- Describe consistent hashing and why it matters for NoSQL databases.
- How does a bloom filter work and why is it important for SSTable-based storage engines?

**Category 7: Anti-Patterns**
- Your team chose MongoDB and the application is slower than the previous PostgreSQL setup. What went wrong?
- A new engineer designed a Cassandra schema that has all queries hitting one partition. How do you fix it?
- Your NoSQL database stores 50 fields but only 5 are ever queried. What design principle was violated?

**Category 8: Migration**
- You are migrating a 10 TB PostgreSQL database to Cassandra. Design the migration plan.
- How do you run a SQL database and a NoSQL database side by side during a migration?
- You need to migrate from MongoDB to DynamoDB with zero downtime. Walk through the steps.

## What the Interviewer Is Actually Testing

When you hear a NoSQL question, the interviewer is evaluating six specific signals:

**Signal 1: Do you pick the right technology for the job?** Many candidates say "use NoSQL" for everything. The best candidates say "for this access pattern, use Cassandra; for this other access pattern, use PostgreSQL; and here is the boundary where one switches to the other."

**Signal 2: Can you reason about consistency?** The interviewer wants to hear you say "eventual consistency is acceptable here because..." or "we need strong consistency here because..." with a specific justification based on the problem, not a generic statement.

**Signal 3: Do you understand data modeling in NoSQL?** Unlike SQL (where you normalize and query freely), NoSQL requires you to design the data model around the query patterns. The interviewer watches whether you ask "what are the read and write patterns?" before proposing a schema.

**Signal 4: Can you handle the operational reality?** Mentioning compaction, hot partitions, read-repair amplification, tombstone accumulation, and GC pauses shows you have operated these systems, not just read about them.

**Signal 5: Do you know what you are giving up?** The strongest signal is when you volunteer the downsides of your choice: "I would use DynamoDB here, which means we lose the ability to do ad-hoc joins. To compensate, we will pre-join the data at write time."

**Signal 6: Can you design for scale?** Starting with a simple design and evolving it through multiple iterations (single region, multi-region, active-active) shows you understand how systems grow.

## Level-by-Level Expectations

### L3 / L4 (Early Career / Mid-Level)

**Expected**: You can explain the four NoSQL types and give examples. You understand that NoSQL trades ACID for scalability. You can choose between SQL and NoSQL for a simple use case. You have heard of the CAP theorem.

**Sample answer to "Should we use SQL or NoSQL for a product catalog?"**:
"I would use a document store like MongoDB for this product catalog because each product has different attributes — a shirt has size and color, a laptop has RAM and storage. With MongoDB, I store each product as a JSON document with only its relevant fields. No ALTER TABLE needed when we add the 'weight' field for shipping items. I would index the fields we query on: category, price range, and name for search. For the inventory and order data, I would keep that in PostgreSQL since those need ACID transactions."

**Interviewer would think**: "Understands the fundamentals, can make a reasonable choice, mentions indexing appropriately."

### L5 (Senior)

**Expected**: You can design a data model for a complex access pattern. You discuss consistency levels, partition key design, and hot partition mitigation unprompted. You bring up operational concerns like compaction and read-repair. You can compare multiple NoSQL databases and justify the choice.

**Sample answer to "Design a leaderboard for a mobile game with 10 million players"**:
"I would use Redis Sorted Sets for the real-time leaderboard. Each set stores player_id with score as the rank. `ZREVRANGE` gives the top 100 in under a millisecond. Updates are O(log N) per player. For durability, I would snapshot to disk every 5 minutes and replicate to a replica Redis instance. But Redis is expensive for all 10 million players in memory — I would only keep the top 10,000 in Redis. The full leaderboard lives in Cassandra, keyed by (game_id, score) with clustering by timestamp. When a player wants to see their rank at position 2,000,000, they query Cassandra, not Redis. I partition by game_id to keep each game's data together, but if a game has a billion players, I would add a score-range prefix to the partition key to avoid hot partitions."

**Interviewer would think**: "Thinks about partial in-memory caching, pre-computes the access pattern boundary between systems, anticipates hot partition problems."

### L6 (Staff / Principal)

**Expected**: You design multi-region active-active systems. You discuss conflict resolution strategies (CRDTs, LWW, application-merged) with specific trade-offs. You design the migration path from an existing system. You account for operational costs, team skill set, and non-functional requirements. You bring up consistency guarantees at the protocol level (Paxos, Raft, quorum calculations).

**Sample answer to "Design a globally-available shopping cart service"**:
"I would use DynamoDB Global Tables with active-active replication. Each region accepts writes independently. The cart is keyed by user_id — a single DynamoDB item with a list of cart item IDs. Conflict resolution uses last-write-wins based on DynamoDB's timestamp, which is sufficient for shopping carts: if two regions accept a write for the same user's cart, one write wins, and the user may lose an item. To prevent that, I would use DynamoDB's conditional update with `attribute_not_exists(cart_item_id)` per item — this way, adding an item that already exists is a no-op rather than overwriting. The conditional update uses DynamoDB transactions, which internally use a Paxos-like protocol across three replicas. For the cart count displayed on every page, I would use eventual consistency (under 1 second staleness). For the checkout operation, I would use strong consistency to verify the cart's exact contents. Cross-region failover: Route53 health checks with a 30-second threshold; if us-east-1 is unhealthy, Route53 shifts traffic to eu-west-1. The shopping cart data is already replicated, so no data loss."

**Interviewer would think**: "Deep protocol knowledge, designs for real-world edge cases (conflict prevention, not just detection), thinks about operational failover with concrete numbers."

### L7 / L8 (Distinguished / Fellow / VP)

**Expected**: You discuss trade-offs at the level of organizational impact. You design systems that teams of 100+ engineers can build and operate. You reference real-world case studies and know why specific companies made their choices. You propose metrics for success and observability. You discuss the economics of storage choices.

**Sample answer to "Should our company migrate from Cassandra to ScyllaDB?"**:
"The decision hinges on our P999 latency requirements and operational costs. Our 200-node Cassandra cluster has P999 read latency of 120ms, driven by GC pauses. ScyllaDB eliminates GC pauses with its C++ shared-nothing architecture, which would bring P999 down to under 20ms — a 6x improvement. On a 200-node cluster saving 100ms per P999 operation at 500,000 ops/sec, that is 50,000 seconds of cumulative latency saved per second — the user-perceived improvement will be dramatic. The migration cost is non-trivial: ScyllaDB uses the same CQL protocol and SSTable format, so we can do a rolling migration with zero downtime. We would set up ScyllaDB nodes as Cassandra peers, let them bootstrap via the existing gossip protocol, and drain the Cassandra nodes over a weekend. Total migration effort: approximately 4 engineering-months. The ROI: reduce our Cassandra node count from 200 to 60 (ScyllaDB claims 3x throughput per node) and save approximately $400,000/year in EC2 costs, plus eliminate the 5 on-call incidents per month related to Cassandra GC pauses. I recommend migrating the three most latency-sensitive clusters first (user profiles, recommendations, viewing history), running them in parallel for 30 days, then proceeding with the remaining clusters in quarterly waves."

**Interviewer would think**: "Thinks in terms of organizational ROI, operational cost, and risk mitigation. References real production metrics. Proposes a phased, reversible migration."

## FAANG Framework for Answering

Use this 6-step framework when answering any NoSQL system design question:

**Step 1: Clarify Access Patterns (30 seconds)**
Ask the interviewer: "What are the read and write patterns? What is the read-to-write ratio? What is the latency requirement at P99? Do queries need to scan ranges or look up individual keys? Are there ad-hoc queries or are all queries known in advance?"

**Step 2: State Your Choice and Justify It (30 seconds)**
"I would use Cassandra because the workload is 90% writes (event tracking), we need linear scalability, and we do not need joins. The trade-off is that read performance degrades under compaction, so we will monitor compaction backlog as a critical metric."

**Step 3: Data Model (60 seconds)**
Write the schema on the whiteboard (or describe it verbally). Include partition key, clustering columns, secondary indexes if any. Explain why this model matches the access patterns.

**Step 4: Consistency and Replication (60 seconds)**
Describe your consistency level choice (CL=LOCAL_QUORUM for reads, CL=QUORUM for writes, or CL=ONE with read-repair). Explain the replication factor. Describe whether you need multi-region and, if so, the replication topology.

**Step 5: Failure Modes and Mitigations (60 seconds)**
Mention the top three failure modes for your chosen system: compaction storms (mitigation: LCS, compaction throughput limit), hot partitions (mitigation: shard key suffix, pre-splitting), tombstone accumulation (mitigation: time-windowed tables, gc_grace_seconds tuning).

**Step 6: Scale-Up Path (60 seconds)**
Describe how your design evolves from 100K users to 10M to 100M: when to add read replicas, when to add a caching layer, when to split into separate clusters by workload, when to go multi-region.

## Phrase Bank

Use these phrases to signal depth during your interview:

- "The read-to-write ratio determines whether I optimize for read performance or write throughput."
- "I would model the data around the access patterns, not the other way around."
- "Partition key cardinality is the single most important design decision in Cassandra."
- "For this workload, eventual consistency is acceptable because [specific reason based on the problem]."
- "I would use a quorum of (RF/2 + 1) to balance consistency and availability."
- "The LSM-tree separates the write path from the read path — writes are always fast, reads pay the cost during compaction."
- "A bloom filter with a 1% false positive rate saves 99% of unnecessary SSTable lookups."
- "I would add a shard key suffix to the partition key to distribute the hot user's writes across N partitions."
- "Tombstones are the hidden cost of deletes in LSM-tree databases — they accumulate until compaction."
- "This is a read-heavy workload, so I would use LeveledCompactionStrategy to minimize read amplification."
- "The system is network-partition-tolerant and eventually consistent, which means we must handle conflicting versions at the application layer."
- "I would set read_repair_chance to 0.1 — high enough to eventually fix inconsistencies, low enough to avoid read-repair storms."
- "Hinted handoff ensures writes are not lost during temporary node failures, but sustained handoff means a node is flapping."
- "For conflict resolution, I would use last-write-wins with client-side timestamps synchronized via NTP, or use DynamoDB's built-in timestamp mechanism."
- "The compaction strategy should match the data lifecycle — STCS for write-heavy, LCS for read-heavy, TWCS for time-series."
- "I would separate real-time serving from analytics by maintaining two clusters with different compaction and caching configurations."
- "The CAP theorem is a theoretical bound, but in practice, the choice is between consistency and availability during a partition — P is not optional in distributed systems."

## Red Flags

**1. Saying "NoSQL is faster than SQL" without qualification.** NoSQL is faster for specific workloads (high write throughput, simple key lookups). For complex analytical queries or multi-row transactions, SQL is faster. If you make this statement, the interviewer will ask "faster at what?" and you must be ready with specifics.

**2. Claiming your system will be "strongly consistent" across regions.** Strong consistency across regions requires synchronous replication, which means every write must wait for the slowest region. The speed of light makes this impossible for global systems. Strong consistency is achievable within a region; across regions, you get eventual consistency.

**3. Designing a Cassandra schema without considering partition key distribution.** If you design a table with `PRIMARY KEY (user_id)` and do not mention that some users generate 100x more traffic than others, the interviewer knows you have not operated Cassandra in production.

**4. Proposing MongoDB for a write-heavy workload without mentioning the global write lock.** MongoDB pre-4.0 had a global database-level write lock. Even MongoDB 4.0+ has document-level locking, not row-level (MongoDB documents are more like rows but can be arbitrarily large). High-write-throughput workloads on MongoDB require careful sharding.

**5. Forgetting to plan for compaction.** If you design a high-write-throughput Cassandra system and never mention compaction, you miss the single most important operational concern. An interviewer at a company that runs Cassandra at scale (Netflix, Apple, Discord) will notice immediately.

**6. Designing a caching layer without an invalidation strategy.** "We will cache everything in Redis" without a plan for cache invalidation leads to serving stale data forever. Mention TTLs, write-through, or event-based invalidation.

**7. Ignoring data model for NoSQL.** If you start designing a NoSQL system by normalizing the data into third normal form (as you would for SQL), you demonstrate that you do not understand how NoSQL data modeling works. Always start with the access patterns, then denormalize to match.

**8. Recommending NoSQL when the problem clearly needs SQL.** If the problem involves multi-row ACID transactions, complex joins, or ad-hoc analytical queries, and you say "use Cassandra," the interviewer ends the signal portion of the interview. Always acknowledge when NoSQL is the wrong choice.

## Practice Drills

### Drill 1: Database Selection (5 minutes)

You are designing a real-time chat application for 500,000 concurrent users. Messages are ephemeral (deleted after 7 days). Users need to see the last 50 messages in a channel. There are 100,000 channels, each receiving 1-100 messages per second.

Answer: Which NoSQL database and data model do you choose? Time yourself: 5 minutes.

**Model answer**: Cassandra with table `channels(channel_id text, message_time timeuuid, user_id text, content text, PRIMARY KEY (channel_id, message_time))` WITH default_time_to_live = 604800. Each channel is a partition. Messages are sorted by timeuuid within the partition. Reading the last 50 messages is a range scan with `ORDER BY message_time DESC LIMIT 50`. TTL automatically deletes messages after 7 days (tombstones, but TWCS mitigates). Replication factor 3, LOCAL_QUORUM for both reads and writes.

### Drill 2: Hot Partition Resolution (5 minutes)

Your DynamoDB table `events` has partition key `user_id`. A celebrity user generates 200 writes per second, causing throttling. The table has 1000 RCU and 1000 WCU provisioned.

Answer: How do you fix this without changing the application code's query pattern? Time yourself: 5 minutes.

**Model answer**: Add a shard key suffix to the partition key. In the DynamoDB table, change the partition key to `user_id_shard = user_id + ":" + random(1-100)`. Write requests pick a random shard suffix, distributing the celebrity's writes across 100 partitions. Read requests must query all 100 shards and merge results in the application. Use DynamoDB's parallel scan or batch get. Increase WCU to 2000 to accommodate the celebrity's peak. If the celebrity's writes are 200/sec distributed across 100 shards, each shard sees only 2 writes/sec.

### Drill 3: Consistency Design (10 minutes)

You are designing a ticket-booking system for concerts. Each concert has 10,000 seats. Two users may try to book the same seat simultaneously.

Answer: Can you use NoSQL for this? If so, how do you prevent double-booking? If not, why? Time yourself: 10 minutes.

**Model answer**: NoSQL can work, but with constraints. Use DynamoDB with conditional writes: the ticket item has `status: "available"`. The booking operation is a conditional update: `UpdateExpression: "SET status = :booked"` with `ConditionExpression: "attribute_exists(ticket_id) AND status = :available"`. DynamoDB's conditional update is atomic per item — it uses a Paxos-like consensus internally. At high concurrency (10,000 seats / 10 seconds = 1,000 writes/sec), DynamoDB's conditional writes handle this cleanly. However, if you need to book multiple seats in one transaction (e.g., "4 seats together"), you need DynamoDB's transaction API, which supports up to 100 items per transaction. For complex booking logic with hold-and-expire patterns, PostgreSQL with SERIALIZABLE isolation is simpler and safer — you should seriously consider starting with SQL for seat booking and using NoSQL only for ancillary data (concert metadata, user profiles).

### Drill 4: Migration Plan (15 minutes)

Your company has 5 TB of user profile data in PostgreSQL. The profiles are JSON blobs with 200 fields per user, 30 million users. Writes are 5,000/sec. Reads are 50,000/sec. The team wants to migrate to DynamoDB for lower latency and auto-scaling.

Answer: Design the zero-downtime migration plan. Time yourself: 15 minutes.

**Model answer**: 
1. Write data: deploy dual-write logic to write to both PostgreSQL and DynamoDB. Application code writes to a local queue (SQS), which asynchronously writes to DynamoDB. The queue handles backpressure.
2. Backfill: run a batch job that reads all 5 TB from PostgreSQL and bulk-loads into DynamoDB using DynamoDB's write-shard mechanism (5,000 WCU per shard, 12 shards = 60,000 writes/sec).
3. Verify: compare checksums per user between PostgreSQL and DynamoDB. Mismatch rate under 0.1% is acceptable.
4. Read migration: flip reads from PostgreSQL to DynamoDB in gradual percentage steps (10%-25%-50%-75%-100%) over 2 weeks. Monitor P99 latency and error rate at each step.
5. Dual-write tear-down: after all reads are on DynamoDB for 30 days, remove the PostgreSQL write path.
6. PostgreSQL retirement: dump PostgreSQL to S3 for archival, then decommission.

### Drill 5: Multi-Region Design (15 minutes)

Your social media app has users in the US, Europe, and Asia. You need each user to see their feed with under 200ms P99 latency. Feeds contain posts from followed users. A post should appear in followers' feeds within 5 seconds.

Answer: Design the database architecture. Time yourself: 15 minutes.

**Model answer**: Use Cassandra with active-active multi-region deployment. Cluster in us-east-1, eu-west-1, ap-southeast-1. Each data center has 12 nodes. Replication factor 3 per data center. Write consistency: LOCAL_QUORUM (writes acknowledged by local DC only, under 5ms). Read consistency: LOCAL_QUORUM. Cross-DC replication: Cassandra's native async replication sends writes from each DC to the others within under 1 second.

Data model: Two tables. `posts_by_user` (partitioned by user_id, clustered by post_timestamp) stores each user's own posts. `feed_by_user` (partitioned by feed_owner_id, clustered by post_timestamp) stores the pre-computed feed for each user.

Feed computation: When user A posts, a background worker reads A's followers from a Redis set, then fan-out writes to each follower's `feed_by_user` partition. The fan-out is batched: for 10,000 followers, use 10 batches of 1,000 writes each, sent asynchronously. Total fan-out time: under 2 seconds.

Cache: Redis in each region caches the top 50 feed entries for the most active 1% of users. Cache hit rate: 80%. Cache miss fall through to Cassandra local read (under 5ms).

This design gives P99 read latency of under 50ms (local Cassandra read plus optional Redis cache hit) and writes visible in under 5 seconds across all regions.
