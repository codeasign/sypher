# Understanding SQL vs NoSQL

## The Startup That Chose Wrong

You are the first engineer at a food delivery startup. Your co-founder wants to launch in two weeks. You need a database. You have heard about PostgreSQL and MongoDB, so you pick one — MongoDB — because it feels faster to build with. No schema. No migrations. Just dump JSON objects in.

Five months later, you have 50,000 orders. Your app shows the wrong delivery status on 3% of orders because an order document has three different shapes in the same collection. Your finance team cannot run a simple "total revenue last month" query without a script that times out. Your CTO asks why adding a "driver tip" field requires updating 12 code paths. Your index on `order.created_at` takes 30 seconds to build because the B-tree has to reconcile mixed document types.

You wish you had picked PostgreSQL.

Three years later, the startup now has 5 million users. Your PostgreSQL monolith handles writes on a single primary. During dinner rush, `SELECT ... FOR UPDATE` contention on the `orders` table pushes p99 latency to 8 seconds. Your team spends weekends fighting lock escalation. Your lead infrastructure engineer says you need to shard — but PostgreSQL horizontal scaling in 2024 is painful without Citus or a managed service that does it for you.

You wish you had picked something else.

## The Insight

Think of SQL databases as a **filing cabinet with labeled folders**. Every folder has the same set of fields. If you need to find all orders from last Tuesday, you open the "orders" drawer, look up the date index card, and pull exactly the ones you need. You know the shape of every folder before you open it. The filing cabinet enforces that every folder in the "orders" drawer has a `total` field and an `order_date` field — if a folder is missing a required field, the cabinet rejects it.

NoSQL databases are a **warehouse of plastic bins**. Each bin holds whatever you want. Order \#1 is a bin with delivery address, items, and payment. Order \#2 is a bin with the same three things plus a customer loyalty number. Order \#3 is a bin that somehow has a nested array of driver locations. No two bins look the same. You can find things by digging through all bins, or by building a separate index wall that tracks where certain values live. But nothing stops anyone from putting a toaster in a bin labeled "orders."

The filing cabinet gives you safety and query power at the cost of flexibility. The warehouse gives you speed and flexibility at the cost of consistency guarantees. Real systems use both — orders go in the filing cabinet, session data goes in the warehouse, and a background job moves data between them.

## Core Vocabulary

- **SQL (Structured Query Language)**: The standard language for querying relational databases. Declarative — you say what you want, not how to get it. The database optimizer figures out the fastest path.

- **NoSQL**: An umbrella term for databases that do not use the relational model. Includes document stores (MongoDB), key-value stores (Redis), wide-column stores (Cassandra), and graph databases (Neo4j).

- **Schema**: The enforced structure of a SQL table — column names, types, constraints. In NoSQL databases, the schema is implicit in the application code, not enforced by the database.

- **ACID**: Atomicity, Consistency, Isolation, Durability — the four guarantees of SQL transactions. A transaction either completes fully or rolls back entirely, the database stays in a valid state, concurrent transactions do not interfere, and committed data survives crashes.

- **BASE**: Basically Available, Soft state, Eventual consistency — the NoSQL trade-off. The database stays available, state changes over time without input, and if you stop writing, replicas converge to the same value.

- **Index**: A data structure (B-tree, LSM tree, hash table) that speeds up lookups by trading write overhead for read speed. Both SQL and NoSQL databases use indexes, but SQL databases optimize for complex queries while NoSQL databases optimize for simple lookups.

- **Sharding**: Splitting data across multiple servers horizontally. Each server holds a subset of rows. SQL databases historically make sharding hard because JOINs and transactions cross shard boundaries. NoSQL databases are built for it from day one.

- **Replication**: Copying data from one server to another for durability and read scaling. Most SQL databases use single-primary replication (one writer, many readers). Many NoSQL databases use multi-primary replication (any node can accept writes).

- **Join**: Combining rows from two or more tables based on a related column. SQL databases support joins natively with sophisticated query planners. NoSQL databases avoid joins by storing denormalized data, pushing join logic to the application.

- **Denormalization**: Storing the same data in multiple places to avoid joins. A MongoDB order document might include the customer name and address directly, duplicating data that would live in a separate `customers` table in SQL. Denormalization speeds reads at the cost of update complexity.

- **Polyglot Persistence**: Using multiple database types in the same system, each chosen for the workload it handles best. A typical stack might use PostgreSQL for transactions, Redis for caching, and Cassandra for time-series analytics.

- **CAP Theorem**: The principle that a distributed database can guarantee at most two of Consistency, Availability, and Partition Tolerance. In practice, partitions happen, so you choose between CP (consistent but unavailable during partitions) and AP (available but possibly inconsistent during partitions).

### Step 2: Query parser and planner (SQL only)

SQL databases parse the query text into an abstract syntax tree, then the query planner generates execution strategies. The planner estimates costs based on table statistics — row counts, index selectivity, join cardinality — and picks the cheapest plan.

```sql
-- The planner considers:
-- 1. Full table scan — read all rows, filter
-- 2. Index scan — use index on total, filter
-- 3. Bitmap scan — combine multiple indexes
SELECT * FROM orders WHERE total > 100 AND status = 'delivered';
```

NoSQL databases skip this step. A MongoDB query hits a single collection and uses a single index per query (except for `$or` queries). There is no join planning, no multi-table optimization.

### Step 4: Index lookup

SQL databases use B-tree indexes by default. A B-tree search for a single row requires traversing from root to leaf — about 3-5 node reads for a table with 10 million rows. Each node read can be from cache or disk.

NoSQL databases use a variety of index structures. MongoDB uses B-trees (WiredTiger engine). Cassandra uses LSM trees — writes go to an in-memory memtable, then flush to SSTable files on disk. Redis uses hash tables for O(1) key lookups.

### Step 5: Fetch rows or documents

SQL returns a row set — multiple rows matching the query, each with the same columns. If you joined tables, rows contain columns from all joined tables.

NoSQL returns documents. A single document might contain nested arrays, embedded sub-documents, or references to other documents (which the application must resolve).

### Step 6: Write path

A SQL write uses a Write-Ahead Log (WAL). The transaction is recorded in the WAL first, then the data page is updated in memory. If the server crashes, the WAL replays on restart. This gives durability without flushing data pages on every write.

A NoSQL write in MongoDB follows the same WAL pattern (journal). In Cassandra, the write goes to the commit log (for durability) and the memtable (in memory). The memtable flushes to SSTable when full.

### Step 7: Transaction management

SQL databases manage concurrent access through locking and MVCC (Multi-Version Concurrency Control). PostgreSQL uses MVCC — each transaction sees a snapshot of data as of its start time. A writer does not block a reader, and a reader does not block a writer. Two writers on the same row do block: the second waits for the first to commit or roll back.

```sql
-- Transaction A: starts, updates row 42
BEGIN;
UPDATE orders SET status = 'shipped' WHERE id = 42;
-- Transaction B: tries to update same row — WAITS
BEGIN;
UPDATE orders SET status = 'delivered' WHERE id = 42;
-- B blocks until A commits or rolls back
```

NoSQL databases handle concurrency differently. MongoDB uses document-level locking with WiredTiger. Cassandra uses last-writer-wins (LWW) with timestamps. If two writes hit the same key simultaneously, the one with the later timestamp wins — even if the earlier write was logically correct. This "no locking" design gives higher throughput under contention but can lose data.

### Step 8: Return results

SQL returns structured rows. The client library (psycopg2, mysql-connector) parses the wire protocol and materializes Python objects. For large result sets, SQL databases use server-side cursors to paginate.

NoSQL returns documents — JSON or BSON. MongoDB returns up to 16MB per document by default. The client library deserializes BSON to native objects.

### Variant 2: Document NoSQL — MongoDB vs Couchbase

Document stores store JSON-like objects and index their contents.

- **MongoDB**: Most popular document store. Rich query language (but no joins). Built-in horizontal sharding via config servers and mongos routers. Used by Uber, Coinbase, Forbes. Default choice for document NoSQL.

- **Couchbase**: Combines document store with caching layer (Membase heritage). N1QL query language provides SQL-like syntax for JSON. Built-in cross-datacenter replication (XDCR). Lower operational overhead than MongoDB for caching-heavy workloads.

### Variant 4: Wide-Column NoSQL — Cassandra vs HBase

Wide-column stores organize data in rows but allow each row to have different columns. They are designed for write-heavy, time-series workloads.

- **Cassandra**: Peer-to-peer architecture (no single point of failure). Each node can accept reads and writes. Tunable consistency — you choose how many replicas must acknowledge a read or write. Linear scalability: double the nodes, double the throughput. Used by Netflix, Apple (75K+ nodes), Discord.

- **HBase**: Built on HDFS. Single master (HMaster) manages region servers. Stronger consistency than Cassandra (CP system). Better for analytics workloads where you need consistent scans. Higher operational complexity — requires ZooKeeper, HDFS, and careful region splitting.

## Mental Model Check

Test your understanding with these five questions.

**Q1: You are building a booking system for a dental clinic. Each appointment has a patient, a dentist, a time slot, and a status. The clinic needs to know if a dentist is double-booked. Should you use SQL or NoSQL?**

A1: SQL. The booking system has relationships (patient, dentist, appointment) and needs strong consistency — you cannot have two appointments for the same dentist at the same time. A SQL transaction with a unique constraint on (dentist_id, time_slot) prevents double-booking at the database level. In NoSQL, you would need application-level locks or conditional updates, which are harder to get right.

**Q2: You need to store 200 million user profiles. Each profile has 5-20 fields. You query by user ID only, and you never join with other tables. What database type?**

A2: NoSQL — specifically a key-value store like DynamoDB or Cassandra. The access pattern is simple (get by ID), there are no joins, and the data volume is large enough that horizontal scaling matters. A SQL database would work but would force you to shard manually or pay for a managed sharded service.

**Q3: Your analytics team needs to run ad-hoc queries combining data from sales, inventory, and customer support. Queries change every week. What do you recommend?**

A3: SQL with a columnar store (PostgreSQL with columnar extension, ClickHouse, or Redshift). Ad-hoc queries benefit from SQL's rich query language — JOIN, GROUP BY, window functions, CTEs — and columnar storage makes aggregate queries over large datasets 10-100x faster than row-oriented storage.

**Q4: In Cassandra, two writes happen at the same time on different nodes for the same row. What value does the database return on read?**

A4: The value with the highest timestamp wins (last-writer-wins reconciliation). If both writes have the same timestamp, Cassandra uses the value that sorts higher lexicographically. This means data loss is possible if two writers produce different values concurrently — Cassandra prioritizes availability and partition tolerance over consistency.

**Q5: You have a PostgreSQL database with a table that stores user sessions. Queries are always exact key lookups on session_id, and you have 500 million rows. Should you use a B-tree index on session_id?**

A5: Yes, a B-tree index on session_id gives O(log N) lookup — about 5-6 node traversals for 500 million rows (since each B-tree level doubles capacity). But a hash index would be even faster for exact lookups (O(1) average). Consider an HASH index in PostgreSQL or switch to Redis (in-memory key-value) if sessions fit in RAM.
