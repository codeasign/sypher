# SQL vs NoSQL — Challenges

These challenges test your understanding of the SQL vs NoSQL decision framework. Each tier demands deeper reasoning: Foundation challenges check recall and comprehension, Design challenges test applied judgment, and Expert challenges require you to navigate ambiguity with no single right answer.

Plan your time accordingly. Foundation challenges should take roughly 5 minutes each, Design challenges roughly 15 minutes each, and Expert challenges roughly 30 minutes each. Write your answers concisely but justify every decision.

**How to approach these challenges**

For Foundation challenges: focus on clarity and precision. Define your terms before using them, and always tie abstract concepts (like CAP theorem or indexing strategies) to specific, concrete scenarios. Use the model answers as a reference for the level of detail expected.

For Design challenges: start by identifying the non-negotiable constraints (latency, throughput, durability, consistency). Then evaluate each database option against those constraints before making your final choice. Show your math for storage and throughput calculations — a design without numbers is speculation. Each Design challenge expects you to weigh trade-offs explicitly: what capability does the chosen database sacrifice in exchange for the one you need?

For Expert challenges: acknowledge the trade-offs explicitly. There is no perfect answer, so your reasoning matters more than your final conclusion. State your assumptions, evaluate at least one alternative approach, and explain why you rejected it. Expert challenges are designed to simulate real-world architectural discussions where the team disagrees on the best approach and the principal engineer must navigate the trade-offs to reach a defensible decision.

**Scoring guidance**

There are no formal points for these challenges. Instead, evaluate yourself against the evaluation criteria for each challenge. If you satisfy all criteria, consider yourself proficient. If you miss one or two, review the model answer and common mistakes to identify gaps. The goal is not to get every detail right on the first try — it is to build an accurate mental model of when and why to choose SQL, NoSQL, or both.

## Tier 1 — Foundation

### Challenge 1: Explain the CAP Theorem Trade-off

**Time:** 5 minutes

Explain how the CAP theorem applies to the choice between a relational database and a document database. Use a concrete example of a real-time chat application.

**Prompt**

Write a short response that covers all of the following:

1.  State the three properties of the CAP theorem and what each means in one sentence.
2.  Explain why a distributed system cannot guarantee all three simultaneously.
3.  Pick one SQL database (for example, PostgreSQL with pgpool) and one NoSQL document database (for example, MongoDB) and describe which two CAP properties each prioritizes.
4.  For the chat application example — users send messages and expect them to appear instantly for all participants — state which CAP trade-off each database would force and why that trade-off matters for chat.

**Evaluation Criteria**

Your response is complete if it:

- Correctly defines Consistency, Availability, and Partition Tolerance.
- States that CP sacrifices Availability during a partition and AP sacrifices Consistency.
- Maps PostgreSQL (with standard replication) to CP and MongoDB (default configuration) to AP.
- Explains that a chat app can tolerate eventual consistency (a few seconds of delay) more than it can tolerate downtime, making AP the natural fit.
- Discusses the practical reality that partition tolerance is not optional in distributed systems, so the real choice is always between CP and AP.
- Acknowledges that the trade-off exists only during a network partition — outside of partitions, both databases provide both consistency and availability.

**Common Mistakes**

- Claiming CAP theorem is about choosing two of three at all times — it applies only during a network partition.
- Labeling a single-node database as CP when it has no partition tolerance at all (single-node databases are CA in the traditional informal framing, but that is only meaningful when no partition has occurred).
- Forgetting that partition tolerance is not optional in a distributed system: if the system is distributed, it must tolerate partitions, so the real choice is CP or AP.
- Confusing consistency in CAP (linearizability) with the ACID consistency property — ACID consistency refers to database invariants and referential integrity, which are entirely different concerns.
- Assuming that choosing AP means accepting inconsistency forever — in practice, AP systems converge to consistency once the partition heals (eventual consistency).

**Model Answer**

The CAP theorem states that a distributed data system can guarantee at most two of three properties: Consistency (every read receives the most recent write or an error), Availability (every request receives a non-error response, without guarantee that it contains the most recent write), and Partition Tolerance (the system continues to operate despite network failures that cause nodes to lose contact).

During a network partition, the system must choose: either refuse responses until the partition heals to guarantee consistency (CP), or return a response even if some nodes are stale to guarantee availability (AP). In practice, partition tolerance is mandatory in any distributed deployment — network failures are inevitable — so the real choice is between CP and AP.

PostgreSQL deployed with a primary-replica setup and a tool such as pgpool is typically configured as CP: if the primary goes down or a network fault isolates replicas, the system may pause writes or refuse reads from replicas to guarantee that every read returns the same, current data. MongoDB with its default settings (primary writes, secondary reads allowed) leans AP: during a partition, secondary replicas continue serving reads even if they have not received the latest write, accepting temporary inconsistency to keep the system available.

A real-time chat application must remain available even when the network is flaky — users would rather see a message appear a few seconds late than see an error. Therefore the AP trade-off (MongoDB-style) is the natural choice: accept that some users may briefly see a slightly stale message order in exchange for the system never refusing requests. An SQL database in strict CP mode would reject writes or block reads during a partition, which would break the chat experience.

**Extension questions for deeper study:** The CAP theorem is often criticized as too coarse-grained. How would you refine your answer using the PACELC extension (which adds a latency-consistency trade-off when there is no partition)? How does the choice change if the chat application requires strong ordering guarantees (for example, messages must appear in the exact order they were sent)?

To make this concrete, imagine two users, Alice and Bob, chatting in a group. A network partition separates the data center that handles Alice's region from the one that handles Bob's region. Under a CP configuration, Alice's message would be rejected because the system cannot confirm that Bob's replica has the latest data. Under an AP configuration, Alice's message is accepted immediately, and Bob will see it once the partition heals (typically within milliseconds to seconds). In a chat application, temporary inconsistency is far more acceptable than message loss or errors.

**Extension — the PACELC trade-off:** The CAP theorem only addresses behavior during a partition. The PACELC extension adds a second dimension: in the absence of a partition (Else), the system must choose between Latency and Consistency. A CP database like PostgreSQL typically prioritizes consistency over latency, meaning it may introduce coordination overhead (locking, two-phase commit) that increases latency during normal operation. An AP database like MongoDB typically prioritizes latency over consistency, accepting that replicas may serve slightly stale data to keep response times low. For a chat application, both during and outside of partitions, the AP+L orientation (availability during partitions, latency over consistency in normal operation) is the right fit.

### Challenge 2: Choose Between SQL and NoSQL for Three Workloads

**Time:** 5 minutes

For each of the following three scenarios, decide whether SQL or NoSQL is the better fit and justify your choice in two or three sentences. Name a specific database product for each.

**Scenarios**

1.  **Booking system** — A hotel reservation platform that must never double-book a room. Transactions involve checking availability, reserving the room, and charging the guest's card as one atomic unit.
2.  **IoT sensor ingestion** — 50,000 temperature sensors each report a reading every 10 seconds. The system must ingest 5 million writes per second with high throughput and minimal coordination overhead. Reads are almost never performed on individual rows; instead, analytics queries aggregate over time windows.
3.  **Product catalog** — An e-commerce site where each product has a varying set of attributes (laptops have CPU speed and RAM; shirts have size and color; furniture has dimensions and material). New product categories are added every quarter, each with its own attribute schema.

**Evaluation Criteria**

Each choice must name a specific database product and provide a concrete reason.

- Scenario 1: SQL — ACID transactions are non-negotiable for double-booking prevention. Correct answer names PostgreSQL or MySQL with InnoDB and mentions atomic transactions or SERIALIZABLE isolation.
- Scenario 2: NoSQL — Time-series or wide-column databases excel at high-ingestion, low-latency writes. Correct answer names TimescaleDB, InfluxDB, or Cassandra and mentions append-heavy write pattern and columnar compression.
- Scenario 3: NoSQL — Document databases handle polymorphic schemas naturally. Correct answer names MongoDB or Couchbase and mentions schema flexibility and the ability to store varying attributes in the same collection without migrations.

**Extension questions for deeper study:** For scenario 1, what if the booking system needs to handle 100,000 concurrent users booking the same popular hotel during a flash sale? Does the SQL choice still hold, or do you need to introduce a queue or sharding layer? For scenario 2, what if the IoT sensors also need real-time anomaly detection, requiring sub-millisecond read latency on the most recent reading from each sensor?

**Common Mistakes**

- Choosing SQL for scenario 2 because of a general preference for relational databases — SQL databases hit write bottlenecks under 5 million writes per second without expensive sharding setups.
- Choosing NoSQL for scenario 1 and claiming that document-level atomicity is sufficient — document databases do not support multi-document ACID transactions with the same guarantees unless explicitly configured, and their distributed transaction performance is significantly worse than that of a well-tuned SQL database for this workload.
- Recommending MongoDB for the product catalog without addressing the need for indexes on polymorphic fields — you would need to create indexes on the common query paths even if the schema is flexible.

**Model Answer**

Scenario 1 — Booking system: SQL. Use PostgreSQL with SERIALIZABLE isolation. The double-booking constraint requires atomic transactions that check inventory and decrement it as a single operation. PostgreSQL's SERIALIZABLE snapshot isolation prevents phantom reads that could cause two bookings to claim the last room. A concrete booking flow would look like this:

```sql
BEGIN;
SELECT available_rooms FROM rooms WHERE room_id = 101 FOR UPDATE;
-- Application checks: available_rooms > 0
UPDATE rooms SET available_rooms = available_rooms - 1 WHERE room_id = 101;
INSERT INTO bookings (guest_id, room_id, check_in, check_out) VALUES (..., 101, ...);
COMMIT;
```

The `FOR UPDATE` lock ensures that no other concurrent transaction reads the same stale count. Without this, two users could both see 1 available room and both book it. NoSQL databases typically lack this level of transaction isolation across multiple documents or records without significant performance penalties.

Scenario 2 — IoT sensor ingestion: NoSQL. Use Apache Cassandra or ScyllaDB. Time-series data with 5 million writes per second demands a database designed for horizontal write scaling. Cassandra's log-structured merge tree (LSM-tree) storage engine makes writes nearly as fast as appending to a log, and its peer-to-peer architecture avoids the single-writer bottleneck of a primary-replica SQL setup. A sensible Cassandra schema uses a partition key of `(sensor_id, date_hour)` to evenly distribute writes across the cluster, with a clustering column of `reading_timestamp` for chronological ordering:

```sql
CREATE TABLE sensor_readings (
    sensor_id text,
    date_hour text,
    reading_timestamp timestamp,
    temperature double,
    humidity double,
    pressure double,
    PRIMARY KEY ((sensor_id, date_hour), reading_timestamp)
);
```

Scenario 3 — Product catalog: NoSQL. Use MongoDB. Products with wildly different attributes (CPU speed for laptops, fabric for shirts) fit naturally into a document model where each document can have a different set of fields. Adding a new product category does not require a schema migration or a separate table join — you simply insert documents with the new attributes. For example, a laptop document and a shirt document can live in the same collection:

```javascript
// Laptop document
{ "_id": "...", "category": "electronics", "name": "UltraBook Pro",
  "attributes": { "cpu_speed": "2.4 GHz", "ram": "16 GB" } }

// Shirt document
{ "_id": "...", "category": "clothing", "name": "Cotton Oxford",
  "attributes": { "size": "M", "color": "Blue", "material": "Cotton" } }
```

In a relational database, this would require either a sprawling table with many NULL columns for each product type, or a complex EAV (Entity-Attribute-Value) pattern that makes queries cumbersome and slow.

### Challenge 3: Indexing and Query Performance

**Time:** 5 minutes

You have a relational database table called `orders` with 50 million rows. The table has columns `order_id` (UUID primary key), `customer_id` (integer), `status` (varchar, one of six values), `created_at` (timestamp), and `total_amount` (decimal). Two queries are running slowly:

- Query A: `SELECT * FROM orders WHERE customer_id = 4921 AND created_at > '2026-01-01';`
- Query B: `SELECT COUNT(*) FROM orders WHERE status = 'shipped';`

Explain, in plain terms, what index or indexes you would create for each query and why that index helps. Then describe how the same workload might be handled differently in a document database like MongoDB.

**Evaluation Criteria**

Your answer must:

- Propose a composite B-tree index for Query A: `(customer_id, created_at)` — covering both equality and range predicates.
- Explain why separate indexes on `customer_id` and `created_at` would be less efficient: the database would need to use one index for the equality filter and then filter the results for the range condition, rather than scanning a single narrow range in the composite index.
- Explain that Query B is slow because `status` has low cardinality (only six values) and a B-tree index on `status` alone would not help much. A partial index or a filtered index on `status = 'shipped'` would be smaller and faster. If counts must be real-time, a materialized view that tracks counts per status is a better solution.
- For MongoDB: describe how a similar compound index on `{ customer_id: 1, created_at: -1 }` would support Query A, and an aggregation pipeline with `$match` and `$count` would handle Query B, noting that MongoDB also struggles with low-cardinality equality filters.
- Explain the difference in execution plan between a full table scan (sequential read of 50 million rows from disk, potentially minutes of work) and an index scan (reading a few thousand index entries, completing in milliseconds). Concrete numbers make the explanation stronger.

**Extension questions for deeper study:** How would your indexing strategy change if Query A also needed to return only `total_amount` rather than all columns, and the table had 200 million rows instead of 50 million? How do covering indexes work in this context? What is the performance difference between a partial index and a materialized view for Query B when the data changes at 1,000 new orders per minute?

**Common Mistakes**

- Proposing separate indexes on `customer_id` and `created_at` — a composite index is more efficient because the database can use the first column for equality and scan a narrow range on the second column, whereas separate indexes would require a bitmap merge or a lookup on one index followed by a filter.
- Suggesting a full-text index — these are not relevant for numeric or date predicates.
- Claiming MongoDB handles low-cardinality counts inherently better — it does not; the same selectivity problem applies because MongoDB's B-tree index on `status` still contains an entry for every document, regardless of cardinality.
- Proposing a covering index for Query B that includes all columns from the table — this would be enormous and wasteful, since `COUNT(*)` only needs to know whether a row matches, not its full content.
- Assuming that adding an index always solves performance problems — an index on `status` for Query B would actually make the query slower than a sequential scan if a large percentage of rows have `status = 'shipped', because reading the index and then fetching rows by pointer is more expensive than a straight table scan.

**Model Answer**

For Query A (find orders for a specific customer after a specific date), create a composite B-tree index on `(customer_id, created_at)`. The database will first locate the exact `customer_id` in the index tree, then scan only the portion of the index where `created_at` is after January 1, 2026. This avoids a full table scan of 50 million rows. The index is a B-tree where the key is a tuple of (customer_id, created_at). The database can use the first column for equality lookups and the second column for range scans because of the index's inherent ordering. If the query were reversed (range on customer_id and equality on created_at), the index would not be as effective.

For Query B (count of shipped orders), an index on `status` alone is not very effective because a large fraction of rows (potentially millions) will have `status = 'shipped'`. The database would still need to read millions of index entries and then perform a count. A better approach is a partial index on `(status)` with a WHERE clause `WHERE status = 'shipped'`, which is far smaller and can answer the count by scanning only the entries in that index:

```sql
CREATE INDEX idx_orders_shipped ON orders(status) WHERE status = 'shipped';
```

If the count is needed frequently and must be fast, create a materialized view that precomputes counts per status and refresh it periodically or on write:

```sql
CREATE MATERIALIZED VIEW order_status_counts AS
SELECT status, COUNT(*) as cnt FROM orders GROUP BY status;
```

This materialized view stores just 6 rows and can answer any status count query in under 1 millisecond.

In MongoDB, you would create a compound index on `{ customer_id: 1, created_at: -1 }` to support the equivalent query. For the count query, you would use `db.orders.countDocuments({ status: 'shipped' })`. If a large portion of documents match `shipped`, MongoDB also struggles because it still must scan all documents matching the filter, even with an index on `status`. A better solution is to use an aggregation pipeline with `$match` and `$count`, or maintain a separate counter collection updated via change streams or application logic. MongoDB's `$count` stage is optimized to count matching documents without retrieving the full document payload, but it still must traverse the index or collection.

## Tier 2 — Design

### Challenge 4: Design the Data Layer for a Social Media Feed

**Time:** 15 minutes

You are designing the data layer for a social media platform that serves 10 million daily active users. Each user's feed shows the 50 most recent posts from people they follow, ordered by creation time.

**Requirements**

- **Scale:** 10 million daily active users. Each user follows 200 other users on average. 500,000 new posts are created per hour.
- **Feed generation:** When a user opens the app, their feed must load in under 500 milliseconds.
- **Write path:** When a user creates a post, it should become visible to their followers within a few seconds.
- **Data size:** Each post record is roughly 2 KB. Store posts for up to 90 days; older posts are archived and no longer appear in feeds.

**Your task**

Answer all of the following:

1.  Choose one primary database (SQL or NoSQL) for storing posts. Justify your choice.
2.  Choose a strategy for feed generation: pull-based (query the database each time a user opens the app) or push-based (precompute feeds when a post is created and store them). Justify your choice.
3.  If you chose the push-based approach, describe the data structure you would use to store precomputed feeds. Specify the database technology and the schema or document structure.
4.  Calculate the approximate storage required for precomputed feeds. Show your math.
5.  Identify one anti-pattern that would cause this design to fail at scale.

**Evaluation Criteria**

- Choice of database must consider write throughput (500,000 posts per hour) and read latency (under 500 ms per feed load).
- Feed generation strategy must account for fan-out: each post must reach 200 followers on average, so 500,000 posts per hour = 100 million fan-out writes per hour.
- Storage calculation for precomputed feeds must be realistic: 50 post IDs per feed times 10 million users yields 500 million feed entries. At roughly 40-100 bytes per entry (post ID plus sort key), this is 20-50 GB.
- The anti-pattern must be something concrete, such as storing full post content in the feed instead of just post IDs, or using a single SQL table for all feeds without partitioning.
- The answer should discuss how the system handles the fan-out write volume without overwhelming the feed storage database — for example, using async worker queues and batching writes.
- The answer should at least acknowledge the celebrity/high-follower problem, even if the proposed solution is not fully fleshed out.

**Common Mistakes**

- Choosing a pull-based (read-path) approach without considering that at 10 million users, even a highly optimized query would need to join a follow graph of 2 billion edges to generate each feed from scratch.
- Choosing a document database for posts but then storing feeds in a relational table, which creates operational complexity and cross-database consistency issues.
- Storing full post content in the feed cache instead of just post IDs and a timestamp, which multiplies storage by roughly 40x.
- Not accounting for the fan-out write volume: 100 million feed-update writes per hour requires a high-throughput, append-friendly store such as Redis sorted sets or Cassandra.
- Designing a schema that requires range queries on non-primary-key columns without proper indexing, leading to full table scans during feed generation.
- Ignoring the celebrity or power-user problem and applying a uniform fan-out strategy to all users regardless of follower count.

**Model Answer**

1.  **Primary database for posts:** NoSQL — Apache Cassandra. Posts are write-heavy (500,000 per hour, roughly 140 writes per second sustained) and each post is self-contained (no complex joins). Cassandra's LSM-tree storage engine handles high write throughput efficiently, and its partition-based data model naturally shards across nodes for horizontal scaling. A reasonable schema would partition by a hash of the post ID and cluster by creation timestamp:

    ```sql
    CREATE TABLE posts (
        post_id timeuuid PRIMARY KEY,
        author_id text,
        content text,
        media_urls list<text>,
        created_at timestamp
    );
    ```

    For secondary lookups (for example, posts by a specific author), you would create a separate materialized view or table keyed by author_id.

2.  **Feed generation strategy:** Push-based (fan-out on write). When a user creates a post, the system writes the post to the primary database and simultaneously inserts a reference (post ID and timestamp) into the feed of every follower. This makes feed reads extremely fast — a single lookup keyed on the follower's user ID returns the 50 most recent entries. The push-based approach shifts the computational cost from read time (which is user-facing and latency-sensitive) to write time (which can be processed asynchronously by background workers).

3.  **Feed storage:** Redis sorted sets. Each user's feed is a sorted set keyed by `feed:{user_id}`. Each member is a post ID, and the score is the post's Unix timestamp. When a post is created, fan-out workers push the post ID and timestamp into the sorted set of each follower using `ZADD`. When the user opens their app, `ZREVRANGEBYSCORE feed:{user_id} 0 +inf LIMIT 0 50` returns the 50 most recent post IDs in roughly 1-10 milliseconds. A background job trims each sorted set to keep only the most recent 500 entries to bound memory usage. The application then fetches the full post content from Cassandra using the post IDs returned from Redis.

4.  **Storage calculation:** 10 million users times 500 entries per sorted set equals 5 billion entries. Each entry stores a post ID (roughly 8 bytes for a numeric ID, or 16 bytes for a UUID) and a score (8 bytes for a double-precision timestamp). Redis overhead is approximately 40 bytes per entry on 64-bit systems. So 5 billion entries times approximately 56 bytes equals 280 GB. If you trim to 200 entries per user, the storage drops to 10 million times 200 times 56 bytes = approximately 112 GB. This is feasible for a Redis cluster with 8-12 shards, each hosting roughly 10-14 GB of data.

5.  **Anti-pattern:** Fan-out on write for celebrity users. A user with 50 million followers would cause 50 million feed-update writes for each of their posts, which could saturate the system. The solution is to handle high-follower accounts with a hybrid approach: for users with more than a follower threshold (for example, 10,000 followers), use a pull-based approach where the feed for those posts is generated on read by merging a small precomputed celebrity feed with the push-based feed of regular accounts. This is exactly the approach used by Twitter (X) for its timeline architecture -- celebrity tweets are fetched separately and merged into the timeline at read time rather than being pushed to millions of feeds on every tweet.

    **Extension questions for deeper study:** How would the design change if the platform adds a "re-share" feature, where re-sharing a post should also fan out to the re-sharer's followers but only if they have not already seen the original post? How would you handle feed hydration (fetching full post content from Cassandra using the post IDs stored in Redis) at 10 million users without overwhelming Cassandra? Would you batch the hydration requests, or use a cache for the post content?

    **Fan-out worker architecture:** The fan-out workers themselves need careful design. When a post is created, the producer publishes a message to a Kafka topic (for example, `fanout-queue`) containing the post ID and author ID. Multiple fan-out worker instances consume from this topic in parallel. Each worker fetches the author's follower IDs from a graph database or a key-value store (for example, a Redis set of followers keyed by `followers:{author_id}`), batches them into groups of 100-500, and issues `ZADD` commands to the Redis cluster. This batching prevents overwhelming Redis with 50 million individual commands. If a worker fails mid-batch, the Kafka message is not committed, and another worker retries the fan-out. Idempotency is important: if `ZADD` is called twice with the same member and score, the second call is a no-op, so retries are safe.

### Challenge 5: Design the Data Layer for a Multi-Tenant SaaS Analytics Dashboard

**Time:** 15 minutes

You are building the data layer for a SaaS platform that provides real-time analytics dashboards to 5,000 business customers. Each customer sees aggregated metrics about their own data: page views, conversion rates, revenue, and user sessions. Customers can run ad-hoc queries on their data with filters by date range, region, device type, and campaign.

**Requirements**

- **Tenancy:** Each customer's data must be completely isolated. A query from customer A must never leak data from customer B.
- **Query latency:** Dashboard queries must return in under 2 seconds. Ad-hoc queries may take up to 10 seconds.
- **Data volume:** Each customer ingests 10 GB of raw event data per day. Events arrive as JSON blobs with varying schemas (different customers track different custom events).
- **Ingestion rate:** 50,000 events per second total across all customers.
- **Schema evolution:** Customers can add new custom dimensions without contacting your team. The schema must support this without migrations.

**Your task**

Answer all of the following:

1.  Choose the primary storage technology for raw event data. SQL or NoSQL? Justify with respect to schema flexibility and ingestion rate.
2.  Choose the query-layer technology for the dashboards. Should it be the same database or a different one? If different, describe how data moves between them.
3.  Describe the tenant isolation strategy in detail. How do you guarantee that customer A cannot query customer B's data, at both the storage layer and the query layer?
4.  Address the schema evolution requirement. How can customers add new custom fields without a migration?

**Evaluation Criteria**

- Choice for raw event storage must handle high-ingestion, schema-on-read data. NoSQL (or a columnar database) is expected.
- The query layer should be different from the ingestion layer — raw JSON blobs are not efficient for ad-hoc analytical queries. An ELT pipeline that transforms raw events into a columnar format (Parquet in a data lake, or a columnar database table) is expected.
- Tenant isolation must be discussed at both the storage layer (separate tables, separate databases, or partition keys) and the query layer (connection-level credentials or query rewriting with tenant ID filters).
- Schema evolution: columnar solutions such as BigQuery or Snowflake support nullable columns and automatic schema detection; a document database with schema-on-read also satisfies this requirement.

**Common Mistakes**

- Storing raw JSON in a columnar database directly — most columnar databases do not support efficient point-ingestion of individual JSON events at 50,000 events per second.
- Using a single SQL database with a tenant_id column for 5,000 tenants — without careful partitioning or separate databases, one tenant's heavy query can degrade performance for all other tenants. The noisy-neighbor problem becomes severe at 5,000 tenants.
- Proposing a solution that requires a schema migration for every new customer dimension when the requirement explicitly forbids that.
- Ignoring the query latency requirement of under 2 seconds for dashboards by suggesting direct queries on raw event data.
- Using a pull-based ETL (Extract, Transform, Load) approach where the analytical store reads directly from the ingestion store on the critical path, adding latency and load to the ingestion layer.
- Proposing different credentials for every tenant at the dashboard layer without addressing credential management at scale — managing 5,000 sets of database credentials introduces its own operational burden.

**Model Answer**

1.  **Raw event storage:** NoSQL — use a document database such as MongoDB or a time-series database such as InfluxDB, or stream raw events into a message queue (Kafka) and then land them as JSON files in object storage (S3 or equivalent). The varying schemas across customers make a fixed relational schema impractical. The 50,000 events per second ingestion rate is manageable with a horizontally sharded NoSQL cluster. However, raw JSON stores are not suitable for the dashboard queries, which brings us to point 2. The total daily data volume is 5,000 customers times 10 GB per customer = 50 TB per day, which further reinforces the need for a scalable, schema-on-read storage layer.

2.  **Query layer:** A different database — a columnar data warehouse such as ClickHouse, BigQuery, or Snowflake. An ELT pipeline (for example, using Apache Spark or a streaming processor) reads raw events from the ingestion store, flattens the common dimensions into columns, and stores the custom dimensions as a variant or JSON column. This separation is essential because columnar databases excel at aggregations over large scans (sum of revenue, count of page views) while NoSQL databases excel at high-ingestion point lookups. The pipeline runs every 5-15 minutes, meaning dashboards see data that is at most 15 minutes old -- acceptable for most analytics use cases. If near-real-time dashboards are required (under 1 minute latency), you would use a streaming ingestion approach directly into the columnar store (for example, ClickHouse's Kafka engine or BigQuery's streaming inserts).

3.  **Tenant isolation at the storage layer:** In the ingestion store, each customer's events are written to a separate partition or separate collection/table. In the columnar warehouse, each customer's data lives in a separate table or a separate schema. Tenant isolation at the query layer: each customer's dashboard connects to the warehouse using credentials scoped to only their table or schema. For ad-hoc queries, a query proxy rewrites every query to include a `WHERE tenant_id = :customer_tenant_id` clause, and the database row-level security policies enforce that no query can scan rows belonging to another tenant. Regular penetration testing verifies that tenant A cannot access tenant B's data through any API endpoint.

    The credential-based approach is the strongest isolation mechanism because even if a bug in the application layer accidentally requests tenant B's data, the database connection itself has no permission to access it. Row-level security (RLS) adds a defense-in-depth layer: even if a query is constructed that lacks the tenant filter, RLS policies on the table automatically append the filter based on the current user's tenant assignment.

4.  **Schema evolution:** The columnar warehouse stores custom dimensions as a single VARIANT or JSON column (for example, `custom_dimensions`). Customers send their new fields as part of the event JSON. The ELT pipeline passes those through without modification. The warehouse does not need a DDL migration — the variant column accepts any structure. The dashboard query layer exposes these custom fields through a dynamic schema discovery mechanism: when a customer adds a new dimension, the dashboard generator reads the schema of the variant column and offers the new field as a filter option. This approach satisfies the requirement that customers add new custom dimensions without contacting your team.

    A concrete example: Customer A decides to start tracking `utm_campaign` as a new dimension. They simply include `{"utm_campaign": "winter_sale"}` in their event JSON from that point forward. The ELT pipeline passes the new field through to the `custom_dimensions` variant column. The next time the dashboard loads, it scans the variant column for the presence of `utm_campaign` and automatically adds it as an available filter. No schema change, no deployment, no contact with your team.

    **Extension questions for deeper study:** What happens when a customer accidentally sends 100x their normal event volume (a misconfigured SDK)? Does your architecture isolate other customers from the impact? How would you implement rate limiting or budget-based ingestion quotas per tenant without adding latency to the write path? What is your strategy for cold-start queries when a new customer first logs into the dashboard and there is no data in the columnar warehouse yet?

**ELT pipeline in detail:** The pipeline reads raw events from the ingestion store every 5 minutes using an Apache Spark batch job. For each batch, it parses the JSON events, extracts the common fields (timestamp, customer_id, event_type, page_url, session_id) into dedicated columns, and writes the remaining fields into the `custom_dimensions` VARIANT column. The transformed data is written to the ClickHouse table using a bulk insert with 10,000-row batches. The pipeline tracks the last processed timestamp in a metadata table, so if the pipeline fails mid-batch, it resumes from the last successful checkpoint. In the event of a schema change (a new common field becomes mandatory), the pipeline is updated by adding the new column to the extraction logic and to the target ClickHouse table schema via an `ALTER TABLE` statement. The deployment of the updated pipeline is a rolling update that causes no data loss.

### Challenge 6: Design a Real-Time Leaderboard

**Time:** 15 minutes

You are building the data layer for a real-time game leaderboard that serves 5 million daily active players. Players earn points by completing challenges, and the leaderboard shows the top 200 players by total points, updated in real time. Each player also sees their own rank and the 10 players above and below them.

**Requirements**

- **Score updates:** 10,000 score updates per second. Each update is a single player's score increasing by some number of points (between 1 and 1,000).
- **Leaderboard read:** The top 200 leaderboard must load in under 100 milliseconds.
- **Player rank lookup:** A player's own rank and neighbors must load in under 100 milliseconds.
- **Data durability:** If the system crashes, no more than 1 second of score updates may be lost.
- **Global scope:** One leaderboard for all players. (No regional or friend-list leaderboards for this version.)

**Your task**

Answer all of the following:

1.  Choose the primary database technology for the leaderboard. Would you use SQL or NoSQL? Name a specific product and justify.
2.  Describe the exact data structure you would use to store scores and ranks. Provide a schema or document structure.
3.  Explain how you would handle the write path: when a player earns points, what operations happen and in what order?
4.  Explain how you would handle the read path: how does the system serve the top 200 and a player's rank-and-neighbors in under 100 milliseconds?
5.  Discuss how you would persist the leaderboard for durability without sacrificing the real-time requirement.

**Evaluation Criteria**

- The choice must address both the write throughput (10,000 updates per second on a single sorted structure) and read latency (under 100 ms). A sorted set data structure is the expected answer.
- The write path must show awareness of the update ordering: increment the player's score, re-sort, and update rankings. If using a sorted set, this is a single `ZINCRBY` operation.
- The read path must describe `ZREVRANGE` for top 200 and `ZREVRANK` plus `ZREVRANGE` for rank neighbors.
- Durability must be addressed — in-memory structures (Redis by default) can lose data on crash. A persistence strategy such as Redis AOF with fsync every second, or a write-ahead log plus periodic snapshots, is expected.
- The answer should consider what happens during peak load (for example, a tournament weekend with 50,000 updates per second) and whether the database choice still holds.
- The answer should address the tie-breaking problem: what happens when two players have the same score, and how does the system determine who ranks higher?

**Common Mistakes**

- Choosing a SQL database with an indexed `score` column — even with an index, updating 10,000 scores per second on the same table causes contention on the index structure, and a query for rank requires counting all rows with higher scores, which is an O(n) operation at 5 million rows.
- Choosing MongoDB with a sorted index on `score` — the `$rank` operator is not built-in and requires a count query of all higher-scoring documents, which also becomes slow at scale.
- Using Redis without a persistence strategy — Redis stores data primarily in memory, and the default snapshot (RDB) policy of saving every 5 minutes could lose up to 5 minutes of score updates.
- Storing each player's rank as a denormalized field and updating it on every score change — this creates a massive write amplification where a single score update potentially causes millions of rank updates.

**Model Answer**

1.  **Database technology:** NoSQL — Redis with the sorted set data structure. Redis's `ZINCRBY`, `ZREVRANGE`, and `ZREVRANK` operations are purpose-built for leaderboards. A single sorted set can hold 5 million members and perform each operation in O(log n) time. 10,000 updates per second is well within a single Redis instance's capability (a modern instance can handle 100,000+ operations per second for simple commands). However, for durability, you must configure Redis properly (see point 5).

2.  **Data structure:** A single Redis sorted set keyed by `leaderboard:global`. Each member is the player's user ID (as a string). The score is the player's total points as an integer. Redis internally maintains the set as a skip list and hash table, giving O(log n) for both insert and rank queries. Example after a few operations:

    ```
    ZADD leaderboard:global 1500 "player_alpha"
    ZINCRBY leaderboard:global 500 "player_beta"
    ZREVRANGE leaderboard:global 0 199 WITHSCORES
    ```

3.  **Write path:** When a player completes a challenge and earns points, the application issues a single `ZINCRBY leaderboard:global <points> "<player_id>"` command. Redis atomically increments the player's score and repositions them in the sorted set. This is the entire write operation — no additional indexes to update, no rank to recalculate. The operation is single-threaded in Redis, so there is no race condition. After the write, the application can optionally publish a message to a stream or pub/sub channel so that connected clients (websocket servers) get notified of ranking changes.

    **Batching for high throughput:** At 10,000 updates per second, each individual `ZINCRBY` command is fast (taking roughly 0.01 milliseconds on modern hardware). However, network round-trips between the application and Redis can become the bottleneck. To optimize further, the application can pipeline commands: instead of sending one `ZINCRBY` at a time and waiting for a response, it can send 100-500 commands in a batch over a single TCP connection and read all responses at once. This reduces network overhead from roughly 0.1 milliseconds per command to effectively zero marginal cost per command. At 10,000 updates per second, pipelining reduces total network overhead from approximately 1 second per second of wall time to negligible.

4.  **Read path:** For the top 200 leaderboard, `ZREVRANGE leaderboard:global 0 199 WITHSCORES` returns the top 200 players and their scores. For a player's rank and neighbors, `ZREVRANK leaderboard:global "<player_id>"` returns the player's rank (0-indexed), then `ZREVRANGE leaderboard:global max(0, rank-10) rank+10 WITHSCORES` returns the 10 players above and below. These operations complete in under 1 millisecond on a Redis instance with 5 million members. The read path never blocks the write path: Redis's single-threaded event loop processes commands in order, so a read command that arrives between two writes simply sees the state after the first write and before the second.

5.  **Durability:** Redis's default RDB snapshot (save every 5 minutes if 100 keys changed) would lose up to 5 minutes of score updates. Configure Redis AOF (Append-Only File) with `appendfsync everysec`. This logs every write operation to disk and fsyncs once per second, ensuring at most 1 second of data loss on crash. The AOF file grows over time; configure automatic rewrite (`auto-aof-rewrite-percentage 100` and `auto-aof-rewrite-min-size 64mb`) to prevent the file from consuming all available disk space. For additional safety, run Redis in a master-replica configuration with automatic failover (Redis Sentinel or Redis Cluster). The replica acknowledges writes, so if the master crashes, the replica has the data. For catastrophic failure recovery, periodically take RDB snapshots from the replica and store them in durable object storage (S3 or equivalent).

    **Extension questions for deeper study:** How would you handle a feature request for weekly leaderboards that reset every Monday? Would you use a separate sorted set per week, or a single sorted set with a weekly prefix in the member ID? How do you prevent score inflation from a cheating player who exploits a bug to earn 1 billion points in one second? What happens to the leaderboard read latency when the most recent write causes a massive score change for a single player (for example, a player was at rank 2,000,000 and jumps to rank 1)?

**Edge case -- tie scores:** If two players have the same score, Redis's sorted set uses lexicographic ordering on the member string as a tiebreaker. This means that if "player_alpha" and "player_beta" both have 5,000 points, "player_alpha" ranks higher because "a" comes before "b". If you need a different tiebreaker (for example, earlier achievement wins), you can encode the tiebreaker into the score as a fractional component: `score = points + (1.0 / max_players) * (max_players - join_order)` or use a composite score approach. The simplest production-grade approach is to append a timestamp suffix to the score decimals so that earlier achievers rank higher among tied scores.

### Challenge 7: Hybrid Architecture — The Transactional Analytics Problem

**Time:** 30 minutes

You are the principal engineer at a growing fintech startup. The company's core product is a payment processing platform that handles 2 million transactions per day. The system has two conflicting data needs:

- **Transactional need:** Every payment must be recorded with ACID guarantees. Double-spending or lost transactions are unacceptable. The transaction write path must sustain 500 writes per second with p99 latency under 50 milliseconds.
- **Analytical need:** The finance team runs ad-hoc queries on the transaction data — daily revenue reports, customer lifetime value calculations, fraud detection queries that scan millions of rows — and these queries take 30-60 seconds on the current setup. When the finance team runs a heavy query at end of month, transaction processing slows down and users see payment failures.

The current architecture uses a single PostgreSQL instance for both workloads. The transaction table has grown to 200 million rows. Financial analysts run aggregate queries directly on the production database.

**Your task**

Design a data architecture that separates the transactional and analytical workloads without compromising either. Address all of the following:

1.  **Separation strategy:** Propose a concrete architecture that isolates the transactional workload from the analytical workload. Name specific database technologies for each side.
2.  **Data synchronization:** Describe how data moves from the transactional store to the analytical store. What consistency guarantees do you provide? How do you handle late-arriving data or corrections (for example, a refund processed 3 days after the original transaction)?
3.  **Query routing:** How does the application know which queries to send to which store? Describe the mechanism.
4.  **Operational considerations:** How do you handle schema changes? If you add a new column to the transaction table, how does that propagate to the analytical store?
5.  **Failure modes:** What happens if the analytical store is unavailable? What happens if the synchronization pipeline breaks for 2 hours? Does the transactional system stay up?

**Evaluation Criteria**

- The architecture must propose a clear split — typically PostgreSQL (or another OLTP database) for transactions and a columnar data warehouse (ClickHouse, BigQuery, Redshift) for analytics.
- Data synchronization must be change data capture (CDC) based — Debezium, logical replication, or a similar mechanism. Batch ETL is acceptable if the latency requirement is relaxed, but the answer must discuss the trade-offs.
- Query routing must be explicit — separate connection strings, separate services, or a query router that inspects the SQL and routes accordingly.
- Schema evolution must be addressed: how new columns flow from source to target without breaking the pipeline.
- Failure modes must cover both the analytical store being down (transactional system stays up) and the CDC pipeline being down (batched catch-up on recovery).

**Common Mistakes**

- Proposing read replicas of PostgreSQL for analytics — read replicas solve read-load distribution but do not isolate the write path. A heavy analytical query on a replica still competes for CPU and I/O on the primary's replication stream, and the replica's query contention can delay replication acknowledgments.
- Using a single database with resource governor or query prioritization — at 200 million rows, full-scan analytical queries inherently use significant I/O and CPU that cannot be eliminated by query prioritization alone.
- Proposing a Lambda architecture (batch + stream) without addressing how corrections (refunds) are handled — corrections become particularly difficult when batch and stream paths produce different aggregations.
- Ignoring the p99 latency requirement of under 50 milliseconds for transactions — any architecture that adds synchronous latency to the write path (for example, dual-write to both databases) violates this requirement.

**Model Answer**

1.  **Separation strategy:** Run PostgreSQL as the transactional (OLTP) store. Run ClickHouse as the analytical (OLAP) store. PostgreSQL handles all payment writes with ACID guarantees and single-row point lookups. ClickHouse handles all aggregation and scan-heavy analytical queries. These are physically separate clusters on separate hardware. The transactional system remains isolated from analytical query load — a heavy analytical query consuming 100% CPU on the ClickHouse cluster has zero impact on payment processing.

    Why ClickHouse instead of a general-purpose data warehouse: ClickHouse's columnar storage, vectorized query execution, and aggressive compression (10:1 or better for financial data) make it ideal for the finance team's workload. A query like "total revenue per day for the last 90 days" on 200 million rows completes in under 1 second on ClickHouse, compared to 30-60 seconds on PostgreSQL.

2.  **Data synchronization:** Use change data capture (CDC) via Debezium connected to PostgreSQL's WAL (write-ahead log). Debezium streams every insert, update, and delete to Apache Kafka. A Kafka consumer reads the change events, transforms them into ClickHouse-compatible row formats, and inserts them into ClickHouse tables. The typical end-to-end latency is under 1 second. Consistency is at-least-once: the CDC pipeline may deliver duplicates, so ClickHouse's `ReplacingMergeTree` engine deduplicates by the transaction's primary key.

    For corrections (refunds processed days later), the CDC pipeline captures the UPDATE on the original transaction row (or the INSERT of the refund row, depending on your schema). ClickHouse's `ReplacingMergeTree` replaces the old row with the updated row when it does a final merge, or you use a `CollapsingMergeTree` where refunds emit a negative entry that cancels the original for aggregation purposes. A concrete example: transaction ID 12345 was originally `$100` and later refunded. The CDC pipeline emits the updated row with `amount = -$100`, and the `CollapsingMergeTree` engine sums both entries (100 + -100 = 0) for accurate aggregations.

    The Kafka layer is critical for reliability. It acts as a durable buffer that decouples the source database from the analytical store. If ClickHouse is down for maintenance, Kafka retains the change events (configure retention to 72 hours). When ClickHouse comes back, the consumer replays from the last committed offset, catching up to real time.

3.  **Query routing:** The application uses two separate connection pools. All payment processing code (insert transaction, check balance, read a single transaction) connects through the PostgreSQL pool. All analytics dashboards, reporting endpoints, and ad-hoc query tools connect through the ClickHouse pool. The application layer makes the routing decision based on the operation type. A lightweight query router service in front of the databases could inspect the SQL — if it starts with `SELECT` and references aggregate functions or scans over date ranges without a specific transaction ID, route to ClickHouse; otherwise route to PostgreSQL. However, explicit routing at the application layer is more reliable because it avoids the risk of the router misclassifying a query.

    For example, a microservice called `payment-service` always uses the PostgreSQL connection pool. A separate `reporting-service` always uses the ClickHouse connection pool. They are separate deployments with separate database credentials, making accidental cross-talk impossible.

4.  **Operational considerations — schema changes:** Add the new column to the PostgreSQL table first. Debezium automatically captures the new column in the change events if it has been configured to include all columns. The Kafka topic schema (AVRO or JSON schema) is updated. The ClickHouse consumer reads the new field from the event. In ClickHouse, you add the new column to the target table using `ALTER TABLE ... ADD COLUMN`. The consumer then writes the new column value into ClickHouse. Existing rows in ClickHouse are backfilled with default values (NULL or 0). This process requires no downtime on either side if the column is nullable. For non-nullable columns with a default value expression, you must backfill the ClickHouse table before enforcing the NOT NULL constraint.

    The order of operations matters: always add the column to PostgreSQL first, then update the target ClickHouse schema, then let the CDC pipeline flow. If you add the column to ClickHouse first without a corresponding source column, the consumer will write NULL values for existing rows and populate it from future change events.

5.  **Failure modes:** If the analytical store (ClickHouse) goes down, the transactional system (PostgreSQL) stays up and processes payments normally. The analytics dashboards will return errors or stale data, but payments are unaffected. When ClickHouse recovers, the CDC pipeline catches up from the Kafka offset, replaying all changes that occurred during the downtime. If the CDC pipeline breaks for 2 hours, Kafka retains the unprocessed change events (assuming sufficient retention is configured — 72 hours is a reasonable default). Once the pipeline is repaired, it resumes consuming from the last committed offset. In both failure scenarios, the transactional PostgreSQL system never pauses or slows down. The only risk is that after a prolonged CDC outage, the initial catch-up may take some time, affecting analytics freshness but not transaction correctness.

    A subtle failure mode to consider: what if the CDC pipeline processes events out of order? For example, an UPDATE arrives before its corresponding INSERT. The ReplacingMergeTree engine handles this gracefully because the merge process replaces rows by primary key regardless of the order in which they were inserted. The final state after the merge is always correct as long as the last event for each primary key is the most recent one. However, you must ensure that the ClickHouse table engine is configured with the proper version column or ordering key to resolve conflicts deterministically.

**Extension questions for deeper study:** How would you handle data retention policies in this architecture? Transaction data must be retained for 7 years per regulatory requirements, but you only need the last 12 months available for fast queries in ClickHouse. How would you tier older data to cheaper storage without breaking the ability to run occasional year-over-year comparisons? Would you use ClickHouse's TTL-based data partitioning, or move cold data to Parquet files in S3 with a separate query engine?

**Anti-patterns to Avoid**

- **Dual-write from the application.** Writing to both PostgreSQL and ClickHouse from the application code (rather than using CDC) introduces a failure domain where one write succeeds and the other fails, leaving the system in an inconsistent state. CDC guarantees that every change committed to PostgreSQL is eventually reflected in ClickHouse without the application needing to manage two write paths.

- **Using PostgreSQL read replicas as the analytical store.** A read replica still shares the underlying storage and replication I/O with the primary. A heavy analytical query on the replica causes replication lag, which can delay the visibility of transactions on the replica and, in extreme cases, cause the replica to fall behind so far that the primary increases its WAL retention, potentially running out of disk space.

- **Running analytical queries during business hours.** Even with a separate analytical store, you should schedule the finance team's heavy queries (end-of-month revenue reports, lifetime value calculations) to run during off-peak hours or in a low-priority queue. A query that scans 200 million rows, even if isolated to ClickHouse, still consumes cluster resources that could affect other analytical queries. Modern data warehouses support workload management (resource pools, query queues) to address this.

### Challenge 8: Multi-Model Data Platform for a Content Management System

**Time:** 30 minutes

You are designing the data layer for a next-generation content management system (CMS). The CMS needs to support three fundamentally different data access patterns:

- **Pattern A — Content storage and retrieval:** Authors create documents with rich text, embedded images, custom metadata fields, and version history. Documents have varying schemas — a blog post has different fields than a product page. The system must support retrieving a single document by ID in under 20 milliseconds.
- **Pattern B — Full-text search:** Users search across all documents by keywords, phrases, and metadata filters. Search results must return in under 200 milliseconds. The index must cover 100 million documents.
- **Pattern C — Relationship queries:** Editors need to see a graph of related content — which documents link to which, which authors contributed to which documents, and which tags are connected to which content. Some queries traverse up to 5 hops (for example, "find all documents authored by people who edited documents tagged 'machine learning'").

**Requirements**

- The system must serve 1,000 read requests per second across all patterns.
- Content updates do not need to be reflected in search or relationship queries instantly — a delay of up to 30 seconds is acceptable.
- The database infrastructure must be manageable by a team of 3 platform engineers. Operational complexity is a real constraint.

**Your task**

Answer all of the following:

1.  **Database selection:** Propose a specific database for each pattern. You may use multiple databases from different families, or you may argue for a single multi-model database. Justify each choice with concrete reasoning about the data model and query patterns.
2.  **Synchronization:** If you use multiple databases, describe how data stays consistent across them. If you use a single database, explain how it satisfies all three patterns without degradation.
3.  **Operational complexity:** Assess the operational cost of your proposed architecture. How many databases does your team need to run, back up, monitor, and upgrade? If you chose multiple databases, defend the added complexity against the alternative of using a single multi-model database such as PostgreSQL with extensions (for example, pgvector, pg_search, or the JSONB data type).

**Evaluation Criteria**

- The answer must provide a concrete, defensible database choice for each pattern, not generic categories.
- If using multiple databases, the synchronization mechanism must be described in enough detail to evaluate its correctness and latency.
- The operational complexity discussion must be honest about trade-offs. Claiming that a single multi-model database is always simpler is not automatically correct — the answer must consider real operational pain points (extension compatibility, backup tooling, resource isolation).
- The answer must address the team size constraint (3 platform engineers).
- The answer should discuss the actual schemas or data models used for each pattern, not just the technology choice.
- The synchronization latency must be within the 30-second window specified in the requirements, and the answer should state the expected latency explicitly.

**Common Mistakes**

- Proposing MongoDB for all three patterns because "it can do everything" — MongoDB has limited full-text search compared to Elasticsearch and no native graph traversal capabilities for multi-hop queries.
- Proposing a graph database (Neo4j) for all three patterns — graph databases handle relationship queries well but are not optimized for full-text search or high-throughput document retrieval by ID.
- Proposing PostgreSQL with all available extensions and claiming it handles all three patterns equally well — while technically possible, running a single PostgreSQL instance that simultaneously serves document lookups, full-text search, and graph queries under 1,000 req/s with mixed workloads puts severe pressure on shared resources (buffer pool, CPU, WAL), and the operational burden of tuning extensions that compete for the same memory and I/O can be higher than running purpose-built databases.
- Ignoring the operational cost constraint — a sprawling architecture with 6+ databases is not manageable by a 3-person team.

**Model Answer**

1.  **Database selection:**

    - **Pattern A (Content storage):** PostgreSQL with JSONB columns. PostgreSQL provides ACID transactions, point-lookup by ID in under 1 millisecond with a primary key index, and schema flexibility via JSONB columns for varying document metadata. Version history can be stored in a separate `document_versions` table linked by foreign key, or by using the document ID with a version suffix in the same table. A typical schema:

    ```sql
    CREATE TABLE documents (
        document_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
        slug VARCHAR(255) UNIQUE NOT NULL,
        title TEXT NOT NULL,
        body TEXT NOT NULL,
        metadata JSONB DEFAULT '{}',
        author_id INT REFERENCES authors(id),
        content_type VARCHAR(50) NOT NULL,
        created_at TIMESTAMPTZ DEFAULT NOW(),
        updated_at TIMESTAMPTZ DEFAULT NOW()
    );

    CREATE TABLE document_versions (
        version_id BIGSERIAL PRIMARY KEY,
        document_id UUID REFERENCES documents(document_id),
        body TEXT NOT NULL,
        metadata JSONB DEFAULT '{}',
        created_at TIMESTAMPTZ DEFAULT NOW()
    );

    CREATE INDEX idx_documents_metadata ON documents USING GIN(metadata);
    CREATE INDEX idx_documents_updated ON documents(updated_at);
    ```

    The `GIN` index on `metadata` enables efficient querying inside the JSONB structure, for example finding all documents where `metadata->>'status' = 'published'`. This gives you schema flexibility without sacrificing query performance. New metadata fields can appear in any document without a migration -- they simply become new keys in the JSONB object. The GIN index automatically covers them.

    - **Pattern B (Full-text search):** Elasticsearch. Elasticsearch's inverted index and relevance scoring are purpose-built for full-text search across 100 million documents. Multi-field filtering (date range, author, tags) combined with keyword search is a single query. Elasticsearch handles this at a far lower latency than PostgreSQL full-text search for the same scale, and it does not consume PostgreSQL's buffer pool or CPU. The Elasticsearch index mapping is designed to support both full-text queries and structured filters:

    ```json
    {
      "settings": { "number_of_shards": 10, "number_of_replicas": 2 },
      "mappings": {
        "properties": {
          "document_id": { "type": "keyword" },
          "title": {
            "type": "text",
            "analyzer": "english",
            "fields": { "keyword": { "type": "keyword" } }
          },
          "body": { "type": "text", "analyzer": "english" },
          "author_name": { "type": "text" },
          "tags": { "type": "keyword" },
          "content_type": { "type": "keyword" },
          "created_at": { "type": "date" },
          "metadata": { "type": "flattened" }
        }
      }
    }
    ```

    The `flattened` type for metadata allows arbitrary key-value pairs without defining each one in the mapping, similar to JSONB in PostgreSQL. The 10-shard configuration distributes the 100 million documents evenly across the cluster, and 2 replicas provide high availability.

    - **Pattern C (Relationship queries):** PostgreSQL with recursive CTEs (Common Table Expressions), using a `links` table and a `document_tags` association table. For the multi-hop query example ("documents authored by people who edited documents tagged 'machine learning'"), a recursive CTE traverses the graph in under 50 milliseconds if the relevant columns are indexed. A dedicated graph database such as Neo4j would be faster for deep traversals (10+ hops), but the requirement is 5 hops at 1,000 queries per second, which PostgreSQL recursive CTEs handle comfortably. This avoids introducing a third database type.

    The supporting schema for relationship queries:

    ```sql
    CREATE TABLE document_links (
        source_id UUID REFERENCES documents(document_id),
        target_id UUID REFERENCES documents(document_id),
        link_type VARCHAR(50),
        PRIMARY KEY (source_id, target_id, link_type)
    );

    CREATE TABLE document_contributors (
        document_id UUID REFERENCES documents(document_id),
        person_id INT REFERENCES authors(id),
        role VARCHAR(50),
        PRIMARY KEY (document_id, person_id, role)
    );

    CREATE TABLE document_tags (
        document_id UUID REFERENCES documents(document_id),
        tag VARCHAR(100),
        PRIMARY KEY (document_id, tag)
    );

    CREATE INDEX idx_links_target ON document_links(target_id);
    CREATE INDEX idx_tags_tag ON document_tags(tag);
    ```

    The recursive CTE for the 5-hop query:

    ```sql
    WITH RECURSIVE machine_learning_editors AS (
        -- Base: find editors of documents tagged 'machine learning'
        SELECT DISTINCT dc.person_id
        FROM document_contributors dc
        JOIN document_tags dt ON dc.document_id = dt.document_id
        WHERE dt.tag = 'machine learning' AND dc.role = 'editor'
    ),
    related_authors AS (
        -- Hop 1-5: traverse document_links and find authors
        SELECT DISTINCT dc2.person_id, 1 AS depth
        FROM machine_learning_editors mle
        JOIN document_contributors dc1 ON mle.person_id = dc1.person_id
        JOIN document_links dl ON dc1.document_id = dl.source_id
        JOIN document_contributors dc2 ON dl.target_id = dc2.document_id
        WHERE dc2.role = 'author'
        UNION
        SELECT DISTINCT dc2.person_id, r.depth + 1
        FROM related_authors r
        JOIN document_contributors dc1 ON r.person_id = dc1.person_id
        JOIN document_links dl ON dc1.document_id = dl.source_id
        JOIN document_contributors dc2 ON dl.target_id = dc2.document_id
        WHERE dc2.role = 'author' AND r.depth < 5
    )
    SELECT DISTINCT person_id FROM related_authors;
    ```

    This query completes in under 50 milliseconds with proper indexing, and it handles up to 5 hops as required. For reference, a graph database would complete the same query in under 10 milliseconds, but the added operational complexity of running a third database (Neo4j) for a 40-millisecond improvement is not justified for a 3-person team.

    Final architecture: **PostgreSQL + Elasticsearch**. Two databases, not three. PostgreSQL handles patterns A and C; Elasticsearch handles pattern B.

2.  **Synchronization:** When a document is created or updated in PostgreSQL, the application writes to PostgreSQL first (the source of truth). After the PostgreSQL write succeeds, it publishes a message to a Redis stream or a lightweight message queue (or alternatively, listens to a PostgreSQL logical replication slot via a CDC tool). A background worker consumes the message, transforms the document into a searchable document structure, and indexes it in Elasticsearch. The target latency for indexing is under 10 seconds, well within the 30-second requirement. For deletes, the same pipeline removes the document from the Elasticsearch index. Consistency is eventually consistent with a 10-second window.

    The transformation step is important: Elasticsearch documents are denormalized for search performance. For example, a document stored in PostgreSQL as a blog post with a foreign key to an author table is indexed in Elasticsearch as a single document that includes the author's name, bio, and profile picture URL. This avoids join queries during search. The mapping in Elasticsearch would look roughly like:

    ```json
    {
      "mappings": {
        "properties": {
          "id": { "type": "keyword" },
          "title": { "type": "text", "analyzer": "english" },
          "body": { "type": "text", "analyzer": "english" },
          "author_name": { "type": "text" },
          "tags": { "type": "keyword" },
          "created_at": { "type": "date" }
        }
      }
    }
    ```

3.  **Operational complexity:** The team runs two database systems: PostgreSQL (for content and relationships) and Elasticsearch (for search). Operational tasks include:

    - **PostgreSQL:** Regular backup (pg_dump or WAL-based continuous archiving), monitoring (connection count, replication lag if read replicas are added), minor version upgrades, vacuum tuning for the JSONB-heavy workload.
    - **Elasticsearch:** Index management (rolling over old indices for performance), snapshot-based backups to object storage, cluster health monitoring, shard rebalancing.
    - **Synchronization pipeline:** A small consumer service (Go or Node.js) that reads from the queue and writes to Elasticsearch. This service needs monitoring (queue depth, indexing error rate) but does not require a dedicated on-call rotation.

    This is a two-database architecture. The operational load is well within the capacity of a 3-person platform team. The alternative of using PostgreSQL with all extensions (pg_search for full-text, plus recursive CTEs for graph) would eliminate one database but create a different kind of operational risk: a heavy search query over 100 million documents in pg_search would consume PostgreSQL's shared buffers and CPU, degrading document lookups (Pattern A) and graph queries (Pattern C). Resource isolation is harder when everything runs in one process.

    The two-database architecture trades a moderate increase in operational surface for clear resource isolation and purpose-optimized performance. Given the team size of 3, this trade-off is sound — each engineer owns one database and the sync pipeline, with clear boundaries and mature tooling for both systems. A practical split of responsibilities: one engineer specializes in PostgreSQL (schema design, backup, performance tuning), one engineer specializes in Elasticsearch (index management, cluster scaling), and the third owns the synchronization pipeline and application integration. All three share on-call rotations for the overall platform.

    **Extension questions for deeper study:** What happens when a content editor renames a tag from "machine-learning" to "ml" -- does the Elasticsearch index get updated immediately, or do search results return stale tag names until the next sync cycle? How would you implement a "related content" feature that returns documents linked within 3 hops of the current document, and should that query go to PostgreSQL (recursive CTE) or Elasticsearch (parent-child relationships)? What is the backup and recovery strategy for Elasticsearch when it holds the only copy of the searchable content (the source of truth is PostgreSQL, but reindexing 100 million documents from scratch takes hours)?

**When to reconsider the single-database approach:** If the team shrinks to 1-2 people, or if the search workload is trivial (under 1 million documents, basic keyword matching only), PostgreSQL with pg_search becomes the more pragmatic choice. The operational simplicity of managing one database offsets the performance trade-offs at lower scale. For 100 million documents with full-text search requirements, the two-database approach is the right call.

**Anti-patterns to Avoid**

- **Polyglot persistence without discipline.** Adding a separate database for every microservice (MongoDB for content, Neo4j for relationships, Elasticsearch for search, Redis for caching, Cassandra for logs) creates an unmanageable operational burden for a small team. The goal is to minimize the number of databases while satisfying the core access patterns.

- **Using the same database for OLTP and search at scale.** Running full-text search queries over 100 million documents inside your transactional database will eventually cause performance degradation for both workloads, no matter how well you tune it.

- **Synchronous dual-writes.** Writing to PostgreSQL and Elasticsearch in the same transaction using a two-phase commit or distributed transaction would violate the 20-millisecond latency requirement for content writes and would introduce a failure domain where a slow Elasticsearch write blocks PostgreSQL.

- **Storing search data in PostgreSQL and building the search index from application logs.** This adds unnecessary complexity and latency. A dedicated search index with a CDC or queue-based pipeline is simpler to reason about and operate.
