# SQL vs NoSQL — Under the Hood

## Internal Architecture

A database is a collection of subsystems that work together. Each subsystem handles a specific concern — parsing, caching, indexing, logging, replication — and each has different trade-offs in SQL vs NoSQL databases.

### How the Components Interact

When you send a query:

1. The **Connection Manager** authenticates you and assigns a backend process or thread.
2. The **Parser** converts your query text into an Abstract Syntax Tree (AST) — a tree of operations.
3. For SQL: the **Query Optimizer** generates 3 to 5 candidate execution plans, estimates their costs using table statistics, and picks the cheapest. For NoSQL: the optimizer simply picks the best index.
4. The **Execution Engine** runs the chosen plan. It calls into the **Buffer Pool** (which caches data pages in memory) and the **Storage Engine** (which manages on-disk data structures).
5. The **Write-Ahead Log** records every mutation for crash recovery. The **Replication Log** streams changes to replicas.
6. The **Storage Engine** organizes data on disk — B-tree pages for PostgreSQL, SSTable files for Cassandra, hash table slots for Redis.

## Data Structures and Algorithms

The choice of data structures is the single most important technical decision a database makes. It determines read speed, write speed, memory usage, and compaction behavior.

### B-Tree (SQL databases, MongoDB)

A B-tree is a self-balancing tree that keeps data sorted and maintains logarithmic search, insert, and delete operations. Every node contains between `t-1` and `2t-1` keys (where `t` is the branching factor, typically 100 to 500).

**B-tree characteristics:**
- Read: O(log_B N) where B is the branching factor. For N=1 billion and B=500, only 3-4 node reads.
- Write: O(log_B N) — find the leaf, write the page. In-place updates can cause page splits (expensive).
- Space: each node is a full page (typically 8KB or 16KB). Up to 50% empty space from page splits.
- Fragmentation: over time, random updates fragment pages. VACUUM (PostgreSQL) or OPTIMIZE TABLE (MySQL) reclaims space.
- Range scans: efficient because leaf nodes form a doubly-linked list.

### LSM Tree (Cassandra, RocksDB, ScyllaDB)

A Log-Structured Merge Tree consists of an in-memory memtable and a hierarchy of on-disk SSTable files. Writes go to the memtable first — a sorted in-memory structure. When the memtable reaches its size threshold (typically 64-256MB), it flushes to disk as an immutable SSTable.

**LSM tree characteristics:**
- Write: O(1) sequential I/O. Writes are 10-100x faster than B-trees on spinning disks, 2-5x on SSDs.
- Read: O(N) in worst case — must check memtable, then each SSTable level. Bloom filters reduce most reads to O(1) by quickly ruling out keys that do not exist.
- Space: higher than B-tree because of multiple copies of the same key across levels. Compaction reclaims space but costs I/O.
- Compaction: the background merge process. There are strategies — size-tiered (Cassandra default), leveled (RocksDB default), and time-window (time-series data).
- Write amplification: each byte written may be written multiple times through compaction. Typical amplification: 10-30x for leveled compaction, 3-5x for size-tiered.

### Hash Table (Redis, Memcached)

Key-value stores use hash tables for O(1) lookup — the simplest possible access pattern.

**Redis hash table characteristics:**
- Read: O(1) average, O(N) during rehashing.
- Memory overhead: 40-60 bytes per key plus value storage. With 1 million keys, that is 40-60MB just for the hash table structure.
- Rehashing: when the load factor exceeds 1, Redis resizes the hash table. This is done incrementally (a few buckets per command) to avoid latency spikes.
- Eviction: when memory is full, Redis evicts keys using one of 8 policies — LRU, LFU, TTL, random, or no-eviction.

### Skip List (Redis Sorted Sets)

Redis uses skip lists for sorted set operations — a probabilistic alternative to balanced trees.

- Insert: O(log N) average, O(N) worst case.
- Range queries: O(log N + M) where M is the range size.
- Memory: each node has 1-32 forward pointers (average 2). Higher memory than a B-tree but simpler implementation.

### Merkle Tree (Cassandra anti-entropy)

Cassandra uses Merkle trees to detect inconsistency between replicas during repair operations.

- Each node builds a hash tree of its data by partition range.
- Comparing Merkle tree roots tells you if two replicas are consistent — no need to compare every row.
- If roots differ, you descend to the subtree that differs, identifying exactly which partitions need repair.

### SQL Read Lifecycle (PostgreSQL)

```sql
-- A simple read
SELECT * FROM orders WHERE id = 42;
```

The query follows this path:

1. Parser converts text to AST: `SELECT (targets: *) FROM (table: orders) WHERE (op: =, col: id, val: 42)`.
2. Optimizer checks if an index on `id` exists. If yes, it estimates the cost of index scan (about 3 node reads for 10M rows) vs sequential scan (about 10M row reads from disk). Index scan wins.
3. Execution engine calls the index access method: traverse B-tree root, internal node, leaf node.
4. The leaf node contains a TID (tuple ID) pointing to the exact location of the row in the heap.
5. The buffer pool fetches the heap page containing the row. If not in memory, it reads from disk.
6. MVCC visibility check: the row has a `xmin` (creating transaction ID) and `xmax` (deleting/updating transaction ID). The engine checks if the current transaction can see this version.
7. The row is returned to the client.

Total path: 3 to 6 I/O operations on cold cache. Under 1ms on warm cache.

### NoSQL Read Lifecycle (Cassandra)

```
SELECT * FROM orders WHERE order_id = 42;
```

The path:

1. Parser identifies the partition key `order_id = 42` and computes the token (consistent hash).
2. The coordinator node forwards the query to the node responsible for that token range.
3. The storage engine checks the bloom filter first — if it says the key does not exist, return immediately (no SSTable access needed).
4. If the bloom filter says the key might exist, check the memtable (in memory, sorted).
5. Check each SSTable in L0, then L1, then L2 — newest to oldest. For each SSTable, check a partition-level bloom filter, then the compression offset map to find the exact byte offset on disk.
6. When found, return to coordinator. Coordinator merges results from multiple replicas (if CLUALL or CL > 1).
7. Return to client.

Total path: potentially many SSTable checks in worst case. Bloom filters make this O(1) on average. Cold cache: 5-20ms. Warm cache: 1-3ms.

## Consistency and State Handling

### Crash Recovery

**SQL (PostgreSQL):** On a crash, PostgreSQL replays the WAL from the last checkpoint. All committed transactions are replayed. All uncommitted transactions are rolled back. The database reaches a consistent state. The recovery time depends on WAL segment size — typically seconds to minutes for most workloads.

**NoSQL (MongoDB):** On a crash, MongoDB replays its journal (WAL equivalent). Journals are flushed every 100ms by default. Data between journal flushes that was not committed is lost. MongoDB also has a "repair" mode for severe corruption, which rebuilds indexes and should be a last resort.

**NoSQL (Cassandra):** On a crash, Cassandra replays the commit log. Aesop-level (Cassandra's anti-entropy) repair detects inconsistency across replicas and fixes it. Hinted handoff (where one node stores writes meant for a downed node) replays when the node returns.

### Concurrent Writes and Write Conflicts

SQL databases handle concurrent writes through locking. PostgreSQL uses row-level locks — two transactions can update different rows simultaneously, but updating the same row serializes the second transaction behind the first.

NoSQL databases handle concurrent writes differently depending on the flavor:

- **MongoDB**: Document-level locking. Two concurrent updates to the same document cause one to wait for the other (WiredTiger engine). Causal consistency guarantees within a session.
- **Cassandra**: Last-writer-wins. Each write includes a timestamp. If two writes arrive at the same partition simultaneously, the one with the higher timestamp wins. If timestamps are equal, the higher value wins lexicographically. Writes never block, but data can be lost.
- **DynamoDB**: Conditional writes via optimistic locking. You read a document with its version number, then write with "if version == N" condition. If another writer incremented the version, your write fails and you retry.

### Stale Reads and Read-Your-Writes

### Read Repair and Anti-Entropy

NoSQL databases that use eventual consistency need mechanisms to fix divergent replicas. Two key approaches:

**Read repair**: When a client reads with a consistency level greater than ONE, the coordinator compares responses from multiple replicas. If one replica returns a stale value, the coordinator updates it with the correct value. This fixes inconsistency in the read path but only for data that is actively read.

**Anti-entropy repair**: A background process that compares replicas and fixes differences. In Cassandra, this uses Merkle trees — each node builds a hash tree of its data, then exchanges tree roots with neighbors. If roots differ, they descend the tree to find the specific partitions that need repair. Run `nodetool repair` regularly (weekly recommended).

## Performance Characteristics

### Latency by Operation Type

The following numbers are approximate p50 values for a single node with data fitting in RAM (cache hit rate 99%):

| Operation | PostgreSQL | MongoDB | Cassandra | Redis |
|-----------|-----------|---------|-----------|-------|
| Key lookup (p50) | 0.3ms | 0.2ms | 0.5ms | 0.05ms |
| Key lookup (p99) | 2ms | 1.5ms | 10ms | 0.3ms |
| Key lookup (p999) | 10ms | 8ms | 50ms | 1ms |
| Range scan 100 rows | 1ms | 3ms | 8ms | N/A |
| Write (single row) | 3ms | 2ms | 1ms | 0.05ms |
| Write (batch 100) | 8ms | 15ms | 2ms | 0.5ms |
| Transaction (2 ops) | 5ms | 4ms | N/A | N/A |
| Aggregation (1M rows) | 500ms | 2s | 5s | N/A |

**Key observations:**
- Redis dominates on latency because it is entirely in-memory.
- Cassandra's p999 read is high due to SSTable lookups across multiple levels.
- PostgreSQL's aggregation speed comes from its columnar-friendly storage and query optimizer.
- MongoDB's batch write latency is high because each document is validated individually.

### Throughput Scaling

Throughput depends on data size, query complexity, and hardware:

| Database | Single node writes/sec | 3-node cluster writes/sec | Scaling type |
|----------|----------------------|--------------------------|--------------|
| PostgreSQL | 15K | 15K (1 writer) | Vertical |
| MongoDB | 50K | 150K (sharded) | Horizontal |
| Cassandra | 200K | 800K | Linear |
| Redis | 500K | 1.5M (cluster) | Horizontal |

Cassandra's linear scaling — double the nodes, double the throughput — is the reason large-scale systems use it. PostgreSQL's single-writer limitation means you must shard at the application layer or use a managed service that handles sharding for you.

### Memory and Disk

| Resource | PostgreSQL per 100GB | MongoDB per 100GB | Cassandra per 100GB |
|----------|---------------------|-------------------|---------------------|
| Memory (warm cache) | 8-16GB | 12-24GB | 4-8GB |
| Disk (raw data) | 100GB | 100GB | 100GB |
| Disk (indexes) | 20-40GB | 30-60GB | 15-30GB |
| Disk (LSM overhead) | N/A | N/A | 30-50GB |
| Write amplification | 1-2x | 2-3x | 10-30x |

Cassandra's write amplification is the hidden cost. Each write byte may be written 10-30 times total due to compaction. On SSDs with a 300 TBW (terabytes written) endurance rating, Cassandra can wear out drives in 2-3 years at high write throughput (100GB/day).

## Configuration and Tuning

### PostgreSQL Tuning

The key PostgreSQL configuration parameters in `postgresql.conf`:

```
# Memory
shared_buffers = 4GB           # 25% of RAM — higher causes cache thrashing
effective_cache_size = 12GB    # OS page cache estimate — helps planner
work_mem = 64MB                # Per-operation sort memory — be careful, per connection
maintenance_work_mem = 1GB     # VACUUM, CREATE INDEX memory

# Write behavior
wal_level = replica            # Replication support
max_wal_size = 8GB             # WAL segment retention — too small = frequent checkpoints
min_wal_size = 2GB
checkpoint_completion_target = 0.9  # Spread checkpoint writes over 90% of interval

# Concurrency
max_connections = 100          # Each connection eats ~10MB RAM
```

**Aggressive tuning for high throughput:**

```
# Use even more memory
shared_buffers = 8GB           # If 32GB RAM, up to 8GB
effective_cache_size = 24GB

# Faster commits
synchronous_commit = off       # Return immediately, WAL in background. Risk: 1s data loss on crash
commit_delay = 100000          # 100ms delay to batch commits. Higher throughput, higher latency

# Plumber tuning
random_page_cost = 1.1         # If all SSD (default 4.0 for HDD)
effective_io_concurrency = 200 # SSD supports more concurrent I/O
```

### MongoDB Tuning

MongoDB configuration in `mongod.conf`:

```
# Storage engine
storage:
  wiredTiger:
    engineConfig:
      cacheSizeGB: 8          # 50% of RAM minus 1GB — WiredTiger uses both cache and journal
    collectionConfig:
      blockCompressor: zstd   # Better compression than snappy (default), more CPU
    indexConfig:
      prefixCompression: true  # Shrinks indexes with common key prefixes

# Operation logging
systemLog:
  destination: file
  path: /var/log/mongodb/mongod.log
  logAppend: true
  verbosity: 0                # Increase to 1 for slow query debugging
```

**Aggressive tuning:**

```
# Page cache
# Outside MongoDB: set vm.dirty_ratio=10, vm.dirty_background_ratio=1
# This ensures the OS flushes dirty pages aggressively, preventing 
# sync storms during journal commits

# WiredTiger cache
cacheSizeGB: 12               # If 32GB RAM and dedicated server
# Journal
# Set journal.commitIntervalMs: 50 (default 100) for less data loss, more fsyncs
```

### Cassandra Tuning

Cassandra configuration in `cassandra.yaml`:

```
# Memory
memtable_allocation_type: heap_buffers  # Off-heap memtable: offheap_objects
memtable_heap_space_in_mb: 1024         # Max heap used by memtables
memtable_offheap_space_in_mb: 1024      # Off-heap memtable space

# Compaction
compaction_throughput_mb_per_sec: 64    # Throttle compaction I/O — increase to 128 on SSDs
concurrent_compactors: 2                # Number of parallel compactions — increase with CPU cores

# Caching
chunk_cache_size_in_mb: 512             # Cache decompressed SSTable chunks
key_cache_size_in_mb: 128               # Cache partition key locations — 128MB covers ~10M keys

# Consistency
# Set per-query, not globally. Default QUORUM for most operations
# For higher throughput: LOCAL_ONE (eventually consistent, fast)
# For strong consistency: LOCAL_QUORUM (costs 2-3x more per operation)

# Read repair
read_repair_chance: 0.1                 # On 10% of reads, check all replicas for consistency
```

**Aggressive tuning:**

```
# Increase compaction throughput
compaction_throughput_mb_per_sec: 256   # Aggressive compaction on SSDs
concurrent_compactors: 4                # Use idle CPU cycles for compaction

# Reduce GC pressure
# JVM tuning: set -Xms8G -Xmx8G (heap, NOT more than 8GB — Cassandra performs poorly with large heaps)
# Use G1GC: -XX:+UseG1GC -XX:MaxGCPauseMillis=200

# Memory tables
memtable_heap_space_in_mb: 512          # Reduce heap memtable — prefer off-heap
memtable_offheap_space_in_mb: 2048      # Move to off-heap, more stable under GC

# Streaming
stream_throughput_outbound_mb_per_sec: 200  # Bootstrap faster when adding nodes
# Hinted handoff
max_hint_window_in_ms: 10800000             # 3 hours — keep hints for down nodes
```

### Common Mistakes

- **PostgreSQL**: Setting `shared_buffers` too high (above 8GB with 32GB RAM causes the OS cache to starve, increasing total I/O).
- **MongoDB**: Not creating indexes for query patterns — every query without an index does a full collection scan.
- **Cassandra**: Using `ALL` consistency level — requires every replica to respond, making the system unavailable if any node is down. Use `QUORUM` or `LOCAL_QUORUM`.
- **Redis**: Using `KEYS *` in production — blocks the event loop for seconds with millions of keys. Use `SCAN` instead.
