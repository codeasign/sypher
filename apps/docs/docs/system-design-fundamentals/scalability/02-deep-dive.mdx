# Scalability Deep Dive

## Internal Architecture of Scaled Systems

Scalability mechanisms are not magic. Each one has a precise internal architecture that governs how it works, where it breaks, and how to tune it.

### Load Balancer Internals

A load balancer is a reverse proxy that maintains a list of healthy backend servers and distributes requests according to a scheduling algorithm.

**Scheduling algorithms:**

| Algorithm | Behavior | Best for |
|-----------|----------|----------|
| Round robin | Distributes sequentially | Equal-capacity servers |
| Least connections | Sends to server with fewest active connections | Variable-length requests |
| IP hash | Same client IP goes to same server | Stateful apps without shared session store |
| Weighted round robin | Servers with higher weight get more requests | Heterogeneous server capacities |
| Random | Random selection | Testing, very large pools |

### Connection Pool Internals

A connection pool maintains a set of persistent database connections that application threads borrow and return.

```
Application thread ──> Borrow connection ──> Execute query ──> Return to pool

Pool configuration:
  min_size = 5        # Connections always kept open (warm pool)
  max_size = 20       # Hard limit (prevents database overload)
  max_idle = 300      # seconds before idle connection is closed
  timeout = 30        # seconds to wait for a connection before error
  max_lifetime = 3600 # seconds before recycling a connection

When all connections are busy:
  New request queues in the pool buffer (up to timeout seconds)
  If buffer fills: "Connection pool exhausted" error returned to client
```

A connection pool of 20 connections on each of 10 app servers means 200 concurrent database connections. PostgreSQL default `max_connections` is 100. You would exhaust the database before you filled the pool.

**The connection math:**

```
PostgreSQL max_connections = 100
App servers = 10
Pool size per server = 20
Total potential connections = 200 (exceeds limit by 2x)
```

Solution: Either increase `max_connections` on PostgreSQL (RAM cost: ~2MB per connection) or reduce pool sizes. The standard formula: `pool_size = (max_connections - 10) / number_of_app_servers`.

**Thread pool sizing:**

```
Formula: threads = cores * (1 + wait_time / compute_time)

CPU-bound (compute_time >> wait_time): threads = cores
  Example: image processing, video encoding

IO-bound (wait_time >> compute_time): threads = cores * 10 to 50
  Example: web serving, database queries

Default (balanced):
  Tomcat default: 200
  Gunicorn: 4-12 per worker
  Node.js: single thread with event loop (async I/O)
```

## Data Structures for Scalability

### Consistent Hashing Ring

Traditional `hash(key) % N` breaks when N changes (adding or removing a server). Every key remaps to a different server, causing massive cache misses or data migration. Consistent hashing solves this.

<AsciiDiagram
  title="Consistent Hashing Ring"
  content={`
                  Hash Ring
          ┌───────────────────┐
          │   Node A          │
          │   ┌───────┐       │
          │   │ Keys  │       │
          │   │ 1-10  │       │
          │   └───────┘       │
          │                   │
   Node D ─┤                   ├─ Node B
   ┌─────┐ │                   │ ┌─────┐
   │Keys │ │                   │ │Keys │
   │31-40│─┤                   ├─│11-20│
   └─────┘ │                   │ └─────┘
           │     ┌───────┐     │
           │     │ Keys  │     │
           │     │21-30  │     │
           │     └───────┘     │
           │   Node C          │
           └───────────────────┘

Adding Node E:
  Key range redistributed only from neighbors
  Keys 26-30 move from C to E
  All other keys stay put

Changing N in hash % N:
  Every key maps to a different node ──> Full cache invalidation
  `}
  alt="Consistent hashing ring with 4 nodes each owning a key range and showing that adding a 5th node only affects adjacent key ranges"
  caption="Consistent hashing: adding a node only redistributes keys from neighbors, not the entire ring"
/>

**How it works:**

```
1. Hash each server node to a position on a circular ring (0 to 2^32 - 1)
2. Hash each key to a position on the same ring
3. Walk clockwise from the key's position to find the nearest server
4. When a server is added: only keys between its position and the next server remap
5. When a server is removed: only its keys remap to the next server clockwise
```

**Virtual nodes** improve distribution. Each physical server maps to 100-200 virtual positions on the ring. This prevents hot spots when nodes have different capacities.

**Real-world usage:**
- Amazon DynamoDB
- Discord (for data sharding)
- Cassandra and Riak
- Akamai CDN

### Distributed Hash Table (DHT)

A DHT extends consistent hashing with peer-to-peer routing. Each node maintains a routing table of `O(log N)` other nodes, enabling key lookup in `O(log N)` hops.

```
Node routing table (for N=10,000 nodes, 14-hop lookup):
  ┌──────────────────────────────────────┐
  │ Finger Table for Node 100            │
  ├──────────┬───────────────┬───────────┤
  │ Interval │ Successor     │ Distance  │
  ├──────────┼───────────────┼───────────┤
  │ 100-200  │ Node 150      │ 50        │
  │ 200-400  │ Node 350      │ 250       │
  │ 400-800  │ Node 720      │ 620       │
  │ 800-1600 │ Node 1400     │ 1300      │
  │ ...      │ ...           │ ...       │
  │ 50000+   │ Node 50       │ 65536     │
  └──────────┴───────────────┴───────────┘
```

Each hop halves the distance to the target. A 10,000-node cluster resolves any key in 14 hops or fewer. Used by Cassandra, Dynamo, and BitTorrent.

### B-Trees for Sharded Databases

A B-Tree is the standard index structure in relational databases. Each node holds `k` keys and `k+1` pointers to child nodes. The tree maintains balance automatically.

```
For a PostgreSQL table with 10 million rows:
  - Each B-Tree node fits in one 8KB page
  - Each node holds ~200 keys (with 12-byte key + 8-byte pointer)
  - Tree depth = log_200(10,000,000) = 3 levels
  - Any row lookup: 3 index reads + 1 heap read = 4 disk reads

For a sharded table (4 shards, 2.5M rows each):
  - Tree depth per shard = log_200(2,500,000) = 3 levels (still)
  - But each shard runs on separate hardware
  - 4 parallel lookups complete in the time of 1 unsharded lookup
```

Sharding reduces the data volume per node, which means each node's B-Tree stays shallow. Write throughput also increases because writes distribute across nodes.

## Consistency During Scaling Events

### Adding Nodes Mid-Operation

Adding a database node while production traffic is flowing requires careful handling. There are three approaches:

**Approach 1: Read-only node (safest)**
```
1. Provision new database instance with replicated schema
2. Add it as a read replica (starts syncing from primary)
3. Wait for replication lag to reach zero (monitor seconds_behind_master)
4. Update connection pool config to include new replica
5. Traffic gradually shifts to new node
6. Zero errors, zero downtime
```

**Approach 2: Resharding (most complex)**

When you need to change the number of shards (e.g., from 4 to 8), data must migrate. Consistent hashing minimizes the migration, but it still happens.

<AsciiDiagram
  title="Resharding: 4 Shards to 8 Shards"
  content={`
Before (4 shards):
  Shard 0: users 0-25%     Shard 1: users 25-50%
  Shard 2: users 50-75%    Shard 3: users 75-100%

Migration process:
  Phase 1: Add 4 new shards (4 to 8)
    All data stays on old shards, new shards are empty
    Write path: dual-write to old + new shard
    Read path: read from old shard

  Phase 2: Backfill new shards
    Background job copies data from old to new shard
    Monitored by comparing row counts and checksums

  Phase 3: Switch reads to new shards
    Read path: read from new shard, fall back to old on miss
    Validate data integrity (sampling 1% of reads)

  Phase 4: Remove old shards
    Write path: new shard only
    Read path: new shard only
    Old shards decommissioned

**Approach 3: Auto-scaling cache nodes**

Redis Cluster handles node addition automatically:
```
CLUSTER MEET <ip> <port>    # Joins new node to cluster
CLUSTER ADDSLOTS <slot...>  # Assign hash slots to new node
# Cluster automatically migrates keys from existing nodes
# Migrations happen in batches of 10 keys
# Client handles MOVED redirects transparently
```

### Hot Spots

A hot spot occurs when one shard, partition, or node receives disproportionate traffic. Common causes:

```
Hot spot scenarios:
  1. Celebrity effect: 1 user generates 10,000x normal traffic
  2. Time-based: All users active at the same hour
  3. Range-based: Keys sorted by creation date, all writes to latest shard
  4. Hash collision: Multiple high-traffic keys map to same partition

Detection:
  - Per-shard latency monitoring (one shard has 5x latency)
  - Partition request count distribution (standard deviation > 3x)
  - Connection pool exhaustion on specific nodes

Solutions:
  - Split hot partition further (sub-partitioning)
  - Add read replicas for hot data
  - Cache aggressively for celebrity content
  - Use write-ahead log (WAL) to batch writes to hot shards
```

### Scaling Efficiency

Not every additional node gives 100% throughput gain. There is always overhead from coordination, networking, and data distribution.

**Amdahl's Law in depth:**

The law states that the maximum speedup of a system is limited by the portion of work that cannot be parallelized.

```
Amdahl's Law:
  Speedup = 1 / ((1 - P) + P/N)

  Where:
    P = proportion of work that can be parallelized
    N = number of nodes
```

**Worked examples at different parallelization ratios:**

```
Example A: 50% parallelizable (P = 0.50)
  2 nodes:   speedup = 1 / (0.50 + 0.50/2) = 1 / 0.75   = 1.33x
  4 nodes:   speedup = 1 / (0.50 + 0.50/4) = 1 / 0.625  = 1.60x
  10 nodes:  speedup = 1 / (0.50 + 0.50/10) = 1 / 0.55  = 1.82x
  100 nodes: speedup = 1 / (0.50 + 0.50/100) = 1 / 0.505 = 1.98x
  Max: 2x speedup (limited by 50% serial work)

Example B: 90% parallelizable (P = 0.90)
  2 nodes:   speedup = 1 / (0.10 + 0.90/2) = 1 / 0.55   = 1.82x
  4 nodes:   speedup = 1 / (0.10 + 0.90/4) = 1 / 0.325  = 3.08x
  10 nodes:  speedup = 1 / (0.10 + 0.90/10) = 1 / 0.19  = 5.26x
  100 nodes: speedup = 1 / (0.10 + 0.90/100) = 1 / 0.109 = 9.17x
  Max: 10x speedup (limited by 10% serial work)

Example C: 95% parallelizable (P = 0.95)
  2 nodes:   speedup = 1 / (0.05 + 0.95/2) = 1 / 0.525  = 1.90x
  5 nodes:   speedup = 1 / (0.05 + 0.95/5) = 1 / 0.24   = 4.17x
  10 nodes:  speedup = 1 / (0.05 + 0.95/10) = 1 / 0.145 = 6.90x
  20 nodes:  speedup = 1 / (0.05 + 0.95/20) = 1 / 0.0975 = 10.26x
  100 nodes: speedup = 1 / (0.05 + 0.95/100) = 1 / 0.0595 = 16.81x
  Max: 20x speedup (limited by 5% serial work)
```

**Practical insight:** Doubling your nodes rarely doubles your throughput. At 95% parallelizability, moving from 10 nodes to 100 nodes (10x increase) only gives a 2.4x speedup gain. The serial 5% dominates at scale. To improve scaling efficiency, identify and reduce serial bottlenecks before adding more nodes.

```
Diminishing returns illustrated:
  95% parallelizable:
    Nodes:    1    2    5    10    20    50    100
    Speedup:  1x  1.9x 4.2x 6.9x 10.3x 14.7x 16.8x
    Gain per node: -  0.9x 0.8x 0.5x  0.3x  0.1x  0.04x

  Each additional node contributes less than the previous one.
```

**Real-world efficiency factors:**

- Network overhead: 0.5-2ms per inter-node call
- Data consistency overhead: 10-30% write throughput penalty for synchronous replication
- Load balancer overhead: 0.1-0.5ms per request
- Serialization/deserialization: varies by format (JSON: ~0.5ms, Protobuf: ~0.1ms, Avro: ~0.15ms)
- Lock contention: each mutex/ lock adds 0.1-1ms of serial wait time

### Instance Configuration

```
Minimum instances: 2 (keep at least N+1 for availability)
Maximum instances: 20 (or budget cap)
Default instance count: 3

Scaling step size:
  - CPU-based: add 2 instances per breach
  - Time-based: pre-warm 5 instances before peak hours (11:00, 18:00)

Cool-down periods:
  After scale-out: 180 seconds (let metrics stabilize before another decision)
  After scale-in: 300 seconds (prevent rapid flapping)

Warm-up time for new instances:
  Container startup: 5-15 seconds
  App initialization: 10-30 seconds
  Connection pool warm-up: 5-10 seconds
  Total: 20-55 seconds before instance serves traffic
```

### Connection Pool Sizing Formula

```
For an application server with 8 cores:
  - Max concurrent worker threads: 8 * 4 = 32 (assumes IO-bound)
  - Each thread needs 1 database connection at peak
  - Connection pool: min(32, max_connections / app_servers)

For 10 app servers with PostgreSQL max_connections = 200:
  - Each pool: 200 / 10 = 20 max
  - Subtract 5 for admin connections: 195 / 10 = 19 max per server
  - Actual: 15 per server (leaves buffer for reporting tools)
```

### Spike Testing

Spike testing answers: does the system survive sudden traffic surges without crashing?

**Common spike scenarios:**
```

   Test: drop load to 50% for 5 min, then surge back to 200%
   Watch: cache warming, connection pool re-establishment

3. Dependency recovery: upstream service comes back online
   Test: fail a dependency for 2 min, then restore it
   Watch: thundering herd, queue backlog drain rate
```

## Thundering Herd Prevention

A thundering herd occurs when a cache miss or resource recovery triggers a stampede of concurrent requests to the same backend. This can cascade into a full outage.

### Strategy 1: Mutex-Based Cache Recompute

Use a distributed lock so only one process recomputes the cached value. Others either wait or serve stale data.

```python
import redis
import time

r = redis.Redis(host="cache-cluster", decode_responses=True)

def get_restaurants(lat, lon):
    cache_key = f"restaurants:{lat}:{lon}"
    lock_key = f"lock:{cache_key}"

    # Attempt cache read
    cached = r.get(cache_key)
    if cached is not None:
        return cached

    # Acquire distributed lock (10 second TTL prevents deadlock)
    lock_acquired = r.set(lock_key, "locked", nx=True, ex=10)
    if lock_acquired:
        try:
            # Double-check cache (another process may have written it)
            cached = r.get(cache_key)
            if cached is not None:
                return cached

            # Expensive computation — only one process runs this
            result = query_database(lat, lon)
            r.setex(cache_key, 60, result)
            return result
        finally:
            r.delete(lock_key)
    else:
        # Lock held by another process — wait and retry
        time.sleep(0.050)  # 50ms backoff
        cached = r.get(cache_key)
        if cached is not None:
            return cached
        # Fallback: compute anyway after waiting
        return query_database(lat, lon)
```

### Strategy 2: Stale-While-Revalidate

Serve stale data immediately while asynchronously refreshing the cache. This eliminates waiting entirely. Cloudflare, Varnish, and Fastly all support this pattern.

```python
def get_restaurants_swr(lat, lon):
    cache_key = f"restaurants:{lat}:{lon}"
    stale_key = f"stale:{cache_key}"

    # Try fresh cache first
    cached = r.get(cache_key)
    if cached is not None:
        return cached

    # Serve stale data immediately
    stale = r.get(stale_key)
    if stale is not None:
        # Trigger async refresh in background
        r.lpush("cache_refresh_queue", cache_key)
        return stale

    # No cache at all — must compute synchronously
    result = query_database(lat, lon)
    r.setex(cache_key, 60, result)
    r.setex(stale_key, 300, result)  # stale lives 5 minutes
    return result
```

### Strategy 3: Probabilistic Early Expiration

Randomize cache expiry to prevent all keys from expiring simultaneously. This is the simplest prevention and works well for bulk key expiration.

```python
import random

def compute_ttl(base_ttl=60, jitter_range=0.2):
    """
    Returns a TTL with +-20% jitter applied.
    base_ttl=60 returns values between 48 and 72 seconds.
    """
    jitter = base_ttl * jitter_range
    return base_ttl + random.uniform(-jitter, jitter)

# Usage:
# r.setex(cache_key, compute_ttl(60, 0.2), result)
```

### Strategy 4: Request Coalescing

Batch concurrent requests for the same resource into a single backend call. This is effective for API gateways and BFF (Backend for Frontend) layers.

```python
import asyncio
from collections import defaultdict

class RequestCoalescer:
    def __init__(self):
        self._pending = defaultdict(list)

    async def get_or_compute(self, key, compute_fn):
        # Fast path: already computing?
        future = asyncio.get_event_loop().create_future()

        if key not in self._pending:
            # First request — start computation
            self._pending[key] = [future]
            try:
                result = await compute_fn()
                for waiter in self._pending[key]:
                    waiter.set_result(result)
            except Exception as e:
                for waiter in self._pending[key]:
                    waiter.set_exception(e)
                raise
            finally:
                del self._pending[key]
        else:
            # Subsequent request — join existing computation
            self._pending[key].append(future)

        return await future
```

### Production Checklist for Thundering Herd Prevention

```
1. Distributed locking:              Redis SET NX or Redlock
2. Stale-while-revalidate:           Serve old data, refresh async
3. TTL jitter:                       Randomize expiration times
4. Request coalescing:               Batch identical requests
5. Cache warming:                    Pre-compute hot keys on deploy
6. Circuit breaker on cache miss:    Limit concurrent recompute ops
7. Rate-limit recompute operations:  Separate throttle for cache fills
8. Graceful degradation:             Serve degraded response on miss
```
