# Scalability Interview Guide

## Question Phrasings

Interview questions about scalability take three forms:

**Form 1: Open-ended design**
```
"Design Twitter."
"Design a URL shortener."
"Design Uber."
"Design a real-time chat system like WhatsApp."
"Design a video streaming platform like YouTube."
"Design a collaborative document editor like Google Docs."
"Design a booking system for a global airline."
"Design a notification system that handles 1 billion pushes per day."
```

Scalability is implicit — the interviewer expects you to discuss how the system handles growth from 1K to 100M users.

**Form 2: Bottleneck diagnosis**
```
"Our API response time increased from 50ms to 2 seconds after
 hitting 10,000 concurrent users. What is wrong?"
"Adding 10 app servers only improved throughput by 20%. Why?"
"Database CPU is at 90% but our query latency is fine. What is
 the next bottleneck to address?"
"We migrated to SSDs but write throughput only improved 10%.
 What are we missing?"
"Our cache hit rate dropped from 98% to 60% overnight. Where
 do you start debugging?"
"Our queue backlog grows every day, but consumer instances are
 at 50% CPU. What is happening?"
"We scaled from 1 to 10 database replicas but read latency
 only improved 30%. Why?"
"Memory usage on app servers keeps rising even though request
 count is flat. What is the most likely cause?"
```

The interviewer gives you a symptom; you must find the root cause.

**Form 3: Scaling strategy**
```
"How would you scale a PostgreSQL database to handle 50,000
 writes per second?"
"Design a cache that can handle 1 million requests per second."
"Design a distributed rate limiter covering 10,000 API keys."
"Design a search index that supports real-time updates at
 100,000 documents per second."
"Design a global leaderboard that updates in real time for
 50 million players."
"Design a recommendation feed that personalizes results for
 100 million users with less than 500ms latency."
"Design a distributed counter that handles 1 billion
 increments per day with strong consistency."
"Design an event-sourced audit log that never loses data and
 supports replay at 50,000 events per second."
"Design a geo-distributed key-value store that prioritizes
 local reads over global consistency."
```

Concrete target with a specific technology. Tests depth of understanding.

## What Interviewers Test

| Competency | What they look for | Red flag |
|-----------|-------------------|----------|
| Bottleneck identification | Finds database, cache, or network bottleneck systematically | Guesses randomly |
| Capacity estimation | Estimates QPS, storage, bandwidth concretely | "It depends" with no numbers |
| Trade-off reasoning | Explains why to shard vs replicate vs cache | Recommends sharding for everything |
| Incremental design | Starts simple, adds complexity as scale increases | Proposes a 20-microservice architecture for 100 users |
| Numbers fluency | Knows PostgreSQL max connections, Redis throughput, network latency | "The database handles it" — no specifics |
| Operational awareness | Discusses monitoring, deployment, rollback | Treats scaling as a one-time design problem |

## Level Expectations

### L3 / Junior — Implementation Focus

"Can you explain how you would add a caching layer to improve performance?"

**Sample answer:**
```
I would add Redis between the application and database.
First, identify the most frequently queried data — for a
social app, that is user profiles and feed posts.

Set a TTL of 60 seconds on cached data. Use a cache-aside
pattern: check cache first, return on hit, query DB on
miss, store result in cache.

Monitor cache hit rate. Target 90%+ for the hot data paths.

If hit rate drops below 80%, either the TTL is too short
or the working set exceeds available memory.
```

**What the interviewer is looking for:** You identify the cache layer, the eviction strategy, and the monitoring metric. You do not over-engineer. You explain the read path clearly. The answer is implementation-ready: another engineer could write code from this description.

### L4 / Mid-Level — Independent Problem Solving

"Design the database schema and query pattern for a news feed that shows the latest 50 posts from followed users."

**Sample answer:**
```
Let me start with the schema. I need two primary tables:

Users table:
  user_id (UUID, primary key)
  username, created_at

Posts table:
  post_id (UUID, primary key)
  user_id (foreign key, indexed)
  content (text)
  created_at (indexed for ordering)

Follows table:
  follower_id (UUID)
  followee_id (UUID)
  composite primary key (follower_id, followee_id)

For the feed query, a naive approach is:

SELECT p.* FROM posts p
JOIN follows f ON p.user_id = f.followee_id
WHERE f.follower_id = :user_id
ORDER BY p.created_at DESC
LIMIT 50;

This works for 1K users but breaks at scale because the
join across follows and posts scans too many rows.

For 10K to 100K users, I add composite indexes:
- (follower_id, created_at) on follows — if we denormalize
  last post time
- (user_id, created_at) on posts — for the per-user lookup

At 100K+ users, I pre-compute the feed. When a user posts,
fan-out that post to all followers' feed caches (Redis sorted
sets, score = timestamp). Each user's feed is a sorted set
capped at 500 entries. This makes feed reads O(log N) and
feed writes fan out to N followers.

The trade-off is write amplification: a user with 10K followers
triggers 10K cache writes. Handle this with async fan-out via
Kafka. For celebrity users (100K+ followers), pull their posts
on-read instead of push on-write.
```

**What the interviewer is looking for:** You design the schema first. You identify the scaling inflection points. You compare push vs pull feed delivery. You consider the celebrity edge case. You communicate trade-offs without being asked.

### L5 / Senior — System Design

"Design Instagram's feed. How does it scale?"

**Sample answer:**
```
Start with a single PostgreSQL database for 10K users.
At 100K users: add read replicas (10:1 read-to-write ratio).
At 1M users: add Redis cache for feed timelines. Pre-compute
timelines on write, cache the last 200 posts per user.
At 10M users: shard the database by user_id (consistent
hashing, 4 shards initially). Use Kafka for async processing
of likes and comments.

Each scaling step is triggered by monitoring: CPU, cache hit
rate, query latency. I do not add complexity until the
numbers demand it.
```

**What the interviewer is looking for:** You present an incremental scaling story with concrete thresholds. You demonstrate that you have actually scaled a system — you know the numbers, the trigger points, and the operational cost of each step. You connect architecture decisions to business metrics.

### L6 / Staff — Multiple Dimensions

"Design a real-time collaboration platform like Google Docs. How do you handle 1M concurrent editors?"

**Sample answer:**
```
The biggest scaling challenge here is consistency, not
throughput. Operational transforms (OT) or CRDTs require
server-side processing per operation.

Architecture:
- 10 WebSocket gateway servers (each handles 100K connections)
- Redis pub/sub broadcasts operations within a document session
- Each active document pinned to one gateway server
- Document storage sharded by document_id across 32 PostgreSQL shards
- Conflict resolution at application layer (OT, 50ms latency budget)
- Load balancer: application-layer routing by document_id (not round-robin)

Key metric: operations per second per document, not users per server.
A popular document with 100 editors generates 1000 ops/sec.
A server can handle ~50 active documents before CPU becomes the bottleneck.
```

**What the interviewer is looking for:** You identify the non-obvious bottleneck (consistency, not throughput). You pick relevant technologies (OT/CRDTs, WebSocket gateway, document-affinity routing). You reason about capacity in the correct units (ops per doc, not users per server). You discuss the consistency-latency trade-off explicitly.

---

### L7-8 / Principal — Organization and Strategy

"How would you design the infrastructure to move from a monolith to microservices safely?"

**Sample answer:**
```
This is not a technology problem — it is an organizational
and operational problem. Microservices will double infrastructure
cost and triple deployment complexity.

Strategy:
1. Identify bounded contexts in the monolith (auth, billing, feed)
2. Extract one service at a time — start with auth (fewest dependencies)
3. Use strangler fig pattern: route traffic to new or old path by header
4. Each extracted service gets its own database, CI/CD, and on-call rotation
5. Invest in observability before extraction (distributed tracing, structured logging)
6. Set a budget: 2x cost increase, 6 months timeline, 1 service per month

Scale the team before the architecture. Conway's Law applies:
the monolith stays until the team can own and operate the services.
```

**What the interviewer is looking for:** You frame the problem as organizational, not technical. You discuss risk mitigation (strangler fig, phased extraction, rollback plan). You set concrete budgets and timelines. You reference Conway's Law and demonstrate awareness that architecture mirrors team structure.

**Step 1: Clarify Requirements**

```
Q: How many users?
A: 100M DAU, 500M MAU. Peak concurrency is 40M users during
   morning commute (8-9 AM ET).

Q: Read vs write ratio?
A: Average user reads 200 tweets and writes 2 tweets per day.
   That is a 100:1 read-to-write ratio.

Q: Latency requirements?
A: Feed load must complete under 500ms at p99. Tweet publish
   should appear in followers' feeds within 5 seconds (near-real-time).

Q: Data persistence?
A: Tweets are permanent. Deleted tweets must disappear within
   24 hours. Likes and retweets are soft-state counters.

Q: Geographic distribution?
A: 60% US, 20% Europe, 15% Asia, 5% rest of world. Need at
   least US-West and EU-West regions for initial deployment.
```

**Step 2: Capacity Estimation**

```
Peak QPS:
- Read: 100M users x 200 reads / 86,400 seconds = ~231K reads/sec
- Write: 100M users x 2 writes / 86,400 seconds = ~2,315 writes/sec
- Peak factor (3x): ~700K reads/sec, ~7K writes/sec during rush hour

Storage:
- Tweet: 280 chars + metadata = ~1KB per tweet
- Storage per user: 2 tweets/day x 1KB x 365 days = ~730KB/user/year
- Total storage: 100M users x 730KB = ~73 TB/year

Bandwidth:
- Average response: 20 tweets x 1KB each + metadata = ~25KB
- Read bandwidth: 25KB x 231K reads/sec = ~5.8 GB/sec
- Write bandwidth: 2KB x 2,315 writes/sec = ~4.6 MB/sec

Cache working set:
- Active users in a day: 100M
- Feed size per user: 200 tweets (sorted set, ~20KB)
- Total cache needed: 100M x 20KB = 2 TB
- Splitting across 50 Redis nodes: 40 GB per node (feasible)
```

**Step 3: Data Model**

```
User table:
  user_id (UUID, primary key)
  screen_name (unique, indexed)
  follower_count (denormalized counter)

Tweet table:
  tweet_id (64-bit auto-increment, primary key)
  user_id (indexed — for user timeline view)
  content (varchar(280) or text for longer tweets)
  created_at (indexed — for sort order)
  like_count, retweet_count (denormalized counters)

Follow table:
  follower_id (UUID)
  followee_id (UUID)
  composite primary key (follower_id, followee_id)
  Secondary index on followee_id for follower list queries

Feed cache (Redis, not a table):
  Key: feed:{user_id}
  Type: sorted set, score = tweet.created_at
  Value: tweet_id
  Cardinality: capped at 500 per user

Shard key: user_id (hash-partitioned across all tables)
Home timeline query: ZREVRANGE feed:{user_id} 0 199
  → fetch tweet content from Redis or fall back to database
```

**Step 4: Core Architecture**

```
1. Load balancer: Application-layer (Layer 7) routing by endpoint.
   /feed/* goes to feed service, /tweet/* goes to tweet service.

2. Feed service: Stateless app servers. Reads feed from Redis.
   Cache miss: fall back to database query.
   Fan-out on write: when user tweets, push tweet_id into all
   followers' Redis sorted sets (async via Kafka).

3. Tweet write service: Validates tweet, stores in database,
   publishes to Kafka topic "new-tweets".

4. Kafka: 32 partitions for new-tweets topic. Partitions by
   user_id so all tweets from one user go to the same partition
   (guarantees per-user order).

5. Fan-out consumer: Reads from Kafka, for each tweet:
   - Fetch follower list from cache or DB
   - Push tweet_id into each follower's Redis sorted set
   - If user has > 100K followers (celebrity), skip fan-out.
     Celebrities use pull-based reads.

6. Redis cluster: 50 nodes. Each node holds ~4M feed sorted sets.
   Slot-per-user hashing ensures even distribution.
```

**Step 5: Scaling Plan**

```
Phase 1 — Single server (0-10K users):
  Everything on one PostgreSQL instance. Feed queries are
  JOINs across follows and tweets tables. Works fine.

Phase 2 — Database replicas (10K-100K users):
  100:1 read-to-write ratio means read replicas give huge gains.
  Add 3 read replicas. Master handles writes, replicas handle
  feed reads with read-committed isolation.

Phase 3 — Feed cache (100K-1M users):
  JOIN queries are too slow. Add Redis feed cache.
  Pre-compute feed on tweet publish. Cache last 200 tweets.
  Cache hit rate targets 95%+.

Phase 4 — Sharding (1M-10M users):
  Single database write master is the bottleneck.
  Shard by user_id across 4 PostgreSQL shards initially.
  Each shard is master-replica pair. Consistent hashing for
  rebalancing.

Phase 5 — Async fan-out + Kafka (10M-100M users):
  Synchronous fan-out during tweet write blocks the request.
  Introduce Kafka between tweet write and fan-out.
  Fan-out consumer scales independently.

Phase 6 — Multi-region (100M+ users):
  Deploy in US-West and EU-West.
  Cross-region replication for data.
  Read from local region, writes go to primary region
  (eventually consistent cross-region).

Phase 7 — Celebrity handling (any scale with power users):
  Users with > 100K followers use pull-based feed.
  When you follow a celebrity, the system marks them.
  Feed requests merge: pull celebrity tweets on-read,
  push regular-user tweets on-write.
```

**Step 6: Trade-offs and Deep Dive**

```
Trade-off: Push vs Pull
  Push (fan-out on write):
    + Reads are O(1) — fast feed load
    - Write amplification: 1 tweet x N followers = N cache writes
    - Celebrity user with 10M followers stalls fan-out
    Decision: Hybrid — push for regular users, pull for celebrities

Trade-off: Strong vs Eventual Consistency
  After writing a tweet, should it appear instantly in your own
  feed? Yes — read-your-writes consistency for the author.
  For followers, eventual consistency within 5 seconds is fine.
  Implementation: Author's feed updated synchronously during
  write. Followers' feeds updated asynchronously via Kafka.

Failure mode: Redis Cluster Down
  If all 50 Redis nodes fail simultaneously, feed service falls
  back to database queries directly. Latency increases from
  50ms to 500ms. Degraded but not down. Redis slaves promote
  automatically. Cache warms over 15 minutes.

Failure mode: Kafka Broker Failure
  Kafaka's replication factor of 3 means one broker failure
  does not lose data. Rebalance completes in ~30 seconds.
  Fan-out is delayed during rebalance, causing 30-second lag
  in tweet delivery.

Failure mode: Database Shard Hotspot
  A viral event (Super Bowl, election) causes one shard to
  handle 10x normal load. Mitigation: pre-identify expected
  hot users. Pre-allocate them to their own shard. Monitoring
  alerts when any shard exceeds 70% CPU.

Monitoring:
  - Redis: cache hit rate per node, memory usage, eviction rate
  - Kafka: consumer lag (alert if > 100K messages behind)
  - Database: replication lag (alert if > 5 seconds), connection
    pool utilization
  - App: p50/p95/p99 feed load latency, tweet publish latency
```

## Phrase Bank

Use these phrases to structure your answers:

**Clarifying scope:**
```
"Let me clarify the scale we are designing for. What is the target number of daily active users?"
"Can you confirm the read-to-write ratio? That determines whether I focus on read replicas or sharding."
"What are the latency requirements at the p99 percentile?"
"Is this a greenfield design or are we migrating an existing system?"
"Should I optimize for throughput or tail latency? Those drive different architectures."
"Are there any hard consistency requirements, or is eventual consistency acceptable for some paths?"
```

**Estimating capacity:**
```
"Assuming 10 million daily active users, each generating 10 read requests and 1 write request per day..."
"With a 10:1 read-to-write ratio, we need 10x more read capacity than write capacity."
"At 100 MB of storage per user, we need approximately 1 PB of storage for 10 million users."
"Cache working set is approximately the top 20% of data that generates 80% of reads."
"Peak traffic is typically 3x the average for consumer applications. Let's design for 3x headroom."
"Each user session holds a WebSocket connection. At 50,000 concurrent connections per server and 10 KB of memory per connection..."
```

**Discussing trade-offs:**
```
"The trade-off here is between consistency and availability. For this feature, eventual consistency is acceptable because..."
"Sharding adds complexity but removes the single write bottleneck. We should shard when writes exceed 5,000/second."
"Read replicas are simpler than caching but add database cost. Redis is faster but requires cache invalidation logic."
"Auto-scaling adds cost unpredictability but prevents outages during traffic spikes."
"Denormalization speeds up reads by 10x but makes writes more expensive and complicates schema migrations."
"Choosing a NoSQL document store trades away joins and transactions for horizontal scalability and flexible schemas."
```

**Deep diving on bottlenecks:**
```
"The most common bottleneck at this scale is the database connection pool. With 100 app servers and 20 connections each..."
"Cache hit rate is the single most important metric. A drop from 95% to 80% means 4x more database load."
"Network bandwidth becomes the bottleneck before CPU in a well-cached system. Each request at 50KB means..."
"When the queue depth grows faster than consumers can process it, you need more partitions and consumer instances."
"The hidden bottleneck is often the garbage collector. Under high allocation pressure, GC pauses can spike p99 latency by 200ms."
"Connection overhead per thread in the app server caps concurrency before CPU does. Consider async I/O for IO-bound workloads."
```

**Transitioning between steps:**
```
"Now that we have the data model, let me walk through how a request flows through the system."
"With the core architecture in place for 1 million users, let me discuss how we would scale to 10 million."
"Given the caching strategy, the next bottleneck to address is database write throughput. Let me talk about sharding."
"Let me consider the failure modes. What happens when the cache cluster goes down?"
"To summarize the architecture before I go deeper: CDN → load balancer → app servers → cache → database."
```

**Handling pushback:**
```
"That is a great point. What if the read-to-write ratio is 1:1? In that case, read replicas provide less benefit and
 I would focus on sharding the writes instead. Let me adjust the design..."

"If the latency requirement is 50ms at p99 rather than 500ms, I would add an in-memory cache layer on the app server
 itself — a local LRU cache in front of the distributed Redis cache."

"What if the data is write-heavy with 90% writes? Then caching helps less because the cache invalidates constantly.
 I would focus on database sharding, connection pooling tuning, and batching writes."

"If 90% of traffic comes from mobile devices, response payload size becomes critical. I would add field selection
 (GraphQL or sparse fieldsets) and compress responses with Brotli."

"If the interviewer says the read-to-write ratio is actually 1000:1, then I would invest heavily in caching and
 CDN — a single write master can easily handle the writes, and 99.9% of traffic should never hit the database."

"If the system must be strongly consistent, we cannot use eventual-consistency caches. I would use read-after-write
 consistency in the cache layer and add version vectors to detect conflicts."

"What if users expect sub-100ms feed loads even on cold start? Then we need to pre-warm the cache during deployment
 and maintain a read-only copy of the database for cache-miss fallback that handles 10,000 QPS."
```

**Talking about costs:**
```
"At 10M users, this architecture costs approximately $50,000 per month. Here is the breakdown: 20 app servers at
 $500 each, a 50-node Redis cluster at $200 each per month, and 4 database shards at $2,000 each."

"The trade-off is that adding a CDN costs roughly $0.01 per GB delivered but reduces origin server costs by 80%.
 At 10 TB/month of content delivery, the CDN pays for itself."

"Using SSDs over HDDs for the database costs 3x more per GB but delivers 10x the IOPS. For the working set of
 500 GB, the additional $1,000 per month is worth the latency improvement."

"Idle capacity for failover is expensive: running 2x capacity costs twice as much but guarantees
 sub-second failover. A cheaper option is running at 60% utilization and relying on auto-scaling to
 absorb a single-node failure."

"Reserved instances reduce compute cost by 40-60% compared to on-demand, but require committing to 1-3 years.
 For stable baseline traffic, reserved is better. Use spot instances for batch processing and canary deployments."

"Data transfer costs between regions are higher than within region. For a multi-region architecture, expect
 cross-region bandwidth to be 10-20% of your total cloud bill."
```

**Wrapping up and summarizing:**
```
"To summarize, we start with a monolith and add complexity only when metrics demand it. The architecture
 transitions through four phases: single database, read replicas, cache layer, and finally sharding."

"The three key decisions I made are: (1) hybrid push-pull for feed delivery, (2) eventual consistency for
 follower feeds with read-your-writes for the author, (3) async processing via Kafka to decouple tweet
 publication from fan-out."

"If I had to pick the single most impactful optimization for this system, it would be the Redis feed cache.
 Without it, every feed load requires a multi-table join across millions of follow relationships."

"The core insight is that this system is read-heavy with a 100:1 read-to-write ratio, so every optimization
 targets the read path. Caching, CDN, and read replicas all reduce load on the write master."

"The failure mode I am most concerned about is a cascade: if Redis fails, the database gets 10x more traffic,
 which fills connection pools, which causes app server timeouts, which triggers more retries. Circuit breakers
 on the database client prevent this."
```

**Transitioning to failure modes:**
```
"Let me shift from the happy path to failure modes. What happens when the Redis cache cluster loses a node?"
"The architecture I described works well in steady state. Now let me consider what breaks and how we recover."
"No design survives contact with production unchanged. Let me discuss three failure scenarios and their mitigations."
"I designed for performance. Now let me design for resilience. The weakest link in this architecture is..."
"How would this system behave during a traffic spike of 10x normal? Let me trace through the bottleneck cascade."
```

## Red Flags (Things that Signal Weak Answers)

| Red flag | What it signals | Better approach |
|----------|----------------|-----------------|
| "We use a load balancer" (and stops there) | Surface-level understanding | Explain algorithm (round robin vs least connections), health checks, TLS termination |
| "We shard the database" without explaining the key | No sharding experience | Discuss shard key selection, rebalancing, hot spots |
| "Add more servers" for every problem | No diagnostic ability | Identify the actual bottleneck first, then propose a targeted solution |
| No mention of monitoring | No operational experience | "We monitor cache hit rate, query latency, and connection pool utilization" |
| "Cassandra solves everything" | Tool-focused, lacks fundamentals | Start with PostgreSQL. Scale with replicas, then cache, then sharding. Only then consider Cassandra |
| No failure mode discussion | Naive design | "This architecture fails if Redis goes down. Here is how we handle that..." |
| One-size architecture (designs same system for 100 and 100M users) | No scaling progression | "At 1K users it is a monolith. At 10K we add replicas. At 100K we add cache..." |
| No numbers | Guesswork | State specific numbers: "PostgreSQL handles 5K writes/second on a c5.4xlarge" |
| No mention of CDN at global scale | Unfamiliar with edge delivery | CDN reduces origin load by 80% for static and cacheable content. Always mention CDN when users are globally distributed |
| Assuming infinite network bandwidth | No real-world deployment experience | "At 5.8 GB/sec bandwidth, a single 10 Gbps NIC handles about 1.25 GB/sec. We need at least 5 NICs or a 40 Gbps link" |
| Ignoring cache invalidation | No production caching experience | Discuss TTL, write-through, cache-aside, and what happens on cache miss. "We invalidate the cache entry when the underlying data changes" |
| Single-region architecture for a global use case | No multi-region experience | "For a global user base, we deploy in at least three regions: US-West, EU-West, and Asia-Pacific. Route by DNS geolocation" |
| No mention of connection pooling | No database scaling experience | "Without connection pooling, 100 app servers x 30 connections = 3,000 connections. PostgreSQL handles ~500 concurrent connections before performance degrades. We add PgBouncer in front" |
| Proposes microservices immediately | Over-engineers | "Start with a monolith. Extract services only when team size and deployment velocity require it" |
| Ignores the cost dimension | Lacks business awareness | "This design costs approximately $X per month. Here is the major cost driver and how to reduce it" |

## Drills

Practice these with a timer. Aim to complete each in 15 minutes.

**Drill 1: URL Shortener**

Design bit.ly for 1 billion URL creations per month. Each URL viewed 10 times on average.

Focus: Capacity estimation (storage for 1B URLs), key generation (distributed unique IDs), cache strategy (hot URLs), database sharding.

**Step-by-step solution outline:**

```
1. Capacity: 1B creations/month = ~385 writes/sec, ~1,155/sec peak.
   10 views per URL = 3,850 reads/sec peak. Storage = 500 GB/year.
2. Key gen: Base-62 encoding (a-z, A-Z, 0-9), 7 characters = ~3.5T unique keys.
   Use Snowflake-style distributed ID service. Pre-generate key batches.
3. Schema: short_key (PK), long_url, created_at, expiration. Shard by short_key hash.
4. Cache: Redis LRU for top 20% of URLs (90% of traffic). 1-day TTL. Write-through on create,
   cache-aside on read miss.
5. Scaling: 0-1M = single DB + in-memory cache. 1M-100M = Redis cluster + read replicas.
   100M-1B = shard + CDN edge caching for redirects.
```

**Drill 2: Rate Limiter**

Design a rate limiter that handles 10 million requests per second across 100 servers.

Focus: Distributed counter (Redis sorted sets vs sliding window), memory efficiency (1KB per user limit vs 10 bytes), consistency (exact vs approximate throttling).

**Step-by-step solution outline:**

```
1. Requirements: 10M req/s across 100 servers. Per-API-key limits (100 req/s).
   Approximate throttling is acceptable (±5%).
2. Algorithm: Sliding window counters. Divide window into 10 sub-windows (100ms each).
   Local counter per server, sync to Redis every 100ms. Memory: ~10 bytes per key.
3. Redis: Hash per API key with TTL of 2 seconds. 10M keys x 10 windows x 10B = ~1 GB.
4. Consistency: Local decision + periodic global sync. Acceptable 10% overshoot during sync interval.
5. Fallback: Redis down → local-only limiting at 2x the global limit.
6. Optimization: Use Count-Min Sketch for 100x memory reduction (overcounts but never undercounts).
```

**Drill 3: Global Chat**

Design WhatsApp for 500 million daily active users. Messages delivered within 100ms.

Focus: WebSocket connection management (100K per server), message routing (presence, delivery, read receipts), storage (per-conversation sharding), multi-region latency.

**Step-by-step solution outline:**

```
1. Connections: 500M DAU x ~2 hours/day = ~41M concurrent. 100K/server = 410 gateways.
   Pinned by user_id hash. Gateway holds (user_id, server_id) mapping.
2. Send flow: User A → gateway → validate → store in DB → Redis pub/sub for conversation →
   all gateways receive → check local mapping → push WebSocket or FCM/APNs notification.
3. Storage: Per-conversation sharding by conversation_id. Append-only log. ~25B messages/day
   at 1KB each = 25 TB/day. Archive to cold storage after 30 days.
4. Multi-region: US, EU, Asia. DNS geo-routing. Inter-region Kafka replication.
   Target: same-region under 50ms, cross-region under 200ms.
5. Offline: On reconnect, batch-deliver unread messages since last_seen_timestamp.
6. Failure: Gateway fails → clients reconnect via DNS in under 1 second.
```

**Drill 4: E-commerce Product Search**

Design Amazon's product search for 10 million products and 100 million users.

Focus: Inverted index sharding, caching top 1% of search queries, auto-complete at scale, personalization without shard-hot-spot issues.

**Step-by-step solution outline:**

```
1. Inverted index: Term → list of (doc_id, frequency, position). Shard by term hash across 16 shards.
   Query: broadcast to all shards → merge → rank → return top-K.
2. Index size: 10M products x 50 terms = 500M entries = ~50 GB. Fits in memory (3.2 GB/shard).
3. Cache: Top 1% of queries generate 80% of traffic. Cache (query → product_ids) in Redis, 5-min TTL.
   Auto-complete trie in Redis: ~500 MB per language.
4. Personalization: Offline-computed user_vector (Spark/Hive). Fetched from user-partitioned Redis at
   query time (avoids index shard hot spots).
5. Scaling: 0-1M products = single Elasticsearch. 1M-10M = sharded ES cluster, 16 nodes.
6. Failure: Index shard fails → incomplete results (missing that shard's terms). Replica takes over in
   ~30 seconds. Degraded mode is acceptable.
```

**Drill 5: Photo Upload Service**

Design Instagram photos for 500 million users. Each upload generates 3 thumbnail sizes, 1 filtered version.

Focus: Async processing pipeline (Kafka + worker pools), CDN distribution, storage tiering (hot vs cold), upload bandwidth management.

**Step-by-step solution outline:**

```
1. Upload: User uploads → app server stores original on local SSD → publish event to Kafka →
   return photo_id immediately. Processing is async.
2. Pipeline (Kafka consumers): Consumer 1 = thumbnails (150/300/640px via libvips).
   Consumer 2 = filter (GPU-accelerated). Consumer 3 = EXIF extraction + NSFW detection + perceptual hash.
   Each consumer scales independently.
3. Storage tiers: Hot = CDN edge (7-day TTL for thumbnails). Warm = cloud storage standard tier.
   Cold = Glacier for originals older than 6 months.
4. Bandwidth: 500M users x 2 photos/day = 1B uploads/day. At 3 MB each = ~35 GB/sec.
   Accept via edge POPs, forward to processing region via private backbone.
5. Storage: Original (3 MB) + 3 thumbnails (700 KB) + 1 filtered (1.5 MB) = ~5.2 MB per photo.
   1B photos/day = 5.2 PB/day. After compression and dedup = ~2 PB/day net.
6. Failure: Worker crashes mid-job → Kafka offset not committed → message reprocessed.
   Idempotency key (photo_id) prevents duplicates.
   CDN failure → serve thumbnails directly from origin storage. Health check detects CDN headers.
```

## Post-Interview Deep Dive Topics

After you complete the main design, the interviewer may drill into specific areas. Prepare for these follow-up questions.

**Data Center Failure**

"How do you handle a full data center failure?"

```
Sample approach:
Design for region isolation. Each region runs independently. Users
are routed via DNS geolocation to their nearest healthy region.

For the Twitter feed example:
- Primary region: US-West
- Failover region: EU-West (also serves European users normally)
- Replication: Each tweet written to US-West is replicated to
  EU-West asynchronously via Kafka MirrorMaker (lag ~2 seconds)
- On US-West failure:
  1. Health check detects US-West is unreachable
  2. DNS update routes all traffic to EU-West (TTL = 60 seconds)
  3. Users experience up to 60 seconds of downtime + 2 seconds of data loss
  4. EU-West serves read-only until replication backlog is cleared
  5. On US-West recovery: replay replication log in reverse direction
- Data loss: up to 2 seconds of tweets. Acceptable for this use case.
```

**Database Migration Without Downtime**

"How do you migrate from one database to another without downtime?"

```
Sample approach:
Use the dual-write pattern with a backfill step:

Phase 1 — Dual writes (both databases active):
  Every write goes to both OldDB and NewDB in the same transaction.
  Use a write-ahead log or a transactional outbox to guarantee at-least-once writes.
  Phase 1 validation: compare read results between OldDB and NewDB
  for the same query. Target 99.9% match before proceeding.

Phase 2 — Backfill:
  Script reads all historical data from OldDB and writes to NewDB.
  Run in batches of 10,000 records. Retry on conflict.
  During backfill, application reads from OldDB.
  Monitor: backfill progress (percentage complete), error rate, latency impact.

Phase 3 — Dual reads:
  Route all reads to NewDB. Keep writes going to both databases.
  Monitor for 1 week. Rollback plan: switch reads back to OldDB if issues found.

Phase 4 — Cutover:
  Stop writes to OldDB. Confirm all data is present in NewDB.
  Remove OldDB connection string from configuration.
  Archive OldDB for 30 days (just in case).

Key risk: schema mismatch. Validate schema before Phase 1.
Automate the rollback: a single config change to revert reads to OldDB.
```

**Handling Traffic Spikes**

"How do you handle a 100x traffic spike (viral event, Super Bowl, flash sale)?"

```
Sample approach:
Three strategies, used in combination:

1. Pre-scaling (proactive):
   - For known events (Super Bowl, Prime Day), pre-provision 5x capacity
   - Warm cache with predicted hot content (competitor analysis, historical data)
   - Pre-scale consumer pools: double the Kafka consumer instances

2. Auto-scaling (reactive, rule-based):
   - CPU > 70% for 2 minutes → add app server instances (cool-down: 10 minutes)
   - Queue depth > 10,000 messages → double consumer count
   - Cache hit rate < 80% → add Redis cluster nodes

3. Degraded mode (graceful fallback):
   - If all scaling mechanisms are exhausted, degrade non-critical features:
     - Disable analytics tracking (saves 20% app server CPU)
     - Disable personalized recommendations (use generic ranking)
     - Throttle non-essential writes (comment likes, view counts)
   - Display a banner: "Some features are temporarily degraded"
   - Never degrade: authentication, read access, basic write functionality

For 100x spikes:
  Static content → CDN absorbs the spike (automatic)
  Dynamic reads → Redis cache warms within 5 minutes
  Writes → queue-based ingestion with backpressure
  Database → connection pooling with queue (reject excess connections gracefully)
```

**Thundering Herd**

"How do you prevent a thundering herd when cache expires for a popular item?"

```
Sample approach:
Three techniques:

1. Cache stampede prevention (mutex lock on cache miss):
   On cache miss, acquire a distributed lock (Redis SETNX with TTL).
   Only the lock holder queries the database.
   Other requests wait (busy-spin with 5ms sleep, up to 500ms).
   Lock holder populates cache, releases lock.

2. Early re-computation:
   Before the TTL expires, if the item is "hot" (requested > N times/sec),
   proactively refresh the cache. This prevents TTL expiration entirely.

3. Staggered TTL:
   Instead of a uniform 60-second TTL, add jitter: TTL = 60 seconds
   +/- 10 seconds (random). This prevents all copies of the cached
   data from expiring simultaneously.

Implementation priority:
  Staggered TTL is simplest — add jitter to all TTLs today.
  Early re-computation for known hot items (celebrity profiles, viral posts).
  Mutex lock as the last resort for cache miss storms.
```

**Scaling the Team**

"How does your architecture change as the engineering team grows from 5 to 500 engineers?"

```
Sample approach:
At 5 engineers, a monolith with a shared database is optimal.
You deploy once a day. Everyone knows the entire codebase.

At 25 engineers, extract 2-3 services. Each service has
a dedicated squad of 5-8 engineers. Service ownership reduces
merge conflicts and deployment coordination overhead.

At 100 engineers, you need:
- Each service has its own database (bounded context per team)
- Clear API contracts (gRPC/Thrift with schema registry)
- Schema ownership: only the owning team changes the database
- Feature flags: each team deploys independently
- SLAs per service: latency, availability, error budgets

At 500 engineers, you need platform engineering:
- Internal developer platform (IDP) for service scaffolding
- Standardized observability (distributed tracing, metrics pipeline)
- Self-service database provisioning
- Dark launches and canary deployments
- Migration framework (strangler fig pattern as a standard tool)

Key insight: The architecture should lag the team size by one phase.
A team of 25 does not yet need a full microservice architecture —
they can stay on a modular monolith with clear package boundaries.
Only extract services when the deployment coordination overhead
exceeds the operational cost of running microservices.
```
