# Authentication at Scale

Authentication is the first system every application builds and the last one you want to redesign under pressure. A naive auth stack that works for 100 users will buckle at 10,000 and catastrophically fail at 10,000,000. This page walks through the architecture progression from a single-server password checker to a globally distributed, multi-region auth platform.

## Naive Implementation: Single-Server Auth

The simplest possible authentication system: one server, one database, one password hashing library.

### What Breaks at 1,000 Users

At 1,000 users, the single-server auth system is still functional but shows warning signs. Bcrypt at cost 12 takes roughly 250ms per hash. If 100 of your 1,000 users log in during a peak minute, the server spends 25 seconds of each minute CPU-bound on hashing alone. The remaining 35 seconds must handle all other requests. Response time fluctuates wildly. The server runs at 40-60% CPU during peak hours. A single deploy takes the entire system offline for 10-30 seconds.

### What Breaks at 100,000 Users

The single server is now in crisis. With 100,000 users, assume 10,000 daily active users (DAU) and 20% of them logging in during peak hour: 2,000 login attempts in one hour, roughly 33 per minute. Bcrypt cost 12 burns 8.3 seconds of CPU per minute, but the real problem is concurrency. The database connection pool (typically 20-50 connections) is exhausted by session lookups, login queries, and signup writes. Queued requests time out. Error rate hits 5-15% during peak windows. The server's single thread of execution means a slow bcrypt computation blocks all other requests. You cannot scale horizontally because sessions are stored in the local database.

### What Breaks at 10,000,000 Users

Impossible to serve from a single server. The database alone would need terabytes of storage for user records and session data. Bcrypt at 250ms per login, with 50,000 peak-hour login attempts, would require 3.5 hours of CPU time per hour — a single server physically cannot keep up. Session token storage in a local SQLite database would exceed disk capacity within weeks. Zero redundancy means any hardware failure, network partition, or power event takes the entire auth system offline. Users cannot log in. The product is dead.

### Auth Service Cluster

The auth service is stateless. Every node in the cluster can handle any request. This is the critical design decision that enables horizontal scaling. Each node runs a lightweight HTTP server (typically 4 vCPU, 16GB RAM) that handles:

- **Password login**: Accept credentials, verify against the user store, issue a session token.
- **Token validation**: Accept a session token, verify it against the session store, return user identity.
- **Session management**: Create, refresh, revoke, and invalidate sessions.
- **Account recovery**: Initiate and complete password reset flows.

The cluster auto-scales based on CPU utilization (target: 60%) and request queue depth (target: fewer than 100 pending requests per node). A typical production cluster runs 10-50 nodes during normal operation and scales to 200+ during traffic spikes.

### Rate Limiter

Rate limiting is a separate concern from authentication logic. A Redis-based rate limiter sits in front of the auth service and enforces:

- **Per-IP rate limit**: 10 login attempts per second, 100 per hour. Prevents brute-force attacks from a single IP.
- **Per-account rate limit**: 5 consecutive failed attempts per minute, then exponential backoff (1 min, 5 min, 15 min, 1 hour). Prevents credential stuffing against specific accounts.
- **Global rate limit**: 50,000 requests per second across all routes. Protects the entire auth cluster from DDoS.
- **Device fingerprint rate limit**: 20 attempts per minute per device fingerprint. Catches distributed attacks that rotate IPs but use the same device profile.

The rate limiter uses a sliding window counter stored in Redis with a TTL equal to the window duration. Each request atomically increments the counter and checks the limit. Redis handles 100,000+ increments per second per node, so the rate limiter is never the bottleneck.

### MFA Service

The MFA service is a separate microservice with its own data store for MFA enrollment records. It exposes three endpoints:

- **POST /mfa/enroll**: Generate a shared secret, store it, return a TOTP provisioning URI or WebAuthn challenge.
- **POST /mfa/verify**: Accept a TOTP code, WebAuthn assertion, or OTP. Return success or failure.
- **POST /mfa/challenge**: Initiate a challenge (send SMS, email OTP, or push notification). Used as a step-up for sensitive operations.

TOTP codes are generated per RFC 6238: a 6-digit code derived from HMAC-SHA1 of the current 30-second time window. The MFA service must account for clock skew (allowing a window of +/- 1 on each side, checking 3 time windows total) and code reuse prevention (one-time use enforced via a Redis set with a 90-second TTL).

### Identity Provider (IdP) Integration Layer

The IdP integration layer handles external authentication flows. It is a facade that abstracts the OAuth2, OpenID Connect, and SAML protocols behind a single internal API. The integration layer manages:

- **Protocol translation**: Converts external identity tokens (Google JWT, Apple identity token, SAML assertion) into a standardized internal user object.
- **Account linking**: Maps external identity provider IDs to internal user records. If the same email address is used with Google and Apple, the IdP layer merges them under one account.
- **Token refresh**: Manages refresh token rotation for external providers. Stores refresh tokens in an encrypted column in the user store.
- **Certificate rotation**: Fetches and caches JWKS (JSON Web Key Sets) from external providers. Rotates cached keys before they expire.

The user store is the source of truth for user records. The auth service reads from and writes to this database for every registration and credential verification.

**Key design decisions for the user store:**

- **Shard by email hash, not user ID**: Email is known at login time before the user ID is known. Hashing the email gives you a deterministic shard for every login attempt without a secondary lookup.
- **Read replicas for the hot path**: Session validation and token refresh are read-heavy operations. Each shard has 2-3 read replicas that handle the 90% of requests that are read-only.
- **Connection pooling via PgBouncer**: Authentication is bursty (traffic spikes during login hours). A pool of persistent connections absorbs the burst without exhausting the database connection limit.
- **Stored procedures for atomic operations**: `INSERT ... ON CONFLICT` for signup, `SELECT ... FOR UPDATE` for password changes that must be atomic.

### Pattern 2: With Redis (Session Cache)

The session store is the most performance-critical component of the auth system. Every authenticated request validates the session token. This must complete in under 5 milliseconds.

**Redis session store design decisions:**

- **Opaque tokens over JWT for sessions**: Opaque tokens (a random 256-bit string) require a Redis lookup but can be instantly revoked. JWT-based sessions allow stateless validation but cannot be revoked without a blocklist. The auth system uses opaque tokens for the primary session and optionally issues a JWT for downstream services that need to verify identity without calling the auth service.
- **Sliding window TTL**: Each successful validation extends the session TTL by 30 minutes. This prevents active users from being logged out during long sessions while ensuring inactive sessions are automatically cleaned up.
- **Absolute expiration**: A session cannot exceed 7 days regardless of activity. The user must re-authenticate weekly. This is a security requirement for most compliance frameworks (SOC 2, PCI DSS, HIPAA).
- **User-to-session mapping**: A Redis set `user_sessions:{user_id}` tracks all active sessions for a user. This enables bulk invalidation when the user changes their password or an admin forces logout.

**Critical security checks for IdP integration:**

- **Validate the state parameter**: The state parameter in the OAuth2 flow prevents CSRF attacks. The auth service generates a random state value, stores it in the user's session (or a signed cookie), and verifies it matches when the callback is received. State values must be single-use with a 5-minute TTL.
- **Verify the id_token signature**: The JWT returned by the IdP must be cryptographically verified using the IdP's JWKS endpoint. Cache the JWKS for 24 hours (or until a `kid` is not found, triggering a refresh). Never accept an unsigned or self-signed token.
- **Check the aud (audience) claim**: The `aud` claim in the id_token must match your application's client ID. Otherwise, a token issued for a different application could be used to authenticate in yours.
- **Account linking security**: When linking an external IdP account to an existing user, require the user to first authenticate with their existing credentials (password or another linked IdP). Otherwise, an attacker who compromises a Google account could link it to any existing account if the email address matches.

### Pattern 4: With Message Queue (Audit Log)

Every authentication event must be recorded for security auditing, compliance, and operational monitoring. The auth service writes events to a message queue, and a separate consumer processes them asynchronously.

**Why Kafka, not a synchronous write:**

If the auth service wrote audit events synchronously to the database, each login would require two database round-trips: one for credential verification and one for the audit record. This doubles the latency of the auth path. Worse, an audit database outage would prevent users from logging in. The queue decouples audit persistence from auth logic. The auth service publishes an event in under 1 millisecond (Kafka produce latency at p99) and returns immediately. The consumer processes the event asynchronously, retrying on failure without affecting the user experience.

### Active-Active vs Active-Passive

Authentication is read-heavy (session validation is 90%+ of requests) and write-light (login creates one session, credential changes are rare). This makes active-active a natural fit: every region independently handles auth requests against its local Redis and user store.

- **Active-active for auth**: Every region runs a full auth stack. Users are routed to the nearest region by DNS latency-based routing. Sessions are replicated across regions asynchronously via Kafka. Write conflicts are rare (a user is unlikely to log in from two regions simultaneously) and the last-write-wins strategy is acceptable.
- **Active-passive for user store**: The primary user store (PostgreSQL) lives in one region (US-East). Other regions maintain read replicas. Writes (signup, password change) are forwarded to the primary region. Read replicas serve the 90%+ of requests that are read-only. This avoids the complexity of multi-region database writes.

### Cross-Region Session Replication

When a user authenticates in US-East, their session must be available in EU-West within seconds — otherwise, a user who travels between regions will be forced to re-authenticate.

The session replication bus works as follows:

1. **Publish**: When a session is created or updated in any region, the auth service publishes a `session_created` or `session_updated` event to a global Kafka topic.
2. **Consume**: Each region runs a consumer that reads from the global topic and writes the session data to the local Redis.
3. **Replicate**: The consumer uses `SET session:{token_id} {data} EX {ttl}` to write the session with the same TTL as the origin region.
4. **Conflict resolution**: If two regions create a session for the same user simultaneously (unlikely but possible), last-write-wins based on the timestamp in the event. The user would need to re-authenticate on one device, which is acceptable.

Typical replication latency: 50-200ms at p99. This means a user who logs in from US-East and immediately flies to EU-West (a 7-hour flight) will find their session replicated. The replication window is important for the edge case where a user logs in, closes their laptop, and opens it on a different network that routes to a different region — the 200ms delay is invisible to the user.

### Regional Failover

If a region becomes unavailable, DNS-based routing redirects traffic to the nearest healthy region. The failover is transparent to the user because sessions are replicated across regions.

```
Failover sequence:
  t=0:   us-east-1 region becomes unreachable (network partition)
  t=1s:  Health check fails (ALB /health endpoint, 3 consecutive failures)
  t=3s:  Route53 health check detects failure, removes us-east-1 from DNS
  t=5s:  DNS TTL expires (60s for standard, but clients may cache longer)
  t=10s: EU-West and AP-SE begin receiving redirected traffic
  t=30s: EU-West auth cluster auto-scales from 15 to 25 nodes
  t=60s: All clients have resolved the new DNS target
  t=5min: Operations team assesses the outage

User impact:
  - Users with active sessions in the failed region: transparent failover
  - Users mid-login at the time of failure: HTTP 502, retry succeeds in new region
  - Users attempting new login: transparent, routed to nearest healthy region
```

## Deployment and Operations

### Zero-Downtime Auth Upgrades

Authentication is the most critical system in your infrastructure. A deployment that causes even 30 seconds of login failures is a production incident. The following practices ensure zero-downtime auth upgrades.

**Blue-green deployment for the auth service:**

```
Phase 1: Provision Green
  - Deploy new auth service version to Green environment
  - Green runs alongside Blue but receives no traffic
  - Validate Green against Blue's database (read-only queries)
  - Run canary: send 1% of traffic to Green, monitor for 5 minutes

Phase 2: Switch Traffic
  - Update load balancer target group to point to Green
  - Connection draining on Blue: 60 seconds
  - All new requests go to Green
  - Blue kept warm for 30 minutes (rollback window)

Phase 3: Validate
  - Monitor login success rate (should be unchanged)
  - Monitor p99 latency (should be within 5% of baseline)
  - Monitor error rate (should be 0% above baseline)
  - If any metric degrades: switch back to Blue

Phase 4: Decommission
  - After 30 minutes with no issues, decommission Blue
  - This becomes the next deployment's Green
```

**Database schema changes for auth:**

The user store schema is the riskiest component to change. Apply the expand-contract pattern:

```
Adding a new MFA method column:
  Phase 1 (Expand):
    ALTER TABLE users ADD COLUMN passkey_credentials JSONB;
    - Application continues to write to mfa_secret (old) and reads from both
    - No downtime, no lock (PostgreSQL ADD COLUMN DEFAULT NULL is instant)

  Phase 2 (Migrate):
    - Background job: for users who enroll a passkey, populate passkey_credentials
    - Application writes to both columns during transition

  Phase 3 (Contract):
    - After 2 weeks, verify passkey_credentials is fully populated
    - ALTER TABLE users DROP COLUMN mfa_secret;
    - This is a separate deployment, 2 weeks after the expand
```

### Operational Runbooks

**Login success rate drops below 90%:**

```
1. Check rate limiter: Is a legitimate user base being throttled?
   → Check per-IP and per-account rate limit counters in Redis
   → If a range of IPs is being rate limited, verify against known attack patterns

2. Check user store: Is the database reachable?
   → Check PostgreSQL connection pool saturation
   → Check replica lag (if read replicas are serving stale data)
   → Check for table locks (pg_locks, long-running queries)

3. Check Redis session store: Can sessions be created?
   → Check Redis memory usage (evictions mean sessions are being dropped)
   → Check Redis cluster health (all nodes reachable, no split-brain)
   → Check for Redis OOM (out-of-memory kills)

4. Check IdP integration: Are external logins failing?
   → Check IdP API response times and error rates
   → Check JWKS cache freshness (expired keys cause JWT validation failures)
   → Check rate limits on IdP API quota

5. Check MFA service: Are MFA challenges failing?
   → Check TOTP clock drift (NTP sync issues)
   → Check SMS/email provider status
   → Check WebAuthn origin validation (must match production domain)
```

**MFA challenge time degrades to 5+ seconds:**

```
1. Check SMS provider: Is Twilio / AWS SNS / third-party provider healthy?
   → Check provider status page
   → Check API response times
   → Switch to fallback provider if available

2. Check email delivery: Is SendGrid / SES / Mailgun healthy?
   → Check delivery queue depth
   → Check IP reputation (blacklisted IPs cause delivery delays)
   → Check DKIM/SPF configuration (misconfigured DNS causes retries)

3. Check TOTP verification: Is clock drift causing window expansion?
   → Check NTP synchronization on all auth nodes
   → Check MFA service logs for time-window errors
   → If clock drift is detected, resync NTP and force re-enrollment for affected users
```

**Suspicious lockout rate spike:**

```
1. Is this a credential stuffing attack?
   → Check IP distribution: single IP or distributed?
   → Check account targeting: random accounts or specific targets?
   → Check timing: uniform rate or bursty pattern?

2. If credential stuffing:
   → Enable CAPTCHA on login page (Cloudflare Turnstile, reCAPTCHA)
   → Increase rate limiting for suspicious IP ranges
   → Enable device fingerprinting to detect automated browsers
   → Notify security team and prepare incident response

3. If attack is confirmed:
   → Block offending IP ranges at the load balancer (WAF rules)
   → Enable challenge-response for all login attempts from suspicious regions
   → Notify affected users (email, in-app notification)
   → Post-mortem: update rate limiting rules, add new detection patterns
```
