## Internal Architecture

A production authentication system is not a single service that checks a password against a hash. It is a pipeline of seven subsystems, each with its own failure modes and performance characteristics.

## Data Structures -- Password Hashing Algorithms

Password hashing is deliberately expensive. The three dominant algorithms -- bcrypt, scrypt, and argon2 -- each use a different strategy to resist brute-force and ASIC/GPU-based attacks.

### bcrypt

bcrypt embeds a cost factor directly into the algorithm. The cost factor (2^cost) determines the number of key expansion rounds. A cost of 12 means 2^12 = 4,096 rounds of Blowfish key expansion.

| Property | Value |
|---|---|
| Algorithm | Blowfish-based key schedule + 64 iterations of the cipher |
| Salt size | 16 bytes (128 bits) |
| Hash output | 184 bits (23 bytes) encoded as 60-character string |
| Memory usage | ~4 KB per hash (fixed, low) |
| Time complexity | O(2^cost) -- each round is the same computation |
| Space complexity | O(1) -- constant memory, no large buffers |
| GPU resistance | Low -- fits easily in GPU shared memory |
| Collision resistance | 2^184 |

The output format embeds everything needed for verification:

```
$2b$12$WlI3qJk9X7M5pR2KvBnDzu4m1Yp8RfTgHjNkLpQwErTyUiOpAsDfGhJkLzXcV
 \_/ \_/ \_____________________________________________________________/
  |   |                          |
  |   cost                    hash + salt
  algorithm prefix
```

### scrypt

scrypt adds memory hardness. It fills a large buffer with pseudo-random data, then reads from that buffer in a data-dependent pattern. This defeats ASIC attackers because the memory footprint is too large for chip-embedded SRAM.

| Property | Value |
|---|---|
| Parameters | N (CPU/memory cost), r (block size), p (parallelization) |
| Typical N | 2^14 = 16,384 (interactive), 2^20 = 1,048,576 (sensitive) |
| Memory usage | 128 * N * r bytes -- typically 16 MB (N=2^14, r=8) |
| Time complexity | O(N * r * p) |
| Space complexity | O(N * r) -- proportional to N and r |
| GPU resistance | High -- memory bandwidth bottleneck |
| Collision resistance | 256-bit (SHA-256 finalization) |

### argon2

argon2 is the winner of the 2015 Password Hashing Competition. It has three variants: argon2d (data-dependent, GPU-resistant), argon2i (data-independent, side-channel resistant), and argon2id (hybrid, recommended).

| Property | Value |
|---|---|
| Parameters | m (memory), t (time), p (parallelism) |
| Typical m | 64 MB (interactive), 256 MB (sensitive) |
| Typical t | 2 (interactive), 3 (sensitive) |
| Typical p | 1 (single thread) |
| Time complexity | O(m * t) -- memory accesses x iterations |
| Space complexity | O(m) -- proportional to memory parameter |
| GPU resistance | Very high -- memory bandwidth bottleneck + data-dependent access |
| Side-channel resistance | High (argon2id) -- first pass uses i variant, rest uses d variant |

### Why Salt Is Required

Without a salt, two users with the same password produce the same hash. An attacker with a precomputed rainbow table can reverse the hash in seconds. With a per-user salt, each password hash is unique, and the attacker must brute-force each hash individually.

### Comparison Summary

| Algorithm | Cost per hash (modern CPU) | Memory per hash | GPU resistance | Recommended for |
|---|---|---|---|---|
| bcrypt cost 10 | ~80 ms | 4 KB | Low | Legacy systems, low-traffic APIs |
| bcrypt cost 12 | ~250 ms | 4 KB | Low | Most web applications (default) |
| bcrypt cost 14 | ~1,000 ms | 4 KB | Low | Admin accounts, high-security |
| scrypt N=2^14 r=8 p=1 | ~150 ms | 16 MB | Medium | General purpose, password managers |
| argon2id m=64MB t=2 p=1 | ~200 ms | 64 MB | High | New systems, high-security apps |
| argon2id m=256MB t=3 p=1 | ~1,200 ms | 256 MB | Very high | Key derivation, cryptocurrency |

## Request Lifecycle -- 12 Steps from Credential to Session

Every login request passes through a 12-step pipeline. Each step has a measurable latency budget.

### Step 1: Credential Submission

The client sends a POST request to `/auth/login` with a JSON body containing `email` and `password`. The credential collector parses the body, validates the Content-Type header, and decodes base64-encoded credentials if present.

### Step 2: Rate Limiting Check

The rate limiter checks three keys: the client IP, the email address, and the global rate. If any exceeds its threshold, the request is rejected with HTTP 429 and a `Retry-After` header.

### Step 3: Credential Validation

The credential validator checks format rules: email format via regex, minimum password length, null byte injection, and SQL injection patterns. Invalid requests are rejected at this stage without touching the database or hasher.

### Step 4: Credential Lookup

The user store queries the credential database by email. If the user does not exist, the system returns a generic "invalid credentials" error (not "user not found") to prevent enumeration attacks.

### Step 5: Password Verification

The password hasher loads the stored hash string, extracts the algorithm identifier, salt, and cost parameters, and runs the hash function with the submitted password. The result is compared to the stored hash in constant time to prevent timing attacks.

### Step 6: MFA Challenge (if enabled)

If the user has MFA enabled, the MFA service generates a challenge. For TOTP, the service verifies the six-digit code against the current time window. For WebAuthn, the service sends a challenge nonce for the authenticator to sign.

### Step 7: Token Issuance

The token issuer generates one of two token types: a signed JWT (stateless, carries claims) or an opaque session ID (stateful, stored in Redis). The choice depends on the system architecture.

### Step 8: Session Creation

The session manager creates a session record in Redis (for opaque sessions) or encodes claims into the JWT (for stateless sessions). The session record includes the user ID, roles, permissions, expiration time, and a refresh token fingerprint.

### Step 9: Audit Logging

The audit logger records the login event: timestamp, user ID, IP address, user agent, success/failure status, and MFA method used. Failed attempts are logged with the reason for failure.

### Step 10: Response Construction

The authentication service constructs the HTTP response. For a successful login, it sets a secure, HttpOnly, SameSite cookie containing the session token or JWT. It also returns a refresh token in the response body (for SPAs to store in memory).

### Step 11: Background Processing

After the response is sent, the system triggers background tasks: updating the last login timestamp, clearing failed attempt counters, and enqueueing a login notification email.

### Step 12: Client-Side Token Storage

The client stores the token (cookie for browser apps, memory for mobile apps) and attaches it to subsequent requests via the `Authorization` header or cookie.

## Consistency and State

Authentication systems face three challenging consistency problems. Each one can cause user-facing issues if not handled correctly.

### Scenario 1: Node Crashes Mid-Signup

**Root cause**: The database commit is durable but the response never reaches the client. The user row exists in a half-verified state.

**Fix options**:
1. Use a signup token: generate a UUID before the DB insert, store it in the session, and only commit the transaction after confirming the token is still valid. If the node crashes, the token is lost and the orphaned row is never committed.
2. Use a cleanup job: run a periodic task that deletes user rows where `email_verified = false` and `created_at < NOW() - INTERVAL 24 HOURS`.
3. Use a two-phase commit protocol: the coordinator node writes a "prepare" record, then waits for acknowledgement from all participants before issuing the "commit". If the coordinator crashes, the recovery process rolls back the partial transaction.

### Throughput Limits

The password hasher is the bottleneck. A single CPU core can handle approximately:

- bcrypt cost 10: 12.5 hashes/second
- bcrypt cost 12: 4 hashes/second
- bcrypt cost 14: 1 hash/second
- argon2id m=64MB t=2: 5 hashes/second
- argon2id m=256MB t=3: 0.8 hashes/second

For a server with 16 cores (32 hyperthreads), the maximum login throughput is:

| Algorithm | Core | Hashes/sec (16 cores) | Max logins/minute |
|---|---|---|---|
| bcrypt cost 10 | 25 ms | 200 | 12,000 |
| bcrypt cost 12 | 250 ms | 64 | 3,840 |
| bcrypt cost 14 | 1,000 ms | 16 | 960 |
| argon2id m=64MB | 200 ms | 80 | 4,800 |
| argon2id m=256MB | 1,200 ms | 13 | 780 |

### Memory Footprint

| Component | Memory per request | Total at 1,000 concurrent logins |
|---|---|---|
| HTTP request buffer | 8 KB | 8 MB |
| Credential validator state | 1 KB | 1 MB |
| bcrypt hash (4 KB) | 4 KB | 4 MB |
| argon2id hash (64 MB) | 64 MB | 64 GB (not practical) |
| Session token | 2 KB | 2 MB |
| Audit log buffer | 1 KB | 1 MB |
| Total per request (bcrypt) | ~16 KB | 16 MB |
| Total per request (argon2id) | ~64 MB + overhead | 64+ GB |

Argon2id with 64 MB memory per hash is not practical for high-concurrency systems. You need to either use bcrypt (4 KB, lower memory) or limit concurrent login requests to a small number of workers. A common pattern is to use a separate, small pool of hashing workers (1-4 cores) and queue login requests behind them.

## Configuration and Tuning

### Password Policy

| Setting | Default | Recommended | Effect |
|---|---|---|---|
| min_length | 8 | 12 | Prevents brute-force on short passwords |
| require_uppercase | false | true | Increases entropy per character |
| require_lowercase | false | true | Increases entropy per character |
| require_digit | false | true | Prevents dictionary-only passwords |
| require_special_char | false | true | Expands character set to 95+ |
| max_length | 128 | 128 | Prevents hash-length DoS attacks |

### Edge Cases and Failure Modes

| Scenario | Impact | Mitigation |
|---|---|---|
| Hash algorithm upgrade | Old passwords become unverifiable | Re-hash on next successful login using the new algorithm, store versioned hash |
| Redis outage | Session validation fails, all users logged out | Use a local cache fallback (5-second TTL) or switch to stateless JWT tokens |
| Clock skew on TOTP | TOTP codes generated at wrong time | Allow a window of +/- 30 seconds (one code before and after current) |
| Database connection pool exhaustion | Login requests queue up waiting for a DB connection | Set a short connection timeout (2 seconds), return 503 to clients |
| Concurrent password change | Old session still valid, new session created, race condition | Use a session version counter: increment on password change, invalidate all sessions with old version |
| Mass account lockout (DoS) | Attacker attempts 5 failed logins on every user account | Use progressive lockout (first lockout: 1 minute, doubling each time) and CAPTCHA after 2 failures |
| Token leaked in URL | JWT or session ID appears in server logs, referrer headers | Only accept tokens in headers or cookies, never in URL query parameters |
| Weak password after migration | Legacy system had no password policy, migrated users have weak hashes | Flag weak passwords on next login, force password change, and re-hash with the current algorithm |
