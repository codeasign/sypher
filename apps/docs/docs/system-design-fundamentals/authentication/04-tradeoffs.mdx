# Authentication — Trade-offs & Failure Modes

Every authentication decision is a compromise. When you choose stronger authentication, you are explicitly trading user convenience for security. A 10-second login delay might stop credential stuffing, but it will also frustrate legitimate users. A hardware-key requirement guarantees phish-proof MFA, but it locks out anyone without the physical token. This page walks you through the trade-off landscape, the failure modes that will test your system, and the anti-patterns that sink production auth at scale.

---

## Core Tension: Security vs. Convenience

Authentication exists on a spectrum. At one end, you have zero friction and zero trust. At the other, you have maximum verification and maximum friction. Your job is to pick the right point on that curve for each user population.

The key insight: **you do not need one authentication strategy for all users**. A dashboard admin uploading financial reports and a read-only API consumer fetching public weather data have very different risk profiles. Align your auth strength to the sensitivity of the resource, not to a one-size-fits-all policy.

### Measuring the Trade-off

You can quantify the security-convenience trade-off using two metrics:

- **False rejection rate (FRR)**: the percentage of legitimate users who fail to authenticate on their first attempt. A high FRR drives users to password reset flows, increases support tickets, and erodes trust.
- **False acceptance rate (FAR)**: the percentage of unauthorized access attempts that succeed. A high FAR means your authentication is not actually blocking attackers.

The goal is to minimize both simultaneously, but every design decision pushes one higher. A strict rate limiter with a low threshold lowers FAR but increases FRR during legitimate burst usage. A complex MFA flow lowers FAR but increases FRR for users who misplace their phone or mistype a code. Your job is to find the Pareto frontier where the next unit of security improvement costs more than a unit of user trust.

### Adaptive Authentication

The most sophisticated systems do not apply a single policy to all users. Instead, they use adaptive authentication that adjusts the required verification level based on contextual signals:

- **Device trust**: a user logging in from a recognized device with a recent cookie skips MFA. An unknown device triggers the full challenge.
- **Geolocation**: a login from the user's home city at a typical hour passes with low friction. A login from a foreign country two hours after the previous session flags for step-up.
- **Behavioral patterns**: typing cadence, mouse movement, and session duration form a baseline. A deviation from the baseline triggers a secondary challenge.
- **Resource sensitivity**: viewing a public profile page requires only a session token. Initiating a wire transfer requires re-authentication with MFA.

Adaptive authentication is the practical answer to the security-convenience dilemma. It applies stronger verification exactly where the risk is highest and steps out of the way when the context is low-risk.

## Decision Matrix

The table below captures the most common authentication decision points you will face when designing a system. Each row isolates one dimension so you can reason about the trade-off independently.

| Decision Point | Option A | Option B | Key Trade-off | When to Choose A | When to Choose B |
|---|---|---|---|---|---|
| Password vs. Passwordless | Email + password | Magic link / passkey | Password reuse risk vs. email delivery dependency | Legacy systems, offline-capable clients | Consumer apps, low-friction sign-up flows |
| Stateless Token vs. Server Session | JWT with self-contained claims | Opaque session ID stored in Redis | No DB lookup on verify vs. instant revocation | Read-heavy APIs with many verify calls | Systems where kill-switch revocation is critical |
| Self-Hosted IdP vs. Cloud IdP | Deploy Keycloak / Dex | Use Auth0 / Cognito / Firebase | Full control and data sovereignty vs. operational simplicity | Regulated industries, on-prem requirements | Startups, teams with no dedicated security engineer |
| MFA Mandatory vs. Optional | Enforce MFA for every login | Let users opt in later | Maximum account protection vs. sign-up friction | Banking, healthcare, admin portals | Content sites, read-only dashboards |
| Short Session TTL vs. Long | 15-minute access + refresh token | 7-day session with sliding expiry | Fewer stolen-token windows vs. frequent re-authentication | Financial dashboards, payment UIs | Social media, content browsing |
| Email OTP vs. SMS OTP vs. Authenticator App | Code delivered via email | Code delivered via SMS | TOTP via authenticator app | No app install needed, still decent security | No app install needed, universal reach | Strongest, no SMS intercept possible | Low-risk flows, backup channel | Users with legacy phones | Power users, enterprise employees |
| Single Origin vs. FDO (Federated) | One auth server | Multiple IdPs via OIDC/SAML | Simple trust model vs. cross-org interoperability | Monolith, single-tenant | Multi-tenant SaaS, enterprise B2B |

These are not binary choices. A production system often blends both sides: you might use JWTs for your public API tier while maintaining server-side sessions for admin panels, or enforce MFA for finance features while keeping it optional for profile settings.

### How to Use the Decision Matrix

The matrix is a conversation starter, not a checklist. For each decision point, you should:

1. **Identify the threat model**: who is trying to access your system, and what happens if they succeed? A bank and a blog have very different answers.
2. **Estimate user tolerance**: how much friction will your user base accept before churning? Enterprise B2B users tolerate more friction than a consumer social app.
3. **Measure the operational cost**: self-hosting an IdP gives you control but requires a team to patch, scale, and monitor it. Cloud IdPs shift that cost to a monthly bill but add vendor lock-in risk.
4. **Plan for evolution**: the right choice for a startup with 100 users is wrong for the same company at 1 million users. Design your auth layer so you can migrate from one side of a decision to the other without rewriting the entire system.

### The Cost of Choosing Wrong

Each wrong decision has a distinct cost profile:

| Wrong Choice | Immediate Cost | Long-Term Cost |
|---|---|---|
| Password-only (no MFA) | Low friction sign-up | Account takeover incidents, user trust erosion |
| Stateless tokens with no revocation | Fast API responses | No way to kill a leaked session; entire user base must re-authenticate after a breach |
| Self-hosted IdP with no dedicated ops | Full data control | Pager fatigue, missed SLA for auth uptime, single point of failure |
| Infinite session TTL | Zero re-auth prompts | Stolen sessions stay valid indefinitely; compliance audit failures |
| SMS-only MFA | Universal reach | SIM swap attacks, 3c per message at scale, delivery delays during carrier outages |

The table is not meant to discourage you from choosing any option — it is meant to make the hidden costs visible so you can budget for the mitigation.

## CAP Theorem Positioning: Auth Is CP

Authentication is a **CP** system in CAP theorem terms — you must prioritize **Consistency** over **Availability** during credential verification.

### Why Consistency Wins

When a user changes their password, every node in your system must agree on the new credential before you can safely accept or reject a login attempt. Serving a stale password hash from a replica that has not received the update yet would let an attacker in with the old credential. Consistency is non-negotiable.

### Partition Scenario Walkthrough

Imagine a network partition splits your authentication service into two groups. A user connects to group A and changes their password. Their friend, still holding the old password, hits group B three seconds later. What happens?

<AsciiDiagram
  id="system-design-fundamentals/authentication/tradeoffs/partition-walkthrough"
  title="Network Partition — Auth Read Strategy"
  alt="Sequence diagram showing two read strategies under a network partition: strongly consistent reads go to the primary and wait, eventually consistent reads serve stale data from a replica"
  content={`
┌────────────────────────────────────────────────────────────────┐
│              Partition — Read Strategy Comparison               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   Strongly Consistent Read:                                     │
│                                                                │
│   Auth Service ──► Primary DB ──► Wait for quorum              │
│       │                    │                                    │
│       │◄─── latest_hash ──│◄─── acknowledged                   │
│       │                    │                                    │
│       └── Verify: CORRECT ──── Access GRANTED                  │
│                                                                │
│   Eventually Consistent Read:                                   │
│                                                                │
│   Auth Service ──► Replica DB ──► old_hash returned            │
│       │                    │                                    │
│       └── Verify with old hash ── Access GRANTED (WRONG!)     │
│                                                                │
│   VERDICT: Strong consistency prevents the stale read.         │
│   The partition delays login but never accepts a stale         │
│   credential.                                                   │
└────────────────────────────────────────────────────────────────┘
`}
  caption="Strong consistency trades a few hundred milliseconds of latency for the guarantee that a credential change is visible everywhere before the next verify."
/>

The correct answer depends on your read strategy:

- **Strongly consistent read** (read from the primary or a quorum): the write must propagate before group B can answer. The friend gets a retry or timeout. The system stays secure.
- **Eventually consistent read** (read from any replica): group B might serve the old hash and let the friend in. This is a security vulnerability disguised as improved availability.

### Handling the Partition in Production

Network partitions are rare but they happen. Here is how production systems deal with the tension between consistency and availability during an auth partition:

| Strategy | What It Does | Trade-Off |
|---|---|---|
| Fail closed | Reject all login attempts until the partition heals | Maximum security, but a total auth freeze during the partition |
| Fail open with degraded mode | Allow logins with cached credentials; require MFA for new logins | Users who logged in recently can access; new users are blocked |
| Local token validation | Validate the JWT signature locally without contacting the auth database | Existing sessions work; password changes and revocations are invisible |
| Primary affinity | Route all auth reads for a user to the primary for N seconds after a write | Prevents the stale-read scenario for the most sensitive window |

Most production systems combine local token validation for active sessions with fail-closed for new login attempts during a partition. Existing users stay online, and the blast radius of the partition is limited to new authentications.

**Rule of thumb**: for the verify-critical path of authentication, always read from the source of truth. Cache the result aggressively so subsequent checks from the same user within seconds are fast, but the first read after a credential change must be consistent.

## Failure Modes

Authentication systems fail in predictable patterns. Below are the most common failure modes you will encounter at scale, each with detection strategy and recovery procedure.

### Failure Mode 1: IdP Outage

Your identity provider — whether self-hosted Keycloak, Auth0, or Cognito — goes down. Every login, token refresh, and user lookup fails simultaneously.

**Detection**: Monitor IdP health endpoint (e.g., `GET /health` or `/.well-known/openid-configuration`). Track 5xx responses from the IdP as a percentage of total auth requests. Alert when the rate exceeds 5% over a one-minute window.

**Recovery**:
1. **If you self-host**: fail over to a standby IdP instance in a different availability zone. Use a global load balancer with health-check-based routing.
2. **If you use a cloud IdP**: enable the provider's failover region. Most major IdPs (Auth0, Cognito) offer multi-region replication; ensure it is configured before the outage.
3. **Emergency bypass**: for internal services, fall back to a local token validation cache. Do **not** bypass auth for external users. A read-only mode that serves cached tokens and rejects new logins is acceptable.
4. **Communicate**: surface a clear status message to users. "We are experiencing a temporary authentication issue. Please try again in a few minutes" is better than a generic 500.

### Failure Mode 2: Database Replication Lag

A user changes their password. The write lands on the primary database but has not replicated to the read replica your auth service queries. The next login attempt reads the old password hash and succeeds.

/>

**Detection**: Track the replication lag metric from your database (e.g., `seconds_behind_master` in MySQL, `replay_lag` in PostgreSQL with logical replication). Alert when lag exceeds 2 seconds on any replica used by the auth service.

**Recovery**:
1. **Read from primary on credential change**: after a password update, pin subsequent reads for that user to the primary database for the next 10-60 seconds. Use a "read-your-writes" consistency guarantee scoped to the auth domain.
2. **Short-circuit with write marker**: set a Redis flag `pwd_changed:{user_id}` with a 30-second TTL when a password is updated. On login, check the flag first. If set, force a primary read.
3. **Reduce replication lag**: if lag is chronic, upgrade replica instance size, increase network bandwidth between primary and replica, or switch to semi-synchronous replication.
4. **Audit log analysis**: replay auth logs for the lag window after an incident to determine whether any compromised accounts need forced session invalidation.

Attackers use password lists from previous data breaches to automate login attempts against your system. Even with basic rate limiting, a distributed botnet can generate thousands of requests per second from thousands of IPs.

**Detection**:
- Sudden spike in `401 Unauthorized` responses (monitor as a percentage of total login attempts).
- Elevated ratio of failed logins to successful logins for a given user population.
- High count of distinct IPs hitting the login endpoint with few requests each (the botnet signature).
- Known breached credential feeds — subscribe to services like Have I Been Pwned for Enterprises or integrate with your cloud provider's credential leak detection.

**Recovery**:
1. **Tiered throttling**: start with per-IP rate limiting (100 requests/minute). After the third failure for a given username, escalate to per-user throttling (1 request/5 seconds). After the tenth failure, lock the account for 15 minutes or require MFA, regardless of whether the next password is correct.
2. **CAPTCHA escalation**: introduce a CAPTCHA challenge after the second failed login on the same account. For headless botnets, this is often a hard stop.
3. **Credential leak check**: compare the attempted password hash against a known compromised credential database. Reject and force password reset if there is a match.
4. **Distributed defense**: use a WAF (Cloudflare, AWS WAF) with bot detection rules. These services identify non-browser user agents, headless browser fingerprints, and requests originating from known VPN or Tor exit nodes.
5. **Notify affected users**: send password-reset emails to accounts that were targeted, advising them to enable MFA.

### Failure Mode 4: Token Leak

A JWT — or, worse, a long-lived refresh token — is leaked via an XSS attack, a compromised client-side log, a man-in-the-middle on an unencrypted network, or a malicious browser extension. The attacker now has a valid session.

**Detection**:
- Anomalous geographic or IP origin for a token that was issued to a different location (e.g., token issued in New York, used in Lagos one minute later).
- Parallel sessions from the same token (same `jti` claim used from two different IPs simultaneously).
- Access to resources the user has never accessed before during their session history.

**Recovery**:
1. **Short TTLs**: keep JWT access token TTLs short (5-15 minutes). A leaked token is only useful for minutes, not days.
2. **Refresh token rotation**: issue a new refresh token with every rotation and invalidate the old one. If a stolen refresh token is used after the legitimate user has already rotated it, the attacker's request fails, and you can flag the account for investigation.
3. **Token blocklist**: maintain a Redis set of revoked token `jti` values with TTL equal to the token's remaining lifetime. Check the blocklist on every protected request.
4. **Forced re-authentication**: when a token leak is confirmed, increment the user's `token_version` in the database. All existing tokens — regardless of expiry — become invalid because the token version claim no longer matches.
5. **Certificate pinning / token binding**: bind the token to a TLS session or device fingerprint so the token is useless on a different machine.

**Detection**:
- Monitor MFA provider health endpoint and response latency. Alert when p99 latency exceeds 5 seconds or error rate exceeds 2%.
- Track the rate of successful MFA challenges versus total MFA challenges initiated. A sudden drop in the completion rate signals provider failure.
- Subscribe to the provider's status page webhook (most major providers offer one).

**Recovery**:
1. **Fallback channel**: offer at least two MFA methods per user. If TOTP is the primary, allow a recovery code or email-based one-time code during the outage window.
2. **Grace period**: if the provider is degraded (not completely down), increase the TOTP validity window from 30 seconds to 90 seconds to account for delivery delays.
3. **Administrative override**: have a manual process (with audit trail) for support engineers to bypass MFA for verified account owners during a confirmed provider outage. Require two-person sign-off and limit override validity to one hour.
4. **Local fallback for TOTP**: if you use TOTP, the TOTP seed is stored on your authentication server — you do not need the external provider to verify the code. The "provider downtime" for TOTP is only about clock skew or the code delivery channel (authenticator app generation is local to the device).
5. **Communicate the outage**: show a message on the login page indicating the secondary factor is unavailable and directing users to an alternative login method.

### Failure Mode 6: Session Fixation

An attacker tricks a user into authenticating with a session identifier the attacker already knows. After the user logs in, the application does not rotate the session ID, so the attacker's pre-set session ID becomes the authenticated session. The attacker now has full access to the user's account.

**Detection**:
- Monitor for a single session ID being used from two different IP addresses before and after a login event.
- Track the rate of session creation without a corresponding login event — a spike may indicate attacker-injected session IDs.
- Audit logs showing the same session ID in a pre-authentication request and a post-authentication request without a rotation event.

**Recovery**:
1. **Always rotate session IDs on login**: generate a new session identifier the moment authentication succeeds. Never reuse the pre-login session ID. Frameworks like Express Session, Django, and Spring Security do this by default — do not disable it.
2. **Regenerate on privilege escalation**: even within an authenticated session, rotate the session ID when the user escalates privileges (e.g., enabling admin mode, verifying a second factor).
3. **Bind session to fingerprint**: associate the session ID with the user's TLS session ID, IP address, or user-agent string. If any of these change mid-session, force re-authentication. This limits the attacker's ability to reuse a fixed session from a different machine.
4. **Short session TTL**: a session fixation attack is only useful as long as the session is valid. Short TTLs limit the window of exploitability.
5. **Accept only server-generated session IDs**: reject any session ID that the client sets via a URL parameter or cookie. Issue session IDs exclusively through the server's Set-Cookie response.

## Anti-Patterns

These patterns appear in production systems more often than you would expect. Each one represents a real incident waiting to happen.

### Anti-Pattern 1: Storing Passwords in Plaintext

You would never do this in 2025, you say. Yet scans of public GitHub repositories regularly find `INSERT INTO users (password) VALUES ('sup3rS3cr3t')` in migration scripts, and internal audit tools uncover plaintext password columns in production databases.

**Why it happens**: a team rushes to ship a prototype, authentication is treated as a non-differentiating feature, and technical debt accumulates. The plaintext column never gets migrated because there are no existing hashes to compare against.

**The fix**: use a strong, slow, salted hashing algorithm. Argon2id is the current gold standard. bcrypt with cost factor 10-12 is an acceptable alternative. scrypt is also viable. Never use MD5, SHA-1, SHA-256 alone (they are fast — too fast for password storage), or a homegrown hash with a secret pepper as the only protection.

```
// DO NOT do this
const password = req.body.password;
const query = `INSERT INTO users (email, password) VALUES (?, ?)`;
db.execute(query, [email, password]);

// Do this instead
const argon2 = require('argon2');
const hash = await argon2.hash(password, { type: argon2.argon2id });
const query = `INSERT INTO users (email, password_hash) VALUES (?, ?)`;
db.execute(query, [email, hash]);
```

### Anti-Pattern 2: Rolling Your Own Crypto

Writing your own encryption algorithm, JWT signing mechanism, or token generation scheme is the fastest way to introduce a vulnerability that no third-party library audit has caught.

**Why it happens**: developers overestimate the difficulty of cryptography or believe a custom scheme is more "secure" because it is obscurer. In reality, every homegrown crypto scheme has failed within months of public deployment.

**The fix**: use standard, audited libraries for everything. For JWT, use `jsonwebtoken` (Node.js), `PyJWT` (Python), or `jjwt` (Java); never construct and sign tokens with raw `HMAC-SHA256`. For encryption, use `libsodium` (via `tweetnacl` or `libsodium.js`). For password hashing, use `argon2` or `bcrypt`. Do not concatenate random bytes and Base64-encode them with a custom alphabet; use `crypto.randomUUID()` or `uuid.v4()`.

### Anti-Pattern 3: Infinite Session Lifetimes

A session that never expires is a permanent credential leak waiting to be exploited. If a user logs into a public kiosk, checks email, and walks away, an infinite session means the next person who sits down is logged in as them.

**Why it happens**: developers want a frictionless user experience. The product team requests "never log out." The implementation skips the expiry check to save a few milliseconds per request.

**The fix**:
- Set a hard maximum session lifetime (e.g., 7 days for consumer apps, 8 hours for enterprise admin panels).
- Implement sliding expiry: reset the expiry timer on every request, but never extend beyond the hard limit.
- Force re-authentication for sensitive operations (password change, MFA removal, payment info update) regardless of session freshness.
- Issue refresh tokens with absolute expiry and rotate them on each use.

### Anti-Pattern 4: No Rate Limiting on Login

Without rate limiting, your login endpoint is an open door for credential stuffing, brute force, and denial-of-service attacks.

**Why it happens**: rate limiting adds latency and complexity. A single authentication server might handle requests from a load balancer, making it hard to track per-IP counts without a shared store. Teams defer it to "the next sprint" that never comes.

**The fix**:
- Implement per-IP rate limiting (e.g., 20 attempts per minute) in your API gateway or reverse proxy layer.
- Implement per-account rate limiting (e.g., 5 attempts per minute per username) in your auth service, backed by Redis.
- Implement per-device fingerprint rate limiting (e.g., 10 attempts per minute per browser fingerprint hash) to catch attackers rotating IPs.
- Return a generic error on failure. Do not distinguish between "user not found" and "wrong password" — that information helps attackers enumerate valid usernames.
- Introduce CAPTCHA after the third failed attempt and progressively harder challenges after each subsequent failure.

### Anti-Pattern 5: Client-Side Auth Decisions

A single-page application checks `localStorage.getItem('isAdmin') === 'true'` and conditionally renders an admin panel. An attacker who inspects the bundle, finds the API endpoint, and calls it directly bypasses the client-side check entirely.

**Why it happens**: teams confuse "hiding the UI element" with "securing the resource." The frontend check feels sufficient during development because the developer never tests the raw API call.

**The fix**:
- Every protected resource must validate authorization on the server. The client view is a UX concern, not a security boundary.
- Never store role or permission data in a place the client can write (localStorage, sessionStorage, client-side cookies). Use an httpOnly cookie for the session token or, if using JWTs, store the token in a secure, httpOnly cookie.
- Validate every API request against the user's actual roles and permissions from a trusted source (the token's claims or a server-side session lookup).
- Apply the principle of least privilege: the server should default to denying access and explicitly allow only what the user needs.

### Anti-Pattern 6: Weak Password Reset Flows

A password reset flow that uses a guessable, predictable, or long-lived reset token is a backdoor into any account. If the reset token is a sequential integer, a six-character numeric code, or a token that never expires, an attacker can enumerate or brute-force their way into any account.

- Generate reset tokens with at least 128 bits of entropy using a cryptographically secure random generator (`crypto.randomBytes(32).toString('hex')` in Node.js, `secrets.token_urlsafe(32)` in Python).

- Invalidate the token after a single use. If the user needs another reset, they must request a new one.
- Rate-limit reset requests per account (1 per minute) and per IP (5 per minute). This prevents an attacker from brute-forcing a short token.
- Never reveal whether an email address is registered. Always return the same message: "If an account with that email exists, a reset link has been sent." This prevents email enumeration.
- Require the user to re-enter their current password (or a recovery code) before allowing a password change from an authenticated session. A leaked session token should not be enough to hijack the password.

## When NOT to Use These Patterns

Not every system needs a full-fledged, multi-factor, token-rotating, IdP-backed authentication stack. Over-engineering auth adds cost, latency, and complexity. Here are scenarios where you can safely dial back.

### Simple Read-Only APIs

If your API serves public, read-only data — a weather forecast, a cryptocurrency price ticker, a documentation site — there is no resource to protect. Authentication adds latency and friction for callers who do not need it.

**What to do instead**: serve the data over HTTPS with optional rate limiting by source IP. If you need usage tracking, issue a lightweight API key (a random UUID stored in a database and passed as a header). No OIDC, no JWTs, no sessions.

### Internal Service-to-Service with mTLS

When two services within a private VPC communicate, mutual TLS (mTLS) provides both encryption and identity verification at the transport layer. Adding a JWT exchange on top is redundant.

**What to do instead**: configure mTLS between services. Each service presents a client certificate signed by your internal CA. The receiving service validates the certificate's CN or SAN against a known allowlist. If you need finer-grained authorization, embed roles in the certificate's extended key usage or use a service mesh (Istio, Linkerd) that supports mTLS-based authorization.

### Prototype with Five Users

If you are building a proof-of-concept for a team of five developers, you do not need OIDC, social login, MFA, token rotation, and a separate authorization service. Use a single shared credential or a simple password file until the prototype validates.

**What to do instead**: use HTTP Basic Auth over HTTPS. Add a simple user table with hashed passwords. Upgrade to a real auth system when you have more than 20 users or when external parties need access.

### Single-Tenant Local Application

An internal tool deployed on a company-managed laptop or local server — a log parser, a deploy dashboard, a local Jupyter notebook server — does not need a distributed authentication system. The authentication boundary is the device itself.

**What to do instead**: use OS-level authentication. The application trusts the system user. If you need an additional barrier, use a single shared password stored in an encrypted environment variable or a bearer token configured at deployment time. Do not build a user registration flow, a password reset endpoint, or an MFA enrollment page for a tool that three people use on their workstations.
