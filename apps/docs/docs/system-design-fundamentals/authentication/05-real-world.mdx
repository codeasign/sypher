# Authentication in Production

Authentication is easy to design on a whiteboard and hard to get right at
 planetary scale. This page walks through real-world architectures from the
 companies that shaped the industry, surveys the open-source and cloud
 ecosystems, and distills hard-won lessons from production incidents.

---

## Company Case Studies

Each of these companies operates at enormous scale and has made deliberate,
 often opinionated, choices about authentication. Understanding why they
 picked what they did — and what broke when those choices were tested — is
 the fastest way to internalize the trade-offs you will face.

---

### Google: Passwordless by Default with Passkeys

Google has been moving away from passwords for years. Their current
 architecture centers on **passkeys** (FIDO2/WebAuthn) as the primary
 authentication factor, with passwords and 2FA codes as fallbacks.

The core insight: a passkey is a key pair stored on the user's device. The
 private key never leaves the device. The public key is registered with
 Google's servers. During authentication, the device proves possession of the
 private key through a cryptographic challenge-response — no shared secret
 crosses the wire.

**Why Google went passwordless:**

- **Phishing resistance.** A passkey is scoped to a specific origin
  (`webauthn:google.com`). Even if a user lands on a lookalike domain, the
  passkey will not release the private key.
- **Reduced credential stuffing.** There is no password hash to leak and no
  shared secret to replay.
- **Better UX on mobile.** On Android, the passkey syncs via the device's
  secure credential manager. On iOS, it lives in iCloud Keychain. The user
  authenticates with their device biometrics — no typing.

Google still supports passwords for legacy clients, but as of 2024 the
 default sign-in flow starts with a passkey prompt. Only if the user has no
 passkey does the system fall back to a password.

**Key architectural choices:**

- Passkey sync is handled by the **platform** (Android, iOS, macOS), not by
  Google. This removes the liability of storing private keys server-side.
- The challenge-response protocol uses **origin-bound** credentials. This
  means a passkey registered for `google.com` cannot be used on
  `g00gle.com`.
- Google's servers never see the private key. The security model shifts from
  "protect stored secrets" to "protect the challenge-response protocol."

**Production numbers:** Over 2 billion passkey authentication events per day
across Google's services as of early 2025.

### Stripe: API Key-Based Auth for Services

Stripe's authentication model is unusual because their primary customers are
 **developers**, not end-users. Every API call to Stripe is authenticated with
 a **secret API key** — a long, high-entropy string that acts as both
 identifier and credential.

Stripe issues three types of keys:

- **Publishable keys** (`pk_live_...`): Used from client-side code. They are
  public and can identify your account but cannot perform sensitive actions.
- **Secret keys** (`sk_live_...`): Used from server-side code. They have full
  account access.
- **Restricted keys** (`rk_live_...`): Scoped to specific permissions. You
  can create keys that can only read charges or only write to customers.

The key itself is a prefix (`sk_live_`) followed by 24 characters of
 base-62 entropy — roughly 128 bits of randomness.

**Why API keys instead of OAuth 2.0 for service-to-service:**

1. **Simplicity.** A single string, a single header. No redirects, no refresh
   tokens, no discovery documents.
2. **Performance.** Key validation is a hash lookup — no asymmetric
   cryptography, no token signature verification at the edge.
3. **Revocation.** You revoke a key and the next request fails. No token
   expiry windows to worry about.

**The trade-off:** API keys are bearer tokens. Anyone who gets hold of your
 key can use it. Stripe mitigates this with:

- **Key prefixes** so logs can redact the secret but identify the key type.
- **IP allowlisting** for secret keys (optional but recommended).
- **Stripe Dashboard** integration — you can roll keys with zero downtime by
  having two active keys and rotating one at a time.
- **Automatic key rotation** for restricted keys used in PCI-scoped contexts.

**Production numbers:** Stripe handles millions of API requests per minute.
Key validation happens at the CDN edge in under 1 ms.

### Cloudflare: Zero Trust with Short-Lived Tokens

Cloudflare's own infrastructure runs on a **Zero Trust** model. No network
 segment is trusted by default. Every request — whether from an employee, an
 API client, or another service — must be authenticated and authorized.

The central mechanism is **short-lived tokens** (SLTs) issued by Cloudflare's
 Access service. These are JWTs with a time-to-live measured in minutes, not
 hours.

**Why short-lived tokens?**

- **Breach containment.** If a token leaks, it is valid for minutes, not days.
  The blast radius is small.
- **No revocation list needed.** By the time a token would appear on a
  revocation list, it has already expired. This is a deliberate architectural
  trade-off: you accept the overhead of re-authentication for the simplicity
  of not maintaining a CRL (Certificate Revocation List) or token blacklist.
- **Device posture checks.** Cloudflare's Access gateway evaluates device
  health (OS version, disk encryption, running processes) at every
  authentication event. Because tokens expire fast, device state is checked
  frequently.

**What about service-to-service auth?** Cloudflare uses **mutual TLS**
 (mTLS) between internal services. Each service has a short-lived X.509
 certificate issued by an internal CA. The certificate is rotated every 24
 hours. This means a compromised service cannot impersonate another service
 for more than 24 hours.

**Production numbers:** Cloudflare's edge processes over 30 million requests
 per second. Every one of them goes through Zero Trust authentication at the
 network edge.

### Okta: IdP Outage Postmortem

In October 2023, Okta experienced a **widespread outage** that locked users
 out of applications that depended on Okta for authentication. The root cause
 was not a security breach — it was a **configuration change** that cascaded
 through tightly coupled subsystems.

**Timeline of the incident:**

1. A configuration change was pushed to Okta's identity management service.
2. The change triggered a bug in the session management subsystem that caused
   an unexpected surge in database connections.
3. Database connection pools were exhausted, causing authentication requests
   to time out or fail.
4. Because Okta's services shared database infrastructure, the failure in
   session management cascaded to other subsystems (token issuance, user
   directory, admin console).
5. The recovery was slowed by the fact that the database pools could not be
   scaled quickly — connection draining took longer than expected.

**Impact:** Over 12,000 organizations were affected. Many could not access
 critical applications (Salesforce, Slack, AWS, GitHub) because those apps
 depend on Okta for SSO.

**Lessons learned from the Okta incident that you should apply to your own
 systems:**

1. **Isolate critical subsystems.** Session management should not share
   database pools with token issuance or user directory. If you cannot afford
   separate databases, use separate connection pools with strict limits.

2. **Test configuration changes in production-like staging.** Many outages
   start with a config push, not a code deploy. Your staging environment must
   mirror production traffic patterns to catch these bugs.

3. **Build gradual rollback capability.** Okta's recovery was slowed because
   connection draining was slow. Design rollback mechanisms that complete in
   seconds, not minutes.

4. **Define blast radius limits.** A single configuration change should not
   be able to take down the entire identity platform. Use circuit breakers
   and bulkheads per tenant or per service.

5. **Communicate proactively.** During an IdP outage, your customers cannot
   log into your status page to check for updates. Have a fallback
   communication channel (email, SMS, social media) that does not depend on
   your own authentication.

## Open Source Implementations

The open-source ecosystem provides several battle-tested authentication
 platforms. Each makes different trade-offs. Here is how to choose.

---

### Keycloak

**What it is:** A full-featured identity and access management (IAM) solution
 from Red Hat. Supports OAuth 2.0, OIDC, SAML 2.0, LDAP federation, and
 social login. Comes with a built-in admin console, user management, and
 customizable login themes.

**Strengths:**
- Batteries included. You get a login page, password reset flows, email
  verification, and user self-service out of the box.
- Social login (Google, GitHub, Facebook) with a few clicks.
- User federation — you can connect Keycloak to an existing LDAP or Active
  Directory without migrating users.
- Built-in session management with token refresh, offline sessions, and
  session revocation.

**Weaknesses:**
- Heavy resource footprint. Keycloak runs on WildFly (JBoss) by default and
  uses a relational database. A production deployment needs at least 1 GB of
  RAM per node.
- Complex configuration. The admin console exposes hundreds of settings.
  Getting them right requires deep understanding of the protocols.
- Slow startup time. Keycloak can take 30-60 seconds to boot, which makes
  autoscaling in Kubernetes challenging.

**When to choose Keycloak:** When you need a complete identity solution with
 minimal custom code. If your team has no identity domain expertise and you
 need SSO, social login, and user management in one package, Keycloak is the
 safest bet.

---

### Ory Hydra

**What it is:** A lightweight OAuth 2.0 and OIDC authorization server from
 the Ory ecosystem. Hydra is strictly an authorization server — it handles
 issuing tokens and validating them, but it does not manage users or provide
 a login UI. You bring your own identity provider and login frontend.

**Strengths:**
- Minimal footprint. Hydra is a Go binary that runs with very little memory.
- Stateless. Hydra can run as a stateless HTTP service with a PostgreSQL
  backend for persistent state. Easy to scale horizontally.
- BYO identity. You control the login flow and user store. Hydra only
  handles the OAuth 2.0 protocol layer.
- Security-first design. Hydra follows OAuth 2.0 Security BCP (Best Current
  Practices) strictly.

**Weaknesses:**
- No user interface. You must build your own login, consent, and account
  management screens.
- No user federation. You must implement or integrate your own user
  directory and authentication logic.
- Smaller community than Keycloak.

**When to choose Ory Hydra:** When you already have a user store and
 authentication logic, and you need a standards-compliant OAuth 2.0/OIDC
 layer on top of it. Hydra is popular among teams that want to own the
 authentication UX but delegate the token issuance protocol.

---

### Dex

**What it is:** A federated OIDC provider from the CoreOS / Cloud Native
 Computing Foundation ecosystem. Dex acts as a **bridge** — it connects
 multiple external identity providers (LDAP, SAML, GitHub, Google, Microsoft,
 OpenID Connect) and exposes them through a single OIDC endpoint.

**Strengths:**
- Connector architecture. Dex supports 15+ identity provider connectors out
  of the box. You configure connectors in a YAML file.
- Kubernetes native. Dex is designed to integrate with Kubernetes clusters.
  Many organizations use Dex to provide `kubectl` authentication through
  their corporate SSO.
- Lightweight. Dex is a single Go binary with a SQLite or PostgreSQL backend.

**Weaknesses:**
- No user management. Dex does not create or store user accounts. It passes
  identity claims through from upstream providers.
- Limited protocol support. Dex supports OIDC but not SAML or OAuth 2.0
  authorization grants beyond the authorization code flow.
- Not an IdP. Dex is a proxy/federation layer, not a full identity provider.

**When to choose Dex:** When you need to unify multiple identity providers
 behind a single OIDC endpoint. Dex is especially useful in Kubernetes
 environments where you need `kubectl` to authenticate through your
 corporate SSO.

---

### Authelia

**What it is:** An open-source authentication and authorization server
 focused on **self-hosting** and **2FA**. Authelia acts as a reverse proxy
 companion — it hooks into nginx, Traefik, or Caddy to protect web
 applications.

**Strengths:**
- Simple deployment. Authelia is a single Go binary with a YAML
  configuration file. You can set it up in minutes behind a reverse proxy.
- Strong 2FA. Supports TOTP, WebAuthn (passkeys), Duo Push, and email/SMS
  one-time codes out of the box.
- Access control rules. You can define per-application, per-user, and
  per-group access policies in YAML.

**Weaknesses:**
- No federation. Authelia does not support OIDC or SAML as an identity
  provider. It only authenticates against its own local database or an
  external LDAP server.
- No user self-service. User registration and password reset are
  administrator-managed.
- Limited scale. Authelia is designed for small to medium deployments
  (home labs, SMBs), not for multi-million user scenarios.

**When to choose Authelia:** When you are self-hosting web applications and
 need a simple, secure authentication gateway with 2FA. Authelia is popular
 in homelab and SMB environments where the overhead of Keycloak or Hydra is
 not justified.

## Cloud Provider Implementations

Each major cloud provider offers a managed identity platform. The choice
 between them often comes down to your existing cloud investment and the
 specific workloads you are running.

---

### AWS Cognito

Amazon Cognito provides user pools (authentication) and identity pools
 (authorization for AWS resources). It is the most widely used managed IdP
 in the AWS ecosystem.

**Strengths:**

- **Deep AWS integration.** Cognito connects naturally with API Gateway,
  ALB, AppSync, and Amplify. You can protect API Gateway endpoints with a
  Cognito user pool authorization without writing a single line of auth code.
- **Customizable authentication flows.** You can define custom challenge
  flows with Lambda triggers (pre-signup, post-authentication, pre-token
  generation, custom message).
- **Federation.** Cognito supports social identity providers (Google,
  Facebook, Amazon, Apple) and SAML/OIDC enterprise providers.
- **Adaptive authentication.** Cognito can evaluate risk factors (IP
  address, user agent, device) and require additional verification for
  high-risk requests.

**Weaknesses:**

- **Pricing at scale.** Cognito charges per monthly active user (MAU). At
  millions of users, this becomes expensive compared to self-hosting
  Keycloak.
- **Limited attribute size.** User pool attributes are capped at 2048 bytes.
  This can be restrictive if you need to store complex profile data.
- **Eventual consistency.** User pool changes can take seconds to propagate.
  This matters if you create a user and immediately need that user to
  authenticate.
- **No built-in MFA for custom flows.** While Cognito supports TOTP and SMS
  MFA, integrating these into custom authentication challenge flows requires
  Lambda orchestration.

**Cost estimate:** For 100,000 MAU, Cognito costs roughly $550/month (first
 50,000 MAU at $0.00550, next 50,000 at $0.00400). Above that, you can
 negotiate enterprise pricing.

---

### GCP Identity Platform

GCP's Identity Platform (formerly Firebase Authentication) is a customer
 identity and access management (CIAM) solution. It provides authentication
 as a service with a focus on client-side SDKs and social login.

**Strengths:**

- **Multi-platform SDKs.** Google provides SDKs for web, iOS, Android,
  Flutter, and React Native. The SDKs handle token refresh, session
  persistence, and reauthentication automatically.
- **Passwordless options.** Built-in support for email link (magic link)
  authentication and phone number authentication with SMS OTP.
- **Tenant support.** Identity Platform supports multi-tenancy out of the
  box, making it suitable for SaaS applications where each customer has
  isolated user directories.
- **Context-aware access.** Integration with Access Transparency and VPC
  Service Controls for enterprise security.

**Weaknesses:**

- **Google Cloud lock-in.** The Identity Platform is tightly coupled to the
  GCP ecosystem. Porting to another cloud is not straightforward.
- **Pricing for phone auth.** Phone number authentication has a per-verification
  cost that adds up quickly at scale.
- **Limited SAML support.** SAML federation is available only at the
  enterprise tier. OIDC is the primary federation protocol.
- **Rate limits.** The default rate limits for sign-in endpoints are
  conservative. You must request quota increases for high-traffic
  applications.

**Cost estimate:** Identity Platform has a free tier (50,000 MAU). Beyond
 that, pricing is $0.00550 per MAU for email/password, plus $0.06 per SMS
 verification. At 100,000 MAU, expect $275/month plus SMS costs.

---

### Azure AD / Microsoft Entra ID

Azure AD (now rebranded as Microsoft Entra ID) is Microsoft's cloud identity
 and access management service. It powers authentication for Microsoft 365,
 Azure, and thousands of enterprise SaaS applications.

**Strengths:**

- **Enterprise federation.** Entra ID is the most mature platform for
  enterprise federation with on-premises Active Directory. If you have an
  on-prem AD, Entra ID connects to it with seamless Kerberos trust.
- **Conditional Access.** You can define granular policies based on user,
  device, location, application, and risk level. This is the most
  sophisticated policy engine among the three cloud providers.
- **Microsoft 365 integration.** If your stack includes Microsoft 365,
  SharePoint, Teams, or Dynamics 365, Entra ID is the natural choice.
- **Privileged Identity Management (PIM).** Just-in-time access elevation
  for administrative roles. Users can request temporary admin access that
  auto-expires.

**Weaknesses:**

- **Complexity.** Entra ID has hundreds of settings, policies, and licensing
  tiers. Even experienced administrators struggle to configure it correctly.
- **Pricing for premium features.** Conditional Access, PIM, and Identity
  Protection require P1 or P2 licenses ($6-$9/user/month). For customer-facing
  apps, this can be cost-prohibitive at scale.
- **External identity limits.** Entra ID External Identities (B2B/B2C) have
  quotas and throttling limits that require Microsoft support intervention
  to raise.
- **Developer experience.** The Microsoft Graph API is powerful but
  complex. SDK support for non-Microsoft environments (Python, Go, Rust) is
  less mature than for .NET.

**Cost estimate:** Entra ID External Identities starts at $0.00325 per MAU
 for the free tier (up to 50,000 MAU). Premium features require P1 or P2
 licenses, which are priced per user, per month. For 100,000 external
 identities, expect $325/month for basic auth.

## Lessons Learned

From the case studies above and years of production experience, here are five
 key lessons you should carry into every authentication system you design.

### Lesson 1: Isolate Authentication Services from Everything Else

The Okta outage showed what happens when authentication shares infrastructure
 with other services. An IdP outage means nobody can log in — not even the
 engineers who are trying to fix it.

**Apply this by:**

- Running authentication services on separate database clusters with
  dedicated connection pools.
- Deploying authentication services in separate Kubernetes namespaces or
  service meshes.
- Having independent autoscaling policies for your auth layer.

### Lesson 2: Short-Lived Tokens Are a Simpler Security Model Than Token Revocation

Cloudflare proved that short TTLs simplify your security architecture. If a
 token expires in 5 minutes, you do not need a revocation list, a
 token-blacklist cache, or a distributed delete operation when a user logs
 out.

**Apply this by:**

- Setting access token TTLs to 5-15 minutes. Force clients to use refresh
  tokens for longer sessions.
- Using refresh token rotation (each refresh invalidates the previous
  refresh token) to detect token theft.
- Accepting the re-authentication overhead as a cost of simpler security.

### Lesson 3: Design for IdP Outages from Day One

If your authentication provider goes down, your application should degrade
 gracefully, not lock every user out. This applies whether you use a cloud
 IdP (Cognito, Entra ID) or self-host (Keycloak, Hydra).

**Apply this by:**

- Caching authentication decisions with a reasonable TTL. If the IdP is
  unreachable, serve the cached decision.
- Using session cookies (server-side sessions) instead of opaque tokens
  when you need offline access. The session is stored in your own database,
  which you can still serve if the IdP is down.
- Having a break-glass admin authentication path that uses a separate
  identity provider for emergency access.

### Lesson 4: Developer Experience Is a Security Property

Stripe's API key model succeeds because it is simple. When authentication is
 hard to use, developers find workarounds — and those workarounds are often
 insecure. A system that is easy to use correctly is more secure than a
 system with theoretically stronger cryptography that nobody implements
 properly.

**Apply this by:**

- Providing SDKs and client libraries that handle token refresh, retry, and
  reauthentication automatically.
- Using consistent authentication patterns across all your services. Do not
  use API keys in one service, OAuth 2.0 in another, and a custom header in
  a third.
- Writing clear, tested getting-started guides. The first authentication
  call a developer makes should succeed in under 5 minutes.

### Lesson 5: Invest in Observability for Authentication

Authentication is the first thing that breaks under load and the hardest
 thing to debug because failures look like permission errors rather than
 component failures. You cannot fix what you cannot see.

**Apply this by:**

- Tracking the full authentication flow as distributed traces. Every hop
  (client to gateway to IdP to user store to token issuer) should be
  visible in your tracing system.
- Setting up dashboards for authentication success and failure rates, token
  issuance latency, and IdP health. Alert on deviations beyond 3 standard
  deviations from the baseline.
- Logging authentication events (success, failure, reason) with structured
  logging. Include a correlation ID that a developer or support engineer can
  use to trace a single authentication request through the entire system.

## Summary

Authentication in production is a balancing act between security, usability,
 and operational complexity. The companies you studied here made different
 choices because their threat models and user bases are different. What they
 share is a deep understanding of the trade-offs and deliberate testing of
 their assumptions.

As you move on to the interview section, you will learn how to communicate
 these trade-offs clearly and design authentication systems that can survive
 the scale and failure modes of production environments.
