Caching questions appear in every FAANG system design interview. They test your understanding of trade-offs, your ability to reason about scale, and your experience with real-world failure modes. This playbook covers the question types, what interviewers look for, how to structure your answer, and the specific phrases and red flags that distinguish a strong response from a weak one.

## How This Topic Appears

Caching questions come in five categories. Recognizing the category tells you what the interviewer is testing and how to structure your answer.

**Direct caching questions:** "Design a cache." "How would you implement caching for a news feed?" "What caching strategy would you use for a social media application?" The caching component is the primary topic. Your answer should lead with caching architecture and trade-offs.

**System design questions with caching as a component:** "Design Twitter." "Design Netflix." "Design a URL shortener." Caching is one part of a larger system. You should mention caching as part of the data flow — but do not spend more than 2-3 minutes on caching unless the interviewer follows up.

**Performance debugging questions:** "A database is overloaded. How do you diagnose and fix it?" "A page used to load in 200ms but now loads in 5 seconds. What do you check?" These questions test whether you think of caching as a diagnostic tool and mitigation strategy.

**Trade-off questions:** "What are the downsides of caching?" "When would you NOT cache something?" "Compare write-through and write-back caching." These test depth of understanding — can you describe the failure modes, not just the benefits?

**Design debate questions:** "Should we use Redis or Memcached for this application?" "Should we cache at the CDN level or the application level?" These test your ability to make concrete technology choices based on requirements.

## What the Interviewer Is Testing

For direct and component caching questions, the interviewer evaluates four signals:

**Depth of understanding:** Do you know the difference between cache-aside, write-through, and write-back? Can you explain what happens internally on a cache miss? Have you thought about eviction policies and TTL? A surface-level answer ("cache is fast storage") scores at the L3 level. A deep answer that describes the hash table, LRU mechanics, and serialization costs scores at L5+.

**Trade-off awareness:** Do you know when NOT to use caching? Can you describe the failure modes of a cache stampede or cold start? Do you understand the consistency implications of caching in a distributed system? This is the single strongest signal for L5 and above — interviewers want to know you can make wise decisions, not just implement patterns.

**Scale reasoning:** Can you calculate the memory needed to cache a given workload? Do you know when a single Redis instance is insufficient and you need a cluster? Can you reason about the network bandwidth and latency implications of your design? This separates candidates who have operated systems at scale from those who have read about caching.

**Operational maturity:** Do you mention monitoring (hit ratio, latency percentiles)? Do you talk about deployment rollouts, cold starts, and circuit breakers? Experienced engineers naturally include operational considerations; junior engineers often skip them.

## Level-by-Level Expectations

Different interview levels require different depths of caching knowledge.

**L3 / Entry-level:** You should be able to define caching, explain why it is useful, and describe the cache-aside pattern at a high level. You should know the difference between local and distributed cache. A reasonable answer: "The application checks the cache first. If the data is there, return it. If not, query the database and store the result in the cache." You do not need to describe eviction algorithms, consistency models, or failure modes in detail.

**L4 / Mid-level:** You should explain two or three caching patterns (cache-aside, write-through, write-back) and when to use each. You should describe the eviction policies (LRU, LFU, TTL) and their trade-offs. You should mention cache stamps and how to mitigate them. You should be able to calculate approximate memory requirements for a given workload.

**L5 / Senior:** You should discuss caching as a distributed systems problem. Describe consistent hashing, replication, and cluster topology. Explain the operational considerations: monitoring hit ratio, deploying cache changes without downtime, handling cold starts. Compare specific technologies (Redis vs Memcached) with concrete differences. Your answer should include trade-offs: "For this workload, I would use write-behind caching because the data changes frequently and we can tolerate brief inconsistency, but we need low write latency."

**L6 / Staff:** You should design a multi-region caching topology with local reads and global invalidation. Discuss cache coherency protocols, cross-region replication lag, and eventual consistency guarantees. Reference real-world systems (Twitter's fan-out, Netflix's EVCache) as evidence for your design choices. Your answer should include specific monitoring and alerting thresholds. You should challenge the premise when appropriate: "Do you actually need caching here, or would optimizing the database queries achieve the same result with less complexity?"

**L7+ / Principal:** You should design a caching system that anticipates organizational needs: cost analysis, capacity planning models, team ownership models, phased rollout strategies. Discuss how the caching design interacts with the organization's microservice boundaries, data ownership, and compliance requirements. Your answer should include a risk assessment of the proposed system and a migration plan from the existing architecture.

## FAANG Framework

Use this 6-step framework when answering any caching-related interview question. It covers all the signals interviewers look for.

**Step 1: Clarify Requirements and Constraints**

Before proposing any solution, ask clarifying questions to narrow the scope:

- What data are we caching? (user profiles, feed items, search results, session data)
- What is the read-to-write ratio? (95% reads and 5% writes is ideal for caching)
- How fresh must the data be? (seconds, minutes, hours — determines TTL and invalidation strategy)
- What is the traffic pattern? (steady, bursty, seasonal — determines capacity planning)
- How many application instances need to share the cache? (single instance, multiple instances, multiple regions)

An example clarifying question: "Are we caching user profiles that change once a month and are read a million times a day, or stock prices that change every second?" The interviewer's answer tells you which caching strategy is appropriate.

**Step 2: Define the Data Access Pattern**

Map out the read and write paths for the specific data:

- How is the data identified? (user ID, content ID, composite key)
- What is the cache key format? (`user:{id}:profile`, `feed:{user_id}:page:{page}`)
- How expensive is a cache miss? (simple indexed query, complex join, external API call)
- What is the acceptable staleness window? (determines TTL)

State your assumptions explicitly: "I will assume user profiles are identified by a UUID, read 100x more often than written, and a staleness of 5 minutes is acceptable."

**Step 3: Choose the Cache Pattern**

Select the primary caching pattern based on the requirements:

- **Cache-aside:** General purpose, most common. Application manages cache population and invalidation.
- **Read-through:** Cache handles its own population on miss. Simpler application code but less control.
- **Write-through:** Every write goes through the cache to the database. Cache is always fresh, writes are slower.
- **Write-behind:** Writes go to the cache, asynchronously flushed to the database. Fast writes, risk of data loss.
- **Refresh-ahead:** Cache proactively refreshes entries before they expire. Reduces misses for popular keys.

Explain why you chose the pattern: "I will use cache-aside because the read-to-write ratio is 100:1, the data is user-specific so global invalidation is straightforward, and cache-aside gives us full control over when and how the cache is populated."

**Step 4: Design the Cache Topology**

Specify the cache infrastructure:

- **Cache type:** Local in-process (Caffeine), distributed (Redis cluster), or both (multi-layer).
- **Data distribution:** Single node, consistent hashing, or Redis cluster hash slots.
- **Replication:** Primary-replica for fault tolerance. Number of replicas.
- **Connection management:** Connection pool size, circuit breaker thresholds, timeouts.

Include sizing: "For 10 million users with 5KB per profile and a 5-minute TTL, assuming 20% of users are active per TTL window, we need 10 million x 5KB x 0.2 = 10GB of cache memory. Adding 30% overhead and one replica gives us 26GB. A 3-node Redis cluster with 10GB per node handles this."

**Step 5: Address Failure Modes and Mitigations**

Demonstrate you have thought about what goes wrong:

- What happens when the cache node crashes? (fail-open to database, circuit breaker)
- What happens during a cache stampede? (probabilistic early expiration, lock around miss)
- What happens during a cold start? (pre-warming, gradual traffic ramp-up)
- What happens when the cache returns stale data? (TTL bound, application tolerates staleness)

This step is the strongest signal for senior-level interviews. Do not skip it.

**Step 6: Define Monitoring and Operational Plan**

Show operational maturity:

- Key metrics: hit ratio (alert on >5% drop), latency (p50/p99/p999), memory usage (alert at 80%), eviction rate.
- Deployment plan: rolling deployment with node drain, warm-up period, canary testing.
- Debugging: `X-Cache` headers in responses, slow-query logging, cache key inspection tools.

This step distinguihes candidates who have operated production systems from those who have only designed them.

## Phrase Bank

These phrases help you communicate caching concepts concisely and professionally during an interview.

**"The cache is a performance optimization, not a source of truth."** Use this when setting context. It immediately communicates that you understand the fundamental relationship between cache and origin.

**"I prefer a short TTL and high hit ratio over a long TTL and perfect freshness."** Use this when discussing TTL strategy. It shows you understand the trade-off and prefer conservative defaults.

**"The staleness window is bounded by the TTL."** Use this when asked about cache consistency. It is a precise way to state that cache may be stale but will eventually be consistent.

**"I would use cache-aside with lock-based deduplication to prevent the thundering herd problem."** Use this when designing a cache for a read-heavy workload. It demonstrates you know both the pattern and its failure mode.

**"The working set determines the cache size, not the total dataset size."** Use this when discussing capacity planning. It shows you understand the difference between all possible data and data frequently accessed.

**"I would deploy with a rolling restart so the cache does not go completely cold."** Use this when discussing deployment strategy. It shows operational awareness.

**"The X-Cache response header is essential for debugging."** Use this when describing monitoring. It shows you have built cache-instrumented systems before.

**"Consistent hashing minimizes reshuffling when nodes are added or removed."** Use this when discussing cluster topology. It shows you understand data distribution mechanics.

**"Write-through guarantees the cache is always fresh but adds latency to every write."** Use this when comparing write strategies. It is a precise, balanced statement.

**"I would set an alert on the eviction rate because a sudden increase indicates the cache is too small for the workload."** Use this when discussing alerting. It shows proactive thinking.

**"Redis is single-threaded for command execution, so a single slow command can block all other operations."** Use this when comparing Redis vs Memcached. It shows deep knowledge of the technology.

**"The difference between p50 and p99 latency for the cache tells you whether there is contention or garbage collection affecting tail latency."** Use this when discussing performance. It shows you operate at the percentile level, not the average level.

**"I would cache at the CDN layer for globally distributed read-heavy content and at the application layer for personalized data."** Use this for multi-layer caching designs. It shows architectural judgment.

**"Probabilistic early expiration refreshes popular entries before they expire, reducing the chance of a stampede."** Use this when discussing cache freshening strategies. It shows you know advanced caching techniques.

**"The memory allocator — jemalloc for Redis, slab allocator for Memcached — significantly affects memory efficiency and fragmentation."** Use this in deep technical discussions. It demonstrates operating-system-level understanding.

## Red Flags

These mistakes weaken your interview response and may cause the interviewer to mark you down.

**Red flag 1: Treating the cache as a source of truth.** Saying "the cache always has the latest data" or "we can rely on the cache for correctness" signals inexperience. The cache is a copy, possibly stale, possibly missing.

**Red flag 2: Ignoring cache misses.** Describing only the happy path (cache hit) without discussing what happens on a cache miss. A complete caching design includes the miss path, the origin query, and the cache population.

**Red flag 3: No TTL.** Proposing a cache without specifying TTL values. Every cache entry must have a TTL. If you forget to mention TTL, the interviewer will ask "how long does data stay in the cache?" and your answer should already address this.

**Red flag 4: Caching everything.** Proposing to cache the entire database without considering which data benefits from caching. Smart engineers identify the top 5-10% of queries that account for 80% of load and cache those.

**Red flag 5: No eviction policy.** Not mentioning what happens when the cache runs out of memory. Every cache has a finite size. You must specify what gets evicted and why.

**Red flag 6: No monitoring.** Designing a caching system without mentioning how to measure its effectiveness. The interviewer will suspect you have not operated a cache in production.

**Red flag 7: Over-engineering.** Proposing a Redis cluster with 20 nodes for 1,000 users. Scale your solution to the requirements. A 2-node Redis cluster (primary + replica) handles 100,000 reads per second — that covers most applications.

**Red flag 8: Ignoring deployment.** Not mentioning how to deploy the cache without downtime or how to handle a cold start. These operational concerns are the difference between a design that works on paper and one that works in production.

## Practice Drills

These five prompts simulate the style and pacing of FAANG system design interviews. Set a timer and practice each one.

**Drill 1 (15 minutes):** Imagine you are designing the caching layer for a news website that displays trending articles to 10 million daily visitors. The articles change every 2-3 hours. Each article includes text, images, metadata, and related article links. Describe your caching strategy, including cache key format, TTL, eviction policy, and how you handle a breaking news event that requires immediate cache invalidation.

**Drill 2 (20 minutes):** An e-commerce platform's product page loads in 8 seconds. The page displays product details, reviews, inventory status, and related products. The database is under 30,000 queries per second and CPU is at 90%. Design a caching solution. Include specific metrics for evaluating success and a rollout plan that avoids downtime.

**Drill 3 (15 minutes):** Compare and contrast Redis and Memcached for a session caching use case. Your application has 500,000 concurrent users, each session is 2KB, and sessions expire after 30 minutes of inactivity. Which would you choose and why? Include specific technical differences that drive your decision.

**Drill 4 (20 minutes):** You are designing a real-time leaderboard for a mobile game with 1 million daily active users. The leaderboard updates every time a player completes a game (millions of updates per day) and is read every time a player opens the leaderboard screen (hundreds of thousands of reads per hour). Describe your caching strategy, including how you handle the high write frequency and the need for sorted ranking.

**Drill 5 (10 minutes):** A colleague proposes caching every API response with a TTL of 24 hours "for performance." Identify three reasons this is a bad idea, and propose a better approach.

### Scoring Rubric for Cache Questions

Understanding how interviewers score caching answers helps you calibrate your response.

**Poor (L3 reject):** Defines caching as "storing data in memory." Describes cache-aside pattern at a high level. Does not mention TTL, eviction, or failure modes. Cannot calculate memory requirements. Does not discuss operations or monitoring.

**Acceptable (L3 hire):** Defines caching with specific terminology. Describes cache-aside with TTL and LRU eviction. Mentions cache stampede as a failure mode. Can approximate memory needs. Mentions hit ratio as a monitoring metric.

**Good (L4-L5 hire):** Compares multiple caching patterns (cache-aside, write-through, write-back) with trade-offs. Discusses eviction policies (LRU, LFU, TTL) with justification. Describes cache stampede prevention (lock around miss, probabilistic early expiration). Calculates memory and throughput with specific numbers. Discusses deployment strategy and cold start mitigation.

**Strong (L5-L6 hire):** Designs multi-layer caching (local + distributed). Discusses data distribution (consistent hashing, hash slots). Compares Redis vs. Memcached with specific performance differences. Describes monitoring strategy with specific alert thresholds. Discusses multi-region topology with cross-region replication. References real-world implementations (Twitter fan-out, Netflix EVCache).

**Exceptional (L6-L7 hire):** Challenges the premise when appropriate. Discusses organizational impact (team ownership, microservice boundaries). Provides cost-benefit analysis. Describes migration strategy from no-cache to caching. Discusses compliance and security implications (PII in cache, encryption, compliance). Mentions advanced techniques (CRDTs for multi-region, Window TinyLFU, probabilistic early expiration).

### Using the Framework in Practice

The 6-step FAANG framework works for any caching question, but the depth at each step varies by question type.

**For direct caching questions ("Design a cache for Twitter timelines"):** Spend 60% of your time on steps 1-3 (clarify, define access pattern, choose pattern). Spend 20% on step 4 (topology). Spend 20% on steps 5-6 (failure modes, monitoring).

**For system design questions with caching as a component ("Design Twitter"):** Spend 10% of your time on caching. Mention caching as part of the data flow, specify the pattern and technology, and move on. Do not dive deep into cache internals unless the interviewer asks.

**For performance debugging questions ("A database is overloaded, what do you do?"):** Mention caching as a mitigation strategy, but start with diagnostics (slow queries, connection pool, CPU analysis). Caching is a solution, not a diagnosis.

**For trade-off questions ("Compare Redis and Memcached"):** Spend 70% on the comparison dimensions (throughput, data types, persistence, clustering, consistency). Spend 30% on when you would choose each, with specific use case examples.

**Follow-up: "How would you debug a sudden drop in cache hit ratio?"**

Structure your answer as a systematic investigation:

1. Check for a recent deployment: did the cache key format change? Did a new version of the application use different key prefixes? Roll back the deployment if the drop coincides with a deploy.
2. Check for TTL changes: did the TTL configuration get modified? A shorter TTL reduces the hit ratio proportionally.
3. Check for data changes: did the workload change? If new users or new content types appeared, previously cached data may no longer be representative of the current traffic.
4. Check the monitoring dashboard: look for patterns in the hit ratio by key prefix, by application instance, and by cache node. A drop limited to one key prefix points to a data-specific issue. A drop across all prefixes points to a system-wide issue.
5. Check the slow log: if the cache is slow, the application may time out on cache reads and fall through to the database, artificially inflating the miss count.
6. Check the cache keys themselves: use `redis-cli --scan --pattern` to sample current keys and verify they match the expected format.

**Follow-up: "How do you handle cache invalidation across microservices?"**

This is a distributed systems problem. Three approaches:

1. **Centralized invalidation service:** A dedicated service that receives invalidation requests from any microservice and broadcasts them to all cache nodes. The invalidation service uses a message queue (Kafka, RabbitMQ) for durability and replayability.

2. **Event-driven invalidation:** Each microservice publishes domain events when data changes (e.g., `UserUpdated`, `OrderPlaced`). A cache invalidation consumer subscribes to all relevant events and issues cache DELETE commands. Each microservice owns its cache keys and subscribes to events that affect those keys.

3. **TTL-based invalidation:** No explicit invalidation. Each microservice accepts a TTL-based staleness window. If a service's data changes, other services may see stale data for up to TTL seconds. This is the simplest approach and works when the staleness window is acceptable.

The choice depends on the consistency requirements. For strong consistency, use approach 1 or 2. For eventual consistency, use approach 3 and set TTLs appropriately.

**Follow-up: "How would you cache a database that is 500GB with only 50GB of cache memory?"**

You cannot cache the entire database. You must cache only the working set — the subset of data accessed frequently. Strategies:

1. **Identify the working set:** Analyze query logs to find the top 10% of keys that account for 90% of reads. Cache only those keys. Use LRU eviction to automatically retain the active working set.

2. **Cache by recency:** Data accessed in the last N minutes is more likely to be accessed again. A cache with LRU eviction naturally retains recently accessed data.

3. **Use a cache hierarchy:** Local in-process cache (10GB) for the hottest data, distributed cache (50GB) for warm data, and direct database access for cold data. The total cache is 60GB, which may cover 80-90% of read traffic if the access pattern is skewed.

4. **Consider SSD-backed cache:** Redis on Flash (Redis Enterprise) or Memcached with SSD tiering can cache more data than fits in RAM, at slightly higher latency. The trade-off is 10-100x slower than RAM but still 10-100x faster than the database.

**Follow-up: "How would you handle cache invalidation for a CDN?"**

CDN invalidation is fundamentally different from application cache invalidation because you cannot delete individual entries on every CDN edge node instantaneously. Strategies:

1. **Short TTL with versioned URLs:** Set a short CDN TTL (5 minutes) and use versioned URLs (`/images/cat.v2.jpg`, `/styles/main.v3.css`). When content changes, deploy it with a new version number. Old versions expire naturally via TTL.

2. **CDN purge API:** Use the CDN provider's purge API to invalidate specific URLs or URL patterns. Most CDNs support instant purging, but propagation to all edge nodes takes 10-60 seconds. Use this for urgent content changes.

3. **Surrogate keys:** Group related content under a surrogate key (also called a cache tag). When any content in the group changes, purge the entire surrogate key. This is more efficient than purging individual URLs.

4. **Origin shield:** A dedicated intermediate cache layer between the CDN edges and the origin server. The origin shield absorbs cold-start traffic from CDN edges, reducing load on the origin.

### Common Interview Mistakes (and How to Fix Them)

These are the most frequent mistakes candidates make in caching-related interviews, based on feedback from FAANG interviewers.

**Mistake 1: Starting with the solution before understanding the problem.** A candidate hears "caching" and immediately starts describing Redis cluster configuration before asking about the data access pattern, read-to-write ratio, and consistency requirements. Fix: always start with clarifying questions. The interviewer is evaluating your problem-solving process, not your ability to recite Redis documentation.

**Mistake 2: Proposing a single cache layer for all data.** A candidate describes one Redis cluster for all cached data — user profiles, feed items, search results, and session data — all with the same TTL and eviction policy. Fix: segment your cache design by data type. Each data type has different access patterns, freshness requirements, and value sizes. Describe separate cache configurations for each.

**Mistake 3: Ignoring the cost of cache misses.** A candidate describes the cache hit path in detail but glosses over what happens on a miss. Fix: for every cache design, describe the miss path: what triggers a miss, how the data is fetched from the origin, how the cache is populated, and how concurrent misses are handled.

**Mistake 4: Over-engineering for the problem scale.** A candidate proposes a 10-node Redis cluster with cross-region replication for a system with 10,000 users. Fix: scale your solution to the requirements. Start with a simple design (single Redis instance with replica) and explain how you would scale it as the system grows. The interviewer wants to see that you can match the solution to the requirements.

**Mistake 5: Not mentioning eviction.** A candidate describes a cache that stores all data forever without considering what happens when memory runs out. Fix: always specify the eviction policy (LRU, LFU, TTL) and justify why it is appropriate for the workload.

**Mistake 6: Treating the cache as a source of truth.** A candidate says "the cache always has the latest data" or "we can rely on the cache for correctness." Fix: the cache is a performance optimization, not a source of truth. The database is the source of truth. The cache may be stale, corrupted, or empty. Design for that.

**Mistake 7: No plan for cache failure.** A candidate describes a system that stops working if the cache goes down. Fix: always describe the fail-open behavior — if the cache is unreachable, the application reads from the database. The cache is a performance optimization, not a critical path dependency.

**Mistake 8: Inconsistent terminology.** A candidate uses "cache miss," "cache miss," and "not in cache" interchangeably, or confuses "TTL" with "eviction." Fix: use precise terminology. TTL is time-based expiration. Eviction is memory-pressure-based removal. Cache miss is a lookup that does not find the key. Cache stampede is a specific failure mode of concurrent misses.

### How to Practice Caching Questions

Caching is a muscle. You build it through deliberate practice, not passive reading.

**1. Whiteboard the data flow.** For every system you design, draw the data flow on a whiteboard (or a piece of paper). Show the request entering the system, hitting the cache, falling through to the database, and populating the cache. Trace through the hit path (green) and the miss path (red). This builds the mental model of the two paths.

**2. Calculate the break-even hit ratio.** For every system you design, calculate the hit ratio required for caching to be a net positive. Use the formula: break-even = cache_lookup_time / database_query_time. This forces you to think about the specific latency numbers for your system.

**3. Practice the failure modes.** For each cache design, ask "what fails?" and describe the symptom, detection, and recovery. Do this for cache stampede, cold start, node crash, network partition, and cache poisoning. This builds the muscle of thinking about failure, which is the strongest signal for senior-level interviews.

**4. Memorize the latency numbers.** Internalize the latency table from the concepts page: local cache (1µs), Redis (1ms), database query (10ms), external API (200ms). These numbers let you make quick calculations during an interview without reaching for a calculator.

**5. Practice the level-appropriate answer.** Record yourself answering the same question at different levels. For L3: define caching, describe cache-aside. For L5: compare cache-aside vs write-through, discuss eviction policies, describe failure modes. For L7: design multi-region topology, discuss cost analysis, describe organizational impact.

### Caching in System Design Interviews: Example Walkthrough

Here is a complete walkthrough of a caching answer for the question "Design Twitter's timeline caching."

**Clarify requirements:** "Before designing, I need to understand the data access pattern. Timelines are read-heavy — a user opens Twitter hundreds of times per day but tweets a few times per week. The read-to-write ratio is approximately 100:1. Each timeline consists of tweets from followed users, ordered reverse-chronologically. Tweets are immutable once posted — they do not change."

**Define the data access pattern:** "The cache key will be `timeline:{user_id}:page:{page_num}`. Each value is a list of tweet IDs (not the full tweet objects). The value size is approximately 1KB for 20 tweet IDs. The TTL will be 5 minutes, with explicit invalidation when a new tweet creates a new cache entry."

**Choose the cache pattern:** "I will use fan-out-on-write with cache-aside for reads. When a user tweets, the tweet ID is written to the timeline cache of every follower. This shifts cost from reads (unbounded) to writes (bounded by follower count). Timeline reads are O(1) — a single cache lookup."

**Design the cache topology:** "Twitter uses a Redis cluster with consistent hashing. Each user's timeline is stored as a Redis list. The cluster size is approximately 20-50 nodes to handle the memory requirements — storing 800 tweet IDs for 300 million users requires approximately 300M x 800 x 8 bytes = 1.9TB. With replication, this doubles to 3.8TB. A 50-node cluster at 16GB each provides 800GB, which is insufficient — Twitter uses a mix of Redis and custom in-memory stores to handle this scale."

**Address failure modes:** "Cache stampede is the primary risk. If a celebrity with 50 million followers tweets, the fan-out writes 50 million cache entries simultaneously. Twitter handles this by using an async queue for fan-out — the tweet is written to a few thousand timelines immediately (for online followers) and the rest are computed lazily on read. Cold starts are handled via persistent Redis RDB snapshots that are loaded on restart."

**Monitoring and operations:** "Twitter monitors the timeline cache hit ratio (target 99%+), the fan-out write rate (tweets per second x average followers), and the cache memory utilization. X-Cache headers in API responses enable debugging. Circuit breakers prevent the cache from being a single point of failure."
