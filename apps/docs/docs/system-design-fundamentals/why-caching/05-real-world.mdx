The best way to understand caching is to study how the largest systems in the world use it. Each company below faced a specific scaling challenge and built a caching solution tailored to their data access patterns. Their approaches reveal patterns you can apply in any system.

## Company Case Studies

**Scale:** 330 million monthly active users, 500 million tweets per day, 40 billion timeline reads per day.

**Problem:** Twitter's core operation is the timeline — a reverse-chronological list of tweets from followed users. The naive approach (query all tweets from followed users, join, sort, deduplicate, render) required joining across thousands of users' tweet streams for a single timeline request. At Twitter's scale, this approach would require tens of thousands of database servers.

**Solution:** Twitter implemented a fan-out-on-write cache. When a user tweets, Twitter loads the list of followers and writes the tweet to each follower's timeline cache. The cache stores a pre-computed list of tweet IDs for each user. When a user opens Twitter, the timeline read is a single cache lookup — O(1) instead of O(followees).

The timeline cache is a Redis-like in-memory store. Each user's timeline is a list of tweet IDs (up to 800 entries per user). The cache uses a mix of Redis and Twemproxy (a Redis proxy) for data distribution. Cache entries have a TTL of 24 hours but are invalidated immediately when a new tweet arrives.

**Result:** 99% of timeline reads are served from cache. The median timeline load takes under 5 milliseconds. Twitter reduced the number of database servers from thousands to hundreds.

**Key lesson:** Fan-out-on-write shifts computation from reads (unbounded, unpredictable) to writes (bounded, predictable). A celebrity with 50 million followers causes 50 million cache writes on a single tweet — but Twitter accepts this cost because reads outnumber writes 100:1.

**Scale:** 260 million subscribers, 1 billion hours of streaming per month, hundreds of microservices.

**Problem:** Netflix's recommendation engine requires computing personalized suggestions for every subscriber. The computation involves collaborative filtering, content-based filtering, and neural network inference — processing that takes 200-500 milliseconds per user. Doing this computation on every page load would require tens of thousands of CPU cores.

**Solution:** Netflix built EVCache, a multi-region Memcached-based caching tier. The recommendation engine runs on a schedule (typically every 4-6 hours) and pre-computes recommendations for every user. The results are stored in EVCache. When a user opens Netflix, the front-end service retrieves the pre-computed recommendations from the nearest EVCache node in under 10 milliseconds.

EVCache spans multiple AWS regions. Each region has its own Memcached cluster. Cross-region replication (asynchronous, rapid mode) keeps clusters in sync. The EVCache client library handles consistent hashing, compression, circuit breaking, and metric reporting.

**Result:** Recommendation load times dropped from 500ms to under 10ms — a 50x improvement. The recommendation computation runs on a batch schedule, spreading CPU cost over hours instead of concentrating it at peak traffic.

**Key lesson:** Batch compute + cache is a powerful pattern. Compute results in bulk when resources are available, cache the results, and serve from cache. This turns expensive per-request computation into cheap cache lookups.

**Scale:** Thousands of AWS customers using DynamoDB, with tables handling millions of requests per second.

**Problem:** DynamoDB provides single-digit-millisecond latency at any scale, but for latency-sensitive applications (ad serving, real-time gaming, live auctions), every millisecond matters. Customers wanted to reduce DynamoDB read latency without changing their application code.

**Solution:** Amazon built DAX (DynamoDB Accelerator), a write-through in-memory cache that implements the same API as DynamoDB. Applications replace the DynamoDB endpoint URL with the DAX cluster endpoint — no code changes required. DAX intercepts all `GetItem`, `Query`, and `Scan` calls, caching results in a Memcached-compatible cluster.

DAX is a write-through cache: every write goes to DAX first, which writes to DynamoDB synchronously and caches the result. This guarantees that the cache always reflects the latest write (unlike cache-aside, which has a staleness window). The trade-off: write latency increases by the cache write time (under 1ms).

**Result:** Read latency for cached items dropped from 5-15ms to under 1ms. Customers achieved 10-50x read throughput without increasing DynamoDB read capacity units, reducing costs by up to 90%.

**Key lesson:** A write-through cache that implements the same API as the origin eliminates the adoption barrier. Applications benefit from caching without code changes. This pattern works well when the origin supports a simple key-value API.

### Discord: Caching Guild Data

**Scale:** 150 million monthly active users, 19 million active servers (guilds), 500 million messages per day.

**Problem:** Discord's core data structure is the "guild" — a server with channels, members, roles, and permissions. Every message interaction requires checking the sender's permissions across multiple dimensions: can they read this channel, write to it, mention @everyone, attach files, and so on. These permission checks involve multiple database lookups. At Discord's scale, doing five database queries per message interaction would require a database fleet of hundreds of servers.

**Solution:** Discord uses a Redis cluster to cache guild state. When a user sends a message, the application loads the guild's entire permission structure from cache in a single lookup. The cached data includes member roles, channel overrides, permission bitfields, and user nickname mappings.

The cache key pattern is `guild:{guild_id}:{channel_id}:permissions:{user_id}`. On member join, role change, or permission update, the cache entry is invalidated. The TTL is set to 5 minutes, but invalidation events typically clear entries within 100 milliseconds.

Discord uses a Redis cluster with 5 shards, each with a replica. The cluster handles over 200,000 operations per second during peak hours. The cache hit ratio is consistently above 98%.

**Result:** Permission checks take under 2 milliseconds from cache versus 50-100 milliseconds from database queries. Discord runs its database on a modest 10-node cluster despite handling hundreds of millions of messages per day.

**Key lesson:** Caching complex authorization structures is one of the highest-value caching use cases. Authorization data changes infrequently but is checked on every user action. A well-cached authorization system can eliminate the need for a large database fleet.

### GitHub: Caching Git Data

**Scale:** 100 million repositories, 40 million developers, millions of Git operations per minute.

**Problem:** GitHub's core operation is serving Git data — repository contents, commit histories, branch lists, and diff comparisons. Git operations are read-heavy and repetitive: the same commit SHA, the same tree object, and the same blob are requested thousands of times per minute by different developers cloning, fetching, and browsing repositories.

**Solution:** GitHub uses a multi-layer caching architecture for Git object data. The first layer is an in-memory LRU cache (Caffeine in the JVM) within each application server that caches recently accessed Git objects. The second layer is a Redis cluster that stores Git object data with repository-scoped keys: `git:object:{repo_id}:{sha}`. The third layer is the Git object database itself (backed by traditional storage).

When a developer requests a Git object, the application checks the local cache first (sub-microsecond), then the Redis cache (sub-millisecond), and only then falls through to the Git object database (10-100ms). The local cache hit ratio is approximately 50% — many objects are requested multiple times within a single request or across requests served by the same instance. The Redis cache hit ratio is an additional 40%, for a combined hit ratio of 90%.

**Result:** Over 90% of Git object reads are served from cache. The Git object database handles only 10% of the read load, reducing the storage infrastructure needed for GitHub's massive dataset. The average Git object read latency is under 2ms.

**Key lesson:** Multi-layer caching with different characteristics at each layer absorbs load at the most cost-effective tier. Microsecond-level local caches handle the hottest data, millisecond-level distributed caches handle the medium-hot data, and the database handles the cold misses.

### Stack Overflow: In-Memory Cache for Q&A Content

**Scale:** 50 million monthly visitors, 20 million questions, 10,000 requests per second at peak.

**Problem:** Stack Overflow's pages are dynamic — they include user-specific data (votes, favorites, accepted answers) but the core content (questions, answers, comments) is the same for all users. Rendering a single question page requires joining across questions, answers, comments, user data, vote counts, and tag metadata. Running this query on every page load would require a massive database cluster.

**Solution:** Stack Overflow uses a Redis cache with a clever composite key structure. The core content of each question page (the question body, all answers, all comments) is rendered server-side into HTML fragments and cached with key `page:{question_id}:html`. The user-specific data (voted, favorited, accepted) is fetched separately and combined at render time. This means the expensive SQL query and HTML rendering happen once per question update, not once per page view.

The cache is invalidated by a sophisticated event system. When a new answer is posted, an answer-created event invalidates the question page cache for that specific question. The next visitor triggers a fresh render of the page with the new answer included.

Stack Overflow also uses Redis for distributed rate limiting, real-time tag counts, and the "hot questions" feed — a pre-computed list of trending questions that is refreshed every 30 seconds.

**Result:** Stack Overflow runs its database on a single Microsoft SQL Server instance (with failover) despite serving 50 million monthly visitors. The Redis cache absorbs the vast majority of read traffic. The database handles primarily writes (new questions, answers, comments) and cache misses.

**Key lesson:** Caching rendered HTML fragments rather than raw data reduces compute cost per request. A rendered question page that takes 200ms to generate from database queries takes under 5ms to serve from cache. The invalidation event system ensures the cache is updated within seconds of a content change.

## Open Source Cache Technologies

The open-source ecosystem provides battle-tested caching technologies for every use case. Choosing the right one depends on your performance requirements, data types, and operational capabilities.

### Redis

The dominant cache for distributed systems. In-memory key-value store with optional persistence, replication, clustering, Lua scripting, and rich data types (strings, hashes, lists, sets, sorted sets, bitmaps, HyperLogLog, streams). Single-threaded event loop with epoll for maximum performance. Used by Twitter, Discord, GitHub, Stack Overflow, and thousands of others.

**Best for:** General-purpose caching, session storage, rate limiting, real-time leaderboards, pub/sub messaging, and any workload that benefits from Redis's rich data types.

**Limitations:** Single-threaded for command execution (a single slow command blocks all others). Memory is expensive — data must fit in RAM. Persistence (RDB/AOF) adds complexity and can cause latency spikes during snapshotting.

**Minimum production configuration:** Redis primary with one replica, maxmemory-policy set to allkeys-lru, protected-mode enabled, requirepass set, and persistence configured (at minimum RDB snapshots every 5 minutes).

### Memcached

Simple, fast, multi-threaded in-memory cache. ASCII protocol, no persistence, no replication, no data types beyond key-value. Multi-threaded architecture gives it an advantage over Redis on multi-core machines for throughput (500,000+ GETs/sec per node). Used by Facebook, Netflix, Wikipedia, YouTube.

**Best for:** High-throughput, simple key-value caching where data loss is acceptable. Memcached excels at caching database query results, API responses, and rendered HTML fragments.

**Limitations:** No persistence, no replication, no data types beyond key-value, no built-in clustering. The slab allocator can cause memory waste if value sizes vary widely. Maximum value size is 1MB by default.

**When to choose Memcached over Redis:** Your workload is purely key-value (no lists, sets, sorted sets). You need the highest possible throughput per dollar. You can tolerate data loss on restart. You have multi-core machines that benefit from Memcached's multi-threaded architecture. You do not need persistence or replication.

### Varnish

HTTP accelerator (reverse proxy cache) that sits in front of web servers. Caches full HTTP responses. Supports complex caching rules via VCL (Varnish Configuration Language). Used by Wikipedia, The Guardian, and thousands of e-commerce sites.

**Best for:** Caching full HTTP responses — HTML pages, API responses, and static assets. Varnish excels when you need fine-grained control over caching behavior: different TTLs per URL pattern, cookie-based cache exclusion, and cache purging by URL pattern.

**Limitations:** Varnish is a reverse proxy — it must sit in front of your web server. It is not a general-purpose key-value store. VCL has a learning curve. Varnish does not support SSL/TLS natively (requires a separate TLS terminator like Hitch or Nginx).

### Nginx Cache

Built-in HTTP caching in the Nginx web server. Caches responses from upstream application servers. Supports cache purging via the `nginx-cache-purge` module. No separate infrastructure needed — the Nginx that serves HTTP requests also caches responses.

**Best for:** Simple HTTP response caching without adding a new service to the infrastructure. If you already use Nginx as a reverse proxy, enabling the cache is a configuration change, not a new deployment.

**Limitations:** Less flexible than Varnish. Cache invalidation is by URL only (no custom rules). Cache storage is disk-based (slower than in-memory caches). The cache is single-node — no built-in clustering or replication.

### Caffeine (JVM)

The highest-performance in-process caching library for Java. Provides near-optimal hit ratio via Window TinyLFU eviction policy. Used by Spring Framework, Cassandra, HBase, and most JVM-based services at Google and LinkedIn.

**Best for:** Java applications that need sub-microsecond local caching. Caffeine achieves near-optimal hit ratios by combining recency and frequency in its eviction policy — it tracks which entries are accessed frequently, not just recently.

**Configuration example:** `Caffeine.newBuilder().maximumSize(10_000).expireAfterWrite(5, TimeUnit.MINUTES).recordStats().build()`

### Guava Cache (JVM)

Google's in-process caching library for Java. Simpler than Caffeine, with fewer tuning options. Provides LRU eviction, TTL, and weak/soft references. A good choice for moderate-performance local caching needs.

## Industry Anti-Patterns in Production

These are real production incidents reported by engineering teams. Each demonstrates a specific caching failure mode.

**GitHub's cache key collision:** GitHub cached repository metadata with a key derived from the repository name. When a repository was renamed, the old key no longer matched the new name. The old data remained in the cache under the old key, consuming memory that could hold active data. GitHub's fix: use a stable repository ID (not the human-readable name) as the cache key component, and invalidate the key on rename.

**Twitter's timeline cache stampede:** Twitter's fan-out-on-write writes tweets to follower timelines at tweet time. When a celebrity with 100 million followers tweets, all 100 million timeline caches must be updated. Before implementing the async fan-out queue, the cache write load spiked so high that Redis nodes became CPU-bound. The fix: process fan-out writes asynchronously through a queue, prioritizing timelines of online followers and deferring offline followers.

**Reddit's cache thundering herd:** Reddit's front page is cached with a fixed TTL of 30 seconds. When the TTL expires for all front page keys simultaneously, every request to the front page triggers a database query to regenerate the cached HTML. Reddit's fix: stagger the TTL with a random jitter (+-5 seconds) and add a mutex that allows only one query per cache key at a time.

**Pinterest's oversized cache values:** Pinterest cached entire PIN objects (up to 100KB each) in Redis. The cache memory filled rapidly, causing high eviction rates and low hit ratios. Pinterest's fix: cache only the PIN ID and metadata (under 1KB), and store the full object in a separate blob store. The application fetches the blob store data on demand, which is acceptable because the full object is only needed when the user scrolls to that specific PIN.

**Etsy's no-TTL memory leak:** Etsy cached template fragments without TTLs, expecting the application to invalidate them explicitly on content changes. Over time, the cache accumulated fragments for every page variation ever generated. Memory usage grew until the Redis cluster ran out of memory. Etsy's fix: add a TTL of 24 hours to all cached fragments, ensuring unused fragments are eventually evicted.

## Cloud Provider Comparison

Each cloud provider offers a managed Redis service. The choice between them depends on your cloud provider strategy and specific requirements.

**AWS ElastiCache for Redis:** The most mature managed Redis service. Supports Redis 7+, clustering (up to 500 shards), encryption at rest and in transit, multi-AZ with automatic failover, and backup to S3. The `serverless` mode (introduced 2023) auto-scales based on traffic and eliminates the need to provision nodes. Use when you are on AWS and want the tightest integration with CloudWatch, IAM, and VPC.

**GCP Memorystore for Redis:** Supports Redis 7+, high availability with standard tier, no standalone clustering (use client-side consistent hashing or Redis Cluster with proxy). Integrates with Cloud Monitoring and Cloud IAM. Lower cost than ElastiCache for equivalent configurations. Use when you are on GCP.

**Azure Cache for Redis:** Supports Redis 7+, clustering, data persistence, geo-replication, and VNet injection. The Premium tier adds data persistence and clustering. Use when you are on Azure.

**Managed vs self-managed:** Managed Redis costs 20-30% more than self-managed but saves the operational cost of a Redis administrator. For teams with fewer than 50 microservices, managed Redis is almost always the right choice. For teams with dedicated infrastructure engineers and cost optimization mandates, self-managed Redis on EC2 instances can achieve 40-50% cost savings by using spot instances, custom AMIs, and optimized instance types.

## Lessons Learned

These patterns emerge from studying caching at scale across dozens of companies:

**Cache keys matter more than anything else.** A poorly designed cache key creates cache collisions (different data mapped to the same key) or cache miss storms (every request generates a unique key). Use a deterministic key format: `{entity_type}:{entity_id}:{sub_resource}`. Include version numbers when the schema changes. Never include timestamps, random numbers, or user-agent strings in cache keys.

**Monitor everything or do not cache at all.** Without hit ratio, latency, and eviction metrics, you are operating blind. Every caching-related outage starts with "We had no idea the hit ratio dropped to 30%." Use the `X-Cache` response header to expose cache status in every response. Log cache operations with timing and result (hit/miss/error).

**Plan for cold starts.** Every cache deployment is a potential cold start. Have a plan for absorbing the full database load during warm-up. Pre-warm the cache when possible. Use persistent storage (Redis RDB/AOF) to minimize warm-up time after restarts.

**TTL is your friend, not your enemy.** Short TTLs give you a safety net. They automatically heal stale data, recover from cache poisoning, and prevent memory leaks from keys that are written but never expire. Start with a short TTL (60 seconds) and increase it as you gain confidence in your invalidation logic.

**Never trust the cache for correctness.** The cache may be stale, corrupted, or empty. Every system component that reads from the cache must handle the case where the cache returns unexpected data. Deserialization errors, missing keys, and corrupted values should be handled gracefully. The database is the source of truth; the cache is a performance optimization.
