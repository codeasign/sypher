A startup launches a social feed product. Day one: five users, one engineer, a single PostgreSQL database on a modest instance. Everything is fast. The feed query — join across users, posts, likes, comments — takes 2 milliseconds. The engineer deploys with a smile.

Day thirty: ten thousand users. The feed query now touches hundreds of rows per request. The database is handling 5,000 reads per second. Response time creeps to 50 milliseconds. Still fine.

Day ninety: one hundred thousand users. The database is handling 50,000 reads per second. The same query that took 2 milliseconds now takes 400 milliseconds. Sometimes 800. The database CPU is pinned at 95%. Connection pools are exhausted. Requests queue up. The engineer gets paged at 2 AM — the database is crashing under the load. Users are tweeting complaints. The company is losing money.

This is the moment every growing system faces. You cannot throw more database hardware at the problem forever. The bottleneck is not the query — it is the sheer repetition of identical work. Every user refresh fires the same joins, the same aggregations, the same scans against the same data. The database computes the same answer over and over, millions of times a day, while most of those answers have not changed since the last request.

The engineer needs a way to serve the most frequently requested data without asking the database every single time.

## What Is Caching

Caching means storing a copy of expensive-to-compute data somewhere faster, so you do not have to recompute it on every request.

Think of a restaurant kitchen. The chef keeps a prep station within arm's reach — salt, oil, chopped vegetables, par-cooked proteins. Reaching to the pantry in the back takes thirty seconds. Reaching to the supplier takes a day. The prep station is a cache. It is small, it is fast, and it is stocked with what the chef needs right now. Refilling it requires walking to the pantry, but the chef only does that when the prep station runs low.

A cache in software works the same way. You trade a small amount of memory for a large reduction in latency and database load. The first time someone requests a piece of data, the system fetches it from the origin (the database, an API, a filesystem) and stores a copy in a faster storage layer. Subsequent requests for the same data skip the origin and read from the cache instead.

The result is transformative. Latency drops from hundreds of milliseconds to under one millisecond. Database load drops by an order of magnitude. The system becomes cheaper to operate because you need fewer database instances. The users get a snappy experience.

## Core Vocabulary

Mastering caching requires precise language. These are the terms you will use every time you design, debug, or discuss a cache.

**Cache hit** — A request finds its data in the cache. This is the fast path. Every cache hit means one database query you did not have to run. High cache hit ratio is the primary goal of any caching strategy.

**Cache miss** — A request does not find its data in the cache. The system must fall through to the origin, fetch the data, optionally populate the cache, and return the result. Cache misses are expensive because they combine the cost of checking the cache with the cost of the origin lookup.

**Cache ratio** — The percentage of requests that result in cache hits. A ratio of 95% means only 5 out of every 100 requests reach the database. Industry benchmarks: 80% is acceptable, 90% is good, 99% is excellent. Below 70%, the cache may be doing more harm than good because the overhead of cache lookups outweighs the savings.

**TTL (Time To Live)** — The maximum time a cached entry is considered valid. After the TTL expires, the entry is evicted or treated as stale. TTL is the primary mechanism for balancing freshness against performance. Short TTL (seconds) means fresh data but lower hit ratio. Long TTL (hours) means high hit ratio but risk of serving stale data.

**Cold cache** — A cache that has just started and contains no data. Every request produces a miss until the cache warms up. Cold starts are dangerous in production because they cause a sudden flood of traffic to the origin — a pattern known as cache stampede.

**Warm cache** — A cache that has been running long enough to accumulate frequently requested data. A warm cache delivers high hit ratios. The warm-up period depends on traffic patterns and TTL settings.

**Origin server** — The authoritative source of data behind the cache. This is the database, API, filesystem, or compute that the cache is protecting. The origin is always truth — the cache is a performance optimization, not a source of truth.

**Cache stampede** — A cascading failure where a large number of cache entries expire simultaneously, causing every request to miss and hammer the origin simultaneously. The origin becomes overloaded, requests slow down, timeouts cascade, and the system can collapse.

**Stale data** — Data in the cache that has been modified at the origin but not yet updated in the cache. Staleness is a consequence of caching. You control it with TTL and invalidation strategies. Some applications tolerate seconds of staleness; others tolerate none.

**Cache hierarchy** — Multiple layers of cache arranged by speed and size. L1 cache is smallest and fastest (CPU cache, nanoseconds). L2 is larger and slower (local memory, microseconds). L3 is largest and slowest (Redis, milliseconds). Requests cascade through the hierarchy: check L1, miss, check L2, miss, check L3, miss, finally hit the origin.

**Local cache** — A cache stored in the memory of the application process itself. Extremely fast (microseconds) but limited by the process memory and lost when the process restarts. Each instance of a distributed application has its own local cache, so data can become inconsistent between instances.

**Distributed cache** — A cache running as a separate service, shared across all application instances. Redis and Memcached are the most common. Slower than local cache (milliseconds) but can store much more data, survives process restarts, and provides a single consistent view.

**Cache-aside** — The most common caching pattern. The application checks the cache first. On a hit, return the data. On a miss, load from the origin, store in the cache, return. The cache and database are independent — the application orchestrates both.

**Write-through** — A caching pattern where every write goes through the cache to the database. The cache is updated synchronously with the write. This keeps the cache always fresh but adds latency to every write.

**Write-back** — A caching pattern where writes go to the cache first and are asynchronously flushed to the database. This makes writes fast but risks data loss if the cache crashes before the flush completes.

**Read-through** — A caching pattern where the cache itself handles populating on a miss. The application requests data from the cache by key; if the key is missing, the cache automatically loads it from the database, caches it, and returns it. The application code is simpler because it does not need to implement the miss path, but the cache must know how to query the database.

**Refresh-ahead** — A proactive caching pattern where the cache refreshes a key before it expires, based on predicted demand. If a key is accessed frequently and its TTL is nearing expiration, the cache preemptively re-fetches the data from the origin. This reduces the chance of a cache miss for popular keys and smooths out load on the origin.

**Cache warming** — The process of populating a cache with data before it is needed, typically after a restart or deployment. Pre-warming avoids the cold-start problem where every initial request produces a miss and overloads the origin. Warming can be done by replaying recent traffic, running batch queries for popular keys, or loading from a persistent snapshot.

**Cache invalidation** — The act of removing or updating a cache entry when the underlying data changes. Invalidation is the hardest problem in caching because it requires the application to know when data has changed and to propagate that knowledge to all cache layers. Strategies include TTL-based expiration, explicit deletion on write, and event-driven invalidation via a message queue.

**Edge cache** — A cache deployed at the network edge, close to end users. CDNs are the most common form of edge cache. They cache HTTP responses at dozens or hundreds of geographically distributed locations, reducing latency for users far from the origin server.

**Slab allocator** — A memory management strategy used by Memcached to reduce fragmentation. Memory is divided into slabs of fixed-size chunks. Values are stored in the smallest slab class that fits. This eliminates external fragmentation but can cause internal fragmentation (unused space within a chunk) and slab imbalance (one size class runs out while another has free chunks).

**Read repair** — A consistency mechanism for caches where, on reading a stale entry, the system detects the staleness and triggers an asynchronous update. This is common in eventually consistent distributed systems where stale reads are detected by comparing version numbers or timestamps.

**Probabilistic early expiration** — A technique that prevents cache stampedes by randomly refreshing entries before they expire. Instead of all entries expiring at their exact TTL, each entry has a probability of being refreshed early that increases as it approaches expiration. This spreads the refresh load across time rather than concentrating it at the TTL boundary.

The diagram shows the fundamental topology. A client request arrives at the application server. The server checks the cache. The cache sits between the application and the database, absorbing the vast majority of read traffic. With a 90% hit ratio, the database sees only 10% of the original read load — a tenfold reduction.

## How It Actually Works

Here is the step-by-step walkthrough of a cache request, from the moment the client sends a request to the moment the response arrives. The scenario: a social feed application where a user requests their timeline.

**Scenario: Cache Hit**

  `}

The numbered steps for a cache hit:

1. The client sends an HTTP GET request to `/feed/uid=42`.
2. The application server receives the request and identifies the resource needed — user 42's feed page 1.
3. The application generates a cache key. A well-designed cache key is deterministic, collision-resistant, and includes all parameters that affect the value. Here: `"feed:42:page=1"`.
4. The application calls the cache client library, which computes a hash of the key and looks up the hash table in the cache server's memory. This is an O(1) operation — constant time regardless of how many keys are stored.
5. The key exists. The cache checks the TTL: was this entry stored less than 300 seconds ago? Yes — 45 seconds have passed. The entry is valid.
6. The cache returns the serialized value (typically JSON, Protocol Buffers, or a compressed binary format). The application deserializes it into the in-memory data structure it needs.
7. The application sets the response header `X-Cache: HIT` and returns the response. Setting this header is a best practice — it lets you debug caching behavior in production without digging through logs.
8. The client renders the feed. Total elapsed time: 3 milliseconds. No database query was performed.

**Scenario: Cache Miss**

<AsciiDiagram
  id="why-caching/cache-miss-flow"

  `}
  caption="Sequence of events when a cache miss occurs"
/>

The numbered steps for a cache miss:

1. The client sends a GET request for a different user's feed — user 99.
2. The application generates the cache key: `"feed:99:page=1"`.
3. The cache lookup finds no matching key. This user has never requested their feed, or the previous entry expired.
4. Before querying the database, the application acquires a distributed lock for this cache key. This is a crucial safeguard: without it, ten concurrent requests for the same missing key would each fire a database query, creating a mini stampede. The lock ensures only one request queries the database; the others either wait or are served once the cache is populated.
5. The application constructs and executes the SQL query. This is a multi-table join across posts, likes, and comments, ordered by timestamp, limited to 20 results.
6. The database executes the query. It scans indexes, joins tables, sorts results. On a cold cache, the query takes 150 milliseconds because the database is under load from all the other cache misses.
7. The application serializes the result set. JSON is the most common format because it is universally readable and compresses well.
8. The application stores the serialized result in the cache with a TTL of 300 seconds (5 minutes). The Redis command is `SET feed:99:page=1 <json> EX 300`.
9. The application releases the lock so other requests for the same key can now find the cached value.
10. The response is returned with `X-Cache: MISS`. The total elapsed time is 158 milliseconds — dominated by the database query.
11. The client renders the feed. On the next request from this user, the cache will return the data in under 5 milliseconds.

The TTL lifecycle determines how long a cache entry lives. When a cache miss occurs and the data is fetched from the origin, the entry is stored with a TTL — say 300 seconds. For the next 300 seconds, any request for that key returns a hit. After 300 seconds, the cache automatically evicts the entry. The next request suffers a miss, refetches from the origin, and resets the clock.

This cycle repeats indefinitely. The TTL is the mechanism that ensures the cache eventually reflects changes at the origin — even if the application never explicitly invalidates an entry.

**Concurrent Request Handling**

<AsciiDiagram
  id="why-caching/concurrent-requests"

    Request 1 ──── Check Cache ── Miss ── Lock ── Query DB ── Populate ── Unlock ── Response
                                              │

                                              │
    Request 3 ──── Check Cache ── Miss ── Wait for Lock ───────────── Check Cache ── Hit ── Response

    Request 1 ──── Check Cache ── Miss ──────────────────── Query DB ────────── Response
    Request 2 ──── Check Cache ── Miss ──────────────────── Query DB ────────── Response
    Request 3 ──── Check Cache ── Miss ──────────────────── Query DB ────────── Response
    Result: 3x database queries for the same data in the same millisecond
  `}
  caption="Lock-based deduplication prevents multiple database queries for the same cache miss"
/>

When multiple requests arrive simultaneously for the same cache key that has expired or does not exist, the naive approach causes each request to query the database independently. This is the thundering herd problem. The solution is lock-based deduplication: the first request to miss acquires a lock, queries the database, and populates the cache. Subsequent requests either block on the lock and then find the cache populated, or they are served a stale-but-acceptable value while the refresh happens in the background.

This pattern is called "lock around the miss" or "mutex-based cache refresh." It is essential for any cache that serves more than a few hundred requests per second.

## Variants and Flavors

Caching is not a single mechanism. Different cache placements and implementations serve different purposes. Understanding which variant to use in which situation is the difference between a system that performs well and one that introduces more problems than it solves.

A local cache lives inside the application process. It is a data structure — a hash map, an LRU cache, or a library like Guava Cache — that stores recently accessed data in the application's own heap memory.

**When to use:** You have a single application instance, or you can tolerate slight inconsistency between instances. The data changes rarely. The working set fits in the application's memory budget. The latency savings from avoiding even a fast network call to Redis matter — for example, sub-100-microsecond lookups for high-frequency computations.

**When not to use:** You have many application instances and need a consistent view of data. Your working set exceeds the memory available in each process. You need to invalidate data globally — invalidating local caches across hundreds of instances is a complex distributed systems problem.

**Example implementations:** Python's `functools.lru_cache`, Java's Caffeine and Guava Cache, Go's `hashicorp/golang-lru`, and ad-hoc `dict`/`map` wrappers.

A distributed cache runs as an independent service that all application instances connect to over the network. It stores data in memory across one or more machines. Redis and Memcached are the dominant implementations.

**When to use:** You have multiple application instances that need to share cached data. Your working set exceeds the memory of a single process. You need centralized control over eviction and TTL. You want your cache to survive application restarts.

**When not to use:** Every network call adds latency — if your performance budget requires microsecond-level lookups, a distributed cache is too slow. You need a simpler setup and can live with per-instance caches.

**Key considerations:** Network latency, serialization costs, connection pool management, and the operational complexity of running a cache cluster. A single Redis instance can handle hundreds of thousands of operations per second, but network round trips add 0.5-5 milliseconds per operation.

Client-side caching stores data on the device making the request — typically the browser. HTTP cache headers (`Cache-Control`, `ETag`, `Last-Modified`) tell the browser how long it can reuse a response without asking the server. Service workers and the Cache API provide programmatic control.

**When to use:** You are serving static assets (images, CSS, JavaScript) or API responses that change infrequently. You want to eliminate network round trips entirely for returning users.

**When not to use:** You need fine-grained control over cache invalidation (browser caches are hard to invalidate). You are serving sensitive data that should not be stored on client devices.

A CDN (Content Delivery Network) cache is a geographically distributed network of cache servers. When a user requests content, the CDN routes the request to the nearest edge node. If that node has the content cached, it serves it directly — eliminating the long cross-continental round trip to the origin server.

**When to use:** You serve content to a global audience. Your content is static or changes infrequently — images, videos, CSS, JavaScript, and even rendered HTML pages. You want to reduce latency for users far from your origin servers.

**When not to use:** You serve highly personalized content that changes per user and per request. You need real-time invalidation — CDN caches take seconds to minutes to propagate invalidations globally.

### Latency Numbers Every Engineer Should Know

When designing a caching system, internalize these latency numbers. They are the basis for every caching decision.

| Operation | Latency | How Many Cache Lookups Fit |
|-----------|---------|---------------------------|
| L1 cache reference (CPU) | 0.5 ns | — |
| Branch mispredict | 5 ns | — |
| L2 cache reference | 7 ns | — |
| Mutex lock/unlock | 25 ns | — |
| Main memory reference | 100 ns | — |
| Compress 1KB with LZ4 | 1,000 ns (1µs) | — |
| Send 1KB over network | 10,000 ns (10µs) | — |
| Local cache lookup (in-process) | 50 ns - 1µs | 1,000,000 per DB query |
| Redis/Memcached (same datacenter) | 500 µs - 5 ms | 100-1,000 per DB query |
| Disk seek | 10 ms | 50 per DB query |
| Database query (simple indexed) | 5-50 ms | — |
| Database query (complex join) | 20-200 ms | — |
| External API call | 200-1000 ms | — |

A local cache lookup is 10,000 to 100,000 times faster than a database query. This gap is why caching is the single most impactful performance optimization available. No amount of query optimization, indexing, or hardware upgrades can match the latency reduction of replacing a 50ms database query with a 1µs cache lookup.

The gap widens under load. A database at 90% CPU has queuing delays that push p99 latency from 50ms to 500ms or more. A cache at 90% memory utilization continues serving lookups at sub-millisecond latency because memory access time does not degrade with utilization. This is the compound benefit of caching: not only does the cache serve data faster, but the database itself gets faster because it has fewer queries to process.

### The Cost of a Cache Miss

A cache miss is more expensive than a direct database query because you pay the overhead of the cache check plus the overhead of the miss handling. With a distributed cache, the round-trip for a cache check is typically 1-5ms. If the miss then requires a 50ms database query, the total is 51-55ms — slightly worse than the direct 50ms query. The system is only faster if the hit ratio is high enough that the average cost (including miss overhead) is lower than the direct query cost.

The break-even hit ratio depends on the relative costs:

- Cache lookup time: C
- Database query time: D
- Hit ratio: H

Average response time = H x C + (1 - H) x (C + D) = C + (1 - H) x D

For the cache to be a net positive, the average response time must be less than the direct database query time D:

C + (1 - H) x D < D
C < H x D
H > C / D

With C = 2ms and D = 50ms, the break-even hit ratio is 2 / 50 = 4%. As long as the hit ratio exceeds 4%, caching improves average response time. In practice, well-designed caches achieve hit ratios of 80-99%, so the break-even analysis is rarely the limiting factor — but it is important to know that there IS a break-even point, and a poorly designed cache can make things worse.

### Cache in the Context of the Full Stack

Caching happens at every layer of the computing stack, from the CPU to the CDN. Understanding the full hierarchy helps you place your cache at the right level.

**Layer 1 — CPU caches (L1, L2, L3):** Hardware-managed caches that store recently accessed memory. L1 is 32KB per core, L2 is 256KB per core, L3 is 8-30MB shared. Access times: 0.5ns to 10ns. The programmer has limited control — CPU caches are automatic.

**Layer 2 — Operating system cache (page cache / buffer cache):** The OS caches recently accessed disk pages in available RAM. When you read a file from disk, the OS keeps it in memory in case you need it again. This is why the first read of a file is slow and subsequent reads are fast. The page cache is automatic but can be tuned via `fadvise` and `mlock`.

**Layer 3 — Application-level in-process cache:** A data structure inside your application process that stores results of expensive computations or database queries. Examples: Python's `functools.lru_cache`, Java's Caffeine. The programmer has full control.

**Layer 4 — Distributed cache:** A separate service (Redis, Memcached) shared across application instances. Survives application restarts. The primary cache layer for multi-instance deployments.

**Layer 5 — CDN / HTTP cache:** A geographically distributed cache that stores full HTTP responses. Operates at the network edge. Examples: CloudFront, Cloudflare, Varnish, Nginx.

**Layer 6 — Client-side cache:** Cache on the user's device. Browser cache, service worker cache, mobile app cache. The most distributed cache layer — data is cached on millions of devices, but you cannot control or invalidate it easily.

Each layer adds latency reduction but also adds complexity. A well-architected system uses the appropriate layers: CDN for global static content, distributed Redis for application data that needs consistency, local caches for hot data that benefits from microsecond access.

## Mental Model Check

These questions test your understanding of the concepts covered in this page. Answer each one before reading the explanation.

**Question 1: A cache has 1,000,000 requests per day. It serves 950,000 from cache and 50,000 from the database. What is the cache hit ratio, and is it good?**

The hit ratio is 950,000 / 1,000,000 = 95%. This is excellent. Only 5% of requests reach the database, meaning the database handles 50,000 requests instead of 1,000,000 — a 20x reduction in load. In most production systems, a hit ratio above 90% is considered good, and above 99% is outstanding.

**Question 2: You deploy a new cache cluster and immediately see a spike in database CPU to 100%. What is happening?**

This is a cold cache problem. The cache starts empty — every request produces a miss and falls through to the database. Until the cache warms up (accumulates entries for the most popular keys), the database bears the full read load. Mitigation strategies include pre-warming the cache with popular data before directing traffic to it, rolling out the cache gradually (canary deployment), or keeping a secondary cache layer active during the warm-up period.

**Question 3: A cache entry has a TTL of 300 seconds. The data changes at the origin after 60 seconds. For how long will clients receive stale data?**

Clients receive stale data for 240 seconds (300 seconds TTL minus 60 seconds elapsed). This is the staleness window. To reduce it, you can either lower the TTL (which reduces hit ratio) or implement explicit invalidation — when data changes at the origin, send a command to delete the cache entry. Explicit invalidation gives you the best of both worlds: low TTL for freshness but with the ability to update immediately when changes occur.

**Question 4: You have 10 application servers each with a local cache. A user's profile is updated on server 1. Server 2 still returns the old profile. Why?**

Each local cache is independent. The update on server 1 populated only server 1's local cache. Server 2's local cache still holds the old value (or nothing at all). This is the fundamental limitation of local caches in a distributed system: they cannot synchronize without an external coordination mechanism. Solutions include using a distributed cache instead, implementing a pub/sub invalidation channel, or accepting the staleness window defined by the local TTL.

**Question 5: At 2:00:00 PM, a popular cache key expires. At 2:00:00.001 PM, 500 concurrent requests arrive for that key. What happens without protection, and how do you fix it?**

Without protection, all 500 requests see a cache miss and simultaneously query the database. This is a cache stampede. The database receives 500 identical queries in the same millisecond, causing a spike that may overwhelm it. The fix is to implement lock-based deduplication (only one request queries the database; the rest wait and then read from cache) or proactive refresh (refresh the cache entry before it expires, using techniques like "early recomputation" where a background job refreshes entries at 70% of their TTL).

**Question 6: An application mixes local caches on 10 application servers with a shared Redis cache. Cache hits on local cache take 5µs. Cache hits on Redis take 2ms. Cache misses hit the database at 50ms. The local cache has a 40% hit ratio, and Redis has a 90% hit ratio (among requests that miss the local cache). What is the average read latency?**

The local cache handles 40% of requests at 5µs. Of the remaining 60%, Redis handles 90% at 2ms. The remaining 10% of that 60% (6% of total) miss and hit the database at 50ms. Average latency = (0.40 x 0.005ms) + (0.60 x 0.90 x 2ms) + (0.60 x 0.10 x 50ms) = 0.002 + 1.08 + 3.0 = 4.082ms. This is a two-layer cache hierarchy. The local cache absorbs the fastest tier, the distributed cache absorbs the middle tier, and the database only sees 6% of requests.

**Question 7: A product manager asks the team to "cache everything to make the site faster." How do you respond?**

Caching everything is not a strategy — it is a recipe for wasted memory, complexity, and stale data. The correct approach is to identify which data benefits from caching: data that is expensive to compute (complex queries, API calls), requested frequently (high read-to-write ratio), and can tolerate some staleness. For each candidate, estimate the expected hit ratio based on access patterns. Only cache data that meets all three criteria. Data that changes every request (real-time status updates) or is already cheap to compute (simple key lookups with good indexes) should not be cached.

**Question 8: A security audit reveals that the cache may contain sensitive user data (PII). What do you do?**

Three options depending on the severity: (1) Encrypt cache values at the application layer before storing. The cache stores only encrypted blobs and cannot read them. (2) Set very short TTLs (60 seconds) so sensitive data is not stored for long. (3) Do not cache sensitive data at all — mark the data type as uncacheable in the application and add an explicit check that skips the cache for these fields. Option 3 is safest for PII. Option 1 is best when you need the performance benefit but must comply with data privacy regulations like GDPR or HIPAA.

**Question 9: A team implements a write-through cache. Write latency increases from 5ms to 12ms. Reads improve from 50ms to 2ms. The read-to-write ratio is 50:1. Is this a good trade-off?**

Average latency per operation = (50/51) x 2ms + (1/51) x 12ms = 1.96ms + 0.24ms = 2.2ms. The original average was (50/51) x 50ms + (1/51) x 5ms = 49.02ms + 0.098ms = 49.1ms. The improvement is from 49.1ms to 2.2ms — a 22x reduction. The 7ms increase in write latency is negligible because writes are so rare (1 out of every 51 operations). This is an excellent trade-off.

**Question 10: A cache cluster has 3 nodes using consistent hashing. One node crashes. What fraction of cached keys become unavailable?**

With consistent hashing and 3 nodes, each node handles approximately one-third of the key space. When one node crashes, the keys it held (approximately 33.3%) are lost. The remaining nodes continue serving their keys. The hit ratio drops from 95% to 95% x 0.667 = 63.4% (assuming uniform key distribution). The database load increases accordingly. Recovery involves the surviving nodes taking over the lost hash slots and gradually repopulating as requests arrive.
