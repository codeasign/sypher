Caching starts simple and becomes complex as scale grows. A single-process hash map works for a hobby project. A global multi-region cache cluster with consistent hashing, replication, and circuit breakers is infrastructure for a billion-user platform. This page walks through the architectural evolution and the patterns that emerge at each scale.

## Naive Implementation

The simplest possible cache is an in-memory hash map inside the application server. Every web framework provides one: Python's `cachetools`, Java's `HashMap`, Go's `sync.Map`, or even a plain `dict`.

**How it works:** On the first request for a key, the application computes the value and stores it in a hash map. On subsequent requests, it checks the hash map first. If the value exists, return it. If not, compute and store.

**Where it breaks at 1,000 users:** The in-memory cache consumes application heap memory. A single server with 1GB of heap can hold roughly 100,000 average-sized cache entries before running out of memory. At 1,000 users each fetching multi-kilobyte responses, memory runs out within minutes. The application spends more CPU time on garbage collection than on serving requests.

**Where it breaks at 100,000 users:** A single application server cannot handle 100,000 concurrent users. You add more servers behind a load balancer. But each server has its own private in-memory cache. User 42's profile is cached on server A. User 43's request lands on server B — cache miss. The effective hit ratio drops from 70% (single server) to 30% (five servers), because the same data is cached independently on each server and no server sees enough traffic to warm its cache for the full dataset.

**Where it breaks at 10,000,000 users:** At this scale, the cache must be shared, distributed, fault-tolerant, and monitored. In-memory hash maps do not survive server restarts. They cannot coordinate invalidation. They offer no visibility into hit ratio or memory usage. The system needs a purpose-built distributed cache.

## Production-Grade Architecture

A production-grade cache architecture decouples the cache from the application, distributes data across multiple nodes, replicates for fault tolerance, and provides operational visibility.

**Sharding (Data Distribution):** The Redis cluster splits the key space into 16,384 hash slots. Each shard (node) owns a subset of slots. When a key is hashed, the hash slot determines which shard holds it. The cluster uses consistent hashing: when adding or removing a shard, only a fraction of keys (roughly 1/N) need to be rehashed to new nodes.

**Replication:** Each shard has one primary and one or more replicas. The primary handles all writes. Replicas replicate asynchronously. If the primary fails, the sentinel or cluster coordinator promotes a replica to primary. Replicas can also serve read traffic, increasing read throughput, but at the cost of potentially serving stale data (replication lag is typically under 10ms on a local network).

**Connection Pooling:** A connection pooler (Envoy, HAProxy, or a Redis-specific proxy like Twemproxy) manages a pool of persistent TCP connections to each Redis shard. This reduces the overhead of connection establishment and maintains a predictable number of connections to the cache.

**Circuit Breaking:** Each application server uses a circuit breaker for cache operations. If cache read errors exceed a threshold (e.g., 50% error rate over 10 seconds), the circuit opens, and the application reads directly from the database until the circuit half-opens and tests whether the cache has recovered.

## Integration Patterns

How the cache integrates with the rest of the system depends on the data source and the access pattern.

This is the cache-aside pattern, the most common integration for read-heavy database workloads. The application manages cache population and invalidation.

The critical detail: to invalidate, you DELETE the cache entry rather than UPDATE it. Deleting forces the next read to cache-miss and fetch the latest value from the database. Updating risks writing a value that does not match what the database would return (because another write may have happened between your update and the read).

When a write occurs, the application publishes an invalidation event to a message queue instead of directly deleting the cache entry. A separate consumer process listens for invalidation events and deletes the corresponding cache entries.

This pattern is useful when:

- The write path and the read path are maintained by different teams. The write team publishes events; the read team manages cache invalidation.
- You need to invalidate multiple cache layers (local + distributed + CDN) on a single write.
- You want to replay invalidation events if the cache goes down and comes back up.

The CDN sits in front of the application server. It caches full HTTP responses (HTML pages, API responses with `Cache-Control` headers). When a request arrives at the CDN edge, the CDN checks its cache. On a hit, it serves the response immediately — no request reaches the application server. On a miss, the CDN forwards the request to the origin (the application server), which may itself have a Redis cache and a database behind it.

This creates a multi-layer cache hierarchy: CDN edge → CDN regional → application cache → database. Each layer absorbs a portion of traffic. A 90% hit ratio at the CDN layer means only 10% of requests reach the application server. A 90% hit ratio at the application cache means only 10% of those 10% reach the database — effectively 1% of original traffic reaches the database.

External API calls are among the most expensive operations in any system — both in latency (200-1000ms) and cost (per-call pricing). Caching external API responses reduces both.

The key difference from database caching: the cache TTL should reflect the API's rate limit window and data freshness requirements. A Twitter API call with a 15-minute rate limit window should be cached for at least 15 minutes to avoid hitting the rate limit during peak traffic. A Stripe balance check should be cached for seconds, not minutes, because users expect near-real-time balance updates.

## Multi-Region and Global Scale

When your users span continents, a single-region cache cluster adds 100-300 milliseconds of latency for users far from the cluster. Multi-region caching solves this by placing cache clusters in each geographic region.

The "local read, global write" pattern is the standard approach for multi-region caching. Application instances read from the nearest cache cluster (1-5ms latency). Writes go to the local primary database and are replicated asynchronously to other regions. The cache in the writing region is invalidated immediately. Caches in other regions invalidate when they receive the replication event — typically within seconds.

The challenge is that a user traveling from Asia to Europe may see stale data until the cache in Europe invalidates. Two strategies mitigate this:

**Global cache with local read-through:** A single global Redis cluster with read-only replicas in each region. Writes go to the global primary. Reads go to the nearest replica. Replication lag is typically under 10ms in well-connected regions.

**Cache invalidation bus:** A cross-region message bus (Kafka with MirrorMaker, or a dedicated invalidation service) propagates invalidation events to all regions. When a user's profile is updated in US-East, an invalidation event for `profile:42` is published and consumed by all regional cache invalidators within seconds.

### Global Cache Topology Trade-offs

Designing a global cache topology requires balancing three competing goals: read latency, write latency, and consistency.

**Single global cluster with regional replicas:** One Redis cluster in a primary region with read-only replicas in other regions. Writes go to the primary (high latency for remote writers). Reads go to the nearest replica (low latency for all readers). Replication lag is 10-100ms across regions. This is the simplest multi-region topology and works well when writes are concentrated in one region.

**Active-active clusters:** Each region has its own primary Redis cluster. Writes go to the local cluster (low latency). Cross-region replication synchronizes the clusters asynchronously. This provides low write latency everywhere but introduces conflict resolution challenges — if a user's profile is updated simultaneously in US-East and EU-West, which update wins? Solutions include last-writer-wins (LWW) with timestamps, or conflict-free replicated data types (CRDTs).

**Regional cache-aside with global invalidation:** Each region has its own cache cluster that is populated from the local database read replica. A global invalidation service propagates invalidation events. This is the most flexible topology — each region's cache is independent and can be configured differently. The trade-off is that invalidation events may arrive later than the replication to the local database replica, causing a brief window where the cache has stale data and the database has the fresh data.

The choice depends on your write distribution. If 90% of writes come from US-East, the single global cluster with replicas is the simplest and most consistent. If writes are evenly distributed across regions, active-active clusters with CRDTs or LWW provide better write latency at the cost of consistency guarantees.

### Regional Capacity Planning

Each region's cache cluster must be sized for that region's traffic. The total traffic is distributed across regions proportionally to the user population. If US-East has 40% of users, EU-West has 30%, and AP-Southeast has 30%, the cache clusters should be sized proportionally.

But the working set per region may be different. US-East users may follow accounts that are primarily US-based, creating a different cache key distribution than AP-Southeast users. The cache in each region should be sized for the region's unique working set, which may be larger or smaller than the proportional share.

A conservative approach: size each region's cache for the full global working set. This provides headroom for traffic shifts (a marketing campaign that drives traffic from one region) and avoids the complexity of per-region capacity planning. The additional memory cost is typically under 20% of the total cache infrastructure cost.

### Cache Key Design Best Practices

The cache key is the foundation of every cache operation. A poorly designed key causes collisions, misses, and operational headaches.

**Deterministic keys:** The same data must always produce the same key. Never include timestamps, random numbers, session IDs, or user-agent strings in the key — doing so guarantees every request generates a unique key, producing a 0% hit ratio. This is the single most common caching bug in production.

**Namespace prefix:** All keys for a given data type should share a prefix. This groups related keys in monitoring tools, enables bulk operations, and prevents collisions between different data types. Convention: `{entity_type}:{entity_id}:{sub_resource}`. Examples: `user:42:profile`, `feed:42:page:1`, `product:8675309:inventory`.

**Version prefix:** Include a version number for schema migrations. When you change the serialization format or the data structure, increment the version. Old entries with the old version are automatically orphaned — the application never looks for the new data under the old key. Example: `v2:user:42:profile`. Without versioning, a schema change causes deserialization errors on every cached entry until the TTL expires (potentially hours or days).

**Fixed-length identifiers:** Use fixed-length identifiers (UUIDs, integers) rather than variable-length strings like email addresses or user names. Fixed-length keys hash uniformly across hash slots. Variable-length keys can create distribution skew: most users have short names, so the hash space is concentrated rather than distributed.

**Key size optimization:** A key of 200 bytes plus a value of 5KB uses 5.2KB total. A key of 20 bytes uses 5.02KB total — a 3.5% savings. For 10 million entries, that is 1.8GB. Optimize key size when memory is a primary concern, but never sacrifice debuggability for a few bytes. A readable key that a human can inspect in `redis-cli` is worth the extra memory.

**Hot key detection and mitigation:** A hot key receives traffic far above the average. Examples: a celebrity's profile during a controversy, a trending product during a sale, a breaking news article. The cache node holding the hot key becomes CPU-bound while other nodes are idle. Mitigation: (1) Replicate the hot key to multiple nodes by adding a shard number to the key (`trending:0`, `trending:1`, through `trending:N`) and reading from a random shard. (2) Cache the hot key at the application layer (local in-process cache) to drastically reduce distributed cache requests. (3) Use Redis RESP3 client-side caching, where the application subscribes to key invalidations and maintains its own local copy.

**Avoid pattern scans in production:** The `KEYS *` command in Redis scans every key and blocks the event loop for seconds on a large cache. Never use it in production. Use `SCAN` for iterative key scanning, or maintain a separate index of cache keys in a Redis set.

### Cache as Cache vs Cache as Database

A critical architectural decision is whether to treat the cache as a performance optimization (read-only, disposable) or as a system of record (persistent, authoritative). This decision affects every other design choice.

**Cache as cache (recommended for most use cases):** The database is the source of truth. The cache is a disposable copy. If the cache node crashes, the system continues — every request falls through to the database. The cache can be rebuilt from database data. This simplifies operations: you can restart the cache cluster without data loss procedures, and you can change cache configurations without migration plans.

**Cache as database (Redis as primary store):** Some systems use Redis as the primary data store, especially for ephemeral data like session state, rate limit counters, and real-time leaderboards. The database, if one exists, is a backup. This approach requires: (1) Redis persistence (RDB snapshots or AOF logs) to survive restarts. (2) Redis replication for fault tolerance. (3) Backup procedures identical to a database. (4) Capacity planning that accounts for data growth, not just working set size.

The two approaches have different operational profiles:

| Aspect | Cache as Optimization | Cache as Database |
|--------|---------------------|-------------------|
| Data loss tolerance | High (rebuild from DB) | Low (data loss is permanent) |
| Persistence | Optional (improves warm-up) | Required |
| Backup | Not needed | Required |
| Replication | Nice to have | Required |
| Scaling | Scale to working set | Scale to total data |
| Cache miss handling | Fall through to DB | Critical failure (data is only in cache) |

Most systems should use the first approach. Only use cache as database for data that is explicitly ephemeral — session tokens that expire in 30 minutes, rate limit counters that reset hourly, real-time leaderboards that exist only for a contest period.

## Capacity Planning

Capacity planning for a cache requires answering four questions: how many requests per second, how much memory per entry, what hit ratio, and how many cache nodes.

**Step 1: Estimate Reads Per Second**

Assume 10 million daily active users (DAU). Each user makes 10 cacheable requests per day. That is 100 million requests per day. Peak traffic is 3x the average (social apps have pronounced peaks around lunch and evening). Average RPS: 100,000,000 / 86,400 = 1,157. Peak RPS: 3,471.

**Step 2: Estimate Memory Per Entry**

Assume each cache entry is 5KB (serialized JSON for a user profile or feed response). With a working set of 10 million unique keys (1 key per user), the naive memory is 10,000,000 x 5KB = 50GB. But only a fraction of users are active at any time. The active working set — keys accessed at least once per TTL window — might be 1 million keys. The memory required for the active working set is 1,000,000 x 5KB = 5GB. With jemalloc overhead of 20-30%, plan for 6.5GB. With a replication factor of 2 (one replica), plan for 13GB total across the cluster.

**Step 3: Estimate Network Bandwidth**

Each cache GET returns 5KB. At 3,471 peak RPS, the incoming bandwidth is negligible (tiny GET commands). The outgoing bandwidth is 3,471 x 5KB = 17MB/s = 136 Mbps. That is well within a single 1Gbps network interface.

**Step 4: Determine Node Count**

A single Redis instance handles 100,000 GETs/sec. You need 3,471 / 100,000 = 0.03 instances for throughput — one instance is more than enough. But you need multiple instances for memory and fault tolerance. With 13GB total memory and a recommended per-instance max of 25GB (to keep RDB snapshots fast), you need one shard. With one replica per shard, you need 2 instances total. For production, add a third node for the sentinel quorum.

**Result:** 3-6 cache nodes (3 Redis instances in a cluster, each with 8-16GB RAM) running on moderate instances (AWS r6g.large or equivalent). Total monthly cost: $150-300. Without caching, you would need 5-10 database read replicas at $500-1,000 each.

## Deployment and Operations

### Zero-Downtime Cache Deploy

Updating cache configuration or software version without causing a cache miss storm requires a rolling deploy strategy:

1. Mark the first node as offline in the connection pooler. No new requests are routed to it.
2. Wait for the node to drain its existing connections (30-60 seconds for Redis with moderate connection lifetimes).
3. Restart the node with the new configuration or software version.
4. The node re-joins the cluster. Its cache is cold — it has no data. But because the cluster uses consistent hashing with replication, the replica can serve requests while the primary warms up, or the other shards cover the hash slots until the node is fully synced.
5. Repeat for each node.

For Redis cluster, the cluster resharding mechanism handles slot migration automatically. The total warm-up time is the time to repopulate one shard's worth of keys — roughly 1/N of the total warm-up time.

### Monitoring

The four signals to monitor for a cache:

**Hit ratio:** The single most important metric. A sudden drop from 95% to 70% indicates a problem — perhaps a code change changed the cache key format, or a deployment cleared the cache. Alert on a drop of more than 5% over 5 minutes.

Track hit ratio at two granularities: (1) Aggregate — the overall ratio across all keys. (2) Per-pattern — the ratio for specific key patterns (`user:*`, `feed:*`). Aggregate hides problems — a drop in `feed:*` ratio from 95% to 10% may be invisible if `user:*` at 99% dominates the aggregate. Most monitoring tools support per-pattern metrics via key prefix tagging.

**Latency:** Monitor p50, p99, and p999 latency for both GET and SET operations. A latency spike indicates network congestion, resource contention, or a slow command blocking the event loop (Redis is single-threaded — a slow `KEYS` or `SORT` command blocks all other operations).

Set different thresholds for different operations. Cache SET operations are typically 10-20% slower than GET operations because SET involves memory allocation, LRU list update, and potential eviction. A p99 SET latency of 10ms is acceptable; a p99 GET latency of 10ms indicates a problem.

**Memory usage:** Monitor used_memory vs maxmemory. If used_memory approaches maxmemory, eviction increases. If the eviction rate is high, the cache is too small for the working set.

Track the fragmentation ratio: `used_memory_rss / used_memory` from Redis INFO output. A ratio above 1.5 indicates significant memory fragmentation, typically caused by storing values of widely varying sizes or by an allocation pattern that confuses jemalloc. Restarting the Redis instance resets fragmentation but causes a cold start — the better fix is to use more uniform value sizes.

**Evictions and expirations:** A high eviction rate (more than 1% of stored keys per hour) indicates the cache is too small. A high expiration rate (keys expiring via TTL) is normal but should be tracked — a sudden spike in expirations means many keys had the same TTL and will expire simultaneously, potentially causing a stampede.

Evictions are the canary in the coal mine. A low eviction rate with high hit ratio means the cache is well-sized. An increasing eviction rate with stable hit ratio means the working set is growing — you need more memory. An increasing eviction rate with decreasing hit ratio means the cache is undersized and actively hurting performance by evicting valuable entries.

**Commands to monitor per second:** Redis INFO provides `total_commands_processed`, which gives the command rate. A sudden drop in command rate indicates a client-side issue (clients cannot connect) or a network issue. A sudden increase accompanied by latency degradation indicates the cache is overloaded.

**Connected clients:** Redis is single-threaded, so each connected client consumes a file descriptor and a small amount of memory for the output buffer. An application with a connection leak accumulates thousands of clients until Redis refuses new connections. Monitor `connected_clients` and alert on sustained increases. The maxclients setting (default 10,000) prevents resource exhaustion but causes connection errors for the application — set it to a value that triggers an alert before being reached.

### Instrumenting the Application

The cache server metrics are valuable, but application-side instrumentation tells you what the user experiences.

**Add X-Cache headers:** Every response should include `X-Cache: HIT` or `X-Cache: MISS`. For multi-layer caches, use `X-Cache: HIT(local), HIT(redis)` or `X-Cache: MISS(local), HIT(redis)`. These headers make debugging straightforward — curl any endpoint and see exactly which cache layer served the response.

**Log cache latency:** Log the time spent in each cache operation, broken down by operation type (GET, SET, DEL) and by key pattern. Use structured logging so you can aggregate by pattern in your log analytics tool. This is essential for identifying slow cache operations.

**Track deserialization failures:** A deserialization error causes the operation to be treated as a cache miss even though the data exists. This inflates the effective miss ratio and increases database load. Log every deserialization failure with the cache key and the error details. Alert on sustained deserialization failure rates above 1%.

**Connection pool metrics:** Expose the cache connection pool's active, idle, and pending (waiting for a connection) counts. If the pending count is consistently above zero, the connection pool is too small. If the idle count is consistently high, the pool is too large and wastes resources.

### Debugging

When something goes wrong with a cache, these diagnostic steps help:

**Check the cache key format.** The most common production bug is a cache key that changes between requests. If the key includes a timestamp, a random number, or a user-agent string, every request produces a unique key and the hit ratio drops to 0%. Look for dynamic components in the key. The fastest way to check: instrument the application to log the cache key for every operation and verify that repeated requests for the same data produce the same key.

**Check TTL consistency.** If some entries are cached for 60 seconds and others for 3600 seconds, the effective hit ratio varies unpredictably. Standardize TTL per data type. Use a TTL registry — a configuration file or database table that defines the TTL for each data type — so all developers use the same values.

**Check serialization errors.** If a value cannot be deserialized (schema mismatch, corrupted data), the application treats every read as a miss. Log deserialization errors and alert on them. Schema changes are the most common cause: you added a field to the serialized object, deployed the schema change, and now the cache contains values in the old format that the new schema cannot deserialize. The fix: use a version field in the serialized data, or use a schema evolution tool like Avro or Protocol Buffers that supports backward compatibility.

**Check connection pool exhaustion.** If the application runs out of connections to Redis, cache operations block waiting for a connection. Monitor the pool's active vs idle connection count. Connection leaks are often caused by not releasing connections back to the pool after a timeout or error — ensure the application always closes or returns connections in a `finally` block or equivalent.

**Check for slow commands blocking Redis.** Redis is single-threaded. A slow command like `KEYS *`, `SMEMBERS` on a large set, or `SORT` on a large list blocks all other operations for the duration. Use Redis's `SLOWLOG` to find commands that took longer than the slowlog threshold (default 10ms). Replace blocking commands with non-blocking alternatives: `SCAN` instead of `KEYS`, `SSCAN` instead of `SMEMBERS`.

**Check for large value sizes.** A single large value (10MB+) causes multiple problems: it consumes memory disproportionally, it takes longer to serialize and deserialize, and it blocks the network output buffer during transmission. Use the `DEBUG OBJECT key` command in Redis to check the serialized length of individual keys. Set a maximum value size policy and enforce it in the application.

**Use the cache debugging toolbox:** Keep these commands ready for production debugging:

- `redis-cli INFO stats` — hit ratio, commands processed, evictions, expired keys
- `redis-cli SLOWLOG GET 10` — last 10 slow commands
- `redis-cli MEMORY STATS` — memory breakdown by type
- `redis-cli CLIENT LIST` — connected clients with their buffers
- `redis-cli MONITOR` — real-time command stream (use carefully in production)
- `redis-cli --bigkeys` — find keys consuming the most memory
- `redis-cli --hotkeys` — find keys accessed most frequently
