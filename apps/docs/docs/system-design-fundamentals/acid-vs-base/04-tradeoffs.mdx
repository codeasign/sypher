# ACID vs BASE — Trade-offs & Failure Modes

The architecture page showed how ACID and BASE coexist. This page maps the hard
choices: when to use each model, what breaks under stress, and how to avoid
common mistakes.

---

## Core Tension

The fundamental trade-off between ACID and BASE is captured by a single
sentence:

**ACID guarantees safety at the cost of availability and performance under
partition. BASE guarantees availability but admits stale reads and conflicting
writes.**

### ACID — Safety First

When a network partition splits an ACID cluster, the partition that does not
contain the primary node cannot process writes. The system chooses consistency
over availability. This is correct for financial systems — better to reject a
transfer than to process it twice.

The cost is invisible until traffic peaks. A single-node ACID database running
at 80% capacity processes 4,000 writes per second with 3ms latency. At 95%
capacity, latency spikes to 50ms as lock contention increases. At 100%,
writes are rejected.

### BASE — Speed First

When a network partition splits a BASE cluster, every partition continues to
accept writes. The system chooses availability over consistency. This is
correct for social media — better to show a stale like count than to show an
error page.

The cost is invisible in the happy path. A BASE system with three replicas in
sync looks strongly consistent 99.9% of the time. But during a partition,
different clients see different states. The system converges when the partition
heals, but temporary contradictions are visible to users.

## Decision Matrix

Use the following decision points to determine whether a data domain should use
ACID or BASE. Each row represents a spectrum — the more your requirements lean
left, the stronger the case for ACID.

| Decision Point | Lean ACID | Lean BASE |
|---|---|---|
| Write Volume | Under 5,000 writes/sec/node | Over 10,000 writes/sec/node |
| Read Latency | Under 5ms required | 5-50ms acceptable |
| Data Loss Tolerance | Zero tolerance | Seconds of data loss acceptable |
| Conflict Handling | Conflicts must be prevented | Conflicts can be resolved |
| Stale Read Tolerance | No stale reads acceptable | Stale reads for seconds OK |
| Cross-Entity Constraints | Foreign keys, cascades required | No cross-key constraints |
| Operation Atomicity | Multi-row operations must be atomic | Single-row operations sufficient |
| Strong Consistency Footguns | BASE requires application-level complexity for correctness | ACID requires manual sharding or vertical scaling for performance |

**Rule of thumb:** If the data domain involves money, contracts, legal
compliance, or safety, use ACID. If the data domain involves user-generated
content, metadata, logs, or any data where "eventually correct" is acceptable,
use BASE.

## Failure Modes

Both systems have characteristic failure modes. Understanding these is
essential for operating them at scale.

### Stale Reads in BASE Systems

A stale read occurs when a client reads from a replica that has not yet
received the latest write. In a three-node Cassandra cluster with consistency
level ONE, a stale read returns data that is up to several seconds old.

**Impact:** Users see outdated information. In a social feed, this means an old
post appears in the feed after the user has seen it. In an e-commerce catalog,
it means showing a product as "in stock" when the last unit was just sold.

**Mitigation:** Raise read consistency to QUORUM. This guarantees that the read
overlaps with at least one replica that received the write. The cost is higher
read latency (must wait for two of three replicas).

### Write Conflicts — Tombstone Resurrection in Cassandra

Cassandra uses tombstones to mark deleted data. A tombstone is a special write
that signals deletion. Tombstones have a time-to-live (default: 10 days).
During this window, a deleted row can be resurrected if a stale replica's
compaction runs and a read repair arrives from a node that still holds the
data.

**Impact:** A user deletes their account. Two weeks later, the account appears
again because a tombstone expired on one replica and a read repair from another
replica restored the data.

**Mitigation:** Do not rely on tombstone expiry for data lifecycle management.
Use explicit compaction strategies (`LeveledCompactionStrategy`) and monitor
tombstone counts. Set `gc_grace_seconds` appropriately (default: 864,000
seconds / 10 days).

### Gossip Partition — Nodes Cannot Converge

If the network splits in a way that prevents gossip from flowing between two
groups of nodes, each group maintains its own view of cluster state. Read
repair and anti-entropy do not operate across the partition. The two groups
diverge indefinitely.

**Impact:** The system operates as two independent clusters. When the partition
heals, merging the diverged state requires manual intervention. This is a rare
but catastrophic failure.

**Mitigation:** Use a gossip failure detection threshold that is appropriate
for your network latency. Monitor gossip convergence metrics. In a multi-data
center setup, ensure seeds are spread across all data centers.

### Hinted Handoff Overflow

If a node is down for an extended period, the coordinator nodes accumulate
hints. If the write rate is high and the node is down for hours, hints can
consume gigabytes of storage. When the node comes back, replaying millions of
hints causes a write storm that can overwhelm the recovering node.

**Impact:** The recovering node falls behind again, creating a death spiral.
The hinted handoff backlog never drains.

**Mitigation:** Reduce hinted handoff duration (default 3 hours). Use
anti-entropy repair instead of relying on handoff for long outages. Monitor
hint counts and alert when they exceed a threshold (for example, 10,000 per
node).

### Read Repair Storms After Node Recovery

When a node recovers after a long outage, every read to its data range
triggers a read repair because the node's Merkle tree diverges from the
healthy nodes. If the recovery happens during peak traffic, the read repair
load compounds with normal traffic.

**Impact:** Read latency spikes across the cluster as every read triggers an
extra write to the recovering node.

**Mitigation:** Run a full anti-entropy repair (`nodetool repair`) on the
recovering node during low-traffic hours before routing traffic to it. Rate
limit read repairs during recovery.

## Anti-Patterns

### Anti-Pattern 1: Using ACID for Everything

Teams with a relational database background often default to ACID for every
data domain. This works until scale reveals the bottleneck.

**Signs:** The database primary is at 90% CPU. Connection pool is exhausted.
Write latency spikes during peak hours. Queries that were fast at 100K rows
are slow at 100M rows.

**Fix:** Separate hot-path data (sessions, logs, analytics, content) into
BASE-compatible stores. Keep ACID only for data that requires it.

### Anti-Pattern 2: Using BASE for Financial Transactions

The opposite mistake: adopting a trendy BASE database for domains that require
ACID guarantees.

**Signs:** Duplicate charges, negative inventory counts, lost payments, ledger
imbalances. The engineering team builds compensating logic for every
operation, recreating ACID guarantees with application code.

**Fix:** Use ACID for financial data. The complexity of implementing ACID at
the application layer exceeds the operational complexity of running an ACID
database. Only consider BASE for financial data if you have a team of
distributed-systems engineers.

### Anti-Pattern 3: Mixing ACID and BASE Without Understanding Edge Cases

A common pattern: write to PostgreSQL (ACID) and asynchronously replicate to
Cassandra (BASE). This works until a write succeeds in PostgreSQL but the
replication fails silently.

**Example:**

1. Order service writes an order to PostgreSQL. Success.
2. Order service publishes a `OrderPlaced` event to Kafka.
3. Kafka consumer is down. The event is lost.
4. The order exists in PostgreSQL but not in Cassandra.
5. Customer support checks the order in the customer-facing system (backed by
   Cassandra) and says "no such order."

**Fix:** Implement the outbox pattern. Write the event to the same ACID
database as the order, in a separate `outbox` table. A separate process reads
the outbox table and publishes events, retrying on failure. This guarantees
at-least-once delivery.

### Anti-Pattern 4: Ignoring Conflict Resolution Strategy

Many teams adopt a BASE database without planning for conflicts. They assume
conflicts are rare and can be handled manually. At scale, conflicts are not
rare — they are a natural consequence of accepting writes at any node.

**Fix:** Before deploying a BASE system, define:
- Which conflict resolution strategy to use (LWW, application merge, CRDT)
- How to detect conflicts (version vectors, timestamps, hashes)
- What to do when a conflict cannot be resolved automatically
- How to alert operators when conflicts accumulate

## When NOT to Use

### When NOT to Use ACID

1. **High-volume append-only logs:** Writing millions of log entries per second
   through an ACID database is expensive and unnecessary. Use Kafka or S3.

2. **Session stores:** Session data is ephemeral. ACID durability guarantees
   are wasted when a session expires in 30 minutes anyway. Use Redis or
   Memcached.

3. **Real-time counters:** Like counts, view counts, and follower counts can
   tolerate temporary inconsistency. ACID would limit update frequency. Use
   Redis or Cassandra with lightweight transactions.

4. **Content delivery / CDN caches:** CDN nodes are inherently eventually
   consistent. ACID on the origin does not help when the edge is stale. Use
   cache invalidation with BASE semantics.

5. **Time-series data:** Stock tickers, sensor readings, and metrics are
   append-only with no cross-row constraints. ACID adds overhead with no
   benefit. Use InfluxDB, TimescaleDB, or S3 with Parquet.

### When NOT to Use BASE

1. **Financial ledgers (General Ledger):** Every entry must balance. A
   temporarily inconsistent ledger can produce incorrect financial reports,
   regulatory violations, or undetected fraud. Must use ACID.

2. **Inventory counts (without compensatory logic):** Selling an item that
   does not exist erodes customer trust. If you use BASE for inventory, you
   must implement compensatory logic: notify the customer, offer a refund, and
   track oversell ratios.

3. **Medical records:** A stale read of a medication list can cause a fatal
   drug interaction. Medical systems must use strong consistency for clinical
   data.

4. **Atomic operations across multiple entities:** Transferring money from
   account A to account B requires atomic debit and credit. BASE cannot
   guarantee this without application-level coordination.

5. **Session-based authentication tokens:** While session *data* can be BASE,
   authentication decisions (is this token valid?) must be strongly consistent
   to prevent replay attacks or unauthorized access.

## ACID Costs That Are Often Overlooked

When teams default to ACID for every data domain, they miss these hidden costs:

### Connection Pool Exhaustion

Every ACID transaction holds a connection to the database while the transaction
is open. If a transaction involves a slow network hop or a user's browser
timeout, the connection is held for seconds. With a pool of 200 connections
and transactions taking 200ms, the database handles 1,000 concurrent
transactions. At 2,000 transactions per second, the pool is exhausted.

**Cost:** 5-20ms queuing delay while waiting for a connection from the pool.
Under peak load, this becomes 100ms+ or transaction rejection.

**Mitigation:** Keep transactions short. Use `SET statement_timeout` in
PostgreSQL to abort transactions that exceed 30 seconds. Use a separate pool
for long-running administrative queries.

### Bloat and Vacuum in MVCC

PostgreSQL's MVCC creates a new row version for every `UPDATE`. Old versions
accumulate until VACUUM reclaims them. At high update rates (10,000+ updates
per second on a single table), VACUUM cannot keep up. The table grows
unboundedly. Index scans slow down because they must skip dead tuples.

**Cost:** 2x-10x storage overhead compared to the active dataset. Index scan
performance degrades by 50% or more when 50% of rows are dead tuples.

**Mitigation:** Use `UPDATE` frequency limits. Batch updates where possible.
Schedule aggressive VACUUM during low-traffic windows. Consider partitioning
high-update tables by time so old partitions can be dropped rather than
vacuumed.

### The Serialization Tax

Serializable isolation requires tracking read-write conflicts. In PostgreSQL,
the SSI (Serializable Snapshot Isolation) implementation tracks which rows and
predicates each transaction has touched. Two transactions that read the same
range of rows and write to different rows within that range can still conflict
if the read predicates overlap.

**Cost:** 10-20% throughput reduction compared to Read Committed. In extreme
cases (long-running transactions scanning large tables), throughput drops 50%.

**Mitigation:** Only use Serializable when the data domain demands it. For most
applications, Read Committed with careful application-level retry logic is
sufficient.

## BASE Costs That Are Often Overlooked

BASE systems have their own hidden costs that teams discover after deployment.

### The Compaction Tax

Cassandra's LSM tree structure requires background compaction. Compaction reads
old SSTables, merges them, and writes new SSTables. This consumes CPU (for
sorting and merging) and I/O (for reading and writing large files). On a
heavily loaded node, compaction can consume 50% of available I/O bandwidth.

**Cost:** Write throughput drops by 30-50% during compaction on the same node.
If compaction falls behind, read latency increases because queries must check
more SSTables.

**Mitigation:** Use Leveled Compaction for read-heavy workloads (less I/O per
compaction). Use Size-Tiered Compaction for write-heavy workloads (less
frequent compaction). Set `compaction_throughput_mb_per_sec` to limit I/O
usage during business hours.

### The Tombstone Tax

Every delete in Cassandra creates a tombstone. Tombstones survive for
`gc_grace_seconds` (default 10 days). If a partition accumulates many
tombstones (for example, a user who deletes and recreates items frequently),
queries on that partition must scan through all the tombstones to find live
data.

**Cost:** A query on a partition with 100,000 tombstones can take 5-10 seconds
while Cassandra skips the tombstones. This blocks the coordinator thread and
increases latency for other queries.

**Mitigation:** Set `gc_grace_seconds` appropriately. For data that is
time-partitioned, use TTL-based expiry instead of explicit deletes. Avoid
frequent delete-recreate patterns — use upsert semantics instead.

### The Gossip Tax

Gossip protocol overhead grows with cluster size. With 100 nodes, each node
exchanges state with 3 random peers per second. That is 300 gossip messages
per second across the cluster — negligible. With 1,000 nodes, it is 3,000
messages per second — still manageable. But the gossip state at each node is
proportional to the number of peers.

**Cost:** Each gossip message carries state for all known nodes. With 500
nodes, a gossip message is approximately 50KB. With 1,000 nodes, it is 100KB.
Multiply by 3 messages per second per node = 300KB/s per node in gossip
traffic. This is small on a 10Gbps network but competes with data traffic.

**Mitigation:** Use multiple seeds to bootstrap new nodes. Monitor gossip
convergence time (should be under 5 seconds for clusters under 100 nodes).
For very large clusters (500+ nodes), use supernode or hierarchical gossip
topologies.

## Cost of Conflict Resolution at Scale

Even LWW (Last-Write-Wins) has costs at scale. Consider an e-commerce catalog
with 10 million products and 100 updates per second.

### LWW Clock Skew

Node A has a clock that is 2 seconds ahead. Node B has a clock that is 2
seconds behind. Node A processes an update to product X at time T+2. Node B
processes an update to the same product at time T (real time). Because Node A's
timestamp is T+2 and Node B's is T, Node A's update always wins — even though
Node B's update happened later in real time.

**Cost:** Lost updates from nodes with slower clocks. In extreme cases, an
administrator's price update can be silently overwritten by a background job
running on a node with a faster clock.

**Mitigation:** Use client-assigned timestamps (the client has one clock
source). Use hybrid logical clocks instead of physical clocks. Monitor clock
skew across nodes and alert on skew exceeding 100ms.

### CRDT Storage Overhead

CRDTs require storing metadata alongside the data. A G-Counter (grow-only
counter) stores a vector of counts, one per node. With 100 nodes, each counter
is 100 integers (800 bytes). A PN-Counter (positive/negative counter) stores
two vectors (200 integers, 1,600 bytes). An LWW-Element-Set stores each element
with a timestamp.

**Cost:** CRDT metadata can consume 50-80% of storage for small payloads. A
system that stores 1 billion small items (each 100 bytes of data) might use
500GB for data and 2TB for CRDT metadata.

**Mitigation:** Only use CRDTs where automatic conflict resolution justifies
the overhead. For simple counters, use Cassandra's built-in counter type
(which uses its own CRDT-like mechanism with lower overhead). Prune old CRDT
metadata during compaction.

## Testing Consistency — Practical Approaches

Testing a BASE system's consistency guarantees requires specialized techniques.

### Jepsen-Style Testing

Jepsen is a testing framework that injects network partitions, node failures,
and clock skew while verifying that the database satisfies its claimed
consistency guarantees.

**Example test:** Deploy a 3-node Cassandra cluster. Write a value X to key K.
Induce a network partition between node 2 and node 3. Write value Y to key K
on node 1. Read key K from node 2. Under CL=QUORUM, the read should return Y.
Under CL=ONE, the read may return X or Y.

**Tool:** Use the `jepsen` library or a simpler custom script that:

1. Generates a sequential history of reads and writes with known expected
   values.
2. Applies fault injection (partition, kill, pause nodes).
3. Collects all returned values and their timestamps.
4. Verifies that the history is valid under the claimed consistency model.

### Application-Level Consistency Checks

In production, instrument your application to detect consistency violations:

1. **Read-your-writes check:** After writing to a BASE store, immediately read
   the same key. Log if the value differs from the expected value.
2. **Cross-system reconciliation:** Run a periodic job that reads the same key
   from all replicas and reports discrepancies.
3. **Business-rule validation:** For example, "inventory count for any SKU
   should never be negative." Write a cron job that scans all inventory counts
   and alerts on violations.

### Chaos Engineering for Consistency

Netflix's Simian Army includes Chaos Kong (simulates a region failure),
Chaos Monkey (randomly kills instances), and Latency Monkey (injects network
latency). Apply similar practices to your data layer:

1. **Node failure:** Kill one Cassandra node per hour during business hours.
   Monitor: do all reads still succeed? Does read repair catch up?
2. **Network partition:** Block traffic between two availability zones for 30
   seconds. Monitor: does the system converge within 5 minutes after the
   partition heals?
3. **Clock skew:** Set one node's clock 5 seconds ahead for 1 minute. Monitor:
   does LWW produce unexpected results? Do clients see "future" timestamps?
