# Understanding ACID vs BASE

Build the mental model that separates two fundamentally different approaches to
database transactions.

---

## The Problem Story

An e-commerce platform runs on a single PostgreSQL database. The schema tracks
inventory, orders, payments, and user sessions in relational tables with strict
foreign-key constraints and serializable isolation. For three years it works
flawlessly.

Then Black Friday arrives.

At 09:42 AM, the database hits 4,800 concurrent write connections. The
connection pool is exhausted. New writes queue and then time out. Customers see
"Something went wrong" when they click "Place Order". The site goes dark for 14
minutes while the engineering team scrambles to kill reporting queries and scale
up the instance.

The post-mortem reveals a painful truth: every order write required an inventory
row lock, a payment transaction, and a session update — all within the same
database transaction. The CTO asks a sharp question:

> *"Can we accept orders even if inventory counts are slightly delayed? Would
> our customers rather have a confirmation now with a correction later, or no
> order at all?"*

This question is the dividing line between **ACID** and **BASE**. It forces you
to decide: do you prioritize correctness or availability? Both are valid. Both
are necessary. The trick is knowing when to use each one.

---

## The Insight

ACID and BASE are not two acronyms on a whiteboard. They are competing
philosophies about how to handle data when the system is under stress, when the
network is slow, or when parts of the system fail.

### ACID — All or Nothing

ACID stands for **Atomicity, Consistency, Isolation, Durability**. It guarantees
that every transaction is a single unit of work that either completes in full or
is rolled back entirely. The database is always in a consistent state. If the
power goes out mid-write, the data is still correct when the machine comes back.

The cost of this guarantee is coordination. Every write acquires locks. Every
transaction holds resources until commit. Under load, this coordination becomes
a bottleneck.

### BASE — Good Enough, Eventually

BASE stands for **Basically Available, Soft state, Eventual consistency**. It
relaxes the ACID guarantees to achieve higher availability and throughput. A
BASE system accepts writes at any node, propagates changes asynchronously, and
resolves conflicts later. The data may be temporarily inconsistent, but it will
converge to a correct state over time.

The cost of this approach is complexity. You must handle stale reads, reconcile
conflicts, and design your application to tolerate temporary inconsistency.

### The Analogy

**ACID is a bank vault.** You deposit cash. The door closes, the lock engages,
and the vault registers the deposit. Nobody can access the vault until the
operation is fully complete. You wait, but you know your money is safe.

**BASE is a library book.** You walk up to a shelf, grab a book, and walk out.
The librarian notices the book is missing later and updates the catalog. A
patron looking for that book between your taking it and the catalog update sees
the book listed as available — a stale read. But eventually, the catalog is
accurate, and the system works for everyone most of the time.

## Core Vocabulary

| Term | Definition |
|---|---|
| **Atomicity** | A transaction is all-or-nothing. No partial writes survive a crash. |
| **Consistency** | Every transaction brings the database from one valid state to another. Constraints, triggers, and cascades are enforced. |
| **Isolation** | Concurrent transactions do not interfere. Each transaction runs as if it were alone. |
| **Durability** | Once committed, a write survives power loss, crashes, and reboots. |
| **Basically Available** | The system responds to every request, even if some nodes are down. No request is rejected. |
| **Soft State** | The system state can change over time without input, as replication catches up. |
| **Eventual Consistency** | If no new writes arrive, all replicas converge to the same value over time. |
| **CAP Theorem** | A distributed system can achieve at most two of: Consistency, Availability, Partition Tolerance. |
| **Strong Consistency** | Every read returns the most recent write. All replicas agree immediately. |
| **Weak Consistency** | Reads may return stale data. No guarantee of recency. |
| **Quorum** | Minimum nodes that must agree for a read or write to succeed. |
| **Read Repair** | During a read, detect stale replicas and update them with the correct value. |
| **Hinted Handoff** | When a replica is down, another node temporarily stores the write and forwards it later. |
| **Gossip Protocol** | Nodes periodically exchange state information to propagate metadata across the cluster. |
| **Conflict Resolution** | When two writes produce different values, the system must decide which one wins or how to merge them. |
| **Version Vector** | A logical clock that tracks causality across replicas to detect conflicting updates. |
| **Last-Write-Wins (LWW)** | A conflict resolution strategy where the write with the latest timestamp is accepted. |

**The 11 steps of an ACID write:**

1. Client issues `BEGIN` to start a transaction.
2. Parser validates the SQL syntax and produces a query plan.
3. Client issues `UPDATE inventory SET quantity = quantity - 1 WHERE id = 42`.
4. The lock manager acquires an exclusive row lock on the target row. Other
   transactions attempting to modify this row are queued.
5. The execution engine checks constraints: does `quantity > 0`? If not, it
   aborts the transaction.
6. The change is appended to the Write-Ahead Log (WAL). This record contains
   the before-and-after image of the row.
7. The WAL segment is fsynced to disk. This ensures crash recovery can replay
   the transaction even if the power fails immediately after.
8. The buffer pool page is updated in memory. The page is marked dirty.
9. The lock is released. Other transactions can now access the row.
10. Client issues `COMMIT`. The commit record is appended to the WAL and fsynced.
11. The database acknowledges the write to the client. The transaction is
    durable.

**Total time:** 2-5 milliseconds for a local write. The bottleneck is step 7
(fsync to disk), which takes 1-3 milliseconds on SSD media.

**The 10 steps of a BASE write:**

1. Client sends a write request to any node in the cluster. That node becomes
   the coordinator for this operation.
2. The coordinator hashes the partition key (inventory item ID 42) to determine
   which nodes own this data range. With replication factor 3, three nodes are
   responsible.
3. The coordinator sends the write to all three replica nodes in parallel.
4. Each replica writes the data to its in-memory memtable. No fsync is required
   at this point. The write is buffered.
5. Each replica acknowledges the write. The coordinator waits for the minimum
   number of acknowledgments defined by the write consistency level.
6. If the consistency level is QUORUM (2 out of 3), the coordinator collects two
   acks and can proceed.
7. The coordinator responds to the client: "Write successful."
8. Behind the scenes, each replica eventually flushes its memtable to an SSTable
   on disk. This is a sequential write, far cheaper than random I/O.
9. If a replica is unreachable, the coordinator stores the write as a
   **hinted handoff** — a small record saying "send this to node 7 when it
   comes back."
10. Periodically, nodes gossip with each other to compare Merkle trees and
    repair any discrepancies. This is **anti-entropy repair**.

**Total time:** 100-500 microseconds for quorum acknowledgment. The client
receives confirmation before any data is written to disk.

## Variants

Not every system is purely ACID or purely BASE. There is a spectrum of
consistency models that trade off strictness for performance.

### Pure ACID — PostgreSQL, MySQL (InnoDB)

These databases provide full ACID guarantees at the highest isolation levels.
Every transaction is serialized, every write is durable, and the system
maintains a single consistent view of data at all times. Use this for financial
ledgers, inventory counts that must not go negative, and legal compliance
records.

### Eventual Consistency — Cassandra, DynamoDB (default), Riak

These are pure BASE systems. Writes are accepted at any node and propagated
asynchronously. Reads may return stale data, but the system converges over time.
Use this for user profiles, product catalogs, session stores, and activity
feeds.

### Tunable Consistency — Cassandra, Riak

These systems let the developer choose the consistency level per operation.

| Consistency Level | Behavior | Use Case |
|---|---|---|
| **ONE** | Read or write from one replica. Fastest, least consistent. | High-throughput logs, counters. |
| **QUORUM** | Majority of replicas must agree. Balance of speed and consistency. | Product catalogs, user sessions. |
| **ALL** | All replicas must agree. Slowest, most consistent. | Critical metadata, configuration. |

If you write at QUORUM and read at QUORUM in a 3-replica cluster, you get
strong consistency because the majority of nodes overlap. This is a common
pattern for systems that need stronger guarantees but cannot use ACID.

### Causal Consistency

If operation A happens before operation B, every replica sees A before B.
Operations that are not causally related (concurrent) may be seen in different
orders. This is stronger than eventual consistency but weaker than strong
consistency. Used in Riak and some DynamoDB configurations.

### Read-Your-Writes

After you write a value, any subsequent read from the same session always sees
that value. This prevents the confusing experience of submitting data and then
seeing the old state. Cassandra provides this via the `CL.LOCAL_QUORUM` pattern
with token-aware routing.

### Monotonic Reads

Once a client reads a value at time T, all subsequent reads return a value that
is at least as recent as time T. The clock does not go backward. This prevents
the disorienting effect of seeing version 3 of a record, then seeing version 2
on the next read.

### Bounded Staleness

The system guarantees that stale data is never older than a time window (for
example, 5 seconds). This is available in Microsoft Azure Cosmos DB as "bounded
staleness" consistency. It provides a predictable upper bound on how stale a
read can be, which is useful for real-time dashboards and monitoring systems.

## Mental Model Check

These five questions test whether you can apply the ACID vs BASE distinction to
real scenarios. Explain your reasoning for each.

**Question 1:** A banking application processes wire transfers. Should it use
ACID or BASE?

**Answer:** ACID. Wire transfers demand atomicity (both sides of the transfer
complete or neither), consistency (total balance is preserved), isolation
(concurrent transfers do not see partial states), and durability (once
committed, the transfer is permanent). BASE could lose money if a node accepts
a withdrawal without checking the balance on another node.

**Question 2:** A social media feed shows posts from friends. A user might see a
post 2 seconds after it is published, or 5 seconds. Is this acceptable?

**Answer:** Yes, this is a BASE use case. Eventual consistency is fine for
social feeds. The cost of ACID (locking, coordination, limited write throughput)
is not justified when a few seconds of staleness has no material impact.

**Question 3:** An e-commerce site takes orders. On Black Friday, the choice is
"reject orders" (ACID, database at capacity) or "accept orders and fix inventory
later" (BASE, occasionally double-sell a hot item). Which is correct?

**Answer:** Neither is universally correct. The inventory system should use ACID
to prevent double-selling. But order intake can use BASE: accept the order,
place it in a queue, and check inventory asynchronously. If the item is sold
out, compensate the customer. This polyglot approach uses ACID where precision
matters and BASE where throughput matters.

**Question 4:** A real-time leaderboard for a gaming platform updates thousands
of times per second. Do players need to see the exact ranking at every instant?

**Answer:** No. Eventual consistency (or bounded staleness of 1-2 seconds) is
acceptable. Players expect slight delays on leaderboards. ACID would limit the
write throughput to a few thousand per second, making the leaderboard
impractical at scale.

**Question 5:** A medical records system stores patient data. A doctor prescribes
a medication that interacts fatally with another drug. The system must detect
this interaction. Which model is required?

**Answer:** ACID. The interaction check requires a consistent view of the
patient's current medication list. A BASE system could miss a recently added
medication if it reads from a stale replica, leading to a fatal outcome. Medical
systems must use strong consistency for clinical data.

## How BASE Systems Reveal Inconsistency — A Worked Example

To understand eventual consistency in practice, trace a specific scenario
through a Cassandra cluster with three replicas and consistency level ONE.

**Setup:** A product catalog with a single row for item SKU-42. The row has a
column `price` = 19.99. Three replicas (nodes A, B, C) all store this row.

**Event 1 — Price update:** An administrator updates the price to 24.99. The
write is coordinated by node A. The coordinator sends the write to all three
replicas. Node A and B acknowledge. Node C is experiencing a GC pause and
does not respond. The coordinator counts 2 of 3 acks — since CL=ONE, it only
needs 1, so the write succeeds immediately. Nodes A and B have the new price
(24.99). Node C still has the old price (19.99).

**Event 2 — Customer reads from node C:** The customer's read request is
routed to node C (the closest node by latency). Node C returns price = 19.99.
The customer sees the old price. They add the item to their cart.

**Event 3 — Checkout:** The customer proceeds to checkout. The checkout
service reads the price again, this time with consistency level QUORUM. The
coordinator reads from two nodes. One returns 24.99, the other returns 19.99.
The coordinator compares the timestamps: 24.99 has a later timestamp, so it
is the winner. The customer is charged 24.99.

**The gap:** Between Event 2 and Event 3, the customer believed the price was
19.99. They might have made a purchasing decision based on that belief. The
system eventually converged to the correct price, but the customer's
experience was inconsistent.

**Mitigation:** The catalog service could set a "stale read warning" flag or
re-read the price from QUORUM when the customer adds to cart. This increases
read latency but prevents the customer from seeing the wrong price.

## The Consistency Spectrum — A Deeper Walkthrough

Consistency is not binary. The spectrum from strict to eventual has well-defined
intermediate points. Understanding these helps you choose the right level of
consistency for each operation.

### Strong Consistency — The ACID End

Every read returns the most recent write. All replicas agree on the order of
all writes. This is the model used by single-node databases and by distributed
systems that use consensus protocols (Raft, Paxos, Spanner's TrueTime).

**Cost:** Every write must be acknowledged by a majority of replicas. A
three-node cluster requires 2 of 3 acks. A five-node cluster requires 3 of 5.
This adds network round trips and reduces throughput.

### Sequential Consistency

Operations from each client appear in the order the client issued them. The
interleaving across clients is consistent with some total order. This is
weaker than strong consistency because two clients may see different
interleavings, but each client's own operations are in order.

**Example:** Client A writes X=1, then X=2. Client B reads X and gets 2. Then
Client B reads X again and gets 1. This is allowed under sequential consistency
because B's first read happens to see the later write, but B's second read
sees the earlier write. The clock does not go backward within a single client's
view, but the order across clients can vary.

### Causal Consistency

If operation A happens before operation B (causal relationship), all clients
see A before B. Operations that are not causally related (concurrent) may be
seen in different orders.

**Example:** Alice posts a comment on a photo. Bob replies to Alice's comment.
Every client sees Alice's comment before Bob's reply. But two comments posted
by different users on different photos may be seen in any order.

**Implementation:** Causal consistency requires tracking dependencies. Each
operation carries a vector clock or a dependency list. The system delays
delivery of an operation until all its dependencies are satisfied.

### Read-Your-Writes Consistency

A weaker guarantee: after a client writes a value, any subsequent read by the
same client returns that value or a later one. Other clients may not see the
write until later.

**Implementation:** The client session is sticky to a specific replica. All
reads and writes go through the same coordinator. The coordinator always
serves the latest value it has seen for that session.

**Cost:** Session stickiness reduces load-balancing flexibility. If the
sticky session's node fails, the client must reconnect and may lose the
read-your-writes guarantee.

### Bounded Staleness

The system guarantees that data is never more than K versions behind or T
seconds old. This is a practical compromise: the system is eventually
consistent, but the staleness is bounded by a known limit.

**Example:** Azure Cosmos DB offers bounded staleness with a maximum of
100,000 operations or 5 seconds behind, whichever comes first. A read that
would return data older than 5 seconds is blocked until the replica catches
up.

**Cost:** The bound requires tracking the staleness of each replica. If a
replica falls behind the bound, reads to that replica cannot be served. This
reduces availability during high-latency periods.

## Failover Behavior — ACID vs BASE

Understanding how each system behaves during a failure is essential for
choosing the right model.

### ACID Failover

When the primary node fails:

1. The system detects the failure (heartbeat timeout, connection failure).
2. A replica is promoted to primary. This requires a manual or automated
   process (Patroni, RDS Multi-AZ, pg_auto_failover).
3. All in-flight transactions on the failed primary are lost. They must be
   retried by the application.
4. The new primary replays the WAL from the last checkpoint. This is ARIES
   recovery in action.
5. The application must reconnect to the new primary. Connection strings or
   DNS entries are updated.

**Downtime:** 30-120 seconds for automated failover. During this window, the
database is unavailable for writes. Reads continue on replicas, but they may
return stale data (the last committed state before the failure).

### BASE Failover

When a node fails:

1. The remaining nodes detect the failure via gossip. The failed node is marked
   as down.
2. Writes intended for the failed node are handled by hinted handoff. The
   coordinator stores the write and will forward it when the node recovers.
3. Reads do not see the failed node. The coordinator reads from the remaining
   replicas. If the consistency level is ONE, reads continue without
   interruption. If ALL, reads fail until the node recovers.
4. When the node recovers, it receives hints from the coordinators. It also
   runs an anti-entropy repair to compare Merkle trees with peers.

**Downtime:** Zero for reads and writes at consistency ONE. Partial for higher
consistency levels. A five-node cluster with RF=3 and CL=ALL loses read
availability when one node fails (2 of 3 replicas cannot satisfy ALL).

## Summary Comparison Table

| Property | ACID | BASE |
|---|---|---|
| Consistency model | Strong (immediate) | Eventual (convergent) |
| Write throughput | 500-5,000 / sec / node | 10,000-100,000 / sec / node |
| Read latency | 0.5-5ms | 1-50ms (depends on CL) |
| Scalability | Vertical (primary bottleneck) | Horizontal (near-linear) |
| Conflict handling | Prevented (locks, isolation) | Resolved (LWW, CRDT, merge) |
| Node failure impact | Writes blocked during failover | Writes continue (hinted handoff) |
| Network partition | Minority side unavailable | Both sides accept writes |
| Cross-region latency | 80-160ms (sync replication) | 1-5ms (async replication) |
| Operational complexity | Moderate (failover, backup) | High (gossip, repair, compaction) |
| Best for | Financial, legal, medical | Social, analytics, content, cache |
