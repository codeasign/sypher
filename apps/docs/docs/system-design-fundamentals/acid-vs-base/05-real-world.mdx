# ACID vs BASE in Production

The trade-offs page laid out the theory. This page examines how four major
companies apply ACID and BASE in their production systems, what broke, and
what they learned.

---

## Uber — ACID for Payments, BASE for Trip History

Uber operates one of the largest hybrid ACID/BASE architectures in production.
Every second, thousands of riders request trips, drivers accept them, and
payments are processed. The data platform spans multiple data stores, each
chosen for its consistency characteristics.

### The Payment System — ACID (MySQL with Schemaless)

Uber's payment processing runs on **Schemaless**, a sharded MySQL layer built
in-house. Schemaless wraps MySQL into a document-store interface while
preserving ACID guarantees within each shard.

**Why ACID:** A payment charge must succeed or fail atomically. If Uber charges
a rider but the driver does not receive the funds, the system is broken. If the
charge fails but the ride is marked as completed, revenue is lost. Each payment
transaction is a single-row document operation in Schemaless, guaranteeing
atomicity and durability.

**Scale:** Uber processes hundreds of millions of payment transactions per day.
Each shard handles a subset of riders, so write throughput scales horizontally
by adding shards. Cross-shard transactions are avoided — payments are
per-rider, so they fit within a single shard.

**Key learning:** Sharding an ACID database works if you can partition by a
natural key (rider ID). Cross-shard transactions require application-level
coordination and are slower. Uber avoids them by designing the data model so
that related data lives on the same shard.

### The Trip History — BASE (Cassandra)

Ride history (past trips, routes, receipts) is stored in **Cassandra**. This
data is append-only per rider. A rider's trip history grows over time, and
they expect to see recent trips near-instantly.

**Why BASE:** Trip history is read-heavy. A rider views their history
frequently, but writes (new trips) are bounded by the number of rides.
Cassandra's partition-per-rider model makes reads fast: a single query to the
rider's partition returns the last N trips. Eventual consistency is acceptable
— a rider seeing a trip that finished 5 seconds ago may not appear in history
for another 2 seconds.

**Key learning:** Uber uses Cassandra's time-to-live (TTL) feature to
automatically expire old trip data. Trips older than a configurable threshold
are deleted transparently. This prevents unbounded storage growth and avoids
manual cleanup.

### The Bridge — Uber's Data Platform

Between the ACID payment system and the BASE trip history, Uber runs a data
platform based on **Apache Kafka** and **HDFS**. Payment events are streamed
from MySQL through Kafka to the trip history system. This is a CDC pipeline
that bridges ACID and BASE, exactly as described in the architecture page.

## Netflix — ACID for Billing, BASE for Everything Else

Netflix operates one of the largest microservice architectures in the world.
Each microservice owns its data, and each data store is chosen for its
consistency characteristics. Netflix's architecture document describes this as
a "data grid" — a set of polyglot data services connected by APIs.

### Billing — ACID (Oracle / PostgreSQL)

Customer billing runs on relational databases. Each charge must be recorded
exactly once. Refunds must be atomic with the charge reversal. Tax calculations
require multi-row ACID guarantees.

**Why ACID:** Billing is the canonical "must be correct" domain. Netflix
initially used Oracle and has migrated some billing workloads to PostgreSQL.
The migration was painstaking — every transaction had to be verified for ACID
compliance before cutover.

**Scale:** Netflix has 260 million subscribers. Each subscriber generates one
billing transaction per month (plus upgrades, downgrades, and pro-rated
charges). That is approximately 100 billing writes per second — well within
the range of a well-tuned PostgreSQL instance with read replicas.

### Everything Else — BASE (Cassandra, EVCache, DynamoDB)

Netflix's non-billing data spans three main BASE systems:

**Cassandra:** User viewing history, recommendations state, A/B test
assignments. These are high-volume, high-write data that can tolerate seconds
of staleness. Netflix runs one of the largest Cassandra deployments in the
world.

**EVCache (Memcached-based):** Session cache, view counts, thumbnail metadata.
EVCache is a distributed key-value store built on Memcached with cross-region
replication. Data is cached with TTL and can be regenerated from the source of
truth.

**DynamoDB:** Some newer services use DynamoDB for its managed scaling and
consistent reads. Netflix treats DynamoDB as a BASE store (using
eventually-consistent reads by default) with strongly-consistent reads for
specific operations like user preferences.

### Key Learning — The Data Grid Abstraction

Netflix exposes data through a set of RESTful data services that abstract the
underlying storage. A recommendation service does not know whether its data
comes from Cassandra, EVCache, or a relational database. This abstraction
allows Netflix to migrate between storage engines without changing service
code.

When Netflix migrated from Oracle to PostgreSQL for billing, the billing
service's API did not change. Only the data service implementation changed.

## Amazon — The Origin of BASE

The term BASE was coined by Amazon's CTO Werner Vogels in a 2007 blog post
accompanying the release of the Dynamo paper. Amazon's architecture is the
origin of the ACID vs BASE distinction.

### The Dynamo Paper — Shopping Cart as BASE

Amazon published the **Dynamo** paper in 2007, describing a key-value store
that traded ACID guarantees for high availability. Dynamo was built to serve
the shopping cart — a humble component that revealed a profound truth.

**The insight:** A shopping cart does not need ACID. If a customer adds an item
to their cart and the write reaches only 2 of 3 replicas, the customer still
sees the item in their cart (from the local replica). If the item is
temporarily lost, the customer can add it again. The cost of a lost cart item
is a minor inconvenience. The cost of rejecting a write during Black Friday is
a lost sale.

**Dynamo features:** Dynamo introduced the core BASE mechanisms:

- **Consistent hashing** for partition placement
- **Vector clocks** for conflict detection
- **Hinted handoff** for availability during node failures
- **Merkle trees** for anti-entropy
- **Configurable quorum** (R + W > N) for tunable consistency

### Order Processing — ACID (Oracle)

While the shopping cart is BASE, order processing is ACID. When a customer
clicks "Place Order", the system must:

1. Check inventory (consistent read across the catalog)
2. Reserve the item (atomic write)
3. Charge the payment method (durable transaction)
4. Create the order record (all steps or none)

Amazon runs one of the largest Oracle deployments for order processing. The
order transaction is ACID. The shopping cart that feeds into it is BASE.

### Key Learning — ACID at the Boundary

Amazon learned that BASE works upstream (shopping cart, recommendations) but
the boundary where money changes hands must be ACID. The transition from BASE
(cart) to ACID (order) is managed by a checkout service that reads the cart
data and processes the ACID transaction. If the ACID transaction fails, the
cart is preserved for retry. The cart data itself is never in ACID storage.

## Discord — MongoDB to Cassandra for Messages

Discord's message storage migration is one of the clearest case studies of
moving from an ACID-ish system to a BASE system for a specific data domain.

### The Problem — MongoDB Write Contention

Discord originally stored all messages in MongoDB. As the platform grew to
millions of concurrent users, MongoDB's document-level locking created write
contention. A busy channel with 10,000 members could receive 100 messages per
second. MongoDB could not sustain this write load on a single replica set.

**The ACID cost:** MongoDB uses document-level locking. Each message insert
acquired a write lock on the channel's document. Contention caused write
latency to spike to 200ms+ on busy channels. Reads also suffered.

### The Solution — Cassandra (BASE)

Discord migrated messages to Cassandra. Each message is a row in a Cassandra
table partitioned by channel ID and clustered by message creation timestamp.

**The BASE trade-off:** Discord accepted eventual consistency for messages.
When a user sends a message, it is written to Cassandra with consistency level
ONE (fastest). The message appears in the channel immediately for the sender
(the write goes to their local coordinator). Other users may see it with a
delay of 100-500 milliseconds. This is acceptable — users expect a brief
transmission delay.

**Key challenge:** Discord had to handle the "missing last message" problem.
When a user refreshes their channel, Cassandra may return slightly different
sets of messages if read repair has not converged. Discord uses a secondary
index with an anti-entropy repair to close this gap.

### Key Learning — Read-Your-Writes for Chat

Discord discovered that a chat app needs read-your-writes consistency: the
sender must always see their own message immediately, even if other users see
it later. Discord implemented this by:

1. Writing the message to Cassandra with a timestamp generated by the client,
   not the server.
2. The client optimistically displays the message before the server confirms it.
3. When the server confirms, the client replaces the optimistic version.
4. If confirmation fails (rare), the client retries.

This is a form of client-side read-your-writes that bridges the BASE
consistency gap for the most critical operation (the sender's experience).

## Open Source and Cloud Providers

### Open Source Databases by Model

**ACID (full transactional support):**

- **PostgreSQL:** Full ACID with serializable snapshot isolation. WAL-based
  replication. The gold standard for open-source ACID.
- **MySQL (InnoDB):** ACID with configurable isolation levels. Group commit for
  WAL. Widely deployed for web applications.
- **SQLite:** Full ACID within a single file. WAL mode for concurrent reads.
  Not distributed. Used for embedded and mobile.
- **CockroachDB:** Distributed ACID with serializable isolation across nodes.
  Uses Raft consensus and HLC (hybrid logical clocks). The closest open-source
  equivalent to Google Spanner.
- **YugabyteDB:** Distributed ACID with PostgreSQL-compatible query layer.
  Uses Raft for replication and DocDB as the distributed document store.

**BASE (eventual consistency):**

- **Apache Cassandra:** Ring-based, gossip-protocol, tunable consistency.
  Merkle trees for anti-entropy. Designed for high write throughput.
- **Riak:** Dynamo-inspired key-value store. CRDT support for automatic
  conflict resolution. Application-level merge for complex data.
- **Voldemort:** LinkedIn's Dynamo clone. Simple key-value with configurable
  quorum. Uses version vectors for conflict detection.
- **Redis with CRDTs:** Redis Enterprise supports CRDT-based replication for
  active-active geo-distribution. Data converges automatically.

**Hybrid (tunable consistency):**

- **FoundationDB:** Provides serializable ACID transactions but exposes a
  key-value interface. Layers on top provide document, relational, or graph
  models.
- **Azure Cosmos DB:** Five consistency levels from strong to eventual.
  Bounded staleness guarantees at most K versions or T time behind.
- **DynamoDB:** Three consistency options (eventual, strong, transactional).
  Strongly-consistent reads are 2x the read capacity unit cost.

### Cloud Provider Offerings

| Cloud | ACID Service | BASE Service |
|---|---|---|
| AWS | RDS (PostgreSQL, MySQL, Oracle, SQL Server) | DynamoDB (eventual/strong reads), S3 (read-after-write consistency) |
| AWS | Aurora (MySQL/PostgreSQL compatible, 6 replicas, async/ sync) | ElastiCache (Redis, Memcached) |
| GCP | Cloud SQL (PostgreSQL, MySQL, SQL Server) | Bigtable (eventual, wide-column), Firestore (strong/eventual) |
| GCP | Cloud Spanner (globally distributed ACID) | BigQuery (analytical, strong read-after-write) |
| Azure | SQL Database (SQL Server, managed) | Cosmos DB (tunable, 5 consistency levels) |
| Azure | Azure Database for PostgreSQL/MySQL | Table Storage (key-value, eventual) |

## Lessons Learned

The companies above share common lessons about ACID and BASE in production.

### Lesson 1: Know Your Consistency Requirements Before Choosing

Amazon chose BASE for the shopping cart because they analyzed the failure modes
and found them acceptable. Discord chose BASE for messages because they
accepted the 500ms convergence window. Netflix chose ACID for billing because
the cost of inconsistency is direct revenue loss. Each decision was made with
clear understanding of what could go wrong.

### Lesson 2: Use ACID for the Source of Truth, BASE for Scale

Every company in this case study uses ACID as the authoritative source of truth
for business-critical data (payments, orders, billing). BASE systems serve
derived views, high-throughput reads, or data with shorter consistency windows.
The ACID system is the system of record; the BASE system is the system of
engagement.

### Lesson 3: Plan for Conflict Resolution

Every BASE deployment produces conflicts. Even with LWW, clock skew produces
anomalies. Even with CRDTs, data types limit what you can express. The teams
that succeed with BASE plan for conflicts before deployment — they define how
conflicts are detected, resolved, and monitored.

### Lesson 4: Test with Network Partitions

The failure modes that break BASE systems (network partitions, node failures,
clock skew) are difficult to reproduce in development. Netflix uses Chaos
Engineering (the Simian Army) to inject these failures in production. Amazon
uses a similar approach with fault-injection testing. Discord tests partition
recovery by physically disconnecting nodes in their staging environment.

If you run a BASE system at scale, introduce network partitions in a staging
environment and observe how long convergence takes. If convergence exceeds
your business tolerance, adjust your consistency levels or replication factor.

## Common Pitfalls in Production BASE Deployments

Teams deploying BASE databases for the first time encounter these patterns
repeatedly. Recognizing them early reduces operational pain.

### Pitfall 1: Not Setting Read Repair Chance

Cassandra's read repair chance defaults to 0.1 (10%). At 10,000 reads per
second across a table with 1 million rows, each row is read-repaired once
every 1,000 seconds (~17 minutes). In a table with 100 million rows, each row
is read-repaired once every 100,000 seconds (~28 hours). Stale data persists
for hours.

**Fix:** Set read repair chance to 1.0 for tables with critical data during
the initial data load, then reduce to 0.2-0.5 for steady state. Run full
anti-entropy repair on a weekly schedule during low traffic.

### Pitfall 2: Overloading a Single Partition

Cassandra partitions data by partition key. If one partition key receives
disproportionate traffic (a viral tweet, a popular product), that partition
becomes a hot spot. All queries for that partition go to a single node
(plus replicas). That node handles 100x the traffic of its peers.

**Fix:** Design partition keys to distribute load evenly. For a social media
app, partition by `(user_id, year_month)` rather than `user_id` alone. This
spreads a single user's data across multiple partitions and nodes.

### Pitfall 3: Ignoring Cluster-Wide Repair Scheduling

Running `nodetool repair` on all nodes simultaneously creates a multiplication
of repair traffic. Each node tries to repair its data with every other node.
The cluster's I/O capacity is consumed by repair traffic, and normal read/write
latency spikes.

**Fix:** Run repairs sequentially, one node at a time, using a repair
scheduler. Use `nodetool repair --parallelism=parallel` but limit to one node
per rack per hour. Monitor repair progress and abort repairs that take longer
than 6 hours.

### Pitfall 4: Assuming Eventual Consistency Means Eventually Correct

Eventual consistency guarantees convergence, but the converged value is not
necessarily the value any particular client intended. Under LWW, if three
clients concurrently update the same key with values A, B, C, the converged
value is whichever had the latest timestamp — not necessarily the intended
result.

**Fix:** For data that needs "correctness" (not just convergence), use
application-level validation after convergence. For example, an inventory
count that converges to 5 items should be validated against the actual
physical count during nightly reconciliation.

## Open Source Ecosystem — Detailed Comparison

The choice between open-source databases often depends on more than just the
ACID/BASE distinction. Below is a detailed comparison of the most popular
options for each model.

### PostgreSQL — The ACID Standard

PostgreSQL is the most widely deployed open-source ACID database. Its
strengths for ACID workloads:

- Full serializable isolation with SSI (no phantom reads, no write skew)
- Extensive data types (JSON, array, range, geometric, custom)
- Write-Ahead Log with group commit for high-throughput writes
- Logical replication (for CDC to BASE systems) and physical replication (for
  high availability)
- Extensibility: extensions like PostGIS (geospatial), pg_partman
  (partitioning), and timescaledb (time-series)

**Weakness:** Single-node write bottleneck. Sharding requires external tools
(Citus, pg_shard, or application-level sharding).

### MySQL with InnoDB — The Web Standard

MySQL with the InnoDB storage engine is the second most popular open-source
ACID database. Its strengths:

- Group commit for WAL (up to 10x write throughput improvement in MySQL 8.0)
- Read replicas with semi-synchronous replication
- Row-level locking with MVCC (Read Committed + Repeatable Read)
- Large ecosystem: Vitess for sharding, ProxySQL for connection pooling,
  Orchestrator for failover management

**Weakness:** The optimizer is weaker than PostgreSQL for complex queries.
Replication lag can be unpredictable under heavy write load.

### Cassandra — The BASE Standard

Apache Cassandra is the most widely deployed open-source BASE database. Its
strengths:

- Linear write scalability: each node adds ~50K writes/second
- Tunable consistency per operation (ONE, QUORUM, ALL, LOCAL_QUORUM, etc.)
- Ring topology with gossip: no single point of failure
- Multi-data center support with configurable replication
- CQL (Cassandra Query Language): SQL-like syntax for easier adoption

**Weakness:** No joins, no foreign keys, no transactions. Read performance
degradation under compaction pressure. Tombstone accumulation from frequent
deletes.

### CockroachDB — Distributed ACID

CockroachDB is the leading open-source distributed ACID database. It combines
the ACID guarantees of PostgreSQL with the horizontal scalability of Cassandra.

- Serializable isolation across all nodes (uses Raft consensus + HLC)
- PostgreSQL-compatible wire protocol (most PostgreSQL drivers work directly)
- Automatic data rebalancing across nodes
- Geo-partitioning: place data in specific geographic locations for
  compliance/latency

**Weakness:** Higher latency than single-node PostgreSQL (Raft consensus adds
a round trip). Lower write throughput than Cassandra (~10K writes/sec/node vs
~50K in Cassandra). Not suitable for sub-millisecond access patterns.

## Cloud Provider Deep Dive — DynamoDB vs Aurora

For teams choosing between AWS-managed services, the DynamoDB vs Aurora
decision is the most common ACID vs BASE choice.

### Amazon DynamoDB (BASE)

DynamoDB is a fully managed key-value and document database. It provides BASE
consistency by default but offers strongly consistent reads (at 2x cost).

| Feature | Eventually Consistent Read | Strongly Consistent Read |
|---|---|---|
| Read cost | 0.5 RCU per 4KB item | 1 RCU per 4KB item |
| Latency (single-digit ms) | 1-3ms | 3-8ms |
| Stale data possible | Yes, up to 1 second | No |
| Availability | 99.99% | 99.99% |

DynamoDB's sweet spot: high-traffic web applications, gaming leaderboards, session stores,
shopping carts, and metadata catalogs. The lack of cross-item constraints is a
limitation for financial data.

### Amazon Aurora (ACID)

Aurora is a MySQL/PostgreSQL-compatible database with distributed storage. It
provides full ACID guarantees.

| Feature | Aurora MySQL | Aurora PostgreSQL |
|---|---|---|
| Write throughput | Up to 200K writes/sec | Up to 100K writes/sec |
| Read replicas | Up to 15 (async) | Up to 15 (async) |
| Failover time | 30-60 seconds | 30-60 seconds |
| Storage | Auto-scaling up to 128TB | Auto-scaling up to 128TB |

Aurora's sweet spot: business-critical applications, financial systems, ERP,
and any workload that requires joins, foreign keys, or cross-row constraints.

### Decision Framework

| Requirement | Choose DynamoDB | Choose Aurora |
|---|---|---|
| Multi-item transactions | Not needed | Required |
| Query flexibility | Single-table access patterns | Complex joins, aggregations |
| Write throughput | Over 10K writes/sec | Under 10K writes/sec |
| Consistency | Eventually consistent OK | Strong consistency required |
| Schema | Schema-less, flexible attributes | Fixed schema with migrations |
| Cost model | Pay per request | Pay per provisioned capacity |

## Migration Horror Stories — Learning from Failures

### The Misconfigured CDC Pipeline

A fintech company set up Debezium CDC from PostgreSQL to Cassandra. The
Cassandra cluster was deployed with RF=2 (replication factor of 2). A single
node failure made the Cassandra cluster unable to satisfy reads at QUORUM
(needs 2 of 2 — one node fails, only 1 remains). The symptom: every read from
Cassandra returned an error. The order history dashboard showed zero data for
4 hours until the failed node was replaced.

**Lesson:** Always set RF=3 in production. QUORUM with RF=2 requires both
nodes — no fault tolerance.

### The Unbounded Tombstone

A social media company stored post likes in Cassandra. When a user unliked a
post, a tombstone was created. Popular posts received millions of likes and
unlikes per day. After 10 days (gc_grace_seconds default), tombstones were
compacted. But during the 10-day window, queries to the popular post's
partition had to scan through millions of tombstones. Read latency for the
most popular post reached 30 seconds.

**Lesson:** Use TTL-based expiry for short-lived data. For like/unlike
patterns, use a counter column or a CRDT that does not create tombstones.

### The Clock Skew Disaster

An e-commerce company used Cassandra with LWW conflict resolution for product
prices. One node in the cluster had a hardware clock that drifted 5 seconds
per day. Over a week, the drift reached 35 seconds. All writes coordinated by
that node carried timestamps 35 seconds in the future. These writes always won
against writes from other nodes. Price updates made by other nodes were
silently overwritten.

**Lesson:** Run NTP on every node with hourly synchronization checks. Monitor
clock skew and alert on any node with skew exceeding 100ms. Use
client-assigned timestamps for critical data.
