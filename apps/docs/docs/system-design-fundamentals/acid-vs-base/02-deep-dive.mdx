# ACID vs BASE — Under the Hood

The previous page built the mental model. This page opens the engine room:
examine the internal data structures, algorithms, and performance
characteristics that make each design tick.

---

## Internal Architecture — ACID System

An ACID database is built around a single-node architecture that guarantees
correctness through careful coordination. Every component is designed to
enforce the four ACID properties.

### The Core Components

**Write-Ahead Log (WAL):** Every modification is recorded in the WAL before it
reaches the data files. The WAL is append-only and sequential, which is fast
for spinning disks and very fast for SSDs. During crash recovery, the database
replays the WAL to reconstruct any lost in-memory changes.

**Lock Manager:** Manages row-level or table-level locks. Implements Two-Phase
Locking (2PL) to ensure serializable isolation. The lock manager tracks which
transactions hold which locks and which transactions are waiting. Deadlock
detection runs periodically — if a cycle is found, one transaction is aborted.

**Buffer Pool:** A memory cache of data pages. The buffer pool reduces disk I/O
by keeping frequently accessed pages in RAM. Dirty pages (modified but not
yet written to disk) are tracked for background flushing. The buffer pool is
managed by a page replacement policy, typically a variant of Clock or LRU.

**Transaction Manager:** Tracks the state of each transaction (active,
preparing, committed, aborted). Maintains a commit log and generates
savepoints for nested transactions. Coordinates with the lock manager to
release locks at commit or abort time.

**ARIES Recovery:** The Algorithm for Recovery and Isolation Exploiting
Semantics is the standard recovery algorithm used by PostgreSQL, Oracle, DB2,
and SQL Server. It uses the WAL to redo committed transactions and undo
uncommitted transactions after a crash, guaranteeing that the database
reaches a consistent state.

## Internal Architecture — BASE System

A BASE database is distributed by design. It sacrifices tight coordination for
scalability and availability. The key components center around replication,
failure detection, and eventual convergence.

### The Core Components

**Partitioner:** Maps each key to a range of nodes. In Cassandra, the partitioner
uses consistent hashing (Murmur3Partitioner) to assign each row to a token
range. When a node joins or leaves, only neighboring ranges shift — O(1/n)
of data moves rather than full rehashing.

**Gossip Protocol:** Each node periodically exchanges state information with up
to three random peers. Gossip runs every second by default. Within seconds,
every node knows about every other node's state (live, dead, joining, leaving).
This is a push-pull protocol: each message contains the sender's state, and
the receiver responds with its own state if its version is newer.

**Hinted Handoff:** If a write target is unavailable, the coordinator stores
the write as a hint (target node ID + data + timestamp). Hints are stored
locally and replayed when the target node reconnects. The default time window
for retaining hints is 3 hours. Beyond that, the hint is discarded and the
write must be repaired through read repair or anti-entropy.

**Read Repair:** When a read request returns, the coordinator compares the
digest (hash) of the data from all replicas. If one replica returns a
different digest, the coordinator fetches the full data from the most recent
replica and writes it to the stale replicas. This happens synchronously before
returning to the client (blocking read repair) or asynchronously after the
response (non-blocking).

**Merkle Trees for Anti-Entropy:** Each node maintains a Merkle tree of its
data for each table. The tree is a balanced hash tree: leaf nodes contain
hashes of individual row ranges, and internal nodes contain hashes of their
children. Nodes exchange Merkle tree roots during gossip. If two nodes have
different root hashes for the same range, they recursively compare children to
find the exact rows that differ, then repair them. This is the anti-entropy
mechanism that converges the cluster over time.

## Data Structures and Algorithms

The internal algorithms define what each system can and cannot do. Below is a
side-by-side reference of the key data structures and their roles.

| Algorithm | ACID Use | BASE Use |
|---|---|---|
| **ARIES Recovery** | Redo/undo after crash using WAL. Standard across relational databases. | Not used. BASE systems rely on hinted handoff and anti-entropy. |
| **Write-Ahead Log** | Sequential append with fsync. Enforces durability. Every write goes here first. | Optional (commit log in Cassandra). Writes go to memtable first, then flushed. |
| **Two-Phase Locking (2PL)** | Acquire locks during transaction, release at end. Ensures serializable isolation. | Not used. Locks would block availability. Systems use conflict resolution instead. |
| **MVCC** | Each transaction sees a snapshot at its start time. Readers do not block writers. | Not used in pure BASE systems. Some (CockroachDB) use MVCC for hybrid ACID. |
| **Gossip Protocol** | Not used. ACID is single-node or primary-replica. | Every BASE node uses gossip to track cluster membership and metadata. |
| **Merkle Tree** | Not used. Single node sees all data directly. | Used for anti-entropy. Nodes compare hashes to find inconsistent rows. |
| **Vector Clocks** | Not used. Single node serializes all writes. | Used to track causality and detect conflicting writes in Dynamo-style systems. |
| **CRDTs** | Not used. Conflicts do not arise under serializable isolation. | Conflict-Free Replicated Data Types (counters, sets, maps) that merge automatically. |
| **Hinted Handoff** | Not applicable. Single primary node handles writes. | Temporary storage for writes intended for unavailable replicas. |
| **Quorum** | Not needed. All data is on one node. | Configurable per operation: ONE, QUORUM, ALL. Governs consistency level. |

## Consistency and State

### ACID — One Truth, Always

In an ACID system, there is exactly one copy of the truth. All reads and writes
go through the same primary node (or primary replica in a read-replica setup).
Isolation levels control how much of that truth concurrent transactions see:

| Isolation Level | Dirty Read | Non-Repeatable | Phantom | Performance |
|---|---|---|---|---|
| Read Uncommitted | Possible | Possible | Possible | Fastest |
| Read Committed | Prevented | Possible | Possible | Fast |
| Repeatable Read | Prevented | Prevented | Possible | Moderate |
| Serializable | Prevented | Prevented | Prevented | Slowest |

Each level trades isolation strictness for throughput. Serializable isolation
is the most expensive because it holds locks for the entire transaction.

### BASE — Multiple Versions, Reconciliation

In a BASE system, truth is temporarily plural. Two concurrent writes to the
same key on different nodes can produce different values. The system must
reconcile these versions when they converge.

**Conflict Resolution Strategies:**

**Last-Write-Wins (LWW):** Each write carries a timestamp (from a coordinator
node or from the client). The write with the highest timestamp wins. This is
the simplest strategy and the default in Cassandra and DynamoDB. Problem:
clock skew between nodes can cause lost updates. If node A has a clock 2
seconds ahead, writes coordinated by node A always win against writes from
node B.

**Application-Level Merge:** When a conflict is detected, both versions are
returned to the client application, which decides how to merge them. This is
used in Amazon Dynamo (the original Dynamo paper) and Riak. For example, a
shopping cart conflict returns both versions, and the application merges the
item lists. This gives the application full control but requires every client
to handle conflicts.

**CRDT-Based Auto-Merge:** Conflict-Free Replicated Data Types (CRDTs) are
data structures that merge automatically without conflicts. A grow-only
counter (G-Counter) uses a vector per node, and merging is the per-element
max function. A last-write-wins element set combines LWW timestamps with set
operations. CRDTs are used in Riak 2.0+, Redis CRDT-based replication, and
SoundCloud's Roshi. They eliminate the need for conflict resolution logic in
the application.

## Performance Characteristics

The performance profiles of ACID and BASE systems diverge dramatically under
load. These numbers are representative for a single node or a small cluster on
modern SSD hardware.

### Write Throughput

**ACID:** 500 to 5,000 writes per second per node. The bottleneck is the fsync
required by the WAL. Each fsync takes 1-3 milliseconds on SSD, limiting writes
to roughly 1,000 per second for a single-threaded synchronous commit. With
group commit (batching multiple fsyncs together), throughput can reach 5,000.
With asynchronous commit (dangerous — can lose data on crash), throughput can
reach 20,000.

**BASE:** 10,000 to 100,000 writes per second per node. There is no fsync on
the write path. The write goes into the memtable (in-memory) and is
acknowledged immediately. Background flushing to SSTables happens as a
sequential write, which is fast and does not block the write path. In
Cassandra, single-node throughput can reach 50,000-100,000 writes per second
for small payloads.

### Read Latency

**ACID:** Under 5 milliseconds for a point query on a primary key in a warm
buffer pool. If the page is in the buffer pool (90%+ hit rate), it is a single
memory lookup. If it requires a disk read, add 5-10 milliseconds.

**BASE:** 1 to 50 milliseconds depending on consistency level. A read at
consistency ONE returns from the nearest replica quickly (1-5ms). A read at
consistency ALL must wait for the slowest replica, which could be 50ms if one
replica is under load or has a GC pause.

### Scalability

**ACID:** Vertical scaling (bigger machines). Adding read replicas helps with
read throughput but the primary write node is the bottleneck. Writes do not
scale horizontally. The maximum write throughput is the throughput of a single
machine.

**BASE:** Horizontal scaling (more machines). Adding nodes increases both read
and write throughput linearly (near-linear). With a replication factor of 3,
adding 3 nodes increases capacity by 1 node worth of throughput.

## Configuration and Tuning

Both systems expose knobs that trade durability or consistency for performance.

### ACID Tuning Parameters

**fsync:** Controls whether the WAL is flushed to disk on every commit. Setting
`fsync = off` in PostgreSQL doubles or triples write throughput but risks data
loss on power failure or OS crash. Only use this on replicas where data can be
re-fetched from the primary.

**Isolation Level:** The default `READ COMMITTED` is fast because it does not
require phantom-read prevention. `SERIALIZABLE` is much slower because it
requires predicate locks or serializable snapshot isolation (SSI) tracking.
Only raise isolation when business logic demands it.

**Checkpoint Frequency:** Checkpoints flush dirty pages from the buffer pool to
data files. Frequent checkpoints reduce crash-recovery time but increase I/O
load. In PostgreSQL, `checkpoint_completion_target = 0.9` spreads the I/O
across 90% of the checkpoint interval. Tuning this balances write performance
against recovery time.

**WAL Size:** A larger WAL absorbs write bursts without forcing checkpoints.
But a giant WAL takes longer to replay after a crash. PostgreSQL's
`max_wal_size = 1GB` is a balanced default. For high-write workloads, 4-16GB
is common.

### BASE Tuning Parameters

**Replication Factor (RF):** The number of nodes that store each row. RF=3 is
the standard for production. RF=1 loses all redundancy. RF=5 is used for
multi-region deployments. Higher RF increases write cost (more nodes to
coordinate) but improves read availability.

**Consistency Level per Operation:** Cassandra allows setting the consistency
level for each individual read and write. Common patterns:

| Pattern | Write CL | Read CL | Property |
|---|---|---|---|
| Fast writes | ONE | ONE | Weak consistency, low latency |
| Balance | QUORUM | QUORUM | Strong consistency for quorum overlap |
| Strongest | ALL | ALL | Slowest, strongest |
| Local datacenter | LOCAL_QUORUM | LOCAL_QUORUM | Cross-datacenter not required |

**Read Repair Chance:** Default is 0.1 (10% of reads trigger read repair).
Higher values (0.5-1.0) converge data faster but consume more read I/O. Lower
values (0.01) reduce read overhead but increase the window of inconsistency.
For critical data, set to 1.0 for the first week after deployment, then
reduce.

**Hinted Handoff Duration:** Default is 3 hours. If a node is down longer than
this, hints are discarded and the node relies on read repair and anti-entropy
to catch up. Shortening this reduces storage for hints (which is small anyway)
but increases the recovery burden on anti-entropy. Extending beyond 3 hours
risks stale hints overwriting newer data.

## MVCC in Detail — How ACID Separates Readers and Writers

Multi-Version Concurrency Control is the mechanism that allows ACID databases
to provide isolation without blocking every reader. Understanding MVCC is
essential for understanding why ACID databases can achieve reasonable
throughput despite their locking overhead.

### How MVCC Works

Each row in an MVCC database can have multiple versions. When a transaction
updates a row, it creates a new version of the row rather than overwriting the
existing one. The old version remains available for other transactions that
started before the update.

**PostgreSQL implements MVCC as follows:**

1. Each row version stores `xmin` (the transaction ID that created it) and
   `xmax` (the transaction ID that deleted or updated it).
2. When a transaction reads a row, it checks the visibility rules: is `xmin`
   committed and visible to my snapshot? Is `xmax` after my snapshot start?
3. If the row is visible, the transaction sees that version. If not, it looks
   at the next older version in the chain.
4. Old versions accumulate in the table (dead tuples). A background process
   called VACUUM reclaims the space.

**Performance implications:**

- Readers never block writers. A `SELECT` can run concurrently with an
  `UPDATE` on the same row. The reader sees the old version.
- Writers do block writers. Two concurrent `UPDATE` statements on the same row
  cause one to wait for the other's commit or abort.
- Dead tuple bloat. Under high update rates, dead tuples accumulate faster
  than VACUUM can reclaim them. This is known as "bloat" and causes index
  scans to slow down because they must skip dead tuples.

### MVCC vs 2PL — Two Isolation Strategies

| Aspect | MVCC (PostgreSQL, Oracle) | 2PL (SQL Server, DB2) |
|---|---|---|
| Readers block writers | No — readers see old version | Yes — readers acquire shared locks |
| Writers block readers | No — readers see old version | Yes — exclusive lock blocks shared lock |
| Writers block writers | Yes — row lock | Yes — row lock |
| Storage overhead | Multiple versions per row | Single version per row |
| Vacuum required | Yes — dead tuple cleanup | No — no dead tuples |
| Best for | Read-heavy workloads with concurrent writes | Write-heavy workloads with short transactions |

### Snapshot Isolation vs Serializable

PostgreSQL's default isolation level is Read Committed, which uses snapshot
isolation per statement. Each statement within a transaction sees the database
as it existed at the start of that statement. This is fast but allows some
anomalies:

- **Non-repeatable reads:** A row read at the beginning of a transaction may
  differ from the same row read later if another transaction committed an
  update in between.
- **Phantom reads:** A query may return different sets of rows at different
  times within the same transaction.

Serializable isolation in PostgreSQL uses Serializable Snapshot Isolation
(SSI), which tracks read-write conflicts and aborts transactions that would
create serialization anomalies. SSI is more expensive because it must track
predicate-based conflicts (not just row-level conflicts).

## LSM Trees — The Storage Engine Behind BASE

While ACID databases typically use B-tree or B+tree structures for on-disk
storage, BASE databases like Cassandra and LevelDB use **Log-Structured Merge
Trees (LSM Trees)** . This choice is not an accident — it is driven by the
BASE philosophy of prioritizing write throughput over read latency.

### How LSM Trees Work

An LSM tree has three layers:

1. **Memtable (in-memory):** Writes land here first. The memtable is an
   in-memory sorted data structure (typically a skip list or red-black tree).
   No disk I/O is required for a write.

2. **SSTables (on-disk, sorted):** When the memtable reaches a threshold (for
   example, 64 MB), it is flushed to disk as an SSTable (Sorted String Table).
   SSTables are immutable — once written, they are never modified.

3. **Compaction (background):** Over time, multiple SSTables accumulate. A
   background compaction process merges overlapping SSTables into a single
   sorted file, discarding tombstones and stale versions.

### LSM Tree Performance

| Operation | B-Tree (ACID) | LSM Tree (BASE) |
|---|---|---|
| Write (in-memory) | Not applicable — writes go to disk pages | Instant — memtable is RAM |
| Write (full) | ~2ms (fsync + page write) | ~0.3ms (memtable + commit log) |
| Read (point query) | ~0.5ms (B-tree traversal) | ~1-5ms (check all SSTables) |
| Range scan | Fast — B-tree is sorted | Variable — depends on SSTable overlap |
| Space amplification | Low — in-place updates | High — multiple SSTable versions |
| Write amplification | Low (1x to 2x) | High (10x to 30x depending on compaction) |

The key trade-off: LSM trees sacrifice read performance (must check multiple
SSTables) and use more disk space (multiple versions) in exchange for much
higher write throughput.

### Compaction Strategies

**Size-Tiered Compaction (STCS):** Used by default in Cassandra. When a table
has N SSTables of similar size, they are compacted into one larger SSTable.
This is simple but creates larger and larger SSTables over time, causing read
latency to increase.

**Leveled Compaction (LCS):** Each level (L0, L1, L2, ...) has a fixed size
limit. SSTables in L1 are the same size. Compaction moves data from L0 to L1
and merges overlapping L1 SSTables into L2. This bounds read latency (at most
N levels to check) at the cost of more write amplification.

**Time-Window Compaction (TWCS):** Used for time-series data. SSTables are
grouped by time window (for example, one file per hour). Compaction only
merges SSTables within the same window. Old windows are dropped via TTL.

## Isolation Anomalies in Detail

Understanding specific isolation anomalies helps you decide which consistency
model and isolation level to use.

### Dirty Read (Prevented by Read Committed)

A dirty read occurs when a transaction reads a value that another transaction
has written but not yet committed. If the other transaction aborts, the first
transaction has read a value that never existed.

```
Transaction A                  Transaction B
UPDATE accounts SET balance    (no operation)
  = balance - 100 WHERE id=1
                               SELECT balance FROM accounts WHERE id=1
                               -- reads 900 (uncommitted!)
ROLLBACK                       -- A aborts, balance is 1000
                               -- but B saw 900, which never existed
```

ACID databases at Read Committed prevent dirty reads by showing only committed
data. BASE systems do not have this concept — there is no transaction boundary.
A read can return any value that any replica currently has.

### Lost Update (Prevented by Serializable)

A lost update occurs when two transactions read the same value, modify it, and
write it back. The second write overwrites the first without incorporating its
change.

```
Transaction A                  Transaction B
SELECT balance WHERE id=1
  -- reads 1000
                               SELECT balance WHERE id=1
                                 -- reads 1000
UPDATE SET balance = 1000 - 100
  WHERE id=1
                               UPDATE SET balance = 1000 - 50
                                 WHERE id=1
COMMIT
                               COMMIT
-- Balance is 950, should be 850
-- A's withdrawal of 100 is lost
```

This can happen at Read Committed isolation. Serializable isolation prevents
it by using predicate locks or SSI. In BASE systems, lost updates are a
feature of LWW conflict resolution — the last write wins, and earlier writes
are lost.

### Write Skew (Prevented by Serializable)

Write skew is a subtle anomaly where two transactions read overlapping data
and write disjoint data, but the result violates a constraint. This is the
most common anomaly that catches developers off guard at Repeatable Read.

```
-- Constraint: at least one doctor must be on call
-- Transaction A: Dr. Smith goes off call
SELECT COUNT(*) FROM on_call WHERE shift='night'
  -- reads 2 (Smith and Jones)
UPDATE on_call SET on_call=false WHERE doctor='Smith'

-- Transaction B: Dr. Jones goes off call (concurrent)
SELECT COUNT(*) FROM on_call WHERE shift='night'
  -- reads 2 (Smith and Jones) — A's update not visible yet
UPDATE on_call SET on_call=false WHERE doctor='Jones'

-- Both commit. Now 0 doctors are on call. Constraint violated.
```

Only Serializable isolation prevents write skew. Read Committed and Repeatable
Read allow it. In BASE systems, write skew is not prevented — it must be
detected at reconciliation time or prevented by application logic.

## The Commit Log in BASE Systems

While BASE systems do not use a WAL in the same way as ACID systems, they do
have a commit log. Understanding the commit log helps explain why BASE systems
are durable despite not fsyncing on every write.

In Cassandra, the commit log is a crash-recovery mechanism:

1. Every write is appended to the commit log on disk (sequential append, no
   fsync required by default).
2. The write is also added to the memtable in memory.
3. If the node crashes, the commit log is replayed on restart to reconstruct
   the memtable.
4. Once the memtable is flushed to an SSTable, the corresponding commit log
   segments can be deleted.

The commit log uses sequential I/O, which is fast even without fsync. If the
node crashes before the commit log is flushed, the last few writes (up to the
`commitlog_sync_period_in_ms` window, default 10 seconds) are lost. This is
the "soft state" property — data loss of a few seconds is acceptable.

`fsync` on the commit log is configurable:

- `periodic` (default): Sync every 10 seconds. Higher throughput, possible 10
  seconds of data loss on crash.
- `batch`: Sync after each write. Lower throughput, zero data loss on crash
  (within a single node).

This configuration is the operational knob that trades durability for
throughput in BASE systems — the same trade-off that ACID databases make with
their `fsync` parameter.

## Practical Diagnostics — How to Measure Consistency

In a production system, you need to measure whether your consistency guarantees
are being met. Here are practical diagnostics for each model.

### ACID Diagnostics

**Transaction conflict rate:** Monitor the rate of serialization failures
(`40001` in PostgreSQL). A rate above 1% indicates that your transactions are
contending too much. Reduce isolation level or redesign access patterns.

**Deadlock count:** Track `deadlocks` in `pg_stat_database`. More than a few
per hour indicates a lock ordering problem. Fix by acquiring locks in a
consistent order across all transactions.

**Replication lag:** In PostgreSQL, monitor `pg_stat_replication` for the
`write_lag`, `flush_lag`, and `replay_lag` values. Lag above 30 seconds during
normal operation indicates network or I/O issues on replicas. A synchronous
replica with lag above 100ms is dangerously slow.

### BASE Diagnostics

**Read repair counts:** In Cassandra, `nodetool cfstats` shows read repair
statistics. A high read repair rate (more than 1% of reads) indicates that
replicas are frequently out of sync. Run a full repair and investigate the
cause — likely a recently down node or network issues.

**Hinted handoff queue:** `nodetool hints` shows the number of pending hints.
A growing queue indicates that nodes are persistently down. Set an alert at
10,000 pending hints per node.

**Gossip convergence:** `nodetool gossipinfo` shows the gossip state of each
node. If nodes show differing `LOAD` or `SCHEMA` values for the same peer,
gossip is not converging. This indicates a network partition or a high rate of
membership changes.

**Compaction backlog:** Monitor the number of pending compactions. A growing
backlog indicates that writes are arriving faster than compaction can keep up.
This causes read performance to degrade as more SSTables accumulate.
