# ACID vs BASE — Interview Playbook

The real-world page showed how companies apply these concepts. This page
prepares you to demonstrate mastery in a system design interview. The ACID vs
BASE distinction appears consistently at FAANG-level interviews, often as a
critical decision point in a larger design.

---

## How This Topic Appears

Interviewers rarely ask "Explain ACID vs BASE" directly (though that happens
at L3/L4). The topic surfaces as a decision point within a system design
problem. Here are the most common phrasings:

### Direct Questions

1. "Explain the difference between ACID and BASE databases."
2. "When would you choose a BASE database over an ACID database? Give an
   example."
3. "What does 'eventual consistency' mean in practice? How do you design for
   it?"
4. "Explain the CAP theorem. Where do ACID and BASE fit in it?"
5. "What are the failure modes of eventually consistent systems?"

### Design Questions With ACID/BASE Decisions

6. "Design the data layer for an e-commerce platform. Which parts need ACID and
   which can be BASE?"
7. "Design a ride-sharing service. How do you handle the payment flow vs the
   trip history storage?"
8. "Design a social media news feed. How do you store millions of posts per
   minute and serve them instantly?"
9. "Design a collaborative document editing system. How do you handle
   concurrent edits?"
10. "Design a global online ticketing system. How do you prevent double-selling
    while serving users worldwide?"
11. "Design an inventory management system. Can you use a BASE store for
    inventory counts?"
12. "Design a payment processing pipeline. Can any part of this be eventually
    consistent?"

---

## What the Interviewer Is Testing

When an interviewer brings up ACID vs BASE, they are assessing:

1. **Do you understand the fundamental trade-off?** The interviewer wants to
   hear that ACID and BASE are a trade-off between consistency and
   availability, not a ranking (one is not better than the other).

2. **Can you map requirements to the right model?** Given a specific data
   domain (payments, sessions, analytics, inventory), can you explain which
   model fits and why?

3. **Do you understand the cost of each choice?** A candidate who says "always
   use ACID" or "always use BASE" fails this test. The correct answer is "it
   depends" followed by concrete reasoning.

4. **Can you design around the chosen model's weaknesses?** If you choose BASE
   for a shopping cart, can you handle stale reads? Can you detect and resolve
   conflicts? If you choose ACID for a high-write workload, do you have a plan
   for scaling?

5. **Do you have hands-on knowledge?** Mentioning specific details — Cassandra
   tombstones, PostgreSQL MVCC, DynamoDB strongly-consistent reads cost,
   CockroachDB Raft transactions — signals real experience versus theoretical
   knowledge.

## Level Expectations

### L3 / Entry Level

**Expectation:** Define ACID and BASE, list the four ACID properties, explain
the basic differences.

**Sample question:** "What is the difference between ACID and BASE?"

**Sample answer:**
> ACID stands for Atomicity, Consistency, Isolation, and Durability. These are
> properties of relational databases like PostgreSQL. Every transaction is
> all-or-nothing (atomicity), the database is always in a valid state
> (consistency), concurrent transactions do not interfere (isolation), and
> committed data survives crashes (durability).
>
> BASE stands for Basically Available, Soft state, and Eventual consistency.
> These are properties of distributed databases like Cassandra. The system
> accepts writes at any node (availability), the state changes as replication
> catches up (soft state), and data converges over time (eventual consistency).
>
> ACID guarantees correctness at the cost of availability under partition. BASE
> guarantees availability at the cost of temporary inconsistency.

### L4 / Mid Level

**Expectation:** Explain trade-offs, give examples of when to use each, discuss
the CAP theorem relationship.

**Sample question:** "Design the data storage for a social media platform. Which
data needs ACID and which can be BASE?"

**Sample answer:**
> A social media platform has several data categories with different consistency
> requirements:
>
> *User profiles* need strong consistency for account details (email, password
> hash) but can use BASE for display preferences and bio text. *Posts and
> feeds* are a clear BASE use case — eventual consistency is fine because
> users expect a brief delay for content to propagate. *Direct messages* need
> read-your-writes consistency but not full ACID — Cassandra with QUORUM
> consistency works. *Ad impressions and billing* need ACID — every impression
> must be recorded accurately for billing.
>
> The platform would use PostgreSQL for user authentication and billing,
> Cassandra or DynamoDB for posts and feeds, and possibly a combination with
> CDC between the systems.

### L5 / Senior Level

**Expectation:** Discuss conflict resolution strategies, anti-patterns,
cross-system consistency patterns (SAGA, CQRS), operational concerns.

**Sample question:** "How would you design a global e-commerce inventory system
that avoids selling the same item twice?"

**Sample answer:**
> I would use a hybrid approach. Inventory reservations must be strongly
> consistent — two customers cannot buy the same item. But inventory queries
> (checking if an item is in stock) can be eventually consistent.
>
> For the reservation path, I would use an ACID database like PostgreSQL with a
> single primary per region. Cross-region inventory for the same SKU would use
> a two-phase approach: reserve locally, then confirm globally using a SAGA.
> If the global confirmation fails (item sold in another region), the local
> reservation is rolled back.
>
> For the query path, I would use a BASE store (Redis or Cassandra) that
> receives CDC events from the ACID inventory system. Most reads go to the
> BASE store. When a customer tries to purchase, the checkout service reads
> from the ACID store for the final check.
>
> Conflict scenarios: if two regions attempt to reserve the last unit of an
> item simultaneously, the system uses a last-write-wins policy at the
> transaction level. One reservation succeeds and the other is compensated
> with a notification to the customer.

### L6 / Staff Level

**Expectation:** Discuss trade-offs with real systems experience, cost analysis,
migration strategies, organizational impact.

**Sample question:** "You are leading a migration from a single MySQL database
to a polyglot persistence architecture. How do you approach it?"

**Sample answer:**
> The priority is zero downtime. The migration proceeds in phases:
>
> Phase 1: Identify data domains by consistency requirements. Session data
> moves first (low risk, pure BASE). Analytics and logs move second (no
> cross-row constraints). Profiles and catalog move third (read-heavy, BASE
> suitable). Payments and orders move last (ACID stays in MySQL for the
> longest time).
>
> Phase 2: Implement CDC using Debezium. MySQL WAL streams changes to Kafka.
> The new BASE stores consume these events to build initial state. Dual-write
> during this phase: write to MySQL and the new store. Compare results for
> correctness verification.
>
> Phase 3: Switch reads. Configure API gateways to read from the new BASE
> stores. Fall back to MySQL if the new store returns an error. Monitor read
> latency and error rates. Keep this running for 2-4 weeks to verify.
>
> Phase 4: Switch writes. Stop writing new data to MySQL for the migrated
> domains. Keep MySQL as an archive. Run comparison jobs to ensure the new
> stores are correct.
>
> The cost of this migration is approximately 6-8 engineering months for the
> core team. The benefit is 10x higher write throughput and lower latency for
> read-heavy domains.

### L7 / Principal Level

**Expectation:** Discuss industry trends, influence database selection across
the organization, define data governance policies.

**Sample question:** "How would you set data consistency policies for a large
organization with dozens of microservices?"

**Sample answer:**
> I would establish three tiers of consistency requirements:
>
> Tier 1 (ACID required): financial transactions, user authentication, legal
> compliance records, inventory counts. These must use ACID databases with
> strict isolation. Any BASE use in these domains requires an architectural
> review and sign-off from a principal engineer.
>
> Tier 2 (Tunable consistency): user profiles (non-auth fields), order history,
> product catalog. These can use ACID or BASE with at least QUORUM
> consistency. Stale reads are bounded to 1 second.
>
> Tier 3 (BASE acceptable): analytics, logs, activity feeds, recommendations,
> session data. These can use BASE with ONE consistency. Stale reads within
> minutes are acceptable.
>
> Each service onboards to this framework through a data governance review.
> The review evaluates: (1) cost of inconsistency, (2) latency requirements,
> (3) write throughput, (4) regulatory constraints. The resulting architecture
> uses ACID for the source of truth and BASE for derived views, with CDC as
> the bridge.

## Phrase Bank

Using precise terminology signals depth. Incorporate these phrases naturally.

### Introducing the Topic

- "ACID and BASE represent a fundamental trade-off between consistency and
  availability — one is not strictly better than the other."
- "The choice between ACID and BASE depends on the consistency requirements of
  each data domain, not the database as a whole."
- "ACID provides strong guarantees but limits horizontal scalability. BASE
  scales horizontally but requires handling eventual consistency."

### Discussing ACID

- "The WAL fsync is the bottleneck — group commit reduces this to one fsync
  per batch."
- "Serializable isolation uses predicate locks or SSI. It guarantees no
  anomalies but can limit throughput."
- "MVCC allows readers to see a consistent snapshot without blocking writers.
  This is how PostgreSQL achieves high concurrency."
- "Deadlock detection in 2PL is typically periodic — the database builds a
  wait-for graph and aborts the youngest transaction in a cycle."

### Discussing BASE

- "Quorum-based reads and writes with R + W > N provide strong consistency
  without full coordination."
- "Vector clocks detect concurrent updates, but the application must handle
  sibling versions."
- "CRDTs are the most elegant solution — they merge automatically without
  conflict resolution logic."
- "The gossip protocol converges in O(log N) rounds, but during a partition,
  the cluster can split into two independent groups."
- "Read repair fixes stale reads at read time, but it does not prevent read
  staleness during high write throughput."

### Making the Trade-Off

- "For this data domain, the cost of inconsistency is [X] and the cost of
  unavailability is [Y]. [X] is higher, so we choose ACID."
- "The inventory system must be strongly consistent for the final check, but
  the browseable catalog can be eventually consistent."
- "We use an outbox pattern to bridge the ACID order system to the BASE
  analytics pipeline."
- "The SAGA pattern handles multi-step transactions across ACID and BASE
  systems, but it requires compensating actions for every step."

## Red Flags

Avoid these mistakes in your answer.

1. **"ACID is always better."** This indicates you have not experienced the
   scalability limitations of single-node databases. The interviewer will
   challenge with a high-scale problem.

2. **"BASE is always better."** This indicates you have not dealt with the
   complexity of eventual consistency or the business impact of inconsistent
   data.

3. **"Eventual consistency means the data is eventually correct."** No —
   eventual consistency means the system converges, but it does not guarantee
   the converged value is the value any particular client wrote.

4. **Ignoring CAP theorem positioning.** Every ACID vs BASE decision is a CAP
   theorem trade-off. Not mentioning it signals a gap in fundamentals.

5. **No concrete numbers.** "ACID is slower" is vague. "ACID databases handle
   roughly 1,000-5,000 writes per second per node on SSD, while BASE systems
   handle 50,000-100,000" is specific and shows experience.

6. **Treating consistency as binary.** Consistency is a spectrum. Mentioning
   tunable consistency levels (ONE, QUORUM, ALL) or bounded staleness shows
   nuance.

7. **Ignoring conflict resolution.** If you propose a BASE system without
   explaining how conflicts are detected and resolved, the interviewer will
   assume you have not operated a BASE system at scale.

8. **Forgetting operational concerns.** A good system design includes
   monitoring, failover, and recovery. Mentioning hinted handoff monitoring,
   gossip convergence checks, and read repair storms shows operational depth.

9. **"ACID and BASE are mutually exclusive."** Many production systems use
   both, bridged by CDC or CQRS. A mature answer shows how they coexist.

10. **No mention of the PACELC theorem.** PACELC extends CAP: if a partition
    occurs (P), trade off A vs C. Otherwise (Else, E), trade off Latency (L)
    vs Consistency (C). ACID systems choose C over L (strong consistency adds
    latency). BASE systems choose L over C (lower latency, weaker consistency).
    Mentioning PACELC at L6+ signals deep distributed-systems knowledge.

## Live Coding Exercise — ACID vs BASE Decision

In some interviews, you will be asked to write a short design document or
pseudocode that demonstrates your ACID vs BASE reasoning. Below is a template
you can adapt.

### Problem Statement

"You are designing the data layer for a customer support ticketing system. Each
ticket has an ID, customer ID, status (open, in-progress, resolved, closed),
priority, assignee, and a conversation thread. The system handles 50,000 new
tickets per day and 500,000 updates per day. Support agents work in shifts
globally.

Which data should use ACID and which should use BASE? Write a design summary."

### Template Answer

```python
# ACID vs BASE Decision — Customer Support Ticketing
# ===================================================

# DATA DOMAIN MAPPING

# ACID (PostgreSQL)
# - Ticket status and assignment (must be strongly consistent)
# - SLA tracking (resolution time must be accurate)
# - Agent shift assignment (no double-booking)
# Reason: Status changes have real-world consequences.
#   A ticket marked "resolved" must not appear as "open" to another agent.
#   SLA violation calculations must be based on accurate timestamps.

# BASE (Cassandra)
# - Conversation thread (append-only messages per ticket)
# - Ticket search index (Elasticsearch)
# - Agent activity log (who updated what, when)
# Reason: Conversation messages are append-only, no cross-message constraints.
#   Search can tolerate indexing lag of 1-2 seconds.
#   Activity logs are never read back in real-time decision paths.

# HYBRID (PostgreSQL + CDC to Cassandra)
# - Ticket metadata changes streamed to Elasticsearch for search
# - Ticket status changes streamed to analytics dashboard

# CONSISTENCY REQUIREMENTS PER OPERATION

def update_ticket_status(ticket_id, new_status, agent_id):
    """
    ACID operation: status change must be atomic.
    Uses PostgreSQL with Read Committed isolation.
    """
    with postgres.transaction():
        # Lock the ticket row to prevent concurrent updates
        ticket = postgres.execute(
            "SELECT * FROM tickets WHERE id = %s FOR UPDATE",
            ticket_id
        )
        # Validate transition (e.g., cannot go from resolved to open)
        if not valid_transition(ticket["status"], new_status):
            raise InvalidTransitionError(ticket["status"], new_status)
        # Update status
        postgres.execute(
            "UPDATE tickets SET status = %s, "
            "updated_at = NOW(), updated_by = %s "
            "WHERE id = %s",
            new_status, agent_id, ticket_id
        )
    # After commit, CDC streams the update to Elasticsearch
    return ticket


def add_conversation_message(ticket_id, author, message):
    """
    BASE operation: messages are append-only, eventual consistency is fine.
    Uses Cassandra with QUORUM write for read-your-writes guarantee.
    """
    message_id = uuid4()
    cassandra.execute(
        """
        INSERT INTO conversation_messages
            (ticket_id, message_id, author, message, created_at)
        VALUES (%s, %s, %s, %s, NOW())
        """,
        ticket_id, message_id, author, message,
        consistency_level=ConsistencyLevel.QUORUM
    )
    return message_id


def search_tickets(search_query):
    """
    BASE read: search results can be 1-2 seconds stale.
    Reads from Elasticsearch, which is CDC-fed from PostgreSQL.
    """
    return elasticsearch.search(
        index="tickets",
        body={"query": {"multi_match": {"query": search_query}}}
    )


# TRADE-OFFS ACCEPTED

# 1. Status change latency: ~5ms (fsync). Acceptable for 50K updates/day.
# 2. Search staleness: up to 2 seconds. Acceptable — agents expect
#    near-real-time, not fully real-time.
# 3. Message consistency: if a node fails, the last 1-2 seconds of
#    conversation messages may be lost. Acceptable — agents can retype.
# 4. No cross-region ACID: tickets are sharded by region. A global ticket
#    search uses Elasticsearch cross-cluster search.
```

## Mock Interview Simulation

Below is a complete transcript of a mock interview question and a model answer.
Use this to practice your delivery.

### Interviewer Question

"I'd like to design the data layer for a collaborative document editing service
like Google Docs. Users create documents, share them with other users, and
multiple users edit the same document simultaneously. How would you think
about ACID vs BASE for this system?"

### Model Answer

**Candidate:** Let me break this down into data domains with different
consistency requirements.

**Domain 1: Document metadata (title, owner, sharing permissions)**

This must be strongly consistent. If two users change sharing permissions at
the same time, the system must converge to a single correct state. An
eventual consistency window here could expose a document to unauthorized
users or lock out authorized users.

I would use ACID, specifically PostgreSQL. The document metadata table has a
row per document. Writes are infrequent (changes to title or permissions
happen once per minute per document at most). ACID guarantees are cheap at
this write volume.

**Domain 2: Document content (the actual text)**

This is the hard part. Google Docs supports real-time collaboration — 100+
users editing the same paragraph simultaneously. ACID locking would serialize
these edits, causing 100ms+ latency per keystroke. That is unacceptable for a
real-time experience.

I would use a CRDT-based approach, which is a form of BASE. Each user's
edits are independent operations (insert character, delete character, format
range). CRDTs merge these operations automatically without conflicts.

The CRDT state is stored in memory for active documents and persisted to a
BASE store (Cassandra) for crash recovery. The BASE store accepts the CRDT
state with eventual consistency — if a server crashes and the last 2 seconds
of edits are lost, only one user's keystrokes are affected and they can retype.

**Domain 3: Document revision history**

Each save creates a revision. Revisions are append-only — once created, they
are never modified. This maps naturally to a BASE store. I would use S3 for
the revision blobs and Cassandra for the revision metadata (document ID,
revision number, timestamp, author).

**Domain 4: Real-time cursor positions**

Each user's cursor position is ephemeral and updated at high frequency (every
100ms). This must be fast and does not need durability. I would use Redis
with TTL (expire cursor positions after 5 seconds of no updates). This is
pure BASE — eventual consistency is inherent because cursor positions are
snapshots that become stale immediately.

**Domain 5: Notifications (user added, comment added)**

Notifications are best-effort. They can be lost without material impact. I
would use Kafka for the event stream and a BASE store for the notification
inbox. The notification inbox can lag by seconds.

**ACID/BASE boundary:**

The key boundary is between document metadata (ACID) and document content
(BASE). Metadata changes are infrequent and must be consistent. Content
changes are high-frequency and must be low-latency. The CRDT approach bridges
this gap: the CRDT algorithm guarantees convergence without ACID coordination.

**Trade-offs accepted:**

1. Content conflicts: If two users make incompatible edits (one deletes a
   paragraph while the other edits it), CRDTs produce a merged result that may
   not be semantically correct. Google Docs solves this with operational
   transformation, a different algorithm that preserves intent. The trade-off
   is algorithmic complexity.

2. Data loss window: If a server crashes before persisting CRDT state to
   Cassandra, the last 1-2 seconds of edits from the user connected to that
   server are lost. Acceptable — the user can retype.

3. Metadata consistency: If the PostgreSQL metadata is unavailable, users
   cannot open documents or change permissions. They can still edit content if
   they have the document loaded (the CRDT state is in memory). This is an
   acceptable degradation path.

### Interviewer Follow-Up

"How do you handle the case where a user edits a document offline and then
comes back online?"

**Candidate:** That is a conflict resolution scenario. The offline edits are
stored locally as CRDT operations. When the user comes online, the local
operations are sent to the server and merged with the server's CRDT state.

If the CRDT merge produces a semantically correct result, great — the merge is
transparent. If not (for example, the user deleted a sentence that another user
had already deleted), the result is shown to both users and they can
reconcile. This is a BASE approach: accept writes offline, merge on reconnect,
and accept temporary divergence.

The metadata (sharing permissions) cannot be changed offline. That requires
ACID and a connection to the server. Offline permission changes are queued and
rejected if they conflict.

## Key Interview Frameworks at a Glance

### CAP-PACELC Decision Matrix

When asked about consistency vs availability, reference both CAP and PACELC:

| Situation | PACELC Choice | Example |
|---|---|---|
| Partition occurs (P) | Choose A (lose C) | Social media feed — stay up, serve stale data |
| Partition occurs (P) | Choose C (lose A) | Payment processing — reject writes rather than double-charge |
| No partition (EL) | Choose C (lose L) | Metadata updates — wait for sync replica ack |
| No partition (EL) | Choose L (lose C) | Session reads — serve from nearest replica, accept staleness |

### Consistency Spectrum Reference

Memorize the consistency spectrum in order from strongest to weakest:

1. **Strict/Linearizable:** All operations appear atomic and instantaneous.
   All replicas agree on a single total order. (Spanner with TrueTime)
2. **Sequential:** Operations from each client appear in program order. The
   interleaving across clients is consistent with some total order. (CockroachDB)
3. **Causal:** Causally-related operations are seen in order by all clients.
   Concurrent operations may be seen in different orders. (Riak, DynamoDB with
   version vectors)
4. **Read-Your-Writes:** A client always reads its own writes. (Cassandra with
   QUORUM writes and token-aware routing)
5. **Monotonic Reads:** Once a client reads a value, all subsequent reads
   return a value at least as recent. (Cassandra with MONOTONIC reads)
6. **Eventual:** All replicas converge over time. No ordering guarantees.
   (Cassandra with CL=ONE, DynamoDB default)
