# ACID vs BASE — Challenges

## Tier 1 — Foundation

These challenges test whether you understand the basic properties and can
classify systems correctly.

### Challenge 1.1: Compare and Classify

For each of the following statements, identify whether it describes an ACID
property or a BASE characteristic, and name the specific property or
characteristic.

1. "A transaction that fails partway through must roll back all changes."
2. "The database accepts writes even when some nodes are unreachable."
3. "Once a transaction is committed, it survives a power failure."
4. "Two transactions running at the same time do not see each other's partial
   results."
5. "A read may return a value that is several seconds old."
6. "The database state can change between queries even when no new input
   arrives."

<details>
<summary>Solution for Challenge 1.1</summary>

1. **ACID — Atomicity.** The all-or-nothing property. In PostgreSQL, this is
   implemented via the WAL: on abort, the WAL records are not committed; on
   replay, any incomplete transactions are rolled back.

2. **BASE — Basically Available.** The system responds to all requests
   regardless of node health. In Cassandra, the coordinator writes to available
   replicas and stores hints for unavailable ones.

3. **ACID — Durability.** Committed data is persisted to non-volatile storage.
   PostgreSQL guarantees this via WAL fsync on every commit.

4. **ACID — Isolation.** Concurrent transactions do not interfere. In
   PostgreSQL, the default isolation level (Read Committed) prevents dirty
   reads but allows non-repeatable reads. Serializable isolation prevents all
   anomalies.

5. **BASE — Eventual consistency.** Replicas converge over time. In Cassandra
   with consistency level ONE, a read might return data from a replica that
   has not yet received the latest write from another replica.

6. **BASE — Soft state.** The system state updates without external input as
   replication catches up. For example, when a Cassandra node that was
   partitioned receives hinted handoff writes, its state changes transparently.
</details>

### Challenge 1.2: CAP Theorem Positioning

Draw the CAP theorem triangle and place the following systems on it:

1. PostgreSQL (single primary, synchronous replicas)
2. Cassandra (RF=3, CL=ONE)
3. Cassandra (RF=3, CL=QUORUM for reads and writes)
4. CockroachDB (distributed ACID)
5. Redis (single node, append-only file enabled)

Explain why each system falls where it does.

<details>
<summary>Solution for Challenge 1.2</summary>

1. **PostgreSQL (single primary, synchronous replicas): CP.** The system is
   consistent (all reads see the same committed state) and partition-tolerant
   (the primary is available as long as it has quorum). During a partition
   that isolates the primary, writes are rejected — availability is sacrificed.

2. **Cassandra (RF=3, CL=ONE): AP.** Reads and writes target one replica. The
   system stays available during partitions (each side accepts reads and
   writes), but data may diverge — consistency is sacrificed.

3. **Cassandra (RF=3, CL=QUORUM): Stronger consistency, but still AP.** With
   R + W > N (read CL + write CL > replication factor), the system provides
   strong consistency when there is no partition. During a partition, the
   minority side may not achieve quorum, reducing availability. The system is
   designed as AP but with tunable consistency that approaches CP.

4. **CockroachDB (distributed ACID): CP.** CockroachDB uses Raft consensus.
   A majority of replicas (quorum) must agree for a write to succeed. During a
   partition, only the side with the majority can write. Availability is
   sacrificed for consistency.

5. **Redis (single node, append-only file): CA.** A single node is not
   partition-tolerant (it is one point of failure). In terms of the CAP
   theorem, a single-node system is CA — it is consistent and available, but
   it does not handle network partitions because there is no network.
</details>

### Challenge 1.3: Identify the Right Model

For each scenario, say whether ACID or BASE is more appropriate and explain your
reasoning in one paragraph.

1. Storing user authentication sessions (login tokens, expiry times)
2. Recording ad impressions for billing (every impression must be counted)
3. A leaderboard for a mobile game (top 100 players, updated in real time)
4. A product catalog for an e-commerce website (10 million products, read-heavy)
5. A voting system for a reality TV show (millions of votes, results announced
   weekly)

<details>
<summary>Solution for Challenge 1.3</summary>

1. **BASE.** Session data is ephemeral. Losing a session forces a user to log in
   again — a minor inconvenience. The high throughput of BASE (Redis handles
   100K+ operations per second) matches the write volume of session creation.
   ACID durability guarantees are wasted on data that expires in 30 minutes.

2. **ACID.** Ad impressions are the basis for billing. Every impression must be
   counted exactly once. An eventual consistency window could cause impressions
   to be missed or counted twice, leading to incorrect billing. ACID
   (specifically atomicity and durability) guarantees that each impression
   recording is all-or-nothing and persisted.

3. **BASE.** Leaderboard updates happen at high frequency. A temporary
   discrepancy (a player seeing their rank as 47 when it is actually 45) is
   acceptable. The BASE system's write throughput handles the update volume.
   Bounded staleness of 1-2 seconds keeps the leaderboard usable.

4. **BASE.** A product catalog is read-heavy with occasional writes (price
   changes, new products). Stale reads of a few seconds are acceptable — a
   customer seeing yesterday's price is not a crisis. Cassandra or DynamoDB
   provides the read throughput needed at scale.

5. **BASE.** The vote count is only finalized when the show announces the
   winner. During the voting window, eventual consistency is fine. The system
   must accept high-volume writes (millions of votes in a short window), which
   favors BASE. The final tally runs against an ACID snapshot for accuracy.
</details>

## Tier 2 — Design

These challenges require you to design a data layer that uses ACID and BASE
appropriately.

### Challenge 2.1: Social Media Platform

Design the data layer for a social media platform with 100 million monthly
active users. The platform stores:

- User profiles (name, bio, avatar URL, email)
- Posts (text, image URLs, timestamps)
- Post likes (user ID + post ID pairs)
- Friend/follow connections
- Direct messages (two-party conversations)
- News feed (aggregated posts from followed users)
- Ad campaigns (targeting rules, budget, spend tracking)

For each data type:
1. State whether it should use ACID or BASE
2. Name a specific database technology
3. Explain the consistency model and any trade-offs

<details>
<summary>Solution for Challenge 2.1</summary>

| Data Type | Model | Technology | Justification |
|---|---|---|---|
| User profiles (auth fields) | ACID | PostgreSQL | Email uniqueness, password hashes must be strongly consistent. |
| User profiles (display fields) | BASE | Cassandra | Bio, avatar, display name can be eventually consistent. |
| Posts | BASE | Cassandra or DynamoDB | Append-only per user. Partition by user ID, cluster by timestamp. Staleness of seconds is acceptable. |
| Post likes | BASE | Redis (counter) or Cassandra | High-write (millions of likes per minute). No cross-key constraints. |
| Friend/follow connections | ACID | PostgreSQL | Must prevent duplicate connections. Eventual consistency could create orphan friend requests. |
| Direct messages | BASE | Cassandra with QUORUM | Requires read-your-writes. Use QUORUM consistency to ensure sender sees their own message. |
| News feed | BASE | Redis (sorted sets) or Cassandra | Pre-computed feed. Generated asynchronously. Stale feed of minutes is acceptable. |
| Ad campaigns | ACID | PostgreSQL | Budget tracking requires strong consistency. Double-spending ad budget causes revenue loss. |

The key insight: the platform uses PostgreSQL for consistency-critical data
(authentication, friends, ad budgets) and Cassandra/DynamoDB/Redis for
high-throughput data (posts, likes, feeds, messages). CDC bridges the two.
</details>

### Challenge 2.2: Payment Processing System

Design the data layer for a payment processing system that handles $1 billion
in transactions per month. The system must:

- Authorize payments (check balance, reserve funds)
- Capture payments (transfer funds from authorization)
- Refund payments (reverse a capture)
- Reconcile with banks (match internal records with bank statements)
- Provide a dashboard showing transaction history
- Detect fraud (real-time scoring of transaction patterns)

Identify which components require ACID and which can use BASE. Explain your
boundary definitions.

<details>
<summary>Solution for Challenge 2.2</summary>

**Must be ACID:**

1. **Authorization + Capture + Refund flow:** These are financial transactions.
   Every authorization must reserve funds atomically. Every capture must
   convert a reservation to a charge without double-booking. Refunds must
   reverse the original transaction. Any inconsistency causes money movement
   errors. Use PostgreSQL with serializable isolation or a sharded MySQL setup.

2. **Ledger (general ledger):** Every debit must match a credit. The ledger is
   the single source of truth. Must use ACID. A BASE ledger would allow
   temporary imbalances that violate accounting principles.

3. **Fraud decisions:** A fraud decision (accept, reject, review) must be
   applied consistently. If one replica says "accept" and another says
   "reject", the system cannot function. Use strongly consistent storage.

**Can be BASE:**

1. **Transaction history (dashboard):** The dashboard can be eventually
   consistent by 1-2 seconds. Use CDC to stream from the ACID payment database
   to a BASE store (Cassandra or Elasticsearch). The dashboard reads from the
   BASE store. If a transaction appears 500ms late, this is acceptable.

2. **Fraud scoring models:** Fraud feature vectors (average transaction amount,
   velocity checks) can be computed on BASE data. Staleness of minutes is
   acceptable because fraud patterns do not change in seconds.

3. **Reconciliation data:** Bank statements arrive daily or weekly. Comparing
   internal records (ACID) with bank data is a batch process that does not
   need real-time consistency.

**Boundary:** The ACID database is the source of truth. All money-moving
operations must go through it. All other operations read from BASE stores that
are CDC-fed from the ACID source. The dashboard never reads directly from the
ACID database — this protects the ACID database from dashboard query load.
</details>

### Challenge 2.3: E-Commerce Platform (Mixed)

Design the data layer for an e-commerce platform similar to Amazon. It requires:

- Product catalog (millions of products, frequently updated prices)
- Shopping cart (per-user, persisted across sessions)
- Inventory counts (must not go negative)
- Order processing (create order, charge, ship)
- User reviews (text, ratings, timestamps)
- Recommendation engine (product affinity, "customers also bought")

Draw a diagram showing which data stores you would use and how data flows
between them.

<details>
<summary>Solution for Challenge 2.3</summary>

The architecture uses four storage systems with clear boundaries:

**PostgreSQL (ACID):** Inventory counts, order records, payment transactions.
Inventory must never go negative — ACID locking prevents this. Orders must be
all-or-nothing (reserve inventory + charge + ship).

**Cassandra (BASE):** Product catalog (read-heavy, eventual consistency fine),
user reviews (append-only per product, reads overwhelm writes), shopping cart
(persisted but not ACID-critical).

**Redis (BASE):** Session cache, real-time inventory display (denormalized
counts with acceptable staleness), recommendation cache (pre-computed
affinity scores).

**Elasticsearch (BASE):** Full-text product search, faceted navigation, review
search. Index updated via CDC from the product catalog.

Data flow:

1. User browses catalog. Reads go to Cassandra (product data) and Redis
   (inventory display count). Both are eventually consistent. The displayed
   inventory count might be slightly stale.

2. User adds item to cart. Write goes to Cassandra. If it fails, the cart
   item is lost — the user can re-add. This is acceptable.

3. User checks out. Checkout service reads from Cassandra (cart) and then
   performs the ACID transaction in PostgreSQL (reserve inventory, create
   order, charge). The inventory reservation uses `SELECT ... FOR UPDATE` to
   prevent double-selling.

4. After order creation, CDC streams the order event to Kafka. Cassandra
   receives the event and updates the user's order history. Elasticsearch
   indexes the order for search.

5. If the inventory reservation fails (item sold out), the entire order
   transaction aborts. The user is notified and the cart is preserved for
   alternative selection.
</details>

## Tier 3 — Expert

These challenges require deep understanding of distributed systems and

### Challenge 3.1: Global Ride-Sharing Platform

Design the data layer for a ride-sharing platform operating in 50 countries.
The platform handles:

1. **Rider requests:** A rider requests a ride. The request must be routed to
   the nearest available driver.
2. **Driver location:** Drivers send GPS coordinates every 3 seconds.
   Approximately 10 million coordinates per minute.
3. **Ride matching:** A ride request must be matched to exactly one driver.
4. **Payment processing:** Fare calculation, surge pricing, and driver payout.
5. **Trip history:** Past rides for riders and drivers.
6. **Fraud detection:** Real-time scoring of ride requests for fraud.
7. **Pricing model:** Dynamic surge pricing based on supply (available drivers)
   and demand (ride requests) in geographic cells.

The system must:
- Serve requests globally with p99 latency under 200ms for ride matching
- Never lose a payment or double-charge a rider
- Handle 1 million concurrent ride requests per hour during peak (NYE)
- Operate in regions with unreliable network connectivity

Design the data architecture with justification for ACID vs BASE choices.

<details>
<summary>Solution for Challenge 3.1</summary>

**Zone-Based Architecture:**

Divide the world into geographic zones (cells of approximately 1km x 1km).
Each zone operates semi-independently. This limits coordination to within a
zone and allows the system to scale horizontally.

**BASE Data (high throughput, eventual consistency acceptable):**

*Driver locations (Redis + Cassandra):* GPS coordinates arrive every 3 seconds
from 5 million drivers. This is 1.6 million writes per second. Redis provides
the sub-millisecond reads needed for real-time matching. Cassandra stores the
historical location stream for analytics. The current location in Redis can be
stale by 3 seconds — acceptable for matching. Use Redis with TTL (expire
location 10 seconds after last update).

*Ride requests (Redis + Kafka):* Requests are published to Redis sorted sets
(per-zone request queue) and streamed to Kafka for durability. If a request
is lost in Redis, the Kafka consumer replays it. Eventual consistency is
acceptable because the rider retries if no match is found.

*Trip history (Cassandra):* Past rides are append-only, read by riders and
drivers infrequently. Cassandra's partition-by-user model makes queries fast.
TTL-based expiry automatically purges data older than the retention policy.

*Fraud scoring (Redis + Cassandra):* Feature vectors (frequency of requests,
distance from typical pickup, payment method velocity) are computed from BASE
data. Staleness of seconds does not affect fraud detection effectiveness.

**ACID Data (strong consistency required):**

*Payment processing (PostgreSQL, per-region):* Each geographic region has an
ACID database for payments. A ride is priced, charged to the rider, and the
driver payout is recorded in a single ACID transaction. This guarantees that
no money is lost. Cross-region payments (a rider from the US taking a ride in
Japan) are handled by the payment gateway, which buffers and retries.

*Surge pricing state (PostgreSQL):* The current surge multiplier for each zone
must be consistent. If two pricing servers compute different multipliers, the
system behaves unpredictably. Use ACID with a single writer per zone.

*Ride matching commitment (PostgreSQL):* When a match is made, the system must
atomically assign the driver, charge the rider's payment method, and record
the match. This is a three-part ACID operation. The match itself is fast
(Redis-based), but the commitment is ACID (PostgreSQL).

**Network Partition Handling:**

In regions with unreliable connectivity, the architecture degrades gracefully:

1. Each zone runs independently. If the zone's PostgreSQL is unreachable, the
   zone can still accept ride requests and store them in a local buffer
   (Cassandra). When connectivity is restored, the buffered requests are
   processed and matched.

2. Driver payouts use offline reconciliation. If a driver completes a ride
   during a network outage, the app records the trip locally and submits it
   when connectivity returns. The ACID payment system validates and processes
   the submitted trip.

3. Surge pricing falls back to a default multiplier during connectivity loss.
   This prevents inconsistent pricing across zones.

**Capacity Planning:**

- Driver location writes: 1.6M writes/sec. Redis cluster of 20 nodes handles
  this. Each node handles 80K writes/sec.
- Ride matching reads: 1M requests/hour peak = 278 requests/sec. Each request
  reads the nearest drivers from Redis (sorted set by GPS) and checks the
  match status in PostgreSQL.
- Payment transactions: 1M rides/hour = 278 transactions/sec. Single-region
  PostgreSQL with 64 cores handles this with group commit.
</details>

### Challenge 3.2: Migrating from Single ACID to Hybrid Without Downtime

Your company runs a monolithic e-commerce platform on a single PostgreSQL
database. The database handles 10,000 writes per second and is starting to hit
its limits. The CTO approves a 6-month migration to a polyglot architecture
using ACID (PostgreSQL) for core transactions and BASE (Cassandra, Redis) for
catalog, cart, sessions, and analytics.

Design the migration plan. It must:

- Have zero downtime (no maintenance windows)
- Allow rollback at any phase
- Validate data correctness continuously
- Provide a timeline with phases

<details>
<summary>Solution for Challenge 3.2</summary>

**Phase 1: Instrumentation and Analysis (Weeks 1-3)**

Deploy comprehensive monitoring on the existing PostgreSQL instance:
- Identify read and write throughput by table
- Identify which tables have multi-row constraints
- Map each table to an ACID or BASE target
- Set up Debezium CDC for PostgreSQL WAL streaming to Kafka

**Milestone:** A documented data domain map. Each table is tagged:
- CORE (ACID): orders, payments, inventory, users (auth fields)
- DERIVED (BASE): catalog, reviews, cart, sessions, user profiles (display)

**Phase 2: Build BASE Stores for DERIVED Data (Weeks 4-8)**

Deploy Cassandra, Redis, and Elasticsearch clusters. These run in parallel
with the existing PostgreSQL. No traffic is routed to them yet.

Set up CDC streams from PostgreSQL to each BASE store:
- Catalog tables → Cassandra
- Session tokens → Redis (with TTL)
- Reviews → Cassandra
- Cart data → Cassandra

Run validation scripts: compare data in BASE stores with the source PostgreSQL
data every hour. Alert on discrepancies.

**Milestone:** BASE stores exist and are CDC-synchronized within 5 seconds of
the PostgreSQL source. Validation scripts show zero discrepancies for 7 days.

**Phase 3: Switch Reads (Weeks 9-14)**

Configure the application to read from BASE stores for DERIVED data. The
application code uses a circuit breaker pattern:

```python
def get_catalog_item(item_id):
    try:
        # Read from Cassandra first
        item = cassandra_client.get(f"catalog:{item_id}")
        if item:
            return item
    except Exception:
        pass  # Fall through to PostgreSQL
    # Fallback to PostgreSQL
    return postgres_client.query(
        "SELECT * FROM catalog WHERE id = %s", item_id
    )
```

This phase runs for 4 weeks. Monitor:
- Read latency: should drop from ~5ms (PostgreSQL) to ~2ms (Cassandra)
- Error rate: should be zero (Cassandra cluster healthy)
- Data consistency: CDC validation continues running

**Rollback point:** Remove the Cassandra code path. Reads go back to
PostgreSQL. Zero impact — the Cassandra cluster was never the sole source.

**Phase 4: Switch Writes for DERIVED Data (Weeks 15-20)**

For DERIVED data, modify the application to write to both PostgreSQL and the
BASE store simultaneously (dual-write). Use a write-through pattern:

```python
def save_cart(user_id, cart_data):
    # Write to BASE (fast)
    cassandra_client.set(f"cart:{user_id}", cart_data)
    # Write to PostgreSQL (slow, backward-compatible)
    postgres_client.execute(
        "INSERT INTO carts (user_id, data) VALUES (%s, %s) "
        "ON CONFLICT (user_id) DO UPDATE SET data = %s",
        user_id, cart_data, cart_data
    )
```

After 2 weeks of dual-write with zero validation errors, remove the PostgreSQL
write for DERIVED data. The BASE store is now the source of truth for DERIVED
data. CDC from PostgreSQL to BASE stops — it is now PostgreSQL that has stale
data.

**Phase 5: Optimize CORE Data (Weeks 21-24)**

The CORE data (orders, payments, inventory) stays in PostgreSQL. Optimize:

1. Shard PostgreSQL by region or by customer ID range (for cross-region
   deployments).
2. Move historical order data to a separate archival store (S3) to reduce the
   active dataset size.
3. Increase checkpoint frequency and WAL size for higher write throughput.

**Rollback point for Phase 4:** The PostgreSQL write path is still present in
the codebase (though not active). Re-enabling it is a configuration change.

**Phase 6: Decommission (Week 24+)**

Remove DERIVED tables from PostgreSQL schema. Deallocate storage. The
PostgreSQL instance is now dedicated to CORE data only, with reduced load and
improved performance.

**Total timeline:** 6 months. Zero downtime. Rollback available at every phase.
</details>
