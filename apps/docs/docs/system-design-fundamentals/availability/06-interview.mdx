# Availability in System Design Interviews

## Core Questions Interviewers Ask

### Question 1: How would you design a highly available system for X?

This is the most common framing. The interviewer gives you a system (URL shortener, chat app, e-commerce platform, payment system) and asks you to design it with high availability in mind.

**What they are testing**:
- Do you instinctively identify SPOFs?
- Do you know the difference between active-passive and active-active?
- Can you calculate availability requirements from business requirements?
- Do you consider the operational complexity of HA mechanisms?

### Question 2: How do you handle a database failure?

**What they are testing**:
- Read replicas, primary failover, replication lag
- Connection pool management during failover
- Data loss scenarios (async vs sync replication)
- Read-your-writes consistency after failover

### Question 3: How would you ensure 99.99% uptime?

**What they are testing**:
- Can you translate "four nines" into concrete engineering decisions?
- Do you understand the cost/benefit of each nine?
- Multi-region architecture, DNS, failover timing

### Question 4: Describe a time you handled a production outage.

**What they are testing**:
- Incident response process
- How you think under pressure
- Postmortem culture — what did you learn?
- Technical depth of your recovery steps

### Question 5: What is the difference between SLA, SLO, and SLI?

**What they are testing**:
- Operational maturity
- Understanding of measurement and targets
- Ability to set realistic commitments

## What the Interviewer Looks For

### Signals of Strong Candidates

| Signal | What It Shows |
|---|---|
| Quotes concrete numbers (TTL, threshold, latency) | Practical experience, not theory |
| Mentions trade-offs unprompted | Systems thinking |
| Uses the correct term for the concept | Depth of knowledge |
| Acknowledges when high availability is not worth the cost | Pragmatism |
| Draws a multi-region diagram with failover paths | Architectural thinking |
| Mentions DNS propagation delay when discussing failover | Real-world experience |
| Discusses monitoring and alerting for availability | Operational maturity |
| References real incidents (GitHub 2018, AWS Kinesis 2020) | Industry awareness |
| Considers data consistency during failover | Distributed systems understanding |
| Proposes load testing and chaos engineering | Validation mindset |

### Red Flags

| Red Flag | Why It Matters |
|---|---|
| "We use a single database with a nightly backup" | No redundancy |
| "DNS failover is instant" | Ignores TTL and DNS caching |
| "99.999% is easy, just add more servers" | Does not understand exponential cost |
| "We do not need health checks, the servers never fail" | Denial of reality |
| "Active-active is always better" | Ignores complexity and consistency |
| "Failover is automated, we never test it" | Testing is essential |
| "The database handles everything" | No understanding of connection pools |
| "We can fix it manually within 5 minutes" | Does not account for human error |

## Level Expectations

### L3 (Entry-Level / New Grad)

**Expectation**: Understand basic concepts. Can define uptime, downtime, SLA. Knows what a SPOF is.

**Good answer**: "A single server is a SPOF. I would add a load balancer and at least two application servers. The database needs a replica for failover."

**Great answer**: "I would design for N+1 redundancy at the application tier. The load balancer does health checks every 5 seconds with a threshold of 3 failures. The database would use async replication with automated failover and a connection pool with retry logic."

### L4 (Mid-Level)

**Expectation**: Can design for a specific availability target. Understands trade-offs. Has hands-on experience with at least one HA mechanism.

**Good answer**: "For 99.9% availability, I would use multi-AZ deployment with ELB and RDS Multi-AZ. The application tier auto-scales across AZs. Health checks are configured with appropriate thresholds."

**Great answer**: "I would design for active-passive across two AZs. The health check would use a deep endpoint that verifies database connectivity, not just a TCP port check. The failover sequence is automated: health check failures trigger a Lambda that promotes the replica and updates Route53 with a 60-second TTL. I would budget 90 seconds of downtime from detection to recovery."

### L5 (Senior)

**Expectation**: Can design multi-region systems. Understands the operational cost of availability. Can discuss CAP theorem positioning.

**Good answer**: "I would use an active-active multi-region design with eventual consistency for reads and a single write region. Conflict resolution is not necessary because writes go to one region. Read replicas in each region serve local traffic. If the write region fails, I promote a different region, accepting 30 seconds of write unavailability."

**Great answer**: "The design depends on the consistency requirements. If the system can tolerate eventual consistency, I would use DynamoDB Global Tables with active-active replication across three regions. If strong consistency is required, I would use Spanner-style synchronous replication with Paxos across regions, accepting the latency cost of ~80ms cross-region round trips. The key trade-off is that DynamoDB provides 99.999% write availability with last-writer-wins conflict resolution, while Spanner provides 99.999% durability but lower write availability during a partition because it requires a majority quorum."

### L6 (Staff / Principal)

**Expectation**: Designs at the organization level. Considers cost, team maturity, operational burden. Can define SLIs and SLOs for the entire system.

**Good answer**: "I would implement an error budget-based approach. Each service defines its SLO. The availability architecture is determined by the SLO. The error budget is monitored in real-time. If a service is at risk of breaching its SLO, we invest in improving availability. If a service has surplus error budget, we can take calculated risks."

**Great answer**: "I would establish an SLO hierarchy. The user-facing SLO is 99.95%. Each downstream service has a stricter SLO to provide headroom. The critical path (payment processing) targets 99.99%. The non-critical path (recommendations) targets 99%. Each team owns their SLO and their error budget. Availability investments are prioritized based on error budget consumption rate. We run weekly chaos experiments against each service to validate the SLO. The cost of availability engineering must not exceed the projected revenue loss from downtime — we calculate this quarterly."

### L7 / L8 (Distinguished / Fellow / VP)

**Expectation**: Sets the availability strategy for the entire organization. Defines standards and culture around reliability.

**Good answer**: "Availability engineering is a cultural investment. I would establish a set of standards — every service must have health checks, circuit breakers, and automated failover. The standards are enforced through a Service Review process. New services must meet the standards before going to production."

**Great answer**: "I would embed availability into the software development lifecycle. Every feature request includes a reliability impact assessment. Every design review includes a section on failure modes. Every on-call incident produces a postmortem with action items. I would fund a team of SREs embedded within product teams rather than a central SRE team — this spreads reliability expertise without creating a bottleneck. The availability target is set per-service based on business criticality, not a single number for the entire organization."

## Phrase Bank — 15+ Key Phrases

Use these phrases in your answers. Each demonstrates specific knowledge.

### Requirement Phase

1. "The availability target is driven by business requirements. For this system, I would target 99.95% because the cost of downtime is approximately $X per minute."
2. "I would start with an SLA of 99.9% and create a roadmap to 99.99% as the system matures."
3. "For read-heavy workloads, I can achieve higher availability for reads by using replicas, while writes have a stricter availability constraint."

### Design Phase

4. "Every component is a potential SPOF. I would eliminate SPOFs starting with the most critical — the database."
5. "I would use active-passive for the database with automated failover. The passive replica receives synchronous replication within the same region."
6. "For the application tier, N+1 redundancy means running one spare instance. With 4 instances needed for peak load, I would run 5."
7. "Health checks must be deep, not shallow. A liveness check pings the port. A readiness check verifies the application can serve a request, which includes database connectivity."
8. "The circuit breaker prevents cascading failures. I would set the threshold at 50% errors over a 60-second window, with a 30-second cooldown."

### Failover Phase

9. "DNS TTL is critical. I would set it to 60 seconds and account for additional propagation delay."
10. "Failover is not instant. Detection takes 15 seconds (3 health checks at 5-second intervals). Promotion takes 10 seconds. Total: 25 seconds of write downtime."
11. "Automated failover introduces risk of false positives. I would use a multi-health-check strategy — the load balancer marks the target unhealthy, but failover only triggers if both the load balancer and an external monitor agree."

### Operational Phase

12. "Error budgets align availability investment with business impact. If a service has surplus error budget, we can deploy faster. If it is burning budget, we halt risky changes."
13. "Post-incident reviews are blameless. The goal is to find system weaknesses, not human mistakes."
14. "Chaos engineering validates that the availability mechanisms actually work. I would start with a single instance termination and escalate to region-level failover drills."

### Trade-off Phase

15. "The CAP theorem applies. For this system, I would choose AP (Availability + Partition Tolerance) because stale reads are acceptable, but downtime is not."
16. "The cost of each nine is exponential. Going from 99.9% to 99.99% costs roughly 3x more. I would only invest if the downtime cost justifies it."

## Five Drills for Interview Preparation

### Drill 1: The Nines Calculator

Practice mapping availability percentages to downtime durations and costs.

Pick a system (e.g., a payment gateway processing $10 million per day) and calculate:
- Daily revenue: $10,000,000
- Revenue per minute: $6,944
- Downtime cost at each nine level

| Nine Level | Downtime/Year | Revenue Lost | Engineering Investment |
|---|---|---|---|
| 99% | 3.65 days | $36.5M | $500K/year |
| 99.9% | 8.76 hours | $3.65M | $1.5M/year |
| 99.99% | 52.56 minutes | $365K | $5M/year |
| 99.999% | 5.26 minutes | $36.5K | $15M/year |

Where is the investment justified? At $36.5M annual downtime loss, $5M for 99.99% is easily justified. At $50K annual downtime loss (a small internal tool), even 99.9% may not be worth the cost.

### Drill 2: Diagram a Multi-Region Architecture

Set a timer for 10 minutes. Draw a multi-region active-passive architecture on paper. Include:

- DNS (Route53) with health checks
- Two regions with load balancers
- Application server pools
- Database primary in region A, replica in region B
- Replication link
- Failover trigger mechanism
- Monitoring

Check your diagram against the architecture in Lesson 3.

### Drill 3: Identify SPOFs in a Given Architecture

Given a description of a system, identify all SPOFs.

Example: "A Django application on a single EC2 instance using a PostgreSQL database on the same instance with an RDS read replica in a different AZ."

SPOFs:
- Single EC2 instance (app + DB on one server)
- If the instance fails, both app and database are gone
- The RDS read replica cannot be promoted automatically — manual intervention needed
- No load balancer — DNS points to the single instance directly
- No health check — no automatic detection

### Drill 4: The Failover Timeline

Given a set of parameters, calculate the worst-case and best-case failover times.

Parameters:
- Health check interval: 5 seconds
- Unhealthy threshold: 3 consecutive failures
- Failover action (promote replica): 8 seconds
- DNS TTL: 60 seconds
- DNS propagation: 0-2x TTL depending on ISP compliance

Worst-case detection: 3 * 5 = 15 seconds (all checks fail right after previous pass)
Worst-case promotion: 8 seconds
Worst-case DNS: 120 seconds (2x TTL for recursive resolvers)
Worst-case total: 143 seconds

Best-case detection: 10 seconds (checks fail on first attempt, 2nd interval = 10 seconds)
Best-case promotion: 8 seconds
Best-case DNS: 0 seconds (client re-resolves immediately)
Best-case total: 18 seconds

### Drill 5: The CAP Decision

For each system, decide CP or AP and justify:
1. Banking transaction system
2. Social media feed
3. Real-time multiplayer game scoreboard
4. DNS system
5. E-commerce product catalog

## Full Interview Walkthrough

Here is a complete mock interview question and answer sequence. Practice reading the question, pausing to think, then delivering the answer.

### Question: "Design Twitter's Tweet Feed with 99.99% Availability"

**Think about:**
- Read-heavy workload (tweet reads vastly outnumber writes)
- Eventual consistency is acceptable for feed display
- Writes must be durable (do not lose tweets)
- 500 million tweets per day, 300 million daily active users

**Sample Answer (Intermediate Level):**

"Let me start by defining the availability requirements. Twitter's tweet feed is read-heavy — roughly 100:1 read-to-write ratio. The target is 99.99% availability, which means about 52 minutes of downtime per year.

For the write path, I would use a CP system. Tweets must be durably stored. I would use a distributed database like Cassandra with a replication factor of 3 across 3 availability zones. Writes require acknowledgment from 2 of 3 replicas (W=2). This tolerates a single replica failure without losing write availability.

For the read path, I would use an AP system with caching. The feed is assembled from the user's follow graph, which can tolerate eventual consistency. If a tweet is 5 seconds late appearing in a follower's feed, that is acceptable. I would use Redis in each availability zone as a local cache. The cache stores pre-computed feed timelines. Write-through: when a user tweets, the fanout service writes to both the database and the cache.

The critical availability mechanism is the fanout service — when a user with 15 million followers tweets, we must write to 15 million timelines. This is an async job. If the fanout service is slow, the tweet is still in the database and will eventually reach followers. The follower reads from their cached timeline, which may be slightly stale but is not empty.

For failover: if the primary Redis cluster in one AZ fails, the application reads from the secondary AZ. If the database replica fails, reads go to the other replicas. The system degrades gracefully — users see slightly older tweets but the feed loads within 200ms.

Monitoring: I would track p99 feed load time (target: 500ms), tweet write latency (target: 200ms), and cache hit ratio (target: 95%)."

**Sample Answer (Senior Level):**

"Let me approach this systematically using the availability framework.

**Requirements**: 99.99% availability. 500M writes/day = 5,787 writes/second peak. 300M daily active users, 100:1 read ratio = 578,700 reads/second peak. The cost of downtime: Twitter's ad revenue is approximately $1M/day, or $694/minute. Multiplied by 2x for reputational damage = $1,388/minute. At 99.99%, monthly downtime cost = 4.32 minutes * $1,388 = $5,996/month. This justifies a significant availability investment.

**SPOFs**: The fanout service is the most critical SPOF. If it fails, no new tweets reach anyone. The database cluster is also critical. The cache is less critical — stale data is acceptable.

**Architecture**: Active-active across 3 regions. Each region has a complete stack. Writes go to the local region's Cassandra cluster. Cross-region replication uses Apache Pulsar for async log-based replication. Reads are served from the local region's Redis cluster.

**Failover for writes**: Cassandra handles AZ failures natively. For a full region failure, the other regions continue accepting writes. Replication lag across regions is typically under 2 seconds.

**Failover for reads**: Each Redis cluster is a Redis Enterprise active-active cluster. If one region goes down, the global load balancer routes traffic to the nearest healthy region. The Redis cluster in the surviving region has the data because of cross-region replication.

**Monitoring**: SLI-based alerting. Error budget for writes: 99.99% of tweet writes must succeed within 1 second. Error budget for reads: 99.99% of feed loads must return within 500ms. Budget consumption is tracked hourly. If the write error budget is being consumed at more than 150% of the allocated rate, the incident response team is paged.

**Validation**: Weekly chaos engineering drills. Every Tuesday at 2 PM, we terminate a random Cassandra node. Every month, we simulate a full region outage using our test cluster. Every quarter, we run a game day where the on-call engineer must handle a simulated cascading failure."

## Mock Interview: "What Happens When Your Primary Database Fails?"

**Interviewer**: "Your primary database server just died. Walk me through what happens next."

**Weak Answer**: "We have a backup. We restore from backup. It takes a few hours but we have monitoring."

**Why this is weak**: No mention of automated failover, no mention of replication lag, no concrete numbers, no discussion of data loss risk, no mention of DNS propagation.

**Strong Answer**:

"Let me walk through the timeline.

**T=0**: The database server crashes. The load balancer's health check detects the failure. Our health check is a readiness probe that runs every 5 seconds with a threshold of 3 consecutive failures. Detection takes 10-15 seconds.

**T=15s**: The health checker marks the primary as unhealthy. This triggers the failover automation. The first step is to verify the primary is truly dead — we check from two separate health check locations to eliminate false positives from a network partition.

**T=20s**: The orchestrator runs a STONITH (Shoot The Other Node In The Head) sequence: it SSHes into the primary and forces a shutdown. If the primary is unreachable, it disconnects the network port via the hypervisor API. This prevents split-brain.

**T=25s**: The orchestrator promotes the read replica to primary. This involves replaying the write-ahead log to apply any transactions that were in transit, then transitioning the replica to read-write mode.

**T=30s**: The application's connection pool detects the primary change. The connection pool is configured with multiple database endpoint addresses — it tries the primary first, and if it fails, connects to the replica that is now the primary. The pool retries with a 100ms backoff.

**T=35s**: The application starts writing to the new primary. Read traffic, which was already going to the replica, continues uninterrupted. Write traffic resumes with a 5-second interruption.

**Data loss assessment**: We use semi-synchronous replication. The primary acknowledges the write to the application, then sends the write to the replica synchronously. If the primary crashes between acknowledging and replicating, we lose at most one write. For a system processing $10M per day, this is acceptable — the cost of syncing every write (adding 2ms per write) is worth avoiding larger data loss.

**Post-incident**: We check for any lost writes using the binary log position at the time of crash. If the new primary has a slightly higher log position than the old primary's last checkpoint, we lost zero data. If the new primary is behind by a few entries, those transactions must be replayed from application logs or accepted as lost.

**Duration**: Total write downtime: 30-45 seconds. Read availability: 100% uninterrupted. Partial data loss: at most one transaction per connection."

## Five Additional High-Signal Topics for Follow-Up Questions

### Topic 1: Synchronous vs Asynchronous Replication

The interviewer might ask: "What replication mode do you use, and why?"

**Key points to cover**:
- **Synchronous**: Zero data loss but slower writes (round-trip latency added). Use for financial systems.
- **Asynchronous**: Faster writes but potential data loss on crash (seconds to minutes of lost data). Use for read-heavy systems where a few seconds of data loss is acceptable.
- **Semi-synchronous**: The primary acknowledges to the client only after at least one replica has confirmed the write. Best trade-off for most systems.

### Topic 2: Fencing and STONITH

The interviewer might ask: "How do you prevent split-brain in an automated failover setup?"

**Key points to cover**:
- Fencing prevents a failed node from interfering with the new primary.
- STONITH: Shoot The Other Node In The Head — forcibly shut down or isolate the old primary.
- Alternative: lease-based fencing (the old primary holds a lease, and if the lease expires, it stops accepting writes).

### Topic 3: Black Friday and Traffic Patterns

The interviewer might ask: "How do you prepare for a 10x traffic spike?"

**Key points to cover**:
- Pre-provision capacity based on historical data + 20% buffer.
- Load test the failover mechanism at peak traffic levels.
- Feature flags to disable non-critical features during peak.
- Queue-based admission control: accept requests, process async, return results via polling.
- Throttle or shed low-priority traffic first (crawlers, batch jobs, free tier).

### Topic 4: Backup and Restore Strategy

The interviewer might ask: "What is your backup strategy, and how does it relate to availability?"

**Key points to cover**:
- RPO (Recovery Point Objective): How much data can you afford to lose? RPO = 1 hour means hourly backups.
- RTO (Recovery Time Objective): How fast must you restore? RTO = 4 hours means you have 4 hours to rebuild from backup.
- Full daily backup + point-in-time recovery via transaction log backups every 5 minutes.
- Automated restore testing: backups that are never tested will fail when needed.

### Topic 5: Incident Response Runbooks

The interviewer might ask: "Walk me through your incident response process."

**Key points to cover**:
- Detection: Monitoring alert fires. Severity levels (SEV1: full outage, 5-minute response time; SEV2: partial degradation, 15-minute response time; SEV3: cosmetic, next business day).
- Triage: The on-call engineer acknowledges, assesses impact, and declares the severity.
- Mitigation: Stop the bleeding (rollback, redirect traffic, restart servers), not root cause analysis.
- Resolution: Apply the fix, verify the system is healthy, update the status page.
- Postmortem: Root cause analysis, timeline reconstruction, action items with owners and due dates.

## Five More Drills for Interview Preparation

### Drill 6: The "Ask Me About Trade-offs" Drill

For each availability mechanism you mention, immediately state the trade-off. Practice this in mock interviews.

| Mechanism | Trade-off |
|---|---|
| "I would use read replicas" | "This adds eventual consistency complexity — users might not see their own writes immediately." |
| "I would use multi-region active-passive" | "The passive region costs nearly as much as the active but handles zero traffic. The cost is worth it for the availability gain." |
| "I would use synchronous replication" | "This adds 2-10ms to every write. For a write-heavy workload, this might be prohibitive." |
| "I would use a circuit breaker" | "During the open period, the feature is completely unavailable. Users see an error instead of a slow response. Sometimes a slow response is better than no response." |
| "I would automate failover" | "Automation introduces false-positive risk. A misconfigured health check could trigger an unnecessary failover." |

### Drill 7: The "Three Architectures" Drill

For any system, be prepared to describe three levels of availability architecture:

**Level 1 (99.9%)**: Single region, multi-AZ, load balancer with health checks, database replica with automated failover, circuit breakers for external dependencies.

**Level 2 (99.99%)**: Multi-region active-passive, Global load balancer with health-check-based failover, cross-region database replication, read replicas in secondary region, automated failover orchestration.

**Level 3 (99.999%)**: Multi-region active-active, anycast routing, distributed database with multi-region writes, CRDT-based conflict resolution, chaos engineering program, 24/7 SRE coverage.

### Drill 8: The "Numbers First" Drill

Always lead with numbers. Never say "fast" — say "under 200ms." Never say "high availability" — say "99.99%." Never say "quick failover" — say "failover completes in under 30 seconds."

Practice converting vague statements to specific numbers:

| Vague | Specific |
|---|---|
| "The system is highly available" | "The system targets 99.95% monthly availability with a 4-hour RTO and 1-hour RPO." |
| "Failover is fast" | "Health checks detect failure in 15 seconds. Automated promotion completes in 30 seconds. Total write downtime: 45 seconds." |
| "We use redundancy" | "N+1 redundancy. 5 instances handle peak traffic. If one fails, 4 handle 100% load at 80% capacity each." |
| "The database is replicated" | "Semi-synchronous replication to one replica in a different AZ. Replication lag: under 100ms during normal operation." |
| "We have monitoring" | "Prometheus scraping every 15 seconds. Alerting based on burn rate: 2% error budget consumed per hour triggers a page." |

### Drill 9: The "Yes, And" Drill

When the interviewer challenges your design, use "yes, and" to acknowledge the concern and add mitigation.

**Interviewer**: "Your circuit breaker would make the feature unavailable during the open period. That seems aggressive."

**Weak response**: "The circuit breaker prevents cascading failure."

**Strong response**: "Yes, and that is a deliberate trade-off. The circuit breaker makes the feature unavailable for 30 seconds while the dependency recovers. During that time, we serve a fallback response: stale cached data. If stale data is not available, we return a specific error message telling the user the feature is temporarily unavailable. This is better than keeping the service in a degraded state for 5+ minutes where every request times out and ties up resources."

### Drill 10: The "Five-Minute Design Sprint"

Set a timer for 5 minutes. Design a highly available version of a familiar system as fast as you can:

- Minute 1: Requirements (target availability, read/write ratio, cost constraints)
- Minute 2: Identify SPOFs (list them)
- Minute 3: Choose redundancy pattern (active-passive vs active-active)
- Minute 4: Design failover (health checks, promotion, DNS)
- Minute 5: Monitoring and validation

Practice with: URL shortener, chat app, e-commerce cart, file upload service, rate limiter.

## Common Mistakes in Availability Interviews

### Mistake 1: Designing for Maximum Availability Without Justification

The interviewer asks you to design Twitter. You immediately propose a multi-region active-active system with CRDT-based conflict resolution. The system would cost $5M+/month to operate. Twitter is a social media platform where 5 minutes of downtime is not a crisis. You over-engineered.

**Fix**: Always ask: "What is the availability target? What is the cost of downtime?" Then design to that target.

### Mistake 2: Ignoring the Cost Dimension

You describe a beautiful multi-region architecture. The interviewer asks "how much does this cost?" You have no idea. You lose credibility because you clearly have never built or operated such a system.

**Fix**: Have rough cost numbers ready. A single EC2 instance: $50-500/month. An RDS instance: $200-2,000/month. Cross-region data transfer: $0.02/GB. A 10-server deployment in 2 regions: approximately $10,000-20,000/month.

### Mistake 3: Failing to Prioritize

You treat all SPOFs as equally important. You discuss the DNS SPOF with the same urgency as the coffee machine SPOF. The interviewer wants to see that you can prioritize: fix the database first, then the load balancer, then the DNS, then the monitoring.

**Fix**: "The most critical SPOF is the database. I would start there. Next, the load balancer. After that, DNS redundancy. Each step adds cost, and I want to get the most availability improvement per dollar."

### Mistake 4: No Validation Plan

You design the system but do not explain how you would validate that it works. The interviewer is left wondering whether you have ever actually built a resilient system.

**Fix**: Always end with validation. "I would validate this design through: (1) load testing to verify capacity, (2) chaos experiments starting with instance termination, (3) quarterly failover drills with timed recovery."

### Mistake 5: Vagueness About the Database Layer

The most common weakness in availability interviews is the database layer. Candidates describe application-tier redundancy well but give vague answers for database failover.

**Fix**: Be specific. "I would use RDS Multi-AZ with synchronous standby in another AZ. Failover is automatic: RDS detects the primary failure, promotes the standby, and updates the DNS endpoint. Typical downtime: 60-120 seconds. For lower RPO, I would use Aurora with 6 replicas across 3 AZs — failover in under 30 seconds with zero data loss."

## What Interviewers Really Want

At the end of any availability-focused question, the interviewer is asking themselves:

1. **Does this candidate understand that downtime is a business problem, not just a technical problem?** You should talk about cost of downtime, SLA penalties, and business impact.

2. **Has this candidate operated a real system?** You should reference specific numbers, configurations, and trade-offs that only come from experience.

3. **Can this candidate prioritize under constraints?** You should ask about the budget, team size, and timeline before diving into the design.

4. **Does this candidate think about failure modes?** You should discuss not just what works, but what breaks and how you recover.

5. **Can this candidate communicate clearly under pressure?** Your answers should be structured, confident, and backed by concrete details.

If you hit all five signals, you will ace the availability question at any company.

## Common Follow-Up Questions

These questions may come after your initial answer. Have answers prepared.

### "How do you handle data loss during failover?"

Key points: Synchronous vs async replication, RPO, WAL archiving, point-in-time recovery. The trade-off is between availability (async: always accepting writes) and durability (sync: no data loss but slower writes).

### "What happens if your monitoring system is down?"

Key points: Redundant monitoring instances, external monitoring from a separate provider (Pingdom, Datadog), self-healing alert if monitoring data stops flowing, manual health check endpoints that work without monitoring.

### "How do you set the health check interval?"

Key points: The interval is a trade-off between detection speed and false positive rate. 5 seconds is standard. For critical services, 1 second with higher threshold (5 consecutive failures). For non-critical services, 15 seconds with lower threshold (2 failures).

### "How do you handle a slow disk that causes 5-second response times?"

Key points: The health check should time out and mark the instance unhealthy. The load balancer removes it. The slow disk is a signal to replace the instance entirely (immutable infrastructure) rather than investigating the root cause of the slowness.

### "What if the replica promotion fails?"

Key points: Second promotion attempt to the next replica. If all replicas fail, the system enters read-only mode. Writes are queued or rejected with a clear error message. The database team investigates the replica state while the system continues serving reads.
