# Availability Concepts

## The 3am Pager Call

It is 3:14 AM on Black Friday. Your phone vibrates off the nightstand. The screen shows a PagerDuty alert: **Payment Gateway — 502 Errors — CRITICAL**.

You stumble to your laptop, heart pounding. The dashboard confirms your worst fear. The payment processing pipeline has been returning HTTP 502 errors for the last four minutes. During those 240 seconds, 12,000 transactions failed. At an average order value of $84.50, that is just over $1 million in failed revenue. The support team is flooding Slack with screenshots from angry customers who cannot complete checkout.

You check the database primary node. It is unresponsive. The replica is serving stale reads — but the write path is completely dead. The auto-increment counter on the orders table threw a deadlock, cascaded into a disk I/O spike, and the primary MySQL node went into crash-recovery loop. There is no automatic failover configured. You scramble to promote the replica manually — a procedure you have never practiced.

Twenty-three minutes later, the system is back online. The damage: $5.8 million in lost revenue, 340,000 abandoned carts, and a 12-point dip in customer satisfaction score that took six weeks to recover.

The root cause was not the deadlock. The root cause was that the system had **no automated availability mechanism**. One database node failed, and the entire business went down with it.

## The Insight

**Availability** is the percentage of time a system is operational and accessible to its users. It is the single most visible non-functional requirement because downtime is immediately measurable — and immediately costly.

Think of a 24-hour convenience store. A store that never closes has high availability. A store that locks the doors at midnight has a scheduled downtime window. A store whose owner goes on vacation without a backup has no availability guarantees. Your system is that store. Every second it is closed, customers walk to a competitor.

The mathematical definition is simple:

```
Availability (%) = (Total Time - Downtime) / Total Time * 100
```

A system that runs for 24 hours and experiences 86.4 seconds of downtime achieves 99.9% availability — three nines. A system with 8.64 seconds of downtime in 24 hours achieves 99.99% — four nines. The difference between three nines and four nines is a factor of 10 in allowed downtime, but the engineering cost grows exponentially.

## Core Vocabulary

### Uptime and Downtime

**Uptime** is the duration a system is reachable, responsive, and producing correct results. **Downtime** is the duration it is not. Downtime splits into two categories:

- **Planned downtime**: Maintenance windows, database migrations, kernel upgrades. You schedule these.
- **Unplanned downtime**: Crashes, network partitions, data corruption. These wake you up at 3 AM.

Planned downtime still counts toward availability if your SLA does not exclude maintenance windows. Many cloud providers subtract planned maintenance from their SLA calculation — read the fine print.

### SLA, SLO, SLI

**SLA (Service Level Agreement)** is the contract between a provider and a customer. It specifies the availability commitment, usually as a percentage, and the penalty for missing it. Example: "Compute API guarantees 99.99% monthly uptime. If uptime falls below, you receive a 10% service credit."

**SLO (Service Level Objective)** is the internal target you set for your system. It should be stricter than the SLA — you want headroom. If your SLA is 99.9%, your SLO might be 99.95%. This buffer absorbs measurement noise and prevents SLA breaches from minor incidents.

**SLI (Service Level Indicator)** is the actual measurement. SLIs answer: What are you measuring, and how? For availability, the SLI is typically the ratio of successful requests to total requests over a time window.

```
SLI = Successful Requests / Total Requests
    = 9,950,000 / 10,000,000
    = 99.5%
```

Choose your SLI definition carefully. Counting every HTTP 200 response ignores slow failures (timeouts that wait 30 seconds before erroring). Counting only responses under 500ms gives a tighter signal.

### The Nines

| Availability Level | Allowed Downtime per Year | Allowed Downtime per Month | Allowed Downtime per Week |
|---|---|---|---|
| 90% (one nine) | 36.5 days | 73 hours | 16.8 hours |
| 99% (two nines) | 3.65 days | 7.3 hours | 1.68 hours |
| 99.9% (three nines) | 8.76 hours | 43.8 minutes | 10.1 minutes |
| 99.95% | 4.38 hours | 21.9 minutes | 5.04 minutes |
| 99.99% (four nines) | 52.56 minutes | 4.38 minutes | 1.01 minutes |
| 99.999% (five nines) | 5.26 minutes | 25.9 seconds | 6.05 seconds |

Each additional nine costs roughly 10x more to engineer. A three-nines web application might run on two load-balanced servers in one data center. A five-nines system requires multi-region active-active deployment, redundant network paths, generator-backed power, and a team of SREs on call 24/7.

### Redundancy

**Redundancy** is the duplication of critical components to increase availability. If one component fails, another takes over.

Types of redundancy:

- **Hardware redundancy**: Multiple servers, disks, network switches, power supplies.
- **Software redundancy**: Multiple application instances, database replicas, DNS nameservers.
- **Geographic redundancy**: Multiple data centers or cloud regions.
- **Time redundancy**: Retry mechanisms that re-attempt failed operations.

Redundancy increases cost linearly but availability logarithmically. Going from one server to two doubles hardware cost but might improve availability from 95% to 99.5%. Going from 10 servers to 20 adds a much smaller gain.

### Failover

**Failover** is the automatic transfer of traffic from a failed component to a healthy one.

- **Active-Passive**: One component handles traffic. The second sits idle, receiving heartbeats. When the active fails, the passive takes over.
- **Active-Active**: Both components handle traffic simultaneously. If one fails, the remaining component absorbs the load.
- **N+1**: N active components plus one standby. The standby can replace any single failure.
- **N+M**: N active components plus M standbys. M provides room for multiple simultaneous failures.

### SPOF (Single Point of Failure)

A **Single Point of Failure** is a component whose failure causes the entire system to become unavailable. Common SPOFs:

- A single database server with no replica
- One load balancer with no backup
- A single network switch connecting all servers
- One DNS record with a single nameserver
- A single power supply or power distribution unit

Eliminating SPOFs is the primary engineering activity in availability design. Every component must have a backup — or the design must tolerate its loss.

### Health Checks

**Health checks** are periodic probes that determine whether a component is alive and serving traffic. Two types:

- **Liveness check**: "Is the process running?" A simple TCP connect to a known port.
- **Readiness check**: "Is the process ready to serve requests?" Verifies that dependencies (database, cache) are reachable and the component is not overloaded.

Health check configuration matters. Too frequent: false positives from transient slowness. Too infrequent: slow failure detection. A common starting point is a 5-second interval with 3 consecutive failures marking the component unhealthy.

### Graceful Degradation

**Graceful degradation** means the system continues to function — with reduced features — when some components fail. Examples:

- An e-commerce site that shows cached product pages when the database is slow, rather than returning an error page
- A video streaming service that drops resolution from 4K to 720p when transcoding capacity is strained
- A messaging app that queues outgoing messages when the server is unreachable, delivering them when connectivity restores

### Maintenance Window

A **maintenance window** is a scheduled period during which the system may be unavailable for planned work. Common during low-traffic hours (2-4 AM). The best systems achieve zero-downtime maintenance through rolling updates and blue-green deployments, eliminating the need for maintenance windows entirely.

## Step by Step: How Availability Mechanisms Work

### Step 1: Health Check and Failure Detection

The system detects that a component has failed. This is the first and most critical step — you cannot recover from a failure you do not know about.

<AsciiDiagram
  id="availability/concepts-health-check"
  title="Health Check Probe Flow"
  alt="A load balancer sends periodic health check requests to an app server, tracking pass/fail counts to decide if the server is healthy or unhealthy."
  content={`
     Every 5 seconds──┐
                      ▼
              ┌─────────────────┐
              │   Load Balancer  │
              │  [Health Check   │
              │   Monitor]       │
              └────────┬─────────┘
                       │
             ┌─────────┼─────────┐
             │         │         │
             ▼         ▼         ▼
        ┌────────┐┌────────┐┌────────┐
        │  App   ││  App   ││  App   │ Fails:
        │ Node 1 ││ Node 2 ││ Node 3 │───▶ Removed
        │ 200 OK ││ 200 OK ││ 503    │     from pool
        └────────┘└────────┘└────────┘

     Threshold: 3 consecutive failures → unhealthy
     Recovery:  2 consecutive successes → healthy
  `}
  caption="Health check with configurable failure and recovery thresholds."
/>

Configuration matters. A health check that pings the process port tells you the process is running — not that it can serve requests. A deep health check connects to the database, runs a lightweight query, and verifies the application logic. Deep checks catch more failure modes but add latency and load. Most production systems use a mix: liveness checks every 5 seconds and deep readiness checks every 30 seconds.

### Step 2: Traffic Reroute

Once a component is marked unhealthy, traffic must be rerouted to healthy components.

<AsciiDiagram

     │  /  │  \\       │         │  /      \\      │
     ▼    ▼    ▼      │         ▼        ▼      │
   ┌──┐ ┌──┐ ┌──┐    │       ┌──┐     ┌──┐     │

   └──┘ └──┘ └──┘    │       └──┘     └──┘     │
                      │                         │

                       │     between N1 and N3  │
  `}
  caption="Traffic distribution shifts after health check failure."

The load balancer typically uses round-robin or least-connections scheduling. When a node is removed, the load balancer distributes its share to the remaining nodes. This is why you need headroom — an N+1 design means any single node can fail without overloading the survivors.

### Step 3: Database Failover

When the primary database fails, the system must promote a replica to primary.

  `}

Failover is not instantaneous. Detection takes 5-15 seconds (3 health check intervals at 5 seconds each). Promotion takes another 2-10 seconds depending on the database engine and the amount of unapplied replication lag. Total downtime from primary failure to resumed writes: 7-25 seconds. This is why four-nines systems measure their allowed monthly downtime in minutes — a single database failover can consume 25% of the monthly budget.

### Step 4: Retry and Fallback

When a request fails, the client or intermediary should retry with exponential backoff.

<AsciiDiagram
  id="availability/concepts-retry-backoff"
  title="Exponential Backoff Retry Pattern"
  alt="Timeline showing a request that fails, then retries at 100ms, 200ms, 400ms, and 800ms intervals before succeeding on the fourth retry."
  content={`
     Request ──▶ Service   Result
        │                   
        ├── 1st attempt ──▶ FAIL (503)
        ├── wait 100ms
        ├── 2nd attempt ──▶ FAIL (503)
        ├── wait 200ms
        ├── 3rd attempt ──▶ FAIL (503)
        ├── wait 400ms
        ├── 4th attempt ──▶ SUCCESS (200)
        │
        Jitter applied: random ±25% 
        to prevent thundering herd
  `}
  caption="Exponential backoff with jitter prevents retry storms."
/>

Retry logic must include a maximum retry count (typically 3-5) and jitter (random ±25% of the backoff interval). Without jitter, all clients retry simultaneously — the thundering herd problem — which can overwhelm a recovering service and cause cascading failure.

### Step 5: Circuit Breaker

A circuit breaker prevents a system from repeatedly calling a failing dependency.

A circuit breaker in the OPEN state returns an error immediately without making a network call. This prevents wasted resources, reduces latency for failing requests, and gives the dependency time to recover.

### Step 7: Health Recovery and Reintegration

After a component recovers, it must be safely reintroduced to the pool. The load balancer should start with a low traffic weight and ramp up. This "slow start" or "warmup" period allows caches to fill, connection pools to establish, and JIT compilers to warm. AWS ELB, for example, uses a configurable deregistration delay and slow start duration.

### Step 8: Post-Incident Review

Every incident produces a postmortem. The goal is not blame — it is improving the system's availability. Key questions:

- How long did it take to detect the failure?
- How long did it take to diagnose the root cause?
- How long did it take to restore service?
- What automated checks would have caught this earlier?
- What manual steps could be automated?

## Variants of Availability Architecture

### Active-Passive (Hot Standby)

One active component serves traffic. One or more passive components run in standby, receiving real-time replication but not serving traffic.

Pros: Simpler to reason about, lower complexity, no conflict resolution needed. Cons: The standby hardware costs nearly as much as the primary but handles zero traffic during normal operation. Failover is not instant — promoting a replica takes time and involves DNS propagation.

## Availability Measurement in Practice

Calculating availability for a real system is more nuanced than the simple formula suggests. Here is how FAANG companies do it.

### Defining the SLI Window

Availability is measured over a specific time window. Common choices:

- **Rolling 30 days**: The standard AWS SLA measurement. Smooths out daily noise but delays detection of prolonged problems.
- **Calendar month**: Matches billing cycles. A bad day early in the month leaves time to recover. A bad day on the 28th breaches the monthly target with no recovery window.
- **Rolling 7 days**: Used for internal SLOs. Faster feedback but more volatile. A 15-minute incident at 99.9% target consumes 1.5 days of error budget.

### Counting Requests Properly

Not all requests are equal. Define what counts as an available request:

**Inclusion criteria**: Every HTTP request that reaches the load balancer counts. Requests blocked by WAF or rate limiting do not — those are user errors, not system failures.

**Exclusion criteria**: Planned maintenance windows, if they are documented and within SLA terms. Requests during a DDoS attack (if the SLA excludes DDoS).

**Success criteria**: The request returns a non-5xx response within the timeout limit. A request that returns HTTP 200 after 25 seconds is not a success for user-facing availability — it failed the user's experience even though the server wrote a 200.

### Example SLI Calculation

A service receives 12,000,000 requests in a 30-day month. The timeout is 500ms.

| Category | Count | Reason |
|---|---|---|
| Total requests | 12,000,000 | |
| HTTP 2xx under 500ms | 11,880,000 | Successful |
| HTTP 2xx over 500ms | 24,000 | Slow — counted as failure |
| HTTP 4xx | 48,000 | User errors — excluded |
| HTTP 5xx | 36,000 | Application errors — failure |
| HTTP 503 (overloaded) | 12,000 | Capacity issue — failure |

Good requests = 11,880,000 (only those under 500ms)
Failures = 24,000 + 36,000 + 12,000 = 72,000
SLI = 11,880,000 / (11,880,000 + 72,000) = 11,880,000 / 11,952,000 = 99.398%

This is below 99.9%. Even though 99.7% of requests returned 2xx, the slow responses pushed the effective availability below the SLO. This is why timeout-bounded SLIs catch quality problems that raw error rates miss.

### Error Budgets

An **error budget** is the amount of downtime a system can tolerate within its SLO window. If your SLO is 99.9% over 30 days, your error budget is 0.1% of the total time = 43.2 minutes.

Error budget tracking changes how teams make decisions:

- **Budget available**: Teams can deploy new features, experiment, and take calculated risks.
- **Budget nearly depleted**: Teams must stop deploying, focus on reliability improvements, and investigate the root cause of the budget consumption.
- **Budget exhausted**: The SLO has been breached. A postmortem is required. The team's next sprint is dedicated to reliability work.

This creates a direct feedback loop between availability and feature velocity. It prevents the common pattern of shipping features first and discovering availability problems later.

## Common Pitfalls in Availability Design

### Pitfall 1: Treating All Failures as Binary

A server is not simply "up" or "down." It can be:

- **Functional but slow**: The TCP port responds but requests time out after 5 seconds.
- **Functional but intermittently failing**: 90% of requests succeed, but 10% fail with random errors.
- **Degraded but serving**: The service works for most users but errors on a specific feature path.
- **Split-state**: The liveness check passes (the process is alive) but the readiness check fails (the process cannot serve traffic).

Treating all failures as binary means you miss partial degradation. Use graduated health states: HEALTHY, DEGRADED, UNHEALTHY, DEAD.

### Pitfall 2: Replication Lag Blindness

You add a read replica for availability, then serve stale data to users who just wrote data. This is the "read-your-writes" consistency problem.

A user submits a product review. The page redirects to the product page. The read replica has not yet picked up the new review because of 500ms replication lag. The user sees "0 reviews" and assumes their submission failed. They submit again. Now there are two identical reviews.

Solutions:
- Read-after-write consistency: For N seconds after a write, route the user's reads to the primary.
- Sticky sessions with read-replica binding: Bind a user to a specific replica for a session duration.
- Replication lag monitoring: Configure an alert if lag exceeds 500ms. Fall back to primary reads during high lag.

### Pitfall 3: Ignoring Client-Side Retries

You configure your server-side HA perfectly. But the client library has its own retry logic with a 30-second timeout and no backoff. When the server has a 2-second blip, every client retries immediately, creating a 10x traffic spike that overwhelms the server.

Audit client-side retry configurations alongside server-side availability mechanisms. Ensure alignments: total client timeout should equal server-side expected recovery time plus one retry interval.

### Pitfall 4: Unbounded Queues

An asynchronous work queue improves availability by decoupling request acceptance from processing. But if the queue is unbounded and processing slows down, the queue grows indefinitely. When the queue eventually fills memory or disk, the entire system crashes.

Always use bounded queues with a rejection policy. In a message queue like SQS, set a maximum message retention period (default 4 days, max 14 days). In an in-memory queue, set a maximum size and a rejection strategy (fail fast, discard oldest, or shed load).

### Pitfall 5: The Single-Metric Dashboard

You monitor "overall uptime" as a single percentage. It looks fine at 99.95%. But digging deeper reveals that each individual user sees 99.0% because failures are concentrated in specific regions or user segments.

Monitor availability by:
- Geographic region
- User plan tier (free vs premium)
- Feature path (search vs checkout vs profile)
- API endpoint
- Time of day

A single aggregated number hides the distribution of failures. Slice the data to find the real story.

## Mental Model Check

Test your understanding with these questions:

1. **SLA Math**: Your system has a 99.95% monthly SLA. So far this month, you have had 12 minutes of unplanned downtime. How much more downtime can you afford before breaching the SLA? (Assume a 30-day month.)

   **Answer**: Total minutes in a 30-day month = 43,200. Allowed downtime at 99.95% = 43,200 * 0.0005 = 21.6 minutes. Used: 12 minutes. Remaining: 9.6 minutes. If your next incident exceeds 9.6 minutes, you breach the SLA.

2. **Failover Timing**: You configure health checks at 5-second intervals with a threshold of 3 failures. Database replica promotion takes 8 seconds. DNS TTL is 60 seconds. What is the worst-case total downtime for a database primary failure affecting write traffic?

   **Answer**: Worst-case detection: 3 * 5 = 15 seconds (the failure happens immediately after a successful check, so the three consecutive failures take 15 seconds). Promotion: 8 seconds. DNS propagation: up to 60 seconds (worst case: clients must wait for DNS TTL to expire before re-resolving to the new primary IP). Total: 15 + 8 + 60 = 83 seconds. In practice, many clients cache DNS beyond the TTL, so actual downtime could be longer. This is why low TTL values and DNS pre-fetching matter.

3. **Redundancy Math**: You run 3 application servers, each capable of handling 500 requests per second (RPS). Your peak traffic is 1,200 RPS. Is this N+1, N+2, or N+0? What happens if one server fails?

   **Answer**: N = number of servers needed at peak = 1,200 / 500 = 2.4, rounded up to 3. You have 3 servers. So N+0 — no redundancy. If one server fails, remaining servers handle 1,200 / 2 = 600 RPS each, exceeding the 500 RPS limit. The system will degrade (slow responses, timeouts). You need 4 servers for N+1. Expected RPS per server should be capacity * 0.8 (80% utilization target for headroom).

4. **Active-Active vs Active-Passive**: You are designing a global payment processing system. Writes must be strongly consistent within a region and eventually consistent across regions. Which architecture variant is appropriate?

   **Answer**: Active-passive with a single write region and multiple read regions. Writes go to the primary region (strong consistency within the write region). Read replicas in other regions serve local reads with eventual consistency. On primary region failure, there is a brief write outage while a replica region promotes to primary. Full active-active is not appropriate because it would require write-write conflict resolution, which breaks strong consistency.

5. **Cost vs Availability**: Your current single-server system costs $500/month and achieves 98% availability. A dual-server active-passive setup costs $1,200/month and achieves 99.9% availability. Your annualized revenue loss from downtime is calculated as: (downtime hours) * $50,000. At what point does the investment pay for itself?

   **Answer**: Current system annual downtime = 365 * 24 * 0.02 = 175.2 hours. Annual revenue loss = 175.2 * $50,000 = $8,760,000. Improved system annual downtime = 365 * 24 * 0.001 = 8.76 hours. Annual revenue loss = 8.76 * $50,000 = $438,000. Annual savings from uptime = $8,760,000 - $438,000 = $8,322,000. Additional cost = ($1,200 - $500) * 12 = $8,400. Payback: $8,400 / $8,322,000 * 365 days = 0.37 days. About 9 hours. The upgrade pays for itself in under a day.
