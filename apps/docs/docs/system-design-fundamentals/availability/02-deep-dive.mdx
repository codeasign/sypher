# Availability Deep Dive

## The Internal Stack

A highly available system is built from multiple layers, each providing a specific mechanism for detecting, isolating, and recovering from failures. Below the application code, an availability stack runs continuously.

The key insight: availability mechanisms at different layers operate at different time scales. DNS changes take 60-300 seconds to propagate. Load balancer health checks detect failure in 5-15 seconds. Circuit breakers react within a single request cycle (milliseconds). Retries handle transient failures in under a second. A robust system uses all layers together — they complement, not replace, each other.

### Layer 1: DNS Redundancy

DNS is the first point of contact. If DNS fails, no traffic reaches your system. DNS redundancy comes in two forms:

- **Multiple A/AAAA records**: DNS round-robin — the client picks an IP at random from the list.
- **Anycast**: Multiple servers share the same IP. BGP routing sends each client to the nearest healthy server.

Anycast is strictly better for availability because it handles server failure without client-side changes — the BGP withdrawal propagates in seconds to minutes, and traffic naturally reroutes to the next-closest server.

### Layer 2: Load Balancer Health Checking

The load balancer is the traffic cop. It must detect failures within seconds and drain connections gracefully.

Health check architecture:

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  Health      │      │  Target      │      │  Target      │

└──────────────┘      └──────────────┘      └──────────────┘
       │                                          │

       ▼                                          ▼
```

The control plane maintains a target state table. Each health check response updates the consecutive success/failure counters. When the failure counter hits the threshold, the target is marked unhealthy and removed from the routing table. This happens in the control plane — the data plane (which forwards actual traffic) checks the routing table before each connection.

## Data Structures and Algorithms

### Failure Detection: The Phi Accrual Algorithm

The standard approach — fixed interval health checks — has a fundamental problem: it cannot distinguish between a dead node and a slow node. The **Phi Accrual Failure Detector**, popularized by Apache Cassandra and Akka, uses a probabilistic approach.

The phi accrual detector is adaptive. In a high-latency environment, it accepts wider gaps before suspecting failure. In a low-latency environment, it detects failure faster. This eliminates the need for manual threshold tuning — the algorithm learns the network's normal behavior.

Common threshold: phi >= 8 means the node is declared dead. This corresponds to approximately 1 in 100 million probability that the node is still alive.

### Gossip Protocol

In large distributed systems, a centralized health monitor is itself a SPOF. Gossip protocols distribute failure detection across all nodes.

     After 3 rounds, all 6 nodes converge
     on the same membership view.
  `}
  caption="Gossip protocol: epidemic-style information dissemination achieves convergence in O(log N) rounds."

Each gossip round:
1. Pick a random node or a small set of nodes (typically 2-3).
2. Exchange membership tables — each entry includes node ID, IP address, incarnation number, and state (ALIVE, SUSPECT, DEAD).
3. Update local membership table with any new information.

Convergence time: O(log N) rounds where N is the number of nodes. For a 100-node cluster, convergence takes approximately 7 rounds at 100ms per round = 700ms total.

The incarnation number prevents stale information from overriding newer state. When a node restarts, it increments its incarnation number. If a gossip message arrives saying "node X is DEAD with incarnation 3," but the local table shows "node X is ALIVE with incarnation 5," the local table wins — the message is stale.

### Heartbeat Mechanisms

Heartbeats are the raw signals that feed failure detectors. Three common mechanisms:

1. **TCP Keepalive**: Built into TCP. Sends empty packets after a configurable idle period. Default: 2 hours. You must reduce this (e.g., 15 seconds) for timely failure detection.

2. **Application-Level Heartbeat**: The application sends "I am alive" messages at a fixed interval (e.g., every 1 second). More reliable than TCP keepalive because it verifies the application process, not just the kernel's TCP stack.

3. **Custom UDP Heartbeat**: A lightweight heartbeat over UDP. Discardable — if a packet is lost, the next heartbeat overwrites it. No connection overhead. Used by systems like etcd and Consul.

### Stage Details

**Stage 2 — Anycast BGP withdrawal**: When a Point of Presence (PoP) fails, the router stops advertising the prefix to BGP peers. BGP convergence takes 30-120 seconds during which some traffic may still route to the dead PoP. DNS-based failover is faster (controlled by TTL) but requires the client to re-resolve.

**Stage 5 — Circuit breaker decision**: The circuit breaker uses a sliding window of recent requests. For each dependency (database, cache, downstream API), it tracks:

- Total requests in window (e.g., last 60 seconds)
- Failed requests in window
- Request volume sufficient for statistical significance (minimum 100 requests)

If the failure rate exceeds the threshold (e.g., 50% in the last 60 seconds), the circuit opens. It stays open for a configurable cooldown (e.g., 30 seconds) before transitioning to half-open.

**Stage 7 — Replica fallback**: A common pattern: try the primary. If it does not respond within 50ms, try a replica. If the replica does not respond within 100ms, read from cache. This cascading timeout pattern maximizes the chance of serving a response while bounding latency.

Split-brain is one of the hardest problems in distributed availability. Prevention strategies:

1. **Quorum-based decisions**: A node only promotes to primary if it can reach a majority of peers.
2. **Fencing (STONITH)**: Shoot The Other Node In The Head. Forcefully shut down the other node before promoting.
3. **Lease-based leadership**: A primary holds a lease from a coordinator. If the lease expires, the node must stop accepting writes. Another node can only acquire the lease after the lease period.
4. **Generation clocks**: Each leadership term has a monotonically increasing number. Writes from an older term are rejected.

### Quorum-Based Consensus

Quorum systems trade availability for consistency. A read/write quorum requires:

- Write quorum: W nodes must acknowledge a write
- Read quorum: R nodes must respond to a read
- Condition: W + R > N (where N is total nodes)

Common configurations:
- **W=R=N/2+1** (majority quorum): Tolerates N/2 node failures. Standard for consensus systems like Raft and Paxos.
- **W=N, R=1**: Strong consistency (all nodes must write), poor write availability. A single node failure blocks writes.
- **W=1, R=N**: Fast writes, strong reads. Write availability is high (tolerates N-1 failures), but reads are slow.

For availability, you would typically choose W=N/2+1, R=N/2+1 — the majority quorum. This gives the best trade-off between availability and consistency.

## Performance: Concrete Numbers

### Failover Time by Method

| Method | Typical Failover Time | Downtime per Month (at 4 nines budget) |
|---|---|---|
| Manual failover | 5-30 minutes | Exceeds 4 nines (52 min/month) by 10-60x |
| Scripted semi-automated | 2-10 minutes | May exceed 4 nines depending on frequency |
| Automated with DNS update | 60-120 seconds | ~8% of budget per event |
| Automated with VIP move | 10-30 seconds | ~2% of budget per event |
| Active-active (no failover) | 0 seconds (no failover needed) | 0% — traffic shifts automatically |

A single manual failover can consume an entire month's four-nines budget. This is why automation is non-negotiable at four nines and above.

### MTTR and MTTF

Two metrics define availability math:

**MTTF (Mean Time To Failure)**: The average time between failures. If a system fails once every 30 days, MTTF = 30 days.

**MTTR (Mean Time To Recovery)**: The average time to restore service after a failure. If recovery takes 15 minutes, MTTR = 15 minutes.

Availability = MTTF / (MTTF + MTTR)

Example:
- MTTF = 720 hours (30 days)
- MTTR = 0.5 hours (30 minutes)
- Availability = 720 / 720.5 = 99.93%

To reach 99.99% with the same MTTF:
- Required MTTR = 720 * (1/0.9999 - 1) = 720 * 0.0001 = 0.072 hours = 4.32 minutes

The path to five nines is either making MTTF astronomically high (99.999% requires MTTF = 720 hours, MTTR = 0.0072 hours = 26 seconds) or reducing MTTR to near zero through active-active architectures.

### Cost of Each Nine

| Availability Level | Approximate Cost Multiplier | Typical Infrastructure |
|---|---|---|
| 95% | 1x | Single server, no redundancy |
| 99% | 1.5x | Single server with RAID, backup |
| 99.9% | 2-3x | Load-balanced servers, database replica |
| 99.95% | 3-5x | Multi-AZ, automated failover |
| 99.99% | 5-10x | Multi-region active-passive |
| 99.999% | 10-20x | Multi-region active-active, chaos engineering, dedicated SRE team |

The cost increases exponentially because each additional nine requires addressing an increasingly rare set of failure modes: power distribution failures, network partition, region-wide outages, cascading failures, and human error during incident response.

## Failure Detection Algorithms in Practice

### Phi Accrual Implementation Details

The phi accrual failure detector is widely used in production systems. Here is how it works at the implementation level:

The detector maintains a sampling window of the last N heartbeat inter-arrival times (typically N = 1000). From these samples, it computes:

- Mean arrival interval (mu): The average time between consecutive heartbeats.
- Variance (sigma-squared): How much arrival times vary.

When the expected heartbeat does not arrive by time t, phi is calculated as:

```
phi = -log10(P(likelihood of interval >= t))

where P assumes a normal distribution with the observed mu and sigma.

Example calculation:
  mu = 1000ms (heartbeat interval)
  sigma = 200ms (some variation in network)
  t since last heartbeat = 2500ms
  
  Standard deviation units from mean: (2500 - 1000) / 200 = 7.5
  P(interval >= 2500ms) = approximately 3.2e-14 (from normal CDF)
  phi = -log10(3.2e-14) = 13.5
  phi = 13.5 >= 8 → declare node DEAD
```

Advantages over fixed-threshold health checks:

- **Adaptive to network conditions**: A high-latency environment naturally has larger mu and sigma, so acceptable gaps are wider. A low-latency environment detects failure faster.
- **Configurable suspicion level**: phi = 1 is a weak suspicion (10% chance the node is dead). Phi = 8 is a strong suspicion. Different thresholds can be used for different actions (phi = 3: stop routing traffic; phi = 8: declare node dead).
- **No hard thresholds**: No need to tune "3 failures in 15 seconds" — the algorithm tunes itself from observed behavior.

### SWIM Protocol

SWIM (Scalable Weakly-consistent Infection-style Protocol) improves on basic gossip. It separates failure detection from membership dissemination:

1. **Failure detection**: Each node periodically picks a random target and sends a ping. If the target does not respond within a timeout, the pinger sends indirect pings through K other nodes (typically K = 3). If all indirect pings also fail, the target is declared SUSPECT and this is disseminated through the gossip.

2. **Membership dissemination**: Suspect and Alive messages propagate through infection-style gossip. Each node periodically gossips its full membership view to a random peer.

SWIM provides provable convergence: the probability that a node remains undetected after a round of gossip decreases exponentially with the number of rounds. Expected detection time: O(log N / log log N) rounds.

### Failure Detection in Kubernetes

Kubernetes uses a simplified failure detection model:

- **kubelet** reports node status to the control plane every `--node-status-update-frequency` (default 10 seconds).
- The **node controller** checks node status every `--node-monitor-period` (default 5 seconds).
- If the node controller does not receive a status update for `--node-monitor-grace-period` (default 40 seconds), it marks the node as UNKNOWN.
- After `--pod-eviction-timeout` (default 5 minutes), the node controller evicts pods from the unknown node.

Total time from node failure to pod reschedule: approximately 5 minutes and 40 seconds. This is conservative to avoid premature eviction during transient network partitions. For faster detection, use liveness probes at the pod level (not the node level).

Key tuning parameters:

- **Gossip interval**: How often each node initiates a gossip round (typically 100ms-1s). Shorter intervals mean faster convergence but more network traffic.
- **Fan-out**: How many peers each node contacts per round (typically 2-3). Higher fan-out converges faster but increases bandwidth per node proportionally.
- **Full vs incremental sync**: Full sync sends the entire membership table (linear in N). Incremental sync sends only changes since last sync (typically constant size). Incremental sync is preferred for large clusters.

## Raft Deep Dive: Log Replication and Availability

Raft provides a practical implementation of distributed consensus with strong availability guarantees for the leader. Understanding Raft's log replication is essential for designing highly available databases.

Raft availability characteristics:

- **Tolerates minority failures**: A 3-node Raft cluster can tolerate 1 node failure and still serve writes. A 5-node cluster tolerates 2 failures.
- **Election timeout**: If the leader fails, followers wait for a randomized election timeout (typically 150-300ms) before starting a new election. The randomization prevents multiple followers from starting elections simultaneously (avoids split votes).
- **Read availability**: Raft can serve reads from the leader (linearizable reads) or from followers (stale reads). For maximum read availability, implement follower reads with periodic safety checks against the leader's commit index.

## Timeouts and Deadlines in Distributed Systems

### The Tail-at-Scale Problem

In a system with 1,000 servers, if each server has a 1% chance of being slow on any request, the probability that at least one server is slow for a given request is:

```
P(no server slow) = (0.99)^1000 = 0.00004
P(at least one slow) = 1 - 0.00004 = 99.996%
```

In a system of 1,000 servers, nearly every request experiences at least one slow server. This is the tail-at-scale problem, and it directly affects availability: a system where every request is slow is effectively unavailable for latency-sensitive use cases.

Solutions:
- **Hedged requests**: Send the same request to two servers. Use the first response. Cancel the second. This reduces p99 latency at the cost of doubling server load.
- **Request hedging by percentile**: If the request is not complete by the p50 latency, send a hedged request. If the original completes first, cancel the hedged request. This adds only 50% load on average instead of 100%.
- **Tied requests**: Send the request to N servers and wait for the first K responses. K=1 gives the fastest response. K=quorum gives strong consistency.

### Connection Pool Exhaustion

Connection pools are a common availability bottleneck. Each database connection consumes a thread on the database server and a file descriptor on both sides.

```
Connection pool config:
maxTotal: 50
maxWaitMillis: 5000
minIdle: 5

Under normal load:
- 30 connections active
- 20 connections idle
- No queuing

Under slow-query load:
- 50 connections active (all busy)
- 0 idle connections
- New requests wait 5000ms for a connection
- After 5000ms, requests throw ConnectionTimeoutException
- User-visible errors: HTTP 503
- Effectively unavailable for writes
```

Prevention strategies:
- **Separate pools for critical paths**: Read pool (max 30), write pool (max 20). A slow analytical query does not starve the write path.
- **Circuit breaker for slow queries**: If a query takes longer than 1 second, terminate it and mark the query pattern as suspicious.
- **Dynamic pool sizing**: Reduce pool size when latency increases to prevent queue buildup. This is counterintuitive — smaller pools under load — but it prevents saturation and preserves throughput.

### Timeout Cascades

A timeout cascade occurs when a downstream service slowdown propagates upward through the call chain, amplifying the impact.

     DB slow 1500ms  →  Payment wait 1500ms  →  Web wait 1500ms  →  Gateway wait 1500ms

  `}

Preventing timeout cascades requires:
- **Cascading timeouts**: Each upstream timeout must be shorter than the downstream timeout. API Gateway (5000ms) > Web App (3000ms) > Payment (2000ms) > Database (1000ms). This way, the inner service times out first and releases its thread.
- **Circuit breaker per dependency**: A slow payment service is isolated. It does not affect the web app's ability to call other services.
- **Bulkheads**: Separate thread pools per downstream dependency. A slow payment service exhausts only the payment thread pool, not the shared thread pool.

### Asynchronous Processing for Availability

Synchronous request-response patterns couple the caller's availability to the callee's availability. Asynchronous processing decouples them:

- **Request acceptance**: The system accepts the request, writes it to a queue, and returns "202 Accepted" immediately.
- **Background processing**: A worker picks up the request from the queue, processes it, and stores the result.
- **Result retrieval**: The client polls for the result or receives a callback.

```
Timeline:
Client ──▶ POST /order ──▶ 202 Accepted {order_id: 123}
                     Queue: <order_123>
                              Worker picks up: processes
                              Worker writes to: ((DB))
Client ──▶ GET /order/123 ──▶ 200 {status: "processing"}
Client ──▶ GET /order/123 ──▶ 200 {status: "completed", data: {...}}
```

The advantage for availability: if the payment processor is slow, the queue buffers the orders. The client sees an immediate acceptance. The system continues accepting requests even when downstream services are degraded.

### Health Check Configuration

| Parameter | Recommended Value | Notes |
|---|---|---|
| Check interval | 5 seconds | Lower values increase CPU/network load |
| Healthy threshold | 2 consecutive successes | Avoids flapping |
| Unhealthy threshold | 3 consecutive failures | Prevents false positives |
| Timeout | 2 seconds | Must be less than check interval |

| Parameter | Recommended Value | Notes |
|---|---|---|
