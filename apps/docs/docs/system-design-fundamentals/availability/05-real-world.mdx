# Availability in the Real World

## Amazon: Multi-AZ Everything

Amazon pioneered the region-and-availability-zone model that has become the standard for cloud availability. An AWS Region (e.g., us-east-1) contains multiple isolated Availability Zones (AZs) — physically separate data centers with independent power, cooling, and networking, connected by low-latency fiber.

Amazon's key availability mechanisms:

- **ELB cross-zone load balancing**: If AZ-1 has 2 healthy instances and AZ-2 has 10, ELB sends 50% of traffic to each AZ. This prevents traffic spikes from overwhelming a single AZ.
- **RDS Multi-AZ**: Synchronous replication to a standby in a different AZ. Failover is automatic within 60-120 seconds. The DNS record for the DB endpoint updates to point to the new primary.
- **Auto-scaling across AZs**: EC2 Auto Scaling groups spread instances across AZs. If one AZ fails, the group replaces instances in a different AZ. Health checks remove unhealthy instances automatically.
- **IMDS (Instance Metadata Service)**: Even EC2 metadata endpoints are highly available — two hardened IP addresses served by redundant infrastructure.

The Simian Army includes:
- **Chaos Monkey**: Randomly terminates EC2 instances
- **Latency Monkey**: Introduces artificial delays to simulate degraded performance
- **Conformity Monkey**: Finds instances that do not conform to best practices and shuts them down
- **Doctor Monkey**: Detects unhealthy instances and reports to the operator
- **Security Monkey**: Finds security vulnerabilities and violations

Netflix's philosophy: if you are not testing failures constantly, you will fail when it matters. By running Chaos Monkey during business hours, teams are incentivized to build self-healing systems that require no human intervention.

## Open Source Availability Tools

### Kubernetes Health Checks

Kubernetes provides two types of probes:

- **livenessProbe**: Determines if the pod is alive. If it fails, kubelet restarts the container.
- **readinessProbe**: Determines if the pod is ready to serve traffic. If it fails, the pod is removed from Service endpoints.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  containers:
  - name: app
    image: myapp:1.0
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
      failureThreshold: 3
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 2
```

Kubernetes also supports **startupProbe** (for slow-starting containers) and **preStop hooks** (for graceful shutdown).

### HAProxy

HAProxy is the gold standard for open-source load balancing. Key availability features:

- **Health checking**: HTTP, TCP, and custom health checks. Configurable intervals and thresholds.
- **Connection draining**: Gracefully remove backends by setting a maintenance weight of 0.
- **Backup servers**: Designate a backup server that takes over if all primary servers fail.
- **Sticky sessions**: Optional session persistence with configurable timeout.

```haproxy
backend web-servers
    balance roundrobin
    option httpchk GET /health HTTP/1.1\r\nHost:\ web.example.com
    default-server check inter 5s fall 3 rise 2
    server app1 10.0.1.10:80 check
    server app2 10.0.1.11:80 check
    server app-backup 10.0.1.20:80 check backup
```

### Keepalived + VRRP

Keepalived implements VRRP (Virtual Router Redundancy Protocol) for IP failover. Two or more servers share a virtual IP. If the master fails, the backup takes over the IP.

```
Master:                       Backup:
┌──────────────────────┐      ┌──────────────────────┐
│ VIP: 10.0.0.100      │      │ (no VIP by default)  │
│ VRRP Priority: 100   │      │ VRRP Priority: 90    │
│ Sends VRRP           │ ────▶│ Monitors VRRP        │
│ advertisements       │      │ advertisements       │
│ every 1 second       │      │ If 3 missed:         │
└──────────────────────┘      │ take over VIP        │
                              └──────────────────────┘
```

VRRP failover takes 3-5 seconds (3 missed advertisements at 1-second intervals plus a skew time). This is fast enough for many internal services but too slow for public-facing web traffic — which is why Cloud Load Balancers (ELB, GCP HTTP LB) use different mechanisms.

## Case Study: Stripe — Idempotency Keys

Stripe processes billions of dollars in payments. Their API must be highly available — a payment authorization failure at a critical moment means a lost sale. One of Stripe's key availability mechanisms is **idempotency keys**.

When a client sends a payment request, it includes an `Idempotency-Key` header with a unique value:

```
POST /v1/charges
Idempotency-Key: 4b9b2d9a-7a9f-4e6e-9a9e-1a2b3c4d5e6f
Content-Type: application/json
{
  "amount": 2000,
  "currency": "usd",
  "source": "tok_visa"
}
```

If the request fails with a network error, the client retries with the same idempotency key. Stripe's API detects the duplicate key and returns the original response — the charge is not duplicated.

Why this is critical for availability: without idempotency keys, a retry-after-timeout pattern is dangerous. If the first request succeeded but the response was lost, the retry creates a duplicate charge. This forces clients to choose between:
- **Retry**: Risk duplicate charges (bad for users)
- **Do not retry**: Risk failed charges when the first request actually failed (bad for revenue)

Idempotency keys eliminate the dilemma. Clients can retry aggressively without side effects.

## Case Study: GitHub 2018 Outage — What You Can Learn

On October 21, 2018, GitHub experienced a 24-hour, 11-minute outage affecting their core services (web UI, APIs, Git operations). The root cause is instructive for every engineer designing for availability.

**What happened**: A planned maintenance operation — replacing a failing 100G optical trunk between US East Coast data centers — caused a brief network partition. Normally, this would be fine: redundant paths should handle it. But a combination of factors caused cascading failure:

1. The partition split the database cluster, causing MySQL to enter a split-brain recovery mode.
2. The automated failover triggered, but the new primary was in a degraded state.
3. When the partition healed, the two database sides had inconsistent data.
4. The manual recovery process took hours because the database cluster's state was complex and the team exercised extreme caution to avoid data loss.

**Key lessons from the postmortem**:

1. **Redundant paths are not enough**: GitHub had redundant network paths between data centers, but the failover was not fast enough to prevent database impact. Test network redundancy under realistic conditions, including the exact failure scenario.

2. **Database failover complexity**: Automated database failover is hard to get right, especially with multi-terabyte datasets and complex replication topologies. GitHub's postmortem recommended faster failover tooling and better testing of partial network failures.

3. **Alerting gaps**: Some critical alerts were missing or incorrectly configured. The team did not know about the database state degradation until customers reported problems. Review alert coverage regularly, especially for edge cases.

4. **Coordinated maintenance windows**: GitHub now coordinates infrastructure changes more carefully. A single fiber maintenance should not cause a global outage — even if the fiber is in a critical path, automated failover should handle it.

Each major cloud provider offers a slightly different set of availability tools. The table below compares how they handle the core availability mechanisms.

### Choosing Between Cloud Providers for Availability

The choice depends on your availability requirements:

- **Best for global anycast**: GCP HTTP LB. A single anycast IP that routes to the nearest healthy backend globally. No DNS propagation delay during failover. The IP does not change.
- **Best for database failover**: AWS RDS Multi-AZ with Aurora. Automated failover within 60-120 seconds. Synchronous replication within the region. Global Database for cross-region reads with under 1 second lag.
- **Best for strong consistency across regions**: Google Spanner. Synchronous replication with Paxos consensus across regions. Strong consistency at global scale. Write latency: 50-200ms depending on distance.
- **Best for simplicity**: Azure Traffic Manager with Azure SQL. DNS-based failover with configurable routing methods (priority, weighted, performance). Suitable for most enterprise workloads.

## Five Lessons from Production

### Lesson 1: Test Failover Regularly

The failover that has never been tested will fail when you need it most. Schedule quarterly failover drills. Automate them. Include them in your on-call training.

A well-run failover drill follows this pattern:

1. **Announce the drill** to all stakeholders 1 week in advance. The drill happens during business hours.
2. **Start the drill**: The chaos engineer terminates the primary database or kills the load balancer.
3. **Observe**: The on-call engineer receives the alert, follows the runbook, and executes the failover.
4. **Measure**: Time to detect, time to diagnose, time to recover. Compare against SLO.
5. **Debrief**: What went well? What went wrong? Update the runbook. Track metrics over time to see improvement.
6. **Rotate**: Make different team members the primary responder each quarter. Everyone should practice.

### Lesson 2: Measure Everything

You cannot improve availability you do not measure. Track:
- SLI: request success rate, latency percentiles
- Time to detect failure
- Time to diagnose root cause
- Time to restore service
- Frequency of false-positive alerts

A measurement practice that has proven effective at Google is the "four golden signals" for monitoring:

1. **Latency**: Time to serve a response. Track both p50 and p99. A p50 of 50ms with p99 of 5 seconds means 1% of users have a terrible experience.
2. **Traffic**: Request rate per second. Watch for unexpected spikes or drop-offs.
3. **Errors**: Explicit failures (5xx) and implicit failures (successful response but wrong result).
4. **Saturation**: How "full" the system is. CPU, memory, disk, connection pool utilization. Saturation at 90%+ is a leading indicator of availability problems.

### Lesson 3: Design for Partial Failures

A single server failing should not cause a full outage. Design so that N-1 servers (or N-1 VMs, N-1 databases) can handle the full load. This means running at no more than 66% capacity during normal operation for an N+1 design.

Partial failure design includes:

- **Graceful degradation**: Define what features work and what features are disabled at each degradation level. Level 1: Cache enabled, no personalization. Level 2: Read-only mode (show data, no writes). Level 3: Static fallback page.
- **Bounded queues**: Every queue must have a maximum size and a rejection policy. An unbounded queue hides the failure until memory is exhausted — then the failure is catastrophic.
- **Client-side timeouts**: Every external call must have a timeout. The timeout should be tight enough that a slow dependency does not cascade. A database call that takes 30 seconds instead of 10ms should not block 30 application server threads.

### Lesson 4: Reduce Blast Radius

When a failure happens, limit its impact. Use bulkheads: separate connection pools, thread pools, and memory limits for different dependencies. A slow database query should not block cache lookups for other users.

Practical blast radius reduction:

- **Cell-based architecture**: Divide the system into isolated cells. Each cell serves a subset of users (e.g., 10,000 users per cell). A failure in one cell affects only those 10,000 users. Cells communicate through asynchronous event buses, not synchronous calls.
- **Shuffle sharding**: A refinement of cell-based architecture. Each user's request touches only a subset of servers (e.g., 5 out of 100). A failure in 1 server affects only 5% of users — but each user is affected only 5% of the time.
- **Circuit breaker per dependency**: If the payment service is slow, the circuit breaker opens for payment calls. The checkout service can still render the cart page — it just shows "payment temporarily unavailable" instead of timing out.

### Lesson 5: Humans Are the Weakest Link

Automated failover is more reliable than manual failover. Every minute of manual intervention is a minute where the engineer might mistype a command, misread a dashboard, or misdiagnose the problem. Automate the entire failover sequence, and verify it works with regular drills.

A study of large-scale outages at major cloud providers found that approximately 70% were caused or exacerbated by human error during incident response. The irony: most of these errors happened because the engineer was trying to fix an automated system that had already failed. The fix is not "train humans better" — the fix is "automate more and test automation frequently."



### Lesson 6: Plan for the Unexpected

The most dangerous availability failure is the one you did not plan for. Common unexpected failures:

- **DNS provider outage**: Your DNS provider (Route53, CloudDNS, Cloudflare) goes down. You cannot update DNS records. Your failover mechanism uses DNS updates. You are stuck.
  - Mitigation: Have a secondary DNS provider. Use a multi-provider DNS configuration where both providers serve the same zone.

- **Certificate expiration**: The TLS certificate for your load balancer expires on a Sunday at 3 AM. All HTTPS traffic fails. Certificate auto-renewal was configured but the ACME client could not reach the CA due to a network change.
  - Mitigation: Monitor certificate expiration dates with 30-day, 7-day, and 1-day alerts. Automate renewal testing — run a mock renewal weekly.

- **IAM / Auth provider failure**: AWS IAM or your OIDC provider is slow or unavailable. New EC2 instances cannot authenticate. Auto scaling cannot add capacity.
  - Mitigation: Use instance profiles with cached credentials (AWS IMDS). Pre-authenticate instance launch templates. Keep a pool of warm instances with valid credentials.

- **Billing issue**: Your cloud account is suspended due to a billing issue. An automated payment failed. All resources are stopped.
  - Mitigation: Configure billing alerts with multiple notification channels. Have a manual override process for critical billing issues. Use a separate billing account for production vs non-production.
