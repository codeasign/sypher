# Availability Challenge

## Tier 1: Core Concepts (Foundation)

### Challenge 1.1 — The Three-Nines Calculator

Your e-commerce platform serves 5 million requests per day with an average revenue of $0.50 per request. The system runs on a single server that achieves 98% uptime.

**Questions:**

A. How many requests fail per day due to downtime?

B. What is the daily revenue lost?

C. If you upgrade to a load-balanced pair of servers achieving 99.5% uptime, what is the new daily revenue loss?

D. The upgrade costs $24,000 per year. How many days until it pays for itself?

**Model Answer:**

A. Daily uptime at 98% = 24 hours * 0.98 = 23.52 hours. Downtime = 0.48 hours = 28.8 minutes = 1,728 seconds. Requests per second = 5,000,000 / 86,400 = 57.87 RPS. Lost requests = 57.87 * 1,728 = 100,000 requests per day.

B. Daily revenue lost = 100,000 * $0.50 = $50,000 per day.

C. At 99.5% uptime, downtime = 24 * 0.005 = 0.12 hours = 7.2 minutes = 432 seconds. Lost requests = 57.87 * 432 = 25,000. Daily revenue lost = $12,500. Savings = $50,000 - $12,500 = $37,500 per day.

D. The upgrade pays for itself in $24,000 / $37,500 = 0.64 days. About 15 hours.

### Challenge 1.2 — SPOF Spotting

List all the SPOFs in this architecture:

- One web server running Nginx + Node.js
- One PostgreSQL database on the same server, storing data on a single SSD
- DNS with a single A record pointing to the server's IP, TTL = 3600 seconds
- One network switch connecting the server to the internet
- One power supply unit
- Daily backup to S3

**Model Answer:**

1. **Web server**: Single server. If it fails, the application is completely unavailable.
2. **Nginx + Node.js co-located**: If Node.js crashes, Nginx has no upstream to proxy to.
3. **PostgreSQL on the same server**: If the server goes down, both app and database are gone. No separate failover possible.
4. **Single SSD**: If the disk fails, all data is lost (backup gap of up to 24 hours).
5. **DNS single A record with 3600-second TTL**: If the IP changes, DNS propagation takes at least 1 hour. During that time, traffic goes to the dead IP.
6. **Single network switch**: If it fails, the server is disconnected.
7. **Single power supply**: If it fails, the server powers off.

The S3 backup is the only partial redundancy — but RPO (Recovery Point Objective) is up to 24 hours.

### Challenge 1.3 — Replication Lag Impact

A social media app has 200,000 daily active users. The system uses a primary database for writes and an async read replica for reads. Replication lag averages 800ms but spikes to 3 seconds during peak traffic. A user posts a comment and immediately refreshes the page.

**Questions:**

A. What does the user see when they refresh?

B. The user refreshes again 3 times over 10 seconds and sees their comment appear and disappear. Why?

C. How would you fix this to provide read-your-writes consistency?

**Model Answer:**

A. The user sees the page without their comment 80% of the time. The async replication has not yet copied the comment to the read replica. The read replica serves the query and returns state before the comment was written.

B. The comment appears and disappears because the read replica's data is lagging but not uniformly stale. During the first refresh, the replica might be at the right log position and shows the comment. During the second refresh, the replica has fallen behind and does not show it. This happens because replication is asynchronous and the lag varies based on replica resource contention and write volume on the primary.

C. Fix options:

1. **Session stickiness with delay**: For 5 seconds after a user writes, route their reads to the primary. This guarantees read-your-writes consistency for the writing user. Other users continue reading from the replica.

2. **Timestamp-based routing**: Each user response includes a timestamp of the last write. The client includes this timestamp in subsequent requests. If the replica's data timestamp is older than the client's last write timestamp, the read is routed to the primary.

3. **Quorum reads**: Read from both the primary and the replica. If the data matches, return it. If they differ, return the primary's data and alert on the discrepancy.

4. **Increase replica capacity**: If the lag is caused by insufficient replica capacity, scale up the replica (more CPU, more memory, or add additional replicas and distribute reads across them).

### Challenge 1.4 — SLA Math

Your service has a monthly SLA of 99.95%. It is the 20th day of a 30-day month. So far this month, the service has experienced 18 minutes of unplanned downtime.

A. How much additional downtime can you afford without breaching the SLA?

B. An incident is detected that will take 25 minutes to resolve. If you proceed with the fix, will you breach the SLA?

**Model Answer:**

A. Total allowed downtime per month = 30 days * 24 hours * 60 minutes = 43,200 minutes. At 99.95%, allowed downtime = 43,200 * (1 - 0.9995) = 21.6 minutes. Already used: 18 minutes. Remaining: 3.6 minutes.

B. A 25-minute fix would consume 25 minutes of downtime. Total would be 18 + 25 = 43 minutes. This exceeds the 21.6-minute budget by 21.4 minutes. The SLA would be breached. You should look for a faster recovery option (e.g., rollback, serve degraded, redirect traffic) or accept the SLA breach and calculate the penalty.

## Tier 2: Design Application (Intermediate)

### Challenge 2.1 — Design for 99.99% Availability

A ride-sharing platform handles 10,000 write requests per second (ride requests, location updates) and 50,000 read requests per second (driver locations, ETAs). The business requires 99.99% availability. The engineering budget is $5 million per year.

Design the availability architecture. Include:

- Application tier redundancy
- Database tier redundancy
- Multi-region strategy
- Failover mechanisms with timing
- Monitoring and alerting

**Model Answer:**

**Application tier**: Use Kubernetes with 100 service replicas spread across 3 AWS availability zones. Run at 60% capacity (60 pods handle full load, 40 provide headroom). Health checks: liveness probe every 10 seconds, readiness probe every 5 seconds with deeper validation (DB connectivity). Circuit breakers for downstream services (matching service, payment service).

**Database tier**: Use a distributed database with multi-AZ support. For the ride-matching database, use Aurora Global Database with a primary in us-east-1 and read replicas in us-west-2 and eu-west-1. Writes go to the primary. Reads are served from the nearest replica. Failover: Aurora automated failover promotes a replica in 30 seconds.

**Multi-region**: Active-passive architecture. Primary region: us-east-1. Standby region: us-west-2. Route53 with latency-based routing and health checks. Failover trigger: 3 consecutive health check failures on the ALB, the database, and the core API endpoint. Estimated failover time: 25 seconds detection + 30 seconds DB promotion + 10 seconds DNS = 65 seconds worst-case. This consumes 65 seconds of the monthly 4-nines budget of 4.38 minutes (263 seconds) — leaving 198 seconds for other incidents.

**Monitoring**: Real-time dashboards tracking request success rate, p99 latency, and replication lag. Alerts at 99.995% (breach of internal SLO) with PagerDuty escalation. Error budget burn rate alerts: if the 30-day rolling availability drops below 99.99%, the on-call engineer is notified within 1 minute.

### Challenge 2.2 — Chaos Engineering Plan

Design a chaos engineering plan for a microservices system with 20 services. Describe:

- Which failure scenarios to test
- The order of testing (least risky to most risky)
- Success criteria (what proves the system survived)
- Rollback plan if chaos causes real damage

**Model Answer:**

**Phase 1 — Single instance failures** (week 1):
Shut down one pod of each service in 60-minute intervals. Verify that remaining pods handle the load without latency increase exceeding 10%. No user impact expected.

**Phase 2 — AZ failure** (week 2):
Terminate all instances in one availability zone. Verify that services in remaining AZs handle full traffic. Run for 30 minutes. Success criteria: no p99 latency increase above 500ms, no error rate above 0.1%.

**Phase 3 — Database replica failure** (week 3):
Kill the read replica. Verify that reads fall through to the primary. Kill the primary (without replication traffic). Verify that automated failover promotes the replica within 60 seconds. Success criteria: write availability returns within 90 seconds, zero data loss.

**Phase 4 — Dependency failure** (week 4):
Block traffic to each downstream service (payment, mapping, notifications). Verify that circuit breakers open within 10 seconds. Verify that the system degrades gracefully (no crash, specific features restricted). Success criteria: core functionality (ride matching, ETA display) continues working.

**Phase 5 — Full region failover** (week 5):
Simulate a complete region outage. Cut all traffic to us-east-1. Verify that Route53 failover directs traffic to us-west-2. Verify that the standby database promotes and serves writes. Run for 2 hours. Success criteria: full service restored within 5 minutes, zero data loss, all features operational within 10 minutes.

**Rollback plan**: All chaos experiments include a kill switch. If error rate exceeds 1% or p99 latency exceeds 1 second, the experiment halts automatically. A human operator reviews and reverts any changes within 60 seconds.

### Challenge 2.4 — Design a Multi-Region Rate Limiter

Design a rate limiter that works across two regions. The rate limit is 100 requests per second per user. The system must enforce this limit even when a user's requests are distributed across both regions.

**Questions:**

A. How do you share rate limit state across regions without introducing a SPOF?

B. What happens if the cross-region sync fails? Do you accept over-rate-limit or under-rate-limit?

C. What is the worst-case deviation from the 100 RPS limit during a partial failure?

**Model Answer:**

A. Use a **token bucket algorithm with local caches and periodic sync**. Each region maintains a local token bucket for each user. The bucket is refilled from a global counter in a distributed database (DynamoDB, Cassandra, or Redis) every 1 second.

```
Local bucket (Region A):              Global counter (DynamoDB):
┌────────────────────────┐           ┌───────────────────┐
│ User 1: 45 tokens      │  sync    │ User 1: 87/100    │
│ User 2: 12 tokens      │  every   │ User 2: 55/100    │
│ User 3: 33 tokens      │  1s      │ User 3: 78/100    │
└────────────────────────┘  ◀───────└───────────────────┘
```

Each region:
1. On request arrival, decrements the local bucket if tokens are available.
2. Every 1 second, syncs local consumption to the global counter.
3. Refreshes the local bucket from the global counter.

The global counter is DynamoDB with strong consistency reads. Each region writes its consumption every second. If a region cannot reach DynamoDB, it continues using its local bucket without refreshing. It accepts the risk of over-approving requests.

B. **Prefer over-rate-limit to under-rate-limit**. Over-rate-limiting (allowing more than 100 RPS) protects availability. If you strictly enforce the limit during a cross-region failure, you might block legitimate traffic from healthy regions. The trade-off: accept slight overage during failures, correct it when sync resumes.

The decision depends on the cost of overage vs the cost of false blocks:

- For an API where 10 extra requests per second causes no harm: over-accept.
- For a payment API where over-approval has financial consequences: over-reject.

C. **Worst-case deviation**: If cross-region sync fails for 10 seconds and both regions are at peak:

- Each region's local bucket starts with 100 tokens.
- Each region consumes at 100 RPS.
- After 10 seconds, each region has consumed 1,000 tokens.
- Global limit: 10 seconds * 100 RPS = 1,000 tokens.
- Actual consumption: 2,000 tokens.
- Over-age: 100%.

For a 10-second sync failure, the system allows 2x the rate limit. This is acceptable for most APIs. For tighter enforcement, reduce the sync interval to 500ms (reduces max overage to 50%) or use a centralized rate limiter with local caching (reduces overage to a few tokens).

## Tier 3: Deep Analysis (Advanced)

### Challenge 3.1 — The Split-Brain Problem

You are designing an active-active database setup across two regions. Each region accepts writes. The replication link between regions has an average latency of 80ms. During a network partition, writes continue in both regions for 15 minutes. After the partition heals, you must reconcile the diverged data.

**Questions:**

A. What conflict resolution strategy would you use?
B. How would you handle writes to the same user account in both regions?
C. What is the worst-case data divergence in terms of number of new records?
D. How would you communicate this to customers?

**Model Answer:**

A. Conflict resolution strategy depends on the data type:

- **User profile fields** (name, email): Last-writer-wins (LWW) with timestamp comparison. The write with the later timestamp wins. Risk: if both regions update the same field simultaneously, one update is lost. Acceptable for non-critical fields.

- **Ride records** (new ride bookings): Assign globally unique ride IDs (UUID v4, probability of collision: effectively zero). Each region generates its own ride IDs. Conflict resolution is not needed for new records — they merge without conflict because they represent different rides.

- **Account balance** (financial data): LWW is not acceptable. Use a CRDT (Conflict-Free Replicated Data Type) approach: an operation-based counter where each region records credit/debit operations rather than the balance value. After partition heals, the operation log is merged — the final balance is the sum of all operations. This is safe but requires a different data model.

B. For writes to the same user account:

Implement per-field LWW with vector clocks. Each write includes a vector clock that tracks causality. If writes are concurrent (vector clocks are incomparable), apply a deterministic merge (e.g., if it is a balance update, sum the operations; if it is a profile update, keep both versions as a multi-value register).

For high-contention scenarios (e.g., coupon code redemption, limited inventory), use a lease-based approach: the user's "home region" owns the account. All writes for that user go through the home region. Reads can be served from any region with a read-replica approach.

C. Worst-case data divergence calculation:

Assume 10,000 writes per second per region. Both regions write for 15 minutes (900 seconds). Total new records: 2 regions * 10,000 RPS * 900 seconds = 18 million new records. Each record is approximately 1KB. Total data divergence: 18 GB.

Merge time depends on the conflict resolution strategy. With LWW, merge is fast (compare timestamps, O(N)). With CRDTs, merge is also O(N). With manual resolution, merge time is proportional to the number of conflicts (the subset of 18M records that have concurrent writes to the same logical key).

D. Customer communication strategy:

- During the outage, a status page message: "We are experiencing elevated error rates. Your data is safe. We will apply updates as they synchronize."
- After the partition heals: "All systems are operational. Recent data may show slight delays in synchronization across regions. Your account reflects your most recent activity. If you notice discrepancies, contact support."
- For affected transactions (if LWW caused data loss): Identify affected users via audit logs. Send personalized emails explaining the situation and offering compensation (ride credits, refunds for affected transactions).

### Challenge 3.2 — Cost-Benefit Analysis

Your startup runs a video streaming service with 500,000 daily active users. Current monthly infrastructure cost: $80,000. Current availability: 99.5% (using multi-AZ active-passive). You are considering two upgrades:

**Option A**: Improve to 99.9% by adding read replicas in 3 regions and implementing automated failover. Additional cost: $35,000/month.

**Option B**: Improve to 99.99% by building a full multi-region active-active architecture. Additional cost: $120,000/month.

The average revenue per user per month is $4. The cost of downtime is $3,000 per minute in direct revenue loss plus $5,000 per minute in estimated reputational damage. Your current monthly burn rate is $150,000, and you have $2 million in funding remaining.

**Questions:**

A. Calculate the cost-benefit ratio of each option.
B. Which option would you recommend, and why?
C. How does the runway affect your decision?

**Model Answer:**

A. **Current state (99.5%)**: Monthly downtime = 30 * 24 * 60 * 0.005 = 216 minutes. Cost of downtime = 216 * ($3,000 + $5,000) = $1,728,000/month. This is clearly unacceptable — the cost of downtime exceeds revenue. The calculation shows why 99.5% is not enough for a revenue-critical service.

**Option A (99.9%)**: Monthly downtime = 43,200 * 0.001 = 43.2 minutes. Cost of downtime = 43.2 * $8,000 = $345,600/month. Benefit vs current: $1,728,000 - $345,600 = $1,382,400/month savings. Additional cost: $35,000/month. ROI = $1,382,400 / $35,000 = 39.5x. Payback period: instant (first month).

**Option B (99.99%)**: Monthly downtime = 43,200 * 0.0001 = 4.32 minutes. Cost of downtime = 4.32 * $8,000 = $34,560/month. Benefit vs Option A: $345,600 - $34,560 = $311,040/month savings. Additional cost vs Option A: $120,000 - $35,000 = $85,000/month. ROI = $311,040 / $85,000 = 3.66x.

B. Recommended option: Start with Option A. The ROI is 39.5x with minimal implementation complexity (adding read replicas is well-understood). Option B provides additional benefit but at much lower ROI. Once the company reaches $10M+ monthly revenue, the marginal benefit of Option B becomes more attractive. At that scale, downtime costs will be proportionally higher.

C. With $2M in funding and $150K/month burn rate, the runway is $2M / ($150K + infra cost). Current runway: $2M / ($150K + $80K) = 8.7 months. Option A runway: $2M / ($150K + $115K) = 7.5 months. Option B runway: $2M / ($150K + $200K) = 5.7 months.

Option A reduces the runway by only 1.2 months while drastically reducing downtime cost. Option B reduces runway by 3 months. Given that Option A provides 39.5x ROI and preserves more runway, it is the clear winner for a startup. Option B can be revisited after the next funding round.

### Challenge 3.3 — Design the Incident Response Runbook

Design a runbook for a scenario where all application servers in the primary region become unresponsive. The system uses an active-passive architecture with a secondary region. The load balancer is an AWS ALB in front of an Auto Scaling Group.

**Questions:**

A. Write a step-by-step runbook for the on-call engineer.

B. Include specific commands, time estimates for each step, and rollback instructions.

C. What part of the runbook could be automated, and what part requires human judgment?

**Model Answer:**

A. **Runbook: Primary Region Application Server Failure**

```
Severity: SEV1 (full outage)
Expected resolution time: 15-30 minutes
```

**Step 1 — Verify the alert (2 minutes)**
```
1. Check CloudWatch dashboard: ALB-5xx-Count
2. Verify ALB healthy host count: aws elbv2 describe-target-health \
   --target-group-arn arn:aws:elasticloadbalancing:...
3. Confirm the issue is not a false alarm: check 3 health check endpoints
   curl -m 5 https://primary.region/healthz
   Expected: 200 OK
   Actual: 5xx or timeout → confirm failure
```

**Step 2 — Try immediate recovery (5 minutes)**
```
4. Force instance refresh in ASG:
   aws autoscaling start-instance-refresh \
     --auto-scaling-group-name web-app-asg \
     --preferences '{"MinHealthyPercentage": 0}'
   // This terminates and replaces all instances
5. Wait 3 minutes for new instances to register
6. Check healthy host count again
```

**Step 3 — If step 2 fails, failover to secondary region (10 minutes)**
```
7. Log into Route53 console
8. Update health check for primary ALB endpoint
   // Set health check status to UNHEALTHY (this triggers automatic DNS failover)
9. Wait for DNS TTL (60 seconds) + propagation buffer (120 seconds)
10. Verify traffic reaching secondary:
    curl -m 5 https://secondary.region/healthz
    Expected: 200 OK
11. Confirm secondary ALB healthy host count > 0
12. Check error rate on secondary region dashboard
```

**Step 4 — Verify recovery (3 minutes)**
```
13. Run end-to-end test: 
    curl -m 10 https://api.yourapp.com/health/full
    Expected: 200 OK with all dependencies green
14. Check CloudWatch metrics: 5xx < 0.1%, latency under 500ms
15. Notify team via Slack #incidents that failover is complete
```

B. **Timeline summary**:

| Step | Action | Duration |
|---|---|---|
| 1 | Verify alert | 2 min |
| 2 | Instance refresh | 5 min |
| 3 | Failover to secondary | 15 min |
| 4 | Verify recovery | 3 min |
| **Total** | | **25 min** |

**Rollback for Step 2**: If new instances also fail, proceed immediately to Step 3 (failover). Do not retry the instance refresh.

**Rollback for Step 3**: If secondary region also fails (unlikely for independent failure), restore primary region DNS and investigate the scope. Possible coordinated failure: shared dependency (e.g., DNS provider, cloud provider control plane).

C. **Automateable parts**:

- **Step 1**: Alert verification can be automated. A Lambda function periodically runs the health check and compares it against the alert. If the alert is a false alarm, it auto-resolves the incident.
- **Step 2**: The instance refresh command is already scriptable. Automate it with a ChatOps command: `/runbook app-server-failure refresh-asg`.
- **Step 4**: Verification can be automated. A post-failover script runs health checks and reports pass/fail.

**Requires human judgment**:

- **Step 3 (failover decision)**: A human should decide whether to trigger a full region failover. Automated failover risks false positives — a misconfigured health check could fail over an entire region. Semi-automated: the system prepares the failover (DNS records ready, secondary scaled up) and prompts the on-call engineer to approve.
- **Rollback decisions**: When to give up on Step 2 and proceed to Step 3 is a judgment call based on the engineer's assessment of root cause.
