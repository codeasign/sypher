## The Doorman Problem

Imagine you run a hotel with 20 different departments: reservations, housekeeping, room service, maintenance, billing, concierge, laundry, valet, gym, pool, spa, events, security, IT support, HR, accounting, marketing, PR, legal, and management. Each department has its own entrance, its own set of rules about who can enter, its own working hours, its own forms to fill out, and its own phone line.

A guest who needs room service, wants to check their bill, and needs a towel replacement would have to walk to three different entrances, fill out three different forms, and remember three different phone numbers. Some entrances require a key card. Some entrances only open during business hours. Some require you to speak a specific language. The guest is frustrated before they even get what they need.

Now hire a front desk receptionist — a single point of contact. The guest walks in the main entrance, tells the receptionist what they need, and the receptionist handles the rest. The receptionist knows which department to route to, what forms are needed, when each department is open, and how to translate between the guest's request and what the department expects. The receptionist also makes sure no unauthorized people get through, limits how many requests one guest can make at once, and keeps a log of every interaction.

That receptionist is your API Gateway.

This is the story of every mobile app or web frontend that talks to a microservices backend. Without a gateway, the client becomes a tangled mess of 20 different API calls, each with its own authentication scheme, rate limits, error formats, and URL structures. With a gateway, the client talks to one address, and the gateway handles the complexity.

## What Is an API Gateway

An API gateway is a server that sits between your clients and your backend services. It acts as a single entry point for all API requests. It handles cross-cutting concerns so individual services do not have to implement them from scratch.

Without a gateway, every service duplicates the same infrastructure logic. With a gateway, you centralize that logic in one place. Each service focuses on its business domain and leaves authentication, rate limiting, logging, and routing to the gateway.

## Core Vocabulary

**API Gateway**
A server that acts as the single entry point for client requests to a system of backend services. It routes requests, aggregates responses, and enforces cross-cutting policies. Every major distributed system uses one.

**Reverse Proxy**
A server that forwards client requests to backend servers on behalf of the client. Unlike a forward proxy (which represents the client), a reverse proxy represents the server. An API gateway is a specialized reverse proxy with added intelligence.

**Routing**
The mechanism by which the gateway examines an incoming request and decides which backend service should handle it. Routing decisions are based on the request path, headers, query parameters, or method. A routing table maps conditions to target services.

**Aggregation**
The gateway calls multiple backend services in parallel and combines their responses into a single response for the client. This reduces the number of round trips the client must make. For a dashboard that needs data from five services, aggregation is critical.

**Rate Limiting**
The gateway enforces a maximum number of requests a client can make within a time window. This protects backend services from overload. Common algorithms are token bucket, leaky bucket, fixed window, and sliding window.

**Authentication**
The gateway verifies the identity of the client before forwarding the request. It validates tokens, API keys, or certificates. This keeps authentication logic out of individual services.

**SSL Termination**
The gateway handles the TLS/SSL handshake and decrypts incoming HTTPS traffic before forwarding it to internal services over plain HTTP. This offloads expensive cryptographic operations from backend services and simplifies certificate management.

**Request/Response Transformation**
The gateway converts incoming requests from one format to another (for example, XML to JSON) or enriches them with additional data before forwarding. It can also transform responses from backend services to a format the client expects.

**Circuit Breaking**
The gateway monitors calls to backend services and stops forwarding requests when a service is failing. After a cooldown period, it probes the service to see if it recovered. This prevents cascading failures.

**Canary Deployment**
The gateway routes a small percentage of traffic to a new version of a service while directing the rest to the stable version. This lets you validate changes in production with minimal risk.

**API Versioning**
The gateway routes requests to different service versions based on the API version specified in the URL or header. This lets you evolve APIs without breaking existing clients.

**Throttling**
A softer form of rate limiting. Instead of rejecting excess requests, the gateway slows them down by queuing them and processing them at a controlled rate. The client sees higher latency but not errors.

## The Request Lifecycle: Ten Steps

When a client sends a request to an API gateway, ten distinct steps happen before the client sees a response. Understanding this lifecycle is the foundation of everything else.

### Step 1: TLS Termination

The client opens a TLS connection to the gateway. The gateway presents its certificate and negotiates a secure session. This is the SSL termination layer. The gateway decrypts the request so it can inspect and process it. Internal traffic between the gateway and backend services typically uses plain HTTP or mTLS, keeping the expensive TLS handshake at the edge.

### Step 2: Connection Management

The gateway accepts the TCP connection and assigns it to a worker thread or event loop. Modern gateways like Envoy use a non-blocking event loop to handle thousands of concurrent connections on a single thread. The connection enters a pool managed by the gateway's connection manager.

### Step 3: Request Parsing

The gateway parses the raw HTTP request into its components: method, path, headers, query parameters, and body. For HTTP/2 or gRPC, the parsing layer also handles multiplexing — multiple requests flowing over a single TCP connection.

### Step 4: Authentication

The gateway extracts credentials from the request: a JWT token in the Authorization header, an API key in a custom header, or a client certificate. It validates the credentials against an authentication service or a local cache. If validation fails, it returns a 401 immediately without touching backend services.

### Step 5: Authorization

After confirming who the client is, the gateway checks what they are allowed to do. This may involve looking up the client's role or permissions in a policy engine. A request to delete a resource may be allowed for admin roles but denied for read-only roles.

### Step 6: Rate Limiting

The gateway checks the client's request count against the configured rate limit. It reads the current count from a distributed counter (typically in Redis) and compares it to the limit. If the limit is exceeded, it returns a 429 Too Many Requests. The response includes Retry-After headers so the client knows when to try again.

### Step 7: Request Transformation

The gateway modifies the request before forwarding. It might add headers (like X-Request-Id for tracing), strip sensitive headers, convert the body format, or inject client IP information. This is also where the gateway might rewrite the URL path to match the internal service's expectations.

### Step 8: Routing

The gateway matches the request against its routing table. The routing table is a set of rules that map request properties to backend service addresses. A common implementation uses a prefix tree (trie) for efficient path matching. The gateway selects the target service, picks an instance from the service registry (via load balancing), and forwards the request.

### Step 9: Backend Call and Circuit Breaking

The gateway sends the transformed request to the chosen backend instance. If the backend returns an error or times out, the circuit breaker checks whether the failure rate has exceeded a threshold. If the circuit is open, the gateway returns a cached response or a fallback error immediately, without calling the backend.

### Step 10: Response Processing

The gateway receives the response from the backend, performs any required transformations (removing internal fields, converting formats, compressing the body), adds response headers (CORS, caching, rate limit status), and sends the response back to the client over the established TLS connection.

## Variants of API Gateway

There is no single API gateway architecture. Different deployment contexts demand different designs.

### BFF Pattern — Backend for Frontend

A dedicated gateway per client type. The mobile app gets its own gateway, the web SPA gets its own gateway, and the IoT devices get their own gateway. Each gateway is optimized for its client's specific needs.

The mobile gateway might aggregate more data (smaller payloads, fewer round trips) while the web gateway might serve richer responses. This prevents one client's requirements from bloating the gateway for all clients.

### Single Gateway

A single shared gateway for all clients. This is the simplest deployment model. One gateway instance (or a cluster of identical instances) handles every incoming request.

This works well when all clients have similar requirements — for example, when you only have one mobile app and no web SPA. The downside is that changing the gateway for one client affects all clients.

### Edge Gateway

Deployed at the network edge, close to the clients geographically. Edge gateways handle traffic from the public internet and sit behind a CDN or a global load balancer.

The edge gateway is the outermost layer of your infrastructure. It handles TLS termination, DDoS protection, IP allowlisting, and basic routing. It forwards requests to internal gateways or directly to services.

### Private Gateway

Deployed inside a private network for internal traffic. No internet-facing endpoints. Used for service-to-service communication within a VPC or data center. Private gateways handle internal routing, internal authentication, and service mesh integration.

A single organization often runs all four variants simultaneously: edge gateways for public traffic, private gateways for internal calls, BFF gateways for each client type, and a management gateway for admin traffic.

### Private Gateway in Detail

A private gateway handles service-to-service traffic within the data center or VPC. It does not terminate TLS from external clients. Instead, it may use mTLS for internal authentication between services.

Private gateways are commonly paired with service mesh sidecars. The private gateway handles cross-cutting concerns at the cluster entry point, while the sidecar proxies handle per-service concerns like retry budgets and locality-aware routing.

## Common Use Cases

API gateways appear in almost every large-scale distributed system. Understanding the common use cases helps you recognize when a gateway is the right tool.

### Mobile Backend

A mobile app needs to call 10 different endpoints to render a single screen. The gateway aggregates these calls into one request. The mobile app sends one request to `/screen/home`, and the gateway calls the user profile, recommendations, notifications, and feed services in parallel, merging the results.

Without the gateway, the mobile app makes 10 round trips over a cellular network. Each round trip adds 50-200 milliseconds of latency. The total load time is 1-2 seconds. With gateway aggregation, the load time is the slowest single service plus 5 milliseconds of gateway overhead — under 200 milliseconds.

### Third-Party API Platform

A company exposes APIs to external developers. The gateway handles authentication (API keys), rate limiting (per-developer quotas), usage tracking (billing per API call), and API versioning.

Each developer gets an API key. The gateway validates the key on every request, checks the developer's rate limit, increments their usage counter, and routes to the correct service version. The gateway also enforces request validation — malformed JSON payloads are rejected before they reach the backend.

### Internal API Platform

A large organization has 200 services and 20 product teams. Each team owns several services. The gateway provides a unified entry point for all internal APIs. Each team registers their services' routes in the gateway. The gateway handles cross-cutting concerns so each team does not need to implement auth, rate limiting, or logging in their services.

The operations team manages the gateway's global configuration (TLS, global rate limits, monitoring). Each product team manages their own routes and service-level plugins.

## How the Gateway Differs from a Load Balancer

A load balancer distributes traffic across multiple instances of the same service. An API gateway does much more. The distinction is important for system design interviews.

| Aspect | Load Balancer | API Gateway |
|---|---|---|
| Primary function | Distribute traffic | Route, transform, secure |
| Protocol awareness | L4 (TCP/UDP) or basic L7 | Full L7 (HTTP, gRPC, WebSocket) |
| Authentication | None | JWT, OAuth2, API keys, mTLS |
| Rate limiting | Basic connection limiting | Per-client, per-route token buckets |
| Request transformation | None | Header/body modification, format conversion |
| Aggregation | None | Parallel service calls |
| Caching | Static content only | Response caching per endpoint |
| Circuit breaking | Basic health checks | Per-service circuit breakers with half-open probes |
| Configuration | Static IP lists | Dynamic routing tables with hot reload |

## Why Services Should Not Implement Their Own Gateway Logic

Each cross-cutting concern that a service implements itself is code that must be written, tested, deployed, monitored, and maintained in every service. Authentication alone, if implemented in 20 services, means 20 copies of the same JWT validation logic, 20 places where a vulnerability could be introduced, and 20 codebases to update when the authentication scheme changes.

Beyond the maintenance burden, services that implement their own cross-cutting logic dilute their core responsibility. A payment service should be focused on processing payments correctly — not on parsing tokens, tracking rate limits, or logging request metrics. The gateway frees services to be single-purpose.

## Mental Model Check

Test your understanding with these five questions. Think through each one before reading the answer.

### Question 1

A mobile app needs to display a user dashboard that aggregates data from four microservices: user profile, recent orders, notifications, and recommendations. Without a gateway, how many round trips does the client make? With a gateway, what changes?

**Answer:** Without a gateway, the client makes four sequential or parallel API calls. The client must know the location of all four services, handle four authentication checks, and merge four responses. With a gateway, the client makes one call to the gateway. The gateway calls the four services in parallel, waits for all responses, merges them, and returns a single payload. The client makes one round trip instead of four.

### Question 2

You are rate limiting API calls. Should the rate limit be enforced at the gateway, at each service, or both? What are the trade-offs?

**Answer:** Enforce rate limiting at the gateway as a first line of defense. Gateway-level rate limiting protects all services at once with a single configuration. Service-level rate limiting adds defense-in-depth but requires each service to implement the same logic. The standard approach is gateway-level limiting for global protection, with per-service limits for specific backend capacity constraints.

### Question 3

A client sends a request with an expired authentication token. Should the gateway or the target service detect this? Why?

**Answer:** The gateway should detect it. Token validation is a stateless operation that the gateway can perform without calling any backend service. If the token is expired, the gateway returns a 401 response immediately, saving the overhead of routing the request, connecting to a backend, and processing it. This protects backend services from processing requests that will inevitably fail.

### Question 4

You have two versions of the same API — v1 and v2. Clients are slowly migrating from v1 to v2. How does the gateway handle this without breaking existing clients?

**Answer:** The gateway routes based on a version indicator in the request URL (for example, /v1/orders vs /v2/orders) or in a header. Each version maps to a different backend service or deployment. As clients migrate, you can gradually reduce the v1 backend instances and increase v2 instances. The gateway never breaks clients — it simply routes according to the declared version.

### Question 5

Your gateway makes a call to a backend service, and that service is taking too long to respond. What options does the gateway have?

**Answer:** The gateway has several options. It can set a timeout and return a 504 Gateway Timeout if the service does not respond in time. It can use a circuit breaker to stop sending requests to the failing service and serve cached responses. It can return a partial response with the data it already has and a note that some data is unavailable. It can also queue the request for retry if the operation is idempotent.

### Question 6

You are designing a gateway for a system with five microservices. Two services need JWT authentication, one needs API key authentication, and two are public (no auth). How does the gateway configuration handle this?

**Answer:** Route-level plugin configuration. The gateway's global configuration sets the default (no auth). The two services that need JWT auth have a JWT plugin attached to their specific routes. The API key service has an API key plugin on its route. Each route specifies only the plugins it needs. No route inherits plugins it does not need. This is the standard approach in Kong, Envoy, and Traefik — plugins are applied per route, not globally.

### Question 7

A client sends a request that triggers three transformation steps: add headers, convert body from XML to JSON, and compress the response. In what order should these transformations run, and why?

**Answer:** The request pipeline runs transformations in this order: add headers first (fast, no body parsing), then convert body format (needs the original body), then compress (needs the transformed body). The response pipeline runs in reverse: decompress (if needed), convert format, then add headers. The order matters because each transformation has a dependency on the previous step's output. Running compression before format conversion would compress the original body format, forcing the converter to decompress first.

### Question 8

A gateway routes requests to three backend services. One service receives a traffic spike from a misconfigured client. How does the gateway prevent this from affecting the other two services?

**Answer:** Connection pooling with bulkheading. Each service gets its own dedicated connection pool. If the misconfigured client exhausts the connection pool for service A, services B and C are unaffected because they have their own pools. The circuit breaker for service A opens, and subsequent requests to service A are failed fast (or served from cache). The gateway also rate-limits the misconfigured client at the gateway level, preventing it from consuming more resources on any service.

### Question 9

You need to add a new authentication method (OAuth2) to an existing gateway that currently only supports API keys. How do you add it without breaking existing clients?

**Answer:** Add the OAuth2 authentication as a new plugin. Attach it to new routes only. Existing routes keep the API key plugin. Clients that use the old routes continue using API keys. New clients use OAuth2 on new routes. After the migration period, you can add the OAuth2 plugin to old routes as a secondary authentication method (either API key or OAuth2 is accepted). This allows a gradual migration without breaking existing clients.

### Question 10

Your gateway is deployed in three regions. A client in Europe makes a request that is routed to the US region because the European gateway is overloaded. What happens to the rate limit?

**Answer:** The rate limit is checked in the US region's Redis cluster. The client's rate limit counter may not have been replicated from the European Redis cluster to the US cluster yet. The client's request is allowed in the US region with a fresh rate counter. This means the client may temporarily exceed their global rate limit by a small margin. This is an acceptable trade-off for multi-region gateways. The alternative — requiring synchronous cross-region rate limit checks — would add 50-200 milliseconds of latency to every request.

## Gateway Metrics and Monitoring

A production gateway emits metrics that tell you the health of the entire system. Understanding these metrics is essential for operating a gateway.

### Key Metrics

**Request rate:** Number of requests per second received by the gateway. Tracked per route, per service, and per status code class (2xx, 4xx, 5xx). A sudden drop in request rate may indicate a client connectivity issue. A sudden increase may indicate a traffic spike or a DDoS attack.

**Latency:** Time from request receipt to response sent. Tracked as p50, p95, and p99. A healthy gateway has p99 under 10 milliseconds for self-hosted deployments. A rising p99 at the gateway while backend latencies are stable indicates the gateway itself is the bottleneck.

**Error rate:** Percentage of requests that return 5xx errors. Tracked per route and per backend service. A rising error rate on a specific route indicates that route's backend service is failing. A rising error rate across all routes indicates a gateway-level issue.

**Active connections:** Number of currently open TCP connections. Each active connection consumes memory for its request/response buffers. A rising connection count without an increase in request rate may indicate slow clients (leitner syndrome) or a DDoS attack.

**Upstream health:** Percentage of healthy backend instances per service. Tracked via active and passive health checks. A dropping upstream health score triggers circuit breaker events.

**Configuration version:** The gateway's current configuration version number. All gateway instances in the cluster should report the same configuration version. Mismatched versions indicate a configuration propagation failure.

## Summary

An API gateway is the single entry point for client requests in a distributed system. It handles TLS termination, authentication, rate limiting, routing, aggregation, request transformation, circuit breaking, and response processing. Without a gateway, every client must manage connections to every service, and every service must duplicate cross-cutting infrastructure logic. With a gateway, clients talk to one address, and services focus on their business domain.

There are four main gateway variants — BFF, single gateway, edge gateway, and private gateway — each suited to different deployment contexts. The choice depends on your client diversity, security requirements, and operational complexity tolerance.

The request lifecycle runs through ten steps from TLS termination to response processing. Each step is a place where the gateway adds value by centralizing logic that would otherwise need to be duplicated across every service in your system.
