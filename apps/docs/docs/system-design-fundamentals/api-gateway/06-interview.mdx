API gateway questions in FAANG interviews fall into three categories: design, troubleshooting, and comparison. Each category tests a different skill.

### Design Questions

**Design an API gateway for a video streaming platform like Netflix.**
This is the most common API gateway interview question. The interviewer wants to see that you understand the gateway's role in a microservices ecosystem, how it handles auth, rate limiting, routing, and aggregation at scale.

**Design a rate limiter for an API gateway.**
A focused systems design question about the rate limiting subsystem. You need to discuss token bucket versus sliding window, in-memory versus distributed (Redis), and how to handle consistency across gateway nodes.

**Design a routing layer for 10,000 microservices.**
This tests your understanding of routing data structures (trie, prefix tree) and performance requirements. The interviewer probes how you handle route conflicts, path parameters, wildcards, and routing table updates without downtime.

**Design a BFF (Backend for Frontend) architecture.**
The interviewer presents a scenario with web, mobile, and IoT clients. They want to see you identify the need for separate gateways per client type and discuss the trade-offs between shared and dedicated gateways.

### Troubleshooting Questions

**Users are getting 429 errors but traffic is normal. How do you debug?**
The interviewer tests your understanding of rate limiter configuration, Redis connectivity, and token bucket state. Start by checking the rate limiter configuration, then verify Redis health, then inspect the client's request frequency.

**After a new deployment, 5 percent of requests return 404. What happened?**
This points to a configuration drift problem. Some gateway instances have the new routes, others do not. Check the configuration version on each gateway node. Verify that the configuration propagation mechanism worked correctly.

**The gateway latency jumped from 5 ms to 200 ms. Where do you look?**
Start with the backend services — a slow service makes the gateway appear slow. Check the p50, p95, and p99 latencies per service. Then check the gateway's own metrics (connection pool depth, circuit breaker state, Redis latency).

**The gateway is consuming 100 percent CPU but request rate is normal. What is happening?**
Possible causes: a misconfigured plugin entering an infinite loop, a TLS handshake storm (too many connections being opened and closed), or a regular expression denial of service (ReDoS) from a malicious payload. Check CPU profiles, open connection count, and plugin execution times.

### Comparison Questions

**Compare Kong, Envoy, and AWS API Gateway.**
The interviewer wants to see that you understand the trade-offs between open-source, managed, and service-mesh-native gateways. Emphasize the difference in latency, customization, and operational overhead.

**When would you choose a service mesh (Istio) over an API gateway?**
API gateways handle north-south traffic (external to internal). Service meshes handle east-west traffic (internal service to internal service). The gateway is for external APIs. The service mesh is for internal communication patterns.

**What is the difference between a reverse proxy and an API gateway?**
A reverse proxy forwards raw requests. An API gateway inspects, transforms, and enriches requests. A reverse proxy cares about which server to connect to. An API gateway cares about who is making the request, whether they are allowed, and how the response should be shaped.

## Testing Signals Interviewers Look For

Interviewers evaluate your API gateway knowledge on four levels. Each level has specific signals they look for in your answers.

### Level 3 (L3) — The Candidate Can Define an API Gateway

Signals:
- Can explain what an API gateway does at a high level
- Knows the basic components: routing, auth, rate limiting
- Can draw a simple diagram with clients, gateway, and services
- Understands why you would use one (reduces client complexity)
- Has implemented or configured a simple gateway (NGINX reverse proxy, Express.js proxy middleware)

What is NOT expected: deep knowledge of routing tries, circuit breaker patterns, or multi-region deployment. At L3, the interviewer is checking that you have working knowledge.

### Level 4 (L4) — The Candidate Can Design a Gateway System

Signals:
- Can design a gateway for a given scale (10,000 RPS, multi-region, 50 services)
- Discusses trade-offs between different gateway architectures
- Understands request lifecycle in detail (authentication, rate limiting, routing, aggregation)
- Considers failure modes (gateway overload, retry storms, configuration drift)
- Mentions observability (metrics, logging, tracing)
- Discusses security (TLS, token validation, IP allowlisting)

### Level 5 (L5) — The Candidate Can Architect Production Gateways

Signals:
- Designs gateway with concrete capacity planning (nodes needed, memory, network bandwidth)
- Discusses BFF pattern and when to use multiple gateways
- Understands distributed rate limiting (Redis Lua scripts, token bucket vs sliding window)
- Designs for fault isolation (bulkheading, circuit breakers, load shedding)
- Compares real-world gateways (Zuul, Kong, Envoy, AWS API Gateway) with specific, accurate details
- Identifies anti-patterns without being prompted

### Level 6 (L6) — The Candidate Can Drive Gateway Strategy

Signals:
- Discusses organizational impact of gateway decisions (team ownership, configuration governance)
- Designs gateway migration strategies (monolith to microservices with gateway as enabler)
- Considers cost optimization (self-hosted vs managed, cache hit ratios, data transfer costs)
- Discusses gateway evolution over time (when to switch from simple proxy to full gateway)
- Articulates the relationship between API gateway, service mesh, and API management platform

## The FAANG Framework for Gateway Questions

When faced with an API gateway design question, use this five-step framework. It mirrors the structure used in Amazon's bar raiser interviews and Google's systems design interviews.

### Step 1: Clarify Requirements

Before drawing anything, ask questions to define the scope.

**Functional requirements:**
- How many services are behind the gateway?
- What types of clients connect? (mobile, web, IoT, third-party)
- What authentication methods are needed? (JWT, API key, OAuth2)
- Is request aggregation needed?
- Are there WebSocket or gRPC endpoints?

**Non-functional requirements:**
- What is the expected peak request rate?
- What is the acceptable latency budget?
- What are the availability requirements? (99.9 percent, 99.99 percent)
- Are there geographic distribution requirements?
- What is the budget for infrastructure? (self-hosted vs managed)

**Example clarification:**
"Before I design the gateway, let me clarify the requirements. You mentioned a video streaming platform. How many services are behind the gateway? What is the peak request rate during a major event like a new season release? What clients need to connect — just mobile and web, or also smart TVs and gaming consoles? And what is the latency requirement for the gateway itself?"

### Step 2: Define the API Surface

Define what routes the gateway exposes and how they map to services.

```
Route Mapping:
  GET  /v2/users/{id}         → user-service
  GET  /v2/users/{id}/history → playback-history-service
  POST /v2/search             → search-service
  GET  /v2/catalog/{id}       → catalog-service
  GET  /v2/recommendations    → recommendation-service
  POST /v2/auth/login         → auth-service
```

Specify which routes require authentication, which require rate limiting, and which are public.

### Step 3: Design the Gateway Architecture

Draw the gateway architecture with all components:

- Global load balancer (DNS-based, geo-routing)
- Gateway cluster (multi-AZ, auto-scaled)
- Configuration store (PostgreSQL, etcd, or file-based)
- Rate limiter state (Redis cluster)
- Service registry (Consul, Kubernetes DNS)
- Monitoring stack (Prometheus, Grafana)

Explain each component's role and how they interact during a single request.

### Step 4: Discuss Key Design Decisions

Explicitly state the trade-offs you are making and why.

**Decision: Use a single shared gateway vs BFF per client type.**
"Since the platform serves mobile, web, and smart TV clients, I recommend a BFF pattern per device type. Each gateway is optimized for its client's needs: the mobile gateway aggregates more data per call (fewer round trips), the web gateway serves richer responses, and the TV gateway streams large chunks of catalog data."

**Decision: Use Redis for rate limiter state with eventual consistency.**
"Rate limiter state is stored in Redis with Lua scripts for atomic token bucket operations. During a network partition, some requests may exceed the rate limit by a small margin. This is acceptable because stopping all traffic is worse than allowing a temporary overage."

**Decision: Use Envoy as the gateway data plane with a custom control plane.**
"Envoy provides the high-performance proxy layer (C++, thread-per-core model, 50K RPS per node). The control plane manages configuration through xDS protocols. This gives us Envoy's performance with our own configuration management layer."

### Step 5: Cover Failure Modes

Every FAANG interview expects you to discuss what breaks. Cover at least three failure modes and their mitigations.

**Gateway overload:** "We use auto-scaling triggered by CPU and connection count. If auto-scaling cannot keep up, the gateway enters load-shedding mode and returns 503 for new requests."

**Backend service failure:** "Each service has its own circuit breaker with per-service thresholds. When a circuit opens, the gateway returns cached responses or a degraded response with available data."

**Configuration drift:** "Configuration changes are rolled out via a canary process. One gateway instance receives the new configuration first. Error rates are monitored for 5 minutes before rolling out to all instances. A rollback mechanism reverts the configuration instantly."

## Fifteen Phrases That Signal Senior-Level Gateway Knowledge

Use these phrases in your interview to demonstrate depth. Each one shows you understand a specific aspect of gateway design that junior candidates miss.

1. "The gateway should be stateless on the request path so we can scale horizontally by adding instances behind the load balancer."

2. "Rate limiting uses a token bucket algorithm with Redis Lua scripts to ensure atomic read-and-decrement operations across gateway nodes."

3. "Configuration changes go through a canary rollout to one gateway node first, with automated rollback if error rates increase."

4. "The routing layer uses a radix tree for O(k) path matching instead of linear regex matching across hundreds of routes."

5. "Connection pooling reduces backend connect latency by 30-60 percent by reusing TCP connections across requests."

6. "The gateway's latency budget is under 5 milliseconds at p99. Each plugin must justify its latency cost per route."

7. "Bulkheading separates connection pools per service so one slow service does not consume all gateway resources."

8. "The circuit breaker uses a half-open state to probe recovery — one request succeeds and the circuit closes, one fails and the circuit reopens."

9. "During a network partition, the rate limiter degrades to local in-memory counters rather than rejecting all traffic."

10. "The gateway should support API versioning through URL prefixes (/v1/, /v2/) with separate upstream targets per version."

11. "Global rate limiting across regions requires cross-region Redis replication, which adds 20-50 ms per request. Per-region rate limiting is usually sufficient."

12. "The configuration store uses a publish-subscribe pattern so all gateway instances converge on the same configuration version within seconds."

13. "Gateway observability must include three pillars: Prometheus metrics for request rates and latencies, structured JSON logs for audit, and OpenTelemetry traces for end-to-end debugging."

14. "The BFF pattern prevents one client type's requirements from bloating the gateway response for all clients."

15. "Self-hosted gateways (Kong, Envoy) offer lower latency and full customization but add operational overhead. Managed gateways (AWS API Gateway) reduce operations but add latency and limit control."

## Eight Red Flags That Lose the Interview

Avoid these mistakes. Each one signals a gap in your understanding that costs you the interview at FAANG level.

1. **"The gateway calls a database to check rate limits."** No — rate limiter state lives in Redis for low-latency access. Databases are for configuration, not per-request state. Calling a database for every request would add 10-50 ms latency.

2. **"All requests go through one gateway instance."** This misses clustering, load balancing, and high availability. Always design for multiple instances across availability zones.

3. **"The gateway handles business logic."** No — the gateway handles cross-cutting concerns (auth, rate limiting, routing). Business logic belongs in services. A gateway that implements business logic is a gateway that requires service-team coordination for every change.

4. **"Rate limiting is done per gateway instance independently."** Without shared state, a client can exceed the rate limit by making requests to different gateway nodes simultaneously. Distributed rate limiting requires shared state in Redis.

5. **"The configuration is hardcoded in the gateway's code."** Configuration changes more frequently than code. Hardcoded config means redeploying the gateway for every route change. Configuration must be hot-reloadable.

6. **"We do not need observability for the gateway."** The gateway is on the critical path for every request. Without observability, you cannot debug any outage. This answer negates all other good signals.

7. **"TLS is handled by each backend service, not the gateway."** TLS termination at the gateway is standard practice. Backend services should not handle TLS — it adds complexity, duplicates certificate management, and wastes CPU on cryptographic operations in every service.

8. **"The gateway never fails because we use Kubernetes to keep it running."** Kubernetes keeps the process running but does not prevent overload, configuration errors, or downstream failures. Kubernetes is not a substitute for circuit breakers, load shedding, and graceful degradation.

## Five Drills to Master API Gateway Design

Practice these drills to build deep, recallable knowledge for your interview.

### Drill 1: Draw the Full Request Lifecycle

Without looking at notes, draw the complete request lifecycle from client DNS resolution to response delivery. Include all twelve steps. Label each step with the data structures and components involved. Repeat until you can draw it in under 5 minutes.

Checklist:
- DNS resolution and global load balancing
- TLS termination
- Connection acceptance and multiplexing
- Request parsing and validation
- Pre-routing middleware (IP filter, rate limit, auth, authz, log, transform)
- Route matching with trie walk
- Service discovery
- Load balancing algorithm selection
- Connection pool checkout
- Backend call with circuit breaker monitoring
- Response transformation and compression
- Response delivery and connection return

### Drill 2: Design a Rate Limiter from Scratch

Take 30 minutes to design a distributed rate limiter for a gateway with 10 nodes handling 50,000 RPS.

Your design must cover:
- Data structure (token bucket with parameters)
- Redis operations (Lua script for atomicity)
- Consistency model (eventual consistency with bounded error)
- Failure handling (local fallback when Redis is unreachable)
- Performance (how many Redis calls per second, what latency)
- Configuration (per-client, per-route, per-Tier limits)

### Drill 3: Compare Three Gateways

Create a comparison table for Kong, Envoy, and AWS API Gateway across ten dimensions:
1. Language and runtime
2. Latency added (p50 and p99)
3. Request throughput per node
4. Plugin/extension mechanism
5. Configuration mechanism
6. Service discovery integration
7. Circuit breaker implementation
8. Rate limiting implementation
9. Observability support
10. Operational complexity

Be able to explain which one you would choose for a given scenario and why.

### Drill 4: Troubleshoot a Production Incident

Take 20 minutes to describe your debugging approach for this scenario:
"The gateway's p99 latency suddenly jumped from 10 ms to 500 ms. Request rate is unchanged. Backend services report normal health. What is the most likely cause, and what is your step-by-step debugging plan?"

Sample answer steps:
1. Check gateway metrics: break down latency by service. Is one service slow, or all services?
2. If one service is slow: check circuit breaker state, connection pool depth for that service, Redis latency for rate limiter calls
3. If all services are slow: check DNS resolution time, check if a new plugin was deployed, check CPU/memory/GC on gateway nodes
4. Check if a configuration change was rolled out recently
5. Check if a client is sending unusually large payloads (body size exceeds typical)

### Drill 5: Build a Migration Plan

Describe how you would migrate a monolith to microservices using an API gateway as the migration enabler.

The plan should follow the strangler fig pattern:
1. Deploy a gateway in front of the monolith. The gateway routes all traffic to the monolith.
2. Extract one service from the monolith (for example, user management). Add a route in the gateway that routes /v2/users to the new service.
3. Migrate clients to /v2/users gradually. Old /v1/users still routes to the monolith.
4. Repeat for each service.
5. Once all routes point to microservices, the monolith is decomissioned.
6. The gateway was the enabler that allowed incremental migration without breaking clients.

Discuss how you handle state that spans the monolith and the new service during migration (dual-writes, shared database, or event-driven synchronization).

## Advanced Questions for Senior Roles

### Design a Gateway for a Real-Time Analytics Platform

A platform ingests 500,000 events per second from IoT devices. Each event must be authenticated, validated, enriched with device metadata, and routed to the appropriate processing pipeline. The gateway must add under 2 milliseconds of latency per event.

**Approach:**
- Use a UDP-based protocol instead of HTTP. HTTP's header parsing and connection management add unacceptable latency.
- Use a single-threaded, event-driven gateway with DPDK (kernel bypass) for network I/O.
- Authentication is a local hash lookup (preloaded device keys), not a network call.
- Enrichment is a local Redis lookup for device metadata.
- Routing is a simple hash-based partition key (device_id mod N partitions).
- No TLS on the ingest path. Events are sent over a private network.
- The gateway batches events into 1,000-event chunks before writing to Kafka.

### Design a Gateway for a Multi-Tenant SaaS Platform

A SaaS platform serves 500 tenants. Each tenant has its own API keys, rate limits, custom authentication schemes, and routing rules. A configuration change for one tenant must not affect other tenants.

**Approach:**
- Use a hierarchical configuration model: global defaults, tenant overrides, and route-specific overrides.
- Tenant configuration is stored per-tenant in the database, not in a shared config file.
- The gateway loads tenant configuration on demand (when the first request for the tenant arrives) and caches it in memory.
- Rate limits are per-tenant with a hard cap per gateway instance. If one tenant's traffic spikes, their rate limit kicks in without affecting other tenants.
- Plugin configurations are also per-tenant. Tenant A's custom auth plugin runs only for Tenant A's routes.
- Tenant isolation via separate connection pools. Tenant A's connections to their backend are in a different pool from Tenant B's.

## Ten Advanced Interview Phrases

Beyond the fifteen core phrases, these demonstrate deeper expertise:

2. "Rate limit counters use a sliding window log approach, not a fixed window. Fixed windows allow burst at the window boundary. Sliding windows provide uniform enforcement."

3. "The gateway supports circuit breaker half-open probes with a configurable success threshold. Three consecutive successful probes close the circuit, preventing flapping."

4. "Configuration validation includes dependency checking. If you remove a plugin that a route depends on, the validator catches the conflict before the configuration is applied."

5. "The gateway uses consistent hashing for cache affinity. It routes requests with the same cache key to the same gateway node, maximizing cache hit rates."

6. "For multi-region deployments, the gateway uses active health checks with a configurable interval. A node is removed from rotation after three consecutive failed health checks at 5-second intervals."

7. "Request transformation uses a template engine. The admin defines the output format as a template with variables extracted from the input request. No custom code required."

8. "The gateway's connection pool uses a lazy eviction strategy. Connections are only closed when the pool exceeds the maximum size. This reduces connection churn."

9. "API versioning uses a content negotiation header, not URL paths. The Accept header specifies the version. This keeps URLs clean and consistent across versions."

10. "The gateway's observability includes RED metrics: Rate (requests per second), Errors (failed requests per second), and Duration (latency distribution). This covers the key health signals for every route."

## Ten Common Interview Questions with Model Answers

### Question 1: What happens when the gateway goes down?

Every request to your system fails. No client can reach any service. The gateway is a single point of failure in the request path. Mitigation: run multiple gateway instances across availability zones behind a load balancer. If one instance fails, the load balancer routes traffic to the remaining instances. For regional failures, use multi-region deployment with DNS failover.

### Question 2: How does the gateway authenticate requests?

The gateway extracts credentials from the request (JWT from the Authorization header, API key from a custom header, or client certificate during TLS handshake). It validates the credentials against a local cache (fast path) or an authentication service (slow path). If validation fails, it returns 401 immediately. If valid, it adds the authenticated user's identity to the request headers before forwarding to the backend service.

### Question 3: How does the gateway handle partial failures?

The gateway uses circuit breakers per service. If a service returns errors or times out, the circuit opens and subsequent requests return cached responses or fallback errors without calling the failing service. Connection pooling with bulkheading ensures one slow service does not consume all gateway resources. Timeouts per service prevent a single slow call from holding the entire request.

### Question 4: Can the gateway add latency to requests?

Yes, the gateway adds 1-6 milliseconds for a well-tuned self-hosted gateway and 30-100 milliseconds for a managed cloud gateway. The overhead comes from TLS termination, request parsing, middleware execution (auth, rate limiting), and route matching. For most applications, this overhead is acceptable compared to the benefits of centralized cross-cutting concerns.

### Question 5: How do you scale the gateway?

The gateway is stateless on the request path, so horizontal scaling is straightforward. Add more instances behind the load balancer. Use auto-scaling based on CPU utilization, request rate, or connection count. Each instance can handle 10,000-50,000 requests per second depending on the gateway software and hardware.

### Question 6: What is the difference between rate limiting and throttling?

Rate limiting rejects requests that exceed a threshold. The client receives a 429 and must retry later. Throttling slows down requests by queuing them and processing them at a controlled rate. The client sees higher latency but no errors. Rate limiting is simpler and more common. Throttling is better for batch operations where retries are expensive.

### Question 7: How do you handle WebSocket connections through a gateway?

The gateway must support WebSocket upgrade (the initial HTTP upgrade request) and then proxy the persistent WebSocket connection to the backend. WebSocket connections are long-lived and consume gateway resources even when idle. The gateway must have separate connection limits for WebSocket connections and set idle timeouts to close stale connections. Some gateways (AWS API Gateway, Kong, Envoy) natively support WebSocket proxying.

### Question 8: How does the gateway handle gRPC calls?

The gateway must understand HTTP/2 (gRPC is built on HTTP/2) and support the gRPC content-type (application/grpc). For gRPC, the gateway can act as a plain proxy (forwarding the binary gRPC payload as-is) or as a gRPC-aware proxy (inspecting and routing based on the gRPC service name and method). Envoy has native gRPC support with load balancing, retries, and circuit breaking for gRPC calls. Kong supports gRPC via a dedicated plugin.

### Question 9: Can the gateway cache responses?

Yes. The gateway can cache GET responses from backend services. Cache storage is typically in Redis or Memcached. The gateway checks the cache before calling the backend. Cache TTL is configurable per route. Cache invalidation happens via TTL expiry, explicit invalidation requests from the backend, or cache tags.

### Question 10: How do you test the gateway under failure conditions?

Use chaos engineering: inject failures in a staging environment and verify the gateway handles them correctly. Kill a gateway instance and verify the load balancer routes traffic to remaining instances. Make a backend service slow and verify the circuit breaker opens. Disconnect Redis and verify the rate limiter degrades gracefully. Simulate a configuration update and verify convergence across all gateway instances.

## Mock Interview: Gateway Design for a Video Platform

Walk through a complete mock interview for the question "Design an API gateway for a video streaming platform."

### Interviewer: "Design an API gateway for a video streaming platform like Netflix."
**Candidate:** "Before I start, let me clarify the scope. How many services are behind the gateway? How many requests per second at peak? What client types need to connect?"

### Interviewer: "Assume 500 services, 100,000 requests per second at peak, and clients include mobile apps, web browsers, smart TVs, and gaming consoles. Latency budget is 10 milliseconds for the gateway."
**Candidate:** "Good. With 500 services and 4 client types, I recommend a BFF pattern. Each client type gets its own gateway optimized for its specific needs. The mobile gateway aggregates data aggressively. The web gateway serves richer responses. The TV gateway streams large chunks of catalog data. The gaming console gateway handles lightweight metadata."

### Interviewer: "Walk me through the architecture."
**Candidate:** "The architecture has three layers:"

**Layer 1 — Global Load Balancer:**
"DNS-based routing directs clients to the nearest region. We deploy in us-east-1, eu-west-1, and ap-southeast-1. The global load balancer detects regional failures and reroutes traffic."

**Layer 2 — BFF Gateways:**
"Each BFF is a cluster of 20 Envoy nodes behind a regional load balancer. Envoy gives us 50,000 RPS per node and under 3 milliseconds latency. Each BFF has its own configuration: routes, plugins, rate limits, and upstream services."

**Layer 3 — Backend Services:**
"500 microservices organized by domain. The gateway discovers services via Kubernetes DNS. Each service has its own connection pool with bulkheading."

### Interviewer: "How does the gateway handle authentication?"
**Candidate:** "Three authentication methods:"
1. JWT tokens for mobile and web clients (validated by the gateway using a cached JWKS key set)
2. API keys for partner integrations (validated against Redis, with PostgreSQL as the backing store)
3. mTLS for internal service-to-service calls (certificates issued by our internal CA)

"The gateway caches authentication results. A JWT token validated once is not re-validated for 5 minutes if the same token is used again. This reduces auth overhead by 90 percent for repeat requests."

### Interviewer: "How do you handle rate limiting at 100,000 RPS?"
**Candidate:** "Token bucket algorithm per client, per route. The bucket state is stored in a Redis cluster with 3 primary nodes and 3 replicas per region. Each gateway instance runs a Lua script on Redis to atomically check and consume tokens."

"Rate limit configuration is per-client-tier: premium customers get 10,000 requests per hour, standard customers get 1,000 per hour. The gateway reads the client's tier from their JWT claims."

### Interviewer: "What about failure modes?"
**Candidate:** "Three key failure modes:"

"Gateway overload: Auto-scaling adds nodes when CPU exceeds 70 percent. If auto-scaling cannot keep up, the gateway enters load-shedding mode. It returns 503 with Retry-After headers for the lowest-priority traffic."

"Backend service failure: Each service has a circuit breaker. If the error rate exceeds 50 percent over a 30-second window, the circuit opens. The gateway returns cached responses or a degraded response. The circuit probes recovery every 30 seconds."

"Configuration drift: Configuration changes are rolled out via a canary process. One gateway node receives the new config first. We monitor error rates for 5 minutes. If errors increase, the config is rolled back automatically."

### Interviewer: "How do you monitor the gateway?"
**Candidate:** "Three pillars:"
1. Prometheus metrics: request rate, error rate, latency (p50/p95/p99), connection count, circuit breaker state, rate limit exceeded count
2. Structured JSON logs: every request is logged with request_id, method, path, status, latency, upstream_service, and user_id
3. Distributed tracing: OpenTelemetry traces every request from client through gateway to backend services. This is essential for debugging latency issues

"The key dashboard shows a single number: the gateway's health score. This is a weighted composite of error rate, latency, and upstream health. If the health score drops below 90 percent, an alert fires."

## Summary

API gateway interview questions test your ability to design, troubleshoot, and compare gateway architectures. FAANG interviewers evaluate candidates at four levels, from basic definition (L3) to strategic architecture (L6). Use the five-step framework — clarify requirements, define the API surface, design the architecture, discuss key decisions, and cover failure modes — to structure your answers. Practice the five drills to build deep recall, and avoid the eight red flags that cost candidates the interview.
