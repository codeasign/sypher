## The Core Tension

When you choose a notification system, you are explicitly trading **delivery completeness** for **delivery cost and user experience**. You can send every notification to every user through every channel and achieve 100 percent delivery. But you will bankrupt your company on SMS fees, get your email domain blacklisted for spam, and cause every user to uninstall your app from notification fatigue. Every design decision in a notification system is a negotiation with this tension.

The fundamental equation is: **Notification value minus notification cost must be positive.** The "cost" is not just money — it is user attention, device battery, network bandwidth, and engineering maintenance. A notification that is ignored, dismissed, or blocked is pure negative value.

## Decision Matrix

| Decision Point | Option A | Option B | Choose A When | Choose B When |
|---------------|----------|----------|--------------|--------------|
| Channel selection | Send all channels | Send preferred channel only | Notification is critical (security, payment) | Notification is informational (like, follow) |
| Batching strategy | Immediate send | Batch by time window | Latency is the priority (chat, alerts) | Cost is the priority (marketing, digests) |
| Retry policy | Aggressive (3 retries, 10s interval) | Conservative (1 retry, 5min interval) | Delivery is critical and provider is reliable | Provider is rate-limited or unreliable |
| Dedup scope | Global (cross-device, cross-channel) | Per-device only | User has multiple devices | User has one device |
| Persistence model | Durable (Kafka, DB) | Ephemeral (in-memory queue) | Notifications must survive restarts | Notifications are time-sensitive and short-lived |
| Template engine | Server-side rendering | Client-side rendering | Consistent branding is required | Customization per device is required |
| Preference default | Opt-in (user must enable) | Opt-out (enabled by default) | Privacy regulations (GDPR, CCPA) | User engagement is the priority |
| Provider failover | Failover to secondary provider | Failover to cheaper channel | Provider reliability is low | Provider cost is the concern |

## CAP Theorem Positioning

The notification system is an **AP** (Available and Partition-tolerant) system. When the network partitions, the system continues to accept and process events, even if it means sacrificing strict consistency.

**What this means in practice:** When the network between the router service and the Redis cluster fails, the router continues to process notifications. It skips the dedup check (accepting potential duplicates) and reads preferences from the database (accepting higher latency). The system stays available.

**What it does NOT mean:** The system does not guarantee that every notification is delivered exactly once. Duplicates can happen. Notifications can be lost (if the Kafka cluster loses a partition). The system trades strict consistency for high availability.

## Failure Modes

### Failure 1: Provider Outage

**What happens:** APNs experiences a regional outage. All push notification requests to the affected APNs endpoint return HTTP 503 or timeout after 30 seconds.

**Why it happens:** APNs is a distributed system run by Apple. Regional outages, DNS failures, and TLS certificate issues can all cause APNs to become unavailable.

**Blast radius:** All push notifications to iOS devices in the affected region fail. For a platform with 60 percent iOS users, this means 60 percent of push notifications are undelivered during the outage. The email and SMS channels are unaffected.

**Detection:** The push worker's health check shows 30 percent error rate. The circuit breaker opens. The monitoring dashboard shows a spike in `delivery.failed_provider_error` metric. PagerDuty alert fires.

**Recovery:**
1. Circuit breaker opens, stopping all requests to the failing APNs endpoint (automatic)
2. Notifications are queued in the `push-outbound` Kafka topic (no data loss)
3. After 60 seconds, the circuit breaker transitions to half-open and sends a probe request
4. If the probe succeeds, the circuit breaker closes and normal delivery resumes
5. If the probe fails, the circuit breaker remains open for another 60 seconds
6. Notifications queued during the outage are delivered when the circuit breaker closes

### Failure 2: Rate Limit Exhaustion

**What happens:** A marketing campaign sends 2 million promotional emails in 30 minutes. SES rate limit is 50,000 emails per hour. The rate limit is exceeded by 40x.

**Why it happens:** The marketing team launches a campaign without coordinating with the notification platform team. The rate limiter configuration did not account for campaign-scale traffic.

**Blast radius:** The SES rate limit is exceeded. SES rejects all email sending requests for 1 hour (the rate limit window). All email notifications — including critical ones like password resets — are blocked.

**Detection:** Email worker logs show HTTP 429 "Too Many Requests" from SES. The rate limit dashboard shows the `email.global` counter at 200 percent of capacity.

**Recovery:**
1. The email queue backs up (notifications are not lost, just delayed)
2. The circuit breaker for SES opens after 10 percent error rate
3. The team identifies the marketing campaign as the source
4. The marketing campaign is paused
5. The SES rate limit is increased (request to AWS support, typically 24 hours)
6. The email queue is drained over the next 2 hours at the new rate limit

### Failure 3: Token Explosion

**What happens:** A user opens the app on 10 different devices. Each device registers a push token. The user receives the same notification on all 10 devices.

**Why it happens:** The token registry has no per-user device limit. The app registers a new token on every device, and the system sends notifications to all registered tokens.

**Blast radius:** The user receives 10 notifications instead of 1. The user is annoyed and disables notifications. The system sends 10x the intended push notifications, increasing provider costs by 10x for that user.

**Detection:** User complaints about "spam from multiple devices." The analytics dashboard shows a spike in `delivery.per_user` for a small number of users.

**Recovery:**
1. Implement a per-user device limit (max 5 devices per user)
2. When a new device registers, evict the oldest device token
3. For notifications, group device tokens by user and send one notification per device group
4. Implement a "last used" timestamp on device tokens — evict tokens not used in 30 days

### Failure 4: Template Rendering Failure

**What happens:** A template contains a syntax error that was not caught in testing. When the template engine tries to render the notification, it throws an exception. The notification is not sent.

**Why it happens:** The template was updated directly in production (bypassing the CI/CD pipeline). The template engine uses a try-catch block, but the exception is not handled gracefully.

**Blast radius:** All notifications using this template fail to render. If the template is the "like" template, all like notifications are silently dropped.

**Detection:** The `template.render.error` metric spikes. The dead-letter queue receives notifications with "template_render_error" reason.

**Recovery:**
1. The template engine must have a fallback template for each notification type
2. When rendering fails, use the fallback template (plain text, no variables)
3. The failed notification is sent using the fallback, and the error is logged
4. The dead-letter queue entry includes the template version and error details for debugging

### Failure 5: Kafka Partition Imbalance

**What happens:** 80 percent of notification events are for 20 percent of users (power users). These users' events all hash to the same 10 Kafka partitions. The remaining 40 partitions are underutilized. The 10 hot partitions have 100,000 lag each.

**Why it happens:** The partition assignment is `hash(recipient_id) % 50`. Power users generate 10x more events than average users, and their IDs are not evenly distributed across the hash space.

**Blast radius:** Notifications for power users are delayed by 5 minutes. Notifications for average users are delivered instantly. The power users — the most engaged users — experience the worst notification performance.

**Detection:** Consumer lag dashboard shows 10 partitions with 100,000+ lag and 40 partitions with under 1,000 lag.

**Recovery:**
1. Increase the number of partitions (requires topic re-creation or partition increase)
2. Use a custom partitioner that considers both recipient ID and event type
3. Implement a salting strategy: `hash(recipient_id + "_" + salt) % 50` with random salt per event

### Anti-Pattern 1: The Synchronous Send

**The mistake:** Making the notification send blocking. The app server calls the notification provider API synchronously and waits for the response before returning the HTTP response to the client.

**Why it seemed reasonable:** Simplicity. Just call the API and wait. No queue, no workers, no async processing.

**What went wrong:** A 500ms provider API call adds 500ms to every user request. At 10,000 requests per second, the app server is blocked for 5,000 seconds per second (impossible — the server falls over). The provider API is called from the user-facing request path, so a provider outage causes the entire application to fail.

**The correct pattern:** Async all the way. The notification event is written to a queue and acknowledged immediately. The provider API call happens in a background worker. The user-facing request path never touches the provider API.

### Anti-Pattern 2: The Single Queue

**The mistake:** All notifications — push, email, SMS — go through a single queue with a single worker pool.

**Why it seemed reasonable:** One queue is simpler than three. The worker can handle all channel types.

**What went wrong:** An SMS delivery failure (Twilio rate limit) blocks the worker. The queue fills up with SMS notifications. Push notifications, which should be delivered in seconds, are delayed by minutes while the worker retries SMS deliveries.

**The correct pattern:** Separate queues per channel. Each channel has its own worker pool with channel-specific rate limits, retry policies, and failure handling.

### Anti-Pattern 3: The Infinite Retry

**The mistake:** Retrying failed notifications indefinitely with no max retry count.

**Why it seemed reasonable:** The notification must be delivered. Retrying forever means it will eventually succeed.

**What went wrong:** A user has an invalid push token. The system retries every 10 seconds for 30 days. That is 259,200 failed API calls to APNs. APNs throttles the connection. The user's invalid token costs $50 in APNs bandwidth.

**The correct pattern:** Max retries = 3. After 3 retries, move to the dead-letter queue. If the notification is critical, fall back to another channel (SMS or email) instead of retrying the same channel.

### Anti-Pattern 4: The Fire-and-Forget

**The mistake:** Sending the notification to the provider and not checking the response.

**Why it seemed reasonable:** The provider accepted the request. The notification was sent. What could go wrong?

**What went wrong:** APNs returns a 410 "Unregistered" status. The system ignores it and continues sending to the same invalid token. The user has uninstalled the app, but the system keeps paying for API calls to a dead endpoint. The token registry is never cleaned up.

**The correct pattern:** Always check the provider response. For invalid tokens, remove the token from the registry. For rate limits, back off. For server errors, retry. The feedback loop is mandatory.

### Anti-Pattern 5: The One-Size-Fits-All Template

**The mistake:** Using the same notification template for push, email, and SMS.

**Why it seemed reasonable:** The content is the same. Why write three templates?

**What went wrong:** The push notification is 200 characters (APNs limit is 128 characters for the alert field). The text is truncated. The SMS notification is 200 characters (SMS limit is 160 characters per segment). The message is split into two SMS messages, costing double. The email notification has no HTML formatting — it is plain text, which looks unprofessional.

**The correct pattern:** One template per channel. The push template is brief (under 128 characters). The SMS template is concise (under 160 characters). The email template is rich (HTML with formatting, images, and CTAs).

## When NOT to Use This

### You Do Not Need a Custom Notification System If:

**Your notification volume is under 10,000 per day:** Managed services can handle this. AWS SNS, Firebase Cloud Messaging, or Twilio Notify provide APIs that work well at this scale. The cost of building and maintaining a custom system exceeds the cost of the managed service.

**You only need one channel (push only):** If your app only sends push notifications, use FCM or APNs directly. The multi-channel complexity is unnecessary. FCM handles Android and iOS push through a single API, which may be sufficient.

**Your users are all in one region and one timezone:** The complexity of multi-region deployment, timezone-aware scheduling, and quiet hours is only justified when your user base spans multiple timezones. A single-region system with a simple cron-based schedule is sufficient.

**You can tolerate best-effort delivery:** If notifications are informational (not critical), you can use a simpler architecture. Best-effort delivery means no retry queue, no dead-letter queue, and no delivery receipts. The system sends the notification once and forgets about it.

### At What Scale Does This Become Necessary?

The custom notification system described in this chapter is justified when:
- You send more than 1 million notifications per day
- You support 3 or more delivery channels (push, email, SMS, in-app)
- Your user base spans multiple timezones
- You have SLAs for notification delivery (99.9 percent delivery rate)
- You need per-user notification preferences and opt-out management
- You have a dedicated platform team to maintain the system
