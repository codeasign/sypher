## The Problem Story

Your photo-sharing app launched three months ago. Day one, 100 signups. You send each one a welcome email — works fine. Day 30, 50,000 users. Someone sends a friend request and your email provider starts throttling you. Day 90, 2 million users. A user likes a photo, the app sends a push notification, the phone buzzes, they open the app for 30 seconds, then close it. Then another like. Then another. By noon the user has uninstalled the app because the buzzing never stopped.

You check your server logs: your notification queue has 8 million pending messages, your SMTP relay is rate-limiting, your push notification provider API key was revoked for exceeding limits, and your database shows connection timeouts because every notification send is a blocking write. At 2am you are paged. The on-call engineer before you set up a quick fix — a single-threaded loop that sends notifications one at a time. That worked for about four hours. Now the queue is 12 million deep and growing. Your users are leaving. Your investors are asking questions. And you realize: you never designed a notification system. You just bolted on API calls whenever someone wanted to tell someone something.

This is the story of every growing application that lacks a notification system. The average smartphone user receives 46 push notifications per day. A messaging app processes billions of events daily. Getting notifications wrong means losing users. Getting them right means building a system that handles volume, respects user preferences, maintains delivery guarantees, and does not collapse under its own weight.

## The Insight

A notification system is a **delivery router**. It takes a message from a producer, figures out who needs to receive it, through which channels, in what format, at what rate, and then delivers it reliably. Think of a postal sorting facility. You drop a letter in the mail. The facility looks up the recipient's address, decides whether to send it by air or ground, stamps it with the right postage, puts it on the right truck, and if the truck breaks down, it tries again. The sender only needs to know the letter arrived.

In an app, the "letter" is a notification event. The "address" is the user's delivery preferences and device tokens. The "trucks" are push notification services (APNs, FCM), email providers (SendGrid, SES), SMS gateways (Twilio), and in-app channels. The "sorting facility" is your notification system — it takes millions of events, classifies them, prioritizes them, formats them, and sends them through the right channels in the right order at the right speed.

The separation of concerns is what makes this pattern powerful. The team building the notification platform does not need to know about every product feature that generates notifications. They define an API contract: send a properly formatted event to this endpoint, and the platform handles everything else. Product teams focus on generating meaningful notification events. The platform team focuses on delivering those events reliably and efficiently.

## Core Vocabulary

**Notification event** — The raw data that something happened: "User A liked your photo." It includes the event type, the actor, the target, and metadata. Events are the input to the system. Without a notification event, nothing happens.

**Channel** — The delivery mechanism: push notification, email, SMS, in-app banner, webhook. Each channel has different capabilities, limits, and user expectations. Email is for detail. Push is for urgency. In-app is for context.

**Recipient** — The user or system that should receive the notification. A recipient has preferences (what channels they accept), tokens (device-specific delivery addresses), and state (online, idle, DND). Notifying the wrong person is worse than not notifying at all.

**Delivery provider** — The external service that actually sends the message: APNs (Apple Push Notification service), FCM (Firebase Cloud Messaging), Twilio (SMS), SendGrid (email). Each provider has its own API, rate limits, retry semantics, and cost structure.

**Template** — A reusable message format with variables. Instead of hardcoding "User A liked your photo" for every combination, you write `{{actor}} liked your {{content}}` and fill in the variables at send time. Templates ensure consistency, support localization, and reduce storage.

**Deduplication** — The logic that prevents sending the same notification twice. If a user likes the same photo twice in one second, you should not send two notifications. Deduplication happens at the event level (same event ID) and the content level (same message).

**Dispatch queue** — A buffer between the event producer and the delivery system. Events go into the queue. Workers pull from the queue at a controlled rate. The queue absorbs spikes, prevents backpressure from providers from crashing the system, and enables retry.

**Rate limiting** — Controlling how many notifications go out per unit time. Per-user (do not bombard a single user), per-channel (APNs has limits), per-provider (SES has sending quotas). Rate limiting prevents your app from getting banned by Apple or Google.

**Delivery receipt** — Confirmation that a notification was delivered. Push notifications have ephemeral receipts (the provider accepted it but cannot guarantee the phone saw it). Email has bounces, opens, and clicks. SMS has delivery reports. The receipt tells you whether to retry or switch channels.

**Preference store** — The database of per-user notification settings: which channels they allow, which event types they want to hear about, quiet hours, and per-device opt-in status. This is the source of truth checked before every send.

**Fan-out** — The process of sending one notification event to multiple recipients. When a user posts a photo, the system fans out notifications to all their followers. Fan-out can be synchronous (send to all now) or asynchronous (chunked over time).

**Quiet hours** — A user-configurable time window during which non-critical notifications are deferred. Notifications are queued and delivered when quiet hours end. Typical quiet hours are 10:00 PM to 7:00 AM in the user's timezone.

## How It Actually Works — Step by Step

### Step 1: Event Production

A service in your application generates an event. A user likes a photo, sends a message, or follows another user. The service constructs a notification event object with the event type, the actor ID, the target entity, and the recipient ID.

**Why this step exists:** Without explicit event construction, you cannot track what is happening. The event object is the contract between services and the notification system. If you skip structured event production, every service invents its own format and the notification system becomes a parsing nightmare.

### Step 2: Event Validation

The ingress endpoint validates the event. Is the recipient ID valid? Does the event type exist in the system? Are all required fields present? Malformed events are rejected immediately with a 400 error.

**Why this step exists:** The rest of the pipeline assumes well-formed events. A single bad event — missing recipient ID, malformed template variables — can crash a worker, stall the queue, or send a notification to the wrong user. Validation at the boundary catches these before they cause damage.

### Step 3: Event Enrichment

The system enriches the event with data needed for delivery. It looks up the recipient's user profile, device tokens, and preferences. It loads the event template. It resolves group memberships if the notification targets a group. This turns the raw event into a fully-specified delivery request.

**Why this step exists:** The producer does not have all the context needed for delivery. The producer knows "User A liked photo B." The notification system needs to know "User A's name is Alice, their avatar URL is..., the recipient's preferred channel is push, the recipient's device token is hex-encoded, and the recipient has liked 5 photos from User A this hour so maybe you should throttle." Enrichment bridges the gap.

### Step 4: Persistence to the Dispatch Queue

The enriched event is written to the dispatch queue — a persistent, durable message queue like Apache Kafka, Amazon SQS, or RabbitMQ. The ingress service receives confirmation that the message was written successfully, then returns a 202 Accepted to the producer.

**Why this step exists:** The dispatch queue is the system's shock absorber. Without it, a spike in event production — a viral post, a bot attack, a Super Bowl ad — would overwhelm downstream providers. The queue decouples production from delivery. Producers write at their speed. The delivery system reads at its speed.

### Step 5: Preference Check

The router pulls the event from the queue and checks the recipient's notification preferences. Does the recipient want to receive this type of notification? On which channel? Are they in quiet hours? Have they opted out entirely?

**Why this step exists:** Sending a notification the user does not want is the fastest way to get them to disable all notifications — or uninstall the app. The preference check is the user's voice in the system. Skipping it means the system decides what the user sees, which is a recipe for notification fatigue and churn.

### Step 6: Deduplication

The system checks whether this notification has already been sent. Duplicates can arise from: retrying a failed produce, a user double-tapping a like button, network retries from the client, or at-least-once queue semantics. The dedup check uses a combination of event ID, recipient ID, and a time window.

**Why this step exists:** Two identical notifications in thirty seconds annoy the user and waste system resources. The dedup logic prevents the most common source of notification spam — the system's own retry mechanisms.

### Step 7: Template Rendering

The template engine takes the event data and renders the notification payload for each channel. A push notification template might produce `{"alert": "Alice liked your photo", "badge": 3, "sound": "default"}`. An email template produces HTML. An SMS template produces 160 characters.

**Why this step exists:** Templates separate content from presentation. The event says "Alice liked a photo." The template system decides how to say that in 40 characters for push, 160 characters for SMS, and 100 words for email. Without templates, formatting logic is scattered across every service, making consistent branding and localization impossible.

### Step 8: Rate Limiting

The rate limiter checks whether sending this notification would exceed any limit:
- Per-user: no more than 5 push notifications per minute per user
- Per-channel: no more than 100 sends per second to APNs
- Per-event-type: no more than 3 like notifications per hour per user

If the notification would exceed a limit, it is either delayed (re-queued with a backoff timestamp) or dropped (if the limit is a hard cap).

**Why this step exists:** Rate limits protect both the user (from notification spam) and the system (from provider throttling). APNs will throttle your connection if you exceed its limits. Users will disable notifications if they are overwhelmed. Rate limiting is the system's self-preservation mechanism.

### Step 9: Channel Dispatch

The notification is placed on the appropriate channel queue. The push queue sends to APNs/FCM. The email queue sends to SendGrid/SES. The SMS queue sends to Twilio. Each channel queue has its own worker pool, configured for that channel's specific rate limits and delivery semantics.

**Why this step exists:** Different channels have different characteristics. Push notifications are ephemeral and fire-and-forget. Email is queued and retried. SMS is expensive and must be confirmed. Separating the queues lets each channel operate independently — a push delivery failure does not block email delivery.

### Step 10: Delivery Feedback

The channel worker receives a delivery receipt from the provider. For push, this is a confirmation from APNs that the message was accepted. For email, it is a bounce or delivery notification. The feedback processor updates the notification status in the database and, if delivery failed, determines whether to retry, fall back to another channel, or give up.

**Why this step exists:** Without feedback, you do not know if your notifications are reaching users. A user who uninstalled your app still has a valid push token — APNs will not tell you until you try to send and get an "Unregistered" response. The feedback loop cleans up invalid tokens, retries failed deliveries, and provides the data you need to monitor delivery health.

## Variants and Flavors

### Variant 1: Fan-Out Notification System

In this variant, a single event produces notifications for many recipients. A user posts a new photo — all their followers should get a notification.

**When to use it:** Social networks, content platforms, collaborative tools — any system where one user's action affects many others.

**When NOT to use it:** When the fan-out factor is too large (millions of followers per user). In that case, use a hybrid approach: fan-out to active followers only, and let inactive followers pull notifications on demand.

Notifications are classified into priority tiers. Critical notifications (account security, payment failures) skip the queue or preempt lower-priority notifications.

## Mental Model Check

**Q1:** A user receives 10 friend requests in 30 seconds. How many notifications should the system send?

**A1:** Probably 1 or 2 — not 10. The dedup logic should detect that all 10 events are of the same type for the same recipient and coalesce them into a single "You have 10 friend requests" notification. Some systems batch by time window — every 30 seconds, coalesce pending notifications of the same type.

**Q2:** What happens if the dispatch queue goes down at the moment a user sends a message?

**A2:** The ingress endpoint fails to write the event and returns a 503 Service Unavailable. The sending service should retry. The message itself is safe — the dispatch queue is only for notifications, not for the message content. The user's message was already saved to the database before the notification event was produced.

**Q3:** Why is there a separate queue for each channel instead of one big queue?

**A3:** Each channel has different rate limits, retry semantics, and delivery characteristics. If push and email share a queue, a push delivery failure (APNs is down) blocks email processing. Separate queues provide failure isolation and let each channel's workers be tuned independently.

**Q4:** A user opts out of push notifications entirely. Later, they change their device. What happens?

**A4:** The preference store says "push: disabled." When the user logs in on the new device, the app registers a new push token. But the preference check runs before the token lookup — so the new token is stored but never used for sending. The user must explicitly re-enable push notifications in settings. This is correct behavior — it prevents silent re-enrollment.

**Q5:** You have 1 million users. Each user sends 5 notifications per day. What is the peak throughput you need to handle?

**A5:** 5 million notifications per day = roughly 58 notifications/second averaged. But traffic is not uniform — peak hours might see 5x average. So you need to handle roughly 300 notifications/second. At the ingress layer, this is trivial. But at the channel provider layer, you must respect provider limits. APNs allows roughly 600 requests/second per connection — you need one APNs connection. Email might be rate-limited to 50/second, so you need a queue to buffer the excess.
