An operations team manages 200 microservices. Each service communicates over HTTP or gRPC. Every team independently implements retry logic, timeout configuration, circuit breakers, metrics export, and distributed tracing headers. The result is duplicated effort, inconsistent behavior, and subtle bugs when one team's retry policy clashes with another team's timeout policy. A service mesh solves this by extracting cross-cutting network concerns into a dedicated infrastructure layer.

---

## The City Traffic Grid Analogy

Imagine a city without a centralized traffic management system. Every driver must individually negotiate each intersection, watch for hazards, decide when to yield, and report their own delays. Some drivers are aggressive with retries (inching forward repeatedly). Others give up too early. There is no unified view of congestion. A service mesh is like installing traffic lights, road signs, and a central traffic control center that manages flow without each driver needing to change how they drive.

## Core Vocabulary

### Service Mesh

A dedicated infrastructure layer for handling service-to-service communication. It is implemented as a set of network proxies deployed alongside application code. The mesh provides traffic management, security, observability, and resiliency features without modifying application code.

### Sidecar Proxy

A network proxy that runs alongside each service instance in a separate container or process. The sidecar intercepts all inbound and outbound traffic. It applies policies for routing, retries, timeouts, authentication, and observability. Envoy is the most common sidecar proxy implementation.

### Data Plane

The collection of all sidecar proxies in the mesh. The data plane handles the actual network traffic between services. Each proxy makes local decisions based on configuration received from the control plane.

### Control Plane

The centralized management component that configures the data plane. The control plane distributes routing rules, service discovery information, TLS certificates, and policy configurations to all sidecar proxies. It does not process application traffic.

### Envoy

A high-performance L7 proxy written in C++ at Lyft. Envoy is the default sidecar proxy for Istio and is used by many other service mesh implementations. It supports HTTP/2, gRPC, TLS termination, circuit breaking, rate limiting, and rich observability.

### Istio

An open-source service mesh platform that uses Envoy as its sidecar proxy. Istio provides a complete control plane with components for traffic management (Pilot), security (Citadel), telemetry (Mixer, deprecated in newer versions), and configuration (Galley). Istio is the most widely adopted service mesh.

### Mutual TLS (mTLS)

Authentication where both sides of a connection present certificates. In a service mesh, mTLS ensures that every service-to-service communication is encrypted and authenticated. The control plane manages certificate issuance and rotation automatically.

### Traffic Routing

The ability to direct requests between services based on rules. You can implement canary deployments, blue-green deployments, A/B testing, and traffic splitting without modifying application code. Rules can match on HTTP headers, source service, destination service, or request properties.

### Observability

The collection of metrics, logs, and traces from all service-to-service communication. A service mesh provides uniform telemetry for every request, including latency, error rates, request volume, and distributed tracing spans. This gives you a complete picture of service health.

### Circuit Breaking

A pattern that prevents cascading failures by stopping requests to a failing service. When a configurable threshold of failures is reached, the circuit breaker trips and subsequent requests fail fast without reaching the downstream service. After a cooldown period, the circuit allows test requests to probe for recovery.

### Fault Injection

Deliberately introducing errors or latency into the mesh to test resiliency. You can inject HTTP errors (503, 500), delays (fixed or variable), or connection failures for specific services or percentages of traffic. This validates that your system handles failures gracefully.

### Telemetry

Automatically collected data about traffic flowing through the mesh. Telemetry includes request-level metrics (latency, status codes, request sizes), distributed traces (span IDs, parent spans, timing), and access logs (source, destination, protocol, response code). The mesh generates this data without any application instrumentation.

## Step by Step: A Request Through the Mesh

When Service A calls Service B in a mesh, the following steps occur.

### Step 1: Application Sends Request

Service A opens a connection to `http://service-b:8080/api/v1/resource`. The application code is unaware of the mesh. It uses standard DNS resolution or a virtual IP.

### Step 2: Sidecar Intercepts Outbound Traffic

The sidecar proxy in Service A's pod intercepts the outbound connection using iptables rules or a transparent proxy (TPROXY). The destination IP and port are redirected to the sidecar's listener port.

### Step 3: Route Lookup

The sidecar consults its local routing table, which was pushed by the control plane via the Route Discovery Service (RDS). It matches the request to a virtual host and route based on the `Host` header, path prefix, or other criteria.

### Step 4: Cluster Selection

The route points to a cluster (a logical group of endpoints). The sidecar uses the Cluster Discovery Service (CDS) configuration to find the cluster definition, which includes the load balancing algorithm (round robin, least request, random, consistent hash).

### Step 5: Endpoint Discovery

The sidecar resolves the cluster to a set of healthy endpoints via the Endpoint Discovery Service (EDS). It filters out unhealthy instances based on active health checks and outlier detection (circuit breaker state).

### Step 6: Load Balancing

The sidecar selects a specific endpoint from the healthy pool. For round robin, it picks the next endpoint in sequence. For least request, it picks the endpoint with the fewest active requests.

### Step 7: TLS Handshake

If mTLS is enabled, the sidecar initiates a TLS handshake with the destination sidecar. It presents its own certificate (issued by the control plane's certificate authority) and verifies the destination's certificate. The certificate includes the service identity (SPIFFE ID).

### Step 8: HTTP/2 or gRPC Multiplexing

The sidecar opens or reuses an HTTP/2 connection to the destination sidecar. Multiple requests from Service A can be multiplexed over a single connection, reducing connection overhead.

### Step 9: Destination Sidecar Receives

The destination sidecar receives the request. It performs inbound policy checks: authentication (mTLS verified), authorization (RBAC rules), and rate limiting. It then forwards the request to the local Service B process.

### Step 10: Response Processing

Service B responds. The response flows back through the destination sidecar, then the source sidecar. The source sidecar records telemetry (latency, response code, request size) and emits distributed tracing spans. If the response is an error, the sidecar may initiate a retry (if configured) or report the failure to the circuit breaker.

## Variants of Service Mesh

### Istio

Istio is the most feature-rich service mesh. It uses Envoy as the sidecar proxy and provides a complete control plane. Istio is backed by Google, IBM, and Lyft. It supports the broadest set of features: traffic management, security, observability, multi-cluster, and extensibility via WebAssembly.

## Key Capabilities

### Traffic Management

- Request routing based on headers, path, service, or weight
- Traffic splitting for canary releases (5% traffic to v2, 95% to v1)
- Fault injection for testing (inject 500ms delay on 10% of requests)
- Timeouts and retries with configurable budgets
- Circuit breaking with configurable thresholds

### Security

- Automatic mTLS between all services
- Certificate rotation every 24 hours by default
- Service identity based on SPIFFE (Secure Production Identity Framework for Everyone)
- Authorization policies at L7 (HTTP methods, paths, headers)
- Audit logging of all authentication decisions

### Observability

- Request-level metrics: latency (p50, p90, p99), error rate, request volume
- Distributed tracing with OpenTelemetry integration
- Access logs with full request metadata
- Service graphs showing topology and traffic flow
- Dashboards in Grafana, Jaeger, Kiali

### Resiliency

- Retry with budget (max 3 retries, 0.1% of requests budget)
- Timeout per request (configurable per route)
- Circuit breaker with configurable thresholds
- Connection pooling with limits
- Outlier detection and automatic ejection

## Mental Model Check

Answer these questions to verify your understanding.

### Question 1

You deploy a new version of Service A with a bug that causes it to return 500 errors for 50% of requests. The old version is still running. How can you route 90% of traffic to the old version and 10% to the new version without changing application code?

**Answer**: Configure a traffic splitting rule in the service mesh. In Istio, you create a `VirtualService` with two `HTTPRoute` destinations, each with a weight. The sidecar proxy routes traffic based on these weights without any application modification.

### Question 2

What is the difference between a service mesh and an API gateway?

**Answer**: An API gateway handles north-south traffic (external clients to internal services). A service mesh handles east-west traffic (service-to-service within the data center). An API gateway operates at the edge of the system. A service mesh operates inside the system. They complement each other: the API gateway authenticates external requests, and the service mesh manages internal communication.

### Question 3

Your team deploys a latency-sensitive service. Adding a sidecar proxy adds 1-2ms of latency per hop. How does this affect a request that traverses 5 services (10 hops including response)?

**Answer**: The added latency is 10-20ms for the round trip (5 outbound hops + 5 response hops). For most applications this is acceptable. For high-frequency trading or real-time systems, you may need to evaluate whether the benefits outweigh the overhead. Some meshes (Linkerd with Rust proxy) reduce this overhead.

### Question 4

You need to encrypt all service-to-service communication. How does a service mesh help compared to implementing TLS in each service?

**Answer**: With a service mesh, you enable mTLS globally with a single configuration change. The control plane automatically issues and rotates certificates. Each sidecar handles TLS termination and origination. Without a mesh, every team must implement TLS in their service, manage certificates, handle rotation, and ensure consistent configuration.

### Question 5

A downstream service is failing intermittently, causing cascading failures. What mesh features can help?

**Answer**: Circuit breaking stops requests to the failing service after a threshold of failures. Retries with a budget handle transient failures. Timeouts prevent requests from hanging indefinitely. Bulkheading limits the number of concurrent requests to the downstream service. These features are configured in the mesh without application changes.

## Summary

A service mesh extracts cross-cutting network concerns from application code into a dedicated infrastructure layer. The data plane consists of sidecar proxies that intercept all service-to-service traffic. The control plane configures the data plane with routing rules, security policies, and observability configuration. The three major implementations are Istio (feature-rich, Envoy-based), Linkerd (lightweight, Rust-based), and Consul Connect (HashiCorp ecosystem). The mesh provides traffic management, security, observability, and resiliency without modifying application code.

## Expanded Core Vocabulary

### SPIFFE

SPIFFE (Secure Production Identity Framework for Everyone) is a standard for identity in distributed systems. In a service mesh, every workload receives a SPIFFE ID in the form `spiffe://trust-domain/service-account`. The sidecar presents this identity during the TLS handshake. The control plane issues SPIFFE-compliant certificates through its certificate authority. This provides a consistent identity model across all services, regardless of the platform or programming language.

### Locality Load Balancing

A feature that routes traffic to the closest healthy endpoint based on geographic or topological locality. In Kubernetes, locality is defined by region, zone, and sub-zone. The mesh prefers endpoints in the same zone before sending traffic to another zone. This reduces latency, lowers cross-zone bandwidth costs, and improves resilience against zone failures. You configure locality weights to control the failover behavior when local endpoints are degraded.

### Outlier Detection

A mechanism that monitors proxy-side statistics for each endpoint and ejects those that exhibit anomalous behavior. The sidecar tracks consecutive 5xx errors, connection failures, and timeout events. When an endpoint exceeds a configurable threshold, it is ejected from the load balancing pool for a specified period. After ejection, the proxy periodically sends probe requests to check if the endpoint has recovered. Outlier detection is distinct from circuit breaking, which operates at the service level rather than the individual endpoint level.

### Header Propagation

The automatic forwarding of specific HTTP headers across service boundaries within the mesh. Distributed tracing headers like `x-request-id`, `x-b3-traceid`, and `x-datadog-trace-id` are propagated by the sidecar proxies without application modification. You can also define custom headers for context propagation, such as tenant IDs, correlation IDs, or authentication tokens. The mesh ensures that headers are preserved across all hops in the request chain.

### Access Logs

Detailed logs of every request that flows through the sidecar proxy. Each log entry contains the source IP, destination IP, protocol, method, path, response code, latency, request size, response size, and connection metadata. The mesh can stream access logs to stdout, local files, or external collectors like Fluentd, Logstash, or Splunk. Access logs are essential for auditing, debugging, and compliance investigations. You can configure the log format and sampling rate to balance detail against storage costs.

### Traffic Mirroring

A technique that copies live traffic to a shadow service without affecting the original request. The sidecar sends a copy of selected requests to a mirror destination while the primary request proceeds normally. The mirror destination processes the request, but its response is discarded. This is used for testing new versions with production traffic, validating configuration changes, and capacity planning. You configure mirroring rules in the VirtualService resource.

### Request Timeout

A per-request timeout enforced by the sidecar proxy. When a request exceeds the configured duration, the proxy returns a 504 Gateway Timeout response to the caller. This prevents slow downstream services from consuming resources indefinitely. You configure timeouts per route or per destination. The timeout is applied at the proxy level, which means it works even if the application does not implement its own timeout logic. Setting appropriate timeouts is one of the most impactful configurations you can make in the mesh.

### Connection Pool

A set of connections between sidecar proxies that are reused for multiple requests. The pool has configurable limits: maximum connections, maximum pending requests, maximum requests per connection, and idle timeout. Connection pooling reduces the overhead of establishing new TCP and TLS connections for every request. The proxy maintains separate pools for each upstream service, and you configure pool settings per destination using a DestinationRule.

## Comparison Table: Istio vs Linkerd vs Consul Connect

Choosing the right service mesh depends on your requirements for feature depth, operational complexity, performance, and ecosystem integration. The following table compares the three major implementations across key dimensions.

| Feature | Istio | Linkerd | Consul Connect |
|---------|-------|---------|----------------|
| Sidecar proxy | Envoy (C++) | linkerd2-proxy (Rust) | Envoy (C++) |
| Resource footprint per sidecar | High (~50MB) | Low (~10MB) | Medium (~50MB) |
| Control plane complexity | High (Pilot, Citadel, Galley, Istiod) | Low (destination, identity, injector) | Medium (Consul server cluster) |
| Supported platforms | Kubernetes, VMs, multi-cluster | Kubernetes only | Kubernetes, VMs, Nomad, Consul |
| Traffic management | VirtualService, DestinationRule, rich routing | ServiceProfiles, basic routing | Intentions, service splits |
| mTLS support | Automatic, SPIFFE-based | Automatic, mutual HTTP/2 | Automatic, SPIFFE-based |
| Protocol support | HTTP/1.1, HTTP/2, gRPC, TCP, WebSocket | HTTP/1.x, HTTP/2, gRPC, TCP | HTTP/1.1, HTTP/2, gRPC, TCP |
| Extensibility | WebAssembly filters, Envoy plugins | None (fixed proxy) | Envoy plugins |
| Multi-cluster support | Yes, mesh federation | Yes, service mirroring | Yes, WAN federation |
| Performance overhead per hop | 2-5ms | 1-2ms | 2-5ms |
| Community | Large, CNCF graduated | Large, CNCF graduated | Medium, HashiCorp backed |
| Learning curve | Steep | Moderate | Moderate |
| Certificate management | Istio CA, pluggable backends | Built-in identity controller | Consul CA, Vault integration |
| Default observability | Prometheus, Grafana, Kiali, Jaeger | Prometheus, Grafana, Jaeger | Prometheus, Grafana, Consul UI |

### When to choose each mesh

**Choose Istio when** you need the broadest feature set, multi-cluster support, or extensibility through WebAssembly. Istio is the best choice for large organizations with complex routing requirements and dedicated platform teams. It supports the widest range of deployment models, including hybrid cloud with VMs and Kubernetes side by side.

**Choose Linkerd when** simplicity and low resource usage are priorities. Linkerd is ideal for teams that want a straightforward mesh with minimal operational overhead. The Rust-based proxy uses significantly less CPU and memory than Envoy, making it suitable for resource-constrained environments. Linkerd's control plane has fewer components, which reduces the surface area for configuration errors.

**Choose Consul Connect when** you already use Consul for service discovery and health checking. Consul Connect provides deep integration with the HashiCorp ecosystem, including Vault for certificate management and Nomad for orchestration. If your infrastructure spans both Kubernetes and non-Kubernetes environments and you already run Consul agents, Consul Connect is the natural choice.

## Configuration Examples: Key Capabilities in Practice

These examples show how to configure common service mesh patterns using Istio's Kubernetes-native API resources. The same capabilities exist in Linkerd and Consul Connect, though the exact resource names and syntax differ.

### Traffic Splitting with Istio VirtualService

The following configuration splits traffic between two versions of a service. The sidecar proxy applies this rule without any application changes.

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-route
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        end-user:
          exact: jason
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v1
      weight: 80
    - destination:
        host: reviews
        subset: v2
      weight: 20
```

This rule routes all traffic from user `jason` to version 2 of the reviews service. All other traffic is split 80/20 between versions 1 and 2. The weighting is applied by the Envoy sidecar, so no load balancer or application code change is needed.

### Circuit Breaking with DestinationRule

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews-circuit-breaker
spec:
  host: reviews
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 10
        http2MaxRequests: 1000
        maxRequestsPerConnection: 10
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 60s
      maxEjectionPercent: 50
```

This configuration limits the connection pool to 100 TCP connections and ejects instances that return five consecutive 5xx errors. Ejected instances are excluded from the load balancing pool for 60 seconds. No more than 50 percent of endpoints can be ejected at once, which ensures that the service remains available even during a partial outage.

### Fault Injection for Resilience Testing

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ratings-fault-injection
spec:
  hosts:
  - ratings
  http:
  - fault:
      delay:
        fixedDelay: 7s
        percentage:
          value: 10
      abort:
        httpStatus: 500
        percentage:
          value: 5
    route:
    - destination:
        host: ratings
        subset: v1
```

This rule injects a 7-second delay on 10 percent of requests and returns a 500 error on 5 percent of requests. You use this configuration to validate that your system handles latency and failures gracefully before they occur in production. Fault injection is one of the most powerful testing tools that a service mesh provides.

### Retry and Timeout Configuration

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ratings-retry
spec:
  hosts:
  - ratings
  http:
  - timeout: 10s
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: gateway-error,connect-failure,refused-stream
    route:
    - destination:
        host: ratings
        subset: v1
```

This rule sets a 10-second total timeout for requests to the ratings service. If the request fails due to a gateway error, connection failure, or refused stream, the sidecar retries up to three times with a 2-second timeout per attempt. The overall request is capped at 10 seconds regardless of how many retries occur.

### Traffic Mirroring Configuration

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: httpbin-mirror
spec:
  hosts:
  - httpbin
  http:
  - route:
    - destination:
        host: httpbin
        subset: v1
      weight: 100
    mirror:
      host: httpbin
      subset: v2
    mirrorPercentage:
      value: 100
```

This configuration mirrors all traffic to version 2 of the httpbin service while the primary request continues to version 1. The mirror responses are discarded. You use this to test version 2 with production traffic before switching users to it. For more conservative testing, set a lower `mirrorPercentage`, such as 10 percent.

## When Does a Service Mesh NOT Help

A service mesh is a powerful tool, but it is not a solution for every problem. Understanding its limitations prevents you from adopting it in situations where it adds complexity without proportional benefit.

### Application-Level Bugs

A service mesh operates at the network layer. It cannot fix bugs in application logic, such as incorrect business rules, data corruption, or race conditions. If Service A sends a malformed JSON payload, the sidecar proxy forwards it faithfully. The mesh does not validate application-level data or enforce schema contracts beyond what the HTTP layer provides.

### Database Performance

The mesh handles service-to-service HTTP and gRPC traffic. It does not intercept database connections by default, and it does not optimize SQL queries, index usage, or connection pooling on the database side. If your bottleneck is a slow query, adding a service mesh will not improve throughput. Database performance requires query optimization, indexing strategies, and caching layers.

### Data Consistency and Transactions

A service mesh does not provide distributed transaction support, saga orchestration, or data consistency guarantees. If you need two-phase commit, event sourcing, or distributed saga coordination across multiple services, you need an application-level framework, not a mesh. The mesh ensures reliable transport, but it does not reason about application state.

### Latency-Sensitive Workloads

Every sidecar proxy adds 1-5ms of latency per hop. For a request that traverses 10 services, this adds 10-50ms of overhead. In high-frequency trading, real-time audio processing, or sub-millisecond RPC systems, this overhead may be unacceptable. For these cases, evaluate sidecar-less mesh alternatives or kernel-level solutions like eBPF that offer lower overhead.

### Small Deployments

If you run five services with simple communication patterns, a service mesh adds significant operational complexity. You must manage sidecar injection, control plane availability, proxy configuration, and certificate rotation. The operational overhead of the mesh may exceed the benefits it provides. For small deployments, application-level retry libraries such as resilience4j or simple TLS configuration are often sufficient.

### Monolithic Architectures

If you deploy a monolith (single process, single deployment), there is no service-to-service traffic to manage. A service mesh adds nothing but overhead. The mesh is only useful when there are multiple services communicating over the network. Do not adopt a service mesh as a forward-looking investment for a monolith that may never be decomposed.

### Protocols the Mesh Does Not Support

Not all service meshes support every protocol. If your services communicate over protocols that the mesh does not understand, the sidecar proxy may fall back to opaque TCP proxying, which loses the ability to apply L7 policies. For example, if your mesh does not support WebSocket and your services rely on it, consider whether the mesh adds value for those flows. Always verify protocol support before adopting a mesh.

## Mental Model Check: Additional Questions

Test your understanding with these additional scenarios.

### Question 6

You need to test a new version of the payment service with production traffic. You want to verify that the new version handles requests correctly without exposing users to failures. What mesh feature do you use, and how do you verify the results?

**Answer**: Use traffic mirroring. Configure the mesh to mirror a percentage of production traffic to the new payment service version. The primary request still goes to the current version, so users are unaffected. To verify results, compare the mirrored responses against the primary responses, or analyze the access logs and metrics from the mirrored service. If the mirrored service shows elevated error rates or latency, you fix the issues before switching users to it.

### Question 7

Your team has 15 microservices. Each service implements retry with exponential backoff independently. You notice that when Service D fails, the retry storms from all upstream services overload it further. How does a service mesh solve this problem differently from application-level retry?

**Answer**: A service mesh centralizes retry configuration in the control plane, making it consistent across all services. You can set a global retry budget (for example, 0.1 percent of requests may be retried) to prevent retry storms. The mesh also provides circuit breaking, which stops all traffic to a failing service rather than repeatedly retrying. With application-level retry, each team configures retry independently, making it harder to detect and prevent retry storms. The mesh gives you a single point of configuration and visibility across all retry behavior.

### Question 8

You deploy a service mesh on a Kubernetes cluster with 500 services. After enabling the mesh, you notice that control plane CPU usage is high and sidecar proxy startup time is slow. What could be the cause, and what mitigations are available?

**Answer**: The control plane may be overwhelmed by the number of sidecar connections. Each sidecar establishes an xDS connection to the control plane. With 500 services and multiple replicas per service, the total number of connections can reach thousands. Mitigations include: tuning the control plane resource limits, enabling push-based configuration updates instead of polling, limiting the scope of configuration pushed to each sidecar using the `Sidecar` resource, and using a dedicated node pool for control plane components. For startup time, configure readiness probes on sidecar containers and use gradual rollouts.

### Question 9

Your security team requires that all service-to-service traffic uses mTLS with certificates that rotate every 12 hours. How does the mesh handle this requirement compared to manual certificate management?

**Answer**: The mesh control plane's certificate authority issues certificates with a 12-hour TTL. Each sidecar proxy automatically requests a new certificate before the current one expires. The sidecar performs the TLS handshake with the new certificate seamlessly, without any application code changes or pod restarts. With manual management, each team would need to implement certificate generation, distribution, rotation, and revocation testing. The mesh automates the entire lifecycle, from issuance to rotation to revocation.

### Question 10

You are evaluating a service mesh for a system that includes both Kubernetes services and legacy virtual machine workloads. Which mesh implementations support this mixed environment?

**Answer**: Istio and Consul Connect support mixed environments with Kubernetes and VMs. Istio requires installing an Istio ingress gateway on the VM and registering the VM workload with the mesh using a WorkloadEntry resource. Consul Connect integrates with the Consul agent running on the VM, which already provides service discovery and health checking. Linkerd supports Kubernetes only. For mixed environments, choose Istio or Consul Connect.

## Expanded Step by Step: Request Lifecycle Detail

The request flow through the mesh involves several internal steps that warrant closer examination.

### Step 2a: Iptables Rule Evaluation

Before the sidecar intercepts traffic, iptables rules redirect inbound and outbound connections. The sidecar init container installs the following rules during pod startup:

- All outbound TCP traffic is redirected to the sidecar's inbound port (15006)
- All inbound TCP traffic not originating from the sidecar itself is redirected to the sidecar's outbound port (15001)
- Traffic to the sidecar's own ports is excluded from redirection

This transparent interception means the application does not know it is communicating through a proxy. The application sends traffic to what it believes is the destination, and the sidecar handles the actual network communication.

### Step 4a: Load Balancing Algorithm Selection

The sidecar supports several load balancing algorithms beyond round robin. The least request algorithm sends each request to the endpoint with the fewest active requests, which provides better load distribution under variable request durations. The consistent hash algorithm routes requests with the same hash key to the same endpoint, which improves cache hit rates. The random algorithm selects endpoints uniformly at random, which works well when all endpoints have similar capacity. You configure the algorithm in the DestinationRule.

### Step 6a: Connection Pool Limits Check

Before selecting an endpoint, the sidecar checks the connection pool limits. If the maximum number of connections to the upstream cluster is reached, the sidecar queues the request. If the queue exceeds the maximum pending requests, the sidecar returns a 503 Service Unavailable response to the caller. This prevents the upstream service from being overwhelmed by concurrent connections. The connection pool limits act as a backpressure mechanism at the proxy level.

### Step 7a: SPIFFE Identity Verification

During the TLS handshake, the destination sidecar extracts the SPIFFE ID from the source sidecar's certificate. The SPIFFE ID has the format `spiffe://cluster.local/ns/namespace/sa/service-account`. The destination sidecar verifies that the SPIFFE ID is authorized to communicate with the destination service using the authorization policies configured in the control plane. If authorization fails, the connection is rejected before any application-level processing occurs. This verification happens entirely at the proxy level.

### Step 9a: HTTP Filter Chain Processing

The destination sidecar processes the request through a chain of HTTP filters before forwarding it to the application. The filter chain may include the following components in order:

- CORS filter: handles cross-origin request validation
- Fault injection filter: injects delays or errors for testing scenarios
- Rate limit filter: enforces request rate limits per source or destination
- RBAC filter: evaluates authorization rules against the request attributes
- Telemetry filter: records metrics, traces, and access log entries

Each filter can modify the request, reject it with a response, or pass it unchanged to the next filter in the chain. The filter chain is fully configurable through the control plane and can be extended with custom WebAssembly filters.
