## Foundation Challenges

### Challenge 1: Sidecar Explanation

**Problem**: Your organization has 50 microservices written in 4 languages (Java, Python, Go, Node.js). Each team implements retry, timeout, and metrics independently. Your manager asks you to explain how a service mesh would solve this problem without requiring each team to change their code. Provide a clear explanation.

**Model Answer**:

A service mesh solves this by inserting a sidecar proxy (such as Envoy) alongside each service. The sidecar intercepts all network traffic via iptables rules in the pod. You configure retry policies, timeouts, and metrics collection centrally in the mesh control plane. The control plane distributes these policies to all sidecar proxies.

No application code changes are needed because:
- iptables transparently redirects traffic through the sidecar
- The sidecar applies policies (retry, timeout) as traffic passes through
- The sidecar emits metrics in a standard format (Prometheus)
- Distributed tracing headers are propagated automatically

The result is consistent behavior regardless of the programming language. Java, Python, Go, and Node.js services all get the same retry count, timeout duration, and metrics format.

Key points to include:
- Sidecar proxy intercepts traffic transparently
- Centralized configuration via control plane
- Language-agnostic because proxies handle network concerns
- No code changes needed at the application layer

### Challenge 2: mTLS Mode Selection

**Problem**: Your team is migrating to a service mesh with mTLS. You have 200 services, 20 of which do not support TLS (they use plain HTTP and cannot be modified in the current quarter). How do you configure mTLS to balance security and backward compatibility?

**Model Answer**:

Use Istio's PERMISSIVE mTLS mode for the migration period.

Configuration:
```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: PERMISSIVE
```

In PERMISSIVE mode:
- Services that support mTLS will use encrypted connections
- Services that only support plain HTTP will continue working with unencrypted connections
- Both modes are accepted by the destination proxy

Migration plan:
1. Set the mesh-wide policy to PERMISSIVE mode
2. Verify that all services communicate correctly
3. Override the policy for each TLS-capable namespace to STRICT mode:

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: strict
  namespace: tls-capable-namespace
spec:
  mtls:
    mode: STRICT
```

4. For the 20 services that cannot support TLS, keep the PERMISSIVE policy on their namespace
5. When those services are updated to support TLS (next quarter), switch their namespace to STRICT
6. Eventually set the mesh-wide default to STRICT

Key points to include:
- PERMISSIVE mode allows gradual migration
- Namespace-level overrides for fine-grained control
- Eventual goal of STRICT mode everywhere
- No downtime during migration

### Challenge 3: Traffic Splitting Configuration

**Problem**: You are deploying version 2 of Service B. You want to send 5% of traffic to the new version while sending 95% to the stable version. Write the VirtualService configuration and explain how the sidecar proxies implement the traffic split.

**Model Answer**:

Istio VirtualService configuration:
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: service-b-routing
spec:
  hosts:
  - service-b
  http:
  - route:
    - destination:
        host: service-b
        subset: stable
      weight: 95
    - destination:
        host: service-b
        subset: canary
      weight: 5
```

DestinationRule defining the subsets:
```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: service-b-subsets
spec:
  host: service-b
  subsets:
  - name: stable
    labels:
      version: v1
  - name: canary
    labels:
      version: v2
```

How the sidecar implements this:

1. The control plane (Pilot) translates this VirtualService into Envoy route configuration
2. The sidecar receives the route configuration via RDS
3. For each request to `service-b`, the sidecar generates a random number between 1 and 100
4. If the number is 1-5, the request routes to the `canary` cluster (version v2 pods)
5. If the number is 6-100, the request routes to the `stable` cluster (version v1 pods)
6. The sidecar uses Envoy's weighted cluster load balancing

Each sidecar independently applies this logic, resulting in approximately 5% of total traffic reaching the canary version.

Key points to include:
- VirtualService defines traffic weights
- DestinationRule defines subsets based on pod labels
- Each sidecar independently evaluates weights using random selection
- Statistical distribution means actual split approximates the configured ratio

## Design Challenges

### Challenge 4: Multi-Cluster Service Mesh Design

**Problem**: Your company operates in three regions (us-east, eu-west, ap-southeast). Each region has its own Kubernetes cluster. You need to deploy a service mesh that allows services in different regions to communicate with mTLS. Services should prefer local endpoints when available and fall back to remote regions if the local service is unhealthy. Design the mesh architecture.

**Model Answer**:

Architecture: Multi-primary Istio mesh with a shared root CA.

Components:
1. Each region runs its own Istio control plane (istiod)
2. A shared root CA issues intermediate certificates for each region's Citadel
3. Each region's Istio is configured with the other regions' API servers for service discovery
4. ServiceEntry resources define remote services with their region and endpoints

Configuration:

Root CA setup:
```
Root CA -> Intermediate CA (us-east) -> Service certificates
        -> Intermediate CA (eu-west) -> Service certificates
        -> Intermediate CA (ap-southeast) -> Service certificates
```

Service discovery cross-region:
```yaml
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: service-b-eu-west
spec:
  hosts:
  - service-b.global
  addresses:
  - 240.0.0.1
  ports:
  - number: 8080
    name: http
    protocol: HTTP
  location: MESH_INTERNAL
  resolution: DNS
  endpoints:
  - address: eu-west-cluster-endpoint
    ports:
      http: 15443
    locality: eu-west
```

Locality-aware routing:
```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: service-b-locality
spec:
  host: service-b.global
  trafficPolicy:
    loadBalancer:
      localityLbSetting:
        enabled: true
        failover:
        - from: us-east
          to: eu-west
```

Traffic flow:
1. A request from a us-east service goes to the local service-b endpoint
2. If no local endpoints are healthy, the request fails over to eu-west
3. If eu-west also has no healthy endpoints, the request fails over to ap-southeast
4. All cross-region traffic uses mTLS with certificates signed by the shared CA

Key points to include:
- Multi-primary model for independent control planes
- Shared root CA for cross-region mTLS
- ServiceEntry for cross-region service discovery
- Locality-aware load balancing with failover priority
- gRPC or VPN connectivity between clusters

### Challenge 5: Observability Stack Design

**Problem**: Design the observability stack for a service mesh with 1000 services and 10,000 requests per second. You need metrics, distributed tracing, and access logs. Describe the components, data flow, storage requirements, and query patterns.

**Model Answer**:

Components:
1. **Metrics**: Prometheus scrapes Envoy metrics (port 15090) from each sidecar. Prometheus stores metrics with 15s resolution. Retention: 30 days for raw metrics, 1 year for aggregated.

2. **Tracing**: Jaeger collects OpenTelemetry-format spans from Envoy. Sampling rate: 1% for production (200 traces/s), 100% for development. Storage: Elasticsearch backend for Jaeger.

3. **Access logs**: Envoy writes structured access logs to stdout. Fluentd daemonset collects logs and sends to Elasticsearch. Example log format:

```
{"timestamp":"2026-07-02T10:45:12.345Z","method":"POST","path":"/api/v1/orders","status":200,"duration":"45ms","source":"service-a","destination":"service-b","traceId":"abc123"}
```

Storage calculations:
- Metrics: 2000 metrics per proxy x 1000 proxies = 2,000,000 metrics. At 15s scrape: 133,333 samples/s. Each sample ~1KB: 133 MB/s for raw ingestion. After compression and aggregation: ~10 GB/day.
- Traces: 1% of 10,000 req/s = 100 traces/s. Average span count: 5 spans per trace. 500 spans/s. Each span ~500 bytes: 250 KB/s. ~21 GB/day.
- Access logs: 10,000 logs/s. Each log ~300 bytes: 3 MB/s. ~260 GB/day.

Query patterns:
- Real-time dashboards: Grafana queries Prometheus for latency, error rate, request volume
- Trace investigation: Jaeger UI to search by service, operation, latency, error
- Log analysis: Kibana to search by trace ID, service, error code
- Service graph: Kiali to visualize topology and traffic flow

Key points to include:
- Separate components for metrics, traces, and logs
- Sampling strategy for tracing to reduce storage
- Storage calculations to justify infrastructure
- Query patterns for different use cases

### Challenge 6: Migration Strategy Design

**Problem**: Your organization has 300 services without a mesh. You need a migration strategy that minimizes risk, allows rollback, and completes within 9 months. Design the migration plan with phases, validation criteria, and rollback procedures.

**Model Answer**:

Phase 0: Foundation (Month 1)
- Deploy Istio control plane in a dedicated namespace
- Configure a sidecar injector with selective injection (label-based, not global)
- Deploy monitoring (Prometheus, Grafana, Jaeger, Kiali)
- Create installation runbooks and team training materials
- Validation: Control plane health checks pass, dashboards show no data yet

Phase 1: Pilot (Month 2)
- Select 3 low-risk services (internal tools, no customer traffic)
- Enable sidecar injection on their namespaces
- Validate: Compare metrics (latency, error rate) before and after mesh
- Test mTLS in PERMISSIVE mode
- Test retry and timeout policies
- Test circuit breaking with fault injection
- Validation: Sidecar metrics match pre-mesh metrics within 5%

Phase 2: Expansion (Months 3-4)
- Enable injection on 30 more services (non-critical, internal-facing)
- Monitor control plane load: CPU, memory, gRPC connections
- Test canary deployment with weighted traffic splitting
- Test mTLS in STRICT mode on PILOT namespace
- Establish alerting for mesh health
- Validation: No increase in P99 latency exceeding 10% for any service

Phase 3: Scale (Months 5-7)
- Enable injection on remaining 267 services
- Monitor and tune resource allocations for proxies
- Implement egress gateway for external API calls
- Train operations team on mesh troubleshooting
- Create runbooks for common issues (certificate expiry, circuit breaker trips)
- Validation: All services have consistent metrics format. Cross-team consistency verified.

Phase 4: Optimization (Months 8-9)
- Remove application-level resilience code from all services
- Set mTLS to STRICT mode mesh-wide
- Fine-tune retry, timeout, and circuit breaker policies
- Implement canary upgrade process for the control plane itself
- Document mesh architecture and operational procedures
- Validation: Operations team can manage mesh independently

Rollback procedures:
- Per-service: Remove the injection label from the namespace, the webhook no longer injects
- Fleet-wide: Disable the webhook, existing sidecars remain until pod restart
- Control plane: Canary upgrade with rollback to previous version
- Data plane: Roll back to previous sidecar image version

Key points to include:
- Phased approach with validation gates
- Selective injection for risk control
- Metrics-based validation criteria
- Rollback procedures at each level
- Team training and documentation
- 9-month timeline with clear milestones

### Challenge 7: Custom Service Mesh Design

**Problem**: Design a service mesh for a high-frequency trading platform with sub-millisecond latency requirements. Off-the-shelf solutions (Istio, Linkerd) add too much overhead. Describe your custom proxy design, what features you include, what you omit, and how you achieve sub-millisecond overhead.

**Model Answer**:

Design principles:
1. Minimize the processing path per request
2. Use lock-free data structures
3. Skip features that are not strictly needed
4. Optimize for the 99th percentile, not the average

Custom proxy design:
- **Language**: Rust (or C++) for zero-GC, deterministic performance
- **Architecture**: Single-threaded event loop per core (similar to DPDK/Seastar)
- **Kernel bypass**: Use AF_XDP or DPDK to avoid kernel network stack
- **Protocol**: Limited to TCP and a single binary protocol (no HTTP parsing)
- **Encryption**: WireGuard instead of mTLS for lower overhead (0.05ms vs 1ms)
- **Configuration**: Static config loaded at startup (no dynamic xDS)
- **Routing**: Hash-based routing with a pre-computed routing table
- **Circuit breaker**: Simple binary state per destination (no sliding window)
- **Metrics**: Minimal set of counters per destination (no histograms per request)

Features omitted:
- HTTP/2 and gRPC support (not needed for custom binary protocol)
- Dynamic route updates (config change requires restart)
- L7 routing (no path/header-based routing)
- Distributed tracing (use packet capture for debugging instead)
- Rate limiting (handled at the application layer)
- Fault injection (not needed for production)
- WASM extensibility (adds overhead)

Performance targets:
- Per-hop latency: 0.1-0.3ms (vs 1-2ms for Istio)
- Per-proxy CPU: 0.1 core at 10,000 req/s
- Per-proxy memory: 20 MB
- No dynamic memory allocation in the request path

Deployment:
- Sidecar runs in a separate container on the same host
- No iptables (use eBPF for traffic interception to avoid kernel overhead)
- Direct communication between proxies on the same host via shared memory

Key points to include:
- Tradeoff: fewer features for lower latency
- Language and architecture choices for performance
- Static configuration eliminates xDS overhead
- Kernel bypass and eBPF for minimal latency
- WireGuard for lightweight encryption

### Challenge 8: Istio Control Plane Scaling

**Problem**: You manage an Istio mesh with 10,000 sidecar proxies. The control plane (istiod) is experiencing high CPU usage and slow xDS updates. Configuration changes take 5-10 minutes to propagate to all proxies. Design a solution to scale the control plane for 10,000+ proxies with sub-minute configuration convergence.

**Model Answer**:

Problem analysis:
- 10,000 proxies each maintain a gRPC stream to istiod
- Each xDS update must push to all 10,000 proxies
- gRPC stream multiplexing causes contention on the control plane
- Full configuration snapshot on each update is too large

Scaling solutions:

1. **Horizontal scaling of istiod**: Deploy 3-5 istiod replicas. Each replica handles a subset of proxies (2000-3000 per replica). Use a leader election mechanism to ensure only one replica responds to configuration changes.

2. **Configuration sharding**: Partition proxies into groups. Each istiod replica handles one group. Add a router layer that assigns proxies to replicas:

```
Ingress -> Router (consistent hash on proxy ID) -> istiod-replica-1 (3000 proxies)
                                                 -> istiod-replica-2 (3000 proxies)
                                                 -> istiod-replica-3 (3000 proxies)
                                                 -> istiod-replica-4 (1000 proxies)
```

3. **Incremental xDS**: Use incremental xDS (ADS) instead of full snapshot pushes. Only send changed resources. This reduces the per-update data size from megabytes to kilobytes.

4. **Delta proxy configuration**: Instead of pushing the same configuration to every proxy, customize per-proxy configuration. A proxy only receives routes for services it communicates with (sidecar.istio.io/proxyCPULimit). This reduces the per-proxy configuration by 80-90%.

5. **gRPC tuning**:
   - Increase `grpc-max-concurrent-streams` on istiod
   - Use connection pooling on the client side
   - Set appropriate timeouts for gRPC streams
   - Enable gRPC keepalive probes

6. **Caching**: Implement a two-level cache:
   - Level 1: istiod caches computed Envoy configuration
   - Level 2: proxies cache their current configuration
   - Only cache misses trigger full computation

Performance targets:
- Configuration convergence: 30 seconds for 90% of proxies
- istiod CPU: less than 4 cores per replica (for 2000 proxies)
- istiod memory: less than 8 GB per replica
- xDS push latency: under 1 second for incremental updates

Monitoring:
- xDS push latency per proxy (P50, P99)
- istiod CPU and memory per replica
- Number of gRPC connections per replica
- Configuration version per proxy (detect drift)

Key points to include:
- Horizontal scaling with sharding
- Incremental xDS reduces data transfer
- Per-proxy configuration customization
- gRPC tuning for large-scale connections
- Caching strategy for repeated computations
- Performance targets for convergence time
