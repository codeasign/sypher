Service mesh architecture evolves from naive application-level implementations through production-grade deployments with full sidecar injection and control plane management. This section covers the naive approach and its problems, the production architecture with Istio and Envoy, integration patterns with existing infrastructure, multi-region deployment considerations, capacity planning, and operational practices.

### Problems with the Naive Approach

**Inconsistent Policies**: Each team chooses different retry counts, timeout values, and circuit breaker thresholds. Service A retries 3 times with a 5-second timeout. Service B retries 5 times with a 2-second timeout. When A calls B, the effective timeout is the minimum of the two timeouts (2s), and the total retry load is 15 requests (3x5). This inconsistency causes unpredictable latency and load patterns.

**Code Duplication**: Every service must implement retry logic, timeout handling, circuit breakers, metrics export, and tracing header propagation. A 200-microservice system has 200 implementations of these patterns. Each implementation has subtle differences and bugs.

**Language Lock-in**: Resilience libraries exist for specific languages (Hystrix for Java, Polly for .NET, resilience4j for Java). Teams using different languages cannot share implementations. Polyglot environments (Python, Go, Node.js, Java, Rust) require multiple library support.

**No Centralized Visibility**: Without a mesh, there is no unified view of service-to-service communication. Each service exports metrics independently. Correlating a request across 5 services requires manual log correlation across 5 different log formats.

**Security Gaps**: mTLS requires each service to manage TLS certificates. Some teams skip encryption entirely. Others use self-signed certificates with no rotation. There is no consistent identity model.

### Sidecar Injection

In Kubernetes, sidecar injection is automated using a mutating webhook admission controller:

1. A pod creation request is sent to the Kubernetes API server
2. The API server calls the sidecar injector webhook (part of the mesh control plane)
3. The webhook mutates the pod spec to add the Envoy sidecar container
4. The webhook also adds:
   - Init container to configure iptables rules
   - Volume mounts for certificates and configuration
   - Environment variables for proxy configuration (pod name, namespace, service account)
5. The modified pod spec is returned to the API server
6. Kubernetes creates the pod with the sidecar

The injection is controlled by namespace labels:
```yaml
metadata:
  labels:
    istio-injection: enabled
```

Or by pod annotations:
```yaml
metadata:
  annotations:
    sidecar.istio.io/inject: "true"
```

### Ingress and Egress Gateways

**Ingress Gateway**: An Envoy proxy deployed at the edge of the mesh. It handles traffic from external clients. Key responsibilities:

- TLS termination for external connections
- Rate limiting for external traffic
- Authentication (JWT validation, OAuth)
- Request routing to internal services
- Load balancing across service instances
- WebSocket and gRPC support

**Egress Gateway**: An Envoy proxy that handles outbound traffic to external services. Key responsibilities:

- Controlled access to external APIs
- TLS origination for external connections
- Traffic monitoring for external calls
- Policy enforcement for external access
- IP address management for external firewalls

### Observability Stack

The mesh integrates with standard observability tools:

**Prometheus**: Scrapes metrics from Envoy proxies (default port 15090). Metrics include request count, duration, size, and TCP statistics. Prometheus stores metrics with 15s resolution.

**Grafana**: Provides dashboards for mesh visualization. Default dashboards include mesh dashboard (global traffic), service dashboard (per-service metrics), and workload dashboard (per-pod metrics).

**Kiali**: A service mesh-specific visualization tool. It shows the service graph, traffic flow, health status, and configuration validation. Kiali uses Prometheus as its data source.

**Jaeger**: Distributed tracing system. Envoy emits spans in Zipkin or OpenTelemetry format. Jaeger collects and visualizes trace data, showing the path of each request through the mesh.

**Elasticsearch**: Stores access logs from Envoy proxies. Each log entry includes source, destination, latency, response code, and metadata.

**Prometheus scraping configuration**: Each Envoy proxy exposes metrics on port 15090 at `/stats/prometheus`. Configure Prometheus to scrape all sidecars:

```yaml
scrape_configs:
- job_name: 'istio-mesh'
  kubernetes_sd_configs:
  - role: pod
  relabel_configs:
  - source_labels: [__meta_kubernetes_pod_container_port_name]
    action: keep
    regex: '.*-envoy-prom'
  - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
    action: replace
    regex: ([^:]+)(?::\\d+)?;(\\d+)
    replacement: $1:$2
    target_label: __address__
  metric_relabel_configs:
  - source_labels: [__name__]
    regex: 'istio_(requests_total|request_duration_milliseconds|request_byte_size|response_byte_size|tcp_.*)'
    action: keep
```

Key metrics to monitor:

| Metric | Description | Alert Threshold |
|--------|-------------|-----------------|
| `istio_requests_total` | Total request count per service | Spike > 2x baseline |
| `istio_request_duration_milliseconds_bucket` | Request duration histogram | p99 > 500ms |
| `istio_request_byte_size` | Request body size | N/A, capacity planning |
| `istio_response_byte_size` | Response body size | Baseline for cost |
| `istio_tcp_sent_bytes_total` | TCP bytes sent | Connection tracking |
| `istio_requests_total{response_code=~"5.*"}` | Error rate | > 1% over 5 minutes |

**Grafana dashboards**: Install the Istio dashboards from the Grafana marketplace or configure them manually:

1. **Mesh Dashboard**: Global view of request volume, success rate, and latency across all services
2. **Service Dashboard**: Per-service metrics with upstream and downstream breakdowns
3. **Workload Dashboard**: Per-pod metrics with CPU, memory, and request concurrency
4. **Performance Dashboard**: Control plane and data plane resource usage

**Jaeger sampling strategies**: Configure Jaeger to sample traces cost-effectively:

```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-default
  namespace: istio-system
spec:
  tracing:
  - providers:
    - name: jaeger
    randomSamplingPercentage: 1.0
    customTags:
      environment:
        literal:
          value: production
```

For high-traffic services, use adaptive sampling: trace 100% of requests under 1000 req/s, then sample 1% above that threshold. This ensures you capture traces for low-traffic services while controlling cost for high-traffic ones.

**Kiali service graph**: Kiali builds a real-time service graph from Prometheus metrics. It shows:

- Services and their dependencies
- Traffic rates with animated edges
- Health status (green/yellow/red per node)
- Configuration validation with error highlighting
- mTLS status per connection (encrypted vs plaintext)
- Request routing visualization for canary deployments

Configure alerting rules for error rate (> 1% over 5 minutes), high latency (p99 > 500ms), and certificate expiry (< 7 days remaining). Use Prometheus alerting rules with the `istio_requests_total`, `istio_request_duration_milliseconds_bucket`, and `istio_agent_cert_expiry_seconds` metrics.

The API gateway and service mesh serve complementary roles:

**API Gateway (north-south)**:
- Handles external client traffic
- Performs authentication (JWT, OAuth, SAML)
- Enforces rate limits per client
- Provides caching for external API calls
- Translates protocols (REST to gRPC)

**Service Mesh (east-west)**:
- Handles internal service traffic
- Provides mTLS encryption
- Implements retry and circuit breaking
- Collects telemetry for all internal calls
- Routes traffic for canary deployments

The integration point is at the ingress: the API gateway forwards requests to internal services through the ingress gateway, which connects to the mesh. The API gateway authenticates and the mesh secures internal traffic.

Multi-cluster deployments use one of two models:

**Primary-Remote Model**:
- One primary cluster runs the control plane
- Remote clusters run only the data plane (sidecars)
- Remote clusters connect to the primary control plane via VPN or peering
- Simpler management, but single control plane is a failure domain

**Multi-Primary Model**:
- Each cluster runs its own control plane
- Control planes federate with each other
- Each cluster is independent in terms of control
- More complex setup, but better isolation

Key considerations for multi-cluster:
- Service discovery across clusters requires DNS or endpoint synchronization
- mTLS between clusters requires cross-CA trust (root CA sharing)
- Latency between clusters affects routing decisions (geo-routing)
- Network connectivity between clusters must be reliable (VPN, Cloud Interconnect, Direct Connect)

### Service Mesh with Legacy Services

Legacy services that cannot run sidecars (due to resource constraints or compatibility) can still participate in the mesh:

1. **Proxy proxy**: Deploy a standalone Envoy proxy alongside the legacy service on the same host
2. **Sidecar per host**: One sidecar for multiple services on the same VM
3. **Mesh expansion**: Istio's VM integration allows VMs to join the mesh using WorkloadEntry and connecting via VPN

### Service Mesh with Message Queues

Message brokers (Kafka, RabbitMQ, NATS) present a challenge for service meshes because they use asynchronous communication. The mesh can still add value:

- **mTLS between producers and brokers**: Encrypt connections from producer sidecars to the broker
- **mTLS between brokers and consumers**: Encrypt connections from the broker to consumer sidecars
- **Observability on broker interactions**: Track produce and consume operations even if the mesh does not intercept the broker protocol
- **Network policy enforcement**: Control which services can produce to or consume from specific topics

However, the mesh cannot intercept traffic that uses non-HTTP protocols at the L7 level unless the protocol is understood. Kafka traffic over plain TCP is passed through as TCP proxy traffic. For advanced Kafka observability, use the Envoy Kafka filter:

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: kafka-filter
spec:
  workloadSelector:
    labels:
      app: kafka-broker
  configPatches:
  - applyTo: NETWORK_FILTER
    match:
      listenerFilter: "envoy.filters.listener.http_inspector"
      context: SIDECAR_INBOUND
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.network.kafka_broker
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.network.kafka_broker.v3.KafkaBroker
          broker_address_pattern: ".*"
          stat_prefix: kafka_istio
```

### Service Mesh with Serverless Functions

Serverless platforms (Knative, AWS Lambda, Google Cloud Run) integrate with service meshes at the ingress layer:

- **Knative with Istio**: Knative uses Istio as its networking layer by default. The Knative activator and queue-proxy work alongside the Istio sidecar
- **Function-to-function calls**: Serverless functions call mesh services through the ingress gateway or directly if they run within the mesh
- **Cold start considerations**: The sidecar must be ready before the function receives traffic. Use readiness gates to synchronize
- **Resource overhead**: Serverless functions have tight resource limits. The sidecar adds 50-100ms to cold start time and 30-60 MB memory overhead

When you deploy Knative with Istio, the request flow is: external request reaches the Knative ingress gateway, which routes to the activator, which routes to the function pod with its sidecar. The sidecar handles mTLS, tracing, and metrics for function-to-service calls.

### MutatingWebhookConfiguration

The sidecar injector registers itself as a `MutatingWebhookConfiguration` resource. When the API server receives a pod creation request, it calls the webhook endpoint before persisting the resource:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: istio-sidecar-injector
webhooks:
- name: sidecar-injector.istio.io
  clientConfig:
    service:
      name: istiod
      namespace: istio-system
      path: /inject
    caBundle: <base64-encoded-ca-cert>
  rules:
  - operations: ["CREATE"]
    apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
  namespaceSelector:
    matchLabels:
      istio-injection: enabled
  reinvocationPolicy: IfNeeded
  sideEffects: None
```

The webhook applies to pod creation in namespaces with the `istio-injection: enabled` label. The `reinvocationPolicy: IfNeeded` ensures the webhook can re-run if other mutating webhooks (service mesh addons, custom admission controllers) modify the pod spec after the initial injection.

### Init Container: iptables Configuration

The init container (`istio-init`) runs with `CAP_NET_ADMIN` to configure iptables rules that redirect traffic through the Envoy sidecar. The key rules are:

```bash
# Redirect all inbound TCP traffic to Envoy (port 15006)
iptables -t nat -A PREROUTING -p tcp -j REDIRECT --to-port 15006

# Redirect all outbound TCP traffic from the application to Envoy (port 15001)
iptables -t nat -A OUTPUT -p tcp -j REDIRECT --to-port 15001

# Exclude traffic to the sidecar itself
iptables -t nat -A OUTPUT -p tcp -d 127.0.0.1 -j RETURN

# Exclude traffic on specified ports (probes, metrics)
iptables -t nat -A OUTPUT -p tcp --dport 15020 -j RETURN
iptables -t nat -A OUTPUT -p tcp --dport 15090 -j RETURN
```

The exclusion list includes ports 15020 (status/probe port) and 15090 (metrics port) so that health checks and Prometheus scraping bypass the sidecar and avoid circular routing. You can extend the exclusion list with the `traffic.sidecar.istio.io/excludeInboundPorts` and `traffic.sidecar.istio.io/excludeOutboundPorts` annotations.

### Envoy Bootstrap Configuration

The sidecar container starts with a bootstrap configuration that points to the control plane (istiod on port 15012), specifies certificate paths for mTLS (cert-chain.pem, key.pem, root-cert.pem), and defines the admin interface on port 15000 for debugging. The node ID is set to `sidecar~<pod_ip>~<pod_name>.<namespace>~<namespace>.svc.cluster.local`.

### Sidecar Lifecycle

The sidecar container uses a `preStop` lifecycle hook to drain connections before shutdown:

```yaml
lifecycle:
  preStop:
    exec:
      command:
      - /bin/sh
      - -c
      - |
        curl -X POST http://127.0.0.1:15000/drain_listeners?inboundonly
        sleep 5
```

This ensures the sidecar stops accepting new connections and drains existing ones before the pod terminates. Without this hook, you see connection reset errors during rolling updates.

### Troubleshooting Injection Failures

Check injection status with `kubectl get namespace <namespace> -o yaml | grep istio-injection`. Verify the webhook with `kubectl get mutatingwebhookconfiguration istio-sidecar-injector -o yaml`. Simulate injection with `istioctl proxy-config inject <deployment-name>`.

Common injection failures: missing namespace label (`istio-injection: enabled`), webhook timeout (must respond within 30 seconds), init container blocked by pod security policies (missing `CAP_NET_ADMIN`), and insufficient node resources for the sidecar.

## Multi-Region Deployment

### Active-Active Mesh

Two or more regions each run a full mesh deployment. Traffic is routed based on geography:

```
User in New York -> us-east1 mesh -> service-b.us-east1
User in Tokyo -> ap-northeast1 mesh -> service-b.ap-northeast1
```

Each region has:
- Its own control plane (multi-primary model)
- Its own data plane
- Its own observability stack

Cross-region calls are used only when necessary (for data dependencies). The mesh routes traffic to the nearest region using DNS-based geo-routing or latency-based routing.

### Disaster Recovery (Active-Passive)

Primary region handles all traffic. Secondary region runs in standby:

- Primary region control plane and data plane are fully active
- Secondary region data plane is active but receives no traffic
- Secondary region control plane is warm (ready to take over)
- On failure, DNS and routing rules are updated to point to the secondary region
- Traffic shifting takes 5-15 minutes for DNS propagation

### Cross-Region Latency

Cross-region latency varies by distance:

| Region Pair | Typical Latency |
|-------------|-----------------|
| us-east1 to us-west1 | 60-70ms |
| us-east1 to eu-west1 | 75-90ms |
| us-west1 to ap-southeast1 | 100-140ms |
| us-east1 to ap-northeast1 | 150-200ms |

The mesh manages cross-region traffic through mTLS with longer timeouts and retry budgets tuned for higher latency.

## Capacity Planning

### Control Plane Sizing

The control plane (istiod) must handle connections from all sidecar proxies:

| Number of Proxies | CPU | Memory | Connections |
|-------------------|-----|--------|-------------|
| 100 | 0.5 core | 1 GB | 100 gRPC |
| 500 | 1 core | 2 GB | 500 gRPC |
| 1000 | 2 core | 4 GB | 1000 gRPC |
| 5000 | 4 core | 8 GB | 5000 gRPC |
| 10000 | 8 core | 16 GB | 10000 gRPC |

For clusters with more than 5000 proxies, horizontal scaling of the control plane (multiple replicas) is recommended. Each replica handles a subset of proxies.

### Data Plane Sizing

Each sidecar proxy requires resources proportional to its request volume:

| Requests/Second | CPU | Memory |
|-----------------|-----|--------|
| 100 | 0.1 core | 64 MB |
| 500 | 0.3 core | 128 MB |
| 1000 | 0.5 core | 256 MB |
| 5000 | 2 core | 512 MB |
| 10000 | 4 core | 1 GB |

Resource limits should be set 2x the expected usage to handle traffic spikes:

```yaml
resources:
  requests:
    cpu: 500m
    memory: 256Mi
  limits:
    cpu: 1000m
    memory: 512Mi
```

### Gateway Sizing

Ingress and egress gateways handle higher traffic volumes than sidecars:

| Throughput | CPU | Memory | Replicas |
|------------|-----|--------|----------|
| 1000 req/s | 1 core | 1 GB | 2 |
| 10000 req/s | 4 core | 4 GB | 4 |
| 100000 req/s | 16 core | 16 GB | 10 |

### Sizing Formulas

Use these formulas to estimate resource requirements before deploying:

**Control plane connections**:
```
total_connections = number_of_proxies * 2
cpu_needed_cores = max(0.5, number_of_proxies * 0.0008)
memory_needed_gb  = max(1.0, number_of_proxies * 0.0016)
```

For example, with 3000 proxies:
- total_connections = 3000 * 2 = 6000 gRPC connections
- cpu_needed_cores = max(0.5, 3000 * 0.0008) = 2.4 cores (round up to 3)
- memory_needed_gb = max(1.0, 3000 * 0.0016) = 4.8 GB (round up to 5)

**Sidecar per-request overhead**:
```
additional_cpu_per_request  = 0.0001 cores
additional_memory_per_request = 0.05 MB
total_sidecar_cpu  = base_cpu + (requests_per_second * additional_cpu_per_request)
total_sidecar_memory = base_memory + (requests_per_second * additional_memory_per_request)
```

For a service handling 2000 req/s:
- total_sidecar_cpu = 0.1 + (2000 * 0.0001) = 0.3 cores
- total_sidecar_memory = 64 + (2000 * 0.05) = 164 MB

**Gateway throughput estimation**:
```
required_replicas = ceil(requests_per_second / (per_replica_throughput * headroom_factor))
headroom_factor = 0.7
```

For 25000 req/s with 5000 req/s per replica:
- required_replicas = ceil(25000 / (5000 * 0.7)) = ceil(7.14) = 8 replicas

Apply a headroom factor of 0.7 to absorb traffic spikes. For production deployments, set the HPA (Horizontal Pod Autoscaler) target at 60% CPU to trigger scaling before resources are exhausted.

### Memory Per Connection Detail

Each sidecar maintains several connections. The fixed overhead is approximately 704 KB (control plane, admin, and metrics endpoints). Variable overhead depends on the upstream connection pool size and concurrent request count. Tune connection pool limits in `DestinationRule` to balance memory usage against latency.

## Network Security Policies and mTLS Configuration

Service meshes provide a security model based on mutual TLS, peer authentication, request authentication, and authorization policies. These four layers form a defense-in-depth strategy for service-to-service communication.

### mTLS Modes

Istio supports three mTLS modes configured through `PeerAuthentication`: `STRICT` (all traffic must use TLS), `PERMISSIVE` (accepts TLS and plaintext, for migration), and `DISABLE` (no TLS, for debugging). Set global mTLS to STRICT in production once all services are part of the mesh. Override per namespace during migration using `PeerAuthentication` with a `selector`. The migration strategy: start with PERMISSIVE globally, monitor for plaintext traffic in Kiali, and switch to STRICT once all services support mTLS.

### AuthorizationPolicy

Authorization policies define who can talk to whom. They follow a DENY-over-ALLOW pattern where DENY rules take precedence. A policy can restrict by source (service account), operation (HTTP method, path), and conditions (JWT claims). Common patterns: default deny all, namespace isolation, environment isolation, and principle of least privilege.

### RequestAuthentication

Request authentication validates JWT tokens at the mesh boundary. The sidecar validates the token before forwarding to the application, returning 401 for invalid tokens. Configure the issuer and JWKS URI in the `RequestAuthentication` resource.

### Certificate Management

Istio manages certificates for mTLS through Citadel. The Root CA is valid for 10 years (manual rotation), the Intermediate CA for 1 year (automatic), and workload certificates for 24 hours (automatic, rotated by sidecar). Check certificate status with `istioctl proxy-config secret <pod-name> -n <namespace>`. Monitor `istio_agent_cert_expiry_seconds` and alert when certificates have fewer than 7 days remaining.

### Security Best Practices

1. **Enable STRICT mTLS**: Disable PERMISSIVE mode once the migration completes
2. **Default deny AuthorizationPolicy**: Apply a deny-all policy in every namespace, then add allow rules
3. **Use unique service accounts**: Every pod should have a unique service account. Do not use the default service account
4. **Restrict egress traffic**: Use Egress Gateway with authorization policies for external API access
5. **Rotate certificates frequently**: The default 24-hour workload certificate lifetime is appropriate. Do not increase it
6. **Audit network traffic**: Enable Envoy access logging and ship logs to Elasticsearch for auditing
7. **Monitor security metrics**: Track mTLS handshake failures, expired certificates, and denied requests

## Operations

### Canary Upgrades

Upgrade the control plane using a canary deployment:
1. Deploy a new version of the control plane (istiod-canary)
2. Configure a small percentage of proxies to connect to the canary
3. Monitor error rates, latency, and resource usage
4. If stable after 30 minutes, promote the new version
5. If errors appear, rollback and investigate

Sidecar upgrades follow the same pattern: update the sidecar image in the injection configuration, the webhook injects new sidecars for new pods. Existing pods are updated by rolling restart.

### Health Checks

The mesh health is monitored through:

- **Control plane health**: istiod readiness probe (port 8080, path /ready)
- **Sidecar health**: Envoy health check (port 15000, path /ready)
- **Gateway health**: Gateway readiness probe (port 15021, path /healthz/ready)
- **Certificate health**: Certificate expiry monitoring (alert if less than 7 days remaining)

### Troubleshooting

Common issues and diagnostics:

**Sidecar not intercepting traffic**: Check iptables rules in the pod:
```
istioctl proxy-config log <pod-name> --level debug
kubectl exec <pod> -c istio-proxy -- iptables-save
```

**Route not matching correctly**: Inspect Envoy configuration:
```
istioctl proxy-config routes <pod-name>
istioctl proxy-config listeners <pod-name> -o json
```

**mTLS handshake failures**: Check certificates:
```
istioctl proxy-config secret <pod-name>
istioctl auth check <source-pod> <destination-pod>
```

**High latency through proxy**: Check connection pool and circuit breaker state:
```
istioctl proxy-config clusters <pod-name> -o json | jq '.[] | {name, circuitBreakers, outlierDetection}'
```

## Day-2 Operations

Beyond initial deployment, a service mesh requires ongoing operational practices for scaling, cost management, and upgrades.

### Backup and Restore

The mesh configuration lives in Kubernetes custom resources (CRDs). Back up all mesh-related resources and store in version control. The control plane is stateless; to restore, reapply the CRD backups and redeploy istiod.

### Scaling Events

During traffic spikes, the data plane scales with application pods via HPA. Ensure HPA accounts for sidecar resource requests -- sidecar CPU is part of the pod CPU metric. Use at least 2 istiod replicas with pod anti-affinity across nodes for control plane high availability.

### Cost Management

Each sidecar adds cost. At typical cloud pricing ($25/core/month, $3/GB/month), a sidecar with 0.5 CPU and 256 MB memory costs about \$13/month. For 1000 sidecars, this is roughly \$13,000/month. Optimize by right-sizing resource requests, using resource quotas, and applying selective injection only to namespaces that need mesh features.

### Upgrade Strategy

Use revision-based upgrades for zero-downtime mesh updates: run `istioctl analyze` before upgrading, install the new revision alongside the existing one, migrate namespaces with `istio.io/rev` labels, roll pods to pick up the new sidecar, and monitor for 30 minutes before cleaning up the old revision.

## Summary

Production service mesh architecture moves from naive application-level resilience code to a dedicated infrastructure layer with sidecar injection, gateways, and a centralized control plane. The architecture integrates with API gateways for north-south traffic and supports multi-cluster and multi-region deployments. Capacity planning for the control plane (istiod) scales with proxy count, and data plane resources scale with request volume. Operational practices include canary upgrades, health checks, and structured troubleshooting.
