## Tier 1: Core Concepts

### Challenge 1: Design a Write-Back Cache from Scratch

**Problem**: Design a write-back cache implementation for a single-node application. Your cache must support `get(key)`, `set(key, value, ttl)`, and must flush dirty entries to a backing store every 500ms. You have 30 minutes to design and pseudo-implement the core components.

**Requirements**:
- Support concurrent reads and writes from multiple application threads.
- Flush dirty entries in batches of up to 100 entries.
- Handle the case where the same key is written multiple times between flushes (coalescing).
- Handle the case where a dirty entry exceeds its TTL before being flushed.
- Provide a mechanism to prevent unbounded memory growth of the dirty buffer.

**Model Answer**:

Your answer should cover these components:

1. **Dirty Buffer**: A concurrent hash map (`ConcurrentHashMap`, `sync.Map`, or sharded map) that stores entries with a dirty flag, value, and timestamp.

2. **Flush Loop**: A background thread that runs every 500ms. It iterates over the dirty buffer, collects entries where `dirty == true`, assembles them into batches of 100, and writes each batch to the backing store.

3. **Coalescing**: When the flush loop iterates, it reads the latest value for each key. If the same key appears in the dirty buffer only once (the map entry was updated in place), no special handling is needed. The flush loop simply reads the current value.

4. **TTL Handling**: When a dirty entry's TTL expires, the entry should still be flushed (it represents a write that the application considers committed). After flushing, the entry can be evicted. Do not discard dirty entries on TTL expiry.

5. **Memory Bound**: Set a maximum size for the dirty buffer (e.g., 10,000 entries). When the limit is reached, apply backpressure: block the `set()` call until the flush loop drains some entries, or reject the write and return an error.

```python
import threading
import time
from dataclasses import dataclass
from typing import Any, Optional

@dataclass
class CacheEntry:
    value: Any
    dirty: bool
    timestamp: float
    ttl: Optional[float]

class WriteBackCache:
    def __init__(self, flush_interval=0.5, batch_size=100, max_dirty=10000):
        self._data = {}  # key -> CacheEntry
        self._lock = threading.RLock()
        self._flush_interval = flush_interval
        self._batch_size = batch_size
        self._max_dirty = max_dirty
        self._running = True
        self._flusher = threading.Thread(target=self._flush_loop, daemon=True)
        self._flusher.start()

    def get(self, key: str) -> Optional[Any]:
        with self._lock:
            entry = self._data.get(key)
            if entry is None:
                return None
            if entry.ttl and (time.time() - entry.timestamp) > entry.ttl:
                # Expired but if dirty, still return it
                if not entry.dirty:
                    del self._data[key]
                    return None
            return entry.value

    def set(self, key: str, value: Any, ttl: Optional[float] = None):
        with self._lock:
            dirty_count = sum(1 for e in self._data.values() if e.dirty)
            if dirty_count >= self._max_dirty:
                raise RuntimeError("Dirty buffer full")
            self._data[key] = CacheEntry(
                value=value,
                dirty=True,
                timestamp=time.time(),
                ttl=ttl
            )

    def _flush_loop(self):
        while self._running:
            time.sleep(self._flush_interval)
            self._flush()

    def _flush(self):
        batch = []
        with self._lock:
            for key, entry in self._data.items():
                if entry.dirty and len(batch) < self._batch_size:
                    batch.append((key, entry.value, entry.ttl))
                    entry.dirty = False
        for key, value, ttl in batch:
            # Write to backing store
            self._write_to_db(key, value, ttl)

    def _write_to_db(self, key, value, ttl):
        # Simulated database write
        pass

    def shutdown(self):
        self._running = False
        self._flusher.join()
        self._flush()  # Final flush of remaining dirty entries
```

### Challenge 2: Data Loss Analysis

**Problem**: Your write-back cache has a flush interval of 1 second. The database batch write takes 100ms. At peak, you handle 50,000 writes per second with a coalescing ratio of 4:1. How many writes can be lost in the worst-case scenario? What happens if the database is unavailable for 30 seconds?

**Model Answer**:

**Worst-case data loss on cache crash**:
- Flush interval: 1,000ms
- Batch write time: 100ms
- Durability window: 1,100ms (1,000ms until next flush + 100ms batch write)
- Writes in flight: 50,000 writes/sec * 1.1s = 55,000 writes
- With 4:1 coalescing: 55,000 / 4 = 13,750 unique keys affected
- If average value size is 1 KB: 55 MB of data lost

**Database unavailable for 30 seconds**:
- New writes accumulated: 50,000 * 30 = 1,500,000 writes
- Coalesced: 375,000 unique keys
- Memory needed: 375 MB (at 1 KB per entry)
- If dirty buffer capacity is exceeded: backpressure activates, writes are throttled or rejected
- After DB recovery: it takes 375,000 / (500 batch size * 100ms) = 75 seconds to drain the backlog
- Total time to fully recover: 30s (outage) + 75s (drain) = 105 seconds

### Challenge 3: WAL Design

**Problem**: Design a Write-Ahead Log for a write-back cache. Your WAL must support crash recovery with no more than 100ms of data loss. The cache handles 20,000 writes per second. Each WAL entry is 200 bytes. You have 1 GB of available disk space for the WAL.

**Model Answer**:

**WAL Format**: Append-only binary log with fixed-size entries. Each entry:
```
┌────────┬──────────┬──────────┬────────┬────────┐
│ CRC32  │ Seq num  │ Key len  │ Value  │ Flags  │
│ (4 B)  │ (8 B)    │ (2 B)    │ (var)  │ (1 B)  │
└────────┴──────────┴──────────┴────────┴────────┘
```

**Sync mode**: Group commit with 50ms sync interval. Every 50ms, accumulated WAL entries are fsynced to disk. This limits maximum data loss to 50ms of writes (plus any in-flight).

**Rotation**: The WAL is rotated every 30 seconds. Each segment is 20,000 * 200 * 30 = 120 MB. With 1 GB of disk, we can keep 8 full segments (4 minutes of history). Old segments are deleted only after all their entries have been flushed to the database.

**Recovery**: On restart, scan all WAL segments in sequence order. Build a map of key to latest value (deduplication). Insert into dirty buffer. Resume flushing. Total recovery time: 1 GB / 2 GB/s (sequential read speed) = 0.5 seconds.

## Tier 2: Applied Design

### Challenge 4: Multi-Data-Type Cache

**Problem**: Your write-back cache must handle three data types with different requirements:

| Data Type | Max Acceptable Data Loss | Write QPS | Max Write Latency |
|-----------|------------------------|-----------|-------------------|
| User settings | 0 (no loss) | 500 | 100ms |
| Like counters | 5 seconds | 80,000 | 2ms |
| Analytics events | 60 seconds | 20,000 | 10ms |

Design a single write-back cache that meets all three requirements. The system has two database shards.

**Model Answer**:

**Design**: Use a single write-back cache with per-key-priority flushing. The cache maintains three separate dirty buffers (or a single buffer with priority tags).

- **User settings** (high priority): Sync WAL. Flush immediately (every 50ms). Batch size of 10. Single flusher thread dedicated to high-priority flushes. Durability window: under 100ms.
- **Like counters** (medium priority): Async WAL with group commit. Flush every 500ms. Batch size of 500. Dedicated flusher thread. Coalescing aggressively (counters are highly coalesceable). Durability window: under 1 second.
- **Analytics events** (low priority): Async WAL (no fsync). Flush every 30 seconds. Batch size of 5,000. Low-priority flusher thread that yields to higher-priority work. Durability window: up to 60 seconds.

**Database sharding**: User settings go to shard 0. Like counters and analytics events go to shard 1. This isolates critical writes from bulk writes.

**Failure handling**: If the database shard for user settings is unavailable, the system immediately alerts. If the analytics shard is unavailable, the circuit breaker opens and writes are buffered for up to 60 seconds before backpressure.

### Challenge 5: Read-Heavy Workload Adaptation

**Problem**: Your write-back cache is handling 100,000 writes/second well with 500ms flush interval. However, you notice that reads to keys that were recently written (and are still dirty) are slower than expected. Investigation reveals that the flusher is holding a read lock on the dirty buffer, blocking concurrent reads. Design a solution that maintains write-back performance while allowing concurrent reads without blocking.

**Model Answer**:

**Solution: Copy-on-write buffer with read optimization**.

Replace the single dirty buffer with a double-buffering scheme:

1. **Active buffer**: New writes go here. Lock-free reads (RCU or atomic load).
2. **Flush snapshot**: When the flusher is ready, it atomically swaps the active buffer with an empty buffer. The flusher works on the snapshot while new writes go to the (now empty) active buffer.

This eliminates read-write contention because reads always access the active buffer, and the flusher works on the snapshot.

**Additional optimization**: Add a Bloom filter for the dirty buffer. A read for key K first checks the Bloom filter. If the filter says "K is not dirty," the read can skip checking the dirty buffer entirely and go straight to the clean cache or database.

```
Read Path:
1. Check Bloom filter for "is K dirty?"
2. If NO: Read from clean cache (fast path)
3. If YES: Read from active dirty buffer (may be dirty, must check)
4. If not in dirty buffer: Read from clean cache or database

Result: 99.9% of reads (for non-dirty keys) skip the dirty buffer check.
```

### Challenge 6: Dead Letter Queue Recovery

**Problem**: After a 10-minute database outage, your write-back cache has 100,000 entries in the retry queue. The database is now healthy. However, the retry queue is attempting to flush entries in the order they failed, which means it is processing stale data. Many of those keys have been updated in the dirty buffer with newer values. How do you handle this without double-writing stale data?

**Model Answer**:

**Solution: Version-based deduplication**.

Each dirty entry carries a version number (monotonic counter). When the flush of version 5 fails and the entry goes to the retry queue, the key may later be updated with version 8 in the dirty buffer.

On retry, before writing to the database, the flusher checks the current version in the dirty buffer:
- If `retry_version < current_version`: The retry entry is stale. Discard it. The current dirty buffer entry will be flushed separately.
- If `retry_version == current_version`: The retry entry is still current. Flush it to the database.
- If key is not in dirty buffer: The entry is no longer in the cache. Flush the retry value to the database (best effort).

```python
def process_retry_queue(self):
    for key, retry_entry in self.retry_queue:
        with self._lock:
            current = self._data.get(key)
            if current and current.version > retry_entry.version:
                # Stale retry, discard
                continue
        # Current or not in cache, flush to DB
        self._write_to_db(key, retry_entry.value)
```

This requires a version field in the dirty buffer entry and the retry queue entry. The version is incremented on every `set()` call for that key.

## Tier 3: Advanced System Design

### Challenge 7: Active-Active Multi-Region Write-Back

**Problem**: Design a write-back cache that operates across three geographic regions (US-East, EU-West, APAC). Each region has its own application servers and write-back cache cluster. Writes in any region must be visible in all regions within 5 seconds (eventual consistency). Each region handles 50,000 writes/second. Conflicts are inevitable: the same user may update their profile from both US and EU within the same second.

Design the system. Address conflict resolution, cross-region flush coordination, and data loss scenarios.

**Model Answer**:

**Architecture**:
- Each region has a local write-back cache cluster (3 nodes per region).
- Each region has a local database instance (primary for reads in that region).
- Cross-region replication uses a CDC stream at the database level (Debezium → Kafka → target region).
- A global flush coordinator manages cross-region flush scheduling.

**Conflict Resolution**: Last-Write-Wins (LWW) with hybrid logical clocks (HLC). Each write is tagged with an HLC timestamp that combines the node's wall clock and a logical counter. HLC eliminates clock skew issues within the same data center and reduces cross-datacenter clock drift impact.

For counters: CRDT delta state. Each region sends the delta (increment by 3) rather than the absolute value (set to 1,502,493). Deltas are merged algebraically.

**Consistency**: Read-your-writes within a region (reads hit local cache, which has the latest write). Cross-region reads may see stale data for up to 5 seconds (the flush + replication delay).

**Data Loss**: Each region has a sync WAL. If a region's cache crashes, the WAL replay recovers dirty data. If an entire region goes down, the other regions continue operating. When the region comes back, it replays its local WAL and syncs from the global CDC stream.

### Challenge 8: Write-Back for Time-Series Data

**Problem**: Design a write-back cache for a time-series metrics system. The system receives 10 million data points per second (metric readings from servers, applications, and infrastructure). Each data point is a tuple: (metric_name, timestamp, value, tags). The database is a time-series database (TSDB) like InfluxDB or TimescaleDB.

The requirements are unusual:
- Data points are immutable (no updates, only inserts)
- The write pattern is append-only and monotonically increasing in timestamp
- Data loss of up to 10 seconds is acceptable
- Query latency for recent data (last 5 minutes) must be under 10ms
- The TSDB can handle batch inserts of up to 100,000 data points in a single write

Design the write-back cache, taking advantage of the time-series workload properties.

**Model Answer**:

**Design**: A specialized write-back cache optimized for append-only time-series data.

**Data Structure**: Instead of a hash map keyed by metric name, use a sorted ring buffer keyed by timestamp. Each buffer segment covers a 1-second window. New data points are appended to the current segment. When the current segment is full (exceeds 500,000 points or 100ms elapsed), it is sealed and moved to the flush queue.

**Flush**: The flusher reads sealed segments in timestamp order and batches them for the TSDB. The batch size is tuned to the TSDB's optimal insert size (100,000 points per batch). Since data points are immutable, there is no coalescing -- but there is also no need for it, because there are no updates to the same key.

**Read optimization**: The ring buffer is kept in memory for the last 5 minutes (300 seconds). Queries for recent data read from the ring buffer without touching the TSDB. This provides under 10ms query latency for recent data.

**Durability**: Since data loss of up to 10 seconds is acceptable, the WAL is async. The WAL is a sequential log on NVMe storage. Recovery replays unflushed segments.

```python
class TimeSeriesWriteBack:
    def __init__(self, segment_duration=1.0, max_segments=300, batch_size=100000):
        self._segments = []  # List of sealed segments
        self._current = Segment()  # Active segment
        self._segment_duration = segment_duration
        self._max_segments = max_segments  # 5 minutes at 1s segments
        self._batch_size = batch_size
        self._flusher = threading.Thread(target=self._flush_loop, daemon=True)
        self._flusher.start()

    def write(self, metric, timestamp, value, tags):
        self._current.append(metric, timestamp, value, tags)
        if self._current.age > self._segment_duration:
            self._seal_current()

    def _seal_current(self):
        segment = self._current
        self._current = Segment()
        self._segments.append(segment)
        if len(self._segments) > self._max_segments:
            # Remove oldest sealed segment (still in DB, no data loss)
            self._segments.pop(0)

    def query_recent(self, metric, start_time, end_time):
        # Search ring buffer only (under 10ms)
        results = []
        for segment in reversed(self._segments):
            if segment.timestamp_range(start_time, end_time):
                results.extend(segment.query(metric, start_time, end_time))
        results.extend(self._current.query(metric, start_time, end_time))
        return results
```

**Key insight**: The immutable, append-only nature of time-series data simplifies write-back significantly. No coalescing, no version conflicts, no causal dependencies. The ring buffer provides read optimization naturally. The only concern is the durability window, which is bounded by the segment duration times the number of unflushed segments.
