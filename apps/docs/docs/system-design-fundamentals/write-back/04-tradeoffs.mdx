## The Core Tension

Write-back caching exists because of a fundamental truth in distributed systems: **you cannot have low latency, high throughput, strong consistency, and full durability all at once.** Every system must choose which properties to sacrifice.

Write-back makes a clear trade: it trades durability and consistency for dramatically higher write throughput and lower latency. The app sees 1ms response times instead of 22ms. The database sees 500 writes per second instead of 50,000. But the system accepts the risk of losing a few seconds of data on a crash and serving stale data on read-replicas.

This is the same trade that underlies nearly every high-performance storage system: Apache Cassandra, Redis with AOF, RocksDB, Kafka, and countless others. They all use some form of write-back because the alternative -- synchronous writes to durable storage -- cannot scale to internet-scale write loads.

## Decision Matrix

Choosing a write strategy requires comparing approaches across multiple dimensions. Use this matrix to decide when write-back is the right choice.

| Dimension | Write-Back | Write-Through | Write-Around |
|-----------|------------|---------------|--------------|
| **Write latency** | 0.1-2ms | 10-30ms | 10-30ms (bypasses cache) |
| **Read latency (hot data)** | 0.1-1ms | 0.1-1ms | 10-30ms (cache miss) |
| **Write throughput** | 10K-100K/s | 500-5K/s | 500-5K/s |
| **Data loss risk** | High (window) | None | None |
| **Read-your-write** | Yes (from cache) | Yes | No (cache miss) |
| **DB write reduction** | 5x-100x (coalescing) | 1x | 1x |
| **Implementation complexity** | High | Low | Low |
| **Recovery time on crash** | Seconds (WAL replay) | Instant | Instant |
| **Operational cost** | Moderate | High (DB load) | High (DB load) |

### Flush Strategy Comparison

| Dimension | Periodic Flush | Threshold Flush | Priority Flush |
|-----------|---------------|-----------------|----------------|
| **Predictable DB load** | Yes | No (bursty) | Partially |
| **Adapts to write rate** | No | Yes | Yes |
| **Durability window variance** | Fixed | Variable | Variable |
| **Implementation complexity** | Low | Low | Medium |
| **Best for** | Steady write loads | Variable traffic | Mixed criticality |

### WAL Mode Comparison

| Dimension | Async WAL | Sync WAL | Group Commit WAL |
|-----------|-----------|----------|-----------------|
| **Write latency added** | 0.1ms | 1-5ms | 0.5-2ms |
| **Data loss on crash** | Last 1-10ms of writes | None | Last 0.1ms |
| **Throughput impact** | Minimal | 30-50% reduction | 10-20% reduction |
| **Best for** | Analytics, counters | Financial, critical | Most production systems |

### Flush Ordering Comparison

| Dimension | Ordered (FIFO) | Unordered | Partial Order |
|-----------|---------------|-----------|---------------|
| **Causal consistency** | Yes | No | Yes (per key group) |
| **Throughput** | Baseline | 20-30% higher | 10-15% higher |
| **Max batch size** | 500 (must preserve order) | Unlimited | 1000 |
| **Best for** | Append logs, audit trails | Independent counters | User profiles, sessions |

## CAP Theorem and Write-Back

Under the CAP theorem, write-back caching is an **AP** system. It chooses Availability and Partition Tolerance over Consistency.

During a network partition that separates the cache from the database, a write-back cache continues to accept writes. This is the "Available" property. The dirty buffer grows because flushes cannot reach the database. When the partition heals, the flusher drains the accumulated dirty entries and the system returns to consistency.

If the partition lasts too long, the dirty buffer hits its capacity limit. At this point, the cache must either apply backpressure (throttle writes) or drop writes. Both choices sacrifice availability, but for different reasons. This is the inherent tension: you can have availability up to the capacity of your dirty buffer, after which you must either degrade or fail.

## Failure Modes

A write-back cache has several well-known failure modes. Understanding them is essential for building a robust system.

### Failure Mode 1: Data Loss on Cache Crash

**What happens**: The cache node crashes before dirty entries are flushed to the database. Without a WAL, all dirty data is lost. With an async WAL, the last few milliseconds of writes may be lost.

**Detection**: On restart, the dirty buffer is empty (no WAL) or partially populated (WAL replay). The database is missing writes that the application considers committed.

**Impact**: Depending on the data, this ranges from acceptable (lost a few analytics events) to catastrophic (lost financial transactions).

**Prevention**: Use a sync WAL (or group commit WAL) to ensure every acknowledged write is on disk before the ACK is sent. Replicate the WAL to a remote storage for added safety.

**Recovery**: With WAL replay, the recovered dirty entries are flushed to the database. Without WAL, the application must detect the inconsistency (read-your-write check) and re-write lost data.

### Failure Mode 2: Dirty Buffer Overflow

**What happens**: Write rate exceeds flush rate consistently. The dirty buffer fills up. The cache must decide what to do: apply backpressure (slow down ACKs) or drop writes.

**Detection**: The dirty ratio metric exceeds the configured threshold. The flush backlog is growing. Write latency increases as backpressure kicks in.

**Impact**: Write latency spikes (backpressure) or data loss (dropped writes). Application performance degrades.

**Prevention**: Configure proper capacity for the dirty buffer. Monitor dirty ratio. Alert on sustained high dirty ratios. Ensure the flusher thread pool is adequately sized.

**Recovery**: If the database was the bottleneck, add database capacity (sharding, read replicas, faster storage). If the flusher was the bottleneck, increase the flusher thread pool and batch size.

### Failure Mode 3: Flush Backlog Spiral

**What happens**: The flush backlog grows. The flusher falls further behind because flushing a larger backlog takes longer. This increases the dirty buffer occupancy, which increases memory pressure, which slows the garbage collector (or allocator), which further slows the flusher. It is a positive feedback loop.

**Detection**: The flush backlog metric shows a sustained increase over multiple flush cycles. The flush latency increases as batches grow.

**Impact**: Eventual dirty buffer overflow (failure mode 2). The system enters a death spiral.

**Prevention**: Implement early backpressure. Do not wait for the dirty buffer to be 80% full -- start throttling at 40%. The goal is to prevent the backlog from growing beyond what the flusher can drain in a single flush cycle.

**Recovery**: Reduce the write rate at the application layer (feature flags, rate limiting, circuit breakers). Increase database capacity. The backlog drains naturally once the write rate drops below the flush rate.

### Failure Mode 4: WAL Corruption

**What happens**: The WAL file is corrupted due to a disk error, power failure during a write, or a software bug. On restart, WAL replay fails or produces incorrect results.

**Detection**: WAL replay fails with checksum errors, parsing errors, or unexpected termination.

**Impact**: Dirty data may be partially or fully lost. Recovery may require manual intervention.

**Prevention**: Use checksums on each WAL entry. Use a CRC-32 or CRC-64 checksum that is verified during replay. Use a journaling or copy-on-write filesystem (ZFS, Btrfs) to reduce the risk of partial WAL writes.

**Recovery**: Truncate the corrupted portion of the WAL (the last entry that fails checksum verification is discarded). Flush all entries up to the corruption point. Log the corruption event for investigation. Consider this a data loss incident and trigger the appropriate process.

### Failure Mode 5: Write Reordering Causing Inconsistency

**What happens**: The application writes K1 then K2, with K2 depending on K1 (e.g., K2 is a comment on a post, K1 is the post counter). The flusher writes K2 before K1 to the database. The database enforces a foreign key constraint and rejects K2. Or worse: no constraint exists, and K2 references a counter that has not been updated yet.

**Detection**: Errors from the database (constraint violations). Or silent logical inconsistency: the database state does not match the application's causal expectations.

**Impact**: Application errors (constraint violations) or silent data corruption (causal chains broken).

**Prevention**: Track causal dependencies explicitly. If two keys are causally related, they must be in the same flush batch or flushed in order. The flush queue supports batch grouping for dependent entries.

**Recovery**: For constraint violations, the flush is retried (the dependent entry may succeed on retry after the dependency is flushed). For silent corruption, periodic consistency checks detect and repair the issue.

## Anti-Patterns

### Anti-Pattern 1: No WAL

Using write-back without a Write-Ahead Log means every cache crash causes data loss. This is acceptable for analytics data (losing the last 5 seconds of page views is fine) but catastrophic for user-facing data.

**Fix**: Always add a WAL. Use async WAL for analytics, group commit WAL for general purposes, and sync WAL for critical data.

### Anti-Pattern 2: Too Large Flush Interval

Setting the flush interval to 30 seconds to maximize batching efficiency. This creates a 30-second durability window. A crash during peak traffic loses 30 seconds of writes. For a system doing 70,000 writes/second, that is over 2 million lost writes.

**Fix**: Keep the flush interval under 1 second for most systems. Use priority-based flushing if you need to batch some writes without delaying critical ones.

### Anti-Pattern 3: No Backpressure

The dirty buffer grows without bound. When memory is exhausted, the cache crashes (OOM), losing all dirty data. This is the most common write-back failure in production.

**Fix**: Implement backpressure at 40% dirty ratio. Throttle writes progressively as the dirty ratio increases. Reject writes when the dirty buffer exceeds 95% capacity.

### Anti-Pattern 4: Ignoring Dirty Ratio

Monitoring only cache hit rate and not dirty ratio. The hit rate looks fine because dirty entries are in the cache. But the dirty ratio is 60% and climbing. The operator does not notice until the cache OOMs.

**Fix**: Add dirty ratio to the monitoring dashboard. Alert when it exceeds 40%.

### Anti-Pattern 5: Single Flusher Thread

The flusher is a single goroutine/thread. When it blocks on a slow database write, no other flushes happen. The dirty buffer grows. Writes back up.

**Fix**: Use a thread pool with at least 4 flusher threads. Partition the dirty buffer so each thread owns a portion.

## When NOT to Use Write-Back

Write-back is not a universal solution. Avoid it in these scenarios:

**Financial transactions**: Debit/credit operations, payment processing, and any operation where data loss means monetary loss. Use write-through or synchronous replication instead.

**Strong consistency required**: Systems where readers must always see the latest write, even if the cache has failed. Write-back provides eventual consistency only.

**Data must not be lost**: Regulatory environments (healthcare, finance) that mandate zero data loss. Write-back always has a durability window, even with sync WAL (the window is just smaller).

**Write-once-read-many workloads**: If data is written once and read many times, write-through or write-around is simpler and sufficient. The batching and coalescing advantages of write-back do not apply.

**Low write volume**: If you are doing fewer than 100 writes/second, the complexity of write-back is not justified. Write-through with a connection pool will handle the load with simpler code.

When you evaluate whether to use write-back, ask: "How much data can I afford to lose?" If the answer is "none," write-back is not for you. If the answer is "a few seconds is acceptable," write-back is a strong candidate.
