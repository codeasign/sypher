## Company Case Studies

The world's largest internet systems all use write-back caching in some form. The specific implementation varies, but the core pattern is always the same: accept writes fast, flush them in batches, and handle the durability and consistency trade-offs explicitly.

### Facebook: Memcached Write-Back for Likes

Facebook's likes system is the canonical example of write-back caching at scale. Every like on a post, photo, or comment is a write. During Super Bowl commercials, Facebook sees hundreds of thousands of likes per second. Write-through would require a database cluster the size of a small country.

**Architecture**: Facebook uses a custom Memcached layer that acts as a write-back cache for like counters. When you double-tap a photo, the like count is incremented in Memcached and acknowledged immediately. A background process (called "McSqueal") tails the Memcached write log and asynchronously applies the updates to MySQL.

**Key insight**: Facebook accepted that losing a few likes in a crash is acceptable. The cultural value is "a like is ephemeral." The user will re-like if the first one disappears. This made write-back the perfect choice.

**Failure experience**: Early versions of McSqueal had a bug where duplicate entries in the tail log caused double-counting. A post would jump from 1,000 likes to 2,000 likes after a crash recovery. Facebook fixed this by making the flush operation idempotent: instead of `SET likes = likes + 1`, they used `SET likes = <absolute value>` with a version check.

### YouTube: View Count Buffer

YouTube's view counter is another canonical write-back system. Every play of a video increments the view count. At YouTube's scale (billions of views per day), writing each view to the database synchronously would be impossible.

**Architecture**: YouTube uses a two-tier write-back approach. The first tier is an in-memory buffer on the front-end servers that aggregates views for 5 seconds. The second tier is a distributed write-back cache (similar to Cloud Bigtable) that buffers and coalesces view count updates before writing to the permanent storage layer.

**Failure experience**: During high-traffic events (music video premieres), the view count write-back cache has been known to overflow, causing a temporary pause in view count updates. YouTube's response was to add backpressure that delays the view count response to the user rather than dropping the view entirely.

### Apache Cassandra: Write-Back Commit Log

Cassandra is a write-back system by design. Every write goes to the commit log (a WAL on disk) and then to the memtable (an in-memory write buffer). The memtable is flushed to SSTables (on-disk immutable data files) when it reaches a threshold size.

**Key insight**: Cassandra's write-back design is why it achieves such high write throughput. The commit log write is the only synchronous disk operation. Everything else is batched and asynchronous. This is also why Cassandra can lose data on a crash if the commit log is not flushed synchronously.

**Failure experience**: With the default `commitlog_sync: periodic` setting, Cassandra may lose up to the commit log sync interval's worth of writes on a node crash. Production deployments at Netflix use `commitlog_sync: batch` for critical data, accepting higher latency for zero data loss.

### Redis AOF (Append-Only File)

Redis provides an optional durability mode called AOF. Every write command is appended to an append-only file on disk. The AOF can be synced to disk every second (default), every write, or never.

**Key insight**: Redis AOF with `appendfsync everysec` is the classic write-back trade-off: accept up to 1 second of data loss for 100x throughput improvement over synchronous writes.

**Failure experience**: Redis's AOF rewrite process can cause latency spikes on small instances. During a rewrite, Redis forks a child process that writes a new AOF file. On memory-constrained systems, the fork can cause the OOM killer to terminate Redis. Production deployments use `auto-aof-rewrite-percentage` set conservatively.

## Open Source Implementations

### Redis AOF

As described above, Redis AOF is a write-back durability mechanism. The in-memory data store is the write-back cache; the AOF file is the WAL. Configuration: `appendonly yes`, `appendfsync everysec`.

### Apache Kafka (Flush on Produce)

Kafka producers can acknowledge writes with varying levels of durability. The `acks` setting controls this: `acks=0` (fire and forget), `acks=1` (leader write, no flush), `acks=all` (all replicas acknowledged). The leader's write is to the in-memory page cache, with periodic flushing to disk controlled by `flush.messages` and `flush.ms`.

### RocksDB WAL

RocksDB uses a WAL for every write. The WAL is written before the memtable is updated. When `DB::Put` returns, the write is in the WAL buffer but may not be flushed to disk. The `WAL::Flush` (with `WAL::Sync`) controls durability. This is write-back with optional synchronous durability.

### MySQL InnoDB Buffer Pool

MySQL's InnoDB storage engine uses a buffer pool that is a write-back cache. Pages are modified in memory and flushed to disk by the background page cleaner thread. Configuration parameters: `innodb_max_dirty_pages_pct` (default 75%), `innodb_io_capacity`, `innodb_flush_neighbors`.

## Cloud Provider Services

### AWS ElastiCache (Redis/Memcached)

AWS ElastiCache supports the write-back pattern through Redis AOF persistence. When enabled, ElastiCache Redis nodes write all commands to an AOF file stored on EBS. On node replacement, the AOF is replayed. The durability is as good as the AOF configuration.

The **DAX (DynamoDB Accelerator)** service is a write-back cache for DynamoDB. Writes to DAX are acknowledged immediately and asynchronously flushed to DynamoDB. This provides lower latency for write-heavy DynamoDB workloads, but with the standard write-back data loss risk.

### GCP Memorystore

Google Cloud's Memorystore for Redis supports AOF persistence similar to AWS ElastiCache. The write-back pattern applies: writes are fast but not durably stored until the AOF or RDB snapshot runs.

## Lessons Learned

After analyzing these real-world systems, five patterns emerge.

### 1. WAL Is Mandatory

Every production write-back system uses a WAL. Without it, every crash causes data loss. With a WAL, data loss is limited to the WAL sync interval. The WAL is the difference between "acceptable data loss" and "production incident."

### 2. Monitor Dirty Ratio

Every system that has failed in production due to write-back issues had the same pattern: the dirty ratio climbed, nobody noticed, and the cache OOM'd. Monitor dirty ratio at all times. Alert on sustained high values.

### 3. Backpressure Saves Systems

When write volume exceeds flush capacity, backpressure is the only thing that prevents catastrophic failure. Cassandra's `memtable_heap_space`, Redis's `client-output-buffer-limit`, and MySQL's `innodb_max_dirty_pages_pct` are all forms of backpressure. Configure them. Test them during load tests.

### 4. Test Crash Recovery

Most teams test the happy path (write, flush, read) but not the crash path. How long does WAL replay take? What happens if the WAL is corrupted? Does the system start accepting writes before recovery completes? Test crash recovery as part of your chaos engineering practice.

### 5. Batch Size Impacts Latency

Larger batches are more efficient but increase the latency for the oldest entry in the batch. A batch of 10,000 entries where the oldest has been waiting for 5 seconds is a bad user experience for interactive workloads. Use batch age as a flush trigger, not just batch size.
