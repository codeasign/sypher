## The Problem: Write-Through Is Slow

Imagine you are building the likes system for a social media app with 100 million daily active users. Every time someone double-taps a photo, your app needs to record a like. This is a write-heavy workload -- potentially 100,000 likes per second during peak hours.

You start with a write-through cache. Each like goes: App writes to cache, cache immediately writes to database, database confirms, cache confirms, app responds. That is four network hops per write, plus the database write latency. At 100,000 writes per second, your database is drowning. Connection pools exhaust. Write queues grow. Users see "failed to save" errors. The pager goes off at 2 AM.

The fundamental problem is that write-through couples the fast in-memory write path to the slow disk-based database write path. The cache cannot acknowledge the write until the database has committed it. Every write pays the full durability tax, even though most of those writes will be read back from the cache moments later.

## The Insight

The breakthrough realization is that most cache writes do not need to be durable immediately. A like counter going from 42,503 to 42,504 does not need to be on disk before the user sees the heart turn red. The user cares about the instant feedback. The database can catch up a few milliseconds later.

Write-back cache accepts writes into the cache immediately and writes them to the database asynchronously later. The app gets an instant acknowledgment. The cache tracks which entries are "dirty" -- modified in cache but not yet written to the database -- and flushes them in batches.

Think of a restaurant: a write-through waiter runs to the kitchen after every single table's order. One table, one trip. A write-back waiter takes orders from five tables, then walks to the kitchen and hands them all to the chef at once. The diners get their food slightly later, but the waiter serves five times as many tables. More importantly, the chef works in batches, which is more efficient.

## Core Vocabulary

**Write-back cache** A caching strategy where writes go to the cache first, are acknowledged immediately, and are propagated to the database asynchronously. This is the core pattern that enables high write throughput.

**Deferred write** A write that has been accepted by the cache but not yet propagated to the database. The system promises to deliver this write eventually, but not right now. This is what enables the latency improvement -- the app does not wait for the database.

**Dirty data** Data in the cache that has been modified but not yet written to the database. Dirty entries are the cache's pending work. They must be tracked carefully because they represent the only copy of the latest write.

**Flush** The process of writing dirty cache entries to the database. Flushing can be periodic (every N milliseconds), threshold-based (when the dirty buffer reaches M entries), or event-driven (when a read-for-miss occurs on a dirty key).

**Write coalescing** The optimization of combining multiple writes to the same key into a single database write. If a counter is incremented ten times in one second, write coalescing sends one write with the final value instead of ten separate writes. This dramatically reduces database load.

**Write buffer** The data structure that holds dirty entries before they are flushed. This can be an in-memory map, a priority queue, or a ring buffer. The buffer's size, eviction policy, and ordering semantics are critical design decisions.

**Durability window** The period between when the cache acknowledges a write and when that write is committed to the database. During this window, a cache crash causes data loss. The durability window is the fundamental trade-off of write-back caching.

**Write-behind** A synonym for write-back where the asynchronous write is explicitly queued. Some systems distinguish "write-behind" as queue-based and "write-back" as buffer-based, but the terms are often used interchangeably.

**Write-back frequency** How often dirty entries are flushed to the database. Higher frequency reduces the durability window but increases database write load. Lower frequency improves batching efficiency but risks data loss.

**Batch write** A group of dirty entries flushed to the database in a single operation. Batching amortizes the cost of database round-trips across many writes. A batch of 100 writes might take only 1.5x the time of a single write, a 60x efficiency gain.

**Write-back acknowledgment** The cache's response to the app confirming the write is accepted. This is NOT a durability guarantee -- it is a "we got it, we will try to persist it" acknowledgment. Apps must understand this distinction.

**Crash recovery** The process of restoring dirty data after a cache failure. If the cache uses a write-ahead log (WAL), it can replay unflushed writes on restart. Without a WAL, all dirty data is lost.

## How It Actually Works

Let us walk through the complete lifecycle of a write in a write-back cache. This is the detailed step-by-step mechanics.

### Step 1: Application Issues a Write

The application calls `cache.set("user:42:likes", 503)`. This is a synchronous call from the app's perspective, but it will return much faster than a database write.

### Step 2: Cache Stores in Dirty Buffer

The cache receives the write and stores it in the dirty buffer. The dirty buffer is an in-memory data structure -- typically a hash map or a concurrent skip list. The entry is stored with a dirty flag set to `true`.

### Step 3: Immediate Acknowledgment

The cache returns success to the application. This happens in under 1 millisecond for local caches or 1-5ms for distributed caches. The application considers the write done and continues processing.

### Step 4: Dirty Flag Propagation

The cache marks the entry as dirty in its metadata. If the entry already exists in the cache, its value is updated and the dirty flag is set. If the entry is new, it is inserted with the dirty flag. The cache also increments a counter tracking how many dirty entries exist.

### Step 5: Background Flusher Activates

A background thread (or a pool of threads) monitors the dirty buffer. The flusher checks two conditions: has the flush interval elapsed, and has the dirty count exceeded the threshold? If either condition is met, the flusher begins a flush cycle.

### Step 6: Flusher Selects Dirty Entries

The flusher iterates over the dirty buffer and collects entries that need to be flushed. It uses a snapshot or a cursor to avoid locking the entire buffer. The selection strategy can be FIFO (oldest dirty entries first), priority-based (certain keys flush sooner), or size-based (largest batches first).

### Step 7: Write Coalescing

If the same key appears multiple times in the dirty buffer (e.g., the like counter was incremented five times), the flusher coalesces these into a single write. Only the latest value matters. Coalescing can reduce database writes by 50-90% in high-frequency write scenarios.

### Step 8: Batch Assembly

The flusher assembles the selected entries into a batch. A typical batch size is 100-1000 entries. The batch is formatted for the target database -- SQL INSERT statements, Cassandra batches, Redis pipeline commands, or a custom protocol.

### Step 9: Database Write

The flusher sends the batch to the database. This is a single round-trip that writes many entries at once. The database commits the batch. If the database supports batch commits, this is significantly faster than individual writes.

### Step 10: Mark as Clean

After the database confirms the write, the flusher marks the cached entries as clean (dirty flag = false). The data is still in the cache for reads, but now it is consistent with the database. The cache can evict clean entries using its normal eviction policy (LRU, LFU, etc.).

### Step 11: Post-Flush Housekeeping

The flusher updates metrics: number of entries flushed, flush duration, batch size, retry count. If any entries failed to write (database error, network timeout), they go into a retry queue. The dirty count is decremented. The flusher checks if another flush cycle is needed (if new dirty entries arrived during the flush).

### Step 12: Cache Eviction

When the cache needs to evict entries to make room for new data, it must handle dirty entries carefully. A dirty entry cannot be evicted until it is flushed. If the eviction policy picks a dirty entry, the cache has two options: trigger an immediate flush of that entry, or skip it and evict a clean entry instead. Most implementations trigger a flush, which adds latency to the eviction path but guarantees no dirty data is lost.

### Concurrent Writes to the Same Key

What happens when the app writes to the same key twice before the first write is flushed? The dirty buffer already has an entry for that key. The cache updates the value in place. The dirty flag stays true. The flusher sees only one entry for that key and writes only the latest value. This is write coalescing in action -- and it is one of the biggest performance wins of write-back caching.

### Read During Dirty Window

If the app reads a key that has a dirty entry in the cache, the cache returns the dirty value. This gives read-after-write consistency without any additional work. The app always sees the latest write, even if it has not reached the database yet.

## Variants of Write-Back

Different applications need different flush behaviors. Here are the common variants.

### Periodic Flush

The flusher runs on a fixed interval. Every 100ms, it flushes all dirty entries. Simple, predictable, easy to reason about. The downside is that under low write load, you still flush, wasting database resources. Under high write load, dirty entries can accumulate between flushes.

### Priority Queue Flush

Different data has different importance. User-facing data (profile updates, settings changes) should flush quickly. Analytics data (page views, click logs) can tolerate longer delays. A priority queue allows the cache to order flushes by priority.

| Priority | Data Type | Max Flush Delay |
|----------|-----------|----------------|
| Critical | Payment status, auth tokens | 50ms |
| High | User profile, settings | 500ms |
| Medium | Likes, comments | 5s |
| Low | Analytics, logs | 30s |
| Bulk | Background aggregates | 60s |

### Write-Behind with Batch Commit

This variant decouples the flusher from the cache entirely. Writes go to a message queue. A separate consumer reads from the queue and writes to the database. This adds a queue for durability (even if the cache crashes, the writes are in the queue) but adds complexity.

## Mental Model Check

Test your understanding with these questions.

**Question 1: What is the durability window in a write-back cache?**

The period between when the cache acknowledges a write to the application and when that write is committed to the database. During this window, a cache failure causes data loss. The window is determined by the flush interval plus the flush execution time.

**Question 2: How does write coalescing improve performance?**

When the same key is written multiple times before a flush, coalescing combines all writes into a single database operation. Only the latest value matters. This can reduce database writes by 50-90% for high-frequency write patterns like counters.

**Question 3: Can a read return stale data in a write-back cache?**

No -- reads are served from the cache, and the cache has the latest value. The staleness is only between the cache and the database. Reads always see the most recent write. However, if the cache is bypassed and the database is read directly, stale data may be seen.

**Question 4: What happens to dirty entries during cache eviction?**

A dirty entry cannot be evicted until it is flushed. The eviction policy must either skip dirty entries (evicting a clean entry instead) or trigger an immediate flush of the dirty entry before evicting it. Most implementations trigger a flush, which adds latency to the eviction path.

**Question 5: When would you choose periodic flush over threshold flush?**

Periodic flush is simpler and produces predictable database load. Choose it when you want consistent flush latency and can tolerate wasted DB writes during low traffic. Choose threshold flush when you want to adapt to traffic patterns and avoid unnecessary database writes.
