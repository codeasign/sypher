## Internal Architecture

A distributed ID generator looks simple from the outside — call `generate()`, get an ID — but internally it is a precisely engineered state machine with three major subsystems: the clock manager, the sequence allocator, and the bit assembler. Each subsystem must handle edge cases (clock jumps, overflow, concurrent access) without external coordination.

**The Clock Manager** is the most critical subsystem. It wraps the operating system's time-of-day clock, maintaining a monotonic view even if the system clock jumps. On Linux the clock manager can use `clock_gettime(CLOCK_REALTIME)` with a guard against backward movement. On containerized platforms (Kubernetes), the clock manager must also account for paused containers whose clock freezes during suspend.

**The Sequence Allocator** uses lock-free atomic operations to increment the per-millisecond counter. In languages with hardware memory-ordering guarantees (C++ `std::atomic`, Java `AtomicLong`, Go `sync/atomic`), the increment cost is under 50 nanoseconds. When the sequence overflows (exceeds 4095), the allocator enters a spin-wait loop: repeatedly checking the clock until the millisecond advances, then resetting the counter.

**The Bit Assembler** performs a single bitwise OR of the shifted components into a 64-bit word. On modern x86-64 processors this is a single cycle operation. The assembler also handles optional formatting (base-62 encoding for URL-safe short IDs, hex for debugging).

## Data Structures and Algorithms

The core data structure is a **64-bit unsigned integer** partitioned into bit fields. No heap allocation, no pointers, no variable-length encoding — the entire state fits in a machine word.

**Structure:**
- Bit 63 (MSB): always 0 (signed integer compatibility)
- Bits 22-62 (41 bits): millisecond timestamp offset from custom epoch
- Bits 12-21 (10 bits): worker ID (0-1023)
- Bits 0-11 (12 bits): sequence number (0-4095)

**Why this structure:** Fixed-width fields enable bitwise extraction and assembly without branching. The timestamp in the high bits ensures IDs sort chronologically when compared as integers. The sequence in the low bits allows the counter to wrap naturally into the timestamp.

| Operation | Data Structure | Time Complexity | Space Complexity |
|---|---|---|---|
| Generate ID | Bit fields in uint64 | O(1) | 8 bytes |
| Parse timestamp | Bit mask + shift | O(1) | — |
| Parse worker ID | Bit mask + shift | O(1) | — |
| Sort by time | Integer comparison | O(1) per compare | — |
| Bulk generation | Atomic increment | O(1) amortized | 8 bytes per ID |

**Algorithm: Spin-Wait Overflow Handler**

```
function generate():
    current_ms = clock.now()
    if current_ms < last_ms:
        handle_skew(current_ms, last_ms)
    if current_ms == last_ms:
        sequence = increment(sequence)
        if sequence == 0:  // overflowed from 4095
            while current_ms <= last_ms:
                current_ms = clock.now()  // spin-wait
            sequence = 0
    else:
        sequence = 0
        last_ms = current_ms
    return assemble(current_ms, worker_id, sequence)
```

This algorithm guarantees no duplicate IDs within a single worker because the sequence resets to zero on each new millisecond. The spin-wait loop is bounded to under 1 millisecond in the worst case.

### Phase 3: ID Storage and Indexing

When the ID is stored in a database, its bit layout determines how efficiently the B-tree index performs. Snowflake IDs with contiguous timestamps insert into adjacent index pages, minimizing page splits. UUID v4 IDs scatter across the entire key space, causing page splits on nearly every insert.

## Consistency and State

### Crash Scenario: Generator Crashes Mid-Sequence

**What happens:** The generator holds its current timestamp and sequence counter only in memory. On restart, the clock has advanced, so the new timestamp is different from the crash timestamp. The sequence resets to zero. No duplicates, no gaps in the monotonic sequence from the system's perspective — the gap in sequence numbers (e.g., sequence left off at 2047, restart picks up at 0) is irrelevant because the timestamp component differs.

### Partition Scenario: Network Split

In a network partition, workers on each side continue generating IDs independently. There is no requirement for cross-worker consensus, so the partition has zero impact on ID generation. However, if the coordination service (assigning worker IDs) is on the other side of the partition, new workers starting during the partition cannot obtain worker IDs.

### Concurrent Write Scenario

Multiple threads calling `generate()` concurrently on the same worker use atomic operations on the sequence counter. With hardware CAS (Compare-And-Swap), this adds under 50 nanoseconds of contention per call.

### Stale Read Scenario

An ID stored in a database replica might not reflect the latest ID if the replica lags behind the primary. This is not a problem for the ID generator itself — it produces IDs, it does not serve them. Application-level read-your-write consistency is handled by the database layer.

## Performance Characteristics

| Metric | Snowflake (single worker) | UUID v4 | UUID v7 | ULID |
|---|---|---|---|---|
| Generation latency p50 | 0.05ms | 0.01ms | 0.03ms | 0.04ms |
| Generation latency p99 | 0.2ms (CAS contention) | 0.05ms | 0.1ms | 0.15ms |
| Generation latency p99.9 | 0.5ms (clock read) | 0.1ms | 0.2ms | 0.3ms |
| Max throughput (single node) | 4M/sec | 10M/sec | 8M/sec | 6M/sec |
| Storage per ID | 8 bytes | 16 bytes | 16 bytes | 16 bytes |
| B-tree insert cost (vs sequential) | 1x (baseline) | 10x (page splits) | 1.5x | 1.5x |
| Memory for generator state | 24 bytes | 0 bytes | 8 bytes | 8 bytes |

**Key insight:** Snowflake is slower per-generation than UUID v4 (clock read adds overhead), but the total system cost is often lower because sequential IDs reduce database write amplification dramatically. At scale, the 10x reduction in I/O from fewer B-tree page splits more than compensates for the 5x increase in generation latency.

## Configuration and Tuning

| Setting | Safe Default | When to Change | What Breaks |
|---|---|---|---|
| Custom epoch | 2020-01-01 | Already past 69-year window (never happens for reasonable epochs) | IDs wrap if timestamp exceeds 41-bit range |
| Worker ID bits | 10 (1024 workers) | Fewer than 64 workers — reclaim bits for timestamp/sequence | Worker count exceeds allocated bits |
| Sequence bits | 12 (4096/ms) | Throughput per worker exceeds 4K/ms | Sequence overflows, generator spins |
| Clock skew threshold | 10ms | Very high accuracy NTP available (1ms) | False positives on small jitter |
| Spin-wait max | 10000 iterations | Real-time systems with tight latency SLAs | Default spin consumes CPU on full sequence |
| Pre-allocate batch | 1000 | Throughput over 100K IDs/sec per process | Larger batch wastes IDs if process crashes |
| Worker ID source | ZooKeeper | Small scale (under 10 workers) — use env var | Manual config is error-prone at scale |

**Tuning guideline for high throughput:** If a single worker must sustain over 4,000 IDs per millisecond, increase the sequence bit width. Moving from 12 to 14 sequence bits supports 16,384 IDs/ms but reduces the timestamp range by 2 bits (from 69 years to 17 years). For most systems, the 4096 default is sufficient — at 4M IDs/sec, a single worker covers all but the largest deployments.
