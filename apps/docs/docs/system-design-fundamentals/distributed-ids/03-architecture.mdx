## Naive Implementation

The simplest distributed ID system uses a shared database table with an auto-increment column as a central counter. Every application server inserts a row, receives the generated ID, and uses it.

**What breaks at 1,000 concurrent users:** The database connection pool (typically 50-200 connections) becomes the bottleneck. Each ID request holds a connection during the INSERT-and-retrieve round-trip. Connection wait times spike from under 1ms to over 100ms.

**What breaks at 100,000 concurrent users:** The database's InnoDB/engine-level mutex on the auto-increment counter serializes all inserts. Actual throughput caps at roughly 10,000 IDs/second on a single database node, regardless of how many app servers are connected. The auto-increment counter is a single machine, single thread, single point of failure.

**What breaks at 10 million users:** The single database node cannot physically handle the write IOPS. Replication lag between primary and replicas means standby nodes cannot take over quickly during a failover. If the primary goes down, the ID sequence may have gaps or — worse — roll back if the auto-increment counter was not flushed to disk.

## Production-Grade Architecture

A production distributed ID system replaces the shared counter with local generation on each application node, coordinated by a lightweight service that assigns worker IDs. The generation itself is entirely local — no database, no network.

The production architecture has three layers:

**Coordination Layer:** A 3-node etcd or ZooKeeper cluster manages worker ID allocation. Workers register on startup with a TTL lease (default 60 seconds). If a worker stops heartbeating, its ID is released after the lease expires. This prevents ID starvation when workers crash without deregistering.

**ID Generation Layer:** Each ID service instance is a stateless HTTP/gRPC server wrapping the Snowflake or ULID generator. Clients call a local (same-AZ) instance to get IDs. The service exposes two endpoints: `GET /id` returns a single ID, `POST /ids?count=N` returns a batch.

**Consumption Layer:** Application services call the ID service at initialization or on demand. For latency-sensitive paths, the application embeds the ID generator library directly as a sidecar process or shared library, eliminating the network hop.

### Pattern 3: Message Queue ID Assignment

When messages in a queue need unique, orderable IDs (e.g., event sourcing), the producer generates the ID before enqueuing.

**Scenario:** An event sourcing system produces 50,000 events/second. Each event needs a unique, time-sortable ID. The producer generates a Snowflake ID before publishing to Kafka, ensuring consumers can order events by ID even if Kafka partitions reorder messages.

## Multi-Region and Global Scale

Global deployments add geographic distribution. Each region runs its own ID service cluster, and the worker ID includes a region prefix to guarantee cross-region uniqueness.

In a multi-region setup, each region's ID generation is fully independent. The region prefix is typically encoded in the highest bits of the worker ID field (e.g., 2 bits for up to 4 regions, leaving 8 bits for workers within each region). Cross-region data replication (e.g., for a global orders database) uses the ID as a conflict-free unique key.

## Capacity Planning

**Scenario:** You are designing an ID system for a social media platform expecting:
- 50 million daily active users
- Peak writes: 25,000 posts/second
- Average writes: 5,000 posts/second
- 10 database shards
- 3 regions (US, EU, APAC)

**Step 1: Peak throughput per region.** Assuming 50% of traffic in US, 30% in EU, 20% in APAC:
- US peak: 12,500 IDs/sec
- EU peak: 7,500 IDs/sec
- APAC peak: 5,000 IDs/sec

**Step 2: Workers required.** With Snowflake's 4,096 IDs/ms per worker (4 million/sec):
- Each region needs only 1 worker to handle peak traffic.
- For redundancy: 3 workers per region (2 active, 1 standby).

**Step 3: Bit allocation.** With 3 regions and 3 workers per region:
- 2 bits for region (supports up to 4 regions)
- 4 bits for worker ID (supports up to 16 workers per region — plenty of headroom)
- Remaining 57 bits for timestamp (41 bits) and sequence (16 bits)
- Result: 1 sign + 41 timestamp + 2 region + 4 worker + 16 sequence = 64 bits

**Step 4: Capacity headroom.** With 16 sequence bits (65,536 IDs/ms per worker), each region's worker can handle 65 million IDs per second — 5,000x the expected peak. The system has vast headroom even with a single worker per region.

**Step 5: Coordination service sizing.** A 3-node ZooKeeper cluster handles millions of heartbeats per day. With only 9 workers total (3 per region), the coordination load is negligible. The cluster should be sized for the total number of worker instances across all regions: 9-15 for this scenario.

**Storage cost estimate:** At 5,000 IDs/second average, the system generates 432 million IDs per day. At 8 bytes per ID, that is 3.5 GB/day of raw ID storage. Index overhead (B-tree internal nodes) adds approximately 0.5 GB/day. Total storage cost at $0.10/GB/month: $1.20/day for the ID column and its index — negligible compared to the row data.

## Deployment and Operations

**Zero-downtime deployment:** ID services are stateless (all state is the worker ID, which is obtained from the coordination service on startup). A rolling deploy starts new pods, which register for new worker IDs, old pods deregister when they shut down. During the transition window, both old and new pods generate IDs on different worker IDs — no conflicts. No blue-green deployment needed.

**Monitoring:** Three metrics are critical for production ID systems:

1. **Generation latency (p50, p99, p99.9):** Alert if p99 exceeds 2ms. Could indicate CAS contention in the sequence allocator or clock read latency from a noisy neighbor on shared hardware.

2. **Clock skew events:** Alert on any detected skew over 5ms. Frequent skew events indicate an NTP configuration problem or a container runtime issue (e.g., paused VM, throttled CPU).

3. **Sequence overflow rate:** If the sequence counter overflows (hits 4095) more than 1% of the time, the worker is operating at the edge of its capacity. Increase sequence bits or add more workers.

**Debugging an ID collision:** Despite the theoretical guarantees, collisions can happen if two workers are accidentally assigned the same worker ID. The fix is never at the application level — it is always a coordination service misconfiguration. The debug procedure is:

1. Query the colliding IDs and extract the worker ID bits.
2. Check the coordination service's worker assignment log for that worker ID.
3. Investigate whether a stale lease caused the duplicate assignment.
4. Fix: shorter lease TTL, stricter deregistration on shutdown, or persistent storage of assigned IDs.

**Upgrading the ID format:** Changing the ID scheme (e.g., from UUID v4 to Snowflake) requires migrating existing records. The safest approach is a dual-write strategy: new records get the new ID format, old records are backfilled in batches. A mapping table bridges the two ID spaces during the migration window. Expect the migration to take 3-6 months for a system with over 100 million records.
