## Tier 1: Foundation

These challenges test your grasp of the basic concepts and terminology. You should be able to answer each in 3-5 minutes.

### Challenge 1.1: Clock Comparison

A distributed key-value store uses vector clocks with 3 nodes (A, B, C). You read a key and receive the following response:

```
value: "hello"
clock: [A=3, B=2, C=1]
```

One second later, you read the same key again and receive:

```
value: "hello"
clock: [A=3, B=2, C=1]
```

**Question:** Did any node modify the key between the two reads? What is the relationship between the two clocks?

<details>
<summary>Model Answer</summary>

No modification occurred between the two reads. The vector clocks are identical: [A=3, B=2, C=1] in both responses. Identical clocks mean identical causal history. The same write that produced the first response also produced the second response. The relationship is "equal" — both clocks represent exactly the same set of events.

The one-second delay between reads does not change anything. Vector clocks only change when a write operation increments a counter. A read is a read-only operation and does not modify the clock.

</details>

### Challenge 1.2: The Missing Edit

A collaborative document editor uses vector clocks to track changes. Alice opens a document and sees:

```
version A — clock: [Alice=1, Bob=0]
content: "Hello world"
```

Bob, working offline, edits the document and later comes back online. Bob's local version is:

```
version B — clock: [Alice=0, Bob=1]
content: "Hello world — edited by Bob"
```

**Question:** When Bob comes online and syncs, what does the system detect? What should the merged document look like?

<details>
<summary>Model Answer</summary>

The system detects that version A [Alice=1, Bob=0] and version B [Alice=0, Bob=1] are concurrent. Neither clock dominates the other: Alice's clock has Alice=1 vs Bob's Alice=0, and Bob's clock has Bob=1 vs Alice's Bob=0. Both nodes made changes without knowing about each other.

The merged document should contain both edits:
- Alice's addition: "Hello world"
- Bob's edit: "Hello world — edited by Bob"

A reasonable merge combines both changes: the document becomes "Hello world — edited by Bob" which includes Alice's original text and Bob's edit. If the application-level merge function detects that Bob's edit fully incorporates Alice's text, no further action is needed.

If the edits were conflicting (e.g., Alice deleted a paragraph and Bob modified it), the system would store both versions and request manual resolution.

</details>

### Challenge 1.3: Clock Size Calculation

A system has 50 nodes. Each key has a vector clock. A particular key has been modified by 8 of the 50 nodes.

**Question:** How many entries does the vector clock for this key have before and after pruning?

<details>
<summary>Model Answer</summary>

Before pruning: 50 entries. The clock has one entry for every known node, even nodes that have never modified this key. All 8 nodes that actually modified the key have non-zero counters. The remaining 42 nodes have zero counters.

After pruning zero-count entries: 8 entries. Pruning removes entries where the counter is zero. Only the 8 nodes that modified the key remain in the clock.

Storage comparison:
- Before pruning: 50 entries x 16 bytes = 800 bytes per key
- After pruning: 8 entries x 16 bytes = 128 bytes per key
- Savings: 84% reduction

For a system with 10 million keys, the storage difference is 8 GB (unpruned) versus 1.28 GB (pruned).

</details>

### Challenge 1.4: Happens-Before

Node A sends a message to Node B with clock [A=2, B=1, C=0]. Node B receives the message, processes it, and sends a reply with clock [A=2, B=2, C=0].

**Question:** Does event A (sending the message) happen-before event B (sending the reply)? How do the vector clocks confirm this?

<details>
<summary>Model Answer</summary>

Yes, event A happens-before event B. The causality chain is:

1. Node A performs event E1 (send message), clock becomes [A=2, B=1, C=0]
2. Node A sends message to B with clock [A=2, B=1, C=0]
3. Node B receives the message, merges clock: max([A=1, B=1, C=0], [A=2, B=1, C=0]) = [A=2, B=1, C=0]
4. Node B performs event E2 (process and send reply), increments B: [A=2, B=2, C=0]

Comparing the clocks:
- E1 clock: [A=2, B=1, C=0]
- E2 clock: [A=2, B=2, C=0]

E1 clock is strictly less than E2 clock in component B (1 < 2), and equal in all other components. Therefore E1 happens-before E2. This is the happens-before relation defined by Lamport: a message send happens-before the corresponding receive, and event order on the same node follows program order.

</details>

## Tier 2: Intermediate

These challenges require 5-10 minutes each and test your ability to apply vector clock concepts to realistic scenarios.

### Challenge 2.1: The Partitioned Shopping Cart

A distributed shopping cart service uses vector clocks with 3 replicas (nodes X, Y, Z). A network partition separates node X from nodes Y and Z for 30 seconds. During the partition:

- Client A adds item "shoes" to the cart on node X.
- Client B adds item "hat" to the same cart on node Y.

Both clients previously read the cart with clock [X=1, Y=1, Z=1] and value ["shirt"].

**Question:** What are the vector clocks for the cart on each side after the writes? When the partition heals, what does the system detect? How should the conflict be resolved?

<details>
<summary>Model Answer</summary>

After the writes:
- Node X: clock [X=2, Y=1, Z=1], value ["shirt", "shoes"]
  (X increments its counter from 1 to 2)
- Node Y: clock [X=1, Y=2, Z=1], value ["shirt", "hat"]
  (Y increments its counter from 1 to 2)
- Node Z: clock [X=1, Y=1, Z=1], value ["shirt"] (no writes on Z)

When the partition heals and the nodes reconcile:

Comparing X's clock [X=2, Y=1, Z=1] with Y's clock [X=1, Y=2, Z=1]:
- X=2 > 1 (X has more)
- Y=1 < 2 (Y has less)
- Z=1 = 1 (equal)
- Neither dominates — CONCURRENT. Conflict detected.

Resolution: Since both add-only operations (no items were removed), the correct merge is the union of items: ["shirt", "shoes", "hat"]. The merged clock is [X=2, Y=2, Z=1] (max per component). Node Z receives the full merged state.

If one operation had been a removal (e.g., "remove shirt"), the merge would be more complex: the system would need to know whether the removal happened before or after the add operations. CRDTs with tombstones handle this case by tracking both additions and deletions separately.

</details>

### Challenge 2.2: The Hot Key

A social media platform stores the like count for popular posts. The like count key for a viral post receives 10,000 writes per second. The system uses vector clocks with 3 replicas.

The vector clock for the post reaches [A=100, B=100, C=100] after 30 seconds of the viral event. The application does not prune the clock.

**Question:** What problems does this cause? How would you redesign the system for this workload while preserving causal ordering?

<details>
<summary>Model Answer</summary>

Problems:

1. **Vector clock bloat:** After 30 seconds, the clock has 300 events across 3 nodes. The clock does not grow in entries (3 entries for 3 nodes), but the counter values become large. A 64-bit counter can hold values up to 2^63, so overflow is not a practical concern.

2. **Comparison cost:** Comparing two clocks with 3 entries is still O(3) — trivially cheap. The counter size does not affect comparison cost; only the number of entries matters.

3. **The real problem is that the like count is a counter, not a versioned data item.** The vector clock approach is wrong for this workload. Each like is an independent increment, not a versioned update that needs causality tracking.

Redesign options:

**Option A: Use a CRDT counter.** A CRDT positive-negative counter (PN-Counter) merges by taking the max of each node's component. The counter value is sum of all components. No conflict resolution needed. Each node maintains its own counter component.

**Option B: Use a single integer counter with atomic increment.** If strong consistency is not required, each replica maintains its own counter and the total is the sum. The Dynamo paper describes this approach for shopping cart item counts.

**Option C: Use Lambda Architecture.** The real-time count from the last hour can be approximate. The accurate count is computed batch-wise. Vector clocks are not needed for either.

</details>

### Challenge 2.3: Clock Recovery After Crash

Node A in a 3-node cluster crashes and restores from a 2-hour-old backup. At the time of the crash, Node A's counter for key K was A=15. The backup has A=5 for that key.

**Question:** What happens when Node A processes new writes after recovery? How can this cause incorrect behavior?

<details>
<summary>Model Answer</summary>

After recovery, Node A has clock [A=5, B=10, C=8] from the backup. The current state on other nodes is [A=15, B=10, C=8]. Node A's clock shows A=5, but the actual counter should be A=15.

Now Node A receives a write from a client whose clock is [A=12, B=10, C=8]. Node A compares this with its own clock:
- Node A's clock: [A=5, B=10, C=8]
- Client clock: [A=12, B=10, C=8]
- Comparison: Client clock is a DESCENDANT of Node A's clock. The write is "fresh" and accepted.

But the client's write [A=12, B=10, C=8] is actually based on events that happened BEFORE Node A's crash (A=15 on other nodes). The write is stale — other nodes have moved past A=15 to A=16, A=17, etc. But Node A, with its restored backup showing A=5, incorrectly treats the write as fresh.

The resulting clock becomes [A=13, B=10, C=8] (Node A increments to 13 after applying the client's write). When Node A syncs with Node B, Node B sees [A=15, B=10, C=8] and [A=13, B=10, C=8]. Now A=13 is concurrent with A=15 (no node B or C differences). The system creates a false version that does not actually represent a concurrent update in the real timeline.

**Mitigation:** Maintain a separate persistent counter for Node A's own clock that is never rolled back, even during restore. The backup should include the node's local counter, not just the per-key clocks. Alternatively, use hybrid logical clocks that include physical time, making it easier to detect that the restored clock is stale.

</details>

## Tier 3: Advanced

These challenges require 10-15 minutes and test system design and analysis skills at a senior level.

### Challenge 3.1: Design a Causal Broadcast System

Design a publish-subscribe system that guarantees causal ordering of messages. The system has 5 broker nodes and 100 publisher clients. Messages published by the same publisher must be delivered in order. Messages from different publishers must be delivered in causal order when a causal relationship exists.

**Question:** How would you use vector clocks to implement causal ordering in this system? What are the tradeoffs in your design?

<details>
<summary>Model Answer</summary>

**Design:**

1. Each publisher-client connection maintains a vector clock. The clock tracks the publisher's own message count plus the "received" counts from other publishers.

2. When a publisher sends a message, it attaches its current vector clock to the message. The publisher increments its own counter in the clock.

3. When a broker receives a message, it buffers the message until the broker can deliver it in causal order. The broker checks the message's vector clock against its current per-subscriber clocks.

4. A message is deliverable to a subscriber when the broker has delivered all messages that causally precede this message. The broker checks: for every node in the message's clock, has the subscriber received that node's messages up to the counter in the message's clock?

5. If not all predecessors are delivered, the broker buffers the message and delivers it later. This is causal order delivery.

**Tradeoffs:**

1. **Buffer size:** Buffering messages for causal ordering can consume significant memory, especially if one publisher is slow. A slow publisher causes message accumulation for all subscribers waiting for that publisher's messages.

2. **Clock size:** With 100 publishers, each vector clock has 100 entries (approximately 1.6 KB per message). For high-throughput systems, the overhead is significant.

3. **Delivery latency:** Causal ordering increases latency because messages must wait for their predecessors to arrive. In a system with 100 publishers, a burst from one publisher can delay messages from other publishers.

4. **Scalability:** 5 brokers is manageable, but scaling to hundreds of brokers with vector clocks would require pruning or hierarchical clocks.

5. **Fault tolerance:** If a publisher disconnects and reconnects, its vector clock must be preserved. Otherwise, the system loses causal ordering for that publisher's messages.

</details>

### Challenge 3.2: The Billion-Key Migration

A company runs a Dynamo-style database with 1 billion keys. The database uses version vectors with 3 replicas per key. The company plans to migrate from version vectors to hybrid logical clocks (HLC) for causal tracking.

**Question:** Design the migration strategy. How do you ensure that no data is lost, no false conflicts are created, and the system remains available throughout the migration?

<details>
<summary>Model Answer</summary>

**Migration strategy: Phased rollout**

Phase 1: Prepare (offline)
- Add HLC storage alongside existing version vector storage in the metadata format.
- Use a feature flag to control which clock system is used for comparison.
- Validate that HLC produces the same comparison results as version vectors for a test corpus.
- Implement an HLC clock that is a drop-in replacement for version vectors in the comparison logic.

Phase 2: Dual-write (online)
- Write operations update both the version vector and the HLC.
- Read comparisons use version vectors (the proven system).
- Monitor the HLC values to ensure they are correctly tracking events.
- Run for 1 week to build confidence.

Phase 3: Gradual read migration (online)
- 5% of reads use HLC for comparison, 95% use version vectors.
- Monitor for any differences in comparison results.
- Increase to 25%, 50%, 75%, 100% over 2 weeks.
- At each stage, if comparison mismatches are detected, roll back to version vectors.

Phase 4: Remove version vectors (online)
- After all reads use HLC for 1 week with zero mismatches, stop updating version vectors.
- Run a background compaction that removes version vector metadata from stored keys.
- Compaction runs at a rate that does not impact production throughput (e.g., 100,000 keys per second).
- Total compaction time for 1 billion keys: 10,000 seconds (approximately 2.8 hours).

Phase 5: Cleanup
- After compaction completes, remove version vector code from the storage engine.
- Update monitoring, alerting, and operational runbooks.

**Key considerations:**
- Rollback plan: if HLC comparison produces incorrect results, the feature flag reverts to version vectors instantly. All keys still have version vector data because dual-write was running.
- Data loss prevention: at no point is the version vector data deleted before the HLC is proven correct. The migration is additive-then-deletive, never subtractive-then-additive.
- Performance impact: dual-write doubles clock metadata update cost. Benchmark to ensure the extra 16 bytes per write does not impact throughput.

</details>

### Challenge 3.3: Multi-Data Center Causal Consistency

Design a system that provides causal consistency across 5 data centers, each with 20 nodes (100 total nodes). The system must support 1 million keys and 100,000 writes per second. A write in one data center must be visible in other data centers within 2 seconds.

**Question:** How do you implement cross-data-center causal consistency with vector clocks? Handle clock size, latency, and conflict resolution.

<details>
<summary>Model Answer</summary>

**Design:**

Use a two-level clock hierarchy:
- Level 1: Per-data-center clocks (5 entries, one per data center)
- Level 2: Per-node clocks within each data center (20 entries per data center)

A cross-data-center write comparison uses only Level 1 clocks (5 entries, 80 bytes). Within a data center, node-level comparisons use Level 2 clocks (20 entries, 320 bytes).

**Write flow:**

1. Client writes to nearest data center's coordinator node.
2. Coordinator increments the data center's Level 1 counter and the node's Level 2 counter.
3. Coordinator acknowledges the write to the client.
4. Asynchronously, the write propagates to other data centers.

**Cross-data-center replication:**

1. Each data center's replication agent aggregates writes into batches (100-1,000 writes per batch).
2. Each batch carries the sending data center's Level 1 clock.
3. Receiving data center merges the Level 1 clock: max per component.
4. If the received Level 1 clock is concurrent with the local Level 1 clock, per-key Level 2 clocks are compared to detect specific conflicts.

**Performance:**

- Clock comparison for cross-data-center writes: O(5) — 5 data center entries, trivial cost.
- Clock comparison for intra-data-center writes: O(20) — 20 node entries, still trivial.
- Storage per key: 5 Level 1 entries + 20 Level 2 entries = 25 entries x 16 bytes = 400 bytes.
- Cross-data-center latency: 100-300ms depending on geographic distance. The 2-second requirement is easily met.

**Conflict resolution:**

For concurrent cross-data-center writes to the same key, use CRDTs for supported types (counters, sets) and application-level resolution for complex types (documents). The two-level clock hierarchy ensures that conflicts are detected at the data center level first, and per-node resolution is only triggered when needed.

</details>
