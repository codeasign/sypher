## How Companies Use It

### Company Case Studies

**Case Study 1: Netflix — Open Connect CDN + Zuul API Gateway**

Netflix operates one of the largest CDNs in the world, serving 220 million subscribers across 190+ countries. In 2023, Netflix accounted for approximately 15% of global downstream internet traffic during peak hours.

The problem: Netflix's streaming architecture in 2012 was a monolithic application that could not scale to global demand. API calls from 60 million devices (Smart TVs, phones, game consoles) hit a single API endpoint. Each API call triggered cascading requests across dozens of internal services. A failure in one internal service (e.g., recommendations) could block the entire streaming page load. The team needed a way to route, throttle, and aggregate API requests at an unprecedented scale while allowing each device type to receive a customized response.

The solution: Netflix built **Zuul**, a reverse-proxy-based API gateway running on top of Nginx and the JVM. Zuul handles:
- Routing: `/api/ios/*` routes to iOS-specific handlers, `/api/tv/*` routes to smart TV handlers
- Rate limiting: per-device-type rate limits (TV devices get higher limits than browsers)
- Request aggregation: a single `/metadata` call fans out to 15 internal services and merges responses
- Circuit breaking: when a downstream service fails (5xx errors exceeding threshold), Zuul opens the circuit and returns a fallback response for 30 seconds, preventing cascading failures
- Request debugging: each request gets a unique trace ID, visible across all microservices

The Open Connect CDN sits in front of the entire stack. Netflix partners with 1,000+ ISPs to deploy Open Connect appliances inside ISP networks. These appliances are reverse proxies optimized for video streaming: they cache popular titles at the ISP edge and serve 95%+ of video traffic from local cache, eliminating 95% of Netflix's transit bandwidth costs.

**Case Study 2: Cloudflare — Global Anycast Reverse Proxy**

Cloudflare operates one of the world's largest reverse proxy networks, handling 20%+ of global web traffic across 310+ data centers in 120+ countries. Any website that uses Cloudflare sends its traffic through Cloudflare's reverse proxy.

The problem: Small-to-medium websites faced the same challenges as Netflix but could not build their own global reverse proxy infrastructure. They needed DDoS protection, TLS termination, caching, and CDN distribution without hiring a team of infrastructure engineers.

The solution: Cloudflare built a global anycast network where each data center announces the same IP address. Every request goes to the nearest Cloudflare data center (typically under 20 ms away). Cloudflare's reverse proxy (a custom HTTP proxy built on Go and LuaJIT called **Cloudflare Proxy**) terminates TLS, applies firewall rules, checks cache, and forwards to the origin.

Key reverse proxy features:
- **DDoS mitigation**: each proxy absorbs attack traffic at the edge. A 2 Tbps DDoS attack is distributed across 310+ data centers, each handling 6 Gbps (trivial for a 100 Gbps-capable data center).
- **Always-on TLS**: Cloudflare terminates TLS for 15+ million domains, presenting the appropriate certificate per domain from a cache of tens of thousands of certificates.
- **Argo Smart Routing**: Cloudflare discovers the fastest path between edge and origin using real-time network measurements, reducing latency by 30% on average.
- **Cache everything**: with Enterprise plans, Cloudflare caches HTML pages (not just static assets). Cache hit rates of 70%+ mean the origin receives only 30% of direct traffic.

**Case Study 3: Twitter/X — Nginx and Twemproxy for Scale**

Twitter (now X) was one of the early large-scale adopters of Nginx as a reverse proxy. In 2011, Twitter served 250 million tweets per day with a Rails-based monolithic application that could not handle the growth. The team introduced a reverse proxy layer between clients and their application servers.

The problem: Twitter's API was a single Rails monolith. As mobile clients proliferated, the API received 10x the traffic of the web frontend. Each API request triggered multiple database queries and cache lookups. The monolith's concurrency was limited by Ruby's GIL — it could handle only 200-300 concurrent requests per process. Scaling required running hundreds of processes across dozens of servers, but the connection management overhead was crippling.

The solution: Twitter placed Nginx as a reverse proxy in front of their Rails application servers. Nginx handled:
- TLS termination (offloading RSA handshake from Ruby processes)
- Connection pooling (thousands of client connections multiplexed over a small pool of backend connections)
- Load balancing across the Rails server pool (least-connections to handle variable request processing times)
- Rate limiting per API key (to prevent abusive clients from starving fair users)

Twitter also built **Twemproxy** (Twitter Proxy, pronounced "tim-proxy"), a lightweight reverse proxy for Redis and Memcached. Twemproxy acts as a proxy between application servers and cache servers, providing:
- Consistent hashing across cache shards (without each app server needing its own hashing logic)
- Connection pooling to reduce the number of connections each app server opens to the cache cluster
- Automatic failover (remove unhealthy cache nodes from the pool)
- Protocol pipelining (combine multiple small cache operations into fewer TCP packets)

Results: Nginx's connection pooling reduced the number of backend Rails processes needed by 5x (from 600 to 120). Twemproxy reduced Redis connection overhead by 95% (each app server used to open 50 connections per Redis node; with Twemproxy, each app server opened 1 connection to the proxy). These changes allowed Twitter to serve 10x more traffic without adding server capacity.

**Case Study 4: GitHub — HAProxy for Zero-Downtime Deployments**

GitHub serves 100 million repositories and 50 million developers. In 2012, GitHub's monolithic Rails application (Unicorn) could not handle the growth. GitHub adopted HAProxy as their reverse proxy for zero-downtime deployments and request routing.

The problem: Deploying a new version of GitHub.com required taking the site offline for 30-60 seconds while Unicorn processes restarted. For a site with millions of daily active developers, 60 seconds of downtime was unacceptable. The team needed a way to route traffic to new processes while old processes drained gracefully.

The solution: GitHub used HAProxy as a TCP-level reverse proxy with a health check that controlled blue-green deployments:

1. HAProxy listens on port 443 (HTTPS) and forwards to a pool of Unicorn workers on a dynamic port range.
2. During deployment, new Unicorn workers start with new code on new ports.
3. HAProxy health check detects the new workers (they respond to HTTP health probes).
4. HAProxy gradually shifts traffic from old workers to new workers.
5. Old workers continue serving existing connections but stop accepting new ones.
6. When the last old connection drains, the old worker shuts down.

This pattern (now called "rolling restart with connection draining") became the standard for zero-downtime deployments. HAProxy's ability to manage connection draining at the TCP level was the key enabler.

GitHub also used HAProxy for their Redis cluster. Each Redis read replica had a HAProxy instance in front. When a replica failed, HAProxy removed it from rotation without any application code changes. The developer experience: database failures became invisible to developers.

Key numbers from GitHub's architecture at scale:
- 50 million requests per minute during peak hours
- 100+ HAProxy instances across their fleet
- Zero-downtime deployments (sub-second traffic shift)
- Redis failover handled in under 10 seconds (HAProxy detection + pool update)

### Open Source Implementations

**Nginx** — The dominant reverse proxy on the web, powering 30%+ of all websites. Nginx's architecture (event-driven, multi-process, shared memory) was covered in the deep dive. Key strength: flexible configuration DSL that handles 80% of reverse proxy use cases without custom code. Weakness: limited built-in service discovery; requires third-party tools (Consul, etcd, or custom Lua scripts) for dynamic upstream management.

**HAProxy** — The gold standard for TCP-level reverse proxying and load balancing. HAProxy excels at connection management, health checking, and failover. It handles 100,000+ concurrent connections per instance on modest hardware. Key strength: the most mature and battle-tested health checking and failover system in open source. Weakness: limited HTTP content inspection compared to Nginx; no built-in cache.

**Envoy** — A modern C++ reverse proxy designed for service mesh architectures (Istio, Consul Connect). Envoy emphasizes dynamic configuration via xDS APIs (discovery service APIs for listeners, clusters, endpoints, secrets, and routes). Key strength: first-class support for gRPC, HTTP/2, HTTP/3, and advanced observability (distributed tracing with OpenTelemetry, access logs with JSON format). Weakness: significantly more complex to configure than Nginx or HAProxy; the dynamic configuration model requires an xDS control plane.

**Caddy** — A Go-based reverse proxy focused on ease of use and automatic HTTPS. Caddy obtains and renews Let's Encrypt certificates automatically with zero configuration. Key strength: the simplest setup of any reverse proxy — `caddy reverse-proxy --from example.com --to localhost:8080` handles TLS, HTTP/2, and compression. Weakness: smaller ecosystem, fewer third-party modules, and lower raw throughput than Nginx (approximately 70% of Nginx's requests/second at high concurrency).

**Traefik** — A cloud-native reverse proxy with built-in service discovery for Docker, Kubernetes, Consul, and etcd. Traefik automatically detects new containers or pods and adds them to the routing table. Key strength: cloud-native design with automatic Let's Encrypt, dashboard UI, and metrics export. Weakness: higher resource usage than Nginx or HAProxy (JVM overhead of 100-200 MB baseline).

### Cloud Provider Implementations

**AWS** — Three main managed options for reverse proxy functionality:

- **Application Load Balancer (ALB)**: Layer 7 load balancer that handles TLS termination, path-based routing, host-based routing, and WebSocket support. ALB scales automatically up to 100,000 requests/second. Cost: ~$22/month per ALB + $0.008 per GB processed. Best for: simple routing with managed scaling. Does NOT provide caching or advanced request manipulation.

- **API Gateway**: Fully managed API gateway with rate limiting, API key management, request transformation, and caching. Handles 10,000 requests/second by default, scalable to 100,000+. Cost: $3.50 per million requests + cache costs. Best for: serverless APIs, per-customer rate limiting, and request/response transformation.

- **CloudFront**: CDN with 450+ edge locations. Provides TLS termination, caching, origin shielding (a reverse proxy at the edge), and Lambda@Edge for custom logic. Cost: $0.085/GB for data transfer out. Best for: global content delivery with edge caching and low latency.

**GCP** — Two primary managed options:

- **HTTP(S) Load Balancer**: Global Layer 7 load balancer with anycast IP, TLS termination, URL-based routing, and Cloud CDN integration. Handles 100,000+ requests/second. Cost: $0.025 per hour + $0.008 per GB. Best for: global routing with integrated CDN.

- **Cloud Armor**: WAF that integrates with the load balancer, providing rate limiting, IP blocking, and OWASP rule sets. Cost: $3-5 per policy per month. Best for: adding DDoS protection and rate limiting without a separate proxy layer.

**Azure** — Two primary managed options:

- **Application Gateway**: Layer 7 load balancer with TLS termination, path-based routing, cookie-based session affinity, and Web Application Firewall (WAF). Can scale up to 200 instances. Cost: ~$25/month + $0.008/GB. Best for: regional HTTP load balancing with WAF.

- **Azure Front Door**: Global Layer 7 reverse proxy with anycast, CDN caching, WAF, and URL-based routing across regions. Handles 100,000+ requests/second with 400+ edge locations. Cost: ~$35/month + $0.01/GB. Best for: global multi-region HTTP load balancing with CDN caching.

**Cloud vs. Self-Managed Decision Guide:**

Choose managed when: your traffic is steady or predictable, you want to minimize operational overhead, you need global anycast routing (hard to set up yourself), or you need compliance certifications (SOC2, PCI).

Choose self-managed when: you have extreme scale requirements that exceed cloud LB limits (300,000+ req/s on a single ALB), you need custom proxy logic (Lua scripts, complex header manipulation), you run multi-cloud and want a consistent proxy layer, or cloud costs exceed self-managed costs by 2x+.

At 50,000 requests/second, AWS ALB costs approximately $1,200/month. Self-managed Nginx on 3 c5.large instances costs approximately $200/month. The 6x cost difference can justify self-management at scale.

### Lessons Learned

**Lesson 1: Rate Limit Every Client, Always.** Every company case study includes a story about a single abusive client taking down the service. Cloudflare sees it daily (DDoS attacks). Twitter saw it with API abuse (a single developer's polling script doing 100 requests/second to the timeline endpoint). Netflix sees it with misbehaving Smart TV apps (a bug that sent infinite retry loops). Configure per-IP rate limits even if you trust your clients. The limit should be generous (100-500 req/s) but non-negotiable. The cost of a rate limit is zero in the steady state; the cost of a missing rate limit is a production incident.

**Lesson 2: Cache Is a Liability, Not Just an Asset.** Every company that introduced caching had to later introduce cache invalidation, and every cache invalidation system eventually introduces a bug. Netflix uses explicit purge APIs with authorization (only the content team can purge cache). Twitter had a notorious bug where a global cache purge (triggered by a deployment script) took down the entire site for 15 minutes because it triggered 10x the normal database load. Cloudflare handles 100,000+ purge requests per day. Design your caching with the assumption that invalidation will be needed and that stale cache is a data quality issue, not just a performance optimization.

**Lesson 3: Connection Draining Is Critical for Zero-Downtime Operations.** GitHub's HAProxy setup was built around connection draining. Netflix's Zuul has a 30-second drain window before shutting down a proxy instance. Cloudflare's edge proxies use health check removal before process termination. Every company that does zero-downtime deployments has a "drain first, then kill" pattern. Failure to drain causes connection resets and user errors. Implement this before you need it: configure your proxy to stop accepting new connections (remove from health check), wait for existing connections to complete (60 seconds default), then terminate.

**Lesson 4: Shared Cache Doubles the Complexity.** Moving from per-worker in-memory cache to a shared Redis cache adds network latency (0.5-2 ms per lookup), serialization overhead, cache invalidation complexity (all workers must coordinate), and a new failure domain (Redis crash can increase backend load 10x). Netflix, Twitter, and GitHub all moved from shared cache back to simpler per-instance cache for latency-sensitive paths after hitting shared-cache bottlenecks. The rule: start with per-instance cache, move to shared cache only when cache hit rates drop below 50% due to multi-instance fragmentation. At 10+ proxy instances, the fragmentation problem becomes severe enough to warrant shared cache.

**Lesson 5: Monitor the Proxy's Proxy.** Your reverse proxy has a health check. But what monitors the monitor? Cloudflare's system has monitoring agents that run outside the proxy network and periodically verify that each edge proxy is reachable and responding correctly. Netflix runs synthetic traffic generators (Chaos Monkey's cousin, "Traffic Monkey") that send fake user requests through the entire proxy stack and verify end-to-end correctness. GitHub's ChatOps system (Hubot) received aggregated proxy metrics every 5 minutes and posted to a Slack channel that every engineer monitored. If your only proxy monitoring is the cloud provider's load balancer health check, you will miss the failures that happen between the health check path and the real traffic path.
