## Build the Mental Model

### The Problem Story

Your startup launches a mobile game. On day one, 500 players join. Your single application server handles every request directly — player authentication, leaderboard queries, match history. Response times sit at 20 milliseconds. Everything feels fast. By day seven, a viral TikTok post drives 50,000 concurrent players. Your servers are receiving 12,000 requests per second. Each request carries the IP address of a real player device, and your application code must handle TLS termination, rate limiting, request routing, authentication, and compression — all in the same process.

At 8:14 PM on day eight, the servers go down. You check the logs: a single player in Brazil sent 400 rapid-fire requests to the leaderboard endpoint, each one triggering an expensive SQL aggregation. That one player consumed 60% of your database connections, starving every other player. Your application server crashed when memory hit 4 GB trying to compress 800 MB of leaderboard data per second. Your origin IP addresses are exposed to the internet, so attackers can target your servers directly.

You are rebuilding the entire infrastructure that night with a single architectural decision: insert a layer between your users and your servers. This layer terminates TLS, distributes incoming traffic across your backend pool, enforces rate limits per client, caches repeated requests, compresses responses, and hides your server topology from the outside world. This layer runs on dedicated hardware, optimized for these specific tasks, freeing your application code to focus on business logic.

### The Insight

A reverse proxy is a server that sits in front of one or more backend servers, intercepting client requests, processing them (rate-limiting, caching, TLS termination, compression), and forwarding them to the appropriate backend — then relaying the response back to the client. The client never communicates directly with the backend.

Think of a post office's central sorting facility. You drop a letter at your local mailbox (the client request). The postal service does not send every letter directly to its destination address. Instead, letters arrive at the central facility (the reverse proxy), where workers sort by zip code (routing), check postage (authentication), bundle bulk mail together (compression and caching), and forward batches to regional distribution centers (backend servers). The recipient never sees the origin post office — they receive mail from the local carrier. The sorting facility shields individual post offices from chaos, balances workload across routes, and rejects letters with insufficient postage before they enter the delivery pipeline.

Six properties emerge from this arrangement that are not present in direct client-to-server communication:

**Abstraction** — Clients see only the proxy address, not the backend topology. You can swap, add, or remove backend servers without client awareness.

**Amplification** — One proxy connection serves many clients through connection pooling, keep-alive, and response caching.

**Protection** — The proxy absorbs attacks (DDoS, slow loris, malformed requests) before they reach application logic.

**Optimization** — The proxy handles expensive operations (TLS, compression, request normalization) on dedicated hardware while backends handle business logic.

**Observability** — All traffic passes through a single chokepoint, giving you centralized metrics, logging, and tracing.

**Control** — Rate limits, access controls, request routing, and header manipulation are enforced at one layer, not scattered across every service.

### Core Vocabulary

**Reverse Proxy** — A gateway server that accepts client requests, forwards them to backend servers, and returns the backend's response to the client. Unlike a forward proxy (which acts on behalf of clients), a reverse proxy acts on behalf of servers. This means the reverse proxy controls what clients can access, while a forward proxy controls what servers clients can reach.

**Origin Server / Backend** — The application server that generates responses. The reverse proxy shields it from direct client contact. Origin servers can be web servers (Nginx, Apache), application servers (Gunicorn, uWSGI), or API services. In a microservice architecture, you may have dozens or hundreds of origin servers behind a single reverse proxy.

**Upstream** — The set of backend servers that the reverse proxy can forward requests to. Each upstream is defined by a name and a list of servers with ports and weights. Nginx uses the `upstream` block; HAProxy uses `backend` stanzas.

**Load Balancing** — The algorithm the proxy uses to select which backend server receives each request. Common strategies include round-robin (next server in sequence), least connections (server with fewest active connections), IP hash (consistent server per client), and random with weighting. The choice affects cache hit rates, connection distribution, and failover behavior.

**TLS Termination** — The proxy decrypts HTTPS traffic from clients before forwarding plain HTTP to backends. This offloads expensive asymmetric cryptography (RSA 2048 or ECDSA) from application servers, centralizes certificate management to one layer, and lets backends handle requests without TLS overhead. Typical single-core throughput for TLS termination is 10,000 to 40,000 handshakes per second on modern hardware, depending on cipher suite.

**Connection Pooling** — The proxy maintains a reusable pool of connections to each backend, avoiding the cost of opening and closing TCP connections per request. Connection setup costs three round trips (TCP handshake + TLS if enabled). Pooling eliminates this overhead. A typical pool size is 100 to 500 connections per backend, keeping idle connections alive for 60 to 300 seconds via keep-alive timeouts.

**Caching** — The proxy stores response bodies (and optionally headers) for a configurable time-to-live (TTL), serving repeated requests from memory instead of forwarding them to the backend. Cache hit rates for static assets commonly reach 90% to 98%. The proxy determines cacheability based on response headers (`Cache-Control`, `Expires`, `ETag`). Cache size is typically configured as a percentage of total RAM, often 10% to 30%.

**WebSocket Upgrading** — When a client initiates a WebSocket connection (via the `Upgrade: websocket` header), the reverse proxy must detect the upgrade handshake, forward it to the backend, and then switch into tunnel mode — passing raw bytes bidirectionally without buffering or interpreting the payload. Not all proxies handle this well; HAProxy and Nginx (with `proxy_http_version 1.1` and `proxy_set_header Upgrade $http_upgrade`) support it natively.

**Health Checks** — The proxy periodically probes backend servers (via HTTP GET, TCP connect, or custom scripts) to determine if they are alive. Unhealthy servers are removed from the upstream pool until they pass health checks again. Active checks run on a timer (every 1 to 30 seconds), while passive checks detect failures during real requests by monitoring response codes and timeouts.

**X-Forwarded-For** — An HTTP header added by the proxy carrying the original client IP address, because the backend sees only the proxy's IP. Standards-track alternatives include `Forwarded` (RFC 7239) and `X-Real-IP`. Without these headers, backend logs, rate limiters, and geo-location all see the proxy's address instead of the actual client.

### How It Actually Works — Step by Step

**Step 1: Client DNS Resolution**

The client resolves the proxy's hostname (e.g., `api.syphergame.com`) via DNS. The DNS response returns the reverse proxy's IP address, not the backend's. This is the first layer of indirection: clients can never bypass the proxy because they do not know where the backends live.

**Step 2: TCP Connection Establishment**

The client opens a TCP connection to the proxy IP on port 443 (HTTPS). The three-way handshake takes about one round trip (2 to 50 milliseconds depending on network distance). The proxy accepts the connection and may reject it immediately based on IP reputation, rate limits, or connection rate limits without ever involving backend logic.

**Step 3: TLS Handshake**

The client and proxy perform the TLS handshake. The proxy presents its certificate (matching the domain). This is where TLS termination happens: the proxy bears the cost of asymmetric cryptography (RSA or ECDSA key exchange), typically consuming 1 to 5 milliseconds of CPU per handshake. The proxy may use session resumption (session IDs or TLS 1.3 session tickets) to reduce subsequent handshakes to zero round trips.

**Step 4: Request Parsing and Validation**

The proxy parses the HTTP request: method, path, headers, and body. It validates the request against configured rules. Requests exceeding maximum header size (default 8 KB) or body size (default 1 MB) are rejected immediately with a 413 or 400 status code. Malformed HTTP (missing Host header, invalid characters in path, HTTP/1.1 without required fields) is rejected with 400 Bad Request. These checks happen before any backend resources are consumed.

**Step 5: Access Control and Authentication**

The proxy checks the request against access control lists. IP-based allowlists and denylists are checked first (O(1) lookup in a hash set). Then the proxy may validate authentication tokens (JWT, OAuth bearer tokens) by checking signature and expiration. Token validation at the proxy prevents unauthenticated requests from ever reaching backend services. A revoked or expired token is rejected with 401 Unauthorized within microseconds, without any database query.

**Step 6: Request Rewriting and Header Manipulation**

The proxy rewrites the request before forwarding it. This includes adding headers (`X-Forwarded-For` with the original client IP, `X-Forwarded-Proto` with the original protocol), stripping internal headers (internal authentication tokens, internal routing directives), and rewriting URL paths (`/api/v2/users` becomes `/internal/users` for a legacy backend). The proxy also normalizes headers — converting multiple `Cookie` headers into a single header, for example, or decompressing request bodies that were compressed by the client.

**Step 7: Cache Lookup**

The proxy computes a cache key from the request method, URI, query parameters, and relevant headers (typically `Host`, `Accept-Encoding`, and `Authorization`). It looks up the key in its in-memory cache (or a shared cache like Redis). On a cache hit (key exists and is not expired), the proxy serves the cached response directly — status code, headers, and body — without contacting the backend. Cache hit latency is under 1 millisecond. Cache miss latency adds only microseconds to the request path.

**Step 8: Load Balancer Selection**

The proxy selects a backend server from the upstream pool using the configured algorithm:

- Round-robin: iterates through the server list in order, wrapping around. Fair distribution but ignores current load. Best when all servers have identical capacity and request cost is uniform.
- Least connections: selects the server with the fewest active connections. Better for variable-length requests (e.g., streaming endpoints that hold connections for seconds versus quick API calls).
- IP hash: hashes the client IP to select a server. Guarantees the same client always reaches the same backend. Useful for session affinity when backends store local state.
- Random with weight: weighted random selection based on server capacity. Used when servers have heterogeneous specs.

**Step 9: Connection Pool Fetch**

The proxy retrieves an idle connection from its connection pool to the selected backend. If no idle connection exists and the pool has capacity, a new connection is created (TCP handshake only — no TLS, since the proxy-to-backend link is plain HTTP). If the pool is full, the request queues until a connection becomes available, up to the proxy's `proxy_read_timeout` (default 60 seconds).

**Step 10: Backend Request and Response**

The proxy forwards the (possibly rewritten) request to the backend over the pooled connection. The backend processes the request normally and returns a response. The proxy receives the response and performs buffering — reading the entire response body into a buffer (up to `proxy_buffer_size`, default 4 KB to 8 KB for headers, and `proxy_buffers` for the body) before sending it to the client. Buffering allows the proxy to:
- Compress the response with gzip or brotli
- Cache the response for future requests
- Retry on the next backend if this one fails mid-response

**Step 11: Response Compression and Cache Storage**

The proxy compresses the response body (if the client sent `Accept-Encoding: gzip` or `br`). Compression reduces bandwidth by 60% to 90% for text content (HTML, JSON, CSS) but adds 10 to 100 milliseconds of CPU latency. After compression, the proxy stores the response in its cache if the response headers indicate cacheability (`Cache-Control: public`, no `Set-Cookie`, TTL set by `max-age` or `Expires`).

**Step 12: Response Delivery**

The proxy sends the response to the client, using the original TLS-encrypted connection. The response includes headers like `X-Cache: HIT` or `X-Cache: MISS` to indicate whether the response came from cache. The proxy may also add `Age` header indicating how long the cached response has been stored. After sending the response, the proxy returns the backend connection to the pool for reuse.

### Variants and Flavors

**Variant 1: Layer 7 Reverse Proxy (Application-Level)**

This is the most common variant. It operates at the HTTP/HTTPS layer, understanding request methods, headers, paths, cookies, and bodies. It can make routing decisions based on any HTTP attribute: route `/api/*` to backend A, `/static/*` to backend B, and `*.png` to a CDN. It supports header manipulation, cookie-based session affinity, and content-based caching.

Use when you need intelligent routing based on request content, TLS termination, and HTTP-level optimizations like caching and compression. The tradeoff is higher CPU and memory overhead because the proxy must parse and buffer HTTP messages.

Do not use when raw throughput is the only concern and all traffic goes to one backend with no need for HTTP-aware processing. In that case, a Layer 4 proxy is simpler and faster.

**Variant 2: Layer 4 Reverse Proxy (Transport-Level)**

Operates at the TCP/UDP layer. It does not inspect HTTP content — it forwards raw TCP streams or UDP datagrams based on destination IP and port. Layer 4 proxies are simpler and faster, processing packets with minimal overhead. HAProxy in TCP mode and IPVS (Linux Virtual Server) are common implementations.

Use when you need raw throughput with minimal latency — for example, forwarding database connections (postgresql://) or non-HTTP protocols (gRPC without HTTP/2 awareness, Redis, Memcached). Layer 4 proxies handle 500,000 to 1,000,000 connections on a single server with modest hardware.

Do not use when you need HTTP-aware routing, caching, header manipulation, or content-based decisions. Layer 4 proxies cannot inspect payloads, so they cannot reject malicious HTTP requests or cache responses.

**Variant 3: API Gateway**

An API gateway is a specialized Layer 7 reverse proxy for microservice architectures. It adds API-specific features: request aggregation (combining multiple backend API calls into one client response), protocol translation (REST to gRPC, SOAP to REST), response transformation (field filtering, data shaping), authentication and authorization, rate limiting per API key, and billing/usage tracking.

Use when you have multiple microservices that need a unified API surface, and you need per-API-key rate limiting, usage analytics, or request/response transformation. API gateways like Kong, AWS API Gateway, and Apigee sit in this category.

Do not use when you have a single monolithic application — the additional features and complexity are unnecessary. A simple Layer 7 proxy (Nginx, Caddy) is sufficient.

**Variant 4: Load Balancer (Hardware)**

Hardware load balancers like F5 BIG-IP and Citrix ADC are purpose-built appliances with custom ASICs for TLS termination, SSL/TLS offload, and traffic management. They handle 100,000 to 1,000,000+ TLS handshakes per second on dedicated hardware. They include DDoS protection hardware, hardware security modules (HSM) for key storage, and redundant power supplies.

Use when you need carrier-grade reliability (99.999% uptime), hardware-accelerated cryptography, or compliance requirements that mandate dedicated appliances (finance, healthcare, government).

Do not use for small to medium deployments. Hardware load balancers cost $10,000 to $100,000+ per appliance. Software proxies (Nginx, HAProxy, Envoy) on commodity hardware handle 100,000+ requests per second at a fraction of the cost. The hardware advantage narrows as software TLS implementations improve (AES-NI instructions, kernel bypass like DPDK).

### Mental Model Check

**Question 1:** Your backend responds in 250 ms. The reverse proxy adds a 5 ms buffer. A client sends a request that hits the cache (10 ms response). What are the two possible response times, and which one tells you the backend is slow?

**Answer:** Cache miss: 255 ms total (5 ms proxy + 250 ms backend). Cache hit: 15 ms (5 ms proxy + 10 ms cache serve). The cache hit response hides backend slowness. To measure real backend health, ignore cached responses or use a separate health check endpoint that bypasses cache.

**Question 2:** Your reverse proxy shows 50,000 concurrent connections but your four backend servers each report only 200 active connections. How is this possible?

**Answer:** The proxy maintains long-lived connections to clients (keep-alive) while using connection pooling to backends. Each client connection stays open between requests, but the proxy multiplexes them over a much smaller pool of backend connections. With 200 connections per backend and four backends, the pool has 800 connections serving 50,000 persistent client connections. The ratio is 62.5:1, which is normal for a proxy with effective keep-alive and connection reuse.

**Question 3:** A reverse proxy stores responses in cache. You update user profile data on the backend, but the proxy continues serving the old data for 60 seconds. What controls this behavior and how do you handle it?

**Answer:** The `Cache-Control: max-age=60` header from the backend response sets the TTL. To handle real-time updates, either set `max-age` to 0 for dynamic endpoints (but lose caching benefits), use cache invalidation by purging the specific cache key when data changes, validate with `ETag` headers (the proxy sends `If-None-Match` and the backend returns 304 Not Modified if unchanged), or use shorter TTLs (5-15 seconds) for user-specific data while keeping long TTLs for public data (60-3600 seconds).

**Question 4:** You configure health checks every 5 seconds. A backend crashes. What is the worst-case window where the proxy still sends traffic to the dead backend?

**Answer:** The worst case is one health check interval plus the request timeout: 5 seconds (check interval) + 60 seconds (default `proxy_read_timeout`) = 65 seconds. During this window, 5 to 50 requests (depending on rate) fail with 502/504 errors before the backend is marked down. Reducing the health check interval to 1 second shrinks the window but increases probe traffic by 5x. Using passive health checks (detecting failures on real requests) can detect crashes within one request timeout.

**Question 5:** You see that your reverse proxy's CPU is at 95% and throughput is 20,000 requests/second. The proxy is doing TLS termination and gzip compression. Backend CPU is at 20%. Should you scale the proxy, the backends, or both?

**Answer:** Scale the proxy first. The proxy is CPU-bound on TLS and compression, while backends are underutilized. Add a second proxy instance behind a DNS round-robin or network load balancer. If TLS is the bottleneck, consider terminating TLS at a dedicated appliance or using a faster cipher suite (ECDHE-ECDSA-AES128-GCM instead of RSA-2048). If compression is the bottleneck, disable compression for API responses (which are typically small) and compress only large static assets. After scaling the proxy, if request volume increases, the backends may then become the bottleneck. Monitor backend CPU after proxy scaling.
