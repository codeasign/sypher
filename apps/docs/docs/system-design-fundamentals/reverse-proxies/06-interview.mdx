## Ace the Interview

### How This Topic Appears

Reverse proxy and load balancing questions appear in three forms: direct knowledge questions, system design breadth questions where the proxy is one component, and debugging/operations questions where you diagnose proxy-related failures.

**Direct Knowledge Questions (expect 2-3 minutes each):**

1. "Explain the difference between a forward proxy and a reverse proxy. When would you use each?"

2. "How does a reverse proxy handle TLS termination? What are the performance implications?"

3. "What load balancing algorithms are you familiar with? When would you choose least-connections over round-robin?"

4. "How does a reverse proxy detect that a backend server is unhealthy? What are active versus passive health checks?"

5. "What is connection pooling and why is it important in a reverse proxy?"

6. "Explain the `X-Forwarded-For` header. Why is it needed and how can it be spoofed?"

**System Design Questions (expect 15-30 minutes each):**

7. "Design the architecture for a multiplayer game backend that handles 1 million concurrent players. How would you use reverse proxies in this system?"

8. "Your video streaming service is experiencing latency spikes during peak hours. Your backend handles 50,000 requests/second but the proxy layer is at 95% CPU. Walk me through the diagnosis and solution."

9. "Design a rate limiting system for a social media API. Where does the reverse proxy fit?"

10. "Your e-commerce site has a flash sale coming up. Traffic is expected to spike 50x normal. Design the proxy layer to handle this."

**Debugging and Operations Questions (expect 5-10 minutes each):**

11. "Users report that some requests return 502 errors while others succeed. All users are on the same network. What do you investigate?"

12. "After deploying a new backend version, you see a 10% increase in 504 errors. What changed and how do you respond?"

13. "Your reverse proxy cache hit rate dropped from 92% to 40% overnight. What could cause this?"

14. "A client reports that their requests consistently return stale data that is 60 seconds old. Your cache TTL is set to 300 seconds. What is happening?"

### What the Interviewer Is Actually Testing

**For direct knowledge questions (types 1-6):** The interviewer checks whether you understand the concept at the systems level, not just the definition. They want to hear about why a concept matters and what breaks without it. Mentioning concrete numbers (TLS handshake takes 1-5 ms, connection pooling reduces backend connections by 10x, health check interval is 5 seconds) separates you from candidates who only know definitions.

**For system design questions (types 7-10):** The interviewer evaluates your ability to decompose a problem into components and reason about each component's constraints. Reverse proxies are expected infrastructure in any design at scale. If your architecture does not include a reverse proxy layer, the interviewer will ask why. They watch for: do you mention caching? Do you address rate limiting? Do you account for TLS overhead? Do you consider multi-region deployment?

**For debugging questions (types 11-14):** The interviewer wants to see your troubleshooting methodology, not the specific answer. Start broad (is it all traffic or a subset?), then narrow (check proxy logs, check backend health, check configuration). Explain what each diagnostic step reveals and how you use that information to rule out causes. They grade on process, not speed.

### Level-by-Level Expectations

**L3/L4 (Entry-level / Early-career):** Expected to describe the basic function of a reverse proxy. Define forward vs. reverse proxy correctly. Name two load balancing algorithms (round-robin and least-connections). Explain TLS termination at a high level ("the proxy decrypts HTTPS traffic before forwarding to backends"). Acceptable answer for "design a proxy layer" includes: put Nginx in front of servers, configure health checks, set up basic caching. No need for deep configuration knowledge or performance numbers.

Sample answer for "What is a reverse proxy?" at L4 level:

"A reverse proxy is a server that sits between clients and backend servers. It accepts client requests, forwards them to one of several backend servers based on a load balancing algorithm, and returns the response to the client. The client never communicates directly with a backend server. This provides several benefits: TLS termination offloads encryption cost from backends, caching reduces request volume to backends, load balancing distributes traffic across multiple servers, and the proxy hides the internal server topology from the internet. Nginx and HAProxy are common reverse proxy implementations."

**L5 (Senior):** Expected to discuss trade-offs between proxy configurations in detail. Compare Nginx vs. HAProxy vs. Envoy with specific use cases. Explain how connection pooling affects memory and throughput. Calculate approximate throughput for a given configuration. Discuss cache invalidation strategies (TTL, purge, stale-while-revalidate). Describe at least three failure modes and their mitigations. Provide concrete numbers: "At 10,000 requests/second with ECDSA certificates and gzip level 3, a 4-core Nginx instance uses about 70% CPU."

Sample answer for "How would you design a proxy layer for a flash sale?" at L5 level:

"A flash sale with 50x normal traffic requires careful proxy design. First, I would front the proxy layer with a CDN that caches as much as possible at the edge: product images, static pages, and even API responses for product details (as long as we accept up to 10-second staleness). For the dynamic endpoints (add-to-cart, checkout), the reverse proxy layer needs aggressive rate limiting — per-IP and per-session — to prevent bots and scalpers from consuming all capacity. I would configure Nginx with a sliding window rate limiter at 20 requests per second per IP for the checkout endpoint, with a burst of 50. I would increase the proxy instance count by 3x before the sale starts using a scheduled auto-scaling action. I would disable gzip compression on the proxy for the checkout flow (saves CPU, the payloads are small). I would set `proxy_read_timeout` to 10 seconds instead of 60 to fail fast on overloaded backends. I would configure the load balancer to use least-connections instead of round-robin because request processing times vary significantly during a flash sale (checkout takes 500 ms, cart takes 50 ms). Finally, I would prepare a static 'sold out' or 'site at capacity' page that the proxy can serve directly, bypassing backends entirely if the system is overwhelmed."

**L6 (Staff / Principal):** Expected to discuss global scale architectures with region-level decision making. Design a multi-region proxy layer with active-active failover. Compare anycast vs. DNS-based routing with cost and complexity analysis. Explain the CAP implications of cross-region caching. Design a circuit-breaking strategy for proxy-to-backend connections. Discuss observability at scale: what metrics, what dashboard layouts, what alert thresholds. Show understanding of cost optimization: "Why would we choose 6 c5.large instances over 3 c5.xlarge instances?" (Availability: more instances = smaller blast radius.)

Sample answer for "Design a global proxy layer for a social media platform" at L6 level:

"I would use a three-tier proxy architecture. Tier 1 is an anycast network of edge proxies (Cloudflare or custom) that cache static content at 310+ locations. Tier 2 is a regional reverse proxy pool in each AWS region where we have compute: us-east-1, eu-west-1, ap-southeast-1. Tier 3 is a service mesh proxy (Envoy) in front of each microservice. The edge proxies return 302 redirects for dynamic requests to the nearest regional proxy, bypassing DNS caching issues. Each regional pool has 8-12 Nginx instances behind a network load balancer, auto-scaled based on CPU (target 60% utilization). The regional proxies maintain a shared Redis cache with 5 GB of hot data per region, populated via cache-aside. Cross-region cache invalidation uses a Kafka topic: when a write occurs in us-east-1, the write service publishes an invalidation event, and all regions consume the topic and evict the stale key. The stale window is typically 200-500 ms (cross-region Kafka propagation). For reads that require global consistency (profile updates, password changes), the proxy sets `Cache-Control: no-cache` so the request always reaches the primary region. For all other reads, we serve from the local cache with a 30-second TTL, accepting temporary inconsistency."

**L7/L8 (Director / Fellow):** Expected to connect proxy architecture to business outcomes. Discuss cost/performance trade-offs in dollar terms. Explain how proxy decisions affect developer experience (deployment velocity, debugging ease). Compare build-vs-buy for proxy infrastructure. Discuss organizational patterns: "Should each team own their proxy configuration, or should there be a platform team?" Show awareness of industry trends: "How does service mesh (Istio, Linkerd) change the role of the reverse proxy?"

### The FAANG Framework for Answering

When asked a system design question involving reverse proxies, use this 6-step framework:

**Step 1: Scope and Constraints.** Clarify scale, latency, and consistency requirements. "Before I design the proxy layer, let me confirm: this needs to handle 10 million daily active users, with a peak of 50,000 requests per second. Average response time must stay under 200 milliseconds. We accept up to 5 seconds of cache staleness for most content but real-time for financial transactions. Is that correct?"

**Step 2: Baseline Component.** Draw the simplest architecture first: clients -> reverse proxy -> backend servers. Explain what each component does at a high level. This ensures the interviewer agrees on fundamentals before you add complexity.

**Step 3: Scale Analysis.** Calculate what breaks at the stated scale. "At 50,000 requests/second with a 50% TLS handshake rate, each Nginx worker needs approximately 3 CPU cores for TLS alone. With 4 workers on an 8-core machine, TLS consumes 75% of CPU. We need at least 3 proxy instances to stay below 70% CPU utilization. Cache hit rate target is 85%, which reduces backend load from 50,000 to 7,500 requests/second."

**Step 4: Failure Modes.** List three things that could go wrong and how the architecture handles them. "A backend fails: the proxy detects it via health checks within 5 seconds and removes it from rotation. A proxy instance fails: the network load balancer redirects traffic to remaining instances. The cache node fails: the proxy falls back to forwarding all requests to the backend — latency increases but the system stays up."

**Step 5: Refinements.** Add features one at a time, explaining the trade-off of each: caching (speed vs. staleness), rate limiting (protection vs. false positives), compression (bandwidth vs. CPU), connection pooling (concurrent capacity vs. memory), circuit breaking (fault isolation vs. reduced capacity). Never add all features at once without justification.

**Step 6: Monitoring and Operations.** Describe how you know the system is healthy. "Three key metrics: proxy CPU (should be under 70%), cache hit rate (should be above 80%), and upstream p99 latency (should be under 300 ms). Dashboard shows these with 15-second granularity. Alert when any metric deviates by 2x from baseline for 5 minutes."

### Phrase Bank

Use these phrases to demonstrate depth. Each includes the context where it is appropriate.

1. "A reverse proxy is a **gateway that sits between client and server, terminating TLS, caching responses, and distributing load across a pool of backends.** " — Opening definition in any proxy question.

2. "The fundamental trade-off is **freshness versus performance: caching speeds up responses but introduces a staleness window.** " — When discussing caching decisions.

3. "TLS termination at the proxy offloads **asymmetric cryptography from application servers, reducing per-request CPU cost by 50-75% for backends that would otherwise do their own TLS.** " — When explaining why the proxy does TLS.

4. "Connection pooling allows the proxy to **multiplex thousands of client connections over a small pool of backend connections, reducing backend connection establishment overhead by an order of magnitude.** " — When explaining proxy efficiency.

5. "Health checks are the **safety net of the proxy layer. Active checks probe every 5 seconds; passive checks detect failures during real requests. Both are needed for fast detection without false positives.** " — When discussing failure handling.

6. "Stale-while-revalidate is the **pragmatic middle ground: serve the stale cached response immediately while asynchronously fetching a fresh one. The client sees cache speed; the server gets accurate data.** " — When balancing caching trade-offs.

7. "The proxy should **fail fast and fail loud: short timeouts (5 seconds connect, 30 seconds read) so failures are detected quickly rather than silently hogging connections.** " — When discussing timeout configuration.

8. "Cache invalidation is harder than caching. **Every cached response has a ticking clock. Once your cache grows beyond 10 GB, you need a purge strategy, not just TTLs.** " — When the interviewer asks about cache management at scale.

9. "At the network level, the proxy is a **man-in-the-middle by design, and that is the whole point. It terminates one TLS connection and creates another, inspecting and potentially modifying the traffic in between.** " — When asked about security implications.

10. "Round-robin is fair in theory but unfair in practice when requests vary in cost. **Least-connections handles variable workloads better because it sends requests to the server with the most available capacity.** " — When choosing a load balancing algorithm.

11. "A 502 means the proxy could not reach the backend. A 504 means the proxy reached the backend but the backend did not respond in time. **The distinction tells you where to look: network layer versus application layer.** " — When debugging proxy errors.

12. "The proxy is the **observability chokepoint: every request passes through it, making it the ideal place to collect metrics, logs, and traces with minimal instrumentation cost.** " — When discussing monitoring.

13. "Auto-scaling the proxy layer should **lag behind traffic by 1-2 minutes to avoid thrashing. Use a moving average of CPU over 60 seconds, not instantaneous values.** " — When designing auto-scaling.

14. "DNS TTL is the **enemy of fast failover. At 300-second TTL, an unhealthy proxy takes 5 minutes to stop receiving traffic. Keep TTL at 30-60 seconds for production.** " — When discussing DNS-based routing.

15. "The proxy should **never be the bottleneck. If proxy CPU exceeds 70%, add instances. Proxy instances are cheap compared to backend instances — it is always worth over-provisioning the proxy layer.** " — When discussing capacity planning.

### Red Flags

These responses will cost you points or disqualify you:

**Red Flag 1: Saying "a reverse proxy" when you mean "load balancer."** They are not the same. A load balancer distributes traffic. A reverse proxy does that plus TLS termination, caching, compression, rewriting, rate limiting, and observability. Confusing them signals you have not worked with proxies in production.

**Red Flag 2: Forgetting X-Forwarded-For.** If you explain a reverse proxy but do not mention that the backend needs the original client IP, you miss a key operational detail. The interviewer will note the gap.

**Red Flag 3: Claiming caching is free.** Every time you mention caching, immediately mention staleness, invalidation, or TTL management. Caching without staleness awareness is a junior-level mistake.

**Red Flag 4: No concrete numbers.** "The proxy handles many requests" is not an interview answer. Use numbers: "A 4-core Nginx instance handles approximately 10,000 requests/second with TLS + compression. Cache hit latency is under 1 millisecond. Health checks run every 5 seconds."

**Red Flag 5: Ignoring the single point of failure.** If your architecture has one proxy instance, the interviewer will ask what happens when it fails. Preempt this by saying "We would deploy at least 2 proxy instances behind a network load balancer for high availability."

**Red Flag 6: Proposing a proxy when the scale does not require one.** For a design with 100 requests/second and one backend, a reverse proxy adds latency and complexity with minimal benefit. Acknowledging when a proxy is unnecessary shows good judgment.

**Red Flag 7: Treating all backends as identical.** A good answer accounts for heterogeneous backends — different CPU, memory, network capacity. Weighted load balancing exists for this reason. Mention it.

**Red Flag 8: Not knowing the difference between Layer 4 and Layer 7.** Layer 4 proxies (TCP) are faster but cannot inspect HTTP content. Layer 7 proxies (HTTP/HTTPS) can route based on URLs, headers, and bodies. Which one do you choose and why? If you cannot answer, study this before the interview.

**Red Flag 9: Overlooking the security surface of the proxy itself.** The proxy has its own attack surface — configuration injection, header injection, cache poisoning, certificate compromise. Mentioning that you would harden the proxy (run as non-root, restrict access to configuration files, use AppArmor/SELinux) separates a production-aware engineer from a textbook engineer.

**Red Flag 10: Presenting the proxy as a magic solution.** "We will just put Nginx in front, and it will handle everything" is not an architecture. The proxy solves specific problems (TLS, caching, routing) and does not solve others (bad database queries, slow application code, memory leaks in backends). Be specific about what the proxy does and does not solve.

### Practice Drills

**Drill 1: Tier 1 — Definition and Comparison (2 minutes)**

Answer: "What is the difference between a reverse proxy, a load balancer, and an API gateway?" Time yourself. Your answer should cover: what each is, where they overlap, where they differ, and when you would use each. Target: 3 specific distinctions in under 2 minutes.

**Drill 2: Tier 2 — Debugging Scenario (5 minutes)**

Scenario: Your production reverse proxy returns 502 errors for 10% of requests. The errors correlate with peak hours but do not affect all endpoints equally — the `/api/users` endpoint has 15% errors while `/api/health` has 0%. Walk through your debugging process aloud. Generate 3 hypotheses, explain how you would test each, and describe the most likely root cause given the pattern. (Hint: the answer involves partial backend pool failure.)

**Drill 3: Tier 3 — Design a Caching Strategy (5 minutes)**

Design the caching strategy for a news website with 5 million daily visitors. Content types: breaking news (seconds old), regular articles (hours old), user comments (minutes old), and static assets (months old). Define TTLs per content type, cache invalidation strategy for breaking news, and how you handle the cache stampede problem when breaking news hits. Provide reasoning for each decision.

**Drill 4: Tier 4 — Multi-Region Proxy Design (10 minutes)**

Design a reverse proxy layer for a global ride-sharing app (Uber, Lyft). Requirements: 50 million users, 15 million rides per day, real-time driver location updates (must be under 100ms), rider payment processing (strong consistency), and promotional banners (cacheable, stale by 1 minute acceptable). Your design must cover: regional proxy topology, cache strategy per data type, cross-region invalidation mechanism, and failover between regions. Present as a diagram with numbered components.

**Drill 5: Tier 5 — Cost Optimization at Scale (10 minutes)**

Your infrastructure uses 30 Nginx proxy instances across 3 regions, costing $6,000/month. Traffic has grown 40% year-over-year for 3 years. Cloud provider costs are the second-largest line item after engineering salary. Optimize this proxy layer for cost without reducing reliability or capacity. Consider: instance sizing (c5.large vs. c5.xlarge), reserved instances vs. on-demand, spot instances for non-critical traffic, moving from Nginx to a cheaper alternative (Caddy, Traefik?), reducing instance count with larger instances, CDN offload, and compression savings. Present a before/after cost model with at least 30% savings shown in dollar terms.
