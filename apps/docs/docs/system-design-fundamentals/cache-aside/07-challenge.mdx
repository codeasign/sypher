## Tier 1: Foundation Challenges

### Challenge 1: Implement Cache-Aside with Fallback

Write a Python function that implements cache-aside with a Redis-like interface and a database fallback.

**API:**
```python
cache = Cache()  # get(key), set(key, value, ttl), delete(key)
db = Database()  # query(sql, params)

def get_user_profile(cache, db, user_id: int) -> dict:
    # Your implementation here
    pass
```

**Requirements:**
- Return cached profile if available
- Query database on cache miss
- Populate cache on miss with 5-minute TTL
- Handle cache connection errors gracefully (fall back to database)
- Handle cache population errors gracefully (do not fail the request)

**Model Answer:**
```python
def get_user_profile(cache, db, user_id: int) -> dict:
    key = f"profile:{user_id}"
    try:
        cached = cache.get(key)
        if cached is not None:
            return json.loads(cached)
    except CacheConnectionError:
        logger.warning("Cache unavailable, falling back to database")
        cached = None

    try:
        result = db.query("SELECT * FROM users WHERE id = ?", [user_id])
        profile = dict(result)

        if cached is None:  # Cache was available, just had a miss
            try:
                cache.set(key, json.dumps(profile), ttl=300)
            except CacheConnectionError:
                logger.warning("Failed to populate cache, continuing")

        return profile
    except DatabaseError:
        logger.error(f"Database query failed for user_id={user_id}")
        raise
```

**Evaluation criteria:**
- Correct cache-aside pattern (check cache, miss -> DB, populate)
- Error handling separates cache errors from database errors
- Cache population failure does not fail the request
- Proper serialization/deserialization

### Challenge 2: Design Cache Key Scheme for an E-Commerce Platform

Design a cache-aside key scheme for an e-commerce platform with the following entities:

- Product (ID, name, description, price, category, stock count)
- User (ID, name, email, shipping address, order history)
- Order (ID, user_id, products, total, status, timestamp)
- Category (ID, name, product count, description)
- Shopping cart (ID, user_id, items, total)

**Requirements:**
- Each entity type with appropriate TTL
- Granular keys so that updating one field does not invalidate all cached data for an entity
- Support for batch invalidation (e.g., all keys for a user)
- Explain TTL choices

**Model Answer:**

```
Cache Key Scheme:

Product:
  product:{id}:metadata    -> {name, description, category_id}  TTL: 1 hour
  product:{id}:price       -> {price, discount}                  TTL: 5 minutes
  product:{id}:stock       -> {stock_count}                      TTL: 30 seconds
  product:{id}:reviews     -> {avg_rating, review_count}         TTL: 10 minutes

User:
  user:{id}:profile        -> {name, email}                      TTL: 1 hour
  user:{id}:address        -> {shipping_address}                 TTL: 1 hour
  user:{id}:order_history  -> [{order_id, status, total}]        TTL: 5 minutes

Order:
  order:{id}               -> {user_id, items, total, status}    TTL: 30 minutes

Category:
  category:{id}:metadata   -> {name, description}                TTL: 1 hour
  category:{id}:count      -> {product_count}                    TTL: 1 hour

Cart:
  cart:{user_id}           -> {items, total}                     TTL: 24 hours

TTL Rationale:
- Stock (30s): Changes with every purchase, needs near-real-time accuracy
- Price (5min): Changes during sales/promotions, but not per-purchase
- Profile (1hr): Rarely changes, hours of staleness acceptable
- Cart (24hr): Persists across sessions, longer TTL reduces abandoned cart misses
```

**Evaluation criteria:**
- Granularity separates frequently changing fields (stock) from stable fields (name)
- Batch invalidation is possible via `product:{id}:*` pattern
- TTLs reflect the update frequency of each field
- Explanation for TTL choices is provided

### Challenge 3: Trace a Cache-Aside Request

Trace the complete lifecycle of a cache-aside read request for the key `order:42` in a system with:
- Redis cache cluster (3 nodes, consistent hashing)
- PostgreSQL primary with read replicas
- Connection pool with 10 connections per node
- Circuit breaker configured with threshold 5 failures in 30 seconds

**Model Answer:**

```
Step 1: Application receives GET /orders/42
Step 2: Application constructs cache key: "order:42"
Step 3: Cache client hashes key using consistent hashing
        hash("order:42") = position on ring -> node R2
Step 4: Cache client acquires connection from R2's pool (connection 3)
Step 5: Cache client sends: GET order:42
Step 6: Redis R2 receives command via event loop
Step 7: Redis performs hash table lookup on key "order:42" (O(1))
Step 8: SCENARIO A - CACHE HIT
        Step 8a: Redis checks TTL: entry valid (set 2 min ago, TTL 30 min)
        Step 8b: Redis moves entry to LRU head (O(1))
        Step 8c: Redis returns serialized value to client
        Step 8d: Client deserializes JSON to dict
        Step 8e: Application returns response
        Total: ~2ms

Step 9: SCENARIO B - CACHE MISS
        Step 9a: Redis returns nil (key not found or expired)
        Step 9b: Client returns None to application
        Step 9c: Application connects to PostgreSQL read replica
        Step 9d: Application sends: SELECT * FROM orders WHERE id = 42
        Step 9e: Database parses SQL, executes, returns row
        Step 9f: Application receives dict: {id: 42, user_id: 7, ...}
        Step 9g: Application serializes to JSON
        Step 9h: Application calls: cache.set("order:42", json, ttl=1800)
        Step 9i: Client hashes key -> node R2, acquires connection
        Step 9j: Client sends: SET order:42 <json> EX 1800
        Step 9k: Redis stores in hash table, adds TTL, moves to LRU head
        Step 9l: Application returns response
        Total: ~45ms

Error Scenario:
        Cache node R2 is unreachable
        Step 4a: Connection attempt fails after 500ms timeout
        Step 4b: Circuit breaker records failure (count = 1/5)
        Step 4c: Client falls back to database directly
        Step 4d: After 5 failures in 30s: circuit opens
        Step 4e: Subsequent requests skip cache entirely for 30s cooldown
```

**Evaluation criteria:**
- Correct sequence of operations
- Distinguishes hit and miss paths
- Includes error handling scenario
- Mentions connection pool and consistent hashing

## Tier 2: Design Challenges

### Challenge 4: Design Cache-Aside for a News Aggregator

Design a cache-aside system for a news aggregator app that scrapes articles from 1,000 sources and serves them to 10 million daily active users.

**Requirements:**
- Each article has: title, summary, source, publish time, category, image URL
- Articles are read 20x more often than they are added
- Breaking news articles need near-real-time freshness (under 10 seconds)
- Evergreen articles can tolerate minutes of staleness
- Viral articles receive 100,000+ reads in the first hour
- The system must serve articles globally with under 200ms p99 latency

**Rubric:**

| Criterion | Excellent (2 pts) | Adequate (1 pt) | Missing (0 pts) |
|-----------|-------------------|-----------------|-----------------|
| Cache key scheme | Different TTLs for breaking vs evergreen, granular keys | Single TTL for all articles | No key scheme |
| Hot key handling | Mutex lock + pre-warming for viral articles | Mentions stampede but no design | No mention |
| Multi-region | Regional cache clusters with global invalidation | Single region cache | No consideration |
| Failure handling | Circuit breaker + graceful degradation + monitoring | Basic fallback only | No fallback |
| Capacity planning | Memory math + bandwidth math + cost estimate | Rough memory estimate only | No math |

**Model Answer:**

```
Architecture:

Multi-region cache-aside with Redis:
- Region US, EU, APAC each have their own Redis cluster
- Each cluster: 3 shards (6 nodes with replication), 16 GB per node
- Local database read replicas in each region as miss fallback

Cache Key Scheme:

  article:{id}:metadata    -> {title, summary, source, category}   TTL: 1 hour
  article:{id}:image_url   -> {image_url}                          TTL: 24 hours
  article:{id}:breaking    -> {breaking_flag, updated_at}          TTL: 10 seconds
  category:{slug}:top      -> [{article_id, title}]               TTL: 60 seconds

  Hot key indicator:
  If article views > 10,000/hour: flag as hot
  Add: article:{id}:hot     -> {view_count}                        TTL: 300 seconds
  Hot articles get additional replica and pre-warming

Cache Stampede Prevention:

  For viral articles:
  - Background worker checks trending detection every 30 seconds
  - When an article crosses 10,000 views/hour, pre-warm all 3 regions
  - Mutex lock per article: SETNX lock:article:{id} EX 5
  - Only the lock holder queries the database for the full article

  For breaking news (category pages):
  - Separate short-TTL cache key for breaking articles
  - Background refresh every 5 seconds for top 100 breaking articles

Capacity Planning:

  10 million DAU * 20 articles/day = 200 million reads/day = 2,315 reads/sec avg
  Peak factor: 5x = 11,575 reads/sec peak
  Hit ratio target: 95%
  Cache hits: 11,000/sec, Misses: 575/sec

  Memory:
  - Top 500,000 articles in cache (most accessed in 1 hour)
  - Each article metadata: 2 KB JSON
  - Total: 500,000 * 2 KB = 1 GB
  - Plus overhead (Redis 30%): 1.3 GB
  - Plus image URLs (small, negligible)
  - Total active cache: approximately 2 GB per region
  - Provision: 3 shards * 16 GB = 48 GB per region (generous headroom)
```

**Evaluation criteria:**
- Different TTLs for different data types (breaking vs evergreen)
- Proactive hot key handling
- Multi-region design with local caches
- Capacity planning with concrete numbers

### Challenge 5: Migrate from Monolithic to Microservices with Cache-Aside

A monolithic e-commerce application uses cache-aside with an in-process memory cache. The company is migrating to microservices. The new services are: Product Service, User Service, Order Service, and Inventory Service. Each service will have its own database and its own cache.

**Requirements:**
- Each service uses cache-aside independently
- A product update (price change) must invalidate the cache in Product Service
- An order placement must invalidate the cache in Inventory Service (stock count) and Order Service (order history)
- The caching strategy must support independent deployment of each service
- Cache consistency must be eventual (under 5 seconds staleness window)

**Rubric:**

| Criterion | Excellent (2 pts) | Adequate (1 pt) | Missing (0 pts) |
|-----------|-------------------|-----------------|-----------------|
| Service boundaries | Each service owns its cache, clear ownership | Shared cache cluster | Single monolith cache |
| Cross-service invalidation | Event bus (Kafka) with topic per service | Direct API calls for invalidation | No cross-service invalidation |
| Independent deployment | Cache-aside supports service-level deployment | Requires coordinated deployment | No consideration |
| Consistency | Explains eventual consistency + TTL bounds | Assumes immediate consistency | No consistency model |

**Model Answer:**

```
Architecture:

Each Service gets its own Redis namespace:
  Product Service:  Redis cluster P (keys: product:*)
  User Service:     Redis cluster U (keys: user:*)
  Order Service:    Redis cluster O (keys: order:*)
  Inventory Service: Redis cluster I (keys: inventory:*)

Cross-Service Invalidation:
  ┌──────────┐     ┌──────────┐     ┌──────────────┐
  │ Order    │────>│ Inval.   │────>│ Inventory    │
  │ Service  │     │ Event Bus│     │ Service      │
  │ (placed) │     │ (Kafka)  │     │ DEL cache    │
  └──────────┘     └──────────┘     └──────────────┘
                          │
                          ├────> Product Service
                          │      DEL product:{id}:price
                          │
                          └────> User Service
                                 DEL user:{id}:order_history

Event format:
{
  "event_type": "inventory.updated",
  "entity_type": "product",
  "entity_id": 42,
  "reason": "order_placed",
  "timestamp": 1719871234
}

Each service subscribes to the events it cares about:
  - Inventory Service subscribes to: inventory.updated
  - Order Service subscribes to: order.placed, order.cancelled
  - Product Service subscribes to: product.updated

Deployment Strategy:
  1. Deploy Order Service v2 (new version)
  2. Order Service v2 publishes invalidation events (new behavior)
  3. Inventory Service v1 subscribes to events (already running)
  4. No coordinated deployment required

Failure Handling:
  - If the event bus is down, services use TTL-based expiry as fallback
  - TTLs are set short enough (30-60 seconds) that staleness is bounded
  - Services log failed invalidation attempts and retry on reconnection
```

**Evaluation criteria:**
- Clear service boundaries for cache ownership
- Event-driven cross-service invalidation
- Independent deployment support
- Graceful degradation when event bus is unavailable

### Challenge 6: Design Cache-Aside for Real-Time Analytics Dashboard

Design a cache-aside system for a real-time analytics dashboard that displays metrics like active users, page views, revenue, and error rates. The dashboard must show data within 5 seconds of the event occurring.

**Requirements:**
- 1 million events per second ingested
- Dashboard refreshes every 10 seconds
- Queries are for pre-computed aggregates (count, sum, avg, p50, p95, p99)
- Historical data (older than 1 hour) is stored in a data warehouse, not a cache
- Cache hit ratio target: 99.5%

**Model Answer:**

```
Architecture:

Stream processing pipeline:
  Events -> Kafka -> Flink (window aggregation) -> Redis cache -> Dashboard

  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Events   │───>│ Kafka    │───>│ Flink   │───>│{Redis}   │───>│ Dashboard│
  │ Stream   │    │ Topic    │    │ Aggreg. │    │ Cache    │    │ Frontend │
  └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘

Cache Key Scheme (pre-computed aggregates written by Flink):

  dashboard:{metric}:current     -> {value, timestamp}         TTL: 30 seconds
  dashboard:{metric}:last_hour   -> [{ts, val}, ...]           TTL: 1 hour
  dashboard:{metric}:p50         -> {value, timestamp}         TTL: 30 seconds
  dashboard:{metric}:p99         -> {value, timestamp}         TTL: 30 seconds

  Examples:
    dashboard:active_users:current     -> {"count": 45231, "ts": 1719871234}
    dashboard:page_views:last_hour     -> [{"ts": 1719871200, "count": 1200}, ...]
    dashboard:revenue:p99              -> {"value": 149.99, "ts": 1719871234}

Cache-Aside Read:
  Dashboard loads by reading from cache:
  - GET dashboard:active_users:current
  - GET dashboard:page_views:last_hour
  - GET dashboard:revenue:p99

  On miss: Return stale data if available, or show "loading" state
  Never query the data warehouse on dashboard refresh (too slow)

Write Path (Flink writes to cache, not cache-aside in the classic sense):
  Flink writes computed aggregates every 5 seconds:
  - SET dashboard:active_users:current <json> EX 30
  - SET dashboard:page_views:last_hour <json> EX 3600 (with sliding window)
  - SET dashboard:revenue:p99 <json> EX 30

Stampede Prevention:
  - Dashboard polls every 10 seconds. If the cache key is missing (Flink
    delayed), serve the previous value (still available if TTL not expired).
  - Redis replication ensures read replicas can serve stale data if the
    primary node where Flink writes is temporarily unavailable.

Capacity:
  50 metrics * 1 KB each = 50 KB active data
  Negligible memory requirements. Redis is not the bottleneck.
  Focus on Flink's ability to compute aggregates within 5 seconds.
```

**Evaluation criteria:**
- Recognizes that this is a write-heavy cache pattern (Flink-to-Redis)
- Cache-aside still applies for the dashboard-to-Redis read path
- Short TTLs match the refresh frequency
- Never queries the warehouse on dashboard load
- Handles Flink delay gracefully

## Tier 3: Expert Challenges

### Challenge 7: Design Cache-Aside for a Global Multi-Tenant SaaS Platform

Design a cache-aside system for a multi-tenant SaaS platform that hosts 10,000 tenants. Each tenant has their own data and configuration. The platform serves 100 million requests per day across all tenants. Some tenants have 5 users (small business), some have 100,000 users (enterprise).

**Requirements:**
- Strong isolation: one tenant's cache usage must not affect another tenant's performance
- Each tenant can configure their own TTLs and cache behavior
- Cache cost is charged back to each tenant based on usage
- Hot tenants (10% of tenants handling 90% of traffic) must not starve cold tenants
- Zero-downtime cache cluster scaling

**Evaluation Dimensions:**

| Dimension | Weight | Excellent | Adequate | Poor |
|-----------|--------|-----------|----------|------|
| Tenant isolation | 30% | Per-tenant cache node pools or size-based quotas | Shared cache with key prefix isolation | No isolation |
| Configurability | 20% | Per-tenant cache settings API with validation | Hardcoded tenant config | No tenant config |
| Cost allocation | 20% | Per-tenant metrics (memory bytes, ops/sec, bandwidth) with billing | Aggregate metrics only | No metrics |
| Fairness | 20% | Weighted eviction or per-tenant LRU | Global LRU (hot tenants evict cold) | No fairness mechanism |
| Operations | 10% | Automated scaling per tenant pool | Manual scaling | No scaling plan |

**Model Answer:**

```
Architecture:

Cache pools per tenant tier:
  Enterprise tenants (100+ users): Dedicated Redis instance (or shard)
  Small tenants (under 10 users): Shared Redis pool with quotas

  ┌─────────────────────────────────────────────────────────────┐
  │                 Cache Cluster Manager                        │
  │                                                              │
  │  Enterprise Pool (dedicated):                                │
  │    tenant_alice:  Redis instance 16 GB                       │
  │    tenant_bob:    Redis instance 8 GB                        │
  │    tenant_corp:   Redis shard in cluster (4 of 12 shards)    │
  │                                                              │
  │  Standard Pool (shared, 9,950 tenants):                      │
  │    Redis cluster: 12 shards * 8 GB = 96 GB total             │
  │    Per-tenant quota: 100 MB max, 10,000 ops/sec max          │
  │    Database-level enforcement:                              │
  │      - Tenant key prefix: "{tenant_id}:{key}"                │
  │      - Redis ACL or key-spaced stats for monitoring          │
  │      - Connection throttling per tenant at the client layer  │
  └─────────────────────────────────────────────────────────────┘

Cache Key Scheme:
  {tenant_id}:{entity}:{id}:{field}
  e.g., acme_corp:user:42:profile
  e.g., smallbiz_llc:product:99:price

Tenant Configuration API:
  PUT /api/v1/cache/config/{tenant_id}
  {
    "ttl_defaults": {
      "user:profile": 600,
      "product:price": 120
    },
    "eviction_policy": "allkeys-lru",
    "max_memory_mb": 100,
    "serialization": "json"
  }

Cost Calculation:
  cost_to_tenant = (memory_bytes_gb * price_per_gb)
                   + (operations_per_sec * price_per_op)
                   + (bandwidth_bytes * price_per_gb)

  Monthly per-tenant billing:
  - Memory: $0.15/GB-hour
  - Operations: $0.001/10K ops
  - Bandwidth: $0.01/GB

Fairness:
  - Per-tenant LRU: each tenant gets their own LRU space within the shared pool
  - A hot tenant cannot evict a cold tenant's entries
  - If a tenant exceeds quota, their keys are evicted (not other tenants')
  - Implementation: proxy layer that separates keyspaces

Zero-Downtime Scaling:
  1. Add new shards to the cluster
  2. Reshard tenants across old + new shards (migrate keys)
  3. Use Redis Cluster's resharding to move slots without downtime
  4. Update DNS entries for dedicated instances
  5. Cache-aside handles misses during migration (transparent)

Failure Mode:
  - If a dedicated Redis instance fails, the enterprise tenant falls back
    to database-only mode. SLA: 99.99% availability for enterprise pool.
  - If the shared pool has a partial failure, affected small tenants
    degrade to database. Non-affected tenants are isolated.
```

**Evaluation criteria:**
- Tenant isolation at the cache infrastructure level, not just key prefixes
- Per-tenant configuration API (not just hardcoded settings)
- Cost allocation with concrete pricing model
- Fairness mechanism (per-tenant LRU)
- Zero-downtime scaling plan

### Challenge 8: Design a Cache-Aside Anti-Entropy System

Design a system that periodically verifies cache-aside correctness across a large distributed system. The system must detect stale cache entries, measure the staleness gap, and optionally correct stale entries.

**Requirements:**
- Run continuously across 100 Redis cache nodes and 50 database shards
- Must not add significant load to either cache or database
- Must detect stale entries within 1 minute of them becoming stale
- Must handle 10 million active cache keys
- Should not double the cache size (sampling approach)
- Correction should be throttled to avoid database spikes

**Evaluation Dimensions:**

| Dimension | Weight | Excellent | Adequate | Poor |
|-----------|--------|-----------|----------|------|
| Sampling | 30% | Adaptive sampling (biased to hot keys), low overhead | Random sampling, configurable rate | No sampling |
| Staleness detection | 30% | Version-based comparison, metadata check | Compare full values | Timestamp only |
| Correction strategy | 20% | Prioritized correction (oldest, most-frequently-read first) | All stale corrected equally | No correction |
| Operational safety | 20% | Throttled correction, circuit breaker, canary mode | Fixed-rate correction | Unbounded correction |

**Model Answer:**

```
Architecture:

Anti-Entropy Service (background process, 3 instances for redundancy):

  ┌─────────────────────────────────────────────────────────┐
  │                 Anti-Entropy Service                      │
  │                                                           │
  │  1. Sampling Phase (every 60 seconds)                     │
  │     - For each cache node: SCAN with COUNT=100            │
  │     - Adaptive sampling: hot keys (accessed > 1000/min)   │
  │       are sampled every cycle; cold keys sampled rarely   │
  │     - Total: 100 nodes * 100 keys = 10,000 keys/cycle     │
  │     - Cycle time: 60 seconds                              │
  │                                                           │
  │  2. Version Check Phase                                   │
  │     - Each cache entry includes a version field:           │
  │       {"data": {...}, "version": 42, "db_updated_at": T}  │
  │     - Anti-entropy reads: cache entry (1 key GET)          │
  │     - Database query: SELECT version FROM table WHERE id=X │
  │       (one column, indexed, fast)                          │
  │     - Compare version numbers                              │
  │     - Detected stale: cache version < database version    │
  │                                                           │
  │  3. Correction Phase (throttled)                          │
  │     - Rate limit: 100 corrections per second              │
  │     - Priority order:                                     │
  │       1. Keys accessed > 10,000/min (most impactful)      │
  │       2. Keys stale for > 5 minutes (oldest staleness)    │
  │       3. Keys with version gap > 10 (far behind)          │
  │     - Correction: DEL cache key (let cache-aside repop)   │
  │     - Alternative: SET cache key with fresh data          │
  │                                                           │
  │  4. Monitoring and Alerting                               │
  │     - Metric: staleness_gap_seconds (p50, p95, p99)       │
  │     - Metric: stale_entries_per_node                      │
  │     - Alert: p95 staleness > 300 seconds (5 minutes)      │
  │     - Alert: stale ratio > 1% of sampled keys             │
  │     - Dashboard: stale heatmap by cache node, key pattern  │
  └─────────────────────────────────────────────────────────┘

Implementation Details:

  Sampling Algorithm:
    Hot pool:   Keys with access count > 1000/min (typically ~1% of keys)
                Sample every cycle (100% coverage)
    Warm pool:  Keys with access count 10-1000/min (typically ~10% of keys)
                Sample every 5 cycles (20% coverage)
    Cold pool:  Keys with access count under 10/min (typically ~89% of keys)
                Sample every 50 cycles (2% coverage)

  Estimated DB Load from Anti-Entropy:
    10,000 samples/cycle * 1 cycle/60s = 167 DB queries/sec
    Index-only lookup: SELECT version FROM table WHERE id = X
    DB load: approximately 3-5 microseconds per query
    Total: 167 * 5us = approximately 0.8 ms/sec CPU = negligible

  Safety Mechanisms:
    - Canary mode: Run in read-only mode for first week (detect, don't correct)
    - Correction circuit breaker: If correction latency > 500ms, pause corrections
    - Backoff: If stale ratio > 5%, reduce sampling rate to avoid overload
    - Alert on any correction that triggers a DB query taking > 1 second

Data Flow:
  ┌──────────┐     ┌─────────────┐     ┌──────────────┐
  │ Cache    │────>│ Anti-Entropy │────>│  Detection   │
  │ SCAN     │     │ Service     │     │  Log: 100    │
  │ 100 keys │     │             │     │  stale found │
  └──────────┘     └──────┬──────┘     └──────────────┘
                          │
                          ▼
                   ┌──────────────┐
                   │  Priority    │
                   │  Queue       │
                   │  1. Hot keys │
                   │  2. Oldest   │
                   └──────┬──────┘
                          │
                          ▼
                   ┌──────────────┐     ┌──────────────┐
                   │  Corrector   │────>│{Cache}       │
                   │  (throttled) │     │ DEL key      │
                   │  100/s max   │     │ (or SET)     │
                   └──────────────┘     └──────────────┘
```

**Evaluation criteria:**
- Adaptive sampling that biases toward hot keys (most impactful)
- Lightweight version comparison (not full value comparison)
- Prioritized correction queue (hot and old first)
- Operational safety mechanisms (canary mode, circuit breaker, throttling)
