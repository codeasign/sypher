## Naive Implementation

The simplest cache-aside implementation uses an in-process cache within a single application server. This is where most teams start and where most systems break.

**Where it breaks:**

**At 1,000 users.** The in-process cache consumes application memory. A Python dict with 100,000 entries (5 KB each) uses 500 MB plus overhead. The application process hits its memory limit, starts swapping, or triggers OOM kills. The Garbage Collector spends more time managing cache objects than serving requests.

**At 100,000 users (multiple instances).** You scale horizontally to 10 application servers. Each server has its own in-process cache. User A's request hits server 1 and populates its cache. User B's request hits server 2 and misses because server 2 has no cache. The effective hit ratio is diluted across N instances:

```
Effective hit ratio = per_instance_hit_ratio / N

Single server:    90% hit ratio
10 servers:       90% / 10 = 9% effective hit ratio
                  (each server only caches 10% of the keys)
```

The database sees 10x more queries than expected.

**At 10 million users (cache stampede).** A popular key expires simultaneously across all 10 instances. All 10 servers miss, query the database, and populate their local caches. The database receives 10 concurrent identical queries. For a popular key with 100,000 requests per second, the database sees a 100,000 QPS spike.

### Consistent Hashing

Consistent hashing distributes keys across cache nodes while minimizing reshuffling when nodes are added or removed. The key is hashed to a position on a ring, and the nearest clockwise node owns that key.

```
Consistent Hash Ring
     ┌─────┐
     │  R1 │<── key "user:42" maps here
     └─────┘
    /       \
   /         \
┌─────┐     ┌─────┐
│ R3  │     │     │
└─────┘     │     │
            │     │
            │     │
   ┌─────┐ │     │
   │     │ └─────┘
   │     │
   │ R2  │<── key "post:99" maps here
   └─────┘

  Adding R4: only ~1/4 of keys move
  vs. modulo-N: ~3/4 of keys move
```

### Connection Pooling

Each application server maintains a pool of TCP connections to each cache node. The pool's size determines concurrency capacity.

```
App Server 1
│
├── Connection Pool for R1: [conn1, conn2, ... conn10]
├── Connection Pool for R2: [conn1, conn2, ... conn10]
├── Connection Pool for R3: [conn1, conn2, ... conn10]
│
│  Each pool:
│  - Max 10 connections
│  - Min 2 idle connections
│  - Timeout: 5 seconds to acquire
│  - Retry on failure: 3 attempts
```

### Circuit Breaker

A circuit breaker prevents the application from repeatedly trying to reach a failed cache node.

```
State Machine:
  CLOSED -> OPEN -> HALF_OPEN -> CLOSED (or OPEN again)

  CLOSED:   Normal operation, requests pass through
  OPEN:     Failures > threshold (e.g., 5 in 30s)
            Requests fail immediately without network call
            Cooldown period: 30 seconds
  HALF_OPEN:After cooldown, allow 1 test request
            Success -> CLOSED
            Failure -> OPEN (restart cooldown)
```

**Pattern:** Reads go through cache-aside with the read replica as the fallback. Writes go to the primary database, then delete the cache key. This separates read and write concerns and leverages database replication for read capacity.

**Key considerations:**
- Replica lag can cause reads to see stale data even after a cache delete. The replica may be 50-500 milliseconds behind the primary.
- Session-level read-after-write consistency: If a user just wrote data, read from the primary for their next request.

**Pattern:** Cache-aside layers on top of NoSQL databases the same way it layers on top of relational databases. The database is already fast (single-digit millisecond queries), but the cache provides additional speed and reduces read capacity costs.

**Key considerations:**
- NoSQL databases already have built-in caching (DynamoDB DAX, MongoDB in-memory engine). Cache-aside may be unnecessary if these are sufficient.
- Cache-aside with DynamoDB is most useful for hot keys that would otherwise consume disproportionate read capacity units.
- The cache reduces read capacity unit consumption, directly lowering AWS costs. Each DynamoDB read costs $0.00000125 per RCU (on-demand). At 10,000 reads per second, a 90% hit ratio saves $0.001125 per second = approximately $2,920 per month.

**Cost example for DynamoDB cache-aside:**
```
Without cache:
  10,000 reads/sec * $0.00000125/RCU * 86,400 sec/day * 30 days
  = $32,400/month in read capacity

With 90% hit ratio cache-aside:
  1,000 DB reads/sec * $0.00000125/RCU * 86,400 * 30
  = $3,240/month in read capacity
  + Redis cache ($450/month)
  = $3,690/month total

Monthly savings: $28,710
```

This economic incentive drives many production cache-aside deployments on DynamoDB. The Redis cluster pays for itself within days.

### With Message Queue (Invalidation)

<AsciiDiagram
  id="cache-aside/message-queue-invalidation"
  title="Cache-Aside with Message Queue for Invalidation"
  alt="Architecture using a message queue to distribute cache invalidation events across all application instances"
  content={`
┌──────────┐    ┌──────────┐    ┌──────────────┐
│  Service  │    │< Queue > │    │  Subscribers  │
│  A        │    │          │    │               │
│  (Writes) │───>│ cache    │───>│  App Server 1 │──> DEL key
│           │    │ invalid. │    │  App Server 2 │──> DEL key
│           │    │ events   │    │  App Server 3 │──> DEL key
└──────────┘    └──────────┘    │  ...          │
                                └──────────────┘

  Event payload:
  {
    "event": "cache.invalidate",
    "key": "user:42:profile",
    "timestamp": 1719871234,
    "reason": "profile_update"
  }
  `}
/>

**Pattern:** When a service updates data, it publishes a cache invalidation event to a message queue. All application instances subscribe to the queue and delete the cache key upon receiving the event.

**Key considerations:**
- At-least-once delivery guarantees that the cache is eventually invalidated.
- Deduplication: The same invalidation event may be delivered multiple times. `DEL` is idempotent.
- Ordering: If two events arrive for the same key (update A then update B), they must be processed in order. Use partitioned queues with the key as the partition key.
- The event bus adds operational complexity. For small deployments (under 50 application servers), direct invalidation via Redis Pub/Sub is a simpler alternative.

**Redis Pub/Sub alternative (simpler, no message queue):**

```
Write path:
  db.update(key, value)
  redis.publish("cache-invalidate", key)

Read path (each app instance subscribes):
  redis.subscribe("cache-invalidate") -> on message: cache.delete(key)
```

Redis Pub/Sub is simpler but has no delivery guarantees. If a subscriber is temporarily disconnected, it misses the invalidation event. For systems where missed invalidations are acceptable (eventual consistency within the TTL window), this is a reasonable trade-off.

**Key considerations:**
- CDN invalidation is expensive (purge requests). Use long TTLs (hours) for static content and short TTLs (minutes) for semi-static content.
- Cache-aside at the application layer handles data that changes in real time (user profiles, inventory counts).
- The two cache layers are independent but complementary.

## Capacity Planning

### Math for 10 Million DAU

**Assumptions:**
- 10 million daily active users
- Each user makes 5 read requests per session
- Peak hour: 20% of daily traffic over 1 hour
- Peak reads per second: `(10,000,000 * 5 * 0.20) / 3600 = 2,778 reads/sec`
- Peak factor (2x headroom): `2,778 * 2 = 5,556 reads/sec`
- Average entry size: 5 KB (serialized)
- Target hit ratio: 90%

**Cache reads:**
```
Total reads/sec:       5,556
Cash hits (90%):       5,000 reads/sec
Cache misses (10%):    556 reads/sec
→ DB queries/sec:      556 reads/sec (manageable)
```

**Memory calculation:**
```
Active keys (accessed in last TTL window):
  Each key accessed every 300 seconds (5-min TTL)
  Unique keys in window = 5,556 read/s * 300 s = 1,666,800 keys
  Memory per key = 5 KB (value) + 72 bytes (overhead) ~= 5 KB
  Total memory = 1,666,800 * 5 KB = 8.3 GB

Add 20% headroom:       10 GB total cache memory
Redis overhead:          10 GB + 2 GB (replication buffer, etc.)
Total Redis cluster:     12 GB
Split across 3 nodes:   4 GB per node
```

**Network bandwidth:**
```
Read throughput:  5,556 req/s * 5 KB = 27 MB/s
Write throughput: 556 populates/s * 5 KB = 2.7 MB/s
Total:            30 MB/s (well under 1 Gbps = 125 MB/s)
```

**Cost estimate (AWS):**
```
3 x cache.r6g.large (4 GB each) = ~$150/month
vs.
Database scaling: Adding 2 read replicas ~$400/month
Cache savings: ~$250/month + lower latency
```

## Deployment

### Zero-Downtime Cache Cluster Updates

Updating a cache cluster without downtime requires careful orchestration.

```
Phase 1: Add new nodes
  - Add new cache nodes to the cluster
  - Consistent hashing redistributes a fraction of keys
  - Traffic gradually shifts to new nodes
  - Monitor hit ratio during redistribution

Phase 2: Verify health
  - Confirm all nodes are accepting traffic
  - Confirm hit ratio returns to baseline
  - Monitor memory usage on new nodes

Phase 3: Remove old nodes
  - Mark old nodes as draining (stop accepting new keys)
  - Wait for existing connections to drain
  - Remove old nodes from the cluster
  - Verify hit ratio after removal
```

### Monitoring

| Metric | Alert Threshold | Action |
|--------|----------------|--------|
| Hit ratio | Below 80% | Check TTL values, eviction policy, key distribution |
| Miss latency (p99) | Above 200 ms | Check database query performance, connection pool |
| Cache memory usage | Above 80% of maxmemory | Increase cluster size, reduce TTL, check for leaks |
| Eviction rate | Above 100/sec | Increase memory, reduce TTL, check for hot keys |
| Connection pool utilization | Above 80% | Increase pool size, check for connection leaks |
| Cache node CPU | Above 70% | Add nodes, check for slow commands |
| Network bandwidth | Above 60% of link capacity | Add nodes, compress values |

### Alerting

```
CRITICAL:
  - Hit ratio < 60% for 5 minutes  (cache is not helping)
  - Cache cluster node down          (capacity reduced)
  - Eviction rate > 1000/sec        (memory exhaustion imminent)

WARNING:
  - Hit ratio < 80% for 15 minutes  (degraded performance)
  - Memory usage > 80%              (plan capacity increase)
  - p99 miss latency > 300ms        (database under load)

INFO:
  - Cache node added/removed
  - Configuration change
  - TTL adjustment applied
```

### Rolling Upgrades

```
1. Health check all nodes: OK
2. Take node R1 offline (CLIENT PAUSE in Redis)
3. Wait for existing connections to drain (10 seconds)
4. Upgrade node R1 (new version, more memory)
5. Bring node R1 back online
6. Monitor: R1 receives traffic, hit ratio recovers
7. Wait 60 seconds for stabilization
8. Repeat for R2, R3
9. Final health check all nodes: OK
```

## Cache-Aside in Containerized Environments

### Kubernetes Deployment

Deploying cache-aside on Kubernetes requires careful configuration of pod placement, resource limits, and networking to maintain performance.

**Pod placement:** Redis pods should use node selectors to land on instances with dedicated SSD or high-memory instance types. Avoid co-locating Redis pods with CPU-intensive application pods on the same node.

**Resource limits:** Redis is memory-bound. Set `resources.limits.memory` to match the `maxmemory` configuration. If Kubernetes kills a Redis pod for exceeding the memory limit (OOMKill), all cache data is lost.

```
apiVersion: v1
kind: Pod
metadata:
  name: redis-cache
spec:
  containers:
  - name: redis
    image: redis:7-alpine
    args: ["--maxmemory", "8gb", "--maxmemory-policy", "allkeys-lru"]
    resources:
      requests:
        memory: "8Gi"
        cpu: "2"
      limits:
        memory: "8Gi"
        cpu: "4"
    ports:
    - containerPort: 6379
```

**Connection handling:** Application pods connect to Redis via a Kubernetes Service (ClusterIP). The service load-balances connections across Redis pods. For sharded Redis clusters, use a StatefulSet with headless service and connect via a Redis Cluster client that discovers nodes through the cluster protocol.

**Data persistence:** Use persistent volumes (SSD-backed) for Redis persistence. Configure Redis AOF (Append Only File) with `appendfsync everysec` for durability with minimal performance impact.

**Rolling updates:** Use `podManagementPolicy: OrderedReady` in the StatefulSet. Update one Redis pod at a time. The cache-aside pattern ensures that when a pod restarts and loses its data, subsequent requests miss the cache and repopulate from the database.

### Service Mesh Integration

In a service mesh environment (Istio, Linkerd), cache-aside traffic benefits from mTLS, telemetry, and traffic management.

**mTLS between app and cache:** Enable mutual TLS for all cache connections. The overhead is approximately 0.5-1 millisecond per connection for the TLS handshake, which is negligible for long-lived connections in the connection pool.

**Circuit breaking at mesh level:** The service mesh can provide circuit breaking for cache connections, detecting when the cache cluster is unhealthy and short-circuiting traffic to avoid timeouts.

**Traffic splitting for canary cache upgrades:** Use traffic splitting to route a percentage of cache reads to a new cache cluster during migration. This allows you to validate a new Redis version or configuration with a small fraction of traffic before switching fully.

### Auto-Scaling Cache Clusters

Cache capacity planning is difficult because memory usage depends on access patterns, TTL values, and entry sizes that change over time. Auto-scaling helps.

**Memory-based scaling:** When Redis memory usage exceeds 80% of maxmemory for 5 consecutive minutes, trigger a scaling event to add a new shard. When memory usage drops below 50% for 1 hour, remove a shard.

```
Scaling policy:
  scale_up:
    metric: cache_memory_usage_percent
    threshold: 80
    duration: 300s  # 5 minutes
    action: add_shard(1)

  scale_down:
    metric: cache_memory_usage_percent
    threshold: 50
    duration: 3600s  # 1 hour
    action: remove_shard(1)
```

**Cold start mitigation for auto-scaling:** When a new shard is added, its cache is empty. Mitigate the cold-start effect by:
1. Gradually redirecting traffic to the new shard (incremental key redistribution)
2. Pre-warming the new shard with hot keys from existing shards
3. Using Redis replication to clone data from an existing shard before re-sharding

## Security Considerations

### Authentication and Authorization

Redis 6+ supports ACL-based authentication. Every cache client must authenticate before executing commands.

```
# Redis ACL configuration
user default off
user cache-app on >strong-password-here ~* +@read +@write -@admin -@dangerous
```

The application uses the `cache-app` user with read and write permissions but no administration capabilities. This prevents accidental or malicious use of dangerous commands like `FLUSHALL`, `KEYS *`, or `CONFIG SET`.

### Encryption in Transit

All cache traffic between application servers and cache nodes should be encrypted with TLS. Redis 6+ supports TLS natively. Enable it with:

```
# Redis configuration
tls-port 6380
port 0  # disable plaintext port
tls-cert-file /etc/redis/redis.crt
tls-key-file /etc/redis/redis.key
tls-ca-cert-file /etc/redis/ca.crt
tls-auth-clients yes
```

The application's cache client must be configured with the CA certificate to verify the server certificate. The overhead of TLS is typically under 5% for Redis operations because most time is spent on network round-trips, not encryption.

### Network Segmentation

Place cache nodes in a private subnet with no direct internet access. Use security groups or firewall rules to allow traffic only from application server subnets and administration jump boxes.

```
AWS Security Group: cache-sg
  Inbound:
    TCP 6379 from app-sg (application servers)
    TCP 6380 from app-sg (TLS port)
    TCP 6379 from bastion-sg (admin access)
  Outbound:
    None required (cache nodes do not initiate outbound connections)
```

### Secrets Management

Cache passwords and TLS certificates must be managed securely. Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets) rather than hardcoding credentials in configuration files.

```
# Bad: hardcoded password in config file
cache:
  host: redis.internal
  password: "super-secret-123"

# Good: read from environment variable or secrets manager
cache:
  host: redis.internal
  password: ${REDIS_PASSWORD}
```

## Operational Runbooks

### Runbook: Cache Node Failure

**Symptoms:** Cache hit ratio drops by approximately 1/N (where N = number of nodes). Database CPU increases. Some cache client connections show timeouts.

**Diagnosis:**
1. Check Redis cluster status: `redis-cli cluster info`
2. Check each node's health: `redis-cli ping` (per node)
3. Check client-side logs for connection errors
4. Verify network connectivity: `traceroute` to cache nodes

**Immediate actions:**
1. Confirm the node is down (not a network partition)
2. If the node has a replica, promote it: `redis-cli failover`
3. If no replica, remove the node from the cluster:
   `redis-cli cluster forget <node-id>`
4. Monitor database load to ensure it can handle the increased traffic

**Recovery:**
1. Restart the failed node
2. Rejoin it to the cluster: `redis-cli cluster meet <ip> <port>`
3. Verify the node re-syncs with its primary
4. Monitor hit ratio as it recovers over the next TTL cycle

### Runbook: Cache Stampede Event

**Symptoms:** Sudden spike in database queries for a specific query pattern. Database CPU spikes. Alert: "Hit ratio dropped below 60%."

**Diagnosis:**
1. Identify the query pattern causing the stampede
2. Check which cache key expired: look at the query pattern
3. Verify the key's TTL was not accidentally shortened

**Immediate actions:**
1. Manually pre-warm the hot key:
   `redis-cli SET <key> <value> EX <ttl>`
2. If the key is popular, use a longer TTL
3. If the stampede continues, implement a rate limiter on the database query

**Prevention:**
1. Add random TTL jitter (+/- 20%) to prevent batch expiry
2. Implement distributed mutex for cache population
3. Add a background refresh layer for known hot keys

## Summary

Scaling cache-aside from a single-server prototype to a production system requires distributed caching, consistent hashing, connection pooling, circuit breakers, and careful capacity planning. The core pattern remains the same -- check cache first, query database on miss, populate cache -- but the infrastructure around it grows to handle multi-region deployment, zero-downtime upgrades, containerized environments, and proactive monitoring.
