## Internal Architecture

A cache-aside system has four layers that work together to serve data at high speed. Understanding the internals of each layer helps you make correct decisions about configuration, capacity, and failure modes.

### The Cache Client

The cache client is a library integrated into your application. It handles:

- **Key hashing and routing.** Maps a key to the correct cache node in a cluster. Consistent hashing minimizes reshuffling when nodes are added or removed.

- **Connection pooling.** Maintains a pool of TCP connections to cache nodes. Pools avoid the overhead of opening a new TCP connection per request (typically 1-3 milliseconds for a new connection).

- **Serialization and deserialization.** Converts between application objects and the byte format stored in the cache. The client decides the serialization format.

- **Error handling and retries.** Detects failed cache nodes, retries on timeout, and may fall back to database on cache failure.

- **Pipelining.** Batches multiple commands into a single network round-trip for higher throughput.

### The Cache Server

The cache server is the middleware that stores key-value pairs in memory. Redis and Memcached are the two dominant implementations. The server handles:

- **Hash table.** The primary data structure for O(1) key lookups. Redis uses a dictionary (hash table) with incremental rehashing.

- **TTL bucket management.** Expired keys are tracked and evicted. Redis uses a combination of lazy expiration (checked on access) and periodic expiration (a background timer scans keys).

- **Eviction policy.** When memory is full, the server decides which keys to remove. LRU is the most common policy.

- **Network I/O.** The server uses an event loop (Redis uses `epoll`/`kqueue` with a single-threaded event loop) to handle thousands of concurrent connections.

- **Protocol parsing.** Redis uses the REdis Serialization Protocol (RESP), a simple text-based protocol. Memcached uses a custom ASCII or binary protocol.

### The Serialization Layer

Data must be converted to bytes before storage and back to objects after retrieval. The serialization layer sits between the client and the server.

```
Application object (dict, struct, protobuf)
         │
         ▼
  serialization (json, msgpack, protobuf)
         │
         ▼
  byte array (UTF-8 or binary)
         │
         ▼
  cache server (stores as bytes)
         │
         ▼
  deserialization
         │
         ▼
  Application object
```

### The Connection Pool

The connection pool is a critical component that is often misconfigured. It manages a set of TCP connections to the cache server, reusing them across requests.

## Data Structures

### Hash Table

The hash table is the core data structure for O(1) key lookup. Both Redis and Memcached use hash tables internally.

**Structure:** An array of buckets. Each bucket is a linked list of entries (to handle hash collisions). The key is hashed, then the hash is taken modulo the table size to find the bucket index.

```
Hash Table (size = 4)
┌──────┐
│  0   │ ──> "user:42" -> "{\"name\":\"Alice\"}"
├──────┤
│  1   │ ──> (empty)
├──────┤
│  2   │ ──> "post:99" -> "{\"title\":\"Hello\"}"
│      │      └──> "config" -> "{\"debug\":true}"
├──────┤
│  3   │ ──> (empty)
└──────┘

Collision: "post:99" and "config" hash to the same bucket
           (chained via linked list)
```

**Resizing:** When the load factor (entries / buckets) exceeds a threshold (typically 1.0), the hash table is resized to approximately double the size. Redis uses incremental rehashing -- it spreads the rehashing work across multiple operations to avoid latency spikes.

### Linked List for LRU Eviction

LRU (Least Recently Used) eviction requires tracking access order. Redis uses a doubly-linked list where each entry has `prev` and `next` pointers.

**On access (read or write):** The entry is moved to the head of the list. This is O(1) thanks to the doubly-linked structure.

**On eviction:** The tail of the list (least recently used) is removed. O(1).

```
LRU Linked List (most recent at head, least recent at tail)

head ──> [user:42] <══> [post:99] <══> [config] <══> [settings] <── tail
         (just accessed)                              (oldest, next evicted)

On access to "config":
  head ──> [config] <══> [user:42] <══> [post:99] <══> [settings] <── tail
  "config" moved to head (O(1))

On eviction (memory full):
  "settings" is removed from tail (O(1))
```

### Min-Heap for TTL Expiry

Redis uses a more efficient approach for TTL tracking. Each key with a TTL is stored in a min-heap (or priority queue) keyed by expiration time.

**Structure:** A binary tree where each parent has a smaller expiration time than its children. The root is always the soonest-to-expire key.

```
TTL Min-Heap
        ┌────┐
        │ 5s │  ── root: expires in 5 seconds
        └────┘
        /    \
    ┌────┐  ┌────┐
    │ 12s│  │ 30s│
    └────┘  └────┘
    /    \
  ┌────┐ ┌────┐
  │ 45s│ │ 60s│
  └────┘ └────┘

Expiry timer pops root every 100ms:
  Root = 5s -> expired, remove, re-heapify
  New root = 12s -> still valid
  Re-heapify: O(log n)
```

Redis combines lazy expiration (check TTL on access) with periodic expiration (a timer that scans the heap). The background timer runs 10 times per second, examining 20 random keys. If more than 25% of those are expired, it continues scanning.

### Step-by-Step Trace

#### Phase 1: Request Arrival

1. HTTP request arrives at the web server on port 443
2. The router matches `/profile/42` to the profile handler
3. The handler extracts the user ID: `42`
4. The handler constructs the cache key: `profile:42`
5. The handler calls the caching layer

#### Phase 2: Cache Lookup

6. The caching layer calls `cache.get("profile:42")`
7. The cache client hashes the key: `hash("profile:42") = 0xBE12F3`
8. Consistent hashing determines the responsible node: `node 3`
9. The client gets a connection from the connection pool (connection 2)
10. The client sends `GET profile:42` over the TCP connection
11. The cache server (Redis) receives the command via its event loop
12. Redis looks up the key in the hash table: O(1) operation
13. Redis checks the TTL: entry was set 200 seconds ago, TTL is 300 seconds, still valid
14. Redis moves the entry to the front of the LRU list: O(1)
15. Redis sends the serialized value back to the client

#### Phase 3: Deserialization and Response (Hit)

16. The cache client receives the raw bytes
17. The cache client deserializes the JSON: `json.loads(bytes)` -> dict
18. The handler receives the dict
19. The handler renders the response (or passes it up the call stack)
20. The HTTP response is sent to the caller

#### Phase 4: Miss Path (if Phase 2 returns nil)

16. The cache returns nil (key not found or expired)
17. The handler receives the miss
18. The handler constructs a database query: `SELECT * FROM profiles WHERE user_id = 42`
19. The handler gets a connection from the database connection pool
20. The database parses the SQL, generates a query plan, executes it
21. The database reads the row from the buffer pool (or disk)
22. The database returns the result row
23. The handler receives the database result
24. The handler serializes the result: `json.dumps(result)` -> bytes
25. The handler calls `cache.set("profile:42", bytes, ttl=300)`
26. The cache client sends the SET command to the cache server
27. The cache server stores the key-value pair in the hash table
28. The cache server adds the TTL to the min-heap
29. The cache server adds the entry to the LRU list (head)
30. The cache server checks memory usage against the maxmemory limit
31. If memory is full, the cache server evicts the LRU tail

#### Phase 5: Response

32. The handler returns the data to the caller
33. Total elapsed time: 2 milliseconds (hit) or 45 milliseconds (miss)

**What happens:** The application successfully fetched the data from the database but the cache SET failed. The application detects the error (via the Redis client exception) and returns the data to the caller. The next request for the same key will miss again, query the database again, and attempt to populate the cache again.

**Impact:** No data loss, but the cache miss penalty is repeated. The database sees more queries than expected until the cache node comes back.

**Mitigation:** Retry logic with exponential backoff. The cache client should retry failed SET operations. If the cache node is down, the application should fall back to database-only mode (degraded state) and periodically retry the cache connection.

### Concurrent Writes Causing Stale Data

Two requests arrive simultaneously. Request A updates a user profile. Request B reads the same profile. With cache-aside, the write path deletes the cache key, and the read path re-populates it.

**What happens:** The write deletes the cache key, but the read arrives between the delete and the write's database commit. The read sees the pre-update state in the database and caches that stale value. The cache now holds stale data until the TTL expires or another write triggers deletion.

**Impact:** Users see stale data for up to the TTL window. This is a consistency violation.

**Mitigation:** Several strategies reduce this window:
- **Write-through:** Update cache synchronously with the database write.
- **Read-after-write consistency:** Use database read-after-write consistency guarantees. Read from the primary database, not replicas, for the first read after a write.
- **Version stamps:** Include a version number in the cache key. The version is incremented on each write. Reads check the version and discard stale cache entries.

**What happens:** The cache client's timeout mechanism triggers. Each `GET` or `SET` operation waits for the configured timeout (typically 100-500 milliseconds) before failing. The application catches the exception and falls back to the database. The database suddenly sees a 10-50x increase in read traffic.

**Impact:** Response times degrade. The database may become overloaded. If the database cannot handle the increased load, the entire system can fail.

**Mitigation:**
- **Circuit breaker:** After N consecutive cache failures, stop trying to reach the cache for a cooldown period. This avoids the timeout penalty on every request. The circuit breaker periodically tests the cache connection.
- **Connection pool isolation:** Separate the cache connection pool from the database connection pool. Cache failures should not consume database connections.
- **Read replicas:** Ensure the database layer has enough read replica capacity to handle the fallback load.
- **Fast timeout:** Use a short cache timeout (20-50 milliseconds) so the fallback is fast. The cache is meant to be fast; if it is slow, it is better to skip it.

  │<───────────────│            │
  │            │            │
  B: GET key       │            │
  ────────────────>│  Expired!  │
  │  nil           │            │
  │<───────────────│            │
  │            │            │
  A: SELECT ...    │            │
  ─────────────────────────────>│
  │  result        │            │
  │<─────────────────────────────│
  │  SET key val   │            │
  │───────────────>│            │
  │            │            │
  B: SELECT ...    │            │
  (same query!)    │            │
  ─────────────────────────────>│
  │  result        │  DUPLICATE │
  │<─────────────────────────────│  WORK!  │
  │  SET key val   │            │
  │───────────────>│  OVERWRITE │
  │            │            │
  Database did 2x work for the same key
  `}
/>

**What happens:** Both requests detect the expired TTL simultaneously. Both query the database. Both populate the cache. The database does twice the work. This is a mini thundering herd.

**Impact:** At small scale, negligible. At high scale (10,000+ requests per second for a hot key), this can cause a database load spike.

**Mitigation:** Use a distributed mutex or a "set if not exists" pattern. The first request sets a temporary lock key in the cache with a short TTL (e.g., `lock:key -> 1, TTL=5s`). Other requests see the lock and wait or fall back to stale data. Redis's `SET NX` (set if not exists) is ideal for this.

## Performance

### Latency Breakdown

| Operation | p50 | p99 | Notes |
|-----------|-----|-----|-------|
| Cache hit (Redis, same region) | 1 ms | 5 ms | Network round-trip + hash lookup + deserialization |
| Cache hit (Memcached, same region) | 0.8 ms | 4 ms | Simpler protocol, no persistence overhead |
| Cache miss (DB query, simple lookup) | 15 ms | 100 ms | Includes cache timeout + DB query + serialization |
| Cache miss (DB query, complex join) | 50 ms | 500 ms | Depends on query complexity and data size |
| Cache SET (2 KB value) | 0.5 ms | 3 ms | Network + hash insert + LRU update |
| Cache DELETE | 0.3 ms | 2 ms | Network + hash remove + LRU unlink |

### Hit Ratio Targets

- **Well-tuned cache-aside:** 85-95% hit ratio
- **Excellent cache-aside:** 95-99% hit ratio
- **Poor cache-aside:** under 70% hit ratio

**The hit ratio directly determines the average request latency:**

```
avg_latency = (hit_ratio * hit_latency) + ((1 - hit_ratio) * miss_latency)

At 90% hit ratio:
  avg = (0.90 * 1ms) + (0.10 * 50ms) = 0.9ms + 5ms = 5.9ms

At 70% hit ratio:
  avg = (0.70 * 1ms) + (0.30 * 50ms) = 0.7ms + 15ms = 15.7ms

At 99% hit ratio:
  avg = (0.99 * 1ms) + (0.01 * 50ms) = 0.99ms + 0.5ms = 1.49ms
```

### Memory Overhead Per Entry

Each cache entry has overhead beyond the raw value size:

| Component | Size | Notes |
|-----------|------|-------|
| Key (string) | Key length + 8 bytes overhead | Redis SDS overhead |
| Value (serialized) | Value length + 8 bytes overhead | Redis SDS overhead |
| Hash table entry | 24 bytes (dictEntry) | Pointers to key, value, next |
| LRU list node | 16 bytes | prev/next pointers |
| TTL tracking | 8 bytes | Expiration timestamp |
| Redis object overhead | 16 bytes | Type, encoding, LRU, refcount |
| **Total overhead per entry** | **72 bytes + key + value** | Approximate |

For a 2 KB value with a 32-byte key:
- Total: 72 + 32 + 2048 = 2152 bytes per entry
- 1 million entries: approximately 2 GB

## Configuration

### Serialization Format

| Format | Speed (serialize) | Speed (deserialize) | Size Ratio | Schema Required |
|--------|-------------------|---------------------|------------|-----------------|
| JSON | 3-5 us/KB | 2-4 us/KB | 1.0x (baseline) | No |
| MessagePack | 2-3 us/KB | 1-2 us/KB | 0.7x | No |
| Protocol Buffers | 1-2 us/KB | 0.5-1 us/KB | 0.5x | Yes (.proto) |
| Avro | 2-3 us/KB | 1-2 us/KB | 0.5x | Yes (schema) |
| Pickle (Python) | 2-4 us/KB | 3-5 us/KB | 0.9x | No (insecure) |

**Recommendation:** Start with JSON for simplicity and debuggability. Move to Protocol Buffers when serialization overhead becomes measurable (above 100,000 reads/second per instance).

### Connection Pool Size

The connection pool size determines how many concurrent requests can talk to the cache simultaneously.

```
pool_size = (max_concurrent_requests * avg_cache_time_ms) / 1000

Example:
  200 concurrent requests, 2ms avg cache time
  pool_size = (200 * 2) / 1000 = 0.4 = 1 (minimum 1)

  But with 200 requests and 50ms avg cache time (misses):
  pool_size = (200 * 50) / 1000 = 10

Rule of thumb: Start with 10-20 connections per cache node.
Monitor for "connection wait" metrics.
```

### Timeouts

| Setting | Value | Rationale |
|---------|-------|-----------|
| Connect timeout | 500 ms | Cache should be fast; if it takes longer, fall back |
| Read timeout | 100 ms | Cache hit should be under 5 ms; 100 ms is generous |
| Retry interval | 50 ms | Don't hammer a failing cache node |
| Max retries | 3 | After 3 retries, the cache node is likely down |

### Retry Policy

```
retry_strategy:
  max_retries: 3
  base_delay_ms: 50
  max_delay_ms: 500
  backoff: exponential  # 50ms, 100ms, 200ms
  retryable_errors:
    - ConnectionError
    - TimeoutError
    - ClusterDownError
  non_retryable_errors:
    - TypeError  # bad serialization
    - OOMError   # cache is out of memory
```

### TTL Defaults

| Data Type | Default TTL | Rationale |
|-----------|-------------|-----------|
| User profile | 300 seconds (5 min) | Changes infrequently, seconds of staleness ok |
| Product catalog | 600 seconds (10 min) | Changes daily, minutes of staleness ok |
| Blog post content | 3600 seconds (1 hour) | Rarely changes after publish |
| Session data | 1800 seconds (30 min) | Security: shorter TTL reduces risk |
| Configuration | 120 seconds (2 min) | Needs reasonably fast propagation |
| Search results | 60 seconds (1 min) | Freshness matters for relevance |
| Trending topics | 30 seconds | Changes fast, needs frequent refresh |

## Summary

Cache-aside at the implementation level is a combination of hash table lookups, LRU eviction, TTL management, and serialization. The performance characteristics are well-understood: cache hits are 10-100x faster than database queries. The failure modes are bounded: network partitions, concurrent writes, and TTL races all have known mitigations. Configuration choices -- serialization format, pool size, timeouts, TTL values -- are the levers you pull to tune the system for your workload.
