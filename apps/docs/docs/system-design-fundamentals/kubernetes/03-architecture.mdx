## Naive Implementation

### Stage 1: Single-Node Cluster (1,000 users)

A team deploys a single-node Kubernetes cluster on a single virtual machine for their internal application. The cluster runs a control plane and a worker node on the same machine (a pattern called "all-in-one" or "single-node" deployment).

**Setup**: A single VM with 4 CPU cores, 16 GB RAM, and a 100 GB SSD. The control plane components (API server, scheduler, controller manager, etcd) and the kubelet all run on this machine. The team deploys three applications as Deployments with 2 replicas each.

**Problems that emerge at this stage**:

The control plane and application workloads compete for resources. When the API server processes a large number of requests during a deployment, applications experience increased latency. A single process crash (etcd disk full, scheduler OOM) takes down the entire cluster. The single node is a single point of failure. If the node fails, all applications are down until the node is restored.

**What breaks**: etcd's fsync latency on the shared disk causes frequent leader election timeouts. The scheduler's filter and score phase consumes CPU that applications need. The API server's memory usage grows with the number of pods and can trigger the OOM killer.

### Stage 2: Multi-Node Single Control Plane (10,000 users, 50 services)

The team separates the control plane and worker nodes. Two dedicated control plane nodes and ten worker nodes form the cluster. Each worker node runs 5-10 pods across 50 services. The team uses a basic deployment strategy: all pods are manually balanced, and the cluster uses the default scheduler configuration.

**Setup**: Two control plane nodes (4 CPU, 16 GB each) with a single etcd instance and a single API server. Ten worker nodes (8 CPU, 32 GB each). The cluster runs 200 pods total.

**Problems that emerge at this stage**:

The single API server and single etcd instance become bottlenecks. During peak traffic (a deployment of 50 services), the API server handles 500 concurrent requests. Watch connections from 10 kubelets and 50 controllers consume API server memory. The etcd leader processes all writes, and a single fsync latency spike (50 ms) delays all cluster operations.

The scheduler runs on the same node as the API server. When the scheduler processes 200 pods in a batch, the API server's response latency increases by 300 ms. During a rolling update, the scheduler falls behind, and pods remain in pending state for 60 seconds.

**What breaks**: The API server's `--max-requests-inflight` limit is hit. Admission webhooks time out. The scheduler queue grows to 100 pending pods. The etcd database size exceeds 1 GB. The controller manager's watch falls behind, triggering a full re-list of all resources.

### Stage 3: HA Multi-Node Cluster (100,000 users, 500 services)

The team deploys a production-grade HA cluster with three control plane nodes, a dedicated etcd cluster, and 50 worker nodes. The cluster runs 1,000 pods across 500 services. The team uses three availability zones for fault tolerance.

**Setup**: Three control plane nodes (8 CPU, 32 GB each) with a load balancer fronting the API servers. A three-node etcd cluster on dedicated instances (4 CPU, 16 GB, SSD). Fifty worker nodes (16 CPU, 64 GB each) spread across three AZs. The cluster runs 1,000 pods.

**Problems that emerge at this stage**:

The API server's watch system handles 100 concurrent watchers from controllers, 50 kubelet watches, and 50 monitoring system watches. Watch events are delivered at 5,000 events per second. The API server's memory usage reaches 8 GB.

The scheduler processes 1,000 pods across 50 nodes. Filtering each pod against 50 nodes takes 500 ms per pod. Schedule throughput drops to 30 pods per second.

The etcd cluster handles 1,000 writes per second and 5,000 reads per second. The etcd database grows to 4 GB. Compaction runs every 30 minutes. etcd's disk usage for the database and WAL reaches 50 GB.

**What breaks**: The controller manager's resync period (10 minutes) causes a thundering herd of API server requests. The kubelet's image garbage collection can delete images that are still needed by pending pods. The EndpointSlice controller creates 500 EndpointSlice resources, and kube-proxy on each node takes 30 seconds to update iptables rules for all services.

## Production-Grade Architecture

A production Kubernetes architecture at 10 million users requires a multi-cluster design with federation, dedicated infrastructure for the control plane, and automated operations.

### Multi-Cluster Topology

A single Kubernetes cluster has hard limits: 5,000 nodes, 150,000 pods, and 300,000 containers. Beyond these limits, the control plane cannot keep up with the watch event volume and etcd cannot handle the write throughput.

Production deployments use multiple clusters organized by:

- **Region**: One or more clusters per geographic region. Each region has independent control planes and worker nodes. Cross-region traffic goes through a global load balancer.
- **Environment**: Separate clusters for development, staging, and production. Stricter RBAC policies and resource guarantees for production clusters.
- **Workload type**: Separate clusters for batch processing, online services, and machine learning. Each cluster has different node types and scheduling policies.
- **Team**: Large organizations use cluster federation or a management cluster that deploys and manages workload clusters.

### Cluster Federation with KubeFed

Kubernetes Federation (KubeFed) synchronizes resources across multiple clusters. A federation control plane runs in a "host" cluster. You define a `FederatedDeployment` with placement rules (which clusters to deploy to), and the federation controller creates the corresponding Deployment in each target cluster.

Federation provides:
- **Cross-cluster service discovery**: DNS records for services across clusters.
- **Replica scheduling**: Distribute pods across clusters based on resource availability and latency.
- **Failover**: When a cluster becomes unhealthy, federated workloads can be rescheduled to healthy clusters.

### Shared Infrastructure Layer

Production clusters share common infrastructure:

- **Container registry**: A centralized registry (ECR, GCR, ACR, or Harbor) stores all container images. Image pull secrets are distributed across all clusters.
- **Monitoring stack**: Prometheus and Grafana scrape metrics from all clusters. Thanos provides long-term storage and global querying.
- **Logging pipeline**: Fluentd or Vector runs as a DaemonSet on each node, shipping logs to a centralized Elasticsearch or Loki cluster.
- **CI/CD system**: A CI/CD pipeline (ArgoCD, Jenkins X, GitLab CI) deploys to all clusters based on git repository state. ArgoCD uses the GitOps pattern where the desired state of the cluster is stored in a Git repository.
- **Certificate management**: cert-manager automatically provisions and renews TLS certificates from Let's Encrypt or internal CAs.
- **Service mesh**: Istio, Linkerd, or Cilium provides mTLS, traffic routing, and observability across all clusters.

## Integration Patterns

### Pattern 1: Sidecar Proxy

A sidecar container runs alongside the main application container in the same pod. The sidecar handles cross-cutting concerns such as logging, monitoring, proxying, and service mesh integration.

**How it works**: The sidecar container shares the same network namespace and volumes as the main container. Traffic to the main container flows through the sidecar proxy. The sidecar injects mTLS, collects metrics, and enforces traffic policies.

**Common use cases**: Istio/Envoy proxy sidecar, Fluentd log collector sidecar, Redis sidecar for caching, Vault agent sidecar for secrets.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  containers:
  - name: main-app
    image: myapp:1.0
    ports:
    - containerPort: 8080
  - name: envoy-proxy
    image: envoyproxy/envoy:v1.28
    ports:
    - containerPort: 9901
    args:
    - -c
    - /etc/envoy/envoy.yaml
    volumeMounts:
    - name: envoy-config
      mountPath: /etc/envoy
  volumes:
  - name: envoy-config
    configMap:
      name: envoy-config
```

### Pattern 2: Init Container for Bootstrapping

An init container runs before the main application container starts. The init container performs setup tasks such as database migrations, file permission changes, or configuration generation. If the init container fails, the pod does not start.

**How it works**: The init container runs to completion (exit code 0). Only after the init container exits successfully does the main container start. Init containers have access to the same volumes as the main container, allowing them to write files that the main container reads.

**Common use cases**: Running database migrations before the app starts, downloading plugins or assets, waiting for a dependent service to become available, generating configuration files from templates.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-init
spec:
  initContainers:
  - name: db-migration
    image: myapp-migration:1.0
    env:
    - name: DB_URL
      value: "postgres://db:5432/mydb"
    command: ["python", "manage.py", "migrate"]
  containers:
  - name: main-app
    image: myapp:1.0
    ports:
    - containerPort: 8080
```

### Pattern 3: DaemonSet for Node-Level Services

A DaemonSet ensures that one pod runs on every node (or a subset of nodes). When a new node joins the cluster, a new DaemonSet pod is automatically created on that node. When a node is removed, the DaemonSet pod is garbage collected.

**How it works**: The DaemonSet controller watches for node additions and removals. For each node that matches the DaemonSet's node selector, the controller creates a pod with `nodeName` set to that node. The controller ensures exactly one pod per selected node.

**Common use cases**: Log collection agents (Fluentd, Filebeat), monitoring agents (Node Exporter, Datadog Agent), networking plugins (Calico, Cilium), kube-proxy, storage daemons (Ceph, GlusterFS).

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      name: fluentd
  template:
    metadata:
      labels:
        name: fluentd
    spec:
      tolerations:
      - key: node-role.kubernetes.io/master
        effect: NoSchedule
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: dockerlog
          mountPath: /var/lib/docker/containers
      volumes:
      - name: varlog
        hostPath:
          path: /var/log
      - name: dockerlog
        hostPath:
          path: /var/lib/docker/containers
```

### Pattern 4: Operator Pattern for Application Management

An Operator extends Kubernetes with custom resource definitions (CRDs) and custom controllers that manage application lifecycle. The Operator encodes domain-specific operational knowledge (backup, restore, scaling, upgrades) as Kubernetes-native controllers.

**How it works**: You define a CRD (e.g., `PostgresCluster`). You deploy the Operator controller, which watches for CRD instances. When you create a `PostgresCluster` resource, the Operator creates the necessary Deployments, Services, ConfigMaps, PersistentVolumeClaims, and Secrets to run a PostgreSQL cluster. The Operator handles failover, backup scheduling, and version upgrades.

**Common Operators**: Prometheus Operator (manages Prometheus instances), cert-manager (manages TLS certificates), Elasticsearch Operator (manages Elasticsearch clusters), Strimzi (manages Kafka clusters), Crossplane (manages cloud infrastructure resources).

```yaml
apiVersion: postgresql.k8s.io/v1
kind: PostgresCluster
metadata:
  name: my-database
spec:
  version: 15
  instances: 3
  storage:
    size: 100Gi
    storageClass: ssd
  backups:
    schedule: "0 2 * * *"
    retention: 30
  users:
  - name: app
    databases:
    - myapp
```

## Multi-Region and Global Scale

### Global Load Balancing

At global scale, traffic enters through a global DNS-based load balancer that routes requests to the nearest regional cluster. The DNS service uses latency-based routing or geolocation routing to direct users to the closest healthy cluster.

```yaml
# Route53 latency-based routing
# us-east-1: 10.0.0.1 (NLB for US East cluster)
# eu-west-1: 10.0.1.1 (NLB for EU West cluster)
# ap-southeast-1: 10.0.2.1 (NLB for AP Southeast cluster)
```

### Cross-Region Data Synchronization

Cross-region state synchronization is one of the hardest problems in multi-cluster Kubernetes. Applications must handle data locality and replication.

**Stateless workloads**: Stateless applications (web servers, API gateways, frontends) can run in any region. The global load balancer distributes traffic. No data synchronization is needed.

**Stateful workloads with regional sharding**: Stateful applications (databases, caches) are sharded by region. Each region's database stores data for users in that region. Cross-region queries are proxied to the correct region.

**Stateful workloads with global replication**: Some applications require global data visibility. These use cross-region data replication (e.g., Active-Active PostgreSQL with pglogical, or Cassandra with cross-region replication). Kubernetes StatefulSets provide stable network identities and persistent storage for each replica.

### Cluster Registry and Service Discovery

At global scale, a cluster registry stores metadata about all clusters: their API server endpoints, regions, available node types, and current capacity. Service mesh control planes (Istio, Consul) can peer across clusters, enabling cross-cluster service discovery.

```
Global Service Discovery:
1. Client in us-east-1 calls service "myapp"
2. Istio in us-east-1 checks its service registry
3. Local pods of "myapp" are available → route locally
4. No local pods → route to us-east-2 or eu-west-1
5. Cross-region routing adds 50-200 ms latency
```

## Capacity Planning

### Compute Capacity

Calculate the total CPU and memory required for your workloads, plus overhead for the control plane, system daemons, and buffer capacity.

```
Given:
- 500 microservices, each requiring 0.5 CPU and 1 GB memory
- 3 replicas per microservice (for HA)
- 20% overhead for system daemons (kubelet, kube-proxy, networking)
- 15% buffer for scaling and node failures
- 3 control plane nodes, each requiring 8 CPU and 32 GB

Worker node calculation:
  Total application CPU = 500 × 0.5 × 3 = 750 CPU cores
  Total application memory = 500 × 1 GB × 3 = 1,500 GB

  System overhead = 750 × 0.20 = 150 CPU cores
  System overhead memory = 1,500 × 0.20 = 300 GB

  Buffer = (750 + 150) × 0.15 = 135 CPU cores
  Buffer memory = (1,500 + 300) × 0.15 = 270 GB

  Total worker CPU = 750 + 150 + 135 = 1,035 CPU cores
  Total worker memory = 1,500 + 300 + 270 = 2,070 GB

  Using 16-CPU, 64-GB nodes:
  Number of worker nodes = max(1,035 / 16, 2,070 / 64) = max(65, 33) = 65 nodes

  Pod density: 500 × 3 = 1,500 pods / 65 nodes = 23 pods per node
```

### etcd Capacity

```
  etcd cluster: 3 nodes, each with 8 GB RAM, 100 GB SSD
  Maximum keys: 1 million (default prevention)
  Maximum database: 8 GB (default quota)
  Write throughput: 1,000 writes/second
  Read throughput: 10,000 reads/second

  For a cluster with 1,500 pods, 500 services, 500 deployments:
  Total etcd keys ≈ 1,500 + 500 + 500 + 500 (endpoints) + 200 (configmaps/secrets)
    = 3,200 keys (well under 1 million limit)

  etcd database size ≈ 3,200 × 2 KB = 6.4 MB
  (Growth to 100 MB with history and events)
```

### Network Capacity

```
  Pod-to-pod communication: 1,500 pods × 10 KB/s average = 15 MB/s
  API server traffic: 65 nodes × 10 KB/s heartbeats = 650 KB/s
  Watch events: 5,000 events/second × 1 KB = 5 MB/s
  Image pulls: 10 images/hour × 500 MB = 1.4 MB/s average

  Total node bandwidth: 15 + 0.65 + 5 + 1.4 = 22 MB/s (176 Mbps)
  Each node should have at least 1 Gbps network interface.
  Nodes should use a dedicated VPC subnet with 1,000+ available IPs.
```

## Deployment and Operations

### GitOps Workflow

The GitOps pattern uses Git as the single source of truth for cluster state. All changes to the cluster are made through pull requests to a Git repository. An operator (ArgoCD or Flux) continuously reconciles the cluster state with the Git repository.

**Workflow**:

1. A developer creates a pull request to update the deployment image tag from v1.0 to v2.0.
2. CI runs tests and builds the new container image, pushing it to the registry.
3. The pull request is reviewed and merged.
4. ArgoCD detects the change in the Git repository and applies the updated Deployment to the cluster.
5. Kubernetes performs a rolling update, gradually replacing v1.0 pods with v2.0 pods.
6. If the update fails, ArgoCD can auto-rollback by reverting to the previous Git commit.

### Cluster Upgrades

Kubernetes releases every three months. Upgrading a production cluster requires careful planning:

1. **Upgrade etcd first**: etcd upgrades are independent of Kubernetes version. Test the etcd upgrade on a non-production cluster first.
2. **Upgrade control plane**: Upgrade the control plane components one node at a time. Each node is drained before upgrade and uncordoned after. The load balancer handles failover.
3. **Upgrade worker nodes**: After the control plane is upgraded, upgrade worker nodes. Use a rolling node pool replacement: new nodes with the updated kubelet version join the cluster, pods are migrated to new nodes, and old nodes are removed.
4. **Upgrade cluster addons**: After the upgrade, update cluster addons (CoreDNS, kube-proxy, CNI plugin, Ingress controller) to versions compatible with the new Kubernetes version.

### Node Pool Management

Production clusters use multiple node pools with different instance types and configurations:

- **General purpose pool**: Standard instances (e.g., m5.large) for most workloads. Autoscaling enabled.
- **CPU-optimized pool**: Compute-optimized instances (e.g., c5.xlarge) for batch processing and CI/CD runners.
- **Memory-optimized pool**: Memory-optimized instances (e.g., r5.2xlarge) for in-memory databases and caches.
- **GPU pool**: GPU instances (e.g., p3.2xlarge) for machine learning training and inference.
- **Spot/Preemptible pool**: Low-cost instances for fault-tolerant batch workloads. Can be interrupted at any time.

Each node pool has taints and tolerations to ensure workloads are scheduled on the correct instance type:

```yaml
# Node pool taint
spec:
  taints:
  - key: "instance-type"
    value: "gpu"
    effect: "NoSchedule"

# Pod toleration
spec:
  tolerations:
  - key: "instance-type"
    value: "gpu"
    operator: "Equal"
    effect: "NoSchedule"
```

### Backup and Disaster Recovery

A production cluster requires backups for both cluster state and application data.

**Cluster state backup**: `etcd` snapshots are the most critical backup. A cron job on each etcd node takes hourly snapshots and uploads them to cloud storage (S3, GCS). The backup includes the snapshot file and a copy of the etcd member's TLS certificates.

**Backup schedule**:
- Hourly: etcd snapshot (keep 24 hours)
- Daily: Full cluster resource export (keep 30 days)
- Monthly: Long-term archive (keep 12 months)

**Disaster recovery procedure**:
1. Provision a new cluster infrastructure (VPC, nodes, load balancers).
2. Restore the latest etcd snapshot to a new etcd cluster.
3. Start the API server, pointing to the restored etcd.
4. Verify that all cluster resources (Deployments, Services, ConfigMaps) are present.
5. Worker nodes rejoin the cluster automatically through their bootstrap tokens.

**Application data backup**: Persistent volume snapshots are taken at the storage layer (EBS snapshots, GCE disk snapshots, Velero). Velero backs up Kubernetes resources and persistent volumes to object storage, enabling namespace-level or cluster-level restore.

## Observability and Monitoring at Scale

### Metrics Pipeline

A production-grade Kubernetes monitoring stack requires collecting metrics at three levels:

**Infrastructure metrics**: Node-level CPU, memory, disk, and network utilization. Collected by Prometheus Node Exporter running as a DaemonSet. Key metrics: `node_cpu_seconds_total`, `node_memory_MemAvailable_bytes`, `node_filesystem_avail_bytes`, `node_network_receive_bytes_total`.

**Kubernetes metrics**: Control plane metrics from the API server, scheduler, controller manager, and etcd. Key metrics: `apiserver_request_duration_seconds`, `scheduler_pod_scheduling_duration_seconds`, `etcd_server_leader_changes_seen_total`.

**Application metrics**: Pod-level application metrics exposed at `/metrics` endpoints. Collected through pod annotations with Prometheus scrape configuration. Key patterns: RED metrics (Rate, Errors, Duration) for request-based services, USE metrics (Utilization, Saturation, Errors) for resource-based services.

### Prometheus Configuration

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: app-monitor
spec:
  selector:
    matchLabels:
      app: myapp
  endpoints:
  - port: metrics
    interval: 15s
    path: /metrics
  namespaceSelector:
    matchNames:
    - production
```

### Logging Pipeline

Log collection follows the pattern: file or stdout on the node, log shipper, centralized storage, and query interface.

**Standard architecture**: Fluentd or Vector runs as a DaemonSet, collecting logs from `/var/log/containers/*.log` on each node. The shipper adds Kubernetes metadata (pod name, namespace, container name, labels) to each log entry. Logs are forwarded to Elasticsearch for search or Loki for cost-effective storage.

**Structured logging best practices**:
- Log in JSON format so the shipper does not need to parse unstructured text
- Include correlation IDs (trace ID, request ID) in every log line
- Use log levels consistently (ERROR for failures, WARN for degradations, INFO for lifecycle events, DEBUG for troubleshooting)
- Avoid logging sensitive data (passwords, tokens, PII) even at DEBUG level

## Security Architecture

### Defense in Depth

A production Kubernetes cluster requires multiple layers of security:

**Layer 1 -- Cluster access**: API server authentication (X.509 certificates, bearer tokens, OIDC, webhook token authentication). API server authorization (RBAC, ABAC, webhook).

**Layer 2 -- Pod security**: Pod Security Standards (Privileged, Baseline, Restricted profiles). Pod Security Admission controller enforces the standard at the namespace level. Container security context (run as non-root, read-only root filesystem, drop Linux capabilities).

**Layer 3 -- Network security**: Network policies for micro-segmentation. Encryption in transit (mTLS through service mesh). Egress controls to prevent data exfiltration.

**Layer 4 -- Image security**: Image scanning for vulnerabilities. Image signing (Notary, Cosign) to verify authenticity. Admission controller webhooks that reject images with critical vulnerabilities.

**Layer 5 -- Secrets management**: Encryption at rest for etcd. External secrets management (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager). Avoid storing secrets in environment variables (use volume mounts).

### Pod Security Standards

Kubernetes 1.23+ includes the Pod Security Admission controller, which enforces pod security standards at the namespace level.

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production-restricted
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

Three security profiles:
- **Privileged**: No restrictions. For system-level pods that need host access.
- **Baseline**: Minimal restrictions. Blocks known privilege escalations. For most application pods.
- **Restricted**: Strong security. Requires running as non-root, read-only root filesystem, and dropping all capabilities. For security-sensitive workloads.

## Network Policies in Detail

Network policies implement micro-segmentation by controlling traffic flow at the IP address and port level. They are enforced by the CNI plugin (Calico, Cilium, Weave Net).

### Default Deny Pattern

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: secure-app
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

### Allow Specific Traffic

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: secure-app
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          name: database
    ports:
    - protocol: TCP
      port: 5432
```

The policy allows only frontend pods to reach backend pods on port 8080, and allows backend pods to reach database pods on port 5432. All other traffic is denied by the default-deny policy.

## Cost Optimization

### Right-Sizing Nodes

Node selection directly affects cluster cost. The right node size balances pod density (larger nodes reduce control plane overhead) with utilization (smaller nodes reduce waste from fragmentation).

**Small nodes (4 CPU, 16 GB)**: Good for development, low-density workloads. Pod fragmentation is high (each pod consumes a percentage of the total node capacity). Control plane overhead is higher per pod because you need more nodes.

**Large nodes (32 CPU, 128 GB)**: Good for production, high-density workloads. Pod fragmentation is lower. Fewer nodes reduce control plane overhead and node management costs. But a large node failure affects more pods.

**Cost calculation for 1,000 pods**:

```text
Using 4-CPU, 16-GB nodes:
  Max pods per node: 30 (kubelet default)
  Nodes required: ceil(1000/30) = 34 nodes
  Cost per hour: 34 x $0.50 = $17.00/hour

Using 16-CPU, 64-GB nodes:
  Max pods per node: 110 (kubelet default)
  Nodes required: ceil(1000/110) = 10 nodes
  Cost per hour: 10 x $2.00 = $20.00/hour

Using 32-CPU, 128-GB nodes:
  Max pods per node: 110
  Nodes required: ceil(1000/110) = 10 nodes
  Cost per hour: 10 x $4.00 = $40.00/hour
```

The 4-CPU nodes are cheapest for raw capacity but add control plane overhead (more nodes = more etcd keys, more watch connections, more node heartbeats). The sweet spot for most production clusters is 8-16 CPU nodes with 32-64 GB memory.

### Cluster Autoscaler Configuration

```yaml
# cluster-autoscaler deployment
spec:
  containers:
  - name: cluster-autoscaler
    command:
    - ./cluster-autoscaler
    - --v=4
    - --stderrthreshold=info
    - --cloud-provider=aws
    - --skip-nodes-with-local-storage=false
    - --expander=least-waste
    - --balance-similar-node-groups=true
    - --scale-down-delay-after-add=10m
    - --scale-down-unneeded-time=10m
    - --max-node-provision-time=15m
    - --nodes=3:20:eks-worker-group-1
    - --nodes=1:10:eks-spot-group-1
```

Key parameters: `scale-down-delay-after-add` prevents rapid scale-down after a scale-up event. `scale-down-unneeded-time` prevents premature scale-down. `expander=least-waste` prefers the node group that minimizes wasted resources. `balance-similar-node-groups` distributes pods across similar instance types for high availability.
