## Company Case Studies

### Netflix: Kubernetes Migration from Titus

Netflix runs one of the largest streaming platforms in the world, serving over 260 million subscribers across 190 countries. Netflix's infrastructure runs on AWS, and for years it used a custom container management platform called Titus for batch and stream processing workloads. In 2023, Netflix announced plans to run its streaming control plane on Kubernetes while keeping Titus for other workloads.

**Why Kubernetes**: Netflix's streaming control plane consists of hundreds of microservices that manage content discovery, personalization, A/B testing, and playback configuration. These services require frequent deployments (500+ per day), canary deployments, and gradual rollouts. Kubernetes provided a standardized deployment model that Titus lacked.

**Architecture**: Netflix runs Kubernetes on Amazon EC2 instances with a custom CNI plugin that integrates with AWS VPC networking. Each pod gets a real VPC IP address, eliminating the overlay network overhead. Netflix uses Spinnaker for CI/CD, which deploys to both Kubernetes and Titus clusters.

**Challenges**: Netflix's scale means it hits Kubernetes limits that most organizations never encounter. The API server handles 10,000+ watch connections from controllers and monitoring systems. etcd processes 5,000+ writes per second during deployment windows. Netflix contributed patches to Kubernetes to improve watch performance and reduce API server memory usage.

**Results**: Deployment velocity increased by 40 percent. Canary deployments that took 30 minutes in Titus now complete in 10 minutes on Kubernetes. Infrastructure costs decreased by 15 percent through better bin-packing and autoscaling. However, the migration took 18 months and required a dedicated team of 12 engineers.

**Lesson learned**: Kubernetes migration at Netflix scale requires deep platform engineering expertise. The team had to fork the Kubernetes scheduler to add custom scoring algorithms for Netflix-specific workload placement requirements.

### Uber: Multi-Region Kubernetes at Planet Scale

Uber runs one of the largest Kubernetes deployments in the world, with clusters spanning 50+ regions and 10,000+ microservices. Uber's platform engineering team built a custom Kubernetes distribution called "uK8s" that extends vanilla Kubernetes with Uber-specific features.

**Why Kubernetes**: Uber's microservices platform grew organically over a decade, resulting in multiple deployment systems (Peloton, Schemaless, Ringpop). Kubernetes provided a unified platform that could replace these systems with a single orchestration layer. The goal was to reduce operational complexity and enable any team to deploy services to any region with a single workflow.

**Architecture**: Each Uber region runs an independent Kubernetes cluster with three control plane nodes and 100-500 worker nodes. Worker nodes use custom Uber-built container images optimized for their specific workload (high-CPU for dispatch, high-memory for data services, GPU for ML). A global control plane (called "CLI") manages deployments across all regions.

**Challenges**: Uber's dispatch service requires sub-50-millisecond p99 latency. The standard Kubernetes service networking (kube-proxy with iptables) added 3-5 milliseconds of latency per hop. Uber replaced kube-proxy with eBPF-based Cilium, reducing network latency to under 1 millisecond per hop.

**Results**: The unified platform reduced deployment time for new services from 2 weeks to 2 hours. Multi-region deployments, previously requiring manual coordination across 10 teams, now execute in 15 minutes through the global control plane. Infrastructure utilization improved by 30 percent through bin-packing and autoscaling.

**Lesson learned**: Vanilla Kubernetes does not scale to Uber's requirements without significant customization. Uber invested heavily in forking and extending Kubernetes components, which required a team of 30+ platform engineers.

### Discord: Stateful Workloads on Kubernetes

Discord runs a real-time communication platform serving 200 million monthly active users. Discord migrated from a monolithic architecture to microservices on Kubernetes, including stateful services that were traditionally considered difficult to run in Kubernetes.

**Why Kubernetes**: Discord's monolithic architecture made it difficult to deploy updates without downtime. A single deploy required restarting the entire service, causing 5 minutes of downtime. Kubernetes enabled zero-downtime deployments through rolling updates.

**Architecture**: Discord runs Kubernetes on their own bare-metal servers in colocation facilities, not in the cloud. This decision was driven by cost: Discord's video and voice traffic would cost millions of dollars per month in egress fees on AWS or GCP. Each cluster runs 50-100 bare-metal nodes with local SSDs for database storage.

**Stateful workloads on Kubernetes**: Discord runs Elixir-based services on Kubernetes using StatefulSets with stable network identities. The Elixir BEAM VM handles state distribution at the application level, so Kubernetes only needs to provide process management and service discovery. Discord also runs Kafka on Kubernetes using the Strimzi operator, with 100+ broker pods across 10 clusters.

**Challenges**: Bare-metal Kubernetes requires manual hardware provisioning, rack-level failure planning, and custom networking (BGP with Calico). Discord's platform team writes custom operators for hardware lifecycle management.

**Results**: Deployment frequency increased from once per week to 50+ times per day. Rolling updates complete in under 30 seconds. Hardware utilization improved from 15 percent (monolith) to 60 percent (Kubernetes with bin-packing).

**Lesson learned**: Running stateful workloads on Kubernetes is viable when the application handles state distribution (Elixir BEAM, Kafka replication) and does not rely on Kubernetes for state management. The key insight is to use Kubernetes for process lifecycle, not data consistency.

### Cloudflare: Edge Computing on Kubernetes

Cloudflare runs a global network spanning 330 cities across 120 countries. Cloudflare uses Kubernetes to manage its edge computing platform, Workers for Platforms, and internal microservices.

**Why Kubernetes**: Cloudflare needed a platform that could run workloads across its global network while abstracting away the differences between data centers. Kubernetes provided the abstraction layer, but Cloudflare had to modify it extensively for edge use cases.

**Architecture**: Each Cloudflare data center runs a lightweight Kubernetes distribution based on K3s. The control plane runs in a centralized location (not at each edge). Each edge cluster has 5-50 worker nodes depending on data center size. A global service mesh (based on Envoy) routes traffic between edge clusters and centralized services.

**Challenges**: Edge computing on Kubernetes required significant changes to the standard Kubernetes model. Cloudflare replaced etcd with a custom distributed key-value store that works at 330-city scale. The scheduler was replaced with a simpler placement algorithm that considers geographic proximity, latency, and data center capacity.

**Results**: Workloads deploy to all 330 data centers in under 60 seconds. Developer teams can deploy edge services without understanding the underlying infrastructure. Resource utilization across the global network improved by 25 percent.

**Lesson learned**: Vanilla Kubernetes is designed for single-region clusters with 100-5,000 nodes. Edge computing requires a fundamentally different architecture with many small clusters. Cloudflare's platform team rebuilt significant portions of Kubernetes to support this model.

### Pinterest: Migration from EC2 to Kubernetes at Scale

Pinterest migrated 2,000+ services from EC2-managed instances to Kubernetes over 18 months. The migration covered 100,000+ pods running across 3,000 nodes.

**Why Kubernetes**: Pinterest's EC2-based deployment required teams to manage their own AMI images, instance sizes, and autoscaling groups. This resulted in inconsistent deployment patterns, wasted resources (average CPU utilization of 12 percent), and slow deployment cycles (30+ minutes per service).

**Architecture**: Pinterest runs Kubernetes on AWS using a custom tool called "Titus-lite" (not related to Netflix's Titus) that manages node lifecycle, instance selection, and autoscaling. Each service gets its own namespace with resource quotas, network policies, and RBAC rules.

**Results**: CPU utilization increased from 12 percent to 45 percent through bin-packing. Deployment time decreased from 30 minutes to 5 minutes. Service ownership became clearer: teams own their Kubernetes manifests instead of shared AMI images.

**Lesson learned**: Gradual migration is critical. Pinterest migrated services one team at a time, starting with low-risk stateless services and progressively moving to stateful services. The migration required extensive documentation, training sessions, and office hours for service owners.

## Open Source Implementations

### Kubernetes (upstream)

The official Kubernetes project on GitHub is the reference implementation. It provides the kube-apiserver, kube-scheduler, kube-controller-manager, kubelet, and kube-proxy binaries. The project is governed by the CNCF (Cloud Native Computing Foundation) and releases a new version every three months.

### K3s

A certified Kubernetes distribution designed for resource-constrained environments. K3s packages the control plane and worker node components into a single binary under 100 MB. It replaces etcd with an embedded SQLite by default (optional etcd for production) and simplifies networking by using Flannel with WireGuard encryption.

Use cases: Edge computing, IoT, Raspberry Pi clusters, development environments, and small-scale deployments (under 100 nodes).

### MicroK8s

A snap-based Kubernetes distribution from Canonical (the company behind Ubuntu). MicroK8s installs with a single command (`snap install microk8s --classic`) and supports single-node and multi-node clusters. It includes add-ons for Istio, Knative, GPU support, and Kubeflow.

Use cases: Development environments, CI/CD pipelines, and small production deployments.

### kind (Kubernetes IN Docker)

A tool for running Kubernetes clusters inside Docker containers. kind is designed for testing Kubernetes itself, not for production workloads. It creates a multi-node cluster using Docker containers as nodes, each running a systemd process inside the container.

Use cases: CI/CD testing, local development, Kubernetes conformance testing, and learning.

### minikube

A tool for running a single-node Kubernetes cluster locally. minikube creates a VM (using VirtualBox, Hyper-V, or Docker) and installs Kubernetes on it. It supports add-ons for dashboard, ingress, metrics-server, and storage provisioner.

Use cases: Local development, learning, and testing Kubernetes configurations.

## Cloud Provider Implementations

### Amazon EKS (Elastic Kubernetes Service)

EKS manages the Kubernetes control plane across three Availability Zones. AWS handles control plane upgrades, patching, and high availability. EKS integrates deeply with AWS services.

**Key features**:
- IAM integration for authentication (aws-iam-authenticator)
- VPC CNI for native VPC networking (each pod gets a VPC IP)
- ALB/NLB Ingress Controller for load balancing
- EBS CSI driver for persistent storage
- EFS CSI driver for shared file storage
- Fargate for serverless compute (no node management)
- ECR for container image storage
- CloudWatch Container Insights for monitoring

**Pricing**: $0.10 per hour per cluster (control plane). Worker nodes are priced separately as EC2 instances or Fargate tasks.

### Google GKE (Google Kubernetes Engine)

GKE offers the most mature managed Kubernetes service, with features not available in other managed offerings.

**Key features**:
- Autopilot mode (fully managed: no node management at all)
- Node auto-repair and auto-upgrade
- Workload Identity (GCP IAM for pods)
- VPC-native networking (Alias IPs)
- GKE Dataplane V2 (eBPF-based networking, replacing kube-proxy)
- Config Connector (manage GCP resources through Kubernetes CRDs)
- Anthos Service Mesh (Istio-based service mesh)
- Cloud Operations for GKE (monitoring, logging, tracing)

**Pricing**: $0.10 per hour per cluster (standard mode). Autopilot mode charges per pod per hour ($0.00338 per vCPU-hour, $0.00039 per GB-hour).

### Azure AKS (Azure Kubernetes Service)

AKS provides integrated Kubernetes with Azure Active Directory and Azure Policy.

**Key features**:
- Azure AD integration for authentication
- Azure Policy for Kubernetes governance (Gatekeeper/OPA)
- Azure CNI with advanced networking (Cilium integration)
- Azure Disk and Azure Files CSI drivers
- Virtual nodes (ACI-based serverless compute)
- Azure Monitor Container Insights
- Azure Defender for Kubernetes (security)

**Pricing**: Free control plane (no hourly charge). Pay only for worker nodes (Azure VMs) and Azure services.

### Other Managed Offerings

**DigitalOcean Kubernetes**: Simple, affordable managed Kubernetes for small-to-medium deployments. Control plane is free. Nodes run on DigitalOcean Droplets at $10/month each.

**IBM Cloud Kubernetes**: Enterprise Kubernetes with integrated IBM Cloud services. Offers both free and standard clusters.

**OVHcloud Managed Kubernetes**: European cloud provider offering Kubernetes at competitive prices.

**Alibaba Cloud ACK (Container Service for Kubernetes)**: The dominant Kubernetes service in China, integrated with Alibaba Cloud services.

## Lessons Learned

### Lesson 1: Start with Managed Services

Every company case study shows that running your own Kubernetes control plane is a significant operational burden. Start with a managed service (EKS, GKE, AKS) and only consider self-managed Kubernetes when you have specific requirements that the managed services cannot meet.

### Lesson 2: Invest in Platform Engineering

Kubernetes is not a turnkey solution. It provides building blocks, not a finished platform. Successful companies invest in platform engineering teams that build abstractions on top of Kubernetes: developer portals (Backstage), CI/CD pipelines (ArgoCD), and internal developer platforms (Humanite, Qovery).

### Lesson 3: Migration is a Marathon, Not a Sprint

Pinterest's 18-month migration, Netflix's multi-year effort, and Uber's phased rollout all show that Kubernetes migration takes time. Plan for a gradual migration: start with stateless services, add stateful services, and only tackle critical path services after the platform is proven.

### Lesson 4: Monitor Everything Early

Kubernetes generates massive amounts of operational data: node metrics, pod metrics, API server metrics, etcd metrics, scheduler metrics, and controller metrics. Set up monitoring and alerting from day one. Common blind spots include etcd disk space, API server latency, and kubelet heartbeat failures.

### Lesson 5: Capacity Planning is Non-Negotiable

Kubernetes autoscaling handles short-term fluctuations, but you must plan for baseline capacity. The cluster autoscaler can only scale within the constraints of your infrastructure. If you run out of instance type availability in a region, the autoscaler cannot create new nodes.

### Lesson 6: Network Policy from Day One

Without network policies, any pod can communicate with any other pod. This creates a flat network where a compromised pod can attack any service in the cluster. Implement network policies (Calico, Cilium, or native Kubernetes NetworkPolicy) from the beginning. Enforce a default-deny policy and explicitly allow required communication paths.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - port: 8080
```

## Additional Case Studies

### Airbnb: Migrating 1,000 Services to Kubernetes

Airbnb migrated its monolith to microservices on Kubernetes over a 3-year period. The migration covered 1,000+ services running on 2,500+ nodes across multiple AWS regions.

**Why Kubernetes**: Airbnb's monolith was deployed using a custom deployment system called "SmartStack" that required manual configuration for each service. Kubernetes provided a standardized deployment model that reduced the per-service configuration overhead from 2 hours to 5 minutes.

**Architecture**: Airbnb runs Kubernetes on Amazon EKS with custom tooling. Each service gets a namespace with resource quotas, network policies, and RBAC. A custom CI/CD pipeline (called "Kubernete") generates Kubernetes manifests from a service definition file that specifies the Docker image, resource requirements, and environment variables.

**Challenges**: Airbnb's monolith had complex dependencies between components. The migration team had to decouple the monolith into independently deployable services before migrating to Kubernetes. This required rewriting shared libraries, extracting database schemas, and implementing service-to-service authentication.

**Results**: Deployment frequency increased from once per week to 200+ times per day. The time to deploy a new service decreased from 2 weeks to 2 hours. Resource utilization improved from 15 percent to 50 percent. The CI/CD pipeline scaled to support 100+ concurrent deployments.

**Lesson learned**: The Kubernetes migration was the easy part. The hard part was the service decomposition. Airbnb spent 18 months decomposing the monolith before the first service moved to Kubernetes.

### Snapchat: Stateful Workloads on Kubernetes at Scale

Snapchat runs one of the largest Kubernetes deployments in production, with clusters spanning multiple data centers and cloud providers. The platform handles 5 billion+ daily operations.

**Why Kubernetes**: Snapchat's platform needs to scale rapidly to handle traffic spikes during events (New Year's Eve, product launches). Kubernetes autoscaling enables the platform to scale from 10,000 pods to 100,000 pods in under 10 minutes.

**Architecture**: Snapchat runs Kubernetes on bare-metal servers in colocation facilities, with a custom control plane that handles the scale. The cluster uses Calico for networking with BGP peering for pod-to-pod communication. A custom operator manages the lifecycle of Snapchat's stateful services (memcached, Cassandra, Kafka).

**Challenges**: At Snapchat's scale, the standard Kubernetes control plane cannot keep up. The API server handles 50,000+ watch connections. etcd processes 10,000+ writes per second. Snapchat's platform team had to optimize the API server (increasing watch cache sizes, tuning etcd compaction, using protobuf encoding) and contribute patches back to the Kubernetes project.

**Results**: The platform handles 10x traffic spikes during peak events. Pods scale from 10,000 to 100,000 in under 10 minutes. The bare-metal infrastructure reduces costs by 60 percent compared to cloud-based alternatives.

**Lesson learned**: At massive scale, the standard Kubernetes architecture hits hard limits. You need a dedicated platform engineering team to optimize, tune, and patch Kubernetes for your specific workload patterns.

### Adidas: E-Commerce on Kubernetes

Adidas migrated its global e-commerce platform to Kubernetes, running 200+ microservices across 5 regions.

**Why Kubernetes**: Adidas's e-commerce platform needed to handle 10x traffic spikes during product launches (new Yeezy drops, Black Friday). The legacy platform could not scale fast enough — provisioning new VMs took 30 minutes, and the application required 10 minutes of warm-up time.

**Architecture**: Adidas runs Kubernetes on Google GKE with Autopilot mode. The platform uses GKE's cluster autoscaler and Horizontal Pod Autoscaler for automatic scaling. A global load balancer routes traffic to the nearest regional cluster based on latency.

**Results**: Traffic spikes during product launches are handled automatically. The platform scales from 500 pods to 5,000 pods in under 5 minutes. Deployment frequency increased from once per week to 50+ times per day. The platform team shrunk from 15 to 5 engineers.

**Lesson learned**: Managed Kubernetes (GKE Autopilot) removes the operational burden of the control plane. For e-commerce workloads that need to scale rapidly but do not have the platform engineering resources of Netflix or Uber, managed Kubernetes is the right choice.

### Emerging Trends

**eBPF replacing iptables**: Cilium uses eBPF (extended Berkeley Packet Filter) to implement networking, security, and observability in the Linux kernel. eBPF replaces kube-proxy's iptables rules, reducing network latency by 50-80 percent and improving throughput by 2-3x. Cilium is the second most popular CNI plugin after Calico and is gaining adoption in cloud-native environments.

**WASM (WebAssembly) workloads**: Kubernetes is starting to support WebAssembly workloads alongside containers. WASM modules are smaller (KB vs MB), start faster (microseconds vs milliseconds), and have stronger sandboxing than containers. Projects like Krustlet and WasmEdge enable WASM workloads on Kubernetes.

**Platform engineering**: The trend is away from raw Kubernetes and toward internal developer platforms built on top of Kubernetes. Backstage (Spotify's developer portal), Humanitec, and Qovery provide abstractions that hide Kubernetes complexity from developers. The developer deploys a "component" and the platform handles the pods, services, and networking.

**FinOps and cost management**: Kubernetes cost management is becoming a critical discipline. Tools like Kubecost, Vantage, and CloudHealth analyze Kubernetes costs by namespace, label, and pod. The practice of "right-sizing" (matching pod resource requests to actual usage) is a standard operational responsibility.

### Cloud Provider Security Features

**EKS security**: EKS integrates with AWS IAM for authentication. Pods assume IAM roles through IRSA (IAM Roles for Service Accounts). EKS supports VPC CNI policies for network security. EKS encrypts etcd at rest using KMS. AWS GuardDuty monitors for suspicious activity in the EKS control plane.

**GKE security**: GKE uses Workload Identity to map Kubernetes service accounts to GCP IAM service accounts. GKE Dataplane V2 (eBPF) provides network policy enforcement. GKE Sandbox (gVisor) provides an additional security layer for multi-tenant clusters. Binary Authorization ensures only signed container images are deployed.

**AKS security**: AKS integrates with Azure AD for authentication. Azure Policy for Kubernetes enforces governance policies (Gatekeeper/OPA based). Azure Defender for Kubernetes provides threat detection for the control plane and nodes. AKS supports confidential computing for encrypting data in use.

### Production Pitfalls and How to Avoid Them

**Pitfall 1: Not setting resource limits on the kube-system namespace**

If you do not set resource limits on system pods (CoreDNS, kube-proxy, CNI DaemonSet), application pods can starve system components. CoreDNS runs out of memory during traffic spikes. Cluster DNS resolution fails. All services become unreachable by DNS name.

**Fix**: Set resource requests and limits on all system components. Use a PriorityClass to ensure system pods are never preempted.

```
PriorityClass: system-cluster-critical (value: 2000000000)
PriorityClass: system-node-critical (value: 2000001000)
PriorityClass: default (value: 0)
```

**Pitfall 2: Running out of node IPs in VPC subnets**

Each pod gets a VPC IP address (on AWS VPC CNI or similar). When the subnet runs out of free IPs, new pods cannot start. This happens gradually and is hard to detect.

**Fix**: Monitor VPC IP usage. Set alerts at 70 percent, 80 percent, and 90 percent utilization. Use larger subnets (/16 or /20). Use prefix delegation (AWS VPC CNI) to increase IP density per node.

**Pitfall 3: Ignoring pod eviction timeouts**

By default, pods on an unreachable node are evicted after 5 minutes. For critical services, 5 minutes of unavailability may violate SLAs.

**Fix**: Reduce `pod-eviction-timeout` for critical services. Use PodDisruptionBudgets. Spread pods across nodes and zones so a single node failure does not take down all replicas.

### Real-World Monitoring Setup

A real-world Kubernetes monitoring stack follows this architecture:

**Prometheus Operator**: Deploys and manages Prometheus instances as Kubernetes-native resources. The Prometheus Operator creates a Prometheus instance per cluster, configured to scrape metrics from all pods with the `prometheus.io/scrape: "true"` annotation.

**Grafana Dashboards**: Pre-built dashboards for Kubernetes monitoring. Key dashboards include: Kubernetes cluster overview (nodes, pods, resource utilization), Kubernetes namespace views (CPU, memory, network per namespace), and Kubernetes control plane health (API server latency, etcd leader changes, scheduler queue depth).

**Alertmanager**: Routes alerts from Prometheus to notification channels. Critical alerts go to PagerDuty for 24/7 on-call response. Warning alerts go to Slack for daytime triage.

### Lessons from Large-Scale Outages

**Outage 1: Kubelet Heartbeat Storm**

A misconfigured monitoring tool created 10,000 pod status updates per second. Each update went through the API server to etcd. etcd's write throughput was exceeded. The API server became unresponsive. All cluster operations stopped for 30 minutes.

**Root cause**: The monitoring tool ran a reconciliation loop every 2 seconds, updating a status annotation on every pod. With 3,000 pods, this created 1,500 writes per second to etcd.

**Fix**: Remove the monitoring tool's status annotations. Use external monitoring that scrapes metrics without writing to the API server.

**Outage 2: Ingress Controller Memory Leak**

An NGINX Ingress controller leaked memory when a service had 10,000 endpoints. The Ingress controller built an in-memory routing table that grew with the number of endpoints. After 48 hours, the Ingress controller consumed 8 GB of RAM and was OOM-killed. All external traffic was dropped for 5 minutes until the pod restarted.

**Root cause**: The service selector matched too many pods (10,000). The Ingress controller loaded all endpoints into its routing table.

**Fix**: Reduce the number of endpoints per service. Use multiple services with more specific selectors. Monitor Ingress controller memory usage and set resource limits.

**Outage 3: Certificate Expiration Cascade**

cert-manager renewed a TLS certificate, but the admission webhook configuration was not updated to use the new certificate. The old certificate expired. Admission webhooks stopped responding. All resource creation and updates failed.

**Root cause**: cert-manager does not automatically update admission webhook configurations when it renews certificates. The webhook configuration still referenced the old certificate.

**Fix**: Use cert-manager's built-in CA injector that automatically updates webhook configurations. Monitor certificate expiration dates. Set up alerts for certificates expiring within 30 days.

### Tips for Your Own Kubernetes Journey

If you are starting with Kubernetes, follow this progression:

**Month 1**: Set up a local cluster using kind (Kubernetes in Docker) or minikube. Deploy a simple web application with a Deployment and Service. Practice kubectl commands. Break things and fix them.

**Month 2**: Learn about ConfigMaps, Secrets, and Ingress. Deploy a three-tier application (frontend, backend, database). Implement readiness and liveness probes. Practice rolling updates and rollbacks.

**Month 3**: Set up a small cluster on a cloud provider (EKS, GKE, or AKS). Configure monitoring with Prometheus and Grafana. Implement RBAC with multiple users and namespaces. Set up a CI/CD pipeline with ArgoCD or Flux.

**Month 4**: Learn about operators (Prometheus Operator, cert-manager). Implement network policies. Set up Horizontal Pod Autoscaling and Cluster Autoscaling. Practice disaster recovery (etcd backup and restore).

**Month 6**: Design and deploy a multi-cluster architecture. Implement a service mesh (Istio or Linkerd). Set up GitOps for multi-cluster deployments. Write a custom Kubernetes operator. Contribute a bug fix or documentation improvement to the Kubernetes project.

### Kubernetes Certification Path

If you want to validate your Kubernetes skills, the CNCF offers three certification paths:

**CKA (Certified Kubernetes Administrator)**: Tests cluster administration skills: installation, configuration, networking, storage, security, and troubleshooting. The exam is a 2-hour hands-on performance test where you perform tasks in a live cluster. This is the most widely recognized Kubernetes certification for operations engineers.

**CKAD (Certified Kubernetes Application Developer)**: Tests application development skills: designing and building applications, using ConfigMaps and Secrets, implementing probes, and debugging applications. The exam is also a 2-hour hands-on test. This certification is for developers who build and deploy applications on Kubernetes.

**CKS (Certified Kubernetes Security Specialist)**: Tests security skills: cluster hardening, supply chain security, runtime security, and compliance. The CKS requires passing the CKA first. This certification is for security engineers and platform engineers responsible for securing Kubernetes clusters.
