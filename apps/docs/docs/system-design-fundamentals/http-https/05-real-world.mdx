## Company Case Studies

### Netflix — Open Connect CDN and HTTP/2 Migration

**Scale**: 260 million subscribers, 15 billion hours of streaming per quarter, 1.5 billion HTTP requests per day across API and control plane traffic.

**Problem**: Netflix's control plane API used HTTP/1.1 with hundreds of connections per device. Each streaming device (TV, phone, laptop) opened 6-8 parallel connections to the API. With 15 million active devices simultaneously, this meant 90-120 million TCP connections in TIME_WAIT state across the infrastructure. Each TIME_WAIT connection consumed kernel memory for 60 seconds (the default TCP TIME_WAIT duration). At peak, this consumed 12 GB of kernel memory on proxy servers, triggering out-of-memory (OOM) kills every 3 hours. The operations team spent 40% of their on-call rotation handling proxy server restarts. Engineers documented the pattern in an internal postmortem titled "The 12GB Problem" after the sixth recurrence in a single quarter.

**Solution**: Netflix migrated the control plane to HTTP/2, reducing connections from 6-8 per device to 1 per device. They deployed Envoy proxy as a sidecar on every EC2 instance, providing HTTP/2 termination and multiplexing at the instance level. The API gateway was updated to support HTTP/2 stream prioritization, ensuring that play-header requests (critical for starting playback) were prioritized over browse and search requests. The migration was rolled out over 9 months in three phases: Phase 1 (months 1-3) enabled HTTP/2 for a subset of devices in a single AWS region. Phase 2 (months 4-6) expanded to all devices in US-East. Phase 3 (months 7-9) rolled out globally.

The most difficult part was not the protocol change itself — it was that Netflix's custom monitoring and alerting system assumed one connection per request. Under HTTP/2, multiple requests shared a connection. The team had to update every monitoring pipeline to track stream-level metrics instead of connection-level metrics. They built a new dashboard showing "streams per connection" as a health indicator: if the average dropped below 3 streams per connection, it meant clients were falling back to HTTP/1.1 and the migration had regressed.

### Uber — HTTP/3 and QUIC for Mobile Riders

**Scale**: 130 million monthly active riders, 25 million trips per day, 1 billion API requests per day from mobile devices.

**Problem**: Uber's mobile app communicates with the backend over HTTPS using HTTP/2 over TCP. Cellular networks experience 2-5% packet loss in good conditions, and 20-40% packet loss in congested areas (concerts, sporting events, rush hour transit). Under packet loss, HTTP/2's TCP-based multiplexing suffered severely — a lost packet on any stream blocked all streams on that connection. Ride requests during peak events at stadiums (50,000 people leaving simultaneously) timed out at rates of 15-25%. The customer support team received 10,000 tickets per day from users whose ride requests timed out and who then took a competitor's service. Uber estimated that each percentage point of request timeout cost $2 million per year in lost revenue.

The problem was architectural, not operational. TCP's in-order delivery guarantee meant that stream 3's packet loss delayed streams 1, 2, 4, and 5. Uber's internal measurement showed that under 5% packet loss, HTTP/2 was 40% slower than HTTP/1.1 — because HTTP/1.1 used 6 independent connections, only the one that lost a packet was affected, while the other 5 continued serving requests.

**Solution**: Uber migrated its ride-hailing API to HTTP/3 over QUIC. The mobile app uses Cronet (Chromium's QUIC stack, ported to mobile) to establish QUIC connections. QUIC's independent stream loss recovery meant a lost packet on the driver-location stream did not affect the ride-request stream. Additionally, QUIC's 0-RTT connection resumption reduced reconnection time from 150ms (TCP + TLS 1.3 on new connection) to 0ms for returning users. This was especially impactful for the 30% of Uber users who use the app while walking between WiFi hotspots and cellular coverage — each network switch previously required a full TCP+TLS handshake.

The migration involved deploying a QUIC-enabled reverse proxy layer in each region. Uber chose Envoy with QUIC support (v1.23+). The Envoy layer terminated QUIC from clients and forwarded requests over HTTP/2 to backend services. This allowed the backend to remain unchanged while the edge layer handled the new protocol. The client library (Cronet) was distributed through the app store as part of regular app updates.

```python
# Uber's HTTP/3 client configuration (simplified)
client_config = {
    'protocol': 'http/3',
    'quic': {
        'max_streams': 256,
        'idle_timeout_seconds': 30,
        'initial_rtt_ms': 200,
        'connection_migration': True,  # Survive WiFi→cellular switch
    },
    'retry': {
        'max_attempts': 3,
        'backoff_ms': [100, 400, 1600],
        'jitter_factor': 0.5,
    }
}
```

**Result**: Ride request timeout rate dropped from 15-25% to under 1% during high-congestion events. Average API latency on cellular dropped from 380ms to 210ms. The connection-migration feature eliminated the "request failed when switching from WiFi to cellular" bug that affected 5% of users. Uber estimated the migration saved $40 million per year in prevented lost trips.

### Discord — Voice Gateway Protocol Over HTTP/2

**Scale**: 200 million monthly active users, 4 billion messages sent per day, 7 million concurrent voice users at peak.

**Problem**: Discord's voice gateway uses a persistent WebSocket connection for signaling (join/leave voice channels, mute/unmute, volume changes). WebSocket connections are long-lived — a user may stay connected for 8-12 hours. Each WebSocket starts as an HTTPS upgrade request. Discord operates 1 million concurrent WebSocket connections per gateway server cluster. The HTTP/1.1 upgrade path required one TCP connection per WebSocket. With 1 million connections, the kernel's epoll event loop spent 30% of its CPU time just iterating over idle file descriptors to check for events. The remaining 70% of CPU was split between actual packet processing (40%) and context switching (30%) as the kernel scheduled workers across the 200+ goroutines handling connection state.

Discord's infrastructure team published a blog post showing that each additional 100,000 connections added 8% CPU utilization from kernel overhead alone — before any application-level processing occurred. The projected cost of scaling to 2 million concurrent users was not linear; it was super-linear because the kernel overhead grew faster than the connection count.

**Solution**: Discord migrated the WebSocket upgrade path to use HTTP/2 on the control plane. The HTTP/2 connection was established once per client, then the WebSocket upgrade used a new stream within that existing connection. This changed the architecture from "1 connection = 1 WebSocket" to "1 connection = many WebSocket streams."

Key innovation: Discord's custom gateway multiplexes signaling, metrics, and logging — three separate WebSocket connections that were previously independent — into three streams on a single HTTP/2 connection. Each stream has its own flow control and priority. The signaling stream gets the highest priority (P1 — cannot drop a user's voice call). The metrics stream gets P2 (latency and quality metrics are useful but not critical). The logging stream gets P3 (debug logs can wait). HTTP/2's native stream prioritization eliminated the need for Discord's custom priority queue.

The migration was invisible to users — the client library handled the protocol switch internally. Discord's beta testers did not report any change in behavior during the 3-month rollout.

### Cloudflare — HTTP/3 Adoption and Performance

**Scale**: 34 million Internet properties proxied, 40 million HTTP requests per second at peak across the global edge network.

**Problem**: Cloudflare operates in 330+ cities worldwide. Their edge servers terminate HTTP/HTTPS connections from clients and forward requests to origin servers. HTTP/2 over TCP was the primary protocol, but TCP's head-of-line blocking was measurable: at 1% packet loss (common on satellite and cellular links), HTTP/2 page load times were 30% slower than HTTP/1.1 (which uses multiple connections, mitigating per-connection loss impact). Cloudflare's engineering team ran a 6-month study measuring page load times across different protocols and network conditions. The data showed that HTTP/2's theoretical advantages over HTTP/1.1 reversed under packet loss — HTTP/2 became the worse choice exactly when users had the worst network conditions.

**Solution**: Cloudflare enabled HTTP/3 for all free, pro, and business customers (enterprise customers opted in). The QUIC transport eliminated TCP's HoL blocking. Cloudflare's edge servers run quiche (their Rust QUIC implementation) which handles 10,000+ QUIC connections per worker process. They built quiche from scratch because no production-ready Rust QUIC stack existed at the time. The project took 18 months and became the reference implementation used by other companies.

Key architectural decision: HTTP/3 terminates at the edge. Between the edge and the origin server, Cloudflare uses HTTP/2 over TCP (or HTTP/1.1 for legacy origins). This hybrid approach provides the best client experience without requiring origin servers to support QUIC. Cloudflare also implemented a "happy eyeballs" connection strategy: the client attempts QUIC and TCP simultaneously, and uses whichever connects first. This ensures that even if QUIC fails (blocked by a firewall), the user experiences no additional delay.

The rollout was gradual: 1% of traffic in month 1, 5% in month 2, 20% in month 3, and 100% by month 6. Cloudflare monitored QUIC connection failure rates and TCP fallback rates to detect firewalls blocking UDP. They found that approximately 3% of corporate networks block UDP entirely, and another 2% degrade UDP traffic (packet loss above 10%). For these users, the TCP fallback was critical.

**Result**: HTTP/3 adoption reached 35% of all client connections by 2026. For clients using HTTP/3, page load time improved by 35% on cellular and 22% on WiFi. Cloudflare reported that HTTP/3 also reduced their bandwidth costs by 8% because QUIC's loss recovery is more efficient than TCP's retransmission mechanism.

### Stripe — Idempotency and HTTP at Payment Scale

**Scale**: Millions of businesses, 1 trillion dollars in payment volume processed annually, 50 million API requests per day.

**Problem**: Payment processing is the worst-case scenario for HTTP retries. A network timeout between Stripe's API and a merchant's server could mean a duplicate charge, a double-shipped order, or a double-billed subscription. Before Stripe introduced idempotency keys, merchants handled this by storing a unique transaction ID on their side and checking for duplicates — a pattern that every merchant implemented differently, usually incorrectly. Stripe's support team fielded 5,000 tickets per month from merchants asking "did this charge go through?" after a timeout.

**Solution**: Stripe introduced the Idempotency-Key header in 2014. Every POST request can include a unique key. Stripe's API checks the key against a 24-hour Redis cache: if the key exists, the API returns the stored response (including the exact status code and body) without processing the request again. If the key does not exist, the request is processed normally, and the response is stored keyed by the idempotency key.

The implementation faced a critical edge case: what if two identical requests arrive simultaneously, both with the same idempotency key? Stripe solved this with Redis atomic locks using SETNX (set if not exists). The first request to acquire the lock processes the payment. The second request waits for the lock to be released and then retrieves the cached response.

### Dropbox — HTTP Download Optimization for Global Sync

**Scale**: 700 million users, 1 billion files synced per day, 2 exabytes of stored data.

**Problem**: Dropbox's desktop client downloads files over HTTPS using HTTP range requests (byte ranges for resumable downloads). A 1 GB file requires 256 range requests of 4 MB each. On a 100ms-latency connection (US-East to US-West), each range request takes at minimum 200ms (one round trip for the request, one for the response). 256 sequential range requests at 200ms each = 51 seconds just for HTTP overhead, before considering bandwidth.

**Solution**: Dropbox optimized the download path in three ways:

First, they moved from sequential range requests to parallel range requests using HTTP/2 multiplexing. 8 parallel streams over a single connection, each requesting a 4 MB range. This reduced the sequential 51-second overhead to approximately 6 seconds (200ms x 256 / 8).

Second, they increased the range size from 4 MB to 16 MB, reducing the number of range requests from 256 to 64. Combined with parallel streams: 64 requests / 8 streams x 200ms = 1.6 seconds overhead.

Third, they switched from HTTP/1.1 to HTTP/2 for the sync API. HTTP/2's header compression (HPACK) reduced the per-range-request overhead from 400 bytes (HTTP/1.1 headers) to 60 bytes (HPACK-compressed), saving bandwidth on the range request metadata.

The total improvement: 51 seconds of overhead reduced to 1.6 seconds — a 32x improvement. Bandwidth utilization also improved because the parallel requests saturated the available throughput rather than leaving gaps between sequential range completions.

## Open Source Implementations

### Nginx — The Universal HTTP Server

Nginx is the most widely deployed HTTP server on the internet, serving 35% of all websites (W3Techs, 2026). Its architecture is event-driven, single-threaded per worker, and nonblocking.

**How Nginx implements HTTP**: Nginx's HTTP module parses requests using a hand-written FSM parser that operates on a ring buffer. The ring buffer avoids memory allocation per connection — the same buffer space is reused across multiple HTTP requests on the same keep-alive connection. The request is passed through a chain of modules (rewrite, access, proxy, fastcgi, etc.). Each module can modify the request, generate a response, or pass to the next module. Response body is generated in a filter chain that handles compression (gzip, brotli), chunked encoding, sub-filtering (find-and-replace in HTML responses), and XSLT transformation.

Nginx's configuration supports all major HTTP versions. For HTTP/2, Nginx implements HPACK header compression in userspace (no dependency on external libraries). For HTTP/3, Nginx added QUIC support in version 1.25 using a fork of Cloudflare's quiche library.

**Key innovation**: Nginx's upstream module implements HTTP load balancing with passive health checks (mark a server as down after 3 consecutive failures), active health checks (health check requests at configurable intervals), circuit breaking (max_fails + fail_timeout), weighted round-robin, least-connections, IP-hash for session persistence, and random with two-choices load balancing. A single Nginx instance can terminate 100,000+ TLS connections and proxy 200,000 req/s on commodity hardware with 4 CPU cores and 8 GB RAM.

**Production configuration patterns**:
- Worker processes = number of CPU cores (auto detects)
- worker_connections = 1024 (can raise to 4096 with sufficient ulimit)
- Keepalive_requests = 1000 (max requests per connection before recycling)
- sendfile on (zero-copy file serving)
- tcp_nopush on (optimize packet count for responses)

### HAProxy — The TCP/HTTP Proxy

HAProxy specializes in high-availability load balancing. It operates in two modes: layer 4 (TCP proxy, no HTTP parsing) and layer 7 (HTTP-aware, full parsing and routing).

**How HAProxy implements HTTP**: In layer 7 mode, HAProxy parses the HTTP request line and headers using a zero-copy parser — it uses pointers into the receive buffer rather than allocating new strings for each header value. This is critical for memory efficiency: parsing a 50-header request creates zero heap allocations. HAProxy supports ACL-based routing with over 1,000 built-in matching criteria: route based on URL path, header values, body content, HTTP method, client IP, SSL session ID, or any combination with AND/OR/NOT logic.

HAProxy's stick-table stores per-client state (rate limits, connection counts, last-seen timestamp, bytes transferred) in a shared memory structure accessible across all worker processes. The stick-table supports atomic increments for rate counting without locks. A single stick-table can track 1 million clients using 64 MB of shared memory.

**Key innovation**: HAProxy's SPOE (Stream Processing Offload Engine) delegates complex processing (WAF, authentication, rate limiting) to external agents via a custom streaming protocol over TCP. The main proxy stays fast because SPOE agents run in separate processes. If an SPOE agent crashes, HAProxy continues operating without the agent's processing — requests pass through with a warning log entry. This gives HAProxy crash isolation that Envoy and Nginx do not provide for plugin code.

### Envoy — The Cloud-Native Proxy

Envoy is a modern proxy designed for service mesh architectures (used by Istio, Consul Connect, AWS App Mesh). It is written in C++ and designed for high throughput in dynamic environments.

**How Envoy implements HTTP**: Envoy implements HTTP/1.1, HTTP/2, and HTTP/3 natively in a single codebase. Its filter chain model allows pluggable processing at each layer:

- **L4 filters**: TCP proxy, TLS inspector (detects TLS vs plaintext by inspecting the first bytes), original destination (pass-through to the original connection target)
- **L7 filters**: HTTP connection manager, router, LB, fault injection (inject delays and errors for chaos testing), rate limiting (local and global), RBAC (role-based access control), gRPC bridge (transcode HTTP/JSON to gRPC/Protobuf), Lua scripting (custom logic without recompiling Envoy)

Envoy's configuration is fully dynamic — it can receive updated routing rules, cluster members, TLS certificates, and listeners from a control plane via the xDS protocol (Aggregated Discovery Service) without restarting. Configuration updates take effect within 100ms of the control plane push.

**Key innovation**: Envoy's out-of-process architecture. It runs as a sidecar alongside each application instance. All inbound and outbound HTTP traffic goes through Envoy, providing consistent mTLS across the entire mesh, distributed tracing (OpenTelemetry trace headers propagated across every HTTP hop), metrics export (standardized Envoy stats format consumable by Prometheus, Datadog, Grafana), and fault injection for chaos engineering. In a Kubernetes cluster with 500 services, Envoy sidecars handle millions of HTTP requests per second while adding under 2ms of latency per hop.

### Redis — HTTP Session Cache for TLS

Redis is not an HTTP server, but it is the most common backing store for TLS session caches and HTTP rate limit counters. A Redis-based TLS session cache enables multi-server session resumption: when a client connects to server A, the session is stored in Redis. When the same client connects to server B (behind the same load balancer), server B retrieves the session from Redis and performs an abbreviated handshake.

**Configuration pattern**:

```
# Nginx configuration for Redis-based TLS session cache
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 24h;
ssl_session_ticket_key /etc/ssl/session_ticket.key;

# Alternative: use Redis for multi-server session sharing
# Requires nginx redis module or Lua scripting
ssl_session_store "lua:redis.call('SET', KEYS[1], ARGV[1], 'EX', 86400)";
ssl_session_fetch_by_lua "
    local res = redis.call('GET', KEYS[1])
    return res
";
```

A single Redis node handles 100,000 session cache queries per second. For higher throughput, Redis Cluster with 6 shards handles 500,000 queries per second with under 1ms p99 latency. The session cache is the most latency-sensitive Redis use case because it sits on the critical path of every new TLS connection.

## Lessons Learned

**Pattern 1: HTTP version upgrades are infrastructure migrations, not configuration changes**. Netflix and Discord both spent 6-12 months migrating from HTTP/1.1 to HTTP/2. The migration touched every layer: client libraries, proxies, load balancers, monitoring dashboards, and alerting rules. Each company found that their HTTP/1.1 infrastructure had implicit assumptions about connection behavior that broke under HTTP/2. Netflix's monitoring assumed one connection per request; under HTTP/2, one connection carried multiple requests, flooding the metrics pipeline with unexpected data.

**Pattern 2: The biggest gains come from reducing connection count, not improving per-request latency**. Netflix's 87% connection reduction eliminated an entire category of failure (OOM kills from TIME_WAIT state). Uber's HTTP/3 migration focused on reducing connection overhead in lossy environments. Discord's consolidation of three WebSocket connections into one HTTP/2 connection reduced CPU usage by 25%. Optimizing connection management consistently yields higher returns than optimizing individual request processing.

**Pattern 3: HTTP/3 is a mobile-first protocol**. Every company that deployed HTTP/3 saw the largest benefits on cellular networks. Cloudflare reported 35% improvement on cellular versus 22% on WiFi. Uber's primary motivation was cellular congestion. Dropbox's parallel range requests improved download speeds on lossy mobile connections by 32x over sequential HTTP/1.1 requests. If your user base is primarily on mobile with varying network quality, HTTP/3 should be a priority.

**Pattern 4: Hybrid protocol stacks are the norm**. No company surveyed uses a single HTTP version end-to-end. Cloudflare uses HTTP/3 at the edge and HTTP/2 to origins. Netflix uses HTTP/2 from client to Envoy proxy and HTTP/1.1 from Envoy to backend services. Discord uses HTTP/2 for WebSocket upgrades and HTTP/1.1 for internal REST calls. Uber uses HTTP/3 at the edge gateway and HTTP/2 internally. The optimal configuration is "fastest protocol available per hop," not "one protocol everywhere."

**Pattern 5: Connection migration is an unsolved problem at scale**. Discord and Netflix both identified connection migration as the next frontier. When a mobile user switches from WiFi to cellular, their TCP connection breaks. HTTP/3's connection migration (QUIC connection ID survives address change) solves this at the transport layer, but application-layer state (auth tokens, session state, partially-completed requests) must still be migrated. Companies are building this themselves because standard solutions do not exist yet. Uber's solution was Cronet's built-in connection migration; Discord's solution was to let WebSocket reconnections restore state from a server-side snapshot.

**Pattern 6: Idempotency is the most important HTTP feature for payment and order systems**. Stripe's idempotency key pattern became an industry standard because it solves the fundamental ambiguity of HTTP retries: "did my POST succeed or not?" Every system that handles money, orders, or state mutations should implement idempotency keys. The implementation is simple (Redis SETNX + TTL) but the impact on reliability is enormous.

**Pattern 7: HTTP monitoring must be stream-aware, not connection-aware**. Under HTTP/1.1, monitoring a connection was equivalent to monitoring a request. Under HTTP/2 and HTTP/3, one connection carries many streams. Companies that migrated to multiplexed protocols had to rebuild their monitoring pipelines to track stream-level metrics. Netflix built a "streams per connection" dashboard. Cloudflare monitors "stream reset rate" as a proxy for connection health. Discord tracks "stream priority distribution" to detect priority inversion bugs. The operational lesson: protocol changes require monitoring changes.
