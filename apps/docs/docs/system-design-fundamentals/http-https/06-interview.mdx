## How This Topic Appears

HTTP and HTTPS appear in three question categories: direct protocol questions, system design questions where HTTP is a component, and debugging questions where HTTP behavior explains a failure. At FAANG, approximately 1 in 5 system design interviews includes an HTTP-specific sub-question, and 1 in 10 coding interviews includes a debugging scenario rooted in HTTP behavior.

### Direct Protocol Questions

These questions test your knowledge of the protocol itself. They appear most frequently in the first 10 minutes of a system design interview, as an entry point before the main design discussion.

1. "Explain what happens when you type https://www.google.com into a browser and press Enter."
   - This is the most common opening question in system design interviews. It tests breadth — can you describe DNS, TCP, TLS, HTTP, rendering — and depth — will you mention caching, keep-alive, HTTP/2, resource loading, rendering pipeline.

2. "What is the difference between HTTP and HTTPS? Walk through the TLS handshake."
   - Tests security fundamentals. The interviewer wants to hear about certificate chains, cipher negotiation, symmetric vs. asymmetric encryption, and the latency cost of the handshake.

3. "How does HTTP/2 differ from HTTP/1.1? What problem does it solve?"
   - Tests protocol evolution awareness. Strong answers mention multiplexing, HPACK, server push, binary framing, and TCP-level head-of-line blocking as a remaining limitation.

4. "Explain HTTP/3 and QUIC. When would you choose it over HTTP/2?"
   - Tests edge knowledge and trade-off thinking. The interviewer wants to hear that QUIC eliminates TCP HOL blocking, uses UDP, supports 0-RTT, and that the trade-off is firewall compatibility.

5. "What status codes have you seen in production and what did they mean?"
   - Tests production experience. The candidate who mentions 429 (rate limited), 502 (bad gateway), 503 (unavailable), 304 (not modified), 401 (unauthorized), 403 (forbidden), 409 (conflict), 412 (precondition failed) with real scenarios is stronger than the candidate who knows 200 and 404.

6. "How do HTTP cookies work? What security properties do they have?"
   - Tests web security knowledge. Expect discussion of Secure, HttpOnly, SameSite attributes, CSRF protection, cookie scoping by domain/path.

7. "What is CORS and why does it exist?"
   - Tests understanding of the same-origin policy and the need for cross-origin resource sharing. Strong answers explain the preflight OPTIONS request and the Access-Control-Allow-Origin header.

8. "What is the difference between Cache-Control: no-cache and Cache-Control: no-store?"
   - Tests caching knowledge. no-cache means cache but revalidate before serving. no-store means do not cache at all. Many engineers get this wrong.

9. "How would you design a retry strategy for an HTTP API client?"
   - Tests systems thinking. The interviewer wants to hear about jittered exponential backoff, idempotency keys, circuit breakers, handling of specific status codes like 429 and 503.

### System Design Questions Where HTTP Is a Component

These questions are primarily about the design of a system, but the HTTP details are a signal that separates good answers from great ones.

10. "Design a URL shortening service like TinyURL."
    - HTTP angle: how does the 301 redirect work? What headers are involved? How do you handle HTTPS for the shortened URL?

11. "Design a rate limiter for an HTTP API."
    - HTTP angle: what status code to return (429), what headers to include (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset, Retry-After), how to communicate rate limit state in a distributed system.

12. "Design WhatsApp — how does the client-server protocol work?"
    - HTTP angle: WebSocket upgrade from HTTPS, long polling fallback for restricted networks, HTTP/2 stream prioritization for message delivery.

13. "Design YouTube — how does video streaming work over HTTP?"
    - HTTP angle: HTTP range requests, chunked transfer encoding, DASH/HLS manifest files delivered over HTTP, CDN edge caching of video segments, byte-range caching policies.

14. "Design a CDN — how does it serve content over HTTPS?"
    - HTTP angle: TLS termination at edge, certificate management at scale (millions of domains), HTTP/3 at edge with HTTP/2 to origin, cache-control headers for invalidation.

15. "Design an API gateway — what HTTP-related features does it provide?"
    - HTTP angle: rate limiting, authentication, request transformation, CORS handling, TLS termination, HTTP version negotiation, response compression, request validation.

### Debugging Questions

These questions test operational experience. They are common in on-site rounds at companies that run their own infrastructure.

16. "Users in Asia report that your API is slow. What could cause this at the HTTP level?"
    - DNS latency from non-optimized resolvers, lack of edge CDN, TCP/TLS handshake overhead over high-latency links, no connection reuse, HTTP/1.1 serialization.

17. "Your API returns 503 intermittently. What checks do you do?"
    - Load balancer logs for upstream response time, backend server health check status, connection pool metrics, garbage collection logs, upstream dependency latency, request queue depth at each layer.

18. "Certificate errors started appearing in production. What is the root cause?"
    - Certificate expired, certificate issued for wrong domain, intermediate CA not included in cert bundle, client's system clock is wrong (certificate not yet valid), OCSP responder is unreachable causing soft-fail.

19. "A deployment caused 502 errors for 10% of users. What happened?"
    - Most common cause: the deployment removed the old servers before the load balancer detected they were unhealthy. The 10% is the traffic that was routed to the decommissioned servers during the DNS/TTL transition window.

20. "Your HTTP/2 service is slower than HTTP/1.1. Why?"
    - Most likely cause: TCP packet loss. HTTP/2's single TCP connection means a lost packet blocks all streams. HTTP/1.1's multiple connections mean only one connection is affected. Check the server's TCP retransmission rate.

## What the Interviewer Is Actually Testing

| Question Type | What Is Tested | Strong Answer Signals | Weak Answer Signals |
|---------------|----------------|----------------------|---------------------|
| "What happens when you type a URL" | Full-stack systems thinking | DNS → TCP → TLS → HTTP → parse → render in order; mention caching, HTTP/2, keep-alive | Only mentions DNS and HTTP; no TLS; no mention of caching |
| "Difference between HTTP and HTTPS" | Security fundamentals | Explain TLS handshake, cert chain, cipher negotiation, symmetric vs asymmetric | "One is secure, one is not" — no mechanism |
| "HTTP/2 vs HTTP/1.1" | Protocol awareness and trade-off thinking | Multiplexing, HPACK, server push, binary framing, TCP HoL blocking | "It is faster" — no why |
| "Design a rate limiter" | Systems design with HTTP-awareness | Mention 429 status, Retry-After header, token bucket vs sliding window, distributed rate limiting | Only mentions algorithm, not HTTP protocol integration |
| "Debug slow API in Asia" | Multi-layer troubleshooting | CDN, regional latency, DNS resolution time, TLS negotiation overhead, connection reuse | Only mentions application code |
| "What status codes" | Production experience | Mentions 429, 503, 502, 304, 401, 403 with real scenarios | Only knows 200 and 404 |

The interviewer is rarely testing whether you can recite RFCs. They are testing whether you understand how the protocol behaves at scale, under failure, and in complex distributed systems. A candidate who says "HTTPS uses TLS which encrypts the connection" passes the knowledge check. A candidate who says "HTTPS adds a 1-3 RTT handshake overhead, costs CPU on the server side, and requires certificate management including renewal and revocation — but protects against MITM and is required for HTTP/2" demonstrates the depth that differentiates a senior engineer.

**The most common mistake**: answering a protocol question purely from theory without grounding it in numbers. "TLS adds overhead" is theory. "TLS 1.3 adds 1 round trip, approximately 40ms at 95th percentile across US coast-to-coast, and each new connection uses about 3KB of server memory for the session state" is an answer that shows production experience.

## Level-by-Level Expectations

### L3/L4 (Entry-level / Early career)

**Expected**: You can describe the HTTP request-response cycle. You know the difference between GET and POST. You understand that HTTPS encrypts traffic. You can list common status codes (200, 301, 400, 401, 403, 404, 500, 502, 503). You understand that DNS converts domain names to IP addresses. You know that browsers open multiple connections for parallelism.

**Not expected**: TLS handshake details, HTTP/2 multiplexing, protocol version trade-offs, connection pooling strategies, certificate management at scale. If asked about these, it is fine to say "I have not worked with that in depth, but I understand the concept at a high level."

**Sample answer to "What happens when you type a URL"**:

"I will start from the top. First, the browser checks its cache for the DNS record. If not found, it asks the OS, then the router, then the ISP's recursive resolver. The resolver queries the root, TLD, and authoritative nameservers to get the IP address. Then the browser opens a TCP connection to that IP on port 443 (since it is HTTPS). It performs a TLS handshake: the server sends its certificate, the browser verifies it against its root CA store, and they negotiate a symmetric key. Then the browser sends an HTTP GET request for the page. The server returns a 200 OK with HTML. The browser parses the HTML, finds references to CSS and JS files, and issues additional GET requests for those. Once all resources are loaded, the page renders."

**This answer would be strong for L3/L4** — it covers the full flow in order and shows understanding of caching, DNS hierarchy, TLS, and resource loading. It would be weak for L5+ (no mention of HTTP/2, keep-alive, CDN, or failure modes).

### L5 (Senior)

**Expected**: Everything from L3/L4, plus: You can explain the trade-offs between HTTP versions. You know how TLS affects latency and can calculate the overhead in round trips. You understand connection pooling, keep-alive timing, and caching headers (max-age, no-cache, no-store, must-revalidate, stale-while-revalidate). You have opinions on when to use HTTP vs HTTPS internally (VPC traffic vs cross-region). You can design a retry strategy with concrete numbers and idempotency handling. You know the difference between HTTP/2 TCP-level HOL blocking and HTTP/1.1 application-level HOL blocking.

**Sample answer to "How would you design an HTTP retry strategy?"**:

"I would use jittered exponential backoff. First attempt, wait a random interval between 100ms and 300ms. Second attempt, wait 400ms to 800ms. Third attempt, wait 1.6s to 3.2s. Maximum 3 retries for non-idempotent requests. For idempotent requests (GET, PUT, DELETE, or POST with an idempotency key), I would allow up to 5 retries.

I would implement a circuit breaker: if 5 consecutive requests to the same downstream service fail, the circuit opens. All requests to that service fail fast for 30 seconds without making network calls. After 30 seconds, I let one request through as a probe. If it succeeds, close the circuit. If it fails, keep it open for another 30 seconds.

For retry on 429 (rate limited), I read the Retry-After header and wait exactly that duration. Do not use exponential backoff for 429 — the server already told you when to retry. For retry on 5xx, use exponential backoff — the server is likely overloaded and needs time to recover. For retry on 4xx (except 429), do not retry at all — the client request is wrong and retrying will not fix it.

I would log every retry attempt and its outcome to a structured log. If retry consumption exceeds 5% of all requests, that is a signal that downstream services are degraded and we need to alert. I would also tag the downstream call in distributed tracing so we can distinguish first attempts from retries in the trace view."

**This answer would be strong for L5** — it shows concrete numbers, edge case handling (idempotency, distinction between 429 and 5xx), a circuit breaker pattern, and observability awareness.

### L6 (Staff)

**Expected**: Everything from L5, plus: You can design a multi-region HTTP infrastructure with active-active routing. You understand the operational implications of protocol choices at scale (connection state memory, CPU overhead of TLS termination, certificate renewal. You have experience debugging HTTP issues in production (retry storms, slowloris attacks, connection pool exhaustion, TLS renegotiation failures). You can articulate how HTTP interacts with service meshes, API gateways, and CDNs. You can estimate capacity requirements for a given traffic pattern. You have a position on HTTP/3 adoption timelines and can defend it.

**Sample answer to "Design the HTTPS infrastructure for a global API serving 100 million requests per day"**:

"I would start with a CDN for TLS termination and static caching — Cloudflare or Fastly at 330+ edge locations. The CDN terminates TLS 1.3 with HTTP/3 support for mobile clients because mobile networks have the highest packet loss and QUIC's independent stream recovery matters most there. The CDN routes to the nearest regional data center over HTTP/2 to minimize origin connection overhead; we do not need QUIC on the edge-to-origin link because that connection runs over dedicated fiber with near-zero packet loss.

In each region, I deploy an Envoy proxy layer with mTLS between all internal services. Envoy handles circuit breaking, retries with jitter, and distributed tracing (OpenTelemetry headers propagated across every HTTP hop). The Envoy configuration is dynamic — it receives routing rules from a control plane (Istio or Consul) so we can reroute traffic without restarts.

The API gateway (Kong or custom) sits behind Envoy and handles authentication (JWT validation), rate limiting (sliding window per API key), and request validation. Rate limit responses include Retry-After and RateLimit-Reset headers so clients can behave correctly. We also set X-Request-Id on every incoming request for traceability.

For the database layer, read replicas in each region serve local read traffic. Writes go to the primary region (us-east). If the primary region fails, global DNS health checks redirect write traffic to us-west within 60 seconds (DNS TTL = 60s). The read replicas may lag by up to 1 second — for read-after-write consistency, the application reads from the primary for 5 seconds after a write.

Every component exposes HTTP metrics: p50/p95/p99 latency, error rate by status code family (2xx, 4xx, 5xx), connection count, TLS handshake duration, and request queue depth. Alerts fire when p99 latency exceeds 500ms for 5 minutes or error rate exceeds 1%. We track the retry amplification factor (total requests / unique requests) as a signal of downstream health.

The infrastructure serves 100 million requests per day = approximately 1,157 req/s average. Peak is approximately 5,000 req/s (5x average). With each server handling 2,000 req/s (conservative for HTTPS), we need 3 servers at peak. With N+2 redundancy across 3 AZs, that is 9 servers. In practice, 12 servers with autoscaling."

**This answer would be strong for L6** — it shows multi-layer thinking, operational awareness, concrete metrics, and trade-off cognizance. It demonstrates that the candidate has built and operated HTTP infrastructure at scale.

### L7/L8 (Principal / Distinguished)

**Expected**: Everything from L6, plus: You can explain how HTTP evolution affects business strategy. You have influenced protocol-level decisions at your company. You can discuss the limitations of HTTP and where it should not be used (real-time streaming, IoT sensor data, high-frequency IPC). You have experience migrating large-scale traffic from one HTTP version to another, including the organizational challenges (multi-team coordination, monitoring rebuilds, phase planning). You can estimate the cost-benefit of protocol migrations in dollar terms.

A principal-level answer might include: "At my previous company, we migrated 100 million users from HTTP/2 to HTTP/3 over 18 months. The key challenge was not the protocol itself — it was that our monitoring and debugging tooling assumed TCP. We had to build QUIC-specific observability before we could confidently cut over. The migration reduced p95 latency by 35% on cellular but only 5% on fiber, so we targeted the rollout to mobile users first. We also discovered that our legacy IDS/IPS system could not inspect QUIC traffic, which required a security review and a temporary HTTP/2 fallback for enterprise customers behind certain firewalls. The total cost of the migration was $2 million in engineering time, but it saved $8 million per year in avoided lost mobile transactions."

## The FAANG Framework for Answering

When an HTTP question comes up, use this 6-step framework:

**Step 1 — Scope the Question (5 seconds)**
"Is this about the full request lifecycle, or a specific component like TLS or caching?"
Clarify whether the interviewer wants breadth or depth. A question like "Explain HTTPS" could mean a 2-minute or 20-minute answer. If the interviewer says "keep it high-level," give the 2-minute version. If they say "go deep on the security," spend 10 minutes on TLS.

**Step 2 — Layer the Answer (10 seconds)**
"I will walk through this from the top of the stack to the bottom: application, presentation, transport, internet."
Layering signals structured thinking. It tells the interviewer you have a mental model. An unstructured stream-of-consciousness answer ("First, the URL is typed, then, um, DNS happens, then there is a connection, then...") signals lack of clarity.

**Step 3 — Add Numbers at Every Layer (throughout)**
Do not say "DNS resolution takes some time." Say "DNS resolution takes 20-120ms for a cache miss, and 0ms for a cache hit with a TTL of 300 seconds."
Numbers make your answer memorable and signal real experience. Engineers who have operated HTTP infrastructure at scale know the numbers. Engineers who have only read about it give qualitative answers.

**Step 4 — Mention the Degraded Path (mid-answer)**
"Under normal conditions, this works. But if the DNS resolver is down, the browser falls back to the next nameserver. If the TLS certificate has expired, the browser shows a warning. If the server is overloaded, it returns 503."
Interviewers look for failure awareness. The candidate who only describes the happy path is perceived as junior. Every experienced engineer knows that production systems fail constantly. Showing that you anticipate failure is a senior signal.

**Step 5 — Show Trade-off Awareness (mid-answer)**
"HTTP/2 multiplexing solves head-of-line blocking at the application layer, but at the TCP layer, a lost packet still blocks all streams. That is why HTTP/3 over QUIC exists — it eliminates TCP-level HOL blocking. The trade-off is that QUIC runs over UDP, which some corporate firewalls block."
Every engineering decision has trade-offs. Explicitly naming them is a senior signal. Candidates who present choices without trade-offs (X is better than Y, full stop) are perceived as inexperienced.

**Step 6 — Land the Plane (10 seconds)**
"To summarize: HTTPS adds a TLS layer to HTTP, costing 1-3 RTT for handshakes and certificate management overhead, but providing encryption, integrity, and authentication that HTTP lacks. The choice depends on whether the data in transit justifies the overhead — for public APIs, always use HTTPS; for internal VPC traffic, evaluate case by case."
End with a concise summary. Do not trail off with "and, um, yeah, that is basically it." A strong closing leaves a lasting impression.

## Phrase Bank

**Opening the answer**:
- "Let me decompose that into layers: application, security, transport, and network."
- "I will walk through the happy path first, then cover failure modes for each step."
- "There are two perspectives on this — the client side and the server side. Let me start with the client."
- "I will anchor this with a concrete number: a typical HTTPS request across the US takes about 120ms at the 50th percentile."
- "Let me scope my answer: I will focus on the HTTPS request lifecycle, from URL entry to first byte of response."

**Explaining trade-offs**:
- "The trade-off is latency versus security. Each added layer of protection costs a round trip."
- "HTTP/2 multiplexing solves application-level HOL blocking but introduces TCP-level HOL blocking. That is why HTTP/3 exists."
- "TLS termination at the load balancer reduces backend CPU by 60-80% but means traffic inside the data center is unencrypted."
- "Keep-alive saves 2-3 RTT per subsequent request but consumes server memory for idle connections."
- "The cost of HTTP/2 over HTTP/1.1 is negligible on fast networks but significant on lossy networks without QUIC."
- "You pay for protocol features in latency, memory, or CPU. There is no free lunch — only the right trade-off for your traffic pattern."

**Describing failure scenarios**:
- "A retry storm amplifies load by the number of clients. With jittered exponential backoff, you spread retries across a time window."
- "Stale DNS is the most common cause of post-migration errors. Always lower TTL to 60s before a migration and keep old servers online for 2x the TTL window."
- "Connection pool exhaustion manifests as 503 errors that correlate with slow queries. The fix is not to increase the pool — it is to fix the slow queries."
- "TLS certificate expiry is a binary failure — either the certificate is valid and everything works, or it is expired and nothing works. Automated renewal is non-negotiable."
- "A 502 Bad Gateway means the load balancer cannot talk to the backend. This is almost always a network connectivity issue, not an application bug."
- "A 503 with no upstream response time data means the request never reached the application — backlog is full or health check is failing."

**Adding production perspective**:
- "At my previous company, we measured a 35% latency improvement for mobile users after migrating to HTTP/3."
- "We found that 80% of our TLS handshakes were redundant because the session cache was too small. Increasing it to 50,000 sessions raised the resumption rate from 60% to 95%."
- "The most expensive HTTP request is not the one that fails — it is the one that succeeds after retries, because you pay the cost N times and cannot tell it was a retry without distributed tracing."
- "In production, we found that HTTP/2 was actually slower than HTTP/1.1 for our API because our network had 2% packet loss from a misconfigured firewall."
- "We reduced our p99 latency from 800ms to 200ms by doing one thing: enabling HTTP/2 keep-alive with connection pooling."

**Ending the answer**:
- "In summary, the right HTTP configuration depends on your traffic pattern, network characteristics, and tolerance for complexity."
- "The key takeaway is that HTTP is not one protocol — it is a family of protocols with different performance characteristics. Choose based on your user's network conditions, not on what is newest."
- "To put it simply: use HTTPS everywhere you can, prefer HTTP/2 as the baseline, consider HTTP/3 for mobile-heavy traffic, and never treat HTTP as a black box."
- "The bottom line: HTTP looks simple on the surface — GET, POST, 200, 404 — but the behavior at scale is determined by the details: connection reuse, protocol version, timeout configuration, and failure handling."

## Red Flags

**Red Flag 1: "HTTP is stateless, so you do not need to worry about server state."**
Reality: HTTP is stateless at the protocol level, but your application layer (sessions, auth tokens, rate limit counters, TLS session caches) adds state. The protocol's statelessness does not make your infrastructure stateless. Load balancers maintain state for connection tracking. TLS session caches are state. Rate limiter counters are state. Claiming "no state" in a distributed system is a strong signal of inexperience.

**Red Flag 2: "HTTPS is always better than HTTP."**
Reality: For internal service-to-service communication in an isolated VPC over dedicated links, HTTPS adds latency and CPU overhead without meaningful security benefit. Evaluate case by case. The cost of TLS is approximately 15-35ms latency per hop and 5-15% CPU overhead on the server. If your internal network is isolated by security groups, firewalls, and private subnets, the additional encryption may not justify the cost.

**Red Flag 3: "HTTP/3 is always better than HTTP/2."**
Reality: HTTP/3 over QUIC uses UDP. Some corporate firewalls and legacy network equipment block or degrade UDP traffic at rates as high as 3-5% of connections. HTTP/3 also has less mature tooling and monitoring support. Tcpdump cannot capture QUIC payloads because they are encrypted with transport-layer keys that are not exposed to the kernel. QUIC-specific tools (qlog, qvis) are newer and less battle-tested than Wireshark for TCP.

**Red Flag 4: "Just increase the connection pool size to fix timeout errors."**
Reality: Connection pool exhaustion is a symptom, not a cause. Increasing the pool size masks the underlying issue and can make things worse by allowing more concurrent slow queries to accumulate, which increases database CPU and memory pressure, which makes queries slower, which worsens the original problem. The correct fix: identify the slow queries, optimize them, add indexes, or scale the database.

**Red Flag 5: "We use 200 OK for everything and put errors in the response body."**
Reality: This breaks every monitoring system, caching layer, and retry mechanism that relies on HTTP status codes. Your API looks healthy while silently failing. The load balancer sees 0% error rate, so no alerts fire. Caches see 200 and store the error response, serving it to future clients. Retry logic sees 200 and does not trigger. This anti-pattern has caused multi-hour outages at major companies where engineers thought "the API is healthy" because the error rates were zero.

**Red Flag 6: "DNS TTL of 86400 is fine because we rarely change our infrastructure."**
Reality: Infrastructure changes more often than you expect — DNS migration, load balancer swap, region failover, CDN reconfiguration. A 24-hour TTL means a day of degraded service for users with cached records. Every major incident postmortem at cloud-scale companies includes at least one mention of "DNS TTL was too long." Set TTL to 60-300 seconds for production records and lower to 60 seconds before any planned migration.

**Red Flag 7: "We do not need to handle 429 because we never rate-limit."**
Reality: If you do not handle 429, your clients will retry aggressively when you eventually need to rate-limit (during a traffic spike, attack, or upstream degradation). The retry storm will make the problem worse. Every HTTP client should handle all status codes gracefully: 2xx is success, 3xx is redirect (follow Location header), 4xx is client error (do not retry except for 429), 5xx is server error (retry with backoff).

**Red Flag 8: "We use a self-signed certificate because it is easier than getting a real one."**
Reality: Every client must either skip certificate verification (curl -k, verify=false) or add the self-signed cert as a pinned exception. The first approach makes your HTTPS effectively HTTP — anyone on the network can MITM. The second approach creates a maintenance burden: every new client, container, or service needs the cert installed. Let's Encrypt issues free certificates in under 30 seconds with ACME automation. There is no excuse for self-signed certs in production in 2026.

**Red Flag 9: "HTTP/2 is slower than HTTP/1.1 for our API, so we stuck with HTTP/1.1."**
Reality: If HTTP/2 is slower than HTTP/1.1 for your API, you likely have TCP packet loss. HTTP/2 multiplexes all streams over one TCP connection — a lost packet blocks all streams. HTTP/1.1 uses multiple TCP connections — only the connection with the lost packet is affected. The solution is to fix the packet loss (check NIC drivers, switch hardware, firewall rules, network congestion) or use HTTP/3 over QUIC (which eliminates TCP-level blocking), not to downgrade to HTTP/1.1.

**Red Flag 10: "We do not need to monitor TLS handshake time."**
Reality: A spike in TLS handshake time is often the first sign of a certificate about to expire, a cipher mismatch causing negotiation fallback, or a MITM device intercepting traffic. Monitoring TLS handshake time (p50 and p99) on every load balancer catches these issues before they become incidents.

## Practice Drills

### Drill 1 — 2 minutes: "Walk through the TLS 1.3 handshake"

**Prompt**: Explain what happens during a TLS 1.3 handshake. Include the messages exchanged, the cryptographic operations, and the latency cost.

**Key points to hit**: ClientHello with supported ciphers and key share. ServerHello with chosen cipher and its key share + certificate + signature. Client verifies cert chain, derives session key. Client sends Finished (encrypted) to confirm. 1-RTT total. 0-RTT with session resumption.

### Drill 2 — 2 minutes: "Explain HTTP status code families"

**Prompt**: List the HTTP status code families (1xx, 2xx, 3xx, 4xx, 5xx) and give two examples per family with a real scenario for each.

**Key points to hit**: 1xx informational (101 Switching Protocols for WebSocket upgrade), 2xx success (200 OK, 202 Accepted for async processing), 3xx redirection (301 Moved Permanently, 304 Not Modified for cache revalidation), 4xx client error (401 Unauthorized, 429 Too Many Requests), 5xx server error (502 Bad Gateway from upstream, 503 Service Unavailable for overload/maintenance).

### Drill 3 — 5 minutes: "Design an HTTP retry strategy"

**Prompt**: Design the retry strategy for a service that calls three downstream APIs with different characteristics: Payment API (POST, idempotency key required), User API (GET, idempotent), and Recommendation API (POST, not idempotent).

**Key points to hit**: Payment API — 3 retries with idempotency key, jittered exponential backoff 100ms/400ms/1.6s. User API — 5 retries, no special handling needed since GET is idempotent. Recommendation API — 1 retry (at most once), no idempotency, circuit breaker opens after 3 consecutive failures. All: handle 429 by reading Retry-After. Log every retry.

### Drill 4 — 10 minutes: "Debug intermittent 503 errors"

**Prompt**: Your team's API returns 503 errors intermittently — about 2% of requests fail. The errors come in bursts: 10 failures in 2 seconds, then 5 minutes of success, then another burst. No deploy happened recently. Walk through your debugging process.

**Key points to hit**: Check load balancer logs for correlation (same backend server? same path? same client IP?). Check database connection pool metrics (active connections vs max, wait time). Check upstream service latency (p99 increased from 50ms to 2s?). Check if a specific query is causing lock contention. Look at garbage collection logs on backend servers (stop-the-world pauses cause all requests to queue). Check if the burst correlates with another team's cron jobs or batch processes.

### Drill 5 — 10 minutes: "Design HTTPS for a global IoT platform"

**Prompt**: 50 million IoT devices (smart sensors, each sending 1KB of telemetry data every 5 minutes). Devices have limited memory (256KB RAM) and may be on lossy cellular networks. Design the HTTPS infrastructure. Protocol version? TLS configuration? Certificate management?

**Key points to hit**: HTTP/3 over QUIC (lossy cellular, connection migration, 0-RTT). TLS 1.3 only (1-RTT handshake, reduced CPU for crypto). Short-lived certificates (30-day validity) with automated renewal via ACME DNS-01 challenge. No HTTP/2 (too much memory for HPACK table on constrained devices). Keep-alive disabled (device mostly sleeping, reconnects on wakeup). Client certificate authentication for device identity. Hardware-backed TLS (ATECC608 secure element chip for key storage). Message size optimization: binary framing over HTTP/3, CBOR instead of JSON for payload.
