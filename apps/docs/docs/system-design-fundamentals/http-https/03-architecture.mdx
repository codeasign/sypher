## Naive Implementation

The simplest HTTP server possible: a single process listening on port 80, accepting one connection at a time, processing the request synchronously, and sending the response. This is the architecture of a Python BaseHTTPServer or a Node.js http.createServer without clustering.

```python
# Simplest possible HTTP server (Python)
from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(200)
        self.send_header('Content-Type', 'text/plain')
        self.end_headers()
        self.wfile.write(b'Hello, world!')

server = HTTPServer(('0.0.0.0', 80), Handler)
server.serve_forever()
```

**What breaks at 1,000 users (10 requests/second)**:
- Nothing. This server handles 10 req/s easily. Response time stays under 5ms.

**What breaks at 100,000 users (1,000 requests/second)**:
- **Thread exhaustion**: Each request creates a thread (or blocks the single thread). The server either hits the OS thread limit (typically 32,767 on Linux) or the Python GIL prevents concurrency. Requests queue up. Latency spikes from 5ms to 30 seconds.
- **Connection limits**: With keep-alive, each client holds a connection open. 100 clients x 6 connections each = 600 concurrent connections. Each connection ties up a thread. The server runs out of file descriptors (default ulimit -n = 1024 on many systems).
- **No TLS**: The naive server listens on port 80 (HTTP). Any sensitive data crosses the network in plaintext. On a shared network, any user can capture traffic with tcpdump.

**What breaks at 10,000,000 users (10,000 requests/second)**:
- **Single point of failure**: One server, one process, one network interface. If the server crashes, the entire service goes down.
- **No load balancing**: 10,000 req/s saturates a single machine's CPU and network bandwidth. A single gigabit NIC maxes out at approximately 1.2 million packets per second — beyond that, packet loss begins.
- **No connection pooling**: Every request opens a new TCP connection (or reuses a single serial connection). Throughput is capped by sequential processing.
- **No caching**: Every request hits the application code. No static asset serving from memory or CDN.
- **No rate limiting**: A single misbehaving client can send 10,000 req/s and degrade service for everyone.

### Layered Evolution

The naive single-server implementation evolves through distinct stages:

1. **Single server** → split into web server + application server + database (3-tier architecture)
2. **Add load balancer** → distribute traffic across multiple app servers (horizontal scaling)
3. **Add CDN** → offload static assets and cache dynamic content at edge
4. **Add API gateway** → centralize authentication, rate limiting, routing
5. **Add service mesh** → manage internal service-to-service HTTP traffic with mTLS, retries, circuit breaking
6. **Multi-region** → deploy in 2+ regions with global load balancing and active-active replication

## Integration Patterns

### Pattern 1: HTTP to Database (Read-Through Cache)

The application receives an HTTP GET request for frequently-read data. Instead of hitting the database on every request, it first checks the cache (Redis/Memcached). On cache hit, return cached data with a Cache-Control header. On cache miss, query the database, populate the cache, and return.

### Pattern 2: HTTP to Message Queue (Async Processing)

The application receives an HTTP POST request for a resource-intensive operation (image processing, email sending, report generation). Instead of blocking the HTTP response on the operation, it enqueues a message and returns 202 Accepted immediately.

```json
// Request
POST /api/process-image
Content-Type: application/json

{"image_url": "https://cdn.example.com/photos/raw/abc.jpg"}

// Response: 202 Accepted
{
  "job_id": "job-7a81f2c4",
  "status": "queued",
  "estimated_completion_seconds": 30
}
```

The worker pool reads from the queue, processes the image, and stores the result. The client polls `GET /api/jobs/{job_id}` to check status (returning 200 with status=complete when done, or 200 with status=pending while processing).

### Pattern 3: HTTP to External API (Gateway Aggregation)

The application receives an HTTP request that requires data from multiple external APIs. Instead of making the client orchestrate multiple calls, the server fans out requests to each external service, collects the responses, and returns a single aggregated response.

```python
# Gateway aggregation pattern
import asyncio
import aiohttp

async def get_user_dashboard(user_id: str) -> dict:
    async with aiohttp.ClientSession() as session:
        # Fan out requests concurrently
        tasks = [
            fetch(session, f"https://api.orders.com/users/{user_id}/recent"),
            fetch(session, f"https://api.recommendations.com/users/{user_id}"),
            fetch(session, f"https://api.notifications.com/users/{user_id}/unread"),
        ]
        recent_orders, recommendations, unread = await asyncio.gather(*tasks)
    
    return {
        "recent_orders": recent_orders,
        "recommendations": recommendations,
        "unread_count": unread["count"],
    }
```

### Pattern 4: HTTP to gRPC (Protocol Bridge)

Many internal services use gRPC (HTTP/2-based, binary protocol) instead of HTTP/1.1 JSON APIs for performance and strong typing. An API gateway or sidecar proxy translates between the external HTTP/JSON interface and the internal gRPC/Protobuf interface.

Feature | HTTP/1.1 JSON | gRPC Protobuf
--------|---------------|--------------
Serialization | Text (JSON), human-readable | Binary (Protobuf), machine-readable
Payload size | ~400 bytes for typical response | ~80 bytes (80% smaller)
Serialization speed | ~2 MB/s (json.dumps in Python) | ~50 MB/s (Protobuf in C++)
Schema enforcement | Optional, runtime errors | Compile-time enforced
Streaming | Server-Sent Events (manual) | Native bidirectional streaming

The translation happens at the gateway: it parses the JSON request, deserializes it into the gRPC request protobuf, sends the gRPC call, receives the protobuf response, and serializes it back to JSON.

**Performance impact of HTTP-to-gRPC bridging**:

| Step | HTTP/1.1 JSON (no bridge) | HTTP-to-gRPC bridge |
|------|---------------------------|---------------------|
| Client sends JSON | ~400 bytes | ~400 bytes |
| Gateway parses JSON | ~50μs | ~50μs |
| Gateway serializes to Protobuf | N/A | ~5μs |
| Internal transfer (gateway to backend) | HTTP/1.1 JSON ~400 bytes | gRPC Protobuf ~80 bytes |
| Backend deserializes | ~50μs | ~5μs |
| Backend processes | Same | Same |
| Backend serializes response | ~50μs | ~5μs |
| Internal transfer back | ~400 bytes JSON | ~80 bytes Protobuf |
| Gateway deserializes Protobuf | N/A | ~5μs |
| Gateway serializes to JSON | N/A | ~50μs |
| Total per-hop overhead | ~150μs + 800 bytes | ~120μs + 160 bytes |

The savings are modest per request but compound across a microservice architecture. In a chain of 5 services, the total bandwidth savings is 5x the per-hop savings. For an API handling 10,000 req/s, gRPC bridging saves approximately 6.4 MB/s of internal network bandwidth.

### Pattern 5: HTTP to CDN (Push Invalidation)

When an origin server updates a resource that is cached by a CDN, the origin must tell the CDN to invalidate the cached copy. This is an HTTP-specific integration pattern.

**Approaches**:

1. **Purge request**: The origin sends an HTTP PURGE request (custom method) to the CDN's cache purge API. The CDN removes the cached object. Next request for that URL fetches fresh content from origin.
   ```
   PURGE /v2/products/42 HTTP/1.1
   Host: cdn.example.com
   Authorization: Bearer <purge-token>
   ```
   Response: 200 OK (purge accepted) or 404 (object not in cache).

2. **Cache-Tag based invalidation**: The origin tags cached responses with Cache-Tag headers:
   ```
   Cache-Tag: product-42, category-electronics, region-us
   ```
   The origin can invalidate all objects matching a tag:
   ```
   POST /purge HTTP/1.1
   Host: cdn.example.com
   Content-Type: application/json
   
   {"tags": ["category-electronics"]}
   ```
   This invalidates every cached object tagged with "category-electronics" — useful when a bulk price update affects all electronics products.

3. **Surrogate-Key based**: Similar to Cache-Tag but with origin-controlled keys. The CDN stores the mapping of surrogate keys to cached objects. Invalidation is by key.

| Approach | Latency to invalidation | API calls per item | Use case |
|----------|------------------------|-------------------|----------|
| PURGE URL | ~100ms | 1 per URL | Individual item update |
| Cache-Tag | ~1s (batch processing) | 1 per tag, covers many items | Category-based update |
| TTL-based (wait for expiry) | TTL duration (e.g., 1 hour) | 0 | Low-priority content |

### Replication Strategy

- **Write path**: All writes go to the primary region (us-east). The database streams changes to replicas in other regions using asynchronous replication (PostgreSQL streaming replication, Aurora cross-region replication).
- **Read path**: Reads are served from the local region's replica. If the replica is behind (lag > 1 second), the application falls back to reading from the primary.
- **Failover**: If the primary region becomes unavailable, the global DNS health check detects the failure within 10 seconds. A secondary region is promoted to primary. DNS TTL must be low (60 seconds) for fast failover.

### Latency Planning

| Route | Geographic Distance | Round-Trip Time (RTT) |
|-------|---------------------|----------------------|
| Client → US-East | Local (within 500 km) | 10-20ms |
| Client → US-East | Cross-continent (e.g., LA → Virginia) | 60-80ms |
| EU → US-East | Transatlantic | 80-110ms |
| Asia → US-East | Transpacific | 140-200ms |
| Asia → EU | Asia-Europe | 150-250ms |

A user in Tokyo connecting to a US-East server experiences 140-200ms of network latency per HTTPS request. With 10 sequential requests for a page load, that is 1.4-2.0 seconds of pure network latency before any server processing. Multi-region deployment reduces this to 10-30ms (local RTT).

## Capacity Planning

### Math for 10 Million Daily Active Users (DAU)

**Assumptions**:
- 10 million DAU
- Each user makes 50 HTTP requests per day (page views + API calls)
- Peak hour = 10x the average hourly rate (80/20 rule — 20% of traffic in peak hour)
- Average response size: 4 KB (API JSON) + 50 KB (HTML + CSS + JS) = 54 KB
- 50th percentile response time target: under 100ms
- 99th percentile response time target: under 500ms

**Calculations**:

Requests per day: 10,000,000 users × 50 requests = 500,000,000 requests/day

Requests per second (average): 500,000,000 / 86,400 = 5,787 req/s

Requests per second (peak): 5,787 × 10 = 57,870 req/s

Data per second (average): 5,787 × 54 KB = 312 MB/s = 2.5 Gbps

Data per second (peak): 57,870 × 54 KB = 3.12 GB/s = 25 Gbps

**Server capacity**:

With a single modern server handling 20,000 req/s (HTTPS, optimized Nginx, 4 workers):
- Average load: 5,787 / 20,000 = 0.29 servers (round to 3 for redundancy)
- Peak load: 57,870 / 20,000 = 2.9 servers (round to 6 for redundancy + headroom)
- Actual deployment: 10 servers across 3 availability zones (N+2 redundancy)

**CPU budget**:
- Each request takes ~0.5ms of CPU time on a modern x86 core (TLS 1.3 with AES-NI)
- Peak CPU: 57,870 req/s × 0.5ms = 28,935 ms/s = 29 cores
- With 10 servers × 8 cores each = 80 cores available. CPU utilization at peak: 29/80 = 36%.

**Memory budget**:
- Each connection: ~30 KB (HTTPS keep-alive)
- Concurrent connections at peak: assume each user has 1 open connection during active use. Peak concurrent users = ~1 million (10% of DAU active simultaneously).
- Connection memory: 1,000,000 × 30 KB = 30 GB
- Per server (10 servers): 3 GB for connection state + 2 GB for application code + 1 GB for cache = 6 GB per server. Use 16 GB instances for headroom.

**Network budget**:
- Peak egress: 25 Gbps. With 10 servers, each needs 2.5 Gbps of network bandwidth. Use instances with 5 Gbps NICs (e.g., AWS c5.4xlarge).

**Key takeaway**: The bottleneck is not CPU or memory — it is network bandwidth and connection count. A server with 64 GB of RAM can handle 2 million connections if it has the network throughput to match.

## Deployment and Operations

### Zero-Downtime Deployments

HTTP has no built-in support for graceful upgrade — closing a connection terminates in-flight requests. Zero-downtime deployment requires orchestration:

1. **Health check endpoints**: The load balancer checks /health (liveness — is the process alive?) and /ready (readiness — is the server accepting traffic?). A server returning 5xx on /ready is removed from the load balancer's rotation.
2. **Graceful shutdown**: When a server receives SIGTERM (from Kubernetes or an orchestrator), it:
   - Stops accepting new connections (close the listen socket)
   - Drains existing connections (wait for in-flight requests to complete, up to a timeout of 30 seconds)
   - Sends a 503 to any new requests on keep-alive connections
   - Closes the process
3. **Rolling update**: New servers start, become healthy, receive traffic. Old servers drain and terminate. The load balancer never sees an all-down state.

### Monitoring Metrics

Every layer of the HTTP stack should be monitored:

| Metric | What It Tells You | Alert Threshold |
|--------|-------------------|-----------------|
| p50/p95/p99 latency | End-user experience | p99 > 1s for 5 minutes |
| Requests per second | Traffic volume | Drop > 50% in 1 minute (possible outage) |
| Error rate (5xx) | Server health | > 1% for 5 minutes |
| Error rate (4xx) | Client misbehavior | > 10% (usually indicates bad client or attack) |
| Active connections | Capacity utilization | > 90% of max for 10 minutes |
| Connection rate (new/s) | Connection churn | Sudden spike (possible SYN flood) |
| TLS handshake time | Certificate expiry or cipher overhead | > 200ms for 5 minutes (possible MITM or cert issue) |
| Cache hit ratio | Cache effectiveness | < 80% for CDN, < 90% for application cache |
| Upstream response time | Backend health | > 500ms (backend performance degradation) |

### Debugging HTTP Issues

**5xx errors on 0.1% of requests, not reproducible locally**:
Check the load balancer's access logs for correlation — specific User-Agent? Specific path? Specific response size? This often points to a backend that runs out of a connection pool for a specific service. Look for patterns in the X-Request-Id or X-Trace-Id headers — if all failing requests share a specific trace ID prefix, the failure is isolated to one backend instance or one request path.

**Slow response times, not correlated with CPU**:
Check connection queue depth on the load balancer. If request queuing exceeds 100ms, the issue is likely a bottleneck in a shared resource — database connection pool, external API rate limit, or TLS handshake on new connections. Use the load balancer's "request queue depth" metric. If it shows queuing, the bottleneck is upstream of the application servers. If it shows no queuing but latency is high, the bottleneck is inside the application servers (database queries, external API calls, garbage collection).

**Intermittent 502 Bad Gateway errors**:
The load balancer cannot establish a TCP connection to the backend. Possible causes: backend crashed and restarted, firewall rule changed, security group was modified, backend's listen backlog is full, or the backend's keep-alive timeout is shorter than the load balancer's idle timeout. Check the backend's /health endpoint directly, bypassing the load balancer. If /health returns 200, the issue is the load balancer's configuration or the network between LB and backend. If /health is unreachable, the backend process is down.

**TLS handshake failures with older clients**:
Check the server's ssl_protocols and ssl_ciphers configuration. If TLS 1.3 only is configured, clients on TLS 1.2 (Windows 7, Safari on iOS 12, some Android devices) cannot connect. Use TLSv1.2 and TLSv1.3 together. Also check that the certificate chain includes intermediate certificates — many servers send only the leaf certificate, leaving the client to fetch intermediates from AIA (Authority Information Access) URLs, which may fail if the client cannot reach those URLs.

**P99 latency spikes every hour at :00**:
This is almost always a cron job, batch process, or scheduled task that triggers at the top of the hour and consumes shared resources. Check cron jobs, database maintenance windows (VACUUM, ANALYZE, index rebuild), log rotation, and monitoring agent collection cycles. The fix is to jitter the start times: instead of starting every cron job at :00, spread them across :00 to :30 with random offsets within the hour.

**Requests from Asia are 10x slower than requests from US**:
The client is likely connecting to a US data center over trans-Pacific fiber, which has 140-200ms RTT. Each HTTP request requires at minimum 1 RTT (for the request-response pair), so each request takes 140-200ms of pure network latency. With 10 sequential requests for a page load (HTML, CSS, JS, 3 images, 4 API calls), total latency is 1.4-2.0 seconds before any server processing. Fix: deploy a CDN with edge PoPs in Asia, or deploy application servers in an Asian region (Singapore, Tokyo, Mumbai).

**HTTP/2 is slower than HTTP/1.1 for your API**:
This is counterintuitive but common on lossy networks. HTTP/2 multiplexes all streams over one TCP connection. A lost TCP packet blocks ALL streams. HTTP/1.1 uses 6-8 independent connections — only the connection with the lost packet is blocked. Check the server's TCP retransmission rate (netstat -s | grep retransmit). If it exceeds 1%, HTTP/2 may be slower than HTTP/1.1. Fix: reduce packet loss (check for faulty NICs, misconfigured switches, or congestion) or migrate to HTTP/3/QUIC which eliminates TCP-level head-of-line blocking.
