# Horizontal Scaling — Interview Playbook

## How This Topic Appears

Horizontal scaling questions appear in system design interviews in four categories.

### Category 1: Direct Questions

The interviewer explicitly asks about horizontal scaling.

- "How would you horizontally scale a stateless web service?"
- "Explain how you would shard a database serving 100 million users."
- "Design a system that auto-scales based on real-time metrics."
- "What happens when you add a new node to a distributed cache cluster?"
- "How do you handle session state when scaling from 1 to 100 application servers?"

### Category 2: Design Questions

Horizontal scaling is the implicit core of the design problem. The interviewer expects you to propose a horizontally scalable architecture.

- "Design a URL shortener like bit.ly."
- "Design a ride-sharing service like Uber."
- "Design a video streaming platform like YouTube."
- "Design a messaging system like WhatsApp."
- "Design a global e-commerce platform handling flash sales."

### Category 3: Debugging Questions

The interviewer describes a problem that occurs at scale and asks for root cause.

- "We added 10 more app servers but throughput only improved by 20%. Why?"
- "Our database shards are unbalanced — one shard holds 60% of the data. How do we fix it?"
- "After deploying a new version, cache hit rate dropped from 95% to 30%. What happened?"
- "Auto-scaling keeps adding and removing instances every 5 minutes. What is wrong?"
- "Cross-region read latency is 200ms for users in Asia even though the replica is in Singapore. What is the issue?"

### Category 4: Trade-off Questions

The interviewer presents a scenario and asks you to compare approaches.

- "Would you use vertical or horizontal scaling for a PostgreSQL database serving 50,000 QPS?"
- "Compare sharding by user_id modulo N vs consistent hashing. When would you pick each?"
- "Should we use sticky sessions or a shared session store?"
- "You have 10 database replicas. Read latency at P50 is 5ms but P99 is 2 seconds. Why?"

## What the Interviewer Is Actually Testing

Interviewers ask about horizontal scaling to evaluate specific competencies:

**System design depth**: Can you reason about trade-offs, not just list components? A candidate who says "I would add a load balancer and more servers" gets a follow-up question: "We did that, and throughput only improved by 30%. What now?"

**Operational awareness**: Do you know what breaks in production? Mentioning connection pooling, circuit breakers, and auto-scaling cooldown periods signals real-world experience.

**Failure mode knowledge**: Can you anticipate what goes wrong? A candidate who spontaneously says "but we need to worry about cache stampede during key expiration" demonstrates depth.

**Numbers sense**: Do your estimates make sense? Saying "we need 10 shards" without calculating QPS and data volume is a red flag.

**Communication clarity**: Can you explain a complex distributed interaction without hand-waving? The diagrams you draw and the step-by-step walkthrough matter more than the final design.

## Level-by-Level Expectations

### L3/L4 (Entry to Mid-Level)

**Expectation**: Understand the basic concept. Can explain stateless vs stateful. Can describe a simple architecture with a load balancer and multiple servers.

**Sample answer to "How would you horizontally scale a web service?"**:
"I would put the application behind a load balancer — an AWS ALB or HAProxy. The application code must be stateless: no session data stored locally, no local file system writes. Session state goes to Redis. Then I can add more application servers behind the load balancer. The load balancer distributes requests in round-robin fashion. If an instance fails, the load balancer's health check detects it and stops routing traffic to it."

**Red flags at this level**:
- Does not mention that the app must be stateless
- Proposes sticky sessions without explaining the trade-off
- Cannot explain what a health check is or how it works

### L5 (Senior Engineer)

**Expectation**: Deep understanding of coordination mechanisms. Can design a sharded database with consistent hashing. Can reason about auto-scaling strategies, connection pool sizing, and replication lag.

**Sample answer to "How would you scale a database beyond one server?"**:
"I would shard the database by user_id using consistent hashing. Consistent hashing minimizes rebalancing when shards are added or removed — only 1/N of keys move instead of all of them. Each shard has a primary for writes and 2-3 replicas for reads. The application layer handles read/write splitting: writes go to each shard's primary, reads go to one of the replicas. I need to monitor replication lag — if it exceeds 200ms, route reads to the primary for that user. For cross-shard queries, I avoid joins entirely — the application collects data from each shard in parallel and merges results."

**Red flags at this level**:
- Cannot calculate connection pool sizes given node count and database limits
- Does not mention replication lag as a concern
- Proposes cross-shard joins without discussing complexity
- Cannot explain the difference between modulo sharding and consistent hashing

### L6 (Staff Engineer)

**Expectation**: Can design a multi-region horizontally scaled system. Understands CAP theorem implications, conflict resolution strategies, and observability requirements. Can discuss organizational impact of scaling decisions.

**Sample answer to "Design a global horizontally scaled system"**:
"I would use an active-passive multi-region architecture with us-east as the primary write region and eu-west and ap-southeast as read replicas. Global DNS routes users to the nearest region via latency-based routing. Within each region, the application is stateless behind an ALB, auto-scaled based on request queue depth and CPU. The database is sharded across 8 nodes per region, with 2 replicas per shard. Cross-region replication is asynchronous — writes in us-east propagate to eu-west and ap-southeast with 50-150ms lag. For read-after-write consistency, a cookie stores the last write timestamp; any read from the same user within 1 second routes to the primary write region. I would choose Cassandra for the AP data (user profiles, activity feeds) and CockroachDB for CP data (payments, inventory). Each team owns end-to-end observability for their service — P50/P95/P99 latency, error budget, and cost per request are tracked in a shared dashboard."

**Red flags at this level**:
- Does not mention conflict resolution for active-active scenarios
- Ignores observability — no discussion of metrics, logging, or tracing
- Cannot articulate the trade-offs between CP and AP databases
- Proposes a design that would fail under the described failure scenario

### L7/L8 (Principal / Distinguished Engineer)

**Expectation**: Can design a system that has survived real production failures. Can articulate lessons learned from operating horizontally scaled systems. Understands the economic trade-offs at each scaling threshold.

**Sample answer to "Tell me about operating a horizontally scaled system at scale"**:
"The hardest lesson was that auto-scaling based on CPU is a lagging indicator. By the time CPU reaches 70%, the system is already degraded for minutes. We switched to a composite metric — request queue depth combined with P99 latency crossing 500ms. This triggers scale-out within 30 seconds instead of 5 minutes. We also learned that each scaling threshold introduces a new coordination bottleneck. At 10 nodes, health check traffic is negligible. At 200 nodes, the health check traffic from every node checking every destination is a significant fraction of our network budget. We switched to gossip-based health propagation. The organizational lesson is that a horizontally scaled system demands a platform team that owns the scaling infrastructure — you cannot expect each service team to independently manage auto-scaling rules, connection pool sizing, and circuit breaker thresholds."

**Red flags at this level**:
- Cannot cite specific production incidents and what was learned
- Does not address organizational or economic factors
- Proposes solutions without discussing operational cost

## The FAANG Framework for Answering

Use this 6-step framework for any horizontal scaling interview question.

### Step 1: Clarify Requirements (2 minutes)

- "How many users? What is the read-to-write ratio? What is the acceptable latency?"
- "What is the data model? Do we need ACID transactions?"
- "Are we designing for a greenfield system or migrating an existing one?"
- "What is the budget for infrastructure costs?"

### Step 2: Define Scale (2 minutes)

- "50 million DAU, 5 requests per user per day, 50:1 read-to-write ratio."
- "Peak throughput: 50M * 5 / 86,400 * 3 = approximately 8,680 RPS."
- "Data volume: 500 GB, growing at 50 GB/month."
- "Target P99 latency: 200ms."

### Step 3: Propose Base Architecture (3 minutes)

- "Stateless app servers behind a load balancer."
- "Read replicas for the database. Sharding if data exceeds 500 GB."
- "Cache layer (Redis) between app and database."
- "Queue (Kafka) for background processing."
- Draw the diagram as you speak.

### Step 4: Walk Through the Request Lifecycle (3 minutes)

- "A request arrives at the load balancer, which selects a healthy app server."
- "The app server checks Redis for cached data. Cache hit: return immediately."
- "Cache miss: query the correct database shard. The shard key is user_id hashed consistently."
- "Writes go to the shard's primary. Reads go to a replica."
- "Non-critical operations are enqueued to Kafka."
- "The response returns through the load balancer to the client."

### Step 5: Discuss Failure Modes and Mitigations (3 minutes)

- "If a shard fails: replicas are promoted. If no replica is available, the shard is unavailable for writes. Read traffic shifts to remaining replicas."
- "If the cache cluster partitions: each node serves the data it has. Stale reads are possible for up to the TTL window."
- "If auto-scaling lags behind a traffic spike: we have 30% buffer capacity and predictive scaling for known events."
- "If cross-region replication is slow: users read from the primary write region until replication catches up."

### Step 6: Identify Bottlenecks and Next Steps (2 minutes)

- "At 100,000 RPS, the database shards need to be rebalanced (hot shard)."
- "At 200 nodes, instance startup time becomes a bottleneck — we need faster container images."
- "At 1 million RPS, the load balancer itself needs horizontal scaling — we deploy multiple LBs behind a virtual IP."

## Red Flags

1. **Proposes sharding without discussing the shard key.** The shard key is the single most important decision in sharding. Not mentioning it means you have not thought about hot spots.

2. **Says "we just add more servers" without discussing statelessness.** If the application stores session data locally, adding servers breaks existing sessions. This reveals a gap in understanding the fundamental prerequisite.

3. **Ignores replication lag.** Every horizontally scaled database has replication lag. Not discussing it, or assuming writes are immediately visible on all replicas, indicates lack of production experience.

4. **Auto-scaling without cooldown.** Proposing auto-scaling rules that trigger on every metric spike without cooldown periods causes thrashing. This is a common mistake of candidates who have read about auto-scaling but never operated it.

5. **Single-region design for a global system.** If you design a system serving users worldwide in a single region, you have ignored 200ms+ latency for distant users. Multi-region is expected for global-scale questions.

6. **No circuit breakers.** If your architecture has service-to-service calls without circuit breakers, a single slow service cascades across the entire system. This is a red flag at L5 and above.

7. **Cannot calculate basic capacity.** If you cannot estimate how many servers, shards, or cache nodes are needed given RPS and data volume estimates, the interviewer loses confidence in your ability to make design decisions.

8. **Ignores monitoring and alerting.** A horizontally scaled system without observability is unmanageable. If you do not mention metrics, dashboards, or alerts, the interviewer knows you have not operated a system at scale.

9. **Proposes eventual consistency without discussing conflict resolution.** If the system can produce conflicting writes (two users updating the same record simultaneously), how are conflicts resolved? Last-write-wins loses data. CRDTs are complex. Not addressing this is a red flag.

10. **Too much complexity for the stated scale.** Proposing 50 database shards for 10,000 users is over-engineering. The interviewer is testing judgment, not the ability to design a Google-scale system.

## Practice Drills

### Drill 1: Database Sharding (5 minutes)

You have a 200 GB PostgreSQL database serving 10,000 QPS (9,000 reads, 1,000 writes). Design a sharding strategy. What shard key do you choose? How many shards? How many replicas per shard? What happens when you need to reshard?

### Drill 2: Cache Layer Design (5 minutes)

You have 50 app servers behind an ALB. Each server has a 1 GB local cache. A deployment replaces all servers with new AMIs. Cache hit rate drops from 90% to 0%. How long until the cache warms up? How would you reduce the impact?

### Drill 3: Auto-Scaling Tuning (5 minutes)

An auto-scaling group fluctuates between 10 and 50 instances every 10 minutes. The cooldown is 60 seconds. CPU averages 40% but spikes to 90% for 30 seconds every 5 minutes. What is wrong with the scaling rules? Propose corrected thresholds.

### Drill 4: Multi-Region Design (15 minutes)

Design a horizontally scaled system that serves users in North America, Europe, and Asia. Each region must have sub-100ms read latency. Writes can be async-replicated across regions. What consistency model do you choose? How do you handle a user who travels from the US to Europe and expects their data to follow them?

### Drill 5: Post-Incident Analysis (15 minutes)

Your horizontally scaled system experienced a P0 outage. At 14:00, traffic spiked from 5,000 to 50,000 RPS. At 14:03, auto-scaling triggered and launched 10 instances. At 14:07, the database connection pool on the primary shard hit 100%. At 14:08, all application servers started returning 503 errors. At 14:12, the on-call engineer identified the issue — a single endpoint querying an unindexed column was creating full table scans. At 14:15, the index was created. At 14:20, traffic was restored. Analyze the timeline. What would you change in the system to prevent a recurrence?
