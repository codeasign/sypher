# Horizontal Scaling — Trade-offs and Failure Modes

## The Core Tension

Horizontal scaling trades operational complexity for near-unbounded capacity.

Every server you add increases capacity by one node's worth, but it also adds one node's worth of coordination overhead — health checks, connection management, data replication, failure detection, and debugging surface area. At small scale (under 10 nodes), the overhead is negligible. At large scale (over 100 nodes), coordination consumes 5-15% of total system resources. At hyperscale (over 1,000 nodes), coordination can consume 30% or more without careful engineering.

## Decision Matrix

| Decision Point | Option A | Option B | Choose A When | Choose B When |
|---|---|---|---|---|
| Stateless vs Stateful | Externalize session to Redis | Sticky sessions on local server | You need linear scaling, fast auto-scaling, any-server-any-request | You cannot afford the 1-2ms Redis hop, or the app is legacy and cannot be refactored |
| Database sharding vs Replicas only | Split data across shards | Add read replicas to single primary | Write throughput exceeds one server, or data exceeds 500GB | Read traffic dominates (10:1+ ratio), data fits on one server |
| Synchronous vs Async replication | Every write confirmed on all replicas | Writes confirmed on primary, async to replicas | Data loss is unacceptable (financial transactions, orders) | Latency is more important than consistency (feeds, profiles, analytics) |
| Round robin vs Least connections LB | Distributes evenly by count | Distributes by active connections | Requests have uniform processing time | Requests vary in processing time (10ms to 10s range) |
| Auto-scaling metrics: CPU vs Queue depth | Scale on CPU utilization | Scale on request queue depth | CPU-bound workloads (compute, rendering, ML inference) | I/O-bound workloads (proxies, API gateways, database connectors) |
| Homogeneous vs Heterogeneous nodes | All nodes same instance type | Mix of instance types by workload | Simple operations, predictable capacity, any-node-any-request | Cost optimization — use spot instances for batch, on-demand for latency-critical |
| StatefulSet vs Deployment in K8s | Deployment with shared PVC | StatefulSet with local storage | Stateless app, easy scaling | Stateful app requiring stable network identity and persistent storage |
| Consistent hashing vs Range sharding | Keys distributed by hash ring | Keys grouped by contiguous ranges | Even distribution is critical, nodes are dynamic | Range queries are common, data has natural boundaries (dates, regions) |

**For horizontally scaled systems, the practical choice is usually AP** for non-critical data and CP for critical data. Running separate systems for different consistency requirements is common. Your user profiles database might be AP (DynamoDB, Cassandra) while your payments database is CP (PostgreSQL with synchronous replication).

### Partition Behavior

When a partition occurs in a horizontally scaled system:

```
Scenario: 6-node database cluster, network partition splits into 2 groups of 3

CP Configuration (quorum writes):
  Write requires 4/6 nodes to acknowledge (majority)
  Both groups have 3 nodes — neither can write
  Result: writes are rejected, reads return pre-partition data
  Impact: service is unavailable for writes until partition heals

AP Configuration (last-write-wins):
  Each node accepts writes independently
  Both groups accept writes
  Result: data diverges across groups
  Impact: reads return inconsistent data until reconciliation
```

## Failure Modes

### Failure 1: Cascading Connection Exhaustion

**What happens**: One database shard becomes slow (full GC pause, slow query spilling to disk). Application server connection pools for that shard fill up as queries take longer. Thread pools on the app servers fill up waiting for database connections. Other shards remain healthy, but their connection pool threads are starved because the app server's thread pool is saturated by the slow shard.

**Blast radius**: All services that share the thread pool with the affected query path.

**Detection**: Thread pool utilization above 90%, database connection pool above 95% for one shard, P99 latency spike.

**Recovery**: Circuit breaker on the database connection pool. Open after 10 consecutive timeouts. This frees thread pool capacity for other shards. Requests to the broken shard fail fast (50ms instead of 30-second timeout).

### Failure 2: Cache Stampede

**What happens**: A popular cache key expires simultaneously across all nodes. All 50 app servers try to recompute the value at the same time, each issuing a database query. The database receives 50 concurrent queries for the same data in under 100ms. If the recomputation is expensive (aggregation query taking 2 seconds), the database is overwhelmed.

**Blast radius**: The database, all queries sharing that database node.

**Detection**: Sudden drop in cache hit rate from 95% to under 10% for a specific key pattern. Database CPU spikes to 100%.

**Recovery**:
1. **Mutex/penetration locking**: Only one server recomputes the cache. Others wait and use the recomputed value.
2. **Stale-while-revalidate**: Serve the stale cached value while recomputing in the background.
3. **Jittered TTLs**: Add random jitter to TTL values (300s + random(0, 60)) to prevent simultaneous expiration.

### Failure 3: Thundering Herd on Restart

**What happens**: A deployment restarts all 50 app servers simultaneously. Each server starts cold — no local cache, no warm connection pools. Each server opens connections to the database, Redis, and Kafka at the same instant. The database sees 50 new connections arrive within seconds. Redis sees 50 simultaneous connection bursts. The result is a brief resource exhaustion at every dependency.

**Blast radius**: Database, Redis, Kafka — all dependencies.

**Detection**: Connection spikes on database and Redis at deployment time. Brief latency surge.

**Recovery**: Rolling deployments with staggered restarts. Kubernetes `maxSurge: 25%` and `maxUnavailable: 25%` ensures only a fraction of nodes restart at once. Connection pool warm-up (pre-establish connections on startup before accepting traffic).

### Failure 4: Inefficient Auto-Scaling

**What happens**: An auto-scaling rule triggers scale-out when CPU exceeds 70% for 5 minutes. Traffic spikes from 10,000 to 50,000 RPS in 2 minutes, triggered by a viral marketing campaign. By the time the auto-scaler detects the threshold (5 minutes) and new instances initialize (8 minutes), 13 minutes have passed. The existing nodes have been at 100% CPU for 11 minutes. Errors cascade.

**Blast radius**: All traffic to the affected tier.

**Detection**: Auto-scaling event logged, but instances launch after the spike has already caused errors.

**Recovery**:
1. Predictive auto-scaling (AWS Predictive Scaling, GCP Predictive Autoscaler)
2. Scheduled scaling for known events (product launches, marketing campaigns)
3. Buffer capacity: keep 30% spare capacity during normal operation
4. Multi-metric scaling: scale on request queue depth (instantaneous) not CPU (lagging)

### Failure 5: Cross-Region Replication Lag

**What happens**: A user in Japan writes data to the us-east primary (150ms write latency). They immediately refresh the page, which reads from the ap-southeast replica. The write has not replicated yet (replication lag: 200ms). The user sees stale data — their write appears lost.

**Blast radius**: Single user experience, but can affect many users simultaneously if a region-wide replication pipeline is slow.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Scatter-Gather Without Bounding

The application queries all shards in parallel for every request instead of routing by shard key.

```
SELECT * FROM orders — routes to all 8 shards in parallel
8 parallel queries, then merge results in app memory
```

At 100 RPS, this generates 800 shard queries per second. At 10,000 RPS, it generates 80,000 shard queries — far beyond what any shard cluster can sustain. The fix is to always include the shard key in queries.

### Anti-Pattern 2: Over-Sharding

Creating 1,000 shards for a 50 GB database. Each shard holds 50 MB of data. The overhead of 1,000 connections, 1,000 connection pools, and 1,000 monitoring dashboards far exceeds the benefit of sharding.

A good rule: each shard should hold at least 10 GB of data or handle at least 500 QPS. If your shards are smaller, you are paying coordination overhead for no benefit.

### Anti-Pattern 3: Synchronous Everything

Every request handler calls 5 internal services synchronously: the user service, the order service, the inventory service, the payment service, and the notification service. Total response time is the sum of all 5 calls. If one is slow, the entire request is slow.

Fix: Identify which calls must be synchronous (the response depends on them) and which can be async (enqueue to Kafka, return immediately). The key is that horizontal scaling exposes these latency chains — fixing them requires both architectural changes and timeouts with fallbacks.

### Anti-Pattern 4: Ignoring Data Locality

App servers in us-east querying a database shard in ap-southeast for every request. The 150ms network round-trip per query means 150ms of unavoidable latency, regardless of how fast the database is.

Fix: Co-locate application and database servers in the same region and preferably the same availability zone. Cross-AZ latency is 1-2ms. Cross-region latency is 50-200ms.

### Anti-Pattern 5: Manual Rebalancing

When a shard runs out of space, a DBA manually moves data to a new shard over a weekend. The migration takes 48 hours, requires application downtime, and risks data loss if the migration script has a bug.

Fix: Automated rebalancing (Cassandra, CockroachDB, MongoDB Atlas) or pre-split shards with room for growth.

## When NOT to Use Horizontal Scaling

Horizontal scaling is not always the answer. Consider these scenarios:

1. **Data fits on one machine**: A 10 GB database serving 500 QPS does not need sharding. Vertical scaling (bigger instance) costs less and is simpler.

2. **Strong consistency is legally mandated**: Financial systems that require ACID transactions across all data. Horizontal scaling's eventual consistency models may violate compliance requirements.

3. **Complex joins and referential integrity**: A normalized schema with 50 tables and foreign key constraints across all of them is extremely difficult to shard. Every join becomes a scatter-gather operation across shards.

4. **Team size is under 5**: Operating a distributed system requires on-call rotation, monitoring infrastructure, and deep debugging skills. A single PostgreSQL instance with read replicas serves most teams at this size.

5. **Variable but predictable traffic under 10,000 RPS**: A well-tuned vertical stack handles this easily. The complexity of horizontal scaling does not pay for itself at this scale.

6. **Legacy application with thread-local state**: Refactoring a monolithic Java EE application with thread-local session state to be stateless takes months. The business case must justify that investment.
