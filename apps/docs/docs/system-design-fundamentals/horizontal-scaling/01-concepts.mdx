# Understanding Horizontal Scaling

## The Ticketmaster Meltdown

On February 25, 2025, Ticketmaster opened presale for Taylor Swift's Eras Tour. Within 30 minutes, 14 million fans attempted to buy tickets. The system processed 3.5 billion bot requests alongside legitimate traffic. Verified fan queues stalled. Users saw spinning loaders for 45 minutes before receiving "An error occurred" messages. The website crashed entirely for 2 hours. By the time service resumed, 80% of presale inventory was held in abandoned carts — phantom reservations that locked out real buyers.

Ticketmaster's system was vertically scaled: massive servers in a single Virginia data center with Oracle databases on $50,000/hour hardware. When the traffic wave hit, every request hit the same bottleneck — the monolithic authentication server, the single inventory database, the one payment pipeline. There was no way to distribute the load because the architecture was never designed to spread across machines.

The Department of Justice investigation that followed cited "insufficient capacity distribution" as a root cause. No single machine on Earth, regardless of cost, could handle 14 million concurrent users.

This is the horizontal scaling problem. You can buy the biggest server on the market — 448 vCPUs, 24 TB of RAM, $100,000 per month — and still hit a ceiling when traffic surges past a million concurrent users. The only way past that ceiling is to spread the work across many machines.

### The Supermarket Checkout Analogy

A single-server system is a grocery store with one checkout lane. On a normal Tuesday, one lane handles 30 customers per hour just fine. On Thanksgiving week, 1,000 customers arrive per hour. The single lane backs up into the aisles. Customers abandon their carts in the cereal aisle.

You have two options. Option one: install a faster POS terminal at the existing lane. Scan items 2x faster, process payments quicker. This is vertical scaling — you get 60 customers per hour, but you still cap out. Option two: open 20 checkout lanes. Each lane handles 30 customers per hour. Total capacity: 600 customers per hour. This is horizontal scaling. But now you need a greeter directing people to the shortest line (load balancer), a policy that any cashier can handle any customer (stateless design), and a way to open and close lanes without shutting down the store (rolling deployment).

A supermarket chain with 500 stores, regional distribution centers, and dynamic staffing based on local demand patterns — that is a horizontally scaled retail system. Each store is a node. The distribution center is a shared state layer. The staffing algorithm is auto-scaling. And the sign directing customers to the shortest checkout lane is a least-connections load balancer.

## Core Vocabulary

**Horizontal scaling (scale out)**: Adding more machines to a system to distribute workload. Contrast with vertical scaling (scale up), which adds resources to a single machine. Horizontal scaling has no theoretical capacity ceiling — you can always add another node. Practical limits come from coordination overhead: each new node adds communication cost proportional to O(log n) in well-designed systems, or O(n) in poorly designed ones.

**Load balancer**: A traffic distribution node that sits between clients and servers. Accepts incoming requests and forwards them to a backend server according to a scheduling policy — round robin, least connections, IP hash, or weighted distribution. Must itself be horizontally scaled (multiple load balancer nodes behind a virtual IP with VRRP/keepalived) to avoid becoming a single point of failure.

**Stateless**: A server design where no session data or user state is stored locally. Every request contains all information needed to process it (authentication token, request body, context). Stateless servers are trivially horizontally scalable — any server can handle any request. Session state is externalized to a shared cache like Redis or a database.

**Stateful**: A server design where session data is stored locally in memory. Requests from the same client must route to the same server (sticky sessions). Horizontal scaling is harder because adding or removing servers requires session migration. WebSocket servers, game servers, and some legacy enterprise applications are inherently stateful.

**Sharding**: Partitioning a database across multiple machines by a shard key. Each shard holds a subset of rows. The shard key determines which server holds which data. A poor shard key creates hot spots — one shard handling 80% of traffic while others sit idle. Consistent hashing distributes keys more evenly and minimizes rebalancing when nodes change.

**Replication**: Copying data from one database server to others. Primary handles writes; replicas handle reads. Asynchronous replication is fast (millisecond latency) but risks data loss. Synchronous replication guarantees no data loss but adds latency proportional to the distance between nodes. PostgreSQL streaming replication, MySQL Group Replication, and MongoDB replica sets are common implementations.

**Auto-scaling**: Dynamically adding or removing server instances based on real-time metrics — CPU utilization, memory pressure, request queue depth, or custom business metrics. AWS Auto Scaling Groups, GCP Managed Instance Groups, and Kubernetes Horizontal Pod Autoscaler implement this pattern. Cooldown periods prevent thrashing (rapid start-stop cycles).

**Connection pool**: A set of pre-established database connections maintained by the application, avoiding TCP handshake overhead per request. Pool size is a critical tuning parameter: too small causes request queuing, too large exhausts database connection limits. PostgreSQL uses approximately 2 MB RAM per connection, so a pool of 100 connections consumes 200 MB before executing a single query.

**Backpressure**: A flow control mechanism where a downstream service signals upstream to slow down when it cannot keep up. HTTP 429 (Too Many Requests), TCP window scaling, and Kafka's `max.in.flight.requests.per.connection` are backpressure implementations. Without backpressure, overwhelmed services crash under cascading failures.

**Idempotency**: The property that an operation can be applied multiple times without changing the result beyond the first application. Critical for horizontal scaling because retries are inevitable — if a load balancer retries a failed request, the second attempt must not create duplicate orders, deduct balances twice, or send duplicate emails. Idempotency keys, conditional updates, and upsert operations are common patterns.

**Leader election**: A process by which distributed nodes select one node to coordinate writes, assign work, or manage shared state. If the leader fails (detected via heartbeat timeout), remaining nodes elect a new leader. Raft and Paxos are consensus algorithms that implement leader election. ZooKeeper, etcd, and Consul provide production-ready leader election services.

**Split brain**: A failure mode where a cluster splits into two groups that each believe they are the primary. Both groups accept writes independently, leading to data divergence that may never reconcile. Prevented by quorum-based decisions (majority vote), fencing (the old leader is forcibly shut down before promoting the new one), and STONITH (Shoot The Other Node In The Head).

Every layer in this diagram can scale independently. The load balancer tier scales by adding more LB nodes behind a virtual IP. The app tier scales by adding instances to the auto-scaling group. When traffic to one layer grows, you scale only that layer.

## How Horizontal Scaling Actually Works

Horizontal scaling is not a single mechanism. It is a composition of several subsystems working together. Here is the step-by-step path a request takes through a horizontally scaled system.

### Step 1: DNS Resolution

A user types `api.example.com` into their browser. The DNS resolver returns not a single IP address, but one of several IP addresses corresponding to the closest regional load balancer. AWS Route53 latency-based routing, Cloudflare's geolocation DNS, and Google Cloud DNS all implement this. The user is routed to us-east-1, eu-west-1, or ap-southeast-1 based on their origin IP.

### Step 2: Load Balancer Receives Request

The regional load balancer (AWS ALB, GCP HTTP LB, or HAProxy) receives the HTTPS request. It terminates the TLS connection, parses the HTTP headers, and selects a backend server from its healthy pool.

### Step 3: Health Check and Server Selection

The load balancer maintains a list of healthy backend servers. Health checks run every 5-15 seconds, pinging `/health` endpoints. Servers that fail 3 consecutive checks are removed from the pool. The scheduling algorithm (round robin for equal-capacity servers, least connections for variable-length requests) selects one healthy server.

### Step 4: Application Server Processes Request

The selected application server receives the request. Since the server is stateless, it can process any request from any user. It extracts the authentication token from the Authorization header and validates it against the shared session store (Redis).

### Step 5: Cache Lookup

Before querying the database, the application checks the distributed cache (Redis cluster). If the data is in cache (cache hit), it returns in under 1 millisecond. If not (cache miss), it queries the database.

### Step 6: Database Query with Shard Routing

If a database query is needed, the application routes it to the correct database shard. The shard key (typically `user_id % number_of_shards` or consistent hashing of `user_id`) determines which shard holds the data.

### Step 7: Write to Primary, Read from Replica

Write operations (INSERT, UPDATE, DELETE) go to the shard's primary database. Read operations go to one of the shard's replicas. The application layer handles read/write splitting using separate connection pools.

### Step 8: Background Job Enqueueing

Non-critical operations — email notifications, image processing, analytics — are enqueued to a message queue (Kafka, RabbitMQ, Amazon SQS) instead of being processed synchronously. The request returns to the client with a 200 status. Background workers consume the queue and process each job.

### Step 9: Response Assembly and Return

The application server assembles the response from cache, database, and any internal service calls. The response traverses back through the load balancer to the client. The entire round trip, under normal conditions, completes in under 100ms.

### Step 10: Monitoring and Scaling Feedback

Metrics from this request — latency, database query time, cache hit rate — are collected by the observability stack (Prometheus, Grafana, Datadog). If aggregate metrics cross thresholds (CPU above 70% for 5 minutes, P99 latency above 500ms), the auto-scaler provisions new instances. If metrics drop, it de-provisions.

## Variants and Flavors

### Stateless Horizontal Scaling

The simplest and most common variant. Application servers hold no local state. Any server can handle any request. Scale by adding servers behind a load balancer.

**When to use:**
- REST APIs and HTTP microservices
- Containerized applications on Kubernetes
- Serverless functions (Lambda, Cloud Functions)
- Any service where the request carries its own context (JWT tokens, request-scoped data)

**When NOT to use:**
- Applications with large in-memory caches that take minutes to warm (each new server starts cold)
- Real-time multiplayer game servers (each game session is a stateful WebSocket connection)
- Legacy enterprise applications with thread-local session storage
- File upload processing that writes to local disk

Servers maintain local state. Requests from the same client must route to the same server. Scaling requires session migration or rebalancing.

**When to use:**
- WebSocket servers (chat, real-time collaboration)
- Game servers (each match exists on one server)

- Legacy applications that cannot be refactored to be stateless

**When NOT to use:**
- Any new system (always prefer stateless design)
- Systems with unpredictable traffic patterns (migration cost spikes during scale events)
- Services that need rapid auto-scaling (session migration takes 30-60 seconds per node)

**When to use:**
- Database exceeds 500 GB (single-node backup takes hours)
- Write throughput exceeds one server's capacity (over 5,000 writes/second)
- Working set exceeds available RAM (caching alone cannot solve it)
- Compliance requires data to stay within specific geographic boundaries (data sovereignty)

**When NOT to use:**
- Data fits on one server (sharding adds complexity for no benefit)
- Queries frequently need cross-shard joins (application must join in memory)
- The data model has complex referential integrity constraints (foreign keys are cross-shard)
- Team lacks operational experience with distributed databases

## Mental Model Check

1. You have 10 app servers behind a round-robin load balancer. Each server takes 200ms per request. The database handles 5,000 QPS. You add 10 more app servers. Throughput increases by 15%. What is the bottleneck?

2. A sharded database distributes users by `user_id % 8`. The shard holding users whose IDs end in 0 or 1 handles 40% of all traffic because your payment-integration partner routes all their users through a single API key that hashes to shard 0. How do you fix the hot spot without moving all data?

3. Your auto-scaling rule adds instances when CPU exceeds 70%. Each instance takes 8 minutes to initialize. Traffic spikes from 1,000 RPS to 50,000 RPS in 2 minutes. How many users see errors before the new instances come online?

4. An application uses sticky sessions for a shopping cart. A deployment restarts 50% of the servers. What happens to users whose session was on a restarted server? How would you prevent this?

5. A cache cluster has 6 nodes with 2 replicas each. A network partition splits the cluster into two groups of 3 nodes each. Which nodes can the application still read from? Which nodes can it write to?

The answers to these questions determine whether your mental model of horizontal scaling matches reality. The gap between what you expect and what happens is where the learning lives. This understanding deepens in the next section, which examines the internal mechanisms — connection pooling, consistent hashing, replication protocols, and the detailed request lifecycle — that make horizontal scaling work under the hood.
