### Saga Execution Coordinator

The coordinator is the brain of the saga runtime. It manages the lifecycle of each saga instance from creation through completion or compensation.

**Saga Instance Pool**: The coordinator maintains an in-memory pool of active saga instances, each with its own state, step sequence, and compensation stack. When a new saga request arrives, the coordinator creates an instance, assigns it a unique saga ID, and registers it in the pool.

**Step Executor**: This component handles the actual remote calls to saga participants. It constructs the request, sends it over the chosen transport (HTTP or gRPC), and enforces timeout and retry policies. The step executor wraps each call in a circuit breaker to handle service failures gracefully.

**Decision Engine**: The decision engine interprets the saga definition — the ordered list of steps and their compensation actions. After each step completes, the decision engine checks the result and determines the next action: proceed to the next step, retry the current step, or begin compensation.

**Compensation Manager**: When a step fails and the decision engine chooses compensation, the compensation manager builds a reverse-ordered list of completed steps and executes their compensation requests in sequence. It tracks which compensations succeeded and which failed, enabling retry or escalation.

### Saga Log

The saga log is the source of truth for every saga instance. It is stored in a durable database (PostgreSQL, DynamoDB, or similar) and survives coordinator crashes.

The log uses three tables:
- **saga_instances**: Tracks the overall status of each saga instance. The coordinator reads this table on startup to find sagas that need recovery.
- **saga_steps**: Records every forward step, its status, and the idempotency key used for the request. This enables the coordinator to skip steps that were already executed during crash recovery.
- **saga_compensations**: Records every compensation attempt, including retry counts. A compensation with status RETRYING indicates the saga is in an incomplete compensated state and needs ongoing attention.

**State transitions in detail:**

1. **PENDING**: The saga request was received and persisted. The coordinator has not yet begun execution.
2. **ACTIVE**: The coordinator is executing forward steps. Steps are recorded in saga_steps as they complete.
3. **COMPLETED**: All forward steps executed successfully. The saga is done.
4. **FAILED**: A forward step failed. The coordinator is about to begin compensation.
5. **COMPENSATING**: Compensation actions are being executed in reverse order.
6. **COMPENSATED**: All compensation actions completed successfully. The system is back to a consistent state.
7. **COMPENSATION_FAILED**: One or more compensation actions could not be completed after exhausting retries. Manual intervention required.

### Detailed Timeline

**Step 1 — Saga Creation (T+0ms):** The customer submits the order. The coordinator creates a saga instance with ID "ord_789" and status PENDING. The saga is persisted to the database immediately so it survives a coordinator crash.

**Step 2 — Pickup (T+5ms):** The coordinator reads the PENDING saga from the database. It transitions the status to ACTIVE and begins executing step 1.

**Step 3 — Reserve Inventory (T+10ms):** The coordinator sends a "reserve item" request to the Inventory Service with idempotency key "ord_789_inv_1". The request includes a timeout of 5 seconds. The coordinator inserts a saga_steps record with step_1 = IN_PROGRESS.

**Step 4 — Inventory Success (T+310ms):** The Inventory Service responds with success. The coordinator updates saga_steps to step_1 = COMPLETED and moves to step 2.

**Step 5 — Process Payment (T+315ms):** The coordinator sends a "charge card" request to the Payment Service with idempotency key "ord_789_pay_2". This involves an external payment gateway, so the timeout is 10 seconds.

**Step 6 — Payment Success (T+720ms):** The Payment Service returns success. The coordinator updates step_2 = COMPLETED.

**Step 7 — Create Shipping Label (T+725ms):** The coordinator calls the Shipping Service with idempotency key "ord_789_shp_3". Timeout is 5 seconds.

**Step 8 — Shipping Failure (T+1,050ms):** The Shipping Service reports that the shipping address is invalid. The coordinator updates step_3 = FAILED.

**Step 9 — Begin Compensation (T+1,055ms):** The coordinator transitions the saga to FAILED, then immediately to COMPENSATING. It reads the saga_steps table to find completed steps: step_1 and step_2. It reverses the order: compensate step_2 first, then step_1.

**Step 10 — Void Payment (T+1,060ms):** The coordinator sends a "void charge" request to the Payment Service with the original charge ID and a compensation-specific idempotency key "ord_789_comp_pay_2".

**Step 11 — Void Success (T+1,460ms):** The Payment Service confirms the void. The coordinator updates comp_2 = COMPLETED.

**Step 12 — Release Inventory (T+1,465ms):** The coordinator sends a "release item" request to the Inventory Service with compensation idempotency key "ord_789_comp_inv_1".

**Step 13 — Release Success (T+1,760ms):** The Inventory Service confirms the release. The coordinator updates comp_1 = COMPLETED.

**Step 14 — Saga Compensated (T+1,765ms):** All compensations succeeded. The coordinator transitions the saga to COMPENSATED status. The coordinator notifies the customer: "Order could not be processed. No charges were made."

## Consistency: The Isolation Problem

The saga pattern guarantees atomicity through compensation, but it does not provide isolation. This is the most important limitation to understand.

### The Dirty Read Problem

After step 1 of the travel booking saga commits, the flight reservation is visible. The saga might later compensate and cancel the reservation. Between step 1 completing and compensation starting, another part of the system can see the reservation and act on it.

**Concrete example:**
1. Saga A books flight AA-1234 (step 1 commits).
2. Saga A books hotel (step 2 commits).
3. Saga B queries available flights. It sees AA-1234 as booked and skips it.
4. Saga A's step 3 fails. Saga A compensates: cancels AA-1234.
5. Now AA-1234 is available again, but Saga B already chose a different flight because it believed AA-1234 was unavailable.

Saga A's intermediate state was visible to Saga B, and Saga B made a decision based on that state. This is a dirty read at the saga level.

### Countermeasures

#### Countermeasure 1: Semantic Lock

Each saga participant holds a "lock" on the resource by setting a flag that other sagas can see. The flag communicates: "this resource is part of an active saga and may be released."

```python
# Semantic lock: saga_holding indicates the resource is tentatively reserved
def reserve_inventory(item_id, quantity, saga_id):
    result = db.execute("""
        UPDATE inventory
        SET quantity_held = quantity_held + %s,
            saga_holding = %s
        WHERE item_id = %s
          AND available_quantity - quantity_held >= %s
    """, (quantity, saga_id, item_id, quantity))

    if result.rowcount == 0:
        raise InsufficientInventoryError()

# Other sagas see the held quantity and adjust their behavior
# The holding saga_id allows the compensation to find and release its hold
```

**How it helps:** Other sagas see the held inventory and do not double-book it. If the saga compensates, the holding is released. If the saga completes, the hold is converted to a confirmed booking.

#### Countermeasure 2: Commutative Updates

Design operations so that the order of execution does not affect the final result. This is especially important for compensation actions that may overlap with forward operations from other sagas.

```python
# Non-commutative: order matters
def add_item_surcharge(order_id, amount):
    # Adding a surcharge after calculating the total changes the total
    order.total += amount

# Commutative: order does not matter
def add_item_to_order(order_id, item_id, quantity):
    # Adding items to an order is commutative with other add_item operations
    # The final order is the same regardless of add order
    order.items.append((item_id, quantity))
```

**How it helps:** If two sagas both modify the same resource, commutative operations ensure the final state is correct regardless of which saga commits or compensates.

#### Countermeasure 3: Pessimistic View

A saga participant re-reads the current state before making a decision, rather than trusting its earlier read. This prevents stale-state decisions.

```python
def process_order_with_pessimistic_view(saga_id, order_id):
    # Step 1: Read current state
    current_state = get_order_state(order_id)

    # Step 2: Validate saga can proceed
    if current_state.get("version", 0) != saga.expected_version:
        raise ConcurrentModificationError("Order was modified by another saga")

    # Step 3: Execute the step with a version check
    result = db.execute("""
        UPDATE orders
        SET status = 'confirmed', version = version + 1
        WHERE order_id = %s AND version = %s
    """, (order_id, saga.expected_version))

    if result.rowcount == 0:
        raise OptimisticLockError()
```

**How it helps:** If another saga modified the order since the first saga read it, the first saga detects the conflict and can re-evaluate its decision.

#### Countermeasure 4: Reread Value

Before performing a compensation action, reread the current value to ensure the compensation is still appropriate. The resource may have been modified by another saga since the original forward operation.

```python
def compensate_hotel_booking(booking_id, saga_id):
    # Reread current status before compensating
    booking = get_booking(booking_id)

    if booking.status == "CANCELLED":
        # Already cancelled by the hotel system or another saga
        # Compensation is a no-op — idempotency ensures safety
        log.info(f"Booking {booking_id} already cancelled. Compensation skipped.")
        return {"status": "ALREADY_COMPENSATED"}

    if booking.status == "CHECKED_IN":
        # Guest already checked in — compensation cannot cancel
        # This is a saga failure that requires manual handling
        raise CompensationNotPossibleError(
            f"Cannot cancel booking {booking_id}: guest checked in"
        )

    # Proceed with cancellation
    result = cancel_booking(booking_id)
    return result
```

**How it helps:** Checking the current value prevents compensations from acting on stale information, avoiding scenarios where a compensation undoes a change that was intentionally modified by another saga.

## Performance: Latency Analysis

The choice between choreography and orchestration has direct consequences on end-to-end latency.

### Choreography Latency

In a choreographed saga, each step emits an event that the next service consumes asynchronously. The latency of each step is the service's processing time plus event bus propagation time.

```
Step 1: Flight booking          = 300 ms processing + 50 ms event publish = 350 ms
Step 2: Hotel booking            = 400 ms processing + 50 ms event pub    = 450 ms
Step 3: Car rental               = 250 ms processing + 50 ms event pub    = 300 ms
Step 4: Payment                  = 800 ms processing                      = 800 ms

Total end-to-end latency: ~1,900 ms
```

The services run in parallel for event consumption — the Flight Service does not wait for the Hotel Service. But each step must finish before the next event is published, so the total latency is sequential.

### Orchestration Latency

In an orchestrated saga, all requests route through the coordinator. Each request incurs an additional network hop.

```
Step 1: Coordinator -> Flight Service               = 2 ms route + 300 ms = 302 ms
Step 2: Coordinator -> Hotel Service                 = 2 ms route + 400 ms = 402 ms
Step 3: Coordinator -> Car Rental Service            = 2 ms route + 250 ms = 252 ms
Step 4: Coordinator -> Payment Service               = 2 ms route + 800 ms = 802 ms

Coordinator saga log writes (sequential, 4 writes):  = 4 x 5 ms             =  20 ms

Total end-to-end latency: ~1,778 ms
```

Orchestration is slightly faster than choreography because there is no event bus propagation. However, the coordinator becomes a bottleneck under high throughput.

### Throughput

- **Choreography**: Event bus handles load distribution. Throughput is limited by the slowest service and the event bus capacity.
- **Orchestration**: The coordinator becomes the bottleneck. A single coordinator can handle approximately 5,000-10,000 saga executions per second on moderate hardware. Beyond that, the coordinator must be sharded or replicated.

## Configuration

### Retry Policies

Every saga implementation must configure retry behavior for both forward and compensation steps.

```yaml
retry:
  forward_steps:
    max_attempts: 3
    initial_interval: 100ms
    max_interval: 5s
    multiplier: 2.0
  compensation_steps:
    max_attempts: 5
    initial_interval: 500ms
    max_interval: 30s
    multiplier: 2.0
  escalation:
    max_compensation_failures: 10
    alert_channel: "saga-failures"
```

Configure these values based on the failure characteristics of each service:

- **Fast-failing services** (validation errors, business rule violations): Do not retry. These failures are deterministic and will fail again.
- **Transient failures** (timeout, 503 Service Unavailable, network error): Retry with exponential backoff. Most transient failures resolve within a few seconds.
- **Rate-limited services** (429 Too Many Requests): Retry with backoff that respects the Retry-After header.

### Timeout Configuration

```yaml
timeout:
  saga_total: 60s          # Max total time for the entire saga
  step_timeout: 10s        # Max time for a single step
  compensation_timeout: 15s # Max time for a single compensation action
  inactivity_timeout: 300s  # Saga with no progress is considered stuck
```

Configure step timeout based on the service's p99 latency. If the Payment Service takes 3 seconds at p99, set the step timeout to 5 seconds. The saga total timeout should be the sum of all step timeouts plus overhead for the coordinator's internal processing.

### Saga Log Persistence

The saga log must be durable to survive coordinator crashes. Choose the database based on throughput and durability requirements.

```sql
-- PostgreSQL schema for saga log (production)
CREATE TABLE saga_instances (
    saga_id        VARCHAR(64) PRIMARY KEY,
    saga_type      VARCHAR(128) NOT NULL,
    status         VARCHAR(32) NOT NULL DEFAULT 'PENDING',
    payload        JSONB,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE saga_steps (
    saga_id         VARCHAR(64) NOT NULL REFERENCES saga_instances(saga_id),
    seq             SMALLINT NOT NULL,
    service_name    VARCHAR(128) NOT NULL,
    action          VARCHAR(64) NOT NULL,
    status          VARCHAR(32) NOT NULL DEFAULT 'PENDING',
    idempotency_key VARCHAR(256) NOT NULL,
    response        JSONB,
    attempted_at    TIMESTAMPTZ,
    PRIMARY KEY (saga_id, seq)
);

CREATE TABLE saga_compensations (
    saga_id         VARCHAR(64) NOT NULL REFERENCES saga_instances(saga_id),
    step_seq        SMALLINT NOT NULL,
    service_name    VARCHAR(128) NOT NULL,
    action          VARCHAR(64) NOT NULL,
    status          VARCHAR(32) NOT NULL DEFAULT 'PENDING',

    last_error      TEXT,
    PRIMARY KEY (saga_id, step_seq)
);
```

For high-throughput scenarios, use DynamoDB or Cassandra instead of PostgreSQL to avoid write contention on the saga_instances table. The saga log schema is designed for point lookups by saga_id — the access pattern is read-heavy during compensation and write-heavy during execution.

### Saga Log Maintenance

The saga log grows continuously. Without maintenance, it becomes a performance and storage problem.

**Data retention:** Completed and compensated sagas can be archived or deleted after a retention period. The typical retention is 90 days for active sagas and 1 year for archived sagas. Audit and compliance requirements may extend the retention to 7 years for financial systems.

```sql
-- Archive sagas older than 90 days
INSERT INTO saga_instances_archive
SELECT * FROM saga_instances
WHERE updated_at < NOW() - INTERVAL '90 days'
  AND status IN ('COMPLETED', 'COMPENSATED');

DELETE FROM saga_instances
WHERE updated_at < NOW() - INTERVAL '90 days'
  AND status IN ('COMPLETED', 'COMPENSATED');
```

**Partitioning strategy:** For PostgreSQL, partition the saga_steps table by month:


### Throughput Formulas

The maximum throughput of an orchestrated saga system is bound by three factors: saga log database write capacity, coordinator CPU capacity, and participant latency.

**Database-bound throughput:**
```
max_throughput = db_write_iops / writes_per_step / steps_per_saga

Example:
  5,000 IOPS provisioned (DynamoDB)
  / 3 writes per step (step_start, step_complete, saga_update)
  / 4 steps per saga
  = 416 sagas/second
```

**Coordinator-bound throughput:**
```
coordinator_capacity = cpu_cores * (1_000 / ms_per_step_processing)

Example:
  8 CPU cores
  * (1,000 / 2ms per step processing)
  = 4,000 steps/second
  / 4 steps per saga
  = 1,000 sagas/second
```

In practice, the database is almost always the bottleneck. Optimize by batching saga log writes, using async writes for non-critical state transitions, and partitioning the saga log across multiple database instances.

### Monitoring the Saga Log

Monitor these metrics on the saga log to detect performance degradation before it affects users:

- **Saga log write latency (p50, p99)**: Sudden increases indicate database contention or IOPS exhaustion.
- **Active saga count by status**: A growing number of ACTIVE sagas that are not progressing indicates stuck sagas.
- **Saga completion rate**: Drop in completion rate signals systemic issues in forward steps.
- **Compensation rate**: Increase in compensation rate signals failures in later steps of the saga.
- **Saga age (seconds in ACTIVE state)**: Sagas that exceed their expected duration may be stuck.
- **Saga log storage growth rate**: Predicts when storage will need expansion or archival.

```python
def metrics_check():
    # Alert if any saga has been ACTIVE for more than 5 minutes
    stuck_sagas = db.query("""
        SELECT COUNT(*) FROM saga_instances
        WHERE status = 'ACTIVE'
          AND updated_at < NOW() - INTERVAL '5 minutes'
    """)
    if stuck_sagas > 0:
        alert("STUCK_SAGAS", f"{stuck_sagas} sagas stuck in ACTIVE state")

    # Alert if compensation rate exceeds 5% in the last minute
    recent = db.query("""
        SELECT
            COUNT(*) FILTER (WHERE status = 'COMPLETED') as completed,
            COUNT(*) FILTER (WHERE status = 'COMPENSATED') as compensated
        FROM saga_instances
        WHERE updated_at > NOW() - INTERVAL '1 minute'
    """)
    total = recent.completed + recent.compensated
    if total > 0 and recent.compensated / total > 0.05:
        alert("HIGH_COMPENSATION_RATE",
              f"Compensation rate: {recent.compensated / total:.1%}")
```

### The Step Execution Loop

The coordinator's step executor runs a tight loop for each step:

```python
def execute_step(saga, step_def):
    step_db_id = persist_step_start(saga.saga_id, step_def.seq)

    for attempt in range(1, step_def.max_retries + 1):
        try:
            idempotency_key = make_key(saga.saga_id, step_def.seq, attempt)

            # Publish command to participant's queue
            queue.publish(
                exchange="saga.commands",
                routing_key=step_def.service,
                message={
                    "saga_id": saga.saga_id,
                    "step_seq": step_def.seq,
                    "action": step_def.action,
                    "payload": step_def.payload,
                    "idempotency_key": idempotency_key,
                    "attempt": attempt
                }
            )

            # Wait for response (blocking or callback-based)
            response = wait_for_event(saga.saga_id, step_def.seq, timeout=step_def.timeout_ms)

            if response["status"] == "COMPLETED":
                persist_step_complete(saga.saga_id, step_def.seq, response["payload"])
                saga.log(f"Step {step_def.seq} completed on attempt {attempt}")
                return StepResult.SUCCESS
            else:
                # Business logic failure (validation, insufficient funds, etc.)
                # Do not retry — the failure is deterministic
                persist_step_failed(saga.saga_id, step_def.seq, response.get("error"))
                return StepResult.FAILED

        except TimeoutError:
            saga.log(f"Step {step_def.seq} timeout on attempt {attempt}")
            if attempt < step_def.max_retries:
                # Schedule retry with exponential backoff
                backoff = min(
                    step_def.initial_interval_ms * (2 ** (attempt - 1)),
                    step_def.max_interval_ms
                )
                schedule_retry(saga.saga_id, step_def.seq, delay_ms=backoff)
                return StepResult.RETRY_SCHEDULED
            else:
                persist_step_failed(saga.saga_id, step_def.seq, "max retries exceeded")
                return StepResult.FAILED

        except ConnectionError:
            saga.log(f"Step {step_def.seq} connection error on attempt {attempt}")
            # Network errors are transient — always retry
            if attempt < step_def.max_retries:
                backoff = min(
                    step_def.initial_interval_ms * (2 ** (attempt - 1)),
                    step_def.max_interval_ms
                )
                schedule_retry(saga.saga_id, step_def.seq, delay_ms=backoff)
                return StepResult.RETRY_SCHEDULED
            else:
                persist_step_failed(saga.saga_id, step_def.seq, "connection failed")
                return StepResult.FAILED

    return StepResult.FAILED
```

This loop handles three distinct failure modes: business logic failures (no retry), timeouts (retry with backoff), and connection errors (retry with backoff). Each retry carries a new idempotency key that includes the attempt number, so the participant can distinguish first-attempt processing from retry processing.
