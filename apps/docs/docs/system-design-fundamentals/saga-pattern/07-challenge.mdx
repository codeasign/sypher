# Saga Pattern — Challenges

Each challenge maps to a specific skill: saga compensation design, failure recovery, and production-scale architecture. Foundation challenges take 5-10 minutes. Design challenges take 15-20 minutes. Expert challenges take 30-45 minutes.

---

## Tier 1: Foundation

These challenges test whether the core concepts are solid.

### Challenge 1.1: Identify the Compensation Gap

A developer designs a two-step saga for a food delivery app:

1. **Create order** — Insert a row in the orders table, decrease restaurant's remaining capacity.
2. **Assign driver** — Call a third-party dispatch API to find a driver. The third-party does not provide a cancellation API.

The saga begins. Step 1 succeeds. Step 2 succeeds. One hour later, the customer cancels. The restaurant's capacity was decremented but never restored because there is no "undo create order" in the system. The restaurant cannot accept new orders until an operator manually restores the capacity.

**Question:** What is missing and how would you fix it?

**Model answer:** The saga is missing a compensating transaction for step 1. The create order step must have a corresponding cancel order action. Without it, the restaurant's capacity cannot be restored. The compensation should be: when a driver assignment fails or the customer cancels, call an "cancel order" API that restores the restaurant's capacity. Even though the driver dispatch API does not support cancellation, the order itself can always be cancelled. The driver assignment step compensation can be a no-op (log the event for manual follow-up) if cancellation is not supported, but the order step must always be compensatable.

### Challenge 1.2: Choreography vs Orchestration — Which Fits?

A startup is building a video processing pipeline with three services:
1. **Transcode Service** — Convert uploaded video to multiple resolutions.
2. **Thumbnail Service** — Generate thumbnail images from the video.
3. **CDN Upload Service** — Upload the processed files to the content delivery network.

The pipeline is linear: transcode must finish before thumbnails can be generated, and thumbnails must finish before CDN upload. If any step fails, all work must be discarded.

**Question:** Would you use choreography or orchestration for this pipeline? Justify your answer.

**Model answer:** Use orchestration. The pipeline is linear and has exactly three steps with a strict ordering requirement. An orchestrator can manage the flow with a simple state machine. The compansations are straightforward: CDN upload failure means delete transcoded files and thumbnails. An orchestrator makes the failure path explicit and the state visible. Choreography would add event-driven complexity with no benefit — the pipeline has no branches, no parallel steps, and no need for services to be added independently.

### Challenge 1.3: Idempotency Key Design

An inventory saga has a step: "reserve item X, quantity 5." The step is idempotent — it checks for the idempotency key before executing. But the idempotency key is constructed as: `{order_id}_{item_id}_{quantity}`.

A customer orders item X, quantity 5. The saga starts. The inventory step executes with key "ord_123_item_x_5". Then the customer modifies the order to item X, quantity 3. The customer submits the change, which creates a new saga with key "ord_123_item_x_3". The inventory service sees a different key and reserves 3 more units of item X. Now 8 units are reserved for a single order.

**Question:** What is wrong with the idempotency key design?

**Model answer:** The idempotency key includes the quantity, which changes when the order is modified. The key should be based on identifiers that do not change across the saga's lifetime: `{order_id}_{item_id}`. If the quantity changes, the new reservation should use the same key, and the inventory service should update the existing reservation rather than creating a new one. A better design: the idempotency key is tied to the order line item, not the quantity. The inventory service uses the key to upsert the reservation — if it exists, update the quantity; if not, create it.

## Tier 2: Design

These challenges test the ability to design real-world saga systems.

### Challenge 2.1: Hotel Cancellation with Partial Compensation

A hotel booking saga has three steps:
1. Book room at Hotel A (non-refundable deposit of $50 charged immediately).
2. Book connecting flight.
3. Book rental car.

Step 1 succeeds and charges $50. Step 2 succeeds. Step 3 fails. The saga compensates.

**Problem:** The hotel has a strict non-refundable policy for deposits. The compensation can cancel the room reservation and make the room available again, but the $50 deposit was already charged and the hotel will not refund it.

**Question:** Design a solution that handles this partial compensation scenario without the platform losing money.

**Model answer options:**

**Option A — Charge the deposit at the end of the saga.** Move the deposit charge from step 1 to after all bookings are confirmed. Step 1 only reserves the room without charging. The deposit is charged as a separate final step after the flight and car are confirmed. This requires the hotel to support "hold room without payment" reservations.

**Option B — Platform insurance model.** Accept that the $50 is lost on failed sagas and treat it as cost of business. Model this as an expected loss rate. If 2% of sagas fail and the average order value is $500, the platform absorbs 2% of $50 deposits = $1 per successful booking. Build this into the pricing.

**Option C — Require cancellation insurance.** Before the deposit step, offer the customer optional cancellation insurance ($5). If the saga fails, the insurance covers the $50 deposit. This shifts the cost to the customer who chooses to opt in.

### Challenge 2.2: Two Coordinators, One Saga

Your team runs two saga coordinator instances behind a load balancer. An order saga is being processed by coordinator A. Coordinator A's network interface experiences a transient failure. The load balancer fails the health check and routes traffic away from A. The request from step 1 comes back to coordinator B, which does not know about the saga.

**Question:** How do you handle this scenario? Design a solution that prevents coordinator B from rejecting the step response.

**Model answer:** Saga state is stored in the saga log database, not in coordinator memory. Both coordinator A and coordinator B read from and write to the same saga log. When coordinator B receives the step response, it looks up the saga instance by saga_id in the saga log. It sees the current state and processes the response accordingly. The coordinator is stateless — any instance can handle any saga step response.

The key requirement: all coordinator instances must share the same saga log and the same message queue. The queue routes commands and events to any available coordinator without affinity. This is already the standard pattern for stateless HTTP services and message consumers.

### Challenge 2.3: API Gateway as Saga Trigger

Your system has an API gateway that receives 1,000 orders per second at peak. Each order triggers a saga with 4 steps. The saga coordinator is a single process.

**Question:** The API gateway cannot wait 2 seconds for each saga to complete before responding to the client. How do you design the flow from the API gateway to the saga coordinator?

**Model answer:** The API gateway does not call the saga coordinator synchronously. Instead:

1. The API gateway receives the order, validates it, and publishes a "NewOrder" event to a message queue.
2. The saga coordinator subscribes to "NewOrder" events and processes them asynchronously.
3. The API gateway immediately responds to the client with: "Order received. Your order ID is ord_789. You will receive a confirmation shortly."
4. The saga coordinator picks up the event, creates a saga instance, and begins execution.
5. When the saga completes, the coordinator publishes a "OrderCompleted" event. A notification service picks up this event and sends an email or push notification to the customer.

This decouples the synchronous HTTP request from the asynchronous saga execution. The client receives a response in under 50ms. The saga processes in the background.

### Challenge 3.1: The Zombie Saga Problem

**Question:** Design a mechanism that prevents coordinator A from executing step 2 at T+8.

**Model answer:** Use a **generation number** or **fencing token** on the saga instance:

```sql
-- Saga instance table with generation number
CREATE TABLE saga_instances (
    saga_id          VARCHAR(64) PRIMARY KEY,
    status           VARCHAR(32),
    generation       INT NOT NULL DEFAULT 1,
    owner            VARCHAR(128)
);

-- Each time a coordinator acquires a saga, it increments the generation
UPDATE saga_instances
SET owner = 'coordinator_B',
    generation = generation + 1
WHERE saga_id = 'saga_001'
  AND generation = (SELECT generation FROM saga_instances WHERE saga_id = 'saga_001');
```

When coordinator B picks up the saga, it increments the generation from 1 to 2. When coordinator A wakes up and tries to execute step 2, it must write to the saga log. The write includes the generation it believes is current (1). The saga log rejects the write because the current generation is 2:

```sql
UPDATE saga_steps
SET status = 'COMPLETED'
WHERE saga_id = 'saga_001'
  AND seq = 2
  AND generation = 1;
-- Rowcount = 0: rejected because generation is stale
```

Coordinator A checks the rowcount. Zero rows updated means its state is stale. Coordinator A re-reads the saga from the log, sees that saga_001 is already COMPLETED, and stops executing.

### Challenge 3.2: Non-Deterministic Compensation

A payment saga has three steps:
1. Reserve funds in currency A (USD).
2. Convert currency A to currency B (EUR) at exchange rate 1 USD = 0.92 EUR.
3. Transfer funds to recipient in currency B.

Step 3 fails because the recipient's bank is unreachable. The saga must compensate steps 1 and 2.

**Problem:** Between step 2 and compensation, the exchange rate changed from 1 USD = 0.92 EUR to 1 USD = 0.85 EUR. If the compensation for step 2 converts EUR back to USD at the new rate, the customer receives fewer USD than they originally reserved. If the compensation uses the original rate, the platform absorbs the exchange rate loss.

**Question:** How should the compensation handle the exchange rate change?

**Model answer:** Never let the compensation be subject to market rates. The currency conversion step should not convert at the current market rate. Instead:

**Option A — Lock the exchange rate:** When the currency conversion is performed, record the locked rate as part of the step's output. The compensation uses the same locked rate so the reversal is guaranteed to return the exact original amount. The booking platform absorbs the difference between the locked rate and the current market rate as a cost of doing business.

```json
{
  "step_2_output": {
    "original_amount_usd": 1000.00,
    "converted_amount_eur": 920.00,
    "locked_rate": 0.9200,
    "rate_valid_until": "2026-07-02T15:30:00Z"
  },
  "step_2_compensation": {
    "reverse_amount_eur": 920.00,
    "reverse_rate": 0.9200,
    "returned_amount_usd": 1000.00
  }
}
```

**Option B — Hold rather than convert:** Instead of converting, reserve USD. Leave the funds in USD until the saga is fully confirmed. Convert only after all steps succeed. This eliminates the exchange rate risk during the saga but delays the conversion, potentially affecting the customer's experience if they expected immediate conversion.

**Option C — Accept market loss:** Treat the currency difference as an expected cost. If the rate moves from 0.92 to 0.85, the platform loses $70 per $1,000 on compensation. At a 1% saga failure rate, that is about 0.7% cost per successful transaction. Model this as a known expense.

### Challenge 3.3: Long-Running Saga with Human Steps

A loan application saga includes a step that requires human approval: "Underwriter reviews application and approves or rejects." This step can take 3-5 business days. The steps before it (credit check, document verification) complete in minutes. The steps after it (fund disbursement, notification) must only execute after approval.

**Question:** How do you model this human step in a saga? What happens if the loan is approved and then disbursement fails one day later — how does the compensation work?

**Model answer:** Model the human review step as a saga step with an extended timeout. When the saga reaches the review step, it publishes a task to a human workflow system (like a task queue or BPMN engine). An underwriter claims the task, reviews the application, and submits the decision (approve or reject) through an admin UI. The submitted decision generates an event that the saga coordinator processes.

Timeout configuration:
```yaml
steps:
  - name: credit_check
    timeout: 30s
  - name: document_verification
    timeout: 60s
  - name: underwriting_review
    timeout: 120h  # 5 business days
    polling_interval: 5m  # check status every 5 minutes
  - name: fund_disbursement
    timeout: 120s
```

If the underwriter approves and the fund disbursement fails 24 hours later, the compensation is complex:

- Step 3 (underwriting) compensation: The underwriter must manually reverse the approval. This is a compensation that requires another human action.
- Steps 1 and 2 compensations: These can be automatic (credit check records can be marked as void, document verification can be invalidated).

The key insight: when a saga has human steps, the compensation may also require human involvement. The system should generate a reversal task in the human workflow system. The underwriter sees both the original approval task and the compensation task with the same application ID. This is a legitimate use case for COMPENSATION_FAILED with manual resolution.

### Challenge 3.4: The At-Least-Once Delivery Problem

A saga coordinator publishes a "reserve inventory" command to a message queue. The queue delivers the message to the Inventory Service. The Inventory Service processes the reservation, writes to its database, and publishes a "step completed" event. The event queue delivers the event to the coordinator. The coordinator updates the saga log.

Now consider this scenario: the Inventory Service writes to its database and publishes the event, but the database write succeeds while the event publish fails. The Inventory Service has a reservation in its database but no event was sent to the coordinator. The coordinator's timeout fires. The coordinator marks the step as FAILED and begins compensation. The compensation sends a "release inventory" command. The Inventory Service releases the inventory that was reserved. Now the inventory is double-released — the reservation was undone by the timeout, and then the compensation also undid it.

**Question:** Design a solution that prevents the inventory from being double-released.

**Model answer:** The root cause is that the Inventory Service's database write and event publish are not atomic. Use the **Outbox pattern** to solve this:

1. The Inventory Service, instead of directly publishing the event, writes the event to an outbox table in its own database, within the same local transaction as the reservation.
2. A separate process (the Outbox Relay) reads the outbox table and publishes events to the event queue.
3. The Outbox Relay deletes or marks the outbox record after successful publish.

```sql
-- Outbox table: same database as the inventory
CREATE TABLE outbox (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_id    VARCHAR(64) NOT NULL,
    event_type      VARCHAR(128) NOT NULL,
    payload         JSONB NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    published_at    TIMESTAMPTZ
);

-- Inventory Service: single transaction for reservation + outbox write
BEGIN;
  UPDATE inventory SET quantity_held = quantity_held + 1 WHERE item_id = 'X';
  INSERT INTO outbox (aggregate_id, event_type, payload)
  VALUES ('ord_789', 'inventory.reserved', '{"item_id": "X", "quantity": 1}');
COMMIT;
```

Now the reservation and the event record are atomic. If the database write succeeds, the outbox record exists. The Outbox Relay will eventually publish it. If the database write fails, nothing changes. The coordinator's timeout may still fire, but when the compensation arrives, the Inventory Service checks the idempotency key. If the reservation was already released by a previous compensation, the release is a no-op.

### Challenge 3.5: Partial Saga Completion with Idempotency Key Collision

A saga has five steps. Steps 1-3 succeed. Step 4 fails. The coordinator begins compensating in reverse order: step 3, step 2, step 1.

The compensation for step 3 uses the same API endpoint as the forward step but with a different action parameter: the forward step sends `{"action": "book", "resource_id": "room_101"}` and the compensation sends `{"action": "cancel", "resource_id": "room_101"}`.

The idempotency key for the forward step is `saga_001_step_3_attempt_1`. The compensation sends a new idempotency key: `saga_001_comp_step_3_attempt_1`.

The Hotel Service checks idempotency by looking up the key. It sees that `saga_001_step_3_attempt_1` was already processed (the forward step) and returns a cached response: "booking confirmed." The compensation is never executed.

**Question:** What is the bug and how do you fix it?

**Model answer:** The bug is that the idempotency key namespace is shared between forward steps and compensation steps. The Hotel Service sees the compensation key and does not realize it is a different operation.

Fix: include the operation type in the idempotency key so that forward and compensation keys are in distinct namespaces.

```
Forward key:  saga_001_forward_step_3_attempt_1
Comp key:     saga_001_compensation_step_3_attempt_1

Or use a prefix:
FWD_001_3_1  vs  COMP_001_3_1
```

Alternatively, the participant can check both the key and the action type. The Hotel Service stores the action type alongside the idempotency key:

```sql
CREATE TABLE idempotency_keys (
    key             VARCHAR(256) PRIMARY KEY,
    action_type     VARCHAR(64) NOT NULL,  -- 'book' or 'cancel'
    response        JSONB,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

When processing a compensation request, the participant checks: "Is there a cached response for this key AND is the action type 'cancel'?" If the key exists but the action type is 'book' (forward step), the participant does not use the cached response — it executes the cancel operation.

### Challenge 3.6: The Cascading Saga Problem

A payment processing system has two sagas that operate on the same user account:

- **Saga A**: Deposit $500 → Exchange USD to EUR → Transfer to external account.
- **Saga B**: Internal transfer $200 from savings to checking.

Saga A starts first. Step 1 (deposit $500) succeeds. Saga B starts. Step 1 of Saga B (internal transfer $200) checks the user's balance: it sees the original balance plus $500 (the deposit from Saga A's step 1, which is committed). Saga B completes successfully.

Saga A's step 2 (exchange USD to EUR) fails. Saga A compensates step 1: removes the $500 deposit.

Now the user's balance is back to the original. But Saga B's internal transfer of $200 was based on the balance that included the $500 deposit. The user effectively received $200 from the internal transfer that was based on funds that were later removed.

**Question:** How do you design the system to prevent this cascading inconsistency?

**Model answer:** This is a classic saga isolation problem. Several countermeasures apply:

**Option A — Pessimistic locking on the user account:** Saga A acquires a lock on the user's account when it starts. Saga B cannot read or modify the account until Saga A completes or compensates. This provides isolation but reduces concurrency.

```python
def acquire_account_lock(account_id, saga_id):
    lock = redis.lock(f"account_lock:{account_id}", timeout=30, blocking_timeout=5)
    if not lock:
        raise SagaLockError(f"Cannot acquire lock on account {account_id}")
    return lock
```

**Option B — Version-based optimistic locking:** Each account has a version number. When Saga A deposits $500, it updates the version. When Saga B reads the account, it notes the version. When Saga B's internal transfer commits, it checks that the version has not changed. If Saga A compensates and changes the version, Saga B's commit fails and must be retried.

**Option C — Saga isolation with balance snapshots:** Each saga reads the account balance at the start and records the snapshot. Compensation actions restore the balance to the snapshot value, not to a computed value. This prevents Saga B from benefiting from funds that were never confirmed.

In practice, Option B (optimistic locking) is the most common approach for financial systems. It provides good concurrency while preventing the cascading inconsistency. The trade-off is that Saga B may need to retry if Saga A's compensation changes the balance between Saga B's read and write.

### Challenge 3.7: Saga Definition Migration

Your team uses a versioned saga definition approach as described in the interview advanced topics. A new version of the booking saga adds a step between the flight booking and hotel booking: "validate travel insurance eligibility." The new step can fail independently of the other steps.

The old saga had steps: [flight, hotel, car, payment]. The new saga has steps: [flight, insurance_check, hotel, car, payment].

At deployment time, 50 sagas are in-flight using version 1 (old definition). These sagas are at various stages: some have completed flight only, some have completed flight and hotel, some have completed all four steps.

**Question:** How do you safely deploy the new saga definition without breaking in-flight sagas?

**Model answer:** The key principle: existing in-flight sagas complete using their recorded version. New sagas use the new version. The coordinator handles this automatically by storing the saga definition version in the saga_instances table.

| In-flight Saga | Current Step | Works With New Definition? | Action |
|---|---|---|---|
| Completed flight only | Step 1 done, step 2 pending (hotel in old, insurance_check in new) | No — step 2 is different | Continue with old definition |
| Completed flight + hotel | Step 2 done, step 3 pending (car in old, hotel in new) | No — step 3 is different | Continue with old definition |
| Completed all 4 steps | Saga is COMPLETED | Not applicable | No action needed |

Since in-flight sagas refer to their recorded version, the coordinator loads the old definition for them and the new definition for new sagas. Both definitions coexist until all version-1 sagas complete.

**Gotcha:** The old services must remain operational until all version-1 sagas complete. If the Hotel Service v2 changes its API, the old hotel endpoint must still be available for version-1 sagas. Plan for a transition period where both API versions are deployed. The transition period duration depends on the maximum saga lifetime — if sagas can take 24 hours, both versions must coexist for at least 24 hours after deployment.

## Tier 4: Beyond the Basics (Extra Credit)

These challenges are not part of the standard interview or course curriculum but represent the frontier of saga pattern thinking.

### Challenge 4.1: Sagas with Blockchain Integration

A supply chain system uses a private blockchain to record provenance events. Each step in the supply chain (manufacturer, shipper, distributor, retailer) must append a record to the blockchain. The blockchain is immutable — once a record is appended, it cannot be undone.

The supply chain saga has four steps, each corresponding to a provenance event. Step 3 (shipper milestone) fails because the shipper's system is offline. The saga must compensate, but step 1 and step 2 records are already on the blockchain.

**Question:** How do you compensate for blockchain records that cannot be deleted?

**Model answer:** Since blockchain records are immutable, compensation cannot delete them. The solution is to append a *correction record* that supersedes the original. The compensation appends a new blockchain record that says "Saga failed — compensating step 1. The original step 1 record is now voided by this correction."

The system reads both the original record and the correction record. Business logic always reads the latest correction for any given saga step. If a correction exists, the original record is treated as voided.

This is the blockchain equivalent of a compensating transaction: the original state is not restored (the original record remains), but the effective state is corrected (the record is no longer treated as valid).

### Challenge 4.2: Sagas with Backpressure

A saga coordinator processes 500 sagas per second. Each saga does 3 database writes to the saga log. The saga log database is provisioned for 2,000 IOPS. At 500 sagas/second with 3 writes each, the coordinator needs 1,500 IOPS — within the provisioned capacity.

A flash sale starts. Traffic spikes to 2,000 sagas per second. The math: 2,000 sagas x 3 writes = 6,000 IOPS needed. The database can only handle 2,000. The saga log write latency rises from 5ms to 800ms. Coordinator threads block waiting for writes. New sagas cannot be accepted. The entire system degrades.

**Question:** Design a backpressure mechanism for the saga coordinator that protects the saga log database without losing sagas.

**Model answer:** Implement admission control at the saga coordinator entry point:

1. **Token bucket limiter:** Accept at most 500 sagas per second (70% of the database capacity). Excess sagas are queued in a "pending" buffer with a 60-second TTL.

2. **Saga log write batching:** Instead of writing each step individually, batch multiple step writes into a single database transaction. Write 10 step completions at once instead of 10 individual writes. This reduces IOPS from 3,000 to 300.

3. **Read-your-writes consistency bypass:** For non-critical state transitions (step start, step update), use asynchronous writes with a write-behind cache. Critical transitions (saga complete, saga fail) use synchronous writes.

4. **Backpressure signal propagation:** When the pending buffer exceeds 80% capacity, the coordinator returns HTTP 429 (Too Many Requests) to the API gateway. The gateway can throttle incoming requests or return "Service temporarily busy" to clients.

The combination of admission control and write batching typically provides 3-5x throughput improvement without adding database capacity.
