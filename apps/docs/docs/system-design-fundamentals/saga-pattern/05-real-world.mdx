# Saga Pattern in Production

## Case Study 1: E-Commerce Order Processing

An e-commerce platform processes 50,000 orders per day across three core services: Inventory, Payment, and Shipping. Each order requires reserving inventory, charging the customer, and creating a shipping label. If any step fails, the entire order must be rolled back.

### Architecture

The platform uses an orchestrated saga with a dedicated coordinator deployed as a Kubernetes service. The coordinator communicates with each service through RabbitMQ command queues and receives results through an event queue.

### Key Lessons from This Implementation

**Lesson 1: Idempotency at the API level catches queue-level duplicates.**
The platform originally relied on RabbitMQ's at-least-once delivery but added API-level idempotency after a queue failover caused 47 duplicate charges in a single day. Each service now checks an idempotency table before processing any command. The cost is negligible — a single indexed read per step.

**Lesson 2: Compensations must handle "already done" gracefully.**
The Payment Service's void endpoint checks whether the charge already exists. If the charge was never captured (the payment step was slow but ultimately timed out), the void is a no-op. If the charge was captured and already voided by a previous compensation attempt, the void is also a no-op. This idempotent compensation design prevents failures from cascading.

**Lesson 3: Manual intervention workflow is part of the system design.**
The team built a simple admin panel showing all COMPENSATION_FAILED sagas. Operators review these daily. The most common cause is the Shipping Service being down during compensation — the operator clicks "Retry compensation" after verifying the service is back online.

## Case Study 2: Travel Booking Platform

A travel booking platform coordinates flights, hotels, car rentals, and insurance across 200+ third-party APIs. Each third party has different APIs, different cancellation policies, and different reliability characteristics.

### Architecture

The platform uses a choreographed saga with an event bus. Each service publishes events when it completes a step. The next service in the chain subscribes to those events. Compensation is handled by each service independently subscribing to failure events.

### Key Lessons

**Lesson 1: Choreography reduces coupling between teams.**
When the Insurance team needed to add cancellation coverage, they only needed to subscribe to the "CarBooked" event and the "CarFailed" event. No existing service had to change. The event schema was the contract, and it was already stable.

**Lesson 2: Compensation failures are expensive in travel.**
If the Flight Service successfully books a non-refundable ticket but a later step fails, the platform must absorb the cost of the cancellation fee. In one incident, a bug caused the Car Rental Service to always fail, triggering compensations that cost the platform $12,000 in non-refundable cancellation fees over 4 hours before the bug was caught.

**Lesson 3: Centralized tracing is mandatory.**
With a choreographed saga, debugging a failure requires tracing events across four services and correlating them with third-party API responses. The team built a centralized saga tracing dashboard that shows the event flow for each booking ID, including the timing of each event and the response from each third party.

## Case Study 3: Fintech Payment Processing

A fintech company processes international payments that involve currency conversion, anti-fraud checks, settlement with partner banks, and notification dispatch. Each payment must be atomic — the customer either sends money and the recipient receives it, or neither happens.

### Architecture

The platform uses an orchestrated saga with AWS Step Functions as the saga coordinator. The saga definition is a JSON state machine defined in the Step Functions console.

```json
{
  "Comment": "International payment saga",
  "StartAt": "FraudCheck",
  "States": {
    "FraudCheck": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:fraud-check",
      "Next": "CurrencyConversion",
      "Catch": [
        {
          "ErrorEquals": ["FraudCheckFailed"],
          "Next": "PaymentFailed"
        }
      ]
    },
    "CurrencyConversion": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:convert-currency",
      "Next": "Settlement",
      "Catch": [
        {
          "ErrorEquals": ["States.ALL"],
          "Next": "ReverseFraudCheck"
        }
      ]
    },
    "Settlement": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:bank-settlement",
      "Next": "NotifyRecipient",
      "Catch": [
        {
          "ErrorEquals": ["States.ALL"],
          "Next": "ReverseCurrencyConversion"
        }
      ]
    },
    "NotifyRecipient": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:send-notification",
      "End": true,
      "Catch": [
        {
          "ErrorEquals": ["States.ALL"],
          "Next": "ReverseSettlement"
        }
      ]
    },
    "ReverseFraudCheck": { "Type": "Task", "Resource": "arn:aws:lambda:clear-fraud-flag", "Next": "PaymentFailed" },
    "ReverseCurrencyConversion": { "Type": "Task", "Resource": "arn:aws:lambda:reverse-conversion", "Next": "ReverseFraudCheck" },
    "ReverseSettlement": { "Type": "Task", "Resource": "arn:aws:lambda:reverse-settlement", "Next": "ReverseCurrencyConversion" },
    "PaymentFailed": { "Type": "Succeed" }
  }
}
```

### Key Lessons

**Lesson 1: Managed orchestration services reduce operational burden.**
Using AWS Step Functions eliminated the need to build and maintain a saga coordinator. Step Functions handles retries, timeouts, state persistence, and dead-letter handling out of the box.

**Lesson 2: Monetary compensations must be proven to auditors.**
Every compensation action in the payment saga must produce an audit trail that proves the customer was not charged and the recipient was not paid. The saga log serves as this audit trail. Each step and compensation is logged with a timestamp, payload, and response. Auditors review these logs quarterly.

**Lesson 3: Long-running compensations (bank settlements can take 24 hours) require async processing.**
Some compensations, like reversing a bank settlement, do not complete instantly. The saga timeout for these steps is set to 48 hours. The coordinator uses a polling pattern — it checks the status of the reversal every 30 minutes until the bank confirms completion.

## Case Study 4: Netflix (Conductor / Orchestration Pioneer)

Netflix built one of the earliest and most influential saga orchestration systems: Netflix Conductor. Conductor manages millions of workflow executions per day across Netflix's microservice architecture.

### Architecture Highlights

- **Orchestration-first design**: Conductor is a dedicated workflow orchestration engine. Workflows are defined as JSON blueprints with tasks, decision points, forks, and joins.
- **Queue-based task execution**: Conductor publishes tasks to queues (SQS or internal). Workers poll for tasks and report results back to Conductor.
- **Task workers are stateless**: Workers take a task input, execute business logic, and return output. Conductor manages state.
- **Built-in retry and compensation**: Each task defines a retry count, timeout, and optional compensation task.

### Key Lessons

**Lesson 1: A dedicated orchestration engine scales better than home-grown coordinators.**
Netflix's early saga implementations were tightly coupled to the application code. Conductor decoupled workflow logic from business logic, allowing different teams to define and manage their own workflows without sharing code.

**Lesson 2: Lineage and traceability are essential at Netflix scale.**
With millions of workflow executions per day, Conductor provides a UI to browse running and completed workflows, inspect task inputs and outputs, and retry failed tasks. This traceability is critical for debugging and compliance.

**Lesson 3: Blueprints enable reuse.**
Conductor workflows are reusable blueprints. The same "fulfill order" blueprint can be parameterized for different product types (digital downloads, physical goods, streaming subscriptions) without duplicating the workflow definition.

## Open Source Tools

### Axon Framework (Java)

Axon provides a saga orchestration engine built on event sourcing and CQRS. It supports both choreographed and orchestrated sagas. Sagas are implemented as event-handling classes that react to events and dispatch commands.

```java
@Saga
public class BookingSaga {

    @Autowired
    private transient CommandGateway commandGateway;

    @StartSaga
    @SagaEventHandler(associationProperty = "flightId")
    public void on(FlightBookedEvent event) {
        // Book hotel after flight is confirmed
        commandGateway.send(new BookHotelCommand(event.getBookingId()));
    }

    @SagaEventHandler(associationProperty = "hotelId")
    public void on(HotelBookedEvent event) {
        // Book car after hotel is confirmed
        commandGateway.send(new ReserveCarCommand(event.getBookingId()));
    }

    @EndSaga
    @SagaEventHandler(associationProperty = "carId")
    public void on(CarBookedEvent event) {
        // All bookings confirmed — end saga
    }
}
```

### Temporal (Go, Java, Python, TypeScript)

Temporal provides a workflow engine with built-in saga support through the `Saga` class. Workflows are written as regular code with full retry, timeout, and compensation support.

```java
// Temporal saga example
public class BookingWorkflowImpl implements BookingWorkflow {

    @Override
    public BookingResult bookTrip(BookingRequest request) {
        Saga saga = new Saga(new Saga.Options.Builder().build());
        BookingResult result = new BookingResult();

        try {
            String flightId = saga.addCompensation(
                () -> flightService.cancel(request.getFlightId()),
                () -> flightService.book(request.getFlightId())
            );
            result.setFlightId(flightId);

            String hotelId = saga.addCompensation(
                () -> hotelService.cancel(request.getHotelId()),
                () -> hotelService.book(request.getHotelId())
            );
            result.setHotelId(hotelId);

            // If car booking fails, saga automatically compensates
            // hotel and flight in reverse order
            String carId = saga.addCompensation(
                () -> carService.cancel(request.getCarId()),
                () -> carService.book(request.getCarId())
            );
            result.setCarId(carId);

        } catch (Exception e) {
            saga.compensate();
            throw new SagaException("Booking failed, compensated", e);
        }

        return result;
    }
}
```

### Eventuate Tram (Java / .NET)

Eventuate Tram provides a saga framework with support for both choreography and orchestration. It uses Apache Kafka or RabbitMQ for event delivery and supports distributed tracing.

### Camunda (Java, REST API)

Camunda provides a BPMN-based workflow engine that can model sagas as business processes. Each step is a service task, and compensation is modeled as a compensation boundary event. Camunda provides a web-based modeler, monitoring dashboard, and REST API.

## Cloud Managed Services

### AWS Step Functions

Step Functions is a fully managed orchestration service. Define workflows as JSON state machines with up to 40,000 start executions per account per region. Key features:

- **Express workflows** for high-volume, short-duration sagas (under 5 minutes).
- **Standard workflows** for long-running sagas (up to 1 year).
- **Built-in retry and error handling** with exponential backoff.
- **Service integrations** with Lambda, ECS, SQS, SNS, DynamoDB.
- **Execution history** stored for 90 days for audit and debugging.

### Azure Durable Functions

Durable Functions extends Azure Functions with workflow orchestration. Write sagas as C# or Python functions with the `OrchestrationTrigger` binding. Key features:

- **Built-in saga support** through the `CallActivityWithRetryAsync` and `ContinueAsNew` APIs.
- **Automatic checkpointing** — orchestrator functions are replayed on restart.
- **Fan-out/fan-in** for parallel step execution.
- **Eternal orchestrations** for long-running sagas.

## Lessons from Production Deployments

### Lesson 1: Start with orchestration, move to choreography only when you have a reason.

Every team interviewed for these case studies started with orchestration. Most stayed with orchestration. The one team that switched to choreography did so specifically to enable the Insurance team to add its service without modifying any existing code. For most use cases, the operational simplicity of a single coordinator outweighs the coupling concerns.

### Lesson 2: Invest in the dead-letter queue workflow early.

When a saga enters COMPENSATION_FAILED, the operations team needs tools to inspect, retry, and manually resolve it. Teams that built these tools after seeing the first production incident regretted not building them during the initial design. The dead-letter queue workflow is not a post-production add-on — it is part of the core architecture.

### Lesson 3: Monitor compensation failure rate as a system health metric.

The rate of compensation failures is a leading indicator of system health. A sudden increase in compensation failures often signals a service degradation or an API contract change that was not communicated. Monitor this metric on the same dashboard as error rates and latencies.

### Lesson 4: Test compensation paths as rigorously as forward paths.

Teams consistently under-test compensation paths. The forward path is tested with unit tests, integration tests, and chaos engineering. The compensation path is tested once during development and never again. Compensation paths should be exercised in every staging deployment, ideally with automated chaos injection that forces compensation mid-saga.

## Additional Case Studies

### Case Study 5: Uber's Ringpop for Saga Coordination

Uber built Ringpop, a distributed coordination layer that manages saga execution across a fleet of services. Ringpop uses consistent hashing to assign saga ownership to specific nodes, and a gossip protocol to propagate state changes.

**Key architectural decisions:**

- **Consistent hashing for saga ownership**: Each saga is assigned to a specific node based on the saga ID hash. This ensures all commands for a saga go to the same coordinator without requiring a central routing layer.
- **Gossip-based state propagation**: When a saga step completes, the coordinator gossips the state change to replica nodes. If the primary coordinator fails, a replica picks up the saga without requiring a database read.
- **Partial replication**: Only the saga coordination metadata is replicated, not the full payload. The payload is stored in a separate key-value store.

**Lessons from Uber's implementation:**

The gossip-based approach reduces saga log database load by 60% compared to a centralized saga log. However, it adds complexity: the gossip protocol must handle network partitions, message ordering, and duplicate suppression. For most teams, a centralized saga log with a replicated database is simpler to operate.

### Case Study 6: Shopify's Saga for Order Fulfillment

Shopify processes hundreds of thousands of orders per day through a saga that coordinates inventory allocation, payment capture, tax calculation, and shipping label generation. The saga runs on a custom orchestration engine built on top of their job queue system.

**Key architectural decisions:**

- **Job queue as saga coordinator**: Each saga step is a job in Shopify's job queue system. The job contains the saga ID, step sequence, and payload. When a job completes successfully, it enqueues the next step's job. When a job fails, it enqueues compensation jobs for previous steps.
- **Idempotency via job deduplication**: The job queue system checks for duplicate jobs by saga ID and step sequence before executing. This prevents duplicate step execution from queue redelivery.
- **Graceful degradation**: If the tax calculation service is down, the saga does not immediately fail. It places the step on a retry queue with a 5-minute delay. Other sagas for different orders continue processing unaffected.

**Performance metrics:**
- 99.5% of sagas complete on the first attempt.
- Average saga completion time: 4.2 seconds (including external API calls for tax calculation and payment processing).
- Compensation rate: 0.3% of all sagas. Most compensations are from payment failures (declined cards).
- Manual intervention rate: 0.02% of all sagas (typically from inventory allocation failures that require operator review).

**Lesson specific to e-commerce:** The inventory allocation step has a 30-minute TTL hold. If the saga does not complete within 30 minutes, the inventory is automatically released. This protects against sagas that stall indefinitely due to coordinator crashes or network partitions. The TTL acts as a safety net for compensation failures — even if the compensation action fails, the inventory is eventually released.

### Case Study 7: A Bank's Payment Settlement Saga

A major bank processes international wire transfers through a saga that coordinates currency conversion, anti-money-laundering (AML) checks, correspondent bank routing, and beneficiary account crediting. The saga is orchestrated via a custom Java-based coordinator deployed on WebLogic.

**Key architectural decisions:**

- **Human-in-the-loop for AML checks**: Step 2 (AML check) is a human step that can take 2-24 hours. The coordinator sets a 48-hour timeout for this step and checks status every 5 minutes.
- **Compensation for irrevocable steps**: Some wire transfers are irrevocable after a certain point (once the money leaves the correspondent bank). The saga is designed to complete all pre-transaction steps before the irrevocable step. Steps after the irrevocable point do not require compensation — they are informational (send confirmation email, update ledger).
- **Saga log as audit trail**: The saga log is stored in Oracle Database with read-only access for auditors. Each step records the operator who performed it, the timestamp, and the full payload. The log is part of the bank's regulatory compliance framework.

**Compliance requirements that shaped the architecture:**

1. **Every compensation must be logged with a reason code.** The saga log stores a `compensation_reason` field that explains why the compensation was triggered (e.g., "AML check failed — customer on watchlist").
2. **Compensations that involve monetary reversals require dual approval.** The compensation action for step 1 (currency conversion) enters a PENDING_APPROVAL state. A second operator must approve the reversal before it executes. This prevents a single coordinator or operator from initiating unauthorized reversals.
3. **The saga log is immutable.** Once a step or compensation is logged, it cannot be modified or deleted. Auditors verify this by checking for gaps in the log sequence and by comparing the log against the database state.

**Lesson specific to finance:** In regulated industries, the saga log is not just an operational tool — it is a compliance artifact. The log must be immutable, queryable by auditors, and retained for the regulatory period (typically 5-7 years). This requirement affects schema design, storage planning, and access control policies.

## Open Source Implementation Comparison

### Axon Framework vs Temporal vs Camunda

| Feature | Axon Framework | Temporal | Camunda |
|---|---|---|---|
| Language support | Java, Kotlin | Go, Java, Python, TypeScript | Java, REST API |
| Saga model | Event-driven (event sourcing) | Workflow-as-code | BPMN modeler |
| Compensation mechanism | Event sourcing reverse events | Saga.compensate() | BPMN compensation boundary |
| Saga log persistence | Event store (RDBMS or NoSQL) | Temporal Server database | Process engine database |
| Retry handling | Built-in command retry | Automatic with configurable policy | BPMN error boundary events |
| Observability | Axon Dashboard | Temporal Web UI | Camunda Cockpit |
| Learning curve | Medium (event sourcing concepts) | Low (write code, not config) | Medium (BPMN notation) |
| Production usage | Mid-sized enterprises | Netflix, Snap, Stripe | Large enterprises |

### Choosing an OSS Framework

| When to Choose | Framework |
|---|---|
| Team already uses event sourcing and CQRS | Axon Framework |
| Team wants to write sagas as regular code | Temporal |
| Team includes business analysts who model workflows visually | Camunda |
| Team is on the JVM and wants minimal external dependencies | Eventuate Tram |
| Team wants a managed solution without operational overhead | AWS Step Functions or Azure Durable Functions |

### Migration Path: From Choreography to Orchestration

Teams often start with choreography because it is perceived as simpler. As the system grows, they migrate to orchestration. The migration path is:

1. **Identify the pain point.** When a bug requires tracing events across four services to find the root cause, or when adding a new service requires modifying three existing service's event handlers, these are signals that orchestration would help.

2. **Build the coordinator incrementally.** Do not rewrite all existing services at once. Start by adding a coordinator that watches existing events and builds a saga log. The coordinator is read-only at this stage — it tracks saga progress without making decisions.

3. **Migrate one saga at a time.** Pick the most failure-prone saga (highest compensation rate) and migrate it to orchestration first. The coordinator begins making decisions for this saga only. Other sagas remain choreographed.

4. **Measure the improvement.** Compare the compensation rate, debugging time, and operator intervention rate before and after migration. Use the data to justify the next migration.
