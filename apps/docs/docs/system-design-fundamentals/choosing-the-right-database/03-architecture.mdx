# Choosing the Right Database at Scale

Every database architecture hits a wall. The wall might be write throughput, storage capacity, query complexity, or operational overhead. The difference between an architecture that survives and one that collapses is how well you anticipate the next wall and how cleanly the architecture lets you step around it. This page traces the evolution from a single-node prototype to a global multi-database deployment.

## Naive Implementation

Start with the simplest possible architecture: one application server talking to one database on the same machine.

**What breaks at 1,000 users?** Nothing. The architecture is adequate for prototyping, small teams, or internal tools. If you are building a weekend project, stop here.

**What breaks at 100,000 users?** The database becomes the bottleneck. Connection pooling is insufficient вҖ” each application thread holds a database connection, and thread count exceeds the database's connection limit. The buffer pool cannot fit the working set, so reads hit disk. A single missing index causes a sequential scan that blocks writes through lock contention. The solution is to separate the application from the database onto different machines, add a connection pooler, and add read replicas.

**What breaks at 10 million users?** The write path is the bottleneck. A single PostgreSQL primary can sustain roughly 5,000-10,000 writes per second. At 10 million active users with even modest write volume per user, you exceed that ceiling. You need either sharding (manual partition across multiple primaries) or a distributed database that handles horizontal scale natively.

**Layer 1: Application servers.** Stateless servers behind a load balancer. No application state stored locally. Database connections managed through a connection pooler.

**Layer 2: Cache.** An in-memory cache (Redis, Memcached) sitting between the application and the database. The cache stores frequently-accessed data (user sessions, product catalog, configuration values) with a TTL. Cache hit rates above 90 percent reduce database load by an order of magnitude.

**Layer 3: Primary database.** The write master. All writes go here. Strongly-consistent reads also go here. The primary is the most carefully provisioned and monitored component in the stack.

**Layer 4: Read replicas.** One or more asynchronous replicas that serve read traffic. Replicas can be stale by 10-100 milliseconds under load. The application routes reads based on staleness tolerance: the user's own data is read from the primary, but the public product catalog can be served from a replica.

**Layer 5: Background workers.** A job queue (Sidekiq, Celery, SQS) for asynchronous operations that do not need to block the response: sending emails, generating thumbnails, running reports, processing refunds.

**Layer 6: Change data capture (CDC).** A CDC pipeline captures every write to the primary database and streams it to secondary systems: the search index (Elasticsearch), the analytics warehouse (BigQuery, Redshift), and the cache for cache invalidation.

### CDC Architecture in Detail

Change data capture works by reading the database's write-ahead log. Debezium is the most widely used CDC tool вҖ” it connects to PostgreSQL's replication slot, reads the WAL, and publishes each change event (INSERT, UPDATE, DELETE) to Apache Kafka. The event includes the before-and-after image of the row (if configured), the transaction ID, and the operation timestamp.

The Kafka consumer for the search index reads these events and updates Elasticsearch. The consumer for the analytics warehouse writes the raw events to Parquet files in S3, which are then loaded into BigQuery or Redshift. The consumer for the cache reads UPDATE and DELETE events and invalidates the corresponding cache keys.

CDC has several advantages over dual-writes (writing to both the database and the secondary system from the application):
- The application does not need to know about the secondary systems. It writes to one database.
- If the secondary system is down, the CDC event waits in Kafka and is replayed when the system recovers.
- The CDC pipeline provides a single, ordered stream of all data changes, which is exactly what an analytics system needs.
- CDC can capture changes made by database administrators directly in SQL, which dual-writes cannot.

The main disadvantage of CDC is operational complexity. You need to run Kafka (or a similar streaming platform), Debezium connectors, and consumers for each target system. The replication lag between the primary database and secondary systems is typically 100 milliseconds to 5 seconds, depending on the CDC pipeline's throughput and the target system's ingestion speed.

### Data Syndication Between Databases

In a polyglot architecture, data often needs to flow between databases. For example, user profile data in MongoDB may need to be available in the analytics system (BigQuery), and order data from PostgreSQL may need to be searchable in Elasticsearch.

The syndication pattern uses a hub-and-spoke model. The primary operational database (PostgreSQL for relational data, MongoDB for document data, Cassandra for time-series data) is the authoritative source. CDC streams from each authoritative source to a central streaming platform (Kafka). Consumers then push data to the target databases: Elasticsearch for search, BigQuery for analytics, Redis for caching.

The syndication pattern prevents a common failure mode: circular replication. If MongoDB streams to PostgreSQL and PostgreSQL streams back to MongoDB, a single update can loop forever. The rule is: each piece of data has exactly one authoritative source, and data flows in one direction from source to secondary systems. Secondary systems never write data back to the authoritative source.

## Integration Patterns

### Pattern 1: Database + Cache for Hot Data

The cache-aside pattern is the most common integration. On a read, the application checks the cache first. On a cache miss, it reads from the database, writes the result to the cache with a TTL, and returns the result. On a write, the application writes to the database and invalidates or updates the cache.

Cache-aside works well for user sessions, product data, and configuration. It works poorly for write-heavy workloads (the cache is constantly invalidated) and for queries that return large result sets (the cache fills with data that is rarely accessed again).

A more sophisticated variant is read-through caching, where the cache layer itself handles the database query on a miss. Redis with RedisJSON or Redis Stack supports read-through caching natively. The application only talks to the cache; the cache handles database communication transparently. This reduces application code complexity but makes the cache configuration more critical вҖ” if the cache goes down, the application loses its data access layer entirely.

### Pattern 2: Primary Database + Search Index

### Pattern 2: Primary Database + Search Index

When your application needs full-text search, faceted navigation, or relevance-ranked results, add a dedicated search index. Write to the primary database first, then asynchronously replicate to the search index via CDC or dual writes.

The search index is never the source of truth. If the index and database disagree on a value, the database wins. The index is rebuilt on failure by replaying the CDC stream from a checkpoint.

### Pattern 3: Operational Database + Analytics Warehouse

Operational databases are optimized for transactional workloads вҖ” many small reads and writes on individual records. Analytics queries вҖ” large aggregations, full-table scans, multi-table joins вҖ” would overwhelm an operational database.

The solution is an extract-load-transform (ELT) pipeline. The CDC stream writes raw data to object storage (S3, GCS). A periodic job loads the raw data into the analytics warehouse (BigQuery, Snowflake, Redshift) and transforms it into dimensional models for reporting.

### Pattern 4: Multi-Database Source of Truth

Some data domains benefit from different database models. A user's profile data (semi-structured) lives in a document store. Their orders (structured, transactional) live in a relational database. Their social connections (graph-shaped) live in a graph database.

Each database is the authoritative source for its domain. Cross-domain queries are handled by the application layer: fetch the user from MongoDB, then fetch their recent orders from Postgres, then fetch friend recommendations from Neo4j, and assemble the response in application code. The application is the join engine.

## Multi-Region and Global Scale

When your users span continents, a single-region deployment guarantees high latency for distant users. A user in Tokyo querying a database in Virginia experiences 150-200 milliseconds of network round-trip time before the database even processes the query.

### Active-Passive (Single Region Writes)

One region handles all writes. Read replicas in other regions serve local read traffic. Replication lag determines how stale remote reads are вҖ” typically 100-500 milliseconds across continents. Failover requires promoting a read replica in another region to primary, which takes 1-5 minutes and risks data loss of un-replicated writes.

### Active-Active (Multi-Region Writes)

Every region accepts writes. Data is synchronously or asynchronously replicated between regions. Active-active architectures require conflict resolution strategies: last-writer-wins (Cassandra), CRDTs (conflict-free replicated data types), or application-level reconciliation. Google Spanner and CockroachDB use consensus protocols to achieve consistent multi-region writes at the cost of 50-150 millisecond write latency for cross-region commits.

### Consistency and Latency Trade-Off

The multi-region consistency trade-off is a spectrum. At one extreme, all writes go to one region and replicas are eventually consistent. At the other extreme, all writes go through a distributed consensus protocol and are globally consistent. The costs are in write latency and availability.

For most applications, the sweet spot is region-local strong consistency with cross-region eventual consistency. User data is consistent within the home region. Cross-region data (global leaderboards, shared content) can tolerate minutes of staleness. This hybrid approach combines the operational simplicity of single-region writes with the latency benefits of regional read replicas.

### Read-Your-Writes Consistency

A critical detail in multi-region architectures: users expect to see their own writes immediately, even if the write was processed in a different region. A user who posts a comment from their phone in Tokyo and then views the page from their laptop in Virginia must see their comment.

This is called read-your-writes consistency. It is strict than eventual consistency but less strict than global linearizability. The common implementation: the application includes the user's home region in the request metadata, and reads for that user are always routed to their home region's primary database. Cross-region replication eventually synchronizes the data to other regions.

The requirement complicates the routing layer. The load balancer must know each user's home region, and the application must be willing to route a read to a potentially distant region if the user's data has not yet propagated locally.

### Read Replica Synchronization

Read replicas introduce a fundamental challenge: keeping all replicas synchronized with the primary while maintaining read availability. The synchronization strategy determines how much staleness your read traffic experiences.

**Transaction-level replication (PostgreSQL, MySQL):** The primary ships each completed transaction's WAL records to replicas. Replicas apply the transactions in order, maintaining a consistent snapshot of the database. Replication lag depends on the network latency between primary and replica and the rate at which the replica applies WAL. A heavily loaded replica can fall behind by 10-100 milliseconds during normal operation and by seconds during traffic spikes. When a replica is promoted to primary, it must apply all remaining WAL segments before accepting writes, which can take minutes if the lag is large.

**Eventual replication with read-repair (Cassandra, DynamoDB):** Data is replicated at write time to a configurable number of nodes (replication factor). Read requests check multiple replicas and repair stale data during the read (read repair). This model allows reads even if some replicas are down or slow, but the consistency guarantee varies by the consistency level configured for each request. With QUORUM consistency, a read is guaranteed to see any write that completed with QUORUM consistency.

**Logical replication (PostgreSQL 10+):** Logical replication decodes WAL into a row-level change stream (INSERT, UPDATE, DELETE) and applies it to target tables. Unlike streaming replication, which requires an identical schema on both sides, logical replication can replicate between different PostgreSQL versions and can filter specific tables or rows. Logical replication is the basis for CDC tools like Debezium.

### Sharding Strategies

Horizontal partitioning вҖ” sharding вҖ” is required when a single database node cannot handle the write throughput or data volume. The key decision is the shard key, which determines how data is distributed.

**Hash sharding:** A hash function is applied to the shard key, and the result determines which shard owns the data. Hash sharding provides uniform distribution вҖ” every shard receives roughly the same number of writes and stores roughly the same amount of data. Hash sharding is the default strategy for DynamoDB, Cassandra, and CockroachDB. The cost: range queries across shards require scatter-gather (query every shard and merge results).

**Range sharding:** Data is divided into contiguous ranges of the shard key. Shard 0 handles keys 0-1,000,000. Shard 1 handles keys 1,000,001-2,000,000. Range sharding supports efficient range queries within a shard вҖ” the query planner knows which shard to query from the range boundaries. The cost: range sharding creates hot spots at the boundaries. If the shard key is a timestamp, all writes go to the "latest" shard while historical shards are idle.

**Directory-based sharding:** A lookup service maps each shard key to a shard. The directory can be updated dynamically as shards split or merge. This approach provides the most flexibility вҖ” you can migrate data between shards without changing the application's routing logic. The cost: the directory is a single point of failure and a potential bottleneck. Vitess (the MySQL sharding layer used by Slack and YouTube) uses a directory-based approach with a topology service (etcd or ZooKeeper) for the directory.

**Choosing the shard key:**
- High cardinality (millions of unique values) вҖ” avoids hot spots.
- Uniform access frequency вҖ” all values are accessed roughly equally.
- Aligned with your most common query вҖ” single-shard queries are fast; cross-shard queries are slow.
- Examples: user_id, tenant_id, customer_id. Counterexamples: status, country, gender, created_at.

Good shard keys have high cardinality and even distribution. User ID is the classic example: a billion users spread evenly across 256 shards gives roughly 4 million users per shard. The hashed user_id ensures that each shard receives roughly the same write throughput.

Bad shard keys have low cardinality or skewed distribution. Sharding by country concentrates all US users on one shard while small countries each get their own underloaded shard. Sharding by creation timestamp concentrates all new data on the last shard (the hot shard problem).

Advanced sharding approaches use consistent hashing (Cassandra, DynamoDB) or range-based splitting with automatic rebalancing (CockroachDB, Bigtable). These handle node addition and removal automatically. The trade-off is that cross-shard queries become complex вҖ” an aggregation across all data requires scatter-gather across every shard.

## Capacity Planning

### Request Rate Estimation

Estimate your peak request rate by starting with your projected daily active users (DAU) and average requests per user per session:

```
Peak QPS = (DAU x requests per user per day x peak factor) / 86,400 seconds

Example:
  1 million DAU x 100 requests/user/day x 2x peak factor / 86,400 = roughly 2,300 QPS
```

Add 30-50 percent headroom for traffic spikes, marketing campaigns, and unexpected viral growth. If your model says 2,300 QPS, design for 3,500 QPS.

### Storage Estimation

```
Storage = (number of records) x (average record size) x (replication factor) x (index overhead)

Example:
  100 million user profiles x 2 KB each x 2 replicas x 1.5x index overhead
  = 100M x 2KB x 2 x 1.5 = 600 GB
```

Add 20 percent for temporary tables, WAL files, and operational overhead. The same workload in a relational database with 5 indexes per table might require 2x-3x more storage than estimated, because each index duplicates the indexed columns.

### Bandwidth Estimation

```
Bandwidth = (request rate) x (average response size)

Example:
  3,500 QPS x 10 KB average response = 35 MB/s = 280 Mbps
```

For multi-region deployments, add replication traffic. Each write shipped to a remote replica adds another 5-50 ms of bandwidth consumption per operation, depending on record size.

### Compute Sizing

A single PostgreSQL node on an 8-core, 64 GB machine handles roughly 5,000 writes/second and 20,000 reads/second before hitting CPU saturation. For higher throughput:

- Add read replicas for read scaling (linear improvement).
- Shard the write path for write scaling (each shard handles its own 5K writes/second).
- Move to a distributed database (CockroachDB, Spanner) that handles both reads and writes at scale across nodes.

The compute cost of a distributed database is typically 2x-3x that of a single-node setup for the same throughput, because each write requires consensus across multiple nodes. The trade-off is automatic scalability versus manual sharding.

## Deployment and Operations

### Zero-Downtime Migrations

Changing a database schema or migrating to a new database without downtime requires the expand-migrate-contract pattern:

1. **Expand:** Add the new schema or database alongside the existing one. Write to both.
2. **Migrate:** Backfill existing data. Run validation queries to ensure consistency.
3. **Contract:** Remove the old schema or database. The migration is complete.

Each phase takes days or weeks. Rollback means stopping the migration and ensuring the old schema is still operational. A production database migration without a rollback plan is not a plan.

### Monitoring

Every production database needs real-time monitoring of these metrics:

- **Latency percentiles** (p50, p99, p99.9) for each query type. Alert if p99 exceeds 2x the baseline.
- **Connection count.** Alert if connections reach 80 percent of the max.
- **Replication lag.** Alert if lag exceeds 10 seconds.
- **Disk usage.** Alert at 80 percent and 90 percent capacity.
- **Cache hit ratio.** Alert if ratio drops below 90 percent for the buffer pool.
- **Slow query log.** Queries longer than 100 ms should be logged and reviewed weekly.
- **Deadlocks and lock waits.** Any deadlock indicates a locking problem that needs investigation.

### Debugging

When a database goes slow, follow this checklist:

1. Check the slow query log. A single bad query can consume 100 percent CPU.
2. Check connection count. Connection storms from application restarts overwhelm even poolers.
3. Check replication lag. A lagging replica cannot serve fresh reads.
4. Check disk I/O. Compaction in LSM-tree databases spikes I/O every few hours.
5. Check the buffer pool hit ratio. A falling ratio means the working set no longer fits in memory.
6. Check for blocking queries. `pg_blocking_pids()` in PostgreSQL shows which query is blocking others.
7. Check application-side connection pools. Leaked connections eventually exhaust the pool.

### Upgrades

Minor version upgrades are typically safe and fix bugs. Major version upgrades change internal data formats and require careful testing. The standard upgrade procedure:

1. Deploy a new database instance running the new version alongside the old instance.
2. Set up replication from the old version to the new version.
3. Run the application against the new version in a staging environment.
4. Cut over traffic during a maintenance window.
5. Keep the old instance running for 48 hours as a rollback target.

For databases that support rolling upgrades (Cassandra, CockroachDB), you can upgrade nodes one at a time with zero downtime. The cluster health must be checked between each node upgrade.

### Backup and Disaster Recovery

A backup strategy has three dimensions: recovery point objective (RPO вҖ” how much data you lose), recovery time objective (RTO вҖ” how long to restore), and cost.

For PostgreSQL, the standard approach is continuous archiving with WAL shipping. A full backup is taken weekly (using pg_basebackup or a snapshot). WAL segments are archived continuously to S3 or equivalent. For recovery, you restore the full backup and replay WAL segments to the desired point in time. RPO is the WAL archive interval (typically 1-5 minutes). RTO is the time to restore the base backup plus replay WAL (typically 1-10 hours for multi-terabyte databases).

For DynamoDB, point-in-time recovery provides continuous backups with 35-day retention. Restoring to any second within the retention window takes minutes regardless of table size. RPO is zero вҖ” you can restore to precisely the point of failure. RTO is 5-30 minutes.

For Cassandra, backup requires snapshotting the data directory on each node. The challenge is ensuring consistency across nodes: a snapshot at time T on node 1 and time T+1 on node 2 may contain inconsistent data for the same logical rows. Repair operations must reconcile differences after restore.

The most common backup failure: nobody tests the restore. Schedule a quarterly restore drill where you restore a production-sized backup to a staging environment and run validation queries. A backup that has not been tested is a wish, not a backup.

### Blue-Green Deployments for Databases

For applications that deploy to production multiple times per day, the database schema change is the bottleneck. Application changes can be deployed in blue-green fashion вҖ” swap traffic between two environments вҖ” but database changes affect both environments simultaneously.

The solution is backward-compatible schema changes applied in phases:
- Add a new column with a default value. The old application code does not use it. The new application code starts using it.
- Backfill data for the new column. Run as a background job.
- Remove references to the old column from application code.
- Drop the old column (a blocking operation on large tables).

This phased approach allows database evolution at application-deployment speed. The constraint is that every schema change must be backward-compatible вҖ” no renames, no type changes that lose information, no NOT NULL constraints on existing data without a default.
