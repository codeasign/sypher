# Choosing the Right Database in Production

Real systems make real database choices under real constraints — existing codebases, limited engineering teams, budget pressure, and time-to-market demands. Studying how successful companies made their database decisions reveals patterns that apply at every scale.

## Company Case Studies

### Netflix: From Oracle to NoSQL to a Polyglot Mesh

Netflix started as a monolithic Java application running on Oracle. By 2008, the Oracle monolith was the bottleneck: schema changes took weeks, scaling required database administrator intervention, and every deployment risked database downtime. The famous 2008 database corruption that prevented DVD shipments for three days triggered the migration to a distributed, cloud-native architecture on AWS.

Netflix's database evolution is a textbook case of deliberate polyglot persistence:

- **DynamoDB** handles user activity (viewing history, ratings, my-list) where high throughput and low latency at global scale are essential. DynamoDB's single-digit-millisecond p99 latency at 100,000+ requests per second is critical for the playback experience.
- **Cassandra** serves the recommendations and personalization engine. Cassandra's wide-column model matches the time-series nature of viewing events, and its multi-region support enables consistent personalization across geographies.
- **MySQL** (via RDS) still powers the billing and subscription system. Financial data needs ACID guarantees, and MySQL with strong consistency provides them at a throughput that billing transactions require — thousands of writes per second, not millions.
- **Elasticsearch** powers the global search function across the catalog of 20,000+ titles with faceted filtering by genre, year, language, and streaming availability.
- **EVCache** (Netflix's Memcached-based distributed cache) reduces database load for high-read-traffic data like the browse experience and member sessions.

### Uber: From PostgreSQL to MySQL to Schemaless to Cassandra

Uber's database journey is defined by explosive growth and the need for write throughput. In 2010, Uber ran on PostgreSQL. As ride requests grew in major cities, PostgreSQL's single-writer bottleneck became the constraint. A single database handling writes for the entire New York City fleet could not keep up.

Uber migrated to MySQL in 2014, then built a custom database layer called Schemaless on top of MySQL. Schemaless provided document-like flexibility with MySQL's durability. The final evolution was a move to Cassandra for the real-time trip data and dispatching events.

Today, Uber's database architecture is tiered:

- **Cassandra** handles real-time trip state and ride matching. Each trip generates hundreds of events (location pings, status changes, driver updates). Cassandra's write throughput scales linearly with nodes, and its multi-region support keeps the system operational even if an entire datacenter goes down.
- **Schemaless (on MySQL)** handles user profiles, driver profiles, and other account data that benefits from document-like flexibility but needs MySQL's replication and durability.
- **PostgreSQL** (returning!) handles the payments and financial reconciliation system, where ACID compliance is non-negotiable and throughput is moderate.
- **Redis** caches frequently accessed data: surge multipliers, nearby driver locations, and fare estimates.

### Discord: MongoDB to Cassandra to ScyllaDB

Discord's database journey shows how read-heavy workloads with time-series access patterns drive database choices. Discord stored all messages in MongoDB. As the platform grew, MongoDB's per-document storage overhead (roughly 30 percent overhead on each document) and read performance became issues at the scale of billions of messages.

The team migrated to Cassandra for message storage. Cassandra's wide-column model perfectly matches the time-series pattern: messages are naturally ordered by timestamp, and reads fetch a contiguous range of recent messages for a channel. Cassandra's per-record overhead is also lower than MongoDB's.

After the Cassandra migration, Discord eventually moved to ScyllaDB, a C++ reimplementation of Cassandra that provides significantly better per-node performance (Scylla claims 10x throughput per node by eliminating Cassandra's JVM garbage collection pauses). ScyllaDB now stores trillions of messages and handles millions of writes per second.

### Cloudflare: A Multi-Provider Hybrid

Cloudflare's database choice is shaped by its position as a reverse proxy for millions of websites. The database must handle both the control plane (configuration management, account data) and the data plane (analytics, logs, cache state).

- **PostgreSQL** powers the core control plane: account data, DNS configuration, firewall rules. Strong consistency is required — if a customer changes their DNS settings, every edge node must converge on the new configuration.
- **ClickHouse** powers the analytics pipeline processing trillions of HTTP request logs per day. ClickHouse's columnar storage and vectorized query execution deliver sub-second aggregations over petabytes of data.
- **Redis** (used internally as a distributed cache) and **Quicksilver** (Cloudflare's custom KV store distributed to 300+ edge locations) provide low-latency access to cached content and configuration.

### GitHub: MySQL at Massive Scale

GitHub runs one of the largest MySQL deployments in the world, powering the core data layer for 100 million repositories, 40 million users, and billions of events. GitHub's database story shows how to scale a relational database to extreme size.

GitHub originally ran on a single MySQL master with multiple read replicas. As the platform grew, the single master became the bottleneck for write throughput. GitHub's solution was horizontal partitioning: they sharded the database by repository ID, distributing repository metadata, issues, pull requests, and comments across 15 MySQL clusters with hundreds of shards each.

Each shard is a standard MySQL instance with a replication topology (one master, two replicas). The application routing layer maps a repository ID to the correct shard using a consistent hash. Cross-shard queries (show me all repositories owned by user X) require scatter-gather across all shards, which GitHub optimized by maintaining summary tables per user that are updated asynchronously.

Key lessons from GitHub's MySQL journey:
- MySQL scales to petabyte size and millions of writes per second when you shard aggressively. The shard key must match the primary access pattern.
- Read replicas handle 10x read traffic before they need to be sharded themselves. The read-to-write ratio at GitHub is roughly 100:1.
- Schema migrations on a sharded database require rolling deploys: add a nullable column to one shard, verify, then proceed to the next. A migration across 100 shards takes days.
- The operational team managing this infrastructure is large — roughly 20 database engineers for 15 clusters — but the per-instance operations are standardized and automated.

### Stack Overflow: The Single Database Done Right

Stack Overflow is famous for running on a single SQL Server instance serving the entire site — 10 billion page views per year, 20 million questions, and 100 million monthly visitors. The architecture shows how far you can push a single relational database with careful optimization.

The key to Stack Overflow's single-database approach:
- No ORM. All queries are hand-written SQL optimized for the specific access pattern. Stack Overflow's team reports that removing Entity Framework (an ORM) and moving to raw ADO.NET queries reduced page load times by 30 percent.
- Aggressive caching at the application level. The SQL Server instance caches query results, but Stack Overflow also caches rendered HTML fragments in memory, reducing database queries per page view to 1-2 on average.
- Denormalized, read-optimized schema. The database schema is designed for the queries the application runs, not for theoretical normalization purity. Tags, for example, are stored as a delimited string on the question row rather than in a separate normalized table.
- The entire database fits in RAM (the buffer pool is roughly 200 GB, and the active database is under 100 GB). Every query results in an in-memory data read, not a disk seek.

Stack Overflow's architecture is the counterpoint to the "you need a distributed database" narrative. For systems where the active data set fits in a single server's RAM and the query pattern is read-heavy with a wide fan-out, a single well-tuned database outperforms distributed alternatives.

### Amazon: A Database for Every Workload

Amazon's internal database story is foundational to the cloud database market. Amazon built DynamoDB in response to the scaling limitations of relational databases in the Amazon.com shopping cart service. DynamoDB's key-value model and consistent hashing ring were designed specifically for high-throughput, low-latency access to simple data (cart contents, session state, product pages).

Today, Amazon's external database strategy (AWS) offers a database for every workload:

- **Amazon RDS/Aurora** for relational workloads requiring SQL and ACID.
- **DynamoDB** for key-value and document workloads at any scale.
- **ElastiCache (Redis/Memcached)** for in-memory caching.
- **Neptune** for graph workloads.
- **Timestream** for time-series data.
- **DocumentDB** for MongoDB-compatible document storage.
- **Keyspaces** for Cassandra-compatible wide-column storage.
- **Quantum Ledger Database (QLDB)** for immutable, cryptographically verifiable transaction logs.

The lesson: Amazon recognized that no single database serves all workloads and built a product for each workload type. The operational insight is that each database has its own management, monitoring, and scaling characteristics — and Amazon is willing to build and maintain each of them because the specialization delivers 10x-100x performance improvements for the right workload.

## Open Source Implementations

### PostgreSQL Extensions

PostgreSQL's extension ecosystem makes it the most adaptable general-purpose database. The extensions effectively add specialized database capabilities without leaving the PostgreSQL ecosystem:

- **PostGIS** adds spatial data types and geospatial queries (point-in-polygon, distance calculations, spatial joins). Used by thousands of location-based applications.
- **pgvector** implements approximate nearest-neighbor search for vector embeddings, enabling semantic search and RAG (retrieval-augmented generation) applications within PostgreSQL — removing the need for a separate vector database.
- **TimescaleDB** turns PostgreSQL into a time-series database with automatic partitioning (hypertables), data retention policies, and time-series-specific query optimizations.
- **Citus** adds transparent sharding to PostgreSQL, distributing data across multiple nodes while preserving the SQL interface.
- **pg_partman** automates table partitioning for range and list partitioning.
- **hypopg** creates virtual indexes to test whether an index would speed up a query without actually building it.

### RocksDB and Its Ecosystem

RocksDB (Facebook's LSM-tree library) has become the embedded storage engine for dozens of databases. Key-value stores, streaming platforms, and graph databases all build on RocksDB:

- **MongoDB** uses WiredTiger (originally a standalone engine, later MongoDB's default).
- **Cassandra** uses its own LSM-tree implementation with similar design principles.
- **InfluxDB** uses its Time-Structured Merge Tree (TSM) for time-series data.
- **Plus more**: Kafka Streams, TiKV, YugabyteDB, Apache Flink, and Pandora all embed RocksDB.

The platform effect is significant: improvements to RocksDB (better compaction algorithms, optimized bloom filters, read-amplification reduction) benefit the entire ecosystem.

### SQLite: The Embedded Database That Runs Everywhere

SQLite is the most deployed database in the world — every smartphone, every browser, every desktop application, and countless embedded systems use it. SQLite is an embedded, serverless, zero-configuration relational database that stores data in a single file.

SQLite's design trade-off is deliberately the opposite of a networked database: it prioritizes simplicity, reliability, and small footprint over concurrency and scale. A SQLite database can handle 50-100 write transactions per second on a typical filesystem (limited by the filesystem's locking granularity) and unlimited read concurrency. The maximum database size is 281 TB, and typical performance is competitive with client-server databases for databases under 1 GB.

SQLite is the right choice for: mobile applications, desktop applications, development and testing, embedded systems, and any application where the data is local to a single device and write concurrency is low. It is the wrong choice for: web applications with concurrent writes from multiple processes, systems requiring network access to the database, or databases expected to grow beyond 100 GB.

### Migration Patterns at Scale: Shopify

Shopify is one of the largest e-commerce platforms in the world, processing hundreds of thousands of requests per second across hundreds of thousands of merchants. Shopify's database architecture has evolved from a single MySQL instance to a sophisticated distributed system, and their migration experience provides valuable lessons.

Shopify started on a single MySQL database — the standard Rails stack of the early 2010s. As the platform grew, the single MySQL instance became the bottleneck. Write throughput was limited by the single master, and the database size exceeded 10 TB, making backups and restores a multi-hour operation.

Shopify's solution was to shard MySQL by merchant ID. Each merchant (store) gets its own database shard. All queries within a merchant's context hit a single shard, providing both isolation and predictable performance. Cross-merchant queries (Shopify's internal analytics) run on a separate analytics cluster replicated from each merchant's shard via CDC.

The migration from a single MySQL instance to a sharded architecture took 2 years and involved:
- Building a proxy layer (based on MySQL Proxy) that routes each query to the correct shard based on merchant_id.
- Writing a backfill tool that copied each merchant's data from the monolithic database to the new sharded database.
- Running both the monolith and sharded system in parallel for 6 months with dual-writes.
- Implementing a reconciliation system that verified data consistency between the old and new systems.
- Training 50+ engineers on the new sharding architecture and access patterns.

The key insight from Shopify's migration: the proxy layer that routes queries to the correct shard is the most critical piece of infrastructure. It must handle shard topology changes (splitting a hot shard, adding capacity), connection pooling per shard, and query routing based on the shard key. Shopify eventually open-sourced their sharding technology as Vitess (originally developed at YouTube and later adopted by Shopify).

### Etsy: MySQL with Online Schema Migration

Etsy is one of the largest MySQL deployments in the e-commerce space, and the company contributed a critical open-source tool: gh-ost (GitHub's Online Schema Migration, originally developed at Etsy). gh-ost allows schema changes on large MySQL tables without downtime, blocking queries, or replication lag.

The tool works by creating a ghost table with the new schema, replaying binlog events from the original table to the ghost table in real-time, and then atomically swapping the tables when the migration is complete. This allows Etsy to run schema migrations on tables with hundreds of millions of rows without any application downtime.

Etsy's database architecture demonstrates how to achieve both scale and availability on MySQL:
- Sharded by user ID across multiple MySQL masters.
- Each shard has a replication topology with one master, two replicas, and automated failover via Orchestrator.
- Schema migrations run via gh-ost across all shards in a rolling fashion.
- The application layer handles cross-shard queries (searching across all users' listings) by fanning out to all shards and merging results.
- Read-heavy queries are served from read replicas, with the application routing read queries based on staleness tolerance.

### YouTube: From MySQL to Bigtable

YouTube's database evolution mirrors the transition from a startup to a Google-scale service. The original YouTube ran on MySQL. As the video catalog grew to millions of videos and billions of views, MySQL's single-master architecture could not handle the write throughput. The team needed a database that could scale horizontally across thousands of machines while serving sub-millisecond lookups on video metadata.

YouTube migrated to Google Bigtable, the wide-column distributed database that runs Google Search, Maps, Gmail, and YouTube. Bigtable's design — a sparse, distributed, persistent multi-dimensional sorted map — matched YouTube's access patterns perfectly: look up a video by ID, range scan videos by upload date for a channel, or scan all videos matching certain metadata criteria.

The key architectural insight: Bigtable's row-level replication (using atomic mutations and row-level transactions) provided the consistency model YouTube needed for video metadata. Video views could be eventually consistent, but video owner and title needed consistency to prevent corruption. Bigtable's single-row transactions provided this guarantee without the overhead of distributed transactions.

### Cloud Provider Database Selection by Use Case

When choosing a managed database on a cloud provider, the decision is often not just about the database technology but about the broader ecosystem. Each cloud provider optimizes their managed databases for different use cases, and the best choice depends on your integration requirements.

**AWS is strongest for:** Serverless key-value workloads (DynamoDB), broadest ecosystem of database types (8 managed database services), and tightest integration with the AWS ecosystem (Lambda, API Gateway, SQS, Kinesis). Choose AWS when you are all-in on the AWS ecosystem and need the widest range of database options.

**GCP is strongest for:** Globally consistent relational workloads (Spanner), serverless document databases (Firestore), and petabyte-scale analytics (BigQuery). Choose GCP when you have global-scale requirements, need strong consistency across regions, or are building a data-intensive application that benefits from BigQuery's serverless analytics.

**Azure is strongest for:** SQL Server workloads (Azure SQL Database), .NET integration, and hybrid cloud deployments that span on-premises and cloud. Choose Azure when you are a Microsoft shop using SQL Server, .NET, and Active Directory, or when you need hybrid cloud capabilities.

### Database Migration Success Stories and Failures

**Success — Slack moving from Cassandra to MySQL:** Slack's original message storage used Cassandra. As the platform grew, the team found that Cassandra's eventual consistency model caused subtle bugs: messages appeared out of order, notifications fired for already-deleted messages, and read repair operations caused unpredictable latency spikes. Slack migrated to MySQL with Vitess (a sharding layer for MySQL) to get strong consistency with horizontal scale. The migration took 18 months and was executed without data loss or downtime.

**Failure — Digg moving from MySQL to Cassandra and back:** In 2010, Digg spent a year migrating from MySQL to Cassandra to handle the scale of social news aggregation. The migration was plagued by operational issues: Cassandra's compaction caused 10x latency spikes, the eventual consistency model confused users who saw inconsistent vote counts, and the engineering team did not have operational experience with distributed databases. Digg eventually migrated back to MySQL. The failed migration contributed to Digg's decline.

**The lesson from both stories:** Migration success depends more on team readiness than on technical superiority. Slack had strong engineering talent and dedicated migration resources. Digg underinvested in operational readiness. The database technology was not the deciding factor in either case.

## Lessons Learned — Five Patterns

**Pattern 1: Start relational, add specialized databases when measured.** Every company in this study started with a relational database. They added NoSQL databases only after measuring a real bottleneck that the relational database could not overcome. The bottleneck was always write throughput, never query expressiveness. This pattern strongly supports the default-to-PostgreSQL guidance.

**Pattern 2: Each database is the sole source of truth for its domain.** In a polyglot architecture, each database owns a specific domain. DynamoDB owns user activity at Netflix. Cassandra owns trip state at Uber. Postgres owns DNS configuration at Cloudflare. No data is split across databases in a way that requires cross-database joins. The application layer is the join engine.

**Pattern 3: CDC is the standard integration pattern.** Every case study uses change data capture to synchronize data between databases. Debezium or Kafka Connect captures writes from the primary operational database and streams them to secondary systems (analytics, search, cache). CDC is preferred over dual-writes because it decouples the application from the secondary system — if the secondary is down, the primary write still succeeds.

**Pattern 4: Managed services reduce the database team headcount needed.** Netflix, Cloudflare, and Amazon run large database teams because they operate at extreme scale or build custom database products. Most companies do not need that. Managed database services (RDS, DynamoDB, Cloud SQL, Cosmos DB) eliminate backup management, patch deployment, hardware provisioning, and failover orchestration. A team of two can operate ten managed database instances that would require a team of six if self-managed.

**Pattern 5: Migrate databases when the pain justifies the risk, not before.** Every migration in these case studies was reactive — the current database was actively causing production incidents, limiting growth, or increasing operational costs. No company migrated to a new database because the new one was theoretically better. The cost and risk of a database migration (6-18 months for major migrations at scale) mean the bar for deciding to migrate must be high.
