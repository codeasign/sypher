## Naive Implementation

### Single Master, Single Replica

The simplest distributed database architecture has one primary node and one replica. The primary handles all writes. The replica handles reads and acts as a failover target.

At 1,000 users, this works fine. The primary processes 50 writes per second. The replica processes 200 reads per second. Replication lag is under 10 milliseconds. Heartbeats between the two nodes keep both healthy.

At 100,000 users, cracks appear. The primary processes 5,000 writes per second. Replication bandwidth becomes a bottleneck. The replica falls behind by 500 milliseconds. Stale reads become noticeable. And the architecture has a fatal flaw: a network partition between the two nodes.

When the link drops, both nodes believe they are alive. The primary continues accepting writes. The replica, unable to reach the primary, also starts accepting writes. Now you have two primaries. This is split-brain. When the partition heals, the two primaries have diverged. Reconciliation is manual and painful.

At 10 million users, this architecture collapses entirely. The primary cannot handle 500,000 writes per second. The replica cannot serve 2 million reads per second. And the single point of failure on both nodes means any partition or crash causes an outage.

## Production-Grade Architecture

A production-grade distributed system that respects CAP constraints has multiple layers. Each layer is designed to handle partitions gracefully.

The architecture has six components. A load balancer layer distributes traffic across nodes. A health check system monitors node liveness. A cluster membership service tracks which nodes are part of the cluster. A consensus layer (for CP systems) or gossip layer (for AP systems) coordinates state. A replication channel moves data between nodes. A storage engine persists data locally.

### Load Balancer Configuration

The load balancer must be partition-aware. Configure it with health checks that detect node liveness. A node that cannot reach a majority of peers should be marked as unhealthy.

For CP systems, the load balancer should route traffic only to the quorum side of a partition. This requires the load balancer to understand the cluster state. In practice, this is achieved by having nodes report their partition status through the health check endpoint. A node in the minority partition returns a 503 status. The load balancer stops sending traffic to it.

For AP systems, the load balancer routes to all nodes. Partition-aware routing is unnecessary because all nodes accept requests. The load balancer simply distributes traffic across all healthy nodes.

## Integration Patterns

### Pattern 1: Relational Database with Read Replicas

A common architecture is a primary PostgreSQL database with multiple read replicas. This is a CP system: consistency is maintained through synchronous replication. During a partition, the primary stops accepting writes if it cannot reach a quorum of replicas.

The read replicas serve stale data during the partition. Applications must handle stale reads gracefully. Use this pattern for systems where strong consistency is required for writes but read staleness is acceptable during partitions.

### Pattern 2: Cache Layer with Write-Through

A cache layer like Redis or Memcached sits in front of a database. The cache is an AP component: it must always be available to serve requests. The database is the CP source of truth.

Writes go to the database first (CP guarantee), then invalidate or update the cache (AP availability). Reads check the cache first. If the cache misses, read from the database and populate the cache.

During a partition between cache nodes, the cache returns stale data for some keys. The database is always consistent. The cache trades consistency for availability.

### Pattern 3: Message Queue with At-Least-Once Delivery

A message queue like Kafka or RabbitMQ is typically configured as AP. Messages must be available for consumption even during partitions. The queue sacrifices ordering guarantees for availability.

Producers write to any partition leader. Consumers read from any partition. During a partition, messages may be duplicated or delivered out of order. Consumers must handle this through idempotency.

Use this pattern when message delivery is more important than message ordering. Log ingestion, event streaming, and analytics pipelines all benefit from AP message queues.

### Pattern 4: CDN for Static Content

A CDN is the ultimate AP system. Content is cached at edge nodes worldwide. Consistency is not guaranteed: an updated CSS file may take minutes to propagate to all edges. But the system never goes down.

CDNs implement a TTL-based consistency model. Content is served from the edge cache until the TTL expires, then fetched from the origin. During a partition, the edge serves stale content beyond the TTL. The origin is isolated but the CDN still serves requests.

Use a CDN when availability is paramount and staleness is acceptable for short periods.

## Multi-Region and Global Scale

Cross-datacenter replication introduces new CAP challenges. The network between regions has higher latency and is more likely to partition. A round trip between us-east-1 and eu-west-1 is 80-100 milliseconds. The probability of packet loss is higher than within a single datacenter.

A global CP system would require synchronous replication across regions. Every write would take 100 milliseconds minimum. Throughput would be limited by the slowest region. This is impractical for most systems.

Instead, global systems use a hybrid approach: CP within each region, AP across regions. Each region has a local quorum that guarantees consistency within the region. Cross-region replication is asynchronous. During a global partition, each region continues operating independently. Data diverges across regions and reconciles when the partition heals.

DynamoDB Global Tables use this pattern. Writes within a region are strongly consistent. Cross-region replication is eventually consistent. A write in us-east-1 may take 1-3 seconds to propagate to eu-west-1.

### Failover Across Regions

When a full region fails, you must fail over to another region. The failover strategy depends on your AP/CP choice.

For AP systems, failover is simple. Route all traffic to the remaining healthy regions. Data may be inconsistent, but the system stays up. Conflict resolution runs after the failed region returns.

For CP systems, failover requires careful coordination. You must ensure only one region is the source of truth. This typically requires a consensus-based leader election across regions. Paxos or Raft across regions is slow. ZooKeeper clusters spanning regions are configured with 5 nodes and a strict quorum requirement.

## Capacity Planning

Walk through the math for a system with 10 million daily active users.

Assume each user makes 10 reads and 2 writes per day. That is 100 million reads and 20 million writes per day.

Peak load is 2x average (traffic is not evenly distributed). Peak seconds per day: 10,000 (approximately 3 hours). Peak read QPS: 100 million / 10,000 seconds * 2 = 20,000. Peak write QPS: 20 million / 10,000 seconds * 2 = 4,000.

For a CP system with per-node throughput of 10,000 writes/second: 4,000 write QPS requires 1 node (with headroom). For reads at 20,000 QPS with 50,000 reads/second per node: 1 node suffices. In practice, use 3 nodes for fault tolerance.

Storage: 10 million users * average 1 KB per user = 10 GB. With 3x replication = 30 GB. Negligible.

Bandwidth: 4,000 writes/second * 1 KB = 4 MB/s write. 20,000 reads/second * 1 KB = 20 MB/s read. Total 24 MB/s. A 1 Gbps network handles this easily.

For an AP system, per-node throughput is 10x higher. 1 node handles 100,000 writes/second. The bottleneck becomes the application layer, not the database.

The takeaway: for 10 million DAU, the database layer is not the bottleneck. The bottleneck is the application layer, the network, and the consistency guarantee overhead.

## Deployment and Operations

### Zero-Downtime Deploy

Deploying a new version of a distributed system requires care. A rolling deploy updates nodes one at a time. During the deploy, the cluster is temporarily in a degraded state.

For CP systems, the rolling deploy must maintain quorum at all times. With N=3, only 1 node can be down during the deploy. Deploy node A, wait for it to join the quorum, deploy node B, wait, deploy node C. Each deploy cycle takes 30-60 seconds. Total deploy time: 2-3 minutes.

For AP systems, the rolling deploy is simpler. Any number of nodes can be down simultaneously (within reason). Deploy all nodes at once using a blue-green pattern: spin up a new cluster alongside the old one, switch traffic, tear down the old cluster.

### Monitoring and Metrics

Monitor these metrics for a CAP-aware system:

**Partition count.** How many partitions have occurred in the last hour. A spike indicates a network problem.

**Quorum loss events.** For CP systems, how often has the minority side been unavailable. Each event means rejected requests.

**Inconsistency window.** For AP systems, how long does data diverge during partitions. Measured as the time between the last consistent state and the first detected inconsistency.

**Hint queue depth.** For AP systems, how many hinted writes are pending. A growing queue indicates a sustained partition.

**Latency p99.** Track separately for normal operation and partition periods. A 10x increase during partitions is expected but should not persist.

**Stale read rate.** For AP systems, what percentage of reads return stale data. Track by key range and node.

### Alert Thresholds

| Metric | Warning | Critical | Action |
|--------|---------|----------|--------|
| Partition duration | > 30 seconds | > 5 minutes | Investigate network |
| Quorum loss | Any occurrence | > 1 minute | Failover or accept degraded |
| Hint queue depth | > 10,000 | > 100,000 | Scale up or investigate |
| Latency p99 (CP) | > 100ms | > 500ms | Check leader election |
| Latency p99 (AP) | > 50ms | > 200ms | Check compaction status |

### Upgrade Strategy

Major version upgrades require a cluster-wide upgrade. For CP systems, this means taking down the old cluster and bringing up the new one. Plan for downtime. Use tools like ZooKeeper's rolling upgrade feature when available.

For AP systems, upgrades are simpler. Because the system tolerates partitions, it tolerates different versions running simultaneously. Upgrade nodes one at a time. Run mixed versions for a short period. The anti-entropy and gossip protocols handle version differences.
