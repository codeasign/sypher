## Internal Architecture

A distributed system that must choose between CP and AP has several internal subsystems. These subsystems interact during normal operation and during partitions.

The architecture consists of five layers. The lowest layer is the network transport, which handles node-to-node communication. Above that is the partition detection layer, which monitors heartbeat signals and timeout thresholds. The consistency protocol layer implements the chosen strategy: consensus for CP systems, conflict resolution for AP systems. The replication layer handles data movement between nodes. The request routing layer directs client requests to the appropriate nodes.

### Partition Detection Mechanics

Partition detection is not binary. A node does not receive a "partition started" interrupt. Instead, it infers a partition from the absence of heartbeats. A heartbeat is a periodic message sent from each node to its peers. If a node misses N consecutive heartbeats within a timeout window T, it suspects a failure.

The challenge is distinguishing between a crashed node and a partitioned node. Both manifest as missing heartbeats. Cassandra uses the Phi Accrual failure detector, which computes a suspicion level based on historical heartbeat arrival times. If the gap is within normal variance, suspicion is low. If the gap exceeds historical patterns, suspicion rises.

The detection threshold directly affects the CAP trade-off. A short timeout detects partitions faster but causes more false positives (unnecessary failovers). A long timeout reduces false positives but extends the window of inconsistency. DynamoDB uses a 10-second timeout. ZooKeeper uses a configurable tick time, typically 2 to 20 seconds.

## Data Structures and Algorithms

### Quorum-Based Voting

Quorum is the minimum number of nodes that must agree on a read or write operation for it to succeed. The formula is standard: Write quorum (W) + Read quorum (R) > Replication factor (N). This ensures that any read quorum overlaps with any write quorum, guaranteeing at least one node has the latest write.

With N=3, you can configure W=2, R=2. This means any write must be acknowledged by 2 of 3 nodes, and any read must get responses from 2 of 3 nodes. The overlap guarantee: any 2 nodes share at least 1 common node, so the read will see the latest write.

The complexity of quorum operations is O(N) per request, where N is the replication factor. Each node must be contacted, and the coordinator must wait for W or R acknowledgments. Network round trips dominate.

### Vector Clocks

Vector clocks capture causality between events in a distributed system. Each node maintains a vector of (node, counter) pairs. When a node performs a write, it increments its own counter. When it receives a write from another node, it merges the remote vector clock: it takes the maximum of each counter and increments its own.

Two events are causally related if one vector clock is strictly greater than the other (every counter is >= the other, and at least one is >). Two events are concurrent if neither vector clock dominates the other. Concurrent events represent conflicts that must be resolved.

Vector clock size grows with the number of nodes that have written a key. In Dynamo, this is bounded by practical pruning: when a node has not written to a key for a configurable period, its entry can be removed from the clock.

The complexity of vector clock comparison is O(N) where N is the number of nodes in the vector. Storing vector clocks adds O(N) storage per key.

### Hinted Handoff

Hinted handoff is the mechanism AP systems use to survive partitions. When a write cannot reach its target replica, the coordinator stores the write locally with a hint: "this belongs to Node B." The coordinator stores the hint in a separate queue, keyed by the unreachable node.

When the coordinator detects that the target node is reachable again, it replays the hinted writes. The target node receives the writes and applies them. The hint queue is drained.

The hint queue is stored in local memory or disk. Its size is bounded: Cassandra limits hints to 3 hours of backlog by default. Beyond this, hints are discarded. You accept the data loss rather than unbounded queue growth.

The complexity of hinted handoff is O(1) per write (store the hint) and O(N) during recovery (replay all pending hints). Hinted handoff does not guarantee consistency on its own. It is a best-effort mechanism.

### Read Repair

Read repair fixes inconsistencies during read operations. When a read request arrives, the coordinator sends the request to all replicas (not just the read quorum). It compares the returned values. If they differ, the coordinator returns the latest value to the client and pushes the correct value to the stale replicas.

Read repair adds latency to read operations because the coordinator waits for all replicas to respond. The complexity is O(N) per read operation.

Cassandra performs read repair probabilistically: you can configure the chance of performing a full read repair on each read (default is 10 percent). Full read repair on every read is too expensive. Probabilistic repair balances consistency and performance.

## The Request Lifecycle

### CP System Write During Partition

The request lifecycle in a CP system during a partition has 10 steps.

Step 1. Client sends a write request to a load balancer. Step 2. Load balancer routes to Node A (which is on the minority side of the partition). Step 3. Node A receives the write request. Step 4. Node A checks its partition status: it has not received heartbeats from Nodes B and C for the timeout period. Step 5. Node A determines it is not part of the quorum. Step 6. Node A rejects the write request with an error: "Not enough replicas available." Step 7. The client receives the error response. Step 8. The client retries (optional, application-dependent). Step 9. The load balancer routes the retry to Node B (on the majority side). Step 10. Node B accepts the write, commits it locally, replicates to Node C (W=2 satisfied), and returns success.

Total latency: approximately 15-50ms for the failed attempt, then 5-15ms for the successful write on the majority side.

### AP System Write During Partition

In an AP system during a partition, the request lifecycle looks different.

Step 1. Client sends a write request. Step 2. Load balancer routes to Node A (isolated). Step 3. Node A receives the write. Step 4. Node A attempts to replicate to Nodes B and C. Both fail. Step 5. Node A stores the write locally. Step 6. Node A creates a hinted handoff entry for each unreachable replica. Step 7. Node A returns success to the client (W=1 is satisfied with the local write). Step 8. When the partition heals, Node A replays the hinted writes to Nodes B and C. Step 9. Nodes B and C apply the writes. Step 10. Data converges.

Total latency: 2-5ms for the write (single-node acknowledgment), plus 10-100ms deferred for hinted handoff replay.

## Consistency and State

### What Happens When a Node Crashes

A node crash differs from a partition. In a crash, the node stops responding entirely. The remaining nodes detect this through missing heartbeats.

In a CP system, the remaining nodes check if they still have a quorum. With N=3 and one crash, there are 2 remaining nodes. W=2 can still be satisfied (both remaining nodes acknowledge). R=2 can still be satisfied. The system continues operating.

In an AP system, the remaining nodes simply continue operating. The crashed node is removed from the replica list for new writes. Writes continue to succeed with the available nodes. When the crashed node returns, it receives the missed writes through hinted handoff or anti-entropy.

### What Happens During a Full Network Partition

A full partition splits the cluster into two groups that cannot communicate. Each group believes the other has crashed.

In a CP system with N=3, a 2-1 split means the group of 2 has the quorum and continues operating. The group of 1 recognizes it has lost quorum and stops accepting writes. When the partition heals, the single node synchronizes from the quorum.

In an AP system with N=3, both groups continue accepting writes. Data diverges. When the partition heals, conflict resolution runs. With last-write-wins, some writes are silently discarded.

### What Happens With Two Simultaneous Writes

In a CP system, two simultaneous writes to the same key go through the consensus protocol. The leader orders the writes. One write is committed first, the second is committed second. Both succeed, and all replicas see both writes in the same order. There is no conflict.

In an AP system, two simultaneous writes to the same key may go to different nodes. The writes are concurrent — they happened without knowledge of each other. With vector clocks, this is detected as a conflict. The system either applies last-write-wins (one write is silently dropped) or exposes the conflict to the application for resolution.

### What Happens With Stale Reads

Stale reads occur in AP systems when a client reads from a node that has not yet received a recent write. This is expected behavior in eventually consistent systems.

In DynamoDB, the default read consistency is eventually consistent. A read may return data that is up to 1 second stale. For strongly consistent reads, you explicitly request them. They cost more RCUs and may fail during partitions with a "SystemUnavailable" error.

The staleness window depends on: replication latency (how fast writes propagate), hint queue depth (how many pending writes exist), and conflict resolution (whether writes are still being reconciled).

## Performance Characteristics

### CP System Performance

CP systems pay a latency penalty for consistency. Every write must go through a consensus protocol, which requires multiple network round trips.

Typical p50 latency for a ZooKeeper write: 1-5 milliseconds. This includes the write to the leader and acknowledgment from a majority of followers. The p99 latency: 50-100 milliseconds, caused by garbage collection pauses, network congestion, or leader election in progress.

Read throughput: 10,000 to 50,000 reads per second per node. Reads are served by any node, but non-leader reads may return slightly stale data (ZooKeeper allows this with a configurable staleness limit).

Write throughput: 1,000 to 10,000 writes per second per node. The bottleneck is the single leader. All writes go through the leader, which serializes them.

### AP System Performance

AP systems are optimized for low latency. Writes can complete on a single node and still succeed.

Typical p50 latency for a Cassandra write: 1 millisecond. The coordinator writes to the local node and returns success. Replication happens asynchronously. The p99 latency: 10-50 milliseconds, caused by compaction events or garbage collection pauses.

Read throughput: 100,000 to 500,000 reads per second per node. Reads are served from local SSDs with no coordination overhead.

Write throughput: 100,000 to 500,000 writes per second per node. The lack of coordination and the ability to write to any node at any time enables massive write throughput.

### The Performance Gap

The throughput gap between CP and AP systems is approximately 10x for writes. This is the cost of strong consistency. Every CP write must coordinate with multiple nodes. Every AP write can succeed locally.

This gap drives architectural decisions. For high-traffic systems where write throughput is critical, AP systems are often chosen despite the consistency trade-off.

## Configuration and Tuning

### Consistency Levels

In Cassandra, you configure consistency per operation. The available levels are:

**ONE.** The coordinator waits for acknowledgment from 1 replica. Fastest, weakest consistency. Use for write-heavy workloads where data loss is acceptable (logging, metrics).

**QUORUM.** The coordinator waits for a majority of replicas. W=2 for N=3. Balanced performance and consistency. Use for general-purpose workloads where you need read-after-write consistency.

**ALL.** The coordinator waits for all replicas. W=3 for N=3. Slowest, strongest consistency. Use only when you must guarantee every replica has the data. A single slow or failed node makes the write fail.

**LOCAL_QUORUM.** Like QUORUM, but only counts replicas in the same datacenter. Use in multi-datacenter deployments to avoid cross-datacenter latency.

**EACH_QUORUM.** Waits for quorum in each datacenter. Slow but provides strong cross-datacenter consistency. Use for global applications that need consistent reads across regions.

### Safe Defaults vs Aggressive Settings

For production deployments, start with these safe defaults:

| Setting | Safe Default | Aggressive | Notes |
|---------|-------------|------------|-------|
| W (write consistency) | LOCAL_QUORUM | ONE | LOCAL_QUORUM for safety, ONE for throughput |
| R (read consistency) | LOCAL_QUORUM | ONE | LOCAL_QUORUM for consistency, ONE for performance |
| Replication factor | 3 | 3 minimum | Never use 1 or 2 in production |
| Hint timeout | 3 hours | 1 hour | Longer = safer, shorter = less storage use |
| Read repair chance | 0.1 | 0.01 | Lower = faster reads, higher = more consistency |

### The W+R > N Rule

The rule W+R > N ensures read-your-writes consistency. With N=3, W=2, R=2 satisfies the rule. W=3, R=1 also satisfies it (3+1 > 3). W=1, R=1 does NOT satisfy it (1+1 < 3). Reads can miss the latest write.

Set W and R based on your workload: read-heavy workloads benefit from low R (fast reads) and high W (consistent writes). Write-heavy workloads benefit from low W (fast writes) and high R (consistent reads).

Never set W+R to N or less unless you can tolerate stale reads. This is a common misconfiguration that leads to inconsistent behavior.
