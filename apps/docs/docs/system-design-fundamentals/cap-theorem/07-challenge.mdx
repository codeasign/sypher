## How to Use These Challenges

Each challenge has a time limit, evaluation criteria, and a model answer. Attempt the challenge within the time limit before reading the model answer. The challenges progress from foundation (5 minutes) through design (15 minutes) to expert (30 minutes).

Passing all challenges at a tier means you have mastered that level. Move to the next tier only after you consistently pass.

---

## Tier 1: Foundation

### Challenge 1.1: Explain the CAP Theorem (5 minutes)

**Prompt.** Write a 300-word explanation of the CAP theorem. Include the definition of each term, the constraint it imposes, and why it matters. Use no jargon beyond the three terms. Your audience is a junior engineer who has heard the term but does not understand it.

**Time limit.** 5 minutes.

**Evaluation criteria.** The explanation defines Consistency, Availability, and Partition Tolerance correctly. It states that the theorem applies during network partitions. It gives one concrete example of a CP system and one example of an AP system. It mentions that CA is not possible in distributed systems.

**Common mistakes.** Defining consistency as "all nodes see the same data" without mentioning that this applies to read-after-write. Stating that the theorem applies at all times rather than only during partitions. Claiming that CA systems are possible.

**Model answer.** The CAP theorem says a distributed data store can only guarantee two of three properties during a network partition. Consistency means every read returns the most recent write. Availability means every request gets a response. Partition Tolerance means the system continues operating despite network failures. During a partition, you must pick two. ZooKeeper is a CP system — it rejects requests on the minority side. Cassandra is an AP system — it accepts writes on both sides. A CA system would sacrifice partition tolerance, which is not a real option in a distributed system. The theorem matters because it forces you to make explicit trade-offs. You cannot build a system that is perfectly consistent and always available across multiple nodes. You must choose which property to sacrifice.

### Challenge 1.2: CP vs AP Decision (5 minutes)

**Prompt.** For each use case, decide whether CP or AP is the right choice. Justify each decision in one sentence.

Use cases: a) DNS resolution service, b) Distributed lock for leader election, c) Shopping cart for an e-commerce site, d) Financial transaction ledger, e) Social media news feed.

**Time limit.** 5 minutes.

**Evaluation criteria.** Each choice is correct and justified. DNS: AP (stale records are acceptable, unavailability is not). Lock service: CP (split-brain is catastrophic). Shopping cart: AP (losing a write is worse than showing stale items). Financial ledger: CP (inconsistency can cause financial loss). Social feed: AP (stale posts are acceptable, downtime is not).

**Common mistakes.** Choosing CP for the shopping cart without considering that the cart must always accept writes. Choosing AP for the lock service without understanding the split-brain risk. No justification provided.

**Model answer.** DNS is AP: stale records are handled by TTL and retries. Lock service is CP: two nodes holding the same lock causes data corruption. Shopping cart is AP: losing a customer's add-to-cart action is worse than showing a slightly stale cart. Financial ledger is CP: inconsistent ledger entries can cause overdrafts or incorrect balances. Social feed is AP: showing a 10-second-old post is fine; showing an error is not.

### Challenge 1.3: Quorum Math (5 minutes)

**Prompt.** You have a 5-node cluster. Calculate the minimum W and R for read-your-writes consistency. Then calculate what happens to availability if one node fails. What if two nodes fail?

**Time limit.** 5 minutes.

**Evaluation criteria.** Correct calculation of W+R > N. Correct determination of quorum size and failure tolerance. N=5, minimum W+R > 5: options include W=3, R=3 (total 6) or W=4, R=2 (total 6). With W=3, R=3, one node failure still allows quorum (3 of 4 remaining nodes). Two node failures: 3 of 3 remaining, still possible. If 3 nodes fail, quorum is lost. Availability degrades gracefully.

**Common mistakes.** Forgetting that W+R must be strictly greater than N, not equal. Confusing replication factor N with cluster size.

**Model answer.** With N=5, W=3, R=3 satisfies W+R > N (6 > 5). The write quorum is 3 nodes. The read quorum is 3 nodes. With one node failure, 4 nodes remain. W=3 is still achievable (3 of 4). With two node failures, 3 nodes remain. W=3 is still achievable (3 of 3). With three node failures, 2 nodes remain. W=3 is not achievable. The system becomes unavailable for writes. Availability degrades from 5 nodes down to 3 nodes for writes.

## Tier 2: Design

### Challenge 2.1: API Rate Limiter Config Store (15 minutes)

**Prompt.** Design a configuration store for a distributed API rate limiter. The rate limiter runs on 100 nodes across 3 datacenters. Configuration changes (rate limits, allowlists, blocklists) are updated by administrators approximately 10 times per day. All 100 nodes must read and apply the latest configuration within 5 seconds of a change. The system must never serve stale configuration that would allow more requests than the configured limit. Availability is important but can tolerate brief unavailability during updates.

Draw the architecture. State whether your design is CP or AP. Justify your choice. Explain what happens during a partition.

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct identification that this is a CP use case. Configuration store must be strongly consistent. ZooKeeper or etcd is the right choice. Architecture shows the config store cluster, the 100 rate limiter nodes as watchers, and the update path. Partition behavior explained: minority side cannot update config, but the majority side continues serving. Rate limiters on the minority side serve the last known configuration.

**Common mistakes.** Using an AP database like Cassandra. Suggesting a file-based configuration pushed to each node. Not handling partitions.

**Model answer.** This is a CP use case. Configuration changes must be strongly consistent. I use etcd with 5 nodes across 3 datacenters. The 100 rate limiter nodes use etcd's watch API to receive real-time configuration updates. On an update, the administrator writes to the etcd leader, which replicates to a quorum of 3 nodes. The watch API pushes the change to all 100 rate limiters within 5 seconds. During a partition, the minority side of etcd rejects writes. Rate limiters connected to the minority side cannot update configuration, but they continue rate-limiting based on the last known configuration. This is acceptable because stale configuration is safe (it was the approved configuration at the time it was loaded), and unavailability during partitions is brief. If the partition isolates a rate limiter from etcd entirely, the rate limiter uses its local cached configuration.

### Challenge 2.2: Global Comment System (15 minutes)

**Prompt.** You are building a comment system for a social network with 500 million monthly active users. Users post comments. Other users read them. Comments must be visible within 2 seconds of posting. Duplicate or out-of-order comments are annoying but not catastrophic. The system must never go down. Design the database layer.

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct identification that this is an AP use case. High availability requirement and tolerance for eventual consistency make AP the right choice. DynamoDB or Cassandra is appropriate. Architecture includes: comment storage, timeline indexing, caching. Partition behavior: both sides accept writes, conflicts resolved by timestamp. Reads served from cache or nearest replica.

**Common mistakes.** Proposing a CP system that could reject writes during partitions. Ignoring the 2-second visibility requirement. Not caching.

**Model answer.** This is AP. Comments must never be lost. I use DynamoDB with eventually consistent reads. The comment table is keyed by (post_id, created_at). Writes use W=1 (fast, always succeeds). Reads use eventually consistent (sub-millisecond, may be 1 second stale). A write-through cache (ElastiCache) serves the most recent comments for each post. During a partition, all writes succeed. Some reads may return slightly stale data. When the partition heals, hinted handoff and read repair restore consistency. Conflict resolution uses last-write-wins by client timestamp. The 2-second visibility requirement is met because writes are visible as soon as they reach any node.

### Challenge 2.3: Hybrid CP-AP Architecture (15 minutes)

**Prompt.** Design a multi-service architecture where one service requires strong consistency (user authentication) and another service requires high availability (user profile updates). Both services share the same user database. How do you design the database layer?

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct separation of concerns. Different consistency levels for different data. Option 1: separate databases for auth and profile. Option 2: single database with tunable consistency. Architecture diagram showing both paths. Justification for each choice.

**Common mistakes.** Forcing both workloads into the same consistency model. Using AP for auth data. Using CP for profile data without considering the availability impact.

**Model answer.** I use two separate databases. Authentication data uses a CP system (etcd-backed or PostgreSQL with synchronous replication). Profile data uses an AP system (Cassandra or DynamoDB). The services are decoupled: auth service reads and writes auth data only; profile service reads and writes profile data only. A user ID links both databases, but there is no cross-database transaction. Auth updates are consistent and may fail during partitions (the user cannot log in). Profile updates always succeed, even during partitions (the user can always update their bio). This hybrid approach respects the different requirements of each workload.

## Tier 3: Expert

### Challenge 3.1: Horizontal Scaling for Billing (30 minutes)

**Prompt.** You are the systems architect for a cloud provider's billing system. The system processes 10 million billing events per hour. Each event represents a usage record for a customer resource (compute hours, storage GB-hours, data transfer GB). Events arrive from 10,000 services across 5 datacenters. The billing system must provide accurate invoices at the end of each billing cycle. A single dollar of overbilling or underbilling across 10 million customers is unacceptable. The system must also be available 99.99 percent of the time because delayed billing events cause revenue leakage.

Design the billing event ingestion pipeline and the billing calculation engine. Address the CAP constraints explicitly. Explain how you handle partitions, data loss, and duplicate events.

**Time limit.** 30 minutes.

**Evaluation criteria.** Deep understanding of CAP applied to a complex system. Recognition that different parts of the pipeline need different CAP properties: ingestion is AP (accept all events), calculation must be CP (correct totals). Use of idempotency keys to handle duplicates. Use of a deterministic reconciliation process. Quantified scale: 10M events/hour = 2,778 events/second peak. Storage: roughly 2 TB/month for raw events.

**Common mistakes.** Using a single CP system for the entire pipeline, causing ingestion failures during partitions. Using AP for the billing calculation, causing incorrect invoices. Not handling duplicate events. Not quantifying storage or throughput.

**Model answer.** The pipeline has three stages with different CAP requirements.

Stage 1: Ingestion (AP). Events arrive from 10,000 services. Each event has a unique idempotency key (resource_id + timestamp + sequence). All events must be accepted. I use Kafka with 16 partitions, replication factor 3, acks=1. Kafka is AP: during a partition, producers continue writing to available partitions. The key insight: losing an event is worse than ingesting a duplicate, because duplicates are detected by idempotency keys.

Stage 2: Deduplication and ordering (CP). The event stream must be processed exactly once. Kafka Streams or Apache Flink reads from Kafka, deduplicates by idempotency key, and orders events by timestamp within each resource_id. This stage requires exactly-once semantics, which is a CP guarantee. Flink's checkpointing with exactly-once mode ensures that every event is processed exactly once, even during failures. The checkpoint state is stored in a CP system (HDFS or S3 with strong consistency).

Stage 3: Calculation (CP). The billing calculation engine reads the deduplicated event stream and computes running totals per customer per billing period. Calculation results are stored in a CP database (Spanner or CockroachDB with strong consistency). During a partition, the calculation engine pauses if it cannot reach the CP database, because an incorrect bill is worse than a delayed bill. The 99.99 percent availability requirement is met by operating multiple calculation instances with leader election.

Duplicate handling: The idempotency key ensures that the same event is not counted twice. If a service retries a failed write, the deduplication stage detects the duplicate by the key and discards it.

Partition handling: If a partition isolates a datacenter, Kafka producers in that datacenter continue writing to local Kafka brokers. The brokers replicate to other datacenters when the partition heals. The deduplication stage processes events from all datacenters. Because events are ordered by timestamp within each resource_id, the final calculation is deterministic regardless of processing order. The billing cycle runs at the end of the month. If a partition is still ongoing, the cycle delays until the partition heals. The business accepts a 24-hour billing delay in extreme cases rather than an incorrect bill.

### Challenge 3.2: Multi-Region Social Graph (30 minutes)

**Prompt.** Design a graph database for a professional social network with 800 million users. The graph stores connections (edges) between users. Each user has an average of 200 connections. The graph must support: read a user's connections (latency under 50ms p99), add a connection (latency under 100ms p99), suggest new connections (batch job, 1 hour latency acceptable), and path traversal (degrees of separation, 500ms p99). The system must be globally available. The strongest consistency requirement is that if User A connects to User B, the connection must be visible to both users within 5 seconds.

**Time limit.** 30 minutes.

**Evaluation criteria.** Understanding that a social graph has different CAP requirements for different operations. Connection adds are write-heavy and need AP (accept all connection requests, propagate within 5 seconds). Path traversal is read-heavy and can tolerate slight staleness. Sizing: 800M users * 200 connections = 160 billion edges. Edge storage: approximately 160B * 64 bytes (user_id pair + timestamp + metadata) = 10 TB for the raw graph. With replication (3x) = 30 TB. Edge creation rate: assuming 10 percent of users create 1 connection per month, approximately 30 new connections per second.

**Common mistakes.** Proposing a single CP database for all graph operations, making writes too slow. Not addressing the 160 billion edges scale. Ignoring the global availability requirement.

**Model answer.** The graph is stored in a custom sharded architecture with different CAP properties per operation.

Connection storage (AP): Each user's adjacency list is stored as a DynamoDB item keyed by user_id. Adding a connection writes to the adjacency lists of both users. Write: W=1, consistent append. If one write succeeds and the other fails (partition), the inconsistency is repaired by an anti-entropy job that scans for half-written connections. The 5-second visibility requirement is met by DynamoDB Global Tables with 1-second replication delay.

Path traversal (AP with local strong consistency): Breadth-first search queries read adjacency lists from local DynamoDB replicas. Stale data (a missing recent connection) causes a slightly incomplete path but does not break correctness. Queries are served from the nearest region.

Connection suggestions (batch, no real-time CAP requirement): A Spark job runs hourly, processing the graph snapshot from the previous hour. The snapshot is a consistent read from DynamoDB at a specific timestamp. The job computes suggested connections using collaborative filtering and writes results to a CP cache (Redis with replication).

Partition handling: During a partition, connection adds succeed in each region. When the partition heals, the anti-entropy job detects and repairs divergent adjacency lists. Path traversal queries continue returning results (may be slightly stale). The system never goes down.

The key insight: a social graph does not need strong consistency for connections. If a connection is delayed by a few seconds, the user experience is acceptable. The system optimizes for availability and write throughput.
