### What Breaks at Each Scale Tier

**At 1,000 users (daily active):** The naive monolith handles everything. Response times are under 50ms. Database CPU sits at 5%. The 8 Gunicorn workers handle 80 concurrent requests without breaking a sweat. Deployments take 30 seconds. No monitoring is needed -- you notice issues because users email you.

**At 10,000 users:** The database CPU hits 40%. Slow queries in the analytics module start appearing. The recommendation engine, which joins 6 tables to generate personalized suggestions, takes 2 seconds per query. You add database indexes and the problem improves. The in-memory cache on the web server reduces redundant queries by 60%.

**At 100,000 users:** The database primary is at 80% CPU. Connection pooling is tuned but the pool frequently saturates at peak hours. You add read replicas -- one for the analytics queries, one for the recommendation engine. Write traffic still goes to the primary. Deployments now take 15 minutes because the CI pipeline has grown. A deployment on Friday takes down the site for 90 seconds during process restart.

**At 1,000,000 users:** The single database primary cannot handle the write throughput. 500 writes/second against a single PostgreSQL instance with 40 tables and 120 indexes creates contention. The in-memory cache is useless because there are now 6 application servers and each has its own cache, resulting in 70% cache misses. The CI pipeline takes 4 hours. A single broken test blocks all deploys. The team spends 30% of engineering time on release coordination.

**At 10,000,000 users:** The monolith is physically impossible. No single database instance can handle the volume. No single process can contain 50 million lines of code. The system must be decomposed.

## Integration Patterns

### Pattern 1: Database-Per-Service

Each microservice owns its data exclusively. No other service directly accesses its database. Data that crosses boundaries is synchronized through API calls or events.

```python
# Video Service -- owns the videos database
class VideoService:
    def get_video(self, video_id):
        # Only this service accesses the videos database
        return self.video_repo.find_by_id(video_id)

    def upload_video(self, user_id, file, metadata):
        video = Video(user_id=user_id, metadata=metadata)
        self.video_repo.save(video)
        # Publish event for other services
        self.event_bus.publish("video.uploaded", {
            "video_id": video.id,
            "user_id": user_id,
            "duration": metadata.duration
        })
        return video

# Notification Service -- consumes the event
class NotificationService:
    def handle_video_uploaded(self, event):
        user = self.auth_client.get_user(event.user_id)  # API call to Auth Service
        self.send_email(user.email, f"Your video {event.video_id} is processing")
```

Key enforcement: the database is accessed only through the owning service's API. No direct cross-service database queries. Database credentials for each service are secrets that only that service knows.

### Pattern 2: Event-Driven Communication

Services communicate through asynchronous events rather than synchronous requests. A service publishes an event to a message broker; interested services consume it and react. This pattern reduces coupling and improves resilience.

```python
# Publishing service (Video Upload)
def on_video_uploaded(video):
    event = {
        "type": "video.uploaded",
        "version": 1,
        "timestamp": datetime.utcnow().isoformat(),
        "data": {
            "video_id": str(video.id),
            "user_id": str(video.user_id),
            "duration_seconds": video.duration,
            "file_size_bytes": video.file_size,
            "content_type": video.content_type
        }
    }
    # Publish with key for partition ordering
    kafka_producer.produce("video.uploaded", key=str(video.id), value=json.dumps(event))

# Consuming service (Transcoding)
def handle_video_uploaded(event):
    video_id = event["data"]["video_id"]
    duration = event["data"]["duration_seconds"]

    if duration > 600:  # Videos over 10 minutes get different transcoding
        transcode_profile = "high_quality"
    else:
        transcode_profile = "standard"

    start_transcoding(video_id, transcode_profile)
```

Event contracts are versioned: when an event schema changes, the publisher increments the version number. Consumers process multiple versions simultaneously during migration windows.

### Pattern 4: Service Mesh

A service mesh moves inter-service communication concerns from application code to an infrastructure layer. A sidecar proxy runs alongside each service instance, handling retries, timeouts, circuit breaking, mutual TLS, and observability.

With a service mesh, service code does not implement retry logic or circuit breakers. The sidecar proxies handle it. The Video Service makes a plain HTTP call to `http://comment-service:8080` -- the sidecar intercepts the call, applies policies, encrypts with mTLS, and retries on failure.

Service mesh implementation using Istio:

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: comment-service-routing
spec:
  hosts:
  - comment-service
  http:
  - match:
    - uri:
        prefix: /api/v1/comments
    route:
    - destination:
        host: comment-service
        port:
          number: 8080
      weight: 90  # 90% traffic to stable
    - destination:
        host: comment-service-canary
        port:
          number: 8080
      weight: 10  # 10% traffic to canary
    retries:
      attempts: 3
      perTryTimeout: 2s
    timeout: 10s
    fault:
      delay:
        percentage:
          value: 1.0  # Inject 1 second delay on 1% of requests
        fixedDelay: 1s
```

## Deployment and Operations

### Zero-Downtime Deploys

Zero-downtime deployment ensures that users never see errors during a service update. The standard pattern for microservices:

**Rolling update with Kubernetes:**
1. A new ReplicaSet is created with the updated container image.
2. The old ReplicaSet has 6 instances; the new one creates 2 instances.
3. Once the new instances pass health checks, traffic is routed to them.
4. Old instances are terminated one at a time, waiting for in-flight requests to complete.
5. Process continues until all 6 instances are replaced.

This works because each service has multiple instances. During the rollout, 4 old + 2 new = 6 total instances serve traffic. Capacity is maintained. Health checks ensure new instances work before receiving traffic.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: comment-service
spec:
  replicas: 6
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2        # Allow 2 extra pods during update
      maxUnavailable: 1  # Only 1 pod can be down at a time
  template:
    spec:
      containers:
      - name: comment-service
        image: streamflow/comment-service:v2.1.3
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /health/live
            port: 8080
          initialDelaySeconds: 15
          periodSeconds: 20
```

### Monitoring and Debugging

Debugging a microservices failure requires correlating data across 12 services. The observability stack provides three pillars:

**Logging**: Each service writes structured JSON logs to stdout. Fluentd collects logs and ships them to Elasticsearch. Each log entry includes `service_name`, `trace_id`, `span_id`, and `request_id` for correlation.

```json
{
  "timestamp": "2026-07-02T14:30:00.123Z",
  "level": "ERROR",
  "service": "video-upload-service",
  "trace_id": "abc123def456",
  "span_id": "span789",
  "message": "Failed to upload file to S3",
  "error": "RequestTimeout",
  "video_id": "vid_98765",
  "file_size_mb": 150,
  "duration_ms": 30020
}
```

**Metrics**: Prometheus scrapes metrics from each service. Key metrics: request count, error rate (5xx/total), latency histogram (p50, p99, p999), active connections, queue depth, database pool utilization. Grafana dashboards show overall system health and per-service drill-downs.

**Distributed Tracing**: Jaeger traces a single request across all services it touches. A video upload trace might show: API Gateway (5ms) -> Auth Service (12ms) -> Video Upload Service (450ms) -> S3 PUT (400ms) -> Kafka publish (5ms). Each span shows exact timing, error state, and tags.

### Database Migrations

Schema changes in a microservices architecture require backward-compatible deployments. The expand-migrate-contract pattern:

1. **Expand**: Add the new column or table alongside existing schema. Both old and new code paths work.
2. **Migrate**: Deploy the updated service code that uses the new schema. Old code paths are deprecated but still present.
3. **Contract**: Remove the old column or table once all instances are running the new code.

```python
# Phase 1: Expand (deploy first)
class VideoRepository:
    def save(self, video):
        # Both old and new columns present
        self.db.execute("""
            INSERT INTO videos (id, user_id, title, format, new_format)
            VALUES (%s, %s, %s, COALESCE(%s, 'mp4'), %s)
        """, [video.id, video.user_id, video.title, video.old_format, video.format])

# Phase 2: Switch to new field (deploy after all instances have new code)
class VideoRepository:
    def save(self, video):
        # Only new field used, old field still exists
        self.db.execute("""
            INSERT INTO videos (id, user_id, title, new_format)
            VALUES (%s, %s, %s, %s)
        """, [video.id, video.user_id, video.title, video.format])

# Phase 3: Remove old column (run migration after code deploy)
# ALTER TABLE videos DROP COLUMN format;
```

This pattern avoids downtime. A migration that requires locking a large table is broken into chunks: `UPDATE videos SET new_format = format WHERE id BETWEEN 0 AND 10000` repeated until all rows are migrated.
