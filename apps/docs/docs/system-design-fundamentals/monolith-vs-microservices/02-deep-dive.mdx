# Monolith vs Microservices -- Under the Hood

## Internal Architecture

A monolith runs as a single operating system process. All modules share memory, thread pools, and class loaders. A microservices system runs as N independent processes -- potentially on different machines -- communicating over the network. The internal structure of each case is fundamentally different.

## Data Structures and Algorithms

### Service Registry

When a microservice starts, it registers its network location with a service registry. When another service needs to call it, the caller queries the registry to find available instances. This is the fundamental data structure that enables dynamic service discovery.

```
Service Registry Entry:
{
  "service_name": "video-upload-service",
  "instance_id": "i-0a1b2c3d4e5f",
  "host": "10.0.1.45",
  "port": 8080,
  "health_check_url": "/health",
  "metadata": {
    "version": "2.1.3",
    "region": "us-east-1",
    "environment": "production"
  },
  "ttl": 30,              // seconds until entry expires
  "registered_at": 1719876542
}
```

The registry stores this data in a distributed key-value store (etcd, Consul) or DNS-based system (Kubernetes CoreDNS). Services send heartbeats every TTL/3 seconds to keep their entry alive. A missing heartbeat for TTL seconds causes the entry to be removed.

### Load Balancing Algorithms

Load balancers distribute requests across service instances using one of these algorithms:

**Round Robin**: Requests are distributed sequentially to each instance in order. Instance 1 gets request 1, instance 2 gets request 2, and so on. Simple but does not account for instance capacity or current load. Works well when instances are homogeneous and requests have similar cost.

**Least Connections**: The load balancer sends the next request to the instance with the fewest active connections. More adaptive than round robin -- it naturally routes traffic away from slower or overloaded instances. The balancer maintains a counter `active_connections[instance_id]` that increments on new connection and decrements on close.

**Weighted Distribution**: Each instance gets a weight based on its capacity (CPU, memory). A 2x instance receives twice the requests of a 1x instance. Weights are set at deploy time and can be adjusted during operations. Algorithm: generate a random number proportional to total weight, find which instance it falls into.

**Consistent Hashing**: Routes requests based on a hash of the request key (user ID, session ID) to ensure the same client reaches the same instance. This preserves session locality. Uses a hash ring: each instance maps to multiple positions on the ring. Key lookup: hash the key, find the next instance clockwise.


### Circuit Breaker State Machine

A circuit breaker prevents cascading failures by stopping requests to a failing service. It implements a three-state machine:

```
State Machine:
  ┌──────────┐     failure > threshold     ┌──────────┐
  │  CLOSED  │ ──────────────────────────>  │  OPEN    │
  │ (normal) │                              │(reject)  │
  └──────────┘                              └──────────┘
       ^                                          │
       │     success > threshold                  │ timeout
       │     ┌──────────────────────────┐         │
       │     │                          │         │
       │     v                          │         v
       │  ┌──────────┐     failure      ┌──────────┐
       └──│  HALF-   │ ──────────────>  │  OPEN    │
          │  OPEN    │                  │(reject)  │
          └──────────┘                  └──────────┘
```

Transition rules:
- **CLOSED**: Requests pass through normally. Count failures. After `failure_count >= threshold` (typically 5) within a rolling window, transition to OPEN.
- **OPEN**: Requests are rejected immediately without calling the downstream service. A timer starts (typical timeout: 30 seconds). When the timeout expires, transition to HALF-OPEN.
- **HALF-OPEN**: Allow a probe request through. If it succeeds, transition to CLOSED and reset the failure count. If it fails, transition back to OPEN and reset the timer.

Implementation in Python-style pseudocode:

```python
class CircuitBreaker:
    def __init__(self, threshold=5, timeout_ms=30000):
        self.state = "CLOSED"
        self.failure_count = 0
        self.threshold = threshold
        self.last_failure_time = 0
        self.timeout_ms = timeout_ms

    def call(self, fn, fallback_fn=None):
        if self.state == "OPEN":
            if (current_time_ms() - self.last_failure_time) > self.timeout_ms:
                self.state = "HALF_OPEN"
            else:
                return fallback_fn() if fallback_fn else None

        try:
            result = fn()
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
                self.failure_count = 0
            return result
        except Exception:
            self.failure_count += 1
            self.last_failure_time = current_time_ms()
            if self.failure_count >= self.threshold:
                self.state = "OPEN"
            return fallback_fn() if fallback_fn else None
```

### Message Routing Tables

In an event-driven microservices system, message brokers maintain routing tables that determine which service receives which event.

```
Kafka Topic Routing Table (internal):
┌──────────────────────┬──────────────────────────┬──────────┐
│ Topic                │ Consumer Group           │ Partition│
├──────────────────────┼──────────────────────────┼──────────┤
│ video.uploaded       │ transcoding-service      │ 0, 1, 2  │
│ video.uploaded       │ notification-service     │ 0, 1, 2  │
│ video.uploaded       │ analytics-service        │ 0, 1, 2  │
│ comment.posted       │ notification-service     │ 0, 1     │
│ comment.posted       │ moderation-service       │ 0, 1     │
│ user.signed_up       │ onboarding-service       │ 0        │
│ user.signed_up       │ email-service            │ 0        │
│ payment.completed    │ billing-service          │ 0, 1, 2  │
│ payment.completed    │ notification-service     │ 0, 1, 2  │
└──────────────────────┴──────────────────────────┴──────────┘
```

Each consumer group represents a logical subscriber. Kafka partitions the topic to distribute load: if the topic has 3 partitions and the consumer group has 3 instances, each instance consumes one partition. Routing decisions are based on the partition key -- typically a business key like `user_id` or `video_id`.

## The Request Lifecycle

### Request Flow Through a Monolith

A user uploads a video through a monolithic application. Here is the exact sequence of events, measured at a single-server deployment:

1. Client POSTs multipart form data to `https://streamflow.example/videos/upload` -- the request arrives at the web server.
2. The web server (Nginx) receives the TCP connection, terminates TLS, and forwards the HTTP request to the application server (Gunicorn) on `127.0.0.1:8000`.
3. Gunicorn assigns the request to a worker thread from its pool (50 threads configured, 14 currently active).
4. The Django middleware chain runs: CORS headers set, CSRF token validated, session loaded from the shared PostgreSQL database (5ms query).
5. The URL router dispatches to `VideoUploadView.post()`.
6. `VideoUploadView` calls `authenticate(request)` which queries the `sessions` table -- another 3ms database query.
7. The handler reads the file stream and writes it to local disk (`/data/uploads/`). For a 50 MB video, this takes 200ms on an SSD.
8. The handler inserts a `Video` record into the `videos` table. The ORM issues an INSERT with 12 columns. Takes 15ms.
9. After the response is sent, the `VideoUploadView` calls `transcode_video.delay(video_id)` -- a Celery async task. This returns control to the client within 50ms but enqueues a background job.
10. The Celery worker picks up the transcode job, spawns an FFmpeg process, and transcodes the video. This consumes 80% CPU for 30 seconds.
11. During transcription, the Celery worker holds a database connection from the shared pool. The pool now has 15 of 50 connections in use.
12. If another request arrives during transcription and the pool is exhausted, the request blocks with a connection timeout error after 30 seconds.

**Total user-facing latency**: approximately 300ms to receive the HTTP 201 response. The transcode happens asynchronously.

### Request Flow Through Microservices

The same operation in a microservices architecture:

1. Client POSTs multipart form data to `https://api.streamflow.example/videos` -- the request hits the API Gateway (Kong).
2. The API Gateway authenticates the request by calling the Auth Service at `http://auth-service:8080/verify-token`. This is an internal gRPC call with a 50ms timeout. The Auth Service queries its own database (2ms) and returns the user ID and roles.
3. The API Gateway routes the authenticated request to the Video Upload Service at `http://video-upload:8080/upload`.
4. The Video Upload Service validates the file type, size (limit: 500 MB), and metadata. It writes the file to S3-compatible object storage (MinIO). Upload to object storage takes 400ms for a 50 MB file over the internal network.
5. The Video Upload Service inserts a record into its own PostgreSQL database (the `videos` table only -- no joins needed). Takes 10ms.
6. The Video Upload Service publishes a `VideoUploadedEvent` to the Kafka topic `video.uploaded` with key `video_id=12345`. This is an async fire-and-forget operation that returns in 5ms.
7. The API Gateway returns HTTP 201 to the client. Total latency: approximately 470ms.
8. Meanwhile, the Transcoding Service -- which subscribes to the `video.uploaded` topic -- receives the event from its Kafka consumer group.
9. The Transcoding Service fetches the video from S3 and starts FFmpeg in a sidecar container with CPU and memory limits (2 vCPUs, 4 GB RAM). This does not affect the Video Upload Service at all.
10. When transcription completes, the Transcoding Service updates its own database and publishes a `VideoTranscodedEvent` to the `video.transcoded` topic.
11. The Notification Service picks up the `VideoTranscodedEvent` and sends a push notification to the user.

**Total user-facing latency**: approximately 470ms for the upload response. The transcode processing is fully isolated from the upload path.

## Consistency and State

### Crashes

**Monolith crash**: The single process dies. All in-flight requests drop. Active database transactions roll back. The shared in-memory cache is lost. Recovery requires restarting the process -- typically 10-30 seconds of downtime. During this time, zero requests are served.

**Microservice crash**: One service instance dies. The API gateway detects it via health check failure (typically 2-3 missed heartbeats over 10-15 seconds). In-flight requests to that instance fail with HTTP 503. The load balancer removes the dead instance from rotation. Other instances serve traffic with zero disruption. If the entire service type crashes (all instances), requests to that specific capability fail, but the rest of the system continues functioning.

### Partitions

A network partition occurs when two services cannot communicate but both are running.

**Monolith**: No network partitions -- all code runs in one process. The monolith either has connectivity to its database or it does not. Simpler failure model.

**Microservices**: Service A cannot reach Service B, but both are healthy. If Service A depends on Service B synchronously, requests that cross the partition fail. If dependent calls use timeouts and circuit breakers, failures are contained. If communication is event-driven, events buffer in Kafka until the partition heals. Data consistency requires careful handling: the reading service may have stale data from before the partition.

### Simultaneous Writes

**Monolith**: The database handles concurrent writes through ACID transactions. Two users updating the same video record use row-level locking. The second write waits for the first transaction to commit. This guarantees serializability.

**Microservices**: Two services may hold copies of the same logical data (user name stored in Auth Service and duplicated in Video Upload Service). If a user updates their name through the Auth Service and the Video Upload Service has a cached copy, stale data exists until an event propagates the change. This is eventual consistency -- the system accepts temporary inconsistency in exchange for availability and partition tolerance.

### Stale Reads

A stale read occurs when a read returns data that does not reflect the latest write. In a monolith, stale reads are rare because all code reads from the same database. In microservices, stale reads are common and expected.

Consider a user changing their display name:
1. User updates display name via the Profile Service. The Auth Service database now has `display_name = "NewName"`.
2. The Auth Service publishes `UserProfileUpdated { user_id: 42, display_name: "NewName" }` to Kafka.
3. The Comment Service has a local cache of display names. It has not yet processed the Kafka event.
4. User views their comment -- the page shows the old display name.
5. Event processes 200ms later -- cache updates. Next page load shows the correct name.

This tradeoff is explicit: microservices sacrifice read-your-writes consistency for independent deployability.

## Performance Characteristics

### Latency

A microservices call chain adds latency at every hop. Here are realistic p50, p99, and p999 latencies for each communication type:

| Hop Type | p50 | p99 | p999 |
|----------|--------|---------|----------|
| In-process function call (monolith) | 0.001ms | 0.01ms | 0.1ms |
| Localhost HTTP to another service | 0.5ms | 3ms | 50ms |
| Network call within same AZ | 1ms | 10ms | 100ms |
| gRPC within same AZ | 0.5ms | 5ms | 30ms |
| Cross-AZ network call | 3ms | 50ms | 500ms |
| Database query (simple, indexed) | 2ms | 20ms | 200ms |
| Kafka publish + consume | 5ms | 100ms | 1000ms |

A monolith serving a page that requires 10 database queries and 50ms of computation completes in approximately 70ms. The same page decomposed into 5 microservices with 2 service hops, 2 database queries per service, and message serialization adds up:

- API Gateway to Auth Service: 1ms
- Auth Service DB query: 2ms
- Auth Service response: 1ms
- API Gateway to Video Service: 1ms
- Video Service DB query: 2ms
- Video Service response: 1ms
- API Gateway to Comment Service: 1ms
- Comment Service DB query: 2ms
- Comment Service response: 1ms
- API Gateway aggregation: 1ms
**Total: approximately 13ms + serialization overhead = 15-20ms**

The microservices version is faster here because each service has a smaller, more focused database. But add two more hops for a more complex query and latency increases. The breakeven point depends on query complexity and data volume.

### Throughput

**Monolith throughput** is limited by the single process. A typical Django/Gunicorn setup with 4 workers handles approximately 400 requests/second on a 4-core server. Bottleneck is the Global Interpreter Lock (GIL) in Python or the thread-per-request model in Java.

**Microservices throughput** scales with instance count. Each service can have 50 instances behind a load balancer, each handling 200 RPS, yielding 10,000 RPS for that service. Different services scale at different rates -- the Video Upload Service needs more instances during peak hours, while the Auth Service needs fewer.

### Memory Footprint

A monolith consumes memory for the full application stack. A Python monolith with Django, DRF, Celery, and Redis client might use 500 MB to 1 GB RSS for the main process.

A microservices architecture consumes more total memory because each service runs its own process with its own runtime. A deployment with 12 service types at 3 instances each (36 processes) might consume:
- 6 Go services at 50 MB each: 300 MB
- 4 Python services at 200 MB each: 800 MB
- 2 Java services at 512 MB each: 1 GB
**Total: 2.1 GB** -- significantly more than the monolith's 1 GB, but the memory is spread across many machines.

### Network Overhead

In a monolith, network calls are zero -- everything is in-process. In microservices, every cross-service call is a network round-trip with TCP connection overhead, serialization/deserialization, and potential retries.

A system handling 10,000 requests/second with an average fan-out of 3 service calls per request generates 30,000 internal requests/second. Each request involves:
- TCP three-way handshake (if not keep-alive): 3 round-trips
- HTTP headers: 200-500 bytes
- JSON serialization/deserialization: 0.1-0.5ms
- gRPC protobuf serialization: 0.02-0.1ms

Using persistent connections and binary protocols (gRPC, Thrift) reduces overhead by 5-10x compared to short-lived HTTP/JSON connections.

## Configuration and Tuning

### Decomposition Strategies

**Domain-first decomposition**: Start by identifying business domains (bounded contexts). Each domain becomes one service. This produces 5-15 services for a typical business application. Netflix uses approximately 700 services for a global streaming platform. Uber has approximately 2,200 services. The number is driven by business complexity, not technical preference.

**Granularity rules of thumb:**
- A service should be ownable by a single team of 6-8 engineers.
- A service should have at most 50 database tables.
- A service should deploy independently of other services at least 95% of the time.
- A new feature should require changes in at most 2 services (ideally 1).
- If you cannot describe what a service does in one sentence, it is too coarse-grained or too fine-grained.

### Service Sizing

Each service should own enough data and logic to be useful independently. Common sizing mistakes:

- **Too large**: The service covers multiple bounded contexts. Teams step on each other. Deployment coordination increases. Extract until each service has a clear single responsibility.
- **Too small**: The service is a CRUD wrapper around a single database table. Teams spend more time on inter-service communication overhead than on business logic. Merge related CRUD operations into larger services.

A well-sized service: manages user notifications. It owns the `notifications`, `notification_templates`, `notification_preferences`, and `notification_delivery_logs` tables (4 tables). It offers 3 API endpoints: `GET /notifications`, `POST /notifications/send`, `PUT /notifications/preferences`. It consumes 3 event types: `UserSignedUp`, `CommentReplied`, `VideoProcessed`. A team of 4 engineers owns it.

### Communication Protocols

| Protocol | Latency | Throughput | Payload Size | Tooling | When to Use |
|----------|---------|------------|--------------|---------|-------------|
| HTTP/JSON (REST) | 1-5ms per call | 500-2K RPS per instance | 1-100 KB | Excellent | External APIs, simple CRUD |
| gRPC (protobuf) | 0.5-2ms per call | 5K-20K RPS per instance | 0.1-10 KB | Good | Internal service-to-service, streaming |
| HTTP/JSON (REST) | 1-5ms per call | 500-2K RPS per instance | 1-100 KB | Excellent | External APIs, simple CRUD |

Choose gRPC for internal traffic where latency matters. Choose HTTP/JSON for external APIs where debuggability matters. Choose message queues for event-driven workflows where immediate response is not required.
