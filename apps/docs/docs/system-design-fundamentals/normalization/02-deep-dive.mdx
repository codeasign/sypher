The key insight: a normalized schema increases the number of tables in a query, which expands the search space for the optimizer. A denormalized query on one table has one access path. A normalized query joining 5 tables has 5 factorial (120) possible join orders. The optimizer must evaluate these efficiently.

### Catalog and Statistics

The optimizer relies on the system catalog, which stores metadata about every table, column, index, and constraint in the normalized schema.

```sql
-- PostgreSQL system catalog queries for normalized schema analysis
SELECT
    relname AS table_name,
    reltuples AS row_count,
    relpages AS disk_pages
FROM pg_class
WHERE relname IN ('customers', 'orders', 'order_items', 'products');

-- Column statistics for join columns
SELECT
    attname AS column_name,
    n_distinct AS estimated_distinct_values,
    most_common_vals,
    most_common_freqs
FROM pg_stats
WHERE tablename = 'orders';
```

The optimizer uses these statistics to estimate the selectivity of each join condition. If `customers` has 100,000 rows and `orders` has 500,000 rows, and the join is on `customer_id`, the optimizer estimates that each customer has approximately 5 orders on average.

## Data Structures and Algorithms

Normalization theory is built on a foundation of discrete mathematics. Understanding these algorithms helps you design schemas and diagnose normalization problems.

### Functional Dependency Analysis

A functional dependency X → Y means that if two rows have the same value for X, they must have the same value for Y. The database enforces this through primary key and unique constraints.

The closure of a set of attributes X under a set of functional dependencies F is the set of all attributes that are functionally determined by X. Written as X+.

**Algorithm: Attribute Closure**

```
Input:  Set of attributes X, set of functional dependencies F
Output: X+ (closure of X under F)

1. closure = X
2. while closure changes:
3.   for each dependency A → B in F:
4.     if A ⊆ closure:
5.       closure = closure ∪ B
6. return closure
```

Example: Given `F = {A → B, B → C, C → D}`, compute `A+`:

- Start: closure = `{A}`
- A → B: A ⊆ `{A}`, so closure = `{A, B}`
- B → C: B ⊆ `{A, B}`, so closure = `{A, B, C}`
- C → D: C ⊆ `{A, B, C}`, so closure = `{A, B, C, D}`
- No more changes. `A+` = `{A, B, C, D}`

This means A determines all four attributes. A is a candidate key.

### Armstrong's Axioms

Three inference rules for deriving all functional dependencies from a given set:

**Reflexivity**: If Y ⊆ X, then X → Y. (Trivial: a set of attributes determines its subsets.)

**Augmentation**: If X → Y, then XZ → YZ for any Z. (Adding the same attributes to both sides preserves the dependency.)

**Transitivity**: If X → Y and Y → Z, then X → Z. (Dependencies chain.)

These three rules are sound (they only produce valid dependencies) and complete (they can derive every valid dependency). Three additional rules follow from them:

**Union**: If X → Y and X → Z, then X → YZ.

**Decomposition**: If X → YZ, then X → Y and X → Z.

**Pseudo-transitivity**: If X → Y and WY → Z, then WX → Z.

### Decomposition Algorithms

When a schema violates a normal form, you decompose it into smaller tables. Two properties are critical:

**Lossless Join**: The decomposition must allow you to reconstruct the original table by joining the decomposed tables without losing or adding rows. If you decompose R into R1 and R2, the join of R1 and R2 must exactly equal R.

**Dependency Preservation**: The decomposition must not make it impossible to enforce the original functional dependencies. Every dependency in F should be enforceable on the individual decomposed tables.

**Algorithm: 3NF Decomposition (Synthesis)**

```
Input:  Relation R, set of functional dependencies F
Output: Decomposition of R into 3NF

1. Compute a minimal cover G of F
2. For each left-hand side X in G:
3.   Create a relation with attributes {X ∪ all Y where X → Y in G}
4.   Choose a key for this relation from X
5. If none of the created relations contains a candidate key of R:
6.   Add a relation with the candidate key attributes
7. Remove any relation that is a subset of another
```

**Algorithm: BCNF Decomposition**

```
Input:  Relation R, set of functional dependencies F
Output: Decomposition of R into BCNF

1. result = {R}
2. while there is a relation Ri in result that is not in BCNF:
3.   Find a FD X → Y in Ri that violates BCNF
4.   Decompose Ri into Ri1 = X ∪ Y and Ri2 = Ri - Y
5.   Replace Ri with Ri1 and Ri2 in result
6. return result
```

BCNF decomposition always produces a lossless join but may not preserve all functional dependencies. The 3NF synthesis algorithm preserves all dependencies but may produce a schema that is not in BCNF.

### Lossless Join Decomposition: The Chase Algorithm

When you decompose a relation, the chase algorithm tests whether the decomposition is lossless. Given a relation R with functional dependencies F, decomposed into R1 and R2:

1. Create a table with one row per Ri and one column per attribute of R.
2. Fill each cell with the column name if that attribute is in Ri, otherwise fill with a distinct subscripted symbol.
3. Apply the functional dependencies in F. If two rows have equal values on the left-hand side of a dependency, make their right-hand side values equal.
4. If any row becomes all unsubscripted symbols (the original attribute names), the join is lossless.

A simpler practical test: A decomposition of R into R1 and R2 is lossless if and only if the common attributes of R1 and R2 contain a candidate key of either R1 or R2. In other words, if `R1 ∩ R2 → R1` or `R1 ∩ R2 → R2`, the decomposition is lossless.

### Dependency Preservation: Why It Matters

Consider `R(A, B, C)` with `F = {A → B, B → C}`. Decompose into `R1(A, B)` and `R2(B, C)`.

- R1 preserves A → B.
- R2 preserves B → C.
- Dependency A → C is implied by transitivity but does not need to be enforced separately because A → B is enforced on R1 and B → C is enforced on R2. Any update that respects these two dependencies will not produce an A → C violation.

Now consider `R(A, B, C, D)` with `F = {A → B, B → C, C → D}`. Decompose into `R1(A, B)` and `R2(C, D)`.

- The dependencies A → B and C → D are preserved.
- The dependency B → C is lost. Neither R1 nor R2 has both B and C.
- The database cannot enforce B → C automatically. The application must enforce it, risking inconsistency.

This is the cost of a non-dependency-preserving decomposition. You trade database-enforced consistency for a more thorough elimination of redundancy.

### Minimal Cover

A minimal cover of a set of functional dependencies F is the smallest set of dependencies that is equivalent to F. Three properties:

1. Every dependency has a single attribute on the right side.
2. No attribute on the left side is extraneous (can be removed without changing the closure).
3. No dependency is redundant (can be removed without changing the closure).

Computing a minimal cover is the first step in the 3NF synthesis algorithm and is essential for understanding the true structure of your data.

## The Query Lifecycle

A SELECT with JOINs on a normalized schema goes through 12 distinct stages. Each stage transforms the query representation.

### Stage 1: Connection and Authentication

The client opens a connection to the database. The database authenticates the client, checks permissions, and assigns a backend process or thread. Connection pooling reuses existing backends to avoid the overhead of forking a new process for each query.

### Stage 2: Query Text Reception

The database receives the SQL text as a string. For prepared statements, the text was already received during the PREPARE phase; the client sends only the parameter values.

```sql
-- Prepared statement for a normalized query
PREPARE get_customer_orders (INT) AS
SELECT c.name, o.order_date, p.name AS product, oi.quantity, oi.unit_price
FROM customers c
JOIN orders o ON c.id = o.customer_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE c.id = $1;
```

### Stage 3: Lexical Analysis

The parser splits the SQL text into tokens: keywords (SELECT, FROM, JOIN, WHERE), identifiers (customers, orders), operators (=, >), literals (date strings, numbers), and punctuation (commas, parentheses).

### Stage 4: Syntax Parsing

The parser builds an Abstract Syntax Tree (AST) from the token stream. The AST represents the query structure as a tree of nodes: SelectStmt, JoinExpr, RangeVar, ColumnRef, A_Const.

### Stage 5: Semantic Analysis and Rewriting

The rewriter resolves names against the catalog. It replaces view references with the view's underlying query (view expansion). Common Table Expressions (CTEs) are inlined or materialized. Column references are resolved to specific table columns.

### Stage 6: Query Graph Construction

The planner builds a query graph showing all tables, join conditions, filter predicates, and projection columns. For a 4-table join, this graph has 4 relation nodes and 3 join condition edges.

### Stage 7: Join Order Enumeration

The planner enumerates join orders. For N tables, there are (2N-2)! / (N-1)! possible join trees. For 4 tables: 120 possibilities. For 6 tables: 30,240 possibilities. The planner uses dynamic programming with the System R algorithm to prune.

```
Join order candidates for 4 tables:
((customers ⋈ orders) ⋈ order_items) ⋈ products   cost: 12,450
((orders ⋈ order_items) ⋈ customers) ⋈ products   cost: 8,230
((orders ⋈ products) ⋈ customers) ⋈ order_items    cost: 45,600
(orders ⋈ (order_items ⋈ products)) ⋈ customers    cost: 3,200  ← best
```

### Stage 8: Access Path Selection

For each table, the optimizer chooses how to access the data:

- Sequential scan: read the entire table (fast for small tables, slow for large ones)
- Index scan: use an index to find matching rows (fast for selective queries)
- Bitmap scan: combine multiple indexes with bitwise operations

For a normalized schema, the optimizer typically uses index scans on foreign key columns. A query filtering by `customer_id = 42` on the `orders` table benefits from an index on `orders.customer_id`.

### Stage 9: Join Method Selection

The optimizer chooses a join algorithm for each pair of relations:

**Nested Loop Join**: For each row in the outer table, scan the inner table for matches. Time complexity: O(N * M). Best when one table is small and the other has an index on the join column. Used for fewer than 10,000 rows in the outer table.

**Hash Join**: Build a hash table on the smaller table, then probe it with each row from the larger table. Time complexity: O(N + M). Best for equi-joins on medium-sized tables. The most common join method for normalized schemas with 10,000 to 1 million rows.

**Merge Join**: Sort both tables on the join column, then merge them in a single pass. Time complexity: O(N log N + M log M). Best for large tables that are already sorted or when the result must be sorted.

### Stage 10: Physical Plan Generation

The optimizer produces a physical plan tree with concrete algorithms at each node. Each node is an iterator that produces tuples for its parent.

```
Physical Plan:
┌──────────────────────────────────────────────────┐
│  Hash Join  (cost=3,200)                         │
│  Join: oi.order_id = o.id                       │
│  ├── Hash Join  (cost=1,800)                     │
│  │   Join: oi.product_id = p.id                 │
│  │   ├── Seq Scan on order_items oi  (cost=800)  │
│  │   └── Hash                                    │
│  │       └── Seq Scan on products p  (cost=400)  │
│  └── Hash                                        │
│      └── Index Scan on orders o  (cost=600)      │
│          Index: o.customer_id = 42               │
└──────────────────────────────────────────────────┘
```

### Stage 11: Execution

The executor runs the physical plan using the volcano model: each operator pulls tuples from its children, processes them, and emits result tuples to its parent. The hash join operator:

1. Reads all rows from the inner child (products) and builds a hash table keyed by `product_id`.
2. Reads rows from the outer child (order_items), hashes each `product_id`, and probes the hash table.
3. For each match, emits a combined tuple to the parent hash join.
4. The parent hash join probes against the orders hash table.
5. The top-level projection removes unneeded columns.

### Stage 12: Result Transmission

The executor sends result tuples to the client. For a query returning 1,000 rows, the database may send them in batches of 100 to balance network overhead and latency. The client receives the result set and renders it.

### Detailed Join Example: Three-Table Normalized Query

Consider this query on a normalized e-commerce schema:

```sql
SELECT c.name, o.order_date, p.name AS product, oi.quantity
FROM customers c
JOIN orders o ON c.id = o.customer_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE c.email = 'alice@example.com'
  AND o.status = 'delivered';
```

The optimizer processes this as follows:

1. **Filter pushdown**: The WHERE clause filters `customers.email = 'alice@example.com'` before any join. This reduces the customers table to 1 row.

2. **Join order selection**: The optimizer starts with the smallest result set (1 customer row) and joins outward. First join: customers with orders. With an index on `orders.customer_id`, this is an index nested loop join: 1 lookup into the index, 20 matching orders.

3. **Join with order_items**: The 20 orders join to order_items via index on `order_items.order_id`. Each order has approximately 3 items, so 60 rows. Filter by `o.status = 'delivered'` reduces to approximately 50 rows.

4. **Join with products**: The 50 order items join to products via primary key index on `products.id`. This is a simple index lookup: 50 lookups, each under 0.5 ms. Total join time: approximately 5 ms.

Without indexes, each step scans the full table: 100K customers, then 500K orders, then 1M order_items, then 10K products. Total scan time: seconds to minutes.

## Consistency and State

Normalized schemas introduce specific consistency challenges during concurrent writes.

### Referential Integrity Enforcement

When a transaction inserts a row into `order_items` with a `product_id`, the database must verify that a matching row exists in `products`. This check requires a lookup on the `products` table. If another transaction concurrently deletes that product, the database uses row-level locks or predicate locking to prevent violations.

PostgreSQL uses a snapshot-based approach (MVCC): the INSERT in `order_items` sees the snapshot of `products` as of the transaction's start time. If a concurrent DELETE committed after the snapshot, the INSERT succeeds but the foreign key points to a deleted row. The database prevents this by taking a `FOR KEY SHARE` lock on the referenced row.

```sql
-- What PostgreSQL does internally for FK enforcement
-- When inserting into order_items with product_id = 100:
SELECT 1 FROM products WHERE id = 100 FOR KEY SHARE;
-- This lock prevents concurrent deletes of product 100
-- until the inserting transaction commits
```

### Cascade Operations

Cascading referential actions propagate changes across normalized tables:

```sql
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);

CREATE TABLE order_items (
    id INT PRIMARY KEY,
    order_id INT REFERENCES orders(id)
        ON DELETE CASCADE
);
```

Deleting a customer with 500 orders triggers:
1. Delete from `order_items` for all 500 orders (potentially thousands of rows).
2. Delete from `orders` for the 500 orders.
3. Delete from `customers` for the one customer.

This cascade can take seconds or minutes depending on the volume. During this time, locks are held on all affected rows, blocking concurrent access. A cascade storm occurs when a single delete cascades to millions of rows, locking a significant portion of the database.

### Constraint Checking Timing

Databases support two modes of constraint checking:

**Immediate constraints** (default): Checked after each statement. If a row inserted into `order_items` references a non-existent `product_id`, the database rejects the statement immediately.

**Deferred constraints**: Checked at transaction commit time. Useful for inserting into multiple related tables where the order of insertion matters. For example, inserting a product and its items in any order, as long as all references are valid by commit time.

```sql
-- Deferred constraint example
BEGIN;
SET CONSTRAINTS fk_order_items_product DEFERRED;
INSERT INTO order_items (id, order_id, product_id, qty)
VALUES (1, 100, 999);  -- product 999 does not exist yet
INSERT INTO products (id, name, price)
VALUES (999, 'New Product', 29.99);
COMMIT;  -- All constraints checked here -- succeeds
```

### Concurrency Control Effects

Normalized schemas spread related data across multiple tables. A single business operation — placing an order — requires inserting into `orders`, `order_items`, and potentially updating `inventory` and `customer_balance`. This multi-table transaction increases the chance of deadlocks.

Deadlock example: Transaction A inserts into `orders` then `order_items`. Transaction B inserts into `order_items` then `orders`. If they acquire locks in different orders, they deadlock. The database detects the cycle and kills one transaction, which must retry.

## Performance Characteristics

Normalization has measurable performance implications. The trade-off is predictable: write operations become faster and more consistent, while read operations become slower due to joins.

### Join Overhead Comparison

| Join Method | Time Complexity | Memory | Best For | Worst For |
|---|---|---|---|---|
| Nested Loop | O(N * M) | O(1) | Small outer table, indexed inner | Large unindexed tables |
| Hash Join | O(N + M) | O(min(N, M)) | Medium equi-joins | Non-equi joins |
| Merge Join | O(N log N + M log M) | O(N + M) | Large sorted tables | Unsorted data |

### Latency Benchmarks

Measured on a normalized e-commerce schema with 100K customers, 500K orders, 1M order items, and 10K products:

| Query Pattern | Denormalized | Normalized (No Index) | Normalized (With Index) |
|---|---|---|---|
| Single order by ID | 1.2 ms | 2.8 ms | 0.4 ms |
| Customer's last 10 orders | 3.1 ms | 15.2 ms | 1.8 ms |
| Monthly revenue report | 45 ms | 2,300 ms | 180 ms |
| Top 10 products by revenue | 180 ms | 4,500 ms | 340 ms |
| Customer with full order history | 2.5 ms | 28 ms | 4.2 ms |

The p50 latency for a 3-table join with proper indexes stays under 5 ms. The p99 latency for a 5-table join without indexes exceeds 500 ms. Indexing foreign key columns is the single highest-impact optimization for normalized schemas.

### Memory vs CPU Trade-off

Hash joins consume memory proportional to the smaller table. A hash join between `orders` (500K rows) and `order_items` (1M rows) builds a hash table on `orders` (approximately 100 MB). If the work_mem setting is under 100 MB, PostgreSQL spills to disk, causing a 10x to 100x slowdown.

```
work_mem = 64 MB  →  hash spills to disk  →  2,300 ms
work_mem = 128 MB →  hash fits in memory  →    180 ms
work_mem = 512 MB →  hash fits comfortably →    175 ms
```

The optimizer chooses the join method based on cost estimates derived from table statistics. You can influence these estimates by updating statistics and adjusting cost parameters.

## Configuration and Tuning

### Normal Form Level Selection

Most production systems target 3NF. The choice depends on workload:

| Normal Form | Write Performance | Read Performance | Consistency | Use Case |
|---|---|---|---|---|
| 1NF only | Fastest | Fastest | Low | Reporting, analytics |
| 2NF | Fast | Moderate | Medium | Simple operational systems |
| 3NF | Moderate | Slower | High | OLTP, financial systems |
| BCNF | Slower | Slowest | Highest | Academic, highly structured data |
| Denormalized | Moderate | Fastest | Low | Read-heavy, caching layers |

### Indexing Strategies for FK Columns

Every foreign key column should have an index. Without an index, the database must scan the entire referencing table for each foreign key check.

```sql
-- Always index foreign key columns
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
CREATE INDEX idx_order_items_order_id ON order_items(order_id);
CREATE INDEX idx_order_items_product_id ON order_items(product_id);

-- Composite indexes for common query patterns
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date DESC);

-- Covering indexes for index-only scans
CREATE INDEX idx_order_items_order_product ON order_items(order_id, product_id)
    INCLUDE (quantity, unit_price);
```

### Denormalization Triggers

Monitor these metrics as signals that denormalization may be necessary:

- Average query response time exceeds 100 ms for frontend queries.
- Join overhead accounts for more than 40 percent of query execution time.
- Read-to-write ratio exceeds 50:1 on a specific query path.
- Materialized views become the primary access pattern for a hot path.

### Database Optimizer Hints

When the optimizer chooses a suboptimal plan, you can guide it:

```sql
-- PostgreSQL: force hash join for a specific query
SET enable_nestloop = off;
SET enable_mergejoin = off;
SELECT c.name, o.total
FROM customers c
JOIN orders o ON c.id = o.customer_id
WHERE c.id = 42;
SET enable_nestloop = on;
SET enable_mergejoin = on;

-- Oracle: optimizer hints
SELECT /*+ LEADING(c o oi) USE_HASH(o) INDEX(o idx_orders_customer_id) */
    c.name, o.order_date, oi.quantity
FROM customers c, orders o, order_items oi
WHERE c.id = o.customer_id
  AND o.id = oi.order_id
  AND c.id = 42;
```

### Workload-Specific Tuning

**OLTP workloads** (many small, fast transactions): Use 3NF. Keep join depth to 3 tables or fewer. Index every FK. Use connection pooling. Set `work_mem` to 4-16 MB per connection.

**OLAP workloads** (large scans, aggregations): Use star schemas (a denormalized fact table with normalized dimension tables). Build materialized views for common aggregations. Use columnar storage.

**Mixed workloads**: Use normalized OLTP tables for writes. Use read replicas with materialized views for reads. The materialized views pull data from the normalized tables into denormalized reporting structures.

## Advanced: Dependency Analysis in Practice

Real-world schemas are more complex than textbook examples. Here is how to analyze a real schema for normalization issues.

### Finding Hidden Dependencies

```sql
-- Detect duplicate customer data across orders
SELECT customer_email, COUNT(DISTINCT customer_name) AS name_count,
       COUNT(DISTINCT customer_address) AS address_count,
       COUNT(*) AS row_count
FROM orders
GROUP BY customer_email
HAVING COUNT(DISTINCT customer_name) > 1
    OR COUNT(DISTINCT customer_address) > 1;
```

This query reveals transitive dependencies in a denormalized schema. If the same email has different names or addresses, you have inconsistency caused by lack of normalization.

### Identifying Candidate Keys

```sql
-- Find columns that could be candidate keys
SELECT column_name, COUNT(DISTINCT column_value) AS distinct_values,
       COUNT(*) AS total_rows
FROM (
    SELECT 'email' AS column_name, customer_email AS column_value
    FROM customers
    UNION ALL
    SELECT 'phone', customer_phone FROM customers
) sub
GROUP BY column_name;
```

If `distinct_values` equals `total_rows`, the column is a candidate key.

### Dependency Preservation Check

After decomposition, verify that all original functional dependencies are preserved:

```sql
-- Check that customer_id → customer_name is preserved
-- in the customers table
SELECT customer_id, COUNT(DISTINCT customer_name)
FROM customers
GROUP BY customer_id
HAVING COUNT(DISTINCT customer_name) > 1;
-- If this returns rows, the dependency is violated
```

## Case Study: Normalizing a Real Schema Under Load

A production team at a mid-size e-commerce company analyzed their order processing schema. The schema had been denormalized for years and was causing weekly data inconsistencies.

**The problem table**:

```sql
CREATE TABLE order_records (
    order_id INT,
    customer_id INT,
    customer_name VARCHAR(200),
    customer_email VARCHAR(255),
    customer_phone VARCHAR(20),
    customer_address TEXT,
    product_id INT,
    product_name VARCHAR(200),
    product_category VARCHAR(100),
    product_price DECIMAL(10,2),
    quantity INT,
    line_total DECIMAL(12,2),
    order_total DECIMAL(12,2),
    order_status VARCHAR(20),
    order_date TIMESTAMP,
    shipped_date TIMESTAMP,
    carrier VARCHAR(50),
    tracking_number VARCHAR(100)
);
```

**The normalization process**:

1. Identify candidate key: `(order_id, product_id)` — each product on an order is one row.

2. 1NF check: Already in 1NF — atomic columns, no repeating groups.

3. 2NF analysis: `product_name`, `product_category`, `product_price` depend only on `product_id` (partial dependency). `customer_name`, `customer_email`, `customer_phone`, `customer_address` depend only on `customer_id` (partial dependency). `order_total`, `order_status`, `order_date`, `shipped_date`, `carrier`, `tracking_number` depend only on `order_id` (partial dependency).

4. 2NF decomposition: Split into `customers`, `orders`, `products`, `order_items`. This eliminates all partial dependencies.

5. 3NF analysis: In the `customers` table, `customer_email` could determine `customer_name` and `customer_phone` — but `customer_id` is the primary key, and no non-key column determines another non-key column. No transitive dependencies found.

6. Result: A 4-table 3NF schema. The team ran a dual-write migration over 4 weeks, writing to both old and new schemas. After verification, they dropped the old denormalized columns.

**Measured outcomes**:
- Insert throughput increased by 15 percent (fewer columns to write per statement).
- Update consistency: zero anomalies in the first 30 days after migration.
- Read latency on the order history page: increased from 3 ms to 12 ms due to joins.
- The team added a materialized view for the order history page, bringing read latency back to 4 ms.
