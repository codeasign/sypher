## Tier 1 — Foundation

These challenges test your ability to recognize normal forms, identify anomalies, and normalize a given schema.

### Challenge 1: Identify the Normal Form

Examine the following table and determine the highest normal form it satisfies. Explain the violations.

```sql
CREATE TABLE enrollment (
    student_id INT,
    student_name VARCHAR(100),
    course_id INT,
    course_name VARCHAR(200),
    instructor_name VARCHAR(100),
    instructor_office VARCHAR(50),
    semester VARCHAR(20),
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id, semester)
);
```

**Questions**:
1. What is the highest normal form this table satisfies?
2. Identify all partial dependencies.
3. Identify all transitive dependencies.
4. Describe the insert anomaly: what fact cannot be stored without having another fact?
5. Describe the update anomaly that occurs if an instructor changes offices.

**Expected solution approach**: The composite key is `(student_id, course_id, semester)`. Columns like `student_name` depend only on `student_id` (partial dependency). `course_name` depends only on `course_id` (partial dependency). `instructor_office` depends on `instructor_name` (transitive dependency through `course_id → instructor_name → instructor_office`). The table is in 1NF. Decompose into `students`, `courses`, `instructors`, `enrollments`, and `semesters` tables.

### Challenge 2: Normalize a Denormalized Schema

The following table tracks employee project assignments. Normalize it to 3NF.

```sql
CREATE TABLE project_assignments (
    project_id INT,
    project_name VARCHAR(200),
    project_manager VARCHAR(100),
    employee_id INT,
    employee_name VARCHAR(100),
    employee_department VARCHAR(100),
    department_head VARCHAR(100),
    role VARCHAR(50),
    hours_allocated INT,
    start_date DATE,
    end_date DATE,
    PRIMARY KEY (project_id, employee_id)
);
```

**Questions**:
1. Identify all functional dependencies.
2. Decompose the table into 3NF. Write the CREATE TABLE statements.
3. Show that the decomposition is lossless by writing the join query that would reconstruct the original data.
4. Does the decomposition preserve all functional dependencies? Identify any that are lost.

**Expected solution**: Decompose into `projects(project_id, project_name, project_manager)`, `employees(employee_id, employee_name, employee_department)`, `departments(department_name, department_head)`, and `assignments(project_id, employee_id, role, hours_allocated, start_date, end_date)`. The dependency `employee_department → department_head` is preserved in the `departments` table. The join query: `SELECT * FROM projects JOIN assignments ON ... JOIN employees ON ... JOIN departments ON ...`.

### Challenge 3: Explain the Anomalies

A university uses a single table to store all student activity:

```sql
CREATE TABLE student_activity (
    student_id INT,
    student_email VARCHAR(100),
    club_id INT,
    club_name VARCHAR(100),
    advisor_name VARCHAR(100),
    event_id INT,
    event_name VARCHAR(200),
    event_date DATE,
    attendance_status VARCHAR(20),
    PRIMARY KEY (student_id, club_id, event_id)
);
```

**Questions**:
1. Give a concrete example of an insert anomaly: you want to add a new club but cannot because no student has joined it yet.
2. Give a concrete example of an update anomaly: changing a club's name requires updating multiple rows.
3. Give a concrete example of a delete anomaly: deleting the last event attendance record for a club also deletes the club's name and advisor.
4. Normalize this table to 3NF. How many tables do you need?

**Expected solution**: 5 tables: `students(student_id, student_email)`, `clubs(club_id, club_name, advisor_name)`, `events(event_id, club_id, event_name, event_date)`, `attendance(student_id, event_id, attendance_status)`, and `club_membership(student_id, club_id)`.

## Tier 2 — Design

These challenges require designing a normalized schema from scratch for a given domain.

### Challenge 4: Social Media Platform

Design a normalized schema for a social media platform with the following requirements:

- Users can create posts (text, image URL, timestamp).
- Users can comment on posts (text, timestamp).
- Users can like posts and comments.
- Users can follow other users.
- Posts can have tags (hashtags).
- Posts can be shared.

**Requirements**:
1. Design the schema to 3NF. Write all CREATE TABLE statements.
2. Identify all foreign key relationships.
3. Add indexes for the most common query patterns.
4. Discuss which query would be most expensive in terms of joins and how you would optimize it.

**Expected solution approach**: 8-10 tables: `users`, `posts`, `comments`, `likes`, `follows`, `tags`, `post_tags`, `shares`. The most expensive query is fetching a user's feed: posts from followed users, with like counts, comment counts, and share counts. This joins 6-7 tables. Optimize with a materialized feed table or a cache.

### Challenge 5: E-Commerce System

Design a normalized schema for an e-commerce platform with:

- Products with categories, variants (size, color), inventory, pricing.
- Customers with addresses, payment methods.
- Orders with items, shipping, discounts.
- Reviews with ratings.

**Additional constraints**:
- A product can have multiple variants, each with its own price and inventory count.
- A customer can have multiple addresses (shipping, billing).
- An order can have multiple items, each referencing a product variant.
- Discounts can be applied at the order level or item level.

**Requirements**:
1. Design the schema to 3NF. Write all CREATE TABLE statements with proper foreign keys.
2. Design the index strategy. Which columns need indexes?
3. Write the query to fetch an order with all items, product details, and discounts. How many tables does it join?
4. Discuss the trade-off: would you add a `product_name` column to `order_items` to avoid the join with `products`? Why or why not?

**Expected solution approach**: 12-14 tables: `products`, `categories`, `product_variants`, `inventory`, `customers`, `addresses`, `payment_methods`, `orders`, `order_items`, `shipments`, `reviews`, `discounts`, `order_discounts`, `item_discounts`. The order detail query joins 6-8 tables. Adding `product_name` to `order_items` is a reasonable denormalization because the product name rarely changes, and it eliminates one join from the most critical query path.

### Challenge 6: Hospital Records System

Design a normalized schema for a hospital with:

- Patients with medical history, allergies, insurance information.
- Doctors with specialties, schedules, departments.
- Appointments with diagnosis, prescriptions, lab orders.
- Billing with procedures, insurance claims, payments.

**Critical constraints**:
- Medical records must never be deleted — only marked as inactive.
- Billing records must be immutable — once finalized, no changes.
- A patient's medical history must be fully traceable across visits.
- Doctors can be affiliated with multiple departments.

**Requirements**:
1. Design the schema to 3NF. Write all CREATE TABLE statements.
2. How do you enforce the immutability of billing records at the schema level?
3. Design the query to fetch a patient's complete medical history across all visits. How many tables does it join?
4. Discuss the normalization trade-off: the `appointments` table has a `diagnosis_code` that references a `diagnosis_codes` lookup table. Would you store the diagnosis description in the `appointments` table to avoid a join? Consider that diagnosis codes change over time.

**Expected solution approach**: 15-18 tables. Immutability: use a `billing_finalized` flag and a database trigger that prevents UPDATE on finalized rows. Medical history query joins 6-8 tables. Storing the diagnosis description denormalized is acceptable because the description is the version that was current at the time of diagnosis — it should not change even if the code description is updated later.

## Tier 3 — Expert

These challenges involve ambiguous normalization decisions with conflicting requirements.

### Challenge 7: Multi-Tenant SaaS with Conflicting Access Patterns

You are designing the database for a multi-tenant SaaS platform that provides analytics dashboards. Each tenant is a company with 10 to 10,000 employees.

**Requirements**:
- **Tenant A** (a large enterprise): 500 employees, 50 dashboard views, 200 data sources. Employees need cross-tenant reporting. They want to query their data alongside industry benchmarks from anonymized aggregate data.
- **Tenant B** (a small startup): 15 employees, 3 dashboards, 2 data sources. They want the simplest possible schema. They do not care about cross-tenant features.
- **Tenant C** (a regulated fintech): 200 employees, strict data isolation requirements. Their data must be physically separate from other tenants. Every schema change must be audited.

**Normalization challenge**:
- A fully normalized schema with shared lookup tables benefits Tenants A and C (cross-tenant queries, auditability).
- A denormalized per-tenant schema benefits Tenant B (simplicity, fast queries).
- Tenant C requires physical isolation, which conflicts with shared lookup tables.

**Questions**:
1. Design a schema strategy that satisfies all three tenants. Describe the architecture.
2. Which tables would you share across tenants? Which would you isolate per tenant?
3. How do you handle Tenant C's requirement for physical isolation while maintaining the normalized structure?
4. Design the migration path: what happens when a Tenant B company grows to 500 employees and needs cross-tenant reporting?
5. Discuss the trade-off: would you use a separate database per tenant (strong isolation, higher operational cost) or a schema-per-tenant within a shared database (weaker isolation, lower cost)?

**Expected solution approach**: Use a hybrid model. Shared lookup tables (industry benchmarks, reference data, aggregate metrics) in a common schema. Per-tenant data tables (dashboards, data sources, queries, user permissions) in tenant-specific schemas or databases. For Tenant C, a separate database instance with logical replication for aggregate data. Migration path: when a tenant outgrows the simple model, run a migration script that splits their data into the normalized shared structure. The trade-off decision depends on regulatory requirements: fintech and healthcare tend toward database-per-tenant; general SaaS can use schema-per-tenant.

### Challenge 8: Ambiguous Normalization for a Real-Time Analytics Platform

You are designing the schema for a real-time analytics platform that tracks user behavior across websites. The platform ingests 100,000 events per second and supports both real-time dashboards and historical analysis.

**Data model**:
- Events: page views, clicks, form submissions, purchases, custom events.
- Each event has a common set of fields (user_id, session_id, timestamp, page_url, user_agent, ip_address) and event-specific fields (purchase_amount, form_field_name, click_coordinates).
- Dimensions: users, sessions, pages, campaigns, referrers.
- Queries: aggregate counts by dimension over time windows (last 5 minutes, last hour, last 7 days).

**The normalization conflict**:
- A fully normalized schema (events table with dimension FKs) supports consistent dimension data. If a campaign name changes, all events reference the updated name through the FK.
- A normalized schema cannot handle 100,000 writes per second. FK checks, index maintenance, and multi-table INSERTs are too slow.
- A denormalized schema (wide events table with dimension values embedded) supports 100,000 writes per second but creates update anomalies when dimension data changes.

**Questions**:
1. Design the schema. Does it make sense to normalize or denormalize the event data?
2. How do you handle the tension between consistent dimension data and high write throughput?
3. Design a strategy for handling dimension changes: what happens when a campaign name changes? Do you update historical events?
4. How do you support real-time dashboards (5-second latency) and historical analysis (10-minute query latency) from the same data model?
5. Discuss the trade-off: would you use a different storage engine for the event data (e.g., ClickHouse, Druid, BigQuery) versus the dimension data (e.g., PostgreSQL, MySQL)?

**Expected solution approach**: The event data should be denormalized for write throughput. Ingest events as wide rows with dimension values embedded. For dimension data, maintain a separate normalized store that is the source of truth. When a dimension changes, update the normalized store and accept that historical events retain the old dimension values. This is the standard approach for analytics systems — historical facts are immutable, dimension values are snapshotted at event time. Use columnar storage (ClickHouse, BigQuery) for event data and relational (PostgreSQL) for dimension data. The real-time dashboard queries the columnar store with a time-range filter. Historical analysis uses the same store with broader time ranges. This architecture is called "lambda architecture" — the hot path serves real-time queries, the cold path serves historical queries.

### Challenge 8 Solution Sketch

Here is a detailed analysis of the normalization decisions for Challenge 8.

**Decision 1: Event data normalization**: The event data should NOT be normalized. At 100K events per second, the write path must be as simple as possible. Each event should be written as a single row with dimension values embedded (denormalized). The FK checks for a normalized event schema would add 100K additional index lookups per second, exceeding the write capacity of a standard RDBMS.

**Decision 2: Dimension store normalization**: The dimension data (campaigns, users, sessions, pages, referrers) should be fully normalized to 3NF. This data changes infrequently and consistency is important. Store it in PostgreSQL or MySQL with proper FK constraints.

**Decision 3: Handling dimension changes**: When a campaign name changes, update the dimension store. Historical events retain the old campaign name. This is called a "type 2 slowly changing dimension" — preserve history by keeping the old value in the fact table. If you need to report on the new campaign name across all history, run an ETL job that updates the event store with the new name.

**Decision 4: Query paths**: 
- Real-time dashboards: query the denormalized event store with time-range filters. Response time: under 5 seconds for the last 60 minutes of data.
- Historical analysis: query the denormalized event store with broader time ranges. Response time: 1-10 minutes depending on the time range.

**Decision 5: Storage engine choice**:
- Event data: ClickHouse, Apache Druid, or Google BigQuery. These are columnar stores optimized for time-range scans and aggregations.
- Dimension data: PostgreSQL or MySQL for consistency and transaction support.

## Bonus Challenge: The Great Debate

Debate both sides of this normalization question (practice with a peer or write both sides):

**Topic**: "A social media platform should store posts in a single denormalized table with all user data embedded."

**For**: Posts are written once and read many times. The user data associated with a post (name, profile picture) should be the version from when the post was created, not the current version. Denormalization captures the snapshot at creation time. Joining with a normalized users table would show the user's current name, not their name when they posted — which may be historically inaccurate.

**Against**: If a user changes their name, all their old posts should reflect the new name to maintain a consistent identity. Normalization ensures that a single update to the users table propagates to all posts through the join. Denormalization would require updating millions of post rows for a single name change.

**Resolution**: The answer depends on product requirements. If preserving historical context matters (a post shows the user's name at the time of posting), denormalize. If maintaining current identity matters (all posts show the user's current name), normalize. Many platforms choose normalized for user names and denormalized for content (post text, images) that is immutable by nature.

## Challenge Solution Templates

### Normalization Decomposition Template

Use this template to decompose any denormalized table into 3NF:

```
Step 1: Identify the functional dependencies
  - List every column pair that has a determination relationship
  - Use SQL to verify: GROUP BY left_side HAVING COUNT(DISTINCT right_side) > 1

Step 2: Identify the candidate keys
  - List all minimal sets of columns that uniquely identify each row
  - The primary key is one of these candidate keys

Step 3: Check 1NF - Are all columns atomic? Is there a primary key?
Step 4: Check 2NF - For composite keys only: does every non-key column depend on the full key?
  - If no: split the table. Each partial dependency gets its own table.
Step 5: Check 3NF - Does any non-key column depend on another non-key column?
  - If yes: split the table. Each transitive dependency gets its own table.
Step 6: Add foreign keys between the resulting tables.
Step 7: Add indexes on all foreign key columns.
```

## Submission Guidelines

For each challenge, produce:

1. **Your normalized schema**: CREATE TABLE statements with proper keys, constraints, and indexes.
2. **Your reasoning**: Explain each normalization decision. Which normal form did you target? Why?
3. **Your trade-off analysis**: What did you choose not to normalize? Why?
4. **Your edge cases**: How does your schema handle null values, duplicate data, concurrent writes, and data deletion?

Compare your solutions against the expected solution approaches outlined above. The best solutions balance theoretical normalization with practical engineering constraints — they are not the most normalized, but the most appropriate for the workload.
