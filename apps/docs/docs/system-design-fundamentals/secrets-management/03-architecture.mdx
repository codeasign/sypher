# Secrets Management at Scale

You have probably experienced the pain of a leaked secret. A developer commits an
API key to a public repository. A CI/CD log captures a database password. A
former employee still has access to production credentials from a shared config
file. These are not edge cases — they are the inevitable outcome of managing
secrets by hand.

At small scale, you can get away with a dotenv file. At FAANG scale, those
approaches fail catastrophically. A compromised database credential can expose
millions of user records. A leaked cloud API key can run up a bill of hundreds
of thousands of dollars in minutes.

This page builds a production-grade secrets management architecture step by
step. You will start with the naive approach, understand why it breaks, and
then construct a HashiCorp Vault deployment with active/standby clustering,
KMS auto-unseal, sidecar injection, and dynamic credentials. You will explore
integration patterns for Kubernetes, CI/CD, and databases, design multi-region
replication, and learn how to capacity-plan at scale.

## The Naive Approach: Hardcoded Secrets

The simplest possible approach is to write secrets directly into your application
code or configuration files. You have probably seen this pattern:

```python
# config.py — DO NOT DO THIS
DB_PASSWORD = "super_secret_123"
API_KEY = "sk_live_abc_def_ghi_12345"
JWT_SECRET = "my_hardcoded_hmac_secret"
STRIPE_WEBHOOK_SECRET = "whsec_1234567890abcdef"
```

This approach works for approximately 30 seconds — until the first time someone
commits these files to version control. From that point on, every developer who
clones the repository, every CI/CD pipeline that runs the tests, and every
deployment artifact that packages the code has a copy of every secret.

The problems compound rapidly:

- **No rotation.** Changing a secret means editing the file, committing the
  change, running a new build, and redeploying every service that uses it. You
  will not do this frequently, which means secrets stay valid for months or
  years.
- **No audit trail.** When a secret is leaked, you have no way to determine who
  accessed it, when, or from where. Every copy of the config file is a forensic
  black hole.
- **Shared credentials.** Every service, every environment, and every developer
  uses the same credential. A single leak compromises everything.
- **No revocation.** If you suspect a leak, your only option is to rotate the
  secret and redeploy — a process that takes hours, not minutes.
- **Compliance violations.** PCI-DSS, SOC 2, HIPAA, and GDPR all require
  encryption, access controls, and audit logging for secrets. Hardcoded secrets
  satisfy none of these requirements.

Even wrapping secrets in environment variables — a common middle ground —
does not solve the underlying problems. Environment variables are still
plaintext, still visible in process listings, still copied into every
deployment artifact, and still impossible to rotate without a restart. They
are marginally better than hardcoded strings but fundamentally inadequate for
any production system that handles sensitive data.

## Production Architecture: HashiCorp Vault

The industry standard for secrets management at scale is HashiCorp Vault. Vault
provides a unified API for storing, accessing, and rotating secrets, with
fine-grained access control, comprehensive audit logging, and pluggable secret
engines that can generate dynamic credentials on demand.

A production Vault deployment is not a single process — it is a distributed
system with multiple components that must be designed for resilience, security,
and performance.

### Cluster Architecture

A production Vault cluster uses an active/standby topology. One active node
handles all requests while standby nodes replicate the data and are ready to
take over if the active node fails. Standby nodes can serve reads directly from
the shared storage backend, improving read throughput in high-traffic
environments.

### Auto-Unseal via KMS

When a Vault node starts, it is sealed — the data is encrypted and unreadable.
In production, you never type an unseal key by hand. Auto-unseal integrates with
a cloud KMS (AWS KMS, GCP Cloud KMS, Azure Key Vault) to automate the process.
On startup, Vault contacts the KMS, decrypts the unseal key, and transitions to
an unsealed state automatically. This means:

- Nodes restart and recover without human intervention.
- The unseal key is never stored alongside the encrypted data.
- Key management is delegated to the cloud provider's HSM-backed KMS.
- Recovery in a disaster takes seconds, not hours.

### Storage Backend

Vault stores its encrypted data in a storage backend. The two primary options
for production are:

- **Raft integrated storage.** Vault manages its own Raft consensus protocol
  directly. No external dependency required. This is the simplest to operate
  and is the recommended approach for most deployments.
- **Consul.** Vault uses Consul as its storage backend. Consul provides a
  battle-tested KV store with strong consistency, health checking, and service
  discovery. This is the traditional approach and still widely used in existing
  deployments.

Both options provide strong consistency, which is essential for a secrets
management system. Eventual consistency would allow a revoked credential to
appear valid for a window of time — an unacceptable risk.

### Audit Logging

Every request to Vault is logged to an audit device. The log captures the
authenticated identity, the path and operation, the timestamp and source IP,
whether the request was allowed by policy, and a HMAC of the secret data (the
secret itself is never logged). Audit logs are forwarded to a SIEM system
(Splunk, Datadog, Elastic) for alerting and forensic analysis. In a FAANG
environment, audit logs generate terabytes per day.

### Failure Domain and Node Sizing

Each Vault node should run on a separate failure domain — a different
availability zone, rack, and network path. In cloud environments, deploy across
three AZs with a minimum of three nodes. Raft consensus requires a majority of
nodes healthy for writes: three nodes tolerates one failure, five tolerates two.

A typical Vault node handling 500 QPS with 10,000 active leases requires
approximately 4 CPU cores, 8 GB of RAM, and a 50 GB SSD. The storage footprint
is small — Vault stores encrypted data — but the write-ahead log can grow
quickly under heavy write loads.

### The Seal/Unseal Lifecycle

When a Vault node starts, it is sealed — it refuses all requests. The seal
protects encrypted data against an attacker who gains access to the storage
backend. Without the unseal key, the data is unreadable.

The unseal process: Vault contacts the KMS with its encrypted unseal key. The
KMS decrypts and returns the key. Vault uses it to decrypt the master key, then
decrypts the storage backend and begins serving requests. This takes less than
30 seconds with auto-unseal. Without it, a human operator would need to provide
multiple Shamir secret shares, which can take hours in a disaster.

## Integration Patterns

A secrets management system is only useful if it integrates with your
infrastructure. You need to connect Vault to Kubernetes, CI/CD pipelines,
databases, and application code — each requires a different auth strategy.

### Kubernetes Sidecar Injection

The most common pattern for integrating Vault with Kubernetes is the sidecar
injector. A Kubernetes mutating admission webhook intercepts pod creation
requests and automatically adds a Vault Agent sidecar container to any pod that
has the appropriate annotation.

The sidecar container authenticates with Vault using the pod's Kubernetes
service account token. Vault validates the token against the Kubernetes API
server, applies the appropriate policies based on the service account and
namespace, and returns the requested secrets. The sidecar writes the secrets to
a shared tmpfs volume that the application container can read.

The key advantage of the sidecar pattern is that your application code does not
need to know anything about Vault. It reads secrets from a file — just as it
would with a mounted secret volume — and the sidecar handles all the
authentication, authorization, and lifecycle management.

### CI/CD Pipeline Integration

Your CI/CD pipelines need access to secrets during the build and deploy process.
A unit test needs a test database password. A deployment step needs a production
API key. A container build needs a registry token. The challenge is granting
these pipelines just enough access — and only for the duration of the build.

The recommended approach is the AppRole authentication method. A CI/CD pipeline
uses a pre-assigned RoleID and a one-time-use SecretID to authenticate with
Vault. The RoleID is a static identifier that identifies the CI/CD pipeline.
The SecretID is a dynamic credential that Vault generates for each pipeline run.

The SecretID delivery is the critical part. The CI/CD platform should generate
the SecretID from Vault at pipeline start time, inject it as an environment
variable, and the pipeline step uses it to authenticate. This ensures that:

- The SecretID is valid only for the duration of the pipeline run.
- If the SecretID is leaked, it can only be used for a single authentication.
- The pipeline's access is scoped to a specific Vault policy.
- Every credential access is logged in the audit trail.

### Database Dynamic Credentials

Static database passwords are one of the most common security vulnerabilities in
production systems. They are shared across services, hard to rotate, and often
embedded in connection strings that propagate through multiple configuration
files. Vault's database secrets engine solves this by generating dynamic
credentials on demand.

When an application needs to connect to a database, it requests credentials from
Vault. Vault connects to the database using a privileged management account,
creates a new database user with a generated password, and returns the
credentials to the application. The credentials have a limited TTL — typically
one hour or less. When the TTL expires, Vault automatically revokes the
credentials and drops the database user.

This approach has profound security implications:

- There are no static database passwords to leak. Every credential is ephemeral.
- An attacker who steals a database credential can use it for at most one hour
  (or less, depending on the TTL).
- Revocation is automatic — you never need to coordinate a password rotation
  across dozens of services.
- Every credential access is logged in the Vault audit trail, giving you a
  complete record of who connected to the database and when.

### Application Code Integration

For use cases where sidecar injection is not an option, your application can
communicate with Vault directly. The pattern is straightforward: authenticate
via one of the supported auth methods (Kubernetes service account, AWS IAM, JWT,
AppRole), request secrets from the appropriate Vault path, cache them in memory,
and renew the lease before it expires.

The Vault client libraries (available for Go, Java, Python, Node.js, Ruby, and
.NET) handle most of this complexity. The key decision is the caching strategy.
A common pattern is to refresh secrets at 60% of the TTL. For a one-hour lease,
you refresh at 36 minutes. This gives a 24-minute buffer for failures and
retries. If the refresh fails, retry with exponential backoff. If all retries
fail, fail open (use the cached secret) and alert — the cached secret is still
valid until the TTL expires.

## Multi-Region Architecture

A single-region Vault deployment is a single point of failure. If the region
goes down, every service that depends on Vault — which is every service — loses
the ability to read secrets, authenticate, and generate credentials. For a
global-scale system, you need a multi-region Vault architecture.

Vault supports two types of replication:

- **Performance replication.** Secret data is replicated from the primary
  cluster to performance standby clusters in other regions. Read requests are
  served locally. Write requests are forwarded to the primary cluster. This is
  suitable for active-active configurations where most operations are reads.
- **Disaster recovery replication.** All data — including policies, auth
  methods, and configuration — is replicated asynchronously to a DR secondary
  cluster. The DR cluster does not serve requests until it is promoted to
  primary. This is suitable for disaster recovery scenarios.

### Performance Replication

You deploy a primary cluster in one region and performance standby clusters in
others. Each performance standby cluster serves reads from the local region.
Write requests are forwarded to the primary cluster, which processes the write
and replicates the data to all performance standby clusters. Writes have
cross-region latency, but reads are local.

This works well for secrets management because the workload is read-heavy. Most
applications read secrets on startup and cache them. The write path — creating
secrets, rotating them, updating policies — is relatively infrequent.

### Disaster Recovery Replication

DR replication copies all data — secrets, policies, auth methods, configuration
— to a DR secondary cluster in a different geographic region. The DR cluster
does not serve requests. It exists purely as a recovery target.

If the primary region fails, you promote the DR cluster to primary. The process
takes 5-15 minutes, depending on data size and DNS TTL. During this window, no
secrets management operations are available. This is an acceptable risk for most
systems — the alternative is permanent loss of the primary region.

### Replication Trade-offs

Performance replication reduces read latency but does not protect against a
complete failure of the primary region. DR replication protects against region
failure but adds operational complexity and requires 5-15 minutes for promotion.

Most FAANG-scale deployments use both: performance replication for low-latency
reads, DR replication for a geographically isolated backup, local KMS in each
region for independent unseal, and cross-region audit forwarding to a
centralized SIEM.

## Capacity Planning

Capacity planning for Vault at scale requires understanding four key dimensions:
request throughput (QPS), storage size, lease count, and network bandwidth. Each
dimension has different scaling characteristics and different bottlenecks.

### Request Throughput (QPS)

Vault's throughput is limited by the storage backend and the cryptographic
operations required for each request. Each read operation requires decryption of
the data. Each write operation requires encryption and replication to the storage
backend.

A single Vault node can handle approximately 500-1000 read QPS and 200-500 write
QPS, depending on the hardware and the size of the secrets. The throughput is
higher for reads because Vault caches frequently accessed secrets in memory.

To scale beyond a single node's capacity, you add more standby nodes. Standby
nodes can serve read requests from the shared storage backend, so read throughput
scales linearly with the number of nodes. Write throughput is limited by the
active node and the storage backend.

### Storage Sizing

Vault's storage footprint is small relative to the data it protects. The
encrypted data is roughly the same size as the plaintext secrets, plus metadata
for each secret (path, version, creation time, lease information).

A medium-sized deployment with 10,000 secrets, 100 policies, and 50 auth method
configurations requires approximately 1-2 GB of storage. A large deployment with
100,000 secrets and 1 million lease entries requires approximately 10-20 GB.

The storage backend's write-ahead log (WAL) can grow significantly under heavy
write loads. Plan for 2-3x the base storage size to accommodate WAL growth.

### Lease Count

Each active lease consumes memory on the Vault node. A lease is created when a
secret is read, a token is generated, or a dynamic credential is issued. The
lease count is the primary driver of memory usage.

A Vault node can handle approximately 100,000 active leases with 8 GB of RAM.
Beyond that, memory pressure forces the node to swap, which degrades performance
and increases latency. For deployments with millions of leases, you need to
distribute the load across multiple nodes or reduce the lease TTL so that leases
expire faster.

### Network Bandwidth

Each Vault request involves a TLS handshake and data transfer. For a deployment
handling 5,000 QPS with an average response size of 4 KB, the network bandwidth
required is approximately 160 Mbps. This is well within the capacity of a single
1 Gbps link, but the latency and connection overhead can become significant at
high QPS.

Connection pooling is essential for high-throughput deployments. The Vault client
libraries support connection pooling, and you should configure the pool size
based on the expected concurrency. A pool of 50-100 connections per client is
typical for high-throughput applications.

### Lease Lifecycle Management

Every secret read, token creation, and dynamic credential generation creates a
lease. If leases are not properly managed, they accumulate and consume memory
until the node runs out of RAM.

Best practices at scale:

- **Set short TTLs.** A one-hour lease is cleaned up within an hour. A 30-day
  lease stays in memory for 30 days. Set TTLs as short as your application can
  tolerate.
- **Renew proactively.** Applications should renew before expiry, not after. The
  client libraries handle this, but verify your configuration.
- **Revoke on shutdown.** Graceful shutdowns should revoke leases to free memory
  and database users immediately.
- **Monitor lease count.** Alert when counts approach node capacity. Use the
  sys/leases endpoint to identify applications that are not managing leases
  properly.

## From Architecture to Trade-Offs

You now have a complete picture of secrets management architecture at scale. A
production Vault cluster with active/standby nodes, KMS auto-unseal, sidecar
injection, dynamic credentials, and multi-region replication forms the backbone
of a secure secrets infrastructure.

But every architectural decision comes with trade-offs. The active/standby
topology provides high availability but adds operational complexity. KMS
auto-unseal enables fast recovery but introduces a cloud provider dependency.
Sidecar injection makes secret retrieval transparent but adds latency to pod
startup. Dynamic credentials provide superior security but require careful lease
management.
