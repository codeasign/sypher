### Challenge 1: Implement a Thread-Safe Queue

**Problem:** Write a thread-safe FIFO queue in Python that supports multiple producers and multiple consumers. The queue must have a maximum capacity. Producers block when the queue is full. Consumers block when the queue is empty.

**Model answer:**

```python
import threading
import collections
import time

class BoundedBlockingQueue:
    def __init__(self, max_size: int):
        self._queue = collections.deque()
        self._max_size = max_size
        self._lock = threading.Lock()
        self._not_empty = threading.Condition(self._lock)
        self._not_full = threading.Condition(self._lock)

    def enqueue(self, item):
        with self._lock:
            while len(self._queue) >= self._max_size:
                self._not_full.wait()  # block until space available
            self._queue.append(item)
            self._not_empty.notify()  # wake one waiting consumer

    def dequeue(self):
        with self._lock:
            while len(self._queue) == 0:
                self._not_empty.wait()  # block until item available
            item = self._queue.popleft()
            self._not_full.notify()  # wake one waiting producer
            return item

    def size(self):
        with self._lock:
            return len(self._queue)
```

**Key discussion points:**
- The `while` loop is essential — it guards against spurious wakeups (a thread can wake even if the condition is false).
- `notify()` wakes one thread instead of `notify_all()` — this avoids the thundering herd problem when multiple producers or consumers are waiting.
- No timeout on wait means a producer blocks indefinitely if the queue stays full. A production version would add timeout and raise `Full` or `Empty`.
- The deque provides O(1) append left and pop right.

### Challenge 2: Implement a Message with TTL

**Problem:** Extend the queue so that messages expire after a configurable TTL. Expired messages should not be returned to consumers. They should be silently dropped or moved to a dead-letter queue.

**Model answer:**

```python
import time
import collections

class TimedMessage:
    def __init__(self, payload, ttl_ms):
        self.payload = payload
        self.created_at = time.monotonic()
        self.expires_at = self.created_at + (ttl_ms / 1000.0)

    def is_expired(self):
        return time.monotonic() >= self.expires_at

class TTLQueue:
    def __init__(self):
        self._queue = collections.deque()

    def enqueue(self, payload, ttl_ms=30000):
        message = TimedMessage(payload, ttl_ms)
        self._queue.append(message)

    def dequeue(self):
        # Discard expired messages at the head
        while self._queue and self._queue[0].is_expired():
            self._queue.popleft()  # silently drop
        if self._queue:
            return self._queue.popleft().payload
        return None
```

**Key discussion points:**
- Lazy expiration (check only on dequeue) is efficient for queues with frequent consumption. For queues that are rarely consumed, a background cleanup thread is needed, or expired messages accumulate memory.
- The TTL is per-message, not per-queue. This allows different expiration times for different job types.
- A DLQ variant would move expired messages to a separate store instead of dropping them.

### Challenge 3: Simulate Consumer Lag

**Problem:** Given a queue with a known enqueue rate and dequeue rate, calculate how long it takes for the queue depth to reach a given threshold and how long the backlog takes to drain after producers stop.

**Model answer:**

```python
def simulate_queue_growth(enqueue_rate=500, dequeue_rate=300,
                          initial_depth=0, threshold=10000):
    """
    Simulate queue growth until threshold is reached.

    Returns: (time_to_threshold_seconds, depth_at_time)
    """
    net_growth_rate = enqueue_rate - dequeue_rate  # 200 msgs/sec
    if net_growth_rate <= 0:
        return float('inf'), initial_depth  # queue never grows

    time_to_threshold = (threshold - initial_depth) / net_growth_rate
    return time_to_threshold, threshold

def simulate_drain_time(queue_depth=10000, dequeue_rate=300):
    """
    How long to drain a queue of given depth at given dequeue rate.
    """
    drain_time = queue_depth / dequeue_rate  # ~33 seconds for 10K / 300
    return drain_time

# Example:
enqueue = 500  # msgs/sec
dequeue = 300  # msgs/sec
threshold = 10000

t_grow, depth = simulate_queue_growth(enqueue, dequeue, 0, threshold)
t_drain = simulate_drain_time(depth, dequeue)

print(f"Queue reaches {depth} in {t_grow:.0f} seconds ({t_grow/60:.1f} min)")
print(f"After producers stop, drain takes {t_drain:.0f} seconds ({t_drain/60:.1f} min)")
```

```text
Output:
Queue reaches 10000 in 50 seconds (0.8 min)
After producers stop, drain takes 33 seconds (0.6 min)
```

**Key discussion points:**
- The calculation assumes constant rates. In practice, the dequeue rate drops as the queue grows because consumers face memory pressure and GC overhead.
- If producers continue during drain, the drain time is infinite unless the dequeue rate exceeds the enqueue rate.
- Real monitoring uses the rate of change of queue depth over the last 5 minutes, not the absolute depth.

### Challenge 4: Design a Webhook Retry System

**Problem:** Design a webhook delivery system that accepts webhook requests from users and delivers them to a destination URL. If the destination returns an error, retry with exponential backoff (1min, 5min, 15min, 1hr, 6hr). After 5 failures, stop retrying and log the failure. Deliver at least 100,000 webhooks per hour.

**Rubric:**

| Aspect | Pass | Excellent |
|--------|------|-----------|
| Queue selection | Names a queue (SQS, Kafka) | Justifies choice with throughput numbers |
| Delayed delivery | Uses `DelaySeconds` or delayed queue | Explains how delayed queues work internally |
| Retry tracking | Tracks retry count per webhook | Uses a separate retry queue per delay level |
| Failure handling | Stops after 5 retries | Moves failed webhooks to DLQ with alert |
| Throughput | Accepts 100K/hr | Shows math: 100K / 3600 = 28/sec |

**Model answer:**

"I would use SQS for this because the throughput requirement is modest (28 messages/second) and SQS provides built-in delay queues and DLQ support.

The architecture uses two queues:

```
Webhook API ──> < Primary Queue: "webhooks" >
                    │
                    ▼
              < Delayed Retry Queue: "webhooks-retry" >
                    │
                    ▼
              [Delivery Workers]
                    │
              ┌─────┴──────┐
              │            │
            200 OK      4xx/5xx/timeout
              │            │
            Complete     Enqueue to retry queue
                          with configurable delay
```

1. The primary queue holds fresh webhooks. Workers poll from this queue and attempt delivery.
2. If delivery fails with a retryable error (5xx, timeout), the worker calculates the delay based on the retry count in the message payload and enqueues to the retry queue with `DelaySeconds`.
3. SQS supports per-message delays up to 15 minutes via `DelaySeconds`. For longer delays (1hr, 6hr), I would use a separate delayed queue with a timer that moves messages to the primary queue when their scheduled time arrives.
4. After 5 failures, the worker sends the webhook to a dead-letter queue and logs the failure details.
5. Monitoring alerts when the DLQ receives any messages, signaling an operator investigation."

```python
import boto3
import json

sqs = boto3.client('sqs')
primary_queue_url = 'https://sqs.us-east-1.amazonaws.com/123/webhooks'
retry_queue_url = 'https://sqs.us-east-1.amazonaws.com/123/webhooks-retry'

# Retry delays in seconds
RETRY_DELAYS = [60, 300, 900, 3600, 21600]  # 1min, 5min, 15min, 1hr, 6hr

def deliver_webhook(webhook):
    url = webhook['destination_url']
    payload = webhook['payload']
    retry_count = webhook.get('retry_count', 0)

    try:
        response = requests.post(url, json=payload, timeout=10)
        if response.status_code < 500:
            # Success or client error (4xx) — do not retry 4xx
            return True
    except requests.Timeout:
        pass  # will retry

    # Failed — decide next step
    if retry_count >= 5:
        # Send to DLQ
        sqs.send_message(
            QueueUrl=dlq_url,
            MessageBody=json.dumps(webhook)
        )
        return False

    # Enqueue with delay
    delay = RETRY_DELAYS[retry_count]
    webhook['retry_count'] = retry_count + 1
    sqs.send_message(
        QueueUrl=retry_queue_url,
        MessageBody=json.dumps(webhook),
        DelaySeconds=min(delay, 900)  # SQS max delay is 15 min
    )
    return False
```

| Throughput | Supports 10K/s | Shows partition math |
| Resource isolation | Shares consumers | Dedicated consumer pools per priority |
| Starvation prevention | N/A | Implements aging or promotion of low-priority jobs |

**Model answer:**

"I would use three separate queues — one per priority level — with dedicated consumer pools for critical jobs and shared capacity for normal and low.

```
┌─────────────┐
│ [Producers]  │
└──────┬──────┘
       │
       │ Tag job with priority level
       ▼
┌─────────────────────────────────────────────┐
│ Priority Router                              │
│ (classifies job, enqueues to correct queue) │
└─────┬──────────────────┬────────────────┬────┘
      │                  │                │

┌────────────┐   ┌────────────┐   ┌────────────┐

│  Queue     │   │   Queue   │   │   Queue    │
│  FIFO      │   │   FIFO    │   │   FIFO     │
│  5K/s      │   │   4K/s    │   │   1K/s     │
└─────┬──────┘   └─────┬──────┘   └─────┬──────┘
      │                │                │
      ▼                ▼                ▼
┌────────────┐   ┌────────────┐   ┌────────────┐
│[Critical   │   │[Normal     │   │[Low       │
│ Workers]   │   │ Workers]   │   │ Workers]  │
│ 50 instance │   │ 20 instance│   │ 5 instance │
│ prefetch=1  │   │ prefetch=5 │   │ prefetch=10│
│ 100ms SLA   │   │ 1min SLA  │   │ 1hr SLA   │
└────────────┘   └────────────┘   └────────────┘
```

**Critical queue:** 50 workers, prefetch=1. Each worker processes one job at a time for minimum latency. At 5,000 jobs/second, each worker handles 100 jobs/second — easily achievable.

**Normal queue:** 20 workers, prefetch=5. Batch processing improves throughput. At 10,000/5000/1000, normal has the highest volume.

**Low queue:** 5 workers, prefetch=10. Fewest resources. If low-priority jobs starve (wait over 55 minutes), promote them to the normal queue by re-enqueuing with updated priority.

The primary risk is head-of-line blocking in the critical queue. If a critical job takes 5 seconds (external API call), it delays all subsequent critical jobs. Mitigations: (1) enforce a processing timeout per job, (2) offer a rush queue for the fastest jobs, (3) use dynamic prefetch that decreases when latency rises."

### Challenge 6: Design a Log Aggregation Pipeline

**Problem:** Design a log aggregation system that ingests 500 GB of log data per day from 10,000 servers. Logs arrive as structured JSON lines. The system supports two consumer groups: real-time alerting (latency under 10 seconds) and archival storage (write to S3 every hour).

**Rubric:**

| Aspect | Pass | Excellent |
|--------|------|-----------|

| Consumer groups | Mentions two consumers | Explains how Kafka consumer groups share partitions |

      ▼                                ▼

Key design decisions:

1. **Partition count (50):** At 500 GB/day = 6 MB/s ingestion. Kafka handles 10 MB/s per partition. 50 partitions gives safety margin and supports 10x growth.

2. **Compression:** Log data compresses 3:1 with gzip and 5:1 with zstd. Enabling compression at the producer reduces network and storage by 60-80%.

3. **Consumer separation:** The alerting consumer reads from the same log but with different consumer group ID. Kafka tracks group offsets separately. The alerting consumer disables batching (deliver each log line immediately). The archival consumer batches aggressively for large S3 writes.

4. **Tiered storage:** With 30-day retention, the first 7 days stay on NVMe (hot), days 8-14 move to HDD (warm), days 15-30 move to S3 (cold). Confluent tiered storage or self-managed with Kafka's `confluent.tier` feature."

### Challenge 7: Exactly-Once Semantics in a Distributed Queue

**Discussion:** Design exactly-once delivery for a queue system distributed across 3 data centers. Producers and consumers exist in any data center. A single message must be delivered to exactly one consumer exactly once, even in the presence of network partitions, broker crashes, and consumer failures.

**Key considerations:**

```
Challenge: Exactly-once across 3 data centers

  DC-1              DC-2              DC-3
 ┌────────┐       ┌────────┐       ┌────────┐
 │Queue A │       │Queue B │       │Queue C │
 │(lead)  │       │(follower)│      │(follower)│
 └────────┘       └────────┘       └────────┘
     │                 │                 │
 ┌────────┐       ┌────────┐       ┌────────┐
 │Consumer│       │Consumer│       │Consumer│
 └────────┘       └────────┘       └────────┘
```

**Problems to solve:**

**Cross-data-center transactions:** A message sent in DC-1 must be replicated to DC-2 and DC-3 before being considered committed. Using a distributed consensus protocol (Raft, Paxos) across data centers adds 80-200ms latency per write. Alternative: use a single leader per partition and accept regional latency.

**Exactly-once at the consumer:** The broker must track delivery state even after the consumer ACKs. If the broker crashes before persisting the ACK, the message may be redelivered. Use a two-phase commit or a write-ahead log for delivery state.

**Transaction IDs:** Each message carries a unique transaction ID. The broker maintains a deduplication window (last N transaction IDs). If a producer retries with the same transaction ID, the broker returns the previous result without processing again.

```python
# Producer-side idempotent send
response = producer.send(
    topic='orders',
    value=order_payload,
    key=str(order.id),
    transaction_id=f"txn-{order.id}",
)
# If the send call fails (timeout, no response),
# the producer retries with the same transaction_id.
# The broker deduplicates.
```

**Trade-offs to discuss:**
- Exactly-once across regions adds 80-200ms to every write. Is it worth it?
- Idempotent consumers are simpler than exactly-once brokers. Can we push the complexity to the consumer?
- PayPal processes 200+ payments/second across 3 data centers using idempotent consumers with strong deduplication keys. They do not use exactly-once brokers.

### Challenge 8: Queue as a Database Backend

**Discussion:** A senior engineer proposes using a queue (Kafka) as the primary database for a new application. All data is written to Kafka. Consumers process the event stream and maintain materialized views. There is no relational database. Evaluate this proposal under three scenarios: a ride-sharing location tracking system, a banking ledger, and a collaborative document editor.

**Analysis:**

**Scenario A — Ride-sharing location tracking:**
*Verdict: Viable.*

Location updates are append-only, time-sensitive, and tolerate eventual consistency. Kafka as the source of truth works: each location update is an event, the MaterializedView is the latest position per driver. If a consumer falls behind, the location data is still accurate once it catches up (old updates are superseded by newer ones). The system can replay all events to reconstruct state after a crash.

```
Driver location updates:
  Kafka Topic: "driver-locations"
  Key: driver_id
  Value: {lat, lng, timestamp, status}

  Materialized view (Kafka Streams / ksqlDB):
  ┌──────────┬────────┬────────┬──────────┐
  │driver_id  │lat     │lng     │status    │
  ├──────────┼────────┼────────┼──────────┤
  │D-001     │37.7749 │-122.419│online    │
  │D-002     │40.7128 │-74.0060│offline   │
  └──────────┴────────┴────────┴──────────┘
```

**Scenario B — Banking ledger:**
*Verdict: Risky, not recommended for primary records.*

A banking ledger requires absolute correctness, strong consistency, and the ability to reverse transactions. Kafka's at-least-once delivery means a deposit event could be processed twice. Kafka's compaction does not guarantee immediate removal of old data. A ledger with a single incorrect entry is a regulatory problem. Use a database for the ledger and a queue for event notifications.

```
Risk: Kafka compaction + banking
  Producer sends: "deposit $100 to account A"
  Consumer processes: balance_A += 100  ✅
  Kafka compacts the topic, removing old offset data
  Consumer crashes and restores from compacted topic
  The "deposit $100" event is gone because compaction
  kept only the last event per key (e.g., "withdraw $50")
  → Account A loses $100

Solution: do NOT use compaction for append-only ledgers
But: Kafka without compaction grows unbounded
```

**Scenario C — Collaborative document editor:**
*Verdict: Viable with caveats.*

Operational transforms (OT) or conflict-free replicated data types (CRDTs) work on event streams. Kafka provides the ordered event log that OT requires. Each edit is an event. Consumers apply edits in order to reconstruct the document state. Multiple consumers maintain independent copies of the document. The caveat: document conflicts must be resolved deterministically, and the OT algorithm must handle gaps in the event stream (a consumer that lags behind may see conflicting edits without historical context).

```
Document edit as Kafka event:
  Topic: "document-updates"
  Key: document_id
  Value: {position: 5, delete: 3, insert: "hello",
          version: 42, user: "alice"}

  Consumer: applies edits in order
            maintains local document state
            kicks out rebalance if consumer crashes
```

**Key discussion points for the interviewer:**

1. Kafka as a database works well for append-only, event-sourced workloads where the latest state per key is sufficient (Kafka Streams, ksqlDB).
2. It fails for workloads that need point-in-time queries, joins across multiple entities, or strong consistency guarantees across related events.
3. The retention/compaction behavior surprises teams — old data disappears silently, and the system cannot answer queries about past state without replaying all events.
4. The operational complexity of Kafka as a database is higher than a relational database. Kafka needs ZooKeeper/KRaft, rebalancing management, partition monitoring, and consumer lag tracking.
5. A compromise pattern: use Kafka as the event store (source of truth for writes) and a database as the read-optimized materialized view. This combines the strengths of both systems.

### Expert Discussion: SLA-Driven Queue Partitioning

**Discussion:** A platform team manages a shared Kafka cluster used by 50 different product teams. Each team has different SLAs: some require sub-50ms p99 delivery, others tolerate 5-second p99 delivery. The cluster has 100 brokers and 10,000 partitions. How should the platform team allocate resources so that one team's noisy workload does not degrade another team's SLA?

**Key considerations:**

**Resource isolation strategies:**

1. **Dedicated brokers per SLA tier:** Assign 10 brokers to the low-latency tier (sub-50ms), 20 brokers to the standard tier (sub-500ms), and 70 brokers to the batch tier (5-second). Each team's topics are pinned to their tier's brokers. The low-latency brokers have lower partition density (50 partitions per broker) and avoid compaction. The batch brokers have higher density (200 partitions per broker) and use compaction.

2. **Quota enforcement:** Kafka supports per-client quotas for network throughput, request rate, and partition storage. The platform team sets quotas per team: Team A gets 100 MB/s produce throughput, Team B gets 20 MB/s. If Team A exceeds its quota, Kafka throttles their producers — other teams are unaffected.

3. **Partition rebalancing strategy:** Use a rack-aware or CPU-aware partition assignor instead of the default range assignor. The assignor distributes partitions so that each broker carries a balanced load. If a broker's CPU exceeds 80%, the partition assignor moves some of its partitions to less-loaded brokers during the next rebalance.

4. **Topic priority with consumer groups:** Low-latency teams use separate consumer groups with higher priority. The broker prioritizes fetch requests from high-priority groups. This is implemented in Kafka through request queues — the broker processes fetch requests from high-priority groups before low-priority groups.

**The trade-off:** Dedicated brokers provide the strongest isolation but waste capacity (a dedicated broker for a low-throughput team runs at 10% utilization). Quota enforcement provides weaker isolation (the broker still handles requests from all teams, just at different rates) but is more resource-efficient. The platform team should start with quotas and only move to dedicated brokers when teams repeatedly hit performance issues.

**How to present this in an interview:** Start by listing the three isolation mechanisms. Then state the trade-off: "Dedicated brokers give the strongest isolation but cost more. Quotas are more efficient but can cause throttling errors. I would start with quotas and only move to dedicated brokers after proving the quota approach is insufficient. The implementation would begin with a monitoring phase — identify which brokers are overloaded, then apply quotas to the heaviest producers, and finally reserve dedicated brokers for the most latency-sensitive teams."

### Expert Discussion: Queue-Integrated Database CDC

**Discussion:** A team wants to use Change Data Capture (CDC) from a PostgreSQL database to feed a Kafka topic. Every insert, update, and delete on the `orders` table should appear as a message in Kafka. The database handles 50,000 transactions/second. The CDC pipeline must not slow down the database. Design the CDC pipeline and discuss the failure modes.

**Key considerations:**

**Implementation strategy:** Debezium is the standard tool for PostgreSQL-to-Kafka CDC. It connects to the database's replication slot and reads the WAL. Each WAL entry is converted to a Kafka message with the before/after image of the row.

```text
PostgreSQL WAL ──> Debezium Connector ──> < Kafka Topic: "dbserver1.public.orders" >
   │                                       │
   │ WAL position: 0/4A1B3C                │ partitions: 12 (by order_id % 12)
   │ Transaction: 100 INSERTs              │ key: order_id (for ordering)
   │ Captured: INSERT, UPDATE, DELETE      │ value: JSON with before/after state
```

**Failure modes to discuss:**

**Replication slot growth:** If the Debezium connector stops consuming the WAL (crash, network partition, Kafka outage), the replication slot accumulates WAL data. PostgreSQL cannot delete WAL segments until all replication slots have consumed them. If the connector is down for hours, the WAL directory fills the disk, and PostgreSQL stops accepting writes. Defense: monitor replication slot lag and alert if the lag exceeds 1 hour. Set up a circuit breaker: if the lag exceeds 4 hours, drop the replication slot (trade-off: CDC data loss on reconnect).

**Schema changes:** An `ALTER TABLE orders ADD COLUMN discount DECIMAL` is executed. The Debezium connector picks up the new column automatically if using Avro with Schema Registry. Without Schema Registry, the connector produces messages with the new schema, and consumers that expect the old schema fail. Defense: always use Schema Registry with CDC. Apply schema changes during low-traffic periods.

**Large transactions:** A bulk update touches 500,000 rows. PostgreSQL generates 500,000 WAL entries. Debezium reads each entry and produces 500,000 Kafka messages. The topic's partition count (12) means 41,000 messages go to each partition. The consumer lag spikes to 41,000 messages per partition. Defense: Debezium can batch WAL entries into fewer Kafka messages using the `heartbeat.interval.ms` and `max.batch.size` settings. For bulk operations, consider using logical replication or a separate bulk data pipeline.

**Idempotency:** A CDC message may be delivered twice if the connector crashes after producing the Kafka message but before committing the WAL offset. The consumer sees the same row change twice. Defense: include the LSN (Log Sequence Number) in each message. The consumer tracks the last processed LSN per row ID and skips messages with a lower LSN.
