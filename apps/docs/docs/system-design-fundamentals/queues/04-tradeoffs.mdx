## The Core Tension: Reliability vs Latency

Every queue system sits on a spectrum between two poles. At one end: maximum reliability with guaranteed delivery and no data loss. At the other: minimum latency with no persistence and potential data loss. No queue system can maximize both.

### The fsync Tax

The bottleneck is physical: `fsync` forces data from the OS page cache to the storage hardware. An NVMe drive with a 4 KB write completes `fsync` in 0.5-2ms. A spinning hard disk takes 5-15ms.

```text
Without fsync (in-memory queue):
  Enqueue: 75 ns
  Throughput: 12M msgs/sec

With fsync every message (durable queue):
  Enqueue: 1.2 ms  (16,000x slower per message)
  Throughput: 8K msgs/sec

With batch fsync (1000 messages):
  Per-message cost: 1.2 ms / 1000 = 1.2 us
  Throughput: 800K msgs/sec
  But: p99 latency of first message = 1.2 ms + batch wait time
```

The decision tree looks like this:

```text
Can the system tolerate data loss on crash?
  ├── YES → In-memory queue (Redis Lists, Disruptor, ZeroMQ)
  │        Latency: < 100 us
  └── NO  → Durable queue (RabbitMQ, SQS, Kafka)
           │
           Can consumers tolerate duplicates?
           ├── YES → At-least-once, batch fsync
           │        Latency: 1-50 ms
           └── NO  → Exactly-once, acks=all
                    Latency: 10-200 ms
```

## Decision Matrix

Use this matrix to match a queue configuration to your application requirements.

| Requirement | In-Memory | Light Durable | Full Durable |
|-------------|-----------|---------------|--------------|
| Max tolerable data loss | Unlimited | 1-5 seconds | Zero |
| Max tolerable latency (p99) | 1 ms | 50 ms | 200 ms |
| Required throughput | 10M/s | 500K/s | 50K/s |
| Consumer duplicates tolerable | Yes | Yes | No |
| Ordering required | No | Per-partition | Global FIFO |
| Budget (infra cost) | Minimal | Moderate | High |
| Operational complexity | Low | Medium | High |
| Message size | Small | Medium | Large |

### When Each Tier Applies

**In-memory (Redis Lists, Disruptor, Aeron):**
- Real-time analytics where a few missed events do not matter
- In-process task scheduling between threads
- Session state that is reconstructed on restart
- Gaming leaderboards where the latest score supersedes earlier ones

**Light durable (SQS Standard, RabbitMQ lazy queues, Kafka acks=1):**
- Webhook delivery with retry on failure
- Email notification queues
- Log shipping and metrics aggregation
- Image/video processing pipelines

**Full durable (SQS FIFO, RabbitMQ quorum queues, Kafka acks=all):**
- Payment processing — a lost transaction is a financial loss
- Order fulfillment — every order must ship exactly once
- Inventory deduction — over- or under-counting costs money
- Event sourcing — event sequence must be complete and ordered

Queues are distributed systems. The CAP theorem applies. Every queue system sacrifices one of consistency, availability, or partition tolerance.

### Practical Implications of CAP Choices

**CP system (consistency + partition tolerance):** During a network partition, a quorum of the minority side is unavailable. Kafka and Pulsar stop accepting writes on the minority side. Reads may stall during leader election.

```
Network partition between Broker A and Brokers B, C:

  Partition 0 leader is on Broker A (minority side)
  Partition 0 has ISR = {A, B, C}

  During partition:
  ┌─────────────────┐     ┌─────────────────┐
  │ Broker A         │     │ Broker B, C      │
  │ Partition 0      │     │ Partition 0      │
  │ (leader)         │     │ (followers)      │
  │                  │     │                  │
  │ Cannot reach     │     │ Cannot reach     │
  │ majority of ISR  │     │ majority of ISR  │
  │                  │     │                  │
  │ Stops accepting  │     │ Elect new leader │
  │ writes           │     │ on Broker B      │
  │                  │     │ Accept writes    │
  └─────────────────┘     └─────────────────┘

  Result: 1 broker unavailable for writes until partition heals
```

**AP system (availability + partition tolerance):** During a partition, both sides accept writes. After the partition heals, the system reconciles. SQS Standard may deliver a message twice or lose a message during reconciliation.

**CA system (consistency + availability):** A single-node broker is CA — there is no partition tolerance because there is only one node. The entire queue is unavailable when the node goes down.

## Five Failure Modes

Every queue system exhibits these failure patterns in production. Understanding them prevents late-night debugging sessions.

### Failure 1: Cascade Failure from Consumer Slowdown

A consumer slows down (high GC pause, external API latency, database contention). Messages back up in the queue. As the queue grows, the broker uses more memory. When the broker exceeds the memory high-water mark, it blocks all producers. Blocked producers cause upstream services to fail.

**Prevention:**
- Set a dead-letter policy with max 3-5 delivery attempts
- Isolate high-volume, non-critical messages into a separate queue
- Monitor consumer lag and auto-scale consumers
- Set per-queue memory limits so one noisy queue does not starve others
- Implement a circuit breaker in the producer: if enqueue fails, fall back to a local file or cache

### Failure 2: Poison Message Loop

A malformed message causes every consumer that touches it to crash. The visibility timeout expires, the message returns to the queue, another consumer picks it up, and the cycle repeats.

```text
Time    Event
T=0s    Message "x=y" enters queue
T=1s    Consumer A receives it, tries to parse JSON, crashes
T=31s   Visibility timeout expires, message returns to queue
T=32s   Consumer B receives it, same crash
T=62s   Back in queue again
T=63s   Consumer C receives it, same crash
T=93s   Max delivery count (3) reached → message moves to DLQ
T=94s   Queue returns to normal processing
```

**Prevention:** Always set `max-delivery-count`. The default in most systems is infinite, which means the poison message loops forever. Set it to 3-5. Monitor the DLQ and alert when it receives messages.

### Failure 3: Slow Drain After Consumer Crash

All consumers of a queue crash simultaneously (deployment rollback, OOM killer). Messages accumulate. The queue grows to millions of messages. When consumers restart, they face a backlog larger than their processing capacity.

```text
Queue depth: 5M messages
Consumer throughput: 100 msg/s
Time to drain: 5,000,000 / 100 = 50,000 seconds = ~14 hours
During drain: new messages also arriving, extending drain time
```

**Prevention:**
- Implement priority queuing: process recent messages first, oldest messages are de-prioritized
- Use a separate "catch-up" consumer pool that scales horizontally
- Under-provision consumers and auto-scale based on queue depth
- Use TTL: if a message is older than N hours, skip it or route to a cold storage queue

### Failure 4: Ordering Collapse

A queue system promises FIFO ordering but a consumer crash causes re-delivery, breaking the guaranteed order. The re-delivered message arrives after newer messages that were already consumed.

```text
Intended order: msg_001 → msg_002 → msg_003 → msg_004

Actual delivery:
  Consumer picks msg_001, starts processing (slow)
  Consumer picks msg_002, processes quickly, ACKs
  Consumer picks msg_003, processes quickly, ACKs
  Consumer picks msg_004, processes quickly, ACKs
  Consumer crashes before ACKing msg_001

  Redelivery: msg_001 arrives after msg_004
  Consumer sees: msg_002, msg_003, msg_004, msg_001

  Ordering violated!
```

**Prevention:**
- Use a single consumer for strict ordering (limits throughput)
- Use partitioned ordering: messages with the same partition key are ordered, messages with different keys are not
- Accept that at-least-once delivery implies possible reordering on redelivery
- Include a sequence number in the message and sort after consumption (event sourcing pattern)

### Failure 5: Unbounded Queue Growth

Producers consistently outpace consumers. The queue grows without bound. The broker runs out of disk space. All brokers in the cluster fill up. The cluster becomes read-only (Kafka) or crashes (RabbitMQ).

```text
Producers: 10,000 msg/s
Consumers: 5,000 msg/s
Net growth: 5,000 msg/s

After 1 hour:  18M messages accumulated
After 1 day:   432M messages → ~860 GB at 2 KB each
After 3 days:  2.6 TB → disk full (assuming 3 TB provisioned)
```

**Prevention:**
- Set retention limits: time-based (keep 7 days) and size-based (keep 500 GB max)
- Monitor the gap between enqueue rate and dequeue rate
- Alert when queue depth exceeds a threshold (10K, 100K, 1M depending on capacity)
- Implement backpressure to producers when queue exceeds 80% of capacity

## Five Anti-Patterns

### Anti-Pattern 1: Queue as Database

Storing permanent data in a queue because it is fast. Queues are optimized for transient messages. They do not support queries, joins, or indexes. When the retention period expires, data is lost.

```
Bad:  Store user profile in queue, query it later
Good: Store user profile in database, use queue to invalidate the cache
```

### Anti-Pattern 2: One Queue for Everything

Routing all messages through a single queue with a catch-all consumer that dispatches by message type. The catch-all consumer becomes a bottleneck and a single point of failure.

```
Bad:  One queue "all-events". Everything goes there.
      Consumer reads every message and dispatches based on "type" field.
      If consumer is slow, email, search, and billing all slow down.

Good: Separate queues: "email-queue", "search-index-queue", "billing-queue".
      Each consumer team owns its queue. One slow queue does not affect others.
```

### Anti-Pattern 3: Ignoring Prefetch

Using the default prefetch count (unlimited in some systems) and wondering why consumers crash with out-of-memory errors. The broker sends all available messages to the consumer. A fast broker can overwhelm a slow consumer.

```
Bad:  No prefetch set. Broker sends 10K messages to consumer immediately.
      Consumer stores them in memory. Consumer runs out of RAM.

Good: Prefetch = 10. Consumer processes 10 at a time.
      Never overwhelmed. Memory usage steady at ~20 MB.
```

### Anti-Pattern 4: Synchronous Wait in Consumer

A consumer receives a message and then makes a synchronous HTTP call that takes 30 seconds. The consumer holds onto the message for 30 seconds. The prefetch limit means no other messages are processed. Throughput drops to 2 messages/minute.

```
Bad:  def handle(msg):
          resp = requests.get("https://slow-external-api.com/process",  # 30 sec timeout
                              data=msg.body, timeout=30)
          # Consumer is blocked for 30 seconds
          ack(msg)

Good: def handle(msg):
          # Enqueue to a retry queue for external API
          enqueue_to_retry_queue(msg)
          ack(msg)  # Return immediately

      # Separate worker for external API calls
      def retry_handler(msg):
          # Retry logic with exponential backoff
          ...
```

### Anti-Pattern 5: No Dead-Letter Queue

No DLQ configured. Poison messages loop forever. Dead consumers from message processing errors drag down the entire consumer fleet. The only recovery is manual queue purging, which loses all messages.

```
Configure a DLQ on day one. It costs nothing. It saves your weekend.
```

## When NOT to Use a Queue

Queues solve decoupling and buffering problems. They introduce operational complexity, latency, and consistency challenges. Avoid them when:

### 1. The Caller Needs the Result

If the response to a request depends on the result of the work — show the user their order confirmation, return search results, validate credit — a queue adds unnecessary indirection. The caller must either poll or receive a webhook, both of which add complexity.

```
Use synchronous RPC when:
  Client waits for the result
  Response time < 500 ms
  The operation is a query, not a command
```

### 2. Total Throughput Is Below 1,000 Ops/Sec

A single database with a connection pool can handle 1,000 writes/second. Adding a queue between the application and database introduces latency, operations overhead, and a new failure mode. If the database is the bottleneck at 1,000 ops/sec, the first step is to optimize queries and add indexes, not introduce a queue.

```
Optimize before queuing:
  1. Add missing indexes
  2. Use read replicas
  3. Connection pooling
  4. Query optimization
  5. Then add a queue if still needed
```

### 3. FIFO Ordering Is Strictly Required Across All Messages

Strict global FIFO means one consumer. One consumer limits throughput to whatever one process can handle (typically 100-10,000 messages/second depending on processing complexity). At that scale, you can also use a database table with an `INSERT ... ORDER BY id LIMIT 1` pattern.

```
Global FIFO constraint:
  ┌─────────────────────────────────┐
  │ At 100 msgs/sec: use a DB table │
  │ At 1000 msgs/sec: use Kafka     │
  │   with single partition         │
  │ Above 1000: accept partitioning │
  │   or rethink the ordering need  │
  └─────────────────────────────────┘
```

### 4. The System Has No Asynchronous Workload

If every request is answered synchronously within the same process, adding a queue introduces overhead without benefit. Queues shine when work happens in the background — after the response is sent, on a schedule, or at the consumer's pace.

```
Queues are valuable when:
  ──────────────────────────
  ✓ Email sending after signup
  ✓ Image resizing after upload
  ✓ Search indexing in background
  ✓ Notification delivery
  ✓ Batch processing (reports, exports)
  ✓ Integration with slow external APIs

Queues add cost when:
  ──────────────────────────
  ✗ Every request must complete synchronously
  ✗ All processing finishes in under 50ms
  ✗ The system has no background work
```

### 5. You Cannot Afford Any Data Loss

No queue system guarantees zero data loss in all scenarios. Even with acks=all and replication factor 3, a correlated failure (rack power loss, cluster-wide software bug) can lose messages. If data must never be lost, the producer must also write to a database in the same transaction.

```
Zero data loss requires:
  1. Producer writes to DB + queue (transactional outbox)
  2. Queue broker with acks=all
  3. Consumer processes idempotently
  4. Reconciliation job compares DB and queue results

If you skip step 1, a rare but possible failure loses data.
If you cannot accept that risk, skip the queue and use a database.
```

### How Failures Compound

A single minor failure can cascade into a system-wide outage if the queue configuration is wrong. Here is a realistic sequence from a production incident:

```text
T=0    A code change increases one consumer's processing time
       from 50ms to 500ms (accidental sleep in a library).
T=1min Consumer lag grows from 0 to 3,000 messages.
T=2min Lag grows to 12,000 messages. Alert fires but is
       classified as noise because past spikes resolved themselves.
T=3min Lag reaches 50,000. Queue depth hits 200 MB in memory.
T=4min Broker memory watermark triggered. All producers blocked.
T=4.5min API servers cannot enqueue messages. HTTP 503 returned.
T=5min  Upstream services that depend on the API also degrade.
T=6min  Site-wide incident declared. The root cause was a single
        450ms extra delay in one consumer, in one queue, which
        propagated through the cascade chain.

Resolution: The consumer was rolled back. The queue drained in 8
minutes. But the 6-minute incident affected all users because a
single queue was shared across critical and non-critical paths.
```

The cascade would not have happened if (a) critical and non-critical queues were isolated, (b) the queue had a throughput limit that dropped messages instead of blocking producers, or (c) the consumer had a processing-time timeout that killed slow handlers.

### Configuration Anti-Patterns

Beyond architectural anti-patterns, specific configuration mistakes cause common failure modes.

**Wrong visibility timeout:** Setting visibility timeout too short (5 seconds for a process that takes 30 seconds) causes every message to time out and be redelivered before processing completes. Each message is processed 2-5x more than necessary.

```text
Visibility timeout = 5 seconds
Processing time = 30 seconds

Timeline:
Consumer picks msg → 5s timeout → msg visible to others → Consumer A
  finishes but ACK is too late → msg goes to Consumer B → both succeed
  → duplicate processing → idempotent consumer required
```

**Wrong prefetch for the workload:** Setting prefetch too high (unlimited) on a consumer that calls a slow external API causes the consumer's in-memory queue to grow unbounded. Each unacknowledged message holds memory and a broker-side slot. The consumer runs out of memory at 100,000 messages.

**Batch size mismatch:** Setting producer batch size to 1 MB when messages average 100 bytes means each batch waits for 10,000 messages or the batch linger time (default 0ms — send immediately). The batch is never filled, and the batching benefit is lost. Solution: set `linger.ms` to 5-10ms to allow small messages to accumulate into a full batch.

**Infinite retries without backoff:** A consumer that retries failed messages immediately and forever creates a tight loop of failure. Each retry attempt wastes CPU, and the message never reaches the DLQ for operator inspection. Always cap retries at 3-5 with exponential backoff.

## Migration Trade-offs

Moving from one queue system to another (RabbitMQ to Kafka, SQS to Kafka, or self-hosted to managed) involves non-trivial trade-offs. These migrations happen in nearly every growing company.

### Dual-Write Pattern

The safest migration strategy is dual-write: every producer writes to both the old and new queue system simultaneously. A consumer from each system processes messages. A reconciliation job compares message counts and offsets.

```text
┌────────────┐
│ [Producer]  │
└──────┬─────┘
       │
       ├──────────────────┐
       │                  │
       ▼                  ▼
┌────────────┐   ┌────────────┐
│< Old Queue >│   │< New Queue >│
│ RabbitMQ    │   │    Kafka    │
└──────┬─────┘   └──────┬─────┘
       │                 │
       ▼                 ▼
┌────────────┐   ┌────────────┐
│[Old Workers]│   │[New Workers]│
└────────────┘   └────────────┘
```

**Pros:** Zero data loss. Rollback is instant — stop the new producer writes and the system operates as before.

**Cons:** Double the infrastructure cost during migration. Two consumer fleets to maintain. Reconciliation complexity.

### Cut-Over Strategy

Once the new queue is verified, the cut-over proceeds in phases:

1. **Phase 1 — Shadow (1 week):** Producers write to both systems. Only old consumers process. New consumers run in shadow mode (process but discard results). Verify message parity > 99.9%.
2. **Phase 2 — Canary (1 week):** Route 1% of producers to the new queue only. Monitor consumer lag, error rates, processing latency. If stable, increase to 10%, then 50%.
3. **Phase 3 — Full cut-over:** All producers write only to the new queue. Old queue reads stop. Old infrastructure decomissioned after 30 days (retention period).

### Data Consistency Verification

During migration, a reconciliation job runs every hour:

```python
def reconcile():
    # Count messages in old queue
    old_count = get_queue_depth_approximate("old-queue")
    # Count messages in new queue (Kafka lag)
    new_lag = get_consumer_group_lag("new-queue-consumers")
    # Count messages produced in last hour (from application logs)
    produced_last_hour = get_produced_count_last_hour()

    expected_remaining = max(0, old_count - produced_last_hour)
    # This is an approximation — queue depth includes unprocessed
    # messages from before the migration window

    if abs(expected_remaining - new_lag) > 100:
        alert("Queue migration parity check failed — investigate")
```

### Trade-off Decision Flowchart

When faced with a queue architecture decision, walk through this decision tree:

```text
Start: Do you need async processing?
  │
  ├── No  → Use synchronous RPC or direct database write.
  │         No queue needed.
  │
  └── Yes → Can you tolerate data loss on crash?
              │
              ├── Yes → In-memory queue (Redis, Disruptor).
              │         Latency: under 1ms. Throughput: millions/s.
              │
              └── No  → Durable queue (Kafka, SQS, RabbitMQ).
                          │
                          Can consumers tolerate duplicates?
                          │
                          ├── Yes → At-least-once delivery.
                          │         Throughput: 100K-500K/s.
                          │         Use batch fsync for performance.
                          │
                          └── No  → Exactly-once delivery.
                                    Throughput: 50K-100K/s.
                                    Use acks=all, idempotent producers.
                                    Accept 40% throughput reduction.
```

## Summary of Trade-offs

```text
Choose in-memory when:  latency under 1ms, data loss on crash acceptable
Choose light durable:   latency under 50ms, occasional duplicates ok
Choose full durable:    zero data loss, ordering, idempotent consumers

Avoid queues when:      caller needs the result, throughput under 1K/s,
                        strict global FIFO, no async work, zero data loss required
```
