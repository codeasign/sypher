## How the Topic Appears

Interviewers rarely ask "Design a queue system" directly. Instead, queues emerge as the solution to a problem the interviewer describes. Here are the ten most common phrasings.

### Phrasing 1: The Obvious Queue

"Design a system for processing background jobs. Users upload videos, and we need to transcode them. The transcode can take 2-10 minutes. Users should be able to leave the page and come back later."

The interviewer is handing you the queue. Take it. The latency requirements (minutes) and async nature (user leaves the page) make a queue the obvious choice.

### Phrasing 2: The Write Scaling Problem

"Design Twitter. Tweets are being written faster than we can persist them. How do you handle 500,000 tweets per second?"

The interviewer expects you to identify that the write path needs a queue to decouple tweet ingestion from tweet persistence. The queue absorbs the burst and feeds the database at a sustainable rate.

### Phrasing 3: The Reliability Trap

"Design a payment processing system. We cannot lose a single transaction. We need to show users real-time confirmation."

The interviewer is testing whether you blindly propose a queue without considering the synchronous confirmation requirement. A queue works for the async settlement, but the real-time confirmation needs a synchronous write path. The correct answer is a hybrid: synchronous for the confirmation, queue for the settlement processing.

### Phrasing 4: The Fan-Out

"Design a notification system. When a user posts, all their 1,000 followers should receive a notification within 30 seconds."

Fan-out is a queue-heavy problem. Pre-compute follower lists, enqueue notification jobs per follower, and distribute the work across consumer workers. The interviewer wants to hear about partition key selection and consumer group scaling.

### Phrasing 5: The Ordering Requirement

"Design a collaborative document editing system like Google Docs. Every edit must be applied in order."

Ordering is the hardest queue constraint. The interviewer expects you to discuss partition keys (document ID as the partition key ensures all edits to one document go to one partition), conflict resolution, and the trade-off between ordering and throughput.

### Phrasing 6: The Rate Limiter

"Design a system to limit API requests to 10 per second per user. How do you handle burst traffic?"

A queue behind a token bucket or sliding window rate limiter. The interviewer wants you to discuss the queue as a smoothing buffer that absorbs bursts and feeds the rate limiter at a steady rate.

### Phrasing 7: The Data Pipeline

"Design a system to process 10 billion events per day from mobile app usage. The data flows to analytics, real-time dashboards, and anomaly detection."

The interviewer expects you to propose Kafka or a similar log-based queue. Discussion points: partitioning by event type, consumer groups for each downstream system, retention policy for replayability, and exactly-once semantics for accurate analytics.

### Phrasing 8: The Dead Letter

"Messages in our queue keep getting stuck. What went wrong?"

A debugging question. The interviewer wants you to walk through poison message scenarios, visibility timeout misconfiguration, prefetch mismatch, and dead-letter queue setup. This tests operational knowledge, not just architectural.

### Phrasing 9: The Migration

"We have 5 terabytes in a RabbitMQ queue. We need to migrate to Kafka with zero data loss. Design the migration."

The interviewer wants to see your understanding of dual-write patterns, cut-over strategies, consumer lag monitoring, and verification of message parity between the old and new systems.

### Phrasing 10: The Distributed Lock

"How would you implement a distributed lock using a queue? Is it a good idea?"

The interviewer is testing whether you recognize that queues are a poor choice for distributed locking. A queue-based lock requires consumers to wait for a lease message, consume it, do work, and re-enqueue the lease. If the consumer crashes, the lease is lost until the visibility timeout expires. ZooKeeper, etcd, or Redis are better choices.

## What the Interviewer Tests

Beneath the problem statement, the interviewer evaluates you on these dimensions:

### Architectural Awareness

Can you identify where a queue fits in a system? Do you know when to use a queue vs a database vs a cache? Can you describe the components: producers, consumers, broker, DLQ, monitoring?

### Trade-Off Articulation

Do you understand the reliability vs latency spectrum? Can you explain why you chose at-least-once delivery over exactly-once? Do you know the CAP positioning of your proposed system?

### Operational Maturity

Do you think about failure modes? How do you handle poison messages? What happens when consumers crash? How do you monitor queue depth and consumer lag? Interviewers at senior levels (L5+) expect this depth.

### Scaling Intuition

Can you estimate the number of partitions needed for 100K messages/second? Do you know how replication factor affects throughput? Can you calculate the storage required for 30 days of retention?

### Implementation Feasibility

Can you sketch the producer and consumer code? Do you know the API calls — `send_message`, `receive_message`, `delete_message`, `basic_ack`? Abstract API descriptions are fine; exact syntax is not required.

## Level Expectations

Different seniority levels face different expectations around queue knowledge.

### L3 / Entry Level — "Recognize the Pattern"

**Expected:** Identify that a queue is useful for async processing. Describe basic producer-consumer flow. Draw a diagram with one queue, one producer, and one consumer.

**Model answer:**

"For a video transcoding system, I would use a queue. Users upload videos through an API. The API enqueues a transcode job and returns immediately. A worker picks up the job, runs ffmpeg, and stores the result. The user can poll for completion."

```
Upload API ──> < Queue > ──> Transcode Worker ──> Storage
```

**Interviewer looks for:** Correct identification of async use case, basic component naming, clear diagram.

### L4 / Mid Level — "Apply to Constraints"

**Expected:** Identify when a queue fits, choose between queue types, handle basic failure scenarios, estimate throughput and storage.

**Model answer:**

"For Twitter's write path, I would use a queue to decouple tweet ingestion from database writes. At 500,000 tweets/second, I need approximately 50 Kafka partitions (10K msgs/sec per partition) with replication factor 3 for durability. I would use at-least-once delivery because a duplicate tweet is acceptable. The queue allows the ingestion API to return in under 10ms while the database writes happen asynchronously at a sustainable rate."

```
Tweet API ──> < Kafka: 50 partitions, RF=3 > ──> Fan-out Service ──> Timeline DB
```

**Interviewer looks for:** Concrete numbers, trade-off awareness, partitioning understanding, failure handling.

### L5 / Senior Level — "Design the System"

**Expected:** Production-quality architecture with monitoring, deployment, and recovery. Multi-region considerations. Capacity planning with concrete numbers. Consumer group management.

**Model answer:**

"For the notification system, I would use Kafka with 100 partitions across 10 brokers. The partition key would be the user ID, so all notifications for one user go to the same partition and are delivered in order. Replication factor of 2 provides availability without writing three copies. The consumer group would have 100 members, one per partition, each maintaining a WebSocket connection to deliver notifications in real time. On consumer failure, Kafka's cooperative rebalancing reassigns the partition in under 10 seconds. I would monitor consumer lag per partition and alert if any partition's lag exceeds 1,000 messages. For exactly-once delivery to the WebSocket, I would use idempotent delivery — each notification has a unique ID, and the client deduplicates on the receiving end."

**Interviewer looks for:** Complete system thinking, operational maturity, concrete metrics, failure handling plans.

### L6 / Staff Level — "Architect the Ecosystem"

**Expected:** Multi-region deployment, cost optimization, team-level migration strategies, org impact. Conversation about trade-offs between self-hosted and managed queue services. Capacity planning for 3-5 years of growth.

**Model answer:**

"I would use a multi-region active-active Kafka deployment with a failover window of 60 seconds. Each region has its own Kafka cluster. A cross-region replication service mirrors critical topics between regions. The cost analysis shows that self-hosted Kafka on EC2 is 40% cheaper than Confluent Cloud at our current scale of 500 TB/day, but the operational burden of managing Kafka requires two dedicated SREs. I would start with Confluent Cloud and migrate to self-hosted when the monthly cost exceeds $50,000. The migration uses a dual-write pattern: every producer writes to both the old SQS queue and the new Kafka topic for 2 weeks, monitored by a reconciliation job that compares message counts every hour."

**Interviewer looks for:** Business-aware architecture, cost modeling, migration strategy, organizational impact.

### L7 / Principal — "Define the Strategy"

**Expected:** Industry-wide perspective. Evaluation of emerging queue technologies (Pulsar, Redpanda, WarpStream). Trade-off of exactly-once vs idempotent consumers across hundreds of services. Definition of the queue standards for the entire organization.

**Model answer:**

"I would standardize the organization on a single queue protocol — either Kafka-native or AMQP 1.0 — to avoid fragmentation. The standard includes a shared schema registry, common consumer group naming conventions, and mandatory DLQ configuration. Non-compliance blocks deployment. I would deprecate point-to-point SQS queues by migrating to a shared Kafka cluster with resource isolation through quota enforcement. For latency-sensitive workloads (under 5ms), I would allow exceptions for an in-memory queue (Aeron or Disruptor) but require load shedding and circuit breakers to prevent data loss."

**Interviewer looks for:** Platform thinking, standardization strategy, organizational leadership, industry knowledge.

### L8 / Distinguished — "Shape the Industry"

**Expected:** Deep understanding of queue internals — log-structured storage, consensus protocols (Raft, KRaft), tiered storage, exactly-once implementation. Ability to compare and contrast the architectural decisions of Kafka, Pulsar, and Redpanda. Vision for where queue technology is heading.

**Model answer:**

"Kafka's log-structured storage was revolutionary in 2011 because it exploited sequential disk bandwidth. By 2026, NVMe random access is 50x faster than spinning disks were in 2011, reducing the advantage of sequential-only designs. Pulsar's separation of compute and storage is better suited for cloud environments where storage is already distributed (S3, EBS). Redpanda's approach of removing the JVM and ZooKeeper dependency provides a 30% throughput improvement for the same hardware. WarpStream's object-store-native architecture eliminates the need for local disks entirely, which is the direction most organizations will move in the next 5 years as S3 throughput continues to increase."

**Interviewer looks for:** Deep internals knowledge, historical context, forward-looking vision, technical credibility.

## The FAANG Answer Framework

Use this structure for every queue-related interview question. It ensures consistent coverage of what interviewers evaluate.

### 1. Frame the Problem (1 minute)

Restate the problem and confirm assumptions. Identify whether a queue is the right tool.

```
"So you want a system where users upload videos and the
transcoding happens asynchronously. The user should be
able to leave the page and come back. A queue is the
right tool here because we need decoupling between the
upload and the processing. Let me confirm a few assumptions..."
```

### 2. Identify the Queue Type (30 seconds)

State which queue system you are using and why.

```
"I'll use Kafka because we need high throughput (50K uploads/hour)
and the ability to replay events if the transcode worker has a bug.
For durability, I'll use acks=all with replication factor 3.
For ordering, each video ID maps to one partition."
```

### 3. Draw the Architecture (2 minutes)

Draw the components and data flow. Label every arrow.

```text
Upload Service ──> < Kafka Topic: "transcode-jobs" >
    │              (100 partitions, RF=3)
    │
    │── (optional) Write job metadata to DB for polling

< DLQ > ◄── Transcode Workers (consumer group, prefetch=1)
    │        └── Poll, transcode, upload to CDN, ACK
    │
    └── Alert if DLQ > 0
```

### 4. Explain the Data Flow (2 minutes)

Walk through one complete message lifecycle.

"A producer sends a job message with video_id, format, and S3 path to Kafka. The broker appends to the partition assigned by hash(video_id). Kafka acks=all ensures all 3 replicas confirm before the producer gets confirmation. A transcode worker polls its assigned partition, processes the video with ffmpeg, uploads the result to S3, and commits the offset."

### 5. Handle Failures (2 minutes)

Cover at least three:

- **Consumer crash:** Visibility timeout / consumer group rebalance
- **Poison message:** Max delivery count -> DLQ
- **Broker crash:** Leader election among ISR
- **Producer failure:** Idempotent producer + retry with deduplication key

### 6. Discuss Trade-offs (1 minute)

Explicitly state what you are giving up.

"I chose Kafka over RabbitMQ because throughput at scale is more important than the flexible routing RabbitMQ offers. This means consumers must know which partition to read from. I lose the ability to use a topic exchange with complex routing keys. If routing needs change, I would add a lightweight relay service that reads from Kafka and publishes to RabbitMQ for specific consumers."

### 7. Quantify and Scale (30 seconds)

Give concrete numbers.

"At 50,000 uploads per day, with 100 partitions, each partition handles 500 messages/day. That is negligible. The system scales to 1 million uploads/day before I need to add partitions. Storage at 10 MB per video metadata: 50 GB/day, 1.5 TB/month, comfortably within one 4 TB NVMe drive per broker."

## Phrase Bank

Use these phrases to demonstrate queue expertise. Replace generic language with specific terminology.

### For Introducing the Queue

Instead of "We put the work somewhere else," say:

"We decouple the write path from the processing path using an asynchronous message queue."

Instead of "It stores the jobs for later," say:

"The queue acts as a buffer that absorbs production bursts and delivers messages to consumers at a sustainable rate."

### For Justifying the Choice

"The reliability requirement drives the choice. At-least-once delivery with consumer-side idempotency provides the right balance of throughput and safety."

"Exactly-once semantics are expensive. I would only use them if duplicate processing causes a correctness bug — financial transactions, inventory deductions."

"Strict FIFO ordering limits throughput to a single consumer. I would rather partition by a logical key and accept per-partition ordering in exchange for 100x the throughput."

### For Discussing Trade-offs

"The core trade-off is reliability versus latency. Every fsync call adds 0.5-2ms. Batching reduces the per-message cost but increases p99 latency because messages wait for the batch to fill."

"The CAP positioning matters here. Kafka with acks=all is a CP system — it prefers consistency over availability during a partition. SQS Standard is AP — it prefers availability and may lose messages during a partition."

### For Explaining Failure Recovery

"We configure a dead-letter queue with three delivery attempts. After three failures, the message is quarantined. An alert fires, and an operator inspects the message."

"Consumer lag is the leading indicator. If lag grows faster than consumers can drain, we auto-scale the consumer group. If it continues growing, we drop non-critical messages."

### For Summary Statements

"The queue allows the producers and consumers to fail independently. A consumer crash does not block producers. A producer burst does not overwhelm consumers."

"The key insight is that the queue is not a database. It is optimized for transient throughput, not long-term storage. Retention policies and compaction prevent unbounded growth."

## Red Flags

Avoid these statements and behaviors.

### Red Flag 1: "The Queue Never Fails"

Acting as if the queue is infinitely reliable. Every queue has failure modes. Discuss them before the interviewer prompts you.

**Instead of:** "The queue delivers every message."

**Say:** "The queue provides at-least-once delivery. Consumers must be idempotent to handle duplicate deliveries."

### Red Flag 2: No Numbers

Answering the capacity question with "it scales" without providing concrete throughput, storage, or latency numbers.

**Instead of:** "Kafka can handle it."

**Say:** "Kafka handles 500K messages/second per node. With 100K messages/second, I need 10 brokers with replication factor 2."

### Red Flag 3: Ignoring the DLQ

Designing a queue system with no mechanism for handling unprocessable messages.

**Instead of:** "Consumers process every message."

**Say:** "Consumers process normal messages. If a message fails three times, it goes to the dead-letter queue. We monitor the DLQ and receive alerts when messages land there."

### Red Flag 4: One Queue for Everything

Routing all event types through a single queue with a catch-all consumer.

**Instead of:** "All events go to one queue, and the consumer dispatches by type."

**Say:** "Critical events like payments and non-critical events like analytics go to separate queues. A payment queue slowdown does not affect analytics throughput."

### Red Flag 5: Ignoring Backpressure

Designing a producer that sends messages faster than the broker can accept, without handling the error.

**Instead of:** "The producer sends and continues."

**Say:** "The producer uses publisher confirms. If the broker returns an error (queue full, memory alarm), the producer retries with exponential backoff and falls back to a local file if retries are exhausted."

### Red Flag 6: No Consumer Lag Monitoring

Assuming consumers always keep up.

**Instead of:** "Consumers process messages as they arrive."

**Say:** "Consumer lag is the primary health metric. We alert if lag exceeds 1,000 messages per partition. If the trend shows lag growing, we auto-scale consumers."

### Red Flag 7: Choosing Exactly-Once Without Justification

Opting for exactly-once delivery because it sounds safer, without considering the throughput impact.

**Instead of:** "I will use exactly-once delivery."

**Say:** "I will use at-least-once delivery with idempotent consumers. Exactly-once reduces throughput by 40% and the correctness benefit is marginal because consumers are designed to handle duplicates."

### Red Flag 8: No Partition Count Plan

Picking a partition count without discussing future growth.

**Instead of:** "I will use 10 partitions."

**Say:** "I will start with 20 partitions based on 100K messages/second target. This provides headroom for 2x growth. When we reach 200K messages/second, I will double partitions to 40 by creating a new topic and migrating consumers."

## Phrase Bank — Extended

Beyond the core phrases, here are ready-to-use statements for specific interview scenarios.

### When Discussing Partitioning

"The partition key determines ordering and parallelism. I would choose a key with high cardinality — user ID, order ID, session ID — so messages spread evenly across partitions. If I use a key with low cardinality, like a boolean flag or US state, some partitions receive all the traffic while others sit idle."

"Rebalancing is disruptive. When a consumer joins or leaves, all consumers in the group stop processing during the rebalance. I would use cooperative rebalancing (incremental rebalance) that reassigns a few partitions at a time instead of stopping the entire group."

### When Discussing Durability

"There are two durability concerns: producer-to-broker and broker-to-consumer. For producer-to-broker, the broker must fsync the message before acknowledging. For broker-to-consumer, the consumer must ACK before the broker deletes. But there is a window: the broker deletes on ACK receipt, not on fsync of the deletion. A crash after the ACK but before the fsync of deletion causes the message to reappear on restart."

### When Discussing Monitoring

"I monitor three metrics: producer error rate, consumer lag, and DLQ depth. Producer error rate tells me if the broker is under pressure. Consumer lag tells me if consumers are keeping up. DLQ depth tells me if messages are failing silently. An alert on DLQ > 0 catches poison messages immediately."

### When Discussing Cost

"Managed queues (SQS, Confluent) cost more than self-hosted but save engineering time. The break-even point varies. At 100 million messages/month on SQS, the cost is approximately $40/month plus data transfer. Self-hosted Kafka on three c5.2xlarge instances costs approximately $600/month in EC2 alone plus operational overhead. Choose managed until the monthly bill exceeds $5,000 or you need custom configurations."

### When Discussing Cache + Queue

"A cache in front of a queue absorbs write bursts. Producers write to Redis in under 100 microseconds. A background drainer reads from Redis and writes to Kafka at a steady rate. This decouples bursty producers from the queue's sustained throughput limit. The risk: if the drainer crashes, messages accumulate in Redis. Redis persistence (AOF) covers this — messages survive the drainer restart."

### When Discussing Compression

"Log data compresses well. JSON with gzip compresses 3:1. With zstd, 5:1 is common. Enabling compression at the Kafka producer reduces network bandwidth and storage by 60-80% with minimal CPU overhead (compression is fast — under 10 microseconds per message). The consumer decompresses transparently."

## Follow-Up Questions to Ask the Interviewer

Asking good questions shows depth. Here are questions that demonstrate queue expertise.

**On requirements:**
"What is the acceptable data loss window? If a broker crashes, can we lose 1 second of messages, or must every message survive?"

"What is the p99 latency requirement end-to-end — from producer send to consumer receipt?"

"Should the system support replay? If a consumer has a bug and processes messages incorrectly, do we need to reprocess them from the beginning?"

**On scaling:**
"What is the expected growth rate for the next 3 years? If we design for 100K messages/second now, should it handle 500K in year 2?"

"Do we need global distribution — producers and consumers in multiple geographic regions?"

**On operations:**
"Who operates the queue system — a dedicated team or the application teams? This affects whether we choose a managed service or self-hosted."

"Is there an existing monitoring infrastructure we integrate with, or are we building from scratch?"

## Level Expectations — Extended Model Answers

### L4 Model Answer — Video Transcoding Queue

**Full answer:**

"I would use SQS or RabbitMQ for a video transcoding queue. The throughput requirement is modest — 1,000 uploads per hour with typical video processing times of 2-10 minutes. Each upload creates one message with the video location (S3 key) and desired output formats.

The architecture has three components:

```text
Upload API ──> < Queue "transcode-jobs" > ──> Worker Fleet
     │                                          │
     │                                          │── Fetch video from S3
     │                                          │── Run ffmpeg for each format
     │                                          │── Upload results to S3
     │                                          │── Update job status in DB
     │                                          │── ACK message
     │
     └── Store job metadata in DB (status: pending)
     └── Return job_id to client

Client polls: GET /jobs/{job_id} → status: pending | processing | done
```

I would use at-least-once delivery because a duplicate transcode wastes resources but does not cause correctness issues. Idempotency is handled by checking the job status in the database before processing: if the status is already 'done', the worker skips the transcode.

The worker pool uses auto-scaling: CloudWatch monitors queue depth. If the depth exceeds 100 messages for more than 2 minutes, it launches additional workers. Each worker handles one message at a time (prefetch=1) to avoid wasting work if a worker crashes.

For failures, I would configure a dead-letter queue with max 3 delivery attempts. If the video file is corrupted and ffmpeg fails every time, the message moves to the DLQ after 3 attempts. An operator checks the DLQ daily and contacts the uploader.

Storage is not a concern: the queue messages are small (~1 KB each), and the DLQ receives messages infrequently."

### L5 Model Answer — Real-Time Analytics Pipeline

**Full answer:**

"I would use Kafka for a real-time analytics pipeline ingesting 500K events/second from mobile SDKs. The events are page views, clicks, purchases, and custom user actions. Each event is a JSON object around 500 bytes.

Architecture:

```text
10M Mobile Devices
     │
     │── POST /events (batched: 100 events per request)
     ▼
[Load Balancer] round-robin across 50 API servers
     │
     │── HTTP/2 connection to Kafka broker
     ▼
< Kafka Topic: "raw-events" >
  150 partitions, 10 brokers, RF=3
  Retention: 7 days (hot), 30 days (S3 via tiered storage)
  Compression: zstd (5:1 compression = 100 GB/day to 20 GB/day stored)
     │
     ├──────────────────────────────────────┐
     │                                      │
     ▼                                      ▼
Consumer Group:                    Consumer Group:
"realtime-dashboard"               "data-lake"
  150 consumers (1 per partition)    10 consumers (batching)
  In-memory (5-minute windows)        Write to S3 as Parquet
  Latency: under 5 seconds            Latency: under 5 minutes
  Aggregates: page views/sec,         Partition: event_date/hour
    revenue/min, active users         Schema evolution via
                                     Glue Schema Registry
     │
     ▼
[Real-time Dashboard] (WebSocket push to product team)
```

Key design decisions:

**Partitions:** 150 partitions across 10 brokers provides 15 partitions per broker. Each partition handles approximately 3,300 events/second — well within Kafka's 10 MB/s per partition limit. If traffic doubles, I double partitions by creating a new topic and migrating consumers.

**Replication factor:** RF=3 is standard for production. With min.insync.replicas=2, the system tolerates one broker failure per partition without blocking writers. At 500K events/second, the cluster writes 1.5 TB/day (500K x 500B x RF=3 x 86,400). After compression (zstd 5:1), that is 30 GB/day on disk for 7 days = 210 GB per broker.

**Consumer groups:** The real-time dashboard consumer disables batching — each event is processed as it arrives. The data-lake consumer batches aggressively: 10,000 events or 1 minute, whichever comes first. Writing individual events to S3 as single objects is expensive in S3 PUT costs and query performance. Batching into large Parquet files optimizes for Athena/Spark reads.

**Schema management:** The raw-events topic uses a schema registry. Every SDK version registers its event schema. New fields are backward-compatible. Breaking changes require a new topic version (raw-events-v2) and a migration period.

**Operational:** I monitor consumer lag per partition. If any partition's lag exceeds 10,000 events, I investigate. Common causes: a slow consumer instance, a partition imbalance, or a GC pause. The DLQ catches events with unknown schemas or malformed payloads."

```text
│   < Kafka Cluster C >                            │
│    Aggregates critical events from all regions   │
│    Global consumers read from here               │
│    Topics: global-events (subset)                │
│    Retention: 90 days                            │
└─────────────────────────────────────────────────┘
```

Critical design decisions:

**Active-passive mirroring:** Producers in each region write to their local cluster. MirrorMaker 2 asynchronously replicates critical topics from each region to the global aggregation cluster. Non-critical topics (debug logs, low-priority analytics) are not replicated — each region's consumers read them locally.

**Partition leader distribution:** Partition leaders are distributed across brokers within a region. During a regional failure, the global cluster continues receiving events from the other regions. The failed region's consumers are redirected to the global cluster with a lag of the replication latency (5 seconds). This means 5 seconds of events from the failed region are delayed but not lost.

**Conflict resolution:** Events carry a timestamp from the producer. The global cluster uses last-writer-wins for events with the same key. This is acceptable for analytics and notifications but would not work for financial transactions.

**Cost model:** Cross-region data transfer costs $0.02/GB. At 200 TB/day, the replication cost is $4,000/day in data transfer alone. This is why only critical topics are replicated — non-critical data saves $3,200/day.

**Disaster recovery RTO/RPO:** If us-east-1 fails entirely, MirrorMaker detects the cluster is unreachable within 15 seconds. Producers in eu-west-1 continue operating. Consumers reading from the global cluster in ap-southeast-1 see events from eu-west-1 with a 5-second delay. The recovery time objective (RTO) is 15 seconds. The recovery point objective (RPO) is 5 seconds — up to 5 seconds of events from us-east-1 may be lost if the failure is catastrophic."

### Drill 2: Trade-off Rapid Fire

2. In-memory or disk-backed for a gaming leaderboard?

### Drill 5: Design a Queue from Scratch

If asked to implement a minimal queue system, sketch these components:

1. **Data structure:** Linked list or circular buffer for in-memory, append-only log for persistence.
2. **Thread safety:** Mutex for single-producer-single-consumer, read-write lock for multiple consumers.

4. **Delivery tracking:** In-flight messages map with visibility timeout thread.

### Drill 6: Partition Count Estimation

You need to estimate the partition count for a new Kafka topic. Walk through the calculation step by step.

**Given:**
- Target throughput: 250,000 messages/second
- Average message size: 1.5 KB
- Replication factor: 3
- Maximum partition throughput (Kafka): 10 MB/s
- Acceptable per-partition lag: 10,000 messages

**Calculation:**

```text
Throughput in MB/s: 250,000 * 1.5 KB = 375 MB/s
Max partition throughput: 10 MB/s
Minimum partitions: 375 / 10 = 37.5, rounded up to 40

Storage per partition per day:
  250,000 / 40 * 1.5 KB * 86,400 = 810 GB per partition per day
  with RF=3: 2.4 TB per partition per day

With 40 partitions on 10 brokers: 4 partitions per broker
Storage per broker: 4 * 2.4 TB = 9.6 TB/day — need 4 x 3.2 TB NVMe per broker

Result: 40 partitions, 10 brokers, 4 partitions per broker.
```

Key insight: the bottleneck is usually not CPU or network but disk I/O for replication traffic. Each message is written 3 times (leader + 2 followers). Write amplification equals replication factor. At 375 MB/s ingress, the cluster writes 1.1 GB/s to disk across all brokers. With 10 brokers, each broker writes 110 MB/s — within NVMe capability (500+ MB/s sustained sequential write).

### Drill 7: Queue Failure Mode Walkthrough

You are on-call and receive a page: "Consumer lag for order-processing queue exceeds 50,000 messages." Walk through your investigation step by step.

**Step 1 — Verify the alert:**
Check the monitoring dashboard. Is the lag growing or stable? If it grows at 1,000 messages/second, you have minutes before the queue backs up. If stable at 50,000, the consumer processes at the same rate as the producer — the queue operates at capacity.

**Step 2 — Check consumer health:**
Are all consumers in the group reporting heartbeats? If a consumer crashed, the group rebalances. During rebalancing, no messages are processed. Is consumer CPU high (processing bottleneck) or low (blocked on external dependency)? Check consumer logs for exceptions. A recent deploy may have introduced a bug.

**Step 3 — Check external dependencies:**
If the consumer calls a database or external API, check its response times. A database query that normally takes 5ms may now take 500ms due to a missing index or lock contention.

**Step 4 — Check upstream:**
Did the producer's throughput increase? A marketing campaign or flash sale could cause 10x normal traffic. If the producer rate exceeds the consumer rate, the queue naturally grows. The fix is either scaling consumers or accepting the backlog.

**Step 5 — Decide on action:**
- If the root cause is a consumer bug: roll back the deployment. Let the backlog drain. Do not skip messages.
- If the root cause is an external dependency slowdown: scale consumers horizontally. Each consumer does less work per message because the dependency is slower, but more consumers share the load.
- If the root cause is a traffic spike: accept the backlog if it drains within the SLA. Scale consumers if the backlog exceeds the SLA window.

### Drill 8: Queue Migration Planning

Your company migrates from RabbitMQ to Kafka. You have 200 queues, 500 consumers, and 1 TB of data in RabbitMQ. The migration must have zero data loss. Outline the plan.

**Phase 1 — Dual-routing setup (2 weeks):**
- Deploy a routing service that reads from RabbitMQ and writes to Kafka (replay all existing messages).
- All new producers write to both RabbitMQ and Kafka simultaneously.
- Shadow consumers read from Kafka but discard results. Verify message parity: for every message in RabbitMQ, is there a matching message in Kafka?

**Phase 2 — Consumer migration (4 weeks):**
- Migrate consumer groups one at a time, starting with the least critical.
- For each consumer group: deploy a Kafka consumer, run it alongside the RabbitMQ consumer for 1 week, compare results, then switch the consumer to read from Kafka only.
- Each migration is reversible: if the Kafka consumer has issues, switch back to RabbitMQ within 5 minutes.

**Phase 3 — Cut-over and decommission (2 weeks):**
- Once all consumers read from Kafka only, stop RabbitMQ producers.
- Retain RabbitMQ data for 30 days (retention period) as a backup.
- After 30 days with no issues, decommission RabbitMQ.

**Biggest risk:** Schema incompatibility. RabbitMQ messages are AMQP format. Kafka messages use custom serialization. The migration must define a common message format (JSON, Avro, Protobuf) and convert during dual-write. Any consumer that bypasses the conversion and reads raw RabbitMQ data will fail on the Kafka side.

### Scenario 1: Onboarding Notifications

### Scenario 2: Real-Time Leaderboard

**What the interviewer expects:**
1. Recognize that the leaderboard is a trade-off: strong consistency means low throughput, high throughput means eventual consistency.
2. Propose a queue to ingest score updates asynchronously.
3. Use a stream processor (Kafka Streams, Flink) to maintain a materialized leaderboard.
4. Use sliding window aggregation: top 100 scores in the last 5 minutes.
5. Discuss that stale data is acceptable in a leaderboard — eventual consistency fits the use case.

### Scenario 3: Dead Letter Disaster

"A production queue has 500,000 messages in the DLQ. The team does not know why. They deleted the DLQ and started over. Two days later, the DLQ has 300,000 messages again. What went wrong?"

**What the interviewer expects:**
1. Identify that deleting the DLQ without investigating the root cause guarantees the problem repeats.
2. Walk through the diagnosis: examine DLQ message content (are they all the same type? same source?), check consumer logs, check for a deployment that changed the processing code.
3. Identify the root cause: likely a malformed message type, a new schema version that consumers cannot handle, or a downstream dependency that is returning errors.
4. Propose fixes: fix the consumer code, add a schema compatibility check in the producer, or add a validation step before the main queue.

### Scenario 5: Audit Log Pipeline

"Design an audit log system that records every data access in a hospital system. HIPAA compliance requires that every access to patient records is logged within 1 minute of the access event. **What the interviewer expects:**

### Scenario 6: Rate-Limited API Gateway

"Design an API gateway that rate-limits requests to 100 requests/second per API key. Burst traffic of 1,000 requests/second must be smoothed to 100/second. Requests that exceed the rate limit are queued and processed when capacity is available."

**What the interviewer expects:**
1. Queue as a smoothing buffer: burst traffic fills the queue, the rate limiter drains it at 100/second.
2. Token bucket or leaky bucket algorithm in front of the queue to determine when to enqueue vs reject.
3. Per-key isolation: one slow API key should not starve other keys.
4. Queue sizing: if the max burst is 1,000/sec and the rate is 100/sec, the queue needs 900 slots of headroom. At 1 KB per request, this is under 1 MB of memory.
5. Visibility timeout consideration: since the consumer (rate-limited API caller) cannot exceed 100 requests/second, the prefetch is set to 1 per consumer.
