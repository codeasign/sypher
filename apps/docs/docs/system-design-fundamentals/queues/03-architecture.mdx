### Limitations of the Single Node

A single-node broker hits three hard limits:

**Throughput ceiling:** One process can handle roughly 500,000 messages per second for in-memory queues, or 50,000 for durable queues with fsync. Beyond this, adding CPU cores provides diminishing returns due to lock contention.

**Storage ceiling:** One machine has a finite disk. At 10,000 messages/second with 4 KB payloads, the broker accumulates 3.4 GB/hour. At 500 GB, it runs out of space in 6 days.

**Availability ceiling:** When the single machine goes down — scheduled maintenance, network partition, hardware failure — the entire queue system stops. Messages queued but not consumed are stranded. Producers receive connection errors.

Production architectures solve all three ceilings simultaneously.

### Partitioning — Horizontal Scaling

Partitioning splits a single logical queue into multiple physical queues (partitions). Each partition runs on a separate broker. Producers are assigned to partitions by a partition key.

```python
def partition_key(message):
    # Hash the user_id to assign to one of N partitions
    user_id = message['user_id']
    partition = hash(user_id) % NUM_PARTITIONS
    return partition
```

**Key design decisions for partitioning:**

**Number of partitions:** More partitions mean more parallelism but more overhead. Each partition adds metadata tracking, replication threads, and file handles. Rule of thumb: start with 2x the number of brokers, then monitor and adjust. Kafka recommends 3-10 partitions per broker.

**Partition key selection:** The partition key determines ordering. Messages with the same key go to the same partition and are ordered. Messages with different keys can be processed in parallel but lose relative ordering.

```
Producer sends:
  {user_id: "alice", event: "login"}    → hash("alice") % 3 = 0 → Partition 0
  {user_id: "bob",   event: "purchase"}  → hash("bob") % 3   = 1 → Partition 1
  {user_id: "alice", event: "logout"}   → hash("alice") % 3 = 0 → Partition 0 (same as login)

Partition 0 order: login, logout  (preserved for alice)
Partition 1 order: purchase      (bob's event, independent)
```

**Partition assignment strategies:**

| Strategy | Behavior | Use Case |
|----------|----------|----------|
| Hash-based | `hash(key) % N` | Even distribution, good for many keys |
| Range-based | Key prefix -> partition | Time-series data, region-based routing |
| Round-robin | Sequential assignment | No ordering needed, maximum throughput |
| Custom | Application-defined | Tenant isolation, priority routing |

### Replication — High Availability

Replication copies each partition to multiple brokers. If the leader fails, a follower takes over. Replication factor of 2 means one leader and one follower. Replication factor of 3 means one leader and two followers.

```text
Replication factor = 3:

                    Broker A          Broker B          Broker C
                    ┌────────┐        ┌────────┐        ┌────────┐
                    │Part 0  │        │Part 1  │        │Part 2  │
                    │LEADER  │        │LEADER  │        │LEADER  │
                    └────────┘        └────────┘        └────────┘
                          │                │                │
                    ┌─────┴─────┐    ┌─────┴─────┐    ┌─────┴─────┐
                    │Part 1 FOLLOWER │Part 2 FOLLOWER │Part 0 FOLLOWER
                    └───────────┘    └───────────┘    └───────────┘
                    ┌────────────┐  ┌────────────┐  ┌────────────┐
                    │Part 2 FOLLOWER│Part 0 FOLLOWER│Part 1 FOLLOWER
                    └────────────┘  └────────────┘  └────────────┘
```

**Replication protocols:**

**Leader-based replication (Kafka, RabbitMQ quorum queues):** The leader accepts writes, replicates to followers, and waits for acknowledgment from a configurable number of in-sync replicas (ISR). Producers can configure `acks`:

- `acks=0`: Fire and forget. Highest throughput. Message may be lost.
- `acks=1`: Leader acknowledges after writing locally. Leader crash before replication = message loss.
- `acks=all` (or `-1`): Leader waits for all in-sync replicas to acknowledge. Strongest durability.

```
acks=all write flow:

Producer     Leader (B0)     Follower (B1)    Follower (B2)
   │              │                │                │
   │── send ─────►│                │                │
   │              │── replicate ──►│                │
   │              │── replicate ──────────────────►│
   │              │◄─ ack ────────│                │
   │              │◄─ ack ─────────────────────────│
   │              │                                │
   │◄─ confirmed ─│  (only after both followers ACK) │
```

### Consumer Groups

Consumers are organized into groups. Each group shares the work of consuming from a set of partitions. Within a group, each partition is consumed by exactly one consumer.

```text
Consumer Group "email-senders":
         ┌──────────────────────┐
         │ Partition 0 ────────│── Consumer A (instance 1)
         │ Partition 1 ────────│── Consumer A
         │ Partition 2 ────────│── Consumer B (instance 2)
         │ Partition 3 ────────│── Consumer B
         └──────────────────────┘

If Consumer B crashes:
         ┌──────────────────────┐
         │ Partition 0 ────────│── Consumer A
         │ Partition 1 ────────│── Consumer A
         │ Partition 2 ────────│── Consumer A (rebalanced)
         │ Partition 3 ────────│── Consumer A (rebalanced)
         └──────────────────────┘
```

The rebalance is automatic. When a consumer joins or leaves, the group coordinator reassigns partitions. During rebalancing, no messages are processed.

## Integration Patterns

Queues do not live in isolation. They connect databases, caches, and external APIs.

### Pattern 1: Database + Queue — The Outbox Pattern

An application writes to a database and needs to send a message to a queue. If the database write succeeds but the queue send fails, the system is inconsistent. The outbox pattern solves this by writing the message to the database in the same transaction as the business data, then a separate process reads the outbox and enqueues messages.

### Multi-Region Design Considerations

**Cross-region latency:** The speed of light imposes a hard minimum latency. New York to London is 40ms. New York to Sydney is 160ms. Replication between these regions adds at least these numbers to end-to-end delivery.

**Active-active vs active-passive:**

| Model | Behavior | Use Case |
|-------|----------|----------|
| Active-passive | One region processes all messages. Other regions route to the active region. | Simpler, no conflict resolution needed. Single region of failure. |
| Active-active | Each region processes local messages and replicates to other regions. | Higher availability, lower local latency. Conflicts possible. |

**Regional queuing with global routing:** Producers in each region write to a local queue. Consumers in the same region process messages immediately. A background replication service consumes from each region's queue and publishes to a global aggregate queue.

## Capacity Planning Math

Before deploying, calculate the resources your queue system needs.

### Input Parameters

| Parameter | Symbol | Example |
|-----------|--------|---------|
| Messages per second (peak) | M | 100,000 |
| Average payload size | S | 2 KB |
| Retention period | R | 7 days |
| Replication factor | RF | 3 |
| Desired p99 latency | L | 50 ms |

### Storage Calculation

```
Daily storage = M * S * 86,400 seconds * RF
             = 100,000 * 2 KB * 86,400 * 3
             = 51,840,000,000 KB
             = ~48 TB per day

Weekly storage = 48 * 7 = 336 TB
```

At 48 TB per day, choose a broker that supports tiered storage (hot data on NVMe, warm data on HDD, cold data in S3).

### Memory Calculation

```
Broker memory needed = (In-flight messages * S)
                    + (Index overhead * M_daily)
                    + (Connection buffers)
                    + (OS page cache)

In-flight (p99 burst): 20,000 messages * 2 KB = 40 MB
Index overhead: 100 million messages * 16 bytes = 1.6 GB
Connection buffers: 10,000 connections * 128 KB = 1.28 GB
OS page cache: ~4 GB (for recent index pages)

Total memory: ~7 GB recommended, 16 GB comfortable
```

### Throughput per Broker

```
With batching (1000 msgs/batch):
  Disk writes: 100,000 / 1000 = 100 fsyncs/sec (comfortable on NVMe)
  Network: 100,000 * (2 KB + 60 bytes overhead) = ~200 Mbps (well under 10 Gbps)
  CPU: ~20% of one core per 10,000 msgs/sec (100K/s ~ 2 cores)
```

### Broker Count

```
Messages per broker = 500,000 / RF / N_brokers
With RF=3 and 100K msgs/sec:
  100K = 500K / 3 / N
  N = 500K / (3 * 100K) = ~1.7

Start with 3 brokers. Scale to 6 when approaching 200K msgs/sec.
```

### Monitoring Alerts

```yaml
# Prometheus alerting rules
groups:
  - name: queue_alerts
    rules:
      - alert: QueueDepthGrowing
        expr: rate(queue_messages_ready[5m]) > rate(queue_messages_consumed[5m]) * 1.2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Queue depth growing faster than consumption"

      - alert: QueueDepthCritical
        expr: queue_messages_ready > 100000
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Queue has over 100,000 unprocessed messages"

      - alert: HighConsumerLag
        expr: kafka_consumer_lag > 50000
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Consumer group lag exceeds 50,000 messages"

      - alert: BrokerDown
        expr: up{job="queue-broker"} < 3
        for: 30s
        labels:
          severity: critical
        annotations:
          summary: "Less than 3 queue brokers are healthy"
```

### Monitoring Alerts Explained

Each alert targets a specific failure mode:

**QueueDepthGrowing:** The 5-minute rate of enqueues exceeds the 5-minute rate of dequeues by 20%. This is an early warning — the gap may close on its own (transient burst) or widen into a backlog. The `for: 5m` window prevents alerting on brief spikes.

**QueueDepthCritical:** The ready message count exceeds 100,000. This fires after 2 minutes of sustained high depth. At this point, the backlog poses a risk to the broker's memory, and manual intervention (scaling consumers, increasing consumer throughput) is likely needed.

**HighConsumerLag:** Kafka consumer lag exceeds 50,000 messages. Unlike queue depth, consumer lag measures the exact offset gap between the last committed message and the latest available message. This is the most direct measure of whether consumers are keeping up.

**BrokerDown:** The number of healthy brokers drops below the cluster minimum (3). This fires after 30 seconds to allow for transient network flaps. When a broker goes down, its partition leaders must be re-elected, which causes a brief pause in service for those partitions.

### Zero-Downtime Consumer Deployment

Deploying new consumer code without dropping messages requires a careful rollout strategy. The standard approach is a rolling deploy with consumer group rebalancing.

```yaml
# Kubernetes rolling update for consumers
apiVersion: apps/v1
kind: Deployment
metadata:
  name: queue-consumer
spec:
  replicas: 20
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 5        # allow 5 extra consumers during update
      maxUnavailable: 5  # allow 5 to be down at once
  template:
    spec:
      terminationGracePeriodSeconds: 60
      containers:
      - name: consumer
        image: myapp/consumer:v2
        lifecycle:
          preStop:
            exec:
              command:
              - /bin/sh
              - -c
              - |
                # Signal consumer to stop accepting new messages
                kill -SIGTERM 1
                # Wait for in-flight messages to complete
                sleep 30
```

The `preStop` hook sends a SIGTERM to the consumer process, which triggers a graceful shutdown: the consumer stops polling for new messages, processes its remaining in-flight messages, and commits offsets before exiting. The `terminationGracePeriodSeconds` must be longer than the visibility timeout plus processing time.

### Queue Security

Queues are a critical path in the system. Securing them requires three layers:

**Authentication:** The broker must verify that every connecting client is who it claims to be. RabbitMQ uses username/password or x.509 certificates. Kafka uses SASL/PLAIN, SASL/SCRAM, or mTLS. SQS uses IAM roles and access keys.

**Authorization:** The broker must verify that the authenticated client has permission to perform the requested operation. RabbitMQ uses virtual host-level permissions. Kafka uses ACLs per topic/group. SQS uses IAM policies on the queue resource.

**Encryption in transit:** All traffic between producers, consumers, and the broker should be encrypted with TLS. This prevents eavesdropping on message payloads and credential theft.

```yaml
# Kafka TLS configuration (broker side)
listeners=PLAINTEXT://0.0.0.0:9092,SSL://0.0.0.0:9093
ssl.keystore.location=/etc/kafka/secrets/kafka.keystore.jks
ssl.keystore.password=${KEYSTORE_PASSWORD}
ssl.key.password=${KEY_PASSWORD}
ssl.truststore.location=/etc/kafka/secrets/kafka.truststore.jks
ssl.truststore.password=${TRUSTSTORE_PASSWORD}
ssl.client.auth=required
```

**Encryption at rest:** If the queue stores sensitive data (PII, payment details), the broker's persistent storage should be encrypted. For Kafka, this means encrypted EBS volumes. For SQS, server-side encryption with KMS is enabled by default.

### Readiness Probes for Consumers

A consumer that is alive but not processing (stuck on a GC pause, waiting on a database lock) should not be receiving messages. A readiness probe checks whether the consumer can actually process work.

```python
# Consumer readiness probe endpoint
@app.get("/health/ready")
def readiness():
    # Check that we can connect to the queue broker
    try:
        broker_ok = check_broker_connection()
    except Exception:
        broker_ok = False

    # Check that recent processing completed successfully
    recent_failures = get_recent_failure_rate(minutes=5)
    processing_ok = recent_failures < 0.01  # less than 1% failures

    # Check memory pressure
    memory_pressure = get_memory_usage()
    memory_ok = memory_pressure < 0.85

    if broker_ok and processing_ok and memory_ok:
        return {"status": "ready"}, 200
    return {"status": "not ready"}, 503
```

Kubernetes uses this endpoint in its readiness probe. If the probe fails, Kubernetes stops sending traffic to the pod and eventually restarts it. This prevents a stuck consumer from holding messages in the in-flight state indefinitely.

### Autoscaling Queue Consumers

Auto-scaling based on queue depth is the standard pattern. When messages accumulate, more consumers are needed. When the queue is empty, excess consumers should be scaled down to save cost.

```yaml
# Horizontal Pod Autoscaler (Kubernetes HPA) based on custom queue metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: queue-consumer-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: queue-consumer
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: External
    external:
      metric:
        name: sqs_approximate_number_of_messages_visible
        selector:
          matchLabels:
            queue_name: transcode-jobs
      target:
        type: AverageValue
        averageValue: 100  # scale up when >100 visible messages per consumer
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5 min before scaling down
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
```

The autoscaler targets a visible message count of 100 per consumer. If the queue has 500 messages and 3 consumers, that is 166 messages per consumer, so the HPA scales up to 5 consumers (500 / 100 = 5). The scale-down stabilization window prevents flapping — rapid scale-up and scale-down as the queue depth oscillates.
