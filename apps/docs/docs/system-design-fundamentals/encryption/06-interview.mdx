# Encryption — Interview Playbook

A targeted interview guide for encryption in distributed systems contexts.
Covers the question landscape, level-appropriate expectations across L3
through L8, a structured FAANG response framework, a curated phrase bank for
articulate answers, common red flags that sink candidates, and five drills
to build fluency.

---

## Question Types

Encryption questions appear in system design interviews in five distinct
forms. Each type tests a different dimension of your knowledge.

### Type 1: Conceptual Foundation

These probe whether you understand the fundamental concepts of encryption,
key management, and cryptographic primitives.

**Examples:**
- What is the difference between symmetric and asymmetric encryption?
- Explain TLS handshake at a high level. What problem does each step solve?
- What is a certificate authority and how does it establish trust?
- What is Perfect Forward Secrecy and why does it matter?
- How does a hash function differ from encryption?

**What interviewers look for:** Clear, precise explanations of cryptographic
concepts without jargon-overload. Ability to distinguish between encryption,
hashing, and signing. Understanding of threat models.

### Type 2: Security Architecture

These test your ability to design systems that use encryption correctly at
scale.

**Examples:**
- How would you encrypt data at rest in a distributed storage system?
- Design a key management system for a platform with 100+ microservices.
- How do you handle encryption in a multi-tenant database?
- What is your approach to encrypting data in transit across multiple data
  centers?
- How would you design a system that supports client-side encryption?

**What interviewers look for:** End-to-end thinking from key generation
through rotation and revocation. Awareness of performance implications.
Practical deployment experience.

### Type 3: Performance and Trade-offs

These test your understanding of the costs and operational implications of
encryption.

**Examples:**
- What is the performance impact of enabling TLS on a high-throughput API?
- When would you choose AES-256-GCM over ChaCha20-Poly1305?
- How does encryption affect database indexing and query performance?
- What is the cost of re-encrypting 10 TB of data when rotating a key?
- How do you balance encryption strength with latency requirements?

**What interviewers look for:** Quantitative reasoning about encryption
overhead. Knowledge of hardware acceleration (AES-NI, QAT). Awareness of
real-world performance numbers.

### Type 4: Compliance and Regulatory

These test your knowledge of how encryption intersects with legal and
regulatory requirements.

**Examples:**
- How does encryption help meet PCI-DSS requirements?
- What encryption strategies would you use for GDPR compliance?
- How do you handle data residency requirements while maintaining encryption?
- What is the difference between encryption at rest and encryption in use?
- How do you handle government requests for encrypted data?

**What interviewers look for:** Understanding of compliance frameworks.
Awareness that encryption is necessary but not sufficient for compliance.
Practical experience with audit requirements.

### Type 5: Incident Response and Operations

These test your ability to handle encryption-related failures and operational
challenges.

**Examples:**
- A signing key was leaked. Walk through the incident response.
- Your application cannot decrypt data after a key rotation. What happened?
- How do you monitor encryption health across a distributed system?
- A customer reports that their encrypted data is corrupted. How do you
  investigate?
- How do you test encryption changes without risking production data?

**What interviewers look for:** Systematic incident response approach.
Understanding of key lifecycle management. Practical operational experience.

## Level Expectations

Interviewers calibrate their expectations by level. The same question about
key management receives very different follow-ups depending on whether you
are interviewing for L3 or L7.

### L3 — Individual Contributor (Entry)

**Focus:** Correct understanding of fundamentals. Ability to implement
standard encryption patterns following established guidelines.

**Sample question:** "Explain how TLS ensures secure communication between a
client and a server."

**Expected answer (L3 pass):**
"TLS uses a combination of asymmetric encryption for key exchange and
symmetric encryption for bulk data. During the handshake, the client and
server agree on a protocol version, select a cipher suite, authenticate the
server using its certificate, and derive a shared session key. The server's
certificate is signed by a certificate authority, which the client trusts.
Once the handshake completes, all data is encrypted with the session key
using a symmetric cipher like AES-GCM. This provides confidentiality,
integrity, and server authentication."

**What separates a strong L3:** Mentioning that the handshake also performs
certificate validation, including checking the certificate chain, expiration,
and revocation status. Showing awareness of TLS 1.3 improvements over 1.2.

**What fails L3:** Confusing TLS with SSL. Thinking encryption alone prevents
tampering without mentioning integrity/MAC. Not understanding the role of
certificate authorities.

### L4 — Mid-Level

**Focus:** Understanding of trade-offs. Ability to design encryption for a
single service. Awareness of key management concerns.

**Sample question:** "Design the encryption layer for a document storage
service. Documents are uploaded by users and retrieved later."

**Expected answer (L4 pass):**
"I would use envelope encryption. Each document gets a unique data encryption
key (DEK) generated with AES-256. The DEK encrypts the document. The DEK
itself is encrypted with a key encryption key (KEK) that is stored in a
hardware security module or key management service. The encrypted DEK is
stored alongside the document metadata. On retrieval, the KEK decrypts the
DEK, and the DEK decrypts the document. This means we never store the raw
DEK, and rotating the KEK does not require re-encrypting every document. For
transit, all API calls use TLS 1.3 with a minimum of 1.2 enforced."

**What separates a strong L4:** Explaining why envelope encryption is
preferred over single-key encryption (key rotation cost, blast radius).
Discussing the trade-off between HSM performance and software key stores.

**What fails L4:** Proposing to store the encryption key in a config file.
Not distinguishing between encryption at rest and in transit. Not having a
key rotation plan.

### L5 — Senior

**Focus:** Multi-service architecture. Key management at scale. Performance
optimization. Security hardening.

**Sample question:** "Design a key management system for a platform with 200
microservices. Services need to encrypt and decrypt data at rest."

**Expected answer (L5 pass):**
"I would use a centralized KMS with a tiered key hierarchy. A master key in
an HSM protects per-team KEKs, which protect per-service DEKs. The KMS
exposes `encrypt()` and `decrypt()` operations with mTLS authentication and
rate limiting. Decrypted DEKs are cached in memory with a short TTL to avoid
HSM throughput bottlenecks. Key rotation happens at three levels: master key
annually via HSM rekey, KEKs quarterly, DEKs on every re-encryption. An
audit log records every key access and rotation. For DR, the master key is
replicated across two geographically separate HSMs."

**What separates a strong L5:** Quantifying KMS throughput and caching
strategy. Discussing blast radius: a compromised KEK exposes one team's
data, while the master key stays protected by the HSM.

**What fails L5:** Proposing a single key for all services. Not considering
HSM throughput limits. Ignoring audit logging. No DR plan for the master key.

### L6 — Staff

**Focus:** Organization-wide encryption strategy. Cross-cutting performance
and security architecture. Regulatory compliance.

**Sample question:** "Our company processes sensitive financial data across
six regions. We need a unified encryption strategy that meets regulatory
requirements in each region while keeping performance acceptable. Design the
system."

**Expected answer (L6 pass):**
"I would establish a three-layer architecture. Transport layer: all
inter-service communication uses mTLS with TLS 1.3 and a curated cipher
suite. Data layer: envelope encryption with region-specific KEKs stored in
regional HSMs — a breach in one region does not compromise another.
Application layer: sensitive fields use field-level encryption with AES-SIV
for deterministic encryption, enabling indexed lookups. Compliance mappings:
GDPR right-to-deletion maps to destroying the user's DEK; PCI-DSS maps to
field-level encryption of cardholder data; data residency laws require KEKs
never leave their region. A compliance dashboard maps each regulatory
requirement to the corresponding encryption control."

**What separates a strong L6:** Knowledge of AES-SIV for queryable encrypted
fields. Regional key isolation. Mapping controls to specific regulations.

**What fails L6:** Single global key across regions. Ignoring data residency.
Not accounting for the performance impact of field-level encryption.

### L7 — Principal / L8 — Distinguished

**Focus:** Long-term cryptography strategy. Industry standards influence.
Organization-wide security posture. Zero-trust architecture.

**Sample question:** "We are migrating our entire infrastructure to a
zero-trust model. Encryption is a core pillar. How do you design the
encryption strategy?"

**Expected answer (L7 pass):**
"Zero-trust encryption means we do not trust the network, infrastructure, or
other services. I would implement four pillars. Pillar one: every
service-to-service call uses mTLS with short-lived certificates via the
SPIFFE identity standard, renewed every 24 hours. Pillar two: all data at
rest uses per-resource DEKs under envelope encryption, KEKs managed by a
centralized KMS enforcing least-privilege policies. Pillar three: sensitive
data in use is protected with confidential computing using AMD SEV-SNP or
Intel TDX enclaves. Pillar four: all keys are hardware-backed in HSMs or
TPMs. I would establish a cross-team cryptography review board to approve
all new schemes and cipher suites. The migration is phased: first mTLS, then
data-at-rest encryption, then confidential computing for sensitive workloads,
each phase with a rollback plan."

**What separates a strong L7:** Knowledge of confidential computing and TEEs.
SPIFFE/SPIRE for identity. Cryptography review board. Phased migration.

**What fails L7:** All-at-once approach that is too expensive. Not considering
operational burden. Ignoring the performance cost of TEEs.

## FAANG Response Framework

Use this structured approach for any encryption system design question. It
mirrors the evaluation criteria used at FAANG companies.

### Step 1: Scope and Clarify (1-2 minutes)

Before diving into design, ask clarifying questions that demonstrate breadth
of thinking.

**Good clarifying questions:**
- "What data classification levels are we protecting — public, internal,
  confidential, or restricted?"
- "Are we designing for encryption at rest, in transit, in use, or all three?"
- "What is the threat model? Are we protecting against external attackers,
  insider threats, nation-state actors, or all of the above?"
- "What compliance requirements apply — PCI-DSS, HIPAA, GDPR, SOC2?"
- "What is the expected scale — data volume, request rate, number of keys?"
- "Are there existing key management infrastructure components we must
  integrate with?"
- "What is the latency budget for encryption and decryption operations?"

**Why this matters:** Encryption is expensive and complex. A candidate who
asks clarifying questions before proposing a solution shows they understand
that encryption is a trade-off, not a checkbox.

### Step 2: Define Requirements (2-3 minutes)

State the requirements explicitly so the interviewer can correct any
misassumptions.

**Functional requirements:**
- Encrypt data at rest (storage-level or field-level)
- Encrypt data in transit (TLS/mTLS between all services)
- Key generation, storage, rotation, and destruction
- Access control for encryption keys (who can encrypt, who can decrypt)
- Audit logging for all cryptographic operations

**Non-functional requirements:**
- Performance: encryption adds latency and CPU overhead — quantify the budget
- Availability: the KMS must be highly available (99.99%+) or services cannot
  decrypt data
- Durability: loss of a key means permanent data loss — backup and recovery
  are critical
- Compliance: meet regulatory requirements for key management and data
  protection
- Operational: key rotation, monitoring, alerting, incident response

### Step 3: High-Level Design (3-5 minutes)

Present the architecture as a layered model. Interviewers want to see you
trace the complete lifecycle of encryption from key generation through data
encryption to key rotation.

**Components to include:**
- **Key Management Service (KMS):** Centralized service for key generation,
  storage, rotation, and access control
- **Hardware Security Module (HSM):** Tamper-resistant hardware that protects
  master keys and performs cryptographic operations
- **Key Hierarchy:** Master Key → KEK (Key Encryption Key) → DEK (Data
  Encryption Key) — envelope encryption
- **Encryption Service / Library:** Client-side library or sidecar that
  handles encryption and decryption using keys from the KMS
- **Audit Log:** Immutable log of all key access, creation, rotation, and
  destruction events
- **Certificate Authority (CA):** Issues and manages TLS certificates for
  service identity

**Flow:**
1. Application requests a new encryption key → KMS generates a DEK and
   encrypts it with the KEK → returns the encrypted DEK to the application
2. Application encrypts data with the DEK → stores the encrypted DEK alongside
   the ciphertext
3. On read, application sends the encrypted DEK to the KMS → KMS decrypts it
   with the KEK → returns the plaintext DEK
4. Application decrypts the ciphertext with the DEK
5. Key rotation: KMS generates a new KEK → re-encrypts all DEKs with the new
   KEK (or lazily on next access)

### Step 4: Deep Dive (5-10 minutes)

The interviewer will choose one area to explore in depth. Be ready for each
of these deep dives.

**Deep Dive A — Key Rotation:**
- Re-encrypting all data is expensive. Envelope encryption solves this: only
  re-encrypt the DEKs, not the data itself.
- Lazy rotation: re-encrypt the DEK with the new KEK on the next read or
  write. Simple but does not immediately protect data encrypted with the
  old key.
- Active rotation: scan all encrypted data and re-encrypt DEKs. Thorough but
  resource-intensive.
- Key versioning: store a `key_version` with each ciphertext so the correct
  KEK is used for decryption.

**Deep Dive B — Performance Optimization:**
- Use hardware acceleration: AES-NI for AES, QAT for compression/crypto.
- Cache decrypted DEKs in memory with a short TTL (e.g., 5 minutes).
- Avoid per-request KMS calls — batch key operations where possible.
- Choose cipher suites wisely: AES-256-GCM is hardware-accelerated on most
  CPUs; ChaCha20-Poly1305 is faster on mobile devices without AES-NI.
- For high-throughput systems, consider dedicated encryption hardware or
  offloading to a sidecar proxy.

**Deep Dive C — Key Storage and Access Control:**
- Master keys: HSM or cloud KMS (AWS KMS, Azure Key Vault, GCP Cloud KMS).
- DEKs: encrypted and stored alongside the data in the application database.
- Access control: IAM policies on the KMS with least-privilege principle.
  Service A can encrypt using KEK-1 but cannot decrypt using KEK-2.
- Audit: every key access must be logged with service identity, operation,
  timestamp, and result.

**Deep Dive D — Encryption in Transit vs. At Rest vs. In Use:**
- In transit: TLS 1.3 with mTLS for service-to-service. Certificate pinning
  for critical paths. Cipher suite restrictions (no weak ciphers).
- At rest: AES-256-GCM or ChaCha20-Poly1305. Envelope encryption for key
  management. File-level, database-level, or field-level depending on
  granularity needs.
- In use: Confidential computing with TEEs (AMD SEV-SNP, Intel TDX, AWS Nitro
  Enclaves). Homomorphic encryption for specific use cases (very expensive).
- Data-in-use is still an emerging space. For most systems, encrypt at rest
  and in transit, and use access controls for data in use.

### Step 5: Trade-offs and Alternatives (2-3 minutes)

Explicitly state what you are trading off and what alternatives exist.

**Encryption trade-offs:**
- Stronger encryption vs. slower performance (AES-256 vs. AES-128, RSA-4096
  vs. RSA-2048)
- Envelope encryption vs. single-key simplicity (more complex but scalable)
- Centralized KMS vs. client-side key management (better control vs. lower
  latency)
- HSM vs. software key store (higher security vs. higher cost and lower
  throughput)
- Full-database encryption vs. field-level encryption (simpler vs. more
  granular)

**Alternatives to mention:**
- **Application-layer encryption:** Encrypt data before it reaches the
  database. Offers fine-grained control but requires more code.
- **Database-native encryption (TDE):** Transparent, no application changes.
  But does not protect against database admins.
- **Client-side encryption:** Data is encrypted on the client before being
  sent to the server. The server never sees plaintext. Used for zero-knowledge
  systems.
- **Tokenization:** Replace sensitive data with tokens. Simpler than encryption
  for PCI compliance but does not provide confidentiality of the original
  data format.

### Step 6: Summary and Operations (1-2 minutes)

Close with operational considerations that show production readiness.

**Operational checklist:**
- Key rotation policy (rotate KEKs quarterly, master keys annually, DEKs
  on every re-encryption)
- Monitoring: KMS latency (P50, P95, P99), encryption/decryption throughput,
  key access counts, failed decryption attempts
- Alerting: spike in decryption failures (potential key compromise), KMS
  latency above threshold, certificate expiry within 30 days
- Backup and recovery: backup master keys in a secondary HSM in a different
  region. Test recovery quarterly.
- Incident response plan: key compromise playbook, key loss playbook,
  certificate revocation process
- Penetration testing: specifically test for key leakage, weak cipher
  acceptance, certificate validation bypass

## Phrase Bank

Use these phrases to express ideas clearly and with authority. Organized by
concept area.

### Explaining Encryption Fundamentals

1. "Encryption provides confidentiality, but not integrity or authentication on
   its own. You need an authenticated encryption mode like GCM to ensure the
   ciphertext has not been tampered with."

2. "Symmetric encryption is fast for bulk data but requires sharing the key.
   Asymmetric encryption solves key distribution but is orders of magnitude
   slower. We use asymmetric for key exchange and symmetric for actual data."

3. "A hash function is one-way — you cannot recover the input. Encryption is
   two-way — with the key, you recover the original data. Different tools."

### Key Management

4. "Envelope encryption is the standard at scale. A DEK encrypts the data, and
   a KEK protects the DEK. Rotating the KEK does not require re-encrypting all
   data, only the DEKs."

5. "The key hierarchy limits blast radius. A compromised DEK exposes one
   resource. A compromised KEK exposes all DEKs it protects. The master key
   is the crown jewel and must be stored in an HSM."

6. "Key rotation is a security and compliance requirement. Lazy rotation with
   key versioning is the pragmatic approach — rotate the KEK and re-encrypt
   DEKs lazily on next access."

### TLS and Networking

7. "TLS 1.3 reduced the handshake from two round trips to one. Combined with
   0-RTT resumption, returning users can send data on the very first packet."

8. "Perfect Forward Secrecy ensures past sessions cannot be decrypted if the
   private key is compromised. TLS 1.3 mandates PFS by removing static RSA
   key exchange — every session uses an ephemeral key."

9. "Mutual TLS is essential for zero-trust. Both sides present certificates,
   eliminating the need for API keys or secrets in service-to-service
   communication."

### Performance and Optimization

10. "AES-NI accelerates AES encryption on modern CPUs. AES-256-GCM can encrypt
    at 5-10 GB/s per core — negligible overhead for most applications."

11. "ChaCha20-Poly1305 is the better choice for mobile devices without AES-NI.
    It is a software-only cipher, faster than AES without hardware support,
    and the default cipher in TLS 1.3 for this reason."

12. "The cost of encryption is not just CPU cycles. Key management, certificate
    lifecycle, and rotation overhead all contribute. The encryption itself is
    often the cheapest part."

### Compliance and Regulation

13. "Encryption is a technical control, not a compliance checkbox. You must
    demonstrate that keys are managed, access is controlled, and rotation
    happens on schedule. The audit trail is as important as the encryption."

14. "Data residency and encryption interact in complex ways. You cannot decrypt
    data in a region where the key does not reside. Your KMS must be regional,
    with keys that never leave their home region."

15. "For GDPR right-to-deletion, encryption provides cryptographic erasure:
    delete the key and the data becomes unrecoverable. This is faster than
    deleting individual records."

### Architecture and Design

16. "Field-level encryption allows different fields to have different keys and
    access policies. Credit card numbers get one key, email addresses another.
    This is essential for PCI-DSS compliance."

17. "Deterministic encryption with AES-SIV produces the same ciphertext for the
    same plaintext, enabling indexed lookups. The trade-off is it reveals
    equality patterns, so use it only for low-cardinality, non-sensitive fields."

18. "Confidential computing with TEEs protects data in use. Data is encrypted
    in memory and decrypted only inside a hardware enclave inaccessible to the
    host OS. This is the frontier of encryption, but adds significant cost."

## Red Flags

These are mistakes and warning signs that interviewers flag. Avoid them in
your answers and watch for them in system designs you review.

### Red Flag 1: "Encryption solves all security problems"

Claiming encryption alone makes a system secure. Encryption provides
confidentiality and integrity, but not authentication or input validation.

**Correct framing:** "Encryption is a critical layer, not a silver bullet. We
still need authentication, authorization, input validation, and monitoring."

### Red Flag 2: "We will just encrypt the entire database"

TDE protects against physical disk theft but not application attacks or
database admin access.

**Correct framing:** "TDE is a baseline. For sensitive data, we need
application-layer or field-level encryption so data is encrypted before
reaching the database."

### Red Flag 3: No key rotation plan

Designing without a rotation mechanism signals inexperience.

**Correct framing:** "Key rotation is built in from day one. Envelope
encryption with key versioning rotates KEKs without re-encrypting all data."

### Red Flag 4: Single key for all data

One key for everything means a single compromise exposes everything.

**Correct framing:** "Unique DEK per resource limits blast radius. The KEK
protects the DEKs and lives in an HSM. A compromised DEK exposes only one
resource."

### Red Flag 5: "We do not need TLS — we are on a private network"

Private networks are not a security boundary. Insider threats and lateral
movement are real risks.

**Correct framing:** "Every inter-service call should use TLS, preferably
mTLS. This is a zero-trust fundamental. The performance cost is negligible."

### Red Flag 6: Storing keys in source code or config files

Keys in code are accessible to too many people and cannot be rotated without
a deploy.

**Correct framing:** "Keys must be in a dedicated KMS or HSM. Application
code never sees raw key material — it calls the KMS to encrypt or decrypt."

### Red Flag 7: Ignoring the performance impact of encryption

Encryption is not free. Proposing it everywhere without cost analysis is a
red flag.

**Correct framing:** "AES-GCM with AES-NI is fast, but asymmetric operations
like RSA signing are expensive. We cache public keys, batch operations, and
measure before deploying."

### Red Flag 8: "We can just use AES-256 and be done"

AES-256 without the right mode is dangerous. ECB is insecure, CBC has padding
oracle attacks, GCM requires proper nonce management.

**Correct framing:** "The mode matters as much as the algorithm. I use
AES-256-GCM for authenticated encryption with a unique nonce per key. For
deterministic encryption, I use AES-SIV."

### Red Flag 9: Confusing encryption and hashing

Using the terms interchangeably or encrypting passwords instead of hashing
them.

**Correct framing:** "Passwords are hashed, not encrypted. Hashing is one-way.
Encryption is reversible. We use AES-GCM for encryption and bcrypt or Argon2
for passwords."

### Red Flag 10: No disaster recovery plan for key loss

Key loss equals permanent data loss. No backup plan is unacceptable.

**Correct framing:** "Key loss is data loss. Master keys are backed up in a
secondary HSM in a different region, encrypted with a recovery key stored
offline in a physical safe. Recovery is tested quarterly."

## Drills

Five drills to build interview fluency. Each should be practiced aloud with
a timer.

### Drill 1: The Elevator Pitch (2 minutes)

**Task:** Explain the difference between symmetric and asymmetric encryption
to a non-technical product manager in 2 minutes. No jargon. Focus on the
trade-off between speed and key distribution.

**Practice script:**
"Symmetric encryption is like a lockbox with one key. Fast, but you must get
the key to the other person securely. Asymmetric encryption is like a mailbox
with a slot. Anyone can drop a letter in (public key), but only the owner can
open it (private key). No shared secret needed. In practice, we use asymmetric
to exchange a temporary key, then symmetric for the actual data — exactly what
TLS does."

**Check:** Can you explain it without using "algorithm," "cipher," "RSA," or "AES"?

### Drill 2: Key Management Design (5 minutes)

**Task:** Design a key management system for a SaaS company with 50 services
and 1 million users. Present your design in 5 minutes.

**Structure:**
1. Clarify (1 min): "What data? Threat model? Compliance requirements?"
2. Key hierarchy (1 min): "Three-tier: master key in HSM, one KEK per team,
   one DEK per user or document."
3. Key lifecycle (1 min): "DEKs generated on first write, encrypted with KEK,
   stored alongside data. KEK never leaves the KMS."
4. Key rotation (1 min): "KEKs rotate quarterly with lazy re-encryption of
   DEKs. Master key rotates annually. Key versions tracked in metadata."
5. Operations (1 min): "KMS exposes REST API with mTLS. Every operation logged.
   Monitor latency, key access counts, failed decryptions. Backup master key
   to a secondary HSM in a different region."

**Check:** Did you cover the key hierarchy, rotation, and operational concerns?

### Drill 3: TLS Deep Dive (5 minutes)

**Task:** Your interviewer says "Walk me through the full TLS 1.3 handshake
and explain what each step accomplishes." You have 5 minutes.

**Structure:**
1. Protocol negotiation (1 min): "Client sends ClientHello with supported TLS
   versions, cipher suites, and key exchange params. Server responds. In TLS
   1.3, this takes one round trip."
2. Key exchange (1.5 min): "ECDHE key exchange. Each side generates an
   ephemeral key pair, exchanges public keys, computes a shared secret.
   This provides Perfect Forward Secrecy."
3. Certificate validation (1.5 min): "Server sends its certificate chain.
   Client validates up to a trusted root CA — checks signature, expiration,
   and revocation via OCSP stapling or CRL."
4. Finished and encrypted data (1 min): "Both sides derive encryption keys
   from the shared secret via a KDF. All data encrypted with AES-GCM or
   ChaCha20-Poly1305."

**Check:** Did you mention ECDHE, PFS, certificate validation, and TLS 1.3
removing weak ciphers?

### Drill 4: Incident Response — Key Compromise (5 minutes)

**Task:** A signing key used for JWT validation has been leaked. Walk through
your incident response plan. You have 5 minutes.

**Structure:**
1. Detection and containment (1 min): "How was the leak detected — alert,
   report, or audit log anomaly? Immediately rotate the compromised key. All
   tokens signed with the old key are invalid. Add the old key ID to a
   revocation list."
2. Assessment (1 min): "Determine the scope: which services use this key,
   what data was accessible, how long was the key exposed. Check audit logs
   for unauthorized access during the exposure window."
3. Remediation (1.5 min): "Generate a new key pair. Update all services that
   validate against this key with the new public key. For services that cannot
   be updated immediately, add the old key to a trusted-but-deprecated set
   with a short TTL. Issue new tokens with the new key."
4. Communication (30 sec): "Notify affected teams, security, and compliance.
   If required, file an incident report. For regulated data, notify the
   relevant authority."
5. Prevention (1 min): "Conduct a root cause analysis. How did the key leak?
   Was it stored in a config file, logged, or exposed through an API? Fix the
   root cause. Add automated scanning for key material in code repositories.
   Implement key rotation automation so that rotation is a button click, not
   a manual process."

**Check:** Did you cover all phases of incident response? Did you prioritize
containment over investigation? Did you mention prevention?

### Drill 5: Encryption Decision Tree (5 minutes)

**Task:** You are advising a product team that needs to protect user data.
They ask: "Should we encrypt at the application layer, database layer, or
both?" Walk through your decision tree. You have 5 minutes.

**Structure:**
1. Ask clarifying questions (1 min): "What is the sensitivity of the data?
   What compliance requirements apply? Who needs access to the data? What is
   the query pattern — do you need to search by encrypted fields?"
2. Decision tree (2.5 min):
   - "If the data is highly sensitive (PII, financial, health) AND you need
     compliance: application-layer encryption with field-level granularity.
     This gives you the strongest protection and audit control."
   - "If the data is moderately sensitive AND you need indexed queries:
     database-layer encryption (TDE) as a baseline with application-layer
     encryption for the most sensitive fields using AES-SIV."
   - "If the data is low sensitivity AND performance is the primary concern:
     TDE is sufficient. The performance overhead is minimal."
   - "If you need zero-knowledge (the server cannot see plaintext):
     client-side encryption. The server stores encrypted data and never
     has access to the keys."
3. Recommendation (1 min): "For most systems, I recommend a layered approach:
   TDE as a baseline protection for data at rest, plus application-layer
   encryption for sensitive fields. This gives you defense in depth without
   sacrificing too much performance."
4. Trade-offs (30 sec): "Application-layer encryption is more work — you
   cannot use database features like indexing, sorting, or searching on
   encrypted fields without deterministic encryption. And you need to manage
   keys yourself."

**Check:** Did you present a decision tree, not a single answer? Did you
ask clarifying questions? Did you explain the trade-offs?

## Quick Reference Card

Keep these numbers and facts in mind for quick recall during interviews.

**Encryption at a glance:**
- Symmetric: AES-128, AES-256, ChaCha20 — fast, single key, bulk data
- Asymmetric: RSA-2048, RSA-4096, ECDSA P-256, Ed25519 — slow, key pair, key
  exchange and signing
- Hashing: SHA-256, SHA-3, BLAKE2 — one-way, integrity, password hashing
  uses bcrypt/Argon2

**Key sizes:**
- AES-128: 128-bit key, 10 rounds, adequate for most use cases
- AES-256: 256-bit key, 14 rounds, recommended for sensitive data
- RSA-2048: 2048-bit key, equivalent to ~112-bit symmetric security
- RSA-4096: 4096-bit key, equivalent to ~128-bit symmetric security
- ECDSA P-256: 256-bit key, equivalent to ~128-bit symmetric security
- Ed25519: 256-bit key, equivalent to ~128-bit symmetric security

**Performance (approximate, per core with AES-NI):**
- AES-256-GCM encryption: 5-10 GB/s
- AES-256-GCM decryption: 5-10 GB/s
- ChaCha20-Poly1305: 1-3 GB/s (software only)
- RSA-2048 sign: ~10,000 ops/s
- RSA-2048 verify: ~200,000 ops/s
- ECDSA P-256 sign: ~50,000 ops/s
- ECDSA P-256 verify: ~25,000 ops/s
- Ed25519 sign: ~100,000 ops/s
- Ed25519 verify: ~50,000 ops/s

**TLS version comparison:**
- TLS 1.2: 2-RTT handshake, supports weak ciphers, PFS optional
- TLS 1.3: 1-RTT handshake (0-RTT with resumption), removes weak ciphers,
  mandates PFS, AEAD ciphers only

**Recommended cipher suites:**
- TLS 1.3: TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256
- TLS 1.2: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384, TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384

**Key rotation intervals:**
- Master key (HSM-backed): annual
- Key encryption keys (KEKs): quarterly
- Data encryption keys (DEKs): on every re-encryption or per-document
- TLS certificates: 90 days (automated via ACME/Let's Encrypt)

**Standard encryption modes:**
- GCM: authenticated encryption, unique nonce per key — recommended
- CBC: requires padding, padding oracle attacks — avoid in new systems
- CTR: stream cipher, no authentication — use only with a separate MAC
- SIV: deterministic, misuse-resistant — use for indexed encrypted fields
- CCM: authenticated, less efficient than GCM

**Critical validation checks for TLS:**
1. Certificate chain is valid and trusted
2. Certificate has not expired
3. Server name matches certificate SAN
4. Certificate not revoked (OCSP or CRL)
5. Cipher suite is in the allowed list
6. Key exchange uses ECDHE (PFS)

**Common ports:**
HTTPS 443, mTLS any port, LDAPS 636, PostgreSQL 5432, MySQL 3306, SQL Server 1433

**Key management acronyms:**
- HSM: Hardware Security Module
- KMS: Key Management Service
- TDE: Transparent Data Encryption
- TEE: Trusted Execution Environment
- CA: Certificate Authority
- PKI: Public Key Infrastructure
- PFS: Perfect Forward Secrecy
- AEAD: Authenticated Encryption with Associated Data
