# Encryption — Trade-offs & Failure Modes

Encryption is not a switch you flip once and forget. Every cipher, every mode, every key-management strategy carries a trade-off between **security**, **performance**, **operational complexity**, and **availability**. When you choose stronger protection, you accept higher CPU cost, greater latency, or the risk of permanent data loss. When you choose speed, you may open a window for attackers.

This page maps those trade-offs so you can make deliberate, defensible choices in your system designs.

### How you should read this

- **Security**: How well does the algorithm resist known cryptanalytic attacks? AES-256 and ChaCha20 are both considered secure for classified data.
- **CPU Cost**: How many cycles per byte does encryption consume? This matters at scale when you encrypt every API response.
- **Latency**: How much wall-clock time does each operation add? Critical for real-time systems like video calls or gaming.
- **HW Acceleration**: Does the algorithm have dedicated CPU instructions (AES-NI, ARMv8 Crypto Extensions)? Without hardware support, software implementations are slower.
- **Key Size**: Larger keys mean more entropy but also more overhead during key exchange and storage.
- **Stream Cipher**: Stream ciphers handle variable-length data without padding, which avoids padding-oracle attacks.

### When to pick each algorithm

**AES-128-GCM** -- Use this as your default for most systems. It provides excellent security, runs fast with hardware acceleration, and has broad library support. The 128-bit key is sufficient for all non-classified data.

**AES-256-GCM** -- Use this when regulations require it (FIPS 140-2 Level 4, NSA Suite B) or when you are encrypting data that must remain confidential for decades. The 256-bit key provides security against quantum brute-force attacks, though Grover's algorithm reduces effective strength to 128 bits.

**ChaCha20-Poly1305** -- Use this on mobile devices or systems without AES hardware acceleration. It is a stream cipher with no padding overhead and performs well in software. Google uses it for TLS on Android.

**XSalsa20-Poly1305** -- Use this when you need long-lived nonces or want to avoid nonce management entirely. The 192-bit nonce lets you generate random nonces without tracking state.

## Failure Modes

Encryption failures are rarely about the cipher being broken. They are almost always about operational mistakes: lost keys, reused nonces, downgraded protocols, or misconfigured systems.

You must design for these failures before they happen.

### 1. Key Loss Equals Data Loss

This is the scariest failure mode because it is irreversible. If you lose the encryption key for data at rest, that data is gone forever. There is no recovery, no backdoor, no master override.

```
┌─────────────────────────────────────────────────────────┐
│             Key Loss Recovery Options                   │
├─────────────┬───────────────────────────────────────────┤
│ Scenario    │ Can you recover the data?                 │
├─────────────┼───────────────────────────────────────────┤
│ Lost DEK    │ No. Data is permanently unrecoverable.    │
├─────────────┼───────────────────────────────────────────┤
│ Lost KEK    │ No. Wrapped DEKs cannot be unwrapped.     │
├─────────────┼───────────────────────────────────────────┤
│ Lost master │ Yes, if you backed up the KMS key         │
│ key in KMS  │ to another region or exported it safely.  │
├─────────────┼───────────────────────────────────────────┤
│ Corrupted   │ No. Even one corrupted byte in the key    │
│ key         │ makes decryption impossible.              │
└─────────────┴───────────────────────────────────────────┘
```

**Mitigations**: Use a key management service (AWS KMS, GCP Cloud KMS, HashiCorp Vault) with automatic key backup and cross-region replication. Store a paper copy of the master key in a safe. Enable key-rotation policies so you never depend on a single key version.

### 2. Nonce Reuse

Nonce (or IV) reuse is catastrophic for authenticated encryption modes like GCM. Reusing a nonce with the same key allows an attacker to recover the authentication key and forge messages.

```
┌─────────────────────────────────────────────────────────┐
│             Nonce Reuse Consequences                    │
├───────────────┬─────────────────────────────────────────┤
│ Mode          │ What happens when nonce repeats         │
├───────────────┼─────────────────────────────────────────┤
│ AES-GCM       │ Auth key leaks. Forger can create       │
│               │ valid ciphertexts.                      │
├───────────────┼─────────────────────────────────────────┤
│ AES-GCM-SIV   │ Much more resistant. Nonce reuse only  │
│               │ leaks that two plaintexts are equal.    │
├───────────────┼─────────────────────────────────────────┤
│ ChaCha20-     │ Keystream repeats. XOR of two           │
│ Poly1305      │ ciphertexts reveals XOR of plaintexts.  │
├───────────────┼─────────────────────────────────────────┤
│ AES-CBC       │ IV reuse leaks block-alignment info.    │
│               │ Less catastrophic but still bad.        │
└───────────────┴─────────────────────────────────────────┘
```

**Mitigations**: Use a deterministic nonce scheme (counter + key-id), or use a mode like AES-GCM-SIV that is nonce-misuse resistant. Never generate nonces with a weak PRNG. For high-throughput systems, a monotonically increasing counter per key is safer than random nonces.

### 3. Downgrade Attacks

A downgrade attack tricks the client and server into negotiating a weaker cipher than either would accept on its own. This is how attackers force connections to use export-grade ciphers or SSL 3.0.

```
┌─────────────────────────────────────────────────────────┐
│             Downgrade Attack Flow                       │
│                                                         │
│   Client                Attacker               Server   │
│     │                      │                    │       │
│     │  "I support AES-    │                    │       │
│     │   256 and AES-128"  │                    │       │
│     │────────────────────>│                    │       │
│     │                     │  "I support only  │       │
│     │                     │   AES-128-CBC"    │       │
│     │                     │───────────────────>│       │
│     │                     │                    │       │
│     │  "I support only    │                    │       │
│     │   AES-128-CBC"      │                    │       │
│     │<────────────────────│                    │       │
│     │                     │                    │       │
│     │  Connection uses    │                    │       │
│     │  AES-128-CBC,       │                    │       │
│     │  vulnerable to      │                    │       │
│     │  padding oracle     │                    │       │
│     │─────────────────────────────────────────>│       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Mitigations**: Pin the minimum acceptable cipher version on the server. Disable all legacy protocols (SSL 3.0, TLS 1.0, TLS 1.1). Use TLS 1.3 exclusively, which removes negotiation of weak ciphers entirely. On the client side, validate the negotiated cipher and abort if it is below your threshold.

### 4. Weak Key Generation

Modern encryption is only as strong as the entropy that feeds your key-generation function. If your random-number generator is predictable, the attacker does not need to break AES -- they can just guess your key.

```
┌─────────────────────────────────────────────────────────┐
│             Sources of Weak Key Generation              │
├──────────────────────────┬──────────────────────────────┤
│ Problem                  │ Real-world example           │
├──────────────────────────┼──────────────────────────────┤
│ Low entropy on boot      │ IoT devices that generate    │
│                          │ keys moments after power-on  │
│                          │ before entropy pool gathers  │
├──────────────────────────┼──────────────────────────────┤
│ Predictable RNG seed     │ Containers started with      │
│                          │ same seed from template      │
├──────────────────────────┼──────────────────────────────┤
│ Math.random() in JS      │ V8 used to seed from         │
│                          │ current time (millis)        │
├──────────────────────────┼──────────────────────────────┤
│ VM snapshot reuse        │ Cloned VMs inherit the      │
│                          │ same /dev/random state       │
├──────────────────────────┼──────────────────────────────┤
│ Hardware RNG failure     │ RdRand returns constant      │
│                          │ on faulty CPU                │
└──────────────────────────┴──────────────────────────────┘
```

**Mitigations**: Always use a cryptographically secure PRNG (`/dev/urandom` on Linux, `secrets` on Python, `crypto.randomBytes` in Node.js). Seed your RNG after boot with at least 256 bits of entropy from multiple sources. On cloud VMs, use hardware security modules or dedicated key-generation services.

### 5. KMS Throttling

When you rely on a cloud KMS to unwrap data keys, every decryption operation becomes an API call. If your traffic spikes, KMS rate limits can cascade into full system failure.

```
┌─────────────────────────────────────────────────────────┐
│             KMS Throttling Cascade                      │
│                                                         │
│    Time     Event                                       │
│    ─────    ─────                                       │
│    T+0s     Normal traffic: 100 req/s KMS calls         │
│    T+1s     Traffic spike to 2000 req/s                 │
│    T+2s     KMS returns HTTP 429 for 50% of calls       │
│    T+3s     Application servers queue KMS requests      │
│    T+4s     Queue backs up to database connection pool  │
│    T+5s     Connection pool exhausted                   │
│    T+6s     Database server starts dropping queries     │
│    T+7s     Health checks fail. Load balancer drains    │
│    T+8s     Service is down. All because of KMS limit.  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Mitigations**: Cache data keys in memory with a TTL so you do not call KMS on every decrypt. Pre-warm the cache during deployments. Implement exponential backoff and circuit breakers for KMS calls. Set up monitoring alarms at 50% of your KMS quota.

## Anti-Patterns

The following patterns are common mistakes that you should avoid in your designs. Each one has caused real-world breaches.

### 1. Rolling Your Own Crypto

This is the most famous anti-pattern in security. Writing your own AES implementation, inventing a new cipher, or implementing a standard algorithm from scratch without a deep understanding of side channels almost always produces broken cryptography.

```
┌─────────────────────────────────────────────────────────┐
│             Why Homegrown Crypto Fails                  │
├──────────────────────────┬──────────────────────────────┤
│ Mistake                  │ Consequence                  │
├──────────────────────────┼──────────────────────────────┤
│ No constant-time ops     │ Timing side channel leaks    │
│                          │ the key via cache timing     │
├──────────────────────────┼──────────────────────────────┤
│ Wrong padding scheme     │ Padding oracle attack         │
│                          │ decrypts ciphertext          │
├──────────────────────────┼──────────────────────────────┤
│ Missing authentication   │ Bit-flipping attacks modify  │
│                          │ ciphertext undetected        │
├──────────────────────────┼──────────────────────────────┤
│ Nonce sequence bug       │ Nonce repeats after         │
│                          │ 2^32 encryptions             │
├──────────────────────────┼──────────────────────────────┤
│ Wrong mode combination   │ Using CBC with ecrypt        │
│                          │ then HMAC with wrong order   │
└──────────────────────────┴──────────────────────────────┘
```

**Fix**: Never implement crypto primitives yourself. Use well-audited libraries: OpenSSL, libsodium, Google Tink, AWS Encryption SDK. These libraries handle edge cases, constant-time execution, and memory clearing for you.

### 2. Fixed IV / Static Nonce

Using a hardcoded initialization vector or a counter that resets on restart is equivalent to using no encryption at all for many modes.

```
┌─────────────────────────────────────────────────────────┐
│             Fixed IV Reuse Consequences                 │
│                                                         │
│   Encrypt("hello") with fixed IV                        │
│       -> ciphertext A                                   │
│                                                         │
│   Encrypt("world") with same fixed IV                   │
│       -> ciphertext B                                   │
│                                                         │
│   Attacker XORs A and B -> XOR of "hello" and "world"   │
│       -> Statistical analysis reveals both plaintexts   │
│                                                         │
│   Outcome: confidentiality is destroyed.                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Fix**: Generate a fresh random nonce for every encryption operation. For GCM, use a 12-byte random nonce. For XSalsa20, use the full 24-byte random nonce. If you must use a deterministic scheme, use a keyed counter that persists across restarts.

### 3. ECB Mode

Electronic Codebook (ECB) mode encrypts each 16-byte block independently. Identical plaintext blocks produce identical ciphertext blocks, which leaks structure.

```
┌─────────────────────────────────────────────────────────┐
│             ECB vs. CBC on an Image                     │
│                                                         │
│   Original Image     ECB Encrypted    CBC Encrypted     │
│                                                         │
│   ┌──────────┐      ┌──────────┐     ┌──────────┐      │
│   │          │      │ ██  ██   │     │ ██████   │      │
│   │   ████   │      │ ████████ │     │ ██ ██ ██ │      │
│   │   ████   │  =>  │  ██████  │  => │ ██████ █ │      │
│   │          │      │ ██ ██ ██ │     │ ██ ███ █ │      │
│   └──────────┘      │ ████████ │     │ ████████ │      │
│                      │  ██ ██  │     │  ██ ██  │      │
│                      └──────────┘     └──────────┘      │
│                                                         │
│   ECB preserves      ECB leakage       Properly         │
│   block patterns     reveals content   randomized       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

For image data, ECB encryption still shows the outline of the original image because identical color blocks map to identical ciphertext blocks. This is not theoretical -- the famous "ECB penguin" image demonstrates the problem visually.

**Fix**: Never use ECB mode. Use GCM, CBC with HMAC, or ChaCha20-Poly1305. If you are encrypting database fields, use a deterministic authenticated encryption mode like AES-SIV.

### 4. Hardcoded Keys

Storing encryption keys in source code, configuration files, or environment variables that end up in version control is one of the most common security violations.

```
┌─────────────────────────────────────────────────────────┐
│             Hardcoded Key Failure Path                  │
│                                                         │
│   Developer commits code with secret key:               │
│     secret_key = "my-super-secret-key-12345"            │
│                                                         │
│   Code pushed to public repo                            │
│       -> Key visible to everyone                        │
│                                                         │
│   Automated scanner finds key                           │
│       -> Key is compromised                             │
│                                                         │
│   You rotate the key                                   │
│       -> All existing ciphertext is now unreadable      │
│          unless you re-encrypt                          │
│                                                         │
│   Lesson: the moment a key touches source control,      │
│   consider it compromised.                              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Fix**: Store keys in a secret manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, Azure Key Vault). Inject them at runtime via environment variables or a secrets API. Add a pre-commit hook that scans for potential keys.

### 5. No Key Rotation

Using the same encryption key for years increases the blast radius of a compromise and reduces the amount of data that can safely be encrypted under a single key.

```
┌─────────────────────────────────────────────────────────┐
│             Key Rotation Strategy Comparison            │
├──────────────┬───────────────┬────────────┬─────────────┤
│ Strategy     │ Security      │ Operational│ Data        │
│              │               │ Cost       │ Re-encrypt  │
├──────────────┼───────────────┼────────────┼─────────────┤
│ No rotation  │ Lowest        │ None       │ None        │
│              │ Key lifetime  │            │             │
│              │ = forever     │            │             │
├──────────────┼───────────────┼────────────┼─────────────┤
│ Envelope     │ Good          │ Medium     │ No          │
│ rotation     │ Rotate KEK   │ Rotate KEK │ Only KEK    │
│ (KEK only)   │ every year    │            │ is changed  │
├──────────────┼───────────────┼────────────┼─────────────┤
│ Full         │ Best          │ High       │ Yes, all    │
│ rotation     │ Rotate DEK   │ Re-encrypt │ data must   │
│ (DEK too)    │ every quarter │ all data   │ be re-read  │
└──────────────┴───────────────┴────────────┴─────────────┘
```

**Fix**: Implement envelope encryption so you can rotate the key-encryption key (KEK) without re-encrypting all your data. Automate rotation with a scheduled job. Set a maximum key lifetime of one year for KEKs and consider quarterly rotation for sensitive data.

## When NOT to Use Encryption

Encryption is not always the right tool. There are scenarios where encryption causes more problems than it solves.

### 1. Data That Needs Search or Indexing

Encrypted data is opaque. You cannot search, sort, filter, or aggregate it without decrypting first, which is expensive and defeats caching.

```
┌─────────────────────────────────────────────────────────┐
│             Encryption vs. Database Operations          │
├────────────────────────┬────────────────┬───────────────┤
│ Database Operation     │ Plaintext      │ Encrypted     │
├────────────────────────┼────────────────┼───────────────┤
│ Exact match lookup     │ Fast (index)   │ Impossible    │
│                        │                │ (must scan)   │
├────────────────────────┼────────────────┼───────────────┤
│ Range query (WHERE     │ Fast (B-tree)  │ Impossible    │
│ created_at > X)        │                │               │
├────────────────────────┼────────────────┼───────────────┤
│ Full-text search       │ Fast (inverted │ Impossible    │
│                        │ index)         │               │
├────────────────────────┼────────────────┼───────────────┤
│ ORDER BY / GROUP BY    │ Fast           │ Impossible    │
├────────────────────────┼────────────────┼───────────────┤
│ JOIN on encrypted col  │ Fast           │ Impossible    │
└────────────────────────┴────────────────┴───────────────┘
```

**Solutions if you need both search and encryption**:

- **Tokenization**: Replace sensitive values with a non-sensitive token that maps back to the original in a secure vault. The token preserves format and can be indexed. This is what payment systems use for credit card numbers.

- **Searchable encryption**: Techniques like blind indexes, homomorphic encryption, or deterministic encryption on specific fields. Searchable encryption is slower and functionally limited. For example, deterministic encryption (same plaintext maps to same ciphertext) allows exact-match lookups but leaks frequency patterns.

- **Application-layer filtering**: Decrypt records in the application and apply filters in memory. This works only if you have a small result set to begin with.

- **Column-level access control**: Instead of encrypting a column, restrict SQL access to it with database permissions and monitor access logs.

### 2. High-Frequency Real-Time Data

If you are encrypting every telemetry event, every clickstream record, or every sensor reading, the CPU overhead adds up fast.

```
┌─────────────────────────────────────────────────────────┐
│             Cost of Encryption at Scale                 │
│                                                         │
│   Events per second: 100,000                            │
│   Payload size: 512 bytes                               │
│   Cipher: AES-256-GCM                                    │
│                                                         │
│   Cost per event: ~0.5 microseconds (with AES-NI)       │
│   Total CPU: 50 ms / second (5% of one core)            │
│                                                         │
│   Without AES-NI (e.g. ARM Cortex-A53):                 │
│   Cost per event: ~3.5 microseconds                     │
│   Total CPU: 350 ms / second (35% of one core)          │
│                                                         │
│   At 1 million events / sec with software AES:          │
│   You need 3.5 dedicated cores just for encryption.     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Alternatives**: Encrypt only the sensitive fields (PII, tokens) rather than the entire record. Use a lightweight cipher like ChaCha20 on constrained devices. Consider whether integrity alone (HMAC) is sufficient if the data is not secret but must be tamper-proof.

### 3. Data That Must Be Shared Broadly

Encryption is fundamentally about restricting access. If dozens of services, dozens of teams, and hundreds of engineers need to read a dataset, encryption adds friction without proportional benefit.

Each service needs the decryption key, expanding the attack surface. Each key rotation requires coordinating across teams. Each new consumer needs key-access provisioning.

**Alternative**: Use network-level isolation (VPC, private subnets, service meshes with mTLS) and access control (IAM, RBAC) instead of application-layer encryption. Encrypt only the fields that contain secrets.

### 4. Data With Very Short Lifetimes

If you delete data within seconds or minutes (session tokens, one-time passwords, ephemeral messages), encryption at the application layer adds latency with little security benefit.

The window for an attacker to read plaintext in memory is already short. The real risk is in the persistence layer (logs, database dumps, backups), which you can protect with full-disk encryption at the storage layer instead.

**Alternative**: Store ephemeral data in memory or in systems with built-in TTL (Redis, DynamoDB TTL). Use TLS for transit and filesystem-level encryption for any persistence that survives the data's lifetime.

## Encryption Overhead in Practice

The overhead of encryption is not just CPU cycles. You should also account for storage and network expansion.

### Storage Overhead

Every encryption scheme adds metadata that must be stored alongside the ciphertext.

```
┌─────────────────────────────────────────────────────────┐
│             Storage Overhead per Encryption              │
├──────────────────────────┬──────────────────────────────┤
│ Component                │ Typical Size                 │
├──────────────────────────┼──────────────────────────────┤
│ Nonce / IV               │ 12 bytes (GCM)              │
├──────────────────────────┼──────────────────────────────┤
│ Auth tag                 │ 16 bytes (GCM)              │
├──────────────────────────┼──────────────────────────────┤
│ Key ID                   │ 4-8 bytes (wrapped DEK ref) │
├──────────────────────────┼──────────────────────────────┤
│ Encrypted DEK (envelope) │ 32-256 bytes (RSA-wrapped)  │
├──────────────────────────┼──────────────────────────────┤
│ Padding (CBC)            │ 1-16 bytes per block        │
├──────────────────────────┼──────────────────────────────┤
│ Total per record (GCM)   │ ~32-60 bytes                │
└──────────────────────────┴──────────────────────────────┘
```

For a database with 10 million encrypted records, the overhead alone consumes roughly 300-600 MB of additional storage. This is negligible for most systems, but you should account for it in capacity planning.

### Network Overhead

Encrypted payloads are slightly larger than their plaintext equivalents due to the auth tag and nonce. For TLS, the overhead is per-record (typically 20-40 bytes per TLS record). For application-layer encryption, the overhead is per-encrypted field.

If you encrypt many small fields individually, the overhead ratio is high. A 16-byte credit-card BIN encrypted with AES-256-GCM becomes 44 bytes -- a 2.75x increase. For large payloads (a 1 MB file), the overhead is negligible (0.003%).

**Guideline**: Encrypt larger payloads rather than many small fields. Batch related fields into a single encrypted blob where possible.

## Putting It Together: A Trade-Off Decision Flow

When you design a new system or review an existing one, walk through these questions in order:

1. **What is your threat model?** Who can access the data if it is not encrypted? An attacker on the network? An attacker with database access? A cloud provider employee? The answers determine whether encryption is needed at all.

2. **What data needs encryption?** Not all data has the same sensitivity. Separate PII, secrets, and internal data. Apply different policies to each category.

3. **What encryption mode fits your access pattern?** If you need random access to records, use AES-GCM with per-record nonces. If you need search, consider deterministic encryption or tokenization. If you need streaming, use ChaCha20.

4. **How do you manage keys?** Use envelope encryption with a cloud KMS. Cache data keys with a short TTL. Automate rotation. Back up master keys across regions.

5. **What happens when something fails?** Plan for key loss, KMS throttling, nonce counter exhaustion, and downgrade attacks. Each failure mode should have a documented mitigation.

6. **How do you test your encryption?** Write tests that verify encryption and decryption round-trips. Test with corrupted ciphertexts. Test with expired keys. Test KMS outage scenarios. Run a chaos experiment where you simulate a key-rotation failure.

## Bridge to Real-World Systems

You now understand the trade-offs, failure modes, and anti-patterns of encryption. In real-world systems, you rarely start from scratch. You work with existing encryption services and infrastructure:

- **AWS** provides KMS for key management, ACM for TLS certificates, and S3 server-side encryption with customer-managed keys (SSE-KMS) or client-side encryption with the AWS Encryption SDK.
- **GCP** provides Cloud KMS, Cloud HSM for hardware-backed keys, and Cloud Storage with customer-managed encryption keys (CMEK).
- **Azure** provides Key Vault for key and secret management, and Storage Service Encryption with customer-managed keys.
- **Open-source** stacks use HashiCorp Vault for secrets management, cert-manager for TLS certificate lifecycle, and SOPS or sealed-secrets for encrypted configuration files.

Each of these services implements the principles you learned here: envelope encryption, key hierarchy, automated rotation, and audit logging. Your job is not to reimplement them, but to compose them into a system that meets your security requirements without sacrificing performance, availability, or operational sanity.

The next page in this series explores how each major cloud provider implements these patterns, so you can make an informed choice for your infrastructure.
