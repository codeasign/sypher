## The Problem Story

A messaging startup called SafeTalk launches to 500,000 users. The founding team of four engineers builds a real-time chat engine with WebSocket connections, push notifications, typing indicators, and read receipts. They use PostgreSQL for message history and Redis for pub-sub delivery. Everything works. The team celebrates the metrics, raises a Series A, and hires aggressively.

On day 90, a security researcher downloads the mobile app and opens the network tab. The researcher sends a message and watches the WebSocket frame вҖ” the message body is plaintext JSON flying across the wire. The researcher inspects the app's local storage and finds every message stored as plaintext in an unencrypted SQLite database. The researcher opens the Android APK, extracts the API base URL, and sends a `GET /api/messages?user_id=382` request. The server responds with 2,400 messages belonging to User 382 вҖ” content, timestamps, attachment URLs, and recipient IDs. All in plaintext. The researcher changes the `user_id` parameter to `383`, then `384`, then `8472`. Every request returns that user's complete message history. The server has no access control on the message history endpoint. The database stores every message as a plaintext string in a `body` column. The entire conversation history of half a million users is accessible to anyone who knows the API endpoint exists.

The researcher sends a responsible disclosure email on day 91. The CTO reads it at 6:47 AM and opens the database console: `SELECT COUNT(*) FROM messages`. Twelve million messages. Every single one readable. The CTO opens the server logs вҖ” the researcher accessed 3,847 user accounts' message histories over 90 minutes. The data was never encrypted. Not in transit. Not at rest. Not anywhere.

The database contains messages like:

"Hey, can you send me the Q4 financial projections?"

"Here is my social security number: 987-65-4321."

"I think I have a brain tumor. I'm scared."

"The password to the company vault is J@nuary2024!"

Every one of those messages was a single `SELECT` statement away from any attacker who found the API endpoint. The company spends the next six months retrofitting encryption: adding TLS to all endpoints, implementing AES-256 column encryption, building a key management system, and running a database migration to re-encrypt twelve million messages. They lose 4,000 users during the six-hour migration. The CTO writes a postmortem titled "How We Spent Nine Months Building a Product That Leaked Every User's Private Data." The most upvoted comment reads: "You should have encrypted from day one."

This is the nightmare that encryption prevents. Not a hypothetical future breach. Not a compliance checkbox. The data you store today will be breached tomorrow. The only question is whether the breach reveals ciphertext or plaintext. Encryption is the difference between a headline that says "Database Exposed, Millions of Records Stolen" and a headline that says "Database Exposed, Data Encrypted, No User Impact."

## The Insight

**Encryption** is the practice of transforming readable data (plaintext) into an unreadable format (ciphertext) using a mathematical algorithm and a secret key, such that only someone with the corresponding key can reverse the transformation.

Think of a locked metal box with a letter inside. You write a message on a piece of paper. You fold the paper. You place it into the box. You close the lid. You snap a padlock through the hasp. You hand the box to a courier who carries it across the city. The courier cannot read the letter вҖ” the box is sealed. A thief intercepts the courier and steals the box. The thief cannot read the letter вҖ” the box is locked. The thief takes the box to a workshop and spends a week trying to break the lock. After seven days of effort, the lock snaps open. The thief reads the letter. The thief has broken the encryption вҖ” but the thief had to work for it. The lock did not stop the determined thief, but it stopped every casual observer, every opportunistic passerby, and every automated data-scraping script that walked past the open box.

The mapping is precise. The letter is the plaintext вҖ” the original readable message. The padlock is the encryption algorithm вҖ” AES, ChaCha20, RSA. The key that opens the padlock is the cryptographic key вҖ” a secret value known only to the sender and the authorized recipient. The locked box is the ciphertext вҖ” the encrypted message that travels across the network or sits in the database. The courier is the network вҖ” the internet, a private connection, or a data center link. The thief is the attacker вҖ” a database breach, a network tap, a man-in-the-middle, or a malicious insider. The hammer is the brute-force attack вҖ” trying every possible key until the box opens. The entire security of the system depends on the strength of the lock and the secrecy of the key. A weak lock (a broken algorithm like DES) can be opened by any thief with a toothpick. A strong lock (AES-256) would take the world's fastest supercomputer billions of years to pick.

## Core Vocabulary

**Encryption** вҖ” The process of transforming plaintext into ciphertext using an algorithm and a key. Encryption provides confidentiality: data that is encrypted cannot be read by anyone who does not possess the key. The security of encryption depends entirely on the secrecy of the key, not the secrecy of the algorithm. Modern encryption algorithms are published, peer-reviewed, and standardized. Security through obscurity (keeping the algorithm secret) is not security.

**Decryption** вҖ” The inverse of encryption. The process of transforming ciphertext back into plaintext using the corresponding algorithm and key. Decryption without the correct key is computationally infeasible for strong algorithms. Attempting decryption with the wrong key produces garbage output, not an error вҖ” a well-designed encryption scheme makes it impossible to distinguish garbage from a valid message without the correct key.

**Plaintext** вҖ” The original, readable data before encryption. Plaintext is what you start with: a message, a file, a password, a database row. Plaintext is the thing you must protect. If plaintext is exposed, the encryption was either not applied, was misconfigured, or was broken. More data breaches result from plaintext exposure than from broken encryption algorithms.

**Ciphertext** вҖ” The unreadable output of encryption. Ciphertext is designed to be indistinguishable from random noise to anyone who does not possess the decryption key. If ciphertext reveals patterns, structure, or statistical correlations with the plaintext, the encryption scheme is weak. In a strong encryption scheme, there is no statistical test that can distinguish ciphertext from truly random data.

**Symmetric Encryption** вҖ” An encryption scheme where the same key is used for both encryption and decryption. Symmetric encryption is fast вҖ” AES-256 with hardware acceleration (AES-NI) can encrypt data at over 1 GB/s per core. The fundamental challenge is key distribution: how do Alice and Bob agree on a shared secret without an attacker intercepting it? If Alice and Bob already have a secure channel to exchange the key, they do not need the encryption. Symmetric encryption is used for bulk data encryption: files, database records, and network traffic payloads.

**Asymmetric Encryption** вҖ” An encryption scheme where two mathematically related keys exist: a public key used for encryption and a private key used for decryption. Asymmetric encryption is slow вҖ” RSA-2048 encryption achieves approximately 5 MB/s, roughly 200 times slower than AES-256. The critical advantage is that the public key can be shared freely. Anyone can encrypt a message using your public key, but only you can decrypt it using your private key. Asymmetric encryption is used for key exchange, digital signatures, and encrypting small payloads like session keys.

**Public Key** вҖ” The key in an asymmetric cryptosystem that is shared openly. The public key encrypts data. It cannot decrypt data. Publishing your public key is safe вҖ” it is like giving everyone a copy of your mailbox's slot. Anyone can drop a letter in, but only you can open the mailbox. Your public key does not need to be kept secret, but it must be authenticated вҖ” a recipient must be able to verify that a given public key actually belongs to you.

**Private Key** вҖ” The key in an asymmetric cryptosystem that is kept secret. The private key decrypts data. It must never be shared, transmitted over an insecure channel, or stored in an unencrypted location. If your private key is compromised, every message encrypted with your public key is readable. Your private key is the master key to your digital identity. Losing it is a security emergency.

**AES (Advanced Encryption Standard)** вҖ” A symmetric block cipher standardized by NIST in 2001. AES operates on 128-bit blocks with key sizes of 128, 192, or 256 bits. AES-256 is the gold standard for symmetric encryption. It is used by the US government for TOP SECRET data (NSA Suite B), by TLS for secure web connections, by disk encryption utilities, and by virtually every encrypted storage system. AES with hardware acceleration (AES-NI) achieves throughput of over 1 GB/s. No practical attack exists against AES-256 вҖ” the best known attack reduces the effective key size from 256 bits to approximately 254 bits, which is still computationally infeasible.

**RSA (Rivest-Shamir-Adleman)** вҖ” An asymmetric encryption algorithm based on the difficulty of factoring the product of two large prime numbers. RSA key sizes are typically 2048 or 4096 bits. RSA-2048 provides approximately 112 bits of security. RSA is used for encrypting shared keys in TLS handshakes, for digital signatures, and for secure email (PGP/GPG). RSA is being gradually replaced by ECC in modern systems because ECC provides equivalent security with smaller keys and faster operation. RSA-2048 is still widely deployed but should not be used for new designs.

**ECC (Elliptic Curve Cryptography)** вҖ” An asymmetric cryptosystem based on the algebraic structure of elliptic curves over finite fields. ECC provides equivalent security to RSA with much smaller key sizes: a 256-bit ECC key provides security comparable to a 3072-bit RSA key. ECC is faster than RSA at equivalent security levels and is the foundation of modern key exchange (ECDH, ECDHE) and digital signatures (ECDSA, EdDSA). Curve25519 (X25519 for key exchange, Ed25519 for signatures) is the recommended curve for new systems.

**TLS (Transport Layer Security)** вҖ” The cryptographic protocol that secures communication over the internet. TLS uses a hybrid approach: asymmetric encryption (ECDHE or RSA) for key exchange and server authentication, then symmetric encryption (AES-GCM or ChaCha20-Poly1305) for bulk data encryption. TLS is what makes HTTPS secure. Every time you see a padlock icon in a browser, TLS is protecting the connection. TLS 1.3 is the current standard вҖ” it reduces handshake latency to one round trip, removes insecure legacy options, and mandates forward secrecy.

**Key Exchange** вҖ” The process by which two parties agree on a shared secret key over an insecure channel without revealing that key to an eavesdropper. The most widely used key exchange protocol is Diffie-Hellman (DH) and its elliptic curve variant ECDH. Key exchange solves the fundamental bootstrapping problem of symmetric encryption: two parties who have never met can agree on a shared secret in plain sight. Key exchange is the foundation of TLS, the Signal Protocol, and virtually every secure communication system.

**Digital Signature** вҖ” A cryptographic mechanism that proves the authenticity, integrity, and non-repudiation of a message. The sender signs a message with their private key. The receiver verifies the signature with the sender's public key. A valid signature proves three things: the message was created by the holder of the private key (authenticity), the message has not been modified since signing (integrity), and the sender cannot later deny having signed it (non-repudiation). Digital signatures are not encryption вҖ” they prove who sent a message but do not hide the message content.

**Hash** вҖ” A one-way mathematical function that maps data of arbitrary size to a fixed-size output (digest). Hashing is not encryption вҖ” encryption is reversible with the key, while hashing is irreversible by design. Cryptographic hash functions (SHA-256, SHA-3, BLAKE2) have three essential properties: collision resistance (no two inputs produce the same output), preimage resistance (given an output, it is infeasible to find any input that produces it), and the avalanche effect (changing one bit of the input changes approximately 50% of the output bits). Hashes are used for password storage, data integrity verification, and as building blocks in encryption schemes (key derivation, HMAC, authenticated encryption).

### Terminology in Practice

These terms appear in every encryption design conversation. Here is how they show up in real engineering discussions:

"We encrypt all data at rest with AES-256-GCM. Each tenant gets a unique data encryption key. The DEK is wrapped with a key encryption key stored in a hardware security module. The KEK never leaves the HSM. An attacker who breaches the database gets encrypted ciphertext, not plaintext. An attacker who breaches the application server still cannot decrypt because the KEK lives in the HSM and access to the HSM requires a signed client certificate."

"The TLS handshake uses ECDHE for key exchange with X25519 and AES-256-GCM for the bulk cipher. The server presents its certificate, which contains its RSA-2048 public key. The client verifies the certificate against a trusted Certificate Authority. The ephemeral ECDHE key exchange ensures forward secrecy вҖ” if the server's long-term private key is compromised tomorrow, today's session keys remain safe."

"We use RSA-OAEP for encrypting small payloads: credit card numbers, API tokens, and symmetric keys. OAEP padding prevents the deterministic encryption weakness of raw RSA. The ciphertext is 256 bytes for a 2048-bit key. The maximum plaintext size is 190 bytes because OAEP overhead consumes 42 bytes of the 256-byte capacity."

"ECC is the right choice for IoT devices. A 256-bit ECC key fits in a single 32-byte register. Key agreement and signature verification complete in under 10 milliseconds on a microcontroller running at 80 MHz. RSA-2048 would take 200 milliseconds on the same hardware and consume 10x the memory for key storage."

## The Big Picture

Encryption in a modern system operates at three distinct layers: symmetric versus asymmetric, the TLS handshake, and encryption at rest versus encryption in transit. Each layer protects against different threats. Understanding where and how to apply each layer is the core skill of encryption system design.

### Symmetric vs Asymmetric Encryption

The two fundamental categories of encryption differ in key structure, performance, and use cases. Symmetric encryption uses one key shared between sender and receiver. Asymmetric encryption uses a key pair: a public key for encryption and a private key for decryption. Production systems use both: asymmetric encryption to exchange a symmetric session key, then symmetric encryption for the actual data.

### Encryption at Rest vs Encryption in Transit

These are two different problems with different threat models. A complete security strategy requires both.

**Encryption in transit** protects data as it moves across the network вҖ” from the browser to the server, from the server to the database, from one microservice to another. If an attacker taps the network cable, compromises a router, or performs a man-in-the-middle attack, the encrypted data is unreadable. TLS is the standard solution for encryption in transit. The threat model is network-level: packet sniffing, DNS spoofing, BGP hijacking, Wi-Fi eavesdropping.

**Encryption at rest** protects data stored on disk вҖ” in a database, in a file system, in an S3 bucket, on a laptop. If an attacker steals the hard drive, compromises a database backup, or gains shell access to the server, the encrypted data is unreadable. AES-256 is the standard solution for encryption at rest. The threat model is storage-level: theft of media, backup compromise, OS-level access, cloud provider access.

### How the Three Pillars Work Together

In a production system, all three layers are active simultaneously. When a user sends a message through your application:

1. The message travels from the user's device to your server over TLS 1.3 (encryption in transit using symmetric + asymmetric hybrid).
2. The server terminates the TLS connection, decrypting the message using the TLS session key (symmetric decryption).
3. The server re-encrypts the message with AES-256-GCM using a data encryption key (DEK) managed by a key management system (encryption at rest with envelope encryption).
4. The encrypted ciphertext is written to the database.
5. The backup process copies the encrypted database to S3, which applies an additional layer of encryption with an S3-managed key (double encryption at rest).
6. If a malicious actor steals the backup file, they must break both the AES-256 data encryption and the S3 server-side encryption to read a single byte.

This is defense in depth. No single encryption layer is assumed to be impenetrable. If one layer is compromised, the other layers remain intact. The attacker must break through multiple independent encryption barriers to reach the plaintext.

## Step by Step: The Complete Encryption Lifecycle

Encryption is not a single operation. It is a sequence of coordinated steps involving key generation, key distribution, data encryption, transmission, decryption, integrity verification, and key rotation. Here is the exact flow, from the moment a user sends a message to the moment the recipient reads it.

### Step 1: Generate Cryptographic Keys

The system generates a key pair for each user. In a typical messaging application, this happens when a user registers for the first time. The server generates an ECC P-256 or X25519 key pair. The private key is encrypted with a passphrase derived from the user's password and stored in a secure keystore on the user's device. The public key is published to a key directory that other users can query.

Key generation must use a cryptographically secure random number generator. The Debian OpenSSL vulnerability of 2008 is a cautionary tale: a single commented-out line of code reduced the entropy pool, causing the RNG to produce only approximately 32,000 possible keys. Every SSL key generated on Debian systems between 2006 and 2008 was one of 32,000 values. An attacker could try all 32,000 and find the private key for any affected server.

### Step 2: Retrieve the Recipient's Public Key

Alice wants to send Bob an encrypted message. Alice's client queries the key directory for Bob's public key. The key directory returns Bob's public key and a certificate or signature that binds the key to Bob's identity. Alice's client verifies this binding.

This key retrieval step is the most vulnerable point in the entire encryption lifecycle. An attacker who can intercept the key directory response and replace Bob's public key with their own can perform a man-in-the-middle attack. The attacker's public key is sent to Alice, Alice encrypts the message for the attacker, the attacker decrypts, reads, re-encrypts with Bob's real key, and forwards to Bob. Neither Alice nor Bob detects the interception. This is why public key infrastructure (PKI) and certificate authorities exist вҖ” they provide a trusted mechanism for verifying public key ownership.

The verification mechanism depends on the threat model:

- **Web (TLS):** Certificate Authorities sign a certificate that binds a domain name to a public key. The browser trusts ~150 CAs. If any CA is compromised, all TLS connections are vulnerable.
- **Messaging (Signal):** Users verify each other's public keys by comparing fingerprints out of band вҖ” scanning a QR code, reading a 12-word phrase aloud, or checking a safety number. No centralized trust.
- **Enterprise (mTLS):** An internal CA issues certificates to services. Each service presents its certificate when connecting to another service. The receiving service verifies the certificate against the CA.

### Step 3: Compose the Plaintext

Alice composes her message. The message enters the application as plaintext in the input field. Alice clicks Send. The application prepares the plaintext for encryption.

The plaintext must never be written to disk, logged, or sent to any analytics service before encryption. A common and catastrophic mistake is logging the raw message body for debugging: `logger.debug("Message from Alice to Bob: {}", messageBody)`. The log file is shipped to a centralized logging service, stored in an S3 bucket, and retained for 90 days. The encryption is bypassed by a single debug log statement. Production systems must audit every code path that touches plaintext to ensure the plaintext flows only to the encryption function and nowhere else.

### Step 4: Generate a Symmetric Session Key and Encrypt

Alice's client generates a random symmetric key вҖ” an AES-256 key (32 bytes of random data from a CSPRNG). This is the session key, used only for this single message. The client encrypts the plaintext with AES-256-GCM using this session key. The encryption produces three outputs: the ciphertext (same length as the plaintext), the authentication tag (16 bytes), and the initialization vector (IV, 12 bytes). The IV is a random nonce that ensures encrypting the same plaintext twice produces different ciphertext.

GCM provides both confidentiality (the ciphertext hides the message) and integrity (the authentication tag detects any modification). These two properties are combined in a single operation вҖ” a significant improvement over older crypto constructions that required separate encryption (AES-CBC) and authentication (HMAC-SHA256).

### Step 5: Wrap the Session Key with Asymmetric Encryption

Alice's client encrypts the AES session key with Bob's public key using RSA-OAEP or ECDH (depending on the protocol). This is the hybrid encryption step. The message is encrypted with a fast symmetric cipher, and the symmetric key is encrypted with a slow asymmetric cipher that only Bob can decrypt.

The RSA-encrypted session key is 256 bytes for a 2048-bit RSA key. The total encrypted payload across the wire is:

```
ciphertext (variable) + IV (12 bytes) + auth tag (16 bytes)
+ encrypted session key (256 bytes) + key identifier (4 bytes)
```

For a 100-byte plaintext message, the encrypted payload is approximately 388 bytes вҖ” about 4x the original size. The asymmetric overhead is constant regardless of plaintext size, so the overhead percentage decreases with larger messages.

### Step 6: Transmit the Encrypted Payload

Alice's client sends the encrypted payload to the server. The payload includes the ciphertext, the IV, the authentication tag, and the encrypted session key. The server receives the payload and stores it in the database. The server cannot decrypt the payload вҖ” it does not have Bob's private key. The server is a dumb storage relay.

This is the critical architectural point of end-to-end encryption (E2EE): the server never holds the decryption keys for user messages. The server is an untrusted relay. Even if the server is fully compromised, the attacker gains access to ciphertext вҖ” not plaintext. The attacker cannot read any message because the private keys live on the users' devices, not on the server.

### Step 7: Decrypt the Payload on the Recipient's Device

Bob's device receives the encrypted payload from the server. Bob's device retrieves Bob's private key from the secure keystore. The device decrypts the encrypted session key using Bob's private key (RSA decryption or ECDH key agreement), recovering the original AES-256 session key. The device then decrypts the ciphertext using AES-256-GCM with the session key and the IV. The device verifies the authentication tag вҖ” if the tag is valid, the plaintext is released to the application. If the tag is invalid, the message is discarded and the user sees an error. The ciphertext was modified in transit.

The decryption latency is dominated by the asymmetric step. RSA-2048 decryption (which uses the private key) takes approximately 1-3 milliseconds. ECDH key agreement takes under 1 millisecond. AES-256-GCM decryption takes under 1 microsecond per kilobyte. For a typical message, total decryption time is under 5 milliseconds.

### Step 8: Ratchet the Keys Forward

After the message is decrypted and read, modern encryption systems move to a new key. This is the ratchet mechanism вҖ” each message advances the key state forward. If a key is compromised, the attacker can decrypt only the single message that used that key.

The Signal Protocol (used by WhatsApp, Signal, and Google Messages) implements a Double Ratchet:

- **Symmetric ratchet:** Each message derives a new key by hashing the previous key. This provides forward secrecy: if the current key is compromised, the attacker can derive future keys (by hashing forward) but cannot derive past keys (hashing is one-way).
- **Asymmetric ratchet (DH ratchet):** After each round-trip of messages, a new Diffie-Hellman key exchange is performed. This provides future secrecy: if the current key is compromised, the attacker cannot derive future keys because the next DH exchange produces a new shared secret that the attacker cannot compute.

### Step 9: Verify Integrity and Authenticate

Before the decrypted plaintext is displayed to Bob, the system verifies the authentication tag from the GCM decryption. The tag is a cryptographic checksum over the ciphertext and the associated data (AAD). During decryption, the tag is recomputed and compared against the attached tag. If the tags do not match, the ciphertext was modified вҖ” either by a network error (bit flip) or by an attacker (bit flip attack).

Integrity verification is mandatory. Without it, an attacker can modify ciphertext in transit and cause the decryption to produce a different plaintext. AES-GCM and ChaCha20-Poly1305 are authenticated encryption modes that prevent malleability. AES-CBC is not вҖ” it provides confidentiality without integrity and requires a separate HMAC.

### Step 10: Display the Plaintext and Clean Up

Bob's application displays the decrypted plaintext. The application ensures the plaintext is displayed but never persisted to disk in unencrypted form. The message is held in memory for the duration of the viewing session and discarded when the view is dismissed. The encrypted ciphertext remains in the database.

The cleanup step is often overlooked. A mobile app that displays a decrypted message and then caches the plaintext in a local SQLite database has defeated the encryption. A thief who steals the phone can open the local database and read every message. Encrypted messaging apps must use encrypted local storage: SQLCipher for local databases, iOS Keychain for secrets, or Android EncryptedSharedPreferences.

The entire lifecycle вҖ” from key generation through decryption and cleanup вҖ” repeats for every message. Each message gets its own session key, its own IV, and its own authentication tag. This per-message encryption ensures that compromising a single message does not compromise any other message.

## Variants and Flavors

Different use cases demand different encryption variants. Here are the four most important constructions for system design interviews and production architecture.

### AES-GCM (Galois/Counter Mode)

AES-GCM is the most widely deployed authenticated encryption mode in the world. It is the default cipher suite for TLS 1.2 and 1.3, the standard for database column encryption, and the recommended choice for most application-level encryption needs.

AES-GCM combines AES in counter mode (CTR) for encryption with Galois message authentication code (GMAC) for integrity. Counter mode encrypts blocks independently, making GCM parallelizable and fast on multi-core CPUs. GMAC uses polynomial multiplication in GF(2^128) вҖ” a mathematical structure that is efficient in hardware but slower in software.

**Parameters and constraints:**

- Key size: 128, 192, or 256 bits. Use 256 bits.
- IV (nonce) size: 12 bytes (recommended by NIST). The IV must be unique per (key, message) pair.
- Tag size: 16 bytes (standard). Never truncate below 8 bytes.
- Maximum plaintext per (key, IV) pair: 64 GB (2^36 bytes). After this, the 32-bit counter wraps.
- **Critical constraint:** Never reuse the IV with the same key. If the IV repeats, the keystream is the same, and XORing two ciphertexts reveals the XOR of the two plaintexts.

### ChaCha20-Poly1305

ChaCha20-Poly1305 is an authenticated encryption construction designed by Daniel J. Bernstein. ChaCha20 is a stream cipher (not a block cipher), and Poly1305 is a polynomial MAC. The combination is a mandatory cipher suite in TLS 1.3 and the only encryption in the WireGuard VPN protocol.

ChaCha20 is optimized for software implementation. Unlike AES, which relies on hardware AES-NI instructions, ChaCha20 runs fast on CPUs without hardware acceleration вҖ” ARM processors, mobile devices, microcontrollers, and older x86 CPUs. ChaCha20 is approximately 3x faster than AES without hardware acceleration. Poly1305 uses simple integer arithmetic (multiplication modulo 2^130 - 5), which is fast in software.

**Parameters and constraints:**

- Key size: 256 bits.
- Nonce size: 12 bytes (IETF variant used in TLS 1.3). Some variants use 8 or 24 bytes.
- Tag size: 16 bytes.
- Maximum plaintext per (key, nonce) pair: effectively unlimited (ChaCha20 supports 2^70 bytes of keystream).
- No ciphertext expansion: ChaCha20 is a stream cipher, so ciphertext length equals plaintext length.

### RSA-OAEP (Optimal Asymmetric Encryption Padding)

RSA raw encryption is deterministic: the same plaintext encrypted with the same key always produces the same ciphertext. Deterministic encryption allows an attacker to build a dictionary of plaintext-ciphertext pairs. OAEP padding solves this by adding random bytes before encryption, ensuring that each encryption of the same plaintext produces a different ciphertext.

OAEP works through a two-round Feistel network using hash functions. The plaintext is padded, XORed with a hash of a random seed, and the seed is XORed with a hash of the padded plaintext. The 256-byte result is encrypted with raw RSA.

### ECDH (Elliptic Curve Diffie-Hellman)

ECDH is the key exchange protocol that enables two parties to agree on a shared secret over an insecure channel. Each party generates an ECC key pair, exchanges public keys, and computes the same shared secret independently.

The security of ECDH rests on the Elliptic Curve Discrete Logarithm Problem (ECDLP): given a generator point G on the curve and a point P = d * G, find the scalar d. For a 256-bit curve (Curve25519 or P-256), the best known attack requires approximately 2^128 operations вҖ” equivalent to breaking AES-128.

## Mental Model Check

Test your understanding of encryption concepts. Work through each question thoroughly before reading the answer.

**Question 1:** A messaging application encrypts messages with AES-256-GCM before sending them to the server. The encryption keys are derived from the user's password, which the server stores as a bcrypt hash. The server stores the encrypted messages alongside the bcrypt hash. An attacker breaches the server and obtains the database. Can the attacker read the messages?

**Answer:** Yes, the attacker can read the messages. The encryption is broken because the server holds both the ciphertext and the means to recover the keys. The attacker extracts a user's bcrypt password hash, cracks the password offline (bcrypt is slow but not impossible for weak passwords), derives the AES encryption key from the password, and decrypts all messages. The fundamental flaw is that the server stores the password-derived key material. True end-to-end encryption requires that encryption keys are generated and stored on the user's device, never on the server. The server should be an untrusted relay that holds only ciphertext. This is the difference between "encrypted at rest" (server-side encryption, vulnerable to server compromise) and "end-to-end encryption" (client-side encryption, resistant to server compromise).

**Question 2:** A system uses AES-256-CBC for encrypting database records. A security review finds that an attacker can modify the ciphertext in such a way that the decrypted plaintext contains a different dollar amount than the original вҖ” changing $100 to $10,000. What is the vulnerability and how do you fix it?

**Answer:** The vulnerability is that AES-CBC is malleable вҖ” it provides confidentiality but not integrity. In CBC mode, flipping a bit in ciphertext block N causes the corresponding bit in plaintext block N+1 to flip in the same position. This is a bit-flipping attack. The attacker does not need to know the key to perform this attack вҖ” the decryption produces a valid plaintext with the attacker's chosen bits. The fix is to use an authenticated encryption mode such as AES-GCM or ChaCha20-Poly1305, which includes an authentication tag that detects any modification of the ciphertext. If the tag does not match during decryption, the operation fails and an error is returned. Alternatively, use encrypt-then-MAC: encrypt with AES-CBC, then compute HMAC-SHA256 over the ciphertext and verify the HMAC before decryption. Never use MAC-then-encrypt (which was the vulnerability in TLS 1.0's CBC mode).

**Question 3:** A TLS 1.2 handshake uses RSA key exchange (not ECDHE). One year later, the server's RSA private key is compromised through a vulnerability in a backup system. An attacker recorded all TLS traffic to this server over the past year. Can the attacker decrypt past sessions? Why? What is the fix?

**Answer:** Yes, the attacker can decrypt all recorded past sessions. The vulnerability is the lack of forward secrecy. In RSA key exchange, the client generates a random pre-master secret, encrypts it with the server's RSA public key, and sends it in the ClientKeyExchange message. The server decrypts with its RSA private key. Both parties derive all session keys from this pre-master secret. An attacker who records the encrypted handshake and later obtains the RSA private key can: (1) extract the encrypted pre-master secret from the recorded ClientKeyExchange message, (2) decrypt it with the now-compromised private key, (3) derive the session keys, and (4) decrypt the entire recorded session. Every past session is compromised. The fix is to use ECDHE (Ephemeral Elliptic Curve Diffie-Hellman) key exchange. In ECDHE, the server generates a fresh ECDH key pair for each session. The ephemeral private key lives only in server memory for the duration of the handshake and is discarded. An attacker who compromises the long-term RSA key one year later cannot decrypt past sessions because the ephemeral private keys were never stored. TLS 1.3 mandates ECDHE and removes RSA key exchange entirely.

**Question 4:** You are designing an encrypted file storage service. Users upload files up to 10 GB. The service must allow users to encrypt files before upload, decrypt after download, support parallel encryption, and allow the server to verify file integrity without having the encryption key. What encryption strategy do you choose?

**Answer:** Use AES-256-GCM with file chunking and a Merkle tree. Split each file into 1 MB chunks. Encrypt each chunk independently with AES-256-GCM using a unique IV per chunk. Store the ciphertext, IV, and authentication tag for each chunk. The chunking approach enables four critical properties: (1) parallel encryption вҖ” each chunk is independent and can be encrypted on separate CPU cores; (2) random access вҖ” the user can download and decrypt chunk 47 without downloading the entire 10 GB file; (3) partial retry вҖ” if chunk 147 fails to decrypt, the user re-uploads or re-downloads only that single chunk; (4) incremental verification вҖ” each chunk's authentication tag can be verified independently. For server-side integrity verification without the encryption key, build a Merkle hash tree over the ciphertext chunks: compute SHA-256 of each encrypted chunk, pair the hashes recursively into a binary tree, and store the 32-byte root hash. The server can verify file integrity by recomputing the Merkle tree from stored ciphertext chunks and comparing the root вҖ” without ever having access to the encryption key. The root hash is signed with the user's private key to prevent tampering. This is the strategy used by encrypted cloud storage services like Tresorit and Cryptomator.

**Question 5:** A company uses a single AES-256 key (KEK) to encrypt all customer database records. The KEK is stored in a configuration file at `/etc/app/key.conf` with permissions `0644` (readable by all users on the server). A developer runs a script that accidentally prints the key to stdout. The stdout is captured by the application logging framework and shipped to a centralized logging service with 90-day retention. The key is now exposed to every employee with logging access вҖ” approximately 200 people. What is the damage and how do you redesign the system to prevent this?

**Answer:** The damage is catastrophic. A single key encrypts every customer record. With the key exposed to 200 people, every customer's personal information, financial data, and private messages can be decrypted. The damage is not limited to current data вҖ” the same key continues to be used for future data, so the attacker can decrypt new data as it is written. The fix requires three architectural changes:

(1) **Key hierarchy with HSM-backed KEK.** Introduce a two-layer hierarchy. A master key (Key Encryption Key, KEK) is stored in a hardware security module (HSM) or cloud KMS. Individual Data Encryption Keys (DEKs) are generated per customer or per record and encrypted with the KEK. The KEK never leaves the HSM. The encrypted DEKs are stored alongside the ciphertext. An attacker who breaches the application server finds encrypted DEKs, not the KEK. To decrypt, the attacker must call the HSM's decrypt API, which requires authentication (client certificate, IAM role, or short-lived token) and authorization (key usage policy).

(2) **Key rotation with limited blast radius.** Each DEK has a lifetime of 30 days or 1 million encryptions. When a DEK expires, the data is re-encrypted with a new DEK. On read, the system tries the current DEK first; if decryption fails, it tries the previous DEK (graceful rotation). After 60 days, old DEKs are permanently deleted. A key exposure at time T compromises data encrypted with keys active within the last 30 days, not all data from the beginning of time.

(3) **Zero-trust key distribution.** The KEK is never written to a file. It is created inside the HSM and stays there. The application authenticates to the HSM and requests encryption/decryption operations. The HSM returns results, never the key. The application never has the raw KEK in memory. Access to the HSM requires short-lived tokens (30-minute TTL) obtained from an identity provider. Logs are scrubbed: any data that looks like it might be a cryptographic key is redacted by the logging pipeline before storage.

The result: even if a developer accidentally prints a variable to stdout, the variable contains only an encrypted DEK or a handle to the HSM вҖ” never the raw key material. The logging service stores ciphertext. The 200 employees with log access see encrypted data they cannot decrypt.
