## Internal Architecture

Every authorization system you will encounter in production — whether it is OAuth2 scopes, AWS IAM policies, or a custom RBAC module — shares a common internal architecture. Understanding this architecture lets you reason about any system you meet.

The core components are the Policy Enforcement Point (PEP), the Policy Decision Point (PDP), the Policy Information Point (PIP), and the Policy Administration Point (PAP). These four roles form the backbone of all authorization decisions.

### PEP — The Gatekeeper

When a request arrives, the PEP is the first component to touch it. The PEP extracts the subject (who is making the request), the action (what they want to do), and the resource (what they want to do it to). It then packages these into an authorization request and sends it to the PDP.

The PEP never makes a decision itself. It is purely a relay and enforcement layer. This separation is critical: you can swap out the PDP without touching the PEP, and vice versa.

### PDP — The Decision Engine

The PDP is the brain of the system. It receives the authorization request and evaluates it against the policies stored in the system. The PDP:

1. Builds an evaluation context from the request — subject, resource, action, environment
2. Resolves applicable policies — which policies match this request
3. Evaluates each policy — runs the rule engine
4. Combines results — applies the combining algorithm (deny-overrides, permit-overrides, first-applicable, etc.)
5. Returns a decision — Permit, Deny, NotApplicable, or Indeterminate

### PIP — The Information Source

The PIP resolves attributes that the PDP needs to make a decision. These attributes come from multiple sources:

- **Subject attributes**: user ID, groups, department, security clearance, location
- **Resource attributes**: resource type, owner, classification, tags
- **Action attributes**: HTTP method, API endpoint, operation name
- **Environment attributes**: time of day, IP address, device type, threat level

The PIP caches resolved attributes aggressively because attribute resolution is often the slowest part of authorization.

### PAP — The Management Interface

The PAP is where administrators define and manage policies. It handles create, read, update, and delete operations on policies, roles, and permissions. The PAP also manages policy versioning, audit trails, and rollout.

Permission inheritance follows a simple rule: a role at any level in the tree inherits all permissions from every ancestor role. If an Editor inherits from Admin, the Editor gains all Admin permissions plus its own. In practice, most systems use additive inheritance — you can only add permissions, never remove them.

The hierarchy is stored as an adjacency list or a nested set in the database:

```sql
-- Adjacency list for role hierarchy
CREATE TABLE roles (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    parent_role_id INT REFERENCES roles(id)
);

-- See all permissions for a user via role traversal
WITH RECURSIVE role_chain AS (
    SELECT r.id, r.parent_role_id
    FROM user_roles ur
    JOIN roles r ON r.id = ur.role_id
    WHERE ur.user_id = ?
    UNION ALL
    SELECT r.id, r.parent_role_id
    FROM roles r
    JOIN role_chain rc ON r.id = rc.parent_role_id
)
SELECT DISTINCT p.*
FROM role_chain rc
JOIN role_permissions rp ON rp.role_id = rc.id
JOIN permissions p ON p.id = rp.permission_id;
```

The recursive query traverses the tree upward, collecting permissions at each level. This is efficient for read-heavy workloads but can be slow for write-heavy role management.

A typical ABAC policy looks like this:

```json
{
  "policy": "engineering-doc-read",
  "effect": "Permit",
  "condition": {
    "all": [
      { "match": "subject.department", "equals": "Engineering" },
      { "match": "resource.sensitivity", "in": ["internal", "public"] },
      { "match": "environment.time", "between": ["09:00", "17:00"] }
    ]
  }
}
```

The policy engine evaluates each condition against the resolved context. If all conditions match, the effect is applied. ABAC policies are expressive but require careful attribute management — if the PIP cannot resolve an attribute, the policy either fails open or fails closed, depending on your configuration.

### Role Hierarchy Trees

The role tree is the foundation of RBAC. Each node is a role, and edges represent inheritance. The tree is stored as an adjacency list in the database, and the PDP traverses it during evaluation.

Complexity analysis:

- **Lookup**: O(1) with an adjacency list and a hash index on role ID
- **Permission resolution**: O(h) where h is the tree height (typically 3-5 levels)
- **Role assignment**: O(1) for leaf nodes, O(h) for internal nodes
- **Memory**: O(n) where n is the number of roles

The tree structure is efficient for read-heavy workloads because the height is bounded. Most organizations have fewer than 10 levels of role hierarchy, so permission resolution completes in microseconds.

### Policy DAGs

ABAC policies form a Directed Acyclic Graph. Each node is a condition or a rule, and edges represent dependencies between conditions. The DAG allows the policy engine to short-circuit evaluation — if a condition fails, all dependent conditions are skipped.

The DAG is evaluated using a topological sort:

```python
def evaluate_policy_dag(policy_dag, context):
    # Topological order ensures dependencies are evaluated first
    order = topological_sort(policy_dag)
    results = {}

    for node_id in order:
        node = policy_dag.nodes[node_id]
        if node.type == "condition":
            results[node_id] = evaluate_condition(node, context)
        elif node.type == "combine":
            deps = [results[d] for d in policy_dag.incoming[node_id]]
            results[node_id] = combine_results(deps, node.algorithm)
        elif node.type == "effect":
            if results[policy_dag.incoming[node_id][0]]:
                return node.effect  # Permit or Deny

    return "NotApplicable"
```

Complexity:

- **Evaluation**: O(n + e) where n is the number of conditions and e is the number of edges
- **Short-circuit savings**: O(n) in the worst case, but typically much less because conditions are ordered by selectivity
- **Memory**: O(n + e) for the adjacency list representation

### ACL Bitmaps

Access Control Lists (ACLs) are the simplest data structure. Each resource has a list of permissions for each user. When you need maximum throughput, ACLs are stored as bitmaps.

A 32-bit bitmap can represent 32 different permissions. A 64-bit bitmap covers 64. The PDP checks a permission by testing a single bit:

```python
# ACL bitmap check
# Bit positions: 0=Read, 1=Write, 2=Delete, 3=Admin, 4=Share, ...
READ_BIT = 1 << 0   # 0b0001
WRITE_BIT = 1 << 1  # 0b0010

def check_permission(acl_bitmap, permission_bit):
    return (acl_bitmap & permission_bit) != 0

# Usage
alice_acl = 0b0011  # Read + Write
can_read = check_permission(alice_acl, READ_BIT)   # True
can_delete = check_permission(alice_acl, 1 << 2)   # False
```

Complexity:

- **Lookup**: O(1) — a single bitwise AND operation
- **Update**: O(1) — set or clear a single bit
- **Memory**: 8 bytes per user-resource pair (or 4 bytes for 32-bit bitmaps)
- **Storage**: O(u * r) where u is users and r is resources — can grow large

ACL bitmaps are ideal for high-throughput systems where the user-resource matrix is sparse but lookups must be fast. They are commonly used in file systems, message queues, and database row-level security.

Let us walk through each step in detail.

### Step 1: Client Sends Request

The client sends an HTTP request (or gRPC call, or message queue message) to the protected resource. The request includes the authentication token, the target resource identifier, and the action to perform.

### Step 2: PEP Intercepts Request

The PEP intercepts the request before it reaches the application logic. In a microservices architecture, the PEP is typically implemented as a middleware layer, a sidecar proxy (like Envoy or Istio), or an API gateway.

### Step 3: Extract Subject, Action, Resource

The PEP parses the request to extract the three key components:

- **Subject**: extracted from the JWT token, session cookie, or API key
- **Action**: extracted from the HTTP method (GET, POST, PUT, DELETE) or the gRPC method name
- **Resource**: extracted from the URL path, request body, or resource identifier header

### Step 4: Build Authorization Request

The PEP constructs an authorization request object:

```json
{
  "subject": {
    "id": "user:alice@corp",
    "token": "eyJhbGciOiJSUzI1NiIs..."
  },
  "action": "read",
  "resource": "document:42",
  "context": {
    "ip": "10.0.1.50",
    "user_agent": "Mozilla/5.0...",
    "timestamp": "2026-07-02T14:30:00Z"
  }
}
```

### Step 5: Forward to PDP

The PEP sends the authorization request to the PDP. This can be an in-process call (same process, same machine), a local RPC (localhost), or a remote call (network). The latency of this hop depends on the deployment model.

### Step 6: Check Decision Cache

Before evaluating, the PDP checks the decision cache. If a cached decision exists for the same subject, action, resource, and environment, and the cache entry has not expired, the PDP returns the cached decision immediately.

Cache key format:

```
cache_key = hash(subject_id + ":" + action + ":" + resource_id + ":" + environment_hash)
```

The cache hit ratio depends on the access pattern. In most systems, it ranges from 60% to 95%.

### Step 7: Resolve Attributes (PIP)

On a cache miss, the PDP contacts the PIP to resolve all attributes needed for evaluation. The PIP queries multiple data sources:

- Subject attributes from the user directory (LDAP, identity provider cache)
- Resource attributes from the resource registry (metadata service, database)
- Environment attributes from the request context and system sensors

The PIP aggregates these attributes into a flat evaluation context. This is the most variable-cost step — it can take 1 ms if everything is cached or 50 ms if multiple backend calls are needed.

### Step 8: Load Applicable Policies

The PDP loads policies that match the request's subject, action, and resource. Policies are indexed by target attributes for fast lookup:

- **Target policy index**: maps (action, resource_type) to policy IDs
- **Scope policy index**: maps (subject_group, resource_scope) to policy IDs
- **Default policy index**: policies that apply to all requests

### Step 9: Evaluate Rules

Each policy contains one or more rules. The PDP evaluates each rule against the evaluation context:

```python
def evaluate_rule(rule, context):
    if not rule.condition(context):
        return "NotApplicable"

    for obligation in rule.obligations:
        execute_obligation(obligation, context)

    return rule.effect  # "Permit" or "Deny"
```

Rules are evaluated in priority order. If a rule matches, its effect is recorded, and evaluation continues unless the combining algorithm short-circuits.

### Step 11: Log Decision

Every authorization decision is logged for audit, debugging, and analytics. The audit log captures:

- The full authorization request (subject, action, resource, context)
- The decision (Permit, Deny, NotApplicable, Indeterminate)
- The policies and rules that contributed to the decision
- The evaluation time in milliseconds
- A unique decision ID for correlation

Centralized logging is critical for compliance (SOC2, HIPAA, GDPR) and for debugging authorization failures.

### Step 12: Cache Decision

The decision is stored in the decision cache so that identical future requests can skip evaluation. The cache entry includes the decision, the policy version at the time of evaluation (for cache invalidation), and a TTL.

### Step 13: Enforce Decision

The PEP receives the decision from the PDP. If the decision is Permit, the PEP allows the request to proceed to the application logic. If Deny, the PEP returns a 403 Forbidden response immediately, without reaching the application.

### Step 14: Return Response

The PEP returns the appropriate HTTP response to the client. For permitted requests, the response includes the requested data. For denied requests, the response includes an error code and a description of why access was denied (if the security policy allows disclosing this information).

### Propagation Mechanisms

There are three common approaches to propagating policy updates:

**1. Polling (Pull Model)**
Each PDP polls the policy store at a configurable interval (typically 30-60 seconds). Simple to implement, but the propagation delay can be up to the polling interval. This is acceptable for most systems because policy changes are infrequent.

**2. Pub/Sub (Push Model)**
The PAP publishes policy change events to a message queue (Kafka, RabbitMQ, NATS). PDPs subscribe to the queue and apply updates immediately. Propagation delay is reduced to message delivery latency (milliseconds), but the system is more complex.

**3. Centralized PDP**
All decisions go through a single PDP, so there is no consistency issue. This sacrifices availability and throughput for consistency. Used in systems where policy consistency is critical, such as banking and healthcare.

### Concurrent Evaluation

When multiple authorization requests arrive simultaneously, the PDP must evaluate them concurrently. This introduces race conditions that can lead to incorrect decisions.

### Handling Stale Decisions

You have several strategies for handling the consistency window:

**Optimistic Revalidation**
After evaluation, check that the policy version has not changed. If it has, re-evaluate. This adds a version check to every decision but catches stale evaluations immediately.

**Lease-Based Caching**
PDPs acquire a lease on policy data. When a PDP holds a lease, it can evaluate without checking for updates. When the lease expires, the PDP must refresh. This bounds the consistency window to the lease duration.

**Policy Version Registry**
A central registry tracks the current policy version. PDPs include the version in each decision. If the client receives a decision with a stale version, it can request re-evaluation from a more recent PDP.

### Consistency Guarantees

| Guarantee | Propagation Delay | Throughput Cost | Use Case |
|-----------|------------------|-----------------|----------|
| Eventual | Up to polling interval (30-60s) | None | Content management, collaboration tools |
| Bounded staleness | Configurable (1-10s) | Low | E-commerce, social media |
| Read-your-writes | Session-scoped (immediate for own changes) | Medium | User settings, profile management |
| Strong | Synchronous update to all PDPs | High | Banking, healthcare, access control |

Most production authorization systems use bounded staleness with a propagation delay of 1-5 seconds. This is fast enough for nearly all use cases, and the complexity cost of strong consistency is rarely justified.


## Performance

Authorization must be fast. Every API request depends on an authorization decision, so the latency of the authorization system adds directly to the latency of every protected endpoint.

### RBAC Performance

RBAC is the fastest authorization model because the data structures are simple and the evaluation logic is straightforward:

| Operation | Median Latency | P99 Latency | Throughput (single core) |
|-----------|---------------|-------------|--------------------------|
| Cache hit decision | 200 microseconds | 500 microseconds | 100,000 decisions/sec |
| Role resolution | 1-3 milliseconds | 10 milliseconds | 30,000 decisions/sec |
| Full policy evaluation | 2-5 milliseconds | 20 milliseconds | 15,000 decisions/sec |
| Role assignment | 5-10 milliseconds | 50 milliseconds | 5,000 assignments/sec |

RBAC is millisecond-scale because the role hierarchy is small (typically 5-20 roles) and the permission resolution is a simple tree traversal. The bottleneck is almost always the database query for role membership, not the policy evaluation itself.

### ABAC Performance

ABAC is more expensive because attribute resolution requires multiple backend calls and the policy evaluation involves complex condition matching:

| Operation | Median Latency | P99 Latency | Throughput (single core) |
|-----------|---------------|-------------|--------------------------|
| Cache hit decision | 300 microseconds | 800 microseconds | 80,000 decisions/sec |
| Attribute resolution (cached) | 3-8 milliseconds | 25 milliseconds | 10,000 decisions/sec |
| Attribute resolution (uncached) | 10-30 milliseconds | 100 milliseconds | 2,000 decisions/sec |
| Full policy evaluation (10 attributes) | 5-15 milliseconds | 50 milliseconds | 5,000 decisions/sec |
| Full policy evaluation (50 attributes) | 20-50 milliseconds | 200 milliseconds | 1,500 decisions/sec |

ABAC latency scales with the number of attributes. Each additional attribute is a potential backend call, and each additional condition is a comparison operation. The key optimization is to resolve attributes lazily — only resolve the attributes that the matching policies actually need.

### Performance Optimization Strategies

**1. Pre-fetch attributes**
Resolve commonly used attributes proactively. If 90% of requests need the subject department, resolve it for every request rather than on demand.

**2. Batch attribute resolution**
Resolve all attributes for a request in a single batch call to the PIP, rather than making individual calls for each attribute.

**3. Parallel policy evaluation**
Evaluate policies in parallel when they are independent. The DAG structure makes this safe — policies with no shared dependencies can be evaluated concurrently.

**4. Compiled policies**
Compile policies into a decision tree or a finite state machine at load time. This transforms O(n) evaluation into O(log n) or even O(1).

```python
# Compiled policy example — transforms conditions into a decision tree
class CompiledPolicy:
    def __init__(self, policy):
        self.decision_tree = self._compile(policy)

    def _compile(self, policy):
        # Transform conditions into a decision tree
        # Each node is a condition, each edge is a result
        tree = DecisionNode()
        for rule in policy.rules:
            self._insert_rule(tree, rule, 0)
        return tree

    def evaluate(self, context):
        node = self.decision_tree
        while node.decision is None:
            result = node.condition.evaluate(context)
            node = node.children[result]
        return node.decision
```

**5. Decision caching**
Cache decisions aggressively. A cache hit is 10-100x faster than a full evaluation. Use a multi-level cache: L1 (in-memory, local to the PDP) and L2 (distributed, shared across PDPs).

### Benchmarking Authorization Systems

When benchmarking, you should measure:

- **Latency**: median, P99, and P999 under load
- **Throughput**: decisions per second at 50% CPU utilization
- **Cache hit ratio**: percentage of decisions served from cache
- **Policy evaluation time**: time spent in the evaluator (excluding attribute resolution)

A well-tuned authorization system should add less than 5 ms of latency to 99% of requests for RBAC, and less than 20 ms for ABAC.


## Configuration

A production authorization system requires careful configuration of the policy storage backend, evaluation caching, and policy versioning. These configuration choices determine the system's performance, consistency, and operational characteristics.

### Policy Storage Backend

The policy store is the source of truth for all policies. Your choice of backend affects latency, throughput, and consistency.

| Backend | Read Latency | Write Latency | Consistency | Use Case |
|---------|-------------|--------------|-------------|----------|
| In-memory (local) | 10-100 microseconds | 100-500 microseconds | Strong | Single-node, low latency |
| Redis | 1-3 milliseconds | 2-5 milliseconds | Eventual | Distributed, high throughput |
| PostgreSQL | 3-10 milliseconds | 5-20 milliseconds | Strong | Moderate scale, audit needs |
| DynamoDB | 5-15 milliseconds | 10-30 milliseconds | Eventual/Strong | Serverless, high scale |
| etcd | 3-8 milliseconds | 10-50 milliseconds | Linearizable | Configuration, leader election |

Choose an in-memory store for the PDP's working set (policies are loaded into memory at startup) and a durable store (PostgreSQL or DynamoDB) for the PAP's management operations.

The PDP loads policies on startup and refreshes them periodically:

```python
class PolicyStore:
    def __init__(self, backend, refresh_interval=60):
        self.backend = backend
        self.refresh_interval = refresh_interval
        self.policies = {}
        self._load_all_policies()

    def _load_all_policies(self):
        raw_policies = self.backend.fetch_all()
        self.policies = {
            p["id"]: self._compile(p)
            for p in raw_policies
        }

    def refresh_loop(self):
        while True:
            time.sleep(self.refresh_interval)
            self._load_all_policies()
```

### Evaluation Caching

Caching is the most impactful performance lever. Configure it carefully:

**L1 Cache (Local, In-Memory)**
- Size: 10,000-100,000 entries
- Eviction: LRU
- TTL: 60-300 seconds
- Storage: dict or concurrent hash map

**L2 Cache (Distributed, Shared)**
- Backend: Redis or Memcached
- Size: 100,000-1,000,000 entries
- Eviction: LRU with TTL
- Serialization: Protocol Buffers or MessagePack for compact storage

**Cache Invalidation Rules**
1. Invalidate on policy version change — if the policy version increments, the entire cache is stale
2. Invalidate on role change — if a user's role assignment changes, invalidate all cache entries for that user
3. Invalidate on attribute change — if a user's department changes, invalidate attribute-dependent cache entries
4. TTL-based expiration — even without changes, entries expire after a configurable TTL

```python
class DecisionCache:
    def __init__(self, l1_size=100000, l2_client=None, default_ttl=120):
        self.l1 = LRUCache(l1_size)
        self.l2 = l2_client  # Redis client
        self.default_ttl = default_ttl
        self.policy_version = 1

    def get(self, key):
        # Check L1 (fastest)
        result = self.l1.get(key)
        if result is not None:
            if result["policy_version"] == self.policy_version:
                return result["decision"]
            self.l1.delete(key)  # Stale entry

        # Check L2 (slower but shared)
        if self.l2:
            raw = self.l2.get(key)
            if raw:
                result = deserialize(raw)
                if result["policy_version"] == self.policy_version:
                    self.l1.put(key, result)  # Promote to L1
                    return result["decision"]
                self.l2.delete(key)

        return None  # Cache miss

    def set(self, key, decision):
        entry = {
            "decision": decision,
            "policy_version": self.policy_version,
            "timestamp": time.time()
        }
        self.l1.put(key, entry)
        if self.l2:
            self.l2.setex(key, self.default_ttl, serialize(entry))

    def invalidate_policy_version(self, new_version):
        self.policy_version = new_version
        self.l1.clear()
        if self.l2:
            self.l2.flushdb()  # Clear all cache entries
```

### Policy Versioning

Policy changes must be versioned to support rollback, audit, and gradual rollout.

**Versioning Scheme**

Use a monotonically increasing version number for the entire policy set:

```
v1.0.0 — Initial policy set
v1.1.0 — Added editor role
v1.2.0 — Modified document read policy
v2.0.0 — Restructured role hierarchy
```

Each version captures the complete state of the policy system. This makes rollback trivial — you simply load the previous version.

**Rollout Strategy**

1. Write the new policy version to the policy store (PAP operation)
2. Increment the global policy version counter
3. PDPs detect the new version (via polling or push notification)
4. PDPs load the new policies and invalidate their caches
5. Monitor for errors — if error rate increases, roll back to the previous version

**Canary Deployment**

For high-risk policy changes, deploy the new version to a subset of PDPs first:

```python
def canary_rollout(new_policy_version, canary_percentage=5):
    # Deploy to canary PDPs
    for pdp in get_canary_pdps(canary_percentage):
        pdp.load_policies(new_policy_version)

    # Monitor for canary period
    time.sleep(300)  # 5 minute observation window

    error_rate = check_error_rate(canary_pdps)
    if error_rate > threshold:
        rollback(canary_pdps, previous_version)
        return False

    # Roll out to remaining PDPs
    for pdp in get_remaining_pdps():
        pdp.load_policies(new_policy_version)

    return True
```

### Configuration Checklist

When configuring an authorization system for production, you should verify:

1. **Storage backend**: durable, highly available, consistent enough for your use case
2. **Cache sizing**: L1 fits in memory, L2 fits in Redis/Memcached with headroom
3. **Cache TTL**: long enough for good hit ratio, short enough for acceptable staleness
4. **Policy versioning**: monotonically increasing, supports rollback
5. **Refresh interval**: fast enough for acceptable propagation delay, slow enough to avoid load spikes
6. **Audit logging**: every decision logged with enough context for debugging
7. **Monitoring**: latency, throughput, cache hit ratio, error rate, policy version drift
8. **Alerting**: threshold-based alerts for high latency, low cache hit ratio, and policy evaluation errors
