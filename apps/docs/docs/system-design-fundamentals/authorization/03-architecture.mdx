# Authorization at Scale

Every system starts simple. You write `if user.role === "admin"` in a controller, deploy it, and move on. That works for a few hundred users. By the time you cross 10,000 users, those checks are scattered across twenty services. At 1 million, you have a policy explosion. At 100 million, you need a dedicated authorization system that runs at the speed of your network, not your application code.

This page walks you through the architectural evolution of authorization: from naive inline checks, through centralized policy engines, into multi-region, multi-layer enforcement. You will see concrete AsciiDiagrams at each stage, capacity planning formulas, and deployment patterns that keep your authorization system running through policy updates, traffic spikes, and regional outages.

## The Naive Approach: Inline Authorization

The simplest possible authorization is a conditional branch in your application code. You check a role, a permission flag, or a group membership right where the request arrives.

```python
# app/controllers/document_controller.py
def get_document(request, document_id):
    user = authenticate(request)
    if user.role != "admin" and user.role != "editor":
        abort(403)
    return db.query(Document).get(document_id)
```

This pattern works for a prototype or an internal tool with fewer than 1,000 users. Every developer on the team can see the logic, there is no network hop for a policy decision, and the mental model is trivial.

**Why it breaks by 1,000 users:**

- **Duplication.** Every endpoint duplicates the same role checks. A policy change requires touching every controller.
- **Inconsistency.** One endpoint checks `user.role === "admin"`, another checks `user.is_admin === True`, a third calls a helper that checks `user.role === "administrator"`. These drift over time.
- **No audit trail.** When a user accesses a document they should not have, there is no centralized record of the decision.

**Why it breaks by 100,000 users:**

- **Role explosion.** You now have admin, editor, contributor, reviewer, viewer, billing-admin, audit-reader, and delegate-admin. Combinatorial checks like `if (user.role in ["admin", "editor", "contributor"]) and document.owner_id == user.id` become unreadable.
- **Attribute-based rules.** You need to check department, document classification, time of day, geographic region, and approval status. Inline conditionals cannot express this without becoming a tangled mess.
- **Performance.** Every request that loads the user object and iterates through permission lists adds latency. At 100,000 requests per minute, those microseconds multiply into seconds of cumulative overhead.

**Why it breaks at 10 million users:**

- **Distributed state.** User roles live in one database, document classifications in another, group memberships in a cache, and hierarchical org structures in a third system. Your inline code needs to fan out to five data sources to make a single decision.
- **No atomicity.** A policy update requires a deployment. You cannot change "all editors can now publish" without shipping new code, running a migration, and hoping every service picks up the change simultaneously.
- **No observability.** You cannot answer "how many requests were denied for this policy" or "which endpoint has the most expensive authorization check." The decision logic is invisible.

## Production Architecture: Centralized Policy Engine

The industry answer to the inline mess is a centralized policy engine. You extract every authorization decision into a dedicated service that evaluates policies against request context and returns a binary allow or deny. The two most common open-source engines are Open Policy Agent (OPA, using Rego) and Casbin (using its own model definition language).

### How It Works

Your application service sends a query to the policy engine. The query contains the subject (who is asking), the action (what they want to do), the resource (what they want to do it to), and any context attributes. The policy engine evaluates this against stored policies and returns a decision.

The policy engine sits outside your application logic. It has its own data store for policies, its own cache for decisions, and its own observability pipeline. Your application becomes a thin client that asks "can this user do this thing?" and trusts the answer.

### Policy as Code

Policies are written in a declarative language, stored in version control, and deployed through a CI/CD pipeline. A Rego policy for document access might look like this:

```rego
package document.authorization

default allow = false

# Admins can access any document
allow {
    input.user.roles[_] == "admin"
}

# Editors can access documents they own
allow {
    input.user.roles[_] == "editor"
    input.resource.owner_id == input.user.id
}

# Reviewers can access unlocked documents
allow {
    input.user.roles[_] == "reviewer"
    input.resource.locked == false
}

# Anyone in the same department can read public documents
allow {
    input.resource.classification == "public"
    input.user.department == input.resource.department
}
```

This policy is a single file, tested with unit tests, reviewed in a pull request, and deployed without touching any application code. When you need to add a rule -- "editors can also publish if the document has been approved" -- you change the policy file, not twenty controllers.

### Decision Caching

Authorization decisions are expensive to compute. A single decision may involve evaluating dozens of rules, fetching attributes from multiple data sources, and running complex set operations. Caching decisions by their input hash saves significant latency.

The cache key is a hash of the subject, action, resource, and context attributes. The cache value is the allow/deny decision plus a time-to-live. You set the TTL based on how quickly the underlying attributes can change. For static roles, you might cache for 300 seconds. For real-time stock trading permissions, you might cache for 5 seconds or skip caching entirely.

Cache hit ratios of 80 to 95 percent are typical for most workloads. The remaining cache misses go through the full evaluation path, which adds 5 to 15 milliseconds of latency depending on the complexity of your policies.

### Policy Engine Performance Characteristics

A well-tuned policy engine handles 10,000 to 50,000 decisions per second on a single node with moderate policies (50 to 200 rules). With caching, the effective throughput can reach 200,000 decisions per second because 90 percent of requests never hit the evaluator.

**Latency budget breakdown:**

| Component | Uncacheable | Cache Hit |
|---|---|---|
| Network (client to engine) | 1-2 ms | 1-2 ms |
| Cache lookup | -- | 0.1 ms |
| Policy evaluation | 3-10 ms | -- |
| Data fetching | 2-5 ms | -- |
| **Total** | **6-17 ms** | **1-2 ms** |

Your application should treat the authorization decision as a blocking call, but you should set a timeout of 50 milliseconds. If the engine does not respond within the timeout, your application should deny the request by default. Authorization failures must be safe, not permissive.

## Integration Patterns

A centralized policy engine does not exist in isolation. It must integrate with the infrastructure that surrounds it: the API gateway that receives requests, the databases that store data, the message queues that carry asynchronous events, and the service mesh that connects everything.

### Integration with API Gateway

The API gateway is the first line of enforcement. You configure the gateway to intercept every request and call the policy engine before routing to the backend service. This pattern is called **pre-authorization**.

The gateway extracts the subject from the authentication token (JWT, session cookie, or API key), constructs the authorization request, and sends it to the policy engine. If the engine denies, the gateway returns a 403 response immediately without ever touching the backend service.

This saves backend capacity. A denial at the gateway costs 2 to 5 milliseconds. A denial that reaches the backend might cost 50 to 200 milliseconds of processing before the authorization check fails.

**Implementation considerations:**

- The gateway must parse the JWT to extract subject claims. This is standard in most API gateways (Kong, Envoy, AWS API Gateway, NGINX).
- The gateway should cache decisions with a short TTL (30 to 60 seconds) to avoid hammering the policy engine on every request.
- The gateway should pass the decision as a header to the backend service so the service does not re-evaluate the same policy. The header might be `X-Authz-Decision: allow` with a signature to prevent tampering.

### Integration with Database (Row-Level Security)

Authorization at the API layer is not enough. A user can bypass the API and connect directly to the database with a read replica, a reporting tool, or a data pipeline. You need row-level security (RLS) inside the database itself.

In PostgreSQL, RLS policies are defined as SQL expressions that the database evaluates on every row access. You can integrate the policy engine by generating RLS policies from the same policy-as-code definitions.

```sql
-- PostgreSQL RLS policy generated from Rego policy
CREATE POLICY document_access ON documents
    USING (
        current_user = 'admin'
        OR (
            owner_id = current_user_id()
            AND current_setting('app.role') = 'editor'
        )
        OR (
            classification = 'public'
            AND department = current_setting('app.department')
        )
    );
```

The challenge is keeping the RLS policies in sync with the policy engine. You can run a cron job that pulls the latest policy definitions from the policy store and regenerates the SQL policies. Alternatively, you can use a proxy layer (such as pgbouncer with a custom auth plugin) that intercepts SQL queries and checks them against the policy engine before executing them.

### Integration with Message Queue

Asynchronous workloads -- background jobs, event processing, batch operations -- need authorization too. A message consumer must check whether the producer had the right to send the message and whether the consumer has the right to process it.

You embed the authorization context in the message envelope. The producer includes a signed authorization token that proves the action was authorized at produce time. The consumer verifies the token and optionally re-evaluates the policy if the consumer's context differs from the producer's.

```json
{
    "event": "document.updated",
    "payload": { "document_id": "doc-123", "new_status": "published" },
    "authz": {
        "subject": "user-456",
        "action": "publish",
        "resource": "document:doc-123",
        "decision": "allow",
        "policy_version": "v2024-03-15",
        "signature": "signed-by-authz-engine"
    }
}
```

The consumer extracts the `authz` block, verifies the signature against the policy engine's public key, and checks that the policy version is still current. If the policy has changed since the message was produced, the consumer re-evaluates. This pattern prevents replay attacks and stale-approval attacks.

### Integration with Service Mesh

In a service mesh (Istio, Linkerd, Consul Connect), every service-to-service call goes through a sidecar proxy. The sidecar can enforce authorization policies at the network level without the application service knowing about it.

You configure the service mesh to call the policy engine on every gRPC or HTTP call between services. The sidecar extracts the source identity from the mTLS certificate, constructs the authorization request, and enforces the decision before forwarding the call.

Service mesh authorization is coarse-grained. It checks whether service A can call service B, not whether a specific user can access a specific resource. For fine-grained authorization, you still need the application-level policy engine call. The two layers complement each other: the mesh provides network-level security, and the policy engine provides resource-level security.

## Multi-Region Architecture

When your system spans multiple geographic regions, the authorization system must run in every region. You cannot route every authorization request to a single region -- the latency would be unacceptable (200 to 500 milliseconds for intercontinental round trips), and a regional outage would take down authorization globally.

### Policy Replication

The policy store is replicated from the primary region to all secondary regions. You use a leader-follower replication model where the primary region accepts policy updates and propagates them to followers.

**Replication strategies:**

- **Synchronous replication.** Every policy update waits for acknowledgment from all regions before completing. This guarantees consistency at the cost of update latency. Acceptable for policy updates that happen once per day.
- **Asynchronous replication.** The primary region acknowledges the update immediately and propagates changes in the background. Followers may serve stale policies for a few seconds. This is the common choice because policy updates are infrequent and a few seconds of staleness is acceptable.
- **Eventual consistency with versioning.** Each policy has a version number. The region evaluates all requests with its current version. If a request carries a context that references a newer version (such as a JWT issued after a policy change), the region fetches the latest policy on demand.

### Decision Consistency

Authorization decisions must be consistent across regions. The same user, action, and resource must produce the same decision in every region. This is harder than it sounds because the context attributes may differ between regions.

For example, if a user's role changes from "editor" to "viewer" in the primary region, the secondary region must learn about this change before it can make correct decisions. The propagation delay between regions creates a window of inconsistency.

**Mitigation strategies:**

- **Short TTL on decision caches.** Set the cache TTL to 30 seconds in multi-region deployments. This limits the inconsistency window to the cache TTL plus the replication lag.
- **Write-through cache invalidation.** When a role changes, the primary region sends an invalidation message to all regions. Each region evicts the affected cache entries immediately.
- **Re-evaluation on context change.** The client includes a context version number in every request. If the region's policy version is older than the context version, the region skips the cache and re-evaluates.

### Regional Failover

If a region loses connectivity to the primary region, it must continue making authorization decisions. The policy engine in each region has a local copy of all policies and can evaluate decisions independently. The only risk is stale context data, which is handled by the consistency mechanisms above.

During a failover, the region should:

1. Continue serving decisions from the local policy cache.
2. Queue any context updates that cannot be replicated.
3. Serve stale decisions for up to 5 minutes before denying.
4. Log every decision made during the isolation period for audit.

When connectivity is restored, the region replays the queued updates and reconciles the decision logs with the primary region.

## Capacity Planning

You need to know how many decisions your authorization system can handle and how to scale it. The key metric is **decisions per second (DPS)**.

### Estimating DPS

Every user action that requires authorization is one decision. For a typical web application, each request generates one to three authorization decisions (one for the API gateway, one for the backend service, and possibly one for a database query).

**Formula:**

```
DPS = (MAU x requests_per_user_per_day / 86400) x decisions_per_request
```

**Example:**

- 10 million MAU
- 50 requests per user per day
- 1.5 decisions per request

```
DPS = (10,000,000 x 50 / 86400) x 1.5
DPS = 5787 x 1.5
DPS = 8680
```

Your policy engine must handle 8,680 decisions per second at peak. With a 2x buffer for traffic spikes, you need capacity for 17,360 DPS.

### Cache Hit Ratio Targets

The cache hit ratio determines how many decisions go through the full evaluation path.

| Cache Hit Ratio | Effective Evaluation DPS | Required Nodes |
|---|---|---|
| 0% (no cache) | 17,360 | 4-6 |
| 50% | 8,680 | 2-3 |
| 80% | 3,472 | 1-2 |
| 95% | 868 | 1 |

**Target: 90% cache hit ratio.** This gives you a 10x reduction in evaluation load. At 17,360 peak DPS with 90% cache hits, you need only 1,736 evaluations per second, which a single well-tuned node can handle.

### Achieving 90% Cache Hit Ratio

- **Cache by input hash.** The key must include subject, action, resource, and all relevant context attributes. A hash of these fields produces a unique key for each decision pattern.
- **Tune TTL by policy type.** Static roles (admin, viewer) can be cached for 5 minutes. Dynamic attributes (stock ownership, real-time clearance) need 10-second TTLs.
- **Pre-warm the cache.** After a deployment, run a batch of the most common request patterns to populate the cache before user traffic arrives.
- **Evict on context change.** When a user's role changes, proactively evict all cache entries for that user. This prevents stale decisions without waiting for TTL expiration.

### Memory and Network

Each cache entry is approximately 200 bytes (key + value + metadata). For 10 million unique decision patterns, you need 2 GB of cache memory. In practice, most deployments have 1 to 5 million active patterns, requiring 200 MB to 1 GB of cache.

The policy engine's network bandwidth is driven by the size of authorization requests. Each request is 500 bytes to 2 KB (subject claims, resource attributes, context). At 17,360 DPS, you need 17 to 35 MBps of network throughput. This is well within the capacity of a single 10 Gbps network interface.

## Deployment: Zero-Downtime Policy Updates

Authorization is critical infrastructure. A bad policy deployment can deny access to every user or, worse, allow access that should be denied. You must deploy policy updates with zero downtime and the ability to roll back instantly.

### Blue-Green Policy Deployment

Maintain two policy stores: the active store and the staging store. The policy engine loads policies from the active store. When you deploy a new policy version, you load it into the staging store, run a validation suite, and then swap the active pointer.

**Validation suite:**

1. **Syntax check.** Does the policy parse correctly? OPA's `rego test` and Casbin's model validation catch syntax errors before they reach production.
2. **Unit tests.** Every rule has a test that verifies its behavior. The test suite must pass before the policy is deployed.
3. **Regression tests.** A set of known decision patterns that must not change. If the new policy changes the decision for a previously verified pattern, the deployment is blocked.
4. **Dry-run evaluation.** The new policy runs in parallel with the old policy. Every decision is evaluated by both versions, and the results are compared. If the new policy disagrees with the old policy on any decision, an alert fires. This is the most powerful validation technique because it catches unintended changes before they affect users.

### Canary Policy Deployment

After validation, deploy the new policy to a subset of traffic. Route 1% of decisions to the new policy version and 99% to the old version. Monitor the new policy's decision distribution, latency, and error rate. If the metrics are stable, increase to 10%, then 50%, then 100%.

**Canary criteria:**

- Decision distribution: the ratio of allow to deny should not change by more than 5%. If the old policy denied 10% of requests and the new policy denies 30%, there is a bug or an intentional tightening that needs review.
- Latency: the new policy's P99 latency should be within 10% of the old policy.
- Error rate: zero errors in the canary. Any error triggers an automatic rollback.

### Rollback

If the canary detects an issue, the policy engine swaps back to the previous version. The rollback is instant because the old policy is still loaded in the active store. The staging store is simply re-pointed to the previous version.

Rollback is not just a technical operation. You must also log the rollback event, notify the team, and preserve the failed policy in a quarantine bucket for post-mortem analysis.

### Monitoring

Your authorization system needs the same monitoring as any critical infrastructure, plus some authorization-specific metrics.

**Required metrics:**

- `authorization.decisions.total` -- total decisions per second, labeled by allow/deny/error
- `authorization.decisions.latency` -- latency distribution (P50, P99, P99.9) for the full evaluation path
- `authorization.cache.hit_ratio` -- cache hit ratio, labeled by policy version
- `authorization.policy.version` -- currently active policy version (gauge)
- `authorization.errors.total` -- errors from the policy engine, labeled by error type
- `authorization.context.staleness` -- age of context data used for decisions (multi-region)

**Alert thresholds:**

| Alert | Threshold | Action |
|---|---|---|
| Error rate spike | > 0.1% errors | Rollback to previous policy version |
| Cache hit ratio drop | < 70% for 5 minutes | Investigate cache key design or TTL configuration |
| P99 latency spike | > 100 ms for 5 minutes | Scale up policy engine nodes |
| Decision distribution shift | Allow/deny ratio changes by > 10% | Review policy changes from last deployment |
| Region isolation | Secondary region cannot reach primary for > 5 minutes | Activate failover mode |

## Summary: Architecture Checklist

When you design an authorization system at scale, use this checklist to validate your architecture:

**Policy Management**

- [ ] Policies are stored in version control as code.
- [ ] Policies are deployed through a CI/CD pipeline with automated testing.
- [ ] Policy changes are validated with dry-run evaluation against production traffic.
- [ ] Policies are versioned and can be rolled back instantly.

**Decision Path**

- [ ] Every authorization request goes through a centralized policy engine.
- [ ] The API gateway enforces coarse-grained authorization before routing to backends.
- [ ] The service mesh enforces network-level authorization between services.
- [ ] The database enforces row-level security as a defense-in-depth layer.
- [ ] Asynchronous messages carry signed authorization tokens for end-to-end verification.

**Caching**

- [ ] Decisions are cached by input hash with configurable TTLs.
- [ ] Cache hit ratio is monitored and kept above 90 percent.
- [ ] Cache entries are proactively evicted on context changes.
- [ ] Cache is pre-warmed after deployments.

**Multi-Region**

- [ ] Every region has a local policy store and policy engine.
- [ ] Policy updates are replicated asynchronously from the primary region.
- [ ] Decision caches have short TTLs (30 seconds) to limit inconsistency windows.
- [ ] Regional failover works independently with local data.

**Observability**

- [ ] Decision count, latency, and cache hit ratio are monitored in real time.
- [ ] Error rate alerts trigger automatic rollback.
- [ ] Decision distribution shifts trigger manual review.
- [ ] All decisions are logged for audit with subject, action, resource, and decision.

You have built a mental model of authorization at scale: from inline checks that work for a hundred users, through centralized policy engines that handle millions, to multi-region architectures that serve billions of requests across the globe. You have seen the integration patterns that embed authorization into every layer of the stack, the capacity planning formulas that tell you how many nodes you need, and the deployment patterns that keep your system running through policy updates.

The next page explores the trade-offs you must make when choosing between different authorization models: role-based access control (RBAC) versus attribute-based access control (ABAC), centralized versus distributed enforcement, and policy-as-code versus policy-as-configuration. There is no single right answer -- the trade-offs depend on your scale, your team, and your compliance requirements.
