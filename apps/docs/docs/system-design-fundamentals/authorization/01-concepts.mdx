## The Healthcare Startup

You join a fast-growing healthcare startup as its tenth engineer. The company stores patient records, treatment plans, lab results, and -- yes -- employee payroll data in the same monolithic application. Authentication exists. Everyone logs in with a company email and a password.

One afternoon, a junior frontend developer notices the "Employee Directory" endpoint returns a `salary` field alongside `name` and `role`. Out of curiosity, they write a quick loop to fetch every employee's salary. Within 30 seconds, they see the CEO's compensation, the CTO's bonus, and your annual raise from six months ago.

No alert fires. No log entry flags the request. The application simply responds: "you are authenticated, here is the data."

A patient later files a complaint about a data broker contacting them with details from their medical record. The investigation traces the leak. The broker paid an intern at the startup who exported thousands of patient profiles through the same unguarded API. The intern was authenticated. The system never asked: *should this user see this data?*

The result: a USD 4.5 million HIPAA settlement, three terminated employees, and a startup that nearly folded before its Series B.

The root cause was not authentication. It was the complete absence of **authorization**.

---

## The Insight

**Authorization** is the process of determining what an authenticated user is allowed to do. It is the gate that stands between "who you are" and "what you can access."

Think of an airport. **Authentication** is your ticket -- it proves you bought a seat on flight UA 792. **Authorization** is your boarding pass -- it determines which gate you may enter, which lounge you may use, and whether you sit in first class or economy. A ticket gets you through security. The boarding pass controls every door after that.

In system design, the authorization layer answers four questions for every request:

- Who is making this request? (identity, established by authentication)
- What resource are they trying to access? (a document, an API endpoint, a database row)
- What action are they trying to perform? (read, write, delete, admin)
- Should this action be allowed? (the policy decision)

## Core Vocabulary

Before you build anything, own these twelve terms. Interviewers use them interchangeably, and precision matters at FAANG level.

| Term | Meaning |
|---|---|
| **Authorization** | The process of granting or denying access to a resource based on policy |
| **Access Control** | The broader discipline of regulating who or what can view or use resources |
| **Authentication** | Verifying identity (who you are) -- distinct from but paired with authorization |
| **RBAC** | Role-Based Access Control -- permissions are assigned to roles, roles are assigned to users |
| **ABAC** | Attribute-Based Access Control -- policies use attributes of user, resource, and environment |
| **PBAC** | Policy-Based Access Control -- a superset that evaluates policies expressed as rules |
| **ACL** | Access Control List -- an explicit list of permissions attached to an object |
| **ReBAC** | Relationship-Based Access Control -- permissions derive from relationships between entities |
| **Policy** | A rule that defines whether access is granted: "managers may read reports in their department" |
| **Permission** | A discrete action privilege: `read:patient-record`, `write:payroll` |
| **Role** | A named collection of permissions: `Admin`, `Editor`, `Viewer` |
| **Scope** | The boundary within which a permission applies: `org:42`, `department:engineering` |
| **Principle of Least Privilege** | Every user should receive only the minimum permissions necessary to do their job |
| **IAM** | Identity and Access Management -- the combined system for authentication, authorization, user management, and audit |

Three key components:

1. **Policy Enforcement Point (PEP):** Intercepts every request before it reaches the resource. If no authorization check exists here, you have the healthcare startup problem.
2. **Policy Decision Point (PDP):** Evaluates the request against stored policies and returns allow or deny. This is the brain.
3. **Policy Information Point (PIP):** Supplies context the PDP needs -- user attributes, resource metadata, environmental data like time of day.

Every production authorization system records every decision in an **audit log**. You cannot prove compliance without it.

## Step by Step: How a Request Gets Authorized

Follow a single request through a real authorization system. An engineer named Priya attempts to READ the patient record with ID `pat-8842`.

### Step 1 -- Request Arrives

Priya's HTTP request reaches the API gateway:

```
GET /api/v2/patients/pat-8842
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
```

The gateway validates the JWT and extracts Priya's identity: `user_id = priya@carecorp.com, role = Nurse, department = Cardiology, org = hos-12`.

### Step 2 -- The PEP Intercepts

The API handler calls the authorization middleware -- the PEP -- before executing any business logic. The PEP extracts:

| Attribute | Value |
|---|---|
| Subject (who) | `priya@carecorp.com` |
| Action (what) | `read` |
| Resource (what target) | `patient-record:pat-8842` |
| Environment (context) | `ip=203.0.113.42, time=14:32:01 UTC, device=managed-laptop` |

### Step 3 -- PEP Calls PDP

The PEP sends an authorization request to the PDP. In a system built with OPA (Open Policy Agent) or AWS Cedar, this is a structured JSON payload:

```json
{
  "input": {
    "user": "priya@carecorp.com",
    "action": "read",
    "resource": {
      "type": "patient-record",
      "id": "pat-8842",
      "department": "cardiology",
      "org": "hos-12"
    },
    "context": {
      "time": "2026-07-02T14:32:01Z",
      "ip": "203.0.113.42",
      "device": "managed-laptop"
    }
  }
}
```

### Step 4 -- Policy Evaluation

The PDP loads policies relevant to this resource type. Here is the rule set that fires:

```
Rule 1: Nurses in Cardiology may read patient records assigned to Cardiology.
Rule 2: No user may read records outside their org.
Rule 3: Read requests from managed devices between 06:00 and 22:00 are allowed.
```

Priya satisfies all three rules: she is a Nurse, her department is Cardiology, record `pat-8842` is assigned to Cardiology, the org matches, the device is managed, and the time is 14:32.

### Step 5 -- Decision

The PDP returns `allow`.

```json
{
  "decision": "allow",
  "evaluated_rules": ["rule-1", "rule-2", "rule-3"],
  "obligations": [
    {"audit": true, "level": "info"}
  ]
}
```

### Step 6 -- Audit Log

The system writes a structured audit entry:

```
{
  "timestamp": "2026-07-02T14:32:01.342Z",
  "user": "priya@carecorp.com",
  "action": "read",
  "resource": "patient-record:pat-8842",
  "decision": "allow",
  "rules_fired": ["rule-1", "rule-2", "rule-3"],
  "request_id": "req-a7f3c9e1"
}
```

### Step 7 -- Resource Access

The PEP caches the decision (TTL: 60 seconds) and forwards the request to the backend service that serves the patient record. The backend does not re-check authorization -- it trusts the PEP.

### Step 8 -- Response

Priya receives the patient record. The entire flow completes in under 15 milliseconds in a warm system.

### The Denied Case

Now imagine Priya attempts to read a record from the `Neurology` department, where she is not assigned.

The PDP evaluates the same three rules. Rule 1 fails: Priya's department is Cardiology, not Neurology. The PDP returns `deny` with the reason `rule-1: department_mismatch`. The PEP returns HTTP `403 Forbidden`. The audit log records the denial. No data leaves the system.

### Caching and Performance

Every authorization check adds latency. In a system handling 50,000 requests per second, even 5 milliseconds per check costs 250 CPU-seconds per second. Solutions:

- **Local PDP cache:** Cache decisions with a TTL (60-300 seconds). Invalidated when policies change.
- **Decision batching:** Send 50-100 concurrent checks in a single PDP call.
- **Sidecar PDP:** Run the PDP as a sidecar on each host -- no network hop.
- **Policy compilation:** Precompile policies into boolean decision trees for O(1) evaluation.

Google's **Zanzibar** (the system behind Google Calendar, Drive, and YouTube) caches ACL checks in a distributed, globally replicated store. It processes 95 percent of checks in under 10 milliseconds using a combination of local cache and a consistent, replicated namespace.

## The Authorization Variants

No single authorization model fits every system. Here are the four major variants you must know for system design interviews.

### RBAC -- Role-Based Access Control

The most widely deployed model. Permissions are assigned to roles, roles are assigned to users. A user may have multiple roles.

**When to use:** Enterprise applications, internal tools, systems with well-defined job functions. GitHub, AWS IAM, and Kubernetes RBAC are canonical examples.

**Limitations:** Role explosion. A hospital with 200 job titles needs 200 roles. A single HR policy exception ("Jessica in Engineering also approves expense reports") requires a custom role for one person.

### ABAC -- Attribute-Based Access Control

Policies use any attribute -- user department, resource classification, time of day, IP range, device type, geographic location. ABAC is far more flexible than RBAC but harder to audit.

**When to use:** Multi-tenant SaaS, healthcare (HIPAA constraints vary by data type), financial systems where compliance rules reference dozens of attributes.

**Real-world example:** A policy in a healthcare ABAC system:

```
ALLOW read ON patient-record
  WHERE user.department = resource.department
    AND resource.confidence_level <= user.clearance_level
    AND context.time BETWEEN "06:00" AND "22:00"
    AND context.device_type = "managed"
```

### ACL -- Access Control Lists

Every resource carries an explicit list of who can do what. ACLs are simple, intuitive, and degenerate to terrible performance at scale.

**When to use:** File systems, shared documents, systems where each resource has unique permissions. Traditional Unix file permissions are an ACL (user/group/other).

**The scaling problem:** A document shared with 10,000 users requires 10,000 entries. Google Drive solved this with **ACL compression** and a separate ACL service (Zanzibar).

### ReBAC -- Relationship-Based Access Control

Permissions are derived from relationships between entities in a graph. "If Alice is a member of the Cardiology group, and Cardiology group has the `viewer` relation on the Cardiology Slack channel, then Alice can view the Cardiology Slack channel."

**When to use:** Social networks, collaborative tools, systems with complex group hierarchies. Google Drive, Dropbox, Slack, and GitHub all use ReBAC-like models.

**Why ReBAC exists:** RBAC cannot express "users who are mentioned in a comment can edit that comment." ACL cannot scale to billions of edges. ReBAC answers graph traversal queries: "can user X reach resource Y through N hops of relationship?"

### When to Use Each Model

| Model | Strengths | Weaknesses | Best For |
|---|---|---|---|
| RBAC | Simple, well-understood, tooling everywhere | Role explosion, rigid | Internal tools, enterprise apps |
| ABAC | Highly flexible, fine-grained, context-aware | Hard to audit, policy sprawl | Healthcare, finance, multi-tenant SaaS |
| ACL | Intuitive, per-resource precision | Does not scale past thousands of entries per resource | File systems, shared docs |
| ReBAC | Scales to billions of edges, handles complex hierarchies | Harder to debug, requires graph DB | Social networks, collaboration platforms |

Production systems often combine models. Google Drive uses ReBAC for sharing and ACL for per-file overrides. AWS IAM uses RBAC (managed policies) with ABAC (condition keys). Kubernetes uses RBAC with ABAC-style attribute checks on pod security policies.

## Mental Model Check

Answer these five questions to verify your understanding.

**Q1:** You are designing the authorization system for a hospital with 12,000 employees, 200 departments, and strict HIPAA regulations that vary by data type. Which model do you choose as your primary approach, and why?

**A1:** ABAC. HIPAA rules reference attributes like patient consent status, treatment relationship, and data classification -- not job titles. An ABAC policy like `ALLOW read ON patient-record WHERE clinician.treatment_relationship = patient.id AND patient.consent_status = "granted"` covers 200 departments without 200 roles. Supplement with RBAC for coarse access (Admin, IT, HR).

**Q2:** Your file-sharing startup has 10 million users. Each user shares folders with an average of 50 collaborators. Using raw ACLs, estimate the storage required for the ACL table.

**A2:** 10 million users x 50 collaborators = 500 million ACL entries. At roughly 80 bytes per entry (user_id + resource_id + permissions mask + timestamp), that is 40 GB plus indexes. Zanzibar at Google handles trillions of ACL entries using delta-based ACLs (store only changes, reconstruct the full ACL from deltas on read) and sharding by object namespace.

**Q3:** A request arrives at your system. The user is authenticated but not authorized. What HTTP status code do you return, and what is the critical step you must not skip?

**A3:** Return `403 Forbidden`. The critical non-skippable step is writing an audit log entry with the denied user, resource, action, and reason. Without this, you cannot detect attack patterns (a single user hitting 1,000 denied endpoints in 30 seconds is a brute-force attempt) and you cannot pass compliance audits.

**Q4:** Your startup uses RBAC with 15 roles. A new regulation requires that a user from the Finance department may only see patient records between 08:00 and 18:00 local time and only from a company-managed device. What do you do?

**A4:** You add ABAC. Extend your policy engine to evaluate context attributes (time, device type) in addition to role. The role determines *what* the user can do; the ABAC rules determine *when and from where*. This is a hybrid RBAC-ABAC model and is the most common pattern in production. Do not expand RBAC -- creating a "Nighttime Finance Finance" role for every time window is unsustainable.

**Q5:** Your service has a permission check that runs on every request and takes 8 milliseconds. You serve 20,000 requests per second. What is the daily CPU cost of authorization checks, and what is your optimization?

**A5:** 8 ms x 20,000 rps = 160 seconds of CPU per second. That is 160 CPU cores fully saturated on authorization alone. Optimization: cache the decision locally with a 60-second TTL. At 95 percent cache hit rate, only 5 percent of requests hit the PDP: 8 ms x 1,000 rps = 8 seconds of CPU per second. The remaining 19,000 requests complete in under 1 ms (local cache lookup). Total: roughly 8 seconds of CPU per second -- a 20x reduction.
