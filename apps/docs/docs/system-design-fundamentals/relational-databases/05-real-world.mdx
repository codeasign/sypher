# Relational Databases in Production

The best way to understand a technology is to see how it breaks and how it is fixed at scale. This page collects real stories from companies that pushed relational databases to their limits — and the engineering they built around them.

## Company Case Studies

### Uber — From MySQL to Schemaless to Docstore

Uber started on a single MySQL instance with a standard schema. By 2015, that instance handled 200+ GB of data and the write throughput was approaching its limit. The team tried sharding MySQL manually, but the operational cost of rebalancing shards and the rigidity of schema changes became a bottleneck.

Uber built **Schemaless** — a layer on top of MySQL that stored data as JSON blobs in a single giant table, with secondary indexes stored in separate tables. This gave them the flexibility of a NoSQL database with MySQL's durability. They later evolved this into **Docstore**, a purpose-built distributed database.

The key insight: MySQL's transactional semantics and replication were rock-solid. The problem was not MySQL. The problem was that schema changes on a 200 GB table took hours. By using MySQL as a key-value store with JSON payloads, they decoupled storage from schema without rewriting the storage engine.

### Shopify — MySQL at E-Commerce Scale

Shopify runs one of the largest MySQL fleets in the world: thousands of instances serving hundreds of thousands of stores. Each merchant's data is isolated by shard key (shop ID). The architecture is unusual: Shopify does not use MySQL's built-in replication heavily. Instead, they use **active-active clusters** with a custom proxy layer.

A defining moment came in 2018 when Shopify migrated their core product database from single-master to a multi-shard architecture. The migration took 18 months. They used a dual-write pattern: every write went to both the old database and the new sharded database. A background job compared the two and reconciled differences. Once the comparison showed zero divergence, they cut over.

### Discord — From MongoDB to Cassandra to ScyllaDB

Discord originally stored all messages in MongoDB. As the platform grew to hundreds of millions of messages per day, MongoDB could not keep up. They migrated to Cassandra, which handled the write volume but had issues with read latency (Cassandra partitions data by hash, making range scans over a channel's message history costly).

Discord's final solution was ScyllaDB, a Cassandra-compatible database written in C++ with a focus on consistent low-latency performance. ScyllaDB handles 1 million+ writes per second across their cluster with p99 read latency under 15 milliseconds.

**Lesson**: Relational databases are not the best choice for every high-volume workload. Discord's data model (append-only messages, no joins, no cross-record transactions) is a natural fit for wide-column stores. The migration path — identify the workload type, choose the right tool, migrate incrementally — applies to any architecture decision.

### Cloudflare — PostgreSQL at the Edge

Cloudflare runs a globally distributed PostgreSQL deployment called **D1** (based on SQLite) and uses PostgreSQL internally for control-plane operations. Their approach to multi-region is pragmatic: they accept that cross-region writes are slow and design their application to route around it.

For D1, they use a primary-followers model where writes go to a primary in one region and replicas in other regions serve reads. Writes that must be consistent are forwarded to the primary. Writes that can tolerate eventual consistency go to the nearest replica and propagate asynchronously.

**Lesson**: Not all writes need global consistency. Classify your writes by consistency requirement and route accordingly. This avoids the latency penalty of synchronous cross-region replication while staying within the relational model.

## Open Source Implementations

**PostgreSQL** — The most advanced open-source relational database. Known for extensibility (custom data types, indexing methods, procedural languages), robust MVCC implementation, and strong adherence to SQL standards. Widely considered the default choice for new projects.

**MySQL** — The most deployed open-source relational database. Known for simplicity, broad ecosystem compatibility, and strong read performance under InnoDB. The default for WordPress, many PHP applications, and a large fraction of the web.

**SQLite** — Embedded relational database. No server process. The entire database is a single file on disk. Used in mobile apps, desktop applications, and embedded devices. Not suitable for concurrent write-heavy workloads.

**CockroachDB** — Distributed SQL database built on a key-value store with a PostgreSQL-compatible wire protocol. Supports multi-region deployments with strong consistency and automatic rebalancing. Uses LSM trees instead of B-trees for the storage engine.

**DuckDB** — Embedded analytical database designed for OLAP workloads. Columnar storage, vectorized execution, zero-dependency. Used for data analysis workflows that would traditionally use Pandas but need to handle datasets larger than available memory.

## Cloud Provider Implementations

**Amazon RDS** — Managed relational database supporting PostgreSQL, MySQL, MariaDB, Oracle, and SQL Server. Handles backups, patching, replication, and failover. Multi-AZ deployments provide synchronous replication across availability zones. Read replicas can be promoted to primary in under a minute.

**Amazon Aurora** — MySQL- and PostgreSQL-compatible database with a cloud-native storage layer. Storage is decoupled from compute: you can have 128 TB of storage per instance, and storage autoscales. Aurora replicates data across 3 availability zones with 6 copies. Failover completes in under 30 seconds. Aurora Serverless v2 scales to hundreds of thousands of transactions per second.

**Google Cloud SQL** — Managed PostgreSQL and MySQL. Supports cross-region replication with automatic failover. Cloud SQL's feature set is smaller than RDS — no provisioned IOPS, smaller maximum storage (30 TB vs. Aurora's 128 TB). The integration with Cloud Run and App Engine is tighter than AWS equivalents.

**Azure Database for PostgreSQL** — Managed PostgreSQL with built-in high availability, automated backups, and zone-redundant deployment. Azure's Hyperscale (Citus) option distributes tables across multiple shard nodes for workloads exceeding 100 GB.

## Lessons Learned

### Pattern 1: Start with a Single Instance, Shard Only When Necessary

Every team that migrated from a single instance to shards said the same thing: sharding is painful. It complicates queries, makes migrations harder, and adds operational overhead. Start with a well-tuned single instance with replicas. Shard only when the write throughput exceeds what one server can handle.

### Pattern 2: Connection Pooling Is Not Optional

Every production deployment needs connection pooling. Without it, connection spikes kill the database. With it, you can absorb traffic surges without tuning `max_connections` up to dangerous levels. PgBouncer (PostgreSQL) and ProxySQL (MySQL) are industry standards.

### Pattern 3: Monitor the Right Things

The most useful database metrics, ranked by how often they predict problems: cache hit ratio, replication lag, query latency (p50/p99), deadlock rate, and autovacuum progress. Disk usage and CPU utilization are less useful — a database with 30% CPU and 50% disk can be in trouble if cache hit ratio dropped from 99% to 85%.

### Pattern 4: Automate Schema Migrations

Schema changes are the most common cause of production database incidents. Use a migration tool (Flyway, Alembic, Liquibase) that runs migrations as part of deployment. Test every migration against a copy of production data. Never run a migration by hand on a production database.

### Pattern 5: Plan for the Database to Become the Bottleneck

At every company that grew past a certain size, the database was the last component to be scaled — and the hardest. The application code can be scaled horizontally trivially (add more app servers). The database requires careful planning. Start thinking about read replicas, caching, and sharding before you need them. The cost of retrofitting these patterns under production pressure is 10x the cost of building them proactively.
