# Relational Databases — Challenges

These challenges test your understanding of relational databases at three levels. Foundation challenges take 5 minutes each. Design challenges take 15 minutes each. Expert challenges take 30 minutes each and require synthesizing multiple concepts.

## Tier 1: Foundation

### Challenge 1: Find the Slow Query (5 minutes)

You have an e-commerce database with these tables:

```sql
CREATE TABLE orders (
  id BIGSERIAL PRIMARY KEY,
  customer_id BIGINT NOT NULL,
  status TEXT NOT NULL,
  total NUMERIC(10,2) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE order_items (
  id BIGSERIAL PRIMARY KEY,
  order_id BIGINT NOT NULL REFERENCES orders(id),
  product_name TEXT NOT NULL,
  quantity INT NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL
);
```

The following query runs every 5 minutes for your finance dashboard and takes 12 seconds:

```sql
SELECT customer_id, SUM(total) AS revenue
FROM orders
WHERE created_at >= now() - interval '30 days'
GROUP BY customer_id
ORDER BY revenue DESC
LIMIT 100;
```

What is the root cause? What index or indexes would you add - **Model Answer:** The query scans the entire `created_at` range from the last 30 days with a GROUP BY on `customer_id`. This is a large sequential scan. Add a composite index on `(created_at, customer_id)` INCLUDE (total) to cover the query. The B-tree will filter on `created_at` efficiently (range scan on the first column) and the second column enables an index-only scan for the GROUP BY. If this query runs against a read replica and the underlying table has many recent orders, also consider a materialized view that pre-aggregates daily revenue and is refreshed every 5 minutes.

### Challenge 2: Detect the Deadlock (5 minutes)

Two transactions run concurrently:

```
Transaction A:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

Transaction B:
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;
UPDATE accounts SET balance = balance + 50 WHERE id = 1;
COMMIT;
```

Under READ COMMITTED isolation, one of these transactions will be chosen as the deadlock victim and aborted. Explain exactly why the deadlock occurs and how to restructure the code to prevent it.

**Model Answer:** Transaction A locks row 1, then requests row 2. Transaction B locks row 2, then requests row 1. Each holds a lock the other needs. The database's deadlock detector runs (usually every second) and detects the cycle. It aborts one transaction (typically the one that accumulated the least work).

To prevent this, always acquire locks in a consistent order: always update account 1 before account 2. Both transactions should update accounts in ascending ID order. This guarantees they never wait on each other. If account IDs are dynamic, sort them before acquiring locks.

### Challenge 3: Identify the Missing Index (5 minutes)

Your team is investigating a slow API endpoint. The raw SQL is:

```sql
SELECT * FROM products
WHERE category = 'Electronics'
  AND price BETWEEN 50 AND 200
ORDER BY rating DESC
LIMIT 20;
```

The products table has 5 million rows. There is an index on `category` alone. The query takes 800 ms. After adding a composite index on `(category, price, rating DESC)`, the query takes 3 ms. Explain in detail why the new index made such a dramatic difference.

**Model Answer:** The original index on `category` returns 1.2 million matching rows for 'Electronics' (a 24% selectivity). The database must then fetch each of those 1.2 million rows from the heap to check the `price` filter and sort by `rating`. This is 1.2 million random page reads plus a sort of 1.2 million rows — expensive.

The new composite index `(category, price, rating DESC)` works in three stages. First, the B-tree narrows to the 'Electronics' subtree. Second, within that subtree, the index is sorted by `price` so the range `BETWEEN 50 AND 200` is a contiguous range scan. Third, within that price range, the index is sorted by `rating DESC` so the `ORDER BY rating DESC LIMIT 20` can read the first 20 entries from the leaf and stop — no sort step needed, no heap fetches (if all needed columns are in the index). This is an **index-only scan** with a late-row-removal cost of zero.

## Tier 2: Design

### Challenge 4: Design a Hotel Reservation System (15 minutes)

Design the database schema and query strategy for a hotel booking system. Requirements:

- Hotels have multiple room types (single, double, suite).
- Each room type has a fixed number of identical rooms.
- A reservation is for a specific room type and date range (check-in to check-out).
- No overbooking: the system must guarantee that two reservations for the same room type do not overlap.
- The system must handle 10,000 bookings per day.

Design the schema, the reservation insertion query, the strategy to prevent overbooking, and the indexing approach.

**Model Answer:**

Core schema:
```sql
CREATE TABLE room_types (
  id SERIAL PRIMARY KEY,
  hotel_id INT NOT NULL REFERENCES hotels(id),
  name TEXT NOT NULL, -- 'Single', 'Double', 'Suite'
  total_rooms INT NOT NULL,
  nightly_rate NUMERIC(10,2) NOT NULL
);

CREATE TABLE reservations (
  id BIGSERIAL PRIMARY KEY,
  room_type_id INT NOT NULL REFERENCES room_types(id),
  guest_name TEXT NOT NULL,
  check_in DATE NOT NULL,
  check_out DATE NOT NULL,
  status TEXT NOT NULL DEFAULT 'confirmed',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  -- Prevent reservation on the same room type for overlapping dates:
  -- application-level check (see below)
);

CREATE INDEX idx_reservations_room_dates
  ON reservations(room_type_id, check_in, check_out);
```

The critical query — check availability before inserting:
```sql
-- Count total booked rooms for a given room type on an overlapping date range
SELECT COUNT(*)
FROM reservations
WHERE room_type_id = $1
  AND status = 'confirmed'
  AND check_in < $3  -- existing check_in < new check_out
  AND check_out > $2; -- existing check_out > new check_in
```

If this count is less than `total_rooms` for that room type, the booking is allowed. To prevent race conditions, use SERIALIZABLE isolation or a row-level lock on the room_type row. In practice, use `SELECT ... FOR UPDATE` on the `room_types` row before the check, which serializes bookings for that room type. This keeps the critical section small and does not lock the entire reservations table.

### Challenge 5: Migration from a Monolith to Sharded Database (15 minutes)

Your social media startup has a single PostgreSQL database with 500 GB of data. The primary table `posts` has 2 billion rows and grows by 10 million rows per day. Write throughput on the primary is approaching the disk IO limit. Design a plan to migrate to a sharded architecture with zero downtime.

**Model Answer:**

Phase 1 (preparation, 2 weeks): Choose the shard key. For a social media app, shard by `user_id` (or `author_id` for posts). This colocates all posts by the same user on one shard, making "show this user's feed" queries single-shard. Use a consistent hash function across a fixed number of virtual shards (e.g., 1024) mapped to physical shards, so rebalancing only moves virtual shards, not individual rows.

Phase 2 (dual-write, 4 weeks): Deploy a shard proxy layer in the application. Every write goes to both the old database and the new sharded database. Reads still come from the old database. Run a background reconciliation job that compares rows between old and new databases and reports discrepancies. Fix any divergence found.

Phase 3 (read cutover, 1 week): Switch reads to the new sharded database. Keep dual-writes running. Monitor query latency, error rates, and replication. Have a rollback plan: switch reads back to the old database if latency spikes or errors appear.

Phase 4 (write cutover, 1 week): Stop writes to the old database. Run one final reconciliation to confirm zero divergence. Decommission the old database. Roll back the application's dual-write logic.

Key risks: cross-shard queries (queries that need data from multiple shards) will be slow or impossible. Identify these queries in Phase 1 and decide: (a) route them to all shards and merge results, (b) redesign the feature to avoid them, or (c) maintain a separate aggregated table.

### Challenge 6: Diagnose an Autovacuum Storm (15 minutes)

Your PostgreSQL database slows to a crawl every afternoon. CPU is at 95%, IO wait is at 40%, and query latency spikes from 5 ms to 800 ms. You look at `pg_stat_activity` and see many autovacuum processes running. Explain the root cause, how to confirm it, and how to fix it.

**Model Answer:**

Root cause: A table with high write throughput (possibly a queue table where rows are INSERTed and then DELETEd, or a frequently updated table) accumulates dead tuples faster than autovacuum can clean them. The default autovacuum settings (scale_factor = 0.2, threshold = 50) mean autovacuum does not fire until 20% of rows are dead. On a 50 million row table, that is 10 million dead tuples before a vacuum cycle starts. When it finally fires, it must scan and clean 10 million dead tuples, consuming CPU and IO.

To confirm: run `SELECT n_dead_tup, n_live_tup, last_autovacuum FROM pg_stat_user_tables WHERE relname = 'your_table';`. If `n_dead_tup` is in the millions and `last_autovacuum` is null or hours ago, the table is starving for vacuum.

Fix: Tune autovacuum for this specific table:
```sql
ALTER TABLE your_table SET (
  autovacuum_vacuum_scale_factor = 0.01,
  autovacuum_vacuum_threshold = 1000,
  autovacuum_vacuum_cost_limit = 1000
);
```

This triggers vacuum when 1% of rows are dead (far sooner) and allows higher IO consumption during vacuum (cost_limit = 1000 instead of the default 200). Also consider: if the table is a queue where rows are processed and deleted, switch to a partitioning scheme where you drop entire partitions instead of deleting individual rows.

## Tier 3: Expert

### Challenge 7: Design a Globally Consistent Ledger (30 minutes)

You are building a financial ledger system that must run in three regions (US, EU, Asia). Every transaction must be globally consistent: if a user transfers money from their US account to their EU account, both sides must see the same balance at all times. The system must handle 50,000 transactions per second globally. Individual transactions cannot fail due to network partitions between regions.

Design the database architecture, including: the data model, the transaction protocol (commit strategy), the consistency model, the conflict resolution strategy, and the failover approach.

**Discussion:**

The core tension here is the CAP theorem applied across regions. Strong global consistency requires serializing writes through a single leader or using a consensus protocol like Raft across all regions. Both approaches add cross-region latency to every write (typically 100-300 ms round trip for a US-to-EU commit). You must decide whether the business actually needs global strong consistency or whether regional consistency with eventual global consistency suffices.

A realistic approach: use a single logical database across all regions with CockroachDB or Spanner, which both use Raft-based consensus. Each transaction commits when a majority of replicas (across regions) acknowledge it. Read requests are served from the local replica if they accept a bounded staleness guarantee, or from the global leader for strongly consistent reads.

For the data model, use an append-only ledger (no UPDATE, only INSERT) with idempotency keys to prevent duplicate transactions. Each entry has: transaction_id (UUID), account_id, amount, timestamp, region_of_origin. Balances are computed as rolling sums via materialized views or periodic aggregation.

The failover strategy: if one region becomes unavailable, transactions continue as long as a majority of replicas remain reachable in the other two regions. When the failed region recovers, it catches up via the Raft log. This requires at least 3 regions and 5 replicas total (2 in the primary region, 2 in the secondary region, 1 in the third).

Latency impact: every write incurs the round-trip time to the nearest replica in another region. For US-to-EU writes, expect 120-180 ms per transaction. The throughput of 50,000 tps is achievable with batching and pipelining (each batch of transactions uses one consensus round).

### Challenge 8: Build a Real-Time Analytics Pipeline on Relational Data (30 minutes)

Your company has a PostgreSQL database serving a web application. The operations team wants a real-time dashboard showing: orders per minute, revenue per hour, top-selling products, and user signup trends. The dashboard must be under 5 seconds stale. Running these queries directly against the production database is not acceptable because they would interfere with the OLTP workload.

Design a system that extracts data from the production database, transforms it, and serves the dashboard. Constraints: you cannot use a third-party SaaS tool. The system must be built on open-source components and PostgreSQL-compatible tools.

**Discussion:**

The solution has four layers: capture, transport, transform, and serve.

Capture layer: Use Debezium to read the WAL stream from the production database. This captures every INSERT, UPDATE, and DELETE as it happens, with minimal overhead (reading the WAL is far cheaper than running queries).

Transport layer: Push the CDC events to Kafka. Kafka absorbs traffic spikes and decouples the producer (Debezium) from the consumer (the transform layer).

Transform layer: Use Kafka Streams or a lightweight SQL engine (ksqlDB or Materialize) to aggregate the raw CDC events into minute-level and hour-level aggregates. For example, count orders per minute, compute rolling revenue sums, and maintain a top-products ranking. The transform layer maintains materialized state that is continuously updated as new events arrive.

Serve layer: Write the aggregates to a separate PostgreSQL instance or to PostgreSQL's native materialized views refreshed continuously (via a small application that applies Kafka events). Materialize is ideal here: it accepts a SQL query and maintains the result as a materialized view updated in real-time. The dashboard queries this materialized view.
