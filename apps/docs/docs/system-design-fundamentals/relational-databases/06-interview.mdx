# Relational Databases --?" Interview Playbook

Relational databases appear in nearly every system design interview. You will not be asked to implement a B-tree from scratch, but you will be expected to explain trade-offs, justify your schema decisions, and identify bottlenecks in a proposed design.

## How This Topic Appears

System design questions that involve databases fall into eight categories. Here is each one with the typical phrasing you will hear.

### Schema Design
- "Design the database schema for a ride-sharing service."
- "How would you model a social media platform's data?"

### Index Strategy
- "Your query is slow --?" how do you diagnose and fix it?"
- "Design an indexing strategy for an e-commerce product catalog."

### Scaling Writes
- "Your database write throughput is maxed out at 10,000 writes per second. How do you scale beyond that?"
- "Design a system that handles 100,000 order placements per minute."

### Replication and High Availability
- "How do you ensure zero downtime during a database migration?"
- "Design a multi-region deployment for a globally distributed application."

### Consistency and Transactions
- "What isolation level would you choose for a banking application?"
- "How do you handle concurrent updates to the same row?"

### Query Performance
- "A query is taking 10 seconds. Walk through your debugging process."
- "How would you optimize a paginated API endpoint backed by a database?"

### Migration and Schema Changes
- "How do you add a NOT NULL column to a 500 GB table without downtime?"
- "Design a zero-downtime migration from a single database to a sharded setup."

### Comparison and Trade-offs
- "PostgreSQL vs. MongoDB for a real-time chat application?"
- "When would you choose a relational database over a document store?"

## What the Interviewer Is Actually Testing

When an interviewer asks about databases, they are evaluating five things:

**Depth of understanding** --?" Do you know how indexes actually work (B-tree structure, not just "it makes things faster")? Can you explain MVCC without hand-waving?

**Trade-off awareness** --?" Do you only list advantages, or do you name the costs? Saying "indexes speed up reads" without mentioning write overhead signals junior-level thinking.

**Operational experience** --?" Have you actually run a database in production? Mentioning connection pooling, replication lag monitoring, or autovacuum tuning separates real experience from textbook knowledge.

**Design judgment** --?" Can you choose the right isolation level, index strategy, or shard key based on the specific constraints of the problem?

**Communication clarity** --?" Can you explain a complex concept (like why REPEATABLE READ does not prevent phantom reads) in plain language?

## Level-by-Level Expectations

### L3/L4 (Entry / Mid-level)

You should be able to:
- Write correct SQL queries with JOINs, GROUP BY, and basic window functions.
- Create a normalized schema for a simple domain (e-commerce, blog, chat).
- Explain primary keys, foreign keys, indexes, and ACID in your own words.
- Identify a slow query and suggest adding an index.
- Describe the difference between Read Committed and Serializable isolation.

**Sample answer (L4):** "For this e-commerce schema, I would put orders and order_items in separate tables with a foreign key. This is 3NF: the order total is computed from line items rather than stored redundantly. I would index the customer_id column in orders because the most common query is 'show my recent orders'."

### L5 (Senior)

Adds to L3/L4:
- Design a sharding strategy and explain the trade-offs of different shard keys.
- Diagnose a production incident involving replication lag or connection exhaustion.
- Plan a zero-downtime migration with dual-writes.
- Compare optimistic vs. pessimistic concurrency control with specific examples.
- Tune PostgreSQL configuration for a write-heavy workload.

**Sample answer (L5):** "I would shard by customer_id hash to distribute writes evenly. The cost is that queries spanning all customers --?" like a daily revenue report --?" must scatter-gather across all shards. I would solve that by routing reporting queries to a separate read replica that aggregates from all shards using Foreign Data Wrappers or a CDC pipeline."

### L6 (Staff)

Adds to L5:
- Design a multi-region active-active database architecture with conflict resolution.
- Evaluate and recommend a database technology stack for the next 3-5 years of company growth.
- Design a data pipeline that guarantees exactly-once semantics from application to database to analytics.
- Identify and prevent systemic failure modes like autovacuum storms or checkpoint storms.

**Sample answer (L6):** "At our scale, the primary bottleneck is WAL write throughput on the leader. I would architect a CDC-based architecture where the primary database captures changes via Debezium into Kafka, and downstream systems --?" search index, cache, analytics --?" consume from Kafka. This decouples read scalability from write throughput. The write path handles 50,000 tps on a single primary with synchronous replication to two followers for durability."

### L7/L8 (Principal / Director)

Adds to L6:
- Define an organization-wide data strategy that spans multiple teams and data stores.
- Evaluate build vs. buy decisions for managed database services.
- Design a data governance framework that balances developer velocity with compliance requirements.
- Lead the technical strategy for a major migration (e.g., Oracle to PostgreSQL across 500 services).

**Sample answer (L7):** "Over the next three years, our data volume will grow from 50 TB to 500 TB. I would standardize on PostgreSQL with Aurora as the managed layer, use RDS Proxy for connection management, and introduce Vitess for sharding when any single shard exceeds 10 TB. The migration plan phases over six quarters: first establish CDC pipelines, then dual-write non-critical services, then cut over service by service with rollback capability at each step."

## FAANG Framework for Answering

When a database question comes up, use this six-step structure. It ensures you cover the right ground without rambling.

### Step 1: Clarify Constraints

Ask questions to narrow scope:
- "Is this a read-heavy or write-heavy workload?"
- "What is the data size today and projected in 2 years?"
- "Is consistency more important than availability, or vice versa?"
- "Do you need to support ad-hoc analytical queries?"

### Step 2: Define the Data Model

Sketch the schema:
- "I would use these tables with these columns and these relationships."
- "Here is my primary key choice and why."
- "This is my normalization strategy."

### Step 3: Identify Access Patterns

List the main queries:
- "The three most common queries are X, Y, and Z."
- "For query X, I would create a composite index on columns A and B."
- "Query Y is a range scan, so a B-tree index on the timestamp column is appropriate."

### Step 4: Choose the Database

Justify relational over alternatives:
- "I am choosing PostgreSQL because we need joins, transactions, and referential integrity."
- "If the write volume exceeds 50,000 per second, I would shard with Vitess or move to CockroachDB."

### Step 5: Address Scale

Describe how the design holds up as data grows:
- "At 10 GB, a single instance with replicas works."
- "At 1 TB, I would add read replicas and a cache layer."
- "At 10 TB, I would shard by customer ID and partition large tables by month."

### Step 6: Identify Risks

Call out failure modes and mitigations:
- "The risk here is replication lag causing stale reads on the reporting dashboard. I would add a staleness check --?" if lag exceeds 5 seconds, route the query to the primary."
- "Connection pool exhaustion during traffic spikes is a risk. PgBouncer with a 200-connection hard limit mitigates this."

## Phrase Bank

Use these phrases to sound confident and precise:

- "I would start with a normalized schema and denormalize selectively based on query profiling."
- "The B-tree index gives us O(log N) lookup time with the leaf nodes linked for efficient range scans."
- "MVCC ensures that readers never block writers and writers never block readers."
- "The write-ahead log is the source of truth for crash recovery. Data files are a cache of the WAL."
- "I would use a composite index on (status, created_at) to cover the most common query pattern."
- "Connection pooling is critical at scale to avoid the overhead of TCP handshake and backend fork per connection."
- "Sharding by hash of the primary key distributes writes evenly but scatters multi-shard queries."
- "Synchronous replication guarantees zero data loss but adds the round-trip time to the replica as write latency."
- "I would design for eventual consistency on read replicas and route writes that need immediate consistency to the primary."
- "Autovacuum in PostgreSQL needs careful tuning on write-heavy tables to prevent bloat."
- "The trade-off is between index coverage for fast reads and index maintenance overhead for writes."
- "I would use a database migration tool with version-controlled, repeatable migrations tested against a staging environment."
- "Point-in-time recovery requires continuous WAL archiving. Without it, you can only recover to the last full backup."
- "The buffer pool hit ratio is the single most important performance metric for a relational database."
- "For a zero-downtime schema migration, I would use a phased approach: add the new column as nullable, backfill data in batches, then add the NOT NULL constraint."
- "I would monitor query latency at the p50, p99, and p999 levels --?" not just the average, which hides tail latency."
- "The planner uses table statistics to estimate row counts. Stale statistics cause bad query plans."

## Red Flags

### Red Flag 1: "I would just add an index"

Without explaining which column, what type of index, and what trade-off. This signals surface-level knowledge. Always name the index type (B-tree, hash, GIN, BRIN), the column(s), and the cost (slower writes, more storage).

### Red Flag 2: "I would use a NoSQL database"

Without explaining why a relational database does not fit. If the problem involves joins, transactions, or ad-hoc queries, NoSQL is the wrong answer. If it involves high write throughput with no joins, NoSQL might be right --?" but you must articulate why.

### Red Flag 3: Confusing replication lag with durability

Saying "with synchronous replication, the replica has the same data as the primary" is correct. Saying "asynchronous replication guarantees no data loss" is dangerously wrong. Be precise about which guarantees apply.

### Red Flag 4: Assuming the database handles everything

Designs that expect the database to handle full-text search, caching, message queuing, and reporting in one instance. A relational database is not a Swiss Army knife. Layer specialized tools as needed.

### Red Flag 5: Ignoring operational concerns

Saying "I would create this index" without mentioning how to create it on a production table without downtime. Say "CREATE INDEX CONCURRENTLY" for PostgreSQL. Mentioning the operational detail shows experience.

### Red Flag 6: Over-normalizing or under-normalizing

A schema in 5NF with 30 tables for a to-do list app is over-engineered. A single table with 50 nullable columns for an e-commerce site is under-normalized. Show judgment about where to land on the normalization spectrum.

### Red Flag 7: No mention of monitoring

Describing a production architecture without any mention of how you know it is working. Mention replication lag, query latency, and cache hit ratio monitoring.

### Red Flag 8: Vague sizing

Saying "add more replicas" without estimating how many. Estimate based on read QPS and per-server capacity. Saying "3 replicas for a workload of 10,000 reads per second with each replica handling 5,000" shows you have done the math.

## Practice Drills

### Drill 1: Schema Design (5 minutes)

Design the schema for a food delivery app with restaurants, menus, orders, and delivery tracking. Normalize to 3NF. Then denormalize one part to optimize the most common query.

**Self-check:** Does your schema allow a restaurant to have multiple menu categories? Can an order have items from multiple restaurants?

### Drill 2: Query Tuning (5 minutes)

Given a table `orders(id, customer_id, status, total, created_at)` with 50 million rows, explain how you would optimize this query:
```sql
SELECT COUNT(*) FROM orders
WHERE status = 'pending' AND created_at > now() - interval '7 days';
```

**Self-check:** Did you suggest a composite index on (status, created_at)? Did you mention the trade-off --?" the index is large because status is low-cardinality?

### Drill 3: Scaling Writes (15 minutes)

Design a system that handles 100,000 new user registrations per hour. Each registration inserts into three tables. Walk through the database architecture from single instance to final sharded design.

**Self-check:** Did you estimate the write throughput (28 inserts/second, easily handled by one instance)? Did you identify the real bottleneck (email uniqueness check on a high-write column)?

### Drill 4: Migration Plan (15 minutes)

You need to rename the `email` column to `primary_email` in a 200 GB table with 500 million rows. The application cannot have downtime. Walk through the complete migration plan.

**Self-check:** Did you mention dual-reading both column names in the application layer first? Did you explain how to backfill in batches? Did you test the rollback plan?

### Drill 5: Multi-Region (15 minutes)

Design a multi-region database architecture for a globally distributed social network. Users in Europe, Asia, and North America should get under 50 ms write latency. Writes made in one region must eventually be visible in all regions.

**Self-check:** Did you identify that synchronous cross-region replication would exceed the 50 ms latency target? Did you propose active-active with conflict resolution or active-passive with local writes and async propagation?
