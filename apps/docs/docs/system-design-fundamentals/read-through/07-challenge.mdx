## Tier 1: Implementation

### Challenge 1.1: Implement a Basic Read-Through Cache

Build a minimal in-process read-through cache with the following API: `get(key)`, `set(key, value, ttl)`, and `delete(key)`. The cache must support a loader function that is called on cache miss.

**Requirements**:
- O(1) get and set operations
- TTL-based expiration (entries are removed after TTL)
- Loader function that returns a Promise (async loading)
- Concurrent miss deduplication
- Maximum cache size with LRU eviction

**Starter code**:

```javascript
class ReadThroughCache {
  constructor(options) {
    // options: { loader, maxSize, defaultTTL }
  }

  async get(key) {
    // Return cached value or load via loader
  }

  set(key, value, ttl) {
    // Store value with TTL
  }

  delete(key) {
    // Remove from cache
  }
}
```

**Model answer**:

```javascript
class ReadThroughCache {
  constructor({ loader, maxSize = 1000, defaultTTL = 300 }) {
    this.loader = loader;
    this.maxSize = maxSize;
    this.defaultTTL = defaultTTL;
    this.cache = new Map();
    this.inFlight = new Map();  // dedup gate
    this.lruKeys = [];
  }

  async get(key) {
    const entry = this.cache.get(key);

    // Cache hit — check TTL
    if (entry && Date.now() - entry.createdAt < entry.ttl * 1000) {
      this._updateLRU(key);
      return entry.value;
    }

    // Miss or expired — check dedup
    if (this.inFlight.has(key)) {
      // Another request is loading this key
      return this.inFlight.get(key).promise;
    }

    // First miss — start loading
    let resolve, reject;
    const promise = new Promise((res, rej) => {
      resolve = res;
      reject = rej;
    });

    this.inFlight.set(key, { promise, resolve, reject });

    try {
      const value = await this.loader(key);
      this.set(key, value, this.defaultTTL);
      resolve(value);
      return value;
    } catch (err) {
      reject(err);
      throw err;
    } finally {
      this.inFlight.delete(key);
    }
  }

  set(key, value, ttl) {
    if (this.cache.has(key)) {
      this._removeLRU(key);
    }

    // Evict if at capacity
    while (this.cache.size >= this.maxSize) {
      const lruKey = this.lruKeys.shift();
      this.cache.delete(lruKey);
    }

    this.cache.set(key, {
      value,
      ttl: ttl || this.defaultTTL,
      createdAt: Date.now()
    });
    this.lruKeys.push(key);
  }

  delete(key) {
    this.cache.delete(key);
    this._removeLRU(key);
  }

  _updateLRU(key) {
    this._removeLRU(key);
    this.lruKeys.push(key);
  }

  _removeLRU(key) {
    const index = this.lruKeys.indexOf(key);
    if (index > -1) {
      this.lruKeys.splice(index, 1);
    }
  }
}

// Usage
const cache = new ReadThroughCache({
  loader: async (key) => {
    const id = key.split(':')[1];
    const response = await fetch(`/api/users/${id}`);
    return response.json();
  },
  maxSize: 5000,
  defaultTTL: 300
});

const user = await cache.get('user:42');
```

### Challenge 1.2: Add Stale-While-Revalidate

Extend the basic read-through cache from Challenge 1.1 with stale-while-revalidate behavior. When a cache entry is expired but within the stale window, return the stale value immediately and asynchronously refresh the cache in the background.

**Requirements**:
- Add a `staleWhileRevalidate` option and a `staleMaxAge` option
- When an entry is past its TTL but within staleMaxAge, return the stale value
- Trigger an async background refresh when stale data is served
- Do not block the response on the background refresh

**Model answer**:

```javascript
class ReadThroughCacheWithStale extends ReadThroughCache {
  constructor(options) {
    super(options);
    this.staleWhileRevalidate = options.staleWhileRevalidate || false;
    this.staleMaxAge = options.staleMaxAge || 3600; // 1 hour
  }

  async get(key) {
    const entry = this.cache.get(key);

    if (!entry) {
      // No entry at all — standard miss
      return super.get(key);
    }

    const age = (Date.now() - entry.createdAt) / 1000;
    const isExpired = age >= entry.ttl;
    const isWithinStaleWindow = age < entry.staleMaxAge || entry.staleMaxAge === 0;

    if (!isExpired) {
      // Fresh — standard hit
      this._updateLRU(key);
      return entry.value;
    }

    if (isExpired && this.staleWhileRevalidate && isWithinStaleWindow) {
      // Serve stale, trigger background refresh
      this._triggerRefresh(key);
      return entry.value;
    }

    // Expired and not in stale window — standard miss
    return super.get(key);
  }

  async _triggerRefresh(key) {
    if (this.inFlight.has(key)) return; // Already refreshing
    try {
      let resolve;
      const promise = new Promise((r) => { resolve = r; });
      this.inFlight.set(key, { promise, resolve });

      const value = await this.loader(key);
      this.set(key, value, this.defaultTTL);
      resolve(value);
    } catch (err) {
      // Silently log — the stale data is already served
      console.error('Background refresh failed for', key, err);
    } finally {
      this.inFlight.delete(key);
    }
  }
}
```

### Challenge 1.3: Add TTL Jitter

Extend the cache to add random jitter to TTL values. When storing a cache entry, add a random offset to the TTL so entries expire at different times. This prevents cache stampedes when many keys are loaded simultaneously with the same TTL.

**Requirements**:
- Add a `ttlJitter` option (percentage of TTL, e.g., 0.1 means +-10%)
- Apply jitter to every `set()` call
- Jitter must be deterministic per key (same jitter for the same key across restarts)

**Model answer**:

```javascript
class ReadThroughCacheWithJitter extends ReadThroughCache {
  constructor(options) {
    super(options);
    this.ttlJitter = options.ttlJitter || 0; // 0 to 1
  }

  // Simple hash function for deterministic jitter
  _hashKey(key) {
    let hash = 0;
    for (let i = 0; i < key.length; i++) {
      const char = key.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash; // Convert to 32-bit integer
    }
    return Math.abs(hash);
  }

  set(key, value, ttl) {
    const baseTtl = ttl || this.defaultTTL;

    if (this.ttlJitter > 0) {
      const hash = this._hashKey(key);
      const normalizedHash = hash / 2147483647; // 0 to 1
      const jitterAmount = baseTtl * this.ttlJitter * normalizedHash;
      const jitteredTtl = baseTtl + jitterAmount;
      super.set(key, value, Math.floor(jitteredTtl));
    } else {
      super.set(key, value, baseTtl);
    }
  }
}
```

## Tier 2: Design

### Challenge 2.1: Design a Read-Through Cache for a Real-Time Leaderboard

Design a read-through cache for a gaming leaderboard that updates every 60 seconds. The leaderboard is read by millions of users but computed by a batch job that runs every 60 seconds.

**Requirements**:
- 10 million users view the leaderboard
- Leaderboard is recomputed every 60 seconds
- Serving stale data is acceptable (up to 30 seconds stale)
- Peak QPS: 50,000 reads/second

**Model answer**:

**Architecture**:
- Use a read-through cache with a 60-second TTL.
- The loader computes the leaderboard by querying the game database and sorting by score.
- Use stale-while-revalidate with a 30-second stale window.
- The batch job writes the computed leaderboard to a staging table. The loader reads from this staging table.

**Key decisions**:
- TTL of 60 seconds matches the compute interval.
- Stale-while-revalidate ensures users never see an error — they get the previous leaderboard while the new one is computed.
- The batch job approach is better than computing on every miss because the computation is expensive.
- Cache stampede is unlikely because the batch job completes in under 10 seconds, and the loader simply reads the staging table — no heavy computation on miss.

**Failure mode**: if the batch job fails, the loader returns stale data from the last successful computation. The cache continues to serve this data until the stale window expires. After 90 seconds (60s TTL + 30s stale window), the loader attempts to compute — if it fails, the circuit breaker opens and the cache returns an error.

### Challenge 2.2: Design a Read-Through Cache for a Social Media Feed

Design a read-through cache that generates and caches personalized social media feeds. Each user's feed is a list of posts from people they follow, ordered by recency and relevance.

**Requirements**:
- 100 million users, 10 million daily active
- Each feed is personalized (different for each user)
- Feed generation is expensive: fan-out query to 300 followed users
- Reads: 1,000 feed requests per second at peak
- TTL: 5 minutes (user expects fresh content)

**Model answer**:

**Architecture**:
- Read-through cache with user ID as the key, the feed post list as the value.
- Loader function executes the fan-out query. This is an expensive operation (300 user queries aggregated).
- Refresh-ahead: the cache proactively refreshes feeds for the most active users (top 1%) before their TTL expires. This eliminates the miss penalty for the highest-traffic users.
- Batch loader: when multiple users' feeds expire simultaneously (e.g., after a deployment), batch the feed generation. Process 100 feeds at a time with a fan-out query that covers all 100 users' followed sets.

**Key decisions**:
- TTL of 5 minutes balances freshness and cache efficiency.
- Refresh-ahead for the top 1% of users covers 80% of feed requests.
- The batch loader reduces database load during cold starts — loading 100 feeds in a batch is faster than loading 100 feeds individually.

**Failure mode**: if the feed generation is slow (over 2 seconds), the circuit breaker opens and the cache serves stale feeds. Users see slightly outdated content but the application remains available.

### Challenge 2.3: Design a Read-Through Cache with Write-Through for an E-Commerce Product Catalog

Design a caching layer for an e-commerce product catalog. Products are read frequently (100,000 reads/second) and updated occasionally by the admin team (10 writes/second).

**Requirements**:
- Product data must be consistent within 10 seconds of an admin update
- Read latency must be under 10ms (p99)
- The product catalog has 1 million products, each 5KB

**Model answer**:

**Architecture**:
- Read-through cache for reads (handles the 100K QPS reads).
- Write-through cache for writes (admin updates go to cache and database atomically).
- Cache invalidation on write: when a product is updated, the write-through path updates the cache entry immediately. The next read returns the fresh data.
- TTL of 60 seconds as a safety net — if the write-through path fails to update the cache, the TTL ensures fresh data is loaded within 60 seconds.

**Key decisions**:
- 10-second consistency requirement means TTL-based invalidation alone is not sufficient — the write-through path is necessary.
- Write-through adds write latency (10ms for cache + 10ms for DB = 20ms total), but write QPS is only 10/second, so this is acceptable.
- Cache size: 1 million products x 5KB = 5GB. With 3x replication: 15GB. A 4-node Redis cluster at 8GB each is sufficient.
- Read QPS = 100K. Each Redis node handles 50K QPS (after replication). Two read replicas per shard handle the load.

## Tier 3: Expert

### Challenge 3.1: Adaptive TTL Based on Access Patterns

Design a read-through cache that dynamically adjusts TTL based on observed access patterns. Frequently accessed keys should get longer TTLs (reducing miss rate). Infrequently accessed keys should get shorter TTLs (reducing memory waste).

**Requirements**:
- Track access frequency per key
- Dynamically compute TTL: `TTL = baseTTL * (1 + frequencyFactor)`
- Maximum TTL cap of 24 hours
- Minimum TTL floor of 60 seconds
- Reset frequency tracking when an entry is reloaded

**Model answer**:

**Architecture**:

```javascript
class AdaptiveTTLCache extends ReadThroughCache {
  constructor(options) {
    super(options);
    this.baseTTL = options.baseTTL || 300;
    this.maxTTL = options.maxTTL || 86400;   // 24 hours
    this.minTTL = options.minTTL || 60;       // 1 minute
    this.frequencyFactor = options.frequencyFactor || 0.5;
    this.frequency = new Map();   // key -> { count, windowStart }
    this.decayInterval = 60000;   // decay frequency every 60s

    // Decay frequency counts periodically
    setInterval(() => this._decayFrequencies(), this.decayInterval);
  }

  get(key) {
    // Record access
    const freq = this.frequency.get(key);
    if (freq) {
      freq.count++;
    } else {
      this.frequency.set(key, { count: 1, windowStart: Date.now() });
    }

    return super.get(key);
  }

  computeAdaptiveTTL(key) {
    const freq = this.frequency.get(key);
    if (!freq || freq.count === 0) return this.baseTTL;

    // Normalize frequency: count divided by time window (in minutes)
    const windowMinutes = Math.max(1, (Date.now() - freq.windowStart) / 60000);
    const accessesPerMinute = freq.count / windowMinutes;

    // Compute TTL: baseTTL * (1 + frequencyFactor * normalizedAccesses)
    const normalizedAccesses = Math.min(accessesPerMinute / 10, 1); // cap at 10/min
    const ttl = Math.floor(
      this.baseTTL * (1 + this.frequencyFactor * normalizedAccesses)
    );

    return Math.max(this.minTTL, Math.min(this.maxTTL, ttl));
  }

  _decayFrequencies() {
    for (const [key, freq] of this.frequency) {
      // Halve the count every decay period (exponential decay)
      freq.count = Math.floor(freq.count / 2);
      if (freq.count === 0) {
        this.frequency.delete(key);
      }
    }
  }
}
```

**Key insight**: adaptive TTL is most beneficial for caches with a long-tail key distribution. The top 1% of keys get the longest TTLs (near 24 hours), while the long-tail keys get shorter TTLs (5-60 minutes). This maximizes hit ratio for the keys that matter most while minimizing memory waste.

### Challenge 3.2: Distributed Read-Through Cache with Consistent Hashing

Design a distributed read-through cache that uses consistent hashing for key distribution across multiple nodes. The cache must support adding and removing nodes with minimal key redistribution.

**Requirements**:
- N cache nodes with consistent hashing
- Virtual nodes (160 per physical node) for balanced distribution
- Each node handles its own read-through logic (loader queries the database directly)
- Graceful handling of node additions and removals

**Model answer**:

**Architecture**:

```
Hash ring with 3 physical nodes x 160 virtual nodes = 480 points on the ring:

                        Node 1 (160 vnodes)
                        ┌────────────────┐
                        │  vnode_1_1     │
                        │  vnode_1_2     │
                        │  ...           │
                        │  vnode_1_160   │
                        └────────────────┘
                               ▲
                               │
        ┌──────────────────────┼──────────────────────┐
        │                      │                      │
  ┌─────┴──────┐        ┌─────┴──────┐        ┌─────┴──────┐
  │  vnode_3_1 │        │  vnode_2_1 │        │  vnode_1_1 │
  │  ...       │        │  ...       │        │  ...       │
  │  vnode_3_160│        │  vnode_2_160│        │  vnode_1_160│
  └────────────┘        └────────────┘        └────────────┘
  Node 3 (160 vnodes)    Node 2 (160 vnodes)    Node 1 (160 vnodes)
```

**Key design decisions**:
- Each cache node runs its own read-through cache (in-process or local Redis). The node is the owner of its keys.
- The consistent hashing ring determines which node handles a given key: `targetNode = hashRing.getNode(hash(key))`.
- Virtual nodes ensure balanced distribution when physical nodes have different capacities.
- On node addition, the new node takes over keys from its neighbors on the ring. These keys become cold on the new node — they must be reloaded from the database.
- On node removal, the neighboring node inherits the keys. If the removal is planned, the keys should be transferred to the new owner before the node is removed.

**Failure mode**: if a cache node fails, its keys are reassigned to the next node on the ring. Those keys are cold on the new node — every read triggers a miss and a database query. The database must handle the additional load. Pre-warming: before removing a node, pre-load its keys onto the neighboring node.

**Code structure for routing**:

```javascript
class ConsistentHashRing {
  constructor(nodes, virtualNodes = 160) {
    this.ring = new Map();  // hash -> nodeId
    this.sortedHashes = [];
    for (const node of nodes) {
      this.addNode(node, virtualNodes);
    }
  }

  addNode(nodeId, virtualNodes = 160) {
    for (let i = 0; i < virtualNodes; i++) {
      const hash = this._hash(`${nodeId}:vnode:${i}`);
      this.ring.set(hash, nodeId);
      this.sortedHashes.push(hash);
    }
    this.sortedHashes.sort((a, b) => a - b);
  }

  getNode(key) {
    const hash = this._hash(key);
    // Find the first hash >= key hash (clockwise)
    const index = this._binarySearch(hash);
    const targetHash = this.sortedHashes[index % this.sortedHashes.length];
    return this.ring.get(targetHash);
  }

  _hash(key) {
    // 32-bit MurmurHash or similar
    let hash = 0;
    for (let i = 0; i < key.length; i++) {
      hash = ((hash << 5) - hash) + key.charCodeAt(i);
      hash = hash & hash;
    }
    return Math.abs(hash);
  }

  _binarySearch(hash) {
    // Simple linear scan for small rings; binary search for large rings
    for (let i = 0; i < this.sortedHashes.length; i++) {
      if (this.sortedHashes[i] >= hash) return i;
    }
    return 0;
  }
}
```
