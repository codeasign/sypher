## Company Case Studies

### Netflix — EVCache with Read-Through

Netflix runs one of the largest read-through cache deployments in the world. EVCache (Ephemeral Volatile Cache) is a distributed read-through cache built on Memcached, serving hundreds of thousands of requests per second across Netflix's microservices.

**Key design decisions at Netflix**:

- **Zone-aware caching**: EVCache runs in multiple AWS availability zones. A read that misses in one zone falls back to another zone before hitting the database. Cross-zone reads are faster than database reads.
- **Automatic fill**: when a microservice reads from EVCache and gets a miss, the service is responsible for filling the cache. Netflix provides client libraries that handle this automatically — the read-through pattern is implemented in the client, not in the cache server.
- **Chunked caching**: large objects (video metadata) are split into chunks and cached independently. The read-through loader reassembles chunks on demand.
- **Ephemeral by design**: EVCache is designed for data loss. If a cache node fails, the cache is rebuilt from the database. There is no replication — loss of a shard means all its keys must be reloaded.

**Lessons from Netflix**:
- Do not replicate cache data — the database is the backup. Replication adds latency and complexity.
- Make the loader service stateless so it can scale independently of the cache.
- Use zone-aware routing to reduce miss penalty — reading from another cache zone is faster than reading from the database.

**Key design decisions at AWS**:

- **Zero configuration**: DAX requires no loader function. The cache knows how to read from DynamoDB because the API is standardized. Any `GetItem` or `Query` call is automatically cache-able.
- **Two-level caching**: DAX caches both individual items (by primary key) and query results (by query parameters + pagination token). Query caching is especially powerful for read-heavy workloads.
- **Write-through option**: DAX can be configured as write-through, writing to DynamoDB and updating the cache atomically. This reduces write latency compared to writing to DynamoDB directly.
- **TTL-aware**: DAX respects DynamoDB TTL. Items with TTL are automatically evicted from the cache when the TTL expires.

**Lessons from Amazon**:
- When the data source has a standard API (like DynamoDB), the loader can be built into the cache system — no user configuration needed.
- Cache both individual items and query results. Query result caching is often more valuable because complex queries are more expensive than point lookups.
- A managed cache service eliminates operational burden — no need to deploy, monitor, or scale the cache cluster manually.

**Key design decisions at LinkedIn**:

- **Two-tier architecture**: the "serving tier" handles reads from cache, the "bootstrap tier" handles cache misses by reading from the database. This separation allows independent scaling of read throughput and miss handling.
- **Disk-backed cache**: Voldemort stores data on disk (B-tree) with an in-memory page cache. This allows caching of datasets larger than memory — the disk is the cache, and the page cache accelerates the most frequently accessed entries.
- **Partition-aware batching**: when a cache partition fails and needs rebuilding, the bootstrap tier batch-loads all keys for that partition from the database. This is much faster than loading one key at a time.

**Lessons from LinkedIn**:
- Separate the serving path from the bootstrap path. The serving path must be fast (sub-millisecond). The bootstrap path can be slower (milliseconds) because it only runs on cache misses.
- Batch loading is essential for partition recovery. A partition with 100,000 keys cannot be rebuilt one key at a time.
- Disk-backed caching allows you to cache datasets larger than memory, at the cost of occasional disk reads.

### PayPal — Read-Through for Fraud Detection

PayPal uses read-through caching to load fraud detection rules and model features. The cache stores pre-computed risk scores and fraud profiles, dramatically reducing the latency of fraud checks.

**Key design decisions at PayPal**:

- **Dual cache: active and standby**: two cache clusters run in parallel. The standby cluster serves stale data when the active cluster fails. This is critical for fraud detection — falling back to the database during a cache failure would add 500ms to every transaction, causing transaction timeouts.
- **Short TTL for fraud data**: fraud profiles change rapidly. PayPal uses TTLs of 60-300 seconds. Refresh-ahead ensures that hot fraud profiles are always fresh.
- **Negative caching**: if a user ID has no fraud profile (clean user), the cache stores this negative result with a longer TTL. This prevents repeated database lookups for clean users.

## Open Source Implementations

### Redis with Read-Through Modules

Redis does not natively support read-through caching — it is a key-value store, not a cache with loaders. However, Redis modules and client-side patterns add read-through behavior:

- **RedisJSON + RediSearch**: store JSON documents in Redis and index them. On a miss, the application loads from the database and stores in Redis. This is cache-aside, but the Redis module handles the JSON storage and querying.
- **Redis with client-side read-through**: libraries like `redis-om` (Node.js), `redis-py` with decorators, and Spring Data Redis provide read-through abstractions. The client library handles the check-then-load pattern, making it invisible to the application developer.
- **Redis Stack**: the Redis Stack distribution includes probabilistic data structures and time-series that can be used for cache hit ratio prediction.

### Caffeine (Java)

Caffeine is the most popular read-through cache library for Java. It provides an in-process read-through cache with highly optimized performance.

```java
// Caffeine read-through cache
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .refreshAfterWrite(1, TimeUnit.MINUTES)  // refresh-ahead
    .build(key -> {
        // This loader runs on cache miss
        String id = key.split(":")[1];
        return userRepository.findById(id);
    });

// Application code — no cache logic
User user = cache.get("user:42");
```

Caffeine's performance comes from:
- **W-TinyLFU eviction**: a frequency-based eviction policy that outperforms LRU for most workloads. Frequently accessed keys are protected from eviction even if they were accessed a long time ago.
- **Ring buffer for recording events**: access events are recorded in a lock-free ring buffer, reducing concurrency overhead.
- **Asynchronous refresh**: `refreshAfterWrite` triggers an async reload, returning the stale value immediately while refreshing in the background.

### Guava Cache (Java)

Guava Cache was the predecessor to Caffeine and remains widely used. It provides a simpler read-through cache implementation.

```java
// Guava Cache read-through
LoadingCache<String, User> cache = CacheBuilder.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(new CacheLoader<String, User>() {
        @Override
        public User load(String key) {
            // Loader runs on cache miss
            return userRepository.findById(extractId(key));
        }

        @Override
        public Map<String, User> loadAll(Iterable<? extends String> keys) {
            // Batch loader for multiple misses
            return userRepository.findByIds(extractIds(keys));
        }
    });
```

### Django Cache Framework (Python)

Django's cache framework supports the read-through pattern via the `get_or_set` method:

```python
# Django read-through pattern
from django.core.cache import cache

def get_user(user_id):
    def load_user():
        return User.objects.get(id=user_id)

    # Read-through: cache.get_or_set handles miss by calling load_user
    return cache.get_or_set(f"user:{user_id}", load_user, timeout=300)
```

### Rails Cache (Ruby)

Ruby on Rails provides `Rails.cache.fetch` which implements read-through:

```ruby
# Rails read-through pattern
user = Rails.cache.fetch("user:#{user_id}", expires_in: 5.minutes) do
  User.find(user_id)
end
```

## Cloud Provider Offerings

### AWS ElastiCache with Read-Through

AWS ElastiCache (Redis/Memcached) does not natively support read-through. AWS recommends implementing read-through in your application using the cache-aside pattern with ElastiCache. However, several patterns achieve read-through behavior:

- **ElastiCache + Lambda**: configure a Lambda function as a "cache miss handler." When ElastiCache returns a miss, the Lambda function loads data from DynamoDB or RDS and populates the cache.
- **ElastiCache + DAX proxy**: for DynamoDB workloads, DAX provides native read-through. For RDS workloads, use a custom proxy that intercepts cache misses and runs the loader.
- **RedisGears**: a Redis module that can trigger server-side functions on key events. You can set up a miss handler that runs a loader when a key is not found.

### GCP Memorystore

Google Cloud Memorystore (Redis) similarly requires application-side read-through logic. Google recommends using Cloud Functions or Cloud Run as the miss handler.

### Azure Redis Cache

Azure Redis Cache provides a "cache-aside" pattern in their documentation. For read-through, Azure recommends using a combination of Redis with Azure Functions as the loader.

## Lessons Learned

### 1. Always deduplicate concurrent misses

The most costly failure in read-through caching is the thundering herd. A single hot key expiring can take down your database. Deduplication is not optional — it is a core requirement.

**Implementation**: use a distributed lock (Redis SET NX, ZooKeeper) with a timeout. The first request acquires the lock and loads the data. Subsequent requests wait on the lock and receive the loaded data.

### 2. Always set a TTL

Entries without a TTL accumulate until the cache runs out of memory. Even for reference data that "never changes," set a TTL of 24 hours. The loader refreshes it once a day, and stale reference data is usually acceptable.

**Implementation**: use a default TTL and override per-key-type when needed. Never set TTL to 0 or null.

### 3. Monitor loader health separately

The cache hit ratio tells you how well the cache is working, but it does not tell you how healthy the loader is. Monitor loader latency, error rate, and throughput as separate metrics.

**Implementation**: emit metrics at each stage of the loader: connection acquisition, query execution, serialization, total duration. Alert on p99 loader latency exceeding 500ms or error rate exceeding 1%.

### 4. Prepare for cold starts

A cold cache (after deployment, scaling event, or cache cluster failure) generates 100% miss rate for the first TTL window. The database must handle this load.

**Implementation**: pre-warm the cache before directing traffic to it. Run a warm-up script that loads the top 10% most-frequently-accessed keys. Use canary deployments where a small percentage of traffic hits the new cache while it warms up.

### 5. Test loader failure scenarios

The loader will fail. The database will be slow. The network will partition. Test these scenarios regularly — in staging and in production (chaos engineering).

**Implementation**:
- **Staging tests**: configure the loader to fail 10% of the time and verify the circuit breaker opens correctly. Configure the database to be slow (100ms latency) and verify the loader timeout fires.
- **Production tests**: use chaos engineering to inject database latency. Monitor the circuit breaker behavior and stale data fallback. Verify that the application continues to serve requests with stale data during the failure injection.

### 6. Instrument every loader call

Every loader call must be instrumented. If you cannot see the latency, error rate, and throughput of the loader, you are flying blind.

**Implementation**: emit metrics at these points in the loader:

```javascript
async function instrumentedLoader(key) {
  const start = Date.now();
  metrics.increment('loader.start', { key });

  try {
    const value = await db.query('SELECT * FROM users WHERE id = $1', [id]);
    metrics.timing('loader.duration', Date.now() - start, { key });
    metrics.increment('loader.success', { key });
    return value;
  } catch (err) {
    metrics.timing('loader.duration', Date.now() - start, { key });
    metrics.increment('loader.error', { key, error: err.code });
    throw err;
  }
}
```

### 7. Version your cache entries

When your data schema evolves, old cache entries contain data in the old format. Without versioning, deserialization fails silently or returns corrupted data.

**Implementation**: include a schema version in each cache entry. Check the version on deserialization and refresh if outdated.

```javascript
const CURRENT_SCHEMA_VERSION = 2;

async function versionedLoader(key) {
  const value = await db.query('SELECT * FROM users WHERE id = $1', [id]);
  return {
    __version: CURRENT_SCHEMA_VERSION,
    name: value.name,
    email: value.email,
    phone: value.phone  // Added in version 2
  };
}

function checkVersion(entry) {
  if (!entry.__version || entry.__version < CURRENT_SCHEMA_VERSION) {
    return null; // treat as miss, force reload
  }
  return entry;
}
```

## Implementation Maturity Model

Read-through cache implementations evolve through five stages of maturity:

### Stage 1: Manual cache-aside (L0)
Every developer writes their own cache logic. No centralized cache layer. Cache behavior is inconsistent across services. Hit ratio is unpredictable.

### Stage 2: In-process read-through (L1)
A single library (Caffeine, Guava) provides read-through behavior within each application process. Loader functions are registered per application. No cross-process deduplication. Cache stampede across application instances.

### Stage 3: Distributed read-through (L2)
A shared cache cluster (Redis, Memcached) with a centralized loader service. Cross-process deduplication, circuit breakers, and rate limiting. Monitoring covers hit ratio, loader latency, and error rate. Loader is deployed as a separate service.

### Stage 4: Multi-region read-through (L3)
Cache clusters in every region, each with a local loader and local database replica. Cross-region cache invalidation. Global monitoring dashboard. Automated failover and disaster recovery.

### Stage 5: Self-tuning read-through (L4)
Adaptive TTL based on access patterns. Automatic pre-warming based on predicted traffic. Machine learning models predict cache misses and pre-load data. Fully automated capacity scaling.

Most organizations operate at Stage 2 or Stage 3. Only very large organizations (Netflix, Amazon, LinkedIn) operate at Stage 4. Stage 5 is aspirational — few organizations have achieved fully self-tuning caches.

## Specific Implementation Patterns

### Pattern: Read-through for session data

Session data is a natural fit for read-through caching. Sessions are read on every request and written infrequently.

```javascript
// Session cache with read-through
const sessionCache = new ReadThroughCache({
  loader: async (sessionId) => {
    // Load session from database on cache miss
    const session = await db.query(
      'SELECT data FROM sessions WHERE id = $1 AND expires_at > NOW()',
      [sessionId]
    );
    return session ? JSON.parse(session.data) : null;
  },
  defaultTTL: 1800,  // 30 minutes — match session TTL
  staleWhileRevalidate: true,
  staleMaxAge: 300   // 5 minutes stale window
});

// Middleware
app.use(async (req, res, next) => {
  const sessionId = req.cookies.sessionId;
  req.session = await sessionCache.get(sessionId);
  next();
});
```

### Pattern: Read-through for configuration

Application configuration rarely changes but is read frequently. Read-through caching with a very long TTL is ideal.

```javascript
// Configuration cache
const configCache = new ReadThroughCache({
  loader: async (key) => {
    const config = await db.query(
      'SELECT value FROM config WHERE key = $1',
      [key]
    );
    return config ? JSON.parse(config.value) : null;
  },
  defaultTTL: 86400,  // 24 hours — configuration changes rarely
  maxSize: 1000       // Configuration is a small dataset
});
```

### Pattern: Read-through with cache warming for ML features

Machine learning feature stores benefit from read-through caching. Features are computed once and read millions of times.

```javascript
// ML feature cache
const featureCache = new ReadThroughCache({
  loader: async (key) => {
    const [userId, featureName] = key.split(':');
    // Compute the feature — this may be expensive
    const userData = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
    const feature = computeFeature(featureName, userData);
    return feature;
  },
  defaultTTL: 3600,          // 1 hour
  refreshAheadWindow: 300,   // Refresh 5 minutes before expiry
  staleWhileRevalidate: true
});

// Pre-warm for batch predictions
async function warmCache(userIds) {
  const features = ['recency', 'frequency', 'monetary', 'tenure'];
  const warmupPromises = [];
  for (const userId of userIds) {
    for (const feature of features) {
      warmupPromises.push(featureCache.get(`${userId}:${feature}`));
    }
  }
  await Promise.all(warmupPromises);
  console.log(`Warmed ${userIds.length * features.length} feature entries`);
}
```
