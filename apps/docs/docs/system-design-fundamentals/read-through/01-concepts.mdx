## The Problem Story

A team of twenty engineers ships a social feed feature. Each engineer builds their own cache layer for user profiles, post metadata, and follower counts. Engineer Alice writes a manual cache-aside pattern — check cache, miss, query database, store in cache, return. Engineer Bob skips the cache entirely for "simplicity." Engineer Charlie stores profiles with a one-hour TTL because it was easy. Engineer Diana forgets to set a TTL at all — her cache entries live forever, consuming memory until the server runs out.

Three months later, the feed is in production. User profiles are inconsistent — some show stale data from Bob's no-cache path, others show five-minute-old data from Alice's cache-aside, and Charlie's users see profiles that are an hour out of date. Diana's server crashes every Tuesday under memory pressure. The team spends two weeks debugging cache-related issues across twenty different implementations.

The root cause is not bad engineers. The root cause is the absence of a single, consistent caching strategy enforced at the infrastructure layer.

## The Insight

Read-through cache solves this by placing a cache layer that owns the loading responsibility. Your application speaks only to the cache. When the cache has the data, it returns it instantly. When the cache does not have the data, it loads it from the database automatically, stores it, and returns it — all without a single line of application-side cache logic.

Think of a library. Cache-aside is like walking to the shelf, finding the book missing, walking to the librarian, requesting the book from the archive, waiting, carrying it back to your desk, and remembering to return it later. Read-through is like a library assistant who checks the shelf for you, fetches missing books from the archive, places them on the shelf, and hands you the book — you never leave your seat.

**The defining property:** the cache is the source of truth for *availability*. The database is the source of truth for *durability*. The cache owns the contract between them.

## Core Vocabulary

**Read-through cache** — A caching pattern where the cache itself loads missing data from the underlying data store on a cache miss. The application never calls the database directly. Why it matters: removes cache logic from application code, centralizing it in the caching layer.

**Cache loader** — A function or callback registered with the cache that knows how to fetch data for a given key from the source of truth. Why it matters: the loader is the only code that touches the database; it is a single point of configuration and testing.

**Cache miss penalty** — The latency incurred when data is not in cache and must be loaded from the database. Why it matters: read-through trades predictable miss latency for simpler application code; the miss penalty determines your p99 tail latency.

**Lazy loading** — Data is loaded into the cache only when first requested, not pre-populated. Why it matters: minimizes cold-start write traffic but means the first request for any key pays the miss penalty.

**Write-through** — A companion pattern where writes go through the cache to the database. Why it matters: read-through is often paired with write-through to keep the cache consistent with the database on writes.

**Cache-aside** — The older pattern where the application checks the cache, then queries the database on a miss, then writes to the cache. Why it matters: read-through is the evolution of cache-aside that moves the responsibility from the application to the cache infrastructure.

**Stale-while-revalidate** — A cache policy that serves stale data while asynchronously refreshing the cache entry in the background. Why it matters: reduces miss penalty by returning data immediately even when the entry is expired, refreshing it before the next request.

**TTL (Time To Live)** — The maximum time a cache entry lives before being considered expired. Why it matters: TTL bounds staleness; choosing the right TTL is the central trade-off between freshness and cache hit ratio.

**Cache provider** — The software library or service that implements the caching logic (Redis, Caffeine, Memcached, Guava Cache). Why it matters: each provider has different read-through semantics, consistency guarantees, and performance characteristics.

**Cache loader strategy** — The policy for how and when the loader is invoked: synchronous, asynchronous, batch, or refresh-ahead. Why it matters: the strategy directly impacts tail latency, database load, and cache hit ratio.

**Refresh-ahead** — A strategy where the cache proactively refreshes an entry before it expires, based on predicted access patterns. Why it matters: eliminates the miss penalty for hot keys by refreshing them preemptively.

**Cache hit ratio** — The percentage of cache lookups that find data in the cache. Why it matters: the single most important metric — a low hit ratio means read-through is adding latency without benefit.

## How It Actually Works

When your application calls `cache.get("user:42")`, a detailed sequence executes inside the read-through cache:

### Step-by-step request lifecycle

1. **Key arrival** — Your application issues `cache.get("user:42")` to the cache client library.

2. **Hashing and routing** — The client hashes the key to determine which cache node holds this entry. If using a distributed cache like Redis Cluster, the key is routed to the correct shard.

3. **Internal lookup** — The cache node checks its internal hash table for an entry matching "user:42." This is an O(1) hash table lookup.

4. **TTL check** — If the entry exists, the cache checks whether the TTL has expired. If the entry is still valid, this is a cache hit — go to step 10.

5. **Miss determination** — If the entry does not exist or the TTL has expired, this is a cache miss. The cache now owns the responsibility of loading the data.

6. **Dedup check** — Before invoking the loader, the cache checks if another request is already loading the same key (concurrent miss deduplication). If so, it waits on that in-flight request rather than duplicating the database call.

7. **Loader invocation** — The cache calls the registered loader function: `loader("user:42")`. This is typically a function you registered at cache initialization time: `cache = ReadThroughCache(loader=fetchUserById)`.

8. **Loader execution** — The loader function queries the database: `SELECT * FROM users WHERE id = 42`. The database returns the row. The loader may also apply transformations, hydrate related objects, or format the data for the cache.

9. **Serialization and storage** — The cache serializes the returned value (JSON, protocol buffers, or a custom serializer) and stores it in the internal hash table with the configured TTL.

10. **Response to application** — The cached value is deserialized (if needed) and returned to your application.

11. **Post-load actions** — If stale-while-revalidate or refresh-ahead is configured, the cache may schedule a background refresh for this key.

### The lookup in code

```javascript
// Application code — no cache logic, just a get call
const user = await cache.get('user:42');
// The cache handles everything else.

// The loader registered at startup:
function fetchUserById(key) {
  const id = key.split(':')[1];
  return db.query('SELECT * FROM users WHERE id = $1', [id]);
}

const cache = new ReadThroughCache({
  loader: fetchUserById,
  ttl: 300,              // 5 minutes
  maxSize: 10000,        // max entries
  staleWhileRevalidate: true
});
```

### Concurrent miss deduplication

## Mental Model Check

**Question 1:** Your application calls `cache.get("profile:7")` and the cache does not have this key. What happens inside the cache?

**Answer:** The cache detects a miss, checks the dedup gate (no other request is loading this key), invokes your registered loader function with key "profile:7", the loader queries the database, returns the profile data, the cache serializes and stores it with the configured TTL, and returns the value to your application. Your code sees a simple return of the profile object.

**Question 2:** How is read-through different from cache-aside?

**Answer:** In cache-aside, your application code checks the cache, queries the database on miss, and writes the result back to cache. In read-through, your application calls `cache.get()` and the cache handles all of that internally. The loader function is registered once at cache initialization, not written in every handler. Read-through eliminates the opportunity for inconsistent cache logic across your codebase.

**Question 3:** What happens if ten requests arrive simultaneously for the same key that is not in cache?

**Answer:** With deduplication, only one request triggers the loader function. The other nine detect that the key is already being loaded and wait on that single in-flight operation. When the loader returns, the result is stored in cache and returned to all ten callers. Without deduplication, all ten would hit the database simultaneously — this is the thundering-herd problem.

**Question 4:** When would you choose stale-while-revalidate over refresh-ahead?

**Answer:** Choose stale-while-revalidate when serving stale data is acceptable and you want to minimize unnecessary background refreshes — the cache only refreshes when a read actually happens after expiration. Choose refresh-ahead when you need zero miss penalty for hot keys and can afford the background load cost. Stale-while-revalidate is better for resources with moderate read frequency; refresh-ahead is better for very hot keys with predictable access patterns.

**Question 5:** What is the correct TTL for a read-through cache entry?

**Answer:** The correct TTL balances freshness requirements against cache hit ratio. For user profile data that changes infrequently (minutes to hours), use 300-3600 seconds. For rapidly changing data like stock prices or leaderboards, use 10-60 seconds. For reference data that never changes (country codes, tax rates), use a very long TTL or no TTL at all. The key insight: set TTL based on how much staleness your application can tolerate, not on how often the data changes — data can change at any time, and TTL only controls when the cache checks for changes.

### Additional mental models

**The library analogy extended**

Imagine a library with three sections: the shelves (cache), the archive (database), and the librarian's desk (loader).

- Cache-aside: you walk to the shelf. The book is not there. You walk to the librarian. You fill out a request form. The librarian fetches the book from the archive. You carry the book back to the shelf and place it there yourself. You check out the book.

- Read-through: you sit at your desk and press a button. The librarian checks the shelf. If the book is there, she brings it to you. If not, she walks to the archive, retrieves the book, places it on the shelf, and brings it to you. You never leave your desk.

- Read-through with stale-while-revalidate: the librarian checks the shelf. The book is there but the page is outdated. She gives you the outdated book immediately (you can still read most of it) and walks to the archive to get the updated version. Next time you press the button, you get the fresh book.

**The vending machine analogy**

A read-through cache is like a vending machine that restocks itself.

- You insert money and press a button (call cache.get).
- If the item is in the machine (cache hit), it drops immediately.
- If the item is empty (cache miss), the machine sends a signal to the warehouse (loader), which brings the item and fills the slot. Then the item drops to you.
- The machine has a limited number of slots (maxSize). If a slot is empty for too long (TTL expired), it is given to another product.
- If the warehouse is closed (database down), the machine shows an "out of stock" message (error) or dispenses an older version of the product (stale data).

**The CPU cache analogy**

Your computer has a multi-level cache hierarchy: L1, L2, L3, and main memory. The read-through cache pattern operates at every level.

- L1 cache: fastest, smallest. Miss goes to L2.
- L2 cache: fast, medium. Miss goes to L3.
- L3 cache: slower, larger. Miss goes to main memory.
- Main memory: slowest, largest. Miss goes to disk.

Each level acts as a read-through cache for the level below. The hardware handles the loading transparently — your code just reads a memory address and gets the value. This is precisely the read-through pattern.

### Common misconceptions

**Misconception 1: "Read-through cache always returns fresh data."**

False. Read-through cache returns data that was fresh at the time of loading. If the database changes after the cache entry was created, the cache serves stale data until the TTL expires or the entry is invalidated. Read-through provides eventual consistency, not strong consistency.

**Misconception 2: "Read-through cache eliminates the need to think about caching."**

False. Read-through eliminates per-request caching logic, but you still need to think about TTL, eviction policies, loader function design, and failure modes. The complexity shifts from the application code to the infrastructure configuration.

**Misconception 3: "Read-through cache is the same as a regular cache with a longer TTL."**

False. The defining feature of read-through is the loader function — the mechanism that automatically populates the cache on a miss. A regular cache with TTL-only still requires the application to populate it.

**Misconception 4: "Read-through cache solves all caching problems."**

False. Read-through is one pattern among many. It is optimal for read-heavy workloads with tolerable staleness. It is not optimal for write-heavy workloads, strongly consistent reads, or small datasets that fit in application memory.

### When you should use read-through

Read-through cache is the right choice when:

1. **Multiple services or teams share the same cache** — the centralized loader ensures consistent caching behavior across all consumers. No team can accidentally implement cache-aside incorrectly.

2. **The read-to-write ratio is heavily skewed toward reads** — 100:1 or higher. Each cache hit saves a database query, and the miss penalty is amortized over many hits.

3. **Stale data is acceptable** — the application can tolerate data that is up to `TTL` seconds old. Most web applications fall into this category (profiles, feeds, catalogs).

4. **The data source has a simple read interface** — a single key maps to a single value (or a batch of keys maps to multiple values). SQL `SELECT BY ID`, DynamoDB `GetItem`, or Redis `GET` are ideal.

5. **The infrastructure team can manage the cache layer** — read-through requires operational investment in the cache cluster, loader service, circuit breakers, and monitoring.

### When you should NOT use read-through

Read-through cache is the wrong choice when:

1. **The write rate is high** — if data changes every few seconds, every cache entry is invalidated before the TTL expires. The cache never serves hits. You are paying for cache infrastructure without any benefit.

2. **Strong consistency is required** — read-through cache is eventually consistent. If your application requires linearizable reads (every read must see the latest write), read-through cannot provide this guarantee.

3. **The dataset is small** — if your entire dataset fits in 1GB or less, load it into application memory at startup. Read-through cache adds unnecessary complexity for small datasets.

4. **The loader is expensive** — if loading one key requires a computation that takes 10 seconds or more, the miss penalty is unacceptable. Pre-compute the data and write it into the cache instead of using read-through.

5. **You need per-request cache control** — read-through applies the same behavior to every request. If you need different caching strategies for different types of requests (sometimes bypass cache, sometimes force refresh), cache-aside gives you more control.

### The loader contract

The loader function is the heart of the read-through cache. Understanding its contract is essential:

```
Input:  key (string)
Output: value (any serializable type) or null (key not found)
Side effects: NONE (the loader must be a pure function)
Timeout: configured per cache (typically 5 seconds)
Retry: configured per cache (typically 2 retries with exponential backoff)
```

The loader must follow these rules:
- It must be idempotent — calling it twice with the same key must produce the same value.
- It must not have side effects — no emails, no counters, no notifications.
- It must handle errors gracefully — return null for not-found keys, throw for database errors.
- It must not call back into the cache it is serving — this creates infinite recursion.
- It should be fast — the loader execution time directly adds to the cache miss penalty.

### The cache hit ratio lifecycle

The cache hit ratio follows a predictable pattern over the lifetime of a cache deployment:

**Phase 1: Cold start (first 5 minutes)** — hit ratio is 0%. Every request is a miss. The database handles 100% of the read load. The loader is under maximum pressure. This phase is the most dangerous for the database.

**Phase 2: Warming (5-30 minutes)** — hit ratio climbs from 0% to 80%. Frequently accessed keys are cached. Database load decreases. The loader handles fewer misses.

**Phase 3: Steady state (30 minutes onward)** — hit ratio stabilizes at 85-98%. The most frequently accessed keys are always in cache. The loader handles only infrequent misses (TTL expirations, cache evictions). Database load is at its minimum.

**Phase 4: Degradation (triggered by deployment, scaling, or failure)** — hit ratio drops. A cache node failure, cluster restart, or key format change causes the cycle to repeat from Phase 1 or Phase 2. The database must handle the increased load during the recovery.

Understanding this lifecycle helps you plan for capacity: the database must have enough headroom to handle Phase 1 load even during steady-state operations. A deployment that restarts the cache cluster effectively resets the hit ratio to zero.

**Key insight**: the cache hit ratio graph over time is an S-curve. Starts flat at zero, climbs steeply as hot keys are cached, then flattens at the maximum achievable hit ratio (determined by cache size relative to unique keys within the TTL window). Monitoring where you are on this curve tells you whether the cache is optimally sized.

### The loader and database interaction

The loader bridges the cache and the database. Understanding how the loader interacts with different database types helps you design better caching strategies.

**With a relational database (PostgreSQL, MySQL)**:
- The loader executes a SELECT query with an exact key match.
- Use database indexes on the columns the loader queries — without an index, every miss triggers a full table scan.
- Use read replicas for the loader to reduce load on the primary database.
- Set statement_timeout on the database side to match the loader timeout.

**With a NoSQL database (DynamoDB, MongoDB)**:
- The loader uses the GetItem or findOne API, which is already optimized for single-key access.
- Batch get operations are more efficient — collect concurrent misses and call BatchGetItem.
- DynamoDB DAX is DynamoDB's native read-through cache — it handles the loader automatically.

**With an external API**:
- The loader makes an HTTP request instead of a database query.
- Add rate limiting to avoid hitting API quotas.
- Cache 4xx responses as errors, not as valid data.
- Retry 5xx responses with exponential backoff.
- Set a shorter TTL because external data changes unpredictably.
