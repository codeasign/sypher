## The Core Tension

Read-through cache simplifies application code but introduces a centralized loader that can become a bottleneck. Every cache miss funnels through the loader function in the same process or service. If the loader is slow, every miss is slow. If the loader fails, every miss fails.

**The tension in one sentence**: read-through trades application simplicity for infrastructure complexity — your application code becomes simpler, but your caching infrastructure must be more reliable than your database.

### The simplicity spectrum

| Pattern | Application complexity | Infrastructure complexity | Developer errors |
|---------|----------------------|------------------------|-----------------|
| No cache | Low | Low | No caching bugs, but slow |
| Cache-aside | Medium | Low | Many (forget cache, wrong TTL, missed invalidation) |
| Read-through | Low | Medium | Few (only the loader can be wrong) |
| Read-through + write-through | Low | High | Fewest (cache handles everything) |

The fewer lines of cache logic your team writes, the fewer cache bugs they introduce. Read-through eliminates the cache-check-and-load pattern from every handler and replaces it with a single registered loader function.

## Decision Matrix

| Dimension | Read-through | Cache-aside | Write-through |
|-----------|-------------|-------------|---------------|
| Application code footprint | Minimal — one loader function | Moderate — check, load, store in every handler | Moderate — write to cache and DB in every handler |
| Cache hit ratio | High (loader ensures data is cached on miss) | High (same) | Highest (data is written to cache on every write) |
| Miss penalty | Higher (includes serialization overhead) | Lower (app controls what to store) | N/A for reads |
| Write consistency | Weak (must pair with write-through or TTL) | Weak (must pair with write-through or TTL) | Strong (synchronous dual write) |
| Operational complexity | Medium (loader service, circuit breaker) | Low (no additional infrastructure) | Medium (write coordination) |
| Cold start behavior | High miss rate until cache warms | Same | Good if data is written frequently |
| Thundering herd risk | Low (dedup gate mitigates) | High (no built-in dedup) | N/A |

### Sync vs async loader

| Aspect | Synchronous loader | Asynchronous loader |
|--------|-------------------|-------------------|
| Application waits | Yes — request blocks until loader completes | No — request gets stale or placeholder data |
| Data freshness | Always fresh on first load | Stale until background load completes |
| Error handling | Error propagates to the caller | Error is logged, stale data is served |
| Use case | User-facing features needing fresh data | Dashboards, analytics, recommendations |
| Latency impact | Miss latency = loader time + cache time | Miss latency = cache time only (near zero) |

### Single vs batch loader

| Aspect | Single-key loader | Batch loader |
|--------|------------------|-------------|
| Database queries per miss | 1 query per key | 1 query for N keys (N up to batch size) |
| Loader complexity | Simple — fetch by ID | More complex — parse keys, group results |
| Cache fill granularity | Per-key | Per-key (each key stored individually) |
| Best for | Random access patterns | Sequential or correlated access patterns |
| Implementation | `loader(key) => value` | `loader(keys[]) => Map<key, value>` |

### Fixed vs dynamic TTL

| Aspect | Fixed TTL | Dynamic TTL |
|--------|-----------|-------------|
| Simplicity | Set once, forget | Compute per key on every load |
| Predictability | Every entry expires at exactly TTL | Entries may have varying lifetimes |
| Adaptability | None — same TTL for all keys | Adjusts based on data characteristics |
| Implementation | `config.ttl = 300` | `loader return { value, ttl: computeTTL(value) }` |
| Best for | Uniform data (all user profiles) | Varied data (profiles + leaderboards in same cache) |

## CAP Theorem and Read-Through

Read-through cache is an AP (Availability and Partition Tolerance) system. When a network partition occurs between the cache and the database, the cache continues to serve data — stale data, if necessary — rather than returning errors.

### Partition scenario

When a network partition separates the cache from the database:

```
Normal state:
  [App] <-> {Cache} <-> ((DB))
  - Reads return fresh data
  - Loader queries the database

During partition:
  [App] <-> {Cache}  ~~X~~  ((DB))
  - Reads for cached keys: return cached data (possibly stale)
  - Reads for uncached keys: loader fails, circuit breaker opens
  - If stale entries exist: return stale data (AP behavior)
  - If no stale entries: return error (unavailability)

After partition resolves:
  [App] <-> {Cache} <-> ((DB))
  - Circuit breaker half-opens, probes succeed
  - Cache reloads stale entries proactively
  - Full functionality restored
```

This AP behavior is appropriate for most read-heavy applications where serving stale data is preferable to serving errors. Applications that require strict consistency should not use read-through caching.

## Failure Modes

### 1. Loader Failure Cascade

The most dangerous failure mode: the loader starts failing (database slow, network issue), but the circuit breaker is not configured. Every miss triggers a slow, failing loader call. All application threads pile up waiting for the loader. Thread pool exhaustion cascades to other services.

**Prevention**: always configure a circuit breaker and a loader timeout. The circuit breaker should open after 5-10 failures in a 60-second window. Loader timeout should not exceed 5 seconds.

### 2. Cache Stampede

When a hot key expires simultaneously across all cache nodes, every request for that key triggers a miss and a loader call. Without deduplication, the database is hit with a wall of identical queries.

  With deduplication:

  t=300s:  Req₁ ──> miss ──> lock ──> loader() ──> ((DB))

`}
/>

**Prevention**: enable deduplication in the loader service. Use refresh-ahead for predictable hot keys. Add jitter to TTL values so entries expire at slightly different times.

### 3. Dead Loader Loop

A misconfigured loader reads from the cache itself, creating an infinite recursion:

```javascript
// WRONG — loader reads from the same cache
async function wrongLoader(key) {
  // This causes infinite recursion!
  const cached = await cache.get(key);
  if (cached) return cached;
  return db.query('SELECT * FROM users WHERE id = $1', [extractId(key)]);
}
```

This pattern causes a stack overflow or an infinite loop. The loader should never call back into the cache it is serving.

**Prevention**: use separate cache instances for the main path and the loader path. Use dependency injection with clear naming: the loader receives a database client, not a cache client.

### 4. Serialization Errors

When the schema of cached data changes (new fields added, fields removed, types changed), deserializing old cache entries can fail.

```javascript
// Version 1 loader (stores UserV1: { name, email })
async function userLoaderV1(key) {
  const row = await db.query('SELECT * FROM users WHERE id = $1', [id]);
  return { name: row.name, email: row.email };
}
// Cache stores: { "name": "Alice", "email": "alice@example.com" }

// Version 2 loader (stores UserV2: { fullName, email, phone })
async function userLoaderV2(key) {
  const row = await db.query('SELECT * FROM users WHERE id = $1', [id]);
  return { fullName: row.name, email: row.email, phone: row.phone };
}
// New code expects { fullName, email, phone }
// Old cache entry has { name, email } — MISSING fullName and phone!
```

**Prevention**: include a schema version in each cache entry. On deserialization, check the version. If the version is outdated, treat the entry as a miss and reload. Use forward-compatible serialization formats like Protocol Buffers or Avro.

### 5. Memory Exhaustion

When cache entries are larger than expected, or when the number of unique keys exceeds the cache capacity, the cache can exhaust memory.

**Scenario**: a new feature stores 1MB user avatar images in the cache instead of small user metadata. The 200GB cache cluster runs out of memory in 15 minutes. The eviction policy evicts everything — all small metadata entries are evicted to make room for avatar images. The hit ratio drops to 5%.

**Prevention**: enforce a maximum entry size. Set per-key-type memory limits. Monitor entry size distributions. Use different caches for different data types (small metadata cache, medium object cache, large blob cache).

## Anti-Patterns

### 1. Loader with Side Effects

The loader should be a pure function — given a key, it returns the corresponding value. It should not send emails, update counters, or trigger notifications.

```javascript
// ANTI-PATTERN: Side effects in loader
async function loaderWithSideEffects(key) {
  const user = await db.query('SELECT * FROM users WHERE id = $1', [id]);
  await emailService.sendWelcomeEmail(user.email); // SIDE EFFECT!
  metrics.increment('user_loaded');                  // Acceptable side effect
  return user;
}
```

Every cache miss triggers the side effect. A cache stampede triggers the side effect hundreds of times. The side effect should happen in the application code, not in the loader.

### 2. Infinite TTL

Setting TTL to `null` or `Infinity` means cache entries live forever. The data becomes increasingly stale. No mechanism exists to refresh it except a server restart.

```javascript
// ANTI-PATTERN: No TTL
const cache = new ReadThroughCache({
  loader: fetchUser,
  ttl: null  // entries never expire!
});
// Week 1: data is fresh
// Week 4: data is stale (users changed names, emails, etc.)
// No way to refresh without restarting the cache
```

Always set a TTL. If you want long-lived cache entries, use a TTL of 24 hours (86400 seconds) so data refreshes at least once per day.

### 3. No Timeout on Loader

Without a timeout, the loader can hang indefinitely if the database connection pool is exhausted, the database is deadlocked, or there is a network issue.

```javascript
// ANTI-PATTERN: No loader timeout
const cache = new ReadThroughCache({
  loader: async (key) => {
    return db.query('SELECT ... FROM ... WHERE id = $1', [id]);
    // No timeout! This query could hang for minutes.
  },
  loaderTimeout: null // BAD: default should be 5000ms
});
```

Always set a loader timeout. Start with 5000ms and adjust based on your p99 query latency. If the loader takes longer than the timeout, fail fast and let the circuit breaker handle the failure.

### 4. Synchronous Loader for Slow Queries

If the loader executes a query that takes 5 seconds, every cache miss blocks the application thread for 5 seconds. For slow queries, use an asynchronous load pattern:

```javascript
// Anti-pattern: synchronous slow loader
function slowLoader(key) {
  const result = db.querySync('SELECT * FROM heavy_report WHERE ...');
  // Blocks the thread for 5 seconds
  return result;
}

// Better: async + stale-while-revalidate
const cache = new ReadThroughCache({
  loader: async (key) => db.query('SELECT * FROM heavy_report WHERE ...'),
  staleWhileRevalidate: true,  // returns stale, refreshes in background
  loaderTimeout: 10000
});
```

### 5. Ignoring Loader Errors

The loader is the most failure-prone component of the read-through cache. Ignoring errors in the loader degrades availability silently.

```javascript
// ANTI-PATTERN: Swallowing errors
async function badLoader(key) {
  try {
    return await db.query('SELECT ...', [id]);
  } catch (err) {
    console.error('Loader error:', err);
    return null; // BAD: returns null, cache stores null
    // Next request also gets null from cache, never retries
  }
}

// Correct: propagate errors
async function goodLoader(key) {
  return await db.query('SELECT ...', [id]);
  // Error propagates → circuit breaker → stale fallback → retry
}
```

## When NOT to Use Read-Through

### Write-heavy workloads

If your application has a high write-to-read ratio (over 1:10), read-through cache adds overhead without benefit. Every write invalidates the cached entry, and the next read must reload it from the database.

**Better approach**: if the workload is write-heavy, consider write-through cache or skip caching entirely.

### Data that changes frequently

If data changes every few seconds but the TTL is 30 seconds, every read is a miss because the entry is invalidated by writes faster than the TTL. The cache adds latency (cache lookup + miss penalty) without providing any benefit.

**Better approach**: use a very short TTL (1-5 seconds) or use a pub-sub invalidation pattern instead of TTL-based expiration.

### Strong consistency requirements

If your application requires linearizability — every read must see the latest write — read-through cache cannot guarantee this. The cache always serves data that may be stale by up to the TTL duration.

**Better approach**: skip caching for strongly consistent reads. Use cache-aside with explicit invalidation on write, and accept that even this provides only eventual consistency.

### Small datasets

If your entire dataset fits in memory, a read-through cache is over-engineered. Load all data into memory at startup and serve it directly from the application.

```javascript
// Over-engineered: reading 1000 config values through cache
// Just load them once:
const config = await db.query('SELECT * FROM config');
const cache = new ReadThroughCache({ ... });  // unnecessary

// Better:
const config = await db.query('SELECT * FROM config');
// Serve from the in-memory config object
```

**Rule of thumb**: if the dataset fits in under 1GB and changes infrequently, load it at startup instead of using read-through cache.

### System of record reads

If you are reading data that must be perfectly accurate — financial transactions, inventory counts, seat availability — do not use read-through cache. The potential staleness introduces business logic errors.

**Better approach**: read directly from the database for system-of-record data. Use read-through cache only for system-of-engagement data (profiles, recommendations, content feeds).

## Trade-Off Deep Dives

### Consistency vs. availability spectrum

Read-through cache lets you trade consistency for availability across a spectrum:

| Configuration | Staleness | Availability during DB outage | Use case |
|---------------|-----------|-------------------------------|----------|
| No stale fallback | None (error on miss) | Low (miss = error) | Finance, inventory |
| Short TTL + no stale | Under 60s | Low | Leaderboards, real-time analytics |
| Short TTL + stale-while-revalidate | Under 60s + stale window | High | Social feeds, news |
| Long TTL + stale-while-revalidate | Under 1h + stale window | Very high | Profiles, reference data |
| No TTL + manual invalidation | Unlimited | Maximum | Configuration, feature flags |

The further right you go on this spectrum, the simpler your application code becomes and the more resilient your system is to database failures. But the more stale data your users may see.

### Hit ratio vs. freshness trade-off

Every read-through cache deployment faces this fundamental trade-off. You cannot maximize both hit ratio and freshness simultaneously.

```javascript
// Freshness-optimal: short TTL, every miss reloads from DB
// Hit ratio: low (may be under 50%)
// Staleness: under 10 seconds
const freshCache = new ReadThroughCache({ defaultTTL: 10 });

// Balanced: moderate TTL, acceptable staleness
// Hit ratio: 85-95%
// Staleness: under 5 minutes
const balancedCache = new ReadThroughCache({ defaultTTL: 300 });

// Hit ratio-optimal: long TTL, very fresh for hot keys
// Hit ratio: 95-99%
// Staleness: up to 1 hour for cold keys, seconds for hot keys
const hitOptimalCache = new ReadThroughCache({
  defaultTTL: 3600,
  refreshAheadWindow: 60
});
```

The right choice depends on your workload. For user profiles, a 5-minute TTL with 95% hit ratio is usually fine. For stock prices, a 10-second TTL with 50% hit ratio is unacceptable — use refresh-ahead instead.

### Cost trade-off

Read-through caching costs money: cache cluster instances, loader service compute, and operational overhead. The benefit is reduced database load and lower read latency. This trade-off must make economic sense.

**Cost calculation example**:

```
Without cache:
  Database: 10 read replicas at $500/month each = $5,000/month
  Database CPU: 60% average (near capacity)

With read-through cache:
  Cache cluster: 4 nodes at $300/month each = $1,200/month
  Loader service: 2 instances at $100/month each = $200/month
  Database: 4 read replicas at $500/month each = $2,000/month
  Total: $3,400/month
  Savings: $1,600/month (32% reduction)

Non-financial benefits:
  - Read latency: 50ms (DB) to 1ms (cache)
  - Database CPU: 60% to 20% (more headroom for writes)
  - Application code is simpler (no cache logic)
```

The cost trade-off favors caching when:
- Database read replicas are expensive (high QPS, large instances)
- Read latency requirements are strict (under 10ms)
- The read-to-write ratio is very high (100:1 or more)
- Multiple services read the same data (cache sharing reduces total cost)

The cost trade-off disfavors caching when:
- The database is already cheap (low QPS, small instances)
- The dataset is small (fits in a single database instance's buffer pool)
- The cache cluster would cost more than the database replicas it replaces

### Operational complexity trade-off

Read-through cache introduces operational complexity in exchange for application simplicity:

**Application simplicity gained**:
- No cache check logic in any handler
- No cache population logic on writes
- No cache invalidation logic
- No cache miss handling
- No cache consistency debugging

**Infrastructure complexity added**:
- Deploy and maintain a cache cluster
- Deploy and maintain a loader service
- Configure circuit breakers, dedup gates, and timeouts
- Monitor cache hit ratio, loader latency, and error rate
- Handle cache cluster failures (node replacement, rebalancing)
- Handle cold starts (pre-warming scripts)
- Upgrade the cache cluster and loader service without downtime

The operational complexity is worth it when the application team is large (20+ engineers) and the database is expensive. It is not worth it for a small team with a simple application and a cheap database.

## Summary

Read-through cache is not a silver bullet. It is a powerful pattern for a specific class of problems: read-heavy workloads where a centralized cache layer can simplify application code and reduce database load. The key trade-offs to evaluate are:

1. **Application simplicity vs. infrastructure complexity**: read-through simplifies your application code but requires you to build and operate a caching infrastructure.

2. **Hit ratio vs. freshness**: longer TTLs improve hit ratios but increase staleness. Use refresh-ahead or stale-while-revalidate to mitigate.

3. **Availability vs. consistency**: read-through is an AP system. It serves stale data during database failures. This is appropriate for most web applications but not for strongly consistent reads.

4. **Cost vs. performance**: caching costs money but reduces database costs and improves latency. Run the economics to determine if caching pays off.

5. **Simplicity vs. control**: read-through removes per-request caching control from the application developer. This is a feature (reduce bugs) or a limitation (reduce flexibility) depending on your perspective.
