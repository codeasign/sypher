## Question Types

### Type 1: "Design a cache for a read-heavy system"

**What they ask**: "Design a caching layer for a social media feed that serves 10 million daily active users. Users read posts 100x more often than they write them."

**What they want**: a system design that includes read-through cache, TTL strategy, miss handling, and capacity planning. The interviewer expects you to compare read-through with cache-aside and justify your choice.

**How to answer**:
1. Start with the requirements: read-heavy, 10M DAU, 100:1 read-write ratio.
2. Explain why read-through is the right choice: 100:1 ratio means the cache will serve most reads, and the miss penalty is acceptable because writes are infrequent.
3. Walk through the architecture: application servers, read-through cache cluster, loader service, database replicas.
4. Cover the miss path: dedup, batch loading, TTL configuration.
5. Address failure modes: circuit breaker, stale fallback, cache stampede prevention.
6. Do capacity planning: QPS, memory, loader throughput.

### Type 2: "How would you handle a cache stampede?"

**What they ask**: "A popular user's profile key expires in your read-through cache, and 10,000 requests arrive simultaneously. How do you prevent the database from being overwhelmed?"

**What they want**: a concrete strategy for deduplication, with an explanation of how it works at the infrastructure level.

**How to answer**:
1. Explain the problem: hot key expiration triggers concurrent misses.
2. Describe deduplication: distributed lock or promise-based dedup gate.
3. Describe complementary strategies: refresh-ahead for predictable hot keys, TTL jitter, stale-while-revalidate.
4. Show the code or pseudocode for the dedup gate.
5. Mention real-world analogies: Netflix EVCache, Caffeine's dedup.

### Type 3: "Read-through vs cache-aside — which would you choose and why?"

**What they ask**: "Compare and contrast read-through cache with cache-aside. When would you use each?"

**What they want**: a balanced comparison that shows you understand the trade-offs, not just the advantages of read-through.

**How to answer**:
1. Define both patterns clearly.
2. List the advantages of read-through: simpler application code, centralized cache logic, consistent miss handling.
3. List the advantages of cache-aside: application controls what to cache, no centralized loader bottleneck, simpler to debug.
4. Provide a decision framework: use read-through when many teams own the same cache, use cache-aside when you need fine-grained control over caching behavior.
5. Mention that read-through is the evolution of cache-aside, not a replacement — both are valid.

### Type 4: "How do you ensure consistency in a read-through cache?"

**What they ask**: "Your application writes data to the database, but the cache still has the old data. How do you ensure the cache is consistent?"

**What they want**: an understanding of the consistency challenges in read-through caching, with practical solutions.

**How to answer**:
1. Explain the problem: read-through cache is eventually consistent by design.
2. Describe TTL-based consistency: set a TTL that matches your staleness tolerance.
3. Describe write-through pattern: combine read-through with write-through for writes.
4. Describe cache invalidation: on write, send an invalidation event to the cache.
5. Describe the trade-off: stronger consistency = higher write latency or more complex infrastructure.

### Type 5: "Design a global read-through cache"

**What they ask**: "Design a caching layer for a global application with users in the US, Europe, and Asia. Every region must read from a local cache with local loaders."

**What they want**: a multi-region architecture with regional cache clusters, database replicas, and cross-region invalidation.

**How to answer**:
1. Describe the multi-region architecture: one cache cluster per region, one database replica per region.
2. Explain the loader pattern: each region's loader reads from its local database replica.
3. Describe cross-region replication: async replication from a primary database to regional replicas.
4. Describe cache invalidation: publish invalidation events to all regions when a write occurs.
5. Address the trade-off: strong consistency across regions is not possible — accept eventual consistency.

## FAANG Framework

### The 5-step cache design framework

**Step 1: Requirements (2 minutes)**
- State the read-to-write ratio (read-heavy? write-heavy? mixed?)
- State the latency requirements (p50, p99, p999)
- State the consistency requirements (strong, eventual, read-your-writes)
- State the scale (DAU, QPS, data size)

**Step 2: Pattern selection (2 minutes)**
- Compare read-through, cache-aside, and write-through
- Justify read-through: "Given the 100:1 read-to-write ratio and the need for simple application code, read-through is the right choice."
- Note the trade-off: "We accept eventual consistency and a slightly higher miss penalty."

**Step 3: Architecture (5 minutes)**
- Draw the architecture: application servers, cache cluster, loader service, database
- Label the data flow: hit path, miss path, loader path
- Show the internal components: hash table, TTL manager, eviction policy, dedup gate

**Step 4: Deep dive (5 minutes)**
- Capacity planning: QPS, memory, loader throughput
- Consistency: TTL, write-through, invalidation
- Failure modes: circuit breaker, stale fallback, cache stampede
- Monitoring: hit ratio, loader latency, error rate

**Step 5: Trade-offs and alternatives (2 minutes)**
- When read-through is not the right choice
- What alternatives exist (cache-aside, write-through, no cache)
- What you would change at 10x scale

## Phrase Bank

### Strong opening phrases

"Read-through cache is the right choice here because the application team should not need to think about caching logic."

"The defining characteristic of read-through is that the cache owns the loading responsibility — the application talks only to the cache."

"Read-through is cache-aside done right — it moves the cache logic from the application to the infrastructure."

### Explaining the miss path

"On a cache miss, the cache invokes the registered loader function, which queries the database, and the cache stores the result before returning it to the application — all transparent to the application."

"The loader is a pure function — it maps a key to a value. The cache handles the lifecycle: invoke on miss, store on success, retry on failure, expire on TTL."

### Comparing patterns

"Cache-aside gives the application full control, but that control comes with responsibility — every developer must implement the pattern correctly."

"Read-through trades flexibility for consistency — you lose the ability to customize caching per-request, but you gain a single, consistent caching behavior across the entire application."

### Handling failure

"The circuit breaker is the safety net — when the loader fails, the circuit opens and the cache serves stale data, preventing a cascade of failures."

"Stale-while-revalidate is the bridge between availability and consistency — the cache serves stale data immediately and refreshes in the background."

### Capacity and scale

"At 10 million DAU with a 95% hit ratio, the loader handles 1,158 QPS — a single database replica can handle this with proper indexing."

"The cache memory requirement is 215GB for 100 million unique keys — a 12-node Redis cluster with 64GB each provides 768GB total, enough for 3x replication."

## Red Flags

### What gets you rejected

**Red flag 1**: "I would just use Redis." — This answer shows no understanding of the cache pattern. Redis is a storage engine, not a cache pattern. Always describe the pattern (read-through) separately from the implementation (Redis, Caffeine, etc.).

**Red flag 2**: "The cache is always consistent." — Claiming that a read-through cache is always consistent with the database shows a lack of understanding of distributed systems. Acknowledge the eventual consistency and describe how you bound staleness with TTL.

**Red flag 3**: "I would set a very long TTL to maximize hit ratio." — This ignores the staleness problem. The interviewer wants to see you balance hit ratio against freshness requirements.

**Red flag 4**: "I would not use a circuit breaker." — Not using a circuit breaker for the loader is a design error. The loader is the most failure-prone component, and without a circuit breaker, a database slowdown can cascade to the entire application.

**Red flag 5**: "I would not handle concurrent misses." — Ignoring cache stampede shows inexperience with production systems. Every read-through cache needs deduplication.

**Red flag 6**: "The loader can do anything — send emails, update counters, trigger notifications." — The loader should be a pure function. Side effects in the loader cause duplicate execution on cache stampedes.

**Red flag 7**: "I would use read-through for everything." — Read-through is not suitable for write-heavy workloads, strongly consistent reads, or small datasets. Acknowledge the limitations.

**Red flag 8**: "I would not monitor the cache hit ratio." — The hit ratio is the single most important metric for a read-through cache. Not monitoring it means you cannot tell if the cache is working.

## Practice Drills

### Drill 1: The three-minute design

Set a timer for three minutes. Design a read-through cache for a photo-sharing application with 5 million DAU. Users view photos 500x more often than they upload them. Cover: requirements, pattern selection, architecture sketch, capacity estimate.

**Check your answer**: Did you mention the 500:1 ratio? Did you choose read-through? Did you estimate QPS and memory? Did you include dedup and circuit breaker?

### Drill 2: The failure postmortem

Scenario: A read-through cache deployment caused a 30-minute production outage. The cache hit ratio dropped to 10% and the database was overwhelmed. Write a postmortem covering: root cause, timeline, impact, and corrective actions.

**Root cause to discover**: a new feature loaded 1MB images into the cache, evicting all small metadata entries. The eviction rate spiked, hit ratio collapsed, and the database received 10x normal query load.

### Drill 3: The migration plan

You are migrating from cache-aside to read-through. Your application has 50 microservices, each with its own cache-aside implementation. Design the migration plan.

**Check your answer**: Did you propose a phased rollout? Did you start with the simplest services (read-only, low traffic)? Did you include a rollback plan? Did you mention the need to update deployment pipelines and monitoring dashboards?

### Drill 4: The capacity puzzle

Your cache cluster has 64GB of memory. The average entry size is 2KB. The TTL is 300 seconds. The read QPS is 50,000. The write QPS is 500. What is the expected hit ratio? What is the expected miss QPS?

**Answer**: 64GB / 2KB = 32 million entries max. With 300s TTL, each entry lasts 300s. At 50,000 QPS, the cache sees 50,000 x 300 = 15 million unique keys per TTL window. The cache holds 32 million entries, so it can hold all unique keys. Expected hit ratio: near 100%. Miss QPS: near 0 (after warmup). But need to account for key distribution — if 1% of keys are very popular, the hit ratio is still near 100% because the cache is large enough.

### Drill 5: The debate

Prepare a two-minute argument for NOT using read-through cache in a specific scenario. Example: "Read-through cache is wrong for a real-time stock trading application because..."

**Answer**: Stock trading requires strong consistency — every read must see the latest trade. Read-through cache is eventually consistent. The TTL-based staleness could cause a user to see a stale stock price, leading to incorrect trading decisions. Use no cache, or use a write-through cache with invalidation on every write.

## Advanced Question Types

### Type 6: "Design a cache for a social media news feed"

**What they ask**: "Design a caching layer for a news feed. Each user has a personalized feed that aggregates posts from 500 followed users. Reads are 1000 QPS. Each feed generation is expensive — it requires querying the recent posts of all 500 followed users and ranking them."

**What they want**: an understanding that personalized feeds are different from simple key-value lookups. The loader is expensive and must be designed carefully.

**How to answer**:
1. Explain that each feed is a unique key per user. The key is `feed:{userId}`.
2. The loader executes a fan-out query: "get recent posts from all 500 followed users, rank by recency/relevance."
3. Use a batch loader that generates multiple feeds at once to amortize the database queries.
4. Use refresh-ahead for the most active users (top 1% of users drive 80% of feed reads).
5. Use stale-while-revalidate — users see their previous feed immediately while the new feed is generated.
6. The loader should timeout after 2 seconds. If the feed generation takes longer, serve stale data.

### Type 7: "How would you invalidate a read-through cache?"

**What they ask**: "When a user updates their profile, the cache still has the old profile. How do you ensure the next read gets the updated data?"

**What they want**: an understanding that read-through alone does not handle writes. They want to see you pair read-through with a write strategy.

**How to answer**:
1. The simplest approach: set an appropriate TTL. If the TTL is 5 minutes, the cache is at most 5 minutes stale.
2. For faster consistency, use write-through: on every write, update both the cache and the database atomically.
3. For explicit invalidation: when a write occurs, delete the cache entry. The next read triggers a miss and reloads from the database.
4. For distributed systems: broadcast an invalidation event to all cache nodes using a pub-sub channel like Redis Pub/Sub or Kafka.
5. Address the trade-off: explicit invalidation is the fastest but adds complexity. TTL-based invalidation is the simplest but has a staleness window.

### Type 8: "Design a read-through cache for a system with 99.999% availability"

**What they ask**: "The system must be available 99.999% of the time (five nines). The cache cannot be a single point of failure. The database can be unavailable for up to 1 minute per year. Design the caching layer."

**What they want**: a highly available read-through cache with redundancy, stale fallback, circuit breakers, and chaos engineering.

**How to answer**:
1. Use a multi-node cache cluster with replication. Each key is replicated across at least two nodes. If one node fails, the replica serves the data.
2. Configure stale-while-revalidate with a 24-hour stale window. During a database outage, the cache serves stale data for up to 24 hours. This covers 99.999% availability (5.26 minutes of downtime per year).
3. The circuit breaker should have a half-open probe interval of 10 seconds. After the database recovers, the cache resumes normal operation within 10 seconds.
4. Use multiple cache clusters in different availability zones. A zone failure does not affect cache availability.
5. The loader service must be deployed with at least 2 instances in each zone. Load balancer health checks detect failed instances.
6. Chaos engineering: regularly kill a cache node, a loader instance, and a database replica. Verify the system continues to serve requests with stale data.

### Type 9: "Compare read-through with write-back cache"

**What they ask**: "When would you use read-through cache versus write-back cache? Design a system that uses both."

**What they want**: understanding that read-through is for reads and write-back is for writes. The two patterns solve different problems.

**How to answer**:
1. Read-through cache optimizes reads — data is loaded lazily on first read. It is ideal for read-heavy workloads.
2. Write-back cache optimizes writes — data is written to the cache first and asynchronously flushed to the database. It is ideal for write-heavy workloads.
3. They can be combined: read-through for reads, write-back for writes. This is called "read-through, write-behind."
4. Example: a user profile service uses read-through for profile reads (frequent) and write-back for profile updates (infrequent). When a user updates their profile, the write-back cache stores the update and asynchronously persists it to the database. The read-through cache returns the updated value from the write-back cache.
5. The challenge: write-back cache can lose data if the cache fails before the data is flushed to the database. Use write-back with acknowledgment: the cache acknowledges the write only after flushing to the database.

### Type 10: "How does read-through cache handle cache stampede?"

**What they ask**: "A cache stampede occurs when multiple requests try to load the same key simultaneously. How does read-through cache prevent this?"

**What they want**: a clear explanation of deduplication at the cache level, with the mechanism explained.

**How to answer**:
1. Define cache stampede: a hot key expires and all concurrent requests trigger misses simultaneously.
2. The read-through cache implements a dedup gate: a concurrent map or distributed lock that tracks in-flight loads.
3. On a miss, the cache checks the dedup gate. If another request is already loading this key, the current request waits on the in-flight promise instead of starting a new load.
4. The dedup gate uses a per-key lock with a timeout. If the loader does not complete within the timeout, the lock is released and the next request becomes the loader.
5. Additional prevention strategies: TTL jitter (randomize expiration times), refresh-ahead (proactively refresh before expiration), stale-while-revalidate (serve stale data and refresh asynchronously).

## Interview Question Phrase Bank — Expanded

### Responding to "What happens on a cache miss?"

"The cache invokes the registered loader function with the key. The loader queries the database, and the cache stores the result before returning to the application. The application never knows there was a miss — it sees a simple return of the value."

### Responding to "How do you choose the TTL?"

"TTL is chosen based on the staleness the application can tolerate, not on how often the data changes. Profile data: 5 minutes. Stock prices: 10 seconds. Configuration: 24 hours. The TTL should be as long as possible without violating the application freshness requirements."

### Responding to "What happens if the database goes down?"

"The circuit breaker opens, and the cache serves stale data from the stale-while-revalidate window. If no stale data exists, the cache returns an error. The application sees degraded but available service."

### Responding to "How do you estimate cache size?"

"Cache size equals the number of unique keys accessed within the TTL window multiplied by the average entry size. For 100 million unique users, each with a 2KB profile, and a 5-minute TTL, the cache needs approximately 200GB. With replication overhead, this becomes 400-600GB."

### Responding to "What is the miss penalty?"

"The miss penalty is the time to execute the loader function plus serialization and storage overhead. For a simple SQL query with an index, expect 10-50ms. For a complex computation, expect 50-500ms. The hit latency is 0.5-3ms. The miss penalty is 10-100x the hit latency."

### Responding to "How do you handle write consistency?"

"Read-through cache alone does not handle writes. I pair it with write-through: every write updates both the cache and the database atomically. This ensures the cache is always consistent with the database on writes. For reads, the TTL provides a bound on staleness."

### Responding to "What are the failure modes?"

"The three most important failure modes are: (1) loader failure cascade — a slow loader blocks all threads, resolved by a circuit breaker; (2) cache stampede — a hot key expiration triggers hundreds of simultaneous database queries, resolved by deduplication; (3) memory exhaustion — large entries evict small entries, destroying the hit ratio, resolved by per-key-type size limits."

### Responding to "How do you monitor a read-through cache?"

"The three most important metrics are: cache hit ratio (target over 90%), loader latency (p99 under 200ms), and loader error rate (under 1%). I also monitor eviction rate, memory usage, and dedup gate in-flight count."

### Responding to "How does read-through work with a distributed cache?"

"In a distributed cache, the consistent hashing ring determines which node holds each key. The read-through behavior is the same — each node handles its own keys. The loader may be centralized (one loader service for all nodes) or distributed (each node has its own loader). The centralized loader has better dedup across nodes; the distributed loader has better isolation."

### Responding to "How do you prevent cold start problems?"

"Before directing traffic to a new cache, warm it up by pre-loading the most frequently accessed keys. Run a warm-up script that reads the top 10% of keys from the database and populates the cache. Use canary deployments — send a small percentage of traffic to the new cache while it warms up. The database must have enough headroom to handle the initial miss storm."

### Responding to "How do you version cache entries?"

"Include a schema version field in each cache entry. On deserialization, check the version. If the version is outdated, treat the entry as a miss and reload from the database. This handles schema evolution without manual cache invalidation."

### Responding to "Can read-through work with a batch loader?"

"Yes. When multiple keys miss simultaneously, the cache can batch them into a single loader call. The loader receives an array of keys and returns a map of keys to values. This is more efficient than N individual loader calls because it reduces database round-trips and connection overhead."

### Responding to "How do you test a read-through cache in production?"

"Use canary deployments and A/B testing. Deploy the cache to a subset of servers and compare latency, error rate, and data freshness with the uncached baseline. Use chaos engineering to inject database failures and verify the circuit breaker and stale fallback work correctly."

## Red Flags — Expanded

### What distinguishes a strong candidate from a weak one

**Weak**: "I would use Redis as a cache." — This shows no understanding of caching patterns. Redis is a storage engine, not a caching pattern.

**Strong**: "I would implement read-through caching with a dedicated loader service, a circuit breaker for database failures, and deduplication to prevent cache stampede. I would use Redis as the cache storage engine." — This shows understanding of both the pattern and the implementation.

**Weak**: "The cache is always consistent because I set a TTL." — This shows a misunderstanding of eventual consistency.

**Strong**: "The cache is eventually consistent. The TTL bounds the staleness to at most 5 minutes. For stronger consistency, I pair read-through with write-through." — This shows understanding of the consistency model.

**Weak**: "I would not monitor the cache." — This is a disqualifying answer.

**Strong**: "I would monitor cache hit ratio, loader latency, error rate, memory usage, and eviction rate. I would alert on hit ratio dropping below 90% or loader latency exceeding 200ms p99." — This shows production readiness.

**Weak**: "Read-through cache works for every scenario." — This shows lack of understanding of trade-offs.

**Strong**: "Read-through cache is optimal for read-heavy workloads with tolerable staleness. For write-heavy workloads, I would use write-through cache. For strongly consistent reads, I would skip caching entirely." — This shows understanding of when to apply the pattern.

### Additional red flags

**Red flag 9**: "I would set the TTL to Infinity." — This shows a fundamental misunderstanding of cache management. Entries would live forever and never reflect database changes.

**Red flag 10**: "The loader can call the cache itself." — This creates infinite recursion. The loader must never call the cache it is serving.

**Red flag 11**: "I would not need a circuit breaker because the database never goes down." — Every database goes down eventually. Not planning for it is a design flaw.

**Red flag 12**: "I would store the entire database in the cache." — This is not feasible for large databases. The cache should store only the most frequently accessed data.

**Red flag 13**: "Read-through cache is the same as cache-aside." — They are different patterns. Read-through centralizes the loading logic in the cache infrastructure. Cache-aside leaves it in the application code.

## Practice Drills — Expanded

### Drill 6: The Amazon interview question

You are designing the product detail page for Amazon. Each page loads product information, seller information, reviews, and recommendations. The page is read 10,000 times per second. Products are updated once per day by sellers. Design the caching strategy.

**Answer**:
- Use read-through cache for product information and seller information. These change infrequently and can be cached with a 24-hour TTL.
- Use read-through cache with a short TTL (5 minutes) for reviews, which change more frequently.
- Use read-through cache with stale-while-revalidate for recommendations, which are computationally expensive to generate.
- Total cache size: 500 million products x 5KB each = 2.5TB. Use a 50-node Redis cluster with 64GB each.
- Write-through for seller updates: when a seller updates a product, the update goes to the cache and the database simultaneously.
- Cache invalidation on stock changes: when inventory changes, invalidate the product cache entry immediately.

### Drill 7: The Twitter interview question

Design a caching layer for Twitter's home timeline. Each user's timeline is composed of tweets from followed users. Timelines are read frequently and must be fresh (under 30 seconds stale). Each timeline generation requires fanning out to 500 followed users and fetching their recent tweets.

**Answer**:
- Key: `timeline:{userId}`. Value: array of tweet IDs (not full tweets).
- TTL: 60 seconds. Stale-while-revalidate with 30-second stale window.
- Loader: fan-out query to 500 followed users' tweet caches. Use batch loading — generate 100 timelines per batch.
- Refresh-ahead for active users: proactively refresh timelines for users who have loaded their timeline in the last 5 minutes.
- Separate tweet content cache: use a separate read-through cache for tweet content, keyed by tweet ID. The timeline cache stores only IDs. The application renders the timeline by fetching each tweet from the tweet cache.
- Cache stampede prevention: the timeline loader is expensive. Deduplication is critical. Use a Redis-based distributed dedup gate with a 10-second timeout.

### Drill 8: The mock interview — 30-minute session

Simulate a 30-minute system design interview with a partner. The question: "Design a caching layer for a ride-sharing app's driver location system."

**Structure**:
- 3 minutes: Requirements gathering (read/write ratio, latency, scale)
- 5 minutes: Pattern selection (read-through vs cache-aside vs write-through)
- 10 minutes: Architecture design (draw the diagram)
- 7 minutes: Deep dive (capacity planning, failure modes, consistency)
- 5 minutes: Trade-offs and alternatives

**Answer key**:
- Driver locations change every 5-10 seconds. Write QPS is high (each driver sends location updates).
- Read QPS is also high (passengers search for nearby drivers).
- Read-through cache with a 10-second TTL for driver locations.
- Write-back cache for driver location updates — the cache aggregates location updates and flushes them to the database every 30 seconds.
- Use geohash-based keys: `location:{geohash_prefix}`. The loader queries all drivers within a geohash region.
- Cache stampede: multiple passengers searching the same area trigger concurrent misses. Use dedup with geohash granularity.

### Drill 9: The code review drill

Review the following read-through cache implementation and identify five issues:

```javascript
const cache = new ReadThroughCache({
  loader: async (key) => {
    const data = await cache.get(key); // Issue 1: cache recursion
    if (data) return data;
    return db.query('SELECT * FROM users WHERE id = $1', [key]);
  },
  defaultTTL: null,     // Issue 2: no TTL
  maxSize: null,        // Issue 3: unlimited size
  loaderTimeout: null,  // Issue 4: no timeout
  staleWhileRevalidate: false,
  refreshAheadWindow: 0
});
```

**Issues**:
1. The loader calls `cache.get(key)` — this creates infinite recursion if the key is not in cache.
2. `defaultTTL: null` means entries never expire. Set a TTL of at least 300 seconds.
3. `maxSize: null` means the cache grows without bound. Set a max size based on available memory.
4. `loaderTimeout: null` means the loader can hang indefinitely. Set a timeout of 5 seconds.
5. No circuit breaker configuration. Add circuit breaker with threshold of 5 failures in 60 seconds.

### Drill 10: The production incident drill

Your team uses a read-through cache for user profiles. At 2:00 PM, the cache hit ratio drops from 95% to 20%. The database CPU spikes to 100%. User-facing latency increases from 50ms to 2 seconds. Diagnose and fix.

**Investigation steps**:
1. Check the cache node metrics: are all nodes still healthy? Is there a node failure?
2. Check the memory usage: has the cache reached its max size? Is the eviction rate spiking?
3. Check the entry size distribution: has a new feature loaded large entries into the cache?
4. Check the deployment log: was there a recent deployment that changed the key format or loader behavior?
5. Check the traffic pattern: is there a spike in traffic to a specific key type?

**Likely root cause**: a new deployment changed the key format from `user:{id}` to `user_profile:{id}`. The cache has all the old keys but the application is asking for the new key format. Every request is a miss, and the database is overwhelmed by the sudden increase in query load.

**Fix**: roll back the key format change. After the rollback, the cache serves hits for the old key format and hit ratio recovers. Plan a migration: deploy the new key format in a canary, and when the cache warms up for the new keys, gradually migrate traffic. Or use dual-read: check both key formats and populate whichever is missing.

## Follow-up Questions and Depth Demonstrations

### Depth: Explaining the dedup gate implementation

Interviewers often want to see that you can implement the dedup gate, not just describe it conceptually.

**Implementation sketch**:

```javascript
class DedupGate {
  constructor() {
    this.inFlight = new Map(); // key -> Promise
  }

  async getOrLoad(key, loader) {
    // Check if this key is already being loaded
    const existing = this.inFlight.get(key);
    if (existing) {
      return existing; // Wait for the in-flight load
    }

    // Start loading and store the promise
    const promise = loader(key).finally(() => {
      this.inFlight.delete(key);
    });
    this.inFlight.set(key, promise);
    return promise;
  }
}

// Usage in the cache:
async get(key) {
  const entry = this.hashTable.get(key);
  if (entry && !this.isExpired(entry)) {
    return entry.value;
  }
  // Read-through: load via dedup gate
  return this.dedupGate.getOrLoad(key, this.loader);
}
```

The key insight: the dedup gate returns a Promise. If the key is already being loaded, the caller gets the same Promise and waits for the same result. If the key is not being loaded, the caller starts the load and stores the Promise for others.

### Depth: Capacity calculation walkthrough

Interviewers want to see you do math, not just describe concepts.

**Walkthrough for a social media application**:

```
Given:
- 10M DAU
- Each user reads 20 profile pages per day
- Peak-to-average ratio: 3:1
- Profile size: 2KB (serialized JSON)
- Desired hit ratio: 95%
- TTL: 300 seconds

Calculations:
- Daily reads = 10M * 20 = 200M reads/day
- Average read QPS = 200M / 86400 = 2,315 QPS
- Peak read QPS = 2,315 * 3 = 6,945 QPS
- Unique users in 300s (TTL window):
  6,945 QPS * 300s = 2,083,500 unique keys
- With 95% hit ratio, the cache must hold 95% of unique keys:
  2,083,500 * 0.95 = 1,979,325 entries
- Memory for entries: 1,979,325 * 2KB = 3.96GB
- Memory for overhead (keys, metadata, LRU): 3.96GB * 1.2 = 4.75GB
- With 2x replication for HA: 9.5GB total
- Cache nodes: 3 nodes at 8GB each = 24GB total (plenty of headroom)

Database load:
- Miss QPS = 6,945 * (1 - 0.95) = 347 QPS
- Each miss triggers 1 database query
- Database must handle 347 additional QPS
- A single database replica handles ~5,000 QPS, so this is well within capacity
```

### Depth: Read-through vs. write-through comparison table

Interviewers may ask you to compare multiple caching patterns in a single answer.

**Comparison of all four patterns**:

| Aspect | Read-Through | Cache-Aside | Write-Through | Write-Back |
|--------|-------------|-------------|---------------|------------|
| Read path | Cache loads on miss | App loads on miss | Cache loads on miss | Cache loads on miss |
| Write path | Not handled (separate mechanism) | App writes to both | Cache writes to DB | Cache writes to DB async |
| App code for reads | `cache.get(key)` | `if cache.has(key)... else db.query()` | `cache.get(key)` | `cache.get(key)` |
| App code for writes | `db.write()` + invalidation | `db.write()` + `cache.set()` | `cache.set(key, value)` | `cache.set(key, value)` |
| Read consistency | Eventual (TTL) | Eventual (TTL) | Eventual (TTL) | Eventual (TTL) |
| Write consistency | N/A | Immediate (dual write) | Immediate (dual write) | Delayed (async flush) |
| Data loss risk | None | None | None | Cache failure = data loss |
| Complexity | Medium | Low | Medium | High |
| Use case | Read-heavy | General-purpose | Read-heavy + consistent writes | Write-heavy |

## Final Interview Tips

1. **Start with the read-to-write ratio**. It is the single most important metric for choosing a caching strategy. State it early: "Given the 100:1 read-to-write ratio, read-through cache is the natural choice."

2. **Draw the architecture**. Even in a phone interview, describe the architecture visually: "The application sits here, the cache sits here, the loader sits here, and the database sits here. Arrows show the data flow."

3. **Address failure modes before the interviewer asks**. Proactively mention the circuit breaker, dedup gate, and stale fallback. This shows deep understanding.

4. **Do the math**. Interviewers love to see capacity calculations. Estimate QPS, memory, and database load. Show your work.

5. **Know when to stop**. Read-through interview questions typically last 10-15 minutes of a 45-minute system design interview. Cover the essentials and move on to the next topic. Do not spend 30 minutes on cache details.

6. **Show you understand the trade-offs**. Every design decision has trade-offs. Acknowledge them: "Read-through simplifies application code but introduces a centralized loader that can become a bottleneck. Here is how I mitigate that..."

7. **Connect to real-world examples**. Mention Netflix, Amazon, or LinkedIn case studies. This shows you know how the pattern works in production, not just in theory.
