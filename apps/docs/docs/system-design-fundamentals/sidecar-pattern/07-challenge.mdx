## Foundation Challenges

### Challenge 1: Logging Sidecar Design

You are the platform engineer for a company with 80 microservices running on Kubernetes. Each service currently writes logs to stdout using its own logging library. The operations team wants a centralized logging pipeline with structured JSON logs, 99.9% delivery guarantee, and a maximum log delivery latency of 10 seconds. The logging infrastructure must handle a peak volume of 5,000 log entries per second across all services.

**Requirements:**
- Design a logging sidecar that replaces the per-service logging libraries.
- The sidecar must read logs from the primary service and forward them to an Elasticsearch cluster.
- The system must tolerate a 30-second Elasticsearch outage without losing log entries.
- Each service must be able to emit logs without knowing about the logging infrastructure.

**Model Answer:**
- Deploy a Fluentd or Vector sidecar container in each pod. Mount an emptyDir volume at /var/log/app in both the primary and sidecar containers.
- The primary service writes structured JSON logs to /var/log/app/service.log as newline-delimited JSON. The sidecar tails this file using inotify for file change detection.
- The sidecar buffers log entries in a circular buffer with a capacity of 200,000 entries. At 5,000 entries per second, this provides 40 seconds of buffer capacity, exceeding the 30-second Elasticsearch outage requirement.
- The sidecar batches log entries in groups of 500 and sends them via HTTP bulk API to Elasticsearch every 5 seconds. This provides a maximum delivery latency of 5 seconds (flush interval) plus 2 seconds (network and processing), well within the 10-second requirement.
- If Elasticsearch is unreachable, the sidecar writes buffered entries to a spillover file on the shared volume. When Elasticsearch recovers, the sidecar first sends the spillover file contents, then resumes normal operation.
- The sidecar tracks its read position in /var/log/app/.fluentd.pos to resume from the correct position after a restart.

### Challenge 2: Library vs. Sidecar Decision

You are designing a new microservice architecture for a financial trading platform. The platform handles 50,000 transactions per second with a maximum allowed latency of 5 milliseconds per transaction. The platform uses Java, C++, and Python services. The platform team wants to implement mTLS, service discovery, and request tracing.

**Requirements:**
- Evaluate whether to use a sidecar proxy or a library-based approach.
- Provide specific latency, resource, and operational cost calculations.
- Justify your recommendation.

**Model Answer:**
- For a latency budget of 5 milliseconds per transaction, the sidecar proxy overhead of 2 to 5 milliseconds (inbound + outbound) consumes 40% to 100% of the budget. This is too high for a line-of-business trading platform.
- The library approach is recommended for this specific use case. Implement mTLS, service discovery, and request tracing as a shared library compiled into each service. The library approach adds 0.01 to 0.1 milliseconds of overhead per transaction, staying within the 5-millisecond budget.
- The library must be implemented in all three languages: Java (for the main trading engine), C++ (for the matching engine), and Python (for analytics). This is a one-time implementation cost of 3 library versions.
- For the latency-insensitive supporting services (monitoring dashboards, reporting, administration), the sidecar proxy can be used. These services do not have the 5-millisecond constraint and benefit from the sidecar's operational simplicity.
- The hybrid approach is the recommended architecture: libraries for the latency-critical trading path, sidecars for all other services.

### Challenge 3: Sidecar Startup Optimization

Your team deploys a service that autoscales rapidly in response to traffic spikes. The service scales from 10 pods to 200 pods in under 2 minutes during a flash sale. Each pod includes an Envoy sidecar proxy. The sidecar's startup time is causing the autoscaler to overshoot: the autoscaler sees that pods are not ready, launches more pods, and when all pods eventually become ready simultaneously, the cluster has 300 pods instead of 200.

**Requirements:**
- Identify the causes of sidecar startup delay.
- Propose optimizations to reduce the startup time.
- Ensure the solution does not compromise security or functionality.

**Model Answer:**
- The sidecar startup delay has three components: image pull (5-15 seconds for Envoy), init container iptables setup (1-2 seconds), and control plane handshake (100-500 milliseconds). Total: 6 to 18 seconds.
- Optimizations: (1) Pre-pull the Envoy sidecar image on all cluster nodes using a DaemonSet that runs the image pull at node startup. This eliminates the image pull time during pod startup, saving 5-15 seconds. (2) Use a sidecar container with native Kubernetes sidecar support (restartPolicy: Always) instead of the init container approach. This allows the sidecar to start in parallel with the primary service rather than sequentially. (3) Reduce the control plane handshake time by configuring the sidecar with a local cache of the bootstrap configuration, eliminating the need for an initial full configuration sync.
- With these optimizations, the sidecar startup time is reduced from 6-18 seconds to 1-3 seconds. The autoscaler can be tuned with a shorter readiness check interval, reducing overshoot from 50% to 10%.

## Design Challenges

### Challenge 4: Multi-Tenant Service Mesh

Design a service mesh for a multi-tenant SaaS platform. The platform hosts 50 tenants, each with 10 to 20 microservices. Each tenant requires isolated networking: tenant A's services cannot communicate with tenant B's services. Each tenant has its own routing rules, rate limits, and TLS requirements. The platform runs on a single Kubernetes cluster with 500 nodes.

**Requirements:**
- Design the sidecar injection strategy for tenant isolation.
- Specify the control plane configuration for tenant-specific policies.
- Address resource allocation and scaling for 50 tenants.
- Include a failover mechanism for control plane outages.

**Model Answer:**
- Use a single Istio service mesh with tenant-aware configuration. Each tenant's services are deployed in a separate Kubernetes namespace. The sidecar injector is configured per-namespace, with each namespace having its own sidecar configuration.
- The control plane (Istiod) distributes tenant-specific configurations using service account-based authorization. Each Envoy sidecar receives only the configuration relevant to its tenant. A global authorization policy prevents cross-tenant traffic: any request from namespace tenant-a to namespace tenant-b is denied by the sidecar proxy.
- Each tenant's rate limits, routing rules, and TLS settings are defined in separate Istio configuration resources (VirtualService, DestinationRule, PeerAuthentication), scoped to the tenant's namespace.
- Resource allocation: each sidecar is allocated 256 MB memory and 500 millicores CPU. For 50 tenants with an average of 15 services each, the total sidecar count is 750. The sidecar overhead is 750 x 256 MB = 192 GB of memory and 750 x 500m = 375 CPU cores.
- Control plane failover: deploy 3 Istiod replicas across availability zones. Each Istiod instance handles all tenants. During a control plane outage, sidecars continue operating with their cached configuration. Configuration changes are not applied until the control plane recovers.

### Challenge 5: Global Multi-Region Service Mesh

Design a global service mesh spanning 3 regions (us-east-1, eu-west-1, ap-southeast-1). Each region has 200 microservices. Services in one region must be able to call services in another region with automatic failover. The system must handle regional outages.

**Requirements:**
- Design the sidecar configuration for cross-region routing.
- Specify the control plane architecture for multi-region operation.
- Address latency and data locality for cross-region traffic.
- Include a disaster recovery plan.

**Model Answer:**
- Deploy a per-region service mesh. Each region has its own Istio control plane (Istiod) and its own Envoy sidecar mesh. The sidecars in each region connect to their local control plane only.
- Cross-region routing is configured using global VirtualServices. Each service's sidecar knows about remote instances of upstream services. The sidecar routes traffic to the local region first. If no local instances are healthy, traffic is routed to the next closest region based on latency.
- The control plane in each region is configured with a global service registry. When a service is deployed in us-east-1, the us-east-1 control plane registers the service. The other regions' control planes discover the new service through a global configuration store (etcd or Consul) that replicates across regions.
- Cross-region traffic is routed through regional gateways (Istio Ingress Gateways). When a service in us-east-1 calls a service in eu-west-1, the traffic flows: us-east-1 sidecar -> us-east-1 gateway -> eu-west-1 gateway -> eu-west-1 sidecar -> destination service. Each hop adds 50-100 milliseconds of cross-region latency.
- Data locality: logging and metrics sidecars forward data to regional backends. Cross-region traffic is minimized. The sidecar's access logs include a region tag to identify where each request was processed.
- Disaster recovery: if a region fails, the remaining regions' control planes detect the failure through the global configuration store. The sidecars update their routing tables to exclude the failed region. Traffic is distributed across the remaining healthy regions.

### Challenge 6: Sidecar for Legacy Migration

You are migrating a legacy monolithic application to microservices. The monolith is written in Java and uses embedded libraries for service discovery, circuit breaking, and metrics collection. The migration plan splits the monolith into 30 microservices over 6 months. During the migration, old and new services must coexist and communicate with each other.

**Requirements:**
- Design a migration strategy that uses the sidecar pattern.
- Both old monolith services and new microservices must participate in the same mesh.
- The migration must be reversible: if a new microservice has issues, traffic must be routed back to the monolith.

**Model Answer:**
- Deploy an Envoy sidecar alongside every service instance, both old and new. The sidecar handles all inter-service communication. The monolith's embedded libraries are bypassed in favor of the sidecar proxy for service discovery and circuit breaking.
- The monolith's existing service is registered with the mesh as a single service. As each microservice is extracted, it is registered as a separate service in the mesh. The sidecar's routing rules determine whether traffic goes to the monolith or the new microservice.
- Canary migration: configure the sidecar to route 10% of traffic to the new microservice and 90% to the monolith. Monitor error rates, latency, and business metrics. Gradually increase the percentage as confidence grows. If issues arise, adjust the sidecar's routing rules to send 100% of traffic back to the monolith.
- The sidecar's routing rules are updated via the control plane without changing any service code. The migration is controlled entirely by configuration changes to the sidecar proxies.
- The monolith continues to use its embedded libraries for internal communication within the monolith process. The sidecar handles only external communication. This avoids the complexity of extracting all internal monolith calls at once.

## Expert Challenges

### Challenge 7: eBPF-Enhanced Sidecar

Design a sidecar proxy that uses eBPF (extended Berkeley Packet Filter) to reduce the latency overhead of traffic redirection. The current iptables-based redirection adds 50-100 microseconds per packet. For a high-frequency trading platform with microsecond latency requirements, this overhead is unacceptable.

**Requirements:**
- Replace iptables-based traffic redirection with eBPF-based redirection.
- Maintain compatibility with the existing Envoy sidecar proxy.
- Reduce the per-packet redirection overhead from 50 microseconds to under 5 microseconds.
- Ensure the solution works in Kubernetes environments.

**Model Answer:**
- Use Cilium's eBPF-based kube-proxy replacement to bypass iptables entirely. Cilium installs eBPF programs directly into the network stack that redirect traffic to the sidecar proxy without traversing the iptables chains.
- The eBPF program attaches to the TC (traffic control) ingress hook on the pod's network interface. For inbound traffic, the eBPF program rewrites the destination IP from the pod IP to the sidecar's IP on the local loopback interface. For outbound traffic, the eBPF program intercepts connect() system calls and redirects the connection to the sidecar's outbound listener.
- The eBPF approach reduces per-packet redirection overhead from 50-100 microseconds (iptables) to 1-5 microseconds (eBPF). This is a 10x to 20x improvement.
- The Envoy sidecar operates unchanged. The iptables rules are simply not installed. The eBPF program handles the traffic redirection that iptables previously handled.
- Compatibility with Kubernetes: Cilium is a CNI plugin that is compatible with standard Kubernetes clusters. The sidecar injection process is modified to install eBPF programs instead of iptables rules. The init container is replaced by a Cilium-managed sidecar lifecycle process.

### Challenge 8: Sidecar Pattern for IoT Edge Devices

Design a sidecar architecture for IoT edge devices. Each device has limited resources (512 MB RAM, dual-core ARM CPU, 10 GB storage). The devices run a container runtime (containerd) and communicate with a cloud backend over an unreliable cellular network. The cloud backend must be able to push configuration updates, collect telemetry, and deploy software updates to the edge devices.

**Requirements:**
- Design a lightweight sidecar that runs on resource-constrained edge devices.
- The sidecar must handle configuration synchronization, telemetry collection, and over-the-air updates.
- The system must tolerate network disconnections of up to 24 hours.
- The sidecar's resource consumption must not exceed 50 MB of RAM and 10% of a CPU core.

**Model Answer:**
- The sidecar is a single binary written in Rust, compiled to a statically linked executable of approximately 5 MB. The binary is deployed as a container alongside the primary application container on each edge device.
- The sidecar implements three functions in a single process. Configuration synchronization: the sidecar polls a cloud configuration endpoint over MQTT (a lightweight protocol suitable for unreliable networks). When the network is disconnected, the sidecar uses the last cached configuration stored on a shared volume.
- Telemetry collection: the sidecar reads telemetry data from a shared volume where the primary application writes its metrics. The sidecar batches telemetry data and sends it over MQTT when the network is available. The batch size is limited to 100 KB to fit within cellular data constraints.
- Over-the-air updates: the sidecar downloads updated container images for the primary application from a container registry. The download uses incremental layers to minimize data transfer. The sidecar verifies the image signature before applying the update. The update is applied by restarting the primary application container.
- Resource consumption: the Rust sidecar binary uses 8 MB of RAM for the base process, plus 10 MB for configuration caching, plus 10 MB for telemetry buffering. Total: 28 MB, well within the 50 MB limit. CPU usage is 5% of a single core during active operation.
- Network resilience: the sidecar implements a store-and-forward pattern. Telemetry data is stored on the shared volume when the network is unavailable. Data is compressed using gzip before transmission, reducing data transfer by 70% to 90%. The sidecar retries failed transmissions with exponential backoff starting at 1 minute and doubling to 1 hour, with a maximum retry duration of 24 hours.
