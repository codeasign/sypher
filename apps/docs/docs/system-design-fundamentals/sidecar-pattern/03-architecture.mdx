## Naive Architecture (Library-Based)

The naive approach to handling cross-cutting concerns is to embed infrastructure logic directly into each service. Every service includes a logging library, a metrics library, a configuration client, and TLS handling code as dependencies. This is the baseline architecture that the sidecar pattern replaces.

**Structure.** Each service instance is a single process. The process includes all infrastructure libraries compiled or interpreted alongside the business logic. The process opens a port for business traffic, writes log files to local storage, emits metrics to a local agent, and manages TLS connections itself.

**Data Flow.** The service writes log entries to a file on disk. A separate log shipper agent (running as a daemonset or host process) reads the log file and forwards entries to the logging system. The service pushes metrics to a local metrics agent over UDP. The service manages its own TLS certificates, loading them from a mounted secret volume.

**Problems.** Each service must implement or include the same infrastructure code. When the logging library is updated, 47 services must be rebuilt and redeployed. Each service consumes memory for its own copy of the infrastructure libraries. The cumulative memory overhead of embedding a 10 MB logging library in 47 services is 470 MB of wasted memory. Error handling for infrastructure concerns varies across services. Some services retry failed log writes. Others silently drop them. There is no centralized policy enforcement.

## Production Architecture (Kubernetes Sidecar + Mesh)

The production architecture adds a sidecar container to every pod. The sidecar handles all cross-cutting concerns. The primary service focuses on business logic.

**Structure.** Each pod contains two containers: the primary service container and the sidecar container. The sidecar is typically a service mesh proxy (Envoy) that handles all network traffic to and from the primary service. Additional sidecar containers can be added for log forwarding, metrics collection, and config synchronization.

**Data Flow.** Inbound traffic arrives at the pod's IP address. Iptables rules redirect the traffic to the sidecar's inbound listener. The sidecar applies filters (authentication, rate limiting, observability), then forwards the traffic to the primary service over localhost. Outbound traffic from the primary service is similarly redirected through the sidecar's outbound listener, which handles service discovery, load balancing, and TLS origination.

**Control Plane.** The sidecar proxies receive configuration from a centralized control plane. The control plane distributes routing rules, service discovery data, TLS certificates, and policy configurations. The control plane also collects telemetry data from the sidecars, providing a centralized view of service-to-service communication.

**Benefits.** The sidecar's configuration is managed centrally. A security patch to the proxy is deployed by updating the sidecar image and rolling pods. The primary service team never touches infrastructure code. Resource usage is predictable: each sidecar consumes a fixed amount of CPU and memory, independent of the service's business logic.

## Integration Architecture

The integration architecture describes how the sidecar pattern connects to external systems: logging backends, monitoring systems, configuration stores, and secret management services.

**Logging Integration.** The log forwarding sidecar writes to a centralized logging system over a reliable transport protocol. For Elasticsearch, the sidecar uses HTTP bulk indexing. For Kafka, the sidecar produces messages to a topic. For S3, the sidecar writes compressed log files at intervals. The sidecar implements backpressure: if the backend is unreachable, the sidecar buffers logs to disk and retries with exponential backoff. The backoff starts at 1 second and doubles to a maximum of 60 seconds.

**Metrics Integration.** The metrics sidecar exposes a Prometheus scrape endpoint on a dedicated port. Prometheus scrapes the sidecar's metrics endpoint at 15-second intervals. The sidecar aggregates metrics from the primary service and adds its own proxy metrics (request count, latency, error rate). The combined metrics are stored in Prometheus and visualized in Grafana dashboards.

**Configuration Integration.** The config sync sidecar connects to a configuration store using a watch-based protocol when available. For etcd, the sidecar uses the watch API to receive real-time change notifications. For Consul, the sidecar uses blocking queries. For AWS AppConfig, the sidecar polls at configurable intervals. The sidecar validates configuration before writing it to the shared volume.

**Secret Integration.** The sidecar may handle secret rotation for the primary service. The sidecar fetches secrets from a secret store (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager), writes them to a shared volume or memory-backed filesystem, and signals the primary service to reload when secrets change. The sidecar handles secret rotation without the primary service needing to implement secret management logic.

## Multi-Region Architecture

In a multi-region deployment, each region runs its own service mesh with its own control plane. Sidecar proxies in each region connect to their local control plane, not to a cross-region control plane. This avoids cross-region latency and network partition issues.

**Cross-Region Traffic.** When a service in us-east-1 calls a service in eu-west-1, the call passes through two sidecar proxies. The origin sidecar in us-east-1 routes the traffic to the cross-region gateway. The gateway forwards the traffic to the destination region. The destination sidecar in eu-west-1 receives the traffic and forwards it to the destination service. Each sidecar applies its own policies independently.

**Data Locality.** Logging and metrics sidecars in each region forward data to regional backends. A centralized observability platform may aggregate data from all regions, but the sidecars write to local storage first. This prevents cross-region data transfer from becoming a bottleneck or a single point of failure.

**Configuration Divergence.** Each region has its own configuration scope. The sidecar in us-east-1 may have different routing rules than the sidecar in eu-west-1. The control plane manages region-specific configurations independently. Configuration changes are rolled out per region, typically starting with a canary region before global rollout.

## Capacity Planning

Capacity planning for sidecar-based deployments requires accounting for the sidecar's resource consumption in addition to the primary service's requirements.

**CPU Allocation.** The sidecar proxy consumes CPU proportionate to the request rate. A planning formula is: sidecar CPU = base CPU (0.1 cores) + (request rate * 0.0001 cores per request per second). For a service handling 5,000 requests per second, the sidecar requires approximately 0.6 CPU cores. The base CPU allocation for the sidecar should be 500 millicores for low-traffic services and 1 to 2 cores for high-traffic services.

**Memory Allocation.** The sidecar proxy's memory consumption is driven by the number of listeners, clusters, routes, and active connections. A planning formula is: sidecar memory = base memory (50 MB) + (number of upstream clusters * 1 MB) + (active connections * 5 KB). For a service with 20 upstream clusters and 1,000 active connections, the sidecar requires approximately 50 + 20 + 5 = 75 MB of memory. A safe allocation is 128 MB to 512 MB per sidecar.

**Network Bandwidth.** The sidecar proxy processes every byte that enters and leaves the pod. The sidecar's bandwidth usage equals the service's bandwidth usage multiplied by 2 (inbound and outbound). For a service that processes 100 MB/s, the sidecar processes 200 MB/s. The sidecar must be provisioned on a node with sufficient network capacity.

**Scaling Considerations.** When scaling a service horizontally, each new pod includes a new sidecar instance. For 100 pods, the sidecar overhead is 100 sidecar instances consuming 50 GB of memory and 50 CPU cores. This overhead is the cost of the sidecar pattern and must be factored into infrastructure cost calculations.

## Operations

Operating a sidecar-based system requires specific practices for monitoring, debugging, and updating the sidecar infrastructure.

**Monitoring the Sidecar.** The sidecar's health must be monitored separately from the primary service. Key metrics include sidecar process uptime, connection pool utilization, configuration revision (should match the expected control plane revision), latency overhead (time spent in the sidecar vs. time spent in the primary service), and error rate (requests that fail in the sidecar vs. requests that fail in the primary service).

**Debugging Sidecar Issues.** When a request fails, the cause may be in the primary service, the sidecar, or the network between them. The standard debugging approach is to check the sidecar's admin endpoint (Envoy's admin interface on port 9901) for configuration state, connection status, and recent errors. The iptables rules can be inspected to verify traffic redirection. The sidecar's access logs show every request that passed through the proxy.

**Sidecar Updates.** The sidecar image is updated independently of the primary service image. A sidecar update is a rolling update of the pod template. The update can be rolled out gradually with a canary deployment: 10% of pods get the new sidecar, 90% keep the old sidecar. If the new sidecar causes issues, the rollout is paused and rolled back.

**Resource Exhaustion.** A sidecar that consumes too much memory triggers an OOM kill. The sidecar container restarts, but the primary service may experience a brief interruption if traffic is redirected through the sidecar. To prevent this, memory limits are set to 150% of the expected peak usage. CPU throttling is preferred over CPU limits: the sidecar can burst above its CPU request when needed.
