# Understanding Denormalization

⏱️ 20 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Define denormalization and explain why it sacrifices write efficiency for read speed
- Recognize when a normalized schema is causing performance problems
- Apply the denormalization step-by-step process to an e-commerce schema
- Distinguish between precomputed columns, materialized views, and star schemas
- Evaluate trade-offs at each stage of denormalization

## Prerequisites

You should already understand:

- Normalization forms (1NF, 2NF, 3NF) from the Normalization topic
- Basic SQL JOIN operations
- The difference between OLTP and OLAP workloads

## The Problem Story

Your team launches a dashboard for the operations team. The schema is pristine — every table is third normal form. The `orders` table has no redundant data. The `customers` table is clean. The `products` table is normalized. You followed every best practice.

```sql
-- The perfectly normalized schema (simplified)
SELECT
  c.name,
  c.email,
  o.order_date,
  o.total,
  oi.quantity,
  oi.unit_price,
  p.name AS product_name,
  p.category,
  p.sku,
  a.street,
  a.city,
  a.zip
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN addresses a ON c.customer_id = a.customer_id;
```

This query joins five tables. In staging with 10,000 customers and 50,000 orders, it runs in 200 milliseconds. The dashboard looks great. You deploy to production.

### The Slow Descent

At 100,000 customers and 2 million orders, that same query takes 2.5 seconds. At 500,000 customers and 15 million orders, it takes 8 seconds. The CTO opens a dashboard report for last month's sales by category and waits 30 seconds for a spinning loader.

```sql
-- The monthly category report the CTO wants
SELECT
  p.category,
  DATE_TRUNC('month', o.order_date) AS month,
  SUM(oi.quantity * oi.unit_price) AS revenue,
  COUNT(DISTINCT c.customer_id) AS unique_customers,
  COUNT(DISTINCT o.order_id) AS order_count
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
GROUP BY p.category, DATE_TRUNC('month', o.order_date);
```

This query joins four tables, aggregates across millions of rows, and hits a massive temporary table in memory. The database is spending most of its time reading indexes, joining rows, and sorting intermediate results. Every dashboard view triggers the same pattern — the database is doing thousands of JOINs per second across the entire page.

The CTO asks: "Why does a simple report take 30 seconds?"

The answer: the schema is perfectly normalized for writes, but the dashboard is doing read-heavy analytics. Normalization eliminated redundancy at the cost of requiring JOINs for every query. The system was designed for transaction processing but is being used for reporting.

### The Fix

You create a denormalized table that stores the data pre-joined:

```sql
CREATE TABLE order_reports AS
SELECT
  c.name AS customer_name,
  c.email,
  o.order_id,
  o.order_date,
  o.total,
  oi.quantity,
  oi.unit_price,
  p.name AS product_name,
  p.category,
  p.sku,
  a.street,
  a.city,
  a.zip
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN addresses a ON c.customer_id = a.customer_id;
```

Now the CTO's category report becomes:

```sql
SELECT
  category,
  DATE_TRUNC('month', order_date) AS month,
  SUM(quantity * unit_price) AS revenue,
  COUNT(DISTINCT customer_name) AS unique_customers,
  COUNT(DISTINCT order_id) AS order_count
FROM order_reports
GROUP BY category, DATE_TRUNC('month', order_date);
```

One table. Zero JOINs. Result: 400 milliseconds instead of 30 seconds.

The trade-off: the `order_reports` table is 3 times larger than the sum of the normalized tables. Every new order requires updating this table or rebuilding it periodically. Data is duplicated across millions of rows. If a customer changes their address, the old address stays in the report table until the next rebuild.

## The Insight

Denormalization is the practice of intentionally adding redundant data to a database schema to improve read performance. It is the opposite of normalization, which removes redundancy to eliminate anomalies and save storage.

Think of it like a restaurant kitchen:

- **Normalized kitchen**: One chef has all ingredients in the walk-in cooler. When an order comes in for a chopped salad, the chef walks to the cooler, gets lettuce, carrots, tomatoes, washes them, chops them, plates them. Efficient storage — every ingredient is in one place. Slow service during dinner rush.
- **Denormalized kitchen**: During the afternoon prep, the chef pre-chops all vegetables and stores them in labeled bins at each station. When a salad order comes in during dinner rush, the line cook grabs pre-chopped lettuce, pre-chopped carrots, pre-chopped tomatoes, and assembles in seconds. Faster service — but some vegetables wilt before they are used.

The dinner rush is your read workload. The pre-chopped vegetables are your denormalized data. The extra storage and waste is the price of speed.

### Why This Matters at Scale

At small scale, JOINs are fast. A normalized schema with a few thousand rows works perfectly for both reads and writes. The overhead of maintaining denormalized copies is not worth the effort.

At medium scale (millions of rows), JOINs start to hurt. A four-table JOIN with aggregation starts taking seconds. Denormalization becomes a practical necessity for any read-heavy workload.

At large scale (billions of rows), JOINs on normalized schemas become prohibitively expensive. Denormalization is not optional — it is the standard pattern for analytics, reporting, and read-heavy services. Every major data warehouse uses denormalized schemas (star schemas) as the default design.

## Core Vocabulary

**Denormalization**: The process of adding redundant data to a database schema by combining data from multiple tables into one table. The goal is to reduce the number of JOINs required for read queries.

**Redundancy**: Data that appears in more than one place. A customer's name stored in both the `customers` table and a denormalized `order_reports` table is redundant. Redundancy is the defining cost of denormalization.

**Read Amplification**: The amount of data a read query must scan to return results. In a normalized schema, each JOIN multiplies the rows scanned. A denormalized schema reduces read amplification by storing data in a single flat structure.

**Write Amplification**: The amount of data that must be written to keep denormalized copies consistent. Every insert, update, or delete in the source tables may require updating multiple denormalized tables, caches, or indexes.

**Materialized View**: A database object that stores the result of a query as a physical table. Unlike a regular view (which is just a saved query), a materialized view persists the data and can be refreshed on a schedule. This is the most common mechanism for implementing denormalization in a relational database.

**Summary Table**: A table that stores pre-aggregated data — for example, daily sales totals by category instead of every individual order line. Summary tables are a specialized form of denormalization that reduces both row count and query complexity.

**ETL (Extract, Transform, Load)**: The pipeline that extracts data from source systems (normalized OLTP databases), transforms it (applies denormalization, aggregation, cleaning), and loads it into a target system (data warehouse, reporting database).

**OLTP (Online Transaction Processing)**: Systems designed for high-volume, low-latency write operations. The schema is typically normalized to minimize write overhead. Examples: order processing systems, user account databases.

**OLAP (Online Analytical Processing)**: Systems designed for complex read queries across large datasets. The schema is typically denormalized to minimize query overhead. Examples: reporting dashboards, analytics platforms, business intelligence tools.

**Star Schema**: A denormalized schema design where a central fact table contains numeric metrics and foreign keys to dimension tables. The fact table is wide (many columns) and deep (many rows). Dimensions are usually denormalized (flat) rather than normalized. The structure looks like a star: one table in the center with tables radiating outward.

**Snowflake Schema**: A variant of star schema where dimension tables are partially normalized into sub-dimensions. For example, instead of storing `category_name` directly in the `product` dimension, the product dimension references a separate `category` table. Snowflake schemas reduce redundancy compared to star schemas but require more JOINs for queries.

## Step by Step: Denormalizing an E-Commerce Schema

Walk through the process of taking a normalized e-commerce schema and gradually denormalizing it. At each step, you will see the trade-off being made.

### Step 1: Identify the Read-Heavy Query

Start by finding the query that accounts for most of your read traffic. In a typical e-commerce dashboard, that is the monthly sales report.

```sql
-- The target query: runs on every dashboard load
SELECT
  DATE_TRUNC('month', o.order_date) AS month,
  p.category,
  SUM(oi.quantity * oi.unit_price) AS revenue,
  COUNT(DISTINCT o.order_id) AS orders
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
GROUP BY 1, 2;
```

**Current tables**: `orders`, `order_items`, `products` — three tables, two JOINs, GROUP BY aggregation.

**Cost**: For 15 million orders and 45 million order items, this query scans 60 million rows and creates a temporary table of 5 million intermediate rows before grouping.

### Step 2: Pre-Join the Most Expensive Relationship

The `order_items` to `products` JOIN is the most expensive because `order_items` is the largest table (45 million rows) and the JOIN requires looking up each product_id.

Create a denormalized copy of `order_items` with product columns included:

```sql
CREATE TABLE order_items_denormalized AS
SELECT
  oi.order_item_id,
  oi.order_id,
  oi.quantity,
  oi.unit_price,
  p.product_id,
  p.name AS product_name,
  p.category,
  p.sku
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id;
```

**Trade-off**: `order_items_denormalized` is 40 percent larger than `order_items` because each row now carries `product_name`, `category`, and `sku`. But the monthly report query now only needs one JOIN instead of two.

**Query improvement**:
```sql
-- Before: 2 JOINs
SELECT ... FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id;

-- After: 1 JOIN
SELECT ... FROM orders o
JOIN order_items_denormalized oi ON o.order_id = oi.order_id;
```

**Latency**: 8 seconds → 5 seconds.

### Step 3: Pre-Join Orders and Customers

The next expensive JOIN is between `orders` and `customers`. Customer data (name, email, tier) is frequently accessed with orders.

```sql
CREATE TABLE orders_denormalized AS
SELECT
  o.order_id,
  o.order_date,
  o.total,
  c.customer_id,
  c.name AS customer_name,
  c.email AS customer_email,
  c.tier AS customer_tier
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;
```

**Trade-off**: `orders_denormalized` is 30 percent larger than `orders` because customer columns are duplicated. If a customer changes their name, the old name persists in the denormalized table until it is refreshed.

**Query improvement**:
```sql
-- Now the monthly report can combine both denormalized tables
SELECT ... FROM orders_denormalized o
JOIN order_items_denormalized oi ON o.order_id = oi.order_id;
```

**Latency**: 5 seconds → 3 seconds.

### Step 4: Create a Full Flat Table

For the fastest possible read performance, combine everything into one flat table:

```sql
CREATE TABLE order_reports AS
SELECT
  o.order_id,
  o.order_date,
  o.total,
  o.customer_name,
  o.customer_email,
  o.customer_tier,
  oi.product_name,
  oi.category,
  oi.sku,
  oi.quantity,
  oi.unit_price
FROM orders_denormalized o
JOIN order_items_denormalized oi ON o.order_id = oi.order_id;
```

**Trade-off**: This table has one row per order item. An order with five items produces five rows, each containing the full customer and order data repeated five times. Storage increases by 3x compared to the normalized schema.

**Query improvement**: Zero JOINs.

**Latency**: 3 seconds → 0.4 seconds.

### Step 5: Add Precomputed Aggregations

Even the flat table requires a GROUP BY. For the most frequent dashboard queries, precompute the aggregation:

```sql
CREATE TABLE daily_category_summary AS
SELECT
  DATE(order_date) AS day,
  category,
  SUM(quantity) AS units_sold,
  SUM(quantity * unit_price) AS revenue,
  COUNT(DISTINCT order_id) AS order_count
FROM order_reports
GROUP BY 1, 2;
```

**Trade-off**: The summary table is 200x smaller than the base flat table (15,000 rows instead of 45 million rows for a year of data). But it is fixed to the specific GROUP BY columns — you cannot drill down into individual orders from this table.

**Query improvement**: No GROUP BY needed. Just a SELECT with optional WHERE filter.

**Latency**: 400 milliseconds → 20 milliseconds.

### Step 6: Handle Customer Address Changes

The CTO now wants to see shipping addresses on the dashboard. You add address columns to the flat table:

```sql
-- Add address columns to order_reports
ALTER TABLE order_reports ADD COLUMN shipping_street VARCHAR(200);
ALTER TABLE order_reports ADD COLUMN shipping_city VARCHAR(100);
ALTER TABLE order_reports ADD COLUMN shipping_zip VARCHAR(20);
```

**Trade-off**: The table grows wider. If a customer moves, the old address stays in the report table. You need a reconciliation job to update the denormalized copy when addresses change.

### Step 7: Set Up an ETL Pipeline

The denormalized tables are now too complex to maintain by hand. Set up a scheduled ETL pipeline:

```sql
-- Refresh the full flat table daily at 2 AM
CREATE OR REPLACE VIEW order_reports_refresh AS
SELECT ... -- (full query from Step 4)

TRUNCATE order_reports;
INSERT INTO order_reports SELECT * FROM order_reports_refresh;
```

For near-real-time updates, use incremental refresh:

```sql
-- Incremental refresh for the last 24 hours
DELETE FROM order_reports
WHERE order_date >= NOW() - INTERVAL '2 days';

INSERT INTO order_reports
SELECT ... -- (full query from Step 4)
WHERE order_date >= NOW() - INTERVAL '2 days';
```

**Trade-off**: Full rebuilds take minutes and block the table during the operation. Incremental refreshes are faster but can miss updates to historical data (customer name changes, address updates).

### Step 8: Monitor and Iterate

Track the health of the denormalized system:

```sql
-- Check data freshness
SELECT MAX(order_date) FROM order_reports;

-- Check for data drift between normalized and denormalized
SELECT COUNT(*) FROM orders o
LEFT JOIN order_reports r ON o.order_id = r.order_id
WHERE r.order_id IS NULL;

-- Measure query improvement
EXPLAIN ANALYZE SELECT ... FROM normalized_query;
EXPLAIN ANALYZE SELECT ... FROM denormalized_query;
```

**Key metrics**: Data freshness lag (should be under one refresh interval), row count consistency (denormalized should match normalized's count), query latency ratio (denormalized should be at least 5x faster).

## Variants of Denormalization

Denormalization is not a single technique. Different scenarios call for different approaches.

### Precomputed Columns

The simplest form: add a column to an existing table that stores a computed value, avoiding repeated calculation.

```sql
-- Instead of computing total on read:
SELECT quantity * unit_price AS total FROM order_items;

-- Store it as a column:
ALTER TABLE order_items ADD COLUMN total DECIMAL(10,2);
UPDATE order_items SET total = quantity * unit_price;

-- Now reads are instant, but writes need to update the column too
```

**Use case**: Expensive computations (complex formulas, concatenations, JSON extractions) that are read far more often than written.

**Trade-off**: The computation happens on write instead of read. If the data is read 100 times for every 1 write, this shifts cost to the less frequent operation.

### Redundant Copies in Different Formats

Store the same data in multiple formats optimized for different access patterns.

```sql
-- Row format for point lookups (OLTP)
CREATE TABLE orders_row (
  order_id BIGINT PRIMARY KEY,
  customer_id INT,
  order_date TIMESTAMP,
  total DECIMAL(10,2)
);

-- Columnar format for analytics (OLAP)
CREATE TABLE orders_columnar (
  order_id BIGINT,
  customer_id INT,
  order_date TIMESTAMP,
  total DECIMAL(10,2)
) WITH (format = 'PARQUET', sort_columns = ('order_date', 'customer_id'));
```

**Use case**: Systems that serve both transactional and analytical workloads on the same data.

**Trade-off**: You maintain two copies of the same data. Storage doubles. Consistency between copies requires a refresh mechanism.

### Materialized Views

A database-native denormalization mechanism. The database manages the redundant storage and provides refresh commands.

```sql
-- Create a materialized view (PostgreSQL syntax)
CREATE MATERIALIZED VIEW monthly_category_sales AS
SELECT
  DATE_TRUNC('month', o.order_date) AS month,
  p.category,
  SUM(oi.quantity * oi.unit_price) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
GROUP BY 1, 2
WITH DATA;

-- Query against the materialized view (same SQL, faster execution)
SELECT * FROM monthly_category_sales
WHERE month >= '2025-01-01';

-- Refresh on demand
REFRESH MATERIALIZED VIEW monthly_category_sales;

-- Refresh concurrently (PostgreSQL 9.4+)—no table lock
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_category_sales;
```

**Eager materialization**: The materialized view is updated immediately when the underlying data changes. Some databases support this (Oracle materialized views with ON COMMIT refresh, SQL Server indexed views). Eager materialization ensures consistency but adds latency to every write transaction.

**Lazy materialization**: The materialized view is updated on a schedule or on demand (REFRESH MATERIALIZED VIEW). Most common pattern. Data is stale between refreshes but writes are not impacted.

**Use case**: Regular reporting queries on frequently changing data. Lazy materialization is the standard approach in data warehouses. Eager materialization is used in specialized scenarios where up-to-date reads are critical.

### Data Warehouse Star Schemas

The dominant pattern for enterprise analytics: a central fact table surrounded by dimension tables. Both fact and dimension tables are denormalized relative to a normalized OLTP schema.

```sql
-- Fact table: contains measures and foreign keys to dimensions
CREATE TABLE fact_sales (
  sale_id BIGINT PRIMARY KEY,
  date_key INT REFERENCES dim_date(date_key),
  product_key INT REFERENCES dim_product(product_key),
  customer_key INT REFERENCES dim_customer(customer_key),
  store_key INT REFERENCES dim_store(store_key),
  quantity INT,
  unit_price DECIMAL(10,2),
  discount DECIMAL(10,2),
  total DECIMAL(10,2)
);

-- Dimension table: denormalized, with all product attributes in one table
CREATE TABLE dim_product (
  product_key INT PRIMARY KEY,
  product_id INT,
  product_name VARCHAR(200),
  category VARCHAR(100),
  subcategory VARCHAR(100),
  brand VARCHAR(100),
  SKU VARCHAR(50),
  unit_cost DECIMAL(10,2),
  current_retail_price DECIMAL(10,2),
  effective_date DATE,
  end_date DATE
);
```

**Use case**: Enterprise business intelligence, reporting, dashboards. Works with every major data warehouse (Snowflake, Redshift, BigQuery, Synapse).

**Trade-off**: The star schema is denormalized by design. Dimensions are wide and flat. Updates to dimension attributes require slowly changing dimension (SCD) strategies rather than simple updates.

### ETL vs ELT

The difference between where the denormalization happens:

**ETL (Extract, Transform, Load)**: Denormalization happens before loading into the target system. A transformation server (e.g., Spark, dbt) reads from source systems, applies joins and aggregations, and writes denormalized tables to the warehouse.

```
[OLTP DB] ──Extract──> [Transform Server] ──Load──> [Data Warehouse]
                |                              |
           Raw tables                    Denormalized tables
```

**ELT (Extract, Load, Transform)**: Raw data is loaded into the warehouse first, then transformed within the warehouse using SQL.

```
[OLTP DB] ──Extract──> [Data Warehouse] ──Transform──> [Denormalized Tables]
                |                              |
           Raw tables                    SQL transforms (dbt, SQL)
```

**Use case**: ETL is better when transformation requires significant compute not available in the data warehouse. ELT is simpler and leverages the warehouse's compute capacity.

## Mental Model Check

Answer these questions to confirm you understand the core concepts.

### Question 1

Your team maintains a normalized schema with customers, orders, and products. The analytics team runs a daily report that JOINs all three tables and aggregates by month. The query takes 45 seconds. Why is this happening?

**Answer**: The normalized schema requires the database to JOIN tables on every query execution. For large tables (millions of rows), JOIN operations create intermediate result sets that consume memory and disk. The aggregation (GROUP BY) then scans the intermediate set. Denormalization would eliminate the JOINs by storing the pre-joined data in a single flat table.

### Question 2

What is the difference between a materialized view and a regular view?

**Answer**: A regular view is a saved SQL query — every time you query it, the database executes the underlying query against the base tables. A materialized view stores the results as a physical table. Querying a materialized view reads the stored data directly, without executing the underlying JOINs. The trade-off is that a materialized view can become stale if the base tables change.

### Question 3

Your system has a read-to-write ratio of 1:1 (equal reads and writes). Should you denormalize?

**Answer**: Probably not. Denormalization shifts work from reads to writes. With equal read and write volume, the write overhead of maintaining redundant copies likely outweighs the read benefit. Consider denormalization only when reads significantly outnumber writes — typically by a factor of 10:1 or more.

### Question 4

A customer changes their shipping address. The denormalized order_report table still shows the old address for two days. Is this a bug?

**Answer**: It depends on the system's consistency requirements. If the system is designed with lazy materialization (batch refresh every night), stale data for two days is expected behavior. The trade-off is accepted: users get fast reads, and the slow background refresh eventually catches up. If the system requires the new address to appear immediately, you need eager materialization or a different architecture.

### Question 5

You are designing a star schema for a retail analytics platform. Should the `dim_product` table contain the category hierarchy (category, subcategory, brand) or should those be separate dimension tables?

**Answer**: In a star schema, the `dim_product` table should contain all product attributes including category hierarchy as columns. This is denormalized by design. The alternative (splitting category into a separate `dim_category` table referenced by a foreign key) would create a snowflake schema, which is a more normalized form. Choose star schema for faster query performance at the cost of redundancy.
