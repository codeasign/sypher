## The Problem Story

A ride-sharing startup launches in three cities. The backend is a single server running on a rented machine: Node.js API, Postgres database, Redis cache. For the first three months this works fine -- the peak is 200 requests per second and the server never breaks 30% CPU. Then the company runs a promotion: "Free rides all Saturday morning." By 9:02 AM the server hits 100% CPU. By 9:03 AM the connection pool is exhausted. By 9:04 AM Postgres stops accepting queries. The app is down for four hours. The promotion cost $80,000 in lost revenue and the CEO learns the word "single point of failure" on a Sunday press call.

This is the load balancer origin story. Every service that grows past a single machine faces the same two questions: how do you split traffic across multiple servers, and how do you keep working when one of them dies.

A second startup watches the first fail and decides to run three identical API servers. They put a round-robin DNS record in front of them. Now traffic is distributed -- but DNS has no awareness of whether a server is alive. One server crashes at 3 AM and DNS keeps sending one-third of requests into a black hole for the next hour (the DNS TTL). Users see intermittent failures. The ops team gets paged at 3:30 AM but by then half the city has switched to the competitor.

A traffic light solves this: it sits at the intersection and directs each car to one of four roads. It knows if a road is blocked (road closed sign) and reroutes around it. It has a predictable pattern that every driver understands. The API servers are the roads. The load balancer is the traffic light -- except the load balancer also sees how many cars each road is handling and can adapt in milliseconds, not on a fixed timer.

## The Insight

A **load balancer** is a traffic cop that sits between clients and servers, distributing incoming requests across a pool of backend instances while continuously checking their health. It abstracts the pool behind a single virtual IP address (VIP), so clients connect to one address and the load balancer decides which backend actually handles each request.

The real-world analogy is a **post office sorting facility**. Mail arrives at the central facility (the VIP). The sorting machines look at each letter's ZIP code, check which delivery trucks are currently at the loading dock, and route the letter onto the correct truck. If a truck breaks down, the sorter knows not to send mail there. If a truck is overloaded, the sorter balances by sending more to emptier trucks. The sender just drops the letter in a mailbox -- they have no idea which truck will carry it or which route the driver takes.

## Core Vocabulary

**Virtual IP (VIP)** -- The single IP address clients connect to. The real backend IPs are hidden behind it. Without a VIP, clients would need to know every backend's IP and manage retries themselves. The VIP decouples the client from the server pool.

**Backend pool** -- The set of servers that process requests. The load balancer maintains this list and updates it as servers are added or removed. If a new server joins the pool during a traffic spike, no client configuration changes -- only the pool list changes.

**Health check** -- A periodic probe (HTTP GET, TCP connect, or ICMP ping) the load balancer sends to each backend. A server that fails three consecutive checks is marked "down" and removed from rotation. Without health checks, load balancers would blindly route to dead servers.

**Scheduling algorithm** -- The rule that picks which backend gets the next request. Round-robin, least connections, IP hash, and consistent hashing are the most common. The choice determines whether traffic spreads evenly or concentrates on a few servers.

**Session persistence (sticky sessions)** -- Ensures all requests from one client go to the same backend. Required when session state is stored in local memory. Without it, a user logged in on server A who hits server B on the next request would appear unauthenticated.

**Connection draining** -- When a backend needs to be taken offline (deploy, maintenance), the load balancer stops sending new connections but lets existing in-flight requests complete. A timeout (30-60 seconds) forces remaining connections closed. Without draining, deployments cause abrupt request failures.

**TLS termination** -- The load balancer decrypts incoming HTTPS traffic and forwards plain HTTP to backends. This offloads CPU-intensive cryptography from application servers and centralizes certificate management. Without it, every backend needs its own TLS certificate and key.

**Active health check vs. passive health check** -- Active: the load balancer probes backends on a fixed schedule. Passive: the load balancer observes real traffic and marks a backend down if it generates too many 5xx errors. Production systems use both.

## How It Actually Works -- Step by Step

A request travels through eight distinct stages inside the load balancer.

### Step 1: Client connects to VIP

The client resolves a DNS name to the VIP (e.g. 1.2.3.4) and opens a TCP connection on port 443.

### Step 3: Protocol inspection

The load balancer parses the HTTP request: method, path, headers, and optionally the body (for layer 7). It extracts whatever fields the routing rules need.

### Step 4: Algorithm selects a backend

The scheduler runs the configured algorithm against the backend pool.

The load balancer opens a new TCP connection to the selected backend (or reuses one from the keep-alive pool) and forwards the request.

### Step 6: Backend processes the request

The backend runs application logic, queries databases, and constructs a response. The load balancer waits (idle timeout is configurable, typically 30-120 seconds).

### Step 7: Response flows back

The backend sends the response to the load balancer. The load balancer may apply transforms: compression, header rewriting, response caching.

### Step 8: Response to client

The load balancer encrypts the response via TLS and sends it to the client over the original connection.

## Variants and Flavors

### Layer 4 (Transport Layer) Load Balancer

Operates at the TCP/UDP level. It sees IP addresses and port numbers but cannot inspect HTTP headers, cookies, or paths.

<AsciiDiagram
  id="load-balancers/l4"
  title="Layer 4 Load Balancer"
  content={`
┌───────────────┐     ┌───────────────────────────────────────────────┐
│   Client      │     │  Layer 4 Load Balancer                        │
│               │     │                                                │
│  TCP:80       │────▶│  ┌────────┐   ┌─────────────┐                 │
│  src:10.0.0.5│     │  │ VIP    │   │ NAT Table   │                 │
│               │     │  │ 1.2.3.4│──▶│              │                 │
│               │     │  └────────┘   │ 10.0.0.5:80 │                 │
│               │     │               │    →         │                 │
│               │     │               │ 10.0.1.2:80 │                 │
│               │     │               └─────────────┘                 │
│               │     │                                                │
│               │     │  Sees: {src_ip, src_port, dst_ip, dst_port,   │
│               │     │         protocol}                              │
│               │     │  Cannot: inspect HTTP headers, cookies, paths  │
└───────────────┘     └───────────────────────────────────────────────┘
`}
  caption="Layer 4 LB is fast (kernel-space NAT/DPDK) but blind to application content. Best for TCP/UDP traffic that does not need content-aware routing."
/>

**Use when**: You need raw throughput and low latency. TLS termination happens at the backend. Examples: game servers, DNS, database read replicas, streaming video.

**Do not use when**: You need content-based routing (/api/v1 vs /api/v2), sticky sessions based on cookies, or rate limiting per URL path.

Operates at the HTTP/HTTPS level. Inspects method, path, headers, cookies, and sometimes body.

**Use when**: You need path-based routing, cookie-based session persistence, rate limiting per endpoint, or TLS termination. Examples: web APIs, microservices gateways, e-commerce sites.

**Do not use when**: You are optimizing for maximum throughput on homogeneous traffic, or your backend handles TLS natively and you want to reduce hops.

### DNS Load Balancing

Not a dedicated appliance. DNS returns multiple IP addresses for one hostname, and clients pick one at random (or round-robin at the resolver level).

**Use when**: Geographic distribution (return different IPs per region), or as a fallback/first-pass distribution layer.

**Do not use when**: You need sub-second failover. DNS TTLs mean clients cache dead IPs for minutes. Always pair DNS LB with a real load balancer behind the VIPs.

## Mental Model Check

**Q1: Your application uses sticky sessions. A server crashes. What happens to users on that server?**

All users whose sessions were on the crashed server lose their session state. The load balancer cannot redirect them because the session data is gone. Solution: use a shared session store (Redis, Memcached) instead of local memory, or use a stateless token (JWT) that any server can validate.

**Q2: A health check sends a TCP SYN to port 3000 every 10 seconds. The backend accepts the SYN but the application is deadlocked. Is the backend healthy?**

No. A TCP health check only verifies that the kernel accepts connections. An HTTP health check would have received a timeout or a 500 error. This is why production systems use application-level health checks (GET /health) that verify database connectivity, cache connectivity, and internal state.

**Q3: You have 5 servers. One becomes 2x slower due to a memory leak. Round-robin keeps sending it equal traffic. What happens?**

The slow server accumulates a queue of pending requests. Response times degrade for everyone because the slow server holds connections open longer. "Least connections" scheduling would detect the backlog and route fewer requests to the slow server.

**Q4: You configure connection draining with a 30-second timeout. A deploy script sends SIGTERM to the old server. What happens in each second?**

The load balancer marks the server as "draining" and stops sending new requests. Existing requests have up to 30 seconds to complete. If a long-running request (e.g., file upload) takes 40 seconds, it gets forcibly disconnected at the 30-second mark. Best practice: set draining timeout to 2x your p99 response time.

**Q5: Clients behind a corporate NAT (5000 users, one public IP) all hit your service. Sticky sessions are based on source IP hash. What breaks?**

All 5000 users map to the same backend server because they share one public IP. That server handles 100% of the traffic while the others sit idle. Solution: use cookie-based stickiness instead of IP hash, or use consistent hashing on a per-request token (session cookie, user ID).
