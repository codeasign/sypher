## Data Structures and Algorithms

### Backend Pool (Hash Map + Sorted List)

The load balancer maintains a hash map keyed by backend ID for O(1) lookup and a separate ordered list for round-robin iteration.

| Property | Value |
|---|---|
| Structure | `map[BackendID]*Backend` + `[]*Backend` |
| Lookup | O(1) |
| Insert/Remove | O(1) for hash map, O(N) for round-robin list rebuild |
| Memory | ~200 bytes per backend |

When a backend is added or removed, the list is rebuilt. This happens rarely (deployments, auto-scaling events) so O(N) rebuild is acceptable.

### Weighted Round-Robin Sequence

Each backend has an integer weight. A backend with weight 3 gets three requests before the scheduler moves to the next backend with weight 1.

| Backend | Weight | Requests per cycle |
|---|---|---|
| A (new, fast) | 5 | 5 |
| B (old, slow) | 1 | 1 |
| C (medium) | 2 | 2 |

A smooth weighted round-robin (used by Nginx) distributes bursts evenly rather than sending all 5 A requests consecutively. The sequence for weights {5, 1, 2} is: A, A, C, A, B, A, C, A.

**Step 1-3**: TCP three-way handshake between client and load balancer. The load balancer's network listener (epoll/kqueue/IOCP) accepts the connection in an event-driven loop. Memory allocated: ~4 KB per connection for socket buffers.

**Step 4**: TLS handshake (one round trip for TLS 1.3, two for TLS 1.2). The load balancer selects the certificate based on SNI (Server Name Indication). Total TLS overhead: 1-3 ms with hardware acceleration, 5-15 ms with software crypto.

**Step 5**: Protocol parsing. For HTTP/1.1, the parser reads the request line and headers into a buffer. An inefficient or malicious request (very long headers, no body termination) is caught here and rejected with 400 Bad Request. Buffer size limit: typically 8-16 KB for headers.

**Step 6**: Scheduler selects a backend. The algorithm runs in O(log N) (min-heap) or O(1) (round-robin with current index pointer). This step adds under 0.001 ms to the critical path.

**Step 7**: Connection to backend. If a keep-alive connection exists in the idle pool, it is reused (no TCP handshake). Otherwise, a new TCP connection is opened. Adding new connections adds latency: ~0.5-2 ms for local network, 5-50 ms across regions.

**Step 8-10**: Backend TCP handshake. The load balancer has a separate event loop for backend connections. Connections time out after a configurable interval (default 5 seconds for connect timeout).

**Step 11-14**: Request forward, backend processing, response return, TLS encryption, response send to client.

### Connection Pool Internals

The connection manager maintains two data structures per backend:

**Idle Pool**: A stack (LIFO) of connections ready for reuse. LIFO maximizes cache locality -- the same CPU core that last used the connection likely still has TCP state in its cache.

**Active Set**: A hash set of connections currently serving a request. When a response arrives, the connection moves from active back to idle.

```
Active: {conn_7, conn_12, conn_19}   -- in use
Idle:   [conn_4, conn_2, conn_15]    -- LIFO stack, conn_4 reused next
```

**Pool limits per backend**:
- `max_connections`: total connections this load balancer opens to one backend. Default: unlimited (risky) or 1024 (safe).
- `max_keepalive_connections`: idle connections kept open. Default: 32.
- When `max_connections` is reached, new requests queue and wait. Queue depth and timeout prevent unbounded accumulation.

## Performance Characteristics

Measured on an m5.xlarge (4 vCPU, 16 GB RAM, 10 Gbps NIC) running HAProxy or Nginx:

| Operation | Latency (p50) | Latency (p99) | p99/p50 ratio |
|---|---|---|---|
| TCP proxy (L4), no TLS | 0.05 ms | 0.12 ms | 2.4x |
| TCP proxy (L4), with TLS | 0.8 ms | 3.2 ms | 4.0x |
| HTTP proxy (L7), no TLS | 0.15 ms | 0.40 ms | 2.7x |
| HTTP proxy (L7), with TLS | 1.1 ms | 4.5 ms | 4.1x |
| HTTP proxy + health check | 0.20 ms | 0.60 ms | 3.0x |
| gRPC proxy (L7), with TLS | 0.9 ms | 4.0 ms | 4.4x |

**Throughput** (same hardware):

| Configuration | Requests/second | Connections | Memory |
|---|---|---|---|
| L4, no logging | 250,000 rps | 100,000 | 450 MB |
| L4, access log | 180,000 rps | 100,000 | 520 MB |
| L7, no logging | 120,000 rps | 50,000 | 680 MB |
| L7, access log | 65,000 rps | 50,000 | 780 MB |
| L7, TLS + logging | 40,000 rps | 25,000 | 920 MB |

**Memory per connection**: ~4-8 KB for socket buffers + ~200 bytes for connection state object. At 500,000 concurrent connections: 2-4 GB of RAM just for connection tracking.

**CPU breakdown** (L7 with TLS): 35% TLS crypto, 25% protocol parsing, 15% kernel networking, 10% scheduler, 10% logging, 5% health checking.

## Configuration and Tuning

### Core Settings

| Setting | Default | What it controls | When to change | What breaks |
|---|---|---|---|---|
| `maxconn` | 2000 | Maximum concurrent connections | Server has >5000 concurrent connections | Connection refused above limit |
| `timeout connect` | 5s | Max time to connect to backend | Backend is cross-region (increase to 10s) | Requests fail with 503 if too low |
| `timeout server` | 50s | Max time to wait for backend response | Backend has long polling (increase to 300s) | Long requests timeout prematurely |
| `timeout client` | 50s | Max idle time on client connection | Public API clients are slow (increase to 120s) | Clients get disconnected mid-upload |
| `timeout keep-alive` | 60s | How long to keep idle connections | Short-lived connections (decrease to 5s) | Resource leaks, too many open FDs |
| `retries` | 3 | Number of retry attempts on failure | Idempotent endpoints only (keep at 3) | Duplicate writes on POST |
| `max-keep-alive-queue` | 100 | Queue requests when all conns busy | High traffic backend (increase to 500) | Request queuing delays |
| `check-interval` | 10s | Time between health checks | Need faster failover (decrease to 2s) | Unnecessary network noise, false positives |
| `check-fall` | 3 | Consecutive failures to mark down | Flaky network (increase to 5) | Slow failover during real failures |
| `check-rise` | 2 | Consecutive successes to mark up | Unstable recovery (increase to 5) | Backend flaps in and out of pool |

### TLS Tuning

| Setting | Default | Effect |
|---|---|---|
| `ssl-min-ver` | TLSv1.2 | Disables TLS 1.0/1.1 (PCI DSS compliance) |
| `ssl-ciphers` | ECDHE+AESGCM | Forward secrecy, hardware-accelerated |
| `ssl-engine` | (none) | Offload to QAT or AES-NI hardware |
| `ssl-session-cache` | true | Cache TLS session tickets, reduce handshake overhead by 60% |
| `ssl-session-cache-timeout` | 300s | Purge stale sessions |

### Logging Impact

Access logging is the single largest performance drain on a load balancer. Each log line requires a write syscall (or buffer flush). At 100,000 rps, buffered logging writes ~20 MB/s. Unbuffered (sync) logging drops throughput by 50-70%.

**Recommendations**:
- Use buffered logging with `log-buffer-size 4096` (HAProxy) or `buffer` directive (Nginx)
- Sample logs at 1:100 for high-traffic endpoints
- Send logs to a separate disk or over UDP to a log aggregator
- Disable access logs in the hot path and rely on metrics export (Prometheus/statsd)

### Resource Limits

Linux kernel tuning for high-connection load balancers:

```
# /etc/sysctl.conf  (required above 100,000 connections)

net.core.somaxconn = 65535           # listen backlog
net.ipv4.tcp_max_syn_backlog = 65535 # SYN queue depth
net.core.rmem_max = 16777216         # max TCP receive buffer (16 MB)
net.core.wmem_max = 16777216         # max TCP send buffer (16 MB)
net.ipv4.tcp_tw_reuse = 1            # reuse TIME_WAIT sockets
net.ipv4.ip_local_port_range = 1024 65535  # ephemeral port range
fs.file-max = 1000000                # max file descriptors
```

Each connection consumes one file descriptor. A load balancer handling 500,000 connections needs `fs.file-max >= 500000 + headroom`.
