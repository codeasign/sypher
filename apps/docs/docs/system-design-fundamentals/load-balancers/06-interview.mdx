## How This Topic Appears

Load balancers are never the headline topic in a system design interview, but they are almost always part of the solution. Expect to discuss them in the context of a larger design or as a focused follow-up question.

### Direct Questions

1. **"Design a rate-limiting load balancer for an API that serves 10,000 clients."** -- Tests understanding of rate limiting at the LB layer vs. application layer. Specifically asking about per-client limits, token bucket vs. sliding window, and what happens when limits are exceeded.

2. **"How would you distribute traffic to a global user base with under 100ms latency?"** -- Tests multi-region load balancing, anycast vs. DNS, regional failover, and data replication.

3. **"Compare L4 and L7 load balancers. When would you use each?"** -- A foundational question that tests whether you understand the OSI model layers and the throughput vs. intelligence trade-off.

4. **"Our service has 5 backend servers. One keeps failing. What do you do?"** -- Tests health checking, circuit breakers, connection draining, and the difference between detection and prevention.

5. **"How does a load balancer know a backend is unhealthy?"** -- Tests active vs. passive health checks, timeouts, and the implications of false positives/negatives.

### Embedded Questions (appear within larger design problems)

6. **"Design WhatsApp."** -- The load balancer sub-question: "How do you ensure a user's WebSocket connection stays pinned to the same server?"

7. **"Design YouTube."** -- The load balancer sub-question: "How do you route users to the nearest video-serving edge server?"

8. **"Design a key-value store."** -- The load balancer sub-question: "How do you distribute read traffic across replicas without overloading any single replica?"

9. **"Design Twitter."** -- The load balancer sub-question: "How do you split read-heavy vs write-heavy traffic?"

10. **"Design Uber."** -- The load balancer sub-question: "How do you ensure that all requests for a specific geographic area go to the same backend?"

### Follow-Up Probes (after you give an initial answer)

11. "What happens when your load balancer's certificate expires at 3 AM?"
12. "You have 10 servers and 28,000 ephemeral ports. What breaks at 30,000 connections?"
13. "How do you deploy a new version of the load balancer without dropping connections?"
14. "Your health check passes but users get errors. What's happening?"
15. "You have 100ms of DNS TTL remaining. The LB fails. What is the user experience?"

## What the Interviewer Is Actually Testing

| Question type | What they look for | Red flag |
|---|---|---|
| "Design a rate-limiting LB" | Do you know token bucket vs. sliding window? Can you reason about memory for 10,000 counters? | Saying "use a library" without explaining the data structure. |
| "Global traffic distribution" | Do you understand anycast, BGP, DNS latency? Can you calculate cross-region replication lag? | Only mentioning round-robin DNS. |
| "L4 vs L7" | Can you articulate the trade-off in concrete terms with numbers? | Vague answer: "L7 is more flexible." |
| "Handling backend failures" | Do you mention circuit breakers, connection draining, slow start? | "The LB automatically handles it." |
| "Health checks" | Can you explain the pros and cons of different check types? | Only describing TCP health checks. |
| Embedded (WhatsApp, YouTube) | Do you realize the question is about load balancing even though it is not explicitly named? | Discussing database replication instead of connection stickiness. |

## Level-by-Level Expectations

### L3/L4 (Entry-Level / New Grad)

**What they expect**: You can define a load balancer and list two algorithms. You know the basic difference between L4 and L7. You can describe round-robin.

**Sample answer to "What is a load balancer?"**:
"A load balancer sits between clients and servers. It accepts incoming requests and distributes them across a pool of backend servers using an algorithm. The most common algorithm is round-robin, where each server gets one request in turn. If a server crashes, the load balancer stops sending requests to it. This makes the application more reliable and scalable because you can add more servers without changing client configuration."

**Sample answer to "What is the difference between L4 and L7?"**:
"L4 load balancers work at the transport layer. They see IP addresses and port numbers but do not inspect the HTTP request content. They are very fast because they just forward TCP packets. L7 load balancers work at the application layer. They can read HTTP headers, cookies, and even the request body. This allows them to route based on URL paths, like sending /api/v1/ to one group of servers and /api/v2/ to another. L7 is slower because of the overhead of parsing HTTP, but more flexible."

### L5 (Senior Engineer)

**What they expect**: You can design a multi-tier load balancing architecture. You know specific algorithms, their time complexities, and when to use each. You have opinions on configuration tuning. You can calculate capacity.

**Sample answer to "Design a global load balancing system for 50 million users"**:
"I would start with anycast routing to direct users to the nearest region. Each region has a three-tier LB stack: a cloud L4 load balancer (AWS NLB or GCP External LB) for DDoS protection, then an internal L7 LB (Nginx or Envoy) for path-based routing and rate limiting, then application load balancers for each microservice. Cross-region traffic uses DNS-based failover with a 30-second TTL. For data, each region has its own database replica with asynchronous replication from a primary region. I would size the L7 tier at 4 c5.4xlarge instances per region, which gives us about 160,000 req/s of L7 capacity with TLS. That gives us headroom for a single-region failure. Health checks run every 5 seconds with 3 failures to mark down. Circuit breakers eject any backend that exceeds 500 active connections or a p99 latency of 1 second."

**Sample answer to "How would you implement sticky sessions at scale?"**:
"I would avoid sticky sessions if possible by externalizing session state to Redis. Each request carries a session token (JWT or opaque session ID) and any backend can look up the session from the shared store. If I had to use sticky sessions for a legacy system, I would use cookie-based stickiness: the LB sets a Set-Cookie header with an encrypted backend identifier on the first request. The client sends this cookie on subsequent requests, and the LB decrypts it to route to the correct backend. If that backend is down, the LB routes to the least-loaded backend despite the cookie, and the application handles session recovery. I would not use IP hash-based stickiness because it breaks behind NAT and proxies."

### L6 (Staff Engineer)

**What they expect**: You can discuss the control plane vs. data plane separation, the CAP trade-off, distributed consensus for health state, and failure modes across multiple regions. You bring up auto-scaling integration, deployment strategies, and cost optimization.

**Sample answer to "How do you ensure load balancer configuration is consistent across 100 instances?"**:
"I would separate the control plane from the data plane. The data plane is stateless -- it has a loaded configuration and forwards packets. The control plane is a distributed configuration service, typically etcd or Consul, with a consensus-based consistency model. Each LB instance runs an agent that watches the control plane for changes. When a change is detected, the agent validates the new config (syntax check, canary deploy to a single instance), then triggers a seamless reload on that instance. A bad config rolls back automatically. The control plane also tracks which LB instances have successfully applied the latest config version. If an instance falls behind by more than two versions, it is quarantined until it catches up. This design gives us CP on the control plane and AP on the data plane."

**Sample answer to "How do you handle a regional load balancer failure?"**:
"I design every region to be independent. If US-East fails, EU-West must keep serving users without data from US-East. This means each region has its own load balancer stack, its own database replicas, and its own cache cluster. Cross-region replication is asynchronous, so there is some data staleness (typically 200-800ms). The failover is handled by DNS health checks: Route 53 or equivalent monitors each region's global VIP (the cloud L4 LB's IP). When the health check fails for 30 seconds, the DNS record for that region is removed. Clients retry DNS after their TTL expires (we set it to 30 seconds), hitting the next-closest region. Users experience a 30-90 second interruption and possibly a few seconds of stale data, but the service does not go completely down."

### L7/L8 (Principal / Distinguished)

**What they expect**: You discuss the system from a cost/benefit perspective across the entire organization. You bring up hardware acceleration, kernel bypass, and custom implementations. You have strong opinions backed by data.

**Sample insight**: "At 500,000 req/s, the TLS overhead costs us $200,000/year in compute. I would offload TLS to a dedicated hardware accelerator (QAT card) or move to a cloud LB that includes TLS termination in its pricing. At 2 million req/s, I would evaluate DPDK or XDP for kernel bypass, which reclaims 30% of CPU cycles from the networking stack."

**Sample insight**: "The real cost of a self-managed load balancer is not the hardware. It is the 2-3 on-call rotations per month dedicated to LB incidents, the 3-hour debugging session when a kernel tuning parameter causes packet loss, and the distraction from product engineering. I would pay the cloud provider premium until the monthly cloud LB bill exceeds $10,000. At that scale, building an internal LB team starts to make sense."

**Sample insight**: "The single best investment for LB reliability is not DPDK or custom hardware. It is a solid canary deployment pipeline for LB config changes. 80% of LB incidents at our scale were caused by a bad config change that passed syntax validation but caused a runtime failure under load. A canary pipeline that sends 1% of traffic to the new config, measures p50/p99/p999 latency and error rate for 60 seconds, and auto-rolls back on any regression would have prevented every one of those incidents."

## The FAANG Framework for Answering

When the interview asks a load-balancer-related question, structure your answer in six steps.

### Step 1: Clarify scope and constraints

"What scale are we designing for? Requests per second, total users, geographic distribution? Is this HTTP traffic or TCP? Are the backends stateless or stateful? What is the budget for LB latency overhead?"

### Step 2: State the core trade-off

"Every load balancer decision is a trade-off between throughput and intelligence. L4 gives me millions of packets per second but no content awareness. L7 gives me rich routing but costs microseconds per request and more memory per connection."

### Step 3: Name the algorithm and why

"For this use case, I would use least-connections because request processing time varies from 5ms to 500ms and we want to avoid overloading slow backends. If we needed session pinning, I would switch to consistent hashing."

### Step 4: Draw the architecture

"Mentally, I am drawing two load balancers in active-active mode, health-checking against a shared etcd-based config store. Each LB has the following internal pipeline..."

### Step 5: Calculate capacity

"With 50,000 req/s and an average response of 25 KB, each LB instance needs 2.5 Gbps throughput. Two instances handle this with 50% headroom. Cost is approximately $800/month in compute and $400/month in cloud LB fees."

### Step 6: Identify failure modes and mitigations

"The main risks are: the LB itself fails (mitigated by active-active), a backend becomes slow and cascades (mitigated by circuit breakers and maxconn limits), and DNS caching causes stale routing after failover (mitigated by 30-second TTLs or anycast)."

## Phrase Bank

| Context | Phrase | Why it works |
|---|---|---|
| Starting an answer | "Before choosing a load balancer strategy, I need to clarify whether the workload is read-heavy or write-heavy, and whether backends are stateful." | Shows you gather requirements before proposing solutions. |
| Comparing L4 and L7 | "L4 operates on IP and port -- it has no concept of an HTTP request. L7 parses HTTP, so it can route based on URL paths, cookies, and headers. The latency cost is roughly 100-200 microseconds per request." | Gives a concrete, numeric comparison. |
| Explaining health checks | "A TCP health check validates the kernel is accepting connections. An HTTP health check validates the application is processing requests. A dependency health check validates the database and cache are reachable. Each level catches different failure modes." | Shows depth of understanding. |
| Handling failures | "I use circuit breakers with a 50% error threshold over a 10-second window. When the breaker trips, the backend is removed from the pool for 30 seconds, then half-opened for traffic probing." | Demonstrates practical, concrete design. |
| Capacity planning | "At 10,000 req/s with 25 KB responses, bandwidth is the bottleneck before CPU. That is 2.5 Gbps, which saturates one 10 Gbps NIC at 25% utilization." | Shows you consider real-world constraints. |
| Session persistence | "I prefer stateless backends with an external session store. Sticky sessions couple client routing to server topology, which breaks during deployments and auto-scaling." | Shows preference for scalable architecture. |
| Global routing | "Anycast gives us a single anycast IP that routes to the nearest healthy region in under 10 seconds. DNS-based failover is simpler but adds 60+ seconds of downtime during a failover." | Compares approaches with concrete trade-offs. |
| Cost optimization | "The marginal cost of one more LB instance is $X/month. The marginal cost of one more backend instance is $Y/month. At this scale, it is cheaper to buy LB capacity than to absorb the latency from overloaded backends." | Demonstrates business awareness. |
| Deployment | "I use blue-green LB deployments: spin up a new LB fleet behind the cloud NLB, validate the new fleet, switch the NLB target group, then drain the old fleet." | Shows operational maturity. |
| Control plane | "The control plane (config, health state) must be CP -- split-brain health status causes cascading failures. The data plane (packet forwarding) must be AP -- dropping packets during a partition is unacceptable." | Shows understanding of distributed systems theory. |

## Red Flags

**Red flag 1: "Round-robin works for everything."** -- Shows you do not understand request variability or stateful backends. Interviewer immediately probes with "what if one server is slower?" and you should have a least-connections answer ready.

**Red flag 2: "I would use sticky sessions so requests always hit the same backend."** -- Without qualification, this suggests you tolerate tight coupling between routing and server topology. The correct answer is "I would avoid sticky sessions by externalizing state. If I needed them, I would use cookie-based stickiness."

**Red flag 3: "Enable retries for everything."** -- Indicates you have never seen a retry storm. The correct answer distinguishes idempotent (GET, PUT, DELETE) from non-idempotent (POST) requests and limits retries to 1-2 with exponential backoff.

**Red flag 4: "The load balancer handles failover automatically."** -- Vague. A real answer specifies: health check interval, consecutive failures to mark down, what happens to in-flight requests, and how clients learn about the failover (DNS TTL, anycast withdrawal).

**Red flag 5: "Health checks just ping the server."** -- Shows shallow understanding. The interviewer will ask "what if the process is alive but the application is deadlocked?" The correct answer describes multi-layer health checks: TCP, HTTP, and dependency verification.

**Red flag 6: "We need maxconn as high as possible for scalability."** -- Indicates unfamiliarity with cascading failures and backpressure. The correct answer sets finite per-backend limits and a circuit breaker.

**Red flag 7: "DNS TTL of 5 minutes is fine."** -- Demonstrates no experience with failover scenarios. In production, a 5-minute DNS TTL means 5 minutes of downtime during failover. The industry standard for critical services is 30-60 seconds.

**Red flag 8: "A load balancer is just Nginx."** -- Shows you know a single implementation but not the concept. The interviewer will test whether you understand load balancers abstractly -- the trade-offs, algorithms, failure modes -- not just one config file format.

**Red flag 9: Never mentions connection draining during deployments.** -- Indicates inexperience with zero-downtime deployments. You must explain how a server is gracefully removed from the pool before being taken offline.

**Red flag 10: "We will just buy a hardware load balancer."** -- At a startup or mid-size company, this signals over-engineering and budget insensitivity. At FAANG scale, hardware LBs are sometimes warranted (F5, A10), but you must justify the FIPS compliance or throughput guarantees.

## Practice Drills

### Drill 1: 2 Minutes -- "Define load balancer, list 3 algorithms, and give a scenario for each."

Set a timer. Practice saying: "A load balancer distributes traffic across backend servers. Round-robin sends one request per server in order -- good for identical servers. Least-connections sends to the server with fewest active connections -- good for variable request processing times. Consistent hashing maps requests to servers based on a key -- good for cache locality."

### Drill 2: 5 Minutes -- "Your API service has 50,000 req/s average, 3 backend servers. One server dies. What happens?"

Practice tracing the sequence: health check detects failure in 15-30 seconds, server removed from pool, remaining 2 servers now handle 25,000 req/s each, if max capacity is 20,000 req/s per server, each server is 25% overloaded, latency increases, queue builds, if no circuit breaker, cascade follows. Mention auto-scaling triggers. Mention DNS if clients use multiple VIPs.

### Drill 3: 10 Minutes -- "Design a load balancing layer for a real-time chat app with 10 million concurrent WebSocket connections."

Sketch on paper or mental: Anycast routing to 5 regions. Cloud L4 LB per region for DDoS. Nginx for WebSocket termination with cookie-based stickiness (user session pinned to a chat server). Each chat server maintains 50,000-100,000 WebSocket connections. Health checks verify WebSocket endpoint. Connection draining (60 seconds) for rolling restarts. Redis for session metadata (not the connection itself). Circuit breaker for chat servers with degraded latency.

### Drill 4: 5 Minutes -- "What metrics would you monitor on a load balancer, and what thresholds trigger an alert?"

List: active connections (alert >80% of max), p99 response time (alert >500ms), 5xx rate (alert >1%), health check failure rate (alert >5%), TLS handshake time (alert >100ms p99), connection drop rate (alert >0), queue depth per backend (alert >100), backend up/down transitions (alert >3 flips in 10 minutes).

### Drill 5: 2 Minutes -- "Explain the CAP trade-off for load balancers."

Practice: "The data plane is AP -- it must forward packets even during a network partition. Dropping packets is worse than forwarding to a potentially dead backend. The control plane is CP -- we use etcd or Zookeeper to agree on which backends are healthy. Split-brain health state could cause all traffic to concentrate on a subset of backends, triggering a cascade."
