**Scale**: 1.2+ billion requests per day during peak hours. Over 200 microservices with dynamic instance counts.

**Problem**: Traditional round-robin between microservices caused cascading failures. When one instance of the "User Service" slowed down, Zuul's default round-robin kept sending it equal traffic, degrading p99 latency for all User Service requests.

**Solution**: Zuul 2 uses a weighted load balancing strategy where each instance's weight is dynamically adjusted based on real-time latency measurements. An instance that responds in 50ms gets higher weight than one responding in 500ms. Eureka's health checks remove unhealthy instances in under 10 seconds.

**Result**: p99 latency dropped from 800ms to 120ms. Instance failures no longer cascade. The dynamic weighting absorbs burst traffic without operator intervention.

**Scale**: 15+ million trips per day, 500+ cities, thousands of microservice instances.

**Problem**: Simple round-robin routing destroyed cache locality. Each node cached city-specific data (drivers, pricing, maps), but a request for San Francisco hit a different node each time, resulting in a 90%+ cache miss rate.

**Solution**: Ringpop implements a consistent hash ring in the application layer. Each node claims a segment of the ring. Requests carry their city ID, and the ring maps the city ID to the owning node. Cache hit rate went from 8% to 85%.

**Result**: Database read load dropped by 77%. p50 latency fell from 45ms to 12ms. Adding or removing nodes only required 1/N of cache entries to be rebalanced.

**Scale**: 150+ million monthly active users. 5+ million concurrent WebSocket connections. 25,000+ server instances.

**Problem**: WebSocket connections last hours or days. Traditional TCP load balancing distributes connections across all available servers, but each client must reconnect to the same session server to maintain guild state. A disconnect during a voice call drops the call.

**Solution**: Nginx uses cookie-based stickiness. On first connection, Nginx sets a `Set-Cookie: route=backend-3`. On reconnect, the client sends the cookie and Nginx routes to `backend-3`. If `backend-3` is down, Nginx routes to the least-loaded available backend.

**Result**: Socket disconnects during maintenance dropped to near zero. The sticky routing allows 24+ hour sessions without rebinding. Nginx alone handles 5+ million concurrent connections on modest hardware.

**Scale**: 20%+ of global web traffic. 330+ cities. 10 million+ requests per second at peak.

**Problem**: Traditional load balancers cannot handle Cloudflare's scale. A single hash ring server would be a bottleneck. Spreading across edge servers required each server to independently make the same routing decisions.

**Solution**: Unimog uses MAGLEV hashing -- a variant of consistent hashing with O(1) lookup (better ring walk) and deterministic behavior across all edge servers. Every edge server independently computes the same backend mapping for a given request, so no coordination is needed.

**Result**: Each edge server independently handles 10 million+ packets per second. No centralized load balancer layer. Adding or removing backends requires no config changes: the hash ring automatically adjusts.

## Open Source Implementations

### Nginx

- **Type**: L7 reverse proxy and load balancer
- **Algorithm**: Round-robin (default), least connections, IP hash, random with two choices
- **Health checks**: Passive (error count triggers removal), active with commercial version
- **Performance**: 40,000-120,000 req/s per instance (L7 with TLS)
- **Config model**: Static file, hot reload via SIGHUP
- **Best for**: HTTP/HTTPS web apps, TLS termination, static file serving with LB
- **Weakness**: No built-in service discovery. Must use DNS or external tool (Consul Template, etcd)

**Nginx snippet**:
```nginx
upstream api_backend {
    least_conn;
    server 10.0.1.2:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.3:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.4:8080 backup;
}

server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;

    location / {
        proxy_pass http://api_backend;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

### HAProxy

- **Type**: L4 and L7 load balancer (TCP + HTTP)
- **Algorithm**: Round-robin, least connections, first (queue-based), source hash, URI hash, hdr() hash
- **Health checks**: Active (HTTP GET/TCP connect/SSL check/LDAP check), passive with observability
- **Performance**: 100,000-250,000 req/s (L4), 50,000-100,000 req/s (L7) per instance
- **Config model**: Static file, seamless reload via `haproxy -sf` (spawn new process, signal old)
- **Best for**: TCP services (databases, message queues), high-throughput L4, WebSocket gateways
- **Weakness**: No native HTTP/2 multiplexing (added in HAProxy 2.0, but not as mature as Envoy)

**HAProxy snippet**:
```haproxy
global
    maxconn 40000
    tune.bufsize 32768

defaults
    mode tcp
    timeout connect 5s
    timeout client 50s
    timeout server 50s
    retries 3

frontend db_read_only
    bind *:5432
    default_backend postgres_replicas

backend postgres_replicas
    balance leastconn
    option tcp-check
    tcp-check connect port 5432
    server replica1 10.0.1.10:5432 check inter 5s fall 3 rise 2
    server replica2 10.0.1.11:5432 check inter 5s fall 3 rise 2
    server replica3 10.0.1.12:5432 check inter 5s fall 3 rise 2 backup
```

### Envoy (Lyft/CNCF)

- **Type**: L7 proxy and service mesh sidecar
- **Algorithm**: Weighted round-robin, least request (with peak EWMA), consistent hashing, random
- **Health checks**: Active (HTTP, TCP, gRPC), passive (outlier detection), degradation endpoint
- **Performance**: 30,000-70,000 req/s per worker (primed for mesh -- richer features = higher cost)
- **Config model**: Dynamic via xDS API (control plane pushes config to data plane), hot reload
- **Best for**: Service mesh (Istio, Consul Connect), gRPC-heavy systems, dynamic routing with circuit breaking
- **Weakness**: Complex configuration. Requires understanding of xDS, ADS, and layer-7 filtering. Overkill for simple setups.

### Key Differences

**AWS NLB vs ALB**: NLB forwards TCP/UDP with minimal overhead (kernel bypass via AWS Nitro). No headers, no cookies, no path routing. ALB understands HTTP/2, WebSockets, gRPC, and path-based routing. NLB costs $5.84/LCU-month base, ALB is slightly cheaper per-request but more expensive at scale.

**GCP HTTP(S) Load Balancer**: Global single anycast IP. One load balancer spans all regions. This is unique -- AWS and Azure require separate LBs per region with DNS routing between them. The GCP LB does TLS termination at the edge and routes via URL maps.

**Azure Traffic Manager**: DNS-based load balancer only, not a real LB. Routes based on performance, geographic, priority, or weighted methods. Must pair with Azure Load Balancer (L4) or Application Gateway (L7) for actual traffic distribution.

## Lessons Learned

**Lesson 1: Health checks must test real behavior.** Every production outage traced to a load balancer involved a health check that passed but the application was broken. The health endpoint should query the database, check cache connectivity, and verify the service can process a transaction. Lightweight health checks are a false sense of security.

**Lesson 2: Circuit break before you cascade.** A single slow backend should be ejected from the pool before it degrades every other backend. Set max connections low (200-500 per backend). Queue requests briefly, then fail over. Do not let one dying server take down the cluster.

**Lesson 3: Consistent hashing beats round-robin for stateful workloads.** When backends have local caches or session state, consistent hashing (Ringpop, MAGLEV) delivers 10x better cache hit rates than round-robin. The cost is slightly higher complexity in membership management.

**Lesson 4: Anycast routing solves the DNS failover problem.** DNS-based failover has a minimum failover time of (TTL + client retry interval) -- typically 60-300 seconds. Anycast BGP failover completes in under 10 seconds because the network layer detects the route withdrawal. Cloud providers charge a premium for anycast IPs, but the reliability gain is worth it for user-facing services.

**Lesson 5: Measure every layer separately.** The load balancer's p50 latency may be 2ms while the backend's p50 latency is 100ms. During incidents, teams often blame the load balancer when the backend is the real problem. Instrument both sides independently (x-request-id tracing, timing headers like `X-Request-Start`).
