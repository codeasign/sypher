## Naive Implementation

The simplest load balancer is a reverse proxy configuration file on a single machine. It works for a few thousand users and fails in predictable ways as scale increases.

### What breaks at 1,000 users

Nothing. A single Nginx instance can handle 10,000-50,000 concurrent connections on modest hardware. At this scale, the naive implementation is sufficient.

### What breaks at 100,000 users

- **Single point of failure**: If the load balancer machine crashes, the entire service goes down. No DNS failover because the VIP is bound to one machine.
- **Connection exhaustion**: The load balancer's kernel buffer runs out. SYN packets get dropped. New clients see connection refused.
- **TLS bottleneck**: TLS decryption consumes 2-3 CPU cores at 10,000 new TLS handshakes per second. The load balancer CPU hits 100% while backends sit at 20% utilization.
- **No graceful scaling**: Adding more backends requires the load balancer to be reconfigured and restarted, causing a brief outage.

### What breaks at 10 million users

- **Network bandwidth**: A single 10 Gbps NIC saturates at 10 million concurrent users even at modest request sizes. Average HTTP response of 50 KB at 50,000 req/s = 20 Gbps -- needs two load balancers just for bandwidth.
- **File descriptor limit**: 500,000 concurrent connections × 2 (client side + backend side) = 1,000,000 file descriptors. Only 655,358 if the kernel is not tuned.
- **Ephemeral port exhaustion**: The load balancer opens connections to backends using ephemeral ports. With default port range 32768-60999 (28,232 ports), only 28,232 distinct backend connections can exist simultaneously before ports are exhausted.
- **DNS caching issues**: If the load balancer's DNS resolution for backend names changes, stale DNS entries cause routing errors.

### Key Design Decisions in Production

**Active-active, not active-standby**: Both load balancers serve traffic. Active-standby wastes 50% of capacity. Active-active requires the load balancers to coordinate state (but for stateless L4 forwarding, coordination is unnecessary).

**Sticky load balancer IPs**: Load balancers get static IPs so upstream firewalls and DNS records do not need to change on every auto-scaling event. Elastic IPs (AWS) or floating IPs (GCP) fulfill this role.

**Separate control plane and data plane**: The data plane forwards packets in kernel space (DPDK, XDP) for maximum throughput. The control plane (config changes, health checks, metrics) runs in user space. This separation allows 10 million+ pps forwarding without context switching.

**Connection limits per backend**: Set `maxconn` per backend to 200-500 to prevent one backend from being overwhelmed during a cascade. Queue excess requests with a timeout (e.g., 1 second) instead of immediately rejecting.

**Global rate limiting**: A shared Redis cluster tracks per-client request counts. The load balancer checks the counter before forwarding:

```
Client IP → hash → Redis key "rl:1.2.3.4" → TTL 1 second
Count > 1000? → Return 429 Too Many Requests
```

### Cross-Region Replication and Latency

| Region pair | One-way latency | Replication lag (typical) |
|---|---|---|
| US-East → US-West | 65 ms | 100-500 ms |
| US-East → EU-West | 80 ms | 200-800 ms |
| US-East → AP-Southeast | 180 ms | 500-2000 ms |
| EU-West → AP-Southeast | 160 ms | 500-1500 ms |

**Read traffic**: Each regions reads from its local replica. Latency is 5-30 ms (within region).

**Write traffic**: Writes go to the primary region (US-East) via internal routing. The app server in Singapore forwards writes to US-East with 180 ms added latency. The user-p95 write latency is 250-300 ms (180 ms cross-region + 70 ms processing).

**Failover**: If US-East fails, DNS health checks detect it (30-second detection window). DNS updates to route traffic to EU-West. EU-West DB becomes the new write primary. TTL propagation takes 60-300 seconds depending on client DNS caching.

### Capacity Planning

**Assumptions for 10 million daily active users**:
- Peak concurrent users: 20% of DAU = 2,000,000
- Requests per user per minute: 2 (checking feed, posting, etc.)
- Peak requests per second: 2,000,000 × 2 / 60 = ~66,667 rps
- Average response size: 25 KB (API JSON)
- Peak bandwidth: 66,667 rps × 25 KB = 1.67 GB/s = ~13.3 Gbps

**L4 load balancer capacity** (per instance):
- 1 instance of AWS NLB can handle ~10 million concurrent connections
- Throughput: up to 100 Gbps per NLB
- Cost: ~$0.008 per LCU (Load Balancer Capacity Unit) hour
- At 66,667 rps + 13.3 Gbps: approximately 20 LCUs
- Monthly cost: 20 × 0.008 × 730 = $116.80

**L7 load balancer capacity** (self-managed Nginx on c5.4xlarge):
- 16 vCPU, 32 GB RAM
- Max throughput with TLS: ~40,000 rps
- Needed for 66,667 rps: 2 instances (active-active)
- Buffer for failover (+100%): 4 instances
- Monthly cost: 4 × ~$0.68/hr × 730 = $1,985.60

**Database read replicas** (behind L4 LB):
- Each Postgres replica handles ~5,000 read qps
- Read ratio: 80% reads, 20% writes
- Reads per second: 66,667 × 0.8 = 53,333 reads/s
- Replicas needed: 53,333 / 5,000 = ~11 replicas

## Deployment and Operations

### Zero-Downtime Configuration Reload

Modifying load balancer configuration (adding a backend, changing a routing rule) traditionally requires a restart, which drops active connections. Modern load balancers support seamless reload.

<AsciiDiagram
  id="load-balancers/reload"
  title="Seamless Config Reload"
  content={`
Time ──────────────────────────────────────────────────────────────▶

Old LB process:
  PID 1234
  ┌──────────────────────────────────────────────────┐
  │ Serves connections on VIP:80                     │
  │                                                  │
  │ conn_1  conn_2  conn_3  ... conn_8472            │
  └──────────────────────────────────────────────────┘

  1. Admin sends SIGHUP (or runs reload command)
  2. Old process forks new process PID 5678
  3. New process inherits listen socket via SO_REUSEPORT
  4. Old process sets graceful shutdown: rejects new conns, drains existing
  5. New process starts accepting connections on VIP:80

New LB process:
  PID 5678
  ┌──────────────────────────────────────────────────┐
  │ Accepts new connections                          │
  │                                                  │
  │ new_conn_1  new_conn_2 ...                       │
  └──────────────────────────────────────────────────┘

  After 60 seconds (timeout client):
  Old process exits with zero active connections
  No connections dropped during the transition.
`}
  caption="Seamless reload uses fork-and-drain: the old process keeps existing connections alive while the new process picks up new ones. No connections are dropped."

### Monitoring Metrics

| Metric | What it reveals | Alert threshold |
|---|---|---|
| `backend_up` (0/1) | Backend health | 0 for 30 seconds |
| `active_connections_percent` | Capacity utilization | >80% for 5 minutes |
| `queue_depth` | Request queue behind backend | >100 for 10 seconds |
| `5xx_rate` | Backend errors (server error) | >1% for 2 minutes |
| `response_time_p99` | User-facing latency | >500 ms for 5 minutes |
| `tls_handshake_time` | TLS negotiation latency | >100 ms (p99) |
| `connection_drops` | Unexpected disconnects | >0 during steady state |
| `health_check_failures` | Backend reliability | >10% failure rate |

### Debugging Common Issues

**Problem: intermittent 503 errors during deploy**

Root cause: The load balancer's connection draining timeout is shorter than the application's graceful shutdown timeout. Application gets SIGTERM, starts refusing connections, but the load balancer still has them in the pool.

Fix: Set load balancer drain timeout to application graceful shutdown timeout + 10 seconds buffer.

**Problem: latency spikes at the top of every hour**

Root cause: Backend log rotation triggers a synchronous flush on all replicas simultaneously. Databases stall for 2-3 seconds.

Fix: Stagger cron jobs across backends. Use the load balancer's slow-start feature to gradually ramp traffic to newly rotated instances.

**Problem: one backend receives 5% more traffic than others with least-connections**

Root cause: New connections are distributed unevenly because the load balancer uses a concurrent hash map without proper locking. The race condition causes some backends to appear to have fewer connections than they actually have.

Fix: Use atomic counters for connection tracking, not shared mutable maps.

### Upgrading Load Balancer Software

| Strategy | Downtime | Complexity | Risk |
|---|---|---|---|
| Blue-green (new LB fleet) | 0 | High | DNS propagation delay |
| Rolling replace behind NLB | 0 | Medium | NLB health check timing |
| Canary (10% → 100%) | 0 | Medium | Partial traffic on new version |
| In-place upgrade | Seconds | Low | Brief cold start |

**Recommended**: Blue-green deployment with an external NLB/CLB in front. Spin up a new load balancer fleet, attach to the NLB target group, verify, then detach the old fleet. Rollback is flipping the target group back.
