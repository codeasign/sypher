# Replication at Scale

Replication in a textbook is simple: set up two databases, configure replication, route reads to replicas. Replication in production serves 10 million daily active users across three continents with 99.995% uptime. This page bridges the gap between the textbook setup and the production-grade architecture that runs at scale.

## Naive Implementation

The simplest possible replication setup is one primary with one replica, configured manually.

```
Application code:
  - Write everything to host:5432 (primary)
  - Read everything from host:5433 (replica)
  - If replica fails, read from primary
  - If primary fails... manually SSH in and promote replica
```

**What breaks at each scale:**

| Scale | What Breaks | Why |
|-------|-------------|-----|
| 1,000 users | Works fine | Single replica handles the read traffic |
| 100,000 users | Replica CPU at 90% | One replica cannot keep up with 100K users' read queries |
| 1 million users | Primary WAL sender saturated | Primary cannot stream WAL to enough replicas |
| 10 million users | Everything fails | No automated failover, no load balancing, no monitoring |

### At 1,000 Users

The setup works. The replica has 10% CPU usage, reads complete in under 10 ms, and the application handles occasional replica outages by falling back to the primary. Manual failover takes 10 minutes but happens only during planned maintenance.

### At 100,000 Users

The replica's CPU is at 90% during peak hours. Queries that took 10 ms now take 200 ms. The application code that hardcodes the replica address cannot add a second replica without a deploy. When the replica fails, the primary absorbs all traffic and its CPU jumps from 20% to 80%. You need more replicas and automated traffic routing.

### At 10 Million Users

The primary handles 50,000 writes per second. Each write must be streamed to six replicas. The primary's network outbound is saturated at 9 Gbps. Replicas three and four are in a different AWS Availability Zone with 2 ms of additional latency, and they lag 8 seconds behind during peak. The manual failover process takes 15 minutes — long enough for users to notice and leave.

The naive setup fails at scale because it lacks automation, monitoring, load distribution, and operational tooling.

### Key Components

**Connection Poolers (PgBouncer, ProxySQL):** Manage thousands of persistent database connections from the application pool. Route writes to the primary connection string and reads to the replica connection string. Provide automatic retry on connection failure.

**Orchestrator (Patroni, Orchestrator for MySQL, Kubernetes Operator):** Monitors cluster health via etcd or ZooKeeper quorum. Detects primary failure and promotes the replica with the highest LSN. Configures the application connection poolers to point to the new primary.

**Read-Only Load Balancer (HAProxy, NLB, or internal DNS):** Distributes read queries across replicas based on health and load. Removes unhealthy replicas from the pool. Supports weighted routing for heterogeneous replica hardware.

**Backup Pipeline (pgBackRest, WAL-G, or cloud snapshot):** Streams WAL to long-term storage. Enables point-in-time recovery to any second. Independent of the replication topology — backups run from a dedicated replica to avoid load on the primary.

## Integration Patterns

### Pattern 1: Relational Database with Read Replicas

The most common pattern: PostgreSQL or MySQL primary with read-only replicas.

```
Application routes:
  - Writes go to the primary connection pool
  - Time-critical reads go to the primary (or sync replica)
  - Bulk reads, reporting, analytics go to replicas
  - Stale-tolerant queries go to any replica
```

### Pattern 2: Cache Invalidation with Replication

Caches sit in front of databases to absorb read traffic. Replication introduces the risk that a cache miss reads from a lagging replica and caches stale data.

```
Solution: The cache invalidation message is sent through a queue that
the primary processes. The primary writes the new value AND publishes
an invalidation event. The cache evicts the stale entry. Subsequent
reads fetch from a replica that has the latest data.
```

### Pattern 3: Message Queue Replication

Kafka and RabbitMQ replicate data across brokers using their own replication protocols. The pattern differs from database replication because queues are ephemeral — they tolerate some data loss.

```
Kafka replication:
  - Each partition has one leader and N followers (ISR = in-sync replicas)
  - acks=all: producer waits for all ISRs to confirm
  - acks=1: producer waits only for the leader
  - acks=0: fire-and-forget
```

### Pattern 4: External API Replication (Change Data Capture)

Capture database changes and replicate them to an external system (search index, cache, analytics pipeline) using CDC.

```
CDC pipeline:
  1. Debezium reads the WAL in real-time
  2. Transforms row changes into Avro or JSON events
  3. Publishes to Kafka topic
  4. Downstream consumers update Elasticsearch, Redis, or data warehouse
```

### Multi-Region Design Decisions

**Option A: One global primary, replicas in every region.** All writes go to the primary region. Replicas serve reads locally. Simple to reason about. Write latency from distant regions is 100-300 ms for each write. Used by companies with a primary user base in one region.

**Option B: Multi-leader, one leader per region.** Each region has a leader that accepts local writes. Leaders replicate to each other asynchronously. Writes are fast (under 10 ms local). Conflict resolution is required. Used by companies with global user bases like Google, Dropbox, and Notion.

**Option C: Application-level sharding by region.** Users are assigned to a regional shard. Each shard is a single-leader cluster in that region. Cross-region queries are handled at the application layer. Used by large SaaS platforms that segment users by geography.

## Capacity Planning

You are launching a social app. Your projections: 5 million users in year one, 50% growth rate. Each user makes 10 reads and 1 write per day. Peak traffic is 5x average. Plan the replication infrastructure.

### Step 1: Compute throughput

```
Daily reads:     5,000,000 users * 10 reads = 50,000,000 reads/day
Daily writes:    5,000,000 users * 1 write  = 5,000,000 writes/day

Peak reads per second (5x average):
  Average reads/second: 50,000,000 / 86,400 = 579 reads/second
  Peak reads/second: 579 * 5 = 2,895 reads/second

Peak writes per second:
  Average writes/second: 5,000,000 / 86,400 = 58 writes/second
  Peak writes/second: 58 * 5 = 290 writes/second
```

### Step 2: Determine primary capacity

A single well-tuned PostgreSQL primary with 16 vCPUs and 64 GB RAM can handle approximately 10,000-20,000 write transactions per second for simple queries. Your peak of 290 writes/second is well within budget. The primary's CPU will spend most of its time on reads that cannot be offloaded: 200 reads/second for session queries that require primary consistency.

### Step 3: Determine replica count

```
Total peak reads: 2,895 reads/second
Primary handles: 200 reads/second (session-consistent reads)
Replicas needed: 2,695 reads/second

One replica handles: ~3,000 reads/second (conservative)
Replicas required: 2,695 / 3,000 = 0.9 = at least 1 replica

Add N+2 redundancy: 3 replicas minimum
  (1 for capacity + 1 for failover buffer + 1 for maintenance)
```

### Step 4: Network and storage

```
WAL bandwidth per write (1 KB per entry):
  290 writes/second * 1 KB = 290 KB/s per replica
  3 replicas * 290 KB/s = 870 KB/s outbound from primary

Replication lag at peak (assuming 10 MB/s replica apply speed):
  WAL generation: 870 KB/s * 8 seconds buffer = 7 MB queue
  Lag: under 1 second

Storage per replica (data + WAL):
  Data: 200 GB (projected)
  WAL retention: 16 GB (16 GB * 1024 segments)
  Total per replica: 216 GB
```

### Recommended Infrastructure (Year 1)

- 1 primary (16 vCPU, 64 GB RAM, 500 GB provisioned IOPS SSD)
- 3 replicas (8 vCPU, 32 GB RAM, 500 GB SSD — replicas need less CPU)
- 1 synchronous replica in same AZ for zero data loss
- 2 asynchronous replicas in different AZs for availability
- 10 Gbps networking on all nodes

## Deployment and Operations

### Zero-Downtime Deploy (Primary Maintenance)

```
1. Set the primary to read-only mode
2. Wait for replicas to catch up (lag = 0)
3. Promote the synchronous replica to new primary
4. Update connection pooler configuration
5. Old primary becomes a replica of new primary
6. Old primary catches up and rejoins the pool
```

### Monitoring

```
Critical metrics (alert on every violation):
  - Replication lag > 30 seconds (warning)
  - Replication lag > 120 seconds (critical)
  - WAL disk usage > 80%
  - Replica disconnected > 5 minutes
  - Primary CPU > 80%

Important metrics (review daily):
  - Replica apply rate (entries/second)
  - Network bandwidth utilization
  - Number of replication slots in use
  - WAL segment generation rate
  - Query latency p50/p95/p99 on replicas
```

### Common Operational Issues

**WAL disk full:** A replica disconnects and the primary retains all WAL segments until the replica reconnects or the slot is dropped. Set `wal_keep_size` to a reasonable limit and alert on WAL disk usage. Create a dashboard that shows WAL generation rate versus retention.

**Replica lag spikes:** Index-heavy operations (ALTER TABLE, CREATE INDEX) on the primary generate large WAL entries that replicas must replay. Run DDL during maintenance windows. Set `max_standby_streaming_delay` to prevent long-running queries on replicas from blocking WAL replay.

**Connection storms:** After a failover, all application instances reconnect simultaneously. Use connection poolers with slow-start. Set `max_connections` on replicas 50% higher than expected to absorb the storm.

**Promoting the wrong replica:** In a crash, two replicas may have nearly identical LSNs. Use quorum-based failover with Patroni, which picks the replica with the highest LSN in the etcd consensus group. Never promote a replica manually unless you have verified its LSN is the highest.

Production replication architecture is about designing for failure from the start. Every component must be redundant, every failure mode must have an automated response, and every metric must be visible.
