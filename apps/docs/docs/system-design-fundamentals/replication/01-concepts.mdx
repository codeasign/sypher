# Understanding Replication

You are the lead engineer for a social media platform with 200,000 daily active users. Your database is a single PostgreSQL instance running on a machine with 64 GB of RAM and 16 vCPUs. Every day, users post status updates, like photos, and send messages. Your database handles it all — until one afternoon at 2:47 PM, when an index finger slips and a celebrity mentions your platform. Traffic spikes from 2,000 requests per second to 18,000 in under a minute. The single database CPU hits 100%, query queue depth grows to 5,000, and within 30 seconds the database is unresponsive. Requests start timing out. Users see white screens. The backup kicks in at 2:49 PM, but by then 14,000 users have experienced errors. Your CEO wants to know why one database cannot handle a few extra users.

The fix seems obvious: get a bigger database. But vertical scaling has limits. The largest cloud instances top out at 24 TB of RAM and 448 vCPUs, and they cost over $400 per hour. Even if you can afford it, a single server is still a single point of failure. When that server goes down — for maintenance, for a power outage, for a kernel panic — your entire application goes down with it. You need a way to distribute the load across multiple servers, and you need a way to survive the loss of any single server without losing data or availability.

You could set up a second database as a hot standby, copying every change from the primary. When the primary fails, the standby takes over within seconds. But now you have a new problem: during normal operation, the standby sits idle, consuming resources but handling no traffic. What if you could route read queries to the standby as well? That doubles your read capacity without adding any extra hardware cost — the second server is already there. This is replication. It is the foundation of every distributed database, every highly available system, and every global-scale application in production today.

## The Insight

Replication is the practice of maintaining identical copies of data across multiple servers. Think of it like a library with branches: the main library holds the master copy of every book, but each neighborhood branch keeps its own copy of the most popular titles. When someone checks out a book from a branch, the main library is notified and records the change. If the main library burns down, each branch still has its collection, and you promote one branch to be the new main library. The data survives. Readers can visit any branch, distributing the traffic instead of funneling everyone downtown.

In database terms, the main library is the **primary** node, the branches are **replica** nodes, and the checkout process is the **replication log** — a sequential record of every write operation applied to the primary.

## Core Vocabulary

<dl>
<dt><strong>Primary (Leader / Master)</strong></dt>
<dd>The authoritative database node that accepts writes. All data changes originate here and flow outward to replicas. <em>Why it matters:</em> If you lose the primary without a replica, you lose the ability to write new data. Every replication architecture designates exactly one primary.</dd>

<dt><strong>Replica (Standby / Follower / Slave)</strong></dt>
<dd>A copy of the primary that serves read traffic. Replicas apply changes from the primary asynchronously or synchronously. <em>Why it matters:</em> Replicas absorb read load and provide failover targets. More replicas means more read capacity but also more replication lag to manage.</dd>

<dt><strong>Replication Log (Write-Ahead Log / Binary Log / Change Data Capture Stream)</strong></dt>
<dd>An append-only sequence of every write operation, committed in order. Replicas read this log to replay changes locally. <em>Why it matters:</em> The replication log determines how fast replicas can keep up, how much bandwidth replication consumes, and how much data you can lose in a crash.</dd>

<dt><strong>Synchronous Replication</strong></dt>
<dd>The primary waits for at least one replica to confirm a write before acknowledging it to the client. <em>Why it matters:</em> Guarantees zero data loss on failover but increases write latency by at least one network round-trip to the replica.</dd>

<dt><strong>Asynchronous Replication</strong></dt>
<dd>The primary acknowledges the write to the client immediately and sends the change to replicas in the background. <em>Why it matters:</em> Lower write latency and higher throughput, but a primary crash before the replica receives the change means permanent data loss.</dd>

<dt><strong>Replication Lag</strong></dt>
<dd>The delay between a write being committed on the primary and appearing on a replica. Measured in seconds or bytes behind. <em>Why it matters:</em> High lag means stale reads — a user writes data and then reads from a replica that has not yet received the change, seeing old information.</dd>

<dt><strong>Failover</strong></dt>
<dd>The process of promoting a replica to become the new primary when the current primary fails. <em>Why it matters:</em> Failover speed determines your recovery time objective (RTO). Automated failover takes seconds; manual failover takes minutes.</dd>

<dt><strong>Split-Brain</strong></dt>
<dd>A dangerous state where two nodes both believe they are the primary, accepting writes independently. <em>Why it matters:</em> Split-brain causes data divergence that is extremely painful to reconcile — two sets of writes that were never merged. Prevention mechanisms (quorum, fencing) are critical.</dd>

<dt><strong>Quorum</strong></dt>
<dd>The minimum number of nodes that must agree on a decision, typically a majority (N/2 + 1). <em>Why it matters:</em> Quorum-based replication prevents split-brain and ensures consistency. A cluster of 5 nodes needs 3 to agree. Lose 2 nodes, and the cluster still operates. Lose 3, and it stops — insufficient quorum.</dd>

<dt><strong>Strong Consistency / Eventual Consistency</strong></dt>
<dd>Strong consistency means every read returns the most recent write. Eventual consistency means reads may return stale data but will converge over time. <em>Why it matters:</em> This is the central trade-off in replication. Strong consistency is safer but slower. Eventual consistency is faster but requires applications to tolerate stale reads.</dd>
</dl>

## How It Actually Works

Replication sounds simple — copy data from A to B. The implementation details are where the complexity lives. Here is the step-by-step lifecycle of a write in a single-leader replication system.

### Step 1: Client Connects to Primary

The application directs all write operations to the primary node. This routing happens either through a connection pooler (PgBouncer, ProxySQL) that knows which node is primary, or through application-level logic that reads a configuration file listing primary and replica addresses.

### Step 2: Primary Commits the Write and Appends to the Log

The primary processes the write, applies it to its local storage engine, and appends a record to the transaction log (write-ahead log in PostgreSQL, binary log in MySQL). This log record contains enough information for a replica to replay the exact same write — either the full row data (statement-based) or the modified bytes (row-based).

### Step 3: Replica Connects and Streams the Log

The replica maintains a persistent TCP connection to the primary (or to a replication slot in PostgreSQL). It sends its current log position — the last change it has applied — and the primary streams all new log entries since that position.

### Step 4: Replica Decodes and Applies Each Entry

The replica receives the raw log data, decodes it according to the replication protocol format, and applies each change to its local storage engine. In PostgreSQL, this is done by the WAL receiver process, which writes to the replica's own WAL and then the WAL application process replays it. In MySQL, the IO thread writes to a relay log, and the SQL thread replays from there.

### Step 5: Replica Signals Its Progress

The replica sends an acknowledgment back to the primary with its updated log position. This acknowledgment is what synchronous replication waits for. In asynchronous mode, the primary does not wait — it continues processing writes and the replica catches up when it can.

### Step 6: Read Traffic Routes to the Replica

The application or load balancer routes incoming read queries to any available replica. The replica returns results from its local copy of the data. If replication lag is present, these results may not include the very latest writes — they reflect the state of the data as of the last log position the replica has applied.

### Step 7: Monitoring and Heartbeats

Each replica sends periodic heartbeat signals to the primary (or to a cluster coordinator like etcd, ZooKeeper, or Patroni). If a replica misses three consecutive heartbeats, the system marks it as down and removes it from the read pool. If the primary misses heartbeats, the system initiates failover — promoting a healthy replica to primary.

## Variants and Flavors

### Single-Leader Replication (Statement-Based, Row-Based, WAL-Based)

This is the most common replication architecture, used by PostgreSQL streaming replication, MySQL replication, and SQL Server Always On.

        ▲          ▲          ▲

        └────── Reads ────────┘

**Use when:** Your workload is read-heavy (80% reads, 20% writes or more). You need high availability with automated failover. You want simplicity — single-leader is the easiest replication model to reason about and operate.

**Do NOT use when:** Your workload is write-heavy with every replica needing to accept writes. Single-leader forces all writes through one node, capping write throughput at what one server can handle. You also should avoid it if your application cannot tolerate even a few seconds of stale reads during normal operation — replication lag is inevitable.

### Multi-Leader Replication

Multiple nodes accept writes, and they replicate changes to each other. Used by MySQL Group Replication, PostgreSQL BDR, and CouchDB.

**Use when:** You have users distributed across multiple geographic regions and you want each region to write to a local leader for latency. Your application can tolerate or resolve conflicts (concurrent writes to the same key on different leaders). You need zero-downtime maintenance across data centers.

**Do NOT use when:** Your application requires strong consistency — multi-leader always allows conflicting writes. If you cannot tolerate even temporary data divergence, single-leader is safer. Also avoid it when your conflict resolution strategy is to ignore conflicts; unhandled conflicts corrupt data silently.

### Leaderless Replication (Dynamo-Style)

Every node accepts reads and writes. No designated primary. Used by Amazon DynamoDB, Cassandra, Riak, and Voldemort.

**Do NOT use when:** You need SQL joins, foreign keys, or transactions across partitions. Leaderless systems are key-value stores at heart and do not support relational features. Also avoid it when your team does not have deep operational experience with distributed systems — leaderless replication is significantly harder to operate and debug.

## Mental Model Check

**Q1: Your application has 99% reads and 1% writes. Which replication architecture is best and why?**

Single-leader replication. The write bottleneck on the primary does not matter at 1% write ratio, and replicas handle the read traffic. This is the pattern used by most content management systems, blog platforms, and reporting dashboards.

**Q2: You deploy replicas in three different AWS regions. A user in Sydney writes data, then a user in London reads the Sydney data 500 milliseconds later and gets stale results. What happened?**

The Sydney write was committed to the primary in Sydney. Replication to the London replica takes approximately 200-400 milliseconds of network latency (Sydney to London is about 160 ms round-trip) plus any processing lag on the replica. At 500 ms after the write, the replica may still be catching up. This is expected behavior with asynchronous replication across geographic distances.

**Q3: What happens if a synchronous replica becomes slow or unavailable?**

The primary cannot acknowledge writes until the synchronous replica confirms them. If the replica is down, every write to the primary hangs. Production systems handle this with a timeout: if the synchronous replica does not respond within a configurable window (typically 1-5 seconds), the primary degrades it to asynchronous mode and continues. This prevents a slow replica from taking down write availability.

**Q4: Your database cluster has five nodes with a replication factor of three. The quorum is set to W=2 and R=2. How many nodes can fail before you cannot write? Before you cannot read?**

With W=2, you need at least two nodes to acknowledge a write. If three nodes fail, only two remain — you can still write. If four nodes fail, only one remains and writes fail. With R=2, same logic: you can still read as long as at least two nodes are available. In practice, if more than two nodes are down the cluster is degraded and operators should intervene.

**Q5: Your application shows a user their own comment appearing, disappearing, and reappearing across page refreshes. What is causing this?**

The user writes a comment (goes to the primary). The next page load hits a replica that has already received the write (comment visible). The next load hits a different replica that is lagging (comment invisible). This is known as "read-your-write inconsistency." Fixes include session stickiness (route users to the same replica), read-after-write consistency (read from primary for N seconds after writing), or switching to synchronous replication.

Replication is the bedrock of distributed data systems. Everything you build on top of it — sharding, caching, global distribution — assumes that replication works correctly.
