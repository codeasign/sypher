# Error Budgets — Building It at Scale

## Naive Implementation: The Single-Service Budget

A naive error budget implementation is simple: collect request counts and error counts for one service, calculate the budget ratio, and alert when the budget is exhausted. This works for a single service with under 1,000 requests per second.

### Architecture at 1,000 Users

At this scale, you can implement error budgets with a simple script that runs every hour:

```python
import time
from datetime import datetime, timedelta

def calculate_error_budget(window_days=30, slo_target=0.999):
    window_start = datetime.now() - timedelta(days=window_days)
    # Query your log system for total and error counts
    total_requests = query_logs("count", window_start)
    error_requests = query_logs("count", window_start, status="5xx")
    
    actual_availability = 1 - (error_requests / total_requests)
    error_budget_remaining = actual_availability / slo_target
    
    return {
        "total_requests": total_requests,
        "error_requests": error_requests,
        "actual_availability": actual_availability,
        "budget_remaining": error_budget_remaining
    }
```

**Breaks at 1,000 requests/second** because:
- The single script cannot keep up with log parsing at higher throughput
- There is no historical storage — budget calculations are lost on server restart
- There is no multi-service support — you need manual aggregation
- No burn rate calculation — you only know budget status once per hour

At this scale, the budget calculation is a Prometheus recording rule:

```prometheus
# Recording rule for 30-day error budget
- record: service:error_budget_ratio_30d
  expr: |
    sum(rate(http_requests_total{status=~"5.."}[30d]))
    /
    sum(rate(http_requests_total[30d]))

- record: service:error_budget_remaining
  expr: |
    1 - (
      sum(rate(http_requests_total{status=~"5.."}[30d]))
      /
      sum(rate(http_requests_total[30d]))
    ) / 0.999  # SLO target
```

**Breaks at 10 million users** because:
- Prometheus single instance cannot handle the cardinality of 30+ services with per-endpoint metrics
- No multi-region support — all metrics come from one data center
- No dependency tracking — you cannot attribute failures across service boundaries
- Burn rate alerts use a single window, missing slow or fast burns

### Architecture at 10 Million Users

At this scale, you need a production-grade error budget system that spans multiple services, multiple teams, and multiple regions.

The production-grade architecture consists of three layers:

**Metric Collection Layer:** Each service runs a metrics agent (Prometheus node exporter or a custom agent) that exposes request counts and error counts as Prometheus metrics. Regional Prometheus instances scrape these agents at 15-second intervals. Multi-region deployments have one Prometheus per region, with Thanos or Cortex providing a global view.

**Calculation Layer:** A centralized budget calculator (implemented as a service, not inline Prometheus rules) handles:
- Per-service budget calculation across all regions
- Burn rate calculation across multiple windows (1-hour, 6-hour, 24-hour)
- Composite budget aggregation for user-facing service groups
- Per-tenant budget tracking for multi-tenant systems
- Historical budget trend analysis for capacity planning

**Action Layer:** The action layer enforces policy based on budget state:
- Continuous deployment pipeline checks the budget before approving a deployment
- PagerDuty integration pages on-call engineers at predefined burn rate thresholds
- Slack notifications provide daily budget status summaries
- Rate limiters automatically activate when budget consumption exceeds emergency thresholds

## Integration Patterns

### Pattern 1: Database-Backed Budget Store

For systems that need accurate, queryable budget history, use a relational database as the budget store.

```
Table: error_budget_state
Columns:
  service_id     VARCHAR(64)   PRIMARY KEY
  slo_target     DECIMAL(5,4)  NOT NULL  -- e.g., 0.999
  window_start   TIMESTAMP     NOT NULL
  window_end     TIMESTAMP     NOT NULL
  total_requests BIGINT        NOT NULL
  error_requests BIGINT        NOT NULL
  budget_ratio   DECIMAL(10,6) NOT NULL
  updated_at     TIMESTAMP     NOT NULL

Table: error_budget_burn_rate
Columns:
  service_id     VARCHAR(64)   PRIMARY KEY
  window_type    VARCHAR(16)   PRIMARY KEY  -- '1h', '6h', '24h'
  burn_rate      DECIMAL(10,4) NOT NULL
  time_to_exhaust INTERVAL     NOT NULL
  updated_at     TIMESTAMP     NOT NULL
```

This pattern allows historical queries like "what was the budget state at the time of the last deployment?" which is essential for post-incident analysis.

### Pattern 2: Cache-Enhanced Budget Calculation

Budget calculations over a 30-day window require scanning millions of data points. A cache reduces the computational load.

### Pattern 3: Queue-Based Burn Rate Alerting

Multi-window burn rate calculation can produce many simultaneous alerts when a major incident occurs. A queue decouples the alert calculation from the notification delivery.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Budget      │ ──▶  │  Alert Queue  │ ──▶  │  Alert       │
│  Calculator  │     │  (Kafka)     │     │  Processor   │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                        ┌─────────────────────────┼─────────────┐
                        │                         │             │
                        ▼                         ▼             ▼
                  ┌──────────────┐          ┌──────────────┐  ┌────┐
                  │  PagerDuty   │          │  Slack       │  │Jira│
                  │  Critical    │          │  Warning     │  │    │
                  └──────────────┘          └──────────────┘  └────┘
```

The queue allows the alert processor to deduplicate, throttle, and prioritize alerts. During a major outage, multiple burn rate windows may fire simultaneously. The alert processor groups them into a single notification.

### Pattern 4: Deployment Gate Integration

Error budgets integrate with the CD pipeline through a webhook API. Before approving a deployment, the CD tool calls the error budget API.

```yaml
# GitHub Actions deployment gate
- name: Check Error Budget
  id: error-budget-check
  run: |
    BUDGET_STATUS=$(curl -s https://budget-api.internal/status/payment-service)
    REMAINING=$(echo $BUDGET_STATUS | jq -r '.budget_remaining')
    if (( $(echo "$REMAINING < 0.25" | bc -l) )); then
      echo "Error budget below 25%. Deployment blocked."
      exit 1
    fi
    echo "Budget status: $REMAINING remaining. Proceeding."
```

The gate checks both total budget remaining and burn rate. Even if the budget is healthy, a high burn rate blocks deployment because it signals an ongoing incident.

Key considerations for multi-region error budgets:

**Regional SLO weighting.** Not all regions are equally important. You can weight regions by traffic volume or business value. A region that handles 50% of traffic should have more influence on the global budget than a region handling 10%.

**Regional autonomy.** Each region should have the ability to freeze deployments independently. A deployment freeze in the AP-SOUTHEAST region (due to budget exhaustion) should not prevent deployments in US-EAST if that region is healthy.

**Cross-region budget transfer.** In an active-active setup, traffic can shift between regions. If US-EAST fails and traffic reroutes to EU-WEST, the EU-WEST region should not be penalized by the extra traffic. Implement traffic-weighted budget adjustment.

## Capacity Planning

Error budget capacity planning answers the question: "How much error budget do we need to support the next 6 months of growth?"

### The Math

Assume 10 million daily active users (DAU) making an average of 20 requests per day. Total daily requests:

```
10,000,000 DAU * 20 requests = 200,000,000 requests per day
```

For a 99.9% SLO over 30 days:

```
Total requests in 30 days = 200,000,000 * 30 = 6,000,000,000
Allowed errors = 6,000,000,000 * 0.001 = 6,000,000 errors
Error budget in requests = 6,000,000 errors per month
Error budget in seconds = 0.001 * 86,400 * 30 = 2,592 seconds
```

Now project growth at 20% month-over-month:

```
Month 1: 200M req/day, 6M error budget
Month 2: 240M req/day, 7.2M error budget
Month 3: 288M req/day, 8.64M error budget
Month 6: 497M req/day, 14.9M error budget
Month 12: 1.03B req/day, 30.9M error budget
```

The error budget in absolute terms grows with traffic, but the SLO stays the same. The challenge is that the infrastructure must also grow to maintain the same error rate. If your error rate increases due to capacity constraints, your budget will be consumed faster.

**Capacity planning recommendations:**
- Monitor the ratio of budget consumption to traffic growth. If both grow at the same rate, your reliability is proportional to your scale — which means you need to grow your infrastructure.
- Plan for 2x headroom. Your infrastructure should be able to handle 2x the current traffic without increasing the error rate. This gives you room to deploy, test, and recover.
- Use the budget consumption trend to predict when you will exhaust the budget before the month ends. If the trend shows 80% budget consumption at week 2 of a 4-week window, you will exhaust the budget by week 3.

### Budget Sizing by Traffic Tier

| Traffic Tier | Daily Requests | SLO | Error Budget (seconds/month) | Infrastructure Cost |
|-------------|---------------|-----|------------------------------|-------------------|
| Small | 1 million | 99.9% | 2,592 | $1,000/month |
| Medium | 100 million | 99.95% | 1,296 | $10,000/month |
| Large | 1 billion | 99.99% | 259 | $50,000/month |
| Enterprise | 10 billion | 99.999% | 25 | $500,000/month |

Each tier costs roughly 5-10x more in infrastructure to achieve the tighter SLO. The error budget shrinks by roughly 10x per tier.

## Deployment and Operations

### Deployment Strategy

The error budget system itself must be deployed with care, because if the budget system goes down, you lose visibility into reliability.

**Blue-green deployment:** Deploy a new version of the budget calculator alongside the existing version. Route 10% of traffic to the new version for 24 hours. If the budget calculations match within 1%, switch all traffic. This ensures the new version does not introduce calculation errors.

**Canary deployment:** Deploy the budget calculator changes to one region first. Monitor for 48 hours. If the budget data in that region matches expectations, roll out globally.

**Feature flags:** Every new budget feature (per-tenant budgets, new burn rate windows) should be behind a feature flag. This allows instant rollback if the change introduces incorrect calculations.

### Data Pipeline Architecture

The error budget data pipeline must handle the volume, velocity, and variety of metric data from hundreds of service instances.

### Pipeline Throughput

```
Data volume per instance:
  - 1 request counter (incrementing at ~100/s)
  - 1 error counter (incrementing at ~1/s)
  - 1 latency histogram (recording ~100 observations/s)
  Total: ~200 data points per second per instance

For 500 instances, each with 3 metric endpoints:
  Total: 500 * 200 * 3 = 300,000 data points per second
  Per minute: 18 million data points
  Per day: 25.9 billion data points
```

The pipeline must handle this throughput with minimal data loss. The collection layer uses batching, compression, and retry logic to maintain reliability.

### Batch Processing Pipeline

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Agent   │────▶│  Buffer  │────▶│  Batch   │────▶│  TSDB    │
│  (Per    │     │  (Local  │     │  Writer  │     │  (Remote) │
│  Instance│     │  Disk)   │     │  (Thrift)│     │  Storage) │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
                      │
                      │ (on network failure)
                      ▼
                 ┌──────────┐
                 │  Retry   │
                 │  Queue   │
                 │  (Disk)  │
                 └──────────┘
```

**Agent:** Runs on each service instance. Buffers metrics in a ring buffer in shared memory. Exposes a /metrics endpoint that the collector scrapes at configurable intervals (default 15 seconds).

**Buffer:** A local disk-bounded queue. When the TSDB is unreachable, the buffer stores up to 1 GB of metric data. The buffer is backed by a write-ahead log (WAL) for crash recovery.

**Batch Writer:** Aggregates metrics from multiple agents into batches of 1,000 data points each. Compresses the batch using Snappy compression (typically 5:1 compression ratio). Sends the compressed batch to the TSDB via HTTP/2.

**TSDB:** Prometheus or Thanos stores the metrics. A single Prometheus instance can handle approximately 10 million samples per second. For higher throughput, use Thanos with horizontal sharding.

### Data Quality Checks

The pipeline includes data quality checks at each stage:

```
Agent-level: Verify counter monotonicity
  - Counter value should never decrease
  - If counter resets (process restart): flag as "counter_reset"
  - Budget calculator handles resets by measuring the delta

Collector-level: Validate metric format
  - Metric names match expected pattern: ^[a-zA-Z_:][a-zA-Z0-9_:]*$
  - Label values are non-empty and under 200 characters
  - Timestamps are within 5 minutes of wall clock time
  - Reject metrics that fail validation with a counted warning

TSDB-level: Verify data continuity
  - No gaps longer than 3x the scrape interval
  - No duplicate timestamps for the same metric+labels combination
  - Data age is within 10% of wall clock (no clock skew)
  - Alert if any quality check fails for more than 5% of metrics
```

### Compression and Retention

```
Raw metric data: ~100 bytes per sample
After Snappy compression: ~20 bytes per sample
Daily storage: 20 bytes * 25.9 billion = 518 GB per day

Retention strategy:
  - 35 days at full resolution (15s intervals): 18 TB
  - 6 months at 5-minute resolution (downsampled): ~10 TB
  - 2 years at 1-hour resolution (downsampled): ~3 TB
  Total compressed storage: ~31 TB

Downsampling reduces storage by:
  - 15s → 5min: 20x reduction
  - 5min → 1h: 12x reduction
  - Combined: 240x reduction from raw
```

The cost of storing 35 days of full-resolution data is the primary budget driver. At $0.10/GB/month for SSD storage, the monthly cost is approximately $1,800 for 18 TB.

## Scaling the Budget Calculator

As the number of services grows, the budget calculator must scale horizontally. The ring buffer approach enables efficient partitioning.

### Partitioning by Service Group

```
Budget Calculator Cluster (3 nodes)

Node 1: Payment, Checkout, Cart (high criticality)
Node 2: Search, Recommendations, Content (medium criticality)
Node 3: Analytics, Reporting, Internal Tools (low criticality)

Each node maintains ring buffers for its assigned services.
Nodes communicate via a shared Redis cache for cross-service queries.
Failure of one node only affects its assigned services.
```

### Sharding by Hash

For larger deployments, shard services across nodes by consistent hashing:

```
hash(service_id) % num_nodes → node assignment

Payment → hash("payment") % 10 = 3 → Node 3
Search → hash("search") % 10 = 7 → Node 7
Cart → hash("cart") % 10 = 1 → Node 1
```

Consistent hashing minimizes rebalancing when nodes are added or removed. Only 1/N of services need to move when a node joins or leaves.

The budget calculator writes to a primary store and replicates to read replicas. Dashboard queries hit the replicas, while deployment gate checks hit the primary.

```
    │  Deployment  │                       │  Dashboard   │  │API │
    │  Gate        │                       │  (Grafana)   │  │    │
    └──────────────┘                       └──────────────┘  └────┘
```

The primary handles writes and deployment gate checks (consistency-critical). Read replicas handle dashboard queries (consistency-tolerant, up to 1-second staleness).

### Storage Cost for Budget Metrics

```
Per-service metric storage:
  - http_requests_total: 1 counter, 1,000 series (per endpoint)
  - http_errors_total: 1 counter, 1,000 series
  - request_duration_seconds: 1 histogram, 1,000 series
  - budget_remaining: 1 gauge, 1 series
  - burn_rate: 3 gauges (1h/6h/24h), 3 series

Total per service: ~2,005 time series
Storage at 15s scrape interval: ~5,760 samples per series per day
Total samples per service per day: ~11.5 million
Storage cost: ~500MB per service per day (compressed)

For 50 services: 25GB per day of storage
For 30-day retention: 750GB of compressed storage
Monthly cost: $500-$1,000 for SSD-backed storage
```

The primary cost driver is the number of time series, not the number of requests. Each unique combination of labels (service, endpoint, status code, region) creates a new time series. Reducing label cardinality by 50% cuts storage costs by 50%.

## Operational Runbook

**Symptom: Budget shows 0% remaining but service appears healthy.**
1. Check the SLI definition. Has it changed recently?
2. Check the Prometheus query. Is the error count query including non-error HTTP codes?
3. Check for metric data gaps. Was Prometheus down for any period during the window?
4. If the SLI is correct and the data is accurate, accept the budget exhaustion — your SLI may be capturing failures that the team has not noticed.

**Symptom: Budget shows 100% remaining but users are complaining.**
1. Your SLI does not match user experience. Review the SLI definition.
2. Check if the error budget window recently reset (e.g., you are on day 1 of a 30-day window).
3. Check if you are measuring the right error types. Are you measuring HTTP errors but users are experiencing timeout errors (which may not return a 5xx)?

**Symptom: Burn rate alert fires every day at the same time.**
1. This is likely a scheduled job or traffic pattern. Identify the recurring traffic source.
2. If it is a batch job that runs daily, exclude it from the burn rate calculation or use a different burn rate threshold for that period.
3. If it is a real pattern of daily degradation, investigate the root cause.

**Symptom: Budget keeps exhausting on the last 3 days of the month.**
1. Your budget window is too short. Consider extending to 30 days.
2. You may have a "month-end" traffic pattern that spikes errors. Analyze the traffic pattern.
3. Consider a rolling window instead of a calendar window. Rolling windows smooth out end-of-month effects.

**Symptom: Budget calculator reports inconsistent values across regions.**
1. Check for clock skew between regions. Prometheus relies on accurate timestamps.
2. Verify that the global aggregator is using the correct timezone for window calculations.
3. Check if regional Prometheus instances have different retention policies.
4. Ensure that the aggregation formula accounts for regional traffic weighting.

**Symptom: Deployment gate check times out.**
1. The budget calculator may be under load. Check CPU and memory usage.
2. The Redis cache may be unavailable. The calculator should fall back to direct TSDB query, which is slower but functional.
3. Check the timeout configuration on the deployment gate. A 5-second timeout is usually sufficient.
4. Consider implementing a circuit breaker on the deployment gate: after 3 consecutive timeouts, allow the deployment with a warning log.

## Reliability of the Error Budget System

The error budget system itself must be reliable. If it goes down, you lose the ability to make data-driven deployment decisions.

### High Availability Design

The budget calculator runs as a stateless service behind a load balancer. Multiple instances handle requests concurrently.

```
┌──────────────┐     ┌──────────────┐
│  Load        │     │  Load        │
│  Balancer 1  │     │  Balancer 2  │
│  (Active)    │     │  (Passive)   │
└──────┬───────┘     └──────────────┘
       │
       ┌──────────────┼──────────────┐
       │              │              │
       ▼              ▼              ▼
┌──────────────┐┌──────────────┐┌──────────────┐
│  Calculator  ││  Calculator  ││  Calculator  │
│  Instance 1  ││  Instance 2  ││  Instance 3  │
│  (us-east-1) ││  (us-east-1) ││  (us-west-2) │
└──────┬───────┘└──────┬───────┘└──────┬───────┘
       │              │              │
       └──────────────┼──────────────┘
                      │
                      ▼
               ┌──────────────┐
               │  Redis       │
               │  Cluster     │
               │  (Cross-     │
               │   Region)    │
               └──────────────┘
```

The calculator instances are deployed across two availability zones. If one zone fails, the load balancer routes traffic to the remaining zone. The Redis cluster is also cross-region, with automatic failover.

### Graceful Degradation

When the budget calculator is unavailable, the system degrades gracefully:

1. **Deployment gate:** Default to "allow" with a warning log. This is a conscious choice — blocking deployments due to a monitoring failure is worse than allowing deployments without monitoring.
2. **Dashboard:** Show "data unavailable" instead of a stale value. Do not show a green budget that is hours old.
3. **Alerting:** The alert manager has a "no data" alert that fires when the budget calculator has not reported for 5 minutes. This alerts the on-call team that the monitoring system is down.
4. **Budget state:** The last known budget state is cached in Redis with a 48-hour TTL. If the calculator is down for less than 48 hours, the cache provides the last known state.

### Testing the Error Budget System

The error budget system must be tested regularly to ensure it works correctly:

1. **Unit tests:** Test the budget calculation algorithm with known inputs and expected outputs. Test edge cases: empty window, all-errors window, all-success window.
2. **Integration tests:** Test the full pipeline: agent → collector → TSDB → calculator → alert. Use synthetic metric data.
3. **Chaos tests:** Randomly kill calculator instances and verify that the system degrades gracefully and recovers.
4. **Load tests:** Verify that the calculator can handle 10x normal traffic without dropping data or exceeding latency targets.
5. **Disaster recovery tests:** Simulate a full region failure. Verify that the system recovers with the correct budget state.
