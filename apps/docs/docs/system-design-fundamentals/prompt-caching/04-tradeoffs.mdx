## The Core Tension

Every prompt caching system faces a fundamental tension: cache freshness versus cache hit ratio. A cache that prioritizes freshness invalidates entries aggressively, shortening TTLs and eagerly purging at every prompt update. This approach guarantees that the model always processes the latest prompt content. But it also means most requests miss the cache and pay the full prefill cost. A cache that prioritizes hit ratio keeps entries alive as long as possible, extending TTLs and resisting invalidation. This approach maximizes latency savings and token cost reduction. But it risks serving stale or incorrect content when the underlying data has changed.

The tension is not a bug. It is the defining design decision of any prompt caching system. The right balance depends on the application. A medical diagnosis assistant must prioritize freshness over hit ratio. A content summarization tool can prioritize hit ratio over freshness. Financial applications must prioritize freshness because stale data could lead to regulatory violations. Customer support chatbots can tolerate some staleness because the knowledge base changes infrequently. The system must be configurable at the prefix level, not just globally, because different prefixes within the same prompt may have different freshness requirements.

The quantitative impact of this trade-off is substantial. For a typical customer support application with a 4,000-token shared prefix, a TTL of 5 minutes results in a hit ratio of approximately 30 percent and average first-token latency of 1,100 milliseconds. A TTL of 1 hour results in a hit ratio of approximately 65 percent and average first-token latency of 550 milliseconds. A TTL of 24 hours results in a hit ratio of approximately 80 percent and average first-token latency of 320 milliseconds. The 50 percentage point difference in hit ratio translates to a 3x difference in latency and a 2x difference in cost.

## Decision Matrix

The following matrix summarizes the trade-offs across key dimensions. Each dimension is analyzed with concrete numbers and scenarios.

| Dimension | Aggressive Freshness | Balanced | Aggressive Hit Ratio |
|---|---|---|---|
| TTL | 1 to 5 minutes | 30 minutes to 2 hours | 6 to 24 hours |
| Invalidation Style | Eager, immediate | Event-driven with delay | Lazy, on next access |
| Hit Ratio | 20 to 40 percent | 50 to 70 percent | 70 to 90 percent |
| Staleness Risk | Very low (under 1%) | Moderate (1 to 5%) | High (5 to 20%) |
| Prefill Cost | High (many misses) | Moderate | Low |
| First Token Latency | 1,000 to 2,000 ms | 200 to 1,000 ms | 50 to 200 ms |
| Cache Management Complexity | Low | Medium | High |
| Memory Efficiency | Low (many short-lived entries) | Moderate | High |
| Best Use Case | Medical, financial, legal | Customer support, education | Static content, documentation |

### Cache Size versus Hit Ratio

The cache size directly determines the hit ratio. A large cache retains more entries, increasing the probability that an incoming prefix matches a cached entry. The relationship follows a power law: doubling the cache size increases the hit ratio by approximately 10 to 15 percentage points, but only up to a saturation point. Beyond the working set size, additional cache capacity provides diminishing returns. The working set size is the number of distinct prefixes that receive enough traffic to justify caching.

### Prefix Length versus Flexibility

Longer prefixes capture more shared context and provide greater latency savings on a hit. But longer prefixes are more likely to be unique to a single request, reducing the hit ratio. A 4,000-token prefix matches fewer requests than a 1,000-token prefix because longer prefixes are more specific. The optimal prefix length is the longest prefix that still produces a meaningful number of matches across requests. In practice, the optimal prefix length is the one that maximizes the product of hit ratio and prefix length.

### Storage Tier versus Latency

RAM storage provides sub-millisecond access latency but is expensive, costing approximately $5 to $10 per GB per month. SSD storage provides 1 to 5 millisecond access latency but is cheaper by a factor of 10, costing approximately $0.50 to $1 per GB per month. The choice between RAM and SSD depends on the hit rate and the latency budget. For a system serving real-time chat applications with a 500-millisecond latency budget, RAM is necessary for the common case. For a batch processing system with a 10-second latency budget, SSD is sufficient.

### Replication versus Storage Cost

Replication improves fault tolerance and availability. A replication factor of 2 doubles the storage cost but guarantees that a single node failure does not cause data loss. A replication factor of 3 triples the storage cost but survives two simultaneous node failures. The trade-off is between the cost of additional storage and the cost of a cache miss cascade after a node failure. A cache miss cascade can increase inference load by 2 to 5x, potentially overwhelming the inference cluster.

## CAP Positioning

Prompt caching systems are AP (Availability and Partition Tolerance) systems in the CAP theorem. The cache prioritizes availability: it will almost always return a response, even if that response is stale or the cache is degraded. The cache trades consistency for availability. Two consecutive requests for the same prefix may return different KV caches if the underlying data has changed between requests. This is acceptable because the prompt is a cooperative optimization: the application controls the prompt content and is responsible for maintaining consistency.

The P in CAP is critical. Cache nodes can fail, networks can partition, and the system must continue to function. When a cache node becomes unreachable, the system routes requests to the remaining nodes. If no node can serve a cache hit, the request falls through to full inference. This graceful degradation is the key to the system's partition tolerance.

## Failure Modes

### Failure Mode 1: Stale Cache Serves Incorrect Content

The most dangerous failure mode. A cached prefix becomes stale because the underlying knowledge base changed, but the cache entry has not been invalidated. The model generates a response based on the old knowledge. A customer asks about the current refund policy. The model responds with last month's policy. The customer receives a refund based on incorrect information. The financial impact can be severe.

**Mitigation**: Short TTLs for dynamic content. Explicit invalidation triggered by content updates. Version tagging with automatic invalidation on version mismatch. Continuous validation of cached entries against the source of truth. Periodic random sampling of cache hits to verify correctness.

### Failure Mode 2: Cache Stampede

A cache stampede occurs when many requests miss the cache simultaneously, all triggering full inference at the same time. This can happen after a deployment when the cache is cold, after a bulk invalidation event, or after a TTL expiry of a very popular prefix. The stampede overloads the inference servers, causing request queuing, timeouts, and cascading failures. The stampede can increase inference load by 5 to 10x within seconds.

**Mitigation**: Cache warming before deployments. Request coalescing (only one request for a given prefix triggers inference; others wait for the result). Staggered TTLs with jitter to prevent synchronized expiry. Rate limiting on cache miss requests.

### Failure Mode 3: Memory Exhaustion

The KV cache entries are large. A single entry can consume 10 GB or more. If the cache size is not properly bounded, a burst of unique prefixes can exhaust all available memory. The eviction controller removes entries, but the eviction itself consumes CPU and I/O. In the worst case, the system enters a death spiral: entries are evicted as fast as they are created, the hit ratio drops to zero, and the eviction overhead consumes the CPU that should be serving requests.

**Mitigation**: Hard memory limits with no-overcommit policy. Admission control that rejects new cache entries when memory usage exceeds a threshold. Monitoring and alerting on memory usage trends. Separate memory pools for L1 and L2 to prevent one from starving the other.

### Failure Mode 4: Network Congestion from Cache Transfers

Moving KV cache tensors between the cache cluster and the inference servers requires high network bandwidth. A single cache hit transfers 10 GB of data. At 100 cache hits per second, the network bandwidth is 100 GB per second. If the network link is insufficient, cache hits become slower than cache misses, defeating the purpose of caching. The network congestion can also affect other services sharing the same network infrastructure.

**Mitigation**: Locality-aware scheduling that routes requests to inference servers with local cache storage. Compression of KV cache tensors before transfer. Tiered storage with L1 on the same machine as the inference server. Network bandwidth reservation for cache transfers.

### Failure Mode 5: False Cache Hits from Semantic Caching

Semantic caching matches prompts based on embedding similarity, not exact text. A false cache hit occurs when the embedding distance is below the threshold but the prompts are semantically different. The model reuses a KV cache that was computed for a different prompt, producing an incorrect or nonsensical response. The false positive rate depends on the embedding quality and the threshold setting.

**Mitigation**: Conservative similarity thresholds. Validation layer that checks the actual prompt text before using the cached KV cache. Fallback to full inference when the similarity is ambiguous. Different thresholds for different prompt types.

### Failure Mode 6: Cache Poisoning

An attacker crafts a prompt that populates the cache with a malicious KV cache entry. When subsequent users trigger a cache hit for a similar prefix, the model generates responses based on the malicious KV cache. The attacker can inject biases, misinformation, or harmful content into the model's output. This is a security vulnerability that can affect all users who share the same cached prefix.

**Mitigation**: Cache entry validation before serving. Rate limiting on cache population. Authentication and authorization for cache writes. Audit logging for all cache operations. Isolated cache per tenant to prevent cross-tenant poisoning.

## Anti-Patterns

### Anti-Pattern 1: Universal Long TTL

Setting a 24-hour TTL for all prefixes regardless of content freshness. This maximizes the hit ratio but guarantees that any mid-day prompt update will serve stale content for up to 24 hours. The correct approach is to set TTL based on the content update frequency of each prefix.

### Anti-Pattern 2: Ignoring Prompt Diversity

Assuming that prompt diversity is low and that a small cache will suffice. In practice, prompt diversity grows with user count. A system with 100,000 users generates 10,000 to 50,000 distinct prompt prefixes per day. The cache must be sized for the actual diversity, not the imagined diversity.

### Anti-Pattern 3: Cache-Only Deployment

Deploying the prompt caching system without a fallback path. If the cache is down, the inference pipeline is broken. Every production deployment must have a fallback path that bypasses the cache and performs full inference. The fallback must be tested regularly to ensure it works under failure conditions.

### Anti-Pattern 4: Treating All Prompts Equally

Applying the same caching policy to all prompts regardless of their criticality. High-stakes prompts (medical, financial, legal) should use strict freshness policies with short TTLs and explicit invalidation. Low-stakes prompts (content summarization, casual chat) can use relaxed policies with long TTLs.

### Anti-Pattern 5: Over-Reliance on Semantic Caching

Assuming that semantic caching is a universal solution and applying it to all prefixes. Semantic caching works well for free-form natural language prompts but poorly for structured prompts with specific formatting requirements. The false cache hit rate for structured prompts can exceed 20 percent.

### Anti-Pattern 6: No Cache Warming

Deploying new code or new prompt templates without precomputing the cache. The first 5,000 requests after a deployment all experience cache misses, causing a latency spike that triggers timeout alarms and frustrates users.

## When NOT to Use

Prompt caching is not always the right solution. Consider the following scenarios where prompt caching provides minimal benefit or is actively harmful.

**Highly Dynamic Prompts**: If every prompt is unique and shares no common prefix with any other prompt, caching provides zero benefit. A system that generates completely novel prompts for each request should not implement prompt caching.

**Very Short Prompts**: If the average prompt length is under 512 tokens, the prefill phase is fast enough that caching overhead exceeds the savings. The Bloom filter check, cache index lookup, and network transfer of the KV cache add latency that may not be recovered.

**Rapidly Changing Content**: If the prompt content changes every 30 seconds, the cache TTL is too short to provide meaningful savings. The system spends more time populating the cache than serving from it.

**Latency-Insensitive Workloads**: If the application has a 30-second latency budget and the full prefill takes only 2 seconds, caching provides minimal user-facing benefit. The operational complexity of the cache may not justify the latency savings.

**Regulatory Constraints**: Some regulated industries require that every inference is auditable and reproducible. Caching the KV cache introduces ambiguity about which content was used for a given response. Full inference provides a clean audit trail.

## Quantitative Trade-off Analysis

### Memory vs Hit Ratio

The relationship between cache memory (number of entries) and hit ratio follows a power law distribution. The first 10 percent of cache entries (the most popular prefixes) typically account for 50 percent of cache hits. The next 20 percent account for 25 percent of hits. The remaining 70 percent account for 25 percent of hits.

For a system with 500 distinct prefixes:
- 50 entries: 50 percent hit ratio (covers the top 10 most popular prefixes)
- 125 entries: 65 percent hit ratio (covers the top 25 prefixes)
- 250 entries: 75 percent hit ratio (covers the top 50 prefixes)
- 500 entries: 82 percent hit ratio (covers the top 100 prefixes)

Beyond 100 entries, each additional entry provides diminishing returns. The system should allocate cache memory to the point where the marginal benefit of an additional entry equals the marginal cost of the memory.

### TTL vs Hit Ratio

The TTL determines how long an entry remains valid. The optimal TTL depends on the access frequency distribution:
- Frequent access (every 5 minutes): TTL should be at least 30 minutes
- Moderate access (every 30 minutes): TTL should be at least 2 hours
- Infrequent access (every 2 hours): TTL should be at least 12 hours
- Rare access (once per day): TTL should be 24 hours

The hit ratio as a function of TTL:
- 5 minutes: 25 percent hit ratio
- 30 minutes: 45 percent hit ratio
- 2 hours: 60 percent hit ratio
- 6 hours: 70 percent hit ratio
- 24 hours: 75 percent hit ratio

Doubling the TTL from 30 minutes to 1 hour increases the hit ratio by 7 percentage points. Doubling from 6 hours to 12 hours increases by only 3 percentage points. The optimal TTL balances the hit ratio gain against the staleness risk.

### Cache Lookup Overhead vs Savings

Cache lookup overhead includes the Bloom filter check (1 microsecond), the index query (200 microseconds to 2 milliseconds), and the KV cache retrieval (200 microseconds to 5 milliseconds). The total overhead is 1 to 7 milliseconds.

The savings from a cache hit depend on the prefix length. For a 4,000-token prefix, the prefill time saved is approximately 1,500 milliseconds. The net benefit of a cache hit is approximately 1,493 to 1,499 milliseconds. The overhead is 0.1 to 0.5% of the savings.

For a 512-token prefix, the prefill time saved is approximately 200 milliseconds. The net benefit is 193 to 199 milliseconds. The overhead is 0.5 to 3.5% of the savings.

As the prefix length decreases, the overhead becomes a larger fraction of the savings. At prefix lengths below 128 tokens, the overhead may exceed the savings, making caching counterproductive.

## Decision Framework

Use this framework to decide whether prompt caching is appropriate for your workload.

**Question 1**: Is there a shared prefix across requests?
- Yes: Proceed to Q2
- No: Do not cache (zero benefit)

**Question 2**: Is the prompt longer than 512 tokens?
- Yes: Proceed to Q3
- No: Do not cache (overhead exceeds savings)

**Question 3**: Does the shared prefix change less frequently than every 5 minutes?
- Yes: Proceed to Q4
- No: Do not cache (TTL too short to provide savings)

**Question 4**: Is the cache sized for the working set?
- Yes: Proceed to implement caching
- No: Either resize the cache or reduce the working set through prompt consolidation

**Question 5**: Is the application sensitive to latency or cost?
- Latency-sensitive: Use aggressive caching with longer TTLs and larger cache
- Cost-sensitive: Use balanced caching with moderate TTLs and exact prefix matching
- Freshness-critical: Use conservative caching with short TTLs and explicit invalidation

## Operational Trade-offs

### Frequency of Cache Warming vs Resource Cost

Warming runs consume GPU resources that could otherwise serve inference requests. The trade-off is between cache freshness and inference capacity.

**Warming every 30 minutes**: Cache is always fresh but consumes 10 percent of GPU capacity.
**Warming every 2 hours**: Cache is moderately fresh, consumes 2.5 percent of GPU capacity.
**Warming every 6 hours**: Cache may be stale for 10 percent of requests, consumes 0.8 percent of GPU capacity.
**Warming on demand only**: Cache is stale after deployment, consumes 0 percent of GPU capacity for warming.

For most production systems, warming every 1 to 2 hours provides the best balance.

### Monitoring Granularity vs Storage Cost

Detailed monitoring (per-prefix hit ratio, per-node latency) is expensive to store but invaluable for debugging.

**Per-request logging**: Full trace of every cache operation. Storage cost: 1 TB per day for 1M requests.
**Sampled logging**: 1 percent of requests are fully logged. Storage cost: 10 GB per day.
**Aggregated metrics**: Hit ratio, eviction rate, and latency at 1-minute granularity. Storage cost: 100 MB per day.
**Average-only metrics**: Hourly averages only. Storage cost: 1 MB per day.

For production systems, use aggregated metrics for alerting and sampled logging with 1 percent sampling for debugging.

## Conflict Resolution: When Freshness and Hit Ratio Clash

### Scenario 1: Knowledge Base Update During Peak Traffic

The knowledge base is updated at 2:00 PM, right in the middle of peak traffic at 2,000 requests per second. The cached prefixes are now stale. The system must choose between serving stale content (high hit ratio, low latency) or invalidating the cache (zero hit ratio, high latency, risk of cache stampede).

**Resolution**: Use a two-phase invalidation. Phase 1: Mark the cache entries as "probationary" and serve them with a warning header. The application monitors the warning headers and decides whether to accept the stale response. Phase 2: After 30 seconds, invalidate the cache entries. The 30-second delay allows the warming pipeline to precompute the new prefixes. The cache stampede risk is reduced because the new prefixes are ready before the invalidation.

### Scenario 2: A/B Testing of Prompt Templates

The team is running an A/B test with two different prompt templates. Template A is the control (70 percent of traffic). Template B is the variant (30 percent of traffic). The cache contains entries for template A. Template B requests miss the cache because the prefix is different.

**Resolution**: Use cache version tags. Tag each entry with the template version. The cache stores entries for both versions. The cache size increases by 30 percent to accommodate the additional entries. The warming pipeline precomputes entries for both templates.

### Scenario 3: Seasonal Traffic Patterns

A retail company has 10x traffic during Black Friday compared to normal days. The cache is sized for normal traffic. During Black Friday, the cache is overwhelmed with new prefixes.

**Resolution**: Use dynamic cache sizing with a pre-allocated Black Friday budget. The cache size is increased by 5x two weeks before Black Friday. The warming pipeline runs every 30 minutes instead of every 2 hours. The eviction policy is temporarily changed to LFU to retain the most popular Black Friday prefixes.

## Long-Term Maintenance Considerations

### Cache Drift Over Time

As the application evolves, the prompt structure changes. The cache hit ratio drifts down over time as the cached prefixes become less relevant. The system must detect and correct this drift.

**Detection**: Monitor the hit ratio trend over 7-day and 30-day windows. A downward trend of 2 percentage points per week indicates cache drift.

**Correction**: Re-run the prefix analysis every 30 days. Update the cache configuration (prefix length, cache size, warming targets) based on the new prefix distribution. The cache may need to be fully invalidated and rebuilt from scratch.

### Prompt Template Versioning

Prompt templates should be versioned. Each version has a unique identifier. The cache uses the version identifier as part of the cache key. When a new version is deployed, the old cache entries are automatically invalidated.

**Versioning scheme**: `{template_version}:{prefix_hash}`. When the template version changes, all cache entries with the old version are invalidated. The new version starts with a cold cache, which is mitigated by the warming pipeline.

### Dependency Management

The cache system depends on the LLM inference engine, the cache cluster, the cache index, and the warming pipeline. Each dependency must be versioned and tested together.

**Dependency matrix**: 
- Cache system v2.0 requires inference engine v1.5+
- Cache system v2.0 requires cache cluster v3.0+
- Cache system v2.0 requires warming pipeline v1.2+

The dependencies are tested in a staging environment that mirrors production. The cache system is deployed only after all dependencies are verified.
