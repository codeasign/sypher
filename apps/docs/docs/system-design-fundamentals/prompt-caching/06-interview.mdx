## Question Types

### Type 1: Design Prompt Caching for an LLM API Service

The interviewer asks you to design a prompt caching system for a large-scale LLM API service serving millions of requests per day. This is the most common question type. The question tests your understanding of the entire caching pipeline: prefix extraction, cache lookup, storage, eviction, and integration with the inference engine. The interviewer will probe your understanding of the KV cache, the prefill phase, and the decode phase.

**Example**: "Design a prompt caching system for an LLM API that serves 10 million requests per day. The average prompt length is 8,000 tokens. Sixty percent of each prompt is shared across requests from the same customer. The system must reduce latency by 50 percent and cost by 40 percent."

### Type 2: Compare Caching Strategies

The interviewer presents two or three prompt caching strategies (exact prefix, semantic, segment-level) and asks you to compare them. The question tests your understanding of the trade-offs between precision, recall, latency, and memory. The interviewer wants to see that you can select the right strategy for the right workload.

**Example**: "Compare exact prefix caching, semantic caching, and segment-level caching. Under what conditions would you choose each one? What are the performance trade-offs?"

### Type 3: Design Cache Invalidation for Dynamic Content

The system prompt changes every 30 minutes. The knowledge base is updated by a separate pipeline. The interviewer asks you to design a cache invalidation strategy that balances freshness against hit ratio. This question tests your understanding of consistency and your ability to design for real-world constraints.

**Example**: "The knowledge base for a customer support chatbot is updated every 30 minutes by a content management system. The cache TTL is currently 1 hour. Design an invalidation strategy that ensures freshness within 5 minutes of an update while maintaining a hit ratio above 50 percent."

### Type 4: Estimate the Capacity Requirements

The interviewer gives you the traffic volume, the model size, the average prompt length, and the target hit ratio. You must estimate the required cache size, the number of cache nodes, the network bandwidth, and the cost. This is a back-of-the-envelope estimation question.

**Example**: "A 70-billion-parameter model serves 500,000 requests per day. The average prompt is 6,000 tokens. The target hit ratio is 60 percent. Estimate the cache size required, the number of cache nodes, and the network bandwidth."

### Type 5: Debug a Production Cache Issue

The hit ratio dropped from 70 percent to 20 percent over the last hour. The interviewer asks you to diagnose the problem. The question tests your systematic debugging skills and your knowledge of cache failure modes.

**Example**: "The cache hit ratio for your prompt caching system dropped from 72 percent to 18 percent in the last hour. The system is a distributed cache with 6 nodes, LRU eviction, and a TTL of 1 hour. Walk me through your debugging process."

### Type 6: Design for Multi-Region

The service is deployed in three regions. Each region has its own inference cluster. The interviewer asks you to design a prompt caching system that works across regions, handles cache replication, and maintains consistency.

**Example**: "Design a prompt caching system for an LLM API deployed in US-East, EU-West, and AP-Southeast. Each region handles 3 million requests per day. Prompts are region-specific but share some common prefixes. Design the cache replication and invalidation strategy."

### Type 7: Integrate with Existing Systems

The company already uses Redis for session caching. The interviewer asks you to design the integration between the existing Redis deployment and the new prompt caching system.

**Example**: "The company has an existing Redis cluster for session caching. They want to add prompt caching. Should they use the same Redis cluster or a separate one? What are the trade-offs?"

### Type 8: Design a Cache Warming Pipeline

The team deploys new prompt templates every week. Each deployment causes a cold start. The interviewer asks you to design a cache warming pipeline that minimizes the cold start impact.

**Example**: "Design a cache warming pipeline for a system that deploys new prompt templates every Monday. The deployment causes 4,000 cache misses in the first minute. Design a warming strategy that reduces the cold start to under 100 cache misses."

### Type 9: Prompt Caching for Code Completion

The company is building a code completion service similar to GitHub Copilot. The interviewer asks you to design a prompt caching strategy optimized for code completion.

**Example**: "Design a prompt caching system for a code completion service. The prompt includes the current file, the open tabs, and the cursor position. How would you structure the cache to maximize reuse across completions within the same file?"

### Type 10: Cost Optimization

The interviewer gives you the current cost breakdown (prefill cost, decode cost, token cost) and asks you to design a cache that minimizes the total cost of ownership.

**Example**: "The current cost per request is $0.01: $0.007 for prefill (70 percent) and $0.003 for decode (30 percent). Design a caching strategy that reduces the cost by 50 percent. Account for the cache storage and network costs."

## Interviewer Testing Goals

The interviewer is testing four things.

**Depth of Understanding**: Can you explain why prompt caching works at the transformer level? Do you understand the KV cache, the prefill phase, and the decode phase? Do you know the difference between caching the prefill and caching the generation?

**System Design Instinct**: Can you design a system that balances trade-offs? Do you know when to use exact prefix caching versus semantic caching? Can you estimate capacity requirements from first principles?

**Failure Mode Awareness**: Do you know what can go wrong? Can you design a system that degrades gracefully under failure conditions? Do you have practical experience with caching in production?

**Communication Clarity**: Can you explain complex concepts clearly? Do you use diagrams effectively? Do you structure your answer with a clear flow?

## Level Expectations

### L3 (Entry Level)

At this level, the interviewer expects a basic understanding of what prompt caching is and why it is useful. You should be able to define the KV cache, explain the difference between prefill and decode, and describe a simple in-memory caching implementation. You should mention cache hit and cache miss. You do not need to discuss distributed caching, eviction policies, or consistency.

**Sample Answer (L3)**: "Prompt caching stores the KV cache from the prefill phase so that requests with the same prompt prefix can skip the prefill and start directly at the decode phase. The KV cache stores the key and value tensors from each attention layer. For a simple implementation, I would use a hash map that maps prompt prefix hashes to KV cache tensors. On a cache hit, I load the cached tensors and skip the prefill. On a cache miss, I run the full prefill and store the result."

### L4 (Mid Level)

At L4, the interviewer expects a deeper understanding of the system design. You should discuss exact prefix caching versus semantic caching, describe the TTL and eviction policy, and mention the integration with the inference server. You should identify the key trade-off between cache hit ratio and memory consumption.

**Sample Answer (L4)**: "I would design a two-tier caching system. L1 stores hot entries in GPU memory for sub-millisecond access. L2 stores warm entries in CPU memory for millisecond access. The eviction policy is LRU with TTL-based expiry. I would use exact prefix caching by default because it is fast and deterministic. For prompts with high variability, I would consider semantic caching with a cosine similarity threshold. The key trade-off is between the hit ratio and the memory consumption. Each cache entry consumes 2.6 MB per token for a 70-billion-parameter model. A 4,000-token prefix consumes 10.4 GB."

### L5 (Senior Level)

At L5, the interviewer expects a complete system design with distributed architecture, fault tolerance, and capacity planning. You should discuss consistent hashing for cache distribution, replication for fault tolerance, Bloom filters for fast miss detection, and cache warming for cold start mitigation.

**Sample Answer (L5)**: "I would design a distributed cache cluster with six nodes. The cache index is stored in a replicated in-memory store using Raft consensus. The prefix matching uses a trie data structure sharded by the first token. A Bloom filter sits in front of the cache index to eliminate cache miss lookups. The cache uses two-tier storage: RAM for hot entries and SSD for warm entries. The cache is warmed before deployments using the previous day's traffic log. The system monitors hit ratio, eviction rate, and lookup latency. Alerts fire when the hit ratio drops below 40 percent."

### L6 (Staff Level)

At L6, the interviewer expects depth in multiple areas plus breadth across the infrastructure. You should discuss multi-region architecture, cross-region cache invalidation, cache-aware prompt engineering, cost modeling, and failure mode analysis with chaos engineering.

**Sample Answer (L6)**: "The system spans three regions with asynchronous cache replication. Each region has its own cache cluster and index. The global cache index maintains regional cache affinity. Writes go through a cross-region message bus for invalidation propagation. I would implement cache-aware prompt engineering as part of the development workflow. The prompt templates include cache optimization as a design goal. The cost model accounts for cache population cost, storage cost, and network transfer cost. The system uses chaos engineering to validate failure modes: cache node crash, network partition, and cache index corruption are tested regularly."

### L7 (Principal Engineer)

At L7, the interviewer expects you to discuss the strategic implications of prompt caching for the business, the interaction with model selection and pricing, and the long-term evolution of the caching approach as models and infrastructure evolve.

**Sample Answer (L7)**: "The prompt caching strategy must align with the business goals. If the goal is user experience, prioritize low latency with aggressive caching. If the goal is cost reduction, prioritize high hit ratio with large caches. The caching strategy also interacts with the model selection. Smaller models have smaller KV caches and may not need caching at all. Larger models benefit more from caching. The long-term evolution includes learned caching policies using reinforcement learning to predict which prefixes to cache and for how long."

### L8 (Distinguished Engineer / Fellow)

At L8, the interviewer expects you to shape the industry discussion on prompt caching, define best practices, and influence the model architecture itself. You should discuss how caching requirements should inform model design: KV cache quantization, attention mechanism optimization, and prompt structure standardization.

## FAANG Framework

Use this five-step framework when answering system design questions about prompt caching.

**Step 1: Clarify Scope and Requirements** -- Ask clarifying questions about the traffic volume, the model size, the average prompt length, the prompt diversity, and the target hit ratio. Ask about the deployment scale (single region or multi-region). Ask about the freshness requirements. Ask about the budget constraints. The answers to these questions determine the entire design. Do not skip this step.

**Step 2: Define the MVP** -- Start with a single-node exact prefix cache. Describe the cache lookup (hash map of prefix to KV cache), the TTL management (configurable TTL per entry), and the eviction policy (LRU). Make sure the interviewer agrees with the baseline before moving to more complex features. The MVP should handle the core use case with minimal complexity.

**Step 3: Scale the Architecture** -- Add distributed caching with consistent hashing, cache index replication, Bloom filter for fast miss detection, and two-tier storage (L1 RAM, L2 SSD). Describe the consistent hashing scheme and how the cache index is distributed. Estimate the number of cache nodes and the network bandwidth. The scaling step addresses the limitations of the MVP.

**Step 4: Add Advanced Features** -- Discuss semantic caching (embeddings + similarity search), segment-level caching (independent segments), cache warming (precompute popular prefixes), and cache-aware prompt engineering (design prompts for cacheability). Describe the trade-offs for each feature and the conditions under which each is useful.

**Step 5: Cover Failure Modes and Operations** -- Discuss cache stampede (request coalescing, jittered TTLs), stale cache (explicit invalidation, version tagging), memory exhaustion (hard limits, admission control), and network congestion (locality-aware scheduling, compression). Describe the monitoring, alerting, and recovery procedures. Mention the deployment strategy (rolling update) and the cold start mitigation (cache warming).

## Phrase Bank

Use these phrases to demonstrate depth during the interview.

- "The KV cache is the intermediate representation of the prompt after the prefill phase. It is a sequence of tensors, one per token per layer, that represent the processed attention state."
- "Exact prefix caching is deterministic and fast but brittle. A single character difference breaks the cache match. This is the trade-off between precision and recall in caching."
- "The Bloom filter is a probabilistic data structure that provides fast cache-miss detection. False positives are possible but false negatives are not."
- "The eviction policy determines which entries to remove when the cache is full. LRU is the default because it works well for workloads with temporal locality. LFU is better for stable popularity distributions."
- "Cache warming is the process of precomputing cache entries before traffic arrives. Without warming, every deployment causes a cold start where the hit ratio drops to zero."
- "The cache size is determined by the product of the number of distinct prefixes, the prefix length, and the per-token KV cache size. For a 70-billion-parameter model, each token consumes approximately 2.6 MB of KV cache."
- "Cache invalidation is the hardest problem in prompt caching. The system must balance freshness against hit ratio. Explicit invalidation is the most reliable approach."
- "Segment-level caching divides the prompt into independently cacheable chunks. This provides more flexibility than prefix caching but adds complexity in segment identification."
- "The network bandwidth between the cache cluster and the inference servers must support moving KV cache tensors at the request rate. At scale, this is often the bottleneck."
- "Cache-aware prompt engineering places shared context at the beginning of the prompt and uses consistent formatting. This simple practice doubles the cache hit ratio."
- "The p50 cache hit latency should be under 5 milliseconds, including the Bloom filter check and the cache index lookup. If the lookup is slower than the prefill, the cache is not worth having."
- "A cache stampede occurs when many requests miss the cache simultaneously, all triggering full inference. Request coalescing and jittered TTLs prevent this."
- "The replication factor determines the fault tolerance of the cache. A factor of 2 doubles the storage cost but ensures that a single node failure does not cause data loss."
- "Semantic caching uses embedding similarity to match prompts. The threshold controls the precision-recall trade-off. A high threshold produces few false positives but also few cache hits."
- "The cache index stores only metadata, not the full KV cache tensors. This keeps the index small and fast. The KV cache tensors are stored in the cache storage tier."

## Red Flags

Interviewers watch for these red flags.

**Red Flag 1: No Mention of Prefill vs Decode** -- If you cannot distinguish the prefill phase from the decode phase, you do not understand how prompt caching works at the model level. This is a fundamental prerequisite.

**Red Flag 2: Assuming Unlimited Cache** -- If you design a cache without memory limits or eviction policies, the interviewer knows you lack production experience. Always discuss memory constraints.

**Red Flag 3: Ignoring Cache Stampede** -- If you do not mention cache stampede or thundering herd, the interviewer knows you have not operated a cache in production.

**Red Flag 4: No Fallback Path** -- If the system cannot function without the cache, the design is not production-ready. Every cache must have a graceful degradation path.

**Red Flag 5: Over-Engineering the MVP** -- If you start with multi-region semantic caching with reinforcement learning, the interviewer knows you cannot prioritize. Start simple.

**Red Flag 6: Ignoring the Prompt Structure** -- If you design a cache without considering the prompt structure, you miss the most important factor determining cache effectiveness.

**Red Flag 7: No Numbers** -- If you cannot estimate the cache size, the latency, or the bandwidth from first principles, the interviewer questions your engineering judgment.

**Red Flag 8: No Monitoring Plan** -- If you design the system without mentioning metrics, dashboards, or alerts, the interviewer knows you have not operated a production system.

### Drill 2: Estimate Cache Size

A service serves 1 million requests per day. Each request has a 6,000-token prompt. The 70-billion-parameter model has 80 layers and a 4,096-dimensional hidden state. Compute the cache size for 500 distinct prefixes with a replication factor of 2. Answer: approximately 10.4 TB.

### Drill 3: Design a Prefix Trie

Draw a prefix trie for the following prefixes: "You are a helpful assistant", "You are a helpful guide", "You are a system", "User asked about X". Show how each prefix is stored as a path from the root to a leaf.

### Drill 4: Write the Eviction Algorithm

Describe the LRU eviction algorithm in pseudocode. Include the data structures (doubly-linked list, hash map) and the time complexity of each operation.

### Drill 5: Design the Cache Index Schema

Design the database schema for the cache index. Include fields for the hash, the storage location, the entry size, the creation timestamp, the access count, and the TTL.

```sql
CREATE TABLE cache_index (
    prefix_hash VARCHAR(64) PRIMARY KEY,
    cache_node_id INT NOT NULL,
    storage_tier ENUM('L1_RAM', 'L2_SSD') NOT NULL,
    storage_offset BIGINT NOT NULL,
    entry_size_bytes BIGINT NOT NULL,
    created_at TIMESTAMP NOT NULL,
    last_accessed_at TIMESTAMP NOT NULL,
    access_count INT NOT NULL DEFAULT 0,
    ttl_expires_at TIMESTAMP NOT NULL,
    version_tag VARCHAR(32),
    tenant_id VARCHAR(64),
    INDEX idx_access (last_accessed_at),
    INDEX idx_ttl (ttl_expires_at)
);
```

### Drill 6: Calculate Network Bandwidth

A cache hit transfers 10 GB of KV cache tensors. The system serves 100 cache hits per second. Calculate the required network bandwidth. Answer: 100 GB per second or 800 Gbps.

### Drill 7: Design a Cache Warming Schedule

A deployment happens at 2:00 PM. Peak traffic is at 8:00 PM. Design a cache warming schedule that ensures the cache is warm before peak traffic.

**Sample Schedule**:
- 1:30 PM: Begin log analysis of previous 24 hours
- 1:45 PM: Identify top 200 most common prefixes
- 1:50 PM: Start batch prefill on 4 warming GPUs
- 2:00 PM: Deploy new code (cache is already warm)
- 2:00 PM to 4:00 PM: Monitor hit ratio (should be above 60 percent)
- 4:00 PM: Run secondary warming for prefixes that emerged after deployment

### Drill 8: Compare Cost Models

A full prefill costs $0.003 per 1,000 input tokens. A cached prefix costs $0.0003 per 1,000 tokens. Calculate the savings for a 6,000-token prompt where 4,000 tokens are cached. Compare with the cache storage cost.

**Answer**: Full prefill cost: 6,000 * $0.003 / 1,000 = $0.018. Cached cost: 4,000 * $0.0003 / 1,000 + 2,000 * $0.003 / 1,000 = $0.0072. Savings per request: $0.0108 (60 percent). At 10,000 requests per day with 70 percent hit rate: daily savings = 7,000 * $0.0108 = $75.60. Monthly savings: $2,268. Cache storage cost: 500 entries * 10.4 GB = 5.2 TB at $0.08/GB/month = $416/month. Net monthly savings: $1,852.

## Mock Interview Transcripts

### Mock Interview 1: Design Prompt Caching for LLM API (L5 Target)

**Interviewer**: "Design a prompt caching system for an LLM API that processes 1 million requests per day."

**Candidate**: "Let me start by asking clarifying questions. What is the average prompt length? How many distinct prompt prefixes do you expect? What is the target hit ratio? What is the latency budget?"

**Interviewer**: "Average prompt is 6,000 tokens. You can assume 200 distinct prefixes. We want at least 60 percent hit ratio. The latency budget is 500 milliseconds."

**Candidate**: "Good. Let me define the scope. With 200 distinct prefixes and a 4,000-token shared prefix, the KV cache for each entry is approximately 10.4 GB for a 70B model. The total cache size is about 2 TB. With a replication factor of 2, that is 4 TB across the cluster. To minimize lookup latency, I will use a Bloom filter for fast miss detection. The Bloom filter eliminates approximately 70 percent of the cache index queries for misses. The prefix matching uses a hash-based index with consistent hashing across 4 cache nodes."

**Interviewer**: "How do you handle the cache miss for the first request of the day?"

**Candidate**: "The first request of the day for each prefix is a cache miss. The system processes it with full prefill and stores the KV cache for subsequent requests. This is expected behavior. To mitigate the cold start, I would implement a cache warming pipeline that precomputes the top 50 prefixes every hour. This covers the majority of the traffic. The warming runs on a separate set of GPU instances and populates the cache before the peak traffic arrives."

**Interviewer**: "What happens when a cache node fails?"

**Candidate**: "With a replication factor of 2, each entry is stored on two nodes. When one node fails, the other node still serves the cache entries. The cache index is updated to remove the failed node. The remaining nodes rebalance the entries. New entries are created with a replication factor of 2 to fill the gap left by the failed node. The system continues to function with reduced capacity until the failed node is replaced."

### Mock Interview 2: Debug Production Cache Issue (L6 Target)

**Interviewer**: "The cache hit ratio dropped from 72 percent to 18 percent in the last hour. Walk me through your debugging process."

**Candidate**: "I start with the metrics dashboard. First, I check the cache node health. If nodes have failed, that explains the drop. Second, I check the eviction rate. A sudden spike in evictions would indicate that the working set size has exceeded the cache capacity. Third, I check the TTL expiry pattern. If a large batch of entries expired simultaneously, that could cause the drop. Fourth, I check the request rate. A sudden increase in unique prefixes would dilute the hit ratio."

**Interviewer**: "All nodes are healthy. The eviction rate is normal. The TTL is not the issue. But the request rate increased by 3x in the last hour."

**Candidate**: "The 3x increase in request rate with normal eviction means the new requests are for different prefixes than what is in the cache. The existing cache was optimized for the old traffic pattern. The new traffic is either from a different user base or from a different application. I would sample the new prefixes and compare them to the cached prefixes. If they are different, the solution is to adjust the cache warming to include the new prefixes. If the new traffic is temporary (a flash crowd), the cache will adapt naturally as the new prefixes populate the cache."

**Interviewer**: "The new traffic is from a marketing campaign that is expected to last 48 hours. What do you do?"

**Candidate**: "I would immediately trigger cache warming for the prefixes from the marketing campaign. I would analyze the campaign prompts to identify the common prefix, and precompute the KV cache for that prefix. This would restore the hit ratio within 15 minutes. I would also increase the cache size temporarily to absorb the additional prefixes, then revert after the campaign ends."

### Mock Interview 3: Choose Between Caching Strategies (L4 Target)

**Interviewer**: "Compare exact prefix caching, semantic caching, and segment-level caching. When would you use each?"

**Candidate**: "Exact prefix caching is the default choice. It is simple, fast, and deterministic. The lookup is O(1) with a hash map, and there are no false positives. I use exact prefix caching when the prompt prefix is guaranteed to be identical across requests. This is common for system prompts, knowledge bases, and instruction templates."

"Semantic caching is useful when the prompt text varies but the meaning is the same. For example, 'What is the refund policy?' and 'Can you tell me about returns?' are different text but the same semantic intent. Semantic caching uses embeddings to find similar prompts. The trade-off is higher lookup latency (5 to 50 milliseconds) and the risk of false positives. I use semantic caching for free-form user queries where exact text matching is too restrictive."

"Segment-level caching is useful when the prompt is assembled from multiple reusable segments. For example, a prompt with a system instruction, a knowledge base, a conversation history, and a current question. Each segment can be cached independently, so changing one segment does not invalidate the cache for the others. I use segment-level caching for complex multi-part prompts where different parts have different update frequencies."

"The choice depends on the prompt structure, the variability of the text, and the tolerance for false positives. In practice, I use hybrid caching: exact prefix for the stable parts, semantic for the variable parts, and segment-level for the overall structure."

## System Design Interview Checklist

Use this checklist to ensure you cover all the key points in a prompt caching system design interview.

**Requirements Clarification**:
- [ ] Traffic volume (requests per day, peak QPS)
- [ ] Model architecture (parameter count, layers, hidden dimension)
- [ ] Prompt characteristics (average length, prefix structure, diversity)
- [ ] Performance requirements (target hit ratio, latency budget, cost constraints)
- [ ] Deployment scale (single region or multi-region, number of inference servers)

**Core Design**:
- [ ] Cache index (hash map of prefix hashes to entry metadata)
- [ ] Prefix extraction (configurable prefix length, token-level extraction)
- [ ] Cache lookup (Bloom filter, index query, storage retrieval)
- [ ] Cache population (on cache miss, store KV cache, update index)
- [ ] TTL management (configurable TTL, extension on hit, background cleanup)

**Advanced Features**:
- [ ] Distributed caching (consistent hashing, sharding, replication)
- [ ] Two-tier storage (L1 RAM for hot entries, L2 SSD for warm entries)
- [ ] Cache warming (precompute popular prefixes before deployment)
- [ ] Cache-aware prompt engineering (shared context at start, consistent formatting)
- [ ] Semantic caching (embeddings, similarity search, threshold tuning)

**Failure Modes**:
- [ ] Cache stampede (request coalescing, jittered TTLs)
- [ ] Stale cache (explicit invalidation, version tagging)
- [ ] Memory exhaustion (hard limits, admission control)
- [ ] Network congestion (locality-aware scheduling, compression)
- [ ] Cache poisoning (input validation, rate limiting, audit logging)

**Operations**:
- [ ] Monitoring metrics (hit ratio, eviction rate, lookup latency, memory usage)
- [ ] Alerting thresholds (hit ratio below 40 percent, eviction rate above 100/s)
- [ ] Deployment strategy (rolling update, cache warming before deploy)
- [ ] Runbooks (low hit ratio, node failure, cold start, stale content)
- [ ] Testing strategy (chaos engineering, load testing, failure mode testing)

## Key Formulas to Memorize

Write these formulas on the whiteboard before you start. They demonstrate quantitative depth.

**KV Cache Size Per Token**:
```
bytes_per_token = L * H * (D / H) * 2 * bytes_per_element * 2
```
Where L = layers, H = heads, D = hidden dimension. The first factor of 2 is for K and V. The second factor of 2 is for prefill and decode.

**Example**: 70B model with L=80, H=64, D=8192, FP16: 80 * 64 * 128 * 2 * 2 * 2 = 5.2 MB per token.

**Total Cache Size**:
```
total_cache = num_prefixes * tokens_per_prefix * bytes_per_token * replication_factor
```

**Network Bandwidth**:
```
bandwidth_gbps = cache_hits_per_second * bytes_per_entry * 8 / 1e9
```

**Cost With Caching**:
```
daily_cost = misses * full_tokens * price_per_token
           + hits * new_tokens * price_per_token
           + hits * cached_tokens * cached_price_per_token
```

**Break-Even Hit Ratio**:
```
break_even = infra_cost / (savings_per_hit * daily_requests)
```
Where infra_cost is the daily cache infrastructure cost and savings_per_hit is the token cost saved per cache hit.

## Common Interview Traps to Avoid

**Trap 1: Focusing Only on the Happy Path**
Interviewers often ask "What happens when the cache is full?" or "What happens when a node crashes?" Be prepared with failure mode analysis.

**Trap 2: Ignoring the Model Architecture**
The KV cache size depends on the model architecture. A 70B model with 80 layers has a much larger KV cache than a 7B model with 32 layers. Always start with the model architecture when estimating cache size.

**Trap 3: Confusing Prompt Caching with Response Caching**
Prompt caching stores the intermediate KV cache, not the final response. Response caching is a separate concept that stores the model's output. The interviewer may be testing whether you understand this distinction.

**Trap 4: Overlooking the Network Bandwidth**
The network bandwidth required to move KV cache tensors is often the bottleneck. Many candidates focus on storage and compute but forget to account for the network.

**Trap 5: Assuming Cache Warming is Optional**
In a production deployment, cache warming is not optional. Without warming, the first wave of requests after a deployment all experience cache misses. The resulting latency spike can trigger cascading failures.

**Trap 6: Not Clarifying the Requirements**
Candidates who jump into the design without asking clarifying questions miss important constraints. Always ask about the model size, prompt length, traffic volume, target hit ratio, and latency budget before designing the system.
