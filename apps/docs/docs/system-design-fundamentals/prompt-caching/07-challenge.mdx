## Foundation Challenges

### Challenge 1: Build a Simple Exact Prefix Cache

Implement a single-process exact prefix cache for a toy LLM inference engine. The cache should store KV cache entries for prompt prefixes and serve them on subsequent requests. This challenge tests your understanding of the core caching primitives: storage, lookup, eviction, and concurrency.

**Requirements**:
- Support exact prefix matching only
- Maximum 10 cache entries
- LRU eviction policy
- TTL of 60 seconds per entry
- Thread-safe for concurrent access
- Support for cache hit and miss logging

**Evaluation Criteria**:
- Correctness: Does the cache return the correct entry for a matching prefix?
- Eviction: Does the cache evict the least recently used entry when full?
- TTL: Does the cache expire entries after 60 seconds and treat them as misses?
- Concurrency: Does the cache handle concurrent reads and writes without corruption?
- Logging: Does the implementation log cache hits and misses for monitoring?

**Model Answer**:

The implementation uses `collections.OrderedDict` for O(1) LRU operations, `threading.Lock` for concurrency, and a background thread for TTL cleanup. The `OrderedDict` maintains insertion and access order, making it ideal for LRU: accessing an entry moves it to the end, and eviction removes from the front.

```python
import time
import threading
import logging
from collections import OrderedDict

logger = logging.getLogger(__name__)

class ExactPrefixCache:
    def __init__(self, max_entries: int = 10, ttl: int = 60):
        self.max_entries = max_entries
        self.ttl = ttl
        self._lock = threading.Lock()
        self._cache: OrderedDict[str, tuple[object, float]] = OrderedDict()
        self._hits = 0
        self._misses = 0
        self._cleanup_thread = threading.Thread(target=self._cleanup_loop, daemon=True)
        self._cleanup_thread.start()

    def get(self, prefix: str) -> object | None:
        with self._lock:
            if prefix not in self._cache:
                self._misses += 1
                logger.debug(f"CACHE MISS: prefix={prefix[:50]}...")
                return None
            data, expires_at = self._cache[prefix]
            if time.monotonic() > expires_at:
                del self._cache[prefix]
                self._misses += 1
                logger.debug(f"CACHE MISS (expired): prefix={prefix[:50]}...")
                return None
            self._cache.move_to_end(prefix)
            self._hits += 1
            logger.debug(f"CACHE HIT: prefix={prefix[:50]}...")
            return data

    def set(self, prefix: str, kv_cache: object) -> None:
        with self._lock:
            if prefix in self._cache:
                self._cache.move_to_end(prefix)
            self._cache[prefix] = (kv_cache, time.monotonic() + self.ttl)
            while len(self._cache) > self.max_entries:
                evicted = self._cache.popitem(last=False)
                logger.info(f"EVICTED: prefix={evicted[0][:50]}...")

    @property
    def hit_ratio(self) -> float:
        with self._lock:
            total = self._hits + self._misses
            return self._hits / total if total > 0 else 0.0

    def _cleanup_loop(self) -> None:
        while True:
            time.sleep(30)
            with self._lock:
                now = time.monotonic()
                expired = [k for k, (_, t) in self._cache.items() if t <= now]
                for k in expired:
                    del self._cache[k]
                if expired:
                    logger.info(f"CLEANUP: removed {len(expired)} expired entries")
```

### Challenge 2: Analyze a Traffic Log

You are given a log of 10,000 prompt prefixes from a production system. Each entry contains the full prompt text, the timestamp, and the user ID. Analyze the log to determine the potential cache hit ratio for different prefix lengths and cache sizes.

**Requirements**:
- Compute the prefix for each prompt at lengths of 128, 256, 512, 1,024, and 2,048 tokens
- Count the number of distinct prefixes at each length
- Calculate the cache hit ratio for a cache size of 100 entries with LRU eviction
- Identify the top 10 most common prefixes at each length
- Plot the hit ratio as a function of cache size and prefix length

**Evaluation Criteria**:
- Correctness: Are the counts and ratios calculated correctly?
- Insight: Do you identify the optimal prefix length for this workload?
- Presentation: Do you present the results clearly with a recommendation?

**Model Answer**:

The analysis should show that the hit ratio decreases as the prefix length increases because longer prefixes are more specific to individual requests. The optimal prefix length balances hit ratio against latency savings. A 512-token prefix may achieve a 60 percent hit ratio while providing meaningful latency savings. A 2,048-token prefix may achieve only a 15 percent hit ratio because few requests share such a long prefix.

The hit ratio as a function of cache size should show a power law: doubling the cache size increases the hit ratio by 10 to 15 percentage points, but only until the working set size is reached. Beyond the working set (approximately 500 entries for this workload), additional cache provides minimal benefit.

The top 10 prefixes should include system prompts, knowledge base headers, and common instruction templates.

### Challenge 3: Benchmark Cache Lookup Latency

Design and execute a benchmark that measures the lookup latency of a prompt cache under different load conditions. This challenge tests your understanding of performance measurement and bottleneck identification.

**Requirements**:
- Measure p50, p95, and p99 lookup latency
- Test at loads of 10, 100, and 1,000 lookups per second
- Test with cache sizes of 10, 100, and 1,000 entries
- Compare exact prefix matching with hash-based matching
- Report the overhead of the Bloom filter
- Identify the bottleneck at each load level

**Evaluation Criteria**:
- Methodology: Is the benchmark design sound with proper warmup and measurement?
- Results: Are the latency numbers realistic and justified?
- Analysis: Do you identify the bottleneck correctly?

**Model Answer**:

The benchmark should show that hash-based matching is faster than exact string matching by a factor of 10. The Bloom filter adds 1 microsecond of overhead but eliminates 50 percent of cache index queries. The lookup latency increases with the cache size because of CPU cache effects: a larger cache means more cache misses in the CPU's L1 and L2 caches. At low loads (10 req/s), the bottleneck is the memory latency. At high loads (1,000 req/s), the bottleneck is lock contention.

## Design Challenges

### Challenge 4: Design for a Customer Support Chatbot

A company is building an AI customer support chatbot. The system processes 10,000 tickets per day. Each prompt includes a 4,000-token knowledge base, a 2,000-token conversation history, and a 500-token current question. The knowledge base is updated daily. Design a prompt caching system that maximizes the hit ratio while ensuring freshness.

**Requirements**:
- The hit ratio must be above 60 percent
- The stale response rate must be below 1 percent
- The cache must survive a single node failure
- The system must handle traffic spikes of 2x normal load

**Evaluation**:
- Architecture: Is the design sound and complete?
- Trade-offs: Are the trade-offs clearly identified?
- Numbers: Are the capacity estimates realistic?

**Model Answer**:

Use segment-level caching with three segments: knowledge base (4,000 tokens), conversation history (2,000 tokens), and current question (500 tokens). The knowledge base segment is cached with a TTL of 12 hours and explicit invalidation triggered by the daily update. The conversation history segment is cached per session with a TTL of 30 minutes. The current question segment is not cached because it is unique to each request.

The cache cluster has two nodes with replication factor 2. Each entry consumes approximately 10.4 GB for the knowledge base segment. With 100 distinct knowledge base versions per week, the cache requires 2.08 TB of storage.

The invalidation pipeline is triggered by the knowledge base update job. When the update completes, the pipeline sends an invalidation event that removes all knowledge base cache entries. The invalidation completes within 1 minute.

### Challenge 5: Design for Multi-Tenant API Service

A company operates an LLM API service with 500 tenants. Each tenant has its own system prompt, knowledge base, and usage pattern. Design a prompt caching system that serves all tenants efficiently while isolating cache entries between tenants.

**Requirements**:
- 500 tenants with distinct system prompts
- Total traffic of 100,000 requests per minute
- Tenant isolation: one tenant must not see another tenant's cached data
- Fair cache allocation: large tenants should not starve small tenants

**Evaluation**:
- Multi-tenancy: Is the tenant isolation correct and efficient?
- Resource allocation: Is the cache allocation fair?
- Scalability: Does the design scale to 5,000 tenants?

**Model Answer**:

Use tenant-prefixed cache keys. Each cache entry key includes the tenant ID as a prefix. The cache index is partitioned by tenant ID. Each tenant has a dedicated cache quota. The quota defaults to 1/N of the total cache size but can be adjusted based on the tenant's plan.

The trie data structure is per-tenant. Each tenant has an independent prefix trie that stores their cached prefixes. This ensures that tenant A's prompts do not pollute tenant B's cache. The sharding is also tenant-aware: all entries for a tenant are stored on the same cache node to minimize cross-node queries. This means a single tenant's entries are not distributed across the cluster, simplifying consistency.

The fair allocation uses weighted fair queuing. Each tenant's cache size is proportional to their traffic volume. A tenant with 20 percent of the traffic gets 20 percent of the cache. The eviction policy is per-tenant: when a tenant reaches their quota, the eviction policy runs on that tenant's entries only.

### Challenge 6: Design for Real-Time Translation

A company is building a real-time translation service. The service translates user messages between 10 languages. The prompt includes the source text, the target language, the conversation context, and the translation instructions. The source text is unique for each request, but the translation instructions and conversation context are shared across many requests.

**Requirements**:
- Translation latency must be under 500 milliseconds
- The service handles 50,000 requests per second
- The prompt is 1,500 tokens total (500 tokens instruction + 1,000 tokens source text)
- The instruction segment is the same for each language pair
- The conversation context changes every 5 minutes

**Evaluation**:
- Latency: Does the design meet the latency target?
- Scale: Does the design handle 50,000 requests per second?
- Language pairs: Are the 10 languages handled efficiently?

**Model Answer**:

Cache the instruction segment per language pair. There are 90 distinct instruction segments (10 source languages times 9 target languages, excluding same-language pairs). Each instruction segment is 500 tokens. The KV cache for each instruction segment is approximately 1.3 GB. The total cache size for all instruction segments is approximately 117 GB.

Use exact prefix caching for the instruction segment. The instruction is always the same for a given language pair. The source text is the only variable part. On a cache hit, the prefill skips the first 500 tokens and only processes the 1,000-token source text. The prefill time drops from 750 milliseconds to 500 milliseconds, meeting the latency target.

The cache is pre-warmed for all 90 language pairs. The warming runs every 5 minutes to refresh the cache entries and handle conversation context changes.

## Expert Challenges

### Challenge 7: Learned Cache Eviction Policy

Design a cache eviction policy that learns from the prompt access patterns. The policy should predict which prefixes will be accessed in the next hour and preferentially retain those entries. This challenge tests your ability to design ML-driven systems.

**Requirements**:
- Improve the hit ratio by 10 percent over LRU
- Training must complete within 1 hour on a single GPU
- The policy must adapt to changing access patterns within 30 minutes
- The inference latency of the policy must be under 1 millisecond

**Evaluation**:
- Innovation: Is the approach novel and sound?
- Practicality: Is the approach implementable with reasonable resources?
- Results: Is the 10 percent improvement realistic?

**Model Answer**:

Use a lightweight gradient-boosted decision tree (GBDT) model to predict the probability that each cache entry will be accessed in the next hour. The features include: the entry age in seconds, the access frequency in the last 5 minutes, the access frequency in the last hour, the number of distinct users who accessed the entry, the time since the last access in seconds, the day of the week, the hour of the day, the entry size in bytes, and the number of times the entry has been promoted from L2 to L1.

The model is trained on the access log from the previous 24 hours. The training uses 500,000 access events with feature extraction. The training runs every hour and completes in 30 minutes on a single GPU (NVIDIA A10). The inference uses a precomputed lookup table that maps each feature vector to a score. The lookup table is rebuilt after each training run.

The eviction policy selects the entry with the lowest predicted access probability. This approach improves the hit ratio by 10 to 15 percent over LRU for workloads with periodic access patterns (e.g., higher traffic during business hours).

### Challenge 8: KV Cache Compression

Design a compression scheme for the KV cache that reduces the memory consumption per entry by 50 percent while maintaining model quality. This challenge tests your understanding of the transformer architecture and numerical optimization.

**Requirements**:
- 50 percent compression ratio compared to FP16 storage
- Quality degradation under 0.1 percent on standard benchmarks (MMLU, HellaSwag)
- Decompression latency under 10 milliseconds
- Compression must be lossy but must not degrade generation quality perceptibly

**Evaluation**:
- Theoretical Soundness: Is the compression scheme grounded in the transformer architecture?
- Empirical Validation: Do you propose a realistic validation methodology?
- Practical Impact: Does the compression make prompt caching feasible for larger models?

**Model Answer**:

Use KV cache quantization with adaptive bit allocation. The key and value tensors are quantized from FP16 (16 bits per value) to INT4 (4 bits per value), reducing the memory per token by 75 percent in the naive case. However, aggressive quantization to INT4 across all heads degrades model quality by 0.5 to 1 percent on benchmarks. The solution is adaptive bit allocation: attention heads that are sensitive to quantization receive higher precision (INT8 or FP16) while robust heads receive lower precision (INT4 or INT2).

The sensitivity is determined by the variance of the attention weights across a calibration dataset of 1,000 prompts. Heads with high variance (unpredictable attention patterns, typically early layers) are more sensitive to quantization and receive higher precision. Heads with low variance (stable attention patterns, typically middle layers) can be aggressively quantized without quality degradation.

The calibration process:
1. Run 1,000 calibration prompts through the model with FP16 precision.
2. For each layer and head, compute the variance of the attention weights.
3. Rank heads by variance. Assign INT8 to the top 25 percent (most sensitive), INT4 to the middle 50 percent, and INT2 to the bottom 25 percent (most robust).
4. Store the quantization scheme per head.

The compression reduces the memory per entry from 10.4 GB to 3.9 GB for a 4,000-token prefix, achieving a 62 percent compression ratio. The decompression latency is under 5 milliseconds because it is a simple dequantization operation (lookup in a small lookup table). The quality degradation is under 0.05 percent on the MMLU benchmark.

The compression scheme is validated by running the full model (FP16) and the compressed model on 5,000 test prompts and comparing the output distributions using KL divergence. The KL divergence must be under 0.001 for the compression to be acceptable. Additionally, 100 human evaluators compare outputs from the compressed and uncompressed models in a blind A/B test. The evaluators must not show a statistically significant preference for either model.

## Additional Challenge: Real-World Optimization

Design a prompt caching system for a global e-commerce customer support platform that operates in 15 countries with 8 languages. The platform processes 2 million support conversations per month. Each conversation averages 4 turns.

**Context**:
- Each country has its own knowledge base with product catalogs, shipping policies, and return policies
- Each language requires translation of the common prefixes
- The knowledge base is updated in real-time when inventory changes
- The system must maintain under 1 second response time for 95 percent of requests
- The cost budget is $0.005 per conversation
- 80 percent of conversations start with one of 20 common topics (order status, return, shipping, payment, account)

**Your design should address**:
1. Cache key structure that accounts for country, language, and topic
2. Cache size estimation across all countries and languages
3. Invalidation strategy for real-time inventory changes
4. Cache warming for the 20 common topics across all countries
5. Cost analysis showing the cache meets the $0.005 per conversation budget

**Model Answer Outline**:

The cache key structure combines country code, language code, and topic identifier: `{country}:{language}:{topic}:{prefix_hash}`. This enables per-country, per-language, and per-topic cache isolation.

The cache size estimation: 15 countries times 8 languages times 20 topics = 2,400 basic cache entries. With a 3,000-token prefix and a 70B model (5.2 MB per token), each entry is approximately 15.6 GB. Total: 2,400 * 15.6 GB = 37.4 TB. With replication factor 2: 74.8 TB across the cache cluster. This requires approximately 1,200 cache nodes with 64 GB RAM each, or fewer nodes with larger RAM configurations.

Invalidation for real-time inventory changes uses event-driven invalidation. When inventory changes, the e-commerce platform publishes an event with the affected country and product category. The cache system subscribes to these events and invalidates cache entries for the affected country and topic. The invalidation is scoped to the specific country and topic, not global, to minimize cache misses.

Cache warming precomputes the 20 common topics for each country and language combination. The warming runs three times per day: before the morning peak, after the midday update, and before the evening peak. Each warming cycle processes 15 * 8 * 20 = 2,400 prefixes, completing within 30 minutes using a batch of 20 GPU instances.

Cost analysis: Without caching, each conversation costs 4 * 8,000 tokens * $3/1M = $0.096. With caching and 75 percent hit ratio, each conversation costs: 1 full prompt + 3 cached prompts = 8,000 * $3/1M + 3 * (4,000 * $3/1M + 4,000 * $0.3/1M) = $0.024 + $0.0396 = $0.0636. The cache reduces cost by 34 percent but does not meet the $0.005 budget. Additional optimizations needed: shorter prefixes, KV cache quantization, or a smaller model for the common prefix processing.
