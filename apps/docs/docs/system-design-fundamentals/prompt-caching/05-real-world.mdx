## Company Case Studies

### Anthropic's Prompt Caching

Anthropic introduced prompt caching for the Claude API in 2024. The feature allows developers to mark a portion of their prompt as a "cache breakpoint." The text before the breakpoint is cached by default and reused across requests. The text after the breakpoint is unique to each request. The breakpoint is specified using a system-level marker in the Messages API.

Anthropic's implementation uses exact prefix caching with configurable breakpoints. The cache is stored on the server side, transparent to the user. The API charges a premium for the initial cache write but offers a 90 percent discount on input tokens for cached prefixes. The pricing structure reflects the cost trade-off: the first request costs more (caching overhead), but subsequent requests cost significantly less.

The key design decisions in Anthropic's implementation include:

- **Explicit breakpoints**: The developer chooses where the cache prefix ends by inserting a cache breakpoint marker. This gives the developer control over what is cached and what is not.
- **Server-side cache**: The cache is managed by the API provider, not the client. The client sends the full prompt every time, and the server detects the repeated prefix.
- **TTL of 5 minutes**: The cache entries expire after 5 minutes of inactivity. Each cache hit extends the TTL by 5 minutes. This prevents cache pollution while maintaining efficiency for active prefixes.
- **Cache size limits**: The maximum cacheable prefix is 4,000 tokens for the Claude Haiku model and 20,000 tokens for Claude Opus. The limits reflect the expected use cases: Haiku is used for simple tasks with short context, Opus for complex reasoning tasks with long context.

Anthropic's documentation provides a clear example. A customer support application sends the knowledge base as the cacheable prefix. The first request costs 4,000 tokens at the full rate plus 2,000 tokens at the full rate. Subsequent requests cost 4,000 tokens at the cached rate (90 percent discount) plus the unique tokens at the full rate. For a system processing 10,000 requests per day with 80 percent cache hits, the monthly savings are approximately 60 percent of the input token cost.

### OpenAI's Cached Prefixes

OpenAI introduced prompt caching for the GPT-4 and GPT-4 Turbo models in late 2024. Unlike Anthropic's explicit breakpoint approach, OpenAI's implementation automatically detects repeated prefixes. When the system detects that the beginning of a prompt matches a prefix that has been processed before, it reuses the cached computation.

The key difference from Anthropic's approach lies in the automation. The developer does not need to insert cache breakpoints. The cache is fully automatic and transparent. This simplicity comes at a cost: the developer has less control over what is cached, how long it persists, and which cache policies apply.

The caching operates at the token level, not the character level. The system tracks the tokenized representation of the prompt prefix and matches against the cached entries. This means that two prompts that differ only in whitespace or formatting (which tokenize identically) will share the cache. This is more flexible than byte-level matching.

The pricing model offers a 50 percent discount on input tokens for cached prefixes. The cache is limited to 1,024 tokens per prefix, which is smaller than Anthropic's limit. OpenAI also implements a session-level cache that persists across requests within the same conversation context.

### GitHub Copilot's Context Caching

GitHub Copilot uses a specialized form of prompt caching tailored to code completion. The prompt includes the current file content (up to 1,024 lines), the surrounding files in the project (up to 10 files), and the recent edit history. Copilot caches the KV cache for the file-level context and only recomputes for the cursor position.

The implementation is a form of segment-level caching. The prompt is divided into three segments: the file header (imports, type definitions, constant declarations), the surrounding code context (function bodies, class definitions), and the cursor position. The file header is cached across all completions within the same file, which is typically 200 to 500 completions per file. The surrounding code context is cached within a session (a few minutes of editing). Only the cursor position is recomputed for each completion.

The caching strategy reduces the prefill time from 500 milliseconds to 50 milliseconds for the average completion. The hit ratio is approximately 85 percent for the file header segment and 70 percent for the context segment. The savings are critical for Copilot's user experience, where completions must appear within 100 milliseconds to feel instantaneous.

### Google's Prefix Caching in Gemini API

Google's Gemini API implements prefix caching as a core feature of the model serving infrastructure. The implementation uses a distributed cache cluster that is shared across all Gemini API requests. The cache uses token-level prefix matching instead of character-level matching. This means the cache operates at the granularity of the model's vocabulary rather than the raw text.

The token-level matching is more efficient because it aligns with the model's internal representation. Two prompts that differ by whitespace in the raw text may be identical at the token level because the tokenizer ignores the whitespace difference. This reduces the number of distinct prefixes and increases the cache hit ratio.

Google reports a 60 to 75 percent cache hit ratio for production workloads. The cache is especially effective for system prompts that are shared across millions of requests. Google also uses the cache for safety checks: the safety classifier prompt is cached and reused for every request.

## Open Source Implementations

### vLLM's Automatic Prefix Caching

vLLM, the popular open-source LLM inference engine, implements automatic prefix caching. The implementation is built into the inference engine and requires no configuration from the user. vLLM maintains a hash-based cache of KV cache entries in GPU memory. When a new request arrives, vLLM computes the hash of the prompt prefix and checks it against the cache.

The vLLM implementation uses GPU memory for the cache, not CPU memory or SSD. This eliminates the network transfer overhead but limits the cache size to the available GPU memory. For a 24 GB GPU running a 7-billion-parameter model, approximately 8 GB is available for the KV cache. This can store approximately 3,000 tokens worth of KV cache, or about 750 tokens for a larger model.

The vLLM implementation supports multiple eviction policies, including LRU and LFU. The default policy is LRU. The cache is automatically cleared when the model is loaded or when the GPU memory is needed for other purposes.

### TGI's Prefix Caching

Hugging Face's Text Generation Inference (TGI) implements prefix caching with a focus on integration with the Hugging Face ecosystem. The TGI implementation uses a combination of exact prefix caching and optional semantic caching.

The exact prefix caching in TGI uses a hash-based approach similar to vLLM. The semantic caching uses sentence embeddings from the `sentence-transformers` library to find similar prompts. The semantic caching is disabled by default and must be explicitly enabled.

TGI's implementation also supports caching across multiple GPUs. The cache is sharded across GPUs using a consistent hashing scheme. A GPU that does not have a local cache hit can forward the request to the GPU that has the cached entry. This cross-GPU cache sharing increases the effective cache size by 4x to 8x for an 8-GPU deployment.

### SGLang's RadixAttention

SGLang, a framework for efficient LLM serving, implements a technique called RadixAttention. RadixAttention uses a radix tree (a compressed trie) to store KV cache entries. The radix tree is more memory efficient than a standard trie because it compresses common prefixes into single nodes. Instead of storing each token as a separate node, the radix tree stores runs of tokens as single nodes.

The RadixAttention implementation supports prefix-level, segment-level, and semantic caching within a single unified framework. The framework automatically determines the caching strategy based on the prompt structure.

SGLang reports a 70 to 90 percent cache hit ratio for production workloads. The radix tree approach reduces the memory overhead of the cache index by 40 percent compared to a hash-based approach, because common prefixes are stored once rather than duplicated.

## Cloud Provider Implementations

### AWS

AWS offers prompt caching through Amazon Bedrock. Bedrock's prompt caching is built into the inference endpoint and is managed by AWS. The user cannot directly configure the cache but can influence caching behavior through prompt design. Placing shared context at the beginning of the prompt increases the likelihood of a cache hit.

AWS also offers dedicated inference endpoints with dedicated cache instances. The dedicated cache is isolated from other users and provides higher hit ratios for high-volume workloads.

### Google Cloud

Google Cloud Vertex AI implements prompt caching for the Gemini models. Vertex AI provides additional configuration options compared to the standard Gemini API, including cache TTL, cache size, and cache warming policies. Vertex AI also integrates the cache with the Agent Builder, which automatically optimizes agent prompts for caching.

### Azure

Azure OpenAI Service implements prompt caching for GPT-4 and GPT-4 Turbo models. Azure provides a cache metrics dashboard that shows hit ratio, miss ratio, and cache size by deployment. The dashboard is integrated with Azure Monitor for alerting and autoscaling.

## Lessons Learned

### Lesson 1: Measure Before Optimizing

Many teams implement prompt caching without measuring the prompt diversity first. They assume a high hit ratio and are disappointed when the actual ratio is 20 percent. The first step in any caching project is to log and analyze the prompt prefixes for a week. The distribution of prefixes determines the potential hit ratio and the required cache size.

### Lesson 2: Design for Monitoring Visibility

A prompt caching system without visibility is a production incident waiting to happen. Every cache operation must be logged and monitored. The hit ratio, miss ratio, eviction rate, and lookup latency should be visible on a dashboard. Alerts should fire when the metrics deviate from the expected range.

### Lesson 3: Plan for Cold Starts

Every deployment, cache resize, and node failure creates a cold start period. During cold starts, the hit ratio drops to zero and climbs back up over time. The system must be designed to handle the increased inference load during cold starts.

### Lesson 4: Cache-Aware Prompt Engineering

The prompt engineering team must be involved in the caching strategy. Prompts that place shared context at the beginning, use consistent formatting, and minimize token-by-token variation achieve higher hit ratios.

### Lesson 5: Test the Failure Modes

Teams often test the happy path (cache hit, low latency, low cost) but neglect the failure modes. The cache node crashes mid-request. The cache index is corrupted. The TTL clock is wrong. Every failure mode must be tested with chaos engineering. The chaos engineering suite should include:

- **Node failure tests**: Randomly kill cache nodes during production traffic. Verify that the system falls back to full inference gracefully.
- **Network partition tests**: Isolate a cache node from the cluster. Verify that the remaining nodes continue to serve requests.
- **Latency injection tests**: Add artificial latency to cache lookups. Verify that the system detects slow lookups and falls back.
- **Data corruption tests**: Corrupt cache entries in storage. Verify that the system detects the corruption and treats the entry as a miss.

The chaos tests must run in staging first, then in production during low traffic. Each test must have a rollback plan and a clear success criterion.

### Lesson 6: Cache Invalidation is the Hardest Problem

Consistency between the cached content and the source of truth is the hardest challenge. Invalidation events are easy to miss. The knowledge base update may happen through a pipeline the cache system does not monitor. Teams must design the invalidation path with the same rigor as the caching path.

**Real-World Invalidation Patterns**:

- **Polling-based invalidation**: The cache system polls the knowledge base for new versions every 5 minutes. If a new version is detected, the cache invalidates all entries with the old version. Simple but not real-time.
- **Event-driven invalidation**: The knowledge base publishes an event on a message queue when it is updated. The cache system subscribes to the queue and invalidates entries when events arrive. Real-time but requires event infrastructure.
- **Hybrid invalidation**: Polling for safety (catch missed events) plus event-driven for timeliness. This is the most robust pattern.

**Common Invalidation Mistakes**:

- Invalidating all entries on any update, even when the update affects only a small subset.
- Not invalidating at all, relying solely on TTL expiry.
- Invalidating too aggressively, causing cache stampedes.
- Not coordinating invalidation across regions, causing inconsistency.

## Emerging Trends

### Model-Native Caching

Future LLM architectures may integrate caching as a first-class feature rather than an add-on. Model-native caching would expose the cache control to the model itself, allowing the model to decide what to cache and for how long. This would eliminate the need for cache breakpoints and manual prefix extraction.

### Cross-Model Cache Sharing

Different models (GPT-4, Claude, Gemini) use different KV cache formats. Cross-model cache sharing would require a common KV cache format. Industry standards for KV cache interchange are emerging, led by the Open Inference Protocol working group.

### Learned Cache Policies

Cache policies are currently hand-tuned (LRU, LFU, fixed TTL). Future systems will use reinforcement learning to learn optimal caching policies from traffic patterns. The learned policy adapts to the workload automatically without manual tuning.

### KV Cache Deduplication

Multiple users may share the same prompt prefix. KV cache deduplication identifies identical prefixes across users and stores the KV cache once. This reduces the cache storage requirement by 30 to 50 percent for workloads with common system prompts.

### Federated Caching

In a federated deployment with multiple LLM providers, the cache is shared across providers. A request to GPT-4 can reuse a KV cache that was computed by Claude, if the models share the same tokenizer and attention mechanism. This is an active research area with significant standardization challenges.

## Community Standards and Best Practices

### Cache Breakpoint Markers

The community is converging on a standard format for cache breakpoint markers. The marker is placed at the point in the prompt where caching should end. The most common formats are:

- Anthropic: `<cache_breakpoint />` in the Messages API
- OpenAI: Automatic detection (no marker needed)
- Open Inference Protocol: `[CACHE_BREAKPOINT]` in the prompt text

### Cache Headers

Standard HTTP headers for prompt caching are emerging:
- `X-Cache-Status`: `HIT`, `MISS`, `STALE`, `BYPASS`
- `X-Cache-TTL`: The remaining TTL in seconds
- `X-Cache-Prefix-Length`: The number of cached tokens
- `X-Cache-Node`: The cache node that served the request

### Cache Metrics Standard

The community recommends standard metric names for cache monitoring:
- `prompt_cache_hits_total`: Total number of cache hits
- `prompt_cache_misses_total`: Total number of cache misses
- `prompt_cache_hit_ratio`: Hit ratio over a sliding window
- `prompt_cache_evictions_total`: Total number of evictions
- `prompt_cache_lookup_duration_ms`: Cache lookup latency in milliseconds
- `prompt_cache_memory_bytes`: Cache memory usage in bytes
- `prompt_cache_entries`: Number of entries in the cache

### Vendor Lock-In Risks

Prompt caching creates vendor lock-in because the KV cache format is specific to each model provider. Migrating from one provider to another requires rebuilding the cache from scratch, which causes a cold start. Mitigation strategies include:

- Using a model-agnostic cache format (e.g., storing the prompt text and re-computing the KV cache on migration)
- Maintaining a warm cache at both providers during migration
- Using a caching layer that abstracts the provider-specific KV cache format

## Interview Questions from Real-World Experience

### Question: How did Cache Warming Impact Your Metrics?

In a production deployment at a customer support company, we observed that cache warming reduced the cold start time from 30 minutes to 5 minutes. Before warming, a deployment at 2:00 PM caused the hit ratio to drop from 75 percent to 0 percent, taking 30 minutes to recover as the cache was repopulated by user traffic. After implementing warming, the hit ratio dropped to 30 percent (the warming covered the top 50 percent of prefixes) and recovered to 70 percent within 5 minutes. The key insight was that warming the top 20 percent of prefixes covered 60 percent of the traffic. Warming the top 50 percent covered 80 percent of the traffic. Beyond that, the marginal benefit of warming additional prefixes diminished rapidly.

### Question: How Did You Handle Multi-Tenant Isolation?

We used tenant-prefixed cache keys with per-tenant Bloom filters. Each tenant had their own Bloom filter so that tenant A's prefixes never pollute tenant B's cache lookups. The cache index was partitioned by tenant ID with configurable per-tenant quotas. A tenant with a high-traffic plan received more cache capacity than a tenant with a low-traffic plan. The isolation was enforced at the index level: a lookup from tenant A could only return entries tagged with tenant A. This prevented cross-tenant data leakage and ensured compliance with data residency requirements.

### Question: What Was the Most Surprising Failure You Encountered?

The most surprising failure was a gradual hit ratio degradation caused by a prompt template change. A developer added a timestamp to the system prompt: "The current date is 2026-01-15." This single line of change caused every prefix to include the date, making each day's cache entries invalid the next day. The hit ratio dropped from 70 percent to 15 percent over 24 hours because every prefix had a different date. The fix was to remove the timestamp from the prompt and instead compute dates dynamically in the application logic. The lesson: any dynamic content in the prefix, even a date, breaks cache reuse.

### Question: How Did You Estimate the Cache Size for Your Deployment?

We used the formula: cache_size = num_prefixes * tokens_per_prefix * bytes_per_token * replication_factor. We logged prompt prefixes for 7 days and counted the number of distinct prefixes. We found that 200 prefixes covered 80 percent of the traffic. With a 4,000-token prefix, a 70B model (5.2 MB per token), and replication factor 2: 200 * 4,000 * 5.2 MB * 2 = 8.32 TB. We provisioned 10 TB of cache storage to allow headroom. After deployment, the actual peak usage was 7.5 TB, confirming the estimate.

### Question: What Was the Network Bandwidth Challenge?

The network bandwidth was the bottleneck we underestimated. Each cache hit transfers 10.4 GB of KV cache tensors. At 100 hits per second, the bandwidth is 1,040 Gbps. Our initial deployment used 40 Gbps network links, which were saturated at 4 cache hits per second. We upgraded to 100 Gbps links and implemented locality-aware scheduling to keep cache hits on the same host. The scheduling improvement doubled the effective throughput without additional network investment.
