## The Problem Story

A mid-sized SaaS company, SupportFlow, built an AI-powered customer support assistant. Every incoming support ticket triggers a call to a large language model. The prompt includes the full knowledge base of 47 product articles, the company's refund and escalation policies, the customer's conversation history with 12 previous messages, and the current ticket context. The system processes 5,000 tickets per day, and each prompt is roughly 12,000 tokens. The LLM provider charges $3 per million input tokens. The math is brutal: 5,000 tickets times 12,000 tokens times 30 days equals 1.8 billion input tokens per month, costing $5,400 per month in input tokens alone. But the real killer is latency. The model spends 80 percent of its inference time processing the repeated knowledge base and policy text, and only 20 percent on the unique ticket content. Users wait 8 to 12 seconds for a response. The support team is drowning.

The engineer assigned to fix this, Priya, takes a closer look at the prompts. She notices that the knowledge base articles are identical across all tickets for a given product. The company policies are the same every single time: the refund policy, the escalation matrix, the SLA definitions, and the product troubleshooting guides. The only variable parts are the customer's name, the conversation history, and the current ticket description. She estimates that 80 percent of every prompt is repeated verbatim across thousands of requests. The company is paying to have the same 9,600 tokens processed from scratch, 150,000 times per month. That is 1.44 billion tokens of wasted computation.

Priya explores alternatives. She could reduce the prompt size by summarizing the knowledge base, but that risks losing critical details that the support agents rely on. She could use a smaller model for the repetitive parts, but that adds architectural complexity and risks inconsistency between two models. She could precompute something on the server side, but the LLM API does not expose internal state. The breakthrough comes when she discovers that recent LLM providers offer a feature called prompt caching. The idea is elegant: if the LLM has already processed the prefix of a prompt, it can reuse the internal representation and skip the most expensive computation. Priya restructures the application to send the knowledge base and policies as a cached prefix, followed by the unique ticket context. The first request of the day takes 10 seconds, but every subsequent request that shares the same prefix takes under 2 seconds. The input token bill drops by 60 percent. The support team is ecstatic.

## The Insight

Prompt caching is a technique that stores the intermediate representation of a prompt prefix so that the LLM can skip the most expensive part of inference when it sees the same prefix again. To understand why this works, you need to know how transformers process text. When an LLM reads a prompt, it does not read it like a human. It converts each token into a vector embedding, then runs those embeddings through a stack of attention layers. In each attention layer, every token computes a weighted sum of every other token. This is the self-attention mechanism, and it is the computational bottleneck of inference. For a prompt of N tokens, each attention layer does O(N^2) work. Doubling the prompt length quadruples the computation. The intermediate results of these attention computations are stored in the key-value cache, or KV cache. The KV cache is what the model reuses when generating new tokens. It represents the processed state of every token seen so far.

The restaurant analogy is helpful here. Imagine a chef running a busy kitchen. Every order includes a base of chopped vegetables, a stock, and a sauce. The first order of the night requires the chef to chop every vegetable, prepare the stock from scratch, and reduce the sauce for 30 minutes. That takes 45 minutes. But once the base is prepared, every subsequent order that uses the same base only needs the final assembly: searing the protein, plating, and adding the garnish. Each subsequent order takes 5 minutes. The pre-chopped vegetables and prepared stock are the KV cache. The first order pays the full cost. The next 50 orders pay only the incremental cost. The kitchen throughput increases by 9x for the same labor cost.

In LLM terms, the first request with a given prompt prefix pays the full prefill cost of processing every token through every attention layer. Subsequent requests that share the same prefix skip the prefill for the shared tokens and only process the new tokens. The cost savings are proportional to the fraction of the prompt that is shared across requests. If 80 percent of the prompt is a shared prefix, the LLM saves 80 percent of the prefill computation. For a 12,000-token prompt where 9,600 tokens are shared, the savings are enormous. The key insight is that prompt caching is not about storing the final answer. It is about storing the intermediate state so the model can pick up where it left off. This is fundamentally different from traditional web caches, which store the final response. A prompt cache stores the computational state, not the output.

To understand why this matters at scale, consider a large-scale customer support system handling 100,000 conversations per day. Each conversation averages 3 turns, so 300,000 prompts per day. If each prompt has a 9,000-token shared prefix and the full prompt is 12,000 tokens, the total prefill computation without caching is 300,000 times 12,000 tokens equals 3.6 billion tokens per day. With caching and an 80 percent hit ratio: 240,000 cache hits (skipping 9,000 tokens each, saving 2.16 billion tokens of prefill) plus 60,000 full prefills (720 million tokens) plus 60,000 first-time prefills that populate the cache (540 million tokens). The total is 1.26 billion tokens, a 65 percent reduction in prefill computation. The latency per request drops from 12 seconds to 2 seconds for cache hits. The cost drops from $10,800 per month to $3,780 per month. These numbers are why every major LLM provider has invested in prompt caching.

## Core Vocabulary

**KV Cache** -- The key-value cache stores the intermediate attention representations (key and value tensors) for each token in the prompt. Every transformer layer maintains its own KV cache. The cache size grows linearly with the number of tokens and layers. For a 70-billion-parameter model with 80 layers and a 4,096-dimensional hidden state, each token consumes roughly 2.6 MB of KV cache across all layers. A 4,000-token prompt consumes about 10 GB of KV cache. The KV cache is the largest consumer of GPU memory during inference.

**Cache Hit** -- A request matches a cached prompt prefix. The LLM skips the prefill computation for the cached tokens and only processes the new tokens. Cache hits reduce latency by 50 to 90 percent depending on the prefix length and model size. The hit ratio is the most important operational metric for a prompt caching system. A cache hit means the model can start generating the first token almost immediately, because it picks up from the cached intermediate state rather than recomputing everything.

**Cache Miss** -- A request does not match any cached prefix. The LLM must process the full prompt through the prefill phase. The cache is populated with the new prefix after the request completes, so subsequent requests with the same prefix benefit. The first request in a sequence always pays the full cost. Cache misses are also called cold starts in the context of a specific prefix.

**Cache Prefix** -- The initial segment of the prompt that is eligible for caching. Providers typically cache only the prompt prefix, not arbitrary positions in the middle. The prefix must be identical across requests for exact prefix caching to work. The prefix length is configurable, typically between 128 and 4,096 tokens. The prefix is the most critical design parameter because it determines both the cache hit ratio and the memory consumption.

**Context Window** -- The maximum number of tokens the LLM can process in a single request. Prompt caching is most effective when the cached prefix consumes a large fraction of the context window. For a 128K context window, a 4,000-token shared prefix reclaims 3 percent of the useful context. For a 32K context window, the same prefix reclaims 12.5 percent. The context window size influences the cache design because larger context windows allow longer prefixes.

**Time-To-Live (TTL)** -- A duration assigned to each cached prefix entry. After the TTL expires, the entry is automatically removed or marked stale. TTL is the primary mechanism for bounding staleness. Typical TTLs range from 5 minutes to 24 hours. Short TTLs reduce stale cache risks but increase the miss rate. Long TTLs maximize the hit ratio but risk serving stale content.

**Cache Invalidation** -- The process of removing or updating cached prefixes when the underlying prompt structure changes. If the knowledge base is updated, all cached prefixes that include the old knowledge base must be invalidated. Invalidation can be explicit (the system sends an invalidation signal) or implicit (TTL expiry). Cache invalidation is famously one of the two hard problems in computer science.

**Cache Warming** -- The practice of precomputing cache entries for anticipated prompts before the traffic arrives. Cache warming is critical for production deployments. Without warming, the first wave of requests after a deployment all experience cache misses. Warming can be triggered by a deployment pipeline, a scheduled job, or a prediction model. The warming process runs the prefill phase for each targeted prefix and stores the resulting KV cache.

**Prefix Matching** -- The algorithm that determines whether an incoming prompt matches a cached prefix. The simplest approach is exact string matching: the prompt must start with the same byte sequence as the cached prefix. More sophisticated approaches use token-level matching or semantic similarity using embeddings. The matching algorithm is the core performance bottleneck of the cache lookup because every request requires a matching operation.

**Semantic Caching** -- An advanced variant that caches and retrieves prefixes based on semantic similarity rather than exact string match. Semantic caching uses embeddings to find similar prompts. It increases the cache hit rate by 10 to 30 percent at the cost of higher lookup latency (5 to 50 milliseconds) and the risk of retrieving semantically different but embedding-similar content.

**Cache Eviction Policy** -- The algorithm that decides which cache entries to remove when the cache is full. Common policies include Least Recently Used (LRU), Least Frequently Used (LFU), and Size-Aware LRU (which considers the memory footprint of each entry). The eviction policy directly impacts the cache hit ratio. A poorly chosen policy can reduce the hit ratio by 20 percent or more.

**Cache Sharding** -- The practice of distributing cache entries across multiple cache nodes. Sharding enables horizontal scaling of the cache but introduces complexity for prefix matching, which may need to query multiple shards or use a consistent hashing scheme. Sharding also affects fault tolerance: if one shard fails, only the entries on that shard are lost.

**Cache-Aware Prompting** -- The practice of designing prompts to maximize cache reuse. Strategies include placing the shared context at the beginning of the prompt, using consistent formatting and ordering of sections, standardizing system instructions across all prompts, and batching similar requests together. Cache-aware prompting is a prompt engineering technique that multiplies the benefits of prompt caching. Companies that adopt cache-aware prompting see 20 to 40 percent higher hit ratios than those that do not.

## How It Actually Works -- Step by Step

The following steps trace the lifecycle of both a cached and an uncached request through the prompt caching system.

### Step 1: A Request Arrives at the API Gateway

A client sends a prompt to the LLM API. The prompt is a sequence of tokens that has been tokenized by the model's tokenizer. The first 4,000 tokens are the shared knowledge base articles and company policies. The remaining 2,000 tokens are the specific customer question and conversation history. The total prompt is 6,000 tokens. The API gateway validates the request format, authenticates the client, and extracts the prompt text. It then forwards the prompt to the cache router for cache checking.

### Step 2: The Cache System Extracts and Hashes the Prefix

The cache router extracts the prefix from the prompt. The prefix length is determined by the cache configuration. In this case, the system is configured to use a 4,096-token prefix, meaning the first 4,096 tokens of the prompt are checked against the cache. The router takes these tokens and computes a cryptographic hash (SHA-256) of the token IDs. This hash serves as the key for the cache index lookup. The hash computation takes approximately 10 microseconds and is negligible compared to the potential savings of a cache hit.

### Step 3: The Bloom Filter Provides Fast Miss Detection

Before querying the cache index, the system checks the Bloom filter. The Bloom filter is a probabilistic data structure that can definitively say "this prefix is not in the cache" but can only say "this prefix might be in the cache." If the Bloom filter says the prefix is not in the cache, the system skips directly to full inference without querying the index. This check is extremely fast: approximately 1 microsecond. In production systems, the Bloom filter eliminates 50 to 80 percent of cache miss index queries, reducing the load on the cache index.

### Step 4: Cache Hit Path -- The Prefix is Found in the Cache

The prefix hash matches an entry in the cache index. The cache entry contains the KV cache tensors for the first 4,000 tokens, computed during a previous request. The system retrieves the entry metadata from the index to determine the storage location. It then fetches the KV cache tensors from the storage tier. The storage tier is determined by the entry's access pattern: hot entries are in L1 (GPU or RAM) with sub-millisecond access, warm entries are in L2 (SSD) with 1 to 5 millisecond access.

### Step 5: Cache Miss Path -- The Prefix is Not in the Cache

The prefix hash does not match any entry in the cache index. This could be because the prefix is genuinely new, the previous entry was evicted due to TTL expiry, the cache is full and the entry was evicted, or the prefix differs by even a single token from the cached version. The system must now perform full inference. It sends the entire 6,000-token prompt to the LLM inference engine. The inference engine runs the full prefill phase, processing all tokens through all 80 transformer layers.

### Step 6: The Decode Phase Generates the Response

After the prefill phase completes (or after loading the cached KV), the model enters the decode phase. In the decode phase, the model generates one token at a time. Each generated token attends to all previous tokens through the self-attention mechanism. The KV cache grows with each generated token because each new token's key and value tensors are appended to the cache. The decode phase is memory-bandwidth-bound because the model must read the entire KV cache from GPU memory for each token.

The decode phase generates tokens until one of three conditions is met: the model generates an end-of-sequence token, the maximum response length is reached, or a stop sequence is detected. Each decode step takes approximately 50 milliseconds for a 70-billion-parameter model.

### Step 7: TTL Management and Extension

Each cache entry has a TTL that determines how long the entry remains valid. The default TTL is 1 hour. The TTL is set when the entry is first created. On subsequent cache hits, the TTL is extended. The extension policy is configurable: the most common policy extends the TTL by the full TTL duration on each hit. So if the TTL is 1 hour and the entry is accessed at minute 55, the TTL is extended by another hour. This ensures that frequently accessed prefixes remain in the cache indefinitely.

A background TTL manager thread scans the cache index every 60 seconds. It identifies entries that have exceeded their TTL and marks them as stale. Stale entries are not served on cache lookups but are not immediately removed from storage. They are reclaimed by the eviction controller when the storage is needed.

### Step 8: Eviction When the Cache is Full

The cache has a finite capacity. When the cache reaches its capacity limit, the eviction controller selects entries to remove. The selection is based on the eviction policy. LRU eviction removes the entry with the oldest access timestamp. LFU eviction removes the entry with the lowest access frequency. The eviction controller runs after each cache write that would exceed the capacity limit.

The eviction process removes the KV cache tensors from the storage tier and removes the metadata from the cache index. The Bloom filter is not updated on eviction. This means the Bloom filter may produce false positives for evicted entries, but this is acceptable because the false positive is resolved by the cache index lookup.

### Step 9: Cache Warming for Deployment

Before a deployment or a traffic spike, the system precomputes cache entries for the most common prompt prefixes. Cache warming ensures that the first requests after the deployment experience cache hits instead of misses. The warming process runs the prefill phase for each targeted prefix and stores the resulting KV cache. The warming typically runs 5 to 30 minutes before the deployment to ensure the cache is fully populated when the new traffic arrives.

The warming pipeline uses the previous day's traffic log to identify the top 100 most common prefixes. It runs the prefill for each prefix in parallel, using a batch of inference servers. The batch processing completes within 10 minutes for a typical workload.

### Step 10: Monitoring and Alerting

The cache system continuously tracks hit ratio, miss ratio, eviction rate, memory usage, and average lookup latency. These metrics are exposed through a monitoring dashboard and streamed to an alerting system. An alert fires if the hit ratio drops below a configurable threshold, indicating a problem with the caching strategy. The monitoring also tracks cache node health, replication lag, and network bandwidth utilization.

## Variants and Flavors

### Exact Prefix Caching

Exact prefix caching is the simplest and most widely supported variant. The cache stores the KV cache for a specific prefix string. An incoming prompt matches the cache only if the beginning of the prompt is byte-for-byte identical to the stored prefix. This variant is fast, deterministic, and easy to implement. The hash-based matching takes O(1) time after the hash computation. The trade-off is that even a single character difference breaks the cache match. A prompt with "User: hello" does not match a cached prefix for "User: hello " (with a trailing space). Exact prefix caching is the default for most providers including Anthropic, OpenAI, and vLLM.

### Semantic Caching

Semantic caching uses embeddings to find prompts that are semantically similar, even if the exact text differs. The system computes an embedding vector for each prompt prefix and stores it alongside the KV cache. On a new request, the system computes the embedding of the incoming prefix and searches for the nearest neighbor in the embedding space. If the distance is below a configurable threshold, the cached KV cache is reused. Semantic caching increases the hit rate by 10 to 30 percent but adds the latency of embedding computation (5 to 50 milliseconds) and nearest-neighbor search (1 to 10 milliseconds with an approximate nearest neighbor index).

The embedding model must be chosen carefully. A sentence transformer model with 384-dimensional embeddings provides a good balance of speed and accuracy. The embedding can be computed on a CPU because it is a lightweight model. The nearest-neighbor search uses a vector database such as FAISS or Annoy.

### Segment-Level Caching

Segment-level caching divides the prompt into reusable segments. Each segment has its own KV cache entry that can be independently cached, evicted, or invalidated. A prompt is assembled from multiple segments, some of which may be cached and some not. This variant is more flexible than prefix caching because it allows cache reuse for prompts that share a middle segment rather than only a prefix. For example, two prompts may share the same knowledge base segment but have different system instructions. With segment-level caching, the knowledge base segment is cached once and reused across both prompts.

The trade-off is increased complexity in segment identification (how to divide the prompt into segments), cache lookup (multiple segments mean multiple lookups), and KV cache assembly (the KV caches from multiple segments must be concatenated in the correct order). The segment boundaries are typically determined by prompt template markers such as `"<system>"`, `"<knowledge_base>"`, and `"<history>"`.

### Hybrid Caching

Hybrid caching combines multiple strategies within a single system. It uses exact prefix caching for the most common prompts where exact matches are expected, semantic caching for free-form natural language prompts where similar-but-not-identical matches are acceptable, and segment-level caching for complex prompt assemblies where different parts have different caching characteristics. The cache system routes each request to the appropriate strategy based on the prompt characteristics and the cache state.

The routing logic uses a classifier that examines the prompt structure. If the prompt starts with a known system prompt marker (exact prefix expected), route to the exact cache. If the prompt is a free-form user query (semantic match expected), route to the semantic cache. If the prompt has multiple segments defined by markers, route to the segment cache.

## Mental Model Check

### Question 1

A team implements prompt caching and sees a 60 percent cache hit ratio. They reorganize their prompts to put the shared context at the beginning and increase the prefix length from 1,024 to 4,096 tokens. The hit ratio increases to 85 percent. Why did the hit ratio improve?

**Answer:** The longer prefix captures more of the shared context. With a 1,024-token prefix, only the first quarter of the knowledge base was cached. Any request that used a different section of the knowledge base after the first 1,024 tokens caused a cache miss because the model had to recompute the entire prefix. With a 4,096-token prefix, the entire knowledge base fits in the cached prefix. The hit ratio is the fraction of the prompt that is shared across requests. Longer prefixes capture more shared content, increasing the hit ratio. This illustrates the importance of cache-aware prompt engineering.

### Question 2

An engineer notices that the cache hit ratio is 90 percent in the first hour after a deployment but drops to 40 percent after 24 hours. What is the most likely cause?

**Answer:** The TTL on the cache entries is too short. Entries expire after a few hours, causing subsequent requests to miss the cache. The engineer should increase the TTL for entries that are accessed frequently and verify that the cache size is sufficient to hold the working set of prefixes. Alternatively, the number of distinct prompts might be growing over time as users submit increasingly diverse queries, and the cache is too small to hold all the distinct prefixes.

### Question 3

A system uses semantic caching with a cosine similarity threshold of 0.95. The hit ratio is 15 percent. The team lowers the threshold to 0.75, and the hit ratio jumps to 55 percent. However, some responses become incorrect because semantically different prompts are matched to the wrong cached prefix. What is the trade-off, and how should the team choose the threshold?

**Answer:** The threshold controls the precision-recall trade-off. A high threshold (0.95) is strict and produces few false positives (high precision) but also few cache hits (low recall). A low threshold (0.75) produces many cache hits (high recall) but also false positives where the cached KV cache is not appropriate for the actual prompt (low precision). The team should measure the response quality degradation as a function of the threshold and choose the threshold that maximizes the hit ratio while keeping the error rate below an acceptable bound.

### Question 4

A company uses prompt caching for a customer support chatbot. The knowledge base is updated daily. The cache TTL is set to 24 hours. After a knowledge base update, the chatbot continues to use the old knowledge base for 24 hours because the cached prefixes are still valid. How should the system handle this?

**Answer:** The system should implement explicit cache invalidation triggered by the knowledge base update. When the knowledge base changes, the system sends an invalidation signal that removes all cache entries containing the old knowledge base prefix. The invalidation signal can be sent via an API call, a message queue event, or a database trigger. After invalidation, the next request with the new knowledge base pays the full prefill cost and populates the cache with the new prefix.

### Question 5

A startup implements prompt caching on a single server with 64 GB of RAM. Each KV cache entry for a 4,000-token prefix consumes approximately 10.4 GB. The server can hold approximately 6 entries. The startup serves 500 distinct prompt prefixes per day. The eviction rate is 95 percent, and the hit ratio is 5 percent. What should the startup do?

**Answer:** The cache is far too small for the working set. The startup has three options. First, increase the cache size by using a larger server with more RAM or a distributed cache cluster. Second, reduce the per-entry memory consumption by using a smaller prefix length, quantizing the KV cache (FP16 to INT8), or using a smaller model. Third, reduce the number of distinct prefixes by standardizing the prompt structure, consolidating similar prompts, and applying cache-aware prompting strategies.

## Prompt Caching vs Traditional Caching

Prompt caching is fundamentally different from traditional web caching. Understanding the differences helps build the right mental model.

| Dimension | Traditional Caching | Prompt Caching |
|---|---|---|
| **Cached Data** | Final response (HTML, JSON, images) | Intermediate computational state (KV tensors) |
| **Cache Unit** | Complete response object | Prefix of the input (tokens) |
| **Cache Key** | URL, query parameters | Token sequence hash |
| **Storage Size** | KB to MB per entry | GB to tens of GB per entry |
| **Storage Medium** | RAM, SSD, or both | GPU memory, RAM, or SSD |
| **Latency Impact** | Eliminates server round trip | Eliminates prefill phase only |
| **Freshness** | Known via HTTP headers (Cache-Control) | Requires custom TTL and invalidation |
| **Complexity** | Mature, well-understood | Newer, evolving rapidly |

## Common Misconceptions

**Misconception 1: Prompt caching stores the LLM's response.** This is incorrect. The cache stores the KV cache tensors from the prefill phase. The response is generated after the cache is loaded.

**Misconception 2: Prompt caching works for any prompt.** Prompt caching requires repeated prefixes across requests. If every prompt is unique, caching provides no benefit.

**Misconception 3: Longer prefixes are always better.** Longer prefixes capture more shared context but reduce the hit ratio because they are more specific. The optimal prefix length balances context capture against reuse frequency.

**Misconception 4: Semantic caching is strictly better than exact prefix caching.** Semantic caching has higher lookup latency, risks false positives, and is not suitable for structured prompts.

**Misconception 5: Cache size should be as large as possible.** Larger caches increase lookup latency due to memory access patterns and increase operational cost. The cache should be sized for the working set, not for all possible prefixes.
