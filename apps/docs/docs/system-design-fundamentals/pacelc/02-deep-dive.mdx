# PACELC — Under the Hood

The internal architecture that implements the PACELC trade-off spans every layer of a distributed database: partition detection, replication management, latency monitoring, and consistency control.

---

### Partition Detector

The partition detector runs continuously on every node. It uses a gossip protocol where each node periodically exchanges status information with a random set of peers (typically 3 per round, every 1 second). When a node fails to hear from a peer within a configurable timeout (default 3 to 10 seconds), it marks that peer as suspect. If multiple nodes agree the peer is unreachable, the cluster declares a partition.

The detection delay is critical. Too fast and transient network jitter causes false positives, forcing unnecessary consistency-mode switches. Too slow and the system remains in Else-mode while a partition exists, allowing stale reads to proliferate. Production systems tune this to 5 to 15 seconds.

### Replication Manager

The replication manager determines how many replicas must acknowledge a write before the coordinator responds to the client. In PA/EL mode, the replication factor is effectively 1. In PC/EC mode, it uses strict quorum: `write_consistency = floor(N/2) + 1` where N is the replication factor.

### Latency Monitor

The latency monitor tracks per-node and cross-datacenter round-trip times. In PA/EL systems, the monitor routes reads to the lowest-latency replica. In PC/EC systems, the monitor still tracks latency but the consistency controller overrides the fast-path decision: the system waits for quorum regardless of whether a fast path is available.

### Consistency Controller

The consistency controller applies the system's PACELC policy. It selects between synchronous and asynchronous replication paths based on the current regime (partition vs normal) and the per-operation consistency level.

## Data Structures and Algorithms

### Quorum Calculations

Quorum is the minimum number of replicas that must participate for a distributed read or write to be considered successful.

The standard formula for a system with N replicas and W (write quorum) and R (read quorum):

- Strong consistency requires: W + R &gt; N
- Write-all/read-one: W = N, R = 1 (maximum consistency, worst write latency)
- Write-one/read-all: W = 1, R = N (minimum write latency, weak read consistency)
- Quorum: W = R = floor(N/2) + 1 (balance)

For N=5 with quorum: W=3, R=3, W+R=6 &gt; 5. Any read quorum of 3 nodes will overlap with any write quorum of 3 nodes by at least 1 node, guaranteeing at least one node has the latest write.

### Hinted Handoff Channels

When a write targets a replica that is temporarily unreachable, a live node stores the write in a hinted handoff buffer. Each entry contains the key, value, timestamp, and the intended replica's identifier. The buffer is stored on disk to survive crashes.

Structure:

```
HintedHandoffEntry {
  key: String (partition key + clustering columns)
  value: Bytes
  timestamp: HybridLogicalClock
  target_replica_id: UUID
  retry_count: uint32
  expiry: TTL (default 3 hours)
}
```

The live node periodically retries delivery to the downed replica. On each successful delivery, the entry is removed. Entries that expire (target replica still down after 3 hours) are promoted to a full repair process.

Complexity: Each hinted handoff write is O(1) on the live node (append to buffer). Delivery is O(N) where N is the number of pending hints. The buffer size grows with the number of concurrent failures and write rate; typical production limits cap buffered hints to 1GB per node.

### Read Repair Queues

Read repair detects and fixes stale replicas during read operations. When a coordinator reads from multiple replicas and finds a version mismatch, it writes the latest version to the stale replicas asynchronously.

Structure:

```
ReadRepairEntry {
  key: String
  latest_value: Bytes
  latest_timestamp: HybridLogicalClock
  stale_replicas: List<UUID>
  created_at: Timestamp
}
```

The read repair queue is bounded to prevent unbounded memory growth. Default size is 64K entries per node. When the queue is full, the system stops doing read repair for new reads — those replicas will be fixed by the next anti-entropy cycle.

### Merkle Trees for Anti-Entropy

Merkle trees enable efficient comparison of data across replicas to find inconsistencies without transferring all data.

Structure:
- Build a balanced binary tree over the partition's key range.
- Each leaf node is a hash of the data in that key range.
- Each internal node is the hash of its children's hashes.
- The root hash represents the entire dataset.

Two replicas compare their Merkle tree roots. If roots match, the data is identical. If roots differ, they recursively descend the tree to find the exact key ranges that diverged. Only the divergent ranges need to be transferred and repaired.

Complexity:
- Building the tree: O(K) where K is the number of unique keys
- Comparison: O(log K) average, O(K) worst case if most ranges differ
- Repair data transferred: proportional to divergent ranges, not full dataset

Cassandra rebuilds Merkle trees every 5 to 30 minutes per table. The tree size is configurable; 64 to 256 leaves per node is typical.

## Consistency and State

### Node Crash — PA/EL Behavior

When a node crashes in a PA/EL system, the coordinator routes writes to a different replica. The replication manager adds the downed node to the hinted handoff target list. Reads from other replicas proceed normally, but may return stale data if the crashed node held the latest version of some keys. Read repair fixes these during subsequent reads.

### Node Crash — PC/EC Behavior

When a node crashes in a PC/EC system, the leader detects the failure through the gossip protocol. If the remaining replicas can form a quorum, writes continue. If quorum is lost (e.g., 2 of 3 replicas down), the system stops accepting writes and returns errors. Reads continue if at least one replica of the read quorum is alive.

### Stale Reads — The Read-Your-Writes Problem

In PA/EL systems, a common failure pattern: a client writes to one replica, then immediately reads from another replica that has not yet received the write. The client sees its earlier write disappear.

This is not a bug — it is the PA/EL contract. Applications that need read-your-writes consistency must use session consistency (read from the same replica that handled the write) or use a higher consistency level.

### Partition Heal — State Reconciliation

When a partition heals, the system enters a reconciliation phase. Each node exchanges its version vectors with peers. Conflicting writes are resolved according to the configured policy (LWW, CRDT merge, explicit application resolution). Hinted handoff writes are delivered. Merkle tree comparisons find any remaining silent divergence.

## Performance Characteristics

### PA/EL (DynamoDB, Cassandra default)

| Metric | Value |
|---|---|
| p50 read latency | 1-3ms |
| p99 read latency | 20-50ms |
| p50 write latency | 1-5ms |
| p99 write latency | 20-50ms |
| Max throughput per node | 10,000-50,000 ops/s |
| Cross-DC replication lag | 50-500ms |
| Memory per GB stored | ~200MB metadata overhead |

### PC/EC (Spanner, Zookeeper)

| Metric | Value |
|---|---|
| p50 read latency | 5-15ms |
| p99 read latency | 100-300ms |
| p50 write latency | 10-50ms |
| p99 write latency | 100-500ms |
| Max throughput per node | 1,000-10,000 ops/s |
| Cross-DC replication lag | 0ms (synchronous) |
| Memory per GB stored | ~500MB metadata overhead |

### Key Differences

The latency gap between PA/EL and PC/EC widens under contention. At 50% CPU saturation, PA/EL systems degrade gracefully because each request touches fewer nodes. PC/EC systems show latency spikes as lock contention and quorum waits compound.

Throughput differences are driven by coordination overhead. A PA/EL write completes in one network round-trip to one node. A PC/EC write requires two round-trips (prepare + commit) to a quorum of nodes. For a 3-replica cluster, that is 6x more network messages.

## Configuration and Tuning

### Read Consistency Levels (Cassandra)

| Level | Behavior | PACELC Regime |
|---|---|---|
| ONE | Return from closest replica. Fast, possibly stale. | PA/EL |
| TWO | Return from 2 replicas, compare timestamps. | Tuned PA/EL |
| QUORUM | Return from floor(N/2)+1 replicas. Strong. | PC/EC |
| ALL | Return from all replicas. Slowest, strongest. | PC/EC max |
| LOCAL_QUORUM | Quorum within local DC. Avoids cross-DC latency. | PC/EC local |
| SERIAL | Linearizable consistency with Paxos. | PC/EC strict |

### Write Consistency Levels (Cassandra)

| Level | Behavior | PACELC Regime |
|---|---|---|
| ANY | Write to any node, even if only hinted handoff. | PA/EL max |
| ONE | Write to 1 replica. Fast. | PA/EL |
| TWO | Write to 2 replicas. | Tuned PA/EL |
| QUORUM | Write to floor(N/2)+1 replicas. | PC/EC |
| ALL | Write to all replicas. | PC/EC max |
| EACH_QUORUM | Quorum in every DC. Strong across DCs. | PC/EC global |

### Replication Factor

RF=3 is the standard for production. It tolerates one node failure without losing quorum. RF=5 tolerates two failures but doubles write amplification. RF=1 is for development only — any node failure causes data loss.

### Hinted Handoff Duration

Default: 3 hours. Tune lower for ephemeral failures (10 minutes if crashes are quickly resolved). Tune higher if node replacements take time. Values above 24 hours accumulate enough hints to cause memory pressure and slow recovery.

### Tuning Guidelines

- For PA/EL: Use CL=ONE for reads and writes. Accept stale reads. Monitor replication lag.
- For PC/EC: Use CL=QUORUM for reads and writes. Accept higher latency. Monitor quorum failures.
- For mixed workloads: Use CL=ONE for 90% of operations and CL=QUORUM for the critical 10%. Use different keyspaces or table settings to separate the paths.
