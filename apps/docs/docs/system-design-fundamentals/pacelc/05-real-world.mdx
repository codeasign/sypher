# PACELC in Production

Real systems implement PACELC trade-offs with concrete mechanisms. The following case studies show how major databases and cloud services handle the partition-regime and else-regime trade-offs in production.

---

## DynamoDB — PA/EL at Planet Scale

DynamoDB is the canonical PA/EL implementation. It was designed at Amazon based on the Dynamo paper, which explicitly chose availability over consistency during partitions and low latency over consistency during normal operation.

### Key Numbers

DynamoDB with default settings at 1000 write capacity units (WCU) delivers:
- p50 write latency: 3-5ms
- p99 write latency: 20-50ms
- Replication lag: 50-200ms in the same region, 200-1000ms cross-region
- ConsistentRead (strong): adds 10-15ms to read latency

### Production Lesson: Shopping Cart at Amazon

Amazon's shopping cart service runs on DynamoDB with PA/EL defaults. The key insight is that a shopping cart does not need strong consistency. If a customer adds an item and it temporarily disappears, they can re-add it. The cost of a rare inconsistency (customer frustration) is far lower than the cost of adding latency to every add-to-cart operation (lost sales due to slow response).

During the 2013 Prime Day, DynamoDB handled 11.8 million cart operations per minute without degradation. The PA/EL design was essential — if every cart write required quorum, the system would have needed 3x the capacity.

### DynamoDB's ConsistentRead Option

DynamoDB offers `ConsistentRead=True` for operations that need strong consistency. This switches from PA/EL to PA/EC for that single read. The write path remains PA/EL, but the read path ensures the returned value is the latest across all replicas. This is used for checkout flows where the final cart total must be accurate.

### Production Lesson: Netflix on Cassandra

Netflix runs Cassandra for its user activity tracking, recommendation data, and viewing history. The key PACELC decision: Netflix uses CL=LOCAL_ONE for writes (PA/EL) and CL=LOCAL_QUORUM for reads that power the recommendation engine.

The rationale: write throughput for user activity is massive (billions of events per day). Using LOCAL_QUORUM for writes would require 3x the cluster size. Since activity data is append-mostly and missing a few events does not break the product, PA/EL writes are optimal. Reads for the recommendation engine use LOCAL_QUORUM because stale recommendations would degrade the user experience measurably.

Netflix reported that this hybrid approach saved approximately 40% in infrastructure costs compared to a uniform QUORUM configuration, while maintaining recommendation quality within acceptable bounds.

### Production Lesson: Apple's Cassandra Deployment

Apple runs one of the largest Cassandra deployments globally for iCloud services. Apple uses CL=LOCAL_QUORUM as the default for user-facing data (contacts, calendars, notes) to ensure synced data across devices is consistent. For analytics and logging, CL=ONE is used.

### Production Lesson: Google Ads

Spanner was built originally for Google's advertising business, which requires globally consistent financial reporting. An advertiser's budget must be deducted exactly once regardless of which data center handles the click event. Spanner's PC/EC guarantees make this possible without complex reconciliation.

The cost is significant: Spanner writes typically take 10-50ms compared to DynamoDB's 1-5ms. Google judged this acceptable because inconsistent ad budget deductions would cost real money through either overspend or underspend.

### Spanner's Read Flexibility

Spanner offers three read modes that map to different points on the PACELC spectrum:

- **Strong reads:** Linearizable. Go through the leader. 5-15ms.
- **Bounded staleness reads:** Client specifies maximum staleness (e.g., 10 seconds). The system reads from any replica that has caught up to that point. 1-5ms.
- **Exact staleness reads:** Client specifies exact timestamp. Useful for point-in-time queries.

This is Spanner's version of tunable consistency — a PC/EC base with optional PA/EL read modes.

### Production Lesson: Cosmos DB in Microsoft 365

Microsoft 365 uses Cosmos DB with Session consistency for most user data. Session consistency guarantees monotonic reads, monotonic writes, and read-your-writes within a single client session. This maps to PA/EL during normal operation with a session-specific consistency guarantee that prevents the most confusing inconsistency pattern (a user seeing their write disappear).

For global configuration data, Microsoft uses Strong consistency. For telemetry, Eventual consistency is used.

## Open Source Implementations

### Cassandra

Cassandra offers the widest range of tunable consistency in open source. It provides 10 consistency levels across reads and writes, covering the entire PA/EL to PC/EC spectrum. The gossip protocol detects partitions. Hinted handoff and read repair provide convergence. Merkle tree anti-entropy runs periodically.

### ScyllaDB

ScyllaDB is a C++ reimplementation of Cassandra that is API-compatible. It offers the same PACELC configuration model but with significantly lower latencies (2-3x faster per node) through a shared-nothing per-core architecture. ScyllaDB's PACELC performance characteristics are the same pattern but with lower absolute latency numbers.

### Riak

Riak was an early Dynamo-style implementation. It used vector clocks for conflict resolution instead of LWW, giving applications more control over merge semantics. Riak's PACELC model: PA/EL default with configurable quorum sizes per bucket.

## Lessons Learned

**Lesson 1: The Else regime matters more than the Partition regime.** A system that is fast and eventually consistent for 99% of operating time but slow and strongly consistent for 1% is better than the reverse.

**Lesson 2: Tunable consistency adds operational complexity.** Teams that configure per-operation consistency levels must test every path in their system. A single misconfigured consistency level can cause correctness bugs.

**Lesson 3: Session consistency is a practical sweet spot.** Read-your-writes guarantees eliminate the most confusing inconsistency pattern without the latency cost of full strong consistency.

**Lesson 4: Cross-region PACELC is different from within-region PACELC.** Within a region, latency is 1-5ms and the cost of consistency is low. Across regions, latency is 50-300ms and the cost of strong consistency is high. Plan accordingly.

**Lesson 5: Monitor both regimes separately.** A system that is healthy in the Else regime may perform completely differently during a partition. Monitor latency and consistency metrics for both regimes independently.
