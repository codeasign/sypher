# PACELC — Trade-offs and Failure Modes

The central insight of PACELC is that consistency always costs latency, even when the network is healthy. The real engineering question is: how much latency are you willing to pay, and for which operations?

---

## The Core Tension

Every read and write in a distributed system involves a hidden negotiation. When a client sends a request, the coordinator must decide how many nodes to consult before responding. Consulting fewer nodes is faster but risks returning stale data. Consulting more nodes is slower but guarantees correctness.

PACELC reveals that this negotiation happens in two different contexts: during abnormal operation (partition) and during normal operation (everyday health). Most decision frameworks focus only on the partition case. PACELC makes the normal-case trade-off explicit and forces architects to define both.

|---|---|---|---|
| Read latency | 1-3ms p50 | 5-15ms p50 | 1-15ms p50 |

|---|---|

## CAP Positioning and PACELC Refinement

CAP Theorem states that a distributed system can provide at most two of Consistency, Availability, and Partition Tolerance. PACELC does not replace CAP. It extends CAP by making explicit what CAP leaves implicit: the trade-off during normal operation.

### What CAP Gets Right

CAP correctly identifies the fundamental constraint during partitions. If the network splits, you cannot have both availability and consistency. This is a hard limit, not a tuning parameter.

### What CAP Misses

CAP is silent on normal operation. Most teams implementing CAP-Theorem-based designs choose AP, reasoning that partitions happen and availability matters. They then run the system for months in healthy-network mode with AP-implied weak consistency, unaware that they have a choice. PACELC surfaces this choice and gives architects the language to talk about it.

## Failure Modes

### Failure 1: Replication Lag Cascade

In PA/EL systems, a write surge causes replication lag to increase. Lagging replicas serve increasingly stale reads. If lag exceeds hinted handoff expiration (default 3 hours), writes are lost. As lag grows, the system's effective consistency degrades from eventual to potentially-never.

**Detection:** Monitor replication lag percentile. Alert at p99 lag &gt; 10 seconds.
**Mitigation:** Throttle writes, add nodes, or switch critical operations to LOCAL_QUORUM.

### Failure 2: Quorum Loss Under Partition

In PC/EC systems, a partition that isolates the minority side causes quorum loss on that side. Clients on the minority side cannot write and may not be able to read. If the partition lasts longer than the application timeout, those clients see errors.

**Detection:** Monitor quorum failure rate. Each quorum failure should trigger an immediate investigation.
**Mitigation:** Deploy 5 replicas instead of 3 for higher partition tolerance. Accept that minority-side unavailability is part of the PC/EC contract.

### Failure 3: Hinted Handoff Buffer Overflow

When a node is down for extended time, the live nodes buffer hinted handoff writes. If the buffer fills, new writes are rejected. The cluster experiences reduced write capacity even though most nodes are healthy.

**Detection:** Monitor hinted handoff queue depth. Alert at 50% capacity.
**Mitigation:** Increase buffer size, reduce hint TTL, or accelerate node recovery.

### Failure 4: Read Repair Storm

When a node comes back after a long outage, its data is significantly stale. Every read that hits it triggers a read repair, which writes the latest data back. If the application has high read volume, the read repair writes can saturate the cluster's write capacity.

**Detection:** Monitor read repair count per second. Alert at 10x baseline.
**Mitigation:** Use Merkle tree anti-entropy to bulk-repair before serving traffic. Let the node catch up gradually.

### Failure 5: Cross-Region Consistency Window

In multi-region PA/EL, a user who writes in us-east-1 and reads in eu-west-1 within 500ms sees their own write as missing. This is not a system failure — it is the contractual behavior. But it feels like a bug to the user.

**Detection:** Monitor user-facing stale-read reports.
**Mitigation:** Use session affinity (pin user to one region for the session duration) or use LOCAL_QUORUM for writes that the user expects to see immediately.

### Failure 6: Clock Skew in LWW

Last-write-wins conflict resolution relies on timestamps. If node clocks skew by more than a few milliseconds, events may be ordered incorrectly. A write that happened first can appear later and overwrite the correct value.

**Detection:** Monitor clock drift between nodes. Alert at drift &gt; 10ms.
**Mitigation:** Use hybrid logical clocks (HLC) or TrueTime instead of wall clocks.

## Anti-Patterns

### Anti-Pattern 1: Consistency Level ALL for Everything

Using CL=ALL for all reads and writes guarantees maximum consistency but creates a system that is fragile under any failure. One slow or down node takes down the entire write path.

**Better:** Use CL=QUORUM for strong consistency. CL=ALL only for operations where the data must be on every node immediately (rare).

### Anti-Pattern 2: Assuming PA/EL Means No Consistency Work

Some teams choose PA/EL and assume eventual consistency means they never need to think about consistency. Then they build features that assume strong consistency (deduplication, unique constraints, sequential counters) and get incorrect results.

**Better:** Map every feature to its consistency requirement. Features that need strong consistency should use a higher consistency level or a different data store.

### Anti-Pattern 3: Hybrid Consistency Without Testing

A team configures per-operation consistency levels but never tests the system under failure. The test suite uses CL=ONE for speed. In production, a critical path uses CL=QUORUM. The team never validates that the quorum path works during a partition.

**Better:** Test every consistency level in your failure injection pipeline. Chaos engineering must cover both regimes.

### Anti-Pattern 4: Global Strong Consistency as Default

Building every operation with EACH_QUORUM because "consistency is always better." This creates a system that is 10x more expensive and 20x slower than needed, for data that never needed strong guarantees.

**Better:** Profile your data access patterns. 80% of operations typically tolerate eventual consistency. Use strong consistency only where the business requires it.

### Anti-Pattern 5: Ignoring the Else Regime

A team analyzes CAP Theorem, decides on AP, and never revisits the decision. They accept 500ms replication lag because "we chose AP, so eventual consistency is expected." They never ask whether they could pay 5ms of latency to reduce lag to 50ms.

**Better:** The Else regime is where you can make targeted investments. Small latency budgets can buy significant consistency improvements when the network is healthy.

## When NOT to Use PACELC

PACELC is a framework for distributed databases that replicate data across multiple nodes. It is not applicable to:

**Single-node databases:** If your entire dataset fits on one machine and you use a single PostgreSQL or MySQL instance, you do not have distributed consistency trade-offs. ACID provides strong consistency without coordination overhead.

**Stateless services:** A service that stores no data and only transforms requests has no consistency trade-off. PACELC only applies to stateful systems.

**Cache-only systems:** A Redis or Memcached instance used purely as a cache with TTL-based expiration is not governed by PACELC. Cache misses are acceptable and do not cause permanent data inconsistency.

**Immutable data stores:** Append-only logs and event stores have different consistency properties. Since data is never updated, the consistency-latency trade-off for writes is different. Reads are straightforward.

**Synchronous replication with single leader:** A single-leader system with synchronous replication to followers does not have the PACELC trade-off in the same way. All writes go through the leader, so there is no write conflict. The trade-off is about read scalability, not consistency vs latency.
