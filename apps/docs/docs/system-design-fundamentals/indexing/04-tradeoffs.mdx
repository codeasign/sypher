## Core Tension

Indexes speed up reads at the cost of write performance and storage. Every index on a table must be maintained on every INSERT, UPDATE (of an indexed column), and DELETE. The cost is proportional to both the number of indexes and their complexity.

**Fundamental equation**:

```
write_cost = base_insert_cost + sum over indexes (index_maintenance_cost)
index_maintenance_cost = O(log n) per index per write

read_gain = original_scan_cost - index_lookup_cost
index_lookup_cost ≈ tree_depth × random_page_cost + matching_rows × heap_fetch_cost
```

This is not merely academic. In production, the impact of adding a single index can be measured directly:
- A single index on a 100-million-row table adds approximately 0.3-1.0ms to each INSERT.
- A covering index with 3 extra columns adds approximately 0.5-1.5ms per INSERT.
- A table with 8 indexes may see INSERT latency increase from 0.5ms (no indexes) to 8-12ms.
- At 10,000 INSERTs per second, each extra index consumes measurable CPU and I/O bandwidth.

The decision to add an index must weigh the read performance gain against the write performance cost. For a table with a 100:1 read-to-write ratio, the trade-off heavily favors indexes. For a 1:1 ratio, every index must be justified.

## Decision Matrix

The following 8 decision points guide index design. Each presents a binary choice with specific trade-offs.

### 1. B-Tree vs Hash

| Decision Factor | B-Tree / B+Tree | Hash |
|---|---|---|
| Query types supported | Equality, range, prefix, sorting, ORDER BY | Equality only |
| Lookup complexity | O(log n) tree traversal | O(1) average, O(n) worst case |
| Range scan | Yes — linked leaf chain enables efficient sequential traversal | No — hash function destroys key ordering |
| ORDER BY support | Yes — index returns rows in sorted order | No — hash ordering is arbitrary |
| Multi-column | Yes — composite index with leftmost prefix | No — single column hash only |
| Disk locality | Good — adjacent keys on same or nearby pages | Poor — hash chains may span overflow pages |
| Crash safety (PostgreSQL) | Always safe | Safe since PostgreSQL 10 (WAL-logged) |
| Size overhead | Moderate — internal nodes add overhead | Lower — no internal nodes, just bucket array |

**Choice rule**: Use B+Tree for everything unless the workload is exclusively single-key equality lookups and there is a verified performance benchmark showing hash index advantage. In practice, the B+Tree is almost always the correct choice.

### 2. Clustered vs Non-Clustered

| Decision Factor | Clustered | Non-Clustered |
|---|---|---|
| Number per table | 1 (determines physical row order) | Many (up to 999 in SQL Server, unlimited in PostgreSQL) |
| Data storage | Table data stored in index leaf pages | Separate from table data — leaf pages hold pointers |
| Range scan efficiency | Excellent — rows physically adjacent, sequential I/O | Poor — rows scattered across heap, random I/O |
| Insert performance | May cause page splits at rightmost insertion point | Affects B+Tree structure only (separate from heap) |
| Update of key column | Row must physically move to maintain sort order | Pointer update only — no row movement needed |

**Choice rule**: Cluster on the primary key or the most common range-scan column. For tables with monotonically increasing keys (auto-increment, timestamp), the rightmost-page hotspot causes write contention — consider a different clustering strategy or hash partitioning.

### 3. Single Column vs Composite

| Decision Factor | Single Column | Composite |
|---|---|---|
| Leftmost prefix applicability | All queries on that column benefit | Only queries filtering on leading columns |
| Size per column | Each extra index adds full B+Tree overhead | One B+Tree covers multiple columns — more compact |
| Optimizer complexity | Simple cost estimation | More complex — optimizer must evaluate column order and selectivity |
| Covering potential | Low — rarely covers all query columns | High — can serve multi-column queries without heap access |

**Choice rule**: Use composite indexes for queries that filter on multiple columns together. The leading column should be the most selective column in the WHERE clause. Use single-column indexes for columns queried independently.

### 4. Covering vs Minimal

| Decision Factor | Covering (with INCLUDE) | Minimal (key columns only) |
|---|---|---|
| Query speed | Fastest possible — no heap access required | Requires heap fetch per matching row |
| Index size | Larger — stores extra payload columns | Smaller — stores only key columns |
| Write overhead | Higher — more data per leaf entry | Lower — per-entry data is minimal |
| Best use case | Hot queries executed thousands of times per second | Infrequent queries, uniqueness enforcement only |

**Choice rule**: Use covering indexes for the top 3-5 queries by execution frequency. For all other queries, use minimal indexes.

### 5. Unique vs Non-Unique

| Factor | Unique | Non-Unique |
|---|---|---|
| Constraint | Prevents duplicate values (database-enforced) | None — multiple rows can have the same key value |
| Optimizer signal | Strong — optimizer knows at most one row matches a key lookup | Weak — optimizer assumes multiple matches |
| Insert overhead | Higher — must check uniqueness for each insert (additional lookup) | Lower — no uniqueness check needed |
| Use case | Primary keys, email addresses, order numbers, username | Foreign keys, status columns, date ranges |

### 6. Dense vs Sparse (BRIN)

| Factor | Dense (B+Tree) | Sparse (BRIN) |
|---|---|---|
| Entries per row | 1 entry per row | 1 entry per block range (typically 1-32 pages) |
| Storage | Full B+Tree — millions of pages for large tables | Tiny — kilobytes for the same data |
| Lookup precision | Exact row location | Points to a range of pages; must scan within the range |
| Maintain cost | Every write updates the B+Tree (O(log n)) | Extremely cheap — updates the range min/max only when a new page boundary is reached |
| Use case | OLTP lookups, point queries | Time-series data, log tables, append-only data |

**Choice rule**: BRIN indexes are the default choice for append-only tables with natural ordering (time-series, logs). They are thousands of times smaller than B+Tree indexes and essentially free to maintain.

### 7. Online vs Offline Build

| Factor | Online (CONCURRENTLY) | Offline (standard) |
|---|---|---|
| Write blocking | None (brief exclusive lock at the end) | Full write block for entire duration |
| Build time | 2x to 4x longer (two full table scans) | Fastest possible (single scan + sort) |
| Resource usage | Higher CPU and I/O (two full scans, sort spool) | Lower total resource consumption |
| Failure handling | Leaves an invalid index that must be manually dropped | Transaction rollback cleans up automatically |

**Choice rule**: Always use CONCURRENTLY in production unless the table is in a maintenance window and writes can be paused.

### 8. Ascending vs Descending

Most indexes are ascending by default. Descending indexes matter when the most recent values appear first:

```sql
-- For queries like "show me the 10 most recent orders":
CREATE INDEX idx_orders_created_desc ON orders (created_at DESC);
-- The index returns rows in descending order without a separate sort step.

-- For mixed sort direction in ORDER BY:
CREATE INDEX idx_orders_customer_date ON orders (customer_id ASC, created_at DESC);
-- Supports: WHERE customer_id = ? ORDER BY created_at DESC
```

## CAP Positioning

Indexes are local to each database node. In a distributed system:

- **Local indexes** (Cassandra, MySQL replicas): Each node maintains its own indexes. No cross-node coordination is needed. Consistency depends on replication lag. Reads that use a local index may return stale data if the replica is behind the primary.

- **Global secondary indexes** (DynamoDB GSI, CockroachDB): The index is distributed across nodes. Writes to the primary table must also update the GSI, which may involve a distributed transaction. This trades availability for consistency — a GSI write that fails may cause the primary write to roll back.

- **Interleaved indexes** (Cloud Spanner): Co-locate index rows with parent rows. Strong consistency is maintained within the interleaved hierarchy, but cross-hierarchy queries require coordination.

The choice of index distribution affects CAP behavior: local indexes favor availability (AP), global indexes favor consistency (CP) at the cost of write latency.

## Failure Modes

### Index Bloat

Index bloat occurs when index pages contain dead entries that have not been reclaimed. In PostgreSQL, this happens when autovacuum falls behind the write volume. A bloated index can consume 2x to 10x its optimal size.

**How bloat accumulates**: Each UPDATE in PostgreSQL creates a new row version (MVCC). The old row version is marked as dead. The index still points to the dead version. Autovacuum must scan the index and remove dead entries. If writes outpace autovacuum processing, dead entries accumulate.

**Symptoms**: Index size grows beyond expected (monitor `pg_relation_size`). The `pg_stat_user_tables.n_dead_tup` value climbs into the millions. VACUUM runs constantly. Query performance degrades because the B+Tree depth increases and leaf chain scans spend time traversing empty entries.

**Quantifiable impact**: A 2GB index that has grown to 8GB due to bloat requires 4x the buffer pool space, 4x the I/O for scans, and 4x the vacuum processing time.

**Fix**: `REINDEX INDEX CONCURRENTLY` or periodic scheduled reindexing. Prevention: tune autovacuum thresholds for heavy-write tables.

### Page Split Storms

When a table receives a burst of inserts that all target the same key range (e.g., a bulk load of time-ordered data), the rightmost leaf page splits repeatedly. Each split cascades upward as parent pages fill.

**Real-world example**: A bulk insert of 10 million time-series records triggers 10 million rightmost-page insertions. The B+Tree grows horizontally (page splits) and vertically (root split). During the storm, concurrent inserts wait on page latches, causing latency spikes from 2ms to 500ms.

**Mitigation**: Lower fill factor to 70-80 percent before bulk loads. Batch inserts in sorted order. Use partitioning to spread inserts across multiple pages. For PostgreSQL, temporarily increase `maintenance_work_mem` for the bulk load session.

### Index Fragmentation

Over time, page splits and deletions cause leaf pages to become physically discontiguous on disk. An index scan that could read a single extent now issues random I/O across dozens of extents. This is distinct from bloat — bloat is about dead entries, fragmentation is about physical ordering.

**Measurement**: PostgreSQL's `pgstattuple` extension provides `avg_leaf_density` (percentage of space occupied by live entries) and `leaf_fragmentation` (percentage of leaf pages not in physical order). Values above 30 percent fragmentation indicate a need for reindexing.

### Stale Statistics

The query optimizer depends on index statistics (cardinality, most common values, correlation between indexed columns and physical order) to choose the best plan. When statistics are stale:

- The optimizer may choose a full table scan over an index that would be more efficient, causing a sudden query regression.
- The optimizer may choose an index that the current data distribution no longer supports.
- Composite index column order assumptions break as data skew changes over time.

**Fix**: Increase `default_statistics_target` in PostgreSQL (default 100, production 500-1000 for large tables). Tune autovacuum's autoanalyze threshold. After bulk operations, manually run `ANALYZE`.

### Deadlocks from Index Locking Order

Multiple transactions updating different rows on the same table can deadlock if they acquire index page locks in different orders. This is especially common with composite indexes where the locking order depends on the key values.

**Example**: Transaction A updates a row with `last_name = 'Jones'` and locks the leaf page for that range. Transaction B updates a row with `last_name = 'Smith'` and locks a different leaf page. But if Transaction A's update also requires a lock on a page that Transaction B holds (due to a different index's page layout), a deadlock occurs. The database resolves the deadlock by aborting one of the transactions.

**Prevention**: Ensure concurrent transactions access rows in a consistent order. Reduce the number of indexes on heavily updated tables. Use `FOR UPDATE NOWAIT` or `SKIP LOCKED` where appropriate.

### Composite Column Order Mistakes

The single most common index design error is incorrect column order in a composite index.

**Wrong**: An index on `(status, created_at)` for a query that filters by `created_at` without `status`. The index cannot be used because `status` is the leading column and is not in the query's WHERE clause.

**Right**: An index on `(customer_id, order_date)` for queries filtering by `customer_id` alone (leading column high selectivity) and also filtering by both `customer_id` and `order_date` together.

**Rule**: Place the column with the most selective filter first. The leftmost prefix rule means the index is only useful for queries that reference the leading column.

## Anti-Patterns

### Indexing Every Column

Blindly indexing every column in a table is a common mistake in early-stage databases. Each index adds write overhead and storage. Most columns are queried infrequently or never directly.

**Example**: A 50-column `users` table with 50 indexes. Every INSERT requires 50 B+Tree updates. The table receives 1,000 INSERTs per second — each index processes 1,000 writes per second. The database spends 80 percent of its CPU time maintaining indexes that are used by fewer than 5 queries.

**Fix**: Profile actual query patterns for two weeks before adding non-primary-key indexes. Monitor `pg_stat_user_indexes.idx_scan` and drop indexes with zero scans.

### Not Indexing Foreign Keys

Foreign key columns are used in JOINs. Without an index on the referencing column (the child-side FK), every DELETE or UPDATE of the parent table triggers a sequential scan of the child table to verify referential integrity.

**Impact**: Deleting a single row from a parent table with 10 million child rows and no FK index takes 30+ seconds. Adding the FK index reduces this to under 1ms.

### Ignoring Composite Index Column Order

The leading column of a composite index must match the query's WHERE clause. Placing a low-selectivity column first wastes the index.

**Wrong**: `INDEX (status, user_id)` — queries filtering on `user_id` alone cannot use this index.
**Right**: `INDEX (user_id, status)` — queries filtering on `user_id` use the leading column; queries filtering on both columns also benefit.

### Index on Low-Cardinality Columns

An index on a column with fewer than 100 distinct values (gender, boolean, status with 3 states) is almost never useful for B+Tree lookups — the index returns too many rows per lookup. A bitmap index helps in data warehouses. In OLTP, the optimizer will likely ignore the index.

### Over-Indexing Write-Heavy Tables

Tables receiving 10,000+ writes per second with 8+ indexes suffer significant write amplification. Each insert modifies 8 B+Trees, each potentially causing page splits and generating WAL entries.

## When NOT to Use an Index

1. **Append-only log tables**: Clickstream events, audit logs, IoT sensor readings. These tables are bulk-scanned for analytics. An index adds write overhead with zero query benefit.

2. **Small tables**: A table with fewer than 1,000 rows fits in a single page. An index lookup requires at least two page reads (index + heap). A sequential scan reads one page. The index is always slower.

3. **Columns never used in WHERE, JOIN, or ORDER BY**: If a column appears only in the SELECT list, indexing it provides no benefit unless it enables an index-only scan.

4. **High-write, low-read tables**: A log buffer table written 100,000 times per second and queried once per hour. The index slows every write. Consider indexing only during active debugging.
