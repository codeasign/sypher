## How This Topic Appears

Indexing questions appear in system design interviews across three formats. Understanding which format you are facing determines how you structure your answer.

**Direct questions** (the interviewer explicitly asks about indexes):
1. "Design an indexing strategy for a ride-sharing application that stores 50 million driver locations."
2. "How would you index a social media feed table that supports queries by user ID and by timestamp?"
3. "What indexes would you create for an e-commerce product catalog with faceted search?"
4. "Explain how B+Trees work and why they are used for database indexes."
5. "Your database has a slow query. Walk through how you would diagnose and fix it."

**Embedded questions** (indexing is part of a broader system design):
6. "How would you store and query messages for a chat application with 1 billion users?"
7. "Design the database schema for a Twitter-like feed. How would you handle the timeline query?"
8. "Design a key-value store that supports range queries."
9. "How would you implement a search feature for a document storage system?"
10. "Your write-heavy application is slowing down. What could be causing it?"

**Debugging questions** (optimization and troubleshooting):
11. "A query that used to be fast is now slow. How do you debug it?"
12. "Your INSERT statements are taking 10x longer than expected in production. What do you check?"
13. "How would you reduce disk space usage in a database that has grown to 10TB?"

---

## What the Interviewer Is Testing

Interviewers assess four dimensions of indexing knowledge:

1. **Conceptual understanding**: Do you know how B+Trees, hash indexes, and inverted indexes work? Can you explain the difference between clustered and non-clustered indexes? Can you draw a B+Tree and trace a lookup?

2. **Practical design judgment**: Given a specific query pattern, can you design the right index? Can you choose between a covering index and a simple index? Can you explain when a partial index makes sense? Can you calculate the storage cost of an index?

3. **Trade-off awareness**: Do you understand that indexes speed up reads at the cost of writes? Can you calculate the impact of an additional index on INSERT throughput? Can you articulate when NOT to use an index?

4. **Operational experience**: Do you know how to monitor index usage, detect bloat, and manage indexes in production without downtime? Can you explain `CREATE INDEX CONCURRENTLY` and its failure modes?

## Level Expectations

### L3 / Entry Level

**Expected**: Can explain B+Tree structure and why databases use indexes. Can create a basic index for a WHERE clause predicate. Understands that indexes speed up reads and slow down writes.

**Sample answer**: "A B+Tree is a balanced tree data structure where each node contains multiple keys and pointers. Leaf nodes store the actual data pointers and are linked together, which makes range scans efficient. An index on `last_name` lets the database find rows with `WHERE last_name = 'Smith'` in O(log n) time instead of scanning all rows. I would create `CREATE INDEX idx_last_name ON users (last_name)`."

**Interviewer tests**: Can the candidate describe the basic data structure? Do they understand the difference between O(log n) and O(n)?

### L4 / Mid Level

**Expected**: Can design a composite index for a multi-column query. Understands the leftmost prefix rule. Can estimate index size. Knows the difference between clustered and non-clustered indexes.

**Sample answer**: "For a query filtering by `(city, status, created_at)`, I would create a composite index on `(city, status, created_at)`. The leftmost prefix rule means this index also helps queries filtering by `city` alone or `(city, status)`. Column order matters: `city` first because it is the most selective. For a 100 million row table, this index would be approximately 3-4GB. I would also consider a partial index if we only query for `status = 'active'`, which would reduce the index size by 60 percent."

**Interviewer tests**: Does the candidate understand the leftmost prefix rule? Can they estimate storage? Do they consider partial indexes as an optimization?

### L5 / Senior Level

**Expected**: Can design a complete indexing strategy for a complex system. Considers write overhead, maintenance, and monitoring. Understands trade-offs between different index types and databases.

**Sample answer**: "For a ride-sharing application, I would use a three-part strategy. First, a partial index on `WHERE status = 'available'` reduces the index size by 60 percent since only available drivers need to be matched. Second, an expression index on geohash grid cells co-locates drivers geographically — each query scans only the grid cells overlapping the bounding box. Third, a covering index on `(geohash_cell, last_heartbeat) INCLUDE (driver_id, lat, lng)` allows index-only scans. The total index size would be approximately 1-2GB for 20 million drivers. I would build the index with `CONCURRENTLY` to avoid downtime, and monitor `idx_scan` to confirm the index is being used."

**Interviewer tests**: Does the candidate think about the full lifecycle? Do they consider operational aspects like online index creation and monitoring?

### L6 / Staff Level

**Expected**: Can design a multi-region indexing strategy. Understands distributed index challenges. Considers consistency implications, cross-region replication, and CAP trade-offs.

**Sample answer**: "For a global multi-region deployment, I would use local indexes on each region's read replicas for geographical queries, and a global secondary index for cross-region queries. Local indexes are eventually consistent with replication lag — the application must tolerate stale reads for the locality query. Global indexes require distributed transactions for consistency, which adds latency. I would partition the data by region, with a local hot index in each region and a global index updated asynchronously via change data capture. The global index would use a different key structure — region ID plus local key — to avoid key conflicts across regions."

**Interviewer tests**: Does the candidate understand the distributed systems implications of indexing? Can they reason about consistency vs latency trade-offs at a global scale?

### L7 / Principal Level

**Expected**: Can design a custom indexing layer from scratch. Understands storage engine internals. Considers all failure modes and recovery strategies.

**Sample answer**: "I would design a custom indexing layer supporting multiple index types. The primary index uses a lock-free B-link-tree for write-heavy workloads, supporting 100,000 inserts per second per node. Secondary indexes use a log-structured merge approach with a 64MB memtable and leveled compaction to balance read and write performance. The system supports online index creation via a two-phase snapshot-and-CDC approach. Failure modes I would address: index corruption detected via periodic page-level checksum verification, bloat monitored via page density metrics, and automatic reindexing when bloat exceeds 30 percent."

## FAANG Framework

Follow this 5-step framework when answering indexing design questions.

### Step 1: Clarify Query Patterns

Ask the interviewer to define the specific read and write patterns that the index must support.

**Questions to ask**:
- "What are the top 3 queries by frequency?"
- "What are the top 3 queries by latency requirement?"
- "What is the ratio of reads to writes?"
- "Are there range queries, or only equality lookups?"
- "What columns appear in WHERE, JOIN, and ORDER BY clauses?"
- "What is the current and projected data volume?"

**Why this matters**: The most common mistake in indexing interviews is jumping to a solution without understanding the query patterns. An index that helps a read-once-per-hour analytics query is different from one that helps a 50,000 QPS login query.

### Step 2: Identify Hot Queries

Prioritize the queries that matter most — those that run most frequently or have the strictest latency targets.

| Priority | Query | Frequency | Latency Target | Index Strategy |
|---|---|---|---|---|
| P0 | Get user by email | 50K/s | 10ms | Unique covering index on email |
| P1 | List orders by user | 10K/s | 50ms | Composite index on (user_id, date DESC) |
| P2 | Search products by name | 5K/s | 200ms | GIN or full-text index |
| P3 | Analytics: daily revenue | 1/s | 5s | No index — sequential scan is acceptable |

**Framework**: "P0 queries get covering indexes. P1 queries get well-designed composite indexes. P2 queries may get specialized index types (GIN, GiST). P3 queries get minimal or no indexing because the low frequency does not justify the write overhead."

### Step 3: Design Indexes

For each P0 and P1 query, design the optimal index:

- **Single vs composite**: Composite for multi-column filtering.
- **Leading column**: Most selective column first.
- **Covering**: INCLUDE all SELECTed columns for hot queries.
- **Partial**: Exclude rows that never match the query.
- **Expression**: For function-based filters.
- **Unique**: For primary key and uniqueness constraints.

**Output for each index**:
```sql
-- P0: login query — get user by email
CREATE UNIQUE INDEX idx_users_email
ON users (email)
INCLUDE (name, avatar_url, created_at);
-- Rationale: Unique constraint enforces no duplicate emails.
-- INCLUDE eliminates heap fetches for the login query.
-- Estimated size: 100M rows * 50B email = 5GB

-- P1: list orders by user with date sort
CREATE INDEX idx_orders_user_date
ON orders (user_id, created_at DESC);
-- Rationale: Supports both equality on user_id and ORDER BY created_at DESC.
-- Leftmost prefix means user_id-only queries also benefit.
```

### Step 4: Estimate Size

Calculate the storage cost of each index.

**Formula**:
```
leaf_pages = ceil(rows * entry_size / (page_size * fill_factor))
internal_pages = leaf_pages / fanout + leaf_pages / fanout^2 + 1
total_size = (leaf_pages + internal_pages) * page_size
```

**Example walkthrough**: "For 100M rows with an 8-byte key, 6-byte pointer, 8KB pages, and 90% fill factor: leaf_pages = 100M * 18 / (8192 * 0.9) = 244,000 pages. Internal pages = 244K / 500 + 244K / 500^2 + 1 = 490 pages. Total = 244,490 * 8KB = 1.96GB."

### Step 5: Explain Trade-offs

For each index, state the trade-off explicitly:

**Read benefit**: "This index reduces query time from 45 seconds to under 10ms. The covering index eliminates 12,000 random heap fetches per query."

**Write cost**: "This index adds approximately 0.5ms per INSERT and 0.3ms per DELETE. At 10,000 writes per second, this index consumes approximately 5 seconds of CPU time per second."

**Storage cost**: "The index adds 2GB of storage for 100M rows. With 5 such indexes, that is 10GB of additional storage."

**Maintenance cost**: "The index will need weekly reindexing to prevent bloat. Each reindex takes 30 minutes with CONCURRENTLY."

## Phrase Bank

Use these phrases during the interview to demonstrate depth.

**Setup phrases**:
- "Before designing indexes, I need to understand the query patterns. What are the top 3 queries by frequency?"
- "Let me start by identifying the read-write ratio for this table."
- "I will design indexes for the P0 queries first, then consider P1 queries."
- "I need to estimate the data volume — how many rows and what is the average row size?"
- "Are the queries equality lookups or range scans? That determines whether a hash or B+Tree index is appropriate."

**Design phrases**:
- "A composite index on `(user_id, created_at DESC)` supports both the JOIN and the ORDER BY clause."
- "I will use a covering index here because this query runs 10,000 times per second and every heap fetch adds latency."
- "A partial index `WHERE status = 'active'` is appropriate because only 5 percent of rows match the query pattern."
- "The leading column should be the most selective column in the WHERE clause."
- "I will use the `INCLUDE` clause to add payload columns without affecting the sort order."
- "For the full-text search query, I would use a GIN index on the tsvector column."

**Trade-off phrases**:
- "The read performance gain justifies the write overhead because the read-to-write ratio is 100:1."
- "This index adds 2GB of storage, which is acceptable given the 10TB database."
- "The additional index will slow INSERTs by approximately 10 percent, which is acceptable for this workload."
- "I would not add an index on this column because it has low cardinality — only 3 distinct values."
- "The covering index eliminates heap fetches but doubles the index size. I will benchmark to confirm the trade-off is worthwhile."

**Operational phrases**:
- "I would create this index with `CONCURRENTLY` to avoid blocking writes in production."
- "I would monitor `idx_scan` in `pg_stat_user_indexes` weekly to confirm the index is being used."
- "I would set up a scheduled reindexing job for indexes with more than 20 percent bloat."
- "I would test the index in a staging environment with production-scale data before deploying."

## Follow-Up Questions and Advanced Topics

After answering the main question, interviewers often probe deeper with follow-ups. Prepare for these:

### Follow-Up 1: Index Interactions

**Interviewer**: "What happens when a table has multiple indexes and a query could use any of them?"

**Expected answer**: The optimizer evaluates each viable index plan — bitmap scan with index AND/OR, or individual index scans with heap fetches. The optimizer picks the plan with the lowest estimated cost. For multi-index queries, the database may use bitmap index scans: each index produces a bitmap of matching page locations, the bitmaps are combined via AND/OR operations, and heap pages are fetched in physical order (reducing random I/O).

```sql
-- Example of bitmap scan plan:
EXPLAIN SELECT * FROM orders WHERE customer_id = 42 OR status = 'pending';
-- May use: BitmapOr -> Bitmap Heap Scan on orders
--   Bitmap Index Scan on idx_orders_customer (customer_id = 42)
--   Bitmap Index Scan on idx_orders_status (status = 'pending')
```

### Follow-Up 2: Expression Index Performance

**Interviewer**: "Does an expression index on `LOWER(email)` get updated on every INSERT even if the email is already lowercase?"

**Expected answer**: Yes. The expression index evaluates `LOWER(email)` for every INSERT or UPDATE that changes the `email` column, regardless of whether the value is already lowercase. The database does not check whether the expression result changed — it performs the evaluation and index update unconditionally. This is a small but measurable overhead.

### Follow-Up 3: Non-Volatile Expressions

**Interviewer**: "Can you create an expression index on an immutable function like `DATE_TRUNC('day', created_at)`? What about a volatile function like `NOW()`?"

**Expected answer**: Immutable functions are allowed because they always produce the same output for the same input. The index stores the result permanently. Volatile functions like `NOW()` are not allowed as expression index components because the stored value would become stale — the index would need constant rebuilding to remain accurate.

### Follow-Up 4: Partial Index and Insert Performance

**Interviewer**: "If I have a partial index `WHERE status = 'active'` and I INSERT a row with `status = 'inactive'`, does the index get updated?"

**Expected answer**: No. The partial index only contains entries for rows that match the WHERE predicate. An INSERT with `status = 'inactive'` does not touch the partial index at all. This is the primary performance benefit of partial indexes — they exclude rows from index maintenance.

### Follow-Up 5: The Buffer Pool and Index Scans

**Interviewer**: "Why might an index scan be slower than a sequential scan even though it reads fewer pages?"

**Expected answer**: An index scan reads pages in random order (following the index's logical order, not the heap's physical order). If the index points to rows scattered across many heap pages, each page read is a random I/O operation. A sequential scan reads pages in physical order, which is much faster on HDD (sequential I/O is 10-100x faster than random) and still faster on SSD (sequential is 2-4x faster). For queries that match more than approximately 5-10 percent of rows, the optimizer may choose a sequential scan over an index scan because the sequential I/O pattern is faster than random heap fetches.

## Anti-Patterns to Avoid in Answers

Beyond the red flags listed above, certain answer patterns undermine an otherwise strong response:

**The Kitchen Sink**: "Let me add every possible index." Stick to the minimum set. Each additional index must be justified.

**The Hand-Wave**: "The database will figure out the best index." The interviewer wants to hear specific design decisions, not a hand-off to the optimizer.

**The Zero-Sum**: "Indexes always make things faster." Acknowledge the write overhead. Show that you understand the full cost.

**The Ivory Tower**: "I would use a perfectly normalized schema with B+Tree indexes on every foreign key." Real-world indexing involves trade-offs, caching, denormalization, and operational constraints.

**The Magic Fix**: "Let me add an index." Before adding an index, explain: (1) what query it helps, (2) at what cost (write/space), (3) how you will verify it works, (4) what monitoring you will set up.

## Practice Questions (Self-Study)

Use these questions to practice the FAANG framework:

1. **Instagram-like photo feed**: Design indexes for a photo-sharing app where the feed shows photos from followed users, sorted by creation time. Table: `photos(photo_id, user_id, image_url, caption, created_at)`. Query: `SELECT * FROM photos WHERE user_id IN (?) ORDER BY created_at DESC LIMIT 30`.

2. **Uber-like driver location**: Design indexes for finding nearby drivers. Table: `drivers(driver_id, lat, lng, status, last_heartbeat)`. Query: `SELECT * FROM drivers WHERE status = 'available' AND lat BETWEEN ? AND ? AND lng BETWEEN ? AND ? ORDER BY last_heartbeat DESC`.

3. **Slack-like message search**: Design indexes for a chat application's message search. Table: `messages(message_id, channel_id, sender_id, content, sent_at)`. Queries: (a) `SELECT * FROM messages WHERE channel_id = ? ORDER BY sent_at DESC LIMIT 100`, (b) `SELECT * FROM messages WHERE content ILIKE '%keyword%' AND channel_id = ?`.

4. **Amazon-like order history**: Design indexes for an e-commerce order history page. Table: `orders(order_id, customer_id, total, status, created_at)`. Query: `SELECT * FROM orders WHERE customer_id = ? ORDER BY created_at DESC LIMIT 20`. Also supports admin view: `SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at ASC LIMIT 100`.

5. **Google Analytics-like page views**: Design indexes for a page view analytics system. Table: `page_views(id, site_id, page_url, visitor_id, viewed_at)`. Query: `SELECT COUNT(*), page_url FROM page_views WHERE site_id = ? AND viewed_at BETWEEN ? AND ? GROUP BY page_url ORDER BY COUNT(*) DESC LIMIT 10`.

For each practice question, go through all 5 steps of the FAANG framework. Write out the SQL index creation commands, estimate the size, and explain the trade-offs.

Avoid these mistakes during the interview. Each signals a lack of depth.

1. **Not knowing B+Tree basics**: "An index stores data in a linked list" or "B+Tree is like a binary search tree." Every database engineer should be able to sketch a B+Tree and explain its balance property and fanout advantage.

2. **Suggesting indexes on every column**: "I would add an index on each column in the table." This shows no understanding of the write overhead trade-off.

3. **Ignoring write overhead**: "The more indexes, the better — they always help." Indexes add cost to every write. The candidate must acknowledge this and quantify it.

4. **Wrong composite column order**: "An index on `(status, user_id)` for a query that filters by `user_id` alone." The leading column is wrong — the index cannot be used.

5. **Ignoring selectivity**: "An index on `gender` with 2 values will speed up the query." Low-cardinality columns rarely benefit from B+Tree indexes.

6. **Not knowing the leftmost prefix rule**: "An index on `(a, b, c)` can be used for a query filtering on `b` alone." The leading column must be in the query's WHERE clause.

7. **No consideration of concurrent index creation**: "We lock the table for 2 hours to build the index." Production indexing must be online.

8. **No monitoring or maintenance plan**: "We add the index and move on." Production indexes need monitoring for usage, bloat, and fragmentation.

9. **Claiming hash indexes are always faster**: "I would use a hash index because it is O(1)." Hash indexes only support equality lookups and cannot be used for range queries or sorting.

10. **Not considering partial indexes**: "I will index the entire table even though 90 percent of rows are filtered out."

## Sample Interview Walkthrough

**Question**: "Design the indexing strategy for a Twitter-like timeline query. The query is: `SELECT tweet_id, content, author_id, created_at FROM tweets WHERE user_id IN (following_ids) ORDER BY created_at DESC LIMIT 50`."

**Step 1 — Clarify**: "I need to understand the data volume. How many tweets per user? How many followed users? Read-write ratio? Latency target?"

**Step 2 — Identify hot queries**: "The timeline query is P0 — runs on every app open, tens of thousands of times per second. Latency target: under 100ms."

**Step 3 — Design indexes**: "The IN-list query cannot be served by a single B+Tree index efficiently. The solution is a fan-out approach: pre-compute each user's timeline in a separate timeline table. The timeline table has a composite primary key on `(user_id, created_at DESC)`."

**Step 4 — Estimate size**: "100M active users, 50 tweets in each timeline = 5B timeline rows. Each row ~200 bytes. Timeline table ~1TB."

**Step 5 — Trade-offs**: "Fan-out approach provides fast reads (single row lookup per user) but amplifies writes — each tweet is written to 200 follower timelines. Write amplification is acceptable because read:write ratio is ~1000:1. I would batch fan-out writes via a background job."
