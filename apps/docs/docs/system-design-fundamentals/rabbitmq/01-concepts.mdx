## The Problem That Message Brokers Solve

A financial trading platform processes 200,000 orders per day. Each order triggers notifications to four services: a risk engine, a compliance auditor, a trade reporter, and a client WebSocket gateway. The original architecture connected each service directly to the order processor via HTTP calls.

This worked for a few hundred orders. As volume grew, failures appeared:

- When the compliance auditor took 30 seconds to respond, every other notification queued behind it — a single slow downstream service blocked all order processing.
- A spike of 10,000 orders in one minute triggered 40,000 concurrent HTTP requests. The order processor ran out of worker threads and dropped connections.
- If the client WebSocket gateway was down for maintenance, orders from that period were never delivered — there was no retry mechanism.
- Adding a fifth notification target meant modifying the order processor code.

These are not problems with the notification logic. They are problems with *how* services communicate. Direct service-to-service calls couple senders to receivers, assume both are available simultaneously, and provide no buffering for traffic spikes.

A message broker sits between services. Services that produce messages (publishers) send them to the broker instead of directly to consumers. Services that process messages (consumers) pull from the broker at their own pace. The broker handles buffering, routing, delivery guarantees, and failure recovery.

**RabbitMQ** is the most widely deployed open-source message broker. It implements the Advanced Message Queuing Protocol (AMQP 0-9-1) and is written in Erlang — a language designed for concurrent, fault-tolerant systems.

## Insight — RabbitMQ as a Post Office

Think of RabbitMQ as a post office, but with programmable sorting rules.

A regular post office works like this: you drop a letter in a mailbox (publish). The postal service sorts letters by destination address (routing). Letters travel through a network to the correct post office (transport). The recipient collects mail from their box (consume).

RabbitMQ extends this model. Instead of one fixed sorting rule (destination address), you define rules called *bindings* that decide which messages go where. A single message can be delivered to multiple queues. Rules can be based on message content, routing keys, or arbitrary headers.

The key innovation is the **exchange** — a routing component that sits between publishers and queues. Publishers never write directly to queues. They send messages to an exchange, and the exchange decides which queues receive the message based on binding rules.

## Core Vocabulary

**Message** — The unit of data transferred through RabbitMQ. A message consists of a payload (bytes — typically JSON or a serialized object), a routing key (a string the exchange uses for routing), and metadata (headers, delivery mode, content type, expiration). In the trading platform, a message might be `\{ "orderId": "ORD-98421", "symbol": "AAPL", "side": "BUY", "qty": 500, "price": 185.30 \}` with a routing key of `"order.created"`.

**Publisher** — Any application that sends messages to an exchange. A publisher does not know which consumers will receive its messages, or if any will. It sends to a named exchange and moves on. The original trading platform had one publisher (the order processor). With RabbitMQ, the order processor publishes once instead of making four HTTP calls.

**Exchange** — The routing engine inside RabbitMQ. Exchanges receive messages from publishers and place them into queues. The exchange type determines the routing algorithm: direct, topic, fanout, or headers. An exchange is bound to zero or more queues through bindings. Each exchange has a name and a type. When you define the binding rules, you are programming the exchange.

**Binding** — A link between an exchange and a queue. A binding includes a routing pattern that tells the exchange which messages go to that queue. For example, binding queue `"risk-engine"` to exchange `"orders"` with routing key `"order.created"` means: every message with routing key `"order.created"` arriving at exchange `"orders"` is copied into queue `"risk-engine"`. Multiple bindings can point to the same queue — a queue receives the union of all messages matching any of its bindings.

**Queue** — A named buffer that stores messages until a consumer processes them. Queues live inside RabbitMQ and are backed by disk or memory depending on configuration. A queue has a name, durability setting (survives broker restart), exclusivity (used by one connection only), and auto-delete setting (removed when last consumer unsubscribes). In the trading platform, each downstream service has its own queue: `risk-engine-queue`, `compliance-queue`, `trade-report-queue`, `ws-gateway-queue`.

**Routing Key** — A string attribute on each message that the exchange uses to decide routing. For a direct exchange, the routing key is compared exactly to binding patterns. For a topic exchange, the routing key is matched against a pattern with wildcards. Routing keys use dot-separated words by convention: `"order.created"`, `"order.cancelled"`, `"trade.executed"`.

**Consumer** — An application that connects to RabbitMQ, subscribes to a queue, and processes messages. Consumers register a callback that RabbitMQ invokes for each delivered message. Consumers can process one message at a time or configure prefetch to process multiple in parallel. After processing, a consumer sends an ACK (acknowledgment) to tell RabbitMQ the message was handled and can be removed.

**AMQP** — Advanced Message Queuing Protocol, version 0-9-1. This is the wire-level protocol RabbitMQ speaks. AMQP defines a set of commands for connection negotiation, channel management, exchange declaration, queue declaration, binding, publishing, consuming, and acknowledgments. It is a binary protocol — efficient compared to HTTP/JSON-based messaging. AMQP 0-9-1 is the default protocol. RabbitMQ also supports MQTT, STOMP, and HTTP via plugins.

**Connection** — A TCP connection between an application and the RabbitMQ server. Connections are long-lived. A single application typically opens one or two connections to RabbitMQ. Establishing a connection requires authentication (username/password) and virtual host selection. Connections are expensive to create — keep them alive and reuse them.

**Channel** — A lightweight virtual connection inside a single TCP connection. A connection can have hundreds of channels. Channels multiplex operations: publishing on channel 1, consuming on channel 2, declaring exchanges on channel 3 — all within the same TCP connection. Channels are cheap to create and close. Most AMQP operations happen on a channel, not directly on a connection.

**Virtual Host (vhost)** — An isolated namespace within a single RabbitMQ server. Each vhost has its own exchanges, queues, bindings, permissions, and policies. Vhosts allow a single RabbitMQ cluster to serve multiple teams or environments with complete isolation. The trading platform might have separate vhosts for development, staging, and production.

**ACK (Acknowledgment)** — A signal from a consumer to RabbitMQ confirming that a message was processed successfully. After receiving an ACK, RabbitMQ removes the message from the queue. If a consumer disconnects without sending an ACK, RabbitMQ re-delivers the message to another consumer. ACKs can be automatic (sent immediately on delivery) or manual (sent after processing is complete). Manual ACKs provide the strongest delivery guarantee.

**NACK (Negative Acknowledgment)** — A signal from a consumer that a message could not be processed. With `requeue=true`, RabbitMQ puts the message back in the queue for another attempt. With `requeue=false`, the message is discarded or sent to a dead-letter exchange. NACKs are useful when a message is malformed — retrying would fail again, so routing to a dead-letter queue for inspection is better.

**Dead-Letter Exchange (DLX)** — An exchange that receives messages that could not be processed normally. Messages are dead-lettered when: a consumer NACKs with `requeue=false`, the message TTL expires, or the queue length limit is exceeded. The dead-letter exchange routes these messages to a dead-letter queue for inspection and reprocessing. This is the safety net for production systems.

**TTL (Time-to-Live)** — A setting on a message or a queue that limits how long a message can sit undelivered. A message with a 5-minute TTL that is not consumed within 5 minutes is either discarded or routed to a dead-letter exchange. TTL prevents stale messages from accumulating. In the trading platform, time-sensitive orders might have a TTL of 30 seconds — if the consumer is down, the order expires rather than executing after a delay.

**Prefetch** — A setting that controls how many messages are sent to a consumer at once. A prefetch of 1 means RabbitMQ sends one message and waits for an ACK before sending the next. A prefetch of 10 means up to 10 messages are outstanding at any time. Higher prefetch improves throughput but increases the risk of message re-delivery if a consumer crashes. The right prefetch depends on message processing time and the cost of reprocessing.

## How Message Routing Works — Step by Step

### Step 1: Declare an Exchange

Before any messages flow, an administrator or application declares an exchange. This creates the exchange in RabbitMQ's metadata store.

```bash
# Declare a direct exchange named "orders"
rabbitmqadmin declare exchange name=orders type=direct
```

In application code using the `pika` Python library:

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.exchange_declare(exchange='orders', exchange_type='direct')
```

The exchange declaration is idempotent — calling it multiple times has no effect if the properties match. If properties differ (e.g., changing type from `direct` to `topic`), RabbitMQ raises an error.

### Step 2: Declare Queues

Each consumer declares its queue. Queues need names and durability settings.

```python
channel.queue_declare(queue='risk-engine-queue', durable=True)
channel.queue_declare(queue='compliance-queue', durable=True)
channel.queue_declare(queue='trade-report-queue', durable=True)
channel.queue_declare(queue='ws-gateway-queue', durable=True)
```

### Step 3: Create Bindings

Bindings connect exchanges to queues with routing patterns.

```python
channel.queue_bind(exchange='orders', queue='risk-engine-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='compliance-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='trade-report-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='ws-gateway-queue', routing_key='order.created')
```

All four queues bind to the `orders` exchange with routing key `order.created`. Every order-created message goes to all four queues.

### Step 4: Bind for Other Routing Keys

The compliance queue also needs order-cancelled messages:

```python
channel.queue_bind(exchange='orders', queue='compliance-queue', routing_key='order.cancelled')
```

Now `compliance-queue` receives messages with routing key `order.created` or `order.cancelled`. Binding rules are additive — a queue receives the union of all routing keys bound to it.

### Step 5: Publisher Sends a Message

The order processor publishes a message:

```python
channel.basic_publish(
    exchange='orders',
    routing_key='order.created',
    body='{"orderId": "ORD-98421", "symbol": "AAPL", "side": "BUY", "qty": 500}',
    properties=pika.BasicProperties(delivery_mode=2)  # persistent
)
```

The publisher specifies the exchange name and routing key. It does not specify queues or consumers. This is the core decoupling — the publisher knows about the exchange, not about consumers.

### Step 6: Exchange Receives the Message

RabbitMQ receives the message on channel 3 (for example) within the publisher's connection. It identifies the exchange `orders` and looks up its type (`direct`).

### Step 7: Exchange Consults Binding Table

For a direct exchange, the routing algorithm is:

1. Look up all bindings on this exchange.
2. Filter bindings where `binding_key == message.routing_key`.
3. For each matching binding, enqueue the message into the bound queue.

With four bindings matching `order.created`, the exchange places the message into all four queues.

### Step 8: Message Stored in Queue

The message is appended to each queue's internal buffer. For durable queues, the message is also written to disk. Each queue maintains a pointer to the next unconsumed message.

### Step 9: Consumer Receives the Message

Each consumer, having subscribed to its queue with `basic_consume`, receives the message. The delivery includes the message body, routing key, and delivery tag (a unique identifier for acknowledgment).

```python
def callback(ch, method, properties, body):
    order = json.loads(body)
    risk_score = calculate_risk(order)
    print(f"Risk score for {order['orderId']}: {risk_score}")
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='risk-engine-queue', on_message_callback=callback)
```

### Step 10: Consumer Acknowledges

After processing, the consumer sends an ACK. RabbitMQ marks the message as processed and removes it from the queue. If the consumer crashes without ACKing, RabbitMQ re-delivers the message to the next available consumer.

## Exchange Types — The Four Routing Strategies

RabbitMQ provides four exchange types, each implementing a different routing algorithm.

### Direct Exchange

Routes messages to queues where the binding key exactly matches the message routing key.

<AsciiDiagram

  content={

                    └───────────────────┘

             ┌─────────────┘       └─────────────┐
             │  "order.created"                  │  "order.created"
             ▼                                    ▼
       ┌──────────┐                        ┌──────────┐
       │ Risk     │                        │Compliance│
       │ Engine   │                        │ Queue    │
       │ Queue    │                        │          │
       └──────────┘                        └──────────┘

       routing_key="order.created"         routing_key="order.created"
       binding_key="order.created"         binding_key="order.created"
`
  }
  alt="Diagram showing a direct exchange routing messages to queues based on exact routing key match"
/>

A message with routing key `"order.created"` goes only to queues bound with the exact key `"order.created"`. A message with `"order.cancelled"` goes only to queues bound with `"order.cancelled"`. Exact match — no wildcards.

**When to use**: Commands or events with a fixed set of known routing keys. Notification types that map 1:1 to processing pipelines.

```python
# Declare
channel.exchange_declare(exchange='orders', exchange_type='direct')

# Bind
channel.queue_bind(exchange='orders', queue='risk-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='compliance-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='compliance-queue', routing_key='order.cancelled')
channel.queue_bind(exchange='orders', queue='audit-queue', routing_key='order.created')
channel.queue_bind(exchange='orders', queue='audit-queue', routing_key='order.cancelled')
channel.queue_bind(exchange='orders', queue='audit-queue', routing_key='order.updated')

# Publish
channel.basic_publish(exchange='orders', routing_key='order.created', body=order_data)
```

### Topic Exchange

Routes messages to queues where the binding pattern matches the routing key, with wildcard support.

<AsciiDiagram
  id="rabbitmq/01-concepts-topic"
  title="Topic Exchange — Pattern-Matched Routing with Wildcards"
  content={

                    `┌───────────────────┐
                    │   Topic Exchange   │
                    │   "events"         │
                    └───────────────────┘
                          │    │    │
            ┌─────────────┘    │    └─────────────┐
            │  "order.*"       │  "trade.*"       │  "*.eu"
            ▼                  ▼                  ▼
       ┌──────────┐      ┌──────────┐      ┌──────────┐
       │ Orders   │      │ Trades   │      │ EU       │
       │ Queue    │      │ Queue    │      │ Queue    │
       └──────────┘      └──────────┘      └──────────┘

       * matches one word
       # matches zero or more words
`
  }
  alt="Diagram showing a topic exchange using wildcard patterns to route messages to multiple queues"
/>

Routing keys in topic exchanges use dot-separated words: `"order.created.eu"`, `"trade.executed.us"`, `"order.cancelled.eu"`.

Two wildcards:
- `*` matches exactly one word: `"order.*"` matches `"order.created"` but not `"order.created.eu"`.
- `#` matches zero or more words: `"order.#"` matches `"order"`, `"order.created"`, and `"order.created.eu"`.

| Binding Pattern | Matches | Does Not Match |
|---|---|---|
| `order.*` | `order.created`, `order.cancelled` | `order.created.eu` |
| `order.#` | `order`, `order.created`, `order.created.eu` | `trade.executed` |
| `*.eu` | `order.created.eu`, `trade.executed.eu` | `order.created.us` |
| `#.critical` | `order.critical`, `order.created.critical` | `order.created.normal` |

**When to use**: Event-driven systems where routing keys have hierarchical structure. Multi-tenant applications where routing includes region, data center, or tenant ID.

```python
# Declare
channel.exchange_declare(exchange='events', exchange_type='topic')

# Bind — get all order events in any region
channel.queue_bind(exchange='events', queue='order-processor', routing_key='order.#')

# Bind — get only trade events from EU
channel.queue_bind(exchange='events', queue='eu-trades', routing_key='trade.executed.eu')

# Publish
channel.basic_publish(exchange='events', routing_key='order.created.eu', body=order_data)
channel.basic_publish(exchange='events', routing_key='trade.executed.us', body=trade_data)
```

### Fanout Exchange

Routes messages to all bound queues, ignoring the routing key entirely.

                    └───────────────────┘
                         │     │     │

                         ▼     ▼     ▼
                    ┌───┐ ┌───┐ ┌───┐

       Every message goes to every bound queue.
       Routing key is ignored.
`
  }
  alt="Diagram showing a fanout exchange broadcasting every message to all bound queues regardless of routing key"
/>

**When to use**: Broadcast scenarios where every consumer needs every message. Cache invalidation events, configuration updates, log distribution, system-wide announcements. Also useful when you want one publisher to feed multiple independent consumers without the publisher knowing about them.

```python
# Declare
channel.exchange_declare(exchange='broadcast', exchange_type='fanout')

# Each consumer binds its queue — no routing key needed
channel.queue_bind(exchange='broadcast', queue='cache-invalidator')
channel.queue_bind(exchange='broadcast', queue='log-writer')
channel.queue_bind(exchange='broadcast', queue='metrics-collector')

# Publish — no routing key
channel.basic_publish(exchange='broadcast', routing_key='', body=config_update)
```

### Headers Exchange

Routes messages based on header attributes rather than routing keys.

<AsciiDiagram
  id="rabbitmq/01-concepts-headers"
  title="Headers Exchange — Route by Message Headers"
  content={

                    │    "router"        │
                    └───────────────────┘

`
  }
  alt="Diagram showing a headers exchange using message header attributes for routing decisions"
/>

Headers exchanges ignore routing keys. Instead, they match on message headers — key-value pairs in the message properties. Bindings specify header criteria with `x-match`:

- `x-match = all` — all specified headers must match (AND logic).
- `x-match = any` — any specified header must match (OR logic).

**When to use**: Complex routing logic that does not fit into hierarchical routing keys. Routing decisions based on many attributes — message type, priority, source service, format version.

```python
# Declare
channel.exchange_declare(exchange='router', exchange_type='headers')

# Bind — match messages where tier=gold AND region=eu
args = {'x-match': 'all', 'tier': 'gold', 'region': 'eu'}
channel.queue_bind(exchange='router', queue='gold-eu-queue', arguments=args)

# Bind — match messages where tier=silver OR tier=gold
args = {'x-match': 'any', 'tier': 'silver', 'tier': 'gold'}
channel.queue_bind(exchange='router', queue='premium-queue', arguments=args)

# Publish — set headers
headers = {'tier': 'gold', 'region': 'eu', 'source': 'web'}
props = pika.BasicProperties(headers=headers)
channel.basic_publish(exchange='router', routing_key='', body=order_data, properties=props)
```

## Exchange Type Decision Matrix

| Exchange Type | Routing Basis | Wildcards | Use Case |
|---|---|---|---|
| Direct | Exact routing key | No | Task distribution, command routing |
| Topic | Pattern matching | `*` and `#` | Event streams with hierarchy |
| Fanout | None (all queues) | N/A | Broadcast, cache invalidation |
| Headers | Message headers | No | Multi-attribute routing |

## Default Exchange

RabbitMQ provides a pre-declared default exchange (named `""` — empty string). It is a direct exchange with a special property: every queue that is created is automatically bound to it with the queue name as the routing key.

This is why you can publish directly to a queue:

```python
# This works because of the default exchange
channel.basic_publish(exchange='', routing_key='risk-engine-queue', body=order_data)
```

The empty exchange name tells RabbitMQ to use the default exchange, which routes by queue name. This is convenient for simple cases but couples the publisher to the queue name. For production systems, always use a named exchange.

## Default Exchange Internals

```python
# Behind the scenes, declaring a queue:
channel.queue_declare(queue='risk-engine-queue')

# Is equivalent to:
channel.queue_declare(queue='risk-engine-queue')
channel.exchange_declare(exchange='amq.direct', type='direct')
channel.queue_bind(exchange='amq.direct', queue='risk-engine-queue',
                    routing_key='risk-engine-queue')
```

The default exchange (`""`) is actually the `amq.direct` exchange with auto-binding for every declared queue. Use it for development and simple scripts. For anything that will run in production, define explicit exchanges and bindings.

## Message Durability and Persistence

Durability in RabbitMQ has three layers.

**Exchange durability**: A durable exchange survives broker restarts. Declare with `durable=True`. Non-durable exchanges are recreated on each restart.

```python
channel.exchange_declare(exchange='orders', exchange_type='direct', durable=True)
```

**Queue durability**: A durable queue survives broker restarts. The queue definition (name, bindings, settings) is persisted. Messages inside a durable queue may still be lost unless each message is also marked persistent.

```python
channel.queue_declare(queue='risk-engine-queue', durable=True)
```

**Message persistence**: A message marked as persistent (`delivery_mode=2`) is written to disk before the publisher receives confirmation. Without persistence, messages in a durable queue are lost if the broker crashes before flushing to disk.

```python
channel.basic_publish(
    exchange='orders',
    routing_key='order.created',
    body=order_data,
    properties=pika.BasicProperties(delivery_mode=2)  # persistent
)
```

**What survives a restart**: Durable exchanges, durable queues, bindings, and persistent messages (that were written to disk). Everything else is lost.

**Performance trade-off**: Persistent messages are 10-20 times slower than non-persistent messages because each message requires a disk fsync. For high-throughput systems where data loss is acceptable (e.g., log streams, cache invalidations), use non-persistent messages. For financial data, use persistent messages with publisher confirms.

## Consumer Acknowledgements

Acknowledgments are critical for reliable message processing. Without them, a processing failure looks like success — the message is removed and lost.

### Automatic Acknowledgement

```python
# auto_ack=True — message is marked delivered as soon as it is sent
channel.basic_consume(queue='risk-engine-queue',
                      on_message_callback=callback,
                      auto_ack=True)
```

The message is removed from the queue immediately upon delivery. If the consumer crashes before processing, the message is lost. Use only for idempotent, non-critical processing or when delivery guarantees are handled elsewhere.

### Manual Acknowledgement

```python
def callback(ch, method, properties, body):
    try:
        process_order(body)
        ch.basic_ack(delivery_tag=method.delivery_tag)
    except Exception:
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)

channel.basic_consume(queue='risk-engine-queue',
                      on_message_callback=callback,
                      auto_ack=False)
```

Manual ACK gives the consumer full control. The message stays in the queue until ACKed. If the consumer dies, RabbitMQ re-delivers the message to another consumer.

### Negative Acknowledgment

```python
# Message is malformed — don't requeue, send to dead-letter queue
ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False)

# Temporary failure — requeue for another attempt
ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)
```

### Reject (Pre-AMQP 0-9-1)

```python
# Older API — equivalent to basic_nack without multiple
ch.basic_reject(delivery_tag=method.delivery_tag, requeue=True)
```

## Prefetch

Prefetch limits how many unacknowledged messages a consumer can have at once.

```python
# Process one message at a time
channel.basic_qos(prefetch_count=1)

# Allow 10 unacknowledged messages
channel.basic_qos(prefetch_count=10)
```

With `prefetch=1`, consumers receive messages in round-robin fashion — each consumer gets one message, processes it, ACKs, and gets the next. This provides fair distribution across consumers.

With `prefetch=10`, a fast consumer can have 10 messages in flight while a slow consumer has 0. This maximizes throughput but can lead to uneven load if processing times vary.

**Rule of thumb**: Set prefetch to `(processing_time_seconds / 0.1)` for sub-second processing, or keep it at 1 for variable processing times. For batch processing where each message takes under 50ms, prefetch of 50-200 is appropriate.

## Dead Letter Exchanges

A dead-letter exchange (DLX) holds messages that could not be processed normally.

Configure a dead-letter exchange on a queue:

```python
args = {
    'x-dead-letter-exchange': 'orders.dlx',
    'x-dead-letter-routing-key': 'dead.orders'
}
channel.queue_declare(queue='risk-engine-queue', arguments=args)

# Declare DLX and dead-letter queue
channel.exchange_declare(exchange='orders.dlx', exchange_type='direct')
channel.queue_declare(queue='dead-orders-queue')
channel.queue_bind(exchange='orders.dlx', queue='dead-orders-queue',
                   routing_key='dead.orders')
```

Messages land in `dead-orders-queue` when:
- A consumer NACKs with `requeue=false`.
- A consumer rejects with `requeue=false`.
- Message TTL expires before delivery.
- Queue length limit is exceeded.

Operators can inspect the dead-letter queue, fix issues, and replay messages through a dedicated republish tool.

## Message TTL

Two levels of TTL control how long a message waits.

**Per-queue TTL**: All messages in the queue expire after a fixed time.

```python
args = {'x-message-ttl': 60000}  # 60 seconds
channel.queue_declare(queue='time-sensitive-queue', arguments=args)
```

**Per-message TTL**: Individual messages have their own expiration.

```python
props = pika.BasicProperties(expiration='30000')  # 30 seconds
channel.basic_publish(exchange='orders', routing_key='order.created',
                      body=body, properties=props)
```

When a message expires, RabbitMQ either discards it or routes it to the dead-letter exchange if configured.

## Queue Length Limit

Prevent queues from growing unbounded during consumer outages.

```python
# Max 10,000 messages in the queue
args = {'x-max-length': 10000}
channel.queue_declare(queue='risk-engine-queue', arguments=args)

# Max 20 MB total
args = {'x-max-length-bytes': 20_000_000}
channel.queue_declare(queue='risk-engine-queue', arguments=args)
```

When a queue exceeds the limit, the oldest messages are dropped or dead-lettered. This protects broker memory and ensures new messages are processed instead of old ones.

## Mental Model Check

Test your understanding of RabbitMQ concepts with these questions.

**1. A publisher sends a message with routing key `"user.created"` to a direct exchange. Queue A is bound with routing key `"user.created"`. Queue B is bound with routing key `"user.*"`. Which queues receive the message?**

Only Queue A. Direct exchanges require an exact match between the message routing key and the binding key. The pattern `"user.*"` is treated as a literal string, not a wildcard — wildcards only work in topic exchanges.

**2. A consumer processes a message, encounters a transient database error, and wants to retry. What should it do?**

Send a NACK with `requeue=True`. The message is placed back at the head of the queue (by default) for immediate re-delivery. For a more robust approach, the consumer should NACK with `requeue=False` and publish the message to a retry queue with a delay — this prevents tight retry loops that could overwhelm the system.

**3. You have 5 consumer instances processing messages from one queue. Each message takes 200ms to process. What prefetch value maximizes throughput without causing excessive redeliveries?**

A prefetch of 5 allows each consumer to have 5 messages in flight. With 5 consumers, that is 25 messages being processed concurrently. If one consumer crashes, up to 5 messages are re-delivered. A prefetch of 10 (50 in-flight) risks 10 messages being re-delivered per crash. Choose prefetch based on the reprocessing cost. For 200ms work, prefetch 5-10 is reasonable — the cost of reprocessing 5-10 messages is under 2 seconds.

**4. A fanout exchange has three bound queues. A message arrives with routing key `"test"`. How many queues receive the message?**

Three. Fanout exchanges ignore routing keys entirely. Every message goes to every bound queue. The routing key could be empty, null, or `"anything"` — the result is the same.

**5. A queue is configured with `x-message-ttl=5000` and `x-dead-letter-exchange=dlx`. A message sits in the queue for 6 seconds without being consumed. What happens?**

The message expires after 5 seconds. RabbitMQ routes it to the dead-letter exchange `dlx` with the original routing key (or the `x-dead-letter-routing-key` if configured). The DLX then routes it according to its own bindings — typically into a dead-letter queue for inspection. The message header includes the reason: `x-first-death-exchange` and `x-first-death-reason` set to `expired`.
