## How to Use These Challenges

Each challenge has a scenario description, requirements, constraints, and a model answer. Attempt the challenge before reading the answer. The challenges increase in complexity from fundamental design to multi-cluster production architectures.

The time estimates are guidelines:
- **Beginner (Tier 1)**: 15-20 minutes each
- **Intermediate (Tier 2)**: 30-45 minutes each

## Tier 1 — Beginner Challenges

### Challenge 1 — Design a Work Queue

**Scenario**: A document processing service converts uploaded PDF files to plain text. Users upload 5,000 PDFs per day. Processing one PDF takes 2-10 seconds depending on file size. The system needs to distribute work across 5 worker processes.

**Requirements**:
1. A message represents one PDF to process. The message contains the file path and upload time.
2. Workers run on separate machines. Each worker processes one PDF at a time.
3. If a worker crashes mid-processing, another worker must pick up the PDF.
4. Workers must not process the same PDF simultaneously.
5. The system must handle a backlog of up to 10,000 unprocessed PDFs.

**Constraints**:
- No message should be lost if the RabbitMQ broker restarts.
- Do not use any exchange — use the default exchange.

**Model Answer**:

```python
# Publisher
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# Durable queue survives broker restart
channel.queue_declare(queue='pdf-queue', durable=True)

def submit_pdf(file_path, upload_time):
    channel.basic_publish(
        exchange='',
        routing_key='pdf-queue',
        body=f'{{"path": "{file_path}", "uploaded": "{upload_time}"}}',
        properties=pika.BasicProperties(delivery_mode=2)  # persistent
    )
```

```python
# Consumer (worker)
import pika
import time

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='pdf-queue', durable=True)

# Process one message at a time — fair distribution
channel.basic_qos(prefetch_count=1)

def process_pdf(ch, method, properties, body):
    pdf_data = json.loads(body)
    print(f"Processing PDF: {pdf_data['path']}")
    try:
        # Simulate PDF processing
        time.sleep(3)
        print(f"Completed: {pdf_data['path']}")
        ch.basic_ack(delivery_tag=method.delivery_tag)
    except Exception as e:
        print(f"Failed: {e}")
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)

channel.basic_consume(queue='pdf-queue', on_message_callback=process_pdf)
channel.start_consuming()
```

**Design decisions explained**:
- `delivery_mode=2` on messages persists them to disk. Combined with a durable queue, messages survive a broker restart.
- `prefetch_count=1` ensures one worker does not hoard all messages — RabbitMQ distributes them round-robin. Each worker gets one message, processes it, ACKs, and gets the next.
- Manual ACK ensures crash safety. If the worker dies during `time.sleep(3)`, the message is still in the queue and RabbitMQ delivers it to another worker.
- The default exchange (`exchange=''`) simplifies the design for a single-queue scenario. For production, use a named exchange.

### Challenge 2 — Design a Log Aggregation Pipeline

**Scenario**: Three microservices generate log entries at rates of 200, 500, and 800 entries per second respectively. All logs are written to a central file for compliance auditing. A separate analytics service reads log entries for real-time monitoring.

**Requirements**:
1. Each microservice publishes log entries to its own topic: `service-a.logs`, `service-b.logs`, `service-c.logs`.
2. The compliance archiver receives ALL log entries from all services.
3. The analytics service receives only `service-b` and `service-c` logs (service-a is legacy and not monitored).
4. New services must be able to join without changing existing consumers.

**Constraints**:
- Use one exchange for all log routing.
- Do not use fanout (too many services would create too many bindings).

**Model Answer**:

Use a topic exchange with a hierarchical routing key pattern: `{service}.{level}.{module}`.

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# Topic exchange for log routing
channel.exchange_declare(exchange='logs', exchange_type='topic')

# Compliance archiver — receives all logs
channel.queue_declare(queue='compliance-archive', durable=True)
channel.queue_bind(exchange='logs', queue='compliance-archive',
                   routing_key='#')  # matches everything

# Analytics — receives service-b and service-c logs
channel.queue_declare(queue='analytics-monitor', durable=True)
channel.queue_bind(exchange='logs', queue='analytics-monitor',
                   routing_key='service-b.#')
channel.queue_bind(exchange='logs', queue='analytics-monitor',
                   routing_key='service-c.#')

# Publishers
def publish_log(service, level, module, message):
    routing_key = f'{service}.{level}.{module}'
    channel.basic_publish(
        exchange='logs',
        routing_key=routing_key,
        body=json.dumps({'service': service, 'msg': message}),
        properties=pika.BasicProperties(delivery_mode=2)
    )

publish_log('service-b', 'ERROR', 'auth', 'Login failed for user 123')
publish_log('service-c', 'INFO', 'api', 'Request processed in 45ms')
```

**Key design**: The topic exchange with `#` wildcard gives the compliance archiver everything. Individual patterns give the analytics service only what it needs. Adding `service-d` requires one new binding for analytics — the archiver gets it automatically.

## Tier 2 — Intermediate Challenges

### Challenge 3 — Design a Reliable Payment Processing Pipeline

**Scenario**: An e-commerce platform processes 50,000 payments per day. Each payment goes through three stages: authorization, fraud check, and settlement. Each stage can fail independently.

**Requirements**:
1. A payment must go through all three stages in order.
2. If fraud check fails, the payment is rejected and must not proceed to settlement.
3. If authorization fails due to a temporary error (e.g., bank timeout), retry up to 3 times with a 30-second delay between retries.
4. If authorization fails on the 4th attempt, the payment is abandoned and stored for manual review.
5. The fraud check service takes 2-5 seconds per payment. The authorization service takes 500ms-2 seconds. The settlement service takes 1-3 seconds.
6. Each stage should have its own pool of worker processes.

**Constraints**:
- Use separate queues for each stage.
- Use dead-letter exchanges for retry and failure handling.

**Model Answer**:

The pipeline uses a dead-letter exchange with TTL for retry. Each stage publishes to the next stage on success, NACKs without requeue on failure.

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# Stage 1: Authorization
channel.exchange_declare(exchange='payments', exchange_type='direct')
channel.exchange_declare(exchange='auth-retry-dlx', exchange_type='direct')

# Authorization queue with retry DLX
channel.queue_declare(
    queue='auth-payments',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'auth-retry-dlx',
        'x-dead-letter-routing-key': 'auth-payments',
        'x-message-ttl': 30000,  # 30-second retry delay
        'x-max-length': 10000
    }
)
channel.queue_bind(exchange='payments', queue='auth-payments',
                   routing_key='payment.new')

# Retry queue — messages from DLX come back here after 30 seconds
channel.queue_declare(
    queue='auth-retry',
    durable=True,
    arguments={
        'x-dead-letter-exchange': '',
        'x-dead-letter-routing-key': 'auth-payments',
        'x-message-ttl': 30000
    }
)
channel.queue_bind(exchange='auth-retry-dlx', queue='auth-retry',
                   routing_key='auth-payments')

# Abandoned payments queue (after 3 retries)
channel.queue_declare(queue='abandoned-payments', durable=True)
channel.queue_bind(exchange='auth-retry-dlx', queue='abandoned-payments',
                   routing_key='abandoned')

# Authorization consumer
def process_auth(ch, method, properties, body):
    payment = json.loads(body)
    retry_count = int(properties.headers.get('x-retry-count', 0)) if properties.headers else 0

    try:
        result = authorize_payment(payment)
        if result['success']:
            # Move to fraud check
            channel.basic_publish(
                exchange='payments',
                routing_key='payment.fraud_check',
                body=body,
                properties=pika.BasicProperties(delivery_mode=2)
            )
            ch.basic_ack(delivery_tag=method.delivery_tag)
        else:
            # Permanent failure — abandon
            ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False)
    except TemporaryError:
        if retry_count < 3:
            # Retry with incremented count
            headers = {'x-retry-count': retry_count + 1}
            channel.basic_publish(
                exchange='auth-retry-dlx',
                routing_key='auth-payments',
                body=body,
                properties=pika.BasicProperties(
                    delivery_mode=2,
                    headers=headers
                )
            )
            ch.basic_ack(delivery_tag=method.delivery_tag)
        else:
            # Abandon after 3 retries
            channel.basic_publish(
                exchange='auth-retry-dlx',
                routing_key='abandoned',
                body=body,
                properties=pika.BasicProperties(delivery_mode=2)
            )
            ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_qos(prefetch_count=5)
channel.basic_consume(queue='auth-payments', on_message_callback=process_auth)
```

**Retry mechanism**: The DLX + TTL pattern routes failed messages to a retry queue, waits 30 seconds (TTL), then routes them back to the original queue. This avoids tight retry loops. After 3 retries, the message goes to the abandoned payments queue for manual review.

**Why not use requeue**: `basic_nack(requeue=true)` places the message at the head of the queue for immediate redelivery. This creates a tight retry loop that consumes CPU and starves other messages. The DLX-TTL pattern provides a genuine delay.

### Challenge 4 — Design a Real-Time Leaderboard for 1 Million Users

**Scenario**: A gaming platform needs to display a real-time leaderboard. Every time a player scores points, the leaderboard must update within 500ms. The system has 1 million daily active players, each generating an average of 50 score events per hour (approximately 14,000 events/second).

**Requirements**:
1. Score events must not be lost.
2. The leaderboard service must process events in order per player (score updates for the same player must be applied sequentially).
3. Multiple leaderboard consumer instances must be able to run for horizontal scaling.
4. The system must continue operating if one consumer instance fails.

**Constraints**:
- Use RabbitMQ for ordering and delivery.
- Each consumer processes a subset of players to maintain per-player ordering.

**Model Answer**:

Use consistent hashing to route each player's score events to the same queue, ensuring per-player ordering. Multiple queues allow horizontal scaling.

```python
import pika
import hashlib

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# Create a fanout exchange to broadcast score events
# to all partition routers
channel.exchange_declare(exchange='score-events', exchange_type='topic')

# Create partition queues — each holds events for a subset of players
PARTITIONS = 10
for i in range(PARTITIONS):
    queue_name = f'leaderboard-partition-{i}'
    channel.queue_declare(queue=queue_name, durable=True)
    # Each partition binds with its partition key pattern
    channel.queue_bind(exchange='score-events', queue=queue_name,
                       routing_key=f'score.partition.{i}')


def get_partition(player_id):
    """Consistent hash to assign player to partition."""
    hash_val = int(hashlib.md5(player_id.encode()).hexdigest(), 16)
    return hash_val % PARTITIONS


def publish_score(player_id, score, game_id):
    partition = get_partition(player_id)
    routing_key = f'score.partition.{partition}'
    message = json.dumps({
        'player_id': player_id,
        'score': score,
        'game_id': game_id,
        'timestamp': time.time()
    })
    channel.basic_publish(
        exchange='score-events',
        routing_key=routing_key,
        body=message,
        properties=pika.BasicProperties(delivery_mode=2)
    )
```

```python
# Consumer for partition 3
def consume_partition(partition_id):
    conn = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
    ch = conn.channel()
    queue_name = f'leaderboard-partition-{partition_id}'

    ch.basic_qos(prefetch_count=1)  # Process one player's event at a time

    def process_score(ch, method, properties, body):
        event = json.loads(body)
        update_leaderboard(event['player_id'], event['score'])
        ch.basic_ack(delivery_tag=method.delivery_tag)

    ch.basic_consume(queue=queue_name, on_message_callback=process_score)
    ch.start_consuming()
```

**Why consistent hashing works**: All score events for player `ABC123` always go to the same partition, so they are processed sequentially on a single consumer. Adding more partitions increases throughput. When a consumer fails, only that partition's players are affected — the other partitions continue operating.

**Leaderboard update**: The consumer maintains an in-memory sorted set (Redis sorted set is the natural choice). Each score event updates the player's score. The leaderboard query reads directly from Redis — no RabbitMQ involved.

## Tier 3 — Advanced Challenges

### Challenge 5 — Design a Multi-Region Event Bus for a Global SaaS Platform

**Scenario**: A SaaS company operates in three regions (US, EU, APAC). Each region runs its own RabbitMQ cluster. Services in each region publish and consume events locally. Some events must be replicated to other regions — for example, a user creation in the US must propagate to EU and APAC.

**Requirements**:
1. Events published in one region must be available in all other regions within 5 seconds.
2. If a region becomes isolated (network partition), events published in that region must be replayed when connectivity is restored.
3. Events must not be duplicated across regions.
4. Each region must operate independently if cross-region links are down — local processing continues.

**Constraints**:
- Network links between regions have 100-300ms latency and 99.9% uptime.
- Event volume: 5,000 events/second per region, average 2 KB per event.

**Model Answer**:

Use RabbitMQ Federation for exchange-level replication and Shovel for reliable queue-to-queue forwarding.

**Federation setup** (configured on each cluster):

```bash
# On US cluster — replicate global events to EU and APAC
rabbitmqctl set_parameter federation-upstream eu-cluster \
    '{"uri": "amqps://user:pass@eu-cluster:5671",
      "ack-mode": "on-confirm",
      "trust-user-id": false,
      "max-hops": 1}'

rabbitmqctl set_parameter federation-upstream apac-cluster \
    '{"uri": "amqps://user:pass@apac-cluster:5671",
      "ack-mode": "on-confirm",
      "max-hops": 1}'

# Apply federation to the global events exchange
rabbitmqctl set_policy federate-global \
    "^global-events$" \
    '{"federation-upstream-set": "all"}' \
    --apply-to exchanges
```

**Deduplication**:

To prevent duplicate event delivery, each event carries a unique event ID. Consumers use an idempotency store to detect duplicates:

```python
def process_event(ch, method, properties, body):
    event = json.loads(body)
    event_id = event['event_id']

    # Check idempotency store (Redis, DynamoDB)
    if idempotency_store.exists(event_id):
        ch.basic_ack(delivery_tag=method.delivery_tag)
        return  # Already processed

    # Process the event
    handle_event(event)

    # Record event as processed
    idempotency_store.record(event_id, ttl=86400)  # 24-hour window
    ch.basic_ack(delivery_tag=method.delivery_tag)
```

**Cross-region reliability**:

- `ack-mode: on-confirm` ensures the upstream cluster waits for the downstream cluster to confirm receipt before removing the message.
- If the link is down, messages queue up in the upstream cluster. On reconnection, they drain in order.
- The `max-hops: 1` setting prevents infinite replication loops (a message replicated from US to EU is not re-replicated back to US).

**Independent operation**: Each cluster has a `local-events` exchange for events that stay within the region and a `global-events` exchange for events that need cross-region replication. If the federation link is down, local processing continues on the `local-events` exchange. Global events pile up in the `global-events` queue and drain when the link recovers.

### Challenge 6 — Diagnose and Fix a Production Incident

**Scenario**: At 14:30 UTC, monitoring alerts fire:

- Queue depth for `order-processing` is 250,000 messages (normal: 5,000).
- Consumer count for this queue dropped from 8 to 1.
- CPU on all RabbitMQ nodes is at 95%.
- Publisher confirms latency increased from 2ms to 5 seconds.
- Some publishers are timing out.

**Your task**: Diagnose the root cause and design a recovery plan. Then make architectural recommendations to prevent recurrence.

**Model Answer**:

**Diagnosis**:

The alert data reveals a cascading failure:

1. **Trigger**: 7 of 8 `order-processing` consumers crashed. This could be a deployment, an unhandled exception in the consumer code, or a resource exhaustion on the consumer machines.

2. **Queue growth**: With 1 consumer processing messages at approximately 10 msg/s (prefetch=1) and publishers producing at approximately 500 msg/s, the queue grows by 490 messages/second. After 500 seconds (approximately 8 minutes), the queue reaches 250,000 messages.

3. **CPU spike**: The queue process is managing 250,000 messages — segment file index grows, compaction runs, and the single Erlang process maxes out one CPU core. Other Erlang scheduler threads also spike because of inter-process communication overhead.

4. **Latency: Publisher confirms slow down because the queue process is saturated — it cannot process new enqueue requests as fast as they arrive. The TCP socket buffer fills, and `basic_publish` blocks.

5. **Publisher timeouts**: If publishers have a 30-second timeout on `wait_for_confirmation()`, they fire alerts and may crash or skip publishing.

**Recovery plan**:

1. **Stop the bleed**: Pause the publishers or throttle them. If possible, set the queue to reject new publishes:
   ```bash
   rabbitmqctl set_policy block-overflow \
     "^order-processing$" '{"overflow": "reject-publish"}' \
     --apply-to queues
   ```

2. **Restore consumers**: Investigate why 7 of 8 consumers crashed. Check consumer logs for `OutOfMemoryError`, `ConnectionResetError`, or similar. Restart the consumer processes.

3. **Monitor queue drain**: With 8 consumers at 10 msg/s each, the queue drains at 80 msg/s. 250,000 messages / 80 msg/s = 3,125 seconds = 52 minutes. This is acceptable if messages are not time-sensitive.

4. **If queue must drain faster**: Temporarily increase consumer instances to 20. Or, if consumer processing is the bottleneck and messages are cumulative (not time-sensitive), purge old messages:
   ```bash
   # Remove all messages older than 30 minutes
   # (Requires plugin: rabbitmq_consistent_hash_exchange or manual implementation)
   ```
   Or use the management UI to delete old messages.

5. **After recovery**: Investigate the root cause. If consumers crashed due to a bug, fix and redeploy. If they crashed due to resource exhaustion, increase memory/CPU on consumer machines.

**Architectural recommendations**:

1. **Set queue length limits**: `x-max-length: 100000` prevents the queue from growing beyond 100,000 messages. At that point, RabbitMQ drops old messages or rejects new publishes rather than letting the queue grow to 250,000.

2. **Dead-letter configuration**: Route overflow or rejected messages to a DLQ instead of losing them.

3. **Consumer health monitoring**: Alert when consumer count drops below the expected minimum (e.g., 6 out of 8 consumers connected). Do not wait for queue depth alerts — they lag 8+ minutes behind the actual problem.

4. **Automatic consumer recovery**: Implement connection recovery with exponential backoff in the consumer code:
   ```python
   def connect_with_retry():
       for attempt in range(5):
           try:
               params = pika.ConnectionParameters(
                   host='rabbitmq',
                   connection_attempts=3,
                   retry_delay=5
               )
               return pika.BlockingConnection(params)
           except Exception:
               wait = 2 ** attempt  # exponential backoff
               time.sleep(wait)
       raise Exception("Failed to connect after 5 attempts")
   ```

5. **Publisher safeguards**: Publishers should use a timeout on confirms and implement circuit breaker — if confirms take longer than 10 seconds, stop publishing and alert:
   ```python
   if not channel.wait_for_confirmation(timeout=10):
       alert("Publisher confirm timeout — possible broker failure")
       circuit_breaker.open()
   ```

6. **Prefetch optimization**: With 8 consumers, set `prefetch_count=10` to allow each consumer to batch process messages. This increases throughput from 10 msg/s per consumer to 30-50 msg/s per consumer, reducing drain time from 52 minutes to 10-15 minutes.
