## Question Types

Redis interview questions fall into four categories. The category determines the depth and scope of the response expected.

### Category 1: Conceptual Questions

**What they test**: Your understanding of Redis fundamentals. These are usually warm-up questions in the first 5 minutes.

**Examples**:
- "What is Redis and what problems does it solve?"
- "How does Redis differ from Memcached?"
- "What data structures does Redis support?"
- "Explain the difference between RDB and AOF persistence."

**Expected level of detail**: L3-L4: 30-second definition with one concrete example. L5+: 2-minute explanation with trade-offs.

### Category 2: Architecture and Design Questions

**What they test**: Your ability to design systems using Redis and reason about trade-offs.

**Examples**:
- "Design a rate limiter using Redis."
- "How would you cache a social media feed using Redis?"
- "Design a leaderboard for a mobile game."
- "How would you implement distributed locking with Redis?"

**Expected level of detail**: L3-L4: Working solution with one approach. L5+: Multiple approaches with trade-off analysis. L6+: 2-3 approaches covering consistency, performance, failure modes.

### Category 3: Deep Dive Questions

**What they test**: Your understanding of Redis internals, performance characteristics, and configuration.

**Examples**:
- "How does the skip list work in Redis sorted sets?"
- "What happens during a Redis failover with Sentinel?"
- "Explain the Redis Cluster hash slot allocation."
- "How does Redis handle eviction with allkeys-lru?"

**Expected level of detail**: L3-L4: High-level overview. L5+: Detailed explanation with time/space complexity. L6+: Implementation details, edge cases, and configuration parameters.

### Category 4: Operational and Troubleshooting Questions

**What they test**: Your production experience with Redis.

**Examples**:
- "My Redis instance is using 5GB more RSS than maxmemory. What could cause this?"
- "The replication lag keeps growing and the replica keeps disconnecting. What do you do?"
- "Redis is returning 'OOM command not allowed when used memory > maxmemory'. What went wrong?"

**Expected level of detail**: L4+: Step-by-step debugging approach. L5+: Root cause analysis with specific commands. L6+: Preventative measures and monitoring strategies.

## Level Expectations

### L3 / Entry Level

**Span**: 0-2 years of experience.

**What you should answer**:
- Basic command knowledge: SET, GET, DEL, EXPIRE, LPUSH, RPOP
- Understand the difference between Redis and Memcached
- Know the common data structures (string, hash, list, set, sorted set)
- Basic TTL and eviction concepts
- Simple caching pattern (cache-aside)

**Acceptable response**: A correct solution using Redis commands. The interviewer will guide you on trade-offs.
- "I would use Redis SET to cache the database query result with a TTL of 3600 seconds."

**Expected depth**: Surface-level understanding. Less depth per concept, but breadth across fundamentals.

### L4 / Mid Level

**Span**: 2-5 years of experience.

**What you should answer**:
- All L3 answers, plus:
- Pipeline vs. transaction differences
- Lua scripting for atomic operations
- Sentinel high-availability concepts
- Replication lag and consistency models
- Eviction policies and when to use each
- Backup and recovery strategies

**Acceptable response**: A working solution with one trade-off explicitly mentioned.
- "I would use a sorted set for the leaderboard with ZADD for updates and ZREVRANGE for queries. The trade-off is memory: each entry costs ~120 bytes, so for 10M users, expect ~1.2GB."

**Expected depth**: Solid understanding with production awareness. Can reason about performance implications.

### L5 / Senior Level

**Span**: 5-8 years of experience.

**What you should answer**:
- All L4 answers, plus:
- Redis Cluster internals (hash slots, resharding, gossip protocol, MOVED/ASK redirects)
- Redlock algorithm and its caveats
- Advanced data structures (streams with consumer groups, HyperLogLog, Bloom filters)
- Performance optimization: pipeline tuning, batch sizing, lazy freeing
- Capacity planning: memory estimation by data type, node sizing, cluster design
- Multi-region replication patterns

**Acceptable response**: Multiple solutions with explicit trade-off analysis and a clear recommendation.
- "For a rate limiter, sorted sets give the most accurate sliding window but cost more memory. For high-throughput APIs, a fixed window counter with INCR and EXPIRE uses under 100 bytes per user and handles 1M+ ops/sec. I would start with the counter approach for cost efficiency and upgrade to sorted sets only if the accuracy requirements demand it."

**Expected depth**: Deep understanding of internals. Can design, deploy, and troubleshoot production Redis at scale.

### L6 / Staff Level

**Span**: 8+ years of experience.

**What you should answer**:
- All L5 answers, plus:
- Custom Redis module development
- Extensive production troubleshooting experience
- Cross-system design (Redis + Kafka + PostgreSQL integration patterns)
- Capacity modeling and cost optimization
- Incident response and post-mortem processes
- Migration strategies between Redis versions or to alternatives

**Acceptable response**: A holistic design covering the entire lifecycle: requirements, solution, trade-offs, failure modes, operational runbook, migration plan, and monitoring strategy.

**Expected depth**: Expert-level knowledge. Can lead an organization's Redis strategy.

### L7 / Principal Level

**Span**: 12+ years of experience.

**What you should answer**:
- All L6 answers, plus:
- Redis source code contributions or deep internals
- Alternative designs: evaluating Redis vs. Dragonfly vs. KeyDB vs. custom solutions
- Organizational scaling: building Redis platforms, self-service patterns, cost allocation
- Industry trends and future directions
- Mentoring and setting standards across the organization

**Acceptable response**: A comprehensive strategy document covering architecture, operations, cost, migration, and growth plan, with clear decision criteria.

### Example: "Design a leaderboard for a multiplayer game"

**R — Requirements**
- 10M players, real-time score updates, top 100 leaderboard
- P99 read latency under 10ms, update latency under 5ms
- Must handle score ties (earliest player wins)
- Weekly reset (leaderboard starts fresh each Monday)

**E — Evaluate**
- Redis sorted set is purpose-built for this: O(log N) updates, O(log N) range queries
- 10M entries x ~120 bytes = ~1.2GB memory — fits in a single node
- ZREVRANGE with WITHSCORES returns top 100 in microseconds
- Memory cost: ~$30/month for a Redis instance with 4GB RAM

**D — Design**
- Key: `leaderboard:game:weekly:2024-W27`
- ZADD to update score, ZINCRBY to increment
- ZREVRANGE 0 99 WITHSCORES for top 100
- ZREVRANK for a player's position
- EXPIREAT 1721347200 (Monday midnight) for auto-cleanup
- Pipelining for batch score updates (100 updates in one round trip)

**I — Identify Trade-offs**
- Memory: 1.2GB for the leaderboard is fine, but if you keep 52 weekly leaderboards (1 year), that is 62GB — use EXPIRE to auto-clean
- Score ties: Redis sorted set uses score as a double. For ties, store score + timestamp as a composite: `score * 100000 + (MAX_TIMESTAMP - timestamp)`. This adds complexity but handles ties correctly.
- Write amplification: Each ZADD is O(log N). At 10M entries, that is ~24 comparisons per update. For 10K concurrent players updating every second, that is 10M ops/hour — a single Redis instance handles this easily.

**S — Summarize**
- "I recommend a Redis sorted set for the leaderboard. It gives O(log N) updates and O(log N) range queries with minimal memory (1.2GB for 10M players). The main trade-off is handling score ties, which requires a composite score encoding. I would use key namespaced by week with EXPIRE for automatic cleanup."

## Phrase Bank

Use these phrases to demonstrate confidence and depth during your interview.

### Opening Phrases

- "Let me start by clarifying the requirements before proposing a solution."
- "Redis is a strong candidate here because of its in-memory performance and data structure flexibility."
- "I will evaluate three approaches, each with different trade-offs."
- "Let me sketch the data model first, then discuss the architecture."

### Data Modeling Phrases

- "I would use a hash for this because it maps closely to the object's field-value structure and supports partial updates with HSET."
- "A sorted set gives us the ordering we need with O(log N) for both insert and range query."
- "I would namespace the keys as `feature:entity:id:field` for consistent discoverability."
- "The TTL should align with the data's natural freshness window — for a news feed, 5 minutes; for user profiles, 1 hour."
- "For multi-key operations, I would use hash tags `{user:123}` to co-locate related keys on the same cluster node."

### Architecture Phrases

- "For availability, I would use Redis Cluster with 3 primaries and 3 replicas across 3 availability zones."
- "Sentinel provides automatic failover, but there is a window of write loss during promotion (async replication lag)."
- "Reads can be served from replicas for scaling, but clients must tolerate eventual consistency."
- "I would use a smart client that handles MOVED and ASK redirects from the Cluster topology."
- "The replication backlog should be sized to absorb the maximum expected disconnect-and-reconnect window."

### Trade-Off Phrases

- "The fundamental trade-off in Redis is memory cost versus speed. RAM is expensive, but sub-millisecond latency is hard to achieve with disk-based stores."
- "Consistency in Redis is eventual by default on replicas. For stronger guarantees, you can use WAIT, but it increases write latency."
- "RDB snapshots are faster to load on restart but lose more data. AOF provides better durability but is slower to load and larger on disk."
- "Eviction policies trade cache hit ratio for memory predictability. allkeys-lfu gives the best hit ratio for skewed workloads but uses more CPU for frequency tracking."
- "Cluster adds operational complexity (resharding, node management) but enables scaling beyond a single node's memory limit."

### Operational Phrases

- "I would monitor evicted_keys, keyspace_hits/keyspace_misses, and master_repl_offset advancements as key health indicators."
- "The first thing I check on a slow Redis instance is the slow log: SLOWLOG GET 50."
- "Memory fragmentation over 1.5 means I should schedule a node restart or enable activedefrag."
- "For replication issues, I check the replication backlog size and ensure it is larger than the maximum burst size."
- "I always separate cache and persistent data stores — different instances with different eviction policies."

### Closing Phrases

- "To summarize, Redis solves this problem well because of [specific data structure] and [performance characteristic]."
- "The main risks are [failure mode 1] and [failure mode 2]. I would mitigate with [specific configuration]."
- "The next step would be to prototype the data model and run a load test at 10x target throughput to identify bottlenecks."

## Red Flags

These are mistakes that signal inexperience with Redis. Avoid them in your interview answers.

### Red Flag 1: Using KEYS in Production

**What it sounds like**: "I would search for all matching keys using KEYS user:*."

**Why it is a red flag**: KEYS blocks the event loop for millions of keys. It can take seconds on a large database, blocking all other operations.

**What to say instead**: "I would use SCAN 0 MATCH user:* to iterate without blocking, or maintain an index set of user keys."

### Red Flag 2: No TTL on Cache Keys

**What it sounds like**: "I will cache the database results in Redis and they persist until I explicitly delete them."

**Why it is a red flag**: Data grows unbounded, cache becomes stale, and the application has no invalidation strategy.

**What to say instead**: "I will set a TTL of 3600 seconds on the cache entry. For critical data, I also invalidate the cache key when the underlying data changes."

### Red Flag 3: No Eviction Policy

**What it sounds like**: "I will let Redis manage memory automatically."

**Why it is a red flag**: Unbounded memory growth leads to OOM kills.

**What to say instead**: "I will set maxmemory to 80% of available RAM and use allkeys-lfu for the cache instance."

### Red Flag 4: Redis as Primary Database

**What it sounds like**: "I will store all application data in Redis because it is fast."

**Why it is a red flag**: Redis has limited persistence guarantees, no rollback, no complex querying, and memory is expensive. It is not designed to replace a database.

**What to say instead**: "Redis will be the cache layer in front of PostgreSQL. The database is the source of truth; Redis improves read performance."

### Red Flag 5: Ignoring Persistence in Production

**What it sounds like**: "Redis is just a cache, so I don't need persistence."

**Why it is a red flag**: If Redis crashes, the cache is cold. All database traffic hits at once (cache stampede), potentially taking down the database.

**What to say instead**: "Even for cache, I enable AOF with appendfsync everysec to minimize data loss on restart and avoid a full cold-cache stampede."

### Red Flag 6: No Replication / Single Point of Failure

**What it sounds like**: "I will run a single Redis instance for the application."

**Why it is a red flag**: Any failure — hardware, network, process crash — causes complete service outage.

**What to say instead**: "I will deploy Redis with at least one replica and Sentinel for automatic failover."

### Red Flag 7: Storing Large Values

**What it sounds like**: "I will store the rendered HTML page (500KB) in Redis."

**Why it is a red flag**: Large values consume RAM inefficiently and block the event loop during retrieval and transmission.

**What to say instead**: "For large objects, I store them in object storage (S3) and keep only the URL or metadata in Redis."

### Red Flag 8: Ignoring Network Round Trips

**What it sounds like**: "I will make individual Redis calls for each item in a loop."

**Why it is a red flag**: N Redis calls = N network round trips. For 1000 items at 0.1ms RTT, that is 100ms of pure network latency.

**What to say instead**: "I will use pipelining or MGET to batch operations into a single round trip."

## Practice Drills

### Drill 1: The 30-Second Definition

Practice answering these in under 30 seconds each:

- "What is Redis?"
- "What is a Redis sorted set?"
- "What is the difference between RDB and AOF?"
- "What is Redis Cluster?"
- "What is cache-aside?"

### Drill 2: The Three-Trade-Off Exercise

For each use case below, identify three trade-offs in using Redis:

1. Storing user sessions
2. Implementing a rate limiter
3. Building a real-time leaderboard
4. Running a task queue
5. Serving a pub/sub event bus

**Example for rate limiter**:
1. **Accuracy vs. memory**: Sliding window with sorted sets uses more memory but is more accurate than fixed window counters.
2. **Atomicity vs. complexity**: A Lua script provides atomic multi-key operations but requires script management and debugging.
3. **Distributed state**: Rate limiter state must be shared across all application instances. Redis provides this, but a Redis failure means no rate limiting until recovery.

### Drill 3: The Failure Mode Walkthrough

For each scenario, walk through the failure mode and recovery steps:

1. Redis primary crashes. What happens? How does Sentinel recover? What data is lost?
2. Redis OOM is killed. What configuration would have prevented this? How do you recover?
3. Replication lag is growing unbounded. What is the root cause? What configuration changes fix it?
4. A Redis Cluster node fails. How does the cluster rebalance? What happens to the hash slots?

### Drill 4: Whiteboard a System

Design and sketch on a whiteboard (or in your head):

1. **Session store**: 100M users, read-heavy, under 5ms p99
2. **News feed cache**: 50M daily active users, each user follows 200 accounts
3. **Real-time analytics counter**: 100K events/sec, 1-second granularity, 30-day retention
4. **Distributed lock service**: 10K locks/sec, 99.999% correctness, multi-datacenter
5. **Gaming leaderboard**: 50M players, real-time updates, top 1000 view, per-game, per-season

### Drill 5: Configuration for a New Service

Given a 32GB server with 4 vCPUs, write the redis.conf for:

1. A general-purpose cache (no persistence needed, eviction on)
2. A session store (persistence needed, high availability)
3. A task queue (persistence needed, no eviction)
4. A rate limiter (no persistence, small data, high throughput)
5. A primary database front (cache-aside, RDB + AOF, LRU eviction)

**Example answer for #1 (cache)**:
```
maxmemory 24576mb
maxmemory-policy allkeys-lfu
maxmemory-samples 10
save ""
appendonly no
lazyfree-lazy-eviction yes
activedefrag yes
timeout 300
tcp-keepalive 60
slowlog-log-slower-than 5000
rename-command KEYS ""
rename-command FLUSHALL ""
rename-command FLUSHDB ""
```
