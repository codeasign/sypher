## The Problem Story

Your startup is growing fast. Last month you had 10,000 users; this month you have 50,000. Your monolithic Rails app is struggling.

You reach for Memcached to cache database queries. It works — your page load times drop from 800ms to 200ms. But soon you need more:

- **Session storage**: you need to store user sessions across multiple web servers, with expiration
- **Rate limiting**: you need to track API calls per user per minute
- **A job queue**: background email processing needs a reliable queue
- **A leaderboard**: your gamification feature needs sorted scores
- **Real-time notifications**: users need to see updates without polling the database

Today you are running **three separate systems**: Memcached for caching, PostgreSQL for session storage (with a cleanup cron job that keeps failing), and RabbitMQ for job queues. Each system has its own operational overhead: monitoring, backups, patching, connection pools.

Your infrastructure team is drowning. Every new data requirement means standing up another service. The operational complexity is growing faster than the feature velocity.

What you need is a single tool that handles all these workloads with sub-millisecond latency, minimal operational overhead, and flexible data structures.

## The Insight

**Redis is an in-memory data structure store that can be used as a cache, database, message broker, and streaming engine — all with sub-millisecond latency.**

Think of Redis as a **Swiss Army knife for data**. Instead of carrying separate tools for every job (a screwdriver for caching, scissors for queues, a corkscrew for sessions), you carry one versatile tool that handles all of them. It fits in your pocket (runs in under 5MB of RAM for small workloads) and handles everything from simple key-value lookups to complex sorted-set operations to publish-subscribe messaging.

Redis stores data primarily in memory, which gives it blazing speed. But it also offers optional persistence to disk for durability. This hybrid model means you can tune it for pure speed (cache-only, no persistence) or near-database durability (AOF persistence with fsync).

The name "Redis" stands for **Remote Dictionary Server**. It started in 2009 when Salvatore Sanfilippo wanted to improve the performance of a real-time web log analyzer for his Italian startup. He built a simple in-memory key-value store with a few extra data structures. Today, Redis is the most popular key-value database in the world, used by Twitter, GitHub, Stack Overflow, Discord, and tens of thousands of other companies.

## Core Vocabulary

### Redis

**Definition**: An open-source, in-memory data structure store used as a cache, database, message broker, and streaming engine.

**Why it matters**: Redis compresses four infrastructure roles into one service, reducing operational complexity. You deploy one Redis cluster instead of a cache server, a session store, a message queue, and a stream processor.

### In-Memory

**Definition**: Data is stored primarily in RAM rather than on disk, enabling sub-millisecond read and write latencies.

**Why it matters**: RAM is 100,000x faster than SSD for random access. This is the fundamental reason Redis can serve 100,000 to 1,000,000 operations per second on a single node.

### RDB (Redis Database File)

**Definition**: A point-in-time snapshot of the dataset, written to disk at configurable intervals. The default persistence mechanism.

**Why it matters**: RDB snapshots provide fast recovery for restart scenarios. You lose at most N minutes of writes (configurable), but the trade-off is minimal performance impact during normal operation since snapshots are forked to a child process.

### AOF (Append-Only File)

**Definition**: A write-ahead log that records every write operation received by the server. Can be configured to fsync every second or every write.

**Why it matters**: AOF provides stronger durability guarantees than RDB. With `appendfsync always`, you lose at most one write on a crash. The trade-off is higher I/O overhead and potentially larger file sizes.

### Replication

**Definition**: Data is copied from a primary Redis instance to one or more replica instances. Replicas serve read queries and provide failover targets.

**Why it matters**: Replication enables read scaling — you distribute read traffic across multiple replicas — and provides high availability. If the primary fails, a replica can be promoted to take over.

### Sentinel

**Definition**: A distributed monitoring and failover system for Redis. A set of Sentinel processes watch primary and replica instances and automatically promote a replica if the primary fails.

**Why it matters**: Sentinel provides automatic failover without manual intervention. You get 99.99% availability with a properly configured Sentinel deployment, compared to manual recovery that can take 10-30 minutes.

### Redis Cluster

**Definition**: Redis's native sharding solution. Data is automatically partitioned across multiple nodes using a 16384-slot hash slot scheme.

**Why it matters**: Redis Cluster lets you scale beyond a single node's memory limit. A cluster with 6 nodes (3 primaries, 3 replicas) can handle 3x the data of a single node while maintaining availability during node failures.

### Data Structures (String, Hash, List, Set, Sorted Set, Stream)

**Definition**: Redis provides specialized data structures, not just simple key-value pairs. Strings hold text or binary, hashes hold field-value pairs, lists hold ordered sequences, sets hold unique members, sorted sets maintain ordered unique members, and streams provide an append-only log.

**Why it matters**: These structures let you model complex data relationships without serializing to JSON or performing application-level sorting. A leaderboard is a single ZADD command, not a 50-line SQL query with ORDER BY and LIMIT.

### Pub/Sub

**Definition**: A messaging pattern where publishers send messages to channels, and subscribers receive messages on channels they have subscribed to. No message persistence — if you are not subscribed, you miss the message.

**Why it matters**: Pub/Sub enables real-time notification patterns with under 1ms latency. You can notify all websocket servers of a user event, broadcast system alerts, or coordinate worker processes.

### Lua Scripting

**Definition**: Redis lets you execute Lua scripts atomically on the server. The entire script runs without interruption from other commands.

**Why it matters**: Lua scripting lets you implement complex multi-key operations atomically without client-side locking. A transfer operation (debit account A, credit account B) runs as a single atomic unit, preventing race conditions.

### TTL (Time To Live)

**Definition**: An expiration time set on a key. Redis automatically deletes the key when the TTL expires.

**Why it matters**: TTL is essential for cache management. You set a TTL of 3600 seconds on a cached database query result, and Redis automatically cleans it up — no cron jobs needed. This prevents stale data and manages memory usage.

### Eviction Policy

**Definition**: The strategy Redis uses to free memory when the `maxmemory` limit is reached. Options include LRU (least recently used), LFU (least frequently used), TTL (shortest TTL first), and noeviction (return errors on writes).

**Why it matters**: Without an eviction policy, Redis fills up and either crashes (OOM kill by the OS) or rejects writes. Choosing the right policy ensures your cache stays warm and useful under memory pressure.

### Transaction

**Definition**: A group of commands executed sequentially and atomically using MULTI/EXEC. Unlike SQL databases, Redis transactions do not support rollback.

**Why it matters**: Transactions ensure atomicity for multi-step operations. When you increment a counter and set a flag in the same transaction, either both happen or neither does.

### Pipeline

**Definition**: A mechanism to send multiple commands to Redis without waiting for individual responses. Commands are buffered and sent in one network round trip.

**Why it matters**: Pipelining reduces network overhead. A loop of 1000 SET commands without pipelining takes 1000 round trips (~100ms on a local network). With pipelining, it takes one round trip (~0.1ms). This is a 1000x improvement.

### GEO

**Definition**: Geospatial indexing commands for storing and querying latitude/longitude coordinates. Supports radius queries, distance calculations, and bounding-box queries.

**Why it matters**: GEO enables location-based features without a separate spatial database. You can find nearby restaurants, track delivery drivers, or compute distances between points with a single command.

## How It Actually Works

### Step 1: Client Connection

A client connects to Redis on port 6379 (default). Redis uses an event-driven, single-threaded reactor pattern. The I/O multiplexer (epoll on Linux, kqueue on macOS, IOCP on Windows via compatibility layer) monitors all client sockets for readability and writability.

### Step 2: Command Parsing

The event loop detects that a client socket has data available. It reads the raw bytes from the socket buffer and parses them according to the **RESP protocol** (Redis Serialization Protocol).

RESP is a simple text-based protocol:
- `*3\r\n` — 3 elements in the array
- `$3\r\nSET\r\n` — bulk string "SET"
- `$4\r\nname\r\n` — bulk string "name"
- `$5\r\nAlice\r\n` — bulk string "Alice"

The parser constructs an internal command structure with the command name and arguments.

### Step 3: Command Dispatch

The parsed command is dispatched to the appropriate handler function. Redis has a lookup table mapping command strings like "SET", "GET", "LPUSH", "ZADD" to their C handler functions.

### Step 4: Data Structure Operation

The handler operates directly on the in-memory data structure. For example, a `GET` on a string key looks up the key in the global dictionary, finds the `redisObject`, and reads the string value.

### Step 5: Response Construction

The handler constructs a RESP response. For a `GET` returning "Alice", the response is `$5\r\nAlice\r\n`. This response is written to the client's output buffer.

### Step 6: Response Transmission

The I/O multiplexer detects that the client socket is writable and writes the buffered response. The client receives the response and parses it.

### Step 7: Optional Persistence (RDB)

Periodically (by default every 900 seconds if at least 1 key changed, every 300 seconds if at least 10 keys, every 60 seconds if at least 10000 keys), Redis forks a child process. The child writes a point-in-time snapshot of the entire dataset to an RDB file.

### Step 9: AOF Rewrite

Over time, the AOF file grows. Redis can rewrite it in the background: it forks a child process that reads the current data state and writes a minimal set of commands to reconstruct it. For example, 1000 `INCR counter` commands become a single `SET counter 1000`.

### Step 10: Replication

When replication is configured, the primary sends a snapshot (RDB) to each connected replica (full sync). After that, it streams every write command to the replicas (partial sync / command propagation).

### Step 11: Eviction

When the `maxmemory` limit is reached, Redis applies the configured eviction policy. Under `allkeys-lru`, it samples a few keys, identifies the least recently used, and evicts them. The sampling approach avoids scanning the entire key space.

### Step 12: Expiration

Redis tracks TTLs for keys with expiration. It uses a **lazy** strategy (check expiration on access and delete if expired) combined with an **active** strategy (a background timer samples 20 keys every 100ms and deletes expired ones). This hybrid approach avoids the overhead of a dedicated expiration thread while keeping memory free.

### Step 13: Multi-Key Operations with Transactions

When a client sends `MULTI`, Redis enters a transaction state. Subsequent commands are queued (not executed). When `EXEC` arrives, Redis executes all queued commands sequentially without interruption. The entire block is atomic.

### Step 14: Lua Script Evaluation

When a client sends `EVAL` with a Lua script, Redis:
1. Creates a Lua sandbox environment
2. Loads the script
3. Executes it atomically (blocking all other operations)
4. Returns the result

Lua scripts run with the same atomicity guarantees as transactions but support complex logic (conditionals, loops, math).

## Mental Model Check

**Question 1**: You run `SET user:1:name "Alice"` followed by `GET user:1:name`. What happens inside Redis?

**Answer**: The SET command parses the key `user:1:name` and value `Alice`, hashes the key to find a bucket in the global dictionary, stores a redisObject with the string "Alice" (encoded as an SDS — Simple Dynamic String), and returns "+OK\r\n". The GET command hashes the same key, locates the same redisObject, reads the SDS, marshals it as a RESP bulk string `$5\r\nAlice\r\n`, writes it to the client output buffer, and returns.

**Question 2**: What happens if you set a TTL of 60 seconds on a key and the server crashes after 30 seconds?

**Answer**: The TTL is stored in memory alongside the key. On a crash with no persistence (both RDB and AOF disabled), the key and its TTL are lost entirely — the server starts with an empty dataset. With RDB persistence and the default save interval (900s), the key might not be in the snapshot, so it would also be lost. With AOF persistence and `appendfsync everysec`, the SET command (including the EXPIRE) would be in the AOF log, and on restart Redis replays the AOF, recreating the key with its remaining TTL — but 30 seconds have passed, so the TTL would be about 30 seconds from the replay time.

**Question 3**: You have 50GB of data but only 10GB of RAM. Can you use Redis?

**Answer**: No — Redis stores the full dataset in memory. If your working dataset exceeds available RAM, the OS will start swapping, destroying Redis's performance. The solution is either: (a) use Redis Cluster to shard the data across multiple nodes (e.g., 5 nodes with 10GB each), (b) use a caching-only pattern with an eviction policy so Redis only holds hot data, or (c) choose a different technology (like a disk-based key-value store) for the full dataset.

**Question 4**: If you send 10,000 SET commands one by one, vs. pipelining them all at once, what's the difference?

**Answer**: Without pipelining, each command incurs one network round trip (RTT). On a local network with 0.1ms RTT, 10,000 commands take ~1 second, plus server processing time (~0.001ms each). With pipelining, all 10,000 commands are sent in one burst — one network RTT (~0.1ms) plus 10,000 server operations (~10ms). Pipelining gives roughly a 100x speedup. The trade-off is that pipelining requires buffering responses on both client and server, consuming memory proportional to the batch size.

**Question 5**: Two clients simultaneously increment the same key with `INCR counter`. Can there be a race condition?

**Answer**: No — `INCR` is atomic in Redis. Since Redis is single-threaded, each command runs to completion before the next one starts. The two `INCR` commands are serialized: the first reads counter (5), increments (6), and writes (6); the second reads counter (6), increments (7), and writes (7). The result is always 7, never lost. However, if you used a non-atomic pattern like `GET` → application increment → `SET`, a race condition would occur: both clients read 5, both set 6, and one increment is lost.
