Redis's internal architecture is a study in elegant simplicity. The entire server runs on a single-threaded event loop with an I/O multiplexer at its core.

### The I/O Multiplexer

Redis does not use threads per connection. Instead, it uses a single-threaded reactor pattern powered by the operating system's I/O multiplexing facilities:

- **Linux**: `epoll_create`, `epoll_ctl`, `epoll_wait` вҖ” scales to hundreds of thousands of sockets
- **macOS/BSD**: `kqueue` вҖ” comparable to epoll
- **Fallback**: `select` / `poll` вҖ” works everywhere, but limits to FD_SETSIZE (typically 1024)

The multiplexer registers interest in read/write events on each client socket. When a socket becomes readable (data arrived), or writable (buffer space available), the event loop wakes up and processes it.

This design means one thread handles all connections, eliminating context-switch overhead and locking complexity. The downside: any slow operation blocks every client.

### The RESP Protocol

Redis clients communicate using RESP (Redis Serialization Protocol), a simple, human-readable protocol:

```
*3           вҶҗ Array of 3 elements
$3           вҶҗ Bulk string of length 3
SET          вҶҗ Command name
$4           вҶҗ Bulk string of length 4
name         вҶҗ Key
$5           вҶҗ Bulk string of length 5
Alice        вҶҗ Value
```

RESP types:
- **Simple String**: `+OK\r\n` вҖ” non-binary, for status replies
- **Error**: `-ERR unknown command\r\n` вҖ” error messages
- **Integer**: `:1\r\n` вҖ” for counters, lengths
- **Bulk String**: `$5\r\nhello\r\n` or `$-1\r\n` for nil вҖ” binary-safe
- **Array**: `*2\r\n$3\r\nfoo\r\n$3\r\nbar\r\n` вҖ” nested collections
- **Null**: `_\r\n` вҖ” RESP3 addition
- **Double**: `,1.5\r\n` вҖ” RESP3 addition
- **Big Number**: `(123456789\r\n` вҖ” RESP3 addition

RESP3 (introduced in Redis 6) adds typed responses that eliminate client-side parsing for types like maps, sets, and timestamps.

### Memory Management

Redis has its own allocator (jemalloc by default) and tracks memory at multiple levels:

- `used_memory`: total bytes allocated by Redis (via jemalloc)
- `used_memory_rss`: resident set size as reported by the OS
- `used_memory_peak`: peak memory usage since startup
- `mem_fragmentation_ratio`: RSS / used_memory вҖ” ratio above 1.5 indicates fragmentation

The `INFO memory` command exposes all of these. This is critical for capacity planning вҖ” a fragmentation ratio of 2.0 means you are using twice the RSS of your actual data.

## Data Structures

### String вҖ” SDS (Simple Dynamic String)

Redis strings are not C strings. They use a custom structure called SDS that stores length and capacity alongside the character data.

```
struct sdshdr {
    int len;     // Current string length
    int free;    // Available free space
    char buf[];  // Flexible array member
};
```

**Benefits over C strings**:
- O(1) length retrieval (no strlen loop)
- Binary-safe (can store null bytes, images, serialized objects)
- Prevents buffer overflows (automatic resizing)
- Pre-allocates extra space to amortize append costs

**Time complexity**: O(1) for get/set, O(N) for append (amortized O(1))

**Encoding**: Strings under 44 bytes use **embstr** (embedded in the redisObject, single allocation). Strings 44 bytes and over use **raw** (separate allocation for SDS and redisObject). The 44-byte threshold arises from the 64-byte jemalloc allocation bucket minus redisObject (16 bytes) and SDS header overhead.

**Memory**: `sizeof(robj) + sizeof(sdshdr) + len + 1` for raw strings; 64 bytes total for embstr strings.

### Hash вҖ” ziplist / hashtable

Redis hashes use two encodings depending on size:

**Ziplist (small hashes)**:
A contiguous memory block storing field-value pairs as a sequence of entries. Used when both the number of fields and the size of each field value are small (default: 512 entries, 64 bytes per entry).

- Layout: `[zlbytes][zltail][zllen][entry1][entry2]...[entryN][zlend]`
- Each entry: `[prev_len][encoding][data]`
- O(N) operations since it requires linear scan
- Extremely memory-efficient вҖ” no pointers, no hash table overhead

**Hashtable (large hashes)**:
Two parallel hash tables (for incremental rehashing). Each entry is a dictEntry struct (24 bytes) plus the key/value redisObjects and their SDS payloads.

- Load factor: 1:1 by default, resizes at 5:1 (grow) or 0.1:1 (shrink)
- O(1) average operations
- Memory overhead: ~72 bytes per entry just for pointers and metadata

### List вҖ” Quicklist

Redis lists use the **quicklist** data structure, which is a doubly-linked list of ziplist nodes. This hybrid combines the memory efficiency of ziplists with the insertion/deletion performance of linked lists.

```
typedef struct quicklist {
    quicklistNode *head;   // First node
    quicklistNode *tail;   // Last node
    unsigned long count;   // Total entries
    unsigned int len;      // Number of nodes
    int fill;              // Fill factor (max ziplist size)
    int compress;          // Compression depth
} quicklist;
```

Each `quicklistNode` holds a ziplist of up to ~8192 entries (configurable via `list-max-ziplist-size`). Nodes at the edges can be partially filled to accommodate push/pop operations without rebalancing.

**Time complexity**: O(1) for LPUSH/RPUSH/LPOP/RPOP at ends, O(N) for LINDEX/LSET with index (traverses from nearest end).

**Compression**: The `list-compress-depth` parameter lets you compress interior nodes using LZF compression. With depth=2, the two nodes at each end remain uncompressed for fast access, while the middle nodes are compressed вҖ” a significant memory savings for long lists.

### Set вҖ” intset / hashtable

Redis sets use two encodings:

**Intset (all integer members)**: A sorted array of integers (16-bit, 32-bit, or 64-bit). Binary search for lookups. Upgraded automatically when a larger integer is added.

- Memory: `sizeof(intset) + (sizeof(type) * card)` вҖ” extremely compact
- O(log N) for lookups due to binary search

**Hashtable (any non-integer member)**: Standard dict with NULL values. O(1) average lookup, membership test, add, and remove.

**Transition**: When the first non-integer member is added, or the set exceeds `set-max-intset-entries` (default 512), it converts from intset to hashtable.

### Sorted Set вҖ” Skiplist + Hash Table

Redis sorted sets use a **dual structure**: a skip list for ordered iteration and a hash table for direct member-score lookups.

```
typedef struct zset {
    dict *dict;           // member вҶ’ score mapping
    zskiplist *zsl;       // ordered structure
} zset;
```

**Skip list**:
- A probabilistic data structure with multiple levels of linked lists
- Each node has a random height (average ~1.33 levels for a 32-level max)
- O(log N) for insert, delete, and search by rank or score
- Span values at each level enable O(log N) rank computation

### Stream вҖ” Radix Tree (Rax)

Redis streams use a **radix tree** (compressed prefix tree, implemented as `rax`) for indexing entries by their millisecond-precision timestamps (entry IDs).

- Each entry ID: `<timestamp>-<sequence_number>` (e.g., `1704067200000-0`)
- The radix tree compresses common prefixes of entry IDs
- Enables O(log N) range queries, O(1) append, O(1) trim
- Consumer groups track their own state using a separate radix tree for pending entries (PEL)

### HyperLogLog

A probabilistic data structure for approximate cardinality counting. Uses 2^14 (16384) registers, each 6 bits, totaling 12KB.

- **Standard error**: 0.81%
- **Mergeable**: multiple HLLs can be merged with UNION
- **O(1)** for PFADD, PFCOUNT, PFMERGE

### Bitmap

Strings operated on at the bit level using `SETBIT`/`GETBIT`/`BITCOUNT`/`BITOP`.

- A 512MB string can represent ~4.3 billion bits
- Offsets are O(1) for individual bit operations
- BITCOUNT uses a variable-precision SWAR algorithm for high-performance population counting

### GEO вҖ” Geohash

Geospatial indexes are implemented on top of sorted sets. Coordinates are encoded as a 52-bit geohash integer and stored as the score in a zset.

- **Encode**: latitude/longitude вҶ’ 52-bit geohash integer via interleaving
- **Decode**: geohash integer вҶ’ bounding box вҶ’ matching members
- **Query**: GEORADIUS computes the geohash range covering the radius and scans the zset

### Rehashing

Redis uses incremental rehashing to avoid blocking the event loop during hash table resizes.

```
Periodic rehashing:
  1. Redis has two hash tables: ht[0] (active) and ht[1] (resize target)
  2. When ht[0] load factor exceeds threshold, allocate ht[1] at 2x size
  3. Time event triggers rehash step: move up to N buckets (default 100) + process current bucket's chain
  4. Each command also does a "rehash step" while it has ht[0] dict access
  5. When ht[0] is empty, swap pointers and free ht[1]
```

This incremental approach means a 10-million-key rehash spreads across thousands of command executions rather than blocking for seconds.

### Blocking Operations

Certain Redis operations block the event loop:

- **BRPOP/BLPOP**: Block until an element is available in a list. Internally, Redis adds the client to a blocking state and re-schedules it when data arrives.
- **BZPOPMIN/BZPOPMAX**: Same for sorted sets.
- **XREAD BLOCK**: Block until new stream data arrives. Uses similar mechanism.
- **WAIT**: Block until N replicas acknowledge the write.

Blocking operations do NOT block the entire server вҖ” only the requesting client. Redis achieves this by tracking blocked clients in a special list and waking them when the relevant key gets data pushed to it.

## Consistency and State

### Async Replication Lag

In standard Redis replication, writes to the primary are asynchronous. This means:

```
Client вҶ’ SET counter 100 вҶ’ Primary (immediately OK)
                           в””в”Җв”Җ (50ms later) вҶ’ Replica (eventually)
```

If the primary crashes between acknowledging the write and propagating it to the replica, the write is lost.

`WAIT` command mitigates this: `SET counter 100\nWAIT 1 1000` tells Redis to block until at least 1 replica has acknowledged the write, with a 1-second timeout. This trades latency for durability.

### Failover with Sentinel

When the primary fails, Sentinel performs automatic failover:

1. **Detection**: Multiple Sentinels must agree the primary is unreachable (quorum). Each Sentinel pings the primary every 1 second. If the primary does not respond within `down-after-milliseconds` (default 30 seconds), a Sentinel marks it as `sdown` (subjectively down). If `quorum` Sentinels agree, it becomes `odown` (objectively down).

2. **Election**: Sentinels use a Raft-like consensus protocol to elect a leader for the failover. Each Sentinel can cast one vote per epoch. The first Sentinel to reach quorum votes becomes the leader.

3. **Replica Selection**: The leader selects the best replica: lowest replication offset (most up-to-date), highest priority config, then lowest run ID.

4. **Promotion**: The selected replica gets `SLAVEOF NO ONE`, making it a primary.

5. **Reconfiguration**: All other replicas get `SLAVEOF <new_primary>` to follow the new primary.

6. **Notification**: The new primary's address is propagated to all connected clients via pub/sub or periodic `SENTINEL get-master-addr-by-name` calls.

### Split-Brain in Cluster

In Redis Cluster, nodes use a **gossip protocol** to exchange state. If a network partition splits the cluster, each side may continue operating independently:

- **Minority side**: If less than half of the primary nodes are reachable, the minority side stops accepting writes after `cluster-node-timeout` (default 15 seconds).
- **Majority side**: Continues normal operation. Replicas on the minority side may attempt to connect to primaries on the majority side.

When the partition heals, the minority side's data is discarded (it gets a full resync from the majority). Writes to the minority side during the partition are lost.

### AOF Rewrite Blocking

AOF rewrite forks a child process. While the fork is running, the parent continues serving clients but tracks all modifications in a rewrite buffer. When the child completes, the parent merges the buffer with the child's output.

The fork itself can be slow on large instances (a 10GB Redis instance takes ~100ms to fork on modern hardware), during which the event loop is paused.

### OOM Kill

If Redis exceeds its `maxmemory` and the eviction policy is set to `noeviction`, write commands return errors. If `maxmemory` is not set, Redis can grow until it exhausts system RAM, at which point the kernel's OOM killer terminates the Redis process.

### Single-Threaded Implications

Since Redis is single-threaded, a slow operation blocks all other clients:

| Operation | Impact | Mitigation |
|-----------|--------|------------|
| `KEYS *` | Blocks for seconds on 10M keys | Use `SCAN` with cursor |
| `SORT` on large list | Blocks for hundreds of ms | Use sorted sets instead |
| `MGET` 10K keys | Blocks for ~1ms | Acceptable in most cases |
| Large value (10MB GET) | Blocks for ~5ms (network I/O) | Keep values small |
| AOF fsync with `always` | Blocks for 0.1-2ms per write | Use `everysec` |

| `client-output-buffer-limit pubsub` | 32mb 8mb 60 | Limits for pub/sub clients (slow consumers get disconnected) |

### Advanced Configuration

| Parameter | Default | Description |
|-----------|---------|-------------|
| `lazyfree-lazy-eviction` | no | Use async free (UNLINK) for evicted keys to avoid blocking |
| `lazyfree-lazy-expire` | no | Use async delete for expired keys |
| `replica-lazy-flush` | no | Use async flush during full resync |
| `activedefrag` | no | Enable automatic memory defragmentation |
| `active-defrag-threshold-lower` | 10 | Minimum fragmentation ratio to start defrag |
| `active-defrag-threshold-upper` | 100 | Maximum fragmentation ratio for aggressive defrag |
| `slowlog-log-slower-than` | 10000 | Log commands slower than N microseconds (10ms) |
| `slowlog-max-len` | 128 | Max entries in slow log |

### Tuning Checklist for Production

1. Set `maxmemory` to 80% of available RAM
2. Choose `maxmemory-policy`: `allkeys-lfu` for caching, `volatile-lru` for mixed workloads, `noeviction` for data you cannot lose
3. Enable `appendonly yes` with `appendfsync everysec` for durability
4. Set `requirepass` with a strong random password
5. Rename or disable `FLUSHALL`, `FLUSHDB`, `CONFIG`, `KEYS`: `rename-command KEYS ""`
6. Set `repl-backlog-size` to 10% of maxmemory for large replication windows
7. Enable `activedefrag yes` for workloads with frequent key churn
8. Set `slowlog-log-slower-than 10000` and monitor the slow log
9. Set `timeout 300` to close idle connections
10. Configure `client-output-buffer-limit` appropriately for your workload
