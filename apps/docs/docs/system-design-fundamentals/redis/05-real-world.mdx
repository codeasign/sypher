**Twitter-specific challenges**:
- Fanout on write vs. fanout on read: Twitter uses a hybrid approach. Celebrities with millions of followers do NOT fanout to all followers вҖ” instead, their tweets are fetched at read time and merged with the pre-computed timeline. Redis stores both the pre-computed lists and the celebrity tweet index.
- Edge caching: Redis instances are deployed in multiple datacenters with local caches to reduce cross-datacenter latency.
- Timeline persistence: Tweets are stored in MySQL for durability. Redis is a materialized cache that can be rebuilt from the database.

**GitHub-specific patterns**:
- **Job retry**: Failed jobs go to a `failed` queue for manual retry via a web interface.
- **Queue priorities**: GitHub runs `critical` queues (password reset emails) with more workers than low-priority queues (weekly digest emails).
- **Worker heartbeats**: Workers update a key in Redis every 5 seconds. A monitoring process detects stalled workers (no heartbeat for 60 seconds) and re-queues their in-flight jobs.
- **Job uniqueness**: Uses Redis sets to prevent duplicate jobs. Before enqueuing, Resque checks if the job signature is already present.

## Open Source Alternatives

### Redis (Self-Hosted)

The original. Open source (BSD 3-clause). Constantly evolving with new data types and features. The community edition lacks some enterprise features (active-active geo-replication, auto-tiering) found in Redis Enterprise.

**Best for**: Teams that want full control over configuration, deployment topology, and version upgrades.

### KeyDB

A multi-threaded fork of Redis by Snap Inc. KeyDB replaces the single-threaded event loop with multiple threads, each handling a subset of connections. It is API-compatible with Redis вҖ” you point the same client driver at it and get multi-threaded performance.

**Performance difference**:
- Single-threaded Redis: ~500K ops/sec on 8 cores (one core used)
- KeyDB with 8 threads: ~3M ops/sec (all cores used)
- Latency: comparable for simple operations, better for pipelining

**Trade-offs**:
- Fewer community resources and integrations than Redis
- Smaller ecosystem of modules and tools
- May lag behind Redis in feature adoption

### Dragonfly

A modern, multi-threaded, cache-compatible alternative written in C++ using a shared-nothing architecture with a log-structured hash table. Claims 25x performance over Redis for multi-threaded workloads.

**Performance**: Up to 4M ops/sec on a single machine with 8 cores.

**CAP properties**: Uses a single primary with opt-in multi-primary replication (Dragonfly v1.0+).

**Trade-offs**:
- Newer and less battle-tested than Redis
- Different internals mean some edge-case behaviors differ
- Smaller ecosystem

## Lessons Learned

### Pattern 1: Key Naming Conventions

Every production Redis deployment needs a consistent key naming convention. The standard pattern is `namespace:key` or `namespace:subspace:key`.

```
# Good
user:1234:profile         # User profile
user:1234:sessions        # User sessions
guild:5678:members        # Guild members
ratelimit:api:user:1234   # Rate limiter
cache:weather:90210       # Cache entry

# Bad
johns_key                  # No namespace
abc123def                  # Meaningless
user_1234_profile          # Inconsistent separator
```

**Why it matters**:
- Namespaces prevent collisions (two features accidentally using the same key)
- `INFO keyspace` groups keys by database; namespacing helps identify which feature owns which keyspace
- `SCAN 0 MATCH user:*` lets you scan a single namespace
- Debugging is easier when key names are self-documenting

### Pattern 2: Use Pipelining for Bulk Operations

A single round-trip to Redis costs ~0.1ms on a local network. If you need to set 10,000 keys:

```bash
# Without pipelining: 10,000 round trips = ~1 second
for key in {1..10000}; do
  redis-cli SET "user:$key:name" "User$key"
done

# With pipelining: 1 round trip = ~1ms
{
  for key in {1..10000}; do
    echo "SET user:$key:name User$key"
  done
} | redis-cli --pipe
```

**Impact**: Pipelining reduces network overhead from 99% of total time to less than 1%. For bulk data loading, initial cache warming, or batch operations, always pipeline.

### Pattern 3: Monitor Slow Logs

```bash
# Configure slow log threshold (microseconds)
CONFIG SET slowlog-log-slower-than 10000  # 10ms

# View last 10 slow commands
SLOWLOG GET 10

# Example output:
# 1) 1) (integer) 42          # ID
#    2) (integer) 1704067200  # Timestamp
#    3) (integer) 12500       # Duration (microseconds)
#    4) 1) "KEYS"             # Command
#       2) "user:*"           # Arguments
#    5) "127.0.0.1:54321"     # Client
#    6) "my-connection-id"    # Connection ID
```

**Watch for**:
- `KEYS *` or `KEYS user:*` (use `SCAN` instead)
- `SORT` on large lists
- `ZRANGE` with very wide ranges
- `GET` on very large values (hundreds of KB or more)
- Lua scripts that run for more than 10ms

### Pattern 4: Plan for Eviction

Do not set and forget `maxmemory-policy`. Consider how eviction affects your application:

- **`allkeys-lru`**: Evicts the least recently used keys. Best for general-purpose caching. Your cache hit ratio will degrade gradually as memory fills.
- **`allkeys-lfu`**: Evicts the least frequently used keys. Better for workloads with popularity skew (a few hot keys get most traffic).
- **`volatile-lru`**: Evicts only keys with TTL set. Good if you have a mix of cache data (with TTL) and persistent data (no TTL).
- **`volatile-ttl`**: Evicts keys with the shortest remaining TTL first. Good if you want to ensure short-lived data is evicted first.
- **`noeviction`**: Return errors on writes when memory is full. Right for persistent data you cannot afford to evict вҖ” but you must ensure the total data fits in maxmemory.

**Monitor eviction rate**: If you see `evicted_keys` growing rapidly in `INFO stats`, your maxmemory is too low, your TTLs are too long, or your eviction policy does not match the workload.

### Pattern 5: Separate Cache and Persistent Data Stores

The most common production mistake is mixing cache data (can be evicted, can be regenerated) with persistent data (must survive restarts) in the same Redis instance with the same eviction policy.

**Wrong approach**:
```
One Redis instance: allkeys-lru
Contains: session data, cached HTML, rate limiter state, leaderboard scores, user profile cache
Problem: A traffic spike evicts rate limiter entries вҶ’ rate limits reset вҶ’ abuse window opens
```

**Right approach**:
```
Redis Instance A (cache):   allkeys-lfu, no persistence, 8GB
  - API response cache
  - Rendered HTML cache

Redis Instance B (state):   volatile-lru, AOF persistence, 16GB
  - Sessions (TTL)
  - Rate limiter data (TTL)
  - Leaderboard scores (no TTL, managed via size limits)

Redis Instance C (queue):   noeviction, AOF persistence, 4GB
  - Task queues
  - Streams
```

**If you must use one instance**:
- Use `volatile-lru` or `volatile-ttl` so keys without TTL are never evicted
- Set TTL on every cache entry explicitly
- Monitor the eviction rate closely
