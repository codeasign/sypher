## Foundation Challenges

### Challenge F1: Implement a Basic Circuit Breaker

Write a circuit breaker in your preferred language (Python, Java, Go, C#, TypeScript, or Rust). Your implementation must:

1. Support three states: CLOSED, OPEN, and HALF_OPEN.
2. Accept configuration: failureCount (integer, default 5), windowSize (seconds, default 10), resetTimeout (seconds, default 30), halfOpenMax (integer, default 3).
3. Expose three methods: `allowRequest()` returns boolean, `recordSuccess()`, `recordFailure()`.
4. Track failures using a sliding window. When the failure count in the window exceeds failureCount, transition to OPEN.
5. After resetTimeout in OPEN state, transition to HALF_OPEN. Allow up to halfOpenMax probe requests in HALF_OPEN. If all succeed, transition to CLOSED. If any fail, transition to OPEN.
6. Be thread-safe. Support 100 concurrent callers using atomic operations or locks.

**Stretch goal:** Implement a rate-based variant that opens when the failure rate exceeds failureRate=50% with minRequests=10.

**Stretch goal:** Implement exponential backoff for the reset timeout starting at resetTimeout, doubling each cycle up to maxResetTimeout=300 seconds.

**Stretch goal:** Add forced-closed and forced-open bypass modes for emergencies.

### Challenge F2: Test Your Circuit Breaker

Write a test suite for your circuit breaker implementation. Cover these scenarios:

1. **State transitions:** Verify CLOSED to OPEN after 5 failures. Verify OPEN to HALF_OPEN after 30-second timeout. Verify HALF_OPEN to CLOSED after 3 successful probes. Verify HALF_OPEN to OPEN after a failed probe.
2. **Sliding window:** Simulate 5 failures at second 0, wait 10 seconds, add 5 more failures. Verify the breaker does not open because the first 5 have expired from the window.
3. **Concurrency:** Launch 50 threads that all call recordFailure() simultaneously. Verify the breaker transitions to OPEN exactly once.
4. **Half-open limit:** Verify that in HALF_OPEN state, only halfOpenMax probes are allowed. The (halfOpenMax + 1)th request should be rejected.
5. **Reset timeout:** Verify the breaker stays OPEN for the configured resetTimeout. Verify it transitions to HALF_OPEN after the timeout expires.
6. **Fallback:** Verify that the fallback is invoked when the breaker is OPEN and not invoked when the breaker is CLOSED.

**Stretch goal:** Write a property-based test with 1000 random success/failure sequences verifying the invariant: the breaker is never OPEN when no failures occurred in the last windowSize seconds.

### Challenge F3: Monitor Your Circuit Breaker

Add metrics emission to your circuit breaker. Emit three metrics:
1. `cb_state` (gauge): 0 for CLOSED, 1 for OPEN, 2 for HALF_OPEN.
2. `cb_transitions` (counter): incremented on every state transition.
3. `cb_rejected` (counter): incremented on every request rejected in OPEN state.

Write a script that prints these metrics every second while your circuit breaker is under load. Use a simulated dependency that fails for 10 seconds every 60 seconds.

**Stretch goal:** Emit metrics to Prometheus and visualize in a Grafana dashboard.

## Design Challenges

### Challenge D1: Design a Circuit Breaker for a Payment Gateway

Design a circuit breaker system for a payment gateway handling $10 million in daily transaction volume. Normal response time is 200ms with a 2-second timeout. The gateway charges $0.10 per transaction.

**Requirements:**
1. Rate limit of 500 req/s. Exceeding this causes a 429 with a 30-second ban.
2. Occasional 500 errors for 5-10 seconds during deployments.
3. Failed transactions due to circuit breaker must be queued for retry without data loss.
4. Maintain 99.5 percent checkout success rate.

**Design considerations:**
- How do you distinguish transient network errors (should retry), rate limit 429 errors (should wait 30s), and true gateway 500 errors (should open breaker)?
- What is the correct reset timeout given the 30-second rate limit ban? Should it be 35 seconds?
- How do you handle the retry queue? Exponential backoff starting at 1 second?
- How do you avoid overwhelming the gateway during recovery? Gradual traffic increase from 1 to 500 req/s over 30 seconds?

**Stretch goal:** Design a canary circuit breaker that routes 1 percent of traffic to a new gateway version and opens if error rate exceeds the current version.

### Challenge D2: Design a Multi-Layer Circuit Breaker for a Microservice Chain

Design a circuit breaker system for an architecture with 50 services and call graph depth of 5 levels.

**Requirements:**
1. Service A calls B calls C calls D. If D is slow, breakers must open faster than failure cascade.
2. Each service has different failure profile: D recovers in 30 seconds, C recovers instantly.
3. System handles 10,000 req/s at peak.

**Design considerations:**
- How do you configure circuit breakers at each level? Should D have shorter timeout (5s) than A (30s)?
- How do you prevent the false positive cascade where one breaker opening causes unnecessary openings upstream?
- How do you propagate root cause through the hierarchy? Add a root-cause header?
- What is the recovery sequence? Deepest first (D to C to B to A)?
- How do you test end-to-end? Chaos engineering with controlled dependency failures?

**Stretch goal:** Design a priority-based circuit breaker where paying customers bypass the breaker during partial failures while free-tier traffic is rejected first.

### Challenge D3: Design a Circuit Breaker for a Real-Time Chat System

Design a circuit breaker for a chat system with 10 million active users using WebSocket connections for real-time messaging and HTTP API for message history.

**Requirements:**
1. Message persistence service occasionally slows during peak hours (7 PM to 10 PM).
2. User presence service has connection storms when popular events start.
3. Notification service has rate limit of 10,000 req/s.

**Design considerations:**
- How does the circuit breaker work for persistent WebSocket connections? Close the WebSocket or stop sending messages?
- How do you handle connection storms where the breaker opens and closes rapidly?
- Is the circuit breaker the right tool for the notification service rate limit, or do you need a rate limiter?
- How do you prioritize messages? Recent messages (last 5 minutes) over older ones?
- What does graceful degradation look like for a chat system? Queue messages locally and send when the breaker closes?

**Stretch goal:** Design a user-level circuit breaker. If a specific user's push notification token is invalid, open a circuit for that user only.

## Expert Challenges

### Challenge E1: Adaptive Circuit Breaker with Machine Learning

Design an adaptive circuit breaker that uses machine learning to predict failures before they happen.

**Requirements:**
1. Learn baseline latency distributions (mean, standard deviation, p50, p99) over a 24-hour rolling window with hourly granularity.
2. Detect anomalies: current latency exceeds 3 standard deviations above the mean, or error rate exceeds 5 percent of baseline.
3. Use exponential smoothing (EWMA with alpha=0.3) to predict recovery time.
4. Adjust parameters (failureThreshold, resetTimeout) based on observed behavior.

**Design considerations:**
- How do you handle the cold start problem with no data for the first 24 hours? Use default configuration during warm-up.
- How do you handle seasonal patterns? The dependency might be 2x slower at peak hours.
- How do you avoid feedback loops? Breaker opens, traffic drops, dependency latency drops, breaker thinks it is healthy. Use separate health probes.
- What is the memory overhead? EWMA requires 3 floating-point values per dependency: approximately 24 bytes.
- How do you validate the model? Shadow mode: run without taking action, log predictions, compare to outcomes.

**Stretch goal:** Implement the adaptive breaker using EWMA. Tune alpha from 0.1 to 0.5 to balance responsiveness and noise.

### Challenge E2: Distributed Circuit Breaker with Consensus

Design a distributed circuit breaker using the Raft consensus algorithm to coordinate state across 5 nodes. When one node opens, all nodes open within 100 milliseconds.

**Requirements:**
1. Tolerate 2 node failures (out of 5).
2. Consensus latency under 100ms for 99.9 percent of transitions.
3. Handle 10,000 circuit breaker operations per second.
4. Handle network partitions: 3-node partition continues, 2-node partition stops.

**Design considerations:**
- How does consensus affect latency? Each transition requires a Raft round trip across 5 nodes.
- How do you avoid the consensus cluster becoming a bottleneck? Only transitions go through consensus, not individual request decisions.
- How do you handle consensus cluster unavailability? Fall back to local-only mode.
- How do you prevent the consensus cluster from causing cascading failure? Keep it independent of application services.
- How do you test under network partitions? Use a simulated network with configurable latency (1-500ms) and partition probability (0-10 percent).

**Stretch goal:** Implement a simulation using the Raft algorithm with configurable network delay, partition probability, and failure characteristics.

### Challenge E3: Circuit Breaker as a Service (CBaaS)

Design a circuit breaker platform offered as a shared infrastructure service to 500 microservices within your organization.

**Requirements:**
1. Each team registers their dependencies. Platform manages all circuit breaker state.
2. HTTP API and gRPC API for checking state and recording outcomes.
3. Dashboard for viewing state across all 500 services.
4. Hot-reloadable YAML configuration without restarting the agent.
5. Handle 100,000 req/s from 500 microservices.

**Design considerations:**
- How do you achieve high availability and low latency? Redis Cluster with local caching on each node.
- How do you isolate state for different services? Namespaced keys: `cb:{service}:{dependency}`.
- How do you handle cold start for new services? Default configuration with 5 failures, 10-second window, 30-second timeout.
- How do you version and roll back configurations? Versioned config store with rollback support.
- How do you charge for the service? $5 per circuit breaker per month for 50 breakers = $250/month per team.
- What is the SLA? 99.99 percent uptime, under 5ms latency at p99.

**Stretch goal:** Design automatic dependency discovery by analyzing service-to-service traffic patterns. Auto-create circuit breakers with sensible defaults when new dependencies are detected.

**Stretch goal:** Design a chaos engineering integration that deliberately opens circuit breakers on a schedule to test resilience. Auto-rollback if error rates exceed 5 percent above baseline.
