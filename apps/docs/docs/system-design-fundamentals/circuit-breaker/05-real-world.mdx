# Circuit Breaker in the Real World

## Netflix: Hystrix and the Birth of Modern Circuit Breaking

Netflix developed Hystrix in 2011 after a series of cascading failures in their streaming platform. The system had grown from 50 microservices in 2009 to over 500 by 2011. Any single service slowdown could ripple through the entire call graph and take down the platform.

### The 2010 Cascading Failure

In 2010, Netflix experienced a cascading failure that took down the entire streaming service for four hours. The root cause was a single content metadata service that slowed from 50 milliseconds to 15 seconds. Every downstream service had a thread pool shared across all dependencies. The content metadata service was called by 30 other services. Within 3 minutes, all thread pools across all 30 services were exhausted. The entire site was down.

The post-mortem identified three root causes:

1. **No thread pool isolation.** A single slow dependency could exhaust the shared thread pool, blocking all other dependencies.
2. **No failure detection.** The system did not detect the slowdown until thread pools were already exhausted.
3. **No fallback.** When the content metadata service failed, there was no degraded experience. The entire page failed.

### The Hystrix Solution

Hystrix introduced three key innovations that became the industry standard:

**1. Per-dependency thread pools (bulkheads).** Each dependency gets a dedicated thread pool. The recommendation service thread pool of 10 threads is separate from the content metadata thread pool of 30 threads. A failure in recommendation does not block content metadata requests. The thread pool sizes are calculated based on each dependency's request rate and latency.

**2. Circuit breaker per thread pool.** Each thread pool has a circuit breaker. The breaker monitors the health of calls made through that pool. When the failure rate exceeds 50 percent in a 10-second window with at least 20 requests, the breaker opens. The breaker stays OPEN for 5 seconds before probing.

**3. Request collapsing.** Multiple concurrent requests for the same resource are collapsed into a single batch call within a 10-millisecond window. This reduced peak load on some dependencies by 90 percent. For example, 100 concurrent requests for movie metadata were collapsed into a single batch request for 100 movie IDs.

### Impact and Metrics

After Hystrix deployment, Netflix reported:
- Cascading failures dropped to zero in the first year after deployment.
- P99 latency during dependency failures dropped from 30 seconds to under 1 second.
- Fallback coverage reached 95 percent of all user-facing API paths.
- The streaming platform achieved 99.99 percent uptime in 2012.
- The Hystrix dashboard provided real-time visibility into every dependency's health, state, and traffic patterns.

Netflix open-sourced Hystrix in 2012. It became the most widely used circuit breaker library in the Java ecosystem, with over 10,000 GitHub stars and adoption at hundreds of companies. Netflix moved to a maintenance-only mode for Hystrix in 2018, recommending Resilience4j for new projects, but the architectural patterns Hystrix established remain the industry standard.

## Amazon: Dynamic and Proactive Circuit Breaking

Amazon uses a variant of the circuit breaker that it calls "dynamic timeout" or "adaptive circuit breaking." Unlike Netflix's count-based approach, Amazon's system adjusts thresholds automatically based on historical performance data.

### The Approach

Each dependency has a performance profile stored in a distributed configuration system (internal equivalent of AWS AppConfig). The profile records:
- Average latency over the last 5 minutes, broken down by operation type.
- P99 latency over the last 5 minutes, broken down by operation type.
- Error rate over the last 5 minutes, broken down by HTTP status code.
- The dependency's capacity in requests per second, measured through load testing.
- Dependency recovery time from previous incidents.

The circuit breaker uses three conditions to open:

1. **Error rate anomaly.** When the current error rate exceeds 3 standard deviations above the historical average, the breaker opens. The standard deviation is computed over a 7-day rolling window to capture weekly patterns.

2. **Latency anomaly.** When the current latency exceeds 2x the historical P99, the breaker opens. This catches slow degradations that do not produce errors.

3. **Proactive capacity-based opening.** When the request rate approaches the dependency's known capacity limit (for example, 90 percent of 12,000 req/s), the breaker opens proactively to prevent overload. This is the key innovation: the breaker opens before failures occur, preventing the dependency from entering a degraded state.

### Proactive Circuit Breaking

Amazon's system predicts when a dependency is about to fail and opens the circuit preemptively. If the request rate is climbing toward the dependency's known capacity of 12,000 req/s, the breaker opens at 10,800 req/s (90 percent). This prevents the dependency from becoming overloaded in the first place.

Proactive breaking requires accurate capacity modeling. Amazon maintains capacity profiles for every service, updated weekly based on load testing and production observation. The capacity model accounts for:
- Request rate (req/s).
- Resource utilization (CPU, memory, database connections).
- External dependencies (downstream services that can become bottlenecks).
- Seasonal patterns (holiday traffic, sales events).

### Impact

Amazon reported that proactive circuit breaking reduced incident duration by 60 percent compared to reactive circuit breaking (waiting for errors to occur). The system prevented an estimated 200+ cascading failures per year across Amazon's internal services.

## Google: Infrastructure-Level Context-Aware Circuit Breaking

Google implements circuit breaking as part of its infrastructure layer, not in application code. The circuit breaker is embedded in the gRPC and HTTP/2 load balancer (part of Google's internal Bandaid and L7 load balancing system).

### Key Differences from Application-Level Circuit Breaking

**1. Application-unaware.** The application does not know circuit breakers exist. The infrastructure intercepts all outbound calls and applies circuit breaking based on response codes, latency, and connection errors. The application code is simpler and cannot accidentally bypass the circuit breaker.

**2. Multiple backend discovery.** When a circuit breaker opens for one backend instance, the load balancer routes traffic to other instances. The circuit breaker operates at the instance level, not the service level. If one payment instance is failing, traffic is redirected to healthy payment instances. This provides more granular protection than service-level breakers.

**3. Graceful degradation via overload feedback.** Google services signal overload through a specific HTTP response header. When a backend is overloaded, it returns a 503 with `Retry-After: 30` and `X-Overload: true`. The load balancer reads these headers and opens a circuit for that backend for the specified duration. This is "client-driven circuit breaking" -- the backend tells the client to stop sending traffic.

**4. Automatic outlier detection.** The load balancer monitors each backend instance for consecutive 5xx errors. After 5 consecutive errors, the instance is ejected from the load balancing pool for 30 seconds (base ejection time). The ejection time doubles on each subsequent ejection.

## Shopify: Circuit Breaker in Ruby at Scale

Shopify operates a large Ruby on Rails monolith that calls hundreds of internal and external services. Ruby's Global Interpreter Lock (GIL) means that thread pool exhaustion is less of a concern (Ruby does not use true OS threads for parallelism), but request queuing and connection pool exhaustion are real problems.

### The 2016 Payment Gateway Outage

In 2016, a payment gateway slowdown caused all 50 Unicorn workers to block waiting for HTTP responses. The Unicorn workers are single-threaded processes. When all 50 workers were blocked on payment calls, no workers were available to serve any other requests. The entire storefront went down for 45 minutes.

### The Semian Solution

Shopify implemented a circuit breaker in Ruby called Semian with the following characteristics:

**Queue-depth-based detection.** Instead of counting failures, Semian monitors the request queue depth at the Rack middleware layer. When the queue depth exceeds 50 requests, the breaker opens for that dependency. This works well for Ruby's single-threaded model where queue depth directly indicates saturation.

**Semaphore-based isolation.** Instead of thread pools (which are ineffective in Ruby due to the GIL), Semian uses SysV semaphores to limit concurrent calls to each dependency. A semaphore with a count of 1 allows one in-flight call at a time across all Unicorn workers. Additional calls queue or fail immediately.

**Shared memory for cross-process coordination.** Ruby on Rails servers typically run multiple Unicorn worker processes (4-16 per machine). Each worker is a separate OS process. Semian uses SysV shared memory to share circuit breaker state across all worker processes on the same machine. This means all workers on a machine share the same circuit breaker state.

**Integration with database and HTTP clients.** Semian integrates with the MySQL2 driver, the Net::HTTP client, the Redis client, and the Memcached client. Each integration is a thin wrapper that calls the circuit breaker before making the actual network call.

### Impact

Shopify reported:
- 90 percent reduction in checkout failures after deploying Semian.
- Circuit breaker detection time dropped from 45 seconds (when the Unicorn queue filled up) to under 5 seconds.
- Semian is now used for all 200+ external dependencies at Shopify.
- The library is open source and maintained by Shopify.

## Open Source Implementations

### Hystrix (Java, Netflix, 2012-2018)

Netflix's Hystrix is the most influential circuit breaker library. It introduced the thread pool isolation pattern, the circuit breaker with sliding window, and request collapsing.

- **Status:** Maintenance mode since 2018. No new features, only critical bug fixes.
- **Best for:** Legacy systems already using Hystrix. Migration to Resilience4j is recommended for new projects.
- **Configuration:** Annotation-based using @HystrixCommand and @HystrixProperty, or programmatic using HystrixCommand.Setter.
- **Metrics:** Hystrix Dashboard (web UI showing real-time circuit breaker state), Turbine (aggregates metrics from multiple instances).
- **JAR size:** Approximately 200 KB.

### Resilience4j (Java, in active development)

Resilience4j is the modern successor to Hystrix. It is lighter (no dependency on Netflix Archaius or Hystrix's 200KB+ JAR), faster (benchmarks show 2x throughput of Hystrix: 15M ops/s versus 8M ops/s), and supports more features.

- **Features:** Circuit breaker, bulkhead, rate limiter, retry, time limiter, cache, and fallback in a single library.
- **Configuration:** Per-instance configuration via builder pattern, YAML files, or properties files. Supports hot reload through Spring Cloud Config.
- **Threading:** Uses semaphore isolation by default (no context switch, lower overhead). Optional thread pool isolation for truly separate thread pools.
- **Metrics:** Integrates with Micrometer, Dropwizard Metrics, and Prometheus. Dashboards available in Grafana.

### Istio / Envoy (Proxy-based, Cloud Native)

Envoy proxy includes circuit breaking at the proxy level. Envoy supports:

- **Cluster-level circuit breaking:** Maximum number of connections (100), pending requests (10), requests (100), and retries (3) per cluster. When any limit is exceeded, the cluster is ejected.
- **Instance-level circuit breaking:** Same limits applied per upstream host. Provides more granular control.
- **Outlier detection:** Automatic circuit breaking based on consecutive 5xx errors (5), consecutive gateway errors (5), or success rate below a threshold (90 percent). Detected instances are ejected from the load balancing pool for 30 seconds (baseEjectionTime).

```
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 10
        http2MaxRequests: 100
        maxRequestsPerConnection: 10
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

## Cloud Provider Implementations

### AWS App Mesh

AWS App Mesh is a service mesh that supports circuit breaking through Envoy proxy integration. You configure circuit breaking in the App Mesh virtual node resource:

```
VirtualNode → VirtualRouter → VirtualService
     │                              │
     │  Envoy sidecar               │  Cluster IP
     │                              │
     │  Circuit breaking config:    │
     │  • maxConnections: 100       │
     │  • maxPendingRequests: 10    │
     │  • maxRequests: 100          │
```

App Mesh supports connection pool limits (TCP and HTTP) but does not support sliding window circuit breaking (outlier detection) natively. For full circuit breaker functionality on AWS, you need to pair App Mesh with a client-side library like Resilience4j or use the application load balancer's circuit breaker features.

### Azure Circuit Breaker (Polly)

Azure's recommended circuit breaker implementation for .NET applications is the Polly library. Polly is a .NET resilience and transient-fault-handling library that supports circuit breaker, retry, bulkhead isolation, timeout, and fallback policies.

```
var circuitBreakerPolicy = Policy
    .Handle<HttpRequestException>()
    .Or<TimeoutException>()
    .CircuitBreakerAsync(
        exceptionsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30)
    );

var result = await circuitBreakerPolicy.ExecuteAsync(() =>
    httpClient.PostAsync(url, content)
);
```

Polly's circuit breaker supports count-based and rate-based opening, configuration via the .NET configuration system, and metrics via Application Insights integration.

## Key Takeaways

- Netflix's Hystrix pioneered the per-dependency thread pool and circuit breaker pattern. It is the most referenced implementation in the industry, with over 10,000 GitHub stars.
- Amazon uses adaptive circuit breaking with proactive detection based on capacity modeling. The breaker opens at 90 percent of capacity to prevent overload before failures occur.
- Google embeds circuit breaking in the infrastructure layer (load balancer), not in application code. It operates at the instance level with zero application code changes required.
- Shopify's Semian library demonstrates circuit breaking for Ruby applications using queue-depth detection and SysV semaphore-based isolation across processes.
- Resilience4j is the modern Java choice: lighter (50 KB JAR), faster (15M ops/s), and more actively developed than Hystrix.
- For cloud-native deployments, Istio/Envoy provides infrastructure-level circuit breaking without changing application code. Configured via Kubernetes DestinationRule resources.
- AWS App Mesh supports connection pool limits but not sliding window outlier detection. Azure recommends Polly for .NET applications with Application Insights integration.
