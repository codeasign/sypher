# Circuit Breaker Tradeoffs

## Core Tension

The fundamental tension in circuit breaker design is between **availability** and **responsiveness**.

If the breaker opens too aggressively (low failure threshold of 2 failures, short window of 5 seconds), you sacrifice availability: the breaker rejects requests that might have succeeded. A 200-millisecond transient hiccup in the dependency triggers 30 seconds of rejected traffic. The dependency returns to health 1 second into the 30-second window, but your users see errors for 29 more seconds. Your availability drops from 99.99 percent to 99.95 percent, which represents 4.3 minutes of additional downtime per month.

If the breaker opens too conservatively (high failure threshold of 50 failures, long window of 120 seconds), you sacrifice responsiveness: the dependency is failing but you keep sending traffic. Every failed request consumes resources: 5 seconds of thread time, a TCP connection, memory for the request and response objects. After 120 seconds, 200 threads are blocked and the service is already in a cascading failure. The breaker opens too late to help.

There is no configuration that perfectly resolves this tension. The right balance depends on your dependency's failure profile and your tolerance for false positives. A critical dependency on a user-facing path needs fast detection with a willingness to accept false positives. A non-critical background dependency can tolerate slower detection.

## Decision Matrix

The following table compares systems with and without circuit breakers across eight dimensions:

| Dimension | Circuit Breaker | No Circuit Breaker | Best for |
|---|---|---|---|
| Failure detection | Automatic within N failures within window | Manual via monitoring dashboards | All production systems with uptime SLAs |
| Cascading failure prevention | Breaks the chain at the first dependency | Cascade continues through all dependents | Critical dependencies (payment, auth, database) |
| False positive handling | Breaker opens unnecessarily, triggers fallback | No false positives (no automatic detection) | Stable dependencies with predictable behavior |
| Recovery speed | Automatic probe and close within timeout | Manual restart, config change, or deployment | Self-healing systems, 24/7 operations |
| Operational complexity | Configuration tuning, metrics monitoring, alerting | None -- just set timeouts | Small teams with limited SRE resources |
| Latency overhead | ~1 microsecond per request in CLOSED state | 0 overhead | Throughput-sensitive systems (100K+ req/s) |
| Fallback integration | Natural pattern with built-in fallback support | Requires custom error handling code | User-facing features with degraded UX acceptable |
| Testing burden | Need failure injection, chaos engineering tests | None | Teams with good QA practices and staging environments |

## CAP Positioning

The circuit breaker pattern is fundamentally an **AP (Availability and Partition Tolerance)** pattern with an **eventual consistency** model.

- **Availability:** The breaker stays available even when the dependency is down. It responds quickly with fallbacks or errors. No request blocks indefinitely. The system as a whole maintains availability even when individual dependencies fail.
- **Partition Tolerance:** The breaker works correctly when the network between caller and dependency is partitioned. It opens on failure and prevents the partition from affecting more resources. The breaker itself does not depend on network connectivity -- it is a local, in-memory component.
- **Consistency:** The breaker does not guarantee that all callers see the same state. Instance A might be OPEN while Instance B is CLOSED. The system converges as traffic patterns and failure signals stabilize. This is eventual consistency: given enough time without new failures, all instances will converge to the same state.

## Failure Modes

### Failure Mode 1: Thundering Herd on Recovery

When a circuit breaker transitions from OPEN to HALF_OPEN, multiple callers might all send probe requests simultaneously. If halfOpenMax is 10 and 10 callers all probe at once, the recovering dependency receives a burst of 10 requests. If the dependency is barely recovered, this burst overwhelms it again.

**Mitigation:** Use a low halfOpenMax (1-3). Stagger probe requests with a random delay of plus or minus 25 percent of the reset timeout.

### Failure Mode 2: False Positive Cascade

Service A opens its breaker for Service B. Service B's traffic drops by 80 percent. Service B's dependency C recovers because of the reduced load. But Service B's breaker for C is still OPEN, so B never sends traffic to C. A false positive at one tier causes a false positive at every downstream tier.

**Mitigation:** Use shorter reset timeouts for downstream breakers. Implement cache fallbacks that absorb the false positive. Use a distributed circuit breaker that shares state.

### Failure Mode 3: Infinite Probe Cycle

The dependency is healthy enough to respond to a probe but unhealthy enough to fail under full load. The breaker transitions between HALF_OPEN and CLOSED repeatedly. Each time it closes, a full traffic burst causes immediate failures and reopens. The system oscillates.

**Mitigation:** Use a larger halfOpenSuccessThreshold (5-10) to require confidence before closing. Introduce a stability period: the breaker must remain in CLOSED state for 30 seconds before allowing full traffic. Gradually increase probe traffic: start with 1 probe, then 5, then 50, then full traffic.

### Failure Mode 4: Cache Poisoning Fallback

The fallback returns a cached response. The cache holds stale data from before the failure. The circuit breaker recovers, but clients continue to see stale data because the fallback was served and cached at the client or edge.

**Mitigation:** Set TTL on fallback cache entries to 30-60 seconds. Use a cache-busting header (`Cache-Control: max-age=30`) on the fallback response. Use a versioned cache key that increments on each recovery.

### Failure Mode 5: Configuration Drift

Different instances of the same service have different circuit breaker configurations. Instance A opens at 5 failures. Instance B opens at 10 failures. The dependency receives uneven traffic. Instance A's callers get fallbacks while Instance B's callers get real responses.

**Mitigation:** Ship configuration through a central config system (Consul, etcd, Kubernetes ConfigMap). Monitor for configuration drift with alerts. Pin configuration to the service version.

### Failure Mode 6: Retry Storm with Breaker Cycling

The caller has a retry policy with 3 retries. The circuit breaker opens after 5 failures. Each retry counts as a separate failure. 2 requests with 3 retries each generate 8 failures (2 originals + 6 retries). The breaker opens after 5 failures, meaning the retries from the second request are rejected. But the retry storm has already consumed resources.

**Mitigation:** Check the circuit breaker before each retry attempt. If the breaker is OPEN, do not retry.

## Anti-Patterns

### Anti-Pattern 1: Single Circuit Breaker for All Dependencies

You wrap your HTTP client in one circuit breaker. Every outgoing call goes through it. When any dependency fails, the breaker opens for all dependencies. A payment failure shuts down inventory and shipping calls too.

**Fix:** Use one circuit breaker per logical dependency. Each dependency has its own configuration, its own sliding window, and its own state.

### Anti-Pattern 2: Circuit Breaker Without Fallback

The breaker opens and the caller receives an exception. No degradation, no fallback. The exception propagates up the call stack, causing a 500 error to the end user.

**Fix:** Always provide a fallback for user-facing services. Even a simple "please try again later" message is better than a blank 500 page. Cache fallbacks are ideal for read-heavy dependencies.

### Anti-Pattern 3: Circuit Breaker as a Retry Replacement

You configure the breaker to open after 3 failures. You also configure a retry with 3 attempts. When the dependency fails, the retry sends 3 requests, all fail, the breaker opens. The caller waited for 3 retry timeouts.

**Fix:** Implement retry with exponential backoff but apply the circuit breaker check before each retry attempt. If the breaker is OPEN, skip the retry and go directly to fallback.

### Anti-Pattern 4: Overly Long Reset Timeout

You configure resetTimeout=300 seconds because you want to be safe. The dependency recovers in 5 seconds. For the next 295 seconds, your system rejects requests to a healthy dependency.

**Fix:** Start with a short resetTimeout of 5-15 seconds. Increase it only if the dependency has a known long recovery time. Use exponential backoff for the reset timeout if the breaker cycles repeatedly.

### Anti-Pattern 5: No Metrics or Alerting

You deploy circuit breakers but do not expose metrics. When the payment service is slow, you notice because customers complain about failed checkout.

**Fix:** Emit metrics for state, transition count, and rejected count. Alert on OPEN state lasting more than 5 minutes.

### Anti-Pattern 6: Circuit Breaker on Cache-Only Dependencies

A Redis cache is a dependency with sub-millisecond latency and no cascading failure risk. Adding a circuit breaker to a Redis call adds overhead without benefit.

**Fix:** Use timeouts and connection pooling for cache dependencies. Reserve circuit breakers for dependencies that can cause cascading failure.

## When NOT to Use a Circuit Breaker

### Idempotent Stateless Dependencies

If the dependency is purely functional -- same input always produces same output, no side effects -- a circuit breaker adds overhead without benefit. Failures are transient and retriable. Exponential backoff with jitter is sufficient. Examples: DNS resolution, static file serving, read-only API endpoints.

### Local-Only Operations

If the dependency is a local file system, an in-memory cache, or a Unix socket, the circuit breaker adds latency to an operation that cannot cascade. A local disk failure means the machine is broken anyway. Examples: local file I/O, shared memory, Unix domain sockets.

### Batch Processing

In batch processing, each unit of work is independent. A failed batch job does not cause thread pool exhaustion for interactive users. Circuit breakers for batch systems add complexity without preventing cascading failure. Use a queue with DLQ (dead letter queue) instead.

### Read-Only Replicas

A read replica has no cascading failure risk. If it is slow, reads can fail over to the primary or another replica. Circuit breakers on read replicas add overhead. Use failover logic instead.

### Systems with Microsecond Latency Requirements

High-frequency trading systems, real-time audio/video processing, and hardware-accelerated systems operate at microsecond latency. The 1-microsecond overhead of a circuit breaker is significant at this scale. Use hardware-level fault isolation instead.

## Key Takeaways

- The core tension is between availability (avoiding false positives that reject healthy requests) and responsiveness (detecting failures quickly before they cascade).
- The decision matrix shows that circuit breakers benefit almost all production systems with critical dependencies, but the configuration varies by dependency type.
- Circuit breakers are an AP pattern -- they prioritize availability and partition tolerance over strict consistency. Each instance makes independent decisions.
- Four common failure modes require specific mitigations: thundering herd (stagger probes), infinite probe cycles (increase success threshold), false positive cascades (shorter timeouts downstream), and cache poisoning (short TTL on fallbacks).
- Six anti-patterns cause circuit breakers to cause more harm than good: single breaker, no fallback, breaker-as-retry, long timeout, no metrics, and cache-only breaker.
- Circuit breakers are not needed for idempotent dependencies, local-only operations, batch processing, read replicas, or microsecond-latency systems.
- Configuration is a tuning problem that depends on the dependency's failure profile, traffic pattern, and recovery characteristics.
