# Circuit Breaker Architecture

## Naive Architecture: Timeout-Only

The simplest approach to handling slow dependencies is a timeout. Every HTTP client has a connect timeout (typically 500ms), a read timeout (typically 2s), and a total timeout (typically 5s). When the dependency exceeds the timeout, the client throws an exception and the thread is released.

### Why Timeouts Alone Fail

A timeout prevents a single thread from blocking forever. It does not prevent cascading failure. Consider a service with 200 threads in its thread pool. Each thread makes a call with a 5-second timeout. If the payment service slows to 30 seconds per request, every thread hits the timeout after 5 seconds. But the thread still spent 5 seconds blocked. If you are receiving 100 requests per second and each request spends 5 seconds waiting, you need 500 threads to keep up. The thread pool has 200. Requests queue up. The queue grows. Memory grows. The service dies.

```
Time:  0s    5s    10s   15s   20s   25s   30s
       │     │     │     │     │     │     │

Thread 1:  [────request 1────]    [────request 6────]
Thread 2:  [────request 2────]    [────request 7────]
Thread 3:  [────request 3────]    [────request 8────]
Thread 4:  [────request 4────]    [────request 9────]
Thread 5:  [────request 5────]    [────request 10───]

Queue:     r6 r7 r8 r9 r10 r11 r12 r13 r14 r15 ...

Result: Thread pool exhausted at t=5s. Queue grows unbounded.
        Memory exhausted at t=15s. Service crashes.
```

### The Timeout-Only Failure Cascade

**Phase 1 (0-5 seconds):** Dependency slows from 200ms to 30s. Threads start blocking for the full 5-second timeout. After 5 seconds, 100 requests have arrived. Each thread blocks for 5 seconds. With a 200-thread pool, all 200 threads are blocked by second 5.

**Phase 2 (5-15 seconds):** Thread pool is exhausted. New requests enter the queue. The queue depth grows by 100 entries per second. At 100 requests per second and 5-second queue wait, the queue holds 500 entries. At 2 KB per request, that is 1 MB of queued data.

**Phase 3 (15-20 seconds):** The service runs out of memory. The OOM killer terminates the JVM or container. All 200 in-flight requests are lost. All 500 queued requests are lost.

**Phase 4 (20-60 seconds):** Downstream services that depended on this service start timing out. Their thread pools fill up. The cascade continues through service A (ordering), service B (inventory), and service C (warehouse).

## Production Architecture: Circuit Breaker + Bulkhead + Fallback

A production-grade architecture combines three patterns:

**Circuit Breaker:** Prevents calls to an unhealthy dependency. Responds immediately with error or fallback. Operates at the request level by checking state before each request and recording outcome after each request.

**Bulkhead:** Limits concurrent calls to each dependency. A dedicated thread pool of 10 threads for the payment service prevents payment problems from affecting the inventory service.

**Fallback:** Provides a degraded response when the circuit is open or the bulkhead is full. Fallbacks enable graceful degradation.

### How They Work Together

When a request arrives to call the payment service:
1. The circuit breaker checks its state. If OPEN, the request is immediately redirected to the fallback. No thread is consumed.
2. If CLOSED or HALF_OPEN, the request enters the payment bulkhead (a dedicated thread pool with 10 threads).
3. If the bulkhead is full (all 10 threads are busy), the request waits in the bulkhead queue (up to 5 entries). If the queue is full, the request is rejected and the fallback is invoked.
4. If a bulkhead thread becomes available, it makes the actual HTTP call to the payment service.
5. If the call succeeds, the result is returned. The circuit breaker records a success.
6. If the call fails, the exception is returned to the caller. The circuit breaker records a failure.

## Integration Patterns

### Pattern 1: Decorator Pattern

Wrap the dependency client with a circuit breaker decorator. The decorator implements the same interface as the original client.

```
interface PaymentClient {
    PaymentResponse charge(PaymentRequest request);
}

class CircuitBreakerDecorator implements PaymentClient {
    private final PaymentClient delegate;
    private final CircuitBreaker breaker;
    private final FallbackProvider<PaymentRequest, PaymentResponse> fallback;

    PaymentResponse charge(PaymentRequest request) {
        if (!breaker.isRequestAllowed()) {
            return fallback.apply(request);
        }
        try {
            PaymentResponse response = delegate.charge(request);
            breaker.recordSuccess();
            return response;
        } catch (Exception e) {
            breaker.recordFailure();
            return fallback.apply(request);
        }
    }
}
```

### Pattern 2: Interceptor Pattern

Use a framework interceptor (Spring AOP, Jakarta EE interceptor, gRPC interceptor) to apply the circuit breaker transparently via annotations.

```
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
public PaymentResponse charge(PaymentRequest request) {
    return paymentClient.charge(request);
}

public PaymentResponse paymentFallback(PaymentRequest request, Throwable t) {
    return PaymentResponse.cached(request.getUserId());
}
```

### Pattern 3: Proxy Pattern (Service Mesh)

Deploy a proxy layer (sidecar, service mesh) that implements circuit breaking. The application code is completely unaware of the circuit breaker. The proxy intercepts all outbound traffic and applies circuit breaking based on configuration.

```
Pod:
  ┌────────────────┐
  │ Application    │
  │ (no CB code)   │──> localhost:8080
  └────────────────┘
         │
         ▼
  ┌────────────────┐
  │ Envoy Sidecar  │──> payment.service:443
  │ (circuit       │
  │  breaker)      │
  └────────────────┘
```

The proxy pattern provides the strongest isolation but adds latency through an additional network hop and operational complexity from managing sidecar lifecycles.

## Multi-Region Circuit Breaker

In a multi-region deployment, each region has its own circuit breaker state:

**Region-Specific Breakage:** The circuit breaker for the payment service in us-east-1 opens independently from the breaker in eu-west-1. This is correct: the payment service in us-east-1 might be unhealthy while the eu-west-1 instance is fine.

**Cross-Region Calls:** If us-east-1 calls the payment service in eu-west-1 as a failover, the local circuit breaker in us-east-1 does not know about the health of the remote instance. You need dedicated cross-region circuit breakers with longer timeouts (30 seconds versus 10 seconds) to account for higher inter-region latency of 100-200ms.

```
us-east-1:
  ┌─────────────────────────────────┐
  │  Checkout Service               │
  │  CB: payment-us-east → timeout=10s │──> payment.us-east-1
  │  CB: payment-eu-west → timeout=30s │──> payment.eu-west-1
  └─────────────────────────────────┘

eu-west-1:
  ┌─────────────────────────────────┐
  │  Checkout Service               │
  │  CB: payment-eu-west → timeout=10s │──> payment.eu-west-1
  │  CB: payment-us-east → timeout=30s │──> payment.us-east-1
  └─────────────────────────────────┘
```

## Capacity Planning with Circuit Breakers

### Thread Pool Calculation

When using bulkheads, each dependency gets a dedicated thread pool. The pool size depends on the request rate, average latency, and target utilization.

Pool size = R * L * (1 / U)

Where R = request rate in req/s, L = average latency in seconds, U = target utilization.

For the payment service with 50 req/s, 200ms latency, and 60 percent utilization:
Pool size = 50 * 0.2 * (1 / 0.6) = 16.6, round up to 17 threads.

When the circuit breaker is OPEN, the bulkhead sees zero traffic. The 17 threads remain idle.

### Queue Sizing

Even with circuit breakers, you need a bounded queue in front of each bulkhead. When the bulkhead is full and the circuit is CLOSED, requests wait in the queue. The queue should be bounded to prevent memory exhaustion.

Queue depth = R * maxWait * (1 - U_avg)

For 50 req/s, 5 seconds max wait, and 50 percent average utilization:
Queue depth = 50 * 5 * (1 - 0.5) = 125 entries.

When the circuit breaker opens, the queue drains rapidly because new requests are fast-failed.

## Operations

### Monitoring Circuit Breakers

You must monitor three metrics for each circuit breaker:

**State:** CLOSED (0), OPEN (1), or HALF_OPEN (2) as a gauge. Alert when the state is OPEN for more than 5 minutes. This indicates the dependency is chronically unhealthy.

**Transition Count:** Number of state transitions per minute as a counter. A high rate of transitions (cycling between OPEN and HALF_OPEN) indicates an unstable dependency. Alert when transitions exceed 10 per minute.

**Rejected Request Count:** Number of requests rejected because the breaker is OPEN as a counter. This measures the blast radius. Alert when rejected count exceeds 1,000 per minute.

### Health Check Integration

The circuit breaker's HALF_OPEN probes can use either the same request path or a dedicated health endpoint:
- Same request path: more realistic but might overwhelm the dependency with complex processing.
- Health endpoint: lighter weight but might succeed when the real endpoint fails.

Best practice: use the health endpoint for the initial HALF_OPEN probe, then graduate to full request probes.

### Deployment Considerations

**New Version Deployment:** When you deploy a new service version, the circuit breaker state resets to CLOSED with an empty sliding window. The new instance has no history of failures. If the dependency is unhealthy, the breaker will open again after failureCount requests.

**Rolling Deployment Risk:** During a rolling deployment, new instances start with CLOSED breakers while old instances might have OPEN breakers. The new instances send full traffic to the dependency, potentially overwhelming it. Mitigation: use a 30-second warm-up period where new instances probe the dependency before accepting full traffic.

### Logging

Every state transition should be logged. Every fallback invocation should be logged. Every probe outcome should be logged.

```
14:23:05.123 [CB] payment.service → OPEN   | failures:5 window:10s timeout:30s
14:23:05.124 [CB] payment.service → closed  | fallback invoked | userId:99482
14:23:35.123 [CB] payment.service → HALF_OPEN | timeout:30s expired
14:23:35.124 [CB] payment.service → probe:1   | status:200 latency:150ms
14:23:35.456 [CB] payment.service → CLOSED  | probes:3/3 succeeded
```

## Key Takeaways

- Timeout-only architecture is insufficient because threads still block waiting for timeout expiration. A circuit breaker prevents thread blocking entirely.
- Production architecture combines circuit breaker (request-level fail-fast), bulkhead (resource-level isolation with dedicated thread pools), and fallback (business-level degradation).
- Each dependency gets its own dedicated thread pool with dependency-specific sizing and queue depth.
- Integration patterns range from decorator (in-code wrapping, most flexible) to interceptor (annotation-based, cleanest) to proxy (service mesh, strongest isolation).
- In multi-region deployments, each region has independent circuit breakers. Cross-region calls need separate breakers with longer timeouts and lower thresholds.
- Capacity planning uses the formula pool size = R * L * (1/U) to size thread pools and bounded queues.
- Monitor state, transition count, and rejected request count. Alert on prolonged OPEN state and rapid cycling.
