# Leader Election — Challenges

Apply your knowledge of leader election through progressively difficult challenges. Each challenge includes a full model answer and a checklist for self-assessment.

## Tier 1: Foundation Challenges

### Challenge 1: Design a Leader Election for a 3-Node Database

A startup runs a 3-node PostgreSQL cluster with streaming replication. Currently, failover is manual. An operator must detect the primary failure, promote a standby, and update the application connection string. This takes 10-15 minutes.

Design an automated leader election system.

**Answer.**

**Architecture:**
- Deploy a consensus-based coordination service (etcd or ZooKeeper) on each of the 3 nodes.
- Each node registers as a potential primary by creating an ephemeral znode.
- The node with the smallest sequence number becomes the primary.
- The other two nodes become standbys and replicate from the primary.
- Applications query the coordination service to discover the current primary.

**Election process:**
1. Node A creates /db/primary/node-0000000001 (ephemeral, sequential).
2. Node B creates /db/primary/node-0000000002.
3. Node C creates /db/primary/node-0000000003.
4. Node A has the smallest sequence number → becomes primary.
5. Node B and C watch the preceding znode (Node A and B respectively).
6. When Node A fails, its ephemeral znode is deleted.
7. Node B's watch fires. Node B becomes the new primary.
8. Node C updates its watch to Node B.

**Election timeout:** None for the election itself (the watch is immediate). The session timeout (5 seconds) determines how long after failure the election begins.

**Checklist:**
- [ ] Each node creates an ephemeral sequential znode on startup.
- [ ] The smallest znode sequence number determines the primary.
- [ ] Applications query the coordination service for the primary address.
- [ ] The system survives any single node failure.
- [ ] A failed node re-joins as a standby, not a primary.

### Challenge 2: Implement a Heartbeat Protocol

Design a heartbeat protocol for a 5-node leader-election cluster where the leader sends heartbeats every 50ms. Each follower maintains a randomized election timeout between 150ms and 300ms.

Specify:
1. The heartbeat message format.
2. What the follower does when it receives a heartbeat.
3. What the follower does when the timeout fires.
4. How the leader detects it has lost quorum.

**Answer.**

**Message format (AppendEntries in Raft terms):**
```
Heartbeat {
  term: 7,              // Leader's current term
  leaderId: 2,          // ID of the leader node
  prevLogIndex: 12,     // Index of last log entry
  prevLogTerm: 6,       // Term of last log entry
  entries: [],          // Empty for heartbeat
  leaderCommit: 10      // Leader's commitIndex
}
```
Total size: approximately 100 bytes.

**On heartbeat receipt (follower):**
1. Compare the heartbeat term against currentTerm.
2. If term < currentTerm, reject the heartbeat and return the higher term.
3. If term >= currentTerm, reset the election timer to 0. The follower stays in follower state.
4. Update currentTerm if the heartbeat term is higher.
5. Update commitIndex with leaderCommit if appropriate.
6. Return success.

**On timeout fire (follower → candidate):**
1. Increment currentTerm.
2. Set votedFor = self.
3. Persist currentTerm and votedFor.
4. Reset election timer to new random value.
5. Send RequestVote RPCs to all other nodes.
6. Wait for responses. If majority achieved, become leader.

**Leader quorum check:**
The leader maintains a lease timer. At least once per lease interval, the leader must receive heartbeat acknowledgments from a majority of nodes. If the leader cannot get acknowledgments from a majority within the lease period (500ms), it steps down voluntarily to prevent split-brain.

**Checklist:**
- [ ] Heartbeat message format includes term, leaderId, prevLogIndex, prevLogTerm, leaderCommit.
- [ ] Follower resets election timer on valid heartbeat.
- [ ] Follower transitions to candidate when timeout fires.
- [ ] Leader checks quorum via lease mechanism.
- [ ] Leader steps down if quorum is lost.

## Tier 2: Intermediate Challenges

### Challenge 3: Handle a Network Partition

A 5-node Raft cluster runs across 3 availability zones (AZs):

- Zone A: Node 1, Node 2
- Zone B: Node 3, Node 4
- Zone C: Node 5

Zone A experiences a complete network outage. Zone B and C can communicate with each other but not with Zone A.

1. What happens to the cluster?
2. Can writes continue?
3. What happens when Zone A recovers?
4. How does the surviving cluster determine the correct leader?

**Answer.**

**During the partition:**

Zone A (Nodes 1, 2) — 2 nodes, minority partition:
- Node A1 and A2 detect leader heartbeat timeout (if leader was in Zone A) or followers detect leader timeout (if leader was elsewhere).
- They can attempt elections, but with only 2 nodes, they cannot reach a majority (3 votes needed).
- No leader is elected in Zone A. Writes are rejected. The cluster is partially unavailable.

Zones B+C (Nodes 3, 4, 5) — 3 nodes, majority partition:
- Nodes 3, 4, and 5 can communicate. They have 3 nodes, which is enough for a majority (2 votes + self).
- An election proceeds normally. The first node to time out becomes candidate.
- With 3 nodes, 2 votes are needed. The new leader is elected.
- Writes continue in the majority partition.

**When Zone A recovers:**
- Nodes 1 and 2 reconnect to the cluster.
- They receive heartbeat messages from the new leader (Node 3, for example) with a higher term.
- Nodes 1 and 2 step down to follower state. They synchronize their logs from the leader.
- Any writes received by the old leader (if one was elected in Zone A) are discarded because they were not committed to a majority.
- The cluster returns to full health with 5 nodes.

**Leader determination:**
Raft guarantees safety because only a node with all committed log entries can be elected. Since the majority partition (Zone B+C) had 3 nodes, any committed entries are in their logs. The minority partition (Zone A) could not commit any entries. Therefore the new leader in Zone B+C has all committed entries, and the Zone A nodes accept its authority.

**Checklist:**
- [ ] Minority partition cannot elect a leader (no quorum).
- [ ] Majority partition elects a leader and continues writes.
- [ ] Recovering nodes step down and sync logs.
- [ ] No committed writes are lost.
- [ ] Uncommitted writes from the minority partition are discarded.

### Challenge 4: Design a Leader Election for a Multi-Tenant SaaS System

A SaaS platform runs 10 shards across 50 nodes. Each shard has 5 nodes. Nodes can serve multiple shards. The platform needs leader election per shard.

Design the election system. Consider:
1. How does each shard elect its leader?
2. What happens when a node hosting multiple shard leaders fails?
3. How does the system rebalance leadership when a node joins?
4. What is the blast radius of a failed election?

**Answer.**

**Architecture:**
- Use ZooKeeper for coordination. Create a znode for each shard: `/shards/shard-0`, `/shards/shard-1`, etc.
- Within each shard's znode, each replicating node creates an ephemeral sequential znode.
- The smallest sequence number within each shard becomes that shard's leader.
- This is identical to the ZooKeeper leader election pattern, applied per shard.

**Node failure:**
If Node X hosts 3 shard leaders (shards 3, 7, 9) and fails, all 3 ephemeral znodes for those shards are deleted. Each shard's follower with the next-smallest znode becomes the new leader. The 3 elections happen independently and in parallel. Total recovery time: one ZooKeeper session timeout (configurable, typically 5-10 seconds).

**Node join:**
When a new node joins, it creates znodes for each shard it should replicate. It obtains a higher sequence number than existing nodes, so it becomes a follower. No leadership rebalance is necessary. The system naturally remains leader until the existing leader fails.

**Blast radius:**
A single failed election affects only one shard. The other 9 shards continue operating. Writes to the affected shard pause during the election (5-10 seconds) but writes to other shards proceed normally.

**Checklist:**
- [ ] Each shard has its own election znodes in ZooKeeper.
- [ ] A node failure triggers elections only for the shards where that node was leader.
- [ ] New nodes join as followers without disrupting existing leaders.
- [ ] Election failures are scoped to a single shard.
- [ ] Recovery time is one session timeout.

## Tier 3: Advanced Challenges

### Challenge 5: Design a Leaderless Cluster with a Twist

A customer demands a system where no single node acts as the leader, but the system must still provide strong consistency for write operations. They want to avoid the complexity of leader election entirely.

Is this possible? If so, design it. If not, explain why.

**Answer.**

This is the foundational question behind consensus protocols that do not require a distinguished leader. Several approaches exist:

**Solution 1: Multi-Paxos with Anycast.** Multi-Paxos does not technically require an elected leader. In classic Paxos, any node can act as a proposer. However, without a distinguished leader, the protocol risks liveness issues (conflicting proposals may never converge). Multi-Paxos typically uses a leader anyway to ensure progress.

**Solution 2: EPaxos (Egalitarian Paxos).** EPaxos allows any node to propose commands without a leader. Commands are ordered using a conflict detection mechanism:
- Each command declares its dependencies.
- Commands that conflict (access the same keys) must be ordered.
- Commands that do not conflict can be executed in parallel.
- EPaxos achieves leaderless consensus with no leader election overhead.

**Solution 3: Conflict-free Data Types (CRDTs).** CRDTs eliminate the need for a leader entirely:
- Every node accepts writes independently.
- Data types are designed so that concurrent updates always converge to the same state.
- No coordination is needed. No leader election. No consensus.
- The trade-off is that CRDTs are limited to specific data types (counters, sets, registers).

**Solution 4: Quorum-based writes without a leader.** In this approach:
- A client sends a write to all replicas.
- The client waits for a quorum (W = N/2 + 1) of successful responses.
- Each replica applies the write independently.
- Read requests also query a quorum (R = N/2 + 1) and return the most recent version.
- No leader is needed. The client coordinates the quorum.

**Recommendation for the customer:**
If the workload involves simple read-modify-write patterns, use CRDTs (Solution 3). If complex transactions are needed, use EPaxos (Solution 2). If the team has operational experience with databases, use quorum-based replication (Solution 4) with a compatible database like Cassandra.

**When leader election is still better:**
Leader election becomes the simpler choice when:
- The workload requires strict serializability.
- The operations are complex (multi-key transactions, stored procedures).
- The team is experienced with Raft or ZooKeeper.
- The system already uses a leader-based database.

**Checklist:**
- [ ] Leaderless consensus is possible (EPaxos, CRDTs, quorum reads/writes).
- [ ] CRDTs eliminate the election problem entirely but limit data types.
- [ ] EPaxos provides leaderless strong consistency with conflict detection.
- [ ] Quorum-based writes with client coordination avoid a leader.
- [ ] Leader election is simpler for complex transactional workloads.
