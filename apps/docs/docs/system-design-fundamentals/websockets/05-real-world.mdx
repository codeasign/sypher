## Company Case Studies

**Problem:** Netflix's content metadata (titles, descriptions, artwork, recommendations) changes constantly as content licensing changes. An HTTP polling approach across the entire catalog would generate billions of requests per day for mostly unchanged data.

**Solution:** Netflix built a dedicated WebSocket service called Zuul (their gateway) with a WebSocket upgrade path. Here is how it works:

### Discord: Real-Time Chat at 150 Million Users

**Scale:** 150 million monthly active users. 50 million concurrent WebSocket connections. 15,000 messages per second at peak.

**Problem:** Discord's chat is inherently real-time. Messages must appear on all recipients' screens within 100ms. Every user is connected to multiple servers (guilds) and channels simultaneously. Broadcasting a message to a guild with 100,000 members requires delivering to 100,000 connections.

**Solution:** Discord uses a custom WebSocket gateway built on Erlang (BEAM VM), chosen for its per-process isolation — each WebSocket connection is a lightweight Erlang process.

### Uber: Real-Time Driver Location and Dispatch

**Scale:** 130 million monthly active users. 5 million drivers. 500,000 concurrent WebSocket connections.

**Problem:** Uber's core product is matching riders with nearby drivers. Each rider's map must show nearby drivers' positions updating every 2-3 seconds. Each driver sends GPS coordinates every 4 seconds. The system must compute matches in real-time while minimizing bandwidth (drivers are on mobile data).

**Solution:** Uber uses a WebSocket-based real-time platform called Ringpop (their own consistent-hashing-based RPC layer) running on Node.js.

### Cloudflare: Global WebSocket Proxy at Edge

**Scale:** Cloudflare's global network proxies WebSocket traffic for millions of websites. They handle over 1 billion WebSocket connections through their edge network of 330+ data centers.

**Problem:** Websites using Cloudflare for CDN and DDoS protection need WebSocket traffic to pass through Cloudflare's proxy without breaking the persistent connection. The proxy must inspect, route, and optimize WebSocket traffic for millions of concurrent connections.

**Solution:** Cloudflare's edge proxy (nginx-based, later replaced by custom Pingora) handles WebSocket upgrade transparently.

**Key patterns used by Cloudflare:**
- **Auto-TLS:** Cloudflare manages TLS termination at the edge, converting wss:// to ws:// for the origin server. This offloads TLS cost from the origin.
- **WebSocket-aware DDoS protection:** The edge proxy inspects WebSocket frame headers for attack patterns without decrypting data frames.
- **Argo Smart Routing:** WebSocket traffic is routed over Cloudflare's private backbone, reducing cross-region latency by 30 percent.
- **Connection coalescing:** Multiple browser tabs to the same origin share a single WebSocket connection through the edge.

**Result:** Websites behind Cloudflare see 30 percent lower WebSocket latency through Argo routing. DDoS attacks targeting WebSocket endpoints are mitigated at the edge before reaching origin servers. Cloudflare handles over 10 million WebSocket connections per data center.

## Open Source Implementations

### Nginx: Reverse Proxy for WebSocket

Nginx supports WebSocket proxying since version 1.3. It is the most common WebSocket proxy in production.

```nginx
# nginx.conf — WebSocket proxy configuration
map $http_upgrade $connection_upgrade {
    default  upgrade;
    ''       close;
}

upstream websocket_backend {
    ip_hash;  # Sticky sessions by client IP
    server 10.0.1.1:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.2:8080 max_fails=3 fail_timeout=30s;
}

server {
    listen 443 ssl;
    server_name ws.example.com;

    ssl_certificate /etc/ssl/certs/example.crt;
    ssl_certificate_key /etc/ssl/private/example.key;

    location /ws {
        proxy_pass http://websocket_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;

        # Critical for WebSocket: increase timeouts
        proxy_read_timeout 3600s;     # Don't timeout idle connections
        proxy_send_timeout 3600s;

        # Buffer for WebSocket frames
        proxy_buffer_size 4k;
        proxy_buffering off;          # Disable buffering for streaming
        
        # Client disconnect detection
        proxy_ignore_client_abort off;
    }
}
```

**How Nginx handles WebSocket:**
- When an HTTP Upgrade request arrives, Nginx establishes a connection to the upstream server
- Nginx then enters a tunnel mode: data from client is forwarded to upstream, data from upstream is forwarded to client
- Nginx does not parse WebSocket frames — it just copies bytes between the two connections
- The tunnel persists until either side closes the connection

**Scaling limit:** Each WebSocket connection consumes two file descriptors in Nginx (one for client, one for upstream). With default ulimit, Nginx handles around 50,000 proxied WebSocket connections before hitting fd limits.

### HAProxy: WebSocket-Aware Load Balancer

HAProxy has better WebSocket support than Nginx. It can:
- Inspect the HTTP Upgrade header to route WebSocket traffic differently from HTTP traffic
- Use stick-tables for session persistence without cookies (source IP or URL parameter)
- Set per-server `maxconn` limits for WebSocket connections
- Provide WebSocket-specific health checks (1002 response)

```haproxy
# haproxy.cfg — WebSocket-specific configuration
frontend websocket_front
    bind *:443 ssl crt /etc/ssl/certs/example.pem
    mode tcp                     # Layer 4 mode for WebSocket
    
    # Use TCP mode for WebSocket, HTTP mode for regular traffic
    tcp-request inspect-delay 5s
    tcp-request content accept if { req_ssl_hello_type 1 }
    
    use_backend ws_backend if { path_beg /ws }
    default_backend http_backend

backend ws_backend
    mode tcp
    balance source              # Source IP hash for sticky sessions
    stick-table type ip size 200k expire 24h  # Stick table
    
    option tcpka                # TCP keepalive
    timeout server 3600000      # 1 hour server timeout
    timeout client 3600000      # 1 hour client timeout
    
    server ws1 10.0.1.1:8080 maxconn 10000
    server ws2 10.0.1.2:8080 maxconn 10000
```

### Redis: Pub/Sub and Connection State

Redis is the most common companion to WebSocket servers. It serves multiple roles:

**Redis Pub/Sub:**
- Lightweight, sub-millisecond publish latency
- No message persistence (messages lost if subscriber disconnected)
- Best for: real-time presence, typing indicators, ephemeral notifications
- Worst for: reliable message delivery, audit trails

**Redis Streams:**
- Messages persist on disk
- Consumer groups with offset tracking
- Best for: cross-node message relay with replay on reconnect
- Worst for: high-throughput (>1M msg/s) — Kafka performs better

**Redis as session store:**
- `SET ws:session:TOKEN userId serverId timestamp EX 86400`
- On reconnect, server reads the token to restore state
- Best for: server restarts and connection migration
- Worst for: very large session data (over 10 MB) — use S3 or DynamoDB

### Kafka: Durable Message Backbone

Kafka is the backbone for WebSocket systems requiring exactly-once delivery guarantees.

**How Kafka fits:**
- **Writers:** WebSocket servers produce messages to Kafka topics
- **Readers:** WebSocket servers consume from Kafka topics to deliver to connected clients
- **Consumer group:** All servers in the pool form a single consumer group. Kafka partitions the message load across servers
- **Offsets:** Each client's last consumed offset is tracked. On reconnect, the server seeks to the client's offset and replays

**Why Kafka over Redis for production systems:** Kafka provides durability (messages survive broker crashes), replayability (re-read any message from any offset), and higher throughput (millions of messages/second). Redis Pub/Sub loses messages on network partition or subscriber disconnection.

## Cloud Provider Implementations

### AWS

| Service | WebSocket Support | Cost Model | Best For |
|---------|------------------|------------|----------|
| API Gateway (V2) | Native WebSocket | Per-connection (0.80 USD/month per 1M connections) + per-message | Serverless WebSocket, low traffic |
| ALB | WebSocket tunnel, 60s idle timeout* | Per-hour + per-LCU | Simple proxying to EC2/ECS |
| NLB | WebSocket tunnel, 350s idle timeout* | Per-hour + per-NLCU | High-throughput, TLS pass-through |
| IoT Core | MQTT over WebSocket | Per-message | IoT devices, message broker |

*Configurable higher.

**AWS API Gateway WebSocket limitations:**
- Maximum message size: 128 KB
- Per-connection maximum throughput: 10 messages/second
- Connection idle timeout: 10-120 minutes (configurable)
- No binary frame support (automatic base64 encoding)
- No custom handshake headers (you cannot extend the upgrade response)

**When to use API Gateway:** Low-traffic side projects. Prototyping. Serverless architectures where you want no infrastructure management.

**When NOT to use API Gateway:** Any application with over 100,000 concurrent connections, binary messages, or high per-connection throughput (over 10 msg/s). You will hit the hard limits, and you cannot bypass them.

## Lessons Learned

### Pattern 1: Externalize State Early

Every company that scaled WebSocket beyond a single server moved state out of process memory. Discord uses ETS (Erlang Term Storage) for session state. Netflix uses Redis + Cassandra. Uber used Ringpop (consistent hashing) but eventually moved to managed Cloud Pub/Sub. The first milestone is 10K concurrent connections — that is when in-process state becomes a scaling bottleneck.

### Pattern 2: Compression Is Mandatory at Scale

Discord saw 40 percent bandwidth reduction with zlib-stream compression. Uber achieved 84 percent reduction by switching from JSON text frames to Protobuf binary frames. Mobile users on metered connections benefit enormously. The CPU cost of compression (1-3 microseconds per 1 KB message) is far lower than the bandwidth cost of uncompressed traffic.

### Pattern 3: Reconnection Strategy Determines Service Reliability

The thundering herd after a server restart can cascade faster than auto-scaling can respond. Every company implements a version of exponential backoff with jitter. Discord uses a 1-2-4-8-16-30-30 second staggered backoff. Netflix uses 1-3-5-10-30 second with random jitter. Cloudflare's edge proxies buffer messages for 15 seconds on client disconnect, allowing the client to reconnect without message loss.

### Pattern 4: Know When to Move Away from Custom Infrastructure

Uber built Ringpop (custom consistent-hashing-based WebSocket routing) and ran it for years. But the operational burden of a custom distributed system (rebalancing, split-brain, debugging) eventually outweighed the benefits. They migrated to GCP Cloud Pub/Sub, a managed service. The migration was expensive but reduced PagerDuty alerts by 80 percent.

### Pattern 5: Binary Frame Protocol Over JSON at High Throughput

Above 10,000 messages/second, JSON serialization and deserialization becomes a significant CPU cost. Every company operating at that scale uses a binary framing protocol: Protobuf, MsgPack, CBOR, or FlatBuffers. The savings are not just in CPU — binary frames are 50-80 percent smaller than equivalent JSON, reducing bandwidth, serialization time, and memory allocation pressure.
