# Capacity Estimation — Tradeoffs and Failure Modes

## Core Tension: Accuracy vs. Speed

Every capacity estimate exists on a spectrum. At one end: a 5-second back-of-envelope calculation that gives you a number within 10x of reality. At the other end: a 2-week perf-testing project that gives you a number within 1.1x of reality. The right place on this spectrum depends on what you are doing with the estimate.

### When to Choose Each Level

**Back-of-envelope (30 seconds):**
- System design interviews. You have 45 minutes total.
- Early architecture brainstorming. You need direction, not precision.
- Comparing two design options. You want to know if option A costs 10x more than option B.
- When you have zero real traffic data.

**Spreadsheet model (1 hour):**
- Quarterly capacity planning for an existing system.
- Cloud budget forecasting for a startup.
- Pre-sales sizing for an enterprise customer.
- When you have analytics data but no load testing infrastructure.

**Production load test (2 weeks):**
- Before a major product launch with 10x expected growth.
- After a significant architecture change.
- When you need SLAs with cost guarantees.
- When previous estimates were wrong by more than 2x.

## Five Major Failure Modes

### Failure Mode 1: Underestimation Causing Outage

The most visible failure. Your estimate says 50 servers, but peak traffic needs 200. The system goes down.

**Root cause:** Wrong assumptions about traffic. Usually the peak-to-average ratio is much higher than expected. A flash sale, a viral post, or a news event can push traffic 50x above average, not 3x.

**Real-world example:** A food delivery app estimated 5x peak for Valentine's Day. Actual traffic was 40x average. The ordering system went down for 45 minutes. Estimated revenue loss: $2.3 million.

**Fix:** Use 5x to 10x peak multiplier for event-driven systems. Build auto-scaling that can double capacity in 2 minutes.

### Failure Mode 2: Overestimation Wasting Budget

The opposite failure. Your estimate says 200 servers, but traffic only needs 20. You waste $36,000 per month on idle compute.

**Root cause:** Using worst-case assumptions for every dimension. Every multiplier compounds. If you overestimate DAU by 2x, sessions by 2x, actions by 2x, and peak by 2x, your estimate is 16x above reality.

**Real-world example:** A startup building an enterprise tool provisioned 100 servers based on a "conservative" estimate. At launch, they had 2,000 users. Server utilization was 3%. They spent $18,000 per month on idle servers for 6 months before rightsizing.

**Fix:** Use realistic assumptions, not worst-case. Build with auto-scaling from day one. Start with a minimum viable footprint and scale up based on actual traffic.

### Failure Mode 3: Ignoring Peak vs. Average

The most common mistake in interviews and real systems. Engineers calculate average QPS and design for that number.

```
Average QPS: 10,000
Provisioned: 10 servers (1,000 QPS each)

But peak QPS is 40,000.
At peak: 40,000 / 10 servers = 4,000 QPS per server
Server max: 1,000 QPS → 4x overload
Result: 5-second response times, timeouts, errors
```

**Fix:** Always estimate peak, not average. Design for peak. Use auto-scaling to handle the difference between average and peak.

### Failure Mode 4: Not Accounting for Growth

A system designed for today's traffic fails in 3 months when user count doubles.

```
Month 1: 5M DAU, 5,000 avg QPS, 10 servers
Month 3: 10M DAU, 10,000 avg QPS, 20 servers needed
Capacity: 10 servers → 50% utilization → 90% → outage

Growth of 100% in 2 months with flat capacity = guaranteed failure.
```

**Fix:** Apply a growth multiplier based on the product's growth rate. SaaS products growing at 15% month-over-month need capacity for 370% of current load after 12 months. Plan quarterly capacity reviews.

### Failure Mode 5: Wrong Read/Write Ratio

The ratio of reads to writes determines database architecture. A 100:1 read-heavy system needs replicas and caching. A 1:10 write-heavy system needs write-optimized storage.

```
Estimated: 90% reads, 10% writes → build with replicas + cache
Actual: 50% reads, 50% writes → replicas underutilized, write path overloaded

Result: cache hit ratio is high, but write latency degrades because
the database is not optimized for heavy writes. Queue for writes
grows unboundedly. System becomes unstable.
```

**Fix:** Validate read/write ratio with real traffic data or similar systems. Do not assume your ratio matches a generic social media profile.

## Anti-Patterns

### Anti-Pattern 1: Estimating to Six Decimal Places

You see estimates like "we need 47.382 servers." This is false precision. You do not know your DAU, actions per session, or peak multiplier to three decimal places — the server count estimate is at best within 30% accuracy. Expressing it to six decimal places is misleading.

**Do this:**
```
Estimated servers: 50 to 70 (range)
Best guess: 60 servers
Confidence: +/- 30%
```

**Not this:**
```
Estimated servers: 62.417
Budget: $12,483.40/month
```

### Anti-Pattern 2: Copy-Pasting from Other Systems

System A social media platform with 50 million DAU uses 200 servers, so System B with 50 million DAU should also use 200 servers.

This ignores different architectures, different features, different action counts, and different peak patterns. Instagram and Twitter both have 500 million+ users, but their storage and bandwidth profiles are completely different — Instagram handles large images and video, Twitter handles short text.

### Anti-Pattern 3: Ignoring Protocols and Encodings

"Each request is 1 KB" — but that is the payload. With HTTP/1.1 headers, TLS handshake overhead, and base64 encoding, the actual wire cost is 3 KB to 5 KB. A 100 million request system at 5 KB instead of 1 KB costs 5x more in bandwidth.

HTTP/2 multiplexing reduces overhead. gzip compression reduces payload. Protobuf encoding reduces size. Each decision affects your bandwidth estimates.

### Anti-Pattern 4: Single-Number Estimates

Estimating only servers without considering storage, bandwidth, or memory leads to surprises. You provision enough compute, but the database runs out of disk space. The network link saturates. The cache eviction rate spikes.

**Do this:**
```
Compute: 60 servers (4 CPU, 16 GB RAM)
Database: 3 nodes (2 TB SSD each)
Cache: 200 GB Redis cluster
Bandwidth: 10 Gbps link
Storage: 10 TB object store (S3 equivalent)
```

**Not this:**
```
We need 60 servers.
```

### Anti-Pattern 5: Forgetting Overhead

A server can handle 1,000 QPS — on a perfect benchmark. In production, the same server handles 300 QPS because of garbage collection, logging, monitoring agents, health checks, background jobs, and OS overhead.

**Rule of thumb:** Production throughput is 30% to 50% of benchmark throughput.

```
Benchmark: 1,000 QPS per server
Production estimate: 300 to 500 QPS per server

Better to estimate low and have headroom than estimate high
and have an outage.
```

## When NOT to Estimate Capacity

Capacity estimation is not always the right tool. Skip it in these situations:

**Tiny systems.** A side project with 100 users does not need capacity estimation. The cost of the estimation time exceeds the potential savings. Just provision one server and move on.

**Prototypes and MVPs.** Speed of iteration matters more than efficiency. Over-provision by 10x and use the cheapest cloud instances. If the prototype fails, you saved nothing by optimizing. If it succeeds, you can optimize later.

**Serverless architectures.** Lambda, Cloud Functions, and similar services handle capacity automatically. You still need to estimate cost (your bill scales with invocations) but you do not need to estimate server counts. The tradeoff is higher per-request cost in exchange for zero capacity management.

**Systems with trivial resource needs.** A service that handles 10 requests per minute does not need a spreadsheet. Just pick the smallest instance type and check it once.

**When data already exists.** If you have production monitoring with real traffic data, use the actual numbers rather than estimates. The monitoring data IS your capacity plan — just add headroom and growth projections.

## Getting Stuck: Analysis Paralysis

A common failure mode that combines both waste and missed deadlines: spending so much time refining the estimate that the system ships late or never ships.

**Symptoms:**
- Tweaking DAU assumptions from 10.2M to 10.5M (adds nothing)
- Running a third load test because the first two had "paging artifacts"
- Waiting for "real user data" when you could ship and measure
- Building a 20-tab spreadsheet for a 2-server system

**Cost matrix for 1 week of over-analysis:**

```
Salary cost of 1 engineer for 1 week: $4,000
Opportunity cost of delayed launch: $50,000 to $500,000 (lost revenue)

Total cost of over-analysis: $54,000 to $504,000 compared to
a 1-hour estimate that would have been within 2x of final needs.
```

**Fix:** Set a timer. Five minutes for interview-level estimates. One hour for quarterly planning. Two weeks for launch-critical estimates. When the timer rings, ship the estimate and add a monitoring dashboard.

## Sensitivity Analysis

Not all assumptions in your estimate matter equally. Sensitivity analysis identifies which assumptions have the biggest impact on the final number.

**Example: Photo-sharing app estimate**

| Assumption | Base Value | 2x Change | Impact on Server Count | Impact on Cost |
|-----------|-----------|-----------|----------------------|---------------|
| DAU | 1M | 2M | +100% servers | +100% cost |
| Photos/user/day | 0.5 | 1.0 | +10% servers | +50% storage cost |
| Photo size | 2 MB | 4 MB | 0% servers | +100% storage cost |
| Peak multiplier | 5x | 10x | +100% servers | +100% compute cost |
| Requests/user | 50 | 100 | +100% servers | +100% compute cost |
| Cache hit ratio | 90% | 95% | -50% DB servers | -50% DB cost |
| Replication factor | 3x | 2x | 0% | -33% storage cost |

**Key findings from sensitivity:**
- DAU, peak multiplier, and requests per user have the largest impact on server count.
- Photo size and photos per user have the largest impact on storage.
- Cache hit rate has outsized impact on database cost.

Focus your estimation effort on the high-sensitivity assumptions. If DAU is uncertain by a factor of 2, your server estimate is uncertain by a factor of 2 regardless of how precisely you estimate everything else.

## Cost Optimization Through Estimation

Capacity estimation directly enables cost optimization. Every dimension you estimate is a dimension you can optimize.

### Compute Optimization

```
Before estimation:
  "We need 100 servers" (guess)

After estimation:
  "We need 50 servers for API traffic, 30 for background jobs, 20 for redundancy"
  → Can use different instance types per workload
  → API servers: compute-optimized (c5)
  → Background jobs: general purpose (m5)
  → Redundancy: spot/preemptible instances (60% discount)
  
  Cost comparison:
  All on-demand general purpose: 100 x $200 = $20,000/month
  Split + spot: 50x$180 + 30x$200 + 20x$80 = $16,600/month
  Savings: $3,400/month (17%)
```

### Storage Optimization

```
Before estimation:
  "Store all data on SSDs"

After estimation:
  "Only 30% of data is accessed weekly. Use hot/cold tiering."
  
  Hot tier (30%): SSD at $5/GB/month
  Cold tier (70%): HDD at $0.5/GB/month
  
  100 TB total:
  All SSD: 100 TB x $5 = $512,000/month
  Tiered: 30 TB x $5 + 70 TB x $0.5 = $189,000/month
  Savings: $323,000/month (63%)
  
  Add archive tier for data older than 90 days:
  Archive (40%): object store at $0.02/GB/month
  30 TB x $5 + 30 TB x $0.5 + 40 TB x $0.02 = $165,000/month
  Total savings from estimation: $347,000/month (68%)
```

### Bandwidth Optimization

```
Before estimation:
  "24/7 CDN at full bandwidth"

After estimation:
  "Peak is 4 hours/day, average is 30% of peak"
  
  Pay-as-you-go CDN: matches actual usage
  Committed use discount: reserve 50% of peak for 30% discount
  
  Monthly transfer: 1,000 TB at $0.08/GB = $80,000
  Reserved 50%: 500 TB at $0.056/GB = $28,000
  Remaining 500 TB at $0.08/GB = $40,000
  Total: $68,000
  Savings: $12,000/month (15%)
```

## When to Re-Estimate

Capacity estimates are not static. Revisit them at these triggers:

| Trigger | Action | Reason |
|---------|--------|--------|
| New product launch | Full re-estimate | Entirely new traffic patterns |
| 2x user growth | Re-estimate all dimensions | Old assumptions are stale |
| New feature with heavy resources | Partial re-estimate | Storage, bandwidth, or compute may spike |
| Cloud bill exceeds estimate by 30%+ | Audit assumptions | Something changed or was wrong |
| Production outages from capacity | Re-estimate + load test | Your estimate failed — find the mistake |
| Architecture change (e.g., new DB) | Full re-estimate | Different resource profile entirely |
| Quarterly planning cycle | Review and adjust | Standard hygiene |

## Summary of Tradeoffs

## The Estimation Cost-Benefit Framework

Every estimation effort should pass a cost-benefit test:

```
Estimation ROI = (Cost of being wrong × Probability of being wrong) / Cost of estimating

If ROI > 1: estimate
If ROI under 1: guess and go
```

**Example 1: Deploying a prototype.**
```
Cost of being wrong: $100 (extra server for a month)
Probability of being wrong without estimation: 80%
Cost of estimating: $200 (1 hour of engineer time)

ROI = ($100 × 80%) / $200 = 0.4
ROI under 1 → Do not estimate. Just deploy and adjust.
```

**Example 2: Launching to 1 million users.**
```
Cost of being wrong: $50,000 (outage, lost revenue, reputation)
Probability of being wrong without estimation: 90%
Cost of estimating: $400 (2 hours of engineer time)

ROI = ($50,000 × 90%) / $400 = 112.5
ROI > 1 → Estimate thoroughly.
```

**Example 3: Building a data center.**
```
Cost of being wrong: $50,000,000 (over-provisioned by 50%)
Probability of being wrong without estimation: 100%
Cost of estimating: $100,000 (team of 5 for 2 weeks)

ROI = ($50,000,000 × 100%) / $100,000 = 500
ROI >> 1 → Invest heavily in estimation.
```

This framework makes the decision quantitative: estimate only when the cost of being wrong significantly exceeds the cost of estimating.

## Uncertainty Propagation

Every assumption in your estimate has uncertainty. These uncertainties compound through multiplication.

**Simple example:**
```
DAU estimate: 10M +/- 30% (range: 7M to 13M)
Actions per user: 20 +/- 20% (range: 16 to 24)
Peak multiplier: 5x +/- 40% (range: 3x to 7x)

Best case: 7M x 16 x 3 = 336M daily actions
Worst case: 13M x 24 x 7 = 2,184M daily actions
Range: 336M to 2,184M — a factor of 6.5x uncertainty
```

Each individual assumption has modest uncertainty (20% to 40%), but combined they create a 6.5x spread. This is why you should not present a single number as your estimate — give a range.

**Better presentation:**
```
Expected QPS: 23,000
Range: 8,000 to 50,000
Confidence: Medium (new system, no historical data)
```

**For interview:**
```
"Based on my assumptions, I estimate roughly 23,000 QPS.
The range could be 10,000 to 50,000 depending on actual DAU and peak ratio.
The most important unknown is the peak multiplier — can you confirm a reasonable value?"
```

## Tooling for Estimation

Spreadsheets, calculators, and reference tables reduce the mental overhead of estimation.

### The Estimation Spreadsheet Template

A good capacity estimation spreadsheet has rows for each dimension and columns for average, peak, and headroom values.

```
Column layout:
  A: Assumption name
  B: Value
  C: Unit
  D: Notes
  E: Confidence (Low/Medium/High)

Row examples:
  DAU                   40,000,000  users   Confirmed from product team     High
  Actions/user/day      20          actions Feed + likes + messages        Medium
  Total daily actions   800,000,000 actions B x C
  Avg QPS               9,259       QPS     D / 86,400
  Peak multiplier       4x          factor  Social media typical
  Peak QPS              37,037      QPS     E x F
  Server capacity       750         QPS     Standard API server benchmark
  Servers needed        50          servers G / H
  Headroom              30%         factor  Standard practice
  Total servers         65          servers I + (I x J)
```

### Pre-Computed Reference Values

Memorize these common conversions to avoid mental math errors:

```
1 million DAU × 10 actions/day = 116 QPS (roughly 100 QPS for mental math)
1 million DAU × 50 MB storage each = 50 TB (before replication)
1 TB/day at 1 KB per write = 11.6 million writes/day
1 Gbps bandwidth = 125 MB/s = 125,000 KB/s
86,400 seconds/day = 100K in rough math (8.6K — 7% error)
```
