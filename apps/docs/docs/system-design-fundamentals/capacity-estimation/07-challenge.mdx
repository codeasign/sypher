# Capacity Estimation Challenge

This section contains three tiers of capacity estimation problems. Each tier increases in difficulty. Work through each problem using the five-minute framework. Model answers are provided after each problem.

---

## Tier 1: URL Shortener

**Problem:** Estimate the infrastructure needed for a URL shortener service like bit.ly. The service has 10 million registered users and 50 million shortened URLs created to date.

**Clarifications:**
- 2 million new shortened URLs created per month
- Each shortened URL gets viewed 10 times on average per month
- Monthly active users: 5 million
- Short URLs expire after 2 years

**Your task:**
1. Estimate DAU
2. Estimate QPS (reads and writes)
3. Estimate storage for 2 years
4. Estimate bandwidth
5. Estimate server count

### Model Answer: Tier 1

**Step 1: Estimate DAU**

```
5 million MAU
Assume 40% daily active rate (typical for utility tools)
= 2 million DAU
```

**Step 2: Estimate QPS**

```
Writes (new short URLs):
  2 million new URLs per month
  / 30 days = 66,667 new URLs per day
  / 86,400 = 0.77 writes per second
  Peak (5x): 4 writes per second

Reads (redirects):
  Each URL is viewed 10 times per month on average
  50 million existing URLs x 10 views/month = 500 million views/month
  Each new URL adds 2M x 10 = 20 million views/month in month 2
  Average over 2 years: roughly 500M views/month
  / 30 days = 16.7M views/day
  / 86,400 = 193 reads per second
  Peak (5x): 965 reads per second

Total peak QPS: 4 writes + 965 reads = 969 QPS
Read-to-write ratio: approximately 250:1
```

**Step 3: Estimate Storage**

```
Each URL entry:
  Short code (7 chars): 7 bytes
  Original URL: 200 bytes average
  Created timestamp: 8 bytes
  Expiry timestamp: 8 bytes
  Click count: 8 bytes
  Metadata: 40 bytes
  Total per entry: roughly 300 bytes

Total entries over 2 years:
  50 million existing + (2 years x 12 months x 2M new/month)
  = 50M + 48M = 98 million entries

Total storage:
  98M x 300 bytes = 29.4 GB

Indexes (primary key on short code): roughly 20% overhead
  = 5.9 GB

Replication (3x): 35.3 GB x 3 = 106 GB

Total raw storage needed: roughly 110 GB
```

**Step 4: Estimate Bandwidth**

```
Write bandwidth:
  4 peak writes per second
  x 300 bytes average request
  = 1.2 KB/s = 9.6 Kbps

Read bandwidth:
  965 peak reads per second
  x 300 bytes response (redirect + metadata)
  = 289.5 KB/s = 2.3 Mbps

Total bandwidth: roughly 2.3 Mbps
```

**Step 5: Estimate Server Count**

```
Peak QPS: 969
Per-server capacity for dynamic requests: 750 QPS
Servers needed: 969 / 750 = 1.3 servers

+ 50% headroom = 2 servers
+ 1 server for redundancy = 3 servers

At this scale, a single web server with a database backend
handles the entire load. Use 3 servers for availability:
2 web + 1 database (or 3 web with a managed database service).

Observation: storage (110 GB) and bandwidth (2.3 Mbps) are trivial.
The system is constrained by availability requirements, not capacity.
```

**Final Answer:**
```
Servers:       3 web (or 2 web + 1 database)
Storage:       150 GB (SSD, providing headroom)
Bandwidth:     under 10 Mbps
Database:     Single PostgreSQL or MySQL instance
Cache:        Optional — 2 GB Redis for hot URL lookups
Cost:         Roughly $300 to $500 per month
```

## Tier 2: Real-Time Chat Application

**Problem:** Estimate the infrastructure needed for a real-time chat application like WhatsApp or Telegram. The app has 100 million DAU.

**Clarifications:**
- Each user sends 40 messages per day on average
- 90% of messages are text (1 KB each)
- 10% of messages are images (500 KB each after compression)
- Messages are stored permanently
- Users belong to groups; group messages are fanned out to all members
- Average group size: 10 members
- 30% of messages are sent to groups, 70% are direct (1-to-1)
- Users have an average of 150 contacts and 20 active groups
- Peak multiplier: 3x

**Your task:**
1. Estimate total messages per day (including group fan-out)
2. Estimate QPS for reads and writes
3. Estimate storage per day and per year
4. Estimate server count
5. Estimate bandwidth
6. Identify the biggest bottleneck

### Model Answer: Tier 2

**Step 1: Total Messages Per Day**

```
Direct messages:
  100M DAU x 40 messages/day x 70% = 2.8B direct messages/day

Group messages:
  100M DAU x 40 messages/day x 30% = 1.2B group message sends/day
  Each group message is fanned out to 10 members
  = 1.2B x 10 = 12B message deliveries/day

Total message writes (including fan-out):
  2.8B + 12B = 14.8B message writes per day

Total message reads (viewing):
  Each user opens the app 10 times per day
  Each open loads 50 recent messages
  = 100M x 10 x 50 = 50B message reads per day
```

**Step 2: QPS**

```
Writes:
  14.8B / 86,400 = 171,296 writes per second average
  Peak (3x): 513,889 writes per second

Text writes: 90% x 171,296 = 154,166/s average, 462,500/s peak
Image writes: 10% x 171,296 = 17,130/s average, 51,389/s peak

Reads:
  50B / 86,400 = 578,704 reads per second average
  Peak (3x): 1,736,111 reads per second
```

**Step 3: Storage**

```
Daily storage growth:
  Text messages: 14.8B x 90% x 1 KB = 13.32 TB/day
  Images: 14.8B x 10% x 500 KB = 740 TB/day
  Total: 753.32 TB/day

Yearly storage (not all messages are permanent; assume 90% retention):
  753.32 TB x 365 x 90% = 247,465 TB/year = 247.5 PB/year

With 3x replication:
  742.5 PB/year of raw storage

For a 5-year retention policy:
  742.5 x 5 = 3,712 PB = 3.7 EB (exabytes) of raw storage
```

**Step 4: Server Count**

```
Write servers (handling message ingestion):
  513,889 peak write QPS
  / 750 QPS per server = 685 servers
  + 30% headroom = 891 servers

Read servers (handling message retrieval):
  1,736,111 peak read QPS
  Each server handles 5,000 QPS with cache (messages cached in Redis)
  / 5,000 = 348 servers
  + 30% headroom = 453 servers

Image processing servers:
  51,389 peak image uploads/s
  Each image needs compression/resizing: 100 ms per image
  51,389 x 0.1 = 5,139 CPU seconds
  / 4 cores per server = 1,285 servers
  + 30% headroom = 1,671 servers

Total servers: 891 + 453 + 1,671 = approximately 3,015 servers
```

**Step 5: Bandwidth**

```
Write bandwidth:
  513,889 peak messages/s
  90% text at 1 KB = 462,500 KB/s = 462 MB/s = 3.7 Gbps
  10% images at 500 KB = 25,694 MB/s = 25.7 GB/s = 205.6 Gbps
  Total inbound: 209.3 Gbps

Read bandwidth:
  1,736,111 peak reads/s
  Average response: 50 messages x 1 KB = 50 KB
  1,736,111 x 50 KB = 86,806 MB/s = 86.8 GB/s = 694 Gbps

  Note: reads are dominated by text. Images are fetched separately.
  Image fetch bandwidth: 5% of reads include an image
  1,736,111 x 5% x 500 KB = 43,403 MB/s = 43.4 GB/s = 347 Gbps

Total outbound: 694 + 347 = 1,041 Gbps
```

**Step 6: Biggest Bottleneck**

The biggest bottleneck is storage. 247.5 PB per year of raw data (742.5 PB replicated) requires a massive object store — comparable to YouTube-level storage volume despite being a "text messaging" app. Images dominate.

The image pipeline is also a bottleneck. 1,671 servers just for image processing means either a very large auto-scaling group or a shift to serverless (Lambda at 51K invocations per second is expensive but manageable).

```
Bottleneck ranking:
  1. Storage: 742.5 PB/year replicated — requires tiered storage
  2. Image processing: 1,671 servers — consider Lambda or dedicated fleet
  3. Outbound bandwidth: 1,041 Gbps — multiple 100 Gbps links needed
  4. Write QPS: 513,889/s — requires sharded database
  5. Read QPS: 1.7M/s — cached, manageable with Redis
```

**Final Answer:**
```
Servers:       ~3,000 (1,700 image processing, 900 write, 400 read)
Database:     Sharded Cassandra or ScyllaDB for messages
Cache:        Redis cluster, 20+ TB for hot messages
Object store: 750+ PB/year (S3/Blob Storage)
CDN:          Serves image downloads
Bandwidth:    1+ Tbps outbound
Image pipeline: Massive auto-scaling group or serverless
Key problem:  Images make this a storage-heavy system,
              not just a message-passing system.
```

## Tier 3: Video Streaming Platform

**Problem:** Estimate the infrastructure needed for a YouTube-scale video streaming platform. The platform has 500 million DAU.

**Clarifications:**
- Users watch 60 minutes of video per day on average
- Average bitrate: 5 Mbps (HD streaming)
- 500 hours of video are uploaded every minute
- Average upload size: 500 MB per minute of video (compressed)
- Videos need transcoding to 3 resolutions (480p, 720p, 1080p)
- Transcoding takes: 2x real-time per resolution (so 1 minute of video = 6 minutes of transcoding CPU)
- 20% of views are live streams, 80% are pre-recorded
- Cache hit ratio for CDN: 90% for popular content
- Peak-to-average ratio: 4x
- Store all videos permanently

**Your task:**
1. Estimate total watch time per day and peak concurrent streams
2. Estimate CDN bandwidth
3. Estimate upload and transcoding capacity
4. Estimate storage per day and total storage
5. Estimate server count for upload and transcoding
6. Estimate total bandwidth
7. Calculate the monthly CDN cost

### Model Answer: Tier 3

**Step 1: Watch Time and Concurrent Streams**

```
Total watch time per day:
  500M DAU x 60 minutes = 30 billion minutes/day
  = 500 million hours/day

Average concurrent streams:
  30B minutes/day / 1,440 minutes/day = 20.8M average concurrent

Peak concurrent streams (4x):
  20.8M x 4 = 83.3M concurrent streams

Peak streaming bandwidth:
  83.3M streams x 5 Mbps = 416,500 Gbps = 416.5 Tbps
```

**Step 2: CDN Bandwidth**

```
CDN handles 90% of views (popular content cache hit).

Peak CDN bandwidth needed:
  416.5 Tbps x 90% = 374.9 Tbps served from CDN
  416.5 Tbps x 10% = 41.7 Tbps served from origin servers

CDN monthly data transfer:
  416.5 Tbps average (not peak, assume 30% of peak sustained)
  = 124.95 Tbps average
  124.95 Tbps = 124,950 Gbps
  x 60 x 60 x 24 x 30 seconds/month
  = 124,950 x 2,592,000 = 323,870,400,000 Gbits/month
  = 40,483,800,000 GB/month = 40.5 exabytes per month
```

**Step 3: Upload and Transcoding**

```
Upload volume:
  500 hours uploaded per minute
  = 500 x 60 = 30,000 hours uploaded per hour
  = 720,000 hours per day

Each hour uploaded:
  Original video at 500 MB/minute = 30 GB/hour

Upload storage per day:
  720,000 hours x 30 GB = 21,600,000 GB/day = 21.6 PB/day

Transcoding:
  500 hours/min x 60 min = 30,000 hours of video uploaded per hour
  Each hour needs 3 resolutions x 2x real-time = 6x real-time transcoding
  Total transcoding: 30,000 hours/hour x 6 = 180,000 transcoding hours per hour
  = 180,000 CPU-hours per hour

  If one server transcodes 4x real-time (one hour of video in 15 minutes):
  30,000 hours/hour upload, need 6x = 180,000 hours transcoding/hour
  Each server does 4 transcoding units (4x real-time)
  Servers needed: 180,000 / 4 = 45,000 servers
```

**Step 4: Storage**

```
Daily upload storage:
  21.6 PB/day of original video

Transcoded copies (3 resolutions):
  Each transcoded copy is roughly 30% smaller than original (compression)
  But there are 3 copies
  
  480p: 30% of original = 9 GB/hour
  720p: 60% of original = 18 GB/hour
  1080p: 100% of original = 30 GB/hour

  Total transcoded per hour uploaded: 9 + 18 + 30 = 57 GB
  Original: 30 GB

  Total storage per hour of upload: 87 GB (original + 3 transcoded copies)

Daily storage growth:
  720,000 hours x 87 GB = 62,640,000 GB/day = 62.64 PB/day
  With 3x replication for durability: 187.9 PB/day

Yearly storage:
  187.9 PB x 365 = 68,583 PB/year = 68.6 EB/year
```

**Step 5: Upload and Transcoding Server Count**

```
Upload servers:
  720,000 hours uploaded per day
  = 30,000 GB/day
  = 347 MB/s average upload
  Peak (4x): 1.4 GB/s

  Each upload server handles 100 concurrent uploads at 10 MB/s each
  = 1 GB/s per server
  1.4 / 1 = 2 upload servers (at peak)
  Add headroom: 4 upload servers

  Actually, upload is trivial. The bottleneck is transcoding.

Transcoding servers:
  45,000 servers for real-time transcoding

  But not all uploads need immediate transcoding.
  80% are pre-recorded: can be transcoded asynchronously within 24 hours
  20% are live: need real-time transcoding

  Pre-recorded: 30,000 hours/hour x 80% = 24,000 hours/hour to transcode within 24 hours
    = 24,000 x 24 hours / 24 hours = 24,000 server-hours (parallelized)
    = roughly 5,000 servers at 5x utilization

  Live: 30,000 x 20% = 6,000 hours/hour of live video, needs immediate transcoding
    = 6,000 x 6 = 36,000 transcoding hours per hour
    Each server does 4x real-time
    36,000 / 4 = 9,000 servers for live

  Total transcoding fleet: 5,000 (batch) + 9,000 (live) = 14,000 servers
```

**Step 6: Total Bandwidth**

```
Outbound (streaming):
  Peak: 416.5 Tbps
  CDN handles 90% = 374.9 Tbps from CDN edge
  Origin handles 10% = 41.7 Tbps

  CDN egress cost varies. At $0.02/GB for large CDN deals:
  40.5 EB/month x $0.02/GB = $810,000,000 per month in CDN costs

Inbound (upload):
  21.6 PB/day upload = 2.5 Tbps average, 10 Tbps peak

Internal bandwidth (between data centers):
  Replication: 187.9 PB/day = 21.7 Tbps average
  Transcoding reads/writes: similar scale
```

**Step 7: Monthly CDN Cost**

```
Monthly CDN data transfer: 40.5 EB
At negotiated YouTube-scale CDN rate: ~$0.005 to $0.01/GB
$0.01/GB x 40.5 x 10^9 GB = $405 million per month

Origin bandwidth:
  Each origin server sends data to CDN at peak
  41.7 Tbps from origin / 10 Gbps per server = 4,170 origin servers

Total estimated monthly cost:
  CDN: $405M at $0.01/GB (YouTube-scale pricing is likely lower)
  Transcoding compute: 14,000 servers x $500/month = $7M
  Storage: 68.6 EB/year, tiered (hot/cold)
    Assume 70% cold at $0.005/GB/month, 30% hot at $0.02/GB/month
    68.6 EB/year = 5.7 EB/month
    5.7 EB x 70% x $0.005 = $20M
    5.7 EB x 30% x $0.02 = $34.2M
    Total storage: $54.2M/month
  Upload compute: negligible
  Internal networking: ~$5M/month
  Data center power/cooling: ~$10M/month

  Total: roughly $481M/month, or $5.8B per year
```

**Final Answer:**

```
Concurrent streams:   83.3M peak
CDN bandwidth:        374.9 Tbps peak
Origin bandwidth:     41.7 Tbps peak
Transcoding servers:  14,000 (9,000 live, 5,000 batch)
Upload ingest:        21.6 PB/day
Total storage:        68.6 EB/year (growing)
Monthly CDN cost:     ~$405M at $0.01/GB
Total monthly cost:   ~$481M

Biggest bottleneck:   CDN cost (84% of total spend)
Second bottleneck:    Storage growth at 189 PB/day replicated
Third bottleneck:     Transcoding latency for live streams

Optimization levers:
1. Reduce bitrate with better codecs (AV1 saves 30% vs. H.264)
2. Tiered storage — cold content to cheap HDD/tape
3. CDN peering — direct connections reduce transit costs
4. Regional distribution — serve from local edge nodes
```

Choose a system you use daily and run the full estimation framework:

1. Define the user base (DAU)
2. Estimate actions per user
