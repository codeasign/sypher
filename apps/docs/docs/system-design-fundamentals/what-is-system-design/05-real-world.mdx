The previous sections covered the theory, the internals, and the trade-offs. This section examines how real companies applied system design to solve specific scaling problems. Each case study includes the scale they operated at, the specific problem they faced, what they built, and the measurable impact.

## Company Case Studies

### Case Study 1: Netflix — Microservices and Chaos Engineering

**Scale:** 260 million subscribers (as of 2025). 5 billion hours of content streamed per month. Peak traffic: 50% of downstream internet bandwidth in North America during evening hours.

**Problem:** In 2008, Netflix ran on a monolithic Java application deployed on a single large server. A database corruption in 2008 caused a three-day outage where DVD shipments stopped. The company realized that their entire business depended on a single Oracle database. They decided to migrate to a cloud-based, microservices architecture before streaming video became their primary business.

**What they built:** Netflix decomposed the monolithic application into over 700 microservices, each responsible for a single capability: user recommendations, video encoding, CDN routing, billing, subscription management, search, and content metadata. Each service has its own database, its own deployment pipeline, and its own team.

Key elements of the architecture:
- **Eureka:** A service registry where each service registers itself and discovers other services via health checks.
- **Zuul:** An API gateway that routes requests to the appropriate service.
- **Hystrix:** A circuit breaker library that prevents cascading failures when a downstream service is slow or unavailable.
- **Chaos Monkey:** A tool that randomly kills production instances during business hours to ensure the system survives instance failures without human intervention.
- **Cassandra:** Used for high-volume, high-availability data like viewing history and recommendations.

### Case Study 2: Uber — Real-Time Dispatch at Global Scale

**Scale:** 130 million monthly active users. 25 million trips per day. Operating in 70+ countries and 10,000+ cities.

**Problem:** Uber's initial architecture was a monolithic Python application called "Micha" that handled dispatch, billing, driver matching, and rider management. As Uber expanded to multiple cities, the monolith could not scale. A single dispatch query evaluated every available driver within a 5km radius for every ride request. In cities with 50,000 drivers, this meant 50 million distance calculations per second — far beyond what Python could handle.

**What they built:** Uber decomposed the monolith into microservices and created domain-specific services for dispatch, pricing, payments, and notifications. The critical innovation was in the dispatch system.

**Dispatch architecture:**
- **Geospatial index:** Instead of checking every driver, use a geospatial index (Google S2 geometry) to find drivers within a specific cell. The world is divided into hierarchical cells. A dispatch query finds the cell containing the rider, checks adjacent cells, and queries only drivers within those cells. This reduces the search space from 50,000 drivers to under 200.
- **Ringpop:** Uber built a consistent hashing ring for distributing driver location updates across servers. Each server handles a subset of geographic cells.
- **Kafka:** Driver location updates (every 4 seconds from each driver's phone) are streamed through Kafka. At 25 million trips per day, that is approximately 6 million location updates per minute.
- **CockroachDB:** A distributed SQL database for transactional data (ride state, payment processing) that is horizontally scalable and ACID-compliant.

### Case Study 3: Instagram — From Monolith to Microservices at Hyperscale

**Scale:** 2 billion monthly active users (2025). 500 million daily active Stories users. 95 million photos and videos uploaded per day. Owned by Meta.

**Problem:** Instagram started as a monolithic Python/Django application running on a single server. By 2010, when Instagram had 14 million users, the monolith was already struggling. The single PostgreSQL database could not keep up with writes. The Redis cache was on the same server as the application, competing for memory. Deployment required a full-system restart.

**What they built:** Instagram's engineering team systematically decomposed the monolith over years. The key architectural decisions:

**Database sharding:** Instagram sharded PostgreSQL by user ID. Each shard (a PostgreSQL instance) stores data for a range of users. The shard key was the user ID modulo the number of shards. When a shard outgrew its capacity, they split it into two shards. A lookup service maps user ID to shard ID.

**Feed generation:** Instagram uses a **pull-based feed**. When you open the app, the feed service queries the 500 most recent photos from each user you follow, merges them sorted by timestamp, and returns the top N. This is the opposite of Twitter's push approach. The difference: Instagram users follow fewer accounts (average 200-300) than Twitter users, making pull feasible.

**Media pipeline:** Uploaded photos go through an asynchronous pipeline: the application writes the photo to S3, emits a message to a queue, and workers resize the photo to 8 different resolutions, extract EXIF data, generate a thumbnail, and run content moderation. The user sees "processing" until all tasks complete.

**Result:** Instagram scaled from 14 million to 2 billion users on the same architectural foundation. The incremental decomposition approach meant no single "rewrite everything" project that risked years of engineering with no user-visible benefit.

**Numbers that matter:**
- 95 million photos uploaded per day
- 8 image resolutions generated per upload = 760 million image processing tasks per day
- 2 billion MAU on a PostgreSQL-backed stack (with 40+ shards)
- Feed generation: under 200ms p99 for users following up to 1,000 accounts

### Case Study 4: Twitter (X) — Timeline Scaling

**Scale:** 330 million monthly active users. 500 million tweets per day. Peak tweet rate: 143,199 tweets per second during major events.

**Problem:** When Twitter launched, user timelines were computed by querying all tweets from followed users at read time. A user following 1,000 accounts would trigger a query joining tweets from those 1,000 accounts. As Twitter grew, this read-time query became impossibly slow. The database could not keep up.

**What they built:** Twitter switched from a **pull model** (query at read time) to a **push model** (pre-compute at write time). When a user tweets, the tweet is copied into the timeline cache of every follower. The timeline read becomes a single cache lookup: "get tweets from my timeline cache."

**The problem with push:** Users with 100 million followers (like celebrities or politicians) cause 100 million cache writes per tweet. Twitter solved this with a **hybrid approach**: users with a small number of followers (under 10,000) get their tweets pushed. Users with a large number of followers (above 10,000) get their tweets pulled at read time.

**Key infrastructure:**
- **Redis clusters:** Each user's timeline is stored in a Redis sorted set, keyed by tweet ID. The sorted set allows pagination (get tweets with IDs lower than X).
- **Fanout service:** A Kafka-based service that reads new tweets and writes them to follower timeline caches. For users with 10,000+ followers, the tweet is added to a "pending pull" list instead.
- **Timeline service:** Reads from Redis. For users who follow high-follower accounts, also queries those accounts' recent tweets and merges them with the cached timeline.

### Case Study 4: Discord — Real-Time Messaging at Scale

**Scale:** 200 million monthly active users. 15 million messages sent per second at peak. 1 billion messages stored per day.

**Problem:** Discord started with MongoDB but hit scaling limits as message volume grew. A single MongoDB shard handled messages for many servers (guilds). When one large guild (like a gaming community with 500,000 members) was active, its messages dominated the shard, causing latency spikes for every other guild on that shard.

**What they built:** Discord migrated from MongoDB to a custom-built distributed database called **ScyllaDB** (a Cassandra-compatible distributed database written in C++). The migration spanned 18 months and required custom tooling to replay all message traffic during the migration window.

**Key architecture decisions:**
- Message storage is **partitioned by guild (server) ID**. All messages for a guild are stored together on the same ScyllaDB node. This ensures that querying a guild's message history is a single-node operation.
- Messages are **immutable** — once written, they are never updated. This simplifies replication and caching.
- **Cache-aside with Redis:** The last 500 messages per guild are cached in Redis. Older messages are read from ScyllaDB.
- **Guild-level rate limiting:** Each guild has a maximum messages-per-second limit. This prevents one large guild from overwhelming the storage system.

### Case Study 5: YouTube — Video Serving at Planetary Scale

**Scale:** 2.5 billion monthly logged-in users. 500 hours of video uploaded every minute. 1 billion hours of video watched per day. Peak bandwidth: 1+ terabit per second.

**Problem:** Serving video at planetary scale requires solving three problems simultaneously: storing exabytes of video, serving it with sub-second start times, and keeping bandwidth costs manageable.

**What they built:**

**Storage:** YouTube stores every video in multiple resolutions (240p through 4K) and formats (MP4, WebM, HLS segments). Each video is stored in Google Cloud Storage (Colossus filesystem) with 3x replication. Total storage: multiple exabytes.

**CDN:** YouTube built its own CDN (Google Global Cache) deployed at over 2,000 internet exchange points worldwide. ISPs host Google cache servers in their own data centers. When a user watches a video, the first request goes to the nearest cache. If the cache does not have the video, it fetches from the origin and caches it for future viewers. Popular videos are cached at the edge. Long-tail videos are served from origin.

**Transcoding pipeline:** When a video is uploaded, it goes through a DAG (directed acyclic graph) of processing stages: demuxing, video decoding, scaling to multiple resolutions, encoding in multiple codecs, audio processing, and packaging. Each stage runs on a separate worker. The pipeline is orchestrated by a custom workflow system.

**Result:** YouTube serves 1 billion hours of video per day with median start time under 2 seconds. The edge caching infrastructure reduces origin bandwidth by 95%+ for popular content.

**Numbers that matter:**
- 2,000+ edge cache locations worldwide
- 500 hours of video uploaded per minute
- 1 billion hours watched per day
- Median video start time: under 2 seconds
- Edge cache hit rate for popular content: over 99%

## Open Source Implementations

The patterns discussed in this course are implemented in production-grade open source software. Studying these systems teaches you how good architecture looks in practice.

### Nginx — Efficient Request Handling

Nginx is a web server and reverse proxy that handles 10,000+ concurrent connections per worker using an event-driven, asynchronous architecture. Unlike Apache (which spawns a thread per connection), Nginx uses a single-threaded event loop with non-blocking I/O.

**Architecture lesson:** Thread-per-connection does not scale. Event-driven architecture with a small number of worker processes (one per CPU core) handles 10x more concurrent connections with 10x less memory.

### Redis — Single-Threaded Speed

Redis is an in-memory data store that uses a single-threaded event loop. All commands are executed sequentially on a single thread. Despite being single-threaded, Redis handles 100,000+ operations per second on modest hardware.

**Architecture lesson:** Avoiding locks and context switches through single-threaded design can outperform multi-threaded designs for specific workloads. Redis proves that simplicity (no locking, no context switching, no contention) beats complexity when the workload fits the model.

### Kafka — Log-Based Architecture

Apache Kafka is a distributed event streaming platform built on a single insight: an immutable, append-only log. Every message is appended to the end of a partition log. Consumers read from the log at their own pace. Messages are never deleted (based on retention policy, not consumption).

**Architecture lesson:** Append-only data structures are the fastest possible write path. By never modifying existing data, Kafka achieves sequential disk writes at disk speed. This is the same insight that powers database write-ahead logs (WAL).

### Cassandra — Hinted Handoff and Gossip

Apache Cassandra is a distributed, wide-column database with no single point of failure. Two key architectural features:

**Hinted handoff:** If a replica node is down when a write arrives, the coordinator stores a "hint" (the data and the intended replica). When the replica comes back, the coordinator delivers the hint. This removes the need for a separate repair process during temporary failures.

**Gossip protocol:** Each node periodically exchanges state information with a few random nodes. Within seconds, every node knows the state of every other node. No central registry needed.

**Architecture lesson:** Decentralized architectures avoid the single point of failure inherent in master-slave setups. Gossip protocols provide eventual consistency without a central coordinator.

## Cloud Provider Implementations

The system design patterns described here are available as managed services from cloud providers. Knowing the mapping between pattern and service is essential for cost-effective design.

| Pattern | AWS | GCP | Azure | When to Use Managed vs Self-Hosted |
|---|---|---|---|---|
| Load balancing | ALB/NLB | Cloud Load Balancing | Azure Load Balancer | Managed: lower ops cost. Self-hosted (HAProxy): more control, lower cost at very high scale. |
| Relational database | RDS/Aurora | Cloud SQL | Azure SQL Database | Managed: automated backups, patching, failover. Self-hosted: 50% cost reduction at enterprise scale. |
| NoSQL database | DynamoDB | Firestore/Datastore | Cosmos DB | Managed: zero ops, auto-scaling. Self-hosted (Cassandra): lower per-request cost at very high volume. |
| In-memory cache | ElastiCache (Redis) | Memorystore | Azure Cache for Redis | Managed: automated failover, patching. Self-hosted: full control over configuration and eviction policies. |
| Message queue | SQS/Kinesis | Pub/Sub | Service Bus | Managed: no server management. Self-hosted (Kafka): lower cost at high throughput, support for replay. |
| Streaming | Kinesis Data Streams | Pub/Sub | Event Hubs | Managed: automatic scaling. Self-hosted (Kafka): more control over partitioning and retention. |
| CDN | CloudFront | Cloud CDN | Azure CDN | Managed: global edge network, easy integration. No competitive self-hosted alternative. |
| Container orchestration | EKS/ECS | GKE | AKS | Managed: reduces Kubernetes ops burden by 70%. Self-hosted: full control, needed for air-gapped environments. |

**The rule of thumb:** Use managed services until the monthly bill exceeds $10,000 for a single service type. Above that threshold, evaluate whether self-hosting saves money. Below that threshold, the operational cost of self-hosting exceeds the infrastructure cost of managed services.

### When to Choose Managed vs Self-Hosted: Detailed Decision Framework

**Choose managed when:**
- Your team has fewer than 5 infrastructure engineers.
- You want to reduce operational burden (backups, patching, failover).
- You need to comply with certifications (SOC2, HIPAA) and want the provider to handle part of the compliance scope.
- Your traffic is predictable and your bill is under $10,000/month per service.
- You need to move fast and cannot afford to become an expert in every infrastructure component.

**Choose self-hosted when:**
- Your traffic volume is high enough that managed service costs exceed the salary of a dedicated infrastructure engineer.
- You need custom configuration that managed services do not support (custom eviction policies, specific hardware, kernel tuning).
- You operate in an air-gapped environment (government, military, on-premises enterprise).
- You have a team of 5+ infrastructure engineers who can run the service better than the cloud provider.
- Your managed service bill for a single component exceeds $50,000/month.

**The hybrid approach:** Most large-scale systems use a mix. Netflix uses managed (AWS) for compute but self-hosts its own CDN (Open Connect). Discord uses managed GCP for compute but self-hosts ScyllaDB. The pattern: use managed services for standard components (load balancers, DNS, compute), self-host for high-volume or custom components (databases, caches, CDN).

### The Failure That Taught the Industry: AWS S3 Outage (2017)

On February 28, 2017, Amazon S3 in us-east-1 experienced a 4-hour outage that affected a significant portion of the internet. Netflix, Slack, Trello, Medium, and thousands of other services were impacted.

**Root cause:** An engineer was debugging a billing system issue and typed a command to remove a small number of servers. A typo in the command caused a much larger set of servers to be removed than intended. The removal cascaded — removing one server caused others to restart, and the restart caused a metadata server overload that prevented re-registration.

**The architectural lesson:** Human error is inevitable. The system must be designed to survive human error at least as well as it survives hardware failure. Amazon's post-mortem identified several safeguards that should have prevented the cascade: slower removal rate, circuit breakers on metadata server load, and automatic re-throttling when error rates increase.

**The fix:** AWS added gradual removal rates, automatic throttling when error rates increase, and a "stop" button that halts all removal operations when an anomaly is detected.

### Migrations That Failed and What They Teach

**The Knight Capital Glitch (2012):** A deployment of new trading software to 7 servers failed — only 1 of the 7 servers received the update. The old and new software ran simultaneously. The old software interpreted market data differently from the new software. Knight Capital lost $460 million in 45 minutes. The company was bankrupt 3 months later.

Lesson: Every deployment must be atomic (all servers or none), and deployment scripts must verify the state match between intended and actual after every step.

**The GitLab Database Incident (2017):** A database administrator accidentally deleted the production database while restoring a replica from backup. The backup was discovered to be non-functional. GitLab lost 6 hours of production data permanently.

Lesson: Test your backup restoration process. A backup that has never been restored is not a backup — it is a hope.

## Lessons Learned

These five lessons distill the experience of the world's largest internet services.

**Lesson 1: Start simple, decompose later.** Netflix started as a monolith. Uber started as a monolith. Twitter started as a monolith. Every company that succeeded with microservices started with a monolith, hit a scaling wall, and decomposed the specific part of the system that hit the wall. No one started with 700 microservices.

**Lesson 2: Measure before you optimize.** Twitter's hybrid push/pull model was not the first idea they tried. They measured the write amplification problem (100 million writes per celebrity tweet) and designed a solution specific to that measurement. Without the measurement, they would have optimized the wrong thing.

**Lesson 3: Partition by the hot key.** Discord's insight was to partition by guild ID so that hot guilds are isolated to their own nodes. If you partition by a field that concentrates traffic, hot shards overload. If you partition by a field that distributes traffic evenly, hot tenants degrade neighbors.

**Lesson 4: Assume everything fails.** Netflix's Chaos Monkey was built on the assumption that servers fail, networks partition, and data centers go offline. By designing for failure from the start, Netflix eliminated the element of surprise. Every failure was a drill, not an emergency.

**Lesson 5: Latency is a product feature.** Amazon computed that 100ms of extra latency cost 1% in revenue. Google found that a 500ms delay reduced search traffic by 20%. Discord treats sub-10ms latency as a non-negotiable feature, not a performance optimization. Latency is not just a technical metric — it directly affects revenue, engagement, and user retention.
