The previous section examined what happens inside each component of a system. This section zooms out to see how components fit together at scale. You will trace a system from its naive starting point through production-grade architecture, across multiple data centers, and through the capacity planning math that determines whether a design survives its first million users.

## Naive Implementation

Every system starts simple. A single server running a web framework, connected to a single database. This works for prototyping, demos, and the first hundred users. It fails catastrophically at scale.

### What Breaks at 1,000 Users

At 1,000 daily active users (DAU), the naive system handles everything with ease. The server CPU sits at 5-10%. The database handles 50-100 QPS with sub-millisecond queries. Response time is under 20ms. The team celebrates their fast application. They think they are done.

### What Breaks at 100,000 Users

At 100,000 DAU, cracks appear. The server handles 5,000-10,000 RPS. The application server CPU hits 80-90%. The database handles 500-1,000 QPS. Response time rises to 200-500ms. Users start seeing "Connection refused" errors during peak hours.

**Specific failures:**
1. **Database connection exhaustion:** The application opens 100 connections. Each connection serves one request at a time. If requests take 200ms, each connection handles 5 RPS. 100 connections handle 500 RPS. At 1,000 RPS, requests queue up. Queued requests time out after 30 seconds. Users see 503 errors.
2. **Thread pool saturation:** The application server has 8 cores running 16 worker threads. Each thread blocks on a database query for 200ms. At 1,000 RPS, each thread handles 5 RPS. 16 threads handle 80 RPS. The remaining 920 RPS queue up. Queue wait time grows to seconds.
3. **Disk I/O:** The database writes 50 MB of logs, 10 MB of analytics, and 20 MB of URL data per minute. The single SSD handles 150 MB/s sequential writes, but the reads compete with writes. Query latency becomes unpredictable.

### What Breaks at 10,000,000 Users

At 10 million DAU, the naive system cannot function at all. The single server would need to handle 500,000-1,000,000 RPS. A single machine cannot provide 1,000 database connections. A single CPU cannot process 1 million requests per second. The system must be redesigned from the ground up.

The specific failures cascade:
1. The database primary saturates at 10,000 QPS. The application serves 100,000 QPS (90% reads from cache, 10% writes to database). Even at 10% write ratio, writes alone are 10,000 QPS, saturating the database.
2. Redis, added as a cache, handles 200,000 QPS for cache hits. But cache misses at 1% of 1M RPS = 10,000 cache misses per second, each hitting the database. The database dies.
3. The single-server network interface saturates. A 10 Gbps NIC handles roughly 1 million small requests per second. Beyond that, packets are dropped. The load balancer marks the server as dead. The system is down.

### Key Architectural Decisions in the Production Design

**Stateless application servers:** No session data stored on the application server. If any server dies, any other server picks up the request. This enables auto-scaling: when CPU crosses 70%, a new server boots in 2 minutes and joins the target group.

**Separate cache cluster:** Redis is not on the application server. It is a separate cluster that all servers share. If an application server dies, the cache is unaffected. If the cache needs more memory, you add Redis nodes without touching application servers.

**Separate queue:** Kafka decouples request handling from background processing. The application writes click events to Kafka and returns. If the analytics system is slow, the application does not slow down. Kafka buffers up to 72 hours of data on disk.

**Read replicas for the database:** Read queries go to replica nodes. Write queries go to the primary. This doubles effective read throughput and removes read contention from write operations. If a read replica fails, the load balancer routes to another replica.

## Integration Patterns

A production system integrates multiple data stores. Each integration pattern optimizes for a specific access pattern.

### Pattern 1: With Relational Database (PostgreSQL)

The relational database is the source of truth. All transactional data — user accounts, URL mappings, billing records — lives in PostgreSQL. The integration pattern is connection pooling with read/write splitting.

```
Application Server
  │
  ├── Write pool → PgBouncer → PostgreSQL Primary
  │   (INSERT, UPDATE, DELETE)
  │
  └── Read pool → PgBouncer → PostgreSQL Replica (x2)
      (SELECT — most queries go here)
```

Write pool: 10 connections. Read pool: 20 connections. Total per server: 30 connections. With 20 servers: 600 connections to PgBouncer, which maps to 200 actual database connections.

### Pattern 2: With Cache (Redis)

The cache stores hot data with a TTL. The integration pattern is cache-aside with write-through for critical data.

**Read path:** Check cache. If hit, return. If miss, query database, populate cache, return.
**Write path:** Write to database. Invalidate cache entry. Do NOT write directly to cache, because the cache is ephemeral — the database is the source of truth.

```python
def get_url(short_key):
    # Cache-aside read
    cached = redis.get(f"url:{short_key}")
    if cached:
        return cached

    url = db.query("SELECT url FROM urls WHERE key = ?", short_key)
    if url:
        redis.setex(f"url:{short_key}", 3600, url)
    return url

def create_url(short_key, original_url):
    # Write-through: write to DB, invalidate cache
    db.execute("INSERT INTO urls (key, url) VALUES (?, ?)", short_key, original_url)
    redis.delete(f"url:{short_key}")  # Invalidate, don't set
    return short_key
```

### Pattern 3: With Message Queue (Kafka)

The message queue decouples the request path from the processing path. The integration pattern is producer-consumer with partitioned topics.

```
Topic: "click-events"
Partitions: 16 (partition by short_key hash)
Retention: 72 hours
Replication factor: 3

Producer (application server):
  - Writes click event JSON to topic
  - Does not wait for consumer acknowledgement
  - Batch rate: 1,000 events or 500ms, whichever comes first

Consumer (analytics worker):
  - Reads from assigned partitions
  - Commits offset after processing
  - Batch INSERT to ClickHouse every 5 seconds
  - On failure: retry with backoff, then send to DLQ
```

### Pattern 4: With Object Storage (S3)

Object storage stores files (images, videos, documents) that are too large or too numerous for a relational database. The integration pattern is "store once, serve through CDN."

**Upload flow:**
1. Application server receives the uploaded file.
2. File is stored in an S3 bucket with a randomly generated key.
3. Database stores the S3 key and metadata (filename, size, content type).
4. CDN is configured with the S3 bucket as origin.

**Read flow:**
1. Application returns the CDN URL to the client (e.g., `https://cdn.example.com/files/{key}`).
2. Client requests the file from the CDN.
3. If CDN has the file cached, it returns it directly (edge hit: under 5ms).
4. If CDN does not have the file (cache miss or first request), it fetches from S3, caches it, and returns it (origin fetch: 50-200ms).

**Key design decisions:**
- **Random keys, not sequential:** Sequential keys (user_1.jpg, user_2.jpg) are guessable. Use UUIDs or random base64 strings.
- **Bucket per environment:** Separate buckets for development, staging, and production. This prevents accidental deletion of production data during testing.
- **Lifecycle policies:** Move files to cheaper storage tiers as they age. S3 Standard → S3 Infrequent Access (30 days) → S3 Glacier (90 days) → delete (365 days).
- **Pre-signed URLs for direct upload:** For large files, generate a pre-signed URL that allows the client to upload directly to S3 without going through the application server. This eliminates the application server as a bottleneck for uploads.

### Pattern 5: With External APIs

External APIs (payment gateways, SMS providers, email services) are the least reliable components in any system. They can be slow, unavailable, or return unexpected errors.

**The integration rules for external APIs:**
1. **Always set timeouts:** Connect timeout: 2 seconds. Read timeout: 10 seconds. Anything slower is a failure.
2. **Use a circuit breaker:** After 5 failures in 60 seconds, stop calling for 30 seconds. Return a cached or degraded response.
3. **Queue non-critical calls:** Email sending can wait 5 minutes. Queue it.
4. **Idempotency keys:** Every payment request carries a unique key. If the request times out and the user retries, the payment gateway sees the same key and does not charge twice.
5. **Limit retries:** Maximum 3 retries with exponential backoff. Beyond that, declare failure and enter degraded mode.

```python
class ExternalAPIClient:
    def __init__(self):
        self.failure_count = 0
        self.circuit_open_until = 0

    def call_with_circuit_breaker(self, request):
        now = time.time()
        if now < self.circuit_open_until:
            raise CircuitBreakerOpen("Service unavailable, skipping")

        try:
            response = requests.post(
                url,
                json=request,
                timeout=(2.0, 10.0),  # connect, read
            )
            self.failure_count = 0
            return response
        except (requests.Timeout, requests.ConnectionError):
            self.failure_count += 1
            if self.failure_count >= 5:
                self.circuit_open_until = now + 30
            raise
```

### Replication Strategy

Multi-region systems need a replication strategy that balances consistency, latency, and data loss risk.

**Synchronous replication:** The primary waits for the replica in the other region to acknowledge before returning. Guarantees zero data loss. Adds 50-100ms to every write (round trip to the remote region). Used for financial transactions where data loss is unacceptable.

**Asynchronous replication:** The primary acknowledges the write immediately. The replica receives the data 50ms-2 seconds later. Risk: if the primary fails before the replica receives the data, writes are lost. Used for most web applications where losing 1-2 seconds of data is acceptable.

**The standard choice for web applications:** Asynchronous replication for the database. This means cross-region read replicas are always slightly behind the primary. If the primary in us-east-1 fails, eu-west-1 might be missing the last 500ms of writes.

### Global Consistency Challenges

Multi-region writes create a consistency challenge. If a user in Europe posts a comment, the write goes to the us-east-1 primary (150ms round trip latency). The user's browser waits 150ms for the write to complete. If the user immediately refreshes to see their comment, the read may go to the eu-west-1 replica, which has not yet received the replication stream (50ms lag). The user does not see their own comment. This is a **read-after-write consistency** violation.

**Solutions for read-after-write consistency:**

1. **Read from primary after write:** After the user performs a write, route all their reads to the primary database for the next 10 seconds. This guarantees they always see their own writes. Trade-off: adds load to the primary.

2. **Cookie-based routing:** Set a cookie on the user's browser after a write. The application server checks for this cookie and routes reads to the primary. No cookie means route to replicas.

3. **Quorum reads:** Read from the primary and all replicas. Take the latest version. This works with versioned data. Trade-off: higher latency (wait for slowest replica).

### Failover Strategy

When the primary region goes down, the system must promote a read replica to primary in another region.

```
Step 1: Health check fails for us-east-1 primary
Step 2: Route53 detects the failure, stops routing to us-east-1
Step 3: Operator (or automation) promotes eu-west-1 replica to primary
Step 4: eu-west-1 database is now writable
Step 5: Route53 is updated: all traffic goes to eu-west-1
Step 6: us-east-1 recovers, becomes a read replica of eu-west-1
```

**Recovery time objective (RTO):** 5 minutes for the failover to complete.
**Recovery point objective (RPO):** 1 second of data loss in the worst case.

### Latency Implications

Cross-region latency matters. A user in Sydney connecting to us-east-1 experiences 150-250ms of network latency before any processing happens. This is why global services deploy in multiple regions and route users to the nearest one.

Route53 latency-based routing: each user's DNS query returns the IP of the load balancer in the region with the lowest latency from that user's location. This ensures Australian users connect to ap-southeast-1, European users connect to eu-west-1, and US users connect to us-east-1.

### DNS Geo-Routing Details

Route53 latency-based routing uses a global network of DNS resolvers to measure latency between each resolver and each region. When a user's device asks "what is the IP of api.example.com?", Route53 returns the IP of the region with the lowest latency from that resolver's location.

```
User in Sydney:
  DNS query → Route53
  Route53 measures:
    us-east-1: 210ms latency
    eu-west-1: 290ms latency
    ap-southeast-1: 15ms latency  ← lowest
  Returns: ALB IP in ap-southeast-1
  User connects to Sydney data center (ap-southeast-1)
  Latency: 15ms + processing time
```

The alternative is **Anycast routing**, where the same IP address is announced from multiple data centers. The internet's BGP routing protocol automatically directs each user to the nearest data center. Cloudflare and Google use Anycast. DNS-based routing is simpler to configure and does not require BGP expertise.

### Data Sovereignty and Compliance

Global deployments must comply with data sovereignty laws. The European Union's GDPR requires that EU citizen data be stored in EU data centers. Brazil's LGPD has similar requirements. China's Cybersecurity Law requires data from Chinese users to be stored in China.

**Implementation:**
1. Collect the user's country during registration (IP geolocation or explicit selection).
2. Store the user's data residency preference in the user profile.
3. Route the user's traffic to the region matching their residency.
4. Never replicate EU user data to non-EU regions (unless explicit consent is obtained).

This means the "global" database is actually a set of regional databases with controlled cross-region replication. Cross-region analytics queries must be computed in each region separately and aggregated centrally.

## Capacity Planning

Capacity planning is the math that turns user counts into infrastructure requirements. It answers the question: how many servers, how much storage, how much bandwidth.

### Planning for 10 Million Daily Active Users (DAU)

Assume a URL shortening service with the following characteristics:
- 10 million DAU
- Each user creates 2 short URLs per day on average
- Each user visits 10 short URLs per day on average
- Peak traffic is 3x average (evening hours)
- Each API call is 2 KB request + 2 KB response

### QPS Calculation

```
Daily writes:     10,000,000 users × 2 URLs/visit = 20,000,000 writes/day
Daily reads:      10,000,000 users × 10 visits = 100,000,000 reads/day
Total operations: 120,000,000 per day

Writes per second (average):   20,000,000 / 86,400 = 231 writes/sec
Reads per second (average):   100,000,000 / 86,400 = 1,157 reads/sec
Total QPS (average):          1,388 QPS

Peak QPS (3x average):        4,164 QPS

Peak write QPS:               693 writes/sec
Peak read QPS:                3,471 reads/sec
```

The peak write rate of 693 writes per second is well within a single PostgreSQL primary's capacity (5,000-10,000 QPS). The peak read rate of 3,471 reads per second with a 90% cache hit rate means 347 database reads per second, easily handled by two read replicas.

### Storage Calculation

```
URL record size:     230 bytes
Daily storage:      20,000,000 × 230 bytes = 4.6 GB/day
Yearly storage:     4.6 GB × 365 = 1.7 TB/year
5-year storage:     1.7 TB × 5 = 8.5 TB

Click event storage:
  Event size:        200 bytes
  Daily events:     100,000,000 × 200 bytes = 20 GB/day
  Yearly events:    20 GB × 365 = 7.3 TB/year
  Retention policy: 90 days for raw events, 5 years for aggregated
  Raw storage:      20 GB × 90 = 1.8 TB
  Aggregated:       ~500 GB (much smaller)
```

### Bandwidth Calculation

```
Inbound traffic (writes):    693 writes/sec × 2 KB = 1.4 MB/s
Outbound traffic (reads):    3,471 reads/sec × 2 KB = 6.9 MB/s
Total inbound per day:       1.4 MB/s × 86,400 = 121 GB/day
Total outbound per day:      6.9 MB/s × 86,400 = 596 GB/day
Monthly bandwidth:           121 + 596 = 717 GB/day × 30 = 21.5 TB/month
```

At $0.09/GB for egress (AWS standard), the monthly bandwidth cost is approximately $1,935. CDN caching for static assets (CSS, JS, images) can reduce this by 40-60%.

### Server Count Calculation

```
Application servers:
  Throughput per server:     10,000 RPS (reasonable for Go/Java)
  Required capacity:         4,164 RPS peak
  Servers needed:            4,164 / 10,000 = 1 (but need redundancy)
  Practical minimum:         3 servers (1 failure tolerance at 40% load)
  Auto-scaling max:          10 servers (for unexpected spikes)

Database servers:
  1 primary (writes)
  2 read replicas (reads, cache miss fallback)
  Total: 3 database servers

Cache nodes:
  Memory needed:             Hot dataset for URLs accessed in last hour
  Assume 10M URLs × 230 bytes = 2.3 GB of hot data
  Plus 30% overhead:        3 GB
  Redis node memory:        8 GB (one node is enough)
  But need redundancy:      2 nodes (primary + replica)

Worker nodes:
  Click processing:         3,471 events/sec peak
  Workers needed:           4 workers (2x redundancy)
```

## Deployment and Operations

An architecture is only as good as its operational tooling. The best design fails if you cannot deploy it, monitor it, or recover it.

### Zero-Downtime Deployment

Rolling deployment replaces servers one at a time without interrupting traffic. The process for a zero-downtime deploy:

1. Drain connections on Server 1 (load balancer sends new requests to other servers).
2. Wait 30 seconds for in-flight requests to complete.
3. Deploy new version to Server 1.
4. Health check passes (returns 200 on /health endpoint).
5. Resume traffic to Server 1.
6. Repeat for each server in the pool.

Total deploy time for 10 servers: approximately 10 minutes. User impact: zero, because each server's connections drain before it goes offline.

**Blue-green deployment** is an alternative: spin up a complete set of new servers (green), run health checks, then switch the load balancer from the old set (blue) to the new set in a single DNS update. Rollback: switch back to blue.

### Monitoring Metrics

You cannot improve what you do not measure. Every production system must expose and monitor these metrics:

**Request metrics (per endpoint):**
- Request rate (RPS)
- Error rate (5xx count / total count)
- Latency distribution (p50, p95, p99, p999)
- Request size distribution

**Application metrics:**
- CPU utilization (target: under 70%)
- Memory utilization (target: under 80%)
- Thread pool utilization (active threads / max threads)
- GC pause time (Java, Go)
- Open file descriptors

**Database metrics:**
- Query rate (QPS)
- Query latency (p50, p95, p99)
- Connection count (active, idle, waiting)
- Replication lag (seconds)
- Cache hit ratio (shared buffers for PostgreSQL)
- Disk I/O utilization

**Cache metrics:**
- Hit rate (target: over 95% for hot data)
- Eviction rate (keys evicted per second)
- Memory usage (percent of maxmemory)
- Command latency (p99 under 5ms)

**Queue metrics:**
- Message rate (produced, consumed)
- Queue depth (messages pending)
- Consumer lag (messages behind head)
- Dead-letter queue count

### Alerting and On-Call

Monitoring is useless without alerting. Every metric must have an alert threshold, and every alert must have a runbook.

**Alert severity levels:**
- **P1 (Critical):** System is down or core functionality is broken. Example: "Error rate exceeds 5% for 5 minutes." Respond within 5 minutes. Runbook must be tested monthly.
- **P2 (High):** Degraded performance for a subset of users. Example: "p99 latency exceeds 500ms for 10 minutes." Respond within 15 minutes.
- **P3 (Medium):** Non-critical component is degraded. Example: "Cache hit rate below 80%." Respond within 1 hour.
- **P4 (Low):** Informational. Example: "Database storage usage exceeds 70%." Respond within 1 week.

**Alert quality checklist:**
- Every alert must be actionable. If there is nothing to do, it is noise, not an alert.
- Every alert must have a runbook link. A runbook is a step-by-step guide for responding.
- Alerts that fire outside business hours must justify waking someone up. If it can wait until morning, make it a P3.

**Runbook structure:**
```
Runbook: DB-PRIMARY-CPU-HIGH
Severity: P2
Symptoms: DB CPU > 80% for 5 minutes
Step 1: Check slow query log (link to dashboard)
Step 2: Identify the slowest query (dashboard link)
Step 3: Kill the query: SELECT pg_terminate_backend(pid)
Step 4: Check if query is from a recent deploy (link to deploy tracker)
Step 5: If from deploy, roll back. If not, add missing index.
Step 6: Verify recovery: CPU should drop below 50% within 2 minutes
Escalation: If CPU stays above 80% after 10 minutes, page senior DBA
```

### Debugging a Production Issue

When a production issue occurs, the debugging process follows a systematic path:

1. **Check the dashboard:** Which metric changed? Latency spike, error rate increase, queue depth growth?
2. **Identify the time range:** When did it start? Correlate with deployments, traffic changes, or external events.
3. **Check the recent deploy:** Did the last deploy change the database query pattern? Add a slow dependency? Change cache TTL?
4. **Check the "last changed" log:** Config changes, feature flags, infrastructure changes.
5. **Check for external triggers:** A competitor ran a promotion. A news article drove traffic. A DNS provider had an outage.
6. **Isolate the component:** Is the application server slow? Database? Cache? External API?
7. **Deep dive on the slow component:** Check the specific query, the specific endpoint, the specific cache key pattern.
8. **Fix and observe:** Rollback a deploy, scale up a pool, clear a cache, throttle a user, kill a slow query.

### Upgrades and Migrations

The hardest operation in system design is changing the data layer while keeping the system running.

**Schema migration strategy:**
1. Add the new column as nullable. Backfill data in batches of 10,000 rows.
2. Deploy application code that writes to both old and new columns.
3. Verify data consistency between old and new.
4. Deploy application code that reads from the new column.
5. Remove the old column migration.

This process takes days or weeks, not minutes. The system stays fully operational throughout.
