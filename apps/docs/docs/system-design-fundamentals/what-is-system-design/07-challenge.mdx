These challenges test your understanding of the entire "What is System Design" topic. They are organized into three tiers that build from foundational knowledge to expert-level ambiguous scenarios.

Each challenge includes a difficulty rating, time estimate, evaluation criteria, and a model answer. Attempt the challenge before reading the answer. System design is a skill that improves only through deliberate practice.

## Tier 1 — Foundation

These challenges test whether you have absorbed the core vocabulary, concepts, and mental models from this topic. Each should take approximately 5 minutes.

### Challenge 1.1: Explain the Terms

**Prompt:** A junior developer on your team asks you to explain the difference between horizontal scaling and vertical scaling. They also ask: "If we have 2,000 requests per second and each server handles 500 requests per second, how many servers do we need with horizontal scaling, and why would we not simply buy a bigger single server?"

Write your explanation as if you are teaching this to a colleague.

**Evaluation criteria:**
- You clearly define both scaling approaches with examples.
- You calculate the server count correctly (4 servers minimum).
- You explain the limitations of vertical scaling (hardware ceiling, cost non-linearity, single point of failure).
- You explain the advantages of horizontal scaling (cost linearity, redundancy, incremental growth).
- You use at least one concrete analogy or diagram description.

**Model answer:**

"Horizontal scaling means adding more machines to your pool. Vertical scaling means making your single machine more powerful — more CPU cores, more RAM, faster disk.

If each server handles 500 RPS and we need 2,000 RPS, we need 4 servers for the capacity. But we would probably run 5-6 servers so that losing one server does not drop us below capacity.

We choose horizontal scaling over vertical scaling for three specific reasons:

First, vertical scaling hits a hardware ceiling. You can only add so much RAM to a single machine. At some point, you cannot buy a bigger server — the largest cloud instances have 128 cores and 512 GB of RAM, and that is the absolute limit. With horizontal scaling, you can add machines indefinitely.

Second, vertical scaling has no redundancy. If your single massive server fails, everything fails. With 5 smaller servers, losing one drops capacity from 2,500 RPS to 2,000 RPS — you are still above your 2,000 RPS requirement.

Third, vertical scaling costs more per unit of capacity. A server with 64 cores might cost 8x a server with 8 cores, but often does not deliver 8x the throughput because of operating system overhead, memory bus contention, and application-level lock contention. Horizontal scaling with smaller machines is typically more cost-effective."

### Challenge 1.2: Identify the Bottleneck

**Prompt:** Your team's web application serves 5,000 requests per second. The average response time is 500ms. The breakdown of average response time by component is:

- Load balancer: 2ms
- Application server (compute): 50ms
- Database query: 30ms (includes network round trip)
- External API call to weather service: 400ms
- Response serialization: 18ms

The external API call is made synchronously on every request. The weather data is displayed on every page. Identify the bottleneck, explain why it is the bottleneck, and propose two different fixes with their trade-offs.

**Evaluation criteria:**
- You identify the external API call as the bottleneck (400ms of 500ms total).
- You explain that removing or optimizing the API call would reduce response time by 80%.
- You propose at least two distinct solutions.
- You discuss trade-offs for each solution.

**Model answer:**

The bottleneck is the external weather API call, which consumes 400ms out of the 500ms total response time. Every other component contributes only 100ms combined. Fixing anything other than the weather API call would reduce response time by a maximum of 20%.

**Fix 1: Cache the weather data.** Store weather data in Redis with a 30-minute TTL. The response time for cached data drops from 400ms to 2ms. This eliminates the bottleneck entirely for cached requests. Trade-off: weather data is up to 30 minutes stale. In a fast-changing weather event, users may see outdated information. For most use cases, this is acceptable.

**Fix 2: Make the API call asynchronous.** Fetch weather data after rendering the page and send it to the browser via a background WebSocket push or a JavaScript AJAX call after the page loads. The page loads in 100ms initially and the weather data arrives 400ms later. Trade-off: increased frontend complexity. Users see the page without weather data for 400ms, then it appears. This may cause layout shifts if not handled properly.

**Fix 3: Fail gracefully with a circuit breaker.** If the weather API is slow or unavailable, serve a cached version or show "Weather unavailable" instead of blocking the page load. This protects users from a 400ms delay when the API has a transient issue. Trade-off: Some users see outdated or no weather data.

The recommended approach is a combination of Fix 1 (caching with 30-minute TTL) and Fix 3 (circuit breaker as fallback). This reduces the median response time to under 100ms.

### Challenge 1.3: Define the Trade-off

**Prompt:** Your team is choosing between a relational database (PostgreSQL) and a NoSQL document database (MongoDB) for a new user profile service. The service stores: user name, email, hashed password, profile picture URL, bio text, follower count, list of follower IDs, and last login timestamp.

Write a short analysis that recommends one database and explains the trade-off.

**Evaluation criteria:**
- You analyze the data structure and access patterns.
- You consider query patterns: lookup by user ID, lookup by email (login), update follower count (write-heavy).
- You make a specific recommendation.
- You explain what you are giving up with your choice.

**Model answer:**

The user profile data includes both structured fields (name, email, password hash) and an array field (follower IDs). The follower IDs could be stored as a JSON array in PostgreSQL or as an array field in MongoDB. Both databases handle this pattern.

The critical factor is the query pattern: lookups by user ID (fast with either), lookups by email for login (needs an index on email), and follower count updates (atomic increment, write-heavy).

**Recommendation: PostgreSQL.** The relational model maps naturally to user profiles. PostgreSQL supports JSONB for array fields like follower IDs if needed. The key advantage is transactional integrity: if a user updates their email and triggers downstream operations (session invalidation, email verification), the transaction ensures atomicity. PostgreSQL's `INSERT ... ON CONFLICT` (upsert) handles the high-write follower counts efficiently.

**Trade-off:** We lose the schemaless flexibility of MongoDB. Adding a new field to the user profile requires a migration: `ALTER TABLE users ADD COLUMN ...`. With MongoDB, you just start writing documents with the new field. The migration overhead is low for this data model (infrequent schema changes) but should be considered if the profile structure evolves rapidly.

**Alternative consideration:** If follower IDs grow to 100,000+ per user and reads of the follower list need to be paginated, PostgreSQL might need a separate `followers` table with an index on `user_id`. MongoDB handles the 100,000-element array with server-side pagination natively. At that scale, MongoDB would be the better choice.

## Tier 2 — Design

These challenges require designing a small system end-to-end. Each should take approximately 15 minutes.

### Challenge 2.1: Design a URL Preview Service

**Prompt:** Design a system that generates preview cards for URLs shared in a messaging app. When a user shares a link, the system fetches the page, extracts the title, description, and image, and stores the preview for 7 days.

Scale: 1 million URLs previewed per day. Each preview requires fetching the page (average 500 KB), parsing the HTML, and downloading the preview image (average 200 KB). The preview must be available in under 2 seconds for 95% of requests.

**Evaluation criteria:**
- Capacity estimation: QPS, storage, bandwidth for fetched pages and images.
- Data model for storing previews (key by URL hash, cache metadata and image path).
- Deduplication: same URL should not be fetched twice within 7 days.
- Async processing: who fetches the page and how?
- Caching: where does the preview data live for fast retrieval?
- Error handling: what happens when a page cannot be fetched or has no metadata?

**Model answer:**

**Capacity estimation:**
- 1 million URLs / 86,400 = 11.6 URLs/second average
- Peak: approximately 35 URLs/second (3x average)
- Each URL fetch: 500 KB page + 200 KB image = 700 KB
- Bandwidth: 35 URLs/sec x 700 KB = 24.5 MB/sec outbound to the internet
- Storage per preview: title (200 bytes) + description (500 bytes) + image path (100 bytes) + URL (200 bytes) + timestamp (8 bytes) = approximately 1 KB
- 7 days of metadata: 7 million x 1 KB = 7 GB
- Image storage: 1 million x 200 KB = 200 GB/day, but only for 7 days = 1.4 TB
- Image storage with deduplication: approximately 30% of URLs share images = 1 TB

**Architecture:**

The system uses an async pattern. When a user shares a URL, the messaging app sends the URL to the preview service. The service returns immediately with a "processing" status. A background worker fetches the page and generates the preview. When complete, the preview is cached and the messaging app receives a notification.

Components:
- **API endpoint:** Accepts URL, returns preview ID immediately. Checks if preview already exists (by URL hash) — if so, returns cached preview immediately.
- **Worker pool:** Pulls URLs from a queue. Each worker fetches the page using a headless browser (for JavaScript-rendered content) or a simpler HTTP parser (for static pages). Invalid or slow URLs are skipped after a 5-second timeout.
- **Image processor:** Downloads the preview image, resizes to 1200x630 pixels (Open Graph standard), compresses to JPEG at 80% quality, stores in S3.
- **Cache:** Redis stores preview metadata (title, description, image URL) keyed by URL hash. TTL: 7 days (matching storage retention).
- **Database:** PostgreSQL stores the URL-to-preview mapping for deduplication. Key: URL hash (SHA256 of normalized URL). Index on hash for fast lookup.

**Deduplication:** Before any fetch, the system checks Redis and PostgreSQL for an existing preview by URL hash. If found, returns cached data. If not, enqueues the fetch. A Bloom filter in front of Redis further reduces redundant checks.

**Error handling:** If the page cannot be fetched (timeout, DNS failure, invalid URL), the preview shows a generic card with the domain name and URL. The failed URL is retried once after 30 minutes. If it fails again, it is moved to a dead-letter queue for manual inspection.

### Challenge 2.2: Design a Notification Service

**Prompt:** Design a notification service that sends push notifications to mobile devices. The system must handle 10 million notifications per day across three channels: push (Firebase/APNs), email, and SMS.

Notifications must be delivered within 5 seconds of being created. Email delivery can take up to 5 minutes. SMS must be delivered within 30 seconds. The system must track delivery status (sent, failed, delivered, read).

**Evaluation criteria:**
- Capacity estimation: notifications per second per channel.
- Queue-based architecture for decoupling notification creation from delivery.
- Channel-specific delivery logic and retry policies.
- Delivery status tracking and idempotency.
- Handling provider failures (Firebase down, SMTP server slow, SMS provider errors).

**Model answer:**

**Capacity estimation:**
- 10 million notifications / 86,400 = 116 notifications/second average
- Peak: approximately 350 notifications/second
- Channel distribution: push 70% (70M/month), email 20% (20M/month), SMS 10% (10M/month)

**Architecture:**

The system uses a producer-consumer pattern with separate queues per channel. Each channel has independent scaling, retry policies, and provider failover.

Components:
- **Notification API:** Accepts `POST /notify` with recipient, channel, template, and parameters. Returns a notification ID immediately. The notification is enqueued.
- **Notification database (PostgreSQL):** Stores each notification with fields: id, user_id, channel, status, created_at, sent_at, delivered_at, error_message.
- **Per-channel queues (Kafka):** Three topics: `notification-push`, `notification-email`, `notification-sms`. Each topic has its own consumer group with channel-specific workers.
- **Push workers:** Call Firebase Cloud Messaging (FCM) for Android and Apple Push Notification Service (APNs) for iOS. Handle device tokens, batched sends (up to 500 per batch for FCM).
- **Email workers:** Use Amazon SES or SendGrid. Handle bounces, complaints, unsubscribe requests. Send within 5 minutes using batched delivery (queue sorting by priority).
- **SMS workers:** Use Twilio or AWS SNS. Handle phone number formatting, carrier errors. Send within 30 seconds with single-message delivery.
- **Delivery tracking service:** Listens to delivery callbacks from FCM, APNs, SES, and Twilio. Updates notification status in PostgreSQL. Webhook delivery status to the calling service.

**Idempotency:** Each notification request includes a client-provided idempotency key. If the same key is received within 24 hours, the system returns the existing notification ID without creating a duplicate.

**Provider failover:**
- Push: Primary = FCM. If FCM returns 503 (server error) more than 5 times in 60 seconds, circuit breaker opens, fallback to APNs (for iOS) or FCM secondary project.
- Email: Primary = SES. If SES queue depth exceeds threshold, spill over to SendGrid.
- SMS: Primary = Twilio. If Twilio fails, fallback to AWS SNS. SMS is the highest priority for provider redundancy because SMS has the tightest delivery SLA.

### Challenge 2.3: Design a File Upload Service

**Prompt:** Design a service that accepts file uploads from users, stores them securely, and generates a shareable link. Files can be up to 2 GB each. The system handles 1 million uploads per day, with an average file size of 5 MB.

Users can upload from any device. The service must be able to resume interrupted uploads. Shared links expire after 30 days.

**Evaluation criteria:**
- Chunked upload for resume support.
- Temporary storage for in-progress uploads versus permanent storage for completed uploads.
- CDN integration for fast download of shared files.
- Virus scanning before the file is marked as available.
- Deduplication (same file uploaded twice should not be stored twice).
- Expiry and cleanup of expired files.

**Model answer:**

**Capacity estimation:**
- 1 million uploads/day, average 5 MB = 5 TB/day inbound
- Peak upload throughput: approximately 5 TB / 8 hours = 625 MB/sec (assuming peak during business hours)
- Storage after deduplication: approximately 60% of raw = 3 TB/day
- 30-day retention: 3 TB x 30 = 90 TB active storage (S3 Standard)
- After 30 days: delete or move to Glacier for long-term archival

**Architecture:**

Components:
- **Upload API:** Accepts multipart upload with chunked transfer. Client sends chunks of 5 MB each with a chunk index. The server acknowledges each chunk. If the connection drops, the client resumes from the last acknowledged chunk. Each upload has a unique upload ID.
- **Chunk storage (S3):** Chunks are stored in a temporary S3 bucket under `uploads/{upload_id}/chunk_{index}`. When all chunks are received, the server initiates an assembly process that concatenates chunks into the final file.
- **Assembly worker:** Reads chunk metadata from PostgreSQL, downloads chunks from S3, concatenates them, computes SHA256 hash, uploads to permanent storage. On completion, deletes chunk storage.
- **Deduplication service:** Before storing permanently, checks if the SHA256 hash already exists. If it does, creates a new reference to the existing file instead of storing a duplicate copy. The reference count is incremented. When all references expire, the file is deleted.
- **Virus scanner:** Lambda function triggered by S3 PUT events. Runs ClamAV on the uploaded file. If clean, the file is marked as available. If infected, the file is quarantined and the uploader is notified.
- **Share link service:** Generates a cryptographically random token (CSPRNG, 128 bits), stored in Redis with a 30-day TTL. The download URL is of the format `https://files.example.com/dl/{token}`.
- **Download service:** Reads the token from Redis, retrieves the file path, generates a signed S3 URL (valid for 1 hour). User downloads directly from S3 through CloudFront CDN.

**Resumable upload flow:**
1. Client sends `POST /upload/init` with file size and name. Server returns upload_id.
2. Client sends chunks: `PUT /upload/{upload_id}/chunk/{index}` with chunk data. Server acknowledges.
3. If interrupted, client queries `GET /upload/{upload_id}/status` to get list of received chunks.
4. Client resumes from the first missing chunk.
5. When all chunks are received, client sends `POST /upload/{upload_id}/complete`. Server assembles.

**Security:** All uploads and downloads over HTTPS. Files are scanned for viruses. File type validation (magic bytes, not file extension). Upload rate limiting: 10 concurrent uploads per user. Max file size: 2 GB per file.

## Tier 3 — Expert

These challenges present ambiguous, open-ended scenarios with no single correct answer. Each should take approximately 30 minutes. The evaluation focuses on your process, not your final design.

### Challenge 3.1: The Cache Stampede

**Prompt:** Your team manages a video recommendation service for a streaming platform with 200 million active users. The system serves personalized recommendations from a Redis cache. The cache is populated by a batch job that runs every 30 minutes and recomputes recommendations for all users.

The batch job reads user engagement data from a data warehouse, runs a machine learning model, and writes the result to Redis. The job takes 45 minutes to complete — because there are 200 million users and the model takes 10ms per user on a 16-core server.

The problem: every 30 minutes, the cache TTL expires for all users simultaneously. At that moment, every user request hits the database (actually the data warehouse, which takes 500ms per query). Latency spikes from 5ms to 500ms for 45 minutes until the batch job repopulates the cache.

Your team is considering three approaches:
1. Stagger the TTLs so each user's cache expires at a different time.
2. Switch from batch processing to incremental updates as users engage with content.
3. Serve stale data from the cache while the new data is being computed.

Evaluate all three approaches. Consider the trade-offs for a 200-million-user system. If you recommend a specific approach, explain why the others are wrong for this scale. If you recommend a hybrid approach, specify the exact split.

**Valid approaches include:**

**Approach 1 — Staggered TTLs:** Assign each user a cache TTL between 25 and 35 minutes using `user_id mod 10_minutes`. The cache expiration load distributes evenly across 10 minutes. The batch job runs continuously and refreshes each user's cache as it expires. The data warehouse sees 200 million / 10 minutes = 333,000 queries per second — which is still too high. This approach reduces the spike but does not eliminate the load on the data warehouse.

**Approach 2 — Incremental updates:** Instead of recomputing all recommendations every 30 minutes, update a user's recommendations only when they watch a video or the video catalog changes. A user who watches 5 videos per day gets their recommendations updated 5 times. The "tail" of inactive users (50%+ of 200 million) keeps their stale recommendations. This reduces compute by 90%. The challenge: the ML model needs global engagement data (trending videos, viral content) to produce good recommendations. Incremental updates cannot capture global trends as effectively as a full recompute. Solution: a hybrid where incremental updates handle the 90% of user-specific changes and a full batch recompute runs once per day for global trends.

**Approach 3 — Serve stale data:** Set the cache TTL to 24 hours. Serve recommendations that are up to 24 hours old. The batch job still runs every 30 minutes but writes to a different Redis key namespace. When the job completes, the application switches to the new namespace. Users always see data that is at most 30 minutes stale, not instantaneously fresh. This completely eliminates the latency spike. The trade-off is recommendation quality: a video that became viral 5 minutes ago will not appear in recommendations for another 25 minutes.

**Recommended hybrid:** Combined approach 3 (serve stale data from old namespace) with approach 1 (staggered writes to the new namespace). The batch job writes results to Redis namespace "v2" incrementally as users are processed. The application serves from namespace "v1" (maximum 30 minutes stale). When the batch job reaches 100% completion, update an atomic config key telling the application to switch to "v2." Users never see a cache miss. The data warehouse is never queried by user requests. Latency stays at 5ms.

### Challenge 3.2: The Unkillable Slow Query

**Prompt:** You are the new infrastructure lead at a mid-size e-commerce company (5 million monthly active users). The engineering team tells you about "the query" — a monthly business intelligence report that runs on the production PostgreSQL primary and takes 6 hours to complete. During those 6 hours, the primary database CPU is at 95%, query latency for user-facing requests goes from 5ms to 1,200ms, and the on-call engineer receives automated alerts but has run out of ideas.

The previous team tried the following fixes, none of which worked:
1. Adding indexes to the tables used by the report query.
2. Rewriting the query using CTEs and subqueries.
3. Killing the query and rerunning it on a different day.

The report uses 15 JOINs across 8 tables. It aggregates 18 months of sales data by product category, region, customer segment, and channel. The result is a 50,000-row report used by the finance team for quarterly planning.

The report cannot be cancelled or deferred — the CFO needs it by the 5th of every month.

Design a solution that eliminates the impact of this query on user-facing traffic without breaking the finance team's deadline. You have a budget of $5,000/month for new infrastructure. Your solution must work within 2 months.

**Valid approaches include:**

**Approach A — Read replica with logical replication:** Provision a read replica of the primary database. Use PostgreSQL logical replication to stream changes to the replica. Run the report query on the replica. The primary database is unaffected. Cost: approximately $400/month for a db.r6g.large instance. The challenge: the replica needs to keep up with writes from the primary. At 5 million MAU, the write rate is likely under 500 writes/sec. A single replica handles this easily. Implementation time: 1 week (provision, configure replication, test, redirect report).

**Approach B — Dedicated analytics warehouse:** Export the relevant tables to a columnar data warehouse (ClickHouse, Redshift). The report query runs on the warehouse instead of PostgreSQL. Cost: approximately $500/month for a small ClickHouse instance. Implementation time: 2-4 weeks (export pipeline, schema mapping, query rewriting). Benefit: columnar storage compresses the data by 3-10x, and the report query runs in minutes instead of hours.

**Approach C — Pre-aggregation with materialized views:** Create materialized views that pre-aggregate the data at multiple granularities (daily, weekly, monthly, by category). The report query uses the pre-aggregated views instead of the raw 18-month dataset. Cost: minimal (storage for materialized views). Implementation time: 1-2 weeks.

**Recommended approach:** Start with Approach A (read replica) as the immediate fix — implementable in 1 week, zero impact on user traffic, well under budget. Then, in parallel, implement Approach C (pre-aggregation) to reduce the report query time from 6 hours to under 10 minutes. The read replica ensures the finance team gets their report on time. The pre-aggregation reduces the infrastructure cost and makes the report runnable on demand rather than monthly.

### Challenge 3.3: The Global Health Platform

**Prompt:** Design a platform that collects health metrics from wearable devices (heart rate, steps, SpO2, sleep stages) for 100 million users worldwide. Each device sends metrics every 5 minutes. Data must be stored for 5 years. Users can view their health trends on a mobile app. Healthcare providers can access patient data with consent.

This scenario tests your ability to reason about data volume, time-series storage, privacy regulations (HIPAA, GDPR), and query patterns under extreme scale.

**Valid approaches include:**

**Data volume estimation:**
- 100 million devices x 288 readings/day (every 5 minutes) = 28.8 billion readings/day
- Each reading: heart rate (4 bytes), steps (4 bytes), SpO2 (2 bytes), sleep stage (1 byte), timestamp (8 bytes), device_id (8 bytes) = 27 bytes
- Plus metadata: user_id (8 bytes), data_type (1 byte) = 9 bytes
- Total: approximately 36 bytes per reading
- Daily storage: 28.8 billion x 36 bytes = 1.04 TB/day
- Yearly storage: 1.04 TB x 365 = 380 TB/year
- 5-year storage: 380 TB x 5 = 1.9 PB

**Key challenges:**
1. Ingest throughput: 28.8 billion writes/day = 333,000 writes/second peak.
2. Storage: 1.9 PB over 5 years.
3. Time-series query pattern: "get heart rate for user X between date A and date B."
4. Privacy: data access controls, encrypted storage, audit logging.
5. Geo-distribution: global user base, data sovereignty laws.

**Recommended architecture:**
- **Ingestion layer:** Devices send data to regional endpoints. Each endpoint runs a Kafka-compatible message broker. Data is partitioned by `user_id mod 1024` to distribute writes evenly.
- **Time-series database:** Apache Cassandra or ScyllaDB. Partition key: `(user_id, date)` where date is the UTC date. Clustering key: timestamp. This ensures that a user's data for a single day is stored on one node. Querying a 7-day trend touches 7 partitions (7 nodes) in parallel.
- **Compression:** Downsample raw data after 90 days. Store 5-minute raw data for 90 days, then aggregate to hourly averages. After 1 year, aggregate to daily averages. This reduces 5-year storage from 1.9 PB to approximately 200 TB.
- **Privacy:** Data encrypted at rest with per-user keys. Access control at the API layer with attribute-based access control (ABAC). Healthcare provider access requires user consent token, valid for a configurable time window.
- **Geo-distribution:** Deploy in 3 regions (US, EU, APAC). Each region stores data for users in that region (data sovereignty). Cross-region access uses replication with consent validation.

### Challenge 3.4: The Social Media Feed Algorithm

**Prompt:** You are designing the content ranking system for a new social media platform targeting 50 million monthly active users. The feed must show each user the 50 most relevant posts from the last 48 hours, sorted by a relevance score that considers: recency, engagement (likes, comments, shares from friends), content type preferences, and ad revenue goals.

The relevance score is computed by a machine learning model that takes 25 milliseconds per post. Each user has approximately 1,500 candidate posts in the last 48 hours (from 500 followed accounts posting 3 times each). The feed must load in under 300ms on mobile devices.

**Valid approaches include:**

**Approach A — Candidate generation + ranking pipeline:** Split the work into two stages. Stage 1 (candidate generation): fetch the 1,500 candidate posts from a Redis cache keyed by `user_id:feed:candidates`. This is a simple sorted set of post IDs from followed accounts. Stage 2 (ranking): apply the ML model to score and sort the candidates. Since each post takes 25ms, scoring 1,500 posts would take 37.5 seconds — far too slow for a 300ms response time.

**Approach B — Pre-computed feed with incremental updates:** Pre-compute each user's top 50 posts in a batch job that runs every 5 minutes. The feed load is a single Redis lookup: `ZREVRANGE user:feed:{user_id}:ranked 0 49`. Response time: under 10ms. Trade-off: the feed is up to 5 minutes stale. Breaking news appears in feeds only after the next batch run.

**Approach C — Two-tier ranking:** Use a lightweight proxy model (0.5ms per post) to rank all 1,500 candidates into a top 200. Then apply the full ML model (25ms per post) to the top 200 only. Total scoring time: 1,500 x 0.5ms + 200 x 25ms = 750ms + 5,000ms = 5.75 seconds. Still too slow for 300ms.

**Approach D — Staged candidate filtering with pre-computed features:** Pre-compute the ML model features for each post (engagement velocity, content type scores, ad value) in a streaming pipeline that updates every minute. At feed load time, use these pre-computed feature vectors to rank candidates in memory. Since the features are pre-computed, the scoring is a simple weighted sum: approximately 0.01ms per post. Scoring 1,500 posts: 15ms. Combined with the 50ms network round trip and cache lookups, the feed loads in under 200ms.

**Recommended approach:** Approach D with a fallback to Approach B for cold-start users (new users with no engagement history). The streaming feature computation pipeline reads engagement events from Kafka, updates per-post feature vectors in Redis, and triggers feed re-ranking for users who follow the affected accounts. This keeps the feed fresh within seconds rather than minutes.
