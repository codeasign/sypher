The previous section showed a complete, scalable architecture. This section examines the uncomfortable truth: every design choice optimizes for one property at the expense of another. Understanding trade-offs is what distinguishes an architect from someone who just reads documentation.

## The Core Tension

System design is the art of making trade-offs deliberately. There is no perfect architecture. There is no free lunch. Every decision has a cost.

**Speed costs consistency.** A cache makes reads fast but serves stale data. An in-memory data store is fast but loses data on restart. An eventually consistent database writes quickly but may return different results to different readers at the same time.

**Availability costs complexity.** Adding redundancy eliminates single points of failure but introduces distributed consensus problems. A primary-replica setup with automatic failover requires a leader election algorithm, a health-check system, and a failover orchestration pipeline.

**Scalability costs simplicity.** A monolithic codebase is easy to understand and debug. A microservices architecture requires service discovery, distributed tracing, circuit breakers, API gateways, and inter-service authentication.

**Performance costs cost.** A cache cluster costs money. Faster hardware costs more. Multi-region deployment doubles your infrastructure bill. At some point, the cost of an additional 10ms of latency reduction exceeds the revenue it generates.

The core tension is this: you must make these trade-offs before you have enough data to know which side is correct. By the time you have the data to validate a trade-off, changing it is expensive.

## Decision Matrix

The decision matrix below captures the most common architectural choices and the conditions that favor each side.

| Decision Point | Option A | Option B | Choose A When | Choose B When |
|---|---|---|---|---|
| Database type | Relational (PostgreSQL) | NoSQL (DynamoDB, MongoDB) | Your data has relationships, joins, or transactions. You need ACID guarantees. | Your data is document-shaped. You need single-digit millisecond latency at any scale. You want to avoid schema migrations. |
| Monolith vs microservices | Monolithic | Microservices | Team size under 15. System under 50,000 LOC. Shipping fast matters more than scaling teams. | Multiple independent teams. Different scaling needs per component. Independent deploy cycles required. |
| Synchronous vs async | Synchronous request/response | Async message queue | The caller needs the result to proceed. Low latency is required. | The caller does not need an immediate response. Traffic bursts need smoothing. You need fault isolation. |
| Cache | Local cache (in-memory) | Distributed cache (Redis) | Single application instance. Data size fits in heap. Staleness is acceptable. | Multiple application instances. Cache must survive restarts. Cache must be shared across instances. |
| Read replicas | No replicas | Read replicas | Write ratio exceeds 50%. Total QPS under 1,000. Stale reads are unacceptable. | Read ratio exceeds 80%. Total QPS exceeds 1,000. 50ms of stale read is acceptable. |
| Data partitioning | No partitioning | Sharding by key | Single instance handles your data volume. No growth beyond 1 TB. | Data exceeds single-instance capacity. Growth is unbounded. Write throughput exceeds single-instance limit. |
| Consistency model | Strong consistency | Eventual consistency | Financial transactions. Inventory counts. User balances. | Social feeds. Analytics. Recommendations. Any data where "latest" vs "almost latest" is indistinguishable. |
| Deployment | Single-region | Multi-region | 99.9% availability is sufficient. User base is regional. Data sovereignty is not required. | 99.99%+ availability required. Global user base. Need geographic latency optimization. Regional disaster recovery required. |

## CAP Theorem Positioning

### The Practical CAP Decision

In practice, every distributed system is AP during a partition. You cannot have CA in a distributed system because partitions are inevitable. The real question is: **how much inconsistency can you tolerate during a partition?**

**DynamoDB and Cassandra choose AP.** During a partition, they serve stale data. Different clients may see different versions of the same data. When the partition heals, they reconcile conflicts using last-write-wins or application-specific conflict resolution.

**Zookeeper and etcd choose CP.** During a partition, if a majority of nodes cannot be reached, the minority side stops serving writes. This means some clients cannot write until the partition heals. This is acceptable for configuration data where consistency matters more than write availability.

**PostgreSQL with streaming replication is CP** during a failure scenario. If the primary fails and the replica has not yet received the last writes, those writes are lost. The system chooses consistency (no partial data) over availability (the lost writes are gone forever).

For most web applications, **AP is the right choice**. A user seeing a slightly stale feed (missing the last 2 seconds of updates) is acceptable. A user seeing an error page is not.

## Failure Modes

A system design must account for the specific ways components fail. Each failure mode has a signature, a root cause, a blast radius, and a recovery strategy.

### Failure Mode 1: Database Primary Overload

**What happens:** The database primary CPU reaches 100%. Queries queue. Response time climbs from 5ms to 500ms to 5 seconds. Connections pile up. The application server connection pool fills with waiting threads. The application stops responding to new requests. The load balancer marks the application servers as unhealthy.

**Root cause:** A query pattern change. A new feature deploys a query that does a full table scan on a 50-million-row table. Or traffic spikes above the database's write throughput capacity. Or a missing index causes a join that takes 30 seconds.

**Blast radius:** Every component that depends on the database. The application servers, the cache warmers, the background workers, the reporting system. In a tightly coupled system, one slow query brings down everything.

**Detection:** Database CPU crosses 80%. Query latency p99 crosses 100ms. Active connections approach max_connections. Slow query log shows queries running longer than 5 seconds.

**Recovery:**
1. Kill the slow query immediately: `SELECT pg_terminate_backend(pid)`.
2. If the query is from a new deploy, roll back the deploy.
3. Add the missing index: `CREATE INDEX CONCURRENTLY ...`.
4. If traffic is the cause, scale up the database instance or add read replicas.
5. Implement query timeout at the application level: kill any query that runs longer than 5 seconds.

### Failure Mode 2: Memory Leak in Application Server

**What happens:** An application server's memory usage grows steadily over days or weeks. The server uses more and more RAM until the OS out-of-memory killer terminates it. The load balancer detects the health check failure and routes traffic to remaining servers. Those servers also have memory leaks, so they fail one by one. The system degrades gradually over hours.

**Root cause:** A reference that is never released. A cache that grows without bound. An event listener that accumulates callbacks. A thread-local variable that persists across requests. In Java, a forgotten static collection. In Python, a global list that items are appended to but never removed from. In Go, goroutines that are created but never exit.

**Blast radius:** The affected application server dies. Other servers take its traffic, which accelerates their own memory leaks. In a cascading failure, all servers in the pool die within minutes of each other.

**Detection:** Memory usage increases monotonically over time. A memory usage trend chart shows steady growth with no plateau. Alert when memory usage exceeds 80% of available RAM after a restart.

**Recovery:**
1. Restart the affected servers to reclaim memory. This is a temporary fix.
2. Generate a heap dump before restarting: `jmap -dump:format=b,file=heap.hprof <pid>`.
3. Analyze the heap dump to find the leaking data structure. Look for classes that have far more instances than expected.
4. Fix the leak and deploy.

**Prevention:** Set a JVM max heap size (`-Xmx`) that is lower than the instance's total RAM. This prevents the application from consuming all memory and forces an OutOfMemoryError that can be handled gracefully (return 503 instead of crashing the OS).

### Failure Mode 3: Downstream API Dependency

**What happens:** Your system calls an external API (payment gateway, SMS provider, weather service). The API slows down or returns errors. If your system has no timeout or circuit breaker, your threads block waiting for the API. Threads fill up. The application stops processing any request — even requests that do not need the external API.

**Root cause:** The external service is overloaded, undergoing maintenance, or buggy. Or your network path to the service has a problem.

**Blast radius:** Any endpoint that depends on the external service fails. If the external call is in the critical path of a high-traffic endpoint, the blast radius is large.

**Detection:** Error rate for the specific external API endpoint spikes. Latency p99 for the endpoint crosses the timeout threshold. Thread pool utilization reaches 100%.

**Recovery:** The circuit breaker opens. The system returns a cached response or a degraded response. The engineering team contacts the external provider. If the provider cannot resolve, implement a fallback to a different provider.

### Failure Mode 3: Cache Failure

**What happens:** The Redis cluster restarts, or a configuration change causes mass eviction, or the network path to Redis is interrupted. Every request that previously hit the cache now misses and hits the database.

**Root cause:** Redis out of memory, restart after upgrade, network partition, or a cache key pattern that floods the cache with unique keys (cache stampede).

**Blast radius:** The database receives 10-100x normal traffic as every read request bypasses the cache. If the database cannot handle this load, it overloads, and the system fails.

**Detection:** Redis error rate increases. Cache hit rate drops from 95% to under 10%. Database QPS spikes 10x.

**Recovery:**
1. If Redis is down, restart or fail over.
2. If the database is being overwhelmed, scale up or add read replicas.
3. If the cache was evicted due to memory pressure, increase maxmemory or add Redis nodes.
4. Re-populate the cache gradually (not all at once — this is the "cache stampede" problem).

### Failure Mode 4: Thundering Herd

**What happens:** A cache entry for a popular URL expires. 1,000 requests arrive simultaneously for the same URL. All 1,000 miss the cache. All 1,000 hit the database. The database handles 1,000 concurrent identical queries.

**Root cause:** A cache key with a short TTL on a high-traffic item. Or a restart that clears the entire cache.

**Blast radius:** The database is hit with N concurrent identical queries where N is the number of application server threads. If N is large (hundreds), the database may overload.

**Recovery:** Implement a **mutex** on the cache key. When the first request misses the cache, it acquires a lock, queries the database, and populates the cache. Other requests that miss the cache either wait briefly for the lock or are served stale data while the cache refreshes.

```python
def get_url_with_mutex(short_key):
    cached = redis.get(f"url:{short_key}")
    if cached:
        return cached

    # Only one thread should query the database
    lock_key = f"lock:url:{short_key}"
    if redis.setnx(lock_key, "1"):  # set if not exists, atomic
        redis.expire(lock_key, 2)  # auto-release after 2s
        url = db.query("SELECT url FROM urls WHERE key = ?", short_key)
        redis.setex(f"url:{short_key}", 3600, url)
        redis.delete(lock_key)
        return url
    else:
        # Another thread is fetching. Wait briefly or serve stale.
        time.sleep(0.05)
        cached = redis.get(f"url:{short_key}")
        if cached:
            return cached
        # Fall back to database as last resort
        return db.query("SELECT url FROM urls WHERE key = ?", short_key)
```

### Failure Mode 6: Data Corruption

**What happens:** A bug in the application writes incorrect data to the database. User balances are set to zero. Product prices are multiplied by 100. User profile privacy settings are changed from "private" to "public." The corrupted data is replicated to all replicas.

**Root cause:** A code defect. A migration that applies incorrectly. A race condition in concurrent writes. An external attacker exploiting an injection vulnerability.

**Blast radius:** All users whose data was affected. If the corruption is in a critical field (balance, price, access control), the blast radius is the entire user base. The corruption propagates to all replicas, making recovery from replicas impossible.

**Detection:** Monitoring alerts when accounting totals do not balance. User support tickets reporting incorrect data. Automated data validation (checksums, referential integrity checks, business rule validation) catching the anomaly.

**Recovery:**
1. Stop the corruption: roll back the deploy, disable the API endpoint, or block the malicious traffic.
2. Restore from backup: point-in-time recovery (PITR) to the last known-good state.
3. Replay valid transactions from the application logs.
4. Validate the restored data against business rules.
5. Restore service.

**Prevention:** The only defense against data corruption is **point-in-time recovery (PITR)** and **immutable backups**. Enable PITR on every database. Test backup restoration quarterly. Run data validation checks continuously. Use the "principle of least privilege" on database access — the application should not have DROP TABLE permissions.

### Failure Mode 7: Dependency Cascade (Domino Effect)

**What happens:** Service A calls Service B, which calls Service C, which calls Service D. When Service D slows down, Service C's threads pile up waiting for D. Service B's threads pile up waiting for C. Service A's threads pile up waiting for B. All four services are now slow. The system-wide capacity drops to the capacity of the slowest service in the chain.

**Root cause:** Synchronous dependency chain with no timeouts, no circuit breakers, and no fallbacks. One slow service propagates latency to every upstream service.

**Blast radius:** Every service in the call chain, and every service that depends on those services. In a deeply nested microservices architecture, a single slow service can bring down 50+ services.

**Detection:** A distributed tracing system (Jaeger, Zipkin) shows that 100% of request time is spent waiting for Service D. Each upstream service shows increasing thread pool utilization.

**Recovery:**
1. Scale up Service D (the root cause).
2. If scaling does not help, roll back the latest deploy to Service D.
3. If the root cause is resource exhaustion, throttle requests to Service D.
4. Implement circuit breakers in Services A, B, and C to fail fast when D is slow.

**Prevention:**
- Set aggressive timeouts on every inter-service call (500ms recommended for internal services).
- Implement circuit breakers on every dependency.
- Use async communication (message queues) for non-critical dependencies.
- Limit the depth of synchronous call chains to a maximum of 3.

### Failure Mode 8: Cascading Region Failure

**What happens:** A data center loses power, or an AWS region experiences a multi-hour outage. All instances in that region go down simultaneously.

**Root cause:** Natural disaster, power grid failure, or cloud provider service disruption. AWS us-east-1 has had multiple multi-hour outages. The 2017 S3 outage was caused by a typo that deleted too many servers.

**Blast radius:** Everything running in the affected region. For single-region systems, total outage. For multi-region systems, partial outage with traffic shifted to remaining regions.

**Recovery:** If you have multi-region deployment with DNS failover, Route53 detects the health check failure and routes traffic to the next region. The database failover must be triggered manually or by automated tooling. Recovery time: 5-30 minutes depending on automation.

## Anti-Patterns Hall of Shame

These are the most common mistakes engineers make. Recognizing them in your own design is a sign of growth.

### Anti-Pattern 1: Over-Engineering

**The symptom:** You design a 12-microservice system with Kafka, Redis, Cassandra, Kubernetes, and service mesh for a project that has 100 users.

**Why it happens:** Engineers over-engineer because it is fun, because they want to use the latest technology, or because they misread the scale requirements.

**The fix:** Start with the simplest thing that works. A monolith with a single database. When you hit a specific scaling limit, fix that limit. Do not build for problems you have not yet encountered.

**The mantra:** "Make it work. Make it right. Make it fast." In that order.

### Anti-Pattern 2: Premature Optimization

**The symptom:** You spend two weeks optimizing a database query that runs 100 times a day and completes in 50ms. You ignore a query that runs 1 million times a day and completes in 500ms.

**Why it happens:** The slow query is on a table you understand well. The fast-but-frequent query involves a complex join you are afraid to touch.

**The fix:** Measure first. Profile production traffic. Find the queries that consume the most total time (latency x frequency). Optimize those. A 500ms query running 1M times is 500,000 seconds per day. A 50ms query running 100 times is 5 seconds per day.

### Anti-Pattern 3: Ignoring Non-Functional Requirements

**The symptom:** You design a complete data model, build all the API endpoints, write comprehensive tests. You never ask: "What happens when the database goes down? What is our latency target? How many requests per second must we handle?"

**Why it happens:** Non-functional requirements (latency, throughput, availability, security) are harder to specify and measure. Teams focus on features because features are visible.

**The fix:** Define non-functional requirements in the design document before writing code. Every design review must include: target latency (p50, p99), target throughput (peak QPS), availability target (nines), and RPO/RTO for disaster recovery.

### Anti-Pattern 4: No Redundancy

**The symptom:** One database server. One application server. One Redis instance. One load balancer. "We will add redundancy later."

**Why it happens:** Cost cutting. "It works fine in development." "The cloud instance is reliable."

**The fix:** A single-instance deployment is not a production system. From day one, run two of everything in the critical path. The cost of two small instances is usually under $100/month. The cost of an outage can be $100,000/hour.

### Anti-Pattern 5: Single Database for Everything

**The symptom:** One PostgreSQL database stores user profiles, URL mappings, click analytics, session data, application logs, billing records, and feature flags.

**Why it happens:** It is the simplest setup. One connection string. One backup. One thing to monitor.

**The fix:** Separate concerns early. User data goes in PostgreSQL (transactional, ACID). Click analytics goes in a time-series database or append-only table. Session data goes in Redis (ephemeral, fast). Logs go to an external logging service. Feature flags go to a configuration service. Mixing transactional and analytical workloads in one database guarantees neither works well at scale.

### Anti-Pattern 6: Ignoring Cold Start

**The symptom:** You deploy a serverless function that processes image uploads. During a quiet hour, no functions run. When the first upload of the hour arrives, the function takes 3 seconds to respond because AWS Lambda needs to load the image processing library into memory.

**Why it happens:** Serverless platforms idle functions after a period of inactivity. The first request after idle triggers a cold start.

**The fix:** Pre-warm serverless functions with scheduled invocations (a CloudWatch event that invokes the function every 5 minutes). Or use provisioned concurrency (keep N instances warm at all times). Or recognize that serverless is a poor fit for latency-sensitive workloads.

### Anti-Pattern 7: Ignoring Network Reliability

**The symptom:** Your system makes 10 internal HTTP calls per user request. No retries. No timeouts. No circuit breakers. When one internal service is slow, all 10 calls queue up and the user request takes 10x longer.

**Why it happens:** Local development and testing environments have zero network latency and zero failure rate. Engineers do not experience network failures during development, so they do not design for them.

**The fix:** Every network call must have:
1. A connection timeout (how long to wait for the TCP handshake).
2. A read timeout (how long to wait for the response body).
3. A retry policy (how many times to retry on failure).
4. A circuit breaker (stop calling after sustained failures).
5. A fallback (what to return when the call fails).

Latency budget per service call: 50ms max. If the service does not respond in 50ms, the circuit is tripped for that request.

### Anti-Pattern 8: Synchronous Everything

**The symptom:** Every operation in the system is synchronous. Creating a new URL sends a confirmation email, logs analytics, updates the user's profile in a search index, and refreshes a materialized view — all before returning the response to the user.

**Why it happens:** It is the simplest programming model: sequential code. No queues. No async workers. No eventual consistency.

**The fix:** Identify operations that do not need to block the response. Email sending: queue it. Analytics logging: batch it. Search index update: eventually update it. Materialized view refresh: schedule it. Only database writes and critical reads should be synchronous. Everything else can wait.

### Anti-Pattern 9: Hardcoded Configuration

**The symptom:** Connection strings, timeouts, retry counts, feature flags, and third-party API keys are hardcoded in the application source code. Changing a database connection string requires a code change, a PR review, a build, and a deploy.

**Why it happens:** It is the fastest way to get started. "We will extract it into configuration later." Later never comes.

**The fix:** Externalize all configuration from code. Use environment variables for secrets, a configuration file for non-sensitive settings, and a feature flag system (LaunchDarkly, Unleash) for behavioral toggles. Configuration changes should be deployable independently from code changes.

```
# Bad: hardcoded
db_url = "postgresql://user:password@db.example.com:5432/prod"

# Good: environment variable
db_url = os.environ.get("DATABASE_URL")
```

### Anti-Pattern 10: Assuming Linear Scaling

**The symptom:** "We have one server handling 1,000 RPS. To handle 10,000 RPS, we add 9 more servers. Simple."

**Why it happens:** Engineers assume that performance scales linearly with the number of servers. In reality, adding servers introduces coordination overhead, network contention, and load balancer capacity limits. At 10 servers, the overhead is negligible. At 1,000 servers, the overhead can consume 20% of total capacity.

**The fix:** Account for coordination overhead in capacity planning. Use the formula: effective_capacity = N x single_node_capacity x (1 - overhead(N)), where overhead increases logarithmically with N. For small N (under 20), overhead is approximately 0. For N=100, overhead is approximately 5%. For N=1,000, overhead is approximately 10-15%.

### Anti-Pattern 11: Not Planning for Data Growth

**The symptom:** "Our database has 500 GB of data. A 1 TB SSD handles it easily." Two years later, the database has 8 TB of data and queries that used to take 2ms now take 200ms because indexes no longer fit in memory.

**Why it happens:** Teams design for current data volume, not projected growth. They assume they can "add more storage later" without considering the query performance implications of larger datasets.

**The fix:** Project data growth for 3 years and design the schema, indexes, and partitioning for that volume from the start. Partition tables by time (monthly or quarterly). Archive old data to cheaper storage. Monitor index-to-RAM ratio — if the active index set exceeds available RAM, add RAM or partition the data.

## When NOT to Use This

Formal system design is not always the right approach. Here is when to skip it.

**The decision matrix captures the most common choices, but every real system is a blend. A typical production system uses a relational database for transactional data, a document store for user-generated content, a cache for hot data, a message queue for async processing, and an object store for files and images. The art is knowing which data goes where.

### Applying the Matrix to a Real Scenario

Imagine you are designing the user profiles service for a social media platform with 50 million users. Each profile has: display name, bio, avatar image, follower count, list of recent posts, and privacy settings.

Walk through the matrix row by row:

**Database type:** Profile data is document-shaped (a user has a name, bio, settings, and a list of posts). No complex joins between profiles. Each profile is read and written independently. This points to NoSQL (DynamoDB or MongoDB). But the follower count requires atomic increments — both DynamoDB and MongoDB support atomic counters. Choose NoSQL.

**Monolith vs microservices:** The profile service is one of 20 planned services. Separate team will own profiles. This points to microservices. But the team has 8 engineers — too small for a full microservices deployment. Recommendation: build the profile service as a separate deployable unit (one service with its own database) but keep the deployment simple (one codebase, one pipeline). This is a "service" without the overhead of a full microservices platform.

**Synchronous vs async:** Reading a profile must be synchronous (the user needs the data now). Updating the profile can be synchronous (the user expects immediate confirmation). But updating the search index and the follower feed can be async.

**Cache:** Profiles are read 100x more than they are written. Each read must be under 10ms. A distributed cache is essential. Every profile read goes to Redis first, then to the database on miss.

**Read replicas:** With NoSQL and a cache hit rate of 95%, the database handles only 50,000 reads per second at peak. A properly configured DynamoDB table handles this without replicas. No replicas needed.

**Data partitioning:** 50 million profiles at 5 KB each = 250 GB. Fits on a single DynamoDB table (unlimited storage, auto-partitioning). No manual sharding needed.

**Consistency:** Follower count can be eventually consistent (if it shows 1,002 instead of 1,003 for 2 seconds, no one notices). User display name should be strongly consistent (a user who just changed their name should see the new name immediately). Solution: DynamoDB strongly consistent reads for user-facing data, eventually consistent reads for derived data.

**Deployment:** Single-region is sufficient for 50 million users in one country. Add multi-region when expanding globally.

## When NOT to Use This:** If you are validating a product idea with fewer than 100 users, you do not need a load balancer, a Redis cluster, or multi-region deployment. You need a single server running a framework. System design starts when you have product-market fit and traffic that hurts.

**When the problem is trivially small:** A script that runs once a day to generate a report does not need system design. A website for a local business with 50 monthly visitors does not need horizontal scaling. The cost of system design (time, complexity, infrastructure) exceeds the benefit.

**When you are the only engineer:** If you are the only developer on a project, the overhead of microservices, message queues, and distributed caching slows you down. Use the simplest architecture that works. Decompose only when the monolith hurts.

**When the requirement explicitly avoids it:** If the specification says "max 500 users, single server, acceptable downtime during maintenance," designing for 10 million DAU with 99.99% availability is over-engineering. Build what the requirements demand.

**When you are solving a business-model problem with technology:** If users leave because your product is confusing, adding a caching layer does not help. If your pricing is wrong, a load balancer does not fix it. System design optimizes technical performance. It does not fix business problems.
