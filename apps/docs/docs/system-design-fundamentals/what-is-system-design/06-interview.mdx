System design interviews are the highest-weighted round in FAANG and top-tier tech company interviews. Unlike coding interviews, there is no single correct answer. The interview evaluates your process: how you approach ambiguity, how you make trade-offs, and how you communicate complex ideas under time constraints.

This section covers exactly how system design interviews work, what interviewers evaluate at each level, and the framework you need to demonstrate senior-level design thinking.

## How This Topic Appears

System design questions fall into four categories. Practice recognizing the category from the phrasing — your approach changes based on what the interviewer is actually asking.

### Direct Questions (Design a Specific System)

These are the most common. The interviewer names a system and asks you to design it.

1. "Design a URL shortening service like TinyURL."
2. "Design a real-time chat application like WhatsApp."
3. "Design a ride-sharing service like Uber."
4. "Design a news feed system like Facebook feed."
5. "Design a video streaming platform like YouTube."
6. "Design a distributed key-value store like DynamoDB."
7. "Design a web crawler that can index the entire internet."
8. "Design a rate-limiting system for a public API."

### Debugging Questions (Find the Problem)

These questions present a scenario where something is wrong and ask you to diagnose and fix it.

9. "Our API latency went from 20ms to 500ms after a deploy. All we changed was adding a new endpoint. What happened and how do we fix it?"
10. "The database CPU is at 100% every day at 2:15 PM for 5 minutes. Nothing is scheduled. What do you check first?"

### Trade-Off Questions (Compare Approaches)

These questions present two approaches and ask you to choose.

11. "Should we use PostgreSQL or DynamoDB for storing user session data?"
12. "Should we use synchronous or asynchronous processing for sending emails?"
13. "Should we deploy in a single region or multiple regions?"

### Estimation Questions (Calculate Capacity)

These questions focus on the quantitative aspect of system design.

14. "Estimate the storage, bandwidth, and server count needed for a service that handles 1 billion photos per day."
15. "How many database read replicas do we need for 50,000 QPS?"

### Recognizing the Question Type Quickly

In a live interview, you have seconds to categorize the question and adjust your approach. Memorize this decision tree:

```
Is the question a specific system name?
  YES → Direct design question. Follow the 6-step framework.
  NO  → Is the question about a performance problem?
         YES → Debugging question. Start with "what changed?"
         NO  → Is the question comparing two approaches?
               YES → Trade-off question. Present both, then recommend.
               NO  → Is the question asking for a number?
                     YES → Estimation question. Show your math.
```

**Example:** "Design WhatsApp" → Direct design. Follow the 6 steps.
**Example:** "Our database got slow after a deploy" → Debugging. Start with "what was in the deploy?"
**Example:** "Should we use REST or gRPC?" → Trade-off. Compare both.
**Example:** "How much storage for 1 billion photos?" → Estimation. Calculate.

**Example:** "Our database got slow after a deploy" → Debugging. Start with "what was in the deploy?"
**Example:** "Should we use REST or gRPC?" → Trade-off. Compare both.
**Example:** "How much storage for 1 billion photos?" → Estimation. Calculate.

### How Interviewers Score Your Answer

Most FAANG companies use a rubric with a 1-4 score on each dimension. Understanding the rubric helps you know where to focus:

| Dimension | Score 1 (Fail) | Score 2 (Borderline) | Score 3 (Pass) | Score 4 (Strong Hire) |
|---|---|---|---|---|
| Requirements | No clarifying questions | Asks 1-2 basic questions | Asks 4-5 relevant questions | Probes constraints, non-functionals, edge cases |
| Capacity | No numbers | Guesses 1-2 numbers | Calculates QPS, storage, bandwidth | All of L3 + peak, growth, cost estimates |
| High-level design | Missing key components | Has components, no data flow | Clear diagram with data flow | Considers trade-offs proactively, mentions alternatives |
| Deep dive | Vague descriptions | Can describe one component in detail | Can describe 2-3 components with specifics | Can discuss failure modes, config, and operations for each |
| Communication | Rambling, no structure | Follows framework with prompting | Clear structured walkthrough | Teaches the interviewer, handles curveballs gracefully |

Aim for Score 3 on every dimension. Score 4 on communication — it is the most heavily weighted dimension at senior levels.

## What the Interviewer Is Actually Testing

Interviewers evaluate your performance on five dimensions. Each dimension maps to a specific skill needed in production engineering.

### Traffic Estimation (Capacity Planning)

**What they look for:** Can you convert a user count into QPS, storage, and bandwidth? Can you reason about peak traffic versus average traffic? Do you know the difference between a daily operation count and a per-second rate?

**Red flag:** Throwing out numbers without a calculation path. Guessing 100 servers with no reasoning.

### Requirements Gathering (Requirements Elicitation)

**What they look for:** Do you ask clarifying questions before designing? Do you understand functional requirements (what the system does) versus non-functional requirements (latency, availability, durability)? Do you push back on ambiguous or contradictory requirements?

**Red flag:** Starting the design immediately without asking a single question. Assuming you know all the requirements.

### Architecture and Components (High-Level Design)

**What they look for:** Can you decompose a system into components? Do you know which components exist (load balancer, cache, database, queue, worker, CDN)? Can you draw a clear component diagram and explain the data flow between components?

**Red flag:** Drawing a single box labeled "server" and a single box labeled "database" for a system that handles 10 million DAU.

### Deep Dive (Detailed Design)

**What they look for:** When asked to go deeper on a specific component, do you have the depth? Can you discuss exactly how the database handles the read pattern? Can you explain the cache key schema? Can you describe the data model with specific fields and types?

**Red flag:** Having only surface-level knowledge. "We use Redis" without explaining what data is cached, what the cache key format is, what TTL is used, or what happens on a cache miss.

### Trade-Off Analysis (Decision Making)

**What they look for:** Do you know that every decision has a cost? Can you articulate what you are giving up when you choose one approach over another? Do you acknowledge trade-offs without being prompted?

**Red flag:** Presenting your design as the only correct approach. "We should use NoSQL because it is fast" without mentioning consistency trade-offs or query limitations.

## Level-by-Level Expectations

Interview expectations are calibrated to the level. A strong answer at L4 is a weak answer at L6. Know what level you are targeting and what the bar is.

### L3 / L4 (Junior / Mid-Level)

**Expectations:** Can follow a structured framework. Can identify basic components. Can handle single-server to small-scale designs.

**Minimum knowledge:** Load balancers, caching (one level), primary-replica database, simple API design.

**Vocabulary to demonstrate:** "single point of failure," "horizontal scaling," "cache hit rate," "read replica."

**Sample answer quality:** A clear, linear walkthrough of the design using the framework. Mentioning at least 2-3 trade-offs unprompted. Unable to go deep on any specific component without guidance.

**What a passing L4 answer looks like (Design URL shortener):**
"You define the requirements: 100M URLs per day, 10:1 read-to-write ratio. You estimate 12,000 RPS peak. You need a database for persistence — PostgreSQL with a primary for writes and two read replicas for reads. You need Redis in front of the database with cache-aside pattern: write to DB, delete cache key; read from cache, miss goes to DB. You need a load balancer in front of stateless application servers. The application generates base62 keys from an auto-increment ID. For analytics, you write click events to Kafka and process them asynchronously."

### L5 (Senior Engineer)

**Expectations:** Can design for scale without hand-holding. Considers failure modes proactively. Discusses monitoring, deployment, and operations.

**Additional knowledge:** Consistent hashing, circuit breakers, multi-region replication, data partitioning strategies, backpressure, graceful degradation.

**Vocabulary to demonstrate:** "circuit breaker," "consistent hashing," "idempotency key," "backpressure," "degraded mode," "eventual consistency," "partition tolerance."

**Sample answer quality:** All of L4 plus explicit failure analysis, multiple trade-off discussions with balanced reasoning, and a clear understanding of when simpler approaches would work.

**What a passing L5 answer adds:**
"And we need to handle failure modes. If Redis goes down and all requests hit the database, the database will overload. We either scale up the database temporarily or implement a local in-memory cache as a second layer. If a database query is slow, we have a 5-second query timeout and a circuit breaker that stops sending queries to the database if error rate exceeds 5%. For deployment, we use rolling updates with health checks so no single server downtime affects users. For monitoring, we track three key metrics: cache hit rate (alert if under 90%), database query latency p99 (alert if over 100ms), and replication lag (alert if over 5 seconds)."

### L6 (Staff Engineer)

**Expectations:** Can design for an entire organization. Considers organizational constraints, team structure, and migration strategies. Balances technical ideal with practical reality.

**Additional knowledge:** Cost modeling, migration strategies (strangler fig pattern), organizational scalability (Conway's law), multi-year roadmaps, build vs buy analysis.

**Vocabulary to demonstrate:** "strangler fig pattern," "Conway's law," "opportunity cost," "ROI analysis," "brownfield vs greenfield," "capacity planning across quarters," "runaway cost."

**Sample answer quality:** All of L5 but acknowledges organizational constraints. Discusses what the first version looks like (simpler) versus the long-term vision. Weighs build-vs-buy. Discusses migration strategy from a legacy system.

**What a passing L6 answer adds:**
"However, we are not building this from scratch. We have an existing monolith that already handles 1M DAU. We use the strangler fig pattern: we build the new URL shortening service alongside the monolith, route a percentage of traffic to it, validate correctness, then incrementally increase traffic. The migration takes 6-9 months. During that time, both systems must coexist with data synced in a consistent format. The team organization also matters: we should structure teams around business capabilities (one team for URL service, one for analytics) rather than technical layers, because Conway's law says the system will mirror the communication structure of the teams that build it."

### L7 / L8 (Principal / Distinguished Engineer)

**Expectations:** Shapes the technical direction of the entire company. Makes decisions that affect multiple organizations. Balances technical debt against business priorities.

**Additional knowledge:** Industry trends, trade-offs at the organizational level, influencing without authority, setting technical standards, defining OKRs for engineering orgs.

**What a passing L7 answer adds:**
Instead of building a custom URL shortener, this L7 evaluates whether the company should buy a URL management platform. They calculate the total cost of ownership: 3 engineers for 18 months to build versus $50K/year for a SaaS solution. They determine that URL shortening is not a competitive advantage — the company's core value is in analytics and user engagement, not URL management. They recommend buying the SaaS platform and integrating it into the analytics pipeline. This frees the engineering team to focus on the company's actual differentiator.

## The FAANG Framework for Answering

This six-step framework is the standard approach taught at top tech companies. Practice it until it becomes automatic.

### Post-Interview Self-Evaluation

After each practice session or real interview, evaluate yourself against this checklist:

**Did I clarify requirements?**
- How many questions did I ask? (Target: 8-10)
- Did I ask about both functional and non-functional requirements?
- Did I ask about scale (DAU, QPS)?
- Did I ask about constraints (budget, team, timeline)?

**Did I estimate capacity?**
- Did I calculate QPS from DAU?
- Did I calculate storage from record size?
- Did I estimate bandwidth?
- Did I use peak (3x average) or average numbers?

**Did I design the high-level architecture?**
- Did I include a load balancer?
- Did I include stateless application servers?
- Did I include a cache?
- Did I include a database with read replicas?
- Did I include a message queue for async processing?
- Did I draw and explain the data flow?

**Did I go deep on at least one component?**
- Did I describe the database schema with specific fields?
- Did I explain the cache key format and TTL?
- Did I describe the API contract?

**Did I discuss trade-offs?**
- Did I mention at least 3 trade-offs unprompted?
- Did I acknowledge alternatives?
- Did I explain why I chose my approach over the alternative?

**Did I identify bottlenecks?**
- Did I name the most likely bottleneck?
- Did I propose a mitigation?

**Did I communicate well?**
- Did I maintain a clear structure (the 6-step framework)?
- Did I respond to the interviewer's hints?
- Did I keep within the time allocation?

### Virtual vs In-Person Interview Differences

System design interviews may be conducted virtually (phone screen or Zoom) or in person (whiteboard). Adapt your approach:

**In-person (whiteboard):**
- Use the whiteboard space efficiently. Draw components from left to right (user side to data side).
- Use different colors for different concerns: green for traffic flow, red for failure points, blue for data stores.
- Write labels on every box and every arrow.
- Leave room for expansion. Do not fill the entire board with the first diagram.
- Take a photo of the whiteboard at the end for your records.

**Virtual (Zoom + shared doc or drawing tool):**
- Screen share a document (Google Docs, Notion) with a table structure, not a blank page. Pre-format: "Requirements:", "Estimation:", "Architecture:", "Deep Dive:", "Trade-offs:", "Bottlenecks:"
- If the tool supports drawing (Miro, Excalidraw, Jamboard), use it. If not, describe the architecture verbally and type bullet points.
- Pause after each section and ask "Does this match your expectations? Should I go deeper on any component?"
- Virtual interviews favor clarity over speed. The interviewer cannot see your body language. Be explicit about what you are drawing or writing.

### Step 1: Clarify Requirements (2-3 minutes)

Ask clarifying questions. Do not start drawing until you have the full picture.

**Questions to ask:**
- "What are the functional requirements? What should this system do?"
- "What are the non-functional requirements? What latency, availability, and durability targets?"
- "How many users? What region?"
- "Is this a greenfield design or a replacement for an existing system?"
- "Are there any constraints I should know about? Budget? Team size? Timeline?"

**Example for URL shortener:**
- "100 million URLs created per day"
- "1 billion redirects per day"
- "Read-to-write ratio of 10:1"
- "p99 latency under 100ms"
- "99.99% availability target"
- "Global user base, multi-region"
- "URLs persist forever"

**Common mistake:** Asking too few questions. The minimum is 5 questions. A good interview answer asks 8-10 questions. An exceptional answer asks follow-up questions based on the interviewer'ss.

**Question categories (memorize these):**

| Category | Example Questions |
|---|---|
| Scale | How many DAU? How many writes per day? Reads per day? |
| Performance | What is the latency target at p50 and p99? |
| Reliability | What is the availability target? RPO? RTO? |
| Geography | Single region or multi-region? Which regions? |
| Data | How long to store data? Any retention or compliance requirements? |
| Team | How many engineers? What is their experience level with this tech stack? |
| Existing | Is this a new system or replacing an existing one? Any migration constraints? |

### Step 2: Estimate Capacity (3-4 minutes)

Convert requirements into concrete numbers.

**Calculations:**
- QPS: traffic per day divided by 86,400
- Peak QPS: average QPS multiplied by 3-5
- Storage: record size multiplied by record count
- Bandwidth: request size multiplied by QPS
- Server count: peak QPS divided by per-server capacity

**Example for URL shortener:**
- 100M writes / 86,400 = 1,157 writes/sec average
- 1B reads / 86,400 = 11,574 reads/sec average
- Peak: 3,500 writes/sec + 35,000 reads/sec = approximately 40,000 RPS peak
- Each URL record: approximately 230 bytes = 23 GB/day
- Yearly storage: 8.4 TB
- Need roughly 20 application servers at 2,000 RPS each

### Step 3: High-Level Design (5-8 minutes)

Draw the component diagram. Show every component and the connections between them.

**Components to consider:**
- DNS routing
- CDN for static assets
- Load balancer (describe the routing algorithm)
- Application servers (stateless, auto-scaled)
- Cache layer (Redis, describe key schema and TTL)
- Primary database (describe type, schema, indexes)
- Read replicas
- Message queue and workers (for async processing)
- Analytics pipeline

**Draw and explain:**
"Users go through Route53 for DNS resolution, then hit an Application Load Balancer. The ALB distributes traffic across a pool of stateless application servers. For reads, the application checks Redis first. On cache miss, it queries PostgreSQL read replicas. For writes, the application writes to the PostgreSQL primary and invalidates the cache key. For analytics, the application publishes events to Kafka workers that batch-insert into ClickHouse."

### Step 4: Deep Dive (10-15 minutes)

The interviewer will choose one component and ask you to go deeper. This is where depth matters.

**Common deep dive topics:**
- **Database schema:** Show the actual SQL tables, indexes, partition strategy.
- **API contracts:** Show the request and schemas for key endpoints.
- **Cache strategy:** Show the cache key format, TTL, eviction policy, read/write pattern.
- **Data flow:** Trace exactly what happens from request to, including error paths.
- **Key generation:** For URL shortener, explain base62 encoding, ID generation, collision handling.

**Example deep dive for database schema:**
"The URLs table is partitioned by created_at month so old partitions can be compressed and potentially archived. The primary key on short_key provides fast point lookups. There is a secondary index on user_id for listing a user's URLs with pagination. The clicks table is append-only with no updates — we use a columnar storage format to reduce storage by 70% compared to row storage."

### Step 5: Discuss Trade-Offs (3-5 minutes)

Proactively discuss what you are giving up with each choice. This step is what separates L4 from L5.

**Trade-offs to discuss:**
- "I chose PostgreSQL over DynamoDB because the URL data has a relational structure and we need joins for analytics. The trade-off is that we cannot scale writes as easily — PostgreSQL requires application-level sharding beyond 30,000 writes/sec."
- "I chose Redis cache-aside over write-through cache because the database is the source of truth and cache loss should be recoverable. The trade-off is a 20-50ms penalty on cache misses for the first request after expiry."
- "I chose asynchronous analytics processing over synchronous because we do not want click tracking to slow down the redirect. The trade-off is that analytics may be up to 5 seconds behind real time."
- "I chose horizontal scaling over vertical scaling because the system needs to handle growth beyond a single machine's limit. The trade-off is operational complexity — managing 20 servers is harder than managing 1."
- "I chose eventually consistent reads for the feed because read latency must be under 10ms. The trade-off is that a user may not see their latest post for 2-3 seconds."

**Pro tip:** After stating the trade-off, acknowledge the alternative: "If the requirements were different — if we needed sub-millisecond write latency at any cost — I would revisit this decision and choose a different approach."

### Step 6: Wrap Up and Identify Bottlenecks (2-3 minutes)

Summarize the design and identify the most likely bottleneck. This shows you are thinking about the system in production, not just on the whiteboard.

**Wrap-up structure:**
"To summarize: we have a stateless, horizontally-scaled web tier behind a load balancer, with Redis caching for reads, PostgreSQL primary-replica for persistence, Kafka for async analytics, and multi-region deployment for availability and latency.

The most likely bottleneck is the database primary write throughput. At 3,500 writes/sec peak, a well-tuned PostgreSQL primary handles this, but as traffic grows, we would need to shard the database by user_id range.

The second bottleneck is the Redis cache capacity during a cache-miss event. If a node fails, the database takes the full 35,000 reads/sec. We mitigate this by running Redis in a cluster with replication and configuring a connection pool limit on the database side."

## Phrase Bank

Use these phrases in your interview answers. Each phrase signals a specific strength.

**For demonstrating experience:**
- "The bottleneck in this system is the write path to the database."
- "I would implement a circuit breaker on the downstream API call."
- "We need idempotency keys to handle retries safely."
- "I would use exponential backoff with jitter for the retry policy."
- "The cache-aside pattern works best here because reads are 10x writes."
- "We partition by user_id to distribute write load evenly across shards."
- "I would migrate using the strangler fig pattern to avoid a big-bang cutover."
- "The p99 latency matters more than p50 for this use case."

**For showing trade-off awareness:**
- "The trade-off here is consistency versus availability."
- "This adds latency but improves durability."
- "We could make it simpler but we lose fault tolerance."
- "This approach optimizes for read throughput at the expense of write latency."
- "The cost of this approach is N dollars per month, but it buys us automatic failover."

**For showing operational maturity:**
- "I would set up a dashboard tracking QPS, error rate, and p99 latency."
- "We need an on-call runbook for database failover."
- "The deploy strategy is rolling update with canary testing."
- "We should have a grace period on backend shutdown to drain in-flight requests."

**For handling ambiguity:**
- "I have two approaches. One prioritizes simplicity. The other prioritizes scalability. Let me present both."
- "Before I commit to a design, I need to understand the latency requirements better."
- "That depends on the team size. With a small team, I would choose the simpler approach."
- "This is a common pattern. The standard approach is X, but in this case, Y might be better because of Z."

## Red Flags

Interviewers watch for these specific behaviors. Each one signals a weakness that can cost you the offer.

**Red Flag 1: Starting the design without clarifying requirements.**
You begin drawing boxes before asking how many users, what latency target, or what features are needed. This signals inexperience with ambiguous problems.

**Red Flag 2: No trade-off discussion.**
You present your design as the only correct solution. You do not mention what you are giving up. This signals inexperience with real-world engineering decisions.

**Red Flag 3: Single-server design for a system with millions of users.**
You draw one application server and one database for a system with 10 million DAU. This signals you have not thought about scale.

**Red Flag 4: No database schema.**
You say "we use a database" but cannot describe the tables, indexes, or partition strategy. This signals shallow knowledge.

**Red Flag 5: No failure handling.**
You describe the happy path only. You do not discuss what happens when the cache fails, the database is slow, or the network has a partition. This signals that you have not operated systems in production.

**Red Flag 6: Over-engineering for the problem.**
You design a 12-service microservices architecture with Kafka, Kubernetes, and multi-region deployment for a system with 100,000 DAU and a 2-person team. This signals poor judgment.

**Red Flag 7: No concrete numbers.**
You say "many users" instead of "10 million DAU." You say "a lot of storage" instead of "8.4 TB per year." This signals that you do not have the quantitative muscle.

**Red Flag 8: Dogmatic technology choices.**
"I only use NoSQL." "Microservices are always the answer." "Kubernetes is too complex, use EC2." Strong opinions without situational awareness signal inflexibility.

**Red Flag 9: Ignoring the database read-to-write ratio.**
You design a cache but do not calculate the cache hit rate impact. You design read replicas but do not calculate how many are needed. This signals you are guessing rather than calculating.

**Red Flag 10: No mention of monitoring or operations.**
You describe a perfect design on a whiteboard but never mention how you would know it is working in production. This signals that you have not operated production systems.

**Red Flag 11: Ignoring the interviewer's hints.**
The interviewer says "what about the read path?" and you keep talking about the write path. The interviewer says "consider cost" and you add more servers. Listen to every hint. The interviewer is telling you exactly what they want to hear next.

**Red Flag 12: Arguing with the interviewer.**
The interviewer suggests an alternative approach. You tell them their approach is wrong. Senior engineers know multiple approaches are valid. Acknowledge their suggestion, compare it to your approach, and either adopt it or explain why your approach fits better for this specific context.

**Red Flag 13: Not knowing the limits of your own design.**
The interviewer asks "what is the biggest bottleneck in your design?" and you say "I think it will handle the load." Every design has a bottleneck. Identify it. Even if you do not fix it, acknowledging it shows you understand the system's limits.

**Red Flag 14: Going too deep too early.**
You spend 15 minutes on the database schema for a 45-minute interview. The interviewer never gets to see your high-level design, capacity estimation, or trade-off analysis. Pace yourself: 2 minutes on requirements, 3 on estimation, 8 on high-level design, 15 on deep dive, 5 on trade-offs, 2 on wrap-up.

**Red Flag 15: Using buzzwords without substance.**
You say "we will use microservices" but cannot explain how services discover each other, what happens when a service is down, or how you handle distributed transactions. Every buzzword must be backed by implementation knowledge.

**Red Flag 16: Forgetting the user.**
You design a technically beautiful system that fails on the user experience. A video platform with perfect CDN architecture but no search feature. A messaging app with distributed consensus but no contact list. Always connect technical decisions to user-facing features.

## Practice Drills

Each drill is a timed prompt. Set a timer and practice in front of a whiteboard or blank document.

### Drill 1: Design a Chat System (15 minutes)

"Design a real-time chat application that supports 10 million DAU. Users can send direct messages and group messages. Messages must appear within 200ms on the recipient's device. The system must store message history for 90 days."

Silence your phone. Set a 15-minute timer. Follow the six-step framework.

### Drill 2: Design a Photo Sharing Service (15 minutes)

"Design a service where users upload photos and share them with followers. 50 million DAU. Each user uploads 2 photos per day on average. Each photo is viewed by 100 followers on average. Photos are 2 MB on average. Response time for the feed must be under 300ms."

### Drill 3: Design a Rate Limiter (10 minutes)

"Design a rate limiter that limits API calls to 100 requests per second per API key. The system must serve 1 million API keys and handle 100,000 requests per second total. Rate limit decisions must be accurate within 5% error margin."

### Drill 4: Debug a Production Issue (10 minutes)

"Our database CPU has been at 100% every day at 2:00 PM for the last week. It lasts for exactly 8 minutes and then returns to 30%. Nothing is scheduled at 2:00 PM. There were no recent deploys. Walk through your debugging process step by step."

### Drill 5: Defend a Trade-Off (5 minutes)

"We are redesigning our user profile service. Our CTO wants to use DynamoDB. Our senior engineer wants to use PostgreSQL. You are the architect. Pick a side and defend it. Then let the interviewer ask you a counter-argument."

### Drill 6: Design a Distributed Counter (10 minutes)

"Design a system that counts the number of active users on a website in real time. The count must be accurate within 5%. The system handles 100,000 concurrent users updating their status every 30 seconds. The count is displayed on a dashboard that refreshes every 10 seconds."

Focus on: write path (updating the counter from 100,000 concurrent users), read path (reading the counter every 10 seconds), consistency model (5% accuracy tolerance allows eventual consistency), and failure handling.

### Drill 7: Design a Search Autocomplete (15 minutes)

"Design a search autocomplete system for an e-commerce site with 50 million products. When a user types 2 characters, the system must suggest the top 10 completions within 50ms. The system handles 10,000 queries per second. Product names and search frequency change daily."

Focus on: trie data structure, prefix matching, frequency-based ranking, precomputation (build the trie offline from search logs), caching (hot prefixes in Redis), character-by-character incrementals.

### Post-Practice Self-Evaluation

After each drill, evaluate yourself using this checklist:

- Did I ask clarifying questions before starting? (Requirements)
- Did I estimate QPS, storage, and bandwidth? (Capacity)
- Did I draw a component diagram with data flow arrows? (High-level design)
- Did the interviewer ask me to go deeper on any component? (Deep dive)
- Did I mention trade-offs without being prompted? (Trade-offs)
- Did I identify a bottleneck and propose a mitigation? (Wrap-up)

Score yourself on each dimension. Focus your next practice session on the weakest dimension. Most engineers need 10-15 practice sessions to reach FAANG interview readiness. Quality of deliberate practice matters more than quantity — a well-structured 15-minute practice with self-evaluation beats a 60-minute unfocused design session.
