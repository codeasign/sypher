## Interview Questions

OIDC questions appear in system design and infrastructure interviews across all FAANG companies. They test whether you understand identity federation, token-based authentication, delegated authorization, and the security boundaries between systems.

### Five Common Questions

**Q1:** "Design a single sign-on (SSO) system for a company with 10,000 employees using 20 SaaS applications."

```
Requirements:
- One login for all internal and external applications
- Support for SAML and OIDC providers
- Session management across applications
- Security: prevent token replay, support MFA
- Failover: no single point of failure for authentication

Design:
- Deploy an identity provider (IdP) — Keycloak, Okta, or
  Azure AD — as a central authentication gateway.
- Each SaaS app registers as a relying party (RP) with the
  IdP, providing a redirect URI and receiving a client ID.
- User visits App A, is redirected to IdP for login.
- IdP authenticates the user (MFA optional), establishes
  an SSO session (cookie or server-side session), and
  issues an ID token and access token to App A.
- User visits App B — App B detects no session, redirects
  to IdP. IdP sees the SSO session, skips login, and
  issues new tokens silently.
- Tokens are signed (RS256 or ES256) and verified by each
  app using the IdP's JWKS endpoint.
- Refresh tokens for long-lived sessions, rotated on each
  use to prevent replay.
- Deploy the IdP across two availability zones with a
  shared session store (Redis with replication).

Security considerations:
- ID tokens are for the client, not for API authorization.
- Access tokens are opaque to the client and validated
  by the resource server.
- Rotate signing keys regularly and keep previous keys
  available during the transition window.
- Rate-limit token endpoint requests to prevent brute force.
```

**Q2:** "How would you design a token service that supports OIDC for a microservices architecture?"

```
Requirements:
- Centralized authentication service
- Each microservice needs to verify user identity
- Tokens must be short-lived (15 minutes) for security
- Support for service-to-service calls (client credentials)
- Must handle 100K authentication requests per second

Design:
- Authentication service exposes OIDC-compliant endpoints:
  /authorize, /token, /userinfo, /jwks, /.well-known/openid-configuration
- On login, the auth service issues three tokens:
  - ID token (JWT, signed, contains sub, name, email, iss, aud)
  - Access token (JWT or opaque, for API authorization)
  - Refresh token (opaque, stored in Redis with TTL)
- Each microservice caches the JWKS response and validates
  ID tokens locally using the JWT library's verify() method.
- Service-to-service auth uses the client credentials grant:
  Service A requests a token with scope "service:read",
  which the auth service grants based on a pre-shared secret.
- Token revocation: maintain a Redis-based blocklist with
  token jti as key and expiration as TTL.
- Horizontal scaling: the auth service is stateless
  (all signing keys in a shared store like Vault), so it
  scales behind a load balancer.

Token validation flow:
┌──────────┐         ┌──────────────┐         ┌──────────────┐
│  Client  │ ──1──>  │  API Gateway │ ──2──>  │  Auth Service│
│          │         │              │         │              │
│          │ <──4──  │              │ <──3──  │              │
└──────────┘         └──────┬───────┘         └──────────────┘
                            │ 5 (with token)
                            ▼
                    ┌──────────────┐
                    │ Microservice │
                    │              │
                    │ 6. Validates │
                    │    token via │
                    │    cached    │
                    │    JWKS      │
                    └──────────────┘
```

**Q3:** "Design an OIDC-compliant authorization code flow for a mobile app."

```
Requirements:
- Mobile app authenticates users via OIDC
- Must support native app redirects (custom URI schemes)
- PKCE is mandatory to prevent authorization code injection
- Support offline access (refresh tokens)
- Handle token expiration gracefully

Design:
- The app uses the authorization code flow with PKCE:
  1. App generates a code_verifier (random 43-128 char string)
     and code_challenge = BASE64URL(SHA256(code_verifier)).
  2. App opens the system browser or embedded web view to
     the IdP's /authorize endpoint with:
     response_type=code, client_id, redirect_uri, scope=openid,
     code_challenge, code_challenge_method=S256, state.
  3. IdP authenticates the user, prompts for consent, and
     redirects to the app's custom URI scheme
     (myapp://callback?code=abc&state=xyz).
  4. App sends the code and code_verifier to the IdP's
     /token endpoint (server-side, not in the app).
  5. IdP verifies BASE64URL(SHA256(code_verifier)) matches
     code_challenge, then issues ID token, access token,
     and refresh token.
  6. App stores tokens in the secure enclave or keychain.
  7. On 401 from the API, the app uses the refresh token
     silently in the background.

Security considerations:
- PKCE prevents interception attacks even if the
  authorization code is intercepted by a malicious app.
- state parameter prevents CSRF attacks on the redirect.
- Use ASWebAuthenticationSession on iOS and Chrome Custom
  Tabs on Android to stay in the same browser context.
- Never store the client_secret in the mobile app bundle.
  Use a public client (no client_secret) with PKCE.
- Refresh tokens must be rotated on each use.
- The userinfo endpoint should be called only when the
  app needs updated claims, not on every launch.
```

**Q4:** "How does OIDC differ from SAML? When would you choose one over the other?"

```
OIDC (modern, JSON-based):
- Uses JWTs for identity tokens (compact, web-friendly)
- Built on OAuth 2.0 authorization framework
- Simpler federated logout (RP-initiated logout, session
  management via iframes or OP-initiated logout)
- Designed for mobile, SPA, and native applications
- Claims are returned as JSON, easy to consume
- Dynamic client registration is supported
- Discovery via .well-known/openid-configuration URL

SAML 2.0 (legacy, XML-based):
- Uses SAML assertions (XML, large payloads)
- Separate protocol from OAuth — no delegation model
- Complex logout (SAML logout profile with multiple
  redirects between IdP and SPs)
- Designed for enterprise web applications
- Claims (attributes) are XML, require XML parsing
- Static client registration via XML metadata exchange
- Discovery via metadata XML files (IdP metadata, SP metadata)

When to choose OIDC:
- You have mobile apps, SPAs, or native clients
- You need delegated authorization (access tokens for APIs)
- You are building a new system from scratch
- Your ecosystem is cloud-native or API-first
- You need simple session management and logout

When to choose SAML:
- You are integrating with legacy enterprise applications
- Your organization has existing SAML-based federation
- Government or regulated industry mandates SAML
- The application supports only SAML (many legacy tools)
- You need attribute-based access control with complex
  XML attribute profiles

Hybrid approaches:
- Many organizations run both: SAML for legacy enterprise
  apps and OIDC for modern cloud apps and APIs.
- IdPs like Okta, Azure AD, and Keycloak support both
  protocols, brokering between them.
```

**Q5:** "Design a secure logout flow that terminates sessions across all OIDC-connected applications."

```
Requirements:
- User logs out of one application, all sessions terminate
- Must revoke tokens at the IdP level
- Must notify each relying party (RP) to clear local sessions
- Must handle the case where some RPs are offline
- Must prevent replay of revoked tokens

Design:
- Single logout (SLO) in OIDC works through the IdP:
  1. User clicks logout in App A.
  2. App A redirects to the IdP's
     /logout?id_token_hint=...&post_logout_redirect_uri=...
  3. IdP clears the SSO session and records all RPs
     that have active sessions for this user.
  4. IdP redirects the user's browser to each RP's logout
     endpoint (front-channel logout via iframes or
     back-channel logout via direct HTTP POST).
  5. Each RP clears its local session and marks tokens
     as revoked in its local cache.
  6. After all RPs acknowledge, the IdP redirects the user
     to the post_logout_redirect_uri.

Token revocation:
- Call the /revoke endpoint on the IdP to invalidate
  refresh tokens immediately.
- For access tokens, set short TTLs (15 minutes) so they
  expire naturally. For immediate revocation, maintain a
  token blocklist checked by the API gateway.

SLO flow diagram:

│ User │  │ App A│  │ IdP  │  │ Other RPs│
└──┬───┘  └──┬───┘  └──┬───┘  └─────┬────┘
   │         │         │            │
   │  1. Logout click  │            │
   │────────>│         │            │
   │         │ 2. SLO  │            │
   │         │ request │            │
   │         │────────>│            │
   │         │         │ 3. Clear   │
   │         │         │ SSO session│
   │         │         │     │      │
   │         │         │ 4. Notify  │
   │         │         │────────────│
   │         │         │     │      │
   │         │         │ 5. Confirm │
   │         │ <───────│────────────│
   │         │ 6. Done │            │
   │ <───────│─────────│            │
   │         │         │            │
```

### Three Advanced Questions

**Q6:** "Design a multi-tenant OIDC provider for a SaaS platform serving 1,000 customers."

```
Requirements:
- Each tenant has its own user directory
- Tenants are isolated (no data leakage across tenants)
- Each tenant configures custom OIDC clients and scopes
- Support custom domains per tenant (tenant.auth.company.com)
- Handle 10 million authentication requests per day

Design:
- Tenant resolution: determine the tenant from the request.
  Options: subdomain (tenant.auth.co.com), custom domain
  (CNAME to shared infrastructure), or path (/auth/tenant/).
- Each tenant has an isolated configuration:
  - Tenant-specific signing keys (RS256 key pair per tenant)
  - Tenant-specific client registrations
  - Tenant-specific claims mapping and scopes
  - Tenant-specific user store (separate DB or schema)
- The /authorize endpoint resolves the tenant from the
  request, loads tenant configuration, and renders the
  tenant's branded login page.
- The /token endpoint signs tokens with the tenant-specific
  key. The JWKS endpoint returns the tenant's public key
  (e.g., /.well-known/jwks.json on the tenant's domain).
- The .well-known/openid-configuration endpoint returns
  tenant-specific endpoints (issuer claim must match
  the tenant's issuer URL).

Data model:
tenants:
  id, name, domain, custom_domain, signing_key_id,
  issuer_url, created_at

clients:
  id, tenant_id, client_id, client_secret_hash,
  redirect_uris[], grant_types[], response_types[],
  token_endpoint_auth_method, scopes[]

users:
  id, tenant_id, email, password_hash, mfa_enabled,
  attributes (JSONB), created_at, updated_at

Scaling:
- Tenant resolution is a fast path — cache tenant configs
  in Redis with a 5-minute TTL.
- Each tenant's signing key is rotated every 90 days with
  a 30-day overlap (previous key still available for
  token validation during the transition).
- Tenant isolation at the database level: use schema-per-tenant
  for PostgreSQL or separate DynamoDB tables per tenant.
- Rate limiting per tenant, not globally (one tenant's
  traffic spike should not affect others).
```

**Q7:** "Design a distributed OIDC session management system that survives IdP node failures."

```
Requirements:
- User sessions survive IdP node crashes
- Sessions are replicated across data centers
- Session revocation is immediate (under 1 second)
- Support 10 million concurrent sessions
- Session time-to-live: 8 hours with sliding window

Design:
- Session store: Redis cluster with replication across
  three availability zones.
- Session data: { sub, sid, auth_time, amr, acr, clients[]
  (list of RP client IDs with active sessions), expires_at }
- On authentication, the IdP writes the session to Redis
  with a TTL of 8 hours, extended by 1 hour on each
  active token refresh.
- On each /token request, the IdP reads the session from
  Redis, extends the TTL, and issues a new token.
- On logout, the IdP deletes the session from Redis and
  publishes a "session:revoked" event to a Redis Pub/Sub
  channel for cross-datacenter propagation.
- Session replication across data centers:
  - Primary datacenter: Redis Cluster (3 nodes, 3 replicas)
  - Secondary datacenter: Redis sentinel or CRDT-based
    replication (Redis Enterprise Active-Active)
  - Cross-datacenter latency: 50-100ms RTT
  - Sessions are eventually consistent across DCs with
    < 200ms convergence window.

Failure modes:
- IdP node fails: other nodes pick up traffic, session
  data is in Redis (persistent, replicated).
- Redis primary fails: sentinel promotes replica in < 10s.
  IdP falls back to local cache during promotion.
- Network partition: IdP nodes without Redis access reject
  authentication requests (fail-closed) but serve cached
  JWKS responses for token validation (read-only).
- Redis cluster full: evict least-recently-used sessions.
  IdP monitors eviction rate and alerts at 80% capacity.

Session recovery:
- If a user's session is evicted from Redis, the next
  token refresh triggers a re-authentication. The IdP
  redirects to the login page, and the user re-authenticates
  (with SSO cookie if the browser still has one).
```

**Q8:** "Design a token exchange and impersonation system for an enterprise OIDC deployment."

```
Requirements:
- Admin users can impersonate regular users for support
- Service accounts can exchange tokens for different scopes
- Token exchange must be audited (who impersonated whom, when)
- Impersonation must be time-limited (max 1 hour)
- The audit trail is immutable and queryable

Design:
- Use the OAuth 2.0 token exchange grant (RFC 8693):
  POST /token
  grant_type=urn:ietf:params:oauth:grant-type:token-exchange
  subject_token=... (admin's access token)
  subject_token_type=urn:ietf:params:oauth:token-type:access_token
  actor_token=... (admin's ID token, proving identity)
  actor_token_type=urn:ietf:params:oauth:token-type:id_token
  requested_token_type=urn:ietf:params:oauth:token-type:access_token
  resource=https://api.example.com
  audience=target_client_id

- The IdP validates both tokens, checks impersonation
  permissions, and issues a new token with:
  - sub: the target user (the impersonated user)
  - act: { sub: admin@company.com, name: "Alice Admin" }
  - scope: restricted to the impersonation scope set
  - exp: 1 hour (max allowed)

- The target API validates the token, sees both the
  subject (target user) and the actor (admin), and logs
  the access accordingly.

- Audit log entry:
  {
    "event": "token_exchange",
    "actor": "admin@company.com",
    "actor_ip": "203.0.113.42",
    "target": "user@company.com",
    "scope": "openid profile email admin:impersonate",
    "client_id": "support-tool",
    "timestamp": "2026-07-02T14:30:00Z",
    "token_jti": "abc123",
    "expires_at": "2026-07-02T15:30:00Z"
  }

- Audit log storage: write-ahead log in a database
  (append-only table with RLS, no UPDATE or DELETE).
  Archive to S3 after 90 days with Athena query capability.

Security considerations:
- Impersonation is a high-risk operation. Require MFA
  for the admin session before allowing token exchange.
- The impersonation scope must be explicitly granted
  to the admin's client, not inherited from the user's scopes.
- Rate-limit impersonation token exchanges: max 10 per
  admin per hour.
- Notify the target user via email when impersonation
  occurs ("Your account was accessed by admin@company.com
  on 2026-07-02 at 14:30 UTC").
```

## Signals Interviewers Look For

### Strong Hire Signals

- Describes the OIDC flow in detail: authorization code, token exchange, ID token validation, userinfo call, refresh token rotation.
- Distinguishes between ID tokens (for the client, identity) and access tokens (for the resource server, authorization).
- Explains PKCE, state parameter, and nonce unprompted, describing exactly what each protects against.
- Brings up JWKS endpoint caching, key rotation, and the `kid` header in JWT validation.
- Quantifies token lifetimes: 15 minutes for access tokens, 7 days for refresh tokens, 8 hours for SSO sessions.
- Discusses real-world IdP deployments (Keycloak, Okta, Azure AD, Auth0) and their tradeoffs.
- Mentions specific OIDC specifications: RFC 6749 (OAuth 2.0), RFC 7636 (PKCE), RFC 7519 (JWT), OpenID Connect Core 1.0.
- Describes the security model: what an attacker can do with each token type and how to mitigate.
- Compares OIDC with SAML, OAuth 2.0, and social login, explaining when each is appropriate.
- Talks about federated logout, RP-initiated logout, and session management across applications.

### Strong Hire Behaviors

What a strong hire does in the first 90 seconds of Q1:

```
"Before I design the SSO system, I need to understand
the requirements. How many applications need to be
integrated? Are these internal or external applications?
What authentication methods are required — password,
MFA, social login, federated identity?"

"Let me break this down into three layers: the
authentication layer (IdP, user store, MFA), the
session layer (SSO session, RP sessions, token
management), and the integration layer (OIDC clients,
redirect URIs, token validation)."

"At the core, I would deploy a standards-compliant
IdP like Keycloak. Each application registers as a
confidential client with a client secret and a
redirect URI. The IdP exposes the standard OIDC
endpoints — /authorize, /token, /userinfo, /jwks,
and the discovery document at /.well-known."

"On the first login, the user is redirected to the
IdP, authenticates, and grants consent. The IdP
issues an authorization code, which the application
exchanges for tokens. The ID token is a JWT signed
with RS256. The application validates it by fetching
the IdP's JWKS, verifying the signature, checking
the issuer and audience claims, and ensuring the
token has not expired."

"On the second application, the user is redirected
to the IdP again. The IdP detects the existing SSO
session (a signed cookie), skips the login prompt,
and issues new tokens silently. This is the
single sign-on experience."
```

### Neutral Signals

- Knows the OIDC flow at a high level but cannot describe token validation steps.
- Mentions JWTs but cannot explain the difference between signed and encrypted tokens.
- Can name OIDC endpoints but cannot describe the request parameters for each.
- Mentions PKCE but cannot explain why it is needed for mobile apps.
- Describes the authorization code flow but cannot articulate the security properties of each step.
- Knows OIDC exists but cannot compare it to SAML or OAuth 2.0.
- Uses "JWT" and "token" interchangeably without distinguishing ID tokens from access tokens.

### Weak Signals

- Cannot explain the difference between authentication (OIDC) and authorization (OAuth 2.0).
- Does not know what PKCE is or why it is required for mobile apps.
- Cannot describe how to validate a JWT (signature, issuer, audience, expiration).
- Proposes storing tokens in localStorage without discussing XSS risks.
- Does not mention the `nonce` parameter or its role in preventing replay attacks.
- Claims JWTs do not need signature verification because they are encoded, not encrypted.
- Cannot name a single production OIDC provider or describe its features.
- Does not know what the `sub` claim represents or why it is required.

## Knowledge Levels

### Level 1: Awareness

You know that OIDC is an authentication protocol built on OAuth 2.0. You understand that it provides identity information in the form of an ID token, typically a JWT. You can name the three parties: end-user, relying party, and identity provider. You know that OIDC enables single sign-on.

### Level 2: Application

You can implement the authorization code flow in a web application. You configure OIDC clients with correct redirect URIs, scopes, and response types. You validate ID tokens: verify the signature against the JWKS endpoint, check the `iss` and `aud` claims, and enforce token expiration. You distinguish between ID tokens (identity, for the client) and access tokens (authorization, for APIs). You can set up PKCE for mobile and SPA clients. You understand the difference between implicit flow, authorization code flow, and hybrid flow.

### Level 3: Mechanism

You understand the cryptographic details: how RS256, ES256, and HS256 signatures work, and why asymmetric signing (RS256, ES256) is preferred over symmetric (HS256) for multi-party systems. You can describe the JWKS rotation process: the IdP publishes a new key pair, includes the new `kid` in the JWT header, and the RP fetches the updated JWKS and caches it. You understand what each claim in an ID token means: `sub`, `iss`, `aud`, `exp`, `iat`, `auth_time`, `nonce`, `acr`, `amr`, `at_hash`, `c_hash`. You can describe the token exchange flow (RFC 8693) and the back-channel logout specification. You understand how OIDC session management works with the `session_state` parameter and postMessage-based iframe communication.

### Level 4: Production

You have deployed and operated an OIDC IdP in production. You have debugged token validation failures, clock skew issues, and JWKS caching problems. You have implemented token revocation with a blocklist, refresh token rotation, and comprehensive token introspection. You understand the performance characteristics of JWT verification at scale (10,000+ verifications per second) and have tuned the JWKS cache TTL to balance fresh keys against endpoint load. You have configured key rotation policies, supervised token migrations, and handled the transition window where old and new keys coexist. You understand how to set up federated logout across multiple applications and have debugged SLO failures caused by iframe blocking or third-party cookie restrictions.

### Level 5: Design

You can design a custom OIDC-compliant IdP from scratch, including the token issuance pipeline, session management, key rotation, and tenant isolation. You understand the security implications of each OIDC extension and can decide when to use them: pairwise pseudonymous identifiers, claims aggregation, distributed claims, and self-issued providers. You can design a token exchange and delegation system for complex enterprise scenarios. You understand how OIDC interacts with OAuth 2.0 resource indicators, token introspection, and token revocation. You can contribute to the OpenID Foundation specifications or implement a new OIDC profile for a novel use case (IoT, decentralized identity, verifiable credentials).

**Trust model and architecture:**

4. "I will use pairwise pseudonymous identifiers (`sub` claim) instead of the user's email or username as the subject. This prevents correlation across different RPs by the IdP — each RP sees a different subject value for the same user."

**Flow selection:**

5. "For a mobile app, PKCE is mandatory. The code challenge binds the authorization code to the specific client instance. Without PKCE, a malicious app that intercepts the authorization code can exchange it for tokens."

8. "The `nonce` parameter in the authentication request is included in the ID token as a claim. The RP validates that the nonce in the token matches the one it sent, preventing replay attacks where an attacker captures an ID token and reuses it."

**Token lifecycle:**

9. "Access tokens should be short-lived — 15 minutes is the standard. Refresh tokens should be rotating: each time the client uses a refresh token, the IdP issues a new refresh token and invalidates the old one. This prevents token replay if a refresh token is compromised."

10. "The token introspection endpoint allows the resource server to check the active state of an access token, including its scopes, expiration, and the client it was issued to. This is necessary for opaque tokens that cannot be validated locally."

**Security and validation:**

18. ## Red Flags

These statements reveal gaps in your understanding. Avoid them.

**Red Flag 1:** "JWT is encrypted, so it is secure."

Wrong. JWTs are signed, not encrypted by default. The payload is base64url-encoded, which is trivially decoded. Anyone with the token can read the claims. The signature proves the token was issued by the IdP and has not been tampered with, but it does not hide the content. If you need confidentiality, use JWE (JSON Web Encryption) — but this is rare because ID tokens are typically transmitted over TLS. Say "JWT is signed with RS256, which provides integrity and authenticity but not confidentiality. The payload is readable by anyone who possesses the token."

**Red Flag 2:** "I will use the implicit flow because it is simpler."

The implicit flow was deprecated in the OAuth 2.0 Security Best Current Practice (BCP) in 2019. It exposes the access token in the URL fragment, making it visible in the browser history, referrer headers, and any JavaScript on the page. The authorization code flow with PKCE achieves the same goal with better security. Say "I use the authorization code flow with PKCE. The implicit flow is deprecated because it exposes tokens in the URL fragment."

**Red Flag 3:** "I will store the access token in localStorage for the SPA."

localStorage is accessible to any JavaScript running on the same origin, making it vulnerable to XSS attacks. If an attacker injects a script, they can read all tokens from localStorage. Cookies (with `HttpOnly`, `Secure`, and `SameSite=Strict`) are safer because they are not accessible to JavaScript. Use the BFF (Backend for Frontend) pattern where the SPA never sees the tokens directly. Say "I store tokens in an `HttpOnly` cookie set by the backend, or use the BFF pattern where the SPA communicates with a backend proxy that manages the tokens."

**Red Flag 4:** "I will use the same client secret for all environments."

Client secrets should be unique per environment (development, staging, production) and per client instance. A leaked development secret should not compromise production. Use environment-specific client registrations in the IdP. Say "Each environment has its own client registration with a unique secret. The production secret is stored in a secrets manager, never in the code repository."

**Red Flag 5:** "I do not need to verify the token signature because the token comes from the IdP over HTTPS."

HTTPS protects the token in transit, but it does not protect against a compromised or malicious client that replays a token. The signature is the only mechanism the RP has to verify that the token was issued by the trusted IdP and has not been modified. Say "I always verify the JWT signature against the IdP's JWKS endpoint. HTTPS protects the transport, but the signature proves the origin and integrity of the token."

**Red Flag 6:** "I will use the `sub` claim as the user's email address."

The `sub` claim is a unique identifier for the user at the IdP, but it is not required to be an email address. Using an email as `sub` creates a privacy issue (the email is exposed to every RP) and a migration problem (if the user changes their email). The `email` claim should be used for the email address, and `sub` should be a persistent, opaque identifier. Say "I use a persistent, opaque UUID as the `sub` claim. The email is available in the `email` claim but is not the subject identifier."

**Red Flag 7:** "All tokens have the same lifetime."

Different token types have different security requirements. Access tokens are short-lived (15 minutes) because they are sent with every API request and are more exposed. Refresh tokens are longer-lived (7-30 days) but are used infrequently and stored more securely. ID tokens are short-lived (15 minutes) because they contain identity information. SSO sessions are longer (8-24 hours) with a sliding window. Say "I set different TTLs: access tokens at 15 minutes, ID tokens at 15 minutes, refresh tokens at 7 days with rotation, and SSO sessions at 8 hours with a sliding window."

**Red Flag 8:** "Logout just clears the local session."

Logout in an OIDC ecosystem must clear sessions at the IdP and all RPs. Clearing the local session in App A leaves the user's SSO session intact at the IdP, and the user can still access App B without re-authenticating. This is a security issue if the user is on a shared computer. Say "Single logout sends the user to the IdP's logout endpoint, which clears the SSO session and notifies all RPs to clear their local sessions through front-channel or back-channel logout."

**Red Flag 9:** "I do not need key rotation because the current key is not compromised."

Key rotation is a security best practice even without a known compromise. Regular rotation limits the impact of an undetected key compromise and reduces the window of vulnerability. The industry standard is to rotate signing keys every 90 days with a 30-day overlap period where both the old and new keys are available for verification. Say "I rotate signing keys every 90 days. The previous key remains available for token validation during a 30-day overlap, and the JWKS endpoint returns both keys. The `kid` header in the JWT tells the RP which key to use for verification."

**Red Flag 10:** "I will use the same OIDC configuration for all clients."

Different clients have different requirements. A confidential web app with a backend can securely store a client secret and use the authorization code flow. A mobile app cannot store a secret and must use PKCE. A machine-to-machine service uses the client credentials grant. Say "I configure OIDC clients according to their type: confidential clients use the authorization code flow with a client secret, public clients use PKCE, and machine clients use the client credentials grant."

### Drill 2: The Token Validator

Given a JWT and its configuration, walk through the validation steps.

**Example JWT header:**

```json
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "key-2026-01"
}
```

**Example JWT payload:**

```json
{
  "iss": "https://auth.example.com",
  "sub": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "aud": ["client-app-123", "api.example.com"],
  "exp": 1893456000,
  "iat": 1713456000,
  "auth_time": 1713455900,
  "nonce": "random-nonce-value",
  "at_hash": "base64url-hash",
  "amr": ["pwd", "otp"],
  "acr": "urn:mace:incommon:iap:silver"
}
```

**Validation steps:**

```
1. Check algorithm: alg is RS256 (asymmetric, acceptable).
   Reject if alg is "none" or HS256 (symmetric, which
   could be used for key confusion attacks).
2. Check typ: header typ is "JWT" (prevents type confusion
   with JWTs used for other purposes).
3. Fetch JWKS from https://auth.example.com/.well-known/jwks.json.
   Find key with kid "key-2026-01". Verify signature.
4. Check iss: matches the expected IdP issuer URL exactly.
5. Check aud: includes our client ID "client-app-123".
   If aud is an array, at least one value must match.
6. Check exp: current time (with 5-minute clock skew
   tolerance) must be before the expiration timestamp.
7. Verify nonce: matches the value sent in the
   authentication request (stored in the session).
8. Verify at_hash: compute SHA-256 of the access token,
   take the left half, base64url-encode, and compare.
9. Use amr and acr: if the operation requires MFA, check
   that amr includes "otp" or "webauthn" and acr meets
   the minimum assurance level.
```

**Practice with:** Generate three JWTs with different configurations and validate each one. Include one with an invalid signature, one with a mismatched audience, and one with an expired timestamp.

**Attack scenarios to diagnose:**

```
Attack 1: Authorization code interception
  - Malicious app on the same device intercepts the
    authorization code from the redirect URI.
  - Without PKCE, the attacker can exchange the code
    for tokens.
  - Mitigation: PKCE (code_challenge and code_verifier).
    The attacker cannot compute the code_verifier from
    the code_challenge (preimage resistance of SHA-256).

Attack 2: CSRF on the redirect URI
  - Attacker crafts a malicious link that initiates the
    OIDC flow and then redirects to the RP's callback
    URL with a stolen authorization code.
  - The RP associates the authenticated session with
    the attacker's account.
  - Mitigation: state parameter with a cryptographically
    random value stored in the user's session. The RP
    verifies the state matches on callback.

Attack 3: Token theft via XSS
  - Attacker injects a script that reads tokens from
    localStorage or sessionStorage.
  - The attacker exfiltrates the tokens and uses them
    to call the API.
  - Mitigation: store tokens in HttpOnly cookies
    (inaccessible to JavaScript). Use the BFF pattern
    where the SPA never sees the tokens.

Attack 4: Refresh token replay
  - Attacker steals a refresh token.
  - The attacker can obtain new access tokens
    indefinitely until the refresh token expires.
  - Mitigation: refresh token rotation. Each time the
    refresh token is used, the IdP issues a new one
    and invalidates the old one. If the old one is
    reused, the IdP detects the replay and revokes
    all tokens for that client.

Attack 5: Mix-up attack
  - Attacker registers a malicious IdP and tricks the
    user into authenticating with it.
  - The RP receives a token from the malicious IdP
    but trusts it because it validates the signature
    (with the malicious IdP's key).
  - Mitigation: the RP stores the expected IdP issuer
    URL in its configuration and validates the `iss`
    claim against it. The RP only accepts tokens from
    the configured IdP, not from any IdP whose JWKS
    it can fetch.
```

**Practice with:** For each attack, trace the full exploit path and propose at least two defense layers. Identify which defenses are mandatory (PKCE, state, HttpOnly) and which are defense-in-depth (nonce, refresh rotation, BFF).

### Drill 4: The Logout Tracer

Given a multi-application OIDC deployment, trace the full logout flow and identify failure points.

**Deployment:**

```
- IdP: auth.company.com (Keycloak)
- App A: app.company.com (React SPA, OIDC client)
- App B: crm.company.com (Angular SPA, OIDC client)
- App C: legacy.company.com (SAML-based, not OIDC)
- App D: partner.company.com (third-party, OIDC client)
```

**Logout flow trace:**

```
1. User clicks "Logout" in App A.
2. App A sends a POST to its own /logout endpoint.
3. App A's backend clears the local session cookie.
4. App A redirects the user to the IdP's /logout endpoint:
   GET https://auth.company.com/logout?
     id_token_hint=...&
     post_logout_redirect_uri=https://app.company.com/logged-out&
     state=xyz
5. IdP receives the logout request. Validates the
   id_token_hint (decodes the ID token, verifies it
   was issued by this IdP, checks the `aud` matches
   App A's client ID).
6. IdP clears the SSO session. Looks up all RPs that
   have active sessions with this user.
7. IdP constructs the front-channel logout page with
   iframes for each RP:
   <iframe src="https://app.company.com/logout?sid=...">
   <iframe src="https://crm.company.com/logout?sid=...">
   <iframe src="https://partner.company.com/logout?sid=...">
8. Each iframe loads the RP's logout endpoint. The RP
   clears the local session for the given session ID.
9. App A's iframe fails to load (third-party cookies
   blocked by the browser). This is the FIRST failure
   point — App A's session is already cleared (step 3),
   but the IdP does not get confirmation.
10. App C uses SAML — the IdP cannot notify it via
    OIDC front-channel logout. This is the SECOND
    failure point — App C's session remains active
    at the SAML SP.
11. After all iframes have loaded (or timed out), the
    IdP redirects the user to the post_logout_redirect_uri
    with the state parameter.
12. App A validates the state parameter and shows the
    "You are logged out" page.

Failure points and mitigations:
- Third-party cookie blocking: use back-channel logout
  (HTTP POST from the IdP to the RP's logout endpoint,
  not dependent on the browser). The IdP sends the
  logout request directly from its server.
- SAML integration: the IdP must also execute the SAML
  SLO profile for SAML-based applications. This is a
  separate logout flow that the IdP initiates after the
  OIDC logout.
- Offline RPs: if App D is unreachable, the IdP should
  log the failed logout attempt and retry with a
  backoff strategy (exponential backoff, max 3 retries).
- User notification: the user should be told which apps
  were successfully logged out and which were not.
```

### Drill 5: The Key Rotation Planner

Given a production OIDC deployment, plan a key rotation that minimizes disruption.

**Current state:**

```
- IdP: Keycloak, deployed in production for 6 months
- Current signing key: RSA-2048, kid="key-2025-01"
- 50 RPs, each caches JWKS with 1-hour TTL
- 10 million active tokens issued with the current key
- Some tokens have a 30-day lifetime (refresh tokens)
- Some RPs have bugs: they do not handle multiple keys
  in the JWKS response
```

**Rotation plan:**

```
Phase 1: Preparation (Week 1)
- Generate a new RSA-2048 key pair with kid="key-2026-01".
- Add the new public key to the JWKS endpoint alongside
  the existing key. The JWKS response now contains:
  {
    "keys": [
      {"kty": "RSA", "kid": "key-2025-01", "use": "sig", ...},
      {"kty": "RSA", "kid": "key-2026-01", "use": "sig", ...}
    ]
  }
- The IdP continues signing with the old key (key-2025-01).
- Monitor: no RP should break because of the additional
  key in the JWKS response. This is the "coexistence test."

Phase 2: Transition (Week 2)
- Switch the IdP to sign new tokens with the new key.
  The JWT header now has kid="key-2026-01".
- Old tokens (signed with key-2025-01) are still valid
  until they expire.
- Both keys remain in the JWKS response.
- Monitor: RP token validation errors. If an RP has a
  bug that prevents it from using the new key, we can
  add that RP's client ID to an allowlist that continues
  receiving tokens signed with the old key.

Phase 3: Deprecation (Week 3+)
- Wait for all tokens signed with key-2025-01 to expire.
  For 15-minute access tokens, this takes 15 minutes.
  For 30-day refresh tokens, this takes 30 days.
- Remove key-2025-01 from the JWKS response.
- Monitor: no RP should be affected because no valid
  token references the old key.
- Archive the old key pair for audit purposes.

Handling buggy RPs:
- Some RPs do not handle multiple keys in JWKS.
  They use the first key in the array, regardless of kid.
  - Mitigation: the IdP can order keys so the new key
    is first (most RPs pick the first key).
  - Alternatively, serve different JWKS responses to
    different RPs based on the client_id (Keycloak
    supports this with its "key provider" SPI).
  - For RPs that hardcode the key, coordinate a
    maintenance window and update the key manually.

Rollback plan:
- If a critical RP breaks during the transition:
  1. Switch the IdP back to signing with the old key.
  2. The new key remains in the JWKS response for
     any RPs that already fetched it.
  3. Debug the RP's issue and re-attempt rotation
     after the fix is deployed.
```

**Practice with these scenarios:**

```
Scenario A: You must rotate keys immediately because
the current key is suspected compromised. How does the
plan change? (Hint: you cannot wait for token expiration.)

Scenario B: You are migrating from RS256 to ES256
(different algorithm, different key type). What
additional considerations apply?

Scenario C: Your IdP is multi-tenant with 1,000 tenants.
Each tenant has its own signing key. How do you
coordinate rotation across all tenants?

Scenario D: Some RPs do not fetch the JWKS endpoint
dynamically. They have the public key hardcoded in
their configuration. How do you handle these RPs
during a rotation?
```

## Key Takeaways

- Every system design interview that touches authentication, identity, or security will include OIDC questions. Prepare with concrete flow details, token validation steps, and security mitigations.
- The four-step framework (trust model, flow, token lifecycle, validation) structures clean answers that cover what the interviewer needs to hear.
- Use specific phrases that signal depth: PKCE, state parameter, nonce, JWKS caching, at_hash, refresh token rotation, pairwise sub, SLO, token binding, step-up authentication.
- Avoid 10 common red flags: claiming JWT is encrypted, using implicit flow, storing tokens in localStorage, skipping signature verification, uniform token lifetimes, ignoring key rotation, inadequate logout, using email as sub, same config for all clients, and ignoring PKCE for mobile apps.
- Run 5 drills until they become automatic: flow selector, token validator, attack simulator, logout tracer, key rotation planner.
- Know at least three real-world OIDC providers and their strengths: Keycloak (open-source, self-hosted, full-featured), Okta / Azure AD (enterprise, high availability, extensive integrations), and Auth0 (developer-friendly, extensible, good for SaaS).
- Know the OIDC specification details: the standard endpoints, the request parameters for each, the claims in the ID token, and the security extensions (PKCE, token binding, DPoP).
- A strong hire quantifies everything: token lifetimes in minutes and days, key rotation in days, JWKS cache TTL in hours, and clock skew tolerance in minutes.
- Continue to the case studies section to apply these patterns to real-world OIDC deployments in detail.
