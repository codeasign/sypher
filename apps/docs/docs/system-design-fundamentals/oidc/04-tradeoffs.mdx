# OIDC -- Trade-offs & Failure Modes

Every authentication protocol makes trade-offs. OIDC is no exception. You chose it because you need a modern, token-based identity layer for your web and mobile applications. But choosing OIDC over alternatives -- or choosing it at all -- comes with a set of engineering tensions, failure modes, and anti-patterns you must understand before you put it in production.

This page covers the hard trade-offs you face when building with OIDC. You will see where OIDC shines, where it breaks, and when you should put it back on the shelf and reach for something else. By the end, you should have a mental model for making the right call in every architecture discussion you join.

The structure is straightforward. First, you examine the fundamental trade-off between OIDC and its predecessor, SAML 2.0 -- what you gain and what you surrender. Then you walk through a structured decision matrix that scores each protocol across the dimensions that matter in real systems. Next comes the CAP analysis of OIDC providers, followed by the specific failure modes you will encounter and the anti-patterns you must avoid. Finally, you cover the situations where OIDC is simply the wrong tool, and you close with a bridge to the practical configuration on the next page.

## The Core Trade-off: OIDC vs SAML

The biggest architectural decision you will make in enterprise identity is choosing between OIDC and SAML. OIDC trades away decades of enterprise-niche maturity for the simplicity and performance that modern applications demand. This is not a subtle trade. It is fundamental, and it affects every layer of your system.

SAML 2.0 was built for a world of SOAP, XML, and enterprise service buses. OIDC was built for a world of REST, JSON, and single-page applications. These are fundamentally different design philosophies, and the choice ripples through your system architecture, your team's skill requirements, your deployment pipeline, and your operational runbook.

### Token Format

OIDC uses JSON Web Tokens (JWTs). A JWT is a compact, URL-safe token that carries claims in JSON. It is easy to parse in any language. It is small enough to fit in a URL query parameter or an HTTP header without fragmentation. SAML assertions are XML documents. An XML Signature envelops the assertion in layers of namespaces, transforms, and canonicalization rules that require a full DOM parser to extract a single attribute.

The practical effect: an OIDC ID token is typically 1-2 KB. A SAML assertion is typically 10-50 KB. The parsing cost scales with the size. On a high-traffic login endpoint, this difference alone can save you hundreds of milliseconds per request.

### Transport and Integration Surface

OIDC speaks REST and JSON. You integrate with it the same way you integrate with any modern API: fetch the discovery document, call the authorization endpoint, call the token endpoint, validate the response. The entire specification is a handful of HTTP requests with well-defined request and response schemas.

SAML 2.0 speaks SOAP and HTTP-Redirect with XML signatures. The integration surface includes WS-* standards (WS-Trust, WS-Federation, WS-Security) that require specialized middleware. Debugging a SAML integration means decoding base64-encoded XML embedded in URL parameters, verifying XML signatures against X.509 certificates, and tracing SOAP faults through enterprise service bus transformations.

### Mobile and Native Support

OIDC was designed with mobile in mind from day one. The authorization code flow with PKCE is the recommended approach for native applications. AppAuth and Android Custom Tabs provide standardized, secure implementations. SAML 2.0 has no native mobile specification. Every mobile SAML integration is a custom, brittle implementation that typically resorts to embedding a WebView -- which means the user sees the full enterprise IdP login page in a cramped browser window with no native UI integration.

You get simplicity and performance from OIDC, but you lose the deep audit-trail capabilities, the decades of enterprise IdP tooling, and the XML signature infrastructure that large regulated organizations have built around SAML. For most modern applications, this is a good trade. For a bank integrating with a government identity federation, it may not be.

The key insight: OIDC is not strictly better than SAML. It is better for the set of problems that modern web and mobile applications face. If your problem set looks like "federate identity across a few thousand enterprise employees who all use the same corporate desktop," SAML may still serve you well.

### Walkthrough of Each Row

**Mobile Support (OIDC: 5, SAML: 1).** This is the decisive row for most modern applications. If your users access your system from phones, OIDC is the only serious choice. SAML's mobile story is essentially absent. The AppAuth SDK for OIDC gives you a standardized, secure mobile authentication flow out of the box. SAML on mobile means embedding a WebView, handling redirects manually, and dealing with cookie-sharing issues between the WebView and the native app. Do not attempt a mobile SAML deployment unless you have a dedicated identity team and a lot of patience.

**Performance (OIDC: 5, SAML: 2).** OIDC's JWT is compact. A typical ID token fits in a single HTTP header. SAML assertions are large XML documents that require a full XML parser to consume. The performance gap grows with the number of attributes in the assertion. A SAML assertion with 20 attributes can be 30-50 KB. The parsing cost, the transport cost, and the signature verification cost all scale with that size.

**Developer Experience (OIDC: 5, SAML: 2).** OIDC libraries exist for every major language and framework. They are well-documented, actively maintained, and follow modern API design patterns. SAML libraries exist, but they are typically wrappers around XML signature verification with complex configuration. Debugging a SAML integration often requires packet captures and XML signature validation tools. Debugging an OIDC integration requires reading a JSON response.

**Enterprise Compliance (OIDC: 3, SAML: 5).** SAML has been deployed in regulated enterprises for two decades. The audit trails, the compliance frameworks, and the governance tooling around SAML are mature. OIDC is newer and has fewer enterprise compliance certifications out of the box. If your compliance auditors require per-assertion audit logs with X.509 certificate chains, SAML is the safer choice.

**Audit Capability (OIDC: 3, SAML: 5).** SAML assertions carry rich XML attributes: authentication context, session index, not-before/not-on-or-after timing, and custom attributes with namespaced URIs. OIDC claims are simpler. You can add custom claims to an OIDC ID token, but the tooling for auditing those claims is less mature.

**Migration Cost (OIDC: 1, SAML: 5).** If you already have SAML deployed across your organization, switching to OIDC means replacing your entire identity infrastructure. The cost of migration includes new SDKs, new configuration, new monitoring, new runbooks, and retraining. SAML costs you nothing because it is already there.

The matrix makes the pattern clear. OIDC dominates on every dimension that matters to a modern web or mobile application. SAML dominates on the dimensions that matter to a regulated enterprise with existing infrastructure. Your decision comes down to which world you operate in.

## CAP Theorem and OIDC Providers

The OIDC provider in your architecture is a system of its own. Like any distributed system, it obeys the CAP theorem. Understanding where your provider sits on the CAP triangle tells you what breaks and when, and it shapes your resilience strategy.

This means your OIDC provider is a single point of failure for authentication. When the provider goes down or gets partitioned, no one can authenticate. There is no "eventual consistency" safety valve -- if the provider cannot verify that it is authoritative, it refuses to sign tokens.

### Why CP Is the Right Choice for Identity

You might wonder: why do OIDC providers choose CP instead of AP? Consider what happens if the provider chose availability over consistency. A partitioned replica would continue issuing tokens with a different signing key. Those tokens would be valid within the partition but invalid in the rest of the system. Users would authenticate in the partition, then be unable to access resources outside it. Worse, if the partition healed, the replica would have issued tokens that the primary does not recognize. Reconciling those tokens would require invalidating a batch of users or accepting tokens signed with a stale key. Both options are worse than simply refusing to issue tokens during the partition.

### Mitigation Strategies

You cannot change the provider's CAP orientation, but you can build your system to survive provider outages.

**Multi-region replication.** Configure your IdP for active-active multi-region deployment. This is available in most enterprise IdP products (Azure AD, Okta, Keycloak with proper clustering). Each region operates independently during a partition. The cost is higher infrastructure complexity and the need to handle cross-region token validation.

**Local token caching.** Cache validated ID tokens and their public keys on your resource servers. A token that was validated before the outage remains valid in cache until it expires. This keeps existing sessions alive even when the provider is unreachable. Set your cache TTL to match your token TTL so cache eviction and token expiration align.

**Offline access tokens (refresh tokens).** Refresh tokens let clients obtain new access tokens without redirecting the user to the provider. During an outage, clients with valid refresh tokens can continue operating as long as the token endpoint is available. If the token endpoint is also down, the refresh token is useless -- but at least the existing access tokens remain valid.

**Circuit breakers with graceful degradation.** Wrap your provider calls in a circuit breaker pattern. When the provider starts returning errors, open the circuit and fall back to cached sessions. When the circuit is open, do not attempt to reach the provider at all -- return cached results immediately. Close the circuit after a backoff interval and try a single request. This prevents the thundering herd problem where every login attempt hits a failing provider simultaneously.

## Failure Modes

You will encounter failures in production. Not all of them, but most of them. Here are the most common ones and how to handle each with specific mitigations you can implement today.

### 1. Provider Down -- No One Can Authenticate

Your OIDC provider goes offline. Every login attempt fails. New users cannot sign up. Existing users whose sessions have expired are locked out. This is the most visible and damaging failure mode because it prevents all user acquisition and re-authentication.

**Mitigation in detail.** First, cache the JWKS (JSON Web Key Set) locally on every resource server. Fetch it on startup, refresh it every 4-6 hours, and never block a request on a JWKS fetch. This ensures that token validation can proceed even if the provider is down. Second, use short-lived access tokens (5-15 minutes) so that during normal operation, the window of vulnerability for a compromised token is small. During a provider outage, those same short-lived tokens become a liability because they expire quickly. The real mitigation is to use refresh tokens with longer lifetimes so clients can re-authenticate without user interaction. Third, implement a fallback authentication mechanism for critical paths. If your application supports multiple identity providers (social login, enterprise SSO, email magic links), route users to an alternative during an outage. Fourth, instrument your provider health checks and alert when the provider is unreachable for more than 30 seconds. Include the provider status on your internal status page so support teams know before users report it.

### 2. Key Rotation Race

The OIDC provider rotates its signing keys. Your client fetches the JWKS set on startup and caches it. The provider issues a new ID token signed with a key that is not in your cache. Your validation fails. The user sees "authentication error" with no clear cause.

This is a classic race condition between the provider's key rotation schedule and your cache refresh interval. It manifests as intermittent failures that correlate with provider maintenance windows. The failures are hard to reproduce because by the time you investigate, the cache has refreshed and the old key is no longer in use.

**Mitigation in detail.** First, set a JWKS cache refresh interval that is shorter than the provider's key rotation interval. Most providers rotate keys every 24-48 hours. A 4-hour cache refresh detects rotations with ample margin. Second, every JWT has a `kid` (key ID) header. Always extract the `kid` and look it up from your cache. If the `kid` is not found, do not reject the token immediately. Instead, fetch the JWKS endpoint synchronously, update your cache, and retry the lookup. If the key is still not found after the fresh fetch, then reject the token. Third, implement exponential backoff for JWKS fetches. If the JWKS endpoint is unreachable, do not hammer it. Start with 1-second retry, double to 2, 4, 8 seconds, up to a 5-minute maximum. Fourth, pre-warm your cache. Before a scheduled provider maintenance window, proactively fetch the JWKS set so your cache has the latest keys before the rotation happens.

### 3. Nonce Replay Attack

An attacker intercepts an authorization code during the redirect. Without a nonce, they can replay that code to obtain a token. With a nonce, the replay is detected because the nonce was already consumed. The attack surface is not just the network -- an open redirect on your own site, a compromised intermediate proxy, or a malicious browser extension can all steal the authorization code.

The nonce is your only defense against authorization code interception attacks in the implicit flow. In the authorization code flow with PKCE, the `code_challenge` and `code_verifier` provide defense against interception, but the nonce still protects against ID token replay within the same session.

**Mitigation in detail.** First, always generate a cryptographically random nonce before initiating the auth request. Use `crypto.randomUUID()` in Node.js, `secrets.token_urlsafe()` in Python, or equivalent in your language. The nonce must have at least 128 bits of entropy. Second, store the nonce in the user's session before redirecting to the provider. This can be a server-side session store, an httpOnly cookie, or a client-side session storage. Third, when the ID token arrives, extract the `nonce` claim and compare it to the stored value. The comparison must be constant-time to prevent timing attacks. Fourth, delete the nonce from the session immediately after verification. One-time use is essential -- the nonce must be consumed on first use. Fifth, set a reasonable nonce expiration (5-10 minutes). If the auth request takes longer, invalidate the nonce and start over. This prevents an attacker from collecting stale nonces and trying to use them later.

### 4. Clock Skew -- Token Validation Fails

Your server's clock is 30 seconds ahead of the OIDC provider's clock. A token that was just issued has an `iat` (issued at) claim in the future from your server's perspective. You reject it. The user gets an authentication error for no reason they can understand.

Clock skew is one of the most common production issues with OIDC. It is silent, hard to debug, and causes intermittent failures that are nearly impossible to reproduce on demand. The failures come and go as time drifts, and they affect only a subset of tokens -- those issued in the window between when the provider's clock ticked and your server's clock ticked.

**Mitigation in detail.** First, most OIDC libraries accept a `leeway` parameter (typically 30-60 seconds) that you pass during token validation. Set this explicitly. Do not rely on the library default because library defaults vary and some libraries default to zero leeway. Second, synchronize all your servers against the same NTP pool. Use a local NTP server in your infrastructure rather than relying on public NTP servers from each server. This ensures all your servers agree on the current time. Third, monitor clock drift as part of your observability. Add a metric for clock drift (in milliseconds) per server, and alert when any server drifts more than 10 seconds from your NTP source. Fourth, if you operate in an air-gapped or isolated environment, run your own NTP server that your OIDC provider and your application servers both use. This eliminates clock skew within your system entirely.

### 5. UserInfo Endpoint Slow

You request the UserInfo endpoint for every API call to get fresh user profile data. The endpoint takes 500ms to respond. Now every authenticated API call has a half-second latency tax. Under load, the endpoint slows further because it queries a database on every call. Your API latency SLO is violated before your application logic even runs.

This failure mode is self-inflicted. The OIDC spec gives you the UserInfo endpoint as a way to get fresh user data, but it does not tell you to call it on every request. That decision is yours, and it is a bad one at scale.

**Mitigation in detail.** First, prefer putting claims directly in the ID token or access token. Use the `claims` parameter during the auth request to specify which claims you need. This eliminates the UserInfo call entirely for read-heavy workloads. Second, if you must use UserInfo, cache the responses aggressively. User profile data changes infrequently. A 5-minute cache TTL is safe for most applications. A 1-hour TTL is safe for profile data that is only displayed on a settings page. Third, use conditional requests. Include the `If-None-Match` header with the `ETag` from the previous response. If the data has not changed, the provider returns a 304 Not Modified with zero response body. This saves bandwidth and processing time on both sides. Fourth, batch UserInfo requests where possible. If you need profile data for multiple users, use a batch endpoint if your provider supports one. If not, make parallel requests rather than sequential ones.

### 6. Token Expiration Cascade

A user's access token expires. The client uses the refresh token to get a new one. But the refresh token has also expired. The user is silently logged out in the middle of their workflow. Data is lost. The user is frustrated.

This happens when you set access token TTL too short (5 minutes) and refresh token TTL too short (1 hour) without considering the user's actual session length. The cascade works like this: the access token expires after 5 minutes. The client attempts to refresh. The refresh token is valid, so it works. But after several refresh cycles, the user has been active for more than an hour. The next refresh attempt fails because the refresh token expired. The user sees "session expired" and loses whatever they were working on.

**Mitigation in detail.** First, align your token TTL strategy with your user's session expectations. For consumer applications, use 15-30 minute access tokens and 24-hour or longer refresh tokens. For enterprise applications where users expect to remain logged in across days, use 24-hour access tokens and 7-day refresh tokens with sliding expiration. Second, implement sliding session expiration. Extend the refresh token TTL on each active use. This prevents mid-work logout for active users while still expiring idle sessions. Third, give the client a grace period. When the refresh token expires, allow a single silent refresh within a short grace window (5-10 minutes after expiration). This handles the edge case where the client's clock is slightly off and attempts the refresh just after expiration. Fourth, save draft state frequently. Do not rely on the authentication session to protect user data. Auto-save form state, editor content, and other user work every 30-60 seconds so that a session expiration loses at most 60 seconds of work.

### 7. IdP Discovery Failure

Your application redirects to the OIDC provider's well-known discovery endpoint (`/.well-known/openid-configuration`) on every login. The discovery endpoint returns 404 or 500. Your application cannot find the authorization endpoint, the token endpoint, or the JWKS URI. No login can proceed.

This failure mode is common in multi-tenant SaaS applications where each tenant has a different OIDC provider. Discovery failures happen when a tenant misconfigures their provider URL, when the provider's discovery endpoint is restructured during a migration, or when network policies block access to the discovery URL.

**Mitigation in detail.** First, cache the discovery document aggressively. Fetch it on application startup and cache it for 24 hours. Refresh it in the background without blocking requests. Second, implement a fallback. If the discovery endpoint fails, use the configured endpoints directly from your tenant configuration. This requires storing the authorization endpoint, token endpoint, and JWKS URI alongside the provider URL in your tenant settings. Third, validate the discovery document on configuration. When a tenant configures a new OIDC provider, validate the discovery document synchronously during configuration rather than lazily on the first login. Fail fast during configuration, not at runtime.

## Anti-Patterns

These are patterns you will see in production code that look reasonable but are dangerously wrong. They are tempting because they save a few lines of code or a few milliseconds of effort. They are dangerous because they open security holes or create hard-to-debug production issues.

### 1. No Nonce Validation

You skip nonce validation because "the authorization code is already secure over HTTPS." You are wrong. The authorization code can be intercepted through an open redirect on your own site, a compromised intermediate proxy, or a browser extension. Without the nonce, an attacker who steals the code can exchange it for a token.

**Why this is dangerous.** The nonce ties the authorization request to the token response. Without it, a stolen authorization code can be used to obtain a token from any client. With it, the attacker would need to know the nonce, which was generated on the initial client and never transmitted over the network in a replayable form.

**The fix.** Always validate the nonce. It is a single string comparison that costs nothing and closes a real attack vector. In the authorization code flow with PKCE, the `code_verifier` provides protection against code interception, but the nonce still protects against ID token replay. Validate both.

### 2. Skipping ID Token Signature Verification

You trust the ID token because it came over HTTPS. You decode it without verifying the signature. An attacker who has compromised the TLS termination point (a reverse proxy, a load balancer, a compromised CA) can forge tokens.

**Why this is dangerous.** TLS protects data in transit between the provider and your client. But what if the attacker is not on the network path? What if they are inside your infrastructure? A compromised reverse proxy that terminates TLS can see and modify the decrypted traffic. If your application trusts the token based solely on having arrived over TLS, the attacker can inject a forged token at the proxy.

**The fix.** Always verify the signature. Fetch the JWKS endpoint once and cache the keys. Validate the `alg` header in the JWT. Reject tokens signed with the `none` algorithm. Reject tokens signed with symmetric keys (`HS256`) unless you absolutely control the shared secret and have verified that no other system has access to it. Use asymmetric algorithms (`RS256`, `ES256`) exclusively. These algorithms use public-key cryptography: your application holds the public key to verify, and only the provider holds the private key to sign.

### 3. Trusting Unverified Claims in the Access Token

You decode the access token (which is opaque to your application in the spec) and read claims from it without verification. The access token is not meant to be consumed by the client -- it is meant to be passed to the resource server. The client should use the ID token for identity information and call the UserInfo endpoint for additional claims.

**Why this is dangerous.** Access tokens may be JWT-format or opaque. If they are JWT-format, decoding them without verification is trivially easy (base64-decode the payload). But an unverified JWT can be tampered with. An attacker who modifies the payload and recomputes the base64 encoding can change the claims. Your application will read the modified claims and make authorization decisions based on forged data.

**The fix.** Use the right token for the right purpose. The ID token is for identity. The UserInfo endpoint is for profile data. The access token is for authorization -- pass it to the resource server and let the resource server verify it. If your access token is a JWT that you need to read, verify its signature first using the same JWKS endpoint you use for ID tokens. Never read claims from an unverified token.

### 4. Mixing ID Tokens and Access Tokens

You treat the ID token as an API credential. You pass it to your backend as the Bearer token. Your backend accepts it. Now you have no way to revoke the credential because ID tokens are not revocable -- they validate entirely by signature.

**Why this is dangerous.** ID tokens are designed to be read and validated by the client, not by the resource server. They are short-lived by design. If you use an ID token as an API credential, you lose the ability to:
- Revoke the credential (there is no revocation endpoint for ID tokens).
- Scope the credential to specific resources (ID tokens carry identity, not authorization).
- Rotate the credential independently of the authentication session.

**The fix.** Access tokens are for API authorization. ID tokens are for client-side identity display. Never use an ID token as an API credential. Use access tokens for backend communication, and use opaque or JWT-based access tokens that can be revoked at the provider. If you need identity information on the resource server, include it as claims in the access token or have the resource server call the UserInfo endpoint.

### 5. Storing Tokens in localStorage

You store tokens in localStorage because it is easy. An XSS vulnerability in your application gives an attacker full access to the token store. They exfiltrate the tokens and impersonate every user.

**Why this is dangerous.** localStorage is accessible from any JavaScript running on the same origin. A single XSS vulnerability -- a reflected XSS in a search parameter, a stored XSS in a user bio -- gives an attacker the ability to read every token from localStorage. They can exfiltrate the tokens to their server and use them to call your APIs as the victim user. The attack persists until the tokens expire or the user changes their password.

**The fix.** For single-page applications, use the authorization code flow with PKCE and keep tokens out of JavaScript-accessible storage entirely. Store tokens in a BFF (Backend for Frontend) pattern where the backend holds the tokens and issues httpOnly session cookies to the frontend. The frontend never sees the tokens. If you cannot use the BFF pattern, store access tokens in memory (a JavaScript variable) and refresh tokens in an httpOnly cookie with Secure, SameSite=Strict, and Path attributes. Never put tokens in localStorage or sessionStorage.

### 6. Assuming the `sub` Claim Is Immutable

You use the `sub` (subject) claim as a foreign key in your database. The provider changes the `sub` format during a migration. Suddenly all your user references are broken. User profiles are orphaned. History is lost.

**Why this is dangerous.** The OIDC spec explicitly says the `sub` is a locally unique identifier within the issuer's scope. It does not say it is stable across provider migrations, tenant migrations, or account merges. Providers change `sub` formats when they migrate between internal systems, when they re-tenant their database, or when they comply with new data privacy regulations.

**The fix.** Treat the `sub` as an opaque identifier that can change. If you need a stable user identifier, ask the provider to include a custom claim that is guaranteed stable across migrations (many enterprise IdPs support this). Alternatively, use a combination of `iss` (issuer URL) and `sub` as your composite key so that if the `sub` changes, you can at least disambiguate by issuer. Build your data model to handle identity migration gracefully -- store the provider name and the external ID as separate columns, and build a user merging tool for the support team.

### 7. Ignoring Token Expiration Until It Hurts

You do not check token expiration on the client because "the server validates it." The user's token expires. The next API call fails with a 401. No refresh logic exists on the client. The user is stuck on a broken page.

**Why this is dangerous.** Silent failures are the most damaging kind. The user sees a broken page with no error message. They refresh. They still see a broken page. They close the tab and open a new one. They are now logged out and have to log in again. If they were in the middle of a workflow, they lose their progress.

**The fix.** Implement proactive token refresh on the client. Check the token's `exp` claim before making an API call. If the token has less than 20% of its TTL remaining, refresh it proactively. This avoids the 401 entirely. For the cases where the token expires before the proactive refresh fires (long-running operations, slow networks), handle the 401 response gracefully. Intercept the 401 in your API client, attempt a single silent refresh, and retry the original request. Only show a session-expired page when the silent refresh itself fails. Do not redirect to the login page on every 401.

### 8. Using `alg: none` in Development and Forgetting to Remove It

You configure your OIDC library to accept unsigned tokens (`alg: none`) during local development because it is faster. You deploy to production without changing the configuration. Anyone can forge a token with `alg: none` and your server will accept it.

**Why this is dangerous.** The `alg: none` token is a JWT with no signature. Verifying a token with `alg: none` means the library does not verify anything. An attacker who knows the token format can forge a token with any claims they want. This is not a theoretical attack -- it has been used in real-world breaches against misconfigured JWT libraries.

**The fix.** Never accept `alg: none` in any environment, including development. Configure your OIDC library to require a specific asymmetric algorithm (RS256 or ES256). If you must use a symmetric algorithm in development, use a strong random secret and switch to asymmetric for production. Validate the `alg` header explicitly in your token validation code and reject any token that does not use the expected algorithm.

## When NOT to Use OIDC

OIDC is not always the right answer. Here are the situations where you should pick something else. Recognizing these situations is as important as knowing how to use OIDC correctly. The common thread across all of them is that OIDC solves a specific problem -- federated identity across organizational boundaries with a web or mobile client. When your problem does not match that description, you should look for a simpler, more targeted solution.

### 1. Legacy Enterprise with SAML Investment

Your organization has a decade of SAML infrastructure. Every internal application speaks SAML. Your identity team knows SAML. Your compliance auditors have approved your SAML implementation.

Do not rip and replace. Add OIDC incrementally for new applications, but let the legacy system keep using SAML. The cost of migrating working SAML integrations far exceeds the benefit of OIDC's simplicity. A migration would require retesting every integration, re-certifying every audit control, retraining every support team member, and running both protocols in parallel during the transition -- months or years of work with no user-facing benefit.

Instead, deploy a federated identity gateway that translates between SAML and OIDC at the boundary. New applications speak OIDC to the gateway. The gateway speaks SAML to the legacy IdP. This gives you the modern integration surface for new development without touching the existing infrastructure.

### 2. Closed Systems with No Third-Party Integrations

You are building a system where the same team controls the client, the API, and the user database. No third-party applications need to authenticate. No federation with external identity providers is required.

In this case, OIDC adds complexity with no benefit. A simple session cookie over HTTPS is simpler, faster, and easier to debug. Add OIDC only when you need to federate identity across organizational boundaries. If your system is a monolith with a single frontend and a single API, use a session cookie with a server-side session store. The cookie is opaque to the client, revocable on the server, and requires no key management, no JWKS fetching, and no token validation library.

### 3. IoT or Device Ecosystems with No Browser

OIDC assumes a browser or a browser-like environment. The authorization code flow, the redirects, the user-agent interaction -- all of these are built around a human with a browser. IoT devices typically have no browser, no user input capability, and no way to handle redirects.

For IoT devices, use the device authorization grant (RFC 8628) if the device has limited input capability. The user authenticates on a separate device (their phone) while the IoT device polls for the token. For machine-to-machine communication, use the client credentials grant directly -- the device authenticates with its client credentials and receives a token without any user interaction. For embedded systems with no network stack capable of HTTPS, consider API keys or mutual TLS instead. OIDC is the wrong tool for a smart light bulb that needs to call a weather API.

### 4. High-Frequency Internal Service Calls

Your services call each other millions of times per hour. Adding OIDC token validation to every call adds latency and load. The token validation itself (fetching JWKS, verifying signatures, checking expiration) is cheap, but the orchestration and error handling around it adds complexity.

For internal service-to-service communication within a trusted network, consider mutual TLS with short-lived certificates or a service mesh instead of OIDC. A service mesh like Istio or Linkerd can handle authentication at the proxy layer, validating certificates for every request without any application code changes. Reserve OIDC for the user-facing authentication boundary. The cost of token validation at the internal service level is not justified when mTLS at the network level provides equivalent security with less complexity.

### 5. Real-Time Systems with Sub-100ms Latency Requirements

Your system processes financial trades, gaming moves, or live video events. Every millisecond counts. OIDC's token validation, even at its fastest, adds overhead. Parsing a JWT, verifying the signature, checking the expiration, and extracting claims takes 1-5 milliseconds in a fast library in a compiled language. In a JavaScript or Python runtime, it can take 10-20 milliseconds.

For these systems, authenticate at the session level (WebSocket handshake or TLS handshake) and then use a lightweight session identifier for subsequent messages. Do not validate a JWT on every message. OIDC is for the authentication handshake, not for real-time data plane traffic. Authenticate once during connection establishment, then use a lightweight session token (a random string or a short-lived HMAC) for the duration of the connection.

### 6. Single-Page Applications with No Backend

You are building a purely client-side SPA that calls third-party APIs directly. The third-party API does not support OIDC. It uses API keys or OAuth 1.0. You are considering adding an OIDC proxy layer just to standardize authentication.

Do not do this. Adding OIDC when none of your downstream consumers support it adds a proxy layer, a token translation step, and a new point of failure. Use the authentication mechanism that your downstream APIs already support. Add OIDC only when it reduces complexity, not when it adds complexity for the sake of consistency.

Here is what you should carry forward from this page into your next architecture discussion, code review, or incident response.

- **OIDC trades enterprise maturity for modern simplicity.** This is the right call for most applications, but know what you are giving up. If your compliance auditors require XML-level audit trails, SAML may still be your answer.

- **Your provider is CP in CAP terms.** Consistency over availability during partitions. Plan for auth outages. Build circuit breakers, cache JWKS locally, and implement offline token support.

- **Seven failure modes will hit you some day.** Provider downtime, key rotation races, nonce replay, clock skew, slow UserInfo endpoints, token expiration cascades, and IdP discovery failure. Mitigate each one before it hits production by implementing the specific mitigations described in this page.

- **Eight anti-patterns will tempt you.** Skip nonce validation. Trust unsigned tokens. Mix ID and access tokens. Store secrets in localStorage. Assume immutable subjects. Ignore expiration until it burns you. Use `alg: none` in development and forget to remove it. Do not fall for any of them. Each one is a production incident waiting to happen.

- **Six situations call for a different protocol.** Legacy SAML environments, closed systems, IoT devices, high-frequency internal calls, real-time systems, and SPAs with no backend all have better authentication options than OIDC. Know when to say no.
