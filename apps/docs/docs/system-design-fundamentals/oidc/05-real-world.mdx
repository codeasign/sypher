# OIDC in Production

OpenID Connect is not just a specification -- it is a production-hardened protocol
running at planetary scale. Every major identity provider (IdP) implements OIDC,
but each does so with distinct architectural choices shaped by its scale, threat
model, and deployment environment.

This page examines how four production OIDC providers work under the hood, surveys
open-source and cloud-hosted alternatives, and distills patterns you can apply
in your own identity architecture.

---

## Case Study 1: Google as an OIDC Provider

Google's OIDC implementation powers billions of sign-ins per day across Google
Workspace, Google Cloud, and third-party applications. Google's architecture is
defined by three imperatives: **global scale**, **low latency**, and **zero-trust
security**.

At the core of Google's OIDC flow is the **Google Authorization Server**, which
handles both the authentication and token issuance steps of the OIDC protocol.

**Key architectural decisions:**

- **Session-based auth, not prompt-based.** Google does not prompt the user for
  credentials with every OIDC request. Instead, it validates the user's existing
  session cookie. If the session is valid, the user is redirected to the consent
  page directly. This reduces authentication latency from seconds to milliseconds.

- **Consent is cached.** Once a user grants consent to a third-party application,
  Google remembers it. On subsequent OIDC requests from the same client, the
  consent step is skipped entirely, and the user is redirected straight to the
  callback with the authorization code.

- **Authorization codes are one-time use.** Google issues an authorization code
  that is valid for 5 minutes and can be used exactly once. The token endpoint
  validates the code, the client ID, the client secret (for confidential clients),
  the redirect URI, and the PKCE code verifier (for public clients) before
  issuing tokens.

- **JWK rotation.** Google publishes its signing keys at
  `https://www.googleapis.com/oauth2/v3/certs`. Keys are rotated on a regular
  schedule. The `kid` (key ID) in the JWT header tells the client which key to
  use for signature verification. Google also publishes a `keys` endpoint that
  includes both current and recently rotated keys, allowing clients to verify
  tokens signed with the previous key during the rotation window.

**Scaling challenges:**

Google's OIDC endpoints handle millions of requests per second. The system uses
a multi-tier cache architecture:

1. **L1 cache** (in-memory, per-frontend-server): Caches JWK sets, client
   metadata, and recent consent decisions. TTL: 60 seconds.
2. **L2 cache** (distributed, Redis-like): Caches session data, authorization
   codes, and pending consent requests. Data is replicated across data centers
   for availability.
3. **Persistent store** (Spanner): The source of truth for client registrations,
   user accounts, and token metadata. Reads are eventually consistent; writes
   require quorum.

The lesson for you: **cache aggressively, but keep the cache TTL short.** Google
caches consent decisions and session state for seconds, not hours. This limits
the blast radius of a stale cache entry while still absorbing the vast majority
of requests.

## Case Study 2: Microsoft Entra ID

Microsoft Entra ID (formerly Azure Active Directory) is Microsoft's cloud identity
service, used by hundreds of millions of users across Microsoft 365, Azure, and
third-party applications. Entra ID implements OIDC with several Microsoft-specific
extensions.

Entra ID's OIDC flow differs from Google's in one critical way: **Entra ID
supports both the authorization code flow and the implicit flow**, though the
implicit flow is deprecated in favor of the authorization code flow with PKCE.

**Entra ID-specific features:**

- **Tenant-scoped issuer.** The `iss` (issuer) claim in Entra ID tokens includes
  the tenant ID (e.g., `https://login.microsoftonline.com/{tenant-id}/v2.0`).
  A multi-tenant application must verify the issuer against a known tenant list,
  not just accept any tenant. This prevents cross-tenant token confusion.

- **Groups overage claim.** When a user belongs to more than 200 groups, Entra
  ID does not include the full group list in the ID token. Instead, it includes
  a `_claim_names` claim that indicates the groups claim is too large, and the
  application must call the Microsoft Graph API to fetch the group membership
  separately. This is a pragmatic design for token size limits -- JWTs should
  be under 8 KB to avoid HTTP header size issues.

- **Conditional Access.** Entra ID integrates with Microsoft's Conditional Access
  engine, which evaluates risk signals (user location, device compliance,
  sign-in risk, user risk) before issuing tokens. If the risk score exceeds a
  threshold, the token is not issued, and the user is prompted for additional
  authentication (MFA, password change, or blocked). This is evaluated at the
  token endpoint, not during the initial authentication.

- **Token lifetime controls.** Administrators can configure access token lifetime
  (default 60-90 minutes), ID token lifetime (default 60 minutes), and refresh
  token lifetime (up to 90 days of inactivity). Short-lived tokens reduce the
  window of a compromised token but increase the load on the token endpoint.

**Performance considerations:**

Entra ID's token endpoint is a global service fronted by Azure Traffic Manager
with anycast routing. The service uses tenant-level sharding: each tenant's
metadata and policies are stored in a dedicated partition. This ensures that
token issuance for one tenant does not affect another tenant's latency.

The lesson for you: **multi-tenant OIDC providers must tenant-scope their
issuer claim.** Without tenant-scoped issuers, a malicious tenant could craft
tokens for another tenant. This is the most common OIDC vulnerability in
multi-tenant applications.

## Case Study 3: Okta

Okta is an independent identity-as-a-service (IDaaS) provider that supports
OIDC at its core. Unlike Google and Microsoft, Okta does not have a consumer
identity business -- it focuses entirely on enterprise identity and workforce
authentication.

Okta's OIDC architecture is built on a **policy engine** that evaluates every
token request against a set of configurable rules. This makes Okta highly
customizable but also introduces complexity in the token issuance path.

**Key architectural features:**

- **App-level policy configuration.** Each OIDC application in Okta has its own
  authentication policy, consent policy, and token policy. This means the same
  user may get different tokens for different applications -- one app might
  require MFA and issue 1-hour tokens, while another accepts password-only
  auth and issues 8-hour tokens.

- **Inline hooks.** Okta supports inline hooks that allow custom logic to
  execute during the token issuance flow. A hook is an HTTP callback to a
  customer-hosted endpoint that receives the token request context and returns
  modifications to the token claims or a denial decision. This allows customers
  to inject custom claims (e.g., department, cost center, custom roles) without
  modifying the directory schema.

- **Token revocation.** Okta provides a token revocation endpoint (`/revoke`)
  that invalidates both access tokens and refresh tokens. Revoked tokens are
  tracked in an in-memory revocation list at the token endpoint. When a token
  is introspected, the revocation list is checked. This is critical for
  workforce scenarios where an employee leaves the company and their sessions
  must be terminated immediately.

- **Rate limiting per application.** Okta enforces rate limits at the application
  level, not just the tenant level. A misbehaving client in one application
  does not affect other applications in the same Okta org. Rate limits are
  configurable per application.

**Lessons from Okta:**

The policy engine architecture is powerful but introduces latency. Each policy
evaluation requires a round-trip to the policy store, and the inline hook
adds an external HTTP call. Okta mitigates this with aggressive caching:
authentication policies are cached in-memory for 30 seconds, and token policies
are cached for 60 seconds.

The lesson for you: **policy-driven token issuance is powerful but cache the
policy, not the token.** Caching the token would prevent revocation. Caching
the policy allows fast evaluation while still respecting policy changes.

## Case Study 4: Auth0

Auth0 (now part of Okta) is a developer-focused identity platform that implemented
OIDC from the ground up as a cloud-native service. Unlike the other providers in
this section, Auth0 was designed for the **developer experience** -- its
configuration is API-first, its documentation is exemplary, and its default
settings are secure by default.

Auth0's architecture is built on a **tenant-per-ISO** model: each tenant runs in
a specific geographic region, and all data for that tenant stays within that
region. This is a deliberate design choice for data residency compliance.

**Auth0's key innovations:**

- **Actions (formerly Rules and Hooks).** Auth0 Actions allow developers to
  inject custom logic into the token issuance pipeline. An Action is a JavaScript
  function that runs before or after token issuance. Actions can:
  - Add custom claims to the ID token (e.g., user's subscription tier)
  - Call external APIs to enrich the token (e.g., fetch user permissions from
    a downstream service)
  - Deny token issuance based on custom logic (e.g., block suspended users)
  - Redirect to a custom error page instead of the default error screen

  Actions run in a sandboxed environment with a 20-second timeout. If the Action
  times out, token issuance fails with an error. This prevents runaway Actions
  from blocking the token endpoint.

- **Organizations.** Auth0's Organizations feature allows a single Auth0 tenant
  to serve multiple customer organizations. Each organization gets its own login
  page, its own branding, and its own set of applications. The OIDC flow is
  modified to include the organization ID in the token request, and the ID token
  includes an `org_id` claim. This is a common pattern for B2B SaaS applications.

- **Attack protection.** Auth0 built a dedicated attack protection system that
  monitors the OIDC flow for abuse:
  - Brute-force protection: locks accounts after N failed login attempts
  - Breached password detection: checks passwords against known breach databases
    (powered by HaveIBeenPwned)
  - Bot detection: uses machine learning to distinguish human logins from bot
    traffic
  - Suspicious IP throttling: rate-limits requests from IPs with a high failure
    rate

- **Custom domains.** Auth0 allows customers to use their own domain for the OIDC
  endpoints (e.g., `login.acme.com` instead of `acme.us.auth0.com`). This is
  implemented by provisioning a TLS certificate for the custom domain and
  configuring a reverse proxy that rewrites the host header. The custom domain
  must be verified by the customer before it becomes active.

**Performance characteristics:**

Auth0's token endpoint is designed for low latency. The median token issuance
time is under 50ms for simple flows (no Actions, no custom claims). Each Action
adds approximately 20-50ms for the sandboxed JavaScript execution, plus any
external HTTP calls made by the Action.

The lesson for you: **customizable token issuance is a feature, not a bug.**
Auth0's Actions allow developers to adapt OIDC without modifying the protocol,
but each Action adds latency. Measure the overhead of your pipeline and set
appropriate timeouts.

## Open Source OIDC Providers

When you need to run your own OIDC provider -- for air-gapped environments,
custom compliance requirements, or cost control -- three open-source projects
dominate the landscape.

### Ory Hydra

Ory Hydra is a lightweight, API-first OIDC provider that implements the OAuth
2.0 and OIDC specifications. It is designed to be embedded into existing
infrastructure: Hydra does not handle user authentication itself. Instead, it
delegates authentication to a separate application via a **login endpoint**.

This separation is Hydra's defining architectural choice. The flow works as
follows:

1. Hydra receives an OIDC authorization request.
2. Hydra redirects the user to the configured login application, passing a
   `login_challenge` parameter.
3. The login application authenticates the user (however it wants -- SAML, LDAP,
   custom database, social login).
4. The login application calls Hydra's Admin API with the `login_challenge` and
   a `subject` identifier.
5. Hydra completes the OIDC flow and issues the token.

**Key strengths:**
- **Authentication-agnostic.** Hydra does not care how you authenticate users.
  Use LDAP, SAML, passwordless, WebAuthn, or a custom database -- Hydra's
  login flow works with any authentication method.
- **Container-native.** Hydra is distributed as a single Docker image with no
  external dependencies for the control plane. The database (PostgreSQL,
  MySQL, CockroachDB, or SQLite) is the only external dependency.
- **OpenAPI-first.** Every Hydra API is defined by an OpenAPI specification.
  Client libraries are auto-generated from the spec.
- **JSON Web Key management.** Hydra includes a built-in JWK management API
  for key rotation. Keys can be stored in the database or in an external
  key management service (KMS).

**Weaknesses:**
- No built-in user interface. You must build the login and consent screens.
- The delegated login flow adds network round-trips. The login application
  must be highly available -- if it is down, no one can log in.
- No built-in user management. Hydra stores only OAuth 2.0 client registrations
  and session data, not user accounts.

### Dex

Dex is a federated OIDC provider that acts as a **bridge** between identity
providers. Dex does not store user accounts. Instead, it authenticates users
against a configured backend (LDAP, SAML, GitHub, Google, Microsoft, or a
custom connector) and issues OIDC tokens for downstream applications.

Dex was originally created by CoreOS (now part of Red Hat) and is widely used
in Kubernetes environments. The standard deployment pattern is: Kubernetes
API server delegates authentication to Dex, and Dex authenticates against
a corporate LDAP or SAML identity provider.

**Key strengths:**
- **Connector-based architecture.** Dex supports connectors for 15+ identity
  backends. Writing a custom connector is straightforward: implement the
  `Connector` interface with `Login` and `Refresh` methods.
- **Kubernetes integration.** Dex is the standard OIDC provider for
  Kubernetes clusters. The `kube-apiserver` authenticates users via Dex,
  and Dex authenticates against the enterprise IdP.
- **Token refresh.** Dex supports refresh tokens for all connector types.
  The refresh flow re-authenticates the user against the backing IdP without
  requiring the user to re-enter credentials.
- **Minimal configuration.** A basic Dex deployment requires only a
  configuration file and a database backend.

**Weaknesses:**
- No user management. Dex is a pure proxy -- it does not store user attributes,
  roles, or group memberships beyond what the connector provides.
- Connector quality varies. The GitHub connector is well-maintained; the LDAP
  connector is functional but lacks advanced features like nested group
  resolution.
- No consent management. Dex does not support the OIDC consent flow. Users
  are authenticated and redirected without seeing a consent screen.

### Keycloak

Keycloak is a full-featured identity and access management (IAM) platform
that includes an OIDC provider, SAML provider, user management, role-based
access control, and a web admin console. It is the most feature-complete
open-source OIDC provider.

**Key strengths:**
- **All-in-one.** User registration, password policies, session management,
  social login, multi-factor authentication, account recovery, and an admin
  console -- all in a single deployment.
- **User federation.** Keycloak can synchronize users from LDAP, Active
  Directory, or any external database. Users are either imported into
  Keycloak's database or accessed on-demand from the external store.
- **Fine-grained authorization.** Beyond OIDC authentication, Keycloak
  supports resource-based authorization, role-based access control, and
  user-managed access (UMA).
- **Admin REST API.** Everything the admin console does can be done via
  the REST API. This is essential for automation and infrastructure-as-code.
- **Themes.** The login pages, consent screens, and email templates are
  fully customizable via themes. Each application can have its own theme.

**Weaknesses:**
- **Heavy.** Keycloak is a Java application (WildFly / Quarkus) that requires
  a relational database (PostgreSQL, MySQL, MariaDB). Minimum memory: 512 MB
  for a small deployment, 2 GB+ for production.
- **Complex configuration.** The admin console exposes hundreds of settings.
  Misconfiguration can lead to security vulnerabilities (e.g., disabling
  PKCE enforcement, setting overly permissive token lifetimes).
- **Database bottleneck.** Every authentication request reads from the
  database. Keycloak caches aggressively (users, sessions, roles) but the
  cache miss path is slow. For high-traffic deployments, you need a
  dedicated database cluster with read replicas.

## Cloud OIDC Services

Every major cloud provider offers an OIDC-compatible identity service. These
services are designed for applications that run on the cloud provider's
infrastructure and need to authenticate users.

### AWS Cognito

Amazon Cognito is AWS's managed identity service. It includes two components:
**Cognito User Pools** (a user directory and OIDC provider) and **Cognito
Identity Pools** (for AWS credentials exchange).

Cognito User Pools implement OIDC with the following characteristics:

- **Hosted UI.** Cognito provides a hosted login page that can be customized
  with your domain and branding. The hosted UI supports social login (Google,
  Facebook, Apple, Amazon) and SAML federation.
- **Lambda triggers.** Cognito supports Lambda triggers at key points in the
  OIDC flow: pre-authentication, post-authentication, pre-token generation,
  and post-confirmation. The pre-token generation trigger is particularly
  useful -- it can add custom claims to the ID token.
- **Adaptive authentication.** Cognito supports risk-based authentication
  using Amazon Fraud Detector. If the login attempt is flagged as high-risk,
  the user is prompted for additional verification.
- **Token customization.** The ID token includes standard claims (`sub`,
  `aud`, `iss`, `exp`, `iat`) plus any custom attributes defined in the
  user pool schema. Custom attributes must be prefixed with `custom:`.

**Cognito limitations:**
- The hosted UI cannot be fully customized beyond CSS changes. The HTML
  structure is fixed.
- Lambda triggers have a 5-second timeout. If the trigger takes longer,
  the authentication fails.
- Cognito does not support the OIDC `claims` parameter for requesting
  individual claims. All claims in the scope are always included.
- Maximum user pool size: 10 million users (soft limit, can be increased).

### Azure AD (Microsoft Entra ID)

Azure AD (now Microsoft Entra ID) is both a cloud identity service and an
OIDC provider. In addition to the enterprise features described in the
Entra ID case study, Azure AD offers:

- **App registrations.** Each application that uses OIDC is registered as
  an app registration in Azure AD. The registration includes the redirect
  URIs, client ID, client secret or certificate, and API permissions.
- **Microsoft Authentication Library (MSAL).** MSAL is a client library
  that handles the OIDC flow automatically. It manages token caching, token
  refresh, and token validation. MSAL is available for .NET, JavaScript,
  Python, Java, Android, and iOS.
- **Conditional Access policies.** Azure AD's Conditional Access engine
  can block or challenge OIDC token requests based on user risk, device
  compliance, location, and application sensitivity.
- **Managed identities.** Azure resources (VMs, App Services, Functions)
  can be assigned a managed identity. The managed identity obtains an OIDC
  token from Azure AD without managing credentials -- Azure handles the
  token lifecycle automatically.

### GCP Identity Platform

Google Cloud's Identity Platform (GCIP) is a customer identity and access
management (CIAM) service that provides OIDC-compatible authentication.

- **Multi-tenancy.** GCIP supports multiple tenants in a single project.
  Each tenant has its own user directory, social login configuration, and
  OIDC client registrations. This is ideal for B2B SaaS applications.
- **Blocking functions.** Cloud Functions that execute during the OIDC
  flow. A blocking function can inspect the authentication request and
  return an error to block the sign-in (e.g., if the user is in a
  blocked country).
- **Passwordless authentication.** GCIP supports email links, phone
  SMS, and anonymous authentication in addition to traditional password
  and social login.
- **Identity platform to Google Cloud mapping.** After authentication,
  GCIP can map the user's identity to a Google Cloud service account.
  This allows the user to access Google Cloud resources (Storage, BigQuery,
  Cloud Run) using their application identity.

## Lessons Learned: Five Key Patterns

After examining these production OIDC systems, five patterns emerge that you
can apply to any identity architecture.

### Pattern 1: Short-Lived Access Tokens, Long-Lived Refresh Tokens

Every production OIDC provider uses the same token lifecycle strategy: access
tokens live for minutes (30-90 minutes), refresh tokens live for days or
weeks (up to 90 days). This is a deliberate tradeoff.

**Why it matters:**
- A compromised access token is dangerous for only 30-90 minutes.
- A compromised refresh token is more dangerous, but refresh tokens are
  stored in secure storage (HTTP-only cookies, secure enclaves, keychains)
  and are never exposed to the browser.
- Short access tokens reduce the need for token revocation. If the token
  is short-lived, you can wait for it to expire rather than revoking it.

**Implementation tip:** Set access token TTL to 15 minutes for high-security
applications, 60 minutes for standard applications. Generate a new refresh
token on every refresh (rotation) and invalidate the previous one. This
limits the window of a compromised refresh token.

### Pattern 2: Cache JWKs, But Honor the `kid` Header

Every OIDC provider publishes a JWK Set (JWKS) at a well-known URL. Clients
must fetch this set to verify the token signature. The `kid` (key ID) in the
JWT header tells the client which key from the set to use.

**Why it matters:**
- Fetching the JWKS on every token validation adds a network round-trip.
  A cache hit returns the key in microseconds.
- The `kid` header allows multiple keys to be active simultaneously. This
  enables key rotation: the old key is still valid for tokens signed before
  the rotation, and the new key is used for new tokens.
- Providers rotate keys on a regular schedule (Google: every 24 hours,
  Microsoft: every 7 days, Okta: configurable). Clients must respect the
  rotation schedule.

**Implementation tip:** Cache the JWKS in memory with a TTL of 5 minutes.
On cache miss, fetch the key set and update the cache. Always verify the
`kid` in the token header matches a key in the cached set. If the `kid`
is not found, fetch the key set fresh (the key may have been rotated).

### Pattern 3: Validate the Issuer, Not Just the Signature

The most common OIDC vulnerability is **issuer confusion** -- an attacker
registers their own OIDC provider, creates a token for their own tenant,
and presents it to your application. If your application only verifies the
signature (which is valid against the attacker's key), it accepts the token.

**Why it matters:**
- The `iss` (issuer) claim must match your application's expected issuer.
  For multi-tenant applications, you must validate against a allowlist of
  known issuers.
- Microsoft Entra ID's tenant-scoped issuer (`https://login.microsoftonline.com/
  {tenant-id}/v2.0`) is the standard defense against this attack.
- Auth0 supports tenant-scoped issuers with the `org_id` claim.

**Implementation tip:** Store the expected issuer as a configuration value
in your application. On token validation, compare the `iss` claim against
the expected value. For multi-tenant applications, maintain a list of
allowed issuers and reject any token whose issuer is not in the list.

### Pattern 4: Separate Authentication from Authorization Claims

OIDC ID tokens should contain authentication claims -- who the user is,
when they authenticated, and how they authenticated. Authorization claims
-- what the user can do -- should come from a separate source.

**Why it matters:**
- ID tokens are signed by the IdP. If the IdP includes authorization claims
  (roles, permissions), the authorization logic is coupled to the IdP.
  Changing the authorization model requires changing the IdP configuration.
- Authorization claims can be stale. If a user's role changes, the ID token
  may still contain the old role until the token expires.
- The OIDC specification recommends keeping ID tokens small. Large ID tokens
  (over 8 KB) cause HTTP header size issues.

**Implementation tip:** Use the ID token only for authentication (verify
the user's identity). Use the `sub` claim (subject) to look up the user's
authorization in your application's database or a dedicated authorization
service. Alternatively, use the access token to call a userinfo endpoint
that returns the user's current authorization state.

### Pattern 5: Plan for Token Revocation

Token revocation is the hardest problem in OIDC. The protocol does not
require tokens to be revoked -- it relies on short expiration times as the
primary revocation mechanism. But there are scenarios where immediate
revocation is required: a user leaves the company, a device is stolen, or
a security incident is detected.

**Why it matters:**
- Without revocation, a compromised token is valid until it expires.
- A 60-minute access token gives an attacker 60 minutes of access.
- OIDC provides a token revocation endpoint (`/revoke`), but not all
  providers implement it, and not all clients use it.

**Implementation tip:** Implement a token revocation list (a deny list of
token `jti` values) that is checked before each API call. The revocation
list is stored in a fast cache (Redis) with a TTL equal to the token's
remaining lifetime. This ensures the list does not grow unbounded. For
additional security, use short access tokens (15 minutes) and rely on
refresh token revocation for long-lived sessions.

## Bridging to the Interview

When you discuss OIDC in a system design interview, you now have the
vocabulary and architecture patterns to speak with authority:

- **Start with the protocol.** "I would use OIDC with the authorization code
  flow and PKCE. The client redirects the user to the IdP, the user
  authenticates, and the IdP returns an authorization code. The client
  exchanges the code for tokens at the backchannel token endpoint."

- **Discuss the token lifecycle.** "Access tokens are short-lived -- 15 to
  60 minutes. Refresh tokens are long-lived -- up to 90 days -- and are
  rotated on each refresh. JWKs are cached with a 5-minute TTL and the
  `kid` header is used for key selection."

- **Address security.** "I validate the issuer, audience, expiration, and
  nonce on every token. I cache the JWKS but always fetch fresh keys if
  the `kid` is not found in the cache. I implement a token revocation list
  in Redis for immediate revocation scenarios."

- **Compare providers.** "For a multi-tenant B2B SaaS application, I would
  use Auth0 or AWS Cognito. For a single-tenant enterprise deployment, I
  would use Keycloak or Ory Hydra. For a Kubernetes-native environment,
  I would use Dex."

- **Acknowledge the tradeoffs.** "Longer token lifetimes reduce load on the
  IdP but increase the window of a compromised token. Custom claims in the
  ID token are convenient but create coupling between the IdP and the
  authorization model. Delegated login (Ory Hydra) is flexible but adds
  latency and an extra dependency."

The best candidates do not just describe OIDC -- they justify every design
decision with a concrete tradeoff analysis grounded in real production
systems. Now you can.
