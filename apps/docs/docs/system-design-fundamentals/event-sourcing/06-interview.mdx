## Question Types

Event sourcing questions in system design interviews fall into five categories. Each tests a different skill.

### Type 1: Design Questions

You are asked to design a system where event sourcing is a natural fit.

**Example:** "Design a banking system that supports account transfers, balance inquiries, and transaction history."

**What they test:** Can you identify when event sourcing is appropriate? Can you design the aggregate, event types, projections, and read models?

**Answer framework:**

1. Identify the core aggregates (BankAccount).
2. Identify the commands (Deposit, Withdraw, Transfer).
3. Identify the events (Deposited, Withdrew, Transferred).
4. Design the event store schema.
5. Design the projections (balance read model, transaction history read model).
6. Address consistency (optimistic concurrency, idempotency).
7. Address scale (snapshots, CQRS, partitioning).

### Type 2: Compare and Contrast Questions

You are asked to compare event sourcing with other approaches.

**Example:** "Compare event sourcing with a traditional CRUD database for an order management system."

**What they test:** Do you understand the trade-offs? Can you make a reasoned recommendation?

**Answer framework:**

1. Define both approaches clearly.
2. List the advantages of event sourcing for this domain (audit trail, time travel, multiple read models).
3. List the disadvantages (complexity, storage growth, eventual consistency).
4. Make a recommendation based on requirements (audit requirements, query patterns, team expertise).

### Type 3: Troubleshooting Questions

You are given a production incident and asked to diagnose and fix it.

**Example:** "Users report that their account balance is incorrect in the UI. The event store has all events, but the read model database was rebuilt 2 hours ago. What could be wrong?"

**What they test:** Operational understanding of event sourcing systems.

**Answer framework:**

1. Check projection lag (is the projection keeping up?).
2. Check the projection rebuild (was the rebuild correct?).
3. Check for missed events (any events published but not processed?).
4. Check for duplicate events (idempotency handling correct?).
5. Check for schema version mismatches (old events through new code?).
6. Check snapshot consistency (if snapshots are used).

### Type 4: Schema Evolution Questions

You are asked how to handle changes to event schemas over time.

**Example:** "You need to add a fee field to the Withdrawal event. Events are already in production. How do you handle this?"

**What they test:** Understanding of upcasting, migration, and backward compatibility.

**Answer framework:**

1. Add a schema version field to the new events (v2).
2. Write an upcaster that transforms v1 events to v2 during replay.
3. Deploy the new projection code that handles v2 events.
4. Verify that old events are correctly upcasted during projection rebuild.
5. Optionally, run a migration to rewrite old events to v2 format.

### Type 5: Scalability Questions

You are asked how to scale an event-sourced system.

**Example:** "Your event store is processing 50,000 events per second. Projections are falling behind. What do you do?"

**What they test:** Understanding of distributed event sourcing, partitioning, and stream processing.

**Answer framework:**

1. Check the bottleneck (event store? projection code? read model database?).
2. Partition projections (each partition handles a subset of streams).
3. Scale projection instances horizontally (one instance per partition).
4. Batch event processing (process 100-1,000 events per transaction).
5. Optimize read model writes (batch inserts, prepared statements, remove unnecessary indexes).

## Levels of Sophistication

### Junior Level (0-2 years)

**What is expected:** You can define event sourcing, describe the basic components, and explain the difference between event sourcing and CRUD.

**Sample answer (junior):**
"Event sourcing stores every state change as an event instead of storing the current state. To get the current state, you replay all events. The main components are the event store (where events live), aggregates (which validate commands and produce events), and projections (which build read models from events)."

**Key phrases to use:**
- "Events are facts — they are immutable."
- "State is derived from replaying events."
- "The event store is append-only."

### Mid Level (2-5 years)

**What is expected:** You can design a medium-complexity event-sourced system, explain the consistency model, and handle schema evolution.

**Sample answer (mid-level):**
"I would model the BankAccount as an aggregate that enforces invariants — for example, the balance cannot go negative. Each command goes through the aggregate, which validates and emits events. Events are stored atomically with optimistic concurrency control using expected version checks. The balance read model is maintained by a projection that subscribes to events. If the read model falls behind, I monitor projection lag and alert when it exceeds 5 seconds."

**Key phrases to use:**
- "Optimistic concurrency with expected version."
- "Snapshots at configurable intervals."
- "Upcasters for schema evolution."
- "Projection lag as a monitoring metric."

### Senior Level (5+ years)

**What is expected:** You can design a distributed, multi-region event-sourced system, handle complex failure scenarios, and make architectural trade-off decisions.

**Sample answer (senior):**
"I would partition the event store by aggregate ID across 64 partitions, with each partition backed by a 3-node EventStoreDB cluster. Global aggregates would be write-located to a single region to avoid distributed consistency problems. Projections would run as independent services, each consuming events from their assigned partitions. The read model would use a shadow-rebuild pattern for zero-downtime deployment. For cross-region deployments, I would replicate events asynchronously using Kafka MirrorMaker with 1-5 second replication lag. The biggest risk is schema evolution across versions, so I would enforce that every event carries a schema version and all upcasters are tested with historical events before deployment."

**Key phrases to use:**
- "Eventual consistency with tunable projection lag."
- "Shadow rebuild pattern for zero-downtime projection updates."
- "Write-located global aggregates for multi-region."
- "Compensation events for error correction."
- "Tiered storage for event retention."

## Phrases That Signal Expertise

Use these phrases naturally in your answers to demonstrate depth:

- "The event store is the source of truth. Everything else — the read model, the cache, the search index — is a derived view that can be rebuilt at any time."
- "Snapshot frequency is a performance tuning parameter. More snapshots mean faster aggregate loads but more write amplification. The right balance depends on the read-to-write ratio of your workload."
- "Optimistic concurrency means we never lock the aggregate. We check the version at write time and retry if there is a conflict. This works well for low-contention aggregates but degrades under high contention."
- "Upcasting happens at read time, not write time. Old events are transformed on-the-fly during replay. This keeps the event store append-only while allowing schema changes."
- "Projection lag is the difference between the latest event position and the last processed position. It is the single most important operational metric in an event-sourced system."
- "A compensation event is an event that corrects a previous event. It preserves the audit trail while fixing errors. There is no delete in the event store."
- "CQRS is not required for event sourcing, but they pair naturally. The event store is the write model; projections build read models. This separation allows independent scaling of reads and writes."
- "The biggest operational risk is untested upcasters. Every schema change must be tested against historical events. A staging environment with a copy of production events is essential."

## Red Flags

Red flags are statements or design choices that indicate a lack of understanding.

| Red Flag | What It Signals | Better Answer |
|----------|----------------|---------------|
| "Events are stored in a regular SQL table and we can update them if needed." | Does not understand immutability. | "Events are immutable. If data is wrong, we append a compensation event." |
| "We use event sourcing for everything — blog posts, comments, todos." | Does not understand trade-offs. | "Event sourcing is best for domains with audit requirements, complex state, or long-lived processes. For simple CRUD, the complexity is not justified." |
| "We never use snapshots because they add complexity." | Does not understand scaling. | "Snapshots are a performance optimization. For aggregates with over 1,000 events, snapshots reduce load time from seconds to milliseconds." |
| "The event store and read model are the same database." | Does not understand separation of concerns. | "The event store is the source of truth. The read model is a derived view. They should be in separate stores." |
| "We can rebuild the read model from the current database state." | Does not understand the role of the event store. | "The read model is always rebuilt from the event store, not from itself. The event store is the single source of truth." |
| "We version events by adding a version field but we never use upcasters." | Does not understand schema evolution. | "Old events need to be transformed to the current schema during replay. Upcasters handle this." |
| "If an event has wrong data, we delete it from the database." | Fundamental misunderstanding. | "Events are immutable. You append a correction or compensation event." |
| "The read model is always consistent with the event store." | Does not understand eventual consistency. | "Read models are eventually consistent. There is always some lag between event storage and projection update." |

## Practice Drills

### Drill 1: Design a Banking System (30 minutes)

Design an event-sourced banking system that supports:
- Account creation with initial deposit.
- Deposits, withdrawals, and transfers between accounts.
- Balance inquiry (current and historical).
- Monthly statement generation.

**Cover:**
- Aggregate design (BankAccount aggregate).
- Event types (AccountOpened, Deposited, Withdrew, Transferred, StatementGenerated).
- Projections (BalanceProjection, TransactionHistoryProjection, StatementProjection).
- Consistency (optimistic concurrency for withdrawals, transfer two-phase append).
- Snapshots (every 100 events for active accounts).
- Schema evolution (add a fee field to Withdrew events).

### Drill 2: Diagnose a Production Incident (20 minutes)

Your team deploys version 2.5 of an event-sourced order management system. After deployment, the ActiveOrders projection shows no orders for the last 3 hours, but the event store has new OrderPlaced events with timestamps in the last 3 hours. What happened and how do you fix it?

**Step-by-step diagnosis:**

1. Check projection lag. If the projection is at position X and the event store is at position Y, the lag is Y - X.
2. Check the projection logs. Are there errors? Look for schema mismatch errors.
3. Check the event types in the last 3 hours. Was a new event type introduced in the deployment that the projection does not handle?
4. Check the OrderPlaced events. Did the schema change in a way the projection code does not expect?
5. Roll back the projection to version 2.4 if the issue is a new event type. Rebuild the projection from scratch.
6. Fix the bug in version 2.6. Add handling for the new event type. Test with old and new events.

### Drill 3: Schema Evolution (20 minutes)

You have 500,000 AccountOpened events in your event store. The current schema includes: `customer_id`, `initial_deposit`, `account_type`. You need to add a `referral_code` field. Events created after today must include the referral code. Old events do not have it.

**Answer:**

1. Update the AccountOpened event class to include an optional `referral_code` field.
2. Bump the schema version to 2.
3. Write an upcaster for version 1 to version 2:

```python
class AccountOpenedUpcasterV1ToV2(Upcaster):
    def can_upcast(self, event_type, version):
        return event_type == "AccountOpened" and version == 1

    def upcast(self, event):
        data = json.loads(event.data)
        # Add default value for new field
        data["referral_code"] = None
        data["schema_version"] = 2
        return [EventRecord(
            event_type=event.event_type,
            data=json.dumps(data),
            metadata=event.metadata
        )]
```

4. Deploy the upcaster. All replay will now see AccountOpened events with the `referral_code` field.
5. New events from the application will include the actual referral code.

### Drill 4: Scale Projections (15 minutes)

Your event store processes 10,000 events per second. The BalanceProjection processes 2,000 events per second on a single instance. The projection is falling behind at a rate of 8,000 events per second.

**Answer:**

The projection cannot keep up. Three solutions:

**Option A: Parallelize by partition.** Split the event stream into 4 partitions (by aggregate ID hash). Run 4 projection instances, each processing one partition. Throughput becomes 4 x 2,000 = 8,000 events per second. Still behind, but better.

**Option B: Batch processing.** Instead of processing one event at a time, batch 100 events and update the read model in a single database transaction. Database round-trips drop from 10,000 to 100 per second. Throughput increases to approximately 8,000 events per second.

**Option C: Partition + batch.** Use 8 partitions with batches of 100 events each. Throughput becomes 8 x 10,000 = 80,000 events per second. The projection catches up and has headroom.

```python
# Batch processing in projection
class BatchBalanceProjection:
    def __init__(self, batch_size: int = 100):
        self.batch = []
        self.batch_size = batch_size

    def handle_event(self, event):
        self.batch.append(event)
        if len(self.batch) >= self.batch_size:
            self.flush()

    def flush(self):
        # Single database transaction for the batch
        with db.transaction():
            for event in self.batch:
                if event.event_type == "FundsDebited":
                    db.execute(
                        "UPDATE account_balances SET balance = balance - ? "
                        "WHERE account_id = ?",
                        (event.data["amount"], event.data["account"])
                    )
                elif event.event_type == "FundsCredited":
                    db.execute(
                        "UPDATE account_balances SET balance = balance + ? "
                        "WHERE account_id = ?",
                        (event.data["amount"], event.data["account"])
                    )
        self.batch = []
```

### Drill 5: Event Sourcing vs CRUD Decision (15 minutes)

You are building a ride-hailing application. Which components benefit from event sourcing?

**Analysis by component:**

- **Ride lifecycle (requested, accepted, picked up, dropped off, paid).** Event sourcing is beneficial. The ride has a complex state machine, audit requirements (driver disputes, fare adjustments), and multiple read models (active rides for the map, ride history for the user, payment records for accounting).

- **User profile (name, phone, payment methods).** CRUD is sufficient. No audit requirement, no complex queries, simple operations.

- **Driver location updates (every 3 seconds per driver, 10,000 drivers).** Neither event sourcing nor CRUD — use a time-series database or Redis for geospatial data. Event sourcing would produce 288 million events per day for location data alone.

- **Fare calculation rules (per-city rate tables).** CRUD is sufficient. Rate tables change infrequently, have no audit requirement, and are simple key-value lookups.

**Recommendation:** Use event sourcing for the ride lifecycle only. Use CRUD for user profiles and rate tables. Use a specialized time-series store for location data.

### Drill 6: Full Design Exercise (45 minutes)

Design an event-sourced inventory management system for a warehouse with 100,000 SKUs. The system must track stock levels, reservations, returns, and supplier orders.

**Part A: Aggregate Design (10 minutes)**

Identify the aggregates and their event types.

<details>
<summary>Model Answer</summary>

**Aggregates:**

1. **SKU Inventory** (per product): tracks stock level, reserved quantity, reorder threshold.
   - Events: StockAdded, StockDeducted, StockReserved, StockReservationReleased, StockAdjusted, ReorderThresholdChanged, StockCountCorrected.

2. **Supplier Order** (per purchase order to supplier): tracks order lifecycle.
   - Events: SupplierOrderPlaced, SupplierOrderConfirmed, SupplierOrderShipped, SupplierOrderReceived, SupplierOrderCancelled.

3. **Warehouse Transfer** (move stock between warehouse locations).
   - Events: TransferInitiated, TransferPicked, TransferReceived, TransferCancelled.

Each aggregate is a consistency boundary. The SKU Inventory aggregate ensures stock does not go negative. The Supplier Order aggregate ensures orders are not double-received. The Warehouse Transfer aggregate ensures stock is not double-counted during transfer.
</details>

**Part B: Projection Design (10 minutes)**

Design the projections and read models needed.

<details>
<summary>Model Answer</summary>

**Projections:**

1. **CurrentStockProjection.** Maintains a SQL table of current stock level per SKU.
   - Handles: StockAdded, StockDeducted, StockReserved, StockReservationReleased, StockCountCorrected, StockAdjusted.
   - Used by: product page, warehouse dashboard, order fulfillment system.

2. **LowStockAlertProjection.** Maintains a table of SKUs where stock is below reorder threshold.
   - Handles: StockDeducted, ReorderThresholdChanged, StockAdded, StockCountCorrected.
   - Used by: procurement team, automated reorder system.

3. **SupplierOrderProjection.** Maintains a document store of open and recent supplier orders.
   - Handles: all SupplierOrder events.
   - Used by: procurement dashboard, receiving team.

4. **WarehouseActivityProjection.** Maintains a time-series table of stock movements for analytics.
   - Handles: StockDeducted, StockAdded, TransferReceived.
   - Used by: operations team, warehouse efficiency reports.
</details>

**Part C: Concurrency and Consistency (10 minutes)**

Two warehouse workers simultaneously pick the last 5 units of SKU-001. The system shows 5 units available. Both workers submit a pick request for 5 units. How does event sourcing prevent overselling?

<details>
<summary>Model Answer</summary>

When the first worker submits the pick request, the system loads the SKU-001 aggregate (version 5, reserved 0, stock 5). The aggregate validates: 5 units available, 0 reserved. It emits StockReserved(5) and StockDeducted(5). The event store appends with expected_version=5. The stream advances to version 6.

When the second worker's pick request arrives, it loaded the aggregate at the same time (version 5). It tries to append with expected_version=5. The event store rejects the append because the stream is now at version 6. The command handler catches the WrongExpectedVersion exception, reloads the aggregate (version 6, stock 0), re-validates (insufficient stock), and returns an error to the second worker.

This is standard optimistic concurrency. No locks are needed. The second worker is automatically rejected and can retry when stock is replenished.
</details>

**Part D: Recovery (15 minutes)**

The database for the low-stock alert projection crashes and all data is lost. Describe the recovery procedure step by step.

<details>
<summary>Model Answer</summary>

1. **Acknowledge the incident.** The procurement team is notified that low-stock alerts are temporarily unavailable. Manual checks are used until the projection is rebuilt.

2. **Reset the projection position.** The projection service is told to start from position 0 (the beginning of the event stream).

3. **Rebuild the projection.** The projection service reads all events from the event store, one by one, and rebuilds the low-stock alert table. With 1 billion events in the event store, this takes approximately 3 hours at 100,000 events per second.

4. **Verify correctness.** After rebuilding, a sample of 1,000 SKUs is checked. The projection is compared against the current stock projection to ensure correctness. Any SKU where stock is below threshold must appear in the alert table.

5. **Swap to the new projection.** If using shadow rebuild, the old (empty) table is atomically swapped with the new (populated) table. The procurement team sees the restored alerts immediately.

6. **Post-mortem.** The root cause is determined (database crash). Monitoring is improved to alert on database health. The recovery time is measured against the SLA target (4 hours).
</details>

## Mock Interview Questions

**Question 1 (5 minutes):** "Explain event sourcing to a non-technical product manager."

**Expected answer structure:**
- "Event sourcing is like a bank statement. Instead of just telling you your current balance, it records every deposit and withdrawal."
- "Every change to data is stored as a permanent, unchangeable record."
- "If you need to know what happened at any point in the past, you just replay the records up to that date."
- "The downside is that it is more complex to build and maintain than a regular database."

**Question 2 (7 minutes):** "What are the most common problems you encounter with event sourcing in production?"

**Expected answer structure:**
- Projection lag (handling high-volume event streams).
- Schema evolution (old events with old formats).
- Event store corruption (backup and recovery procedures).
- Snapshot inconsistency (verification and recovery).
- Team learning curve (training and documentation).

**Question 3 (10 minutes):** "Design an event-sourced system for a food delivery platform. Focus on the order lifecycle."

**Expected answer structure:**
- Identify aggregates: Order, Restaurant, DeliveryDriver, Payment.
- Identify events: OrderPlaced, OrderAccepted, OrderPrepared, OrderPickedUp, OrderDelivered, PaymentProcessed.
- Identify projections: ActiveOrders (for the customer app), RestaurantQueue (for restaurants), DriverAssignment (for dispatchers).
- Address concurrency: Two drivers cannot accept the same order (optimistic concurrency on the Order aggregate).
- Address failure: A driver fails to deliver (compensation events, reassignment saga).

## Common Mistakes in Interviews

Avoid these mistakes when answering event sourcing questions in system design interviews.

**Mistake 1: Treating event sourcing as a silver bullet.**

Do not say "I would use event sourcing for everything." Instead, explain which components benefit from event sourcing and which do not. For the ride-hailing system: event sourcing makes sense for the ride lifecycle but not for driver location updates.

**Mistake 2: Forgetting to mention eventual consistency.**

If you design an event-sourced system and describe the read model returning data immediately after a write, you have missed a key point. Explain that read models are eventually consistent — there is a lag between writing an event and seeing its effect in the read model. If the requirement demands read-your-writes consistency, explain how you would handle it (synchronous projection, reading from the aggregate, or reading from the event store for the current request).

**Mistake 3: Ignoring schema evolution.**

Many candidates design an event-sourced system but never mention how the event schema changes over time. Interviewers look for this. When you present your design, add a note: "Events carry a schema version number, and we use upcasters to transform old events during replay."

**Mistake 4: Not addressing storage growth.**

Interviewers may ask: "The event store has been running for 5 years. It is 10 TB. What do you do?" Prepare an answer covering tiered storage (hot SSD, warm HDD, cold S3), retention policies, and archiving. Show that you have thought about long-term operational costs.

**Mistake 5: Confusing commands with events.**

In an interview, you might be asked: "What events does the BankAccount emit?" If you answer "Withdraw" (a command) instead of "Withdrew" or "WithdrawalExecuted" (an event), you have made a category error. Commands are requests in the imperative tense; events are facts in the past tense. Always use past tense for events.

**Mistake 6: Overcomplicating the initial design.**

Start simple. A common interview arc is: "Design a banking system." You begin with a simple event store and one projection. Then the interviewer asks about performance, and you add snapshots. Then about scale, and you add CQRS. Then about multi-region, and you add replication. Do not jump to the most complex architecture in the first 30 seconds. Let the interviewer guide you to higher complexity as the conversation progresses.

## Summary

Interview success with event sourcing requires:

- **Conceptual clarity.** You can explain events, event stores, aggregates, snapshots, and projections without hesitating.
- **Trade-off awareness.** You know when event sourcing is appropriate and when it is not.
- **Operational understanding.** You can diagnose projection lag, schema evolution failures, and other production issues.
- **Scaling knowledge.** You can design partitioned, multi-region event sourcing architectures.

Practice the seven drills above and internalize the key phrases and red flags. Focus on understanding the *why* behind each design decision, not just the *what*.
