## Case Studies

### 1. Event Store DB: Financial Trading Platform

A European trading platform processes over 500,000 trades per day, each generating multiple events (order placed, filled, partially filled, cancelled, settled). They migrated from a traditional SQL-based order management system to EventStoreDB in 2019.

**The problem:** The old system stored only the current state of each order. When a trade dispute arose, reconstructing what happened required digging through application logs, database change data capture streams, and manual reconciliation. The process took 3-5 days per dispute.

**The solution:** Every order lifecycle event is stored as an immutable event in EventStoreDB. The order aggregate is rebuilt from its event stream on every command. Read models are built from three projections: one for active orders (fast queries), one for historical orders (archival), and one for regulatory reporting.

**Results:**

- Trade dispute resolution time dropped from 3-5 days to under 30 minutes.
- Regulatory audits that previously required a dedicated team of 3 for 2 weeks now require 1 person for 2 days.
- New query capabilities (for example, "show me all orders that were modified between 2 PM and 3 PM on June 15") are implemented by adding a new projection, not by migrating the database schema.
- The event store has grown to 12 TB over 5 years, managed with tiered storage (hot events on SSD, events older than 6 months on S3, archived after 3 years).

**Key lesson:** The investment in event sourcing was justified by the regulatory compliance requirement. The ability to answer any historical question about any order in minutes, not days, is a competitive advantage in the financial industry.

### 2. Greg Young's Original Work: Event Sourcing in Finance

Greg Young formalized the event sourcing pattern as part of Domain-Driven Design and CQRS in the late 2000s. His work at a large financial institution in London demonstrated event sourcing for a foreign exchange trading system.

**The insight:** The system processed currency trades with complex validation rules. Each trade could be modified by multiple parties (trader, compliance officer, settlement team) before final execution. The traditional CRUD approach required complex locking and conflict resolution. Event sourcing simplified the model: each modification was a new event, and the trade state was always the result of replaying all events in order.

**Key contributions from Greg Young's work:**

- The concept of the aggregate as a consistency boundary, ensuring that all events for a single entity are processed atomically.
- The recommendation to use event sourcing within a single bounded context, not across the entire system.
- The pattern of snapshots as a pure performance optimization, not a correctness mechanism.
- The distinction between commands (requests that can be rejected) and events (facts that are always accepted).

**Key lesson:** Event sourcing is a domain modeling technique, not a database technology. The value comes from thinking about your domain in terms of events, not from the event store software.

### 3. Axon Framework: Insurance Claims Processing

A large European insurance company uses the Axon Framework to process claims across 14 different product lines (auto, health, life, property, travel). The system handles 2 million claims per year, each with a lifecycle of 30-90 days involving multiple approvals, rejections, resubmissions, and payments.

**The problem:** Each claim had a complex lifecycle with state transitions that depended on the claim type, the claimant's history, the policy terms, and regulatory requirements. The old monolithic system used a single massive `claims` table with 247 columns, and business logic was scattered across stored procedures, triggers, and application code. Adding a new claim type required 6-9 months of development.

**The solution:** The team redesigned the system using Axon Framework with event sourcing. Each claim type is a separate aggregate. The business logic for each claim type is encapsulated in its aggregate, and events drive the state machine.

**Results:**

- New claim type development time dropped from 6-9 months to 4-6 weeks.
- Compliance audits are now handled by replaying events for the specific claim, not by hiring external auditors to review the system.
- The system was migrated to a microservices architecture by splitting the monolith along aggregate boundaries.
- The event store (Axon Server) now contains 1.5 billion events across 10 years of operation.

**Key lesson:** Event sourcing excels in domains with complex state machines and long-lived processes. The ability to replay events to understand the exact sequence of state transitions is invaluable for debugging and compliance.

### 4. Event Sourcing in E-Commerce: Order Management

A large e-commerce platform (processing 10 million orders per month) uses event sourcing for their order management system. Each order generates 15-30 events from placement through delivery.

**The problem:** The order management system had grown to support 40+ order statuses, 200+ state transitions, and integrations with 15 external systems (payment processors, shipping carriers, fraud detection, inventory management). The CRUD-based system had become unmanageable — adding a new order status required modifying 50+ code paths, and debugging order issues required correlating logs from 15 services.

**The solution:** The order system was rebuilt with event sourcing. Each order is an aggregate. External integrations are handled by event handlers that subscribe to order events. The read model (a document database) provides fast order lookup for the customer-facing application.

**Results:**

- Adding a new order status now requires adding one new event type and updating the relevant projections. No database schema changes are needed.
- Debugging order issues: engineers query the event stream for a specific order and see the exact sequence of events.
- External system integration failures are handled by retrying the event handler, not by manual database fixes.
- The system has been running in production for 4 years with zero data loss incidents.

**Key lesson:** Event sourcing is particularly valuable in systems with many state transitions and external integrations. Each event is a recovery point: if an integration fails, you can retry the event handler without data loss.

## Open Source Tools

### EventStoreDB

EventStoreDB is the most popular purpose-built event store. It was created by Greg Young and is maintained by Event Store Ltd.

**Key features:**

- Append-only event storage with optimistic concurrency.
- Built-in projection engine (JavaScript-based).
- Clustered mode for high availability (3 or 5 nodes).
- HTTP and gRPC APIs.
- Subscription model for event processing.
- Persistent subscriptions (at-least-once delivery) and volatile subscriptions (in-memory, single consumer).
- Built-in snapshots.
- Event metadata (correlation ID, causation ID) built into the data model.

**When to use:** You need a purpose-built event store with strong consistency guarantees and a projection engine. EventStoreDB is the most mature option.

**When not to use:** You need a multi-tenant, cloud-managed event store. EventStoreDB is self-hosted or available as Event Store Cloud.

### Apache Kafka as Event Store

Kafka is not an event store — it is a distributed log. However, many teams use Kafka as their event store, storing events as Kafka messages and using Kafka's log compaction for snapshot-like retention.

**Key features:**

- Extremely high throughput (1 million+ messages/second per cluster).
- Built-in partitioning and replication.
- Log compaction (retains the latest value per key — useful for state snapshots).
- Kafka Streams for stream processing.
- Kafka Connect for integration with external systems.
- Mature ecosystem with extensive tooling.

**When to use:** You need extreme throughput, already have a Kafka investment, or need stream processing alongside event storage.

**When not to use:** You need fine-grained concurrency control (optimistic concurrency per stream). Kafka does not provide built-in expected-version checking. You need built-in projections (Kafka requires external stream processors).

```python
# Using Kafka as an event store with concurrency control
# Implemented via a compacted topic for stream versions
class KafkaEventStore:
    def __init__(self, bootstrap_servers: str):
        self.producer = KafkaProducer(bootstrap_servers=bootstrap_servers)
        # Compacted topic tracks {stream_id: version}
        self.version_topic = "stream-versions"

    def append(self, stream_id: str, expected_version: int,
               events: list[EventRecord]) -> list[AppendResult]:

        # Check current version (from compacted topic)
        current_version = self._get_version(stream_id)
        if current_version != expected_version:
            raise WrongExpectedVersion(stream_id, current_version, expected_version)

        results = []
        for event in events:
            current_version += 1
            event.version = current_version
            future = self.producer.send(
                topic="events",
                key=stream_id.encode(),
                value=event.serialize()
            )
            # Wait for acknowledgment
            record_metadata = future.get(timeout=5)
            results.append(AppendResult(
                stream_id=stream_id,
                version=event.version,
                offset=record_metadata.offset
            ))

        # Update version in compacted topic
        self.producer.send(
            topic=self.version_topic,
            key=stream_id.encode(),
            value=str(current_version).encode()
        )

        return results
```

### Axon Framework

Axon is a Java framework that implements the CQRS and event sourcing patterns. It provides aggregate modeling, event store integration, and projection management.

**Key features:**

- Aggregate modeling with annotations (`@Aggregate`, `@CommandHandler`, `@EventSourcingHandler`).
- Built-in event store (Axon Server) or pluggable storage (JPA, MongoDB, Kafka).
- Event bus and event handling with automatic retry.
- Deadline handling (scheduling future events).
- Saga orchestration for long-running transactions.
- Integration with Spring Boot.

**When to use:** You are building a Java application with complex domain logic and want a framework that handles event sourcing infrastructure.

**When not to use:** You are not using Java, or your application is simple enough that event sourcing overhead is not justified.

## Cloud Services

### AWS EventBridge

AWS EventBridge is a serverless event bus that can be used as part of an event sourcing architecture. It is not a full event store — it does not provide append-only storage or event replay — but it can serve as the event bus for publishing events to projections.

**Capabilities:**

- Event ingestion from AWS services and custom applications.
- Event filtering and routing to targets (Lambda, SQS, Step Functions).
- Event archiving and replay (up to 14 days of archival).
- Schema registry for event versioning.

**Limitations:**

- Maximum event size of 256 KB.
- 14-day event retention limit (not suitable as a long-term event store).
- No optimistic concurrency control.
- No built-in aggregate support.

**Best use case:** Use EventBridge as the event bus in a larger event sourcing architecture. Store events in a durable event store (EventStoreDB, DynamoDB, or S3) and use EventBridge for routing events to projections.

### Azure Event Grid

Azure Event Grid is a serverless event routing service similar to AWS EventBridge. It provides event ingestion, filtering, and routing to Azure services.

**Capabilities:**

- Event ingestion from Azure services and custom applications.
- Event filtering and routing to Azure Functions, Service Bus, Event Hubs.
- Dead-letter events for failed deliveries.
- Event schemas and versioning.

**Limitations:**

- 1 MB maximum event size.
- 24-hour event retention (for dead-letter).
- No event replay capability.
- No optimistic concurrency.

**Best use case:** Use Event Grid as the event distribution layer in Azure-based event sourcing systems. Use Event Hubs or Cosmos DB as the durable event store.

## Lessons from Production

### Lesson 1: Start with a Single Bounded Context

Do not try to event source your entire system at once. Choose one bounded context — the one with the most complex state management or the strongest audit requirements — and implement event sourcing there. Leave the rest of the system on CRUD. This limits the learning curve and the operational risk.

### Lesson 2: Invest in Tooling Early

Event sourcing requires tooling that CRUD systems do not:

- **Event viewer.** A UI for browsing the event stream, filtering by event type, aggregate, or time range.
- **Projection rebuild tool.** A CLI or API for triggering a rebuild of a specific projection.
- **Consistency checker.** A tool that periodically compares aggregate state from replay vs snapshot.
- **Schema evolution dashboard.** A view of which event schema versions are in use and how many events need upcasting.

Build these tools as part of the initial implementation, not as an afterthought. Without them, debugging production issues is significantly harder.

### Lesson 3: Monitor Projection Lag

Projection lag is the most important operational metric for an event-sourced system. It tells you if your projections are keeping up with the event volume. Set up alerts for when lag exceeds your SLA (typically 5-30 seconds, depending on your domain).

### Lesson 4: Plan for Event Retention

The event store grows without bound. Plan for this from day one:

- Choose a retention policy (hot storage for 90 days, warm storage for 1 year, cold storage beyond).
- Implement tiered storage or archiving.
- Budget for SSD storage costs.
- Test the archiving and recovery process before you need it.

### Lesson 5: Test Schema Evolution

Schema evolution is the most common source of production failures in event-sourced systems. Test it thoroughly:

- Write integration tests that replay old events through new projections.
- Run a staging environment with a copy of production events.
- Practice the upcasting and rebuild process before each deployment that changes event schemas.

### Lesson 6: Document the Event Types

Maintain a catalog of all event types in your system. For each event type, document:

- The event name, schema version, and data fields.
- When it is emitted (which command, which aggregate).
- Which projections handle it.
- The schema evolution history (what changed in each version).

This catalog is essential for onboarding new team members and for debugging production issues.

### Lesson 7: Practice Disaster Recovery

Every event-sourced system needs a documented disaster recovery plan. Test it quarterly.

**Recovery scenarios to practice:**

| Scenario | Recovery Action | Target Time |
|----------|----------------|-------------|
| Event store primary node fails | Failover to replica | Under 1 minute |
| Entire event store cluster lost | Restore from backup | Under 4 hours |
| Read model database corrupted | Rebuild from event store | Under 1 hour |
| Projection code bug deployed | Roll back projection, rebuild | Under 30 minutes |
| Event schema migration fails | Roll back to old upcasters | Under 15 minutes |

**Recovery drill for read model loss:**

```bash
# Step 1: Stop all write traffic to prevent event accumulation
# (or let events accumulate — the read model will catch up)

# Step 2: Reset the projection position
curl -X POST /projections/balance/reset

# Step 3: Start projection rebuild from scratch
# Projection reads all events from position 0
# This may take 30 minutes to 4 hours depending on event volume

# Step 4: Monitor rebuild progress
curl /projections/balance/progress
# Expected output: {"processed": 5000000, "total": 10000000, "eta_seconds": 1800}

# Step 5: Verify read model consistency with a sample
python verify_read_model.py --sample-size 10000

# Step 6: Redirect traffic to rebuilt read model
# For shadow rebuild: swap the table pointer
curl -X POST /projections/balance/swap
```

### Lesson 8: Consider the Total Cost of Ownership

The following table shows the estimated monthly infrastructure cost for event-sourced systems at different scales:

| Scale | Events/Day | Infrastructure | Est. Monthly Cost |
|-------|-----------|---------------|-------------------|
| Small | 10,000 | 1 PostgreSQL (event store), 1 app server | $150 |
| Medium | 500,000 | 3-node EventStoreDB, 2 projections, 1 Read Model DB | $1,200 |
| Large | 50,000,000 | 9-node EventStoreDB cluster, 10 projection services, 3 read model DBs, caching layer | $15,000 |
| Enterprise | 1,000,000,000+ | Multi-region EventStoreDB, Kafka for replication, 50+ projections, global caching | $100,000+ |

At the enterprise level, the cost is justified by regulatory compliance requirements and the cost of audit failures. For a medium-scale application, the $1,200/month cost may be a significant increase over a simple CRUD database that costs $50/month.

## Production Deployment Checklist

Before deploying an event-sourced system to production, verify each item:

**Event Store**

- [ ] Event store running in clustered mode (minimum 3 nodes).
- [ ] Backup strategy configured and tested (full backup daily, incremental every hour).
- [ ] Restore procedure documented and tested in staging within the last 30 days.
- [ ] Monitoring configured: disk usage, write latency, read latency, cluster health.
- [ ] Retention policy defined (hot/warm/cold/archive tiers).
- [ ] Connection pooling configured for the event store client.

**Projections**

- [ ] Each projection has its own database or schema (no shared read model DB).
- [ ] Projection position checkpointing enabled (resume from last processed event on restart).
- [ ] Shadow rebuild capability implemented and tested.
- [ ] Projection lag monitoring with alerts (warning at 30 seconds, critical at 5 minutes).
- [ ] Dead-letter queue configured for events that fail processing.

**Commands and Aggregates**

- [ ] Idempotency keys implemented for all commands.
- [ ] Optimistic concurrency enforced on all event appends.
- [ ] Rate limiting configured per aggregate to prevent event flooding.
- [ ] Command validation logic tested with unit tests and integration tests.

**Schema Evolution**

- [ ] Upcaster registry implemented with at least one version of backward compatibility.
- [ ] Integration tests cover replaying old events through new upcasters.
- [ ] Staging environment has a copy of production events for schema change testing.

**Operations**

- [ ] Event viewer tool deployed (browse events by stream, type, or time range).
- [ ] Consistency checker implemented (periodically compares snapshot state to replay state).
- [ ] Runbook documented for: read model rebuild, event store failover, schema migration.
- [ ] On-call team trained on event sourcing concepts and common failure modes.
- [ ] Cost budget set for event store storage with projected growth for the next 12 months.

## Summary

Event sourcing is used in production across finance, insurance, e-commerce, and other domains where auditability, complex state management, and long-lived processes are requirements. The technology choices range from purpose-built event stores (EventStoreDB) to general-purpose distributed logs (Kafka) to cloud services (EventBridge, Event Grid).

The common thread across all successful deployments: event sourcing is applied to a bounded context where the benefits clearly outweigh the costs, and the team invests in the operational tooling and testing practices that make event sourcing maintainable.
