## How to Use This Section

Each challenge tests a specific skill.

Tier 1 verifies you understand core concepts and vocabulary: hit rate,
eviction policies, cache stampedes, caching strategies, and the reasoning
behind caching decisions. These are quick checks of conceptual grounding
that should take 5 to 10 minutes each.

Tier 2 tests your ability to apply the pattern under real-world constraints
with concrete numbers: you must calculate memory requirements, design key
structures with appropriate TTLs, handle hot keys, build invalidation
pipelines, and design read and write paths with measurable latency targets.
These are full design exercises requiring 20 to 30 minutes each.

Tier 3 pushes you into ambiguous territory where there is no single correct
answer: you must navigate trade-offs between consistency and latency, justify
design decisions under budget and team constraints, and evaluate alternative
approaches with incomplete information. These simulate real architecture
decisions where the answer depends on context, constraints, and priorities.

Attempt each challenge without looking at the answer first. Write down your
response or speak it aloud as you would in an interview. For Tier 2, force
yourself to do the math before reading the model answer. For Tier 3, try to
arrive at at least two defensible positions before reading the guide. After
each challenge, compare against the answer and note where your reasoning
differed or where you missed a consideration.

## Tier 1: Foundation

These challenges verify that you have absorbed the core concepts. Each one

### Challenge 1.1: Cache Hit Rate Calculation

You run a news website serving 2 million article page views per day. Each
article requires a database query taking 45 milliseconds. You deploy a Redis
cache storing rendered article HTML. Over 24 hours, you measure 1,840,000
cache hits and 160,000 cache misses.

**Questions:**

(a) Calculate the cache hit rate as a percentage. Show the formula.

(b) How many milliseconds of database query time did the cache save over
24 hours? Express in milliseconds, seconds, and hours.

(c) On average, how many milliseconds per request did the cache save compared
to a no-cache setup? Assume cache hit latency is 3 milliseconds (network +
Redis lookup). Express the improvement as a multiplier.

(d) Each new article requires a 15-minute index build before appearing in
search results. How would you handle caching during that build window?
Describe two approaches and recommend one.

---

**Model Answer:**

(a) Hit Rate = (Cache Hits) / (Total Requests) * 100

    = 1,840,000 / 2,000,000 * 100

    = 92 percent.

92 out of every 100 page views are served from cache without touching the
database. Only 8 out of 100 trigger a database query.

(b) The cache saved 45ms of database time for each hit.

    Total saved = 1,840,000 * 45 ms = 82,800,000 milliseconds.

    Divide by 1,000 to get seconds: 82,800 seconds.

    Divide by 3,600 to get hours: 23 hours.

The cache saves nearly an entire database server-day of work every 24 hours.

(c) Without cache: every request takes 45ms.

    With cache: 92 percent take 3ms (hit), 8 percent take 45ms (miss).

    Average = (0.92 * 3) + (0.08 * 45) = 2.76 + 3.6 = 6.36ms.

    Savings = 45 - 6.36 = 38.64ms per request.

    Multiplier: 45 / 6.36 = 7.07x faster.

The uncached page loads in 45ms (noticeable delay). The cached page loads in
approximately 6ms (feels instantaneous).

(d) Approach 1: Set a 15-minute TTL. First read after index build triggers a
miss and refetches with indexed metadata. Penalizes the first reader.

Approach 2: Populate cache at publish time. Publishing pipeline inserts into
DB and immediately SETs the cache key with rendered HTML. Index build runs
asynchronously. First reader always gets a cache hit.

Recommendation: Approach 2. It does not penalize direct-access users. The
cost is one extra Redis SET command in the publish pipeline.

### Challenge 1.2: Eviction Policy Selection

Three different use cases run on the same Redis instance. For each workload,
recommend the most appropriate eviction policy.

**Workload A:** A product catalog with 500,000 items. 10 percent (50,000)
receive 90 percent of traffic. The remaining 450,000 are rarely accessed but
must be available if searched for. Cache holds approximately 200,000 entries.

**Workload B:** A session store for 2 million active sessions. Each session
is accessed on every API request. Sessions have a 24-hour sliding TTL. Cache
holds 1.5 million sessions.

**Workload C:** A news content cache. 1,000 new articles published per day.
Each article gets a traffic burst in the first 2 hours, then near zero. Cache
holds 50,000 entries.

---

**Model Answer:**

**Workload A -- allkeys-lfu**

The access pattern is heavily skewed. LFU preserves the frequently accessed
10 percent while evicting the long tail. With 200,000 slots and only 50,000
hot products, the hot set fits. LFU's Morris counter with logarithmic decay
ensures a suddenly popular product accumulates frequency quickly.

LRU would evict a product viewed 1,000 times yesterday but unviewed for 2
hours. LFU keeps it through low-traffic periods. allkeys-random would be
disastrous, evicting the best-seller with the same probability as a
discontinued item.

**Workload B -- allkeys-lru**

LRU keeps the most recently active sessions. Inactive sessions expire via TTL.
Evicting the least recently accessed correlates well with sessions inactive
the longest. volatile-ttl is also viable (all sessions have TTLs) but LRU is
safer because it relies on actual access rather than configured TTL values.
A session touched 5 minutes ago with a 24-hour TTL stays cached under LRU.
volatile-ttl might evict it prematurely.

LFU is wrong: a session with 100 requests yesterday would have high frequency,
blocking new sessions today.

**Workload C -- allkeys-lru**

News has recency-based access. LRU keeps the newest articles. As traffic bursts
fade and new articles arrive, old ones are evicted naturally.

LFU is wrong: a morning article with 10,000 views would occupy cache all day
despite being irrelevant by afternoon. volatile-ttl works if all articles have
TTLs matching the news cycle (for example, 4 hours).

### Challenge 1.3: Explaining Cache Stampede

Every 5 minutes, the database CPU spikes from 30 percent to 95 percent for
30 seconds, then returns to normal. The spike coincides with cache TTL expiry
for a set of popular keys.

**Questions:**

(a) Explain the phenomenon using a real-world analogy.

(b) Describe three mitigations ranked from simplest to most complex.

(c) For each mitigation, identify a downside or trade-off.

---

**Model Answer:**

(a) **The buffet line analogy.** A lunch buffet restocks dessert every 5
minutes. 500 people are waiting. When the fresh tray arrives, everyone rushes
the buffet simultaneously. The kitchen (your database) is overwhelmed serving
500 people at once. The same happens when a popular cache key expires: all
application servers miss simultaneously and all fetch from the database at
the same instant. That is a cache stampede.

(b) **Mitigation 1 -- TTL jitter (simplest).** Add random variation:

    ttl = 300 + random.randint(-30, 30)

With a 60-second spread, entries expire at different times. One-line code
change in the cache SET function.

**Mitigation 2 -- Request coalescing (moderate).** Only one server fetches
from DB on a miss. Others wait and then read from cache.

    SET lock:{key} server_id NX EX 5

First request acquires lock and fetches from DB. Others poll cache every 10ms
until data appears.

**Mitigation 3 -- Refresh-ahead (most complex).** Proactively refresh before
TTL expires.

    if entry.ttl_remaining < REFRESH_THRESHOLD:
        spawn_background_refresh(key)

Current request gets old value (no wait). Background task fetches fresh data
before entry expires.

(c) **TTL jitter trade-off:** Maximum staleness increases to 330 seconds
instead of 300. Does not protect against cache cold starts from restarts.

**Request coalescing trade-off:** Waiting requests experience +50ms latency
(the DB fetch time). Lock system can bottleneck if thousands of keys expire
simultaneously.

**Refresh-ahead trade-off:** Background refreshes consume DB capacity during
idle periods. Must ensure refreshes complete before TTL expiry.

### Challenge 2.1: E-Commerce Product Cache

**Context:** E-commerce platform with 10 million products, 5 million DAU,
50,000 requests per second at peak.

**Constraints:**

- Database reads take 80ms (5 joined queries).
- Price and inventory change in real time.
- Metadata changes weekly or less.
- 6-node Redis cluster (3M + 3R), 8 GB RAM per node.
- Target p95 read latency under 10ms.
- Budget $1,200 per month.

**Tasks:**

(1) Design the cache key structure with at least 4 patterns and TTLs.

(2) Justify eviction policy and memory allocation.

(3) Explain price and inventory invalidation. Happy path and failure case.

(4) Calculate memory for the hot set. 2 KB per entry, 80/20 rule, 30 percent
overhead.

(5) Describe mitigation for a flash sale causing 500,000 RPS on one product.

---

**Evaluation Rubric:**

| Criterion | Weak (1) | Adequate (2) | Strong (3) |
|-----------|----------|--------------|------------|
| Key structure | One key | Two keys | 4+ keys with distinct TTLs |
| TTL reasoning | Same for all | Different TTLs | TTLs tied to change frequency |
| Invalidation | TTL only | TTL + delete | TTL + delete + hot-path |
| Capacity | No numbers | Rough estimate | Concrete with overhead |
| Flash sale | Ignores it | Mentions hot key | Specific mitigation plan |

---

**Model Answer Walkthrough:**

**(1) Cache Key Structure:**

```
product:{id}:metadata          TTL: 3600s   1.2 KB
product:{id}:price             TTL: 30s     0.2 KB
product:{id}:inventory         TTL: 15s     0.15 KB
category:{id}:related:{pid}    TTL: 3600s   0.3 KB
flash_sale:{pid}:burst         TTL: 10s     2 KB (replicated)
```

Separate data by change frequency. Price and inventory have short TTLs (real-
time changes). Metadata and related have long TTLs (weekly changes). This
avoids unnecessary misses for slow-changing data.

**(2) Eviction Policy and Memory:**

Policy: allkeys-lfu. Top 10 percent of products (1M) get 90 percent of traffic.
LFU preserves popular products through low-traffic periods. LRU would evict a
product unviewed for 2 hours overnight, causing an 80ms DB query next morning.

Memory allocation from 24 GB total master memory:

```
Metadata:        4 GB
Price:           2 GB
Inventory:       1 GB
Related:         2 GB
Shared pool:    15 GB
```

If a flash sale needs extra space, LFU evicts lower-frequency entries.

**(3) Price and Inventory Invalidation:**

Happy path: Write handler updates DB, then `DEL product:{id}:price`. Next read
misses, fetches fresh price, repopulates. 30-second TTL is safety net.

For inventory, use CDC pipeline. Debezium captures WAL changes, publishes to
Kafka. Consumer deletes inventory key. 100-500ms latency, acceptable.

Do NOT use write-through. Adding 1-2ms cache write to 10,000 updates/second
makes the cache a write bottleneck. Invalidation on write is correct.

**(4) Capacity Calculation:**

Hot set: 20 percent of products (2M) get 80 percent of traffic.

Per product: 1.2 + 0.2 + 0.15 + 0.3 = 1.85 KB.

Hot set memory: 2,000,000 * 1.85 KB = 3,700,000 KB = 3.7 GB.

With 30 percent overhead: 3.7 * 1.3 = 4.81 GB.

Available: 3 masters * 8 GB * 0.75 = 18 GB.

Required: 4.81 GB. Available: 18 GB. Headroom: 13 GB (27 percent utilization).

**(5) Flash Sale Mitigation:**

A single product at 500,000 RPS saturates its Redis node (~150K GET/s max).

```
Step 1: Local cache on each of 100 app servers (2-second TTL).
        Absorbs 499,000 RPS. Only 100 RPS reach Redis.

Step 2: Dedicated key flash_sale:{pid}:burst replicated to all 3 masters.
        Reads pick a random replica.

Step 3: API gateway rate limiting at 200,000 RPS. Excess returns 429.

Step 4: Pre-warm cache 2 minutes before the sale.
        Run GET operations on all masters to populate cache.
```

### Challenge 2.2: Multi-Service Shared Cache

**Context:** Three services share one Redis cluster.

- **Service A (Search):** 200,000 QPS. 10 KB per entry. TTL: 2 min.
- **Service B (Profiles):** 150,000 reads/s, 500 writes/s. 500 B per entry.
  TTL: 5 min. Immediate invalidation on write.
- **Service C (Counters):** 50,000 INCR/s. 100 B per entry. No TTL.

Cluster: 6 nodes (3M + 3R), 16 GB per node.

**Tasks:**

(1) Design an isolation strategy. Two options and a recommendation.

(2) Calculate memory. Hot sets: Service A = 500K, B = 4M, C = 10M. 30 percent
overhead.

(3) Service A entries are 10 KB, 10x B and 100x C. How does size disparity
affect eviction? Design a strategy to prevent unfair eviction.

(4) Design an invalidation pipeline for Service B. Handle 500 writes/s and
5,000 writes/s bursts.

(5) List monitoring metrics with alert thresholds.

---

**Evaluation Rubric:**

| Criterion | Weak | Adequate | Strong |
|-----------|------|----------|--------|
| Isolation | No isolation | Separate DBs | Prefix isolation or separate instances |
| Memory | No numbers | Rough estimate | Concrete with overhead per service |
| Size disparity | Not addressed | Mentions problem | Separate pools per service |
| Invalidation | Delete each write | Batch deletes | Queue-based with deduplication |
| Monitoring | No metrics | Generic metrics | Specific metrics with thresholds |

---

**Model Answer Walkthrough:**

**(1) Isolation Strategy:**

Option A: Separate Redis databases (DB 0, 1, 2). Weak isolation: maxmemory is
cluster-wide. Service A's memory spike evicts B and C's even on different DBs.

Option B: Separate instances on dedicated nodes. Strongest isolation. Three
6-node clusters at $690/month each = $2,070/month.

Option C: Single cluster with prefix isolation: `search:*`, `profiles:*`,
`counters:*`. Monitor per-prefix memory via MEMORY USAGE. Alert when any
prefix exceeds its allocation. Zero additional cost.

Recommendation: Option C. If a service consistently exceeds its budget,
upgrade that service to Option B.

**(2) Capacity Calculation:**

```
Service A: 500,000 * 10 KB * 1.3 = 6.5 GB
Service B: 4,000,000 * 500 B * 1.3 = 2.6 GB
Service C: 10,000,000 * 100 B * 1.3 = 1.3 GB

Total: 6.5 + 2.6 + 1.3 = 10.4 GB

Available: 3 * 16 GB * 0.75 = 36 GB

Utilization: 10.4 / 36 = 29 percent. Headroom: 25.6 GB.
```

**(3) Size Disparity Handling:**

One search entry (10 KB) takes the memory of 20 profiles (500 B) or 100
counters (100 B). Under shared LRU, one large entry pushes out 100 small
entries even if the small entries are accessed more frequently.

Strategy: Run three Redis instances per node, each with its own maxmemory.

```
Node 1:
  Port 6379: Service A   8 GB  allkeys-lru
  Port 6380: Service B   4 GB  allkeys-lfu
  Port 6381: Service C   2 GB  allkeys-lfu
```

Each has independent eviction. Service A cannot evict Service C. Uses 14 GB
of 16 GB per node. Trade-off: 9 processes instead of 3 to manage.

**(4) Invalidation Pipeline:**

```python
# Write handler
def update_profile(user_id, data):
    db.execute("UPDATE profiles SET ... WHERE id = %s", user_id, data)
    redis.lpush("invalidation:queue", f"profile:{user_id}")

# Background worker (every 100ms)
def process_queue():
    msgs = redis.brpop("invalidation:queue", timeout=0, count=100)
    unique = set(msgs)             # Deduplication
    pipe = redis.pipeline()
    for key in unique:
        pipe.delete(key)
    pipe.execute()
```

Deduplication is critical at 5,000 writes/s. Without it, the same user_id
updated twice in 100ms generates two DELETEs. Reduces DELETE rate by 30-50%.

Max staleness: 100ms + 1ms = 101ms. Acceptable.

**(5) Monitoring Metrics:**

- Hit rate by prefix. Alert: below 80 percent for 5 consecutive minutes.
- Eviction rate by node. Alert: over 500/s sustained for 1 minute.
- Memory by prefix. Alert: exceeds allocation by 20%.
- Command latency p99. Alert: over 15ms for any operation.
- Connected clients. Alert: over 80 percent of maxclients.
- Network throughput. Alert: over 80 percent of capacity.

### Challenge 2.3: Real-Time Leaderboard Cache

**Context:** Mobile gaming platform with 20 million MAU, 500 games. Global and
friend leaderboards. Real-time score submissions.

**Constraints:**

- 100,000 score submissions/s at peak.
- 500,000 leaderboard reads/s at peak.
- Show top 100 players and requesting player's rank.
- Friend leaderboards: top 20 friends plus rank among friends.
- Scores must reflect within 2 seconds.
- Redis sorted sets available.
- Each game: up to 10 million entries.

**Tasks:**

(1) Design cache key structure. Why sorted sets?

(2) Describe the write path. How do you meet the 2-second requirement?

(3) Calculate memory. 60 bytes per member. 100 bytes skip list overhead.
500 games, 2 million active players each.

(4) Top 10 leaderboards get 80 percent of reads. Prevent node overload.

(5) Design the read path. How many operations? Expected latency?

---

**Evaluation Rubric:**

| Criterion | Weak | Adequate | Strong |
|-----------|------|----------|--------|
| Key structure | One key | Two keys | Tiered keys with different TTLs |
| Write path | ZADD only | ZADD + batch | ZADD + async friend update |
| Memory | No numbers | Rough estimate | Concrete with skip list overhead |
| Hot leaderboard | Not addressed | Mentions replication | Local cache + read replicas |
| Read path | Single ZREVRANGE | ZREVRANGE + ZRANK | Pipelined with cached snapshot |

---

**Model Answer Walkthrough:**

**(1) Cache Key Structure:**

```
leaderboard:global:{game_id}
    Sorted set. Members: "player:{id}", scores: integer.
    ZADD (O(log n)), ZREVRANGE (O(log n + M)), ZREVRANK (O(log n)).

leaderboard:friends:{game_id}:{player_id}
    Sorted set. Only friends who play this game. Materialized view.

leaderboard:top100:{game_id}
    Cached JSON string. Refreshed every 30s.

leaderboard:rank:{game_id}:{player_id}
    String. TTL: 10s. Converts ZREVRANK (50us) to GET (5us).
```

Sorted sets are correct because no other data structure provides O(log n)
insertion, rank query, and range retrieval simultaneously.

**(2) Write Path:**

```python
def submit_score(player_id, game_id, score):
    # Step 1: Validate
    if not validate_score(player_id, score):
        return error

    # Step 2: Update global leaderboard
    redis.zadd(f"leaderboard:global:{game_id}", {f"player:{player_id}": score})

    # Step 3: Publish event for async friend updates
    redis.xadd(f"score:updates:{game_id}", {
        "player_id": player_id,
        "score": score
    })

    # Step 4: Delete rank cache
    redis.delete(f"leaderboard:rank:{game_id}:{player_id}")
```

Step 2 completes in under 1ms. Score is immediately visible. Friend leaderboards
update within 500ms (async worker, one batch cycle).

Do NOT update friend leaderboards synchronously. 200 friends = 200 ZADDs per
submission. At 100K submissions/s: 20 million ZADD/s, far beyond capacity.

**(3) Memory Calculation:**

Global: 2M players * 160 bytes = 320 MB per game. 500 games: 160 GB.

Friend: 50 friends * 20 entries * 160 bytes = 3.2 KB per leaderboard.
Cache lazily with 1-hour TTL. Only 50K active per game.
50K * 3.2 KB = 160 MB per game. 500 games: 80 GB.

Total: 160 + 80 = 240 GB. With 30 percent overhead: 312 GB.
With replication: 624 GB. Nodes (32 GB, 24 GB usable): 26.

**(4) Hot Leaderboard Handling:**

Local cache on each app server for `leaderboard:top100:{game_id}` (5s TTL).
200 app servers turn 200,000 RPS into 2,000 RPS at Redis.

Read-replica distribution across all replicas.

Cached rank key with 30-second TTL. 90 percent of rank queries hit cache.

**(5) Read Path:**

```
Step 1: Local GET top100 snapshot          [0ms, 95% hit]
Step 2: Redis GET top100 snapshot          [2ms, 99% hit]
Step 3: If miss: ZREVRANGE 0 99 + cache   [50us]
Step 4: Local GET rank                     [0ms, 90% hit]
Step 5: Redis GET rank                     [2ms]
Step 6: If miss: ZREVRANK                  [50us]
Step 7: If rank > 99: ZREVRANGE rank-5..rank+5 [20-100us]
```

Expected latency: 2-5ms fully cached, 10-30ms cold. Cold reads under 1%.

## Tier 3: Expert Challenges

These challenges have no single correct answer. They test your ability to
navigate ambiguity, weigh trade-offs, and make justifiable design decisions
under uncertainty. Budget 30 to 45 minutes per challenge. The discussion
guides present multiple valid approaches -- your goal is to articulate why
your chosen approach is better for your specific context.

### Challenge 3.1: Global Cache with 200ms Write Propagation

You are the cache architect for a global social media platform with 500
million MAU. Data centers in US-East, EU-West, and AP-Southeast. Each region
has its own cache cluster.

A user in London comments on a photo by a user in Sydney. The comment must
appear within 5 seconds. The photo's cached view count must increment within
5 seconds. Analytics tolerates 5 minutes of delay.

Current design: 5-minute TTLs in each region. 15 percent of users complain
about stale cross-region data.

**Constraints:**

- Cross-region latency: US-EU: 65ms. US-AP: 120ms. EU-AP: 150ms.
- Each region: 10 nodes (5M + 5R), 32 GB per node.
- Budget: $50,000. Team: 4 engineers, 2 weeks.
- No globally distributed database available.

**Tasks:**

(1) Propose an architecture meeting the 5-second requirement for user content
while keeping the 5-minute bound for analytics. Describe data flow and
invalidation for each data type.

(2) Calculate latency impact. London user posts a comment. How much additional
latency for the writer? When does Sydney see it?

(3) Design an evolutionary path: what now (within budget and time), what later
when the requirement tightens to 30ms?

(4) Evaluate Redis Enterprise active-active geo-replication as an alternative.
Build vs. buy analysis considering cost, lock-in, and complexity.

(5) Two users in different regions like the same photo simultaneously.
Two conflict resolution strategies? Recommend one for Phase 1 and one for
Phase 2.

---

**Discussion Guide:**

**(1) Architecture:**

Primary-region (active-passive) architecture with an invalidation bus. US-East
is the write primary. All writes go to US-East, then propagate to EU-West and
AP-Southeast via Kafka.

For user content (comments, likes): write handler in London sends to US-East.
US-East updates primary cache and publishes invalidation event. Event propagates
via Kafka topic replication. EU-West consumers receive within 65ms, delete the
stale key, and optionally pre-populate the fresh value. Next read fetches fresh
data. Total propagation: 65-150ms, well within 5 seconds.

For analytics (view counts): write-behind pattern. Counts accumulate locally
as INCR operations. Every 5 minutes, a background job flushes aggregated counts
to the primary database. Avoids propagating millions of increments per second.

Key insight: not all data needs the same freshness. Separating user-facing
content (5-second requirement) from analytics (5-minute requirement) avoids
over-engineering the entire system.

**(2) Latency Calculation:**

London writer: 65ms (London to US-East) + 2ms (Redis write) + 1ms (Kafka
publish) = 68ms additional. Total POST response ~70ms, under 100ms perceived
instantaneity threshold.

Sydney reader: 68ms (London to US-East) + 120ms (Kafka AP-Southeast) + 2ms
(consumer) + 2ms (cache update) = ~192ms. Meets 5-second requirement by wide
margin.

**(3) Evolutionary Path:**

Phase 1 (now, $50K, 2 weeks): Primary-region invalidation bus. Deploy Kafka
brokers in each region (3 in US-East, 2 consumers in EU-West and AP-Southeast).
Cost: ~$20,000 for 7 Kafka instances. Engineering: 2 weeks. Achieves 5-second
requirement with 65-150ms propagation.

Phase 2 (6 months, 30ms requirement): Active-active CRDT system. Writes go to
nearest regional cache (2ms local), propagate asynchronously (500ms max).
Writer no longer waits 68ms for primary region. Requires Redis Enterprise or
custom CRDT. Budget: $100,000+.

**(4) Build vs. Buy (Redis Enterprise):**

Build:
+ Full control and understanding
+ Zero vendor lock-in
+ $20,000 one-time (within budget)
+ Works with standard Redis
- 2 weeks engineering
- Ongoing maintenance

Buy:
+ No pipeline to build
+ Automatic CRDT conflict resolution
+ Built-in monitoring
- $30,000-$50,000 per year (exceeds first-year budget)
- Vendor lock-in (black box reconciliation)
- Limited CRDT data type support

Recommendation: Build in Phase 1. Meets requirements, stays within budget,
builds team expertise. Evaluate Redis Enterprise for Phase 2.

**(5) Conflict Resolution:**

Strategy A (LWW): Timestamp wins. Simple but loses increments. Two simultaneous
likes at T=100.0 and T=100.1: only the later counted. 10 to 11 instead of 12.

Strategy B (CRDT Counter): Each region has counter vector. Total = sum. Both
increments preserved. 10 to 12 correctly. Requires CRDT support.

Strategy C (Deduplicated Event Log): Each like has UUID. Stored in set:
`likes:{photo_id}:events`. Sets unioned during reconciliation. Duplicates
ignored. Works with standard Redis. 8 MB per region for 100K likes/s.

Recommendation: Phase 1 uses Strategy C (no CRDT needed). Phase 2 uses Redis
Enterprise CRDT counters for automatic resolution.

### Challenge 3.2: Cache or Not to Cache

Staff engineer for a weather data API startup. 25 million requests/day. Single
Postgres database.

Query times: current weather 15ms, forecast 50ms, historical 200-500ms.

CTO wants Redis: estimates 80 percent DB load reduction and 50 percent latency
improvement. Budget: $3,000/month.

Traffic analysis:

| Endpoint | Req/day | Size | Cacheability | Updates |
|----------|---------|------|-------------|---------|
| Current | 15M | 4 KB | High | Every 15 min |
| Forecast | 7M | 30 KB | Medium | Every hour |
| Historical | 3M | 100KB-1MB | Low | Never |
| Alerts | 500K | 2 KB | High | Few/day |

Unique locations/day: 500K (current), 200K (forecast), 50K (historical).

**Tasks:**

(1) Analyze the CTO's estimates. Calculate hit rate and latency for each
endpoint.

(2) Recommend which endpoints to cache and which not to. Justify each.

(3) Design cache keys and TTLs for cached endpoints. Propose alternatives
for uncached endpoints.

(4) Calculate whether $3,000/month is sufficient. ElastiCache pricing:
r6g.large (13 GB) = $115/mo, r6g.xlarge (26 GB) = $230/mo.

(5) CTO insists on caching everything. Argue against caching historical data
with specific numbers and a better alternative.

(6) After deployment, current weather hit rate is below expectations. 40
percent of requests use new coordinates. Propose two strategies with trade-offs.

---

**Discussion Guide:**

**(1) Hit Rate and Latency Analysis:**

Current weather: 15M req/day, 500K locations. 30 req/location/day.
Hit rate: 29/30 = 96.7 percent. DB queries saved: 14.5M/day.
Latency: without=15ms, with=(0.967*3)+(0.033*15)=3.4ms. 4.4x improvement.

Forecast: 7M req/day, 200K locations. 35 req/location/day.
Hit rate: 34/35 = 97.1 percent. DB queries saved: 6.8M/day.
Latency: without=50ms, with=(0.971*3)+(0.029*50)=4.35ms. 11.5x improvement.

Historical: 3M req/day, 50K locations. 60 req/location/day. 98.3 percent hit
rate. But 50K * 500KB = 25 GB + 30 percent = 32.5 GB. Dominates cache budget.

Overall DB reduction: (0.60*0.967)+(0.28*0.971) = 85.2 percent. Exceeds 80%.

Overall latency: Without=(0.60*15)+(0.28*50)+(0.12*350)=65ms. With=(0.60*3.4)+

Historical queries dominate the average.

**(2) Cache Recommendations:**

Cache current weather. Best candidate: 96.7 percent hit rate, 4 KB, 2.6 GB
with overhead. Reduces 15M DB queries to 500K.

Cache forecast. 97.1 percent. Compress with zstd to 10 KB. 2.6 GB with
overhead.

Cache alerts. 2 KB each. Negligible memory. Critical latency requirement.

Do NOT cache historical. 32.5 GB for hot set. Immutable data: database buffer
cache with proper indexing serves it efficiently.

**(3) Cache Keys:**

```
weather:current:{lat}:{lon}       TTL: 900s
weather:forecast:{lat}:{lon}      TTL: 3600s (zstd compressed)
weather:alert:{alert_id}          No TTL, explicit invalidation
```

Alternative for historical: Materialized table in Postgres with unique
constraint on (lat, lon, start_date, end_date). First: 200-500ms (compute +
store). Subsequent: 2-5ms (indexed read). Zero infrastructure cost.

**(5) Argument Against Historical Data:**

50,000 queries * 500 KB avg = 25 GB. With 30 percent overhead: 32.5 GB.
With replication: 65 GB.

r6g.xlarge: 65 / (26 * 0.75) = 4 nodes. 4 * $230 = $920/month just for
historical. Plus growth: 50K new entries per day.

Hit rate: 98.3 percent. Only 1.7 percent (50K/day) are first-time queries.
DB serves them at 200-500ms. Total time saved: 50,000 * 350ms * 30 = 146
hours/month. DB runs at 25 percent CPU. Savings negligible.

Better alternative: Materialized table in Postgres. Zero cost. Same 2-5ms
for repeats. First-query penalty acceptable for analytics dashboards. Saves
$920/month.

**(6) Improving Hit Rate:**

40 percent new coordinates reduce effective hit rate to 58 percent.

Strategy A: Round coordinates to 0.1-degree grid. Unique locations drop from
500K to ~50K. Users at nearby coordinates share cache entries. Trade-off:
10 km grid cell accuracy. Acceptable for general weather, not for aviation.

Strategy B: Pre-warm from 10,000 NOAA weather stations. Fetch all stations
every 15 minutes. Covers first requests within 20 km of any station. 0.1 CPU
cost. Coverage gap: remote locations under 5 percent.

Strategy C: Predictive pre-fetching. Track popular geohash prefixes. Pre-fetch
top 1,000 cells. Complex to implement but covers 80 percent of new coordinates.

Recommendation: Implement grid rounding (58% to 85%) and station pre-warming
(85% to 92%). Remaining 8 percent: rare locations. DB handles 20K additional
queries/day at 15ms = 300 seconds DB time, negligible.
