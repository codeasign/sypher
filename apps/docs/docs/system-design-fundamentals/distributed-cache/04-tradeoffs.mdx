## Core Tension

The central tension in distributed caching is the trade-off between speed and correctness. A cache exists to be fast: serving data from RAM in microseconds rather than from disk in milliseconds. But speed demands compromise. Storing a copy of the data means the cache can diverge from the source of truth. The faster you want reads, the more you must tolerate staleness, complexity, or cost.

This tension manifests in three dimensions:

**Staleness vs. Latency**: A cache with a 1-second TTL is very fresh but serves from cache only if a request arrived in the last second. Hit rate is low, origin load is high. A cache with a 1-hour TTL has a high hit rate and low latency but can serve data that is an hour out of date. Every caching decision picks a point on this curve.

**Consistency vs. Availability**: A write-through cache is consistent but less available: if the cache node is unreachable, the write fails. A cache-aside setup with TTL is highly available: you can always hit the origin if the cache is down. But between a write and the TTL expiry, the cache is stale.

**Memory vs. Engineering Cost**: Adding more cache nodes improves hit rate and reduces database load. But each node costs money and adds operational complexity. At some point, the marginal hit rate improvement from one more node does not justify the cost.

## CAP Theorem and Caching

The CAP theorem states that a distributed data system can provide at most two of Consistency, Availability, and Partition Tolerance. Distributed caches consistently choose AP: Availability and Partition Tolerance over strong Consistency.

**Partition**: When the network link between the cache and database breaks, a cache-aside system remains available: it serves cached data (possibly stale) and accepts writes that will be reconciled later. A write-through system becomes unavailable because it cannot complete the synchronous write to both cache and database.

**Consistency trade-off**: The cache guarantees eventual consistency within the TTL window. After a write to the database, the cache transitions through a stale state until the TTL expires or an invalidation event arrives.

## Failure Modes

### Failure Mode 1: Cache Stampede (Thundering Herd)

**Scenario**: A hot key expires simultaneously. 10,000 application servers detect the miss and all attempt to recompute the value from the database. The database, which comfortably handles 5,000 queries per second, receives 10,000 in under 100 milliseconds. Connection pool saturates. Query queue fills. Database CPU spikes to 100 percent. Latency climbs from 20ms to 5 seconds.

**Impact**: 
- Database overload: 5 to 10 minutes of degraded performance
- Cascading failures: other services sharing the database are impacted
- Cache warming: even after the database recovers, the cache must repopulate, prolonging the period of low hit rate

**Root cause**: The hot key had a uniform TTL across all servers. When the TTL expired, all servers missed simultaneously. No request coalescing was in place.

**Mitigation**:
- Add jitter to TTLs: `ttl = base_ttl + random(-jitter, +jitter)`. This spreads expiration times.
- Implement request coalescing: one thread fetches, others wait.
- Use refresh-ahead: proactively refresh before expiry for known hot keys.
- Use a local cache in front of the distributed cache for the hottest keys.

### Failure Mode 2: Hot Key Overload

**Scenario**: A celebrity tweet goes viral. The tweet key `tweet:84213` accounts for 80 percent of all cache reads. The cache node holding this key handles 400,000 requests per second. Its CPU is pinned at 100 percent. Other requests to the same node experience 500ms latency. The node sends high-latency signals, and the cluster marks it as failing.

**Impact**:
- Single node saturated
- Collateral damage: all keys on the same node experience high latency
- Potential cascading failover if the node is marked as down

**Root cause**: Consistent hashing placed the hot key on a single physical node. No local caching on application servers. No read replicas for the hot key.

**Mitigation**:
- Local cache: application servers cache the hot key in memory with a short TTL (2 seconds). This reduces distributed cache load from 400,000 RPS to 400 RPS (one per application server).
- Hot key replication: store the hot key on multiple cache nodes. Reads pick a random replica.
- Virtual nodes: ensure virtual node count is high enough to spread load statistically.
- Rate limiting: cap requests to the hot key at the application layer.

### Failure Mode 3: Cache Penetration

**Scenario**: An attacker sends requests for non-existent product IDs: `product:-1`, `product:99999999`, `product:ABCD`. None of these keys exist in the database. Each request causes a cache miss (value not in cache because it was never stored), then a database miss (row does not exist), and does not populate the cache. Every request hits the database.

**Impact**:
- Effective bypass of the entire cache layer
- Database load increases 100x for the targeted keys
- Attack can be low-bandwidth: 100 requests per second to 100 non-existent keys generates 10,000 database queries per second

**Root cause**: The cache does not store negative results. The application does not differentiate between "key not cached" and "key does not exist."

**Mitigation**:
- Cache null results: store a sentinel value for non-existent keys with a short TTL (30 to 60 seconds).
- Bloom filter: maintain a Bloom filter of all valid product IDs. Before checking the cache or database, verify the key might exist. Bloom filters use approximately 10 MB for 10 million items with a 1 percent false positive rate.
- Parameter validation: reject clearly invalid keys (negative IDs, non-numeric formats) at the API gateway.

```python
def get_product(product_id):
    if not is_valid_id(product_id):
        return None  # Reject at application layer

    # Bloom filter check
    if not bloom_filter.might_contain(f"product:{product_id}"):
        return None  # Definitely does not exist

    cache_key = f"product:{product_id}"
    result = cache.get(cache_key)
    if result is not None:
        return result if result != NULL_SENTINEL else None

    product = db.query("SELECT * FROM products WHERE id = %s", product_id)
    if product is None:
        cache.set(cache_key, NULL_SENTINEL, ex=60)
        return None

    cache.set(cache_key, serialize(product), ex=3600)
    return product
```

### Failure Mode 4: Replication Lag and Stale Reads

**Scenario**: A user updates their profile picture. The write goes to the database but the cache invalidation event is delayed by 2 seconds due to a CDC pipeline backlog. Meanwhile, the user refreshes the page and sees the old picture.

**Impact**:
- Poor user experience: users do not see their own changes
- Data inconsistency: other users see stale data for longer than expected
- Write conflicts: stale read might be followed by a write based on stale data

**Root cause**: Asynchronous replication or CDC pipeline has inherent latency. Under high write volume, the backlog grows.

**Mitigation**:
- Write-through: accept the latency penalty for user-specific data.
- Read-your-write consistency: let the user bypass the cache for their own recent writes. Use a write timestamp or session flag.
- Invalidation at write time: the write handler explicitly deletes the cache key synchronously, not via CDC.

### Failure Mode 5: Memory Exhaustion and OOM

**Scenario**: A configuration error sets `maxmemory` to 0 (unlimited) on a 32 GB cache node. The cache stores 30 GB of data. A traffic spike causes the node to store 5 GB more. The kernel OOM killer terminates the Redis process. The replica promotes, but the new master also has high memory. Both nodes crash-loop.

**Impact**:
- Complete loss of cached data for that shard
- Database receives full read load for all keys in that shard
- Crash-loop prevents stable recovery

**Root cause**: No memory limit. No swap limit in container. No memory monitoring and alerting.

**Mitigation**:
- Always set `maxmemory` to 70 to 80 percent of available RAM.
- Set container memory limits equal to `maxmemory` plus 20 percent overhead.
- Monitor used memory and eviction rate. Alert when memory exceeds 75 percent of maxmemory.
- Configure swap for emergency headroom (but degrade gracefully with `vm.swappiness=1`).

## Anti-Patterns

### Anti-Pattern 1: Cache-as-Database

Storing critical, non-recoverable data exclusively in the cache. The cache is not designed for durability. When the node restarts, the data is gone.

**Why it fails**: Cache nodes can fail, restart, or be evicted. Data that cannot be regenerated must live in a database with ACID properties.

**Fix**: Always have a durable source of truth. The cache is an accelerator, not a storage system. If you need durable key-value storage, use a database like DynamoDB, Cassandra, or FoundationDB.

### Anti-Pattern 2: One TTL to Rule Them All

Using the same TTL for all cached data regardless of access pattern or freshness requirements.

**Why it fails**: Hot keys expire simultaneously, causing stampedes. Infrequently accessed keys waste memory. Time-sensitive data (stock prices) becomes stale too quickly; reference data (country codes) is refreshed too often.

**Fix**: Set TTLs per data category:
- Reference data (country codes, tax rates): 24 hours or no TTL with explicit invalidation
- User profiles: 5 minutes
- Session data: 24 hours with sliding expiry
- Real-time data (stock prices, leaderboards): 30 seconds or TTL matching update frequency

### Anti-Pattern 3: Caching Everything

Caching every piece of data the application touches, including data that is rarely read, is cheap to compute, or changes frequently.

**Why it fails**: Wasted memory. Each cache entry consumes memory plus overhead. Data that is written more often than read has a negative caching ROI. Data that is cheap to compute (simple aggregation of a few rows) adds more complexity than it saves.

**Fix**: Cache only data that has a high read-to-write ratio (over 10:1), is expensive to compute (multi-query joins, API calls to external services), or is accessed many times in a short window.

### Anti-Pattern 4: Ignoring Cache Warming

Starting a new cluster or node and expecting full performance immediately. The cache is cold: every request misses until the cache fills.

**Why it fails**: A cold cache means 100 percent miss rate for the first 5 to 30 minutes. All requests hit the database. If the database cannot handle the full read load, the application fails.

**Fix**: Warm the cache before directing traffic to it. Export data from a live cache node and load it into the new node. For Redis, use `RDB` file transfer or replication from a live node. Pre-compute hot keys in a preparation script.

### Anti-Pattern 5: Over-Eviction

Setting `maxmemory` too low, causing aggressive eviction that removes entries moments after they are stored.

**Why it fails**: The cache becomes a pass-through. Every entry is evicted before it can serve a hit. Memory is wasted on the overhead of storing and evicting entries that never deliver value.

**Fix**: Monitor `evicted_keys` in Redis `INFO stats`. If evicted_keys per second exceeds 1 percent of total keys, increase `maxmemory` or scale out. Use `allkeys-lru` to at least keep popular entries.

### Anti-Pattern 6: Shared Cache with Noisy Neighbors

Using the same cache cluster for multiple services or features without isolation. One service's traffic spike evicts another service's cached data.

**Why it fails**: The search service caches 10 GB of index data. The recommendation service suddenly needs 8 GB for a new feature. Evictions spike, and the search service hit rate drops from 95 percent to 60 percent. Database load doubles.

**Fix**: Use separate cache instances or separate logical databases per service. In Redis, use different DB numbers (0-15) or different key prefixes with `maxmemory-policy` adjusted per DB. For stronger isolation, use separate clusters.

### Anti-Pattern 7: Long TTL for Mutable Data

Setting a multi-hour TTL on data that users can modify through the application.

**Why it fails**: Users update their data and do not see the change for hours. Customer support tickets spike. Trust in the application erodes.

**Fix**: For mutable data, either invalidate the cache on every write (use event-driven approach) or keep TTLs under 5 minutes and explain staleness in the UI. For user-facing data, strongly prefer invalidation.

## When NOT to Use

A distributed cache is not the right tool for every problem. Recognize these situations.

**When data changes faster than it is read**: If every piece of data is written once and read once, the cache introduces overhead without benefit. A real-time analytics pipeline that processes 1 million events per second and reads each event zero times after processing should not use a distributed cache. Use a stream processor instead.

**When consistency requirements are absolute**: If the application cannot tolerate any stale data, a cache adds risk. Financial trading systems that need to show the exact account balance should not cache the balance. Use database reads with appropriate isolation levels.

**When data size exceeds available memory by 100x**: If you have 10 TB of data and budget for 100 GB of cache, the hit rate will be low for most access patterns unless access is extremely skewed. A cache serving 1 percent of your dataset cannot deliver a high hit rate. Consider alternative patterns: database read replicas, materialized views, or data warehouse pre-aggregation.

**When the application has a single server**: A distributed cache adds network latency, serialization, and operational complexity. If you have a single application server, an in-process cache or a local Redis instance on the same machine is simpler and faster. Only use a distributed cache when you have two or more application servers that need a shared view of cached data.

**When you cannot tolerate cache infrastructure costs**: A production-grade Redis cluster with 3 masters and 3 replicas on AWS costs approximately $500 to $1,500 per month depending on instance size. For a small application with moderate traffic, this cost might exceed the benefit. Evaluate the break-even point: how many database replicas would you need to replace the cache?

**When data access follows no pattern**: If every request is for a different, random key, the cache hit rate will be near zero. A dictionary serving random words from a 500,000-word vocabulary will have a hit rate of 0.02 percent per user. Caching serves locality of reference. Without it, the cache is wasted.

**Evaluation checklist** before adopting a distributed cache:

| Question | If Yes | If No |
|----------|--------|-------|
| Read-to-write ratio over 5:1? | Consider caching | Probably not |
| Stale data acceptable up to TTL? | Cache is viable | Use write-through or skip |
| Can you afford cache infrastructure? | Proceed | Use database optimizations first |
| Do you have multiple application servers? | Distributed cache helps | Use local cache |
| Is data access skewed (80/20 rule)? | High hit rate expected | Low hit rate expected |
| Is data size small enough to fit in memory? | Good candidate | Consider read replicas |
