## Naive Implementation

### Version 1: Single Reverse Proxy (Good for 1,000 Users)

The simplest possible CDN is a single Nginx server configured as a reverse proxy cache. It sits in front of your origin server and caches responses on disk.

```nginx
# /etc/nginx/sites-available/cdn.conf
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=cdn_cache:10m
                 max_size=10g inactive=60m use_temp_path=off;

server {
    listen 80;
    server_name cdn.example.com;

    location / {
        proxy_cache cdn_cache;
        proxy_cache_key "$scheme$request_method$host$request_uri";
        proxy_cache_valid 200 302 60m;
        proxy_cache_valid 404 1m;
        proxy_pass http://origin-server:8080;
        add_header X-Cache-Status $upstream_cache_status;
    }
}
```

**What works:** HTTP responses get cached. Repeated requests for the same URL return from disk. The `X-Cache-Status` header tells you HIT or MISS.

**What breaks at 1,000 users:** Nothing. This setup handles 1,000 concurrent users comfortably with 10GB of disk cache.

### What Breaks at 100,000 Users

At 100,000 concurrent users, five problems emerge simultaneously:

1. **Single point of failure** — The Nginx server crashes. All traffic stops. No redundancy.
2. **Cache stampede** — When the cache expires for a popular URL, all 100,000 users miss simultaneously. Each sends a request to the origin, which dies under the load.
3. **TLS bottleneck** — Nginx performs TLS termination on a single CPU core. 100,000 connections exceed its capacity. Handshake latency climbs to 10 seconds.
4. **Disk I/O saturation** — 100,000 requests per second hitting the same SSD causes read queuing. p99 latency goes from 2ms to 500ms.
5. **Single geographic location** — Users in other continents still experience 200-600ms latency.

### What Breaks at 10 Million Users

All the 100K problems magnify, plus new ones:

1. **Global latency** — A single server in us-east-1 gives Australian users 200ms RTT minimum. Your global user base sees median latency of 180ms.
2. **Origin saturation** — Even with 99 percent cache hit ratio, 10M users × 10 requests/user/day = 100M requests. 1 percent misses = 1M requests/day to origin = 12 requests/second sustained. But peak traffic is 10x average, so 120 req/s to origin. One server can handle this. The real problem is flash crowds: a viral post spikes to 100K req/s instantly.
3. **Cache capacity** — 10M users × 500KB average asset size × 100 assets each = 5TB of unique content. A single 10GB cache evicts everything constantly. Hit ratio drops to 20 percent.
4. **DNS resolution** — Your single IP address cannot route users to different servers by location. Every user hits the same server regardless of where they are.

## Integration Patterns

### Pattern 1: CDN + Database (Read-Heavy Workload)

A CDN can cache database query results, dramatically reducing database read load. This works best for data that changes infrequently and is read many times.

### Cross-Region Replication for Cache Warming

For predictable content (video uploads, scheduled publications), proactively replicate content across all regions before users request it.

```python
# Simplified cache warming — pre-fetch content across all regions
REGIONS = {
    "us-east": "https://nyc-cdn.example.com",
    "eu-west": "https://fra-cdn.example.com",
    "ap-southeast": "https://sin-cdn.example.com",
    "sa-east": "https://gru-cdn.example.com",
}

def warm_cache(content_paths, regions=REGIONS):
    for path in content_paths:
        for region, pop_url in regions.items():
            url = f"{pop_url}{path}"
            response = requests.get(url, headers={"X-Warmup": "true"})
            if response.status_code == 200:
                log(f"Warmed {region}: {path}")
            else:
                log(f"Warmup failed for {region}: {path}")
```

Pre-warming reduces p95 latency during launches from 400ms (cold cache) to 20ms (warm cache).

### Failover and Disaster Recovery

| Scenario | Impact | Failover Mechanism | RTO | RPO |
|----------|--------|-------------------|-----|-----|
| Single edge server failure | 1/N capacity loss | LB routes around failed server | 1s | N/A |
| Single POP failure | Regional capacity loss | Anycast withdraws BGP route, traffic shifts to nearest POP | 60s | N/A |
| Regional cache loss | Increased origin load | Traffic goes directly to origin shield | 1s | N/A |
| Origin shield failure | All edges lose shield | Bypass shield, edges talk directly to origin | 1s | N/A |
| Origin cluster failure | New content cannot be served | Serve stale cached content | ~0s | TTL remaining |
| Control plane failure | No purges, no config changes | Read-only mode, cached configs continue | ~0s | N/A |

## Capacity Planning

### Math for 10 Million Daily Active Users

**Assumptions:**
- 10M DAU
- 50 requests per user per day (page loads, asset fetches)
- 500KB average response size (HTML + CSS + JS + images)
- 50 percent of requests are cacheable static assets
- Target cache hit ratio: 95 percent

**Traffic calculations:**

```
Total requests per day: 10,000,000 × 50 = 500,000,000
Cacheable requests per day: 500,000,000 × 0.50 = 250,000,000
Cache hits per day: 250,000,000 × 0.95 = 237,500,000
Cache misses per day: 250,000,000 × 0.05 = 12,500,000

Peak requests per second (10x average):
  Average RPS: 500,000,000 / 86,400 = 5,787 RPS
  Peak RPS: 57,870 RPS

Cacheable peak RPS: 57,870 × 0.50 = 28,935 RPS
  Hits: 28,935 × 0.95 = 27,488 RPS
  Misses: 28,935 × 0.05 = 1,447 RPS

Bandwidth:
  Peak egress: 28,935 RPS × 500KB = 14.5 GB/s = 116 Gbps
  Per POP (assume 20 POPs share load evenly): 116 / 20 = 5.8 Gbps per POP
    → Comfortable for a 10Gbps uplink (58% utilization)
    → Tight for a 1Gbps uplink (would saturate at 580%)

Storage (cache capacity):
  Unique content set: 10M users × 100 assets each = 1B objects
  Total size: 1B × 500KB = 500TB
  With 95% requests hitting 5% of content (Zipf distribution):
    Hot set: 500TB × 0.05 = 25TB needs to fit in cache
    Per POP: 25TB / 20 POPs = 1.25TB per POP
    → 4 × 3.84TB NVMe drives per server (15TB raw, 7.5TB usable RAID-10)
    → 4 edge servers per POP = 30TB usable per POP

Edge servers needed:
  Requests per edge server: 28,935 RPS peak / 20 POPs = 1,447 RPS per POP
  Each edge server handles ~5,000 RPS comfortably
  Servers per POP: max(4, ceil(1,447 / 5,000)) = 4 servers per POP
  Total servers: 20 POPs × 4 = 80 edge servers

Origin cluster:
  Cache misses per second (peak): 1,447 RPS to origin
  Each origin server handles 500 RPS
  Origin servers needed: ceil(1,447 / 500) = 3 servers
  Plus 1 for redundancy: 4 origin servers
```

**Cost estimate (monthly, approximate):**

| Component | Qty | Unit Cost | Monthly |
|-----------|-----|-----------|---------|
| Edge servers (POPs) | 80 | $500/mo | $40,000 |
| Bandwidth (116 Gbps × 0.5 utilization) | 50 Tbps-mo | $0.01/GB | $150,000 |
| Load balancers (2 per POP) | 40 | $200/mo | $8,000 |
| Origin servers | 4 | $1,000/mo | $4,000 |
| Object storage (origin) | 100TB | $0.023/GB | $2,300 |
| DNS (anycast) | 1 | $1,000/mo | $1,000 |
| Colocation (20 POPs) | 20 | $3,000/mo | $60,000 |
| **Total** | | | **$265,300/mo** |

This is why public CDNs (Cloudflare, Akamai, Fastly) charge $500-$5,000/month for their service — they amortize infrastructure across thousands of customers. Building your own CDN makes sense only above 100 Gbps sustained throughput.

## Deployment and Operations

### Zero-Downtime Deployment

Deploying a new version of the CDN software must not interrupt active connections or empty the cache.

```python
# Rolling deployment procedure for a POP
def deploy_rollout(pop_id, version):
    # 1. Mark server as draining (stop sending new connections)
    lb_api.set_weight(pop_id, "edge-1", 0)  # Stops new traffic
    
    # 2. Wait for in-flight requests to complete (max 60s)
    wait_for_active_connections("edge-1", timeout=60)
    
    # 3. Deploy new version
    deploy_software("edge-1", version)
    
    # 4. Warm the cache (pre-fetch top 1000 URLs from healthy servers)
    warm_cache_from_sibling("edge-1")
    
    # 5. Health check (10 requests, expect under 50ms p50)
    if health_check("edge-1"):
        lb_api.set_weight(pop_id, "edge-1", 100)  # Restore traffic
    else:
        rollback("edge-1", previous_version)
    
    # 6. Move to next server, stagger 30s apart
```

Key principle: never deploy all servers in a POP simultaneously. Deploy one at a time, verify health, then proceed. Total deployment time for a 4-server POP: 3 minutes.

| Metric | Alert Threshold | Why |
|--------|----------------|-----|
| Cache hit ratio | Below 85% for 5 minutes | Either TTLs too short or cache keys too granular |
| p50 latency | Above 20ms for 5 minutes | Server under load or upstream slow |
| p99 latency | Above 200ms for 5 minutes | Upstream timeout or resource contention |
| Origin load | Above 80% of capacity | CDN becoming ineffective, scale origin |
| Error rate (5xx) | Above 0.1% for 1 minute | Origin or cache failure |
| Stale served rate | Above 0.5% for 5 minutes | Origin reliability issue or stale config |
| Purge completion time | Above 30s for any purge | Control plane or network issue |
| POP CPU utilization | Above 80% for 10 minutes | Scale out, add servers |
| POP disk usage | Above 70% for 1 hour | Eviction not keeping up, add capacity |
| TLS handshake time | Above 100ms p50 | SSL optimization needed or certificate issue |

### Debugging Common Problems

**Symptom: Cache hit ratio suddenly drops from 95% to 40%**

Check: Did a deployment change cache key templates? Did query parameters change? Is a new version of the app embedding random tokens in URLs? Most common cause: a developer added a UUID to CSS URLs for cache busting without versioning the path.

Fix: Normalize cache keys to strip the UUID parameter. Add the version to the path instead.

**Symptom: Users in Asia report errors but US users are fine**

Check: Is the Asia POP reachable? Did a submarine cable get cut? Are Asian edge servers at capacity? Run `curl` from a server in Singapore to the origin.

Fix: If cable cut, route traffic to a different POP (US West or Australia). If capacity, add servers to the POP.

**Symptom: Origin CPU spikes at the top of every hour**

Check: Are multiple TTLs set to exactly 3600 seconds? If objects were cached at the top of the hour, they all expire at the top of the next hour, causing a thundering herd of revalidation requests.

Fix: Add jitter to TTLs. Set `s-maxage=3500+rand(200)` so expiry times are spread across the hour window.

**Symptom: CDN serves wrong content variant (e.g., desktop site to mobile)**

Check: Is the cache key template including the `User-Agent` header? Is it including `Accept` headers? If two different devices map to the same cache key but expect different content, the first request's response gets served to everyone.

Fix: Include the device classification (not the full User-Agent) in the cache key. Normalize the User-Agent header to a category value (mobile, desktop, tablet).

### Upgrades and Maintenance

CDN software upgrades follow a strict schedule:

1. **Control plane first:** Upgrade the purge service, config store, and monitoring. These are internal-facing and do not affect user traffic.
2. **POP edge servers:** Roll through one server at a time per POP. Verify cache hit ratio does not drop after each upgrade.
3. **Origin shield:** Upgrade during low traffic (3 AM local time in the shield's region).
4. **Origin cluster:** Standard blue-green deployment behind a load balancer.

Each upgrade includes a rollback plan. The rollback command must be tested during the upgrade dry run, not discovered during the actual incident.
