## The Problem Story

Imagine you run a global e-commerce platform based in New York. Your product catalog images, CSS files, and JavaScript bundles all live on a single web server in a data center just outside Manhattan. A customer in Tokyo opens your homepage. Her browser sends a request that crosses the Pacific Ocean, traverses a dozen internet exchange points, and finally reaches your New York server 280 milliseconds later. Your server assembles the HTML, sends it back across the ocean, and another 280 milliseconds pass before the page starts rendering. That is over half a second of pure network latency before any content even appears.

Now multiply this by thousands of simultaneous users spread across six continents. Every image, every script, every font file makes the same round trip. The Tokyo user experiences 560ms latency on every resource. The Sydney user sees over 600ms. The server in New York, meanwhile, handles ten thousand concurrent connections for the same static assets over and over. CPU usage climbs, network bandwidth saturates, and the whole site slows down for everyone — including customers in New Jersey who are physically 20 kilometers from the data center. A 100-millisecond delay in page load reduces conversion rates by 7 percent at this scale. At 600 milliseconds, you are bleeding revenue from every market outside North America.

The root cause is not server performance. The server can generate a response in 20 milliseconds. The problem is distance. Light in fiber optic cable travels at roughly two-thirds the speed of light in a vacuum — about 200,000 kilometers per second. The circumference of Earth is 40,000 kilometers. A signal literally cannot circle the planet faster than 200 milliseconds even under ideal physics. Geography is the bottleneck that no amount of server-side optimization can fix.

## The Insight

A Content Delivery Network (CDN) is a geographically distributed network of proxy servers that caches content closer to end users, so the network traverses fewer miles per request.

Think of a central post office system. If every letter sent from any city had to first go to a single national sorting facility in one central location, a letter from San Francisco to Los Angeles would travel through Chicago first. That is how the internet works without a CDN — every request routes to the origin server regardless of where the user sits. A CDN is like having regional post offices in every major city. A letter dropped in San Francisco bound for Los Angeles goes to the San Francisco regional office, which forwards it directly to the Los Angeles regional office, which delivers it locally. The central facility exists but only handles letters that no regional office can process. Your mail arrives faster because it travels fewer hops over shorter distances.

## Core Vocabulary

**Origin server** — The definitive source of truth for your content. Every file originates here. The CDN fetches from the origin when no cached copy exists. If the origin goes down, cached content still serves but nothing new can be cached.

**Edge server (POP — Point of Presence)** — A server in a CDN data center located close to end users. Edge servers store cached content and serve user requests. A global CDN operates hundreds or thousands of POPs across dozens of countries.

**Cache hit** — A request that finds its content already stored on the edge server. Cache hits return in 10-50 milliseconds because no upstream fetch is needed. High cache hit ratio (90 percent or above) is the primary CDN performance metric.

**Cache miss** — A request where the edge server does not have the content. The edge must fetch from the origin server or an upstream layer, adding one full round trip of latency. Cache misses dominate the p99 latency tail.

**Cache key** — The identifier used to look up a cached object, typically the full URL including query parameters. Variations in cache keys determine whether a cached object serves or a new request goes to origin.

**Time to Live (TTL)** — The duration a cached object remains valid before the edge server must revalidate with the origin. Short TTLs (seconds) ensure freshness for dynamic content. Long TTLs (days) maximize cache hits for static assets.

**Origin shield (intermediate layer)** — An intermediate cache layer that sits between edge servers and the origin server. When multiple edges miss simultaneously (a flash crowd), the shield deduplicates requests so the origin sees only one fetch instead of hundreds.

**Purge / invalidation** — Removing cached content from edge servers before its TTL expires. Used when content changes unexpectedly (a price update, a corrected image). Purge operations must propagate to every POP globally, which takes seconds to minutes depending on CDN architecture.

**Regional cache (parent layer)** — A larger cache that serves a geographic region (for example, all of Western Europe). Edge servers in that region fetch from the regional cache before reaching the origin. This adds a second caching layer that dramatically reduces origin load.

**Anycast** — A routing technique where the same IP address is announced from multiple locations worldwide. Traffic automatically routes to the nearest POP. Anycast is how a single DNS resolution maps a user to their closest edge server without the client knowing about the geography.

**Stale-while-revalidate** — A serving strategy where the edge returns a cached object that has expired (stale) while simultaneously fetching a fresh copy in the background. Users see zero latency; the next request gets the new version. This masks cache miss latency at the cost of momentarily serving slightly outdated content.

**Cache control headers** — HTTP headers (`Cache-Control`, `Expires`, `ETag`, `Last-Modified`) that the origin server sends to tell the CDN what to cache, for how long, and how to validate freshness. These headers are the contract between origin and CDN.

## How It Actually Works — Step by Step

A user in Berlin visits your website at `https://www.example.com/index.html`. The user's browser performs a DNS lookup on `www.example.com`. Because your DNS provider uses a CDN, the DNS response returns an anycast IP address that routes to the nearest CDN POP — in this case, the Frankfurt POP.

### Step 5: Origin Responds, Caches Fill

The origin server receives the single request from the shield, generates the response (20ms server time), and returns it. The shield caches it with a TTL header. The shield forwards the response to the regional cache, which caches it. The regional cache forwards to the Frankfurt edge POP, which caches it and responds to the browser.

### Step 7: TTL Expiry Triggers Revalidation

After one hour, the TTL on `/index.html` expires at the Frankfurt edge. The next request triggers a conditional GET with an `If-None-Match` header containing the ETag. If the content has not changed, the origin returns HTTP 304 Not Modified with no body. The edge updates the TTL without transferring the full content.

## Variants and Flavors

### Variant 1: CDN for Static Assets Only

The simplest and most common CDN use case. Static assets — images, CSS, JavaScript, fonts — rarely change and benefit from multi-year TTLs.

<AsciiDiagram
  id="cdns/variant-static"
  title="Variant 1 — Static asset CDN with long TTLs"

│             │                  │                  │

└────────────┘   logo.png +       │  TTL = 365 days  │
                  Cache-Control:  │  No version check │
                  public,         │  Serve directly   │
                  max-age=31536000│                   │
                                  └─────────────────┘

  Versioned URLs (logo.v2.png) enable cache busting without purge.
  `}
  caption="Static asset CDN uses maximum TTLs and versioned URLs for zero origin traffic on repeat views."
/>

**When to use:** Your site serves images, fonts, or JavaScript bundles that change infrequently. Cache hit ratio exceeds 95 percent.

**When not to use:** Content changes every few seconds (live scores, stock tickers). The TTL would be so short that cache overhead exceeds benefits.

### Variant 2: CDN for Dynamic Content (Push CDN / DCDN)

The CDN does not just cache — it processes dynamic content closer to the user. Edge computing platforms (Cloudflare Workers, Fastly Compute@Edge, AWS Lambda@Edge) execute code at the POP.

**When to use:** You need personalization, A/B testing, authentication checks, or API aggregation at the edge. User-perceived latency drops dramatically because the compute happens in the same city as the user.

**When not to use:** Your dynamic logic requires a centralized database (ACID transactions, complex joins). Edge compute does not have low-latency access to a single database from every POP — latency between POPs is 100-300ms.

**When to use:** Distributing files over 100MB to a global audience — software installers, game updates, firmware images, video files.

**When not to use:** Distributing files under 1MB in size. The overhead of range request handling and parallel connection management does not justify the improvement.

## Mental Model Check

**Question 1:** You deploy a new version of your site. Users in New York see the new content immediately, but users in Tokyo see the old content for another 30 minutes. What happened?

**Answer:** The Tokyo edge POP still has the old content cached with a long TTL. Your purge request either did not reach that POP, or the purge was for a different cache key (for example, the URL changed but you purged the old URL). Solution: use versioned URLs (`style.v2.css`) so old cache keys are naturally abandoned, or verify the purge API is covering the correct URL pattern globally.

---

**Question 2:** You observe that your cache hit ratio is 65 percent. The origin server is under heavy load. What is the most likely cause?

**Answer:** Either your TTLs are too short, or you have many unique cache keys (for example, URLs with session IDs or timestamps in query parameters). Every unique key creates a separate cache entry, so rarely-repeated URLs always miss. Solution: normalize cache keys by stripping irrelevant query parameters, and increase TTLs for content that does not change frequently.

---

**Question 3:** After adding a CDN, your dynamic API endpoints now return errors. What went wrong?

**Answer:** The CDN likely cached a POST response or cached a 5xx error response. POST requests should never be cached by default. Error responses should have `Cache-Control: no-store` headers. If the CDN cached a 503 response, it served the error to all subsequent users until the TTL expired. Solution: ensure your origin sends correct `Cache-Control` headers on error responses, and configure the CDN to cache only GET requests.

---

**Question 4:** You have 10 edge POPs worldwide. At 2:00 PM UTC, every POP simultaneously experiences a cache miss for the same new content. What is the origin load?

**Answer:** Without an origin shield, the origin would receive 10 requests simultaneously (one from each POP's edge cache). With an origin shield, only 1 request reaches the origin (the shield deduplicates). This is why the origin shield layer is critical — a flash crowd of 100 POPs would still send only 1 request to the origin through the shield.

---

**Question 5:** Your video streaming CDN works well for 480p and 720p but buffers frequently for 4K streams. What is the bottleneck?

**Answer:** 4K video requires 25-50 Mbps sustained throughput. CDN edges typically have gigabit ports shared across many users. If the edge POP serves 50 concurrent 4K streams at 40 Mbps each, that is 2 Gbps — exceeding a single edge server's network capacity. The fix is either: (a) configure the CDN to serve large video from dedicated streaming servers with higher bandwidth allocation, or (b) use a CDN specialized for video that pre-positions content and uses connection pooling.
