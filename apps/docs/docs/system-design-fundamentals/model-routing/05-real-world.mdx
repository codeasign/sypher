## Company Case Studies

### OpenAI: API Gateway Routing

OpenAI operates one of the largest model routing systems in production. Every API call to OpenAI's endpoints passes through a routing layer that determines which model instance serves the request. The routing system handles millions of requests per minute across dozens of model variants.

The routing architecture has three layers. First, an API gateway that authenticates requests, validates parameters, and applies rate limits. Second, a model router that maps each request to a specific model deployment based on the model name, availability zone, and capacity. Third, an instance scheduler that assigns requests to individual GPU instances running the model.

The key challenge OpenAI solves is capacity management across model variants. GPT-4, GPT-4o, GPT-4o-mini, and each fine-tuned model have different capacity pools. The router must balance load across instances, handle hot spots (a new model launch that floods capacity), and gracefully degrade when instances fail.

OpenAI's routing decisions are based on:
- **Model availability:** Which model variants are deployed in which regions
- **Capacity headroom:** Current queue depth and estimated time to completion per instance
- **Latency optimization:** Route to the nearest available instance with capacity
- **Cost optimization:** Batch requests to the same model to maximize GPU utilization

The system also handles the "cold start" problem for new model deployments. When a new model version is deployed, traffic is gradually shifted from old to new instances, with the router monitoring error rates and latency at each step.

### Anthropic: Claude Model Selection

Anthropic's routing system focuses on capability matching rather than cost optimization. Since Anthropic primarily offers Claude models (Haiku, Sonnet, Opus) through their API, the routing challenge is determining which model variant best matches each request's complexity and safety requirements.

The routing system uses a prompt classifier that analyzes the request's complexity, domain, and safety sensitivity. Simple classification tasks route to Claude Haiku. Complex reasoning or code generation routes to Claude Sonnet or Opus. Requests involving sensitive topics receive additional safety processing before routing.

### LangChain and LangSmith: Routing Chains

LangChain provides a framework for building LLM applications, and LangSmith is their observability and evaluation platform. In the LangChain ecosystem, model routing is implemented through "routing chains" -- programmable decision points that direct requests to different models or chains based on the input.

The LangChain routing approach is developer-centric. Instead of an opaque routing layer, developers write explicit routing logic using LangChain's expression language. This gives maximum flexibility but requires developers to maintain the routing logic themselves.

### ML Infrastructure Companies: Portkey, Helicone, and OpenRouter

Several companies have built dedicated model routing infrastructure as a service. These platforms provide a unified gateway that handles routing, fallback, caching, and observability across multiple LLM providers.

**Portkey** provides a model gateway with AI-powered routing. Their system uses a prompt classifier to determine the optimal model for each request, considering cost, latency, and quality. They support fallback chains, retry logic, and A/B testing of different models. Portkey's routing is configurable through a dashboard, allowing teams to update routing policies without code changes.

**Helicone** focuses on observability-first routing. Their system collects detailed metrics on every request -- latency, cost, token usage, error rates -- and uses this data to inform routing decisions. Helicone's approach is to let teams start with simple routing (route everything to one model), collect data, and then iteratively refine routing policies based on real usage patterns.

**OpenRouter** provides a marketplace-style routing system. When you send a request to OpenRouter, their system selects the best model from multiple providers based on availability, price, and latency. OpenRouter handles provider failover transparently -- if one provider is down, the request is routed to another provider offering the same model.

## Open Source Implementations

Several open source projects provide model routing capabilities that you can self-host:

**AI Gateway (Portkey):** Portkey's AI Gateway is an open source model gateway that handles routing, fallback, caching, and load balancing across LLM providers. It supports 250+ models and provides a unified API interface. The gateway is implemented in Node.js and can be deployed as a sidecar or standalone service. It supports weighted routing (send 70 percent of traffic to GPT-4o, 30 percent to Claude Sonnet for A/B testing), fallback chains, and circuit breakers.

**LiteLLM:** A Python library that provides a unified interface for calling 100+ LLMs. LiteLLM handles model routing through its "router" abstraction, which supports fallback, load balancing, and cost tracking. The router can be configured with YAML or Python dictionaries. LiteLLM is lighter than AI Gateway -- it is a library rather than a standalone service -- making it suitable for smaller deployments or as part of a larger application.

**OpenRouter (community):** OpenRouter's open source components provide routing logic for selecting models across providers based on availability and pricing. The system maintains a registry of available models, their current pricing, and health status. It is designed for applications that need to switch between providers transparently.

**Semantic Router:** A specialized library for semantic routing -- routing decisions based on the meaning of the input rather than keywords. It uses embedding models to classify requests and can be used independently or as a component in a larger routing system.

**Trade-offs between open source options:**

| Option | Deployment | Routing Logic | Best For |
|--------|-----------|--------------|----------|
| AI Gateway | Sidecar or standalone | Rule-based, ML-based, weighted | Production, multi-provider setups |
| LiteLLM | Library (embedded) | Rule-based, fallback chains | Python applications, simpler setups |
| OpenRouter | Standalone service | Provider-based, availability-first | Consumer-facing apps needing failover |
| Semantic Router | Library | Embedding-based semantic routing | Applications needing content-aware routing |

## Cloud Provider Implementations

### AWS Bedrock Model Invocation

AWS Bedrock provides a managed service for accessing foundation models from multiple providers. Bedrock's routing system handles model selection within the AWS ecosystem. When you invoke a model through Bedrock, the service routes your request to the appropriate model endpoint based on the model ID, provisioned throughput, and region.

Bedrock supports two routing modes: on-demand (serverless, pay-per-token) and provisioned throughput (reserved capacity). The routing system directs requests to the appropriate infrastructure based on the mode. For provisioned throughput, the router ensures requests go to the correct provisioned instances.

### GCP Vertex AI Model Garden

Google Cloud's Vertex AI provides a "Model Garden" that includes Google's models (Gemini, PaLM) and third-party models. Vertex AI's routing system focuses on model selection within the GCP ecosystem. It supports model versioning, A/B testing, and traffic splitting between model versions.

Vertex AI's routing is particularly strong for multimodal models -- it routes requests based on the input modality (text, image, video, audio) to the appropriate model endpoint. The routing system is integrated with GCP's monitoring and logging, providing observability out of the box.

### Azure OpenAI Routing

Microsoft Azure's OpenAI service provides routing across OpenAI models deployed in Azure regions. The routing system handles regional availability, capacity management, and data residency requirements. Azure's routing is tightly integrated with their global network, routing requests to the nearest available endpoint.

Azure supports "data zone routing" for compliance -- requests from EU customers can be routed to EU-only endpoints, ensuring data never leaves the region. This is a critical feature for enterprise customers with data residency requirements.

## Lessons Learned

### Lesson 1: Start Simple, Measure Everything

The most common mistake teams make is building an elaborate routing system before they have data. Start by routing all traffic to a single model, instrument every request with latency, cost, and quality metrics, and analyze the distribution. Only after you understand your traffic patterns should you build routing logic. Teams that skip this step end up optimizing based on assumptions, not data.

### Lesson 2: Routing Accuracy Is a Spectrum, Not a Binary

No classifier achieves 100 percent accuracy. A 95 percent accurate router means 1 in 20 requests is misclassified. You need to decide what happens to those misclassified requests. The safest approach is to route low-confidence classifications to the most capable model (conservative routing). The most cost-effective approach is to route them to the cheapest model that meets a minimum quality threshold. The right choice depends on your application's tolerance for quality degradation.

### Lesson 3: Fallback Chains Are Not Optional

Every model can fail -- rate limits, outages, latency spikes, degraded quality. A routing system without fallback chains is fragile. Design each route with at least two fallback options. The fallback chain should be carefully ordered: the first fallback should be a model with similar capabilities, the second should be a model with broader capabilities (but higher cost), and the third should be a graceful degradation path (e.g., a cached response or a message asking the user to retry later).

### Lesson 4: Cost Drift Is Real and Must Be Monitored

Model pricing changes. Providers change pricing tiers, introduce new models, and deprecate old ones. Your routing decisions, which were optimal when configured, become suboptimal over time. Set up alerts for cost drift: if the average cost per request increases by more than 10 percent over a week, investigate. Re-evaluate routing policies monthly against current pricing.

### Lesson 5: The Routing System Itself Must Be Observable

A routing system adds a layer of indirection. If the routing system fails, all requests fail. You need comprehensive observability: routing decision latency, classification distribution, fallback rate, cache hit ratio, and registry health. Every routing decision should be traceable -- given a request ID, you should be able to see why it was routed to a particular model. Without this observability, debugging routing issues becomes guesswork.

## Metrics and Key Performance Indicators

A model routing system generates dozens of metrics. Here are the most important ones to monitor and the targets they should meet.

### Routing Accuracy

Routing accuracy measures the percentage of requests that are sent to the best model for that request. This is the most important metric because it captures whether the routing system is achieving its goal.

| Metric | Target | Warning | Critical |
|--------|--------|---------|----------|
| Classification accuracy | Over 95 percent | Under 90 percent | Under 80 percent |
| Fallback rate | Under 3 percent | Over 5 percent | Over 10 percent |
| Misclassification rate (sampled) | Under 5 percent | Over 8 percent | Over 15 percent |
| Low-confidence rate | Under 15 percent | Over 25 percent | Over 40 percent |

### Cost Efficiency

Cost metrics measure whether the routing system is delivering the expected savings.

| Metric | Target | Warning | Critical |
|--------|--------|---------|----------|
| Cost per request | Under budget | 10 percent over budget | 25 percent over budget |
| Cost savings vs. single-model baseline | 40-70 percent | Under 30 percent | Under 20 percent |
| Routing overhead cost | Under 1 percent of total cost | Over 3 percent | Over 5 percent |
| Fallback cost premium | Under 5 percent | Over 10 percent | Over 20 percent |

### Performance

Latency and throughput metrics measure the routing system's operational health.

| Metric | Target | Warning | Critical |

## Real-World Cost Analysis

To understand the concrete financial impact of model routing, consider a real-world example based on a customer support platform that processes 5 million conversations per month.

### Without Routing

The system uses GPT-4o for every request. Each conversation averages 4 requests, for a total of 20 million requests per month. Each request averages 500 input tokens and 200 output tokens.

Monthly cost calculation:
- Input tokens: 20M * 500 = 10B input tokens
- Output tokens: 20M * 200 = 4B output tokens
- GPT-4o cost: (10B / 1M) * $2.50 + (4B / 1M) * $10.00 = $25,000 + $40,000 = $65,000 per month

### With Routing

Traffic distribution after analysis: 60 percent simple (FAQ, passwords, status checks), 25 percent moderate (account issues, billing questions), 15 percent complex (technical support, complaints, legal).

Simple requests go to GPT-4o-mini ($0.15/$0.60 per 1M tokens). Moderate requests go to Claude Sonnet ($3.00/$15.00 per 1M tokens). Complex requests go to GPT-4o ($2.50/$10.00 per 1M tokens).

Monthly cost calculation:
- Simple: 12M requests * 700 tokens = 8.4B tokens. Cost: (8.4B / 1M) * $0.15 input + $0.60 output = $1,260 + $5,040 = $6,300
- Moderate: 5M requests * 700 tokens = 3.5B tokens. Cost: (3.5B / 1M) * $3.00 input + $15.00 output = $10,500 + $52,500 = $63,000
- Complex: 3M requests * 700 tokens = 2.1B tokens. Cost: (2.1B / 1M) * $2.50 input + $10.00 output = $5,250 + $21,000 = $26,250
- Total with routing: $6,300 + $63,000 + $26,250 = $95,550 per month

Wait -- that is higher than without routing! This is because moderate requests were sent to Claude Sonnet, which is more expensive than GPT-4o for this volume. The routing policy needs adjustment.

### Optimized Routing

After analysis, the team realizes that GPT-4o-mini costs $0.15/$0.60, while Claude Sonnet costs $3.00/$15.00 -- a 20x difference. For moderate requests, the routing table is updated to use GPT-4o ($2.50/$10.00) instead of Claude Sonnet, because GPT-4o is cheaper for this volume.

- Simple: $6,300 (GPT-4o-mini)
- Moderate: 5M requests * 700 tokens = 3.5B tokens. GPT-4o cost: (3.5B / 1M) * $2.50 + $10.00 = $8,750 + $35,000 = $43,750
- Complex: $26,250 (GPT-4o)
- Total with optimized routing: $6,300 + $43,750 + $26,250 = $76,300

Monthly savings: $65,000 (original) to $76,300 (with routing) -- wait, that is a cost increase. This shows that routing is not always cost-saving if the mid-tier model is more expensive than the single model baseline.

The key insight: **Model routing saves money only when you can route a significant fraction of traffic to models that are genuinely cheaper than your current single-model baseline.** If your "moderate complexity" mid-tier model costs more than the baseline, routing actually increases cost.

### The Correct Approach

In this example, the correct routing strategy is:
- Simple (60 percent): GPT-4o-mini ($0.15/$0.60)
- Moderate + Complex (40 percent): GPT-4o ($2.50/$10.00) -- same as the original single model

Total monthly cost: $6,300 + $43,750 = $50,050

Monthly savings: $65,000 - $50,050 = $14,950 (23 percent reduction)

This example illustrates a crucial lesson: **routing saves money by identifying the fraction of traffic that can use a cheaper model, not by distributing traffic across multiple expensive models.** The 60 percent of simple traffic going to GPT-4o-mini provides all the savings. The remaining 40 percent can stay on the original model.

## Provider-Specific Routing Considerations

Each model provider has different characteristics that affect routing decisions. Understanding these differences is critical for designing a routing system that works across multiple providers.

### OpenAI

OpenAI offers a range of models with distinct pricing tiers: GPT-4o ($2.50/$10.00 per 1M tokens), GPT-4o-mini ($0.15/$0.60), GPT-4 Turbo, and legacy models. OpenAI's API is the most widely used and has the most mature SDK support.

**Routing considerations:**
- OpenAI has a single API endpoint for all models -- the model name is specified in the request body. The router does not need to switch endpoints, just model names.
- Rate limits are per-model and per-organization tier. The router must track rate limit headers and back off when approaching limits.
- OpenAI's models are hosted in US-based data centers by default. For EU customers, Azure OpenAI provides regional hosting.
- Pricing changes are announced in advance but can still affect routing economics when they take effect.

**Recommended routing approach for OpenAI-only setups:** Use a simple model gateway that dynamically selects the model name based on routing decisions. No multi-endpoint complexity.

### Anthropic

Anthropic offers Claude Opus (most capable, $15.00/$60.00), Claude Sonnet ($3.00/$15.00), and Claude Haiku ($0.25/$1.25). Anthropic's API uses a different endpoint than OpenAI's, requiring the gateway to switch API endpoints.

**Routing considerations:**
- Each Claude model has its own API endpoint. The router must know which endpoint to call for each model.
- Claude models have different context window sizes: Opus and Sonnet support 200K tokens, Haiku supports 200K tokens. This affects capability matching for long-document requests.
- Anthropic's API uses a different authentication mechanism (x-api-key header) that the gateway must handle.
- Claude models are particularly strong at structured analysis, code generation, and multi-turn conversations -- these request types should be routed to Claude models when available.

**Recommended routing approach for Anthropic + OpenAI:** Use a model gateway that abstracts provider endpoints. The gateway maintains a mapping of model names to API endpoints, authentication methods, and request formats. The router only needs to specify the model name; the gateway handles provider-specific orchestration.

### Google Gemini

Google offers Gemini 1.5 Pro and Gemini 1.5 Flash through their Vertex AI and AI Studio APIs. Pricing is competitive with OpenAI: Gemini 1.5 Flash at $0.075/$0.30 per 1M tokens is one of the cheapest capable models.

**Routing considerations:**
- Gemini models support very large context windows (up to 1 million tokens for Pro). For long-context requests, Gemini may be the only practical option.
- Google's API is accessed through GCP authentication (service accounts, OAuth), which is different from OpenAI's API key model.
- Gemini Flash offers excellent speed (under 500ms for typical requests) making it ideal for latency-sensitive routing tiers.
- Vertex AI provides enterprise-grade features (VPC controls, audit logging, private endpoints) that may be required for regulated industries.

### Model Selection by Provider Strength

Different providers excel at different request types. A well-designed routing system considers provider strengths, not just model names:

| Request Type | Best Provider | Reason |
|-------------|--------------|--------|
| Code generation | Anthropic (Claude Sonnet) or OpenAI (GPT-4o) | Strongest code understanding and generation |
| Long document analysis | Google (Gemini 1.5 Pro) | 1M token context window |
| Simple classification | OpenAI (GPT-4o-mini) | Fastest response time, lowest cost |
| Multi-step reasoning | Anthropic (Claude Opus) | Strongest reasoning capabilities |
| Multilingual content | Google (Gemini 1.5 Flash) | Best non-English performance at low cost |
| Structured data extraction | OpenAI (GPT-4o) | Strong JSON mode and function calling |

## Future Trends in Model Routing

The model routing landscape is evolving rapidly. Understanding where the industry is heading helps you build routing systems that will remain relevant as the ecosystem changes.

### Trend 1: Model Convergence

As foundation models improve, the capability gap between "cheap" and "expensive" models is narrowing. GPT-4o-mini today matches GPT-3.5's capabilities from 18 months ago. If this trend continues, the cost savings from routing may diminish because even "cheap" models will be capable enough for most tasks.

**Impact on routing:** The routing system must adapt to shrinking capability gaps. Instead of routing based on capability tiers, routing may shift to latency-based and availability-based decisions. The model router becomes more of a load balancer than a capability matcher.

### Trend 2: Provider-Specific Specialization

Model providers are differentiating by specialization rather than general capability. Some models excel at code, others at reasoning, others at multimodal tasks. Routing decisions will increasingly be about matching request type to provider specialty rather than matching complexity to model tier.

**Impact on routing:** The routing table needs to capture provider-specific strengths, not just cost and latency. The classifier needs to identify not just request complexity but the domain and modality of the request.

### Trend 3: Dynamic Pricing

Model providers are introducing dynamic pricing -- off-peak discounts, reserved capacity pricing, and spot instances for LLM inference. These pricing models make static routing tables suboptimal.

**Impact on routing:** The routing system must incorporate a real-time pricing feed and re-optimize routing decisions dynamically. This adds complexity but also increases potential savings.

### Trend 4: Routing as Infrastructure

Model routing is evolving from an application-level pattern to an infrastructure concern. Just as load balancers and API gateways are standard infrastructure components, model routing gateways will become a standard part of the AI infrastructure stack.

**Impact on routing:** Future routing systems will be configured through infrastructure-as-code (Terraform, Pulumi) rather than application code. Teams will choose from managed routing services (Portkey, Helicone) instead of building their own.

## Implementation Blueprint: Building a Router in a Weekend

If you want to build a basic model router for your application, here is a concrete implementation plan that a single developer can complete in a weekend.

### Saturday Morning: Set Up the Foundation

Create a Python service with FastAPI that accepts requests and routes them to models. Set up a configuration file for the routing table.

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import json

app = FastAPI()

class RoutingRequest(BaseModel):
    text: str
    user_id: str = None
    max_latency_ms: int = 5000
    max_cost_cents: float = 1.0

class ModelGateway:
    def __init__(self, config_path="routing_config.json"):
        with open(config_path) as f:
            self.config = json.load(f)
        self.routing_table = self.config["routing_table"]
    
    def classify(self, text: str) -> str:
        # Simple keyword-based classifier
        text_lower = text.lower()
        for category, rules in self.routing_table.items():
            for keyword in rules["keywords"]:
```

Implement the fallback chain and basic metrics collection. Use a simple in-memory store for circuit breaker state.

```python
class CircuitBreaker:
    def __init__(self, threshold=5, cooldown_seconds=30):
        self.failures = {}
        self.threshold = threshold
        self.cooldown = cooldown_seconds
    
    def record_failure(self, model: str):
        now = time.time()
        if model not in self.failures:
            self.failures[model] = []
        self.failures[model].append(now)
        # Remove entries older than cooldown
        self.failures[model] = [t for t in self.failures[model] 
                                if now - t < self.cooldown]
    
    def is_open(self, model: str) -> bool:
        return len(self.failures.get(model, [])) >= self.threshold
```

```python
import threading

class RoutingTableCache:
    def __init__(self, registry_url, refresh_interval=60):
        self.registry_url = registry_url
        self.refresh_interval = refresh_interval
        self.cache = {}
        self.lock = threading.Lock()
        self._start_refresh()
    
    def _refresh(self):
        while True:
            time.sleep(self.refresh_interval)
            # Fetch from registry
            new_table = self._fetch_from_registry()
            with self.lock:
                self.cache = new_table
    
    def get(self, key):
        with self.lock:
            return self.cache.get(key)
```
