## Real-World Systems

Production systems at scale have faced every consistency failure mode. The following case studies show how leading companies made tradeoffs and what broke when they got them wrong.

## Case Study 1: Google Spanner — External Consistency via TrueTime

Google Spanner is the first globally distributed database to provide **external consistency** (linearizability with a global commit order) across data centers at planetary scale.

### Architecture

Spanner uses TrueTime, Google's time-synchronization infrastructure, to assign globally meaningful timestamps to writes. TrueTime exposes a bounded uncertainty interval `[earliest, latest]` for the current time.

### How TrueTime Works

TrueTime is the core innovation that makes Spanner possible. It is not a single clock but a distributed time infrastructure deployed across every Spanner data center.

**Two reference sources.** Each data center runs both GPS receivers and atomic clocks. GPS provides microsecond-level accuracy but can fail due to signal obstruction, antenna issues, or satellite outages. Atomic clocks provide nanosecond-precision local timekeeping but drift over days. By combining both sources and cross-checking them, TrueTime maintains a credible uncertainty bound.

**Uncertainty bound e (epsilon).** TrueTime returns an interval `[TT_earliest(), TT_latest()]` instead of a single time value. The width of this interval is the uncertainty bound e, which TrueTime guarantees contains the actual time. In practice, e ranges from 1ms to 7ms depending on network distance and clock skew between data centers. Google's infrastructure monitors and publishes this bound continuously.

**Lease-based polling.** Each Spanner server polls a local TrueTime daemon for the current interval. The daemon maintains its clock by synchronizing with the GPS and atomic clock infrastructure every 30 seconds. Between polls, the daemon increases the uncertainty bound to account for clock drift. If a server cannot reach the TrueTime daemon, it eventually rejects operations rather than risk violating consistency.

**Commit wait.** When a Spanner write transaction commits, the coordinator assigns a timestamp T and runs Paxos to replicate the write. The coordinator then waits until `TT_after(T)` returns true — meaning real time has passed the commit timestamp plus the uncertainty bound. This wait guarantees that any subsequent read, no matter which data center it originates from, will see a local TrueTime interval that is entirely past T. Without commit wait, a read at a different data center could have a clock so far behind that it misses the write.

This mechanism gives Spanner **external consistency**: if transaction A commits before transaction B starts, then A's timestamp is strictly less than B's timestamp. No clock drift between data centers can invert the order.

### Paxos Groups and Directory-Based Splitting

Spanner divides data into **directories**, which are sets of contiguous key-value pairs. Each directory is replicated across a **Paxos group** — typically 3 to 5 replicas spread across zones. The Paxos group runs the Paxos consensus algorithm to commit writes with linearizability within the group.

A single Spanner server (called a **tablet**) can host multiple Paxos groups. Directories can be split and moved between Paxos groups for load balancing. When a directory grows too large, Spanner splits it. When a tablet becomes hot, Spanner moves directories to other tablets.

### Read-Write Transaction Path

A Spanner read-write transaction follows these steps:

1. **Begin:** The client sends a BeginTransaction request to the local Spanner node. The node becomes the transaction coordinator.

2. **Read phase:** The coordinator sends reads to the Paxos group leaders for the relevant directories. Group leaders acquire read locks and return the current values with their commit timestamps.

3. **Buffered writes:** The client buffers writes locally. No data is sent to replicas yet.

4. **Commit:** The client sends Commit. The coordinator picks a group as the **leader group** (the one with the largest pending write set). The coordinator assigns a commit timestamp T using `TT_now().latest`.

5. **Paxos commit:** The coordinator sends the write to all Paxos groups involved. Each group runs Paxos to commit the write at timestamp T. If any group fails, the entire transaction aborts.

6. **Commit wait:** The coordinator waits until `TT_after(T)` returns true.

7. **Release locks:** Read locks are released and the coordinator tells the client the transaction committed.

### Key Insight

Spanner achieves strong consistency at scale without sacrificing availability because TrueTime eliminates the tradeoff between consistency and speed. Instead of waiting for a second round trip to confirm ordering, Spanner uses the guaranteed clock bound to order writes across data centers.

### What Broke Before Spanner

Before Spanner, Google's systems used Bigtable (eventual consistency) for many applications. Engineers at Google reported that 15-20% of application code was devoted to handling consistency edge cases that would not exist with a strongly consistent database. Spanner eliminated that complexity at the cost of requiring atomic clocks and GPS receivers in every data center.

## Case Study 2: Amazon DynamoDB — Tunable Consistency at Scale

DynamoDB is Amazon's fully managed NoSQL database. It provides tunable consistency: per-request choice between eventual (default), strongly consistent reads, and transaction support.

### Architecture

DynamoDB is built on a Dynamo-style ring architecture with consistent hashing, vector clocks, and hinted handoff.

### Adaptive Capacity and Auto-Scaling

DynamoDB's storage nodes are organized into partitions, each managed by a leader replica and two follower replicas. Each partition handles a range of hash keys.

**Adaptive capacity** is DynamoDB's response to hot keys and uneven access patterns. When a single partition receives disproportionately high traffic, DynamoDB automatically splits the partition and redistributes the data. This splitting happens without downtime. For example, if a single product in an e-commerce catalog receives 50% of all read traffic, adaptive capacity moves that product's data to a dedicated partition with its own RCU/WCU allocation. This prevents a hot key from throttling all other items in the original partition.

**Auto-scaling** adjusts the provisioned throughput based on actual usage patterns. DynamoDB Auto Scaling uses a target utilization percentage (default 70%). When consumed capacity exceeds this target for a sustained period, DynamoDB increases the provisioned capacity. When utilization drops below 70%, it decreases capacity after a cool-down period. This matters for consistency: under-provisioned tables throttle requests, which causes retries, which increases the time window where replicas can diverge.

### DynamoDB Accelerator (DAX)

DAX is an in-memory cache for DynamoDB that sits between your application and the DynamoDB service. DAX provides two benefits for consistency:

**Read-through with TTL.** DAX caches read results with a configurable TTL (default 5 minutes). For eventually consistent reads, DAX serves cached data directly without hitting DynamoDB, cutting read latency from single-digit milliseconds to microseconds. When the TTL expires, the next read fetches fresh data from DynamoDB.

**Write-through invalidation.** When your application writes to DynamoDB through DAX, DAX invalidates the corresponding cached item. The next read must go to DynamoDB and picks up the latest version. This does not guarantee read-after-write consistency — a concurrent read from a different DAX node might still read stale data — but it reduces the window of inconsistency significantly.

DAX is appropriate for read-heavy, eventually consistent workloads like leaderboards, product catalogs, and session stores where sub-millisecond response time matters more than absolute freshness.

### Global Tables: Multi-Region Replication

DynamoDB Global Tables replicate data across AWS regions with last-writer-wins (LWW) conflict resolution. Each region maintains a full replica of the table. Writes in any region propagate to all other regions within one second typically.

### Key Insight

Amazon's key innovation was making consistency a per-request tuning parameter rather than a database-wide configuration. The DynamoDB paper (2007) showed that most data does not need strong consistency, and those that do can pay the cost selectively.

### What Broke

DynamoDB's eventual consistency default caused incidents at early Amazon adopters. The 2011 Amazon outage was partly attributed to cascading consistency failures: a DynamoDB table used for session state returned stale session data, causing authentication failures that propagated across services. Amazon responded by adding stronger consistency options.

## Case Study 5: Discord — From Cassandra to Scylla for Consistency

Discord migrated from Cassandra to ScyllaDB after encountering consistency-related issues that impacted user experience.

### The Problem

Discord stored chat messages in Cassandra with N=3, W=2, R=1 (eventual reads). Users occasionally experienced:

- **Disappearing messages:** A message appeared, then vanished on refresh.
- **Out-of-order messages:** Messages appeared in different orders on different clients.
- **Notification desync:** The notification count showed "5 new messages" but only 3 appeared.

The root cause: R=1 (eventual reads) meant reads hit any one of three replicas. If that replica was behind on replication, the user saw stale data.

### The Solution

Discord moved to ScyllaDB (a Cassandra-compatible database) with stricter consistency settings and better repair mechanisms.

## Case Study 6: LinkedIn — Distributed Counters With Cassandra CRDTs

LinkedIn operates one of the largest professional social networks, serving over 1 billion members. One of its most visible features is the profile view counter — the number of times a member's profile has been viewed.

### The Problem

Profile views are a high-write, low-read workload. Every page visit to a profile creates a write. The total write rate across LinkedIn's member base exceeded 500,000 writes per second during peak hours. Storing each view as an individual row in a relational database would require enormous write capacity. Furthermore, the exact count is not critical — a member seeing 142 instead of 143 views does not change the user experience. The system can tolerate bounded inaccuracy.

### The Solution

LinkedIn built its profile view counter on **Apache Cassandra** with a **CRDT (Conflict-Free Replicated Data Type)** counter approach.

**Cassandra counters.** Cassandra provides a built-in counter data type that supports increment and decrement operations. Each counter operation is idempotent — applying the same increment twice is safe because Cassandra uses a distributed counter implementation that tracks per-node contributions.

**CRDT counter mechanics.** The Cassandra counter uses a **PN-Counter** (Positive-Negative Counter) under the hood. Each node maintains two vectors: one for increments and one for decrements. The total count is the sum of all increments minus the sum of all decrements across all nodes. When a read request arrives, the coordinator queries all replicas and merges their vectors by taking the element-wise maximum of each vector. This merge operation is commutative, associative, and idempotent — the defining properties of a CRDT.

**Eventual consistency is acceptable.** LinkedIn chose eventual consistency because:
- Profile view counts do not need to be exact — a difference of 1-2% is imperceptible in the UI.
- The CRDT counter converges automatically without complex conflict resolution logic.
- Writes never block on replica coordination — Cassandra accepts the write as soon as one replica acknowledges it.
- If a node is down, its counter contribution is replayed via hinted handoff when it recovers.

### Key Numbers

- Write throughput: 500,000+ counter updates per second at peak.
- p99 read latency: 5ms (eventually consistent read from one replica).
- Staleness during normal operation: less than 100ms.
- Maximum observed counter drift during partition: 5% between nodes.
- Storage per counter: approximately 100 bytes (two vectors per node in the replica set).

### What You Can Learn

LinkedIn's counter is a textbook case of choosing eventual consistency correctly. The key criteria: the data is additive, small errors are invisible in the UI, and the write volume would make strong consistency prohibitively expensive. If LinkedIn had chosen a strongly consistent counter with quorum writes, they would have needed to scale their Cassandra cluster to approximately 3x the nodes to maintain the same write throughput.

## Case Study 7: GitHub — From MySQL to Vitess With Sharded Consistency

GitHub hosts over 200 million repositories and serves more than 100 million developers. For most of its history, GitHub ran on a single MySQL master with read replicas. As the platform grew, this architecture hit limits.

### The Problem

By 2018, GitHub's primary MySQL database was handling approximately 5,000 queries per second (QPS) during peak hours with a single write master. The database was 300GB in size, and schema changes took hours or days. GitHub needed to scale writes horizontally. Sharding was the only viable path, but sharding introduces consistency challenges.

### The Migration to Vitess

GitHub migrated to **Vitess**, an open-source database clustering system for MySQL. Vitess provides horizontal sharding with automatic resharding, connection pooling, and query routing. Under the hood, Vitess uses **etcd** (Raft-based consensus) for metadata management and coordination.

### Consistency Challenges in Sharded Databases

**Challenge 1: Cross-shard transactions.** A GitHub operation like "create a pull request with labels" might need to write to the repositories shard, the issues shard, and the labels shard. Vitess supports distributed transactions using two-phase commit (2PC), but 2PC adds latency proportional to the number of shards involved. Each phase requires a round trip to each shard coordinator, and the prepare phase must hold locks until the commit phase completes. GitHub optimized by minimizing cross-shard transactions: they designed the schema so that most operations touch a single shard.

**Challenge 2: Fan-out queries.** When a query does not specify a sharding key, Vitess must fan the query out to every shard. For example, a search for "all repositories owned by user X" if the sharding key is repository ID instead of user ID. Each shard executes a partial query and returns results to the Vitess gateway, which merges them. During the merge, the gateway must handle partial failures — one shard might time out. If a shard fails, the gateway can either return partial results or fail the entire query. GitHub chose to return partial results for non-critical queries and fail for critical ones.

**Challenge 3: Read-after-write consistency within a session.** After a user pushes a commit, the web UI must show that commit on the next page load. With multiple shards and read replicas, a read might go to a replica that has not yet received the write. Vitess addresses this with **replica routing hints**: a session parameter that tells Vitess to prefer the shard's master for reads that must see recent writes. GitHub marks any read that occurs within 5 seconds of a write as a "critical read" and routes it to the master.

**Challenge 4: Resharding.** When a shard grows too large (GitHub's threshold was 300GB per shard), Vitess splits it into two or more smaller shards. During resharding, both the old and new shards must serve queries simultaneously. Vitess uses a **copy-and-catch-up** strategy: it copies data from the old shard to the new shards while replaying the binlog of ongoing writes. Once the new shards catch up, the Vitess gateway switches to routing queries to the new shards. During the switch, there is a brief window (typically under 1 second) where a write might be applied to the old shard but not yet visible on the new shard.

### Key Numbers

- Pre-migration: single MySQL master at 5,000 QPS, 300GB database.
- Post-migration: 30+ shards, each under 300GB.
- Cross-shard transaction latency: additional 15-20ms per shard.
- Fan-out query latency: maximum of all shard latencies, not average.
- Read-after-write window for critical reads: 5 seconds.
- Resharding downtime: zero; copy-and-catch-up runs online.

## Eight Lessons From Production

**Lesson 1: Eventual consistency causes visible user-facing bugs in chat and feed applications.**

Discord's disappearing messages, Facebook's (now Meta's) wrong notification counts, and Twitter's (now X's) timeline inconsistencies all trace back to eventual reads. Users notice staleness when they interact with their own data. If the user just performed an action and the system UI shows a different state, they perceive the application as broken even if the data converges seconds later.

**Lesson 2: Tunable consistency lets you pay for what you need.**

DynamoDB's pricing model encodes this: eventual reads cost half of strong reads. Most data can be eventual. Know which data is which. A good rule of thumb: user-owned data (profile, settings, messages) needs strong consistency. System-owned data (logs, metrics, aggregates) can tolerate eventual consistency.

**Lesson 3: Coordination services prove strong consistency is viable at small scale.**

etcd and ZooKeeper handle 10K writes/second with single-millisecond latency. Coordination metadata does not need petabytes of scale. Use strong consistency for coordination; use weaker models for bulk data. Do not use a coordination service as a primary database — the throughput limits will constrict application growth.

**Lesson 4: Clock synchronization changes the game.**

Spanner's TrueTime eliminates the need for a second consensus round trip by replacing it with guaranteed clock bounds. This is the frontier of strongly consistent distributed databases. When evaluating databases, ask about their time-synchronization scheme. If they rely on NTP alone, their consistency guarantees degrade as clock skew grows.

**Lesson 5: Migration from eventual to strong is painful but eliminates a hidden tax.**

Before Spanner, Google engineers spent 15-20% of their code on consistency edge cases. After migration, that complexity disappeared. The "consistency tax" on application code is real and measurable. When choosing a database for a new project, consider not just the operational cost of strong consistency but the engineering cost of eventual consistency.

**Lesson 6: Strong consistency is a competitive advantage for developer productivity.**

When your database guarantees linearizability, your application developers write procedural code that matches their mental model. A read after a write returns the written value. A transaction either commits or aborts atomically. This eliminates an entire class of bugs: race conditions, stale reads, phantom writes. Google estimated that Spanner reduced the bug rate for transaction-handling code by 10x compared to Bigtable-based applications. The productivity gain came not from the database itself but from the reduced cognitive load on developers — they no longer needed to think about which replica they were reading from or whether a write had propagated.

**Lesson 7: Tunable consistency is the winning pattern — one size does not fit all.**

DynamoDB's per-request consistency choice is the most practical pattern to emerge from two decades of distributed systems research. A single application has data with different consistency needs. User authentication tokens need linearizable reads. Product recommendation scores can be 30 seconds stale. Activity feeds can be eventually consistent. By exposing consistency as a per-request parameter, DynamoDB lets you optimize for latency and cost on the 95% of reads while paying for strong consistency on the critical 5%. This pattern has been adopted by other systems: Azure Cosmos DB offers five consistency levels per request, and MongoDB supports read concern levels ranging from "local" (eventual) to "linearizable."

**Lesson 8: Consistency failures compound across service boundaries — test the entire call chain.**

A consistency failure in one service rarely stays contained. Consider a typical read path: your API gateway to your auth service to your database to your cache. If the database is strongly consistent but the cache serves stale data, the user sees the stale data. If the auth service is eventually consistent and returns a stale session token, the API gateway rejects a valid request. If service A writes to database X and service B reads from database Y (replicated from X with a 5-second delay), service B operates on stale data for 5 seconds.

LinkedIn encountered exactly this pattern in 2015. A cascading consistency failure across three services caused 45 minutes of degraded availability. The root cause was a configuration change in service A that increased its write rate to database X. The replication lag from database X to database Y spiked. Service B, reading from database Y, started seeing stale data. Service C, which depended on consistent output from both A and B, began throwing errors. The error rate propagated to the API gateway, which started rejecting requests.

The fix was not to change any database's consistency model. It was to add write-through caching in service B, test the full call chain under load, and add a circuit breaker that degraded gracefully when cross-service consistency was violated.

**Test strategy for cross-service consistency:**

1. **Replication lag injection:** Simulate 1s, 5s, and 30s of replication lag in your staging environment and observe which services degrade.
2. **Partial failure testing:** Take a downstream database offline and verify that upstream services degrade gracefully rather than crash.
3. **Timestamp drift injection:** Introduce artificial clock skew between services and verify that timestamp-based conflict resolution still converges.
4. **End-to-end consistency assertions:** Write integration tests that write a value to one service and immediately read it from another service, asserting the read returns a value no older than N seconds.

## Key Takeaways

- Google Spanner achieves external consistency across continents using TrueTime clock synchronization and Paxos. Its directory-based splitting allows sub-millisecond load balancing across Paxos groups.
- Amazon DynamoDB pioneered tunable consistency per operation — eventual by default, strong on request. DAX caches eventually consistent reads for microsecond latency, and Global Tables provide multi-region replication with LWW conflict resolution.
- Cassandra's AP design prioritizes availability but caused visible consistency bugs at Discord and Netflix. CRDT counters (as LinkedIn uses) provide an elegant solution for distributed counters where bounded inaccuracy is acceptable.
- ZooKeeper and etcd prove strong consistency is practical for coordination metadata at 10K writes/sec. Do not use them as primary databases — their throughput limits are intentional.
- GitHub's migration from MySQL to Vitess shows that sharding introduces three consistency challenges: cross-shard transactions, fan-out queries, and read-after-write sessions. Each has a workable solution, but none is free.
- Discord's migration from Cassandra (eventual) to ScyllaDB (strong reads) eliminated disappearing messages at the cost of 3ms additional latency. Stale-read rate dropped from 5% to 0.01%.
- The "consistency tax" on application code is real — Google estimated 15-20% of code dealt with consistency edge cases before Spanner.
- Tunable consistency is the winning pattern: use strong for critical paths, eventual for everything else.
- Consistency failures compound across service boundaries. Test the entire call chain under replication lag to prevent cascading failures.
