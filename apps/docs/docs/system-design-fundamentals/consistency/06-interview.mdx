## Interview Questions

Consistency questions appear in every system design interview. They test whether you understand the tradeoffs between correctness, latency, and availability.

### Five Common Questions

**Q1:** "Design a distributed key-value store. How do you handle consistency?"

**Q2:** "Our e-commerce site oversells products under high traffic. How do you fix it?"

**Q3:** "Design a distributed counter that never loses increments."

**Q4:** "Design a distributed locking service with strong consistency guarantees."

**Q5:** "Our chat application shows messages disappearing. Diagnose and fix."

### Three Advanced Questions

**Q6:** "Design a globally distributed database with external consistency."

**Q7:** "Design a conflict-free data structure for collaborative document editing."

**Q8:** "Design a coordination service for 10,000 microservices."

### Five Additional Questions

**Q9:** "Design a distributed cache with invalidation."

Model answer outline:

```
Requirements:
- Cache sits in front of a primary database
- Multiple cache nodes across data centers
- Data changes invalidate cache entries
- Staleness tolerance: under 100ms for hot keys
- Throughput target: 100K reads per second

Design:
- Use a write-through cache pattern: every write to the
  primary database also updates the cache synchronously.
- For multi-node caches, use a publish-subscribe channel
  (Redis Pub/Sub or Apache Kafka) to broadcast invalidation
  messages to all cache nodes.
- Each cache node subscribes to invalidation topics and
  evicts or refreshes entries on notification.
- For stale-tolerant reads, use time-to-live (TTL) as a
  safety net: set TTL to 60 seconds so stale entries are
  eventually evicted even if invalidation messages are lost.

Consistency dimension:
- Read-after-write: route cache-miss reads to the primary
  database to guarantee the latest value.
- Stale reads: accept a 100ms window where a cache node
  serves a stale value before the invalidation arrives.
- Edge case: if the invalidation message arrives before the
  write is committed to the primary, the next read misses
  the cache and hits a stale primary replica. Mitigate with
  quorum reads on the primary database.

Failure modes:
- Cache node failure: traffic rebuilds the cache from the
  primary database on miss (thundering herd — use
  request coalescing).
- Message bus failure: fall back to TTL-based eviction
  and direct cache invalidation on write.
```

**Q10:** "How does MongoDB handle consistency?"

Model answer outline:

```
MongoDB consistency model:
- By default, MongoDB provides strong consistency on
  reads from the primary node in a replica set.
- Write concern {w: "majority"} waits for the write to
  be committed to a majority of replica set members.
- Read concern "majority" returns data that has been
  committed by a majority of the replica set.
- Read concern "linearizable" guarantees that the read
  returns the most recent write acknowledged by a
  majority, at the cost of higher latency.

Tradeoffs to discuss:
- Default read preference ("primary") gives strong
  consistency but cannot serve reads if the primary
  is unavailable.
- Secondary reads ("primaryPreferred" or "secondary")
  improve availability but may return stale data.
- Causal consistency sessions in MongoDB allow a client
  to read its own writes across replica set members
  with the `afterClusterTime` field.

Edge cases:
- Rollback: if a secondary becomes primary before
  receiving all replicated writes, the un-replicated
  writes are rolled back. The application must handle
  these with idempotent retry or conflict detection.
- Multi-document transactions in MongoDB provide ACID
  within a replica set but incur coordination overhead.
```

**Q11:** "Design a distributed sequence number generator."

Model answer outline:

```
Requirements:
- Generate globally unique, monotonically increasing IDs
- Handle 100K requests per second
- Survive node failures without gaps or duplicates
- Must work without a central coordinator

Approach options (with tradeoffs):

Option A: Snowflake-style IDs (Twitter Snowflake)
   ┌─────────┬──────────┬────────────┬────────────┐
   │ 1 bit   │ 41 bits  │ 10 bits    │ 12 bits    │
   │ sign    │ timestamp│ worker ID  │ sequence   │
   └─────────┴──────────┴────────────┴────────────┘

- No coordination between workers.
- Each worker allocates a unique ID from its config.
- Sequence resets every millisecond.
- Maximum 4096 IDs per millisecond per worker.
- Time sync via NTP is required — clock skew causes
  collisions. Mitigate with worker-side clock monitoring
  and a safety margin of 10 seconds drift tolerance.

Option B: Database-backed segments
- Each worker claims a range of IDs from a database:
  "UPDATE sequence SET next_id = next_id + 1000
   WHERE name = 'orders';"
- Workers cache the range locally.
- If a worker dies, the unused segment is lost (gaps).
- Tune segment size: 1000 for 100K TPS means 100
  database queries per second for ID allocation.

Consistency dimension:
- Snowflake IDs are unique by construction: no two
  workers share the same worker ID + timestamp.
- Segment-based IDs require a lock or transaction on
  the database to claim a range, which becomes a
  bottleneck at scale.

Failure modes:
- NTP clock skew beyond 10ms: duplicate IDs.
  Mitigate by checking clock monotonicity before
  each batch and blocking until clock catches up.
- Database leader failover during segment allocation:
  the new leader may assign overlapping ranges.
  Mitigate with sticky allocation or fencing tokens.
```

**Q12:** "How would you design read-after-write consistency in a Key-Value store?"

Model answer outline:

```
Problem statement:
A client writes a value, then immediately reads.
Without read-after-write consistency, the read may hit
a different replica that has not yet received the write.

Design strategies:

Strategy 1: Session pinning (sticky sessions)
- Route all reads and writes from the same client
  session to the same replica (the one that accepted
  the write).
- Implementation: hash the session ID to select a
  replica. On replica failure, the session must be
  re-pinned or fall back to quorum reads.
- Limitation: if the pinned replica fails before
  replicating the write, the client sees the old value.

Strategy 2: Read-your-writes timestamp tracking
- Client maintains a timestamp of its last write.
- On read, the client sends the timestamp to any
  replica. If the replica's value is older than the
  timestamp, it waits until the write propagates or
  proxies the read to a replica with the latest value.
- Implementation: use vector clocks or Hybrid Logical
  Clocks (HLC) to track causality.

Strategy 3: Quorum reads with W+R > N
- Write quorum W includes at least one replica beyond
  the minimum, guaranteeing that some replicas overlap.
- Read quorum R includes enough replicas that at least
  one has the latest write.
- Example: N=3, W=2, R=2 ensures at least one replica
  in the read set overlaps with the write set.

Tradeoff comparison:
┌─────────────────┬────────────┬────────────┬────────────┐
│ Strategy        │ Latency    │ Complexity │ Partition  │
│                 │ (p99)      │            │ tolerance  │
├─────────────────┼────────────┼────────────┼────────────┤
│ Session pinning │ 2-5ms      │ Low        │ Low        │
│ Timestamp       │ 5-15ms     │ Medium     │ Medium     │
│ Quorum          │ 10-30ms    │ Low        │ Medium     │
└─────────────────┴────────────┴────────────┴────────────┘
```

**Q13:** "Design a consistent hash ring with virtual nodes for Dynamo-style replication."

Model answer outline:

```
Requirements:
- Distribute keys across N physical nodes
- Minimal key redistribution when nodes join or leave
- Each key replicated to R nodes for fault tolerance
- Handle hot spots with virtual nodes (vnodes)

Design:

Step 1: Create the hash ring
- Define a 128-bit or 256-bit hash space (e.g.,
  MD5 or SHA-1 hash of key).
- Arrange the hash space as a ring from 0 to 2^128 - 1.
- Place each physical node at multiple points on the
  ring by hashing "node_id:vnode_index" for each of
  150-200 virtual nodes per physical node.

Step 2: Place keys
- Hash the key to a point on the ring.
- Travel clockwise from the hashed position.
- Assign the key to the first K virtual nodes encountered.
- Map each virtual node to its physical node.
- Each key is stored on the R physical nodes that own the
  first R virtual nodes clockwise from the key position.

Step 3: Replication with preference list
- The preference list for a key is the list of R physical
  nodes responsible for that key.
- Writes go to all R nodes in the preference list.
- Reads query all R nodes and return the latest version
  based on vector clock comparison.

Consistency dimension:
- With N=3, W=2, R=2, the system provides strong
  consistency for Ansible-available nodes.
- With W=1, R=1, the system provides eventual consistency
  with a convergence window determined by hinted handoff
  and read repair.

Edge cases:
- Node addition: a new physical node with 200 virtual
  nodes claims ownership of keys between its placement
  points and the previous virtual node. Only those keys
  are redistributed.
- Node failure: virtual nodes on the failed physical
  node are removed from the ring. Keys are reassigned
  to the next available virtual node. Hinted handoff
  buffers writes until the node recovers.
```

## Signals Interviewers Look For

### Strong Hire Signals

- Identifies the consistency model needed before proposing a system design.
- Draws consistency-availability (CAP) tradeoffs explicitly and ties them to business requirements.
- Uses quorum math (W+R > N) naturally when discussing read and write paths.
- Explains how Raft or Paxos achieves consensus without hand-waving.
- Mentions read-repair, hinted handoff, or CRDTs as operational mechanisms.
- Discusses real-world systems (Spanner, DynamoDB, etcd) and why they made specific design choices.
- Asks clarifying questions about staleness tolerance before proposing solutions.
- Articulates the difference between quorum consistency and linearizability, noting that W+R > N guarantees strong read guarantees but does not guarantee total ordering across all operations.
- Quantifies staleness windows in milliseconds, not hand-wavy terms like "eventually."
- Draws a detailed write path: client sends request to leader, leader appends to log, replicates to followers, waits for majority acknowledgment, then commits.
- Brings up clock synchronization issues unprompted when discussing timestamp-based ordering.

### Strong Hire Behaviors

What a strong hire does in the first 90 seconds of Q1:

```
"Before I propose a design, I need to understand the
requirements. What is the staleness tolerance for
this key-value store? Is the workload read-heavy or
write-heavy? What are the throughput and latency SLAs?"

"If the workload demands strong consistency — for
example, account balances — I would use linearizable
consistency with Raft. A 5-node Raft cluster gives
us fault tolerance with W=3, R=3. That gives
W+R=6 > 5, guaranteeing strong consistency while
tolerating up to 2 node failures."

"On the write path: the client sends the write to the
Raft leader. The leader appends the entry to its log,
replicates it to the followers, and commits once a
majority acknowledges. On the read path: I route
reads through the leader for linearizability."

"If the workload can tolerate staleness — for example,
a product catalog — I would offer tunable consistency.
The client can choose between leader reads (strong)
or replica reads (eventual, under 100ms staleness).
Read repair and hinted handoff converge replicas
in the background."
```

### Neutral Signals

- Knows CAP theorem but cannot explain how to apply it to a specific design.
- Mentions Raft by name but cannot describe log replication steps.
- Proposes strong consistency for everything.
- Can describe DynamoDB's tunable consistency at a high level.
- Mentions quorum math but does not calculate concrete W and R values for a given N.
- Talks about vector clocks but cannot describe how to detect a conflict from two clock values.

### Weak Signals

- Insists on ACID for every distributed system.
- Claims eventual consistency is "always wrong" or "only for caching."
- Cannot define the difference between linearizable and sequential consistency.
- Does not mention network partitions.
- Proposes only one consistency strategy for the entire system.
- Says "just use a database" without specifying which database and why.
- Cannot name a single real-world system that uses each consistency model.

## Knowledge Levels

### Level 1: Awareness

You know that consistency exists as a concept. You understand CAP theorem at a memorization level.

### Level 2: Application

You can choose the right consistency model for a given system. You use quorum math (W+R > N) to reason about read and write behavior. You know the tradeoffs between strong and eventual consistency. You understand that read-after-write consistency is a common middle ground.

### Level 3: Mechanism

You understand how Raft achieves consensus: leader election, log replication, majority commitment. You can describe vector clocks and how they detect concurrent writes. You understand CRDT merge semantics. You can explain read repair and hinted handoff at the wire-protocol level.

### Level 4: Production

You have debugged a consistency violation in production. You know how to migrate from eventual to strong consistency. You understand the performance characteristics of quorum reads under load. You can reason about the interaction between consistency and other system properties (caching, CDN, async processing). You have tuned Raft election timeouts to avoid split-brain scenarios in high-latency networks.

### Level 5: Design

You can design a new consistency protocol or modify an existing one for a novel workload. You understand the implications of TrueTime on distributed consensus. You could implement Raft from the specification. You understand the limitations of FLP impossibility and how practical systems circumvent it with failure detectors and timeouts.

## 26 Phrases to Use

Use these phrases naturally in your answer. They signal depth.

**Requirement gathering:**
1. "What is the staleness tolerance for this data?"
2. "Does the same client read their own writes within a session?"
3. "How does this system behave during a network partition — can it reject writes?"
4. "What is the read-to-write ratio for this workload?"
5. "Is the operation idempotent? Does the client retry on timeout?"
6. "What is the p99 latency SLA? Does it differ between reads and writes?"

**Model selection:**
7. "I need W+R > N to guarantee strong consistency."
8. "This data can tolerate eventual consistency with a 2-second convergence window."
9. "Causal consistency is sufficient here because operations only need to respect happens-before, not total order."
10. "Read-after-write consistency solves the 'my update disappeared' problem without the cost of global strong consistency."
11. "For this workload, tunable consistency lets the client choose between latency and correctness per request."
12. "Monotonic reads guarantee that a client never sees a value older than the last one it observed."

**Mechanism design:**
13. "I will use a 3-node Raft cluster with synchronous replication for the write path."
14. "Vector clocks track causal history. If neither clock dominates, we have a conflict needing resolution."
15. "CRDTs merge deterministically without a coordinator. For a counter, each node tracks its own increments."
16. "Read repair on the read path fixes stale replicas without requiring synchronous writes."
17. "Session pinning ensures that all reads from a client within a session go to the replica that accepted the write, which prevents the phantom read problem."
18. "Read repair propagates the latest version to stale replicas on each read, converging over multiple read cycles in under 100ms within a single data center."
19. "CRDT merge semantics ensure that concurrent updates from any two nodes produce the same deterministic result regardless of arrival order — convergence is guaranteed by the algebraic properties of the data type."
20. "The write-ahead log persists every mutation before applying it, guaranteeing durability. On recovery, the node replays committed entries from the log to restore state."
21. "Snapshot isolation guarantees that a read transaction sees a consistent snapshot of the database as of the transaction start time, with no dirty reads or non-repeatable reads."
22. "Multi-version concurrency control maintains separate versions of each row per transaction, allowing reads to proceed without blocking writes and providing snapshot isolation."
23. "Gossip protocol converges membership state across the cluster within O(log N) rounds, bounded by the failure detection interval of 500ms per round."
24. "Hinted handoff stores the write on a coordinator node when the target replica is unavailable, replaying it within 60 seconds when the replica recovers."
25. "Quorum consistency with W+R > N provides strong read guarantees but does not guarantee linearizability — a stale leader could serve an old value during an election window."
26. "When a partition heals, the two sides reconcile divergent state through anti-entropy repair. The Merkle tree comparison identifies differing key ranges in O(log N) time."

**Validation:**
27. "During a network partition, the minority partition will block writes — that is the CP choice."
28. "If the leader fails, uncommitted log entries may be lost. Clients must retry with idempotency keys."
29. "LWW loses the older write. If data loss is unacceptable, use CRDTs or application-level merge."
30. "Read-after-write requires session pinning. Without it, the next request may hit a different replica."
31. "If the clock skew exceeds the safety margin, timestamp-based ordering breaks. Use Hybrid Logical Clocks to bound this risk."
32. "Network delays can cause a client to receive a success response for a write that the cluster has not committed. Implement client-side timeouts and retry with idempotency tokens."

## 13 Red Flags

These statements reveal gaps in your understanding. Avoid them.

**Red Flag 1:** "I will use ACID in a distributed database."

Wrong for distributed systems. ACID implies a single-node transaction manager. In distributed systems, ACID is either impossible (isolation) or prohibitively expensive (atomicity across nodes). Say "I use linearizable transactions via Raft" instead.

**Red Flag 2:** "CAP theorem says you can only have two of three."

Wrong. During a partition, you choose between CP and AP. Outside a partition, you can have both consistency and availability. Say "CAP classifies behavior during partitions, not normal operation."

**Red Flag 3:** "Eventual consistency eventually becomes consistent."

Too vague. Be specific: "Eventual consistency converges within a bounded window. With Cassandra and W=2, R=2, the convergence window is approximately 100ms in the same data center."

**Red Flag 4:** "Just use strong consistency everywhere."

Reveals a lack of understanding about the latency and availability costs. Strong consistency halves throughput and multiplies latency. No production system uses strong consistency for all data.

**Red Flag 5:** "I will use Paxos because it is the standard."

Paxos may be the original consensus protocol, but it is notoriously hard to implement correctly. Nobody builds new systems on Paxos. Raft is the standard. Say "I deploy Raft because of its understandable leader-based model."

**Red Flag 6:** "Consistency is a database problem, not an application problem."

Wrong. Application-level decisions (session handling, retry logic, conflict resolution) directly affect the consistency the user experiences. Say "The database provides guarantees, but the application must handle edge cases like retries and session pinning."

**Red Flag 7:** "Leaderless replication is always better for availability."

Leaderless systems (Dynamo, Cassandra) handle node failures more gracefully but introduce write conflicts. There is no free lunch. Say "Leaderless replication improves availability during partitions but requires conflict resolution on reads."

**Red Flag 8:** "My system will never have a network partition."

Partitions are inevitable in any distributed system spanning more than one machine. Design for them. Say "I assume the network will partition at some point. My design handles it by..."

**Red Flag 9:** "Two-phase commit solves all distributed consistency problems."

Two-phase commit (2PC) is a distributed atomic commit protocol, not a consensus protocol. 2PC blocks on coordinator failure (it is a blocking protocol). It provides atomicity but not fault tolerance. If the coordinator fails after sending the PREPARE message but before sending COMMIT, participants are stuck in a prepared state holding locks indefinitely. Say "I use three-phase commit for non-blocking atomic commit, or Raft for consensus-based replication."

**Red Flag 10:** "Just use a timestamp to order writes" without discussing clock skew.

Timestamps from wall clocks are not reliable across machines. NTP synchronization gives precision of approximately 1-50ms depending on the data center topology, but clock skew can exceed 100ms during network congestion. Two events on different machines can get the same timestamp or out-of-order timestamps. Say "I use Hybrid Logical Clocks (HLC) or TrueTime to bound clock uncertainty to within 10ms for ordering."

**Red Flag 11:** Not knowing the difference between ZooKeeper and etcd.

Both are coordination services, but they have different architectures and guarantees. ZooKeeper uses Zab (a Paxos variant) and provides sequential consistency with linearizable writes. etcd uses Raft and provides linearizable reads. ZooKeeper sessions are time-bound (session timeout, typically 10-30 seconds). etcd uses lease-based heartbeats (default 1 second). Say "I choose etcd when I need linearizable reads and a simpler leader-based model. I choose ZooKeeper when I need hierarchical znodes and Apache Kafka integration."

**Red Flag 12:** Claiming that CRDTs solve everything.

CRDTs eliminate the need for a central coordinator but have limitations. State-based CRDTs (CvRDTs) require the entire state to be propagated on each sync, which can be large for complex data types. Operation-based CRDTs (CmRDTs) require reliable delivery of operations, which is hard in an unreliable network. CRDTs also struggle with schema evolution and garbage collection of tombstones. Say "CRDTs are a good fit for specific data types like counters, sets, and maps, but they require careful design for custom data structures."

**Red Flag 13:** "I will use eventual consistency because it is faster" without quantifying the staleness window.

This reveals a lack of precision. The interviewer will ask "how fast?" and "how stale?" Be specific: "I use eventual consistency with a 200ms convergence window. Writes are acknowledged in under 5ms. Reads return in under 2ms from the nearest replica. If the replica is more than 200ms behind the leader, the read misses and hits the primary."

## 8 Drills

Practice these drills in mock interviews to internalize consistency reasoning.

### Drill 1: The Quorum Calculator

Given N (replication factor) and desired consistency level, calculate W and R.

**Example:** N=5, you need strong consistency but want to minimize write latency.

```
Question: What W and R do you choose?
Answer:    W=3, R=3 -> W+R=6 > 5, strong.
           W=2, R=4 -> W+R=6 > 5, also strong, faster writes.
           W=1, R=5 -> W+R=6 > 5, fastest writes, slowest reads.

Better answer: W=3, R=3. Tolerates 2 node failures.
               W=2, R=4 tolerates 3 failures for writes,
               2 for reads. Pick based on read:write ratio.
```

**Practice with:** N=3, N=5, N=7. Calculate (W,R) pairs for strong. Calculate minimum (W,R) to tolerate 1 or 2 node failures.

### Drill 2: The Partition Simulation

Given a system diagram, simulate a network partition and explain what happens.

**Example:** 5-node Raft cluster, split into group of 3 and group of 2.

```
Group of 3: Has quorum (3 of 5). Elects leader. Accepts writes.
Group of 2: No quorum. Cannot elect leader. Rejects writes.
On recovery: Group of 2 rejoins, catches up on log entries.
Writes from minority during partition: rejected by the protocol,
but a client might have received a success response before
the partition started and the value might not be replicated.
```

**Practice with:** Dynamo-style N=3, partition splits 2 vs 1. What happens to reads? Writes?

### Drill 3: The Staleness Estimator

Given replication topology and network latency, estimate the maximum staleness window.

**Example:** Leader in us-east-1. Async replica in eu-west-1. RTT is 80ms. Replication batch interval is 20ms.

```
Max staleness = RTT + batch_interval + buffer
              = 80ms + 20ms + 10ms
              = 110ms

During that window, a read from the eu-west replica
returns the stale value. For strict deadlines (e.g.,
you promised users 50ms staleness), this means eu-west
reads must go through the leader or a quorum.
```

**Practice with:** 3-region topology (us-east, eu-west, ap-southeast). Calculate staleness for each region.

### Drill 4: The Conflict Scenario

Given a sequence of writes from two clients, determine whether there is a conflict.

**Example:**

```
Client A: write(x=1) at T=0
Client B: write(x=2) at T=1
Client A: write(x=3) at T=2
Client B: read(x) at T=3
```

With vector clocks: Client A's writes produce v_A=[A:2], Client B's write produces v_B=[A:1, B:1]. The read sees both versions. Neither dominates. Conflict.

**Practice with:** Generate three sequences of concurrent writes and determine which produce conflicts and which are causally ordered.

### Drill 5: The Migration Planner

Given a system currently using eventual consistency, plan the migration to strong consistency.

```
Step 1: Audit all data access patterns.
Step 2: Classify data into tiers (critical, important, non-critical).
Step 3: Add per-request consistency flag (default: eventual).
Step 4: Shadow-read with strong for critical data (5% of traffic).
Step 5: Compare results, measure staleness, fix issues.
Step 6: Roll out strong for critical paths.
Step 7: Monitor p99 latency and error rate.
Step 8: Set up rollback: flip the flag back to eventual.
```

**Practice with:** Map each step to a specific system (e.g., an e-commerce platform with inventory, catalog, user profiles, and analytics).

### Drill 6: The Protocol Selector

Given a system description, choose the appropriate protocol and justify your choice.

**Example:** You need to replicate a configuration file across 10 servers. All servers must agree on the contents at all times. Writes are infrequent (once per hour). Read-after-write consistency is required.

```
Candidate protocols:
- Raft: leader-based consensus, easy to implement.
- Paxos: classic consensus, harder to implement.
- Zab: used by ZooKeeper, similar to Raft.
- Gossip: eventual, no consensus.
- CRDT: no consensus, commutative updates.

Choice: Raft with N=5. Three voting members handle
consensus. Two non-voting observers replicate the
committed log asynchronously.

Rationale:
- Writes are infrequent, so leader election overhead
  (300ms) is negligible.
- Raft provides linearizable reads through the leader.
- N=3 would survive 1 failure; N=5 survives 2.
- Gossip would not guarantee read-after-write.
- CRDTs are overkill for a configuration file that
  is replaced atomically, not merged.
```

Drill scenarios to practice:

```
Scenario A: A distributed counter shared across 100
nodes, each incrementing 1000 times per second.
Partitions are rare but possible.

Which protocol? Why?
```

```
Scenario B: A DNS-like service mapping hostnames to
IP addresses. Reads are 99.99% of traffic. Staleness
of 60 seconds is acceptable.

Which protocol? Why?
```

```
Scenario C: An auction system where bids on the same
item must be totally ordered. Bids arrive at 10K TPS.

Which protocol? Why?
```

### Drill 7: The Compose Drill

Given a stack of three services — API, database, cache — trace a write through the stack and identify where consistency can break.

**System topology:**

```
[Client] --> [API Service (3 nodes)]
                 |
                 +--> (( Primary DB - Primary replica ))
                 |        +--> (( Read replica 1 ))
                 |        +--> (( Read replica 2 ))
                 |
                 +--> { Cache cluster (Redis, 3 nodes) }
```

**Write flow:**

```
1. Client POST /users {name: "Alice", email: "a@b.com"}
2. API node receives request, validations pass.
3. API node writes to primary DB: INSERT INTO users ...
4. Primary DB commits and asynchronously replicates
   to read replicas.
5. API node updates Redis cache: SET users:1 ...
6. API node returns 201 to client.

BREAK 1: Step 4 is async. If a read hits a read replica
before replication completes, the read sees a missing row.

BREAK 2: Step 5 updates the cache, but if another API
node served a GET for user 1 between step 3 and step 5,
that node cached the empty result (cache-aside pattern).

BREAK 3: If the primary DB fails between step 3 and
step 5, the cache has stale data that must be invalidated
on failover.

BREAK 4: If the Redis write fails (step 5), the API
node must decide whether to return 201 (the DB write
succeeded but the cache is stale) or 500 (fail the
entire request).

BREAK 5: Concurrent writes from two API nodes to the
same user row cause a lost update if the DB does not
use optimistic locking.
```

**Trace through each break and propose a fix:**

```
Break 1: Route reads through the primary for 1 second
after write, or use a read-after-write timestamp vector
that includes the DB commit sequence number.

Break 2: Use cache-invalidation-on-write (delete the
cache key before writing to DB) or compare-and-swap
in Redis.

Break 3: Subscribe to DB failover events and
invalidate the entire cache partition on failover.

Break 4: Use a write-through cache pattern: if the
cache update fails, return 200 but log the failure
for async repair.

Break 5: Add an optimistic lock column (version number)
to the user table. The UPDATE statement checks the
version matches and increments it atomically.
```

### Drill 8: The Tradeoff Quantifier

Given two alternative designs, calculate the latency, throughput, and availability difference in concrete numbers.

**Scenario:** E-commerce checkout system. 1000 writes per second. 10,000 reads per second. N=3. You must decide between always-strong and tunable consistency.

**Design A: Always strong (W=3, R=2, leader reads)**

```
Write path: client -> leader -> replicate to 2 followers
            -> wait for 2 acks -> commit
Write latency: 2ms (local write) + 2ms (RTT to 2 followers)
               + 0.5ms (commit) = 4.5ms p50, 12ms p99

Read path: leader read
Read latency: 2ms p50, 5ms p99

Write throughput (leader bottleneck):
  Leader handles 1000 writes/sec + 10,000 reads/sec
  = 11,000 operations/sec on the leader node.

Availability:
  Tolerates 1 node failure (still has 2 of 3).
  If the leader fails: 300ms election downtime,
  then reads resume. Writes resume after election.

Total write throughput with N=3, W=3:
  Limited by the slowest follower. If followers
  have p99 latency of 10ms, max write throughput
  is approximately 100 writes/sec per follower
  throughput before queuing. At 1000 writes/sec,
  you need to partition across multiple Raft groups.
```

**Design B: Tunable (W=2, R=2 for critical; W=1, R=1 for non-critical)**

```
Write path (critical): client -> any replica -> replicate
            to 1 other replica -> ack
Write latency: 1.5ms p50, 4ms p99 (one fewer RTT)

Write path (non-critical): client -> any replica -> ack
Write latency: 0.5ms p50, 1ms p99 (no replication wait)

Read path (critical): quorum read (2 of 3)
Read latency: 3ms p50, 8ms p99 (query 2, wait for faster)

Read path (non-critical): closest replica read
Read latency: 0.5ms p50, 2ms p99

Write throughput (critical): 5000 writes/sec per node
Write throughput (non-critical): 20,000 writes/sec per node

Availability:
  Tolerates 1 node failure for critical operations
  (W=2 means 1 failure leaves 2 nodes, still can write).
  Tolerates 2 node failures for non-critical operations
  (W=1 means 1 live node can accept writes).

Staleness for non-critical reads:
  Max staleness = gossip propagation time + clock skew
                = 100ms (intra-DC) + 10ms buffer
                = 110ms max staleness window
```

**Quantified difference:**

```
Metric                Design A          Design B
Write latency p99     12ms              4ms (critical)
                                         1ms (non-critical)
Read latency p99       5ms              8ms (critical)
                                         2ms (non-critical)
Write throughput      ~1000/sec         ~5000/sec (critical)
                                        ~20,000/sec (non-critical)
Read throughput       ~10,000/sec       ~10,000/sec
Node failures         Tolerates 1       Tolerates 1 (critical)
  tolerated                              Tolerates 2 (non-critical)
Leader election       300ms             None (leaderless)
  downtime

Verdict: Design B provides 3x lower write latency and
5x higher write throughput at the cost of 3ms higher
read latency for critical operations and a 110ms
staleness window for non-critical reads.
Choose Design B when the business can tolerate 110ms
staleness for 90% of reads. Choose Design A when
every read must be linearizable.
```

## Key Takeaways

- Every system design interview includes a consistency dimension — prepare for it with concrete numbers and specific protocol names.
- The four-step framework (requirements, model, mechanism, validation) structures clean answers that cover what the interviewer needs to hear.
- Use specific phrases that signal depth: quorum math, vector clock comparison, session pinning, read repair, hint replay, gossip convergence.
- Avoid 13 common red flags that reveal shallow understanding: vague CAP usage, ACID for distributed, strong everywhere, Paxos default, ignoring partitions, 2PC as a cure-all, timestamps without clock sync, CRDT blind spots.
- Run 8 drills until they become automatic: quorum calculator, partition simulation, staleness estimator, conflict scenario, migration planner, protocol selector, compose drill, tradeoff quantifier.
- Know at least three real-world systems and their consistency models: Spanner (external consistency), DynamoDB (tunable), Cassandra or etcd (CP versus AP positioning).
- A strong hire quantifies everything — staleness windows in milliseconds, quorum math with concrete N, W, R values, and latency numbers for each path in the architecture.
- Continue to the case studies section to apply these patterns to real-world consistency problems in detail.
