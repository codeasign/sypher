## Tier 1: Foundation

These challenges test your grasp of the basic concepts and terminology. You should be able to answer each in 3-5 minutes.

### Challenge 1.1: The Disappearing Like

A social media platform shows the like count on posts. User A likes a post. When User A refreshes, the count shows 142. User B, viewing the same post from another region, sees 141 likes. The count eventually shows 142 for User B after 4 seconds.

**Question:** Identify the consistency model at play. Explain what User B experienced and why.

<details>
<summary>Model Answer</summary>

The system uses eventual consistency. User B's read hit a replica that had not yet received the update from User A's write. The write took approximately 4 seconds to propagate across regions.

The specific sequence:

1. User A sends a like — the write reaches the leader in region 1.
2. The leader acknowledges the write and increments the count to 142.
3. Replication to region 2 begins asynchronously.
4. User B reads from a replica in region 2 before replication completes — sees 141.
5. After 4 seconds, the replication arrives at region 2. Subsequent reads return 142.

This is acceptable behavior for a social media like count. The staleness window (4 seconds) is invisible to most users, and the cost of strong consistency (2-5x higher write latency) would degrade the posting experience for all users.

**Alternative answer if user experience demands better:** Implement read-after-write consistency by pinning the writer's session to the leader for 5 seconds after a write. Subsequent refreshes from User A would always see the latest count, without requiring all users to pay the strong-consistency cost.

</details>

### Challenge 1.2: The Quorum Calculator

You are designing a distributed database with 7 replicas (N=7). The workload is 80% reads, 20% writes. The application requires strong consistency.

**Question:** Calculate the minimum W and R values that satisfy the strong consistency requirement. Then suggest a (W, R) pair optimized for this read-heavy workload. Explain why.

<details>
<summary>Model Answer</summary>

The strong consistency formula is W + R > N.

With N=7:
- Minimum pairs where W+R > 7:
  - W=1, R=7 (W+R=8 > 7)
  - W=2, R=6 (W+R=8 > 7)
  - W=3, R=5 (W+R=8 > 7)
  - W=4, R=4 (W+R=8 > 7)
  - W=7, R=1 (W+R=8 > 7)

For a read-heavy workload (80% reads), optimize for fast reads by minimizing R and increasing W:

- **Recommended: W=4, R=4.** This balances both paths tolerates 3 node failures for writes and 3 for reads. With 7 replicas, you can lose 3 nodes and still operate.
- **Alternative: W=6, R=2.** Reads are fast (only need 2 responses) but writes are slow (need 6 of 7). This works if reads dominate heavily and write throughput is not a bottleneck.
- **Not recommended: W=1, R=7.** Reads must contact every node — the slowest node sets the read latency. A single slow or overloaded node multiplies p99 read latency by 7x.

The W=4, R=4 pair provides fault tolerance (tolerates 3 node failures), reasonable latency on both paths (parallel reads to 4 of 7 nodes), and meets the strong consistency requirement.

</details>

### Challenge 1.3: Session Consistency

A user updates their profile picture on a social network. The user refreshes the page and sees the old picture. Refreshing again 3 seconds later shows the updated picture.

**Question:** What consistency guarantee is violated? Propose a fix that does not require global strong consistency.

<details>
<summary>Model Answer</summary>

Read-after-write consistency (also called read-your-writes consistency) is violated. The user performed a write (profile picture upload) and then performed a read (page refresh) that did not reflect their own write.

This is the most visible consistency violation in user-facing systems because the user knows they just performed an action and expects to see the result.

**Fix without global strong consistency:** Implement session pinning with read-after-write guarantee.

1. When the user uploads a picture, the server creates a session cookie with a replica ID.
2. For the next N seconds (typically 5-30), the user's reads are pinned to the replica that received the write (the leader).
3. After the pinning window expires, reads can go to any replica.

```python
# Pseudocode for session pinning
class SessionPinnedClient:
    def __init__(self):
        self.pinned_replica = None
        self.pin_expiry = 0

    def write_profile(self, user_id, data):
        response = leader.write(user_id, data)
        self.pinned_replica = leader.id
        self.pin_expiry = time.now() + 5  # 5 second pin
        return response

    def read_profile(self, user_id):
        if time.now() < self.pin_expiry:
            return self.pinned_replica.read(user_id)
        return nearest_replica.read(user_id)  # eventual
```

This adds no extra latency to writes and minimal routing logic to reads. Only the writing user pays the "strong" cost, and only for 5 seconds.

</details>

### Challenge 2.1: Distributed Ticket Counter

A ticketing platform sells 10,000 tickets for a concert. The system uses 3 web servers and a load balancer. Under high traffic, the system oversells tickets because two servers read the same remaining count and both decrement it.

**Question:** Design a distributed counter that never oversells. Provide at least two approaches ranked by performance.

<details>
<summary>Model Answer</summary>

**Approach 1: Raft-based counter (strong consistency, moderate throughput)**

Use a 3-node Raft cluster dedicated to the counter. Every ticket sale is a linearizable compare-and-swap operation:

```
1. Client requests to buy 1 ticket.
2. Coordinator sends read to Raft cluster → current count = 234.
3. If count > 0, coordinator sends "CAS count=234 → 233" to Raft.
4. Raft replicates the operation. If CAS succeeds, ticket is sold.
5. If CAS fails (someone else bought in between), retry from step 2.
```

Throughput: approximately 5,000-10,000 tickets/second (limited by Raft leader).

```python
class RaftCounter:
    def __init__(self, raft_client):
        self.raft = raft_client

    def buy_ticket(self) -> bool:
        while True:
            # Atomic read-modify-write via compare-and-swap
            current = self.raft.linearizable_read("tickets_remaining")
            if current <= 0:
                return False  # sold out
            success = self.raft.compare_and_swap(
                key="tickets_remaining",
                expected=current,
                new_value=current - 1
            )
            if success:
                return True
            # CAS failed → retry
```

**Approach 2: Sharded batch counter (high throughput, eventual + reconciliation)**

Pre-allocate ticket blocks to each server:

```
Server A gets: tickets 1-3334
Server B gets: tickets 3335-6667
Server C gets: tickets 6668-10000

Each server sells from its own block independently.
No coordination needed between servers.
```

This eliminates consistency issues entirely for the common case. The tradeoff: if Server A's block sells out but Server C still has tickets, you may need a secondary allocation step. However, for a single concert event, pre-allocating evenly removes all synchronization overhead.

Throughput: 100,000+ tickets/second (limited only by each server's local database).

**Approach 3 (bonus): CRDT G-Counter with a monotonically increasing sell order**

Use a grow-only counter (G-Counter) where each replica tracks its own increments. Total sold is the sum of all replicas. The ticket sale is valid if total sold < capacity.

```
Replica A increments its P counter: P_A += 1
Replica B increments its P counter: P_B += 1
Total sold = P_A + P_B + P_C
Sell is accepted if total sold <= 10,000
```

This allows concurrent increments without locks. The potential oversell is at most N-1 tickets (one per replica racing). For a 10,000-ticket event, a 2-ticket oversell is tolerable; you refund the last 2 buyers.

</details>

### Challenge 2.2: Multi-Region Chat

A chat application operates in 3 regions. Messages must appear in order within a conversation. Users in different regions see messages in different orders and occasionally miss messages.

**Question:** Design the consistency model for the chat system. Provide the specific (W, R) configuration and explain how you handle ordering.

<details>
<summary>Model Answer</summary>

**Consistency model: Causal consistency with monotonic reads.**

Causal consistency preserves the order of causally related messages (a reply after the original message) but allows concurrent messages from different users to appear in different orders, which is acceptable in chat.

**Configuration:**

```
Replication factor: N=3 (one replica per region)
Write quorum: W=2 (must ACK in 2 of 3 regions)
Read quorum: R=2 (consult 2 replicas on read)

W+R = 4 > 3 = N → strong consistency for each write
But ordering is causal, not linearizable.
```

**Ordering mechanism:**

Each message carries a vector clock. The client's vector clock is included in every read request. The server returns messages that are causally after the client's clock, plus any concurrent messages.

```json
{
  "conversation_id": "abc123",
  "messages": [
    {
      "id": "msg_456",
      "author": "alice",
      "text": "See you at 8pm",
      "vector_clock": {"region_a": 5, "region_b": 3, "region_c": 4}
    },
    {
      "id": "msg_457",
      "author": "bob",
      "text": "Perfect!",
      "vector_clock": {"region_a": 5, "region_b": 4, "region_c": 4}
    }
  ],
  "client_clock": {"region_a": 4, "region_b": 3, "region_c": 4}
}
```

The client tracks its own vector clock. On each read, it sends its clock. The server returns only messages that are causally after the client's clock. The client merges the new messages into its local view, respecting causal order.

**Message delivery guarantee:** Messages are delivered in causal order, not wall-clock order. If Alice sends "See you at 8pm" and Bob replies "Perfect!", all clients see "See you at 8pm" before "Perfect!" because the reply carries a vector clock that depends on the original message. Concurrent messages from unrelated conversations may appear in different orders on different clients — this is acceptable in chat.

</details>

### Challenge 2.3: Inventory Read Path

An e-commerce platform experiences a "flash crash" during Black Friday: a database failover caused 4 seconds of stale inventory reads, resulting in 200 oversold items across 50 SKUs.

**Question:** Design a read path that prevents stale reads during database failover without increasing p99 read latency by more than 5ms.

<details>
<summary>Model Answer</summary>

**Root cause:** The failover promoted an async replica to primary. The replica was 200ms behind on replication. For 4 seconds after failover, reads returned inventory counts from before the last 200ms of writes.

**Solution: Quorum reads with a read coordinator.**

Replace single-node reads with a lightweight quorum read against 2 of 3 database nodes.

```
Before (single-node):
  App → Read Replica A → returns v=3 (stale during failover)

After (quorum read):
  App → Read Coordinator
    → sends read to Replica A (v=3) and Replica B (v=5)
    → returns v=5 (latest)
    → if replicas disagree, checks version vector
```

**Implementation:**

```python
class QuorumReadCoordinator:
    def __init__(self, nodes: list[str], R: int = 2):
        self.nodes = nodes  # 3 database nodes
        self.R = R

    def read_inventory(self, sku: str) -> int:
        results = []
        for node in self.nodes[:self.R]:
            version, count = node.read_with_version(f"inv:{sku}")
            results.append((version, count))

        # Merge: return the highest version
        latest = max(results, key=lambda x: x[0])
        return latest[1]
```

**Latency impact:** Parallel reads to 2 nodes mean the slowest node sets the response time. In practice, with same-DC latency of 0.5ms per node, the p99 increases from 0.5ms (single read) to approximately 1ms (quorum read waiting for the slower of two).

**Additional safeguard: Read-repair.** If the quorum detects version disagreement, the coordinator writes the latest version back to the stale node asynchronously. This prevents future stale reads from that node without blocking the current read.

</details>

### Challenge 3.1: Global Financial Ledger

Design a globally distributed financial ledger system that supports:

- 50 million accounts
- 10,000 transactions per second globally
- Strong consistency (no double-spending, no negative balances)
- Multi-region deployment (US, EU, Asia)
- p99 write latency under 200ms
- 99.999% uptime

<details>
<summary>Model Answer</summary>

**Architecture overview:**

Use Google Spanner as the reference architecture — external consistency via TrueTime-style clock synchronization with Raft-based consensus within each shard.

**Sharding:**

Partition accounts by account ID hash into 256 shards. Each shard is an independent Raft group of 5 nodes deployed across 3 regions (US: 2, EU: 2, Asia: 1 — or similar distribution).

```
Shard 0 (accounts 0000-0FFF): Raft group with 5 nodes
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │US-EA1│ │US-EA2│ │EU-WE1│ │EU-WE2│ │AS-SE1│
  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘

Shard 1 (accounts 1000-1FFF): same topology
... 256 shards total
```

**Consistency mechanism:**

Each shard runs Raft for linearizable operations. Cross-shard transactions use two-phase commit with a transaction coordinator.

```
Transaction: Transfer $100 from Account A (Shard 0) to Account B (Shard 1)

1. Coordinator contacts Shard 0's leader: "prepare: debit A $100"
   → Shard 0 checks balance, locks A, responds "prepared"
2. Coordinator contacts Shard 1's leader: "prepare: credit B $100"
   → Shard 1 locks B, responds "prepared"
3. Coordinator sends "commit" to both shards
   → Both shards apply the change atomically
4. Coordinator sends "done" to client
```

**Clock synchronization:**

Use GPS clocks in each data center with a bounded uncertainty of 2ms (achievable with GPS + PTP). This allows TrueTime-style commit wait: after committing a transaction, the system waits for the uncertainty bound to expire before making the result visible, guaranteeing linearizability.

**Performance budget:**

| Operation | Time |
|---|---|
| Intra-shard Raft commit | 10ms (2 RTT × 5ms intra-region) |
| Two-phase prepare | 20ms (prepare to 2 shards in parallel) |
| Two-phase commit | 10ms |
| Commit wait (clock uncertainty) | 2ms |
| Total cross-shard tx | ~42ms |
| Intra-shard tx | ~10ms |

To achieve 10,000 TPS globally:
- 256 shards × ~40 TPS per shard = 10,240 TPS
- Each shard can handle ~40-50 linearizable transactions per second with Raft
- Throughput scales linearly with shard count

**Failover handling:**

- If a Raft leader fails, a new leader is elected within 300ms.
- If an entire region fails, shard leaders are re-elected in remaining regions.
- Read quorum: R=3 out of 5. Write quorum: W=3 out of 5.
- During a single-region failure: still 3 of 5 nodes available, W=3 R=3 works.

</details>

### Challenge 3.2: Conflict-Free Shopping Cart

Design a shopping cart system using CRDTs for e-commerce. The cart must:
- Support concurrent adds/removes from multiple devices
- Never lose items (no LWW data loss)
- Merge automatically across sessions
- Handle 50 million concurrent carts

<details>
<summary>Model Answer</summary>

**Data structure: Observed-Removed Set (OR-Set)**

An OR-Set is a CRDT that supports add and remove operations without conflicts. Each element has a unique tag. Removing an element adds the tag to the tombstone set. Adding an element with the same tag after removal is a no-op.

```
OR-Set shopping cart:

Elements (items currently in cart):
  {"product_A": [tag_1], "product_B": [tag_2]}

Tombstones (removed items):
  {"product_C": [tag_3], "product_D": [tag_4]}

Merge rule: union of elements minus tombstones.
```

**Cart representation:**

```json
{
  "cart_id": "user_42_cart",
  "items": {
    "product_123": {
      "quantity": 2,
      "added": [{"device": "iphone", "tag": "t_1"}, {"device": "web", "tag": "t_2"}]
    },
    "product_456": {
      "quantity": 1,
      "added": [{"device": "iphone", "tag": "t_3"}]
    }
  },
  "removed": {
    "product_789": [{"device": "web", "tag": "t_4"}]
  },
  "version_vector": {"device_iphone": 3, "device_web": 2, "device_api": 1}
}
```

**Operation semantics:**

```python
class ORSetCart:
    def add_item(self, product_id: str, quantity: int, device: str):
        # Add with a unique tag per device
        tag = f"{device}_{self.next_seq(device)}"
        if product_id not in self.items:
            self.items[product_id] = {"quantity": 0, "added": []}
        self.items[product_id]["quantity"] += quantity
        self.items[product_id]["added"].append({"device": device, "tag": tag})
        # If this product was previously removed, the new add wins

    def remove_item(self, product_id: str, device: str):
        if product_id in self.items:
            tag = f"{device}_{self.next_seq(device)}"
            # Move all add-tags to tombstone
            for add_record in self.items[product_id]["added"]:
                self.removed.setdefault(product_id, []).append(add_record)
            del self.items[product_id]

    def merge(self, other_cart):
        # Merge items: union of items
        for pid, item_data in other_cart.items.items():
            if pid in self.removed:
                # Check if all tombstones are superseded by new adds
                new_tags = {(a["device"], a["tag"]) for a in item_data["added"]}
                removed_tags = {(r["device"], r["tag"]) for r in self.removed[pid]}
                if not new_tags - removed_tags:
                    continue  # all removed, skip
            # Normal merge logic
            self.items.setdefault(pid, {"quantity": 0, "added": []})
            self.items[pid]["quantity"] += item_data["quantity"]
            self.items[pid]["added"].extend(item_data["added"])

        # Merge tombstones: union
        for pid, tombstones in other_cart.removed.items():
            self.removed.setdefault(pid, [])
            self.removed[pid].extend(tombstones)

        # Clean up: remove items whose add-tags are all tombstoned
        for pid in list(self.items.keys()):
            if pid in self.removed:
                all_tags = {(a["device"], a["tag"]) for a in self.items[pid]["added"]}
                removed_tags = {(r["device"], r["tag"]) for r in self.removed[pid]}
                if all_tags.issubset(removed_tags):
                    del self.items[pid]
```

**Concurrent add and remove from different devices:**

```
Device A: adds product X with tag t_1 at T=0
Device B: removes product X with tag t_2 at T=0 (concurrent)
         OR-Set merge: product X appears in both state.
         Result: depends on merge order.
         - If merge sees add after remove → product X is added.
         - If merge sees remove after add → product X is removed.

Resolution: Since both operations are concurrent, the OR-Set preserves
both. On merge, the system must pick a consistent state. Rules:
- If no add-tag is tombstoned by a concurrent remove tag: keep item.
- If an add-tag is tombstoned: remove item.
In practice, the last merge to apply determines the result, but
because both sides see the same state after merge, convergence is
guaranteed.
```

**Performance at 50M carts:**

- Each cart is a separate document keyed by user ID.
- Carts are stored in DynamoDB (tunable consistency) or Cassandra (eventual).
- Merge happens on read: the device sends its local copy, the server merges it with the stored copy, and returns the merged result.
- Read repair propagates updates to replicas.

**Complexity:** Each OR-Set operation is O(number of items in the cart). With an average cart size of 5-10 items and a maximum of 100, this is negligible. Storage for tombstones grows with the total number of unique items ever added to a cart, which for most users is under 500 items.

</details>
