## The Five Incidents That Define the Problem

Five real-world scenarios illustrate why consistency dominates system design discussions.

**Scenario 1: The Disappearing Comment**

Alice writes a comment on a popular blog post. She refreshes the page: her comment appears with a count of 47 comments. Bob, viewing the same post from a different region, refreshes 10 seconds later. He sees 43 comments. Alice's comment is missing from his view entirely. Both users contact support — f--?s--, one confused, one frustrated. The system's database replicas have not yet synchronized.

**Scenario 2: The Phantom Bank Balance**

You check your bank account balance on your phone at 9:00 AM. It shows $3,200. You pay $1,200 for rent using your laptop at 9:01 AM. At 9:03 AM, you open the mobile app to confirm. It still shows $3,200. The mobile app reads from a read replica that has not received the latest write.

**Scenario 3: The Oversold Concert**

An e-commerce site sells 500 tickets for a concert. Two web servers each read the current ticket count at the same moment: both see 3 remaining tickets. Server A sells ticket 4 of 3 to Customer X. Server B sells ticket 5 of 3 to Customer Y at nearly the same instant. The site has sold 5 tickets for 3 available slots. The payment systems are upstream of the consistency guarantee — f--?s--, the database accepted both writes because neither server knew about the other's concurrent transaction.

**Scenario 4: The Collaborative Document Ghost Edition**

Two engineers edit the same design document simultaneously. Engineer A deletes an outdated section and adds a new architecture diagram. Engineer B, working from a cached copy that has not received the deletion, cannot see the removed section and adds a paragraph that references the deleted material. Both save. The document now contains a broken cross-reference and a phantom section header pointing to text that no longer exists. The collaboration layer used last-writer-wins on individual paragraphs but did not merge structural changes, producing a corrupted document that neither engineer authored.

**Scenario 5: The Leaderboard That Lied**

A gaming platform displays a live leaderboard during an esports tournament. Player X completes a level with 95,000 points at 14:32:01. Player Y, playing simultaneously, also posts 95,000 points at 14:32:03. The leaderboard should show both at rank 1 with a tie-breaker by completion time. Instead, a read replica that has only received Player X's score shows Player X at rank 1 and Player Y missing entirely. Viewers in different geographic regions see different leaderboards depending on which replica serves their request. The tournament organizers receive complaints from viewers accusing the platform of cheating.

All five incidents share the same root cause: **nodes disagree about the current state of data because replication has not completed**.

## Insight

Consistency means that all nodes in a distributed system see the same data at the same time, or that a read operation returns the most recent write.

Compare these communication patterns in everyday life:

| Pattern | Example | Consistency Model |
|---|---|---|
| Group chat with lag | Alice sends a message; Bob receives it 30 seconds later, Charlie 45 seconds later, and Diana never receives it (phone offline). | Eventual consistency |
| Phone call | Both participants hear each other in real-time. Every word is delivered in the order spoken, and neither hears an incomplete version of the conversation. | Strong (linearizable) consistency |
| Library book checkout | The main branch catalog updates instantly when you check out a book. A neighboring branch still shows the book as available for 30 minutes until the nightly sync runs. | Eventual consistency with bounded convergence |
| Shared whiteboard | Two people draw on different parts of a whiteboard simultaneously. Observers at opposite ends of the room perceive the drawings in different orders because they saw each stroke at slightly different times. | Causal consistency |
| Git commit history | All developers on a team push to the same repository. After a pull, every developer sees the identical commit graph in the same topological order. | Sequential consistency |
| Bank ATM network | You withdraw cash from an ATM in New York. When you check your balance at a teller in Los Angeles 10 minutes later, the withdrawal is reflected. The inter-bank settlement runs on a batch cycle but the customer-facing balance is read-your-writes. | Read-after-write consistency |
| DNS record update | A domain's A record is changed. Users querying different DNS resolvers see either the old IP or the new IP depending on TTL expiry. After 24 to 48 hours, all resolvers converge. | Eventual consistency with TTL bounds |
| Google Docs collaborative editing | Within a single document, edits appear in causal order for each collaborator. If Alice types "A" and Bob types "B" at the same instant, both see both characters, but the exact interleaving may differ on slow networks. | Causal consistency |
| Airline seat inventory | Two travel agents book the same last seat on a flight. One succeeds, one gets an error. The reservation system uses a single master database with a locking mechanism. | Linearizable consistency |

The group chat lets you operate quickly — f--?s--, you send your message and move on. Coworkers see it when they reconnect. The phone call requires both parties to be synchronized, which adds coordination delay but guarantees that neither party sees a contradictory version of the conversation.

Distributed systems map directly to this tradeoff. The cost of coordination (network round trips, leader election overhead, quorum commitments) determines how consistent the system can be while maintaining acceptable latency.

## Core Vocabulary

Master these 22 terms before moving to the deep dive.

**Strong consistency** — f--?s--, Any read returns the most recent write. Equivalent to operating on a single-node system. Requires coordination across replicas before acknowledging a write.

**Eventual consistency** — f--?s--, Given enough time without new writes, all replicas converge to the same value. Reads may return stale data during the convergence window.

**Linearizability** — f--?s--, The strongest form of consistency. Every operation appears to take effect atomically at some point between its invocation and completion. Operations are totally ordered.

**Sequential consistency** — f--?s--, Operations from each process appear in program order. Operations from different processes may interleave arbitrarily, but all processes see the same interleaving.

**Causal consistency** — f--?s--, Operations that are causally related (A happens-before B) are seen in that order by all processes. Concurrent operations may be seen in different orders.

**Read-after-write consistency** — f--?s--, After a write, the writer (and only the writer) always sees the updated value. Also called read-your-writes consistency.

**Monotonic reads** — f--?s--, If a process reads a value for a given key, any subsequent read for that key returns the same or a newer value. The read timestamp never goes backward.

**Monotonic writes** — f--?s--, Writes from a given process are applied in the order they were issued. A write is not accepted until preceding writes for that process have been applied.

**Quorum** — f--?s--, The minimum number of nodes that must agree for a read or write to be considered successful. Typically expressed as R (read quorum) and W (write quorum), with N being total replicas.

**Consensus** — f--?s--, A distributed algorithm that gets multiple nodes to agree on a single value. The foundation of strongly consistent replication.

**Raft** — f--?s--, A consensus algorithm designed for understandability. Uses leader election, log replication, and safety guarantees. The de facto standard for new distributed systems.

**Paxos** — f--?s--, The original consensus algorithm. Proven correct but notoriously difficult to implement correctly. Used internally by Google Chubby and Apache ZooKeeper (Zab is a variant).

**Split-brain** — f--?s--, A failure mode where a network partition divides nodes into two groups, each believing it is the primary. Both groups accept writes, leading to divergence.

**Version vector** — f--?s--, A data structure that tracks the causal history of a replica. A list of (node, operation count) pairs. Used to detect concurrent updates.

**Conflict resolution** — f--?s--, The mechanism for merging divergent values when two or more nodes have independently modified the same data. Strategies include last-writer-wins (LWW), merge functions, CRDT merge rules, or application-level resolution.

**Last-writer-wins (LWW)** — f--?s--, The write with the most recent timestamp is accepted. All other writes are discarded. Simple but loses data.

**CRDT (Conflict-free Replicated Data Type)** — f--?s--, A data structure designed so that concurrent updates from different nodes can be merged without conflicts, producing the same result on all nodes.

**Gossip protocol** — f--?s--, A peer-to-peer communication mechanism where each node periodically exchanges state information with a random subset of other nodes. Information spreads exponentially: after O(log N) rounds, all nodes in an N-node cluster have received the update. Used by Cassandra, Riak, and Redis Cluster for failure detection and metadata propagation.

**Anti-entropy** — f--?s--, A background reconciliation process in which nodes compare their data for a given key range and exchange missing updates. Anti-entropy uses Merkle trees to efficiently detect differences without transferring the full dataset. Unlike read repair (which fires on-demand), anti-entropy runs continuously and catches every divergence.

**Read repair** — f--?s--, An on-demand consistency mechanism triggered during a read request. The coordinator compares version information across replica responses. If any replica returns an older version, the coordinator writes the latest value to that replica before returning the result to the client. Limits the window of inconsistency to the interval between reads.

**Hinted handoff** — f--?s--, A durability mechanism for when a replica is temporarily unreachable. The coordinator stores the write locally with a hint identifying the intended replica. When the replica comes back online, the coordinator forwards the hinted write. Hinted handoff prevents data loss during brief outages but does not guarantee permanent replication — f--?s--, hints can be lost if the coordinator itself fails before forwarding.

**Slack quorum** — f--?s--, A relaxed quorum variant that accepts writes when fewer replicas respond than the strict quorum requires, as long as some configurable slack threshold is met. Used by Cassandra's ANY write consistency level, where a write succeeds if at least one node (including hinted handoff coordinators) accepts it. Increases write availability during failures at the cost of consistency guarantees.

**Probabilistic bounded staleness (PBS)** — f--?s--, A formal model from Amazon DynamoDB that quantifies the probability of stale reads given R, W, N, and the distribution of replica lag. Instead of guaranteeing "no stale reads," PBS answers the question "what percentage of reads will see a write within K milliseconds?" DynamoDB's design uses PBS to offer SLAs on staleness — f--?s--, for example, 99.9% of reads see writes within 100 ms when W=2, R=2, N=3.

**Hybrid logical clocks (HLC)** — f--?s--, A clock mechanism that combines physical wall-clock time with a logical counter to track causality across distributed nodes. HLCs stay close to physical time (useful for human-readable timestamps in logs and LWW decisions) while preserving the happens-before properties of vector clocks. Used by CockroachDB and Spanner to assign commit timestamps without centralized coordination.

The diagram illustrates three key points:

- Write to leader: The client sends a write to Node A (the leader), which updates its local value to v=2.
- Replication in flight: Node A sends the update to Nodes B and C, but this takes measurable time (typically milliseconds to seconds).
- Stale read: A client reading from Node B before replication completes sees v=1 — f--?s--, the stale value. The window between write and full replication is where consistency violations occur.

### Step 2: Coordinator determines the replication set

The coordinator node knows the topology: which nodes hold replicas of this key. In a Dynamo-style ring, it computes the preference list. In a Raft cluster, it forwards to the leader.

### Step 3: Coordinator sends write to all replicas

Node A fans out the write to Nodes B and C. The behavior differs by consistency level:

- Strong consistency: A waits for acknowledgment from a majority quorum before responding to the client.
- Eventual consistency: A acknowledges immediately and replicates in the background.

### Step 4: Client waits for quorum acknowledgment

If W=2 and N=3, the coordinator can respond to the client after 2 nodes acknowledge. The third node may still be processing, but the system considers the write durable.

### Step 5: Client issues a read

The client now sends a GET request for the same key.

### Step 6: Coordinator chooses read sources

- For strong consistency: the coordinator reads from enough nodes (quorum) and returns the latest version.
- For eventual consistency: the coordinator reads from the nearest replica and returns whatever value that node holds.

### Step 8: Write conflict scenario

Two clients write concurrently. Without coordination, the last write accepted by each replica may differ.

### Step 9: Resolution strategies

Three main approaches to resolving conflicts:

- **Read-repair**: On read, the coordinator detects version disagreement and writes the latest version to stale nodes.
- **Last-writer-wins**: Timestamp comparison picks the winner. All other writes are lost.
- **CRDT merge**: Data structures designed so concurrent updates merge deterministically without data loss.

### Step 10: Convergence

After sufficient time with no new writes, all replicas converge to the same value. This is eventual consistency in action. The convergence time depends on network latency, replication strategy, and conflict resolution.

### Step 11: Hinted handoff when a replica is down

When one or more replicas are unreachable during the write fan-out, the coordinator cannot send the write to all N nodes. Instead of dropping the write, the coordinator stores it locally in a hints file — f--?s--, a holding area that records the intended replica and the data. Once the replica recovers and reconnects, the coordinator forwards the pending hints.

Dynamo-style systems (Cassandra, Riak) use hinted handoff with configurable timeout — f--?s--, typically a few hours before hints are discarded.

### Step 12: Read repair propagation

Read repair closes the consistency gap on every read path. When the coordinator receives responses from replicas, it compares the version vectors or timestamps. Any replica returning an older version than the latest receives an asynchronous write to bring it up to date.

The read repair flow works in three phases:

- **Phase 1 — f--?s--, Collect**: The coordinator sends a digest request (metadata only, not full data) to all replicas in the read set. Each replica returns a version summary.
- **Phase 2 — f--?s--, Compare**: The coordinator identifies which replicas hold stale versions by sorting the responses by version timestamp or vector clock.
- **Phase 3 — f--?s--, Repair**: The coordinator sends the latest full value to all stale replicas. The client receives the response immediately — f--?s--, the repair runs in the background.

Read repair does not require extra consistency nodes at write time. Its effectiveness depends on the application's read-to-write ratio: systems with frequent reads converge faster than systems with infrequent reads. In high-read workloads, read repair keeps the system clean without background anti-entropy.

### Step 13: Anti-entropy and gossip-based sync

While read repair handles convergence opportunistically, anti-entropy runs continuously in the background regardless of client traffic. Each node periodically picks a random partner node and compares the data they hold for a given key range.

A naive comparison — f--?s--, send all keys and values to the partner — f--?s--, is bandwidth-prohibitive for datasets with millions of keys. Anti-entropy solves this with Merkle trees:

- Each node builds a Merkle tree over its data, where each leaf is a hash of a key's value and each internal node is a hash of its children.
- Nodes exchange root hashes. If the roots match, the entire subtree is identical — f--?s--, no further work needed.
- If roots differ, nodes recursively descend the tree to find the exact leaf where values diverge.
- Only the divergent key-value pairs are transferred.

Merkle tree comparison requires O(log N) rounds of messages for N keys, compared to O(N) for a full dataset transfer. Cassandra runs anti-entropy every 5 minutes by default, configurable per keyspace.

### Step 14: Conflict resolution — f--?s--, LWW versus CRDT

When concurrent writes produce divergent values on different replicas, the system must resolve the conflict deterministically — f--?s--, every replica must reach the same final value independently. Two approaches dominate.

**Last-writer-wins (LWW)** uses a per-key timestamp assigned by the coordinator or the client. The write with the highest timestamp wins. All other writes are silently discarded.

```
# LWW resolution pseudocode
def resolve(values):
  # Sort by timestamp descending, return highest
  return max(values, key=lambda v: v.timestamp)

# Given concurrent writes:
#  Node A: v=42, ts=1000
#  Node B: v=99, ts=1002
# Result: v=99 wins because ts=1002 &gt; ts=1000
# v=42 is lost permanently
```

LWW requires synchronized clocks (NTP) and tolerates timestamp ties only up to the clock precision. Apache Cassandra uses LWW for all mutation operations by default when no other resolution is configured.

**CRDTs (Conflict-free Replicated Data Types)** are data structures whose merge operation is associative, commutative, and idempotent. Every node that receives concurrent writes applies the same merge rules and arrives at the same result.

Common CRDTs include:

- **G-Counter (grow-only counter)**: Each node maintains its own increment in a vector. The total is the sum of all vector entries. Merging takes the element-wise maximum.
- **PN-Counter (positive-negative counter)**: Two G-Counters — f--?s--, one for increments, one for decrements. The value is the difference.
- **G-Set (grow-only set)**: Elements added to the set are never removed. Merging is set union.
- **OR-Set (observed-remove set)**: Supports both add and remove. Each element carries a unique tag. Adds insert the tag; removes tombstone the tag. An element is present if its tag exists in the add-set but not in the remove-set.
- **LWW-Register**: Each update carries a wall-clock timestamp. The value with the latest timestamp wins — f--?s--, a CRDT version of LWW with deterministic merge semantics across all replicas.

Choose LWW when data loss is acceptable and throughput is the priority. Choose CRDTs when every write matters and you need deterministic convergence — f--?s--, for example, collaborative editing applications where no keystroke should vanish.

### Step 15: Convergence detection

The final step in the consistency lifecycle is determining when all replicas agree. Convergence detection answers the question: has the system reached a state where every node holds the same value for every key - Three detection mechanisms are used in practice:

- **Gossip state exchange**: Each node broadcasts its Merkle tree root hash to a random subset of peers. When all peer responses match the local root hash, the node considers itself converged for that key range. The system is fully converged when every node's root hashes match across all peer pairs.
- **Vector clock comparison**: The coordinator tracks the maximum version seen across all replicas during a read. If every replica returns the same version vector, convergence is verified for that key.
- **Quorum verification**: A dedicated audit process reads from all N replicas with R=N and compares the returned values. This is expensive but provides a definitive answer. AWS DynamoDB offers a `ConsistentRead` parameter that effectively does this for single-key accesses.

Convergence detection is not typically exposed to application code. Instead, the system provides visibility through operational metrics: Cassandra exposes `RepairedDataTracking`, Riak has `riak-admin cluster partition-status`, and DynamoDB surfaces replication lag in CloudWatch metrics.

Seven major consistency models, ordered from strongest to weakest.

Every operation appears to take effect atomically at a single point in time. All nodes agree on the order of all operations.

### 2. Sequential Consistency

Operations from each client appear in program order. Operations from different clients can be interleaved, but all clients see the same interleaving.

**Use when:** You need ordering guarantees within a session but can tolerate reordering across clients.

| Application | Why Sequential Consistency Matters |
|---|---|
| Log aggregation system | Each producer's log entries are read in the order they were written; entries from different producers may interleave |
| Multiplayer game state sync | Each player's actions are broadcast in the order they occurred; the server assigns a global sequence for reconciliation |

### 4. Read-after-write (Read-your-writes) Consistency

After a process writes a value, that process always sees the updated value. Other processes may still see stale data.

**Use when:** Session-based applications: user updates a profile picture, then refreshes the page. The user expects their own change to appear.

**Avoid when:** Multi-user publishing where readers must see the latest state.

| Application | Why Read-after-write Matters |
|---|---|
| User profile settings | You change your email address and refresh — f--?s--, the page must show the new address immediately |
| Shopping cart | You add an item and navigate to the cart page — f--?s--, the item must appear |
| Content management system | An editor publishes a draft and previews the live page — f--?s--, the published version must render |

### 6. Eventual Consistency

Given enough time with no writes, all replicas converge to the same value. No bound on convergence time.

**Use when:** DNS, content delivery networks, social media feeds, analytics.

**Avoid when:** Banking, inventory, distributed coordination.

| Application | Why Eventual Consistency Is Acceptable |
|---|---|
| DNS record propagation | TTL defines the convergence bound. A few minutes of stale IP resolution is tolerable for most services |
| CDN cache invalidation | Stale assets are served for seconds to minutes. Fresh assets reach all edge nodes within the TTL window |
| Social media like counts | A variance of a few likes between user views does not affect user experience. The count converges in seconds |
| Analytics dashboards | Hourly report data does not need millisecond precision. A 5-minute staleness window is standard |

### 7. Probabilistic Bounded Staleness (PBS)

Probabilistic bounded staleness is not a separate consistency model but a formal framework for quantifying how stale a read can be under a given configuration. Introduced by Amazon in the DynamoDB paper, PBS answers the question: given R, W, and N, what is the probability that a read returns a value older than K milliseconds - PBS works by modeling the replica lag distribution. Each replica has a measurable delay between receiving a write and making it visible to reads. The lag distribution is typically right-skewed: most replicas converge in under 10 ms, but tail latencies can extend to hundreds of milliseconds.

The PBS formula estimates the stale read probability:

```
P(stale) — f--,°<? (number of non-overlapping replica sets) / (total possible replica sets)

For N=3, W=2, R=2:
- W=2 means at least 2 replicas hold the latest write
- R=2 means the coordinator reads from 2 replicas
- The probability that neither of the 2 read replicas overlaps with the 2 write replicas:

  P(stale) = C(N - W, R) / C(N, R) = C(1, 2) / C(3, 2) = 0 / 3 = 0

For N=3, W=1, R=1:
  P(stale) = C(2, 1) / C(3, 1) = 2/3 = 66.7%
```

In production, DynamoDB uses this model to offer SLAs such as: 99.9% of reads return data within 100 ms of the most recent write when W=2, R=2, N=3.

**Use when:** You need a quantifiable staleness SLA rather than a qualitative "eventual." E-commerce product catalogs, session management, user preference storage.

**Avoid when:** You need absolute guarantees. PBS deals in probabilities — f--?s--, it cannot prevent the 0.1% tail event of a stale read.

| Configuration | Stale Read Probability | Typical p99 Latency |
|---|---|---|
| W=3, R=3, N=3 | 0% (strong consistency) | +2-5 ms per write |
| W=2, R=2, N=3 | 0% (quorum overlap guaranteed) | +1-3 ms per write |
| W=1, R=3, N=3 | 0% (read quorum covers all) | +1 ms per write (fast write) |
| W=2, R=1, N=3 | 33% (one of three reads could miss) | Fast reads |
| W=1, R=1, N=3 | 66% (likely stale) | Fastest path |

## Mental Model

When reasoning about consistency in a system design interview, ask yourself these eight questions in order:

**Question 1: Does my client care if it reads stale data?**

If staleness causes incorrect behavior or a poor user experience, you need a stronger consistency model. If not, eventual consistency saves cost and latency.

**Question 2: Can I tolerate a write being lost?**

Last-writer-wins loses concurrent writes. If data cannot be lost, use CRDTs or a consensus-based approach.

**Question 3: Does the same client read and write within a single session?**

If yes, read-after-write consistency is the minimum requirement. Users expect their own changes to be visible immediately.

**Question 4: Do I need total ordering of all operations?**

Financial ledgers, distributed locks, and coordination services need total ordering. Most applications do not.

**Question 5: What are my latency and availability targets?**

Strong consistency adds 2-3 round trips for quorum coordination. If p99 latency must stay under 10 ms, eventual consistency may be the only option for multi-region deployments.

**Question 6: Can concurrent writes be resolved deterministically?**

If your system allows two clients to write the same key at the same time from different nodes, you must decide how those conflicting writes are resolved. Three options exist:

- **Last-writer-wins**: Pick the write with the highest timestamp. Simple, fast, but the losing write vanishes permanently. Acceptable when data loss is tolerable and throughput is the priority.
- **CRDT merge**: Use data structures whose merge function is associative, commutative, and idempotent. All concurrent writes survive. The merge result is identical on every replica. Used in collaborative editing (Automerge, Yjs), multi-master replication (Redis CRDTs), and Riak.
- **Application-level resolution**: Return all conflicting versions to the client and let application logic decide. Used in Amazon DynamoDB's conditional writes and versioned reads. Gives the application full control but pushes complexity to the caller.

Choose LWW when throughput is the priority and occasional data loss is acceptable. Choose CRDTs when no write can be discarded. Choose application-level resolution when the business logic for merging is too complex to encode in infrastructure.

**Question 7: Do you need to detect conflicts or prevent them?**

Two fundamentally different strategies exist for handling concurrent writes, and the choice determines your entire system architecture.

- **Detection**: Allow concurrent writes, then find and resolve divergences after the fact. Detection mechanisms include version vectors (compare replica histories), Merkle tree comparison (anti-entropy), read repair (detect staleness during reads), and gossip state exchange (propagate version metadata). Detection scales better because writes proceed without coordination, but recovery from detected conflicts is complex and may lose data.

- **Prevention**: Ensure that only one node can accept writes for a given key at any time. Prevention mechanisms include single leader replication (Raft), distributed locking (ZooKeeper), lease-based authority (Spanner), and Paxos-based consensus (each write goes through a full consensus round). Prevention adds coordination latency on every write but eliminates the recovery path entirely.

Choose detection when write throughput is your top priority and eventual consistency is acceptable. Choose prevention when correctness is non-negotiable and you can afford the latency cost of coordination.

A concrete comparison: Amazon DynamoDB uses detection (vector clocks, read repair) and handles millions of writes per second across regions. Google Spanner uses prevention (TrueTime, Paxos) and guarantees external consistency at the cost of 10-50 ms write latency across regions. Neither approach is universally correct — f--?s--, the right choice depends on your workload's tolerance for stale reads versus write latency.

**Question 8: How much latency can the write path tolerate?**

The latency budget of your write path directly determines which consistency mechanisms are feasible:

| Write Latency Budget | Available Mechanisms | Example Systems |
|---|---|---|
| Below 1 ms | Local writes only, no replication | Embedded databases (SQLite), single-node Redis |
| 1-5 ms | Async replication, hinted handoff, eventual consistency | Cassandra with W=1, Riak, DynamoDB with eventual consistency |
| 5-15 ms | Quorum writes (W=N/2+1), Raft commit, leader replication | Raft clusters, Kafka with acks=all, etcd |
| 15-50 ms | Multi-region quorum, synchronous cross-DC replication | Spanner, CockroachDB, multi-region Kafka |
| Above 50 ms | Batch replication, CDC pipelines, offline sync | Data warehouse replication, mobile offline-first apps |

If the system requires single-digit millisecond writes, you cannot run a full Raft consensus round trip across three data centers — f--?s--, the speed of light limits cross-continent round trips to roughly 30 ms minimum. In this case, you must use eventual consistency with hinted handoff and read repair, or accept a multi-region compromise where writes are acknowledged locally and replicated asynchronously.

A concrete example: Cassandra with CL=ONE (write to one node, no quorum) achieves p99 write latencies of 1-3 ms on a typical three-node cluster in the same region. Changing to CL=QUORUM (W=2 of 3) increases p99 to 5-10 ms as the coordinator waits for the second acknowledgment. Changing to CL=ALL (W=3 of 3) pushes p99 to 8-15 ms because the slowest replica sets the pace. Every millisecond of write latency imposes a real cost on user-facing workloads — f--?s--, API gateway timeouts, connection pool exhaustion, and degraded UX.

## Key Takeaways

- Consistency governs what a distributed system allows you to see when reading after a write.
- Stronger consistency means more coordination, higher latency, and lower availability during partitions.
- Eventual consistency enables low latency and high availability but creates a window of stale reads.
- Seven major consistency models form a spectrum: linearizable, sequential, causal, read-after-write, monotonic reads, eventual, and probabilistic bounded staleness.
- Choose the weakest consistency model that your application can tolerate — f--?s--, not the strongest one you can implement.
- Quorum sizes (R, W, N) directly tune the consistency level of a Dynamo-style system.
- Stale reads, write conflicts, and convergence delays are the three failure modes of weak consistency.
- Consistency is a tradeoff axis, not a binary property — f--?s--, most production systems use tunable consistency.
- Hinted handoff, read repair, and anti-entropy form a three-layer defense against divergence in eventually consistent systems.
- Probabilistic bounded staleness gives you a quantified SLA on staleness instead of a vague "eventually" promise.
- The latency budget of your write path dictates which consistency mechanisms are physically possible for your deployment topology.

This foundation prepares you for the deep dive into how consensus protocols, quorum mechanisms, and conflict resolution strategies actually work under the hood.
