## How This Topic Appears

TCP vs UDP questions appear in three contexts: fundamentals, system design, and debugging/troubleshooting.

### Categorization

**Foundation Questions (Easy — Phone Screen / L3)**

1. "Explain the difference between TCP and UDP. When would you use each?"
2. "Walk me through what happens when you type a URL into a browser and press Enter."
3. "Describe the TCP three-way handshake. What happens if a SYN is lost?"
4. "How does TCP ensure reliable delivery?"
5. "What is a port number? What are well-known ports?"

**System Design Questions (Medium-Hard — L4/L5 Onsite)**

6. "Design a real-time multiplayer game. Would you use TCP or UDP for player position updates? Why?"
7. "Design a video streaming service. How does the transport protocol choice affect buffering and latency?"
8. "Design a monitoring system that collects metrics from 100,000 servers every 10 seconds. Should the collectors use TCP or UDP?"
9. "A client reports that file uploads to your service stall after 4 KB. Diagnose the problem."

**Advanced / Expert Questions (Hard — L6+)**

10. "Design a transport protocol for a Mars rover with a 40-minute round trip. TCP is unusable. What do you do?"
11. "Your team's latency-critical service uses UDP. A network engineer says 'UDP is unreliable, switch to TCP.' How do you respond?"
12. "Compare TCP Reno, CUBIC, and BBR. Under what conditions would each outperform the others?"

## What the Interviewer Is Actually Testing

| Question Type | What They Test | Red Flag | Green Flag |
|--------------|---------------|----------|------------|
| "Explain the difference" | Do you actually understand, or did you memorize a list? | Lists 3 differences without explaining why they exist | Explains the core trade-off (reliability vs. latency) and traces each protocol feature back to that choice |
| "Walk me through a URL" | Do you have a mental model of the full stack? | Jumps from DNS to HTTP without mentioning transport | Names TCP as the transport layer, explains SYN/SYN-ACK/ACK, mentions port 80 |
| "Design a game transport" | Can you apply transport knowledge to a concrete system? | Says "UDP" without explaining what happens to lost packets | Defines loss budget per message type, describes UDP for state sync and TCP for critical events, mentions interpolation |
| "Diagnose 4 KB stall" | Do you know the Nagle algorithm? | Says "network issue" and stops | Immediately identifies Nagle's algorithm, explains TCP_NODELAY, asks about delayed ACK |
| "Design a protocol for Mars" | Can you innovate beyond textbook answers? | Says "use UDP with retransmission" (does not compute — 40 min RTT) | Describes forward error correction, file carousel, packet erasure encoding, or rate-controlled data streams |

## Level-by-Level Expectations

### L3 / L4 (Entry Level, 0-2 years experience)

**Expected knowledge**: Basic understanding of TCP vs UDP. Can explain the three-way handshake. Knows that TCP is reliable, UDP is not. Can give examples (HTTP uses TCP, DNS uses UDP).

**Sample answer to "Explain the difference"**:
"TCP is a connection-oriented protocol that guarantees data arrives in order without loss. It establishes a connection with a three-way handshake (SYN, SYN-ACK, ACK), numbers every byte, and retransmits lost packets. UDP is a connectionless protocol that sends datagrams without guarantees. TCP is used when reliability matters — HTTP, file transfers, email. UDP is used when speed matters — DNS, video streaming, gaming. The trade-off is latency: TCP adds one round trip for setup and retransmission delay on loss."

**What they need to improve**: Understanding of congestion control, flow control, and real-world failure modes. Connecting protocol behavior to application-level impact.

### L5 (Senior, 3-6 years)

**Expected knowledge**: Deep understanding of congestion control algorithms, buffer sizing, and TCP tuning. Can design a system using both protocols together. Understands the kernel architecture implications.

**Sample answer to "Design a real-time multiplayer game"**:
"I would use UDP for player position updates and TCP for game-critical events like weapon pickups and scoring. Player positions update at 20 Hz; losing one update is invisible because the next one arrives in 50ms with a newer position. Packet 10 supersedes packet 9. But a 'player picked up the flag' event must arrive — I would send that over a reliable UDP sub-channel with application-level ACKs and a 200ms retry timeout. I would also implement dead reckoning: the client predicts player positions between updates using velocity vectors, and corrects when a position update arrives."

**What they need to improve**: Deep kernel-level understanding, multi-region scaling considerations, and advanced failure mode analysis.

### L6 (Staff, 7-10 years)

**Expected knowledge**: Can design a TCP-like protocol from scratch. Knows the Linux kernel TCP implementation details. Has opinions on CUBIC vs BBR. Can reason about transport decisions at network scale (100K+ connections, multi-region).

**Sample answer to "Design a transport for Mars"**:
"With a 40-minute round trip, TCP is fundamentally broken. The RTO would be over 40 minutes, and TCP's slow start would take hours to reach full bandwidth. I would design a UDP-based file carousel with erasure coding. Split the data into N blocks, add K redundant blocks using Reed-Solomon encoding, and continuously broadcast all N+K blocks as a carousel. The receiver collects enough blocks (any N of N+K) to reconstruct the data. No ACKs needed. The carousel rate equals the link bandwidth divided by block size. This trades guaranteed delivery for constant-rate streaming — perfect for deep-space communication where retransmission time exceeds the mission timeline."

**What they need to improve**: Usually nothing technical — the expectation is communication, mentorship, and trade-off articulation.

### L7 / L8 (Principal / Distinguished, 10+ years)

**Expected knowledge**: Can influence the organization's networking strategy. Understands the business impact of transport choices. Has implemented or overseen transport layer changes at scale.

**Sample answer to "Respond to the network engineer"**:
"Your concern about UPD's lack of reliability is valid. Let me walk through why we chose it and the mitigations we've built. Our real-time voice service needs under 200ms end-to-end latency. At 5% packet loss on our network path, TCP would retransmit old packets that arive too late for playback. We have built three mitigations: Opus codec with packet loss concealment handles 20ms gaps, forward error correction duplicates every other packet, and the jitter buffer absorbs 80ms of jitter. Our measured subjective call quality at 5% loss is 4.2 out of 5. The business impact: our engineering months building these mitigations versus the lost revenue from 200ms-latency calls that users would abandon."

## The FAANG Framework for Answering

Use this 6-step framework for any system design question involving transport protocol choice.

### Step 1: State the Constraint

Begin with the fundamental constraint of the system being designed.

*"The primary constraint is ___ . Everything else follows from this."*

Examples:
- Real-time voice: "The primary constraint is end-to-end latency under 200ms."
- Database writes: "The primary constraint is exactly-once delivery of every write."
- Video streaming: "The primary constraint is zero rebuffering events during playback."

### Step 2: Define Loss Budget

Define how much packet loss the application can tolerate at the transport layer before the user notices.

*"The acceptable loss budget is ___ %. At this rate, the user experiences ___ ."*

Examples:
- Game position updates: "1-10% loss. Each lost update is replaced by the next one 50ms later."
- Video call: "0.5-2% loss. Above 2%, the video codec cannot conceal the artifacts."
- Payment transaction: "0% loss. Losing one transaction creates a financial discrepancy."

### Step 3: Choose Transport

Based on the constraint and loss budget, choose TCP, UDP, or a mixed approach.

*"For ___ traffic, I will use ___ because ___ . For ___ traffic, I will use ___ because ___ ."*

### Step 4: Mitigate Weaknesses

For your chosen transport, describe how you address its downsides.

*"To mitigate TCP's head-of-line blocking, we will ___."*
*"To mitigate UDP's lack of reliability, we will ___."*

### Step 5: Describe the Additive Layer

Describe any application-level protocol features you add on top of the transport.

*"On top of UDP, we will add ___ : sequence numbers for gap detection, a retransmission timer for critical messages with a 200ms timeout, and a heartbeart every 5 seconds."*

### Step 6: Quantify the Result

Estimate the expected latency, throughput, and reliability metrics.

*"With this design, we expect ___ microseconds p50 latency, ___ throughput at peak, and ___ % reliability under normal conditions."*

## Phrase Bank

The following phrases are ready-to-use for interviews. Each includes the context where it is most effective.

| Phrase | When To Use | Why It Works |
|--------|-------------|--------------|
| "The fundamental trade-off in transport protocols is between reliability and latency. Every protocol feature traces back to this trade-off." | Opening statement on TCP vs UDP questions | Sets up the framework before listing differences |
| "TCP guarantees in-order delivery, which creates head-of-line blocking: if packet 3 is lost, packets 4-10 must wait in the receive buffer even though they arrived." | Explaining TCP's weakness in streaming contexts | Shows you understand the mechanism, not just the symptom |
| "The three-way handshake adds one round trip before any data flows. On a 200ms transatlantic link, that is 200ms of dead time before your HTTP request leaves the client." | Discussing real-world latency impact | Quantifies the abstract concept |
| "UDP's checksum is optional in IPv4. If you have hardware checksum offload on the NIC, corrupted packets can slip through. Application-level CRC-32 is cheap insurance." | Advanced UDP reliability discussion | Shows awareness of kernel-to-NIC offload behavior |
| "Selective ACKs (SACK) let TCP recover multiple lost packets in one round trip instead of retransmitting everything after the first loss. This is essential on links with burst loss." | Advanced TCP tuning | Demonstrates protocol-level optimization knowledge |
| "BBR paces at the measured bandwidth, not the loss-detected bandwidth. It does not fill the buffer until it drops — it fills the buffer to the actual bottleneck rate. This means zero bufferbloat." | Explaining BBR vs CUBIC choice | Shows you understand modern congestion control |
| "For real-time systems, I use a jitter buffer at the receiver — typically 40-80ms depth — to convert the transport's variable latency into fixed latency for the audio/video decoder." | Real-time system design | Shows you know the receiver-side architecture |
| "TCP's TIME_WAIT state holds the connection for 60-120 seconds after close to prevent delayed duplicate segments from corrupting a new connection. At high connection rates, this exhausts ephemeral ports." | High-connection-rate system design | Connects protocol behavior to operational reality |

| "UDP with application-level reliability is more complex but gives you partial reliability exactly where you need it. The right choice is not the same protocol for all messages within a system." | Mixed-protocol architecture | Demonstrates nuanced thinking |
| "A SYN flood works because TCP allocates memory before the handshake completes. | "QUIC eliminates head-of-line blocking at the transport level by providing per-stream ordering rather than per-connection ordering. One lost packet in stream A does not block streams B through Z." | Modern protocol evolution | Shows you know the current state of the art |

## Red Flags

### Red Flag 1: "UDP is unreliable, so I would never use it."

**Why it is a red flag**: Shows you do not understand the application-level trade-off. Netflix uses UDP for streaming. Discord uses UDP for voice. DNS uses UDP by default. The question is not whether UDP is reliable — it is whether your application can afford the overhead of reliability.

**The right answer**: State the constraints, define the loss budget, and choose the transport that meets the requirements with the least overhead.

### Red Flag 2: "I would use TCP for everything. It is more reliable."

**Why it is a red flag**: Shows you have never shipped a latency-sensitive system. You are choosing the safe answer instead of analyzing the problem.

**The right answer**: Analyze the system's latency budget, throughput requirements, and loss tolerance before choosing.

### Red Flag 3: "TCP has no downsides at scale."

**Why it is a red flag**: Shows inexperience with production systems at scale. TCP's memory overhead per connection (128 KB+), head-of-line blocking, TIME_WAIT accumulation, and retransmission storms are well-documented pain points.

**The right answer**: Describe the downsides of TCP at scale and how you would mitigate each one.

### Red Flag 4: "UDP is faster because it has smaller headers."

**Why it is a red flag**: The 8- vs 20-byte header difference matters only for ultra-small payloads (DNS, NTP). For typical payloads (1 KB+), the 1.2% header savings is negligible. The real speed difference comes from no handshake, no congestion control, and no ACK processing.

**The right answer**: Explain the actual mechanisms that make UDP faster — connectionless design, zero ACK overhead, no flow control, no congestion window.

### Red Flag 5: Ignoring application-level reliability when choosing UDP.

**Why it is a red flag**: Saying "I choose UDP" and not describing how lost packets are handled shows you have not thought through the consequences. In production, UDP without application-level reliability mechanisms is unusable for most workloads.

**The right answer**: Always describe the reliability layer you would add on top of UDP: sequence numbers, heartbeats, retransmission for critical messages, forward error correction, or jitter buffering.

### Red Flag 6: "DNS uses UDP because it is faster" — no nuance.

**Why it is a red flag**: DNS uses UDP because DNS responses under 512 bytes fit in one packet, and a lost DNS query can be retried by the resolver with a different server. DNS also uses TCP for zone transfers and responses over 512 bytes. The choice is nuanced.

**The right answer**: DNS uses UDP for standard queries because TCP's setup overhead would triple the latency for a 40-byte query. DNS falls back to TCP for large responses and uses TCP exclusively for zone transfers where data integrity is critical.

### Red Flag 7: Not knowing the difference between L4 and L7 load balancers.

**Why it is a red flag**: A senior engineer working on distributed systems must know that L4 (NLB) supports UDP and TCP, while L7 (ALB) supports only HTTP/2 over TCP. Choosing the wrong load balancer type breaks the architecture.

**The right answer**: Specify whether you are using L4 or L7 load balancing based on whether the application needs transport-layer (UDP, raw TCP) or application-layer (HTTP, cookies) routing.

### Red Flag 8: "I would not use TCP because of the overhead."

**Why it is a red flag**: "Overhead" is vague. You must quantify the overhead in terms of latency (handshake RTT), throughput (congestion window), and resource usage (per-connection memory). Vague answers signal shallow understanding.

**The right answer**: Quantify: "TCP adds 1 RTT before data flows, 128 KB per connection for kernel buffers, and approximately 3% CPU overhead for ACK processing compared to UDP."

## Practice Drills

### Drill 1 (2 minutes): Explain the three-way handshake

Timer starts now. Explain the TCP three-way handshake with sequence numbers, what each state transition means, and what happens if a SYN is lost. Define initial sequence number randomization and why it matters for security.

**After 2 minutes**: Check if you covered: SYN (client sends ISN), SYN-ACK (server sends ISN + ack of client ISN+1), ACK (client ack of server ISN+1), state transitions (CLOSED -> SYN_SENT -> ESTABLISHED), ISN randomization to prevent TCP hijacking.

### Drill 2 (5 minutes): Design a transport for a real-time stock price feed

A financial data provider pushes 500,000 price updates per second. Each update is 100 bytes. The system serves 10,000 subscribers. Updates are sent over a multicast-like fan-out. You must choose the transport protocol. Defend your choice.

**Model answer**: UDP per subscriber socket. At 500,000 updates/sec x 100 bytes = 50 MB/s, TCP's ACK overhead (500,000 ACKs/sec per subscriber, times 10,000 subscribers) creates an impossible 5 billion ACKs per second. UDP eliminates ACKs entirely. Loss mitigation: each update carries a monotonic sequence number. Subscribers detect gaps and request a retransmit of missing batches via a separate TCP channel. 0.01% loss means 50 missed updates per second — acceptable for most trading strategies that use the feed (they interpolate between received values).

### Drill 3 (5 minutes): Diagnose a timeout problem

A mobile app uploads images to a server. Uploads under 4 KB complete in 50ms. Uploads over 4 KB take 200ms and regularly time out at 30 seconds. What is the most likely cause?

**Model answer**: Nagle's algorithm combined with delayed ACK. The client sends the first 1460 bytes (one TCP segment). The server delays its ACK for the standard 200ms (delayed ACK timer). Meanwhile, Nagle holds the remaining bytes because it has not received an ACK for the first segment. After 200ms, the server sends the ACK, the client flushes the buffer, and the remaining data transfers normally. For uploads under 4 KB (approximately 3 segments), the 200ms delay is noticeable but not fatal. For larger uploads, multiple Nagle-ACK cycles compound, and if the mobile device switches networks mid-upload, the connection breaks. Fix: enable `TCP_NODELAY` on the client socket and disable delayed ACK on the server (`TCP_QUICKACK`).

### Drill 4 (10 minutes): Design a reliable broadcast system for 1000 servers

You must deliver configuration updates (2 KB) to 1000 servers. Reliability is critical — a missed update means the server runs stale configuration. Design the transport mechanism. Compare TCP unicast, UDP multicast, and UDP with gossip.

**Model answer**: Use a combination of UDP gossip for dissemination and TCP for confirmation. The config publisher sends the update to 10 seed servers via TCP (each seed confirms receipt). Seed servers broadcast the update to the remaining 990 servers via UDP multicast or UDP gossip. Servers that receive the UDP broadcast send a confirmation back via TCP. After 30 seconds, the publisher lists servers that have not confirmed and sends a direct TCP push to each. This gives the speed of UDP broadcast (all 1000 servers notified in under 200ms) with the reliability of TCP confirmation (every server either ACKs or gets a point-to-point retry).

### Drill 5 (10 minutes): Design WebRTC's transport

Explain why WebRTC uses DTLS over UDP rather than raw UDP or TCP. How does ICE (Interactive Connectivity Establishment) work with UDP through NATs? What happens when UDP is blocked by a firewall?

**Model answer**: WebRTC uses DTLS over UDP because: (1) UDP provides sub-200ms latency needed for real-time voice/video, (2) DTLS adds encryption equivalent to TLS without the multi-round-trip setup, (3) DTLS runs over UDP (a datagram protocol) and handles packet loss gracefully unlike TLS over TCP which would block on retransmission. ICE uses STUN (UDP-based) to discover the server reflexive address (public IP:port behind NAT). If the NAT is symmetric, ICE falls back to TURN (TCP relay through a cloud server). When a firewall blocks UDP, WebRTC falls back to TURN over TCP, which increases latency by 30-50ms but maintains connectivity. This is why WebRTC applications recommend opening UDP port 3478 on firewalls.
