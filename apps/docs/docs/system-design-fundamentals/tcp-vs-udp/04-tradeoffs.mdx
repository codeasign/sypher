## The Core Tension

**Every network protocol trades reliability for latency, and the transport layer is where that trade-off is made permanent.**

TCP guarantees delivery, ordering, and integrity. UDP guarantees none of those things. The tension is inescapable: you cannot add reliability without adding delay, and you cannot remove it without accepting data loss. The question is never "which is better?" — it is "what is the cost of a lost packet in your system?"

A lost stock trade costs millions. A lost video frame is invisible. Your transport choice must match the cost of loss in your domain.

## Decision Matrix

| Decision | Option A (TCP) | Option B (UDP) | Choose A When | Choose B When |
|----------|---------------|----------------|---------------|---------------|
| Transport for database writes | Guaranteed ordered delivery, ACK per batch, 1-5ms latency | Custom reliability on UDP, 0.2-1ms latency | Regulatory compliance requires write audit trail | Internal metrics DB where 0.001% loss is acceptable |
| Real-time video streaming | Buffers 2-5 seconds, smooth playback, no frame loss | 100-500ms latency, possible frame drops, adaptive bitrate | On-demand streaming (Netflix, YouTube — buffering is free) | Live streaming, video calls, gaming (latency budget under 200ms) |
| DNS resolution | Reliable, zone transfers use TCP, queries under 512 bytes use TCP fallback | Fast queries (single packet), no handshake overhead, less than 1ms resolution | Zone transfers where data integrity beats speed | Regular DNS queries where speed and server capacity matter |
| Multiplayer game state sync | Guaranteed position updates, 50ms extra latency from ACK wait | Best-effort position updates at 20Hz, interpolate missing frames | Turn-based strategy (move validity critical) | FPS or racing game (50ms extra latency causes motion sickness) |
| Cloud load balancer health checks | TCP health check confirms application is accepting connections | UDP health check confirms only that the process is running | Workloads where accepting connections implies healthy application | Stateless services (DNS, media servers) where UDP availability equals health |
| Service discovery / gossip | Reliable node list sync, slower convergence (30-60 seconds) | Fast epidemic broadcast (5-10 seconds convergence), tolerates loss | Small clusters under 50 nodes where consistency matters | Large clusters above 500 nodes where convergence speed matters |

## CAP Theorem Positioning

TCP is a **CP** class of transport: it chooses Consistency (ordered, reliable delivery) over Availability (if the network is partitioned, TCP connections stall rather than deliver corrupt data). TCP connections eventually become available again when the partition heals.

UDP is an **AP** class of transport: it chooses Availability (datagrams flow even during partitions) over Consistency (packets may be lost, duplicated, or reordered). UDP never stalls, but it does not promise your data arrives intact.

## Failure Modes

### Failure 1: TCP Retransmission Storm

**What happens**: A single packet loss on a high-bandwidth link triggers fast retransmit, which halves the congestion window. The sender halves its rate, which creates bursty traffic as the window reopens. This burst causes more loss. The congestion window collapses to 1 segment (MSS = 1460 bytes), and the link drops from 10 Gbps to approximately 10 Mbps.

**Root cause**: A buffer overflow at a congested router that drops one packet out of 10,000.

**Blast radius**: All flows sharing the congested router experience throughput collapse. Recovery takes multiple seconds.

**Detection**: Monitor `tcpi_lost` in `/proc/net/tcp`. If a connection shows `tcpi_lost > 0` for more than 10 consecutive monitoring intervals, alert.

**Recovery**: Switch from CUBIC to BBR congestion control. BBR paces at the measured bandwidth rather than probing for loss, so it avoids congestion collapse entirely. Apply this at the kernel level: `sysctl -w net.ipv4.tcp_congestion_control=bbr`.

### Failure 2: UDP Receive Queue Overflow (Datagram Black Hole)

**What happens**: A burst of 10,000 datagrams arrives at the server in 5 milliseconds. The kernel's receive queue (default 208 KB) can hold approximately 200 maximum-sized datagrams. Datagrams 201 through 10,000 are dropped. The application sees no error — the kernel increments `UdpInErrors` silently.

**Root cause**: The application processing rate is slower than the incoming datagram rate, even for 5 milliseconds.

**Blast radius**: All clients whose datagrams arrived during the 5ms burst lose their data. If each datagram carries a different client's DNS query, 9,800 clients see DNS timeouts.

**Detection**: Monitor `netstat -s | grep UdpInErrors` at 1-second intervals. Any increase means datagram loss.

**Recovery**: Increase `SO_RCVBUF` to 4 MB on the socket descriptor. Use `SO_REUSEPORT` to spread incoming datagrams across multiple worker threads, each with its own receive queue.

### Failure 3: TCP Head-of-Line Blocking

**What happens**: Stream A sends packet 1 (seq=100), packet 2 (seq=200), packet 3 (seq=300). Packet 2 is lost. The receiver holds packet 3 in the receive buffer but cannot deliver it because byte 200-299 is missing. The application's read() call blocks. Meanwhile, streams B, C, and D (multiplexed over the same TCP connection via HTTP/2) also stall — their packets arrived, but the ordered byte-stream abstraction prevents delivery past the gap.

**Root cause**: TCP's ordered delivery guarantee — the receiver cannot hand data to the application out of order.

**Blast radius**: Every stream sharing the TCP connection stalls. For HTTP/2 with 100 concurrent requests on one TCP connection, all 100 requests wait for that one retransmission.

**Detection**: Monitor application-level response latency for all streams on a connection. If p50 is 5ms and p99 spikes to 200ms, head-of-line blocking is likely.

**Recovery**: Switch to QUIC (HTTP/3), which runs on UDP and provides per-stream ordering without cross-stream blocking. Or open multiple TCP connections and shard streams across them — though this increases connection overhead.

### Failure 4: TCP SYN Flood

**What happens**: An attacker sends 100,000 SYN packets per second with spoofed source IPs. The server allocates a TCB for each SYN and puts the connection in SYN_RCVD state. Memory fills within seconds. Legitimate clients cannot connect.

**Root cause**: The TCP state machine allocates memory before the handshake completes, and the attacker never sends the third ACK.

**Blast radius**: Complete denial of service for all TCP services on the target machine.

**Detection**: `netstat -s | grep "SYN_RCVD"`. If the count exceeds 10,000, a SYN flood is active. Also monitor `TcpExtTCPAbortOnOverflow` and `TcpExtListenOverflows`.

**Recovery**: Enable `tcp_syncookies = 1`. The kernel encodes connection state in the SYN-ACK sequence number and allocates zero memory until the handshake completes. This is effective against any SYN flood that does not saturate the network link itself.

### Failure 5: UDP Amplification Attack

**What happens**: An attacker sends a small DNS query (60 bytes) with a spoofed source IP to a public DNS resolver. The resolver responds with a DNS response of up to 3000 bytes to the victim's IP. The amplification factor is 50x. The attacker sends 20 Gbps of queries (60-byte packets), and the victim receives 1000 Gbps of response traffic.

**Root cause**: UDP's lack of handshake means the server cannot distinguish a legitimate query from a spoofed one.

**Blast radius**: The entire network path to the victim at the amplified bandwidth level. ISPs drop traffic, causing collateral damage to other customers sharing the same infrastructure.

**Detection**: Monitor outbound traffic volume on DNS servers. A sudden 50x increase in response-to-query ratio indicates amplification.

**Recovery**: Apply BCP38 (source IP validation) at the network edge to prevent spoofed packets from leaving your network. Use Response Rate Limiting (RRL) on DNS servers. On the receiving side, block the attack source IP ranges at the edge firewall.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Custom TCP Retry Layer on Top of UDP

**Mistake**: Implementing your own TCP on top of UDP — sequence numbers, ACKs, retransmission, flow control — in the application layer.

**Why it seemed reasonable**: "TCP is slow. We need faster-than-TCP reliability. Let's build a lightweight version."

**Production impact**: The custom implementation misses edge cases that TCP's 40+ years of refinement have solved: congestion control (TCP Reno, CUBIC, BBR), adaptive RTO (Jacobson's algorithm), selective ACKs, and buffer auto-tuning. Your version either uses too much bandwidth (congestion collapse) or too little (10% of link capacity). DNS-over-QUIC implementations mature enough to be exceptions; yours is not.

**Correct pattern**: Use TCP with tuning (BBR congestion control, `TCP_NODELAY`, increased buffer sizes). If you need faster-than-TCP reliability, use QUIC (Google's battle-tested UDP+reliability layer) or a well-reviewed RTP/RTCP implementation.

### Anti-Pattern 2: TCP for Real-Time Game State

**Mistake**: Using TCP to synchronize player positions in a multiplayer FPS or racing game.

**Why it seemed reasonable**: "We need reliable position data. Players should not teleport."

**Production impact**: Under packet loss, TCP retransmits old position updates while the game world moves forward. Players experience rubber-banding: they receive position 1, then position 1 again (retransmitted), then position 5, with all intermediate positions lost. The head-of-line blocking guarantees the player receives old data while the game waits for the retransmission.

**Correct pattern**: UDP with per-message importance. Position updates (20 Hz, 50 bytes each) are fire-and-forget. Only game-critical actions (pick up weapon, deal damage, score goal) are sent over a thin reliable sub-channel with application-level ACKs and a 200ms timeout.

### Anti-Pattern 3: UDP for Database Writes

**Mistake**: Writing financial transaction data over UDP because "most packets arrive."

**Why it seemed reasonable**: "We benchmarked and UDP is 5x faster. We just need to add checksums."

**Production impact**: A 0.01% packet loss rate on a system processing 100,000 transactions per second means 10 lost transactions per second, or 864,000 lost transactions per day. The database's balance sheet is wrong by millions of dollars, and no error or alarm fired because UDP returned success to every system call.

**Correct pattern**: Always use TCP for database writes. If write throughput is the bottleneck, batch writes (PostgreSQL COPY, MySQL LOAD DATA, or application-level batching in Kafka) rather than switching to UDP.

### Anti-Pattern 4: Silently Ignoring UDP Receive Buffer Drops

**Mistake**: Running a high-throughput UDP service and not monitoring `UdpInErrors`. Over 5% of datagrams are dropped silently, and the application does not notice because it never checks for gaps.

**Why it seemed reasonable**: "UDP is lossy by design. We budgeted for 1% loss."

**Production impact**: The 5% drop rate (including burst drops) causes application-level retry storms. Clients retry, sending more datagrams, causing more drops, causing more retries. The system enters a collapse loop. Budgeting for 1% uniform loss does not account for burst losses at 50%.

**Correct pattern**: Monitor `UdpInErrors` every second. Alert if it increases at all. Set `SO_RCVBUF` to at least 4 MB on production servers. Implement application-level sequence numbers and a gap-detection timer.

### Anti-Pattern 5: One TCP Connection per Request

**Mistake**: Opening a new TCP connection for every HTTP request to a database or caching layer.

**Why it seemed reasonable**: "Connection pooling adds complexity. We open and close quickly."

**Production impact**: At 10,000 requests per second, each connection costs 3 packets for setup (SYN, SYN-ACK, ACK) and 4 for teardown (FIN, ACK, FIN, ACK) = 7 packets overhead minimum. That is 70,000 packets per second just for connection management, consuming 15-20% of CPU. Each connection also enters TIME_WAIT for 60 seconds, consuming ephemeral ports. Ports exhaust in 60 seconds at 10,000/sec x 60 seconds = 600,000 ports — exceeding the 16-bit range.

**Correct pattern**: Use connection pooling (HikariCP, c3p0, Redis connection pool). One TCP connection handles thousands of sequential or multiplexed requests. The connection overhead is paid once.

## When NOT to Use This

### When TCP Is Wrong

- **Sub-10ms latency required end-to-end**: If your system must respond in under 10ms, the TCP handshake (1 RTT) uses the entire budget before any application logic runs. Use QUIC or raw UDP.
- **Packet loss exceeds 1%**: TCP's congestion window collapses at loss rates above 1%. Throughput drops to under 100 Mbps on a 10 Gbps link. Use UDP and design the application to tolerate loss.
- **100,000+ simultaneous connections**: At this scale, the kernel's per-connection memory overhead (128 KB each = 12.8 GB) becomes a capacity problem before your application does. Use UDP or connection rate limiting.

### When UDP Is Wrong

- **Financial transactions, payment processing, or any regulated data**: Regulations (PCI-DSS, SOC2, SOX) require audit trails for all data. TCP's guaranteed delivery with kernel-level ACKs provides the transport-layer proof. UDP does not.
- **Database replication**: Losing a single write transaction corrupts the replica state. The replica cannot detect the gap without application-level sequence numbers that duplicate TCP's functionality.
- **Low data rate with high reliability requirement**: If you send 10 messages per second and every message must arrive, TCP's overhead is negligible (20-60 bytes per packet on 1 KB messages = 2-6% overhead). Use TCP and avoid reinventing reliability.

### Simpler Alternatives

- **Need reliable multicast?** Use QUIC streams or implement application-level fan-out over TCP to each receiver. Do not try to make UDP reliable by hand.
- **Need exactly-once delivery?** That is an application problem, not a transport problem. TCP gives you at-most-once or at-least-once (depending on retransmission behavior). Exactly-once requires application-level deduplication with idempotency keys.
- **Need ulta-low latency (under 1ms)?** Use shared memory for inter-process communication, RDMA for inter-machine, and avoid the transport layer entirely. TCP and UDP both add microseconds of kernel overhead per message.
