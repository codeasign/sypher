## How This Topic Appears in Interviews

Partitioning questions appear in three forms: direct "design a data partitioning strategy" questions, implicit questions about sharding in larger system design problems, and troubleshooting questions about hot spots and rebalancing.

### Direct Phrasings

1. "Design a data partitioning strategy for a ride-hailing platform that stores 5 billion trips per year."

2. "How would you shard a user database for a social network with 500 million users?"

3. "Compare hash partitioning and range partitioning. When would you use each?"

4. "Design a partition key for a time-series metrics system ingesting 10 million data points per second."

5. "How does consistent hashing work? Walk through adding and removing nodes."

6. "Our database has a hot spot — one shard receives 10x more traffic than others. Diagnose and fix."

7. "Design a multi-tenant SaaS database partitioned by tenant. How do you handle a tenant with 100x more data than average?"

8. "Our Cassandra cluster has large partition warnings. What causes this and how do you fix it?"

9. "Design a resharding strategy that migrates from 8 to 64 shards without downtime."

10. "Design a system that partitions event data across both time and geographic region."

### Implicit in Larger Design Problems

11. "Design Twitter." — The tweets table must be partitioned, and the partition key is a critical design decision.

12. "Design YouTube." — Video metadata and view counts each need different partitioning strategies.

13. "Design Uber." — Trip data partitioning by city is a central challenge.

14. "Design a distributed rate limiter." — Partitioning counters by user or by IP affects accuracy and cost.

15. "Design a global chat application." — Message storage must be partitioned by conversation or by server.

## What the Interviewer Is Testing

Partitioning questions test four distinct capabilities:

**1. Data modeling intuition.** Can you identify the natural partition key from the requirements? Do you know that the partition key must have high cardinality and uniform access frequency? Can you estimate data size and partition count?

**2. Failure mode awareness.** Do you know what happens when a partition key is poorly chosen? Can you describe hot spots, data skew, and rebalancing storms? Do you have mitigation strategies ready?

**3. Tradeoff articulation.** Can you explain why hash partitioning is better than range for uniform distribution, and why range is better for time-series queries? Can you discuss the tradeoff between query flexibility and scalability?

**4. Operational maturity.** Do you understand online resharding? Can you describe dual-writes, VReplication, and zero-downtime migration? Do you plan for rebalancing in your architecture from the start?

## FAANG Framework for Partitioning Questions

Use this 6-step framework in your interview response. It mirrors the system design process but focuses specifically on partitioning.

### Step 1: Gather Requirements

**Ask clarifying questions:**
- What is the data size? (Rows, columns, total storage)
- What is the throughput? (Reads and writes per second)
- What are the dominant query patterns? (List the top 3 SQL queries)
- Is there a natural partition key? (Column appearing in 80+ percent of WHERE clauses)
- What are the consistency requirements? (ACID across all data? Read-your-writes?)
- Is there a retention policy? (Can old data be dropped?)

**Bad:** "I will use hash partitioning." (No justification.)
**Good:** "The dominant query is 'get rides for user X'. The rides table has 5 billion rows. User_id is in 90 percent of queries. I will use user_id as the partition key."

### Step 2: Estimate Data Size

**Calculate:**
- Row size: Sum of column sizes (or use averages for variable-length fields)
- Total rows: Current + growth projection (assume 2-3 year planning horizon)
- Total data: Row count x row size x replication factor
- Target shard count: Total data / target shard size (50-200 GB)

**Example:** "5 billion trips at 1 KB each = 5 TB. With 20 percent annual growth for 2 years: 7.2 TB. With replication factor 3: 21.6 TB total storage. Target 200 GB per shard: 108 shards. I will round to 128 shards for binary alignment."

### Step 3: Choose Partition Key

**Evaluate candidates:**
- Candidate 1: user_id. Cardinality: 50M. Appears in 90 percent of queries. Uniform distribution? Yes (well-distributed user IDs). Verdict: Strong.
- Candidate 2: created_at. Cardinality: 5B values. Appears in 30 percent of queries. Uniform distribution? No (recent data is hotter). Verdict: Weak for reads, strong for time-series.
- Candidate 3: city. Cardinality: 10,000. Appears in 60 percent of queries (Uber case). Uniform distribution? No (one city dominates). Verdict: Acceptable if combined with hash.

**Select and justify:** "I choose user_id as the partition key with hash partitioning. It has high cardinality, appears in 90 percent of queries, and hash partitioning ensures uniform distribution. For time-range queries, I will create a separate materialized view partitioned by date for the analytics team."

### Step 4: Plan for Skew

**Identify potential skew:**
- Power users: Top 1 percent of users generate 20 percent of traffic.
- Celebrities: A single user might receive 100x more queries than average.
- Time-of-day: All shards see higher traffic during peak hours, but no shard is disproportionately affected.

**Mitigation plan:**
- For power users: Add a read cache (Redis) in front of the hot shard. Cache user profiles and recent data aggressively.
- For celebrities: Split the celebrity's data across multiple sub-keys. The application fans out reads to all sub-keys and merges.
- For general imbalance: Consistent hashing with virtual nodes. Rebalance when any shard exceeds 150 percent of average load.

### Step 5: Handle Rebalancing

**Describe the approach:**
- Use consistent hashing so that adding a node moves only K/N keys.
- Use virtual nodes (150 per physical node) for fine-grained distribution.
- Pre-split to 4x the expected node count. Logical partitions are reassigned without moving data.
- Implement dual-write during migration: writes go to both old and new locations.
- Rate-limit data transfer to 20 percent of network bandwidth.
- Verify consistency with checksum comparison before switching reads.

### Complete Worked Example: Partitioning Instagram-Like Media Platform

**Problem:** "Design data partitioning for a photo-sharing platform with 1 billion users. Each user uploads photos, follows other users, and views a feed of photos from followed users."

**Step 1: Requirements gathering.**
- Data: photos (2 MB average, 500M new per day), user profiles (1 KB, 1B users), follows graph (10 bytes per edge, 100B edges).
- Queries: feed (get latest photos from followed users) = 60 percent, profile page = 20 percent, upload = 10 percent, search = 10 percent.
- Partition key candidates: user_id appears in feed and profile queries (80 percent), photo_id in none.

**Step 2: Estimate data size.**
- Photos: 500M/day x 2 MB = 1 PB/day. 30-day retention = 30 PB.
- User profiles: 1B x 1 KB = 1 TB (negligible).
- Follows graph: 100B edges x 10 bytes = 1 TB (manageable on a single graph DB node).
- Shard count: 30 PB / 200 GB per shard = 150,000 shards. Too many. Use object storage for photos (S3) and index metadata separately.
- Metadata shards: 500M photos/day x 200 bytes (metadata only) = 100 GB/day. 30 days = 3 TB. Use 32 shards for metadata.

**Step 3: Choose partition key.**
- Metadata: partition by `user_id` hash. 80 percent of queries include user_id.
- Photos: stored in S3. Partition by date (YYYY/MM/DD) for time-based access pattern.
- Follows graph: partition by `follower_id` hash for feed generation.
- Feed cache: Redis cluster, keyed by user_id. Contains pre-computed feed (latest 200 photos from followed users).

**Step 4: Plan for skew.**
- Celebrity users: single author followed by 100M users. Their photos must be fanned out to all followers. Mitigation: queue-based fan-out with 10 workers per celebrity. User's own cache partition handles the fan-out. Cache each photo independently so followers see it when the fan-out completes.
- Hot photos: a viral photo viewed 10M times in 1 hour. Mitigation: CDN cache at edge. The photo is served from edge cache, never hitting the database.

**Step 5: Handle rebalancing.**
- Metadata shards: 32 shards pre-split. Consistent hashing for modulus-free resharding.
- Object storage: no rebalancing needed — S3 handles distribution transparently.
- Feed cache: Redis Cluster with consistent hashing. Add nodes as memory grows.

**Step 6: Explain tradeoffs.**
- "I am optimizing for the feed and profile queries that account for 80 percent of traffic by using user_id as the partition key. The tradeoff is that cross-user queries (find all photos taken in New York) require scatter-gather. I accept this tradeoff because those queries are rare and can be served by a separate Elasticsearch index."
- "I am splitting photo storage from photo metadata. This means most queries (feed, profile) only touch the small metadata shards, while photo data is served from S3 through CDN. The tradeoff is that fetching photo metadata and photo bytes requires two separate lookups. I accept this because the CDN's edge caching eliminates the second lookup for 99 percent of requests."

### Step 6 (General): Explain Trade-offs

**Explicitly state what you are giving up:**
- "I am choosing user_id as the partition key. This optimizes for the 90 percent of user-level queries at the cost of making cross-user analytical queries expensive scatter-gather operations. The analytics team will need a separate data warehouse."
- "I am using hash partitioning for uniform distribution. This means range queries on user_id (like 'all users created in January') cannot prune to a single partition. If range queries become critical, I will add a secondary index on created_at with a partitioned lookup table."
- "I am using consistent hashing instead of mod-N. This makes rebalancing cheaper at the cost of slightly more complex routing logic. The routing table must be stored in an external metadata service (etcd) rather than computed on the fly."

- "Sub-sharding distributes a hot key across multiple physical partitions using a suffix-based key scheme."
- "Cache shielding places a distributed cache in front of a hot shard to absorb read traffic."
- "A hot partition creates a system bottleneck where one shard's throughput limits the whole cluster."

**Performance**
- "Single-shard queries achieve 1-5ms latency with a well-chosen partition key."
- "Scatter-gather queries incur latency equal to the slowest responding shard plus coordination overhead."
- "Data skew reduces effective cluster capacity to the utilization of the hottest shard."
- "Partition pruning is the mechanism by which the query optimizer skips irrelevant partitions."

**Operational**
- "Dual-writes during migration ensure no data loss when switching routing tables."
- "A grace period between read switch and write switch allows rollback if consistency checks fail."
- "Partition size monitoring detects skew before it impacts query performance."

**Architecture**

## Red Flags

## Red Flags

Eight statements or behaviors that signal a lack of depth in partitioning knowledge. Avoid these in interviews.

**Red Flag 1: "I will just use auto-increment ID as the partition key."**
Why it is wrong: Auto-increment IDs cluster writes on the last partition. All new rows go to one shard until the ID range fills. This creates an instant hot spot on the highest-numbered partition.

**Red Flag 2: "I will add more partitions when the data grows."**
Why it is wrong: Adding partitions with mod-N hashing changes the mapping for nearly all existing keys. All data must be moved. This is not a lightweight operation. Use consistent hashing or pre-split from the start.

**Red Flag 3: "Cross-shard queries are fine; I will just fan out to all shards."**
Why it is wrong: Scatter-gather for every query defeats the purpose of partitioning. If cross-shard queries are common, the system should not be partitioned by that key. Either choose a different partition key or use a separate system for global queries.

**Red Flag 4: "Partitioning is always better than a single database."**
Why it is wrong: Partitioning adds complexity, reduces query flexibility, and requires operational maturity. A single database with proper indexing handles 100 GB efficiently. Partitioning is a scaling strategy for when a single node is insufficient.

**Red Flag 5: "I will store all related data in the same database regardless of partition."**
Why it is wrong: In a partitioned system, a "database" is actually a set of independent shard databases. Cross-shard joins are not possible without a coordinator. The schema must be designed so that related data is co-located on the same shard.

**Red Flag 6: "I do not need to worry about hot spots because hash partitioning distributes uniformly."**
Why it is wrong: Hash partitioning distributes keys uniformly by count, not by access frequency. If 1 percent of users generate 50 percent of queries, hash distribution does not prevent hot spots. Access patterns matter as much as key distribution.

**Red Flag 7: "I will handle resharding in a maintenance window."**
Why it is wrong: A maintenance window for resharding at scale means hours of downtime. This is unacceptable for systems that aim for 99.99 percent availability. Online resharding with dual-writes, backfill, and atomic routing table updates is the expected approach.

**Red Flag 8: "I do not need to monitor partition health after setup."**
Why it is wrong: Partition imbalance develops over time as data grows and access patterns change. A partition that is balanced on day one can be 10x skewed by day 100. Continuous monitoring with per-shard latency, size, and throughput metrics is essential.

**Red Flag 9: "I will use auto-increment ID as the shard key with range partitioning."**
Why it is wrong: Auto-increment IDs with range partitioning create a hot spot on the last partition. All new writes go to the highest-numbered partition because new IDs are always larger than existing ones. The last partition receives 100 percent of write traffic while the others are idle. This completely defeats the purpose of partitioning. Use hash partitioning on a high-cardinality key instead.

**Red Flag 10: "I will put all my data in one database and partition it."**
Why it is wrong: Database-level partitioning (PostgreSQL declarative partitioning, MySQL partitioning) splits a table within a single database instance. It does not distribute data across multiple servers. The database is still a single point of failure. The total throughput is still limited by one machine. For horizontal scaling, the data must be split across multiple database instances (sharding), not just partitioned within a single database.

## Interview Walkthrough: Design Partitioning for a Social Platform

### Problem Statement

"Design a data partitioning strategy for a social platform with 200 million monthly active users. Users post text, images, and videos. The feed query ('show posts from people I follow') must load in under 500ms."

### Step-by-Step Solution

**1. Requirements gathering.**
- 200M MAU, 50M daily active users.
- Each user posts 2x per day on average: 100M posts/day.
- Posts: 1 KB text + 100 KB media average. 100M x 100 KB = 10 TB/day of media.
- Feed read: Each user reads 100 feed items per day: 5B feed reads/day. 58,000 reads/second peak.
- Dominant query: "SELECT posts FROM feed WHERE user_id = ? ORDER BY created_at DESC LIMIT 50."

**2. Partition key selection.**
- The feed query filters by user_id. user_id appears in 99 percent of read queries.
- user_id has high cardinality (200M distinct values).
- Hash partitioning by user_id ensures uniform distribution.
- Posts and feed generation should be sharded by the same user_id for co-location.

**3. Capacity estimation.**
- 100M new posts per day. 10 TB of media per day.
- 30-day retention for primary feed: 3B posts, 300 TB.
- Long-term storage: archival to S3 with partition by date.
- Shard count: 300 TB / 200 GB per shard = 1,500 shards. With over-provisioning for 2 years: 3,000 shards.
- Scaling: 256 physical nodes, each managing 12 logical shards.

**4. Architecture.**
- Hash partitioning by user_id across 3,000 logical shards.
- Consistent hashing for node assignment (virtual nodes).
- Routing via a stateless proxy layer (NGINX + Lua) that maps user_id to shard.
- Feed generation: fan-out-on-write. When user A posts, write to the feeds of all followers. Each follower's feed is stored on their shard by user_id.
- Media storage: separate object store (S3) partitioned by date + user_id hash prefix.

**5. Hot spot mitigation.**
- Celebrity users: distribute their feed generation across multiple workers. Cache their recent posts in a Redis cluster.
- Peak hours: burst capacity via read replicas per shard.
- Imbalance detection: alert when any shard exceeds 150 percent of average CPU or storage.

**6. Rebalancing.**
- Pre-split to 3,000 logical shards. Add physical nodes as data grows. Reassign logical shards without moving data.
- If logical shards must be split: Vitess-style VReplication with dual-writes, backfill, atomic cutover.

**7. Tradeoffs.**
- Optimized for user-level feed queries at the cost of cross-user analytical queries.
- Feed generation uses fan-out-on-write, which is write-heavy. Writes scale with shard count.
- Post deletion requires deleting from all followers' feeds — a scatter operation.

## Interview Cheat Sheet: Partitioning Key Concepts at a Glance

**Partition key selection** — Always pick the column that appears in 80 percent+ of WHERE clauses. It must have high cardinality (many distinct values) and uniform access frequency.

**Strategy selection:**
- Range: time-series, retention-driven, time-range queries dominate.
- Hash: opaque IDs, uniform distribution needed, equality lookups dominate.
- List: geographic region, tenant ID, category values with cardinality under 10,000.
- Composite: both range and hash needed — range for retention, hash for distribution.

**Shard count formula:** Total data (including growth for 2 years) / target shard size (50-200 GB). Round up to the nearest power of 2 for binary alignment.

**Consistent hashing:** Hash ring with virtual nodes (150 per physical node). When a node joins/leaves, only K/N keys move. Key is assigned to the nearest virtual node clockwise from its hash position.

**Three types of rebalancing:**
1. Consistent hashing: data moves to immediate neighbors only. Best for incremental growth.
2. Pre-split: logical partitions (1024) remapped to physical nodes. No data movement for scale-out.
3. Mod-N rehash: all keys remap. Expensive. Avoid.

**Hot spot mitigation:**
- Sub-shard hot keys across multiple physical partitions.
- Cache layer (Redis, CDN) absorbs 99 percent of reads to hot key.
- Rate-limit writes to hot shard. Shed load before the shard fails.

**Cross-shard query types:**
- Scatter-gather: send to all shards, merge results. Latency = slowest shard.
- Coordinator merge: collect partial results, sort/aggregate. Memory = O(N + K).
- Distributed join: join tables across shards. Use co-location or broadcast join.

**Failover process (3 replicas):**
- Primary fails → heartbeat missed for 3 intervals.
- Replica with highest WAL position promoted.
- Routing table updated → queries routed to new primary.
- New replica provisioned from snapshot + WAL replay.
- Total: 10-60 seconds.

**Three must-monitor metrics:**
- Per-shard p99 latency vs cluster median.
- Partition size ratio (largest / smallest).
- Per-shard replication lag in seconds.

**Interview answer structure for any partitioning question:**
1. Ask about query patterns (what columns appear in WHERE clauses)
2. Identify the natural partition key (must appear in 80 percent+ of queries)
3. Calculate data size and shard count (total data / 200 GB target per shard)
4. Choose strategy (hash, range, list, or composite based on query patterns)
5. Address hot spots (cache layer, sub-sharding, virtual nodes)
6. Explain rebalancing strategy (consistent hashing, pre-split, or online resharding)
7. Describe tradeoffs (cross-shard queries, distributed transactions, operational complexity)

**Three sentences to always include in your answer:**
- "The partition key must match the dominant access pattern."
- "Hash partitioning distributes uniformly by key count, but access frequency skew requires separate mitigation."
- "I will design for rebalancing from the start using consistent hashing with virtual nodes."

**Time management for partitioning interview questions:**
- 0-2 minutes: Ask clarifying questions about data size, throughput, query patterns, and consistency requirements.
- 2-5 minutes: Estimate data size, calculate shard count, and identify the partition key candidate.
- 5-10 minutes: Describe the architecture — routing layer, partition strategy, and cross-shard query handling.
- 10-15 minutes: Address failure modes — hot spots, data skew, rebalancing, and disaster recovery.
- 15-20 minutes: Explain tradeoffs — what you optimized for and what you sacrificed.
- 20-25 minutes: Handle follow-up questions and edge cases (what if the partition key changes, what if a shard fails).

## Common Mistakes in Partitioning Interviews

Avoid these mistakes that signal surface-level understanding:

**Mistake: Jumping to a solution without understanding the query pattern.** The candidate says "I'll use hash partitioning by user_id" before asking what the queries look like. The interviewer is evaluating whether you match the partition key to the access pattern. Always start with: "Can you tell me about the dominant query patterns? What columns appear in the WHERE clause most frequently?"

**Mistake: Ignoring the growth trajectory.** The candidate designs for the current data size without considering growth. A design that works for 1 TB fails at 10 TB. Always ask: "What is the expected growth rate and planning horizon?" Design for 2-3 years of growth.

**Mistake: Over-engineering for a small dataset.** The candidate proposes Vitess + 128 shards + consistent hashing for a 50 GB dataset that gets 1,000 queries per day. The interviewer is evaluating whether you understand when partitioning is not needed. Propose partitioning only when it is justified by data size, throughput, or availability requirements.

**Mistake: Not mentioning cross-shard queries.** The candidate designs a partitioning scheme and does not address what happens when a query does not include the partition key. The interviewer is evaluating whether you understand the tradeoff. Always explain: "This design optimizes for queries with the partition key. For queries without it, the system must use scatter-gather, which is slower."

**Mistake: Proposing a fixed partition count without a rebalancing plan.** The candidate says "64 shards" without explaining how to add more. The interviewer is evaluating whether you plan for growth. Always describe the rebalancing strategy: "I will use consistent hashing with virtual nodes so that adding shards moves only K/N of the data."

**Mistake: Forgetting about hot spots.** The candidate says "hash partitioning distributes data uniformly" without addressing access skew. The interviewer is evaluating whether you anticipate failure modes. Always say: "Hash distribution is uniform by key count, but access frequency may not be. I will plan for hot spots by monitoring per-shard query latency and implementing sub-sharding for hot keys."

## Additional Interview Questions and Model Answers

### Q11: How do you choose between range partitioning and hash partitioning?

**Model answer:** The choice depends on the dominant query pattern. Range partitioning is appropriate when queries filter by a continuous range of the partition key and when data retention is required. Time-series data, logs, and historical records all fit this pattern. Hash partitioning is appropriate when the partition key is opaque (UUID, user ID), when uniform distribution is important, and when range queries on the partition key are rare.

Use a decision matrix: if the WHERE clause contains a range predicate on the partition key in 50 percent or more of queries, prefer range partitioning. If the WHERE clause contains an equality predicate on the partition key, prefer hash partitioning. If both patterns are common, use composite partitioning — range at the top level for retention, hash within each range for distribution.

### Q12: How do you handle a hot spot on a hash-partitioned table?

**Model answer:** A hot spot can arise from two causes: data skew (one partition holds more data) or access skew (one partition receives more queries). Each requires a different mitigation.

For data skew: use consistent hashing with virtual nodes. The hot partition's 150 virtual nodes are spread across all physical nodes, so no single physical node is overloaded. Alternatively, use range partitioning within the hash partition to further subdivide the data.

For access skew: the hot key (celebrity user, popular product) is accessed more frequently than other keys. Mitigation: sub-shard the hot key across multiple physical partitions by appending a suffix. For user 42, create keys user_42_0 through user_42_15. Distribute read traffic across all 16 sub-keys. The application fans out reads and merges results. This increases read latency for the hot key (from one shard query to 16 parallel shard queries) but distributes the load across 16 shards instead of 1.

For both types: add a cache layer in front of the hot shard. A Redis cache with 60-second TTL absorbs 99 percent of reads to the hot key, reducing shard load by 100x. Cache shielding is the simplest and most effective mitigation for access skew.

### Q13: How does consistent hashing differ from simple mod-N hashing for rebalancing?

**Model answer:** Mod-N hashing assigns a key to partition `hash(key) % N`. When N changes to N', every key's partition assignment changes for approximately `(N' - N) / N'` of keys. For a change from 8 to 16 shards, 50 percent of keys move. For a change from 8 to 9 shards, 89 percent of keys move — a massive data migration for a tiny capacity increase.

Consistent hashing solves this by placing both nodes and keys on a hash ring (typically a 64-bit circular address space). Each physical node maps to 150 virtual nodes at different positions on the ring. A key is assigned to the nearest virtual node clockwise from its hash position. When a node joins or leaves, only the keys whose virtual nodes are immediately adjacent on the ring are affected. This is approximately 1/N of total keys for N physical nodes. For a change from 8 to 9 nodes with consistent hashing, only 11 percent of keys move (approximately 1/9), compared to 89 percent with mod-N.

The tradeoff: consistent hashing requires a routing table lookup (stored in memory, maintained via a metadata service), while mod-N hashing computes the partition assignment without any external state. Consistent hashing also introduces the complexity of virtual nodes and ring management. The operational benefit of minimal data movement during rebalancing outweighs the complexity cost for any system that expects to grow.

### Q14: Design a partitioning strategy for a real-time leaderboard.

**Model answer:** A real-time leaderboard for an online game with 10 million active players. The leaderboard shows the top 100 players by score, updated every 5 seconds.

Partitioning approach: Use Redis Sorted Sets as the storage layer. Redis automatically partitions data across a Redis Cluster using hash slots (16,384 slots distributed across nodes). The partition key is the game level or tournament ID. Each level's leaderboard is a separate sorted set.

For a single global leaderboard: the sorted set key is "leaderboard:global" with player IDs as members and scores as values. This single sorted set fits in memory (10M members x 50 bytes = 500 MB). No horizontal partitioning is needed for the leaderboard itself — one Redis instance handles it.

For per-level leaderboards (100 levels, 100K players each): the sorted set key is `"leaderboard:level:{level_id}"`. Redis Cluster distributes the 100 sorted sets across nodes by hash slot. Each sorted set is small (100K x 50 bytes = 5 MB) and fits entirely in memory.

For cross-level aggregation (combine scores across all levels for a global leaderboard): the application reads all 100 sorted sets, merges by summing player scores, and returns the top 100. This is a scatter-gather across Redis Cluster nodes. With 100 sorted sets on a 6-node cluster, the scatter-gather takes 50-100ms — acceptable for a 5-second refresh interval.

Alternative: use a materialized view with a streaming pipeline. Player scores are written to a Kafka topic partitioned by level_id. A stream processor (Kafka Streams, Flink) aggregates scores per player and writes to a Redis sorted set. The stream processor handles the cross-level aggregation in the stream processing layer, so the Redis query is a simple sorted set range query — no scatter-gather needed.

### Q15: What are the most common causes of data loss in partitioned systems?

**Model answer:** Four causes account for the majority of data loss incidents in partitioned systems:

1. Dual-write failure during resharding. The application writes to both the old and new shard locations. If one of the writes fails and the error is not handled, the data exists in only one location. If the migration continues without detecting the partial write, the data is lost. Mitigation: make dual-writes atomic using a transaction or a two-phase commit between the shards. If atomicity is not possible, run a reconciliation job after each batch of dual-writes.

2. Replica promotion with asynchronous replication. When a shard primary fails and a replica is promoted, any writes that were acknowledged by the primary but not replicated to the replica are lost. Mitigation: use synchronous replication (semi-sync in MySQL, synchronous_commit in PostgreSQL) to ensure every acknowledged write exists on at least one replica.

3. Routing table staleness. The routing table points to an old shard that has been retired. Writes go to a shard that is no longer in the topology, and the data is never visible to queries. Mitigation: implement a grace period after routing table updates. The old shard remains available for 24 hours after the new shard is active. Run a reconciliation job to catch any writes that hit the old shard during the grace period.

4. Partition drop with unarchived data. A retention policy drops a partition that still contains data needed for compliance or audit. Mitigation: enforce a two-step deletion process. First, archive the partition to cold storage. Second, only drop after confirming the archive is complete and accessible.
