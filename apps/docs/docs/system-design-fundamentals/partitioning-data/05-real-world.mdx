## How Companies Partition Data at Scale

The following case studies show how four major technology companies implemented partitioning to solve specific scalability challenges. Each case study begins with the problem, describes the architecture, and explains the key design decisions.

---

## Case Study 1: Uber вҖ” Trip Data Partitioned by City and Date

### The Problem

Uber processes 25 million trips per day across 70 countries and 10,000 cities. Each trip generates a record with tens of columns: pickup location, dropoff location, route, fare, driver, rider, timestamp, surge multiplier, payment method, and ratings. The trip table grows by 25 million rows per day вҖ” approximately 5 billion rows per year.

The original architecture stored all trips in a single PostgreSQL database. By 2015, the database had reached 50 TB. Queries for "show all trips in San Francisco on March 15" scanned the entire table and took over 30 seconds. `ALTER TABLE` operations locked the table for up to 45 minutes. The database could not keep up with the 10,000 writes per second during peak hours.

### The Architecture

Uber built Schemaless, a partitioned document database on top of MySQL. The key innovation was a two-level partitioning scheme: trips were partitioned first by city, then by day within each city.

### Key Design Decisions

**City as partition key.** Uber's analysis showed that 90 percent of queries filter by city. A dispatcher or a rider rarely queries across cities вҖ” they care about their current city. City-level sharding means that 90 percent of queries prune to exactly one shard group.

**Date as sub-partition.** Within each city, data is partitioned by day. This enables two critical operations: time-range queries (show trips from yesterday) prune to exactly one daily partition, and data retention is implemented by dropping daily partitions older than 90 days. Dropping a partition is a metadata operation that takes milliseconds.

**Schemaless document store.** Uber stored trip data as JSON documents in MySQL. This avoided schema changes during the rapid evolution of the trip data model. The database had no fixed schema вҖ” the application interpreted the JSON. This was a pragmatic choice, not a best practice, but it allowed Uber to iterate quickly.

**Ringpop for consistent hashing.** Uber used Ringpop, a consistent hashing library, to assign shard groups to physical nodes. When a node joined or left, only the affected shard groups moved. Ringpop used virtual nodes (128 per physical node) for even distribution.

### Results

- Query latency for dispatcher queries: reduced from 30 seconds to under 50ms.
- Table lock contention: eliminated. Maintenance operations run per-partition.
- Write throughput: scales horizontally. Each shard group handles 2,000-5,000 writes per second.
- Data retention: 90-day rolling window. Dropping 365 partitions per year (one per day per city) is automated.

### Lessons Learned

- City was the right partition key because it matched the dominant query pattern. Verify this with query log analysis before committing.
- Two-level partitioning is essential for time-series data at Uber's scale. A single level would not provide both geographic isolation and time-based retention.
- Schemaless JSON was a pragmatic tradeoff that sacrificed queryability for development velocity. Later, Uber migrated to DocStore (a more structured system).

## Case Study 2: Discord вҖ” Channel Data Sharded by Guild ID

### The Problem

Discord stores every message sent in every channel across every server (guild). As of 2023, Discord had over 500 million messages sent per day across millions of guilds. The messages table in Cassandra grew to tens of billions of rows. Queries to load a channel's message history вҖ” "show the last 50 messages in channel 12345" вҖ” began to slow down as partition sizes grew.

### The Architecture

Discord uses Cassandra, a distributed key-value store with automatic partitioning. Cassandra partitions data by the primary key's partition key component. Discord chose guild_id as the partition key and channel_id + message_id as the clustering key.

### Key Design Decisions

**Guild ID as partition key.** All messages for a guild must be co-located on a single Cassandra node. This allows the query "load channel history for the current guild" to hit exactly one node. Cross-guild queries are not needed вҖ” a user is always viewing messages in one guild at a time.

**Channel ID and message ID as clustering columns.** Within a guild partition, messages are ordered by channel_id (ascending) and then message_id (descending). Loading the last 50 messages in a channel is a sequential read of the first 50 rows in the clustering range вҖ” the fastest possible Cassandra read pattern.

**Large partition problem.** Discord's largest guilds have over 1 million messages in a single channel. The partition for that guild exceeds Cassandra's recommended maximum of 100 MB per partition. Discord implemented a two-tier approach: the most recent 100,000 messages per channel are stored in the primary partition, and older messages are moved to a separate cold storage system.

### Results

- Message load latency: 2-5ms for channel history queries.
- Write throughput: Cassandra handles 500 million writes per day across the cluster.
- Availability: Cassandra's AP model means no downtime during node failures. Discord has not lost message availability since the migration.

### Lessons Learned

- Partition key must match the query pattern. Guild_id works because 99 percent of queries are within a guild.
- Cassandra's partition size limit (100 MB recommended) is real. Plan for it. Monitor partition sizes with `nodetool cfstats`.
- Large partitions are a Cassandra anti-pattern. Discord's cold storage tier was an architectural response to this limitation.

## Case Study 3: Netflix вҖ” Video Metadata Sharded by Title ID

### The Problem

Netflix's catalog contains over 18,000 titles, each with rich metadata: title, description, cast, crew, genres, ratings, availability regions, audio tracks, subtitle tracks, artwork URLs, and licensing terms. The metadata is stored in DynamoDB and accessed by virtually every service вҖ” the browse service, search service, recommendation engine, and player service all read metadata for every request.

The original architecture stored all metadata in a single DynamoDB table with title_id as the partition key. As the catalog grew and metadata complexity increased, the table exceeded 10 TB. The 3 KB per-title average grew to 50 KB as artwork URLs and multi-language descriptions were added. Query latency for batch operations (loading metadata for 200 titles) increased from 20ms to 150ms.

### The Architecture

Netflix partitioned metadata across multiple DynamoDB tables by title type and access pattern. Each table uses title_id as the partition key but stores different subsets of columns.

### Key Design Decisions

**Vertical partitioning across tables.** Netflix applied vertical partitioning first: frequently accessed columns go into a hot table, large rarely accessed columns go into a cold table, and relationship data goes into a separate table. This is the same pattern as vertical partitioning but applied at the DynamoDB table level.

**Hot table is tiny.** The hot table contains only the columns needed for browsing and search results: title, year, rating, genres, runtime, and box office. At 200 bytes per title, the entire hot table is 3.6 MB вҖ” it fits in a single DynamoDB DAX cache node. Queries never hit disk.

**Cold table stores large payloads.** The cold table holds multi-language descriptions, artwork URLs, and other large fields. It is accessed only when a user opens a title detail page. The cold table is partitioned by the same title_id, so the application can fetch hot and cold data in parallel with two DynamoDB queries.

### Results

- Browse query latency: 1-3ms (hot table, cached in DAX). Down from 20ms.
- Detail page latency: 5-10ms (hot table + cold table in parallel).
- DynamoDB throughput costs: reduced by 60 percent because the hot table is 10 percent of the original data size.
- Cache hit rate: 99.8 percent for the hot table.

### Lessons Learned

- Vertical partitioning before horizontal partitioning. Netflix's biggest win came from splitting columns, not rows. The hot table is so small that it does not need horizontal sharding at all.
- Right-size the partition key. Netflix uses title_id, which has exactly 18,000 distinct values. This is enough for DynamoDB to distribute evenly.
- DAX caching eliminates the hot table read bottleneck. When the hot table fits in cache, horizontal scaling of reads is unnecessary.

## Case Study 4: Cloudflare вҖ” DNS Query Logs Partitioned by Time and Data Center

### The Problem

Cloudflare handles 20 percent of the world's web traffic. Every DNS query (tens of millions per second) generates a log entry. Analyzing these logs requires querying petabytes of data across hundreds of data centers. The logs are stored in ClickHouse, a columnar analytics database.

### The Architecture

Cloudflare partitions DNS logs by time (hourly) and data center. The partition key is a composite of `(event_date, datacenter_id)`.

### Key Design Decisions

**Time-date composite partition key.** The partition key `(event_date, datacenter_id)` serves two purposes. Time-range queries prune to the relevant date. Data center queries prune to the relevant data center. Most queries filter by both dimensions.

**Sub-partitioning by hour.** Within each (date, DC) partition, ClickHouse organizes data by hour. A query for "the last hour from IAD" scans one 18 GB sub-partition instead of the full 450 GB daily partition.

**Data retention via partition drop.** Cloudflare retains DNS logs for 90 days. Dropping the oldest day's partitions is a metadata operation вҖ” ClickHouse removes the partition directory from the filesystem. No `DELETE FROM` scanning petabytes.

### Results

- Query latency: 200ms for common queries (down from 30+ seconds on the previous Hive-based system).
- Data ingestion: 10 million events per second across all data centers.
- Storage efficiency: 10:1 compression ratio on DNS logs.
- Retention cost: nearly zero. Dropping partitions is free.

### Lessons Learned

- Composite partition keys are essential for multi-dimensional queries. A single partition key would not handle the time + data center query pattern.
- Time-based partitioning with retention is the killer feature. Partition drop is dramatically cheaper than row deletion.
- ClickHouse's partition pruning is extremely efficient. The query planner uses min/max indexes per partition to skip unnecessary partitions before scanning.

## Case Study 5: YouTube вҖ” Vitess at Google Scale

### The Problem

YouTube's user data вҖ” subscriptions, watch history, playlists, preferences вҖ” was stored in a single large MySQL database. By 2010, the database had reached 50 TB. A single `ALTER TABLE` for a schema change took days because MySQL 5.5 created a temporary table for the entire 50 TB dataset. Schema changes blocked writes for hours. YouTube's rapid feature development required weekly schema changes. The monolithic database was the bottleneck on product velocity.

### The Architecture

Google built Vitess to solve YouTube's MySQL scalability problem. Vitess introduced a sharding layer on top of MySQL that provided transparent partitioning, online schema migration, and automated resharding.

**Keyspace ID abstraction.** Vitess derives a keyspace ID from the primary key of each row. For YouTube's user table, the keyspace ID is based on `user_id`. Vitess supports both range-based and hash-based sharding. Range-based sharding maps key ranges to shards. Hash-based sharding computes `hash(keyspace_id) % shard_count`. The application provides the keyspace ID вҖ” it does not need to know the shard topology.

**VTGate routing proxy.** All queries go through VTGate, a MySQL-compatible proxy. VTGate parses each SQL query, extracts the keyspace ID from the `WHERE` clause, looks up the target shard in the topology service, and forwards the query. VTGate handles scatter-gather queries by fanning out to all relevant shards and merging results.

**VReplication for resharding.** Vitess's VReplication technology enables online shard splitting and merging. VReplication uses row-based streaming replication to copy data from a source shard to one or more destination shards. The destination shards filter rows based on their new key range. VReplication tracks replication lag using GTID-based position tracking. When lag reaches near-zero seconds, Vitess performs an atomic cutover: the routing table is updated to point to the new shards, and write traffic is redirected atomically.

**Online schema migration (Online DDL).** Vitess supports schema changes through VReplication. When a schema change is requested, Vitess creates a new table with the new schema on each shard, starts VReplication from the old table to the new table, and performs a cutover when replication catches up. This avoids the `ALTER TABLE` lock that plagued YouTube's original architecture. The application sees zero downtime during schema changes.

### Results

- Schema changes: reduced from days to minutes. Online DDL eliminated lock contention.
- Write throughput: scales linearly with shard count. YouTube added shards as the user base grew from 100 million to 2 billion monthly active users.
- Resharding: runs online with zero application downtime. VReplication handles the data movement without blocking queries.
- MySQL remains the storage engine. Vitess adds partitioning capabilities without changing the underlying database. YouTube's DBAs continue to use their existing MySQL expertise.

### Lessons Learned

- The keyspace ID abstraction allows Vitess to support both range and hash partitioning with the same architecture. The application does not care about the partitioning strategy at the query level вҖ” it only provides the keyspace ID in its data model.
- Online DDL is one of the most valuable features of partitioned systems. The ability to evolve the schema without locking the table is a primary motivation for partitioning at YouTube's scale.
- Vitess's architecture proved that MySQL could scale to YouTube-sized workloads. Vitess became an independent CNCF project and is now used by Slack, Square, Pinterest, and Nozzle. The project graduated from CNCF incubation in 2023.

## Open Source and Cloud Partitioning Tools

### Open Source

**Vitess.** MySQL-compatible sharding layer. Used by YouTube, Slack, and Square. Provides transparent sharding with VTGate proxy, online schema migrations, and automated resharding. Supports both range-based and hash-based sharding.

**Citus (now Azure Cosmos DB for PostgreSQL).** PostgreSQL sharding extension. Distributes tables across worker nodes by a distribution column. Supports co-located joins, distributed queries, and reference tables. Citus handles the partitioning transparently вҖ” the application queries a single PostgreSQL endpoint.

**Cassandra.** Peer-to-peer distributed database with automatic partitioning. Uses consistent hashing with virtual nodes. The partition key is specified in the primary key definition. Cassandra handles partition splitting, replication, and rebalancing automatically.

**MongoDB.** Document database with native sharding. Supports range, hash, and zone-based sharding. The shard key is specified at collection creation. MongoDB uses a config server replica set to store the routing table. Shard balancing is automatic.

**Elasticsearch.** Search engine with index sharding. Each index is partitioned into primary shards and replica shards. Shard count is specified at index creation. Rebalancing happens automatically when nodes join or leave.

**Kafka.** Distributed event streaming platform. Topics are partitioned across brokers. Partition count determines the maximum parallelism for consumers. Key-based partitioning ensures ordering within a partition. Kafka's partition assignment to brokers is managed by the controller broker using a rack-aware algorithm. When a broker fails, the controller reassigns partition leadership to remaining brokers. Kafka's partition rebalancing protocol (cooperative rebalance) rebalances consumer group assignments without stopping all consumers.

### Deployment Considerations for Open Source Tools

Each tool has specific operational considerations for partitioning:

**Vitess:** Requires a topo service (etcd, ZooKeeper, or Consul) for shard topology storage. Each shard runs a VTTablet process that manages a MySQL instance. VTGate proxies are stateless and scale horizontally. Online DDL requires VReplication, which needs sufficient WAL retention on the source shard. Recommended: at least 3 VTGate instances behind a load balancer, 3 topo service nodes, and a 3-node Orchestrator cluster for automated failover.

**Citus:** The coordinator node is a single point of entry. If the coordinator fails, no new queries can be routed. Deploy the coordinator with a hot standby using streaming replication. Worker nodes are independent PostgreSQL instances. Citus co-location requires that tables share the same distribution column and the same shard count. Add workers by running `SELECT master_add_node('new-worker-hostname')`.

**Cassandra:** Uses the gossip protocol for node discovery and failure detection. The partition key is specified in the table schema. Each node owns a range of the consistent hash ring. Virtual nodes (vnodes) are enabled by default (256 per node). `nodetool repair` must run periodically to ensure consistency across replicas. Cassandra does not support automatic partition splitting вҖ” partition count is determined by the number of tokens, which is configured at cluster creation.

**MongoDB:** Sharding is configured through a shard key specified at collection creation. MongoDB uses a config server replica set (3 nodes minimum) to store the routing table. Mongos routers are stateless and scale horizontally. The balancer process automatically migrates chunks between shards when distribution is uneven. Chunk splitting happens at a configurable size threshold (default 64 MB per chunk).

**Elasticsearch:** Shard count is configured at index creation and cannot be changed without reindexing. The `_split` API can increase shard count for an index by a factor of 2, but it requires the index to be read-only during the split. Elasticsearch's cluster allocation system automatically distributes shards across nodes. Use shard allocation awareness to distribute shards across availability zones.

### Cloud Providers

**AWS Aurora.** Auto-sharding storage. Aurora's storage layer is automatically partitioned into 10 GB segments across 3 availability zones. The database node sees a single logical volume. No manual sharding needed.

**GCP Cloud Spanner.** Globally distributed relational database with automatic partitioning (splitting). Spanner splits and merges tablet ranges automatically based on load. The application sees a single database. Partitioning is completely transparent.

**Azure Cosmos DB.** Globally distributed NoSQL database. Partition key is specified at container creation. Cosmos DB automatically manages physical partition splits. Logical partitions are limited to 20 GB each. Cosmos DB supports multiple consistency levels ranging from strong (linearizable) to eventual (bounded staleness). The partition key choice determines RU (request unit) throughput distribution вҖ” choose a key with high cardinality to distribute throughput evenly across physical partitions.

### Migrating Between Partition Strategies

Teams sometimes need to change their partitioning strategy as workloads evolve. The most common migration is from range partitioning to hash partitioning, or vice versa.

**From range to hash:** A system initially partitioned by date (monthly ranges) discovers that the dominant query pattern has shifted from time-range queries to user-level queries. All queries now filter by user_id, but the data is partitioned by month. Every user-level query is a scatter-gather across 12+ monthly partitions. Migration: create a new hash-partitioned table by user_id, enable dual-writes (new data goes to both tables), backfill historical data from the range-partitioned table to the hash-partitioned table, verify consistency, switch reads to the new table, drop the old table.

**From hash to range:** A system initially partitioned by user_id (hash) discovers that the dominant query pattern has shifted from user-level queries to time-range analytical queries. The engineering team needs to run monthly aggregates. Every aggregate query is a scatter-gather across 256 shards. Migration: create a new range-partitioned table by month, enable dual-writes, backfill historical data, verify consistency, switch reads. The hash-partitioned table remains as the write-optimized store for user-level queries. The range-partitioned table becomes the read-optimized store for analytics. The system now operates two partitioned tables with different strategies вҖ” a form of CQRS.

**Key insight:** Migrating between partition strategies is feasible but expensive. The migration requires the same dual-write, backfill, verify, and cutover process as the initial partitioning migration. Budget 4-8 weeks for the migration on a system with 5+ TB of data. The cost is justified only if the query performance improvement is at least 10x for the dominant query pattern.

## Common Migration Patterns

Each company in this section followed a distinct migration path from monolithic to partitioned. The patterns reveal useful lessons.

**Uber's migration pattern: Schema-less first, structured later.** Uber built Schemaless as a thin partition layer on top of existing MySQL instances. The team did not rewrite the entire storage layer at once. Instead, they added a routing layer (Ringpop + dispatcher) that sat in front of the existing MySQL instances and gradually migrated data from monolithic tables to partitioned shards. This incremental approach meant Uber could start partitioning the highest-traffic cities first and expand to the rest over weeks, not months.

**Discord's migration pattern: Cassandra as the partitioned foundation.** Discord chose Cassandra from the start for its message storage, with the understanding that Cassandra's automatic partitioning would handle growth. The migration was not from monolithic to partitioned, but from a prototype storage (flat files) to a production-grade partitioned database. Discord's key insight was choosing a database that already had partitioning built in, avoiding the need to build a custom partition layer.

**Netflix's migration pattern: Vertical split before horizontal shard.** Netflix's first partitioning step was vertical: split the wide metadata table into hot and cold tables. This required no routing layer, no rebalancing, no cross-shard coordination. Only after the hot table proved too small to need horizontal sharding did Netflix consider DynamoDB sharding вҖ” and ultimately did not need horizontal sharding because the hot table fits in DAX cache.

**Cloudflare's migration pattern: Replace, not migrate.** Cloudflare's DNS logs were stored in Apache Hive on HDFS. The migration to ClickHouse was a full replacement: new data was written to both systems during the migration window, historical data was backfilled into ClickHouse in parallel, and the old Hive system was decommissioned after verification. Cloudflare chose to rebuild rather than incrementally partition the existing Hive setup.

## Lessons Learned

**1. Match the partition key to the access pattern.** Every case study validates this rule. Uber partitions by city because 90 percent of queries filter by city. Discord partitions by guild because users always view messages within one guild. Analyze query logs before choosing a partition key.

**2. Plan for rebalancing from day one.** The naive architecture (mod-N hashing, hardcoded routing) works at day one and fails at month six. Build the consistent hashing ring and routing layer before the first production load. Adding these after the data grows is painful.

**3. Avoid cross-partition queries.** Design the schema so that related data lives on the same partition. Co-locate orders with their customer. Co-locate messages with their channel. Every cross-partition query is a tax on performance.

**4. Monitor partition imbalance.** Track partition size, query latency, and write throughput per partition. Set alerts before imbalance causes production issues. Automated rebalancing is worth the investment.

**5. Test partition management operations.** Test resharding, partition splits, and rebalancing in a staging environment with production-scale data. These operations are the most common source of production incidents in partitioned systems.

**6. Vertical partitioning first, horizontal second.** Netflix's biggest performance win came from splitting hot and cold columns. The hot table was so small it did not need horizontal sharding at all. Vertical partitioning is free вҖ” it requires no routing layer, no rebalancing, and no cross-shard coordination.

**7. Time-based partitioning for retention.** If data has a retention window, partition by time. Dropping a partition is orders of magnitude cheaper than running `DELETE FROM ... WHERE timestamp < ...`. This applies to both databases (PostgreSQL, MySQL) and analytics engines (ClickHouse, BigQuery).

**8. Two-level partitioning is often necessary.** One level of partitioning is not enough for complex data. Uber uses city then date. Cloudflare uses date then data center. Composite partitioning provides both dimensions of pruning without requiring a single complex partition key.

**9. Test partition failure recovery.** The most dangerous assumption in partitioned systems is that replica promotion works. Test it. Select a shard at random during peak traffic, kill its primary, and measure the time to full recovery. Run this drill quarterly. A partition that fails during the drill is a partition that will fail during a real incident.

**10. Budget headroom for rebalancing.** Never run a partitioned cluster above 60 percent disk capacity. Rebalancing requires copying data between nodes. If the cluster is at 90 percent capacity, there is no room for the temporary data copies that rebalancing creates. The rebalancing process will fail with disk-full errors, leaving the cluster in an inconsistent state. Budget 40 percent headroom for operations.

**11. Invest in automation before the first incident.** A rebalancing runbook that requires 15 manual steps will fail when executed under pressure by an on-call engineer at 3 AM. Automate partition health checks, rebalancing triggers, and failover procedures before they are needed. The automation investment pays for itself in the first incident it prevents.

**12. Consider whether partitioning is needed at all.** The case studies in this section come from companies operating at planetary scale вҖ” billions of operations per day, petabytes of data. For 95 percent of applications, a well-tuned monolithic database with proper indexing, connection pooling, and caching handles the workload efficiently. The operational cost of partitioning (monitoring, rebalancing, schema changes, cross-shard query handling) is justified only when a single node is insufficient for throughput, storage, or availability requirements.
