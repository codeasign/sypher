## Core Tension

Partitioning improves scalability and performance for single-partition operations but makes cross-partition operations expensive or impossible. This is the fundamental tension of partitioned systems: **you trade query flexibility for data scalability.**

A system with 64 shards can handle 64 times the write throughput of a single node. Each shard stores 1/64 of the data, so single-shard queries finish 64 times faster — in theory. But the same system cannot efficiently execute a query that needs data from all 64 shards. A simple `JOIN` between orders and customers that works in 2 milliseconds on a monolithic database becomes a 5-second scatter-gather on a sharded database if the tables are not co-located on the same shard.

The tension manifests in every design decision:

- **Single-shard transactions** are fast and ACID. Cross-shard transactions require 2PC or Sagas, adding 10-50ms of latency per transaction.
- **Partition-aligned indexes** are small and efficient. Global indexes require maintaining coordination across all shards.
- **Referential integrity** within a shard is enforced by the database. Referential integrity across shards must be enforced by the application.

Every partitioning decision is an answer to the question: "How much cross-partition query capability am I willing to sacrifice for scalability?"

### Decision 1: Hash Key vs Range Key

**Hash key advantage:** Uniform distribution. No hot spots from key ordering. Every shard receives roughly equal write load. This is the default choice for most systems.

**Range key advantage:** Natural partition pruning for time-range queries. Efficient data retention (drop old partitions). Co-located data within a range — queries for "last 30 days" hit exactly one partition.

**Tradeoff example:** A ride-hailing service partitions by user_id (hash). Queries for "my last 30 rides" are fast. But the business team's query for "total rides in March" must scan all shards. If the service instead partitions by ride_date (range), the March query prunes to one partition, but user-level queries become scatter-gather because any user's rides span multiple months.

**Resolution:** Use composite partitioning. Primary partition by user_id for application queries. Maintain a secondary materialized view partitioned by date for analytics.

### Decision 2: Few Large Shards vs Many Small Shards

**Few large shards (4-16):** Simpler operations. Fewer connections. Lower cross-shard query latency (fewer participants to merge). But each shard is a larger failure domain. Backups are slower. Reshards are more painful.

**Many small shards (64-1024):** Better load distribution. Smaller failure domains. Faster per-shard maintenance. But more connections, larger routing tables, higher chance of any single shard having issues.

**Rule of thumb:** Start with enough shards so that each holds 50-200 GB at peak projected size. For 5 TB: 25-100 shards.

### Decision 3: Application-Level vs DB-Level Partitioning

**Application-level partitioning** means the application code computes the shard and directs the query. The database knows nothing about the partitioning scheme.

**Advantages:** Full control over routing logic. Can implement custom partition algorithms. No database vendor lock-in.

**Disadvantages:** Every application service must implement the same routing logic. Schema changes require updating all services. Harder to add new services that need the same data.

**Database-level partitioning** means the database manages partitions natively. PostgreSQL declarative partitioning, MySQL partitioning, and Citus distributed tables all manage partitioning at the database layer.

**Advantages:** Transparent to applications. Queries automatically prune. Maintenance operations are SQL commands. Schema changes propagate to all partitions.

**Disadvantages:** Limited to the database's built-in strategies. Vendor lock-in. Cross-database queries are not supported.

### Decision 4: Vertical vs Horizontal Partitioning

**Vertical partitioning** splits columns. Suitable for wide tables with rarely accessed large fields. Does not scale write throughput.

**Horizontal partitioning** splits rows. Scales write throughput linearly with shard count. The standard approach for distributed systems.

**Tradeoff:** Vertical partitioning is often combined with horizontal. First split wide columns vertically (hot vs cold schema), then shard both resulting tables horizontally.

### Decision 5: Synchronous vs Asynchronous Rebalancing

**Synchronous rebalancing** blocks reads and writes to a partition while it moves. Simple to implement. Guarantees consistency. Causes downtime.

**Asynchronous rebalancing** moves data in the background while the partition remains available. Writes during migration are dual-written. Consistency is eventually achieved. No downtime. Complex to implement.

**Recommendation:** Always choose asynchronous rebalancing for production systems. The downtime from synchronous rebalancing at scale is measured in hours, not minutes.

### Decision 6: Fixed vs Dynamic Partition Count

**Fixed partitions** are defined at creation. The number never changes. Growth is handled by increasing the resources per partition or adding new partitions by pre-splitting.

**Dynamic partitions** split and merge automatically based on size thresholds. No up-front planning required.

**Recommendation:** Fixed partitions for predictable workloads. Dynamic partitions for workloads with unpredictable growth or spiky traffic patterns.

### Decision 7: Single-Region vs Multi-Region

**Single-region** is simpler. One routing table. No cross-region replication. Lower latency for local users. Higher latency for global users.

**Multi-region** requires geo-partitioning or active-active replication. Higher complexity. GDPR compliance possible. Global latency is uniformly low.

## CAP Positioning

Distributed partitioned systems must be classified along the CAP theorem axis.

**AP Systems (Availability + Partition Tolerance):** Most partitioned databases choose availability over consistency during partition events. Cassandra, DynamoDB (default), and Couchbase fall into this category. They use eventual consistency across replicas. A network partition between shard replicas results in both sides accepting writes. When the partition heals, conflict resolution (LWW or CRDT) reconciles the divergence.

**CP Systems (Consistency + Partition Tolerance):** Some partitioned systems choose consistency over availability. Google Spanner, CockroachDB, and MongoDB (with majority write concern) are CP. **The practical choice:** Most partitioned systems are AP at the shard-to-shard level (shard mapping, routing) but may be CP within a shard (consistent replication within a single shard). This hybrid approach provides the best balance for most workloads.

## Failure Modes

### Hot Spots from Poor Key Choice

The most common failure mode in production. A partition key that seems reasonable at design time becomes a bottleneck as usage patterns evolve.

**Example:** A SaaS platform partitions tenants using `tenant_id % 64`. One tenant grows to 500,000 users while the average is 5,000. That tenant's shard receives 100x the traffic of other shards. The shard's CPU is pinned at 98 percent. Query latency for that tenant degrades from 5ms to 500ms. All users on that tenant — including non-vip users — experience the slowdown simultaneously.

**Detection:** Monitor per-shard query latency and CPU utilization. Set alert thresholds at 2x the average. Use heat maps that show traffic per partition over time.

**Mitigation:** Split the hot partition by sub-sharding the hot key. For tenant-based sharding, distribute the large tenant across multiple sub-keys (tenant_0, tenant_1, ..., tenant_15) and route queries round-robin within the tenant's sub-keys.

### Data Skew

Uneven data distribution across partitions, even with hash partitioning, when the data itself has a skewed distribution.

**Example:** A social media platform partitions posts by `user_id % 256`. The distribution is perfect: 1/256 of users per shard. But 5 percent of users are power users who each post 1,000x more than the average user. Those power users' shards hold 50x more data than the average shard. The shards grow at different rates. Some reach 500 GB while others are at 10 GB.

**Detection:** Track partition size as a percentage of average. Any partition exceeding 150 percent of the average is skewed.

**Mitigation:** Use consistent hashing with virtual nodes — small imbalances are resolved by reassigning virtual nodes. For significant skew, apply range partitioning within each shard based on a secondary dimension.

**Root cause:** The rebalancing process consumes network bandwidth, CPU, and disk I/O on both source and destination nodes. If the system is already operating near capacity, rebalancing pushes it over the edge. The overloaded nodes fail or become unresponsive, triggering more rebalancing, which overloads more nodes.

**Prevention:**
- Never run a cluster above 60 percent capacity. Reserve headroom for rebalancing storms.
- Rate-limit data transfer during rebalancing. Allow only 20 percent of network bandwidth to be consumed by rebalancing traffic.
- Use incremental rebalancing: move data in small batches, verifying health after each batch.
- Implement circuit breakers: if a node's latency exceeds a threshold, pause rebalancing to that node.

### Orphaned Partitions

After a rebalancing or shard split, a partition is left in an inconsistent state. Some data has moved to the new partition, some has not. Queries return partial results or miss data entirely.

**Root cause:** Race condition between the replication stream and the routing table update. A write arrives at the old partition after the routing table has been updated to point to the new partition. The write is applied to the old partition but never replicated to the new one.

**Prevention:** Implement a grace period during which both old and new partitions accept writes. After the grace period, run a reconciliation job that compares checksums on the old and new partitions. Only remove the old partition after reconciliation passes.

### Cross-Shard Transaction Failures

A distributed transaction spanning three shards fails after the first two have committed. The third shard rejects the transaction (key conflict, constraint violation, timeout). The first two shards have already applied changes that cannot be easily undone.

**Detection:** The application receives a transaction error for the third shard. The first two shards show committed changes. A manual reconciliation script must identify and revert the partial commits.

**Prevention:** Use the Saga pattern instead of distributed transactions. Each step in the saga has a compensating action that undoes it. If step 3 fails, the saga executor invokes compensation for step 2 and step 1.

### Zombie Records During Migration

During a live migration between partitioning schemas, some records are written to the new location but also survive in the old location. Subsequent queries may return duplicate results.

**Root cause:** Dual-writes are active during migration. A record is written to both old and new partitions. The migration switchover is supposed to stop writes to the old partition, but a delayed writer (a long-held connection, a queued message) writes to the old partition after the switchover.

**Prevention:** At switchover, set the old partition to read-only at the database level, not just the application level. Run a deduplication job after switchover to identify and remove zombie records.

## Anti-Patterns

### Monolithic Hot Key

**Anti-pattern:** A single partition key value accounts for a disproportionate share of data or traffic. The classic example: partitioning a social media platform by `user_id` without accounting for celebrity users. A user with 50 million followers generates 100x the traffic of an average user. Their single shard is overwhelmed.

**Fix:** Shuffle the celebrity's data across multiple sub-keys. Append a suffix to the user_id: `celebrity_0`, `celebrity_1`, ..., `celebrity_15`. Distribute reads across these sub-keys. The application must know to fan out reads to all sub-keys and merge results.

### Re-Partitioning by Rehashing Live

**Anti-pattern:** Changing the partitioning function from `hash(key) % N` to `hash(key) % M` on a live production system without a migration plan. This invalidates all existing partition assignments. Every key maps to a potentially different partition. All data must be moved. The system is inconsistent during the move.

**Fix:** Use consistent hashing with virtual nodes. Adding or removing nodes only affects immediate neighbors on the ring. Or pre-split partitions at 4x the expected node count and rebalance logical partitions to physical nodes without changing the hash function.

### Monotonically Increasing Keys as Hash Shard Keys

**Anti-pattern:** Using a monotonically increasing value (auto-increment ID, timestamp) as the partition key for hash partitioning. Hash partitioning distributes keys uniformly only if the keys themselves have high cardinality across the range. Auto-increment IDs cluster new data in the same hash bucket: all new inserts go to the same bucket until the hash range boundary is crossed.

**Fix:** Combine the monotonically increasing key with a high-cardinality key. For example, use `HASH(user_id || order_id)` instead of `HASH(order_id)` alone. Or use range partitioning instead of hash when the partition key is time-based.

### Too Many Small Shards

**Anti-pattern:** Creating 1,000 shards for a 50 GB dataset. Each shard holds 50 MB. The overhead of connections, routing metadata, and coordination far exceeds the benefit of partitioning. The system spends more time coordinating than querying.

**Fix:** A partition should be at least 10 GB for transactional workloads and 50 GB for analytical workloads. The sweet spot for most databases is 50-200 GB per partition.

## Cost Implications of Partitioning

Partitioning introduces hidden costs beyond infrastructure. Understanding these costs is essential for making the build-vs-buy decision.

### Infrastructure Cost

Partitioning requires more infrastructure than a monolithic database. Each shard requires its own compute, memory, storage, and networking. With replication factor 3, a 5 TB dataset requires 15 TB of storage and 3x the compute. The cost is linear with shard count.

**Cost comparison for a 5 TB workload:**

| Architecture | Compute Nodes | Storage | Monthly Cost (est.) |
|---|---|---|---|
| Monolithic (vertical scale) | 1 large instance (64 vCPU, 512 GB) | 6 TB SSD | $4,500 |
| Partitioned (8 shards, RF=3) | 24 instances (8 vCPU, 64 GB each) | 15 TB SSD | $9,600 |
| Partitioned (32 shards, RF=3) | 96 instances (4 vCPU, 32 GB each) | 15 TB SSD | $14,400 |

Partitioning costs 2-3x more than a monolithic approach at the same data size. The cost is justified by the throughput and scalability benefits that a single node cannot deliver.

### Operational Cost

The operational cost of partitioning is harder to quantify but more significant:

- **Engineering time.** Designing, implementing, and testing the partitioning scheme takes 4-8 weeks for an experienced team.
- **Monitoring complexity.** Per-shard monitoring requires more dashboards, more alert rules, and more on-call context.
- **Migration cost.** Resharding is a major engineering project. Each resharding event costs 2-4 weeks of engineering time plus migration infrastructure.
- **Debugging difficulty.** A slow query on a monolithic database is straightforward to debug. A slow query on a partitioned system requires identifying which shard is slow, understanding that shard's load, and determining whether the issue is local to the shard or systemic.

### When the Cost Is Worth It

- Write throughput exceeds 10,000 writes/second (the practical limit of a single PostgreSQL instance).
- Data size exceeds 5 TB (the practical limit for online DDL operations).
- The system requires 99.99 percent availability that a single node cannot provide (maintenance windows are unacceptable).
- The workload is geographically distributed and requires data locality for latency or compliance.

When none of these conditions are met, the cost of partitioning exceeds the benefit.

## Monitoring Strategies for Partition Health

Partition imbalance develops gradually. A system that is perfectly balanced on day one can be 10x skewed by month six as access patterns shift. Continuous monitoring with the right signals catches imbalance before it causes production issues.

### Key Signals

**Partition size ratio.** Track the ratio of the largest partition to the smallest partition. A healthy system stays under 1.5x. When this ratio exceeds 3x, rebalancing should be triggered.

**Throughput coefficient of variation (CV).** Track the CV of write throughput across partitions. CV = standard deviation / mean. A CV below 0.2 indicates balanced load. CV above 0.5 indicates significant skew — some partitions handle much more traffic than others.

**Logistic growth rate per partition.** Track the weekly growth rate of each partition. A partition growing at 10 percent per week while the cluster average is 2 percent per week is a leading indicator of an emerging hot spot. It will reach imbalance in 10-15 weeks if unchecked.

### Automated Remediation

For systems with automated rebalancing (Cassandra, Elasticsearch, MongoDB), set the rebalancing thresholds conservatively:

- Cassandra: `nodetool cleanup` after node addition. Set `stream_throughput_outbound_megabits_per_sec` to 200 Mbps to avoid saturating network links during rebalance.
- Elasticsearch: Set `cluster.routing.allocation.balance.shard` to 0.45 (default 0.55) to trigger rebalancing earlier. Set `cluster.routing.allocation.cluster_concurrent_rebalance` to 2 to avoid overwhelming the cluster.
- MongoDB: Set `balancerWindow` to off-peak hours. Set `chunkSize` to 64 MB (default) for balanced chunk distribution.

For systems without automated rebalancing (Vitess, custom sharding), build a monitoring pipeline that triggers alerts rather than automated remediation. Automated shard splitting is risky without the safety guarantees that purpose-built systems provide.

## When NOT to Use Partitioning

Partitioning is not always the right answer. In these scenarios, a monolithic database outperforms a partitioned one.

**Small datasets under 100 GB.** A single PostgreSQL instance with proper indexing handles 100 GB efficiently. Adding partitioning adds complexity without measurable benefit. The operational overhead of managing 10 ten-GB shards is higher than managing one 100 GB database.

**Workloads with frequent cross-shard queries.** If 30 percent or more of all queries do not include the partition key, partitioning will slow down the system rather than speed it up. Partitioning optimizes for partition-local queries. Global queries become scatter-gather, which is slower than scanning a single table of equivalent total size.

**Systems that need ACID across all data.** If the application requires transactions across arbitrary rows in the same table, partitioning breaks ACID guarantees. A transaction that updates user 42's balance and reads user 99's balance spans at least two shards. Distributed transactions (2PC) are slow, fragile, and not supported by all databases.

**Rapidly changing schemas.** If the schema changes weekly (new columns, modified data types, new indexes), managing those changes across 64 shards is painful. Each schema change must be rolled out to every shard. Partitioned schema migrations require careful sequencing to avoid locks and data inconsistencies.

**Low throughput requirements.** A system handling 1,000 writes per second and 5,000 reads per second on a single PostgreSQL instance (tuning for 10,000 IOPS SSD) does not need partitioning. Partitioning adds operational complexity for zero throughput gain.

**Startup or MVP phase.** A product with 1,000 users and uncertain growth should not start with partitioning. Build the monolithic version first. If the product grows to 100,000 users, then evaluate whether partitioning is needed. Premature partitioning slows development velocity — every schema change requires updating routing logic, and debugging queries requires understanding the partition topology. The opportunity cost of premature partitioning is higher than the migration cost from monolithic to partitioned later.

**Compliance requiring single-region data storage with audit trails.** Some compliance frameworks (SOC 2 Type II, PCI DSS) require data to remain in a single audited environment. Partitioning across multiple regions or clouds creates additional audit scope. Each shard must meet the same compliance requirements, multiplying the audit burden. A single large database in a single region is simpler to audit and certify.

**Legacy systems with tightly coupled data.** Applications designed with cross-table foreign keys and distributed transactions across most operations are hard to partition. Every foreign key that crosses partition boundaries must be replaced with application-level referential integrity. Every cross-table join becomes a scatter-gather or requires data duplication. The migration effort may exceed the benefit, especially for applications planned for retirement within 1-2 years.

**Teams without dedicated database operations support.** Partitioning requires ongoing operational expertise: monitoring per-shard health metrics, managing partition splits and merges, troubleshooting cross-shard query performance, and maintaining the routing layer. If the team does not have a DBA or SRE with partitioned-database experience, the operational risk outweighs the scalability benefit. Consider a managed database service (Aurora, Cloud SQL, Spanner) that handles scaling transparently.

**Analytical systems scanning most partitions anyway.** If the workload is predominantly full-table scans for analytics, partitioning does not help. The query touches every partition regardless of partition pruning. Columnar storage formats (Parquet, ORC) and data skipping (min/max statistics per file) provide better optimization for analytical workloads.

## Partitioning Strategy Selection: Scenario-Based Guide

The following scenarios map common real-world situations to the best partitioning approach. Each scenario includes the dominant query pattern and recommended strategy.

**Scenario 1: User profile data for a social network.** Queries: "get profile for user X" (99 percent). Team size: 5 engineers. Data: 1 TB. Growth: 50 percent per year. Partitioning recommendation: hash by user_id across 8 shards with consistent hashing. The routing layer is a simple library in the application code (no proxy). The team is small and the access pattern is simple — do not over-engineer with Vitess or a dedicated proxy layer until the team grows.

**Scenario 2: IoT sensor readings at 10M events/second.** Queries: "get readings for sensor X over the last hour" (80 percent), "get average reading across all sensors in region Y" (20 percent). Team size: 10 engineers. Data: 10 TB/day. Partitioning recommendation: range by timestamp (hourly), secondary hash by sensor_id. Use ClickHouse with its built-in partition+order scheme. The time-range queries prune to the relevant hourly partition. The cross-sensor queries scan one hour's partition but benefit from ClickHouse's columnar vectorized execution.

**Scenario 3: Multi-tenant SaaS platform with 1,000 tenants.** Queries: "get data for tenant X" (95 percent). Tenants range from 100 users (small) to 100,000 users (large). Partitioning recommendation: list by tenant_id, but with a tenant capacity tier. Create 4 tiers: small (T1: under 1K users), medium (T2: 1K-10K), large (T3: 10K-50K), and enterprise (T4: 50K+). Each tier has a different number of shards per tenant. T1 tenants share one shard (100 tenants per shard). T2 tenants get a dedicated shard. T3 tenants get 4 shards (hash within tenant). T4 tenants get a dedicated cluster. This tiered approach avoids the problem of one large tenant overwhelming a shard shared by many small tenants.

**Scenario 4: Time-series monitoring with sub-second query requirements.** Queries: "show CPU for server X in the last 5 seconds" (99.9 percent). Data: 1M data points/sec. Partitioning recommendation: range by timestamp (seconds). Each second is a partition. Use a ring buffer of the last 3600 partitions (1 hour). Older partitions are automatically dropped. The "last 5 seconds" query reads exactly 5 partitions. Use RedisTimeSeries or TimescaleDB's hypertables for native time-series partitioning with automatic retention.

**Scenario 5: Global e-commerce catalog with inventory.** Queries: "show product X details" (60 percent), "search by category" (30 percent), "check inventory for order" (10 percent). Partitioning recommendation: composite. First, vertically split the product table: hot (product_id, name, price, stock, category_id) and cold (product_id, description, specs, images). Second, hash the hot table by product_id across 32 shards for product detail queries. Third, maintain a separate Elasticsearch index for category search. The inventory check goes to the hot table — it filters by product_id (single shard) and checks stock in under 5ms.

## Migration Strategy: Moving from Monolithic to Partitioned

Migrating from a monolithic database to a partitioned system is one of the riskiest operations in data engineering. A structured approach reduces risk.

### Phase 1: Audit and Plan (2-4 weeks)

**Analyze query logs** to identify the dominant access pattern. Run `pg_stat_statements` (PostgreSQL) or `performance_schema` (MySQL) to capture the top 50 queries by frequency. For each query, identify which column appears in the WHERE clause. The column that appears most frequently is the partition key candidate.

**Measure data distribution.** For the candidate partition key, run `SELECT key, count(*) FROM table GROUP BY key ORDER BY count(*) DESC LIMIT 100`. Check for skew: are the top 10 keys accounting for more than 20 percent of rows? If so, the key is a poor partition candidate and a composite key or sub-sharding strategy is needed.

**Benchmark query performance.** For the top 10 queries, measure: average latency, max latency, rows scanned, and whether index usage is optimal. These benchmarks become the baseline for measuring improvement after partitioning.

### Phase 2: Build the Partitioned System (4-8 weeks)

**Set up the routing layer.** Deploy the proxy (Vitess, ProxySQL, or a custom router). Configure the routing table with the initial partition mapping. Load test the proxy to ensure it can handle peak throughput without becoming a bottleneck.

**Create the partitioned schema.** Create the partitioned table on each shard. For a new system being migrated from monolithic, this means creating the table structure on each shard without data.

**Implement dual-writes.** Configure the application to write each row to both the monolithic database and the partitioned system. Initially, reads come from the monolithic database only. This ensures that the partitioned system has current data when the cutover happens. Monitor write latency and error rates for both paths.

### Phase 3: Backfill and Verify (2-4 weeks)

**Backfill historical data.** Use a parallel export from the monolithic database and import into the partitioned system. Run 8-16 parallel worker threads, each responsible for a range of the partition key. For 5 TB of data on 10 Gbps networking: approximately 4-8 hours for the initial backfill.

**Verify consistency.** After backfill, compare row counts between the monolithic database and each shard. Run checksum comparisons on a 1 percent sample of rows. If mismatches are found, identify the root cause (race condition with dual-writes, missing rows during backfill, data type conversion issues) and re-run the affected range.

### Phase 4: Cutover (1-2 hours)

**Stop writes to the monolithic database.** Set the monolithic database to read-only. The application's dual-write path now fails on the monolithic side. This is the moment of truth — the partitioned system must handle all writes correctly.

**Run final verification.** Compare the last few minutes of writes between the monolithic database and each shard. If all rows match, proceed.

**Switch reads to the partitioned system.** Update the routing configuration. The application now reads from the partitioned system. Monitor query latency and error rates closely for the first hour.

**Monitor.** Watch per-shard latency, error rates, and replication lag for 24 hours. Keep the monolithic database in read-only mode as a rollback option.

### Phase 5: Decommission (1-2 weeks)

**Remove dual-write code** from the application. The monolithic database is no longer receiving writes.

**Archive the monolithic database** to cold storage. Keep it available for 30 days as a safety net.

**Remove the monolithic database** from production infrastructure after confirming no rollback is needed.

**Total migration timeline:** 8-16 weeks for a system of 5 TB with 100 shards. Rushing the migration increases the risk of data loss or extended downtime.

## Disaster Recovery for Partitioned Systems

A partitioned system introduces unique disaster recovery challenges. Each shard is an independent database, which means the recovery strategy must handle both single-shard failures and full-cluster failures.

### Recovery Point Objective (RPO) Per Shard

RPO measures how much data loss is acceptable in a failure. In a partitioned system, RPO can differ per shard based on the shard's data criticality.

**Hot shards** (current month's data, active users): RPO of 0 seconds. Use synchronous replication. Every write must be acknowledged by at least one replica before the primary acknowledges the client. This guarantees no data loss if the primary fails. The tradeoff is higher write latency (the primary waits for the replica's acknowledgment).

**Warm shards** (last 3 months of data, recently active users): RPO of 5-60 seconds. Use asynchronous replication with a short lag target. The primary acknowledges writes immediately without waiting for replicas. If the primary fails, up to 5-60 seconds of writes may be lost. Acceptable for most transactional workloads.

**Cold shards** (archived data, older than 6 months): RPO of 24 hours. Use daily snapshots to S3 or Glacier. The primary can be a single node with no replicas. If the primary fails, restore from the latest snapshot. Data loss is limited to the last 24 hours of writes to the cold shard.

### Recovery Time Objective (RTO) Per Shard

RTO measures how long it takes to restore service after a failure. Partitioned systems can achieve lower RTO by parallelizing recovery.

**Single shard failure:** RTO of 5-15 minutes. The replica is promoted to primary (30 seconds). The routing table is updated (5 seconds). The new replica is provisioned from the promoted primary (10-15 minutes for a 200 GB shard). Total: 10-15 minutes.

**Multiple shard failure (rack or AZ failure):** RTO of 15-30 minutes. All replicas in the affected AZ are promoted simultaneously. The routing table is updated for all affected shards. New replicas are provisioned for all affected shards in parallel. Total: 15-30 minutes.

**Full cluster failure (region failure):** RTO of 1-4 hours. Restore all shards from the latest backups in a different region. Restore the routing table from the backup catalog. Verify consistency across all shards. Start traffic. Total: 1-4 hours depending on backup size and restore parallelism.

### Disaster Recovery Drill Protocol

Run this drill quarterly to validate RPO and RTO assumptions:

1. Select one shard at random. Record the current time and the last confirmed write.
2. Kill the shard's primary process. Do not tell the on-call engineer in advance.
3. Measure the time until the replica is promoted and the routing table is updated.
4. Measure the time until the shard is fully recovered (new replica provisioned).
5. After recovery, compare the last confirmed write before the kill with the data on the recovered shard. Calculate actual data loss (RPO achieved).
6. Document: actual RTO vs target RTO, actual RPO vs target RPO, and any issues encountered during the drill.
7. Fix any issues found. Repeat the drill next quarter.
