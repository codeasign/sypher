The previous section covered real-world implementations. This section is your interview playbook for system design interviews focused on AI agent architecture at FAANG and similar companies. You will learn how this topic appears, what interviewers test at each level, a structured framework for answering, a phrase bank, red flags to avoid, and practice drills.

## How This Topic Appears

AI agent architecture questions appear in system design interviews in several forms. Here are the most common phrasings, organized by the type of question.

**Design a customer support agent:**
1. "Design a system where users can ask questions about their account and an AI agent resolves their issues."
2. "Design an AI-powered customer support system that can handle billing, technical support, and account management."
3. "How would you build a system where an AI agent can access multiple internal tools to help customers?"

**Design a code generation agent:**
4. "Design a system where an AI agent can generate code, run it, and fix errors automatically."
5. "How would you build an AI-powered software development assistant that can write, test, and debug code?"

**Design a research agent:**
6. "Design a system where an AI agent researches a topic by searching multiple sources, synthesizing findings, and producing a report."
7. "How would you build a system that can answer complex questions by retrieving and analyzing information from multiple data sources?"

**Design a multi-agent system:**
8. "Design a system where multiple AI agents collaborate to solve a complex problem."
9. "How would you design a system with specialized agents that communicate and coordinate with each other?"

**Design an agent platform:**
10. "Design a platform that lets non-technical users create and deploy AI agents that interact with their company's tools and data."
11. "How would you build a general-purpose agent framework that can be customized for different business domains?"

## Question Difficulty Mapping

Not all agent architecture questions test the same skills. Interviewers calibrate the difficulty of the question to the level they are hiring for. Here is how each question type maps to typical difficulty levels and what the interviewer expects at each level.

**Customer Support Agent (L3-L5):**
This is the most common entry-level agent question. At L3, you describe the ReAct loop and name the components. At L4, you add memory management, error handling, and tool schema validation. At L5, you add capacity planning, cost estimation, and trade-off analysis. The interviewer expects you to recognize that this is a straightforward single-agent pattern and not over-engineer it.

**Code Generation Agent (L4-L6):**
This question tests your understanding of the feedback loop between generation and execution. At L4, you describe the plan-generate-execute cycle. At L5, you add sandbox security, self-healing logic, and multi-file state management. At L6, you discuss the architecture of a code generation platform that supports multiple languages, frameworks, and project types. The interviewer expects you to understand that code generation is qualitatively different from text generation because the output must be executable.

**Research Agent (L4-L6):**
This question tests your understanding of the ReAct loop with external search. At L4, you describe the search-read-synthesize loop. At L5, you add source credibility scoring, citation management, and loop safety. At L6, you design a multi-source research system with cross-referencing, contradiction detection, and iterative refinement. The interviewer expects you to recognize that the hard problem is not the search itself but the synthesis of potentially contradictory information.

**Multi-Agent System (L5-L7):**
This question is reserved for senior and staff levels. At L5, you identify when a single agent is insufficient and describe the supervisor pattern. At L6, you compare multiple coordination patterns including supervisor, round-robin, broadcast, and auction, and justify your choice. At L7, you design the communication protocol, shared memory model, and conflict resolution mechanism. The interviewer expects you to understand that multi-agent systems are a last resort, not a default choice.

**Agent Platform (L6-L8):**
This is the hardest question type and is used for staff, principal, and distinguished level interviews. At L6, you design the platform architecture with tool registry, memory management, and monitoring. At L7, you add multi-tenancy, policy enforcement, cost governance, and organizational adoption strategy. At L8, you design an ecosystem where agents from different teams can discover and use each other's capabilities safely. The interviewer expects you to think about the platform as a product, not just infrastructure.

**Financial Trading Agent (L6-L8):**
This question is a variant that tests risk-awareness and regulatory compliance. At L6, you design the agent with explicit guardrails and kill switches. At L7, you add policy engine, audit trail, and compliance reporting. At L8, you design the system to handle market-wide events, cascading failures, and regulatory inquiries. The interviewer expects you to understand that in regulated domains, safety is the primary constraint, not performance.

**Key insight:** Interviewers rarely ask an L3 candidate to design a multi-agent system, and they rarely ask an L7 candidate to design a customer support agent. If you are interviewing at a specific level, focus your preparation on the question types that match your target level. If the interviewer asks a question that seems below your level, they are testing whether you can add depth and sophistication to a simple problem.

## What the Interviewer Is Actually Testing

Each question type tests different aspects of your system design knowledge.

**For "design a customer support agent" questions:**
- Can you identify the key components of an agent system (orchestrator, planner, executor, tools, memory)?
- Do you understand how tool selection and tool calling work?
- Can you handle the trade-off between autonomy (agent decides what to do) and safety (agent must not do harmful things)?
- Do you consider error handling, retries, and fallback paths?

**For "design a code generation agent" questions:**
- Do you understand the plan-generate-execute loop?
- Can you design a self-healing system that detects and fixes errors?
- Do you consider the sandbox security implications of code execution?
- Can you handle the state management problem of multi-file code generation?

**For "design a research agent" questions:**
- Do you understand the ReAct loop and when to use it?
- Can you design a memory system that manages context across multiple search iterations?
- Do you consider source credibility, citation, and hallucination mitigation?
- Can you handle the "agent stuck in a loop" problem?

**For "design a multi-agent system" questions:**
- Can you identify when a single agent is insufficient and multiple agents are needed?
- Do you understand the supervisor pattern and its alternatives?
- Can you design agent communication protocols, shared memory, and conflict resolution?
- Do you consider the cost and latency implications of multi-agent coordination?

**For "design an agent platform" questions:**
- Can you design a system that is both flexible (supports many use cases) and safe (prevents misuse)?
- Do you understand how to make agent capabilities configurable without code changes?
- Can you design a tool registry, plugin system, or extension mechanism?
- Do you consider the multi-tenancy, isolation, and security implications?

## Level-by-Level Expectations

### L3 / L4 (Entry to Mid-Level)

**Expectation:** You can identify the components of an agent system and explain how they work together. You understand the basic agent loop. You can sketch a simple architecture. At this level, the interviewer is checking that you have hands-on experience with agent systems and are not just repeating buzzwords.

**What you should demonstrate:**
- You can name the components of an agent system (orchestrator, planner, executor, tools, memory)
- You can explain the ReAct loop in your own words
- You can sketch a simple architecture diagram
- You understand the difference between a single LLM call and an agent loop
- You can identify the tools the agent needs for a specific use case

**What you should avoid:**
- Do not describe a single LLM call as an agent — the interviewer will immediately know you lack hands-on experience
- Do not hand-wave the agent loop — explain how it actually works
- Do not forget to mention memory management — this is a common L4 blind spot

**Model answer for "Design a customer support agent" (L4):**

"I would design a single-agent ReAct system. The core components are: an orchestrator that receives the user request and manages the conversation, an LLM that reasons about the request and decides what to do, a set of tools that the agent can call (look up account, search knowledge base, create ticket), and a memory store that maintains conversation history.

The flow is: user request arrives, the orchestrator wraps it in a system prompt with tool definitions, sends it to the LLM, the LLM either returns a tool call or a final response, if it is a tool call, the orchestrator executes it and feeds the result back to the LLM, this repeats until the LLM returns a final response to the user.

For the customer support use case, the tools would be: get_account_by_email, search_knowledge_base, create_support_ticket, and escalate_to_human. The system prompt would instruct the agent to always verify the user's identity before performing account changes, and to escalate to a human agent if the issue cannot be resolved in 5 iterations."

**What the interviewer looks for:** Clear understanding of the agent loop, correct identification of components, awareness of tool calling mechanics, basic error handling.

### L5 (Senior)

**Expectation:** You can design a production-grade system with multiple components, handle failure modes, and make trade-off decisions. You understand the cost and performance implications of your design. At L5, the interviewer expects you to have built and operated agent systems in production, not just prototyped them.

**What you should demonstrate:**
- Capacity planning: estimate QPS, tokens, cost, and latency for the proposed system
- Trade-off analysis: explain why you chose one pattern over another
- Failure mode awareness: discuss tool failures, infinite loops, context overflow, and their mitigations
- Production considerations: monitoring, deployment, A/B testing, rollback
- Decoupling: designing components that can scale independently

**What you should avoid:**
- Do not propose a system that cannot scale — every component must be horizontally scalable
- Do not ignore cost — show that you know what LLM inference costs and how to estimate it
- Do not forget observability — agent behavior is non-deterministic and you must be able to see what happened
- Do not propose a monolithic architecture — the interviewer expects a decoupled, service-oriented design

**Model answer for "Design a customer support agent" (L5):**

"I would design an asynchronous agent system with a decoupled orchestrator, planner, and executor. The architecture uses a task queue for resilience and horizontal scaling, a memory store for session persistence, and a tool service mesh for independent tool scaling.

The request flow is: user sends a message to the API gateway, which authenticates and creates a session. The orchestrator classifies the request and publishes a planning task to the queue. The planner service picks up the task, generates a plan (a DAG of steps), and publishes execution tasks. The executor worker pool processes each task by calling the appropriate tool. Results are stored in the memory store. When all tasks complete, the orchestrator synthesizes the final response.

For 10 million DAU, I estimate 300 requests per second at peak. Each request averages 8 agent iterations, with 4,000 input tokens and 200 output tokens per iteration. The LLM cost is approximately $0.35 per request, or $175,000 per day. The tool execution requires 30 instances per tool type for 2x headroom. Memory store needs 180 MB of active cache and 50 GB of daily storage.

The key trade-off is autonomy versus safety. For a customer support agent, I prioritize safety. I would use explicit tool routing with human-in-the-loop for account changes, implement strict tool schema validation, and set a maximum of 10 iterations with automatic escalation to human support."

**What the interviewer looks for:** Capacity planning, cost estimation, trade-off analysis, failure mode awareness, production-grade design thinking.

### L6 (Staff)

**Expectation:** You can design a multi-agent system or a platform that other teams use. You identify the hard problems that others miss. You understand the business and organizational implications of your design. At L6, the interviewer expects you to think beyond the individual agent and consider the system-of-systems implications.

**What you should demonstrate:**
- Platform thinking: designing for reuse, extensibility, and multi-tenancy
- Hard problem identification: naming the problems that others miss (cross-tool policy enforcement, isolation between tenants, cost allocation)
- Organizational awareness: understanding how the platform affects team structure, development velocity, and operational burden
- Trade-off articulation: explaining why you chose a specific multi-agent pattern over alternatives
- Risk mitigation: designing safety mechanisms that scale across thousands of agents

**What you should avoid:**
- Do not design for a single use case — at L6, you are designing for multiple current and future use cases
- Do not ignore the human element — how will teams adopt the platform? How will they debug agent failures?
- Do not propose a solution that requires heroic effort to maintain — the platform must be self-service
- Do not forget about governance — who decides what agents are allowed to do?

**Model answer for "Design a multi-agent platform" (L6):**

"I would design a platform that lets teams define, deploy, and monitor AI agents without writing agent infrastructure code. The platform has three layers: the agent runtime, the capability layer, and the management layer.

The agent runtime is a generic agent loop that supports multiple patterns (ReAct, Plan-and-Execute, Supervisor). The loop is controlled by a configuration that specifies the pattern, the LLM model, the iteration limit, and the error handling policy.

The capability layer is a tool registry and a knowledge base connector. Teams register tools as OpenAPI specifications with authentication, rate limits, and safety annotations. The knowledge base connector supports multiple vector stores and provides a unified retrieval interface.

The management layer provides monitoring, cost tracking, and A/B testing. Every agent run produces a trace that includes the full thought process, tool calls, and observations. Teams can compare agent versions side by side on success rate, cost, and latency.

The hard problems are: isolation between agents from different teams (each agent needs its own namespace for tools and state), safety across tool calls (a tool registered for one use case should not be callable by an agent in a different use case), and cost allocation (each agent run must be attributable to a team and a use case for billing).

The organizational implication is that the agent platform team should be separate from the teams that build agents on top of the platform. The platform team owns the runtime, the tool registry, and the management layer. The business teams own the agent configuration, the tool implementations, and the prompt templates."

**What the interviewer looks for:** Platform thinking, multi-tenancy, cross-cutting concerns, organizational awareness, identification of truly hard problems.

### L7 / L8 (Principal / Distinguished)

**Expectation:** You can design a system that changes the way the company operates. You identify the strategic implications of your design. You connect technical decisions to business outcomes. At L7/L8, the interviewer is evaluating whether you can shape the company's technical strategy around AI agents.

**What you should demonstrate:**
- Strategic vision: the agent platform is not just infrastructure — it is a competitive advantage that changes how the company operates
- Business metrics: connect technical decisions to revenue, cost, customer satisfaction, and developer productivity
- Ecosystem thinking: how does the agent platform interact with the company's existing systems, data, and workflows?
- Risk-reward analysis: what is the cost of not building this capability? What is the risk of building it wrong?
- Leadership: how will you convince teams to adopt the platform? How will you measure success?

**What you should avoid:**
- Do not focus on implementation details — at this level, the strategic impact matters more than the API design
- Do not ignore the competitive landscape — how does this compare to what competitors are doing?
- Do not propose a solution that requires buying a vendor product — at this level, you are building differentiating technology
- Do not forget that you are influencing, not commanding — how will you get buy-in from other teams and leaders?

**Model answer for "Design a company-wide agent platform" (L7):**

"I would design an agent platform that is a first-class product within the company, not an infrastructure project. The platform must solve three problems: capability discoverability, safety at scale, and cost governance.

Capability discoverability means that any team can register a tool, and any agent can discover and use that tool — with appropriate permissions. The platform maintains a service catalog of all registered tools, their capabilities, their rate limits, and their cost. Agents use the catalog at runtime to select the best tool for each task.

Safety at scale means that the platform must prevent the agent from doing harm even when the agent is given broad capabilities. The platform implements a policy engine that evaluates each tool call against a set of policies. Policies are defined at the tool level, the agent level, the user level, and the organizational level. The policy engine runs before each tool call and can block, modify, or approve the call.

Cost governance means that the platform provides visibility into the cost of every agent run, every tool call, and every LLM token. The platform enforces cost budgets at the agent level, the team level, and the organizational level. When a budget is exceeded, the platform can throttle, alert, or stop agent runs.

The strategic implication is that the agent platform becomes a competitive advantage. The speed at which teams can build and deploy agents directly correlates to the company's ability to serve customers, operate efficiently, and innovate. The platform should be designed so that a team can go from idea to production agent in under one week.

The KPIs for the platform are: time to first agent (target: 1 week), agent success rate (target: 95%), cost per agent run (target: under $0.10), and agent adoption rate (target: 80% of eligible teams within 6 months)."

**What the interviewer looks for:** Strategic thinking, business awareness, policy and governance design, ability to define and measure success, organizational impact analysis.

## The FAANG Framework for Answering

Use this 6-step framework to structure your answer to any AI agent architecture question.

### Step 1: Clarify Scope and Requirements (2 minutes)

Before designing, clarify what you are building. Ask questions to narrow the scope.

**Phrases:**
- "Let me make sure I understand the scope. Are we designing a single-purpose agent, or a platform that can support multiple types of agents?"
- "What are the non-functional requirements? Latency? Throughput? Cost constraints?"
- "What tools or data sources does the agent need to access?"
- "What is the risk tolerance? Can the agent make mistakes, or must every action be verified?"
- "How many users will this serve? What is the expected scale?"

### Step 2: Identify the Agent Pattern (2 minutes)

Choose the right agent pattern for the use case. Justify your choice.

**Phrases:**
- "For this use case, I recommend a ReAct agent because the next step depends on the result of the previous step."
- "I would use a Plan-and-Execute pattern here because the steps are known in advance and the dependencies are clear."
- "This requires a multi-agent supervisor pattern because the system needs expertise in multiple domains."
- "A single tool-use agent is sufficient here because the number of tools is small and the reasoning is simple."

### Step 3: Sketch the High-Level Architecture (5 minutes)

Draw the major components and data flows. Label each component.

**Phrases:**
- "The architecture has five layers: the ingress layer, the orchestrator, the agent runtime, the tool layer, and the persistence layer."
- "The orchestrator receives the request, classifies it, and initializes the agent loop."
- "The agent runtime runs the ReAct loop: think, act, observe, repeat."
- "Tools are registered in a tool registry and called through a tool executor that handles retries, timeouts, and error mapping."
- "The memory store maintains conversation history, agent state, and tool results."

### Step 4: Deep Dive into Key Components (5 minutes)

Pick 2-3 components and explain them in detail. Show depth.

**Phrases:**
- "Let me go deeper into the tool calling mechanism. When the LLM returns a tool call, the executor validates the schema, checks permissions, executes the tool with a timeout, and captures the result. If the tool fails, the executor retries up to 3 times with exponential backoff before returning an error to the agent loop."
- "The memory management is critical. I use a sliding window of 10 recent messages plus a compressed summary of the conversation history. When the token count exceeds 80% of the context window, the system summarizes the oldest messages and replaces them with a summary token."
- "The loop safety mechanism detects infinite loops by checking for repeated tool calls within a sliding window of 5 iterations. If the same tool call appears 3 times with the same parameters, the agent is forced to try a different approach or stop."

### Step 5: Address Failure Modes and Trade-offs (3 minutes)

Show that you understand the risks and have mitigation strategies.

**Phrases:**
- "The main trade-off is autonomy versus safety. I chose a ReAct pattern with tool routing validation to balance both. The agent can make decisions autonomously, but every tool call is validated against the tool schema and checked for safety policies."
- "The cascading failure risk is handled by the tool service mesh. Each tool runs independently with its own circuit breaker. If one tool fails, the agent can continue with the remaining tools and produce a partial response."
- "Context overflow is mitigated by the memory management system. I use a dual strategy: a sliding window for recent context and a summarization step that compresses old context when the token count exceeds the threshold."

### Step 6: Summarize and Wrap Up (1 minute)

Recap the key decisions and why you made them.

**Phrases:**
- "To summarize, I designed a multi-agent supervisor system with a central orchestrator, specialized agents for billing and technical support, a shared memory store, and a human-in-the-loop escalation path for high-risk actions. The system prioritizes safety over autonomy while maintaining the flexibility to handle diverse customer issues."
- "The key design decisions are: ReAct pattern for the agent loop, decoupled orchestrator and executor for independent scaling, sliding window memory management with summarization, and a policy engine for tool call safety."

## Phrase Bank

Use these phrases to sound authoritative and structured in your interview.

**For framing the problem:**
- "Before I design the architecture, let me clarify the scope and requirements."
- "Let me identify the key non-functional requirements that will drive the architecture."
- "The first decision is which agent pattern fits this use case."

**For describing the architecture:**
- "The system is organized into five layers: ingress, orchestration, agent runtime, tools, and persistence."
- "The orchestrator is the central coordinator that manages the agent lifecycle."
- "The agent loop runs a ReAct cycle: receive context, reason, select action, execute, observe, and repeat."
- "Tools are registered in a tool registry and called through a stateless executor service."

**For discussing trade-offs:**
- "The fundamental tension is between autonomy and control. More autonomy means more capability but less predictability."
- "I chose asynchronous processing with a task queue to decouple request ingestion from agent execution."
- "The trade-off for memory management is between context completeness and token cost."
- "Parallel tool calls improve latency but introduce race conditions. I use parallel execution for independent read-only tools and sequential execution for tools that modify state."

**For addressing failure modes:**
- "Tool failures are handled by the executor with retry logic and exponential backoff."
- "Infinite loops are detected by a pattern matcher that checks for repeated tool call sequences."
- "Context overflow is prevented by proactive memory management with summarization."
- "The system degrades gracefully by skipping non-critical tools and producing partial responses."

**For framing the problem:**
- "Let me outline the scope by asking three questions: who is the user, what tools do they need, and what is the risk tolerance?"
- "The first step is to define the boundary of the agent system: what decisions does the agent make autonomously, and what requires human approval?"
- "I need to understand the expected scale because it will drive the architectural decisions around synchronization, state management, and cost optimization."

**For describing the architecture:**
- "The orchestrator manages the agent lifecycle: initialization, iteration, and termination. It is the only component that holds the full conversation state."
- "The tool executor is a stateless service that maps tool names to tool implementations, validates parameters, enforces rate limits, and captures results."
- "The memory manager handles three types of memory: working memory for the current conversation, episodic memory for past interactions, and semantic memory for learned knowledge."
- "The policy engine sits between the agent and the tools, intercepting every tool call for permission checking, rate limit enforcement, and safety validation."
- "Each agent instance runs in its own container with a dedicated context window, preventing interference between concurrent sessions."

**For discussing trade-offs:**
- "Synchronous processing gives lower latency but lower throughput. Asynchronous processing with a queue gives higher throughput but adds latency. For interactive agents, I use a hybrid approach."
- "The trade-off between prompt detail and token cost is a continuous optimization problem. Detailed prompts improve accuracy but increase cost and latency. I use dynamic prompt compression based on the complexity of the request."
- "Caching tool results improves latency and reduces cost but risks stale data. I use time-based cache invalidation with a TTL that depends on the tool type: short TTL for fast-changing data, longer TTL for static data."
- "Specialized small models for specific tasks are cheaper and faster than one large model for everything. The trade-off is the complexity of routing between models and maintaining multiple model deployments."

**For addressing failure modes:**
- "The circuit breaker pattern prevents cascading failures. If a tool fails three times in rapid succession, the circuit breaker opens and the agent skips that tool for a cooldown period."
- "Hallucination detection is done by cross-referencing tool results with the agent's reasoning. If the LLM claims a fact that contradicts the tool result, the agent is forced to re-evaluate."
- "The dead letter queue captures agent runs that exceed the maximum iteration limit. Each dead letter is reviewed manually to identify systemic issues in the agent design."
- "Graceful degradation means the agent can produce a useful response even when some tools are unavailable. The system prompt includes fallback instructions for each tool."

**For discussing scale:**
- "At 300 requests per second peak, the system requires approximately 30 agent worker instances."
- "The memory store needs 180 MB of active cache and 50 GB of daily storage for session data."
- "The LLM cost is the dominant cost factor at approximately $0.35 per request."
- "The tool service mesh scales independently based on the capacity needs of each tool."
- "The orchestrator is stateless and horizontally scalable. The bottleneck is the LLM API, not the orchestration layer."
- "Tool execution is the most latency-sensitive path. Each tool call adds 100 to 500 milliseconds. The total agent time is the sum of LLM inference time plus tool execution time plus iteration overhead."
- "Database sharding is based on session ID. Each agent session maps to exactly one shard, ensuring consistent read-after-write semantics for conversation history."
- "The maximum throughput of the system is limited by the LLM API rate limit. I use a token bucket algorithm on the orchestrator to stay within the rate limit while maximizing throughput."

## Red Flags

These are specific behaviors that signal to an interviewer that you do not understand agent architecture.

**Red Flag 1: You call the LLM once and return the response.** This shows you do not understand the iterative nature of agents. The entire point of an agent is the loop. If you are not describing a loop, you are describing a chatbot, not an agent.

**Red Flag 2: You ignore tool failures.** If your design assumes every tool call succeeds, you have not thought about production reality. Every tool call can fail. Every tool call can time out. Every tool call can return garbage data. You must handle all of these.

**Red Flag 3: You have no memory management.** If your design stores all conversation history forever and never manages the context window, your agent will break on the third or fourth request when the context window fills. Memory management is not optional.

**Red Flag 4: You give the agent unlimited iterations.** If your design has no max_iterations, no cost limit, and no time limit, you are building a system that can run forever and cost infinite money. Every agent run must have hard limits.

**Red Flag 5: You do not distinguish between tool call patterns.** If you describe all tool calls as the same mechanism, you miss the important distinction between read-only tools (safe to call any time) and write tools (require confirmation). The agent must handle these differently.

**Red Flag 6: You design a single agent for everything.** If your agent has 50 tools, covers 20 use cases, and has a 5,000-word system prompt, you are building the kitchen sink anti-pattern. Specialized agents with focused prompts and limited tool sets outperform monolithic agents.

**Red Flag 7: You ignore cost.** LLM calls are expensive. If your design does not consider the cost of each agent iteration, the cost of tool calls, and the total cost per request, you are building a system that will be too expensive to operate at scale.

**Red Flag 8: You have no observability.** If your design does not include logging, tracing, and metrics for the agent loop, you will be unable to debug, optimize, or improve the system. Agent behavior is non-deterministic — observability is mandatory.

**Red Flag 9: You treat all tool calls as equal.** You have not distinguished between read-only tools (safe to call anytime) and write tools (require confirmation). The agent should handle these differently. Read-only tools can be called freely. Write tools require permission checks, confirmation prompts, and audit logging.

**Red Flag 10: You ignore the human.** Your design has no mechanism for the agent to ask for help, escalate to a human, or incorporate human feedback. Every agent system needs a human-in-the-loop path, even if it is rarely used.

**Red Flag 11: You propose a multi-agent system when a single agent would suffice.** Multi-agent systems add complexity, latency, and cost. If a single agent with a well-designed tool set can handle the use case, do not add the complexity of multi-agent coordination. The interviewer will see this as over-engineering.

**Red Flag 12: You do not discuss rate limiting.** LLM APIs have rate limits. External APIs have rate limits. Your agent system must handle rate limiting gracefully. If you do not mention rate limiting, the interviewer will assume you have not operated an agent system at scale.

**Red Flag 13: You assume the LLM never makes mistakes.** The LLM can generate incorrect tool calls, misinterpret tool results, produce hallucinated reasoning, and go off-topic. Your design must handle all of these. If you assume the LLM is perfect, you have not deployed an agent system in production.

**Red Flag 14: You have no plan for prompt versioning.** Agent prompts change frequently. Each change can alter behavior. Without versioning, you cannot roll back a bad prompt deployment or A/B test prompt changes. The interviewer will expect you to have a prompt management strategy.
