## Naive Implementation

The simplest agent implementation is a single process that runs the agent loop synchronously. The agent receives a request, decomposes it, executes tools, and returns a response — all in one blocking call.

This implementation works for development and testing but fails in production. A single slow agent run blocks the entire service. A crash loses all in-flight agent state. There is no way to scale to multiple users. The naive approach has a maximum throughput of approximately 1 request per 30-60 seconds per process.

### Component Details

**API Gateway:** Handles authentication, rate limiting, request routing, and response caching. Converts synchronous HTTP requests into asynchronous tasks. Returns a task ID immediately so the client can poll for results. The gateway also handles WebSocket connections for streaming agent responses — as each iteration completes, the result is streamed to the client in real time.

**Orchestrator Service:** Manages the lifecycle of each agent run. Maintains state in a distributed cache (Redis). Publishes tasks to the task queue. Listens for completion events and returns results. The orchestrator is the only component that has a global view of all active agent runs. It can enforce cross-session policies, detect anomalies (a single user running 100 concurrent agents), and implement circuit breakers (stop all new agent runs if the system is degraded).

**Task Queue:** A durable message queue (Redis Streams, SQS, or RabbitMQ) that holds pending agent tasks. Enables asynchronous processing, load leveling, and retry on worker failure. The queue must support priority levels — critical support tickets should be processed before routine queries. The queue also supports delayed delivery for scheduled agent tasks ("run this analysis every day at 9 AM").

**Agent Worker Pool:** A pool of worker processes that execute the agent loop. Each worker runs an LLM client and a set of tool connectors. Workers are stateless — they load state from the cache and persist results back. Workers can scale horizontally based on queue depth. Each worker has a maximum concurrency setting (how many agent loops it can run in parallel) and a graceful shutdown mechanism (finish the current iteration, then stop accepting new tasks).

**Planner Service:** A dedicated service that handles plan generation. Can be scaled separately from the executor workers. The planner benefits from caching — identical or similar requests produce identical plans. The planner also maintains a plan quality score based on how often agents complete plans without modification. Poor plans are flagged for human review and prompt improvement.

**Tool Service Mesh:** A set of tool services that wrap external APIs, databases, and compute resources. Each tool service runs independently with its own scaling, retry, and rate-limiting policies. The tool mesh includes a circuit breaker pattern — if a tool service fails more than 50% of requests in a 60-second window, the circuit breaker opens and the agent receives a "tool unavailable" response instead of waiting for a timeout. The tool mesh also handles authentication translation — the agent uses internal credentials, and the tool mesh translates them to the external API's authentication mechanism.

**Memory Store:** A persistent store for agent state, conversation history, and tool results. Typically a combination of a document database (for full conversation history) and a cache (for active session state). The memory store implements time-to-live (TTL) policies: active sessions are retained for 24 hours, completed sessions for 30 days, and raw tool results for 7 days. The memory store also supports versioning — if a session is replayed, the agent sees the state as it was at the time of the original run, not the current state.

### Scaling Each Component

Each component scales independently based on its specific bottleneck:

| Component | Bottleneck | Scaling Strategy |
|-----------|-----------|-----------------|
| API Gateway | Connections per second | Add more gateway instances behind a load balancer |
| Orchestrator | Session state updates | Partition by user ID hash, add replicas per partition |
| Task Queue | Queue throughput | Partition into multiple queues by priority or agent type |
| Agent Workers | LLM API rate limits | Scale workers to match LLM provider's rate limit allocation |
| Planner | LLM calls per second | Cache identical plans, scale planning model separately |
| Tool Mesh | External API rate limits | Each tool service scales independently based on its API quota |
| Memory Store | Read/write throughput | Redis cluster for cache, read replicas for document store |

### Observability Pipeline

A production agent system must emit observability data at every layer:

- **Agent-level:** Each agent run emits a trace with every thought, tool call, observation, and loop iteration. The trace includes timestamps, token counts, and latency breakdowns. Traces are stored for 30 days for debugging and analysis.
- **Tool-level:** Each tool call emits a metric: success/failure, latency, retry count, and output size. Tool metrics are used for SLO tracking and capacity planning.
- **LLM-level:** Each LLM call emits a metric: model, tokens used, latency, cost, and response type (tool call vs. text). LLM metrics are used for cost allocation and model selection tuning.
- **System-level:** Standard infrastructure metrics: CPU, memory, queue depth, connection pool utilization, and error rates.

## Integration Patterns

Production agent systems integrate with external systems through several well-defined patterns.

### Pattern 1: API Tool Integration

The agent calls external APIs through a proxy service that handles authentication, rate limiting, and response transformation.

```
Agent ──> Tool Proxy ──> External API (Slack, Salesforce, GitHub, etc.)
              │
              ├── Authentication (OAuth, API key)
              ├── Rate limiting (per API quota)
              ├── Response transformation (JSON → structured)
              └── Error handling (retry, fallback, circuit breaker)
```

### Pattern 2: Knowledge Base Integration (RAG)

The agent retrieves relevant information from a knowledge base before reasoning. This is the Retrieval-Augmented Generation (RAG) pattern applied to agent systems.

### Pattern 3: Human-in-the-Loop Approval

For high-stakes actions, the agent pauses and waits for human approval before proceeding.

```
Agent: "I need to transfer $50,000 from account A to account B."
Human-in-the-Loop: [PENDING] Awaiting manager approval

System:
  1. Agent identifies a high-risk action (transfer over $10,000)
  2. Agent pauses execution, stores pending action in memory
  3. Notification sent to approver ("Manager John, approve transfer?")
  4. Approver reviews and approves or rejects
  5. Agent resumes: if approved, execute transfer; if rejected, notify user
  6. Session continues with the result
```

### Pattern 4: Event-Driven Triggers

Instead of waiting for a user request, the agent can be triggered by events from external systems.

```
Event: New support ticket created (priority: critical)
  └──> Event Bridge ──> Agent Orchestrator
       └──> Agent: "Analyze the ticket, find similar resolved issues,
            draft a response, and assign to the right team"
       └──> Actions: Search knowledge base, classify priority,
            generate response, update ticket system
```

### Pattern 5: Streaming Output

The agent streams intermediate results to the user as they become available, rather than waiting for the complete response. This pattern uses Server-Sent Events or WebSockets to push incremental updates.

```
User: "What are the top 3 competitors and their market share?"

Agent stream:
  [1s] "Thinking about the best approach..."
  [3s] "Searching for market share data..."
  [5s] "Finding competitors in this space..."
  [7s] "Analyzing market share distribution..."
  [10s] "Competitor 1: Acme Corp, 34% market share"
  [11s] "Competitor 2: Beta Inc, 28% market share"
  [12s] "Competitor 3: Gamma LLC, 15% market share"
  [14s] "Complete analysis with comparison table..."
```

Streaming improves user perception of latency significantly. A 15-second agent run feels fast when the user sees progress every 1-2 seconds. The same 15-second run feels slow when the user stares at a loading spinner. Streaming also gives the user the option to cancel early if the agent is going in the wrong direction.

### Pattern 6: Scheduled and Batch Processing

Agents can run on a schedule or process batches of work without user interaction. The orchestrator supports cron-style scheduling and batch job submission.

```
Schedule: "Every Monday at 9 AM, analyze the sales pipeline"
Batch: "Process these 10,000 support tickets and classify them"
```

Scheduled agents use the same architecture as interactive agents but with a scheduler component that triggers agent runs at predefined times. Batch processing uses the same worker pool but reads inputs from a queue or file rather than from a user chat.

### Data Sovereignty and Compliance

In multi-region deployments, data sovereignty regulations (GDPR in Europe, CCPA in California, PIPL in China) require that user data stays within specific geographic boundaries. The agent system must respect these boundaries.

The solution is data residency tagging. Each user session is tagged with the user's home region. The agent runtime ensures that:

- User data (conversation history, tool results, agent state) is stored only in the user's home region
- LLM calls route to inference endpoints in the same region as the user's data
- Tool calls that access external APIs respect the user's data residency — some tools may be prohibited in certain regions
- Cross-region data transfer is logged and audited for compliance

When a user travels from one region to another, the agent can continue from the user's home region with potentially higher latency, or the agent can migrate the session to the local region with explicit user consent.

### Disaster Recovery

The agent system must survive a full region outage. The disaster recovery strategy has three tiers:

**Tier 1: Active-Passive (cold standby).** A second region is provisioned but not running. When the primary region fails, the secondary region starts. Recovery time: 30 minutes. Data loss: up to 5 minutes of session state (last async replication). Suitable for development and staging environments.

**Tier 2: Active-Active (warm standby).** Both regions are running and serving traffic. User sessions are sticky to one region, but the other region can take over immediately. Recovery time: 1 minute. Data loss: up to 30 seconds of session state. Suitable for most production environments.

**Tier 3: Active-Active (hot standby).** Both regions are running and serving traffic. User sessions are replicated synchronously. Recovery time: zero. Data loss: zero. Requires synchronous replication, which adds latency to every state write. Suitable for financial and compliance-critical environments.

### Key Design Decisions for Multi-Region

**Region Affinity:** Route users to the nearest region based on DNS or latency-based routing. The agent state is stored in the user's home region. Cross-region reads are served from the global store with eventual consistency.

**State Propagation:** Agent state is replicated across regions asynchronously. If the user moves from the US to Europe, the agent can continue the conversation from the new region. Active sessions have a 15-minute TTL in the regional cache.

**Tool Availability:** Some tools are region-specific (local databases, region-restricted APIs). The tool mesh routes to the appropriate regional instance. Global tools (web search, public APIs) are available from any region.

**LLM Provider:** LLM inference endpoints are typically available in multiple regions. Route to the nearest LLM endpoint. Cache identical plans across regions to reduce LLM calls.

## Capacity Planning

For a system serving 10 million daily active users, the capacity planning mathematics is as follows.

### Assumptions

- 10 million DAU
- 5% of users make an agent request per day = 500,000 requests per day
- Peak hour factor: 10% of daily traffic = 50,000 requests per hour
- Peak minute factor: 20% of hourly traffic = 16,667 requests per minute
- Peak QPS: 16,667 / 60 = 278 requests per second (rounded to 300)

### Agent Request Characteristics

- Average agent iterations per request: 8
- Average LLM tokens per iteration: 4,000 input + 200 output = 4,200 tokens
- Average tool calls per iteration: 1.2
- Average tool execution time: 500ms
- Average agent run duration: 12 seconds

### LLM Token Throughput

```
Requests per second: 300
Iterations per request: 8
Total iterations per second: 300 x 8 = 2,400
Tokens per iteration: 4,200
Total tokens per second: 2,400 x 4,200 = 10,080,000 tokens/second

Hourly token consumption: 10,080,000 x 3,600 = 36.3 billion tokens/hour
Daily token consumption: 36.3 x 24 = 871 billion tokens/day

At $0.01 per 1K input tokens and $0.03 per 1K output tokens:
  Average cost per 1K tokens: (4000 x 0.01 + 200 x 0.03) / 4200 = $0.0105
  Cost per request: 8 x 4200 x 0.0105 / 1000 = $0.35
  Daily cost: 500,000 x $0.35 = $175,000 per day
  Monthly cost: $175,000 x 30 = $5,250,000
```

### Tool Execution Capacity

```
Tool calls per second: 300 x 8 x 1.2 = 2,880 tool calls/second
Average tool execution time: 500ms
Concurrent tool executions needed: 2,880 x 0.5 = 1,440

If each tool service handles 100 concurrent calls:
  Number of tool service instances: 1,440 / 100 = 15 instances
  With 2x headroom: 30 instances per tool type
```

### Memory Store Sizing

```
Active sessions: 300 requests/second x 12 seconds = 3,600 concurrent sessions
State per session: 50 KB (plan, observations, conversation)
Total memory needed: 3,600 x 50 KB = 180 MB (fits in a single Redis instance)

Daily storage: 500,000 requests x 100 KB per request = 50 GB/day
Monthly storage: 50 GB x 30 = 1.5 TB
Yearly storage: 1.5 TB x 12 = 18 TB (requires S3 or similar object storage)
```

## Deployment and Operations

Running AI agents in production requires operational practices beyond traditional microservices.

### Zero-Downtime Agent Deployment

Agent systems have long-running stateful operations (the agent loop). Standard blue-green deployment does not work because in-flight agents would be lost. The deployment strategy:

1. **Start new workers:** Spin up a new pool of agent workers with the new code.
2. **Drain old workers:** Stop routing new tasks to the old worker pool. Existing agents on old workers complete naturally.
3. **Migrate gracefully:** The orchestrator tracks which worker version handles each session. New sessions go to new workers. Old sessions stay on old workers.
4. **Cutover:** When all old sessions complete, terminate the old worker pool.

### Monitoring Agent Behavior

Standard infrastructure metrics (CPU, memory, latency) are not enough. Agent-specific metrics are essential.

**Success Rate:** The percentage of agent runs that complete without error. A successful agent run is one that produces a useful response, not just one that avoids crash. Target: 98%+.

**Loop Count Distribution:** The number of iterations per agent run. A spike in high loop counts indicates poor planning or tool failures. Track the p50, p90, and p99 of loop counts.

**Tool Error Rate:** The percentage of tool calls that fail. Track per tool type. A failing tool should trigger an alert and potentially a circuit breaker.

**Retry Rate:** The percentage of tool calls that require retries. High retry rates indicate tool instability or aggressive timeouts.

**Context Window Utilization:** The percentage of the LLM context window used at each iteration. Approaching 100% indicates memory management issues.

**Cost Per Request:** The total cost of LLM tokens plus tool execution for each agent run. Track per user, per request type, and per agent version.

### Incident Response for Agent Systems

Agent systems have unique incident response requirements. Standard runbooks do not cover agent-specific failures.

**Incident Type: Agent producing wrong answers.** The agent is responding to users but the answers are factually incorrect. Diagnosis: check the LLM response traces for hallucination patterns, check tool call results for errors that the agent ignored, check the system prompt for drift. Mitigation: rollback the agent prompt to the previous version, increase the temperature to reduce the agent's confidence in uncertain responses, add a fact-checking step to the agent loop.

**Incident Type: Agent making unexpected tool calls.** The agent is calling tools that it should not be calling, or calling tools with unexpected parameters. Diagnosis: review the tool call trace for patterns, check if the LLM is misinterpreting the tool descriptions. Mitigation: update tool descriptions to be more specific, add input validation rules, restrict tool access to the minimum necessary set.

**Incident Type: Agent loop timeout.** Agent runs are consistently hitting the max iteration limit without completing. Diagnosis: check if the planner is generating overly complex plans, check if tool calls are failing and requiring retries, check if the user requests are unusually complex. Mitigation: increase the max iteration limit, add better error recovery to reduce retry iterations, implement a "simplify" step that encourages the agent to produce shorter plans.

**Incident Type: Cost spike.** The daily LLM cost has doubled for no apparent reason. Diagnosis: check the number of agent runs, check the average iterations per run, check the token consumption per iteration, check if a model update changed behavior. Mitigation: reduce the max iteration limit, switch to a cheaper LLM model for routine tasks, add cost budgets with automatic throttling.

### Cost Optimization Strategies

LLM costs dominate the operating budget for agent systems. Several strategies reduce cost without sacrificing quality.

**Strategy 1: Caching.** Cache identical or similar agent plans. If 10% of requests are identical ("what is the weather in San Francisco?"), caching the plan saves the LLM call for that request. More importantly, cache tool call results. If multiple agents ask the database for the same customer record within a 60-second window, the cached result serves all but the first request. Plan caching can reduce LLM calls by 10-20%. Tool result caching can reduce external API calls by 30-50%.

**Strategy 2: Tiered models.** Use a fast, cheap model (GPT-4o-mini at $0.15 per million input tokens) for routine tool calls and a powerful model (GPT-4o at $2.50 per million input tokens) for complex reasoning. The ratio of cheap to expensive calls should be at least 5:1. This alone can cut LLM costs by 60-70% while maintaining quality on the hard problems.

**Strategy 3: Early termination.** Many agent runs can produce a useful result before all tasks complete. If the agent has enough information to answer the user's question after 3 iterations, do not run the remaining 5 iterations. Implement a "sufficiency check" after each iteration that evaluates whether the accumulated observations are sufficient to produce a final response. Early termination can reduce average iteration count by 30-40%.

**Strategy 4: Batch tool calls.** Instead of making 5 individual API calls, combine them into a single batch call if the tool supports it. Batching reduces both cost and latency. A batch of 5 searches costs the same as 1 search in terms of tool infrastructure but returns 5 results. Batch processing also reduces the number of agent iterations because the agent waits for one batch result instead of five sequential results.

**Strategy 5: Prompt compression.** The system prompt and tool descriptions consume thousands of tokens per iteration. Compress tool descriptions by removing redundant information, using shorter names, and consolidating similar tools. Every 1,000 tokens saved per iteration saves $15 per million requests. For a system processing 4 million requests per month, compressing prompts by 1,000 tokens saves $60 per month.

### Security Architecture

Agent systems introduce unique security challenges. The agent has access to tool calls that can read, write, and modify data. An attacker who compromises the agent's LLM can execute arbitrary tool calls. Security must be designed into the architecture from day one.

**Defense 1: Tool permission levels.** Assign each tool a permission level: read (data retrieval only), write (data modification), exec (code execution). The agent can call read-level tools without restriction. Write-level tools require a confirmation prompt. Exec-level tools require explicit human approval and run in a sandboxed environment with network isolation and resource limits.

**Defense 2: Input validation.** Validate every tool call parameter against an allowlist. If a tool expects an email address, validate that the parameter is a valid email format. If a tool expects a database query, validate it against a list of allowed query patterns. This prevents prompt injection attacks that attempt to call tools with malicious parameters. Validation happens before the tool is invoked, not after.

**Defense 3: Output sanitization.** Tool results may contain sensitive data (PII, API keys, internal IP addresses). Sanitize tool output before passing it to the LLM for reasoning. Strip or mask PII, redact API keys and passwords, and truncate excessively large results. The LLM should never see raw sensitive data unless it is essential for the task.

**Defense 4: Session isolation.** Each agent session runs in an isolated context. One session cannot access another session's state, tools, or results. This prevents cross-session data leakage and limits the blast radius of a compromised session. Session isolation is enforced at the state store level using namespace prefixes.

**Defense 5: Audit logging.** Every tool call, observation, and decision is logged with a timestamp, session ID, user ID, and the full input and output. Logs are append-only and immutable. Any security incident can be fully reconstructed from the logs. Audit logs are retained for a minimum of 90 days, and access to the logs requires administrative privileges.

### A/B Testing Agent Prompts

Agent prompts are code. Changes to system prompts, tool descriptions, or planning instructions should be versioned and tested.

```
Experiment: Increase temperature from 0.2 to 0.4
  Control: temperature = 0.2 (50% of traffic)
  Variant: temperature = 0.4 (50% of traffic)

Metrics to compare:
  - Success rate (did the agent complete the task?)
  - Average loop count (more iterations = less efficient)
  - User satisfaction score (from post-interaction survey)
  - Cost per request (higher temperature = more tokens)
  - Tool error rate (higher temperature = more creative but less reliable tool selection)

Duration: 2 days or 10,000 requests per variant, whichever is longer.
```

Agent prompts should be stored in a version control system, code-reviewed, and deployed through the same CI/CD pipeline as the application code. Store prompts as templates with variables for runtime injection.
