## Tier 1: Foundation

### Challenge 1: Base-62 Encoding Depth

**Prompt:** You are building the key generation module for a URL shortener. Your team decides to use base-62 encoding with the alphabet `0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ`. You need to implement the encode and decode functions and verify your understanding of the namespace.

**Part A — Encode and Decode (10 minutes):** Implement the `encode_base62` and `decode_base62` functions. The encoder accepts a non-negative integer and returns the shortest string that represents it. Zero encodes to `0`. The decoder accepts a string and returns the integer. The string `100` decodes to 3844 (62^2). Include input validation: reject negative numbers in encode, reject invalid characters in decode.

**Part B — Namespace Math (5 minutes):** Answer these questions with calculations:

1. How many unique short codes exist at each length from 1 to 8 characters?
2. At 10 million new URLs per day, how many days until the 6-character namespace is 50 percent exhausted? The 7-character namespace?
3. If you remove 5 visually ambiguous characters (0, O, I, l, 1), reducing the alphabet to 57 characters, how many 7-character codes can you generate? What percentage of the base-62 namespace do you lose?
4. A URL shortener has 500 million mappings using 7-character codes. What percentage of the namespace is consumed?

**Part C — Performance Requirement (5 minutes):** Write a test that verifies your encode and decode functions complete in under 1 microsecond per call for random inputs. Explain how you would optimize if the functions are slower than the target.

**Model Answer:**

Part A:
```python
ALPHABET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
BASE = len(ALPHABET)

def encode_base62(num: int) -> str:
    if num < 0:
        raise ValueError("Input must be non-negative")
    if num == 0:
        return ALPHABET[0]
    result = []
    while num > 0:
        result.append(ALPHABET[num % BASE])
        num //= BASE
    return "".join(reversed(result))

def decode_base62(code: str) -> int:
    if not code:
        raise ValueError("Input must not be empty")
    result = 0
    for char in code:
        idx = ALPHABET.find(char)
        if idx == -1:
            raise ValueError(f"Invalid character: {char}")
        result = result * BASE + idx
    return result
```

Part B:
1. 62^1 = 62. 62^2 = 3,844. 62^3 = 238,328. 62^4 = 14,776,336. 62^5 = 916,132,832. 62^6 = 56,800,235,584. 62^7 = 3,521,614,606,208. 62^8 = 218,340,105,584,896.
2. 6-character: 56.8 billion / 2 = 28.4 billion. At 10 million per day: 2,840 days (7.8 years). 7-character: 3.5 trillion / 2 = 1.76 trillion. At 10 million per day: 176,000 days (482 years).
3. 57^7 = 1,954,897,237,353. Lost: (62^7 - 57^7) / 62^7 = (3.52e12 - 1.95e12) / 3.52e12 = 44.5 percent of the namespace.
4. 500 million / 3.5 trillion = 0.014 percent. Less than one one-hundredth of a percent.

**Common Mistakes:**
- Off-by-one in length calculation: a base-62 string of length N starts at 62^(N-1), not 62^N.
- Characters reversed: the remainder represents the least significant digit, which goes at the end of the string.
- Not handling zero: zero is a valid input and must encode to a single character, not an empty string.
- Forgetting that removing ambiguous characters costs 44.5 percent of the namespace, not just 5/62 = 8 percent. The cost compounds across every character position.

### Challenge 2: Collision Handling

**Prompt:** Your URL shortener uses hash-based key generation. The short code is derived from the first 4 bytes of an MD5 hash of the long URL, encoded in base-62 and truncated to 7 characters. You currently have 100 million mappings.

**Part A — Collision Probability (5 minutes):** Calculate the collision probability for the next 1,000 creates. Then calculate the probability that the next create collides with any of the 100 million existing entries. Use these numbers to assess whether hash-based generation is viable at your scale.

**Part B — Resolution Strategy (10 minutes):** Design a collision resolution strategy that handles the case where two different URLs produce the same short code. Your strategy must guarantee eventual uniqueness and must not degrade the throughput of the create endpoint by more than 5 percent.

**Part C — Deduplication Logic (5 minutes):** If the same URL is submitted twice, the system should return the existing short code instead of creating a new one. Integrate deduplication into your collision handling logic. What happens during a race condition where two users submit the same URL simultaneously?

**Model Answer:**

Part A:
Collision probability for the next 1,000 creates: P(collision in any of 1,000 creates) = 1 - (1 - 1/62^7)^1000.
62^7 = 3.5e12. 1/3.5e12 = 2.86e-13. (1 - 2.86e-13)^1000 ≈ 1 - 1000 * 2.86e-13 ≈ 1 - 2.86e-10.
Probability is approximately 1 in 3.5 billion. Negligible.

Probability that the next create collides with one of 100 million existing entries: the birthday paradox says P(collision between any two entries out of N) = N^2 / (2 * 62^7). At N = 100 million: (1e8)^2 / (2 * 3.5e12) = 1e16 / 7e12 = 1,428 expected collisions. The next create has a collision probability of 100 million / 3.5e12 = 1 in 35,000. Not negligible at this scale.

Part B:
```python
def create_short_url(url: str, max_attempts: int = 5) -> str:
    normalized_url = url.strip().lower()
    for attempt in range(max_attempts):
        if attempt == 0:
            input_str = normalized_url
        else:
            input_str = f"{normalized_url}:{attempt}"
        digest = hashlib.md5(input_str.encode()).digest()
        value = int.from_bytes(digest[:4], byteorder="big") % (62 ** 7)
        code = encode_base62(value).rjust(7, "0")
        existing = db.lookup(code)
        if existing is None:
            db.insert(code, url)
            return code
        if existing == url:
            return code  # Duplicate, return existing
    raise CollisionLimitError("Max collision retries exhausted")
```

The saturation strategy uses double hashing: append a counter to the URL and rehash. The maximum of 5 attempts keeps the worst-case create latency at 5 * 5ms = 25ms, a 25 percent increase that still meets the 50ms target. In practice, the first retry succeeds 99.997 percent of the time.

Part C:
The collision check in the loop also serves as deduplication. When the code already exists and maps to the same URL, the function returns it immediately. For the race condition where two users submit the same URL simultaneously, both requestors compute the same hash. The database INSERT uses a UNIQUE constraint. The second INSERT fails, and the application falls through to the collision check, which finds that the existing mapping matches and returns the existing code. This is safe without explicit locking.

**Common Mistakes:**
- Using the birthday paradox formula incorrectly: the formula N^2 / (2 * M) estimates collisions among N items in a space of M, not the probability that a single new item collides with N existing items.
- Unlimited retry loops: without a maximum, a pathological case with repeated collisions blocks the create endpoint indefinitely. Always set a maximum.
- Not handling the insert race condition: two concurrent inserts for the same code cause a database constraint violation. The application must catch this and retry or resolve.

### Challenge 3: Redirect vs Proxy Trade-offs

**Prompt:** Your URL shortener currently uses HTTP 302 redirects. A product manager asks whether the system should switch to an HTTP proxy approach instead — where the shortener fetches the destination content and returns it directly to the client, rather than redirecting the client to the destination.

**Part A — Comparison Table (10 minutes):** Create a comparison of redirect (302) versus proxy for the following dimensions: bandwidth cost, latency, caching behavior, content type support, analytics accuracy, security implications, scalability, and legal liability. For each dimension, explain which approach is better and why.

**Part B — Decision (5 minutes):** Under what circumstances would the proxy approach be the better choice? Under what circumstances must you use a redirect? Provide specific examples.

**Part C — Hybrid Approach (5 minutes):** Design a hybrid approach that uses redirects for most links but switches to proxy mode for specific use cases. How would you detect which mode to use? What does the data model look like?

**Model Answer:**

Part A:
| Dimension | Redirect (302) | Proxy | Winner |
|-----------|---------------|-------|--------|
| Bandwidth cost | Low. The shortener sends a 300-byte response. The client fetches the destination directly. | High. The shortener downloads the full content (average 2 MB per page) and serves it to the client. Bandwidth increases by 10,000x. | Redirect |
| Latency | The client receives the redirect in 10ms, then fetches the destination in 500ms. Total: 510ms. | The shortener fetches the destination (500ms) and serves it to the client. The client gets content in 510ms — same total. | Tie |
| Caching | CDN can cache 301 responses. Browser caches the redirect. | CDN can cache proxied content (HTML, images). Caching works for both. | Tie |
| Content type support | Any content type. The destination server handles it. | Only HTTP content. Cannot proxy HTTPS streams, WebSockets, or binary downloads. | Redirect |
| Analytics accuracy | The shortener sees every redirect request. Click events are emitted per request. | The shortener sees every content request. Click events are emitted per request. | Tie |
| Security | The destination is visible in the Location header. Users can inspect the URL before visiting. | The destination is hidden. Users cannot see where they are going. Makes phishing easier to hide. | Redirect |
| Scalability | The shortener handles 300-byte responses. Scales to millions of req/s with moderate infrastructure. | The shortener must download and serve full content. Scales to thousands of req/s before bandwidth saturates. | Redirect |
| Legal liability | The shortener is not hosting the content. The destination host is responsible. | The shortener is hosting and serving the content. Legal liability for DMCA, defamation, copyright. | Redirect |

Part B:
Use proxy mode when: the destination content is behind an authentication wall that the shortener's users cannot access, the destination is slow or unreliable and the shortener can cache the content, or you need to transform the content (inject analytics scripts, remove tracking parameters, translate text).

Use redirect mode when: you need to scale to millions of requests, you serve any content type beyond basic HTTP, you want to minimize legal liability, or you want users to see the destination URL in their browser address bar.

Part C:
Add a `proxy_enabled` flag to the URL mappings table. When true, the redirect engine fetches the destination content and serves it. When false or NULL, the engine issues a standard 302 redirect.

Detection logic:
- If the destination URL starts with `https://` and sets Content-Security-Policy headers, use redirect.
- If the destination is a large file (video, PDF, ZIP), use redirect. The shortener should not proxy 500 MB files.
- If the destination is behind a login wall and the shortener has authorized access, use proxy.
- If the destination is known to be slow (response time over 5 seconds), use proxy with caching.

```json
{
  "short_code": "camp1",
  "long_url": "https://example.com/whitepaper.pdf",
  "proxy_enabled": true,
  "proxy_ttl": 86400,
  "proxy_transform": "inject_analytics"
}
```

**Common Mistakes:**
- Assuming proxy is always better because "it hides the destination." Proxy mode hides the destination from the user too, creating a security risk and a legal liability.
- Not accounting for the 10,000x bandwidth multiplication. A proxy shortener at 15,000 req/s would consume 30 GB/s of bandwidth, costing millions per month.
- Forgetting that proxy mode breaks HTTPS. The shortener terminates the client's HTTPS connection and establishes a new HTTPS connection to the destination. The client cannot verify the destination's certificate.

## Tier 2: Design

### Challenge 4: 100 Million URLs with Analytics

**Prompt:** Design a URL shortener for a marketing analytics platform. The system must handle:
- 100 million total active short URLs
- 10 million new shorten operations per day
- 500 million redirects per day
- Per-link analytics: click count, referrer domain, country, device type, hourly time-series
- Custom slugs: users can specify their own short codes (up to 20 characters, alphanumeric plus hyphens)
- Link expiration: configurable TTL from 1 hour to permanent
- 99.99 percent availability
- Global audience: users in North America, Europe, and Asia

**Scale Estimates (5 minutes):** Calculate:
- Peak redirect throughput (assume 60 percent of daily traffic during 8 business hours)
- Storage for 3 years of URL mappings
- Storage for 90 days of raw click events
- Cache memory for the hot set (assume 20 percent of URLs receive 80 percent of traffic)
- Write throughput for new URLs at peak

**Data Model (10 minutes):** Design the database schema for URL mappings and click events. Include indexes, partitioning strategy, and column types. Explain why you chose each type and index.

**Caching Strategy (5 minutes):** Design a multi-tier cache architecture. What is cached at each tier? What are the TTLs? How do you handle cache invalidation when a link is updated or deleted?

**Analytics Pipeline (10 minutes):** Design the click event pipeline from redirect to dashboard. Include: event schema, message queue configuration, stream processing logic, storage layers, and dashboard query patterns. The dashboard must refresh within 5 seconds.

**Model Answer:**

Scale Estimates:
- Peak redirect throughput: 500 million / 8 hours = 62.5 million per hour = 17,361 req/s.
- URL mapping storage (3 years): 10 million per day * 365 * 3 = 10.95 billion mappings. At 300 bytes per row (8-byte short_code, 200-byte average long_url, 16-byte UUID, timestamps, overhead): approximately 3.3 TB.
- Click event storage (90 days): 500 million per day * 90 = 45 billion events. At 200 bytes per event (short_code, timestamp, country code, device type, referrer domain, session ID): approximately 9 TB.
- Cache hot set: 20 percent of 100 million URLs receive 80 percent of traffic. Hot set = 20 million entries. At 344 bytes per entry * 3 replicas = 20.6 GB of Redis memory.
- Write throughput peak: 10 million creates per day, 60 percent in 8 business hours = 6 million / 8 hours = 208 creates per second.

Data Model:
```sql
CREATE TABLE url_mappings (
    short_code VARCHAR(8) NOT NULL,
    long_url VARCHAR(2048) NOT NULL,
    owner_id CHAR(36) NOT NULL,
    created_at TIMESTAMP(6) NOT NULL,
    expires_at TIMESTAMP(6) NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    is_custom BOOLEAN NOT NULL DEFAULT FALSE,
    click_count BIGINT NOT NULL DEFAULT 0,
    metadata JSON NULL,
    PRIMARY KEY (short_code),
    INDEX idx_owner_id (owner_id),
    INDEX idx_expires_at (expires_at),
    INDEX idx_created_at (created_at)
) PARTITION BY KEY (short_code) PARTITIONS 128;
```

The click events table uses a write-optimized schema. Each redirect emits one event row. The table is partitioned by time (monthly) for efficient retention management:
```sql
CREATE TABLE click_events (
    event_id CHAR(36) NOT NULL,
    short_code VARCHAR(8) NOT NULL,
    timestamp TIMESTAMP(6) NOT NULL,
    country CHAR(2),
    device_type VARCHAR(10),
    referrer_domain VARCHAR(255),
    session_id CHAR(36),
    user_agent TEXT,
    PRIMARY KEY (event_id, timestamp)
) PARTITION BY RANGE (TO_DAYS(timestamp)) (
    PARTITION p_current VALUES LESS THAN (TO_DAYS('2025-08-01')),
    PARTITION p_next VALUES LESS THAN MAXVALUE
);
```

Caching Strategy:
- L1 (application server): local LRU cache. 10,000 entries per server. TTL: 60 seconds. Stores short_code -> long_url mappings.
- L2 (Redis cluster): 10 primary nodes, 3 replicas each. 20 GB per node. TTL: 24 hours. Stores short_code -> (long_url, expires_at, is_active).
- Cache invalidation: on update or delete, the application sends an explicit DEL command to Redis. The cache is write-through on create. For CDN caching, use surrogate keys to purge all cached redirects for a specific short code when it is updated.

Analytics Pipeline:
1. The redirect engine emits click events to an in-memory buffer. The buffer batches 500 events or 100ms of time, then flushes to Kafka.
2. Kafka topic has 32 partitions with 3x replication. Retention: 7 days. Partition key: short_code for ordered processing per code.
3. Kafka Streams consumer: deduplicates by event_id (5-minute window), enriches with GeoIP (country, region, city) and device classification, aggregates into 5-minute time buckets per short_code.
4. Aggregated data (short_code, timestamp_bucket, count, country_breakdown, device_breakdown) is written to a time-series database (ClickHouse). Raw events are stored in a separate ClickHouse table with 90-day retention.
5. The dashboard queries the aggregated table for sub-second responses. It queries the raw table only for drill-down investigations.

**Common Mistakes:**
- Not accounting for the long tail: 80 percent of traffic comes from 20 percent of URLs, but the remaining 80 percent of URLs still need to be in the database. A cache sized for only the top 10 percent misses too many requests.
- Using a single analytics table: mixing raw events and aggregated data in the same table makes both writes and queries slow. Separate the pipeline into hot (aggregated) and cold (raw) paths.
- Not partitioning click events: 45 billion events over 90 days without partitioning makes queries scan the entire table. Monthly partitioning lets queries scan one month at a time and drops old partitions instantly.

### Challenge 5: Custom Slugs with Analytics

**Prompt:** Design the custom slug subsystem for a URL shortener. The system must support:
- User-specified short codes (slugs) of 4 to 20 alphanumeric characters plus hyphens
- Auto-generated codes that must not collide with custom slugs
- Slug changes: users can change the destination URL of a slug without changing the slug itself
- Slug ownership verification: only the owner can modify or delete their slug
- Slug release: inactive slugs are released after 90 days for reuse by other users
- Per-slug analytics with comparison to auto-generated codes

**Data Model (10 minutes):** Design the database schema that supports custom slugs alongside auto-generated codes. Pay special attention to the namespace collision problem: how do you prevent a custom slug from matching an auto-generated code?

**Collision Prevention (5 minutes):** Auto-generated codes use 7-character base-62 strings. Custom slugs can be 4-20 characters. A custom slug of "aB3xK9" would collide with a generated code. Design a strategy that prevents this collision without requiring a check against the generated namespace for every custom slug submission.

**Slug Release (10 minutes):** Design the slug release mechanism. After 90 days of inactivity, a slug becomes available for other users. "Inactivity" means no redirects and no edits for 90 consecutive days. The previous owner must receive a notification 30 days before release. Design the release process to prevent races where two users attempt to claim the same released slug simultaneously.

**Analytics Comparison (5 minutes):** A marketing manager wants to know whether custom slugs perform better than auto-generated codes in terms of click-through rate. Design the analytics comparison. What metrics do you compare? How do you control for the fact that custom slugs are used by different types of campaigns than auto-generated codes?

**Model Answer:**

Data Model:
The custom slugs table extends the URL mappings schema. A separate `custom_slugs` table tracks ownership and activity:
```sql
CREATE TABLE custom_slugs (
    slug VARCHAR(20) NOT NULL,
    owner_id CHAR(36) NOT NULL,
    destination_url VARCHAR(2048) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_accessed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    is_released BOOLEAN NOT NULL DEFAULT FALSE,
    release_notification_sent BOOLEAN NOT NULL DEFAULT FALSE,
    PRIMARY KEY (slug),
    INDEX idx_owner_id (owner_id),
    INDEX idx_last_accessed (last_accessed_at),
    INDEX idx_released (is_released, last_accessed_at)
);
```

Collision Prevention:
Reserve all 7-character base-62 strings that could be auto-generated as reserved. When a user submits a custom slug, the system checks whether the slug is a valid auto-generated code by attempting to decode it:

```python
def is_reserved_code(slug: str) -> bool:
    if len(slug) != 7:
        return False
    try:
        value = decode_base62(slug)
        return value < (62 ** 7)
    except ValueError:
        return False
```

If `is_reserved_code` returns True, the custom slug is rejected. This check runs in under 1 microsecond. Additionally, auto-generated codes are never checked against the custom slugs table during redirect lookup because the tables are separate: look up in `url_mappings` first, then in `custom_slugs` if not found.

Slug Release:
A background job runs daily at 02:00 UTC:
1. Query `custom_slugs` WHERE `is_released = FALSE` AND `last_accessed_at < NOW() - INTERVAL 60 DAY`. For each matching slug, send a notification to the owner: "Your slug X will be released in 30 days."
2. Set `release_notification_sent = TRUE`.
3. Query again WHERE `is_released = FALSE` AND `release_notification_sent = TRUE` AND `last_accessed_at < NOW() - INTERVAL 90 DAY`. Set `is_released = TRUE`.

When a user claims a released slug, the claim uses a database transaction with row-level locking:
```python
def claim_slug(slug: str, owner_id: str, destination_url: str):
    with db.transaction():
        existing = db.query(
            "SELECT slug FROM custom_slugs WHERE slug = %s FOR UPDATE", slug
        )
        if existing:
            row = db.query(
                "SELECT is_released FROM custom_slugs WHERE slug = %s", slug
            )
            if not row["is_released"]:
                raise SlugTakenError("Slug is currently owned")
        db.execute(
            "INSERT INTO custom_slugs (slug, owner_id, destination_url) "
            "VALUES (%s, %s, %s) "
            "ON DUPLICATE KEY UPDATE owner_id = VALUES(owner_id), "
            "destination_url = VALUES(destination_url), "
            "is_released = FALSE",
            (slug, owner_id, destination_url)
        )
```

Analytics Comparison:
The analytics system compares CTR for custom slugs versus auto-generated codes within the same campaign category. Use a cohort analysis: group slugs by creation channel (marketing dashboard, API, web UI) and compare CTR within each channel. Custom slugs created through the marketing dashboard are compared to auto-generated codes created through the same dashboard.

```sql
SELECT
    CASE WHEN c.is_custom THEN 'custom' ELSE 'generated' END AS slug_type,
    AVG(clicks / impressions) AS avg_ctr,
    COUNT(*) AS total_links
FROM url_mappings u
JOIN (
    SELECT short_code, COUNT(*) AS clicks, COUNT(DISTINCT session_id) AS impressions
    FROM click_events
    WHERE timestamp > NOW() - INTERVAL 30 DAY
    GROUP BY short_code
) a ON u.short_code = a.short_code
WHERE u.creation_channel = 'marketing_dashboard'
GROUP BY slug_type;
```

**Common Mistakes:**
- Using a single `url_mappings` table for both custom slugs and auto-generated codes. This creates a hot spot on the primary key index and makes the reservation system complex. Separate tables with separate lookup paths.
- Not notifying the owner before releasing a slug. Releasing a slug without notice breaks links that the owner may still be using through cached redirects.
- Releasing a slug but allowing the old mapping to continue serving redirects. The old mapping must be deleted or redirected to an error page.

### Challenge 6: QR Code Redirect System

**Prompt:** Design a URL shortener optimized for QR codes printed on physical materials: product packaging, business cards, billboards, and magazine ads.

**Constraints:**
- The QR code must be scannable at a minimum size of 8mm x 8mm
- The short URL must be a maximum of 24 characters including the domain (`https://s.co/` is 14 characters, leaving 10 for the short code)
- The link must never expire for printed materials (a business card lasts 5 years)
- The QR code must be readable under suboptimal conditions: low light, damaged surface, 45-degree angle
- Analytics must track scan time, geographic location, device type, and repeat scans
- The system must support bulk creation: a marketing team prints 500,000 product packages at once, each with a unique QR code

**QR Code Optimization (10 minutes):** QR codes have four encoding modes: numeric, alphanumeric, byte/binary, and kanji. Alphanumeric mode encodes 45 characters per cell, the densest mode for URL payloads. Your short domain is `https://s.co/`, which is 14 characters. You have 10 characters remaining. Design a short code alphabet that maximizes QR code scan reliability. Which characters do you exclude and why? What is the maximum namespace size?

**Bulk Creation Pipeline (10 minutes):** Design the bulk creation pipeline for 500,000 unique QR codes. Each code points to a unique destination URL (product-specific landing pages with tracking). The pipeline must complete within 4 hours. Design the key generation strategy, the batch insert process, and the QR code image generation step.

**Offline Redirect Cache (10 minutes):** A QR code on a product package must work even when the user opens it in a region with poor connectivity. The initial page load may fail. Design a strategy that makes the redirect work even when the network is unreliable. Consider: PWA service workers, preload hints, DNS prefetch, and browser caching.

**Model Answer:**

QR Code Optimization:
The short code alphabet should exclude characters that are easily confused in low-resolution QR scans: 0 (zero), O (uppercase oh), I (uppercase eye), l (lowercase ell), 1 (one). This leaves an alphabet of 57 characters (10 digits minus 0 and 1, plus 26 lowercase minus l, plus 26 uppercase minus O and I).

With a 10-character code and 57-character alphabet: 57^10 = 3.6 x 10^17 possible codes. This is 100,000 times larger than the 7-character base-62 namespace. For 500,000 products, this is effectively infinite.

QR code error correction level should be H (30 percent recovery). At version 2 (25x25 cells) with alphanumeric mode and error correction H, the maximum payload is 20 characters. Your 24-character URL (14 domain + 10 code) fits in version 2 with error correction M (15 percent recovery). For damaged surfaces, version 2 with error correction H fits 20 characters, which is tight. Use version 3 (29x29 cells) with error correction H for maximum reliability on damaged surfaces.

Bulk Creation Pipeline:
The key generation strategy uses a sequential counter for bulk creation. Each of the 500,000 codes receives the next ID from a database sequence. The ID is encoded in the reduced 57-character alphabet:

```python
ALPHABET_QR = "23456789abcdefghjkmnpqrstuvwxyzABCDEFGHJKLMNPQRSTUVWXYZ"
BASE_QR = len(ALPHABET_QR)  # 57

def generate_qr_code(counter: int) -> str:
    code = []
    for _ in range(10):
        code.append(ALPHABET_QR[counter % BASE_QR])
        counter //= BASE_QR
    return "".join(reversed(code))
```

Batch insert pipeline:
1. Read the product list CSV: 500,000 rows with product_id and landing page URL.
2. Allocate ID ranges in batches of 10,000 from the database sequence.
3. For each batch, generate the short codes, prepare the INSERT statements.
4. Batch INSERT into the database (10,000 rows per transaction).
5. Generate QR code images in parallel using a worker pool (50 workers). Each QR code is a version 3 image (29x29 cells) with error correction H, output as PNG at 300 DPI.
6. Store the mapping: product_id -> short_code -> QR code image file path.

The total time: 500,000 / 10,000 per transaction = 50 transactions for inserts (approximately 5 minutes). QR code generation: 500,000 / 50 workers = 10,000 images per worker. At 200ms per image: 2,000 seconds = 33 minutes. Total pipeline time: under 1 hour, well within the 4-hour limit.

Offline Redirect Cache:
The user scans the QR code and the browser navigates to the short URL. If the network is unavailable, the browser cannot resolve the short domain or fetch the redirect.

Strategy 1: Service Worker pre-cache. If the user has visited the short domain before, a service worker caches the redirect response. On the next visit (even offline), the service worker returns the cached redirect.

Strategy 2: DNS prefetch with HTTP header. Include a `Link: <https://s.co/>; rel=dns-prefetch` HTTP header or use `<link rel="dns-prefetch" href="https://s.co/">` in the referring page's HTML. This pre-resolves the DNS for the short domain.

Strategy 3: 301 redirect with aggressive caching. Use 301 redirects with Cache-Control: public, max-age=31536000, immutable. The browser caches the redirect for 1 year. If the user has ever visited a short URL while online, the browser has the redirect cached and can follow it even without a network connection. This is the most practical offline strategy.

For the QR code use case, use 301 redirects exclusively. The analytics data comes from the first visit (when the QR code is scanned for the first time). Subsequent visits are cached and do not need analytics. If the product team needs per-visit analytics, embed tracking parameters in the destination URL instead of relying on shortener click events.

**Common Mistakes:**
- Using the standard base-62 alphabet for QR codes without removing ambiguous characters. The characters 0, O, I, l, 1 are difficult to distinguish in low-resolution scans, increasing the error rate.
- Not considering the QR code version limit. At version 2 (25x25 cells), a 24-character URL with error correction H does not fit. You must either shorten the domain or increase the QR code version, which increases the printed size.
- Forgetting that QR codes on physical materials cannot be updated. The printed link is permanent. The destination URL can change, but the short code cannot. Use a URL shortener that supports destination updates without changing the short code.
- Using 302 redirects for printed materials. The user may visit the link once. If the shortener is down, the link is broken forever for that visitor. Use 301 so the browser caches the redirect and does not depend on the shortener for future visits.

## Tier 3: Expert

### Challenge 7: Multi-Tenant URL Shortener for Enterprise with SLA Tiers

**Prompt:** Design a multi-tenant URL shortener for enterprise customers. Each customer is a company with its own branding (custom short domain), user management, analytics access, and billing plan. The system has three SLA tiers:

- **Free tier**: Max 1,000 short URLs per month. Shared infrastructure. Redirect latency P99 under 100ms. No custom domain. Basic click counts only. Rate limited to 10 creates per minute.
- **Business tier**: Max 100,000 short URLs per month. Dedicated Redis cache shard. Redirect latency P99 under 20ms. Custom short domain. Full analytics (geographic, device, referrer, 1-year retention). API access with 100 req/s rate limit. Rate limited to 1,000 creates per minute.
- **Enterprise tier**: Unlimited short URLs. Dedicated cache cluster. Dedicated database read replicas. Redirect latency P99 under 10ms. Custom short domain with SSL. Full analytics with 3-year retention and data export. API access with 10,000 req/s rate limit. SSO integration. Audit logging of all operations. 99.995 percent availability SLA.

The system has 500 free-tier customers, 50 business-tier customers, and 5 enterprise-tier customers. The enterprise customers generate 80 percent of the traffic.

**Tenant Isolation Model (10 minutes):** Design the tenant isolation model. How do you provide dedicated cache shards and database replicas for enterprise customers while keeping the free tier on shared infrastructure? What is the data isolation strategy for the URL mappings and analytics tables?

**SLA-Aware Routing (10 minutes):** Design the request routing layer that enforces SLA targets. Enterprise requests must always hit dedicated infrastructure. Free tier requests are routed to shared infrastructure. How does the CDN or API gateway know which tier a request belongs to? How do you prevent a free-tier traffic spike from affecting enterprise latency?

**Billing and Quota Enforcement (10 minutes):** Design the quota enforcement system for the three tiers. The free tier is limited to 1,000 URLs per month and 10 creates per minute. Business tier is limited to 100,000 URLs per month and 1,000 creates per minute. Enterprise tier has no limits. How do you count and enforce these quotas efficiently? What happens when a customer exceeds their quota? How do you handle the transition between billing cycles?

**Failure Isolation (10 minutes):** Design the failure isolation model. A free-tier cache node fails and causes errors for all free-tier customers. How do you prevent this from affecting business and enterprise customers? An enterprise customer's dedicated database replica fails. How do you maintain their 99.995 percent availability SLA?

**Model Answer:**

Tenant Isolation Model:
The system uses a tenant-aware data model. Each tenant has a `tenant_id` and a `tier`. The URL mapping table includes `tenant_id` as a partition key for analytics isolation, but redirect lookups use only the short code (which is globally unique across all tenants).

The short code incorporates a tenant prefix. Each tenant's short codes start with a unique prefix. For free-tier customers, the prefix is 2 characters. For business-tier, it is 1 character. For enterprise-tier, it is a single custom character chosen by the customer. This enables tenant-aware routing based on the URL path.

Infrastructure mapping:
- Free tier: shared application servers, shared Redis cluster (10 nodes), shared MySQL cluster (1 primary + 3 replicas).
- Business tier: dedicated Redis shard (3 nodes, 1 primary + 2 replicas), shared application servers with QoS weights, shared MySQL cluster with dedicated connection pool.
- Enterprise tier: dedicated application servers (auto-scaled), dedicated Redis cluster (6 nodes, 3 primary + 3 replicas), dedicated MySQL read replicas (2 per enterprise customer, replicating from the shared primary).

SLA-Aware Routing:
The CDN or API gateway inspects the first character of the short code path to determine the tenant tier. The tenant tier mapping is stored in a configuration database with a local cache on each API gateway node:

```python
def route_request(short_code: str):
    prefix = short_code[0]
    tier_config = tenant_cache.get(prefix)
    if not tier_config:
        tier_config = db.query(
            "SELECT tier, dedicated_cluster_endpoint FROM tenant_config "
            "WHERE prefix = %s", prefix
        )
        tenant_cache.set(prefix, tier_config)
    if tier_config["tier"] == "enterprise":
        forward_to_backend(tier_config["dedicated_cluster_endpoint"])
    elif tier_config["tier"] == "business":
        forward_to_backend("business-app-pool", qos_weight=10)
    else:
        forward_to_backend("free-app-pool", qos_weight=1)
```

QoS at the application layer uses weighted fair queuing. Business tier requests are dispatched with weight 10, free tier requests with weight 1. A business tier request gets 10x the processing capacity of a free tier request.

Billing and Quota Enforcement:
Quota counting uses Redis counters with a sliding window. For the free tier:

```python
def check_create_quota(tenant_id: str, tier: str):
    if tier == "enterprise":
        return True

    monthly_key = f"quota:monthly:{tenant_id}:{datetime.now().strftime('%Y%m')}"
    monthly_count = redis.incr(monthly_key)

    if tier == "free" and monthly_count > 1000:
        return False

    minute_key = f"quota:minute:{tenant_id}:{datetime.now().strftime('%Y%m%d%H%M')}"
    minute_count = redis.incr(minute_key)
    redis.expire(minute_key, 120)

    limits = {"free": 10, "business": 1000}
    if minute_count > limits.get(tier, float("inf")):
        return False

    return True
```

When a customer exceeds their monthly quota, the create endpoint returns 403 Forbidden with a message explaining the limit and a link to upgrade. Existing short URLs continue to serve redirects even when the quota is exceeded. Only new creates are blocked.

At the end of the billing cycle, the monthly counters are reset. A billing job runs at midnight on the 1st of each month, resetting all monthly quota counters. If a customer upgraded mid-cycle, their counter is adjusted: new limit minus the current count equals remaining allowance for the rest of the cycle.

Failure Isolation:
Free-tier cache failure: each Redis shard is labeled with its tier. A free-tier shard failure does not affect business or enterprise shards because they are on different Redis nodes (or different clusters entirely). The application uses separate Redis connection pools per tier, so a connection timeout to a free-tier shard does not block business-tier traffic.

Enterprise database replica failure: enterprise customers have two dedicated replicas. If one fails, the application routes all read queries to the remaining replica. The 99.995 percent SLA allows 26.3 seconds of downtime per week. A single replica failure is detected within 5 seconds by the health checker. The remaining replica handles the load while a replacement replica is provisioned from the database snapshot (approximately 15 minutes). If both replicas fail simultaneously, the application falls back to the shared database cluster with degraded latency but continued availability.

**Common Mistakes:**
- Using tenant_id only as a column without prefix-based routing. Without prefix routing, the API gateway cannot distinguish tenant tiers from the URL path alone. Every request must go to a centralized routing service, adding latency.
- Sharing Redis connection pools across tiers. A connection pool timeout for one tenant blocks connection acquisition for other tenants. Use separate connection pools per tier.
- Not resetting quotas at the billing cycle boundary with proper handling for mid-cycle upgrades. A customer who upgrades on day 15 should not have to wait until day 30 for their new quota to apply.

### Challenge 8: Global Real-Time URL Analytics Pipeline

**Prompt:** Design a real-time analytics pipeline for a URL shortener that processes 1 trillion redirects per month across 6 geographic regions. The pipeline must:

- Process 385,000 click events per second (average), 1 million per second peak
- Display per-code click counts within 5 seconds of the redirect
- Display geographic distribution within 30 seconds
- Support drill-down queries: total clicks, clicks by country, clicks by city, clicks by device type, clicks by referrer domain, clicks over time (1-minute, 1-hour, 1-day granularity)
- Store raw events for 7 days, aggregated data for 2 years
- Handle late-arriving events (up to 5 minutes delayed) without double-counting
- Provide per-customer analytics isolation for the multi-tenant system from Challenge 7
- Cost: under $0.01 per 10,000 events processed

**Ingestion Layer (10 minutes):** Design the event ingestion layer. Each click event is 200 bytes. At 385,000 events per second (77 MB/s, 6.7 TB/day), how do you ingest, buffer, and distribute events? Design the Kafka or equivalent topology. How many partitions? What is the replication factor? How do you handle a regional Kafka cluster failure?

**Stream Processing (10 minutes):** Design the stream processing layer. Events must be deduplicated, enriched with geographic and device data, and aggregated into time-bucketed counts. Should you use Kafka Streams, Apache Flink, Apache Spark Streaming, or a custom processor? Justify your choice based on latency requirements, state management, and exactly-once semantics.

**Aggregation and Storage (10 minutes):** Design the aggregation and storage architecture. Click counts must be queryable at multiple granularities (1-minute, 1-hour, 1-day). Geographic data requires hierarchical aggregation (country -> region -> city). Design the data model for each aggregation level and calculate the storage requirement for 2 years.

**Late-Arriving Events (5 minutes):** A click event may arrive 30 seconds to 5 minutes after the actual redirect due to network buffering, mobile device offline queues, or Kafka consumer lag. Design the late-event handling strategy. How do you update already-computed aggregates without double-counting?

**Cost Optimization (5 minutes):** The pipeline must operate at under $0.01 per 10,000 events. Calculate the monthly processing cost at 1 trillion events per month. Identify the top cost drivers and design optimizations for each.

**Model Answer:**

Ingestion Layer:
Use Kafka with 256 partitions, 3x replication factor. Each partition handles approximately 1,500 events per second at peak (385,000 / 256 = 1,504). A single Kafka broker handles 10,000 events per second comfortably, so 256 partitions across 16 brokers (16 partitions per broker) provides headroom.

Events are produced with the short code as the partition key. This ensures all events for the same short code go to the same partition, enabling ordered processing per code. The partition key is a hash of the short code: `abs(hash(short_code)) % 256`.

Regional event flow: each region has a local Kafka cluster. Events are produced to the local cluster. A Kafka MirrorMaker 2 instance replicates events from all regional clusters to a central aggregation cluster. This prevents WAN latency from affecting event production: the redirect engine writes to a local Kafka broker in under 5ms.

Stream Processing:
Use Apache Flink with exactly-once semantics. Flink provides the required state management (windowed aggregation with 5-minute session windows for late events), exactly-once processing via checkpointing to S3, and sub-second processing latency for the 5-second freshness requirement.

The Flink job topology:
1. Source: Kafka consumer reading from the 256 partitions.
2. FlatMap: deserialize, validate, discard malformed events.
3. KeyBy short_code.
4. Deduplication using event_id with a 10-minute TTL state store (RocksDB-backed, approximately 1 TB for 7 days of event IDs).
5. AsyncIO enrichment: GeoIP lookup (MaxMind database, 500 MB, loaded into memory per task manager), device classification (regex-based, 10 MB).
6. Window: 1-minute tumbling windows with allowed lateness of 5 minutes.
7. Aggregate: count, country histogram, device histogram, referrer domain histogram.
8. Sink: write aggregated results to ClickHouse, raw events to Parquet files in S3.

Aggregation and Storage:
Three-tier aggregation model:

Tier 1 (hot, 7 days): Per-minute counts in ClickHouse.
```sql
CREATE TABLE clicks_per_minute (
    short_code String,
    timestamp DateTime,
    country FixedString(2),
    device_type LowCardinality(String),
    referrer_domain String,
    count UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(timestamp)
ORDER BY (short_code, timestamp);
```

Tier 2 (warm, 90 days): Per-hour counts. Rolled up from per-minute data every hour.
```sql
CREATE TABLE clicks_per_hour (
    short_code String,
    timestamp DateTime,
    country FixedString(2),
    device_type LowCardinality(String),
    count UInt64
) ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(timestamp)
ORDER BY (short_code, timestamp);
```

Tier 3 (cold, 2 years): Per-day counts. Rolled up from per-hour data daily.

Storage: Per-minute data for 7 days: 385,000 events/s * 60 seconds * 60 minutes * 24 * 7 = 2.3 trillion raw events. Aggregated to per-minute per-code: with 10 million active codes, each producing 1,440 rows per day (one per minute) = 14.4 billion rows per day for the per-minute table. At 100 bytes per row: 1.4 TB per day. This is too large. Optimize by only storing per-minute data for the top 100,000 codes by traffic. For all other codes, aggregate directly to per-hour. This reduces per-minute storage by a factor of 100.

Late-Arriving Events:
Flink handles late events through allowed lateness and side outputs. Configure the 1-minute tumbling window with `allowedLateness(5 minutes)`. When a late event arrives within that window, Flink triggers an incremental update to the window's aggregate and emits an updated result.

The sink (ClickHouse) uses `ReplacingMergeTree` with a version column. Each update to a per-minute aggregate increments the version. The ClickHouse query reads only the latest version per (short_code, timestamp) pair, ignoring stale updates.

For events arriving more than 5 minutes late, Flink sends them to a side output. A separate hourly job processes the side output and applies corrections to the hourly aggregates. This job runs every 6 hours and processes batches of late events, updating the hourly counts using a differential approach: subtract the duplicate events from the first count, add the corrected count.

Cost Optimization:
Monthly cost calculation at 1 trillion events:
- Kafka cluster: 16 brokers at $1,200/month = $19,200. Cost per 10,000 events: $19,200 / (10^12 / 10^4) = $0.000192.
- Flink cluster: 100 task managers at $800/month = $80,000. Cost per 10,000 events: $0.0008.
- ClickHouse storage: 7-day raw (40 TB compressed) + 90-day warm (80 TB compressed) + 2-year cold (200 TB compressed) = 320 TB. At $0.08/GB-month for EBS + $0.01/GB-month for S3: approximately $25,600/month. Cost per 10,000 events: $0.000256.
- Event enrichment databases: GeoIP MaxMind license $2,000/month. Negligible per-event cost.
- S3 Parquet storage: 7 days of raw events, compressed to 50 bytes per event = 50 TB. At $0.023/GB = $1,150/month. Cost per 10,000 events: $0.0000115.

Total: $19,200 + $80,000 + $25,600 + $2,000 + $1,150 = $127,950 per month. Cost per 10,000 events: $0.00128. Well under the $0.01 target.

Top cost driver: Flink compute at $80,000/month (63 percent of total cost). Optimization: use spot instances for non-critical task managers (70 percent discount, reducing compute cost to $24,000/month). Use reserved instances for the stateful task managers that hold the deduplication state store.

**Common Mistakes:**
- Not accounting for the 5-minute allowed lateness in the window design. If late events are simply discarded, click counts are underreported by 0.5-2 percent, which is unacceptable for enterprise analytics.
- Storing raw events in a queryable database. Raw events at 385,000/s generate 2.3 TB per day uncompressed. Storing these in ClickHouse or PostgreSQL would cost $5,000+/day. Compress and store raw events in Parquet on S3 for $10/day. Query the aggregated tables, not the raw events.
- Using the same aggregation granularity for all short codes. The top 0.1 percent of codes generate 80 percent of traffic. Storing per-minute data for the 99.9 percent of codes that each receive fewer than 100 clicks per day wastes storage. Throttle aggregation granularity based on traffic volume.
