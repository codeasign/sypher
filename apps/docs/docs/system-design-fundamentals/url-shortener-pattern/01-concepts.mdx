## Problem Story

Imagine you run a social media platform that allows 140 characters per post. A user wants to share an article whose URL is 180 characters long: `https://www.example.com/blog/2025/07/top-ten-strategies-for-optimizing-cloud-native-microservice-deployments-with-kubernetes-and-docker-containers-in-production-environments`. That single link consumes the entire tweet with 40 characters to spare. The user pastes the link, the preview API fetches the metadata, and the service truncates the message. The post never sends.

Now multiply this by ten million users sharing links every day. Your platform stores billions of full-length URLs in the database, each one consuming storage, slowing down reads, and inflating backup windows. Spam detection systems must scan the full URL path for malicious patterns. Analytics pipelines parse character-by-character to classify link categories. The sheer **weight** of raw URLs becomes an infrastructure tax on every subsystem that touches them.

This is not just a character-count problem. It is a **data compression, storage optimization, and lookup performance** problem disguised as a user experience annoyance. Every system that handles shared links faces three pressures:

Consider a concrete example. A marketing campaign sends 10 million email messages, each containing a 220-character tracking URL. The email itself is 600 characters. The tracking URL consumes 37 percent of the message payload. The email provider charges by volume. The tracking parameters in the URL expose campaign details: `utm_source`, `utm_medium`, `campaign_id`, `user_id`, `session_token`. Anyone who intercepts the email learns the campaign structure and the user's identity within the system.

A URL shortener reduces that 220-character URL to 24 characters. The email payload drops by 33 percent. The tracking parameters are hidden behind an opaque short code. The campaign structure is invisible to anyone who sees the email. The shortener also centralizes click analytics -- instead of parsing query parameters across 10 million email clients, the system reads click events from a single analytics pipeline.

The problem appears in many contexts beyond social media:

- **Messaging platforms**: SMS charges by character. A 200-character URL turns a single-segment message into a multi-segment message at double the cost. Shortening the URL keeps it in a single segment.

- **Print media**: QR codes on billboards, bus stops, and magazines must encode a URL that humans can also type manually. A 24-character short URL is practical for both scanning and manual entry.

- **Voice assistants**: Smart speakers read URLs aloud. A short URL like "short dot link slash a b three x k nine" is pronounceable. A long URL with query parameters is not.

- **API payloads**: REST APIs that include resource links in responses benefit from shorter representations. A paginated response with 100 items each carrying a 200-character self-link adds 20 KB of overhead. Short URLs reduce that to 2.4 KB.

- **Database storage**: Variable-length text columns in MySQL with character set utf8mb4 consume 4 bytes per character. A 200-character URL consumes 800 bytes per row. With one billion rows, that is 800 GB for URLs alone. A 24-character short code consumes 96 bytes per row.

- **CDN costs**: When a short URL is shared on a social platform, the platform's preview service fetches the destination URL to generate a link preview. The shortener's redirect is cached by the CDN. The origin server handles fewer requests.

The URL shortener pattern addresses all three pressures. It transforms a long, variable-length, privacy-sensitive string into a short, fixed-size, opaque identifier. The mapping is stored in a database, and the short code is what travels across the wire. The original URL is only resolved at the moment of redirect -- a controlled, auditable moment.

## The Insight

The core insight behind URL shortening is surprisingly simple: **replace a variable-length string with a fixed-length token in a compact alphabet, and use the token as a primary lookup key.** The token must be globally unique within the system, short enough to save space, and generated without coordinating with every other node in real time.

Think of a coat check in a busy restaurant. You hand over your heavy winter coat (the long URL). The attendant hangs it on a numbered hook (the database row) and hands you a small ticket with a number on it (the short code). Later, you return the ticket, the attendant looks up hook 42, and hands back your coat. The ticket is tiny. It does not reveal what the coat looks like or who it belongs to. And the restaurant issues thousands of tickets without any two being the same.

The URL shortener works the same way. The long URL is stored in a database row keyed by a unique identifier. That identifier is encoded into a short string using an alphabet that avoids ambiguous characters -- no `0` versus `O`, no `1` versus `l` or `I`. The encoded string is what you share. When someone visits the short URL, the system decodes the identifier, looks up the row, and issues an HTTP redirect to the original location.

The magic is in the encoding scheme. A 64-bit integer encoded in base-62 (digits, lowercase letters, uppercase letters) produces a 7-character string that can represent up to 3.5 trillion unique values. That is enough for most systems. With 8 characters, you reach 218 trillion. The math works in your favor because each additional character multiplies the namespace exponentially.

## Core Vocabulary

Before you dive deeper, establish a shared vocabulary. These terms appear throughout every URL shortener discussion. Mastering this vocabulary lets you read any shortener design document and immediately understand the architectural choices.

**Short code** -- The fixed-length token that replaces the long URL. Usually 6 to 8 characters drawn from a base-62 alphabet. Examples: `aB3xK9`, `7qP2mN`.

**Slug** -- Another name for the short code, borrowed from content management systems. Some teams use "slug" and "code" interchangeably.

**Redirect** -- The HTTP response that sends a client from the short URL to the original destination. Two types exist: 301 (permanent, cached by browsers) and 302 (temporary, not cached).

**Base-62 encoding** -- An encoding scheme that uses 62 characters (0-9, a-z, A-Z) to represent numeric identifiers. More compact than decimal or hexadecimal. Each position carries 62 times the weight of the position to its right.

**Base-64 encoding** -- A variant that adds `-` and `_` to the alphabet (URL-safe version) or `+` and `/` (standard version). Slightly denser than base-62 but introduces characters that may need encoding in some contexts.

**Snowflake ID** -- A 64-bit unique identifier generation scheme that splits bits across timestamp, worker ID, sequence number, and a custom epoch. Used by many shortener systems to avoid database round-trips during generation.

**Hash function** -- A deterministic function that maps an input of arbitrary size to a fixed-size output. Used to derive short codes from long URLs. Common choices: MD5, MurmurHash, or a custom hash.

**Collision** -- The event where two different long URLs produce the same short code. Collision handling is a critical design decision.

**Key lookup** -- The database or cache operation that resolves a short code into its corresponding long URL. This operation must complete in single-digit milliseconds.

**TTL** -- Time-to-live. The duration a short code remains valid before expiring. Some systems use permanent TTLs; others use time-based expiration for temporary links.

**Click tracking** -- The system that records every redirect event for analytics. Includes timestamp, referrer, user agent, geographic location, and device type.

**Redirect chain** -- A sequence of HTTP redirects that may include intermediate hops for analytics, click counting, or bot filtering before reaching the final destination.

**Link preview** -- Metadata (title, description, image) that social platforms and messaging apps fetch from the destination URL before displaying a shared link.

## How It Works

### Step 1: User submits a long URL

A user sends a POST request to `/shorten` with a JSON body containing the long URL. The API layer validates the URL: it must be well-formed, use HTTP or HTTPS, and not point to a known malicious domain. The service also checks whether this exact URL was already shortened by the same user or within the system. If deduplication is enabled, an existing short code is returned instead of creating a new one.

The API request and response format typically looks like this:

```json
POST /api/shorten
Content-Type: application/json

{
  "url": "https://www.example.com/blog/2025/07/top-ten-strategies",
  "custom_slug": null,
  "ttl_seconds": null
}

Response 200:
{
  "short_code": "aB3xK9",
  "short_url": "https://short.link/aB3xK9",
  "long_url": "https://www.example.com/blog/2025/07/top-ten-strategies",
  "created_at": "2025-07-03T12:00:00Z",
  "expires_at": null
}
```

The API layer performs five validations before proceeding:

1. **URL format**: The input must be a valid URL with a scheme (http or https). The system uses a standard URL parser to reject malformed strings.
2. **Length limit**: The maximum URL length is typically 2048 characters, matching the practical limit for most web servers and browsers.
3. **Blocklist check**: The domain is checked against a blocklist of known phishing, malware, and spam hosts. This check uses a Bloom filter for fast lookup.
4. **Deduplication check**: If the system is configured for deduplication, the service computes a hash of the normalized URL (lowercased, trailing slash stripped, query parameters sorted) and checks for an existing mapping.
5. **Rate limiting**: The user's IP address and account identifier are checked against rate limits. Typical limits are 100 shorten requests per minute per user.

If any validation fails, the API returns an appropriate error response with a descriptive message. The client must handle these errors and retry with corrected input.

### Step 2: Key generation runs

The key generation engine produces a unique identifier. Three strategies dominate, each with different trade-offs for determinism, coordination, and throughput.

**Hash-based**: Compute a hash of the long URL (MD5, SHA-256, or MurmurHash) and take the first N bytes of the digest. Encode those bytes in base-62. This is deterministic -- the same URL always produces the same short code. If a collision occurs, append a salt or increment a counter until the code is unique.

The hash-based approach is the most common choice for small to medium systems because it requires no coordination between servers. Each server can independently compute the short code for any URL. The downside is that hash collisions, while rare, require a retry loop, and the hash function may produce biased output depending on the input distribution.

```python
import hashlib
import base62

def generate_short_code(url: str, length: int = 7) -> str:
    digest = hashlib.md5(url.encode()).digest()
    # Take first 4 bytes (32 bits) of the digest
    value = int.from_bytes(digest[:4], byteorder='big')
    return base62.encode(value)[:length]
```

**Sequential**: Maintain a global counter in a database or a service like Redis. Issue an INCR command to get the next integer. Encode that integer in base-62. This produces shorter codes for earlier entries, guarantees uniqueness without collision checks, and produces monotonically increasing codes that are friendly to B-tree index performance.

The sequential approach is the simplest to implement correctly. The counter becomes a single point of contention at high throughput, but this can be mitigated with batch allocation. The sequential order also leaks information -- a competitor can estimate your total number of shortened URLs by creating a few links and observing the codes.

```python
import redis
import base62

def generate_sequential_code(redis_client: redis.Redis) -> str:
    next_id = redis_client.incr('url_shortener:counter')
    return base62.encode(next_id)
```

**Snowflake-based**: Generate a 64-bit ID from a Snowflake-style worker. The ID embeds a timestamp (41 bits), a worker identifier (10 bits), and a sequence number (12 bits). Encode the ID in base-62. This is unique across workers without coordination and is time-ordered, which helps with B-tree index performance.

The Snowflake approach is the most scalable for distributed systems. Each worker generates IDs independently without any network round-trip, supporting up to 4096 IDs per millisecond per worker. The trade-off is clock dependency -- if the system clock drifts backward, IDs may collide.

```python
import time
import base62

class SnowflakeGenerator:
    def __init__(self, worker_id: int, epoch: int = 1700000000000):
        self.worker_id = worker_id
        self.epoch = epoch
        self.sequence = 0
        self.last_timestamp = -1

    def next_id(self) -> int:
        timestamp = int(time.time() * 1000) - self.epoch
        if timestamp == self.last_timestamp:
            self.sequence = (self.sequence + 1) & 0xFFF
        else:
            self.sequence = 0
            self.last_timestamp = timestamp
        return (timestamp << 22) | (self.worker_id << 12) | self.sequence

    def generate_code(self) -> str:
        return base62.encode(self.next_id())
```

### Step 3: Collision check

For hash-based generation, the system checks whether the short code already exists in the database. If it does, the system compares the associated long URL. A match means the URL was already shortened -- return the existing code. A mismatch means a collision -- append a salt and regenerate the hash. Sequential and Snowflake strategies skip this step because they guarantee uniqueness by construction.

The collision check is a critical performance consideration. For a 7-character base-62 code, the probability of a collision after N entries follows the birthday paradox approximation:

```
P(collision) ~= N^2 / (2 * 62^7)
```

With 1 million entries, the collision probability is approximately 0.00014 (1 in 7,000). With 10 million entries, it rises to approximately 0.014 (1 in 70). At 100 million entries, it reaches approximately 0.93 (essentially guaranteed). This means that hash-based shorteners must handle collisions as a normal operational concern, not an edge case.

The collision resolution strategy typically uses a technique called "double hashing": append a counter to the input URL and rehash until a unique code is found:

```python
def generate_unique_code(url: str, db) -> str:
    code = generate_short_code(url)
    attempt = 0
    while db.exists(code):
        if db.lookup(code) == url:
            return code  # Duplicate, return existing
        attempt += 1
        code = generate_short_code(f"{url}:{attempt}")
    return code
```

### Step 4: Persist the mapping

The service writes a row to the database: short code, long URL, creation timestamp, optional expiration time, and an owner identifier. The write goes to the primary database node. A successful write confirms the short code is now permanently associated with the long URL.

The database schema uses the short code as the primary key for optimal lookup performance:

```sql
CREATE TABLE url_mappings (
    short_code VARCHAR(8) PRIMARY KEY,
    long_url VARCHAR(2048) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP NULL,
    owner_id VARCHAR(64) NULL,
    click_count BIGINT NOT NULL DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    INDEX idx_expires_at (expires_at),
    INDEX idx_owner_id (owner_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

The write transaction must be durable. Most systems use synchronous replication to at least one replica before acknowledging the write to the client. This ensures that a database node failure does not lose the mapping.

### Step 5: Populate the cache

The service writes the mapping to the cache layer (typically Redis or Memcached) with the short code as the key and the long URL as the value. The cache entry includes a TTL that determines how long the mapping stays hot. Popular links remain cached for hours or days.

Cache writes use a write-through pattern: the service writes to both the database and the cache within the same request. This ensures that the cache is always populated for newly created links, preventing a cache-miss stampede when the first visitor arrives.

The cache key format is simple: `short:{code}`. The value is the long URL. The TTL varies by link type:
- **Default links**: 24 hours TTL, refreshed on each access
- **Popular links** (above 1000 clicks/hour): 7 days TTL
- **Custom slug links**: 48 hours TTL
- **Expiring links**: TTL matches the remaining link lifetime

### Step 6: Return the short URL

The service constructs the complete short URL by combining the base domain with the short code: `https://short.link/aB3xK9`. It returns this URL to the client as a JSON response. The full round trip -- from submission to response -- should complete in under 50 milliseconds.

The response includes metadata that the client can use for display and debugging:

```json
{
  "short_code": "aB3xK9",
  "short_url": "https://short.link/aB3xK9",
  "long_url": "https://www.example.com/blog/2025/07/top-ten-strategies",
  "created_at": "2025-07-03T12:00:00Z",
  "expires_at": null,
  "is_new": true
}
```

The `is_new` flag distinguishes between a newly created mapping and a returned duplicate. This lets the client show different UI messages for each case.

### Step 7: User shares the short URL

The short URL travels across email, SMS, social media, messaging apps, or print media. It is 30 to 40 characters instead of 150 to 300. It fits anywhere. It reveals nothing about the destination. It is indistinguishable from any other short URL.

The short URL behaves differently depending on the sharing channel:

- **Email clients**: Many email clients prefetch all links in a message to generate link previews and check for malicious URLs. This means the short URL receives its first click (a prefetch) within seconds of the email being opened, even if the human recipient never clicks the link. Analytics systems must distinguish between prefetch events and human clicks by examining the user agent header.

- **SMS and messaging apps**: Most messaging apps do not prefetch links on receipt. The first click is the human tapping the link. This provides more accurate click data but also means the first visitor experiences a cache miss and a 2-5 millisecond database lookup.

- **Social media platforms**: Platforms like Twitter, LinkedIn, and Facebook fetch link previews by following the redirect and reading the destination page's Open Graph meta tags. The preview fetch counts as a click event, inflating click counts. Some shorteners exclude known bot user agents from the click count.

- **Print media**: QR codes and printed short URLs have no click feedback. The creator cannot know how many times the printed code is scanned unless the shortener's analytics pipeline captures the redirect events. The short URL must be stable and never expire for print use cases.

### Step 8: Visitor clicks the short URL

A user clicks the short URL in their browser. The browser sends a GET request to the shortener service at `https://short.url/aB3xK9`. The service parses the path, extracts the short code, and initiates the redirect flow.

The DNS resolution for the short domain is typically handled by a CDN provider that also provides DDoS protection. The request hits the CDN edge node closest to the visitor. If CDN caching is enabled and this short URL has been visited before, the CDN may serve a cached redirect response without forwarding to the origin server.

### Step 9: Cache lookup

The service first checks the cache. The cache key is the short code. If the cache contains the mapping, the service reads the long URL from the cache in under 1 millisecond and proceeds to the redirect step. This is the fast path and handles 90 percent or more of requests for popular links.

The cache lookup is a single Redis GET command. The response is either the long URL (cache hit) or a nil response (cache miss). The service uses connection pooling to minimize TCP connection overhead. Each application server maintains a pool of 10-50 persistent connections to the Redis cluster.

Cache hit latency breakdown:
- Redis round-trip: 0.3-0.8 milliseconds
- Response parsing: 0.1-0.2 milliseconds
- Total: 0.4-1.0 milliseconds

The service uses a consistent hashing scheme to distribute cache keys across Redis cluster nodes. This ensures that the cache lookup for a given short code always hits the same node, maximizing cache locality.

### Step 10: Database lookup (cache miss)

If the cache returns a miss, the service queries the database. The query uses the short code as the primary key -- a B-tree index makes this a single lookup, typically 2 to 5 milliseconds. If the row exists, the service populates the cache for future requests and proceeds. If the row does not exist, the service returns a 404 response.

The database query is a simple primary key lookup:

```sql
SELECT long_url, expires_at, is_active
FROM url_mappings
WHERE short_code = 'aB3xK9'
LIMIT 1;
```

The query plan for this operation uses a primary key index lookup, which is the fastest possible access path in InnoDB. The database reads a single page (16 KB) from the clustered index, which may require one or two disk I/O operations if the page is not in the buffer pool.

After the database returns the row, the service populates the cache asynchronously using a write-behind pattern. The cache entry is set with the configured TTL to prevent the next request from hitting the database again. For a link that receives 1000 requests per second, the cache saves 999 database queries per second after the first miss.

### Step 10b: Cache miss stampede protection

A cache miss stampede occurs when a popular short URL expires from the cache and thousands of requests simultaneously hit the database before any of them repopulate the cache. This can overwhelm the database.

Protection strategies include:
- **Early expiration**: Refresh the cache entry before it expires when the link is still receiving requests. This is known as "read-through with proactive refresh."
- **Mutex on cache miss**: Only one request per short code performs the database lookup. Other requests wait for the first request to populate the cache. This is implemented with Redis SET NX with a short TTL.
- **Renewal queue**: When a cache entry expires, the request enqueues a background job to refresh the cache. The current request waits for the job to complete. Subsequent requests wait for the same job.

### Step 11: Enrich the redirect

Before sending the redirect, the service may enrich the response. It checks for link expiration and returns a 410 Gone if the link expired. It may perform bot detection to avoid counting automated requests as clicks. It may add tracking parameters to the destination URL for analytics on the destination side.

Enrichment includes these checks run in order:

1. **Expiration check**: If `expires_at` is set and the current time is past the expiration, return HTTP 410 Gone. The response body includes a message indicating the link has expired.
2. **Active flag check**: If `is_active` is false, return HTTP 410 Gone. This allows manual deactivation of links.
3. **Bot detection**: The user agent is checked against a list of known bots (Googlebot, Twitterbot, Facebook crawler, Bingbot, etc.). Bot clicks are counted separately from human clicks in the analytics pipeline.
4. **Rate limiting per code**: If a single short code exceeds 10,000 requests per second, the system may rate-limit the redirect to protect downstream infrastructure. This is rare but necessary for viral links.
5. **Destination health check**: For premium users, the system may verify the destination URL is reachable before redirecting. If the destination returns 5xx, the system returns a 503 with a message indicating the destination is unavailable.

### Step 12: HTTP redirect

The service sends an HTTP response with status code 301 (Moved Permanently) or 302 (Found) and a Location header pointing to the original URL. A 301 response tells browsers to cache the redirect -- subsequent visits skip the shortener entirely. A 302 response tells browsers to check the shortener each time, which supports click tracking and rotating analytics.

The full HTTP response for a 302 redirect:

```
HTTP/1.1 302 Found
Location: https://www.example.com/blog/2025/07/top-ten-strategies
Cache-Control: no-cache, no-store, must-revalidate
Content-Type: text/plain
Content-Length: 0
```

For a 301 redirect:

```
HTTP/1.1 301 Moved Permanently
Location: https://www.example.com/blog/2025/07/top-ten-strategies
Cache-Control: public, max-age=31536000, immutable
Content-Type: text/plain
Content-Length: 0
```

The choice between 301 and 302 has significant operational implications:
- **301 (permanent)**: Browsers and CDNs cache the redirect for up to one year. The shortener serves fewer requests, reducing infrastructure costs. However, click tracking is inaccurate because repeat visits bypass the shortener entirely. Analytics data is lost for returning visitors.
- **302 (temporary)**: Every visit hits the shortener. Click tracking is accurate. The shortener can update the destination URL dynamically. The trade-off is higher infrastructure cost because every request reaches the origin server.
- **303 (see other)**: Used for POST-redirect-GET patterns. Rarely used in URL shorteners but useful when the short URL is submitted via a form.
- **307 (temporary redirect)**: Preserves the HTTP method. If someone sends a POST to a short URL, the redirect re-issues the POST to the destination. This is important for API use cases.

### Step 13: Analytics pipeline records the click

Whether before or after the redirect, the service emits a click event. The event includes the short code, timestamp, referrer, user agent, IP address, and geographic data. These events flow into a message queue (Kafka, Kinesis, or RabbitMQ) and are processed asynchronously by a stream processor that aggregates counts, generates reports, and feeds analytics dashboards.

In this variant, users specify their own short code instead of receiving a system-generated one. A marketing team might want `https://short.link/black-friday` instead of `https://short.link/x7K2aP`. The system checks availability and reserves the custom slug.

Custom slugs require additional constraints:
- **Uniqueness domain**: Custom slugs and generated slugs share the same namespace. The system must prevent a generated slug from colliding with a reserved custom slug.
- **Length limits**: Most systems restrict custom slugs to 20 to 30 characters to avoid abuse.
- **Profanity filter**: Custom slugs are user-supplied text, requiring validation against offensive or misleading content.
- **Premium tier**: Many services charge for custom slugs and offer them only to paid accounts.

Implementation considerations for time-limited shorteners:
- **Database index**: The `expires_at` column must be indexed for efficient cleanup queries.
- **Cleanup strategy**: Batch delete expired rows nightly or use TTL-based table partitioning.
- **Cache behavior**: Cache entries should use the same TTL as the link itself. Serving a cached redirect for an expired link is a correctness bug.
- **Renewal**: Some systems allow the creator to extend the TTL before expiration, which updates both the database row and the cache entry.

- **Delete-before-redirect**: Delete the row within the same database transaction before sending the redirect response. If the redirect fails, the link is still consumed -- this is intentional to prevent replay attacks.

- **Cleanup sweep**: A background job periodically removes consumed rows that were soft-deleted as a safety net for any missed deletions.

### Variant 4: QR-Code-Optimized Shortener

This variant optimizes the short code for QR code scanning. The alphabet omits characters that are easily confused in low-resolution scans. The code length is fixed at 6 characters to produce the smallest possible QR code payload.

QR codes encode data more efficiently with alphanumeric mode (45 characters per cell) versus byte mode. A 6-character short code in the subset `0123456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghjkmnpqrstuvwxyz` (removing `O`, `0`, `I`, `l`, `1`) produces a QR code version 2 (25x25 cells) that scans reliably even on damaged packaging.

The QR-optimized variant introduces a custom encoding step. The base-57 integer is first converted to a 6-character string in the reduced alphabet. The string is then appended to the short domain. The QR code encodes the complete short URL, which is shorter than encoding the original long URL:

- Long URL in QR: 180 characters = version 4 QR (33x33 cells)
- Short URL in QR: 24 characters = version 2 QR (25x25 cells)

The 33 percent reduction in QR code size means the code occupies less print space and scans at smaller physical sizes. On a business card, a version 2 QR code is 15mm x 15mm while a version 4 code is 20mm x 20mm.

### Variant 5: Parameterized Dynamic Shortener

This variant allows the short URL to accept query parameters that are appended to the destination URL at redirect time. A creator sets the base destination to `https://example.com/landing` and shares the short URL as `https://short.link/camp1?utm_source={source}&user_id={uid}`. When a visitor clicks the short URL, the system parses the query parameters from the short URL, interpolates them into the destination template, and redirects to the complete resolved URL.

### Variant 6: Blockchain-Verified Shortener

This variant records short code mappings on a blockchain for tamper-proof auditability. Each mapping is written to a smart contract that emits an event when the mapping is created or updated. The short code is derived from the blockchain transaction hash, ensuring global uniqueness without a central coordinator.

The blockchain variant trades latency for trust. Creating a short URL takes 10-30 seconds (the time to mine a transaction) instead of 50 milliseconds. The redirect path reads from a centralized cache backed by the blockchain, achieving sub-100-millisecond lookup times while maintaining the audit trail on-chain.

This variant is used for:
- **Content authenticity**: Publishing platforms that need to prove a link was created at a specific time
- **Decentralized applications**: Systems that cannot rely on a single URL shortener service
- **Regulatory compliance**: Industries that require immutable audit logs of all shortened links

## Mental Model Check

Test your understanding with these five questions. Cover the answer and explain it before checking.

**Question 1**: Why do most URL shorteners use exactly 7 characters for base-62 short codes?

<details>
<summary>Answer</summary>

Seven characters of base-62 provide 62^7 = 3.5 trillion combinations. This is enough for most systems to operate for years without exhausting the namespace. Eight characters (62^8 = 218 trillion) is overkill for all but the largest systems. Six characters (62^6 = 56 billion) may be too few for high-volume systems. Seven hits the sweet spot between namespace size and URL brevity.
</details>

**Question 2**: What happens if two users submit the same long URL to a hash-based shortener?

<details>
<summary>Answer</summary>

In a hash-based shortener, the same long URL always produces the same short code. The second submission finds the existing mapping and returns the existing short URL. This is intentional deduplication -- the same long URL maps to exactly one short code. This conserves the key namespace and lets the system detect which URLs are popular based on how often their short codes are retrieved by different users.
</details>

**Question 3**: Why is a 301 redirect dangerous for click tracking?

<details>
<summary>Answer</summary>

A 301 (Moved Permanently) redirect is cached by browsers. On the second visit, most browsers skip the shortener entirely and navigate directly to the destination URL. This means the shortener never sees the second click -- click counts are underreported. A 302 (Found) redirect is not cached, so every visit hits the shortener and gets counted. The trade-off is latency: 301 redirects are faster for repeat visitors, while 302 redirects provide accurate analytics.
</details>

**Question 4**: How does a sequential ID generator handle write contention at high throughput?

<details>
<summary>Answer</summary>

A single INCR command on a Redis key handles up to 100,000 increments per second on a single node. For higher throughput, the system can use segmented counters: each application instance reserves a batch of IDs (for example, 10,000 at a time) from a central counter and distributes them locally. This reduces contention to one reservation per batch per instance rather than one INCR per short code.
</details>

**Question 5**: What is the storage cost of 1 billion short URLs in a relational database?

<details>
<summary>Answer</summary>

Each row stores a 7-character short code (7 bytes), a long URL averaging 200 bytes (up to 2048), a 8-byte timestamp, and a 4-byte expiration. With indexes, each row consumes approximately 300 bytes. One billion rows require roughly 300 GB of raw storage plus 50 percent overhead for indexes and replication, totaling approximately 450 GB. This fits on a single large server but is better distributed across shards for query performance.
</details>

**Question 6**: What happens when a user creates a short URL for a 301-redirected destination?

<details>
<summary>Answer</summary>

The shortener resolves to the intermediate URL, which then issues its own 301 redirect. The visitor experiences a redirect chain: short URL to shortener (302), shortener to intermediate URL (301), intermediate URL to final destination (301). This can add latency and complicate debugging. Some shorteners follow the chain during creation to detect loops and resolve to the final destination directly. Others store as-is and let the chain happen at redirect time.
</details>

**Question 7**: Why do some shorteners use base-57 instead of base-62?

<details>
<summary>Answer</summary>

Base-57 removes five visually ambiguous characters from base-62: `0` (zero), `O` (uppercase oh), `I` (uppercase eye), `l` (lowercase ell), and `1` (one). This makes short codes easier to read aloud, type manually, and recognize in print. The namespace shrinks from 3.5 trillion to approximately 1.9 trillion for 7-character codes, which is still sufficient for most systems.
</details>

**Question 8**: How do you prevent users from shortening URLs to malicious destinations?

<details>
<summary>Answer</summary>

Prevention requires a multi-layer approach. At creation time, check the destination against blocklists of known phishing, malware, and spam domains. Use a URL scanner (Google Safe Browsing API, VirusTotal) to classify the destination. If the URL is suspicious, flag it for manual review or require the user to authenticate. At redirect time, check the destination again -- a URL that was safe at creation time may have been compromised. Some systems require all shortened links to be reviewed before they become active, accepting the delay for security.
</details>

**Question 9**: How does a shortener handle 1 million redirects per second during a viral event?

<details>
<summary>Answer</summary>

A short URL going viral creates a hot key problem. One short code receives millions of requests per second, overwhelming a single cache node. The solution combines several techniques: pre-warm the cache as soon as the link is created, use a replicated cache with multiple read replicas, shard the cache key across nodes, and apply aggressive CDN caching with a 301 redirect for maximum browser caching. The redirect itself should be a static response served from the CDN edge, not from the application server. For extreme cases, generate a signed URL that bypasses the shortener entirely.
</details>

**Question 10**: What is the difference between base-62 encoding and base-64 URL-safe encoding?

<details>
<summary>Answer</summary>

Base-62 uses digits (0-9), lowercase (a-z), and uppercase (A-Z). Base-64 URL-safe uses the same 62 characters plus `-` and `_`. Base-64 packs 6 bits per character versus base-62's 5.95 bits. For 7-character codes, base-62 represents 3.5 trillion values; base-64 URL-safe represents 4.4 trillion values (64^7). The difference is marginal at 7 characters but becomes more significant at shorter code lengths. For a 5-character code, base-64 (1.07 billion) offers nearly double the capacity of base-62 (916 million).
</details>

## Summary

The URL shortener pattern solves a deceptively simple problem with deep architectural implications. What looks like a character-count fix reveals itself as a lesson in key generation, database indexing, caching strategy, and distributed system design. The core abstractions -- short code, base-62 encoding, redirect, analytics pipeline -- appear across web frameworks, content management systems, marketing platforms, and security tools.

Key takeaways from this mental model:

- **The namespace is exponential**: Each additional character multiplies the addressable space by the alphabet size. Six characters is enough for most systems; seven characters is comfortable; eight characters is future-proof.
- **Key generation strategy drives everything**: Sequential counters are simple but create a single point of contention. Hash-based generation is deterministic but requires collision handling. Snowflake IDs are decentralized but add clock dependency.
- **The cache is not optional**: A shortener without a cache fails under load. The cache absorbs 90 percent or more of read traffic and protects the database from redirect storms.
- **Analytics must be async**: Click events should never block the redirect path. Decouple analytics collection from redirect delivery using a message queue.
- **Security is multi-layer**: Blocklists at creation time, rechecking at redirect time, and abuse detection on the analytics pipeline form a defense-in-depth strategy.
