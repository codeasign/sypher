## Naive Implementation

Every dashboard platform starts simple. A single server with a time series database, a query API, and a web frontend. This works for small teams, but breaks at predictable thresholds.

### 1,000 Users — Single Server

At 1,000 daily active users, a single server running Prometheus, Grafana, and PostgreSQL on an 8-core machine with 32GB RAM and 1TB SSD handles the load without breaking a sweat. The time series database ingests approximately 100,000 samples per second and serves 50 dashboard queries per second. Memory utilization sits at 40%, CPU at 20%. The entire system costs approximately $200 per month in cloud infrastructure.

### 100,000 Users — Bottlenecks Emerge

At 100,000 daily active users, the single server shows its limits. The time series database now ingests 5 million samples per second and handles 5,000 dashboard queries per second. The disk I/O queue fills: the same SSD that is writing incoming metrics must also read historical data for queries. Read latency for dashboard queries climbs from 50ms to 800ms. CPU utilization hits 85% at peak, driven primarily by label index lookups and query aggregation.

The server experiences its first out-of-memory kill when a dashboard with 200 panels loads simultaneously with a 30-day query that tries to scan a billion data points. The operations team adds swap space, which makes the dashboard so slow that engineers stop using it. The on-call rotation now relies on terminal-based alerts instead of charts. The system is functional but painful.

### 10 Million Users — Complete Breakdown

At 10 million daily active users, the single server architecture collapses. The ingestion rate exceeds 500 million samples per second — far beyond what a single machine can handle. Disk I/O becomes the bottleneck: 50 SSDs worth of write throughput funneled through one PCIe bus. Query latency is measured in minutes, not milliseconds. Dashboard pages time out before they finish loading.

The naive architecture fails because it violates three principles of distributed systems: separation of concerns (ingestion and querying compete for the same resources), horizontal scalability (one server cannot grow), and fault isolation (a single bad query can crash the entire system).

## Production-Grade Architecture

A production dashboard platform separates each concern into its own tier. The architecture uses four tiers: ingestion, storage, query, and presentation. Each tier scales independently and is isolated from failures in other tiers.

## Service-Level Objectives (SLOs) as Design Drivers

SLOs should drive the design of the dashboard platform, not just the metrics displayed on it. When you define SLOs for the dashboard platform itself, you make explicit what level of service the operations team can expect.

### Dashboard Platform SLOs

- **Availability SLO: 99.9%.** The dashboard platform should be available 99.9% of the time, which allows approximately 8.7 hours of downtime per year. This SLO is lower than the services it monitors because the dashboard platform is a tool, not the system itself. If the dashboard platform is down, the services continue running; the team just cannot see the metrics visually. Alerts still fire through the alerting system.

- **Query latency SLO: p99 under 2 seconds.** The 99th percentile of dashboard queries should complete in under 2 seconds. This ensures that the dashboard loads in a reasonable time for 99% of viewers. The remaining 1% of queries (complex ad-hoc queries over long time ranges) are expected to take longer and are not subject to this SLO.

- **Data freshness SLO: data is at most 60 seconds old.** The most recent data point on any dashboard panel should be no more than 60 seconds old. This ensures that the on-call engineer is looking at current data, not stale data. The freshness is driven by the scrape interval (15 seconds) plus the query and rendering time (5-15 seconds), so the 60-second SLO is achievable with room to spare.

- **Storage durability SLO: 99.9999999% (11 nines).** Metric data should not be lost. The WAL and replication ensure that data survives disk failures, node crashes, and region failures. The 11-nines target means that if you store 1 billion data points, you expect to lose fewer than 1.

### SLO Monitoring for the Dashboard Platform

The dashboard platform monitors its own SLOs using a separate, minimal monitoring stack. This stack consists of a single Prometheus instance and a single Grafana instance running on different infrastructure (different cloud account, different region). The meta-monitoring stack tracks:

- Dashboard platform availability (from synthetic checks)
- Dashboard query latency (p50, p95, p99 from query API metrics)
- Data freshness (time since last successful scrape for each target)
- Storage usage (disk, memory, series count)

The meta-monitoring dashboard is the only dashboard that does not run on the dashboard platform. It is the fallback that the on-call engineer uses when the main dashboard platform is unavailable.

## Integration Patterns

### Pattern 1: Metrics Database (Primary Storage)

All metric data is stored in a purpose-built time series database (Prometheus, VictoriaMetrics, Thanos, InfluxDB). The TSDB handles the write path, compression, rollups, and range queries. This is the default integration pattern because every dashboard panel ultimately reads from the TSDB.

```
Pros: Optimized for time series workloads, excellent compression,
  efficient range scans, built-in rollups
Cons: Limited to numeric data, no native support for logs or traces,
  requires separate infrastructure for high cardinality

Integration: Dashboard API → PromQL query → TSDB → JSON response
```

### Pattern 2: Cache Layer (Query Acceleration)

A Redis or Memcached layer sits between the dashboard API and the TSDB. Each unique query result is cached with a TTL equal to the dashboard's refresh interval. Cache hits return in under 1 millisecond, compared to 50-500ms for TSDB queries. The cache also absorbs duplicate queries from multiple users viewing the same dashboard.

### Pattern 3: Message Queue (Ingestion Buffer)

A Kafka or RabbitMQ queue sits between the scrape targets and the TSDB ingestion layer. The queue acts as a shock absorber: when the TSDB is slow to flush (compaction running, GC pause), the queue buffers incoming metrics instead of dropping them. The queue also enables multiple consumers: one consumer writes to the hot TSDB, another writes to a cold storage archive, a third computes real-time rollups.

```
Pros: Decouples ingestion from storage, absorbs write spikes,
  enables multiple consumers, replay on failure
Cons: Adds 50-200ms of latency to the ingestion path,
  requires Kafka cluster management

Integration: Service → Metrics Agent → Kafka Topic → TSDB Ingestion
```

### Pattern 4: CDN (Static Asset Delivery)

The dashboard web application (HTML, JavaScript, CSS, fonts, images) is served from a CDN, not from the dashboard server. CDN edge nodes cache the static assets for 24 hours and serve them from the closest geographic location to the viewer. This reduces page load time for the SPA shell from 300ms to 30ms and offloads all static-serving traffic from the dashboard servers.

```
Pros: Sub-50ms asset delivery, offloads web server, global coverage
Cons: Only applies to static assets, not API responses
  (API responses are cacheable only if they have a TTL)
```

## Capacity Planning

### Scenario: 10 Million DAU Dashboard Platform

You need to design a dashboard platform for a service with 10 million daily active users. Each user triggers metric emissions from 40 backend services running across 500 hosts.

### Step 1: Compute Ingestion Rate

```
Metrics per host: 200 (CPU, memory, disk, network, per-service metrics)
Scrape interval: 15 seconds
Hosts: 500
Samples per scrape: 500 hosts × 200 metrics = 100,000 samples
Samples per second: 100,000 / 15 = 6,667 samples/second

Application-level metrics (per request):
  Requests per second: 50,000
  Metrics per request: 5 (latency, status, bytes in/out, active sessions)
  Samples per second: 50,000 × 5 = 250,000 samples/second

Total ingestion rate: 256,667 samples/second ≈ 260K samples/second
Daily ingestion: 260K × 86,400 = 22.5 billion samples/day
```

### Step 2: Compute Storage Requirements

```
Compressed size per sample: 1.37 bytes (Gorilla compression)
Daily raw storage: 22.5B × 1.37 B = 30.8 GB/day
Raw retention (7 days): 30.8 × 7 = 216 GB
Monthly raw storage: 30.8 × 30 = 924 GB
Rollup factor (1m to 1h): 60x reduction
Monthly rollup storage: 924 / 60 = 15.4 GB
Total for 7 days raw + 13 months rollup: 216 + (15.4 × 13) = 416 GB
```

### Step 3: Compute Query Throughput

```
Dashboard views per day: 10M DAU × 0.5 views/user = 5M dashboard views
Panels per dashboard: 12
Queries per dashboard load: 12 (one per panel)
Queries per dashboard refresh: 12 (refresh every 30s, 2 refreshes/min)

Total queries per second at peak (2x average):
  Average QPS: (5M dashboard views × 12 panels) / 86,400 = 695 QPS
  Peak QPS: 695 × 2 = 1,390 QPS
  With 70% cache hit rate: 1,390 × 0.3 = 417 actual TSDB queries/second
```

### Step 4: Compute Cluster Size

```
Ingestion nodes:
  Each ingestion node handles 100K samples/second
  Required: 260K / 100K = 3 nodes (round up to 4 for redundancy)

Storage shards:
  Each shard handles 50K active time series
  Active series: 500 hosts × 200 metrics = 100,000 series
  Required: 100K / 50K = 2 shards (use 4 for headroom and replication)

Query nodes:
  Each query node handles 200 QPS (TSDB queries)
  Required: 417 / 200 = 3 nodes (round up to 4 for redundancy)

Total cluster: 4 ingestion + 4 storage + 4 query = 12 nodes
Plus: 2 load balancers, 2 Redis cache nodes, 1 CDN configuration
Estimated monthly infrastructure cost: $8,000-$12,000
```

## Deployment and Operations

### Deployment Model

The dashboard platform is deployed as a set of containerized microservices orchestrated by Kubernetes. Each tier (ingestion, storage, query, presentation) has its own deployment manifest, horizontal pod autoscaler, and pod disruption budget.

```yaml

  namespace: observability

  replicas: 4
  selector:

      app: dashboard-query

    metadata:

        app: dashboard-query
```

### Operational Runbooks

**Slow Dashboard Load** — Check query API latency: if p99 > 500ms, check Redis cache hit rate. If cache hit rate is below 50%, increase cache TTL from 30 to 60 seconds. If cache is fine, check TSDB query latency: if p99 > 2 seconds, add more storage shards or increase rollup resolution.

**Missing Data on Dashboard** — Check scrape target health: if targets are down, restart the metrics agent. If targets are up, check the ingestion pipeline: Kafka lag should be under 10 seconds. If Kafka lag is high, add ingestion nodes. If all pipelines are healthy, check that the dashboard query time range matches the data retention window.

**High Storage Usage** — Check series churn rate: a high series churn (rapid creation and deletion of time series) generates index overhead. Reduce labels with high cardinality (session IDs, user IDs). Increase rollup retention and decrease raw retention if storage is critically low.

## Monitoring the Monitor

Every dashboard platform must be monitored itself. The monitoring system that tracks your services cannot meaningfully monitor itself — if the dashboard platform is down, the monitoring alerts for the dashboard platform are also unreachable.

**Health check endpoints.** Every dashboard component exposes a `/healthz` endpoint (liveness, is the process running?) and a `/readyz` endpoint (readiness, can the process accept traffic?). The load balancer uses the readiness probe to route traffic. A separate, independent monitoring system (a simple bash script on a different server) pings the health endpoints every 10 seconds and alerts if any component is unhealthy.

**Synthetic dashboard tests.** A synthetic test script runs every 60 seconds from an external monitoring service (Pingdom, StatusCake, or a simple Lambda function). The script loads the most critical dashboard, waits for all panels to render, and reports the page load time. If the dashboard takes longer than 5 seconds to load, an alert fires. If the dashboard returns a 5xx error, a critical alert fires. This synthetic test runs from outside the production network, so it detects network-level failures that internal health checks miss.

**Dashboard usage metrics.** The dashboard platform tracks its own usage metrics: number of dashboard views per hour, average page load time, number of queries per second, cache hit rate, and error rate. These metrics are displayed on a "dashboard of dashboards" — a meta-dashboard that shows the health of the dashboard platform itself. This meta-dashboard is the first thing the on-call engineer checks when they suspect the dashboard platform is degraded.

## Observing the Dashboard Platform

The dashboard platform must be observable itself. Without observability, you cannot tell whether a slow dashboard is caused by a network issue, a TSDB bottleneck, a cache miss, or a browser rendering problem.

### Instrumentation Points

Every component of the dashboard platform must export metrics that allow the operations team to diagnose problems:

**Ingestion tier metrics:**
- `scrape_duration_seconds`: how long each scrape took (p50, p99)
- `scrape_samples_post_metric_relabeling`: number of samples collected per scrape
- `scrape_targets_up`: number of scrape targets that responded successfully
- `scrape_targets_down`: number of scrape targets that did not respond
- `ingestion_queue_length`: how many samples are waiting to be written to the TSDB
- `ingestion_queue_duration_seconds`: how long samples wait in the ingestion queue

**Storage tier metrics:**
- `tsdb_wal_writes_total`: number of WAL write operations
- `tsdb_wal_write_duration_seconds`: how long each WAL write takes
- `tsdb_memtable_size_bytes`: current MemTable size
- `tsdb_memtable_flush_duration_seconds`: how long each MemTable flush takes
- `tsdb_chunks_in_memory`: number of compressed chunks in the chunk cache
- `tsdb_series_total`: total number of active time series
- `tsdb_index_memory_bytes`: memory used by the inverted index

**Query tier metrics:**
- `query_duration_seconds`: query execution time (p50, p95, p99)
- `query_cache_hits_total`: number of cache hits
- `query_cache_misses_total`: number of cache misses
- `query_concurrent`: current number of concurrent queries
- `query_samples_scanned`: number of data points scanned per query
- `query_slow_queries_detected`: number of queries that exceeded the slow query threshold

**Presentation tier metrics:**
- `dashboard_load_duration_seconds`: time to load a complete dashboard (p50, p99)
- `dashboard_panels_per_page`: number of panels per dashboard
- `dashboard_refresh_latency_seconds`: how long the refresh cycle takes
- `dashboard_views_per_hour`: how many times each dashboard is viewed

### Continuous Dashboard Performance Monitoring

The operations team runs synthetic dashboard loads every 60 seconds from an external monitoring service. The synthetic load opens the most critical dashboard (the one the on-call engineer uses), waits for all panels to render, and reports the total load time.

If the synthetic dashboard load takes longer than 5 seconds, a warning alert fires. If it takes longer than 10 seconds, a critical alert fires. If the dashboard returns a 5xx error, a page is sent to the on-call engineer.

The synthetic load results are stored in a separate monitoring system (not the dashboard platform itself) so that they are available even when the dashboard platform is down. The last 100 synthetic load results are displayed on a status page that the operations team can access independently.

### Dashboard Platform Runbook

Every failure mode of the dashboard platform has a corresponding runbook entry. The runbook entry describes the symptoms, the probable cause, and the steps to fix it.

**Symptom: Dashboard shows "No data" for all panels.**
Probable cause: TSDB is down or unreachable.
Fixes: Check the TSDB health endpoint. If the TSDB process is down, restart it. If the TSDB is up but not responding, check the disk space (a full disk causes the TSDB to stop accepting writes). If the disk is full, delete old data blocks or increase the disk size.

**Symptom: Dashboard loads but panels show stale data.**
Probable cause: Scrape agent is not reaching the targets.
Fixes: Check the scrape target list in the Prometheus configuration. Run a manual scrape against one of the targets to verify the target is responding. If the target is up but Prometheus is not scraping it, check the network firewall rules.

**Symptom: Dashboard takes longer than 10 seconds to load.**
Probable cause: One or more panels are running expensive queries.
Fixes: Check the query API's slow query log. Identify the slow query and either rewrite it to be more efficient, add a recording rule to pre-compute it, or increase the cache TTL for that query.

**Symptom: Dashboard crashes the browser tab.**
Probable cause: A panel is rendering too many data points (often caused by a query that returns 100,000+ data points for a single chart).
Fixes: Limit the query to return at most 10,000 data points. Increase the rollup resolution for long time ranges. Use a table with sparklines instead of a line chart for high-cardinality data.

## Zero-Downtime Upgrades

Dashboard platforms must be upgraded without interrupting the monitoring pipeline. A rolling upgrade strategy ensures that at least one replica of each component is always available.

**Ingestion tier upgrade.** Ingestion nodes are stateless — they scrape metrics and forward them to Kafka. A rolling upgrade of ingestion nodes causes no data loss because the scrape targets are independently addressable. If the scrape agent on one node is restarting, the other nodes continue scraping. After the upgrade, the restarted node resumes scraping.

**Storage tier upgrade.** Storage nodes require careful handling because they hold data. The upgrade process is: mark the node as read-only (drain), wait for the node to flush its MemTable to disk, upgrade the binary, restart the node, and mark it as read-write. During the upgrade, the node is read-only — it cannot accept new writes but can still serve existing data. The other nodes in the cluster handle the write load.

**Query tier upgrade.** Query nodes are stateless — they accept queries, check the cache, and forward to the storage tier. A rolling upgrade of query nodes causes no query loss because the load balancer routes traffic away from the upgrading node. During the upgrade, queries are served by the remaining nodes. After the upgrade, the node re-joins the load balancer pool.

## Cost Optimization

Dashboard platforms can be expensive to run at scale. The major cost drivers are storage (SSD for hot data, S3 for cold data), compute (ingestion and query nodes), and network (data transfer between regions).

**Reduce storage cost.** The most effective cost optimization is reducing the raw data retention period. Raw data at 10-second resolution is expensive. Reducing raw retention from 30 days to 7 days saves 75% of storage cost. The remaining 23 days of trend data is served from 1-hour rollups, which are 60x smaller than raw data.

**Reduce compute cost.** Query caching is the most effective compute cost optimization. A 60-second cache TTL reduces the query load on the TSDB by 80%. For a cluster that costs $5,000 per month, caching saves $4,000 per month. The remaining $1,000 in compute cost is for the cache misses and the cache layer itself.

**Reduce network cost.** In a multi-region setup, cross-region data transfer is expensive. The solution is to keep dashboard queries within the region. Each region's dashboards query only the local TSDB. The global view uses only pre-aggregated rollups, which are pushed to each region once per day. This reduces cross-region data transfer from terabytes per day to megabytes per day.

## Federation Patterns

Federation is the practice of connecting multiple Prometheus or TSDB instances to form a query hierarchy. There are three common federation patterns, each with different trade-offs.

**Hierarchical federation.** A central Prometheus server scrapes selected metrics from multiple leaf Prometheus servers. The central server stores aggregated data for cross-service dashboards. Each leaf server stores detailed data for service-specific dashboards. This pattern works well for organizations with clear team boundaries. The trade-off is that the central server becomes a single point of failure for cross-service dashboards.

**Functional federation.** Different Prometheus servers are responsible for different metric categories. One server handles infrastructure metrics (CPU, memory, disk). Another handles application metrics (request rate, error rate, latency). A third handles business metrics (revenue, active users). The dashboard queries the appropriate server based on the metric category. This pattern works well when metric categories have different retention requirements and query patterns.

**Temporal federation.** Different TSDB instances handle different time ranges. A Prometheus instance handles the last 7 days of data at 15-second resolution. A Thanos store handles 7-30 days at 1-minute resolution. A long-term storage (S3) handles 30 days to 13 months at 1-hour resolution. The query layer automatically routes queries to the appropriate instance based on the time range. This pattern optimizes storage cost by matching resolution to retention.

## Alert Integration

Dashboards and alerts are complementary systems. The dashboard shows what is happening now. The alert tells you when something needs attention. Integrating alerts into the dashboard creates a closed feedback loop.

**Alert annotations on dashboards.** When an alert fires, the dashboard automatically shows an annotation at the time the alert fired. The annotation includes the alert name, severity, and a link to the alert details. This lets the on-call engineer see not just the metric spike but also the automated response to the spike.

**Dashboard links from alerts.** Every alert notification includes a link to the relevant dashboard with the time range set to the alert window. The on-call engineer clicks the link and sees the dashboard pre-configured to the time of the incident. This eliminates the 2-minute overhead of navigating to the correct dashboard and setting the time range.

**Silence integration.** When a dashboard panel is showing an anomaly that is expected (known maintenance, planned deployment), the engineer can silence the associated alerts directly from the dashboard. The silence is time-bounded and automatically expires. This reduces alert fatigue by correlating dashboard observations with alert suppression.

## Disaster Recovery and Backup

Dashboard systems must have a disaster recovery plan that covers data loss, region failure, and operational mistakes.

**Backup strategy.** Three types of data must be backed up: the time series database (metric data), the dashboard definitions (dashboard-as-code files in Git), and the configuration (Prometheus config, alerting rules, recording rules). The TSDB data is backed up to S3 or GCS every 6 hours using a snapshot utility (Prometheus has a built-in snapshot API). Dashboard definitions are already in Git, so they are backed up by the existing Git backup process. Configuration is backed up as part of the infrastructure-as-code pipeline.

**Recovery point objective (RPO).** The TSDB backup runs every 6 hours, so the maximum data loss is 6 hours. For critical metrics, the RPO is reduced to 1 hour by using a secondary TSDB that receives the same data via a Kafka consumer. The secondary TSDB is in a different region. If the primary region fails, the secondary region has all metric data with at most 1 hour of data loss.

**Recovery time objective (RTO).** The target RTO is 30 minutes: within 30 minutes of a disaster declaration, the dashboard system should be fully operational. The RTO is achieved by maintaining a hot standby in a secondary region. The standby is configured with the same dashboard definitions and scrape targets. In the event of a primary region failure, the DNS record is updated to point to the secondary region. The DNS propagation takes 5-10 minutes, and the secondary region's TSDB is already running with the most recent data.

**Cross-region backup validation.** The backup is verified monthly by restoring it to a test environment and running a synthetic dashboard load. The test verifies that the dashboard loads correctly, the metrics are present, and the query latency is within acceptable ranges. Any backup that fails the test is investigated and the backup process is fixed.

## Dashboard Templates and Automation

At scale, manual dashboard creation does not work. Each team creates dashboards differently, leading to inconsistency and gaps in coverage. Automation ensures that every service has a minimum viable dashboard.

### Auto-Generated Baseline Dashboard

When a new service is deployed and registered in the service catalog, the dashboard platform automatically creates a baseline dashboard. The baseline dashboard includes:

- **Four golden signals:** latency (p50, p95, p99), traffic (requests per second), errors (error rate percentage), and saturation (CPU, memory, connection pool)
- **Deployment annotations:** the dashboard automatically overlays deployment events from the CI/CD pipeline
- **Dependency panels:** the dashboard shows the health of the service's dependencies (databases, caches, queues) by querying dependency-specific metrics
- **Runbook link:** a panel at the bottom of the dashboard links to the service's runbook

The baseline dashboard is generated from a template. The template uses the service's metadata (name, team, dependencies) from the service catalog to fill in the panel configurations.

### Service Catalog Integration

The service catalog is the source of truth for service metadata. The dashboard platform reads from the catalog to know which services exist, which team owns each service, what dependencies each service has, and what metrics each service exposes.

When a service is added to the catalog, a CI/CD job triggers the dashboard generation pipeline. The pipeline creates the baseline dashboard, configures the scrape targets in Prometheus, sets up the recording rules for the four golden signals, and opens a pull request for the service team to customize the dashboard.

### Dashboard Versioning

Dashboard definitions are versioned alongside the service code. Each dashboard has a version number, and changes are tracked through Git. The dashboard platform stores all versions and can roll back to any previous version.

Versioning is important for incident postmortems. When an incident occurs, the dashboard platform freezes the dashboard version at the time of the incident. The postmortem team reviews the frozen dashboard to understand what signals were available at the time of the incident. This prevents hindsight bias: the team cannot claim "the dashboard should have shown X" because they are looking at the dashboard as it was, not as it is now.

## Advanced Query Optimization

### Predicate Pushdown

When a query asks for `rate(http_requests_total{service="auth-api"}[5m])`, the TSDB can push the label filter `service="auth-api"` down to the storage layer. The storage layer reads only the chunks for matching series, not all series. This optimization reduces I/O by the selectivity factor. If 1 out of 50 services is selected, I/O is reduced by 98%.

### Vector Matching

When a query joins two metrics (e.g., error rate divided by request rate to compute error percentage), the TSDB must align the two time series by timestamp. Vector matching pairs data points with the same timestamp from both series. This operation is O(N) where N is the number of data points, and is efficient when both series have the same timestamp alignment. Mismatched alignment (one series has gaps) requires interpolation, which is slower.

### Subquery Optimization

A query like `avg_over_time(rate(http_requests_total[5m])[1h:])` computes the rate over 5-minute windows, then averages those rates over 1 hour. This is a subquery, and it is expensive because it materializes the intermediate rate computation before averaging. The optimization is to use recording rules that pre-compute the hourly rate as a separate metric. The dashboard query then reads the pre-computed metric instead of computing it on the fly.

```yaml
# Recording rule: pre-compute hourly average request rate
groups:
  - name: request_rates
    interval: 5m
    rules:
      - record: job:http_requests_total:rate1h
        expr: avg_over_time(rate(http_requests_total[5m])[1h:])
```

This recording rule runs every 5 minutes and stores the result as a new time series. The dashboard query reads `job:http_requests_total:rate1h` instead of computing the subquery. The query is 10-50x faster because the computation was already done.

## Real-World Incident: Dashboard Platform Outage

In 2020, a major cloud provider experienced a dashboard platform outage that affected thousands of customers. The root cause was a misconfigured scrape target that caused the TSDB's inverted index to grow uncontrollably, consuming all available memory and triggering an OOM kill.

**Timeline:**
- 14:00 UTC: A new customer deployed a service with a misconfigured metrics exporter that emitted metrics with 10 million unique label combinations (cardinality explosion).
- 14:05 UTC: The TSDB's inverted index grew from 500 MB to 8 GB as it indexed the new labels.
- 14:07 UTC: The TSDB ran out of memory and the OOM killer terminated the process.
- 14:07-14:15 UTC: All dashboards showed "No data" for every panel. The operations team received hundreds of alerts from services that detected the dashboard platform was down.
- 14:15 UTC: The TSDB restarted and began recovering from the WAL. The recovery process replayed the high-cardinality labels, causing another OOM kill.
- 14:15-14:30 UTC: The TSDB entered a crash loop, restarting every 2-3 minutes and immediately running out of memory.
- 14:30 UTC: The operations team identified the problematic scrape target and removed it from the scrape configuration.
- 14:32 UTC: The TSDB restarted successfully without the high-cardinality labels.
- 14:35 UTC: Dashboards began showing data again. Total outage duration: 35 minutes.

**Lessons learned:**
1. Label validation at ingestion time prevents cardinality explosions. The incident would have been avoided if the scrape target had label validation enabled.
2. The WAL replay during recovery replayed the same high-cardinality labels, causing the crash loop. The recovery process should skip high-cardinality labels after detecting a crash loop.
3. The dashboard platform should have a "circuit breaker" that automatically disconnects scrape targets that cause the TSDB to exceed memory thresholds.

## Capacity Planning Walkthrough: 50 Million DAU

Let us walk through a full capacity planning exercise for a dashboard platform serving 50 million daily active users.

**Step 1: Estimate the number of services and hosts.**
For 50 million DAU, you need approximately 5,000 microservices running across 10,000 hosts. Each service exposes 100 metrics. Each host exposes 200 infrastructure metrics. Total metrics: (5,000 x 100) + (10,000 x 200) = 500,000 + 2,000,000 = 2.5 million metrics.

**Step 2: Estimate the ingestion rate.**
At a 15-second scrape interval, each metric produces 5,760 data points per day. Total data points per day: 2.5 million x 5,760 = 14.4 billion. Data points per second: 14.4 billion / 86,400 = 166,667.

**Step 3: Estimate storage requirements.**
Compressed size per data point: 1.37 bytes. Raw data per day: 14.4 billion x 1.37 bytes = 19.7 GB. Raw data for 7 days: 19.7 GB x 7 = 138 GB. One-minute rollups: 14.4 billion / 6 = 2.4 billion points per day. At 0.17 bytes per point = 408 MB per day. For 30 days: 12.2 GB. One-hour rollups: 2.4 billion / 60 = 40 million points per day. At 0.17 bytes per point = 6.8 MB per day. For 13 months: 2.7 GB. Total storage: 138 GB + 12.2 GB + 2.7 GB = 153 GB.

**Step 4: Estimate query throughput.**
50 million DAU. Each user views 0.5 dashboards per day. Total dashboard views: 25 million. Panels per dashboard: 12. Queries per dashboard load: 12. Total queries per day: 25 million x 12 = 300 million. Queries per second (average): 300 million / 86,400 = 3,472. Queries per second (peak, 2x): 6,944. With 70% cache hit rate: 2,083 TSDB queries per second.

**Step 5: Estimate cluster size.**
Ingestion nodes: 166,667 / 100,000 = 2 nodes (round to 4 for redundancy). Storage shards: 2.5 million series / 50,000 = 50 shards (round to 64 for overhead). Query nodes: 2,083 / 200 = 11 nodes (round to 16 for redundancy). Cache nodes: 3 Redis nodes. Total: 4 + 64 + 16 + 3 = 87 nodes.

**Step 6: Estimate cost.**
Each node: $500/month (c5.2xlarge equivalent). Total compute: 87 x $500 = $43,500/month. Storage: 153 GB SSD at $0.08/GB/month = $12/month. Total: $43,512/month. Annual cost: $522,144.
