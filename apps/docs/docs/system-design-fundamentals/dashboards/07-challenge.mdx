## Tier 1: Foundation

### Challenge 1: Design a Single-Service Dashboard

You are building a dashboard for a single web service that handles user authentication. The service has 10 instances running behind a load balancer. List the 8 most important metrics you would show on the dashboard and explain why each metric matters.

**Model Answer:**

The 8 metrics, in priority order from top to bottom:

1. **Error rate (percentage of 5xx responses)** — The single most important health indicator. If errors are above 1%, the service is degraded. This needs to be the first panel the on-call engineer sees.

2. **P99 latency** — The second most important indicator. If latency exceeds 1 second, users are experiencing slow responses. This panel should be adjacent to the error rate so the engineer can correlate them.

3. **Request rate (requests per second)** — The throughput metric. A sudden drop in request rate indicates a routing problem (load balancer, DNS). A sudden spike indicates a traffic surge or DDoS.

4. **CPU utilization (average and max across instances)** — The resource metric. If CPU exceeds 80%, the service is approaching capacity. The max across instances catches the hot instance that the average hides.

5. **Memory utilization (average and max)** — The memory metric. A memory leak shows as a steady upward trend. The panel should show the 24-hour trend, not just the current value.

6. **Active connections** — The concurrency metric. If active connections exceed the connection pool limit, new requests will be queued or rejected. This metric often correlates with latency spikes.

7. **Database query latency (p50, p99)** — The dependency metric. If the database is slow, the service is slow. This panel helps the engineer determine whether the service is the problem or the database is the problem.

8. **Error breakdown by HTTP status code** — The diagnostic metric. A spike in 401s means authentication failures. A spike in 503s means the service is overloaded. A spike in 500s means a code bug. The breakdown tells the engineer where to look.

### Challenge 2: Identify the Dashboard Problem

A dashboard shows the following metrics for a payment processing service. The on-call engineer reports that the dashboard is not useful during incidents. Identify at least 5 problems with the dashboard description.

```
Dashboard: Payment Service Overview
- Panel 1: CPU usage (average across all hosts, last 7 days)
- Panel 2: Memory usage (average across all hosts, last 7 days)
- Panel 3: Disk I/O (each host as a separate line, 12 hosts, all lines in blue)
- Panel 4: Network throughput (stacked area chart, 12 hosts stacked)
- Panel 5: Requests per second (last 7 days)
- Panel 6: Error count (last 7 days, no percentage)
- Panel 7: Database connections (current value, no trend)
- Panel 8: Queue depth (current value, no trend)
- Panel 9: GC pause time (last 7 days)
- Panel 10: Average response time (last 7 days)
```

**Model Answer:**

1. **No error rate percentage.** Error count alone is misleading — 100 errors out of 1,000 requests is a 10% error rate (critical). 100 errors out of 100,000 requests is a 0.1% error rate (acceptable). The dashboard shows the raw count but not the rate, so the engineer cannot judge severity.

2. **Averaging hides outliers.** CPU and memory are averaged across all 12 hosts. One host at 100% CPU is invisible when averaged with 11 hosts at 40%. The dashboard should show the max or p99 across hosts, not the average.

3. **No latency metric.** The dashboard shows "average response time" but not p99 latency. The average is misleading because a single slow request can pull the average up, but the 99th percentile latency tells you what actual users experience. The metric should be p99 latency, not average.

4. **No SLO or threshold lines.** None of the panels have threshold lines. The dashboard shows data but does not tell the engineer whether the data is good or bad. Every panel should have a horizontal line marking the acceptable threshold.

5. **No previous-period comparison.** None of the panels overlay the previous period (last hour, yesterday, last week). The engineer cannot tell whether the current value is normal, improving, or degrading. A 200ms latency is bad unless it was 500ms yesterday.

6. **No correlation with deployments.** The dashboard has no annotations for deployments, configuration changes, or incidents. When a metric spikes, the engineer has no way to tell whether a deployment caused it. Every dashboard should overlay deployment annotations.

7. **Inconsistent time ranges.** Some panels show the last 7 days (CPU, memory, requests) while others show current values (database connections, queue depth). The engineer cannot compare metrics across panels that use different time ranges. All panels should default to the same time range.

### Challenge 3: Optimize a Dashboard Query

A dashboard panel shows the p99 latency for 50 services over the last 30 days. The query takes 45 seconds to complete. The dashboard has 12 such panels. The page times out after 60 seconds. Identify the problem and propose three optimizations.

**Model Answer:**

The problem is that the query scans raw data over 30 days. A 30-day range at 10-second resolution means 259,200 data points per time series. With 50 services, the query scans 12.96 million data points. With 12 panels, the total scan is 155 million data points — far too many for a dashboard load.

**Optimization 1: Use rollups.** Change the query to read from 1-hour rollup tables instead of raw data. A 30-day query against 1-hour rollups reads 720 data points per service (instead of 259,200). This is a 360x reduction in data scanned. The p99 of hourly rollups is not the same as the p99 of raw data, but it is accurate enough for a 30-day trend view.

**Optimization 2: Reduce the default time range.** Change the dashboard default from 30 days to 6 hours. The operational use case for this dashboard is "what is happening right now," not "what happened last month." The 30-day view should be available but only when the user explicitly selects it. This reduces the default query time from 45 seconds to approximately 1 second.

**Optimization 3: Cache the 30-day query.** The 30-day p99 latency for 50 services changes slowly — at most a few data points per refresh cycle. Cache the result of this query with a 1-hour TTL. The first user to load the dashboard waits 45 seconds (acceptable for a slow query). Every subsequent user for the next hour gets the result from cache in under 1 second.

## Tier 2: Design

### Challenge 4: Design a Real-Time Dashboard for a Video Streaming Platform

You are building a real-time dashboard that shows the health of a video streaming platform serving 10 million concurrent viewers. The platform has 200 microservices across 3 regions. The dashboard must update every 10 seconds and show the last 6 hours of data. Design the ingestion pipeline, storage system, and query architecture.

**Model Answer:**

**Ingestion pipeline.** Each microservice exposes a `/metrics` endpoint that returns metrics in Prometheus exposition format. A regional scrape agent polls each service every 10 seconds. The scraped data is sent to a regional Kafka topic. The Kafka topic has 3 partitions per region, partitioned by service ID. This ensures that metrics for the same service are always in the same partition, enabling ordered processing.

**Storage system.** Each region runs a dedicated VictoriaMetrics cluster. VictoriaMetrics is chosen because it handles 10x more active time series than Prometheus on the same hardware, and the platform has approximately 200 services x 200 metrics per service = 40,000 active time series. The cluster has 3 ingestion nodes and 3 storage nodes. Raw data is stored for 7 days at 10-second resolution. One-minute rollups are stored for 30 days. One-hour rollups are stored for 13 months.

**Query architecture.** A regional query API serves dashboard queries from the regional VictoriaMetrics cluster. The query API has a Redis cache with a 30-second TTL. A global query API aggregates results from all three regions for the cross-region overview dashboard. The global API queries only the rollup tables (1-hour resolution), never the raw data.

**Dashboard layout.** The top row shows 4 global health indicators: total streaming errors, average bitrate, p99 startup latency, and active viewers. The second row shows per-region health (3 panels, one per region). The third row shows the top 10 services by error rate with sparklines. The dashboard refreshes every 10 seconds.

### Challenge 5: Design a Multi-Tenant Dashboard Platform

You are building a dashboard platform for a company with 50 engineering teams. Each team manages 5-20 microservices. Each team must see only their own metrics. The platform must support 1,000 concurrent dashboard viewers. Design the authentication, authorization, data isolation, and rate limiting.

**Model Answer:**

**Authentication.** The dashboard platform integrates with the company's existing SSO provider (Okta, Azure AD, Google Workspace). Every dashboard viewer authenticates through SSO. The dashboard API receives a JWT token with the user's identity and team membership.

**Authorization.** Each team has a namespace in the metric store. The metric namespace is a prefix on the metric name: `team-payments.request_latency`, `team-auth.error_rate`. The authorization layer at the query API checks the JWT token and rewrites the query to include only the namespaces the user is authorized to see. For example, a user from team-payments sees `team-payments.*` and `shared-infra.*` but not `team-auth.*`.

**Data isolation.** The time series database is sharded by namespace. All metrics for team-payments go to shards 1-4. All metrics for team-auth go to shards 5-8. This ensures that one team's heavy query does not affect another team's metrics. The shard assignment is stored in a configuration database and cached in the query API.

**Rate limiting.** Each team has a quota of 500 queries per second. Teams that exceed the quota receive a 429 (Too Many Requests) response. The quota is enforced by a token bucket algorithm in the query API. The dashboard frontend shows a warning banner when the team is approaching the rate limit. Each team can request a quota increase through a self-service form.

**Dashboard ownership.** Each dashboard has a metadata field with the team owner. The dashboard platform sends a weekly summary of dashboard views and metric freshness. Dashboards that have not been viewed in 90 days are automatically archived. Dashboards with metrics that have not emitted data in 7 days show a warning banner.

### Challenge 6: Design a Cost-Optimized Dashboard at Massive Scale

Your company needs to store 10 years of metric data for compliance purposes. The data volume is 1 billion data points per day. The budget is $50,000 per year for storage. Design the storage tier with cost calculations.

**Model Answer:**

**Data volume.** 1 billion data points per day at 1.37 bytes per point (Gorilla compressed) = 1.37 GB per day = 500 GB per year.

**Tier 1: Hot storage (7 days).** Raw 10-second resolution data stored on SSD. 7 days x 1.37 GB = 9.6 GB. Cost: $0.08/GB/month x 9.6 GB = $0.77/month x 12 = $9.22/year.

**Tier 2: Warm storage (30 days).** One-minute rollup data stored on HDD. 1 billion points per day / 6 (from 10s to 1m) = 167 million points per day. At 0.17 bytes per point (rollup compression) = 28 MB per day. 30 days = 840 MB. Cost: $0.02/GB/month x 0.84 GB = $0.02/month x 12 = $0.20/year.

**Tier 3: Cold storage (13 months).** One-hour rollup data stored on S3 Standard. 167 million points per day / 60 (from 1m to 1h) = 2.8 million points per day. At 0.17 bytes per point = 0.5 MB per day. 395 days = 193 MB. Cost: $0.023/GB/month x 0.19 GB = $0.004/month x 12 = $0.05/year.

**Tier 4: Archive (10 years).** Daily rollup data stored on S3 Glacier Deep Archive. 2.8 million points per day / 24 (from 1h to 1d) = 117,000 points per day. At 0.17 bytes per point = 20 KB per day. 3,650 days = 71 MB. Cost: $0.001/GB/month x 0.07 GB = $0.00007/month x 12 = $0.0008/year.

**Total annual storage cost: approximately $10.** This is well within the $50,000 budget. The key insight is that compression and rollups reduce the data volume by orders of magnitude as the data ages. The 10-year archive costs essentially nothing because the daily rollup is so small.

## Tier 3: Expert

### Challenge 7: Design a System That Detects Dashboard Anomalies Automatically

You are building a system that watches dashboards and automatically detects anomalies. The system should alert the on-call engineer when a metric deviates from its expected pattern. The system must handle 50,000 metrics across 500 dashboards. Design the anomaly detection algorithm, the alerting pipeline, and the integration with existing dashboards.

**Discussion points:**

The key challenge is distinguishing between genuine anomalies and normal patterns. A metric that spikes every day at 9:00 AM (the daily traffic surge) should not trigger an alert. A metric that spikes at 2:00 AM (outside normal traffic patterns) should trigger an alert.

**Seasonal decomposition.** Use a time series decomposition algorithm that separates each metric into three components: trend (long-term direction), seasonality (daily and weekly patterns), and residual (the remainder after removing trend and seasonality). The anomaly detector watches the residual component. A residual that exceeds 3 standard deviations from the mean is an anomaly. This approach automatically handles daily and weekly patterns without manual configuration.

**Multi-metric correlation.** A single metric spike is often a false alarm. But a correlated spike across multiple metrics (error rate AND latency AND database connections all spiking at the same time) is almost certainly a real incident. The anomaly detector should compute pairwise correlations between metrics and only alert when at least 3 correlated metrics trigger simultaneously.

**Integration with dashboards.** The anomaly detector annotates the dashboard chart with a red marker at each detected anomaly. The on-call engineer sees the anomaly marker alongside the deployment annotation, making it easy to determine whether the anomaly was caused by a deployment. The anomaly detector also generates a summary dashboard that shows the top 5 anomalies in the last hour, ranked by severity.

### Challenge 8: Design a Dashboard System for a Fintech Company with Regulatory Requirements

A fintech company needs a dashboard system that meets regulatory requirements: all metric data must be immutable (no deletion or modification), queries must be auditable (every query is logged with the user and timestamp), and the system must provide 99.99% availability. The company processes 1 million transactions per day. Design the system with regulatory compliance in mind.

**Discussion points:**

**Immutability.** The time series database uses a write-once, append-only storage format. Once a data point is written, it can never be modified or deleted. The database never compacts or merges blocks — it only writes new blocks. This means the storage requirement is higher than a standard TSDB (no compaction savings), but every data point is cryptographically verifiable. The database writes a SHA-256 hash of each block to a blockchain-based audit log.

**Query audit trail.** Every query to the dashboard API is logged with the user identity, timestamp, query string, and result size. The audit log is stored in a separate, append-only database that is never pruned. The compliance team has read-only access to the audit log. The dashboard API returns a unique query ID with every response, which the compliance team can use to trace the query from the frontend to the storage layer.

**High availability.** The system is deployed across 3 independent regions (active-active-active). Each region runs the full dashboard stack. In the event of a regional failure, the remaining 2 regions handle the full load. The system uses a global load balancer with health checks that route traffic away from the failed region within 5 seconds. The dashboard SLO is 99.99% availability, which allows 52 minutes of downtime per year.

**Data residency.** Transaction data must remain in the country of origin. The dashboard platform enforces data residency at the storage layer: metrics from EU transactions are stored only in EU regions. The global view dashboard shows only aggregated data (1-hour rollups) from all regions, never raw data. Each region's dashboard API enforces that users can only query metrics from their authorized region.

**Access control.** The dashboard API implements role-based access control (RBAC) with three roles: viewer (can view dashboards but not modify them), editor (can create and modify dashboards for their team), and admin (can manage users, teams, and system configuration). All role changes are logged and require two-person approval.

## Case Study: Design a Kubernetes Cluster Dashboard

You are tasked with designing a dashboard for a Kubernetes cluster running 50 microservices across 20 nodes. The dashboard will be used by the platform team (5 engineers) and the service teams (50 engineers). Design the dashboard hierarchy and the key metrics for each level.

**Model Answer:**

The dashboard hierarchy has three levels.

**Level 1: Cluster Overview (platform team, 1 dashboard).** This dashboard shows the health of the entire cluster. The top row shows: total pods (running vs pending vs failed), node CPU utilization (average and max), node memory utilization (average and max), and pod restart rate. The second row shows the top 5 services by resource usage (CPU, memory, network). The third row shows the top 5 nodes by resource usage. The dashboard refreshes every 30 seconds and shows the last 6 hours.

**Level 2: Namespace Overview (service teams, 1 per namespace).** This dashboard shows the health of a specific namespace. The top row shows: deployments by status (available replicas vs desired replicas), pod CPU usage (average and max across pods), pod memory usage, and error rate (5xx responses from the ingress). The second row shows the top 5 pods by resource usage. The third row shows the service discovery status (endpoints registered vs endpoints ready). The dashboard refreshes every 30 seconds.

**Level 3: Pod Detail (debugging, on-demand).** This dashboard shows the health of a specific pod. It shows: CPU usage (with the pod's resource request and limit as threshold lines), memory usage, network I/O, disk I/O, and the last 100 log lines. The dashboard is accessed by clicking a pod name in the namespace overview dashboard. It refreshes every 15 seconds and shows the last 1 hour.

## Case Study: Design a Dashboard for a Database Cluster

Design a dashboard for a PostgreSQL database cluster with 3 nodes (primary + 2 replicas). The database handles 10,000 transactions per second. The dashboard is used by the database reliability team.

**Model Answer:**

The dashboard has 8 panels organized in a 4x2 grid.

**Panel 1: Transactions per second.** A line chart showing the number of transactions per second, with a separate line for the primary and each replica. Threshold line at 15,000 TPS (warning) and 20,000 TPS (critical). This panel is the most important — it shows the database throughput.

**Panel 2: Replication lag.** A line chart showing the replication lag in seconds for each replica. A lag of under 1 second is normal. A lag of 5+ seconds indicates the replicas are falling behind. Threshold line at 5 seconds (warning) and 30 seconds (critical). This panel is the second most important — it shows whether the replicas are keeping up with the primary.

**Panel 3: Connection count.** A gauge showing the current number of database connections, with a threshold line at 80% of the connection pool limit. A single stat panel (big number) with a sparkline showing the trend.

**Panel 4: Query latency.** A line chart showing p50, p95, and p99 query latency. Threshold line at 100ms (p50), 500ms (p95), and 1 second (p99). This panel shows whether queries are slow.

**Panel 5: Cache hit ratio.** A gauge showing the PostgreSQL shared buffer hit ratio. A hit ratio of 99%+ is normal. Below 95% indicates the cache is too small. This panel is important for diagnosing read performance issues.

**Panel 6: Disk I/O.** A line chart showing disk read and write throughput in MB/s. The chart shows a separate line for each node. High write throughput on the primary is normal. High write throughput on a replica is unusual and indicates a problem.

**Panel 7: Active queries.** A table showing the top 5 currently running queries by duration, with the query text, the user, and the database. This panel is useful for identifying long-running queries that are blocking other queries.

**Panel 8: WAL size.** A line chart showing the Write-Ahead Log size on each node. A growing WAL size indicates that the replicas are not keeping up (the primary keeps writing WAL because the replicas have not acknowledged the last write). A large WAL can cause the primary to run out of disk space.

## Case Study: Design a Real-Time Gaming Dashboard

Design a dashboard for an online multiplayer game with 1 million concurrent players. The dashboard is used by the game operations team (20 engineers) to monitor server health, player experience, and revenue.

**Model Answer:**

The dashboard system has four tiers, each serving a different audience.

**Tier 1: Executive Overview (C-level and producer).** A single-page dashboard with 6 large KPI panels: concurrent players (1.2M), revenue per minute ($4,200), average match latency (85ms), crash rate (0.3%), match abandonment rate (2.1%), and new user conversion rate (18%). Each panel shows the current value and a comparison to the same time last week. The dashboard refreshes every 60 seconds.

**Tier 2: Game Operations (ops team).** A dashboard with 12 panels showing real-time server health. The top row shows: total active game servers (2,400), servers by status (online, overloaded, offline), average CPU per server (62%), and average memory per server (71%). The second row shows the top 5 regions by player count and the top 5 regions by error rate. The third row shows a world map heatmap of player latency. The dashboard refreshes every 15 seconds.

**Tier 3: Matchmaking (matchmaking team).** A dashboard with 8 panels focused on the matchmaking system. The top row shows: matchmaking requests per minute, average time to find a match (12 seconds), match quality score (85/100), and match abandonment rate (2.1%). The second row shows a histogram of players by skill level (ELO distribution) and a heatmap of match quality by time of day. The dashboard refreshes every 30 seconds.

**Tier 4: Anti-Cheat (security team).** A dashboard with 6 panels focused on cheat detection. The top row shows: anti-cheat reports per minute, confirmed cheaters banned (per hour), false positive rate (0.01%), and reports under investigation. The second row shows the top 5 cheat types by volume (aimbot, wallhack, speedhack, esp, other) and a trend chart showing cheat detections over the last 7 days. The dashboard refreshes every 60 seconds.

**Key metrics for gaming dashboards:**
- Concurrent players (the most important metric — shows the game's health)
- Match latency (p50, p95, p99 — player experience)
- Server CPU and memory (infrastructure capacity)
- Match abandonment rate (symptom of problems — players leaving before the match starts)
- Crash rate (quality metric — how often the game crashes)
- Anti-cheat metrics (security and fairness)

## Case Study: Design an SLO Tracking Dashboard

Design a dashboard that tracks SLO attainment for 10 services. Each service has a 99.9% availability SLO over a 30-day rolling window. The dashboard is used by the VP of Engineering and the service teams.

**Model Answer:**

The SLO dashboard has three tiers.

**Tier 1: Executive Summary (1 panel).** A single big-number panel showing the organization-wide SLO attainment: the percentage of services currently meeting their SLO. If all 10 services are meeting their 99.9% target, the panel shows "100%." If 2 services are failing their SLO, the panel shows "80%." The panel is green when 100%, yellow when 80-99%, and red when below 80%. This panel is the only thing the VP looks at during the daily standup.

**Tier 2: Service Breakdown (10 panels in a table layout).** A table where each row is a service and the columns are: service name, SLO attainment (percentage), error budget remaining (minutes), burn rate (1x, 2x, 5x, 10x), and a sparkline showing the 7-day trend. Services are sorted by error budget remaining, with the service closest to exhausting its budget at the top. The table is color-coded: green rows for services with over 50% error budget remaining, yellow for 25-50%, red for under 25%.

**Tier 3: Service Detail (one per service, accessed by clicking a row).** A detail dashboard showing the selected service's SLO metrics over the full 30-day window. The top panel shows the SLO attainment as a percentage with a threshold line at 99.9%. The second panel shows the error budget burn rate over time, with annotations for deployments and incidents. The third panel shows the top 3 contributors to error budget consumption. The dashboard shows both the current 30-day window and a comparison to the previous 30-day window to show whether the service is improving or degrading.

**Key metrics:** SLO attainment (percentage), error budget remaining (minutes or requests), burn rate (multiplier of target), time to exhaustion (estimated time until error budget is exhausted at current burn rate).

## Case Study: Design a Dashboard for a Content Delivery Network

Design a dashboard for a CDN with 200 edge locations serving 1 million requests per second. The dashboard is used by the CDN operations team of 12 engineers.

**Model Answer:**

The CDN dashboard has three tiers.

**Tier 1: Global Overview.** This dashboard shows the health of the entire CDN. The top row shows: total requests per second, global cache hit ratio, global error rate (5xx from origin), and total data transfer in Gbps. The second row shows a world map heatmap of request density and latency by region. The third row shows the top 5 edge locations by request volume and the bottom 5 by error rate. The dashboard refreshes every 30 seconds.

**Tier 2: Regional Dashboard (one per geographic region).** This dashboard shows the health of all edge locations in a specific region. The top row shows the regional cache hit ratio, regional error rate, and regional average latency. The second row shows each edge location as a row in a table with columns: location name, requests per second, cache hit ratio, error rate, average latency, and a sparkline showing the 1-hour trend. The dashboard refreshes every 30 seconds.

**Tier 3: Edge Location Dashboard (one per location).** This dashboard shows the health of a single edge location. The top row shows: requests per second, cache hit ratio by content type (static, dynamic, API), origin response time (p50, p95, p99), and TLS handshake time. The second row shows the top 10 most-requested URLs by request count. The third row shows system metrics: CPU, memory, disk usage, and network I/O. The dashboard refreshes every 15 seconds.

## Case Study: Design a Dashboard Migration Strategy

Your organization currently uses 200 Grafana dashboards backed by a single Prometheus server. You are migrating to a new platform using Thanos and multiple Prometheus servers. Design the migration strategy with zero downtime and continuous monitoring coverage.

**Model Answer:**

The migration follows four phases over 8 weeks.

**Phase 1: Audit (Week 1).** Audit all 200 dashboards. For each dashboard, determine: is it actively used? (check the view count in Grafana's usage statistics). Does it still have valid queries? (check each query against the current Prometheus data). Who owns it? (check the dashboard metadata or ask the team). The audit will find that approximately 30% of dashboards are unused, 20% have broken queries, and 50% are actively maintained.

**Phase 2: Side-by-side deployment (Weeks 2-3).** Deploy the Thanos system alongside the existing Prometheus server. Configure Thanos to scrape the same targets as the existing Prometheus. Import the 100 actively maintained dashboards into Grafana using the dashboard API. Run both systems in parallel for 2 weeks, comparing query results to ensure the new system produces the same data.

**Phase 3: Cutover (Week 4).** Announce the cutover date. On the cutover day, update the DNS records to point the Grafana URL to the new system's load balancer. Keep the old Prometheus server running for 7 more days as a fallback. During the cutover week, monitor the new system's query latency and error rate. If any critical dashboard shows degraded performance, switch back to the old system.

**Phase 4: Cleanup (Weeks 5-8).** After the cutover, decommission the old Prometheus server. Archive the 60 unused dashboards (they can be restored if needed). Fix the 40 dashboards with broken queries and re-enable them. Train the teams on the new system's features (longer retention, cross-service queries, multi-region support).

**Key success metrics for the migration:**
- 100% of critical dashboards are available at all times during the migration
- Dashboard query latency is equal to or better than the old system (p99 under 500ms)
- No more than 5 support tickets per week during the cutover phase
- All teams have completed training within 2 weeks of the cutover

**Key metrics for CDN dashboards:**
- Cache hit ratio (the single most important CDN metric — a low hit ratio means the CDN is not effective)
- Origin response time (how long the origin server takes to respond to cache misses)
- TLS handshake time (how long the TLS connection takes to establish)
- Data transfer per location (used for capacity planning and billing)
- Error rate by edge location (identifies failing edge locations)

## Case Study: Design a Multi-Cloud Dashboard

Design a dashboard system that monitors services running across AWS, GCP, and Azure. Each cloud provider has its own monitoring tools (CloudWatch, Google Cloud Monitoring, Azure Monitor). The dashboard must provide a unified view across all three providers without vendor lock-in.

**Model Answer:**

The dashboard uses a unified metric layer that abstracts away provider-specific differences.

**Unified metric taxonomy.** A common set of metric names is defined across all three providers. The taxonomy maps provider-specific metric names to the unified names: `aws.ec2.cpu` → `cloud.cpu.utilization`, `gcp.compute.instance.cpu.utilization` → `cloud.cpu.utilization`, `azure.vm.percentage.cpu` → `cloud.cpu.utilization`. The mapping is configured in a central configuration database and applied by the ingestion pipeline.

**Provider-specific scraper agents.** Each cloud provider has a dedicated scraper agent that reads metrics from the provider's monitoring API and converts them to the unified format. The AWS scraper reads from CloudWatch, the GCP scraper reads from Google Cloud Monitoring, and the Azure scraper reads from Azure Monitor. Each scraper runs in its own region and is isolated from the others.

**Unified TSDB.** All scraped metrics are written to a single TSDB cluster (Thanos or VictoriaMetrics). The unified TSDB provides a single query interface for all dashboards. Dashboard queries use the unified metric names and are provider-agnostic.

**Provider-specific dashboards.** Each cloud provider has a dedicated dashboard that shows provider-specific metrics (AWS has EC2 instance health, GCP has GKE cluster status, Azure has Azure SQL performance). These dashboards use the unified metric names with the `cloud=aws` label filter.

**Cross-cloud dashboard.** A global dashboard shows the health of all services across all three providers. The top row shows cloud-level health: AWS health (99.95%), GCP health (99.97%), Azure health (99.92%). The second row shows the top 5 services by error rate across all clouds. The third row shows cost per cloud provider (a business metric that helps optimize multi-cloud spending).

## Appendix: Self-Evaluation Rubric

Use this rubric to evaluate your own dashboard designs. Score yourself from 1 (novice) to 5 (expert) in each dimension. A score of 30 or higher indicates you are ready for a senior-level system design discussion on dashboards.

### Data Ingestion (score 1-5)

| Score | Description |
|-------|-------------|
| 1 | Single Prometheus server scraping all services |
| 2 | Multiple Prometheus servers with federation |
| 3 | Message queue (Kafka) between scrapers and TSDB |
| 4 | Sharded ingestion with tenant isolation and rate limiting |
| 5 | Adaptive scrape intervals, edge aggregation, and intelligent sampling |

### Storage Design (score 1-5)

| Score | Description |
|-------|-------------|
| 1 | All data stored in a single TSDB at full resolution |
| 2 | Multiple TSDB instances with basic sharding |
| 3 | Compression (Gorilla), rollups at 3 tiers, inverted index |
| 4 | Multi-tier storage (SSD hot, HDD warm, S3 cold), consistent hashing |
| 5 | Cost-optimized tiered storage with adaptive retention policies |

### Query Performance (score 1-5)

| Score | Description |
|-------|-------------|
| 1 | Every dashboard query hits the TSDB directly |
| 2 | Query result caching with fixed TTL |
| 3 | Multi-level cache (browser, CDN, Redis, TSDB), query timeout |
| 4 | Query prioritization, tenant-level rate limiting, scan budget |
| 5 | Predictive pre-fetching, materialized views for common queries |

### Dashboard Layout (score 1-5)

| Score | Description |
|-------|-------------|
| 1 | Random arrangement of 20+ panels |
| 2 | Top-down hierarchy with 12-15 panels, consistent color scheme |
| 3 | Drill-down paths, annotations, previous-period comparisons |
| 4 | SLO-based layout, burn-rate panels, incident dashboard templates |
| 5 | Adaptive layouts, automatic panel generation from service catalogs |

### Scalability and Reliability (score 1-5)

| Score | Description |
|-------|-------------|

## Final Words

Dashboards are the most visible artifact of your monitoring infrastructure. Every engineer in the organization uses them. Every incident starts with someone opening a dashboard. The quality of your dashboards determines how quickly your team can detect, diagnose, and resolve incidents.

The principles in this chapter apply to any scale, from a startup with 5 services to an enterprise with 10,000 services. Start with the four golden signals. Use the F-pattern layout. Show thresholds on every panel. Add annotations for every deployment. Review your dashboards quarterly. The time you invest in dashboard design pays back every time an engineer opens a dashboard instead of running a terminal command.

## Challenge Solutions Verification

After completing each challenge, verify your solution against these criteria:

**Challenge 1 (Single-Service Dashboard):** Verify that you have exactly 8 metrics, that the error rate is the first metric, and that each metric has a specific reason for being on the dashboard. If any metric is included "because it seems useful," remove it.

**Challenge 2 (Identify Dashboard Problem):** Verify that you identified at least 5 problems. The most important problems are: no error rate percentage, averaging hides outliers, and no latency metric. If you missed these three, review the trade-offs section.

**Challenge 3 (Optimize a Dashboard Query):** Verify that you proposed at least 3 optimizations. The rollup optimization is the most impactful. If you did not mention rollups, review the deep dive section.

**Challenge 4 (Real-Time Video Dashboard):** Verify that your design handles 10 million concurrent viewers, updates every 10 seconds, and shows the last 6 hours. The key insight is that each region must have its own dashboard stack to avoid cross-region latency.

**Challenge 5 (Multi-Tenant Dashboard):** Verify that your design includes authentication (SSO), authorization (namespace-based RBAC), data isolation (sharding by namespace), and rate limiting (per-team quota). If you missed rate limiting, review the architecture section.

**Challenge 6 (Cost-Optimized Storage):** Verify that your design includes four storage tiers (hot, warm, cold, archive) and that the total cost is under $50,000 per year. The key insight is that rollups reduce storage by orders of magnitude.

**Challenge 7 (Anomaly Detection):** Verify that your design includes seasonal decomposition, multi-metric correlation, and dashboard integration. The key insight is that a single-metric spike is often a false alarm, but a correlated spike across multiple metrics is almost certainly a real incident.

**Challenge 8 (Fintech Dashboard):** Verify that your design includes immutability, audit trail, 99.99% availability, data residency, and RBAC. The key insight is that regulatory requirements drive the architecture more than performance requirements.
