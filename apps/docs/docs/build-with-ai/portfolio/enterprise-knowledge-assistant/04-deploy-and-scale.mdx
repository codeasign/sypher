## Deploy It

### Docker Compose

```yaml
version: "3.8"
services:
  knowbot:
    build: .
    ports:
      - "8005:8005"
    environment:
      - DB_PATH=/data/knowbot.db
      - CHROMA_PATH=/data/chroma
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
      - SLACK_BOT_TOKEN=${SLACK_BOT_TOKEN:-}
      - SLACK_APP_TOKEN=${SLACK_APP_TOKEN:-}
      - NOTION_API_KEY=${NOTION_API_KEY:-}
    volumes:
      - ./data:/data
```

## Scale It

### Capacity Model

| Component | 10 users | 100 users | 1,000 users |
|---|---|---|---|
| ChromaDB queries | 50/day | 500/day | 5,000/day |
| LLM calls | 30/day | 300/day | 3,000/day |
| Monthly cost | ~$2 | ~$15 | ~$150 |
| Storage | 50MB | 500MB | 5GB |

### First Bottleneck

ChromaDB collection-per-tenant works well until 100+ tenants. At that point, switch to a single collection with tenant_id metadata field and row-level filtering.

## Portfolio Presentation

### 3-Sentence Summary

"I built KnowBot, a multi-tenant enterprise knowledge assistant that indexes documents from Notion, GitHub, and Markdown sources and provides a unified chat interface with tenant-scoped access control. It uses ChromaDB collection-per-tenant for natural data isolation, SQLite for caching and audit logging, and Claude Haiku for cost-effective query responses. The system handles 100+ queries/day at under $0.02 per query and includes Slack integration for easy access."

### 5 Technical Decisions to Highlight

1. **Collection-per-tenant** — zero chance of cross-tenant data leaks
2. **Modular ingestion** — add a new source type with one module
3. **Query caching** — 40% cache hit rate reduces cost and latency
4. **Slack-first interface** — meets users where they already work
5. **Audit logging** — every query is traceable for compliance
