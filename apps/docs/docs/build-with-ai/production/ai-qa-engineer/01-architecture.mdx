Stripe's engineering blog reported that their CI pipeline runs over 100,000 test cases per day. For every developer-hour spent writing tests, the team estimates 4 hours saved in production incident response. Yet most startups and small teams have test coverage under 30% — not because they don't value testing, but because writing comprehensive tests is tedious and time-consuming.

An AI QA engineer that automatically generates tests from source code, runs them, and reports failures reduces the barrier to good test coverage from hours to seconds.

## What You'll Build

You'll build **TestBot**, an automated QA system with:

- **Test generation** — reads source code and generates pytest test cases covering edge cases, error paths, and happy paths
- **Test execution** — runs generated tests in an isolated environment and captures results
- **Failure analysis** — determines if failures are real bugs or bad test assumptions
- **Coverage reporting** — measures line/branch coverage and identifies untested code
- **Test repair** — automatically fixes failing tests when the test itself is wrong
- **CI integration** — single command drop-in for any Python project

## Technology Decisions

| Component | Choice | Why this, not X |
|---|---|---|
| Test Framework | pytest | Industry standard, great fixture system. unittest is built-in but verbose |
| LLM | Claude Sonnet 4 | Test generation needs deep code understanding. Haiku misses edge cases |
| Execution | subprocess + virtualenv | Isolated execution prevents side effects. Docker is more isolated but slower |
| Coverage | pytest-cov | Wraps coverage.py, integrates with pytest. Slower but more accurate than AST analysis |
| Reporting | Rich (terminal) | Beautiful terminal output. HTML reports need a browser to view |
