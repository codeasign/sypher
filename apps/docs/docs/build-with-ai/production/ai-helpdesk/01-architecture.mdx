Intercom's 2024 customer support benchmark report found that companies receive an average of 1,200 support tickets per month, with 62% requiring under 5 minutes of agent time to resolve. That's 744 hours per month spent on tickets that could be fully or partially automated — a cost of roughly $18,000/month for a mid-size support team at $25/hour. The problem compounds during product launches, when ticket volume can spike 5-10x and wait times balloon to hours.

A well-built AI helpdesk can auto-resolve 40-60% of tickets on first contact, route the rest to the right agent, and surface the most relevant KB articles for every response.

## What You'll Build

You'll build **HelpBot**, an intelligent helpdesk system with:

- **Ticket ingestion** via simulated email/webhook endpoint that categorizes, prioritizes, and routes tickets
- **Intent classification** using Claude Haiku — identifies the issue type, urgency, and required team
- **Auto-resolution engine** that searches your knowledge base and generates a complete answer when confidence is high
- **Escalation workflow** that hands off to human agents with full context when auto-resolution can't handle it
- **Analytics dashboard** (simple CLI reports) showing resolution rate, response times, and common issues
- **Ticket status tracking** — open, in-progress, resolved, escalated

### Data Flow

1. Ticket arrives via API → stored in SQLite with status `open`
2. Intent classification runs: identifies category (billing/technical/account), sentiment, and priority
3. KB search runs against indexed articles using hybrid search (vector + keyword)
4. Confidence score computed: if over 0.8, auto-generate answer and set status `resolved`
5. If under 0.8, assign to the appropriate team and set status `escalated`
6. Human agent sees full ticket context + LLM draft answer on the escalation

## Technology Decisions

| Component | Choice | Why this, not X |
|---|---|---|
| LLM | Claude Haiku 3.5 | Fast and cheap for classification (under $0.01 per ticket). Sonnet would add accuracy but 5x cost for marginal gain |
| KB Storage | SQLite + ChromaDB | Same pattern as the RAG system — keeps dependencies minimal. Notion KB would require API access per-tenant |
| Ticket Store | SQLite | Simple, relational, works for single-team. PostgreSQL needed at multi-team scale |
| Classification | Structured output via LLM | No training data needed. Fine-tuned classifier is more accurate but requires labeled historical tickets |
| Escalation | Rule-based (confidence threshold) | Simple, auditable, easy to tune. ML-based routing is overkill for early stages |

## Data Model

```python
from pydantic import BaseModel, Field
from typing import Optional, Literal
from datetime import datetime
import uuid

class Ticket(BaseModel):
    id: str = uuid.uuid4().hex[:12]
    subject: str
    description: str
    customer_email: str
    status: Literal["open", "in_progress", "resolved", "escalated"] = "open"
    priority: Literal["low", "medium", "high", "critical"] = "medium"
    category: Optional[str] = None
    assigned_team: Optional[str] = None
    confidence_score: float = 0.0
    resolution: Optional[str] = None
    created_at: str = datetime.utcnow().isoformat()
    resolved_at: Optional[str] = None

class KBArticle(BaseModel):
    id: str = uuid.uuid4().hex[:12]
    title: str
    content: str
    category: str
    tags: list[str] = []

class ClassificationResult(BaseModel):
    category: Literal["billing", "technical", "account", "general", "feature_request"]
    priority: Literal["low", "medium", "high", "critical"]
    sentiment: Literal["positive", "neutral", "frustrated", "angry"]
    required_team: Literal["support", "engineering", "billing", "product"]
    confidence: float = Field(..., ge=0.0, le=1.0)
    requires_escalation: bool
    escalation_reason: Optional[str] = None
```

## API Surface

| Method | Path | Input | Output |
|---|---|---|---|
| POST | `/api/v1/tickets` | `{ subject, description, customer_email }` | `{ ticket_id, status, confidence, resolution }` |
| GET | `/api/v1/tickets/{id}` | path param | `{ ticket details }` |
| GET | `/api/v1/tickets` | `?status=&team=` | `[{ tickets }]` |
| POST | `/api/v1/kb/articles` | `{ title, content, category, tags }` | `{ article_id }` |
| GET | `/api/v1/kb/search` | `?q=` | `[{ articles }]` |
