It is Tuesday afternoon. You just walked out of back-to-back meetings — a sprint retro, a product review, and a cross-team sync. Your notebook has three pages of scribbled notes, a voice memo, and a calendar invite that says "Next Steps" with no next steps. By Friday, nobody remembers who was supposed to follow up on the migration deadline. The information was all said out loud; it just wasn't captured in a way anyone could act on.

A meeting transcript is raw data. A good summary is an action plan. The difference is structure: extracting who said what, what was decided, who owns each action item, and what the blockers are. In this project you'll build a tool that takes a meeting transcript — paste one in from Otter.ai, Microsoft Teams, or Zoom — and produces a clean, structured summary with decisions, action items, open questions, and a timeline of discussion.

## What You'll Build

A Python CLI tool that processes meeting transcripts and produces structured summaries. The system:

- Accepts raw transcript text or a text file path as input
- Analyses the transcript using an LLM to extract key discussion points, decisions, and action items
- Outputs a structured Markdown summary with sections for overview, discussion topics, decisions, action items, and open questions
- Supports different meeting types (standup, retro, planning, 1:1) with tailored summary templates
- Optionally saves summaries to a local file with timestamps for future reference

## Concepts You Need

**Structured output from unstructured text.** An LLM can read a rambling conversation and extract a structured schema from it — decisions, owners, deadlines. Instead of asking "summarise this," you specify the exact output format (JSON with `decisions`, `action_items`, `open_questions` arrays) and the model fills it in. This is called structured output or constrained generation.

**Prompt engineering for information extraction.** The quality of the summary depends entirely on how you frame the extraction task. A generic "summarise this meeting" prompt produces a paragraph. A prompt that says "List every decision made, the person who made it, and the context" produces a structured table. You'll learn to write extraction prompts that yield specific, consistent fields.

**Meeting-type-specific templates.** A daily standup has a different structure than a sprint retro. A standup has three questions (what I did, what I'll do, blockers). A retro has a format (what went well, what to improve, action items). By detecting the meeting type and using a tailored prompt template, you get summaries that match the format people expect from each meeting genre.

### Step 2: Define the meeting type detection

Add a function that reads the transcript and identifies the meeting type:

```python
def detect_meeting_type(transcript: str) -> str:
    """Detect the meeting type from the transcript content."""
    keywords = {
        "standup": ["standup", "yesterday", "today", "blocker", "what did you"],
        "retro": ["retro", "sprint retrospective", "what went well", "start doing"],
        "planning": ["planning", "sprint planning", "estimate", "story points", "backlog"],
        "one-on-one": ["1:1", "one-on-one", "1-1", "career", "feedback"],
        "brainstorming": ["brainstorm", "ideate", "think of ideas", "what if"],
    }
    transcript_lower = transcript.lower()
    scores = {}
    for meeting_type, words in keywords.items():
        score = sum(1 for word in words if word in transcript_lower)
        if score > 0:
            scores[meeting_type] = score
    if not scores:
        return "general"
    return max(scores, key=scores.get)
```


**What happened:** The function uses keyword scoring to guess the meeting type. This is a simple heuristic — lightweight, fast, and accurate enough for most transcripts. The detected type determines which prompt template is used for summarisation.

### Step 5: Build the summarisation function

Add the core function that calls the LLM:

```python
def summarize_transcript(
    transcript: str,
    meeting_type: str = None,
    custom_context: str = None
) -> str:
    """Summarize a meeting transcript using the LLM."""
    if not meeting_type:
        meeting_type = detect_meeting_type(transcript)

    template = PROMPT_TEMPLATES.get(meeting_type, PROMPT_TEMPLATES["general"])

    system_prompt = (
        f"{template}\n\n"
        "Be specific: include names, numbers, and concrete details from the transcript. "
        "If a decision or action item has no clear owner, note that. "
        "Do not add information not present in the transcript."
    )

    user_message = f"Transcript:\n\n{transcript}"
    if custom_context:
        user_message = f"Context: {custom_context}\n\n{user_message}"

    response = client.messages.create(
        model="claude-3-5-haiku-20241022",
        max_tokens=2000,
        system=system_prompt,
        messages=[{"role": "user", "content": user_message}]
    )

    summary = response.content[0].text
    input_tokens = response.usage.input_tokens
    output_tokens = response.usage.output_tokens
    cost = (input_tokens * 3e-6) + (output_tokens * 15e-6)
    print(f"Processed {input_tokens} input tokens, {output_tokens} output tokens "
          f"(~${cost:.4f})")

    return summary
```

### Step 6: Add file I/O support

Add functions to read from a file and save the summary:

```python
def load_transcript(file_path: str) -> str:
    """Load a transcript from a text file."""
    path = Path(file_path)
    if not path.exists():
        print(f"Error: file not found: {file_path}")
        sys.exit(1)
    return path.read_text(encoding="utf-8")

def save_summary(summary: str, output_dir: str = "summaries") -> str:
    """Save the summary to a timestamped file."""
    output_path = Path(output_dir)
    output_path.mkdir(exist_ok=True)
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
    filename = output_path / f"meeting_summary_{timestamp}.md"
    filename.write_text(summary, encoding="utf-8")
    print(f"Summary saved to: {filename}")
    return str(filename)
```

**What happened:** `load_transcript` reads a transcript file from disk. `save_summary` writes the generated summary to a timestamped Markdown file in a `summaries/` directory. This makes it easy to keep a running archive of meeting summaries.

### Step 7: Build the interactive CLI

Add the main function that ties everything together:

```python
def main():
    print("=" * 50)
    print("  AI Meeting Notes Summarizer")
    print("=" * 50)

    source = input("\nEnter transcript text (T) or load from file (F)? ").strip().lower()

    if source == "f":
        file_path = input("Enter transcript file path: ").strip()
        transcript = load_transcript(file_path)
        print(f"Loaded {len(transcript)} characters from {file_path}")
    else:
        print("Paste the transcript below. Press Enter, then Ctrl+Z (Windows)")
        print("or Ctrl+D (macOS/Linux) on a blank line to finish:\n")
        lines = []
        try:
            while True:
                line = input()
                lines.append(line)
        except EOFError:
            pass
        transcript = "\n".join(lines)

    if not transcript.strip():
        print("Error: no transcript provided")
        sys.exit(1)

    # Detect and show type
    meeting_type = detect_meeting_type(transcript)
    print(f"\nDetected meeting type: {meeting_type}")

    # Optional custom context
    context = input("Add custom context (optional, press Enter to skip): ").strip()
    if not context:
        context = None

    print("\nGenerating summary...\n")
    summary = summarize_transcript(transcript, meeting_type, context)
    print("\n" + "=" * 50)
    print(summary)
    print("=" * 50)

    # Save
    save = input("\nSave summary to file? (y/n): ").strip().lower()
    if save == "y":
        save_summary(summary)

if __name__ == "__main__":
    main()
```

**What happened:** The main function presents a choice: paste transcript text directly or load from a file. It detects the meeting type, optionally accepts custom context (e.g. "This is the Q4 planning meeting"), generates the summary, and offers to save it.

## Understand What Happened

**What each API call actually did.** The Anthropic API call sent the system prompt (the meeting-type-specific template) plus the transcript as the user message. The model processed the input, extracted the structured fields, and returned a formatted Markdown summary. The system prompt acted as a constraint — telling the model what fields to extract and how to format them.

**Token usage and cost for a typical run.** A 10-minute standup transcript (~500 words) uses roughly 800–1,000 input tokens and generates 150–250 output tokens. At Claude Haiku pricing, that's about $0.001–$0.002 per meeting. An hour-long meeting transcript (~3,000 words) uses ~4,000–5,000 input tokens and costs ~$0.015. A full week of daily standups costs under $0.01.

**Performance characteristics.** The model is the bottleneck at 1–3 seconds per call. Loading from a file and detecting the meeting type are both instant. The system is IO-bound on the API call, not CPU-bound.

**3 common mistakes teams make with this pattern:**

- **Not including speakers in the transcript.** If you strip speaker labels, the LLM cannot attribute action items. Always include who said what — the summary quality drops sharply without speaker attribution.

- **Using a single generic prompt for all meetings.** A standup summary that includes "What Went Well" sections looks wrong. Meeting-type-specific templates produce summaries that match audience expectations and are immediately useful without mental restructuring.

- **Not handling very long transcripts.** A 2-hour planning meeting can exceed the context window if you send the full transcript. Split long transcripts into segments (e.g. by agenda item), summarise each segment, then summarise the segment summaries. This hierarchical approach handles any transcript length.
