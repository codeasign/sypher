You open a 50-page research report your colleague sent over Slack. Somewhere in there is the answer to the question your manager just asked in standup. You Ctrl+F for a keyword, skim three false positives, scroll some more, and eventually give up and re-read the whole executive summary. You are not alone — knowledge workers spend nearly 10 hours per week searching for and consolidating information across documents. The problem isn't that the information is missing; it's trapped inside PDFs designed for print, not for querying.

Retrieval-Augmented Generation (RAG) changes this. Instead of reading a PDF manually, you extract its text, split it into meaningful chunks, embed each chunk into a vector space, and let a language model find and read the most relevant passages for whatever question you ask. The result is a conversational interface to your documents — ask "What were the Q3 revenue drivers?" and get a paragraph synthesised from the actual pages that mention revenue, complete with context.

In this project you'll build exactly that: a command-line chat-with-PDF tool that loads a document, indexes it, and lets you ask questions in a natural back-and-forth conversation. By the end you'll understand the core RAG pipeline that powers tools like ChatGPT retrieval, Notion AI, and enterprise search systems.

## What You'll Build

A Python CLI tool that takes a PDF path and enters an interactive Q&A session. The system:

- Extracts text from any PDF (single or multi-page)
- Splits the text into overlapping chunks for precise retrieval
- Generates embeddings for each chunk using a local Sentence-Transformers model
- Stores embeddings in an in-memory FAISS vector index
- Retrieves the top-K most relevant chunks for each user question
- Passes the retrieved chunks as context to an LLM (Claude Haiku via API)
- Maintains conversation history so follow-up questions stay coherent

## Concepts You Need

**Text chunking with overlap.** Documents are too long to fit in an LLM context window, and small passages are easier to retrieve precisely. You split the text into chunks of a fixed size (e.g. 500 tokens) with a small overlap (e.g. 50 tokens) so that sentences aren't cut mid-thought. The overlap means a sentence that straddles two chunks appears in both, so retrieval doesn't miss it. For example, `"The Q3 revenue grew by 12% due to strength in the enterprise segment."` split at token 500 might lose the enterprise segment reference — overlap preserves it.

**Text embeddings for semantic search.** An embedding is a list of floats (a vector) that captures the meaning of a piece of text. Two chunks with similar meanings produce vectors that are close together in the vector space. You measure closeness with cosine similarity. Unlike keyword search (which fails on synonyms and paraphrases), embedding search finds "What were the main drivers of Q3 growth?" even when the PDF only says "enterprise segment strength drove revenue."

**FAISS for fast similarity search.** FAISS (Facebook AI Similarity Search) is an in-memory vector index that can find the nearest neighbours of a query vector across thousands of chunks in milliseconds. You build the index once at startup, then query it for every user question. For a 50-page PDF with ~150 chunks, a single query takes under 5 milliseconds on a laptop CPU.

**Retrieval-Augmented Generation (RAG).** The pattern that combines retrieval with generation: you retrieve the most relevant chunks from your index, inject them into a prompt along with the user's question, and ask the LLM to answer based only on that context. This grounds the LLM's answer in your document rather than its training data, reducing hallucinations and making answers verifiable.

### Step 2: Extract text from a PDF

Add a function that uses PyMuPDF (fitz) to read every page:

```python
import fitz  # PyMuPDF

def extract_text(pdf_path: str) -> str:
    """Extract all text from a PDF file."""
    doc = fitz.open(pdf_path)
    text_parts = []
    for page_num, page in enumerate(doc, 1):
        page_text = page.get_text()
        if page_text.strip():
            text_parts.append(f"--- Page {page_num} ---\n{page_text}")
    doc.close()
    full_text = "\n\n".join(text_parts)
    print(f"Extracted {len(full_text)} characters from {len(doc)} pages")
    return full_text
```

**What happened:** PyMuPDF opens the PDF and iterates every page. `page.get_text()` extracts the readable text while discarding layout information. Each page is prefixed with a page number marker so the LLM can cite which page a fact came from.

### Step 3: Chunk the text with overlap

Add a function that splits the extracted text into overlapping chunks:

```python
from typing import List

def chunk_text(text: str, chunk_size: int = 500, overlap: int = 50) -> List[str]:
    """Split text into overlapping chunks of approximately chunk_size tokens."""
    words = text.split()
    chunks = []
    start = 0
    while start < len(words):
        end = min(start + chunk_size, len(words))
        chunk = " ".join(words[start:end])
        chunks.append(chunk)
        if end == len(words):
            break
        start = end - overlap
    print(f"Created {len(chunks)} chunks")
    return chunks
```

**What happened:** The function splits the text into word tokens, then slides a window of `chunk_size` words across the text, stepping forward by `chunk_size - overlap` words each time. This means the last `overlap` words of one chunk are also the first `overlap` words of the next, preserving sentence continuity.

### Step 4: Load the embedding model and build the FAISS index

Add functions to load a Sentence-Transformers model and build the index:

```python
import numpy as np
import faiss
from sentence_transformers import SentenceTransformer

def load_embedding_model() -> SentenceTransformer:
    """Load a lightweight embedding model."""
    print("Loading embedding model...")
    model = SentenceTransformer("all-MiniLM-L6-v2")
    print("Embedding model loaded")
    return model

def build_index(embeddings: np.ndarray) -> faiss.IndexFlatL2:
    """Build a FAISS index from embedding vectors."""
    dimension = embeddings.shape[1]
    index = faiss.IndexFlatL2(dimension)
    index.add(embeddings)
    print(f"Index built with {index.ntotal} vectors, dimension {dimension}")
    return index
```

```bash
# Expected output:
# Loading embedding model...
# Embedding model loaded
# Index built with N vectors, dimension 384
```

**What happened:** `all-MiniLM-L6-v2` is a small, fast embedding model that produces 384-dimensional vectors. It runs locally on your CPU — no API call needed, no cost per query. FAISS stores these vectors in an `IndexFlatL2` (brute-force L2 distance) index. For a document with ~150 chunks, the index build takes under 2 seconds on a modern laptop.

### Step 5: Index a PDF end-to-end

Add the function that ties extraction, chunking, and indexing together:

```python
def index_pdf(pdf_path: str, embedding_model: SentenceTransformer) -> tuple:
    """Extract, chunk, and index a PDF. Returns (chunks, index)."""
    print(f"Indexing: {pdf_path}")
    raw_text = extract_text(pdf_path)
    chunks = chunk_text(raw_text)
    chunk_embeddings = embedding_model.encode(chunks, show_progress_bar=True)
    index = build_index(np.array(chunk_embeddings).astype("float32"))
    print(f"Indexed {len(chunks)} chunks")
    return chunks, index
```

**What happened:** This is the full indexing pipeline. The embedding model encodes each chunk into a 384-dimensional vector. FAISS is handed the full array of vectors and builds the index in memory. The function returns both the original text chunks (needed for retrieval display) and the vector index.

### Step 6: Retrieve relevant chunks

Add the retrieval function:

```python
def retrieve_chunks(
    query: str,
    chunks: List[str],
    index: faiss.IndexFlatL2,
    embedding_model: SentenceTransformer,
    top_k: int = 5
) -> List[str]:
    """Retrieve the most relevant chunks for a query."""
    query_embedding = embedding_model.encode([query])
    distances, indices = index.search(np.array(query_embedding).astype("float32"), top_k)
    retrieved = [chunks[i] for i in indices[0]]
    print(f"Retrieved {len(retrieved)} chunks for query")
    return retrieved
```

**What happened:** The query is embedded using the same model, then FAISS finds the `top_k` nearest neighbours in the vector space. The `distances` array tells you how far each chunk is from the query (lower = more similar). The corresponding chunk texts are returned so they can be used as context for the LLM.

### Step 7: Generate an answer with Claude Haiku

Add the generation function:

```python
def generate_answer(
    query: str,
    context_chunks: List[str],
    conversation_history: List[dict]
) -> str:
    """Generate an answer using Claude Haiku with retrieved context."""
    context = "\n\n".join(
        f"[Chunk {i+1}] {chunk}"
        for i, chunk in enumerate(context_chunks)
    )
    system_prompt = (
        "You are a helpful assistant that answers questions based on the provided "
        "document context. Answer concisely and accurately. If the context does not "
        "contain the answer, say so — do not make up information. "
        "Cite the relevant chunk numbers when possible."
    )
    messages = conversation_history + [
        {
            "role": "user",
            "content": f"Context from the document:\n\n{context}\n\n"
                       f"Question: {query}"
        }
    ]
    response = client.messages.create(
        model="claude-3-5-haiku-20241022",
        max_tokens=1000,
        system=system_prompt,
        messages=messages
    )
    answer = response.content[0].text
    # Track token usage
    input_tokens = response.usage.input_tokens
    output_tokens = response.usage.output_tokens
    print(f"  Tokens: {input_tokens} in / {output_tokens} out "
          f"(~${input_tokens * 3e-6 + output_tokens * 15e-6:.4f})")
    return answer
```

**What happened:** The retrieved chunks are assembled into a prompt with the system instruction to answer from context only. The conversation history is included so the LLM can follow multi-turn conversations ("tell me more about that" still works). The response's token usage is printed so you can track cost — a typical answer costs under $0.002.

### Step 8: Build the interactive CLI loop

Add the main loop that ties everything together:

```python
import shutil

def main():
    pdf_path = input("Enter PDF path: ").strip()
    if not os.path.exists(pdf_path):
        print(f"File not found: {pdf_path}")
        sys.exit(1)

    embedding_model = load_embedding_model()
    chunks, index = index_pdf(pdf_path, embedding_model)

    conversation_history = []
    print("\n" + "=" * 50)
    print("Ready! Ask questions about your PDF.")
    print("Type 'quit' to exit, 'history' to see conversation, 'clear' to reset")
    print("=" * 50 + "\n")

    while True:
        query = input("> ").strip()
        if query.lower() in ("quit", "exit"):
            break
        if query.lower() == "history":
            for msg in conversation_history:
                role = msg["role"].upper()
                content = msg["content"][:100]
                print(f"  {role}: {content}...")
            continue
        if query.lower() == "clear":
            conversation_history = []
            print("Conversation cleared")
            continue
        if not query:
            continue

        retrieved = retrieve_chunks(query, chunks, index, embedding_model)
        answer = generate_answer(query, retrieved, conversation_history)
        print(f"\n{answer}\n")

        conversation_history.append({"role": "user", "content": query})
        conversation_history.append({"role": "assistant", "content": answer})

        # Keep history manageable
        if len(conversation_history) > 10:
            conversation_history = conversation_history[-10:]

if __name__ == "__main__":
    main()
```

**What happened:** The main function prompts for a PDF path, indexes it, then enters an interactive loop. Each question is embedded, used to retrieve relevant chunks, and passed to the LLM with conversation history. The loop supports `history` (view conversation), `clear` (reset history), and `quit` (exit).

## Understand What Happened

**What each API call actually did.** The Sentence-Transformer model call (`model.encode()`) created a 384-dimensional vector representation of each chunk — this ran entirely on your CPU, no network, no cost. The FAISS index search (`index.search()`) compared your query vector against all 90 chunk vectors using L2 distance and returned the 5 closest matches in under 5 milliseconds. The Anthropic API call sent the retrieved chunks plus your question to Claude Haiku, which generated a grounded answer using only the provided context.

**Token usage and cost for a typical run.** Indexing a 10-page PDF costs $0.00 — embedding is local. Each question costs approximately 1,000–1,500 input tokens (for the system prompt, context chunks, and conversation history) and 100–300 output tokens, totalling roughly $0.001–$0.002 per question. A full 30-minute study session with 20 questions costs under $0.05.

**Performance characteristics.** The embedding model loads in ~2 seconds on first run and encodes ~100 chunks in under 3 seconds. FAISS search is sub-millisecond for a few hundred vectors. The API call is the bottleneck at 1–3 seconds per question. The system is entirely CPU-bound for the local parts and network-bound for the API call.

**3 common mistakes teams make with this pattern:**

- **Chunking without overlap.** Without overlap, a sentence that spans two chunks may be incomplete in both, causing retrieval to miss it. The answer then lacks context that the document actually contains. Always use at least 10–15% overlap relative to chunk size.

- **Using the same model for indexing and querying.** If you index with one embedding model and query with a different one, the vector spaces are incompatible and retrieval quality drops sharply. Always use the same `SentenceTransformer` model instance (or at minimum the same model name) for both sides.

- **Ignoring the system prompt guardrails.** Without a prompt that says "answer from context only, say if you don't know," the LLM is likely to hallucinate facts that sound plausible but aren't in the document. This defeats the purpose of RAG. Always include a strict context-grounding instruction.
