## Summary

This page covers three variations on Binary Search that each handle a trickier version of the same problem. Find First/Last Occurrence answer "where does this repeated value start or end?" in a sorted list with duplicates. Search Rotated Array answers "can I still binary-search a sorted list that's been spun around, like a deck of cards cut in the middle?" All three still work by repeatedly narrowing down a range, just with a slightly smarter rule for which half to keep вҖ” so they're all roughly as fast as regular Binary Search.

## Pseudocode

```text
findFirstOccurrence(array, target):
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. Look at the middle item.
4. If it matches the target, remember this position as the best answer so far, then keep looking to the left for an even earlier match.
5. If the middle item is too small, move low past it.
6. If the middle item is too big, move high before it.
7. When low passes high, report the best answer remembered (or "not found" if none was ever recorded).

findLastOccurrence(array, target):
Same as above, but on a match, remember the position and keep looking to the right for an even later match instead.

searchRotated(array, target):
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. Look at the middle item. If it matches the target, stop and report its position.
4. Figure out which half (left or right of the middle) is still in normal sorted order вҖ” the list was cut and rejoined at one point, so only one half is "clean."
5. If the target's value would fit inside that clean half's range, search there; otherwise search the other half.
6. If low passes high with no match, report "not found."
```

### Recursive Version

```text
findFirstOccurrence(array, target, low, high):
1. If low has passed high, report "not found."
2. Look at the middle item between low and high.
3. If it matches the target, save that position, then recurse on the LEFT half (low to mid - 1) вҖ” if that recursive call finds nothing, fall back to the position saved here.
4. If the middle item is too small, recurse on the right half (mid + 1 to high).
5. If the middle item is too big, recurse on the left half (low to mid - 1).

findLastOccurrence(array, target, low, high):
Same shape, but on a match recurse on the RIGHT half (mid + 1 to high) instead, falling back to the saved position.

searchRotated(array, target, low, high):
1. If low has passed high, report "not found."
2. Look at the middle item. If it matches the target, stop and report its position.
3. Work out which half (low to mid, or mid to high) is still in clean sorted order.
4. If the target's value fits inside that clean half's range, recurse into that half; otherwise recurse into the other half.
```

## Worked Example

These variants only work on a list that's already sorted (or, for the rotated case, sorted and then cut once and rejoined) вҖ” like looking up a word in a dictionary instead of a shuffled stack of index cards.

## Complexity

- **Best case:** for Search Rotated Array, if the very first middle guess happens to be the target, you're done in one step (`O(1)`). The occurrence variants don't get this shortcut вҖ” even a lucky first match doesn't end the search early, since they must keep narrowing to confirm no earlier or later match exists.
- **Average case:** all three variants still cut the remaining range roughly in half every step, the same way plain Binary Search does вҖ” that's about `log2(n)` steps for `n` items (`O(log n)`).
- **Worst case:** even when the target isn't in the list at all, or you need to squeeze all the way to the exact edge of a run of duplicates, you keep halving the range until it's empty вҖ” still `O(log n)`, just with every step used up.
- **Space:** the iterative version above only moves a couple of pointers, so it uses a constant, tiny amount of extra memory (`O(1)`). The recursive version adds one stack frame per halving, so it uses a small, growing amount of memory proportional to the number of steps (`O(log n)`).

## Applications

- C++'s `std::lower_bound` / `std::upper_bound` and Python's `bisect.bisect_left` / `bisect.bisect_right` implement exactly the find-first/find-last idea, for inserting into or querying ranges of sorted data.
- Database engines answer range queries on a sorted index вҖ” "every order placed on this exact date" вҖ” by finding the first and last matching row instead of scanning every row.
- Log analysis tools binary-search a sorted-by-timestamp log file to find the first entry at or after a given time, or the last entry before a cutoff.
- Version control tools like `git bisect` narrow down a sorted sequence of commits to find the exact one that introduced a bug вҖ” the same "keep halving" idea as Search Rotated Array applied to a linear history.
- Autocomplete and dictionary lookups find the first and last index of every word sharing a prefix in a sorted word list, using the same occurrence-boundary trick.
