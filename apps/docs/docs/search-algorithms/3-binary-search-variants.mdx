## Summary

This page covers three variations on Binary Search that each handle a trickier version of the same problem. Find First/Last Occurrence answer "where does this repeated value start or end?" in a sorted list with duplicates. Search Rotated Array answers "can I still binary-search a sorted list that's been spun around, like a deck of cards cut in the middle?" All three still work by repeatedly narrowing down a range, just with a slightly smarter rule for which half to keep — so they're all roughly as fast as regular Binary Search.

## Pseudocode

```text
findFirstOccurrence(array, target):
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. Look at the middle item.
4. If it matches the target, remember this position as the best answer so far, then keep looking to the left for an even earlier match.
5. If the middle item is too small, move low past it.
6. If the middle item is too big, move high before it.
7. When low passes high, report the best answer remembered (or "not found" if none was ever recorded).

findLastOccurrence(array, target):
Same as above, but on a match, remember the position and keep looking to the right for an even later match instead.

searchRotated(array, target):
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. Look at the middle item. If it matches the target, stop and report its position.
4. Figure out which half (left or right of the middle) is still in normal sorted order — the list was cut and rejoined at one point, so only one half is "clean."
5. If the target's value would fit inside that clean half's range, search there; otherwise search the other half.
6. If low passes high with no match, report "not found."
```

## How It Works

- **Find First/Last Occurrence:** picture a sorted list `[1, 2, 2, 2, 3]` and you want to know where the run of `2`s starts. Regular Binary Search would stop the instant it finds any `2`. This variant is more stubborn: when it finds a `2`, it writes down "found one here" but keeps narrowing toward the left (for "first") or right (for "last") to see if there's an even earlier or later `2`. It only stops for good once the search range runs out, at which point the last position it wrote down is the true boundary.
- **Search Rotated Array:** picture the sorted list `[4, 5, 6, 7, 0, 1, 2]` in the diagram above — it's sorted, but it's been cut after `7` and the front chunk moved to the back, like cutting a deck of cards. Looking at the middle item (`7`, at index 3) tells us the left side (`4` through `7`) is still in clean ascending order. Since the target `0` doesn't fall inside that range, we know it must be hiding in the other half, so we search there instead — index 4, where `0` sits.
- This only works if the list was sorted before being "cut" — a list rotated in this specific way, with no duplicate values for the rotated variant. A list in random order breaks the trick.
- How fast is it? All three variants still cut the range roughly in half every step, just like plain Binary Search, so they're all fast — best case you find it almost instantly, worst case you narrow down until only one spot is left (this is still `O(log n)` time). The occurrence variants do a little extra work even after finding a match, since they keep looking for a better boundary instead of stopping right away.
