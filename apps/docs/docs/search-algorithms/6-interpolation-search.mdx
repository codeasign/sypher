## Summary

Interpolation Search finds something in a sorted list of numbers by guessing where the target probably is, based on its value, instead of always guessing the middle. Think of a phone book: if you're looking for "Zamora," you'd flip near the very end, not the middle, because you know the alphabet is spread out evenly. This algorithm does the same math trick with numbers. When the numbers are spread out evenly, it's even faster than Binary Search — but if the numbers are bunched up unevenly, its guesses get bad and it can slow down a lot.

## Pseudocode

```text
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. If every remaining number in the range is the same value, just check whether it equals the target directly, and stop.
4. Otherwise, guess a position based on how close the target's value is to the low end versus the high end of the current range — like estimating a page number by value.
5. If that guessed position falls outside the current range, report "not found."
6. If the item at the guessed position is the target, stop and report its position.
7. If it's smaller than the target, keep searching to the right of the guess.
8. If it's bigger than the target, keep searching to the left of the guess.
9. If low ever passes high without a match, report "not found."
```

## Worked Example

We'll search for `85` in this sorted list of ten evenly spaced numbers: `10, 20, 30, ..., 100`. This only works because the list is sorted and the values are spread out uniformly — the guess is only trustworthy if the numbers rise at a fairly steady rate, the same way you can only estimate a page by letter if the alphabet is evenly spread through a phone book.

Instead of guessing the middle (index 4 or 5), the formula weights the guess toward where `85` would land if the values were a straight line — index 7. `array[7]` is `80`, which is smaller than `85`, so the target must be further right. We move `low` up to index 8.

Only two values are left: `90` and `100`. Plugging them into the formula gives a probe position of `7` — but that index is outside the current `low`-to-`high` range. That's the algorithm's built-in safety check catching an impossible guess, and it's how Interpolation Search knows `85` isn't in the list without checking every remaining item one by one.

## Complexity

- **Best case:** If the very first guess lands exactly on the target — which happens a lot with evenly spread numbers, since the formula computes an almost-exact position on the first try — the search finishes almost instantly (`O(1)`).
- **Average case:** On numbers that are spread out evenly, each guess gets far closer to the target than Binary Search's plain middle-guess, so it typically takes only about `log(log(n))` guesses — for a million items, sometimes fewer than 5 (`O(log log n)`).
- **Worst case:** If the numbers are bunched up unevenly — a few huge outliers next to a lot of small, close-together numbers — the guesses can be way off almost every time, and it degrades to checking almost every item one by one, same as Linear Search (`O(n)`).
- **Space:** It only tracks a couple of numbers — the current `low`, `high`, and the guessed position — so it uses constant extra memory (`O(1)`).

The catch is that all of this speed depends on the data being roughly evenly spread. Binary Search doesn't care how the values are distributed because it always guesses the middle; Interpolation Search only pays off when its value-based guess is actually a good one.

## Applications

- **Looking up a name in a phone book or a word in a dictionary** — the classic real-world picture for this algorithm, since names and words are roughly evenly spread across the alphabet.
- **Searching large sorted numeric database indexes**, such as sequential order IDs, invoice numbers, or auto-incrementing primary keys, where the evenly spaced keys let interpolation outperform a plain middle-guessing search.
- **Searching sorted log files or time-series records indexed by timestamp**, since timestamps recorded at regular intervals are close to uniformly distributed.
- **Locating values in sorted numeric sensor or telemetry data** in IoT and monitoring pipelines, where readings are indexed evenly over time.
