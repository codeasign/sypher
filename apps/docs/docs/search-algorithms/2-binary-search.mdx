## Summary

Binary Search finds something in a sorted list by always checking the middle item first, then throwing away the half of the list that can't have the answer. Imagine guessing a number between 1 and 100: instead of counting up one by one, you'd guess 50, hear "higher" or "lower," and jump straight to the middle of whichever half is left. It repeats that "check the middle, discard a half" move until it finds the target or runs out of list. Because it throws away half the remaining items every single step, it's dramatically faster than checking one item at a time — but it only works if the list is already sorted.

## Pseudocode

```text
1. Set a "low" marker at the start of the list and a "high" marker at the end.
2. While the low marker hasn't passed the high marker:
3. Find the item exactly in the middle of low and high.
4. If that middle item is the target, stop and report its position.
5. If the middle item is smaller than the target, the target must be further right — move the low marker to just past the middle.
6. If the middle item is bigger than the target, the target must be further left — move the high marker to just before the middle.
7. If the low marker ever passes the high marker without a match, report "not found."
```

### Recursive Version

Binary search also has a natural recursive shape: instead of a loop, it calls itself on a smaller half of the list each time.

```text
1. If the low marker has passed the high marker, report "not found."
2. Find the item exactly in the middle of low and high.
3. If that middle item is the target, stop and report its position.
4. If the middle item is smaller than the target, call yourself again, but only on the right half (low = middle + 1).
5. If the middle item is bigger than the target, call yourself again, but only on the left half (high = middle - 1).
```

## Complexity

- **Best case:** the target is the very first middle item you check, so it's found instantly (`O(1)`).
- **Average case:** each comparison throws away half of what's left, so it typically takes only a handful of steps even on large lists (`O(log n)`).
- **Worst case:** the target is missing, or it's the last item left after every possible halving, but halving shrinks things so fast this still finishes quickly (`O(log n)`).
- **Space:** the iterative version only needs a couple of markers, no extra memory that grows with the list (`O(1)`). The recursive version needs a little extra memory for each nested call, growing with the number of halvings (`O(log n)`).

## Applications

- Standard library search functions, like Python's `bisect` module and Java's `Arrays.binarySearch` / `Collections.binarySearch`, use binary search under the hood.
- Version control bisecting, like `git bisect`, uses the same "check the middle, discard a half" idea to find the commit that introduced a bug.
- Database and index lookups on sorted keys (e.g. B-tree index searches) narrow down a match the same way instead of scanning every row.
- Autocomplete and spell-check dictionaries that store words in sorted order use it to check whether a word exists without scanning the whole list.
- Debugging techniques like finding the point where a monotonic value (e.g. a sensor reading) crosses a threshold, by repeatedly checking the middle of a range.
