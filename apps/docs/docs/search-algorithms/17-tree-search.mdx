## Summary

Tree Search looks for a value in a branching tree structure by exploring one branch all the way down before backing up and trying the next one, keeping track of the exact trail of values it followed. Think of it like exploring a family tree, checking a person, then all of their descendants down one side, before moving to the other side. Because the values aren't arranged in any particular order, there's no shortcut вҖ” the search has to check nodes one at a time until it finds a match, so it gets slower the more nodes the tree has.

## Pseudocode

```text
1. Build the tree first: turn the flat list of numbers into a tree of connected nodes, where each number's two "children" come from specific positions further along in the list.
2. Start at the root node. Add its value to your trail.
3. If this node's value is the target, stop вҖ” you found it, and the trail so far is your answer.
4. Otherwise, try going left: repeat this whole process on the left child.
5. If going left found the target, stop and report the trail.
6. Otherwise, try going right: repeat this whole process on the right child.
7. If going right found the target, stop and report the trail.
8. If neither side found it, remove this node's value from the trail (you're backing out of a dead end) and report "not found" back up to whoever called you.
9. If you run out of tree (an empty spot), that spot has nothing to check вҖ” immediately report "not found."
```

### Recursive Version

```text
1. Define a helper step, search(node, target, trail), that looks at one node at a time.
2. If node is empty, report "not found" back to whoever called you вҖ” there's nothing here to check.
3. Add this node's value to the trail.
4. If this node's value is the target, report "found" and stop.
5. Call search on this node's left child. If that call reports "found," pass "found" straight back up without checking the right side.
6. Otherwise, call search on this node's right child. If that call reports "found," pass "found" straight back up.
7. If neither child found it, remove this node's value from the trail вҖ” you're backing out of a dead end вҖ” and report "not found" back up.
8. Kick the whole thing off by calling search(root, target, an empty trail).
```

## Complexity

- **Best case:** If the target happens to be the very first node checked вҖ” the root вҖ” it's basically instant (`O(1)`).
- **Average case:** Since the tree's values aren't sorted, there's no shortcut to skip a branch, so on a typical tree you end up checking a good chunk of the nodes before finding a match (`O(n)`).
- **Worst case:** If the target is missing entirely, or it's the very last node the search happens to reach, every single node gets visited once (`O(n)`) вҖ” there's no way to rule out a branch just by comparing numbers.
- **Space:** The search keeps a trail from the root down to wherever it currently is, plus one waiting call for every level it has descended into. For a balanced tree, that trail only ever gets about as deep as `log2(n)` levels (`O(log n)`); but for a heavily lopsided tree that's basically one long chain, the trail can grow as long as every node in the tree (`O(n)`) вҖ” this is where how balanced the tree is really matters.

## Applications

- **File system search utilities**, like the Unix `find` command or Java's `Files.walkFileTree`, walk a directory tree branch by branch to locate a matching file, since folders aren't stored in any searchable order.
- **DOM tree traversal in web browsers**, such as `Node.contains()` or a hand-rolled `querySelector` fallback, walks down the page's element tree to find a specific node.
- **XML and JSON tree parsers**, like Python's `xml.etree.ElementTree.iter()`, search a parsed document tree by descending into every branch looking for a matching tag or key.
- **Nested menu or category "find item" features**, such as a file manager's search or a CMS's category browser, walk down a tree of folders or categories one branch at a time because there's no sort order to exploit.
- **Game dialogue and decision trees**, where an NPC or game engine walks down branches of a conversation or decision tree looking for the node that matches the player's current state.
