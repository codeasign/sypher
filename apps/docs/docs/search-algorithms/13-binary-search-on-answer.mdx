## Summary

Binary Search on Answer solves a different kind of problem: instead of searching for a value hiding in a list, you search for the best number that satisfies some rule — here, the slowest eating speed that still lets Koko finish all her banana piles in time. The trick is to guess a speed, check if it works, and use that yes/no answer to guess higher or lower next time, cutting the range of possible speeds in half each round, just like guessing a number between 1 and 100. It's much faster than trying every possible speed one at a time.

## Pseudocode

```text
1. canFinish(piles, hours, speed): check if eating at this speed finishes all piles within the hour limit.
   1a. Add up the hours needed for each pile at this speed (round up any partial hour).
   1b. If the total ever goes over the hour limit, this speed is too slow — report "no."
   1c. Otherwise, report "yes."
2. Set the slowest possible speed to 1, and the fastest possible speed to the size of the biggest pile.
3. While the slow guess is still less than the fast guess:
4.     Guess the middle speed between them.
5.     Check with canFinish whether that middle speed works.
6.     If it works, it might be possible to go even slower — lower the fast guess to this middle speed.
7.     If it doesn't work, you need to go faster — raise the slow guess just above this middle speed.
8. When the two guesses meet, that number is the slowest speed that still works.
```

## How It Works

- This only works because the yes/no check is "one-directional": if a speed is fast enough to finish in time, every faster speed also finishes in time. If a speed is too slow, every slower speed is too slow too. That means as you slide from slow to fast along the number line, the answer flips from "no" to "yes" exactly once — which is exactly what lets you throw away half the guesses each round, the same way sorted order lets plain binary search throw away half a list.
- In the diagram, round 1 guesses the middle speed, `6`. Checking it takes `6` hours total, which fits inside the `8`-hour limit, so `6` works. Since we want the slowest speed that still works, there's no need to check anything faster than `6` — throw away the top half.
- Round 2 guesses `3`, the middle of what's left. That needs `10` hours — too slow. Since `3` didn't work, nothing slower than `3` will either, so throw away the bottom half.
- Round 3 guesses `5`. That needs `7` hours, which fits — so `5` works, and we can rule out anything faster than `5`.
- Round 4 guesses `4`. That needs exactly `8` hours, right at the limit — so `4` still works. The two guesses meet at `4`, meaning `4` is the slowest speed that finishes in time.
- If the very first guess happens to be the answer, you're done almost immediately. Usually, though, you keep cutting the range of possible speeds in half until only one candidate is left. Each guess also has to add up hours across every pile, so the total work is roughly "how many piles" multiplied by "how many times you can cut the speed range in half" (in Big-O terms, that's O(n log m) time, where n is the number of piles and m is the biggest pile — much faster than trying every possible speed one at a time).
