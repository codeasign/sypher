## Summary

Ternary Search is like Binary Search's cousin: instead of splitting a sorted list into two halves, it splits it into three parts using two probe points, then throws away whichever parts can't contain the target. It sounds like it should be faster since it eliminates more at once, but each round costs more work (two comparisons instead of one), and the extra cost usually cancels out the benefit — so for plain list lookups it ends up a bit slower than Binary Search in practice.

## Pseudocode

```text
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. Pick two probe points that split the range into three roughly equal parts: mid1 (about a third of the way in) and mid2 (about two-thirds of the way in).
4. If the item at mid1 is the target, stop and report its position.
5. If the item at mid2 is the target, stop and report its position.
6. If the target is smaller than the item at mid1, it must be in the first part — keep only that part.
7. If the target is bigger than the item at mid2, it must be in the last part — keep only that part.
8. Otherwise the target must be in the middle part — keep only that part.
9. If low ever passes high without a match, report "not found."
```

### Recursive Version

```text
1. If low has passed high, report "not found" and stop.
2. Otherwise, pick two probe points that split the range into three roughly equal parts: mid1 and mid2.
3. If the item at mid1 is the target, stop and report its position.
4. If the item at mid2 is the target, stop and report its position.
5. If the target is smaller than the item at mid1, repeat this whole process on just the first part (from low to mid1 minus one).
6. If the target is bigger than the item at mid2, repeat this whole process on just the last part (from mid2 plus one to high).
7. Otherwise, repeat this whole process on just the middle part (from mid1 plus one to mid2 minus one).
```

## Worked Example

Let's trace Ternary Search over the array `[2, 4, 6, 8, 10, 12, 14, 16, 18]`, looking for the target `10`. Just like Binary Search, this only works on a sorted list — the questions "is the target smaller than this probe?" and "is the target bigger than that probe?" only make sense if the numbers are already in order.

## Complexity

- **Best case:** If the target sits at one of the very first two probes, it's found almost instantly (`O(1)`).
- **Average case:** Each round throws away two-thirds of whatever is left, so it takes roughly log-base-3 of the list size rounds, with two comparisons per round (`O(log n)`).
- **Worst case:** The same shrink-by-two-thirds pattern still applies even when the target isn't in the list at all — you just keep narrowing until nothing is left to check (`O(log n)`).
- **Space:** The iterative version only ever needs a couple of pointers, no matter how big the list is (`O(1)`); the recursive version needs a little more room because each call waits on the call stack until a base case is hit (`O(log n)`).

## Applications

- **Finding the peak of a unimodal function** — a curve that rises then falls, or falls then rises, with exactly one high point or low point — is a classic use case in numerical optimization when there's no derivative to lean on.
- **Tuning a single hyperparameter** against a convex loss curve in machine learning can use Ternary Search to zero in on the best value instead of testing every candidate one by one.
- **Golden-section search**, a well-known one-dimensional optimization technique in operations research, is built on the same "split into three parts, discard the two that can't hold the answer" idea as Ternary Search.
- **Competitive programming problems** that ask for the point where a function stops increasing and starts decreasing routinely reach for Ternary Search as the standard tool.
- **Plain sorted-array lookups** can technically use Ternary Search as a stand-in for Binary Search, though it's rarely picked for that job in real code since it does more comparisons per round for no real gain.
