## Summary

Breadth-First Search explores a graph or tree one whole "ring" at a time, visiting everything close by before moving further out. Imagine dropping a pebble in a pond and watching the ripples spread outward evenly in all directions — that's BFS: it visits all of a starting point's direct neighbors first, then all of their neighbors, and so on. Because it spreads out evenly, it's the algorithm to use when you want the shortest route measured in number of steps. It gets slower the more places and connections there are, but it never wastes time backtracking.

## Pseudocode

```text
1. If the starting point isn't in the graph, report an empty result.
2. Keep a checklist of visited places, and a waiting line (a queue) starting with just the starting point. Mark the starting point visited right away.
3. While the waiting line isn't empty, take the place that's been waiting longest off the front of the line.
4. Add it to the result list.
5. Look at each of its neighbors. For any neighbor not already checked off, mark it visited and add it to the back of the waiting line.
6. Repeat until the waiting line is empty, then report the result list.
```

## How It Works

- In the diagram, we start at `A` (that's "ring 0"). We mark it visited and add its neighbors `B` and `C` to the waiting line — that's ring 1.
- We take `B` off the front of the line and visit it. `B`'s neighbor `D` gets added to the back of the line.
- We take `C` off the line and visit it next. `C` also points to `D`, but `D` is already checked off, so nothing new gets added.
- Finally we take `D` off the line and visit it — that's ring 2. The line is now empty, so we're done.
- Visit order: `A, B, C, D` — notice both ring-1 nodes (`B` and `C`) get visited before the ring-2 node (`D`), which is the whole point of exploring in rings.
- Because BFS always finishes an entire ring before moving to the next one, the first time it reaches any place, it's guaranteed to be by the shortest possible route (counted in number of steps) — that's why BFS is the go-to choice for "what's the fewest number of hops to get from here to there."
- How fast is it? Just like DFS, every place and every connection has to be looked at once, so there's no lucky shortcut — it always takes about the same amount of work for a given graph, growing with how big the graph is (this is called `O(V + E)` time, where `V` is the number of places and `E` is the number of connections). The tradeoff is memory: BFS can end up holding an entire wide ring of places in its waiting line at once, which can use more memory than DFS on a graph that's wide rather than deep.
