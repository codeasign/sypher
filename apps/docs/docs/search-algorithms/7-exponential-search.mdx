## Summary

Exponential Search is a fast way to find something in a sorted list by first guessing a rough neighborhood, then zooming in. It grows a "reach" by doubling it — 1, 2, 4, 8, 16, and so on — until it jumps past where the target could be, then it runs a Binary Search inside just that small stretch to pin down the exact spot. Think of flipping through a big sorted phone book by jumping further and further ahead each time, then narrowing down between your last two jumps once you've overshot the name. It only works on sorted lists, but it's especially quick when the item you want is near the front, and it's also handy when you don't even know how long the list is.

## Pseudocode

```text
1. If the list is empty, report "not found."
2. Check the first item. If it matches the target, report position 0.
3. Set a "reach" counter to 1.
4. While the reach hasn't passed the end of the list, and the item at the reach is still smaller than the target, double the reach.
5. Now the target (if it's in the list) must be somewhere between half the current reach and the current reach (or the end of the list, whichever comes first).
6. Run a Binary Search inside just that narrow window: check the middle item, and keep narrowing to the left half or right half depending on whether the middle is too big or too small.
7. If that search finds a match, report its position. Otherwise, report "not found."
```

## Worked Example

This walks one fixed example through both phases of the algorithm: a sorted list of 16 odd numbers, searching for `25`. Watch the reach double outward until it overshoots the target, then watch Binary Search take over inside the small window it landed on. This only works because the list is already sorted — the "still too small, keep doubling" trick only makes sense when values only ever go up as you move right, the same way you can only jump ahead in a phone book because the names are alphabetical.

## Complexity

- **Best:** If the target is the very first item in the list, it's found before any doubling even starts — almost instant (`O(1)`).
- **Average:** The doubling phase only takes a handful of steps because it grows so fast, and the Binary Search step inside the window it lands on is just as quick as an ordinary Binary Search over that smaller stretch (`O(log i)`, where `i` is roughly how far into the list the target sits).
- **Worst:** If the target is near the end of the list, or missing entirely, the reach has to double all the way out to the end before Binary Search even begins — but that's still far fewer checks than looking at every item one by one (`O(log n)`).
- **Space:** It only ever needs to remember a couple of numbers at a time — the current reach and the low/high boundaries — no matter how big the list is (`O(1)`).

## Applications

- **TimSort's "galloping mode"** — the hybrid sort used by Python's built-in `sorted()`/`list.sort()` and by Java's `Arrays.sort()` for objects uses an exponential-search-style probe to quickly find how many elements from one sorted run belong before merging continues, instead of comparing one element at a time.
- **Searching a list of unknown or unbounded size** — the classic "search in a sorted array with no known length" problem, useful for APIs that stream back sorted, paginated results without ever exposing a total count up front.
- **Recent-first sorted logs and timestamps** — tools that expect the item being searched for (like "the first entry after this time") to usually sit near the front of a sorted, timestamp-ordered list benefit from the doubling phase's speed near the start.
- **Large sorted files or key blocks on disk** — probing with a doubling offset before narrowing with Binary Search can mean fewer expensive reads than starting a Binary Search across the entire range immediately.
