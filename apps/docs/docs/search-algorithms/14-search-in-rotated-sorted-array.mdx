## Summary

A rotated sorted array is a list that was originally in order from smallest to biggest, but then got "cut" at some point and the front piece was moved to the back — so it looks like two sorted chunks stuck together (like `[4,5,6,7,0,1,2]`, which was `[0,1,2,4,5,6,7]` before the cut). This algorithm finds a target number in that kind of list without checking every item. The trick: at every step, at least one of the two halves you're looking at is still perfectly in order, so you can figure out which half that is and use ordinary binary search reasoning on it. It's nearly as fast as searching a normal sorted list — it only slows down a little the more items you add, instead of slowing down in proportion to every single item.

## Pseudocode

```text
1. Set low to the first index and high to the last index.
2. While low is at or before high:
3.     Look at the middle item between low and high.
4.     If it matches the target, stop — you found it. Report its position.
5.     Figure out which half, left or right, is still in normal sorted order.
6.     If the left half is in order:
7.         If the target could fit inside that left half's range of values, search the left half.
8.         Otherwise, search the right half.
9.     If the right half is in order instead:
10.        If the target could fit inside that right half's range of values, search the right half.
11.        Otherwise, search the left half.
12. If low passes high without a match, report "not found."
```

### Recursive Version

```text
1. Define a helper step that takes a low and a high boundary.
2. If low is past high, report "not found" — there's nothing left to check.
3. Look at the middle item between low and high.
4. If it matches the target, report its position and stop.
5. Figure out which half, left or right, is still in normal sorted order.
6. If the left half is in order:
7.     If the target could fit inside that left half's range of values, repeat this whole step on the left half.
8.     Otherwise, repeat this whole step on the right half.
9. If the right half is in order instead:
10.    If the target could fit inside that right half's range of values, repeat this whole step on the right half.
11.    Otherwise, repeat this whole step on the left half.
```

## Worked Example

This only works if the array is sorted apart from a single rotation — like a deck of cards that was cut once and the bottom half placed on top, rather than fully shuffled. Take the array `[15, 18, 2, 4, 7, 10, 13]`, which is the sorted list `[2, 4, 7, 10, 13, 15, 18]` rotated so its last two items moved to the front. Search it for the target `13`.

## Complexity

- **Best case:** if the very first middle item you check is the target, you're done immediately — instant (`O(1)`).
- **Average case:** like ordinary binary search, each step throws away about half of whatever's left to check, so the number of steps grows very slowly as the list grows (`O(log n)`).
- **Worst case:** even when the target is missing entirely, you still only ever cut the remaining range in half each time, so it still finishes in about the same small number of steps (`O(log n)`).
- **Space:** the iterative version only keeps a couple of pointers, no matter how big the list is (`O(1)`). The recursive version keeps one extra "layer" on the call stack per step, so it uses a small, slowly growing amount of extra memory (`O(log n)`).

## Applications

- **Ring buffers and circular queues** in embedded systems and audio/network streaming, where the write position wraps back to the start once it reaches the end, leaving a "rotated" order in memory.
- **Consistent-hashing rings** in distributed databases like Apache Cassandra and Amazon DynamoDB, where keys are looked up on a circular identifier space that has effectively been rotated relative to where you start scanning.
- **Log rotation buffers** in systems like syslog and journald, where the oldest entries get overwritten in place and the "start" of the readable log is wherever the last write landed, not at index 0.
- **Round-robin task scheduling** in operating systems, where the next task to run wraps around a fixed-size list of processes once it reaches the end.
- **Version-control and backup rotation schemes** that reuse a fixed number of slots in a cycle, where finding a specific backup by date means searching a rotated sequence of slot numbers.
