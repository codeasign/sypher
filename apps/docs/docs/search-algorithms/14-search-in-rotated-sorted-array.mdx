## Summary

A rotated sorted array is a list that was originally in order from smallest to biggest, but then got "cut" at some point and the front piece was moved to the back — so it looks like two sorted chunks stuck together (like `[4,5,6,7,0,1,2]`, which was `[0,1,2,4,5,6,7]` before the cut). This algorithm finds a target number in that kind of list without checking every item. The trick: at every step, at least one of the two halves you're looking at is still perfectly in order, so you can figure out which half that is and use ordinary binary search reasoning on it. It's nearly as fast as searching a normal sorted list.

## Pseudocode

```text
1. Set low to the first index and high to the last index.
2. While low is at or before high:
3.     Look at the middle item between low and high.
4.     If it matches the target, stop — you found it. Report its position.
5.     Figure out which half, left or right, is still in normal sorted order.
6.     If the left half is in order:
7.         If the target could fit inside that left half's range of values, search the left half.
8.         Otherwise, search the right half.
9.     If the right half is in order instead:
10.        If the target could fit inside that right half's range of values, search the right half.
11.        Otherwise, search the left half.
12. If low passes high without a match, report "not found."
```

## How It Works

- This works even though the list isn't fully sorted, because a rotated sorted array is really just two sorted pieces glued together at one "rotation point" — the spot where the numbers suddenly drop instead of climbing (between `7` and `0` in the diagram).
- At every step, you're looking at a middle item that splits the remaining range into two halves. One of those halves is guaranteed to still be in normal ascending order — the rotation point can only be hiding in the other one.
- In step 1, the left half (`4,5,6,7`) is in normal order. The target `0` doesn't fall between `4` and `7`, so it can't be hiding in that half — search the other half instead, the one that contains the rotation point.
- In step 2, the new left half (`0,1`) is in normal order, and this time the target `0` does fall in that range — so search this half.
- In step 3, the range has shrunk to a single item, `0`, which matches the target. Done.
- If the target happens to be exactly in the middle early on, you find it almost right away. Otherwise, you keep splitting the range in half, checking one item each time, the same number of steps as plain binary search takes on a fully sorted list of the same size (in Big-O terms, this is "O(log n)" time and "O(1)" extra memory — this approach assumes all the numbers in the list are different from each other; if there are lots of repeated values it can occasionally slow down).
