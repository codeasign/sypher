## Summary

Jump Search finds something in a sorted list by leaping forward in fixed-size chunks (blocks) instead of checking every item, and then walking normally through the one block where the target must be hiding. Imagine flipping through a phone book by jumping ahead 20 pages at a time until you overshoot the name you want, then flipping page by page from there. It's slower than Binary Search, but it only ever moves forward — never jumps backward — which matters for some kinds of data. It's faster than checking every item one by one, but not as fast as Binary Search.

## Pseudocode

```text
1. If the list is empty, report "not found" right away.
2. Pick a block size — roughly the square root of the list's length.
3. Starting from the beginning, leap forward by one block size at a time, each time checking the last item in the block you just landed on.
4. Keep leaping forward as long as that last item is still smaller than the target.
5. Once a block's last item is no longer smaller than the target (or you reach the end of the list), stop leaping — the target must be in this block, if it's anywhere.
6. Walk through that one block item by item, checking each one against the target.
7. If you find a match, report its position. If you reach the end of the block with no match, report "not found."
```

## Worked Example

This only works because the list is already sorted — the "still smaller, keep leaping" trick depends on values only ever going up as you move right, the same way you can only skip ahead confidently in a dictionary because the words are in order.

## Complexity

- **Best case:** If the very first item you check while leaping (or the first item in the final block) already happens to be the target, you find it almost instantly (`O(1)`).
- **Average case:** On a typical search, you leap through roughly √n blocks and then walk through part of one block — together that comes out to about `O(√n)`.
- **Worst case:** Even when the target is the last item of the last block you scan, you never leap through more than √n blocks and never walk more than √n items inside the final block, so it's still `O(√n)`.
- **Space:** Jump Search doesn't build any extra structures — it only tracks a couple of numbers (the current jump position and the previous one) as it goes, so it uses constant extra memory (`O(1)`).

The block size matters a lot here: it's chosen to be roughly `sqrt(n)` because that's the size that keeps the number of jumps and the number of steps inside the final block balanced. Pick a smaller block and you leap more often; pick a bigger block and you walk more once you land — √n is the sweet spot that minimizes the total work.

## Applications

- **Sequential-access storage like tape drives**, where jumping backward is expensive or impossible — Jump Search only ever moves forward, which fits hardware that reads much faster in one direction than the other.
- **Searching large sorted files read in fixed-size blocks from disk**, where each block read has a fixed cost — jumping block by block instead of checking every record cuts down the number of expensive reads.
- **Block-skipping strategies in database and log-scanning engines**, where sorted data is grouped into pages or segments and a cheap "peek at the last value in this segment" check decides whether to skip the whole segment.
- **Teaching the tradeoff between Linear and Binary Search**, since Jump Search sits in between them and is a common textbook and interview stepping stone for understanding why block size and access cost matter.
