## Summary

Depth-First Search is a way to explore a graph or tree by picking one path and following it as far as it goes before backtracking to try another. Think of exploring a maze by always taking the next unexplored turn and only backing up once you hit a dead end — that's DFS. It works on connected networks of any shape, and it keeps a checklist of places already visited so it never loops forever, and it gets slower the more places and connections there are to explore.

## Pseudocode

```text
1. If the starting point isn't in the graph, report an empty result.
2. Keep a checklist of visited places, and a "to-do" pile (a stack) starting with just the starting point.
3. While the to-do pile isn't empty, take the most recently added place off the pile.
4. If it's already on the checklist, skip it and go back to step 3.
5. Otherwise, mark it visited and add it to the result list.
6. Add each of its neighbors that aren't already checked off onto the to-do pile.
7. Repeat until the to-do pile is empty, then report the result list.
```

### Recursive Version

DFS has a natural recursive shape: "visit a place, then visit each of its neighbors the same way."

```text
1. If the starting point isn't in the graph, report an empty result.
2. Keep a checklist of visited places and a result list, both empty at first.
3. To "visit" a place: if it's already on the checklist, stop and do nothing.
4. Otherwise, mark it visited and add it to the result list.
5. For each of that place's neighbors, in order, "visit" the neighbor the same way.
6. Kick everything off by "visiting" the starting point once.
7. Once that first visit fully finishes, report the result list.
```

## Worked Example

Here's a small tree we'll trace all the way through:

```text
        A
       / \
      B   C
      |   |
      D   E
      |
      F
```

A has two children, B and C. B leads down to D, which leads down to F. C leads down to E, a dead end. We'll follow the iterative (stack-based) version above, pushing a node's neighbors in reverse order so the leftmost neighbor ends up on top of the stack and gets explored first.

## Complexity

- **Best case:** Even when things go smoothly, DFS still has to look at every reachable place and every connection between them at least once to be sure it's fully explored — there's no shortcut case (`O(V + E)`).
- **Average case:** On a typical graph, DFS visits every vertex once and follows every edge once, so its speed scales directly with the graph's total size (`O(V + E)`).
- **Worst case:** Even on a densely connected graph where almost every place connects to many others, DFS still only touches each connection once, so it stays at the same pace (`O(V + E)`).
- **Space:** It needs room to remember which places it already visited and to hold the stack of places still waiting to be explored, which in the worst case (a long chain) can hold nearly every vertex at once (`O(V)`).

## Applications

- **Maze and puzzle solving** — following one corridor all the way to a dead end before backtracking to try the next one is exactly how DFS explores a graph.
- **Topological sorting for build order** — tools like `make`, npm, and other package managers use a DFS-based traversal to figure out which dependencies must be installed or built before others.
- **Cycle detection** — DFS can spot circular dependencies (like two npm packages that depend on each other) or deadlocks in resource-allocation graphs by noticing when it revisits a node that's still on its current path.
- **Finding connected components** — the "flood fill" bucket tool in image editors like MS Paint and Photoshop uses a DFS-style spread to fill every connected pixel of the same color.
- **Generating solutions with backtracking** — solvers for puzzles like Sudoku or N-Queens use DFS to try one choice fully before backing up and trying another.
