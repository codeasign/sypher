## Summary

This algorithm finds the very first and very last position of a value in a sorted list, even when that value shows up many times in a row. It does this by running binary search twice: once hunting for the leftmost matching spot, and once hunting for the rightmost. Picture a sorted shelf of numbered folders where several folders in a row all say "8" — this finds exactly where that run of "8"s starts and where it ends. It stays fast even with lots of duplicates, because both searches jump straight to the edges instead of checking every matching folder one by one.

## Pseudocode

```text
Finding the FIRST occurrence:
1. Look at the whole list as your search window.
2. Check the middle item of the window.
3. If it's smaller than the target, the target must be further right — shrink the window to the right half.
4. Otherwise (it's equal to or bigger than the target): if it equals the target, remember this spot as your best guess so far. Then shrink the window to the left half, in case there's an even earlier match.
5. Repeat steps 2-4 until the window is empty.
6. Your last remembered guess is the first occurrence (or "not found" if you never got a match).

Finding the LAST occurrence:
1. Look at the whole list as your search window again.
2. Check the middle item of the window.
3. If it's bigger than the target, the target must be further left — shrink the window to the left half.
4. Otherwise (it's equal to or smaller than the target): if it equals the target, remember this spot as your best guess so far. Then shrink the window to the right half, in case there's an even later match.
5. Repeat steps 2-4 until the window is empty.
6. Your last remembered guess is the last occurrence.

Putting it together:
1. Run the "find first" search.
2. If nothing was found, report "not found" for both and stop.
3. Otherwise, also run the "find last" search.
4. Report both positions.
```

### Recursive Version

Both the "find first" and "find last" searches can be written as a function that calls itself on a smaller window instead of looping — the same idea as plain binary search, just with an extra "keep going anyway" twist when it finds a match.

```text
FIND-FIRST(list, target, low, high):
1. If low is past high, the window is empty — stop and report "no match in this window."
2. Check the middle item of the window.
3. If it's smaller than the target, call FIND-FIRST again on just the right half.
4. If it's equal to the target: call FIND-FIRST again on just the left half (to look for an earlier match); if that call finds nothing, this middle spot is the answer.
5. If it's bigger than the target, call FIND-FIRST again on just the left half.

FIND-LAST(list, target, low, high):
1. If low is past high, the window is empty — stop and report "no match in this window."
2. Check the middle item of the window.
3. If it's bigger than the target, call FIND-LAST again on just the left half.
4. If it's equal to the target: call FIND-LAST again on just the right half (to look for a later match); if that call finds nothing, this middle spot is the answer.
5. If it's smaller than the target, call FIND-LAST again on just the right half.
```

## Worked Example

Here's one fixed example we'll trace through every step: the sorted array `[2, 4, 4, 4, 4, 7, 9]` (indices 0-6), searching for target `4`. This value appears four times in a row, at indices 1, 2, 3, and 4. This only works because the array is already sorted — the same way looking up a word in a dictionary only works because the words are in order.

Now `low = 0` and `high = 0`, so `mid = 0`, which holds `2`. Since `2` is smaller than the target, `low` moves to `1`, which is now past `high`, so the window is empty and the search stops. The last recorded guess, index 1, is the true first occurrence.

/>

Because index 3 matched, `low` moves to `mid + 1 = 4`, shrinking the window to indices 4-6. The new `mid = 5` holds `7`, which is bigger than the target, so `high` moves to `mid - 1 = 4` and the guess stays at index 3 for now.

Now `low = 4` and `high = 4`, so `mid = 4`, which holds `4` — another match. This becomes the new best guess, and `low` moves to `5`, which is now past `high`, so the window is empty and the search stops. The last recorded guess, index 4, is the true last occurrence.

Putting both searches together: the first occurrence of `4` is at index 1, and the last occurrence is at index 4 — a run of four matching values found in just a handful of comparisons instead of scanning past every one of them.

## Complexity

- **Best case:** if the middle item checked happens to be an edge of the matching run right away, one comparison per search settles it (`O(1)`).
- **Average case:** each search keeps cutting the window in half, so it takes only a handful of checks even for a huge list (`O(log n)`).
- **Worst case:** same as average — the window still halves every step no matter how the values are arranged, since duplicates don't slow either search down (`O(log n)`).
- **Space:** the iterative version only needs a couple of pointers, so it uses barely any extra memory (`O(1)`); the recursive version needs a bit of memory for each nested call, growing with how many times the window can be halved (`O(log n)`).

## Applications

- Counting how many times a value occurs in a sorted list or log file, by subtracting the first index from the last.
- Range queries in databases and search indexes — finding all rows or records whose sorted key equals a given value.
- Finding the boundaries of a group in sorted, bucketed datasets, such as all events in a given minute inside a sorted timestamp array.
- Autocomplete and dictionary lookups, where you need every word sharing a sorted prefix, not just one match.
- Version-control and changelog tools that binary-search sorted commit or release lists to find the first and last entry affected by a change.
