## Implementation

Below is a complete Rust 2021 edition implementation of the Interpreter pattern using a Stock Market domain. The pattern defines a composable expression grammar for stock filtering. The `Expression` trait declares the `interpret()` method. Terminal expressions (`PriceGreaterThan`, `VolumeGreaterThan`, `TickerEquals`, `SectorEquals`) implement atomic comparisons against a `StockData` struct. Nonterminal expressions (`AndExpr`, `OrExpr`, `NotExpr`) compose expressions logically. The `StockScreener` evaluates an expression tree against a portfolio, collecting matching stocks. Trait objects (`Box<dyn Expression>`) provide runtime polymorphism.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   <<trait>>                          │
//   │   Expression (AbstractExpression)    │ ← OOP: Abstraction
//   ├──────────────────────────────────────┤    ISP: single method
//   │   fn interpret(&self, &StockData)    │
//   │       -> bool                        │
//   └────────────────┬─────────────────────┘
//                    ▲ impl for
//   ┌────────────────┴─────────────────────┐
//   │                │                     │
//   ┌────────────────┐  ┌──────────────────┐
//   │ TerminalExprs  │  │ NonterminalExprs │ ← Polymorphism via
//   │ (PriceCheck,   │  │ (AndExpr,        │   Box<dyn Expression>
//   │  VolumeCheck,  │  │  OrExpr,         │   trait objects
//   │  TickerMatch,  │  │  NotExpr)        │
//   │  SectorMatch)  │  │                  │
//   └────────────────┘  └──────────────────┘
//
//   ┌──────────────────────────────────────┐
//   │ StockScreener (Client)               │ ← SRP: only screens
//   │ + screen(&[StockData], &dyn Express) │    DIP: depends on
//   └──────────────────────────────────────┘    Expression trait
```

#### OOP Concepts Applied

1. **Abstraction** — The `Expression` trait defines the contract. All client code works with `&dyn Expression` trait objects.
2. **Encapsulation** — Each expression struct stores its private fields and exposes only `interpret()` through the trait.
3. **Inheritance** — Rust uses trait implementation rather than class inheritance. Each struct implements the `Expression` trait.
4. **Polymorphism** — `screen()` takes `&dyn Expression` and calls `interpret()` through dynamic dispatch via the vtable.

#### SOLID Principles Applied

5. **SRP** — Each expression struct evaluates exactly one grammar rule. `StockScreener` only iterates and collects.
6. **OCP** — New expression types implement the `Expression` trait without modifying existing code.
7. **LSP** — Every struct implementing `Expression` is usable wherever `&dyn Expression` is expected.
8. **ISP** — The `Expression` trait is a single method — as small as possible.
9. **DIP** — `StockScreener` depends on the `Expression` trait abstraction, not on concrete structs.

### StockData (Context)

```rust
// ── Context ──
#[derive(Debug, Clone)]
struct StockData {
    ticker: String,
    price: f64,
    volume: u64,
    sector: String,
}
```

### Expression (AbstractExpression)

```rust
// ── Abstract Expression (Trait) ──
trait Expression {
    fn interpret(&self, stock: &StockData) -> bool;
}
```

### PriceGreaterThan (TerminalExpression)

```rust
struct PriceGreaterThan {
    threshold: f64,
}

impl PriceGreaterThan {
    fn new(threshold: f64) -> Self {
        Self { threshold }
    }
}

impl Expression for PriceGreaterThan {
    fn interpret(&self, stock: &StockData) -> bool {
        stock.price > self.threshold
    }
}
```

### VolumeGreaterThan (TerminalExpression)

struct VolumeGreaterThan {

### AndExpr (NonterminalExpression)

```rust
struct AndExpr {
    left: Box<dyn Expression>,
    right: Box<dyn Expression>,
}

impl AndExpr {
    fn new(left: Box<dyn Expression>, right: Box<dyn Expression>) -> Self {
        Self { left, right }
    }
}

impl Expression for AndExpr {
    fn interpret(&self, stock: &StockData) -> bool {
        self.left.interpret(stock) && self.right.interpret(stock)
    }
}
```

### OrExpr (NonterminalExpression)

```rust
struct OrExpr {
    left: Box<dyn Expression>,
    right: Box<dyn Expression>,
}

impl OrExpr {
    fn new(left: Box<dyn Expression>, right: Box<dyn Expression>) -> Self {
        Self { left, right }
    }
}

impl Expression for OrExpr {
    fn interpret(&self, stock: &StockData) -> bool {
        self.left.interpret(stock) || self.right.interpret(stock)
    }
}
```

### NotExpr (NonterminalExpression)

```rust
struct NotExpr {
    expr: Box<dyn Expression>,
}

impl NotExpr {
    fn new(expr: Box<dyn Expression>) -> Self {
        Self { expr }
    }
}

impl Expression for NotExpr {
    fn interpret(&self, stock: &StockData) -> bool {
        !self.expr.interpret(stock)
    }
}
```

### StockScreener (Client)

```rust
// ── Client ──
struct StockScreener;

impl StockScreener {
    fn screen(&self, stocks: &[StockData], expression: &dyn Expression) -> Vec<StockData> {
        stocks
            .iter()
            .filter(|s| expression.interpret(s))
            .cloned()
            .collect()
    }
}
```

### Language-Specific Notes

- **OOP concepts** — Rust does not have classes or classical inheritance. Traits provide abstraction, and `Box<dyn Expression>` enables runtime polymorphism through vtable-based dynamic dispatch. This is the idiomatic Rust approach to the Interpreter pattern's recursive expression tree.
- **SOLID in practice** — Rust's trait system naturally enforces ISP: a trait should have one method (trait segregation). LSP is guaranteed by the type system — any `&dyn Expression` reference works with any implementor. OCP is achieved by implementing the `Expression` trait on new structs without modifying existing code.
- **Idiomatic patterns** — `Box::new(...)` for heap allocation of trait objects. `#[derive(Debug, Clone)]` for automatic trait implementations on the context struct. `vec![]` macro for list creation. `.iter().filter().cloned().collect()` for functional-style collection filtering. `to_uppercase()` for Unicode-aware string normalization.
- **Ownership** — The `interpret()` method takes `&self` and `&StockData` (shared references), meaning multiple expressions can evaluate against the same stock data simultaneously without ownership conflicts. The `screen()` method returns `Vec<StockData>` (owned copies via `.cloned()`) so the caller has full ownership of results.
