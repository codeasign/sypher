## Intent

Encapsulate a request as an object, thereby letting you parameterize clients with different requests, queue or log requests, and support undoable operations. The Command pattern decouples the object that invokes an operation from the one that knows how to perform it.

In the stock market domain, this means treating every trade order — buy, sell, or cancel — as a first-class object that can be created, stored, executed, reversed, and replayed independently. The trader or algorithmic system (client) never directly manipulates a portfolio; instead it creates command objects and submits them to a broker (invoker), which executes them against the market (receiver).

## The Problem Without This Pattern

Imagine building a trading application where every investor class directly calls methods on a `Stock` object whenever a trade is needed.

```python
# Problem: Tight coupling — trading logic scattered everywhere
class Investor:
    def buy_stock(self, stock, quantity, price):
        stock.shares += quantity
        stock.cash -= quantity * price
        print(f"{self.name} bought {quantity} shares of {stock.ticker}")

    def sell_stock(self, stock, quantity, price):
        if stock.shares >= quantity:
            stock.shares -= quantity
            stock.cash += quantity * price
            print(f"{self.name} sold {quantity} shares of {stock.ticker}")
        else:
            print("Insufficient shares")
```

What breaks:

- **No undo capability.** Once `buy_stock` runs, there is no way to reverse it without writing separate reversal logic in every investor class.
- **No queuing or scheduling.** If the market is closed, orders cannot be deferred — the call site must handle timing itself.
- **No audit log.** Each investor class would need its own logging, creating duplication and inconsistency.
- **Hard to extend.** Adding a limit order, stop-loss, or bracket order requires modifying every investor class.
- **Violates Open/Closed Principle.** Every new order type forces changes across multiple callers.

The root cause is that requests are method calls — transient and tightly coupled. You cannot store them, pass them around, compose them, or reverse them cleanly.

## Structure

The Command pattern organizes five participants. The Client creates a ConcreteCommand, setting its Receiver. The Invoker stores and triggers the Command. The Command calls operations on the Receiver when executed.

### Participants

| Participant | Role | Stock Market Analogy |
|---|---|---|
| **Command** | Declares the interface for executing and undoing an operation | The `Order` interface every trade type must implement |
| **ConcreteCommand** | Implements `execute()` and `undo()` by calling methods on the Receiver | `BuyOrder`, `SellOrder`, `CancelOrder` |
| **Receiver** | Knows how to perform the actual business logic | `Stock` — manages share counts and valuations |
| **Invoker** | Stores commands and triggers their execution | `Broker` — accepts orders, executes them, maintains history |
| **Client** | Creates ConcreteCommand objects and sets their Receiver | The trader or automated trading system |

## Pseudocode — Stock Market Example

The following pseudocode demonstrates a trading system using the Command pattern. A broker accepts buy and sell orders for stocks, executes them, supports cancellation (undo), and maintains a complete order history.

```
// PSEUDOCODE — Command Pattern in Stock Market Trading
// Participants labeled with [ROLE]

// ── [RECEIVER] ──
class Stock:
  fields: ticker, price, shares
 
  method buy(quantity, price):
    shares = shares + quantity
    print("Bought N of TICKER")
 
  method sell(quantity, price):
    if shares >= quantity:
      shares = shares - quantity
      print("Sold N of TICKER")
    else:
      print("Insufficient shares")
 
  method getInfo():
    return "TICKER: $PRICE/share, SHARES outstanding"

// ── [COMMAND] ──
interface Order:
  method execute()
  method undo()

// ── [CONCRETE COMMAND] ──
class BuyOrder implements Order:
  fields: stock, quantity, price
 
  method execute():
    stock.buy(quantity, price)
 
  method undo():
    stock.sell(quantity, price)

// ── [CONCRETE COMMAND] ──
class SellOrder implements Order:
  fields: stock, quantity, price
 
  method execute():
    stock.sell(quantity, price)
 
  method undo():
    stock.buy(quantity, price)

// ── [CONCRETE COMMAND] ──
class CancelOrder implements Order:
  fields: targetOrder, wasExecuted
 
  method execute():
    wasExecuted = true
    targetOrder.undo()
 
  method undo():
    if wasExecuted:
      targetOrder.execute()

// ── [INVOKER] ──
class Broker:
  fields: orderHistory (list of orders)
 
  method placeOrder(order):
    order.execute()
    orderHistory.append(order)
    print("Order executed and recorded")
 
  method cancelLastOrder():
    if orderHistory not empty:
      lastOrder = orderHistory.pop()
      cancelCmd = CancelOrder(lastOrder)
      cancelCmd.execute()
      orderHistory.append(cancelCmd)
 
  method showHistory():
    for each order in orderHistory:
      print(order details)

// ── [CLIENT] ──
aapl = Stock("AAPL", 175.50, 1000)
msft = Stock("MSFT", 378.90, 1200)
broker = Broker()

broker.placeOrder(BuyOrder(aapl, 100, 175.50))
broker.placeOrder(SellOrder(msft, 50, 378.90))
broker.cancelLastOrder()
broker.showHistory()
```

## How It Works — Step by Step

1. **Client creates a Stock (Receiver).** The investor or trading algorithm instantiates a `Stock` object with a ticker symbol, current price, and initial share count. The Stock knows how to adjust its position when trades occur.

2. **Client creates a ConcreteCommand and binds it to the Receiver.** For example, `BuyOrder(aapl, 100, 175.50)` creates a command object that stores a reference to the `aapl` Stock, the quantity (100), and the price ($175.50). The command does not execute yet — it is a packaged request.

3. **Client passes the command to the Invoker.** The client calls `broker.placeOrder(buyOrder)`, handing the command object to the Broker. The Broker accepts any `Order` implementation without knowing its concrete type.

4. **Invoker calls `execute()` on the Command.** Inside `placeOrder`, the Broker calls `buyOrder.execute()`. This delegates to `BuyOrder.execute()`, which calls `aapl.buy(100, 175.50)` on the Receiver.

5. **Receiver performs the actual operation.** The Stock's `buy` method increments its share count by 100 and records the transaction. The business logic — validation, position tracking, logging — stays inside Stock.

6. **Invoker stores the executed Command in its history.** The Broker appends the executed order to `orderHistory`. This makes the operation reviewable and reversible later.

7. **Undo reverses the operation.** When `broker.cancelLastOrder()` is called, the Broker wraps the original order in a `CancelOrder` command and calls `execute()`. The `CancelOrder.execute()` calls `undo()` on the original order, which invokes the inverse operation on Stock — a sell reverses a buy, and vice versa.

8. **History review.** The Broker can replay the entire `orderHistory`, printing each order's details. This provides an auditable trail of all trading activity.

## When to Use

- You need to parameterize objects with operations — for example, a `Broker` that accepts any type of order (buy, sell, cancel) without knowing its implementation.
- You need to queue, schedule, or log requests. A trading system may receive orders while the market is closed and must execute them in sequence when the market opens.
- You need to support undo or rollback. A brokerage platform must allow traders to cancel pending orders or reverse executed trades within a settlement window.
- You need to support macros or composite operations. A portfolio rebalancing strategy might create a batch of buy and sell orders executed atomically.
- You need to audit or replay operations. Regulatory compliance requires a complete, immutable order history that can be replayed for inspection.
- You need to decouple the sender of a request from its receiver. An algorithmic trading engine should not depend on the concrete stock implementations it trades.

## When NOT to Use

- The operation has no reversible side effects, and you do not need queuing, logging, or auditing. A simple function call suffices.
- The system has only one or two fixed operations that never change. Introducing the Command pattern adds unnecessary abstraction.
- Performance is critical and the overhead of creating command objects for every operation is measurable in your hot path. High-frequency trading at nanosecond scale may bypass this pattern.
- The undo logic is impossible to implement correctly because operations have irreversible side effects (e.g., a trade that settles externally cannot be truly undone).
- You are building a throwaway prototype with no requirements for extensibility, undo, or auditing. Start simple and refactor when the need arises.

## Real-World Occurrences

- **Swing GUI toolkit (Java).** Swing's `Action` interface (and `AbstractAction`) is a canonical Command pattern implementation. Buttons, menu items, and keybindings all share the same `Action` objects, enabling undo, enabling/disabling, and shared behavior across UI components.

- **.NET System.Windows.Input.ICommand.** WPF, MAUI, and Blazor use `ICommand` to bind UI actions to view models. Commands support `CanExecute` for enabling/disabling controls and `Execute` for performing the action. The pattern is central to MVVM architecture.

- **Python's `cmd` module.** The standard library `cmd.Cmd` class uses the Command pattern for building line-oriented command interpreters. Each command is a `do_XXX` method, and the framework dispatches based on user input, supporting help, history, and extensibility.

- **Redux / NgRx (JavaScript).** While not a textbook Command pattern, Redux actions are command-like objects with a `type` and payload. The store (invoker) dispatches actions (commands) to reducers (receivers), and the action log enables time-travel debugging (undo/replay).

- **Undo/redo in graphics editors.** Adobe Photoshop, Figma, and GIMP use the Command pattern extensively. Every user action (brush stroke, filter, layer change) is a command object stored in an undo stack. Ctrl+Z pops the stack and calls `undo()` on each command.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Strategy** | Strategy configures an algorithm's behavior; Command encapsulates a request as an object | Use Strategy when you need to swap algorithms at runtime (e.g., different pricing models). Use Command when you need to queue, log, or undo operations |
| **Memento** | Memento captures and externalizes object state for later restoration | Use Memento when state snapshots are more natural than inverse operations (e.g., saving a document state before each edit). Combine Memento with Command for robust undo |
| **Observer** | Observer notifies dependents when a subject changes state | Use Observer when many objects need to react to state changes (e.g., price tickers updating dashboards). Use Command when the focus is on encapsulating and controlling a request |
| **Chain of Responsibility** | CoR passes a request along a chain until one handler processes it | Use CoR when the handler for a request is unknown until runtime (e.g., order routing to different exchanges). Use Command when a single known handler processes the request |
| **Template Method** | Template Method defines the skeleton of an algorithm, letting subclasses fill in steps | Use Template Method when you have a fixed algorithm structure with variant steps. Use Command when the entire operation is the variant unit |
| **Composite** | Composite treats individual objects and compositions uniformly | Combine with Command to create macro commands — a `CompositeCommand` that contains multiple sub-commands executed as one unit (e.g., a portfolio rebalance as a batch of orders) |
