TypeScript brings static typing to the Composite pattern, giving the `CarPart` hierarchy compile-time safety that plain JavaScript lacks. An abstract `CarPart` class declares the interface with typed return values. Leaf classes (`Wheel`, `BrakePad`, `Piston`, `SparkPlug`) extend it and implement the data methods directly. The `CompositePart` abstract class holds a typed `children: CarPart[]` array and delegates `getWeight()`, `getPrice()`, and `getPartCount()` to each child via `reduce`. Concrete composites (`Engine`, `Suspension`, `Car`) extend `CompositePart`, inheriting the aggregation logic while adding domain-specific properties.

## How the Implementation Works

TypeScript's access modifiers (`protected`, `private`, `public`) control visibility of the children list and individual part fields. The abstract `CarPart` class enforces that all subclasses implement `getWeight()`, `getPrice()`, and `getPartCount()`. The `CompositePart` class provides the shared aggregation logic that concrete composites inherit. TypeScript's type system ensures that only `CarPart` instances can be added as children, preventing runtime type errors.

### OOP Principles Applied

| Principle | How It Is Applied |
|---|---|
| **Encapsulation** | Fields like `weight`, `price`, and `children` are `protected` or `private`, accessed only through public getters |
| **Inheritance** | Every concrete class extends either `CarPart` (leaves) or `CompositePart` (assemblies), both inheriting the interface shape |
| **Polymorphism** | The `processPart(part: CarPart)` function works identically on leaf and composite arguments, dispatching to the correct implementation at runtime |
| **Open/Closed** | New leaf types (e.g., `Turbocharger`) or composite types (e.g., `ExhaustSystem`) extend existing classes without modifying any client code or `CompositePart` |
| **Single Responsibility** | Leaves store a single part; `CompositePart` manages tree aggregation; concrete assembly classes add domain-specific fields |
| **Liskov Substitution** | A `CompositePart` is a `CarPart`; a `Wheel` is a `CarPart`. Any function typed `(p: CarPart) => void` accepts both |

// Composite Pattern — Car Parts Hierarchy (TypeScript)
abstract class CarPart {
  protected name: string;

  constructor(name: string) {
    this.name = name;
  }

  getName(): string {
    return this.name;
  }

  abstract getWeight(): number;
  abstract getPrice(): number;
  abstract getPartCount(): number;

  addChild(part: CarPart): void {
    throw new Error(
      `addChild() is not supported on ${this.constructor.name}`
    );
  }

  removeChild(part: CarPart): void {
    throw new Error(
      `removeChild() is not supported on ${this.constructor.name}`
    );
  }

  getChild(index: number): CarPart {
    throw new Error(
      `getChild() is not supported on ${this.constructor.name}`
    );
  }
}

### Composite

```typescript
abstract class CompositePart extends CarPart {
  protected children: CarPart[];

  constructor(name: string) {
    super(name);
    this.children = [];
  }

  getWeight(): number {
    return this.children.reduce(
      (sum: number, child: CarPart) => sum + child.getWeight(),
      0
    );
  }

  getPrice(): number {
    return this.children.reduce(
      (sum: number, child: CarPart) => sum + child.getPrice(),
      0
    );
  }

  getPartCount(): number {
    return this.children.reduce(
      (count: number, child: CarPart) => count + child.getPartCount(),
      0
    );
  }

  addChild(part: CarPart): void {
    if (!(part instanceof CarPart)) {
      throw new Error('Child must be a CarPart instance');
    }
    this.children.push(part);
  }

  removeChild(part: CarPart): void {
    const index = this.children.indexOf(part);
    if (index !== -1) {
      this.children.splice(index, 1);
    }
  }

  getChild(index: number): CarPart {
    if (index < 0 || index >= this.children.length) {
      throw new Error(`Child index ${index} out of bounds`);
    }
    return this.children[index];
  }

  getChildren(): CarPart[] {
    return [...this.children];
  }
}
```

### Print Helper

```typescript
function printPartsTree(part: CarPart, indent: string = ''): void {
  const weight = part.getWeight().toFixed(2);
  const price = part.getPrice().toFixed(2);
  const count = part.getPartCount();
  const type = part instanceof CompositePart ? 'Assembly' : 'Part';
  console.log(
    `${indent}${type}: ${part.getName()} | Weight: ${weight} kg | Price: $${price} | Parts: ${count}`
  );
  if (part instanceof CompositePart) {
    for (const child of part.getChildren()) {
      printPartsTree(child, indent + '  ');
    }
  }
}
```

## Language-Specific Notes

- The abstract `CarPart` class uses TypeScript's `abstract` keyword on methods, providing compile-time enforcement that every concrete class implements the required methods.
- Access modifiers (`private`, `protected`) prevent external code from directly mutating the children array or individual part weights and prices, enforcing encapsulation.
- The `CompositePart` class is also declared `abstract` — it extends `CarPart` with aggregation logic but is not meant to be instantiated directly. Only concrete composites like `Engine`, `Suspension`, and `Car` can be constructed.
- The `instanceof` guard in `addChild()` provides a runtime safety net, though TypeScript's type system already catches type mismatches at compile time.
- The spread operator `[...this.children]` in `getChildren()` returns a shallow copy, protecting the internal array from external mutation.
- To compile and run: `npx tsc composite.ts --target ES2020 && node composite.js` — or use `ts-node` for direct execution.
