Rust implements the Composite pattern using traits and enums rather than classical inheritance. A `CarPart` trait defines the common interface. Two concrete types, `Leaf` and `Composite`, implement the trait differently. The `Leaf` struct holds a single part's data. The `Composite` struct holds a `Vec<Box<dyn CarPart>>` of children and delegates `get_weight()`, `get_price()`, and `get_part_count()` to each. Unlike object-oriented languages, Rust uses `Box<dyn CarPart>` for dynamic dispatch because the trait is object-safe. This approach gives Rust the same uniform treatment of parts and assemblies while leveraging the language's ownership model and zero-cost abstractions.

## How the Implementation Works

Rust's `CarPart` trait defines the interface with `&self` methods. The `Leaf` struct implements the trait by returning its stored fields. The `Composite` struct implements the trait by calling `iter().map(...).sum()` on its children vector. Since Rust does not have inheritance, each concrete composite type (`Engine`, `Suspension`, `Car`) wraps a `Composite` and delegates to it, or uses `Composite` directly with a string name. The driver uses `Box::new()` to allocate leaf and composite objects on the heap and then calls trait methods through dynamic dispatch.

### OOP Principles Applied

| Principle | How It Is Applied |
|---|---|
| **Encapsulation** | Fields are private (`name`, `weight`, `price`, `children`); access is through trait methods and public associated functions |
| **Inheritance (Composition-based)** | Rust has no class inheritance, but `Composite` structs contain a `Vec<Box<dyn CarPart>>` and reuse the aggregation logic |
| **Polymorphism** | `Box<dyn CarPart>` provides runtime polymorphism; the `print_parts_tree()` function accepts `&dyn CarPart` and dispatches dynamically |
| **Open/Closed** | Adding a new leaf type (e.g., `Turbocharger`) or a new composite type (e.g., `ExhaustSystem`) requires no changes to existing traversal code |
| **Single Responsibility** | `Leaf` owns a single part's data; `Composite` owns aggregation; builder functions own construction |
| **Liskov Substitution** | A `&dyn CarPart` reference can point to a `Leaf` or `Composite` interchangeably; calling `get_weight()` works identically |

```rust
// Composite Pattern — Car Parts Hierarchy (Rust)
trait CarPart {
```

### Composite

```rust
struct Composite {
    name: String,
    children: Vec<Box<dyn CarPart>>,
}

impl Composite {
    fn new(name: &str) -> Self {
        Composite {
            name: name.to_string(),
            children: Vec::new(),
        }
    }

    fn add_child(&mut self, child: Box<dyn CarPart>) {
        self.children.push(child);
    }

    fn children(&self) -> &[Box<dyn CarPart>] {
        &self.children
    }
}

impl CarPart for Composite {
    fn name(&self) -> &str {
        &self.name
    }

    fn get_weight(&self) -> f64 {
        self.children
            .iter()
            .map(|child| child.get_weight())
            .sum()
    }

    fn get_price(&self) -> f64 {
        self.children
            .iter()
            .map(|child| child.get_price())
            .sum()
    }

    fn get_part_count(&self) -> u32 {
        self.children
            .iter()
            .map(|child| child.get_part_count())
            .sum()
    }
}
```

### Print Helper

```rust
fn print_parts_tree(part: &dyn CarPart, indent: &str) {
    let weight = format!("{:.2}", part.get_weight());
    let price = format!("{:.2}", part.get_price());
    let count = part.get_part_count();

    // Check if this is a composite by attempting downcast
    let kind = if part.name() == "Engine"
        || part.name() == "Suspension"
        || part.name().contains("Honda")
    {
        // We know these are composites by construction
        "Assembly"
    } else {
        // Check if the part count is > 1 — composite heuristic
        if count > 1 && part.get_weight() > 10.0 {
            "Assembly"
        } else {
            "Part"
        }
    };

    println!(
        "{}{}: {} | Weight: {} kg | Price: ${} | Parts: {}",
        indent, kind, part.name(), weight, price, count
    );
}
```

```rust
    let mut car = Car::new("Honda", "Civic", 2025);

        "Assembly: {} | Weight: {:.2} kg | Price: ${:.2} | Parts: {}",

    println!("Total price: ${:.2}", car.get_price());
```

## Language-Specific Notes

- Rust has no class inheritance or `instanceof` operator. The Composite pattern is implemented using a trait (`CarPart`) with two implementors: `Leaf` (for individual parts) and `Composite` (for assemblies). Concrete composites like `Engine` and `Car` wrap a `Composite` internally and delegate to it.
- Dynamic dispatch uses `Box<dyn CarPart>` (heap-allocated trait objects) because the `CarPart` trait is object-safe — all methods take `&self` and return owned values, not `Self`.
- The `Leaf` struct is generic enough to serve all simple part types, while specialized structs like `Wheel` and `Piston` wrap a `Leaf` and delegate to it, adding domain-specific fields like `diameter_inches` and `bore_mm`.
- Child aggregation uses Rust's iterator chain: `self.children.iter().map(|c| c.get_weight()).sum()`, which is as concise as LINQ or streams in other languages.
- Rust's ownership model means that `add_child` takes ownership of the child via `Box<dyn CarPart>`, preventing aliasing issues at compile time. The tree is a single-owner hierarchy.
- Compile and run with `rustc composite.rs && ./composite` or `cargo new composite && cargo run` — no external dependencies required.
