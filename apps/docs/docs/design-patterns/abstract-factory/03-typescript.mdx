## Implementation

Below is a complete TypeScript implementation of the Abstract Factory pattern using a banking domain. The `BankFactory` interface defines three creation methods, and two concrete factories (`PersonalBankingFactory` and `BusinessBankingFactory`) produce families of related account objects. TypeScript's type system enforces the abstract factory contract at compile time: any attempt to omit a creation method or return the wrong product type produces a compilation error. The client code depends only on the `BankFactory` and `Account` interfaces.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌─────────────────────────────────────────────────────┐
//   │   <<interface>> BankFactory                         │
//   ├─────────────────────────────────────────────────────┤
//   │ + createSavingsAccount(holder, amount): Account     │  ← OOP: Abstraction
//   │ + createCheckingAccount(holder, amount): Account    │     ISP: focused
//   │ + createLoanAccount(holder, amount): Account        │     creation methods
//   └─────────────┬───────────────────────┬───────────────┘
//                 │ implements            │ implements
//   ┌─────────────┴─────────────┐  ┌──────┴────────────────┐
//   │ PersonalBankingFactory    │  │ BusinessBankingFactory│
//   │ ── creates Personal       │  │ ── creates Business   │  ← OOP: Polymorphism
//   │    product family         │  │    product family     │     LSP: substitutable
//   └─────────────┬─────────────┘  └──────┬────────────────┘
//                 │                       │
//                 ▼                       ▼
//   ┌────────────────────────┐  ┌────────────────────────┐
//   │ PersonalSavingsAccount │  │ BusinessSavingsAccount │
//   │ PersonalCheckingAccount│  │ BusinessCheckingAccount│  ← OOP: Inheritance
//   │ PersonalLoanAccount    │  │ BusinessLoanAccount    │     (implements Account)
//   └─────────────┬──────────┘  └──────┬─────────────────┘
//                 │                    │
//                 └───────┬────────────┘
//                         │ implements
//                  ┌──────┴──────────┐
//                  │ <<interface>>   │
//                  │ Account         │  ← OOP: Abstraction
//                  ├─────────────────┤     ISP: minimal interface
//                  │ getDetails()    │
//                  │ getBalance()    │
//                  │ getMonthlyFee() │
//                  └─────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `BankFactory` and `Account` are interfaces that define the contracts. The client code depends on these interfaces, never on the concrete classes.
2. **Encapsulation** — Each account class hides its internal state with private fields (`private readonly holder`). Access is provided through public methods only.
3. **Inheritance** — All six concrete account classes implement the `Account` interface. Both concrete factory classes implement the `BankFactory` interface.
4. **Polymorphism** — The driver stores all accounts in an `Account[]` array and calls `getDetails()` on each. TypeScript's structural type system ensures the correct method runs per concrete type.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` creates only personal banking products. `BusinessBankingFactory` creates only business banking products. Each account class represents exactly one product type.
2. **OCP** — Adding a `PremiumBankingFactory` class requires no changes to the `BankFactory` interface, existing factories, or existing account classes.
3. **LSP** — Every concrete account type can substitute `Account` without breaking the client. The driver treats all accounts uniformly.
4. **ISP** — The `Account` interface is minimal: three methods. The `BankFactory` interface is also minimal: three creation methods. No class implements methods it does not need.
5. **DIP** — Client code depends on the `BankFactory` and `Account` interfaces, not on concrete factory or product types.

### Account

```typescript
// Abstract Factory Pattern — TypeScript (strict) — Banking example
// ===================================================================
// Participants: BankFactory (AbstractFactory), PersonalBankingFactory
//   and BusinessBankingFactory (ConcreteFactory), Account (AbstractProduct),
//   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
//   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
//   (ConcreteProducts)
// OOP: abstraction, encapsulation, inheritance, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ===================================================================
// ── Abstract Product Interface ──
// OOP role: Abstraction + ISP (defines the contract for all account types)
// SOLID: ISP (minimal interface), DIP (clients depend on this)
interface Account {
  getDetails(): string;
  getBalance(): number;
  getMonthlyFee(): number;
}
```

### PersonalSavingsAccount

```typescript
// ── Concrete Product Family 1: Personal Banking Accounts ──
// OOP role: Inheritance (implements Account) + Polymorphism
// SOLID: LSP (substitutable for Account), OCP (extensible), SRP (one responsibility)
class PersonalSavingsAccount implements Account {
  private readonly holder: string;
  private readonly balance: number;
  private readonly interestRate: number = 3.5;
  private readonly monthlyFee: number = 5.0;

  constructor(holder: string, initialDeposit: number) {
    this.holder = holder;
    this.balance = initialDeposit;
  }

  getDetails(): string {
    return `Personal Savings[${this.holder}]: $${this.balance.toFixed(2)} @ ${this.interestRate}% APY`;
  }

  getBalance(): number {
    return this.balance;
  }

  getMonthlyFee(): number {
    return this.monthlyFee;
  }
}
```

### BankFactory

```typescript
// ── Abstract Factory Interface ──
// OOP role: Abstraction + Encapsulation (encapsulates product family creation)
// SOLID: SRP (creates accounts only), DIP (clients depend on this abstraction)
interface BankFactory {
  createSavingsAccount(holder: string, amount: number): Account;
  createCheckingAccount(holder: string, amount: number): Account;
  createLoanAccount(holder: string, amount: number): Account;
}
```


### displayAccounts

```typescript
// ── Client / Composition Root ──
// OOP role: Composition over inheritance
// SOLID: DIP (depends on BankFactory and Account abstractions), SRP (orchestrates only)
function displayAccounts(label: string, factory: BankFactory): void {
  console.log(`=== ${label} ===`);
  const accounts: Account[] = [
    factory.createSavingsAccount('Alice', 1000.0),
    factory.createCheckingAccount('Bob', 2500.0),
    factory.createLoanAccount('Carol', 50000.0),
  ];

  for (const account of accounts) {
    console.log(`  ${account.getDetails()}`);
    console.log(`    Balance: $${account.getBalance().toFixed(2)}, Monthly Fee: $${account.getMonthlyFee().toFixed(2)}`);
  }
  console.log('');
}
```

### Main / Driver

```typescript
// ── Main / Driver ──
function main(): void {
  const personalFactory: BankFactory = new PersonalBankingFactory();
  const businessFactory: BankFactory = new BusinessBankingFactory();

  displayAccounts('Personal Banking Accounts', personalFactory);
  displayAccounts('Business Banking Accounts', businessFactory);

  // Demonstrate family consistency — all accounts from the same factory
  // share the same fee structure and product family characteristics
  console.log('=== Cross-Family Comparison ===');
  const personalSavings: Account = personalFactory.createSavingsAccount('Alice', 1000.0);
  const businessSavings: Account = businessFactory.createSavingsAccount('Acme Corp', 50000.0);

  console.log(`Personal Savings monthly fee: $${personalSavings.getMonthlyFee().toFixed(2)}`);
  console.log(`Business Savings monthly fee: $${businessSavings.getMonthlyFee().toFixed(2)}`);

  const personalLoan: Account = personalFactory.createLoanAccount('Carol', 50000.0);
  const businessLoan: Account = businessFactory.createLoanAccount('Initech', 500000.0);

  console.log(`Personal Loan APR: ${personalLoan.getDetails()}`);
  console.log(`Business Loan APR: ${businessLoan.getDetails()}`);
}

main();
```

### Language-Specific Notes

- **Structural type system** — TypeScript uses structural (duck) typing, not nominal typing. A class satisfies an interface if it has the correct shape, even without an explicit `implements` clause. The `implements` keyword is used here for readability and compiler-enforced contract checking.
- **`readonly` modifier** — TypeScript's `readonly` modifier on class fields prevents reassignment after construction. This is enforced at compile time and does not affect the emitted JavaScript. Combined with `private`, it provides compile-time encapsulation.
- **No `private` at runtime** — TypeScript's `private` keyword is a compile-time-only construct. It is erased during compilation to JavaScript. In contrast, JavaScript's `#` private fields (used in the JS page) are enforced at the engine level. For TypeScript, the `private` keyword is the idiomatic choice for compile-time encapsulation.
- **Interface-based design** — TypeScript is the ideal language for the Abstract Factory pattern because interfaces are a first-class language feature. The `BankFactory` interface enforces the complete contract at compile time: any concrete factory missing a method produces a type error.
- **`strict` mode** — These examples use TypeScript's `strict` mode, which enables `noImplicitAny`, `strictNullChecks`, `strictFunctionTypes`, and other strict checks. All function parameters and return types are explicitly annotated for maximum type safety.
