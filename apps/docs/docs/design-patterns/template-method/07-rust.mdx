## Implementation

Below is a complete Rust implementation of the Template Method pattern using a stock market trading domain. Unlike class-based languages, Rust achieves the Template Method pattern through a combination of traits (for the interface contract) and a struct with a method that calls closures or trait methods. This implementation uses a `TradingStrategy` trait for the primitive operations and a standalone `execute_trade()` function as the template method — idiomatic Rust that separates the algorithm skeleton from the data that varies. Concrete strategies `MovingAverageCrossover` and `MeanReversion` implement the `TradingStrategy` trait and are exercised with realistic AAPL, GOOGL, and MSFT price data.

### How the Implementation Works

#### Rust-Specific Design

Rust does not have classical inheritance, so the Template Method pattern must be adapted to Rust's ownership model and trait system:

- **Trait for the interface** — `TradingStrategy` is a trait that defines the primitive operations (`analyze_market`, `calc_position_size`) and hooks (`pre_validate`, `apply_risk_controls`) with default implementations.
- **Generic free function for the template method** — Instead of a method on a base class, `execute_trade()` is a standalone generic function that accepts any type implementing `TradingStrategy`. This is more idiomatic in Rust than trying to simulate inheritance.
- **Concrete structs implement the trait** — `MovingAverageCrossover` and `MeanReversion` are plain structs that implement `TradingStrategy`, overriding the methods they need.

#### OOP Concepts Applied (adapted to Rust)

1. **Abstraction** — The `TradingStrategy` trait defines the contract. Concrete types hide their internal constants and implement the trait methods.
2. **Encapsulation** — Each strategy struct stores its configuration as private fields. Implementation details of `analyze_market()` are hidden from callers.
3. **No classical inheritance** — Rust uses composition and trait implementation instead. The shared `Order` type is a standalone struct used by both strategies.
4. **Polymorphism via generics** — The `execute_trade()` function is generic over `T: TradingStrategy`, providing compile-time polymorphism (static dispatch). Boxed trait objects could provide dynamic dispatch if needed.

#### SOLID Principles Applied

5. **SRP** — The template function owns the algorithm structure; each strategy struct owns its analysis. The `Order` struct owns order representation.
6. **OCP** — Adding a new strategy requires only a new struct implementing `TradingStrategy`. The template function and existing strategies remain unchanged.
7. **LSP** — Any type implementing `TradingStrategy` can be passed to `execute_trade()`. The function guarantees the same execution flow for all strategies.
8. **ISP** — The `TradingStrategy` trait has only two required methods (no default). Hooks are provided as default trait methods that implementors may override.
9. **DIP** — `execute_trade()` depends on the `TradingStrategy` trait (abstraction), not on concrete strategy types.

### Data Types

```rust
// Template Method Pattern — Rust — Stock Market example
// ======================================================
// Participants: TradingStrategy (trait),
// MovingAverageCrossover, MeanReversion (concrete structs)
// execute_trade() — generic template function
// OOP adapted: trait abstraction, impl polymorphism, composition
// SOLID: SRP, OCP, LSP, ISP, DIP
// ======================================================

use std::time::{SystemTime, UNIX_EPOCH};

// ── Data Types ──

#[derive(Debug, Clone)]
struct TradingDecision {
    signal: i32,  // 1 = BUY, -1 = SELL, 0 = HOLD
    confidence: f64,
}

#[derive(Debug, Clone)]
struct Order {
    ticker: String,
    shares: u32,
    signal: i32,
    order_id: String,
    timestamp: String,
    status: String,
}

impl Order {
    fn new(ticker: &str, shares: u32, signal: i32) -> Self {
        let now = SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap_or_default()
            .as_millis();
        Order {
            ticker: ticker.to_string(),
            shares,
            signal,
            order_id: format!("ORD-{}", now),
            timestamp: "2026-07-05T10:15:30".to_string(),
            status: "FILLED".to_string(),
        }
    }
}
```

### TradingStrategy (Trait)

```rust
// ── Trait: TradingStrategy ──
// OOP role: Abstraction + ISP
// SOLID: ISP, DIP
trait TradingStrategy {
    /// Required: analyze market data and return a trading decision.
    fn analyze_market(&self, price_data: &[f64]) -> TradingDecision;

    /// Required: determine how many shares to trade.
    fn calc_position_size(&self, confidence: f64, max_position: u32) -> u32;

    /// Hook: pre-validate the signal before trading. Default: check minimum confidence.
    fn pre_validate(&self, _signal: i32, confidence: f64) -> bool {
        confidence >= 0.15
    }

    /// Hook: apply risk controls. Default: allow all trades.
    fn apply_risk_controls(&self, _ticker: &str, _shares: u32) -> bool {
        true
    }

    /// Default min confidence for default pre_validate.
    fn min_confidence(&self) -> f64 {
        0.15
    }

    /// Default max position size.
    fn max_position(&self) -> u32 {
        1000
    }
}
```

### Template Method (Free Function)

```rust
// ── Template Method (free function) ──
// Instead of a method on a base class, this is a generic function
// that accepts any type implementing TradingStrategy.
// This is the idiomatic Rust equivalent of the template method.
fn execute_trade<T: TradingStrategy>(
    strategy: &T,
    ticker: &str,
    price_data: &[f64],
) {
    let decision = strategy.analyze_market(price_data);

    // Hook 1: Pre-validate the signal
    if !strategy.pre_validate(decision.signal, decision.confidence) {
        println!(
            "[{}] Trade rejected by pre-validation (confidence: {:.4})",
            ticker,
            decision.confidence
        );
        return;
    }

    let shares = strategy.calc_position_size(decision.confidence, strategy.max_position());
    if shares == 0 {
        println!("[{}] No trade: position size is zero", ticker);
        return;
    }

    // Hook 2: Apply risk controls
    if !strategy.apply_risk_controls(ticker, shares) {
        println!(
            "[{}] Trade rejected by risk controls ({} shares)",
            ticker, shares
        );
        return;
    }

    let order = Order::new(ticker, shares, decision.signal);
    log_trade(ticker, shares, decision.signal, decision.confidence, &order);
}
```

### LogTrade (Concrete Shared Function)

```rust
// ── Concrete shared function (analogous to a base class concrete method) ──
fn log_trade(ticker: &str, shares: u32, signal: i32, confidence: f64, order: &Order) {
    let direction = if signal > 0 {
        "BUY"
    } else if signal < 0 {
        "SELL"
    } else {
        "HOLD"
    };
    println!(
        "[2026-07-05, 10:15:30 AM] {}: {} {} shares \
         (confidence: {:.4}) — Order#{} [{}]",
        ticker, direction, shares, confidence, order.order_id, order.status
    );
}
```

### MovingAverageCrossover (Concrete Strategy)

```rust
// ── Concrete Strategy 1: MovingAverageCrossover ──
// OOP role: Implements TradingStrategy trait
// SOLID: LSP, OCP, SRP
struct MovingAverageCrossover;

impl MovingAverageCrossover {
    fn moving_average(&self, data: &[f64], window: usize) -> f64 {
        let start = data.len() - window;
        let sum: f64 = data[start..].iter().sum();
        sum / window as f64
    }
}

impl TradingStrategy for MovingAverageCrossover {
    fn analyze_market(&self, price_data: &[f64]) -> TradingDecision {
        const SHORT: usize = 3;
        const LONG: usize = 8;

        if price_data.len() < LONG {
            return TradingDecision {
                signal: 0,
                confidence: 0.0,
            };
        }

        let short_ma = self.moving_average(price_data, SHORT);
        let long_ma = self.moving_average(price_data, LONG);

        // Compute previous period moving averages for crossover detection
        let prev_data = &price_data[..price_data.len() - 1];
        let prev_short = self.moving_average(prev_data, SHORT);
        let prev_long = self.moving_average(prev_data, LONG);

        if short_ma > long_ma && prev_short <= prev_long {
            let confidence = f64::min((short_ma - long_ma) / long_ma, 1.0);
            return TradingDecision {
                signal: 1,
                confidence,
            };
        } else if short_ma < long_ma && prev_short >= prev_long {
            let confidence = f64::min((long_ma - short_ma) / long_ma, 1.0);
            return TradingDecision {
                signal: -1,
                confidence,
            };
        }

        TradingDecision {
            signal: 0,
            confidence: 0.0,
        }
    }

    fn calc_position_size(&self, confidence: f64, max_position: u32) -> u32 {
        let scaled = max_position as f64 * f64::min(confidence * 1.2, 1.0);
        scaled.floor() as u32
    }
}
```

### MeanReversion (Concrete Strategy)

```rust
// ── Concrete Strategy 2: MeanReversion ──
// OOP role: Implements TradingStrategy trait
// SOLID: LSP, OCP, SRP
struct MeanReversion;

impl TradingStrategy for MeanReversion {
    fn analyze_market(&self, price_data: &[f64]) -> TradingDecision {
        const LOOKBACK: usize = 10;
        const THRESHOLD: f64 = 0.02;

        if price_data.len() < LOOKBACK + 1 {
            return TradingDecision {
                signal: 0,
                confidence: 0.0,
            };
        }

        let current_price = price_data[price_data.len() - 1];
        let recent: &[f64] = &price_data[price_data.len() - LOOKBACK..];
        let mean_price: f64 = recent.iter().sum::<f64>() / LOOKBACK as f64;
        let deviation = (mean_price - current_price) / mean_price;

        if deviation > THRESHOLD {
            return TradingDecision {
                signal: 1,
                confidence: f64::min(deviation, 0.5),
            };
        } else if deviation < -THRESHOLD {
            return TradingDecision {
                signal: -1,
                confidence: f64::min(deviation.abs(), 0.5),
            };
        }

        TradingDecision {
            signal: 0,
            confidence: 0.0,
        }
    }

    fn calc_position_size(&self, confidence: f64, max_position: u32) -> u32 {
        // Mean reversion is more conservative: 60% of standard sizing
        let scaled = max_position as f64 * confidence * 0.6;
        scaled.floor() as u32
    }

    // Override hook: tighter risk controls
    fn apply_risk_controls(&self, _ticker: &str, shares: u32) -> bool {
        const MAX_TRADE_SIZE: u32 = 500;
        if shares > MAX_TRADE_SIZE {
            return false;
        }
        // Call the default hook via the trait (must use fully qualified syntax)
        <Self as TradingStrategy>::apply_risk_controls(self, _ticker, shares)
    }

    // Override hook: stricter confidence threshold
    fn pre_validate(&self, signal: i32, confidence: f64) -> bool {
        const MIN_CONFIDENCE_STRICT: f64 = 0.25;
        confidence >= MIN_CONFIDENCE_STRICT
    }
}
```

### Language-Specific Notes

- **Traits instead of abstract classes** — Rust does not have inheritance or abstract classes. The `TradingStrategy` trait defines the contract with required methods (no default body) and hooks (with default body). This is more flexible than inheritance because types from different module hierarchies can implement the same trait.
- **Generic template function** — `execute_trade<T: TradingStrategy>()` is a free function generic over the strategy type. This is the idiomatic Rust equivalent of a template method. It provides static dispatch (monomorphization) at compile time, resulting in zero runtime overhead compared to virtual method dispatch in class-based languages.
- **Trait objects for heterogeneous collections** — The driver uses `Vec<Box<dyn TradingStrategy>>` for runtime polymorphism when storing different strategies in a collection. This uses dynamic dispatch (vtable pointers), which has a small runtime cost but allows mixing strategy types in a single vector.
- **`<Self as TradingStrategy>::` syntax** — In the `apply_risk_controls` override in `MeanReversion`, calling the default trait implementation requires fully qualified syntax: `<Self as TradingStrategy>::apply_risk_controls(...)`. This is because Rust traits are not classes — default methods must be explicitly referenced when you want to call them from an override.
- **No `Order` ownership complexity** — The template function creates `Order` locally and passes it to `log_trade()` by reference. Rust's ownership model ensures no aliasing or dangling references. The `Order::new()` constructor uses `format!` for string generation, which is both safe and idiomatic.
