## Implementation

Below is a complete JavaScript implementation of the Singleton pattern using a banking domain. The `BankConfiguration` class ensures that every department in the bank — Loan Department, Transaction Service, and Compliance System — accesses the same configuration instance. This guarantees consistency for interest rates, fee structures, regulatory limits, and API endpoints across the entire banking application. The implementation uses ES2022+ private static fields for the instance and private instance fields for the configuration properties.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   BankConfiguration (Singleton)      │  ← OOP: Encapsulation
//   ├──────────────────────────────────────┤     SRP: holds config data
//   │   - #instance (private static)       │     DIP: clients depend on
//   │   - #baseInterestRate (private)      │      this concrete class
//   │   - #savingsInterestRate (private)   │
//   │   - #wireTransferFee (private)       │
//   │   - #maxTransactionLimit (private)   │
//   │   - #minBalanceRequirement (private) │
//   │   - #kycApiEndpoint (private)        │
//   │   - #bankName (private)              │
//   ├──────────────────────────────────────┤
//   │   + static getInstance()             │  ← Global access point
//   │   + getBaseInterestRate()            │
//   │   + getSavingsInterestRate()         │
//   │   + getWireTransferFee()             │
//   │   + getMaxTransactionLimit()         │
//   │   + getMinBalanceRequirement()       │
//   │   + getKycApiEndpoint()              │
//   │   + getBankName()                    │

```

#### OOP Concepts Applied

1. **Encapsulation** — The `BankConfiguration` class hides all configuration data behind private fields (`#baseInterestRate`, `#wireTransferFee`, etc.) and exposes only read-only getter methods. The private static `#instance` field ensures that the singleton instance cannot be accessed or modified from outside the class.

2. **Abstraction** — Clients interact with `BankConfiguration` through a well-defined set of getter methods. They do not need to know how the configuration is loaded, stored, or managed.

3. **Polymorphism** — Not directly applicable to the Singleton pattern itself, but the three client classes (`LoanDepartment`, `TransactionService`, `ComplianceSystem`) all use the same `BankConfiguration` interface polymorphically through the getter methods.

#### SOLID Principles Applied

4. **SRP** — `BankConfiguration` has a single responsibility: holding and providing access to banking configuration. The instance control (ensuring uniqueness) is a secondary concern handled by the `getInstance()` method. The client classes each have their own distinct responsibilities — loan processing, transaction handling, and compliance verification — and they delegate configuration access to the singleton.

5. **OCP** — The system is open for extension: new configuration properties can be added to `BankConfiguration` without modifying any client code. New client classes can access the same singleton without changing the singleton's implementation.

6. **LSP** — While there is no inheritance hierarchy in the singleton, the principle holds for the client classes: they all treat the `BankConfiguration` instance consistently through the same getter interface.

7. **ISP** — The `BankConfiguration` interface is small and focused: each getter method returns a single configuration value. No client is forced to depend on methods it does not use.

8. **DIP** — High-level client classes depend on the concrete `BankConfiguration` singleton, but the abstraction is through the getter interface. Dependency injection could be used to pass the singleton to clients, further decoupling them.

### BankConfiguration (Singleton)

```javascript
// Singleton Pattern — JavaScript — Banking example
// ==================================================
// Participants: BankConfiguration (Singleton),
// LoanDepartment, TransactionService, ComplianceSystem (Clients)
// OOP: encapsulation, abstraction
// SOLID: SRP, OCP, ISP, DIP
// ==================================================

// ── Singleton ──
class BankConfiguration {
  // Private static field holds the single instance
  static #instance = null;

  // Private constructor prevents external instantiation via new
  // Only getInstance() may call this constructor
  constructor() {
    if (BankConfiguration.#instance !== null) {
      throw new Error("Use BankConfiguration.getInstance() to access the singleton");
    }
    this.#baseInterestRate = 4.5;
    this.#savingsInterestRate = 3.75;
    this.#wireTransferFee = 25.00;
    this.#maxTransactionLimit = 50000.00;
    this.#minBalanceRequirement = 500.00;
    this.#kycApiEndpoint = "https://api.globaltrustbank.com/kyc/v2";
    this.#bankName = "Global Trust Bank";
    this.#swiftCode = "GTBKUS44";
    this.#reserveRequirement = 0.10;
  }

  // Private instance fields — configuration properties
  #baseInterestRate;
  #savingsInterestRate;
  #wireTransferFee;
  #maxTransactionLimit;
  #minBalanceRequirement;
  #kycApiEndpoint;
  #bankName;
  #swiftCode;
  #reserveRequirement;

  // Public static method — global access point
  static getInstance() {
    if (BankConfiguration.#instance === null) {
      BankConfiguration.#instance = new BankConfiguration();
    }
    return BankConfiguration.#instance;
  }

  // Public getter methods — read-only access to configuration
  getBaseInterestRate() {
    return this.#baseInterestRate;
  }

  getSavingsInterestRate() {
    return this.#savingsInterestRate;
  }

  getWireTransferFee() {
    return this.#wireTransferFee;
  }

  getMaxTransactionLimit() {
    return this.#maxTransactionLimit;
  }

  getMinBalanceRequirement() {
    return this.#minBalanceRequirement;
  }

  getKycApiEndpoint() {
    return this.#kycApiEndpoint;
  }

  getBankName() {
    return this.#bankName;
  }

  getSwiftCode() {
    return this.#swiftCode;
  }

  getReserveRequirement() {
    return this.#reserveRequirement;
  }
}
```

### LoanDepartment (Client 1)

```javascript
// ── Client 1: LoanDepartment ──
class LoanDepartment {
  processLoanApplication(amount, creditScore) {
    const config = BankConfiguration.getInstance();
    let rate = config.getBaseInterestRate();
    if (creditScore > 700) {
      rate = rate - 1.5;
    }
    const totalInterest = amount * (rate / 100);
    console.log(`LoanDepartment: Processing $${amount.toFixed(2)} loan application`);
    console.log(`  Interest rate: ${rate}%`);
    console.log(`  Annual interest: $${totalInterest.toFixed(2)}`);
    console.log(`  Bank: ${config.getBankName()}`);
    console.log(`  SWIFT: ${config.getSwiftCode()}`);
  }
}
```

### TransactionService (Client 2)

```javascript
// ── Client 2: TransactionService ──
class TransactionService {
  processWireTransfer(amount) {
    const config = BankConfiguration.getInstance();
    if (amount > config.getMaxTransactionLimit()) {
      console.log(`TransactionService: REJECTED — $${amount.toFixed(2)} exceeds limit of $${config.getMaxTransactionLimit().toFixed(2)}`);
      return;
    }
    const fee = config.getWireTransferFee();
    const total = amount + fee;
    console.log(`TransactionService: Processing wire transfer of $${amount.toFixed(2)}`);
    console.log(`  Wire fee: $${fee.toFixed(2)}`);
    console.log(`  Total debit: $${total.toFixed(2)}`);
    console.log(`  Bank: ${config.getBankName()}`);
  }

  checkSavingsRate() {
    const config = BankConfiguration.getInstance();
    const rate = config.getSavingsInterestRate();
    console.log(`TransactionService: Current savings interest rate is ${rate}%`);
  }
}
```

### ComplianceSystem (Client 3)

```javascript
// ── Client 3: ComplianceSystem ──
class ComplianceSystem {
  verifyCustomer(customerId) {
    const config = BankConfiguration.getInstance();
    const endpoint = config.getKycApiEndpoint();
    const minBalance = config.getMinBalanceRequirement();
    console.log(`ComplianceSystem: Verifying customer ${customerId}`);
    console.log(`  KYC endpoint: ${endpoint}`);
    console.log(`  Minimum balance requirement: $${minBalance.toFixed(2)}`);
    console.log(`  Reserve requirement: ${(config.getReserveRequirement() * 100)}%`);
  }
}
```

### Main / Driver

```javascript
// ── Main / Driver ──
function main() {
  console.log("=== Banking System Initialization ===\n");

  // Create client services
  const loanDept = new LoanDepartment();
  const transactionService = new TransactionService();
  const compliance = new ComplianceSystem();

  // Exercise each client — they all use the same BankConfiguration singleton
  loanDept.processLoanApplication(250000.00, 720);
  console.log();
  transactionService.processWireTransfer(15000.00);
  console.log();
  transactionService.checkSavingsRate();
  console.log();
  compliance.verifyCustomer("CUST-4201");
  console.log();

  // Verify singleton identity
  console.log("=== Singleton Identity Verification ===");
  const ref1 = BankConfiguration.getInstance();
  const ref2 = BankConfiguration.getInstance();
  const ref3 = BankConfiguration.getInstance();
  console.log(`ref1 === ref2: ${ref1 === ref2}`);
  console.log(`ref2 === ref3: ${ref2 === ref3}`);
  console.log(`All three references point to the same instance: ${(ref1 === ref2) && (ref2 === ref3)}`);
}

main();
```

### Language-Specific Notes

- **Private fields** — JavaScript ES2022+ supports private fields with the `#` prefix. `#instance` is a private static field, and `#baseInterestRate` etc. are private instance fields. These are truly private at the language level — they cannot be accessed from outside the class, even through reflection or `Object.keys()`.

- **Static initialization** — The `static #instance = null` declaration initializes the static field when the class is first loaded. The `getInstance()` method lazily creates the instance on first access, which is the standard Singleton approach.

- **Constructor guard** — The constructor throws an error if called directly when an instance already exists. This prevents accidental creation of a second instance via `new BankConfiguration()` after the first has been created.

- **No thread-safety** — JavaScript is single-threaded in its execution model (excluding Web Workers), so the simple `if (instance === null) instance = new ...` pattern is safe without locks. In a multi-threaded environment like Node.js Worker Threads, this pattern would need additional synchronization.

- **Module caching** — Node.js natively singletons module exports. An alternative approach in Node.js would be to export a pre-configured instance from the module, leveraging the module cache as the singleton mechanism instead of implementing it manually in the class.
