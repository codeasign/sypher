## Implementation

Below is a complete C# implementation of the Singleton pattern using a banking domain. The `BankConfiguration` class uses `Lazy<T>` from .NET — the simplest and most thread-safe singleton mechanism in C#. The `Lazy<T>` type guarantees that the initialization function runs exactly once and is thread-safe by default. Three client classes — `LoanDepartment`, `TransactionService`, and `ComplianceSystem` — all access the same `BankConfiguration` instance, ensuring consistent interest rates, fee structures, and regulatory settings across the entire banking application.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   BankConfiguration (Singleton)      │  ← OOP: Encapsulation
//   ├──────────────────────────────────────┤     SRP: holds config data
//   │   <<static Lazy<BankConfiguration>>  │
//   │   - _lazy: Lazy<BankConfiguration>   │  ← Thread-safe by default
//   ├──────────────────────────────────────┤
//   │   - baseInterestRate: double         │
//   │   - wireTransferFee: double          │
//   │   - ... (other config fields)        │
//   ├──────────────────────────────────────┤
//   │   + static Instance: BankConfig      │  ← global access point
//   │   + BaseInterestRate: double         │
//   │   + WireTransferFee: double          │
//   └──────────────────────────────────────┘
```

#### OOP Concepts Applied

1. **Encapsulation** — The `BankConfiguration` class hides all data behind `private readonly` fields and exposes only public read-only properties. The `private` constructor prevents external instantiation. The `Lazy<T>` field is also private, encapsulating the singleton lifecycle.

2. **Abstraction** — Clients interact with `BankConfiguration` through public properties without knowledge of how configuration is loaded or how the singleton instance is managed.

3. **Polymorphism** — Not directly applicable to the Singleton pattern, but the three client classes all use the same `BankConfiguration` interface through its public properties.

#### SOLID Principles Applied

4. **SRP** — `BankConfiguration` is responsible only for holding configuration data. The `Lazy<T>` mechanism handles thread-safe initialization, separating concerns.

5. **OCP** — New configuration properties can be added to `BankConfiguration` without modifying any client code. The system is open for extension.

6. **LSP** — While there is no inheritance hierarchy, the typed properties provide consistent behavior across all clients.

7. **ISP** — Each property is individually focused. A client that needs only the wire transfer fee does not depend on properties related to KYC endpoints.

8. **DIP** — High-level client classes depend on `BankConfiguration` through its public properties, not on the details of configuration loading or instance management.

### BankConfiguration (Singleton)

```csharp
// Singleton Pattern — C# — Banking example
// ==========================================
// Participants: BankConfiguration (Singleton),
// LoanDepartment, TransactionService, ComplianceSystem (Clients)
// OOP: encapsulation, abstraction
// SOLID: SRP, OCP, ISP, DIP
// ==========================================

// ── Singleton ──
public class BankConfiguration
{
    // Lazy<T> provides thread-safe lazy initialization
    // LazyThreadSafetyMode.ExecutionAndPublication is the default
    private static readonly Lazy<BankConfiguration> _lazy =
        new(() => new BankConfiguration());

    // Private constructor — prevents external instantiation
    private BankConfiguration()
    {
        // Load all configuration values
        BaseInterestRate = 4.5;
        SavingsInterestRate = 3.75;
        WireTransferFee = 25.00;
        MaxTransactionLimit = 50000.00;
        MinBalanceRequirement = 500.00;
        KycApiEndpoint = "https://api.globaltrustbank.com/kyc/v2";
        BankName = "Global Trust Bank";
        SwiftCode = "GTBKUS44";
        ReserveRequirement = 0.10;
    }

    // Public static property — global access point
    public static BankConfiguration Instance => _lazy.Value;

    // Public read-only properties — configuration data
    public double BaseInterestRate { get; }
    public double SavingsInterestRate { get; }
    public double WireTransferFee { get; }
    public double MaxTransactionLimit { get; }
    public double MinBalanceRequirement { get; }
    public string KycApiEndpoint { get; }
    public string BankName { get; }
    public string SwiftCode { get; }
    public double ReserveRequirement { get; }
}
```

### LoanDepartment (Client 1)

```csharp
// ── Client 1: LoanDepartment ──
public class LoanDepartment
{
    public void ProcessLoanApplication(double amount, int creditScore)
    {
        BankConfiguration config = BankConfiguration.Instance;
        double rate = config.BaseInterestRate;
        if (creditScore > 700)
        {
            rate -= 1.5;
        }
        double annualInterest = amount * (rate / 100.0);
        Console.WriteLine($"LoanDepartment: Processing {amount:C2} loan application");
        Console.WriteLine($"  Interest rate: {rate}%");
        Console.WriteLine($"  Annual interest: {annualInterest:C2}");
        Console.WriteLine($"  Bank: {config.BankName}");
        Console.WriteLine($"  SWIFT: {config.SwiftCode}");
    }
}
```

### TransactionService (Client 2)

```csharp
// ── Client 2: TransactionService ──
public class TransactionService
{
    public void ProcessWireTransfer(double amount)
    {
        BankConfiguration config = BankConfiguration.Instance;
        if (amount > config.MaxTransactionLimit)
        {
            Console.WriteLine($"TransactionService: REJECTED — {amount:C2} exceeds limit of {config.MaxTransactionLimit:C2}");
            return;
        }
        double fee = config.WireTransferFee;
        double total = amount + fee;
        Console.WriteLine($"TransactionService: Processing wire transfer of {amount:C2}");
        Console.WriteLine($"  Wire fee: {fee:C2}");
        Console.WriteLine($"  Total debit: {total:C2}");
        Console.WriteLine($"  Bank: {config.BankName}");
    }

    public void CheckSavingsRate()
    {
        BankConfiguration config = BankConfiguration.Instance;
        double rate = config.SavingsInterestRate;
        Console.WriteLine($"TransactionService: Current savings interest rate is {rate}%");
    }
}
```

### Main / Driver

```csharp
// ── Main / Driver ──
public class Program
{
    public static void Main(string[] args)
    {
        Console.WriteLine("=== Banking System Initialization ===\n");

        var loanDept = new LoanDepartment();
        var transactionService = new TransactionService();
        var compliance = new ComplianceSystem();

        loanDept.ProcessLoanApplication(250000.00, 720);
        Console.WriteLine();
        transactionService.ProcessWireTransfer(15000.00);
        Console.WriteLine();
        transactionService.CheckSavingsRate();
        Console.WriteLine();
        compliance.VerifyCustomer("CUST-4201");
        Console.WriteLine();

        // Singleton identity verification
        Console.WriteLine("=== Singleton Identity Verification ===");
        var ref1 = BankConfiguration.Instance;
        var ref2 = BankConfiguration.Instance;
        var ref3 = BankConfiguration.Instance;
        bool refEq12 = ReferenceEquals(ref1, ref2);
        bool refEq23 = ReferenceEquals(ref2, ref3);
        Console.WriteLine($"ref1 == ref2: {refEq12}");
        Console.WriteLine($"ref2 == ref3: {refEq23}");
        bool allSame = refEq12 && refEq23;
        Console.WriteLine($"All references point to the same instance: {allSame}");
    }
}
```

### Language-Specific Notes

- **`Lazy<T>` for thread safety** — The `Lazy<T>` class in .NET provides a simple, thread-safe mechanism for lazy initialization. By default, `Lazy<T>` uses `LazyThreadSafetyMode.ExecutionAndPublication`, which ensures that the initialization function runs exactly once and that all threads see the same value. This is the recommended approach for singletons in modern C#.

- **Expression-bodied property** — `public static BankConfiguration Instance => _lazy.Value;` uses C#'s expression-bodied member syntax. The `Instance` property is read-only and returns the `Lazy<T>.Value`, which triggers initialization on first access.

- **`ReferenceEquals` for identity** — C# overloads the `==` operator for reference types, but `ReferenceEquals` is the unambiguous way to check reference identity. The driver uses `ReferenceEquals` to verify all references point to the same instance.

- **`readonly` fields and init-only properties** — All configuration properties are declared as `{ get; }` which makes them init-only (set once in the constructor) and immutable after construction. The `_lazy` field is `readonly`, ensuring the `Lazy<T>` reference itself never changes.

- **`volatile` vs `Lazy<T>`** — Before `Lazy<T>` was introduced in .NET Framework 4.0, singletons required `volatile` static fields and double-checked locking. `Lazy<T>` eliminates this boilerplate entirely. For most applications, `Lazy<T>` is both simpler and more performant than manual locking.

- **ASP.NET Core dependency injection** — In ASP.NET Core, the preferred approach is to let the DI container manage singleton lifetime by registering the service as `AddSingleton<T>()`. This gives the same singleton guarantee without implementing the pattern manually. The manual implementation is still useful for non-DI applications or libraries.
