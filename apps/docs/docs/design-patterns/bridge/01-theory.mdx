## Intent

The Bridge pattern decouples an abstraction from its implementation so that the two can vary independently. Instead of binding an abstraction to a fixed implementation through inheritance, the Bridge pattern uses composition: the abstraction holds a reference to an implementation interface, and both sides can be extended without affecting the other. This eliminates the combinatorial explosion of subclasses that occurs when you try to cover every combination of abstraction and implementation through inheritance alone.

## The Problem Without This Pattern

Consider a car manufacturing system. You have different car types вҖ” Sedan, SUV, Truck вҖ” and different engine types вҖ” GasEngine, ElectricEngine, HybridEngine. Without the Bridge pattern, you would need a separate class for every combination:

```
Car                            CarWithEngine
в”ңв”Җв”Җ Sedan                      в”ңв”Җв”Җ SedanGas
в”Ӯ   в”ңв”Җв”Җ SedanGas               в”ңв”Җв”Җ SedanElectric
в”Ӯ   в”ңв”Җв”Җ SedanElectric          в”ңв”Җв”Җ SedanHybrid
в”Ӯ   в””в”Җв”Җ SedanHybrid            в”ңв”Җв”Җ SUVGas
в”ңв”Җв”Җ SUV                        в”ңв”Җв”Җ SUVElectric
в”Ӯ   в”ңв”Җв”Җ SUVGas                 в”ңв”Җв”Җ SUVHybrid
в”Ӯ   в”ңв”Җв”Җ SUVElectric            в”ңв”Җв”Җ TruckGas
```

Adding a new car type (Convertible) or a new engine type (HydrogenEngine) multiplies the number of classes. With 4 car types and 4 engine types, you need 16 classes. With 5 and 5, you need 25. This is the combinatorial explosion problem.

### Inheritance Lock-In

```python
# BAD: Inheritance-based approach вҖ” every combination is a new class
class Car:
    def drive(self): ...

class Sedan(Car):
    def drive(self): return "Sedan moving"

class SedanGas(Sedan):
    def drive(self): return "Sedan with gas engine moving"

class SedanElectric(Sedan):
    def drive(self): return "Sedan with electric engine moving"

# Adding SUV forces 3+ more classes. Adding Truck forces 3+ more.
# Adding a Hydrogen engine forces changes across all 3 car types.
```

The inheritance approach creates a rigid hierarchy. Each new car type requires a new subclass for every engine type. Each new engine type requires a new subclass for every car type. The abstraction (car type) and the implementation (engine type) are permanently coupled through the class hierarchy.

### Code Duplication Across the Hierarchy

The inheritance approach forces duplicated logic. If all Sedan variants share a trunk-opening mechanism, you must either implement it in each subclass (SedanGas, SedanElectric, SedanHybrid) or push it to a shared parent вҖ” but the shared parent (Sedan) no longer exists in a flat hierarchy because the engine type is baked into the class name.

```python
# BAD: Duplicate trunk logic in every Sedan variant
class SedanGas(Sedan):
    def drive(self):
        return "Sedan with gas engine driving"
    def open_trunk(self):
        return "Trunk opens (15.5 cu ft)"

class SedanElectric(Sedan):
    def drive(self):
        return "Sedan with electric engine driving"
    def open_trunk(self):
        return "Trunk opens (15.5 cu ft)"  # Same as above вҖ” duplicated

class SedanHybrid(Sedan):
    def drive(self):
        return "Sedan with hybrid engine driving"
    def open_trunk(self):
        return "Trunk opens (15.5 cu ft)"  # Duplicated again
```

This duplication becomes maintenance nightmare. Changing trunk capacity from 15.5 to 16.0 cu ft requires editing every Sedan variant. Forgetting one variant introduces a subtle bug that may go undetected until production.

### The N x M Problem in the Real World

In a real automotive configurator, the dimensions of variation are rarely limited to two. Consider:

```
Car types:         Sedan, SUV, Truck, Coupe, Convertible, Wagon  (6 types)
Engine types:      Gas, Diesel, Electric, Hybrid, Hydrogen          (5 types)
Transmission:      Manual, Automatic, CVT, DCT                      (4 types)
Drivetrain:        FWD, RWD, AWD                                    (3 types)
```

Without the Bridge pattern, the number of classes needed would be 6 x 5 x 4 x 3 = 360 classes. With the Bridge pattern, you need 6 + 5 + 4 + 3 = 18 classes. The Bridge pattern collapses multiplicative complexity into additive complexity. This is why the Bridge pattern is essential in systems with multiple orthogonal dimensions of variation.

### Participants

1. **Abstraction (Car)** вҖ” Defines the high-level interface. Holds a reference to the Implementor (Engine) and delegates engine-specific work to it. The client interacts with the Abstraction, not the Implementor directly. In the Cars domain, `Car` defines `drive()` and delegates to `engine.start()` and `engine.sound()`.

2. **RefinedAbstraction (Sedan, SUV, Truck)** вҖ” Extends the Abstraction with type-specific behavior. Sedan adds `openTrunk()` for sedan-specific cargo access. SUV adds `engage4WD()` for off-road traction control. Truck adds payload and towing specifications. Each RefinedAbstraction implements `drive()` in a way that uses the engine reference polymorphically.

3. **Implementor (Engine interface)** вҖ” Defines the low-level interface for all engine types. Declares `start()`, `sound()`, and `fuelType()` methods that all concrete engines must implement. This interface is independent of the car type hierarchy. The Engine does not know whether it is installed in a Sedan, SUV, or Truck.

4. **ConcreteImplementor (GasEngine, ElectricEngine, HybridEngine)** вҖ” Implements the Engine interface with specific behavior. GasEngine uses combustion logic with cylinder count and displacement. ElectricEngine uses battery capacity and motor power. HybridEngine combines both strategies and coordinates between them. Each ConcreteImplementor provides its own `start()` and `sound()` while conforming to the same interface.

## Pseudocode вҖ” Cars Example

```
// PSEUDOCODE вҖ” Bridge Pattern in Cars Domain
// Participants labeled with [ROLE]

// в”Җв”Җ [IMPLEMENTOR] в”Җв”Җ
interface Engine:
    // The implementation interface вҖ” all engines must support these
    method start() -> void
    method sound() -> string
    method fuelType() -> string

// в”Җв”Җ [CONCRETE IMPLEMENTOR: GasEngine] в”Җв”Җ
class GasEngine implements Engine:
    private cylinders: int
    private displacement: float  // in liters

    method __init__(cylinders: int, displacement: float):
        this.cylinders = cylinders
        this.displacement = displacement

    method start():
        print("Gas engine fires: *vroom vroom*")
        // Cranks the starter, injects fuel, ignites spark plugs

    method sound():
        return "VROOOOM" + "." * cylinders

    method fuelType():
        return "Gasoline (" + displacement + "L " + cylinders + "-cyl)"

// в”Җв”Җ [CONCRETE IMPLEMENTOR: ElectricEngine] в”Җв”Җ
class ElectricEngine implements Engine:
    private batteryCapacity: float  // in kWh
    private motorPower: int         // in kW

    method __init__(batteryCapacity: float, motorPower: int):
        this.batteryCapacity = batteryCapacity
        this.motorPower = motorPower

    method start():
        print("Electric motor engages: *whirrr*")
        // Inverter powers the stator windings, rotor spins

    method sound():
        return "WHIIIIRRRR"

    method fuelType():
        return "Electric (" + batteryCapacity + "kWh, " + motorPower + "kW)"

// в”Җв”Җ [CONCRETE IMPLEMENTOR: HybridEngine] в”Җв”Җ
class HybridEngine implements Engine:
    private gasEngine: GasEngine
    private electricEngine: ElectricEngine
    private batteryLevel: int  // percentage 0-100

    method __init__(gasEngine: GasEngine, electricEngine: ElectricEngine,
                    batteryLevel: int):
        this.gasEngine = gasEngine
        this.electricEngine = electricEngine
        this.batteryLevel = batteryLevel

    method start():
        electricEngine.start()  // Electric motor spins up first
        if batteryLevel < 20:
            gasEngine.start()   // Gas engine kicks in when battery is low

    method sound():
        if batteryLevel >= 20:
            return electricEngine.sound()
        return gasEngine.sound() + " then " + electricEngine.sound()

    method fuelType():
        return "Hybrid (Gas + Electric, battery at " + batteryLevel + "%)"

// в”Җв”Җ [ABSTRACTION] в”Җв”Җ
abstract class Car:
    protected engine: Engine  // Bridge to the implementation

    method __init__(engine: Engine):
        this.engine = engine   // Composition: Car HAS-A Engine

    method drive() -> string:
        // Abstract вҖ” each car type drives differently
        abstract

    method startEngine():
        engine.start()
        print("Car is ready.")

    method getSpecs() -> string:
        return "Car with " + engine.fuelType()

// в”Җв”Җ [REFINED ABSTRACTION: Sedan] в”Җв”Җ
class Sedan extends Car:
    private trunkCapacity: float  // in cubic feet

    method __init__(engine: Engine, trunkCapacity: float):
        super.__init__(engine)
        this.trunkCapacity = trunkCapacity

    method drive():
        engine.start()
        return "Sedan cruising smoothly on the highway. " + engine.sound()

    method openTrunk():
        print("Trunk opens (" + trunkCapacity + " cu ft).")

    method getSpecs():
        return "Sedan with " + engine.fuelType() +
               " | Trunk: " + trunkCapacity + " cu ft"

// в”Җв”Җ [REFINED ABSTRACTION: SUV] в”Җв”Җ
class SUV extends Car:
    private groundClearance: float  // in inches
    private awd: boolean

    method __init__(engine: Engine, groundClearance: float, awd: boolean):
        super.__init__(engine)
        this.groundClearance = groundClearance
        this.awd = awd

    method drive():
        engine.start()
        return "SUV tackling off-road terrain. " + engine.sound()

    method engage4WD():
        if awd:
            print("All-wheel drive engaged. Traction optimized.")
        else:
            print("This SUV does not have AWD.")

    method getSpecs():
        return "SUV with " + engine.fuelType() +
               " | Ground: " + groundClearance + "\" " +
               ("AWD" if awd else "2WD")

// в”Җв”Җ [REFINED ABSTRACTION: Truck] в”Җв”Җ
class Truck extends Car:
    private payloadCapacity: float  // in pounds
    private towingCapacity: float   // in pounds

    method __init__(engine: Engine, payloadCapacity: float, towingCapacity: float):
        super.__init__(engine)
        this.payloadCapacity = payloadCapacity
        this.towingCapacity = towingCapacity

    method drive():
        engine.start()
        return "Truck hauling heavy load. " + engine.sound()

    method getSpecs():
        return "Truck with " + engine.fuelType() +
               " | Payload: " + payloadCapacity + "lbs | " +
               "Towing: " + towingCapacity + "lbs"

// в”Җв”Җ CLIENT CODE в”Җв”Җ
gasEngine = new GasEngine(6, 3.5)
sedan = new Sedan(gasEngine, 15.5)
print(sedan.getSpecs())
// Output: Sedan with Gasoline (3.5L 6-cyl) | Trunk: 15.5 cu ft
print(sedan.drive())
// Output: Gas engine fires: *vroom vroom*
//         Sedan cruising smoothly on the highway. VROOOOM......

electricEngine = new ElectricEngine(75.0, 250)
suv = new SUV(electricEngine, 8.7, true)
print(suv.getSpecs())
// Output: SUV with Electric (75.0kWh, 250kW) | Ground: 8.7" AWD

hybridEngine = new HybridEngine(
    new GasEngine(4, 2.0),
    new ElectricEngine(12.0, 80),
    35
)
truck = new Truck(hybridEngine, 2000, 7500)
print(truck.getSpecs())
// Output: Truck with Hybrid (Gas + Electric, battery at 35%) | Payload: 2000lbs | Towing: 7500lbs
```

## How It Works вҖ” Step by Step

1. **Define the Implementor interface.** The Engine interface declares the operations that all engines must support: `start()`, `sound()`, and `fuelType()`. This interface is the bridge point вҖ” it is the only thing the Car abstraction knows about engines. The interface must be stable: once Car subclasses start calling `engine.sound()`, changing the interface signature breaks all clients. Good interface design from the start is critical.

2. **Create ConcreteImplementors.** Each engine type (GasEngine, ElectricEngine, HybridEngine) implements the Engine interface with its own internal logic. GasEngine uses cylinder count and displacement to determine its sound profile. ElectricEngine uses battery capacity and motor power вҖ” it produces no combustion sound, only electric motor whir. HybridEngine composes a gas and electric engine together and conditionally delegates based on battery level. Each ConcreteImplementor is completely independent of the Car hierarchy.

3. **Define the Abstraction.** The Car abstract class declares the high-level `drive()` method and holds a reference to an Engine via composition. Car delegates engine-specific operations to the Engine reference rather than implementing them directly. The choice of composition over inheritance is the key architectural decision: Car has-an Engine instead of Car is-a specific engine type.

4. **Create RefinedAbstractions.** Sedan, SUV, and Truck extend Car with their own drive behavior and type-specific methods. Sedan adds `openTrunk()` for its unique cargo access. SUV adds `engage4WD()` for off-road traction. Truck adds payload and towing specifications. Each RefinedAbstraction calls `engine.start()` and `engine.sound()` inside its `drive()` method, but the exact implementation of those engine methods depends on the ConcreteImplementor that was injected at construction time.

5. **The client composes Abstraction with Implementor.** The client creates a ConcreteImplementor (e.g., `ElectricEngine` with 75 kWh and 250 kW) and passes it to the Car constructor. This composition is the key insight of the Bridge pattern: the client decides which implementation to use at runtime. The same Sedan class can produce a gasoline sedan, an electric sedan, or a hybrid sedan depending on which engine object is passed to its constructor.

6. **The client calls only the Abstraction interface.** The client calls `car.drive()` and `car.getSpecs()` without knowing whether the car has a gas, electric, or hybrid engine. The Car abstraction delegates to the Engine implementation behind the scenes. The two hierarchies remain independent: you can add a HydrogenEngine without modifying any Car subclass, and you can add a Convertible car type without modifying any Engine subclass.

7. **Both hierarchies can grow independently.** Adding a new engine type (HydrogenEngine) requires one new class that implements Engine, and that engine can be used in any existing or future Car subclass. Adding a new car type (Convertible) requires one new class that extends Car, and that car can use any existing or future Engine implementation. No existing code changes. The Bridge pattern eliminates the N x M class explosion by replacing multiplicative inheritance with additive composition.

## When to Use

- **You want to avoid a permanent binding between an abstraction and its implementation.** The Car should not be hardcoded to a specific engine. The Bridge pattern lets you swap engines at runtime through composition so that a Sedan can be gasoline today, electric tomorrow, and hybrid next month вҖ” all without any code change to the Sedan class.

- **Both the abstraction and the implementation should be extensible through subclassing.** You need to add new car types and new engine types independently, and neither hierarchy should depend on the other's concrete types. The Bridge pattern keeps these two hierarchies separate, allowing each to grow without affecting the other.

- **Changes to the implementation should not affect the client.** The client code that calls `car.drive()` should not change when you add a new engine type. The abstraction insulates the client from implementation changes. This is the Dependency Inversion Principle in action: high-level modules (Car clients) depend on abstractions (Car), not on concrete details.

- **You have a combinatorial explosion of classes.** If you find yourself creating classes like `SedanGas`, `SedanElectric`, `SedanHybrid`, `SUVGas`, `SUVElectric`, `SUVHybrid`, `TruckGas`, `TruckElectric`, `TruckHybrid` вҖ” you need the Bridge pattern to collapse the N x M hierarchy into N + M classes. This is the primary signal that triggers Bridge pattern adoption.

- **You want to share an implementation among multiple objects.** Multiple car types can share the same engine instance. A `GasEngine(6, 3.5)` with the same specification can be used in a Sedan, an SUV, and a Truck without duplicating code or creating separate engine instances for each car. This is especially valuable when engine objects are expensive to create or carry heavy state.

- **The implementation details should be hidden from the client.** The client interacts with Car and does not need to know about engine internals like cylinder count, battery capacity, or motor power ratings. The Car's `getSpecs()` method provides a human-readable summary, but the client never accesses engine parameters directly.

- **Your domain has multiple orthogonal dimensions of variation.** When a system has several independent axes of change вҖ” such as car type, engine type, transmission type, and drivetrain вҖ” the Bridge pattern can be applied recursively. Each dimension becomes its own hierarchy, connected through composition at the next level up. This prevents the combinatorial explosion that would occur with a flat inheritance tree.

## When NOT to Use

- **There is only one implementation and no foreseeable need for more.** If your application will ever have only one engine type, the Bridge pattern adds unnecessary indirection. A simple inheritance hierarchy with `Car` and `Sedan`, `SUV`, `Truck` subclasses is sufficient. Introducing the Bridge pattern preemptively for a single engine type is premature abstraction.

- **The abstraction and implementation are permanently and intrinsically tied.** A Car that physically cannot have an engine other than its built-in one does not benefit from the Bridge pattern. The pattern is only useful when both sides can vary. If your domain has exactly one engine type for all eternity, the interface and indirection add complexity without benefit.

- **You are working in a performance-critical code path with millions of instantiations.** The Bridge pattern adds an extra level of indirection (the Engine reference) and dynamic dispatch (virtual method calls through the interface). In hot loops that allocate and destroy thousands of objects per second, this overhead can matter. For most business applications, the overhead is negligible.

- **The abstraction hierarchy is stable and small.** If you have exactly one car type and one engine type and no plans to add more, the Bridge pattern is over-engineering. A simple Car class with inline engine logic вҖ” or even a function that accepts an engine type parameter вҖ” is simpler, more readable, and equally maintainable.

- **The team is unfamiliar with the pattern and the domain is simple.** For a codebase with three car types and two engine types that never changes, the Bridge pattern adds conceptual overhead without proportional benefit. The pattern is a tool for managing complexity, not a checkbox to include in every design. Simple domains deserve simple solutions.

- **Performance matters more than flexibility in the specific code path.** If the engine start operation is called in a real-time control loop with microsecond deadlines, the virtual dispatch overhead of the Bridge pattern may be unacceptable. In such cases, monomorphized code or compile-time polymorphism (e.g., C++ templates, Rust generics) may be more appropriate.

## Real-World Occurrences

1. **Java AWT and Swing вҖ” Component and Peer architecture.** The Java Abstract Window Toolkit uses the Bridge pattern to separate GUI components (Button, TextField, List) from their platform-specific implementations (MotifButton, WindowsButton, MacButton). The Component hierarchy is the Abstraction, and the Peer interface is the Implementor. This allows Java to render the same component on different operating systems without changing the component hierarchy. When a Java application creates a `Button`, the AWT framework selects the correct platform peer at runtime, and the button draws itself using native OS widgets.

2. **JDBC (Java Database Connectivity).** The JDBC API is a textbook Bridge pattern. The `java.sql.Connection`, `Statement`, `ResultSet`, and `PreparedStatement` classes form the Abstraction hierarchy. The JDBC drivers (MySQL Driver, PostgreSQL Driver, Oracle Driver, SQLite Driver) are the ConcreteImplementors. Application code writes SQL queries against the JDBC interfaces, and the selected driver handles database-specific protocol communication, connection pooling, and result set formatting. Adding a new database requires only a new driver JAR вҖ” no application code changes.

3. **.NET Stream classes.** The .NET `System.IO.Stream` abstract class is the Abstraction. Concrete subclasses (`FileStream`, `MemoryStream`, `NetworkStream`, `GZipStream`, `CryptoStream`) are RefinedAbstractions. The underlying OS handle, buffer, or socket is the Implementor. The `Stream` class delegates read/write/seek operations to the platform-specific implementation. This lets .NET code read from files, memory buffers, network sockets, compressed archives, and encrypted data through the same `Stream` interface. The `Stream` abstraction is extended with decorators (`BufferedStream`, `CryptoStream`) that wrap other streams вҖ” a combination of Bridge and Decorator patterns.

4. **Embedded systems and HAL (Hardware Abstraction Layer).** Automotive ECUs (Electronic Control Units) use the Bridge pattern extensively through hardware abstraction layers. The abstraction layer defines interfaces for sensors (temperature, pressure, speed, oxygen) and actuators (fuel injectors, throttle valves, ignition coils, motor drivers). The hardware-specific implementation handles register-level communication, I2C/SPI/CAN bus protocols, and pin mapping. The same sensor abstraction can work with Bosch, Continental, or Denso hardware without changing the engine control logic. When an OEM switches sensor suppliers, only the ConcreteImplementor changes.

5. **Cross-platform GUI frameworks (Qt, wxWidgets, GTK).** Qt uses the Bridge pattern between its widget classes (QPushButton, QSlider, QTable, QTreeView) and their platform-specific paint implementations (QStyle). The QWidget is the Abstraction, and QStyle / QPaintDevice are the Implementor interfaces. This separation allows Qt applications to render natively on Windows (using GDI/GDI+), macOS (using Quartz), and Linux (using X11 or Wayland) from the same codebase. The same QPushButton looks like a Windows button on Windows, a macOS button on macOS, and a GTK-style button on Linux вҖ” all without application code changes.

6. **Operating system virtual file systems.** The VFS (Virtual File System) layer in Linux, FreeBSD, and Windows uses the Bridge pattern. The abstraction defines file operations (open, read, write, close, seek, mmap) through a standard interface that all application code calls. The implementation handles filesystem-specific on-disk structures (ext4, NTFS, FAT32, ZFS, Btrfs, XFS). Adding a new filesystem requires implementing the VFS interface for that filesystem вҖ” no changes to the VFS dispatch layer or to application code. The VFS Bridge is what allows Linux to support dozens of filesystems transparently.

7. **Graphics rendering APIs (Skia, Cairo, Direct2D).** Google's Skia graphics library uses the Bridge pattern to separate its drawing abstraction from platform-specific backends. Skia's `SkCanvas` is the Abstraction вҖ” application code calls `drawRect()`, `drawCircle()`, `drawText()`. The backend implementations (GPU via OpenGL/Vulkan/Metal, CPU raster, PDF, SVG) are the ConcreteImplementors. The same Skia code runs on Android, Chrome, Flutter, and Fuchsia, producing correct output through different rendering backends.

8. **Device drivers in monolithic kernels.** The Linux kernel's device driver model is a Bridge pattern at scale. The abstraction defines `struct file_operations` вҖ” a set of function pointers for open, release, read, write, ioctl, mmap, and poll. Each device driver (ConcreteImplementor) fills in these function pointers with hardware-specific logic. The VFS layer (Abstraction) calls through these pointers without knowing whether the underlying device is a SATA SSD, NVMe drive, USB flash drive, or network block device. This Bridge architecture is what allows Linux to support thousands of hardware devices through a single system call interface.

## Key Implementation Considerations

### 1. Interface Design Is Critical

The Engine interface is the contract between the two hierarchies. Design it carefully from the start because changing it later requires updating all ConcreteImplementors. A well-designed Engine interface:

- Declares only the operations that all engines genuinely share. `start()`, `sound()`, and `fuelType()` are universal across all engine types.
- Avoids engine-type-specific methods at the interface level. A method like `chargeBattery()` belongs on ElectricEngine, not on the Engine interface, because GasEngine does not need it.
- Provides a stable abstraction that new engine types can implement without modifying existing code.

If the interface proves too restrictive, consider adding default method implementations (in Java 8+, C#, or Kotlin) so that new methods do not break existing implementations.

### 2. Constructor Injection vs Setter Injection

The Bridge pattern typically uses constructor injection to bind the abstraction to the implementation:

```java
// Constructor injection вҖ” preferred for Bridge
Car sedan = new Sedan(new GasEngine(6, 3.5), 15.5);
```

Constructor injection ensures the Car is always in a valid state вҖ” it cannot exist without an engine. Setter injection is an alternative but introduces a temporal coupling window where the Car exists without an engine:

```java
// Setter injection вҖ” allows incomplete state
Sedan sedan = new Sedan(15.5);
sedan.setEngine(new GasEngine(6, 3.5));  // Car existed without engine briefly
```

Constructor injection is the recommended approach because it guarantees the invariant that every Car has an Engine.

### 3. Shared vs Exclusive Engine Instances

Engines can be shared across car instances or dedicated per car:

```java
// Shared engine вҖ” one V8 powers multiple cars (same spec)
Engine v8 = new GasEngine(8, 5.3);
Car sedan = new Sedan(v8, 15.5);
Car suv = new SUV(v8, 10.2, true);
// Both cars share the same engine specification object

// Exclusive engine вҖ” each car gets its own engine instance
Car sedan2 = new Sedan(new GasEngine(8, 5.3), 16.0);
Car suv2 = new SUV(new GasEngine(8, 5.3), 10.2, true);
```

Sharing is useful when the Engine object carries expensive state (e.g., a connection pool to a diagnostic system). Exclusive instances are simpler when Engines are lightweight value objects.

### 4. Combining Bridge with Other Patterns

The Bridge pattern combines naturally with several other patterns:

- **Abstract Factory** вҖ” Creates the appropriate engine for a given car configuration. The factory encapsulates the decision of which ConcreteImplementor to instantiate.
- **Factory Method** вҖ” Each Car subclass can override a factory method that provides the default engine type.
- **Prototype** вҖ” Clone a template car-engine configuration rather than constructing from scratch.
- **Singleton** вҖ” A single engine specification object can be shared across cars.

### 5. Performance and Memory Considerations

The Bridge pattern adds:
- One pointer-sized field per Abstraction instance (the Engine reference)
- One virtual method dispatch per engine operation
- One heap allocation for each Box/dyn or shared_ptr

For most applications, this overhead is negligible. For real-time control systems or embedded devices with strict memory budgets, consider:
- Using static polymorphism (C++ templates, Rust generics) when the engine type is known at compile time
- Sharing engine instances across cars to reduce memory allocation
- Using a flyweight pattern for engine specifications that repeat across many cars

### 6. Testing Each Hierarchy Independently

The Bridge pattern makes testing straightforward because each hierarchy can be tested in isolation:

```java
// Test the Engine hierarchy independently
void testElectricEngine() {
    Engine e = new ElectricEngine(75.0, 250);
    assertEquals("WHIIIIRRRR", e.sound());
    assertEquals("Electric (75.0kWh, 250kW)", e.fuelType());
}

// Test the Car hierarchy with a mock Engine
void testSedanWithMockEngine() {
    Engine mock = mock(Engine.class);
    when(mock.sound()).thenReturn("MOCK");
    Sedan sedan = new Sedan(mock, 15.5);
    assertTrue(sedan.drive().contains("MOCK"));
}
```

The independent testability is a significant maintainability advantage, especially in large codebases with many car and engine types.

## Integration with Modern Architectures

### Dependency Injection Frameworks

In Spring (Java), ASP.NET Core (C#), or Guice, the Bridge pattern integrates naturally with dependency injection:

```java
@Configuration
class CarConfig {
    @Bean
    @Scope("prototype")
    public Engine gasEngine() {
        return new GasEngine(6, 3.5);
    }

    @Bean
    @Scope("prototype")
    public Car sedan() {
        return new Sedan(gasEngine(), 15.5);
    }
}
```

The DI container manages the composition of Car and Engine, and swapping engines becomes a configuration change rather than a code change.

### Microservices and API Versioning

The Bridge pattern appears at the architectural level in microservice design. The service interface (Abstraction) is the API contract. The implementation (ConcreteImplementor) can be versioned independently. As long as the API contract remains stable, the implementation can change, be migrated to a different language, or be replaced entirely without affecting clients.

### Plugin Architectures

The Bridge pattern is the foundation of plugin systems. The Engine interface becomes the plugin contract. Third-party developers implement new engines (HydrogenEngine, RotaryEngine, SteamEngine) without modifying the Car hierarchy. The plugin is loaded at runtime and composed with existing Car types through the bridge.

## Typical Interaction Flow

The sequence of method calls when a client drives a car illustrates how the Bridge pattern routes behavior through the abstraction to the implementation:

1. The client calls `sedan.drive()` (on the Abstraction).
2. Inside `drive()`, the Sedan calls `this.engine.start()` (delegating to the Implementor).
3. The Engine's `start()` method executes the engine-specific startup logic (printing to console, spinning up the motor, etc.).
4. Control returns to `drive()`, which then calls `this.engine.sound()` (another delegation to the Implementor).
5. The Engine's `sound()` method returns the engine-specific sound string.
6. The Sedan's `drive()` method returns a combined string that includes the engine sound.

The client never directly calls `engine.start()` or `engine.sound()`. All engine operations are mediated through the Car abstraction. This indirection is what makes the two hierarchies independent: the Sedan does not know whether the engine is gas, electric, or hybrid, and the engine does not know whether it is in a sedan, SUV, or truck.

## Consequences

### Benefits

- **Decoupled hierarchies.** The abstraction and implementation can vary independently. Adding a new car type does not affect any engine class and vice versa. This is the primary benefit and the reason the pattern exists.

- **Elimination of combinatorial explosion.** The Bridge pattern replaces N x M classes with N + M classes. For large values of N and M, this dramatically reduces the number of classes and simplifies the codebase.

- **Runtime flexibility.** The engine is injected at construction time, which means the same Car subclass can work with different engines. The client decides which implementation to use, potentially based on runtime configuration, user preference, or factory output.

- **Improved testability.** Each hierarchy can be tested independently. You can test Sedan logic with a mock Engine that returns canned sound strings. You can test GasEngine logic with a test harness that checks start/stop behavior. Mocking the Engine interface in tests is straightforward.

- **Clean separation of concerns.** The Car abstraction handles car-level behavior (driving dynamics, cargo access, drivetrain). The Engine implementation handles engine-level behavior (combustion, electric motor control, hybrid coordination). Neither crosses into the other's domain.

- **Open/Closed compliance.** Both hierarchies are open for extension (new car types, new engine types) but closed for modification. Adding a Convertible requires no changes to existing code вҖ” just a new class extending Car.

### Liabilities

- **Increased complexity.** The Bridge pattern introduces an extra interface, an extra level of indirection, and the architectural discipline of keeping two hierarchies separate. For simple domains, this is over-engineering.

- **Runtime indirection overhead.** Every engine operation involves a virtual method call through the interface reference. In performance-critical paths, this dispatch overhead can accumulate.

- **Interface stability pressure.** The Engine interface must be designed correctly from the start. If a new engine type needs a method that existing engines do not support, all existing implementations must be updated or a default must be provided.

- **Abstraction leakage.** There is a risk that car-specific concerns leak into the engine interface or vice versa. For example, if a HybridEngine needs to report battery level, that requirement must be part of the Engine interface, potentially forcing gas and electric engines to implement dummy battery-level methods.

- **Construction complexity.** The client must know about both the car type and the engine type to construct the composite object. This often requires a Factory or Abstract Factory to encapsulate the construction logic and prevent clients from depending on both hierarchies.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Adapter** | Adapter makes unrelated classes work together by converting one interface to another. Bridge separates an abstraction from its implementation intentionally, from the start. | Choose Adapter when you have existing incompatible interfaces that need to cooperate and you cannot change either side. Choose Bridge when you are designing the abstraction and implementation to vary independently from the beginning. |
| **Strategy** | Strategy lets you swap algorithms (behavior) at runtime through a common interface. Bridge lets you swap both the abstraction and the implementation independently. The intent is structural (decoupling hierarchies) rather than behavioral (swapping algorithms). | Choose Strategy when you need to vary only the behavior of a single object, not the object's type hierarchy. Choose Bridge when you have two orthogonal dimensions of variation (car type and engine type). |
| **Abstract Factory** | Abstract Factory creates families of related objects. Bridge separates an abstraction from its implementation. They often work together: Abstract Factory can configure a Bridge with a specific implementation. | Choose Abstract Factory when you need to ensure that objects from a family are used together (e.g., sedan + gas engine + manual transmission). Choose Bridge when the goal is to decouple two hierarchies so they can vary independently. |
| **State** | State changes an object's behavior when its internal state changes. Bridge separates an abstraction from its implementation permanently. State is about dynamic behavior change; Bridge is about static structural decoupling. | Choose State when an object must change its behavior at runtime based on internal state transitions (e.g., engine idle vs accelerating vs braking). Choose Bridge when you want to prevent a permanent binding between abstraction and implementation. |
| **Template Method** | Template Method defines the skeleton of an algorithm in a base class and lets subclasses override steps. Bridge uses composition to delegate implementation to a separate hierarchy. | Choose Template Method when the variation is within a single hierarchy and steps of an algorithm change (e.g., Car defines drive skeleton with start, accelerate, steer steps). Choose Bridge when the variation spans two orthogonal dimensions (car type and engine type). |
