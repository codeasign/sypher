The Bridge pattern in TypeScript benefits from formal interface definitions and compile-time type safety. The `Engine` interface declares the contract that all engines must fulfill, and the `Car` abstract class holds a typed `Engine` reference. Three concrete engine classes implement the `Engine` interface, and three car subclasses extend `Car`. TypeScript's type system catches mismatches at compile time — passing an object that lacks `start()`, `sound()`, or `fuelType()` produces a compilation error rather than a runtime failure. Generic constraints and union types can further refine how engines and cars compose without losing the independent hierarchy design.

## How the Implementation Works

TypeScript's `Engine` interface defines the exact shape every engine must satisfy. The `Car` abstract class accepts an `Engine` in its constructor and stores it as a protected property. Concrete car classes (Sedan, SUV, Truck) override `drive()` and call `this.engine.start()` and `this.engine.sound()`, relying on TypeScript's type system to guarantee those methods exist. The HybridEngine class takes two `Engine` objects in its constructor — a gas engine and an electric engine — and conditionally delegates to one or both based on battery level. Private fields with the `#` prefix keep engine parameters encapsulated at runtime.

### OOP Concepts Applied

- **Abstraction** — The `Engine` interface abstracts away engine internals. The `Car` abstract class defines the car contract. Neither side knows the other's concrete details.
- **Encapsulation** — `GasEngine` hides cylinder count and displacement. `ElectricEngine` hides battery capacity and motor power. `HybridEngine` hides the delegation logic between its two sub-engines.
- **Polymorphism** — `Car.drive()` calls `this.engine.start()` polymorphically. The same method name invokes different engine behaviors based on the actual Engine implementation injected.
- **Composition** — `Car` has an `Engine` reference (composition over inheritance). `HybridEngine` composes two `Engine` references, showing that implementations can also be composed.

### SOLID Principles Applied

- **Single Responsibility Principle** — Each class handles exactly one dimension of variation: GasEngine handles combustion logic, Sedan handles sedan driving behavior, and so on.
- **Open/Closed Principle** — Adding a `DieselEngine` requires one new class implementing `Engine`. Adding a `Convertible` requires one new class extending `Car`. No existing code changes.
- **Liskov Substitution Principle** — Any `Engine` can replace another in any `Car` subclass. The Car code calls only the `Engine` interface methods, which all implementations satisfy.
- **Interface Segregation Principle** — The `Engine` interface has exactly three methods. No engine is forced to implement irrelevant operations.
- **Dependency Inversion Principle** — `Car` depends on the `Engine` interface, not on concrete engine classes. Concrete engines depend on the same interface, not on Car subclasses.

```typescript
// Bridge Pattern — TypeScript — Cars Domain
// ============================================
// Participants: Car (Abstraction), Sedan/SUV/Truck (RefinedAbstractions),
// Engine (Implementor interface), GasEngine/ElectricEngine/HybridEngine
// (ConcreteImplementors)
// OOP: abstraction, encapsulation, polymorphism, composition
// SOLID: SRP, OCP, LSP, ISP, DIP
// ============================================

// ── Participant 1: Engine (Implementor Interface) ──
// OOP: Abstraction — defines the contract all engines must satisfy
// SOLID: ISP — minimal interface; DIP — Car depends on this abstraction
interface Engine {
  start(): void;
  sound(): string;
  fuelType(): string;
}
```

### Participant 2: GasEngine (ConcreteImplementor)

```typescript
// OOP: Encapsulation — internal parameters are private
// SOLID: SRP — implements only gas combustion logic
class GasEngine implements Engine {
  #cylinders: number;
  #displacement: number;

  constructor(cylinders: number, displacement: number) {
    this.#cylinders = cylinders;
    this.#displacement = displacement;
  }

  start(): void {
    console.log('Gas engine fires: *vroom vroom*');
  }

  sound(): string {
    return 'VROOOOM' + '.'.repeat(this.#cylinders);
  }

  fuelType(): string {
    return `Gasoline (${this.#displacement}L ${this.#cylinders}-cyl)`;
  }
}
```

### Participant 3: ElectricEngine (ConcreteImplementor)

```typescript
// OOP: Encapsulation — battery and motor specs are private
// SOLID: SRP — implements only electric motor logic
class ElectricEngine implements Engine {
  #batteryCapacity: number;
  #motorPower: number;

  constructor(batteryCapacity: number, motorPower: number) {
    this.#batteryCapacity = batteryCapacity;
    this.#motorPower = motorPower;
  }

  start(): void {
    console.log('Electric motor engages: *whirrr*');
  }

  sound(): string {
    return 'WHIIIIRRRR';
  }

  fuelType(): string {
    return `Electric (${this.#batteryCapacity}kWh, ${this.#motorPower}kW)`;
  }
}
```

### Participant 4: HybridEngine (ConcreteImplementor)

```typescript
// OOP: Composition — wraps both GasEngine and ElectricEngine
// SOLID: SRP — coordinates delegation without duplicating engine logic
class HybridEngine implements Engine {
  #gasEngine: Engine;
  #electricEngine: Engine;
  #batteryLevel: number;

  constructor(gasEngine: Engine, electricEngine: Engine, batteryLevel: number) {
    this.#gasEngine = gasEngine;
    this.#electricEngine = electricEngine;
    this.#batteryLevel = batteryLevel;
  }

  start(): void {
    this.#electricEngine.start();
    if (this.#batteryLevel < 20) {
      console.log('Battery low — gas engine kicks in.');
      this.#gasEngine.start();
    } else {
      console.log('Running on electric power — gas engine idle.');
    }
  }

  sound(): string {
    if (this.#batteryLevel >= 20) {
      return this.#electricEngine.sound();
    }
    return this.#gasEngine.sound() + ' then ' + this.#electricEngine.sound();
  }

  fuelType(): string {
    return `Hybrid (Gas + Electric, battery at ${this.#batteryLevel}%)`;
  }
}
```

### Participant 5: Car (Abstraction)

```typescript
// OOP: Abstraction — defines the car interface and holds Engine reference
// SOLID: DIP — depends on Engine interface, not concrete implementations
abstract class Car {
  protected engine: Engine;

  constructor(engine: Engine) {
    this.engine = engine;
  }

  abstract drive(): string;

  getSpecs(): string {
    return `Car with ${this.engine.fuelType()}`;
  }
}
```

### Participant 6: Sedan (RefinedAbstraction)

```typescript
// OOP: Polymorphism — overrides drive() with sedan-specific behavior
// SOLID: SRP — handles only sedan driving mechanics
class Sedan extends Car {
  #trunkCapacity: number;

  constructor(engine: Engine, trunkCapacity: number) {
    super(engine);
    this.#trunkCapacity = trunkCapacity;
  }

  drive(): string {
    this.engine.start();
    return `Sedan cruising smoothly on the highway. ${this.engine.sound()}`;
  }

  openTrunk(): string {
    return `Trunk opens (${this.#trunkCapacity} cu ft).`;
  }

  getSpecs(): string {
    return `Sedan with ${this.engine.fuelType()} | Trunk: ${this.#trunkCapacity} cu ft`;
  }
}
```

### Participant 7: SUV (RefinedAbstraction)

```typescript
// OOP: Polymorphism — overrides drive() with SUV-specific behavior
// SOLID: SRP — handles only SUV driving mechanics
class SUV extends Car {
  #groundClearance: number;
  #awd: boolean;

  constructor(engine: Engine, groundClearance: number, awd: boolean) {
    super(engine);
    this.#groundClearance = groundClearance;
    this.#awd = awd;
  }

  drive(): string {
    this.engine.start();
    return `SUV tackling off-road terrain. ${this.engine.sound()}`;
  }

  engage4WD(): string {
    if (this.#awd) {
      return 'All-wheel drive engaged. Traction optimized.';
    }
    return 'This SUV does not have AWD.';
  }

  getSpecs(): string {
    const drivetrain = this.#awd ? 'AWD' : '2WD';
    return `SUV with ${this.engine.fuelType()} | Ground: ${this.#groundClearance}" ${drivetrain}`;
  }
}
```

  console.log(gasSuv.drive());

}

### Language-Specific Notes

- TypeScript's `interface Engine` provides compile-time type safety. Passing an object that does not satisfy the `Engine` contract to a `Car` constructor produces a compiler error — a significant advantage over plain JavaScript's runtime duck typing.
- The `abstract class Car` with `abstract drive(): string` enforces that all concrete car types implement the `drive()` method. TypeScript's compiler flags any subclass that forgets to implement it.
- Private class fields with `#` (ES2022+) provide both compile-time privacy enforcement and runtime encapsulation. Unlike TypeScript's `private` keyword, `#` fields are truly private at runtime in JavaScript engines.
- The `HybridEngine` constructor accepts two `Engine` objects, demonstrating that the Bridge pattern's implementation side can also use composition. The `Engine` interface ensures type safety even when composing implementations.
- TypeScript's structural type system means any object with the correct shape satisfies the `Engine` interface, even without an explicit `implements Engine` clause. This enables interop with plain JavaScript objects.
- The driver creates five distinct car-engine configurations (Sedan+Gas, SUV+Electric, Truck+Hybrid, Sedan+Electric, SUV+Gas), demonstrating that both hierarchies vary independently — the core benefit of the Bridge pattern.
