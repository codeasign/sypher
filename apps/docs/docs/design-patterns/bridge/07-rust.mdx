The Bridge pattern in Rust uses a `trait` to define the Engine contract and a generic `Car` struct that holds a `Box<dyn Engine>` to achieve runtime polymorphism. The `Engine` trait declares `start()`, `sound()`, and `fuel_type()`. Three concrete engine structs (`GasEngine`, `ElectricEngine`, `HybridEngine`) implement the `Engine` trait. Three Car types (`Sedan`, `SUV`, `Truck`) are generic structs that each hold a `Box<dyn Engine>` and implement the common behavior through their own `drive()` and `get_specs()` methods. Rust's ownership model and trait system provide compile-time guarantees that the engine reference is always valid, while `dyn Engine` enables runtime dispatch so the same car code works with any engine type.

## How the Implementation Works

Rust's `Engine` trait defines the three required methods. Each engine struct implements `Engine` and stores its parameters as private fields. The `HybridEngine` holds two `Box<dyn Engine>` values — one for the gas sub-engine and one for the electric sub-engine — and delegates conditionally. Each Car struct (`Sedan`, `SUV`, `Truck`) holds an `engine: Box<dyn Engine>` field and implements its own `drive()` and `get_specs()` methods. Rust's `println!` macro and `format!` macro handle string formatting. The `dyn Engine` syntax enables runtime polymorphism: the concrete engine type is erased at compile time but dispatched correctly at runtime through a vtable pointer.

### OOP Concepts Applied (Rust's Equivalent)

- **Abstraction** — The `Engine` trait defines the abstraction. Car structs call trait methods without knowing the concrete engine type.
- **Encapsulation** — Each engine struct hides its fields (cylinders, battery_kwh) as non-public (private by default in Rust). Car structs also keep their fields private.
- **Polymorphism** — `dyn Engine` provides runtime polymorphism via vtable dispatch. Calling `engine.start()` dispatches to the correct implementation based on the concrete type behind the trait object.
- **Composition** — Each Car struct holds a `Box<dyn Engine>`. `HybridEngine` composes two `Box<dyn Engine>` values, demonstrating composition at the implementation level.

### SOLID Principles Applied

- **Single Responsibility Principle** — Each engine struct handles one engine type. Each car struct handles one car type.
- **Open/Closed Principle** — New engine types (`HydrogenEngine`) require one new struct implementing `Engine`. New car types (`Convertible`) require one new struct. No existing code changes.
- **Liskov Substitution Principle** — Any `Box<dyn Engine>` can be used in any Car struct. The Car code calls only trait methods, which all implementations provide.
- **Interface Segregation Principle** — The `Engine` trait has three methods. No implementation is forced to implement unused methods.
- **Dependency Inversion Principle** — Each Car struct depends on the `Engine` trait (abstraction), not on concrete engine types.

```rust
// Bridge Pattern — Rust — Cars Domain
// =====================================
// Participants: Car structs (Abstraction), Sedan/SUV/Truck (RefinedAbstractions),
// Engine trait (Implementor), GasEngine/ElectricEngine/HybridEngine
// (ConcreteImplementors)
// OOP equivalents: trait (abstraction), private fields (encapsulation),
// dyn Engine (polymorphism), Box<dyn Engine> (composition)
// =====================================

// ── Participant 1: Engine (Implementor Trait) ──
// OOP: Abstraction — defines the contract all engines must satisfy
// SOLID: ISP — minimal three-method interface
trait Engine {
    fn start(&self);
    fn sound(&self) -> String;
    fn fuel_type(&self) -> String;
}
```

### Participant 2: GasEngine (ConcreteImplementor)

```rust
// OOP: Encapsulation — fields are private by default
// SOLID: SRP — implements only gas combustion logic
struct GasEngine {
    cylinders: u32,
    displacement: f64,
}

impl GasEngine {
    fn new(cylinders: u32, displacement: f64) -> Self {
        GasEngine { cylinders, displacement }
    }
}

impl Engine for GasEngine {
    fn start(&self) {
        println!("Gas engine fires: *vroom vroom*");
    }

    fn sound(&self) -> String {
        format!("VROOOOM{}", ".".repeat(self.cylinders as usize))
    }

    fn fuel_type(&self) -> String {
        format!("Gasoline ({:.1}L {}-cyl)", self.displacement, self.cylinders)
    }
}
```

### Participant 4: HybridEngine (ConcreteImplementor)

```rust
// OOP: Composition — wraps both a gas and electric engine via Box<dyn Engine>
// SOLID: SRP — coordinates delegation without duplicating engine logic
struct HybridEngine {
    gas_engine: Box<dyn Engine>,
    electric_engine: Box<dyn Engine>,
    battery_level: u8,
}

impl HybridEngine {
    fn new(gas_engine: Box<dyn Engine>, electric_engine: Box<dyn Engine>,
           battery_level: u8) -> Self {
        HybridEngine { gas_engine, electric_engine, battery_level }
    }
}

impl Engine for HybridEngine {
    fn start(&self) {
        self.electric_engine.start();
        if self.battery_level < 20 {
            println!("Battery low — gas engine kicks in.");
            self.gas_engine.start();
        } else {
            println!("Running on electric power — gas engine idle.");
        }
    }

    fn sound(&self) -> String {
        if self.battery_level >= 20 {
            return self.electric_engine.sound();
        }
        format!("{} then {}", self.gas_engine.sound(), self.electric_engine.sound())
    }

    fn fuel_type(&self) -> String {
        format!("Hybrid (Gas + Electric, battery at {}%)", self.battery_level)
    }
}
```

### Participant 5: Sedan (RefinedAbstraction)

```rust
// OOP: Composition — holds Box<dyn Engine>
// SOLID: SRP — handles only sedan driving mechanics
struct Sedan {
    engine: Box<dyn Engine>,
    trunk_capacity: f64,
}

impl Sedan {
    fn new(engine: Box<dyn Engine>, trunk_capacity: f64) -> Self {
        Sedan { engine, trunk_capacity }
    }

    fn drive(&self) -> String {
        self.engine.start();
        format!("Sedan cruising smoothly on the highway. {}", self.engine.sound())
    }

    fn open_trunk(&self) -> String {
        format!("Trunk opens ({:.1} cu ft).", self.trunk_capacity)
    }

    fn get_specs(&self) -> String {
        format!("Sedan with {} | Trunk: {:.1} cu ft",
                self.engine.fuel_type(), self.trunk_capacity)
    }
}
```

### Participant 6: SUV (RefinedAbstraction)

```rust
// OOP: Composition — holds Box<dyn Engine>
// SOLID: SRP — handles only SUV driving mechanics
struct SUV {
    engine: Box<dyn Engine>,
    ground_clearance: f64,
    awd: bool,
}

impl SUV {
    fn new(engine: Box<dyn Engine>, ground_clearance: f64, awd: bool) -> Self {
        SUV { engine, ground_clearance, awd }
    }

    fn drive(&self) -> String {
        self.engine.start();
        format!("SUV tackling off-road terrain. {}", self.engine.sound())
    }

    fn engage_4wd(&self) -> String {
        if self.awd {
            "All-wheel drive engaged. Traction optimized.".to_string()
        } else {
            "This SUV does not have AWD.".to_string()
        }
    }

    fn get_specs(&self) -> String {
        let drivetrain = if self.awd { "AWD" } else { "2WD" };
        format!("SUV with {} | Ground: {:.1}\" {}",
                self.engine.fuel_type(), self.ground_clearance, drivetrain)
    }
}
```

### Language-Specific Notes

- Rust does not have classes or inheritance. The Bridge pattern is implemented using trait objects (`Box<dyn Engine>`) and composition. Each Car struct holds its engine as a trait object, providing runtime polymorphism through vtable dispatch.
- Rust's `trait Engine` defines the interface. Any struct that implements all three trait methods is automatically an `Engine`. The compiler checks trait implementation completeness at compile time.
- `Box<dyn Engine>` is a heap-allocated trait object. The `Box` ensures the engine value has a known size (a pointer) and the `dyn` keyword enables runtime method dispatch through a hidden vtable pointer.
- Rust's `println!` and `format!` macros use compile-time format string checking. Format specifiers like `{:.1}` (one decimal place) and `{}` (Display trait) are validated at compile time, preventing format string bugs.
- All fields in Rust structs are private by default (no `pub` keyword used here). This provides encapsulation without requiring a separate access modifier — external code cannot read or write engine parameters directly.
- The `HybridEngine` demonstrates nested trait objects: `gas_engine: Box<dyn Engine>` and `electric_engine: Box<dyn Engine>`. Multiple levels of boxing are needed because trait objects are unsized — the `Box` provides the sized pointer that the struct needs.
