The State pattern allows an object to alter its behavior when its internal state changes. This Python implementation models a stock market order lifecycle, where an Order transitions through states such as New, Submitted, PartiallyFilled, Filled, Cancelled, and Rejected. Each concrete state encapsulates its own validation rules and transition logic as a standalone class, keeping the Order context clean and maintainable while enforcing the legal state graph of a real trading system.

## How the Implementation Works

### Class Diagram in Pseudocode

```text
from abc import ABC, abstractmethod

class OrderState(ABC):
  @abstractmethod
  handle_submit(order)          — raise error by default
  @abstractmethod
  handle_fill(order, qty, price)  — raise error by default
  @abstractmethod
  handle_cancel(order)          — raise error by default
  @abstractmethod
  handle_reject(order, reason)   — raise error by default
  @abstractmethod
  get_status_label()            — return state name string
  @abstractmethod
  get_allowed_operations()      — return list of operation names

class NewState(OrderState):
  handle_submit()       — transition to SubmittedState
  handle_cancel()       — transition to CancelledState
  get_status_label()    — "New"
  get_allowed_ops()     — ["submit", "cancel"]

class SubmittedState(OrderState):
  handle_fill(qty, prc) — transition to PartiallyFilled or Filled
  handle_cancel()       — transition to CancelledState
  handle_reject(reason) — transition to RejectedState
  get_status_label()    — "Submitted"
  get_allowed_ops()     — ["fill", "cancel"]

class PartiallyFilledState(OrderState):
  handle_fill(qty, prc) — transition to Filled if fully filled
  handle_cancel()       — transition to CancelledState
  get_status_label()    — "PartiallyFilled"
  get_allowed_ops()     — ["fill", "cancel"]

class FilledState(OrderState):       # terminal
  get_status_label()    — "Filled"
  get_allowed_ops()     — []

class CancelledState(OrderState):    # terminal
  get_status_label()    — "Cancelled"
  get_allowed_ops()     — []

class RejectedState(OrderState):     # terminal
  get_status_label()    — "Rejected"
  get_allowed_ops()     — []

@dataclass
class Order:
  symbol: str
  quantity: int
  filled_quantity: int = 0
  price: float = 0.0
  state: OrderState = field(default_factory=NewState)

  submit()             — delegates to state
  fill(qty, price)     — delegates to state
  cancel()             — delegates to state
  reject(reason)       — delegates to state
  get_status()         — state.get_status_label()
  set_state(s)         — update internal state
  add_fill(qty, price) — increment filled quantity
  is_fully_filled()    — filled_quantity >= quantity
```

### OOP Concepts Applied

- **Abstraction** — The abstract `OrderState` class defines the contract via `@abstractmethod` decorators; callers interact through the interface without knowing the concrete state class name.
- **Encapsulation** — Each concrete state owns its transition rules and validation logic. The `Order` dataclass encapsulates all order data with a private-like convention (no direct access to internals from outside the class).
- **Inheritance** — Concrete states inherit from the abstract `OrderState` base, overriding only the methods that represent valid transitions for that state. The base class provides default error-returning implementations for all operations.
- **Polymorphism** — The `Order` delegates every public method to `self.state` typed as `OrderState`; each concrete state provides its own behavior for the same set of method signatures.

### SOLID Principles Applied

- **Single Responsibility** — Each state class handles exactly the subset of lifecycle operations valid for its phase. `SubmittedState` manages fills, cancellations, and rejections; `NewState` only handles submission and pre-submission cancellation.
- **Open/Closed** — New states (e.g., `ExpiredState`, `StoppedState`) can be added by subclassing `OrderState` without modifying any existing state class or the `Order` context.
- **Liskov Substitution** — Any `OrderState` subclass can replace another in the `Order` context without breaking correctness because each subclass adheres to the same abstract method signatures.
- **Interface Segregation** — The `OrderState` abstract class declares focused, state-specific methods (`handle_submit`, `handle_fill`, `handle_cancel`, `handle_reject`) rather than a single monolithic method that branches internally on state type.
- **Dependency Inversion** — `Order` depends on the abstract `OrderState` base class, never on concrete state implementations. States are discovered and swapped at runtime.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field


# ── OrderState (Abstract Base Class) ─────────────────────────

class OrderState(ABC):
    """Abstract interface for all order lifecycle states."""

    def handle_submit(self, order: "Order") -> str:
        status = self.get_status_label().lower()
        return f"ERROR: Cannot submit — order is already {status}"

    def handle_fill(self, order: "Order", quantity: int, price: float) -> str:
        status = self.get_status_label().lower()
        return f"ERROR: Cannot fill — order is already {status}"

    def handle_cancel(self, order: "Order") -> str:
        status = self.get_status_label().lower()
        return f"ERROR: Cannot cancel — order is already {status}"

    def handle_reject(self, order: "Order", reason: str) -> str:
        status = self.get_status_label().lower()
        return f"ERROR: Cannot reject — order is already {status}"

    @abstractmethod
    def get_status_label(self) -> str:
        ...

    @abstractmethod
    def get_allowed_operations(self) -> list[str]:
        ...
```

### SubmittedState

```python
class SubmittedState(OrderState):
    def handle_fill(self, order: "Order", quantity: int, price: float) -> str:
        symbol = order.symbol
        remaining = order.remaining
        if quantity >= remaining:
            order.add_fill(remaining, price)
            order.set_state(FilledState())
            return f"{symbol}: Filled {order.filled_quantity} shares at ${price:.2f}"
        order.add_fill(quantity, price)
        order.set_state(PartiallyFilledState())
        return f"{symbol}: Partially filled {order.filled_quantity}/{order.quantity} shares at ${price:.2f}"

    def handle_cancel(self, order: "Order") -> str:
        symbol = order.symbol
        order.set_state(CancelledState())
        return f"{symbol}: Submitted -> {order.get_status()}"

    def handle_reject(self, order: "Order", reason: str) -> str:
        symbol = order.symbol
        order.set_state(RejectedState())
        return f"{symbol}: Rejected — {reason}"

    def get_status_label(self) -> str:
        return "Submitted"

    def get_allowed_operations(self) -> list[str]:
        return ["fill", "cancel"]
```

### Order (Context)

```python
# ── Order (Context) ────────────────────────────────────────────

@dataclass
class Order:
    symbol: str
    quantity: int
    filled_quantity: int = 0
    price: float = 0.0
    state: OrderState = field(default_factory=NewState)

    @property
    def remaining(self) -> int:
        return self.quantity - self.filled_quantity

    def set_state(self, state: OrderState) -> None:
        self.state = state

    def add_fill(self, qty: int, price: float) -> None:
        self.filled_quantity += qty
        self.price = price

    def get_status(self) -> str:
        return self.state.get_status_label()

    def get_allowed_operations(self) -> list[str]:
        return self.state.get_allowed_operations()

    def is_fully_filled(self) -> bool:
        return self.filled_quantity >= self.quantity

    def submit(self) -> str:
        return self.state.handle_submit(self)

    def fill(self, quantity: int, price: float) -> str:
        return self.state.handle_fill(self, quantity, price)

    def cancel(self) -> str:
        return self.state.handle_cancel(self)

    def reject(self, reason: str) -> str:
        return self.state.handle_reject(self, reason)
```

### Main / Driver

```python
# ── Driver ────────────────────────────────────────────────────

if __name__ == "__main__":
    print("--- Order 1: AAPL 100 @ $175.50 ---")
    order1 = Order(symbol="AAPL", quantity=100, price=175.50)
    print(order1.submit())
    print(order1.fill(100, 175.50))
    print(order1.cancel())

    print()
    print("--- Order 2: GOOGL 50 @ $140.25 ---")
    order2 = Order(symbol="GOOGL", quantity=50, price=140.25)
    print(order2.submit())
    print(order2.fill(20, 140.25))
    print(order2.cancel())

    print()
    print("--- Order 3: MSFT 75 @ $378.80 ---")
    order3 = Order(symbol="MSFT", quantity=75, price=378.80)
    print(order3.submit())
    print(order3.reject("Insufficient funds"))

    print()
    print("=== Final States ===")
    print(f"AAPL: {order1.get_status()}")
    print(f"GOOGL: {order2.get_status()}")
    print(f"MSFT: {order3.get_status()}")
```

## Language-Specific Notes

- Python's `abc.ABC` and `@abstractmethod` provide a first-class mechanism for defining the `OrderState` interface. The `@abstractmethod` decorator on `get_status_label` and `get_allowed_operations` forces every concrete subclass to implement them, catching missing methods at instantiation time.
- The `@dataclass` decorator eliminates boilerplate for the `Order` context class — `__init__`, `__repr__`, and `__eq__` are generated automatically from field declarations, keeping the code concise and focused on behavior.
- The `field(default_factory=NewState)` pattern in the `Order` dataclass initializes the state to a fresh `NewState` instance for every order object without sharing a single state reference across instances.
- Type hints (`list[str]`, `-> str`, `"Order"` forward reference) make the state transition contracts self-documenting and enable IDE type checking and autocompletion.
- Python's `f"{price:.2f}"` format specifier produces consistent two-decimal-place output for dollar amounts, matching the expected output format without manual rounding or string manipulation.
- The `@property` decorator on `remaining` provides a computed property that stays in sync with `filled_quantity` changes automatically, eliminating the risk of a stale cached value when the fill quantity is updated.
