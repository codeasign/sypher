The Rust 2021 edition implementation of the State pattern models a stock market order lifecycle. Each order transitions through well-defined states — New, Submitted, PartiallyFilled, Filled, Cancelled, and Rejected — with the `Order` struct delegating state-specific behavior to the current `OrderState` trait object. Rust's ownership model requires an ownership-based approach to state transitions, where `self: Box<Self>` consumes the old state and produces a new one, making the pattern both memory-safe and free of garbage collection.

## How the Implementation Works

### Class Diagram (Pseudocode)

```text
trait OrderState {
    + handle_submit(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState>
    + handle_fill(self: Box<Self>, order: &mut Order, i32, f64) -> Box<dyn OrderState>
    + handle_cancel(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState>
    + handle_reject(self: Box<Self>, order: &mut Order, &str) -> Box<dyn OrderState>
    + get_status_label(&self) -> &str
    + get_allowed_operations(&self) -> Vec<&str>
}

struct NewState;             impl OrderState { /* submit -> SubmittedState, cancel -> CancelledState */ }
struct SubmittedState;       impl OrderState { /* fill -> FilledState or PartiallyFilledState, cancel -> CancelledState, reject -> RejectedState */ }
struct PartiallyFilledState; impl OrderState { /* fill -> FilledState, cancel -> CancelledState */ }
struct FilledState;          impl OrderState { /* terminal -- all operations fail, return self */ }
struct CancelledState;       impl OrderState { /* terminal -- all operations fail, return self */ }
struct RejectedState;        impl OrderState { /* terminal -- all operations fail, return self */ }

struct Order {                     // Context
    - symbol: String
    - quantity: i32
    - filled_quantity: i32
    - price: f64
    - state: Box<dyn OrderState>
    + submit(&mut self)              // mem::replace(&mut self.state, ..).handle_submit(self)
    + fill(&mut self, qty, price)    // mem::replace(&mut self.state, ..).handle_fill(self, ..)
    + cancel(&mut self)              // mem::replace(&mut self.state, ..).handle_cancel(self)
    + reject(&mut self, reason)      // mem::replace(&mut self.state, ..).handle_reject(self, ..)
    + get_status(&self) -> &str      // delegates to state.get_status_label()
    + set_state(Box<dyn OrderState>) // replaces current state
    + add_fill(qty, price)           // accumulates filled quantity
    + is_fully_filled() -> bool      // checks if filled_quantity >= quantity
}
```

### OOP Concepts Applied

1. **Abstraction** — The `OrderState` trait defines the contract for all lifecycle states. The `Order` holds a `Box<dyn OrderState>` and never references concrete state structs directly — it calls methods on the trait abstraction.

2. **Encapsulation** — Each state struct encapsulates its own transition rules, validation logic, and error messages. The `Order` struct's fields are private, and `set_state()` is the only mechanism for changing the current state.

3. **Polymorphism** — The `Order` calls methods on `Box<dyn OrderState>`, and Rust's trait object dispatch routes the call to the correct concrete implementation at runtime — without any `match` statements on status strings.

4. **Ownership-based state management** — Unlike garbage-collected languages, Rust uses `self: Box<Self>` in the trait signature to consume the old state during transitions. The `std::mem::replace` pattern safely swaps out the current state, passes ownership to the handler, and stores the new state returned by the handler.

### SOLID Principles Applied

5. **SRP** — `Order` manages only order data and state delegation. Each state struct handles only the behavior for one specific lifecycle phase. No struct mixes responsibilities.

6. **OCP** — Adding a new state (e.g., `PendingReviewState`) requires creating one new struct implementing `OrderState`. No existing state structs need modification unless they transition to the new state.

7. **LSP** — Every concrete state struct is a valid substitute for `dyn OrderState`. The `Order` calls the same trait methods regardless of which state is active, and every state provides correct behavior for its phase.

8. **ISP** — The `OrderState` trait defines exactly the methods a state needs: four lifecycle handlers, a label getter, and an operations query. No struct implements irrelevant methods.

9. **DIP** — `Order` depends on the `dyn OrderState` trait, not on concrete state structs. States are injected via `set_state()` and are fully interchangeable without changing the `Order` struct.

## Implementation

```rust
trait OrderState {
    fn handle_submit(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState>;
    fn handle_fill(self: Box<Self>, order: &mut Order, quantity: i32, price: f64) -> Box<dyn OrderState>;
    fn handle_cancel(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState>;
    fn handle_reject(self: Box<Self>, order: &mut Order, reason: &str) -> Box<dyn OrderState>;
    fn get_status_label(&self) -> &str;
    fn get_allowed_operations(&self) -> Vec<&str>;
}

struct NewState;
struct SubmittedState;
struct PartiallyFilledState;
struct FilledState;
struct CancelledState;
struct RejectedState;
```

### NewState

```rust
impl OrderState for NewState {
    fn handle_submit(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState> {
        println!("{}: New -> Submitted", order.symbol);
        Box::new(SubmittedState)
    }

    fn handle_fill(self: Box<Self>, order: &mut Order, _quantity: i32, _price: f64) -> Box<dyn OrderState> {
        println!("ERROR: Cannot fill — order must be submitted first");
        self
    }

    fn handle_cancel(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState> {
        println!("{}: New -> Cancelled", order.symbol);
        Box::new(CancelledState)
    }

    fn handle_reject(self: Box<Self>, order: &mut Order, _reason: &str) -> Box<dyn OrderState> {
        println!("ERROR: Cannot reject — order is not yet submitted");
        self
    }

    fn get_status_label(&self) -> &str { "New" }
    fn get_allowed_operations(&self) -> Vec<&str> { vec!["submit", "cancel"] }
}
```

### SubmittedState

```rust
impl OrderState for SubmittedState {
    fn handle_submit(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState> {
        println!("ERROR: Cannot submit — order is already submitted");
        self
    }

    fn handle_fill(self: Box<Self>, order: &mut Order, quantity: i32, price: f64) -> Box<dyn OrderState> {
        order.add_fill(quantity, price);
        if order.is_fully_filled() {
            println!("{}: Filled {} shares at ${:.2}", order.symbol, order.filled_quantity, price);
            Box::new(FilledState)
        } else {
            println!("{}: Partially filled {}/{} shares at ${:.2}", order.symbol, order.filled_quantity, order.quantity, price);
            Box::new(PartiallyFilledState)
        }
    }

    fn handle_cancel(self: Box<Self>, order: &mut Order) -> Box<dyn OrderState> {
        println!("{}: Submitted -> Cancelled", order.symbol);
        Box::new(CancelledState)
    }

    fn handle_reject(self: Box<Self>, order: &mut Order, reason: &str) -> Box<dyn OrderState> {
        println!("{}: Rejected — {}", order.symbol, reason);
        Box::new(RejectedState)
    }

    fn get_status_label(&self) -> &str { "Submitted" }
    fn get_allowed_operations(&self) -> Vec<&str> { vec!["fill", "cancel", "reject"] }
}
```

### Order (Context)

```rust
struct Order {
    symbol: String,
    quantity: i32,
    filled_quantity: i32,
    price: f64,
    state: Box<dyn OrderState>,
}

impl Order {
    fn new(symbol: &str, quantity: i32, price: f64) -> Self {
        Order {
            symbol: symbol.to_string(),
            quantity,
            filled_quantity: 0,
            price,
            state: Box::new(NewState),
        }
    }

    fn set_state(&mut self, new_state: Box<dyn OrderState>) {
        self.state = new_state;
    }

    fn submit(&mut self) {
        let state = std::mem::replace(&mut self.state, Box::new(NewState));
        self.state = state.handle_submit(self);
    }

    fn fill(&mut self, quantity: i32, price: f64) {
        let state = std::mem::replace(&mut self.state, Box::new(NewState));
        self.state = state.handle_fill(self, quantity, price);
    }

    fn cancel(&mut self) {
        let state = std::mem::replace(&mut self.state, Box::new(NewState));
        self.state = state.handle_cancel(self);
    }

    fn reject(&mut self, reason: &str) {
        let state = std::mem::replace(&mut self.state, Box::new(NewState));
        self.state = state.handle_reject(self, reason);
    }

    fn get_status(&self) -> &str {
        self.state.get_status_label()
    }

    fn get_allowed_operations(&self) -> Vec<&str> {
        self.state.get_allowed_operations()
    }

    fn add_fill(&mut self, quantity: i32, _price: f64) {
        self.filled_quantity += quantity;
    }

    fn is_fully_filled(&self) -> bool {
        self.filled_quantity >= self.quantity
    }
}
```

### Main / Driver

```rust
fn main() {
    println!("--- Order 1: AAPL 100 @ $175.50 ---");
    let mut order1 = Order::new("AAPL", 100, 175.50);
    order1.submit();
    order1.fill(100, 175.50);
    order1.cancel();

    println!("\n--- Order 2: GOOGL 50 @ $140.25 ---");
    let mut order2 = Order::new("GOOGL", 50, 140.25);
    order2.submit();
    order2.fill(20, 140.25);
    order2.cancel();

    println!("\n--- Order 3: MSFT 75 @ $378.80 ---");
    let mut order3 = Order::new("MSFT", 75, 378.80);
    order3.submit();
    order3.reject("Insufficient funds");

    println!("\n=== Final States ===");
    println!("AAPL: {}", order1.get_status());
    println!("GOOGL: {}", order2.get_status());
    println!("MSFT: {}", order3.get_status());
}
```

### Language-Specific Notes

- **`self: Box<Self>` ownership pattern** — The `OrderState` trait uses `self: Box<Self>` instead of `&self` for all transition methods. This means calling `handle_submit` consumes the current state, preventing the same state object from being used after a transition. The handler returns a new `Box<dyn OrderState>`, which the `Order` stores as its new state. This is the idiomatic Rust approach to the State pattern and avoids any borrow-checker conflicts.

- **`std::mem::replace` for safe swapping** — The `submit()`, `fill()`, `cancel()`, and `reject()` methods on `Order` use `std::mem::replace(&mut self.state, Box::new(NewState))` to temporarily swap in a dummy state, take ownership of the real state, call the handler, and store the returned state. This pattern satisfies the borrow checker by ensuring `self.state` is never borrowed when the handler receives `&mut self`.

- **Zero-sized state structs** — All six state structs (`NewState`, `SubmittedState`, etc.) are zero-sized types with no fields. They exist only as vtable pointers inside the `Box<dyn OrderState>`. This means `Box::new(NewState)` allocates only the heap memory for the trait object's data pointer and vtable, not for any state-specific data.

- **Terminal states return `self`** — `FilledState`, `CancelledState`, and `RejectedState` are terminal states. When an invalid operation is attempted (e.g., cancel on a filled order), the handler prints an error and returns `self` — the same state object, unchanged. This ensures the `Order`'s state field is always valid and never left in a `None` state.

- **`Vec<&str>` for allowed operations** — The `get_allowed_operations()` method returns `Vec<&str>`, borrowing string literals from the program's static data. Terminal states return `vec![]` (empty vector), while active states return their valid operations. This avoids allocation of owned `String` values for read-only operation names.

- **No `match` or `enum` for state dispatch** — While Rust's `enum` and `match` are a natural fit for state machines, this implementation uses trait objects (`Box<dyn OrderState>`) to demonstrate the GoF State pattern's polymorphic approach. An `enum`-based type-state pattern would be more idiomatic for simple state machines, but trait objects better illustrate the pattern's intent of delegating behavior to interchangeable state objects.
