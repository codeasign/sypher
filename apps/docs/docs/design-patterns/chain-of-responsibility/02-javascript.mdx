The **Chain of Responsibility** pattern allows a request to travel through a sequence of handlers until one of them processes it. In the stock market domain, trade orders must be approved based on their total dollar value. Different trading personnel have distinct approval limits, forming a natural chain of responsibility.

## How the Implementation Works

### Class Diagram (Pseudocode)

```text
Participants:
  - OrderApprover (Abstract Handler)
    - #nextHandler: OrderApprover | null
    - setNext(handler): OrderApprover
    - approveOrder(order): string
  - TradeOrder (Request)
    - #ticker, #quantity, #pricePerShare, #orderType
    - get totalValue(): number
  - JuniorTrader extends OrderApprover  (limit = $1,000)
  - SeniorTrader extends OrderApprover  (limit = $10,000)
  - PortfolioManager extends OrderApprover  (limit = $100,000)
  - RiskDirector extends OrderApprover  (no limit, final handler)

OOP Roles:
  Abstract Handler       → OrderApprover base class
  Concrete Handlers      → JuniorTrader, SeniorTrader, PortfolioManager, RiskDirector
  Request Object         → TradeOrder
  Client / Chain Builder → main driver

SOLID Principles:
  Single Responsibility  — Each handler knows only its own approval limit
  Open/Closed            — New approval roles can be added without modifying existing handlers
  Liskov Substitution    — Any handler subclass can replace OrderApprover without breaking the chain
  Interface Segregation  — The handler interface is minimal (setNext + approveOrder)
  Dependency Inversion   — Handlers depend on the abstract OrderApprover, not concrete implementations
```

### OOP Concepts Applied

1. **Inheritance** — Concrete handlers extend the abstract `OrderApprover` class, reusing the chain-building logic in `setNext()`.
2. **Polymorphism** — The client calls `approveOrder()` on the first handler without knowing (or caring) which concrete type eventually processes the request.
3. **Encapsulation** — Each handler's approval limit and reference to the next handler are private fields, hidden from outside access.
4. **Abstraction** — The `OrderApprover` base class defines the contract (`approveOrder` and `setNext`) while leaving the actual approval logic to subclasses.

### SOLID Principles Applied

5. **Single Responsibility Principle** — Each handler class has exactly one job: deciding whether an order falls within its approval limit.
6. **Open/Closed Principle** — Adding a new approval tier (e.g., `ComplianceOfficer`) requires only a new handler subclass and a chain link — no existing handler code changes.
7. **Liskov Substitution Principle** — Any handler can replace another in the chain as long as it extends `OrderApprover`; the chain logic remains correct.
8. **Interface Segregation Principle** — The handler interface consists of just two methods: `setNext()` to build the chain and `approveOrder()` to handle a request.
9. **Dependency Inversion Principle** — High-level chain logic depends on the abstract `OrderApprover` class, not on concrete trader or manager implementations.

### Complete JavaScript Implementation

### TradeOrder (Request)
```javascript
'use strict';

class TradeOrder {
  #ticker;
  #quantity;
  #pricePerShare;
  #orderType;

  constructor(ticker, quantity, pricePerShare, orderType) {
    this.#ticker = ticker;
    this.#quantity = quantity;
    this.#pricePerShare = pricePerShare;
    this.#orderType = orderType;
    Object.freeze(this);
  }

  get ticker() {
    return this.#ticker;
  }

  get quantity() {
    return this.#quantity;
  }

  get pricePerShare() {
    return this.#pricePerShare;
  }

  get orderType() {
    return this.#orderType;
  }

  get totalValue() {
    return this.#quantity * this.#pricePerShare;
  }
}
```

### OrderApprover (Abstract Handler)
```javascript
class OrderApprover {
  #nextHandler = null;

  setNext(handler) {
    this.#nextHandler = handler;
    return handler;
  }

  approveOrder(order) {
    throw new Error('Subclass must implement approveOrder()');
  }

  _next(order) {
    if (this.#nextHandler) {
      return this.#nextHandler.approveOrder(order);
    }
    return `REJECTED: ${order.ticker} — no approver available for $${order.totalValue}`;
  }

  _getNextHandler() {
    return this.#nextHandler;
  }
}
```

### JuniorTrader (Concrete Handler)
```javascript
class JuniorTrader extends OrderApprover {
  #limit = 1000;
  #role = 'JuniorTrader';

  approveOrder(order) {
    const value = order.totalValue;
    if (value <= this.#limit) {
      return `APPROVED by ${this.#role}: ${order.orderType.toUpperCase()} ${order.quantity} shares of ${order.ticker} at $${order.pricePerShare} (total: $${value})`;
    }
    return this._next(order);
  }
}
```

### Main / Driver
```javascript
function main() {
  const junior = new JuniorTrader();
  const senior = new SeniorTrader();
  const pm = new PortfolioManager();
  const director = new RiskDirector();

  // Build the chain: JuniorTrader -> SeniorTrader -> PortfolioManager -> RiskDirector
  junior.setNext(senior).setNext(pm).setNext(director);

  const orders = [
    new TradeOrder('AAPL', 10, 150, 'buy'),
    new TradeOrder('GOOGL', 50, 175, 'buy'),
    new TradeOrder('MSFT', 1000, 200, 'sell'),
  ];

  for (const order of orders) {
    console.log(`--- Processing ${order.ticker} ---`);
    const result = junior.approveOrder(order);
    console.log(result);
    console.log('');
  }
}

main();
```

### Expected Output

```bash
--- Processing AAPL ---
APPROVED by SeniorTrader: BUY 10 shares of AAPL at $150 (total: $1500)

--- Processing GOOGL ---
APPROVED by SeniorTrader: BUY 50 shares of GOOGL at $175 (total: $8750)

--- Processing MSFT ---
APPROVED by RiskDirector: SELL 1000 shares of MSFT at $200 (total: $200000)
```

Note: The AAPL order has a total value of $1,500 (10 x $150), which exceeds the JuniorTrader limit of $1,000. It is approved by SeniorTrader instead. The GOOGL order ($8,750) exceeds the JuniorTrader limit but is within the SeniorTrader limit of $10,000, so SeniorTrader approves it. The MSFT order ($200,000) exceeds all limits up to RiskDirector, the final authority.

### Language-Specific Notes

- **Private fields (`#`)** enforce true encapsulation at the engine level — `#nextHandler` and `#limit` are inaccessible from outside the class, eliminating naming convention reliance on underscore prefixes.
- **`Object.freeze(this)` in the constructor** makes `TradeOrder` instances immutable after creation, which is critical for audit trails in financial systems.
- **Fluent chain-building** is supported by returning `handler` from `setNext()`, enabling the concise `junior.setNext(senior).setNext(pm).setNext(director)` call.
- **No interfaces in JavaScript** — the abstract handler uses a base class with a throwing method in the parent to enforce the contract, checked at runtime rather than compile time.
- **Dynamic dispatch** works naturally: calling `approveOrder()` on the first handler automatically delegates through the chain via `this._next()` until a handler claims the request.
- **The helper `_getNextHandler()`** is exposed only for testing or inspection; production code exclusively uses `approveOrder()` to traverse the chain.
