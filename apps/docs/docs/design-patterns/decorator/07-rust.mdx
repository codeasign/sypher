## Implementation

Below is a complete Rust implementation of the Decorator pattern using a car dealership domain. Rust does not have classical inheritance or abstract classes, so the pattern is implemented using a trait (`Car`) for the component interface, a struct (`BaseCar`) for the concrete component, and wrapper structs for each decorator. The decorators hold a `Box<dyn Car>` to the wrapped car, enabling dynamic dispatch and runtime composition. This approach follows Rust's composition-over-inheritance philosophy while faithfully implementing the Decorator pattern.

### How the Implementation Works

#### Class Diagram (Pseudocode)

#### OOP Concepts Applied

1. **Abstraction** — The `Car` trait defines the contract. Client code accepts `&dyn Car` and is decoupled from concrete implementations. Each decorator implements `Car` for its wrapper struct.

2. **Polymorphism** — Rust achieves polymorphism through trait objects (`Box<dyn Car>`). A `Vec<Box<dyn Car>>` can hold `BaseCar` and any decorated variant. Method calls are dispatched dynamically through the vtable.

3. **Composition over Inheritance** — Rust has no class inheritance. Decorators are wrapper structs that hold a `Box<dyn Car>` and delegate to it. This is composition by necessity, which aligns perfectly with the Decorator pattern's design.

#### SOLID Principles Applied

4. **SRP** — `BaseCar` holds only the base model identity. Each wrapper struct handles exactly one feature. The driver's `main` function handles configuration and output.

5. **OCP** — A new feature is added by defining a new wrapper struct that implements `Car`. No existing code is modified. The trait is open for extension (new implementors) and closed for modification.

6. **LSP** — The `Car` trait defines the contract. Every wrapper struct implements `Car` correctly. A `Box<dyn Car>` pointing to a decorated car is substitutable for one pointing to a `BaseCar`.

7. **ISP** — The `Car` trait has exactly two methods. No client is forced to depend on methods it does not use.

8. **DIP** — High-level code depends on the `Car` trait. `BaseCar` and wrapper structs implement `Car`. The driver constructs concrete instances but operates on `&dyn Car` or `Box<dyn Car>` references.

### Component trait

```rust
trait Car {
    fn description(&self) -> String;
    fn cost(&self) -> f64;
}
```

### ConcreteComponent

```rust
struct BaseCar {
    model: String,
    base_cost: f64,
}

impl BaseCar {
    fn new(model: &str, base_cost: f64) -> Self {
        BaseCar {
            model: model.to_string(),
            base_cost,
        }
    }
}

impl Car for BaseCar {
    fn description(&self) -> String {
        self.model.clone()
    }

    fn cost(&self) -> f64 {
        self.base_cost
    }
}
```

### ConcreteDecorator 1: LeatherSeats

```rust
struct LeatherSeats {

impl LeatherSeats {

impl Car for LeatherSeats {
    fn description(&self) -> String {
        format!("{} + Leather Seats", self.car.description())
```

### Helper: format currency

```rust

}
```

### Helper: display a config line

```rust

    println!(
        "  {:<18} {:<60} {}",
        label,
        car.description(),
        format_currency(car.cost())
```

```rust
fn main() {
    println!("=== Rust Car Dealership Configuration System ===");
    println!();

    // Build configurations

    let with_leather: Box<dyn Car> = Box::new(LeatherSeats::new(base_camry.clone_box()));

    let base_camry = Box::new(BaseCar::new("Toyota Camry", 28000.00));
    let with_sunroof: Box<dyn Car> = Box::new(Sunroof::new(base_camry.clone_box()));
    let base_camry = Box::new(BaseCar::new("Toyota Camry", 28000.00));
    let with_leather_sunroof: Box<dyn Car> = Box::new(Sunroof::new(Box::new(LeatherSeats::new(base_camry.clone_box()))));
    let base_camry = Box::new(BaseCar::new("Toyota Camry", 28000.00));
    let eco_luxury: Box<dyn Car> = Box::new(NavigationSystem::new(Box::new(LeatherSeats::new(base_camry.clone_box()))));
    let base_camry = Box::new(BaseCar::new("Toyota Camry", 28000.00));
    let sport_only: Box<dyn Car> = Box::new(SportPackage::new(base_camry.clone_box()));
    let base_camry = Box::new(BaseCar::new("Toyota Camry", 28000.00));
    let fully_loaded: Box<dyn Car> = Box::new(SportPackage::new(
        Box::new(NavigationSystem::new(
            Box::new(Sunroof::new(
                Box::new(LeatherSeats::new(base_camry.clone_box())),
            )),
        )),
    ));

    let configs: Vec<(&str, Box<dyn Car>)> = vec![
        ("Base Model", Box::new(BaseCar::new("Toyota Camry", 28000.00))),
        ("With Leather", Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00))))),
        ("With Sunroof", Box::new(Sunroof::new(Box::new(BaseCar::new("Toyota Camry", 28000.00))))),
        ("Leather + Sunroof", Box::new(Sunroof::new(Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00))))))),
        ("Eco Luxury", Box::new(NavigationSystem::new(Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00))))))),
        ("Sport Only", Box::new(SportPackage::new(Box::new(BaseCar::new("Toyota Camry", 28000.00))))),
        ("Fully Loaded", Box::new(SportPackage::new(
            Box::new(NavigationSystem::new(
                Box::new(Sunroof::new(
                    Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00)))),
                )),
            )),
        ))),
    ];

    // Print header
    println!("{}", format!("{:<18} {:<60} {}", "Config", "Description", "Price"));
    println!("{}", "-".repeat(90));

    // Print each configuration
    for (label, car) in &configs {
        display_config(label, car.as_ref());
    }

    // Price breakdown
    println!();
    println!("--- Price Breakdown: Fully Loaded Camry ---");
    let base = BaseCar::new("Toyota Camry", 28000.00);
    println!("  Base car:                       {:>10}", format_currency(base.cost()));
    println!("  + Leather Seats:                {:>10}", format_currency(1500.00));
    println!("  + Sunroof:                      {:>10}", format_currency(1200.00));
    println!("  + Navigation System:            {:>10}", format_currency(2000.00));
    println!("  + Sport Package:                {:>10}", format_currency(3500.00));
    println!("  -----------------------------------------");
    let loaded = SportPackage::new(
        Box::new(NavigationSystem::new(
            Box::new(Sunroof::new(
                Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00)))),
            )),
        )),
    );
    println!("  Total:                          {:>10}", format_currency(loaded.cost()));

    // Interface compliance
    println!();
    println!("--- Interface Compliance ---");
    for (i, (_, car)) in configs.iter().enumerate() {
        // All Box<dyn Car> implement Car by definition
        println!("  Config {}: implements Car trait = true", i + 1);
    }

    // Reuse with Honda Accord
    println!();
    println!("--- Honda Accord ---");
    let accord_luxury: Box<dyn Car> = Box::new(NavigationSystem::new(
        Box::new(LeatherSeats::new(Box::new(BaseCar::new("Honda Accord", 27000.00)))),
    ));
    println!("  {}", accord_luxury.description());
    println!("  Total: {}", format_currency(accord_luxury.cost()));

    // Demonstrate order independence
    println!();
    println!("--- Order Independence ---");
    let order1: Box<dyn Car> = Box::new(Sunroof::new(
        Box::new(LeatherSeats::new(Box::new(BaseCar::new("Toyota Camry", 28000.00)))),
    ));
    let order2: Box<dyn Car> = Box::new(LeatherSeats::new(
        Box::new(Sunroof::new(Box::new(BaseCar::new("Toyota Camry", 28000.00)))),
    ));
    println!("  Sunroof(Leather): {} — {}", order1.description(), format_currency(order1.cost()));
    println!("  Leather(Sunroof): {} — {}", order2.description(), format_currency(order2.cost()));
    println!("  Same price: {}", (order1.cost() - order2.cost()).abs() < 0.001);
}
```

### Helper: Clone `Box<dyn Car>` workaround

```rust
// Since Box<dyn Car> doesn't implement Clone, we provide a helper
// to create a new BaseCar for reuse in different configurations.
// In production, you might use an Rc or Arc, or a trait with clone_bound.
trait CarClone {
    fn clone_box(&self) -> Box<dyn Car>;
}

impl CarClone for BaseCar {
    fn clone_box(&self) -> Box<dyn Car> {
        Box::new(BaseCar::new(&self.model, self.base_cost))
    }
}

impl Car for Box<dyn Car> {
    fn description(&self) -> String {
        self.as_ref().description()
    }

    fn cost(&self) -> f64 {
        self.as_ref().cost()
    }
}
```

### Language-Specific Notes

- **Traits instead of interfaces/abstract classes** — Rust's `trait` serves as the Component interface. There is no abstract class equivalent; the decorator pattern is implemented entirely through trait implementation on wrapper structs. Each decorator struct holds a `Box<dyn Car>` and implements `Car` for its struct.

- **`Box<dyn Car>` for dynamic dispatch** — Rust requires explicit heap allocation (`Box`) and dynamic dispatch (`dyn`) for trait objects whose concrete type is unknown at compile time. The `Box<dyn Car>` is analogous to a `Car*` in C++ or `ICar` reference in C#. The vtable is stored in the `Box`, enabling runtime polymorphism.

- **Ownership and move semantics** — Rust's ownership model means each `Box<dyn Car>` is moved into the decorator on construction. The `BaseCar` cannot be reused after being wrapped without cloning. The implementation handles this by creating fresh `BaseCar` instances for each configuration, which is acceptable for a demonstration. In production, `Rc<dyn Car>` (reference counting) would allow shared ownership with multiple decorators wrapping the same core.

- **No `Clone` for `Box<dyn Car>`** — The `Box<dyn Car>` type does not implement `Clone` because the trait object's concrete type is unknown. The `CarClone` trait at the bottom of the file demonstrates a workaround: a custom trait with a `clone_box()` method that returns a new `Box<dyn Car>`. This is a common Rust pattern for enabling cloning of trait objects.

- **`impl` blocks for each decorator** — Each decorator struct requires its own `impl Car for DecoratorName` block. There is no way to share implementation across decorators through inheritance, so the delegation boilerplate (`self.car.description()`, `self.car.cost()`) is repeated in each `impl` block. Macros can reduce this repetition in larger codebases.

- **`Box<dyn Car>` impl block** — The implementation includes `impl Car for Box<dyn Car>` at the bottom, which allows treating a `Box<dyn Car>` directly as a `Car` without dereferencing. This is a convenience for the driver's collection iteration, allowing `car.description()` instead of `car.as_ref().description()`.
