The Builder pattern in TypeScript leverages the language's type system to construct complex loan applications with compile-time safety. A generic `LoanApplicationBuilder` interface defines the construction steps, three concrete builders implement loan-specific logic, and a `LoanDirector` orchestrates the building process. TypeScript's strict null checks ensure that no field is accessed before it is set, its readonly modifiers prevent accidental mutation after construction, and its interface system enforces the builder contract across all implementations.

## How the Implementation Works

### Class Diagram (Pseudocode)

The `LoanApplicationBuilder` interface declares the construction steps as a generic contract. Each concrete builder assembles a `LoanApplication` product independently, adding loan-type-specific methods beyond the interface. The `LoanDirector` accepts any builder through the interface and orchestrates the standard construction sequence. Clients configure a builder with specialized data (property info for mortgages, vehicle details for auto loans) before passing it to the director.

### OOP Concepts Applied

- **Abstraction:** The `LoanApplicationBuilder` interface hides internal construction details. Clients and the director interact only through the builder contract, never touching private fields.
- **Encapsulation:** Each concrete builder keeps its fields private (`private` keyword), exposing only the builder methods. Internal state is protected from external modification during the multi-step construction process.
- **Inheritance:** The product type `LoanApplication` serves as a shared base; optional fields for mortgages and auto loans extend the shape polymorphically through optional properties.
- **Polymorphism:** `LoanDirector` accepts any `LoanApplicationBuilder`, treating `PersonalLoanBuilder`, `MortgageBuilder`, and `AutoLoanBuilder` uniformly through the interface.

### SOLID Principles Applied

- **SRP:** Each builder handles exactly one loan type. The director owns only the construction sequence. The display function owns only output formatting — no class does more than one thing.
- **OCP:** Adding a new loan type (for example, `StudentLoanBuilder`) requires zero changes to existing builders or the director. The system is open for extension, closed for modification.
- **LSP:** All concrete builders implement `LoanApplicationBuilder<LoanApplication>`. Any builder is substitutable wherever the interface is accepted without changing correctness.
- **ISP:** The builder interface is minimal and focused — `setPersonalInfo`, `setLoanTerms`, `setInterestRate`, `addDocument`, `getResult`. Extra methods like `setPropertyInfo` live only on the concrete builders that need them.
- **DIP:** `LoanDirector` depends on the abstract `LoanApplicationBuilder` interface, not on any concrete builder. Builders also depend on the abstraction — both sides invert the traditional dependency direction.

```typescript
// Builder Pattern — TypeScript — Banking example
// ================================================
// Participants: LoanApplicationBuilder (interface), PersonalLoanBuilder,
// MortgageBuilder, AutoLoanBuilder (ConcreteBuilders),
// LoanDirector (Director), LoanApplication (Product)
// OOP: abstraction, encapsulation, inheritance, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ================================================

// ─── Product types ───────────────────────────────────────────────────
// PersonalInfo captures the applicant's identity and employment status.
interface PersonalInfo {
  readonly name: string;
  readonly ssnLast4: string;
  readonly employment: string;
}

// LoanApplication is the final product assembled by the builder. It
// includes optional fields for mortgage and auto-loan specifics.
interface LoanApplication {
  readonly personalInfo: PersonalInfo | null;
  readonly loanType: string;
  readonly amount: number;
  readonly termMonths: number;
  readonly interestRate: number;
  readonly documents: ReadonlyArray<{ type: string; ref: string }>;
  readonly propertyAddress?: string;
  readonly appraisedValue?: number;
  readonly downPayment?: number;
  readonly vehicleDescription?: string;
  readonly vin?: string;
  readonly vehicleValue?: number;
  readonly dealership?: string;
}
```

### Builder Interface
```typescript
// ─── Builder interface ───────────────────────────────────────────────
// Generic over the product type, allowing concrete builders to return
// specialized subtypes while maintaining a uniform construction API.
interface LoanApplicationBuilder<T extends LoanApplication> {
  setPersonalInfo(name: string, ssn: string, employment: string): this;
  setLoanTerms(type: string, amount: number, termMonths: number): this;
  setInterestRate(rate: number): this;
  addDocument(type: string, ref: string): this;
  getResult(): T;
}
```

### PersonalLoanBuilder (ConcreteBuilder)
```typescript
// ─── Concrete Builder: PersonalLoanBuilder ───────────────────────────
class PersonalLoanBuilder implements LoanApplicationBuilder<LoanApplication> {
  private personalInfo: PersonalInfo | null = null;
  private loanType = '';
  private amount = 0;
  private termMonths = 0;
  private interestRate = 0;
  private readonly docs: Array<{ type: string; ref: string }> = [];

  setPersonalInfo(name: string, ssn: string, employment: string): this {
    this.personalInfo = { name, ssnLast4: ssn.slice(-4), employment };
    return this;
  }

  setLoanTerms(type: string, amount: number, termMonths: number): this {
    this.loanType = type;
    this.amount = amount;
    this.termMonths = termMonths;
    return this;
  }

  setInterestRate(rate: number): this {
    this.interestRate = rate;
    return this;
  }

  addDocument(type: string, ref: string): this {
    this.docs.push({ type, ref });
    return this;
  }

  getResult(): LoanApplication {
    return {
      personalInfo: this.personalInfo,
      loanType: this.loanType,
      amount: this.amount,
      termMonths: this.termMonths,
      interestRate: this.interestRate,
      documents: [...this.docs],
    };
  }
}
```

### LoanDirector (Director)
```typescript
// ─── Director ────────────────────────────────────────────────────────
// Accepts any LoanApplicationBuilder and orchestrates the step-by-step
// construction for each loan type. The director never knows which
// concrete builder it is using.
class LoanDirector {
  constructPersonalLoan(builder: LoanApplicationBuilder<LoanApplication>): LoanApplication {
    return builder
      .setPersonalInfo('Alice Johnson', '123-45-6789', 'Software Engineer')
      .setLoanTerms('Personal', 15000, 60)
      .setInterestRate(11.99)
      .addDocument('ID', 'Driver License #DL-987654')
      .addDocument('Income Verification', 'Paystub #2025-Q4')
      .getResult();
  }

  constructMortgage(builder: LoanApplicationBuilder<LoanApplication>): LoanApplication {
    return builder
      .setPersonalInfo('Bob Martinez', '987-65-4321', 'Teacher')
      .setLoanTerms('Mortgage', 350000, 360)
      .setInterestRate(6.25)
      .addDocument('ID', 'Passport #P-445566')
      .addDocument('Tax Returns', 'IRS 1040 2025')
      .addDocument('Appraisal Report', 'Appraisal #A-78901')
      .getResult();
  }

  constructAutoLoan(builder: LoanApplicationBuilder<LoanApplication>): LoanApplication {
    return builder
      .setPersonalInfo('Carol Chen', '456-78-9012', 'Nurse')
      .setLoanTerms('Auto', 32000, 60)
      .setInterestRate(7.49)
      .addDocument('ID', 'Driver License #DL-332211')
      .addDocument('Vehicle Registration', 'Reg #REG-2026-0042')
      .getResult();
  }
}
```

### Display Helper
```typescript
// ─── Display helper ──────────────────────────────────────────────────
function formatCurrency(amount: number): string {
  return `$${amount.toFixed(2)}`;
}

function displayLoanApplication(app: LoanApplication, label: string): void {
  const info = app.personalInfo!;
  console.log(`=== ${label} ===`);
  console.log(
    `Applicant: ${info.name} | Loan: ${formatCurrency(app.amount)} over ` +
    `${app.termMonths}mo @ ${app.interestRate}% APR`
  );

  if (app.propertyAddress) {
    console.log(
      `Property: ${app.propertyAddress} (Appraised: ${formatCurrency(app.appraisedValue!)})`
    );
    const loanAmount = app.amount - app.downPayment!;
    const ltv = (loanAmount / app.appraisedValue!) * 100;
    console.log(
      `Down Payment: ${formatCurrency(app.downPayment!)} | LTV: ${ltv.toFixed(2)}%`
    );
  }

  if (app.vehicleDescription) {
    console.log(
      `Vehicle: ${app.vehicleDescription} ` +
      `(VIN: ${app.vin}, Value: ${formatCurrency(app.vehicleValue!)})`
    );
    console.log(`Dealership: ${app.dealership}`);
  }

  const docLabels = app.documents.map((d) => d.type).join(', ');
  console.log(`Documents: ${docLabels}`);
  console.log();
}
```

### Main / Driver
```typescript
// ─── Driver ──────────────────────────────────────────────────────────
function main(): void {
  const director = new LoanDirector();

  // 1. Personal Loan — Alice Johnson
  const personalBuilder = new PersonalLoanBuilder();
  const personalLoan = director.constructPersonalLoan(personalBuilder);
  displayLoanApplication(personalLoan, 'Personal Loan Application');

  // 2. Mortgage — Bob Martinez
  // Property-specific data is set on the builder BEFORE passing it to
  // the director. The director fills in the common fields via the
  // interface; the builder retains the mortgage-specific state.
  const mortgageBuilder = new MortgageBuilder();
  mortgageBuilder.setPropertyInfo('123 Main St, Springfield', 450000, 70000);
  const mortgage = director.constructMortgage(mortgageBuilder);
  displayLoanApplication(mortgage, 'Mortgage Application');

  // 3. Auto Loan — Carol Chen
  // Vehicle details and dealership are pre-configured on the builder
  // before the director fills in the standard loan fields.
  const autoBuilder = new AutoLoanBuilder();
  autoBuilder
    .setVehicleInfo('2024 Honda Accord', '1HGCV1F34PA012345', 35000)
    .setDealership('City Motors');
  const autoLoan = director.constructAutoLoan(autoBuilder);
  displayLoanApplication(autoLoan, 'Auto Loan Application');
}

main();
```

### Language-Specific Notes

- **Strict null checks:** The `PersonalInfo | null` type on `personalInfo` forces every builder method to handle the uninitialized state at compile time. TypeScript will refuse to compile code that accesses `personalInfo` before `setPersonalInfo` is called, preventing a whole class of runtime errors.
- **Interfaces as contracts:** The `LoanApplicationBuilder` interface with the generic bound `<T extends LoanApplication>` provides a compile-time contract that every builder must fulfill. Any mismatch in method signatures or return types is caught at compile time, not at runtime.
- **Readonly and immutability:** The `readonly` modifier on `LoanApplication` fields and `ReadonlyArray` for the documents list prevent consumers from mutating a completed application. Once `getResult()` returns the product, it is effectively frozen — the builder pattern combined with readonly delivers safety at both ends of the construction pipeline.
- **Fluent chaining with `this` return types:** Every builder method returns `this`, enabling the fluent method chaining used throughout the director. TypeScript correctly infers the narrowed return type, so chaining across builder-specific methods (for example, `setVehicleInfo().setDealership()`) preserves full type information.
- **Discriminated optional fields:** The `LoanApplication` interface models loan-type-specific data through optional properties (`propertyAddress?`, `vin?`, etc.). The display function uses type narrowing (`if (app.propertyAddress)`) to branch correctly at runtime without runtime type checks or casting.
- **Generics for flexibility:** The `LoanApplicationBuilder<T>` generic parameter allows future builders to return specialized subtypes (for example, a `MortgageBuilder` returning `MortgageApplication extends LoanApplication` with typed property fields) while the director continues to work with the base `LoanApplication` contract unchanged.
