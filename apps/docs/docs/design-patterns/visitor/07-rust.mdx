## Implementation

Below is a complete Rust implementation of the Visitor pattern using a stock market portfolio domain. The `Portfolio` holds a collection of financial instruments (Stocks and Bonds), and each `Visitor` trait implementation performs a different analysis: computing total value, calculating annual dividend income, assessing risk, and generating a formatted report. Rust's trait system naturally models the Visitor pattern's double dispatch mechanism through trait objects (`Box<dyn Element>`) and a `visit` method per concrete type.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   trait Element                      │  <- OOP: Abstraction
//   ├──────────────────────────────────────┤     ISP: minimal accept contract
//   │   fn accept(&self, visitor:          │
//   │       &dyn Visitor)                  │
//   └────────────────┬─────────────────────┘
//                    ▲ impl
//   ┌────────────────┴──────────────────┬──┐
//   │                                   │
//   ┌──────────────────┐   ┌──────────────────────┐
//   │     Stock        │   │      Bond            │  <- OOP: Polymorphism
//   ├──────────────────┤   ├──────────────────────┤     LSP: both implement
//   │ ticker: String   │   │ name: String         │     Element trait,
//   │ shares: u32      │   │ face_value: f64      │     substitutable
//   │ price: f64       │   │ coupon_rate: f64     │     in Portfolio
//   │ div_yield: f64   │   │ years_to_mat: u32    │
//   └──────────────────┘   └──────────────────────┘
//
//   ┌──────────────────────────────────────┐
//   │   trait Visitor                      │  <- OOP: Abstraction
//   ├──────────────────────────────────────┤     ISP: one visit method per type
//   │   fn visit_stock(&self, &Stock)      │
//   │   fn visit_bond(&self, &Bond)        │
//   └────────────────┬─────────────────────┘
//                    ▲ impl
//   ┌────────────────┴──────────────────┬──┐
//   │ ValueVisitor  │ DividendVisitor  │  │  <- SRP: one algorithm per visitor
//   │ RiskVisitor   │ ReportVisitor    │  │     OCP: new operations = new
//   └───────────────────────────────────┘       visitor structs
```

#### OOP Concepts Applied

1. **Abstraction** -- The `Element` and `Visitor` traits define the contracts. Client code depends on `&dyn Element` and `&dyn Visitor`, not on concrete types.
2. **Encapsulation** -- Each struct hides its internal fields. Visitors access data only through the struct's public fields or getter methods.
3. **Inheritance (via trait implementation)** -- `Stock` and `Bond` both implement `Element`. They share the `accept` interface but each provides its own dispatch logic.
4. **Polymorphism (via trait objects)** -- `Portfolio` holds `Vec<Box<dyn Element>>`. The `accept_all` method calls `accept` on each element, and the correct visit method is dispatched at runtime.

#### SOLID Principles Applied

5. **SRP** -- Each visitor struct has exactly one responsibility: calculate value, compute dividends, assess risk, or generate a report. The `Portfolio` only manages the collection.
6. **OCP** -- Adding a new analysis requires only a new struct implementing `Visitor`. No existing code needs modification. The element types are closed for modification, operations are open for extension.
7. **LSP** -- Any `Element` implementor can be substituted into `Portfolio` without breaking the client. The `accept_all` method works uniformly regardless of the concrete element type.
8. **ISP** -- The `Visitor` trait has exactly as many methods as there are element types (two: `visit_stock` and `visit_bond`). No implementor is forced to provide methods it does not need.
9. **DIP** -- `Portfolio` depends on the `Element` trait abstraction, not on concrete `Stock` or `Bond` types. Visitors depend on the `Visitor` trait contract. High-level modules are insulated from low-level implementation details.

### Element (Trait)

```rust
// OOP role: Abstraction
// SOLID: ISP (minimal contract), DIP (depended upon abstraction)
trait Element {
    fn accept(&self, visitor: &dyn Visitor);
}
```

### Visitor (Trait)

```rust
// OOP role: Abstraction
// SOLID: ISP (one method per element type), OCP (extensible with new visitors)
trait Visitor {
    fn visit_stock(&self, stock: &Stock);
    fn visit_bond(&self, bond: &Bond);
}
```

### Stock (Concrete Element)

```rust
// OOP role: Polymorphism (implements Element)
// SOLID: SRP (represents only stock data), LSP (substitutable for Element)
#[derive(Clone)]
struct Stock {
    ticker: String,
    company_name: String,
    shares: u32,
    price_per_share: f64,
    dividend_yield: f64,
}

impl Stock {
    fn new(ticker: &str, company_name: &str, shares: u32, price: f64, yield_: f64) -> Self {
        Stock {
            ticker: ticker.to_string(),
            company_name: company_name.to_string(),
            shares,
            price_per_share: price,
            dividend_yield: yield_,
        }
    }
}

impl Element for Stock {
    fn accept(&self, visitor: &dyn Visitor) {
        visitor.visit_stock(self);
    }
}
```

### ValueVisitorCell (Concrete Visitor)

```rust
// OOP role: Polymorphism (implements Visitor)
// SOLID: SRP (only calculates total value), OCP (new analysis = new visitor)
struct ValueVisitorCell {
    total_value: Cell<f64>,
}

impl ValueVisitorCell {
    fn new() -> Self {
        ValueVisitorCell { total_value: Cell::new(0.0) }
    }

    fn get_total_value(&self) -> f64 {
        self.total_value.get()
    }
}

impl Visitor for ValueVisitorCell {
    fn visit_stock(&self, stock: &Stock) {
        let value = stock.shares as f64 * stock.price_per_share;
        self.total_value.set(self.total_value.get() + value);
    }

    fn visit_bond(&self, bond: &Bond) {
        self.total_value.set(self.total_value.get() + bond.face_value);
    }
}
```

### DividendVisitorCell (Concrete Visitor)

```rust
// OOP role: Polymorphism (implements Visitor)
// SOLID: SRP (only calculates annual income)
struct DividendVisitorCell {
    total_dividend: Cell<f64>,
}

impl DividendVisitorCell {
    fn new() -> Self {
        DividendVisitorCell { total_dividend: Cell::new(0.0) }
    }

    fn get_total_dividend(&self) -> f64 {
        self.total_dividend.get()
    }
}

impl Visitor for DividendVisitorCell {
    fn visit_stock(&self, stock: &Stock) {
        let value = stock.shares as f64 * stock.price_per_share;
        let dividend = value * stock.dividend_yield;
        self.total_dividend.set(self.total_dividend.get() + dividend);
    }

    fn visit_bond(&self, bond: &Bond) {
        let coupon = bond.face_value * bond.coupon_rate;
        self.total_dividend.set(self.total_dividend.get() + coupon);
    }
}
```

### RiskVisitor (Concrete Visitor)

```rust
// OOP role: Polymorphism (implements Visitor)
// SOLID: SRP (only assesses risk)
struct RiskVisitor {
    results: RefCell<Vec<String>>,
}

impl RiskVisitor {
    fn new() -> Self {
        RiskVisitor { results: RefCell::new(Vec::new()) }
    }

    fn get_results(&self) -> Vec<String> {
        self.results.borrow().clone()
    }
}

impl Visitor for RiskVisitor {
    fn visit_stock(&self, stock: &Stock) {
        let assessment = if stock.dividend_yield > 0.04 {
            format!("{}: LOW RISK -- established dividend payer", stock.ticker)
        } else if stock.dividend_yield > 0.01 {
            format!("{}: MEDIUM RISK -- moderate dividend", stock.ticker)
        } else {
            format!("{}: HIGH RISK -- low or no dividend", stock.ticker)
        };
        self.results.borrow_mut().push(assessment);
    }

    fn visit_bond(&self, bond: &Bond) {
        let assessment = if bond.years_to_maturity > 20 {
            format!("{}: MEDIUM RISK -- long maturity", bond.name)
        } else if bond.years_to_maturity > 10 {
            format!("{}: LOW RISK -- medium maturity", bond.name)
        } else {
            format!("{}: LOW RISK -- short maturity", bond.name)
        };
        self.results.borrow_mut().push(assessment);
    }
}
```

### ReportVisitor (Concrete Visitor)

```rust
// OOP role: Polymorphism (implements Visitor)
// SOLID: SRP (only generates formatted report)
struct ReportVisitor {
    lines: RefCell<Vec<String>>,
}

impl ReportVisitor {
    fn new() -> Self {
        ReportVisitor { lines: RefCell::new(Vec::new()) }
    }

    fn get_report(&self) -> String {
        let lines = self.lines.borrow();
        let mut report = String::from("\nPortfolio Report:\n");
        report.push_str("| Ticker  | Name                 | Qty/Mat | Price/Val  | Mkt Value  | Income    |\n");
        report.push_str("|---------+----------------------+---------+------------+------------+-----------|\n");
        for line in lines.iter() {
            report.push_str(line);
            report.push('\n');
        }
        report
    }
}

impl Visitor for ReportVisitor {
    fn visit_stock(&self, stock: &Stock) {
        let value = stock.shares as f64 * stock.price_per_share;
        let income = value * stock.dividend_yield;
        let line = format!(
            "| {:<6} | {:<20} | {:>5}  | ${:>8.2} | ${:>8.2} | ${:>8.2} |",
            stock.ticker, stock.company_name, stock.shares, stock.price_per_share, value, income
        );
        self.lines.borrow_mut().push(line);
    }

    fn visit_bond(&self, bond: &Bond) {
        let income = bond.face_value * bond.coupon_rate;
        let line = format!(
            "| {:<6} | {:<20} | {:>2}yr  | ${:>8.2} | ${:>8.2} | ${:>8.2} |",
            bond.name, "Bond", bond.years_to_maturity, bond.face_value, bond.face_value, income
        );
        self.lines.borrow_mut().push(line);
    }
}
```

### Portfolio (ObjectStructure)

```rust
// OOP role: Composition (aggregates elements)
// SOLID: SRP (manages collection only), DIP (depends on Element trait)
struct Portfolio {
    elements: Vec<Box<dyn Element>>,
}

impl Portfolio {
    fn new() -> Self {
        Portfolio { elements: Vec::new() }
    }

    fn add(&mut self, element: Box<dyn Element>) {
        self.elements.push(element);
    }

    fn accept_all(&self, visitor: &dyn Visitor) {
        for element in &self.elements {
            element.accept(visitor);
        }
    }
}
```

### Main / Driver

```rust
fn main() {
    let mut portfolio = Portfolio::new();

    // Add stocks with realistic data
    portfolio.add(Box::new(Stock::new("AAPL", "Apple Inc.", 100, 175.50, 0.005)));
    portfolio.add(Box::new(Stock::new("GOOGL", "Alphabet Inc.", 50, 140.25, 0.004)));
    portfolio.add(Box::new(Stock::new("MSFT", "Microsoft Corp.", 80, 380.75, 0.007)));
    portfolio.add(Box::new(Stock::new("JPM", "JPMorgan Chase", 60, 155.30, 0.048)));

    // Add bonds
    portfolio.add(Box::new(Bond::new("US10Y", 10000.0, 0.043, 10)));
    portfolio.add(Box::new(Bond::new("US30Y", 5000.0, 0.048, 30)));

    // 1. ValueVisitor -- calculate total portfolio value
    let value_visitor = ValueVisitorCell::new();
    portfolio.accept_all(&value_visitor);
    println!("Total Portfolio Value: ${:.2}", value_visitor.get_total_value());

    // 2. DividendVisitor -- calculate annual income
    let div_visitor = DividendVisitorCell::new();
    portfolio.accept_all(&div_visitor);
    println!("Annual Dividend/Coupon Income: ${:.2}", div_visitor.get_total_dividend());

    // 3. RiskVisitor -- assess risk for each holding
    let risk_visitor = RiskVisitor::new();
    portfolio.accept_all(&risk_visitor);
    println!("\nRisk Assessment:");
    for result in risk_visitor.get_results() {
        println!("  {}", result);
    }

    // 4. ReportVisitor -- generate formatted report
    let report_visitor = ReportVisitor::new();
    portfolio.accept_all(&report_visitor);
    println!("{}", report_visitor.get_report());
}
```

### Language-Specific Notes

- **Traits over classes** -- Rust uses `trait` definitions instead of abstract classes or interfaces. The `Element` and `Visitor` traits correspond to the pattern's abstract participants. Concrete types implement these traits with `impl Trait for Type` blocks.
- **Interior mutability for visitor state** -- Rust's ownership model requires explicit handling of mutable state. The `ValueVisitorCell` and `DividendVisitorCell` use `Cell<f64>` for interior mutability (allowing mutation through an `&self` reference). The `RiskVisitor` and `ReportVisitor` use `RefCell<Vec<String>>` for collection types. This is the idiomatic Rust approach to the Visitor pattern where the visit methods take `&self`.
- **Trait objects for polymorphism** -- `Box<dyn Element>` and `&dyn Visitor` provide runtime polymorphism, analogous to virtual method dispatch in OOP languages. The `Portfolio` holds `Vec<Box<dyn Element>>`, allowing heterogeneous collections of stocks and bonds.
- **No inheritance hierarchy** -- Unlike Java or C#, Rust has no class inheritance. The pattern is modeled entirely through trait implementation. Each concrete type independently implements the `Element` trait, and the `accept` method manually dispatches to the correct visitor method.
- **Pattern matching alternative** -- For simpler cases, Rust's `match` on enums can replace the Visitor pattern entirely. However, the Visitor pattern is still useful when you need to add new operations without modifying the element types, especially when the number of operations grows faster than the number of types.
- **Zero-cost abstractions** -- Rust's trait objects have a small runtime cost (vtable lookup). For performance-critical code, you could use generics with monomorphization instead of `dyn` trait objects, but for most use cases the trait object approach is idiomatic and sufficiently fast.
