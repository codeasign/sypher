The Factory Method pattern in C# 12 / .NET 8 maps naturally to the language's interface-first design philosophy. The `IAccount` interface declares the product contract, concrete records `SavingsAccount`, `CheckingAccount`, and `LoanAccount` implement it, and `AccountFactory` provides a virtual factory method that subclasses override. C#'s primary constructors, `readonly record struct` value semantics, and built-in `decimal` type for financial calculations make the banking domain implementation both type-safe and expressive.

## How the Implementation Works

### Class Diagram (Pseudocode)

```
interface IAccount
    +AccountType: string        // "Savings", "Checking", "Loan"
    +Balance: decimal
    +Deposit(amount: decimal): void
    +Withdraw(amount: decimal): bool
    +GetStatement(): string

record SavingsAccount : IAccount
    -Owner: string
    -Balance: decimal
    -InterestRate: decimal
    -MinBalance: decimal
    +AccountType => "Savings"
    +Deposit: adds amount to balance
    +Withdraw: fails if below minimum balance
    +GetStatement: formatted savings statement

record CheckingAccount : IAccount
    -Owner: string
    -Balance: decimal
    -OverdraftLimit: decimal
    +AccountType => "Checking"
    +Deposit: adds amount to balance
    +Withdraw: allows overdraft up to limit
    +GetStatement: formatted checking statement

record LoanAccount : IAccount
    -Owner: string
    -Principal: decimal
    -InterestRate: decimal
    -TermMonths: int
    -MonthlyPayment: decimal
    +AccountType => "Loan"
    +Deposit: applies payment to principal
    +Withdraw: not allowed (loan disbursement only)
    +GetStatement: formatted loan statement

class AccountFactory
    +CreateAccount(type, owner, initialDeposit): IAccount
    +OpenAndFundAccount(type, owner, initialDeposit): IAccount
```

### OOP Concepts Applied

| Concept | Application |
|---|---|
| **Abstraction** | `IAccount` exposes only `Deposit`, `Withdraw`, and `GetStatement` hiding each account type's internal calculation details |
| **Encapsulation** | Fields like `InterestRate`, `OverdraftLimit`, and `Principal` are `private` or `init`-only; mutation happens only through well-defined methods |
| **Inheritance** | All three concrete records implement `IAccount` sharing the same polymorphic interface contract |
| **Polymorphism** | The factory returns `IAccount`; callers invoke methods without knowing whether the runtime type is Savings, Checking, or Loan |

### SOLID Principles Applied

| Principle | How the Code Applies It |
|---|---|
| **SRP** | Each account record has one reason to change (its own business rules); `AccountFactory` has one responsibility (creating configured account instances) |
| **OCP** | A new `BusinessAccount` record implementing `IAccount` requires zero changes to existing accounts or the factory's polymorphic callers |
| **LSP** | Every concrete account satisfies the `IAccount` contract invariants; a `SavingsAccount` can substitute `IAccount` anywhere without surprising behavior |
| **ISP** | `IAccount` is narrowly focused on account operations — no forced dependencies on unrelated concerns like customer notification or reporting |
| **DIP** | `AccountFactory` depends on the `IAccount` abstraction, not on concrete records; callers depend only on `IAccount` and the factory abstraction |

## Complete Implementation

```csharp
// Factory Method Pattern — Banking Domain (C# 12 / .NET 8)
//
// Participants:
//   IAccount            (Product)         — interface contract for all accounts
//   SavingsAccount      (ConcreteProduct) — savings with interest and minimum balance
//   CheckingAccount     (ConcreteProduct) — checking with overdraft protection
//   LoanAccount         (ConcreteProduct) — amortizing loan with monthly payments
//   AccountFactory      (Creator)         — declares the factory method CreateAccount
//
// OOP: Abstraction (IAccount), Encapsulation (private fields), Inheritance (interface impl),
//      Polymorphism (callers use IAccount)
// SOLID: SRP (one reason to change per class), OCP (new accounts need no existing changes),
//        LSP (all accounts satisfy IAccount), ISP (focused interface), DIP (factory depends on abstraction)

using System.Globalization;
```

### IAccount (Product Interface)
```csharp
/// <summary>Product interface that all account types implement.</summary>
public interface IAccount
{
    string AccountType { get; }
    string Owner { get; }
    decimal Balance { get; }
    void Deposit(decimal amount);
    bool Withdraw(decimal amount);
    string GetStatement();
}
```

### SavingsAccount (Concrete Product)
```csharp
/// <summary>
/// Savings account with configurable interest rate and minimum balance requirement.
/// OCP: New account variants extend IAccount without modifying this class.
/// LSP: Fully substitutable for IAccount — all invariants documented.
/// </summary>
public record SavingsAccount : IAccount
{
    // SRP: This record owns only savings-specific business rules.
    private readonly decimal _interestRate;    // Annual rate, e.g. 0.035m = 3.5%
    private readonly decimal _minimumBalance;

    public string AccountType => "Savings";
    public string Owner { get; }
    public decimal Balance { get; private set; }

    public SavingsAccount(
        string owner, decimal initialDeposit,
        decimal interestRate = 0.025m, decimal minimumBalance = 100.00m)
    {
        Owner = owner;
        Balance = initialDeposit;
        _interestRate = interestRate;
        _minimumBalance = minimumBalance;
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Deposit amount must be positive.", nameof(amount));
        Balance += amount;
    }

    public bool Withdraw(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Withdrawal amount must be positive.", nameof(amount));

        decimal projectedBalance = Balance - amount;
        if (projectedBalance < _minimumBalance)
            return false;   // Cannot go below minimum balance.

        Balance -= amount;
        return true;
    }

    public string GetStatement()
    {
        decimal monthlyInterest = Balance * (_interestRate / 12m);
        return string.Create(CultureInfo.InvariantCulture,
            $"""
            ═══════════════════════════════════
              SAVINGS ACCOUNT STATEMENT
              Owner:       {Owner}
              Balance:     {Balance:C}
              Interest:    {_interestRate:P1} APR
              Min Balance: {_minimumBalance:C}
              Monthly Int: {monthlyInterest:C}
            ═══════════════════════════════════
            """);
    }
}
```

### CheckingAccount (Concrete Product)
```csharp
/// <summary>
/// Checking account with overdraft protection up to a configurable limit.
/// OCP, LSP: Same pattern as SavingsAccount — independently extendable.
/// </summary>
public record CheckingAccount : IAccount
{
    private readonly decimal _overdraftLimit;

    public string AccountType => "Checking";
    public string Owner { get; }
    public decimal Balance { get; private set; }

    public CheckingAccount(
        string owner, decimal initialDeposit,
        decimal overdraftLimit = 500.00m)
    {
        Owner = owner;
        Balance = initialDeposit;
        _overdraftLimit = overdraftLimit;
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Deposit amount must be positive.", nameof(amount));
        Balance += amount;
    }

    public bool Withdraw(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Withdrawal amount must be positive.", nameof(amount));

        decimal projectedBalance = Balance - amount;
        if (projectedBalance < -_overdraftLimit)
            return false;   // Exceeds overdraft protection.

        Balance -= amount;
        return true;
    }

    public string GetStatement()
    {
        decimal availableOverdraft = _overdraftLimit + Balance;
        return string.Create(CultureInfo.InvariantCulture,
            $"""
            ═══════════════════════════════════
              CHECKING ACCOUNT STATEMENT
              Owner:             {Owner}
              Balance:           {Balance:C}
              Overdraft Limit:   {_overdraftLimit:C}
              Available Overdft: {availableOverdraft:C}
            ═══════════════════════════════════
            """);
    }
}
```

### LoanAccount (Concrete Product)
```csharp
/// <summary>
/// Amortizing loan account. Withdrawals represent loan disbursement (once);
/// deposits represent payments that reduce principal.
/// ISP: Focused only on loan-specific operations via IAccount.
/// </summary>
public record LoanAccount : IAccount
{
    private readonly decimal _interestRate;          // Annual rate
    private readonly int _termMonths;
    private readonly decimal _monthlyPayment;
    private int _paymentsMade;

    public string AccountType => "Loan";
    public string Owner { get; }
    public decimal Balance { get; private set; }     // Remaining principal
    public decimal Principal { get; }                 // Original loan amount

    public LoanAccount(
        string owner, decimal principal,
        decimal interestRate, int termMonths)
    {
        Owner = owner;
        Principal = principal;
        Balance = principal;       // Full principal owed at origination
        _interestRate = interestRate;
        _termMonths = termMonths;

        // Standard amortization formula: M = P * [r(1+r)^n] / [(1+r)^n - 1]
        decimal monthlyRate = interestRate / 12m;
        decimal factor = (decimal)Math.Pow((double)(1m + monthlyRate), termMonths);
        _monthlyPayment = principal * (monthlyRate * factor) / (factor - 1m);
        _paymentsMade = 0;
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentException("Payment amount must be positive.", nameof(amount));

        Balance -= amount;
        if (Balance < 0) Balance = 0m;   // Overpayment handled gracefully
        _paymentsMade++;
    }

    public bool Withdraw(decimal amount)
    {
        // Loan disbursement: allowed only once at origination (no balance yet).
        // For this simulation, withdrawals are not permitted after funding.
        return false;
    }

    public string GetStatement()
    {
        int remainingMonths = _termMonths - _paymentsMade;
        return string.Create(CultureInfo.InvariantCulture,
            $"""
            ═══════════════════════════════════
              LOAN ACCOUNT STATEMENT
              Owner:          {Owner}
              Principal:      {Principal:C}
              Balance Owed:   {Balance:C}
              Rate:           {_interestRate:P1} APR
              Term:           {_termMonths} months
              Monthly Pmt:    {_monthlyPayment:C}
              Payments Made:  {_paymentsMade}
              Remaining:      {remainingMonths} payments
            ═══════════════════════════════════
            """);
    }
}
```

### AccountFactory (Creator)
```csharp
/// <summary>
/// Factory that creates and funds bank accounts based on a type code.
/// DIP: Depends on IAccount abstraction, not concrete record types.
/// SRP: Single responsibility — account instantiation and initial funding.
/// </summary>
public class AccountFactory
{
    /// <summary>Factory method — subclasses could override to provide variants.</summary>
    public virtual IAccount CreateAccount(string type, string owner, decimal initialDeposit)
    {
        return type.ToLowerInvariant() switch
        {
            "savings" => new SavingsAccount(owner, initialDeposit),
            "checking" => new CheckingAccount(owner, initialDeposit),
            "loan" => new LoanAccount(owner, initialDeposit, 0.065m, 60),
            _ => throw new ArgumentException($"Unknown account type: {type}", nameof(type))
        };
    }

    /// <summary>
    /// Template method: creates an account, deposits initial funds, and prints a confirmation.
    /// Demonstrates how the factory method is called within a larger workflow.
    /// </summary>
    public IAccount OpenAndFundAccount(string type, string owner, decimal initialDeposit)
    {
        Console.WriteLine($"  Opening {type} account for {owner} with {initialDeposit:C}...");
        IAccount account = CreateAccount(type, owner, initialDeposit);

        if (initialDeposit > 0 && account.AccountType != "Loan")
        {
            account.Deposit(0);   // Balance already set by constructor; no-op for illustration
        }

        Console.WriteLine($"  Created {account.AccountType} account #{account.GetHashCode():X} for {owner}");
        Console.WriteLine($"  Starting balance: {account.Balance:C}");
        Console.WriteLine();
        return account;
    }
}
```

## Language-Specific Notes

- **C# records for immutable product state**: Using `record` (or `readonly record struct`) for `SavingsAccount`, `CheckingAccount`, and `LoanAccount` gives value-based equality, `with` expressions for non-destructive mutation, and `init`-only property setters — all of which reinforce encapsulation and reduce accidental state corruption.

- **Interfaces with default implementations**: C# 8+ allows default interface methods, which can provide shared product behavior (e.g., a default `GetStatement` format) while still letting each concrete record override it. The example above deliberately avoids defaults to keep the interface narrowly focused per ISP.

- **Primary constructors and `private readonly` fields**: C# 12 primary constructors on records reduce boilerplate for injected dependencies like `_interestRate` and `_overdraftLimit`. The combination of `private readonly` fields with `init`-only auto-properties enforces that configuration is set at construction time and immutable afterward.

- **Polymorphic dispatch with `switch` expressions**: The factory method uses a `switch` expression (not a chain of `if/else`) to map type strings to concrete products. This is idiomatic C# and aligns with OCP — adding a new type only requires a new arm in the switch and a new record, without touching existing case bodies.

- **`decimal` for financial calculations**: C#'s `decimal` type (28-29 significant digits, base-10 floating point) is the correct choice for monetary values. It avoids the binary rounding errors of `float`/`double` (e.g., `0.1m + 0.2m == 0.3m` is `true`), making it suitable for interest accrual and amortization schedules.

- **Factory method as a template method hook**: In C#, the factory method often pairs with a template method (here `OpenAndFundAccount`) that calls `CreateAccount` as part of a larger workflow. This demonstrates how the GoF pattern composes with other creational and behavioral patterns — particularly useful in ASP.NET Core dependency injection scenarios where the factory resolves runtime account types.
