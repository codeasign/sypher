The Flyweight pattern is a structural design pattern that enables sharing fine-grained objects at scale by separating intrinsic state (shared, context-independent) from extrinsic state (unique, context-dependent). Instead of creating thousands of nearly identical objects that each carry the same data, the pattern stores shared data once in a flyweight object and reuses that object across many contexts. The client supplies the extrinsic state each time it uses the flyweight, so the flyweight can remain stateless or carry only what is common across all uses.

What makes the Flyweight pattern powerful is that it turns object identity from a liability into an asset. In naive object-oriented designs, every distinct conceptual object gets its own heap allocation, and identical data is duplicated across thousands of instances. The Flyweight pattern recognizes that many objects share the same underlying data and that sharing that data reduces memory consumption proportionally to the ratio of duplication. For systems where the number of objects is bounded only by user input or data volume — such as a car configurator that lets customers build millions of unique vehicle configurations — the Flyweight pattern can be the difference between a system that runs on a laptop and one that crashes a server.

## The Problem Without This Pattern

Consider a car manufacturer's online configurator. A customer selects a car model, chooses an engine, picks wheels, selects a paint color, and adds optional features. The system must represent each configured car as an object so it can be displayed, priced, saved, and compared.

Without the Flyweight pattern, every car configuration stores its own copy of every specification object:

```python
# BROKEN — every car stores its own copy of every spec object
class CarConfiguration:
    def __init__(self, vin, owner, license_plate, production_date,
                 engine_model, engine_hp, engine_displacement, engine_cylinders,
                 wheel_diameter, wheel_width, wheel_material,
                 paint_code, paint_name, paint_finish,
                 trim_name, trim_features):
        self.vin = vin
        self.owner = owner
        self.license_plate = license_plate
        self.production_date = production_date
        # Intrinsic data duplicated in every car
        self.engine_model = engine_model
        self.engine_hp = engine_hp
        self.engine_displacement = engine_displacement
        self.engine_cylinders = engine_cylinders
        self.wheel_diameter = wheel_diameter
        self.wheel_width = wheel_width
        self.wheel_material = wheel_material
        self.paint_code = paint_code
        self.paint_name = paint_name
        self.paint_finish = paint_finish
        self.trim_name = trim_name
        self.trim_features = trim_features
```

What breaks:

- **Memory explosion** — A car configurator may have only 50 engine variants, 30 wheel types, 40 paint colors, and 20 trim levels — a total of 140 unique specification objects. But if the configurator generates 100,000 saved car configurations, and each car stores its own copy of every spec field, the system allocates memory for 14 million spec data points instead of 140. If each spec object is 200 bytes, that is 28 MB of duplicates instead of 28 KB of shared data. As the number of configurations grows into the millions, the waste becomes catastrophic.

- **Cache-hostile memory layout** — When every car configuration holds its own copies of the same spec data, the CPU cache cannot hold as many car objects. A cache miss on a spec field that is identical across thousands of cars forces the same data to be loaded from main memory over and over. This degrades the performance of operations like filtering, sorting, and bulk pricing.

- **Update inconsistency** — When the manufacturer updates the horsepower of the 2.0L Turbo engine from 250 to 260, every CarConfiguration that uses that engine must be updated individually. In a naive implementation, the update requires iterating over all 100,000 configurations, finding the ones with matching engine model strings, and updating each one. If any configuration is missed, the system displays inconsistent horsepower values for the same engine.

- **Object identity loss** — Without the Flyweight pattern, there is no canonical object representing the 2.0L Turbo engine. Two CarConfiguration objects that both use the same engine have separate EngineSpec objects with the same field values. The system cannot use reference equality to check whether two cars share the same engine. Every comparison must be a field-by-field structural comparison, which is slower and more error-prone.

- **Serialization bloat** — When saving car configurations to a database or sending them over the network, every car's configuration carries its own copy of every spec field. A 100,000-car export produces a payload that is orders of magnitude larger than necessary. The database stores the same engine specs 100,000 times instead of once.

The Flyweight pattern solves all of these problems by separating the specification data that is shared across many cars (intrinsic state) from the per-car identification data that is unique to each vehicle (extrinsic state). The spec objects are created once, stored in a factory, and shared across all car configurations that use them.

### Participants

- **Flyweight (`CarSpec`)** — Declares the interface through which flyweights can receive and act on extrinsic state. In the Cars domain, the interface exposes a `display()` method that returns a formatted description of the spec. The interface is intentionally small so that all concrete flyweights can implement it uniformly.

- **ConcreteFlyweight (`EngineSpec`, `WheelSpec`, `PaintSpec`)** — Implements the `CarSpec` interface and stores the intrinsic state (the spec data that is shared across many car configurations). Each concrete flyweight is immutable after construction: its fields (engine model, horsepower, wheel diameter, paint code, etc.) never change. Immutability is essential because the flyweight is shared across multiple clients, and any mutation would affect all cars referencing that flyweight.

- **FlyweightFactory** — Creates and manages flyweight objects. It maintains a cache (typically a `Map` or `Dictionary`) of existing flyweights keyed by a unique identifier such as the engine model name or paint code. When a client requests a flyweight, the factory checks the cache first. If the flyweight exists, it returns the cached instance. If not, it creates a new one, stores it in the cache, and returns it. The factory ensures that flyweights are shared rather than duplicated.

- **Client (`CarConfiguration`)** — Maintains references to one or more flyweight objects and computes or stores the extrinsic state (VIN, owner, license plate, production date). The client passes extrinsic state to the flyweight when calling its methods. Each `CarConfiguration` object holds references to shared `EngineSpec`, `WheelSpec`, and `PaintSpec` flyweights, plus its own unique identification data.

### How the Participants Interact

The interaction follows a consistent sequence. The client creates a `CarConfiguration` by requesting the appropriate spec flyweights from the factory. The factory returns existing flyweights or creates new ones. The client stores the flyweight references alongside its own extrinsic state. When the client needs to display the car's configuration, it delegates to the flyweight's `display()` method, which formats the intrinsic state. The extrinsic state is appended by the client or passed as a parameter to the flyweight method.

## Pseudocode — Cars Example

The following pseudocode shows how the Flyweight pattern works in a car configurator application:

```
// PSEUDOCODE — Flyweight Pattern in Cars Domain
// Participants labeled with [ROLE]

// [FLYWEIGHT] — Interface for all shared spec objects
interface CarSpec:
    method display(): string

// [CONCRETE FLYWEIGHT] — Engine specification
class EngineSpec implements CarSpec:
    fields:
        model: string = ""
        horsepower: int = 0
        displacement: float = 0.0
        cylinders: int = 0
        fuel_type: string = ""

    method EngineSpec(model, hp, disp, cyl, fuel):
        this.model = model
        this.horsepower = hp
        this.displacement = disp
        this.cylinders = cyl
        this.fuel_type = fuel

    method display() -> string:
        return format engine spec

// [CONCRETE FLYWEIGHT] — Wheel specification
class WheelSpec implements CarSpec:
    fields:
        diameter: float = 0.0
        width: int = 0
        material: string = ""
        brand: string = ""

    method WheelSpec(dia, w, mat, brand):
        this.diameter = dia
        this.width = w
        this.material = mat
        this.brand = brand

    method display() -> string:
        return format wheel spec

// [CONCRETE FLYWEIGHT] — Paint specification
class PaintSpec implements CarSpec:
    fields:
        code: string = ""
        name: string = ""
        finish: string = ""
        hex_color: string = ""

    method PaintSpec(code, name, finish, hex):
        this.code = code
        this.name = name
        this.finish = finish
        this.hex_color = hex

    method display() -> string:
        return format paint spec

// [FLYWEIGHT FACTORY] — Manages cached spec objects
class CarSpecFactory:
    fields:
        engines: Map<string, EngineSpec> = empty
        wheels: Map<string, WheelSpec> = empty
        paints: Map<string, PaintSpec> = empty

    method getEngine(model, hp, disp, cyl, fuel) -> EngineSpec:
        if engines contains model:
            return engines[model]
        engine = new EngineSpec(model, hp, disp, cyl, fuel)
        engines[model] = engine
        return engine

    method getWheel(key, dia, w, mat, brand) -> WheelSpec:
        if wheels contains key:
            return wheels[key]
        wheel = new WheelSpec(dia, w, mat, brand)
        wheels[key] = wheel
        return wheel

    method getPaint(code, name, finish, hex) -> PaintSpec:
        if paints contains code:
            return paints[code]
        paint = new PaintSpec(code, name, finish, hex)
        paints[code] = paint
        return paint

    method getStats() -> string:
        return format cache sizes

// [CLIENT] — Car configuration with extrinsic state
class CarConfiguration:
    fields:
        vin: string
        owner: string
        license_plate: string
        production_date: Date
        engine: EngineSpec
        wheels: WheelSpec
        paint: PaintSpec

    method CarConfiguration(vin, owner, plate, date, engine, wheels, paint):
        this.vin = vin
        this.owner = owner
        // ... store extrinsic state
        this.engine = engine
        this.wheels = wheels
        this.paint = paint

    method display() -> string:
        return format extrinsic + delegate to flyweights

// Driver
factory = CarSpecFactory()
// Create flyweights (shared across cars)
engine2L = factory.getEngine("2.0L Turbo", 250, 2.0, 4, "Gasoline")
wheel18 = factory.getWheel("18x8", 18.0, 8, "Alloy", "Sport")
paintRed = factory.getPaint("R3", "Racing Red", "Gloss", "#FF0000")
// Create cars with shared specs
car1 = CarConfiguration("VIN001", "Alice", "ABC-123", "2026-01-15", engine2L, wheel18, paintRed)
car2 = CarConfiguration("VIN002", "Bob", "XYZ-789", "2026-02-20", engine2L, wheel18, paintRed)
print(car1.display())
print(car2.display())
print(factory.getStats())
```

The Flyweight pattern's sharing workflow unfolds across nine distinct steps:

1. **Client requests a flyweight** — When a customer configures a car, the system needs an `EngineSpec` for the selected engine. The client calls `flyweightFactory.getEngine("2.0L Turbo", 250, 2.0, 4, "Gasoline")`. The client passes the unique key and the construction parameters for the flyweight. The factory's interface is designed so that the client does not need to know whether the flyweight will be created or retrieved from cache.

2. **Factory checks the cache** — The `FlyweightFactory` looks up the key `"2.0L Turbo"` in its internal `Map<string, EngineSpec>`. The cache check is an O(1) hash map lookup. If the key exists, the factory returns the existing flyweight without creating a new one. If the key does not exist, the factory proceeds to create a new flyweight.

3. **Cache miss: factory creates a new flyweight** — On the first request for the 2.0L Turbo engine, the cache does not contain the key. The factory constructs a new `EngineSpec` object with the provided parameters: 250 horsepower, 2.0L displacement, 4 cylinders, gasoline fuel. The new flyweight is fully initialized and immutable. The factory stores the flyweight in the cache under the key `"2.0L Turbo"`.

4. **Factory returns the flyweight to the client** — The factory returns the `EngineSpec` reference to the first `CarConfiguration` object. The client stores this reference alongside its own extrinsic state (VIN, owner, license plate, production date). The first car is now configured with the shared engine spec.

5. **Second client requests the same flyweight** — A different `CarConfiguration` object (for a different customer's car) requests the same engine via the same factory call. The second car also needs the 2.0L Turbo engine. The client does not know that the engine already exists in the cache — it simply makes the same request with the same parameters.

6. **Cache hit: factory returns existing flyweight** — The factory performs a cache lookup and finds the `EngineSpec` for `"2.0L Turbo"` that was created in step 3. Instead of creating a new `EngineSpec` object, the factory returns the existing one. No new memory is allocated for the engine spec. The second car now shares the same `EngineSpec` object as the first car.

7. **Factory returns the shared flyweight** — The factory returns the same `EngineSpec` reference to the second car. Both car configurations now point to the same flyweight object in memory. This is the key insight of the Flyweight pattern: the two cars are distinct objects, but they share the underlying spec data.

8. **Both cars hold the same flyweight reference** — `CarConfiguration` 1 (Alice, VIN001) and `CarConfiguration` 2 (Bob, VIN002) both hold a reference to the same `EngineSpec` object. The intrinsic state (engine model, horsepower, displacement, cylinders, fuel type) is stored once. The extrinsic state (VIN, owner, license plate, production date) is stored separately in each car. The memory savings are proportional to the number of cars that share the same engine spec.

9. **Display delegates to flyweight** — When either car's `display()` method is called, it formats the extrinsic state directly and delegates the spec formatting to the flyweight's `display()` method. The flyweight formats its intrinsic state without needing to know which car is calling it. The output for each car includes both the unique extrinsic data (VIN, owner) and the shared spec data (engine model, horsepower) from the flyweight.

The same process repeats for wheels and paint. Each `CarConfiguration` holds three flyweight references (engine, wheels, paint) plus its own extrinsic state. With 100,000 cars, 50 engine variants, 30 wheel types, and 40 paint colors, the total number of spec objects is 120 (50 + 30 + 40) instead of 300,000. The memory savings are substantial: 99.96% reduction in spec object count.

## When to Use

- **The application creates a large number of objects that consume most of the available memory** — If your car configurator creates hundreds of thousands of `CarConfiguration` objects, and each one stores its own copy of spec data, the application will run out of memory or trigger excessive garbage collection. The Flyweight pattern reduces the object count for spec data to the number of unique configurations, which is orders of magnitude smaller than the number of car instances.

- **The objects share a large amount of intrinsic state that can be factored out** — In the car configurator, every car configuration includes engine specs, wheel specs, and paint specs. These specs are identical across many cars. The amount of shared data (engine model, horsepower, wheel diameter, paint code, etc.) is large relative to the unique per-car data (VIN, owner, license plate). The Flyweight pattern is most effective when the intrinsic state is significantly larger than the extrinsic state.

- **The object identity is determined by the combination of intrinsic state** — If two cars with the same engine, wheels, and paint are considered to have the same "spec profile," the Flyweight pattern provides a natural representation. The spec profile is the flyweight, and the per-car details are extrinsic. The factory cache ensures that the spec profile is a canonical representation, so reference equality works for spec comparison.

- **The application needs to update shared data consistently** — When the manufacturer updates the 2.0L Turbo engine's horsepower from 250 to 260, updating the single `EngineSpec` flyweight immediately affects all cars that reference it. No iteration over all car configurations is needed. The update is O(1) instead of O(N).

- **The application serializes or transmits objects over the network** — Sharing flyweights reduces serialization payload size because the spec data is stored once and referenced by key. When transmitting car configurations over the network, only the flyweight keys need to be sent, not the full spec data. The receiving end reconstructs the flyweights from its own factory.

- **Performance profiling confirms that object allocation is a bottleneck** — The Flyweight pattern reduces object allocation count, which reduces GC pressure and improves throughput. In a high-throughput car configurator API, reduced allocation means lower latency per request and higher throughput under load.

- **The intrinsic state is immutable** — Flyweights must be immutable because they are shared across multiple clients. If the intrinsic state changes over time, sharing becomes unsafe. The Cars domain naturally satisfies this constraint: engine specs, wheel specs, and paint specs do not change once defined.

## When NOT to Use

- **The number of objects is small** — If the car configurator creates only a few hundred car configurations, the memory savings from sharing flyweights is negligible. The overhead of the factory, the cache, and the indirection is not justified. Use simple value objects or structs instead.

- **Objects have little or no intrinsic state** — If the car configuration has only extrinsic state (every car is completely unique with no shared specs), the Flyweight pattern provides no benefit. The factory would create one flyweight per car, which is the same as creating the objects directly, but with extra indirection overhead.

- **Intrinsic state is frequently mutated** — If engine specs change frequently and retroactively affect all cars, the flyweight cannot be shared safely without synchronization. Every mutation of a shared flyweight must be visible to all threads, which requires locks or atomic operations. In this case, consider using immutable flyweights and replacing them atomically in the factory.

- **The cost of looking up flyweights exceeds the cost of creating new objects** — If the factory's cache lookup is expensive (e.g., cache misses are common, key computation is slow, or the cache is distributed across a network), the overhead of the Flyweight pattern may outweigh the memory savings. The pattern is only beneficial when cache hit rates are high.

- **Memory is not a constraint** — If the application runs on a server with ample memory and the number of car configurations is bounded, the Flyweight pattern's complexity and indirection are not worth the memory savings. Simpler designs with direct object composition are preferable.

- **The application requires unique object identity for every instance** — If the system must distinguish between two cars with the same specs by something other than extrinsic state (e.g., if each car must have a unique spec object for auditing or tracking), the Flyweight pattern breaks this requirement. Each car must have its own spec objects, even if they are structurally identical.

- **Thread safety introduces unacceptable complexity** — In a multi-threaded car configurator, the factory must be thread-safe. If the factory is accessed concurrently, cache operations must be synchronized. This can become a bottleneck if the factory is under heavy load. In such cases, consider using concurrent hash maps or lock-free data structures.

## Real-World Occurrences

1. **Java String Interning** — Java's `String.intern()` method is a classic Flyweight pattern implementation. The JVM maintains a string constant pool (the flyweight factory) that stores unique string literals. When a string is interned, the JVM checks the pool and returns the canonical instance if it exists, or adds the string to the pool if it does not. This is why `"hello" == "hello"` is `true` in Java — both literals reference the same interned string object. The Flyweight pattern here reduces memory usage for strings that appear frequently in application code.

2. **Integer Cache in Java and .NET** — Java's `Integer.valueOf(int)` caches `Integer` objects for values in the range -128 to 127 (configurable). When you call `Integer.valueOf(100)`, the JVM returns a cached flyweight object rather than creating a new `Integer` instance. Similarly, .NET's `Integer` cache stores values from -128 to 127. This is a textbook Flyweight pattern where the integer value is the intrinsic state and the factory returns cached instances for frequently used values.

3. **Text Rendering Systems** — In document editors and web browsers, characters in a font are flyweights. A single character glyph object (e.g., the letter 'A' in 12pt Arial) is shared across all occurrences of that character in the document. The extrinsic state — the character's position on the page, its color, its size — is stored separately for each occurrence. This is the original GoF example of the Flyweight pattern, and it remains relevant in modern text rendering engines like HarfBuzz, ICU, and Skia.

4. **Game Development Entity Systems** — Game engines use the Flyweight pattern extensively to share 3D models, textures, animation data, and sound effects across multiple game entities. A single `Model` object (e.g., a tree mesh) is shared across thousands of tree instances, each with its own extrinsic state (position, rotation, scale, health). The Unity engine's `ScriptableObject` system and Unreal Engine's data assets are modern examples of flyweight factories in game development.

5. **Database Connection Pooling** — While connection pooling is primarily an Object Pool pattern, it shares the Flyweight principle of reusing shared, expensive-to-create objects. The database connection configuration (host, port, database name, credentials) is intrinsic state shared across all connections in the pool. The extrinsic state (transaction context, active statement, session variables) is per-connection.

6. **CSS Class Sharing in Web Browsers** — When a web page uses CSS classes, the browser stores the style rules once per class and shares them across all DOM elements that reference that class. Each DOM element stores only its class name (extrinsic), not the full set of style rules. The browser's CSS engine acts as a flyweight factory, resolving class names to shared style objects. This is why CSS class-based styling is more memory-efficient than inline styles.

7. **Apache Spark's Broadcast Variables** — In distributed computing, Spark's broadcast variables allow the driver to share a read-only object (intrinsic state) across all worker nodes without serializing it for every task. The broadcast variable is a flyweight: the data is stored once per worker node, and all tasks on that node share the same reference. Without broadcast variables, the same data would be serialized and deserialized for every task, wasting CPU and network bandwidth.

8. **ORM Identity Maps** — Object-Relational Mapping frameworks like Hibernate (Java), Entity Framework (C#), and SQLAlchemy (Python) use identity maps to ensure that each database row maps to exactly one in-memory object per session. The identity map is a flyweight factory: when the ORM loads a row, it checks the identity map first. If the object exists, it returns the cached instance. If not, it creates a new one and stores it in the map. This prevents duplicate objects for the same database row and ensures that changes to the object are reflected consistently across the session.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Singleton** | Singleton ensures exactly one instance of a class globally. Flyweight creates multiple shared instances, each representing a unique combination of intrinsic state. | Choose Singleton for global configuration objects (e.g., a single `CarConfiguratorSettings`). Choose Flyweight when you need many shared objects keyed by their intrinsic state. |
| **Object Pool** | Object Pool reuses objects that are expensive to create but does not share them across clients simultaneously. Flyweight shares objects across clients concurrently. | Choose Object Pool for reusable resources like database connections or thread pools. Choose Flyweight for read-only data that can be shared simultaneously across many clients. |
| **Proxy** | Proxy provides a surrogate or placeholder for another object to control access. Flyweight provides a shared object to reduce memory usage. | Choose Proxy when you need lazy loading, access control, or logging. Choose Flyweight when you need to reduce memory usage by sharing common data. |
| **Composite** | Composite lets clients treat individual objects and compositions uniformly. Flyweight enables sharing of fine-grained objects. | Choose Composite when you need tree structures with part-whole hierarchies. Choose Flyweight when you need to share data across many small objects. |
| **Facade** | Facade provides a simplified interface to a complex subsystem. Flyweight provides a mechanism for sharing fine-grained objects. | Choose Facade when you need to simplify client interaction with a complex system. Choose Flyweight when you need to reduce memory usage through object sharing. |

The key distinction between Flyweight and other structural patterns is its focus on memory optimization through object sharing. Singleton shares one object globally. Object Pool reuses objects but does not share them concurrently. Proxy controls access to a single object. Composite builds tree structures. Flyweight specifically targets the scenario where a large number of objects can share a smaller set of intrinsic data.

## Implementation Considerations

Implementing the Flyweight pattern requires attention to several design decisions that affect correctness, performance, and maintainability.

### Intrinsic vs Extrinsic State Separation

The most critical design decision is determining which state is intrinsic (shared) and which is extrinsic (per-client). The dividing line is not always obvious and depends on the application's access patterns.

**Intrinsic state** should be:
- Invariant across all uses of the flyweight
- Immutable after creation
- Relatively expensive to store per-instance
- Keyed by a natural identifier (engine model name, paint code)

**Extrinsic state** should be:
- Varies per client or per use
- Potentially mutable
- Relatively small compared to intrinsic state
- Computed or stored by the client

In the Cars domain, the separation is natural: engine specs, wheel specs, and paint specs are intrinsic because they are defined by the manufacturer and do not change per car. VIN, owner, license plate, and production date are extrinsic because they are unique to each car.

### Flyweight Immutability

Flyweights must be effectively immutable. If a shared flyweight's state changes, all clients that reference it are affected. This can lead to subtle bugs where modifying one car's engine spec inadvertently changes the spec for all other cars that share the same engine flyweight.

To enforce immutability:
- Make all fields `final` (Java), `readonly` (C#), or frozen (JavaScript `Object.freeze()`)
- Do not expose setter methods
- Return defensive copies or immutable views of any collection fields
- In languages like Python, use `@property` with read-only accessors and store values in private attributes

If the intrinsic state must change (e.g., the manufacturer updates engine specifications), the correct approach is to create a new flyweight and update the factory. The old flyweight continues to be used by existing cars, and new cars use the new flyweight. This avoids mutating shared state.

### Thread Safety

In a multi-threaded environment, the flyweight factory must be thread-safe. The factory's cache is a shared mutable data structure that can be accessed concurrently by multiple threads.

Thread safety strategies:
- **Synchronized access** — Use `synchronized` methods (Java) or `lock` statements (C#) to protect cache access. This is simple but can become a bottleneck under high concurrency.
- **Concurrent collections** — Use `ConcurrentHashMap` (Java), `ConcurrentDictionary` (C#), or equivalent in other languages. These provide thread-safe access without explicit synchronization.
- **Double-checked locking** — Check the cache, then synchronize only on cache miss. This reduces contention for cache hits but requires careful implementation to avoid race conditions.
- **Immutable factory** — Create the flyweight cache at startup with all known flyweight objects. The factory becomes read-only, eliminating the need for synchronization entirely.

### Flyweight Identity

When flyweights are shared, object identity becomes meaningful. Two clients that reference the same flyweight key reference the same object in memory. This means reference equality (`==` in Java, `===` in JavaScript, `is` in Python) can be used to check whether two car configurations use the same engine spec.

However, this also means that the system must not rely on object identity for things that should be distinct. If two cars have the same engine model but different serial numbers, the engine serial number should be part of the extrinsic state, not the intrinsic state.

### Factory Cache Size

The flyweight factory's cache grows with the number of unique flyweight keys. If the domain has a bounded set of possible values (e.g., 50 engine models, 30 wheel types, 40 paint colors), the cache size is bounded and predictable. If the keys are unbounded (e.g., user-defined custom paint colors), the cache can grow indefinitely, and you may need to implement eviction policies.

Cache eviction strategies:
- **LRU (Least Recently Used)** — Evict flyweights that have not been accessed recently. Useful when some flyweights become obsolete over time.
- **Weak references** — Use weak references in the cache so that flyweights are garbage-collected when no clients reference them. Java's `WeakHashMap` and Python's `weakref.WeakValueDictionary` support this.
- **TTL (Time To Live)** — Evict flyweights after a certain time period. Useful when spec data becomes stale (e.g., when the manufacturer releases new model year specs).
- **No eviction** — If the number of unique flyweights is bounded and small, no eviction is needed. The cache holds all flyweights for the lifetime of the application.

## Benefits and Trade-offs

### Benefits

- **Reduced memory usage** — The most significant benefit. With 100,000 car configurations, 50 engine variants, 30 wheel types, and 40 paint colors, the total number of spec objects is reduced from 300,000 to 120. This is a 99.96% reduction in spec object count.

- **Reduced allocation and GC pressure** — Fewer objects means fewer allocations and less work for the garbage collector. In a high-throughput car configurator API, this translates to lower latency and higher throughput.

- **Consistent updates** — Updating a single flyweight affects all clients that reference it. When the manufacturer updates an engine's horsepower, the change is immediately visible across all car configurations that use that engine.

- **Canonical representation** — The factory ensures that each unique spec is represented by exactly one object. This enables reference equality checks for spec comparison. Two cars with the same engine can be identified by comparing engine flyweight references rather than performing field-by-field comparison.

- **Natural separation of concerns** — The intrinsic/extrinsic state separation aligns with the domain's conceptual model. Spec data is managed by the manufacturer and shared across all cars. Identification data is per-car and managed by the customer.

### Trade-offs

- **Increased complexity** — The Flyweight pattern adds a factory, a cache, and an interface. The separation of intrinsic and extrinsic state must be carefully designed and maintained. Developers must understand which state goes where.

- **Indirection overhead** — Accessing a flyweight requires a cache lookup (or a hash map lookup on every access). This adds a small performance cost compared to direct field access. In systems where the cache hit rate is high, this overhead is negligible.

- **Thread safety burden** — The factory must be thread-safe in concurrent environments. This adds synchronization overhead or requires concurrent collections.

- **Immutable constraint** — Flyweights must be immutable, which may conflict with other design requirements. If the intrinsic state must change, new flyweights must be created and the cache must be updated.

- **Debugging complexity** — When debugging, it can be confusing that two car configurations share the same spec object. Developers must understand that the flyweight is shared and that modifying it would affect multiple cars.

- **Premature optimization risk** — The Flyweight pattern is a memory optimization technique. Applying it before identifying a real memory bottleneck adds complexity without benefit. Profile first, optimize second.

## When Flyweight Fails in Practice

Despite its memory benefits, the Flyweight pattern can create problems in several scenarios:

- **Mutable intrinsic state** — If the intrinsic state is accidentally mutated, every client that references the flyweight is affected. In the Cars domain, if a developer adds a setter to `EngineSpec` and calls it, all cars with that engine see the change. This can corrupt data across the entire application.

- **Cache pollution** — If the factory key is not unique, different intrinsic states can map to the same key, causing incorrect sharing. For example, if two different engine models have the same model name string but different internal configurations, the factory would incorrectly return the first engine for both.

- **Memory leak via cache** — If the cache holds references to flyweights that are no longer needed, the memory cannot be reclaimed. In the Cars domain, this is usually not a problem because the set of engine specs is bounded. However, if the system allows user-defined custom specs, the cache can grow unboundedly.

- **Extrinsic state explosion** — If the separation between intrinsic and extrinsic state is poorly designed, the extrinsic state can become as large as the intrinsic state, defeating the purpose of the pattern. For example, if each car's paint spec includes a custom paint formula that is unique to that car, the paint spec should not be a flyweight.

## Performance Analysis

When evaluating whether the Flyweight pattern is appropriate for a given scenario, it is helpful to quantify the memory savings and the overhead. The following analysis uses the car configurator as a running example.

### Memory Savings Calculation

Let:
- `N` = number of car configurations (clients)
- `E` = number of unique engine specs
- `W` = number of unique wheel specs
- `P` = number of unique paint specs
- `S_int` = size of each intrinsic state object (bytes)
- `S_ext` = size of each extrinsic state object (bytes)
- `S_ref` = size of a reference/pointer (bytes)

**Without Flyweight** (each car stores its own copies of all specs):
```
Total memory = N * (S_int + S_ext) per spec type
For all three spec types: 3 * N * (S_int + S_ext)
```

**With Flyweight** (specs shared, cars store references):
```
Flyweight memory = (E + W + P) * S_int
Car configurations memory = N * (S_ext + 3 * S_ref)
Total = (E + W + P) * S_int + N * (S_ext + 3 * S_ref)
```

**Memory savings**:
```
Savings = 3 * N * S_int - (E + W + P) * S_int - 3 * N * S_ref
        = S_int * (3N - E - W - P) - 3 * N * S_ref
```

With realistic values for a large car configurator:
- `N = 1,000,000` car configurations
- `E = 50`, `W = 30`, `P = 40` unique specs
- `S_int = 200` bytes per spec object (strings, numbers)
- `S_ext = 100` bytes per car (VIN, owner, plate, date)
- `S_ref = 8` bytes (64-bit pointer)

**Without Flyweight**: `3 * 1,000,000 * (200 + 100) = 900 MB`
**With Flyweight**: `(50 + 30 + 40) * 200 + 1,000,000 * (100 + 24) = 24 KB + 124 MB = ~124 MB`
**Savings**: `900 MB - 124 MB = 776 MB` (86% reduction)

The savings increase linearly with the number of car configurations. At 10 million configurations, the savings exceed 7.7 GB.

### Cache Hit Rate

The cache hit rate determines the performance benefit of the Flyweight pattern. In the Cars domain, the cache hit rate is typically high because the number of unique spec combinations is much smaller than the number of car configurations.

If the factory processes `N` requests for flyweights and there are `U` unique flyweights, the cache hit rate is `(N - U) / N`. With `N = 1,000,000` and `U = 120` (50 + 30 + 40), the hit rate is `(1,000,000 - 120) / 1,000,000 = 99.988%`. This means 999,880 out of every 1,000,000 requests are cache hits, and only 120 requests require creating a new flyweight.

### Overhead

The Flyweight pattern introduces overhead in the form of:
- **Factory lookup**: Each `getEngine()`, `getWheel()`, or `getPaint()` call performs a hash map lookup. With a well-tuned hash map, this is O(1) amortized and adds negligible overhead (typically less than 100 nanoseconds per lookup).
- **Memory for cache**: The factory's hash maps store pointers to all unique flyweights. With 120 unique flyweights and 8-byte pointers, the cache overhead is approximately 960 bytes plus hash map bookkeeping (typically a few KB total).
- **Reference pointer storage**: Each car configuration stores 3 references (24 bytes on 64-bit) instead of inline spec data. This is a small cost compared to the memory savings.

The overhead is trivially small compared to the memory savings, making the Flyweight pattern one of the most cost-effective memory optimization techniques available.

### When the Numbers Change

The Flyweight pattern becomes less beneficial when:
- The ratio `N / U` is small (few clients per unique flyweight)
- The intrinsic state is small relative to the reference pointer size
- The extrinsic state dominates the total memory per client

For the Cars domain, `N / U` is typically very large (thousands or millions of car configurations per unique spec combination), and the intrinsic state (engine model strings, paint codes, wheel dimensions) is significantly larger than a pointer reference. This makes the Cars domain an ideal candidate for the Flyweight pattern.

## Summary

The Flyweight pattern is a structural design pattern that reduces memory usage by sharing common data across multiple objects. It separates intrinsic state (shared, context-independent data) from extrinsic state (unique, context-dependent data). The intrinsic state is stored in flyweight objects that are created and managed by a flyweight factory. The factory ensures that each unique combination of intrinsic state is represented by exactly one flyweight object. Clients reference the flyweight and supply the extrinsic state separately.

In the Cars domain, the Flyweight pattern enables a car configurator to represent thousands or millions of car configurations using a small number of shared spec objects. Engine specs, wheel specs, and paint specs are flyweights, shared across all cars that use them. The per-car extrinsic state (VIN, owner, license plate, production date) is stored in each car configuration object. The result is a system that can handle millions of car configurations without running out of memory, while maintaining consistent spec data across all cars.

The key design decisions when implementing the Flyweight pattern include: correctly separating intrinsic and extrinsic state, ensuring flyweight immutability, making the factory thread-safe, and choosing an appropriate cache eviction strategy. The pattern is most effective when the number of unique flyweights is small relative to the number of clients, and when the intrinsic state is significantly larger than the extrinsic state.
