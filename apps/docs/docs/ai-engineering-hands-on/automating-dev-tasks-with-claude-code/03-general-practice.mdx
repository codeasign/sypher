These exercises progress from basic automation patterns to complex multi-step pipelines. Each exercise builds on the previous one, so work through them in order.

## Tier 1 -- Beginner

These exercises focus on the fundamental mechanics of running Claude Code in automated contexts.

### Exercise 1: Run Claude Code with a One-Shot Prompt

Write a script that invokes Claude Code with a single prompt to review a file and outputs the results in JSON format.

**Task:** Create a shell script called `review-file.sh` that accepts a file path as an argument, invokes Claude Code to review that file, and saves the review to a JSON file.

**Hint:** Use the `-p` flag for the prompt and `--output-format json` for structured output. Pass the file path as part of the prompt string. Use `$1` to access the first argument in the shell script.

**Solution:**

```bash
#!/bin/bash
# review-file.sh -- Review a file using Claude Code and save results as JSON

FILE_PATH="$1"

if [ -z "$FILE_PATH" ]; then
  echo "Usage: ./review-file.sh <file-path>"
  exit 1
fi

if [ ! -f "$FILE_PATH" ]; then
  echo "Error: File '$FILE_PATH' not found"
  exit 1
fi

OUTPUT_FILE="review-$(basename "$FILE_PATH")-$(date +%s).json"

claude -p "Review the file $FILE_PATH. Check for:
1. Code quality issues
2. Missing error handling
3. Potential bugs
4. Style violations

Output a JSON object with keys: summary, issues (array of objects with severity, line, description), recommendations (array of strings)" \
  --dangerously-skip-permissions \
  --allowedTools Read \
  --output-format json \
  --shutdown-timeout 120000 > "$OUTPUT_FILE" 2>&1

echo "Review saved to $OUTPUT_FILE"
```

Make the script executable and test it:

```bash
chmod +x review-file.sh
./review-file.sh src/main.py
```

Expected output: A JSON file is created with the review results. The file contains a summary, an array of issues with severity levels and line numbers, and an array of recommendations.

### Exercise 2: Generate a README from a Codebase

Write a script that uses Claude Code to generate a README file for a project directory.

**Task:** Create a shell script called `generate-readme.sh` that takes a directory path, asks Claude Code to analyze the directory and generate a README, and saves the result to `README.md` in that directory.

**Hint:** Tell Claude Code to read the key files in the directory first (like `package.json`, `setup.py`, or `requirements.txt`) to understand the project. Use the `Read` and `Write` tools.

**Solution:**

```bash
#!/bin/bash
# generate-readme.sh -- Generate a README for a project directory

PROJECT_DIR="${1:-.}"

if [ ! -d "$PROJECT_DIR" ]; then
  echo "Error: Directory '$PROJECT_DIR' not found"
  exit 1
fi

cd "$PROJECT_DIR" || exit 1

claude -p "Analyze this project directory and generate a comprehensive README.md file. Include:
1. Project name and description (infer from code)
2. Installation instructions
3. Usage examples
4. API documentation (if applicable)
5. Configuration options
6. Development setup instructions

Write the README to README.md using the Write tool." \
  --dangerously-skip-permissions \
  --allowedTools Read,Write,Grep,Glob \
  --output-format json \
  --shutdown-timeout 180000 > readme-generation-log.json 2>&1

echo "README generation complete. Check README.md"
```

Expected output: A `README.md` file is created in the project directory with a comprehensive description, installation instructions, and usage examples. The generation log is saved to `readme-generation-log.json`.

### Exercise 3: Batch Process Multiple Files

Create a script that processes multiple files in a directory, running Claude Code on each one and collecting the results.

**Task:** Create a shell script called `batch-review.sh` that iterates over all Python files in a directory, reviews each one with Claude Code, and aggregates the results into a single summary report.

**Hint:** Use a loop in the shell script to iterate over files. Give each Claude Code invocation a unique output file name. After all reviews are complete, use a final Claude Code invocation to summarize the aggregate results.

**Solution:**

```bash
#!/bin/bash
# batch-review.sh -- Review all Python files in a directory

TARGET_DIR="${1:-src}"
REPORT_DIR="batch-reviews-$(date +%s)"
SUMMARY_FILE="$REPORT_DIR/summary.json"

mkdir -p "$REPORT_DIR"

for FILE in "$TARGET_DIR"/*.py; do
  if [ ! -f "$FILE" ]; then
    echo "No Python files found in $TARGET_DIR"
    exit 0
  fi

  BASENAME=$(basename "$FILE" .py)
  OUTPUT_FILE="$REPORT_DIR/review-$BASENAME.json"

  echo "Reviewing $FILE..."

  claude -p "Review the Python file $FILE. Output a JSON object with:
  - file: the filename
  - issues: array of objects with severity (low/medium/high), line, description
  - quality_score: integer from 1 to 10
  - summary: one-sentence overview" \
    --dangerously-skip-permissions \
    --allowedTools Read \
    --output-format json \
    --shutdown-timeout 60000 > "$OUTPUT_FILE" 2>&1

  echo "  Review saved to $OUTPUT_FILE"
done

echo "Generating aggregate summary..."

claude -p "Read all JSON review files in the directory $REPORT_DIR. Aggregate the results into a single JSON summary with:
  - total_files: count
  - average_quality: mean quality score
  - total_issues: count
  - issues_by_severity: breakdown by severity
  - top_recommendations: top 5 most important recommendations across all files

Save the summary to $SUMMARY_FILE using the Write tool." \
  --dangerously-skip-permissions \
  --allowedTools Read,Write,Grep \
  --output-format json \
  --shutdown-timeout 180000 > "$REPORT_DIR/aggregation-log.json" 2>&1

echo "Batch review complete. Results in $REPORT_DIR/"
echo "Summary: $SUMMARY_FILE"
```

Expected output: A directory called `batch-reviews-<timestamp>/` containing individual review files for each Python file and an aggregate summary. The summary includes counts, average quality scores, and prioritized recommendations.

### Exercise 4: Run Claude Code in Dry-Run Mode

Create a script that uses Claude Code with read-only tools to simulate a review without making any changes.

**Task:** Create a shell script called `dry-run-review.sh` that runs Claude Code with only read tools and produces a report of what changes it would recommend, but does not actually make any changes.

**Hint:** Use `--allowedTools Read,Grep,Glob` to restrict Claude Code to read-only tools. Ask Claude Code to describe what it would change but not to use any write tools.

**Solution:**

```bash
#!/bin/bash
# dry-run-review.sh -- Simulate a review without making changes

TARGET="${1:-.}"

if [ ! -d "$TARGET" ] && [ ! -f "$TARGET" ]; then
  echo "Usage: ./dry-run-review.sh <file-or-directory>"
  exit 1
fi

claude -p "Perform a dry-run review of $TARGET. Do NOT make any changes. Instead, produce a JSON report describing:
1. What changes you would recommend
2. Which files would be affected
3. What the impact of each change would be
4. Priority level for each change (critical/important/nice-to-have)

Output a JSON object with keys: recommendations (array of objects with file, change_description, impact, priority), summary, estimated_effort" \
  --dangerously-skip-permissions \
  --allowedTools Read,Grep,Glob \
  --output-format json \
  --shutdown-timeout 120000 > dry-run-report.json 2>&1

echo "Dry-run review complete. Report saved to dry-run-report.json"
echo "No changes were made to any files."
```

Expected output: A JSON report file with recommendations, affected files, and priority levels. No files in the codebase are modified.

## Tier 2 -- Intermediate

These exercises combine Claude Code automation with concepts from other AI engineering topics.

### Exercise 5: Prompt Template Pipeline

Combine prompt engineering patterns with Claude Code automation. Create a system that uses different prompt templates for different review scenarios.

**Task:** Create a directory of prompt templates and a script that selects the appropriate template based on the file type being reviewed.

**Hint:** Store prompt templates as separate text files. Use a case statement in the shell script to select the template based on the file extension. Inject the file path into the template before passing it to Claude Code.

**Solution:**

First, create a templates directory:

```bash
mkdir -p .claude-templates
```

Create a template for Python files (`.claude-templates/python-review.txt`):

```txt
Review this Python file: {FILE_PATH}

Focus on:
1. PEP 8 compliance
2. Type hint usage
3. Exception handling
4. Function docstrings
5. Import organization

Output a JSON object with keys: summary, pep8_issues (array), type_hint_issues (array), error_handling_issues (array), overall_score (integer 1-10)
```

Create a template for JavaScript files (`.claude-templates/javascript-review.txt`):

```txt
Review this JavaScript file: {FILE_PATH}

Focus on:
1. ESLint rule compliance
2. Async/await usage
3. Error handling in promises
4. Variable scoping
5. Function documentation

Output a JSON object with keys: summary, lint_issues (array), async_issues (array), scoping_issues (array), overall_score (integer 1-10)
```

Create the script (`template-review.sh`):

```bash
#!/bin/bash
# template-review.sh -- Use prompt templates for file review

FILE_PATH="$1"

if [ -z "$FILE_PATH" ] || [ ! -f "$FILE_PATH" ]; then
  echo "Usage: ./template-review.sh <file-path>"
  exit 1
fi

EXTENSION="${FILE_PATH##*.}"

case "$EXTENSION" in
  py)
    TEMPLATE=".claude-templates/python-review.txt"
    ;;
  js|jsx|ts|tsx)
    TEMPLATE=".claude-templates/javascript-review.txt"
    ;;
  *)
    echo "No template found for .$EXTENSION files"
    echo "Using generic template"
    TEMPLATE=".claude-templates/generic-review.txt"
    ;;
esac

if [ ! -f "$TEMPLATE" ]; then
  echo "Error: Template '$TEMPLATE' not found"
  exit 1
fi

PROMPT=$(sed "s|{FILE_PATH}|$FILE_PATH|g" "$TEMPLATE")

claude -p "$PROMPT" \
  --dangerously-skip-permissions \
  --allowedTools Read \
  --output-format json \
  --shutdown-timeout 120000 > "review-$(basename "$FILE_PATH").json" 2>&1

echo "Review saved to review-$(basename "$FILE_PATH").json"
```

Expected output: The script selects the correct prompt template based on the file extension and generates a review that is specific to the language. The Python review focuses on PEP 8 and type hints, while the JavaScript review focuses on ESLint rules and async patterns.

### Exercise 6: Structured Output Processing Pipeline

Combine structured output prompting with CI automation. Create a pipeline that processes Claude Code JSON output and generates different artifacts based on the content.

**Task:** Create a script that runs Claude Code to review a codebase, then processes the JSON output to generate a Markdown report, a CSV issue tracker, and a summary for Slack.

**Hint:** Use `jq` to extract different parts of the JSON output. Use shell redirection to create each output format. The Markdown report is for the team wiki, the CSV is for the issue tracker, and the summary is for Slack notifications.

**Solution:**

```bash
#!/bin/bash
# process-review-output.sh -- Generate multiple artifacts from review JSON

REVIEW_INPUT="${1:-claude-review-output.json}"
REPORT_DIR="review-artifacts-$(date +%s)"

mkdir -p "$REPORT_DIR"

if [ ! -f "$REVIEW_INPUT" ]; then
  echo "Error: Review input file '$REVIEW_INPUT' not found"
  echo "Run Claude Code review first: claude -p 'review code' --output-format json > $REVIEW_INPUT"
  exit 1
fi

# Generate Markdown report
echo "# Code Review Report" > "$REPORT_DIR/report.md"
echo "Generated: $(date)" >> "$REPORT_DIR/report.md"
echo "" >> "$REPORT_DIR/report.md"
jq -r '.summary // "No summary"' "$REVIEW_INPUT" >> "$REPORT_DIR/report.md"
echo "" >> "$REPORT_DIR/report.md"
echo "## Issues Found" >> "$REPORT_DIR/report.md"
jq -r '.issues[]? | "- **\(.severity):** \(.description) (line \(.line))"' "$REVIEW_INPUT" >> "$REPORT_DIR/report.md" 2>/dev/null
echo "" >> "$REPORT_DIR/report.md"
echo "## Recommendations" >> "$REPORT_DIR/report.md"
jq -r '.recommendations[]? | "- \(.)"' "$REVIEW_INPUT" >> "$REPORT_DIR/report.md" 2>/dev/null

echo "Markdown report generated: $REPORT_DIR/report.md"

# Generate CSV issue tracker
echo "severity,file,line,description" > "$REPORT_DIR/issues.csv"
jq -r '.issues[]? | [.severity, .file, (.line | tostring), .description] | @csv' "$REVIEW_INPUT" >> "$REPORT_DIR/issues.csv" 2>/dev/null

echo "CSV issue tracker generated: $REPORT_DIR/issues.csv"

# Generate Slack summary
echo "Claude Code Review Summary:" > "$REPORT_DIR/slack-summary.txt
jq -r '"Files reviewed: " + (.files_reviewed | length | tostring)' "$REVIEW_INPUT" >> "$REPORT_DIR/slack-summary.txt" 2>/dev/null
jq -r '"Issues found: " + (.issues | length | tostring)' "$REVIEW_INPUT" >> "$REPORT_DIR/slack-summary.txt" 2>/dev/null
jq -r '.summary' "$REVIEW_INPUT" >> "$REPORT_DIR/slack-summary.txt" 2>/dev/null

echo "Slack summary generated: $REPORT_DIR/slack-summary.txt"
echo "All artifacts saved to $REPORT_DIR/"
```

Expected output: Three artifacts are generated from the single JSON review output: a Markdown report for the wiki, a CSV file for the issue tracker, and a plain-text summary for Slack notifications.

### Exercise 7: Multi-Step Automation with Guardrails

Combine loop safety patterns with Claude Code automation. Create a pipeline that runs Claude Code with a circuit breaker and maximum turn limit.

**Task:** Create a script that runs Claude Code in a loop with a maximum iteration count, checking the output after each iteration and stopping if the output indicates the task is complete or if a safety limit is reached.

**Hint:** Use a `while` loop with a counter. After each Claude Code invocation, check the JSON output for a `task_complete` field. Break the loop if complete, or if the counter exceeds the maximum. Use `CLAUDE_CODE_MAX_TURNS` to limit each individual invocation.

**Solution:**

```bash
#!/bin/bash
# safe-automation.sh -- Multi-step automation with guardrails

TASK="$1"
MAX_ITERATIONS="${2:-5}"
ITERATION=0
LOG_FILE="automation-log-$(date +%s).json"

if [ -z "$TASK" ]; then
  echo "Usage: ./safe-automation.sh <task-description> [max-iterations]"
  exit 1
fi

echo "Starting automation with max $MAX_ITERATIONS iterations"
echo "Task: $TASK"

while [ $ITERATION -lt $MAX_ITERATIONS ]; do
  ITERATION=$((ITERATION + 1))
  echo ""
  echo "=== Iteration $ITERATION of $MAX_ITERATIONS ==="

  OUTPUT_FILE="iteration-$ITERATION-output.json"

  CLAUDE_CODE_MAX_TURNS=20 \
  claude -p "$TASK

  After completing what you can, output a JSON object with:
  - status: complete or incomplete
  - progress: description of what was accomplished
  - remaining: what still needs to be done
  - task_complete: true or false
  - iteration: $ITERATION" \
    --dangerously-skip-permissions \
    --allowedTools Read,Edit,Write,Grep,Glob,Bash \
    --output-format json \
    --shutdown-timeout 120000 > "$OUTPUT_FILE" 2>&1

  TASK_COMPLETE=$(jq -r '.task_complete // false' "$OUTPUT_FILE")
  PROGRESS=$(jq -r '.progress // "No progress reported"' "$OUTPUT_FILE")
  REMAINING=$(jq -r '.remaining // "Unknown"' "$OUTPUT_FILE")

  echo "Progress: $PROGRESS"
  echo "Remaining: $REMAINING"
  echo "Task complete: $TASK_COMPLETE"

  # Save iteration to aggregate log
  echo "{\"iteration\": $ITERATION, \"task_complete\": $TASK_COMPLETE, \"progress\": \"$PROGRESS\", \"remaining\": \"$REMAINING\"}" >> "$LOG_FILE"

  if [ "$TASK_COMPLETE" = "true" ]; then
    echo ""
    echo "Task completed in $ITERATION iterations"
    break
  fi

  # Update the task for the next iteration based on remaining work
  TASK="Continue the remaining work: $REMAINING"
done

if [ $ITERATION -eq $MAX_ITERATIONS ] && [ "$TASK_COMPLETE" != "true" ]; then
  echo ""
  echo "WARNING: Maximum iterations ($MAX_ITERATIONS) reached without completion"
  echo "Remaining work: $(jq -r '.remaining // "Unknown"' "iteration-$MAX_ITERATIONS-output.json")"
  echo "Consider increasing max-iterations or refining the task"
fi

echo ""
echo "Automation log saved to $LOG_FILE"
```

Expected output: The script runs Claude Code in a loop, each time passing the remaining work from the previous iteration. It stops when the task is complete or when the maximum iteration count is reached. The aggregate log shows the progress of each iteration.

## Tier 3 -- Advanced

These exercises are open-ended scenarios that require you to design and build a complete automation solution.

### Exercise 8: Multi-Stage CI/CD Pipeline with Claude Code

Design a CI/CD pipeline that uses Claude Code at three different stages: pre-commit, PR review, and release.

**Scenario:** Your team has a monorepo with a Python backend and a JavaScript frontend. You need to automate quality checks at every stage of the development lifecycle.

**Requirements:**

- A pre-commit hook that uses Claude Code to check for secrets, large files, and formatting issues before allowing a commit
- A PR review workflow that runs Claude Code to review the full diff, generate tests, and update documentation
- A release workflow that uses Claude Code to generate release notes, update the changelog, and bump version numbers

**Considerations:**

- Each stage has different `--allowedTools` requirements
- The pre-commit hook must be fast (under 30 seconds) to avoid frustrating developers
- The PR review workflow must post results as PR comments
- The release workflow must create a GitHub release with auto-generated notes

**No single solution is provided for this exercise. Design your own solution and test it on a sample repository.**

### Exercise 9: Cron-Based Code Quality Monitoring

Design a system that runs Claude Code on a weekly schedule to monitor code quality trends across the entire codebase.

**Scenario:** Your team wants to track code quality metrics over time. Every Sunday night, a cron job runs Claude Code against the main branch and generates a quality report that is posted to the team Slack channel.

**Requirements:**

- A GitHub Actions workflow triggered by a cron schedule (weekly on Sunday)
- Claude Code analyzes the entire codebase for quality issues
- The output is compared with the previous week's report to show trends
- A summary is posted to Slack with highlights and regressions
- A detailed report is stored as a GitHub Pages artifact

**Considerations:**

- Full codebase analysis is expensive. How do you manage API costs?
- How do you handle the case where the analysis takes longer than the CI timeout?
- How do you store and compare historical data?

**No single solution is provided for this exercise. Design your own solution and test it on a sample repository.**

### Exercise 10: Multi-Repository Automation Coordinator

Design a system that coordinates Claude Code automation across multiple repositories.

**Scenario:** Your organization has 15 microservice repositories, each with its own CI pipeline. You need to run consistent code quality checks across all repositories and aggregate the results into a single dashboard.

**Requirements:**

- A central coordinator script that triggers Claude Code runs in each repository
- Consistent prompt templates and review criteria across all repositories
- An aggregated dashboard that shows quality metrics for each repository
- A notification system that alerts teams when their repository quality drops below a threshold

**Considerations:**

- How do you handle rate limits when running against 15 repositories simultaneously?
- How do you ensure consistent prompt templates across repositories?
- How do you handle repositories with different languages and frameworks?
- How do you store and serve the dashboard?

**No single solution is provided for this exercise. Design your own solution and test it on a sample set of repositories.**
