You arrive at your desk on Monday morning. Five pull requests are waiting for review, each one touching code you barely remember writing. The CI pipeline failed on three of them -- not because the logic was wrong, but because formatting was off, a docstring was missing, and one test file was never generated. You spend the next two hours clicking through diffs, running formatters, writing test stubs, and updating documentation that went out of date the moment the PR was opened. By the time you get to your actual feature work, it is almost lunch. This is not a one-time problem. This is every Monday. Every PR. Every release cycle.

The frustration compounds. You know that the code review checklist is the same every time: does it have tests? Are the docs updated? Does the code follow project conventions? Does the CI pass? You are manually checking boxes that a machine could check faster and more consistently. When you try to skip a step to save time, a bug slips through, and you spend twice as long debugging it later. The repetitive nature of these tasks leads to review fatigue, and your reviews become less thorough over time. You start missing things that matter because you are exhausted by the things that do not.

The cost is real. A developer who spends two hours per day on manual review and maintenance tasks loses over 40 hours per month -- an entire work week. Multiply that across a team of ten, and you are losing ten developer-weeks every month to tasks that follow predictable patterns. The solution is not to work faster or to be more disciplined. The solution is to automate the patterns so that you can focus on the decisions that require human judgment.

Consider the release day scenario. Your team has been working on a major feature for two weeks. The code is merged, the tests pass, but you still need to update the changelog, bump the version number, regenerate the API documentation, and write release notes. Each of these tasks is simple on its own, but together they consume an entire afternoon. One team member forgets to update the changelog, and the next release has incorrect version history. Another team member manually updates the version number but forgets to update the dependency file, and the build breaks. These are not bugs in the code. These are bugs in the process. Automation eliminates the process bugs by making the routine steps consistent and repeatable.

## The Concept

Automating development tasks with Claude Code means configuring Claude Code to run programmatically in CI/CD pipelines, git hooks, and scheduled jobs so that it can execute code review, test generation, documentation updates, and batch refactoring without manual intervention.

Think of it like a factory quality-control robot. On a factory assembly line, every product passes through a series of automated inspection stations. Each station checks one thing: dimensions, weight, seal integrity, label placement. The robot does not replace the human quality inspector -- it handles the repeatable checks so the human can focus on the products that fail inspection and need judgment calls. Claude Code in your CI pipeline is the same idea. It handles the predictable checks and automated tasks -- reviewing diffs, generating tests, updating docs, running formatters -- so you can focus on the architectural decisions, the design trade-offs, and the code that actually needs human attention.

## Real-World Use Cases

Before you dive into the mechanics, it helps to see the concrete scenarios where Claude Code automation delivers the most value. Each use case below represents a pattern you can adapt to your own workflow.

### Automated PR Review

Every pull request goes through the same checklist: does it have tests? Are there security issues? Does it follow the project style guide? Does the documentation need updating? A human reviewer spends 15 to 30 minutes on each PR checking these boxes. With Claude Code, the initial review happens in seconds. The reviewer gets a structured report of issues before they even open the PR, and they can focus on the logic and design decisions that matter.

### Test Generation on Code Changes

When a developer adds a new function or modifies an existing one, the corresponding test file needs updating. This is often the first thing that gets skipped when the deadline is tight. Claude Code can examine the diff, identify changed functions, and generate test stubs or complete test cases that follow the project's existing test conventions. The developer reviews and adjusts the tests rather than writing them from scratch.

### Documentation Sync

Code documentation drifts out of sync with the codebase over time. Function signatures change, parameters are added or removed, and the docstrings stay frozen in the past. Claude Code can scan the codebase for discrepancies between the code and its documentation, then generate updates that bring the docs back in sync. This is especially valuable for API documentation where accuracy is critical.

### Batch Refactoring

When you rename a function, update an import path, or change a configuration key, the change needs to propagate across every file that references it. Doing this manually is error-prone -- it is easy to miss a file in a large codebase. Claude Code can find all references, apply the change consistently, and verify that nothing is broken afterward.

### Release Automation

Creating a release involves updating the changelog, bumping the version number, generating release notes, updating dependency files, and creating a git tag. Each step is simple, but the sequence is easy to get wrong. Claude Code can execute the entire release workflow: read the commit history, categorize the changes, generate release notes, update version files, and create the GitHub release.

## CLI Reference

The following table covers the key flags and commands you use when running Claude Code in automated contexts.

| Flag | Syntax | Purpose | Example |
| --- | --- | --- | --- |
| Dangerously skip permissions | `--dangerously-skip-permissions` | Bypass all permission prompts for unattended CI runs | `claude -p "review this diff" --dangerously-skip-permissions` |
| Print mode | `-p "PROMPT"` or `--print "PROMPT"` | Run a single prompt and exit immediately | `claude -p "Generate tests for src/utils.py"` |
| Allowed tools | `--allowedTools TOOL1,TOOL2` | Restrict Claude Code to a specific set of tools | `claude --allowedTools Read,Edit,Bash` |
| Output format | `--output-format json` | Emit structured JSON for programmatic parsing | `claude -p "Review this diff" --output-format json` |
| Model selection | `--model MODEL_ID` | Pin a specific Claude model version | `claude --model claude-sonnet-4-20250514` |
| Resume session | `--resume SESSION_FILE` | Continue a previous session from a saved state | `claude --resume review-session.json` |
| Max threads | `--max-threads N` | Limit the number of concurrent tool operations | `claude --max-threads 2` |
| Verbose logging | `--verbose` | Enable detailed log output for debugging | `claude -p "run task" --verbose` |
| Shutdown timeout | `--shutdown-timeout MS` | Set a hard timeout for the Claude Code process | `claude --shutdown-timeout 300000` |
| Process mode | `--process FILE` | Execute a predefined process specification | `claude --process review-process.json` |

## How It Actually Works -- Step by Step

Understanding what happens under the hood lets you design reliable automation and debug it when things go wrong. Each step below reveals a layer of the automation stack.

### Step 1: The Trigger Event Arrives

Every automation pipeline starts with an event. The event source determines when and how Claude Code runs. There are three main trigger types.

       │                    │                     │
       │                    │                     │

                            │
                            ▼
                   ┌────────────────┐
                   │ CI Trigger     │
                   │ Webhook        │
                   │ Receiver       │
                   └────────────────┘`} />

When you push to a repository, GitHub sends a webhook payload to the CI system. The payload contains the commit hash, branch name, author information, and a list of changed files. The CI system receives this payload and checks whether the event matches the workflow trigger conditions. For example, a workflow configured to run on `pull_request` events will ignore push events. For cron-based automation, the CI system uses its built-in scheduler to fire the pipeline at the specified time, and the event payload contains only the schedule time and the branch to check out.

The event metadata is important because it shapes what Claude Code does. A PR event includes the base branch and the head branch, so Claude Code can diff them. A push event includes the commit message, which provides context for the changes. A cron event has no specific change context, so Claude Code must analyze the codebase as a whole.

### Step 2: The CI Pipeline Spawns Claude Code

The CI pipeline definition -- typically a YAML file in the repository -- contains a step that invokes Claude Code. The pipeline first checks out the repository, installs dependencies, and then runs Claude Code with the appropriate flags and prompts.

<AsciiDiagram id="automating-dev-tasks-with-claude-code/step2" title="CI pipeline spawning Claude Code as a job step" content={`┌──────────────────────────────────────────────────────┐

│     └──────────────────────────────────┘              │

└──────────────────────────────────────────────────────┘`} />

The CI runner checks out the repository, installs any dependencies, and then runs Claude Code with the appropriate flags. The `--dangerously-skip-permissions` flag is essential for unattended operation -- without it, Claude Code would prompt for permission on every tool call and the pipeline would hang forever. The `--allowedTools` flag restricts what Claude Code can do, providing a safety boundary. The `--output-format json` flag makes the output machine-readable so the CI step can parse it.

The environment setup matters. The CI runner must have node installed (for Claude Code CLI), the repository must be checked out with full git history (so Claude Code can see the diff), and the `ANTHROPIC_API_KEY` must be passed as an environment variable from the CI secrets store. Missing any of these causes the step to fail.

### Step 3: Claude Code Authenticates and Initializes

When Claude Code starts, it reads the `ANTHROPIC_API_KEY` environment variable to authenticate with the Anthropic API. If this variable is not set, the process exits with an authentication error. Once authenticated, Claude Code loads the repository context -- it reads the files in the working directory, identifies the project structure, and builds an initial understanding of the codebase.

<AsciiDiagram id="automating-dev-tasks-with-claude-code/step3" title="Claude Code initialization and authentication flow" content={`┌────────────────────────────────────────────────────┐

│  4. Identify project structure and language          │
│  5. Load key files into context                      │
│  6. Ready for task execution                         │
└────────────────────────────────────────────────────┘`} />

This initialization step is critical for automation. The context window determines how much of the codebase Claude Code can see at once. For large repositories, you may need to narrow the scope by passing a specific file path or diff in the prompt rather than expecting Claude Code to explore the entire codebase. The initialization also checks for a `.claude` configuration file in the project root, which can define project-specific settings like allowed tools, custom prompts, or ignored files.

### Step 4: Claude Code Executes the Task

With the context loaded, Claude Code begins executing the assigned task. It uses the tools available to it -- restricted by the `--allowedTools` flag -- to read files, edit code, run commands, and examine the repository.

<AsciiDiagram id="automating-dev-tasks-with-claude-code/step4" title="Claude Code executing tools within the allowed tool set" content={`┌────────────────────────────────────────────────────────┐

│  3. Determine changes needed                             │

              │ Allowed Tools (set by  │
              │ --allowedTools flag)   │
              │                        │
              │ Read    Edit    Bash   │
              │ Grep    Glob    Write  │
              └────────────────────────┘`} />

The execution loop is the core of Claude Code's automation capability. For each step in the loop, Claude Code decides which tool to call, what parameters to pass, and how to interpret the results. This loop continues until the task is complete or a limit is reached. The `--shutdown-timeout` flag provides a hard stop: if Claude Code has not finished within the specified time, the process is killed.

The tool call pattern matters for cost and reliability. Each tool call consumes tokens for the request and the response. Excessive tool calls -- especially calls that read large files unnecessarily -- can drive up costs and increase the risk of hitting the timeout. A well-designed prompt guides Claude Code to be efficient with its tool calls. For example, instead of asking Claude Code to "explore the codebase," give it specific file paths to examine.

There are three common tool call patterns that emerge in automation:

**Single-pass pattern.** Claude Code reads the necessary files, makes a determination, and produces output. This is the fastest and cheapest pattern, suitable for simple review tasks. The tool call sequence is typically Read, Read, Read, then output.

**Iterative refinement pattern.** Claude Code reads files, makes a change, reads the result, adjusts, and repeats. This is suitable for tasks like test generation where the first attempt may need correction. The tool call sequence is Read, Write, Read, Edit, Read, output.

**Exploratory pattern.** Claude Code searches the codebase to find relevant files, then processes them. This is suitable for tasks like batch refactoring where the affected files are not known in advance. The tool call sequence is Grep or Glob, then Read on each match, then Edit on each file.

### Step 5: Claude Code Produces Output

Once the task is complete, Claude Code produces output based on the `--output-format` flag. In JSON mode, the output is a structured object that includes the status, the changes made, the files affected, and any messages or comments.

The structured output is designed for programmatic consumption. The CI pipeline step that invoked Claude Code can parse this JSON and decide what to do next: post a comment on the PR, create a new issue, fail the build, or pass the results to a notification system. The output includes a status field that indicates whether the task succeeded, failed, or was interrupted by the timeout. You should always check the status field before processing the rest of the output. If the status is `timeout`, the output may be incomplete.

### Step 6: The CI Pipeline Acts on the Results

The final step belongs to the CI pipeline. The pipeline reads the Claude Code output and takes the appropriate action. This is typically a shell script or a GitHub Actions step that runs after Claude Code exits.

The CI pipeline acts as the router. It decides what to do with the output based on the context. If the automation was triggered by a PR, the results are posted as a comment on that PR. If the automation was triggered by a cron schedule, the results might be used to create a new PR with the changes. If the automation found critical issues, the pipeline might fail the build to prevent merging.

The routing logic is typically implemented as a series of conditional checks in the CI pipeline YAML. For example, in GitHub Actions, you use the `if:` keyword to conditionally run steps based on the output of previous steps. A common pattern is to check whether the Claude Code output contains issues with severity `high`, and if so, fail the build and notify the team.

## Setting Up Your First Automation

Getting started with Claude Code automation requires three things: a trigger, a task definition, and an output handler. The following diagram shows the minimum setup.

│                                                                  │

│  │              │    │                  │    │ Handler      │   │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘`} />

To set up your first automation, follow these steps:

1. Create the workflow file in `.github/workflows/` with the appropriate trigger event
2. Add a step that installs Claude Code via npm
3. Add a step that invokes Claude Code with the task prompt and the required flags
4. Add a step that parses the JSON output and takes action
5. Push the workflow file to the repository and test it with a real PR

The key to a successful first automation is keeping it simple. Start with a single task -- reviewing a PR diff -- and a single output action -- posting a comment. Once that works, layer on additional tasks like test generation or documentation updates.

Here is a concrete example of a minimal workflow file that runs Claude Code on every PR push:

```yaml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: Run review
        run: |
          claude -p "Review the diff in this PR. \
          Output JSON with summary and issues." \
            --dangerously-skip-permissions \
            --allowedTools Read,Grep,Glob \
            --output-format json \
            --shutdown-timeout 180000 > review.json
      - name: Post comment
        run: |
          SUMMARY=$(jq -r '.summary' review.json)
          gh pr comment ${{ github.event.pull_request.number }} \
            --body "## Claude Code Review\n\n$SUMMARY"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

This workflow is about 25 lines of YAML. It checks out the repository, installs Claude Code, runs a review, and posts the results as a PR comment. The entire setup takes less than 10 minutes from start to finish.

## Common Automation Patterns

As you build more automations, you will encounter recurring patterns that solve common problems. Recognizing these patterns saves you from designing the same solution twice.

### The Fire-and-Forget Pattern

Trigger a Claude Code task and do not wait for the result. The task runs, produces output, and the pipeline moves on. This pattern is useful for tasks where the result is not needed immediately -- for example, generating a weekly code quality report that is posted to a dashboard.

```yaml
- name: Run weekly quality report
  run: |
    claude -p "Analyze code quality and save report to reports/" \
      --dangerously-skip-permissions \
      --allowedTools Read,Write,Grep,Glob \
      --output-format json \
      --shutdown-timeout 300000
  continue-on-error: true
```

The `continue-on-error: true` flag ensures the pipeline does not fail if the Claude Code task fails. This is appropriate for informational tasks that should not block the CI pipeline.

### The Gatekeeper Pattern

Claude Code runs a review and the pipeline fails if the review finds critical issues. This pattern is useful for enforcing quality gates before merging.

```yaml
- name: Quality gate review
  run: |
    claude -p "Review this PR. If you find any HIGH severity issues, \
    output status: fail. Otherwise output status: pass." \
      --dangerously-skip-permissions \
      --allowedTools Read,Grep,Glob \
      --output-format json > gate.json
    STATUS=$(jq -r '.status' gate.json)
    if [ "$STATUS" = "fail" ]; then
      echo "Quality gate failed"
      exit 1
    fi
```

The gatekeeper pattern is the most common automation pattern for PR workflows. It prevents merges that introduce regressions, missing tests, or documentation gaps.

### The Branch-and-PR Pattern

Claude Code creates a new branch with its changes and opens a PR for human review. This pattern is useful for tasks where the changes need human approval before merging -- for example, dependency updates or batch refactoring.

```yaml
- name: Create PR with changes
  run: |
    git checkout -b claude-auto-changes
    git add -A
    git commit -m "Auto-generated changes"
    git push origin claude-auto-changes
    gh pr create --title "Auto-generated changes" --body "Review required"
```

The branch-and-PR pattern gives you a safety net. Every change Claude Code makes is reviewed by a human before it reaches the main branch.

### The Notification Pattern

Claude Code sends its results to a notification system -- Slack, email, or a dashboard. This pattern is useful for scheduled tasks where the results are informational rather than actionable.

```bash
# Send Claude Code results to Slack
REPORT=$(claude -p "Summarize code quality" --output-format json)
curl -X POST -H "Content-type: application/json" \
  --data "{\"text\": \"$REPORT\"}" \
  "$SLACK_WEBHOOK_URL"
```

Each pattern serves a different purpose. The fire-and-forget pattern is for background tasks. The gatekeeper pattern is for quality enforcement. The branch-and-PR pattern is for changes that need review. The notification pattern is for reporting.

## Cost, Limits, and Safety

Running Claude Code in CI has real implications for cost, reliability, and security. Understanding these constraints helps you design automation that is both effective and safe.

### API Costs

Each Claude Code invocation consumes tokens from the Anthropic API. The cost depends on the model used, the length of the prompt, the size of the codebase context, and the number of tool calls made during execution. A typical code review of a 500-line diff might cost between 0.10 and 0.50 dollars per run. A full test generation run across a module could cost 1.00 to 3.00 dollars. For a team running 50 automated reviews per day, the monthly cost can range from 150 to 750 dollars.

To give you a concrete sense of the cost structure, here is a breakdown by task type:

- **PR review (500-line diff):** 0.10 to 0.50 dollars, 30 to 60 seconds
- **Test generation (single module):** 0.50 to 2.00 dollars, 1 to 3 minutes
- **Documentation sync (full repo):** 1.00 to 5.00 dollars, 2 to 5 minutes
- **Batch refactoring (10 files):** 0.50 to 3.00 dollars, 1 to 4 minutes
- **Release notes generation:** 0.20 to 1.00 dollars, 30 to 90 seconds

To manage costs, consider these strategies:

- Use `--allowedTools` to restrict tool usage and prevent unnecessary exploration
- Set `--shutdown-timeout` to bound the maximum runtime
- Use `--max-threads 1` to prevent parallel operations that consume more tokens
- Pin a specific model version with `--model` to avoid unexpected cost changes
- Run Claude Code only on changed files rather than the full codebase
- Use `CLAUDE_CODE_MAX_TOKENS` to cap total token consumption per run

### Rate Limits

The Anthropic API imposes rate limits on requests per minute and tokens per minute. When running Claude Code in CI, multiple concurrent pipeline runs can hit these limits. This is especially common when multiple PRs are opened simultaneously or when a cron-based automation runs alongside PR-based automation.

The rate limits depend on your API tier. At the standard tier, you can expect approximately 50 requests per minute and 100,000 tokens per minute. If your team has 10 developers who each push 3 times per day, and each push triggers a Claude Code review, you could have up to 30 concurrent requests at peak times. This is within the standard rate limit, but if you also run a nightly full-codebase scan, the combined load could exceed the limit.

To handle rate limits:

- Stagger cron-based automation to avoid peak hours
- Implement retry logic with exponential backoff in the CI pipeline step
- Use a queue system for concurrent automation requests
- Monitor API usage and adjust limits accordingly
- Consider upgrading to a higher API tier if your team scales up

### Safety with --allowedTools

The `--allowedTools` flag is your primary safety mechanism. Without it, Claude Code has access to the full tool set, including `Bash` and `Write`, which could modify files or run arbitrary commands. In a CI context, this is a security risk.

A typical safe configuration for code review:

```bash
claude -p "Review the diff in this PR" \
  --allowedTools Read,Grep,Glob \
  --dangerously-skip-permissions \
  --output-format json
```

This configuration only allows Claude Code to read files and search the codebase. It cannot edit files or run commands. For tasks that require editing, such as test generation or documentation updates, add `Edit` and `Write` to the allowed tools list, but be aware that this introduces the risk of unintended modifications. The principle of least privilege applies: give Claude Code only the tools it needs for the specific task.

### Environment Variables

Claude Code reads the following environment variables in CI contexts:

- `ANTHROPIC_API_KEY` -- required. The API key for authentication. Never hardcode this in your CI configuration. Use CI secrets or environment variable management.
- `CLAUDE_CODE_MAX_TOKENS` -- optional. Limits the total tokens used per session. Set this to prevent runaway costs.
- `CLAUDE_CODE_MAX_TURNS` -- optional. Limits the number of tool call turns per session. Set this to prevent infinite loops.
- `CLAUDE_CODE_DISABLE_VERSION_CHECK` -- optional. Disables the version check for faster startup in CI.
- `CLAUDE_CODE_CONFIG_PATH` -- optional. Specifies a custom path to the Claude Code configuration file.

### Avoiding Runaway Loops

The most dangerous failure mode in CI automation is a runaway loop: Claude Code gets stuck in a cycle of making changes, reviewing them, making more changes, and never stopping. This can consume thousands of dollars in API costs before someone notices.

To prevent runaway loops:

- Always set `--shutdown-timeout` to a reasonable value (300 seconds is a good starting point)
- Set `CLAUDE_CODE_MAX_TOKENS` to limit total token consumption
- Set `CLAUDE_CODE_MAX_TURNS` to limit the number of tool call iterations
- Implement a circuit breaker in the CI pipeline that kills the step if it runs longer than expected
- Review the output of the first few automated runs to ensure the loop terminates naturally
- Use `--max-threads 1` to prevent parallel tool calls that can amplify a runaway loop

### Security Considerations

When running Claude Code in CI, you are granting an AI system access to your codebase. This has security implications that you must address.

**API key management.** The `ANTHROPIC_API_KEY` is a credential that grants access to the Claude API. Store it in the CI secrets store, not in the repository. Never commit the key to a file, even in a private repository.

**Scope of access.** The `--allowedTools` flag controls what Claude Code can do, but it does not control what files it can read. Claude Code can read any file in the working directory. If your repository contains secrets, configuration files with passwords, or other sensitive data, Claude Code can access them during automation. Keep sensitive data out of the repository or use environment variables for secrets.

**Output handling.** The JSON output from Claude Code may contain code snippets, file contents, or other information from the codebase. If the output is posted to a public PR or a public channel, this information becomes public. Be mindful of what you do with the output.

**Branch protection.** When Claude Code creates commits or branches, consider whether those changes should be subject to the same review process as human changes. Configure branch protection rules that require approval for auto-generated PRs.

## When to Use This / When Not To

Automation with Claude Code is powerful, but it is not the right tool for every situation. The key is knowing when to automate and when to leave the task to a human.

### When to Use

**Automated PR reviews.** Use Claude Code to review every PR for common issues: missing tests, missing documentation, formatting violations, dead code, and security antipatterns. This catches the easy problems before they reach a human reviewer, so the human can focus on logic and design.

**Test generation.** When you add a new function or module, use Claude Code to generate the initial test suite. The generated tests will not be perfect, but they give you a starting point that is faster than writing from scratch.

**Documentation updates.** Use Claude Code to update API documentation, README files, and inline comments when code changes. This is especially valuable for large codebases where documentation drifts out of sync with the code.

**Batch refactoring.** When you need to apply a consistent change across many files -- renaming a function, updating an import path, changing a configuration key -- use Claude Code to make the changes programmatically.

**Scheduled maintenance tasks.** Use cron-based automation to run weekly dependency updates, security scans, or code quality reports.

**Release automation.** Use Claude Code to generate release notes, update the changelog, bump version numbers, and create GitHub releases.

### When Not To Use

**Production deployments without review.** Never let Claude Code deploy to production without human approval. The cost of a bad deployment far outweighs the convenience of automation.

**Security-sensitive operations.** Do not use Claude Code for tasks that involve secrets management, access control changes, or security policy modifications. The risk of unintended exposure is too high.

**When you need deterministic results.** Claude Code is a language model, not a deterministic program. The same prompt can produce different outputs on different runs. If you need exact reproducibility, use a traditional script instead.

**When cost is prohibitive.** For very large codebases or very frequent runs, the API costs can add up quickly. Calculate the expected cost before automating and compare it to the value of the time saved.

**When the task requires human judgment.** Code review is not just about finding bugs. It is about understanding the intent of the code, evaluating trade-offs, and making architectural decisions. Claude Code can flag issues, but it cannot replace a thoughtful human review.

**When the codebase contains sensitive data.** If your repository contains secrets, credentials, or personally identifiable information, running Claude Code automation on it exposes that data to the API. Review the codebase for sensitive data before enabling automation.

## Debugging Automation Failures

When your Claude Code automation fails, the problem is usually in one of four areas. Knowing where to look saves debugging time.

### Authentication Failures

If Claude Code exits immediately with no output, the API key is likely missing or invalid. Check that the `ANTHROPIC_API_KEY` environment variable is set in the CI environment. Verify that the API key has not expired and has the necessary permissions.

Debugging steps:

```bash
# Verify the API key is set
echo "Key length: ${#ANTHROPIC_API_KEY}"
# Check the first few characters to confirm it is the right key
echo "Key starts with: ${ANTHROPIC_API_KEY:0:5}..."
```

### Timeout Failures

If Claude Code produces incomplete output or exits with a timeout status, the `--shutdown-timeout` is too low for the task. Increase the timeout value, or optimize the prompt to reduce the number of tool calls.

The typical solution is to double the timeout and run the task again. If it times out again, the prompt is too broad -- narrow it to a specific scope.

### Tool Restriction Failures

If the output contains error messages about denied tool access, the `--allowedTools` list is missing a tool that Claude Code needs. Review the error message to see which tool was denied, and add it to the list if appropriate.

For example, if the error says "Write tool is not in the allowed tools list" and the task needs to create files, add `Write` to the list. Be careful not to add tools that the task does not need.

### Output Parsing Failures

If the CI pipeline fails at the JSON parsing step, the Claude Code output is not valid JSON. This can happen if the output is truncated, if Claude Code included explanatory text outside the JSON block, or if the JSON structure changed between runs.

Add a validation step before parsing:

```bash
# Validate JSON before parsing
if ! jq empty claude-output.json 2>/dev/null; then
  echo "Error: Invalid JSON output"
  cat claude-output.json
  exit 1
fi
```

### Pipeline Logic Failures

If the automation runs but produces the wrong results, the pipeline logic is incorrect. For example, the trigger event might not match what you expect, or the output action might be routing to the wrong destination.

Check the CI logs to verify each step executed in the expected order. Add `--verbose` to the Claude Code invocation to get detailed logs of the execution loop.

You are setting up a cron-based automation that runs Claude Code on a schedule. Your team has 20 developers who each push 3 times per day, and you also want to run a full codebase scan every night. What is the primary concern you need to address?

Answer: Rate limiting. The combination of PR-based automation (up to 60 triggers per day from pushes) and cron-based automation (one nightly scan) can hit the API rate limits. You should stagger the cron-based automation to run during low-traffic hours and implement retry logic for API requests.

Your CI pipeline runs Claude Code to generate tests for every new PR. You notice that some runs take over 5 minutes and cost more than expected. What two flags can you use to bound the cost and runtime?

Answer: `--shutdown-timeout` to set a hard timeout (for example, `--shutdown-timeout 120000` for 2 minutes) and `--allowedTools` to restrict Claude Code to only the tools it needs (for example, `--allowedTools Read,Edit,Write` for test generation). You can also set `CLAUDE_CODE_MAX_TOKENS` and `CLAUDE_CODE_MAX_TURNS` environment variables to further limit resource consumption.

You want Claude Code to review a PR diff but not make any changes. What `--allowedTools` configuration should you use?

Answer: `--allowedTools Read,Grep,Glob`. This restricts Claude Code to read-only tools. It can examine files and search the codebase, but it cannot edit files, write new files, or run commands. This is the safest configuration for review-only tasks.

Your automated Claude Code task generates a test file and a documentation update based on changes in a PR. The task runs, creates the files, and then exits. The CI pipeline needs to create a new commit with these changes. What output format should you use, and why?

Answer: `--output-format json`. The JSON output provides structured data about what files were created or modified, the status of the task, and any messages. The CI pipeline can parse this JSON to determine which files to stage and commit, rather than trying to parse unstructured text output.
