You have just been handed a 200,000-line codebase you have never seen. The ticket says "fix the flaky checkout test." You open the repo, scan the directory tree, and feel the familiar dread: where do you even start? You grep for `checkout`, find forty-seven matches across six files, open three of them, get lost in a maze of imports, and forty-five minutes later you have not written a single line of code. That is the problem Claude Code solves — not by replacing you, but by becoming your pair programmer who has already read the whole repo.

Or consider this: you need to refactor a payment processing module from synchronous HTTP calls to an async event-driven pattern. You know the architecture you want, but the implementation requires touching twelve files, running five migration commands, and updating three test suites. By hand, that is a full afternoon of context-switching, one error-prone search-and-replace at a time. With Claude Code, you describe the target architecture in a few sentences, and it plans the entire refactor, proposes each file change, and waits for your approval before executing.

The pain is not about writing code — it is about the overhead of context recovery every time you switch files, the mental stack of tracking what changed where, and the fear of missing an edge case across a hundred files. Claude Code removes that overhead by keeping your entire project in context and executing your instructions across the whole codebase at once.

## The Concept

Claude Code is an agentic coding tool that runs in your terminal, reads your entire codebase into context, and executes tasks you give it in natural language. It can edit files, run commands, search code, write git commits, and loop on a task until it is done — all while you review and approve every action. Think of it as a senior engineer who works at your keyboard: you describe what needs to happen, it proposes the implementation, and you decide whether to accept it.

The key insight is that Claude Code is not an autocomplete or a chat widget embedded in your editor. It is a full terminal-based agent that has access to tools — reading files, editing files, running shell commands, searching code — and uses those tools to accomplish goals you define. Every tool call is visible to you, and every file write requires your approval by default.

## CLI Reference

The following table covers every core command and flag you need to get started with Claude Code. All syntax shown is current as of Claude Code v2.x.

| Command / Flag | What it does | Example |
|---|---|---|
| `claude` | Launch Claude Code in the current directory | `claude` |
| `claude -p "prompt"` | Run a one-shot prompt and exit (non-interactive) | `claude -p "explain this project"` |
| `claude --model` | Choose a model: `claude-sonnet-4-20250514` (default) or `claude-opus-4-20250514` | `claude --model opus` |
| `claude --resume` | Resume the most recent session | `claude --resume` |
| `claude --allowedTools` | Permit tools without prompting for this session | `claude --allowedTools "Bash, Edit, Write"` |
| `/help` | Show available slash commands inside a session | `/help` |
| `/clear` | Clear the conversation history inside a session | `/clear` |
| `/status` | Show current git status, diff, and session info | `/status` |
| `Ctrl+C` | Interrupt the current response or exit the session | Press Ctrl+C |
| `claude execute "cmd"` | Run a shell command directly through Claude Code | `claude execute "npm test"` |
| `claude --verbose` | Show full tool-by-tool execution details | `claude --verbose` |
| `claude --output-format json` | Output structured JSON for scripting | `claude --output-format json -p "summarize"` |
| `claude --config` | Open project-level settings in an editor | `claude --config` |
| `claude --version` | Print the installed version | `claude --version` |
| `claude update` | Update Claude Code to the latest version | `claude update` |

### Model Comparison

The two models available for Claude Code differ significantly in capability and cost:

| Model | Best For | Speed | Cost |
|---|---|---|---|
| `claude-sonnet-4-20250514` | Day-to-day coding, debugging, refactoring, file edits | Fast | Lower |
| `claude-opus-4-20250514` | Complex architecture, multi-file orchestration, deep reasoning | Slower | Higher |

Sonnet is the default for a reason: it handles the vast majority of coding tasks at a good balance of speed and intelligence. Switch to Opus when you are working on a problem that requires deep architectural reasoning — designing a new service boundary, untangling a complex dependency graph, or planning a multi-step migration.

## How It Actually Works — Step by Step

### Step 1: You Install Claude Code

Claude Code is distributed as an npm package. You install it globally:

```bash
npm install -g @anthropic-ai/claude-code
```

This installs the `claude` command globally on your system. The package bundles the Claude Code agent runtime, the tool execution engine, and the terminal user interface. After installation, verify it worked:

```bash
claude --version
```

Expected output (version may vary):

```
@anthropic-ai/claude-code/2.1.200 (win32 x64)
```

You must have an active Anthropic account. On first launch, Claude Code prompts you to authenticate via OAuth — it opens a browser window where you log in to your Anthropic account and authorize the CLI. After that, a session token is stored locally and used for all subsequent requests.

Using npm is the only supported installation method. Alternative package managers are not officially supported, but you may be able to install via yarn or pnpm with the global flag:

```bash
yarn global add @anthropic-ai/claude-code
# or
pnpm add -g @anthropic-ai/claude-code
```

Note that the package is quite large — it bundles the Claude model interface, the tool execution engine, the terminal UI, and several support libraries. The download is approximately 50-100 MB depending on your platform. Installation usually completes within 30-60 seconds on a standard internet connection.

### Platform-Specific Notes

**Windows:** npm global packages install to `%APPDATA%\npm\`. Make sure this directory is in your PATH. Claude Code stores session tokens in `%USERPROFILE%\.claude\`.

**macOS / Linux:** npm global packages install to `/usr/local/lib/node_modules/` or the nvm-managed equivalent. Claude Code stores session tokens in `~/.claude/`.

**CI/CD Environments:** In automated environments where you cannot use OAuth, set the `ANTHROPIC_API_KEY` environment variable before launching Claude Code. The `-p` flag is especially useful in scripts and CI pipelines because it exits cleanly with a zero exit code on success.

### Step 2: You Launch Claude and Set Context

When you run `claude` from your project root, the agent scans the directory structure, reads `.gitignore` to know which files to skip, and loads your project files into its context window. It does not load every file blindly — it reads the directory tree first (using the `ls` tool internally) and only reads individual files when it needs their content to answer your question or execute your task.

```bash
cd /path/to/your/project
claude
```

The terminal clears and shows the Claude Code prompt — a `▶` symbol indicating the agent is ready. The model loaded by default is `claude-sonnet-4-20250514`, which provides fast responses for most coding tasks.

### Step 3: You Describe Your Task

At the `▶` prompt, you type a natural language description of what you want done. This is the most important part of using Claude Code effectively. Good task descriptions include:

- **What you want done:** "Add input validation to the signup form"
- **Where to look:** "in the `src/auth/` directory"
- **Constraints:** "Use zod for validation, do not modify the database layer"
- **Acceptance criteria:** "Tests should pass and the form should reject empty emails"

Poor task descriptions get poor results:

> "Fix the bug" — too vague, does not say which bug or where.

Better:

> "The signup form crashes when the email field is empty. Find the crash in `src/auth/signup.ts` and add zod validation to prevent it. Keep the existing test structure in `src/auth/__tests__/` and make sure all tests pass."

When you write a good task description, Claude Code immediately starts working. It does not ask clarifying questions by default — it assumes you gave it enough context and proceeds. If the task description is ambiguous, Claude Code does its best to interpret it, which may not match what you intended.

To refine the task during execution, you can:

- **Add context mid-task:** "Also check for the same issue in the admin form"
- **Change direction:** "Actually, use yup instead of zod"
- **Ask for explanation:** "Explain why you chose that approach"

Each follow-up prompt builds on the previous conversation history, so Claude Code remembers the entire context of the task.

Claude Code processes your description and begins planning. It reads the relevant files you mentioned, searches for the patterns you described, and builds a mental model of the change needed before proposing anything.

### Step 4: Claude Reads Files and Plans the Work

Claude Code uses a set of tools to explore your codebase. The most commonly used tools are:

| Tool | What it does | Example usage |
|---|---|---|
| `Read` | Reads a file's content into context | Read a source file to understand its structure |
| `Grep` | Searches file contents with regex | Find all usages of a function |
| `Glob` | Finds files matching a pattern | Find all test files in a directory |
| `Bash` | Runs a shell command | Run `npm test` or `git status` |
| `Edit` | Makes a targeted edit to a file | Replace a block of code |
| `Write` | Writes or overwrites a file | Create a new file from scratch |

When you give a task, Claude Code uses these tools in sequence. It might start with `Glob` to find relevant files, then `Read` a few of them to understand the code, then `Grep` for specific patterns to find all touch points, and finally propose edits using `Edit` or `Write`.

Each tool call is logged to the terminal. With default settings, you see the tool name and a summary of what it does. With `--verbose`, you see the full input and output of each tool, which is useful for debugging or learning.

The order of tool calls is not hardcoded. Claude Code decides which tool to use based on the current goal. For example:

- If you ask "find all places where we handle errors," Claude uses `Grep` first to locate the patterns
- If you ask "read the main config file," Claude uses `Glob` or `Bash ls` to find it, then `Read` to load the content
- If you ask "add a new endpoint," Claude reads the existing route definitions first, then plans the edit

### Step 5: You Approve or Modify the Plan

Once Claude Code has a plan, it presents its proposed changes to you. By default, every file write requires your explicit approval. The approval prompt shows:

- The file path being changed
- A diff of what will change (additions in green, removals in red)
- The specific tool being used (Edit, Write, or Bash)

You have three options at each prompt:

- **Approve (Y):** Allow the change to proceed
- **Deny (N):** Reject this specific change
- **Edit (E):** Modify what Claude proposed before approving

This permission model is the central safety mechanism in Claude Code. It ensures you never lose control of what gets written to your filesystem. You can also pre-approve tools using `--allowedTools` to speed up routine tasks:

```bash
claude --allowedTools "Bash, Edit, Write, Read"
```

But use this sparingly. The approval prompt is your safety net against accidental damage.

### Step 6: Claude Executes and Iterates

After you approve each change, Claude Code applies it. The Edit tool makes precise, targeted modifications — it does not rewrite entire files unless you explicitly ask for a full rewrite. This precision means only the lines you need to change are modified, reducing the risk of unintended side effects.

After making changes, Claude Code typically runs verification commands automatically. If you asked for a bug fix, it might run:

```bash
npm test
```

If the tests pass, Claude reports success. If they fail, it reads the test output, identifies the issue, and fixes it. This loop — propose, approve, execute, verify — repeats until the task is complete or you interrupt it.

If at any point you see Claude heading in the wrong direction, press **Ctrl+C** to interrupt the current response, then rephrase your instructions. A second Ctrl+C exits the session entirely.

### Step 7: You Exit Claude Code

When your task is complete, type `/bye` or press **Ctrl+C** twice to exit. Claude Code also exits when you close the terminal window. If you want to return to the same session later, use:

```bash
claude --resume
```

This reloads the conversation history from the previous session, including all file context and changes that were discussed. Note that the resumed session does not automatically re-read files — it relies on the context that was built during the original session.

### Step 8: Understanding the Permission Model in Depth

The Claude Code permission model has three tiers:

**Tier 1 — Prompt on every tool use (default):** Every time Claude Code wants to read a file, edit a file, or run a command, it asks for your approval. This is the safest mode and recommended for new users.

**Tier 2 — Pre-approved tools via `--allowedTools`:** You specify which tools Claude can use without asking. This is good for routine tasks where you trust Claude's judgment:

```bash
claude --allowedTools "Bash, Edit, Write, Read"
```

**Tier 3 — Session-level approvals:** Inside a session, you can type `yes` to approve all remaining tool calls for this session. Use this when Claude is on a roll and you want to speed things up.

The permission model applies to all tools that modify the filesystem or run external commands. Read-only tools (Read, Grep, Glob) are typically pre-approved in practice because they cannot damage your project.

### Behind the Scenes: What Happens During Tool Execution

When Claude Code executes a tool, several things happen at once:

1. **The tool call is constructed.** Claude Code serializes the tool name, parameters, and expected output format into a structured request.
2. **The approval prompt renders.** If approvals are enabled, the terminal shows the tool, its parameters, and (for Edit/Write) a diff of the changes.
3. **Approval is awaited.** Claude Code pauses and waits for your input. This is not a timeout — the session remains active indefinitely while waiting.
4. **The tool executes.** After approval, the tool runs against your project. Edit modifies the file in-place. Bash runs the command and captures stdout/stderr. Write creates or overwrites the file.
5. **The result is returned.** Claude Code reads the output of the tool (file content for Read, search results for Grep, command output for Bash) and incorporates it into its reasoning.
6. **The plan continues.** Claude Code evaluates whether the tool output matches the expected outcome. If yes, it proceeds to the next step. If no, it retries or asks for clarification.

This cycle happens rapidly. A single task might involve 5-20 tool calls in sequence, each taking a few seconds to propose, approve, and execute.

### Context Window Management

Claude Code has a limited context window (the amount of text it can process at once). With large codebases, managing the context window is important:

- Claude Code prioritizes recently read files and the current conversation
- Older context may be summarized or dropped when the window is full
- The `/status` command shows context usage information
- Use `/clear` to reset the conversation and free up context space
- Creating a `.claudeignore` prevents unnecessary files from consuming context

If Claude Code seems to "forget" earlier parts of the conversation, it is likely because the context window has filled up. Use `/clear` to reset and provide fresh context.

## Cost, Limits, and Safety

### How Claude Code Billing Works

Claude Code uses a separate billing system from the Anthropic API. Instead of per-token pricing, Claude Code bills by session usage. Charges are based on the number of requests and the model used. As of July 2026:

| Model | Relative Cost |
|---|---|
| `claude-sonnet-4-20250514` | Lower per-request cost |
| `claude-opus-4-20250514` | Higher per-request cost (approximately 3-5x Sonnet) |

Each interactive session consumes multiple requests — one for each prompt you send and each follow-up. A typical debugging session might use 10-20 requests. A complex refactoring session across many files might use 50-100 requests.

To estimate the cost of a session, multiply the number of prompts you expect to send by the per-request cost for the model you are using. For example, a Sonnet session with 20 prompts costs approximately 20 times the Sonnet per-request rate. An Opus session handling the same task would cost 3-5 times more.

Claude Code does not charge for idle time, only for requests. You can leave a session open indefinitely without incurring charges as long as you are not sending prompts.

### Rate Limits

Claude Code has rate limits that vary by account tier:

- **Free / trial tier:** Limited requests per day
- **Paid tier:** Higher limits, typically sufficient for daily development work
- **Enterprise tier:** Custom limits

If you hit a rate limit, Claude Code pauses and waits before retrying. You see a message like "Rate limit approaching — slowing down." This is normal and does not lose your work.

### Avoiding Runaway Costs

The most common financial mistake with Claude Code is letting it run unchecked on large-scale refactoring tasks without reviewing costs. Follow these rules:

1. **Start with Sonnet.** Use `claude-sonnet-4-20250514` by default. Only switch to Opus when you genuinely need deeper reasoning.
2. **Use the `-p` flag for quick questions.** One-shot prompts using `claude -p "question"` cost one request and exit immediately, no session overhead.
3. **Set a budget.** Before starting a large refactoring task, decide how many iterations you are willing to pay for.
4. **Monitor your usage.** Check your Anthropic console to see session counts and costs.
5. **Interrupt early.** If Claude goes in the wrong direction, press Ctrl+C immediately rather than letting it waste requests.

### Environment Variables

Claude Code reads these environment variables for configuration:

| Variable | What it controls |
|---|---|
| `ANTHROPIC_API_KEY` | API key for authentication (alternative to OAuth) |
| `CLAUDE_CODE_ALLOWED_TOOLS` | Comma-separated list of pre-approved tools |
| `CLAUDE_CODE_MODEL` | Default model override |
| `CLAUDE_CODE_VERBOSE` | Enable verbose logging (set to `true`) |
| `HOME` or `USERPROFILE` | Where Claude Code stores session tokens |
| `CLAUDE_CODE_DISABLE_TELEMETRY` | Disable usage telemetry (set to `1`) |

### Security Considerations

**Never share your session tokens.** Claude Code stores authentication tokens in your home directory. If someone gains access to your machine, they can use Claude Code as you.

**Review every diff before approving.** Even though Claude Code is powerful, it can make mistakes. Always read the proposed changes before approving them.

**Use a `.gitignore` for secrets.** Claude Code respects `.gitignore` and skips those files. Keep API keys, `.env` files, and credentials in your `.gitignore`.

**Prefer the `-p` flag for sensitive code.** When working with sensitive code, use `claude -p "one task"` to avoid leaving a long session history on disk.

### .claudeignore

Just as `.gitignore` tells git which files to skip, `.claudeignore` tells Claude Code which files to skip when scanning your project. Create a `.claudeignore` file in your project root:

```
*.log
dist/
build/
.env
*.min.js
```

Files in `.claudeignore` are never read into context, which saves context window space and prevents Claude Code from processing generated or sensitive files. Unlike `.gitignore`, Claude Code does not read `.claudeignore` automatically — you must create it explicitly.

### Claude Code Configuration

You can configure Claude Code per-project by creating a `.claude/settings.json` file:

```json
{
  "model": "claude-sonnet-4-20250514",
  "allowedTools": ["Bash", "Edit", "Write", "Read", "Grep", "Glob"],
  "verbose": false
}
```

This is useful for teams that want consistent Claude Code behavior across all developers. The settings file is also where you define custom slash commands and hooks.

### Custom Slash Commands

You can create custom slash commands by adding them to your `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": {
      "Bash": "echo 'Running: $TOOL_INPUT'"
    }
  },
  "slashCommands": [
    {
      "name": "lint",
      "description": "Run linter across the project",
      "prompt": "Run eslint on the src/ directory and fix any auto-fixable issues"
    },
    {
      "name": "deploy",
      "description": "Deploy the current branch",
      "prompt": "Run the deploy script for the current branch. Check git status first, then run npm run build && npm run deploy"
    }
  ]
}
```

Custom slash commands appear in the `/help` output and can be invoked by typing `/lint` or `/deploy` at the prompt. This is a powerful way to standardize common workflows.

## When to Use This / When Not To

**Use Claude Code when:**

- You are exploring an unfamiliar codebase and need to understand how it works
- You need to make a change that touches many files (refactoring, renaming, migrating)
- You are debugging a complex issue that spans multiple modules
- You want to write tests for existing code without manually typing every test
- You are doing repetitive code transformations that would take hours by hand
- You need to generate boilerplate (new components, API endpoints, configuration files)

**Do not use Claude Code when:**

- You are writing code that should come from your own expertise and judgment — Claude Code is a tool, not a replacement for your architectural decisions
- You are working with highly sensitive or proprietary code that cannot be sent to an external API
- You need pixel-perfect UI work — Claude Code has no visual output and cannot verify visual correctness
- You are in an environment without internet access — Claude Code requires a connection to the Anthropic API
- Your task is a quick single-line fix — it is faster to just edit the file yourself than to describe it and approve the change
- You are learning to code for the first time — you should understand the code you are writing before delegating it

The honest trade-off is speed versus control. Claude Code makes you faster by removing context-switching overhead, but it adds the overhead of describing tasks and reviewing changes. For small, well-understood changes, the manual approach is faster. For large, multi-file changes, Claude Code wins.

### Productivity Tips

- **Be specific about files and line numbers** when you know where the change belongs. This reduces Claude Code's search time.
- **Use `/clear` between unrelated tasks.** Starting fresh prevents context pollution from the previous task.
- **One task per session.** For complex projects, split your work into focused sessions rather than one giant session.
- **Review the plan before approving changes.** Claude Code often explains its approach before making edits. Read this explanation — if the plan is wrong, stop it early.
- **Use `--allowedTools` for trusted tasks.** If you are doing routine refactoring that you have verified before, pre-approve tools to speed up the workflow.
- **Combine `-p` with `--output-format json`** for scripting. You can parse JSON output in scripts to build automated code review pipelines or CI checks:

```bash
claude --output-format json -p "check for security vulnerabilities" | jq '.summary'
```

## Mental Model Check

**Question 1:** What is the difference between `claude` and `claude -p "prompt"`?

  `claude` launches an interactive session where you can have a multi-turn conversation with Claude Code, asking follow-up questions and approving changes one by one. `claude -p "prompt"` runs a single prompt and exits immediately — it is non-interactive and useful for one-shot questions or for scripting Claude Code into automated workflows.

---

**Question 2:** Why does Claude Code require your approval before writing files by default?

  Approval is the safety mechanism that prevents unintended changes to your codebase. Claude Code can misinterpret your instructions, propose incorrect implementations, or make edits that conflict with other parts of the system you did not mention. Reviewing each diff before approving ensures you stay in control. You can relax this with `--allowedTools` when you trust the task.

---

**Question 3:** When would you use `claude-sonnet-4-20250514` versus `claude-opus-4-20250514`?

  Use Sonnet for day-to-day coding tasks: debugging, writing tests, simple refactoring, code generation. It is fast and cost-effective. Use Opus only for tasks that require deep architectural reasoning: designing new system components, untangling complex dependency graphs, or planning major multi-step migrations. Opus is slower and more expensive, so reserve it for the hard problems.

---

**Question 4:** What happens when you press Ctrl+C during a Claude Code response?

  The first Ctrl+C interrupts Claude Code mid-response, stopping the current tool execution or text generation. You can then rephrase your instructions. Pressing Ctrl+C a second time exits the session entirely, discarding the conversation history unless you later use `--resume`.

---

**Question 5:** What should you include in a good task description for Claude Code?

  A good task description includes: what you want done, where in the codebase to look, any constraints or preferences (libraries, patterns, exclusions), and acceptance criteria. For example: "Add zod input validation to the signup form in `src/auth/signup.ts`. Do not modify the database layer. Use the existing test structure in `src/auth/__tests__/`. Verify all existing tests still pass."

---

**Question 6:** What is the purpose of `.claudeignore` and how is it different from `.gitignore`?

  `.claudeignore` tells Claude Code which files to skip when scanning your project — keeping generated files, logs, and build artifacts out of context. It is similar to `.gitignore` but is not automatically created. You must create it explicitly. Unlike `.gitignore`, which prevents git from tracking files, `.claudeignore` only prevents Claude Code from reading them into context.

**Question 7:** What does the `--verbose` flag do and when should you use it?

  `--verbose` shows every tool call with full input and output details. Use it when you want to understand exactly what Claude Code is doing behind the scenes, or when debugging unexpected behavior. In normal usage, the default (non-verbose) mode is sufficient — it shows enough information to understand what is happening without overwhelming you with details.
