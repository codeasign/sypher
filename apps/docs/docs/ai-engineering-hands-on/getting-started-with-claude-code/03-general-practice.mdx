## Tier 1 — Beginner

### Exercise 1: Explain a File with a One-Shot Prompt

**Task:** Create a small Python script with a bug in it, then use `claude -p` to ask Claude Code to find the bug. Do not enter interactive mode — use the one-shot flag.

Create a file called `buggy.py`:

```python
def calculate_total(prices):
    total = 0
    for price in prices:
        total += price
    return total

def apply_discount(total, discount):
    return total - discount

def main():
    items = [10.99, 5.49, 3.99, 12.49]
    result = calculate_total(items)
    print(f"Total: ${result}")
    final = apply_discount(result, "5.00")
    print(f"After discount: ${final}")

if __name__ == "__main__":
    main()
```

Now run:

```bash
claude -p "read buggy.py and find any bugs. Explain what is wrong and how to fix it."
```

**Hint:** The bug is a type mismatch. Look at the `apply_discount` function call.

**Expected outcome:** Claude Code identifies that `discount` is passed as a string `"5.00"` instead of a float `5.00`, which causes a `TypeError` at runtime. It suggests changing the call to `apply_discount(result, 5.00)`.

### Solution

Claude Code's response should include:

```
Bug found: In the main() function, apply_discount(result, "5.00") passes
the discount as a string. The apply_discount function expects a number,
so this will raise: TypeError: unsupported operand type(s) for -: 'float' and 'str'

Fix: Change "5.00" to 5.00 on the apply_discount call.
```

This exercise demonstrates the most basic use of Claude Code: asking a question about a file and getting an answer, all without starting an interactive session.

**Hint:** The `/help` command lists all available slash commands. The `/status` command shows git status, directory, and session info.

### Exercise 3: Create a `.claudeignore` File

**Task:** Create a `.claudeignore` file in your practice project and verify that Claude Code respects it.

Create a large log file:

```bash
cd /tmp/claude-code-practice
for i in $(seq 1 1000); do echo "This is log line $i" >> debug.log; done
```

Now run a one-shot prompt:

```bash
claude -p "list all files in the project and their sizes"
```

Note which files Claude Code lists. Now create `.claudeignore`:

```
*.log
```

Run the same prompt again:

```bash
claude -p "list all files in the project and their sizes"
```

**Hint:** The first run lists `debug.log` as a 30KB+ file. After creating `.claudeignore`, the second run should not mention `debug.log` at all.

**Expected outcome:** Before `.claudeignore`, Claude Code reads `debug.log` and may comment on its size. After `.claudeignore`, Claude Code skips the file entirely and does not mention it.

### Solution

```bash
echo "*.log" > /tmp/claude-code-practice/.claudeignore
```

After running the prompt again, Claude Code's output should not include any reference to `debug.log` or `.log` files. This confirms that `.claudeignore` is working correctly. The `debug.log` file still exists on disk — Claude Code simply ignores it.

### Exercise 4: Compare Model Outputs

**Task:** Ask the same question to both Sonnet and Opus, and compare the responses.

```bash
claude --model sonnet -p "explain what a closure is in Python in three sentences"
```

Then:

```bash
claude --model opus -p "explain what a closure is in Python in three sentences"
```

**Hint:** Save each response to a file for easy comparison.

**Expected outcome:** Both models explain closures correctly, but Opus may provide more depth or nuance even within the three-sentence limit.

### Solution

```bash
claude --model sonnet -p "explain what a closure is in Python in three sentences" > sonnet-response.txt
claude --model opus -p "explain what a closure is in Python in three sentences" > opus-response.txt
diff sonnet-response.txt opus-response.txt
```

The `diff` command shows the differences between the two responses. You may notice that Opus produces more detailed explanations with better examples, while Sonnet gives more concise, direct answers. This illustrates the trade-off: Opus for depth, Sonnet for speed.

## Tier 2 — Intermediate

### Exercise 1: Resume a Multi-Turn Debugging Session

**Task:** Start an interactive session, debug a problem across multiple turns, exit, then resume the session and continue debugging.

Create a file with a logic error:

```python
# discount.py
def calculate_discount(price, is_member):
    if is_member:
        discount = 0.2
    else:
        discount = 0.5
    return price * (1 - discount)
```

Start a session:

```bash
cd /tmp/claude-code-practice
claude
```

Run three turns:

```
Turn 1: read discount.py and explain what it does
Turn 2: there's a bug in the discount logic. Can you find it?
Turn 3: fix the bug
```

Approve the fix when Claude Code proposes it. Then exit with `/bye`.

Now resume the session:

```bash
claude --resume
```

Ask:

```
what was the bug you found in the discount.py file?
```

**Hint:** The bug is that non-members get a 50% discount (`0.5`) while members get only 20% (`0.2`). The logic is inverted.

**Expected outcome:** The resumed session remembers the context. Claude Code explains that the discount values were swapped — members should get a higher discount (0.2, or 20% off) versus non-members getting a smaller discount (0.5, or 50% off). After the fix, members get the correct 20% discount and non-members get 0% discount.

### Solution

The bug in the original code is that `is_member` being `True` gives a 20% discount, while `is_member` being `False` gives a 50% discount. This is inverted — non-members get a better deal than members.

The fix Claude Code should propose:

```python
def calculate_discount(price, is_member):
    if is_member:
        discount = 0.2  # 20% off for members
    else:
        discount = 0.0  # 0% off for non-members
    return price * (1 - discount)
```

The key learning from this exercise is that `--resume` preserves the session history, including all file reads, discussions, and changes made. You can pick up exactly where you left off.

### Exercise 2: Use `--verbose` to Understand Tool Execution

**Task:** Run a simple prompt with `--verbose` and observe the tool calls Claude Code makes.

```bash
claude --verbose -p "read buggy.py and count the number of functions"
```

**Hint:** Look for the `Read` tool call in the output. You will see the exact file content that Claude Code reads.

**Expected outcome:** With `--verbose`, you see every tool call, including the `Read` tool that loads the file and the `Grep` or `Glob` tool that may be used to verify the file exists. The output shows the raw input and output of each tool.

### Solution

The verbose output should show something like:

```
[Tool: Read]
  Input: {"path": "/tmp/claude-code-practice/buggy.py"}
  Output: (file content displayed here)
[Tool: Bash]
  Input: grep -c "^def " /tmp/claude-code-practice/buggy.py
  Output: 3
```

The `Read` tool loads the file content. The `Bash` tool runs `grep` to count the function definitions. This transparency is what makes `--verbose` valuable for debugging.

## Tier 3 — Advanced

### Exercise 1: Multi-Step Task Pipeline

**Task:** Build a script that uses Claude Code to perform a sequence of code quality checks on a project, reporting results in a structured format. Use `--output-format json` for machine-readable output.

Create the project structure:

```bash
mkdir -p /tmp/claude-pipeline/src
cd /tmp/claude-pipeline
git init
```

Create a few files with intentional issues:

```python
# src/app.py
import os
import sys

def get_data():
    return os.environ.get("SECRET_KEY")

def process(items):
    result = []
    for i in range(len(items)):
        result.append(items[i] * 2)
    return result

def save(data):
    f = open("output.txt", "w")
    f.write(data)
    # file not closed!

if __name__ == "__main__":
    data = get_data()
    print(f"Processing {len(data)} items")
```

```python
# src/test_app.py
from app import process

def test_process():
    assert process([1, 2, 3]) == [2, 4, 6]
    assert process([]) == []

def test_process_negative():
    assert process([-1, 0, 1]) == [-2, 0, 2]
```

Now write a shell script that runs Claude Code in a pipeline:

```bash
#!/bin/bash
# claude_pipeline.sh

echo "=== Claude Code Pipeline ==="
echo "Step 1: Code review"
claude --output-format json -p "review the code in src/ for bugs and security issues" > review.json

echo "Step 2: Suggest improvements"
claude -p "read src/app.py and suggest 3 specific improvements" > improvements.txt

echo "Step 3: Run tests"
cd src && python -m pytest test_app.py -v 2>&1; cd ..

echo ""
echo "Pipeline complete. Results:"
echo "- Review: review.json"
echo "- Improvements: improvements.txt"
```

**Hint:** The `--output-format json` flag makes Claude Code output structured JSON that you can parse with `jq` or other tools. The security issue is that `SECRET_KEY` should not be read from environment variables and printed.

**Expected outcome:** The pipeline runs three steps sequentially. The review identifies the missing `f.close()` call and the insecure `SECRET_KEY` handling. The improvements suggest using `with open(...)` for file handling. The pytest step runs the tests and reports results.

### Solution

Run the pipeline:

```bash
chmod +x claude_pipeline.sh
./claude_pipeline.sh
```

Expected output:

```
=== Claude Code Pipeline ===
Step 1: Code review
Step 2: Suggest improvements
Step 3: Run tests
=========================
=== test session starts ===
collected 2 items
test_app.py::test_process PASSED
test_app.py::test_process_negative PASSED
=========================
Pipeline complete.
Results:
- Review: review.json
- Improvements: improvements.txt
```

The `review.json` file contains structured output with Claude Code's findings. `improvements.txt` contains the three suggestions. The tests pass, confirming the logic is correct despite the code quality issues.

### Exercise 2: Automated Scripting with Claude Code

**Task:** Write a script that uses Claude Code to generate boilerplate code for a new REST API endpoint, including the route handler, tests, and documentation stub.

Create a script that automates endpoint creation:

```bash
#!/bin/bash
# create_endpoint.sh
# Usage: ./create_endpoint.sh <endpoint_name> <http_method> <path>

NAME=$1
METHOD=$2
PATH=$3

if [ -z "$NAME" ] || [ -z "$METHOD" ] || [ -z "$PATH" ]; then
    echo "Usage: $0 <endpoint_name> <http_method> <path>"
    exit 1
fi

echo "Creating endpoint: $NAME ($METHOD $PATH)"

# Create the route handler
claude -p "create a Python Flask route handler for $METHOD $PATH called $NAME. Include request validation, error handling, and a docstring. Save it to src/routes/${NAME}.py"

# Create the test file
claude -p "read src/routes/${NAME}.py and create a pytest test file for it. Include tests for success case, validation error, and edge cases. Save to tests/test_${NAME}.py"

# Create the documentation stub
claude -p "create a markdown documentation stub for the $NAME endpoint ($METHOD $PATH). Describe the request/response format, parameters, and error codes. Save to docs/endpoints/${NAME}.md"

echo "Endpoint $NAME created."
echo "Files:"
echo "  src/routes/${NAME}.py"
echo "  tests/test_${NAME}.py"
echo "  docs/endpoints/${NAME}.md"
```

**Hint:** Create the required directories first: `mkdir -p src/routes tests docs/endpoints`. The script chains three Claude Code calls, each building on the output of the previous one.

**Expected outcome:** The script creates three files for a new endpoint. The route handler includes validation and error handling. The tests cover the main cases. The documentation stub describes the API.

### Solution

Create the directories and run the script:

```bash
mkdir -p /tmp/claude-pipeline/src/routes /tmp/claude-pipeline/tests /tmp/claude-pipeline/docs/endpoints
cd /tmp/claude-pipeline
./create_endpoint.sh create_user POST /api/users
```

Claude Code creates three files:

`src/routes/create_user.py`:
```python
from flask import Blueprint, request, jsonify

create_user_bp = Blueprint("create_user", __name__)

@create_user_bp.route("/api/users", methods=["POST"])
def create_user():
    """
    Create a new user.

    Expects JSON body with 'name' and 'email' fields.
    Returns the created user ID and a 201 status.
    """
    data = request.get_json()
    if not data:
        return jsonify({"error": "Request body must be JSON"}), 400

    name = data.get("name")
    email = data.get("email")
    if not name or not email:
        return jsonify({"error": "name and email are required"}), 400

    # TODO: Insert into database
    user_id = 1
    return jsonify({"id": user_id, "name": name, "email": email}), 201
```

`tests/test_create_user.py`:
```python
import pytest
from src.routes.create_user import create_user

# Tests for the create_user endpoint
```

`docs/endpoints/create_user.md`:
```markdown
# Create User Endpoint

**Method:** POST
**Path:** /api/users
```

This exercise demonstrates how Claude Code can be integrated into automated workflows, generating consistent code across multiple files based on a single description.

### Exercise 3: Security Audit with Claude Code

**Task:** Use Claude Code to perform a security audit of a codebase, identifying common vulnerabilities and suggesting fixes.

Create a file with multiple security issues:

```python
# vulnerable_app.py
import os
import sqlite3
import hashlib

DB_PASSWORD = "supersecret123"

def get_db():
    return sqlite3.connect("app.db")

def login(username, password):
    db = get_db()
    query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"
    cursor = db.execute(query)
    user = cursor.fetchone()
    db.close()
    return user

def store_password(password):
    return hashlib.md5(password.encode()).hexdigest()

def admin_dashboard(user_id):
    token = request.args.get("token")
    if token == "admin123":
        return "Welcome admin"
    return "Access denied"

def save_file(filename, content):
    with open(f"/uploads/{filename}", "w") as f:
        f.write(content)
```

Run a security audit:

```bash
claude -p "read vulnerable_app.py and identify all security vulnerabilities. For each one, explain the risk and suggest a fix. Rank them by severity."
```

**Hint:** Look for SQL injection, hardcoded passwords, weak hashing, and path traversal vulnerabilities.

**Expected outcome:** Claude Code identifies at least 5 security issues:
1. Hardcoded database password (`DB_PASSWORD`)
2. SQL injection vulnerability in the `login` function (string interpolation)
3. Weak password hashing (MD5 instead of bcrypt or argon2)
4. Hardcoded admin token (`"admin123"`)
5. Path traversal risk in `save_file` (no sanitization of `filename`)

### Solution

Claude Code's response should identify these vulnerabilities ranked by severity:

**Critical:** SQL injection in `login()` — the query uses string interpolation with user input. An attacker can input `' OR 1=1 --` to bypass authentication.

**High:** Hardcoded credentials — `DB_PASSWORD` and `"admin123"` token are visible in source code.

**High:** Weak password hashing — MD5 is trivially reversible. Use `bcrypt` or `hashlib.pbkdf2_hmac` instead.

**Medium:** Path traversal in `save_file()` — a filename like `../../etc/passwd` could overwrite system files. Sanitize with `os.path.basename()`.

**Low:** No input validation — usernames and passwords are not validated before use.

To fix the most critical issue (SQL injection), Claude Code would suggest:

```python
def login(username, password):
    db = get_db()
    query = "SELECT * FROM users WHERE username = ? AND password = ?"
    cursor = db.execute(query, (username, password))
    user = cursor.fetchone()
    db.close()
    return user
```

This exercise demonstrates Claude Code's ability to act as a security reviewer, catching vulnerabilities that might be missed in manual review.
