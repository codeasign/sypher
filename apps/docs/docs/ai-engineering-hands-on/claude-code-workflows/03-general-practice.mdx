```python
# test_old_utils.py
from old_utils import get_user, save_user, delete_user
```

### Exercise 2: Review and Fix a Buggy Module

**Task:** Create a file called `buggy_math.py` with the following intentionally buggy code. Do not fix it yourself — use Claude Code's `/review` and `/fix` commands to find and fix the bugs.

```python
def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)

def divide_list(items, divisor):
    result = []
    for i in range(len(items) + 1):
        result.append(items[i] / divisor)
    return result

def fibonacci(n):
    if n <= 0:
        return []
    if n == 1:
        return [0]
    fib = [0, 1]
    for i in range(2, n):
        fib.append(fib[i-1] + fib[i-2])
    return fib
```

**Hint:** Run `/review --effort high` to get a thorough analysis. The `divide_list` function has an off-by-one error that will cause an `IndexError`. The `factorial` function is correct but has no negative-number guard. Read the review findings carefully — some are bugs, some are suggestions. Use `/fix` to apply only the bug fixes.

<details>
<summary>Solution</summary>

Save the buggy code as `buggy_math.py`, commit it, then launch Claude Code:

```
/review --effort high
```

Claude should find at least these issues:

1. **`divide_list` off-by-one error**: `range(len(items) + 1)` iterates one too many times, causing `IndexError` on the last iteration. Fix: use `range(len(items))`.
2. **`factorial` no negative guard**: `factorial(-1)` causes infinite recursion. Fix: add `if n < 0: raise ValueError(...)`.
3. **`fibonacci` empty string in docstring**: The function could use a docstring.

To fix, run:

```
/fix
```

Claude will propose fixes. Approve the bug fixes (issues 1 and 2). You can reject the docstring suggestion if you prefer. After fixing, verify with tests:

```python
# test_buggy_math.py
from buggy_math import factorial, divide_list, fibonacci

def test_factorial():
    assert factorial(5) == 120
    assert factorial(0) == 1

def test_divide_list():
    assert divide_list([10, 20, 30], 10) == [1.0, 2.0, 3.0]

def test_fibonacci():
    assert fibonacci(5) == [0, 1, 1, 2, 3]
    assert fibonacci(1) == [0]
    assert fibonacci(0) == []
```

```bash
python -m pytest test_buggy_math.py -v
```

All tests should pass.

</details>

### Exercise 3: Automate a Repetitive Check with `/loop`

**Task:** Create a Python script called `status_server.py` that simulates a service that becomes ready after a random delay of 5-15 seconds. Use Claude Code's `/loop` command to poll the server every 3 seconds and report when it is ready.

Use this starter script:

```python
# status_server.py
import random
import time
import sys

READY_AT = time.time() + random.uniform(5, 15)

def is_ready():
    return time.time() >= READY_AT

def health_check():
    if is_ready():
        print("READY")
        return True
    else:
        remaining = int(READY_AT - time.time())
        print(f"NOT_READY (ready in ~{remaining}s)")
        return False

if __name__ == "__main__":
    result = health_check()
    sys.exit(0 if result else 1)
```

**Hint:** Start a background terminal and run `python status_server.py` in a loop to simulate a polling check. Then use Claude Code's `/loop` command with `--max-iterations 10` to try the check up to 10 times and stop when the server is ready. The `/loop` command runs any prompt repeatedly, so you can prompt it to run `python status_server.py` and report the result.

<details>
<summary>Solution</summary>

Save the server script and commit it. In Claude Code, run:

```
/loop --max-iterations 10 3s "Run 'python status_server.py' and report the status. Stop looping when the output is READY."
```

Claude will execute the prompt every 3 seconds. Each iteration:

1. Runs `python status_server.py`
2. Reads the output
3. Reports the status
4. Continues looping if output is `NOT_READY`, stops if `READY`

Expected output over several iterations:

```text
Iteration 1: Output was NOT_READY (ready in ~10s). Continuing.
Iteration 2: Output was NOT_READY (ready in ~7s). Continuing.
Iteration 3: Output was NOT_READY (ready in ~4s). Continuing.
Iteration 4: Output was READY. Stopping.
```

Without `--max-iterations 10`, the loop would run indefinitely (until the model's timeout). The max iterations guardrail prevents runaway costs.

</details>

### Exercise 1: Multi-File Refactor with Permission Scoping

**Task:** Create a small project with three modules that share a common data format. Then use Claude Code to refactor the data format from dictionaries to dataclasses across all three files. Pre-approve only the tools needed for this refactor.

Create these three files:

```python
# order_parser.py
def parse_order(data: dict) -> dict:
    return {
        "order_id": data["id"],
        "customer": data["customer"],
        "items": data["items"],
        "total": sum(item["price"] * item["qty"] for item in data["items"])
    }
```

```python
# order_validator.py
def validate_order(order: dict) -> bool:
    required = ["order_id", "customer", "items", "total"]
    return all(field in order for field in required)
```

```python
# order_reporter.py
def generate_report(orders: list[dict]) -> str:
    lines = ["Order Report", "============="]
    for order in orders:
        lines.append(f"Order {order['order_id']} - {order['customer']}: ${order['total']:.2f}")
    return "\n".join(lines)
```

**Hint:** Launch Claude Code with `--allowedTools` scoped to only `Read, Edit, Write, Grep, Glob` — you do not need `Bash` for this task. To define a dataclass, describe the fields you want: `order_id: str`, `customer: str`, `items: list`, `total: float`.

<details>
<summary>Solution</summary>

Launch Claude Code with scoped permissions:

```bash
claude --allowedTools "Read, Edit, Write, Grep, Glob"
```

Then describe the refactor:

```
/plan Refactor the three order modules to use a shared Order dataclass instead of raw dictionaries.

Create a new file order.py with:
from dataclasses import dataclass

@dataclass
class Order:
    order_id: str
    customer: str
    items: list
    total: float

Then update order_parser.py to return Order(...) instead of a dict.
Update order_validator.py to accept Order and check its attributes.
Update order_reporter.py to accept list[Order] and access attributes directly.
```

Claude will propose a plan with file-by-file changes. Approve it.

After the refactor, create a quick test to verify:

```python
# test_orders.py
from order import Order
from order_parser import parse_order
from order_validator import validate_order
from order_reporter import generate_report

data = {"id": "ORD-001", "customer": "Alice", "items": [{"price": 10, "qty": 2}, {"price": 5, "qty": 1}]}
order = parse_order(data)
assert isinstance(order, Order)
assert order.order_id == "ORD-001"
assert validate_order(order) is True
report = generate_report([order])
assert "Alice" in report
print("All integration checks passed.")
```

Run it: `python test_orders.py`

Expected output: `All integration checks passed.`

</details>

### Exercise 2: Batch Process Files with Claude

**Task:** Create 10 small JSON files in a `data/` directory, each containing a simple record with `id`, `name`, and `value` fields. Use Claude Code to read all files, compute the sum of all `value` fields, and write a summary report.

Create the data files with a small script:

```bash
mkdir data
for i in $(seq 1 10); do
  echo "{\"id\": $i, \"name\": \"item$i\", \"value\": $((RANDOM % 100))}" > data/file_$i.json
done
```

**Hint:** Describe the task as a batch processing workflow. Claude will use `Glob` to find all JSON files, `Read` to open each one, and `Write` to create the summary. You do not need to list every file — let Claude discover them.

<details>
<summary>Solution</summary>

Launch Claude Code and run:

```
Read all JSON files in the data/ directory. Sum the "value" fields across all files. Write a summary report to summary.txt with the total sum and a per-file breakdown.
```

Claude will:
1. Use `Glob` to find `data/*.json`
2. Read each file
3. Parse the JSON content
4. Compute the sum of all `value` fields
5. Write `summary.txt`

After the task, check the output:

```bash
cat summary.txt
```

The file will contain something like:

```text
Batch Summary Report
====================
Total files processed: 10
Sum of all values: 487

Per-file breakdown:
  data/file_1.json: value=42
  data/file_2.json: value=87
  ...
```

Permission note: Each `Read` call requires approval unless you pre-approved it. For 10 files, this means 10 prompts. To reduce friction, pre-approve `Read` with `--allowedTools "Read, Write, Glob"` at startup.

### Exercise 1: Automated Code Quality Pipeline

**Scenario:** You are responsible for code quality on a team of five developers. Every PR should pass linting, type checking, and unit tests before merging. You want to automate this as a Claude Code workflow that runs on every PR.

**Task:** Design a Claude Code workflow that:
1. Checks out a PR branch
2. Runs the linter (flake8 or pylint)
3. Runs the type checker (mypy)
4. Runs the test suite (pytest)
5. Reports all results in a single summary
6. If any step fails, analyzes the failures and proposes fixes

Create a small project with intentional lint, type, and test issues. Then run your workflow against it.

<details>
<summary>Solution</summary>

Create a project with issues:

```python
# messy_module.py
import os
import sys

def process_data(items):  # Missing type hint
    result = []
    for i in range(len(items)):  # Unnecessary range(len(...))
        result.append(items[i] * 2)
    return result

def greet(name):
    print(f"Hello, {name}")  # Should return string, not print
```

```python
# test_messy.py
from messy_module import process_data

class TestMessy:
    def test_process_data(self):
        assert process_data([1, 2, 3]) == [2, 4, 6]  # Will fail: returns [2, 4, 6] (correct)
        # Missing test for empty list
```

Launch Claude Code and run:

```
/plan Create a quality pipeline for this project. I want to:
1. Run flake8 on messy_module.py and record the findings
2. Run mypy on messy_module.py and record the findings
3. Run pytest and record the results
4. Compile a single summary report listing all issues found
5. Then fix all issues and re-run to confirm they are resolved

Output the summary to quality_report.txt.
```

Claude will execute each step sequentially, compiling the report. After fixing, the second run should show zero issues.

Expected quality report content:

```text

  messy_module.py:1:1 - F401 'os' imported but unused
  messy_module.py:2:1 - F401 'sys' imported but unused
```

### Exercise 2: Database Migration Workflow

**Scenario:** You are migrating a legacy flat-file storage system to SQLite. The project has 5 JSON files in a `data/` directory, each containing an array of records. Each record has `id`, `name`, `email`, and `created_at` fields.

**Task:** Use Claude Code to:
1. Read all 5 JSON files and understand the schema
2. Create a SQLite database with an appropriate schema
3. Write a migration script that imports all data from the JSON files into SQLite
4. Write verification queries to confirm the data matches
5. Run the migration and verification

<details>
<summary>Solution</summary>

First, create the JSON data files:

```bash
mkdir data_legacy
for i in $(seq 1 5); do
  python -c "
import json, random
from datetime import datetime, timedelta
users = []
for j in range(10):
    days_ago = random.randint(0, 365)
    users.append({
        'id': (($i - 1) * 10) + j + 1,
        'name': f'user_{(($i - 1) * 10) + j + 1}',
        'email': f'user{(($i - 1) * 10) + j + 1}@example.com',
        'created_at': (datetime.now() - timedelta(days=days_ago)).isoformat()
    })
with open(f'data_legacy/batch_$i.json', 'w') as f:
    json.dump(users, f, indent=2)
"
done
```

Launch Claude Code and run:

```
/plan Migrate the data from the 5 JSON files in data_legacy/ into a SQLite database called users.db.

Steps:
1. Read all JSON files to understand the schema
2. Create users.db with a users table matching the JSON structure
3. Write a migration script that inserts all records into SQLite
4. Verify the migration by querying total count and sampling records
```

Claude will create `migrate.py` and run it. Verify the result:

```bash
sqlite3 users.db "SELECT COUNT(*) FROM users;"
# Expected: 50

sqlite3 users.db "SELECT * FROM users LIMIT 3;"
```

This exercise demonstrates a real-world pattern: using Claude to handle the mechanical parts of a migration (reading source data, generating schema, writing migration code) while you focus on data integrity and edge cases.

</details>

### Exercise 3: Incident Response Runbook

**Scenario:** You are on call for a service that occasionally goes down. You want a Claude Code workflow that automates the initial incident response: checking service health, examining recent logs, checking recent deployments, and collecting diagnostic information into a single report.

**Task:** Create a set of simulated incident artifacts (log files, health check scripts, deployment records) and then build a Claude Code workflow that:
1. Runs a health check script
2. If the health check fails, reads the last 50 lines of the log file
3. Checks the recent git log for recent changes
4. Compiles all information into an incident report
5. Re-runs the health check on a loop until the service recovers

<details>
<summary>Solution</summary>

Create simulated incident artifacts:

```bash
mkdir incident-practice
cd incident-practice
git init
```

```python
# health_check.py
import random
import sys
import time

# Simulates a service that recovers after 30 seconds
START_TIME = time.time()
RECOVERY_TIME = 30  # seconds

def check():
    elapsed = time.time() - START_TIME
    if elapsed < RECOVERY_TIME:
        print(f"Service UNHEALTHY (elapsed: {int(elapsed)}s)")
        sys.exit(1)
    else:
        print("Service HEALTHY")
        sys.exit(0)

if __name__ == "__main__":
    check()
```

```python
# generate_logs.py
import random

log_lines = [
    "2026-07-04 10:00:01 INFO Starting service",
    "2026-07-04 10:00:02 INFO Database connection established",
    "2026-07-04 10:00:03 WARN High memory usage detected: 85%",
    "2026-07-04 10:00:04 ERROR Connection pool exhausted",
    "2026-07-04 10:00:05 ERROR Request timeout on /api/users",
    "2026-07-04 10:00:06 WARN Retrying database connection",
    "2026-07-04 10:00:07 INFO Database reconnected",
    "2026-07-04 10:00:08 WARN Memory usage: 82%",
    "2026-07-04 10:00:09 ERROR Health check failed",
    "2026-07-04 10:00:10 CRITICAL Service entering degraded mode",
]

with open("service.log", "w") as f:
    for line in log_lines:
        f.write(line + "\n")
```

Commit the artifacts:

```bash
git add .
git commit -m "Initial commit with incident practice files"
```

Launch Claude Code and run:

```
/plan Create an incident response workflow:

1. Run python health_check.py to check if the service is healthy
2. If it exits with code 1 (unhealthy), read the last 50 lines of service.log
3. Check git log --oneline -5 for recent deployments
4. Compile everything into incident_report.md
5. Use /loop to re-run the health check every 5 seconds and update the report when the service recovers

The incident report should include:
- Timestamp of the incident
- Health check result
- Relevant log entries (especially ERROR and CRITICAL lines)
- Recent git changes
- Recommended next steps
```

After executing, check the report:

```bash
cat incident_report.md
```

The workflow demonstrates how Claude can automate the first 5 minutes of incident response — collecting data, correlating signals, and producing a structured report — while you focus on the actual remediation.

### Exercise 4 (Optional): Custom Slash Command Workflow

**Scenario:** Your team has a common task that you repeat several times a day: creating a new API endpoint with a standard template (route handler, validation, tests, documentation). You want to build a custom Claude Code slash command that automates this entire workflow.

**Task:** Review the Writing Custom Slash Commands topic, then create a custom slash command that generates a new API endpoint scaffold. The command should:
1. Ask for the endpoint name and HTTP method
2. Create the route handler file with a template
3. Create the test file with starter tests
4. Create a documentation snippet
5. Register the endpoint in the router

<details>
<summary>Solution</summary>

Create a `.claude/commands/` directory and write the custom command:

```bash
mkdir -p .claude/commands
```

```bash
# .claude/commands/new-endpoint
Create a new API endpoint scaffold.

When the user says /new-endpoint <name> <method>:
1. Create the route handler at src/routes/<name>.py with a standard FastAPI template
2. Create the test file at tests/test_<name>.py with GET/POST templates
3. Create a doc snippet at docs/endpoints/<name>.md
4. Register in src/router/__init__.py
```

Launch Claude Code and try it:

```
/new-endpoint users GET
```

Claude should create all the scaffold files. This demonstrates how custom slash commands turn multi-step workflows into single commands that your whole team can use.

</details>
