## Prerequisites

- Python 3.10 or later installed on your machine
- `pip` available (comes with Python)
- Node.js and `npx` available (for the MCP Inspector — install from nodejs.org if missing)
- A terminal open in your workspace directory

## Step 1: Set Up the Project

Create a new directory for your project and set up a virtual environment.

```bash
mkdir weather-mcp-server
cd weather-mcp-server
python -m venv .venv
```

Activate the virtual environment:

- **macOS / Linux:**
  ```bash
  source .venv/bin/activate
  ```

- **Windows (Command Prompt):**
  ```cmd
  .venv\Scripts\activate.bat
  ```

- **Windows (PowerShell):**
  ```powershell
  .venv\Scripts\Activate.ps1
  ```

After activation, your prompt should show `(.venv)` at the start.

Create a `pyproject.toml` file:

```toml
[project]
name = "weather-mcp-server"
version = "0.1.0"
description = "A simple MCP server that returns weather forecasts"
requires-python = ">=3.10"
dependencies = [
    "mcp>=1.0.0",
]
```

Create a `requirements.txt` file (alternative for pip-only workflows):

```
mcp>=1.0.0
```

Install the dependency:

```bash
pip install -r requirements.txt
```

**Expected output:**
```
Collecting mcp>=1.0.0
  ...
Successfully installed mcp-1.x.x pydantic-2.x.x ...
```

**Explanation:** The `mcp` package is the official Python MCP SDK. It pulls in `pydantic` for data validation. This is the only external dependency you need — the server itself is pure Python.

## Step 2: Create the Server with FastMCP

Create a file called `server.py` in the project directory:

```python
from mcp.server import FastMCP

# Create the server instance with a descriptive name
server = FastMCP("WeatherServer")
```

At this point, the server exists but has no functionality. The name `"WeatherServer"` is displayed in the client UI when the server is connected.

Add a `__main__` block so the server can be run directly:

```python
if __name__ == "__main__":
    server.run(transport="stdio")
```

**Full file so far:**

```python
from mcp.server import FastMCP

server = FastMCP("WeatherServer")

if __name__ == "__main__":
    server.run(transport="stdio")
```

**Explanation:** `FastMCP` is the high-level API. It automatically handles JSON-RPC message parsing, tool discovery, and response serialization. The `run()` method starts the event loop on stdio — it reads from stdin and writes to stdout.

## Step 3: Register the `get_forecast` Tool

Add a tool that accepts a city name and returns mock weather data. The SDK infers the input schema from your function signature.

```python
@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city.

    Args:
        city: The name of the city to get the forecast for
        units: Temperature units — "celsius" or "fahrenheit"
    """
    # Normalize the city name
    city_key = city.lower().strip()

    # Mock weather database
    forecasts = {
        "tokyo": {
            "celsius": "18C, partly cloudy with mild breeze",
            "fahrenheit": "64F, partly cloudy with mild breeze",
        },
        "london": {
            "celsius": "12C, overcast with light drizzle",
            "fahrenheit": "54F, overcast with light drizzle",
        },
        "new york": {
            "celsius": "22C, sunny and clear skies",
            "fahrenheit": "72F, sunny and clear skies",
        },
        "paris": {
            "celsius": "15C, light rain expected",
            "fahrenheit": "59F, light rain expected",
        },
        "sydney": {
            "celsius": "26C, warm and humid",
            "fahrenheit": "79F, warm and humid",
        },
    }

    entry = forecasts.get(city_key)
    if not entry:
        return f"Sorry, no forecast data available for {city}. Try: Tokyo, London, New York, Paris, or Sydney."

    return f"Forecast for {city}: {entry.get(units, entry['celsius'])}"
```

**Full file so far:**

```python
from mcp.server import FastMCP

server = FastMCP("WeatherServer")

@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city."""
    city_key = city.lower().strip()
    forecasts = {
        "tokyo": {"celsius": "18C, partly cloudy", "fahrenheit": "64F, partly cloudy"},
        "london": {"celsius": "12C, overcast", "fahrenheit": "54F, overcast"},
        "new york": {"celsius": "22C, sunny", "fahrenheit": "72F, sunny"},
        "paris": {"celsius": "15C, light rain", "fahrenheit": "59F, light rain"},
        "sydney": {"celsius": "26C, warm", "fahrenheit": "79F, warm"},
    }
    entry = forecasts.get(city_key)
    if not entry:
        return f"No forecast available for {city}"
    return f"Forecast for {city}: {entry.get(units, entry['celsius'])}"

if __name__ == "__main__":
    server.run(transport="stdio")
```

**Explanation:** The `@server.tool()` decorator registers the function as a discoverable tool. When the client sends a `tools/list` request, the SDK returns this tool's name, description, and input schema. When the client sends a `tools/call` request with arguments, the SDK validates them and invokes the function. The docstring becomes the tool description that the LLM reads.

The function signature `(city: str, units: str = "celsius")` generates this schema:

```json
{
  "type": "object",
  "properties": {
    "city": {"type": "string", "description": ""},
    "units": {
      "type": "string",
      "description": "",
      "default": "celsius"
    }
  },
  "required": ["city"]
}
```

## Step 4: Register a Resource

Add a resource that exposes weather data at a URI pattern. Resources let the LLM "read" data by URI rather than calling a tool.

```python
@server.resource("weather://forecast/{city}")
async def forecast_resource(city: str) -> str:
    """Weather data for a city, accessible as a resource URI."""
    # Reuse the forecast logic
    city_key = city.lower().strip()
    forecasts = {
        "tokyo": "18C, partly cloudy",
        "london": "12C, overcast",
        "new york": "22C, sunny",
        "paris": "15C, light rain",
        "sydney": "26C, warm",
    }
    entry = forecasts.get(city_key)
    if not entry:
        return f"No data for {city}"
    return f"Current conditions in {city}: {entry}"
```

**Final server.py:**

```python
        "sydney": {"celsius": "26C, warm", "fahrenheit": "79F, warm"},
    }
    entry = forecasts.get(city_key)
    if not entry:
        return f"No forecast available for {city}"
    return f"Forecast for {city}: {entry.get(units, entry['celsius'])}"

@server.resource("weather://forecast/{city}")

```

**Explanation:** The `@server.resource()` decorator takes a URI pattern with `{city}` as a path parameter. When the client reads `weather://forecast/tokyo`, the SDK extracts `city=tokyo` and calls the handler. Resources give the LLM a file-like way to access data, useful for information lookups that do not need action parameters.

## Step 5: Run the Server via stdio

Start the server:

```bash
python server.py
```

The terminal will appear to hang with no output. This is correct — the server is waiting for JSON-RPC messages on stdin.

To verify it is running, open another terminal in the same directory and test it manually:

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' | python server.py
```

**Expected output:**
```json
{"jsonrpc":"2.0","id":1,"result":{"tools":[{"name":"get_forecast","description":"Get the current weather forecast for a given city.","inputSchema":{"type":"object","properties":{"city":{"type":"string"},"units":{"type":"string","default":"celsius"}},"required":["city"]}}]}}
```

Press **Ctrl+C** to stop the server after testing.

**Explanation:** The server reads one JSON-RPC message from stdin, processes it, writes the response to stdout, and then exits. In normal operation, the client keeps stdin open and sends multiple messages. The `run()` method enters an event loop that handles this continuous communication.

## Step 6: Test with MCP Inspector

The MCP Inspector is a browser-based debugging tool for MCP servers. It connects to your server and lets you manually invoke tools and read resources.

In a new terminal (keep your server stopped), run:

```bash
npx @anthropic/mcp-inspector
```

**Expected output:**
```
Starting MCP Inspector...
  ➡  Web UI: http://localhost:5173
```

Open `http://localhost:5173` in your browser. You will see:

1. A text input labeled **Command** — enter `python` (or the full path to your Python executable).
2. A text input labeled **Arguments** — enter `server.py` (or the full path to your server file).
3. Click **Connect**.

After connecting, the Inspector shows:

- **Tools tab:** Lists the `get_forecast` tool with its description and input schema.
- **Resources tab:** Lists the `weather://forecast/{city}` resource template.
- **Prompt tab:** (empty — no prompts were registered).

To test the tool:

1. Click the **Tools** tab.
2. Find `get_forecast` and click **Call**.
3. In the arguments field, enter:
   ```json
   {"city": "Tokyo", "units": "celsius"}
   ```
4. Click **Execute**.

**Expected result:**
```json
{
  "content": [
    {
      "type": "text",
      "text": "Forecast for Tokyo: 18C, partly cloudy"
    }
  ]
}
```

To test the resource:

1. Click the **Resources** tab.
2. Find `weather://forecast/{city}`.
3. Enter `weather://forecast/paris`.
4. Click **Read**.

**Expected result:**
```json
{
  "contents": [
    {
      "uri": "weather://forecast/paris",
      "mimeType": "text/plain",
      "text": "Current conditions in paris: 15C, light rain"
    }
  ]
}
```

**Explanation:** The MCP Inspector starts a web UI that acts as an MCP client. It spawns your server as a subprocess, connects to its stdio, and sends JSON-RPC messages when you click buttons. This is the primary debugging workflow during MCP server development — you can test tools and resources without setting up an LLM client.

## Step 7: Verify Tools Are Discoverable

The MCP Inspector already verified discoverability in Step 6 when it listed the tool after connecting. You can also verify programmatically by running a quick test.

Create `test_discover.py` in the same directory:

```python
"""Quick test script to verify MCP server tool discovery."""
import subprocess
import json

def test_tool_discovery():
    result = subprocess.run(
        ["python", "server.py"],
        input=json.dumps({
            "jsonrpc": "2.0",
            "id": 1,
            "method": "tools/list",
            "params": {}
        }),
        capture_output=True,
        text=True,
        timeout=5,
    )
    response = json.loads(result.stdout)
    tools = response.get("result", {}).get("tools", [])
    tool_names = [t["name"] for t in tools]
    assert "get_forecast" in tool_names, f"Expected get_forecast, got {tool_names}"
    print(f"Tool discovery passed. Found tools: {tool_names}")

if __name__ == "__main__":
    test_tool_discovery()
```

Run it:

```bash
python test_discover.py
```

**Expected output:**
```
Tool discovery passed. Found tools: ['get_forecast']
```

**Explanation:** The test sends a single JSON-RPC `tools/list` request to the server and checks that the response includes the expected tool name. This programmatic verification is useful for automated testing in CI pipelines.
