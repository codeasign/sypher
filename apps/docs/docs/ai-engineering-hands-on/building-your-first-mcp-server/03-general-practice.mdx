## Tier 1: Beginner Exercises

These exercises extend the basic Weather MCP Server with additional tools and resources. Complete the guided practice before starting these.

### Exercise 1.1: Add a `get_alerts` Tool

Add a new tool that returns weather alerts for a given region.

**Requirements:**
- Tool name: `get_alerts`
- Parameters: `region: str` (required), `severity: str` (optional, default `"all"`)
- Returns a list of active alerts with type, severity, and description
- Include at least three regions with mock data

**Starter code:**

```python
@server.tool()
async def get_alerts(region: str, severity: str = "all") -> str:
    """Get active weather alerts for a region."""
    alerts_db = {
        "california": [
            {"type": "heat advisory", "severity": "moderate", "description": "Temperatures expected to reach 38C"},
            {"type": "fire danger", "severity": "high", "description": "Elevated fire risk due to dry conditions"},
        ],
        "florida": [
            {"type": "thunderstorm watch", "severity": "moderate", "description": "Scattered thunderstorms expected"},
            {"type": "flood advisory", "severity": "elevated", "description": "Urban flooding possible in low-lying areas"},
        ],
        "colorado": [
            {"type": "winter storm warning", "severity": "high", "description": "Heavy snowfall expected above 2500m"},
        ],
    }
    alerts = alerts_db.get(region.lower().strip(), [])
    if severity != "all":
        alerts = [a for a in alerts if a["severity"] == severity]
    if not alerts:
        return f"No alerts for {region}" + (f" at {severity} severity" if severity != "all" else "")
    lines = [f"Active alerts for {region}:"]
    for a in alerts:
        lines.append(f"  - {a['type']} ({a['severity']}): {a['description']}")
    return "\n".join(lines)
```

**Verify with MCP Inspector:**
- Call `get_alerts` with `{"region": "California"}` — expect two alerts.
- Call `get_alerts` with `{"region": "California", "severity": "high"}` — expect one alert.
- Call `get_alerts` with `{"region": "Texas"}` — expect the no-alerts response.

@server.tool()
async def get_uv_index(city: str, date: str = "today") -> str:

### Exercise 2.1: Add Input Validation

Wrap your tool parameters with validation logic. Validate that `city` and `region` are not empty, do not contain special characters, and are within a reasonable length.

**Requirements:**
- Create a `validate_city_name(name: str) -> bool` function
- Reject empty strings, names shorter than 2 characters, names with digits or special characters
- Reject names longer than 50 characters
- Return a clear error message when validation fails

**Starter code:**

```python
import re

def validate_city_name(name: str) -> tuple[bool, str]:
    """Validate a city name. Returns (is_valid, error_message)."""
    if not name or not name.strip():
        return False, "City name cannot be empty"
    cleaned = name.strip()
    if len(cleaned) < 2:
        return False, f"City name '{cleaned}' is too short (min 2 characters)"
    if len(cleaned) > 50:
        return False, f"City name is too long ({len(cleaned)} characters, max 50)"
    if not re.match(r"^[a-zA-Z\s\-\.']+$", cleaned):
        return False, f"City name '{cleaned}' contains invalid characters (letters, spaces, hyphens, and periods only)"
    return True, ""

@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city."""
    is_valid, error = validate_city_name(city)
    if not is_valid:
        return f"Validation error: {error}"
    # ... rest of the function
```

**Verify:**
- Call `get_forecast` with an empty city — expect a validation error message.
- Call `get_forecast` with `"a"` — expect a "too short" error.
- Call `get_forecast` with a string containing digits — expect an invalid characters error.

### Exercise 2.2: Structured Error Handling

Replace plain string error returns with structured error responses using the MCP error format.

**Requirements:**
- Define a `McpError` class or use the SDK's error types
- Return errors with error codes: `-32602` for invalid params, `-32603` for internal errors
- Catch unexpected exceptions in your tool handlers and return structured errors

**Starter code:**

```python
# Import error types from the SDK — use try/except for import guard
try:
    from mcp.types import ErrorData
    HAS_MCP_ERRORS = True
except ImportError:
    HAS_MCP_ERRORS = False

class ToolError(Exception):
    """Custom error for tool-level issues."""
    def __init__(self, message: str, code: int = -32602):
        self.code = code
        self.message = message
        super().__init__(self.message)

@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city."""
    try:
        is_valid, error = validate_city_name(city)
        if not is_valid:
            raise ToolError(error, code=-32602)

        forecasts = {
            "tokyo": {"celsius": "18C, partly cloudy", "fahrenheit": "64F, partly cloudy"},
            "london": {"celsius": "12C, overcast", "fahrenheit": "54F, overcast"},
            "new york": {"celsius": "22C, sunny", "fahrenheit": "72F, sunny"},
        }
        entry = forecasts.get(city.lower().strip())
        if not entry:
            return f"No forecast available for {city}"

        return f"Forecast for {city}: {entry.get(units, entry['celsius'])}"

    except ToolError as e:
        # Re-raise as a structured error
        raise e
    except Exception as e:
        raise ToolError(f"Internal error: {str(e)}", code=-32603)
```

**Verify:**
- Call with invalid input — expect a structured error response in the Inspector rather than a plain text message.
- The Inspector shows errors in red with the error code.

### Exercise 2.4: Rate Limiting

Add a simple rate limiter that prevents a single tool from being called more than 10 times per minute.

**Requirements:**
- Track timestamps of calls per tool name
- If more than 10 calls happen in a 60-second window, return a rate-limit error
- Reset the counter after 60 seconds of inactivity
- Log a warning when rate limits are hit

**Starter code:**

```python
import time
from collections import defaultdict
from collections import deque

class RateLimiter:
    """Simple sliding-window rate limiter per tool."""

    def __init__(self, max_calls: int = 10, window_seconds: int = 60):
        self.max_calls = max_calls
        self.window_seconds = window_seconds
        self.calls: dict[str, deque] = defaultdict(deque)

    def check(self, tool_name: str) -> tuple[bool, str]:
        """Check if the tool call is allowed. Returns (allowed, message)."""
        now = time.time()
        window = self.calls[tool_name]

        # Remove old entries outside the window
        while window and window[0] < now - self.window_seconds:
            window.popleft()

        if len(window) >= self.max_calls:
            oldest = window[0]
            retry_after = int(self.window_seconds - (now - oldest))
            return False, f"Rate limit exceeded for {tool_name}. Retry in {retry_after}s"

        window.append(now)
        return True, ""

rate_limiter = RateLimiter()

@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city."""
    allowed, msg = rate_limiter.check("get_forecast")
    if not allowed:
        logger.warning(f"Rate limit hit: get_forecast")
        return f"Error: {msg}"
    # ... rest of the function
```

**Verify:**
- Call `get_forecast` 11 times rapidly — the 11th call should return a rate-limit error message.
- Wait 60 seconds (or reduce the window for testing) — calls should succeed again.

## Tier 3: Advanced Exercises

These exercises add remote access, authentication, and real external API integration.

### Exercise 3.1: Switch to SSE Transport

Modify the server to use SSE (Server-Sent Events) transport instead of stdio, enabling remote access from network clients.

**Requirements:**
- Change `transport` from `"stdio"` to `"sse"` in the `run()` call
- Set a custom `host` and `port` (e.g., `0.0.0.0:8000`)
- Test connecting from another machine on the same network
- Add CORS headers if needed for browser-based clients

**Starter code:**

```python
from mcp.server import FastMCP

server = FastMCP(
    "WeatherServer",
    host="0.0.0.0",
    port=8000,
)

# ... tool and resource registrations ...

if __name__ == "__main__":
    server.run(transport="sse")
```

Run the server:

```bash
python server.py
```

**Expected output:**
```
INFO:     Started server process [12345]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8000
```

Test from another terminal on the same machine:

```bash
curl -N http://localhost:8000/sse
```

**Expected output:**
```
data: {"jsonrpc":"2.0","method":"endpoint","params":{"uri":"http://localhost:8000/message"}}
```

This confirms the SSE endpoint is active and waiting for a client to connect.

**Explanation:** SSE transport uses HTTP Server-Sent Events for the server-to-client direction and HTTP POST for the client-to-server direction. The server runs an HTTP server (using uvicorn under the hood) and listens for connections. Any MCP client on the network can connect to it if the port is accessible.

### Exercise 3.3: Call a Real External Weather API

Replace mock data with calls to a real weather API, handling HTTP requests, response parsing, and error cases.

**Requirements:**
- Use the `httpx` library to call a real weather API (e.g., Open-Meteo which is free and requires no API key)
- Parse the JSON response and format it for the LLM
- Handle network errors, timeouts, and invalid responses gracefully
- Add a timeout of 10 seconds for each external call

**Starter code:**

```python
import httpx

OPEN_METEO_URL = "https://api.open-meteo.com/v1/forecast"

# City to coordinates mapping
CITY_COORDS = {
    "tokyo": {"lat": 35.6762, "lon": 139.6503},
    "london": {"lat": 51.5074, "lon": -0.1278},
    "new york": {"lat": 40.7128, "lon": -74.0060},
    "paris": {"lat": 48.8566, "lon": 2.3522},
    "sydney": {"lat": -33.8688, "lon": 151.2093},
}

@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the current weather forecast for a given city using real data."""
    city_key = city.lower().strip()
    coords = CITY_COORDS.get(city_key)
    if not coords:
        known = ", ".join(CITY_COORDS.keys())
        return f"Unknown city '{city}'. Supported cities: {known}"

    temp_unit = "celsius" if units == "celsius" else "fahrenheit"

    try:
        async with httpx.AsyncClient(timeout=10.0) as client:
            response = await client.get(
                OPEN_METEO_URL,
                params={
                    "latitude": coords["lat"],
                    "longitude": coords["lon"],
                    "current_weather": "true",
                    "temperature_unit": temp_unit,
                },
            )
            response.raise_for_status()
            data = response.json()
            weather = data.get("current_weather", {})
            temp = weather.get("temperature", "N/A")
            wmo_code = weather.get("weathercode", 0)
            conditions = {
                0: "clear sky", 1: "mainly clear", 2: "partly cloudy",
                3: "overcast", 45: "foggy", 48: "depositing rime fog",
                51: "light drizzle", 53: "moderate drizzle", 55: "dense drizzle",
                61: "slight rain", 63: "moderate rain", 65: "heavy rain",
                71: "slight snow", 73: "moderate snow", 75: "heavy snow",
                80: "slight rain showers", 81: "moderate rain showers", 82: "violent rain showers",
                95: "thunderstorm", 96: "thunderstorm with slight hail", 99: "thunderstorm with heavy hail",
            }.get(wmo_code, f"code {wmo_code}")
            return f"Current conditions in {city}: {temp} degrees {units}, {conditions}"

    except httpx.TimeoutException:
        return f"Error: Weather API timed out for {city}. Try again later."
    except httpx.HTTPStatusError as e:
        return f"Error: Weather API returned status {e.response.status_code} for {city}."
    except Exception as e:
        return f"Error fetching weather data: {str(e)}"
```
