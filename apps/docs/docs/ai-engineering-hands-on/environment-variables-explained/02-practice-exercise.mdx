# Environment Variables Explained from Zero -- Practice Exercise

⏱ 15 minutes · Difficulty: Beginner

In this exercise you will set up a real `.env` file, write a Python script that reads environment variables, and verify that secrets stay out of your source code.

## Setup

### What you need

- Python 3.8 or later installed (check with `python --version`)
- A terminal (Command Prompt, PowerShell, Terminal.app, or any terminal emulator)
- A text editor (VS Code, PyCharm, or even Notepad works)
- An API key from any AI provider (optional -- you can complete the exercise with a fake key)
- An internet connection (only needed to install `python-dotenv`)

### Verify your Python installation

Open your terminal and run:

```bash
python --version
```

You should see output like:

```
Python 3.12.0
```

The exact version number does not matter as long as it starts with 3. If you see a `command not found` error, try `python3 --version` instead. On some systems, the command is `python3` and `python` points to Python 2.

### Create a working directory

Open your terminal and run these commands:

```bash
mkdir env-vars-exercise
cd env-vars-exercise
```

This creates a folder called `env-vars-exercise` and moves into it. All your files for this exercise go in this folder.

### Create a .gitignore file

Create a file called `.gitignore` in the `env-vars-exercise` folder with this content:

```bash
# .gitignore for env-vars-exercise
.env
__pycache__/
*.pyc
```

The `.env` line tells Git to ignore any file named `.env`.

### Install python-dotenv

Run this in your terminal (make sure you are in the `env-vars-exercise` folder):

```bash
pip install python-dotenv
```

You should see output like:

```
Collecting python-dotenv
  Downloading python_dotenv-1.0.1-py3-none-any.whl (19 kB)
Installing collected packages: python-dotenv
Successfully installed python-dotenv-1.0.1
```

The version number may differ slightly, but the key message is "Successfully installed".

## The Build

### Step 1: Create your .env file

Create a file called `.env` in the `env-vars-exercise` folder with this content:

```bash
# My environment variables
MY_APP_NAME=EnvVarExplorer
MY_FAVORITE_NUMBER=42
GREETING=Hello from the environment
API_KEY=sk-placeholder-do-not-use
```

Every line in a `.env` file follows the format `KEY=VALUE`. Lines that start with `#` are comments and are ignored. No spaces around the `=` sign.

### Step 2: Write a Python script that reads one variable directly

Create a file called `step2_direct.py` with this content:

```python
import os

# Try to read a variable that may or may not exist
app_name = os.environ.get("MY_APP_NAME", "DefaultApp")
print(f"App name from environment: {app_name}")
```

Run it:

```bash
python step2_direct.py
```

Expected output:

```
App name from environment: DefaultApp
```

Notice the output is `DefaultApp`, not `EnvVarExplorer`. That is because you have NOT loaded the `.env` file yet. Python only sees variables that were set in the shell or inherited from the parent process. The `.env` file is just a file on disk -- Python does not read it automatically.

### Step 3: Write a script that uses python-dotenv to load the .env file

Create a file called `step3_loaded.py` with this content:

```python
import os
from dotenv import load_dotenv

# Load variables from .env into the environment
load_dotenv()

# Now read the variables
app_name = os.environ.get("MY_APP_NAME", "DefaultApp")
favorite_number = os.environ.get("MY_FAVORITE_NUMBER", "0")
greeting = os.environ.get("GREETING", "No greeting found")

print(f"App name: {app_name}")
print(f"Favorite number: {favorite_number}")
print(f"Greeting: {greeting}")
```

Run it:

```bash
python step3_loaded.py
```

Expected output:

```
App name: EnvVarExplorer
Favorite number: 42
Greeting: Hello from the environment
```

Now Python sees the variables because `load_dotenv()` read them from the `.env` file and added them to `os.environ`.

### Step 4: Write a script that handles a missing variable gracefully

Not every variable you try to read will exist. A robust script handles missing variables without crashing.

Create a file called `step4_robust.py`:

```python
import os
from dotenv import load_dotenv

load_dotenv()

def get_env_or_warn(var_name):
    """Read an env var and warn if it is missing."""
    value = os.environ.get(var_name)
    if value is None:
        print(f"WARNING: {var_name} is not set")
        return None
    return value

# Read a variable that exists
api_key = get_env_or_warn("API_KEY")
print(f"API_KEY: {api_key}")

# Read a variable that does NOT exist
missing = get_env_or_warn("DATABASE_URL")
print(f"DATABASE_URL: {missing}")
```

Run it:

```bash
python step4_robust.py
```

Expected output:

```
WARNING: DATABASE_URL is not set
API_KEY: sk-placeholder-do-not-use
DATABASE_URL: None
```

The script did not crash when `DATABASE_URL` was missing. It printed a warning and continued.

### Step 5: Write a script that requires a variable and crashes if missing

Sometimes a variable is essential. If your database password is not set, you cannot connect. Crashing with a clear error message is better than failing mysteriously halfway through.

Create a file called `step5_required.py`:

```python
import os
from dotenv import load_dotenv

load_dotenv()

# Get a required variable
api_key = os.environ.get("API_KEY")
if not api_key:
    raise ValueError(
        "API_KEY is not set. "
        "Create a .env file with API_KEY=your-key and run load_dotenv()"
    )

print(f"API key loaded. Last 4 characters: ...{api_key[-4:]}")
```

Run it:

```bash
python step5_required.py
```

Expected output:

```
API key loaded. Last 4 characters: ...use
```

Now change your `.env` file so `API_KEY` is missing or commented out:

```bash
# API_KEY=sk-placeholder-do-not-use
```

Run the script again:

```bash
python step5_required.py
```

Expected output:

```
Traceback (most recent call last):
  File "step5_required.py", line 10, in <module>
    raise ValueError(
ValueError: API_KEY is not set. Create a .env file with API_KEY=your-key and run load_dotenv()
```

This is a deliberate crash with a helpful message. The script tells you exactly what went wrong and how to fix it.

Restore your `.env` file by uncommenting the `API_KEY` line before moving on.

### Step 6: Set a variable from the terminal and see it override .env

What happens when you set the same variable in both the terminal and the `.env` file? The terminal wins, because it was set before `load_dotenv()` runs.

Create a file called `step6_override.py`:

```python
import os
from dotenv import load_dotenv

# First, read what is already in the environment
pre_load = os.environ.get("GREETING", "not set yet")
print(f"Before load_dotenv: GREETING={pre_load}")

# Now load the .env file
load_dotenv()

# Read again after loading
post_load = os.environ.get("GREETING", "still not set")
print(f"After load_dotenv: GREETING={post_load}")
```

First run it normally:

```bash
python step6_override.py
```

Expected output:

```
Before load_dotenv: GREETING=not set yet
After load_dotenv: GREETING=Hello from the environment
```

Now set the variable in your terminal and run again:

**macOS/Linux:**
```bash
export GREETING="I win from the terminal"
python step6_override.py
```

**PowerShell (Windows):**
```powershell
$env:GREETING="I win from the terminal"
python step6_override.py
```

Expected output:

```
Before load_dotenv: GREETING=I win from the terminal
After load_dotenv: GREETING=I win from the terminal
```

The value set in the terminal takes precedence. `load_dotenv()` will not overwrite a variable that already exists in the environment. This is by design: it lets you override `.env` values temporarily for testing.

## Common Errors

### Error 1: ModuleNotFoundError: No module named 'dotenv'

```
ModuleNotFoundError: No module named 'dotenv'
```

**Cause:** You did not install `python-dotenv`.

**Fix:** Run `pip install python-dotenv` in your terminal. Make sure you are in the same virtual environment or Python installation where you run your script.

### Error 2: KeyError when reading an env var

```
KeyError: 'API_KEY'
```

**Cause:** You used `os.environ["API_KEY"]` instead of `os.environ.get("API_KEY")`, and the variable was not set.

**Fix:** Either set the variable in your `.env` file and call `load_dotenv()` before reading it, or use `.get()` with a default value.

### Error 3: Variables from .env appear as None or default values

```
My env var is: DefaultValue (when you expected the real value)
```

**Cause:** You called `load_dotenv()` too late -- after you already tried to read the variable. Or you forgot to call `load_dotenv()` entirely.

**Fix:** Verify that `load_dotenv()` is called at the very top of your script, before any `os.environ.get()` calls. Check that your `.env` file is in the same directory where you run the script.

### Error 4: .env file accidentally committed to GitHub

You pushed to GitHub and later noticed your `.env` file appeared in the remote repository.

**Cause:** You did not add `.env` to `.gitignore` before your first commit.

**Fix:** This is serious. Rotate any API keys that were exposed. Then add `.env` to `.gitignore`, remove the file from git tracking with `git rm --cached .env`, commit, and push. But assume any secret in that commit is compromised.

```bash
echo ".env" >> .gitignore
git rm --cached .env
git add .gitignore
git commit -m "Remove .env from tracking and add to .gitignore"
git push
```

Then regenerate all API keys that were in the exposed `.env` file.

### Error 5: TypeError when using an env var as a number

```
Traceback (most recent call last):
  File "app.py", line 8, in <module>
    total = count + 5
TypeError: can only concatenate str (not "int") to str
```

**Cause:** You read a number from an environment variable and tried to use it as an integer, but `os.environ` always returns strings. `os.environ["COUNT"]` gives you `"5"` (a string), not `5` (an integer).

**Fix:** Convert the value explicitly with `int()` or `float()`:

```python
count = int(os.environ.get("COUNT", "0"))
total = count + 5  # now works, both are integers
```

### Error 6: .env file is in the wrong directory

You placed `.env` in your home directory or Desktop folder, but your script is running from a different folder. `load_dotenv()` looks for `.env` in the current working directory -- the folder you are in when you run the script, not the folder where the `.env` file physically lives.

**Fix:** Either move `.env` into the same directory where you run the script, or pass the full path to `load_dotenv()`:

```python
from dotenv import load_dotenv
load_dotenv("/absolute/path/to/your/.env")
```

To check which directory Python considers "current":

```python
import os
print("Current working directory:", os.getcwd())
```

Place your `.env` file there, and `load_dotenv()` will find it.

## What You Just Built

You built a complete, reusable pattern for keeping secrets out of your source code:

- A `.gitignore` file that protects `.env` from accidental commits
- A `.env` file that stores your secrets and configuration in plain text
- A Python script that uses `load_dotenv()` to load those variables into `os.environ`
- A robust pattern for handling missing variables with clear error messages instead of silent failures
- Knowledge of how terminal-set variables override `.env` values, giving you runtime control over configuration
- Experience converting string environment variables to the correct Python types (int, bool)

This is the exact same pattern used by professional AI engineers, web developers, and DevOps teams. Every major framework and deployment platform -- Django, Flask, FastAPI, Heroku, Vercel, AWS Lambda, Docker -- uses environment variables for secrets and configuration.

From this point forward, you should never hardcode an API key in your source code again. Every project you start should begin with three files: `.gitignore`, `.env` (never committed), and your main Python script that calls `load_dotenv()` at the top.

### Quick checklist for every new project

1. Create the project directory
2. Create `.gitignore` with `.env` in it
3. Run `git init` to turn it into a git repository
4. Create `.env` with your secrets
5. Write your Python code with `os.environ.get()` calls
6. Call `load_dotenv()` at the top of your script
7. Run `git check-ignore .env` to confirm the file is protected
8. Create your first commit knowing your secrets are safe
