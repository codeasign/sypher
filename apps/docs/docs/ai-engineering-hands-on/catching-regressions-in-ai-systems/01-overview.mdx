Your team updates the system prompt for your customer support bot — a small change, adding "Be concise" to the instructions. You deploy. For the next three days, everything looks fine. Then your support ticket volume spikes by 30%. Customers are complaining that the bot sounds rude. You check — yes, the responses are shorter, but more importantly, the bot has stopped using polite openings and closings. The system prompt change — which looked harmless — caused the model to drop a behavioral pattern that was never explicitly in the prompt but emerged from the original instruction set.

This is a regression in an AI system. Unlike a traditional software regression where a function starts returning wrong values, an AI regression is a degradation in quality along one or more behavioral dimensions. The output is still valid, grammatical, and on-topic — just worse in a way that is hard to detect without systematic evaluation.

## The Concept

**A regression in an AI system is a statistically significant drop in quality along a measured dimension between two versions of the system.** Think of it like a golfer's performance. A golfer does not miss every shot when they are in a slump — they hit slightly more slices, slightly fewer greens in regulation, putt slightly worse. No single shot tells you they have regressed, but tracking scores over 18 holes reveals the trend. For AI, eval scores across a golden dataset are your scorecard, and a regression is a statistically significant downward trend across multiple rounds.

## Regression Detection Methods Reference

| Method | What It Detects | How to Use It |
|---|---|---|
| `scipy.stats.ttest_ind` | Drop in mean score between two versions | `t, p = ttest_ind(baseline_scores, candidate_scores)` |
| `scipy.stats.levene` | Increase in variance (more unpredictable outputs) | `stat, p = levene(baseline_scores, candidate_scores)` |
| `numpy.percentile` comparison | Tail degradation — are the worst outputs getting worse? | `np.percentile(scores, 5)` compared across versions |
| `scipy.stats.ks_2samp` | Any change in the full distribution shape (not just mean) | `stat, p = ks_2samp(baseline, candidate)` |
| Effect size (Cohen's d) | Practical significance — is the drop meaningful? | `d = (mean1 - mean2) / pooled_std` |
| Confusion matrix per category | Changes in error types (hallucination vs omission vs verbosity) | `sklearn.metrics.confusion_matrix(categories)` |
| `run chart` / control chart | Visual tracking of scores over time with upper/lower control limits | Plot scores with UCL = mean + 3*std and LCL = mean - 3*std |
| `pytest.approx` with tolerance regression | Floating-point comparison against a baseline | `assert current_mean == pytest.approx(baseline_mean, abs=0.05)` |

## How It Actually Works — Step by Step

### Step 1: Establish a Baseline

Before you can detect regressions, you must have a baseline — a set of scores from your current production system. Run your full eval suite on the golden dataset with the current model version and prompt configuration. Record the mean, standard deviation, and percentiles for each dimension. This is your anchor.

### Step 2: Run the Candidate

When you have a new version to test — a model update, a prompt change, a different temperature setting — you run the exact same eval suite on the exact same golden dataset. The only variable that changes is the system itself.

### Step 3: Compare Distributions, Not Points

This is the critical step. You do not compare individual scores — you compare the distributions. A regression can take three forms:

- **Mean shift:** The average quality dropped. This is what most people check first.
- **Variance increase:** The average stayed the same, but the outputs are less predictable. Some are better, some are much worse.
- **Tail degradation:** The worst outputs got worse. The mean might be unchanged, but the 5th percentile dropped. This is often the first sign of trouble.

### Step 4: Run Statistical Tests

Use Welch's t-test to check for mean shift. Use Levene's test to check for variance change. Use the Kolmogorov-Smirnov test to check for any distributional change. A p-value below 0.05 suggests the change is real, not noise.

### Step 5: Evaluate Practical Significance

A statistically significant difference is not necessarily a practically important one. If the mean dropped from 0.870 to 0.869 with p < 0.001, that is statistically significant but meaningless. Compute Cohen's d (effect size) to understand the magnitude. A d of 0.2 is small, 0.5 is medium, 0.8 is large.

### Step 6: Decide and Investigate

If a regression is both statistically significant and practically significant, block the deployment and investigate. Look at the specific prompts where scores dropped the most — those are your regression failure modes. Fix the prompt or model configuration, re-run, and confirm the regression is resolved.

## Cost, Limits, and Safety

**Cost of regression testing:** Each regression check requires running the full eval suite on both the baseline and the candidate. If your golden dataset has 200 prompts and each eval costs 3 judge calls (dimensions), that is 600 API calls per version. Run each version 3 times for statistical stability and you are at 1,800 calls per regression check. Budget for this — regression testing is not free.

**Choosing your golden dataset size:** Start with 50-100 prompts. Expand to 200-500 prompts as your system matures. Beyond 500, the marginal value of each additional prompt decreases while cost increases linearly. Focus on prompt quality — 50 well-crafted edge cases catch more regressions than 500 randomly sampled prompts.

**False positives from statistical tests:** With many dimensions and frequent testing, you will get false positives by chance. Apply a Bonferroni correction when testing multiple dimensions: divide your alpha threshold by the number of dimensions. If you test 4 dimensions, use alpha = 0.05 / 4 = 0.0125.

**Regression in safety dimensions:** If a safety score drops (even a small amount), treat it as a high-severity regression. A model that becomes slightly more likely to produce harmful outputs is unacceptable — deploy only after confirming safety scores are stable or improved.

## When to Use This / When Not To

**Use regression detection when:** You are making any change to your AI system that will be deployed to production — prompt updates, model version changes, temperature adjustments, RAG configuration changes, or system instruction modifications. Every change is a candidate for regression.

**Do not rely solely on regression detection when:** You are making a major architectural change (e.g., switching from a cloud model to a local model, changing the RAG chunking strategy). For major changes, run a full manual evaluation of 50-100 outputs in addition to the automated regression suite.

**Make regression testing part of your CI/CD pipeline:** Every PR that changes prompt templates, model configurations, or system instructions should trigger the regression test suite. A PR that causes a regression should be blocked from merging until the issue is resolved.

## Mental Model Check

**Q1:** What is the difference between a statistically significant regression and a practically significant one?

**A1:** Statistical significance means the drop is unlikely to be due to chance (p < 0.05). Practical significance means the drop is large enough to matter in production (e.g., Cohen's d > 0.3). A change can be statistically significant but practically meaningless with a large enough sample.

**Q2:** Why should you track the 5th percentile (P5) of scores in addition to the mean?

**A2:** The P5 tells you about the tail of the distribution — the worst-case outputs. The mean can stay the same while the tail gets worse. If your system's worst outputs are getting worse (less safe, more hallucinated), that is a serious regression even if the average looks fine.

**Q3:** What is the Bonferroni correction and when would you use it?

**A3:** The Bonferroni correction divides the alpha threshold by the number of statistical tests being run simultaneously. Use it when testing multiple dimensions to avoid false positives. If testing 5 dimensions with alpha=0.05, use alpha=0.01 per test.

**Q4:** How does Levene's test complement a t-test in regression detection?

**A4:** A t-test checks for mean shift. Levene's test checks for variance change. A system can have the same mean but become more unpredictable (higher variance) — this is a regression that the t-test would miss but Levene's test would catch.

**Q5:** If you run a regression test and find a drop in faithfulness but no change in relevance, what does that tell you?

**A5:** It tells you the new version is still on-topic (relevance is fine) but is making more factual errors (faithfulness dropped). This narrows your investigation: look at how the change affected the model's tendency to hallucinate or misstate facts, rather than its ability to stay on topic.
