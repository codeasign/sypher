Imagine you built a customer support bot using a script. You mapped every known question to a hard-coded answer. "Where is my order?" triggers a lookup. "What are your hours?" returns a schedule. "How do I reset my password?" sends a link. It works perfectly for the first week. Then a user types "I ordered something three weeks ago and it still hasn't arrived, and also I want to change the delivery address, and can you tell me if the blue one is in stock instead?" Your script sees no exact match. None of your keyword triggers fire. It crashes, returns a generic error, or — worst of all — silently does nothing. The user gets frustrated and escalates to a human. Your "automation" just created more work.

Now consider a different scenario. You built a data processing pipeline that downloads a CSV, cleans the data, computes some statistics, and writes a report. It works on the test file. Then a real-world API returns a CSV with a slightly different column name. The pipeline breaks. You add a fix. Next week, the date format changes. Another fix. Over six months, the pipeline becomes a tangle of special cases, each one addressing a specific variation that someone encountered once. The maintenance burden outweighs the automation benefit.

This is the fundamental limit of hard-coded automation. Every decision path, every branch, every edge case must be anticipated in advance. The real world does not cooperate. APIs return fields in unexpected formats. Users phrase questions in ways you never imagined. Business rules change. A system that cannot adapt to novelty is brittle by design. You have seen this pattern in every scripted workflow you have ever maintained: the initial version works, then each new requirement adds conditional branches, and over time the code becomes an unmaintainable tangle of special cases. There is a better way.

Consider a third scenario: you work at a company where the team lead asks you to "find all the Jira tickets tagged 'urgent' that were updated in the last 24 hours, cross-reference them with the latest deployment status from Jenkins, and post a summary in Slack." With a script, you would need to write code to authenticate with three different APIs, handle pagination on the Jira results, parse Jenkins JSON output, format the Slack message, handle every possible error from each service, and add retry logic for network failures. The script would take days to write and would break the first time any API changed its response format. With an agent, you define three tools (search_jira, check_jenkins, post_slack), give the agent the goal in natural language, and let it figure out the rest.

What if instead of anticipating every possible input and mapping it to a fixed action, you built a system that could understand the intent, reason about what to do, and adapt when things go wrong? That is the promise of an AI agent.

## The Concept

An AI agent is a system that uses a large language model (LLM) to perceive its environment, decide what to do, take action via tools, and learn from the result — iterating until the task is done. Instead of following a predetermined checklist, the agent reasons about its goal at each step and chooses the next action based on the current situation.

Think of a human assistant. If you tell them "book a conference room for Tuesday at 2 PM for eight people," they do not follow a rigid script. They check room availability, consider which rooms fit eight people, check for conflicts, send the booking, and confirm back to you. If the room they wanted is taken, they try the next one without asking you to re-explain the goal. If the booking system is down, they wait and retry, or notify you of the delay. If you said "eight people" but the room only fits six, they flag the conflict. The assistant adapts.

An AI agent works the same way: you give it a goal in natural language, and it figures out the sequence of steps, adapts when things go wrong, and reports back when done. The difference is that the agent's "thinking" happens inside an LLM, and its "actions" are tool calls that your code executes. The agent does not have general intelligence or common sense — it has a powerful pattern-matching engine, a set of tools you define, and a loop that lets it iterate toward a goal.

The three core components that make this possible are:

- **LLM**: The reasoning engine that understands the goal, plans steps, and decides when the task is complete. The LLM does not execute actions — it reasons about what to do and requests actions via structured tool calls.
- **Tools**: Functions the LLM can invoke — calling APIs, reading files, sending emails, querying databases. The LLM does not execute these itself; it requests them, and your application code runs them. This separation is critical for safety and control.
- **Memory**: The conversation history and any state the agent accumulates across turns, allowing it to maintain context and build on previous reasoning. Without memory, the agent would treat each step as a brand new problem with no context.

These three elements work together in a loop: the LLM perceives (reads the latest input and history), thinks (reasons about what to do next), and acts (either calls a tool or returns a final answer). The loop is the core pattern — it is what distinguishes an agent from a one-shot LLM completion.

To make this concrete, imagine the user says "What is the weather in Tokyo and should I bring an umbrella?" The agent does not have a pre-written branch for this. Instead, the LLM perceives the question, thinks "I need current weather data for Tokyo, which I can get via the get_weather tool," acts by requesting a tool call, perceives the tool result ("temperature: 22°C, conditions: rainy"), thinks "22°C and rainy means a light jacket and umbrella are recommended," and acts by returning a final answer. The entire interaction is dynamic — the agent constructed its own decision tree at runtime based on the specific request.

## Key Concepts Reference

The table below maps the terminology you will encounter when building agents with the Anthropic API. Every term here maps directly to a parameter or field in the API request and response.

| Concept | What It Means | How You Use It |
|---|---|---|
| System Prompt | The core instructions that define the agent's persona, constraints, and behavior | Sent as the `system` parameter in the API request; sets the rules the agent follows for the entire conversation |
| Tool Definition | A JSON schema describing a function the agent can call, including its name, description, and parameters | Passed as the `tools` array in the API request; the LLM uses the description to decide when to call it |
| Message | A single turn in the conversation with a `role` (user or assistant) and `content` | Sent as the `messages` array; each exchange adds one or two messages |
| Content Block | A piece of content within a message — text, tool_use, or tool_result | The `content` field of a message is a list of blocks; the LLM can mix text and tool_use blocks in one response |
| tool_use | The LLM's request to invoke a tool, identified by an `id`, `name`, and `input` | Your application reads the `stop_reason` and, if it is `tool_use`, executes the named tool with the provided input |
| tool_result | The output returned after executing a tool, linked to the original tool_use by its `id` | You construct a message with role `user` containing a `tool_result` content block and append it to the conversation |
| stop_reason | The LLM's signal for why it stopped generating — `end_turn`, `tool_use`, `max_tokens`, or `stop_sequence` | `end_turn` means the agent considers the task complete; `tool_use` means it is requesting a tool call |
| Max Tokens | The maximum number of tokens the LLM can generate in a single response | Set via the `max_tokens` parameter; also acts as a safety limit to prevent unbounded generation |
| Temperature | Controls randomness in the LLM's output (0.0 to 1.0) | Lower values (0.0–0.3) produce more deterministic, focused behavior for agent tasks |

## How It Actually Works — Step by Step

The agent loop is not magic. It is a predictable cycle of eight steps that repeats until the task is done. Understanding each step in detail is essential before you write your first agent. Each step corresponds to a specific action in your code, a specific field in the API request or response, or both.

### Step 1: User Provides a Goal

The user sends a message in natural language. This could be "Find the latest sales report and email it to the team" or "Check the weather in Tokyo and suggest activities based on the forecast." The message becomes the first entry in the conversation history. At this point, the agent has no context beyond the system prompt and the user's message. It does not know what tools exist or what the user expects — it only has the text. The quality of the user's message matters: a vague goal like "help me" is much harder for the agent to act on than a specific goal like "find all Jira tickets assigned to me that are past their due date."

The user message is added to the messages array with role `"user"`. This is the starting state for every agent session.

```json
{
  "role": "user",
  "content": "What is the weather in Tokyo and should I bring an umbrella?"
}
```

### Step 2: System Prompt Sets the Rules

Before the first user message, you define a system prompt. This is a set of instructions that governs the agent's behavior: who it is (persona), what it can do (available tools), how it should handle ambiguity, and any constraints (budget limits, safety rules, output format preferences). The system prompt is sent once and applies for the entire conversation.

A well-crafted system prompt is the single most important factor in agent reliability. It tells the LLM things like "always ask for confirmation before sending an email" or "if you are not sure about a parameter, ask the user instead of guessing." Without a good system prompt, the agent may behave unpredictably — guessing parameters, calling tools unnecessarily, or failing to ask for clarification when needed.

Example system prompt for a weather agent:

```text
You are a helpful weather assistant. You have access to a weather lookup tool.
Always ask the user which city they want weather for if they do not specify one.
When providing the answer, include temperature, conditions, and a practical
recommendation (e.g., do you need an umbrella, a jacket, sunscreen).
Never make up weather data. Only report what the tool returns.
```

### Step 3: LLM Receives the Prompt and Tools

The API request bundles the system prompt, the conversation history (messages), and the tool definitions (tools). The LLM processes all three together. The tool definitions are especially important — they are JSON schemas that tell the LLM what functions exist, what each one does, and what parameters it expects.

The LLM does not "see" the tool as code. It sees a JSON description: a name, a text description, and a schema of inputs. The quality of the description directly affects how well the LLM chooses which tool to call and when. A vague description like "gets weather data" is far less useful than "gets the current weather for a specific city and returns temperature, conditions, and humidity as a JSON object." The LLM uses the description to decide relevance.

Here is what a tool definition looks like as JSON:

```json
{
  "name": "get_weather",
  "description": "Get the current weather for a city. Returns temperature, conditions, and humidity.",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "The city name, e.g. Tokyo, London, New York"
      }
    },
    "required": ["city"]
  }
}
```

### Step 4: LLM Decides What to Do

The LLM reasons about the current state and produces a response. This response contains one or more content blocks. If the LLM decides it can answer directly from its existing knowledge, it generates a text block with `stop_reason: "end_turn"`. If it needs more information or needs to take an action, it generates one or more `tool_use` content blocks with `stop_reason: "tool_use"`.

The LLM can also generate a mix: a text block explaining what it is about to do, followed by a tool_use block. For example, the response might include "I will check the weather in Tokyo for you now" as a text block and then a tool_use block with `name: "get_weather"` and `input: {"city": "Tokyo"}`. This is normal — the LLM reveals its reasoning before taking action.

The LLM can also request multiple tools in a single response. For example, if the user asks "What is the weather in Tokyo and what time is it in London?" the LLM might generate two tool_use blocks in one response: one for `get_weather` and one for `get_current_time`. Your code must handle both.

### Step 5: Application Executes the Tool

When `stop_reason` is `tool_use`, your application reads the tool name and input from the content block. You then execute the corresponding function — this is your code, running locally. The function could be anything: calling a REST API, reading a file, running a database query, or computing a value.

This is the step where you validate the input before executing. Never trust the LLM's input blindly — always validate parameters against your own rules before running a tool. The LLM might guess a parameter value if the user did not provide one, or it might construct a file path that you do not want it to access. Your code is the gatekeeper. Validate the tool name, validate the parameters, and only execute if everything checks out.

Code sketch for tool execution:

```python
def execute_tool(name, input_data):
    if name == "get_weather":
        city = input_data.get("city", "")
        if not city or not isinstance(city, str):
            return "Error: city must be a non-empty string"
        # ... call the weather API ...
    elif name == "read_file":
        filepath = input_data.get("filepath", "")
        if ".." in filepath or not filepath.startswith("/allowed/path"):
            return "Error: invalid file path"
        # ... read the file ...
    else:
        return f"Error: unknown tool '{name}'"
```

### Step 6: Tool Result Goes Back to the LLM

You create a new message with role `"user"` containing a `tool_result` content block. This block includes the `tool_use_id` (to link it to the original request) and the output of the tool execution. You append this message to the conversation history and send the updated messages array back to the LLM.

The `tool_use_id` is critical — it links the result to the original request. Without the correct ID, the LLM cannot associate the result with the tool call it made. The ID comes from the `tool_use` content block in the LLM's response and must be copied exactly into the `tool_result` block.

Code sketch for returning a tool result:

```python
# After executing the tool, append the result to the conversation
messages.append({
    "role": "assistant",
    "content": response.content  # The LLM's response with the tool_use blocks
})
messages.append({
    "role": "user",
    "content": [
        {
            "type": "tool_result",
            "tool_use_id": tool_use_block.id,
            "content": tool_result_string
        }
    ]
})
```

Note that the assistant response is appended first, then the tool_result. This preserves the conversation structure: the LLM said something (assistant), then the tool returned a result (user). The role is `"user"` for tool results because the result is new information entering the conversation from outside the LLM.

### Step 7: Loop Continues

The LLM now has the tool result in its context. It incorporates this new information into its reasoning. It may call another tool (if it needs more data) or produce a final answer. This loop repeats until the LLM signals `stop_reason: "end_turn"` or a safety limit is reached. Each iteration adds to the conversation history, which means the context window usage grows with each cycle.

### Step 8: Final Answer

When the LLM decides the task is complete, it produces a text block with `stop_reason: "end_turn"`. Your application reads this text and returns it to the user as the final answer. The agent session is complete. The conversation history now contains the full trace of every decision, every tool call, and every intermediate result.

## Cost, Limits, and Safety

Building an agent is not just about getting the loop right. You must also account for the operational realities of running an agent in production. The same flexibility that makes agents powerful also makes them expensive and potentially dangerous if not properly constrained.

### Token Costs Per Iteration

Every agent loop iteration consumes tokens in both directions:

- **Input tokens**: The system prompt, conversation history, tool definitions, and the latest tool_result are all sent to the LLM on each turn. The system prompt and tool definitions are constant across iterations, but the conversation history grows with every exchange. After 10 iterations, the conversation history alone could be thousands of tokens.
- **Output tokens**: The LLM's response — whether a tool_use request or a final answer — consumes output tokens. Output tokens are typically priced higher than input tokens on most API pricing plans.

A single agent run with 3 tool calls might consume 4x–10x the tokens of a simple completion because the full context (system prompt + tool definitions + growing conversation) is sent on every iteration. This is not a bug — it is the cost of maintaining context across multiple reasoning steps.

Example budget for a typical agent session:

| Component | Approximate Tokens |
|---|---|
| System prompt | 300–800 |
| Tool definitions (2–3 tools) | 400–1,000 |
| Each user message | 50–300 |
| Each LLM response (tool_use) | 100–400 |
| Each tool_result message | 50–500 |
| Final response | 200–800 |

Running a 5-iteration agent session with these numbers would consume roughly 3,000–8,000 input tokens and 1,000–3,000 output tokens. Multiply by the per-token pricing to get the cost per session.

### Rate Limits

API providers enforce rate limits (requests per minute, tokens per minute). A looping agent can hit these limits quickly if each iteration makes a separate API call without pacing. If you exceed the rate limit, the API returns a 429 HTTP status code with a `rate_limit_error` error type.

Common rate limit patterns:

- **Requests per minute (RPM)**: Maximum number of API calls per minute. For the Anthropic API, this varies by tier — typically 50–1,000 RPM.
- **Tokens per minute (TPM)**: Maximum number of tokens processed per minute. This is usually the tighter constraint for agents with large system prompts and tool definitions.
- **Concurrent requests**: Maximum number of simultaneous API calls. Your agent loop is sequential (one call at a time), so this is less likely to be an issue unless you run multiple agent sessions in parallel.

If your agent runs in a loop that makes 10 API calls within a few seconds, you may exceed the RPM limit. Add a small delay between iterations or implement a token bucket rate limiter. The safest approach is to implement exponential backoff: wait 1 second after the first 429, then 2 seconds, then 4, then 8, up to a maximum of 30–60 seconds.

### Runaway Loop Safety

The most common failure mode of an early agent is an infinite loop: the LLM keeps calling tools, getting results, and calling more tools without ever deciding the task is done. This can happen for several reasons:

- The tool results are ambiguous or surprising, prompting the LLM to investigate further.
- The task is genuinely open-ended (e.g., "research everything about this topic").
- The LLM gets stuck in a cycle where tool A depends on tool B, and tool B depends on tool A.
- A bug in the tool implementation returns unexpected results that confuse the LLM.

You must implement at least these safeguards:

- **Max iterations**: Hard limit on the number of tool call cycles (start with 10–25).
- **Max tokens**: Set `max_tokens` to a reasonable ceiling (2,048–4,096).
- **Timeout**: Abort the agent if total wall-clock time exceeds a threshold (30–120 seconds).
- **Human interrupt**: Provide a mechanism to terminate the agent mid-execution.
- **Duplicate detection**: Detect if the LLM is making the same tool call with the same parameters repeatedly.

### API Key Security

Never hardcode API keys in source code. Always load them from environment variables:

```python
import os
from anthropic import Anthropic

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
```

The Anthropic SDK reads `ANTHROPIC_API_KEY` from the environment automatically if no key is passed, so `Anthropic()` with no arguments is the safest pattern. This means you never need to write the key in your code at all.

Common mistakes to avoid:

- Committing `.env` files to version control (add `.env` to `.gitignore`).
- Hardcoding keys in test files or example scripts.
- Logging API keys in debug output.
- Sharing API keys across team members (each person should have their own key).

### Context Window Limits

Every LLM has a maximum context window (the total input tokens it can process per request). As the conversation grows across iterations, you approach this limit. The Claude Sonnet 4 model, for example, has a 200K token context window, but you will typically want to stay well below the maximum to leave room for the LLM's response.

Strategies to handle context growth:

- **Truncation**: Drop the oldest messages when the token count approaches the limit. This is the simplest approach but loses early context.
- **Summarization**: Condense earlier parts of the conversation into a summary. Replace the oldest messages with a single summary message. This preserves context in compressed form.
- **Sliding window**: Keep only the most recent N turns plus the system prompt. The agent loses awareness of very early steps but maintains focus on the current state.
- **Token counting**: Use the API's response usage information to track how many tokens you are consuming and proactively manage the conversation length.

## When to Use This / When Not To

The agent pattern is powerful but not universal. Understanding when it adds value and when it adds unnecessary complexity is essential for making good architectural decisions.

### Use an agent when:

- The task requires multiple steps that cannot be predetermined. The agent decides the sequence at runtime based on the results of each step.
- The task involves conditional logic: "if this API returns X, do Y; if it returns Z, do W." The agent can evaluate the condition and choose the next action.
- The user's goal is expressed in natural language and may vary in phrasing or scope. The agent's LLM understands the intent even when the wording changes.
- You need the system to adapt to partial failures (a tool returns an error, so try a fallback). The agent can interpret the error and choose an alternative approach.
- The task benefits from reasoning: it is not a simple lookup or transformation. The agent can reason about the best approach, combine information from multiple sources, and handle ambiguity.
- The requirements change frequently. An agent with a well-written system prompt can adapt to new requirements without code changes — you can often change behavior by editing the system prompt rather than rewriting code.
- You need to combine data from multiple sources in a flexible way. The agent can decide which tools to call and in what order based on the user's request, without you hard-coding the orchestration logic.
- The task is exploratory. The user does not know exactly what they need until they see intermediate results. The agent can iterate, refine, and adapt based on partial findings.

### Do not use an agent when:

- A simple completion suffices. If the task is "summarize this text" or "translate this sentence," a single LLM call with no tools is cheaper, faster, and less error-prone. Adding an agent loop to a task that does not need tools adds latency and cost for no benefit.
- Human-in-the-loop is required for every decision. If the task needs human approval at each step, the autonomy of an agent adds complexity without benefit. A guided workflow with explicit human approval steps is a better fit.
- A RAG pipeline is a better fit. If the task is "answer questions about a specific document set," a retrieval-augmented generation system with a fixed retrieval step is simpler and more predictable than an agent deciding when to search. The agent pattern adds unnecessary complexity when the retrieval strategy is fixed.
- The cost and latency of multiple LLM calls are unacceptable. Each iteration adds 1–5 seconds and consumes tokens. If the use case demands sub-second responses, an agent is the wrong architecture. Look at caching strategies or simpler completions instead.
- The safety risk is too high. If a wrong tool call could delete data, send an unauthorized email, or trigger a financial transaction, you need strong guardrails or a human approval step before execution. An agent with autonomous tool calling is not appropriate for high-risk operations without a human-in-the-loop layer.
- The task is deterministic. If the inputs and outputs are well-defined and the logic is straightforward, a regular function or script is faster, cheaper, and more predictable than an agent. A function that takes a number and returns its square root does not need an LLM.
- You need guaranteed reproducibility. LLM outputs are non-deterministic by nature. The same input can produce different tool call decisions across runs. If you need exactly the same result every time, a deterministic script is the right choice.
- The environment is fully controlled. If you know every possible input and every expected output, there is no need for the flexibility an agent provides. The agent's adaptability becomes dead weight.

Your code should read the `tool_use` content block(s) from the response, extract the `name` and `input`, execute the corresponding function, construct a `tool_result` message, append it to the conversation history, and send the updated messages back to the LLM for the next iteration.

### Question 3

Why can a single agent session consume significantly more tokens than a simple LLM completion?

<details>
<summary>Show answer</summary>

Each agent iteration sends the full system prompt, tool definitions, and the entire conversation history (which grows with every turn) as input tokens. In addition, each tool call produces output tokens (the LLM's tool_use response) plus the tool_result tokens you send back. A session with multiple tool call cycles can easily consume 4x–10x the tokens of a single completion call.
</details>

### Question 4

What is the most common failure mode of a newly built agent, and what are two safeguards to prevent it?

<details>
<summary>Show answer</summary>

The most common failure mode is a **runaway loop** where the LLM keeps calling tools without ever deciding the task is done. Safeguards include: a **max iterations** limit (e.g., stop after 10–25 tool calls), a **max tokens** ceiling per response, and a **wall-clock timeout** that aborts the agent if it runs too long.
</details>
