You have an idea for an app that can summarize emails, answer questions about your documentation, or generate code from natural language. You know an LLM can do it. But every guide you find assumes you already know how to make an HTTP request to an API, parse JSON, and handle authentication. So you copy-paste a curl command from a blog post, hardcode your API key, and run it. It works — once. Then you push the code to GitHub, and a bot scans your repo, finds the key, and you get an email from OpenAI saying your account has been compromised. Your bill spikes to four hundred dollars from a cryptominer running on your leaked key.

That scenario plays out hundreds of times a month. The OpenAI API is just a web request. But getting it right — securely, reliably, cost-effectively — means understanding what happens between your Python process and OpenAI's servers. This topic gives you that understanding.

## The Concept

Calling the OpenAI API means sending a structured HTTP request to a remote server that runs a large language model, waiting for it to generate text, and parsing the response. Think of it like ordering from a restaurant kitchen: you write down your order on a specific form (the request), hand it to the waiter (the API client), the kitchen (the model) prepares the meal (generates tokens), and the waiter brings it back (the response). The menu (available models) and prices are posted upfront.

## API Reference

| Method / Parameter | What It Does | Example |
|---|---|---|
| `pip install openai` | Installs the official OpenAI Python SDK (v2.44.0) | `pip install openai` |
| `from openai import OpenAI` | Imports the client class | `from openai import OpenAI` |
| `client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])` | Creates a client that reads your API key from an environment variable | `client = OpenAI()` — reads from `OPENAI_API_KEY` by default |
| `client.chat.completions.create(model="gpt-4o", messages=[...])` | Sends a chat completion request to the specified model | `client.chat.completions.create(model="gpt-4o", messages=[{"role":"user","content":"Hello"}])` |
| `messages` parameter | Array of message objects with `role` and `content`. Roles: `developer` (system prompt), `user`, `assistant` | `messages=[{"role":"developer","content":"You are a helpful assistant"}, {"role":"user","content":"What is Python?"}]` |
| `temperature` parameter (0.0–2.0) | Controls randomness. Lower values produce focused, deterministic outputs | `temperature=0.0` |
| `max_tokens` parameter | Maximum tokens the model can generate in the response | `max_tokens=500` |
| `stream` parameter (bool) | If true, receive tokens incrementally as they are generated | `stream=True` |
| `response_format` parameter | Constrain the output to a JSON schema (structured output) | `response_format={"type":"json_schema","json_schema":{...}}` |
| `completion.choices[0].message.content` | Accesses the response text from a completion | `print(completion.choices[0].message.content)` |
| `completion.usage` | Contains token usage stats (prompt_tokens, completion_tokens, total_tokens) | `print(completion.usage.total_tokens)` |

## How It Actually Works — Step by Step

### Step 1: Your Code Constructs a Request

When you call `client.chat.completions.create()`, the SDK builds a Python dictionary containing your model choice, messages, and parameters. It serializes this into a JSON payload.

### Step 2: The SDK Sends an HTTPS Request

The OpenAI SDK uses the `httpx` library under the hood. It sends a POST request to `https://api.openai.com/v1/chat/completions` with:

- Your API key in the `Authorization: Bearer <key>` header
- The `Content-Type: application/json` header
- The JSON payload from Step 1 as the request body

The SDK never exposes your key in the URL or in the request body — it only goes in the header, which is encrypted by HTTPS.

### Step 3: OpenAI Authenticates and Routes

OpenAI's servers receive your request and:

1. Validate your API key against their database
2. Check your account has available credits or is on a paid plan
3. Check rate limits (how many requests you've made in the last minute)
4. Route the request to the appropriate model server

If any check fails, you get an error response — no tokens are consumed and you are not charged.

### Step 4: The Model Generates a Response

The model server loads the specified model (e.g., GPT-4o) and runs inference:

1. Your messages are tokenized — split into tokens (roughly 0.75 words per token for English)
2. The tokens are passed through the model's neural network layers
3. The model generates one token at a time, each step predicting the most likely next token
4. Generation stops when the model produces a stop token, reaches `max_tokens`, or hits a content filter

### Step 5: The Response is Serialized and Returned

OpenAI's server packages the generated tokens into a structured JSON response:

```json
{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1723456789,
  "model": "gpt-4o",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Python is a high-level programming language..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 10,
    "completion_tokens": 45,
    "total_tokens": 55
  }
}
```

This JSON is sent back over HTTPS to the SDK.

### Step 6: The SDK Parses the Response

The OpenAI SDK deserializes the JSON into a `ChatCompletion` Python object. You access the generated text through:

```python
response_text = completion.choices[0].message.content
```

Each field has a specific meaning:

| Field | What It Tells You |
|---|---|
| `choices[0].message.content` | The generated response text |
| `choices[0].finish_reason` | Why generation stopped: `"stop"`, `"length"`, `"content_filter"`, or `"tool_calls"` |
| `usage.prompt_tokens` | Number of tokens in your input (what you are billed for) |
| `usage.completion_tokens` | Number of tokens in the response (what you are billed for) |
| `usage.total_tokens` | Sum of prompt and completion tokens |

## Cost, Limits, and Safety

### Token Costs (approximate, as of 2026)

| Model | Input (per 1M tokens) | Output (per 1M tokens) |
|---|---|---|
| `gpt-5.5` | $15.00 | $60.00 |
| `gpt-4o` | $2.50 | $10.00 |
| `gpt-4o-mini` | $0.15 | $0.60 |
| `o3` | $10.00 | $40.00 |
| `o4-mini` | $1.10 | $4.40 |

A single typical call to `gpt-4o` with 500 input tokens and 200 output tokens costs roughly `(500/1M * $2.50) + (200/1M * $10.00)` = $0.00325 — less than one cent. At 1,000 calls per day, that becomes about $3.25 per day.

### Rate Limits

Free-tier accounts: 3 requests per minute, 200 per day. Tier 1 paid: 500 RPM, 10,000 per day. Higher tiers available based on usage history.

### Safety Rules

**Never hardcode an API key in your code.** Always read it from an environment variable:

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
```

Set the variable in your terminal:

```bash
export OPENAI_API_KEY="sk-proj-your-key-here"
```

Or create a `.env` file (add `.env` to `.gitignore` first):

```
OPENAI_API_KEY=sk-proj-your-key-here
```

**Set usage limits** in the OpenAI dashboard under Billing → Usage limits. Enable a hard cap (e.g., $10/month) so a runaway loop cannot blow your budget.

**Watch for unbounded loops.** Every call to `client.chat.completions.create()` costs money. If you call it inside a `while True` loop without a break condition, your bill increases with every iteration.

## When to Use This / When Not To

**Use the OpenAI API when:**
- You need a powerful, general-purpose model with strong instruction following
- You want the latest frontier model (GPT-5.5, GPT-4o)
- You need multimodal capabilities (image understanding with GPT-4o)
- You do not need to run the model locally

**Do not use the OpenAI API when:**
- You need to process sensitive data that cannot leave your network
- You need offline operation (no internet connection)
- Your application makes millions of low-complexity calls — consider a smaller model like `gpt-4o-mini` or a local alternative
- You are building a high-frequency feature — each call has network latency (typically 500ms–5s)

## Mental Model Check

**Question 1:** What is the minimum information you must send to the OpenAI API to get a response?

  The model name and at least one message with the user role. For example:
  ```python
  client.chat.completions.create(
      model="gpt-4o",
      messages=[{"role": "user", "content": "Hello"}]
  )
  ```

---

**Question 2:** Where does your API key go in the request, and why does it matter?

  It goes in the `Authorization: Bearer` HTTP header, not in the URL or body. This matters because HTTPS encrypts the headers, so your key is protected in transit. Never put it in a URL parameter or JSON body.

---

**Question 3:** How is the response text accessed after a successful API call?

  Through `completion.choices[0].message.content`. The `choices` array supports multiple completions (n parameter), and index 0 is the first (often only) one.

---

**Question 4:** If the model stops generating before finishing its answer, what field tells you why?

  The `finish_reason` field on the choice object. `"stop"` means the model finished naturally. `"length"` means it hit `max_tokens`. `"content_filter"` means the response was flagged. `"tool_calls"` means the model wants to use a tool.

---

**Question 5:** What three pieces of information does the `usage` object provide, and why should you log them?

  `prompt_tokens`, `completion_tokens`, and `total_tokens`. You should log them to track and predict costs, detect unexpectedly long responses, and monitor usage patterns for billing alerts.
