You are building an AI-powered code assistant. It needs to read files from the project directory, query a local SQLite database for user records, search the web for documentation, post messages to a Slack channel when tasks complete, and take screenshots of the browser during debugging. Without MCP, you would write five separate integrations: a file reader, a database connector, a web scraping pipeline, a Slack API client, and a Puppeteer script. Each would have its own authentication, its own error handling, its own configuration format, and its own way of exposing results to the LLM. You would maintain them all yourself, and every new tool would mean another custom adapter.

Existing MCP servers eliminate that work. They are pre-built adapters that give LLM applications access to real-world tools and data sources through a single, standard protocol. Instead of writing five bespoke integrations, you install five MCP servers and configure them once. The MCP client in your AI application handles discovery, capability negotiation, and communication with every server through the same interface.

## The Concept

**Existing MCP servers are pre-built adapters that give LLM applications access to real-world tools and data sources through a standard protocol.**

Think of an MCP server the way you think of a USB peripheral. You do not write a new driver every time you plug in a keyboard, a webcam, or an external drive. The USB standard defines how the device announces itself and how the operating system talks to it, and the manufacturer ships a driver that speaks that standard. An MCP server is the same idea for AI applications: the server announces the tools it offers and the shape of their inputs, and any MCP-compatible client (Claude Code, Claude Desktop, or a custom application built on the MCP SDK) can plug in and start calling those tools immediately, with zero custom integration code.

The servers already exist because the problems they solve are common ones. Almost every AI application eventually needs to read a file, query a database, search the web, or interact with a developer platform like GitHub. Rather than every team solving these problems in isolation, Anthropic, other vendors, and the open source community publish and maintain servers for exactly these needs. Using one means you inherit someone else's testing, edge-case handling, and ongoing maintenance as the underlying API changes. You just point your client at it and go.

## API Reference Table

The following table covers the most widely used MCP servers available today. Each entry shows the package name, what it does, and how to install or run it. Note that this ecosystem moves quickly: some early reference servers have been archived and replaced by better-maintained alternatives.

| Server | Package / Project | What It Does | Install / Setup |
|---|---|---|---|
| Filesystem | `@modelcontextprotocol/server-filesystem` (maintained by Anthropic) | Read, write, list, move, and search files within an explicitly allowed directory | `npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/dir` |
| GitHub | `github/github-mcp-server` (official GitHub project; replaces the archived `@modelcontextprotocol/server-github`) | Read and create issues and pull requests, search code, inspect repos, manage branches | Remote: point client at `https://api.githubcopilot.com/mcp` with bearer token. Local: `docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN ghcr.io/github/github-mcp-server` |
| SQLite | `mcp-server-sqlite` (Anthropic original, now archived) | Run queries against a local SQLite file and inspect its schema | `uvx mcp-server-sqlite --db-path /path/to/db.sqlite` — for production use a community fork instead |
| Postgres | `postgres-mcp` (Postgres MCP Pro, Crystal DBA) | Query execution, EXPLAIN plan analysis, index tuning suggestions, and database health checks | `pipx install postgres-mcp` then run with `--access-mode=restricted` for read-only |
| Puppeteer / Playwright | `@playwright/mcp` (Microsoft's official browser automation server; replaces the archived Puppeteer server) | Drive a real browser: navigate, click, type, take screenshots, read the accessibility tree | `npx @playwright/mcp@latest` (add `--headless` for CI-style use) |
| Slack | `slack-mcp-server` (community server by korotovsky, the most widely adopted) | Read channel history, list users, post messages and thread replies | `npx -y slack-mcp-server` with a Slack token supplied via environment variable |
| Memory | `@modelcontextprotocol/server-memory` (maintained by Anthropic) | Store facts as a persistent knowledge graph the model can write to and query across sessions | `npx -y @modelcontextprotocol/server-memory` |
| Brave Search | `@brave/brave-search-mcp-server` (maintained by Brave Software) | Web search and local business search results backed by the Brave Search API | `npx -y @brave/brave-search-mcp-server` with `BRAVE_API_KEY` set in the environment |
| Fetch / Web | `@modelcontextprotocol/server-fetch` (maintained by Anthropic) | Fetch and extract content from web pages, returned as markdown text | `npx -y @modelcontextprotocol/server-fetch` |
| Sequential Thinking | `@modelcontextprotocol/server-sequential-thinking` (maintained by Anthropic) | Enable structured, step-by-step reasoning with a visible thought process that the model can revisit and revise | `npx -y @modelcontextprotocol/server-sequential-thinking` |

A few things worth internalizing from this table before you install anything. First, "official reference server" does not always mean "current best option" — the MCP ecosystem moves fast enough that some of the earliest servers (SQLite, Puppeteer, the original GitHub server) have been archived in favor of better-maintained replacements. Second, every server that touches a paid API (Brave Search, GitHub, most Slack setups) needs a credential of its own, supplied through an environment variable, never hardcoded into a config file you might commit to source control. Third, the install pattern is almost always one of two shapes: `npx -y <package>` for Node-based servers, or `uvx <package>` / `pipx install <package>` for Python-based ones. Once you recognize that pattern, evaluating a new MCP server you have never used before takes seconds.

### Step 1: Discovering What Servers Exist

Before you install anything, you need to know what is available. The main catalog is the `modelcontextprotocol/servers` repository on GitHub, which lists the reference servers still under active maintenance (filesystem, memory, fetch, git, sequential-thinking, time) plus links to community and vendor-maintained servers for everything else (databases, GitHub, Slack, search providers, and dozens more). Vendors like Brave, GitHub, and Microsoft also publish and maintain their own servers directly, independent of that central list.

You can also find MCP servers through:

- **npm registry** — Search for packages with the `@modelcontextprotocol` or `@anthropic` scope, or use the `mcp-server` keyword. This is the largest collection of MCP servers.
- **PyPI** — Python-based MCP servers distributed as pip packages, useful when you need Python library access.
- **GitHub** — Many servers are developed in the open. The official repositories live under `github.com/modelcontextprotocol/servers`. Community servers are tagged with `mcp-server`.
- **MCP directories** — Community-maintained lists of MCP servers organized by category (filesystem, database, communication, search, developer tools).

When evaluating a server, check the last commit date, the number of downloads or stars, and whether it documents its tools and resources clearly. A server with recent commits, real issue-tracker activity, and meaningful adoption is one whose bugs other people have already found and whose security holes get patched. A server last updated over a year ago, with no responses to open issues, is a server you would be debugging alone.

### Step 2: Installing the Server

Most MCP servers do not need a separate install step at all. You run them on demand with `npx` (for Node-based servers) or `uvx` (for Python-based servers), and the tool cache downloads them the first time and reuses that cache afterward. This is the same pattern you would use to run a one-off CLI tool without permanently installing it globally.

```bash
# Node-based server, run on demand (no explicit install needed)
npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/dir

# Python-based server, run on demand
uvx mcp-server-sqlite --db-path ./sample.db

# Permanent installation for pinning a specific version
npm install -g @modelcontextprotocol/server-filesystem@1.2.3

# Python permanent installation
pip install postgres-mcp

# Homebrew installation
brew install mcp-server-sequential-thinking

# Git clone for servers under active development
git clone https://github.com/modelcontextprotocol/servers.git
cd servers/src/brave-search
npm install
```

Using `npx` is convenient for exploration because the server is fetched and cached on first run. For production use, you may want to pin a specific version by installing the package explicitly with `npm install -g` or `pip install`. This ensures that updates to the package do not change behavior unexpectedly.

### Step 3: Registering the Server with Your Client

A server sitting on disk does nothing until a client knows to launch it. The configuration you provide tells the client what command to run, what arguments to pass, and what environment variables to set.

**Claude Code** — Register a server with the `claude mcp add` command:

```bash
claude mcp add --transport stdio filesystem -- npx -y @modelcontextprotocol/server-filesystem /path/to/allowed/dir
```

**Claude Desktop** — Register the server in `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@anthropic/mcp-server-filesystem",
        "/Users/you/projects"
      ]
    },
    "github": {
      "command": "npx",
      "args": [
        "-y",
        "github/github-mcp-server"
      ],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    },
    "brave-search": {
      "command": "npx",
      "args": [
        "-y",
        "@brave/brave-search-mcp-server"
      ],
      "env": {
        "BRAVE_API_KEY": "your_brave_api_key"
      }
    }
  }
}
```

**CLAUDE.md configuration** (for Claude Code projects that use project-level config):

```markdown
## MCP Servers

### filesystem
- command: npx
- args: -y @anthropic/mcp-server-filesystem /Users/you/projects

### github
- command: npx
- args: -y github/github-mcp-server
- env: GITHUB_PERSONAL_ACCESS_TOKEN=ghp_your_token_here
```

Any environment variables you specify in the server configuration (an API key, a database path, an allowed directory) are passed into that server's child process environment, not into the client's own environment. This keeps credentials scoped to exactly the one server that needs them.

### Step 4: Server Startup and Lifecycle

When your client starts (or when you first reference a newly registered server), it spawns the server as a child process using the command and arguments from the configuration. The server initializes its own state on startup:

- Opens a database connection and verifies it is alive
- Checks that the allowed directory exists and is readable
- Verifies that required API keys are present in the environment
- Loads any configuration files the server may need
- Sets up internal data structures for the knowledge graph or search index

The server process stays alive for the duration of the client session. When the client shuts down, it sends a shutdown notification and the server cleans up resources (closes file handles, disconnects from databases, releases API connections).

Each server runs as its own operating system process. This means a crash in one server does not affect the others — the client detects the disconnected server, marks it unavailable, and continues communicating with the remaining ones. Process isolation is one of the core architectural benefits of MCP.

### Step 5: Capability Negotiation

Once the server process is running, the client and server perform a handshake over the transport (standard input/output for local servers, SSE for remote ones). This is where the two sides agree on what they can do together.

The client sends an `initialize` request describing the protocol version it speaks and its own capabilities:

```json
{
  "protocolVersion": "2025-03-26",
  "capabilities": {
    "roots": {
      "listChanged": true
    },
    "sampling": {}
  },
  "clientInfo": {
    "name": "claude-desktop",
    "version": "1.0.0"
  }
}
```

The server responds with its protocol version and a declaration of what it offers:

```json
{
  "protocolVersion": "2025-03-26",
  "capabilities": {
    "tools": {},
    "resources": {
      "subscribe": true,
      "listChanged": true
    }
  },
  "serverInfo": {
    "name": "mcp-server-filesystem",
    "version": "0.1.0"
  }
}
```

If the versions are compatible, the client follows up with a `tools/list` request, and the server responds with every tool it offers — each tool's name, a description, and a JSON Schema describing its inputs:

```json
{
  "tools": [
    {
      "name": "read_file",
      "description": "Read the complete contents of a file from the allowed directory",
      "inputSchema": {
        "type": "object",
        "properties": {
          "path": {
            "type": "string",
            "description": "Path to the file, relative to the allowed directory or absolute within it"
          }
        },
        "required": ["path"]
      }
    },
    {
      "name": "write_file",
      "description": "Write content to a file within the allowed directory",
      "inputSchema": {
        "type": "object",
        "properties": {
          "path": { "type": "string" },
          "content": { "type": "string" }
        },
        "required": ["path", "content"]
      }
    },
    {
      "name": "list_directory",
      "description": "List all files and directories in a given path within the allowed directory",
      "inputSchema": {
        "type": "object",
        "properties": {
          "path": { "type": "string" }
        },
        "required": ["path"]
      }
    }
  ]
}
```

This is the moment your AI application actually learns what a given server can do. Nothing is hardcoded on the client side. The same client code that reads this tool list from a filesystem server also reads it from a database server, a search server, or a Slack server.

### Step 6: Tool Calls in Action

With the tool list available, your AI application can now decide, based on what you asked, which tool to call and with what arguments. This is ordinary tool-use behavior on the model's part, the same mechanism used for any function call. The sequence looks like this:

1. The LLM generates a response that includes a tool call request.
2. The client formats this into an MCP `tools/call` JSON-RPC request.
3. The client routes the request to the correct server process based on the tool name.
4. The server receives the request, executes the tool, and returns the result.
5. The client passes the result back to the LLM as the tool's output.
6. The LLM continues reasoning with the new information.

```bash
# What the client sends internally (conceptual JSON-RPC)
# Method: tools/call
# Params: { name: "read_file", arguments: { path: "config.json" } }
#
# What the server returns:
# { content: [{ type: "text", text: "...file contents..." }] }
```

### Step 7: Handling Results and Chaining Calls

The model sees the tool's result the same way it would see the result of any function call, and decides whether it has what it needs or whether another tool call is required. A single user request — "find the open pull requests that touch the billing module and summarize them" — might trigger a sequence of several calls to the GitHub server: one to search for relevant PRs, several more to fetch each PR's details, and possibly one to read a changed file. From your point of view as the developer, none of this sequencing was hand-coded. It falls out naturally from the model having tools available and reasoning about how to use them.

```python
# Conceptual example of what happens under the hood
# when an LLM calls a tool on an MCP server

import json

# The MCP client sends a tool call request
tool_call_request = {
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
        "name": "search",
        "arguments": {
            "query": "MCP protocol specification",
            "count": 5
        }
    }
}

# The server processes the request and returns results
# This happens over the transport layer (stdio or SSE)
server_response = {
    "jsonrpc": "2.0",
    "id": 1,
    "result": {
        "content": [
            {
                "type": "text",
                "text": "Search results for 'MCP protocol specification':\n\n1. Model Context Protocol Specification\n   https://spec.modelcontextprotocol.io\n\n2. MCP: The Missing Link for LLM Tool Access\n   https://example.com/mcp-guide\n\n3. Getting Started with MCP Servers\n   https://example.com/mcp-servers"
            }
        ],
        "isError": False
    }
}
```

### Step 8: Server Shutdown and Reconnection

MCP servers are separate processes, so a crash in one does not take down your AI application. The client detects the disconnected server, marks it unavailable, and — depending on the client — either automatically reconnects or waits for you to manually reconnect it. This process isolation is one of the underrated benefits of the architecture: a buggy or overloaded database server does not corrupt or hang the AI application driving it.

When the client shuts down normally, it sends a `shutdown` notification to each server process. The server responds by cleaning up: closing file handles, disconnecting from databases, releasing API connections, and flushing any in-memory state to disk if applicable.

## Connecting Multiple Servers Simultaneously

One of the most powerful features of MCP is connecting multiple servers to the same client at the same time. Your AI application can access:

- **The filesystem** for reading and writing project files
- **GitHub** for issue tracking and code review
- **A database** for querying application data
- **Brave Search** for finding documentation online
- **Slack** for team communication
- **Playwright** for browser automation
- **Memory** for persistent fact storage across sessions

Each server operates independently. They do not conflict because each tool has a unique name (scoped to its server) and each resource has a unique URI. The client manages routing: when the LLM calls a tool, the client identifies which server owns that tool (based on the tool list returned during capability negotiation) and forwards the request to the correct process.

## Cost, Limits, and Safety

Every server you connect inherits the cost structure and limits of whatever it wraps, and those costs are easy to lose track of once tool calls are happening automatically inside a conversation instead of one command at a time in a terminal.

### API Costs Are Per-Server, and Some Are Metered

- **Brave Search** — Free tier caps out at a fixed number of queries per month. Past that, you pay per query or per thousand queries depending on your plan.
- **GitHub** — API enforces rate limits (thousands of requests per hour for an authenticated token, far fewer for unauthenticated access). Hitting that ceiling mid-conversation surfaces as a tool error unless the server itself implements backoff.
- **Slack** — Free for workspace-level operations within standard rate limits. Heavy usage may require a paid Slack plan.
- **Postgres** — Database server costs are separate. MCP does not add its own charges.
- **Filesystem, SQLite, Memory, Sequential Thinking** — Free, entirely local, no API costs.

### Rate Limits Compound When a Model Decides the Number of Calls

A human developer calling the GitHub API manually thinks about each request. A model given a `search_issues` tool and an open-ended instruction like "find every issue related to authentication" may call the tool many times in a row: once per page of results, once per repository, once per retry after a transient error. Watch your usage on any metered server closely during your first sessions, and prefer servers that cap page size and result count by default. Implement retry logic with exponential backoff for transient failures, and consider using the Memory MCP server to cache frequent API responses.

### Filesystem and Database Access Carry Real Security Implications

A filesystem server configured to allow access to your home directory, rather than a single project folder, gives the model the ability to read anything you can read — SSH keys, browser session tokens, other projects' secrets — the moment a prompt (yours or an injected one from a document the model is processing) asks it to. Always scope a filesystem server to the narrowest directory the task actually needs.

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@anthropic/mcp-server-filesystem",
        "/Users/you/projects/current-project-only"
      ]
    }
  }
}
```

The same logic applies doubly to database servers. A Postgres or SQLite server given unrestricted write access can drop tables or corrupt data from a single malformed or manipulated request. Every database MCP server worth using offers a read-only or restricted access mode. Enable it by default and only grant write access for the specific task that needs it.

### Credentials Belong in Environment Variables

A `GITHUB_PERSONAL_ACCESS_TOKEN`, a `BRAVE_API_KEY`, a database connection string with an embedded password — all of these should be read from your shell environment or a `.env` file that is excluded from version control, and referenced by name in your MCP server configuration, not pasted into it directly.

```bash
# Good: pass credentials via environment variables
# The config references the env var name, not the value
export BRAVE_API_KEY="your_key_here"
npx -y @brave/brave-search-mcp-server

# Never: hardcode credentials in the config file or command line
npx -y @brave/brave-search-mcp-server --api-key "sk_live_..."  # BAD
```

### Credential Management Best Practices

1. Use environment variables for all API keys and tokens.
2. Add credential files to `.gitignore` so they are never committed.
3. Use a `.env` file for local development and a secrets manager in production.
4. Rotate tokens regularly and revoke any that are exposed.
5. Grant the minimum permissions required. A GitHub token used only for reading issues does not need admin access.
6. For database servers, prefer read-only connections by default. Use a database user with `SELECT`-only privileges.

## When to Use Existing MCP Servers

Reach for an existing MCP server whenever the capability you need is common enough that someone else has almost certainly already built and hardened it.

**Use a pre-built server when:**

- **The tool or service is common** — Filesystem access, database queries, web search, and version control operations are well-solved problems. The official servers are maintained and documented.
- **You want to move fast** — Installing a pre-built server takes minutes. Writing a custom integration takes hours or days.
- **The server is well-maintained** — Official servers under the `@modelcontextprotocol` or `@anthropic` scope, or vendor-published servers like `@playwright/mcp` or `github/github-mcp-server`, receive regular updates and fixes.
- **You need standard functionality** — If your use case matches the tools and resources the server provides, there is no reason to build from scratch.
- **The server is local** — Local servers (filesystem, SQLite, memory) have zero latency to external APIs and no ongoing costs.

**Prefer the popular, actively maintained option over the obscure one**, even when the obscure one looks like a closer fit on paper. A server with recent commits, real issue-tracker activity, and meaningful adoption is one whose bugs other people have already found and whose security holes get patched. A server last updated over a year ago, with no responses to open issues, is a server you would be debugging alone. For anything touching credentials or write access to real systems, that risk is rarely worth it.

## When to Build a Custom Server Instead

A custom MCP server makes sense when:

- **The tool or service is niche** — If you need access to an internal API, a proprietary database, or a specialized tool, no pre-built server exists. A thin custom server wrapping one internal API is often both safer and simpler than trying to adapt a general-purpose server.
- **The existing server is unmaintained** — Servers with no recent commits, unresolved issues, or poor documentation are risky dependencies. If the only option is an abandoned server, building your own is the safer path.
- **You need different security boundaries** — An existing server may expose too much functionality. For example, a filesystem server that allows all file operations when you only need read access. Or a database server that offers unrestricted write access when you need fine-grained per-table permissions.
- **You need fine-grained authorization** — Different tools available to different users, or different scopes based on the request context. A generic pre-built server has no concept of your authorization model.
- **The server has latency requirements you cannot meet** — Some servers start slowly or add overhead from their dependency tree. A minimal custom server may be more efficient.
- **You need to combine multiple tools** — If your use case always calls two tools in sequence, you may want a single server that exposes one combined operation.

Do not reach for an MCP server at all when a direct API call would suffice. If your application only ever needs to make one kind of call to one kind of service, and no LLM is deciding when or how to call it, you do not need the protocol overhead of a server, a client, and capability negotiation. A plain SDK call to that service's API is simpler, faster, and has one less moving part to keep running.

## Mental Model Check

Test your understanding of using existing MCP servers with these questions. Try to answer each before reading the explanation.

### Q1: You want to add web search capability to your AI assistant. What do you need to do?

**Answer:** You need to install and configure the Brave Search MCP server. First, obtain a Brave Search API key from the Brave Search API website. Then, configure the server in your `claude_desktop_config.json` or register it with `claude mcp add`. The server requires the `BRAVE_API_KEY` environment variable. Once configured, restart your client. During startup, the client will spawn the server process, perform capability negotiation, and discover the `search` tool. Your AI assistant can then call `search` with your query to get web results.

### Q2: You installed the GitHub MCP server, but your AI assistant says it cannot find the `create_issue` tool. What could be wrong?

**Answer:** Several things could be wrong. First, check that the server started successfully by looking at the client logs for initialization errors. Second, verify that your GitHub personal access token has the required permissions (it needs `issues:write` scope to create issues). Third, confirm that capability negotiation completed — if the server failed to initialize (e.g., due to a bad token), it would not have sent its tool list. Fourth, check that you are using the correct package: the original `@modelcontextprotocol/server-github` has been archived. Use `github/github-mcp-server` instead. Fifth, try running the server directly from the command line to see any error output before it gets swallowed by the client.

### Q3: A filesystem MCP server is configured with `/Users/you/projects` as the allowed directory. Can the AI assistant read `/etc/passwd`?

**Answer:** No. The filesystem server should reject any path that is not within the allowed directory. A well-implemented filesystem server normalizes paths, resolves symlinks, and verifies that the resulting absolute path starts with the allowed directory prefix. A path like `/etc/passwd` or `../../etc/passwd` is rejected with an error. However, not all servers implement these checks correctly. Always verify that the MCP server you use implements proper path restriction, and test it with at least one traversal attempt to confirm. This is particularly important if the allowed directory contains sensitive subdirectories you do not want the model to read.

### Q4: Your application uses three MCP servers. The Postgres server crashes. Do the other two stop working?

**Answer:** No. MCP servers run as independent operating system processes. If the Postgres server crashes, the client receives a transport error for that specific server and marks it unavailable, but continues communicating with the filesystem and Brave Search servers normally. The AI assistant may notice that database queries are unavailable and tell you about it, but file reads and web searches continue to function. Some MCP clients support automatic reconnection — the client periodically tries to restart the crashed server and, if successful, re-runs capability negotiation. Check your client documentation to see if auto-reconnect is supported.

### Q5: You need your AI assistant to query a PostgreSQL database and then post the results to a Slack channel. Should you use one MCP server or two?

**Answer:** Use two separate MCP servers — one for PostgreSQL and one for Slack. Each server handles one domain. The AI assistant can call the PostgreSQL server to run a query, receive the results, and then call the Slack server to post them. This is the intended pattern: servers are single-purpose adapters, and the AI assistant orchestrates the workflow by calling them in sequence. If you find yourself needing this exact combination frequently (querying the database and posting to Slack as a single operation), you could build a custom server that combines both — but start with two separate servers. The single-purpose approach is simpler to maintain, easier to debug (you can test each server independently), and more reusable (you can reuse the same PostgreSQL server with a different notification channel later).
