These exercises let you practice installing, configuring, and working with existing MCP servers on your own. Each tier increases in complexity, from a single server with a clear goal to multi-server orchestration and custom configuration. Choose the tier that matches your comfort level, or work through all three in order.

## How to Use These Exercises

Each exercise follows the same structure: a scenario description, step-by-step instructions, an expected outcome, and check-your-understanding questions. Some exercises include troubleshooting tips for common issues. The exercises are designed to be completed in order within each tier, but you can skip around if you already have experience with certain servers.

Every exercise expects you to run real commands in your terminal and test the results. You will learn more by debugging failures than by getting everything right the first time — when something breaks, the troubleshooting steps and deepening questions will help you understand why.

## Before You Begin

These exercises assume you have:

- **Claude Code** installed and authenticated
- **Node.js 18+** available (`node --version` to verify)
- **npm** available (for `npx`-based servers)
- **Docker** available (for GitHub server exercises — optional if you use the remote transport alternative)
- **A GitHub personal access token** with `repo` and `issues` scopes
- **A Brave Search API key** (sign up at `brave.com/search/api` for the free tier)
- **A clean test directory** where you can freely create and delete files

Set your credentials as environment variables before starting any exercise that needs them:

```bash
export GITHUB_PERSONAL_ACCESS_TOKEN=ghp_your_token_here
export BRAVE_API_KEY=your_brave_api_key_here
```

If you skip this step, the servers will start but return authentication errors when you try to use them. Each exercise tells you which servers it needs, so you can set up credentials only for the ones you plan to try.

This is also a good time to create a sandbox project directory:

```bash
mkdir -p ~/mcp-sandbox
cd ~/mcp-sandbox
echo "# MCP Sandbox" > README.md
mkdir src tests docs
cat > src/app.py << 'EOF'
def hello():
    print("hello from mcp sandbox")

if __name__ == "__main__":
    hello()
EOF
cat > tests/test_app.py << 'EOF'
import pytest
from src.app import hello

def test_hello():
    assert hello() is None
EOF
```

You will use this directory for exercises that involve the filesystem server. Having a real project structure with multiple files makes the exercises more realistic.

## Tier 1: Beginner — Single Server Setup and Use

These exercises focus on installing and using one MCP server at a time. Each one reinforces the core lifecycle: install, register, verify, use, remove.

### Exercise 1.1: Memory Server — Persistent Facts

Set up the Memory MCP server, which maintains a knowledge graph that Claude can write to and query across a session.

**Instructions:**

1. Register the Memory server with Claude Code using `@modelcontextprotocol/server-memory`.
2. Verify the server is running with `claude mcp list`.
3. Start a Claude Code session and ask Claude to remember several facts about you — your name, your favorite programming language, the project you are working on.
4. End the session and start a new one. Ask Claude what it remembers about you.

**Expected outcome:** The first session stores facts in the knowledge graph. The second session retrieves them — the Memory server persists across sessions because it writes to a local file on disk. If the second session does not remember anything, check whether the server was configured with a `--memory-file` flag pointing to a persistent path; the default writes to a file in the current working directory.

**Check your understanding:** Why does the Memory server use a persistent knowledge graph rather than relying on the model's context window? (The context window is ephemeral — it lasts only for the duration of a single session and is lost when the session ends. The knowledge graph persists to disk, so facts survive across sessions, restarts, and even client upgrades.) What happens to stored facts when you remove the Memory server's registration? (The facts are stored in a file on disk. Removing the registration just stops Claude Code from launching the server; the data file remains and will be available again if you re-register the server pointing at the same file.)

### Exercise 1.2: Fetch Server — Download Web Content

Set up the Fetch MCP server, which downloads a URL and returns its contents as markdown. This is useful for having Claude read documentation, blog posts, or other web pages without needing a full search integration.

**Instructions:**

1. Register the Fetch server with Claude Code using `@modelcontextprotocol/server-fetch`.
2. Start a Claude Code session and ask Claude to fetch and summarize a real documentation page — for example, the FastAPI documentation at `https://fastapi.tiangolo.com` or the Python `requests` library docs at `https://requests.readthedocs.io`.
3. Try fetching a page that requires JavaScript rendering — a Single Page Application that loads content dynamically. What happens? Why?

**Expected outcome:** Plain HTML pages download and convert to markdown cleanly. Pages that depend on client-side JavaScript may return incomplete or empty content, because the Fetch server sends an HTTP request and parses the HTML response — it does not run a browser.

**Check your understanding:** The Fetch server does not execute JavaScript. For what kinds of pages is this server useful? (Documentation pages, blog posts, API reference pages — any page where the content is in the initial HTML.) When would you need a browser automation server like Playwright instead? (When the page loads content dynamically with JavaScript — SPAs, paginated content behind a "load more" button, or any page that requires user interaction like clicking or form submission.)

### Exercise 1.3: Playwright Server — Browser Automation

Set up the Playwright MCP server (`@playwright/mcp`), which drives a real browser. Unlike the Fetch server, this one can execute JavaScript, click buttons, fill forms, and take screenshots.

**Instructions:**

1. Register the Playwright server with Claude Code using `npx @playwright/mcp@latest`.
2. Start a Claude Code session and ask Claude to navigate to a web page, take a screenshot, and save it to your project directory.
3. Ask Claude to find a specific piece of information on a page — for example, "Go to the MDN documentation for `Array.map` and tell me what arguments the callback function receives."

**Expected outcome:** Claude launches a headless browser, navigates to the URL, reads the accessibility tree or extracts text, and returns the information. Screenshots are saved as PNG files to your filesystem through the filesystem server (not through Playwright itself — you will need the filesystem server registered to save files).

**Check your understanding:** Why is Playwright a more appropriate choice than the Fetch server for a page that has a "load more" button or an interactive form? What are the trade-offs in startup time and resource usage between a browser automation server and a plain HTTP fetch server?

### Exercise 1.4: Time Server — Current Date and Time

Set up the Time MCP server (`@modelcontextprotocol/server-time`), which provides current date and time information in various time zones.

**Instructions:**

1. Register the Time server with Claude Code.
2. Ask Claude "What time is it right now in Tokyo, London, and New York?"
3. Ask Claude "What is the current UTC time, and what Unix timestamp corresponds to it?"
4. Ask Claude "If a meeting is scheduled for 3 PM Eastern Time on July 15, 2026, what time is that in Singapore?"

**Expected outcome:** Claude uses the time server to get accurate current time data and timezone conversions. The server provides structured time data that Claude can format and explain.

**Check your understanding:** Why does this server exist when you could just ask the model to tell you the current time? What inherent limitation of LLMs does this server address?

## Tier 2: Intermediate — Combining Multiple Servers

These exercises require you to run two or more MCP servers simultaneously and ask Claude to coordinate their use.

### Exercise 2.1: Research and Save

Set up the Brave Search server and the Filesystem server together — the most common two-server combination for research tasks.

**What you will observe:** When you give Claude a prompt that requires both web search and file writing, it automatically figures out which server to call at each step. The search comes from Brave Search; the write operation goes through the Filesystem server. You never tell it which server to use — the model reasons about the available tools and picks the right one.

**Instructions:**

1. Register both servers if they are not already registered.
2. Ask Claude: "Search the web for three tutorials about building MCP servers. For each one, save the title, URL, and a one-paragraph summary to a file called `docs/mcp-tutorials.md`."
3. Verify the file was created and has the expected content by asking Claude to read it back to you.
4. Ask Claude to update the file by adding a new section: "Add a fourth tutorial that covers a specific aspect of MCP — security best practices."

**Expected outcome:** Claude calls the Brave Search server to find tutorials, then writes the compiled results to a file through the Filesystem server. The entire task is a single natural language prompt. When you ask for an update, Claude reads the existing file (Filesystem server), searches for new content (Brave Search server), and writes the updated version (Filesystem server again).

**Challenge variant:** Add a step asking Claude to then fetch the full content of each tutorial using the Fetch server and save the full text. You will need three servers — Brave Search, Fetch, and Filesystem — working in sequence. This variant is more realistic because many research workflows involve: search for sources, fetch the full content, synthesize, and save — the exact pattern MCP servers are designed to support.

**Troubleshooting tip:** If Claude saves the file but you cannot find it, ask Claude "Where did you save the file?" The model knows the path it used. The file may be in the directory you passed to the filesystem server during registration, which may differ from your current working directory.

### Exercise 2.2: GitHub Issues from Search Results

Set up the Brave Search server and the GitHub server together. This exercise simulates a common developer workflow: discover a project online, then explore its GitHub presence.

**Instructions:**

1. Register both servers.
2. Ask Claude: "Search the web for open source projects related to AI safety that are actively maintained. Find their GitHub repositories and check if any of them have issues labeled 'help wanted' or 'good first issue'. Summarize the top three projects and list one 'good first issue' from each."
3. Optional extension: "Create a new file called `docs/contribution-opportunities.md` with the same information."

**Expected outcome:** Claude uses Brave Search to find projects, then switches to the GitHub server to inspect each project's issues. If the file extension is used, the Filesystem server becomes a third participant. This exercise demonstrates how the model chooses between servers based on what each step requires — it is a dynamic decision made at runtime, not a pre-scripted pipeline.

**Troubleshooting tip:** If Claude seems stuck or slow, it might be making many individual GitHub API calls. The model does not know how many issues a repository has until it asks. A repository with hundreds of labeled issues could trigger many tool calls. If this happens, narrow the scope: "Find just the top 3 'good first issue' labels across all AI safety projects, not every individual issue."

**Check your understanding:** How does the model decide which server to call for each step of this task? Does it plan the sequence ahead of time, or does it decide step by step? (The model typically plans a rough sequence upfront — "first I need to search, then I need to check GitHub" — but the exact tool calls are decided step by step as the results of each call become available. This is the same reasoning-vs-execution pattern discussed in agent fundamentals.)

### Exercise 2.3: Document Your Workspace

Set up the Filesystem server, the GitHub server, and a memory server together — a three-server exercise that mirrors a real developer workflow.

**What makes this interesting:** Unlike the previous exercises that used servers sequentially (search, then write), this one requires the model to cycle between servers multiple times as it discovers new information. It reads from the filesystem, checks against GitHub, writes to a file, and remembers a fact for later — all in a single session.

**Instructions:**

1. Register all three servers (Filesystem, GitHub, Memory).
2. Ask Claude: "Explore my project directory. Find every file that contains the string 'TODO' or 'FIXME'. For each file, extract the relevant lines. Check whether each file is part of a GitHub repository by examining any git remote configuration. Save the report to `docs/todo-audit.md`, and tell the memory server that we found outstanding TODOs so future sessions know about them."
3. After the task completes, end the session and start a new one.
4. In the new session, ask Claude: "What did we find about TODOs in the project last time?"

**Expected outcome:** In the first session, Claude uses the filesystem server to search the project (using `search_files` or reading individual files), the GitHub server to check repository information, the filesystem server again to write the report, and the memory server to store a persistent fact. In the second session, the memory server should recall the stored fact — if it does not, check whether the memory server was configured with a persistent storage path.

**Check your understanding:** The Memory server persists data across sessions by writing to a local file. Where does it store that file, and what happens to the stored data if you delete that file or switch to a different working directory? (The default storage location is a file in the current working directory. Switching directories means starting with an empty knowledge graph — specify `--memory-file` with an absolute path for persistent storage across directories.)

### Exercise 2.4: Multi-Source Research Brief

Set up Brave Search, Fetch, and Filesystem servers — a three-server pipeline that mirrors a real research workflow.

**Why this matters:** This is the most common real-world pattern for MCP servers in knowledge work. A human researcher would spend 30-60 minutes doing what Claude does in under a minute: discover sources, read them, synthesize, and save. The MCP server model makes this available as a natural language request with zero integration code.

**Instructions:**

1. Register all three servers (Brave Search, Fetch, Filesystem).
2. Ask Claude: "I need a research brief on 'vector databases for RAG'. Search the web for the top articles, fetch the full content of the top three results, synthesize them into a concise brief with key takeaways, and save it as `docs/vector-db-brief.md`."
3. Read the saved file. Does it contain a genuine synthesis or just a list of bullet points from each article?
4. Ask Claude to expand the brief by searching for one more aspect: "Add a section about performance benchmarks comparing different vector databases."

**Expected outcome:** Claude orchestrates a sophisticated multi-step pipeline: search (Brave Search), fetch each result (Fetch), synthesize in its response, and write the final version (Filesystem). Each server is used at the right stage of the pipeline. The expansion step shows that Claude can revisit the search step with a refined query and update the saved file accordingly.

**Evaluation criteria for your brief:**

- Does it include specific product names (Pinecone, Weaviate, Qdrant, Milvus) rather than generic descriptions?
- Does it mention specific metrics or benchmarks, or only high-level concepts?
- Does it cite sources (even informally like "according to article X") so you know where the information came from?
- If the brief is vague or generic, ask Claude to be more specific and include exact numbers or comparisons.

**Check your understanding:** What happens if the Brave Search server returns only two relevant results instead of three? Does the pipeline fail, or does Claude adapt? (Claude adapts — it works with whatever results it gets. The Fetch server may fail on one of the URLs if the page is behind a login wall or returns an error, and Claude will note the failure and proceed with the results it did get. This graceful degradation is another benefit of letting the model orchestrate the pipeline rather than hardcoding it.)

## Tier 3: Advanced — Custom Configuration and Troubleshooting

These exercises push you beyond the basic install-and-use pattern into configuration management, security awareness, and debugging.

### Exercise 3.1: Scoped Filesystem Access

Set up two filesystem servers that point at different directories, then test the boundary between them.

**Instructions:**

1. Register two filesystem servers with different names and different allowed directories:
   - `fs-project` pointed at your project directory
   - `fs-config` pointed at `~/.config` or another configuration directory
2. In a Claude Code session, ask to read a file from your project (should succeed via `fs-project`).
3. Ask to read a file from the config directory (should succeed via `fs-config`).
4. Ask to copy a file from the config directory into the project directory.

**Expected outcome:** Steps 2 and 3 work through their respective servers. Step 4 is a cross-server operation — Claude can read the source file through `fs-config` and write through `fs-project`, but it requires two separate tool calls. Notice that Claude cannot simply "copy" a file across server boundaries in one call; it must read through one server and write through the other.

**Deepening question:** What security principle does this two-server architecture enforce? How does it compare to a single filesystem server given full access to your home directory?

### Exercise 3.2: Diagnose a Misconfigured Server

You are given a correctly registered server that does not work. Your job is to diagnose why. This is the most common support scenario when working with MCP servers — the server starts, it registers, but tool calls fail.

**Setup scenario:** Register the Brave Search server without setting the `BRAVE_API_KEY` environment variable. The server starts up fine because the missing key does not prevent initialization. But when you ask Claude to search the web, the server returns an error.

**Instructions:**

1. Do not set `BRAVE_API_KEY` before registration. Register the server as normal with `claude mcp add`.
2. Run `claude mcp list` to confirm the server shows as "running." Notice that the server starts without complaint even though the key is missing — it does not validate credentials during initialization.
3. Start a Claude Code session and ask Claude a search question, for example: "Search the web for the latest version of Node.js."
4. Observe the error message carefully. It comes from the MCP server, not from Claude itself. The message will say something like "API key not configured" or "Authentication failed" — this is the Brave Search server reporting that it cannot authenticate with the Brave API.
5. Now set the environment variable in your terminal: `export BRAVE_API_KEY=your_key_here`.
6. Try the same search question again without restarting Claude Code. Does it work?
7. If it does not work, exit the Claude Code session and start a new one. Does it work now?

**Expected outcome:** The server starts without the key because the server does not validate the key at startup — it checks only when a search request arrives. Setting the variable after the server is already running may not fix the issue if Claude Code caches the original environment; you may need to close and restart the session. This is a common debugging scenario when working with API-key-backed MCP servers.

**Deepening question:** How would you design an MCP server that validates its credentials at startup and refuses to initialize if they are missing? What trade-off does that approach introduce? (Failing early makes debugging easier, but it makes the server impossible to start if the credential service is temporarily unavailable — a pattern called "startup dependency coupling.")

### Additional Diagnostic Exercise: Debug a Port Conflict

Some MCP servers (especially those using HTTP/SSE transport instead of stdio) bind to a network port. If that port is already in use, the server fails to start.

**Scenario:** You have registered a server that uses SSE transport, and you suspect another process is already using the port it needs.

**Instructions:**

1. Find which process is using a specific port: on macOS/Linux, run `lsof -i :PORT_NUMBER`; on Windows, run `netstat -ano | findstr :PORT_NUMBER`.
2. Identify whether the conflicting process is another MCP server, a development server, or a system service.
3. Decide how to resolve the conflict: either stop the conflicting process, or reconfigure the MCP server to use a different port.

**Expected outcome:** You learn how to diagnose port conflicts and resolve them — a skill you will need when running multiple MCP servers locally, especially during development.

**Deepening question:** Why do most MCP servers default to stdio transport instead of HTTP/SSE? What advantage does stdio offer over HTTP for local development? (Stdio avoids port management entirely, is simpler to configure, and has lower latency because there is no TCP handshake overhead.)

### Additional Diagnostic Exercise: Environment Variable Scope

This exercise reveals a subtle issue: environment variables set in one shell session may not be visible to MCP servers launched from a different session.

**Scenario:** You open two terminal windows. In terminal A, you set `BRAVE_API_KEY=mykey` and register the Brave Search server. In terminal B, you start Claude Code.

**Instructions:**

1. Open two terminal windows.
2. In terminal A, set `BRAVE_API_KEY` and register the Brave Search server.
3. In terminal B, start Claude Code and try to use the Brave Search server.
4. If it fails, the issue is that the server process inherited the environment from terminal A, but the Claude Code client in terminal B does not have the variable.

**Expected outcome:** You understand that MCP server processes inherit environment variables from the shell where `claude mcp add` or `claude` was launched — not from the shell where you set the variable. The fix is to either set the variable in your shell profile (`~/.zshrc`, `~/.bashrc`, or system environment variables) or to launch Claude Code from the same terminal where the variable is set.

**Deepening question:** How would you configure environment variables so that they are available to MCP servers regardless of which terminal launches Claude Code? (Answer: set them in your shell profile or in Claude Code's settings file if it supports environment variable overrides for servers.)

### Exercise 3.3: Customize a Server's Configuration

Some MCP servers expose configuration options beyond their defaults. This exercise explores the filesystem server's search tool.

**Instructions:**

1. Register the filesystem server with a directory that has a deliberately complex nested structure (create a few levels of subdirectories with files).
2. Ask Claude to "find all JavaScript files in the project that contain the word 'async'." Note the result.
3. Now ask Claude to "use the search tool with a case-insensitive pattern to find all files containing 'TODO'." Observe whether the server supports that option.

**Expected outcome:** The filesystem server's `search_files` tool accepts glob patterns and optionally content filtering. Asking for the right kind of search reveals whether the server supports advanced features like regex, case sensitivity, or file type filtering. Different MCP servers expose different levels of configuration through their tool schemas.

**Deepening question:** What is the difference between a server that exposes many parameters on its tools and a server that exposes few? Which is easier to use, and which is more flexible?

### Exercise 3.4: Build a Portable Server Configuration

Create a reusable server configuration file that you could share with a teammate or check into a project repository (with secrets handled safely).

**Instructions:**

1. Create a JSON file called `mcp-servers.json` in your project directory that contains configurations for the Filesystem server (scoped to the project directory) and the GitHub server (using a placeholder for the token).
2. The file should have a structure like this, using an environment variable reference instead of a hardcoded token:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "${PROJECT_DIR:-.}"]
    },
    "github": {
      "command": "docker",
      "args": ["run", "-i", "--rm", "-e", "GITHUB_PERSONAL_ACCESS_TOKEN", "ghcr.io/github/github-mcp-server"]
    }
  }
}
```

3. Document in a comment (or in a `## Setup` section in the JSON file) that the user must have `GITHUB_PERSONAL_ACCESS_TOKEN` set in their environment and Node.js installed.
4. Test that the configuration works by asking Claude to verify both servers are functional.

**Expected outcome:** You end up with a portable, project-scoped MCP server configuration that any teammate (or your future self on a new machine) can apply after setting their credentials. The token is never in the file — it comes from the environment.

**Deepening question:** What would you add to this configuration to make it production-ready? Consider monitoring, logging levels, resource limits on the Docker container, and error recovery policies.

### Exercise 3.5: Server Discovery and Evaluation

This exercise simulates a real-world scenario where you need to evaluate an unfamiliar MCP server before trusting it with credentials.

**Scenario:** A teammate tells you about "an MCP server for Postgres" and asks you to set it up for the team. Your job is to find, evaluate, and configure it — without actually installing it on a production system.

**Instructions:**

1. Search the web for "Postgres MCP server" and identify the current best-maintained option. The overview page mentions `postgres-mcp` by Crystal DBA, but the ecosystem may have changed.
2. Evaluate the server on these criteria:
   - **Last update:** When was the package last published on npm or PyPI?
   - **Security model:** Does it support a read-only mode? How does it handle credentials?
   - **Transport:** Does it use stdio or SSE? Can it run as a child process, or does it need a network port?
   - **License:** Is it open source? What license?
   - **Adoption:** How many downloads or GitHub stars does it have?
3. Based on your evaluation, write a brief recommendation: "Yes, use this server" or "No, avoid this server" with your reasoning.

**Expected outcome:** You learn how to evaluate an MCP server before trusting it — a critical skill because not all community servers are well-maintained or secure. The evaluation criteria used here (update recency, security model, adoption) apply to any MCP server you consider integrating into a real project.

**Check your understanding:** If a server has not been updated in 18 months but has 10,000 weekly downloads, does that make it more trustworthy or less? (It is ambiguous: high downloads suggest many people use it and find it working, but no recent updates mean known bugs and API changes in the underlying service may never be patched. This is exactly the kind of judgment call you will face with real servers.)

### Exercise 3.6: Transport Type Trade-Offs

All the servers in these exercises use stdio transport (the server runs as a child process of the client). But MCP also supports SSE (Server-Sent Events) transport for remote servers.

**Scenario:** You need to share an MCP server with a teammate who works on a different machine. A stdio server only works on your machine. An SSE server could be accessed remotely.

**Instructions:**

1. Research what it would take to make the filesystem server available over SSE instead of stdio. The server package itself does not support SSE — you would need to wrap it.
2. Sketch the architecture: the filesystem server runs on machine A, an SSE proxy runs alongside it, and machine B's Claude Code connects to the SSE endpoint.
3. Answer these questions:
   - What security risks does remote filesystem access introduce?
   - How would you authenticate the remote client?
   - Under what circumstances would the latency of SSE be a problem?

**Expected outcome:** You understand that stdio is simpler and more secure (no network exposure) but limited to the local machine. SSE enables remote access but introduces authentication, encryption, and latency concerns. Most production setups start with stdio and only add SSE when a genuine multi-machine use case arises.

**Deepening question:** If you needed to run an MCP server in a cloud environment and have your local Claude Code connect to it, what additional infrastructure would you need compared to running it locally? (Consider networking, authentication, TLS, and process management.)

## What Next

After completing these exercises, you have experienced the full lifecycle of working with existing MCP servers: installing from npm or Docker, registering with `claude mcp add`, verifying the server is connected, using it in natural language prompts, combining multiple servers for compound tasks, and diagnosing configuration issues. The skills transfer directly to any other MCP server in the ecosystem — the protocol is the same, the registration command is the same, and the pattern of asking Claude "what tools do you have" and letting it figure out the right one to call is the same.

To go further, pick one server from the reference table in the overview page that you have not tried yet — the Postgres server or the Slack server — and work through the same install-verify-use cycle on your own. The only thing that changes between servers is the package name, the arguments, and the credentials you need.

## Quick Reference: Common MCP Server Registration Commands

Keep this cheat sheet handy as you work through the exercises:

| Server | Registration Command |
|---|---|
| Filesystem | `claude mcp add --transport stdio filesystem -- npx -y @modelcontextprotocol/server-filesystem /path/to/dir` |
| GitHub (Docker) | `claude mcp add --transport stdio github -- docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN ghcr.io/github/github-mcp-server` |
| Brave Search | `claude mcp add --transport stdio brave-search -- npx -y @brave/brave-search-mcp-server` |
| Memory | `claude mcp add --transport stdio memory -- npx -y @modelcontextprotocol/server-memory` |
| Fetch | `claude mcp add --transport stdio fetch -- npx -y @modelcontextprotocol/server-fetch` |
| Playwright | `claude mcp add --transport stdio playwright -- npx @playwright/mcp@latest` |
| Time | `claude mcp add --transport stdio time -- npx -y @modelcontextprotocol/server-time` |

Remove a server when you are done:

```bash
claude mcp remove <server-name>
```

List all registered servers and their status:

```bash
claude mcp list
```
