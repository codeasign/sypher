You are about to set up three real MCP servers, connect them to Claude Code, and use them to complete practical tasks. By the end of this exercise, you will have a filesystem server reading and writing project files for you, a GitHub server managing pull requests and issues on your behalf, and a Brave Search server answering questions with live web results. The same pattern — install, configure, use — applies to every other MCP server you encounter later.

## Prerequisites

Before you begin, make sure you have these ready:

- **Node.js 18+** installed (`node --version` to check)
- **Claude Code** installed and authenticated
- **A GitHub personal access token** with `repo` and `issues` scopes (generate one at `github.com/settings/tokens`)
- **A Brave Search API key** (sign up at `brave.com/search/api` for the free tier)
- **A temporary project directory** — create a folder called `mcp-playground` somewhere you can safely create and delete files

```bash
mkdir -p ~/mcp-playground
cd ~/mcp-playground
git init
echo "# MCP Playground" > README.md
mkdir src docs
echo "console.log('hello from mcp')" > src/index.js
echo "# Project Docs" > docs/overview.md
```

You will also need your GitHub personal access token and Brave API key stored in environment variables so the MCP servers can pick them up without you pasting secrets into config files.

```bash
export GITHUB_TOKEN=ghp_your_token_here
export BRAVE_API_KEY=your_brave_api_key_here
```

On Windows, use `$env:GITHUB_TOKEN = "ghp_your_token_here"` in PowerShell, or set them through your system environment variables panel. On macOS and Linux, you can also add the `export` lines to your `~/.zshrc` or `~/.bashrc` so they persist across terminal sessions.

## Part 1: Filesystem Server

The filesystem server is the simplest MCP server to set up and the easiest to verify works. It gives Claude the ability to read, write, and search files within a directory you explicitly allow — nothing outside that directory is accessible.

### Step 1.1: Register the Server

Open a terminal and register the filesystem server with Claude Code, pointing it at the `mcp-playground` directory you just created:

```bash
claude mcp add --transport stdio filesystem -- npx -y @modelcontextprotocol/server-filesystem ~/mcp-playground
```

This command tells Claude Code: "When you start, spawn a process that runs `npx -y @modelcontextprotocol/server-filesystem ~/mcp-playground`. Name this server `filesystem`." The `npx -y` flag auto-accepts the download without prompting, so the server starts immediately.

### Step 1.2: Verify the Server Is Running

```bash
claude mcp list
```

You should see `filesystem` in the output, along with its status and a count of available tools. If the server failed to start, the list will show it as disconnected — check that the path you passed exists and that you have Node.js 18+ installed.

### Step 1.3: Use the Server

Start a Claude Code session and try the following prompts one at a time. Watch how Claude uses the filesystem tools to complete each request.

```bash
claude
```

Once inside the session:

- "List all files in my project directory."
- "Read the README.md file and summarize what the project is about."
- "Create a new file called `src/utils.py` with a function that reverses a string."
- "Search for any files containing the word 'hello'."
- "Move `docs/overview.md` to `docs/readme-backup.md`."

After each prompt, examine the tool call details Claude displays. Notice that every operation returns a structured result — text content, a list of matching files, or a confirmation message. The model never has direct filesystem access; it goes through the MCP server, which enforces the directory boundary you configured.

### Step 1.4: Test the Directory Boundary

Try asking Claude to read a file outside the allowed directory:

- "Read the file at `/etc/hostname`."
- "List files in my home directory."

Watch what happens. The server returns a permission error — it refuses to operate outside the `~/mcp-playground` boundary you set during registration. This is not Claude refusing; it is the MCP server enforcing its access control. If you need access to a different directory, you register a separate server (or a second instance of the filesystem server with a different name and path).

### Step 1.5: Check What Your Configuration Looks Like

Claude Code stores MCP server configurations in its settings file. Take a look at what was written:

```bash
cat ~/.claude/settings.json
```

You will see something like this (the exact path depends on your platform):

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "C:\\Users\\you\\mcp-playground"
      ]
    }
  }
}
```

If you wanted to use the same server in Claude Desktop, you would copy this exact JSON block into `claude_desktop_config.json`. The configuration format is identical across clients.

## Part 2: GitHub Server

The GitHub server gives Claude the ability to interact with the GitHub API — create and browse issues, manage pull requests, search repositories, and inspect code. You need a personal access token with the appropriate scopes.

### Step 2.1: Register the Server

Register the GitHub MCP server, passing your token through an environment variable:

```bash
claude mcp add --transport stdio github -- docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN npx -y @modelcontextprotocol/server-github
```

Wait — that is the old, archived server. The current official server from GitHub ships as a Docker image. Use this instead:

```bash
claude mcp add --transport stdio github -- docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN ghcr.io/github/github-mcp-server
```

If you do not have Docker installed and cannot use it, there is an alternative setup using the remote HTTP transport that connects directly to GitHub's managed MCP endpoint. Use the `claude mcp add` command with the remote transport pointing at `https://api.githubcopilot.com/mcp`, and your token supplied as a bearer token. The Docker approach is simpler for local experimentation.

### Step 2.2: Handle Docker Issues (If Any)

If Docker is not available on your system, or if you prefer not to run a Docker container, register the server using the remote transport instead:

```bash
# Remove the Docker-based server first if you registered it
claude mcp remove github

# Re-register with the remote HTTP transport
claude mcp add --transport sse github -- https://api.githubcopilot.com/mcp
```

Note that the remote transport option requires you to configure authentication through Claude Code's settings rather than through environment variables. Check the GitHub MCP server documentation for the most up-to-date setup instructions, because the endpoint URL and authentication method may have changed since this was written.

### Step 2.3: Verify and Use the Server

With the server registered (either Docker or remote), start Claude Code and try these prompts:

- "List the open issues in the repository `vercel/next.js`."
- "Create an issue in my own `mcp-playground` repository (or in any test repo you control) with the title 'Test issue from MCP' and the body 'Created through the GitHub MCP server.'"
- "Search for repositories on GitHub that use the Model Context Protocol."
- "Show me the contents of the README for `modelcontextprotocol/servers`."

Observe how Claude uses tools like `search_issues`, `create_issue`, `search_repositories`, and `get_repository` to fulfill each request. The model is reasoning about which tool fits which question — you never tell it "call `search_issues`" directly.

### Step 2.4: Make a Multi-Step Request

Try a prompt that requires multiple tool calls:

- "Find repositories related to MCP that have open issues labeled 'good first issue', pick the one with the most stars, and summarize what its README says about contributing."

This single prompt triggers a chain of tool calls: one to search repositories, another to look up the most-starred one, another to read its README, and possibly another to list issues. Claude sequences these automatically because the GitHub server makes all the tools available.

## Part 3: Brave Search Server

The Brave Search server gives Claude the ability to search the live web and return real results. Unlike the filesystem and GitHub servers, this one requires an API key from a third-party service.

### Step 3.1: Get Your API Key

Go to `brave.com/search/api` and sign up for the free tier. The free plan gives you a limited number of queries per month — enough for learning and experimentation. Once you have your key, set it as an environment variable:

```bash
export BRAVE_API_KEY=your_brave_api_key_here
```

### Step 3.2: Register the Server

```bash
claude mcp add --transport stdio brave-search -- npx -y @brave/brave-search-mcp-server
```

The server reads the `BRAVE_API_KEY` environment variable from the environment when it starts. If the variable is not set, the server will start but return errors when you try to search, because it cannot authenticate with Brave's API.

### Step 3.3: Verify and Use the Server

Start Claude Code and try these prompts:

- "Search the web for the latest news about AI agents."
- "Find information about the Model Context Protocol and summarize what it is."
- "What are the top restaurants in San Francisco according to Brave Search?"
- "Compare the search results for 'MCP server' and 'Model Context Protocol' — are they about the same thing?"

For each response, note how Claude attributes the information. The model sees the search results as structured data — titles, URLs, descriptions — and presents them to you in natural language, but it is not making up the facts; they come from Brave's search index.

### Step 3.4: Understand the Cost Implications

Look at the Brave Search dashboard after running several queries. The free tier typically caps at 2,000 queries per month. If you run a session that makes dozens of search calls, you can exhaust that cap quickly. Ask Claude how many searches it is performing:

- "How many search calls did you just make to answer my questions?"

The model can tell you how many tool calls it made. This awareness is important because when you add a search MCP server to a real project, you should monitor usage — especially if the model is operating autonomously without human confirmation before each search.

Now that you have all three servers registered, start Claude Code and try a compound task that uses all of them:

This task uses:
- **Brave Search** — to find the testing framework
- **GitHub** — to look up issues
- **Filesystem** — to write the results

**Compound task 2:** "Read the README.md in my project, search the web for ways to improve it, and create a GitHub issue in the `modelcontextprotocol/servers` repo suggesting one improvement based on what you learned."

Watch how Claude orchestrates these across three different servers, each running as its own process. It does not know ahead of time which server to call for each step — it evaluates each subtask against the available tools and picks the right server dynamically.

## Part 5: Advanced — Extending What You Built

Now that the three servers are running, take a few minutes to experiment with variations that deepen your understanding of how MCP servers work.

### Experiment 5.1: Multiple Filesystem Servers with Different Scopes

Register a second filesystem server pointed at a different location:

```bash
claude mcp add --transport stdio fs-docs -- npx -y @modelcontextprotocol/server-filesystem ~/Documents
```

Now start a Claude Code session and ask: "Copy the README.md from my mcp-playground project into my Documents folder." Watch how Claude accesses the first filesystem server to read the file and the second one to write it. Because each server is an independent process with its own directory boundary, Claude cannot copy the file in a single tool call — it must read through one server and write through the other. This is a feature, not a limitation: it means the model cannot accidentally write sensitive data from one scope into another unless you explicitly set up the servers to permit it.

### Experiment 5.2: Rename Servers and Re-register

Remove the `brave-search` server and re-register it under a different name:

```bash
claude mcp remove brave-search
claude mcp add --transport stdio web-search -- npx -y @brave/brave-search-mcp-server
```

Ask Claude to search the web. Does it know which server to call, or does it need you to clarify? The model sees the server name and tool descriptions in its context — if the server name is descriptive enough ("web-search" vs "brave-search"), the model can usually figure out the right server to call. This experiment shows that server naming matters for usability.

### Experiment 5.3: Observe What Happens When a Server Crashes

Start a Claude Code session. While inside the session, manually kill the filesystem server process. On most systems, you can find the process using `ps aux | grep mcp-filesystem` and kill it with `kill <PID>`. Then ask Claude to read a file from the project directory. What happens?

The client detects the server disconnect and reports the server as unavailable. Depending on the client, Claude may automatically retry by restarting the server, or it may ask you to reconnect. This demonstrates process isolation in action — the GitHub and Brave Search servers continue working normally while the filesystem server is down.

## Part 6: Troubleshooting Common Issues

If any part of this exercise did not work as expected, here are the most common problems and their solutions.

### Server Fails to Start

If `claude mcp list` shows the server as "disconnected" or its status is not "running":

1. Check that the package name is spelled correctly. The filesystem server is `@modelcontextprotocol/server-filesystem` — a common mistake is typing `server-file-system` or omitting the `@modelcontextprotocol/` scope.
2. Check that any paths you passed actually exist. The filesystem server will refuse to start if the directory you gave it does not exist or is not readable.
3. For the GitHub Docker server, confirm Docker is running: `docker ps` should show a running daemon. If you get "Cannot connect to the Docker daemon", start Docker Desktop or your system's Docker service first.

### Server Runs but Returns Errors

If the server starts but tool calls fail:

1. Check environment variables. The Brave Search server needs `BRAVE_API_KEY` set in the environment where Claude Code starts. If you set the variable in a terminal window and then launched Claude Code from a different terminal (or from a desktop shortcut), the variable may not be available. Try: `export BRAVE_API_KEY=your_key_here && claude` to make sure the variable reaches the server process.
2. Check network access. The GitHub server needs to reach `api.github.com`. The Brave Search server needs to reach `api.search.brave.com`. If you are behind a corporate proxy, you may need to configure the proxy in the server's environment.

### Authentication Errors

If you see "401 Unauthorized" or "403 Forbidden":

1. For GitHub: verify your personal access token has the right scopes. Go to `github.com/settings/tokens` and check that the token includes `repo` (for private repos) and at minimum `public_repo` (for public repos). Regenerate the token if necessary.
2. For Brave Search: check your API key dashboard at `brave.com/search/api` to confirm your key is active and has not exceeded its rate limit. The free tier caps at 2,000 requests per month.

### Docker-Specific Problems

If you are using the Docker-based GitHub server:

1. On macOS and Windows, Docker Desktop must be running. Run `docker info` to verify.
2. The first pull of the `ghcr.io/github/github-mcp-server` image can take a minute or more. If the server seems to hang, wait for the download to complete.
3. On Linux, you may need `sudo` to run Docker, or configure your user to be in the `docker` group. If you run into permission issues, try: `sudo claude mcp add --transport stdio github -- sudo docker run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN ghcr.io/github/github-mcp-server`

## Part 7: Cleanup

When you are done experimenting, you can remove the servers from your Claude Code configuration:

```bash
claude mcp remove filesystem
claude mcp remove github
claude mcp remove brave-search
```

Removing a server does not uninstall the underlying package — `npx` caches it, and Docker images remain on disk. It just removes the registration entry from Claude Code's settings, so the server will not start automatically next time.

To fully clean up the filesystem server's cache:

```bash
# Find the npx cache location for the filesystem package
npx -y @modelcontextprotocol/server-filesystem --help 2>/dev/null || true
# The cache lives under ~/.npm/_npx/ on most systems
```

You do not need to clean the cache unless you are running low on disk space. The cached packages are harmless and will speed up the next `npx` invocation that needs them.

## Part 8: Reflection Questions

Answer these questions in your own words after completing the exercise. They will help solidify what you learned.

1. **What is the difference between installing a server and registering a server?** Installing happens when `npx` downloads the package to its cache (or Docker pulls the image). Registering happens when you run `claude mcp add`, which writes a configuration entry telling Claude Code how to launch the server. The server is not running until you start a Claude Code session, and the package is not permanently installed on your system path — it is cached in `npx`'s cache and runs on demand.

2. **Why can one server access the filesystem while another accesses GitHub, even though both are connected to the same Claude Code session?** Each server is an independent process with its own environment, its own credentials, and its own allowed operations. The filesystem server has access to the local disk; the GitHub server has a token that grants access to GitHub's API. They do not share state, credentials, or capabilities. The only thing they share is the protocol (MCP) and the client (Claude Code) that routes tool calls to the correct server based on which tool the model chose.

3. **What happens if the GitHub server crashes while the Brave Search server is still running?** The GitHub server's crash does not affect the Brave Search server or the Claude Code client itself. The client detects the disconnection and marks the GitHub server as unavailable. You can continue using the Brave Search server normally. When you fix the GitHub server and restart the session, it reconnects. This process isolation is a deliberate architectural choice.

4. **Why does the filesystem server need a directory path at registration time, while the Brave Search server does not?** The filesystem server's entire purpose is to gate access to a specific directory — it needs to know which directory to allow. The Brave Search server's purpose is to query a web API, which does not need a filesystem path. Each server's required arguments reflect what it needs to do its job. A database server would need a connection string; a Slack server would need a token.

5. **If you had to connect Claude Code to a Jira instance, how would you approach finding the right MCP server?** You would start by searching the `modelcontextprotocol/servers` repository on GitHub and checking the community-maintained server list. You would also search npm and GitHub for "mcp server jira" to find community packages. Before trusting one, you would check its last update date, the number of users, whether there are open issues about security, and whether the maintainer responds to reports. This is the same discovery process used for any third-party dependency.

## Summary

You set up three MCP servers, each with a different transport and authentication model:

| Server | Transport | Authentication | Key Configuration |
|---|---|---|---|
| Filesystem | stdio (child process) | None (directory boundary) | Allowed root path |
| GitHub | stdio (Docker) or SSE (remote) | GitHub token via env var or bearer token | GITHUB_PERSONAL_ACCESS_TOKEN |
| Brave Search | stdio (child process) | API key via env var | BRAVE_API_KEY |

Every server followed the same lifecycle: install (or pull a Docker image), register with `claude mcp add`, verify with `claude mcp list`, and use inside a Claude Code session. The registration syntax varied slightly — the filesystem server needed a path argument, the GitHub server needed a Docker invocation with an environment variable, the Brave Search server needed no arguments at all — but the overall shape was identical.

This is the pattern you will use for every MCP server you encounter. The ecosystem includes hundreds more: Postgres, Slack, Memory, Playwright, and dozens of community-maintained servers for everything from Jira to Snowflake to Google Drive. Install, configure, use — the same three steps, the same protocol, the same `claude mcp add` command.
