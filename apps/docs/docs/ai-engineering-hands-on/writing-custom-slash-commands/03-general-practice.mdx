## Tiered Practice Exercises

These exercises progress from foundational command creation to advanced multi-file workflows. Attempt each tier in order. Solutions are at the end.

## Beginner Tier

### Exercise B1: Create a Status Command

Build a custom slash command that displays the current project status in a formatted report. It should show the branch name, last commit message, and list of uncommitted files.

**Task**: Create `.claude/commands/status.md` with a command that runs `git status --short`, `git log --oneline -1`, and `git rev-parse --abbrev-ref HEAD`, then formats the output into a readable report.

**Hint**: Use `{{branch_name}}` for the branch and `{{diff}}` for the uncommitted changes. The frontmatter should set a descriptive `name` and `description`, and limit `tools` to `[Bash, Read]` since this command only inspects, never modifies.

**Solution**:

```markdown
---
name: status
description: Show formatted project status with branch, last commit, and uncommitted changes
tools: [Bash, Read]
---

You are generating a project status report.

Current branch: {{branch_name}}

Run the following commands and format the output into a clean report:

1. `git status --short` to show uncommitted changes
2. `git log --oneline -3` to show recent commits
3. `git rev-parse --abbrev-ref HEAD` to confirm the current branch

Format the output like this:

## Project Status
- Branch: {{branch_name}}
- Recent commits: (list)
- Uncommitted changes: (count)

Then list each changed file with its status (M for modified, A for added, D for deleted).
```

### Exercise B2: Add a Trigger Pattern

Extend the status command from B1 so it automatically suggests itself when you are looking at any project file.

**Task**: Edit the frontmatter of `.claude/commands/status.md` to add a `triggers` key that matches all common source files.

**Hint**: The `triggers` key accepts a list of glob patterns. Use `["*"]` to match any file, or be more specific with `["*.py", "*.js", "*.ts", "*.md", "*.json", "*.yaml"]`.

**Solution**:

```markdown
---
name: status
description: Show formatted project status with branch, last commit, and uncommitted changes
triggers: ["*.py", "*.js", "*.ts", "*.md", "*.json", "*.yaml", "*.yml", "*.go", "*.rs", "*.java"]
tools: [Bash, Read]
---

You are generating a project status report.

Current branch: {{branch_name}}

Run the following commands and format the output into a clean report:

1. `git status --short` to show uncommitted changes
2. `git log --oneline -3` to show recent commits
3. `git rev-parse --abbrev-ref HEAD` to confirm the current branch

Format the output like this:

## Project Status
- Branch: {{branch_name}}
- Recent commits: (list)
- Uncommitted changes: (count)

Then list each changed file with its status.
```

### Exercise B3: Command with Context

Create a command that reads the current file and explains what it does. This command should use `{{current_file}}` to target the file you are editing.

**Task**: Create `.claude/commands/explain.md` that reads the file at `{{current_file}}` and produces a summary of what the code does, what dependencies it imports, and what side effects it may have.

**Hint**: The `{{current_file}}` variable contains the path of the file currently active in the editor context. Your command should check if this variable is empty and handle that case gracefully.

**Solution**:

```markdown
---
name: explain
description: Explain the current file -- imports, purpose, and side effects
triggers: ["*.py", "*.js", "*.ts", "*.go", "*.rs"]
tools: [Read]
---

You are explaining the file at {{current_file}}.

If {{current_file}} is empty, inform the user that no file is currently selected
and ask them to open a file first.

Otherwise, read the file and produce a structured explanation:

1. What does this file do? (one sentence summary)
2. What imports does it use? (list them)
3. What are the main functions or classes? (list with brief descriptions)
4. What side effects exist? (file I/O, network calls, global state changes)
5. What would break if this file were deleted? (downstream dependencies)

Keep the explanation concise. Aim for a developer who is new to this project
to understand the file's role after reading your explanation.
```

### Exercise B4: Permission-Scoped Grep

Create a command that searches the codebase for a specific pattern and reports all matches. This command should be strictly read-only.

**Task**: Create `.claude/commands/find.md` that uses `Grep` to search for a pattern you specify and returns a formatted list of matches with file paths and line numbers.

**Hint**: Use `Grep` in the `tools` list. The command should ask you what pattern to search for. Since `{{selected_text}}` can capture highlighted text, you can use it to pre-populate the search term.

**Solution**:

```markdown
---
name: find
description: Search the codebase for a pattern and show formatted results
triggers: ["*"]
tools: [Grep, Read]
---

Search the codebase for the pattern specified by the user.

If {{selected_text}} is not empty, use it as the search term.
Otherwise, ask the user what pattern to search for.

For each match found, report:
- File path (relative to project root)
- Line number
- The matching line content (trimmed to 120 characters)

If there are more than 50 matches, show the first 50 and report the total count.
If there are no matches, say so clearly.

Do not modify any files -- this is a read-only search.
```

## Intermediate Tier

### Exercise I1: Multi-Step Review Pipeline

Combine two commands into a pipeline. The first command lints the code, and the second reviews the lint output. Create both commands and chain them.

**Task**: Create `/lint` that runs your project's linter (e.g., `flake8`, `eslint`, `golangci-lint`) and parses the output into a structured report. Then create `/review` that depends on the lint command and runs after it.

**Hint**: Use the prompt body of `/review` to instruct the model to first run `/lint`, then review the remaining issues. The `tools` list for `/review` should include `Skill` if you want it to invoke the lint command as a skill.

**Solution**:

Create `.claude/commands/lint.md`:

```markdown
---
name: lint
description: Run the project linter and format results into a structured report
tools: [Bash, Read]
---

Run the linter for this project. Detect the project type automatically:

- Python: run `flake8 . --count --statistics`
- JavaScript/TypeScript: run `npx eslint . --format compact`
- Go: run `golangci-lint run ./...`
- Rust: run `cargo clippy --no-deps`

If the linter is not installed, suggest installing it and show the install command.

Format the output as a table with columns: File, Line, Error Code, Description.
Group by severity if the linter supports it.
```

Create `.claude/commands/review.md`:

```markdown
---
name: review
description: Run lint then review remaining issues in the diff
tools: [Bash, Read, Grep]
---

Run the lint command first to catch style issues.
Then review the current diff ({{diff}}) for logic issues that the linter would not catch.

Focus on:
- Logic errors and edge cases
- Security vulnerabilities
- Performance bottlenecks
- API misuse

Do not repeat issues the linter already found. The output should have two sections:
1. Lint results (summary only)
2. Deep review findings (with file paths, line numbers, and explanations)
```

### Exercise I2: Settings-Driven Command with User Overrides

Create a command that reads behavior from `.claude/settings.json` and allows the user to override settings via `settings.local.json`.

**Task**: Create a `/deploy` command that reads a `deploy` configuration block from `settings.json`, but allows the user to override the target environment in `settings.local.json`.

**Hint**: The command should read both files, merge them (with `settings.local.json` taking precedence), and then execute the deploy steps based on the merged configuration.

**Solution**:

First, add a deploy configuration to `.claude/settings.json`:

```json
{
  "hooks": {},
  "deploy": {
    "default_environment": "staging",
    "build_command": "npm run build",
    "test_before_deploy": true,
    "environments": {
      "staging": {
        "url": "https://staging.example.com",
        "branch": "main"
      },
      "production": {
        "url": "https://example.com",
        "branch": "release"
      }
    }
  }
}
```

Then create `.claude/commands/deploy.md`:

```markdown
---
name: deploy
description: Deploy to the configured environment with pre-deploy checks
tools: [Bash, Read, Edit]
---

Read the deploy configuration from .claude/settings.json.
Then read .claude/settings.local.json to check for user overrides.

If settings.local.json has a deploy.default_environment key, use that
instead of the project default.

Current branch: {{branch_name}}

Determine the target environment:
1. If the user specified one (e.g., "production"), use that
2. Otherwise, use the default from configuration (staging)

Check that the current branch matches the configured branch for the target
environment. If not, warn the user and ask for confirmation.

If test_before_deploy is true, run the tests. If tests fail, abort the deploy.

Run the build command. If it succeeds, report the deploy URL and confirm
the target environment.
```

### Exercise I3: Command with Dynamic Model Selection

Create a command that chooses the model based on the task complexity. Simple tasks use a fast model, complex tasks use a capable model.

**Task**: Create a `/smart-review` command that checks the size of the diff. If the diff is small (under 50 lines), use `claude-sonnet-4-20250514`. If it is large, use `claude-opus-4-20250514`. The frontmatter should set no model, and the prompt body should instruct the model to make the decision.

**Hint**: Since the frontmatter `model` key is static, you cannot switch models dynamically from the prompt body. Instead, create two separate command files and have the prompt in a third dispatcher command decide which one to invoke.

**Solution**:

Create `.claude/commands/smart-review.md`:

```markdown
---
name: smart-review
description: Review the diff, selecting the model based on diff size
tools: [Bash, Read]
---

Count the number of lines in the current diff. Use `git diff | wc -l` to
get the count.

If the diff is 50 lines or fewer, use the forte model (Sonnet) by running
the /review-fast command. If the diff is larger than 50 lines, use the
opus model by running the /review-deep command.

Report which model was selected and why.
```

Create `.claude/commands/review-fast.md`:

```markdown
---
name: review-fast
description: Quick review for small diffs using the default model
model: claude-sonnet-4-20250514
tools: [Bash, Read, Grep]
---

Perform a concise code review of this diff:

{{diff}}

Focus on the most important issues only. Limit to 5 findings maximum.
Use CRITICAL and WARNING severity levels only -- skip suggestions.
```

Create `.claude/commands/review-deep.md`:

```markdown
---
name: review-deep
description: Thorough review for large diffs using the capable model
model: claude-opus-4-20250514
tools: [Bash, Read, Grep]
---

Perform a thorough code review of this large diff:

{{diff}}

Do not limit the number of findings. Use all three severity levels:
CRITICAL, WARNING, and SUGGESTION. For each finding, include the file
path, line number, and a detailed explanation of the issue and how to
fix it. End with a summary with counts per severity level.
```

### Exercise I4: Command with External API Integration

Create a command that calls an external API (via `WebFetch` or `curl`) and incorporates the response into the command output.

**Task**: Create a `/docs` command that takes a package name, fetches its documentation summary from a public API, and returns a formatted digest.

**Hint**: Use `WebFetch` in the `tools` list. The command can use `pypi.org` or `npmjs.com` JSON APIs to fetch package metadata.

**Solution**:

```markdown
---
name: docs
description: Fetch and summarize documentation for a package
tools: [WebFetch, Read]
---

Ask the user which package and ecosystem they want documentation for
(PyPI, npm, crates.io, etc.).

If {{selected_text}} is not empty, use it as the package name.

Fetch the package metadata from the appropriate registry API:
- PyPI: `https://pypi.org/pypi/{package_name}/json`
- npm: `https://registry.npmjs.org/{package_name}`
- crates.io: `https://crates.io/api/v1/crates/{package_name}`

Extract and display:
1. Package name and version
2. Description (one paragraph)
3. Dependencies (top 10)
4. Recent releases (last 3)
5. Homepage URL

If the package is not found, report the error and suggest similar package names.
```

## Advanced Tier

### Exercise A1: Stateful Command with Caching

Build a command that caches its results to avoid redundant computation. The cache should expire after a configurable timeout.

**Task**: Create a `/health` command that checks the health of all services in a `docker-compose.yml` file. If the health check was run within the last 60 seconds, return the cached result instead of re-running the checks.

**Hint**: The command can write a cache file to `.claude/health-cache.json` with a timestamp. Before running the health checks, read the cache file and check if the timestamp is within the TTL window. Use `Read` and `Write` tools for cache I/O.

**Solution**:

```markdown
---
name: health
description: Check service health with caching (60s TTL)
tools: [Bash, Read, Write]
---

Check the health of services defined in docker-compose.yml.

First, read the cache file at .claude/health-cache.json if it exists.
Parse the timestamp from the cache. If the cache was written within the
last 60 seconds, return the cached report and do not re-run checks.

If the cache is expired or missing:
1. Run `docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"`
2. Run `docker-compose ps --services` to list expected services
3. For each service running, check if the health endpoint responds
   (e.g., `curl -s -o /dev/null -w "%{http_code}" http://localhost:PORT/health`)
4. Write the report to .claude/health-cache.json with a timestamp

Format the report as a table:

| Service | Status | Port | Health Endpoint | HTTP Code |
|---------|--------|------|-----------------|-----------|
| api     | Up     | 8080 | /health         | 200       |

Include the cache timestamp in the report so the user knows when the
check was last run.
```

### Exercise A2: Multi-Repository Orchestrator

Build a command that operates across multiple repositories. This is useful for monorepo setups or when you need to run the same command across several projects.

**Task**: Create a `/cross-review` command that takes a list of repository paths, reads the current diff from each one, and produces a single combined review report.

**Hint**: The command should prompt you for the list of repository paths, or read them from a configuration file at `.claude/cross-review-config.json`. Use `Bash` to run `git diff` in each repository.

**Solution**:

Create `.claude/cross-review-config.json`:

```json
{
  "repositories": [
    "../frontend",
    "../backend",
    "../shared-lib"
  ],
  "max_diff_lines_per_repo": 200,
  "include_uncommitted": true
}
```

Create `.claude/commands/cross-review.md`:

```markdown
---
name: cross-review
description: Review changes across multiple repositories
tools: [Bash, Read, Grep]
---

Read the configuration from .claude/cross-review-config.json to get the
list of repository paths.

For each repository in the list:
1. Navigate to the repository directory
2. Run `git rev-parse --abbrev-ref HEAD` to get the current branch
3. Run `git diff --stat` to get a summary of changes
4. Run `git diff` to get the full diff (limit to max_diff_lines_per_repo lines)
5. If the diff exceeds the limit, truncate and note the truncation

Combine all diffs into a single review report. Organize by repository:

## Repository: ../frontend (branch: main)
Changes: 3 files modified
(Review findings...)

## Repository: ../backend (branch: feature/api)
Changes: 7 files modified
(Review findings...)

For each repository, include:
- Number of files changed
- Number of insertions and deletions
- Security-sensitive patterns found
- Potential integration issues between repositories

If a repository path does not exist or is not a git repo, skip it with
a warning.
```

### Exercise A3: Self-Modifying Command

Build a command that can update its own behavior based on project patterns. This is the most advanced pattern -- a command that learns from your project and adapts.

**Task**: Create a `/learn` command that scans the project's commit history, identifies common patterns in commit messages, and generates a commit message template that future commits can use.

**Hint**: The command should use `git log --oneline -50` to analyze recent commit messages, detect a pattern (e.g., `feat:`, `fix:`, `chore:` prefixes), and write a configuration file that another command reads. The command should be self-aware: it should not modify its own `.md` file, but it can write to a separate config file.

**Solution**:

```markdown
---
name: learn
description: Analyze project patterns and configure command behavior
tools: [Bash, Read, Write, Grep]
---

Analyze your project's conventions by examining:
1. Commit message patterns: run `git log --oneline -50`
2. File naming conventions: look at the top-level directory structure
3. Code style: check for `.editorconfig`, `.prettierrc`, `pyproject.toml`, `eslint.config.js`
4. Branch naming: check the last 10 branches with `git branch -l --sort=-committerdate | head -10`

From the commit messages, detect the most common prefix pattern:
- Conventional commits (feat:, fix:, chore:, docs:, refactor:)
- Simple prefixes (Add, Fix, Update, Remove)
- No consistent pattern

Write the detected conventions to .claude/project-patterns.json:

```json
{
  "commit_style": "conventional",
  "prefixes": ["feat", "fix", "chore", "docs", "refactor"],
  "branch_naming": "kebab-case",
  "detected_at": "{{timestamp}}"
}
```

After writing the file, summarize what was detected and suggest which
existing commands could be updated to use these conventions. Do not
modify any command files -- only report the suggestions.

If .claude/project-patterns.json already exists, compare the new
analysis with the old one and report what changed.
```

## Solutions

### B1 Solution

The `/status` command is shown in the exercise above. It uses `{{branch_name}}` and `{{diff}}` template variables combined with shell commands to produce a formatted status report.

### B2 Solution

The updated frontmatter with triggers is shown in the exercise above. The `triggers` key accepts a list of glob patterns that match file types the command is relevant for.

### B3 Solution

The `/explain` command is shown in the exercise above. Key detail: the `{{current_file}}` variable is empty when no file is active in the editor, so the command must handle that case.

### B4 Solution

The `/find` command is shown in the exercise above. Note that `Grep` is a separate tool from `Bash` -- you must include it explicitly in the `tools` list.

### I1 Solution

The two-command pipeline with `/lint` and `/review` is shown in the exercise above. The chaining is done through the prompt body of `/review`, which instructs the model to run `/lint` first.

### I2 Solution

The `/deploy` command with `settings.json` and `settings.local.json` merging is shown in the exercise above. The key concept is that `settings.local.json` overrides are user-specific and should not be committed to the repo.

### I3 Solution

The three-command setup (`/smart-review`, `/review-fast`, `/review-deep`) is shown in the exercise above. The dispatcher `/smart-review` uses `Bash` to count diff lines and then instructs the model accordingly.

### I4 Solution

The `/docs` command with external API integration is shown in the exercise above. The `WebFetch` tool is essential for this pattern.

## Reflection Questions

After completing the exercises, consider these questions:

1. How would your approach to command design change if you were building commands for a team of 20 developers instead of yourself?

2. What security review process would you recommend before accepting a custom slash command from a pull request submitted by a teammate?

3. How would you test a custom slash command that writes files? What guardrails would you put in place to prevent accidental data loss?

4. If a command needs to prompt the user for input mid-execution (for example, "which environment do you want to deploy to?"), how would you design the prompt body to handle this?

5. How would you version and migrate custom commands when Claude Code releases a breaking change to the frontmatter schema?
