You have been running Claude Code for two weeks and the `/help` menu shows exactly seven built-in commands. Every morning you type the same instruction: "run the linter, then run the tests, then report the coverage." You paste the same five-line system prompt before every code review session. You have aliased the whole thing in your shell as `alias review="claude -p 'review the diff'"` but that is a band-aid, not a solution. What you actually want is a single keystroke: `/review`. A command that knows exactly what you mean, sets up the right model and permissions, and runs your multi-step routine end to end. That is exactly what custom slash commands give you.

Custom slash commands turn your recurring workflows into first-class CLI commands. They live in your project, travel with your repo, and give you back the seconds that add up to hours every week.

The same pain extends beyond solo work. On a team of five developers, each person has their own review process memorized in their head. One person runs tests first, then lints, then reviews. Another reviews first, then lints, then tests. The output is inconsistent. The code review comments use different formats. The linting rules get applied differently. When you commit a custom slash command to the repo, you encode the team's agreed workflow in a file that everyone can discover and use with the same two keystrokes. The process becomes consistent, documented, and auditable without any ceremony beyond the `.md` file itself.

## The Concept

Custom slash commands are `.md` files in the `.claude/commands/` directory that Claude Code reads and registers as new entries in the slash command menu. Each file has YAML frontmatter that defines the command name, description, trigger conditions, model preference, permitted tools, and the prompt body that executes when you run the command. Think of them as executable prompt templates -- like shell aliases for AI, but with structured metadata, permission scoping, and template variable substitution.

## CLI Reference

The following table covers every command, flag, and file element you need to write and manage custom slash commands in Claude Code v2.x.

| Element / Flag | What it does | Example |
|---|---|---|
| `.claude/commands/<name>.md` | Create a custom slash command file | `.claude/commands/review.md` |
| `--- name` frontmatter key | Display name shown in the `/help` menu | `name: review` |
| `--- description` frontmatter key | One-line summary in `/help` | `description: Run linter, tests, and coverage` |
| `--- triggers` frontmatter key | File patterns that auto-suggest this command | `triggers: ["*.py", "*.js"]` |
| `--- model` frontmatter key | Override the model for this command | `model: claude-opus-4-20250514` |
| `--- tools` frontmatter key | Category-level tool permission whitelist | `tools: [Bash, Read, Edit, Write]` |
| `--- allowedTools` frontmatter key | Explicit tool allowlist (supersedes tools) | `allowedTools: [Bash, Read]` |
| `--- permissionLevel` frontmatter | Permission prompt behavior: allow-all, restricted, or default | `permissionLevel: "allow-all"` |
| `{{branch_name}}` template variable | Injected from git branch state | `You are on branch {{branch_name}}` |
| `{{current_file}}` template variable | Injected from session editor context | `Review {{current_file}}` |
| `{{diff}}` template variable | Injected with the working tree diff | `Analyze this diff: {{diff}}` |
| `{{clipboard}}` template variable | Injected with system clipboard content | `Explain: {{clipboard}}` |
| `{{selected_text}}` template variable | Injected with selected terminal text | `Fix this: {{selected_text}}` |
| `{{project_root}}` template variable | Resolved to the project root absolute path | `/home/user/project` |
| `{{timestamp}}` template variable | Current local date and time | `2026-07-04T14:30:00` |
| `claude --config` | Open project-level settings.json for editing | `claude --config` |
| `claude --verbose` | Show full tool-by-tool execution details | `claude --verbose -p "/review"` |
| `settings.json hooks` | JSON configuration for automated trigger behaviors | See hooks section below |
| `settings.local.json` | User-local overrides for project settings | `.claude/settings.local.json` |
| `CLAUDE_CODE_DISABLE_HOOKS` | Environment variable to disable all hooks | `export CLAUDE_CODE_DISABLE_HOOKS=true` |

## How It Actually Works -- Step by Step

### Step 1: File Discovery

When you type `/` in a Claude Code session, the tool reads the contents of `.claude/commands/` in your project root. Every `.md` file in that directory becomes a candidate. Claude Code builds an in-memory index mapping each file's frontmatter `name` field to its path. If no `name` is specified in the frontmatter, the filename (minus the `.md` extension) is used as the command name. Files in subdirectories are not registered -- only the flat list of `.md` files directly inside `.claude/commands/` is scanned.

The discovery process checks two locations in order. First, it looks in the project-level `.claude/commands/` directory. Then it checks the global directory at `~/.claude/commands/`. If the same command name exists in both locations, the project-level version takes precedence. This means you can override a global command with a project-specific version by placing a file with the same name in the project directory.

File system permissions also play a role. If the `.claude/commands/` directory exists but is not readable, or if individual files have restrictive permissions, those commands are silently skipped -- no error is shown. Always verify that your command file is world-readable (644 permissions) so Claude Code can access it.

The index is refreshed each time you open the command menu. You do not need to restart the Claude Code session to pick up new commands. Simply create or modify the `.md` file, then type `/` in the session, and the new command appears.

### Step 2: Frontmatter Parsing

The YAML frontmatter is the command's configuration manifest. It must be the first thing in the file, delimited by `---` on its own line before and after. Each frontmatter key controls a specific dimension of behavior. Here is a complete reference of every supported key:

| Frontmatter Key | Required | Type | Default | Description |
|---|---|---|---|---|
| `name` | No | String | Filename stem | Display name shown in `/help` and used for matching |
| `description` | No | String | Empty | One-line summary shown in `/help` beside the name |
| `triggers` | No | Array of strings | None | Glob patterns that auto-suggest this command |
| `model` | No | String | Session default | Override the model for this specific command |
| `tools` | No | Array of strings | Session defaults | Broad category-level tool permission whitelist |
| `allowedTools` | No | Array of strings | tools value | Explicit tool allowlist that supersedes tools |
| `permissionLevel` | No | String | "default" | Controls permission prompt granularity |

Once Claude Code locates the file, it parses the YAML frontmatter block between the `---` delimiters. The frontmatter is the command's configuration manifest. Each key controls a specific dimension of behavior:

- **name**: The command identifier displayed in `/help` and used for matching. Must be unique within `.claude/commands/`. If two files have the same `name`, the behavior is undefined -- one will silently shadow the other. Always use unique names.
- **description**: Shown in the `/help` menu and used for tab-completion hints. Keep it under 80 characters. This is the first thing your teammates see when they run `/help`, so make it descriptive enough to distinguish the command from others. "Run linter" is too vague. "Run flake8 linter on staged Python files and report results" is better.
- **triggers**: A list of glob patterns. When you select text or a file matching these patterns, Claude Code suggests the command automatically. For example, `triggers: ["*.py", "*.js"]` makes the command appear when you are looking at Python or JavaScript files. Triggers do not run the command automatically -- they only surface it in the suggestion menu. Use hooks for automatic execution.
- **model**: Override the session model for this specific command. Useful when a task needs a stronger model (Opus for code review) or a cheaper one (Sonnet for simple linting). The model must be one that Claude Code supports -- check `claude --help` for the current list. If you specify a model name that does not exist, the command falls back to the session default.
- **tools**: The tool categories this command is allowed to use. Possible values include `Bash`, `Read`, `Edit`, `Write`, `Glob`, `Grep`, `WebSearch`, `WebFetch`, `Skill`, and others. If omitted, the command inherits the session's current tool permissions. This is the most common source of runtime errors -- commands that try to use a tool not in the allowlist will fail with a permission error.
- **allowedTools**: A more explicit allowlist that supersedes the broad `tools` key when both are present. Use this for fine-grained control when a command should only be able to read files, never edit them. The intent of `allowedTools` is to tighten security beyond what `tools` provides. A command with `tools: [Bash, Read, Edit, Write]` and `allowedTools: [Bash, Read]` effectively has only `Bash` and `Read` permissions.
- **permissionLevel**: Controls how aggressively Claude Code prompts for permission. Use `"allow-all"` for fully trusted commands that should run without confirmation on every tool call, or omit it to prompt per-action which is the default. The `"restricted"` value adds extra prompts even for typically safe operations -- use this when running commands from untrusted sources.

### Step 3: Template Variable Substitution

After parsing the frontmatter, Claude Code processes the prompt body below the frontmatter for template variables. These are placeholders written as `{{variable_name}}` that get replaced with runtime values before the prompt is sent to the model.

Available template variables:

| Variable | Source | Example output |
|---|---|---|
| `{{branch_name}}` | `git rev-parse --abbrev-ref HEAD` | `feature/add-login` |
| `{{current_file}}` | Active file in editor context | `src/auth/login.ts` |
| `{{diff}}` | `git diff` output (staged + unstaged) | Unified diff text |
| `{{clipboard}}` | System clipboard content | Any text on your clipboard |
| `{{selected_text}}` | Text selected in the terminal | The highlighted text |
| `{{project_root}}` | Absolute path to the project root | `/home/user/project` |
| `{{timestamp}}` | Current local date and time | `2026-07-04T14:30:00` |

If a variable cannot be resolved (for example, `{{diff}}` when there is no git repo), the variable is left as-is in the prompt or replaced with an empty string depending on the variable. Always design your prompts to handle empty variable substitutions gracefully.

### Step 4: Permission Resolution

Before executing any tools referenced in the command prompt, Claude Code checks the `tools` and `allowedTools` frontmatter keys against the session's current permission configuration. If a `permissionLevel` of `"allow-all"` is set and the session trusts the command source (the command is in the project's own `.claude/commands/` directory), tool calls proceed without prompting. Otherwise, each tool invocation triggers the standard permission prompt.

The permission resolution follows a specific priority chain. First, `allowedTools` is checked -- this is the most specific and restrictive list. If `allowedTools` is present, only tools in that list are permitted regardless of what `tools` says. If `allowedTools` is absent, the broader `tools` list is used. If both are absent, the command inherits the session's current tool permissions, which may have been set by the `--allowedTools` launch flag or earlier commands.

The permission model also distinguishes between project-level commands and global commands. Commands in the project's `.claude/commands/` directory inherit the project's permission scopes. Commands in the global `.claude/commands/` directory (inside your home directory) use global settings. This separation means you can have strict permissions for work projects and looser permissions for personal projects without changing any configuration.

This is the layer where safety matters most. A custom command that says "write to every file in the repo" could be dangerous if you run it without reviewing what it does. Always inspect the frontmatter of a command you did not write yourself before running it. The frontmatter is the safety contract -- it declares what the command intends to do. If the prompt body contradicts the frontmatter, one of them is lying, and you should not run the command until you understand why.

### Step 5: Prompt Execution

The resolved prompt is sent to the model specified in the frontmatter (or the session default if none is specified). The model receives the prompt exactly as written, with all template variables expanded. It then uses the permitted tools to carry out the task. This is no different from typing the same instructions manually -- except the instructions are consistent, repeatable, and version-controlled.

The execution phase is where the model interprets your instructions and decides which tools to call, in what order, and with what arguments. The model sees the same context it would see in a regular session: the current file tree, the git state, and the conversation history. The only difference is the initial prompt came from a file instead of being typed by you.

During execution, the model may call tools multiple times, read files, run commands, and iterate on the output. The full session history is preserved, so if the command needs to loop or refine its approach, it can. The `--verbose` flag is particularly useful during development to see exactly which tools the model calls and what arguments it passes.

The execution phase is where the quality of your prompt body matters most. A well-written prompt body produces reliable, consistent results. A poorly-written one produces variable output that may miss the mark. Here are the characteristics of effective prompt bodies:

- **Specific instructions**: Instead of "review the code," say "identify security vulnerabilities, type mismatches, and performance bottlenecks in the diff."
- **Output format specification**: Tell the model exactly how to format its output. "Return a markdown table with columns: File, Line, Severity, Description" produces consistent output every time.
- **Edge case handling**: Include instructions for what to do when common conditions are not met. "If the diff is empty, report that there are no changes to review."
- **Self-termination**: Include a stop condition. "After producing the report, stop and present the results to the user." Without this, the model may continue analyzing beyond what is needed.

The model also has access to the same session context it would have in a regular conversation. This means the command can reference files the model has already read, variables it has already computed, and decisions it has already made. The context is not reset between command invocations within the same session.

### Step 6: Chaining Commands

Commands can be chained by writing a prompt that triggers another command as part of its execution. For example, a `/full-review` command might start by running `/lint` to check code style, then analyze the remaining issues with the full model. Chaining is done through the prompt body itself -- you describe the sequence of steps and the model executes them in order.

There are two approaches to chaining. The first is explicit chaining: the prompt body of command A says "First run the /lint command, then review the remaining issues." The model interprets this as a multi-step instruction and executes each step sequentially. The second is implicit chaining: a hook in `settings.json` is configured to run command A, which then invokes command B as part of its prompt. The hook acts as the trigger, and the chaining happens inside the command.

For more complex automation, combine custom slash commands with `settings.json` hooks. Hooks let you run commands automatically before or after certain events, such as before a commit, when Claude Code starts, or when a file matching a specific pattern changes. The hooks system is the bridge between manual command invocation and fully automated workflows.

### Step 7: Sharing Commands via Version Control

The final step in the lifecycle is sharing. Since custom command files are plain `.md` files in your project directory, you commit them to the repository just like any other source file. When a teammate clones the repo and opens it in Claude Code, the commands are automatically available -- no install step, no configuration, no documentation to read.

Sharing via version control has implications for command design. A command that works on your machine with your specific file paths will break on a teammate's machine. Use template variables (`{{project_root}}`, `{{branch_name}}`) instead of hardcoded paths. Use environment variables for API keys instead of embedding them in the command. Write the `description` field clearly because that is what appears in `/help` on your teammate's terminal.

If you want to share a command without committing it to the repo (for example, a personal workflow that is not relevant to the team), place it in your global `.claude/commands/` directory instead. Global commands are available in every project but are not visible to anyone else who clones the repo.

## Hooks and Settings Integration

Custom slash commands become dramatically more powerful when combined with the Claude Code hooks system. Hooks are JSON configurations in `.claude/settings.json` that trigger commands automatically on specific events.

### Supported Hook Events

| Hook Event | Trigger | Use Case |
|---|---|---|
| `pre-commit` | Fires before `git commit` completes | Run linting, review, or tests before allowing a commit |
| `post-commit` | Fires after `git commit` completes | Update changelog, notify team, tag the release |
| `on-start` | Fires when Claude Code session begins | Load project context, check for updates, verify env vars |
| `on-file-change` | Fires when a file matching a pattern changes | Auto-lint on save, auto-format on edit |
| `pre-execute` | Fires before every tool execution | Log tool usage, enforce safety checks |

### Hook Configuration Structure

Each hook entry in `settings.json` specifies the command to run and optional metadata:

```json
{
  "hooks": {
    "pre-commit": [
      {
        "command": "review",
        "description": "Run code review before commit"
      }
    ],
    "on-start": [
      {
        "command": "status",
        "description": "Show project status on session start"
      }
    ],
    "on-file-change": [
      {
        "command": "lint",
        "description": "Auto-lint on file save",
        "pattern": "*.py"
      }
    ]
  }
}
```

The `description` field in a hook entry is shown to you when the hook fires, so you know why the command is running. The optional `pattern` field in `on-file-change` hooks restricts which file changes trigger the command.

### Settings.local.json Overrides

User-local settings live in `.claude/settings.local.json` and follow the same structure as `settings.json`. Any key present in `settings.local.json` takes precedence over the same key in `settings.json`. This is useful for:

- **Disabling hooks locally**: A teammate who does not want the pre-commit review hook can override it in their local settings without affecting the team's configuration.
- **Adding personal hooks**: A developer can add personal hooks that only fire in their session.
- **Overriding tool permissions**: A user can tighten their local tool permissions without affecting the team's command files.

Example `.claude/settings.local.json` that disables the pre-commit hook:

```json
{
  "hooks": {
    "pre-commit": []
  }
}
```

Note that `settings.local.json` should be added to `.gitignore` so each team member's local preferences stay personal.

## Cost, Limits, and Safety

### Cost Implications

Every custom slash command consumes tokens on every invocation. The cost depends on three factors:

1. **Model selection**: Using `model: claude-opus-4-20250514` costs roughly 3-5x more per token than `model: claude-sonnet-4-20250514`. Always pick the cheapest model that can complete the task reliably.
2. **Prompt length**: Template variables like `{{diff}}` can inject thousands of tokens. A large diff from a monorepo could exhaust your context window before the model even starts reasoning.
3. **Tool execution tokens**: Every tool call (Bash, Read, Write) consumes additional tokens for the tool result. A command that reads 20 files and writes 5 more will use significantly more tokens than one that reads 3 files.

Estimate your per-command cost before running it in a loop or hook. A command that costs $0.05 per run and fires on every file save could burn through a substantial budget in a day. To estimate cost, run the command once with `--verbose` and note the token usage reported in the output. Multiply by your expected daily or weekly run count to get a rough budget.

Token usage also varies by the size of template variable substitution. The `{{diff}}` variable on a branch with 50 changed files could inject 10,000 tokens or more. The `{{clipboard}}` variable could inject megabytes of text if you have a large block of code copied. Consider adding a line in the prompt body that instructs the model to truncate large inputs: "If the diff exceeds 2000 lines, summarize the types of changes rather than reviewing every line."

### Rate Limits and Guardrails

Custom slash commands are subject to the same Anthropic API rate limits as any other Claude Code session. If you set a command to loop (via a cron-like hook in settings.json), you can hit rate limits quickly. The following environment variables help you control this:

| Variable | Purpose |
|---|---|
| `ANTHROPIC_API_KEY` | API key for authentication |
| `ANTHROPIC_AUTH_TOKEN` | Alternative auth token |
| `CLAUDE_CODE_ALLOWED_TOOLS` | Default tool allowlist for the session |
| `CLAUDE_CODE_DISABLE_HOOKS` | Set to `"true"` to disable all hooks temporarily |
| `CLAUDE_CODE_MAX_TOKENS` | Maximum tokens per response |

### How to Avoid Runaway Loops

A hook-triggered command that writes a file which then triggers another hook could create an infinite loop. Follow these safety practices:

- **Never write to `.claude/commands/` or `.claude/settings.json` from a command body.** This can create a feedback loop where a command modifies its own configuration.
- **Set `tools` to the minimum set required.** A command that only reads files should not have `Write` or `Edit` in its `tools` list.
- **Use `allowedTools` for explicit denials.** If a command should never call `Bash`, list everything except `Bash` in `allowedTools`.
- **Test new commands with a dry run first.** Run the command in a clean directory or with `--verbose` so you can see every tool call before it executes.
- **Set `CLAUDE_CODE_DISABLE_HOOKS=true` during development** so your half-written command does not accidentally fire on real project files.
- **Add a run counter or timeout check** in the prompt body. For example: "If this command has already run 5 times in the last minute, stop and report an error." This prevents runaway loops even if the hook fires repeatedly.
- **Use a sentinel file** to detect re-entrant calls. The command can write a `.claude/.running` file at the start and delete it at the end. If the file already exists when the command starts, abort immediately -- another instance is already running.

### Permission Scoping

Commands defined in `.claude/commands/` within your project inherit the project's permission scopes. Commands in the global `.claude/` directory (your home directory) use global settings. You can control this with `permissionLevel` in the frontmatter:

- **Omitted (default)**: Standard permission prompts per tool invocation. Each time the command calls a tool, you see a prompt asking for approval. This is the safest mode for commands you do not fully trust.
- **`"allow-all"`**: No prompts for allowed tools within this command. Use this only for commands you have read and verified. A command with `allow-all` permission can run any tool in its `tools` list without asking you again.
- **`"restricted"`**: Extra prompts even for typically safe operations. Use this when running commands from untrusted sources or when you want to audit every single action the command takes.

The permission system operates on a principle of least privilege. A command should only have the permissions it needs to do its job and no more. A `/status` command that only runs `git status` needs only `Bash` permission, not `Edit` or `Write`. A `/review` command that inspects files needs `Read` and `Bash`, but not `Write`.

When a command's `tools` list conflicts with the session-level `--allowedTools` flag, the more restrictive setting wins. If you launch Claude Code with `--allowedTools "Read, Grep"` and a command requests `tools: [Bash, Edit, Write]`, the command will only have `Read` and `Grep` available. The session-level restriction acts as a safety net that cannot be overridden by any command.

This layered permission model protects you from accidents. Even if a command has `allow-all` set in its frontmatter, if your session was started with restrictive global tool permissions, the command cannot escalate beyond those boundaries.

### Environment Variables for Safety

The following environment variables act as safety controls for custom slash commands:

| Variable | Purpose | Example |
|---|---|---|
| `CLAUDE_CODE_DISABLE_HOOKS` | Disable all hooks temporarily | `export CLAUDE_CODE_DISABLE_HOOKS=true` |
| `CLAUDE_CODE_ALLOWED_TOOLS` | Global tool allowlist for all commands | `export CLAUDE_CODE_ALLOWED_TOOLS="Read,Grep"` |
| `CLAUDE_CODE_MAX_TOKENS` | Cap token usage per session | `export CLAUDE_CODE_MAX_TOKENS=32000` |
| `CLAUDE_CODE_LOG_LEVEL` | Set logging verbosity for debugging | `export CLAUDE_CODE_LOG_LEVEL=debug` |

Setting `CLAUDE_CODE_DISABLE_HOOKS=true` is the emergency brake. If a hook-triggered command enters an infinite loop or behaves unexpectedly, setting this variable stops all hooks immediately without needing to edit any configuration files.

## When to Use This / When Not To

Custom slash commands are a force multiplier for workflows you repeat more than three times. They eliminate the cognitive overhead of remembering the exact prompt, the exact tool permissions, and the exact model configuration for a recurring task. If you find yourself copy-pasting the same instruction block from a notes file, that is a candidate for a custom command.

**Use custom slash commands when:**

- You have a multi-step review or build workflow you run multiple times per day
- You want to enforce a consistent prompt across a team by committing the command file to the repo
- You need a command that always uses a specific model (Opus for code review, Sonnet for linting)
- You want to configure a hook that fires automatically on session start, before commit, or on file change
- You have tool permission requirements that differ from your session defaults (a read-only command should never have write access)

**Do not use custom slash commands when:**

- The task is a one-off. Writing a command file for something you will do once is wasted effort.
- The workflow depends on information you cannot capture in template variables. If the prompt needs human judgment at every step, a command cannot replace that interaction.
- The command would require dangerous permissions on untrusted code. Never run a custom command from a third-party repository without reading its frontmatter and prompt body first.
- The sequence is better expressed as a shell script. Custom slash commands are for AI-guided workflows, not pure automation. If your workflow has no AI reasoning component -- just running three shell commands in sequence -- a Makefile or shell alias is simpler and faster.
- The command environment is unstable. If your project has unreliable network access, frequently changes directory structure, or has inconsistent dependencies, a custom command will fail in confusing ways. The model will try to diagnose the failure, which costs tokens and time, when a simple shell script would just print the error and exit.
- The command exposes sensitive information. A command that reads environment variables and prints them as part of a report should be designed carefully. The prompt body should not echo API keys or secrets back to the output.

The position to take: custom slash commands excel at workflows where the AI's reasoning adds value at each step. The linter output needs interpretation. The test failures need diagnosis. The diff needs contextual review. If you are just running commands without interpretation, you do not need an AI -- use a shell script. The boundary between automation and AI-guided workflow is the decision point: if every step is deterministic (run this, then run that, then run that), shell scripting wins. If any step involves interpretation (check the output, decide what to do next, adjust parameters), an AI command wins.

## Designing Effective Commands

Writing a custom slash command that delivers consistent, reliable results requires more than copying a template. The quality of the prompt body determines the quality of the output. Here are design principles drawn from real-world usage across teams.

### Start with the Output in Mind

The most effective commands specify the output format before the analysis steps. When the model knows exactly what to produce, it shapes its reasoning to fit that format. A command that says "return a markdown table with columns" produces more consistent results than one that says "report the findings." The output format acts as a constraint that focuses the model on the right dimensions of the problem.

### Handle the Empty Case

Every template variable can be empty. `{{diff}}` is empty when there is no diff. `{{current_file}}` is empty when no file is selected. `{{selected_text}}` is empty when nothing is highlighted. A robust command checks for these conditions explicitly: "If the diff is empty, report that there are no changes to review and suggest running the command after making edits." Without this check, the model may hallucinate content or produce confusing output.

### One Command, One Responsibility

A command that reviews code, deploys to production, and sends a Slack notification is three commands stacked in a trench coat. Split it into `/review`, `/deploy`, and `/notify`. Then chain them with hooks. Single-responsibility commands are easier to test, easier to share, and easier to reason about. They also fail more gracefully -- if the deploy step fails, the review results are not lost.

### Test with Verbose Mode

Before relying on a command in your daily workflow, run it with `--verbose` at least once. This shows every tool call the model makes, every argument it passes, and every intermediate result. You will see whether the command reads unnecessary files, calls tools in an unexpected order, or attempts actions you did not intend. One verbose run often reveals issues that would otherwise surface as confusing behavior weeks later.

### Version Your Commands

When you update a command file, commit the change with a clear message describing what changed. If a teammate was relying on a specific behavior, they need to know when it changed. The commit history of `.claude/commands/` files serves as a changelog for your team's shared automation.

## Mental Model Check

### Question 1

Where must custom slash command files be located, and what naming convention do they follow?

<details>
<summary>Answer</summary>

They must be placed directly inside the `.claude/commands/` directory (not in subdirectories) as `.md` files. The filename minus the `.md` extension becomes the command name unless the frontmatter specifies a `name` key. For example, `.claude/commands/review.md` registers as `/review`.
</details>

### Question 2

What happens when a template variable like `{{diff}}` cannot be resolved because the project is not a git repository?

<details>
<summary>Answer</summary>

The `{{diff}}` variable resolves to an empty string since there is no git diff to compute. Other variables like `{{branch_name}}` also resolve to empty strings. Your command prompt should handle this gracefully -- for example, by saying "review the following changes" and then showing nothing, rather than crashing or producing an error. Design your prompts with conditional phrasing like "If there is a diff, review it; otherwise, summarize the current state of the project."
</details>

### Question 3

What is the difference between the `tools` and `allowedTools` frontmatter keys?

<details>
<summary>Answer</summary>

`tools` is a broad category-based allowlist (for example, `tools: [Bash, Read, Edit]`) that grants access to entire tool categories. `allowedTools` is a more explicit, granular allowlist that supersedes `tools` when both are present. If a tool appears in `tools` but not in `allowedTools`, the more restrictive list wins. Use `tools` for simple allowlisting and `allowedTools` when you need fine-grained control, such as allowing `Read` but explicitly denying `Edit` on a command that should only inspect files.
</details>

### Question 4

Describe a scenario where setting `permissionLevel: allow-all` would be appropriate, and one where it would be dangerous.

<details>
<summary>Answer</summary>

**Appropriate**: A `/status` command that only runs `git status`, `git log --oneline -5`, and reads a few config files. These operations are read-only and have no destructive potential. Approving each tool call individually would be tedious friction with no safety benefit.

**Dangerous**: A command from a third-party repository that you have not read the full prompt body of. Even if the frontmatter claims only to read files, the prompt body could instruct the model to overwrite configuration files, delete branches, or exfiltrate data. Always inspect the full file before running a command with `allow-all` permissions.
</details>

### Question 5

How can you share a custom slash command with your team, and what are the implications of doing so?

<details>
<summary>Answer</summary>

You commit the `.md` file to the repository, alongside your project source code. Any teammate who clones the repo and uses Claude Code in that directory will see the command in their `/help` menu automatically -- no additional setup required.

The implications are that you now own the command as a piece of shared infrastructure. Changes to the command affect everyone on the team. You should version the command thoughtfully, add a `description` that clearly explains what it does, and avoid hardcoding paths or values specific to your machine. Use template variables for environment-specific values. If the command needs external API keys, document which environment variables must be set and add a check in the prompt body that verifies the key is present before proceeding.
</details>
