## Practice Exercise: Build a Custom Code Review Command

In this exercise you will build a custom slash command called `/review` that performs a structured code review on the current diff. It will read the working tree changes, analyze them against configurable rules, and produce a formatted report with severity ratings.

By the end of this exercise, your command will:

- Automatically detect the current branch and diff from git state
- Apply configurable review rules from a project configuration file
- Produce a markdown report grouped by severity (critical, warning, suggestion)
- Respect the `Bash`, `Read`, and `Grep` tools only -- no write access
- Support a `--verbose` flag for detailed in-context reasoning

### Setup Requirements

Create a new directory for your command files. You will work inside the project you want to add the command to.

```bash
mkdir -p .claude/commands
```

Verify the directory was created:

```bash
ls -la .claude/commands/
```

Expected output (a new empty directory):

```
drwxr-xr-x  2 user user   64 Jul  4 10:00 .
drwxr-xr-x  3 user user   96 Jul  4 10:00 ..
```

You also need a project that has git initialized and at least one commit with uncommitted changes. If your current project has no changes, make a trivial edit to any file:

```bash
echo "# test change" >> README.md
```

If you are starting from scratch, initialize a test project:

```bash
mkdir test-slash-commands && cd test-slash-commands
git init
echo "# Test Project" > README.md
git add README.md && git commit -m "Initial commit"
echo "print('hello')" > hello.py
```

This gives you a clean git repo with one commit and one uncommitted file -- the perfect test environment for a custom slash command that uses `{{diff}}` and `{{branch_name}}`.

### Step 1: Create the Command File

Create a new file at `.claude/commands/review.md` with the following frontmatter and an initial prompt body.

```markdown
---
name: review
description: Run a structured code review on the current diff
triggers: ["*.py", "*.js", "*.ts", "*.java", "*.go", "*.rs"]
model: claude-sonnet-4-20250514
tools: [Bash, Read, Grep]
permissionLevel: allow-all
---

You are performing a code review on the changes in this project.

Current branch: {{branch_name}}

Review the following diff:

{{diff}}

Follow these rules:
1. Identify all changes in the diff
2. Check for common issues: syntax errors, type mismatches, security vulnerabilities, performance problems, and style violations
3. Group findings by severity: CRITICAL, WARNING, SUGGESTION
4. For each finding, include the file path, line number, and a specific explanation
5. End with a summary that counts findings by severity

If {{diff}} is empty, report that there are no changes to review.
```

Save this file and verify it is in the correct location:

```bash
ls -la .claude/commands/review.md
```

You should see the file listed with a size of approximately 600-800 bytes.

### Step 2: Verify the Command Is Registered

Launch Claude Code in your project directory and check that the command appears in the help menu.

```bash
claude -p "list all available slash commands" --allowedTools "Bash"
```

Alternatively, launch interactively and type `/help`:

```bash
claude
# Then type: /help
```

Look for `review` in the output. If you see it, the command is registered. If not, check that the file is in the correct directory and that the frontmatter is valid YAML.

### Step 3: Add a Configuration File

Create a configuration file that the command can read to customize the review rules. This shows how a custom command can reference project-level configuration.

Create `.claude/review-config.json`:

```json
{
  "severity_levels": ["CRITICAL", "WARNING", "SUGGESTION"],
  "checks": {
    "security": {
      "enabled": true,
      "patterns": [
        "hardcoded.*key",
        "password.*=",
        "api_key.*=",
        "exec\\(",
        "eval\\("
      ]
    },
    "performance": {
      "enabled": true,
      "patterns": [
        "for.*in.*range\\(len\\(",
        "\.append\\(.*for",
        "O\\(n\\^2\\)"
      ]
    },
    "style": {
      "enabled": true,
      "max_line_length": 100,
      "check_import_order": true
    }
  },
  "output_format": "markdown"
}
```

Now update the command prompt to load and use the configuration. Replace the content of `.claude/commands/review.md` with:

```markdown
---
name: review
description: Run a structured code review on the current diff
triggers: ["*.py", "*.js", "*.ts", "*.java", "*.go", "*.rs"]
model: claude-sonnet-4-20250514
tools: [Bash, Read, Grep]
---

You are performing a code review on the changes in this project.

Current branch: {{branch_name}}

First, read the configuration file at .claude/review-config.json to get the review rules and severity levels.

Then, review the following diff:

{{diff}}

Follow the rules from the configuration file. For each check category that is enabled, scan the diff for matching patterns. Group findings by severity as defined in the config. Output the report in the format specified in the config file.

If {{diff}} is empty, report that there are no changes to review.

After the review, suggest one improvement to the review configuration that would catch more issues in this project.
```

### Step 4: Add a Help Subcommand

Extend the command to support an optional `--help` flag by checking the first word of the prompt. This is a common pattern for making commands self-documenting.

Create `.claude/commands/review-help.md`:

```markdown
---
name: review-help
description: Show detailed usage information for the /review command
model: claude-sonnet-4-20250514
tools: [Read]
---

Show the user the documentation for the /review command.

The /review command performs a structured code review on the current diff.

Usage:
  /review              Run a full review on the current diff
  /review --verbose    Show detailed reasoning for each finding
  /review --help       Show this help message

Configuration:
  .claude/review-config.json  Controls which checks run and output format

Template variables used:
  {{branch_name}}  Current git branch
  {{diff}}         Current working tree diff

Required permissions:
  Bash, Read, Grep  (no write access)

To customize the review, edit .claude/review-config.json and adjust the
checks, severity levels, or output format.
```

### Step 5: Test the Command with a Dry Run

Before running the command on real code, test it with a dry run to see what it would do. Use `--verbose` to see every tool call the command makes.

```bash
claude -p "run the /review command on the current diff" --verbose --allowedTools "Bash, Read, Grep"
```

This runs the command in non-interactive mode with verbose output. You will see each step the model takes: reading the config file, examining the diff, and producing the report.

If you see errors, go back and check that the file paths and frontmatter are correct.

### Step 6: Set Up a Hook for Automatic Review on Commit

This step shows how to combine a custom slash command with a `settings.json` hook. The hook runs the review command automatically before every commit.

Create or edit `.claude/settings.json`:

```json
{
  "hooks": {
    "pre-commit": [
      {
        "command": "review",
        "description": "Run code review before commit"
      }
    ]
  }
}
```

Now, every time you run `git commit`, Claude Code will automatically trigger the `/review` command before the commit completes. If the review finds critical issues, you can cancel the commit and fix them.

To temporarily disable all hooks without deleting the configuration:

```bash
export CLAUDE_CODE_DISABLE_HOOKS=true
```

Set it back to run hooks again:

```bash
unset CLAUDE_CODE_DISABLE_HOOKS
```

### Step 7: Add the Command to Version Control

The final step is to commit your custom command files so the entire team can use them.

```bash
git add .claude/commands/review.md
git add .claude/commands/review-help.md
git add .claude/review-config.json
git add .claude/settings.json
```

Do not commit `.claude/settings.local.json` -- that file is for user-local overrides and should be in `.gitignore`.

```bash
echo ".claude/settings.local.json" >> .gitignore
git add .gitignore
git commit -m "Add /review custom slash command with pre-commit hook"
```

Now any teammate who clones the repo and runs Claude Code will have the `/review` command available automatically.

### Step 8: Verify the Full Flow

Run the command one final time on a real change to confirm everything works end to end.

```bash
# Make a change
echo "def hello(): print('hello world')" >> test.py

# Run the review command
claude -p "run /review" --allowedTools "Bash, Read, Grep"
```

You should see output that includes:

- The current branch name
- The diff of the change you just made
- A categorized report with severity levels
- At least one finding (the new file or the change itself)
- A summary count

If the command runs without errors and produces a useful review, your custom slash command is working correctly.

## Common Errors at This Stage

### Error 1: Command not found in /help menu

```
No matching command found. Type /help to see available commands.
```

**Cause**: The `.md` file is not in the correct directory, or the frontmatter is invalid YAML.

**Fix**: Verify the file is at `.claude/commands/<name>.md` -- not in a subdirectory, not in the wrong location. Check the frontmatter has no trailing spaces, no tabs mixed with spaces, and all keys are lowercase. Run `claude --version` to confirm you are on a version that supports custom commands (v2.x or later).

### Error 2: Permission denied for tool

```
Tool "Bash" is not allowed by this command's frontmatter.
```

**Cause**: The `tools` or `allowedTools` list in the frontmatter does not include the tool the command is trying to use.

**Fix**: Add the missing tool to the `tools` list in the frontmatter. If you intentionally want to block it, adjust the command prompt to avoid using that tool. Remember that `allowedTools` supersedes `tools` -- if both are present, the intersection is enforced.

### Error 3: Template variable not substituted

```
The prompt contains {{diff}} as literal text instead of the actual diff.
```

**Cause**: The template variable name is misspelled, or the variable is not supported in the current context.

**Fix**: Check the exact spelling of the variable. Valid variables are `{{branch_name}}`, `{{current_file}}`, `{{diff}}`, `{{clipboard}}`, `{{selected_text}}`, `{{project_root}}`, and `{{timestamp}}`. Note that `{{diff}}` only works in a git repository with uncommitted changes. If you are in a directory without a git repo, the variable resolves to an empty string.

### Error 4: Hook not firing on commit

```
git commit runs but the /review command does not execute.
```

**Cause**: The hook configuration in `settings.json` has a syntax error, or `CLAUDE_CODE_DISABLE_HOOKS` is set to `true`.

**Fix**: Validate the JSON in `.claude/settings.json` -- a trailing comma or missing bracket will silently disable the entire hooks block. Check that `CLAUDE_CODE_DISABLE_HOOKS` is not set by running `echo $CLAUDE_CODE_DISABLE_HOOKS`. If it returns `true`, unset it with `unset CLAUDE_CODE_DISABLE_HOOKS`.

### Error 5: Command runs but uses the wrong model

```
The command executes but seems less capable than expected (responses are shallow).
```

**Cause**: The `model` frontmatter key is misspelled, set to the wrong model, or the session default is overriding it.

**Fix**: Check the frontmatter for an exact match: `model: claude-sonnet-4-20250514` or `model: claude-opus-4-20250514`. Model names must match exactly what Claude Code accepts. If the key is missing entirely, the command inherits the session default.

### Error 6: Settings.local.json overrides not taking effect

```
You set a value in settings.local.json but the command still uses the project default.
```

**Cause**: `settings.local.json` must be in the same `.claude/` directory as `settings.json` and must use the same JSON structure.

**Fix**: Verify the file path is `.claude/settings.local.json` (not `.claude/settings.json.local` or any other variation). The file must be valid JSON. User-local overrides use the same keys as the project settings -- any key present in `settings.local.json` takes precedence over the same key in `settings.json`.

## What You Just Built

- **A custom slash command file** at `.claude/commands/review.md` with frontmatter controlling name, description, triggers, model, and tools
- **A configuration file** at `.claude/review-config.json` that the command reads to customize its behavior without editing the command itself
- **A help command** at `.claude/commands/review-help.md` that documents the primary command's usage, creating a self-documenting pair of commands
- **A pre-commit hook** in `.claude/settings.json` that automatically runs the review before every commit, catching issues before they reach the repository
- **A version-controlled shareable workflow** that any teammate can use by simply cloning the repo -- no additional setup or documentation required
- **Template variable integration** with `{{branch_name}}` and `{{diff}}` that injects runtime context into the command prompt without hardcoding any values

### Extension Challenges

If you finish the core exercise, try these extensions:

1. **Multi-language support**: Modify the review-config.json to have language-specific rules. Python files should check for PEP 8 violations, JavaScript files should check for async/await patterns, and Go files should check for error handling. The command should detect the file type and apply the appropriate rules.

2. **Auto-fix mode**: Create a `/review-fix` command that not only reports issues but also fixes them. This command should have `Edit` and `Write` in its `tools` list, and its prompt body should instruct the model to apply the fixes it recommends. Add a safety prompt that requires confirmation before writing changes.

3. **Summary-only mode**: Add a `--summary` variation of the review command that only shows the count of findings by severity without listing every finding. Create a separate command file or add conditional logic to the existing prompt body.

4. **Notification hook**: Add a `post-commit` hook in `settings.json` that runs a `/notify` command after every commit. The notifiy command should summarize what was committed and (optionally) send a notification. For the notification part, use `WebFetch` to call a webhook URL.

5. **Review threshold**: Add a quality gate to the pre-commit hook. If the review finds more than 5 critical issues, the commit should be blocked. If it finds fewer, the commit should proceed. This requires the hook to evaluate the output of the review command and make a decision based on the result.
