These exercises assume you already have the weather MCP server from the practice exercise connected to Claude Code or Claude Desktop. Work through the tiers in order — later exercises assume you can already do the earlier ones without a walkthrough.

## Tier 1 — Beginner

### Exercise 1: Register a second server with the filesystem server

**Task:** Register the official `@modelcontextprotocol/server-filesystem` server alongside your weather server, scoped to a single directory of your choice, and confirm both servers appear when you list your configured servers.

**Hint:** This server is distributed as an npm package and launched with `npx`, not `uv` — the `command` and `args` fields will look different from your weather server's entry, but the overall shape of the config is identical.

#### Solution

Using Claude Code:

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /home/you/Desktop
```

Or by editing `claude_desktop_config.json` directly:

```json
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": ["--directory", "/home/you/projects/weather", "run", "weather.py"]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/you/Desktop"]
    }
  }
}
```

Verify with:

```bash
claude mcp list
```

Expected result:

```text
weather: uv run --directory /home/you/projects/weather weather.py - ✓ Connected
filesystem: npx -y @modelcontextprotocol/server-filesystem /home/you/Desktop - ✓ Connected
```

### Exercise 2: Test a server independently with the MCP Inspector in CLI mode

**Task:** Instead of the browser-based Inspector UI, use the Inspector's `--cli` mode to list your weather server's tools directly from the terminal, without opening a browser.

**Hint:** The `--cli` flag switches the Inspector from an interactive UI to a scriptable command that takes a `--method` argument.

#### Solution

```bash
npx @modelcontextprotocol/inspector --cli uv run weather.py --method tools/list
```

Expected result — a JSON array describing both tools and their input schemas:

```json
{
  "tools": [
    {
      "name": "get_alerts",
      "description": "Get weather alerts for a US state.",
      "inputSchema": { "type": "object", "properties": { "state": { "type": "string" } }, "required": ["state"] }
    },
    {
      "name": "get_forecast",
      "description": "Get weather forecast for a location.",
      "inputSchema": { "type": "object", "properties": { "latitude": { "type": "number" }, "longitude": { "type": "number" } }, "required": ["latitude", "longitude"] }
    }
  ]
}
```

This mode is especially useful for confirming a server's tool list has not silently changed after an edit, without leaving your terminal.

### Exercise 3: Call a tool directly through the Inspector CLI

**Task:** Using the same `--cli` mode, call `get_alerts` for the state of Florida directly, bypassing Claude entirely.

**Hint:** The `--method tools/call` mode needs both `--tool-name` and one or more `--tool-arg key=value` pairs.

#### Solution

```bash
npx @modelcontextprotocol/inspector --cli uv run weather.py --method tools/call --tool-name get_alerts --tool-arg state=FL
```

Expected result (contents vary depending on real current alerts):

```text
{
  "content": [
    {
      "type": "text",
      "text": "No active alerts for this state."
    }
  ]
}
```

This confirms the tool's handler runs correctly independent of any AI client — useful whenever you want to separate "is my server broken" from "is my Claude configuration broken."

### Exercise 4: Remove a server and confirm it is gone

**Task:** Remove the filesystem server you registered in Exercise 1, and confirm Claude Code no longer lists it.

**Hint:** There is a single command for this, and it takes only the server's name.

#### Solution

```bash
claude mcp remove filesystem
```

```bash
claude mcp list
```

Expected result:

```text
weather: uv run --directory /home/you/projects/weather weather.py - ✓ Connected
```

The `filesystem` entry no longer appears in the list, and if you edited `claude_desktop_config.json` by hand instead, confirm its entry has been deleted from the file's `mcpServers` object.

## Tier 2 — Intermediate

### Exercise 5: Diagnose a deliberately broken configuration

**Task:** Take a working weather server config and deliberately introduce a bug — a relative path instead of absolute, or a single backslash on Windows — then use the tools from this lesson to identify exactly what broke, without looking at the diff you made.

**Hint:** Start with `claude mcp get weather` or the Claude Desktop connector menu, then narrow down to a JSON validator if the server shows as disconnected rather than simply missing.

#### Solution

Introduce the bug, for example changing:

```json
"args": ["--directory", "/home/you/projects/weather", "run", "weather.py"]
```

to a relative path:

```json
"args": ["--directory", "./weather", "run", "weather.py"]
```

Restart Claude and check status:

```bash
claude mcp get weather
```

Expected result — a failure, since `./weather` resolves relative to wherever Claude's own process happens to be running, not your project directory:

```text
weather
  Status: ✗ Failed to connect
  Command: uv --directory ./weather run weather.py
  Error: directory not found
```

The fix is to replace the relative path with the absolute one from `pwd` (or `cd` on Windows Command Prompt). The general lesson: any time a server that worked when you ran it by hand fails only when Claude launches it, suspect a relative path or an assumption about the current working directory first.

### Exercise 6: Manage environment variables across two servers with different needs

**Task:** Add a second server definition that requires an API key (you can simulate this with any placeholder value), while keeping the keyless weather server as-is, and confirm each server receives only the environment variables it needs.

**Hint:** The `env` block is per-server, not global — a variable defined for one server entry is not automatically visible to another.

#### Solution

```json
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": ["--directory", "/home/you/projects/weather", "run", "weather.py"]
    },
    "search": {
      "command": "npx",
      "args": ["-y", "@example/search-mcp-server"],
      "env": {
        "SEARCH_API_KEY": "${SEARCH_API_KEY}"
      }
    }
  }
}
```

Set the real value only in your shell, never in the file:

```bash
export SEARCH_API_KEY="your-actual-key-value"
```

Confirm with Claude Code that the variable resolved for the correct server only:

```bash
claude mcp get search
```

Expected result shows the `env` block with the variable present for `search` and confirms `weather` has no `env` block at all, since it never needed one.

### Exercise 7: Recover from a corrupted JSON config

**Task:** Deliberately remove a closing brace from `claude_desktop_config.json`, observe that Claude Desktop shows no MCP servers at all, and use a validator to locate and fix the exact error.

**Hint:** Do not guess by eye — pipe the file through a JSON parser and read the line and column number it reports.

#### Solution

```bash
python -m json.tool < claude_desktop_config.json
```

Expected result when the file is broken:

```text
json.decoder.JSONDecodeError: Expecting ',' delimiter: line 11 column 1 (char 210)
```

Open the file at that line, restore the missing brace or comma, and re-run the validator until it prints the parsed JSON back with no error. Restart Claude Desktop fully and confirm the connector menu shows your servers again.

### Exercise 8: Restart a single server without restarting Claude

**Task:** After making a code change to `weather.py` (add a print statement to `get_alerts`'s docstring, for instance), pick up the change without quitting your entire Claude Code session.

**Hint:** Claude Code has a slash command specifically for reconnecting one server.

#### Solution

Inside an interactive Claude Code session:

```text
/mcp
```

Select the `weather` server from the panel and choose to reconnect or restart it. This tears down the old child process and spawns a fresh one from your updated `weather.py`, without losing your conversation history or having to reconfigure anything.

## Tier 3 — Advanced

### Exercise 9: Convert the weather server to a remote HTTP-based server

**Task:** Take the stdio weather server and describe, in concrete steps, what would need to change for a teammate to connect to it as a remote server over the network instead of running their own local copy.

There is no single correct answer here, but a strong approach looks like this: the `FastMCP` instance's `mcp.run(transport="stdio")` call would need to change to an HTTP-capable transport (the current Python SDK supports running FastMCP over Streamable HTTP), and the server would need to run as a persistent process on a host reachable by your team, rather than being spawned fresh by each person's own Claude session. On the client side, instead of a `command`/`args` stdio entry, each teammate's configuration would use a `type: "http"` entry pointing at that host's URL:

```json
{
  "mcpServers": {
    "weather": {
      "type": "http",
      "url": "https://weather-mcp.yourteam.internal/mcp"
    }
  }
}
```

The trade-off worth calling out: you gain the ability to share one running server across a whole team without everyone needing `uv` and the project checked out locally, but you take on the responsibility of actually hosting and keeping that process alive, plus you introduce a new network dependency and a new place secrets could leak if the endpoint is not properly access-controlled. A weekend personal project almost never justifies this migration; a team-shared internal tool almost always does.

### Exercise 10: Build a launcher script that switches servers based on an environment flag

**Task:** Write a small wrapper script that Claude launches instead of your server directly, which reads an environment variable to decide whether to run the weather server in a "verbose logging" mode or a "quiet" mode, without needing two separate config entries.

A strong approach is a thin shell (or Python) wrapper that Claude's `command` field points to instead of `uv` directly:

```bash
#!/usr/bin/env bash
# launch-weather.sh
if [ "$WEATHER_VERBOSE" = "true" ]; then
    exec uv run --directory /home/you/projects/weather weather.py --log-level debug
else
    exec uv run --directory /home/you/projects/weather weather.py
fi
```

```json
{
  "mcpServers": {
    "weather": {
      "command": "/home/you/projects/weather/launch-weather.sh",
      "env": {
        "WEATHER_VERBOSE": "${WEATHER_VERBOSE}"
      }
    }
  }
}
```

An alternative approach is to skip the wrapper script entirely and instead have `weather.py` itself read the environment variable and branch on it internally — simpler to maintain for a single server, but less reusable if you plan to apply the same "verbose flag" pattern across many different servers, where a shared launcher script avoids duplicating the same branching logic in every server's source.

### Exercise 11: Pass a credential to a server without ever letting Claude's config file hold the real value

**Task:** Design a way to give an MCP server access to a real secret (an API key, a database password) such that the secret never appears in `claude_desktop_config.json`, `.mcp.json`, or any file that could plausibly be committed to version control, even by accident.

The core approach already shown in this lesson — `${VAR_NAME}` expansion reading from your shell environment — solves most of this, but it is worth going one step further for genuinely sensitive credentials. Rather than relying on a plain shell `export` (which can leak through shell history, process listings on some systems, or a misconfigured CI log), store the secret in your OS-level credential manager or a secrets manager, and have a small wrapper script fetch it at launch time and export it only into the child process's environment, never into your interactive shell:

```bash
#!/usr/bin/env bash
# launch-with-secret.sh
export DB_PASSWORD="$(security find-generic-password -a weather-server -w)"
exec uv run --directory /home/you/projects/weather weather.py
```

Point Claude's `command` field at this wrapper instead of at `uv` directly, and omit an `env` block for the secret entirely, since the wrapper injects it before your server process ever starts. The trade-off is added complexity — a script to maintain per platform — against a real reduction in how many places the secret's plaintext value ever touches disk. For a personal project this is often more caution than the risk warrants; for a credential that could cause real damage if leaked, it is worth the extra step.
