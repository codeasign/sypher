## The Problem Story

You spent the last two lessons building an MCP server. It has a clean tool schema with properly typed parameters, descriptive names, sensible default values, and helpful error messages when something goes wrong inside a handler. You tested it from the terminal a dozen times: `uv run weather.py` followed by manual JSON-RPC messages through stdin, and every single tool returned exactly the result you expected. You are proud of this server, and you should be — it is a small but complete piece of working software.

Then you open Claude Desktop, type something that should trigger your weather tool — "What is the wind speed at Sea-Tac?" — and nothing happens. Claude responds with a generic answer about checking weather websites or offers to search the web. Your server, which works perfectly when invoked directly, is invisible to Claude. There is no error message, no crash, and no hint in the UI that anything went wrong, because from Claude's point of view, your server never existed in the first place. Claude is not ignoring your server; it does not know your server exists.

You open the settings menu in Claude Desktop, hunting for something that looks like a place to register a custom tool. You scan every tab, every submenu, every icon. You find nothing obvious. There is no "Add Server" button, no "Browse for MCP Server" file picker, and no prompt at first launch asking where your servers live. Eventually you discover that Claude Desktop does not scan your filesystem for MCP servers, does not detect running processes on common ports, and does not offer any kind of auto-discovery. It reads exactly one JSON file, stored at exactly one platform-specific path, and if your server is not described inside that file using precisely the right keys and structure, it remains invisible no matter how well it runs on its own.

You find the file. You add an entry. You restart Claude Desktop. Now instead of nothing, you see a red icon next to your server name — or worse, a gray one with no tooltip. Claude Desktop knows there is supposed to be a server, but it cannot start it. Was the path wrong? Did a missing comma break the JSON? Did you use a single backslash in a Windows path when JSON needed a double backslash? Did you forget an environment variable that the server needs at launch? You are debugging blind because the configuration file gives you a generic failure indicator and nothing else — no log line, no stack trace, no indication of which field caused the problem.

This is the exact moment where most people give up on building custom MCP servers, not because the MCP protocol itself is hard — writing tool handlers, defining schemas, and returning structured results are all straightforward once you have done it once — but because the *connection* step, the part that is supposed to be "just configuration," turns out to have its own failure modes, its own file format quirks, its own path-resolution pitfalls, and its own debugging tools that nobody mentioned when you were learning to write tool handlers. Every one of these failures is simple in isolation: a missing escape character, a relative path that resolves differently from a different working directory, a process that prints a log line to stdout instead of stderr and corrupts the message stream. But they do not feel simple when you are facing them for the first time without a mental model of what Claude is actually doing when it reads that JSON file and spawns your server.

The good news is that once you understand what Claude is actually doing — how it parses the config file, how it resolves environment variables, how it spawns the child process, how the transport layer carries messages, and how the initialize handshake negotiates capabilities — the opaque parts stop being opaque. Every failure maps to a specific step in a specific sequence, and once you know which step failed, you know exactly where to look for the fix. This lesson walks through that entire sequence: how to tell Claude Desktop and Claude Code about a server you wrote yourself, what each step in the connection process actually does, and how to see what went wrong when a step does not succeed.

The story you just read is not hypothetical — it is exactly what happens to most developers the first time they try to connect a custom MCP server, and it happens in a predictable sequence. Stage one is confusion: the server works from the terminal but not from Claude, and you have no idea why. Stage two is discovery: you find the configuration file, make an edit, and restart. Stage three is the second failure: the server appears but shows an error state, and the error message is either absent or vague. Stage four is the turning point: you learn about the MCP Inspector, use it to test the connection independently, and finally see a concrete error message that tells you what is actually wrong. Every developer who has connected a custom MCP server has gone through this sequence, and the purpose of this lesson is to compress you from stage one directly to stage four by explaining what the configuration file is actually doing, what failure modes exist at each step, and what tools you have to diagnose each one.

There is a secondary pattern at work here that is worth naming explicitly. When you are building an MCP server and testing it from the terminal, you are acting as both the client and the developer — you type JSON-RPC messages through stdin, and you read the responses from stdout. That works great for validating that the server logic is correct, but it completely skips the configuration step, the spawn step, and the connection step. Passing that first test only proves that the tool handlers are correct. It does not prove that Claude can find, launch, or talk to the server. To prove the connection works, you need to test the path that Claude actually uses: a configuration file, a process spawn command, and a transport channel that Claude itself opened. The debug pattern that most developers miss is that the server's internal correctness and the server's discoverability through Claude are two completely separate things, and you need to test them separately.

The debugging workflow that emerges from this separation is straightforward but counterintuitive the first time you do it. When a server does not show up in Claude's tool list, your first check should not be the server code at all — it should be the configuration file syntax. If the configuration is valid, your second check should be whether the process starts at all when Claude runs the exact command from the config, which you can test by copying that command into a terminal. If the process starts, your third check should be whether the initialize handshake succeeds, which the MCP Inspector or `claude mcp get` can tell you. Each check rules out one entire box from the diagram below, and by the time you reach the server code, you already know the problem is there rather than guessing at every layer at once.

## The Concept

Connecting a custom MCP server to Claude means configuring Claude's MCP client to discover, launch, and communicate with your server process over stdio or another supported transport.

Think of it like plugging a new appliance into your house's electrical system. The appliance — your MCP server — knows how to do its job perfectly on its own workbench. It has its own power cord, its own internal circuits, and its own user manual. But to run inside the house, it needs to be wired into a circuit that the house already understands: a specific voltage, a specific plug shape, a specific breaker rating. Claude's configuration file is the wiring diagram. It tells the house exactly which circuit to open, what appliance to plug in, what power settings (environment variables) to supply, and what type of connection (stdio or network) the appliance expects.

This metaphor is more literal than it seems. A stdio server plugs into Claude's stdin and stdout the same way an appliance plugs into a wall outlet — the physical connection is simple, but it only works if the plug shape matches and the circuit is live. An HTTP-based server plugs into a network socket instead, like an appliance with a detachable power cord that connects to an extension lead running to a different room. The MCP client does not care which transport you choose; it cares that the transport is configured consistently on both sides, because mismatched expectations — a server that speaks SSE to a client expecting stdio, or vice versa — produce the same kind of failure as a European plug in a North American outlet: the pins look close enough to fit, but nothing conducts.

There is one more layer to the appliance metaphor that is worth extending because it maps directly to a concept in the MCP protocol. Once the appliance is plugged in and turned on, it does not just sit there waiting — it sends a periodic "heartbeat" signal to confirm it is still running, and so does the house's circuit. The MCP equivalent is the `ping` message, a lightweight JSON-RPC request that Claude sends to the server periodically and that the server must respond to. If the server does not respond to a ping within a reasonable timeout, Claude treats the connection as dead and transitions the server to the disconnected state. This is how Claude detects a server that has hung or crashed silently, without waiting for a tool call to fail first. Your server does not need to implement ping handling explicitly in most cases — the MCP SDK does it automatically — but you should be aware that a server that blocks on a long-running operation without responding to pings will be treated as disconnected before the operation completes.

Concretely, "connecting" means one of three things depending on which Claude product you use. For Claude Desktop, you add an entry to `claude_desktop_config.json`, a JSON file stored in your operating system's application support directory. This file is the single source of truth for the desktop app — there is no other way to register a server, no plugin system, and no dynamic discovery. Every server you want available in Claude Desktop must be listed here before the application starts.

For Claude Code, you run `claude mcp add` from the command line, which writes the server definition into a scoped configuration file. The `--scope` flag determines where: `--scope project` writes to `.mcp.json` in your project root (checked into version control, shared with the team), `--scope user` writes to `~/.claude.json` (personal, available across all projects you open), and `--scope local` writes to `.mcp.json` but with entries in `.gitignore` so it stays on your machine alone. Each scope has a purpose, and choosing the right one determines whether your server travels with the code or stays linked to your personal environment.

For a one-off session — a CI run, a demo environment, or a quick test — you point the `--mcp-config` flag at any JSON file on disk. This mode does not persist the server definition anywhere, so it is perfect for temporary setups where you do not want to modify your project's `.mcp.json` or your personal `~/.claude.json`. The `--strict-mcp-config` variant goes further: it causes Claude Code to refuse to start if the config file has any issues — a missing file, an invalid JSON structure, or any server that cannot be loaded — which is useful in automated pipelines where silent config failures would waste time.

In every case, you are answering the same three questions for Claude's MCP client: what command starts my server, what transport does it speak, and what environment does it need to run. Get those three answers right, and the connection works. Get any one of them wrong, and you are back to the red icon with no explanation. The three-questions framework is your mental checklist: before you even open the config file, write down the exact command that starts your server, the exact transport it uses, and the exact environment variables it needs. If the config file does not match that checklist, the failure is in the config. If the config matches but the server still does not appear, the failure is somewhere further down the chain — and you know which checks to run next.

## API, SDK, and CLI Reference

The table below covers every configuration surface, command, and tool you will use to connect and debug a custom MCP server, verified against the current Model Context Protocol documentation and Claude Code CLI reference. Each entry includes the exact syntax, what it does in practice, and a concrete example you can adapt to your own server.

Before reading the table, it helps to group these items into four categories conceptually. The first category is **configuration sources**: the files and commands that tell Claude about your server's existence (`claude_desktop_config.json`, `.mcp.json`, `~/.claude.json`, `--mcp-config`). The second category is **registration commands**: the CLI tools that create and manage entries in those configuration files (`claude mcp add`, `claude mcp add-json`, `claude mcp remove`, `claude mcp update`). The third category is **inspection commands**: the tools that show you what Claude actually knows about a server (`claude mcp list`, `claude mcp get`, `/mcp` slash command). The fourth category is **debugging tools**: the standalone utilities that let you test a server's transport and protocol without Claude involvement at all (MCP Inspector, the `--strict-mcp-config` flag for automated validation).

CLAUDE.md is worth calling out by name because it is easy to assume it configures MCP servers, and it does not. CLAUDE.md is a project instructions file — Claude reads it for context about your codebase, conventions, and guidelines, but it has no special syntax for registering a server process. The right place to document that "this project expects the `weather` MCP server to be connected" is a note in CLAUDE.md so any developer who joins the project knows what to install. The right place to actually *register* that server is `.mcp.json` (project scope, checked into version control) or `~/.claude.json` (personal scope, machine-specific). The two files serve complementary purposes: one tells humans what servers they need, and the other tells Claude how to start them.

Nothing in this picture is magic. Claude Desktop and Claude Code both use the same underlying MCP client logic — they read a configuration source, they use that configuration to start a process or open a network connection, and then they speak a fixed sequence of JSON-RPC messages over whatever transport that process expects. Every failure you will ever debug happens at one of the four boxes in this diagram:

1. **Config box** — a bad file path, invalid JSON syntax, or a missing key
2. **Spawn box** — a command that does not exist, a relative path that does not resolve, or a process that crashes immediately
3. **Transport box** — a stdio server that writes log output to stdout and corrupts the message stream, or a network connection that times out
4. **Protocol box** — a version mismatch during the handshake, tools that never get registered, or handler code that throws an uncaught exception

The rest of this lesson walks through each box in order. When something goes wrong with your own server, your first debugging question should be "which box?" — and once you know the answer, you know exactly where on this diagram to look.

There is a critical insight hidden in this diagram that is worth stating explicitly: the configuration source and the transport layer are the two places where most developers waste the most debugging time, because the failures there do not produce visible errors. A bad config entry produces no servers at all — no error message, no crash, just absence. A corrupt transport — a stdio server that writes log lines to stdout — produces protocol errors that look like the server's logic is broken, but the logic never even runs. The spawn box and the protocol box, by contrast, tend to produce visible errors: a process that cannot start produces a "command not found" or a crash report, and a handshake failure produces a version mismatch message. The invisible failures — boxes 1 and 3 — are the ones you need to learn to recognize by symptom rather than by error message.

## How It Actually Works — Step by Step

### Step 1: Claude reads the configuration source

At startup, Claude Desktop opens `claude_desktop_config.json` and parses it as JSON. If the file does not exist, has invalid syntax, or the `mcpServers` key is missing or not an object, Claude Desktop simply shows no MCP servers at all — there is no crash banner, no startup error, and no warning in the UI. It moves on as if you never intended to connect any servers, and you would have no way to tell it did anything wrong unless you already knew to check the file format.

This silent-default behavior is intentional on the client side — the MCP spec defines servers as optional infrastructure, so the client is not required to alert you when none are configured — but it creates a debugging gap: the failure mode of "wrong config" and the failure mode of "no servers installed at all" look identical. You cannot distinguish them without opening the file and verifying its contents.

Claude Code does the same thing but from more places at once, and it merges them according to a priority order. At session startup, it checks for servers in up to three locations: `.mcp.json` in your project root (shared, checked into git), `~/.claude.json` (personal, scoped across all projects), and any file passed with `--mcp-config` on the command line. If the same server name appears in more than one source, the most specific scope wins — local scope overrides project scope, which overrides user scope. This means you can have a personal `~/.claude.json` that adds a debugging server you use on every project, while your team's `.mcp.json` only contains the servers relevant to that specific codebase, and they coexist without conflict.

This step is entirely static file reading — no process has been started yet, and no network connection has been opened. Every bug you can introduce in this step is a JSON or path bug: a trailing comma that makes the entire file unparseable, a Windows path with single backslashes that the JSON parser interprets as escape sequences, a server entry nested one level too deep inside an extra object, or a typo in the `mcpServers` key name that causes the entire block to be silently ignored. All of these produce the same result — no servers, no error — which is why the first thing you should do when a server fails to appear is open the configuration file in a JSON-aware editor and make sure it parses.

There is a practical implication here that is easy to overlook: JSON does not allow comments, so you cannot leave yourself notes about which server is for what purpose. If you accumulate multiple server entries in your config file, make the server names self-documenting — use names like `weather-dev` and `weather-prod` rather than just `weather` — and keep a separate plain-text note elsewhere about what each server does and what environment it expects. A server definition that is syntactically valid but semantically wrong because it points to a stale path or a retired API will parse successfully and show up in Claude's list, only to fail on the first real tool call.

### Step 2: Claude validates and spawns the server process

Once the configuration is parsed and the server entries are extracted, Claude Code or Claude Desktop looks at each entry and decides how to start it. For a stdio server, this means calling the equivalent of `command args[0] args[1] ...` as a child process, with the `env` block from the config merged into that process's environment. This is a standard operating system process spawn — the same `CreateProcess` call on Windows or `fork` + `exec` on macOS and Linux that every other application uses to start a subprocess. There is no sandboxing layer, no special permission boundary, and no injection of additional security context beyond what the parent process already has. The child process runs as you, with all of your filesystem and network access.

For an HTTP or SSE server, no process is spawned at all — Claude instead opens a network connection to the URL given in the config entry. In this case, the `command` and `args` fields are irrelevant; only the `url` field and any `headers` or `env` fields that configure the connection matter. The remote server's lifecycle is entirely the responsibility of whoever deploys and maintains it, which means you also need to handle health checks, restarts, and connection recovery on the server side rather than relying on Claude to manage them.

This is the step where the most common class of configuration errors appears: "command not found" when the executable is not on Claude's `PATH`, or "relative path does not resolve" when a path like `./weather.py` works from your terminal but not from Claude. The fundamental issue is that Claude spawns the process using its own working directory — typically the application bundle directory for Claude Desktop, or the current shell directory for Claude Code — and not the directory where you happen to keep your terminal open when you tested the server manually. This mismatch is exactly why every official example uses an absolute path rather than a relative one: absolute paths work the same way regardless of which directory the parent process considers its current working directory.

For Claude Desktop on Windows, path resolution has an additional trap. A JSON string like `"C:\Users\you\weather.py"` is parsed as `C:Usersyouweather.py` because `\U` and `\w` and `\y` are all escape sequences in JSON — `\U` is an invalid escape that will cause a parse error, while `\w` gets silently converted to `w`. The correct form is either `"C:\\Users\\you\\weather.py"` with doubled backslashes or `"C:/Users/you/weather.py"` with forward slashes, which Windows handles without issue in most modern APIs.

There is one more subtlety about the spawn step that only shows up in practice: how long should Claude wait for the server to start before giving up? For stdio servers, Claude typically waits for the process to write its `initialize` response to stdout. If the process crashes during startup or is still initializing when the timeout expires, the spawn appears to have succeeded — the process handle existed for a moment — but the connection never materializes. The error message Claude shows will say something about a failed handshake rather than a failed spawn, which can mislead you into debugging the protocol when the real problem was a slow or crashing initialization in the server code. If a server consistently fails to connect after what looks like a correct configuration, add a short delay or a health-check log line at the start of your server's main function to confirm the process actually reaches the point where it listens on stdin.

### Step 3: The transport layer connects

Once the process is running or the network connection is open, Claude and your server need a shared channel for structured messages. For stdio servers, that channel is the child process's standard input and standard output: every JSON-RPC message is written as a single line to stdin and read as a single line from stdout on the other side. There is no framing protocol beyond a newline character — the entire message up to the newline is one complete JSON-RPC request or response. This newline-delimited JSON format is sometimes called NDJSON or JSON-lines, and it means that a single extra newline inside a message string can split it into two incomplete fragments that neither side can parse.

This is also why a stdio server must never write ordinary log output to standard output: a stray `print()` in Python or `console.log()` in Node.js inserts bytes into the message stream that are not valid JSON-RPC, corrupting whatever message follows and making the server appear, from Claude's side, to send garbage. The classic symptom of a server that logs to stdout is that the first tool call works correctly — the initial `print()` happened before any messages were exchanged — but subsequent calls fail with parsing errors because each log line shifts the message boundaries. Log lines belong on standard error, which Claude captures separately and may surface in its debug output without interfering with the protocol stream. If you are already using a logging framework in your server, make sure it outputs to stderr by default, or configure it explicitly with a `log_stream=stderr` or equivalent option.

For remote servers, the transport is HTTP-based instead of pipe-based. The current recommended transport for remote MCP servers is Streamable HTTP, which uses standard HTTP request-response semantics with the ability to stream multiple notifications over a single connection. The older SSE transport is still supported for compatibility with existing servers, but it is considered legacy for new development. Both carry the same JSON-RPC messages — initialize, listTools, callTool, ping, and notifications — they just deliver them over a network socket instead of an operating-system process pipe. The choice between them affects how you deploy the server and who manages its lifecycle, but the protocol layer on top of the transport is identical.

The transport layer has one more property that matters for debugging: it is transparent to the protocol messages. Claude does not care whether a JSON-RPC response arrived through a pipe or through an HTTP response body; it processes the same response object either way. This means you can test a stdio server locally during development, then deploy the same server behind an HTTP endpoint for team-wide access, without changing a single line of tool-handling code — you only change the transport configuration in the server's entry point and the corresponding config entry in Claude. The transport abstraction is one of the cleanest design decisions in the MCP specification, because it keeps your business logic completely separate from the deployment topology.

A practical debugging technique that leverages this transport transparency is to test your server's transport layer independently using a simple JSON-RPC echo pattern. Before connecting the server to Claude at all, you can send a `ping` request manually through the same transport mechanism your server expects. For a stdio server, this means piping a well-formed `ping` JSON-RPC message through stdin and checking that you get a `pong` response on stdout. For an HTTP server, it means sending a `POST` to the server's endpoint with the same JSON body and checking the response. If the ping-pong works, the transport layer is functional and the issue is somewhere in the protocol or tool logic above it. If the ping-pong fails, the transport itself is broken, and there is no point debugging anything further up the stack until you fix it.

The `ping` message is worth understanding in detail because it plays a role in the connection lifecycle that goes beyond debugging. Claude sends `ping` requests to connected servers at periodic intervals to verify that the server is still alive and responsive. If the server does not respond within a timeout period, Claude transitions it to the disconnected state and stops routing tool calls to it. This means a server that enters an infinite loop, hangs on a slow external API call, or deadlocks on a shared resource will eventually be detected and disconnected, even if the tool handler itself never crashes. Most MCP SDKs handle `ping` responses automatically in a background thread or event loop, so you do not need to write handler code for them, but you need to ensure that your server's main event loop is responsive enough to process incoming `ping` messages even while a tool handler is running. A server that processes tool calls synchronously on a single thread will not respond to `ping` until the current tool call completes, which means a long-running tool call — anything over about thirty seconds — risks being interpreted as a dead connection.

### Step 4: The initialize handshake

With a transport channel open in both directions, Claude sends an `initialize` request. This is a mandatory JSON-RPC request that carries Claude's protocol version number, a list of client capabilities (support for tool notifications, resource subscriptions, prompt templates, and so on), and an optional client identification string. Your server must respond with an `initialize` result that includes its own protocol version, its own supported capabilities, and identification information (server name and version). Until this response is received and processed, Claude will not attempt any further communication — not `listTools`, not `callTool`, not even a `ping` health check. The connection is in a "not ready" state that is invisible to you as a user but clearly defined in the protocol.

In the MCP specification, the initialize handshake is the only request-response pair that has special ordering rules. The `initialized` notification that follows — sent by Claude to confirm it received the server's response — is technically a notification rather than a request, which means the server does not need to reply to it. But the server should not process any other requests from Claude until it receives this notification, because the protocol version negotiation is not complete until both sides have acknowledged each other's capabilities. Most MCP SDKs handle this sequencing automatically, but if you are implementing a server from scratch without an SDK, you must implement this ordering check yourself or risk processing a `callTool` with the wrong protocol assumptions.

If the protocol versions are incompatible — for example, your server was built against an older version of the MCP SDK that does not support the version Claude advertises — the handshake fails. If your server never responds because it crashed during startup, the handshake times out. If the response is malformed, parsing fails. In every case, the connection fails right here, before a single tool has been discovered or called, and Claude marks the server as errored. This is the step where the `/mcp` slash command in Claude Code and the `claude mcp get <name>` command become most useful: the error message they surface will often tell you exactly whether the handshake succeeded or not.

The handshake is the same regardless of whether you are connecting to your own hand-written server or a published one. It is baked into the MCP specification itself, so there is nothing project-specific to configure for it. Your job is only to make sure your server's MCP SDK is a version recent enough to speak a protocol version Claude understands, which for the Python and TypeScript SDKs means keeping them reasonably up to date — any version from the last year should work without issue.

### Step 5: Capability exchange and tool discovery

After the handshake succeeds, Claude sends a `listTools` request. This is a simple request with no arguments: it asks the server to enumerate every tool it exposes, along with the full JSON Schema description of each tool's parameters. Your server responds with a list of tools, where each tool has a name (unique within the server), a human-readable description, and an `inputSchema` object that defines the shape of valid arguments using standard JSON Schema keywords — `type`, `properties`, `required`, `description` on individual properties, and occasionally `enum` or `pattern` constraints.

The quality of your tool descriptions directly affects how well Claude decides when to call each tool. A description like `"Returns weather data for a location"` is significantly less useful to the model than `"Returns current weather conditions including temperature, wind speed, humidity, and precipitation probability for a given latitude and longitude. Use this tool when the user asks about weather at a specific location."` The model uses the tool description to decide which tool matches a given user request, so a short or vague description means Claude may call the wrong tool or fail to call any tool at all, even when the right one is available. Write tool descriptions as if you are writing documentation for another developer who has never seen your code — because from the model's perspective, that is exactly what they are.

Claude stores this tool list in memory for the duration of the session. It is how Claude Desktop's connector menu populates the tool names you see when you click the attachment icon. It is how Claude Code decides which tool to call when it encounters a request that matches a tool description. And it is how the MCP Inspector displays the tool schema in its browser-based UI — the Inspector sends the same `listTools` request and renders whatever comes back.

If your server registers no tools at all — for example, because a decorator was misapplied, a registration call never ran due to a conditional that evaluated to false, or the server is missing the SDK call that registers handlers — the handshake still succeeds, and the transport still works, but the tool list comes back empty. Claude will behave as if the server barely exists: there will be a server name in the status display, but no tools to invoke, which can be confusing because the server itself is "connected" while being completely unusable. This is a debugging scenario that looks like a transport or config problem but is actually a tool-registration problem inside your server code.

The tool schema you return from `listTools` also determines how well Claude can use your tools. Every parameter should have a clear description that tells the model what kind of value to supply. A parameter described as `"The latitude of the location"` is better than one with no description at all, but `"The latitude of the location in decimal degrees, between -90 and 90, using the WGS84 coordinate system. Positive values are north of the equator."` is significantly better because it gives the model enough context to generate correct values even without seeing example data. Parameters that accept enumerated values should use an `enum` constraint in the schema rather than describing the valid values in the description text, because the model treats `enum` as a hard constraint while text in a description is advisory.

### Step 6: The tool call lifecycle

When you ask Claude to do something that matches one of your server's tools, Claude constructs a `callTool` request. This request includes the tool name — exactly as it appears in the `listTools` response — and an arguments object whose keys and types match the `inputSchema` you declared. It is Claude's responsibility to generate arguments that satisfy the schema, but it is your server's responsibility to validate them again before acting, for the same reason you never trust user input in a web application: the model can hallucinate parameter values, omit required fields, or supply data types that do not match the schema despite its best efforts. Server-side validation is not optional, even for tools that seem straightforward.

There are three patterns for structuring the server-side validation, and the right one depends on how much logic you are willing to write. The simplest pattern is to let the SDK validate the schema before it reaches your handler — most MCP SDKs do this automatically, rejecting calls that do not match the declared schema with a clear error message. The second pattern is to add explicit checks inside your handler for the fields that matter most, catching cases where the schema validation passed (because the data types were technically correct) but the values do not make sense for your use case — a latitude of 200 degrees passes a `number` type check but is not a valid coordinate. The third pattern, which you should only use when performance is critical and you have full control over the client, is to trust the schema entirely and skip handler-level validation — but this is risky because a future version of the model or a different client might generate arguments that the original schema did not anticipate.

Your server's handler runs whatever logic the tool requires — calling an external API, querying a database, reading a file, performing a calculation — and returns a result. The result is a list of `content` items, each with a `type` field. The most common type is `"text"`, a plain string. You can also return structured `"json"` items, `"image"` items for binary data, or even `"resource"` items that reference resources the server has exposed. If something goes wrong inside the handler, you should return a result with `isError: true` rather than letting the exception propagate uncaught, because an uncaught exception closes the transport and disconnects the server, while a well-formed error result lets Claude explain the problem to the user and continue the conversation.

Claude incorporates the tool result into its next response, and if the task needs another tool call — for example, fetching weather data for a second location after displaying the first — the cycle repeats. Each round trip is independent: Claude can call the same tool multiple times with different arguments, or call different tools in any sequence, or decide based on one tool's result that no further tools are needed. Your server has no say in the order of calls beyond expressing it through the tool schema you provide — a property marked as `required` will tend to be filled in, but an optional property may or may not be provided based on what the model infers from the conversation context.

There is an important timing consideration for tool calls: Claude may call multiple tools in parallel if the task allows it. For example, if you ask Claude to look up the weather in Seattle and Portland, Claude may send two `callTool` requests simultaneously rather than waiting for the first to complete before sending the second. Your server should handle concurrent requests correctly — using thread-safe data structures, avoiding shared mutable state between handler invocations, and ensuring that database connections and API clients are designed for concurrent access. A server that assumes single-threaded sequential calls will encounter race conditions, data corruption, or deadlocks when Claude calls it in parallel.

### Step 7: Error handling and disconnection

If any step from 1 through 6 fails at any point, Claude does not crash the entire session. It marks that individual server as disconnected or errored and continues the conversation without it. The rest of your servers keep working, and Claude Desktop or Claude Code continues to respond to your prompts — it just pretends the failed server does not exist for the rest of that session. This graceful degradation is essential for a development workflow where you might have half a dozen servers registered and only one is under active development: a bug in your new server does not take down your notes tool or your database schema explorer.

A stdio server that exits unexpectedly is detected the moment its process handle closes. If the server's process crashes with a segmentation fault or an unhandled exception, Claude sees a closed pipe and transitions the server to the errored state. A remote server that drops its HTTP connection is detected when the HTTP stream ends or a keepalive timeout fires. In both Claude Desktop and Claude Code, you can see this state: a red icon next to the server name in Desktop, or an "errored" status from `claude mcp list` in Claude Code. The `claude mcp get <name>` command will often surface the last known error — a process exit code, an HTTP status code, or an error message from the transport layer — giving you a concrete starting point for debugging.

The recovery strategy differs by product. In Claude Code, you can use `claude mcp connect <name>` to attempt a reconnection without restarting the entire session. This re-runs the spawn and handshake sequence from the beginning, which is useful when you have fixed the underlying issue and want to test it in the same conversation. In Claude Desktop, there is no equivalent command — the only recovery option is to quit and restart the application, which re-reads the config file and re-attempts every server connection from scratch.

This step is the one most people skip when things go wrong. The natural instinct is to jump to "the server does not work" and start changing the config file or rewriting the handler code. But the right first step is always to check what Claude actually reported: is the server errored, or is it connected with zero tools, or is it not appearing at all? Each of those three symptoms points to a different box in the big-picture diagram, and the fastest path to a fix is ruling out the wrong boxes before you change anything.

There is an additional error-handling layer within the tool call itself that is separate from connection errors. When your server receives a `callTool` request and the handler runs successfully but returns an error condition — for example, the external API is unavailable or the requested data does not exist — you should return a tool result with the `isError` field set to `true`. This tells Claude that the tool executed but encountered a business-logic failure, which is different from a transport or protocol failure. Claude will report the error to the user and continue the conversation, rather than treating it as a server crash and transitioning the entire server to the errored state. Using `isError` correctly is the difference between a server that appears unreliable — constantly disconnecting — and a server that handles failures gracefully within the conversation flow. When you set `isError: true` on a result, the server continues running and can accept further tool calls in the same conversation. When you let an exception propagate out of the handler without catching it, the MCP SDK typically terminates the connection, and Claude transitions the server to the errored state.

Because Claude Code reads from multiple configuration files and merges them, understanding the scope hierarchy is essential for debugging a server that appears in one session but not another, or that uses the wrong version of a command.

One practical workflow that takes advantage of the scope system is to use project scope for servers that need specific environment variables (like a database connection string that differs per developer) and keep only the server name and command in the shared `.mcp.json`, leaving the `env` block empty. Each developer sets their own environment variables at the shell level, and the `${VAR_NAME}` expansion picks them up at launch time. This keeps the shared config file clean of secrets while still giving every developer the same server registration.

Another useful pattern is the `--scope local` option, which creates an entry in `.mcp.json` but adds it to `.gitignore` automatically. This is the right choice for servers that are specific to your local development environment — a file-system exploration tool, a local database inspector, or a server that accesses credentials stored only on your machine. The server definition lives in the project directory where you can manage it alongside other project files, but it never accidentally gets committed to version control and shared with the whole team.

## Configuration File Examples

Before diving into the debugging workflow, it helps to see the full shape of a configuration file with multiple servers so you can recognize the pattern you are aiming for. The three examples below cover the most common scenarios you will encounter.

### Claude Desktop multi-server config

This is what a complete `claude_desktop_config.json` looks like with two servers, one using stdio and one using a network transport. The `mcpServers` key contains one entry per server, and each entry has the same basic structure: either a `command` with `args` for stdio, or a `url` for HTTP-based transport.

```json
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": [
        "--directory",
        "C:/Users/you/projects/weather-server",
        "run",
        "weather.py"
      ],
      "env": {
        "API_KEY": "${WEATHER_API_KEY}"
      }
    },
    "notes": {
      "command": "node",
      "args": [
        "C:/Users/you/projects/notes-server/index.js"
      ]
    }
  }
}
```

Each server name — `weather` and `notes` in this example — is the human-readable name that appears in Claude Desktop's connector menu. You can edit these names freely, but they must be unique within a single config file because Claude uses the name as the key in its internal server registry.

### Project-scoped `.mcp.json` for Claude Code

This is what the same configuration looks like when stored in a project's `.mcp.json` for use with Claude Code. Notice the structure is identical to the `mcpServers` block from the Desktop config, because the two products share the same underlying server definition format.

```json
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": [
        "--directory",
        "/absolute/path/to/weather-server",
        "run",
        "weather.py"
      ]
    }
  }
}
```

The key difference between this file and the Desktop config is scope: `.mcp.json` lives in the project directory and is typically committed to version control, so it should not contain secrets or environment-specific paths. Use `${VAR_NAME}` expansion for anything that differs between developers, and keep the file focused on what every developer needs to run the project's servers.

### One-off session with `--mcp-config`

When you pass a config file with `--mcp-config`, the format is the same as `.mcp.json`. The difference is purely in lifecycle: the file is read once at session startup and never referenced again. This is useful for testing a server in isolation without modifying any persistent configuration.

```json
{
  "mcpServers": {
    "debug-server": {
      "command": "python",
      "args": [
        "C:/Users/you/debug/server.py"
      ],
      "env": {
        "DEBUG": "true",
        "LOG_LEVEL": "debug"
      }
    }
  }
}
```

When you run `claude --mcp-config ./debug-config.json`, Claude Code starts a session that has access to the debug server but does not write anything to `.mcp.json` or `~/.claude.json`. Any servers you normally have registered through those files are still available — the `--mcp-config` file is additive, not a replacement — unless you also pass `--strict-mcp-config` to enforce a precisely defined set of servers.

## Practical Debugging Workflow

When a server does not appear in Claude or shows an error state, work through these checks in order. Each check rules out an entire layer of the stack and narrows down where the problem actually lives.

### Check 1: Validate the configuration file syntax

Open your configuration file in a JSON-aware editor or run it through a JSON validator. A single missing comma, an extra trailing comma, or a single backslash in a Windows path will make the entire file unparseable, and Claude will silently treat it as if no servers are configured. This is the most common cause of "the server does not show up at all" and takes ten seconds to verify.

If the JSON is valid, check the structure: the top-level key must be `mcpServers` (not `mcp_server`, not `MCPServers`, not `servers`), and each entry under it must be an object with at least a `command` string. If you are using a remote server, the entry needs a `url` field instead of `command` and `args`. If both `command` and `url` are missing, Claude cannot start or connect to the server, and it will silently skip the entry.

### Check 2: Verify the command works from your terminal

Copy the exact `command` and `args` from the configuration file, concatenate them into a shell command, and run it from any directory — not from your project directory, because Claude may not be running from there either. If the command fails with "command not found" or a file-not-found error, the configuration needs an absolute path or a more specific command. If the command succeeds and the server starts (printing its initialization message or waiting for input), the spawn itself is functional.

This check catches the most insidious class of errors: commands that work when you test them from the directory where the script lives but fail when run from any other directory. Testing from a random directory on your filesystem — your desktop, your home folder, or even a system directory — proves that the path resolution does not depend on the current working directory.

### Check 3: Test the transport layer independently

If the command works from the terminal but fails when Claude runs it, the issue may be in the transport layer. Use the MCP Inspector to test the connection: run `npx @modelcontextprotocol/inspector <your-command>` with the exact same command and arguments from the configuration file. The Inspector opens a browser window showing the connected server, its tools, and their schemas. If the Inspector connects successfully, the transport layer and the initialize handshake are both working, and the problem is somewhere in how Claude is calling the server — possibly a permission issue or a scope collision.

If the Inspector fails to connect, the problem is in the transport or handshake layer. The Inspector shows error messages that are more detailed than what Claude Desktop surfaces, including JSON-RPC error codes, protocol version mismatches, and timeouts. Read the error message carefully: a version mismatch tells you to update your SDK, a timeout tells you the server is starting too slowly, and a parse error tells you the server is outputting garbage to stdout.

### Check 4: Check Claude's reported status

Run `claude mcp list` to see all registered servers and their statuses. A server that shows "errored" has a configuration or runtime problem. A server that shows "connected" but has zero tools has a tool-registration problem. A server that does not appear at all has a scope or file-location problem. Each status points to a different root cause, so check the status before changing anything.

Run `claude mcp get <name>` for a specific server to see its full configuration and the last error message. The error message is often the single most useful piece of debugging information available — it may say "process exited with code 1" (startup crash), "connection refused" (wrong URL or port), or "protocol version mismatch" (SDK incompatibility). Write down the exact error message before you start making changes, because the error message is the symptom, and fixing the symptom without understanding it leads to guesswork.

### Check 5: Isolate the server from other servers

If you have multiple servers registered and one is failing, temporarily remove the others from the configuration file and test only the failing server in isolation. A server that crashes on startup due to a port conflict or a resource contention issue may appear to have a configuration problem when the real issue is that another server is using a resource it needs. This is especially common with HTTP-based servers that bind to specific ports — if two servers try to bind to the same port, the second one to start will fail with an address-in-use error that looks like a configuration problem but is actually a port collision.

These five checks cover every layer of the connection process: configuration, spawn, transport, protocol, and tool registration. If you work through them in order, you will always find the root cause before you start changing code. If you skip steps, you risk changing the wrong thing — rewriting a tool handler when the problem was a missing environment variable, or editing a config file when the problem was a server that registered zero tools. The discipline of checking from the bottom up is the single most important debugging skill for MCP connections.

A concrete example illustrates how this workflow plays out in practice. Imagine you have a weather server that you registered with `claude mcp add`, and when you run `claude mcp get weather`, the status shows "errored" with exit code 1. Following the checks in order: first, you verify the `.mcp.json` file parses correctly — it does. Second, you copy the command from the config into a terminal — `uv run weather.py` — and it runs without error from your project directory. Third, you test it from a different directory entirely, and it fails with "file not found." Now you know the issue: the relative path `weather.py` only works when the working directory is the project root, but Claude Code spawns the process from the directory where you launched it, which might not be the project root. The fix is to add `--directory` with an absolute path to the args array. You made it to check 2 of 5 before finding the root cause, and you solved it by changing the config file, not the server code.

This example illustrates why the bottom-up order matters. If you had started by reading the server source code (check 5 territory), you would have spent ten minutes reading perfectly good handler code before discovering the issue was a relative path in the config file. The checks are ordered from the simplest and fastest to the most complex and time-consuming, so you always find the easy fix before you start investigating the hard one.

## Cost, Limits, and Safety

Connecting a broken server does not cost you API tokens by itself. A failed JSON parse, a process that cannot start, a handshake that times out — none of these events trigger a model call, because the model never gets involved. The cost implications start once the handshake succeeds and Claude begins using the server's tools: every callTool request and its result becomes part of the conversation context, and Claude pays token costs for the full round trip. A tool that returns a large, unfiltered blob of data — an entire log file, a full database table, a raw API response with thousands of fields — burns through tokens on every single turn that references it, not just the turn where the call happened, because the model sees the full conversation history including previous tool results on every subsequent request.

There are two cost patterns worth understanding explicitly. The first is the "expensive tool" pattern: a tool that does a cheap computation but returns a large result, like a file reader that returns the entire contents of a 100 KB configuration file. The token cost of that result is incurred once when the tool is called, but it is incurred again on every subsequent model request because the conversation history includes the full tool result. If you ask Claude to do three more things after reading the file, you pay for the file contents three more times. The fix for this pattern is to make your tools return summaries or filtered results by default and only return the full data when explicitly requested through a parameter.

The second pattern is the "chatty tool" pattern: a task that requires many small tool calls in sequence. For example, asking Claude to look up the weather for ten cities results in ten separate callTool requests, each with its own round trip and its own entry in the conversation history. The token cost per call is small, but the cumulative cost of ten round trips plus ten result blocks in the conversation is larger than a single call that accepts an array of cities. If you find yourself making many similar calls in the same conversation, consider whether your tool schema could accept a batch parameter that processes multiple items in one handler invocation.

There is a third pattern that is less obvious but equally important: the "deep context" problem. If a tool call happens early in a long conversation, the token cost of that tool's result — including the full input and output of the handler — is paid on every subsequent model turn because the conversation history does not get compressed retroactively. A 2,000-token tool result from turn 3 is still in the context window at turn 30, contributing to the total token count and therefore the cost of every API call made after it. This is not a problem with short conversations of a few turns, but it becomes significant in extended coding sessions where Claude might make dozens or hundreds of tool calls over the course of hours. The practical implication is that tool handlers should return the minimum content needed for the model to understand the result, not the maximum content the tool could theoretically produce.

The most common way people waste real time, rather than tokens, is a relative path in the `command` or `args` field. Claude spawns your server process from its own working directory — not your terminal's working directory — so `"args": ["run", "weather.py"]` without an explicit directory flag will fail to find `weather.py` even though the exact same command works fine when you run it by hand from your project folder. This single mismatch accounts for more support questions about MCP server configuration than any other issue, because it contradicts the normal developer experience of running scripts from a project root. Always use an absolute path in configuration files, on every platform.

On Windows, path resolution has the JSON-escape trap mentioned earlier, but it also has a subtler trap: a path with spaces. If your username contains a space — `C:\Users\Jane Doe\weather\weather.py` — you need to make sure the space is inside a single argument, not split across two. If `args` contains `["C:\Users\Jane", "Doe\weather\weather.py"]`, the space splits the path into two arguments and Claude's spawn call passes them separately, which will not resolve to any file. The correct form is either `["C:\Users\Jane Doe\weather\weather.py"]` as a single argument, or split the path into the part before the space and the part after — but in practice, keeping the entire path as one array element is simpler and less error-prone.

Environment variables deserve the same care you would give an API key in any other context: never write a secret directly into `claude_desktop_config.json`, `.mcp.json`, or any file you might commit to version control. Use the `${VAR_NAME}` expansion syntax so the actual value comes from your shell environment at launch time, and keep `.mcp.json` limited to non-secret defaults if it is meant to be shared with a team. If a referenced environment variable is not set when Claude launches, most implementations either fail the server start silently or pass an empty string through — neither of which raises an alarm on its own. You need to check `claude mcp get <name>` or the equivalent Desktop logs after any change to confirm the value actually landed, because a missing environment variable produces silent failure rather than a helpful error.

Safety considerations go beyond secrets. A `mcpServers` entry is, at its core, a directive that says "run this command with these arguments and this environment every time Claude starts." If you ever add a server definition you copied from an untrusted source without reading it, you are handing that source the ability to run arbitrary code on your machine every time Claude launches. The tool permission system in Claude Code protects you from unexpected tool calls — Claude will ask before calling a tool that accesses the filesystem or network — but it only protects you from tool calls you can see. It does nothing about what a malicious server does inside its own handler code, before any tool call is made, or during startup. Treat a new MCP server config the same way you would treat a new package dependency: read what it launches before you trust it, and prefer scopes that match how much you actually want to share a given server.

There is also a subtler safety consideration around the `env` block in server configs. When you set an environment variable in the config file, that variable is passed to the server process but it is also potentially visible to any other process that can read the parent process's environment at launch time. On a shared machine or a CI environment where multiple processes run under the same user account, an environment variable set through the MCP config could leak to a co-located process that reads `/proc/self/environ` or the equivalent. If you are running Claude on a shared development server, consider whether a hardcoded environment variable in a config file is the right approach or whether a more secure secret-management system would be safer.

There are limits to how many servers Claude can manage simultaneously, though these limits are practical rather than hard-coded. Each stdio server runs as a separate process with its own memory footprint, file descriptors, and stdio pipes. On a typical development machine, running more than ten to fifteen stdio servers at once can consume significant system resources, especially if any of them have expensive initialization routines. If you find yourself registering many servers, consider which ones you actually need active at the same time and move the rest to a separate configuration file that you load on demand with `--mcp-config`. Each additional server also adds to Claude's startup time, because the client needs to spawn the process, wait for the handshake, and call `listTools` before the session is fully initialized.

Rate limiting is another practical concern, though it applies to the underlying APIs your server calls rather than to MCP itself. When Claude calls a tool that hits an external API — a weather service, a database, a search engine — the tool handler should respect the API's rate limits and return appropriate errors when limits are reached. A tool that exceeds rate limits and starts returning HTTP 429 responses or connection timeouts will appear unreliable to Claude, and the model may stop calling it even when the tool itself is working correctly. A simple approach is to wrap external API calls in a retry-with-backoff loop and return any rate-limit errors as `isError` results with a clear explanation of when to retry.

## When to Use / When Not To

Use Claude Desktop's `claude_desktop_config.json` when you are working in the desktop application and want a small number of servers available across every conversation you have. This is the natural home for personal productivity servers: a notes tool that stores quick ideas, a calendar reader that checks your schedule, a personal weather server like the one from the previous lesson, or a file-search tool that works across your documents folder. These servers are tied to your machine and your workflow, not to any specific project, so putting them in a single machine-wide configuration file makes sense. The trade-off is that every server registered here appears in every conversation, even conversations where none of those tools are relevant. If you accumulate many servers, consider whether you actually want all of them active in every chat.

There is a subtle usability consideration with Claude Desktop that is worth knowing about before you register a dozen servers: Claude Desktop loads every server defined in the config file at startup, and it keeps them loaded for the entire session. Each server consumes system resources — memory for the process, file descriptors for the pipes — even if you never ask Claude to call its tools. A server that performs an expensive initialization, like loading a large machine-learning model or connecting to a remote database on startup, will slow down every launch of Claude Desktop regardless of whether you use that server in any given conversation. If you have servers with expensive initialization, consider keeping them in a separate configuration file that you swap in only when needed, rather than adding them to the main desktop config.

Use Claude Code's `claude mcp add` and scoped configuration when you are working from the terminal and want servers tied to a specific codebase. Project-scoped servers (`.mcp.json`, committed to version control) are the natural home for development tools that every team member needs — a database schema explorer, a deployment tool, a linter that runs across the codebase, or a documentation search tool that queries the project's internal wiki. When a new developer clones the repository and runs `claude` for the first time, the project-scoped servers are already available because their definitions travel with the code. Personal-scope servers (`~/.claude.json`) are for tools you use across multiple projects but do not want to impose on your team — a personal note-taker, a to-do list manager, or a bookmark search tool that is specific to your own setup.

Choose stdio transport for any server that only needs to run on your own machine, alongside your own Claude session. Stdio is simpler to configure (no port numbers, no URLs, no firewall rules), it has no network exposure (the process communicates over pipes that only the parent process can access), and its process lifecycle is fully managed by Claude — Claude spawns the server when it is needed and terminates it when the session ends, so you never have to remember to start or stop it separately. Stdio servers are inherently single-user and single-session, which is the right trade-off for development and personal productivity tools.

Choose an HTTP-based transport when the server needs to run independently of any single Claude session. This is the right choice for a shared service that multiple team members access concurrently, a data pipeline server that stays alive between sessions and caches expensive computations, or a server hosted on a machine without Claude installed that is accessed remotely. Streamable HTTP is the recommended transport for new remote servers, supporting both request-response and notification patterns over standard HTTP. The older SSE transport is still accepted for compatibility with existing servers that use it, but if you are building a new remote server from scratch, start with Streamable HTTP.

Do not reach for a custom MCP connection when the task is a one-time script you will run once and never again. Writing a proper MCP server with tool schemas, error handling, and a configuration file has setup cost that only pays off across repeated use. A single-use script is better served by a direct Python or Node.js command that you run on demand, or by a Claude Code custom slash command if you want to invoke it repeatedly from within a session. Do not default to SSE transport for a new server today; it still works for compatibility with existing infrastructure, but current MCP guidance favors Streamable HTTP for anything you are building from scratch, and using a legacy transport for a new project adds complexity with no benefit.

Do not connect a server you have not read the source of, especially one that requires broad filesystem or network access. The MCP permission model protects you from tool calls you can see — it prompts you before a tool reads your files or sends data to a network endpoint — but a server with a `command` field runs arbitrary code before any tool permission check occurs. If you download an MCP server from an untrusted source and register it with Claude, you are effectively giving that source the ability to run code on your machine during every Claude session, with all the access privileges of your user account. Treat an MCP server binary the same way you treat any other executable you download from the internet: inspect it before you run it, and do not run it if you cannot verify its origin.

Here is a quick decision tree for choosing the right approach to connecting your server. Start by asking whether the server is personal or shared. If it is personal and you always use Claude Desktop, put it in `claude_desktop_config.json`. If it is personal and you always use Claude Code, register it with `--scope user` so it is available across all your projects. If it is shared with a team, register it with `--scope project` so `.mcp.json` can be committed to version control. If it is temporary, use `--mcp-config` with a one-off file. If your server needs to run independently of any Claude session — as a background service accessible from multiple clients — use Streamable HTTP transport regardless of which Claude product you use. If your server only runs on your machine alongside your own Claude session, use stdio transport for its simpler configuration and automatic lifecycle management.

There is also a practical consideration around how you manage server updates over time. When you need to change a server's configuration — update its command, add an environment variable, or change its transport — you have two choices. The first is to remove the server entry and re-add it with the new configuration, which works but requires you to type the full command again. The second is to use `claude mcp update`, which modifies the existing entry in place and preserves any fields you do not change. The `update` command is especially useful when you are iterating on a server during development and need to change only the command or arguments without retyping the entire configuration block.

The transport decision also affects how you handle authentication, which is worth calling out separately because it is easy to overlook when you are just getting started. A stdio server inherits your machine's local security context — it runs as you, with your permissions, and there is no network-based authentication to configure because there is no network. An HTTP-based server, by contrast, is exposed on a network port and needs to authenticate incoming connections from Claude. The MCP specification does not mandate a specific authentication mechanism for HTTP transports, so you need to decide how your server verifies that an incoming connection is actually Claude and not some other client. The most common approach is to include an authentication header in the config file's `headers` field and validate it on the server side, using a shared secret that both Claude and the server know but that is not exposed to the network at large.

Here is a summary table of the decision factors:

| Factor | Choose stdio | Choose HTTP/SSE |
|---|---|---|
| Server lifecycle | Managed by Claude | Managed by you |
| Number of clients | One (current Claude session) | Many (team-wide or service) |
| Network exposure | None (pipes only) | Port must be accessible |
| Authentication | None needed (local process) | Must configure explicitly |
| Configuration complexity | Low (command + args) | Medium (URL + headers + auth) |
| Persistence across sessions | No (starts fresh each time) | Yes (can cache state) |
| Deployment scope | Single machine | Any reachable network

## Mental Model Check

These five questions check whether you have built the right mental model of how the connection process works. Each question targets a specific misconception that commonly trips up developers when they first start debugging connection failures. Try to answer each one before reading the explanation — the gap between what you guessed and what the explanation says is exactly where your debugging intuition needs to grow.

### Question 1

Where does Claude Desktop look for its MCP server configuration, and what happens if the file is missing or malformed?

Claude Desktop reads a single file — `claude_desktop_config.json`, located under your platform's application support directory. On Windows, that is `%APPDATA%\Claude\claude_desktop_config.json`, which typically expands to `C:\Users\<YourUsername>\AppData\Roaming\Claude\claude_desktop_config.json`. On macOS, it is `~/Library/Application Support/Claude/claude_desktop_config.json`. If the file is missing, contains invalid JSON, or is missing the `mcpServers` key, Claude Desktop does not crash or display an error dialog — it simply behaves as though no MCP servers are configured, as if the setting does not exist. This silent-default behavior is by design, but it means the first debugging step for a server that does not appear should always be to open the file in a JSON-aware editor and verify its syntax, because no amount of restarting the app will fix a file that does not parse.

The file location differs between platforms, but the file format is identical. This means you can prepare a configuration file on one platform and share it with someone on another platform, as long as you are careful about path separators — forward slashes in paths work on both macOS and Windows, while backslashes only work on Windows and may cause issues there as well if you use single backslashes instead of doubled ones. If you maintain a shared configuration file across platforms, use forward slashes for all paths and let each operating system resolve them in its own way.

### Question 2

You configured a server with a relative path like `"args": ["run", "weather.py"]`, and it worked fine when you ran the same command from your terminal, but it fails when Claude launches it. Why?

Claude spawns your server process using its own working directory, not the directory where your terminal happens to be sitting when you tested the command manually. On Claude Desktop, the working directory is typically the application bundle directory, which is almost certainly not the directory containing `weather.py`. On Claude Code, the working directory is the directory where you launched Claude Code, which may or may not be the same directory where `weather.py` lives — and even if it matches today, it may not match tomorrow when you start Claude Code from a different terminal. A relative path that resolves correctly in one directory will fail, or worse, silently resolve to a different file with the same name in a different directory, when Claude launches the process from a different location. Always use an absolute path in the `command` and `args` fields of any MCP server configuration, and prefer forward slashes even on Windows to avoid JSON escape-sequence issues.

The most reliable way to avoid this issue is to use an absolute path in the config file directly. If you are using a tool like `uv` that expects a relative path to a script, use the `--directory` flag with an absolute path to set the working directory for the child process explicitly. This pattern — absolute path for the directory, relative path for the script within that directory — works because the directory is absolute and independent of Claude's working directory, while the script reference is relative only to that directory. The config entry looks like `"args": ["--directory", "C:/Users/you/weather", "run", "weather.py"]`, which works from any working directory on any platform.

### Question 3

What is the difference between stdio transport and an HTTP-based transport, and how does that affect who manages the server's process lifecycle?

With stdio transport, Claude spawns your server as a child process and communicates over that process's standard input and output pipes. Claude fully owns the process lifecycle: it starts the server when Claude launches (or when it first needs it), and it terminates the server when the session ends or when the server becomes disconnected. You do not start or stop a stdio server manually; Claude does it for you, which is both the convenience and the constraint — convenient because you never have to remember to start it, constraining because the server does not survive beyond a single Claude session.

With an HTTP-based transport — Streamable HTTP or the legacy SSE — Claude does not spawn any process at all. It connects to a URL that some already-running server exposes, and that server's lifecycle is managed independently of Claude's session. The server can be started by a systemd service, a Docker container, or a manual terminal session, and it continues running even after Claude is closed, ready for the next connection. This is the right choice when multiple Claude clients need access to the same server, or when the server performs long-running tasks that should not be interrupted when a single conversation ends.

The process lifecycle difference has a practical consequence for development workflows. With stdio transport, every time you change your server code and want to test it, you need to restart Claude so that the old process is terminated and a new one is spawned with the updated code. With an HTTP-based transport, you can restart the server independently — kill the old process, start the new one — and Claude will reconnect on the next tool call without needing to restart the session. If you are doing rapid iteration on server code, consider developing against an HTTP transport even if you plan to deploy with stdio, because the faster iteration cycle outweighs the extra configuration complexity during development.

### Question 4

After you add a server to `claude_desktop_config.json`, does Claude Desktop pick it up immediately, or do you need to do something else first?

You need to fully quit and restart Claude Desktop. Claude Desktop reads the configuration file exactly once, at application startup — closing the application window is not the same as quitting the application. On both Windows and macOS, the application continues running in the system tray or menu bar after the window is closed, and it does not re-read the configuration file when the window is re-opened. A running session will not notice that the file changed underneath it, no matter how long you wait. The correct sequence is: edit the file, save it, fully exit Claude Desktop (right-click the system tray icon and select Quit on Windows, or use the menu bar Quit command on macOS), then launch Claude Desktop again. If the server still does not appear after a restart, the issue is almost certainly in the file format or the server configuration, not a caching problem.

This behavior is different from Claude Code, which re-reads its configuration at the start of each new session — you can edit a `.mcp.json` file while Claude Code is running, close the session, open a new one, and the changes take effect immediately without needing to restart a desktop application. If you switch between Claude Desktop and Claude Code frequently, it helps to remember which product caches its config for the session (Desktop) and which re-reads it every time (Code), because the debugging steps for a config change are different for each.

### Question 5

Your server's `initialize` handshake succeeds, but Claude reports that it has no tools available from that server. At which step in the connection process does this problem originate, and what does that tell you about where to look for the bug?

This happens after the handshake, during the `listTools` step, which is Step 5 in the connection sequence. Since the handshake succeeded, you know that the transport layer is working correctly, the process spawn succeeded, the protocol version is compatible, and the capability negotiation completed — all of the "infrastructure" steps that come before tool discovery are fine. The bug lives specifically in how your server registers or exposes its tools. Common causes include a missing tool decorator or registration function call, a conditional that prevents tool registration under the current environment, a handler that was registered under a different name than intended, or an `inputSchema` that failed to compile and caused the tool to be silently skipped. Debugging should focus on the server's tool-registration code, not on the configuration file, the transport setup, or the handshake logic — those are all verified working by the successful `initialize`.

A systematic debugging approach for this scenario is to add a log statement right after the server's tool registration call that prints the name and count of every registered tool. When you connect the server to the MCP Inspector, you will see this log on stderr (if the server logs to stderr, which it should) along with the tool count. If the count is zero, you know the registration call never ran or produced no results. If the count is correct but Claude still sees no tools, the issue is in how the server is returning the tool list in response to `listTools` — perhaps the response format does not match what the client expects. Most MCP SDKs handle this automatically, but if you are implementing the protocol manually, the `listTools` response must follow the exact JSON-RPC structure and content schema that the specification defines.
