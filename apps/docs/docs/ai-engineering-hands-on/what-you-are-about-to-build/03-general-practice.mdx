# Practice Exercises

These exercises help you deepen your understanding of the AI Research Engine you built. They are organized by difficulty so you can find the right level for your current skill.

Each exercise tells you what to build and what the expected behavior should be. Try to write the solution yourself before looking at hints or checking online.

## Tier 1 — Beginner

These exercises modify the existing `research.py` script in simple ways. Each one changes five lines of code or fewer.

### Exercise 1.1: Change the AI's Personality

Modify the system prompt in the `ask_ai()` function so the AI responds in a different style.

**Requirements:**
- Change the system prompt to make the AI respond like a pirate
- Change the system prompt to make the AI respond like a strict teacher
- Run the script and ask the same question with both personalities

**Expected behavior:** The same question produces a completely different style of answer depending on which system prompt you use.

**Hints:**
- The system prompt is the `system` parameter in `client.messages.create()`
- Pirate example: `"You are a pirate research assistant. Answer every question like a salty sea captain."`
- Teacher example: `"You are a strict but fair teacher. Answer every question with precision and demand correct terminology."`

**Reflection question:** You changed only one string in your code, and the entire behavior of the application shifted. What does this tell you about where the intelligence of an AI application lives?

### Exercise 1.2: Add a Timestamp to the Log

Modify the `save_exchange()` function to include the current date and time with each entry.

**Requirements:**
- Import `datetime` from the Python standard library
- Use `datetime.datetime.now()` to get the current timestamp
- Write the timestamp on its own line before each question-answer pair

**Expected output in research_log.txt:**

```
[2026-07-03 14:30:22]
Q: What is machine learning?
A: Machine learning is a subset of artificial intelligence...
------------------------------------------------------------
[2026-07-03 14:35:10]
Q: What is deep learning?
A: Deep learning is a subset of machine learning...
------------------------------------------------------------
```

**Hints:**
- Add `from datetime import datetime` at the top of the file
- Add this line inside `save_exchange()`: `timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")`
- Write `f"[{timestamp}]\n"` to the file before the question line

**Reflection question:** Why is a timestamp useful when you have many entries in your research log?

### Exercise 1.3: Add a Greeting Counter

Add a counter that shows how many questions you have asked in the current session.

**Requirements:**
- Track how many questions the user asks in one session
- Display the count at the start of each answer (e.g., "Answer 1:", "Answer 2:")
- Reset the counter to 1 each time the script starts

**Expected behavior:**

```
Your question: What is Python?
Asking the AI Research Engine...

--- Answer 1 ---
Python is a high-level programming language...
```

**Hints:**
- Add a variable `question_count = 0` before the while loop in `main()`
- Increment it inside the loop with `question_count += 1`
- Print it as part of the answer display

**Reflection question:** What other session-level statistics could you track and display?

### Exercise 1.4: Validate Question Length

Add validation that rejects questions shorter than 3 characters or longer than 2000 characters.

**Requirements:**
- Check the question length before sending it to the API
- If the question is too short, print a message and prompt again
- If the question is too long, truncate it to 2000 characters and notify the user

**Expected behavior:**

```
Your question: hi
Question is too short. Please ask a meaningful question.
Your question:
```

```
Your question: [pastes a 5000-character question]
Question was truncated to 2000 characters.
Asking the AI Research Engine...
```

**Hints:**
- Add the check in the `main()` function after reading the question
- Use `len(question)` to check length
- Use `question[:2000]` to truncate

**Reflection question:** Why would you want to limit question length? What happens if you send a very long prompt to the API?

## Tier 2 — Intermediate

These exercises extend the script with new features. Each one requires adding 10-30 lines of code.

### Exercise 2.1: Show Token Usage

The API response includes token counts for input, output, and total. Display these counts alongside the answer.

**Requirements:**
- Access the `message.usage` object from the API response
- Display input tokens, output tokens, and total tokens after each answer
- At the end of the session, display the total tokens used across all questions

**Expected behavior:**

```
Answer:
Python is a high-level programming language...

Token usage: 24 input + 156 output = 180 total tokens
```

At session exit:

```
Session summary: 3 questions, 512 total tokens used
Research log saved to research_log.txt
Goodbye!
```

**Hints:**
- The usage data is in `message.usage.input_tokens` and `message.usage.output_tokens`
- Store the cumulative total in a variable that persists across loop iterations
- The `anthropic.types.Message` object contains the usage information

**Reflection question:** Why does token count matter? How could tracking tokens help you manage API costs?

### Exercise 2.2: Session-Based Log Files

Instead of appending everything to one file, create a new log file for each session with a timestamp in the filename.

**Requirements:**
- Generate a filename like `research_2026-07-03_14-30-22.txt` when the script starts
- All questions and answers from that session go into that file
- The original `research_log.txt` behavior should work alongside this (save to both, or offer a choice)

**Expected behavior:**

The script creates a file like `research_2026-07-03_14-30-22.txt` in the project folder, containing only the current session's conversation.

**Hints:**
- Generate the filename once at the start of `main()` using `datetime.now().strftime()`
- Create a function `get_session_filename()` that returns the formatted name
- Pass the filename to `save_exchange()` as a parameter

**Reflection question:** When would session-level logs be more useful than a single cumulative log? When would the single log be better?

### Exercise 2.3: Markdown Export

Add a function that exports the research log as a markdown file suitable for viewing in any markdown viewer.

**Requirements:**
- Write a function `export_to_markdown()` that reads `research_log.txt` and creates `research_log.md`
- Format each exchange as a markdown blockquote for the question and normal text for the answer
- Add a heading with the export date at the top

**Expected output in research_log.md:**

```
# AI Research Log
Exported on 2026-07-03

> **Q:** What is machine learning?

Machine learning is a subset of artificial intelligence...

---

> **Q:** What is deep learning?

Deep learning is a subset of machine learning...

---
```

**Hints:**
- Parse the log file line by line
- Lines starting with `Q:` become `> **Q:**` lines
- Lines starting with `A:` become the answer lines
- Separator lines (`---`) become `---` in markdown too
- Call `export_to_markdown()` at the end of the session

**Reflection question:** Why is markdown a good format for viewing research logs? What other export formats could be useful?

### Exercise 2.4: Configurable Temperature

Add a command-line argument that lets you control the `temperature` parameter of the LLM, which affects how creative or random the responses are.

**Requirements:**
- Accept a `--temperature` argument when running the script (e.g., `python research.py --temperature 0.7`)
- Temperature should be a float between 0.0 and 1.0
- Default to 0.7 if no argument is given
- Pass the temperature to the API call

**Expected behavior:**

```bash
python research.py --temperature 0.0
```

This produces very focused, deterministic answers.

```bash
python research.py --temperature 1.0
```

This produces more creative, varied answers.

**Hints:**
- Use `sys.argv` or the `argparse` module to read command-line arguments
- The `argparse` module is cleaner: `parser.add_argument("--temperature", type=float, default=0.7)`
- Pass the temperature to the API: `client.messages.create(temperature=temperature)`

**Reflection question:** What happens when temperature is set to 0.0? What happens at 2.0 (try it if the model supports it)? Why would you choose different temperatures for different tasks?

## Tier 3 — Advanced

These exercises require significant code additions or architectural changes. They are designed to stretch your skills and prepare you for the later lessons in this course.

### Exercise 3.1: Multi-Model Comparison

Build a script that sends the same question to multiple models and compares their responses side by side.

**Requirements:**
- Define a list of at least three different models (e.g., `claude-sonnet-4-20250514`, `claude-3-haiku-20240307`, `claude-3-opus-20240229`)
- For each model, send the same question and capture the response
- Print the responses with clear headers showing which model generated each
- Display the response time and token usage for each model

**Expected behavior:**

```
$ python compare_models.py
Question: Explain recursion in one paragraph.

--- Model: claude-3-haiku ---
Time: 1.2s | Tokens: 150
Answer: Recursion is a programming technique where a function...
----------------------------------------

--- Model: claude-sonnet-4 ---
Time: 2.8s | Tokens: 220
Answer: Recursion occurs when a function calls itself...
----------------------------------------

--- Model: claude-3-opus ---
Time: 4.5s | Tokens: 310
Answer: Recursion is a method of solving problems where...
----------------------------------------
```

**Hints:**
- Create a new file `compare_models.py` (keep your original `research.py` intact)
- Wrap the API call in a function that accepts the model name as a parameter
- Use `time.time()` to measure response duration
- Store results in a list of dictionaries for easy comparison

**Reflection question:** Based on the speed, cost, and quality differences you observed, which model would you use for a chatbot? Which for a one-time research deep dive? Which for batch-processing thousands of simple queries?

### Exercise 3.2: Batch Question Processor

Build a script that reads questions from a text file and processes them all automatically, saving the results to a single output file.

**Requirements:**
- Accept an input filename as a command-line argument: `python batch_ask.py questions.txt`
- The input file contains one question per line
- Process each question through the API
- Save all questions and answers to an output file with a timestamp
- Print progress updates as each question is processed

**Expected input file (questions.txt):**

```
What is the capital of France?
What is the boiling point of water in Celsius?
Who wrote Romeo and Juliet?
What is the square root of 144?
```

**Expected output:**

```
$ python batch_ask.py questions.txt
Processing question 1 of 4: What is the capital of France?
  Done (2.1s)
Processing question 2 of 4: What is the boiling point of water in Celsius?
  Done (1.8s)
Processing question 3 of 4: Who wrote Romeo and Juliet?
  Done (2.3s)
Processing question 4 of 4: What is the square root of 144?
  Done (1.5s)
All questions processed. Output saved to batch_output_2026-07-03.txt
```

**Hints:**
- Use `sys.argv` to get the filename argument
- Read the file with `open(filename, "r")` and iterate over lines
- Call the same `ask_ai()` function you wrote in the original exercise
- Track progress with a counter formatted as `f"Processing question {i} of {total}"`
- Use `time.time()` to measure and display elapsed time per question
- Handle potential API errors for individual questions without stopping the batch

**Reflection question:** Batch processing is useful for automated workflows. What kinds of tasks would you automate this way? How would you handle a batch where some questions fail while others succeed?

### Exercise 3.3: Interactive Chat History Viewer

Build a companion script that reads a research log file and lets you browse through it interactively.

**Requirements:**
- Read a research log file (default: `research_log.txt` or accept a filename argument)
- Display a numbered list of all question-answer pairs in the log
- Let the user select a number to view the full exchange
- Support commands: `list` (show all entries), `view N` (show entry N), `search keyword` (find entries), `quit`

**Expected behavior:**

```
$ python view_log.py
Log file: research_log.txt (12 entries)
Commands: list, view N, search keyword, quit

> list
  1: What is machine learning?
  2: What is deep learning?
  3: What is the difference between AI and ML?
  4: Explain neural networks
  ...

> view 2
--- Entry 2 ---
Q: What is deep learning?
A: Deep learning is a subset of machine learning that uses
   neural networks with multiple layers...
--------------------------------------------------------

> search neural
Found 3 matching entries:
  1: What is machine learning?
  4: Explain neural networks
  7: What are convolutional neural networks?
```

**Hints:**
- Parse the log file into a list of dictionaries, each with `question` and `answer` keys
- Use a `while True` loop for the interactive command prompt
- For the search feature, use case-insensitive matching with `keyword.lower() in text.lower()`
- The `view_log.py` script does not call any API — it only reads files

**Reflection question:** This script separates reading from generating. Why is it useful to have a dedicated viewer for your research log? How does this separation of concerns apply to larger software projects?

## Going Further

Here are ideas for continuing to extend your AI Research Engine on your own:

- **Voice input:** Use the `speech_recognition` Python library to accept spoken questions
- **Web interface:** Build a simple HTML page using Flask or FastAPI that wraps your research engine
- **Multiple languages:** Ask questions in different languages and compare the AI's fluency
- **Question suggestions:** Have the AI suggest three follow-up questions after each answer
- **Summarize session:** Add a command that asks the AI to summarize everything discussed in the current session

Each of these extensions introduces a real-world engineering concept (speech processing, web frameworks, multilingual support, summarization) in a practical, immediately useful way.

## After the Exercises

When you complete all three tiers of exercises, you will have:

- A solid understanding of how your AI Research Engine works under the hood
- Experience modifying and extending a working Python script
- Familiarity with the API request-response cycle
- The ability to debug common issues independently
- A foundation for the more advanced topics that follow

The most important skill you are building here is not any specific Python syntax or API knowledge. It is the ability to take a working piece of software, understand what it does, and change it to do something new. That is the core skill of software engineering, and you have just practiced it with real AI technology.

Keep your `research.py` file and your exercise modifications. You will revisit them in later lessons as you add prompt engineering, structured output, memory, and retrieval capabilities.

When you are ready, move on to the next lesson where you will learn how prompts work under the hood and how to engineer them for precise, reliable results.
