⏱ 15 minutes · Difficulty: Beginner

## The Problem

Imagine you want to build something that thinks. Not a calculator that returns a number, not a search engine that returns links — something that understands a question written in plain English and writes back a thoughtful, original answer.

You have an idea: a research assistant that can help you learn any topic. You type "explain quantum entanglement like I am ten years old," and instead of getting links, you get a clear explanation written just for you. You follow up with "what does that mean for real-world technology?" and the assistant remembers the context and builds on it.

This is the kind of tool that would change how you learn, how you work, how you explore ideas. But you have no idea where to start.

Do you need a data science degree? Do you need to train your own AI model? Do you need a team of engineers and a million-dollar cloud budget?

The answer to all three is no. By the end of this course, you will build exactly this tool from scratch using nothing more than Python, a few free or low-cost services, and under 100 lines of code.

## The Concept

You will build what this course calls the **AI Research Engine** — a command-line program that connects to a large language model (LLM) via an API and lets you hold intelligent conversations about any topic.

Here is the one-sentence concept: you write questions in your terminal, a Python script sends them to an AI model hosted by a company like Anthropic or OpenAI, the model generates a response, and the script prints that response back to you and saves the full conversation to a file.

That is it. That is the core idea. Everything else you will learn — prompt engineering, structured output, embeddings, retrieval-augmented generation, multi-agent systems — is a way to make that simple loop more powerful.

The course is organized into these major phases:

| Phase | What You Will Build |
|-------|-------------------|

Each phase is one or more lessons in this course. You build something real in every lesson.

## How It Works

The flow above breaks down into a sequence of concrete steps. Each step maps to a few lines of Python code that you will write yourself.

### Step 1: You Type a Question

You run the script from your terminal and it prints a prompt:

```
$ python research.py
Welcome to the AI Research Engine
Type your question (or 'quit' to exit):

Your question:
```

The script uses Python's built-in `input()` function to capture what you type. This single line of code is the entry point for everything that follows.

### Step 2: The Script Loads Your API Key

The script reads your API key from an environment variable called `ANTHROPIC_API_KEY`. This is a secret string that identifies you to the LLM provider. It is stored outside your code so you never accidentally commit it to version control.

```python
import os
api_key = os.environ["ANTHROPIC_API_KEY"]
```

The `os.environ` dictionary gives your script access to all environment variables set on your system. If the key is missing, the script stops with a clear error — no crashes, no mystery.

### Step 3: The Script Builds the Prompt

Your question gets wrapped in a structured message format that the API expects. This format includes a system prompt that sets the AI's behavior and a user message with your actual question.

```python
import anthropic

client = anthropic.Anthropic(api_key=api_key)

message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system="You are a helpful research assistant. Answer questions clearly and accurately.",
    messages=[
        {"role": "user", "content": user_question}
    ]
)
```

The `model` parameter tells the API which LLM to use. Different models offer different trade-offs between speed, cost, and capability. The `max_tokens` parameter sets a ceiling on how long the response can be — 1024 tokens is roughly 750 words, enough for a thorough paragraph or a short essay. The `system` parameter is your first encounter with prompt engineering: it sets the AI's persona and behavioral constraints.

The `messages` list is where the conversation history lives. Right now it has only one message (your current question), but in later lessons you will include past questions and answers so the AI remembers context across a conversation.

The SDK handles all the low-level details: serializing your data to JSON, encrypting the HTTP request with TLS, managing the connection pool, and parsing the response. You never see a raw HTTP request or worry about network error handling — the SDK does it for you.

### Step 4: The LLM Processes Your Request

This is where the magic happens — but it is not actually magic. The LLM is a neural network with billions of parameters that has been trained on a vast corpus of text. When it receives your prompt, it predicts the most useful sequence of tokens (words and sub-words) that answer your question.

From your perspective, this is a single line of code:

```python
response_text = message.content[0].text
```

But behind the API, the model is running a forward pass through its neural network — matrix multiplications at an enormous scale — all in under a few seconds.

### Step 5: The Script Displays the Response

The response from the API comes back as a JSON object containing the generated text plus metadata (model used, token counts, stop reason). You extract the text and print it:

```python
print(f"\\nAI Research Engine says:\\n")
print(response_text)

The `response_text` variable contains the model's generated text as a plain Python string — no special formatting, no control characters, just words. You can process it further: count words, extract keywords, save it to a database, or feed it into another function.
```

The output looks like this in your terminal:

```
AI Research Engine says:

Great question! The sky appears blue because of a phenomenon
called Rayleigh scattering. Sunlight is made up of all the
colors of the rainbow. When sunlight reaches Earth's atmosphere,
it collides with tiny gas molecules — mostly nitrogen and oxygen.
...
```

### Step 6: The Script Saves the Conversation

Before asking if you want to continue, the script appends the exchange to a file. This builds a searchable history over time:

```python
with open("research_log.txt", "a", encoding="utf-8") as f:
    f.write(f"Q: {user_question}\\n")
    f.write(f"A: {response_text}\\n")
    f.write("-" * 60 + "\\n")
```

This is your AI's memory. Future lessons will expand this simple text file into a proper database with search and retrieval.

### Step 7: The Script Loops for Follow-Up Questions

Finally, the script asks if you have another question:

```python
again = input("\\nAsk another question? (y/n): ")
if again.lower() != "y":
    print("Goodbye! Research log saved to research_log.txt")
    break
```

This loop is what turns a one-shot question-answer tool into a conversational research assistant. Your questions build on each other, and the log file captures the full conversation.

## Cost, Limits, and Safety

Building with LLMs is powerful, but you need to understand the real-world constraints.

### API Pricing

LLM APIs are not free, but they are very cheap for learning and experimentation. Here are approximate costs as of mid-2026:

| Model | Input Cost (per million tokens) | Output Cost (per million tokens) | Typical cost per question |
|-------|-------------------------------|--------------------------------|--------------------------|
| Claude Haiku | $0.25 | $1.25 | Less than $0.001 |
| Claude Sonnet | $3.00 | $15.00 | Less than $0.01 |
| Claude Opus | $15.00 | $75.00 | Less than $0.05 |
| GPT-4o mini | $0.15 | $0.60 | Less than $0.001 |
| GPT-4o | $2.50 | $10.00 | Less than $0.01 |

A typical question with a 500-token input and 300-token output using Claude Sonnet costs about $0.003. You can run hundreds of experiments for under one dollar.

To put this in perspective, here is a realistic cost projection for learning and development:

| Activity | Questions | Estimated Cost (Sonnet) | Estimated Cost (Haiku) |
|---------|-----------|----------------------|----------------------|
| One lesson's practice exercises | 20 | $0.06 | $0.01 |
| One week of daily practice | 140 | $0.42 | $0.07 |
| Building a prototype over a month | 500 | $1.50 | $0.25 |
| Running a small production bot for a day | 10,000 | $30.00 | $5.00 |

The cost scales with usage. For learning, your total API spend for the entire course will likely be under $5.00.

### Rate Limits

API providers limit how many requests you can send per minute. For learning purposes these limits are generous — you will not hit them during normal practice. If you do, the API returns a 429 status code and your script should retry with a backoff delay.

### Token Limits

Each model has a maximum context window (the total number of tokens it can process in one request). Current models support between 8K and 200K tokens. A token is roughly 0.75 English words, so a 200K context window can handle about 150,000 words — equivalent to a long novel.

### Safety Considerations

LLMs are powerful tools with important limitations:

- **They can be wrong.** LLMs sometimes produce confident-sounding but incorrect answers (hallucination). Never use AI-generated answers for medical, legal, or financial decisions without verification.
- **They reflect their training data.** The model's knowledge and biases come from the internet text it was trained on. Always maintain a critical mindset.
- **Your data goes to the API provider.** When you send a question to an API, the text leaves your machine. Never send sensitive personal information, passwords, or proprietary code to third-party APIs.
- **No privacy in free tiers.** Free API tiers may use your prompts for model training and improvement. Always read the provider's data usage policy.

### Safe API Key Handling

Follow these rules every time you work with API keys:

- Store keys in environment variables, never in your source code
- Add `.env` and `*.env` to your `.gitignore` file
- Never commit a file that contains a key
- If a key is accidentally exposed, revoke it immediately through the provider's dashboard
- Use a separate key for development and production

## When to Use What You Build

The AI Research Engine pattern — user prompt → LLM API → response — is the foundation for thousands of real-world applications.

### Use This Approach When:

- You need to generate natural language content (summaries, explanations, creative writing)
- You want to build a conversational tool (chatbot, tutor, research assistant)
- You need to extract or transform text at scale (classify, translate, paraphrase)
- You want to build an agent that takes actions based on natural language instructions
- You need to combine AI reasoning with your own data (RAG, document analysis)
- You are prototyping an AI product idea and need to validate it quickly

### Do Not Use This Approach When:

- You need guaranteed exact output every time (use traditional code with unit tests instead)
- You are handling sensitive personal or medical data without a data-processing agreement
- You need millisecond-level latency (LLM calls take 1-10 seconds minimum)
- The cost per invocation would exceed the value of the result
- You need deterministic, auditable results for regulatory compliance
- A simple lookup table, search engine, or rule-based system would solve the problem

To help you decide, here is a comparison of when an LLM-based approach beats traditional software and vice versa:

| Task | LLM Approach | Traditional Approach | Winner |
|------|-------------|-------------------|--------|
| Answer an open-ended question | LLM generates a natural explanation | Search engine returns links | LLM (synthesis) |
| Calculate a mathematical expression | LLM guesses the result | Calculator returns exact answer | Traditional (correctness) |
| Summarize a long article | LLM writes a concise summary | Algorithm extracts first paragraph | LLM (quality) |
| Sort a list of 10,000 numbers | LLM attempts the sort | Python's `sorted()` does it instantly | Traditional (speed) |
| Write a poem about AI | LLM generates original poetry | Template fills in blanks | LLM (creativity) |
| Authenticate a user login | LLM is never asked this | Password hashing + comparison | Traditional (security) |

The pattern is clear: use LLMs for tasks that require understanding, synthesis, creativity, and natural language. Use traditional code for tasks that require precision, speed, and determinism. The most powerful applications combine both.

## Mental Model Check

Answer these five questions to confirm you understand the big picture. Each question has a single correct answer. Try to answer before checking the explanation.

### Question 1

What is the primary role of the API in the AI Research Engine?

A. To store your conversation history
B. To connect your Python script to the LLM running on the provider's servers
C. To compile your Python code into machine code
D. To display the AI's response in a formatted window

**Answer: B.** The API is the bridge between your script and the LLM. Your script sends a request containing your prompt to the API endpoint, and the API returns the model's generated response. It does not store conversations (A), compile code (C), or handle display formatting (D).

### Question 2

Why do you store your API key in an environment variable instead of writing it directly in your Python script?

A. Environment variables make the script run faster
B. Python cannot process strings longer than 255 characters
C. So the key never appears in your source code and cannot be accidentally committed to git
D. Environment variables automatically rotate the key every 24 hours

**Answer: C.** Hardcoding keys in source code is a security risk. If you commit the file to git, the key is exposed to everyone with repository access. Environment variables keep the key outside your code — it is loaded at runtime from your operating system's environment.

### Question 3

What happens inside the LLM when your script sends a prompt?

A. The prompt is stored in a database and compared against similar queries
B. A search engine finds web pages matching your question and returns snippets
C. A neural network processes the prompt, predicts useful tokens, and generates a response
D. A human operator reads your question and types a response

**Answer: C.** The LLM is a neural network, not a database or search engine. It generates responses by predicting which sequence of tokens (words and sub-words) best answers your prompt. It does not look up pre-written answers (A), search the web (B), or employ humans (D).

### Question 4

If you ask an LLM for medical advice, what should you do with the answer?

A. Follow it exactly because AI models are trained on medical literature
B. Verify it with a qualified medical professional before taking any action
C. Use it as your primary diagnosis since it costs less than a doctor visit
D. Share it on social media as medical fact

**Answer: B.** LLMs can produce confident-sounding but incorrect information (hallucination). Never use AI-generated answers for medical, legal, or financial decisions without independent verification from a qualified professional.
