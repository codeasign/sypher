## The Problem

You build a customer-facing chatbot. It passes every test. It answers questions about account balances, order history, and shipping status with impressive accuracy. Your users love it -- at first.

Then the emails start coming in. "The bot told me my account was locked, but it could not unlock it." "It showed me my order history, but I could not cancel the order." "It knew my shipping address was wrong, but I had to call support to change it."

Your chatbot is a brilliant analyst locked in a room with no hands. It can read, it can reason, but it cannot act. Every time a user needs something done -- a refund, an update, a calculation -- the chatbot hands them a detailed instruction manual and says "you do it." The user has to switch tabs, log into another system, follow the steps, and come back. The friction is real. Users stop using the chatbot. They call support instead.

You look at your code. You already have functions for everything the chatbot is being asked to do: `lookup_account()`, `process_refund()`, `update_address()`, `cancel_order()`. The functions exist. The data is accessible. The only missing piece is the bridge between the LLM's reasoning and the functions that get things done.

This is the exact problem that tool use -- also called function calling -- was designed to solve.

## The Concept

Tool use is the mechanism by which an LLM can request the execution of external functions -- API calls, database queries, computations -- and incorporate the results into its response. Instead of only generating text, the LLM can say "I need to call this function with these arguments," wait for your application to execute the function, and then use the result to continue its response.

Imagine your team hires a brilliant analyst. You sit them at a desk and say "answer questions." They are good at reasoning, but they have no computer, no phone, no access to any data. They can only guess. That is what an LLM without tools is like.

Now give that analyst a computer with a database query tool, a web browser, and a calculator. Suddenly they can look up real numbers, compute accurate totals, and verify facts. They are the same analyst with the same reasoning ability -- but now they can act on the real world and bring real data back into their reasoning process.

That is tool use. The LLM stays the reasoning engine. Your application provides the tools. Together, they accomplish what neither can do alone.

### The Core Idea

When you send a request to an LLM API, you include a set of tool definitions. Each tool definition describes a function that your application can execute: what the function does, what parameters it accepts, and what data types those parameters expect. The LLM reads these definitions alongside the user's message and decides whether any of the tools would help answer the request.

If the LLM determines a tool is needed, it does not execute the tool itself. It produces a structured tool use request -- a JSON object containing the name of the tool and the arguments it wants to pass. Your application receives this request, executes the corresponding function with those arguments, and sends the result back to the LLM. The LLM then incorporates the result into its response.

You are always in control. The LLM can only request a tool call -- it cannot execute anything. Your application decides whether to honor the request, validate the arguments, and run the function. This separation is the key design principle: the LLM reasons, your application acts.

### How the Conversation Looks

Understanding the message format is essential for debugging. Here is what a tool use conversation looks like on the wire:

```
Turn 1:
  User: "What is the weather in Tokyo?"
  Assistant: [ToolUseBlock: get_weather(city="Tokyo")]
  User: [ToolResultBlock: "Weather in Tokyo: 22 C, Sunny"]

Turn 2:
  Assistant: "The weather in Tokyo is 22 C and sunny."
```

The user sends the first message. The assistant responds with a tool use block instead of text. The user sends back a tool result block. The assistant then produces the final text response. From the API's perspective, the tool result is a user message -- but it is a structured content block, not a text string.

Your application code must manage this conversation flow. You send the assistant's response (which includes the `tool_use` blocks) and the tool results (as a user message) back to the API on each turn. The API does not manage state for you -- you must accumulate and send the full message history on every request.

## API and SDK Reference

The following table shows every relevant field and structure for working with tool use using the Anthropic Python SDK. All syntax shown uses current SDK version 0.42.0 or later.

| Concept | Field or Structure | Purpose | Example or Notes |
| --- | --- | --- | --- |
| Tool definition | `tools` parameter in `client.messages.create()` | Pass an array of tool definitions to the API | `tools=[{"name": "get_weather", "description": "...", "input_schema": {...}}]` |
| Tool name | `name` (string) | Unique identifier for the tool; must match the function you will call | `"get_weather"` -- use snake_case, 1-64 characters |
| Tool description | `description` (string) | Tells the LLM when and how to use the tool | `"Get the current weather for a city. Use this when the user asks about weather conditions."` |
| Input schema | `input_schema` (dict) | JSON Schema describing the parameters the tool accepts | `{"type": "object", "properties": {"city": {"type": "string"}}, "required": ["city"]}` |
| Tool use request | `stop_reason: "tool_use"` | The response signals that the LLM wants to call a tool | Check `message.stop_reason` before processing content blocks |
| Tool use content block | `ToolUseBlock` with `id`, `name`, `input` | The structured request from the LLM containing the tool name and arguments | `block.name`, `block.input` give you the tool name and a dict of arguments |
| Tool result | `tool_result` content block | Send the function's output back to the LLM | `{"type": "tool_result", "tool_use_id": block.id, "content": str(result)}` |
| Tool choice | `tool_choice` parameter | Control how the LLM selects which tool to call | `{"type": "auto"}`, `{"type": "any"}`, or `{"type": "tool", "name": "get_weather"}` |
| Auto mode | `tool_choice: {"type": "auto"}` | LLM decides whether to call a tool or respond with text | Default behavior; the LLM uses its judgment |
| Any mode | `tool_choice: {"type": "any"}` | LLM must call at least one tool on every turn | Useful for classification or extraction pipelines where you always need structured output |
| Specific tool | `tool_choice: {"type": "tool", "name": "get_weather"}` | Force the LLM to call a specific tool | Useful for testing a single tool or routing |
| Max tool calls limit | Use `max_tokens` and application loop control | Prevent infinite tool call loops by limiting turns | Your code should enforce a maximum number of tool call iterations |
| Tool result size | `content` string in `tool_result` block | The function output that the LLM will read | Keep results concise; large results consume tokens and may be truncated |
| Multiple tools | Array of tool definitions | Pass multiple tools in a single request | The LLM can choose among them or call several in parallel |
| Parallel tool calls | Multiple `tool_use` blocks in one response | The LLM can request multiple independent tool calls at once | Each block has a unique `id`; execute them independently and return results separately |

## How It Actually Works -- Step by Step

Understanding each part of the tool use lifecycle helps you build reliable, debuggable agents.

### Step 1: You Define the Tools

Every tool use interaction starts with tool definitions. You include them in the API request alongside your messages. The LLM reads these definitions before generating a response.

<AsciiDiagram id="tool-use-and-function-calling/tool-definition" title="Structure of a tool definition sent to the LLM API" content={`┌────────────────────────────────────────────────────────────┐

Each tool definition has three parts: a unique name, a description that tells the LLM what the tool does and when to use it, and an `input_schema` that describes the parameters. The `input_schema` follows JSON Schema format. You specify the type of each parameter, a description that helps the LLM fill it correctly, and a `required` array listing which parameters must be provided.

The description field is more important than you might think. The LLM uses the description to decide whether this tool applies to the current situation. A vague description like "weather tool" gives the LLM little guidance. A precise description like "Get the current weather for any city. Call this when the user asks about temperature, conditions, or forecast for a specific location" helps the LLM make the right choice.

Here is an example of how you send tools in the API request:

```python
import os
from anthropic import Anthropic

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=[
        {
            "name": "get_weather",
            "description": "Get the current weather for a city. Returns temperature, conditions, humidity, and wind speed.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "The name of the city, e.g. 'Tokyo' or 'New York'"
                    },
                    "units": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "Temperature unit preference"
                    }
                },
                "required": ["city"]
            }
        }
    ],
    messages=[
        {"role": "user", "content": "What is the weather in Tokyo?"}
    ]
)
```

The `tools` parameter accepts an array of tool definition dictionaries. Each dictionary must have `name`, `description`, and `input_schema` keys.

### Step 2: The LLM Decides to Call a Tool

When the LLM receives the request, it reads the user message and the tool definitions. It considers whether any of the tools would help answer the question. If the answer is yes, it generates a `ToolUseBlock` instead of a text response.

<AsciiDiagram id="tool-use-and-function-calling/tool-call-lifecycle" title="The LLM receives tool definitions and decides to produce a tool_use block" content={`┌────────────────────────────────────────────────────────────┐
│  LLM Decision Process                                         │
├────────────────────────────────────────────────────────────┤
│  1. Read user message: "What is the weather in Tokyo?"       │
│  2. Read tool definitions: [get_weather, send_email, ...]    │
│  3. Does any tool help answer this question?  --- YES       │
│  4. Which tool?  --- get_weather                             │
│  5. What arguments?  --- city="Tokyo", units="celsius"       │
│  6. Stop with stop_reason = "tool_use"                       │
│  7. Produce ToolUseBlock with id, name, input                │
└────────────────────────────────────────────────────────────┘`} />

The LLM's response has `stop_reason` set to `"tool_use"` instead of the usual `"end_turn"`. The `content` array contains one or more `ToolUseBlock` objects instead of text blocks. Each `ToolUseBlock` has:

- `id`: A unique identifier like `"toolu_abc123"` that you will use when returning the result
- `name`: The name of the tool to call, matching one of your definitions
- `input`: A dictionary of arguments that the LLM has populated based on the `input_schema`

Your code checks for `stop_reason` to decide what to do next:

```python
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=[...],  # tool definitions
    messages=[{"role": "user", "content": "What is the weather in Tokyo?"}]
)

if response.stop_reason == "tool_use":
    for block in response.content:
        if block.type == "tool_use":
            tool_name = block.name
            tool_args = block.input
            tool_id = block.id
            print(f"LLM wants to call: {tool_name}")
            print(f"With arguments: {tool_args}")
```

### Step 3: Your Application Executes the Function

The LLM cannot execute anything. It can only request. Your application receives the `ToolUseBlock`, validates the arguments, and runs the corresponding function.

The execution is entirely your responsibility. You map the tool name from the LLM to an actual function in your codebase. You pass the arguments from the `input` dict to the function. You handle errors, missing arguments, and edge cases.

```python
def get_weather(city: str, units: str = "celsius") -> str:
    """Mock weather function for demonstration."""
    weather_data = {
        "Tokyo": {"temp": 22, "conditions": "Partly cloudy", "humidity": 65},
        "New York": {"temp": 18, "conditions": "Sunny", "humidity": 55},
        "London": {"temp": 14, "conditions": "Rainy", "humidity": 80},
    }
    city_data = weather_data.get(city, {"temp": 20, "conditions": "Unknown", "humidity": 50})
    if units == "fahrenheit":
        city_data["temp"] = round(city_data["temp"] * 9/5 + 32)
    return f"Weather in {city}: {city_data['temp']}°{'F' if units == 'fahrenheit' else 'C'}, {city_data['conditions']}, Humidity: {city_data['humidity']}%"


if block.name == "get_weather":
    result = get_weather(**block.input)
```

Your function should return a string or a list of content blocks. The return value becomes the content of the `tool_result` block you send back to the LLM.

### Step 4: You Return the Tool Result

Once the function executes, you package the result into a `tool_result` content block and send it back to the LLM in a follow-up message. You must reference the original tool use block's `id` so the LLM knows which tool call this result belongs to.

```python
# Continue the conversation with the tool result
messages = [
    {"role": "user", "content": "What is the weather in Tokyo?"},
    {"role": "assistant", "content": response.content},
    {"role": "user", "content": [
        {
            "type": "tool_result",
            "tool_use_id": tool_id,
            "content": result
        }
    ]}
]

final_response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=[...],
    messages=messages
)

print(final_response.content[0].text)
```

The conversation history must include everything: the original user message, the assistant's response (which contains the `tool_use` block), and the user message that contains the `tool_result`. The LLM needs the full context to understand what happened and produce a coherent final response.

### Step 5: The LLM Incorporates the Result

When the LLM receives the `tool_result`, it reads the content and incorporates it into its reasoning. It treats the tool result as reliable information -- the equivalent of looking up a fact in a trusted database. It then generates a text response that answers the original question using the tool result.

When you receive multiple `tool_use` blocks in one response, you execute all of them independently -- they do not depend on each other. You then return all the results in a single follow-up message:

```python
if response.stop_reason == "tool_use":
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            if block.name == "get_weather":
                result = get_weather(**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })

    messages.append({"role": "assistant", "content": response.content})
    messages.append({"role": "user", "content": tool_results})

    final = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        tools=[...],
        messages=messages
    )
```

The LLM receives all tool results and synthesizes the information into a coherent answer that covers both cities.

## Cost, Limits, and Safety

Tool use adds capabilities, but it also adds complexity and cost. Understanding the trade-offs helps you design efficient systems.

### Token Cost of Tool Definitions

Every tool definition you send in the API request consumes tokens. The `name`, `description`, and `input_schema` are all part of the prompt. The more tools you define and the more detailed your schemas are, the more tokens you consume on every request.

For a typical tool with a name, a one-sentence description, and a schema with two or three parameters, the cost is roughly 50 to 100 tokens per tool definition. If you define ten tools with detailed schemas, you add 500 to 1000 tokens to every request. This does not sound like much, but in a conversation with fifty turns, that is 25,000 to 50,000 extra tokens just for the tool definitions.

Strategies to manage this cost:

- Only include tools that are relevant to the current conversation context. Do not pass every tool you have ever written.
- Keep descriptions concise but precise. Every word in the description costs tokens.
- Use `input_schema` properties sparingly. Only include parameters the LLM actually needs to set.

### Large Tool Schemas

If your tool accepts a complex object with many nested properties, the schema can grow rapidly. A schema with five nested levels and arrays of objects can easily consume 500+ tokens for a single tool. Consider whether the LLM truly needs to set all those parameters, or whether your application can set some of them automatically.

### Deep Recursion of Tool Calls

A tool call loop that does not terminate can consume significant tokens and time. If the LLM calls a tool, gets the result, calls another tool, and keeps going, you can burn through your token budget quickly. Always enforce a maximum number of tool call iterations in your application code:

```python
MAX_TOOL_CALLS = 10
tool_call_count = 0

while tool_call_count < MAX_TOOL_CALLS:
    response = client.messages.create(...)
    if response.stop_reason != "tool_use":
        break
    tool_call_count += 1
    # handle tool calls
```

### Input Validation

The LLM generates arguments based on your schema, but the arguments are only as reliable as the schema and the LLM's understanding of it. Always validate the arguments before passing them to your functions. The LLM might provide a string where you expect an integer, or omit a required parameter if your schema is ambiguous.

```python
def validate_and_call(block):
    tool_name = block.name
    args = block.input

    if tool_name == "get_weather":
        if "city" not in args or not isinstance(args["city"], str):
            return "Error: 'city' parameter is required and must be a string."
        if "units" in args and args["units"] not in ["celsius", "fahrenheit"]:
            args["units"] = "celsius"
        return get_weather(**args)

    return f"Error: Unknown tool '{tool_name}'."
```

### No Automatic Execution

The LLM cannot call tools on its own. It can only request. Your application is the gatekeeper. You decide which tools to honor, whether the arguments are valid, and whether the tool call is safe to execute. This separation is a security feature -- it prevents the LLM from executing arbitrary actions without oversight.

## When to Use Tool Use and When Not To

### Use Tool Use When

- You need real-time data from an external system: weather, stock prices, database records, user account information.
- You need to perform calculations or transformations that require precision: currency conversion, tax calculation, date math.
- You need to trigger side effects under user control: sending an email, creating a support ticket, updating a record.
- You need structured information retrieval where the query depends on the conversation context.

### Do Not Use Tool Use When

- You could achieve the same result with post-processing of the LLM's output. If the LLM can return JSON and you can extract the information from it, you do not need a tool.
- You want the LLM to generate and execute arbitrary code. Tool use is for calling known, pre-defined functions, not for executing arbitrary scripts. If you need code execution, use a dedicated sandboxed environment.
- The operation is trivial and always the same. If every conversation starts with the same lookup, consider including the data in the system prompt instead.

### Tool Choice Guidance

- **Auto mode** (`{"type": "auto"}`): Use this for general-purpose assistants where the LLM should decide whether a tool is needed. This is the default and works well for most use cases.
- **Any mode** (`{"type": "any"}`): Use this when every turn must produce a tool call. For example, a classification pipeline where every user message maps to a tool call, or a structured data extraction flow.
- **Specific tool** (`{"type": "tool", "name": "get_weather"}`): Use this for testing a single tool, or when you know exactly which tool the LLM should use regardless of the user's message. This is also useful in multi-agent architectures where a router agent has already decided which specialist tool to invoke.

### Tool Design Patterns

As you build more complex agents, these design patterns will help you structure your tools effectively.

**The single-responsibility pattern.** Each tool should do one thing and do it well. A tool that both fetches weather data and sends an email is harder to maintain and confuses the LLM. Split concerns into separate tools with clear names.

**The idempotency pattern.** Design tools so that calling them multiple times with the same arguments produces the same result. The LLM may call the same tool twice if it is uncertain, and idempotent tools prevent duplicate side effects like charging a customer twice.

**The summary pattern.** When a tool returns a large dataset, return a summary or the most relevant subset rather than the full data. The LLM does not need every row of a 10,000-row query result -- it needs the aggregate or the top few rows.

**The error-as-data pattern.** When a tool fails, return a structured error message as the tool result rather than raising an exception. The LLM can read the error, understand what went wrong, and attempt to correct the arguments on the next turn. This is more robust than crashing the agent loop.

## Common Pitfalls and Debugging Tips

Even experienced developers run into issues when integrating tool use for the first time. Here are the most common problems and how to diagnose them.

### Pitfall 1: The LLM Never Calls Any Tools

If your LLM always responds with text and never produces a `tool_use` stop reason, your tool definitions are likely too vague or missing key context. The LLM decides whether to call a tool based on the `description` field. If the description does not clearly tell the LLM when the tool applies, it will default to answering from its training data.

**Debugging steps:**

1. Read your tool descriptions as if you are the LLM. Would you know exactly when to call each tool based on the description alone?
2. Make descriptions actionable. Instead of "Gets weather data," write "Call this when the user asks about the current temperature, conditions, or forecast for a specific city location."
3. Include example use cases in the description. "For queries like 'What is the weather in Paris?', use this tool."
4. Reduce the number of tools. If you pass ten tools, the LLM has more options to consider and may choose none.

### Pitfall 2: The LLM Calls the Wrong Tool

The LLM selected the wrong tool from your list. For example, you have `get_weather` and `get_stock_price`, and the LLM calls `get_weather` when the user asks about stock prices.

**Debugging steps:**

1. Make your tool names more descriptive. `get_current_stock_price_by_ticker` is harder to confuse with `get_weather` than `price_tool`.
2. Review the descriptions for overlap. If both descriptions mention "real-time data," the LLM may conflate them.
3. Add `tool_choice: {"type": "tool", "name": "get_stock_price"}` to force the LLM to use a specific tool during testing. If it works in forced mode but fails in auto mode, the issue is in how the LLM chooses between tools.

### Pitfall 3: Tool Arguments Are Wrong

The LLM populates the `input` dict with incorrect or nonsensical values. For example, passing `city="sunny"` instead of `city="Miami"`.

**Debugging steps:**

1. Add detailed descriptions to each property in the `input_schema`. Tell the LLM what values are valid and what the property represents.
2. Use `enum` constraints wherever possible. If a parameter only accepts three values, list them in an `enum` array. This dramatically reduces incorrect values.
3. Validate arguments before calling the function and return a clear error message as the tool result. The LLM will read the error and attempt to correct the arguments on the next turn.
4. Test with `tool_choice: {"type": "any"}` to see if the LLM can populate arguments correctly when it is forced to use the tool.

### Pitfall 4: The Agent Loops Infinitely

The LLM calls a tool, gets the result, calls another tool, gets the result, and keeps going without ever producing a final text response. This can burn through your token budget and frustrate users.

**Debugging steps:**

1. Always enforce a maximum number of tool call iterations in your code. Start with a low limit like 3 or 5 and increase it as needed.
2. Check whether your tool results contain information that the LLM can use to make a decision. If every tool result says "try another tool," the LLM will keep looping.
3. Add a fallback response for when the loop limit is reached. The fallback should tell the user what was accomplished and what could not be done.
4. Monitor the `stop_reason` at each turn. If it is always `tool_use` and never `end_turn`, the LLM is stuck in a decision loop.

### Pitfall 5: Context Window Exceeded

Tool use conversations can grow large quickly. Each tool definition, each tool call request, and each tool result increases the conversation length. If your agent runs for many turns, you can hit the context window limit.

**Debugging steps:**

1. Track `message.usage.input_tokens` and `message.usage.output_tokens` after each API call. This tells you how close you are to the model's context limit.
2. Keep tool results concise. Return summaries instead of raw data dumps.
3. Consider whether older turns can be summarized and removed from the conversation history. For long-running agents, implement a sliding window that keeps only the most recent N turns and a summary of the rest.
4. Switch to a model with a larger context window if your use case requires many tool call turns.

### Pitfall 6: Rate Limiting and Throttling

Each tool call loop iteration makes one or more API requests. If your agent calls many tools per turn and runs many turns, you can hit API rate limits.

**Debugging steps:**

1. Add exponential backoff to your API calls. The Anthropic SDK includes built-in retry logic, but you may need to increase the retry limit for long-running agents.
2. Cache tool results that are likely to be the same across turns. If the LLM asks for the same data twice, return the cached result instead of making a new API call.
3. Consider using streaming to start processing tool results as they arrive, reducing the total wall-clock time of the agent loop.

## Mental Model Check

Test your understanding with these questions.

### Question 1

You send a request with a tool defined and the LLM responds with `stop_reason: "end_turn"` and a text block. What does this mean?

**Answer:** The LLM determined that no tool was needed to answer the question. It generated a text response directly. This is normal -- the LLM uses tools only when it decides they are necessary.

### Question 2

Your tool definition has `input_schema` with `"required": ["city"]`, but the LLM's tool use block has `input: {"country": "Japan"}` without a city. What went wrong?

**Answer:** The LLM may have misunderstood the schema or the description. The description of the `city` parameter may have been ambiguous, or the LLM may have conflated city and country. Check your parameter descriptions and ensure they clearly explain what value the LLM should provide.

### Question 3

You receive a response with three `tool_use` blocks. Can you execute them in any order?

**Answer:** If the tools are independent -- meaning none of them depends on the result of another -- you can execute them in any order, including in parallel. If one tool's output is needed as input to another, you must execute them sequentially in a multi-turn flow.

### Question 4

You enforce a maximum of 5 tool call turns. The agent has called 5 tools and still has not produced a text response. What should you do?

**Answer:** Return a final response to the user that says the agent could not complete the task within the limit. Optionally, return the partial results that were gathered so far. In your code, you should break out of the loop after the maximum is reached and generate a fallback response.

### Question 5

You define a tool with a 500-word description and a schema with 20 properties. How does this affect your API usage?

**Answer:** Every word in the description and every property in the schema consumes tokens on every request. This tool definition will add hundreds of tokens to the prompt on every turn, increasing cost and latency. Consider reducing the description to two sentences and including only the most commonly used properties in the schema. Less commonly used parameters can be optional with defaults.
