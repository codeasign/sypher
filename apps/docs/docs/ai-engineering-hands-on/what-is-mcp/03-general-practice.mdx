These exercises assume you have completed the guided practice exercise and have the filesystem MCP server registered with Claude Code. Work through the tiers in order — later exercises build on concepts from earlier ones.

## Tier 1 — Beginner

### Exercise 1: Explore server capabilities by listing tools without starting a session

**Task:** Before starting an interactive Claude Code session, use the CLI to inspect what tools the filesystem server exposes, and identify which ones can perform read operations versus write operations versus discovery operations, all without giving the model any input at all.

**Hint:** The `claude mcp get` command already shows tool names and descriptions. Categorize each of the nine tools into read, write, or discovery.

#### Solution

```bash
claude mcp get filesystem
```

```text
filesystem
  Status: ✓ Connected
  Command: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp
  Tools:
    read_file (Read the complete contents of a file from the file system)
    read_multiple_files (Read the contents of multiple files simultaneously)
    write_file (Write content to a file)
    create_directory (Create a new directory)
    list_directory (Get a listing of all files in a directory)
    move_file (Move or rename a file or directory)
    search_files (Recursively search for files and directories matching a pattern)
    get_file_info (Get file information and metadata)
    list_allowed_directories (Returns the list of directories that this server is allowed to work with)
```

**Read tools:** `read_file`, `read_multiple_files`, `get_file_info`

**Write tools:** `write_file`, `create_directory`, `move_file`

**Discovery tools:** `list_directory`, `search_files`, `list_allowed_directories`

This categorization matters for safety — you know at a glance which tools you would never pre-approve (the write tools) versus which ones are safe to allow freely (the discovery tools). The descriptions are the only signal the model has too, which is why clear, honest descriptions matter.

### Exercise 2: Search for a file across multiple directories

**Task:** Create a few test files at different levels of your project directory (for example, `src/utils/helper.py` and `tests/test_helper.py`), then ask Claude to find all Python files in your project without telling it the file names. Observe how the model uses `search_files` with a pattern to discover files you never mentioned.

**Hint:** The `search_files` tool takes a `pattern` argument — a glob-style string like `**/*.py`.

#### Solution

First, create test files at different directory levels:

```bash
mkdir -p src/utils tests
touch src/utils/helper.py src/utils/__init__.py tests/test_helper.py tests/__init__.py
```

In a Claude Code session, ask:

```text
Find all Python files in my project.
```

Claude will request to call `search_files` with a pattern like `**/*.py`. After approval, you will see:

```text
I found the following Python files in your project:
- src/utils/helper.py
- src/utils/__init__.py
- tests/test_helper.py
- tests/__init__.py
```

The model discovered files at multiple levels of the directory tree that you never explicitly named. It used the pattern `**/*.py` to match recursively, and the server returned paths relative to the allowed root.

### Exercise 3: Read multiple files in one tool call

**Task:** Using the same test files from Exercise 2, ask Claude to read the contents of two files at the same time, and observe how the server's `read_multiple_files` tool handles a batch request in a single round trip rather than making separate calls.

**Hint:** Phrase your request to explicitly ask for multiple files in one sentence.

#### Solution

In your Claude Code session, ask:

```text
Read src/utils/__init__.py and src/utils/helper.py and tell me what each one contains.
```

Claude will request to call `read_multiple_files` with a `paths` array containing both paths. The response will show both files' contents in a single result, and Claude will present them side by side. This is more efficient than making two separate `read_file` calls because it saves a round trip between the model and the server.

### Exercise 4: Compare tool descriptions to actual behavior

**Task:** Pick one tool from the filesystem server, read its description from `claude mcp get filesystem`, and then test whether the description accurately predicts what arguments the tool accepts. For example, does `get_file_info` return only metadata, or does it also return file contents?

**Hint:** Try calling the tool with different argument combinations and observe what the model reports back.

#### Solution

From the tool list, `get_file_info` is described as `"Get file information and metadata"`. Ask Claude:

```text
Show me the file info for mcp-test.txt.
```

Claude will call `get_file_info` and return metadata like:

```text
File: mcp-test.txt
Size: 67 bytes
Created: 2026-07-04T10:30:00.000Z
Modified: 2026-07-04T10:30:00.000Z
Type: file
```

The description is accurate — it returns metadata, not contents. Good tool descriptions like this one make it easy for the model to choose the right tool for the job. A misleading description, like calling this tool "file content reader," would cause the model to call it when it actually needs `read_file`, and the result would be useless.

## Tier 2 — Intermediate

### Exercise 5: Register the filesystem server with multiple allowed directories

**Task:** Remove your existing filesystem server and re-register it with two separate allowed directories — your project directory and a second directory of your choice (for example, a `~/Documents` or `~/Desktop` folder). Confirm the server can access files in both but still refuses access outside them.

**Hint:** The `@modelcontextprotocol/server-filesystem` package accepts multiple directory arguments on the command line.

#### Solution

Remove the existing server:

```bash
claude mcp remove filesystem
```

Register with two directories:

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp /home/you/Documents
```

Verify:

```bash
claude mcp get filesystem
```

```text
filesystem
  Status: ✓ Connected
  Command: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp /home/you/Documents
  Allowed directories:
    /home/you/projects/learn-mcp
    /home/you/Documents
```

In a Claude Code session, test that both directories are readable:

```text
Read the file mcp-test.txt in my project, and also list the files in my Documents folder.
```

Claude will call `read_file` for your project file and `list_directory` for Documents, and both should succeed.

Then test that access outside both is still blocked:

```text
List the files in /tmp.
```

The tool call will fail with an error message like:

```text
Error: Directory /tmp is not within allowed directories.
```

The server enforces all boundaries simultaneously — adding more allowed directories does not weaken any of them.

### Exercise 6: Compare stdio servers with different scope restrictions

**Task:** Register a second instance of the filesystem server under a different name with a narrower scope (for example, a subdirectory of your project like `src`). Start a Claude Code session and compare what each server can reach, then remove one and observe the effect on the remaining server.

**Hint:** You can have multiple MCP servers with the same underlying package but different configurations running simultaneously, as long as their names are different.

#### Solution

```bash
claude mcp add filesystem-src -- npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp/src
```

```bash
claude mcp list
```

```text
filesystem: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp - ✓ Connected
filesystem-src: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp/src - ✓ Connected
```

In a Claude Code session, ask:

```text
Read the file src/utils/helper.py.
```

Claude will choose one of the two servers that can reach this path. Both can reach it, so the model may pick either. Now ask:

```text
Read the file mcp-test.txt in the project root.
```

Only `filesystem` can reach the root — `filesystem-src` is restricted to `src/`. Claude will attempt the call through the appropriate server and succeed or fail depending on which it chooses. This demonstrates why MCP servers should be scoped as narrowly as possible: a server with a wider scope can satisfy more requests, but a server with a narrower scope is safer because it literally cannot touch anything outside its boundaries.

Remove the extra server:

```bash
claude mcp remove filesystem-src
```

### Exercise 7: Write a tool through Claude and verify the result outside of Claude

**Task:** Ask Claude to create a new file in your project using the `write_file` tool, then exit Claude Code and verify the file exists on disk with the exact content you requested. This confirms that the tool actually touches the real filesystem, not a sandboxed or simulated one.

**Hint:** Watch the permission prompt carefully — this is a write operation, which deserves more scrutiny than a read.

#### Solution

In a Claude Code session, ask:

```text
Create a file called greeter.py in my project that contains a Python function called greet that takes a name and returns "Hello, {name}!".
```

Claude will request to call `write_file` with the path and content. After you approve, the tool executes and Claude confirms the file was created.

Exit Claude Code:

```text
/exit
```

Verify the file on disk:

```bash
cat greeter.py
```

```python
def greet(name: str) -> str:
    return f"Hello, {name}!"
```

The file is real. It persists after the Claude Code session ends, and you can use it with Python or edit it with any editor. The `write_file` tool went through the full MCP round trip — model reasoning, JSON-RPC transport, server handler execution, real filesystem write — and the result is indistinguishable from you having written the file yourself.

Remove the test file when done:

```bash
rm greeter.py
```

## Tier 3 — Advanced

### Exercise 8: Design an MCP server architecture for a team of four engineers

**Task:** Your team of four engineers needs access to the following capabilities through their AI coding assistant: a shared Postgres database for querying user data, a shared Jira API for ticket lookups and updates, individual project filesystem access, and a shared internal documentation search. Each engineer works in their own project directory and has individual Jira credentials, but the database credentials are shared.

Design the MCP server architecture answering these questions:

- How many separate MCP servers would you define, and what would each one wrap?
- Which servers should be local (stdio) and which should be remote (HTTP/SSE)?
- How would you handle credentials: shared ones versus per-engineer ones?
- What scope restriction would you set on each server?

There is no single correct answer, but a strong approach considers these trade-offs.

#### Solution

A solid architecture separates concerns into four servers:

1. **Filesystem server** — local stdio, one per engineer, scoped to their individual project directory. Each engineer registers their own with their own allowed directory. No credentials needed. This is the simplest component.

2. **Database server** — could be local stdio with a shared configuration file checked into the repo containing only the connection host and database name, while the actual password comes from an environment variable (`${DB_PASSWORD}`). Each engineer sets `DB_PASSWORD` in their own shell profile. The server runs as a local process spawned by each engineer's Claude session.

3. **Jira server** — like the database server, local stdio per engineer, but each engineer's personal Jira API token goes into their own `${JIRA_TOKEN}` environment variable so credentials never appear in any shared file. The server wraps the Jira REST API.

4. **Documentation search server** — this is the best candidate for a remote (HTTP/SSE) server. If the team maintains an internal docs site, a single shared server running on a team-accessible host lets all four engineers query documentation without each one running their own copy. The trade-off: you must host and keep it running, but you gain a single source of truth for documentation search without per-engineer setup overhead.

The key design insight: keep servers with per-user state (files, credentials) local and per-user, and only elevate shared, stateless capabilities to a remote server where the hosting cost is justified by the elimination of duplicated setup.

### Exercise 9: Compare the MCP approach to a direct function call approach for a specific use case

**Task:** Your teammate proposes skipping MCP entirely for their filesystem reading needs. They want to write a single Python function that reads files and call it directly from a script, instead of installing the filesystem MCP server. Write a short analysis comparing the two approaches for a team of one versus a team of five, including setup time, maintenance burden, and security considerations.

**Hint:** Consider that MCP's process boundary provides isolation that a direct import does not, but that isolation also means additional overhead.

#### Solution

For a team of one, the direct approach wins on simplicity:

```python
def read_file(path: str) -> str:
    with open(path, "r") as f:
        return f.read()
```

This is five lines of code, no dependencies, no configuration files, and no protocol. If you are the only person who will ever use this, and you are only using it from one codebase, there is no good reason to wrap it in MCP.

For a team of five, the picture changes. Each engineer would either copy the helper function into their own project or import it from a shared library. If the helper function changes — adding encoding detection, error handling for binary files — every copy needs updating. More importantly, there is no permission system: the calling script has the same filesystem access as the user running it. A model-driven agent using this function could read or write any file the user can access, with no approval gate.

The MCP approach shifts the balance:

- **Setup time:** higher for one person (install npx, run `claude mcp add`), roughly the same for five people (the command is the same for everyone, and is documented once).
- **Maintenance burden:** lower for the team. When the package updates, each engineer re-runs `npx -y` and gets the new version. No shared library to update and coordinate.
- **Security:** higher for MCP. The process boundary means the server only accesses the directories you explicitly allow, and the permission gate means you approve every call before it executes. A direct function call has no such gate — it reads whatever path you give it.
- **Flexibility:** MCP wins here. The same filesystem server also works with Claude Desktop, IDE plugins, or any future MCP-compatible client. The direct function call only works in the codebase where you defined it.

The verdict: for a single developer on a single project, use the direct function call. For a team, or for any project where the filesystem access will be used by a model-driven agent rather than deterministic code, MCP is worth the setup overhead.

### Exercise 10: Trace the full MCP message flow for a tool call using the Inspector in CLI mode

**Task:** Instead of relying on an interactive session with Claude Code, use the MCP Inspector to send a `tools/call` message directly to the filesystem server and observe the exact JSON-RPC request and response, bypassing the model's reasoning entirely. This gives you a clear picture of what the wire format looks like without any AI involved.

**Hint:** The Inspector's `--cli` mode takes `--method tools/call`, `--tool-name`, and `--tool-arg` flags. Use `--tool-arg path=mcp-test.txt` to read the test file.

#### Solution

```bash
npx @modelcontextprotocol/inspector --cli npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp --method tools/call --tool-name read_file --tool-arg path=mcp-test.txt
```

```json
{
  "content": [
    {
      "type": "text",
      "text": "Hello from the filesystem MCP server. If you can read this, MCP is working."
    }
  ],
  "isError": false
}
```

This is exactly the same JSON structure the model would receive in a Claude Code session, shown to you directly without the model interpreting it. The `content` array holds the result, and `isError: false` signals success.

Compare this to what happens with an invalid path:

```bash
npx @modelcontextprotocol/inspector --cli npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp --method tools/call --tool-name read_file --tool-arg path=nonexistent.txt
```

```json
{
  "content": [
    {
      "type": "text",
      "text": "Error: File nonexistent.txt does not exist."
    }
  ],
  "isError": true
}
```

The same response shape, but with `isError: true` and an explanation in the text. The model would see this exact structure and use it to decide what to do next — try a different path, list the directory first, or tell you the file was not found.

Understanding this wire format is valuable for debugging. When a tool call behaves unexpectedly, you can isolate whether the problem is in the server (test it directly with the Inspector), in the model's reasoning (too vague a description), or in the client (not passing the right context). Without the Inspector, you can only observe the final output and guess which layer caused the issue.
