These exercises reinforce the agent loop concepts from the overview and the
practice exercise. Each tier increases in difficulty. Start with Beginner if
this is your first time working with agent loops, or jump to the tier that
matches your experience level.

Use the same setup as the practice exercise: Python 3.10+, the `anthropic`
SDK, and an `ANTHROPIC_API_KEY` environment variable. All exercises build on
the core loop pattern you already built.

```python
import os
from dotenv import load_dotenv
from anthropic import Anthropic

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))
```

## Tier 1: Beginner

These exercises focus on the fundamentals of the agent loop. You will build
simple loops, add basic tools, and observe how the loop structure works in
practice.

### Exercise 1.1: Count the Loop

Write a minimal agent loop that does nothing but print the iteration count on
each pass. The agent should have one tool: `echo(message: str)` that returns
the message it receives. Run the loop with `max_iterations=5` and observe how
the token usage grows each iteration even though no real work is happening.

```python
tools = [
    {
        "name": "echo",
        "description": "Echo back the message you pass to it",
        "input_schema": {
            "type": "object",
            "properties": {
                "message": {
                    "type": "string",
                    "description": "The message to echo"
                }
            },
            "required": ["message"]
        }
    }
]
```

**Check your understanding**: Why does the input token count increase even
though the agent is not accumulating useful information? Answer: The messages
array still appends every tool call and tool result, so the context grows
with each iteration even when the content is trivial.

**Expected outcome**: You see 5 iterations, each with increasing input token
counts. The agent calls `echo` repeatedly until hitting the iteration limit.

**Hint**: The agent may stop early with `end_turn` if it decides to answer
instead of calling the tool. If this happens, prompt it more explicitly, for
example: "Use the echo tool five times in a row."

### Exercise 1.2: Add a Calculator Tool

Extend your agent loop with a `calculate(expression: str)` tool that evaluates
mathematical expressions. Use Python's `eval()` function carefully -- only allow basic
arithmetic operators.

```python
import re

def calculate(expression):
    """Safely evaluate a math expression."""
    # Only allow digits, operators, parentheses, spaces, and decimal points
    if not re.match(r'^[0-9+\-*/(). ]+$', expression):
        return "Error: invalid characters in expression"
    try:
        result = eval(expression)
        return str(result)
    except Exception as e:
        return f"Error: {e}"
```

Ask your agent: "What is (15 + 7) * 3 - 22 / 2?"

**Check your understanding**: Look at the stop reason on iteration 1. Did the
model call the tool directly, or did it explain what it was going to do first?
The model often adds a thinking preamble before calling a tool. This is
normal and expected -- the model is reasoning about the problem before acting.

**Expected outcome**: The agent calls `calculate("(15 + 7) * 3 - 22 / 2")`,
gets a result, and returns the final answer in natural language.

### Exercise 1.3: Multi-Tool Agent

Give your agent three tools: `roll_die(sides: int)`, `flip_coin()`, and
`pick_random_number(min: int, max: int)`. Ask it: "Simulate rolling a 6-sided
die, then flipping a coin. Tell me both results."

```python
import random

def roll_die(sides):
    result = random.randint(1, sides)
    return f"Rolled a d{sides}: {result}"

def flip_coin():
    result = random.choice(["heads", "tails"])
    return f"Coin flip: {result}"

def pick_random_number(min_val, max_val):
    result = random.randint(min_val, max_val)
    return f"Random number between {min_val} and {max_val}: {result}"
```

**Check your understanding**: Does the agent call both tools in a single
response (batching), or does it call one tool per iteration? Look at the
response content length in the first iteration to see how many tool_use
blocks it contains.

**Expected outcome**: The agent calls two tools (roll_die and flip_coin) and
returns the combined results. This may happen in one iteration if the model
batches tool calls, or across two iterations.

**Stretch goal**: Ask the agent "Roll a d20 and tell me if the result is
greater than 15." Observe how the agent processes the tool result and makes a
conditional decision based on it -- this demonstrates the reasoning capability
of the loop.

### Exercise 1.4: Tool Result Dependent Logic

Give your agent a `search_knowledge_base(query: str)` tool that returns
predefined answers for a set of queries. Ask it a multi-part question that
requires following up on the initial answer.

```python
knowledge_base = {
    "capital of France": "Paris",
    "population of Paris": "2.1 million",
    "area of Paris": "105 square kilometers",
    "language spoken in France": "French",
    "currency of France": "Euro",
}

def search_knowledge_base(query):
    """Search the knowledge base for an answer."""
    query_lower = query.lower().strip()
    for key, value in knowledge_base.items():
        if key in query_lower or query_lower in key:
            return value
    return "No information found for that query. Ask something else."
```

Ask: "What is the capital of France? What is its population? What language do
they speak there?"

**Check your understanding**: How many iterations did the loop need? Did the
agent ask about all three facts in separate tool calls? This shows how the
loop enables the agent to build a complete picture by combining multiple
lookups.

**Expected outcome**: The agent calls `search_knowledge_base` three times (or
fewer if the agent batches), then produces a comprehensive answer combining
all three facts.

## Tier 2: Intermediate

These exercises add complexity to the loop itself, including token tracking,
checkpoints, and dynamic behavior.

### Exercise 2.1: Token Cost Tracker

Build a wrapper around your agent loop that calculates and displays the
estimated cost of each run. Use a pricing dictionary and print a cost summary
when the loop finishes.

```python
PRICING = {
    "claude-sonnet-4-20250514": {
        "input_per_million": 3.00,
        "output_per_million": 15.00,
    }
}

def calculate_cost(model, input_tokens, output_tokens):
    pricing = PRICING.get(model)
    if not pricing:
        return 0.0
    input_cost = (input_tokens / 1_000_000) * pricing["input_per_million"]
    output_cost = (output_tokens / 1_000_000) * pricing["output_per_million"]
    return round(input_cost + output_cost, 6)
```

Modify your loop to call `calculate_cost` after each iteration and accumulate
the total. Print a final report:

```
Cost Report for run_agent_loop
==============================
Model: claude-sonnet-4-20250514
Iterations: 3
Total input tokens:  2,499
Total output tokens:   216
Estimated cost: $0.0107
```

**Check your understanding**: Which costs more -- input or output tokens? For
Claude Sonnet 4, output tokens cost 5x more per token. But in an agent loop,
input tokens often dominate because the context grows with each iteration.
Calculate the ratio for your run.

**Expected outcome**: A clear cost report showing the per-iteration and total
cost breakdown.

**Stretch goal**: Add a warning when estimated cost exceeds $0.10 and ask for
confirmation before continuing to the next iteration.

### Exercise 2.2: Human-in-the-Loop Approval Gate

Add a pause mechanism to your agent loop. When the model calls a specific
tool (for example, `send_email`, `delete_file`, or `charge_credit_card`),
pause the loop and ask a human to approve the action.

```python
HUMAN_APPROVAL_TOOLS = [
    "send_email",
    "delete_file",
    "charge_credit_card",
    "modify_database",
]

def require_human_approval(tool_name, tool_args):
    """Return True if human approves, False otherwise."""
    print(f"\n*** HUMAN APPROVAL REQUIRED ***")
    print(f"Tool: {tool_name}")
    print(f"Arguments: {tool_args}")
    response = input("Approve this action? (y/n): ").strip().lower()
    return response == "y"
```

In your loop, before executing a tool, check if it is in the approval list:

```python
if block.name in HUMAN_APPROVAL_TOOLS:
    if not require_human_approval(block.name, block.input):
        result = f"Tool call {block.name} was rejected by a human operator."
        # Tell the model the action was rejected
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result,
            }],
        })
        continue
```

Test with tools like `send_email(to="admin@example.com", subject="Test",
body="This is a test email")` and `generate_report(format="pdf")`. Only
`send_email` should trigger the approval gate.

**Check your understanding**: What happens if the human rejects the tool call?
The agent receives the rejection as a tool result and can decide what to do
next. It might ask for clarification, try a different approach, or explain
why the action was needed. This is the Observe-Think-Act cycle working
through a rejection.

**Expected outcome**: The loop pauses on `send_email`, waits for human input,
and either proceeds or skips based on the response. Other tools run without
interruption.

### Exercise 2.3: Retry on Tool Failure

Modify your loop to handle tool execution errors gracefully. Instead of
crashing when a tool raises an exception, catch the error and return a
meaningful error message to the model so it can try a different approach.

```python
def execute_tool_with_retry(tool_name, tool_args, max_retries=2):
    """Execute a tool with retry logic."""
    last_error = None
    for attempt in range(1, max_retries + 1):
        try:
            result = execute_tool(tool_name, tool_args)
            return result
        except Exception as e:
            last_error = str(e)
            print(f"  Tool {tool_name} failed (attempt {attempt}/{max_retries}): {last_error}")
    return f"Error: Tool {tool_name} failed after {max_retries} attempts. Last error: {last_error}"
```

Now make one of your tools fail intermittently:

```python
def unreliable_weather(location):
    """Simulate an unreliable API that fails 40% of the time."""
    import random
    if random.random() < 0.4:
        raise ConnectionError("Weather API timed out. Try again.")
    return f"Weather in {location}: 22 degrees, sunny"
```

**Check your understanding**: Does the model try a different approach when
the tool fails, or does it call the same tool again with the same arguments?
The answer depends on the error message returned. If the error says "try
again", the model will retry. If it says "location not found", the model will
try a different location or give up.

**Expected outcome**: The loop handles tool failures without crashing. The
model receives the error in the tool result and decides how to proceed.

### Exercise 2.4: Iteration Budget Based on Complexity

Instead of a fixed `max_iterations`, implement a dynamic budget that
allocates more iterations to complex tasks. Use the user request length as a
proxy for complexity.

```python
def calculate_budget(user_request, min_iterations=3, max_iterations=25):
    """Allocate iterations based on request complexity."""
    word_count = len(user_request.split())
    # Simple request: 1-10 words -> 3 iterations
    # Medium request: 10-30 words -> 8 iterations
    # Complex request: 30+ words -> 15 iterations
    if word_count < 10:
        return min_iterations
    elif word_count < 30:
        return 8
    elif word_count < 100:
        return 15
    else:
        return max_iterations
```

Test with three requests of increasing complexity:

- Short: "What is the weather in Paris?"
- Medium: "What is the weather in Paris, London, Tokyo, and Sydney? Then tell me which city has the best weather."
- Long: A paragraph describing a multi-step research task

**Check your understanding**: Does the short request actually use all 3
allocated iterations, or does it finish earlier? The iteration budget is a
maximum, not a target. Most simple tasks finish in 1-2 iterations regardless
of the budget.

**Expected outcome**: The iteration budget adapts to the request. Simple
queries complete early within a small budget. Complex queries get more room
to explore.

## Tier 3: Advanced

These exercises push the agent loop to its limits, addressing real-world
challenges like context management and adaptive behavior.

### Exercise 3.1: Context Window Trimmer

Build a context trimming system that summarizes early conversation history
when the token count approaches the context window limit. This is essential
for agents that run many iterations.

Implement a `trim_context` function that:

1. Counts total tokens in the messages array (use a simple estimate: 4
   characters per token)
2. When the total exceeds 150,000 estimated tokens, triggers a trim
3. Keeps the last 5 messages (tool calls and results) as-is
4. Summarizes everything before those into a single message
5. Returns the trimmed messages array

```python
def estimate_tokens(messages):
    """Rough estimate of token count for a messages array."""
    total = 0
    for msg in messages:
        content = str(msg.get("content", ""))
        if isinstance(content, list):
            for block in content:
                total += len(str(block.get("content", ""))) // 4
                total += len(str(block.get("text", ""))) // 4
        else:
            total += len(content) // 4
    return total


def trim_context(messages, max_tokens=150000, keep_recent=5):
    """Trim messages by summarizing older ones when context gets too large."""
    if estimate_tokens(messages) < max_tokens:
        return messages, False

    print(f"Trimming context: {estimate_tokens(messages)} tokens > {max_tokens} limit")

    # Keep the most recent messages
    recent = messages[-keep_recent:] if len(messages) > keep_recent else messages
    # Summarize the older messages
    older = messages[:-keep_recent] if len(messages) > keep_recent else []

    if not older:
        return messages, False

    # Ask the model to summarize the older conversation
    summary_prompt = (
        "Summarize the conversation so far in 2-3 sentences. "
        "Focus on what has been accomplished and what is still pending."
    )
    summary_response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        messages=older + [{"role": "user", "content": summary_prompt}],
    )

    summary = summary_response.content[0].text
    trimmed = [
        {"role": "assistant", "content": f"[Previous conversation summary: {summary}]"}
    ] + recent

    print(f"Trimmed context to {estimate_tokens(trimmed)} tokens")
    return trimmed, True
```

Integrate `trim_context` into your loop so it runs before each API call. Test
by creating a task that generates many tool calls (for example, "Call the
echo tool 20 times").

**Check your understanding**: What information is lost during summarization?
The raw tool results are replaced by a summary, so fine-grained details from
early tool calls are no longer available. If the agent needs to refer back to
a specific detail from early in the conversation, it cannot -- it only has
the summary.

**Expected outcome**: The loop runs many iterations without hitting the
context window limit. When the token count approaches the threshold, the
messages array is summarized and trimmed, and the loop continues.

**Stretch goal**: Instead of a single summary, implement a hierarchical
approach. Summarize the oldest messages into a summary block, then when that
grows too large, summarize multiple summary blocks into a higher-level
summary.

### Exercise 3.2: Dynamic Tool Addition

Build an agent loop that can add new tools at runtime based on the model's
requests. The model can define a new tool by calling a special
`create_tool(name, description, implementation_code)` function, and the loop
registers it for future use.

```python
# This tool lets the model create new tools on the fly
dynamic_tools = []

register_tool_schema = {
    "name": "register_tool",
    "description": "Create a new tool that the agent can use. Provide a name, description, and Python expression for the implementation.",
    "input_schema": {
        "type": "object",
        "properties": {
            "tool_name": {
                "type": "string",
                "description": "The name of the new tool (snake_case)"
            },
            "description": {
                "type": "string",
                "description": "A clear description of what the tool does"
            },
            "implementation": {
                "type": "string",
                "description": "A Python expression that takes **kwargs and returns a string result"
            }
        },
        "required": ["tool_name", "description", "implementation"]
    }
}
```

When the model calls `register_tool`, parse the implementation, wrap it in a
callable function, and add it to the tools list for future API calls.

**Check your understanding**: What are the security implications of letting
the model define code that executes? The implementation string is passed to
`eval()` or `exec()`, which can run arbitrary code. For a production system,
you would need a sandboxed execution environment. This exercise is for
understanding the flexibility of dynamic tool systems, not for production use.

**Expected outcome**: The model can create a new tool (for example, a
`convert_currency` tool) and then use it in subsequent iterations of the same
loop.

**Stretch goal**: Add a `list_tools()` function that returns all currently
registered tools, so the model can discover what it has already created.

### Exercise 3.3: Parallel Tool Execution Batching

The Anthropic API supports multiple tool calls in a single response. Modify
your loop to handle multiple `tool_use` blocks in one response by executing
them in parallel using `concurrent.futures`.

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def execute_tools_parallel(tool_calls):
    """Execute multiple tool calls in parallel and return results."""
    results = []
    with ThreadPoolExecutor(max_workers=5) as executor:
        future_to_tool = {
            executor.submit(execute_tool, block.name, block.input): block
            for block in tool_calls
        }
        for future in as_completed(future_to_tool):
            block = future_to_tool[future]
            try:
                result = future.result()
                results.append((block, result))
            except Exception as e:
                results.append((block, f"Error: {e}"))
    return results
```

In the loop, instead of handling tool calls one at a time, collect all
`tool_use` blocks from the response and execute them in parallel:

```python
if response.stop_reason == "tool_use":
    tool_blocks = [block for block in response.content if block.type == "tool_use"]

    # Execute all tools in parallel
    results = execute_tools_parallel(tool_blocks)

    # First append the assistant message with all tool_use blocks
    messages.append({
        "role": "assistant",
        "content": tool_blocks,
    })

    # Then append all tool results
    for block, result in results:
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result,
            }],
        })
```

Test with a request that naturally leads to multiple independent tool calls:
"What is the weather in Tokyo, London, San Francisco, and Sydney?"

**Check your understanding**: Does the parallel execution reduce total wall
clock time compared to sequential execution? Yes -- if each tool call takes 1
second, sequential execution of 4 tools takes 4 seconds, while parallel
execution takes approximately 1 second (plus overhead).

**Expected outcome**: All four weather lookups are executed in parallel,
reducing the time to complete the request. The model receives all results and
produces a combined answer.

## Putting It All Together

Once you complete all three tiers, you will have built:

- A working agent loop with proper stop reason handling
- Token usage tracking and cost estimation
- Human-in-the-loop approval gates for sensitive actions
- Error resilience with retry logic
- Dynamic iteration budgets
- Context window trimming for long-running agents
- Parallel tool execution for performance
- Dynamic tool registration

These are the same capabilities that production agent systems implement. The
difference is that you built them yourself, line by line, which means you
understand exactly how they work and can debug them when something goes
wrong.

The agent loop is not a framework feature. It is a fundamental pattern. What
you built in these exercises is the foundation of every agent system you will
encounter in your career.
