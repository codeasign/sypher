You spend three days building an agent. You test it manually five times. Every time it calls the right tool, formats the output correctly, and returns a perfect result. You push it to production feeling confident. On the fourth request -- the one from your most important customer -- the agent calls a tool with the wrong parameters, returns garbage, and you have no record of what happened. No logs. No trace. No way to replay the conversation. The customer is frustrated, your manager is asking questions, and you are staring at an empty terminal wondering what your agent actually did.

That is the nightmare of running a black-box agent. You built something that makes autonomous decisions, but you did not build the instrumentation to understand those decisions after the fact. When an agent works, it feels like magic. When it fails, it feels like betrayal -- because you cannot see inside its head.

The problem gets worse as your agent grows. A single-turn tool call is easy to debug: you see the input and the output. But a multi-step agent that calls five tools across three loop iterations, making decisions at each step based on previous results -- that is a chain of reasoning that disappears the moment the response is returned. If step three goes wrong, was it step three itself, or a bad decision at step two that poisoned step three? Without evaluation and debugging infrastructure, you cannot answer that question.

Consider a real scenario. Your agent is a customer support bot that handles refund requests. It has access to three tools: lookup order, check refund policy, and issue refund. In testing, it always calls lookup order first, then checks the policy, and only calls issue refund if the policy allows it. In production, on a specific order with a partial refund and a promo code discount, the agent calls issue refund without checking the policy. The refund is processed incorrectly. The order total was calculated with a discount, and the agent refunds the full amount instead of the discounted amount. You lose money, and you did not even know it happened until the finance team flagged it three weeks later.

The root cause is subtle. The system prompt said "check the refund policy before issuing a refund," but the agent interpreted "check the policy" as checking the policy duration window, not the refund amount calculation. A subtle ambiguity in the prompt that never surfaced in testing because the test cases all used simple orders. Without logging every tool call, its inputs, and its outputs -- and without an evaluation pipeline that tests edge cases -- you would never catch this.

The failure is not the agent's fault. It is the fault of the system you built around the agent. You shipped an agent without observability, without evaluation, without a test suite. You treated it like a deterministic function, but it is a stochastic decision-maker. Every input can produce different outputs, and you need the infrastructure to measure that variability.

Another scenario to consider: a code generation agent that produces unit tests. You give it a Python function and ask it to generate test cases. In testing, it always produces correct, runnable tests. In production, on a function with a complex decorator chain, the agent generates tests that import modules that do not exist. The tests fail at import time, and the CI pipeline breaks. The developer assigned to fix the tests spends an hour debugging the generated code, then disables the agent entirely. The agent was never evaluated on decorated functions, so the failure mode was invisible until it hit production.

These scenarios share a common pattern. The agent works on the cases you tested, fails on the cases you did not test, and you have no way to know which cases are missing from your test coverage. The only defense is systematic evaluation: a structured process that measures agent behavior across a representative set of inputs, identifies failure modes, and tracks improvements over time. Without evaluation, you are flying blind. You are deploying a system that makes autonomous decisions, and you have no instrument panel.

## The Concept

Evaluating and debugging agents means systematically measuring agent performance, tracing decision paths, identifying failure modes, and iterating -- so you can trust what your agent does when you are not watching.

Think of it like a flight data recorder on an aircraft. Every commercial airplane has a black box that records every instrument reading, every control input, every communication, for the entire flight. When something goes wrong, investigators do not guess. They pull the black box, replay the data, and pinpoint exactly where the chain of events went wrong. The black box is not there because planes crash often. It is there because when they do crash, you need to know why.

Your agent needs the same thing. Not because you expect it to fail constantly, but because when it does fail, you need to understand the failure. You instrument your agent with logging, tracing, and evaluation metrics so that every decision is recorded, every token is accounted for, and every failure can be classified and fixed.

The evaluation part is the maintenance schedule. An aircraft does not just fly until something breaks. It has a maintenance schedule: after every flight, after every 100 flight hours, after every 1000 cycles. Each inspection checks specific systems against defined thresholds. Your agent should have a similar cadence: evaluate every response against quality criteria, run regression tests after every prompt change, and periodically audit trajectories against a held-out test set.

The debugging part is the diagnostic toolkit. When a component fails on an aircraft, mechanics do not replace parts at random. They follow a diagnostic flowchart: check power, check signal, check mechanical linkage, check control surface. Your agent debugging should follow the same pattern: check the log, replay the trajectory, classify the failure, isolate the root cause, fix it, and confirm the fix with a re-run.

The combination of evaluation and debugging forms a continuous improvement cycle. You do not just evaluate once and deploy. You evaluate, find failures, debug them, fix them, re-evaluate to confirm the fix, and then deploy. And you keep evaluating after deployment because production traffic will always find edge cases your test suite missed. Every production failure becomes a new test case, expanding your dataset and making the agent more robust over time.

## API and SDK Reference

The following concepts and tools are the building blocks of any agent evaluation and debugging system.

| Concept | Purpose | Key Metrics or Tools |
| --- | --- | --- |
| Trajectory evaluation | Score whether the agent took the optimal path through tools and reasoning steps | Path efficiency, unnecessary tool calls, loop iterations count |
| Task completion rate | Percentage of tasks where the agent achieved the stated goal | Success rate, partial completion rate, failure rate |
| Tool call accuracy | How often the agent calls the correct tool with the correct parameters | Correct tool rate, correct parameter rate, hallucinated tool rate |
| Cost per task | Total token cost to complete a single agent run | Input tokens, output tokens, cache hits, total USD |
| Latency per loop iteration | Time taken for each observe-think-act cycle | Time to first tool call, time between iterations, total wall time |
| Logging and tracing | Recording every step of the agent loop for post-hoc analysis | JSON log files, structured events, OpenTelemetry spans, Langfuse traces |
| Evaluation datasets | Curated sets of test cases with expected outcomes | Test case count, coverage by tool, edge case coverage |
| A/B testing agent prompts | Compare two prompt variants against the same test set | Win rate, regression rate, statistical significance |

### Trajectory Evaluation

Trajectory evaluation scores the path the agent took through the problem space. Did it call the right tools in the right order? Did it make unnecessary intermediate calls? Did it loop more times than needed? This is the most informative evaluation because it reveals process failures, not just outcome failures. An agent that reaches the right answer through a wildly inefficient path will eventually break on a harder problem.

To implement trajectory evaluation, you define an expected sequence of tool calls for each test case. The evaluator compares the actual sequence against the expected sequence and computes a match score. Exact matches score 1.0. Partial matches (correct tools in wrong order) score lower. Paths with extra tools score a penalty for each unnecessary call.

The trajectory score formula you can use in your code: for each expected tool in the expected sequence, check if the agent called it at the corresponding position. Count matches, divide by total expected tools. Subtract a penalty for any unexpected tool calls. The final score ranges from 0.0 (completely wrong path) to 1.0 (perfect path with no extra steps).

Trajectory evaluation is especially valuable for detecting shortcut behavior. An agent might learn to skip a validation step because in your test data the validation always passes. When a real production case hits the validation step with invalid data, the agent has no experience handling it and fails. Trajectory evaluation catches this because the agent's path diverges from the expected path, even though the outcome might have been correct in the test data.

### Task Completion Rate

This is the simplest metric: did the agent do what you asked? Define a success criteria for each task (the user got their refund, the code was generated, the question was answered) and measure the fraction of tasks where the agent met the criteria. Be precise about what counts as success. Partial success matters too -- the agent might have answered the question but left out an important caveat.

Task completion rate is the first metric you should track because it is easy to measure and directly reflects user value. A task completion rate below 80 percent means your agent needs significant improvement before it is production-ready. Above 95 percent means you can focus on finer-grained metrics like trajectory efficiency and cost optimization.

When defining task completion, you need to operationalize the concept of "success." For a refund agent, success means the refund was processed and the correct amount was returned. For a research agent, success means the answer is factually correct and cites sources. For a code agent, success means the code compiles, passes tests, and follows project conventions. Write these criteria down explicitly in your evaluation dataset so every team member applies the same standard.

Partial completion scoring helps you distinguish between agents that almost work and agents that completely fail. A score of 0.7 on a refund task means the agent identified the right order but used the wrong refund amount. That is better than a score of 0.0 where the agent called the wrong tool entirely. The partial score tells you how close the fix is.

### Tool Call Accuracy

Your agent is only as good as its tool calls. Measure two things: did it call the right tool, and did it pass the right parameters? A tool call accuracy of 80 percent means one in five tool calls is wrong, which is almost certainly unacceptable in production. Track which tools are most often misused. If one specific tool is consistently called with wrong parameters, the tool description or the prompt needs fixing.

Break tool call accuracy down by tool. If the agent misuses `get_weather` once in a hundred calls but misuses `issue_refund` once in every ten calls, your debugging effort should focus on the `issue_refund` tool. The tool description may be ambiguous, the parameters may be poorly named, or the prompt may give conflicting instructions about when to use it.

Tool call accuracy has two components. Tool selection accuracy asks: did the agent pick the right tool from the available options? Parameter accuracy asks: did the agent pass the right arguments to that tool? These are independent metrics. An agent can have 100 percent tool selection accuracy (always picks the right tool) but 60 percent parameter accuracy (frequently passes wrong arguments). Track both separately and address them with different fixes. Tool selection issues are usually prompt-level problems. Parameter issues are usually tool-description-level problems.

To measure parameter accuracy, define the expected parameter values for each tool call in your test case. The evaluator compares the actual parameter values against the expected values. For numeric parameters, allow a tolerance. For string parameters, check for exact match or semantic match. For optional parameters, check that the agent only passes them when they are relevant.

### Cost per Task

Each agent run costs money. Track the token usage per run -- input tokens (the system prompt, conversation history, tool results) and output tokens (the assistant's reasoning and tool call blocks). Divide by task to get cost per task. Set a budget per task and alert when costs spike. A sudden increase in cost often means the agent is looping more times than expected or returning excessively long reasoning traces.

Cost tracking also reveals inefficiencies. If your agent spends 80 percent of its tokens on input (the conversation history growing with every loop iteration), you may need to implement conversation summarization or sliding window context management. If output tokens are unexpectedly high, the agent may be generating overly verbose reasoning traces that add latency and cost without improving quality.

### Latency per Loop Iteration

Time matters in production. Track the wall-clock time for each observe-think-act cycle. If a single iteration takes more than 10 seconds, users will notice. If the agent makes five iterations, that is 50 seconds of waiting. Measure latency and set thresholds. High latency often correlates with complex reasoning or large context windows.

Latency metrics help you decide when to optimize. If the LLM call itself is the bottleneck (low time-to-first-token), consider a faster model or prompt caching. If the tool execution is the bottleneck (high time-in-tool), optimize the tool implementation or add parallel tool execution. If the number of loop iterations is the bottleneck, improve the prompt to reduce unnecessary reasoning steps.

### Logging and Tracing

This is your agent's black box. Every step must be recorded: the user's input, the system prompt, the assistant's response (including tool call blocks), the tool results, the timing, and the token counts. Store these in a structured format (JSON Lines is popular) with timestamps, step numbers, and unique run IDs. For production systems, use a tracing platform like Langfuse, Arize, or OpenTelemetry to visualize the full decision path.

Logging is the foundation everything else depends on. Without logs, you cannot evaluate trajectories, track costs, detect loops, or debug failures. Start with file-based logging (JSON Lines files in a local directory) and graduate to a tracing platform when you need multi-service trace correlation, real-time dashboards, or team-wide access to logs.

### Evaluation Datasets

You cannot evaluate an agent without test cases. Build a dataset of realistic inputs with expected outcomes. Each test case should include the input, the expected tool calls (in order), the expected final output, and any edge case markers (boundary values, missing data, ambiguous requests). Start with 10 to 20 cases and grow to 100 or more as the agent becomes more complex.

A good evaluation dataset covers three categories: happy path cases (standard inputs that should work every time), edge cases (missing data, unusual values, boundary conditions), and adversarial cases (inputs designed to confuse or trick the agent). Each category reveals different failure modes.

### A/B Testing Agent Prompts

Every time you change a prompt, you risk regressions. Run the old prompt (A) and the new prompt (B) against your evaluation dataset and measure which one wins. A change is safe only if B performs at least as well as A on every metric. Track prompt versions explicitly in your logs so you can always trace which prompt produced which behavior.

A/B testing should be automated. Define a comparison script that runs both prompts against the same dataset, computes metrics for each, and produces a diff report. If the diff shows regressions, the new prompt should not be deployed until the regressions are fixed or accepted with explicit justification.

The evaluation and debugging workflow is a closed loop that feeds improvements back into the agent.

The loop starts with an agent run. Every step is logged -- user input, assistant response, tool calls, tool results, timing, and token counts. The logged data is evaluated against defined metrics: did the agent complete the task? Did it take the optimal path? Was it cost-effective? When a failure is identified, it is classified into a failure type. The root cause is isolated by replaying the trajectory. A fix is applied -- typically a prompt change, a tool description update, or a guardrail addition. The agent is re-run against the same test case, and the results are compared to the original run to confirm the fix worked without causing regressions.

## How It Actually Works -- Step by Step

Building evaluation and debugging infrastructure requires layering observability on top of your agent loop. Each step below adds a specific capability to your system. You should implement them in this order because each step depends on the previous one.

### Step 1: Logging Every Step

The foundation of all evaluation is logging. Every interaction with the LLM, every tool call, and every decision point must be recorded with enough detail to reconstruct the agent's state at any moment.

In practice, logging means writing a JSON object for each significant event in the agent loop. Each log entry should include a unique run ID, a timestamp, a step number, and the event data. Use a structured logging library or write directly to a JSON Lines file. The goal is to create a complete, replayable record of every agent interaction.

What should you log at minimum? Run ID to group all events from one agent invocation. Timestamp for each event to measure timing. Step number to reconstruct the sequence. Event type so you can filter by tool calls vs LLM responses vs final outputs. Token counts for cost tracking. Tool name, input, and output for trajectory reconstruction. Finish reason to detect truncations or refusals.

A typical log entry looks like this in practice:

```json
{
  "run_id": "run_abc123",
  "timestamp": "2026-07-04T14:30:00.123Z",
  "step_number": 3,
  "event_type": "tool_call",
  "tool_name": "lookup_order",
  "tool_input": {"order_id": "ORD-789"},
  "tool_output": {"status": "delivered", "amount": 49.99},
  "duration_ms": 234,
  "input_tokens": 1250,
  "output_tokens": 87,
  "loop_count": 2
}
```

Every log entry includes the contextual fields (run_id, timestamp, step_number) plus event-specific fields. The run_id links all entries from the same agent invocation. The step_number gives you ordering even if timestamps are slightly off due to clock skew or async writes.

### Step 2: Token Accounting

Tokens are your agent's fuel and your budget. You need to track input and output tokens for every LLM call per loop iteration.

The Anthropic API returns token usage in the response object under `usage.input_tokens`, `usage.output_tokens`, and optionally `usage.cache_read_input_tokens` and `usage.cache_write_input_tokens` when prompt caching is enabled. Sum these across all iterations in a run to get the total cost per task.

Track running totals so you can detect runaway loops early. An agent that suddenly consumes 10 times the expected tokens is likely stuck in a loop or processing an unexpectedly large context. Set a maximum token budget per run and terminate the agent if it exceeds the budget.

Store token counts per iteration and per run in your logs. This lets you answer questions like "which test case consumed the most tokens" and "did the token count increase after the last prompt change." Token tracking also reveals context window growth -- if the input tokens grow steadily across loop iterations, the conversation history is accumulating without pruning.

### Step 3: Trajectory Evaluation

Trajectory evaluation scores the path, not just the destination. An agent that reaches the right answer by calling five unnecessary tools is not a good agent -- it is an inefficient one that will fail under load or on edge cases.

Define an expected trajectory for each test case: the sequence of tools the agent should call, the parameters it should pass, and the decision rules it should follow. Then compare the actual trajectory against the expected trajectory. Score on exact match (same tools in the same order) and on partial match (correct tools but wrong order, or correct tools with extra steps).

Trajectory evaluation catches subtle failures that outcome evaluation misses. An agent that issues a refund without checking the policy produces the same outcome as one that checks the policy first -- if the policy allows the refund. But on the next test case where the policy disallows it, the shortcut becomes a disaster. Trajectory evaluation forces the agent to follow the correct process, not just reach the correct outcome.

Define your trajectory scoring function to penalize both missing steps and extra steps. A perfect trajectory matches the expected tool sequence exactly. A missing step (agent skipped a required tool call) is a major penalty. An extra step (agent called an unnecessary tool) is a minor penalty. Wrong order (agent called the right tools but in the wrong sequence) is a medium penalty.

In code, your trajectory evaluation function might look like this:

```python
def score_trajectory(actual_tools: list, expected_tools: list) -> float:
    matches = sum(1 for a, e in zip(actual_tools, expected_tools) if a == e)
    match_score = matches / max(len(expected_tools), 1)
    extra_penalty = max(0, len(actual_tools) - len(expected_tools)) * 0.1
    return max(0.0, match_score - extra_penalty)
```

This function gives a perfect 1.0 when the agent calls exactly the expected tools in order. It penalizes extra tool calls at 0.1 per extra call. It scores partial matches proportionally -- if the agent gets 3 out of 4 expected tools right, the match score is 0.75.

### Step 4: Outcome Evaluation

Outcome evaluation is simpler: did the agent achieve the user's goal? Define a success function for each test case that checks the final state.

For a refund agent, the outcome is successful if the refund was issued correctly. For a code generation agent, the outcome is successful if the generated code compiles and passes tests. For a question-answering agent, the outcome is successful if the answer contains the correct information.

Outcome evaluation is necessary but not sufficient. An agent can produce the right outcome through the wrong process, and that agent will fail on edge cases. Always pair outcome evaluation with trajectory evaluation.

Write your outcome evaluator as a function that takes the logged trajectory and returns a score from 0 to 1. A score of 1 means the task was completed perfectly. A score of 0.5 means partial completion -- the agent achieved the main goal but missed a subtask. A score of 0 means the agent failed entirely.

For outcome evaluation, you have several options. Keyword matching is the simplest: check if the final output contains certain words or phrases. LLM-as-judge is more sophisticated: ask another model to evaluate whether the output meets the criteria. Human review is the gold standard but does not scale. Use keyword matching for rapid iteration, LLM-as-judge for automated evaluation, and human review for spot checks before deployment.

### Step 5: Failure Classification

When an agent fails, classify the failure. This determines what fix to apply.

The six failure types and their common fixes break down as follows. You can implement failure classification as a function that takes the logged trajectory and the expected trajectory and returns the failure type.

```python
def classify_failure(actual: list, expected: list, error: str = "") -> str:
    if error:
        return "Tool Error"
    if not actual and expected:
        return "Refusal"
    if len(actual) > 3 and actual[-3:] == actual[-4:-1]:
        return "Loop"
    if any(t not in [e["tool"] for e in expected] for t in actual):
        return "Hallucinated Tool"
    if len(actual) != len(expected):
        return "Wrong Tool"
    for a, e in zip(actual, expected):
        if a["tool"] != e["tool"]:
            return "Wrong Tool"
        if a["params"] != e["params"]:
            return "Wrong Parameters"
    return "None"
```

This classification function checks failures in order of severity: tool errors first (hardware or infrastructure), then refusals (agent declined to act), then loops (stuck in repetition), then hallucinated tools (invented names), then wrong tool selection, and finally wrong parameters. Each failure type points to a different fix strategy.

**Tool Error**: The tool itself failed -- API timeout, authentication denied, rate limit exceeded. This is usually an infrastructure issue, not an agent issue. Fix by adding retries, improving error handling in the tool, or adjusting rate limits. Add retry logic with exponential backoff to your tool execution layer.

**Wrong Tool**: The agent called a tool that is technically valid but wrong for the current task. For example, calling issue_refund when it should have called lookup_order first. This is a prompt issue -- the agent did not understand the correct decision flow. Fix by clarifying the tool descriptions or adding decision rules in the system prompt.

**Hallucinated Tool**: The agent called a tool name that does not exist. This happens when the LLM misremembers the tool name from the system prompt. Fix by making tool names more distinctive or adding validation that checks tool names against a whitelist before execution. Whitelist validation should happen before any tool is executed.

**Loop**: The agent is stuck in a repeating cycle of tool calls. It calls the same tool with the same parameters, gets the same result, and calls it again. Fix by setting a maximum loop count and by designing tools that produce different results on repeated calls. Also consider adding a circuit breaker that terminates the loop after N identical consecutive tool calls.

**Refusal**: The agent declined to act, saying something like "I cannot process this request" or "I do not have permission to do that." This happens when the prompt guardrails are too strict or the task is genuinely outside the agent's scope. Fix by adjusting the system prompt's refusal guidelines or adding a tool that lets the agent ask for clarification.

**Wrong Parameters**: The agent chose the correct tool but passed incorrect parameters. This is common with tools that have complex parameter schemas. Fix by improving the tool description, adding parameter validation in the tool itself, or simplifying the parameter structure. Each parameter should have a clear description of what value to pass and when.

### Step 6: Regression Testing

Every time you change the agent's prompt, tool definitions, or system configuration, you risk breaking something that was working before. Regression testing catches these breaks before they reach production.

Maintain a test suite of evaluation cases. Each case has an input, an expected trajectory, and an expected outcome. After every change, run the entire test suite and compare the results against the baseline. A regression is when a test case that previously passed now fails.

│  │ Step 1: lookup_order(ORD-456)  ✓  order found      │              │
│  │ Step 2: check_refund_policy(ORD-456)  ✓  allowed  │              │
│  │ Step 3: get_discounted_amount(ORD-456)  ✓  $34.99 │              │
│  │ Step 4: issue_refund(amount=34.99)  ✓  discounted  │              │
│  │ Outcome: CORRECT                                   │              │
│  └────────────────────────────────────────────────────┘              │
└─────────────────────────────────────────────────────────────────────┘`} />

The before-and-after comparison is the heart of regression testing. Run the test suite against the old prompt (the baseline), then against the new prompt. Compare every metric: trajectory score, outcome score, cost, latency. A change is safe only if the new prompt does not regress on any metric.

Store the baseline results alongside your test suite. Use a versioned directory structure where each prompt version has its own evaluation results file. This lets you roll back to the previous prompt instantly if the new one causes regressions. Keep a changelog of what changed between prompt versions so you can trace which change caused a regression.

A practical approach is to store baselines as JSON files with the prompt version hash in the filename:

```text
baselines/
  baseline_prompt_v1.json
  baseline_prompt_v2.json
```

Each baseline file contains a dict mapping test case IDs to their trajectory scores, outcome scores, cost, and latency. The regression pipeline loads the latest baseline, runs the current agent, and computes the diff.

```python
def check_regression(current: dict, baseline: dict, threshold: float = 0.1) -> list:
    regressions = []
    for test_id, current_score in current.items():
        baseline_score = baseline.get(test_id)
        if baseline_score is not None:
            delta = current_score - baseline_score
            if delta < -threshold:
                regressions.append((test_id, baseline_score, current_score, delta))
    return regressions
```

When a regression is detected, the pipeline should block deployment and surface the regression details in a report: which test case regressed, what the baseline score was, what the current score is, and what the delta is. This report goes to the developer who made the prompt change, who can then investigate the root cause before re-attempting the deployment.

## Real-World Evaluation Example

To make these concepts concrete, consider a complete evaluation workflow for a customer support agent that handles refund requests.

### The Agent Setup

Your agent has three tools: `lookup_order(order_id)` returns order details, `check_refund_policy(order_id)` returns whether the order is eligible for a refund and the maximum refund amount, and `issue_refund(order_id, amount)` processes the refund. The system prompt instructs the agent to always look up the order first, then check the policy, and only issue a refund if the policy allows it.

### The Evaluation Dataset

You build a test suite with 10 cases. Three are happy-path cases: a standard order, a rush order, and a digital download. Three are edge cases: an order with a discount code, an order that is already refunded, and an order with a partial refund. Two are error cases: an order ID that does not exist and a tool that returns a timeout error. Two are adversarial cases: a user asking for a refund in a language the agent was not trained on and a user asking for a refund on an item that is explicitly non-refundable.

### Running the Evaluation

You run the agent against all 10 cases. The results show that 8 cases pass (score above 0.9), but two cases fail. The discounted order case scores 0.3 because the agent refunds the full amount instead of the discounted amount. The non-refundable item case scores 0.0 because the agent issues a refund despite the policy saying the item is non-refundable.

### Debugging the Failures

You examine the logs for the discounted order case. The trajectory shows: lookup_order (correct), check_refund_policy (correct), issue_refund with amount=49.99. The expected trajectory is: lookup_order, check_refund_policy, issue_refund with amount=34.99. The failure is classified as Wrong Parameters -- the agent chose the right tool but passed the wrong amount.

You examine the logs for the non-refundable item case. The trajectory shows: lookup_order (correct), issue_refund (skipped check_refund_policy). The failure is classified as Wrong Tool -- the agent called issue_refund when it should have called check_refund_policy first. The system prompt says "check the refund policy before issuing a refund," but the agent interpreted this as optional.

### Applying the Fixes

For the first failure, you update the tool description for `issue_refund` to say: "amount is the final refund amount after any discounts or promotions have been applied." This clarifies that the amount must be the discounted amount, not the full amount.

For the second failure, you update the system prompt to say: "You MUST call check_refund_policy before calling issue_refund. If the policy says the item is non-refundable, do not call issue_refund. Instead, explain to the user why the refund cannot be processed."

### Re-running and Comparing

You run the full test suite again with the updated prompt and tool descriptions. Both failing cases now pass. The discounted order case scores 1.0 (trajectory and outcome correct). The non-refundable item case scores 1.0 (agent correctly calls check_refund_policy, gets a denial, and explains to the user).

You compare the before-and-after metrics across all 10 cases. All 8 previously passing cases still pass. The average trajectory score improved from 0.85 to 0.97. The average outcome score improved from 0.82 to 0.96. The fix did not cause regressions.

This example illustrates the full evaluation loop: build a dataset, run the agent, identify failures, classify them, apply targeted fixes, and confirm the fixes work without regressions.

## Cost, Limits, and Safety

### Token Budgets and Cost Controls

Set a maximum token budget per agent run to prevent runaway costs. If your average task costs 5,000 tokens, set a budget of 20,000 tokens (4x the average) and terminate the agent if it exceeds the budget. This prevents a single stuck agent from burning through your API quota.

Track your evaluation costs per run and per day. If you run a 100-case test suite twice a day, each case averaging 5,000 tokens at roughly 3 dollars per million input tokens, your daily evaluation cost is approximately 3 dollars. Over a month, that is about 90 dollars. This is a reasonable cost for a production agent, but it should be budgeted explicitly so it is not a surprise.

Consider using a cheaper model for evaluation runs if you only need trajectory-level feedback and not perfect output quality. You can evaluate trajectories with a smaller model and reserve the full model for final outcome evaluation.

### Cost of Evaluation Runs

Running evaluation datasets costs money. Each test case requires at least one full agent run, and each run consumes input and output tokens. If your test suite has 100 cases and each case costs 5 cents, a full evaluation run costs 5 dollars. Run the full suite only when you change the prompt or tools.

For daily or hourly evaluation, use a smaller smoke test suite (5 to 10 cases) that covers the most critical paths. Run the full suite on every prompt change or before deployment. Consider using a cheaper model (like claude-sonnet-4-20250514 is already a good balance) for evaluation runs to reduce costs, but be aware that the evaluation may not perfectly reflect production behavior if you use a different model.

Be aware that evaluation itself introduces measurement overhead. The logging infrastructure adds latency to each agent step (writing to a file or sending to a tracing service takes time). In production, use async logging or batch writes to minimize the overhead. In evaluation, the overhead is acceptable because accuracy matters more than speed.

### Privacy Concerns with Logging User Data

Your agent logs will contain user inputs and possibly user data returned by tools. If you log everything without redaction, you create a privacy and compliance risk.

Never log raw passwords, API keys, social security numbers, or credit card information. If your tools process sensitive data, implement a redaction layer in the logging middleware that scrubs sensitive fields before writing them to the log file.

For GDPR, CCPA, or HIPAA compliance, you may need to delete logs after a retention period, obtain user consent before logging, or store logs in a specific geographic region. Consult your legal team for the requirements that apply to your use case.

### PII Redaction in Logs

Implement PII redaction as a filter function that runs before the log entry is written. Use regex patterns or a PII detection library to identify and replace sensitive values with placeholders.

```python
import re

def redact_pii(log_entry: dict) -> dict:
    entry = log_entry.copy()
    if "tool_input" in entry and isinstance(entry["tool_input"], str):
        entry["tool_input"] = re.sub(
            r"\b\d{3}-\d{2}-\d{4}\b", "[SSN_REDACTED]", entry["tool_input"]
        )
        entry["tool_input"] = re.sub(
            r"\b\d{16}\b", "[CC_REDACTED]", entry["tool_input"]
        )
    return entry
```

Apply the redaction before writing the log entry. Store the redacted version in your main log file. If you need the unredacted data for debugging, store it in a separate, access-controlled location with a short retention period and explicit audit logging of who accesses it.

### Staging vs Production Evaluation

Never evaluate a new prompt directly in production. Run all evaluations in a staging environment that mirrors production but serves no real users. The staging environment should have the same tools (pointed at test data), the same model configuration, and the same prompt.

Once the staging evaluation passes, deploy to production with canary or gradual rollout. Monitor production metrics for the first 24 hours at a reduced traffic level before full rollout.

Set up separate logging pipelines for staging and production. Staging logs can be verbose (log every detail for debugging). Production logs should be efficient (async, sampled, redacted) to minimize overhead and privacy risk. Production logs should also have a retention policy -- keep them for 30 to 90 days depending on compliance requirements, then archive or delete them.

## When to Use This and When Not To

### Heavy Evaluation for Production Agents

If your agent handles money, personal data, or critical infrastructure, invest in heavy evaluation. Run a full test suite (100 plus cases) before every deployment. Maintain trajectory baselines for every prompt version. Monitor cost and latency in real time. Set up automated alerts for metric regressions. Have a human review every prompt change before it reaches production.

### Light Evaluation for Prototypes

If you are building a prototype or a personal assistant that handles non-critical tasks, start with light evaluation. Log every step to a JSON file, define 5 to 10 test cases, and manually review the logs when something looks wrong. Add more evaluation infrastructure as the agent's responsibility grows.

The light approach is better than no approach. Many teams skip evaluation entirely because the full infrastructure feels overwhelming. Start with a single JSON log file and three test cases. That already puts you ahead of teams running black-box agents with no observability.

### Automated Evals vs Human Review

Automated evals catch process failures (wrong tool, wrong order) and quantitative regressions (cost spike, latency spike). Human review catches qualitative failures (wrong tone, misunderstood context, subtle errors in reasoning). Use automated evals as the first gate and human review as the second gate.

For high-stakes tasks, always have a human review the evaluation results before deploying a prompt change. For low-stakes tasks, automated evals with clear pass-fail thresholds are sufficient. The key is defining "stakes" explicitly: does a failure cause financial loss, data exposure, or reputational damage? If yes, involve a human.

### When the Overhead Is Worth It

Evaluation overhead is worth it when the cost of a failure exceeds the cost of the evaluation infrastructure. If one bad agent run loses 1,000 dollars, spending 50 dollars on evaluation is obvious. If one bad agent run costs 1 dollar in wasted time, spending 50 dollars on evaluation is harder to justify.

Start with the minimum evaluation that gives you confidence, and add more evaluation as the agent's scope and responsibility increase. The incremental approach works well: start with logging, add outcome evaluation, add trajectory evaluation, build a test suite, automate regression testing, and finally add a dashboard. Each step adds confidence without requiring the full infrastructure upfront.

### When Not to Use This

Do not invest in heavy evaluation infrastructure for a one-off script or a prototype you plan to discard. If your agent runs a single task on a single input and you can inspect the output manually, skip the evaluation pipeline. The overhead of building test suites, maintaining baselines, and running regression pipelines is not justified for code you will throw away.

Do not use automated evaluation as a substitute for human judgment on tasks that require subjective quality assessment. If your agent writes marketing copy, the quality of the output depends on brand voice, tone, and audience understanding -- metrics that are hard to automate. Use automated evals for structural checks and human review for quality.

Do not deploy evaluation infrastructure that collects more data than you can analyze. Logging every detail of every agent run is valuable only if you have the capacity to review the logs and act on the findings. If your logs grow faster than your ability to review them, you have created a data landfill, not a debugging tool. Implement sampling or alerting thresholds to focus your attention on the most important signals.

## Mental Model Check

Test your understanding with these questions.

**Question 1**: Your agent called the correct tool but passed the wrong parameter. What failure type is this, and what fix should you consider?

**Answer**: This is a Wrong Parameters failure. The fix is to improve the tool description so the LLM understands what each parameter means, add parameter validation in the tool itself, or simplify the parameter schema to reduce ambiguity. You may also need to add an example in the tool description showing the correct parameter format.

**Question 2**: You run your evaluation test suite and every test passes. Do you deploy the new prompt immediately?

**Answer**: No. Passing the test suite means the prompt does not regress on your known test cases, but it may still fail on cases you have not tested. Deploy gradually with monitoring. Also verify that the test suite covers edge cases, not just the happy path. A test suite with 100 percent coverage of known patterns still misses unknown patterns.

**Question 3**: What is the difference between trajectory evaluation and outcome evaluation?

**Answer**: Trajectory evaluation scores the path the agent took (the tools called, the order, the decisions made). Outcome evaluation scores whether the agent achieved the goal. An agent can produce the right outcome through the wrong path, so you need both. Always run both evaluations and flag cases where trajectory and outcome diverge.

**Question 4**: Your agent is stuck in a loop, calling the same tool with the same parameters and getting the same result. What two changes would you make?

**Answer**: First, add a maximum loop count to the agent loop so it stops after N iterations. Second, check whether the tool returns the same result every time -- if it does, the LLM has no new information to change its decision, and you need to make the tool stateful or add a different tool that provides context. Also consider adding a detection mechanism that terminates the loop after three identical consecutive tool calls.

**Question 5**: A user reports that the agent sometimes refunds the full amount instead of the discounted amount. Your test suite uses only simple orders without discounts. What do you do?

**Answer**: Add a test case with a discounted order to your evaluation dataset. Run it against the current agent to reproduce the failure. Classify the failure (likely Wrong Parameters) and fix the prompt or tool description to clarify how discount amounts should be calculated. Then verify the fix passes the new test case and does not regress on the existing cases. Update your deployment checklist to require test coverage for discount scenarios going forward.

**Question 6**: You add logging to your agent and discover that test case costs range from 0.02 dollars to 0.80 dollars. What does this tell you?

**Answer**: The variance is extremely high -- some test cases cost 40 times more than others. Investigate the expensive cases. They likely involve more loop iterations, larger tool results being fed back into the context, or more verbose reasoning traces. You may need to optimize those specific paths by improving the prompt to converge faster, truncating large tool results, or setting a token budget per run.

**Question 7**: Your evaluation dataset has 100 test cases, all of which are happy-path scenarios. Your agent scores 0.98 on the dataset but fails on the second production request. What went wrong?

**Answer**: Your evaluation dataset has no edge cases. The agent is optimized for the happy path and fails on any deviation. Add test cases that cover missing data, ambiguous inputs, boundary values, tool errors, and refusal scenarios. A good dataset should have roughly 60 percent happy-path cases, 30 percent edge cases, and 10 percent adversarial or failure cases. Without edge case coverage, your evaluation score is misleading.
