You just deployed a summarization feature. Your product manager sends you a test article: a 2,000-word earnings report. The summary your model produces says "the company's revenue grew by 15%." That is correct. The full report said revenue grew by 15% in North America but declined by 8% in Europe. The summary is technically true but misleading — it implies overall growth was 15% when the net was much lower. Traditional testing cannot catch this because the output is a valid, grammatical, and partially correct sentence. The only way to flag it is an evaluation that checks whether the summary fairly represents the full source.

Writing evals — evaluation tests — is the skill of turning fuzzy quality criteria into measurable, automated checks. This topic covers the specific techniques and patterns for building evals that catch real failures without producing false positives on every acceptable variation.

## The Concept

**An eval is a programmatic check that scores an LLM output against a quality dimension, then compares the score against a threshold to produce a pass or fail result.** Think of it like a spell-checker for quality: the spell-checker does not know whether your sentence is true, but it knows whether each word is spelled correctly. Similarly, a relevance eval does not check if the answer is correct — it checks whether the answer is on-topic. A factuality eval checks whether the claims in the output are supported by the source material. Each eval measures one dimension, and you compose multiple evals to get a complete picture.

## Eval Methods and Metrics Reference

| Method / Metric | What It Measures | How It Works |
|---|---|---|
| `AnswerRelevancyMetric()` from `deepeval` | How well the response addresses the prompt | Compares embedding similarity between response and prompt |
| `G-Eval` (chain-of-thought scoring) | Multi-dimensional quality using LLM-as-judge with CoT | LLM generates step-by-step reasoning, then outputs a score 1-5 |
| `FaithfulnessMetric()` from `deepeval` | Whether the response contains claims unsupported by the context | Splits response into claims, checks each against source |
| `HallucinationMetric()` from `deepeval` | Whether the model invented information | Claims extracted and verified against provided context |
| `BERTScore` (precision, recall, F1) | Lexical and semantic similarity to a reference | Token-level pairwise cosine similarity between reference and candidate |
| `ROUGE-L` (recall-oriented) | How much of the reference content is covered by the response | Longest common subsequence between reference and candidate |
| `Custom metric` via `deepeval` | Your own scoring logic | Subclass `DeepEvalMetric` and implement `measure()` |

## How It Actually Works — Step by Step

### Step 1: Split the Response into Atomic Claims

When you run a faithfulness eval, the first thing it does is decompose the LLM response into individual, checkable claims. For the response "Revenue grew 15% globally, driven by North American growth," the eval identifies two claims: (1) revenue grew 15% globally, (2) growth was driven by North America. Each claim becomes a separate unit of verification.

### Step 2: Retrieve Supporting Context

Each claim is checked against the source material. In a RAG system, the source is the retrieved documents. In a summarization task, the source is the original text. The eval locates the sentences in the source that are most relevant to each claim using embedding similarity search.

### Step 3: LLM Verification of Each Claim

For each claim-context pair, the eval sends a verification prompt to an LLM judge: "Does the source support the following claim? Answer yes or no and explain." This is a binary check per claim, not a score — it either passes or fails.

### Step 4: Aggregate into a Score

The final faithfulness score is the fraction of supported claims: `supported_claims / total_claims`. If 3 out of 4 claims are supported, the score is 0.75. This is why the score is always between 0.0 and 1.0 — it is a percentage, not an arbitrary scale.

### Step 5: Compare Against Threshold

The score is compared to your configured threshold. A threshold of 0.8 means at least 80% of claims must be supported. The eval outputs PASS or FAIL.

### Step 6: Log and Report

Every eval run produces a log entry with the score, the individual claim results, and the reasoning. This is critical for debugging — when an eval fails, you need to see which claims were unsupported and why.

## Cost, Limits, and Safety

**Eval cost per response:** Each eval call consumes tokens. A faithfulness eval with 5 claims generates roughly 5 judge prompts (one per claim) plus the context tokens. Estimate 500-1,000 input tokens per claim verification. For 100 responses with 5 claims each, that is 500 judge calls at roughly 500-750 tokens each.

**Choosing the right judge model:** Use a fast, cheap model (Haiku, GPT-4o-mini, Llama 3 8B) for initial screening. Reserve expensive judges (Sonnet, GPT-4) for edge cases and low-scoring samples. This tiered approach reduces eval costs by 60-80% compared to using a single expensive judge for everything.

**False positives from eval:** Evals themselves can be wrong. A faithfulness eval might flag a correct claim as unsupported because the judge model misunderstood the context. Always sample and manually review eval results before acting on them. The eval is a signal, not a verdict.

**Eval data leakage:** If you use the same model for both inference and evaluation, you risk the eval model being biased toward the inference model's style. Use a different model family for evaluation (e.g., Claude judges GPT, GPT judges Llama).

## When to Use This / When Not To

**Use structured evals when:** You need to track quality over time, catch regressions between model versions, or compare different prompt strategies. Evals are essential for any AI system that will be maintained beyond a single sprint.

**Do not use evals when:** You are prototyping and iterating rapidly. Early in development, manual review of 10-20 outputs is more informative than building a complex eval pipeline. Premature evaluation infrastructure slows down exploration.

**Do not use LLM-based evals for:** Deterministic checks (JSON validity, length limits, regex patterns). Use programmatic assertions for those — they are faster, cheaper, and more reliable. Reserve LLM-based evals for semantic quality dimensions.

## Mental Model Check

**Q1:** What is the key difference between a faithfulness eval and a relevance eval?

**A1:** Faithfulness checks whether the response is supported by the source material (claims are true). Relevance checks whether the response addresses the prompt (topic is correct). A response can be faithful but irrelevant, or relevant but unfaithful.

**Q2:** Why does a faithfulness eval decompose the response into individual claims instead of scoring the whole response at once?

**A2:** Scoring the whole response at once gives a single score that hides partial failures. The response might be 80% correct and 20% hallucinated. Decomposing into claims reveals exactly which parts are wrong and why.

**Q3:** What is the minimum dataset size for a reliable eval pipeline?

**A3:** For coarse quality signals, 20-50 prompt-response pairs can reveal major issues. For reliable regression detection with statistical significance, aim for 100-200 pairs. Smaller datasets produce high-variance scores that make it hard to distinguish real changes from noise.

**Q4:** How do you choose the right threshold for an eval metric?

**A4:** Start with a baseline run on known-good outputs and compute the 5th percentile score. Set the threshold at or slightly below that value. Then run on known-bad outputs and verify the threshold catches them. Adjust iteratively.

**Q5:** If your eval shows a score of 0.72 with threshold 0.8, is the system definitely broken?

**A5:** Not necessarily. The eval could be wrong (false positive), the test prompt could be unusually hard, or the judge model could be having a bad day. Always investigate low-scoring samples before taking action. A single eval failure is a flag, not a conclusion.
