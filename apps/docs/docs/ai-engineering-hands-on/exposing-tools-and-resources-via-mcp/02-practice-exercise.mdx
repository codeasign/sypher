## Goal

In this exercise you will build a Weather MCP server from scratch and extend it with tools, static resources, template resources, and a prompt template. You will then test everything using the MCP Inspector.

By the end you will have a working server that:

- Lists available weather regions as a static resource
- Returns forecast data for any city via a template resource
- Lets the LLM query forecasts with typed parameters via a tool
- Lets the LLM submit structured weather observations via a tool
- Provides a reusable prompt template for weather reports

## Setup

Create a new directory and set up a Python project:

```bash
mkdir weather-mcp-server
cd weather-mcp-server
python -m venv .venv
source .venv/bin/activate   # On Windows: .venv\Scripts\activate
pip install mcp httpx
```

Create a file named `weather_server.py`. This is where all your code will go.

## Step 1: Bootstrap the Server

Start with the minimal MCP server skeleton:

```python
import json
import random
from mcp.server import Server
from mcp.server.models import InitializationOptions

server = Server("weather-server")

if __name__ == "__main__":
    from mcp.server.stdio import stdio_server

    async def main():
        async with stdio_server() as (read_stream, write_stream):
            await server.run(
                read_stream,
                write_stream,
                InitializationOptions(
                    server_name="weather-server",
                    server_version="0.1.0",
                ),
            )

    import asyncio
    asyncio.run(main())
```

Run the server to verify it starts without errors:

```bash
python weather_server.py
```

It should sit there waiting for input on stdin. Press Ctrl+C to stop it.

## Step 2: Add a Static Resource

Register a static resource that returns a list of available weather regions:

```python
@server.resource(
    uri="weather://regions",
    name="Weather Regions",
    mime_type="application/json",
)
async def get_regions() -> str:
    regions = [
        {"id": "uk", "name": "United Kingdom"},
        {"id": "us", "name": "United States"},
        {"id": "eu", "name": "Europe"},
        {"id": "asia", "name": "Asia"},
        {"id": "au", "name": "Australia"},
    ]
    return json.dumps({"regions": regions})
```

Place this decorator right after `server = Server("weather-server")` and before the `if __name__` block.

Restart the server and test with a quick script. Create `test_regions.py`:

```python
import asyncio
from mcp.client.stdio import stdio_client
from mcp import ClientSession

async def main():
    async with stdio_client(["python", "weather_server.py"]) as (read, write):

```

Run it:

```bash
python test_regions.py
```

You should see the `weather://regions` resource listed.

## Step 3: Add a Tool -- get_forecast

Now add a tool that returns a simulated forecast for any city:

```python
@server.tool(name="get_forecast", description="Get the weather forecast for a city")
async def get_forecast(
    city: str = "London",
    days: int = 7,
    units: str = "metric",
) -> str:
    """Return a simulated forecast for the given city.

    Args:
        city: Name of the city to get forecast for.
        days: Number of forecast days (1-14).
        units: Temperature units, "metric" or "imperial".
    """
    if days < 1 or days > 14:
        return f"Error: days must be between 1 and 14, got {days}"
    if units not in ("metric", "imperial"):
        return f"Error: units must be 'metric' or 'imperial', got '{units}'"

    forecasts = []
    for day in range(1, days + 1):
        temp = random.randint(10, 35) if units == "metric" else random.randint(50, 95)
        conditions = random.choice(["Sunny", "Cloudy", "Rainy", "Windy", "Partly Cloudy"])
        forecasts.append({
            "day": day,
            "temperature": temp,
            "unit": "C" if units == "metric" else "F",
            "conditions": conditions,
        })

    return json.dumps({"city": city, "forecast": forecasts})
```

Notice the validation at the top of the handler. This is your safety layer: even though the JSON Schema constrains types, you add semantic validation for ranges and allowed values.

## Step 4: Add a Tool -- submit_weather_report

Add a second tool that accepts a structured weather observation:

```python
@server.tool(name="submit_weather_report", description="Submit a weather observation")
async def submit_weather_report(
    city: str,
    temperature: float,
    conditions: str,
    humidity: float = 50.0,
    wind_speed: float = 10.0,
) -> str:
    """Record a weather observation.

    Args:
        city: City where observation was taken.
        temperature: Observed temperature.
        conditions: e.g. Sunny, Rainy, Cloudy.
        humidity: Relative humidity percentage (0-100).
        wind_speed: Wind speed in km/h or mph.
    """
    if humidity < 0 or humidity > 100:
        return f"Error: humidity must be 0-100, got {humidity}"
    if wind_speed < 0:
        return f"Error: wind_speed must be non-negative, got {wind_speed}"

    # In a real server you would store this in a database
    report = {
        "city": city,
        "temperature": temperature,
        "conditions": conditions,
        "humidity": humidity,
        "wind_speed": wind_speed,
    }
    print(f"Report received: {json.dumps(report)}")
    return json.dumps({"status": "accepted", "report": report})
```

This tool demonstrates a side-effect operation: it accepts data from the LLM and persists it (here just printing to stdout, but in production you would write to a database or API).

## Step 5: Add a Template Resource

Now add a template resource that lets the LLM read forecast data by city URI:

```python
@server.resource(
    uri="weather://forecast/{city}",
    name="Weather Forecast for City",
    mime_type="application/json",
)
async def get_forecast_resource(city: str) -> str:
    """Return a 7-day forecast for the given city."""
    forecasts = []
    for day in range(1, 8):
        temp = random.randint(10, 35)
        conditions = random.choice(["Sunny", "Cloudy", "Rainy", "Windy", "Partly Cloudy"])
        forecasts.append({
            "day": day,
            "temperature": temp,
            "unit": "C",
            "conditions": conditions,
        })
    return json.dumps({"city": city, "forecast": forecasts})
```

The parameter `city` is extracted from the URI path `{city}` and passed as the function argument. The LLM can now read a forecast by accessing `weather://forecast/london` or `weather://forecast/tokyo`.

## Step 6: Add a Prompt Template

Add a reusable prompt that generates a weather report summary:

```python
@server.prompt(name="weather_report", description="Generate a weather report for a city")
async def weather_report_prompt(city: str) -> str:
    return f"""You are a professional meteorologist. Generate a detailed weather report for {city}.

Please include:
1. A summary of current conditions
2. The 7-day forecast in a table
3. Any weather advisories or warnings
4. Recommendations for residents

Use the get_forecast tool to fetch real data.
"""
```

This prompt template saves the LLM from having to construct this instruction from scratch every time. The user simply asks for a weather report, the LLM invokes the prompt, and the prompt reminds it to use the tool to fetch real data.

## Step 7: Assemble the Full Server

Here is the complete `weather_server.py`:

import random
import asyncio
from mcp.server import Server
from mcp.server.models import InitializationOptions

server = Server("weather-server")

@server.resource(
    uri="weather://regions",
    name="Weather Regions",
    mime_type="application/json",
)
async def get_regions() -> str:
    regions = [
        {"id": "uk", "name": "United Kingdom"},
        {"id": "us", "name": "United States"},
        {"id": "eu", "name": "Europe"},
        {"id": "asia", "name": "Asia"},
        {"id": "au", "name": "Australia"},

    return json.dumps({"regions": regions})

    uri="weather://forecast/{city}",
    name="Weather Forecast for City",
    mime_type="application/json",

    forecasts = []

        forecasts.append({
            "day": day,
            "temperature": temp,
            "unit": "C",
            "conditions": conditions,
        })

@server.tool(name="get_forecast", description="Get the weather forecast for a city")
async def get_forecast(
    city: str = "London",

            "temperature": temp,

## Step 8: Test with MCP Inspector

The MCP Inspector is a web-based tool that lets you connect to any MCP server and interactively test its tools, resources, and prompts.

Install the inspector:

```bash
npx @modelcontextprotocol/inspector
```

Start your weather server in one terminal:

```bash
python weather_server.py
```

In another terminal, start the inspector and point it at your server:

```bash
npx @modelcontextprotocol/inspector python weather_server.py
```

Open your browser to the URL shown by the inspector (usually `http://localhost:5173`). You should see:

1. **Resources tab:** Lists `weather://regions` and `weather://forecast/{city}`. Click on `weather://regions` to read the regions data. Try reading `weather://forecast/london` to see the template resource in action.

2. **Tools tab:** Lists `get_forecast` and `submit_weather_report`. Click on `get_forecast` to see its JSON Schema. Enter a city name and click "Call Tool" to execute it.

3. **Prompts tab:** Lists the `weather_report` prompt. Select it, provide a city, and see the generated prompt text.

## Step 9: Verify Everything Works

Run through this checklist to confirm all primitives are working:

1. **Static resource:** Read `weather://regions` and confirm you see the list of 5 regions.
2. **Template resource:** Read `weather://forecast/tokyo` and confirm you get a 7-day forecast.
3. **Tool with defaults:** Call `get_forecast()` with no arguments. It should return a 7-day forecast for London in metric units.
4. **Tool with arguments:** Call `get_forecast(city="Paris", days=3, units="imperial")`. Confirm 3 days in Fahrenheit.
5. **Tool with validation:** Call `get_forecast(days=20)`. Confirm you get the validation error message.
6. **Submit tool:** Call `submit_weather_report(city="Berlin", temperature=22.5, conditions="Sunny", humidity=45)`. Confirm it returns an accepted status.
7. **Prompt:** Generate the `weather_report` prompt for `"Sydney"`. Confirm the output includes the full meteorologist briefing instructions.

## Extension Ideas

If you finish early, try these enhancements:

1. **Real weather data.** Replace `random.randint` with calls to a real weather API like OpenWeatherMap or WeatherAPI. Store the API key in an environment variable.

2. **Persistence for reports.** Store submitted weather reports in a SQLite database and add a tool to query recent reports.

3. **Resource completion.** Add a completion callback to the `weather://forecast/{city}` template that suggests city names as the user types.

4. **Error handling.** Make the server return structured error responses instead of simple strings when validation fails.

5. **Logging.** Use the `Context` object to log every tool call and resource read with timestamps and parameters.
