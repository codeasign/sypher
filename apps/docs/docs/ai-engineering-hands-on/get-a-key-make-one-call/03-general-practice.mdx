These exercises reinforce the key-flow and first-call workflow you just completed. They are organised into three tiers. Tier 1 focuses on single concepts. Tier 2 combines multiple skills. Tier 3 presents real scenarios you might encounter when working with API keys and authentication.

Each exercise follows the same structure: a task description, a hint if you get stuck, and a complete solution with explanation. Attempt each exercise on your own before looking at the solution.

## Tier 1 — Beginner

### Exercise 1: Verify Your Environment Variable Is Set

**Task.** Before running any script that uses an API key, you should confirm the environment variable is set in your current terminal session. Write a one-line Python command (run directly in the terminal, not in a script file) that prints the value of `ANTHROPIC_API_KEY` if it is set, or prints "NOT SET" if it is not.

**Hint.** Use `python3 -c "..."` to run a short Python expression without creating a file. The `os.getenv()` function returns the value or `None` if the variable is missing, which you can check with `or`.

**Solution.**

```bash
python3 -c "import os; key = os.getenv('ANTHROPIC_API_KEY'); print(key if key else 'NOT SET')"
```

If the variable is set, you see the first few characters of your key printed. If not, you see `NOT SET`. This is a fast way to check your environment before running any script. The `os.getenv()` function is safer than `os.environ[]` here because it returns `None` instead of raising an error when the variable is missing.

You can also check from the terminal directly:

```bash
echo $ANTHROPIC_API_KEY
```

On Windows PowerShell:

```powershell
echo $env:ANTHROPIC_API_KEY
```

But the Python approach works identically on every operating system, which is why it is useful in scripts.

### Exercise 2: Send a Different Prompt

**Task.** Modify the script from the practice exercise so that instead of asking the model to say hello, you ask it a question you actually want answered. Pick any factual question — "What is the capital of Mongolia?", "Explain what a variable is in Python", "Who wrote The Great Gatsby?" — and run the script. Confirm you get a real answer.

**Hint.** Only the `content` string inside the messages list needs to change. Everything else stays the same.

**Solution.**

```python
import os
from dotenv import load_dotenv
import anthropic

load_dotenv()

api_key = os.environ["ANTHROPIC_API_KEY"]
client = anthropic.Anthropic(api_key=api_key)

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=200,
    messages=[
        {"role": "user", "content": "What is the capital of Mongolia?"}
    ]
)

print(response.content[0].text)
```

Expected output (approximate):

```
The capital of Mongolia is Ulaanbaatar.
```

The model returns a real, factual response. This proves you are not getting a canned greeting — the model is actually processing your prompt and generating a relevant answer. Try three different questions to see the range of responses.

### Exercise 3: Print the Full Response Object

**Task.** The `response` object contains more than just the generated text. Write a script that prints the entire response object (not just the text) so you can see its structure. Then identify which part of the object holds the text, which holds the token usage, and which holds the response ID.

**Hint.** Use `print(response)` or `print(vars(response))` to inspect the object. The `dir()` function lists all attributes.

**Solution.**

```python
import os
from dotenv import load_dotenv
import anthropic

load_dotenv()

api_key = os.environ["ANTHROPIC_API_KEY"]
client = anthropic.Anthropic(api_key=api_key)

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=50,
    messages=[
        {"role": "user", "content": "Say the word banana."}
    ]
)

print("=== Full Response ===")
print(response)
print()
print("=== Response attributes ===")
print(dir(response))
print()
print("=== Content text ===")
print(response.content[0].text)
print()
print("=== Token usage ===")
print(f"Input: {response.usage.input_tokens}")
print(f"Output: {response.usage.output_tokens}")
print(f"Response ID: {response.id}")
```

The output shows the full response object structure. You can see that `response.content` is a list of content blocks (usually one), each with a `text` attribute. The `response.usage` object has `input_tokens` and `output_tokens`. The `response.id` is a unique identifier for this API call, useful for debugging or contacting support about a specific request.

### Exercise 4: Handle a Missing Key Gracefully

**Task.** Write a script that checks whether the API key exists before making the API call. If the key is missing, print a helpful error message and exit instead of crashing with a `KeyError`. If the key exists, proceed with the API call.

**Hint.** Use `os.getenv("ANTHROPIC_API_KEY")` instead of `os.environ["ANTHROPIC_API_KEY"]`. If it returns `None`, print a message and call `exit(1)`.

**Solution.**

```python
import os
from dotenv import load_dotenv
import anthropic

load_dotenv()

api_key = os.getenv("ANTHROPIC_API_KEY")

if api_key is None:
    print("ERROR: ANTHROPIC_API_KEY is not set.")
    print("Create a .env file in this directory with:")
    print('  ANTHROPIC_API_KEY=sk-ant-...')
    print("Then run this script again.")
    exit(1)

client = anthropic.Anthropic(api_key=api_key)

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=50,
    messages=[
        {"role": "user", "content": "Say hello."}
    ]
)

print(response.content[0].text)
```

This pattern is called a guard clause. It checks the prerequisites before doing any real work. If the key is missing, the script tells you exactly what to do instead of showing a cryptic Python stack trace. This friendly error handling is standard practice in production code.

## Tier 2 — Intermediate

### Exercise 5: Switch to OpenAI

**Task.** The practice exercise used the Anthropic API. Now repeat the entire workflow with OpenAI instead. Sign up for an OpenAI account (or use an existing one), generate an API key, store it in your `.env` file as `OPENAI_API_KEY`, install the OpenAI SDK, and write a script that makes a call to the OpenAI API and prints the response.

**Hint.** The OpenAI Python SDK is installed with `pip install openai`. The client is created with `openai.OpenAI(api_key=...)`. The method is `client.chat.completions.create(model="gpt-4o", messages=[...])`. The response text is accessed via `response.choices[0].message.content`.

**Solution.**

First, install the OpenAI SDK:

```bash
pip install openai
```

Add the key to your `.env` file:

```
OPENAI_API_KEY=sk-your-openai-key-here
```

Then write the script:

```python
import os
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()

api_key = os.environ["OPENAI_API_KEY"]
client = OpenAI(api_key=api_key)

response = client.chat.completions.create(
    model="gpt-4o",
    max_tokens=100,
    messages=[
        {"role": "user", "content": "Say hello in one short sentence."}
    ]
)

print(response.choices[0].message.content)
```

Run it:

```bash
python3 my_openai_call.py
```

Expected output (approximate):

```
Hello! It's great to connect with you.
```

The structure is similar to Anthropic but with different naming: `chat.completions.create` instead of `messages.create`, and `response.choices[0].message.content` instead of `response.content[0].text`. The pattern — install SDK, create client, call method, read response — is identical across providers. This is the skill that lets you work with any LLM API.

### Exercise 6: Run the Same Prompt Against Two Providers

**Task.** Write a single script that calls both the Anthropic API and the OpenAI API with the same prompt. Print both responses side by side so you can compare them. Handle the case where one provider's key is missing but the other is present.

**Hint.** Use the guard clause pattern from Exercise 4 but separately for each key. Store each response in a variable and print both at the end. Make sure both keys are in your `.env` file.

**Solution.**

```python
import os
from dotenv import load_dotenv
import anthropic
from openai import OpenAI

load_dotenv()

anthropic_key = os.getenv("ANTHROPIC_API_KEY")
openai_key = os.getenv("OPENAI_API_KEY")

prompt = "Explain what an API key is in one sentence."

# Call Anthropic if key is available
if anthropic_key:
    anthropic_client = anthropic.Anthropic(api_key=anthropic_key)
    anthropic_response = anthropic_client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=100,
        messages=[{"role": "user", "content": prompt}]
    )
    print("=== Anthropic ===")
    print(anthropic_response.content[0].text)
else:
    print("Skipping Anthropic: ANTHROPIC_API_KEY not set")

print()

# Call OpenAI if key is available
if openai_key:
    openai_client = OpenAI(api_key=openai_key)
    openai_response = openai_client.chat.completions.create(
        model="gpt-4o",
        max_tokens=100,
        messages=[{"role": "user", "content": prompt}]
    )
    print("=== OpenAI ===")
    print(openai_response.choices[0].message.content)
else:
    print("Skipping OpenAI: OPENAI_API_KEY not set")
```

This script runs without crashing even if one key is missing. Try it with both keys set, then try it with only one key set. Notice how each model phrases the same concept differently. This multi-provider pattern is common in production systems that use one provider as a fallback for another.

### Exercise 7: Measure Response Time

**Task.** Modify your script to measure how long the API call takes, from the moment you call `client.messages.create()` to the moment you receive the response. Print the elapsed time in seconds. Run it three times and note whether the time is consistent.

**Hint.** Use `time.time()` before and after the API call. Subtract the start from the end to get the elapsed time.

**Solution.**

```python
import os
import time
from dotenv import load_dotenv
import anthropic

load_dotenv()

api_key = os.environ["ANTHROPIC_API_KEY"]
client = anthropic.Anthropic(api_key=api_key)

start = time.time()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=100,
    messages=[
        {"role": "user", "content": "What is 2 + 2?"}
    ]
)

elapsed = time.time() - start

print(response.content[0].text)
print(f"\nResponse time: {elapsed:.2f} seconds")
print(f"Input tokens: {response.usage.input_tokens}")
print(f"Output tokens: {response.usage.output_tokens}")
```

Expected output (approximate, time will vary):

```
2 + 2 equals 4.

Response time: 1.34 seconds
Input tokens: 12
Output tokens: 5
```

Response times typically range from 0.5 to 3 seconds for short prompts. Factors that affect response time include: model size (larger models are slower), your geographic distance from the server, the current load on the provider, and the number of output tokens. Tracking response time helps you set appropriate timeouts and manage user expectations.

## Tier 3 — Advanced

### Exercise 8: You Ran Out of Free Credits Mid-Session

**Scenario.** You have been experimenting with the Anthropic API for an hour. You made dozens of successful calls. Suddenly, every call fails with a 429 or 402 error. You check the console and see you have exhausted your free trial credits or hit your spending limit. You still need to test three more prompts for a project due tomorrow.

**Task.** Without spending real money (or minimal money), find a way to keep testing. You have three options: switch to a free provider, use a local model, or set up a new free trial account. Choose one approach, implement it, and make one successful call using the alternative setup.

**Solution.**

The quickest path is to switch to a free tier of a different provider. Here are three viable approaches:

**Option A: Use OpenAI's free trial.** If you have not used OpenAI before, sign up for a new account at platform.openai.com. New accounts get $5-10 of free credits. Add the key to your `.env` file as `OPENAI_API_KEY` and use the OpenAI script from Exercise 5.

**Option B: Use a local model with Ollama.** If you have a reasonably modern computer, install Ollama from ollama.com and run a small model locally:

```bash
# Install Ollama, then:
ollama pull llama3.2:1b
ollama run llama3.2:1b  # Interactive test
```

This runs entirely on your machine with no API key and no cost. The model is much less capable than Claude or GPT-4, but it is good enough to test your script structure and prompt logic.

**Option C: Set a lower spending limit.** If you have credits remaining but hit an automated cap, go to the Anthropic console billing section and increase or remove your spending limit. Set a hard cap of $5 so you control the maximum spend. Then continue testing.

For the actual implementation, the Ollama approach is most instructive because it shows that the request-response pattern works the same way with a local model:

```python
import requests
import json

response = requests.post(
    "http://localhost:11434/api/generate",
    json={
        "model": "llama3.2:1b",
        "prompt": "Say hello in one sentence.",
        "stream": False
    }
)

print(response.json()["response"])
```

No API key needed, no environment variables, no cost. The trade-off is quality: a 1-billion-parameter local model produces noticeably worse output than a frontier model. But for prototyping and learning, free is hard to beat.

### Exercise 9: A Key Was Accidentally Committed to a Public Repo

**Scenario.** You were working quickly, made a commit that included your `.env` file, pushed to a public GitHub repository, and realized five minutes later what happened. The key is now public. Anyone can find it by scanning your repo.

**Task.** Describe the immediate steps you must take, in order, to secure your account. Then simulate the fix: write the commands to revoke the key and generate a new one (using the provider console, which you describe), and update your local `.env` file.

**Solution.**

**Immediate steps in order:**

1. **Revoke the compromised key immediately.** Go to the Anthropic console (console.anthropic.com), navigate to API Keys, find the exposed key, and click Revoke or Delete. This invalidates the key instantly. Any further requests using that key will be rejected.

2. **Generate a new key.** In the same console section, create a new API key. Give it a name to distinguish it from the old one.

3. **Update your local `.env` file.** Replace the old key value with the new one.

4. **Remove the key from git history.** Force-remove the `.env` file from the repository history:

```bash
# Remove .env from git tracking (but keep local copy)
git rm --cached .env

# Rewrite history to remove .env from all commits
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch .env" \
  --prune-empty --tag-name-filter cat -- --all

# Force push the cleaned history
git push origin --force --all
```

**Important:** Even after rewriting history, the key was briefly public. Assume it was copied by automated scanners within seconds of your push. Revocation is the only reliable fix — history rewriting prevents future exposure but does not undo the damage.

5. **Check your billing page.** Go to the provider console billing section and scan for any unusual usage spikes during the window the key was exposed. If you see unexpected charges, contact support.

**Prevention for the future:**

Add `.env` to `.gitignore` before creating the file, not after. Use a pre-commit hook that scans for API key patterns and blocks commits that contain them. Store API keys in a password manager, not in a plain text file on your desktop.

### Exercise 10: Your Script Works Sometimes and Fails Other Times

**Scenario.** You wrote a script that calls the API every 2 seconds in a loop. Sometimes it works for 20 iterations. Sometimes it fails on iteration 3 with a rate limit error. The failures are intermittent and hard to reproduce.

**Task.** Diagnose the problem and redesign the script to handle it robustly. Write a version that retries automatically when it gets a rate limit error, waits an appropriate amount of time, and logs each attempt so you can see the pattern.

**Solution.**

The problem is that rate limits are not always exact. Free-tier accounts typically allow 5-50 requests per minute with a burst allowance. If you make 10 requests in quick succession, you may hit the limit even if you stay under the per-minute average. The fix is to add retry logic with exponential backoff.

```python
import os
import time
from dotenv import load_dotenv
import anthropic

load_dotenv()

api_key = os.environ["ANTHROPIC_API_KEY"]
client = anthropic.Anthropic(api_key=api_key)

def call_with_retry(prompt, max_retries=3):
    for attempt in range(max_retries):
        try:
            start = time.time()
            response = client.messages.create(
                model="claude-sonnet-4-20250514",
                max_tokens=50,
                messages=[{"role": "user", "content": prompt}]
            )
            elapsed = time.time() - start
            print(f"Attempt {attempt + 1}: Success ({elapsed:.2f}s)")
            return response
        except anthropic.RateLimitError as e:
            wait = 2 ** attempt  # Exponential backoff: 1s, 2s, 4s
            print(f"Attempt {attempt + 1}: Rate limited. Waiting {wait}s...")
            time.sleep(wait)
    
    print("All retries exhausted.")
    return None

# Test with 5 rapid calls
for i in range(5):
    print(f"\nCall {i + 1}:")
    response = call_with_retry(f"Count from {i * 5 + 1} to {i * 5 + 5}.")
    if response:
        print(f"  Response: {response.content[0].text}")
        print(f"  Tokens: {response.usage.output_tokens}")
    time.sleep(1.5)  # Polite delay between groups
```

The exponential backoff — waiting 1 second, then 2 seconds, then 4 seconds — gives the rate limit window time to reset. The `max_retries=3` prevents infinite retries. The logging shows you exactly what happened: which attempts succeeded, which were rate limited, and how long you waited.

In production, you would also add jitter (a small random delay) to prevent multiple retrying clients from syncing up, and you would log to a file instead of printing to the console. This basic retry pattern is the foundation for all production API interaction.
