# Using OpenRouter for Multi-Provider Access

A single API gateway that connects you to hundreds of LLM providers with one integration.

## The Problem Story

You spend two weeks building an AI-powered customer support agent. You pick a single provider -- GPT-4o -- because it gives the best responses. Your code calls a single endpoint. Your architecture has one dependency. Your tests pass. You deploy to production and everything works.

Then the provider goes down.

Not a rate-limit spike. Not a brief 503 that a retry would fix. A full regional outage that lasts six hours. Your support agent is silent. Customers cannot get help. Your team scrambles to find another model, but your code is hardcoded to one API format, one base URL, one authentication scheme. There is no quick switch.

You get through the outage. Management asks: "What if this happens again?" You know the right answer is multi-provider support. But building it yourself means maintaining N different SDKs, N authentication flows, N request formats, N error-handling layers. Every provider has its own quirks. OpenAI uses a chat completions format. Anthropic uses a different messages format. Google uses yet another. Writing an abstraction layer that normalizes all of them is a project in itself.

Even if you solve the reliability problem, you still have a cost problem. Different providers charge different prices for equivalent capability. The cheapest model that can handle your task changes week to week. Without a dynamic routing layer, you either overpay for a top-tier model every time or you manually track pricing tables and update your code constantly.

This is exactly the problem OpenRouter exists to solve.

## The Concept

OpenRouter is a unified API gateway that sits between your application and dozens of LLM providers. You send one request in one format, and OpenRouter routes it to the best available provider based on your preferences for cost, latency, throughput, or reliability.

Instead of managing N separate API integrations, you manage one. Instead of hardcoding a model string, you specify a routing policy. Instead of handling each provider's error format, you get standardized error responses. Instead of manually checking which providers are up, OpenRouter monitors provider health and fails over automatically.

The key insight is that OpenRouter uses the OpenAI-compatible chat completions format. Any SDK that speaks the OpenAI API can talk to OpenRouter by changing the base URL and API key. This means you get multi-provider access without learning a new SDK.

### What OpenRouter provides

- **Unified endpoint**: One POST endpoint for all models from all providers
- **Automatic failover**: When one provider is down, routes to another
- **Cost optimization**: Route by cheapest price, lowest latency, or highest throughput
- **Model fallback chains**: Try model A, then B, then C on different providers
- **Provider preferences**: Allowlist, blocklist, or pin order of providers
- **Zero data retention**: Route only to providers that do not store your data
- **Bring your own key**: Use existing provider API keys through OpenRouter

## API / SDK / CLI Reference

The table below covers the essential OpenRouter API features you will use when building multi-provider applications.

| Feature | Endpoint / Header | Syntax / Example |
|---|---|---|
| Chat Completions | `POST https://openrouter.ai/api/v1/chat/completions` | `curl -X POST "https://openrouter.ai/api/v1/chat/completions" -H "Authorization: Bearer $OPENROUTER_API_KEY" -d '{"model":"openai/gpt-4o","messages":[{"role":"user","content":"Hello"}]}'` |
| List Models | `GET https://openrouter.ai/api/v1/models` | `curl "https://openrouter.ai/api/v1/models" -H "Authorization: Bearer $OPENROUTER_API_KEY"` |
| Authentication | `Authorization: Bearer <key>` header | Set the environment variable `OPENROUTER_API_KEY` and pass it in every request header |
| Provider Routing | `provider` object in request body | `{"provider": {"order": ["Anthropic", "OpenAI"], "allow_fallbacks": false}}` -- pins providers in exact order and disables automatic fallback |
| Model Fallback | `models` array in request body | `{"models": ["openai/gpt-4o", "anthropic/claude-sonnet-4", "google/gemini-2.0-flash-lite"]}` -- walks the list in order if each model fails |
| Price Ceiling | `provider.max_price` object in request body | `{"provider": {"max_price": {"prompt": 2e-7, "completion": 2e-6}}}` -- caps prompt and completion token prices in dollars |
| Rate Limits | Free tier: 50 requests/day, 20 RPM. Paid tier ($10+ credits): no platform-level limits | OpenRouter does not enforce rate limits for pay-as-you-go users, but individual providers may enforce their own |
| Pricing Endpoint | `GET https://openrouter.ai/api/v1/models` includes `pricing` field per model | Each model entry contains `{"pricing": {"prompt": "2e-7", "completion": "2e-6"}}` with per-million-token prices as strings |

### Using the OpenAI SDK with OpenRouter

The OpenAI Python SDK works as a drop-in client for OpenRouter. You only change two values: the base URL and the API key.

```python
from openai import OpenAI

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key="OPENROUTER_API_KEY",  # set via environment variable
)

response = client.chat.completions.create(
    model="openai/gpt-4o",
    messages=[{"role": "user", "content": "Hello, world!"}],
)
```

OpenRouter recognizes the standard OpenAI request body fields and adds its own routing extensions through the `provider` and `models` parameters.

## How It Works -- Step by Step

The process of making a multi-provider request through OpenRouter involves several stages. Each stage is a separate concern that you can configure independently.

### Step 1: Choose a model identifier

OpenRouter uses a `provider/model` slug format to identify every model on the platform. The slug uses the provider name followed by a forward slash and the model name as the provider identifies it.

Common slug patterns:

```
openai/gpt-4o
anthropic/claude-sonnet-4-20250514
google/gemini-2.0-flash-001
meta-llama/llama-4-maverick-17b-128e-instruct
mistralai/mistral-large-2411
deepseek/deepseek-chat
```

You can also use router suffixes to express intent without specifying a concrete model:

| Suffix | Behavior | Use Case |
|---|---|---|
| `:nitro` | Route to the fastest provider for this model | User-facing chat where latency matters |
| `:floor` | Route to the cheapest provider for this model | Batch jobs where cost is the priority |

For example, `openai/gpt-4o:nitro` routes to the fastest GPT-4o provider, and `openai/gpt-4o:floor` routes to the cheapest.

### Step 2: Configure provider routing

The `provider` object in the request body controls how OpenRouter selects which provider serves your request. You can specify ordering, sorting, allowlists, blocklists, and quality constraints.

```json
{
  "model": "anthropic/claude-sonnet-4-20250514",
  "provider": {
    "order": ["Anthropic", "Amazon Bedrock"],
    "allow_fallbacks": false,
    "data_collection": "deny"
  }
}
```

This configuration tells OpenRouter: try Anthropic first, then fall back to Amazon Bedrock if Anthropic is unavailable. Do not fall back to any other provider. Only route to providers that do not train on your data.

The full set of provider preference fields:

| Field | Type | Description |
|---|---|---|
| `order` | `string[]` | Provider slugs in exact priority order |
| `allow_fallbacks` | `boolean` | Whether to allow backup providers (default: true) |
| `sort` | `string` | Sort by `"price"`, `"throughput"`, or `"latency"` |
| `only` | `string[]` | Allowlist of provider slugs |
| `ignore` | `string[]` | Blocklist of provider slugs to skip |
| `max_price` | `object` | Price cap `{prompt, completion, image}` per token |
| `data_collection` | `"allow"` or `"deny"` | Whether providers may store request data |
| `zdr` | `boolean` | Require zero data retention endpoints only |
| `quantizations` | `string[]` | Filter by quantization level (e.g., `"int4"`, `"int8"`) |

When you set `sort` or `order`, OpenRouter disables its automatic load balancing and applies your preferences directly.

### Step 3: Set up model fallback chains

Model fallback lets you specify an ordered list of models to try. If the first model fails (outage, rate limit, or error), OpenRouter tries the next one in the list. This is different from provider fallback -- provider fallback tries the same model on a different provider, while model fallback tries a completely different model.

```json
{
  "models": [
    "openai/gpt-4o",
    "anthropic/claude-sonnet-4",
    "google/gemini-2.0-flash-lite",
    "mistralai/mistral-small-2501"
  ],
  "provider": {
    "allow_fallbacks": true
  }
}
```

When you pass a `models` array, OpenRouter walks the list in order. If the first model returns a successful response, that is what you get. If it errors out, OpenRouter tries the next model. This continues until either a model succeeds or the list is exhausted.

### Step 4: Send the request

Once you have configured your model, routing preferences, and fallback chain, you send the request to OpenRouter. The request body follows the OpenAI chat completions format with additional OpenRouter-specific fields.

```python
import os
from openai import OpenAI

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key=os.environ["OPENROUTER_API_KEY"],
)

response = client.chat.completions.create(
    model="openai/gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Explain what OpenRouter does in one sentence."},
    ],
    temperature=0.7,
    max_tokens=200,
)
```

The response object returned by the OpenAI SDK is identical in structure regardless of which provider handled the request on the backend.

```json
{
  "id": "gen-xxxxxxxxxxxx",
  "object": "chat.completion",
  "created": 1730000000,
  "model": "openai/gpt-4o",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "OpenRouter is a unified API gateway that routes your requests to the best available LLM provider based on your preferences for cost, latency, and reliability."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 25,
    "completion_tokens": 30,
    "total_tokens": 55
  }
}
```

### Step 5: Handle provider metadata

OpenRouter optionally returns metadata about which provider handled your request. This is useful for debugging, billing attribution, and understanding routing behavior.

To opt in, set the `X-OpenRouter-Metadata` header to `true` on your request:

```python
response = client.chat.completions.create(
    model="openai/gpt-4o",
    messages=[{"role": "user", "content": "Hello"}],
    extra_headers={
        "X-OpenRouter-Metadata": "true"
    }
)
```

When metadata is enabled, the response includes a `provider` field in `response.body` showing which provider was selected and why. You can also see the model slug that was actually used, which may differ from your requested model if fallback routing kicked in.

```json
{
  "provider": "Anthropic",
  "model": "anthropic/claude-sonnet-4-20250514",
  "reason": "Primary provider was available and within price constraints"
}
```

### Step 6: Set up streaming for real-time responses

OpenRouter supports streaming through server-sent events (SSE), the same way OpenAI does. Streaming is essential for chat applications where you want to show tokens as they are generated rather than waiting for the complete response.

```python
import os
from openai import OpenAI

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key=os.environ["OPENROUTER_API_KEY"],
)

stream = client.chat.completions.create(
    model="openai/gpt-4o",
    messages=[{"role": "user", "content": "Count from 1 to 5 slowly."}],
    stream=True,
)

for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="")
```

Each chunk in the stream contains a `delta` with the next token. When streaming through OpenRouter, the provider failover logic still applies for the initial connection, but once a stream is established it runs to completion on the same provider.

## Cost, Limits, and Safety

### Token pricing

OpenRouter passes through provider pricing with no markup. You pay exactly what the provider charges per million tokens. The 5.5% platform fee applies to credit purchases, not to individual API calls.

Each model in the models list endpoint includes a `pricing` field with per-million-token prices as strings:

```json
{
  "pricing": {
    "prompt": "2.5e-7",
    "completion": "1e-6"
  }
}
```

These values are in dollars per token. To calculate the cost of a request:

```
prompt_cost = prompt_tokens * float(pricing["prompt"])
completion_cost = completion_tokens * float(pricing["completion"])
total_cost = prompt_cost + completion_cost
```

### Using the :floor suffix to minimize cost

Append `:floor` to any model slug to automatically route to the cheapest available provider for that model:

```python
response = client.chat.completions.create(
    model="openai/gpt-4o:floor",
    messages=[{"role": "user", "content": "Hello"}],
)
```

### Setting a price ceiling

Use the `provider.max_price` field to enforce a hard cost cap per token. If no provider can serve the request within the price ceiling, OpenRouter returns an error rather than exceeding your budget:

```json
{
  "model": "openai/gpt-4o",
  "provider": {
    "max_price": {
      "prompt": 3e-7,
      "completion": 2e-6
    }
  }
}
```

### Rate limits

OpenRouter rate limits depend on your plan:

| Plan | Rate Limit |
|---|---|
| Free tier (no credits purchased) | 50 requests per day, 20 requests per minute |
| Pay-as-you-go (with $10+ in credits) | No platform-level rate limits. Individual providers may enforce their own limits. |
| Enterprise | Optional dedicated rate limits and SLA |

Free-tier requests that fail due to errors still count against your daily quota.

### Safety tips

- Always set `provider.max_price` to prevent unexpected costs from routing to an expensive provider
- Use `provider.allow_fallbacks: false` when you need strict provider pinning for compliance
- Use `data_collection: "deny"` and `zdr: true` for sensitive data
- Set a budget alert in the OpenRouter dashboard to get notified when spending exceeds a threshold
- Test with cheap models first and validate token usage before scaling to production traffic
- Enable `X-OpenRouter-Metadata` in development to understand which provider is handling each request

### Bring your own key (BYOK)

If you already have API keys with providers, you can bring them to OpenRouter. This is typically cheaper because you bypass OpenRouter's routing entirely for that provider and pay only OpenRouter's BYOK fee (5% after the first 1 million free requests per month).

BYOK is configured per provider in the OpenRouter dashboard under the Keys section. Once configured, requests for that provider model are forwarded using your key.

## When to Use / When Not To

### When to use OpenRouter

- **Multi-provider redundancy**: Your application needs high availability and cannot tolerate a single provider outage
- **Cost optimization**: You want to automatically route to the cheapest provider for each request
- **Model experimentation**: You want to compare responses from different models on the same task
- **Geographic compliance**: You need to restrict data processing to specific regions
- **Rapid prototyping**: You want to try many models without creating accounts with each provider
- **Capability matching**: You want to route simple queries to cheap models and complex queries to powerful models

### When not to use OpenRouter

- **Single-provider commitment**: You have an exclusive contract with one provider and do not need failover
- **Provider-specific features**: You rely on features that only exist in a provider's native SDK and are not exposed through the OpenAI-compatible format
- **Maximum latency sensitivity**: Routing through a gateway adds minimal overhead (typically under 50ms), but if every millisecond matters, direct provider access is faster
- **Tight compliance requirements**: Some organizations require direct contractual relationships with each provider, which a gateway intermediary complicates
- **Free provider access**: If the provider offers free tier access through their own endpoint, you may not need a gateway

## Mental Model Check

Test your understanding of OpenRouter with these questions.

### Question 1

You are building a customer support chatbot that must stay online even if one provider has an outage. You want to prioritize Anthropic but fall back to OpenAI if Anthropic is unavailable.

Which configuration should you use?

**Answer**: Set `provider.order` to `["Anthropic", "OpenAI"]` with `allow_fallbacks` set to `true` (the default). OpenRouter tries Anthropic first. If Anthropic returns an error, it falls back to OpenAI. If both fail, you can add a `models` array with additional model options.

```json
{
  "model": "anthropic/claude-sonnet-4",
  "provider": {
    "order": ["Anthropic", "OpenAI"],
    "allow_fallbacks": true
  }
}
```

### Question 2

You have a batch job that processes millions of records overnight. Cost is the primary concern. Which routing strategy minimizes cost?

**Answer**: Use the `:floor` suffix on your model slug. This routes to the cheapest available provider for that model. Combined with `provider.max_price`, you can enforce a hard cost ceiling.

```python
response = client.chat.completions.create(
    model="openai/gpt-4o:floor",
    provider={"max_price": {"prompt": 3e-7, "completion": 2e-6}},
    messages=[{"role": "user", "content": "Process this record."}],
)
```

### Question 3

You send a request to OpenRouter with `model: "openai/gpt-4o"` but the response contains text generated by Google Gemini. How is this possible?

**Answer**: This happens when model fallback or provider fallback is enabled. The primary model is unavailable, and OpenRouter routes to an available model that can handle the request. To prevent this, set `allow_fallbacks: false` in the provider object and do not use the `models` array. Alternatively, check the `X-OpenRouter-Metadata` response to see which provider handled the request.

### Question 4

Your application processes sensitive financial data. You need to ensure that no provider stores or trains on your prompts and completions. What configuration do you need?

**Answer**: Set `data_collection: "deny"` and `zdr: true` in the provider object. This restricts routing to providers that do not collect training data and that offer zero data retention endpoints.

```json
{
  "provider": {
    "data_collection": "deny",
    "zdr": true
  }
}
```

### Question 5

You want to try three different models on the same prompt to compare their responses. What is the most efficient way to do this with OpenRouter?

**Answer**: Send three separate requests, each with a different model slug in the `model` field. Do not use the `models` array (which is for fallback, not parallel comparison). To ensure each response comes from the intended model, disable fallbacks:

```python
models_to_test = ["openai/gpt-4o", "anthropic/claude-sonnet-4", "google/gemini-2.0-flash-001"]

for model in models_to_test:
    response = client.chat.completions.create(
        model=model,
        provider={"allow_fallbacks": False},
        messages=[{"role": "user", "content": "Summarize AI engineering in three bullet points."}],
    )
    print(f"--- {model} ---")
    print(response.choices[0].message.content)
```
