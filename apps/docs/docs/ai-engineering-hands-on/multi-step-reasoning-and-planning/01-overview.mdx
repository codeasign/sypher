You ask your agent to "clean up the project directory." It immediately runs `rm -rf /tmp/project` without checking if there is a backup. The files are gone. You forgot to mention that "clean up" means "archive old log files, keep the source code, and move the build artifacts to a staging folder." The agent had no way to know what you meant because it never asked, never planned, and never checked its assumptions before acting.

This is the single most dangerous failure mode in agentic systems: acting without thinking. An agent that barrels into the first action it thinks of is not intelligent — it is a loaded weapon. You have seen this pattern before: the agent books a flight before confirming the dates, deletes a branch before checking if the work was merged, sends an email before proofreading it, or makes confident assumptions about what you meant instead of gathering information first. The root cause is always the same — the agent skipped the planning phase.

## The Concept

Multi-step reasoning and planning is the capability of an agent to break a complex goal into sub-steps, reason about dependencies, execute them in order, and adjust the plan when something goes wrong. Think of a project manager who creates a work breakdown structure, assigns dependencies, monitors progress, and re-plans when a task runs late. That manager does not start building without a blueprint. The same discipline applies to agents.

A planner agent is fundamentally different from a reactive agent. A reactive agent sees a query and immediately produces an answer or action. A planner agent pauses, decomposes the goal, checks what information it needs, verifies assumptions, executes each step deliberately, and verifies results before moving on. This pause-and-plan phase is what separates a reliable agent from a chaotic one.

The planning process mirrors how you approach complex tasks in your own work. When you build a feature, you do not start writing code immediately. You think about the requirements, the data model, the API endpoints, the edge cases, and the testing strategy. You create a mental plan, even if you do not write it down. An agent needs to do the same thing explicitly, because it cannot hold a mental model the way you can.

## API/SDK Reference

### Chain-of-Thought Reasoning

Chain-of-thought (CoT) is the foundational technique for multi-step reasoning. Instead of asking the model to produce an answer directly, you ask it to reason step by step. The model generates intermediate reasoning tokens that lead to a more accurate final answer.

```python
from anthropic import Anthropic

client = Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=2048,
    messages=[
        {
            "role": "user",
            "content": "A store has a 20% discount on all items. A customer buys a $50 shirt and a $30 pair of pants. What is the total after discount? Think step by step."
        }
    ]
)
print(response.content[0].text)
```

### ReAct (Reasoning + Acting)

ReAct interleaves reasoning traces with tool calls. The model thinks out loud, calls a tool, observes the result, and continues reasoning. This pattern prevents the model from committing to an action before understanding the context.

```python
SYSTEM_PROMPT = """
You are an agent that follows the ReAct pattern.
For each step, output:
1) Thought: what you need to figure out
2) Action: the tool call you will make
3) Observation: the result of the tool call
4) Final: your conclusion when you have enough information
"""
```

### Plan-and-Execute Pattern

In the plan-and-execute pattern, the agent first generates a complete plan as a structured list of steps, then executes each step. This is different from ReAct, where reasoning and acting are interleaved. Plan-and-execute separates the two phases explicitly.

```python
plan = agent.create_plan(
    goal="Compare pricing of OpenAI vs Anthropic"
)
# Returns:
# [
#   "Find OpenAI API pricing page",
#   "Find Anthropic API pricing page",
#   "Extract per-token costs for GPT-4o and Claude Sonnet",
#   "Compare input/output pricing across models",
#   "Summarize findings in a comparison table"
# ]
```

### Task Decomposition

Task decomposition is the process of breaking a high-level goal into smaller, executable sub-tasks. The LLM analyzes the goal and generates a list of sub-tasks that, when executed in order, achieve the original goal.

```python

            }
        ]
```

### Sub-Task Scheduling

Once you have a list of sub-tasks with dependencies, you need to schedule them. Tasks with no dependencies can run in parallel. Tasks with dependencies must wait for their prerequisites to complete.

```python
def schedule_tasks(tasks: list[dict]) -> list[list[dict]]:
    """
    Group tasks into sequential batches.
    Batch 0: tasks with no dependencies.
    Batch 1: tasks whose dependencies are all in batch 0.
    Batch N: tasks whose dependencies are all in batches 0 through N-1.
    """
    scheduled = []
    remaining = list(tasks)
    while remaining:
        batch = [t for t in remaining if all(
            any(d == completed["step"] for completed in sum(scheduled, []))
            for d in t["depends_on"]
        )]
        if not batch:
            break
        scheduled.append(batch)
        remaining = [t for t in remaining if t not in batch]
    return scheduled
```

### Dependency Tracking

Dependency tracking ensures that each sub-task executes only after its prerequisites are satisfied. This prevents ordering errors like searching for something before you know the search terms.

```python
class DependencyTracker:
    def __init__(self):
        self.completed = set()
        self.failed = set()

    def can_execute(self, task: dict) -> bool:
        return all(dep in self.completed for dep in task["depends_on"])

    def mark_completed(self, step: int):
        self.completed.add(step)

    def mark_failed(self, step: int):
        self.failed.add(step)
```

### Re-Planning on Failure

When a step fails, the agent needs to re-plan. It should not blindly continue with the remaining plan. The re-planning process evaluates which remaining steps are affected by the failure and adjusts them.

```python
def replan(original_plan: list[dict], failed_step: int, error: str) -> list[dict]:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": f"""
Original plan: {json.dumps(original_plan, indent=2)}
Step {failed_step} failed with error: {error}

Create a revised plan that handles this failure.
Return a JSON array of {{'step': int, 'task': str, 'depends_on': list[int]}}.
                """
            }
        ]
    )
    return parse_json(response.content[0].text)
```

## How It Actually Works — Step by Step

### Step 1: Goal Analysis

When you give your agent a goal, the first thing it must do is analyze what you actually asked for. The agent reads your request and identifies the key components: what information is needed, what tools are available, what success looks like, and what could go wrong.

The LLM generates an analysis that extracts the implicit requirements from your request. If you say "compare API pricing," the agent identifies that it needs to know which APIs, what pricing dimensions (input tokens, output tokens, caching, rate limits), and what format for the comparison.

### Step 2: Dependency Mapping

Once you have the sub-tasks, you need to figure out which ones depend on each other. This is critical because executing tasks in the wrong order wastes time and produces wrong results.

The agent identifies dependencies by examining each sub-task and asking: "What information do I need before I can do this?" If Task C requires the output of Task A, then Task C depends on Task A. Tasks with no dependencies can run in parallel.

```python
def map_dependencies(sub_tasks: list[str]) -> list[dict]:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": f"""
Given these sub-tasks: {json.dumps(sub_tasks)}

For each sub-task, identify which other sub-tasks it depends on.
Return a JSON array of {{"task": str, "depends_on": list[int]}}.
Use indices starting at 0.
                """
            }
        ]
    )
    return json.loads(response.content[0].text)
```

Common dependency types include:
- **Data dependency**: Task B needs the output of Task A
- **Resource dependency**: Task B needs a tool or connection established by Task A
- **Conditional dependency**: Task B only needs to run if Task A found something
- **Sequencing dependency**: Task B must run after Task A for logical ordering

### Step 3: Execution Ordering

With dependencies mapped, you determine what runs first, what runs in parallel, and what runs next. This is the scheduling phase. The goal is to minimize total execution time by running independent tasks in parallel.

Here is the scheduling implementation:

```python
def schedule_tasks(tasks_with_deps: list[dict]) -> list[list[int]]:
    """Returns batches of task indices that can run in parallel."""
    batches = []
    completed = set()
    remaining = set(range(len(tasks_with_deps)))

    while remaining:
        batch = []
        for i in list(remaining):
            deps = tasks_with_deps[i]["depends_on"]
            if all(d in completed for d in deps):
                batch.append(i)
        if not batch:
            break
        for i in batch:
            remaining.remove(i)
        batches.append(batch)
        completed.update(batch)

    return batches

tasks = [
    {"task": "Fetch OpenAI pricing", "depends_on": []},
    {"task": "Fetch Anthropic pricing", "depends_on": []},
    {"task": "Parse OpenAI rates", "depends_on": [0]},
    {"task": "Parse Anthropic rates", "depends_on": [1]},
    {"task": "Build comparison", "depends_on": [2, 3]},
]

batches = schedule_tasks(tasks)
for i, batch in enumerate(batches):
    print(f"Batch {i}: tasks {batch} -- {[tasks[b]['task'] for b in batch]}")
```

### Step 4: Step Execution

Each sub-task is executed as a tool call, an API call, or a sub-agent invocation. The agent passes the required context from completed dependencies to the current task. This is where the actual work happens.

```python
class PlanningAgent:
    def __init__(self):
        self.client = Anthropic()
        self.tracker = DependencyTracker()

    def execute_step(self, task: dict, context: dict) -> str:
        response = self.client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=2048,
            messages=[
                {
                    "role": "user",
                    "content": f"""
Task: {task['task']}

Context from completed steps: {json.dumps(context, indent=2)}

Execute this task and return the result.
"""
                }
            ]
        )
        return response.content[0].text
```

Each step execution follows this cycle:
1. **Receive input**: the task description and context from previous steps
2. **Process**: the LLM generates the response or the tool executes the action
3. **Produce output**: structured result that the next step can consume
4. **Verify**: check that the output is valid before passing it along

### Step 5: Result Verification

After each step executes, the agent verifies that the result is valid. This is not the same as the LLM being confident -- verification uses explicit checks like schema validation, range checks, or a verification LLM call.

```python
def verify_result(step_name: str, result: str, success_criteria: str) -> bool:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=256,
        messages=[
            {
                "role": "user",
                "content": f"""
Step: {step_name}
Result: {result}
Success criteria: {success_criteria}

Did this step succeed? Answer only YES or NO.
"""
            }
        ]
    )
    return "YES" in response.content[0].text.upper()
```

If verification fails, the agent can retry the step (if the error is transient or fixable), or trigger re-planning (if the step cannot succeed as originally defined).

### Step 6: Re-Planning on Failure

When a step fails verification, the agent must re-plan. This is the most important safety mechanism in a planning agent. Without re-planning, the agent continues executing a plan that is now invalid, producing garbage results.

```python
def handle_step_failure(
    plan: list[dict],
    failed_step: int,
    error: str,
    completed_results: dict
) -> tuple[list[dict], str]:
    """
    Returns (revised_plan, status) where status is 'continue' or 'abort'.
    """
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": f"""
Original plan: {json.dumps(plan, indent=2)}
Step {failed_step} failed: {plan[failed_step]['task']}
Error: {error}
Completed steps: {json.dumps(completed_results, indent=2)}

Can we continue? If yes, provide a revised plan starting from step {failed_step}.
If no, explain why we must abort.

Return JSON: {{"can_continue": bool, "revised_plan": list, "reason": str}}
"""
            }
        ]
    )
    decision = json.loads(response.content[0].text)

    if decision["can_continue"]:
        return decision["revised_plan"], "continue"
    else:
        return plan, f"abort: {decision['reason']}"
```

### Step 7: Completion Verification

After all sub-tasks are executed, the agent checks whether the original goal is met. This is a final verification step that ensures nothing was missed. The agent reviews the original goal against the accumulated results and confirms completeness.

```python
def verify_completion(original_goal: str, all_results: dict) -> dict:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        messages=[
            {
                "role": "user",
                "content": f"""
Original goal: {original_goal}
All step results: {json.dumps(all_results, indent=2)}

Is the goal fully met? Return JSON:
{{"goal_met": bool, "missing_items": list, "summary": str}}
"""
            }
        ]
    )
    return json.loads(response.content[0].text)
```

If the goal is not fully met, the agent can create a new plan for the missing items or report to you what remains.

## Cost, Limits, and Safety

### Token Cost of Long Planning Chains

Every planning step consumes tokens. A single ReAct loop with a thought, action, and observation typically costs 200-500 tokens. A plan with 10 steps costs 2,000-5,000 tokens just for reasoning. If the agent re-plans, you pay for the re-planning tokens too.

Consider the token budget carefully. A research task that takes 10 steps with 2 re-planning cycles costs roughly 12,000 tokens in reasoning alone, plus the tool call tokens. At Claude Sonnet pricing, that is approximately $0.04 per task. For a batch of 1,000 tasks, that is $40 in reasoning overhead.

### Depth-First vs Breadth-First Planning

**Depth-first planning** explores one path to completion, going deep before going wide. This is efficient when the first path is likely correct but wastes effort if the first path fails midway. Token cost is proportional to the depth of exploration.

**Breadth-first planning** explores multiple possible plans before choosing the best one. This is more robust for uncertain tasks but consumes more tokens upfront. Token cost is proportional to the number of plans explored.

For most agent tasks, depth-first with re-planning on failure is the right balance. The agent makes a plan, executes it, and re-plans only when something goes wrong. This avoids the token waste of exploring alternatives that may never be needed.

### Setting Max Planning Depth

Always set a maximum planning depth to prevent infinite loops or excessive planning. A good default is 5-10 steps per plan, with a maximum of 3 re-planning cycles.

```python
class BoundedPlanner:
    MAX_PLAN_DEPTH = 10
    MAX_REPLAN_CYCLES = 3

    def __init__(self):
        self.replan_count = 0

    def create_plan(self, goal: str) -> list[dict]:
        plan = self._generate_plan(goal)
        if len(plan) > self.MAX_PLAN_DEPTH:
            plan = self._consolidate_plan(plan)
        return plan

    def can_replan(self) -> bool:
        return self.replan_count < self.MAX_REPLAN_CYCLES

    def replan(self, plan: list[dict], failed_step: int, error: str) -> list[dict]:
        if not self.can_replan():
            raise RuntimeError("Max re-plan cycles exceeded")
        self.replan_count += 1
        return self._generate_revised_plan(plan, failed_step, error)
```

### Handling Plan Failure Gracefully

When a plan cannot be completed, the agent must fail gracefully. This means reporting to you what was accomplished, what failed, and why. A hard crash or silent partial output is unacceptable.

Graceful failure includes:
1. Listing completed steps and their results
2. Identifying the step that failed and why
3. Suggesting alternative approaches you could try
4. Preserving partial results so you do not have to start over

### Cost of Re-Planning

Re-planning is expensive. Each re-plan cycle costs roughly as much as the original plan generation. If re-planning happens frequently, your costs multiply. Monitor re-plan rates in production and investigate agents that re-plan more than once per task.

## When to Use This / When Not To

**Use multi-step reasoning and planning when:**
- The task requires multiple coordinated actions
- Steps have dependencies that must be respected
- The cost of a wrong action is high (deleting files, booking travel, sending messages)
- You need the agent to explain its reasoning and show its work
- The task is complex enough that a single LLM call cannot handle it

**Do not use multi-step planning when:**
- The task is a simple question that a single LLM call can answer
- The planning overhead exceeds the value of the task (do not plan for 500 tokens to answer a 50-token question)
- The task is purely creative and benefits from improvisation rather than rigid planning
- The task is deterministic and can be hardcoded as a simple script
- You need sub-second response time -- planning adds latency to every request

The general rule: if you can describe the task in one sentence and do it in two steps, you do not need planning. If the task requires more than two conditional steps, planning will save you time and errors.

## Mental Model Check

**Question 1:** What is the difference between a reactive agent and a planning agent, and why does the distinction matter for safety?

  A reactive agent takes the first action it thinks of without analyzing the goal first. A planning agent pauses to decompose the goal, map dependencies, and schedule execution. The distinction matters because reactive agents can perform destructive actions (deleting files, sending messages) without understanding the context, while planning agents verify assumptions before acting.

**Question 2:** Why does chain-of-thought reasoning improve planning quality?

  CoT gives the model token space to compute intermediate results. Without CoT, the model must jump from the goal directly to the plan in one step, which often misses dependencies or edge cases. With CoT, the model reasons through each decomposition decision explicitly, producing more complete and accurate plans.

**Question 3:** How do you decide between sequential and parallel execution for sub-tasks?

  Sub-tasks with no dependencies on each other can run in parallel. Sub-tasks where one requires the output of another must run sequentially. The scheduler groups independent tasks into batches and executes each batch in parallel, reducing total execution time.

---

**Question 4:** What happens when a step in the plan fails verification?

  The agent should trigger re-planning. First, it checks if the failure can be worked around by replacing the failed step with an alternative. If not, it reports the failure to you with details about what was accomplished and what remains. The agent never continues executing the remaining plan as if nothing happened.

---

**Question 5:** What is the right balance between planning depth and execution cost?

  For most tasks, a plan of 5-10 steps with a maximum of 3 re-plan cycles is appropriate. The guiding principle is that planning overhead should not exceed the value of the task. A task that takes 10 seconds to execute manually does not need a 50-step plan. A task that involves deleting production data needs thorough planning even if it takes longer. Monitor re-plan rates and adjust limits based on your tolerance for cost versus safety.
