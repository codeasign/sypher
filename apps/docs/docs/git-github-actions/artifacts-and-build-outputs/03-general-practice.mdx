## Tier 1 — Beginner

### Exercise 1: Upload a single file

**Task:** Create a workflow that writes "Hello, artifacts!" to a file named `greeting.txt`, then uploads it as an artifact named `greeting`.

**Hint:** Use `echo` to write the file and `actions/upload-artifact@v4` to upload it.

<details>
<summary>Solution</summary>

```yaml
name: Single File Artifact
on: workflow_dispatch

jobs:
  upload:
    runs-on: ubuntu-latest
    steps:
      - name: Create file
        run: echo "Hello, artifacts!" > greeting.txt
      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: greeting
          path: greeting.txt
```

After running, download the `greeting` artifact from the Actions tab. It will be a zip file containing `greeting.txt`.
</details>

### Exercise 2: Download a specific artifact

**Task:** Create a workflow with two jobs. The first uploads a file called `data.txt`. The second downloads that artifact and prints its contents.

**Hint:** Use `actions/download-artifact@v4` in the second job with the same `name` as the upload.

<details>
<summary>Solution</summary>

```yaml
name: Download Specific
on: workflow_dispatch

jobs:
  create:
    runs-on: ubuntu-latest
    steps:
      - name: Create data
        run: echo "Secret sauce recipe" > data.txt
      - name: Upload
        uses: actions/upload-artifact@v4
        with:
          name: my-data
          path: data.txt

  consume:
    needs: create
    runs-on: ubuntu-latest
    steps:
      - name: Download
        uses: actions/download-artifact@v4
        with:
          name: my-data
      - name: Print contents
        run: cat data.txt
```

The second job downloads and prints the file contents.
</details>

### Exercise 3: Set a custom retention period

**Task:** Upload an artifact named `temp-build` that automatically deletes after 2 days instead of the default 90.

**Hint:** Use the `retention-days` input on `actions/upload-artifact@v4`.

<details>
<summary>Solution</summary>

```yaml
name: Short Retention
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Temporary build output" > output.bin
      - uses: actions/upload-artifact@v4
        with:
          name: temp-build
          path: output.bin
          retention-days: 2
```

The artifact will be deleted after 2 days regardless of the repository's default retention setting.
</details>

## Tier 2 — Intermediate

### Exercise 4: Upload and download with custom paths

**Task:** Create a workflow where the build job creates files in a subdirectory `output/release/`, uploads the entire `output/` directory as an artifact, and the deploy job downloads it to a custom path `/tmp/deploy-files/`.

**Hint:** Both `upload-artifact` and `download-artifact` accept a `path` input for custom locations.

<details>
<summary>Solution</summary>

```yaml
name: Custom Paths
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Create files
        run: |
          mkdir -p output/release
          echo "binary v1.0" > output/release/app.bin
          echo "config" > output/release/config.yml
      - name: Upload directory
        uses: actions/upload-artifact@v4
        with:
          name: release-files
          path: output/

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Download to custom path
        uses: actions/download-artifact@v4
        with:
          name: release-files
          path: /tmp/deploy-files/
      - name: Verify path
        run: |
          ls -la /tmp/deploy-files/release/
          cat /tmp/deploy-files/release/config.yml
```

The files are extracted into `/tmp/deploy-files/release/` — the artifact's internal directory structure is preserved.
</details>

### Exercise 5: Cache Python dependencies

**Task:** Create a workflow that uses `actions/cache@v4` to cache Python pip dependencies. The cache key should include a hash of `requirements.txt`. Add a restore-key fallback that matches any previous Python cache.

**Hint:** The pip cache directory is `~/.cache/pip` on Linux and macOS, and `~\AppData\Local\pip\Cache` on Windows.

<details>
<summary>Solution</summary>

```yaml
name: Cached Dependencies
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-${{ runner.os }}-${{ hashFiles('**/requirements.txt') }}
          restore-keys: |
            pip-${{ runner.os }}-
            pip-
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Test
        run: python -c "import pytest; print('pytest cached!')"
```

On the first run, pip installs from scratch and the cache is saved. On subsequent runs (with the same requirements.txt), pip installs from the cache — reducing install time from ~30 seconds to ~2 seconds.
</details>

### Exercise 6: Upload artifacts with glob patterns

**Task:** Create a workflow that generates multiple log files (`debug.log`, `error.log`, `access.log`) in a `logs/` directory, then uploads only the `error.log` file as an artifact using a glob pattern.

**Hint:** Use a glob pattern like `logs/error.log` or `**/error.log` for the `path` input.

<details>
<summary>Solution</summary>

```yaml
name: Glob Pattern
on: workflow_dispatch

jobs:
  generate:
    runs-on: ubuntu-latest
    steps:
      - name: Create logs
        run: |
          mkdir -p logs
          echo "[DEBUG] detail" > logs/debug.log
          echo "[ERROR] failure" > logs/error.log
          echo "[ACCESS] GET /" > logs/access.log
      - name: Upload only error logs
        uses: actions/upload-artifact@v4
        with:
          name: error-logs
          path: logs/error.log
      - name: Upload all logs except debug
        uses: actions/upload-artifact@v4
        with:
          name: all-except-debug
          path: |
            logs/*.log
            !logs/debug.log
```

Glob patterns support exclusion with `!` prefix. The second artifact includes `error.log` and `access.log` but not `debug.log`.
</details>

## Tier 3 — Advanced

### Exercise 7: Debug an artifact workflow

**Scenario:** This workflow has multiple issues. Find and fix them:

```yaml
name: Broken Artifacts
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: mkdir -p build && echo "binary" > build/output
      - uses: actions/upload-artifact@v4
        with:
          name: Build Output
          path: build/

  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: Build Output
      - name: Run tests
        run: cat build/output
      - uses: actions/upload-artifact@v4
        with:
          name: test-results.xml
          path: /tmp/results/

  deploy:
    needs: [build, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
      - run: ls -R
```

<details>
<summary>Solution</summary>

There are at least 4 issues:

1. **Artifact name with space**: `name: Build Output` — while technically valid, artifact names with spaces are harder to reference later. Use kebab-case or underscores: `build-output`.

2. **Downloaded artifact extract location**: The `test` job downloads `Build Output` but does not specify a `path`. By default, download-artifact extracts into the current working directory. The extracted files will be in a subdirectory named after the artifact (e.g., `Build Output/output`). The step `cat build/output` expects the file at `./build/output`, which is not where it is actually placed.

3. **Upload step with no files**: The test job uploads `path: /tmp/results/` but never creates that directory or any files in it. This will produce a warning "No files were found."

4. **Deploy downloads all artifacts without name**: The deploy job uses `actions/download-artifact@v4` without specifying a name, which downloads ALL artifacts. This is correct but may be unintended if the workflow grows.

Fixed version:

```yaml
name: Fixed Artifacts
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: mkdir -p build && echo "binary" > build/output
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: build/

  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: ./downloaded
      - name: Run tests
        run: cat ./downloaded/output
      - name: Create test results
        run: |
          mkdir -p /tmp/results
          echo "All tests passed" > /tmp/results/test-results.xml
      - uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: /tmp/results/test-results.xml

  deploy:
    needs: [build, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: ./deploy-files
      - run: |
          echo "Deploying..."
          ls -la ./deploy-files/
```

Key fixes:
- Changed artifact names to kebab-case
- Added `path: ./downloaded` in the download step so files land at a predictable location
- Created the test results file before uploading
- Deploy job downloads only the build artifact, not all artifacts
</details>

### Exercise 8: Design a multi-platform release pipeline

**Scenario:** Design a workflow that:
1. Builds a Go application on Linux, macOS, and Windows
2. Names each binary with the OS and architecture in the filename (e.g., `app-linux-amd64`, `app-darwin-amd64`, `app-windows-amd64.exe`)
3. Uploads each binary as a separate artifact
4. Has a `release` job that downloads all binaries, creates a GitHub release, and attaches all binaries to the release
5. Only runs when a tag `v*.*.*` is pushed
6. Uses `GITHUB_TOKEN` with `contents: write` permission

<details>
<summary>Solution</summary>

```go
// main.go — simple Go application
package main

import "fmt"

func main() {
    fmt.Println("Hello from multi-platform build!")
}
```

```yaml
name: Multi-Platform Release
on:
  push:
    tags:
      - 'v*.*.*'

permissions:
  contents: write

jobs:
  build:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        include:
          - os: ubuntu-latest
            goos: linux
            goarch: amd64
            ext: ''
          - os: macos-latest
            goos: darwin
            goarch: amd64
            ext: ''
          - os: windows-latest
            goos: windows
            goarch: amd64
            ext: '.exe'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.21'
      - name: Build binary
        run: |
          GOOS=${{ matrix.goos }} GOARCH=${{ matrix.goarch }} go build -o app-${{ matrix.goos }}-${{ matrix.goarch }}${{ matrix.ext }} .
      - name: Upload binary
        uses: actions/upload-artifact@v4
        with:
          name: binary-${{ matrix.goos }}-${{ matrix.goarch }}
          path: app-${{ matrix.goos }}-${{ matrix.goarch }}${{ matrix.ext }}

  release:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Download all binaries
        uses: actions/download-artifact@v4
        with:
          path: ./binaries
      - name: List binaries
        run: find ./binaries -type f -exec ls -la {} \;
      - name: Create Release
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          gh release create ${{ github.ref_name }} \
            ./binaries/*/* \
            --title "Release ${{ github.ref_name }}" \
            --notes "Automated release for ${{ github.ref_name }}"
```

When a tag like `v1.0.0` is pushed, this workflow:
1. Builds the Go app for 3 platforms in parallel (3 matrix jobs)
2. Each job uploads its binary as a uniquely-named artifact
3. The release job downloads all artifacts and creates a GitHub release with all binaries attached

Other valid approaches include using `gox` for cross-compilation, combining all binaries into a single artifact, or using the `softprops/action-gh-release` action instead of the CLI.
</details>

### Exercise 9: Optimize workflow with selective artifact handling

**Scenario:** Your monorepo has 3 packages: `frontend`, `backend`, and `shared`. Each has its own test suite. Design a workflow that:

1. Determines which packages changed in the current push (use `git diff`)
2. Only builds and tests the changed packages
3. Uploads test reports as artifacts for only the changed packages
4. Deploys only if all changed packages pass
5. Uses caching for both pip (backend) and npm (frontend) dependencies
6. Wall-clock time should be minimized — if only `frontend/` changes, do not run backend tests

<details>
<summary>Solution</summary>

```yaml
name: Selective CI
on: [push, workflow_dispatch]

jobs:
  detect:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.changes.outputs.frontend }}
      backend: ${{ steps.changes.outputs.backend }}
      shared: ${{ steps.changes.outputs.shared }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2
      - id: changes
        run: |
          if [ "${{ github.event_name }}" = "workflow_dispatch" ]; then
            echo "frontend=true" >> $GITHUB_OUTPUT
            echo "backend=true" >> $GITHUB_OUTPUT
            echo "shared=true" >> $GITHUB_OUTPUT
          else
            echo "frontend=$(git diff --name-only HEAD~1 HEAD | grep -c '^frontend/')" >> $GITHUB_OUTPUT
            echo "backend=$(git diff --name-only HEAD~1 HEAD | grep -c '^backend/')" >> $GITHUB_OUTPUT
            echo "shared=$(git diff --name-only HEAD~1 HEAD | grep -c '^shared/')" >> $GITHUB_OUTPUT
          fi

  frontend-test:
    needs: detect
    if: needs.detect.outputs.frontend > 0
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Cache npm
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: npm-${{ hashFiles('frontend/package-lock.json') }}
          restore-keys: npm-
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
        working-directory: ./frontend
      - run: npm test
        working-directory: ./frontend
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: frontend-test-report
          path: frontend/test-results/
          retention-days: 14

  backend-test:
    needs: detect
    if: needs.detect.outputs.backend > 0 || needs.detect.outputs.shared > 0
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-${{ hashFiles('backend/requirements.txt') }}
          restore-keys: pip-
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
        working-directory: ./backend
      - run: pytest -v --junitxml=test-results/results.xml
        working-directory: ./backend
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: backend-test-report
          path: backend/test-results/
          retention-days: 14

  deploy:
    needs: [detect, frontend-test, backend-test]
    if: always()
    runs-on: ubuntu-latest
    steps:
      - name: Check all tests passed
        run: |
          if [ "${{ needs.frontend-test.result }}" = "failure" ] || [ "${{ needs.backend-test.result }}" = "failure" ]; then
            echo "Tests failed — skipping deploy"
            exit 1
          fi
          echo "All tests passed — deploying"
```

Key design decisions:
- The `detect` job uses `git diff` to determine which packages changed
- Frontend and backend jobs only run if their packages changed
- Backend also runs if `shared/` changed (since backend depends on shared)
- Each job uploads test reports as artifacts with unique names
- The deploy job checks the result of each test job using `needs.<id>.result`
- Caching is per-package with scoped keys to avoid cache collisions
</details>
