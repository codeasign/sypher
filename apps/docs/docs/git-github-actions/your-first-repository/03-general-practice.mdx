The exercises below are organized by difficulty. Start with Tier 1 to build muscle memory for the basic commands, then progress through Tier 2 and Tier 3 as you gain confidence. Each exercise specifies exact commands, expected output, and verification steps.

## Tier 1 — Beginner

These exercises focus on single commands. Complete each one in a terminal.

### Exercise 1: Initialize a Repository

Create a new directory called `practice-beginner` and initialize it as a Git repository.

```bash
mkdir practice-beginner
cd practice-beginner
git init
```

**Expected output:**

```
Initialized empty Git repository in .../practice-beginner/.git/
```

**Verification:**

```bash
ls -a
```

You should see `.git` (a hidden directory) and nothing else. The `.git` directory is the repository.

**If you get stuck:** Make sure you are inside the `practice-beginner` directory. Run `pwd` to check your current directory path. If you see a different path, use `cd practice-beginner` to navigate to the correct directory.

### Exercise 2: Stage and Commit a Single File

Inside the `practice-beginner` directory, create a file named `hello.txt` with the content "Hello, Git!" and commit it.

```bash
echo "Hello, Git!" > hello.txt
git add hello.txt
git commit -m "add hello.txt"
```

**Expected output:**

```
[main (root-commit) abc1234] add hello.txt
 1 file changed, 1 insertion(+)
 create mode 100644 hello.txt
```

The commit hash (`abc1234`) will be different on your machine. That is expected — each commit gets a unique hash.

**Verification:**

```bash
git log --oneline
```

You should see a single line showing your commit.

**If you get stuck:** Run `git status` at any point. If it shows `hello.txt` as untracked, you forgot to run `git add hello.txt`. If it shows "nothing to commit, working tree clean" but you have not committed, you may have already committed — check with `git log`.

### Exercise 3: Commit Multiple Files

Create two more files and commit them together in a single commit.

```bash
echo "Line 1" > file-a.txt
echo "Line 2" > file-b.txt
git add file-a.txt file-b.txt
git commit -m "add two more files"
```

**Expected output:**

```
[main def5678] add two more files
 2 files changed, 2 insertions(+)
 create mode 100644 file-a.txt
 create mode 100644 file-b.txt
```

Note that Git reports "2 files changed, 2 insertions" — both files were included in this single commit.

**Verification:**

```bash
git log --oneline
```

You should see two commits now. The most recent one is at the top.

### Exercise 4: View the Commit History

Use `git log` with different flags to see your commit history in various formats.

```bash
git log --oneline
```

**Expected output:**

```
def5678 (HEAD -> main) add two more files
abc1234 add hello.txt
```

Now try the detailed view:

```bash
git log
```

This shows the full commit hash, author, date, and message for each commit. Press `q` to exit the log viewer.

**Verification:** Count the commits. You should have exactly two. Each commit message should match what you wrote.

## Tier 2 — Intermediate

These exercises require multiple steps and introduce the full lifecycle.

### Exercise 5: Create a Repository from Scratch

This exercise simulates a real-world scenario. You will create a project directory, initialize a repository, create three files across two commits, and push to a remote.

Create a new directory outside of `practice-beginner`:

```bash
cd ..
mkdir project-alpha
cd project-alpha
git init
```

Create three files with meaningful content:

```bash
echo "# Project Alpha" > README.md
echo "print('Hello from Project Alpha')" > main.py
echo "requests>=2.28.0" > requirements.txt
```

Stage and commit all three files:

```bash
git add README.md main.py requirements.txt
git commit -m "initial commit: add README, main.py, and requirements"
```

Now simulate development: edit `main.py` to add a function:

```bash
echo "def greet(name):" >> main.py
echo "    return f'Hello, {name}'" >> main.py
echo "" >> main.py
echo "if __name__ == '__main__':" >> main.py
echo "    print(greet('World'))" >> main.py
```

Stage only the changed file and commit:

```bash
git add main.py
git commit -m "add greet function"
```

**Verification:**

```bash
git log --oneline
```

You should see two commits. The first commit includes three files; the second commit includes only `main.py`.

**Expected output:**

```
f6e5d4c (HEAD -> main) add greet function
a1b2c3d initial commit: add README, main.py, and requirements
```

### Exercise 6: Fix a Detached HEAD State

A detached HEAD state happens when you check out a commit directly instead of a branch. Practice entering and leaving this state.

Create a new repository for this exercise:

```bash
cd ..
mkdir practice-detached
cd practice-detached
git init
echo "commit 1" > file.txt
git add file.txt && git commit -m "first commit"
echo "commit 2" >> file.txt
git add file.txt && git commit -m "second commit"
echo "commit 3" >> file.txt
git add file.txt && git commit -m "third commit"
```

Now check out the first commit by its hash:

```bash
git log --oneline
```

Copy the hash of the first commit (the bottom one) and run:

```bash
git checkout <first-commit-hash>
```

**Expected output:**

```
Note: switching to '<hash>'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.
```

You are now in detached HEAD state. Your `HEAD` points directly to a commit, not to a branch. To return to a normal state:

```bash
git checkout main
```

**Expected output:**

```
Previous HEAD position was a1b2c3d first commit
Switched to branch 'main'
```

You are back on the `main` branch. Your working directory now shows the latest version of `file.txt` with all three lines.

**Verification:** Run `git log --oneline` and confirm you see all three commits. Your HEAD is pointing to `main`.

### Exercise 7: Recover from a Bad Commit

Sometimes you commit something you should not have. Practice recovering from a bad commit.

Stay in the `practice-detached` directory (or create a new one if you prefer).

```bash
echo "BAD CODE - DO NOT USE" >> file.txt
git add file.txt
git commit -m "accidental bad commit"
```

You have made a commit you regret. To undo it while keeping the changes in your working directory:

```bash
git reset --soft HEAD~1
```

This moves the branch pointer back one commit but keeps the file changes staged. Check the status:

```bash
git status
```

You will see that `file.txt` is still modified. To discard the changes entirely:

```bash
git reset --hard HEAD
```

This resets both the staging area and the working directory to the last commit. The bad changes are gone.

**Verification:** Run `git log --oneline` and confirm the bad commit is no longer in the history. Run `cat file.txt` and confirm the "BAD CODE" line is not present.

**Important:** `git reset --hard` permanently discards uncommitted changes. Only use it when you are certain you want to throw away your work.

### Exercise 8: Stage Selectively

This exercise demonstrates partial staging — staging only some of your changes while leaving others uncommitted.

```bash
cd ..
mkdir practice-selective
cd practice-selective
git init
echo "alpha" > alpha.txt
echo "beta" > beta.txt
echo "gamma" > gamma.txt
git add alpha.txt beta.txt gamma.txt
git commit -m "initial commit: three files"
```

Now modify each file:

```bash
echo "change 1" >> alpha.txt
echo "change 2" >> beta.txt
echo "change 3" >> gamma.txt
```

Stage only `alpha.txt` and `gamma.txt`:

```bash
git add alpha.txt gamma.txt
```

Check the status:

```bash
git status -s
```

**Expected output:**

```
M  alpha.txt
 M beta.txt
M  gamma.txt
```

The `M` in the first column (staged) means `alpha.txt` and `gamma.txt` are staged. The `M` in the second column (working tree) means `beta.txt` is modified but not staged.

Commit the staged files:

```bash
git commit -m "update alpha and gamma"
```

Verify that `beta.txt` remains modified:

```bash
git status -s
```

You should see only `beta.txt` listed as modified.

## Tier 3 — Advanced

These exercises simulate real-world scenarios you will encounter as you work on larger projects.

### Exercise 9: Create a Repository from an Existing Project

You have a project directory with files already in it. You need to add Git version control after the fact without losing any data.

Create a simulated existing project:

```bash
cd ..
mkdir existing-project
cd existing-project
echo "console.log('hello');" > app.js
echo "node_modules/" > .gitignore
echo "# My App" > README.md
mkdir src
echo "export const greet = (name) => name;" > src/utils.js
```

This directory already has files and a subdirectory. Now initialize the repository:

```bash
git init
git add -A
git commit -m "initial commit: add existing project files"
```

The `-A` flag stages all files and directories, including the `src/` subdirectory. Verify everything was committed:

```bash
git log --oneline
git show --stat HEAD
```

The `git show --stat HEAD` command shows which files were included in the latest commit. You should see `app.js`, `.gitignore`, `README.md`, and `src/utils.js`.

**Challenge:** Create a public repository on GitHub named `existing-project` and push this repository to it. Use the commands from the practice exercise section if you need a reminder.

### Exercise 10: Handle a Rejected Push

This exercise simulates a realistic collaboration scenario where two developers push to the same repository.

You need two separate directories for this exercise. Create them side by side:

```bash
cd ..
mkdir repo-central
cd repo-central
git init
echo "# Central Repository" > README.md
git add README.md
git commit -m "initial commit"
```

Now clone this repository into two separate working directories:

```bash
cd ..
git clone repo-central developer-alice
git clone repo-central developer-bob
```

In `developer-alice`:

```bash
cd developer-alice
echo "Alice's changes" >> README.md
git add README.md
git commit -m "alice updates README"
```

In `developer-bob`:

```bash
cd ../developer-bob
echo "Bob's changes" >> README.md
git add README.md
git commit -m "bob updates README"
```

Now push from Alice first:

```bash
cd ../developer-alice
git push
```

Then try to push from Bob:

```bash
cd ../developer-bob
git push
```

**Expected error:**

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to '../repo-central'
```

Bob's push is rejected because Alice pushed first. Bob's local history does not include Alice's changes. To fix this, Bob must pull first:

```bash
git pull --rebase origin main
```

This pulls Alice's changes and re-applies Bob's commit on top of them. Then push:

```bash
git push
```

**Verification:** In the `repo-central` directory, check the log:

```bash
cd ../repo-central
git log --oneline
```

You should see both commits: Alice's update and Bob's update. The order in the log depends on the timestamps and the rebase.

### Exercise 11: Work with Multiple Remotes

This exercise simulates a fork-based workflow where you have two remotes: `origin` (your fork) and `upstream` (the original repository).

Create a local repository:

```bash
cd ..
mkdir multi-remote
cd multi-remote
git init
echo "# Multi-Remote Demo" > README.md
git add README.md
git commit -m "initial commit"
```

Add two remotes with different URLs:

```bash
git remote add origin https://github.com/your-username/multi-remote.git
git remote add upstream https://github.com/upstream-owner/multi-remote.git
```

Verify both remotes:

```bash
git remote -v
```

**Expected output:**

```
origin    https://github.com/your-username/multi-remote.git (fetch)
origin    https://github.com/your-username/multi-remote.git (push)
upstream  https://github.com/upstream-owner/multi-remote.git (fetch)
upstream  https://github.com/upstream-owner/multi-remote.git (push)
```

Practice fetching from the upstream remote:

```bash
git fetch upstream
```

This downloads commits from the upstream repository but does not merge them. You can view upstream branches with:

```bash
git branch -r
```

To pull changes from upstream into your main branch:

```bash
git pull upstream main
```

This fetches and merges changes from the upstream repository's `main` branch into your current branch.

**Challenge:** Rename the `upstream` remote to `original` using `git remote rename upstream original`. Then remove the `original` remote with `git remote remove original`. Verify the removal with `git remote -v`.

## Summary of Exercises

| Tier | Exercise | Skill Practiced | Key Command |
|---|---|---|---|
| Beginner | 1 | Repository creation | `git init` |
| Beginner | 2 | Single file commit | `git add`, `git commit` |
| Beginner | 3 | Multi-file commit | `git add <file1> <file2>` |
| Beginner | 4 | History viewing | `git log --oneline` |
| Intermediate | 5 | Full lifecycle | `git init` through `git commit` |
| Intermediate | 6 | Detached HEAD recovery | `git checkout <hash>`, `git checkout main` |
| Intermediate | 7 | Undoing a commit | `git reset --soft HEAD~1` |
| Intermediate | 8 | Selective staging | `git add <file>` |
| Advanced | 9 | Existing project repo | `git add -A` |
| Advanced | 10 | Rejected push resolution | `git pull --rebase` |
| Advanced | 11 | Multiple remotes | `git remote add`, `git fetch` |

Complete all 11 exercises to build a solid foundation in Git repository management. Each exercise builds on the previous one, so work through them in order.
