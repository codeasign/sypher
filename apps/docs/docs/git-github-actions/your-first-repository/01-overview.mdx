You have been coding for months. You have files scattered across your desktop, a projects folder with folders named `final`, `final-v2`, `final-v3-actually`, and a sinking feeling that you lost something important last week. You made a change to a script, saved over the old version, and now the old version is gone. You cannot undo. You cannot go back. You cannot see what changed. You are coding blind, one save at a time, with no safety net.

This is how most developers start. You learn to write code before you learn to protect it. And then one day you delete a file you needed, or you refactor a function and break three things you did not know depended on it, and you have no way to recover. You spend two days rewriting what you lost. You promise yourself you will never let this happen again.

That promise is what brings you here. A Git repository is the tool that keeps your code safe, tracks every change, and lets you go back to any point in your project's history. By the end of this section, you will go from coding blind to having a complete, versioned, backed-up project that you can share with anyone.

The story of the `final-v3-actually` folder is a common one. You have a project that has evolved over time, and each iteration creates a new folder because you are afraid to lose the old version. You end up with ten copies of the same project, none of which you can confidently say is the right one. You waste time searching through files, comparing them manually, and wondering which version had the feature you need. A repository eliminates this entirely. Instead of duplicating folders, you create commits — labeled snapshots that you can browse, compare, and restore at any time. The single `project/` folder contains the latest version, and Git remembers every version that came before it.

## The Concept

A repository — often called a "repo" — is a directory managed by Git. It contains all your project files *and* the complete history of every change ever made to them. Think of it as a time machine for your code.

The full lifecycle of a repository follows a simple loop:

```
Working directory → Staging area → Local commit → Remote push
```

You start with files on your computer. You tell Git which files to track. You package those changes into a commit — a snapshot of your project at a moment in time. Then you push that commit to a remote server like GitHub so it is backed up and shareable.

Here is every action in the lifecycle:

- **Init** — Turn an ordinary directory into a Git repository. This creates a hidden `.git` folder that stores all version history and configuration.
- **Add** — Move changes from your working directory into the staging area. This selects which files will be included in the next commit.
- **Commit** — Save the staged changes as a permanent snapshot in your local repository. Each commit gets a unique identifier and stores the full state of every staged file.
- **Remote add** — Connect your local repository to a remote server. This creates a named link (usually `origin`) that points to a URL on GitHub or another Git server.
- **Push** — Upload your local commits to the remote repository. This makes your changes visible to others and creates a backup off your machine.
- **Pull** — Download changes from the remote repository to your local one. This keeps your local copy in sync with the remote when others have pushed changes.

These six actions form the foundation of everything you will do with Git. Master them, and you master the daily workflow of every professional developer.

### Git vs GitHub: What Is the Difference

A common point of confusion for new developers is the difference between Git and GitHub. They are not the same thing, and understanding the distinction is important.

**Git** is the version control software that runs on your computer. It is a command-line tool that manages your repository locally. It tracks changes, creates commits, and handles branches. Git works entirely offline — you can create repositories, make commits, and explore history without an internet connection. Git is free and open source.

**GitHub** is a web-based hosting service for Git repositories. It provides a remote server where you can store copies of your repositories, a web interface for browsing code, collaboration features like pull requests and issues, and automation tools like GitHub Actions. GitHub requires an internet connection and a free account.

The relationship is simple: Git is the engine, GitHub is the garage. Git does the work of version control. GitHub stores the results and lets you share them.

There are alternatives to GitHub: GitLab, Bitbucket, SourceHut, and self-hosted options like Gitea. All of them use Git under the hood. The skills you learn here work with any of them.

### What Makes a Repository Different from a Regular Folder

A regular folder on your computer has no memory. When you save a file, the previous version is overwritten. When you delete a file, it goes to the trash, but after you empty the trash, it is gone. There is no record of what the file looked like last week, who edited it, or why.

A Git repository adds a layer of memory on top of your folder. Every time you commit, Git records:

- The exact content of every file you committed.
- The name and email of the person who made the changes.
- The date and time of the commit.
- A message describing what changed and why.
- A pointer to the previous commit, forming a chain of history.

This chain of commits is the core of version control. You can walk backward through it to see exactly how your project evolved. You can compare any two commits to see what changed. You can revert to any previous commit if something goes wrong.

### The Three States of Git

Every file in a Git repository exists in one of three states at any given time:

- **Modified** — The file has been changed in your working directory but has not been staged or committed yet. Git knows the file is different from the last commit but has not recorded the new version.
- **Staged** — The file has been added to the staging area. You have told Git that this version of the file should be included in the next commit. The staging area is a buffer between your working directory and the repository.
- **Committed** — The file has been saved as a permanent snapshot in the local repository. The commit is stored in the `.git/objects/` database and is part of the project's history.

Understanding these three states is the key to understanding Git. Every command you run moves files between these states.

## Command / Syntax Reference

The table below lists every command you need to create and manage a repository from scratch. Each row shows the command, its syntax, the most common flags, and a concrete example.

| Command | Syntax | Common Flags | What It Does | Example |
|---|---|---|---|---|
| `git init` | `git init [directory]` | None | Creates a new Git repository in the current directory (or a specified directory) | `git init` |
| `git add` | `git add <path>` | `-A` (all changes), `.` (current directory) | Stages changes for the next commit | `git add README.md` |
| `git commit` | `git commit -m "<message>"` | `-m` (message inline), `--amend` (edit last commit) | Creates a permanent snapshot of staged changes | `git commit -m "initial commit"` |
| `git remote add` | `git remote add <name> <url>` | None | Links your local repo to a remote server | `git remote add origin https://github.com/user/repo.git` |
| `git push` | `git push <remote> <branch>` | `-u` (set upstream), `--force` (override remote) | Uploads commits to the remote repository | `git push -u origin main` |
| `git pull` | `git pull <remote> <branch>` | `--rebase` (rebase instead of merge) | Downloads and merges remote changes into your local branch | `git pull origin main` |
| `git status` | `git status` | `-s` (short format) | Shows which files are staged, unstaged, or untracked | `git status` |
| `git log` | `git log` | `--oneline` (compact), `--graph` (visual branches) | Displays the commit history | `git log --oneline` |
| `git clone` | `git clone <url>` | None | Creates a local copy of a remote repository | `git clone https://github.com/user/repo.git` |
| `git fetch` | `git fetch <remote>` | None | Downloads commits from a remote without merging | `git fetch origin` |

**Flags explained:**

- `-m` — Passes the commit message directly on the command line instead of opening an editor. Always use this for quick commits. Without `-m`, Git opens your default text editor (Vim, Nano, or VS Code) and waits for you to type a message.
- `-u` — Sets the upstream branch so future pushes can use just `git push` without specifying remote and branch. You only need to do this once per branch.
- `-A` — Stages all changes including new, modified, and deleted files. Equivalent to `git add --all`. Use this when you want to stage everything in one command.
- `--oneline` — Shows each commit on a single line with a shortened hash and the commit message. Much easier to scan than the full log.
- `-s` — Shows the status in a compact format with two-character codes: `??` for untracked, `M` for modified, `A` for added. The first column is the staging area, the second column is the working directory.
- `--amend` — Modifies the most recent commit instead of creating a new one. Use this to fix a commit message or add forgotten files to the last commit.
- `--rebase` — When pulling, reapplies your local commits on top of the remote changes instead of creating a merge commit. This keeps the history linear and cleaner.
- `--force` — Overwrites the remote branch with your local history. Use with extreme caution — it can delete commits on the remote that other people may have based their work on.

## How It Actually Works — Step by Step

### Step 1: Repository Initialization

When you run `git init`, Git creates a hidden folder called `.git` inside your project directory. This folder is the repository. It contains everything Git needs to track your project: configuration files, the object database, references to branches and tags, and the staging area.

The `.git` directory is the engine room. The `objects/` directory will store every version of every file you commit — each version is compressed and stored as a "blob" identified by its SHA-1 hash. The `refs/` directory tracks branches and tags. The `HEAD` file records which branch you are currently on — it contains a reference like `ref: refs/heads/main`. The `index` file is the staging area, which maps filenames to their staged content blobs. The `config` file stores repository-specific settings like remote URLs and user overrides.

You will almost never touch these files directly, but knowing they exist helps you understand what Git is doing behind the scenes. When you run a Git command, it reads and writes to these files. The `.git` directory is the source of truth for everything Git knows about your project.

Your project files remain exactly where they are. Git does not move or copy them. It just adds a hidden folder that starts watching them.

### Step 2: Staging — The Intermediate Area

When you run `git add README.md`, Git does not commit the file immediately. Instead, it copies the file's content into the staging area (the `index` file inside `.git`). The staging area is a holding zone where you assemble the next commit.

Why have a staging area? Because it lets you control exactly what goes into each commit. You might edit three files but only want to commit two of them. You stage only those two, commit, and the third file stays modified in your working directory, ready for a future commit. This keeps your commits focused and your history clean.

The staging area also lets you review changes before committing. You can run `git diff --staged` to see exactly what will be included in the next commit, line by line. This is your last chance to catch mistakes before they become permanent history.

### Step 3: Commit Object Creation

When you run `git commit -m "initial commit"`, Git takes everything in the staging area and creates a commit object. This object is stored in the `.git/objects/` directory. The commit object is a binary file that contains:

- A **tree** object that maps filenames to their content blobs. The tree is like a directory listing — it tells Git which files were part of this commit and which blob objects contain their content.
- A **parent** pointer to the previous commit (none for the first commit, which is called a "root commit"). This pointer forms the chain of history.
- The **author** and **committer** metadata (name, email, timestamp). These can be different if someone applied a patch on behalf of another person.
- The **commit message** that you provided with `-m`.

Each commit gets a SHA-1 hash — a 40-character hexadecimal string like `a1b2c3d4e5f6789abcdef0123456789abcdef01` — that uniquely identifies it. This hash is how Git refers to that specific snapshot forever. The hash is computed from the commit content, so it is impossible for two different commits to have the same hash.

### Step 4: Remote Connection

A local repository is isolated. To back it up or share it, you need a remote. When you run `git remote add origin https://github.com/your-username/your-repo.git`, Git stores the remote URL in the `.git/config` file under the nickname `origin`. The name `origin` is conventional — it means "the primary remote." You can add more remotes with different names later.

The `.git/config` entry looks like this:

```ini
[remote "origin"]
    url = https://github.com/your-username/your-repo.git
    fetch = +refs/heads/*:refs/remotes/origin/*
```

This tells Git: "when you need to reach the remote named `origin`, use this URL. When you fetch from it, download all branches and store them under `refs/remotes/origin/`."

You can have multiple remotes pointing to different services. For example, you might have `origin` pointing to GitHub and `backup` pointing to GitLab. Each remote is independent — you push and pull from them separately.

### Step 5: Push — Uploading to Remote

When you run `git push -u origin main`, Git:

1. Looks up the remote URL for `origin` in the `.git/config` file.
2. Compresses the commits in your local repository that the remote does not have yet into a "packfile" to minimize transfer size.
3. Transfers those commits to the remote server via HTTPS or SSH.
4. Updates the remote's `refs/heads/main` to point to the latest commit.
5. Records the upstream tracking relationship so future `git push` commands default to `origin main`.

The `-u` (or `--set-upstream`) flag is important. It sets up a tracking relationship between your local branch and the remote branch. After you run this once, you can just type `git push` or `git pull` and Git will know which remote and branch to use. Without it, you would have to type `git push origin main` every time.

### Step 6: Pull — Downloading from Remote

When you run `git pull origin main`, Git:

1. Downloads the commits from the remote that your local repository does not have.
2. Updates your local `origin/main` tracking branch.
3. Merges the remote changes into your current local branch.

The pull command is actually a combination of two commands: `git fetch` (download the data) and `git merge` (integrate it into your branch). You can run these separately if you want more control:

```bash
git fetch origin      # downloads commits but does not merge
git log --oneline origin/main  # review what changed
git merge origin/main          # merge the fetched changes
```

This is how you stay in sync with a team. Other developers push their changes, and you pull them down to incorporate them into your work. If you have local commits that are not on the remote, Git creates a merge commit that combines both histories. If you prefer a linear history, use `git pull --rebase` instead, which re-applies your local commits on top of the remote changes.

### Step 7: Checking Status — Your Dashboard

The `git status` command is your dashboard. It tells you exactly what is happening in your repository at any moment. Run it frequently — before and after every operation — to build a clear mental model of what Git is doing.

```bash
git status
```

The output tells you three things:

- Which branch you are on.
- Whether you have commits that have not been pushed.
- Whether you have changes in your working directory and staging area.

A clean status looks like this:

```
On branch main
nothing to commit, working tree clean
```

This means every change has been committed and your local branch is up to date with the remote. This is a good state to be in before switching branches or pulling new changes.

## When to Use This / When Not To

**Use a repository when:**

- You are starting a new project, even a small one. The cost of `git init` is zero; the cost of losing work is high. A project with a single file benefits from version control just as much as a project with a thousand files.
- You want to experiment with changes without fear of breaking something permanently. Commit the working version, experiment, and if things go wrong, you can always go back to the last commit. This is called "branching," and it is one of Git's most powerful features.
- You need to share your code with others. A remote repository on GitHub is the standard way to distribute and collaborate on code. You can share a link instead of emailing zip files.
- You want a backup of your work. Every push to GitHub is a copy of your code on a different server. If your laptop is lost, stolen, or damaged, your code survives on GitHub.
- You need to track why changes were made. Commit messages create a diary of your project's development. Six months from now, you will thank yourself for writing clear commit messages — they tell you not just what changed, but why.
- You are working on a team. Git is the standard tool for collaborative development. It handles multiple people editing the same files, resolves conflicts, and keeps a complete record of who changed what and when.
- You are deploying software. Git integrates with deployment tools like GitHub Actions, Vercel, Netlify, and Heroku. Pushing to a branch can trigger an automatic deployment to production.

**Do not use a repository when:**

- You are storing large binary files (videos, compiled executables, datasets over 100 MB). Git is optimized for text files and does not handle large binaries well. Every clone of the repository includes every version of every binary file, so the repository size grows quickly. Use Git LFS (Large File Storage) or a dedicated storage solution like AWS S3 instead.
- You are working with sensitive credentials or secrets. Never commit API keys, passwords, or private keys to a repository. Git history is permanent — once a secret is committed, it is in the history forever, even if you delete it in a later commit. Use environment variables or a secrets manager like Doppler or HashiCorp Vault.
- You are working on a throwaway one-off script that you will never revisit. Even then, it takes three seconds to run `git init`, so consider whether "throwaway" really means "throwaway." Most developers have regretted not versioning a script they thought was temporary.
- Your project has a single, never-changing file that you never edit. If you write a file once and never touch it again, a repository adds no value. But if you ever edit it, even once, you will wish you had version history.

## How to Think About Git

The single most important mental model for Git is this: **a repository is a chain of snapshots.** Each commit is a photograph of your entire project at a specific moment. The chain of commits tells the story of your project from start to finish.

When you add a file, you are saying "include this in the next photo." When you commit, you are taking the photo. When you push, you are sending copies of the photos to a remote album. When you pull, you are downloading photos someone else took.

Git does not store differences or patches by default. It stores the full content of every file in every commit. It uses compression and delta encoding under the hood to save space, but conceptually, every commit is a complete snapshot. This is why checking out any commit in history is instant — Git does not need to replay changes. It just restores the files from that snapshot.

### Common Misconceptions

**Misconception 1: Git stores only the changes (diffs) between versions.** This is false. Git stores the full content of every file in every commit. It compresses and deduplicates internally, but the model is complete snapshots, not patches. This is why Git operations are fast — restoring a file from any commit is a direct read, not a replay of changes.

**Misconception 2: You need to be online to use Git.** This is false. Git is a local tool. You can create repositories, make commits, view history, and create branches entirely offline. Only `git push` and `git pull` require an internet connection because they communicate with a remote server.

**Misconception 3: If you delete a file, Git deletes it too.** This is false. Deleting a file from your working directory does not delete it from the repository. The file remains in every commit that contains it. You can restore it at any time from any commit where it existed.

**Misconception 4: Git is only for large projects.** This is false. Git works just as well for a single-file project as it does for a million-line codebase. The overhead is the same: a single `.git` folder. Many developers use Git for personal notes, configuration files, and even writing projects.

## Common Terminology

Here are the terms you will encounter as you work with Git repositories. Understanding these terms makes the documentation and error messages much easier to read.

- **Repository (repo)** — A directory managed by Git, containing your project files and their complete history. The repository is the `.git` folder, not your project folder.
- **Commit** — A snapshot of your project at a specific point in time. Each commit has a unique hash, a message, an author, and a timestamp.
- **Branch** — A movable pointer to a specific commit. The default branch is `main`. Branches let you work on multiple versions of your project simultaneously.
- **Remote** — A copy of your repository hosted on another server, typically GitHub. The default remote is named `origin`.
- **Staging area (index)** — A file (`.git/index`) that holds the list of changes to be included in the next commit. You add files to it with `git add`.
- **Working directory (working tree)** — The directory on your computer where you edit files. It is one of the three locations Git tracks (working directory, staging area, repository).
- **HEAD** — A pointer to the commit you are currently viewing. In normal operation, HEAD points to a branch, which points to a commit. In detached HEAD state, HEAD points directly to a commit.
- **SHA-1 hash** — A 40-character hexadecimal string that uniquely identifies a commit. You will usually see the abbreviated form (7 characters) in `git log --oneline` output.
- **Upstream** — The remote branch that your local branch is tracking. When you run `git push -u origin main`, you set `origin/main` as the upstream of your local `main` branch.
- **Clone** — A copy of a remote repository on your local machine. Cloning copies all commits, branches, and tags.
- **Fetch** — Downloading commits from a remote without merging them into your local branch. Fetch is a safe way to see what others have done without affecting your work.
- **Merge** — Combining two branches into one. Git creates a merge commit that has two parent commits.

## Repository Quick Reference Checklist

Before you create a repository, run through this checklist to make sure you have everything in place:

**Before you start:**
- [ ] Git is installed: run `git --version` to verify.
- [ ] Your name and email are configured: `git config --global user.name` and `git config --global user.email`.
- [ ] You have a GitHub account and are signed in.
- [ ] You have a directory for your project (create it with `mkdir project-name`).

**First-time setup:**
- [ ] Run `git init` inside your project directory.
- [ ] Create a `README.md` file describing your project.
- [ ] Run `git add README.md` to stage it.
- [ ] Run `git commit -m "initial commit"` to commit it.

**Remote setup:**
- [ ] Create a repository on GitHub (do not add README, .gitignore, or license).
- [ ] Run `git remote add origin <url>` to link your local repo to GitHub.
- [ ] Run `git push -u origin main` to push your code and set the upstream.

**Ongoing workflow:**
- [ ] Run `git status` before and after every operation.
- [ ] Run `git add <file>` to stage changes before committing.
- [ ] Run `git commit -m "message"` to create snapshots.
- [ ] Run `git push` to upload your commits.
- [ ] Run `git pull` to download others' changes.
