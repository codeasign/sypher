This exercise walks you through creating your first repository from scratch. You will initialize a local Git repository, create a file, commit it, connect to GitHub, and push your code. By the end, you will have a working repository hosted on GitHub that you can share, back up, and build upon.

## Setup

Before you begin, make sure you have the following ready:

- **Git installed** on your machine. Open a terminal and run `git --version`. If you see a version number like `git version 2.40.0`, you are ready. If you see an error, install Git from https://git-scm.com/downloads.
- **A GitHub account**. Go to https://github.com/join and create an account if you do not have one. It takes less than two minutes.
- **A terminal open** in an empty directory. Create one with `mkdir my-first-repo && cd my-first-repo` or use any empty folder you have.
- **Git configured** with your name and email. Run these two commands, replacing the values with your own:

```bash
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"
```

These commands set the name and email that will appear on every commit you make. The `--global` flag applies this setting to all repositories on your machine. You can override it for a single repository by omitting `--global` and running the commands inside a repo.

## The Walkthrough

### Step 1: Initialize the Repository

Run this command in your empty directory:

```bash
git init
```

You should see output like:

```
Initialized empty Git repository in C:/Users/you/my-first-repo/.git/
```

Git has created a hidden `.git` folder inside your directory. Your directory is now a repository. Verify it with:

```bash
ls -a
```

You will see the `.git` directory listed. On Windows, use `dir /a` to show hidden files.

### Step 2: Create a README File

Every project should have a README file that describes what it does. Create one with a simple description:

```bash
echo "# My First Repository" > README.md
```

This creates a file called `README.md` containing a single Markdown heading. You can also create the file manually in your editor and write the same content.

### Step 3: Check the Status

Before you stage anything, check the current state of your repository. This is a habit you should build from day one. Run `git status` frequently to see what Git knows about your project:

```bash
git status
```

You will see output like this:

```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present
```

Git tells you that `README.md` exists in your directory but is not being tracked. Git sees it but is not saving any version history for it yet. The file is in the "untracked" state.

### Step 4: Stage the File

Tell Git to start tracking `README.md` by staging it:

```bash
git add README.md
```

Run `git status` again:

```bash
git status
```

The output changes:

```
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md
```

The file is now in the staging area, ready to be committed. The message "Changes to be committed" means Git has registered the file's current content and is waiting for you to create a snapshot.

### Step 5: Commit the File

Create your first commit:

```bash
git commit -m "initial commit"
```

Output:

```
[main (root-commit) a1b2c3d] initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

You have just created your first commit. The output tells you:

- `main` — the branch you committed on.
- `(root-commit)` — this is the first commit in the repository. It has no parent commit.
- `a1b2c3d` — the abbreviated commit hash (the full hash is 40 characters).
- `initial commit` — the commit message you provided.
- `1 file changed, 1 insertion(+)` — one file was modified with one line added.

Run `git log` to see your commit in the history:

```bash
git log --oneline
```

```
a1b2c3d (HEAD -> main) initial commit
```

This is your project's history. Right now it has one entry.

### Step 6: Create a Repository on GitHub

Now you need a remote home for your repository. Go to GitHub in your browser:

1. Click the **+** icon in the top-right corner and select **New repository**.
2. Enter the repository name: `my-first-repo`.
3. Leave the description optional.
4. Leave the repository **Public** (or select Private if you prefer).
5. **Do not** check "Add a README file" — you already created one locally.
6. **Do not** add a .gitignore or license.
7. Click **Create repository**.

GitHub will show you a page with commands. Look for the section labeled "push an existing repository from the command line." You will use those commands in the next step.

**Important note about authentication:** Starting in 2021, GitHub no longer accepts account passwords for Git operations. You must use a personal access token instead. If you do not have one, create it now:

1. Click your profile photo in the top-right corner of GitHub and select **Settings**.
2. Scroll down to **Developer settings** (left sidebar).
3. Click **Personal access tokens** > **Tokens (classic)**.
4. Click **Generate new token** > **Generate new token (classic)**.
5. Give the token a name like "my-first-repo-token".
6. Set the expiration to 90 days (or "No expiration" for a permanent token).
7. Select the `repo` scope (it selects all sub-items automatically).
8. Scroll to the bottom and click **Generate token**.
9. Copy the token string immediately. It looks like `ghp_xxxxxxxxxxxxxxxxxxxx`. You cannot see it again after you leave this page.

Keep this token in a secure place. You will use it as your password when Git prompts you for credentials.

### Step 7: Connect Your Local Repository to GitHub

Copy the remote URL from the GitHub page. It will look like one of these:

- HTTPS: `https://github.com/your-username/my-first-repo.git`
- SSH: `git@github.com:your-username/my-first-repo.git`

Run this command to add the remote:

```bash
git remote add origin https://github.com/your-username/my-first-repo.git
```

Replace `your-username` with your actual GitHub username. Verify the remote was added:

```bash
git remote -v
```

You should see two lines:

```
origin   https://github.com/your-username/my-first-repo.git (fetch)
origin   https://github.com/your-username/my-first-repo.git (push)
```

Your local repository is now linked to the remote. The nickname `origin` refers to the GitHub URL.

### Step 8: Push Your Commit to GitHub

Push your local commit to the remote repository:

```bash
git push -u origin main
```

Git will prompt you for your GitHub username and password (or personal access token). If you have two-factor authentication enabled, use a personal access token instead of your password. Enter your credentials, and you should see output like:

```
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 229 bytes | 229.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
To https://github.com/your-username/my-first-repo.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.
```

Your code is now on GitHub. Refresh your repository page in the browser. You will see `README.md` displayed with the content "# My First Repository."

### Step 9: Make Changes and Commit Again

Edit `README.md` to add more content. Open the file and add a new line:

```markdown
# My First Repository

This is my first Git repository. I created it to learn version control.
```

Save the file. Check the status:

```bash
git status
```

```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   README.md
```

Git sees that `README.md` has changed from the version in the last commit. The file is "modified" but not staged. Stage and commit it:

```bash
git add README.md
git commit -m "add description to README"
```

### Step 10: Push Again

Since you already set the upstream with `-u` in step 8, you can now push with just:

```bash
git push
```

Git remembers that your local `main` branch pushes to `origin main`. Verify on GitHub that the updated README appears.

### Step 11: Verify Everything on GitHub

Go to your repository page on GitHub. You should see:

- Your repository name at the top.
- Your README content displayed below the file list.
- A link showing "2 commits" — click it to see both commits in the history.
- The commit messages "initial commit" and "add description to README."

## Common Errors

### Error 1: "fatal: not a git repository"

**Full error message:**

```
fatal: not a git repository (or any of the parent directories): .git
```

**Why it happens:** You ran a Git command (like `git add` or `git commit`) in a directory that is not a Git repository. There is no `.git` folder in the current directory or any parent directory.

**How to fix it:** Run `git init` to turn the current directory into a Git repository, or navigate to the correct directory that already contains a `.git` folder.

### Error 2: "fatal: No configured push destination"

**Full error message:**

```
fatal: No configured push destination.
Either specify the URL from the command-line via --set-upstream or configure the remote repository using

    git remote add <name> <url>

and then push using the remote name

    git push <name>
```

**Why it happens:** You ran `git push` without setting up a remote first. Git does not know where to send your commits.

**How to fix it:** Create a repository on GitHub, then run:

```bash
git remote add origin https://github.com/your-username/your-repo.git
git push -u origin main
```

### Error 3: "error: failed to push some refs" (non-fast-forward)

**Full error message:**

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'https://github.com/your-username/my-first-repo.git'
hint: Updates were rejected because the remote contains work that you do not
hint: have locally. This is usually caused by another repository pushing to
hint: the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
```

**Why it happens:** The remote repository has commits that your local repository does not have. This typically happens when you initialized a GitHub repository with a README, .gitignore, or license file, and then tried to push a local repository that started from scratch. The two histories do not share a common ancestor.

**How to fix it:** Pull the remote changes first, then push:

```bash
git pull origin main --allow-unrelated-histories
git push
```

The `--allow-unrelated-histories` flag tells Git to merge two branches that do not share a common commit. This is necessary when you are combining a local repository with a remote one that was initialized separately.

### Error 4: "fatal: authentication failed"

**Full error message:**

```
fatal: Authentication failed for 'https://github.com/your-username/my-first-repo.git'
```

**Why it happens:** Git could not verify your identity when connecting to GitHub. This happens when you use a password directly instead of a personal access token, or when your token has expired. GitHub removed password authentication for Git operations in August 2021.

**How to fix it:** Generate a personal access token on GitHub:

1. Go to GitHub Settings > Developer settings > Personal access tokens > Tokens (classic).
2. Click **Generate new token**.
3. Give it a name like "git-push" and select the `repo` scope.
4. Click **Generate token** and copy the token immediately.
5. When Git prompts for a password, paste the token instead.

If you want Git to remember your credentials so you do not have to enter them every time, enable the credential helper:

```bash
git config --global credential.helper cache
```

This caches your credentials for 15 minutes by default. You can extend the timeout:

```bash
git config --global credential.helper "cache --timeout=3600"
```

This caches credentials for one hour.

### Error 5: "fatal: The current branch main has no upstream branch"

**Full error message:**

```
fatal: The current branch main has no upstream branch.
To push the current branch and set the remote as upstream, use

    git push --set-upstream origin main
```

**Why it happens:** You ran `git push` without having set the upstream tracking relationship. Git does not know which remote and branch to push to.

**How to fix it:** Run the command with the `-u` flag:

```bash
git push -u origin main
```

After this, the upstream is set and you can use `git push` without additional arguments.

## What You Just Learned

- **Creating a repository:** `git init` transforms any directory into a Git repository by creating a hidden `.git` folder.
- **Staging and committing:** `git add` moves files to the staging area, and `git commit` creates a permanent snapshot of the staged changes.
- **Connecting to a remote:** `git remote add origin <url>` links your local repository to a GitHub repository.
- **Pushing code:** `git push -u origin main` uploads your commits to GitHub and sets the upstream tracking branch.
- **Viewing history:** `git status` shows the current state of your working directory and staging area, and `git log --oneline` shows the commit history.
- **Recovering from errors:** You now know how to fix the five most common Git errors: not a repository, no push destination, rejected push due to non-fast-forward, authentication failure, and no upstream branch.

## Next Steps

You have completed your first repository workflow. Here is what you can do next:

- **Explore the GitHub interface.** Click around your repository page. Look at the "Insights" tab to see contributor statistics. Try editing a file directly on GitHub and then pulling the change to your local machine.
- **Add a second file.** Create a new file in your local repository, commit it, and push it. See how the commit history grows.
- **Practice on a different machine.** Clone your repository to another computer using `git clone <url>`. This is how you access your code from anywhere.
- **Share your repository.** Send the GitHub URL to a friend. They can clone it, make changes, and you can learn how to pull their contributions.
