Complete these exercises on your own to prove you can design, debug, and optimize matrix builds.

## Tier 1 — Beginner

### Exercise 1: Single-Axis Matrix for Python Versions

**Task:** Create a matrix build that tests a Python project on Python 3.10, 3.11, and 3.12. Each job should install dependencies from `requirements.txt` and run `pytest`.

**Hint:** Use `actions/setup-python@v5` with `python-version: ${{ matrix.python }}` and define `matrix: { python: ['3.10', '3.11', '3.12'] }`.

**Solution:**

```yaml
name: Python Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python: ['3.10', '3.11', '3.12']

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python }}
      - run: pip install -r requirements.txt
      - run: pytest
```

Three jobs run in parallel, each with a different Python version. The `strategy` keyword is required for the matrix to work. Each job runs independently and reports its own status.

### Exercise 2: Two-Axis Matrix (OS × Language Version)

**Task:** Create a matrix build that tests a Go project on Ubuntu and macOS, with Go versions 1.21 and 1.22. Use `actions/setup-go@v5`.

**Hint:** Define `matrix: { os: [ubuntu-latest, macos-latest], go: ['1.21', '1.22'] }` and set `runs-on: ${{ matrix.os }}`.

**Solution:**

```yaml
name: Go Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
        go: ['1.21', '1.22']

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: ${{ matrix.go }}
      - run: go build ./...
      - run: go test ./...
```

The matrix produces 4 jobs (2 OS × 2 Go versions). Each job runs on the specified OS with the specified Go version. The `go build` and `go test` commands are the same for every combination.

### Exercise 3: Using `max-parallel` to Limit Concurrency

**Task:** Create a matrix build with 8 job combinations but limit execution to only 2 concurrent jobs at a time.

**Hint:** Add `max-parallel: 2` to the `strategy` block.

**Solution:**

```yaml
name: Limited Matrix

on: [push]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
      max-parallel: 2
      fail-fast: true

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

Only 2 of the 9 jobs run at any time. The remaining 7 wait in the queue. This is useful for controlling runner minute consumption or when the CI infrastructure has limited capacity.

## Tier 2 — Intermediate

### Exercise 1: Matrix with Exclude for Unsupported Combinations

**Task:** Create a matrix build for a Java project that tests on Ubuntu and Windows with Java 17, 21, and 22. Exclude Windows + Java 17 because the project's native library is not supported on that combination.

**Hint:** Use `exclude: [{ os: windows-latest, java: 17 }]`.

**Solution:**

```yaml
name: Java Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        java: [17, 21, 22]
        exclude:
          - os: windows-latest
            java: 17

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: ${{ matrix.java }}
      - run: mvn test
```

The matrix produces 5 jobs instead of 6. The Windows + Java 17 combination is excluded. The `exclude` block is evaluated after the Cartesian product, so it cleanly removes that specific pair.

### Exercise 2: Matrix with Include for an Extra Configuration

**Task:** Create a matrix build that tests Node 18, 20, and 22 on Ubuntu. Additionally, include a macOS + Node 22 job that is marked as experimental and allowed to fail.

**Hint:** Use `include` to add the macOS + Node 22 job with an `experimental` flag. Use `continue-on-error: ${{ matrix.experimental }}`.

**Solution:**

```yaml
name: Matrix with Include

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20, 22]
        include:
          - os: macos-latest
            node: 22
            experimental: true

    continue-on-error: ${{ matrix.experimental || false }}
    runs-on: ${{ matrix.os || 'ubuntu-latest' }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

The `include` adds a job that would not exist in the Cartesian product. The `experimental` flag is set to `true` only for this job, and `continue-on-error` allows it to fail without affecting the overall workflow status.

### Exercise 3: Matrix Outputs to a Downstream Job

**Task:** Create a matrix build that runs tests across 3 Node versions and then passes the test results to a downstream "report" job that prints a summary.

**Hint:** Use `needs: test` on the report job. The report job runs after all matrix jobs complete.

**Solution:**

```yaml
name: Matrix with Report

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test

  report:
    needs: test
    runs-on: ubuntu-latest
    if: always()
    steps:
      - run: |
          echo "All matrix jobs completed."
          echo "Node versions tested: 18, 20, 22"
          echo "Trigger: ${{ github.event_name }} on ${{ github.ref_name }}"
```

The `report` job runs after all 3 matrix jobs finish. The `if: always()` ensures it runs even if one of the matrix jobs failed — useful for sending notifications or generating reports regardless of individual job outcomes.

## Tier 3 — Advanced

### Exercise 1: Debug a Broken Matrix Build

**Scenario:** Your team has this matrix build that is supposed to test on Ubuntu and Windows with Node 18 and 20, but it is failing. Diagnose and fix all issues.

```yaml
name: CI

on: push

jobs:
  test
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

**Issues to identify:**

1. **Missing colon after `test`:** `test` should be `test:` (YAML syntax error — the workflow will fail to parse)
2. **No `timeout-minutes`:** A hung test on Windows could run for 6 hours
3. **No `fail-fast` strategy:** If one job fails, the others continue running unnecessarily, wasting runner minutes

**One valid fix:**

```yaml
name: CI

on: push

jobs:
  test:
    runs-on: ${{ matrix.os }}
    timeout-minutes: 10
    strategy:
      fail-fast: true
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

### Exercise 2: Design a Compatibility Matrix

**Scenario:** You maintain a React component library that must support:
- React 18.2 and 19.0
- Node 18, 20, and 22
- Ubuntu and Windows
- One "latest" configuration on macOS that is experimental

Design a matrix that covers all supported combinations while keeping the total job count under 15. The macOS job should not block the pipeline.

**Approach:**

There is no single correct answer, but a strong approach:

```yaml
name: Component Library Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
        react: ['18.2', '19.0']
        exclude:
          - os: windows-latest
            node: 18
            react: '19.0'
        include:
          - os: macos-latest
            node: 22
            react: '19.0'
            experimental: true

    continue-on-error: ${{ matrix.experimental || false }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm install react@${{ matrix.react }} react-dom@${{ matrix.react }}
      - run: npm test
```

This produces 11 jobs (2 × 3 × 2 = 12, minus 1 exclude, plus 1 include). The macOS job is experimental. The full matrix covers React 18.2 and 19.0 across all OS and Node combinations where they are supported.

### Exercise 3: Split a Large Matrix into Two Workflows

**Scenario:** Your matrix build has grown to 20 combinations (5 OS × 4 Node versions). It takes too long to run all 20 on every push. Design a strategy where a fast subset runs on every push and the full matrix runs on a schedule.

**Approach:**

Create two workflow files:

**Fast CI (`.github/workflows/ci-fast.yml`)** — runs on every push:

```yaml
name: Fast CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

**Full Compatibility (`.github/workflows/ci-full.yml`)** — runs nightly:

```yaml
name: Full Compatibility Matrix

on:
  schedule:
    - cron: '0 6 * * 1-5'  # Weekdays at 6 AM UTC

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest, ubuntu-20.04, windows-2019]
        node: [18, 20, 22]
      max-parallel: 5

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

The fast CI gives developers feedback in under 3 minutes. The full matrix runs overnight and catches edge cases. If the full matrix fails, a notification goes to the team in the morning. This approach saves approximately 80 percent of runner minutes compared to running all 20 combinations on every push.
