Your Python library supports Python 3.9, 3.10, 3.11, 3.12, and 3.13 — but your CI only tests on 3.12. A user on 3.9 opens an issue: `dict.union()` doesn't exist in their version, breaking your library. You never caught it because CI only ran one configuration. Matrix builds let you test every supported version, every OS, every dependency combination — all from a single workflow definition, not five copies of the same file.

## The Concept

A matrix build runs the same job logic across multiple configurations defined by variables. Instead of writing one workflow per OS-version pair, you define a strategy matrix that generates every combination automatically, and GitHub Actions runs them in parallel as separate jobs.

Think of it like a vending machine that accepts multiple coin types. The vending mechanism is the same — insert coins, press button, get snack — but it runs that mechanism for US quarters, UK 2-pound coins, and EU 2-euro coins in parallel to validate all three at once.

## Core Matrix Build YAML Keys

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `strategy` | Parent key for matrix configuration | No | `strategy: { matrix: {...} }` |
| `strategy.matrix` | Defines the axes and values to cross-product | No | `matrix: { os: [ubuntu, windows], node: [18, 20] }` |
| `strategy.fail-fast` | Cancel remaining matrix jobs when one fails | No | `fail-fast: true` |
| `strategy.max-parallel` | Limit concurrent matrix jobs | No | `max-parallel: 3` |
| `${{ matrix.axis }}` | Reference a matrix variable in the job | No | `runs-on: ${{ matrix.os }}` |
| `include` | Add extra config for specific matrix combinations | No | `include: [{ os: macos, node: 20, experimental: true }]` |
| `exclude` | Remove specific combinations from the matrix | No | `exclude: [{ os: windows, node: 18 }]` |

## How Matrix Builds Actually Work — Step by Step

### Step 1: Define the Matrix Axes

You declare the variable axes in `strategy.matrix`. Each axis is an array of values. GitHub Actions computes the Cartesian product of all axes to produce N job specifications:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20, 22]
```

This produces 3 × 3 = 9 job specifications. Each one will run independently with `matrix.os` and `matrix.node` set to a specific combination.

### Step 2: Reference Matrix Variables in the Job

Within the job, you use `${{ matrix.axis-name }}` to substitute the current combination's value:

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
```

GitHub Actions expands `${{ matrix.os }}` to, e.g., `ubuntu-latest` for the first job and `windows-latest` for the second before the runner starts. Each job sees only its own values.

### Step 3: Jobs Run in Parallel (Subject to Limits)

All 9 jobs are enqueued immediately. By default, GitHub runs up to 256 concurrent matrix jobs across all workflows in a repository. You can cap this with `max-parallel`:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]
  max-parallel: 3
```

With `max-parallel: 3`, only 3 of the 6 jobs run at once. The remaining 3 wait until a slot opens. This prevents CI from overwhelming shared infrastructure or consuming all your runner minutes in one burst.

### Step 4: Fail-Fast Behavior (Default: Cancel All)

When one matrix job fails, `fail-fast: true` (the default) tells GitHub to cancel all other in-progress matrix jobs for this workflow run. This saves runner minutes — if a Node 18 build fails because the package doesn't support that version, there is no point running Node 20 and Node 22 until someone fixes the root cause.

```yaml
strategy:
  fail-fast: true  # default — cancel siblings on first failure
  matrix:
    node: [18, 20, 22]
```

Set `fail-fast: false` when you need the full matrix result regardless of individual failures — for example, a release-validation matrix where you need to know if a Windows-specific bug exists even when Linux already failed.

### Step 5: Include and Exclude Refine the Matrix

Use `include` to add extra configurations that would not be generated by the Cartesian product alone — for example, marking one combination as experimental:

```yaml
strategy:
  matrix:
    node: [18, 20]
    os: [ubuntu-latest]
    include:
      - node: 22
        os: ubuntu-latest
        experimental: true
```

Use `exclude` to remove combinations that do not apply:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]
    exclude:
      - os: windows-latest
        node: 18  # Windows + Node 18 not supported
```

`include` adds entries that would not exist in the cross-product. `exclude` removes entries that would. Both are evaluated after the Cartesian product.

### Step 6: Conditional Steps Within Matrix Jobs

You can use `if` conditions to run steps only for specific matrix combinations:

```yaml
steps:
  - run: ./deploy.sh
    if: matrix.node == 20 && matrix.os == 'ubuntu-latest'
```

This is useful for deploy steps that should run only once across the whole matrix, or for experimental jobs that should not block the overall status:

```yaml
continue-on-error: ${{ matrix.experimental }}
```

## When to Use Matrix Builds / When Not To

**Use matrix builds when:**

- Your library or application must support multiple runtime versions (Python 3.9 through 3.13, Node 18 through 22)
- Your code runs on multiple operating systems and OS-specific bugs are common
- You need to test against multiple dependency versions (different package managers, database versions, or SDK versions)
- You want to include experimental configurations that should not block the merge

**Consider a simpler approach when:**

- You support a single runtime version on a single OS — a single job is simpler and uses one runner minute instead of N
- Your matrix would generate more than 10–20 jobs — large matrices are expensive and slow. Break them into separate workflows (fast CI matrix, daily compatibility matrix)
- You are testing multiple dependency versions that rarely break — a rotating cron job that runs the full matrix weekly may be more cost-effective than running all combinations on every push

The rule: matrix builds for what your users actually run. If your published package targets Node 18 LTS and Node 20 LTS, test those two — not 14 through 22 just to be thorough. Each matrix axis multiplies cost and completion time.

## Full Matrix Build Example

```yaml
name: CI Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
        include:
          - os: macos-latest
            node: 22
            experimental: true

    continue-on-error: ${{ matrix.experimental }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

This workflow tests on Ubuntu and Windows with Node 18, 20, and 22, plus an experimental macOS + Node 22 job. The experimental job is allowed to fail without blocking the overall status.
