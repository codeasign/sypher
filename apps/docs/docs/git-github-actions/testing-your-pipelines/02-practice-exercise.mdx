## Setup

Before you start, make sure you have the following ready:

- Docker Desktop or Docker Engine installed and running (`act` requires Docker).
- A GitHub account with a repository where you can create and run workflows.
- The `gh` CLI tool installed and authenticated (`gh auth login`).
- Node.js 18 or later installed locally.
- A terminal you are comfortable running bash commands in.

Install `act` using one of the following methods:

**macOS:**
```bash
brew install act
```

**Windows (using winget):**
```bash
winget install nektos.act
```

**Windows (using Scoop):**
```bash
scoop install act
```

**Linux:**
```bash
curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash
```

**Verify the installation:**
```bash
act --version
```

Expected output (version may vary):
```
act version 0.2.63
```

Create a new directory for this exercise:

```bash
mkdir -p ~/github-actions-testing
cd ~/github-actions-testing
```

Create a simple Node.js project to test with:

```bash
npm init -y
echo '{"name": "test-project","version": "1.0.0"}' > package.json
echo "console.log('hello world');" > index.js
```

Initialize a git repository and create the workflow directory:

```bash
git init
mkdir -p .github/workflows
```

## The Walkthrough

### Step 1: Install `act` and Run a Dry Run on a New Workflow

Create your first workflow file at `.github/workflows/test.yml`:

```yaml
name: Test Pipeline
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:

jobs:
  lint:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: echo "Node version: ${{ matrix.node-version }}"
      - run: npm ci
      - run: node -e "console.log('lint passed for node ' + process.version)"

  build:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: node index.js

  test:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: node -e "console.log('tests passed')"
```

Now run a dry run. This validates the YAML, expands the matrix, and prints the job graph without executing anything:

```bash
cd ~/github-actions-testing
act --dry-run -W .github/workflows/test.yml
```

Expected output (approximately):

```
[Test Pipeline/lint (18)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Test Pipeline/lint (20)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Test Pipeline/build] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Test Pipeline/test] 🚀  Start image=catthehacker/ubuntu:act-22.04

Job graph:
    ┌── lint (18) ──┐
    ├── lint (20) ──┤
    └─────┬─────────┘
          ▼
    ┌── build ──┐
          ▼
    ┌── test ───┘

4 jobs to run. 0 total steps. (dry-run)
```

**Checkpoint:** The dry run should show 4 jobs: `lint (18)`, `lint (20)`, `build`, and `test`. If you see this, the workflow YAML is valid and the matrix expansion is correct.

### Step 2: Run a Specific Job with `act`

Now run just the `build` job locally. This tests the job in isolation without running the `lint` and `test` jobs.

```bash
act -j build -W .github/workflows/test.yml
```

`act` will pull the Docker image (if not cached) and execute the steps. The output will look something like this:

```

[Test Pipeline/build]   🐳  docker run image=catthehacker/ubuntu:act-22.04 entrypoint=[] cmd=[])
```

### Step 3: Intentionally Break the Workflow and Test the Failure

Now introduce a deliberate error to see how `act` reports failures. Add a step that always fails:

Edit `.github/workflows/test.yml` and add this step to the `build` job:

```yaml
      - name: Intentional failure
        run: exit 1
```

Run the build job again:

```bash
act -j build -W .github/workflows/test.yml
```

Expected output (approximately):

```
[Test Pipeline/build]   ⭐  Run Main Intentional failure
[Test Pipeline/build]   ❌  Failure - Main Intentional failure
[Test Pipeline/build] exit 'failure'
[Test Pipeline/build]   🏁  Job failed
Error: exit with `FAILURE`: 1
```

**Checkpoint:** The job should fail at the "Intentional failure" step. The error message should clearly show the exit code and the step name. This is how you validate failure scenarios locally without waiting for GitHub.

Remove the intentional failure step before continuing:

```bash
# Edit the file and remove the "Intentional failure" step
# Or use sed:
sed -i '/Intentional failure/,+2d' .github/workflows/test.yml
```

### Step 4: Test with Custom Event Payloads

Some workflows behave differently based on the triggering event. For example, you might run different jobs on a `pull_request` event vs a `push` event. To test this, create a custom event payload.

Create a file named `event.json`:

```json
{
  "action": "opened",
  "pull_request": {
    "head": {
      "ref": "feature-branch"
    },
    "base": {
      "ref": "main"
    },
    "number": 42
  },
  "repository": {
    "full_name": "test-user/test-repo"
  }
}
```

Update your workflow to use `github.event.action`:

```yaml
name: Test Pipeline
on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize]
  workflow_dispatch:

jobs:
  lint:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: echo "Node version: ${{ matrix.node-version }}"
      - run: npm ci
      - run: node -e "console.log('lint passed for node ' + process.version)"

  pr-check:
    runs-on: ubuntu-latest
    if: ${{ github.event_name == 'pull_request' }}
    steps:
      - run: echo "PR #${{ github.event.number }} opened"
      - run: echo "Branch: ${{ github.event.pull_request.head.ref }} → ${{ github.event.pull_request.base.ref }}"

  build:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: node index.js

  test:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: node -e "console.log('tests passed')"
```

Now run the workflow with the custom event payload:

```bash
act pull_request -e event.json -W .github/workflows/test.yml
```

Expected output should include the `pr-check` job, which only runs on pull_request events:

```
[Test Pipeline/pr-check] ⭐  Run Main echo "PR #42 opened"
[Test Pipeline/pr-check] ✅  Success - Main echo "PR #42 opened"
```

Now run with the push event (no custom payload needed):

```bash
act -W .github/workflows/test.yml
```

The `pr-check` job should be skipped because the event is `push`, not `pull_request`.

**Checkpoint:** You have now validated event-driven workflow behavior locally. The `pr-check` job runs only when you specify `act pull_request`, and is skipped for `act push`.

### Step 5: Test Artifact Upload and Download Locally

Artifacts are a key mechanism for passing data between jobs. Test this locally to verify that your artifact paths and names are correct.

Add artifact steps to your workflow:

```yaml
name: Test Pipeline
on: [push, workflow_dispatch]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Generate build output
        run: |
          mkdir -p dist
          echo "version: 1.0.0" > dist/build-info.txt
          echo "commit: ${{ github.sha }}" >> dist/build-info.txt
          node -e "console.log('build output generated')"
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/

  verify:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: dist/
      - name: Verify artifact content
        run: |
          cat dist/build-info.txt
          echo "Artifact verified successfully"
```

Run the workflow locally:

```bash
act -W .github/workflows/test.yml
```

**Checkpoint:** The `build` job should create the artifact and upload it. The `verify` job should download it and print its contents. This confirms that your artifact names and paths are correct before you push to GitHub.

Note: `act` may not fully support artifact upload/download in all versions. If the artifact step fails, it is a limitation of `act` rather than your workflow. In that case, push to GitHub to verify the artifact behavior there.

### Step 6: Test with `gh` CLI on GitHub

Now push your workflow to GitHub and use the `gh` CLI to trigger and inspect runs.

First, push your code:

```bash
git remote add origin https://github.com/YOUR_USER/YOUR_REPO.git
git add -A
git commit -m "Add workflow with artifact testing"
git push -u origin main
```

Trigger the workflow manually:

```bash
gh workflow run test.yml
```

Watch the run in real-time:

```bash
gh run watch
```

After the run completes, view the logs:

```bash
# List recent runs
gh run list

# View logs for a specific run
gh run view <run-id> --log

# Download artifacts
gh run download <run-id>
```

**Checkpoint:** You should see the workflow trigger, run through all jobs, and complete. The `gh run watch` command should show live output of each step.

### Step 7: Test a Matrix Strategy with Multiple Variants

Add a more complex matrix strategy to your workflow to test how matrix expansion works with `act --dry-run`.

Create a new workflow file at `.github/workflows/matrix-test.yml`:

```yaml
name: Matrix Test
on: [push, workflow_dispatch]

jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        os: [ubuntu-latest]
        node: [16, 18, 20]
        include:
          - os: ubuntu-latest
            node: 20
            coverage: true
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: node -e "console.log('Node ' + process.version + ', coverage: ${{ matrix.coverage }}')"
```

Run the dry run:

```bash
act --dry-run -W .github/workflows/matrix-test.yml
```

Expected output:

```
[Matrix Test/build (16)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (18)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (20)] 🚀  Start image=catthehacker/ubuntu:act-22.04

3 jobs to run. 0 total steps. (dry-run)
```

**Checkpoint:** The matrix should expand to 3 jobs (Node 16, 18, 20). The `include` entry for Node 20 with coverage should create a job with `coverage: true`. The dry run shows you the exact matrix expansion before you execute any steps.

### Step 8: Test a Reusable Workflow Locally

Reusable workflows let you define a workflow once and call it from multiple other workflows. Test this locally to verify the interface.

Create the reusable workflow at `.github/workflows/reusable-lint.yml`:

```yaml
name: Reusable Lint
on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string
      fail-on-warning:
        required: false
        type: boolean
        default: false
    outputs:
      lint-status:
        description: "Exit code of the lint step"
        value: ${{ jobs.lint.outputs.status }}

jobs:
  lint:
    runs-on: ubuntu-latest
    outputs:
      status: ${{ steps.result.outputs.status }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - id: result
        run: |
          if [ "${{ inputs.fail-on-warning }}" == "true" ]; then
            echo "Running strict lint..."
            node -e "console.log('strict lint passed')"
            echo "status=passed" >> $GITHUB_OUTPUT
          else
            echo "Running normal lint..."
            node -e "console.log('lint passed')"
            echo "status=passed" >> $GITHUB_OUTPUT
          fi
```

Create the caller workflow at `.github/workflows/caller.yml`:

```yaml
name: Caller Workflow
on: [push, workflow_dispatch]

jobs:
  call-lint:
    uses: ./.github/workflows/reusable-lint.yml
    with:
      node-version: "20"
      fail-on-warning: false

  after-lint:
    runs-on: ubuntu-latest
    needs: call-lint
    steps:
      - run: echo "Lint status: ${{ needs.call-lint.outputs.lint-status }}"
```

Run the caller workflow locally:

```bash
act -W .github/workflows/caller.yml
```

**Checkpoint:** The workflow should call the reusable workflow, run lint with Node 20, and pass the output to the `after-lint` job. This confirms that the reusable workflow interface (inputs and outputs) is correctly defined.

## Common Errors at This Stage

### Error 1: `act` cannot find Docker

```
Error: docker is not available. Please install Docker first.
```

**Fix:** Make sure Docker Desktop or Docker Engine is installed and running. On Windows, check that Docker Desktop is running in the system tray. On macOS, check the Docker menu bar icon. On Linux, run `docker ps` to verify the daemon is running.

### Error 2: `act` fails with "workflow not found"

```
Error: no workflows found in .github/workflows/
```

**Fix:** Make sure your workflow file is in the correct directory. The path should be `.github/workflows/` relative to your project root (where you run `act` from). Use `act --list` to see available workflows.

### Error 3: Secrets not available in `act`

```
Error: secret SLACK_WEBHOOK_URL is required but not provided
```

**Fix:** `act` does not have access to GitHub secrets by default. You can provide secrets in two ways:
- Create a `.secrets` file in your project root with `KEY=VALUE` pairs.
- Pass secrets via command line: `act -s SLACK_WEBHOOK_URL=my_webhook_url`

### Error 4: `act` uses a different base image than GitHub

```
The step works locally but fails on GitHub (or vice versa).
```

**Fix:** `act` uses community-built Docker images that mimic GitHub runners but are not identical. You can specify a specific image with the `-P` flag:
```bash
act -P ubuntu-latest=catthehacker/ubuntu:act-22.04
```

### Error 5: `actions/upload-artifact` fails in `act`

```
Error: Failed to create artifact: upload-artifact is not fully supported in act
```

**Fix:** Artifact upload/download support in `act` is limited. Use the `--artifact-server-path` flag to configure a local directory for artifacts:
```bash
act --artifact-server-path /tmp/artifacts
```

### Error 6: `gh run watch` shows "no runs found"

```
no runs found
```

**Fix:** Make sure the workflow has been triggered at least once. Use `gh workflow list` to verify the workflow exists, then `gh workflow run <workflow-name>` to trigger it. Wait a few seconds for the run to start before watching.
