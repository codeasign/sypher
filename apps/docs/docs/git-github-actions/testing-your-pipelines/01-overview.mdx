You pushed a branch, opened a PR, and waited. Ten minutes later, the CI run failed. You look at the logs and realize the `setup-node` action is failing because your workflow references Node 18 but the matrix only defines Node 20. This is a syntax error in your YAML — a variable name mismatch between `matrix.node-version` and the `node-version` input. You fix it, push again, wait another ten minutes, and the same thing happens with a different mismatch. Each iteration costs ten minutes of your time and ten minutes of runner time. The real problem is not the YAML mistake: it is that you pushed untested pipeline code to production.

## The Concept

Pipeline-as-code testing means applying the same rigor to your CI/CD workflow files that you apply to your application code: validate syntax, test individual steps in isolation, run the full workflow locally before pushing, and inspect intermediate state with artifacts. Think of it like a dress rehearsal before a Broadway show — the cast runs through every scene, tests the lighting cues, checks that the trap door opens on cue, and fixes problems before the paying audience arrives. Your CI pipeline is the same: the production run is the show, and you should rehearse it first.

## Command / Syntax Reference

| Command / Concept | Purpose | Example |
|---|---|---|
| `act` (nektos/act) | Runs GitHub Actions workflows locally using Docker containers. The primary tool for offline pipeline testing. | `act -j test -W .github/workflows/ci.yml` |
| `act --dry-run` | Simulates a workflow run without executing any steps. Validates the job graph, matrix expansion, and variable resolution. | `act --dry-run -j build` |
| `act --list` | Lists all available workflows and jobs that `act` can run. | `act --list` |
| `act -e <event.json>` | Runs a workflow with a custom event payload, useful for testing `workflow_dispatch` and `pull_request` triggers. | `act workflow_dispatch -e event.json` |
| `gh workflow run` | Triggers a workflow on GitHub from the CLI. Useful for manual testing after pushing. | `gh workflow run ci.yml -f branch=main` |
| `gh run watch` | Watches a workflow run in real-time from the terminal. | `gh run watch <run-id>` |
| `gh run view --log` | Views the full log output of a completed workflow run. | `gh run view <run-id> --log` |
| `actions/upload-artifact` / `actions/download-artifact` | Persists and retrieves files between jobs, enabling verification of intermediate pipeline state. | `uses: actions/upload-artifact@v4 with: name: build` |
| `workflow_dispatch` with inputs | Allows triggering workflows with custom parameters for targeted testing of specific jobs or scenarios. | `on: workflow_dispatch: inputs: debug: type: boolean` |
| Composite action local testing | Running a composite action by referencing its local path with `./` prefix before publishing. | `uses: ./.github/actions/my-action` |

## How It Actually Works — Step by Step

### Step 1: Writing the Workflow File

You write your workflow YAML in `.github/workflows/`. The file defines triggers, jobs, steps, and matrix strategies. At this stage, the file is just text on disk — it has not been validated or executed.

```yaml
# .github/workflows/ci.yml
name: CI
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:
    inputs:
      debug:
        type: boolean
        default: false

jobs:
  lint:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm run lint
```

### Step 2: Dry-Run Validation with `act --dry-run`

Before running anything, you validate the workflow graph. `act --dry-run` parses the YAML, resolves variables, expands the matrix, and prints the job graph without executing anything.

### Step 3: Running a Specific Job Locally with `act`

Once the dry run looks correct, you run a specific job with `act -j <job-name>`. This executes the job in a Docker container that mimics the GitHub Actions runner environment. You see real output, real exit codes, and real errors.

```bash
act -j lint -W .github/workflows/ci.yml
```

If the job passes locally, you can be confident it will pass on GitHub (assuming the Docker container accurately mirrors the GitHub runner). If it fails, you can iterate on the fix in seconds instead of minutes.

### Step 4: Testing Event Payloads with Custom Triggers

Some workflows behave differently depending on the triggering event. A `pull_request` workflow might run a different set of checks than a `push` workflow. You can test these by providing a custom event JSON file.

```json
{
  "action": "opened",
  "pull_request": {
    "head": { "ref": "feature-branch" },
    "base": { "ref": "main" },
    "number": 42
  }
}
```

```bash
act pull_request -e event.json -j test
```

### Step 5: Pushing and Testing with `gh` CLI

After local testing passes, you push to GitHub. The `gh` CLI lets you trigger a specific workflow and watch its output without leaving the terminal.

```bash
# Trigger a workflow with custom inputs
gh workflow run ci.yml -f debug=true

# Watch the run in real-time
gh run watch

# View logs after completion
gh run view --log
```

### Step 6: Verifying Intermediate State with Artifacts

Complex workflows often produce intermediate artifacts that need inspection. You can upload artifacts at any point and download them to verify correctness.

```yaml
- name: Build application
  run: npm run build

- name: Upload build output for inspection
  uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/

- name: Download and verify in downstream job
  uses: actions/download-artifact@v4
  with:
    name: build-output
    path: dist/
```

## When to Use This / When Not To

### Use Pipeline Testing Techniques When

- You are writing a new workflow from scratch and want to validate it before pushing.
- You are modifying an existing workflow and need to test the changes without disrupting the main branch.
- You are working with matrix builds and want to verify the matrix expansion produces the expected combinations.
- You are testing reusable workflows or composite actions that are consumed by multiple repositories.
- You need to debug a workflow that behaves differently on GitHub than you expect from reading the YAML.
- You want to verify artifact content between pipeline stages without deploying to production.

### Do Not Use Pipeline Testing Techniques When

- Your workflow is a simple one-step lint job that has been running unchanged for months.
- You do not have Docker installed on your machine (`act` requires Docker).
- You are working on a Windows workflow and `act` does not fully support the Windows runner image.
- You need to test GitHub-specific features that `act` cannot emulate (like deployment environments with protection rules).
- The workflow executes actions that require real GitHub tokens or secrets that you cannot expose locally.

### Pipeline Testing Strategy Matrix

| Testing Level | Tool | What It Catches | Time Cost |
|---|---|---|---|
| Syntax validation | `act --dry-run` | YAML errors, matrix expansion bugs, missing variables | 1 second |
| Single job | `act -j jobname` | Step ordering, action compatibility, script errors | 1-5 minutes |
| Full workflow | `act -W workflow.yml` | Job dependency issues, artifact passing, concurrency | 5-15 minutes |
| Event payload | `act trigger -e event.json` | Conditional step logic, trigger-specific behavior | 1-5 minutes |
| Production verification | `gh workflow run` | Runner-specific issues, secret availability, environment rules | 5-30 minutes |
| Post-mortem inspection | `gh run view --log` | Failure root cause, log output, timing | 1-2 minutes |
