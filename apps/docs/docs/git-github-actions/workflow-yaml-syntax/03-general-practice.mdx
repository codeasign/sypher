## Tier 1 — Beginner

### Exercise 1: Fix the indentation

**Task:** This workflow has indentation errors. Fix it so it passes YAML validation:

```yaml
name: Bad Indentation
on: push
jobs:
build:
  runs-on: ubuntu-latest
  steps:
  - run: echo "Hello"
```

<details>
<summary>Solution</summary>

```yaml
name: Bad Indentation
on: push
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello"
```

The key fix: `build` must be indented 2 spaces under `jobs`, and `steps` items must be indented 2 spaces under `steps`.
</details>

### Exercise 2: Add a matrix strategy

**Task:** Modify the workflow below so it runs on both `ubuntu-latest` and `macos-latest` using a matrix strategy:

```yaml
name: Single OS
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Running on ${{ runner.os }}"
```

<details>
<summary>Solution</summary>

```yaml
name: Matrix OS
on: workflow_dispatch

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
    steps:
      - run: echo "Running on ${{ matrix.os }}"
```

Expected output: Two parallel jobs, one per OS. The `runner.os` context is also available but `matrix.os` is more explicit for this case.
</details>

### Exercise 3: Add a comment

**Task:** Add a YAML comment to the workflow below explaining what the `if` condition does:

```yaml
name: Comment Practice
on: push
jobs:
  deploy:
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - run: echo "Deploying"
```

<details>
<summary>Solution</summary>

```yaml
name: Comment Practice
on: push
jobs:
  deploy:
    runs-on: ubuntu-latest
    # Only deploy when pushing to the main branch
    if: github.ref == 'refs/heads/main'
    steps:
      - run: echo "Deploying"
```

Comments in YAML start with `#`. They are ignored by the parser and by GitHub Actions.
</details>

## Tier 2 — Intermediate

### Exercise 4: Convert inline flow to block YAML

**Task:** This workflow uses inline flow syntax (`{}` and `[]`). Convert it to block YAML (with indentation and dashes):

```yaml
name: Inline Flow
on: {push: {branches: [main, develop]}}
jobs:
  build: {runs-on: ubuntu-latest, steps: [{run: "echo Building"}]}
```

<details>
<summary>Solution</summary>

```yaml
name: Inline Flow
on:
  push:
    branches:
      - main
      - develop
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building"
```

Block YAML is more verbose but much easier to read, especially as workflows grow. Use inline flow only for simple, short lists.
</details>

### Exercise 5: Use `fromJson` to build a dynamic matrix

**Task:** Create a workflow where the `generate` job outputs a JSON matrix as a string, and the `build` job uses `fromJson()${{ fromJson(needs.generate.outputs.matrix) }}` to consume it. The matrix should test Node.js versions 16, 18, and 20 on Ubuntu only.

<details>
<summary>Solution</summary>

```yaml
name: Dynamic Matrix
on: workflow_dispatch

jobs:
  generate:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: echo 'matrix={"node":[16,18,20]}' >> $GITHUB_OUTPUT

  build:
    needs: generate
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJson(needs.generate.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: node --version
```

Expected outcome: Three parallel jobs, each running `node --version` with a different Node version. The `generate` job runs first, produces the matrix JSON, and then 3 `build` jobs fan out.
</details>

### Exercise 6: Merge multiple anchors

**Task:** Create a workflow with two anchors: `&checkout` (contains the checkout step) and `&python-setup` (contains Python setup). Use `<<:` merge syntax to include both in two jobs — `lint` and `test`. The `test` job should also have an additional step that runs `pytest`.

<details>
<summary>Solution</summary>

```yaml
name: Anchor Merge
on: workflow_dispatch

checkout-setup: &checkout
  - uses: actions/checkout@v4

python-setup: &python-setup
  - uses: actions/setup-python@v5
    with:
      python-version: '3.11'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      <<: *checkout
      - *python-setup
      - run: pip install flake8 && flake8 .

  test:
    runs-on: ubuntu-latest
    steps:
      <<: *checkout
      - *python-setup
      - run: pip install pytest && pytest
```

Note: The `<<:` merge key merges a mapping into the current mapping. For sequences (like `steps`), you cannot use `<<:` — instead, you can reference the anchor directly with `- *anchor-name`. This is a YAML limitation: anchors work on both mappings and sequences, but `<<:` only works on mappings.
</details>

## Tier 3 — Advanced

### Exercise 7: Debug an invalid YAML file

**Scenario:** This workflow file fails to parse. Find and fix all errors:

```yaml
name: Broken YAML
on: push
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: |
          npm ci
      -name: Run tests
        run: npm test
      - run:  echo "completed"
      - uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: ./test-results/
```

<details>
<summary>Solution</summary>

There are at least 4 issues:

1. **`pull_request` is not indented under `on`**: `pull_request` is at the same level as `on` instead of being a child. It should be indented 2 spaces under `on`.

2. **Missing colon after `-name`**: `-name: Run tests` should be `- name: Run tests`. The space after the dash is required.

3. **Extra space in `run:`**: `- run:  echo "completed"` has two spaces after `run:`. While this is technically valid YAML (the extra space is just whitespace), it is inconsistent.

4. **`with:` nested incorrectly**: `with:` and its children need to be properly indented under `- uses:`.

Fixed version:

```yaml
name: Broken YAML
on:
  push:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: |
          npm ci
      - name: Run tests
        run: npm test
      - run: echo "completed"
      - uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: ./test-results/
```
</details>

### Exercise 8: Design a workflow that uses every YAML feature

**Scenario:** Design a single workflow file that demonstrates all of the following YAML features in a meaningful way (not just present, but each one serves a purpose):

- Mappings and sequences
- Multi-line string (`|`)
- Inline flow (`{}` or `[]`)
- YAML anchor (`&`) and alias (`*`)
- Expression (`${{ }}`)
- `if` condition
- `needs` dependency
- Job output

<details>
<summary>Solution</summary>

```yaml
name: YAML Feature Showcase
on: [push, workflow_dispatch]  # inline flow sequence

env: &global-env  # anchor definition
  NODE_ENV: test
  LOG_LEVEL: debug

jobs:
  prepare:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.get-version.outputs.version }}
    steps:
      - uses: actions/checkout@v4
      - id: get-version
        run: |  # multi-line string
          VERSION=$(node -e "console.log('1.0.0')")
          echo "version=$VERSION" >> $GITHUB_OUTPUT
          echo "Prepared version $VERSION"

  build:
    needs: prepare
    runs-on: ubuntu-latest
    env: *global-env  # anchor alias
    if: github.ref_name == 'main'  # expression + condition
    steps:
      - uses: actions/checkout@v4
      - run: echo "Building version ${{ needs.prepare.outputs.version }}"
```

This workflow uses:
- **Inline flow**: `[push, workflow_dispatch]` for the trigger list
- **Anchor/alias**: `&global-env` and `*global-env` to share environment variables
- **Multi-line string**: `|` for the version extraction script
- **Expression**: `${{ needs.prepare.outputs.version }}` to reference output
- **`if` condition**: Only builds on main
- **`needs`**: `build` waits for `prepare`
- **Job output**: `prepare` outputs the version string
</details>

### Exercise 9: Refactor a duplicated workflow

**Scenario:** The workflow below has the same 3 steps repeated in 4 jobs. Refactor it using YAML anchors and aliases to eliminate all duplication:

```yaml
name: Duplicated Steps
on: workflow_dispatch

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install flake8 && flake8 .

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install pytest && pytest

  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install mypy && mypy .

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install build && python -m build
```

<details>
<summary>Solution</summary>

```yaml
name: Unduplicated Steps
on: workflow_dispatch

checkout-and-python: &setup
  - uses: actions/checkout@v4
  - uses: actions/setup-python@v5
    with:
      python-version: '3.11'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - *setup
      - run: pip install flake8 && flake8 .

  test:
    runs-on: ubuntu-latest
    steps:
      - *setup
      - run: pip install pytest && pytest

  type-check:
    runs-on: ubuntu-latest
    steps:
      - *setup
      - run: pip install mypy && mypy .

  build:
    runs-on: ubuntu-latest
    steps:
      - *setup
      - run: pip install build && python -m build
```

The `&setup` anchor captures the shared checkout+Python setup sequence. Each job references it with `- *setup` (an alias). This reduces the file from 40 lines to 26 lines and makes it easy to change the Python version in one place.

Note: This uses a sequence anchor (`&setup` pointing to a list), not a mapping anchor. The `- *setup` syntax references the entire list and inserts it at that position.
</details>
