You open a pull request and the CI pipeline doesn't run. You check the workflow file — it looks fine. You check the Actions tab — "Invalid workflow file." You stare at the YAML for 20 minutes before realizing a missing colon on line 7 broke the entire file. YAML is unforgiving of syntax mistakes, and GitHub Actions adds its own rules on top. Understanding the exact syntax — not just the concepts — is the difference between a workflow that runs and one that silently fails.

## The Concept

A GitHub Actions workflow is a YAML file with a specific structure: top-level keys define the workflow's name, triggers, jobs, and configuration. YAML uses indentation (2 spaces per level) to define nesting, and GitHub Actions adds reserved keywords like `on`, `jobs`, `steps`, `uses`, and `run` that must appear at specific nesting levels.

Think of it as a blueprint for a building. The top-level keys are the floors (foundation, ground floor, first floor). Each floor has rooms (jobs), and each room has furniture (steps). The blueprint language (YAML) has strict rules about how you draw walls and doors (indentation, colons, dashes). One misplaced line and the building inspector (GitHub) rejects the whole plan.

## Key YAML Syntax Elements

| Element | Syntax | Purpose |
|---------|--------|---------|
| Mapping (dict) | `key: value` | Key-value pair with a colon and space |
| Sequence (list) | `- item` | Dash-space prefix for each list item |
| Nested mapping | `key:\n  subkey: value` | Indent 2 spaces for each nesting level |
| Multi-line string | `key: \|` or `key: >` | `\|` preserves newlines, `>` folds lines |
| Inline flow | `{key: value, key2: value2}` | Compact JSON-like syntax for simple mappings |
| Comments | `# comment` | Ignored by YAML parser |
| Anchors | `&anchor` and `*anchor` | Reuse a block of YAML elsewhere |
| Boolean | `true`, `false`, `yes`, `no` | Case-insensitive; be careful with `on` and `off` |
| String quoting | `"value"` or `'value'` | Double quotes for escape sequences, single for literal |
| `${{ }}` | Expression syntax | GitHub Actions expressions, not YAML |
| `!` tag | `!reference` | Used by some actions for type tags |

## How It Actually Works — Step by Step

### Step 1: GitHub reads the YAML file

When a trigger event fires, GitHub locates every `.yml` and `.yaml` file in the `.github/workflows/` directory. It parses each file using the standard YAML 1.1 parser (the `yaml` Go library) and converts the YAML document into an internal data structure.

### Step 2: The parser validates YAML structure

The YAML parser checks basic syntax: every key has a colon, indentation is consistent (you cannot mix 2-space and 4-space indentation in the same file), and list items are properly marked with `- `. If the YAML is invalid, GitHub immediately reports the error and does not attempt to parse further.

### Step 3: GitHub validates the Actions-specific schema

After basic YAML parsing, GitHub checks the parsed structure against the Actions workflow schema:

- `on` must be present and contain valid event names
- `jobs` must be a map, not a scalar
- `jobs.<id>.runs-on` must be a string or list of strings
- `steps[*]` must contain either `uses` or `run` (or both)

If the schema validation fails, the error is shown in the Actions tab with a line number.

### Step 4: Expression interpolation

Before any step runs, GitHub evaluates expressions inside `${{ }}` markers. These expressions can reference:
- `github.*` — context about the event and repository
- `env.*` — environment variable values
- `job.*` — current job status
- `steps.*` — outputs from previous steps
- `secrets.*` — encrypted secrets
- `matrix.*` — matrix strategy variables

### Step 5: The workflow runner executes the parsed steps

The runner receives the fully resolved YAML — with all expressions evaluated — and begins executing steps in order. Each step's `run` command is passed to the shell (bash by default on Linux/macOS, PowerShell on Windows), and `uses` actions are downloaded from the Marketplace or a GitHub repository.

### Step 6: Job outputs and status propagation

After steps complete, the runner collects:
- Step-level outputs (from `echo "::set-output name=key::value"` or `$GITHUB_OUTPUT`)
- Job-level status (success, failure, cancelled, skipped)
- These values flow to dependent jobs via `needs` and `if` conditions

## When to Use YAML Syntax Features / When Not To

### Use anchors (`&`) and aliases (`*`) when:
- You have repeated blocks of YAML in multiple jobs (e.g., the same checkout + setup steps)
- You want to define a shared "defaults" block and reference it in several places

### Do not use anchors when:
- The workflow is simple (fewer than 3 jobs) — anchors add indirection that makes debugging harder
- You are working with a team unfamiliar with YAML anchors — explicit duplication is clearer

### Use `${{ }}` expressions when:
- You need to reference dynamic values like branch names, commit SHAs, or environment variables
- You need to conditionally skip a step based on a previous step's output

### Do not use expressions when:
- A static string works — `${{ }}` adds parsing overhead and is not needed for hardcoded values
- Inside `with:` inputs that accept booleans — use `true`/`false` without quotes, not `${{ true }}`

### Use multi-line `|` when:
- A `run:` step has multiple shell commands that should run sequentially
- You want to keep the script readable with proper line breaks

### Use `>` when:
- You have a long string that should be a single line in the output but is hard to read as one line in the file
