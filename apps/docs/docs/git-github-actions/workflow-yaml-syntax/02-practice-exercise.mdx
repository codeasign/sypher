You will create a workflow file from scratch, progressively adding complexity as you learn each YAML structure. By the end you will have a workflow that uses mappings, sequences, multi-line strings, expressions, and conditional logic.

## The Walkthrough

### Step 1: Write the simplest valid workflow

Create `.github/workflows/simple.yml`:

```yaml
name: Simple Workflow
on: workflow_dispatch
jobs:
  hello:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello, YAML!"
```

This is the minimum viable workflow. It has a name (`Simple Workflow`), a single trigger (`workflow_dispatch` for manual run), one job (`hello`), and one step that prints text.

**Checkpoint:** Validate the YAML syntax locally:

```bash
# On macOS/Linux — use Python's yaml parser
python3 -c "import yaml; yaml.safe_load(open('.github/workflows/simple.yml')); print('Valid YAML')"
```

If you get "Valid YAML," the file is syntactically correct.

### Step 2: Add multiple triggers using sequence syntax

Replace `on: workflow_dispatch` with a list of triggers. Create `.github/workflows/multi-trigger.yml`:

```yaml
name: Multi-Trigger
on:
  push:
    branches:
      - main
      - develop
  pull_request:
    types:
      - opened
      - synchronize
  workflow_dispatch:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Linting on branch ${{ github.ref_name }}"
```

Notice the YAML structure: `on` is a mapping, not a scalar. `push` and `pull_request` are sub-keys under `on`. `branches` is a list (sequence) under `push`.

**Checkpoint:**

```bash
python3 -c "import yaml; yaml.safe_load(open('.github/workflows/multi-trigger.yml')); print('Valid YAML')"
```

### Step 3: Add a multi-line run step

Create `.github/workflows/multiline.yml`:

```yaml
name: Multi-Line Step
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Multi-line script
        run: |
          echo "Step 1: Installing dependencies"
          pip install -r requirements.txt 2>/dev/null || echo "No requirements file"
          echo "Step 2: Running tests"
          python app.py
          echo "Step 3: Done"
```

The `|` character after `run:` tells YAML that the following indented block is a literal multi-line string. Each line is executed as a separate shell command.

### Step 4: Use expressions to pass dynamic values

Create `.github/workflows/expressions.yml`:

```yaml
name: Expression Demo
on: workflow_dispatch

jobs:
  show-context:
    runs-on: ubuntu-latest
    steps:
      - name: Show event information
        run: |
          echo "Triggered by: ${{ github.actor }}"
          echo "Branch: ${{ github.ref_name }}"
          echo "Repository: ${{ github.repository }}"
          echo "Run ID: ${{ github.run_id }}"
      - name: Conditional step
        if: github.ref_name == 'main'
        run: echo "This only runs on main"
```

**Expected output when run on main:**

```
Triggered by: your-username
Branch: main
Repository: your-username/yaml-syntax-practice
Run ID: 12345
This only runs on main
```

**Expected output when run on a feature branch:**
```
Triggered by: your-username
Branch: feature/my-feature
Repository: your-username/yaml-syntax-practice
Run ID: 12346
```
The last line is absent because the `if` condition evaluated to `false`.

### Step 5: Use strategy matrix and job outputs

Create `.github/workflows/matrix-with-outputs.yml`:

```yaml
name: Matrix with Outputs
on: workflow_dispatch

jobs:
  generate:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: echo "matrix={\"os\":[\"ubuntu-latest\",\"macos-latest\"],\"node\":[18,20]}" >> $GITHUB_OUTPUT

  build:
    needs: generate
    runs-on: ${{ matrix.os }}
    strategy:
      matrix: ${{ fromJson(needs.generate.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - name: Running on ${{ matrix.os }}
        run: echo "Building with Node ${{ matrix.node }} on ${{ matrix.os }}"
```

This demonstrates dynamic matrix generation. The `generate` job produces a JSON string that defines the matrix. The `build` job uses `fromJson()` to parse it into a real matrix, producing 4 combinations (2 OSes x 2 Node versions).

### Step 6: Use re-usable YAML anchors

Anchors let you define a block once and reuse it. Create `.github/workflows/anchors.yml`:

```yaml
name: YAML Anchors Demo
on: workflow_dispatch

defaults: &global-defaults
  run:
    shell: bash

jobs:
  job-a:
    <<: *global-defaults
    runs-on: ubuntu-latest
    steps:
      - run: echo "Job A uses the global defaults"

  job-b:
    <<: *global-defaults
    runs-on: ubuntu-latest
    steps:
      - run: echo "Job B also uses bash shell via the anchor"
```

**Checkpoint:**

```bash
python3 -c "
import yaml
data = yaml.safe_load(open('.github/workflows/anchors.yml'))
print('Valid YAML')
print('Jobs:', list(data.get('jobs', {}).keys()))
"
```

## Common Errors at This Stage

### Error 1: "Invalid value" for `on`
**Message:** `"on" must be a string, an object, or a list of strings.`

**Why:** You wrote `on:` with no value, or you used a scalar where a list was expected.

**Fix:** Always provide a valid trigger:
```yaml
# Bad
on:

# Good — scalar (single trigger)
on: push

# Good — list (multiple triggers)
on: [push, pull_request]

# Good — object with options
on:
  push:
    branches: [main]
```

### Error 2: "Steps must contain either 'uses' or 'run'"
**Message:** `Each step in a job must contain a "uses" or "run" key.`

**Why:** You created an empty step (no `uses` or `run`), or you mis-indented a step's content.

**Fix:** Every step needs either `uses`, `run`, or both:
```yaml
steps:
  - name: My step
    # Missing: uses or run
```

```yaml
steps:
  - name: My step
    run: echo "Fixed!"
```

### Error 3: "Mapping values are not allowed here"
**Message:** `mapping values are not allowed in this context` from the YAML parser.

**Why:** A colon is in the wrong place or indentation is inconsistent.

**Fix:** Check that you have exactly 2 spaces per nesting level and every key is followed by `: `:
```yaml
# Bad
jobs:
  build:
  runs-on: ubuntu-latest  # Wrong: same indent as build

# Good
jobs:
  build:
    runs-on: ubuntu-latest  # 2 spaces deeper than build
```

### Error 4: Expression not evaluated — shows literally as `${{ ... }}`
**Message:** The log literally shows `${{ github.actor }}` instead of the username.

**Why:** The expression is inside single quotes `'${{ ... }}'` in a `run:` command. Single quotes prevent shell evaluation, but in GitHub Actions, expressions are evaluated by the runner before the shell sees them. However, if the entire `run:` value is quoted in single quotes in the YAML, the `$` is still evaluated by Actions — so this error is actually more subtle. Make sure the expression is not inside a YAML double-quoted string where `$` would need escaping:
```yaml
# Bad
- run: echo '${{ github.actor }}'  # Shell gets the literal string '${{...}}'

# Good
- run: echo "${{ github.actor }}"  # Shell expands properly
- run: echo ${{ github.actor }}    # Also works (but less safe with spaces)
```

## What You Just Learned

- YAML uses 2-space indentation to define nesting in workflow files
- `on`, `jobs`, `steps`, `uses`, and `run` are the core structural keys
- `|` creates multi-line literal strings for multi-command steps
- `${{ }}` expressions reference contexts like `github`, `env`, and `matrix`
- YAML anchors (`&`/`*`) and merge keys (`<<:`) reduce duplication
- The `fromJson()` function converts strings to objects for dynamic configuration
- Every step needs either a `run` command or a `uses` action reference
