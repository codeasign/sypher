## Setup

For this walkthrough, you need a GitHub repository with GitHub Actions enabled. You also need Node.js 18 or later installed locally. If you do not have a repository ready, create one now.

```bash
# Create a new repository directory and initialize it
mkdir quality-gates-demo
cd quality-gates-demo
git init
```

```bash
# Create a minimal Node.js project
npm init -y
```

```bash
# Install a linter and test framework
npm install --save-dev eslint jest
```

```bash
# Initialize ESLint
npx eslint --init
```

You also need to push this repository to GitHub. Create a repository on GitHub (do not initialize it with a README) and then run:

```bash
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/YOUR-USERNAME/quality-gates-demo.git
git push -u origin main
```

## The Walkthrough

You will build a quality-gated pipeline from scratch. Each step adds a gate. By the end, your pipeline will have five gates: lint, format, unit test, integration test, and deployment approval.

### Step 1: Create the Trigger Gate

The first gate decides when the pipeline runs at all. Create a workflow file that only triggers on pushes to main and pull requests targeting main.

```bash
mkdir -p .github/workflows
```

Create the file `.github/workflows/quality-gates.yml`:

```yaml
name: Quality Gates
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx eslint src/
```

Commit and push this file:

```bash
git add .github/workflows/quality-gates.yml
git commit -m "Add trigger gate for main branch"
git push origin main
```

Expected output:

```
remote: Create a pull request for 'main' on GitHub by visiting:
remote:      https://github.com/YOUR-USERNAME/quality-gates-demo/pull/new/main
To https://github.com/YOUR-USERNAME/quality-gates-demo.git
 * [new branch]      main -> main
```

After pushing, go to the Actions tab in your GitHub repository. You should see the workflow run.

### Step 2: Add a Lint Gate with Fail-Fast

Add a lint job that runs ESLint on all source files. If linting fails, no further jobs run. This is the fail-fast principle.

Update `.github/workflows/quality-gates.yml`:

```yaml
name: Quality Gates
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx eslint src/ --max-warnings 0

  format-check:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx prettier --check src/
```

The `needs: lint` line creates the dependency chain. The `--max-warnings 0` flag tells ESLint to treat warnings as errors. This is a quality gate: no warnings allowed.

Commit and push:

```bash
git add .github/workflows/quality-gates.yml
git commit -m "Add lint gate with fail-fast dependency"
git push origin main
```

Checkpoint: The workflow should have two jobs. The format-check job should show a pending status until lint finishes. If lint fails, format-check is skipped.

### Step 3: Add a Unit Test Gate with Coverage Threshold

Now add a unit test job that enforces a minimum coverage threshold. This is a numeric quality gate: if coverage drops below 80%, the pipeline fails.

Create a sample source file and test:

```bash
mkdir src
```

Create `src/calculator.js`:

```javascript
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

function divide(a, b) {
  if (b === 0) {
    throw new Error("Division by zero");
  }
  return a / b;
}

module.exports = { add, subtract, multiply, divide };
```

Create `src/calculator.test.js`:

```javascript
const { add, subtract, multiply, divide } = require("./calculator");

describe("Calculator", () => {
  test("adds two numbers", () => {
    expect(add(2, 3)).toBe(5);
  });

  test("subtracts two numbers", () => {
    expect(subtract(10, 4)).toBe(6);
  });

  test("multiplies two numbers", () => {
    expect(multiply(3, 4)).toBe(12);
  });

  test("divides two numbers", () => {
    expect(divide(10, 2)).toBe(5);
  });

  test("throws on division by zero", () => {
    expect(() => divide(5, 0)).toThrow("Division by zero");
  });
});
```

Update the workflow to add the test gate:

```yaml
name: Quality Gates
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx eslint src/ --max-warnings 0

  format-check:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx prettier --check src/

  unit-tests:
    needs: format-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx jest --coverage --coverageThreshold='{"global":{"lines":80}}'
```

Commit and push:

```bash
git add .
git commit -m "Add unit test gate with coverage threshold"
git push origin main
```

Expected output in the Actions tab:

```

  Calculator
    ✓ adds two numbers
    ✓ subtracts two numbers
    ✓ multiplies two numbers
    ✓ divides two numbers
```

### Step 4: Add an Integration Test Gate with Timeout

Integration tests are slower and more expensive. Add a timeout gate to prevent runaway tests from consuming runner minutes.

Create `src/api.test.js`:

```javascript
function fetchUser(id) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (id > 0) {
        resolve({ id, name: "Alice" });
      } else {
        reject(new Error("Invalid user ID"));
      }
    }, 100);
  });
}

describe("User API", () => {
  test("fetches a user by ID", async () => {
    const user = await fetchUser(1);
    expect(user.name).toBe("Alice");
  });

  test("rejects invalid ID", async () => {
    await expect(fetchUser(-1)).rejects.toThrow("Invalid user ID");
  });
});
```

Update the workflow:

```yaml
  integration-tests:
    needs: unit-tests
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx jest --testMatch='**/*.integration.test.js' --forceExit
```

Since the test file is named `.test.js` and not `.integration.test.js`, you can also run all tests. For now, just run jest against all test files:

```yaml
  integration-tests:
    needs: unit-tests
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npx jest
```

Commit and push:

```bash
git add .
git commit -m "Add integration test gate with timeout"
git push origin main
```

### Step 5: Add a Conditional Deployment Gate

The final gate is a deployment job that only runs on the main branch and only when all previous jobs pass. It also uses a GitHub environment so you can add required reviewers later.

```yaml
  deploy:
    needs: integration-tests
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npm run build --if-present
      - run: echo "Deploying to production..."
```

The `if` condition is a quality gate: deployment only happens on pushes to main. Pull request runs never deploy. The `environment: production` line creates a manual approval gate if you configure required reviewers in the repository settings.

To set up the environment, go to your GitHub repository, click Settings, then Environments, then create a new environment called `production`. Add a required reviewer (yourself or a teammate).

Commit and push:

```bash
git add .
git commit -m "Add conditional deployment gate"
git push origin main
```

Create a pull request to test the pipeline end to end. Create a new branch, make a change, and open a PR.

```bash

```

Modify `src/calculator.js` to introduce a deliberate lint error:

```javascript
function add(a,b){
  return a + b;
}
```

Commit and push:

```bash

```

Open a pull request on GitHub from `test-quality-gates` to `main`. The pipeline should trigger, and the lint job should fail.

## Common Errors at This Stage

### Error 1: Job fails because the dependency chain is not satisfied

```
Error: This job is waiting for the lint job to pass
```

The `needs` key creates a hard dependency. If the job name in `needs` does not match exactly, the workflow fails validation. The names are case-sensitive and must match the job keys in the YAML file.

Fix: Verify that the job name in `needs:` matches the exact job key name. For example, `needs: lint` matches the job key `lint:` but not `Lint:` or `lint-job:`.

### Error 2: ESLint fails with "Parsing error: The keyword 'const' is reserved"

```
D:\projects\quality-gates-demo\src\calculator.js
  1:1  error  Parsing error: The keyword 'const' is reserved
```

This happens when ESLint is configured for an older ECMAScript version. Your ESLint config file (`.eslintrc.json`) likely defaults to ES5.

Fix: Update `.eslintrc.json` to set `ecmaVersion` to the latest:

```json
{
  "parserOptions": {
    "ecmaVersion": "latest"
  }
}
```

### Error 3: Jest cannot find the coverage threshold

```
Invalid coverage threshold configuration: {"global":{"lines":80}}
```

The JSON syntax in the command line can be tricky. The shell may interpret the quotes differently.

Fix: Move the coverage threshold to `package.json`:

```json
{
  "jest": {
    "coverageThreshold": {
      "global": {
        "lines": 80
      }
    }
  }
}
```

Then run jest without the command-line flag:

```bash
npx jest --coverage
```

### Error 4: The deploy job runs on pull requests when it should not

The deploy job uses `if: github.ref == 'refs/heads/main'` but this check alone is not enough. Pull requests from a branch named `main` would also match.

Fix: Add the event name check to the condition:

```yaml
if: github.ref == 'refs/heads/main' && github.event_name == 'push'
```

This ensures deployment only happens on direct pushes to main, not on pull requests targeting main.

## What You Just Learned

- You structured a CI/CD pipeline as a series of five quality gates, each with a specific enforcement role.
- You used `needs` to create a dependency chain that stops the pipeline when any gate fails.
- You added a coverage threshold gate that enforces a minimum 80% line coverage.
- You added a timeout gate that prevents runaway integration tests from wasting runner minutes.
- You added a conditional deployment gate that only runs on the main branch.
- You configured a GitHub environment with required reviewers as a manual approval gate.
- You learned how to debug common pipeline errors related to dependencies, parser configuration, coverage thresholds, and conditional logic.
