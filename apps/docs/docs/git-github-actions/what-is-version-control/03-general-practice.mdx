These exercises reinforce the version control concepts you learned. Work through them in order. Each tier builds on the previous one.

## Tier 1: Beginner

These exercises focus on single commands. Complete each one in a terminal with Git available.

### Exercise 1.1: Initialize a Repository

**Task:** Create a new directory called `my-scripts` and initialize it as a Git repository.

<details>
<summary>Hint</summary>

Use `mkdir` to create the directory, `cd` to enter it, and `git init` to initialize the repository. Verify with `git status`.
</details>

<details>
<summary>Solution</summary>

```bash
mkdir my-scripts
cd my-scripts
git init
# Expected output: Initialized empty Git repository in .../my-scripts/.git/
git status
# Expected output: On branch master, No commits yet, nothing to commit
```

After running these commands, the `my-scripts` directory contains a `.git` subdirectory. The repository is ready to track files.
</details>

### Exercise 1.2: Stage a File Without Committing

**Task:** Inside the `my-scripts` repository you created in Exercise 1.1, create a file named `hello.sh` containing `echo "Hello World"` and stage it with `git add`. Do not commit it yet. Verify the file is staged.

<details>
<summary>Hint</summary>

Use `echo "echo Hello World" > hello.sh` to create the file. Use `git add hello.sh` to stage it. Use `git status` to verify it appears under "Changes to be committed."
</details>

<details>
<summary>Solution</summary>

```bash
# Create the file
echo "echo Hello World" > hello.sh
# Stage it
git add hello.sh
# Verify it is staged
git status
# Expected output: Shows hello.sh under "Changes to be committed"
```

The file is now in the staging area. If you run `git commit`, this file would be included in the snapshot. Until you commit, the file is not yet part of the permanent history.
</details>

### Exercise 1.3: View Commit History in Compact Format

**Task:** Navigate to the `~/git-first-repo` repository you created in the walkthrough (or create a new repository with two commits). View the commit history using the compact one-line format.

<details>
<summary>Hint</summary>

Use `git log --oneline` to see commits in compact format. If you do not have the walkthrough repository, create one: `mkdir test-repo && cd test-repo && git init && echo "a" > a.txt && git add a.txt && git commit -m "first" && echo "b" > b.txt && git add b.txt && git commit -m "second"`.
</details>

<details>
<summary>Solution</summary>

```bash
# Navigate to a repository with commits
cd ~/git-first-repo
# View compact history
git log --oneline
# Expected output:
# c8d9e0f (HEAD -> master) Add hello.py script
# e5f6g7a Add project description to README
# a1b2c3d Initial commit: add README.md
```

Each line shows the abbreviated commit hash (7 characters) and the commit message. The `HEAD -> master` marker indicates that the latest commit is the one your working directory is based on.
</details>

### Exercise 1.4: Detect Modified Files

**Task:** In any repository with at least one committed file, modify the file by adding a new line of text. Run `git status` to confirm the file is modified. Then run `git diff` to see the exact changes.

<details>
<summary>Hint</summary>

Use `echo "new line" >> filename` to append a line to an existing file. Then run `git status` to see the file listed as "modified." Then run `git diff filename` to see the exact lines added.
</details>

<details>
<summary>Solution</summary>

```bash
# Append a line to the README
echo "A new line of content" >> README.md
# Check status
git status
# Output: README.md listed under "Changes not staged for commit"
# View the diff
git diff README.md
# Output shows the + line you added
```

The `git diff` command shows the difference between your working directory and the last committed version. Lines starting with `+` are additions. Lines starting with `-` would be deletions. This is how you inspect changes before staging them.
</details>

## Tier 2: Intermediate

These exercises combine multiple commands into a single workflow.

### Exercise 2.1: Complete Edit-Stage-Commit Cycle

**Task:** Create a new repository called `project-alpha`. Create a file called `index.html` with the content `<h1>Project Alpha</h1>`. Stage it and commit it with the message "Add index.html." Then modify the file to add a second line `<p>Version 2</p>`, view the diff, stage the change, and commit again. Finally, view the full history.

<details>
<summary>Hint</summary>

You need to run the cycle twice: once for the initial file creation, once for the modification. Between the two cycles, use `git diff` to inspect the changes before staging. Use `git log --oneline` to see both commits at the end.
</details>

<details>
<summary>Solution</summary>

```bash
# Create and initialize
mkdir project-alpha
cd project-alpha
git init

# First cycle: create and commit
echo "<h1>Project Alpha</h1>" > index.html
git add index.html
git commit -m "Add index.html"

# Second cycle: modify and commit
echo "<p>Version 2</p>" >> index.html
git diff index.html
# Expected: shows the added line
git add index.html
git commit -m "Add paragraph to index.html"

# View history
git log --oneline
# Expected: 2 commits shown
```

You have completed the full edit-stage-commit cycle twice. Each cycle created a permanent snapshot in the repository's history. The `git diff` command between the cycles let you inspect the exact change before committing.
</details>

### Exercise 2.2: Selective Staging

**Task:** Create a repository called `selective-repo`. Create three files: `app.py`, `utils.py`, and `tests.py`. Stage only `app.py` and `utils.py`. Commit with the message "Add app and utils modules." Then check that `tests.py` remains untracked. Finally, stage and commit `tests.py` separately.

<details>
<summary>Hint</summary>

Use `git add` with specific filenames to stage only the files you want. Do not use `git add .` because that would stage all files. After the first commit, run `git status` to confirm `tests.py` is still untracked.
</details>

<details>
<summary>Solution</summary>

```bash
mkdir selective-repo
cd selective-repo
git init

# Create three files
echo "def main():" > app.py
echo "    pass" >> app.py
echo "def helper():" > utils.py
echo "    pass" >> utils.py
echo "def test_main():" > tests.py
echo "    assert True" >> tests.py

# Stage only two files
git add app.py utils.py
git status
# Expected: app.py and utils.py staged, tests.py untracked

git commit -m "Add app and utils modules"
# Expected: 2 files changed

git status
# Expected: tests.py still untracked

# Now stage and commit the third file
git add tests.py
git commit -m "Add tests module"
# Expected: 1 file changed

git log --oneline
# Expected: 2 commits
```

Selective staging lets you build commits that contain related changes. This is a key practice for maintaining a clean project history. Each commit should represent a single logical change.
</details>

### Exercise 2.3: Compare Two Commits

**Task:** Use the `project-alpha` repository from Exercise 2.1. View the differences between the first commit and the most recent commit. Identify which lines were added between them.

<details>
<summary>Hint</summary>

First find the commit hashes using `git log --oneline`. Then use `git diff <hash1>..<hash2>` to compare the two commits. The first hash should be the older commit, the second the newer one.
</details>

<details>
<summary>Solution</summary>

```bash
cd project-alpha

# Find the commit hashes
git log --oneline
# Example output:
# b2c3d4e (HEAD -> master) Add paragraph to index.html
# a1b2c3d Add index.html

# Compare the first and second commits
git diff a1b2c3d..b2c3d4e
# Expected: shows the + line for the paragraph added
```

The `git diff` with two commit hashes shows the aggregate changes between those two points in history. This is useful for reviewing what changed between releases or identifying when a bug was introduced.
</details>

### Exercise 2.4: Clone a Repository and Explore Its History

**Task:** Clone a public repository from GitHub and explore its history. Use `https://github.com/git/git-scm.com.git` (the Git website repository) or any other public repository. After cloning, view the commit history in `--oneline` format, view the latest 5 commits, and check the status.

<details>
<summary>Hint</summary>

Use `git clone <url>` to clone. This creates a new directory with the repository name. Use `cd` to enter it. Use `git log --oneline -5` to see the latest 5 commits. Use `git status` to see the current state.
</details>

<details>
<summary>Solution</summary>

```bash
# Clone a public repository
git clone https://github.com/git/git-scm.com.git
cd git-scm.com

# View the latest 5 commits
git log --oneline -5
# Expected: shows 5 commits

# View the full graph
git log --oneline --graph --all | head -10
# Expected: shows branching structure

# Check status
git status
# Expected: On branch main, nothing to commit
```

Cloning downloads the entire repository — every commit, every branch, every tag. You now have a full copy of the project's history on your machine. You can browse it, search it, and even create new branches, all without an internet connection.
</details>

## Tier 3: Advanced

These exercises present ambiguous real-world scenarios. There is no single correct answer. The solution walks through one strong approach and notes that alternatives exist.

### Exercise 3.1: Recovering from Accidental Sensitive Data in a Commit

**Scenario:** You are working on a web application. You created a repository, added all your files, and made your first commit. Five minutes later you realize that one of the files, `config.py`, contains a hardcoded API key. You have already committed it. The commit has not been pushed to any remote repository yet. How do you remove the API key from the commit history?

**Additional context:** The `config.py` file also contains other settings that you need to keep. You cannot simply delete the file. You need to edit it and then update the commit.

**Consider:**
- What is the simplest approach that removes the secret from the commit?
- What if you had already pushed the commit to GitHub?
- What if other people have already cloned the repository?

<details>
<summary>Hint</summary>

You have not pushed yet, so the history is only on your machine. You can edit the file to remove the API key and then amend the commit using `git commit --amend`. However, `--amend` only changes the most recent commit. If the secret is in an older commit, you need `git rebase` or `git filter-branch`. For this scenario, assume the secret is in the most recent commit.
</details>

<details>
<summary>Solution</summary>

**One strong approach: amend the commit**

Since you caught the mistake immediately and the commit is the most recent one, you can fix it with `git commit --amend`:

```bash
# 1. Edit config.py to remove the API key
# Open config.py in your editor and delete the line with the API key

# 2. View the diff to confirm the change
git diff config.py
# Shows the API key line being removed

# 3. Stage the fixed file
git add config.py

# 4. Amend the previous commit
git commit --amend
# This opens your editor to modify the commit message
# Save and close to create the replacement commit
```

After `git commit --amend`, Git creates a NEW commit that replaces the original. The commit hash changes. The original commit (with the API key) is no longer referenced and will eventually be cleaned up by Git's garbage collection.

**Why this works:** `git commit --amend` takes the current staging area and uses it to create a new commit, replacing the previous commit's parent pointer. The old commit is orphaned.

**What if the commit had already been pushed?** Do NOT use `--amend` on a pushed commit if others have based work on it. Instead, follow these steps:
1. Remove the API key from `config.py` and commit the fix as a new commit.
2. Use `git push` normally.
3. The API key is still visible in the history, so you should also rotate the key (generate a new one).

**What if the secret is in an older commit (not the most recent)?**
- If not pushed: use `git rebase -i` to edit the specific commit.
- If pushed: use `git filter-branch` or `git filter-repo` to rewrite history, and coordinate with all team members to rebase their work on the new history.

**Critical note:** Amending or rebasing commits that have been pushed to a shared repository causes problems for anyone else who has cloned the repository. They will see divergent histories and need to `git pull --rebase` or re-clone. Communicate with your team before rewriting shared history.

**Alternative approaches:**
- Use `git reset --soft HEAD~1` to undo the commit while keeping the files staged, then fix the file and commit again.
- Use `git restore --staged` and `git reset` for a more manual approach.
- For deeply buried secrets, use `git filter-repo` (a dedicated tool) or `BFG Repo-Cleaner` (a Java-based tool).
</details>

### Exercise 3.2: Retroactively Versioning an Existing Project

**Scenario:** You have been working on a data analysis project for two weeks. The project directory is at `/home/you/data-analysis/` and contains 15 files. You did not use version control. You have made significant changes over the two weeks — you have modified files, deleted some, and added new ones. Your project is working correctly now, but you have no history. How do you start tracking this project with Git while preserving as much context as possible?

**Consider:**
- Can you create a meaningful history after the fact?
- What information do you lose by starting version control now?
- Should you create one commit with everything, or split the history into logical stages?

<details>
<summary>Hint</summary>

You cannot recreate the exact history because you did not save intermediate versions. But you can create a meaningful starting point. The simplest approach is to initialize a repository and commit the current state. If you have backup copies, email attachments, or notes from different stages, you can reconstruct a more detailed history.
</details>

<details>
<summary>Solution</summary>

**One strong approach: initialize and commit the current state**

The most pragmatic approach is to start tracking from the current state and move forward:

```bash
# 1. Navigate to the project directory
cd /home/you/data-analysis

# 2. Initialize the repository
git init

# 3. Create a .gitignore file to exclude generated files
echo "*.pyc" > .gitignore
echo "__pycache__/" >> .gitignore
echo ".env" >> .gitignore
echo "data/raw/*.csv" >> .gitignore

# 4. Stage all project files
git add .

# 5. Commit the current state
git commit -m "Initial commit: data analysis project"
```

**What you gain:** From this point forward, every change is tracked. You can experiment, refactor, and roll back with confidence.

**What you lose:**
- You cannot see what changed during the first two weeks.
- You cannot revert to a specific earlier version.
- You cannot attribute specific changes to specific dates or authors.

**If you want a richer initial history (alternative approach):**

If you have backup copies, email attachments, or notes from different stages, you can reconstruct a more detailed history:

```bash
# Create a temporary directory for reconstruction
mkdir /tmp/reconstruction
cd /tmp/reconstruction
git init

# Stage and commit each known version in order
# Version 1: initial prototype (from backup or memory)
cp -r /backups/data-analysis-v1/* .
git add .
git commit -m "Initial prototype: data loading and basic stats"

# Version 2: added visualization (from a different backup)
cp -r /backups/data-analysis-v2/* .
git add .
git commit -m "Add visualization module with matplotlib charts"

# Version 3: current state
cp -r /home/you/data-analysis/* .
git add .
git commit -m "Add regression analysis and report generation"
```

**Why this matters:** A reconstructed history gives you the ability to:
- Use `git bisect` to find when a bug was introduced (if you can identify the version where the bug appeared).
- Use `git blame` to see when each line was added (though the attribution will be for the reconstruction date).
- Create branches from known points in the project's evolution.

**Critical note:** Be honest about reconstructed history. If you are committing code that was written over a period of time, use commit messages that acknowledge the reconstruction. Do not backdate commits unless you are certain of the dates, and understand that `git log` timestamps are metadata that can be verified.

**Alternative approaches:**
- Use `git init` with a single initial commit and then use `git log --after` and `git log --before` to filter by date going forward.
- If you have a CI/CD system or build artifacts, check if they contain timestamps that can help reconstruct the timeline.
- Use `git stash` to save work-in-progress while you set up the initial commit.
</details>

### Exercise 3.3: Choosing Between Local, Centralized, and Distributed Version Control

**Scenario:** You are advising a small team of three designers who work on large Adobe Illustrator files (50 MB to 200 MB each). They need to collaborate on these files, see who changed what, and revert to previous versions. They have tried Dropbox but it does not provide meaningful version history for binary files, and they have tried copying files manually (the `final-v3` problem) but it is chaotic. They ask you whether Git is the right solution.

**Consider:**
- How does Git handle binary files compared to text files?
- What are the storage implications of tracking 200 MB files?
- Are there version control tools designed specifically for binary or design files?

<details>
<summary>Hint</summary>

Git works with any file type, but it is optimized for text. Binary files do not compress well with Git's delta compression. Every version of a 200 MB Illustrator file becomes a full 200 MB object in the repository. Git also cannot show meaningful diffs for binary files — it can only tell you "this file is different."
</details>

<details>
<summary>Solution</summary>

**One strong approach: use Git with Git LFS (Large File Storage)**

Git is not ideal for large binary files out of the box, but Git LFS extends Git to handle them efficiently:

```bash
# Install Git LFS
git lfs install

# In your repository, configure LFS to track Illustrator files
git lfs track "*.ai"
git lfs track "*.eps"
git add .gitattributes
git commit -m "Configure Git LFS for design files"
```

**How Git LFS works:** Instead of storing the binary file in the repository, Git LFS stores a small text pointer in the repository and stores the actual binary file on a remote server. When you clone, Git LFS downloads the current versions of the binary files. Old versions are stored on the server and downloaded on demand.

**Pros of Git + Git LFS:**
- Full version history for design files.
- Meaningful commit messages describing what changed.
- Integration with existing Git workflows (branches, pull requests, code review).
- Offline access to the latest version of files.
- Multiple hosting options (GitHub, GitLab, Bitbucket all support LFS).

**Cons of Git + Git LFS:**
- LFS has storage quotas on most hosting platforms (GitHub gives 1 GB free, then charges for additional storage).
- Git cannot show meaningful diffs for Illustrator files — you cannot see "these three lines changed."
- Designers need to learn Git commands, which is a significant learning curve.
- LFS operations can be slow for very large files.
- Conflicts on binary files cannot be merged automatically — someone must choose one version or redo the work.

**Alternative approaches:**

1. **Specialized design version control:** Tools like **Abstract** (for Sketch), **Kactus** (for Sketch), or **Pixelapse** are designed for design file version control. They provide visual diffs, layer-level comparison, and a UI designed for designers rather than developers. However, many of these tools have been discontinued or acquired.

2. **Perforce Helix Core:** Perforce is a centralized version control system that handles large binary files natively. It is used by game development studios and visual effects companies. It uses a checkout-lock model where a file must be checked out before editing, preventing merge conflicts. Perforce is free for small teams (up to 5 users) but expensive at scale.

3. **Backup-based approach:** If the team only needs the latest version and a few backups, a combination of cloud storage (Google Drive, Dropbox) with naming conventions and a shared spreadsheet tracker may be sufficient. This is not version control, but it may be simpler for a non-technical team.

4. **Hybrid approach:** Use Git for code and configuration files, and use a separate system (like a shared network drive with versioned backups) for design files. This matches each tool to the type of content it handles best.

**Recommendation for this team:**

If the designers are willing to learn basic Git commands, use Git + Git LFS hosted on GitHub or GitLab. Set up a `.gitignore` that excludes all large design files from regular Git tracking and uses LFS for them instead. Establish a workflow where only one person edits a design file at a time (to avoid merge conflicts). Accept that Git cannot show "what changed" in a design file — the commit message must describe the change manually.

If the designers are not willing to learn Git, use Perforce or a cloud storage solution with manual version management. The learning curve of Git for non-developers is a real cost, and the team should weigh that against the benefits of version control.

**Critical note:** There is no perfect answer for this scenario. The right choice depends on the team's technical skill, budget, and willingness to change their workflow. The worst choice is to force Git without LFS and end up with a 10 GB repository that nobody can clone or maintain.
</details>
