You are halfway through fixing a critical bug in `auth.py` when your manager asks you to drop everything and review a hotfix on the `main` branch. You have three files modified, two staged, and one untracked. Committing a half-baked bug fix is not an option, and throwing away your changes is not an option either. You need a way to set your dirty working directory aside, switch branches, do the hotfix, and then come back to exactly where you left off. That is exactly what `git stash` does.

The `git stash` command takes your uncommitted changes (both staged and unstaged), saves them onto a stack, and restores your working directory to match the last commit. Later, you can re-apply those changes exactly as they were.

Think of it like a clean desk policy. You are assembling a complex circuit board with components spread across your desk. Someone needs the workbench for a different project. You sweep everything into a labeled tray, put the tray on a shelf, clear the workbench, and let the other person work. When they are done, you pull the tray down and put everything back exactly where it was.

## Command and Syntax Reference

| Command | Flag | Description | Example |
|---|---|---|---|
| `git stash push` | `-m` | Save dirty working directory with a message | `git stash push -m "wip: auth bug fix"` |
| `git stash push` | `--keep-index` | Stash unstaged changes only, keep staged files | `git stash push --keep-index` |
| `git stash push` | `--include-untracked` | Also stash untracked files | `git stash push --include-untracked -u` |
| `git stash push` | `--all` | Stash every file including ignored ones | `git stash push --all` |
| `git stash pop` | (none) | Apply the top stash and remove it from the stack | `git stash pop` |
| `git stash apply` | `stash@{n}` | Apply a specific stash without removing it | `git stash apply stash@{2}` |
| `git stash drop` | `stash@{n}` | Delete a specific stash from the stack | `git stash drop stash@{1}` |
| `git stash list` | (none) | Show all stashes in the stack | `git stash list` |
| `git stash show` | `-p` | Show the diff of a stash (with `-p` for full patch) | `git stash show -p stash@{0}` |
| `git stash branch` | `<name>` | Create a new branch from the commit where a stash was made and apply the stash | `git stash branch hotfix/auth stash@{0}` |
| `git stash clear` | (none) | Remove all stashes from the stack | `git stash clear` |

**Windows vs macOS/Linux notes:** All commands are identical across platforms. The only difference is path format when using `git stash push -- <pathspec>` — on Windows, use forward slashes (`git stash push -- src/auth.py`) or escaped backslashes. On macOS and Linux, forward slashes are standard.

## How It Actually Works — Step by Step

### Step 1: You have a dirty working directory

When you run `git stash push`, Git does not simply save your files as-is. It creates a special commit-like object called a **stash entry** that is attached to your current `HEAD` commit. This entry records the exact state of your tracked files (both the index and the working tree). Git then resets the index and working tree to match `HEAD`, leaving your branch clean.

### Step 2: Git creates the stash entry

Internally, `git stash push` creates two commits: one for the current index state and one for the working tree state. These commits are stored on a special ref called `refs/stash`, which is a stack. The most recent stash is always `refs/stash`, and older stashes are reachable via `stash@{N}`.

The stash entry stores the exact diff between your working tree and HEAD, not the full files. This keeps the stash small and fast. Git uses the same delta compression it uses for regular commits.

### Step 3: Git resets the working directory

After recording the stash entry, Git runs `git reset --hard HEAD` internally. This resets both the index and the working tree to match the current HEAD commit. Your working directory is now clean — exactly as if you had just checked out the branch.

### Step 4: You switch branches and do your work

With a clean working directory, you can now safely run `git checkout main` or `git switch main`. There is no risk of merge conflicts because there are no dirty files to carry over. You fix the hotfix, commit it, and push it. The stash sits untouched on `refs/stash` the entire time. You can create multiple stashes — each new one pushes the previous ones down the stack.

### Step 5: You restore the stash with pop or apply

When you run `git stash pop`, Git looks at `refs/stash` (the top of the stack), computes the diff between the stash's working tree commit and the commit it was based on, and applies that diff to your current working directory. If you are on the same branch you started from, the diff applies cleanly. If you are on a different branch, Git attempts a three-way merge.

`git stash pop` applies the stash and then runs `git stash drop` to remove it from the stack. `git stash apply` does the same merge but leaves the stash on the stack.

### Step 6: Merge conflicts during stash pop or apply

If the files you stashed have been modified in the branch you popped onto, Git performs a three-way merge between the stash working tree, the original HEAD, and the current HEAD. If there are conflicts, Git leaves the conflict markers in the files and does not drop the stash — even if you used `pop`. This is a safety mechanism: Git keeps the stash so you can inspect the conflicts and retry.

To resolve, fix the conflict markers, stage the files, and then run `git stash drop` manually to clean up. `git stash branch` is a good alternative: it creates a new branch from the commit where the stash was originally made and applies the stash there, avoiding the merge entirely.

## When to Use This

**Use git stash when:**
- You need to switch branches urgently but your working directory is dirty.
- You want to test a theory by temporarily discarding changes and then bringing them back.
- You need to pull in remote changes but have local modifications that would be overwritten.
- You want to move staged changes to a different branch (stash, switch, pop).
- You are experimenting with a change and want to save it temporarily without creating a permanent commit.

**Do not use git stash when:**
- The changes represent a complete logical unit — commit them instead. Stashing is for work-in-progress, not finished work.
- You need the changes to be visible to collaborators — stashes are local only and never pushed.
- You have more than 5-10 stashes accumulating — this signals you should create a feature branch or use `git stash branch` to turn stashes into proper branches.
- The changes are large and complex — a branch with commits is easier to review, rebase, and share.
- You need to stash the same changes repeatedly — create a branch and commit your work properly.
