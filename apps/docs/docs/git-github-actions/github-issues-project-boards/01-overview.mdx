Your team has 15 bugs, 8 feature requests, and 3 technical debt items all floating in Slack messages, email threads, and sticky notes. No one knows what is being worked on, what is blocked, or what was already fixed. GitHub Issues gives every task a permanent home, and Project Boards let you track work through stages from backlog to done.

## The Concept

An issue is a trackable unit of work: a bug report, feature request, question, or task. Each issue has a unique number, title, description, labels, assignees, milestones, and comments. A Project Board is a kanban-style view that organizes issues into columns (e.g., To Do, In Progress, Done) and tracks their status as they move through the workflow.

Think of issues as individual index cards on a wall. Each card has a description of what needs to happen, who is responsible, and what category it falls under. The Project Board is the wall itself, divided into columns. As work progresses, you move cards from one column to the next. Anyone on the team can look at the wall and instantly know the state of every task.

## Command / Syntax Reference

| Command / Action | What it does | Example |
|---|---|---|
| `gh issue create` | Create a new issue. | `gh issue create --title "Bug" --body "..."` |
| `gh issue list` | List issues matching a filter. | `gh issue list --label bug --assignee @me` |
| `gh issue view <number>` | View issue details and comments. | `gh issue view 42` |
| `gh issue close <number>` | Close an issue. | `gh issue close 42` |
| `gh issue reopen <number>` | Reopen a closed issue. | `gh issue reopen 42` |
| `gh issue comment <number> --body "<text>"` | Add a comment to an issue. | `gh issue comment 42 --body "Fixed in PR #43"` |
| `gh issue edit <number> --add-label "<label>"` | Add a label to an issue. | `gh issue edit 42 --add-label "bug"` |
| `gh project list` | List project boards for the repository. | `gh project list` |
| `gh project item-add <project-id> --url <issue-url>` | Add an issue to a project board. | `gh project item-add 1 --url "$(gh issue view 42 --json url --jq .url)"` |

**Issue templates (`.github/ISSUE_TEMPLATE/`):**

| File | Purpose |
|---|---|
| `bug_report.md` | Template for reporting bugs — steps to reproduce, expected vs. actual behavior, environment |
| `feature_request.md` | Template for proposing features — problem statement, proposed solution, alternatives |
| `config.yml` | Configuration for the issue template chooser (blank issues allowed or not) |

**Issue labels (common conventions):**

| Label | Meaning |
|---|---|
| `bug` | Something is not working as expected |
| `enhancement` | New feature or improvement |
| `documentation` | Documentation changes |
| `good first issue` | Good for new contributors |
| `help wanted` | Maintainer is looking for help |
| `priority: high` | Needs immediate attention |
| `status: blocked` | Cannot proceed until something else is resolved |

**Windows note:** All `gh` commands work identically on Windows. Issue templates use Markdown files and can be created in any text editor.

## How It Actually Works — Step by Step

### Step 1: Create an issue

```bash
gh issue create --title "Login page crashes on empty email" \
  --label bug \
  --assignee @me \
  --body "## Steps to Reproduce
1. Go to /login
2. Leave email field empty
3. Click Submit

## Expected
Show validation error 'Email is required'

## Actual
Page crashes with 500 error

## Environment
- Browser: Chrome 120
- OS: Windows 11"
```

GitHub creates issue #42 with:
- A unique number (42)
- The title, body, and metadata
- Status: `open`
- Labels: `bug`
- Assignee: you

### Step 2: Add the issue to a project board

```bash
gh project item-add 1 --url "$(gh issue view 42 --json url --jq .url)"
```

The issue appears in the project board's default column (usually "To Do" or "Backlog"). You can drag it to another column in the web UI.

### Step 3: Work on the issue — link it to a PR

Create a branch linked to the issue:

```bash
git switch -c fix-login-crash
```

In the PR description, reference the issue:

```bash
gh pr create --title "Fix login crash on empty email" \
  --body "Fixes #42

Adds validation for empty email field before database query."
```

The phrase `Fixes #42` (or `Closes #42`, `Resolves #42`) creates a link between the PR and the issue. When the PR is merged, GitHub automatically closes the issue.

### Step 4: Track progress on the project board

As the PR moves through review, you (or automation) move the issue card across columns:
- **To Do** → **In Progress** (when you assign yourself and start working)
- **In Progress** → **Review** (when you open the PR)
- **Review** → **Done** (when the PR is merged)

### Step 5: Automate with GitHub Actions

You can automate board movements. Example workflow (`.github/workflows/auto-move.yml`):

```yaml
name: Auto-move issues
on:
  pull_request:
    types: [opened, closed]
jobs:
  move:
    runs-on: ubuntu-latest
    steps:
      - uses: alex-page/github-project-automation-plus@v0.8.0
        with:
          project: Project Board Name
          column: In Progress
          repo-token: ${{ secrets.GITHUB_TOKEN }}
```

### Step 6: Close and archive

When the PR merges and the issue auto-closes, the card moves to the "Done" column automatically (if configured). After a release, you can archive completed cards to keep the board focused on active work.

## When to Use This / When Not To

**Use Issues when:**
- You need to track a bug, feature, or task with discussion and history
- You want to assign work to specific team members
- You need to organize work by labels, milestones, or projects
- You want to link code changes (PRs) to specific tracked items

**Do not use Issues when:**
- The task is a quick internal note that does not need tracking — use a TODO comment or a wiki page
- The work is already tracked in an external system (Jira, Trello, Asana) and syncing would be redundant
- You are asking a question that will be answered once and never referenced again — use Discussions instead

**Use Project Boards when:**
- You need a visual overview of work in progress across the team
- You want to track the status of multiple issues through a defined workflow
- You need to prioritize work in a shared space

**Do not use Project Boards when:**
- You are the only person on the project — a simple task list in the issue tracker is sufficient
- The project has no defined workflow stages — start with issues only and add a board when you need it
- You need complex dependency tracking or Gantt charts — use a dedicated project management tool
