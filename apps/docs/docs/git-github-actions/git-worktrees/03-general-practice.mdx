## Tier 1 — Beginner

### Exercise 1

**Task:** Create a repo, add a worktree on the same branch as the main worktree. What happens? (Hint: Git prevents this.)

<details>
<summary>Hint</summary>
`git worktree add ../other main` should fail because `main` is already checked out in the main worktree.
</details>

### Solution

```bash
mkdir -p ~/wt-beginner-1 && cd ~/wt-beginner-1
git init repo && cd repo
echo "init" > f.txt && git add f.txt && git commit -m "Init"
git worktree add ../other main 2>&1 || echo "Blocked as expected"
```

Git refuses with `fatal: 'main' is already checked out at '...'`.

---

### Exercise 2

**Task:** Create a repo with 2 branches. Add a worktree for each branch. Verify `git worktree list` shows both worktrees. Remove both worktrees.

<details>
<summary>Hint</summary>
Create `branch-a` and `branch-b`. Use `git worktree add ../path-a branch-a` and `git worktree add ../path-b branch-b`. List with `git worktree list`. Remove with `git worktree remove`.
</details>

### Solution

```bash
mkdir -p ~/wt-beginner-2 && cd ~/wt-beginner-2
git init repo && cd repo
echo "init" > f.txt && git add f.txt && git commit -m "Init"
git branch branch-a
git branch branch-b
git worktree add ../wt-a branch-a
git worktree add ../wt-b branch-b
git worktree list
git worktree remove ../wt-a && git worktree remove ../wt-b
```

### Exercise 3

**Task:** Create a worktree with a new branch using `-b`. Verify the branch exists in the main worktree with `git branch --list`.

<details>
<summary>Hint</summary>
`git worktree add -b new-branch ../new-path`. From the main worktree, `git branch --list` shows `new-branch`.
</details>

### Solution

```bash
mkdir -p ~/wt-beginner-3 && cd ~/wt-beginner-3
git init repo && cd repo && echo "init" > f.txt && git add f.txt && git commit -m "Init"
git worktree add -b my-feature ../feature
cd ../repo && git branch --list
```

`my-feature` appears in the branch list.

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a repo, make a commit on `main`, create a worktree for `main`, then try to create another worktree also on `main`. Git should refuse. Then move the branch in the second worktree to a different commit and try again.

<details>
<summary>Hint</summary>
`git worktree add ../wt2 main` fails because `main` is already in `wt1`. To work around, create a new branch at the same commit for the second worktree: `git worktree add -b main-alt ../wt2 main`.
</details>

### Solution

```bash
mkdir -p ~/wt-inter-1 && cd ~/wt-inter-1
git init repo && cd repo
echo "A" > f.txt && git add f.txt && git commit -m "A"
echo "B" >> f.txt && git add f.txt && git commit -m "B"
git worktree add ../wt1 main
git worktree add ../wt2 main 2>&1 || echo "Blocked — main already checked out"
# Workaround: create a new branch at the same commit
git worktree add -b temp-main ../wt2 main
git worktree list
```

### Exercise 2

**Task:** Create a repo with 3 commits. Create 3 worktrees, each on a different commit (not necessarily a branch). Use detached HEAD worktrees.

<details>
<summary>Hint</summary>
Use `git worktree add ../wt1 HEAD~2` to check out a specific commit in detached HEAD. Same for `HEAD~1` and `HEAD`.
</details>

### Solution

```bash
mkdir -p ~/wt-inter-2 && cd ~/wt-inter-2
git init repo && cd repo
for i in 1 2 3; do echo "$i" > f.txt && git add f.txt && git commit -m "C$i"; done
git worktree add ../wt1 HEAD~2
git worktree add ../wt2 HEAD~1
git worktree add ../wt3 HEAD
git worktree list
```

Each worktree has a different commit checked out in detached HEAD.

### Exercise 3

**Task:** Create a worktree, lock it, verify it is locked with `git worktree list`, then unlock it. Try to remove a locked worktree without unlocking — it should fail.

<details>
<summary>Hint</summary>
`git worktree lock <path>` locks it. `git worktree list` shows `locked` next to the path. `git worktree remove <path>` fails on locked worktrees. `git worktree unlock <path>` unlocks it.
</details>

### Solution

```bash
mkdir -p ~/wt-inter-3 && cd ~/wt-inter-3
git init repo && cd repo && echo "init" > f.txt && git add f.txt && git commit -m "Init"
git worktree add ../locked-wt main
git worktree lock ../locked-wt
git worktree list
git worktree remove ../locked-wt 2>&1 || echo "Blocked — worktree is locked"
git worktree unlock ../locked-wt
git worktree remove ../locked-wt
```

## Tier 3 — Advanced

### Exercise 1: Parallel feature development with worktrees

**Scenario:** You need to work on three features simultaneously: `feature/search`, `feature/auth`, and `feature/dashboard`. Each has a long build time. You want to have all three checked out in separate directories so you can work on them without context-switching overhead.

**Task:** Create a repo with a base script. Create 3 branches from `main`. Create a worktree for each branch. Write a script that builds and tests all three worktrees in parallel using `git worktree foreach`-like logic.

<details>
<summary>Hint</summary>
Create the branches, then `git worktree add -b feature/search ../search`, etc. Use `for d in ../search ../auth ../dashboard; do (cd "$d" && make test) & done; wait` to run in parallel.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/wt-adv-1 && cd ~/wt-adv-1
git init repo && cd repo
echo "base" > app.py && git add app.py && git commit -m "Base"
# Create branches
for branch in feature/search feature/auth feature/dashboard; do
  git branch $branch
done
# Create worktrees
for branch in feature/search feature/auth feature/dashboard; do
  name=${branch#feature/}
  git worktree add -b $branch ../$name 2>/dev/null || git worktree add ../$name $branch
done
# Simulate parallel build
for d in ../search ../auth ../dashboard; do
  (cd "$d" && echo "Building in $(pwd)..." && python -c "print('OK')") &
done
wait
echo "All builds complete"
```

**Alternative approach:** Use a CI script that triggers builds in each worktree. The worktree approach avoids the need to clone the repo three times.

### Exercise 2: PR review workflow with worktrees

**Scenario:** You have 3 open pull requests to review. Each PR is on a different branch. You want to check out each PR in its own worktree so you can review them side by side, run tests, and leave comments — all without interrupting your current work.

**Task:** Create a repo with 3 PR branches (simulated by 3 local branches). Use a script to create a worktree for each PR branch, run the tests, and report the results. Remove the worktrees after review.

<details>
<summary>Hint</summary>
Use `gh pr checkout <number>` to check out a PR branch (or `git fetch origin pull/<id>/head:pr/<id>`). Create worktrees with `git worktree add ../pr-<id>`. Run tests in each. Remove with `git worktree remove`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/wt-adv-2 && cd ~/wt-adv-2
git init repo && cd repo && echo "base" > app.py && git add app.py && git commit -m "Base"
# Simulate 3 PR branches
for i in 1 2 3; do
  git branch pr/$i
  git switch pr/$i
  echo "Feature $i" >> app.py
  git add app.py && git commit -m "PR $i feature"
  git switch main
done
# Review script
cat > review-prs.sh << 'EOF'
#!/bin/bash
PRS=$(git branch --list 'pr/*' | sed 's/.*pr\///')
for pr in $PRS; do
  echo "=== Reviewing PR #$pr ==="
  wt_path="../pr-review-$pr"
  git worktree add $wt_path pr/$pr 2>/dev/null
  (cd $wt_path && echo "Files:" && ls && echo "Tests: python -c 'print(\"OK\")'")
done
echo "=== Review complete ==="
git worktree list | grep pr-review
EOF
chmod +x review-prs.sh
./review-prs.sh
# Cleanup after review
for pr in $(git branch --list 'pr/*' | sed 's/.*pr\///'); do
  git worktree remove ../pr-review-$pr 2>/dev/null || true
done
```

**Alternative approach:** Use `gh pr checkout` to fetch PR branches from a real GitHub repo. The worktree approach is the same — create worktrees for each PR branch, review, remove.

### Exercise 3: Worktree-based release management

**Scenario:** Your release process requires:
1. Create a release branch from `main`
2. Bump the version number
3. Run the full test suite
4. Build the distribution
5. Tag the release
6. Merge the release branch back to `main`

You want to do all of this without touching your main working directory. You also want to keep the release branch around for hotfixes.

**Task:** Automate the release process using a worktree. Write a script that:
1. Creates a worktree on a new `release/vX.Y` branch
2. Bumps the version in the worktree
3. Runs tests
4. Tags the release
5. Reports success
6. Keeps the worktree for hotfixes

<details>
<summary>Hint</summary>
Use `git worktree add -b release/v1.0 ../release-v1.0 main`. In the worktree, modify the version file, commit, tag. The tag is shared with the main repo because worktrees share refs.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/wt-adv-3 && cd ~/wt-adv-3
git init repo && cd repo
echo "VERSION = '0.1.0'" > version.py
echo "print('app running')" > app.py
git add . && git commit -m "Initial project"
```

The release script:

```bash
cat > create-release.sh << 'EOF'
#!/bin/bash
VERSION=${1:?Usage: $0 <version>}
BRANCH="release/v$VERSION"
TAG="v$VERSION"
WORKTREE="../release-$VERSION"
echo "=== Creating release $VERSION ==="
# Create worktree
git worktree add -b "$BRANCH" "$WORKTREE" main
if [ $? -ne 0 ]; then
  echo "ERROR: Could not create worktree"
  exit 1
fi
# Bump version in worktree
cd "$WORKTREE"
echo "VERSION = '$VERSION'" > version.py
git add version.py
git commit -m "Bump version to $VERSION"
# Run tests (simulated)
echo "Running tests..."
python -c "print('All tests passed')" || {
  echo "Tests failed — aborting"
  cd - > /dev/null
  git worktree remove "$WORKTREE"
  exit 1
}
# Tag the release
git tag -a "$TAG" -m "Release $VERSION"
echo "=== Release $VERSION complete ==="
echo "Tag: $TAG"
echo "Branch: $BRANCH"
echo "Worktree: $WORKTREE (keep for hotfixes)"
cd - > /dev/null
EOF
chmod +x create-release.sh
./create-release.sh 1.0.0
```

Verify:

```bash
git branch --list 'release/*'
git tag --list 'v*'
git worktree list
```

**Alternative approach:** Use a CI/CD pipeline (GitHub Actions) for releases. The worktree approach is useful for local testing and when you want to keep the release branch immediately available for hotfixes without switching away from your current work.
