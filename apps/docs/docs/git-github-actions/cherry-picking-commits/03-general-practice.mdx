Work through these exercises in a terminal. Tier 1 builds single-command muscle memory. Tier 2 combines cherry-picking with prior Git skills. Tier 3 presents real-world backporting and recovery scenarios.

## Tier 1 — Beginner

### Exercise 1

**Task:** Create a repo with `main` (2 commits) and `feature` (1 commit on top). Cherry-pick the single feature commit onto `main`. Verify it appears as a new commit on `main` with a different hash.

<details>
<summary>Hint</summary>
Use `git cherry-pick <hash>` from `main`. Compare the original hash on `feature` with the new hash on `main` using `git log`.
</details>

### Solution

```bash
mkdir -p ~/cherry-beginner-1 && cd ~/cherry-beginner-1
git init
echo "C1" > f.txt && git add f.txt && git commit -m "C1"
echo "C2" >> f.txt && git add f.txt && git commit -m "C2"
git switch -c feature
echo "F1" >> f.txt && git add f.txt && git commit -m "F1"
ORIG_HASH=$(git rev-parse HEAD)
git switch main
git cherry-pick $ORIG_HASH
NEW_HASH=$(git rev-parse HEAD)
echo "Original: $ORIG_HASH"
echo "Cherry-pick: $NEW_HASH"
test "$ORIG_HASH" != "$NEW_HASH" && echo "Different hashes - confirmed copy" || echo "Same hash!"
```

### Exercise 3

**Task:** Cherry-pick a commit that causes a conflict, then abort the cherry-pick. Verify you are back to the pre-pick state.

<details>
<summary>Hint</summary>
Modify the same line on both branches. Cherry-pick from the divergent branch. When the conflict appears, run `git cherry-pick --abort`.
</details>

### Solution

```bash
mkdir -p ~/cherry-beginner-3 && cd ~/cherry-beginner-3
git init && echo "same line" > f.txt && git add f.txt && git commit -m "Base"
echo "main version" > f.txt && git add f.txt && git commit -m "Main change"
git switch -c feature HEAD~1
echo "feature version" > f.txt && git add f.txt && git commit -m "Feature change"
git switch main
git cherry-pick <hash-of-feature-commit>
# Conflict
git cherry-pick --abort
git status
```

`git status` shows "nothing to commit, working tree clean."

### Exercise 4

**Task:** Use `git cherry-pick -x` to pick a commit. Verify the commit message includes the original hash reference.

<details>
<summary>Hint</summary>
After cherry-picking with `-x`, use `git log -1 --format="%B"` to see the full commit message including the cherry-pick annotation.
</details>

### Solution

```bash
mkdir -p ~/cherry-beginner-4 && cd ~/cherry-beginner-4
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c source
echo "target change" >> f.txt && git add f.txt && git commit -m "Important fix"
HASH=$(git rev-parse HEAD)
git switch main
git cherry-pick -x $HASH
git log -1 --format="%B"
```

The output shows "Important fix" followed by a blank line and `(cherry picked from commit <hash>)`.

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a repo with 3 branches: `main`, `feature-a` (2 commits), `feature-b` (1 commit based on `feature-a`). Cherry-pick only `feature-b`'s commit onto `main` — without getting `feature-a`'s commits.

<details>
<summary>Hint</summary>
Find `feature-b`'s commit hash with `git log feature-b`. The commit on `feature-b` that is not on `feature-a` is the tip commit.
</details>

### Solution

```bash
mkdir -p ~/cherry-inter-1 && cd ~/cherry-inter-1
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feature-a
echo "A1" >> f.txt && git add f.txt && git commit -m "Feature A - 1"
echo "A2" >> f.txt && git add f.txt && git commit -m "Feature A - 2"
git switch -c feature-b
echo "B1" >> f.txt && git add f.txt && git commit -m "Feature B - 1"
HASH=$(git rev-parse HEAD)
git switch main
git cherry-pick $HASH
git log --oneline
```

Only "Feature B - 1" appears on `main`. No feature A commits.

### Exercise 2

**Task:** Cherry-pick a range of 3 commits from `feature` onto `main` using the range syntax. Verify all 3 appear as separate commits on `main` and are not squashed.

<details>
<summary>Hint</summary>
Use `git cherry-pick <start>..<end>` where `<start>` is the commit BEFORE the first one you want and `<end>` is the last one. Or use `git cherry-pick main..feature` from `main`.
</details>

### Solution

```bash
mkdir -p ~/cherry-inter-2 && cd ~/cherry-inter-2
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feature
echo "F1" >> f.txt && git add f.txt && git commit -m "Feat 1"
echo "F2" >> f.txt && git add f.txt && git commit -m "Feat 2"
echo "F3" >> f.txt && git add f.txt && git commit -m "Feat 3"
git switch main
git cherry-pick main..feature
git log --oneline
```

3 new commits on `main` — "Feat 1", "Feat 2", "Feat 3". All separate, not squashed.

### Exercise 3

**Task:** Cherry-pick a commit from a branch, then use `git show` on both the original and the cherry-pick to compare their diffs. Confirm the diffs are identical but the hash, date, and parent differ.

<details>
<summary>Hint</summary>
Use `git show --format="%H %P %ai" <hash>` to display hash, parent hash, and author date. Compare the outputs.
</details>

### Solution

```bash
mkdir -p ~/cherry-inter-3 && cd ~/cherry-inter-3
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c source
echo "fix" >> f.txt && git add f.txt && git commit -m "Bug fix"
ORIG=$(git rev-parse HEAD)
git switch main
git cherry-pick $ORIG
NEW=$(git rev-parse HEAD)
echo "=== Original ===" && git show --format="%H %P %ai" -s $ORIG
echo "=== Cherry-pick ===" && git show --format="%H %P %ai" -s $NEW
echo "=== Diff comparison ==="
git diff $ORIG^..$ORIG > /tmp/orig.diff
git diff $NEW^..$NEW > /tmp/new.diff
diff /tmp/orig.diff /tmp/new.diff && echo "Diffs are identical" || echo "Diffs differ"
```

The hashes, parent hashes, and dates differ. The diffs are identical.

## Tier 3 — Advanced

### Exercise 1: Backport a fix across multiple release branches

**Scenario:** A critical security fix was committed on `main` as commit `abc1234`. You need to backport it to three active release branches: `release/v1`, `release/v2`, `release/v3`. Each branch has diverged from `main` significantly.

**Task:** Cherry-pick the same fix to all three release branches. Then verify the fix exists on each branch by checking that the commit message (with `-x` annotation) is present. Handle any conflicts that arise — each branch may need a different resolution.

<details>
<summary>Hint</summary>
Checkout each release branch in turn and `git cherry-pick -x <hash>`. Resolve conflicts if they occur. Use `git log --all --grep="cherry picked from"` to verify all backports.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/cherry-adv-1 && cd ~/cherry-adv-1
git init
# Create common base
echo "base" > app.py && git add app.py && git commit -m "Base"
# Create release branches
for v in 1 2 3; do
  git branch release/v$v
  git switch release/v$v
  echo "release v$v change" > app.py && git add app.py && git commit -m "v$v changes"
done
# Create the fix on main
git switch main
echo "SECURITY FIX" >> app.py && git add app.py && git commit -m "Fix auth bypass"
FIX_HASH=$(git rev-parse HEAD)
# Backport to each release branch
for v in 1 2 3; do
  git switch release/v$v
  git cherry-pick -x $FIX_HASH 2>/dev/null || {
    echo "Conflict on release/v$v - resolving"
    echo "SECURITY FIX" >> app.py
    git add app.py
    git cherry-pick --continue --no-edit 2>/dev/null || true
  }
done
# Verify
git log --all --grep="cherry picked from" --oneline
```

All three release branches have the fix. Each cherry-pick has a different hash but the same original source reference.

**Alternative approach:** For many branches, write a shell loop. For branches with very different codebases, you may need to adapt the fix per branch rather than cherry-picking blindly.

### Exercise 2: Recover a single commit from a deleted branch

**Scenario:** Someone deleted a branch that had a critical fix, but the commit is still in Git's object store (not yet garbage-collected). You need to find and cherry-pick it without the branch reference.

**Task:** Simulate this: create a commit on a branch, delete the branch, then use `git reflog` or `git fsck` to find the commit hash and cherry-pick it to `main`.

<details>
<summary>Hint</summary>
After deleting the branch, use `git reflog` to find the commit. The reflog records every HEAD movement, including checkouts and commits. Find the commit message and hash, then cherry-pick it.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/cherry-adv-2 && cd ~/cherry-adv-2
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch -c doomed
echo "critical fix" >> f.txt && git add f.txt && git commit -m "Critical security fix"
FIX_HASH=$(git rev-parse HEAD)
git switch main
git branch -D doomed
# Branch is gone, but we have the hash
git cherry-pick $FIX_HASH
git log --oneline -2
```

If you did not save the hash before deleting:

```bash
git reflog
# Look for: "commit: Critical security fix"
# Copy the hash from the reflog entry
git cherry-pick <hash-from-reflog>
```

**Alternative approach:** `git fsck --lost-found` finds dangling commits. This works even if the reflog has expired (default 90 days).

### Exercise 3: Cherry-pick orchestration — selective backport with dependencies

**Scenario:** A feature branch has 10 commits. Commits 1-3 set up a new database schema. Commits 4-5 add a data access layer that depends on the schema. Commits 6-10 add a feature that uses the data layer. Your `stable` branch needs only the database schema and data layer (commits 1-5) — not the feature that uses them. You must cherry-pick only commits 1-5, in order, onto `stable`.

**Task:** Create this scenario. Cherry-pick commits 1-5 onto `stable` in the correct order. Verify that the code on `stable` has the schema and data layer but NOT the feature code from commits 6-10.

<details>
<summary>Hint</summary>
Use `git cherry-pick <commit1> <commit2> <commit3> <commit4> <commit5>` in one command. Or, now that cherry-pick accepts multiple arguments, list the hashes explicitly. To find the hashes, use `git log feature --oneline`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/cherry-adv-3 && cd ~/cherry-adv-3
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch main
# Create feature branch with 10 commits
git switch -c feature
echo "schema v1" > schema.sql && git add schema.sql && git commit -m "C1: Create schema"
echo "schema v2" > schema.sql && git add schema.sql && git commit -m "C2: Update schema"
echo "index" >> schema.sql && git add schema.sql && git commit -m "C3: Add index"
echo "data layer" > dal.py && git add dal.py && git commit -m "C4: Add data access layer"
echo "dal utils" >> dal.py && git add dal.py && git commit -m "C5: Add DAL utilities"
echo "feature UI" > ui.py && git add ui.py && git commit -m "C6: Add feature UI"
echo "feature logic" > logic.py && git add logic.py && git commit -m "C7: Add feature logic"
echo "feature tests" > tests.py && git add tests.py && git commit -m "C8: Add tests"
echo "feature config" > config.py && git add config.py && git commit -m "C9: Add config"
echo "feature docs" > docs.md && git add docs.md && git commit -m "C10: Add docs"
```

Cherry-pick only commits 1-5:

```bash
HASHES=$(git log --oneline feature | tail -5 | awk '{print $1}' | tac)
# This gives C1, C2, C3, C4, C5 in order
for h in $HASHES; do
  git cherry-pick $h
done
# Verify
ls *.py *.sql
```

`stable` has `schema.sql` and `dal.py` but NOT `ui.py`, `logic.py`, etc.

**Alternative approach:** Use `git cherry-pick <C1>^..<C5>` where `C1` is the first commit hash. The `^` suffix includes `C1` in the range (range is normally exclusive of the start).
