You merged a pull request on Friday afternoon. All checks passed green. By Monday morning, your team's on-call dashboards showed three production alerts, and the deploy had to be rolled back. The root cause was a null pointer exception that should have been caught by a unit test. That test passed during CI but failed on the next push and then passed again two pushes later. Nobody flagged it because the CI result was green, and nobody had noticed that the test had been passing and failing unpredictably for two weeks. The test was flaky, and that flakiness had eroded the team's trust in the entire test suite enough that the real failure got buried in the noise.

**A flaky test is a test that passes and fails on the same code without any code changes.** It is not a test that reveals a real bug. It is a test that reveals instability in the test environment, the test itself, or the interaction between the test and the system under test. Think of a flaky test like a smoke detector that goes off whenever you cook toast. After the third false alarm, you stop paying attention to it entirely. When the house actually catches fire, the alarm goes off and nobody moves. Flaky tests are the same: they train your team to ignore CI failures.

## Command and Syntax Reference

The following table covers the key tools, YAML keys, and pytest options for detecting, managing, and mitigating flaky tests in CI.

| Tool / Key / Option | Purpose | Example |
|---|---|---|
| `pytest --reruns N` (pytest-rerunfailures) | Automatically reruns a failed test up to N times before marking it as failed | `pytest --reruns 3` |
| `pytest --reruns-delay N` | Adds a delay in seconds between reruns to allow transient conditions to resolve | `pytest --reruns 3 --reruns-delay 5` |
| `pytest --flaky-attempts N` (pytest-flaky) | Runs the entire test suite up to N times, reporting success if any run passes | `pytest --flaky-attempts 3` |
| `jobs.<id>.strategy.fail-fast: false` | Prevents one failed matrix job from cancelling other jobs, allowing flaky-detection analysis | `fail-fast: false` |
| `jobs.<id>.continue-on-error: true` | Marks a job as allowed to fail without blocking the pipeline result | `continue-on-error: true` |
| `steps.<id>.continue-on-error: true` | Marks a single step as allowed to fail, useful for non-critical flaky checks | `continue-on-error: true` |
| `pytest --last-failed` | Re-runs only the tests that failed in the previous run, used in retry strategies | `pytest --last-failed --reruns 2` |
| `pytest -x` (exitfirst) | Stops the test run after the first failure, useful for fast feedback in retry loops | `pytest -x --reruns 2` |
| `pytest --cache-clear` | Clears the pytest cache before running, prevents stale failure data from affecting results | `pytest --cache-clear` |
| Quarantine directory | A directory of tests that are excluded from the main test suite and run separately for monitoring | `tests/quarantine/` |
| Flake rate tracking | Recording the pass/fail ratio of each test over time to identify flaky candidates | `psql -c "INSERT INTO flake_log VALUES(...)"` |

## How It Actually Works — Step by Step

Here is how flaky tests emerge, how they affect your pipeline, and how you manage them over time.

### Step 1: A Test Is Born (Passing)

A developer writes a test that passes locally. The test involves a network call to a third-party API. The developer's local network is fast and reliable, so the test always passes.

```python
def test_user_fetch():
    response = requests.get("https://api.example.com/users/1")
    assert response.status_code == 200
    assert response.json()["name"] == "Alice"
```

This test passes every time on the developer's machine. In CI, it passes 90% of the time. In the remaining 10%, the API returns a 503 or a timeout. The test is already flaky, but nobody has noticed yet.

### Step 2: The First Retry Click

The first time a flaky test fails in CI, someone clicks the "Re-run failed jobs" button. The test passes on retry. Nobody investigates. The mental model shifts from "the test failed" to "the test is flaky."

### Step 3: Detection Through Tracking

You cannot manage what you do not measure. The first step to managing flaky tests is tracking their failure rate. Each test run should record which tests passed and which failed, and whether the code changed between runs.

```bash
pytest --json-report --json-report-file=test-report.json
```

Then parse the report to identify tests with a high failure rate over the last 50 runs:

```python
# analyze_flakes.py
import json
from collections import defaultdict

history = defaultdict(list)
for run in get_last_50_runs():
    for test in run["tests"]:
        history[test["nodeid"]].append(test["outcome"])

threshold = 0.05  # 5% failure rate
for test_id, outcomes in history.items():
    failures = outcomes.count("failed")
    rate = failures / len(outcomes)
    if rate >= threshold:
        print(f"FLAKY: {test_id} ({rate:.1%} failure rate)")
```

### Step 5: Retry Strategies in GitHub Actions

When a test is known to be flaky but you are still investigating, use retry strategies to keep the pipeline moving. The key is to distinguish between retry for detection and retry as a workaround.

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - name: Run tests with flaky retry
        run: |
          pytest --reruns 3 --reruns-delay 2 \
            --junitxml=report.xml \
            --json-report --json-report-file=flake-report.json
      - name: Upload flake report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: flake-report
          path: flake-report.json
```

The `--reruns 3` flag retries each failed test up to 3 times with a 2-second delay between attempts. The `if: always()` on the upload step ensures the report is captured even when the test step fails.

### Step 6: Fix or Delete

Every flaky test has a root cause. The most common causes are:

- **Async timing:** The test does not wait long enough for an async operation to complete.
- **Test order dependency:** The test assumes a certain state left by a previous test.
- **Network flakiness:** The test depends on an external service that is not always available.
- **Resource leaks:** The test creates files, database connections, or threads that it does not clean up.
- **Shared mutable state:** The test modifies global state that affects other tests.

The fix for each is specific, but the decision framework is consistent:

| Cause | Fix | Time to Fix | Fix or Quarantine? |
|---|---|---|---|
| Async timing | Increase wait time or use a retry assertion | 10 minutes | Fix |
| Test order dependency | Add `autouse` fixtures or `reset_state` hooks | 30 minutes | Fix |
| Network flakiness | Mock the network call | 20 minutes | Fix |
| Resource leaks | Add cleanup in `teardown` or `finally` blocks | 15 minutes | Fix |
| Shared mutable state | Isolate state per test | 45 minutes | Fix |
| Non-deterministic algorithm | Use a fixed seed | 5 minutes | Fix |
| Environmental race condition | Sleep or retry | Variable | Quarantine |

If a test has been in quarantine for 4 weeks without being fixed, delete it. The cost of maintaining a flaky test forever exceeds the cost of writing a new one.

## When to Use These Strategies

Use flaky test management strategies when:

- **Your team routinely clicks "re-run failed jobs" without investigating.** This is the clearest signal that flaky tests are eroding trust.
- **You cannot remember the last time you saw a CI failure that was a real bug.** If every red X is a flaky test, your CI is no longer a quality gate.
- **The test suite has more than 200 tests.** Below 200 tests, you can manage flakiness manually. Above that, you need automated detection.
- **You deploy to production more than once per week.** High deployment frequency requires a reliable test suite. One flaky test per week breaks the trust.
- **Your team has more than five developers.** With more developers, the noise from flaky tests multiplies because each developer triggers the pipeline more often.

## When Not to Use These Strategies

Avoid heavy flaky test management when:

- **Your test suite has fewer than 50 tests.** The overhead of tracking, quarantining, and reviewing is not worth it. Just fix the flaky test immediately.
- **You are in early prototyping.** If the product direction changes weekly, the tests will be rewritten anyway. Accept flakiness and focus on speed.
- **The flaky test is a known, accepted false positive that you are about to delete.** Do not quarantine a test you plan to delete tomorrow. Just delete it now.
- **Your CI runners are unreliable.** If the runner itself crashes or the network infrastructure is unstable, the problem is not the test. Fix the infrastructure first.
