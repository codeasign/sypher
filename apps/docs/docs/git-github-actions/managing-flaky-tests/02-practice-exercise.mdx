## Setup

For this walkthrough, you need Python 3.9 or later, pip, and a GitHub repository with GitHub Actions enabled. You will create a test suite with deliberately flaky tests, then build a pipeline that detects, retries, and quarantines them.

```bash
# Create a project directory
mkdir flaky-test-demo
cd flaky-test-demo
git init
```

```bash
# Install pytest and the flaky test plugins
pip install pytest pytest-rerunfailures pytest-flaky pytest-json-report
```

```bash
# Create the project structure
mkdir -p tests/quarantine
```

Create a minimal pytest configuration file `pytest.ini`:

```ini
[pytest]
testpaths = tests
markers =
    flaky: marks a test as known to be flaky
    slow: marks a test as slow
```

Initialize the git repository and push to GitHub:

```bash
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/YOUR-USERNAME/flaky-test-demo.git
git push -u origin main
```

## The Walkthrough

You will build a flaky test management system from scratch. Each step introduces a new tool or pattern for detecting and managing flaky tests.

### Step 1: Create a Deliberately Flaky Test

Create a test that passes 80% of the time and fails 20% of the time. This simulates a real flaky test caused by an unreliable network call or race condition.

Create `tests/test_flaky_login.py`:

```python
import random
import time


def test_login_success():
    """
    Simulates a login API call that sometimes fails due to network latency.
    This is a deliberately flaky test for demonstration purposes.
    """
    time.sleep(0.1)
    if random.random() < 0.2:
        raise AssertionError("Login failed: timeout waiting for response")
    assert True
```

Run the test multiple times locally to see the flakiness:

```bash
for i in $(seq 1 10); do
  pytest tests/test_flaky_login.py -q 2>&1 | tail -1
done
```

Expected output (varies due to randomness):

```
1 passed in 0.15s
1 passed in 0.12s
1 failed in 0.13s
1 passed in 0.14s
1 passed in 0.11s
1 failed in 0.13s
1 passed in 0.12s
1 passed in 0.15s
1 passed in 0.11s
1 passed in 0.14s
```

The test passes most of the time but fails occasionally. This is the exact behavior of a real flaky test.

### Step 2: Add Retry with pytest-rerunfailures

Install the rerunfailures plugin and use it to retry the flaky test automatically.

```bash
pip install pytest-rerunfailures
```

Run the test with the `--reruns` flag:

```bash
pytest tests/test_flaky_login.py --reruns 3 --reruns-delay 1 -v
```

Expected output:

```
tests/test_flaky_login.py::test_login_success RERUN
tests/test_flaky_login.py::test_login_success RERUN
tests/test_flaky_login.py::test_login_success PASSED
```

The `RERUN` lines show that the test failed on the first attempt and succeeded on the second. The `--reruns 3` flag tells pytest to retry up to 3 times. The `--reruns-delay 1` adds a 1-second delay between retries.

Checkpoint: You now have automatic retry for flaky tests. The test will pass as long as it succeeds within 3 attempts.

### Step 3: Detect Flaky Tests with pytest-flaky

The `pytest-flaky` plugin runs the entire test suite multiple times and reports the results. This is a detection tool, not a retry tool.

```bash
pip install pytest-flaky
```

Create a test that is more consistently flaky `tests/test_flaky_api.py`:

```python
import requests


class TestUserAPI:
    def test_get_user(self):
        """
        Tests fetching a user from an external API.
        The external API is unreliable, causing this test to be flaky.
        """
        try:
            response = requests.get(
                "https://jsonplaceholder.typicode.com/users/1",
                timeout=2
            )
            assert response.status_code == 200
            assert response.json()["name"] == "Leanne Graham"
        except requests.RequestException:
            raise AssertionError("API request failed: network error")

    def test_get_posts(self):
        """Related test that also depends on the external API."""
        try:
            response = requests.get(
                "https://jsonplaceholder.typicode.com/posts",
                timeout=2
            )
            assert response.status_code == 200
            assert len(response.json()) > 0
        except requests.RequestException:
            raise AssertionError("API request failed: network error")
```

Run the tests with the `--flaky-attempts` flag:

```bash
pytest tests/test_flaky_api.py --flaky-attempts 3 -v
```

Expected output:

```
tests/test_flaky_api.py::TestUserAPI::test_get_user PASSED
tests/test_flaky_api.py::TestUserAPI::test_get_posts PASSED
```

The `--flaky-attempts 3` flag runs the entire test suite up to 3 times. If the test passes on any attempt, it is reported as passed. This is useful for detection but dangerous for gating, because it can mask real failures.

### Step 4: Build a Flaky Test Detection Pipeline in GitHub Actions

Create a workflow that runs the test suite, captures flaky test data, and uploads the report as an artifact.

Create `.github/workflows/flake-detection.yml`:

```yaml
name: Flake Detection
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  detect-flakes:
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-rerunfailures pytest-json-report
      - name: Run tests with flaky detection
        run: |
          pytest tests/ \
            --reruns 2 \
            --reruns-delay 1 \
            --json-report \
            --json-report-file=flake-report.json \
            -v
      - name: Upload flake report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: flake-report
          path: flake-report.json
```

The `continue-on-error: true` at the job level means the pipeline does not fail even if the tests fail. The report is captured and uploaded regardless.

Commit and push:

```bash
git add .
git commit -m "Add flaky test detection pipeline with report upload"
git push origin main
```

### Step 5: Implement the Quarantine Pattern

Now create a gating pipeline that separates the main test suite from the quarantine suite. The main suite blocks the pipeline. The quarantine suite runs alongside it but does not block.

Create `.github/workflows/quality-gates.yml`:

```yaml
name: Quality Gates
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  main-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest
      - name: Run main test suite (gating)
        run: |
          pytest tests/ \
            --ignore=tests/quarantine \
            --junitxml=results.xml
      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: results.xml

  quarantine-tests:
    needs: main-tests
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-json-report
      - name: Run quarantine suite (non-gating)
        run: |
          pytest tests/quarantine/ \
            --reruns 3 \
            --json-report \
            --json-report-file=quarantine-report.json
      - name: Upload quarantine report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: quarantine-report
          path: quarantine-report.json
```

The `--ignore=tests/quarantine` flag in the main suite excludes the quarantine directory. The quarantine job runs with `continue-on-error: true`, so it never blocks the pipeline.

Move the flaky test into quarantine:

```bash
mv tests/test_flaky_login.py tests/quarantine/
```

Commit and push:

```bash
git add .
git commit -m "Implement quarantine pattern for flaky tests"
git push origin main
```

### Step 6: Add Automated Flake Rate Tracking

Create a script that tracks flake rate over time by parsing the JSON reports from each pipeline run.

Create `scripts/track_flakes.py`:

```python
#!/usr/bin/env python3
"""
Tracks flaky test rates by parsing JSON reports from CI runs.
Run this script after each pipeline run to update the flake history.
"""
import json
import os
from datetime import datetime
from collections import defaultdict

HISTORY_FILE = "flake-history.json"
FLAKE_THRESHOLD = 0.05  # 5% failure rate triggers flagging


def load_history():
    if os.path.exists(HISTORY_FILE):
        with open(HISTORY_FILE) as f:
            return json.load(f)
    return []


def save_history(history):
    with open(HISTORY_FILE, "w") as f:
        json.dump(history, f, indent=2)


def parse_report(report_path):
    with open(report_path) as f:
        report = json.load(f)

    run_id = report.get("created", datetime.now().isoformat())
    results = defaultdict(list)

    for test in report.get("tests", []):
        nodeid = test.get("nodeid", "unknown")
        outcome = test.get("outcome", "unknown")
        retries = test.get("callspec", {}).get("reruns", 0)
        results[nodeid].append({
            "outcome": outcome,
            "retries": retries,
            "run_id": run_id,
        })

    return results


def identify_flakes(history):
    """Identify tests with a flake rate above the threshold."""
    test_outcomes = defaultdict(list)

    for run in history:
        for test_id, outcomes in run.items():
            for outcome in outcomes:
                test_outcomes[test_id].append(outcome["outcome"])

    flaky_tests = []
    for test_id, outcomes in test_outcomes.items():
        failures = outcomes.count("failed")
        total = len(outcomes)
        if total > 0:
            rate = failures / total
            if rate >= FLAKE_THRESHOLD:
                flaky_tests.append({
                    "test_id": test_id,
                    "failure_rate": round(rate, 3),
                    "total_runs": total,
                    "total_failures": failures,
                })

    return sorted(flaky_tests, key=lambda x: x["failure_rate"], reverse=True)


if __name__ == "__main__":
    import sys

    if len(sys.argv) < 2:
        print("Usage: track_flakes.py <report.json>")
        sys.exit(1)

    report_path = sys.argv[1]
    history = load_history()
    new_results = parse_report(report_path)
    history.append(new_results)
    save_history(history)

    flaky_tests = identify_flakes(history)
    print(f"Flake history updated. Total runs tracked: {len(history)}")
    print(f"Tests with flake rate above {FLAKE_THRESHOLD:.0%}:")
    for test in flaky_tests:
        print(f"  {test['test_id']}: {test['failure_rate']:.1%} "
              f"({test['total_failures']}/{test['total_runs']})")
```

Run the tracking script against the report from your last pipeline run:

```bash
python scripts/track_flakes.py flake-report.json
```

Expected output:

```
Flake history updated. Total runs tracked: 1
Tests with flake rate above 5%:
  tests/test_flaky_api.py::TestUserAPI::test_get_user: 20.0% (1/5)
```

## Common Errors at This Stage

### Error 1: pytest-rerunfailures not installed

```
UsageError: --reruns is not available. Please install pytest-rerunfailures
```

The rerunfailures plugin is installed separately from pytest. It is not included in the base pytest package.

Fix:

```bash
pip install pytest-rerunfailures
```

Then verify the installation:

```bash
pytest --help | grep reruns
```

You should see `--reruns=RERUNS` in the output.

### Error 2: JSON report file not created

```
json_report.py: could not create report file: flake-report.json
```

This happens when the directory for the report file does not exist. The `--json-report-file` option expects the directory to exist.

Fix: Create the directory or use a path that exists:

```bash
mkdir -p reports
pytest --json-report --json-report-file=reports/flake-report.json
```

### Error 3: Quarantine directory not ignored

```
FAILED tests/quarantine/test_flaky_login.py::test_login_success - AssertionError...
```

The `--ignore` flag requires the correct path. If you run pytest from the project root and the quarantine directory is `tests/quarantine/`, the ignore path must be relative to the project root.

Fix:

```bash
pytest tests/ --ignore=tests/quarantine
```

Use the path relative to the current working directory, not an absolute path.

### Error 4: Flaky test detection shows false positives from legitimate failures

Your tracking script reports a test as flaky, but the test failure was caused by a real code change. This is the most dangerous false positive in flaky test management.

Fix: Add a code-change check to your tracking script. Only count a test as flaky if it failed on the same code that passed on the previous run:

```python
def is_same_code(run_a, run_b):
    """Check if the codebase is the same between two runs by comparing commit hashes."""
    return run_a.get("commit") == run_b.get("commit")
```

Include the commit hash in your report by adding it to the pytest metadata:

```python
# conftest.py
def pytest_report_header():
    import subprocess
    commit = subprocess.check_output(["git", "rev-parse", "HEAD"]).strip().decode()
    return [f"commit: {commit}"]
```

- You used `pytest-flaky` with `--flaky-attempts 3` to detect flaky tests by running the suite multiple times.
- You built a two-track pipeline with a gating main suite and a non-gating quarantine suite.
- You created a flake rate tracking script that records test outcomes over time and flags tests with a failure rate above 5%.

- You built a complete workflow that uploads flake reports as artifacts for post-run analysis.
