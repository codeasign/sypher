The exercises below are organized by difficulty. Start with Tier 1 even if you feel confident — the beginner exercises establish a baseline. After completing each exercise, verify your work before moving to the next one.

---

## Tier 1 — Beginner

These exercises involve single-command configuration tasks. Each one can be completed in under 30 seconds once you know the command.

### Exercise 1.1: Verify your Git installation

Open a terminal and confirm that Git is installed and available on your PATH. Write the command you used and the version number you received.

**Checklist:**

- [ ] You ran `git --version`
- [ ] The output contained a version number (for example, `git version 2.45.1`)
- [ ] If the command failed, you installed Git using the appropriate method for your OS
- [ ] After installation, you opened a new terminal window and verified again

**Stretch goal:** Run `which git` (macOS/Linux) or `where git` (Windows) to see the exact path of the Git executable. Write down that path.

### Exercise 1.2: Set your Git identity

Set your name and email address at the global level. Use your real name and the email address you plan to use for your Git commits.

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
```

**Checklist:**

- [ ] You ran `git config --global user.name "Your Name"` with your actual name
- [ ] You ran `git config --global user.email "your.email@example.com"` with your actual email
- [ ] You verified with `git config user.name` and `git config user.email`
- [ ] The output matches what you set

**Stretch goal:** Check what happens if you omit the `--global` flag while you are not inside a Git repository. Run `git config user.name` — does it set a value or produce an error? Write down the result.

### Exercise 1.3: Set your default editor and branch name

Configure Git to use your preferred editor and to create new repositories with the default branch name `main`.

```bash
git config --global core.editor "code --wait"
git config --global init.defaultBranch main
```

**Checklist:**

- [ ] You set `core.editor` to your preferred editor
- [ ] You set `init.defaultBranch` to `main`
- [ ] You verified both settings with `git config core.editor` and `git config init.defaultBranch`

**Stretch goal:** If you use VS Code, verify that the `--wait` flag works by running `git commit` without a `-m` flag in a test repository. VS Code should open with a commit message template. Close the editor window without typing a message — Git should abort the commit with a message like "Aborting commit due to empty commit message."

### Exercise 1.4: View your full configuration

Run `git config --list --show-origin` and examine the output.

**Checklist:**

- [ ] You ran the command and saw output
- [ ] You can identify which settings come from the system-level config file
- [ ] You can identify which settings come from your global `~/.gitconfig`
- [ ] You can see your `user.name`, `user.email`, `core.editor`, and `init.defaultBranch` in the output

**Stretch goal:** Redirect the output to a text file for later reference:

```bash
# Windows PowerShell
git config --list --show-origin > git-config-backup.txt

# macOS / Linux
git config --list --show-origin > ~/git-config-backup.txt
```

## Tier 2 — Intermediate

These exercises involve multiple commands and require you to think about how configuration levels interact.

### Exercise 2.1: Set up Git aliases

Git aliases let you create shortcuts for frequently used commands. Instead of typing `git status`, you can type `git st`. Instead of `git log --oneline --graph --all`, you can type `git tree`.

Set up the following aliases:

```bash
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.tree "log --oneline --graph --all --decorate"
```

**Checklist:**

- [ ] Each alias was set with `git config --global`
- [ ] `git st` produces the same output as `git status`
- [ ] `git tree` produces a graph of commits
- [ ] Run `git config --list --show-origin` and confirm the aliases appear in the output

**Stretch goal:** Create an alias called `git last` that shows the most recent commit, and an alias called `git unstage` that runs `git reset HEAD --`:

```bash
git config --global alias.last "log -1 HEAD"
git config --global alias.unstage "reset HEAD --"
```

### Exercise 2.2: Create a project-specific config override

Create a test directory, initialize it as a Git repository, and set a local identity that overrides your global identity.

```bash
mkdir ~/test-project
cd ~/test-project
git init
git config --local user.name "Test User (Local)"
git config --local user.email "test-local@example.com"
```

**Checklist:**

- [ ] You ran `git init` in the test directory
- [ ] You set `user.name` and `user.email` with `--local`
- [ ] From inside `~/test-project`, `git config user.name` shows `Test User (Local)` (the local value)
- [ ] From outside `~/test-project` (for example, `cd ~`), `git config user.name` shows your global name
- [ ] Run `git config --list --show-origin` from inside `~/test-project` and confirm both the global and local values appear, with the local file listed second

**Stretch goal:** Open the `.git/config` file in your test repository with a text editor and find the `[user]` section. The file should look like this:

```ini
[user]
        name = Test User (Local)
        email = test-local@example.com
```

### Exercise 2.3: Configure different line-ending behavior

Line ending handling differs between Windows (CRLF) and macOS/Linux (LF). Configure Git to handle this automatically based on your operating system.

```bash
# Windows: convert LF to CRLF on checkout, CRLF to LF on commit
git config --global core.autocrlf true

# macOS / Linux: convert CRLF to LF on commit only, do not convert on checkout
git config --global core.autocrlf input
```

**Checklist:**

- [ ] You set `core.autocrlf` to the correct value for your OS
- [ ] You verified with `git config core.autocrlf`
- [ ] The value is `true` on Windows or `input` on macOS/Linux

**Stretch goal:** Read the documentation for `core.autocrlf` by running `git help config`. Search for the `core.autocrlf` section. Write down what the `false` setting does.

### Exercise 2.4: Configure a commit template

Create a commit message template file that Git will pre-fill every time you run `git commit` without `-m`.

```bash
# Create the template file
echo "# Summary: " > ~/.git-commit-template.txt
echo "# " >> ~/.git-commit-template.txt
echo "# Description:" >> ~/.git-commit-template.txt
echo "# " >> ~/.git-commit-template.txt
echo "# Closes: " >> ~/.git-commit-template.txt

# Tell Git to use it
git config --global commit.template ~/.git-commit-template.txt
```

Now create a test repository, stage a file, and run `git commit` (without `-m`). Your editor should open with the template pre-filled.

**Checklist:**

- [ ] The template file exists at `~/.git-commit-template.txt`
- [ ] `git config commit.template` points to the correct path
- [ ] When you run `git commit` without `-m`, your editor opens with the template text

**Stretch goal:** Add a blank line at the top of the template file. Git treats everything after the first blank line as the commit body — the first line before the blank line is the commit subject. This lets you separate the subject line from the body automatically.

## Tier 3 — Advanced

These exercises simulate real-world scenarios that professional developers encounter.

### Exercise 3.1: Set up a team-standard `.gitconfig` file

You are joining a team that has standardized on specific Git settings. The team's standard `.gitconfig` excerpt is shown below. Apply these settings to your own global configuration.

```yaml
# Team standard settings
[user]
    name = Your Name
    email = your.name@company.com
[core]
    editor = code --wait
    autocrlf = true
    whitespace = trailing-space,space-before-tab
[init]
    defaultBranch = main
[alias]
    ci = commit -v
    st = status -sb
    ll = log --oneline --decorate --graph -20
    unstage = reset HEAD --
    amend = commit --amend --no-edit
[push]
    default = current
[fetch]
    prune = true
    pruneTags = true
```

**Checklist:**

- [ ] Each setting is applied using `git config --global`
- [ ] `git config --list` shows all `[core]`, `[alias]`, `[push]`, and `[fetch]` settings
- [ ] The `whitespace` setting is set to `trailing-space,space-before-tab`
- [ ] `git ll` shows the last 20 commits in a graph format
- [ ] `git amend` runs `git commit --amend --no-edit` (amends the last commit without opening an editor)

**Stretch goal:** Write the entire configuration block directly to `~/.gitconfig` by using a heredoc:

```bash
# macOS / Linux
cat >> ~/.gitconfig << 'EOF'

[push]
    default = current
[fetch]
    prune = true
    pruneTags = true
EOF
```

**Note:** Be careful when redirecting to `~/.gitconfig` — if you use `>` instead of `>>`, you overwrite the entire file. Always use `>>` to append.

### Exercise 3.2: Troubleshoot a PATH issue

You installed Git on Windows, but when you open Command Prompt and type `git --version`, you get this error:

```
'git' is not recognized as an internal or external command,
operable program or batch file.
```

You installed Git to the default location: `C:\Program Files\Git`. Simulate the troubleshooting process:

**Step 1:** Check if the `git.exe` file actually exists at the installation path:

```bash
# Windows Command Prompt
dir "C:\Program Files\Git\cmd\git.exe"

# Windows PowerShell
Test-Path "C:\Program Files\Git\cmd\git.exe"
```

**Step 2:** Check the current PATH value:

```bash
# Windows Command Prompt
echo %PATH%

# Windows PowerShell
$env:Path -split ";"
```

**Step 3:** Look for `C:\Program Files\Git\cmd` in the PATH output. If it is missing, you need to add it.

**Step 4:** Add Git to the PATH (if it is missing):

```bash
# Windows PowerShell (temporary — lasts only for this terminal session)
$env:Path += ";C:\Program Files\Git\cmd"

# Windows PowerShell (permanent — persists across restarts)
[Environment]::SetEnvironmentVariable(
    "Path",
    [Environment]::GetEnvironmentVariable("Path", "Machine") + ";C:\Program Files\Git\cmd",
    "Machine"
)
```

**Step 5:** Open a new terminal and run `git --version`.

**Checklist:**

- [ ] You verified that `git.exe` exists at the expected path
- [ ] You checked whether `C:\Program Files\Git\cmd` is in your PATH
- [ ] If Git was missing from PATH, you added it (temporarily or permanently)
- [ ] `git --version` works in a new terminal window

**Stretch goal:** On macOS or Linux, simulate the same exercise by installing Git to a non-standard location (for example, `~/apps/git`) and adding it to your PATH manually. This teaches you to handle situations where Git is installed but not on the system PATH.

### Exercise 3.3: Manage multiple Git identities

You have two Git identities: one for personal projects and one for your employer. Set up your global config to use your personal identity, then create a separate directory for work projects that uses your work identity.

```bash
# 1. Set global identity (personal)
git config --global user.name "Ada Lovelace"
git config --global user.email "ada.personal@example.com"

# 2. Create a work directory and clone (or init) a repo there
mkdir ~/work-projects
cd ~/work-projects
git init
# or: git clone https://github.com/your-company/project.git

# 3. Set local identity for work
git config --local user.name "Ada Lovelace (Company)"
git config --local user.email "ada@company.com"
```

**Verification test:**

```bash
# From inside ~/work-projects, Git should show the work identity
cd ~/work-projects
git config user.name    # Ada Lovelace (Company)
git config user.email   # ada@company.com

# From outside ~/work-projects, Git should show the personal identity
cd ~
git config user.name    # Ada Lovelace
git config user.email   # ada.personal@example.com
```

**Advanced approach — conditional includes (Git 2.13+):**

If you have Git 2.13 or later, you can automate identity switching using conditional includes. Instead of manually setting `--local` for each work repository, you configure Git to load a different config file when you are inside the `~/work-projects` directory:

```bash
# Create a separate config file for work
git config --file ~/.gitconfig-work user.name "Ada Lovelace (Company)"
git config --file ~/.gitconfig-work user.email "ada@company.com"

# Add the conditional include to your main ~/.gitconfig
git config --global includeIf.gitdir:~/work-projects/.path ~/.gitconfig-work
```

After this setup, any repository inside `~/work-projects` automatically uses the work identity. You never need to run `git config --local` for identity again.

**Checklist:**

- [ ] You set up two distinct identities (personal and work)
- [ ] You verified that `git config user.name` returns different values depending on the current directory
- [ ] (Optional) You set up a conditional include for automatic identity switching
- [ ] You confirmed the conditional include works by running `git config --list --show-origin` from inside a work project directory

**Stretch goal:** Add a third identity for open-source contributions. Use a different email address and a different commit template. Set up a separate directory for open-source projects (`~/oss-projects`) and configure a conditional include that loads the open-source identity automatically.

## Self-Assessment Checklist

After completing all three tiers, use this checklist to confirm you are ready to work with Git in any environment.

**Installation:**

- [ ] You can install Git on Windows using winget, Chocolatey, or the standalone installer.
- [ ] You can install Git on macOS using Homebrew or Xcode Command Line Tools.
- [ ] You can install Git on any Linux distribution using the correct package manager.
- [ ] You can verify the installation with `git --version` in a new terminal window.
- [ ] You understand what PATH is and can add Git to PATH manually if needed.

**Configuration:**

- [ ] You can set `user.name` and `user.email` at the global level.
- [ ] You can set `core.editor` and `init.defaultBranch`.
- [ ] You can view your full configuration with `git config --list --show-origin`.
- [ ] You can create local overrides for a specific project.
- [ ] You can create Git aliases for frequently used commands.
- [ ] You can configure `core.autocrlf` for your operating system.

**Advanced:**

- [ ] You can set up a team-standard `.gitconfig` with multiple sections.
- [ ] You can troubleshoot PATH issues when Git is installed but not found.
- [ ] You can manage multiple Git identities for different projects.
- [ ] You can use conditional includes to automate identity switching.
