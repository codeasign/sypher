You try again, this time in Command Prompt:

Your Mac-using colleague says to open Terminal and run `git --version`. You do that (on Windows), and the macOS command does the same thing — nothing. You search "install git." You close all 6 browser tabs and walk away. Git stays uninstalled. The tutorial remains unfinished. This experience — getting blocked before you can write a single command — is more common than it should be. The rest of this page exists so you never have to repeat it.

The frustration is not that Git is hard to install. It is that the installation process makes assumptions about your environment that may not hold. The tutorial assumes you have Git. The installer assumes you understand PATH. The error message tells you something is wrong, but it does not tell you what to do about it. Each assumption adds one more barrier between you and your first commit.

This page removes every assumption. By the time you finish reading, you will be able to install Git on any operating system, configure it exactly the way you want, and debug any installation or configuration error that comes up.

## The Concept

Git is a **distributed version control system** that tracks changes to files over time. It was created by Linus Torvalds in 2005 to manage the Linux kernel source code. Today it is the standard tool for version control in software development — used by individual developers, open-source projects, and enterprise teams alike.

Before you can use Git, you need two things:

1. **The Git executable installed on your machine.** This gives you the `git` command in your terminal.
2. **Configuration that tells Git who you are.** Git stamps every commit with a name and email address. If these are not set, Git refuses to create commits.

Installation is not a one-time event — you may need to install Git on multiple machines (work laptop, personal computer, a cloud VM) or update it as new versions ship. Configuration is also not a one-time event — you may need different identities for different projects or teams.

Git organizes its configuration into three levels:

| Level | Scope | Config File Location | Override Priority |
|-------|-------|---------------------|-------------------|
| System | Every user on the machine | `/etc/gitconfig` (Linux/macOS) or `C:\Program Files\Git\etc\gitconfig` (Windows) | Lowest |
| Global | Your user account on the machine | `~/.gitconfig` or `~/.config/git/config` | Medium |
| Local | One Git repository | `.git/config` inside that repository | Highest |

When you run `git config`, you choose which level to write to by passing a flag. If you do not specify one, the default is local (if you are inside a repository) or global (if you are not). The three levels form a chain: a setting defined at the system level can be overridden at the global level, which can be overridden at the local level. Git resolves any setting by starting at the most specific level (local) and falling back to the next level if the key is not found.

**The .gitconfig file format.** The global configuration file (`~/.gitconfig`) is a plain text file in INI format. Each section is marked by a section name in square brackets, and each setting is a key-value pair on its own line. Here is what a typical `.gitconfig` looks like:

```ini
[user]
        name = Ada Lovelace
        email = ada@example.com
[core]
        editor = code --wait
        autocrlf = true
[init]
        defaultBranch = main
[alias]
        st = status
        co = checkout
        br = branch
```

The file format is straightforward: `[section]` headers, followed by indented `key = value` lines. You can edit this file directly with a text editor instead of using `git config` commands. Editing directly is useful when you want to copy settings from one machine to another or when you need to make many changes at once. However, `git config` is safer because it handles formatting and prevents syntax errors.

**How Git finds its config files.** Git searches for configuration files in the following order:

1. System: `/etc/gitconfig` (Linux/macOS) or `C:\Program Files\Git\etc\gitconfig` (Windows). Requires administrator privileges to modify.
2. Global: `~/.gitconfig` (also checks `~/.config/git/config` on Linux). Specific to your user account.
3. Local: `.git/config` inside the current repository. Only exists after you run `git init` or `git clone`.

Git also respects two environment variables that can override these file paths:

- `GIT_CONFIG_SYSTEM` — overrides the system config file path.
- `GIT_CONFIG_GLOBAL` — overrides the global config file path.

These environment variables are rarely used in everyday development but are important to know about because CI/CD systems and containerized environments sometimes set them to control Git behavior without modifying files.

## Command and Syntax Reference

The table below covers every command you need to install Git and set it up for daily use.

| Command | Syntax | Flags / Options | Example | What It Does |
|---------|--------|-----------------|---------|--------------|
| Check Git version | `git --version` | None | `git --version` → `git version 2.45.1.windows.1` | Confirms Git is installed and prints the version number. If Git is not installed, this command produces an error. |
| Install on Debian/Ubuntu | `apt-get install git` | `-y` to skip confirmation | `sudo apt-get install git -y` | Installs the latest Git package from the distribution's repository. Requires `sudo`. |
| Install on Fedora/RHEL | `dnf install git` | None | `sudo dnf install git` | Installs Git using the DNF package manager. Replaces older `yum` on RHEL 8 and later. |
| Install on macOS (Homebrew) | `brew install git` | None | `brew install git` | Installs Git via Homebrew. If Homebrew is not installed, you must install it first from `brew.sh`. |
| Install on macOS (Xcode) | `xcode-select --install` | None | `xcode-select --install` | Installs the Xcode Command Line Tools, which include Git. |
| Install on Windows (winget) | `winget install Git.Git` | `--silent` to skip UI | `winget install Git.Git --silent` | Installs Git for Windows using the built-in Windows package manager (Windows 10 1809 and later). |
| Set user name | `git config --global user.name "Your Name"` | `--global`, `--local`, `--system` | `git config --global user.name "Ada Lovelace"` | Sets the name that appears in every commit you create. |
| Set user email | `git config --global user.email "ada@example.com"` | `--global`, `--local`, `--system` | `git config --global user.email "ada@example.com"` | Sets the email that appears in every commit you create. |
| Set default editor | `git config --global core.editor EDITOR` | `--global` | `git config --global core.editor "code --wait"` | Sets the text editor Git opens for commit messages, rebase instructions, and other interactive input. |
| Set default branch name | `git config --global init.defaultBranch main` | `--global` | `git config --global init.defaultBranch main` | Changes the default branch name from `master` to `main` when you run `git init`. |
| List all config | `git config --list` | `--global`, `--local`, `--system`, `--show-origin` | `git config --list --show-origin` | Prints all settings Git is currently using, merging the three config levels. With `--show-origin`, it also prints the file path where each setting was defined. |
| Get one value | `git config KEY` | None | `git config user.name` → `Ada Lovelace` | Prints the value of a single key. |
| Get all values for a key | `git config --get-all KEY` | `--global`, `--local`, `--system` | `git config --get-all user.name` | Prints every value Git finds for the key across all config levels. Useful when the same key is set in multiple files. |
| Unset a key | `git config --unset KEY` | `--global`, `--local`, `--system` | `git config --global --unset user.name` | Removes a key from the specified config level. |

## How It Actually Works — Step by Step

### Step 1: Download the Git Installer

Git is distributed as a native installer for each operating system. The download page at `git-scm.com/downloads` automatically detects your OS and offers the correct package.

For Windows, the installer is an `.exe` file (roughly 50 to 70 MB depending on the version). For macOS, you can download a `.dmg` file or use Homebrew. For Linux, you use the system package manager rather than a downloaded installer — the package manager pulls Git from the distribution's repository.

The file you download is not Git itself; it is an installer that unpacks Git onto your system. This distinction matters because the installer also handles two critical setup tasks: adding Git to your PATH environment variable and associating file types with Git.

### Step 2: Run the Installer

**Windows.** The Git for Windows installer is a graphical wizard with 10 to 12 screens. The most important screen is "Adjusting your PATH environment," which gives you three options:

- **"Use Git from Git Bash only"** — Git is available only inside the Git Bash terminal that ships with the installer. Not recommended.
- **"Use Git from the Windows Command Prompt"** — Git is available in Command Prompt and PowerShell. This is the recommended choice.
- **"Use Git and optional Unix tools from the Command Prompt"** — Git plus a collection of Unix utilities (grep, sed, awk) are added to PATH. These tools can conflict with existing Windows tools. Use this only if you know you need the Unix tools.

You should choose the second option. This adds `C:\Program Files\Git\cmd` to your system PATH variable, which makes `git` available in every terminal window.

**macOS.** If you use Homebrew, the command `brew install git` handles everything — it downloads the source, compiles it, and places the binary in `/usr/local/bin/git`. If you use the `.dmg` installer, you open the disk image, run the `.pkg` file, and follow the wizard (which has 3 to 4 screens). If you use `xcode-select --install`, a system dialog appears, downloads the Command Line Tools package (roughly 150 MB), and installs Git alongside compilers and other developer utilities.

**Linux.** Every major distribution includes Git in its package repository:

```bash
# Debian / Ubuntu / Linux Mint
sudo apt-get update
sudo apt-get install git -y

# Fedora / RHEL / CentOS
sudo dnf install git

# openSUSE
sudo zypper install git

# Arch Linux / Manjaro
sudo pacman -S git
```

No manual PATH configuration is needed — the package manager places the `git` binary in `/usr/bin/git`, which is already in the default PATH on all Linux distributions.

### Step 3: Verify the Installation

After the installer finishes, you open a **new** terminal window — not the one that was open before the install. Old terminal windows still hold the old PATH value in memory. In a new terminal, run:

```bash
git --version
```

You should see output like:

```
git version 2.45.1
```

The exact version number depends on when you installed. If you see a version number, Git is installed and available on your PATH. If you see an error, the installer did not add Git to your PATH, or you are still using an old terminal window.

### Step 4: Check for an Existing .gitconfig File

Git reads its global configuration from `~/.gitconfig`. You can check whether this file exists:

```bash
# On macOS and Linux
cat ~/.gitconfig

# On Windows (PowerShell)
Get-Content ~\.gitconfig

# On Windows (CMD)
type %USERPROFILE%\.gitconfig
```

If the file does not exist, that is normal — Git creates it automatically the first time you run `git config --global` with a write command. If the file does exist but is empty or contains only a few lines, you can add to it safely.

### Step 5: Set Your Identity

Git requires two pieces of information to create commits: a name and an email address. These are stored as `user.name` and `user.email`.

```bash
git config --global user.name "Ada Lovelace"
git config --global user.email "ada@example.com"
```

The `--global` flag writes these values to `~/.gitconfig`. Every commit you create from this point forward uses this name and email. If you need a different identity for a specific project, you navigate into that project's directory and use `--local`:

```bash
cd ~/projects/work-repo
git config --local user.name "Ada L. (Work)"
git config --local user.email "ada@company.com"
```

This writes to `.git/config` inside that repository. The local values override the global ones for that project only.

### Step 6: Configure Your Editor and Default Branch

Two additional settings make Git more pleasant to use:

```bash
# Set VS Code as the default editor for commit messages
git config --global core.editor "code --wait"

# Set the default branch name to "main" for new repositories
git config --global init.defaultBranch main
```

The `core.editor` setting controls which editor Git opens when it needs you to type a commit message (or resolve merge conflicts). The `--wait` flag tells VS Code to block the terminal until you close the editor window. Without it, Git would think you finished typing before you actually did.

The `init.defaultBranch` setting controls the name Git assigns to the first branch in a new repository. The historical default was `master`, but many teams now use `main`. Setting this to `main` at the global level means every repository you initialize from now on starts with a branch called `main`.

### Step 7: Review Your Configuration

To see everything you have configured:

```bash
git config --list --show-origin
```

This prints every setting Git knows about, prefixed by the file path where each setting is defined:

```
file:C:/Program Files/Git/etc/gitconfig    core.symlinks=false
file:C:/Program Files/Git/etc/gitconfig    core.autocrlf=true
file:C:/Users/ada/.gitconfig               user.name=Ada Lovelace
file:C:/Users/ada/.gitconfig               user.email=ada@example.com
file:C:/Users/ada/.gitconfig               core.editor=code --wait
file:C:/Users/ada/.gitconfig               init.defaultBranch=main
```

The `--show-origin` flag is invaluable when you are debugging unexpected behavior. If Git uses a value you did not expect, the origin tells you which file set it.

### Step 8: Test Your Configuration with a Real Commit

The best way to confirm that your installation and configuration work correctly is to create a test repository, stage a file, and make a commit. This exercises every component of your setup: the Git executable, the PATH, your identity settings, and your editor configuration.

```bash
# Create a temporary directory and enter it
mkdir ~/git-test
cd ~/git-test

# Initialize a new Git repository
git init

# Create a test file
echo "Hello, Git!" > README.md

# Stage the file
git add README.md

# Commit the file
git commit -m "Initial commit: test installation"
```

If every command succeeds, Git is installed and configured correctly. You should see output like:

```
[main (root-commit) a1b2c3d] Initial commit: test installation
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

If the commit fails with "Please tell me who you are," your identity settings are missing or incorrect. Return to Step 5 and verify your `user.name` and `user.email` settings.

If the commit opens your editor unexpectedly, you may have omitted the `-m` flag. If you configured `core.editor` and the editor does not open, verify that the editor path is correct — for example, `code --wait` requires VS Code to be on your PATH.

### Step 9: Clean Up Your Test Repository

After verifying that everything works, remove the test repository to keep your home directory clean:

```bash
cd ~
rm -rf ~/git-test
```

On Windows PowerShell, use:

```powershell
cd ~
Remove-Item -Recurse -Force ~/git-test
```

This step is optional but recommended. The test repository served its purpose — you confirmed that Git is installed, configured, and ready to use.

### Step 10: Understand the Config File Format on Disk

Now that you have set values with `git config`, open the global config file in a text editor and examine its structure:

```bash
# macOS / Linux
cat ~/.gitconfig

# Windows PowerShell
Get-Content ~\.gitconfig
```

You should see a file that looks like this:

```ini
[user]
        name = Ada Lovelace
        email = ada@example.com
[core]
        editor = code --wait
[init]
        defaultBranch = main
```

Each `[section]` header groups related settings. The indentation between the section header and the key-value pairs is a single tab character. Git is strict about this format — if you edit the file manually, you must preserve the tab indentation and the `[section]` syntax.

You can also create a brand new config file from scratch by writing it directly:

```bash
cat > ~/.gitconfig-custom << 'EOF'
[user]
        name = Ada Lovelace
        email = ada@example.com
[core]
        editor = vim
[init]
        defaultBranch = main
EOF

# Tell Git to use this custom config instead of the default
git config --global include.path ~/.gitconfig-custom
```

This `include.path` directive tells Git to load the settings from `~/.gitconfig-custom` as if they were part of the main config file. This is useful for organizing settings into separate files by category — for example, one file for identity settings, another for aliases, and a third for tool-specific settings.

## When to Use This and When Not To

### Install Git when:

- You are setting up a new machine for development and Git is not yet installed.
- You are on a shared server or cloud VM that does not have Git.
- Your current Git version is older than 2.28 (versions before 2.28 lack `init.defaultBranch`).
- You want to use a different Git identity for a specific project.
- You are setting up a CI/CD pipeline that needs to clone repositories.
- You are preparing a Docker image that will run Git commands during the build process.

### Do not install Git when:

- Git is already installed and the version is current (check with `git --version`).
- You are on a restricted machine where you cannot install software. In this case, you can use a cloud-based Git environment such as GitHub Codespaces or GitLab Web IDE.
- You are only browsing code, not contributing. You can view repositories through a web browser without Git installed.
- You are using a managed development environment that already includes Git (for example, GitHub Codespaces, Replit, or Gitpod).
- You are running a production server that does not need Git — installing Git on a production server increases the attack surface and is unnecessary if the server only serves traffic.

### Configure at the right level:

- Use **system** for settings that every user on the machine should share — typically only system administrators use this level. For example, a team of developers sharing a build server might set `core.autocrlf` at the system level so everyone gets the same line-ending behavior.
- Use **global** for your personal identity and preferences — this is the level you will use 95 percent of the time. Your name, email, editor, and default branch name belong here.
- Use **local** when you need different settings for one project (different email for work versus personal projects, different editor, different commit template).

### Config priority in practice:

The three-level system is not just academic — it solves real problems. Consider this scenario:

- Your system config sets `core.autocrlf = true` (Windows, CRLF line endings).
- Your global config sets `user.name = Ada Lovelace` and `user.email = ada@example.com`.
- You work on a Linux project that requires `core.autocrlf = input` and a different email.

You can set the local config in that project's directory to override both `core.autocrlf` and `user.email` without affecting any other project. The system-level `core.autocrlf` is still active for every other repository, but the local value takes precedence in this one.
