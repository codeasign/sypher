The following exercises progress from basic reliability concepts to advanced failure triage strategies. Each exercise includes a hint to guide you and a full solution to check against.

## Tier 1 — Beginner

These exercises focus on the fundamental building blocks of pipeline reliability: timeouts, retries, and failure conditions.

### Exercise 1.1: Add a Job-Level Timeout

Your team has a workflow that occasionally hangs during the dependency installation step. The workflow runs indefinitely and blocks the runner queue. Add a job-level timeout to prevent this.

**Workflow file:**
```yaml
name: Build
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
```

**Hint:** The `timeout-minutes` key can be set at the job level. Consider what a reasonable maximum time is for a full build pipeline.

**Solution:**
```yaml
name: Build
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
        timeout-minutes: 5
      - run: npm run build
        timeout-minutes: 8
```

The job-level timeout of 15 minutes acts as a safety net. The step-level timeouts provide more granular control: if `npm ci` takes more than 5 minutes, it is likely a network issue, and if `npm run build` takes more than 8 minutes, there is likely a performance regression.

### Exercise 1.2: Use `if: failure()` to Upload Logs on Failure

Your workflow runs tests, but when the tests fail, you have no way to inspect what happened because the test output is lost. Add a step that uploads test logs only when the test step fails.

**Workflow file:**
```yaml
name: Test
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test -- --reporters=default --outputFile=test-results.json
```

**Hint:** Use `if: ${{ failure() }}` on a step that uploads the test results file. The `failure()` function returns `true` when any previous step in the job has failed.

**Solution:**
```yaml
name: Test
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test -- --reporters=default --outputFile=test-results.json
      - name: Upload test results on failure
        if: ${{ failure() }}
        uses: actions/upload-artifact@v4
        with:
          name: test-results-${{ github.run_id }}
          path: test-results.json
```

This ensures that test results are only uploaded when the tests fail. On success, the artifact upload step is skipped, saving storage space and upload time.

### Exercise 1.3: Mark a Non-Critical Step with `continue-on-error`

Your workflow has a code coverage step that you want to track but do not want to block the build. If coverage drops below the threshold, the step should show a warning in the logs, but the rest of the pipeline should continue.

**Workflow file:**
```yaml
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - name: Check coverage
        run: |
          COVERAGE=72
          if [ "$COVERAGE" -lt 80 ]; then
            echo "Coverage $COVERAGE% is below 80% threshold"
            exit 1
          fi
      - run: echo "Deploying..."
```

**Hint:** Add `continue-on-error: true` to the coverage check step so the workflow continues even if coverage is below the threshold. The step will show as failed in the logs but the job will not fail.

**Solution:**
```yaml
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - name: Check coverage
        continue-on-error: true
        run: |
          COVERAGE=72
          if [ "$COVERAGE" -lt 80 ]; then
            echo "Coverage $COVERAGE% is below 80% threshold"
            exit 1
          fi
      - run: echo "Deploying..."
```

The step will fail with exit code 1 and print the warning, but the `continue-on-error: true` flag means the job continues to the "Deploying..." step. The coverage warning is visible in the logs but does not block the deployment.

### Exercise 1.4: Use `if: always()` to Upload Logs Regardless of Outcome

You want to collect all logs from every workflow run, whether it passes or fails. This is useful for auditing and post-mortem analysis.

**Workflow file:**
```yaml
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

**Hint:** Use `if: ${{ always() }}` on a step that runs at the end of the job. The `always()` function returns `true` regardless of whether previous steps passed or failed.

**Solution:**
```yaml
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - name: Upload logs
        if: ${{ always() }}
        uses: actions/upload-artifact@v4
        with:
          name: build-logs-${{ github.run_id }}
          path: /var/log/*.log
```

With `if: always()`, the log upload step runs even if `npm test` fails. This ensures you always have access to diagnostic information for every run.

## Tier 2 — Intermediate

These exercises combine multiple reliability techniques into cohesive strategies.

### Exercise 2.1: Implement a Retry with Exponential Backoff

Your workflow makes an HTTP call to an external API that occasionally returns 429 (Too Many Requests) or 503 (Service Unavailable). Implement a retry with exponential backoff that retries up to 5 times with increasing delays.

**Workflow snippet:**
```yaml
      - name: Fetch external data
        run: |
          curl -s https://api.example.com/data > data.json
```

**Hint:** Implement the retry logic as a shell script in the `run` block. Use a `for` loop, `sleep` with exponential backoff, and check the HTTP status code. The formula for delay can be `delay = base * 2^attempt` where `base` is 2 seconds.

**Solution:**
```yaml
      - name: Fetch external data with exponential backoff
        run: |
          MAX_ATTEMPTS=5
          BASE_DELAY=2
          for i in $(seq 1 $MAX_ATTEMPTS); do
            echo "Attempt $i of $MAX_ATTEMPTS"
            HTTP_CODE=$(curl -s -o data.json -w "%{http_code}" https://api.example.com/data)
            if [ "$HTTP_CODE" -eq 200 ]; then
              echo "Success on attempt $i"
              exit 0
            fi
            echo "HTTP $HTTP_CODE, retrying in $((BASE_DELAY * 2**i)) seconds..."
            sleep $((BASE_DELAY * 2**i))
          done
          echo "All $MAX_ATTEMPTS attempts failed"
          exit 1
```

The exponential backoff increases the delay between retries: 2s, 4s, 8s, 16s, 32s. This gives the API time to recover from transient overload without hammering it with rapid retries.

### Exercise 2.2: Categorize Failures and Send Different Notifications

Create a workflow with three jobs: `build`, `test`, and `deploy`. If `build` fails, notify the infrastructure team. If `test` fails, notify the QA team. If `deploy` fails, notify the platform team. Each notification should go to a different Slack channel.

**Workflow file:**
```yaml
name: CI/CD
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
  test:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
  deploy:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - run: echo "Deploying..."
```

**Hint:** Create separate notification jobs for each failure scenario. Use `needs` and `if: failure()` to target the specific job that failed. You can use different secrets for different Slack webhook URLs. Consider using `github.job` to identify which job failed.

**Solution:**
```yaml
name: CI/CD
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  notify-build-failure:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ failure() && needs.build.result == 'failure' }}
    steps:
      - run: echo "Notifying infra team about build failure"
      - name: Send Slack notification
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "Build failed — notify infrastructure team"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.INFRA_SLACK_WEBHOOK }}

  test:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  notify-test-failure:
    runs-on: ubuntu-latest
    needs: test
    if: ${{ failure() && needs.test.result == 'failure' }}
    steps:
      - run: echo "Notifying QA team about test failure"
      - name: Send Slack notification
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "Tests failed — notify QA team"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.QA_SLACK_WEBHOOK }}

  deploy:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - run: echo "Deploying..."

  notify-deploy-failure:
    runs-on: ubuntu-latest
    needs: deploy
    if: ${{ failure() && needs.deploy.result == 'failure' }}
    steps:
      - run: echo "Notifying platform team about deploy failure"
      - name: Send Slack notification
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text": "Deploy failed — notify platform team"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.PLATFORM_SLACK_WEBHOOK }}
```

Each notification job depends on its corresponding job and checks `needs.<job>.result == 'failure'` to ensure it only fires when that specific job failed. This prevents the infra team from being paged when a test fails.

### Exercise 2.3: Add `workflow_dispatch` with Comprehensive Inputs

Your workflow needs to support manual re-runs with the following capabilities:
- Select which job to run (all, build-only, test-only, deploy-only).
- Enable debug mode, which adds verbose logging to every step.
- Skip the deploy step for testing purposes.
- Specify a custom branch to check out.

**Hint:** Use `workflow_dispatch` with `inputs` of type `choice`, `boolean`, and `string`. Use `if` conditions to check the input values throughout the workflow.

**Solution:**
```yaml
name: CI/CD
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:
    inputs:
      job-selector:
        description: "Select which job to run"
        type: choice
        options:
          - all
          - build-only
          - test-only
          - deploy-only
        default: all
      debug:
        description: "Enable debug logging"
        type: boolean
        default: false
      skip-deploy:
        description: "Skip the deploy step"
        type: boolean
        default: false
      target-branch:
        description: "Branch to check out"
        type: string
        default: ""

jobs:
  build:
    runs-on: ubuntu-latest
    if: ${{ inputs.job-selector == 'all' || inputs.job-selector == 'build-only' }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ inputs.target-branch || github.ref }}
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Build
        run: npm run build
      - name: Debug info
        if: ${{ inputs.debug }}
        run: |
          echo "Node version: $(node --version)"
          echo "npm version: $(npm --version)"
          echo "OS: $(uname -a)"
          echo "Branch: ${{ inputs.target-branch || github.ref }}"

  test:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ (inputs.job-selector == 'all' || inputs.job-selector == 'test-only') && !cancelled() }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ inputs.target-branch || github.ref }}
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
        env:
          DEBUG: ${{ inputs.debug && 'true' || 'false' }}

  deploy:
    runs-on: ubuntu-latest
    needs: test
    if: ${{ inputs.job-selector == 'all' && inputs.skip-deploy != true && !cancelled() }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ inputs.target-branch || github.ref }}
      - run: echo "Deploying to production..."
```

The `workflow_dispatch` inputs give you fine-grained control over which jobs run, what branch they use, and whether debug logging is enabled. This is essential for debugging production issues without running the full pipeline.

### Exercise 2.4: Handle `fail-fast` in a Matrix Build

Your matrix build runs tests on 4 operating systems (ubuntu, windows, macos, and a legacy ubuntu-20.04). If one OS fails, the others should still complete so you can see the full picture of which OS versions are affected.

**Workflow snippet:**
```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest, ubuntu-20.04]
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

**Hint:** Set `fail-fast: false` at the strategy level. This tells GitHub Actions to continue running the remaining matrix jobs even if one of them fails.

**Solution:**
```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest, ubuntu-20.04]
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

With `fail-fast: false`, if the Windows job fails due to a path separator issue, the macOS and Ubuntu jobs will still run. This gives you a complete picture of which platforms are affected by a change. Without it, the first failure cancels all in-progress matrix jobs, hiding potentially useful information.

## Tier 3 — Advanced

These exercises require you to design complete reliability strategies from scratch.

### Exercise 3.1: Design a Self-Healing Pipeline

Your pipeline has three stages: lint, test, and deploy. Each stage can fail for different reasons:

- Lint fails intermittently due to a race condition in the linter cache.
- Tests fail occasionally due to network timeouts when downloading test fixtures.
- Deploy fails when the deployment target is undergoing maintenance.

Design a pipeline that:
- Retries lint up to 2 times with a 10-second delay between attempts.
- Retries tests up to 3 times with exponential backoff (5s, 25s, 125s).
- Does not retry deploy — instead, it posts a message to a deployment channel and waits for manual approval.
- Uploads all logs regardless of outcome.
- Sends a summary notification to Slack at the end of the run.

**Reasoning-based solution:**

```yaml
name: Self-Healing Pipeline
on: [push, workflow_dispatch]

jobs:
  lint:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Lint with retry
        run: |
          for i in 1 2 3; do
            echo "Lint attempt $i of 3"
            if npm run lint; then
              echo "Lint passed on attempt $i"
              exit 0
            fi
            if [ $i -lt 3 ]; then
              echo "Lint failed, retrying in 10 seconds..."
              sleep 10
            fi
          done
          echo "Lint failed after 3 attempts"
          exit 1
        timeout-minutes: 3

  test:
    runs-on: ubuntu-latest
    needs: lint
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Test with exponential backoff
        run: |
          DELAYS=(5 25 125)
          for i in 0 1 2; do
            echo "Test attempt $((i+1)) of 3"
            if npm test; then
              echo "Tests passed on attempt $((i+1))"
              exit 0
            fi
            if [ $i -lt 2 ]; then
              echo "Tests failed, retrying in ${DELAYS[$i]} seconds..."
              sleep ${DELAYS[$i]}
            fi
          done
          echo "Tests failed after 3 attempts"
          exit 1
        timeout-minutes: 10

  deploy:
    runs-on: ubuntu-latest
    needs: test
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Attempt deployment
        id: deploy
        continue-on-error: true
        run: |
          echo "Attempting deployment..."
          if curl -s -o /dev/null -w "%{http_code}" https://deploy.example.com/status | grep -q "200"; then
            echo "Deploy target is available"
            echo "Deploying..."
            echo "Deployment complete"
          else
            echo "Deploy target is undergoing maintenance"
            exit 1
          fi
      - name: Notify on deploy failure
        if: ${{ steps.deploy.outcome == 'failure' }}
        run: |
          echo "Deployment failed — target is undergoing maintenance"
          echo "Posting to deployment channel for manual approval..."

  upload-logs:
    runs-on: ubuntu-latest
    needs: [lint, test, deploy]
    if: ${{ always() }}
    steps:
      - name: Upload all logs
        uses: actions/upload-artifact@v4
        with:
          name: pipeline-logs-${{ github.run_id }}
          path: /tmp/*.log

  notify-summary:
    runs-on: ubuntu-latest
    needs: [lint, test, deploy]
    if: ${{ always() }}
    steps:
      - name: Build summary
        run: |
          echo "## Pipeline Summary" >> $GITHUB_STEP_SUMMARY
          echo "Lint: ${{ needs.lint.result }}" >> $GITHUB_STEP_SUMMARY
          echo "Test: ${{ needs.test.result }}" >> $GITHUB_STEP_SUMMARY
          echo "Deploy: ${{ needs.deploy.result }}" >> $GITHUB_STEP_SUMMARY
      - name: Send Slack summary
        run: |
          echo "Sending summary to Slack..."
          echo "Lint: ${{ needs.lint.result }}"
          echo "Test: ${{ needs.test.result }}"
          echo "Deploy: ${{ needs.deploy.result }}"
```

The key design decisions in this solution are:
- Lint uses a simple fixed-delay retry because the failure is cache-related and resolves quickly.
- Tests use exponential backoff because network timeouts are resolved by waiting longer.
- Deploy uses `continue-on-error` with notification instead of retry because maintenance windows are not resolved by retrying.
- The `upload-logs` and `notify-summary` jobs use `if: always()` to run regardless of outcome.
- The summary uses `needs.<job>.result` to report the status of each job individually.

### Exercise 3.2: Build a Failure Dashboard with GitHub Artifacts

Design a workflow that collects failure data from multiple jobs, aggregates it into a JSON report, and uploads it as an artifact. The report should include:
- The job name and result for each job.
- The duration of each job.
- The error message of the first failing step in each job.
- The runner name that executed each job.
- A timestamp of when the workflow was triggered.

**Reasoning-based solution:**

```yaml
name: Failure Dashboard
on: [push, workflow_dispatch]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - name: Run lint
        continue-on-error: true
        id: lint-step
        run: |
          echo "Running lint..."
          npm run lint 2>&1 || { echo "LINT_ERROR=true" >> $GITHUB_ENV; echo "Lint failed" >> error.log; }
      - name: Record lint metadata
        run: |
          echo "{\"job\":\"lint\",\"result\":\"${{ steps.lint-step.outcome }}\",\"duration\":\"$(date -u +%s)\",\"runner\":\"${{ runner.name }}\"}" > lint-meta.json

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - name: Run tests
        continue-on-error: true
        id: test-step
        run: |
          echo "Running tests..."
          npm test 2>&1 || { echo "TEST_ERROR=true" >> $GITHUB_ENV; echo "Tests failed" >> error.log; }
      - name: Record test metadata
        run: |
          echo "{\"job\":\"test\",\"result\":\"${{ steps.test-step.outcome }}\",\"duration\":\"$(date -u +%s)\",\"runner\":\"${{ runner.name }}\"}" > test-meta.json

  build-report:
    runs-on: ubuntu-latest
    needs: [lint, test]
    if: ${{ always() }}
    steps:
      - name: Collect metadata from all jobs
        run: |
          echo "{" > report.json
          echo "\"workflow\": \"${{ github.workflow }}\"," >> report.json
          echo "\"run_id\": ${{ github.run_id }}," >> report.json
          echo "\"timestamp\": \"$(date -u -Iseconds)\"," >> report.json
          echo "\"branch\": \"${{ github.ref_name }}\"," >> report.json
          echo "\"commit\": \"${{ github.sha }}\"," >> report.json
          echo "\"jobs\": {" >> report.json
          echo "  \"lint\": { \"result\": \"${{ needs.lint.result }}\" }," >> report.json
          echo "  \"test\": { \"result\": \"${{ needs.test.result }}\" }" >> report.json
          echo "}," >> report.json
          echo "\"trigger\": \"${{ github.event_name }}\"" >> report.json
          echo "}" >> report.json
      - name: Upload aggregated report
        uses: actions/upload-artifact@v4
        with:
          name: failure-report-${{ github.run_id }}
          path: report.json
```

This report structure allows you to aggregate failure data across runs and analyze trends. You could extend this by adding a separate job that downloads and merges reports from multiple runs to create a historical view.

### Exercise 3.3: Implement Graceful Degradation for External Dependencies

Your pipeline depends on three external services: npm registry, a Docker registry, and a secrets manager. If any of these services is unavailable, the pipeline should degrade gracefully rather than hard-failing. Design a workflow that:
- Checks each service availability before using it.
- If npm registry is down, uses a local cache or mirror.
- If Docker registry is down, skips Docker-related steps.
- If secrets manager is down, falls back to locally available credentials.
- Logs which services were unavailable and includes this in the final report.

**Reasoning-based solution:**

```yaml
name: Graceful Degradation Pipeline
on: [push, workflow_dispatch]

jobs:
  check-services:
    runs-on: ubuntu-latest
    outputs:
      npm-status: ${{ steps.check-npm.outputs.status }}
      docker-status: ${{ steps.check-docker.outputs.status }}
      secrets-status: ${{ steps.check-secrets.outputs.status }}
    steps:
      - id: check-npm
        run: |
          if curl -s -o /dev/null -w "%{http_code}" https://registry.npmjs.org/ | grep -q "200"; then
            echo "status=available" >> $GITHUB_OUTPUT
          else
            echo "status=unavailable" >> $GITHUB_OUTPUT
          fi
      - id: check-docker
        run: |
          if curl -s -o /dev/null -w "%{http_code}" https://registry-1.docker.io/ | grep -q "200"; then
            echo "status=available" >> $GITHUB_OUTPUT
          else
            echo "status=unavailable" >> $GITHUB_OUTPUT
          fi
      - id: check-secrets
        run: |
          if [ -n "${{ secrets.AWS_ACCESS_KEY_ID }}" ]; then
            echo "status=available" >> $GITHUB_OUTPUT
          else
            echo "status=unavailable" >> $GITHUB_OUTPUT
          fi

  build:
    runs-on: ubuntu-latest
    needs: check-services
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        if: ${{ needs.check-services.outputs.npm-status == 'available' }}
        run: npm ci
      - name: Install from cache fallback
        if: ${{ needs.check-services.outputs.npm-status != 'available' }}
        run: |
          echo "npm registry unavailable, using local cache"
          if [ -d "node_modules" ]; then
            echo "Using cached node_modules"
          else
            echo "No cache available, installing from local registry mirror"
            npm ci --registry https://npm-mirror.internal.example.com
          fi
      - name: Build Docker image
        if: ${{ needs.check-services.outputs.docker-status == 'available' }}
        run: docker build -t myapp .
      - name: Skip Docker build
        if: ${{ needs.check-services.outputs.docker-status != 'available' }}
        run: echo "Docker registry unavailable, skipping Docker build"
      - name: Deploy with secrets
        if: ${{ needs.check-services.outputs.secrets-status == 'available' }}
        run: echo "Deploying with AWS credentials from secrets manager"
      - name: Deploy without secrets
        if: ${{ needs.check-services.outputs.secrets-status != 'available' }}
        run: |
          echo "Secrets manager unavailable, using local credentials"
          echo "Deploying to staging with local credentials"

  report-degradation:
    runs-on: ubuntu-latest
    needs: [check-services, build]
    if: ${{ always() }}
    steps:
      - name: Generate degradation report
        run: |
          echo "## Service Degradation Report" >> $GITHUB_STEP_SUMMARY
          echo "npm: ${{ needs.check-services.outputs.npm-status }}" >> $GITHUB_STEP_SUMMARY
          echo "Docker: ${{ needs.check-services.outputs.docker-status }}" >> $GITHUB_STEP_SUMMARY
          echo "Secrets: ${{ needs.check-services.outputs.secrets-status }}" >> $GITHUB_STEP_SUMMARY
          echo "Build result: ${{ needs.build.result }}" >> $GITHUB_STEP_SUMMARY
      - name: Upload degradation report
        uses: actions/upload-artifact@v4
        with:
          name: degradation-report-${{ github.run_id }}
          path: /tmp/step-summary-*.md
```

The graceful degradation pattern ensures your pipeline remains functional even when external dependencies are unavailable. Each service check is independent, so one failure does not cascade. The degradation report provides visibility into which services were affected, allowing you to investigate and restore service without an emergency pipeline fix.
