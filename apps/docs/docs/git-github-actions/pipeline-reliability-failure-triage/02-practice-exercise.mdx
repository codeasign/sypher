## Setup

Before you start, make sure you have the following ready:

- A GitHub account with a repository where you can create and run workflows.
- A GitHub personal access token (classic or fine-grained) with `repo` and `workflow` scopes.
- A text editor for writing YAML files.
- A Slack webhook URL (optional, for the notification step). If you do not have one, you can use a dummy URL and inspect the logs instead.

Create a new directory for this exercise:

```bash
mkdir -p ~/github-actions-reliability
cd ~/github-actions-reliability
```

Create a simple Node.js project to test with:

```bash
npm init -y
echo "console.log('hello world');" > index.js
```

Initialize a git repository and create the workflow directory:

```bash
git init
mkdir -p .github/workflows
```

## The Walkthrough

### Step 1: Create a Baseline Unreliable Workflow

Create a workflow that has no reliability features. This is your starting point — the pipeline that fails on every transient error.

Create `.github/workflows/ci.yml` with the following content:

```yaml
name: CI
on: [push, workflow_dispatch]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
      - run: echo "Build complete"
```

Commit and push this to your repository:

```bash
git add -A
git commit -m "Add baseline unreliable workflow"
git remote add origin https://github.com/YOUR_USER/YOUR_REPO.git
git push -u origin main
```

**Checkpoint:** Go to the Actions tab in your repository. You should see the workflow run. It will pass because there is nothing wrong with it yet. The point is to establish a baseline.

### Step 2: Simulate a Transient Failure

Add a step that simulates a transient failure — a network timeout or a flaky download. This will make the pipeline fail intermittently, which is the problem you need to solve.

Replace the content of `.github/workflows/ci.yml` with:

```yaml
name: CI
on: [push, workflow_dispatch]
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
        timeout-minutes: 5
      - name: Simulate flaky network call
        run: |
          if [ $((RANDOM % 3)) -eq 0 ]; then
            echo "Simulating network timeout..."
            sleep 30
            exit 1
          else
            echo "Network call succeeded"
          fi
      - run: npm test
      - run: echo "Build complete"
```

Push this change:

```bash
git add -A
git commit -m "Add flaky network simulation"
git push
```

**Checkpoint:** Run the workflow multiple times (use `workflow_dispatch` to trigger it manually). About one in three runs will fail with the simulated timeout. This is the unreliable pipeline you need to fix.

### Step 3: Add the Retry Mechanism

Now add the retry mechanism. Use `actions/retry-step` to retry the flaky step up to 3 times. This is the core reliability feature.

Update `.github/workflows/ci.yml`:

```yaml
name: CI
on: [push, workflow_dispatch]
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
        timeout-minutes: 5
      - name: Flaky network call with retry
        uses: actions/retry-step@v1
        with:
          max_attempts: 3
          retry_on: exit
          command: |
            if [ $((RANDOM % 3)) -eq 0 ]; then
              echo "Simulating network timeout..."
              sleep 30
              exit 1
            else
              echo "Network call succeeded"
            fi
          timeout_minutes: 2
      - run: npm test
      - run: echo "Build complete"
```

Push this change:

```bash
git add -A
git commit -m "Add retry mechanism for flaky network step"
git push
```

**Checkpoint:** Run the workflow multiple times. Even though the step fails about one in three times, the retry mechanism catches it and re-runs the step. The overall workflow should succeed much more often. The probability of all 3 attempts failing is `(1/3)^3 = 1/27`, so about 96% of runs should succeed.

### Step 4: Add Timeout Management

In the previous step, you added `timeout_minutes: 2` to the retry step. Now add timeout management to the job level as well. This prevents a hung step from blocking the runner indefinitely.

Update the workflow to add a job-level timeout and a diagnostic step that runs on failure:

```yaml
name: CI
on: [push, workflow_dispatch]
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
        timeout-minutes: 5
      - name: Flaky network call with retry
        uses: actions/retry-step@v1
        with:
          max_attempts: 3
          retry_on: exit
          command: |
            if [ $((RANDOM % 3)) -eq 0 ]; then
              echo "Simulating network timeout..."
              sleep 30
              exit 1
            else
              echo "Network call succeeded"
            fi
          timeout_minutes: 2
      - run: npm test
        timeout-minutes: 3
      - run: echo "Build complete"
        timeout-minutes: 1

  diagnostics:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ failure() }}
    steps:
      - run: echo "The build job failed. Collecting diagnostics..."
      - uses: actions/upload-artifact@v4
        with:
          name: diagnostics-${{ github.run_id }}
          path: /tmp/*.log
```

Push this change:

```bash
git add -A
git commit -m "Add timeout management and diagnostics job"
git push
```

**Checkpoint:** Trigger a workflow run. If the `build` job fails, the `diagnostics` job should run automatically (because of `if: ${{ failure() }}`). If the build succeeds, the diagnostics job is skipped.

### Step 5: Add Notifications with Failure Categorization

Now add a notification step that sends a Slack message when the workflow fails. The message should include the run URL and a categorization of the failure.

Add a Slack notification job to your workflow:

```yaml
  notify:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ failure() }}
    steps:
      - name: Send Slack notification
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {
              "text": "Pipeline Failure: ${{ github.workflow }} run ${{ github.run_id }}",
              "blocks": [
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "Workflow *${{ github.workflow }}* failed in *${{ github.repository }}*"
                  }
                },
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "Run ID: ${{ github.run_id }}\nBranch: ${{ github.ref_name }}\nCommit: ${{ github.sha }}"
                  }
                },
                {
                  "type": "actions",
                  "elements": [
                    {
                      "type": "button",
                      "text": { "type": "plain_text", "text": "View Run" },
                      "url": "${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
                    }
                  ]
                }
              ]
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}

  always-upload-logs:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ always() }}
    steps:
      - name: Upload all logs
        uses: actions/upload-artifact@v4
        with:
          name: full-logs-${{ github.run_id }}
          path: /var/log/*.log
```

Push this change:

```bash
git add -A
git commit -m "Add Slack notification and log upload on always"
git push
```

**Checkpoint:** If you set up a Slack webhook URL as a secret named `SLACK_WEBHOOK_URL`, you should receive a Slack message when the workflow fails. If you did not set up Slack, the step will fail silently (which is fine for this exercise). The `always-upload-logs` job should run regardless of whether the build passes or fails.

### Step 6: Add `workflow_dispatch` with Inputs for Manual Re-run

Add a `workflow_dispatch` trigger with custom inputs that let you control the behavior of the workflow when you re-run it manually. This is useful for debugging — you can enable debug logging, skip certain steps, or target specific jobs.

Update the `on` section of your workflow:

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:
    inputs:
      debug:
        description: "Enable debug logging"
        type: boolean
        default: false
      skip-tests:
        description: "Skip the test step"
        type: boolean
        default: false
      target-job:
        description: "Run only specific job (build, diagnostics, notify)"
        type: choice
        options:
          - all
          - build
          - diagnostics
          - notify
```

Then use these inputs in your steps:

```yaml
      - name: Debug logging
        if: ${{ inputs.debug == true }}
        run: echo "DEBUG=true" >> $GITHUB_ENV

      - name: Run tests
        if: ${{ inputs.skip-tests != true }}
        run: npm test
```

Push this change:

```bash
git add -A
git commit -m "Add workflow_dispatch inputs for manual control"
git push
```

**Checkpoint:** Go to the Actions tab, click on your workflow, and click "Run workflow". You should see the input fields: `debug` (checkbox), `skip-tests` (checkbox), and `target-job` (dropdown). Try running with different combinations.

### Step 7: Implement `continue-on-error` for Non-Critical Steps

Some steps in your pipeline are not critical — a code lint warning, a code coverage check, a performance benchmark. If these fail, the pipeline should log the failure and continue, not block the entire build.

Add a non-critical lint step with `continue-on-error`:

```yaml
      - name: Lint check (non-critical)
        continue-on-error: true
        run: |
          echo "Running linter..."
          if [ $((RANDOM % 2)) -eq 0 ]; then
            echo "Lint found 3 warnings"
            exit 1
          else
            echo "Lint passed"
          fi
```

Push this change:

```bash
git add -A
git commit -m "Add non-critical lint step with continue-on-error"
git push
```

**Checkpoint:** Run the workflow multiple times. When the lint step fails, the workflow should continue to the next steps. The overall build should still succeed even if the lint step fails.

### Step 8: Final Workflow — Full Reliability

Here is the complete workflow with all reliability features integrated. Compare this with your starting point from Step 1.

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:
    inputs:
      debug:
        description: "Enable debug logging"
        type: boolean
        default: false
      skip-tests:
        description: "Skip the test step"
        type: boolean
        default: false
      target-job:
        description: "Run only specific job"
        type: choice
        options:
          - all
          - build
          - diagnostics
          - notify
        default: all

jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci
        timeout-minutes: 5

      - name: Lint check (non-critical)
        continue-on-error: true
        run: |
          echo "Running linter..."
          if [ $((RANDOM % 2)) -eq 0 ]; then
            echo "Lint found 3 warnings"
            exit 1
          else
            echo "Lint passed"
          fi

      - name: Flaky network call with retry
        uses: actions/retry-step@v1
        with:
          max_attempts: 3
          retry_on: exit
          command: |
            if [ $((RANDOM % 3)) -eq 0 ]; then
              echo "Simulating network timeout..."
              sleep 30
              exit 1
            else
              echo "Network call succeeded"
            fi
          timeout_minutes: 2

      - name: Run tests
        if: ${{ inputs.skip-tests != true }}
        run: npm test
        timeout-minutes: 3

      - run: echo "Build complete"
        timeout-minutes: 1

  diagnostics:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ failure() }}
    steps:
      - run: echo "Build failed. Collecting diagnostics..."
      - run: df -h
      - run: free -m
      - run: echo "Runner: ${{ runner.name }}"
      - uses: actions/upload-artifact@v4
        with:
          name: diagnostics-${{ github.run_id }}
          path: /tmp/*.log

  notify:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ failure() }}
    steps:
      - name: Collect failure info
        run: |
          echo "Workflow: ${{ github.workflow }}"
          echo "Run ID: ${{ github.run_id }}"
          echo "Branch: ${{ github.ref_name }}"
          echo "Commit: ${{ github.sha }}"

  always-upload-logs:
    runs-on: ubuntu-latest
    needs: build
    if: ${{ always() }}
    steps:
      - uses: actions/upload-artifact@v4
        with:
          name: full-logs-${{ github.run_id }}
          path: /tmp/*.log
```

## Common Errors at This Stage

### Error 1: `actions/retry-step` not found

```
Error: Unable to resolve action `actions/retry-step@v1`, unable to find version `v1`
```

**Fix:** The `actions/retry-step` is a community action, not an official GitHub action. You may need to use a specific version or a different action. Check the Marketplace for the latest version. Alternatively, implement retry logic manually with a shell loop:

```yaml
- name: Flaky step with manual retry
  run: |
    for i in {1..3}; do
      echo "Attempt $i of 3"
      if <your_command>; then
        echo "Success on attempt $i"
        exit 0
      fi
      sleep 5
    done
    echo "All 3 attempts failed"
    exit 1
```

### Error 2: `failure()` condition not triggering

```
The diagnostics job is skipped even though the build job failed.
```

**Fix:** The `if: ${{ failure() }}` condition checks the status of the job it depends on (via `needs`). Make sure you have `needs: build` in the diagnostics job. Also, `failure()` only triggers when at least one dependency has failed. If all dependencies are skipped, `failure()` does not trigger.

### Error 3: Workflow times out despite retries

```
The job was cancelled because it exceeded 10 minutes.
```

**Fix:** The `timeout-minutes` at the job level caps the total time for all steps combined. If your retry step takes 3 attempts x 2 minutes each = 6 minutes, plus other steps, you might exceed the job limit. Calculate the worst-case total time and set `timeout-minutes` accordingly. Alternatively, reduce `timeout_minutes` on individual retry steps.

### Error 4: Secrets not available in forked PRs

```
Error: slackapi/slack-github-action@v1 — secret SLACK_WEBHOOK_URL not found
```

**Fix:** By default, GitHub does not pass secrets to workflows triggered from forked repositories. This is a security feature. If you need notifications for forked PRs, consider using a different approach, such as using `pull_request_target` (with caution) or a third-party service like `pull_request` comment checks.

### Error 5: `continue-on-error` step still causes the job to fail

```
Job failed because a step with continue-on-error: true failed
```

**Fix:** This happens when a subsequent step also fails. `continue-on-error` only prevents the current step from failing the job. If any other step fails, the job still fails. Check whether another step is failing after the `continue-on-error` step.

## What You Just Learned

- You created a baseline unreliable workflow and observed how transient failures cause random pipeline failures.
- You added the `actions/retry-step` action to retry flaky steps up to 3 times, increasing reliability from 66% to 96%.
- You configured `timeout-minutes` at both the job and step level to prevent hung runs from blocking the runner queue.
- You implemented a diagnostics job that runs only on failure using `if: ${{ failure() }}` and uploads logs using `actions/upload-artifact`.
- You added a notification job that sends Slack messages when the pipeline fails, with full context about the run.
- You used `workflow_dispatch` with input parameters to control debug logging, skip tests, and target specific jobs for manual re-runs.
- You applied `continue-on-error` to non-critical steps so that lint warnings and other soft failures do not block the pipeline.
- You built a complete reliability-aware workflow that distinguishes between transient failures, code failures, and infrastructure failures.
