Your team has been fighting the same CI failure for three weeks. A test flakes out on one runner, the network drops mid-dependency install, and a perfectly good build gets a red X because SpotVM 47 had a corrupt cache. Nobody knows whether the failure is real or just noise, so every red build triggers a Slack thread, three people stop what they are doing to investigate, and fifteen minutes later someone says "re-run it" and it passes. The pipeline is not trusted. When the pipeline is not trusted, nobody trusts the green builds either, and the entire CI/CD system becomes a rubber stamp that everyone ignores.

## The Concept

Pipeline reliability is the practice of making your CI/CD workflows resilient to transient failures so that every red build signals a real problem, not infrastructure noise. Think of it like a commercial airline pre-flight checklist: the plane (your workflow) runs through the same checks every time, and if a sensor glitches, the system retries before grounding the plane. You do not cancel a flight because a gust of wind bumped a reading. Similarly, you should not fail a build because npm registry timed out for 200 milliseconds.

## Command / Syntax Reference

| YAML Key / Concept | Purpose | Example |
|---|---|---|
| `timeout-minutes` | Caps how long a job or step can run before GitHub kills it. Prevents hung runners from blocking the queue. | `timeout-minutes: 10` |
| `continue-on-error` | Marks a step as non-critical. The workflow continues even if this step fails. | `continue-on-error: true` |
| `if: ${{ failure() }}` | Runs a step only when a previous step in the same job has failed. Used for cleanup, diagnostics, or notifications. | `if: ${{ failure() }}` |
| `if: ${{ always() }}` | Runs a step regardless of pass or fail. Useful for uploading logs or artifacts on any outcome. | `if: ${{ always() }}` |
| `if: ${{ cancelled() }}` | Runs a step only when the workflow was cancelled manually. | `if: ${{ cancelled() }}` |
| `actions/retry-step` | A community action that retries a step up to N times with a delay between attempts. | `uses: actions/retry-step@v1` with `max_attempts: 3` |
| `strategy.fail-fast` | When true, cancels all in-progress jobs in a matrix if any one fails. Set to false to let other matrix variants finish. | `strategy: fail-fast: false` |
| `workflow_dispatch` | Allows manual re-triggering of a workflow with optional input parameters for re-running specific jobs. | `on: workflow_dispatch: inputs:` |
| `github.run_attempt` | A context variable that increments each time a workflow is re-run. Useful for conditional retry logic. | `if: ${{ github.run_attempt < 3 }}` |
| `actions/upload-artifact` / `actions/download-artifact` | Persists logs, test reports, and build outputs across jobs for post-mortem analysis. | `uses: actions/upload-artifact@v4` |

## How It Actually Works — Step by Step

### Step 1: A Job Starts and the Runner Picks It Up

GitHub provisions a runner from its pool. The runner clones your repository, sets up the workspace, and begins executing the first step. At this point, anything can go wrong: the runner might be slow, the network might be congested, or the Docker layer cache might be stale.

### Step 3: The Retry Mechanism Kicks In

If the failure is transient, the retry mechanism activates. Using `actions/retry-step` or a custom retry loop, the step re-runs with exponential backoff. Each attempt increments `github.run_attempt`, which the workflow can check to decide whether to keep trying.

```yaml
- name: Install dependencies
  uses: actions/retry-step@v1
  with:
    max_attempts: 3
    retry_on: timeout
    command: npm ci
    timeout_minutes: 5
```

### Step 4: The Final Outcome Is Decided

After all retries are exhausted, the pipeline decides the final outcome. If the step still fails and it is an infrastructure issue, the pipeline notifies the operations team and continues. If it is a code issue, the pipeline fails the job and notifies the development team. If a step is marked with `continue-on-error: true`, the pipeline logs the failure and moves on regardless.

### Step 5: Notification and Log Collection

Regardless of the outcome, the pipeline collects logs and artifacts. The `if: ${{ always() }}` condition ensures that log uploads run even when previous steps failed. Notifications are sent to the appropriate channel based on the failure category.

```yaml
- name: Upload test results
  if: ${{ always() }}
  uses: actions/upload-artifact@v4
  with:
    name: test-results-${{ github.run_id }}
    path: test-results/

- name: Notify on failure
  if: ${{ failure() }}
  uses: slackapi/slack-github-action@v1
  with:
    payload: |
      { "text": "Pipeline failed: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}" }
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

### Step 6: Manual Re-run with `workflow_dispatch`

When a developer investigates a failure and fixes the root cause, they can re-run the workflow manually using `workflow_dispatch`. This is also useful for re-running failed jobs without re-running successful ones, which saves time and runner minutes.

```yaml
on:
  workflow_dispatch:
    inputs:
      run_id:
        description: "Run ID to re-run"
        required: false
      job_filter:
        description: "Only re-run specific jobs (comma separated)"
        required: false
```

## When to Use This / When Not To

### Use Pipeline Reliability Techniques When

- Your pipeline fails intermittently and you cannot reproduce the failure locally.
- You have a large matrix build where one variant failing should not cancel the rest.
- You need to distinguish between "the build is broken" and "the runner is having a bad day."
- Your team spends more time investigating CI failures than fixing actual bugs.
- You have external dependencies (npm, PyPI, Docker Hub) that occasionally timeout.
- You want to collect logs and artifacts even from failed builds for post-mortem analysis.

### Do Not Use Pipeline Reliability Techniques When

- Every failure is a real code failure. Retrying will only mask the problem and waste runner minutes.
- You have not yet investigated the root cause of flaky tests. Retrying a flaky test hides the flakiness.
- You are using `continue-on-error` as a default for all steps. This turns off all failure signaling.
- Your infrastructure is consistently unreliable (runners crashing every build). Fix the infrastructure first, then add retries.
- You are on a free GitHub plan with limited runner minutes. Retries consume extra minutes.

### Failure Category Reference

| Category | Examples | Appropriate Response |
|---|---|---|
| Transient | Network timeout, npm registry 503, DNS failure, Docker pull timeout, cache corruption | Retry up to 3 times with backoff |
| Infrastructure | Runner offline, disk full, Docker daemon crashed, GitHub Actions outage | Notify ops, do not retry |
| Code | Compilation error, test assertion failure, lint error, missing module | Fail immediately, notify dev |
| Environment | Missing secret, wrong Node version, incorrect env var, expired token | Fail immediately, notify dev |
| Test Flake | Non-deterministic test, race condition, timing-dependent test | Mark as `continue-on-error`, file bug, fix later |
