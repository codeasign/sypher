Your workflow passes every time you run it locally. You push, the runner picks it up, and five seconds later you see a red X and a cryptic error: "Error: Process completed with exit code 1." No stack trace, no line number, no hint. You cannot SSH into the runner because it is ephemeral — it is created, runs your job, and disappears in under a minute. The error says "permission denied" but you have no idea which file, which command, or which step caused it. You try adding `echo` statements and pushing again, but each push takes two minutes, and you have made five pushes already. You are debugging blind.

This scenario plays out every day. A workflow that runs on your macOS laptop fails on the Ubuntu runner because a dependency is installed globally on your machine but not in the runner's clean environment. Or a file path works on Windows but the backslash escaping breaks in the YAML parser. Or a secret is spelled correctly in your `.env` file but missing from the repository's secrets. Or a Node.js version works fine in your local `nvm` but is not available in the runner's tool cache. You cannot replicate the runner environment locally — the runner is a clean Ubuntu VM with a specific set of pre-installed tools (Node, Python, Docker, Git, and about 200 other packages at specific versions), a specific directory layout (`/home/runner/work/`), and a specific set of environment variables (`CI=true`, `GITHUB_*`, `RUNNER_*`, `ACTIONS_*`) that GitHub sets automatically. The only way to debug is to bring the runner's internals to the surface.

The three most common failure patterns that require debugging are:

1. **Environment mismatch** — A tool or library works on your machine because it is installed globally, but the runner does not have it. For example, you use `ffmpeg` in a script but the Ubuntu runner does not include `ffmpeg` by default. Or you assume `python3` is on the PATH, but the runner only has Python available through `actions/setup-python`.

2. **Permission issues** — The runner runs as a non-root user with limited filesystem access. If your workflow writes to `/usr/` or `/opt/` without `sudo`, it fails. If it tries to access another repository's files or secrets, the `GITHUB_TOKEN` permissions may not allow it.

3. **Context evaluation errors** — An expression like `${{ secrets.DEPLOY_KEY }}` evaluates to an empty string because the secret does not exist or is misspelled. The workflow does not fail at this point — it runs with an empty value, which causes a confusing failure later.

## The Concept

Debugging GitHub Actions Workflows means using a set of diagnostic tools — debug logging, step inspection, live terminal sessions, workflow validation, and local runner simulation — to find and fix failures that you cannot reproduce on your own machine. GitHub Actions runners are ephemeral and environment-specific, so a workflow that works on your macOS laptop may fail on the Ubuntu runner because of a different PATH, a missing tool, a stricter file permission, or an environment variable that only exists on the runner.

The fundamental challenge is that you are writing code that runs on someone else's machine. You define the steps, but the runner controls the environment. The runner decides which shell to use (`bash` on Linux and macOS, `pwsh` on Windows), which user to run as, which directories are available, which tools are pre-installed, and which environment variables are set automatically. When your workflow fails, the error message is often the only clue you get — and it is usually too vague to identify the root cause.

The debugging toolkit includes five categories of tools:

- **Verbose logging** — Enable `ACTIONS_STEP_DEBUG` and `ACTIONS_RUNNER_DEBUG` to see every command, expansion, and environment variable the runner evaluates. These are environment variables and secrets that tell the runner to emit debug-level log lines. Without them, the runner only shows stdout and stderr from your steps. With them, you see the runner's internal diagnostics.

- **Live SSH access** — Inject a `tmate` action into your workflow to pause the runner and let you SSH in to inspect the filesystem, processes, and environment in real time. This is the closest you can get to "logging into the server" — you run `ls`, `cat`, `env`, `which`, and any other command you would run on a production machine.

- **Local simulation** — Run `nektos/act` to execute your workflow on your own machine, iterating in seconds instead of minutes. `act` uses Docker containers that mimic the GitHub runner's operating system and toolset. It is not a perfect replica (the images are maintained by the community, not GitHub), but it is fast enough for rapid iteration.

- **Conditional execution** — Use `continue-on-error`, `if: always()`, and `if: failure()` to isolate which step fails and to run cleanup or diagnostic steps regardless of the failure. These YAML keywords let you control the flow of the job during debugging, allowing you to skip past a failure to see what happens downstream.

- **Context inspection** — Print the `github`, `env`, and `runner` contexts with `toJSON()` to see exactly what the runner sees. The `github` context contains the full webhook payload — the branch, commit, PR number, sender, repository, and more. The `env` context shows all environment variables available to the step. The `runner` context tells you the OS, architecture, and debug mode status.

## Command / Syntax Reference

### Debug Logging Configuration

| Key | Description | Example |
|-----|-------------|---------|
| `ACTIONS_RUNNER_DEBUG=true` | Set as a repository secret or environment variable to enable runner-level debug logging. Produces extra log lines showing how the runner processes the workflow, downloads the job, and sets up the workspace. | `ACTIONS_RUNNER_DEBUG: true` in workflow `env` block |
| `ACTIONS_STEP_DEBUG=true` | Set as a repository secret or environment variable to enable step-level debug logging. Each step logs every command it runs, every expression evaluation, and the expanded values of all variables. | `ACTIONS_STEP_DEBUG: true` in workflow `env` block |
| `ACTIONS_STEP_DEBUG` as repo secret | Add `ACTIONS_STEP_DEBUG` as a repository secret with value `true` to enable debug logging for all workflows in the repository without modifying any workflow files. Toggle it on and off by adding or deleting the secret. | Settings > Secrets and variables > Actions > New repository secret |

### Interactive Debugging

| Key | Description | Example |
|-----|-------------|---------|
| `mxschmitt/action-tmate` | GitHub Action that installs and starts a tmate SSH session on the runner. You connect via the tmate URL printed in the logs, inspect the runner interactively, and disconnect when done. The runner pauses until you disconnect. | `- uses: mxschmitt/action-tmate@v3` |
| `nektos/act` | CLI tool that runs GitHub Actions workflows locally using Docker containers. Useful for fast iteration without pushing to GitHub. Supports `-j` for job selection, `-W` for workflow file, `--secret-file` for secrets, and `--env-file` for environment variables. | `act -j build -W .github/workflows/ci.yml` |
| `gh actions` commands | GitHub CLI commands for inspecting, re-running, and managing workflow runs. `gh run list` lists recent runs, `gh run view` shows details, `gh run view --log` streams logs, `gh run rerun` re-runs a failed job. | `gh run view --log --job=build` |

### Conditional Execution

| Key | Description | Example |
|-----|-------------|---------|
| `workflow_dispatch` with debug inputs | Add a `workflow_dispatch` trigger with an input field for debug mode. The workflow checks the input and conditionally enables debug logging, extra diagnostics, or verbose output. | `if: ${{ github.event.inputs.debug == 'true' }}` |
| `if: always()` | Run a step regardless of whether previous steps succeeded or failed. Useful for cleanup or diagnostic steps that must run after a failure. Overrides the default failure behavior where the job stops at the first failed step. | `if: always()` |
| `if: failure()` | Run a step only when a previous step in the job has failed. Returns `true` when any step in the job has exited with a non-zero code. Useful for notification or log-upload steps that should only run on failure. | `if: failure()` |
| `continue-on-error: true` | Mark a step as non-blocking. The step is allowed to fail without failing the entire job. The job status reflects the failure but continues to subsequent steps. The step icon shows a warning instead of an error. | `continue-on-error: true` |

### Context and Inspection

| Key | Description | Example |
|-----|-------------|---------|
| `github.event` context | The full webhook event payload that triggered the workflow. Contains information about the branch, commit, PR, sender, repository, and more. The structure differs depending on the trigger event. | `${{ toJSON(github.event) }}` |
| `runner.os` and `runner.debug` | `runner.os` returns the operating system of the runner (Linux, Windows, macOS). `runner.debug` returns `true` when `ACTIONS_STEP_DEBUG` or `ACTIONS_RUNNER_DEBUG` is enabled. Both are useful for conditional debugging steps. | `${{ runner.os }}` and `${{ runner.debug }}` |
| `toJSON()` | Converts any context object to a JSON string so you can print it in a log for inspection. Works with `github`, `env`, `secrets`, `vars`, `runner`, and `strategy` contexts. | `${{ toJSON(github) }}` |
| Workflow visualization graph | GitHub displays a visual graph of workflow runs in the Actions tab showing job status, duration, and dependencies. Each job is a node, and arrows show the `needs` relationships. A failed job is highlighted in red. | Actions tab > workflow run > Graph view |

## How It Actually Works — Step by Step

### Step 1: Workflow YAML Parsing and Runner Assignment

When you push to a branch that triggers a workflow, GitHub parses the YAML file, validates its structure, resolves expression syntax (`${{ }}`), and assigns the job to an available runner that matches the `runs-on` label. If the YAML has a syntax error, the workflow fails at this stage before any step runs — you see the error in the Actions tab immediately.

The parsing step is the first place where debugging begins. If your workflow file has a missing colon, an extra space, or an improperly indented line, the parser rejects it and you see a red error before any job starts. The error message usually tells you the line number, which makes YAML syntax errors the easiest failures to fix.

### Step 2: Runner Agent Initialization

The runner agent (a small application that runs on the compute host) receives the job specification, creates a working directory, checks out the repository (if you use `actions/checkout`), sets up environment variables from the workflow, secrets, and contexts, and prepares the step execution environment. This is where `ACTIONS_RUNNER_DEBUG` produces its first log lines — you see the runner downloading the job, setting up the workspace, and resolving secrets.

During initialization, the runner performs several critical operations:

- It creates a unique working directory under `_work/` on the runner's filesystem, named after the repository and the run ID
- It resolves all `${{ }}` expressions in the workflow, including `secrets.*`, `github.*`, `env.*`, and `vars.*`
- It populates the `GITHUB_*` environment variables: `GITHUB_WORKSPACE`, `GITHUB_SHA`, `GITHUB_REF`, `GITHUB_ACTION`, `GITHUB_ACTOR`, `GITHUB_EVENT_NAME`, and many more
- It creates the `$RUNNER_TEMP` directory for temporary files
- It sets up the `PATH` with the pre-installed tool locations

If `ACTIONS_RUNNER_DEBUG` is enabled, you see each of these steps logged in the runner output. For example, you see lines like:

```
##[debug]Evaluating: secrets.MY_SECRET
##[debug]Evaluating: github.ref_name
##[debug]Setting env 'GITHUB_WORKSPACE' to '/home/runner/work/my-repo/my-repo'
```

This is invaluable when a secret is missing or a context expression returns an unexpected value. Without debug logging, you only see the final result — an empty string where a value should be.

### Step 3: Job Execution with Step Isolation

Each step in the job runs in its own process with its own working directory. The runner executes steps sequentially, captures stdout and stderr for each step, and streams the output to GitHub in real time. If a step fails (exit code != 0), the runner stops the job and marks subsequent steps as skipped — unless `continue-on-error: true` is set, in which case the job continues.

Step isolation means that environment variables set in one step do not persist to the next step unless you use `$GITHUB_ENV` or `$GITHUB_OUTPUT`. Similarly, the working directory resets to `$GITHUB_WORKSPACE` for each step. This is a common source of confusion: you `cd` into a directory in one step, and the next step is back at the workspace root.

The runner also enforces a timeout on each step. The default timeout is 360 minutes (6 hours) for each job, but individual steps can time out sooner if the `timeout-minutes` key is set. When a step times out, the runner kills its process and marks the step as failed with a timeout error. This is distinct from an exit code failure — a timeout produces a different error message and is often caused by an infinite loop, a hung network request, or a test suite that takes longer than expected.

When `ACTIONS_STEP_DEBUG` is enabled, the runner logs the exact command string before executing it, including all expanded variables. This is the single most useful debugging feature because it shows you exactly what the runner sees:

```
##[debug]/usr/bin/bash --noprofile --norc -e -o pipefail /home/runner/work/_temp/01234567-89ab-cdef-0123-456789abcdef.sh
```

The command is written to a temporary script file and executed. The `-e` flag means the shell exits on the first error (which is why `exit 1` stops the job). The `-o pipefail` flag means a pipeline fails if any command in the pipeline fails. If your step uses pipes, this flag can cause a failure that you did not expect — for example, `echo "ok" | grep "notfound"` would fail even though the `echo` succeeded, because `grep` returns exit code 1 when it finds no match.

### Step 4: Log Streaming to GitHub

As each step runs, the runner streams its stdout and stderr to GitHub's log ingestion service. You see the logs in real time in the Actions tab. When `ACTIONS_STEP_DEBUG` is enabled, the runner also emits debug-level log lines that show every expression evaluation, every variable expansion, and every command the step executes. These debug lines are prefixed with `##[debug]` in the raw log output.

The log streaming architecture is designed for low latency. Each line of output is sent to GitHub as a separate event with a timestamp. The Actions tab groups these lines by step and displays them in order. If you download the raw log archive, you get a single file per job with all the debug lines, timestamps, and exit codes preserved.

The `##[debug]` prefix is important: when you view logs in the Actions UI, debug lines are collapsed by default. You have to expand the "Debug" section within each step to see them. This keeps the normal log view clean while making the full detail available when you need it. In the raw log download, debug lines are inline with regular output, so you can `grep` for `##[debug]` to extract only the diagnostic information.

### Step 5: Diagnostic Data Collection

After the job completes (or fails), you can download the raw logs, inspect the workflow visualization graph, re-run failed jobs with debug logging enabled, or trigger a new run with `workflow_dispatch` inputs that enable extra diagnostics. The `gh run view --log` command lets you search and filter logs from the terminal. For persistent failures, you can add a tmate step to the workflow, push, and SSH into the live runner to investigate the environment interactively.

The workflow visualization graph is particularly useful for matrix builds and multi-job workflows. Each job is a node in the graph, and the `needs` relationships are shown as arrows. A failed job is highlighted in red, and you can click on any job to see its step-by-step output. The graph view also shows the duration of each job, which helps identify performance issues alongside correctness issues.

The `gh` CLI gives you terminal-based access to the same data. This is useful when you want to grep through logs from multiple runs, or when you want to download logs for analysis in a script:

```bash
# List recent runs with their IDs and statuses
gh run list --limit 5 --json databaseId,status,conclusion,headBranch

# Download logs for a specific run as a zip file
gh run download 1234567890

# View logs and filter for error patterns
gh run view 1234567890 --log | grep -E "error|Error|FAIL|exit code"
```

The `gh run download` command downloads the raw log archive as a zip file, which you can then extract and search locally. This is useful when you want to share logs with a teammate, archive them for later analysis, or process them with a script.

```bash
# List recent runs
gh run list --limit 5

# View logs for a specific run and job
gh run view 1234567890 --log --job build | grep "error"

# Re-run a failed job with debug logging (if you already set the secret)
gh run rerun 1234567890 --failed
```

### Step 6: Fix and Iterate

Once you identify the root cause — a missing environment variable, a wrong file path, a permission issue, a tool that is not installed on the runner — you edit the workflow file, commit, and push. The new run uses the same debugging tools you already set up, so you can verify the fix immediately. When the workflow passes, you remove the tmate step and disable debug logging (or keep debug logging as a secret so you can toggle it on demand without modifying the workflow file).

The iteration cycle follows this pattern:

1. Push triggers a workflow run (2 minutes)
2. The run fails with a cryptic error
3. You enable debug logging (via secret or environment variable)
4. You re-run the failed job (30 seconds to start)
5. You inspect the debug logs and identify the issue
6. You fix the workflow file, commit, and push
7. The new run passes (or fails with a different error — go back to step 2)

The goal is to minimize the time spent in steps 2-4. Using `act` locally eliminates step 1 entirely (no push needed). Using `ACTIONS_STEP_DEBUG` as a repository secret means you enable it once and it applies to all subsequent runs without modifying the workflow file. Using tmate eliminates guesswork when the failure is environment-specific.

## When to Use / When Not To

**Use debug mode (`ACTIONS_STEP_DEBUG=true`) for every new workflow you write.** It costs nothing and gives you full visibility into the runner's execution. Keep it as a repository secret so you can toggle it on and off without modifying workflow files. The debug logs are collapsed by default in the UI, so they do not add noise to normal viewing — you only see them when you expand the debug section. This is a set-and-forget debugging tool.

**Use tmate only for stubborn failures that you cannot reproduce with `act` or debug logs.** A tmate session is interactive — you poke around the runner, run commands, inspect files — but it leaves the runner running until you disconnect, which costs billable minutes. Use it as a last resort. A typical tmate session lasts 5-10 minutes, which is enough to run `env`, `ls`, `which`, and `cat` on a few files. If you need more time, you are probably not looking in the right places.

**Use `nektos/act` for fast iteration during development.** Running `act` locally takes seconds instead of the minutes it takes to push to GitHub and wait for a runner. Use it to validate your workflow logic before you push. Do not rely on `act` alone — it uses Docker containers that approximate the GitHub runner environment but do not match it exactly. The community-maintained images may lack tools that the real runner has, or may have different versions. Always push to verify the final workflow.

**Use `continue-on-error` and `if: always()` during debugging** to isolate failures. Let a step fail without blocking the rest of the job so you can see which steps pass and which fail. Remove these after debugging — a workflow that ignores failures is not a CI pipeline, it is a notification system. The `if: always()` flag is the exception: keep it on cleanup steps that should always run, even in production workflows.

**Do not push secrets or sensitive data into debug logs.** When `ACTIONS_STEP_DEBUG` is enabled, the runner logs every variable expansion, including the values of secrets. If you print a secret to the log, it is visible to anyone who can view the workflow run. Use `echo "***"` to mask secrets, or inspect secrets indirectly through their effect on the workflow rather than their raw value. GitHub automatically masks secrets in log output, but the masking only applies to the exact secret value — if you transform the secret (e.g., base64-encode it), the transformed value is not masked and leaks the secret.

**Use `if: failure()` for notification and reporting steps, not for debugging.** A step that sends a Slack or email notification when the workflow fails is a production pattern, not a debugging pattern. During debugging, you want to see the failure directly in the logs, not in a notification that you will ignore after the tenth run.

**Use the workflow visualization graph for multi-job workflows.** When a workflow has three or more jobs with complex `needs` dependencies, the graph view in the Actions tab shows you the execution order and which jobs depend on which. A failed job is highlighted in red, and the graph makes it obvious whether a downstream job failed because of a dependency or because of its own logic. Access the graph from the workflow run page by clicking the "Graph" tab.

**Do not commit debugging steps to the main branch.** The `tmate` step, the `continue-on-error` flags, and the extra `echo` statements are temporary tools. Commit them to a feature branch or a debugging branch, but remove them before merging to `main`. A workflow that has `continue-on-error: true` on every step is not a CI pipeline — it is a silent failure generator that teaches you to ignore red Xs.

**Use `workflow_dispatch` with debug inputs for production workflows that you need to debug occasionally.** Instead of editing the workflow file every time you need to debug, add a `workflow_dispatch` trigger with a boolean input called `debug`. When you trigger the workflow manually, you check the debug box, and the workflow enables extra logging and diagnostics conditionally. This approach keeps the workflow file clean while giving you on-demand debugging capability.
