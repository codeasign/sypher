You will create a deliberately broken GitHub Actions workflow, push it to a repository, and then use every debugging technique in the toolkit — step debug logging, local simulation with `act`, `continue-on-error`, `if: always()`, tmate SSH sessions, and context inspection — to find and fix each failure. By the end of this exercise, you will have a repeatable debugging process that you can apply to any workflow.

## Setup

Create a new repository on GitHub called `debugging-actions-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/debugging-actions-practice.git
cd debugging-actions-practice
```

Create the workflow directory:

```bash
mkdir -p .github/workflows
```

Create a deliberately broken workflow. Save this as `.github/workflows/broken.yml`:

```yaml
name: Broken Workflow

on: [push]

env:
  APP_ENV: production
  REQUIRED_TOKEN: ${{ secrets.MY_SECRET }}

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Print environment
        run: echo "Running in $APP_ENV mode"

      - name: Use a missing variable
        run: echo "The secret value is $REQUIRED_TOKEN"

      - name: Simulate a build
        run: |
          mkdir -p dist
          echo "build output" > dist/output.txt
          exit 1

      - name: Deploy (should never run if build fails)
        run: echo "Deploying to $APP_ENV..."

      - name: Cleanup
        run: echo "Cleaning up temporary files..."
```

This workflow has three deliberate problems:
1. Step "Simulate a build" always exits with code 1 (failure)
2. The "Deploy" step will be skipped when build fails, so you never see the deployment message
3. The "Cleanup" step is also skipped — cleanup should run regardless

Push the broken workflow:

```bash
git add -A
git commit -m "Add broken workflow"
git push origin main
```

Visit the Actions tab in your repository. You will see a failed run with a red X.

### Installing `nektos/act` for Local Simulation

You need `act` to run workflows locally without pushing. Install it for your operating system:

**macOS:**

```bash
brew install act
```

**Linux (Ubuntu/Debian):**

```bash
curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash
```

**Windows (using winget):**

```bash
winget install nektos.act
```

**Windows (using Chocolatey):**

```bash
choco install act-cli
```

**Windows (using Scoop):**

```bash
scoop install act
```

Verify the installation:

```bash
act --version
```

## Walkthrough

### Step 1: Observe the Failure

Open the Actions tab in your repository. Click on the failed run. You will see the `build` job highlighted in red. Click on it to expand the step list. The "Simulate a build" step shows `exit code 1`. The "Deploy" and "Cleanup" steps are marked as skipped — they never ran because the job stopped at the first failure.

This is the default behavior: a step that exits with a non-zero code stops the entire job. All subsequent steps are cancelled.

### Step 2: Enable Step Debug Logging with a Secret

Debug logging is the fastest way to see exactly what the runner sees. Add `ACTIONS_STEP_DEBUG` as a repository secret:

1. Go to your repository Settings > Secrets and variables > Actions
2. Click "New repository secret"
3. Name: `ACTIONS_STEP_DEBUG`
4. Value: `true`
5. Click "Add secret"

Now re-run the failed workflow:
1. Go to the Actions tab and click on the failed run
2. Click "Re-run jobs" > "Re-run failed jobs"

The workflow runs again, but this time every step logs its internal state. Click on any step to see the debug output — each command is prefixed with `##[debug]` showing the expanded environment variables, the working directory, and the exact command string that the runner executed.

The `ACTIONS_STEP_DEBUG` secret works for every workflow in the repository without modifying any workflow files. You can toggle it on and off by deleting or editing the secret.

### Step 3: Re-Run with Full Debug Logging

For even more detail, add `ACTIONS_RUNNER_DEBUG` alongside the step debug log. Add another repository secret:

1. Go to Settings > Secrets and variables > Actions
2. Click "New repository secret"
3. Name: `ACTIONS_RUNNER_DEBUG`
4. Value: `true`

Now trigger a new run by pushing an empty commit:

```bash
git commit --allow-empty -m "Trigger run with full debug logging"
git push origin main
```

When you view the logs, the runner-level debug lines show how the runner downloaded the job, set up the workspace, and resolved the `secrets.MY_SECRET` reference. Note that the secret value is masked in the logs — you will see `***` instead of the actual value. This is GitHub's secret masking in action.

### Step 4: Run the Workflow Locally with `act`

Pushing every experiment costs time. Use `act` to run the workflow on your machine. From the repository root:

```bash
act -j build -W .github/workflows/broken.yml
```

`act` pulls Docker images that simulate the GitHub runner environment, checks out the repository, and runs each step. You see the same output you would see on GitHub. The difference is that `act` runs in seconds, not minutes, and you can edit the workflow file and re-run without pushing.

When you run the broken workflow with `act`, you see the same failure — the "Simulate a build" step exits with code 1. Now edit the workflow file to fix just that step:

```yaml
      - name: Simulate a build
        run: |
          mkdir -p dist
          echo "build output" > dist/output.txt
          # exit 1   <-- remove this line
```

Run `act` again. The build step passes. You verified the fix locally in under 10 seconds.

### Step 5: Add `continue-on-error` to Isolate the Failing Step

Revert the fix so the workflow is broken again. Now add `continue-on-error: true` to the "Simulate a build" step:

```yaml
      - name: Simulate a build
        continue-on-error: true
        run: |
          mkdir -p dist
          echo "build output" > dist/output.txt
          exit 1
```

The `continue-on-error` flag tells the runner to allow this step to fail without stopping the job. The step shows a warning instead of an error, and the job continues to the next steps.

Push this change and watch the Actions tab. The "Simulate a build" step shows a yellow warning icon (failure allowed), and the "Deploy" and "Cleanup" steps now run. You can see the full flow and confirm that only the build simulation is broken — the rest of the workflow is structurally sound.

### Step 6: Add `if: always()` for Cleanup Steps

The "Cleanup" step should run whether the build succeeds or fails. Add `if: always()`:

```yaml
      - name: Cleanup
        if: always()
        run: echo "Cleaning up temporary files..."
```

The `if: always()` condition overrides the default skip-on-failure behavior. This step now runs after every step completes, regardless of the job status.

Remove `continue-on-error: true` from the build step so it fails hard again. Push and observe: the "Simulate a build" step fails, the "Deploy" step is skipped (it does not have `if: always()`), but the "Cleanup" step runs because you specifically asked it to run always.

### Step 7: SSH Into the Runner with tmate

For failures that you cannot reproduce locally — different operating system, different PATH, a tool that is installed differently on the runner — you need live access to the runner environment. The `mxschmitt/action-tmate` action starts a tmate SSH session on the runner.

Add the tmate step after the checkout step:

```yaml
      - uses: actions/checkout@v4

      - name: Debug with tmate
        uses: mxschmitt/action-tmate@v3
```

Push and go to the Actions tab. The workflow will pause at the tmate step. Open the step log — you will see a line like:

```
Connection established: ssh <random-hash>@nyc1.tmate.io
```

Copy the SSH address. Open a terminal on your machine and connect:

```bash
ssh <random-hash>@nyc1.tmate.io
```

You are now inside the GitHub runner container. You can run any command:

```bash
# Check the operating system
cat /etc/os-release

# Check the working directory
pwd && ls -la

# Check environment variables
env | sort

# Check which tools are available
which node python3 docker git

# Check the repository files
ls -la $GITHUB_WORKSPACE
```

When you are done investigating, disconnect:

```bash
exit
```

The tmate step completes and the workflow continues. Delete the tmate step from the workflow when you have found the issue — it adds billable time to every run.

### Step 8: Inspect the `github` Context and Fix the Workflow

The `github` context contains everything the runner knows about the event that triggered the workflow. Print it to see the full payload:

```yaml
      - name: Inspect github context
        run: echo '${{ toJSON(github) }}'
```

Push and look at the output. You will see the `github` object with fields like `repository`, `actor`, `ref`, `sha`, `event_name`, and more. This is particularly useful when debugging complex triggers — for example, when a `pull_request` target behaves differently than a `push` trigger.

Now fix the workflow completely. Restore the broken step to pass (remove `exit 1`) and remove the temporary debugging steps. The final fixed workflow should look like this:

```yaml
name: Fixed Workflow

on: [push]

env:
  APP_ENV: production
  REQUIRED_TOKEN: ${{ secrets.MY_SECRET }}

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Print environment
        run: echo "Running in $APP_ENV mode"

      - name: Simulate a build
        run: |
          mkdir -p dist
          echo "build output" > dist/output.txt

      - name: Deploy
        run: echo "Deploying to $APP_ENV..."

      - name: Cleanup
        if: always()
        run: echo "Cleaning up temporary files..."
```

Push the fixed workflow. All steps should pass with green checkmarks.

## Common Errors

### "Error: Process completed with exit code 1"

This is the most common error and the least informative. It means a command in your step returned a non-zero exit code. Any of these can cause it:

- A test suite failed (e.g., `jest`, `pytest`, `go test`)
- A linter found violations and returned exit code 1
- A shell command failed (e.g., `mkdir` on a path that already exists, `rm` on a file that does not exist)
- A script you called with `run:` threw an unhandled exception

**Fix:** Enable `ACTIONS_STEP_DEBUG` to see the exact command and its expanded arguments. Or add `continue-on-error: true` temporarily to let the job continue and inspect the output of subsequent steps. Use `set -x` in your shell command to print every command before execution.

### "Error: Resource not accessible by integration"

The runner tried to access a GitHub resource (a secret, an environment, a deployment, a package registry) that the `GITHUB_TOKEN` or the workflow's permissions do not allow. This typically happens when:

- Your workflow uses `secrets:` but the secret does not exist in the repository
- Your workflow accesses the GitHub API without sufficient `permissions:` in the workflow
- You are using `GITHUB_TOKEN` but the token does not have the required scope
- You are trying to write to a protected branch

**Fix:** Check the `permissions:` block at the top of your workflow. For most operations, this is sufficient:

```yaml
permissions:
  contents: read
  issues: write
  pull-requests: write
```

Also verify that the secret exists in Settings > Secrets and variables > Actions.

### "Secret not found"

The runner resolved a `${{ secrets.SOMETHING }}` expression but the secret does not exist in the repository, organization, or environment. The workflow does not fail — the expression evaluates to an empty string, which can cause subtle bugs. If `ACTIONS_STEP_DEBUG` is enabled, you will see the empty expansion in the debug log.

**Fix:** Verify the secret name is spelled correctly. Secrets in repository settings are case-sensitive. Use `toJSON(secrets)` to list available secret names (the values are masked). Add `${{ secrets.SOMETHING != '' }}` in a step to check whether the secret exists before using it.

### "docker: command not found" when using `act`

When you run `act` and get `docker: command not found`, it means Docker is not installed or the Docker daemon is not running. `act` requires Docker to spin up the container environment that simulates the GitHub runner.

**Fix (macOS):** Install Docker Desktop from docker.com and start it. Verify with `docker info`.

**Fix (Linux):** Install Docker Engine:

```bash
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io
sudo usermod -aG docker $USER
```

Log out and back in for the group change to take effect.

**Fix (Windows):** Install Docker Desktop from docker.com and start it. Verify with `docker info` in PowerShell.

After Docker is running, `act` should work without errors.

### "The `github` context is not available outside of `steps`"

The `github` context is not available in the `env` block at the workflow or job level — it is only available in step-level expressions. If you write:

```yaml
env:
  BRANCH_NAME: ${{ github.ref_name }}
```

You get an empty string at best or a cryptic error at worst. The same applies to the `runner`, `secrets`, and `vars` contexts.

**Fix:** Move the expression into the step that needs it:

```yaml
- name: Print branch
  run: echo "Branch is ${{ github.ref_name }}"
```

Or assign it in a step's output and reference that output in later steps.

## What You Just Learned

- **Secrets can control debug logging without touching workflow files.** Setting `ACTIONS_STEP_DEBUG` and `ACTIONS_RUNNER_DEBUG` as repository secrets enables verbose logging for every workflow in the repository. You toggle debugging on and off by adding or removing the secret, not by editing YAML files.

- **`continue-on-error` isolates failures.** Adding `continue-on-error: true` to a step lets it fail without stopping the job, so you can see which subsequent steps pass or fail. Use it during debugging and remove it when you are done.

- **`if: always()` ensures cleanup runs.** Steps that free resources, upload logs, or post notifications should use `if: always()` so they execute even when the job fails. Without it, these steps are skipped when any earlier step fails.

- **`nektos/act` gives you sub-second iteration.** Running a workflow locally with `act` takes seconds instead of minutes. Use it to validate workflow logic, test fixes, and iterate quickly. But verify the final workflow on GitHub — `act` uses Docker, not the actual runner environment.

- **tmate gives you a live terminal on the runner.** When a failure only happens on the GitHub runner and you cannot reproduce it locally, inject a tmate step to SSH into the runner and inspect the environment, filesystem, and processes interactively. Remove the tmate step after debugging to avoid burning billable minutes.

- **`toJSON()` exposes what the runner sees.** Printing `${{ toJSON(github) }}` or `${{ toJSON(env) }}` reveals the exact objects and values the runner uses. This is especially useful when debugging conditional expressions, event triggers, and context-dependent behaviors.
