## Tier 1 — Beginner

### Exercise 1: Create a simple reusable workflow with no inputs

**Task:** Create a reusable workflow file `.github/workflows/greeting.yml` that has `on: workflow_call` as its trigger and contains one job called `say-hello` that runs on `ubuntu-latest` and prints "Hello from reusable workflow" using a `run:` step.

<details>
<summary>Solution</summary>

```yaml
# .github/workflows/greeting.yml
name: Greeting
on: workflow_call

jobs:
  say-hello:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello from reusable workflow"
```

This is the minimum viable reusable workflow. It has no inputs, no secrets, no outputs. Any caller can invoke it with a single line: `uses: owner/repo/.github/workflows/greeting.yml@main`.
</details>

### Exercise 2: Call the reusable workflow from a caller workflow

**Task:** Create a caller workflow `.github/workflows/caller.yml` that is triggered `on: push` to `main` and calls the reusable greeting workflow from Exercise 1 using the `uses:` keyword.

<details>
<summary>Solution</summary>

```yaml
# .github/workflows/caller.yml
name: Caller
on:
  push:
    branches: [main]

jobs:
  greet:
    uses: your-username/your-repo/.github/workflows/greeting.yml@main
```

Replace `your-username/your-repo` with the actual owner and repository name. When you push to `main`, GitHub Actions will: (1) trigger the caller workflow, (2) find the `greet` job with `uses:`, (3) start a reusable workflow run, and (4) execute the `say-hello` job.
</details>

### Exercise 3: Create a composite action that runs two shell commands

**Task:** Create a composite action at `.github/actions/hello-action/action.yml` that runs two shell commands. The commands should print "Hello" and then print "World" on separate lines. Each step must specify `shell: bash`.

<details>
<summary>Solution</summary>

```yaml
# .github/actions/hello-action/action.yml
name: "Hello World Action"
description: "Prints Hello and World on separate lines"
runs:
  using: composite
  steps:
    - shell: bash
      run: echo "Hello"
    - shell: bash
      run: echo "World"
```

The composite action defines `runs.using: composite` and two steps. Each step has `shell: bash` — this is required because composite actions do not inherit a default shell from the workflow. The action is used as `uses: ./ .github/actions/hello-action` from the same repo or `uses: owner/repo/.github/actions/hello-action@ref` from another repo.
</details>

### Exercise 4: Pass a string input to a reusable workflow

**Task:** Create a reusable workflow that accepts an `input` called `name` of type `string` (required). The workflow should print `"Hello, <name>!"` where the placeholder is replaced with the value of the input. Then create a caller workflow that passes the name `"World"` as the input.

<details>
<summary>Solution</summary>

Reusable workflow `.github/workflows/greeting.yml`:

```yaml
# .github/workflows/greeting.yml
name: Personalized Greeting
on:
  workflow_call:
    inputs:
      name:
        required: true
        type: string

jobs:
  greet:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello, ${{ inputs.name }}!"
```

Caller workflow `.github/workflows/say-hello.yml`:

```yaml
# .github/workflows/say-hello.yml
name: Say Hello
on: workflow_dispatch

jobs:
  greet:
    uses: your-username/your-repo/.github/workflows/greeting.yml@main
    with:
      name: "World"
```

When the caller runs, it passes the `name` input to the reusable workflow. The reusable workflow receives it via `${{ inputs.name }}` and prints "Hello, World!". Inputs are always strings by default. The `type: string` declaration is optional but recommended for clarity.
</details>

## Tier 2 — Intermediate

### Exercise 5: Create a reusable workflow with multiple input types

**Task:** Create a reusable workflow that accepts three inputs: `app-name` (string, required), `debug-mode` (boolean, default `false`), and `retry-count` (number, default `3`). The workflow should print the values of all three inputs. Create a caller that passes `app-name: "my-app"`, `debug-mode: true`, and `retry-count: 5`.

<details>
<summary>Solution</summary>

Reusable workflow `.github/workflows/config-check.yml`:

```yaml
# .github/workflows/config-check.yml
name: Config Check
on:
  workflow_call:
    inputs:
      app-name:
        required: true
        type: string
      debug-mode:
        required: false
        type: boolean
        default: false
      retry-count:
        required: false
        type: number
        default: 3

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "App name: ${{ inputs.app-name }}"
          echo "Debug mode: ${{ inputs.debug-mode }}"
          echo "Retry count: ${{ inputs.retry-count }}"
```

Caller workflow:

```yaml
# .github/workflows/deploy-app.yml
name: Deploy App
on: workflow_dispatch

jobs:
  config-check:
    uses: your-username/your-repo/.github/workflows/config-check.yml@main
    with:
      app-name: "my-app"
      debug-mode: true
      retry-count: 5
```

GitHub Actions supports three input types for reusable workflows: `string`, `boolean`, and `number`. Booleans and numbers are converted to their native types inside the workflow — you can use them in `if:` conditions and arithmetic. The output will show the values: `my-app`, `true`, and `5`.
</details>

### Exercise 6: Create a composite action with inputs

**Task:** Create a composite action at `.github/actions/greet-person/action.yml` that takes an input `name` (required, default "Friend") and `greeting` (required, default "Hello"). The action should print the greeting and name combined and also set a step output called `message` that contains the full greeting string. The composite action should have two steps: one to construct the message and one to display it.

<details>
<summary>Solution</summary>

```yaml
# .github/actions/greet-person/action.yml
name: "Greet Person"
description: "Prints a personalized greeting"
inputs:
  name:
    description: "Name of the person to greet"
    required: false
    default: "Friend"
  greeting:
    description: "Greeting type"
    required: false
    default: "Hello"
outputs:
  message:
    description: "The full greeting message"
    value: ${{ steps.construct.outputs.message }}
runs:
  using: composite
  steps:
    - id: construct
      shell: bash
      run: |
        FULL_MESSAGE="${{ inputs.greeting }}, ${{ inputs.name }}!"
        echo "message=$FULL_MESSAGE" >> $GITHUB_OUTPUT
    - shell: bash
      run: echo "${{ steps.construct.outputs.message }}"
```

Usage in a workflow:

```yaml
steps:
  - uses: ./.github/actions/greet-person
    with:
      name: "Alice"
      greeting: "Hi"
```

The composite action defines both `inputs` and `outputs`. The first step (`construct`) builds the message string and writes it to `$GITHUB_OUTPUT`. The second step reads the output via `${{ steps.construct.outputs.message }}` and prints it. The action also exposes the message as an action-level output via the `outputs.message.value` declaration.
</details>

### Exercise 7: Pass secrets using `secrets: inherit`

**Task:** Create a reusable workflow that expects a secret called `api-key` (required). The workflow should have one job that runs on `ubuntu-latest` and prints a masked version of the secret (the first 4 characters followed by asterisks). Create a caller that uses `secrets: inherit` to pass all secrets to the reusable workflow.

<details>
<summary>Solution</summary>

Reusable workflow `.github/workflows/secret-check.yml`:

```yaml
# .github/workflows/secret-check.yml
name: Secret Check
on:
  workflow_call:
    secrets:
      api-key:
        required: true

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - name: Check secret length
        run: |
          KEY_LENGTH=$(echo -n "${{ secrets.api-key }}" | wc -c)
          echo "API key length: $KEY_LENGTH characters"
      - name: Show masked secret
        run: |
          PREFIX=$(echo -n "${{ secrets.api-key }}" | head -c 4)
          echo "Secret starts with: ${PREFIX}****"
```

Caller workflow:

```yaml
# .github/workflows/run-check.yml
name: Run Secret Check
on: workflow_dispatch

jobs:
  check:
    uses: your-username/your-repo/.github/workflows/secret-check.yml@main
    secrets: inherit
```

The `secrets: inherit` keyword passes every secret the caller workflow has access to. This is convenient but should be used with caution — it gives the reusable workflow access to all caller secrets, including ones it may not need. For production, use explicit secret passing with `secrets: api-key: ${{ secrets.API_KEY }}` instead.
</details>

### Exercise 8: Use workflow_call outputs to pass data back to the caller

**Task:** Create a reusable workflow with one job that generates a random build ID and exposes it as a `workflow_call` output. Create a caller workflow that reads the build ID and prints it in a subsequent job.

<details>
<summary>Solution</summary>

Reusable workflow `.github/workflows/build-id-generator.yml`:

```yaml
# .github/workflows/build-id-generator.yml
name: Build ID Generator
on:
  workflow_call:
    outputs:
      build-id:
        description: "Generated build ID"
        value: ${{ jobs.generate.outputs.id }}

jobs:
  generate:
    runs-on: ubuntu-latest
    outputs:
      id: ${{ steps.create-id.outputs.id }}
    steps:
      - id: create-id
        run: |
          TIMESTAMP=$(date +%s)
          SHORT_SHA=${GITHUB_SHA:0:7}
          echo "id=build-${TIMESTAMP}-${SHORT_SHA}" >> $GITHUB_OUTPUT
```

Caller workflow:

```yaml
# .github/workflows/use-build-id.yml
name: Use Build ID
on: workflow_dispatch

jobs:
  generate-id:
    uses: your-username/your-repo/.github/workflows/build-id-generator.yml@main

  report:
    needs: generate-id
    runs-on: ubuntu-latest
    steps:
      - name: Print build ID
        run: echo "Build ID is ${{ needs.generate-id.outputs.build-id }}"
```

The output flows through two levels. The `generate` job defines a job-level output (`id`). The reusable workflow maps this to a `workflow_call` output (`build-id`). The caller accesses it via `needs.generate-id.outputs.build-id`. The `report` job will print something like `Build ID is build-1720123456-a1b2c3d`.
</details>

## Tier 3 — Advanced

### Exercise 9: Design a reusable deployment workflow with environment-specific configuration

**Task:** Design a reusable deployment workflow that deploys to `dev`, `staging`, or `prod` based on a caller-provided input called `environment` (string, required, limited to `dev`, `staging`, `prod` via `options`). Each environment requires different secrets: `dev` needs a `dev-token`, `staging` needs a `staging-token`, and `prod` needs a `prod-token`. The workflow should also have a manual approval step for the `prod` environment using a `reviewers` input (string). The workflow should print the deployment target and environment without actually deploying.

<details>
<summary>Solution</summary>

```yaml
# .github/workflows/deploy-environment.yml
name: Deploy to Environment
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
        # Note: the GitHub Actions UI does not support enum/options
        # validation for reusable workflow inputs. Document this
        # constraint for callers and validate inside the workflow.
      reviewers:
        required: false
        type: string
        default: ""
    secrets:
      dev-token:
        required: false
      staging-token:
        required: false
      prod-token:
        required: false

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - name: Validate environment
        run: |
          case "${{ inputs.environment }}" in
            dev) echo "Deploying to dev" ;;
            staging) echo "Deploying to staging" ;;
            prod) echo "Deploying to prod" ;;
            *)
              echo "Error: Invalid environment '${{ inputs.environment }}'"
              exit 1
              ;;
          esac

  deploy:
    needs: validate
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - uses: actions/checkout@v4
      - name: Select token based on environment
        run: |
          if [ "${{ inputs.environment }}" = "dev" ]; then
            echo "Using dev token"
            # Use: ${{ secrets.dev-token }}
          elif [ "${{ inputs.environment }}" = "staging" ]; then
            echo "Using staging token"
            # Use: ${{ secrets.staging-token }}
          elif [ "${{ inputs.environment }}" = "prod" ]; then
            echo "Using prod token"
            # Use: ${{ secrets.prod-token }}
          fi
      - name: Deploy
        run: echo "Deploying to ${{ inputs.environment }}"
```

Caller workflow:

```yaml
# .github/workflows/deploy-all.yml
name: Deploy All
on: workflow_dispatch

jobs:
  deploy-dev:
    uses: your-username/your-repo/.github/workflows/deploy-environment.yml@main
    with:
      environment: dev
    secrets:
      dev-token: ${{ secrets.DEV_TOKEN }}

  deploy-staging:
    needs: deploy-dev
    uses: your-username/your-repo/.github/workflows/deploy-environment.yml@main
    with:
      environment: staging
    secrets:
      staging-token: ${{ secrets.STAGING_TOKEN }}

  deploy-prod:
    needs: deploy-staging
    uses: your-username/your-repo/.github/workflows/deploy-environment.yml@main
    with:
      environment: prod
      reviewers: "team-leads"
    secrets:
      prod-token: ${{ secrets.PROD_TOKEN }}
```

This demonstrates a real-world deployment pattern. The reusable workflow is called three times — once per environment. Each call passes different secrets. The `environment:` field in the `deploy` job maps to a GitHub Environment, which supports approval gates for `prod`. The callers chain with `needs:` so that dev deploys first, then staging, then prod.
</details>

### Exercise 10: Create a matrix of reusable workflow calls

**Task:** Create a caller workflow that uses a matrix strategy to call the same reusable workflow with different parameters. The reusable workflow should accept `language` (string) and `version` (string) inputs and print `"Testing <language> version <version>"` where the placeholders are replaced by the input values. The matrix should test three languages (python, node, go) with two versions each (python: 3.11/3.12, node: 18/20, go: 1.21/1.22).

<details>
<summary>Solution</summary>

Reusable workflow `.github/workflows/test-language.yml`:

```yaml
# .github/workflows/test-language.yml
name: Test Language
on:
  workflow_call:
    inputs:
      language:
        required: true
        type: string
      version:
        required: true
        type: string

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Testing ${{ inputs.language }} version ${{ inputs.version }}"
```

Caller workflow:

```yaml
# .github/workflows/matrix-caller.yml
name: Matrix Language Tests
on: workflow_dispatch

jobs:
  test-languages:
    strategy:
      matrix:
        language: [python, node, go]
        version:
          - { language: python, versions: [3.11, 3.12] }
          - { language: node, versions: [18, 20] }
          - { language: go, versions: [1.21, 1.22] }
    uses: your-username/your-repo/.github/workflows/test-language.yml@main
    with:
      language: ${{ matrix.language }}
      version: ${{ matrix.version.versions[0] }}
```

Note: Reusable workflows with matrix strategies have a limitation — you cannot use `strategy.matrix` directly in a job that uses `uses:`. Instead, you must define the matrix in the caller and pass each combination as individual inputs. For a more dynamic approach, you can use a preparatory job to generate the matrix and then use `fromJson()`:

```yaml
name: Dynamic Matrix Caller
on: workflow_dispatch

jobs:
  generate:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - id: set-matrix
        run: |
          echo 'matrix={"include":[
            {"language":"python","version":"3.11"},
            {"language":"python","version":"3.12"},
            {"language":"node","version":"18"},
            {"language":"node","version":"20"},
            {"language":"go","version":"1.21"},
            {"language":"go","version":"1.22"}
          ]}' >> $GITHUB_OUTPUT

  test:
    needs: generate
    strategy:
      matrix: ${{ fromJson(needs.generate.outputs.matrix) }}
    uses: your-username/your-repo/.github/workflows/test-language.yml@main
    with:
      language: ${{ matrix.language }}
      version: ${{ matrix.version }}
```

This pattern generates a dynamic matrix in a preparatory job and passes each combination to the reusable workflow. Each matrix entry calls the reusable workflow once, resulting in 6 parallel workflow runs.
</details>

### Exercise 11: Build a composite action with nested actions and error handling

**Task:** Build a composite action that does the following: (1) checks out code, (2) runs a user-provided build command, (3) runs a user-provided test command, (4) cleans up temporary files regardless of whether the build or test step failed (using `if: always()` or `if: failure()`), and (5) uploads a build log even on failure. The action should accept `build-command` (string, required) and `test-command` (string, required) as inputs.

<details>
<summary>Solution</summary>

```yaml
# .github/actions/build-and-test/action.yml
name: "Build and Test"
description: "Builds and tests a project with cleanup on failure"
inputs:
  build-command:
    description: "Command to build the project"
    required: true
  test-command:
    description: "Command to test the project"
    required: true
  cleanup-command:
    description: "Command to clean up after build"
    required: false
    default: "rm -rf ./build ./dist"
runs:
  using: composite
  steps:
    - uses: actions/checkout@v4

    - id: build-log
      shell: bash
      run: echo "build_log=build-output.txt" >> $GITHUB_OUTPUT

    - name: Run build
      id: build
      shell: bash
      run: |
        echo "Starting build at $(date)" | tee build-output.txt
        ${{ inputs.build-command }} 2>&1 | tee -a build-output.txt
        echo "Build exit code: $?" >> build-output.txt

    - name: Run tests
      id: test
      if: success()
      shell: bash
      run: |
        echo "Starting tests at $(date)" | tee -a build-output.txt
        ${{ inputs.test-command }} 2>&1 | tee -a build-output.txt
        echo "Test exit code: $?" >> build-output.txt

    - name: Cleanup temporary files
      if: always()
      shell: bash
      run: |
        echo "Running cleanup..."
        ${{ inputs.cleanup-command }}
        echo "Cleanup complete"

    - name: Upload build log
      if: always()
      uses: actions/upload-artifact@v4
      with:
        name: build-output-${{ github.run_id }}
        path: build-output.txt
```

Usage in a workflow:

```yaml
name: Build and Test Project
on: push

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: your-username/your-repo/.github/actions/build-and-test@main
        with:
          build-command: "npm ci && npm run build"
          test-command: "npm test"
          cleanup-command: "rm -rf ./node_modules ./dist"
```

Key points in this exercise:

- **`if: always()`** on the cleanup and upload steps ensures they run regardless of build or test failure. This is critical for debugging CI failures — without it, a failed build would skip cleanup and leave the workspace in an inconsistent state.

- **`if: success()`** on the test step means tests only run if the build succeeded. You could also use `if: always()` to run tests regardless of build status, but that is unusual.

- **`if: failure()`** would run a step only when a previous step failed. You could use it for failure-specific cleanup or diagnostics:

```yaml
- name: Debug on failure
  if: failure()
  shell: bash
  run: |
    echo "Build or test failed. Environment info:"
    uname -a
    df -h
```

- **Nested actions** — the composite action uses `actions/checkout@v4` and `actions/upload-artifact@v4` inside its steps. This is allowed in composite actions and is a powerful way to compose higher-level actions from marketplace building blocks.

- **Error handling hierarchy**: `if: always()` runs unconditionally; `if: success()` runs only when all previous steps succeeded; `if: failure()` runs only when a previous step failed; `if: cancelled()` runs only when the workflow was cancelled.
</details>

### Exercise 12: Refactor a duplicated workflow into reusable and composite components

**Scenario:** The workflow below is copy-pasted across 5 microservice repositories. Refactor it into a reusable workflow and a composite action such that all 5 repos can replace their duplicated workflow with a single `uses:` line and minimal configuration.

Original duplicated workflow in each repo:

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
```

<details>
<summary>Solution</summary>

First, create a composite action for the repeated setup steps. In the shared repository, create `.github/actions/node-setup/action.yml`:

```yaml
# .github/actions/node-setup/action.yml
name: "Node.js Setup"
description: "Checks out code, installs Node.js, and installs dependencies"
inputs:
  node-version:
    description: "Node.js version to use"
    required: false
    default: "20"
runs:
  using: composite
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
    - shell: bash
      run: npm ci
```

Then create the reusable workflow. In the shared repository, create `.github/workflows/node-ci.yml`:

```yaml
# .github/workflows/node-ci.yml
name: Node.js CI
on:
  workflow_call:
    inputs:
      node-version:
        required: false
        type: string
        default: "20"

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: your-org/shared-workflows/.github/actions/node-setup@main
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: your-org/shared-workflows/.github/actions/node-setup@main
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm test

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: your-org/shared-workflows/.github/actions/node-setup@main
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm run build
```

Now each of the 5 microservice repos replaces their 40-line CI workflow with this:

```yaml
# Each microservice repo replaces its entire ci.yml with this:
name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  ci:
    uses: your-org/shared-workflows/.github/workflows/node-ci.yml@v1
    with:
      node-version: "20"
```

This reduces each repo's workflow from ~40 lines to ~12 lines. If you need to change the Node version or add a new step (like security scanning), you update the reusable workflow once and all 5 repos pick up the change immediately.
</details>

## Bonus: Self-Assessment

After completing these exercises, you should be able to:

- Distinguish between reusable workflows (job-level reuse, separate runner) and composite actions (step-level reuse, same runner)
- Create a reusable workflow with `on: workflow_call` and define inputs, secrets, and outputs
- Call a reusable workflow from another workflow using the `uses:` syntax with a proper ref
- Create a composite action with `runs.using: composite` and explain why `shell:` is required
- Pass data between callers and reusable workflows using `workflow_call` outputs
- Use `secrets: inherit` and explicit secret passing appropriately
- Handle failures in composite actions using `if: always()`, `if: success()`, and `if: failure()`
- Design a matrix of reusable workflow calls for parallel testing across multiple configurations
- Refactor duplicated workflows into shared reusable and composite components
