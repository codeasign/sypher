You will build a reusable CI workflow and a composite setup action, then wire them together across two repositories. By the end you will have a centralized CI pipeline that any repository in your organization can call with a single `uses:` line.

## Setup

Create two repositories. The first holds the shared workflows and actions. The second is a consumer that calls them.

```bash
# Create the shared workflows repository
gh repo create shared-workflows --public --clone
```

```bash
# Create the consumer application repository
gh repo create my-consumer-app --public --clone
```

```bash
# Create the directory structure in the shared-workflows repo
mkdir -p .github/workflows
mkdir -p .github/actions/setup-project
```

```bash
# Create a sample application in the consumer repo for testing
mkdir -p .github/workflows
cat > app.py << 'EOF'
def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
    assert add(0, 0) == 0
    print("All tests passed!")

if __name__ == "__main__":
    test_add()
EOF
```

## The Walkthrough

### Step 1: Create a reusable workflow that runs lint and test

In the `shared-workflows` repository, create `.github/workflows/ci-checks.yml`:

```yaml
name: CI Checks
on: workflow_call

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install flake8
      - run: flake8 . --max-line-length=100

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install pytest
      - run: pytest
```

The key element is `on: workflow_call`. This single line marks the workflow as reusable. Without it, the workflow can only run on its own triggers — it cannot be called from another workflow.

**Checkpoint:** Push this file to the shared-workflows repository:

```bash
git add .github/workflows/ci-checks.yml
git commit -m "Add reusable CI workflow"
git push origin main
```

### Step 2: Add inputs for node-version and os

The reusable workflow is hardcoded to Python 3.11 on Ubuntu. Make it flexible with inputs. Update `.github/workflows/ci-checks.yml`:

```yaml
name: CI Checks
on:
  workflow_call:
    inputs:
      python-version:
        required: false
        type: string
        default: "3.11"
      os:
        required: false
        type: string
        default: ubuntu-latest

jobs:
  lint:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ inputs.python-version }}
      - run: pip install flake8
      - run: flake8 . --max-line-length=100

  test:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ inputs.python-version }}
      - run: pip install pytest
      - run: pytest
```

Inputs are defined under `on: workflow_call: inputs:`. Each input can have `required`, `type`, and `default`. The `type` must be one of `string`, `number`, `boolean`, `environment`, or `choice`. Inside the jobs, you reference inputs with `${{ inputs.input-name }}`.

**Checkpoint:** Commit and push:

```bash
git add .github/workflows/ci-checks.yml
git commit -m "Add inputs for python-version and os"
git push origin main
```

### Step 3: Add secrets for an API token

Some CI steps need access to secrets — for example, a package registry token or an API key for integration tests. Add a secret input to the reusable workflow. Update `.github/workflows/ci-checks.yml`:

```yaml
name: CI Checks
on:
  workflow_call:
    inputs:
      python-version:
        required: false
        type: string
        default: "3.11"
      os:
        required: false
        type: string
        default: ubuntu-latest
    secrets:
      registry-token:
        required: false

jobs:
  lint:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ inputs.python-version }}
      - run: pip install flake8
      - run: flake8 . --max-line-length=100

  test:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ inputs.python-version }}
      - run: pip install pytest
      - run: pytest
      - name: Authenticated registry check
        if: secrets.registry-token != ''
        run: echo "Registry token is available" > /dev/null
```

Secrets under `workflow_call` are declared with `secrets: secret-name: { required: true/false }`. Inside the workflow, you reference them with `${{ secrets.secret-name }}`. The `if:` condition uses `secrets.registry-token != ''` to skip the step if no token was provided.

**Checkpoint:** Commit and push:

```bash
git add .github/workflows/ci-checks.yml
git commit -m "Add optional registry-token secret"
git push origin main
```

### Step 4: Create a consumer workflow that calls the reusable workflow

Switch to the `my-consumer-app` repository. Create `.github/workflows/ci.yml`:

```yaml
name: Consumer CI
on:
  push:
    branches: [main]
  pull_request:
  workflow_dispatch:

jobs:
  ci-checks:
    uses: your-username/shared-workflows/.github/workflows/ci-checks.yml@main
    with:
      python-version: "3.12"
      os: ubuntu-latest
    secrets:
      registry-token: ${{ secrets.REGISTRY_TOKEN }}
```

The `uses:` syntax follows the pattern `owner/repo/.github/workflows/workflow-name.yml@ref`. The `@ref` can be a branch (`@main`), a tag (`@v1`), or a commit SHA (`@a1b2c3d`). Using a tag or SHA is strongly recommended for production — a branch reference changes every time you push, which can break callers unexpectedly.

**Checkpoint:** Commit and push the consumer workflow:

```bash
git add .github/workflows/ci.yml

```

Navigate to the Actions tab of the consumer repo. You should see the CI workflow run. Click into it and observe that the `ci-checks` job actually runs the reusable workflow's jobs (`lint` and `test`) as if they were defined locally.

### Step 5: Create a composite action for project setup

The reusable workflow duplicates the checkout and Python setup steps across both jobs. Extract this into a composite action. In the `shared-workflows` repository, create `.github/actions/setup-project/action.yml`:

```yaml
name: "Setup Python Project"
description: "Checks out code, sets up Python, and installs dependencies"
inputs:
  python-version:
    description: "Python version to use"
    required: false
    default: "3.11"
runs:
  using: composite
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: ${{ inputs.python-version }}
    - shell: bash
      run: pip install --upgrade pip
    - shell: bash
      run: pip install flake8 pytest
```

A composite action requires three things:
- `runs.using: composite` — tells GitHub this is a composite action
- `runs.steps` — a list of steps, same as a workflow step list
- `inputs:` (optional) — input definitions, just like any action

Each step in a composite action must specify `shell:` for `run:` commands. This is a key difference from workflow steps, where `shell:` defaults to `bash` on Linux runners. In composite actions, you must always set `shell:` explicitly.

### Step 6: Update the reusable workflow to use the composite action

Now replace the duplicated steps in the reusable workflow. Update `.github/workflows/ci-checks.yml`:

```yaml
name: CI Checks
on:
  workflow_call:
    inputs:
      python-version:
        required: false
        type: string
        default: "3.11"
      os:
        required: false
        type: string
        default: ubuntu-latest
    secrets:
      registry-token:
        required: false

jobs:
  lint:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: flake8 . --max-line-length=100

  test:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: pytest
      - name: Authenticated registry check
        if: secrets.registry-token != ''
        run: echo "Registry token is available" > /dev/null
```

The `uses:` path for a composite action follows the same pattern as a reusable workflow: `owner/repo/.github/actions/action-name@ref`. The action name is the directory name containing `action.yml`.

**Checkpoint:** Commit and push both files:

```bash
git add .github/actions/setup-project/action.yml
git add .github/workflows/ci-checks.yml
git commit -m "Add composite setup action and use it in CI workflow"
git push origin main
```

Now trigger the consumer workflow again. The reusable workflow will use the composite action for setup, reducing duplication and making the CI workflow cleaner.

### Step 7: Add workflow_call outputs and use them in the caller

A reusable workflow can pass data back to the caller using `on: workflow_call: outputs:`. Add a build output to the reusable workflow. First, add a build job. Update `.github/workflows/ci-checks.yml`:

```yaml
name: CI Checks
on:
  workflow_call:
    inputs:
      python-version:
        required: false
        type: string
        default: "3.11"
      os:
        required: false
        type: string
        default: ubuntu-latest
    secrets:
      registry-token:
        required: false
    outputs:
      build-status:
        description: "Status of the build job"
        value: ${{ jobs.build.outputs.status }}

jobs:
  lint:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: flake8 . --max-line-length=100

  test:
    runs-on: ${{ inputs.os }}
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: pytest
      - name: Authenticated registry check
        if: secrets.registry-token != ''
        run: echo "Registry token is available" > /dev/null

  build:
    runs-on: ${{ inputs.os }}
    needs: [lint, test]
    outputs:
      status: ${{ steps.build-outcome.outputs.status }}
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: |
          echo "Building application..."
          python -m compileall .
          echo "Build complete"
      - id: build-outcome
        run: echo "status=success" >> $GITHUB_OUTPUT
```

Now update the consumer workflow in `my-consumer-app` to use the output. Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy
on:
  workflow_dispatch:

jobs:
  ci-checks:
    uses: your-username/shared-workflows/.github/workflows/ci-checks.yml@main
    with:
      python-version: "3.12"
      os: ubuntu-latest

  deploy:
    needs: ci-checks
    runs-on: ubuntu-latest
    if: needs.ci-checks.outputs.build-status == 'success'
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying application..."
```

The `deploy` job uses `needs: ci-checks` to wait for the reusable workflow to complete, then checks `needs.ci-checks.outputs.build-status` to decide whether to deploy. This is the same pattern as using outputs between jobs in a single workflow, but the output comes from a separate workflow.

### Step 8: Test that changes propagate to the caller

Make a change to the reusable workflow and verify it propagates without touching the consumer. In `shared-workflows`, modify `.github/workflows/ci-checks.yml` — add a new step to the lint job:

```yaml
    steps:
      - uses: your-username/shared-workflows/.github/actions/setup-project@main
        with:
          python-version: ${{ inputs.python-version }}
      - run: flake8 . --max-line-length=100
      - run: echo "Linting completed at $(date)"  # new line
```

Commit and push to `shared-workflows`. Now trigger the consumer workflow again — either by pushing to `my-consumer-app` or manually via `workflow_dispatch`. The consumer will automatically pick up the change because it references `@main` (or whatever ref you used). This is the power of reusable workflows: one change in the shared repository updates every caller.

## Common Errors

### Error 1: "Called workflow failed"

**Message:** `Called workflow 'your-username/shared-workflows/.github/workflows/ci-checks.yml@main' failed with error: ...`

**Why:** The reusable workflow itself failed — a job inside it failed, or the workflow file has a syntax error. The error is often a generic message that does not tell you which job failed.

**Fix:** Go to the shared-workflows repository and look at the Actions tab. You will see the individual run of the reusable workflow with detailed logs for each job. Alternatively, add `ACTIONS_STEP_DEBUG: true` as a secret to enable debug logging:

```yaml
jobs:
  ci-checks:
    uses: your-username/shared-workflows/.github/workflows/ci-checks.yml@main
    with:
      python-version: "3.12"
      os: ubuntu-latest
    secrets:
      registry-token: ${{ secrets.REGISTRY_TOKEN }}
      ACTIONS_STEP_DEBUG: true
```

### Error 2: "Invalid workflow reference"

**Message:** `Invalid workflow reference: 'your-username/shared-workflows/.github/workflows/ci-checks.yml'`

**Why:** The `uses:` path is incorrect. Common causes: the workflow file does not exist at that path, the repository name is wrong, the ref (branch/tag) does not exist, or the file is not in `.github/workflows/`.

**Fix:** Verify the exact path. The format is strict: `owner/repo/.github/workflows/filename.yml@ref`. The file must exist in the `.github/workflows/` directory of the specified repository at the specified ref. Check that the repository is public (or that you have appropriate access to a private repository):

```yaml
# Correct
uses: my-org/shared-workflows/.github/workflows/ci-checks.yml@v1

# Common mistakes:
# Missing .github/workflows/ prefix
uses: my-org/shared-workflows/ci-checks.yml@v1
# Wrong ref
uses: my-org/shared-workflows/.github/workflows/ci-checks.yml@nonexistent-branch
# Wrong file extension
uses: my-org/shared-workflows/.github/workflows/ci-checks.yaml@main
```

### Error 3: "Secret input required but not passed"

**Message:** `Called workflow '...' requires secret 'registry-token' but was not provided by the caller.`

**Why:** The reusable workflow declares a secret with `required: true`, but the caller does not pass it. This is a hard failure — the caller cannot proceed.

**Fix:** Either pass the secret from the caller, or make the secret optional in the reusable workflow:

```yaml
# Make the secret optional (recommended if it is not always needed)
on:
  workflow_call:
    secrets:
      registry-token:
        required: false
```

```yaml
# Pass the secret from the caller
jobs:
  ci-checks:
    uses: ...
    secrets:
      registry-token: ${{ secrets.REGISTRY_TOKEN }}
```

### Error 4: "Composite action does not support `uses` at top level"

**Message:** `Invalid action definition: 'uses' is not allowed in the top-level of a composite action. Only 'runs' is allowed.`

**Why:** You placed a `uses:` key at the top level of `action.yml` instead of inside `runs.steps`. The top level of a composite action only supports `name`, `description`, `inputs`, `outputs`, and `runs`.

**Fix:** Move the `uses:` inside `runs.steps`:

```yaml
# Bad — uses at the top level
name: Setup Project
uses: actions/checkout@v4  # Wrong!
runs:
  using: composite
  steps:
    - run: echo hello
      shell: bash
```

```yaml
# Good — uses inside runs.steps
name: Setup Project
runs:
  using: composite
  steps:
    - uses: actions/checkout@v4  # Correct
    - run: echo hello
      shell: bash
```

## What You Just Learned

- `on: workflow_call` marks a workflow as reusable, allowing it to be called from other workflows
- Reusable workflows run on their own runner instance with their own workspace, separate from the caller
- Inputs, secrets, and outputs are declared under `on: workflow_call` and referenced with `${{ inputs.name }}`, `${{ secrets.name }}`, and `${{ jobs.job-id.outputs.name }}`
- Composite actions use `runs.using: composite` and `runs.steps` to package multiple steps into a reusable unit that runs inside the caller's job
- The `uses:` syntax for reusable workflows follows `owner/repo/.github/workflows/file.yml@ref`, while composite actions use `owner/repo/.github/actions/name@ref`
- `secrets: inherit` passes all caller secrets to the reusable workflow — use it carefully and prefer explicit secret passing
- Changes to a reusable workflow propagate to all callers automatically when you use a branch ref, or explicitly when you update a tag ref
