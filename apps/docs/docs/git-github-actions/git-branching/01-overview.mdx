You are working on a team feature and your teammate pushes a breaking change to `main`. Suddenly your working directory is a minefield of half-finished code and broken dependencies. You cannot commit your incomplete work, and you cannot revert your teammate without losing your context. You need a way to isolate work-in-progress from the stable line вҖ” and that is exactly what branches give you.

## The Concept

A branch is a movable pointer to one specific commit. When you create a branch, Git does not copy any files вҖ” it writes a 41-byte file containing the target commit hash. Switching branches tells Git to rewrite your working directory and staging area to match that commit's snapshot.

Think of branches like labeled bookmarks in a choose-your-own-adventure novel. The main storyline is one path; you place a bookmark before trying a different ending. If the alternate path fails, you return to the bookmark. If it succeeds, you merge the path back. No pages are duplicated вҖ” only the bookmark moves.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git branch` | List local branches. The current branch gets a `*` prefix. | `git branch` |
| `git branch <name>` | Create a new branch at the current HEAD. No switch happens. | `git branch feature-login` |
| `git branch -d <name>` | Delete a fully merged branch safely. | `git branch -d old-branch` |
| `git branch -D <name>` | Force-delete a branch regardless of merge status. | `git branch -D abandonned-feature` |
| `git switch <name>` | Switch to an existing branch (Git 2.23+). Safe вҖ” refuses if changes would be lost. | `git switch feature-login` |
| `git switch -c <name>` | Create a branch AND switch to it in one command. | `git switch -c new-feature` |
| `git checkout -b <name>` | Legacy equivalent of `git switch -c`. Still works everywhere. | `git checkout -b new-feature` |
| `git branch -m <old> <new>` | Rename a branch. | `git branch -m bugfix bug-fix` |
| `git branch -a` | List all branches including remote-tracking ones. | `git branch -a` |
| `git branch -vv` | List branches with their upstream tracking branch and last commit message. | `git branch -vv` |

**Windows note:** On Windows, Git uses the same commands вҖ” no difference. File paths use forward slashes in Git commands (`git add src/file.py`), which works on both OSes.

## How It Actually Works вҖ” Step by Step

### Step 1: Starting state вҖ” one branch, linear history

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2))  вҶҗ main
                               вҶҗ HEAD
```

Your repository has one branch called `main` pointing at commit `C2`. `HEAD` is a symbolic reference pointing at `main`, so `HEAD` вҶ’ `main` вҶ’ `C2`. When you run `git log`, you see commits `C2`, `C1`, `C0`.

### Step 2: Create a branch вҖ” no new commit, just a new pointer

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2))  вҶҗ main, feature-login
                               вҶҗ HEAD
```

Running `git branch feature-login` creates a new pointer file at `.git/refs/heads/feature-login` containing the hash of `C2`. Nothing else changes. Your working directory is identical. Both `main` and `feature-login` point at the same commit.

### Step 3: Switch branches вҖ” HEAD moves, working tree rewrites

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2))  вҶҗ main
                               вҶҗ feature-login
                               вҶҗ HEAD
```

Running `git switch feature-login` updates `.git/HEAD` to point at `feature-login` instead of `main`. Git then compares the tree of `C2` (the target) with the current working directory. Since they match (you just created the branch), no files change on disk.

### Step 4: Make a new commit on the branch вҖ” history diverges

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2))  вҶҗ main
                               в”Ӯ
                               в””в”Җв”Җв–¶ ((D0))  вҶҗ feature-login
                                            вҶҗ HEAD
```

You edit a file, stage it with `git add`, and run `git commit`. Git creates a new commit `D0` whose parent is `C2`. The `feature-login` pointer moves forward to `D0`. `main` stays at `C2`. History has diverged вҖ” you now have two lines of development.

### Step 5: Make more commits вҖ” branches drift apart

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2)) в”Җв”Җв–¶ ((E0))  вҶҗ main
                               в”Ӯ
                               в””в”Җв”Җв–¶ ((D0)) в”Җв”Җв–¶ ((D1))  вҶҗ feature-login
                                                       вҶҗ HEAD
```

Meanwhile, someone committed `E0` on `main`. Now each branch has commits the other does not. The divergence distance is the set of commits reachable from one branch but not the other.

### Step 6: List branches and see the divergence

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2)) в”Җв”Җв–¶ ((E0))  вҶҗ main
                               в”Ӯ
                               в””в”Җв”Җв–¶ ((D0)) в”Җв”Җв–¶ ((D1))  вҶҗ feature-login
                                                       вҶҗ HEAD
```

Running `git branch -v` shows both branches with their latest commit message. Running `git log main..feature-login` shows commits reachable from `feature-login` but not from `main` вҖ” in this diagram, `D0` and `D1`. This is how you review what your branch changed before merging.

## When to Use This / When Not To

**Use branches when:**
- You are starting work on a feature that will take more than one commit
- You need to fix a bug in production while mid-way through a feature
- You want to experiment without risk to the stable codebase
- You are collaborating and need a clean place for code review before merging

**Do not use branches when:**
- You are fixing a one-line typo on a solo project вҖ” commit directly to `main` after staging
- You are the only developer and never need to revisit old states вҖ” a linear history is simpler
- You are still exploring and have not made any commit worth branching from

The cost of a branch is near-zero (41 bytes for the pointer file), so the default should be "branch early, branch often." The real cost is the mental overhead of remembering which branch you are on вҖ” `git status` and your shell prompt (configured to show the branch name) solve this.
