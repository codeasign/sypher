# Security Scanning with CodeQL and Dependabot

## The Problem

Imagine you maintain a production Node.js service. Six months ago, someone added `lodash@4.17.15` to handle deep object merges. You shipped the feature, moved on, and nobody thought about that dependency again.

In the real world, `lodash@4.17.15` has a known **prototype pollution vulnerability** (CVE-2020-8203). An attacker can craft a malicious JSON payload that pollutes `Object.prototype`, injecting properties that bypass validation checks. Your authentication middleware might silently start accepting unexpected fields. Or consider `log4j` (CVE-2021-44228) — a Java logging library so widely used that the RCE vulnerability earned a CVSS score of 10.0, the maximum. Companies spent months patching systems because they had no automated way to discover which of their repos depended on the vulnerable version.

Maybe you have seen this happen. A security advisory goes out on a Tuesday. Your team scrambles to find everywhere the vulnerable package is used. Someone creates a JIRA ticket. Two weeks later, the ticket is still open because the person who added the dependency left the team. In the meantime, production is exposed.

Vulnerable dependencies don't announce themselves. You need automated tools that continuously monitor what you depend on and what your own code does. That is where **Dependabot** and **CodeQL** come in.

## The Concept

**Dependabot** and **CodeQL** are GitHub-native security tools that work together to protect your repositories. They cover two different layers of the security surface.

### Dependabot — Dependency Vulnerability Scanning

Dependabot monitors your project's dependency manifest files (like `package.json`, `requirements.txt`, `Gemfile`, `Dockerfile`) against the **GitHub Advisory Database**. When it finds a match — your project depends on a package version that has a known vulnerability — Dependabot opens a pull request that bumps the dependency to the minimum patched version.

Dependabot runs on a schedule you define (daily, weekly, monthly). It can target multiple package ecosystems in the same repository. You can configure labels, assignees, reviewers, and limit how many open PRs Dependabot creates at once. It can also be configured to ignore certain dependencies or allow only specific update types.

### CodeQL — Static Application Security Testing (SAST)

CodeQL is GitHub's semantic code analysis engine. Instead of scanning for strings or regex patterns, CodeQL treats your codebase as a **relational database**. It runs queries written in QL (CodeQL's query language) against that database to find security vulnerabilities and code quality issues.

CodeQL can detect:
- SQL injection
- Cross-site scripting (XSS)
- Hardcoded credentials and API keys
- Path traversal
- Insecure deserialization
- Server-side request forgery (SSRF)
- Command injection
- Buffer overflows (in C and C++)

CodeQL runs during CI — typically triggered on pushes to `main` and on pull requests. Results appear as alerts in the **Security tab** of your repository, with severity levels, affected lines, and remediation guidance. You can also export results in SARIF format and upload them to GitHub for centralized triage.

### How They Work Together

Dependabot catches the **dependencies you import**. CodeQL catches the **vulnerabilities you write**. Together, they form a defense-in-depth security strategy. Dependabot prevents you from shipping known-bad packages; CodeQL prevents you from shipping known-bad code patterns.

## Command and Configuration Reference

Below are the key configuration files, workflow actions, and commands you need.

### Dependabot Configuration

Dependabot is configured in `.github/dependabot.yml` at the root of your repository.

| Field | Description | Required | Example |
|---|---|---|---|
| `version` | Schema version (must be `2`) | Yes | `version: 2` |
| `updates` | Array of update configurations | Yes | See below |
| `package-ecosystem` | Package manager to scan | Yes per entry | `npm`, `pip`, `docker`, `github-actions` |
| `directory` | Location of manifest | Yes per entry | `/`, `/backend` |
| `schedule.interval` | How often to check | Yes per entry | `daily`, `weekly`, `monthly` |
| `schedule.time` | Time of day (HH:MM) | No | `09:00` |
| `schedule.timezone` | Timezone for schedule | No | `America/New_York` |
| `schedule.day` | Day of week for weekly | No | `monday` |
| `open-pull-requests-limit` | Max concurrent PRs | No (default 5) | `10` |
| `labels` | Labels on Dependabot PRs | No | `["dependencies", "security"]` |
| `assignees` | Users to assign | No | `["alice", "bob"]` |
| `reviewers` | Team/user reviewers | No | `["team-security"]` |
| `allow` | Filter by update type | No | `[{"dependency-type": "direct"}]` |
| `ignore` | Skip specific deps | No | `[{"dependency-name": "express"}]` |
| `target-branch` | Branch for PRs | No | `develop` |

**Minimal Example:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
      day: monday
      time: "09:00"
    open-pull-requests-limit: 10
    labels:
      - dependencies
      - security
  - package-ecosystem: docker
    directory: /
    schedule:
      interval: weekly
```

**With Allow and Ignore Rules:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
    allow:
      - dependency-type: direct
    ignore:
      - dependency-name: express
        versions:
          - ">=5.x"
      - dependency-name: "@types/*"
```

### CodeQL Workflow Actions

CodeQL runs through a GitHub Actions workflow using the `github/codeql-action` family of actions.

| Action / Field | Description | Required | Example |
|---|---|---|---|
| `github/codeql-action/init@v3` | Initialize CodeQL analysis | Yes | `uses: github/codeql-action/init@v3` |
| `languages` | Languages to analyze | Yes with init | `["javascript", "python"]` |
| `queries` | Query suites to run | No | `security-and-quality`, `security-extended` |
| `github/codeql-action/analyze@v3` | Run queries and upload results | Yes | `uses: github/codeql-action/analyze@v3` |
| `output` | SARIF output file | No | `../results.sarif` |
| `upload-sarif` | Upload SARIF to GitHub | No | `true` |
| `category` | Categorize results | No | `security-and-quality` |
| `paths` | Include only specific paths | No | `["src/**"]` |
| `paths-ignore` | Exclude specific paths | No | `["**/test/**", "**/*.test.js"]` |
| `github/codeql-action/upload-sarif@v3` | Upload SARIF separately | Alternative | See below |

**Minimal CodeQL Workflow:**

```yaml
# .github/workflows/codeql.yml
name: CodeQL Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: "0 8 * * 1"

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    strategy:
      fail-fast: false
      matrix:
        language: [javascript, python]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          category: "/language:${{matrix.language}}"
```

**With Path Filters:**

```yaml
      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-extended      

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          paths: src
          paths-ignore: |
            src/vendor
            **/*.test.js
```

### Dependabot Auto-Approve and Auto-Merge

You can automate the merge of Dependabot PRs to reduce manual overhead for low-risk updates.

```yaml
# .github/workflows/dependabot-auto-merge.yml
name: Dependabot Auto-Merge

on: pull_request_target

permissions:
  pull-requests: write
  contents: write

jobs:
  auto-merge:
    runs-on: ubuntu-latest
    if: github.actor == 'dependabot[bot]'
    steps:
      - name: Fetch Dependabot metadata
        id: metadata
        uses: dependabot/fetch-metadata@v2
        with:
          alert-lookup: true

      - name: Approve a PR
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr review --approve "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}

      - name: Enable auto-merge
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr merge --auto --squash "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}
```

### Query Suites Reference

| Suite Name | Description | Coverage |
|---|---|---|
| `security-extended` | Moderate-precision security queries | More results, minor noise |
| `security-and-quality` | Security + code quality queries | Broadest coverage |
| `security-experimental` | Experimental and lower-precision queries | Research-grade |
| Default (no queries set) | High-precision security queries only | Lowest noise |

## How It Actually Works

### Dependabot Internals

Dependabot runs as a GitHub-native service triggered by your schedule configuration. Here is what happens each time it runs:

│ ncy file │   │ versions │   │ Advisory │   │ safe     │   │ assignee │
│          │   │          │   │ DB       │   │ version  │   │ & labels │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
`} />

1. **Fetch manifest.** Dependabot reads your dependency files (e.g., `package-lock.json`, `yarn.lock`, `Gemfile.lock`) from the configured directory.
2. **Parse versions.** It extracts every dependency name and its pinned version from the lock file.
3. **Match against Advisory DB.** Each dependency-version pair is checked against the GitHub Advisory Database, which aggregates CVEs from NVD, GitHub security advisories, and other sources.
4. **Create PR with safe version.** If a vulnerability is found, Dependabot determines the minimum patched version and creates a pull request that updates the version.
5. **Notify the team.** Labels, assignees, and reviewers are applied as configured, and alerts appear in the repository Security tab.

CodeQL follows a different pipeline that runs during CI rather than on a schedule.

1. **Build a CodeQL database.** The `github/codeql-action/init` step extracts a relational representation of your code. For compiled languages (C++, Java, Go), it intercepts the build process. For interpreted languages (JavaScript, Python), it extracts directly from the source tree.
2. **Run security queries.** The `github/codeql-action/analyze` step runs the configured query suite against the database. Standard queries cover the OWASP Top 10. The `security-and-quality` suite adds hundreds of additional queries.
3. **Generate SARIF output.** CodeQL produces a SARIF (Static Analysis Results Interchange Format) file. SARIF is an OASIS standard format for static analysis results.
4. **Upload results to Security tab.** The `github/codeql-action/analyze` action automatically uploads the SARIF file unless you disable it. Alerts appear in the repository Security tab, grouped by query and file location.

## When to Use Security Scanning

**Use Dependabot and CodeQL when:**
- Your repository has external dependencies (npm, pip, Maven, NuGet, etc.)
- Your team maintains a production application or service
- You accept pull requests from external contributors
- You need to meet compliance requirements (SOC 2, PCI-DSS, FedRAMP)
- You want automated guardrails against common vulnerability classes
- You have a CI pipeline that can run analysis on every push

**You might skip or limit them when:**
- You work on a personal toy project with no dependencies
- Your project has zero external dependencies (rare but possible)
- You already use an equivalent enterprise SAST tool with mandatory results
- You are prototyping rapidly and YAML configuration overhead is not justified
- The repository contains only documentation or configuration files

For production repositories, security scanning should never be optional. The cost of a single missed vulnerability — in incident response time, data breach liability, and reputational damage — far exceeds the cost of configuring a YAML file.

## Key Takeaways

- **Dependabot** scans your dependencies against the GitHub Advisory Database and opens pull requests with patched versions automatically.
- **CodeQL** analyzes your source code for security vulnerabilities using semantic queries and reports results as SARIF to the Security tab.
- **Dependabot runs on a schedule** (daily/weekly/monthly). **CodeQL runs on every push and PR** you configure it for.
- **You configure both through YAML files** committed to `.github/` in your repository.
- **The Security tab** is your single pane of glass for both dependency alerts and code-level alerts.
- **Query suites** let you trade off between noise and coverage: default (high precision), `security-extended`, or `security-and-quality` (broadest).
- **Auto-approve and auto-merge** can reduce operational overhead for low-risk dependency updates.
- **SARIF upload** lets you centralize results from multiple analysis tools into GitHub.
