In this walkthrough, you will simulate a two-person collaboration: one repository on GitHub, two local clones. Do not worry if you do not have a partner — you can use two terminal windows on your own machine.

## Setup

You need a throwaway GitHub repository and a local directory. Complete these steps before starting the walkthrough.

### Step S1: Create a GitHub Repository

1. Log in to GitHub and click the green "New" button next to your repositories list, or go to `https://github.com/new`
2. Name the repository `git-remote-practice`. Do not initialize it with a README, .gitignore, or license — you want an empty repository
3. Click "Create repository"
4. Copy the HTTPS URL shown on the quick-setup page. It looks like `https://github.com/<your-username>/git-remote-practice.git`

### Step S2: Set Up Your Local Directory

On Windows, open PowerShell (or Git Bash). On macOS or Linux, open a terminal.

```bash
mkdir ~/git-remote-practice
cd ~/git-remote-practice
git init
```

Confirm the directory is empty:

```bash
git status
```

Expected output:

```
On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)
```

### Step S3: Make an Initial Commit

```bash
echo "hello from the local repo" > README.md
git add README.md
git commit -m "initial commit"
```

Expected output:

```
[main (root-commit) a1b2c3d] initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

Your commit hash will be different — that is normal.

### Step S4: Connect Your Remote

```bash
git remote add origin https://github.com/<your-username>/git-remote-practice.git
```

Verify the remote was added:

```bash
git remote -v
```

Expected output:

```
origin  https://github.com/<your-username>/git-remote-practice.git (fetch)
origin  https://github.com/<your-username>/git-remote-practice.git (push)
```

Two URLs appear — one for fetch, one for push. They are usually the same, but you can configure them separately if needed.

### Step S5: Push the Initial Commit

```bash
git push -u origin main
```

This uploads your commit and sets `main` to track `origin/main`. The `-u` flag means you can run plain `git push` and `git pull` afterward.

## The Walkthrough

You are Developer A. A second developer — Developer B — will clone the same repository and make changes. You will handle conflicts, fetch remote changes, and push your own work.

### Step 1: Developer B Clones the Repository

Open a second terminal window. Navigate to a different directory (for example, `~/git-remote-practice-B`).

```bash
cd ~
git clone https://github.com/<your-username>/git-remote-practice.git git-remote-practice-B
cd git-remote-practice-B
```

Expected output:

```
Cloning into 'git-remote-practice-B'...
remote: Enumerating objects: 3, done.
remote: Counting objects: 100% (3/3), done.
remote: Total 3 (delta 0), reused 0 (delta 0)
Receiving objects: 100% (3/3), done.
```

Developer B now has a complete copy of the repository, including the `README.md` file and the full commit history. The `origin` remote was automatically configured to point to the same GitHub URL.

**Checkpoint:** Verify both clones are in sync.

In Developer B's terminal:

```bash
git log --oneline
```

Expected output:

```
a1b2c3d (HEAD -> main, origin/main, origin/HEAD) initial commit
```

The `origin/main` and `origin/HEAD` refs confirm that the remote-tracking branch was also cloned.

### Step 2: Developer B Makes a Commit and Pushes

In Developer B's terminal, add a line to the README:

```bash
echo "line added by Developer B" >> README.md
git add README.md
git commit -m "B adds a line"
git push
```

Expected output:

```
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 8 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (3/3), 320 bytes | 320.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0)
To https://github.com/<your-username>/git-remote-practice.git
   a1b2c3d..e5f6g7h  main -> main
```

The remote on GitHub now has commit `e5f6g7h`. Your local clone (Developer A) does not know this yet.

**Checkpoint:** In Developer A's terminal, run:

```bash
git log --oneline --all
```

Expected output:

```
a1b2c3d (HEAD -> main, origin/main) initial commit
```

Developer A's `origin/main` still points at the original commit. The remote-tracking branch is stale.

### Step 3: Developer A Fetches the Remote Changes

Back in Developer A's terminal, run:

```bash
git fetch origin
```

Expected output:

```
remote: Enumerating objects: 5, done.
remote: Counting objects: 100% (5/5), done.
remote: Total 3 (delta 0), reused 0 (delta 0)
Unpacking objects: 100% (3/3), 314 bytes | 314.00 KiB/s, done.
From https://github.com/<your-username>/git-remote-practice.git
   a1b2c3d..e5f6g7h  main       -> origin/main
```

Git downloaded the new commit and updated `origin/main` to `e5f6g7h`. Your working tree and your local `main` branch are unchanged — fetch is read-only.

**Checkpoint:** View the remote-tracking branch:

```bash
git log origin/main --oneline
```

Expected output:

```
e5f6g7h (origin/main) B adds a line
a1b2c3d initial commit
```

### Step 4: Developer A Pulls to Merge Remote Changes

Now integrate Developer B's work into your local branch:

```bash
git pull origin main
```

Expected output:

```
Updating a1b2c3d..e5f6g7h
Fast-forward
 README.md | 2 ++
 1 file changed, 2 insertions(+)
```

Because you had not committed anything locally since the initial commit, git performed a **fast-forward merge** — it simply moved your `main` pointer forward to match `origin/main`. No merge commit was created.

**Checkpoint:** Verify the merged state:

```bash
git log --oneline --all
```

Expected output:

```
e5f6g7h (HEAD -> main, origin/main) B adds a line
a1b2c3d initial commit
```

Your local `main` and `origin/main` now point at the same commit again.

### Step 5: Developer A Makes a Commit and Pushes

Now it is your turn to add a change. In Developer A's terminal:

```bash
echo "line added by Developer A" >> README.md
git add README.md
git commit -m "A adds a line"
git push
```

Expected output:

```
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 8 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (3/3), 320 bytes | 320.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0)
To https://github.com/<your-username>/git-remote-practice.git
   e5f6g7h..f8g9h0i  main -> main
```

**Checkpoint:** In Developer B's terminal, run `git fetch` and check the log:

```bash
git fetch origin
git log origin/main --oneline
```

Expected output:

```
f8g9h0i (origin/main) A adds a line
e5f6g7h B adds a line
a1b2c3d initial commit
```

### Step 6: Both Developers Make Conflicting Changes

Now you will create a real conflict — the most common pain point in collaborative git.

In Developer A's terminal, change the first line of `README.md`:

```bash
echo "Developer A was here" > README.md
git add README.md
git commit -m "A overwrites README"
git push
```

The `>` operator overwrites the file. Expected push output:

```
... f8g9h0i..a1b2c3  main -> main
```

In Developer B's terminal, also change the first line differently:

```bash
echo "Developer B was here" > README.md
git add README.md
git commit -m "B overwrites README"
```

Now try to push:

```bash
git push
```

Expected output — a rejection:

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'https://github.com/...'
hint: Updates were rejected because the remote contains work that you do not
hint: have locally. This is usually caused by another repository pushing to
hint: the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
hint: See the 'Note about fast-forwards' in 'git push --help' for details.
```

This is the non-fast-forward rejection. The remote has commit A's change that is not in B's history. Git refuses to overwrite it.

**Checkpoint:** Verify B's local history:

```bash
git log --oneline --all
```

Expected output:

```
a1b2c3 (HEAD -> main) B overwrites README
f8g9h0i (origin/main) A adds a line
e5f6g7h B adds a line
a1b2c3d initial commit
```

`origin/main` is ahead of `main` due to the fetch that happened from the `git push` rejection check. Actually, git push does fetch before rejecting — so `origin/main` is now at A's commit.

### Step 7: Developer B Pulls and Resolves the Conflict

Developer B must pull and resolve:

```bash
git pull origin main
```

Expected output:

```
Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.
```

Git attempted to merge but could not because both sides changed the same region of the same file. Open `README.md` in a text editor. You will see:

```
<<<<<<< HEAD
Developer B was here
=======
Developer A was here
>>>>>>> a1b2c3d... A overwrites README
```

The section between `<<<<<<< HEAD` and `=======` is your local change. The section between `=======` and `>>>>>>>` is the incoming change from the remote.

Edit the file to keep the resolution you want. For example, combine both lines:

```
Developer A and Developer B were here
```

After editing, save the file. Then:

```bash
git add README.md
git commit -m "resolve merge conflict between A and B"
git push
```

Expected output:

```
... g1h2i3j -> main -> main
```

**Checkpoint:** Verify the history on Developer B:

```bash
git log --oneline --all --graph
```

Expected output:

```
*   g1h2i3j (HEAD -> main, origin/main) resolve merge conflict between A and B
|\
| * a1b2c3d A overwrites README
* | a1b2c3 B overwrites README
|/
* f8g9h0i A adds a line
* e5f6g7h B adds a line
* a1b2c3d initial commit
```

The merge commit `g1h2i3j` has two parents: B's commit and A's commit. The `--graph` flag shows the branching structure.

### Step 8: Developer A Pulls the Conflicting Change

Back in Developer A's terminal, the remote is now ahead. Pull to get the merge commit:

```bash
git pull origin main
```

Expected output:

```
Updating a1b2c3d..g1h2i3j
Fast-forward
 README.md | 1 +
 1 file changed, 1 insertion(+)
```

This is a fast-forward — A had no new commits, so git just moved `main` forward.

**Checkpoint:** Both developers are now synchronized. Verify:

```bash
git log --oneline --all
```

Expected output:

```
g1h2i3j (HEAD -> main, origin/main) resolve merge conflict between A and B
a1b2c3d A overwrites README
a1b2c3 B overwrites README
f8g9h0i A adds a line
e5f6g7h B adds a line
a1b2c3d initial commit
```

Both Developer A and Developer B see the same history. The local and remote are in sync.

## Common Errors at This Stage

### Error 1: Non-Fast-Forward Rejection

**Exact message:**

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'https://github.com/...'
hint: Updates were rejected because the remote contains work that you do not
hint: have locally.
```

**Why it happens:** Someone pushed to the remote after your last fetch or pull. Your local branch has diverged from the remote branch. Git refuses to push because it would overwrite commits you have not seen.

**Fix:** Run `git pull origin main` to merge the remote changes, then `git push` again.

### Error 2: Merge Conflict During Pull

**Exact message:**

```
Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.
```

**Why it happens:** Both you and another developer changed the same lines in the same file. Git cannot decide which version to keep.

**Fix:** Open the conflicted file, find the `<<<<<<<`, `=======`, `>>>>>>>` markers, edit the file to the correct content, then `git add <file>` and `git commit` to complete the merge.

### Error 3: No Such Remote

**Exact message:**

```
fatal: 'origin' does not appear to be a git repository
fatal: Could not read from remote repository.
```

**Why it happens:** You ran `git push` or `git pull` in a repository that has no remote configured, or you misspelled the remote name.

**Fix:** Run `git remote -v` to see your configured remotes. If none exist, add one with `git remote add origin <url>`.

### Error 4: Permission Denied (Public Key)

**Exact message:**

```
git@github.com: Permission denied (publickey).
fatal: Could not read from remote repository.
```

**Why it happens:** You are using SSH authentication but your SSH key is not registered with GitHub, or you are using the wrong key.

**Fix:** Add your public key to GitHub at `https://github.com/settings/keys`. Test with `ssh -T git@github.com`. Alternatively, switch to the HTTPS URL: `git remote set-url origin https://github.com/<username>/<repo>.git`.

### Error 5: Failed to Push Some Refs (No Reason Given)

**Exact message:**

```
 ! [rejected]        main -> main (remote rejected)
error: failed to push some refs to 'https://github.com/...'
```

**Why it happens:** The remote repository has branch protection rules (for example, requiring a pull request review before merging). This is configured on GitHub, not in git itself.

**Fix:** Create a pull request instead of pushing directly to the protected branch. Use `git push origin feature-branch` and open a PR on GitHub.

## What You Just Learned

- **`git remote add`** connects your local repository to a remote URL. The name `origin` is a convention, not a requirement — you can name remotes anything.
- **`git fetch`** downloads new commits from the remote and updates your remote-tracking branches (`origin/main`) without touching your working tree or local branches. It is safe to run at any time.
- **`git pull`** runs `git fetch` followed by a merge. It is the fastest way to get your local branch in sync, but it can create merge commits you did not expect.
- **`git push`** uploads your local commits to the remote. It fails if your history has diverged — you must pull first.
- **Merge conflicts** happen when two people change the same part of the same file. Git marks the conflict with `<<<<<<<`, `=======`, and `>>>>>>>` markers. You edit the file, stage it, and commit to resolve.
- **Remote-tracking branches** (`origin/main`, `origin/develop`) are your local cache of what the remote looked like last time you checked. They are updated by fetch, pull, and push, but you never commit directly to them.
