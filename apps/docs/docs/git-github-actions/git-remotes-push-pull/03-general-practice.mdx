These exercises assume you have a GitHub account and can create throwaway repositories. Each exercise stands alone — you can start from scratch or use a practice repo you already created.

## Tier 1 — Beginner

### Exercise 1: Clone a Repository

Clone the `git-scm` book repository to your local machine.

**Task:** Run the command that downloads the entire repository from `https://github.com/git-scm/git-scm.com` to a directory named `git-scm-book`.

<details>
<summary>Hint</summary>

The `git clone` command takes a URL and an optional directory name as its second argument.
</details>

<details>
<summary>Solution</summary>

```bash
git clone https://github.com/git-scm/git-scm.com.git git-scm-book
```

This downloads the repository, creates the `git-scm-book` directory, configures `origin` to point at the cloned URL, and checks out the default branch. Verify with:

```bash
cd git-scm-book
git remote -v
```

You should see two lines — one for fetch, one for push — both pointing at the same URL.
</details>

### Exercise 2: Add a Second Remote

You have a repository that already has `origin` set. Add a second remote called `upstream` pointing to the original project.

**Task:** Given that `origin` points to your fork at `https://github.com/yourname/project.git`, add `upstream` pointing to `https://github.com/original/project.git`.

<details>
<summary>Hint</summary>

Use `git remote add` with a name (`upstream`) and a URL. The URL is the second argument.
</details>

<details>
<summary>Solution</summary>

```bash
git remote add upstream https://github.com/original/project.git
```

Verify with:

```bash
git remote -v
```

Expected output:

```
origin    https://github.com/yourname/project.git (fetch)
origin    https://github.com/yourname/project.git (push)
upstream  https://github.com/original/project.git (fetch)
upstream  https://github.com/original/project.git (push)
```

You now have two remotes. You can fetch from `upstream` to get the original project's latest changes without pushing to it.
</details>

### Exercise 3: Push to a Specific Remote Branch

You have a local branch `feature-login` and want to push it to `origin` under the same name.

**Task:** Push the local branch `feature-login` to `origin` and set the upstream tracking so that future `git push` and `git pull` from this branch work without arguments.

<details>
<summary>Hint</summary>

The `-u` flag (short for `--set-upstream`) sets the tracking relationship. You must specify the remote name and the branch name.
</details>

<details>
<summary>Solution</summary>

```bash
git push -u origin feature-login
```

Expected output:

```
 * [new branch]      feature-login -> feature-login
```

After this, running `git push` from `feature-login` will push to `origin/feature-login` automatically. Check the tracking relationship:

```bash
git branch -vv
```

You should see `feature-login` with `[origin/feature-login]` listed next to it.
</details>

### Exercise 4: Fetch Remote Changes Without Merging

You know the remote has new commits because a notification from GitHub told you. You are in the middle of editing a file and do not want to disturb your working tree.

**Task:** Download the latest changes from `origin` without modifying your working directory or your current branch.

<details>
<summary>Hint</summary>

There is a single command that contacts the remote and downloads commits but touches nothing else.
</details>

<details>
<summary>Solution</summary>

```bash
git fetch origin
```

This downloads all new commits from all branches on `origin` and updates your remote-tracking branches (`origin/main`, `origin/develop`, and so on). Your working directory, staged changes, and current branch pointer are untouched. After fetching, you can inspect the new commits:

```bash
git log origin/main --oneline -5
```

You can then decide when to merge — immediately, after you finish your current edit, or never.
</details>

## Tier 2 — Intermediate

### Exercise 5: Sync a Fork with Its Upstream

You forked a repository on GitHub three weeks ago. The original project has moved ahead by 12 commits. You want to bring your fork's `main` branch up to date.

**Task:** Working in your local clone, fetch the upstream changes, merge them into your local `main`, and push the result to your fork on GitHub. Assume `origin` points to your fork and `upstream` points to the original project.

<details>
<summary>Hint</summary>

This is a three-step process: fetch from `upstream`, merge into your local `main`, then push to `origin`. You need to be on the `main` branch for the merge step.
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Make sure you are on the main branch
git checkout main

# Step 2: Fetch the latest commits from the original project
git fetch upstream

# Step 3: Merge the upstream's main into your local main
git merge upstream/main

# Step 4: Push the updated main to your fork
git push origin main
```

If step 3 produces a fast-forward, your local `main` simply moved forward. If you or a teammate made commits to your fork's `main`, a merge commit is created or a conflict arises. In that case, resolve the conflict, commit, then push.

An alternative approach uses `git pull upstream main` instead of separate `fetch` + `merge`:

```bash
git checkout main
git pull upstream main
git push origin main
```

This works because `git pull` is `git fetch` followed by `git merge` — the same two operations in one command.
</details>

### Exercise 6: Delete a Remote Branch

You pushed a feature branch called `experiment-ui` three weeks ago. The feature was abandoned. You want to clean up the remote.

**Task:** Delete the branch `experiment-ui` from `origin` using a single git push command. Do not delete your local branch — only the remote copy.

<details>
<summary>Hint</summary>

The `git push` command accepts a `--delete` flag, or you can use the colon refspec syntax: `git push origin :experiment-ui`.
</details>

<details>
<summary>Solution</summary>

```bash
git push origin --delete experiment-ui
```

Expected output:

```
 - [deleted]         experiment-ui
```

The colon syntax is equivalent:

```bash
git push origin :experiment-ui
```

This sends an empty ref to the remote's `experiment-ui` ref, which has the effect of deleting it. Your local branch remains intact:

```bash
git branch  # experiment-ui is still listed
```

To clean up the local branch too, run:

```bash
git branch -d experiment-ui
```

The `-d` flag deletes the local branch only if it has been fully merged. Use `-D` (force delete) if it has not.
</details>

### Exercise 7: Pull with Rebase to Avoid a Merge Commit

Your team prefers a linear history. You have one local commit on `main` that the remote does not have. The remote has two commits you do not have.

**Task:** Integrate the remote changes into your local branch without creating a merge commit. Use `git pull` with a specific flag.

<details>
<summary>Hint</summary>

The `--rebase` flag changes how pull integrates the remote commits. Instead of creating a merge commit, it replays your local commits on top of the remote branch.
</details>

<details>
<summary>Solution</summary>

```bash
git pull --rebase origin main
```

What happens internally:

1. Git fetches the two new commits from the remote
2. Git temporarily removes your local commit from the branch
3. Git applies the two remote commits to your branch
4. Git replays your local commit on top of the new tip

The resulting history is linear:

```
A ──> B ──> C (remote) ──> D (your commit, replayed)
```

Without `--rebase`, you would get a merge commit:

```
A ──> B ──> C ──> E (merge commit)
   └──> D ──────────┘
```

If a conflict occurs during the rebase, git pauses and lets you resolve each commit one at a time:

```bash
# After resolving the conflict in the file
git add <file>
git rebase --continue
```

To abort the rebase entirely:

```bash
git rebase --abort
```

Only use `git pull --rebase` on branches that you have not shared with others. Rebasing shared branches rewrites history and causes problems for teammates.
</details>

### Exercise 8: Push a Local Branch to a Different Remote Name

You have two remotes: `origin` (your personal fork) and `company` (your organization's internal mirror). You want to push your `feature-x` branch to `company` but not to `origin`.

**Task:** Push the `feature-x` branch to `company/feature-x` without setting the upstream.

<details>
<summary>Hint</summary>

You can omit the `-u` flag. The command is `git push <remote> <local-branch>:<remote-branch>`.
</details>

<details>
<summary>Solution</summary>

```bash
git push company feature-x:feature-x
```

Because the local and remote branch names are the same, you can also write:

```bash
git push company feature-x
```

Without `-u`, git does not set the upstream tracking. After this push, running `git push` from `feature-x` without arguments will still push to `origin/feature-x` (if the upstream was set there previously) or fail with an error about no upstream.

To verify where the push went:

```bash
git remote -v
git branch -vv
```

Use the colon syntax `local:remote` when the branch names differ on each side:

```bash
git push company feature-x:deployment/feature-x
```
</details>

## Tier 3 — Advanced

### Exercise 9: Force Push After a Sensitive Commit

You accidentally pushed a commit containing an API key to the `main` branch of your personal project repository. The key is now visible on GitHub. You have already rotated the key, but you need to remove the commit from the history.

**Scenario:** Your repository has three commits on `main`:

```
a1b2c3d (HEAD -> main, origin/main) Add README
e5f6g7h Add API key (this should never have been pushed)
f8g9h0i Initial commit
```

You want to remove commit `e5f6g7h` so that the remote history appears as if it never existed. You work alone on this personal project, so no one else depends on the branch.

**Task:** Remove the middle commit from the local history and force-push the corrected history to `origin`.

<details>
<summary>Hint</summary>

Use `git rebase -i` to drop the commit, or `git reset --soft` to uncommit it. Then you must force-push because the remote will reject the non-fast-forward update. Consider `--force-with-lease` for safety.
</details>

<details>
<summary>Solution</summary>

**Approach 1: Interactive rebase**

```bash
# Rebase the last 3 commits
git rebase -i HEAD~3
```

An editor opens with:

```
pick f8g9h0i Initial commit
pick e5f6g7h Add API key
pick a1b2c3d Add README
```

Change the line for `e5f6g7h` from `pick` to `drop` (or simply delete the line):

```
pick f8g9h0i Initial commit
drop e5f6g7h Add API key
pick a1b2c3d Add README
```

Save and exit. Git rewrites history, skipping the dropped commit. Commit `a1b2c3d` gets a new hash because its parent changed.

**Approach 2: Interactive rebase single-commit**

```bash
git rebase --onto HEAD~2 HEAD~1
```

This says: take everything from `HEAD~1` (the commit after `e5f6g7h`) forward and replay it on top of `HEAD~2` (the commit before `e5f6g7h`), effectively dropping `e5f6g7h`.

**After either approach**, force-push with lease:

```bash
git push --force-with-lease origin main
```

`--force-with-lease` checks that your remote-tracking branch (`origin/main`) still matches the remote before overwriting. If someone else pushed in the meantime, the push is rejected — preventing accidental data loss.

```bash
git push --force origin main  # also works, less safe
```

**Important security note:** Even after force-pushing, the commit with the API key is still accessible on GitHub for any user who knows the commit hash. GitHub's support team can purge it permanently. For truly sensitive data, contact GitHub support and rotate all affected credentials immediately.

**Why not `git reset`?**

```bash
git reset --soft HEAD~2
git commit -m "Add README"
git push --force-with-lease origin main
```

This also works. It moves the branch pointer back two commits, then re-creates the top commit. The result is the same: the middle commit is gone. The choice between rebase and reset is stylistic — rebase is more explicit about what you are removing.
</details>

### Exercise 10: Diverged Histories — You Need Both Changes

You and a teammate both pushed to `main` without pulling first. Your local history and the remote history have diverged. You cannot push because of a non-fast-forward rejection. You cannot reset because you want to keep both sets of changes.

**Scenario:**

Your local `main`:

```
D ──> E (your commits)
│
A ──> B ──> C (origin/main, stale)
```

Remote `main`:

```
A ──> B ──> C ──> F ──> G (teammate's commits)
```

Your goal is to produce a `main` that contains all five commits (D, E, F, G) plus a merge commit.

**Task:** Pull the remote changes, merge them with your local work, and push the result. Use `git fetch` and `git merge` explicitly rather than `git pull` so you can observe each step.

<details>
<summary>Hint</summary>

Do not use `git pull`. Run `git fetch` first, inspect the state, then `git merge` to combine. After the merge, push normally.
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Fetch the remote state
git fetch origin
```

Your repository now looks like:

```
D ──> E (main)
│
A ──> B ──> C ──> F ──> G (origin/main)
```

```bash
# Step 2: Merge the remote-tracking branch into your local branch
git merge origin/main
```

Git creates a merge commit H. The state becomes:

```
D ──> E ──> H (main)
│         │
└──> F ──> G (origin/main)
```

```bash
# Step 3: Push the merged result
git push origin main
```

This push succeeds because H contains all commits that the remote had (F, G) plus your commits (D, E). The fast-forward check passes — the remote's `main` is an ancestor of your new `main`.

**Alternative: rebase instead of merge**

```bash
git fetch origin
git rebase origin/main
```

Your commits D and E are replayed on top of the remote's tip:

```
A ──> B ──> C ──> F ──> G ──> D' ──> E' (main, rebased)
```

The commit hashes change (D' and E' are new commits), so the history is linear. Then push:

```bash
git push origin main
```

This works because the rebased history is a fast-forward from the remote's perspective.

**Trade-off analysis:**

| Approach | History | When to use |
|---|---|---|
| Merge | Non-linear, preserves the exact timeline | Shared branches, team workflows, any time you want to preserve the parallel development context |
| Rebase | Linear, cleaner graph | Personal branches, feature branches before merging into main, when you want a clean commit log |

Neither approach is universally correct. The merge approach is safer on shared branches because it does not rewrite commit hashes. The rebase approach produces a cleaner history but requires force-push if you have already shared your commits with anyone.
</details>

### Exercise 11: Partial Fetch — Get Only One Branch

You are working on a monorepo with 40 branches. Fetching all of them takes 30 seconds and clutters your terminal with output about branches you do not care about.

**Scenario:** You only need to see the latest commits on `release/v2.5` from `origin`. You do not want to fetch any other branch.

**Task:** Fetch only the `release/v2.5` branch from `origin` and update only the corresponding remote-tracking branch.

<details>
<summary>Hint</summary>

`git fetch` accepts a remote name followed by a branch name. You can also use the refspec syntax to control exactly which remote-tracking branch is updated.
</details>

<details>
<summary>Solution</summary>

**Simple approach — fetch one branch:**

```bash
git fetch origin release/v2.5
```

This downloads new commits for `release/v2.5` and updates `origin/release/v2.5`. No other branches are fetched.

**Precise approach — explicit refspec:**

```bash
git fetch origin release/v2.5:refs/remotes/origin/release/v2.5
```

This explicitly maps the remote's `release/v2.5` to the local remote-tracking `origin/release/v2.5`. The shorter form above does the same thing implicitly.

**Verify the result:**

```bash
git log origin/release/v2.5 --oneline -5
```

Only the specified branch was fetched. To confirm no other branches were updated:

```bash
git branch -r
```

Compare with the output of `git ls-remote origin` to see which branches exist on the remote but were not fetched.

**Why this matters:**

In a monorepo with dozens of branches, fetching all of them every time wastes bandwidth and time. A partial fetch:

- Downloads only the refs you explicitly request
- Skips the negotiation handshake for branches you do not care about
- Keeps your remote-tracking branch list clean

If you want to fetch all branches but prune stale ones from your local tracking:

```bash
git fetch origin --prune
```

This fetches everything and removes any remote-tracking branch that no longer exists on the remote.
</details>
