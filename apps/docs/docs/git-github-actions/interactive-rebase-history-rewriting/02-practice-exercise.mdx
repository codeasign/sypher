You will create a repository with commits that need cleanup, then use interactive rebase to squash, reword, reorder, and drop commits. Every command works identically on Windows, macOS, and Linux — but the interactive editor differs.

## Setup

```bash
mkdir -p ~/interactive-rebase-lab
cd ~/interactive-rebase-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the base and a messy branch:

```bash
echo "Initial content" > project.py
git add project.py
git commit -m "Initial commit"
echo "def hello():" >> project.py
echo "    print('Hello')" >> project.py
git add project.py
git commit -m "Add hello function"
echo "def goodbye():" >> project.py
echo "    print('Goodbye')" >> project.py
git add project.py
git commit -m "Add goodbye function"
```

Now create a branch with commits that need cleanup:

```bash
git switch -c messy-feature
echo "    print('debug')" >> project.py
git add project.py
git commit -m "WIP debugging"
echo "def calculate():" >> project.py
echo "    return 42" >> project.py
git add project.py
git commit -m "asdf"
echo "    # Fixed the calculation" >> project.py
git add project.py
git commit -m "typo fix"
echo "    return 42 * 2" >> project.py
git add project.py
git commit -m "Actually implement calculate"
echo "def validate():" >> project.py
echo "    pass" >> project.py
git add project.py
git commit -m "Add validate stub"
```

Verify the messy history:

```bash
git log --oneline
```

Expected output — 6 commits on top of `main`:

```
a1b2c3d (HEAD -> messy-feature) Add validate stub
b2c3d4e Actually implement calculate
c3d4e5f typo fix
d4e5f6a asdf
e5f6a7b WIP debugging
f6a7b8c Add goodbye function
g7h8i9k (main) Add hello function
h8i9j0k Initial commit
```

## The Walkthrough

### Step 1: Configure your editor (if needed)

Interactive rebase opens a text editor. If you have not configured one:

```bash
# VS Code (Windows, macOS, Linux)
git config --global core.editor "code --wait"

# Or for nano (simpler on Linux/macOS)
# git config --global core.editor "nano"
```

**Windows note:** If `code --wait` does not work, VS Code may not be in your PATH. Use Notepad instead: `git config --global core.editor "notepad"`. On macOS, `nano` works out of the box.

### Step 2: Start an interactive rebase of the last 5 commits

We want to clean up the 5 feature commits above the "Add goodbye function" commit:

```bash
git rebase -i HEAD~5
```

Your editor opens with a todo list:

```
pick e5f6a7b WIP debugging
pick d4e5f6a asdf
pick c3d4e5f typo fix
pick b2c3d4e Actually implement calculate
pick a1b2c3d Add validate stub

# Rebase f6a7b8c..a1b2c3d onto f6a7b8c (5 commands)
#
# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
# f, fixup <commit> = like "squash", but discard this commit's log message
# x, exec <command> = run command (the rest of the line) using shell
# d, drop <commit> = remove commit
#
# These lines can be re-ordered; they are executed from top to bottom.
```

### Step 3: Squash and reword — edit the todo list

Change the todo list to:

```
pick e5f6a7b WIP debugging
fixup d4e5f6a asdf
pick c3d4e5f typo fix
fixup b2c3d4e Actually implement calculate
reword a1b2c3d Add validate stub
```

This means:
- Keep "WIP debugging" as is
- Merge "asdf" into "WIP debugging", discarding the "asdf" message (fixup)
- Keep "typo fix" as is
- Merge "Actually implement calculate" into "typo fix", discarding its message (fixup)
- Edit the message for "Add validate stub" (reword)

Save and close the editor.

### Step 4: Git applies the squash/fixup actions

Git replays the commits. The fixup actions produce output:

```
Successfully rebased and updated refs/heads/messy-feature.
```

If there is an output silence with no errors, the rebase succeeded. Then the reword action triggers — your editor opens again with the message for "Add validate stub".

### Step 5: Reword the last commit

Change the message from:

```
Add validate stub
```

To:

```
Add validate function skeleton
```

Save and close. Git creates the new commit with the updated message.

### Step 6: Verify the cleaned-up history

```bash
git log --oneline
```

Expected output:

```
abc1234 (HEAD -> messy-feature) Add validate function skeleton
def5678 typo fix
ghi9012 WIP debugging
f6a7b8c Add goodbye function
g7h8i9k Add hello function
h8i9j0k Initial commit
```

The 5 messy commits have become 3 clean commits. The "asdf" and "Actually implement calculate" messages are gone — their changes were squashed into the commits above them.

```
  Before (6 commits):                                   After (3 commits):
  a1b2c3d Add validate stub                             abc1234 Add validate function skeleton
  b2c3d4e Actually implement calculate    ────▶         def5678 typo fix (includes "Actually implement calculate")
  c3d4e5f typo fix                                      ghi9012 WIP debugging (includes "asdf")
  d4e5f6a asdf
  e5f6a7b WIP debugging
```

**Checkpoint:** `git diff main..HEAD` shows the total diff of all feature changes — identical to before the rebase. The content is the same; only the commit structure changed.

### Step 7: Reorder commits (practice)

Create a fresh scenario to practice reordering:

```bash
git switch main
git switch -c reorder-practice
echo "feature A part 1" > a.txt && git add a.txt && git commit -m "A1"
echo "feature B part 1" > b.txt && git add b.txt && git commit -m "B1"
echo "feature A part 2" >> a.txt && git add a.txt && git commit -m "A2"
echo "feature B part 2" >> b.txt && git add b.txt && git commit -m "B2"
```

Start interactive rebase:

```bash
git rebase -i HEAD~4
```

In the editor, reorder the commits so the A commits come first, then the B commits:

```
pick <hash> A1
pick <hash> A2
pick <hash> B1
pick <hash> B2
```

(Simply move the lines in the editor so A1 and A2 are consecutive, then B1 and B2.)

Save and close. Verify:

```bash
git log --oneline
```

Expected output — A commits appear first, then B commits.

```
abc1234 B2
def5678 B1
ghi9012 A2
jkl3456 A1
```

**Under the hood:** Reordering works because each commit is a full snapshot of the changes. Git applies the diffs in the new order. If A2 depends on A1 but is placed before A1, Git will detect a conflict and pause for resolution.

### Step 8: Drop a commit

Still on `reorder-practice`, remove the middle commit:

```bash
git rebase -i HEAD~4
```

Change the todo list to drop one of the commits:

```
pick <hash> A1
drop <hash> A2
pick <hash> B1
pick <hash> B2
```

Save and close. Verify:

```bash
git log --oneline
```

The dropped commit's changes are gone from history. The file `a.txt` only contains "feature A part 1" — the second line "feature A part 2" is not present.

## Common Errors at This Stage

### Error 1: `error: cannot 'squash' without a previous commit`

You tried to squash a commit but it is the first commit in the rebase range (or the first commit in the repository). Squash merges a commit into the one before it — there is no previous commit.

**Fix:** If it is the first commit in the range but not the repo's root, expand the range (e.g., use `HEAD~N+1`). If it is the root commit, use `git rebase -i --root` instead.

### Error 2: After squashing, the file content is wrong

You squashed commits that touched the same area of a file, and Git could not apply the combined diff cleanly. Git paused the rebase with a conflict marker.

**Fix:** Resolve the conflict like any merge conflict: edit the file, remove markers, `git add`, then `git rebase --continue`. The combined commit will contain the resolved content.

### Error 3: `git rebase --edit-todo` does not do anything

You ran `git rebase --edit-todo` but the editor opened with an empty file or the original todo unchanged. This happens when you are not in the middle of an interactive rebase.

**Fix:** `--edit-todo` only works during a paused rebase (e.g., after an `edit` or conflict pause). If you want to restart the rebase plan, first abort with `git rebase --abort`, then re-run `git rebase -i`.

### Error 4: After interactive rebase, the branch cannot be pushed

```
! [rejected]        messy-feature -> messy-feature (non-fast-forward)
```

You rebased commits that were already on the remote. The remote still has the old commit hashes, and your local has new ones.

**Fix:** If you are the only person using this branch, use `git push --force-with-lease`. If others have pulled the old commits, coordinate with them first — they will need to rebase their work on top of the rewritten branch.

## What You Just Learned

- `git rebase -i HEAD~N` opens a todo list of the last N commits
- Actions: `pick` (keep), `reword` (change message), `squash` (combine + merge messages), `fixup` (combine + discard message), `drop` (remove)
- Reorder todo lines to change commit order — Git replays diffs in the new sequence
- Squash and fixup combine multiple commits into one, reducing history noise
- Dropping a commit removes its changes permanently from the branch
- The total diff of the branch is the same after rebasing — only the commit structure changes
- Never rebase commits that have been pushed to a shared branch
