Work through these exercises in a terminal. Tier 1 builds single-action muscle memory. Tier 2 combines multiple rebase actions. Tier 3 presents scenarios where you must decide how to clean up complex histories.

## Tier 1 — Beginner

### Exercise 1

**Task:** Create 4 commits on a branch. Use interactive rebase to reword the second commit's message from "second" to "second commit (updated)".

<details>
<summary>Hint</summary>
Run `git rebase -i HEAD~4`. Change `pick` to `reword` (or `r`) on the line for the second commit. Save, then edit the message in the next editor.
</details>

### Solution

```bash
mkdir -p ~/interactive-beginner-1 && cd ~/interactive-beginner-1
git init
echo "1" > f.txt && git add f.txt && git commit -m "first"
echo "2" >> f.txt && git add f.txt && git commit -m "second"
echo "3" >> f.txt && git add f.txt && git commit -m "third"
echo "4" >> f.txt && git add f.txt && git commit -m "fourth"
git rebase -i HEAD~4
# In the editor: change 'pick' to 'reword' (or 'r') on the second commit line
# In the next editor: change "second" to "second commit (updated)"
git log --oneline
```

The second commit message now reads "second commit (updated)".

### Exercise 2

**Task:** Create 3 commits where the third is a small fixup. Use interactive rebase to squash the third into the first (not the second) by reordering the todo list.

<details>
<summary>Hint</summary>
Reorder the todo lines so the fixup commit comes right after the first commit, then use `fixup` (or `f`).
</details>

### Solution

```bash
mkdir -p ~/interactive-beginner-2 && cd ~/interactive-beginner-2
git init
echo "base content" > main.py && git add main.py && git commit -m "Add main module"
echo "print('hi')" >> main.py && git add main.py && git commit -m "Add greeting"
echo "print('hello world')" >> main.py && git add main.py && git commit -m "minor fix"
git rebase -i HEAD~3
# Reorder: move the "minor fix" line right after "Add main module", change to 'fixup'
git log --oneline
```

The log shows 2 commits: "Add main module" (includes the fixup content) and "Add greeting".

### Exercise 3

**Task:** Create 5 commits. Drop the third commit using interactive rebase. Verify the dropped commit's changes are not in the final file.

<details>
<summary>Hint</summary>
Use `git rebase -i HEAD~5`. Change `pick` to `drop` (or `d`) on the third commit line.
</details>

### Solution

```bash
mkdir -p ~/interactive-beginner-3 && cd ~/interactive-beginner-3
git init
echo "1" > f.txt && git add f.txt && git commit -m "C1"
echo "2" >> f.txt && git add f.txt && git commit -m "C2"
echo "TO_BE_DROPPED" >> f.txt && git add f.txt && git commit -m "C3 - drop me"
echo "4" >> f.txt && git add f.txt && git commit -m "C4"
echo "5" >> f.txt && git add f.txt && git commit -m "C5"
git rebase -i HEAD~5
# Change 'pick' to 'drop' on the C3 line
cat f.txt
```

`f.txt` contains 1, 2, 4, 5 — the "TO_BE_DROPPED" line is gone.

### Exercise 4

**Task:** Create 3 commits. Use `fixup` to squash the second commit into the first, discarding the second commit's message. Confirm the final commit message is the first commit's message.

<details>
<summary>Hint</summary>
Use `git rebase -i HEAD~3`. Change `pick` to `fixup` (or `f`) on the second commit line.
</details>

### Solution

```bash
mkdir -p ~/interactive-beginner-4 && cd ~/interactive-beginner-4

echo "main work" > f.txt && git add f.txt && git commit -m "Main work"
echo "fix" >> f.txt && git add f.txt && git commit -m "Quick fix"
echo "more" >> f.txt && git add f.txt && git commit -m "More work"

# Change 'pick' to 'fixup' on the "Quick fix" line
git log --oneline
```

The log shows 2 commits. The first commit message is "Main work" — "Quick fix" was absorbed without its message appearing.

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a branch with 5 commits. Use interactive rebase to:
1. Squash commits 2 and 3 together (keep both messages)
2. Reword commit 4
3. Drop commit 5
Do this in a single rebase session.

<details>
<summary>Hint</summary>
Run `git rebase -i HEAD~5`. Apply `squash` to commit 3 (merges into commit 2), `reword` to commit 4, `drop` to commit 5. Save once to apply the whole plan.
</details>

### Solution

```bash
mkdir -p ~/interactive-inter-1 && cd ~/interactive-inter-1
git init
for i in 1 2 3 4 5; do echo "$i" > f.txt && git add f.txt && git commit -m "C$i"; done
git rebase -i HEAD~5
# todo: pick C1, pick C2, squash C3, reword C4, drop C5
# In squash editor: keep both C2 and C3 messages
# In reword editor: change C4 message to "Updated C4"
git log --oneline
```

Expected result: 3 commits (C1, C2+squashed C3, C4 reworded). C5 is gone.

### Exercise 2

**Task:** Create 4 commits where the second commit adds a file that the fourth commit depends on. Reorder the commits so the file is added earlier. The rebase should succeed without conflicts because the dependency is satisfied.

<details>
<summary>Hint</summary>
Create commits: C1 (file A), C2 (file B), C3 (edit A), C4 (edit B). The dependency is C4 on C2. If you move C4 before C2, it will conflict. Move C2 before C3 instead.
</details>

### Solution

```bash
mkdir -p ~/interactive-inter-2 && cd ~/interactive-inter-2
git init
echo "A1" > a.txt && git add a.txt && git commit -m "Create A"
echo "B1" > b.txt && git add b.txt && git commit -m "Create B"
echo "A2" >> a.txt && git add a.txt && git commit -m "Update A"
echo "B2" >> b.txt && git add b.txt && git commit -m "Update B"
git rebase -i HEAD~4
# Reorder so B commits are first:
# pick <C2> Create B
# pick <C4> Update B
# pick <C1> Create A
# pick <C3> Update A
git log --oneline
```

The history now shows B-related commits first, then A-related commits. No conflicts because dependencies are satisfied.

### Exercise 3

**Task:** Create a branch with 3 commits. Use `edit` on the second commit to split it into two separate commits. The final history should have 4 commits instead of 3.

<details>
<summary>Hint</summary>
Use `git rebase -i HEAD~3` with `edit` on the second commit. When Git pauses, `git reset HEAD~1` to unstage the commit's changes, then `git add` and `git commit` twice to create two commits, then `git rebase --continue`.
</details>

### Solution

```bash
mkdir -p ~/interactive-inter-3 && cd ~/interactive-inter-3
git init
echo "C1" > f.txt && git add f.txt && git commit -m "C1"
echo "C2a" > f.txt && echo "C2b" >> f.txt && git add f.txt && git commit -m "C2 (to split)"
echo "C3" >> f.txt && git add f.txt && git commit -m "C3"
git rebase -i HEAD~3
# Change 'pick' to 'edit' on C2 line
# Git pauses at C2
git reset HEAD~1
# Now C2a and C2b changes are unstaged
echo "C2a" > f.txt && git add f.txt && git commit -m "C2 part A"
echo "C2b" >> f.txt && git add f.txt && git commit -m "C2 part B"
git rebase --continue
git log --oneline
```

4 commits: C1, C2 part A, C2 part B, C3.

## Tier 3 — Advanced

### Exercise 1: Clean up a real-world messy pull request

**Scenario:** You have a branch with 12 commits. Among them:
- 2 "fix typo" commits that should be squashed into the original commits they fix
- 1 "WIP" commit that should be dropped entirely
- 1 commit with a debug API key that must be removed from history (not just the file — the commit must not exist)
- Several commits with messages like "update", "fix", "asdf" that need descriptive messages

**Task:** Do not actually create 12 commits. Write the interactive rebase todo list you would use, showing each commit's action and the reasoning. Then create a reduced version (4 commits with similar issues) and execute it.

<details>
<summary>Hint</summary>
For removing a secret, use `drop` on the commit, or `edit` to selectively remove the secret line from the commit's content. If the secret was added in one commit and modified in another, `edit` the first commit that introduced it.
</details>

### Solution — one valid approach

Reduced scenario:

```bash
mkdir -p ~/interactive-adv-1 && cd ~/interactive-adv-1
git init
echo "start" > config.py && git add config.py && git commit -m "Initial config"
echo "SECRET=abc123" >> config.py && git add config.py && git commit -m "Add API config"
echo "print('debug')" > debug.py && git add debug.py && git commit -m "WIP debugging"
echo "SECRET=xyz789" >> config.py && git add config.py && git commit -m "Update secret"
echo "real_feature" > feature.py && git add feature.py && git commit -m "asdf"
echo "typo fix" > fix.py && git add fix.py && git commit -m "fix typo"
```

The clean-up plan:

```
pick <hash> Initial config
edit <hash> Add API config          # Remove the secret line, keep the config file
fixup <hash> Update secret          # Absorb the secret update into config (or drop)
drop <hash> WIP debugging           # Remove debug file entirely
reword <hash> asdf                  # Change to "Add real_feature module"
fixup <hash> fix typo               # Squash typo fix into real_feature
```

Execute:

```bash
git rebase -i HEAD~6
# Set the actions as above
# When Git pauses at "Add API config":
sed -i '/SECRET/d' config.py
git add config.py
git rebase --continue
# When the reword editor opens, change "asdf" to "Add real_feature module"
git log --oneline
git diff main..HEAD
```

No commit contains the secret. The debug file never existed in the rewritten history. The final 3 commits tell a clean story.

**Alternative approach:** Use `git filter-branch` or `git filter-repo` for large-scale secret removal across many commits. Interactive rebase works for small, targeted cleanup where you know exactly which commits to edit.

### Exercise 2: Rebase a branch that has merge commits

**Scenario:** Your feature branch includes merge commits from pulling `main` into it several times. These merge commits complicate the history. You want to remove the merge bubbles and create a clean linear history.

**Task:** Create a branch with 3 feature commits and 2 merge commits (simulate pulling `main` twice). Use interactive rebase to linearize the history — the final branch should have exactly 3 commits, all on top of `main`, with no merge commits.

<details>
<summary>Hint</summary>
By default, `git rebase` drops merge commits and replays only non-merge commits. Use `git rebase -i main` (not `HEAD~N`) to rebase the entire branch onto `main`. Git automatically skips merge commits.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/interactive-adv-2 && cd ~/interactive-adv-2
git init
echo "main1" > f.txt && git add f.txt && git commit -m "Main 1"
echo "main2" >> f.txt && git add f.txt && git commit -m "Main 2"
git switch -c feature
echo "feat1" >> f.txt && git add f.txt && git commit -m "Feature 1"
git switch main
echo "main3" >> f.txt && git add f.txt && git commit -m "Main 3"
git switch feature
git merge main -m "Merge main into feature"
echo "feat2" >> f.txt && git add f.txt && git commit -m "Feature 2"
git switch main
echo "main4" >> f.txt && git add f.txt && git commit -m "Main 4"
git switch feature
git merge main -m "Merge main again"
echo "feat3" >> f.txt && git add f.txt && git commit -m "Feature 3"
git log --oneline --graph --all
```

The graph shows merge bubbles. Now linearize:

```bash
git rebase -i main
# Git presents only the 3 feature commits (merge commits are automatically dropped)
# Optionally squash or reword
git log --oneline --graph
```

The graph is linear. 3 feature commits on top of "Main 4". No merge commits.

**Why this works:** Git's rebase operation automatically skips commits with more than one parent (merge commits). It only replays commits that have exactly one parent.

### Exercise 3: History archaeology — recover changes from a dropped commit

**Scenario:** During an interactive rebase, you dropped a commit that contained important changes. You did not realize the mistake until after the rebase completed. The commit's changes are gone from the branch, and the original commit is no longer reachable from any branch.

**Task:** Create this scenario: drop a commit during rebase, then recover its changes using `git reflog` + `git cherry-pick`. Do not recreate the branch from scratch.

<details>
<summary>Hint</summary>
After the rebase, `git reflog` shows the old HEAD positions. Find the entry showing the pre-rebase state, `git cherry-pick <hash>` the dropped commit's hash from the reflog, then commit the recovered changes.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/interactive-adv-3 && cd ~/interactive-adv-3
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
echo "important work" >> f.txt && git add f.txt && git commit -m "Important work (do not lose)"
echo "final touch" >> f.txt && git add f.txt && git commit -m "Final touch"
git rebase -i HEAD~2
# Drop "Important work" (accidentally)
# Verify it is gone:
cat f.txt
# "important work" is missing!
```

Recover:

```bash
# Find the dropped commit's hash
git reflog
# Look for the entry: HEAD@{1} rebase -i (finish): HEAD~2
# Actually look for the commit before the rebase started
# The dropped commit's hash is visible in reflog
git cherry-pick <hash-of-important-work>
```

If the cherry-pick succeeds, the changes are back. If it conflicts (because "final touch" was already applied), resolve and continue.

**Alternative approach:** Create a branch at the reflog entry pointing to the pre-rebase HEAD, then use `git diff` to extract the changes as a patch and apply them manually: `git branch rescue HEAD@{1}` then `git diff rescue..HEAD > patch.diff && git apply patch.diff`.
