You shipped version 1.0 two months ago. A user reports a bug that was present in 1.0 but is now fixed in your latest code. You need to check out exactly what was shipped, not the current development branch. You also need to create version 1.1 with a hotfix, and you need anyone who clones the repo to be able to find the exact code that was deployed. Branches move — tags are permanent markers.

## The Concept

A tag is a fixed, non-moving reference to a specific commit. Unlike branches, tags do not change when new commits are made. Lightweight tags are just a pointer (like a branch that never moves). Annotated tags store the tagger name, date, message, and can be cryptographically signed.

Think of branches as sticky notes you move around as you work, and tags as permanent ink labels on a filing cabinet. Once you label a folder "v1.0", that label stays on that exact folder forever — even as you add new folders to the drawer. Anyone who looks for v1.0 finds exactly what was shipped.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git tag` | List all tags. | `git tag` |
| `git tag -l "<pattern>"` | List tags matching a pattern. | `git tag -l "v1.*"` |
| `git tag <name>` | Create a lightweight tag at the current commit. | `git tag v1.0` |
| `git tag <name> <commit>` | Create a tag at a specific commit. | `git tag v1.0 a1b2c3d` |
| `git tag -a <name> -m "<msg>"` | Create an annotated tag with a message. | `git tag -a v1.0 -m "Release v1.0"` |
| `git tag -s <name> -m "<msg>"` | Create a signed annotated tag (GPG-signed). | `git tag -s v1.0 -m "Release v1.0"` |
| `git tag -d <name>` | Delete a local tag. | `git tag -d v1.0` |
| `git push origin <tag>` | Push a specific tag to the remote. | `git push origin v1.0` |
| `git push origin --tags` | Push all tags to the remote. | `git push origin --tags` |
| `git push origin --delete <tag>` | Delete a tag on the remote. | `git push origin --delete v1.0` |
| `git checkout <tag>` | Check out a tag in detached HEAD state. | `git checkout v1.0` |
| `git describe` | Describe the current commit relative to the nearest tag. | `git describe` |
| `git tag --contains <commit>` | List tags that contain a specific commit. | `git tag --contains a1b2c3d` |

**Windows note:** All tag commands work identically. For GPG signing on Windows, you need to install Gpg4win and configure Git: `git config --global gpg.program "C:\Program Files (x86)\GnuPG\bin\gpg.exe"`.

```
│                                         ▼                 │

│                                                           │

```

## How It Actually Works — Step by Step

### Step 1: Create a lightweight tag

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2))  ← main
                                ← HEAD
```

Running `git tag v1.0` creates a file at `.git/refs/tags/v1.0` containing the hash of `C2`. This is a 41-byte file — exactly the same structure as a branch reference, but Git never updates it automatically.

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2))  ← main, v1.0
                                ← HEAD
```

### Step 2: Create an annotated tag

Running `git tag -a v1.1 -m "Release v1.1"` does more work:

1. Git creates a tag object in `.git/objects/` containing the tagger name, email, date, message, and the hash of the target commit
2. The tag object gets its own hash (like a commit does)
3. Git writes the tag object's hash to `.git/refs/tags/v1.1`

```
  tag object v1.1 {
    object: C3
    type: commit
    tag: v1.1
    tagger: Your Name <email> 2026-07-03
    message: Release v1.1
  }
```

You can inspect the tag object with `git cat-file -p v1.1`.

### Step 3: List and search tags

```
  git tag -l "v1.*"
  v1.0
  v1.1
```

Tags are listed alphabetically. If you use semantic versioning (v1.0.0, v1.1.0, v1.10.0), note that `v1.10.0` sorts between `v1.1.0` and `v1.2.0` — not after `v1.9.0`. Use `git tag --sort=version:refname` for correct semver sorting.

### Step 4: Check out a tag

```
  git checkout v1.0
```

Git checks out the commit `C2` and detaches HEAD:

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2))  ← main, v1.0
                                ← HEAD (detached)
```

You are now in detached HEAD state. Any commits you make here will not belong to any branch and will be lost when you switch away (unless you create a branch first).

### Step 5: Push tags to a remote

```
  git push origin v1.0
```

The remote receives the tag and stores it in `.git/refs/tags/v1.0`. Anyone who clones the repo after this will get the tag. `git push origin --tags` pushes all local tags that are not already on the remote.

### Step 6: Use `git describe` to identify a commit

```
  git describe
  v1.1-3-gabc1234
```

This output means: the nearest tag reachable from the current commit is `v1.1`, and the current commit is 3 commits ahead of that tag, with abbreviated hash `abc1234`. The `g` prefix stands for "git" — it distinguishes the hash from similar-looking numbers.

This is how many build systems generate version strings for pre-release builds (e.g., `1.1.0-alpha.3+abc1234`).

## When to Use This / When Not To

**Use tags when:**
- You are shipping a release — tag the exact commit that was deployed
- You need a permanent, non-moving reference for historical or audit purposes
- You want to mark a specific commit for cherry-picking or bisect boundaries
- You are creating a release artifact and want to embed the version in the build

**Do not use tags when:**
- You need to update the reference — tags are permanent. Delete and re-create if you made a mistake, but never reuse a tag name for a different commit
- You want to track ongoing work — use a branch instead
- You are marking an internal-only commit that does not correspond to a release or milestone
- You are using the tag name as a variable — if you need "latest stable" that moves, use a branch named `stable` and update it with `git branch -f stable <commit>`

**Lightweight vs. annotated:** Use annotated tags for releases (they store the tagger, date, and message). Use lightweight tags for personal bookmarks (e.g., "this is where I left off debugging"). The Git documentation recommends annotated tags for public releases because they store the full metadata.
