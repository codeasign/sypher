You will create a GitHub repository, add a workflow file, trigger it by pushing code, and watch the run execute in the Actions tab. Every step works on Windows, macOS, and Linux — the only difference is that you need a web browser to see the GitHub Actions UI.

## Setup

1. A GitHub account (sign up at [github.com](https://github.com) if you do not have one).
2. Git installed and configured (`git --version` should return a version number).
3. A terminal (PowerShell on Windows, Terminal on macOS, or any shell on Linux).

Create a new repository on GitHub (do not initialize with a README):

```bash
# Log in to GitHub from the CLI
gh auth login
```

```bash
# Create a new repository
gh repo create actions-fundamentals-practice --public --clone
```

```bash
# Move into the repository directory
cd actions-fundamentals-practice
```

Create a simple Python script to test:

```bash
# Create a test file
echo "print('Hello, GitHub Actions!')" > test.py
```

```bash
# Stage and commit
git add test.py
git commit -m "Add initial test script"
```

## The Walkthrough

### Step 1: Create the workflows directory

```bash
mkdir -p .github/workflows
```

GitHub looks for workflow files in `.github/workflows/` at the root of your repository. Any `.yml` file in that directory is a valid workflow.

### Step 2: Write your first workflow

Create `.github/workflows/ci.yml` with the following content:

```yaml
name: CI Pipeline
on:
  push:
    branches: [ main ]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Run the script
        run: python test.py
      - name: Verify exit code
        run: |
          echo "Script ran successfully"
          echo "Run ID: ${{ github.run_id }}"
```

**Checkpoint:** Verify the file exists and is valid YAML:

```bash
cat .github/workflows/ci.yml
```

Expected output should show the full YAML content you just wrote.

### Step 3: Commit and push the workflow

```bash
git add .github/workflows/ci.yml
git commit -m "Add first CI workflow"
```

```bash
git push origin main
```

When you push, GitHub detects the workflow file. The `on: push: branches: [main]` trigger fires, and a new workflow run begins.

### Step 4: Watch the run in the Actions tab

Open your browser and navigate to:

```
https://github.com/<your-username>/actions-fundamentals-practice/actions
```

You should see a running workflow named "CI Pipeline." Click on it to see the live log output.

**Expected log output (abbreviated):**

```
Run actions/checkout@v4
  Syncing repository: your-username/actions-fundamentals-practice
  ...

Run actions/setup-python@v5
  Successfully set up Python 3.11.xx

Run python test.py
  Hello, GitHub Actions!

Run echo "Script ran successfully"
  Script ran successfully
  Run ID: 1234567890
```

### Step 5: Trigger the workflow manually

Go to the Actions tab, click the "CI Pipeline" workflow, and click the "Run workflow" dropdown. Select "Branch: main" and click the green "Run workflow" button.

This triggers the `workflow_dispatch` event. A new run starts even though you did not push any code.

### Step 6: Add a failing step and see what happens

Edit `.github/workflows/ci.yml` to add a step that fails:

```yaml
name: CI Pipeline
on:
  push:
    branches: [ main ]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Run the script
        run: python test.py
      - name: This step will fail
        run: exit 1
      - name: This step will never run
        run: echo "I never execute"
```

```bash
git add .github/workflows/ci.yml
git commit -m "Add a failing step to see failure behavior"
```

```bash
git push origin main
```

In the Actions tab, you will see the run fail at the "This step will fail" step. The "This step will never run" step is skipped entirely — GitHub stops the job at the first non-zero exit code.

### Step 7: Use continue-on-error

Replace the failing step with one that uses `continue-on-error`:

```yaml
name: CI Pipeline
on:
  push:
    branches: [ main ]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Run the script
        run: python test.py
      - name: This step fails but does not block
        continue-on-error: true
        run: exit 1
      - name: This step runs anyway
        run: echo "I still execute despite the failure"
```

```bash
git add .github/workflows/ci.yml
git commit -m "Demonstrate continue-on-error"
```

```bash
git push origin main
```

Now the workflow succeeds even though a step failed. The job is marked as green (success) because `continue-on-error: true` tells GitHub that a failure in this step is acceptable.

### Step 8: View the workflow run summary

In the Actions tab, click on the most recent successful run. Scroll down to see:

- **Job status**: green checkmark or red X
- **Step duration**: how long each step took
- **Annotations**: any warnings or errors GitHub attached to the run
- **Artifacts**: if any were uploaded, they appear here

## Common Errors at This Stage

### Error 1: "No workflow runs"
**Message:** The Actions tab shows "No workflow runs yet."

**Why:** Either the workflow file is not in the correct directory, or it has a `.yaml` extension that is not being picked up (GitHub recognizes both `.yml` and `.yaml`, but double-check the file is in `.github/workflows/`).

**Fix:** Verify the file path:
```bash
ls .github/workflows/
```
Ensure the file is there. If not, move it:
```bash
mv your-file.yml .github/workflows/ci.yml
```

### Error 2: "Invalid workflow file"
**Message:** "YAML syntax error — mapping values are not allowed here" or similar.

**Why:** YAML is sensitive to indentation. A single misplaced space causes the entire file to be invalid.

**Fix:** Check your indentation. Every level of nesting should use exactly 2 spaces:
```yaml
# Correct
jobs:
  test:        # 2 spaces indent
    runs-on:   # 4 spaces
      ubuntu-latest  # 6 spaces

# Wrong
jobs:
  test:
  runs-on: ubuntu-latest  # Wrong level — this is a sibling of test, not a child
```

### Error 3: "Process completed with exit code 1"
**Message:** The step log ends with "Error: Process completed with exit code 1."

**Why:** A command in a `run:` step returned a non-zero exit code. This is the expected behavior — GitHub Actions treats any non-zero exit as a failure.

**Fix:** Check the step's log output above the error line to see which command actually failed. Add `set -x` at the start of a multi-line `run:` block to print each command before executing it:
```yaml
- name: Debug script
  run: |
    set -x
    python test.py
    echo "This line runs"
```

### Error 4: "Resource not accessible by integration"
**Message:** The step log shows "Resource not accessible by integration" or a 403 error.

**Why:** The default `GITHUB_TOKEN` does not have permission to access certain resources (like writing to the repository or creating releases).

**Fix:** Add explicit permissions to the workflow:
```yaml
permissions:
  contents: write
  issues: read
```

## What You Just Learned

- A workflow is a YAML file in `.github/workflows/` that defines an automated process
- Workflows are triggered by events (push, workflow_dispatch, etc.)
- Each workflow contains one or more jobs, and each job runs on a fresh runner
- Jobs contain ordered steps that execute sequentially
- A step that exits with a non-zero code fails the job by default
- `continue-on-error: true` lets you mark a step's failure as acceptable
- You can trigger workflows both automatically (push) and manually (workflow_dispatch)
