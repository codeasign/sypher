## Tier 1 — Beginner

### Exercise 1: Create a workflow with two jobs

**Task:** Create a workflow file named `multi-job.yml` that defines two jobs — `build` and `test`. The `test` job should depend on the `build` job (use `needs`). Both jobs should run on `ubuntu-latest`. Each job should have a single step that prints the job name.

**Hint:** The `needs` key on a job tells GitHub to wait until the listed jobs complete before starting this one.

<details>
<summary>Solution</summary>

File: `.github/workflows/multi-job.yml`

```yaml
name: Multi-Job Workflow
on:
  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building the project..."
  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Testing the project..."
```

**Expected outcome:** When you manually trigger this workflow, the `build` job runs first. After it completes, the `test` job starts. If you remove the `needs: build` line, both jobs run in parallel.
</details>

### Exercise 2: Use a matrix strategy

**Task:** Create a workflow that runs the same job across three Python versions: `3.10`, `3.11`, and `3.12`. Use `actions/setup-python@v5` to set up each version. The job should run `python --version` to verify.

**Hint:** The `strategy.matrix` key lets you define variables that produce a job variant for each combination.

<details>
<summary>Solution</summary>

File: `.github/workflows/matrix.yml`

```yaml
name: Matrix Build
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.10', '3.11', '3.12']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: python --version
```

**Expected outcome:** Three parallel jobs appear in the Actions tab, each running a different Python version. Each job logs something like `Python 3.11.x`.
</details>

### Exercise 3: Add a timeout

**Task:** Modify a workflow so that a job automatically stops if it runs for more than 2 minutes. Add a step that sleeps for 5 minutes to verify the timeout works.

**Hint:** Add `timeout-minutes` at the job level or step level.

<details>
<summary>Solution</summary>

File: `.github/workflows/timeout-test.yml`

```yaml
name: Timeout Test
on: workflow_dispatch

jobs:
  slow-job:
    runs-on: ubuntu-latest
    timeout-minutes: 2
    steps:
      - run: echo "Starting long-running step..."
      - run: sleep 300
      - run: echo "This line never runs"
```

**Expected outcome:** After 2 minutes, the job is cancelled by GitHub. The log shows "Job was cancelled because it exceeded the maximum time of 2 minutes."
</details>

## Tier 2 — Intermediate

### Exercise 4: Conditional job execution

**Task:** Create a workflow with a job that only runs when the push event comes from a branch named `develop`. The job should print the branch name using the `github.ref_name` context variable.

**Hint:** Use a job-level `if` condition. The `github.ref_name` context variable holds the branch or tag name.

<details>
<summary>Solution</summary>

File: `.github/workflows/conditional.yml`

```yaml
name: Conditional Job
on:
  push:
    branches:
      - develop

jobs:
  deploy-develop:
    runs-on: ubuntu-latest
    if: github.ref_name == 'develop'
    steps:
      - run: echo "Deploying from branch ${{ github.ref_name }}"
```

**Expected outcome:** Pushing to `develop` triggers the workflow and the job runs. Pushing to any other branch does not trigger the workflow at all (because the `on.push.branches` filter excludes them).
</details>

### Exercise 5: Reusable workflow reference

**Task:** You have a workflow that checks out code and runs tests. Extract the test steps into a separate reusable workflow file and call it from your main workflow.

**Hint:** A reusable workflow must have `on: workflow_call` at the top level. Call it from another workflow with `uses: owner/repo/.github/workflows/reusable.yml@branch`.

<details>
<summary>Solution</summary>

File: `.github/workflows/reusable-test.yml`

```yaml
name: Reusable Test Suite
on:
  workflow_call:

jobs:
  run-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running tests..."
      - run: echo "All tests passed"
```

File: `.github/workflows/main.yml`

```yaml
name: Main Pipeline
on: [push]

jobs:
  call-tests:
    uses: ./.github/workflows/reusable-test.yml
  deploy:
    needs: call-tests
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."
```

**Expected outcome:** The main workflow calls the reusable workflow. The reusable workflow's jobs appear as direct children of the caller's job graph — you see `call-tests` and `deploy` as sibling jobs in the Actions tab.
</details>

### Exercise 6: Cancel duplicate runs

**Task:** Configure concurrency so that if you push twice in quick succession, the previous run is cancelled and only the latest push runs.

**Hint:** Use the `concurrency` top-level key with a group name and `cancel-in-progress: true`.

<details>
<summary>Solution</summary>

File: `.github/workflows/cancel-dupes.yml`

```yaml
name: Cancel Duplicates
on: [push]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building for ${{ github.ref }}"
      - run: sleep 30
      - run: echo "Build complete"
```

**Expected outcome:** Push commit A. While it is still running, push commit B. Commit A's workflow run is automatically cancelled (status: "Cancelled"), and commit B's run proceeds.
</details>

## Tier 3 — Advanced

### Exercise 7: Debug a broken workflow

**Scenario:** Your teammate created this workflow, but it fails every time. Identify and fix the issues:

```yaml
name: Broken Workflow
on: push

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install flake8
        run: pip install flake8
      - name: Lint
        run: flake8 .
  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: pytest
      - name: Generate report
        run: echo "## Test Results" > report.md
      - uses: actions/upload-artifact@v4
        with
          name: test-report
          path: report.md
```

<details>
<summary>Solution</summary>

There are at least 3 issues:

1. **Missing colon** on line 21: `with` should be `with:` — YAML requires the colon after the key name.
2. **Missing Python setup**: `pip install flake8` assumes Python is already installed. It is on `ubuntu-latest`, but the better practice is to use `actions/setup-python` first.
3. **No workflow_dispatch trigger**: You cannot test this manually — you must push every time.

Fixed version:

```yaml
name: Fixed Workflow
on: [push, workflow_dispatch]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install flake8
        run: pip install flake8
      - name: Lint
        run: flake8 .
  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install pytest
        run: pip install pytest
      - name: Run tests
        run: pytest
      - name: Generate report
        run: echo "## Test Results" > report.md
      - uses: actions/upload-artifact@v4
        with:
          name: test-report
          path: report.md
```

Other valid approaches include pinning exact Python patch versions in the matrix or using a Docker container with pre-installed tools.
</details>

### Exercise 8: Design a PR quality gate

**Scenario:** Your team wants a workflow that enforces quality checks on every pull request to `main`:

- Run linting (any linter for your language of choice)
- Run unit tests
- Check that test coverage is at least 80%
- Only allow merging if all checks pass
- Post a comment on the PR with the coverage percentage

**Constraints:** You cannot use third-party actions beyond `actions/checkout@v4`, `actions/setup-python@v5`, and `actions/github-script@v7`. Design the workflow structure — you do not need to write the full implementation of the coverage check.

<details>
<summary>Solution</summary>

```yaml
name: PR Quality Gate
on:
  pull_request:
    branches: [ main ]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install flake8
      - run: flake8 .

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install pytest pytest-cov
      - name: Run tests with coverage
        run: pytest --cov=src --cov-report=term-missing --cov-fail-under=80
      - name: Post coverage comment
        uses: actions/github-script@v7
        with:
          script: |
            const { execSync } = require('child_process');
            const output = execSync('pytest --cov=src --cov-report=term-missing').toString();
            const match = output.match(/TOTAL\s+\d+\s+\d+\s+(\d+)%/);
            const coverage = match ? match[1] : 'unknown';
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `Coverage: ${coverage}%`
            });
```

Other valid approaches include using a coverage badge service, running the PR comment as a separate job with `if: success()`, or using a matrix for multiple Python versions. The key design requirement is that `--cov-fail-under` blocks merging by failing the step if coverage drops below 80%.
</details>

### Exercise 9: Optimize for speed

**Scenario:** Your workflow takes 12 minutes to run — 3 minutes of dependency installation and 9 minutes of tests across 3 operating systems. You need to get it under 4 minutes. Describe at least 2 strategies and implement one with actual YAML.

<details>
<summary>Solution</summary>

Strategy 1 — **Parallelize across a matrix**: Run tests on all 3 OSes simultaneously instead of sequentially:

```yaml
name: Fast Parallel Build
on: [push]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest
```

This brings wall-clock time down to the slowest single OS (~3 min deps + ~4 min tests = ~7 min) since all three run in parallel.

Strategy 2 — **Cache dependencies**: Use `actions/cache@v4` to store installed packages between runs:

```yaml
name: Cached Dependencies
on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-${{ hashFiles('requirements.txt') }}
          restore-keys: |
            pip-
      - run: pip install -r requirements.txt
      - run: pytest
```

Combined effect: cache hits in ~10 seconds instead of 3 minutes of installation, and parallel OS matrix runs finish in ~4 minutes (the longest single-OS test suite). This is the strongest approach — use both together.

Other valid strategies: splitting tests by directory into separate jobs, using a more powerful runner (`ubuntu-latest-4core`), or reducing the test matrix to only the OSes your users actually run on.
</details>
