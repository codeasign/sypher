You will create a repository, add a CI workflow, configure required status checks via the GitHub API, and observe how the merge gate behaves when checks pass, fail, and fall out of date. By the end you will understand the full lifecycle of a gated merge.

## Setup

Create a new repository for this practice session:

```bash
gh repo create status-checks-practice --public --clone
```

```bash
# Create the workflows directory
mkdir -p .github/workflows
```

```bash
# Create a simple Node.js project with a test
cat > package.json << 'EOF'
{
  "name": "status-checks-practice",
  "version": "1.0.0",
  "scripts": {
    "test": "node test.js"
  }
}
EOF
```

```bash
# Create a test file
cat > test.js << 'EOF'
const assert = require('assert');

function add(a, b) {
  return a + b;
}

// Basic arithmetic test
assert.strictEqual(add(2, 3), 5, '2 + 3 should equal 5');
assert.strictEqual(add(-1, 1), 0, '-1 + 1 should equal 0');
assert.strictEqual(add(0, 0), 0, '0 + 0 should equal 0');

console.log('All tests passed!');
EOF
```

```bash
# Create a workflow file
cat > .github/workflows/ci.yml << 'EOF'
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: node test.js

  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Lint passed (placeholder)"
```

```bash
# Commit and push
git add -A
git commit -m "Initial commit: project setup with CI workflow"
git push origin main
```

## Step 1: Enable Required Status Checks

Use the GitHub API to set up branch protection on `main` with two required status checks: `"Unit Tests"` and `"Lint"`.

```bash
# Enable required status checks on main
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Replace `:owner` and `:repo` with your GitHub username and `status-checks-practice`.

```bash
# Verify the configuration
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks'
```

You should see output showing the `contexts` array with `"Unit Tests"` and `"Lint"`, plus `strict: false`.

## Step 2: Create a Branch with a Breaking Change

Now create a branch and introduce a test failure to see the gate in action.

```bash
# Create and switch to a new branch
git checkout -b feature/broken-test
```

```bash
# Introduce a failing test
cat > test.js << 'EOF'
const assert = require('assert');

function add(a, b) {
  return a + b;
}

assert.strictEqual(add(2, 3), 5, '2 + 3 should equal 5');
assert.strictEqual(add(-1, 1), 0, '-1 + 1 should equal 0');
assert.strictEqual(add(0, 0), 1, '0 + 0 should equal 1'); // Intentional bug

console.log('All tests passed!');
EOF
```

```bash
# Commit and push the broken branch
git add -A
git commit -m "Introduce intentional test failure"
git push origin feature/broken-test
```

## Step 3: Open a Pull Request and Observe the Gate

```bash
# Open a pull request
gh pr create \
  --base main \
  --head feature/broken-test \
  --title "Test: breaking change" \
  --body "This PR intentionally breaks the test to demonstrate required status checks."
```

```bash
# Check the PR status
gh pr view --json state,mergeable,statusCheckRollup
```

Watch the `statusCheckRollup` output. You should see:

- `"Unit Tests"` -- `"FAILURE"` or `"ERROR"` (the test assertion failed)
- `"Lint"` -- `"SUCCESS"` (the placeholder lint step passed)
- `mergeable` -- `"CONFLICTING"` or `"BLOCKED"` (the merge is blocked by the failing check)

## Step 4: Fix the Code and Push

Now fix the broken test and push the corrected code.

```bash
# Fix the intentional bug
cat > test.js << 'EOF'
const assert = require('assert');

function add(a, b) {
  return a + b;
}

assert.strictEqual(add(2, 3), 5, '2 + 3 should equal 5');
assert.strictEqual(add(-1, 1), 0, '-1 + 1 should equal 0');
assert.strictEqual(add(0, 0), 0, '0 + 0 should equal 0');

console.log('All tests passed!');
EOF
```

```bash
# Commit and push the fix
git add -A
git commit -m "Fix: correct the 0 + 0 assertion"
git push origin feature/broken-test
```

```bash
# Check the PR status again
gh pr view --json state,mergeable,statusCheckRollup
```

Now you should see both checks passing and `mergeable: "MERGEABLE"`.

```bash
# Merge the PR (now allowed)
gh pr merge --squash --delete-branch
```

## Step 5: Enable Strict Mode

Strict mode requires the PR branch to be up to date with the base branch before merging. This prevents a scenario where a PR passes checks against an older version of `main`, but `main` has since received new commits that could break the PR's changes.

```bash
# Update branch protection to strict mode
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

```bash
# Verify strict mode is enabled
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks'
```

The output should now show `"strict": true`.


```bash

gh pr create \

  --head feature/strict-test \
  --title "Test: strict mode" \
```

Wait for the checks to pass on this PR. Then advance the `main` branch independently:

```bash
# Push a new commit directly to main (simulating another PR merging first)

git pull origin main

git add -A
git commit -m "Simulate another PR merging first"
git push origin main
```

Now return to the `feature/strict-test` branch and check the PR status:

## Step 7: Update the Branch and Merge

Update the PR branch to include the latest `main` changes, then merge.

```bash
# Update the branch with the latest main
git checkout feature/strict-test
git merge main
git push origin feature/strict-test
```

Wait for the checks to re-run on the updated branch, then verify they pass:

```bash
gh pr view --json state,mergeable,statusCheckRollup
```

When both checks pass and the branch is up to date, merge:

```bash
gh pr merge --squash --delete-branch
```

## Common Errors

### "Required status check expected"

**Symptom:** The merge button says "Required status check was not found" even though the check clearly ran.

**Cause:** The required status check name in the branch protection configuration does not exactly match the check run name. The comparison is case-sensitive and must match the full string including any matrix suffixes.

**Fix:** Check the exact check run name:

```bash
# List all check runs for the latest commit on the PR branch
gh api repos/:owner/:repo/commits/HEAD/check-runs \
  --jq '.check_runs[].name'
```

Then update the protection rule with the exact names:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

### "No status check found for ref"

**Symptom:** The PR shows "Cheks are pending" indefinitely, and the check run never appears.

**Cause:** The workflow did not trigger because the commit did not match the workflow's `on:` trigger conditions. Common causes: pushing to a branch that is not covered by the `branches:` filter, or the workflow file has a syntax error.

**Fix:** Verify the workflow ran:

```bash
# List recent workflow runs for the branch
gh run list --branch feature/broken-test --limit 5
```

If no runs appear, check the workflow file for syntax errors:

```bash
# Validate the workflow file
gh api repos/:owner/:repo/actions/workflows/ci.yml \
  --jq '.state'
```

### "This branch is out of date with the base branch"

**Symptom:** The merge button is grayed out, checks all passed, but the message says the branch is out of date.

**Cause:** Strict mode (`strict: true`) is enabled, and the base branch has new commits that are not in the PR branch. The branch protection rule requires the PR branch to be up to date.

**Fix:** Update the branch from the base:

```bash
git checkout feature/your-branch
git pull origin main
git push origin feature/your-branch
```

The checks will re-run against the updated branch. After they pass, the merge button becomes available.

### "Check run conclusion missing"

**Symptom:** A check run completes but the branch protection rule does not recognize its conclusion. The check appears as "Expected -- Waiting for status to be reported."

**Cause:** The check run was created by a different GitHub App than expected, or the check run name does not match the required context exactly. This can also happen if the check run was created after the branch protection rule was last cached.

**Fix:** Re-trigger the check run evaluation by pushing a new commit, closing and reopening the PR, or updating the branch protection configuration:

```bash
# Re-apply the branch protection rule to refresh the cache
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

## What You Just Learned

- You configured required status checks on a protected branch using the GitHub API, transitioning from an unprotected branch to one that blocks merges on failing checks.

- You observed the merge gate in action: a PR with a failing test showed a grayed-out merge button, and the same PR with passing tests showed a green merge button.

- You enabled strict mode and saw how advancing the base branch while a PR is open blocks the merge until the PR branch is updated -- even when all checks previously passed.

- You diagnosed and resolved the four most common errors encountered when working with required status checks: name mismatches, missing triggers, stale branches, and missing conclusions.

- You used `gh api` to inspect check run names, workflow run status, and branch protection configuration, building a mental model of how GitHub evaluates mergeability.

- Strict mode is the closest you can get to a "guaranteed green" merge without additional tooling. It ensures that every commit on the base branch has been validated against the latest version of the codebase.
