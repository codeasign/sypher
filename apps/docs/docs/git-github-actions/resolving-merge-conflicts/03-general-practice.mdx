## Tier 1 — Beginner

### Exercise 1

**Task:** Create two branches that each add a new function to the same Python file — one adds `def square(n)` at line 10, the other adds `def cube(n)` at line 10. Merge them. Does a conflict occur? Why or why not?

<details>
<summary>Hint</summary>
Git merges changes on different lines within the same file without conflict. Both branches added a function at the same line number, so the result depends on context — if the surrounding content matched, Git auto-merges. Try it.
</details>

### Solution

```bash
mkdir -p ~/conflict-beginner-1 && cd ~/conflict-beginner-1
git init
cat > math.py << 'EOF'
def add(a, b):
    return a + b
EOF
git add math.py && git commit -m "Base"
git switch -c branch-square
echo -e '\ndef square(n):\n    return n * n' >> math.py
git add math.py && git commit -m "Add square"
git switch main
git switch -c branch-cube
echo -e '\ndef cube(n):\n    return n ** 3' >> math.py
git add math.py && git commit -m "Add cube"
git switch main
git merge branch-square
git merge branch-cube
```

No conflict — Git auto-merged because both additions appended to the end of the file and the context (the last line of the base file) matched on both sides. The result contains both `square` and `cube`.

### Exercise 2

**Task:** Create a conflict, then resolve it by accepting the entire "ours" version using `git checkout --ours`. Verify no conflict markers remain.

<details>
<summary>Hint</summary>
Create a simple file, branch, change the same line on both branches. Merge to trigger the conflict, then use `git checkout --ours file.py` and stage/commit.
</details>

### Solution

```bash
mkdir -p ~/conflict-beginner-2 && cd ~/conflict-beginner-2
git init
echo "line1" > f.txt && git add f.txt && git commit -m "Base"
git switch -c side-a
sed -i 's/line1/line1-A/' f.txt && git add f.txt && git commit -m "A change"
git switch main
sed -i 's/line1/line1-B/' f.txt && git add f.txt && git commit -m "B change"
git merge side-a
# Conflict triggered
git checkout --ours f.txt
git add f.txt && git commit -m "Accept ours"
cat f.txt
```

`f.txt` contains "line1-B" — the `main` version. No markers present.

### Exercise 3

**Task:** Create a conflict, abort the merge, then confirm you are back to the exact pre-merge state. Verify with `git log` and `git status`.

<details>
<summary>Hint</summary>
Use `git merge --abort` after the conflict appears. Run `git diff --stat` to confirm the working tree is clean.
</details>

### Solution

```bash
mkdir -p ~/conflict-beginner-3 && cd ~/conflict-beginner-3
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch -c feature
sed -i 's/start/feature change/' f.txt && git add f.txt && git commit -m "Feature"
git switch main
sed -i 's/start/main change/' f.txt && git add f.txt && git commit -m "Main"
git merge feature
# Conflict
git merge --abort
git status
git log --oneline
```

`git status` reports "nothing to commit, working tree clean." `git log` shows the two pre-merge commits and no merge commit or conflicted state.

### Exercise 4

**Task:** During a conflict, use `git show :1:file.txt`, `:2:file.txt`, and `:3:file.txt` to view all three stages. Compare their contents.

<details>
<summary>Hint</summary>
After triggering the conflict, inspect each index stage. Stage 1 is the merge base, stage 2 is "ours", stage 3 is "theirs".
</details>

### Solution

```bash
mkdir -p ~/conflict-beginner-4 && cd ~/conflict-beginner-4
git init && echo "initial value" > config.txt && git add config.txt && git commit -m "Base"
git switch -c alt
echo "alt value" > config.txt && git add config.txt && git commit -m "Alt"
git switch main
echo "main value" > config.txt && git add config.txt && git commit -m "Main"
git merge alt
# Now inspect the stages
echo "=== Stage 1 (base) ===" && git show :1:config.txt
echo "=== Stage 2 (ours) ===" && git show :2:config.txt
echo "=== Stage 3 (theirs) ===" && git show :3:config.txt
```

Three different values appear:
- Stage 1: "initial value"
- Stage 2: "main value"
- Stage 3: "alt value"

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a 3-way conflict involving multiple files. Branch A modifies `utils.py` and `db.py`. Branch B modifies `utils.py` and `api.py`. After merging, one file should auto-merge and one should conflict. Identify which is which before resolving.

<details>
<summary>Hint</summary>
Make each branch change different lines of `utils.py` (auto-merge) or the same line (conflict). Only one of the two modified files needs to conflict. Use `git status` after the merge attempt to see which file is listed under "both modified".
</details>

### Solution

```bash
mkdir -p ~/conflict-inter-1 && cd ~/conflict-inter-1
git init
echo "utility function" > utils.py
echo "database helper" > db.py
echo "api route" > api.py
git add . && git commit -m "Base"
git switch -c branch-a
sed -i 's/utility function/util_v2/' utils.py
sed -i 's/database helper/db_v2/' db.py
git add . && git commit -m "Branch A changes"
git switch main
sed -i 's/utility function/util_v3/' utils.py
sed -i 's/api route/api_v2/' api.py
git add . && git commit -m "Branch B changes"
git merge branch-a
```

Expected result:
- `api.py`: auto-merged (only branch B changed it)
- `db.py`: auto-merged (only branch A changed it)
- `utils.py`: auto-merged (both changed the same line, but Git can sometimes auto-merge if the changes are compatible — or conflicts if they are different

Actually, since both branches changed the same line of `utils.py` to different values, this will conflict:

```
Auto-merging utils.py
CONFLICT (content): Merge conflict in utils.py
Auto-merging db.py
Merge of db.py made by the 'ort' strategy.
Auto-merging api.py
Merge of api.py made by the 'ort' strategy.
```

Resolve by editing `utils.py`:

```bash
sed -i 's/<<<<<<< HEAD.*/utility function v3/' utils.py
sed -i '/^=======$/d' utils.py
sed -i '/^>>>>>>> branch-a$/d' utils.py
git add utils.py
git commit -m "Merge branch-a: keep utility v3, accept both db and api changes"
```

### Exercise 2

**Task:** Create a conflict where the correct resolution combines elements from both sides rather than picking one. For a JSON configuration file, one branch renames a key, the other changes the value under the old key. Resolve by applying the rename AND the new value.

<details>
<summary>Hint</summary>
Set up a JSON config where branch A changes `"timeout": 30` to `"request_timeout": 30` (key rename) and branch B changes `"timeout": 30` to `"timeout": 60` (value change). The correct merge is `"request_timeout": 60`.
</details>

### Solution

```bash
mkdir -p ~/conflict-inter-2 && cd ~/conflict-inter-2
git init
cat > config.json << 'EOF'
{
    "host": "localhost",
    "timeout": 30,
    "retries": 3
}
EOF
git add config.json && git commit -m "Base config"
git switch -c branch-rename
sed -i 's/"timeout": 30/"request_timeout": 30/' config.json
git add config.json && git commit -m "Rename timeout key"
git switch main
sed -i 's/"timeout": 30/"timeout": 60/' config.json
git add config.json && git commit -m "Increase timeout"
git merge branch-rename
```

Conflict occurs. The correct resolved content:

```json
{
    "host": "localhost",
    "request_timeout": 60,
    "retries": 3
}
```

Edit and commit:

```bash
cat > config.json << 'EOF'
{
    "host": "localhost",
    "request_timeout": 60,
    "retries": 3
}
EOF
git add config.json
git commit -m "Merge: rename timeout key AND use new value 60"
```

### Exercise 3

**Task:** Create a conflict, then use `git log --merge` to identify the exact commits that caused it. Use the output to explain to a colleague why the conflict occurred: "Commit X changed line Y, and commit Z changed the same line."

<details>
<summary>Hint</summary>
After the conflict, `git log --merge` shows only commits from both branches that touch conflicted files. Use `git show <hash>` on each to see the specific changes.
</details>

### Solution

```bash
mkdir -p ~/conflict-inter-3 && cd ~/conflict-inter-3
git init && echo "version = 1" > app.py && git add app.py && git commit -m "v1"
echo "version = 2" > app.py && git add app.py && git commit -m "v2"
git switch -c feature-x
echo "version = 3" > app.py && git add app.py && git commit -m "v3-x"
git switch main
echo "version = 4" > app.py && git add app.py && git commit -m "v4-main"
git merge feature-x
git log --merge
```

The output shows exactly two commits — the one on `main` and the one on `feature-x` that both modified `app.py`:

```
commit <hash-main>
    v4-main

commit <hash-feature>
    v3-x
```

Explain: "The conflict is because commit `v4-main` changed `app.py` line 1 to `version = 4`, and commit `v3-x` changed the same line to `version = 3`. Both branches changed the same line of the same file after the common ancestor `v2`."

## Tier 3 — Advanced

### Exercise 1: Multi-file conflict with domain knowledge

**Scenario:** You are merging a feature branch into `main`. Three files conflict: a database migration script, a configuration file, and a test file. Each requires different resolution logic:

- **Migration script:** Both versions are correct, they just ran at different timestamps. Keep both — run them sequentially.
- **Config file:** One side added `max_connections` to the database section, the other side removed the entire database section. The correct resolution is keeping `max_connections` and keeping the database section.
- **Test file:** Both sides added different test cases at the end of the file. Keep all test cases.

**Task:** Set up this scenario and resolve each file appropriately.

<details>
<summary>Hint</summary>
Create each file with clear section boundaries. After the conflict, resolve each file differently: for the migration script, combine both timestamp blocks; for config, keep the database section with `max_connections`; for tests, keep both test functions.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/conflict-adv-1 && cd ~/conflict-adv-1
git init
echo "CREATE TABLE users (id INT);" > 001_create_users.sql
echo "db { host=localhost }" > config.toml
echo "def test_a(): pass" > test_app.py
git add . && git commit -m "Base"

git switch -c feature-x
echo "ALTER TABLE users ADD email TEXT;" >> 001_create_users.sql
echo "max_connections=100" >> config.toml
echo "def test_feature_x(): pass" >> test_app.py
git add . && git commit -m "Feature X"

git switch main
echo "CREATE INDEX idx_users ON users(id);" >> 001_create_users.sql
sed -i 's/db { host=localhost }/prod { host=prod.internal }/' config.toml
echo "def test_main(): pass" >> test_app.py
git add . && git commit -m "Main change"

git merge feature-x
```

Resolve:

```bash
# Migration script — keep both
cat > 001_create_users.sql << 'EOF'
CREATE TABLE users (id INT);
ALTER TABLE users ADD email TEXT;
CREATE INDEX idx_users ON users(id);
EOF

# Config — keep database section + max_connections
cat > config.toml << 'EOF'
db { host=localhost }
max_connections=100
EOF

# Test file — keep both test functions
# (Auto-merge may have already worked on this if functions were on different lines)
# Verify:
cat test_app.py
# If markers remain, combine:
cat > test_app.py << 'EOF'
def test_a(): pass
def test_feature_x(): pass
def test_main(): pass
EOF

git add .
git commit -m "Merge feature-x: resolved migrations, config, and tests"
```

### Exercise 2: Binary file conflict simulation

**Scenario:** Two team members each regenerated a compiled CSS file (or a lock file) on their branches from different source inputs. Git cannot merge binary files or generated artifacts meaningfully because it cannot apply delta logic.

**Task:** Create two branches that each regenerate a simulated "lock file" (a hash-based manifest). When the merge conflicts, resolve it by:
1. Choosing `--theirs` for the lock file
2. Regenerating it from the merged source (simulated by running a "generator" command)
3. Committing the regenerated version

<details>
<summary>Hint</summary>
Use a text file that simulates a lock file with an `md5sum` of the source. Each branch modifies both the source file and the lock file. After the merge conflict on the lock file, pick one side, then regenerate by re-running the hash.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/conflict-adv-2 && cd ~/conflict-adv-2
git init

# Source and its "lock file"
echo "src v1" > source.txt
echo "lock: $(md5sum source.txt | cut -d' ' -f1)" > lock.txt
git add . && git commit -m "Initial"

git switch -c branch-a
echo "src v2 from branch A" > source.txt
echo "lock: $(md5sum source.txt | cut -d' ' -f1)" > lock.txt
git add . && git commit -m "Branch A update"

git switch main
echo "src v2 from main" > source.txt
echo "lock: $(md5sum source.txt | cut -d' ' -f1)" > lock.txt
git add . && git commit -m "Main update"

git merge branch-a
```

Conflict in `lock.txt`. Resolve:

```bash
# Accept theirs for lock.txt (keep their lock file)
git checkout --theirs lock.txt
# But the source might need merging too — resolve source.txt manually
# then regenerate the lock
git checkout --ours source.txt  # keep main's source
# Or edit source.txt to merge both — then regenerate lock
# For example, merge both versions:
echo -e "src v2 from main\nsrc v2 from branch A" > source.txt
# Regenerate lock
echo "lock: $(md5sum source.txt | cut -d' ' -f1)" > lock.txt
git add .
git commit -m "Merge: combined sources, regenerated lock file"
```

**Alternative approach:** Accept both `--ours` for both files, then manually merge sources and regenerate. The key principle: generated files are never manually merged — pick one parent's version and regenerate from the merged source.

### Exercise 3: The silent conflict trap

**Scenario:** You merge branch `feature` into `main`. Git reports zero conflicts. The merge commit is created. But a critical function that was correct on both branches individually now returns wrong results because the two branches made contradictory assumptions about a shared data structure's schema.

**Task:** Set up a scenario where:
- Branch A adds a validation function that expects field `email` to be a string
- Branch B changes the schema so `email` is now a list of strings
- Both branches pass their own tests individually
- The merge creates no text conflict
- The merged code breaks because `feature`'s function receives a list where it expects a string

Detect this by running the test suite after the merge. Then write a test that catches this class of error.

<details>
<summary>Hint</summary>
Create a `schema.py` and a `validate.py`. Branch A changes `validate.py`, branch B changes `schema.py`. They do not touch the same files, so Git auto-merges. But the combined logic is wrong. The fix needs to update the validator to handle both cases.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/conflict-adv-3 && cd ~/conflict-adv-3
git init

cat > schema.py << 'EOF'
USER_SCHEMA = {
    "email": str
}
EOF

cat > validate.py << 'EOF'
from schema import USER_SCHEMA


def validate_user(data):
    for field, expected_type in USER_SCHEMA.items():
        if not isinstance(data.get(field), expected_type):
            raise ValueError(f"{field} must be {expected_type}")
    return True
EOF

cat > test_validate.py << 'EOF'
from validate import validate_user


def test_valid_user():
    assert validate_user({"email": "a@b.com"}) is True
EOF

git add . && git commit -m "Initial schema + validator + tests"

git switch -c branch-new-feature
cat > validate.py << 'EOF'
from schema import USER_SCHEMA


def validate_user(data):
    for field, expected_type in USER_SCHEMA.items():
        if not isinstance(data.get(field), expected_type):
            raise ValueError(f"{field} must be {expected_type}")
    return True


def send_welcome_email(user):
    # New feature: sends email to single address
    addr = user["email"]  # Assumes string
    return f"Sending to {addr}"
EOF

cat > test_feature.py << 'EOF'
from validate import send_welcome_email


def test_welcome():
    result = send_welcome_email({"email": "a@b.com"})
    assert "a@b.com" in result
EOF

git add . && git commit -m "Add welcome email feature"
git switch main

cat > schema.py << 'EOF'
USER_SCHEMA = {
    "email": list
}
EOF

cat > test_schema.py << 'EOF'
from schema import USER_SCHEMA


def test_email_is_list():
    assert USER_SCHEMA["email"] is list
EOF

git add . && git commit -m "Change email schema to list"
git merge branch-new-feature
```

No conflict. Run the tests:

```bash
python -m pytest test_validate.py test_feature.py test_schema.py 2>&1 || true
```

`test_welcome` fails because `send_welcome_email` calls `user["email"]` and gets a list, then tries to format it as a string. But `test_validate` and `test_schema` both pass individually because neither exercises the other branch's code path.

The correct fix:

```python
def send_welcome_email(user):
    emails = user["email"]
    if isinstance(emails, list):
        addr = emails[0]
    else:
        addr = emails
    return f"Sending to {addr}"
```

**Key lesson:** Zero-conflict merges can still introduce bugs. Always run the full test suite after any merge, and write integration tests that exercise cross-module behavior.
