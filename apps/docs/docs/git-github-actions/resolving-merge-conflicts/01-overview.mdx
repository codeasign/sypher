You run `git merge feature` and get a wall of text you did not expect: `CONFLICT (content): Merge conflict in app.py`. Your terminal turns into a battlefield of `<<<<<<<`, `=======`, and `>>>>>>>` markers. The commit you wanted to make is blocked until you decide which lines to keep. This is a merge conflict, and it is not a crisis вҖ” it is Git asking you to make a decision it cannot make for you.

## The Concept

A merge conflict occurs when Git's three-way merge algorithm detects that two branches changed the same part of the same file in incompatible ways. Git does not know which version to keep, so it stops the merge and asks you to decide.

Imagine two editors each correcting the same sentence in a paragraph. One changes "The quick brown fox" to "The speedy brown fox." The other changes it to "The quick orange fox." A human editor must decide which word was meant to change вҖ” or combine both to "The speedy orange fox." Git is that human editor's assistant that highlights each disputed sentence and waits for a decision.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git merge --abort` | Cancel the merge and restore the pre-merge state. Safe вҖ” no changes are lost. | `git merge --abort` |
| `git diff --name-only --diff-filter=U` | List every file with unresolved conflicts (unmerged paths). | `git diff --name-only --diff-filter=U` |
| `git diff` | Show the conflict differences between the working tree and the staged state. | `git diff` |
| `git checkout --ours <file>` | Accept the current branch's version of a conflicted file, discarding theirs. | `git checkout --ours app.py` |
| `git checkout --theirs <file>` | Accept the incoming branch's version of a conflicted file, discarding ours. | `git checkout --theirs app.py` |
| `git log --merge` | Show commits from both branches that touch the conflicted files. | `git log --merge` |
| `git mergetool` | Launch a visual merge tool (requires configuration). | `git mergetool` |
| `git show :1:<file>` | Show the merge-base (common ancestor) version of a file during a conflict. | `git show :1:app.py` |
| `git show :2:<file>` | Show the "ours" (current branch) version during a conflict. | `git show :2:app.py` |
| `git show :3:<file>` | Show the "theirs" (merging branch) version during a conflict. | `git show :3:app.py` |

**Windows note:** All commands work identically on Windows. Visual merge tools like VS Code or `winmerge` work with `git mergetool` on Windows. To configure VS Code: `git config merge.tool vscode` and `git config mergetool.vscode.cmd 'code --wait "$MERGED"'`.

## How It Actually Works вҖ” Step by Step

### Step 1: Git computes the three snapshots

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2))  в”Җв”Җв–¶ ((C3))  вҶҗ main
                               в”Ӯ
                               в””в”Җв”Җв–¶ ((D0)) в”Җв”Җв–¶ ((D1))  вҶҗ feature
```

When you run `git merge feature` (on `main`, where HEAD вҶ’ main вҶ’ C3), Git finds three versions for each file:

- **Stage 1 (base):** the file as it existed at `C2` вҖ” the merge base
- **Stage 2 (ours):** the file as it exists at `C3` вҖ” your current branch
- **Stage 3 (theirs):** the file as it exists at `D1` вҖ” the branch being merged

These are stored in Git's index (staging area) as `:1:file.py`, `:2:file.py`, and `:3:file.py`.

### Step 2: Git processes each file through the merge matrix

For each file, Git performs the comparison table:

| Ours (C3) vs Base (C2) | Theirs (D1) vs Base (C2) | Result |
|---|---|---|
| Unchanged | Unchanged | Keep base version. No action. |
| Changed | Unchanged | Use ours. Auto-merge. |
| Unchanged | Changed | Use theirs. Auto-merge. |
| Changed | Changed вҖ” different lines | Auto-merge by combining. |
| Changed | Changed вҖ” same lines | **CONFLICT.** Stop and mark. |

When the last row occurs, Git writes conflict markers into the working tree file and moves on to the next file.

### Step 3: Git writes conflict markers into the file

```
  def greet():
  <<<<<<< HEAD
      name = "Git"
  =======
      print("Hi")
  >>>>>>> feature
      return name
```

Git replaces the conflicting region with:

- `<<<<<<< HEAD` вҖ” start of our version (current branch)
- `=======` вҖ” divider between our version and their version
- `>>>>>>> feature` вҖ” end of their version (the merging branch)

Git continues processing other files. It stages all non-conflicted files automatically. Conflicted files are left unstaged with both versions present inside the markers.

### Step 4: Git pauses the merge

```
  ((C0)) в”Җв”Җв–¶ ((C1)) в”Җв”Җв–¶ ((C2)) в”Җв”Җв–¶ ((C3))  вҶҗ main (MERGE_HEAD вҶ’ D1)
                               в”Ӯ              вҶҗ HEAD still at main
                               в””в”Җв”Җв–¶ ((D0)) в”Җв”Җв–¶ ((D1))
```

Git creates a special state file `.git/MERGE_HEAD` containing the hash of `D1`. This tells Git that a merge is in progress. Any subsequent Git command checks for this file вҖ” if present, Git refuses most operations until you resolve or abort.

Running `git status` during a conflict shows:

```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  both modified:   app.py

no changes added to commit (use "git add" and/or "git commit -a")
```

### Step 5: You resolve and complete

You edit the conflicted file to remove the markers and decide the final content:

```
  def greet():
      name = "World"
      print("Hello")
      return name
```

Then stage and commit:

```bash
git add app.py
git commit -m "Merge feature into main"
```

Git reads the staged snapshot, combines it with the parents (`C3` and `D1`), and creates the merge commit:

```
  ((C0)) в”Җв”Җв–¶ ... в”Җв”Җв–¶ ((C3)) в”Җв”Җв–¶ ((M0))  вҶҗ main
                               в”Ӯ    /    вҶҗ HEAD
                               в””в”Җв”Җв–¶ ((D1))  вҶҗ feature
```

## When to Use This / When Not To

**Resolve conflicts yourself when:**
- Only one file or a small region is conflicted вҖ” manual editing is fastest
- You know both sides of the code well enough to make the right call
- The conflict is text-based and the correct resolution is a combination of both changes

**Use `--ours` or `--theirs` when:**
- One side is a trivial formatting change and the other is a semantic change вҖ” keep the semantic one
- The conflicted file is a generated artifact (lock file, compiled output) вҖ” take either version and regenerate
- You rebased and the same fix was already applied differently on both branches

**Use `git merge --abort` when:**
- The merge introduces too many conflicts to resolve confidently
- You realize you merged the wrong branch
- Your test suite fails before the merge, and the conflict is a symptom of deeper issues вҖ” abort, fix the branch, then re-merge

**Do not stay in a conflicted state indefinitely.** A half-resolved merge in your working tree blocks all other Git operations. Either complete it or abort it. If you need to pause for hours, `git merge --abort` is always safe вҖ” the commits on both branches are unchanged.
