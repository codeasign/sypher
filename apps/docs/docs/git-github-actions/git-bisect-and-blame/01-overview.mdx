A test started failing three weeks ago, and no one knows which commit caused it. There are 200 commits in the range. Scrolling through every commit by hand would take an hour. You need a way to find the exact commit that introduced the bug — not by reading every diff, but by narrowing the search space exponentially. Git bisect does this using binary search, and git blame tells you who last touched a specific line when you find the culprit.

## The Concept

`git bisect` performs a binary search through your commit history to find the exact commit that introduced a bug. You mark a known-good commit and a known-bad commit, then Git checks out the midpoint commit, you test it, mark it good or bad, and Git halves the range again. In 7 steps you can find the culprit among 128 commits.

`git blame` annotates every line of a file with the commit hash, author, and date of the last modification. It tells you who to talk to about a specific line — not who to blame, but who has the most context.

Think of bisect as a detective narrowing down when a crime happened. Instead of interviewing every person on a 200-person guest list, you split the list in half, ask "was the crime before or after this point?" and repeat. After 8 questions, you have the one person. Blame is the property tag on each item — it tells you who last moved it.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git bisect start` | Begin a bisect session. | `git bisect start` |
| `git bisect start <bad> <good>` | Start bisect with known-good and known-bad commits in one command. | `git bisect start HEAD 1.0.0` |
| `git bisect good` | Mark the current commit as good (bug not present). | `git bisect good` |
| `git bisect bad` | Mark the current commit as bad (bug present). | `git bisect bad` |
| `git bisect good <commit>` | Mark a specific commit as good without checking it out. | `git bisect good v1.0` |
| `git bisect bad <commit>` | Mark a specific commit as bad without checking it out. | `git bisect bad HEAD~10` |
| `git bisect skip` | Skip the current commit (cannot test — e.g., build failure). | `git bisect skip` |
| `git bisect reset` | End the bisect session and return to the original HEAD. | `git bisect reset` |
| `git bisect log` | Show the bisect log (all good/bad marks so far). | `git bisect log` |
| `git bisect run <script>` | Automate bisect using a script that exits 0 for good, non-0 for bad. | `git bisect run pytest` |
| `git blame <file>` | Show the last commit that modified each line of a file. | `git blame app.py` |
| `git blame -L <start>,<end> <file>` | Blame only a specific line range. | `git blame -L 10,20 app.py` |
| `git blame -w <file>` | Ignore whitespace changes when blaming. | `git blame -w app.py` |
| `git blame -e <file>` | Show author email instead of name. | `git blame -e app.py` |

**Windows note:** All commands work identically. For `git bisect run` on Windows, use a batch file or PowerShell script: `git bisect run powershell.exe -Command "python test.py"`. The script must exit with code 0 for good, non-0 for bad.

## The Big Picture

```
┌──────────────────────────────────────────────────────────┐
│              Git Bisect Binary Search                     │
│                                                           │
│  Commits:  C0 ← C1 ← C2 ← C3 ← C4 ← C5 ← C6 ← C7 ← C8  │
│  Status:   G    G    G    ?    ?    ?    ?    ?    B      │
│                                                           │
│  Step 1: Check midpoint C4 — is it good or bad?          │
│  If C4 is good: bug is in C5-C8 (4 commits)              │
│  If C4 is bad:  bug is in C1-C4 (4 commits)              │
│                                                           │
│  Each step halves the remaining range.                    │
│  128 commits → 7 steps.  N commits → log2(N) steps.      │
└──────────────────────────────────────────────────────────┘
```

## How It Actually Works — Step by Step

### Step 1: Initialize the bisect range

```
  ((C0)) ──▶ ((C1)) ──▶ ... ──▶ ((C8))  ← main (buggy)
                                    ↑
  Known good: v1.0 (= C0). Known bad: HEAD (= C8).
```

You run `git bisect start HEAD v1.0`. Git stores the good and bad commit hashes in `.git/BISECT_START`, `.git/BISECT_GOOD`, and `.git/BISECT_BAD`. Git calculates the midpoint commit using a graph-theoretic algorithm that accounts for merge commits and skipped commits.

### Step 2: Git checks out the midpoint

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3)) ──▶ ((C4))  ← HEAD (detached)
                                                       bisect: roughly 4 steps remaining
```

Git checks out commit `C4` — the midpoint between `C0` and `C8`. HEAD is detached. You run your test: `python -m pytest test_app.py`. The test passes (or fails).

### Step 3: Mark the commit and narrow the range

You mark it good or bad. If the test passes (bug not present), you run `git bisect good`:

```
  Bisecting: 2 steps remaining to test after this (roughly C2-C5)
  ((C2))  ← HEAD (detached)
```

Git has narrowed the range to `C4` (exclusive) to `C8` (inclusive) — 4 commits remain. It checks out `C2` (the new midpoint). You test again.

### Step 4: Repeat until the first bad commit is found

After each test and mark, the range halves. With 8 commits, it takes 3-4 steps:

```
  Step 1: Test C4 → good. Range: C5-C8 (4 commits)
  Step 2: Test C6 → bad.  Range: C5-C6 (2 commits)
  Step 3: Test C5 → good. Range: C6 (1 commit)
```

When the range narrows to a single commit, Git reports:

```
  a1b2c3d is the first bad commit
  commit a1b2c3d
  Author: Jane Doe
  Date:   Mon Jun 15 14:30:00 2026

      Refactor config parsing

  app.py | 2 +-
  1 file changed, 1 insertion(+), 1 deletion(-)
```

### Step 5: Reset the bisect session

```bash
git bisect reset
```

Git restores your original HEAD position and deletes the bisect state files from `.git/`.

### Step 6: Use blame to identify the specific line

Now that you know the commit, you want to see exactly which line caused the problem:

```bash
git blame -L 42,45 app.py
```

Expected output:

```
a1b2c3d (Jane Doe 2026-06-15)     config = load_config()
b2c3d4e (John Roe 2026-06-10)     if config.debug:
a1b2c3d (Jane Doe 2026-06-15)         debug_mode = config.debug
c3d4e5f (Alice Smith 2026-06-05)     return debug_mode
```

Line 42 and 44 were last modified by Jane Doe in commit `a1b2c3d` — the same commit bisect identified. John Roe's line 43 was last modified in a different commit.

## When to Use This / When Not To

**Use bisect when:**
- A test failure or bug has no obvious cause and spans many commits
- You need to find the exact commit that introduced a regression for reverting or cherry-picking
- The bug can be detected by a script (use `git bisect run` for full automation)

**Do not use bisect when:**
- The bug is intermittent (flaky test) — bisect needs deterministic good/bad classification
- The range includes commits that do not compile — use `git bisect skip` for those, or start with a narrower range
- There are merge commits that make the DAG non-linear — bisect handles this, but `bisect skip` may be needed for complex merge bubbles

**Use blame when:**
- You need to understand who last changed a specific line of code
- You are reviewing a pull request and want context on why a line was written
- You need to find the right person to ask about a piece of code

**Do not use blame when:**
- A line was last changed by a global formatting commit — `git blame -w` ignores whitespace-only changes to see the real author
- The line was copied from another file — blame shows the copy source, not the original author
- You want to assign fault — the last modifier is not necessarily the author of the bug; the bug may have been introduced in an earlier commit and only surfaced after a refactor
