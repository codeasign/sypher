You will create a release automation workflow that builds a project, creates a GitHub Release with artifacts, publishes to a package registry, and generates release notes — all triggered by pushing a Git tag.

## Setup

Create a repository on GitHub called `release-automation-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/release-automation-practice.git
cd release-automation-practice
```

Set up a Node.js library project:

```bash
npm init -y
mkdir -p .github/workflows
```

Update `package.json`:

```json
{
  "name": "@your-username/release-automation-practice",
  "version": "1.0.0",
  "description": "A library to practice release automation",
  "main": "index.js",
  "scripts": {
    "test": "jest",
    "build": "mkdir -p dist && cp index.js dist/",
    "lint": "echo 'Lint passed'"
  }
}
```

Create `index.js`:

```javascript
function greet(name) {
  return `Hello, ${name}!`;
}

function add(a, b) {
  return a + b;
}

module.exports = { greet, add };
```

Create `index.test.js`:

```javascript
const { greet, add } = require('./index');

test('greet returns greeting', () => {
  expect(greet('World')).toBe('Hello, World!');
});

test('add returns sum', () => {
  expect(add(2, 3)).toBe(5);
});
```

Create a `CHANGELOG.md`:

```markdown
# Changelog

## 1.0.0

- Initial release
- Added greet function
- Added add function
```

Install the test runner:

```bash
npm install --save-dev jest
```

Verify tests pass:

```bash
npm test
```

Initial commit:

```bash
git add .
git commit -m "Initial commit with library code"
git push origin main
```

## The Walkthrough

### Step 1: Create a Tag-Triggered Release Workflow

Create `.github/workflows/release.yml`:

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      packages: write

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Extract version
        run: echo "VERSION=${GITHUB_REF_NAME#v}" >> $GITHUB_ENV

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build

      - name: Package artifact
        run: |
          zip -r app-${{ env.VERSION }}.zip dist/ package.json README.md

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          name: Release ${{ env.VERSION }}
          body_path: CHANGELOG.md
          files: app-${{ env.VERSION }}.zip
          generate_release_notes: true
          draft: false
          prerelease: false
```

This workflow triggers on any tag matching `v*`. It extracts the version number from the tag, runs tests, builds the project, and creates a GitHub Release.

### Step 2: Create a Tag and Trigger the Release

Create a Git tag and push it:

```bash
git tag v1.0.0
git push origin v1.0.0
```

Go to the Actions tab. You should see a new workflow run triggered by the tag push.

### Step 3: Verify the Release

After the workflow completes, go to your repository's Releases page (right sidebar, below "About"). You should see:

```
Release v1.0.0
  ✅ app-1.0.0.zip attached (1.2 MB)
  📝 Generated release notes from merged PRs
  🏷️ Tag: v1.0.0
```

Click into the release. You should see the zip file listed under "Assets" and the release notes displayed in the body.

### Step 4: Create a Pre-release Workflow

Modify the workflow to detect pre-release tags:

```yaml
- name: Create GitHub Release
  uses: softprops/action-gh-release@v2
  with:
    name: Release ${{ env.VERSION }}
    body_path: CHANGELOG.md
    files: app-${{ env.VERSION }}.zip
    generate_release_notes: true
    draft: false
    prerelease: ${{ contains(github.ref_name, 'rc') || contains(github.ref_name, 'beta') || contains(github.ref_name, 'alpha') }}
```

The `prerelease` field is set to `true` when the tag contains `rc`, `beta`, or `alpha`. This marks the release as "Pre-release" in the GitHub UI.

Commit and push:

```bash
git add .github/workflows/release.yml
git commit -m "Add pre-release detection for rc/beta/alpha tags"
git push origin main
```

### Step 5: Create a Pre-release Tag

Create and push a release candidate tag:

```bash
git tag v1.1.0-rc.1
git push origin v1.1.0-rc.1
```

The workflow runs again. This time, the release is marked as "Pre-release" in the GitHub UI, with an orange ⚠️ label.

### Step 6: Add npm Publishing

Add a publish step to the workflow:

```yaml
- name: Publish to npm
  run: |
    echo "//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}" > .npmrc
    npm publish --access public
  if: ${{ !contains(github.ref_name, 'rc') && !contains(github.ref_name, 'beta') }}
```

The `if` condition prevents pre-release versions from being published to npm (or you can invert it to publish pre-releases with a `--tag next` flag).

For this exercise, you do not need a real npm token. Instead, simulate the publish step:

```yaml
- name: Simulate npm publish
  run: |
    echo "Would publish version ${{ env.VERSION }} to npm"
    echo "npm publish --access public"
```

Commit and push:

```bash
git add .github/workflows/release.yml
git commit -m "Add npm publish step to release workflow"
git push origin main
```

### Step 7: Add a Draft Release Workflow

Some teams prefer to review releases before publishing. Create a draft release workflow:

```yaml
name: Draft Release

on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Release version (e.g., 1.2.0)'
        required: true
      prerelease:
        description: 'Is this a pre-release?'
        type: boolean
        default: false

jobs:
  draft:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Update version
        run: |
          npm version ${{ github.event.inputs.version }} --no-git-tag-version
          echo "VERSION=${{ github.event.inputs.version }}" >> $GITHUB_ENV

      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build

      - name: Package
        run: zip -r app-${{ env.VERSION }}.zip dist/

      - name: Create Draft Release
        uses: softprops/action-gh-release@v2
        with:
          name: Release ${{ env.VERSION }}
          draft: true
          prerelease: ${{ github.event.inputs.prerelease == 'true' }}
          files: app-${{ env.VERSION }}.zip
          generate_release_notes: true
```

This workflow is triggered manually from the Actions tab. It creates a draft release that can be reviewed and published from the GitHub UI.

### Step 8: Add a Notification Step

Add a notification step to the release workflow:

```yaml
- name: Notify team
  run: |
    echo "::notice::Release ${{ env.VERSION }} published!"
    echo "Release URL: https://github.com/${{ github.repository }}/releases/tag/${{ github.ref_name }}"
```

This outputs a notice in the Actions log. For real notifications, you would add a Slack, email, or webhook step.

Update the release workflow and commit:

```bash
git add .github/workflows/release.yml
git commit -m "Add release notification step"
git push origin main
```

## Common Errors at This Stage

### Error 1: Workflow does not trigger on tag push

```
No workflow runs triggered by tag v1.0.0
```

**Why it happens:** The `on: push: tags: [v*]` pattern matches the tag name, not the branch name. If the tag was pushed but the workflow file is not on the default branch, it will not trigger.

**Fix:** Ensure the workflow file is on the default branch (usually `main` or `master`). GitHub Actions only reads workflow files from the default branch for tag-triggered workflows.

### Error 2: `softprops/action-gh-release` fails with "Not Found"

```
Error: Not Found
```

**Why it happens:** The `GITHUB_TOKEN` does not have `contents: write` permission, which is required to create releases.

**Fix:** Add the permissions block:

```yaml
jobs:
  release:
    permissions:
      contents: write
```

### Error 3: `npm publish` fails with "403 Forbidden"

```
npm ERR! 403 403 Forbidden - PUT https://registry.npmjs.org/@your-username/release-automation-practice
```

**Why it happens:** The package name `@your-username/release-automation-practice` is scoped but not published to npm, or the `NPM_TOKEN` is missing or invalid.

**Fix:** For this exercise, use the simulated publish step. For real publishing:
1. Create an npm token at npmjs.com/settings/tokens
2. Add it as a repository secret: `NPM_TOKEN`
3. Ensure the package name is unique on npm

### Error 4: Version tag already exists

```
git tag v1.0.0
fatal: tag 'v1.0.0' already exists
```

**Why it happens:** Git tags are unique across the repository. You cannot create a tag that already exists without deleting it first.

**Fix:** Use a different version number, or delete the existing tag:

```bash
# Delete local tag
git tag -d v1.0.0

# Delete remote tag (if previously pushed)
git push origin --delete v1.0.0
```
