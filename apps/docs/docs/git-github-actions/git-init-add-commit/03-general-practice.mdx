## Tier 1 — Beginner

### Exercise 1.1: Initialize a Repository

**Task:** Create a directory called `my-first-repo` anywhere on your filesystem, then initialize it as a Git repository. Verify the repository exists without using `ls` or `dir` to peek at the `.git` folder.

<details>
<summary>Hint</summary>

Use two commands: `mkdir` to create the directory and `git init` inside it. Use `git status` to verify — it will show "No commits yet" if the repository was created successfully.

</details>

<details>
<summary>Solution</summary>

```bash
mkdir my-first-repo
cd my-first-repo
git init
git status
```

Expected output includes `On branch main` and `No commits yet`. If you see `fatal: not a git repository`, you are not inside the directory or `git init` was not run.

</details>

### Exercise 1.2: Stage a Single File

**Task:** Inside `my-first-repo`, create a file called `hello.txt` with the text `Hello, Git!`. Stage only this file. Verify it is staged but not yet committed.

<details>
<summary>Hint</summary>

Create the file with `echo` or a text editor. Use `git add hello.txt` to stage it. Use `git status` to confirm it appears under "Changes to be committed."

</details>

<details>
<summary>Solution</summary>

```bash
echo "Hello, Git!" > hello.txt
git add hello.txt
git status
```

Output shows:

```
Changes to be committed:
  new file:   hello.txt
```

The file is staged and ready for commit. If you run `git status` before `git add`, the file appears under "Untracked files."

</details>

### Exercise 1.3: Commit with a Message

**Task:** Commit the staged `hello.txt` with the message `add hello greeting`. After committing, verify the commit exists in the log. Confirm the commit contains your configured name and email.

<details>
<summary>Hint</summary>

Use `git commit -m "add hello greeting"`. Then run `git log` to see the commit. If `git log` shows `Author identity unknown`, run `git config user.name "Your Name"` and `git config user.email "you@example.com"` first.

</details>

<details>
<summary>Solution</summary>

```bash
git commit -m "add hello greeting"
git log
```

Output shows:

```
commit a1b2c3d... (HEAD -> main)
Author: Your Name <you@example.com>
Date:   Fri Jul 3 10:00:00 2026 -0400

    add hello greeting
```

The `Author` line shows your configured identity. If the name or email is wrong, run `git config` again and make a new commit — the previous commit still has the old identity because commits are immutable once created.

</details>

### Exercise 1.4: Stage All Files at Once

**Task:** Create two new files, `file_a.txt` and `file_b.txt`, each with any content. Stage both files with a single command. Verify both appear as staged.

<details>
<summary>Hint</summary>

Create the files. Then use `git add .` to stage everything in the current directory. The `.` means "current directory and all subdirectories." Alternatively, `git add -A` stages all changes across the entire repository.

</details>

<details>
<summary>Solution</summary>

```bash
echo "Content A" > file_a.txt
echo "Content B" > file_b.txt
git add .
git status
```

Output shows:

```
Changes to be committed:
  new file:   file_a.txt
  new file:   file_b.txt
```

Both files are staged. If you had existing modified files, `git add .` would stage those too. To stage only new files, use `git add *.txt` to match by extension.

</details>

## Tier 2 — Intermediate

### Exercise 2.1: Staging Selectively with `git add -p`

**Task:** Create a file called `calculator.py` with the following content:

```python
def add(a, b):
    return a + b

def multiply(a, b):
    return a * b
```

Stage only the `add` function without staging the `multiply` function. Commit the staged portion with message `add addition function`. Then stage and commit the `multiply` function separately.

<details>
<summary>Hint</summary>

Use `git add -p calculator.py`. Git will show you each "hunk" of the file and prompt `Stage this hunk [y,n,q,a,d,/,j,J,g,e,?]?`. Type `y` for the first hunk and `n` for the second. After committing the first hunk, stage and commit the second.

</details>

<details>
<summary>Solution</summary>

```bash
git add -p calculator.py
```

Git splits the file into hunks. When prompted, answer `y` for the first hunk (the `add` function) and `n` for the second hunk (the `multiply` function).

```bash
git commit -m "add addition function"
```

Now the working directory still has the `multiply` function unstaged. Stage and commit it:

```bash
git add calculator.py
git commit -m "add multiplication function"
git log --oneline
```

Output:

```
* 8f7a6b5 (HEAD -> main) add multiplication function
* 3c4d5e6 add addition function
* a1b2c3d add hello greeting
```

Each function has its own commit. This is the recommended practice — one logical change per commit. If a bug is found in `multiply`, you can revert that commit without losing `add`.

</details>

### Exercise 2.2: Unstage a File

**Task:** Create three files, `page1.md`, `page2.md`, and `page3.md`. Stage all three with `git add .`. Then unstage `page2.md` only, leaving the other two staged. Verify the result with `git status`.

<details>
<summary>Hint</summary>

Use `git restore --staged page2.md` to remove `page2.md` from the staging area. The file stays on disk unchanged — only its staged status is removed.

</details>

<details>
<summary>Solution</summary>

```bash
touch page1.md page2.md page3.md
git add .
git status
```

All three files appear under "Changes to be committed."

```bash
git restore --staged page2.md
git status
```

Output:

```
Changes to be committed:
  new file:   page1.md
  new file:   page3.md

Untracked files:
  page2.md
```

`page2.md` moved back to "Untracked files." The file content is unchanged. The old command for this was `git reset HEAD page2.md`, which still works but `git restore --staged` is the modern approach introduced in Git 2.23.

</details>

### Exercise 2.3: Amend the Last Commit

**Task:** The last commit message you wrote is incomplete. It says `add addition function` but you also fixed a typo in the file. Change the commit message to `add addition function and fix typo`. Do not create a new commit.

<details>
<summary>Hint</summary>

Use `git commit --amend -m "add addition function and fix typo"`. This replaces the last commit with a new one that has the same content and parent but a different message.

</details>

<details>
<summary>Solution</summary>

```bash
git commit --amend -m "add addition function and fix typo"
git log --oneline
```

Output:

```
* 3c4d5e6 (HEAD -> main) add addition function and fix typo
* a1b2c3d add hello greeting
```

Note that the commit hash may change. Amending creates a brand new commit object — the old one still exists in the object database but is now unreachable. Never amend commits that have been pushed to a shared repository, because anyone who pulled the old commit will have a divergent history.

</details>

### Exercise 2.4: Discard Unstaged Changes

**Task:** Open `calculator.py` and add a broken line like `def broken(): return 1 / 0`. Save the file. Do not stage the change. Discard the modification so the file returns to its last committed state.

<details>
<summary>Hint</summary>

Use `git restore calculator.py` to discard all unstaged changes in the file. Git replaces the file content with the version from the last commit. This is irreversible — the changes are gone permanently.

</details>

<details>
<summary>Solution</summary>

```bash
git restore calculator.py
cat calculator.py
```

The file content is back to:

```python
def add(a, b):
    return a + b

def multiply(a, b):
    return a * b
```

`git status` confirms nothing is modified. The old command for this was `git checkout -- calculator.py`. `git restore` is the modern equivalent. Use this with caution: any unsaved work in that file is gone after the command runs.

</details>

## Tier 3 — Advanced

### Exercise 3.1: You Accidentally Staged a Large File

**Scenario:** You are working on a data analysis project. You ran `git add .` and staged everything, including a 200 MB CSV file called `raw_data.csv` that should never be in version control. You have not committed yet. You need to unstage only that file and add it to `.gitignore` so it never gets staged again.

<details>
<summary>Solution and Reasoning</summary>

Step 1: Unstage the large file without affecting other staged files:

```bash
git restore --staged raw_data.csv
```

Step 2: Create or edit `.gitignore` to prevent future staging:

```bash
echo "raw_data.csv" >> .gitignore
```

Step 3: Stage the `.gitignore` file:

```bash
git add .gitignore
```

Step 4: Verify the staging area:

```bash
git status
```

`raw_data.csv` should no longer appear in "Changes to be committed," and `.gitignore` should appear there. The file remains on disk unchanged — it just will not be tracked.

Why not delete the file? Because it is your data file. Git should track code, not data. The `.gitignore` pattern can be a specific filename, a wildcard like `*.csv`, or a directory like `data/`. For many large data files, `*.csv` is more practical than listing each file.

An alternative approach would be to use `git rm --cached raw_data.csv` after the fact, but that only helps if the file was already committed. Since you caught it before committing, `git restore --staged` is the cleaner tool.

</details>

### Exercise 3.2: You Need to Find When a Line Was Introduced

**Scenario:** A bug was introduced somewhere in `calculator.py`. The line `return a * b` produces wrong results for negative numbers. You need to find which commit introduced this line, when, and by whom. The file has been committed multiple times.

<details>
<summary>Solution and Reasoning</summary>

Use `git blame` to annotate every line with the commit that last modified it:

```bash
git blame calculator.py
```

Output:

```
3c4d5e6e (Your Name 2026-07-03 10:05:00 -0400 1) def add(a, b):
3c4d5e6e (Your Name 2026-07-03 10:05:00 -0400 2)     return a + b
8f7a6b5a (Your Name 2026-07-03 10:10:00 -0400 3)
8f7a6b5a (Your Name 2026-07-03 10:10:00 -0400 4) def multiply(a, b):
8f7a6b5a (Your Name 2026-07-03 10:10:00 -0400 5)     return a * b
```

Line 5 shows commit `8f7a6b5a` authored by you on July 3. Now inspect that commit:

```bash
git show 8f7a6b5a
```

This shows the full diff of the commit, including the commit message and author. You now know exactly when and why that line was introduced.

If the repository had many contributors and thousands of commits, `git blame` works the same way — it walks the DAG backward through every parent link until it finds the commit that introduced each line. Git's blame is linear in the number of commits that touched the file, which makes it efficient even on large repositories. A limit: `git blame` shows the last commit that modified a line, so if a later commit restyled whitespace on that line, blame points to the formatting commit, not the original logic commit. Use `git blame -w` to ignore whitespace changes.

</details>

### Exercise 3.3: You Need to Recover a Deleted File from an Earlier Commit

**Scenario:** You deleted `notes.txt` three commits ago and have made two commits since then. You did not realize you needed it until now. The file is not in the working directory, not staged, and not in the current commit. You need to restore it to its state from before it was deleted.

<details>
<summary>Solution and Reasoning</summary>

First, find the commit where `notes.txt` still existed:

```bash
git log --oneline -- notes.txt
```

This filters the log to only commits that modified `notes.txt`. If the file was deleted, the deletion commit will also appear. Look for the most recent commit where the file existed (the commit before the deletion).

Suppose the deletion commit was `b2c3d4e` and the commit before that was `a1b2c3d`. Restore the file from that earlier commit:

```bash
git restore --source=a1b2c3d -- notes.txt
```

Verify the file is back:

```bash
cat notes.txt
ls notes.txt
```

The file is now in your working directory but not staged. Stage it if you want to keep it:

```bash
git add notes.txt
git commit -m "restore notes.txt from commit a1b2c3d"
```

Alternative approach using `git checkout`:

```bash
git checkout a1b2c3d -- notes.txt
```

Both approaches extract the file content from a specific commit into your working directory without altering the commit history. The `--` tells Git that `notes.txt` is a file path and not a branch name — this is necessary when a file name could be confused with a branch or tag.

Why not just `git revert` the deletion commit? Reverting would undo everything that commit did, which might include other changes you want to keep. Restoring a single file is safer when you only need that one file back. If the deletion commit only deleted that one file, `git revert b2c3d4e` would be equivalent and more explicit about the intent.

</details>
