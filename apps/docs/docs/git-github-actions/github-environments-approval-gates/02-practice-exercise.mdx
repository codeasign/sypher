You will set up GitHub Environments with protection rules, create a workflow that deploys to staging automatically and production only after approval, and observe the approval process in action.

## Setup

Create a repository on GitHub called `environments-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/environments-practice.git
cd environments-practice
```

Set up a simple project:

```bash
mkdir -p .github/workflows
```

Create `index.js`:

```javascript
console.log("Deploying application...");
```

Create a `package.json`:

```json
{
  "name": "environments-practice",
  "version": "1.0.0",
  "scripts": {
    "test": "node -e \"console.log('Tests passed')\""
  }
}
```

Create a deploy script `deploy.sh`:

```bash
#!/bin/bash
echo "Deploying to $1 environment..."
echo "Version: $(cat package.json | grep version | head -1 | awk -F: '{ print $2 }' | sed 's/[",]//g' | tr -d ' ')"

if [ "$1" = "staging" ]; then
  echo "Staging URL: https://staging.example.com"
  exit 0
elif [ "$1" = "production" ]; then
  echo "Production URL: https://example.com"
  exit 0
else
  echo "Unknown environment: $1"
  exit 1
fi
```

On Windows, also create `deploy.ps1`:

```powershell
param($Environment)
Write-Output "Deploying to $Environment environment..."
Write-Output "Production URL: https://example.com"
```

Make the script executable (macOS/Linux):

```bash
chmod +x deploy.sh
```

Commit the initial files:

```bash
git add .
git commit -m "Add project files for environment practice"
git push origin main
```

## The Walkthrough

### Step 1: Create GitHub Environments

Go to your repository on GitHub: **Settings > Environments > New environment**.

Create **staging**:
- Environment name: `staging`
- No protection rules (deployments happen automatically)

Create **production**:
- Environment name: `production`
- Required reviewers: `1` (add your own GitHub username)
- Wait timer: `1` minute
- Deployment branches: `main`

### Step 2: Create a Workflow with Environments

Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy with Environments

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.example.com
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        run: bash deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://example.com
    concurrency:
      group: production-deploy
      cancel-in-progress: false
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        run: bash deploy.sh production
```

Commit and push:

```bash
git add .github/workflows/deploy.yml
git commit -m "Add environment-aware deployment workflow"
git push origin main
```

### Step 3: Watch the Pipeline Hit the Approval Gate

Go to the Actions tab. You should see:

```
Deploy with Environments — Run #1
  ├── test — ✅ Passed
  ├── deploy-staging — ✅ Passed
  └── deploy-production — ⏸ Waiting for approval
```

The `deploy-production` job is paused at the approval gate. In the GitHub UI, you can see:

```
⏳ This job is waiting for approval from a reviewer

Environment: production
Branch: main

[Approve] [Reject]
```

The job also has a 1-minute wait timer counting down before the Approve/Reject buttons become active.

### Step 4: Approve the Production Deployment

Since you added your own username as a required reviewer, navigate to the Actions tab and click **Review deployments**:

1. Click on the workflow run
2. Click the "Review deployments" button
3. Select the `production` environment
4. Click **Approve and deploy**

The job immediately starts executing:

```
deploy-production — ✅ Passed
  \_ Deploy to production
     Deploying to production environment...
     Production URL: https://example.com
```

### Step 5: Test a Rejected Deployment

Push a change and this time reject the deployment:

```bash
echo "// small change" >> index.js
git add .
git commit -m "Small change to test rejection"
git push origin main
```

When the pipeline reaches the approval gate, click **Review deployments** and select **Reject**.

The job is marked as skipped:

```
deploy-production — ⏹ Rejected
  Reason: Rejected by reviewer
```

The workflow is not marked as failed — it is marked as "Skipped" because the rejection was a deliberate decision, not a failure. Downstream jobs (if any) would also be skipped.

### Step 6: Add Environment-Specific Secrets

Go to Settings > Environments > production > Add secret.

Add a secret called `PRODUCTION_API_KEY` with any value (e.g., `prod-key-abc123`).

Update the deployment workflow to use it:

```yaml
deploy-production:
  needs: deploy-staging
  runs-on: ubuntu-latest
  environment:
    name: production
    url: https://example.com
  steps:
    - uses: actions/checkout@v4
    - name: Deploy to production
      run: |
        echo "Deploying with API key: ${PRODUCTION_API_KEY:0:4}..."
        bash deploy.sh production
      env:
        PRODUCTION_API_KEY: ${{ secrets.PRODUCTION_API_KEY }}
```

Only the `deploy-production` job (which references `environment: production`) can access the `PRODUCTION_API_KEY` secret. The `test` and `deploy-staging` jobs cannot see it.

Commit and push:

```bash
git add .github/workflows/deploy.yml
git commit -m "Add environment-specific secret for production"
git push origin main
```

### Step 7: Verify Secret Isolation

In the workflow run, expand the `deploy-staging` job logs. You should see that the `PRODUCTION_API_KEY` is not available — the step would show `Deploying with API key: ...` as `Deploying with API key: ....` (empty) because the secret is not injected into staging jobs.

Expand the `deploy-production` job logs (after approving). It should show:

```
Deploying with API key: prod...
```

This confirms that environment-scoped secrets are only available to jobs that reference that environment.

### Step 8: Add a Deployment Protection Branch Rule

Go to Settings > Environments > production > Edit protection rules.

Under "Deployment branches," change from "All branches" to "Selected branches" and add `main`.

Now try pushing a change from a branch other than `main`:

```bash
git checkout -b feature/test
echo "// feature change" >> index.js
git add .
git commit -m "Feature change from non-main branch"
git push origin feature/test
```

The workflow triggers (since `on: push` matches any branch), but the `deploy-production` job is immediately skipped:

```
deploy-production — ⏹ Skipped
  Reason: Environment protection rule not met
```

The job did not even reach the approval gate — GitHub evaluated the branch restriction and rejected the deployment before provisioning a runner. This saves runner minutes and prevents accidental deployment from non-main branches.

## Common Errors at This Stage

### Error 1: "Environment protection rule not met" on every push

```
Environment protection rule not met
Branch: main
Deployment branches: main
```

**Why it happens:** The deployment branch restriction is configured to allow only certain branches, but the branch name does not match exactly. The restriction is case-sensitive and checks the full ref.

**Fix:** Ensure the branch name matches exactly. If you are deploying from `main`, the rule should allow `main`. If you are deploying from `refs/heads/main`, that is the full ref — the UI expects just the branch name.

### Error 2: Approval gate never appears; job runs instantly

```
deploy-production — ✅ Passed (no approval requested)
```

**Why it happens:** You forgot to add a required reviewer to the production environment, or the environment name in the workflow does not match the configured environment name (case-sensitive).

**Fix:** Go to Settings > Environments > production and verify:
1. Required reviewers is checked
2. At least one reviewer is added
3. The environment name in YAML matches exactly: `environment: production` (not `Production` or `prod`)

### Error 3: Reviewer cannot approve

```
You do not have permission to approve this deployment.
```

**Why it happens:** The user attempting to approve does not have write access to the repository. Only users with write, maintain, or admin roles can approve deployments.

**Fix:** Add the user as a collaborator with write access (Settings > Collaborators > Add people), or add their team as a required reviewer.

### Error 4: Secrets are empty in the deployment job

```
Deploying with API key: ....
```

**Why it happens:** The environment secret exists but is not being injected because the job does not reference the correct environment name, or the secret was added to the wrong environment.

**Fix:** Verify:
1. The secret is added to the correct environment (Settings > Environments > production > Secrets)
2. The workflow job references that environment: `environment: production`
3. The secret is passed correctly: `${{ secrets.PRODUCTION_API_KEY }}` (not `${{ secrets.PROD_API_KEY }}` if the secret is named differently)

## What You Just Learned

- You created GitHub Environments (`staging` and `production`) with protection rules
- You configured a required reviewer approval gate for production deployments
- You observed the approval gate pausing the pipeline and waiting for human input
- You approved and rejected deployments and saw both outcomes
- You added environment-specific secrets that are only available to the production job
- You configured a deployment branch restriction to prevent non-main deployments
- You verified secret isolation between environments
