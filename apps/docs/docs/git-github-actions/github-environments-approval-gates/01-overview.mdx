Your CD pipeline deploys to production on every green commit. One Friday afternoon, a junior developer merges a PR with a misconfigured database connection string. Within 30 seconds, the pipeline deploys it to production. All users see a 500 error page for 12 minutes while you scramble to roll back. An approval gate would have caught it: the deploy paused and waited for a senior engineer to review the change before touching production — turning a 30-second auto-deploy into a reviewed, deliberate action.

## The Concept

A GitHub Environment is a named deployment target (staging, production, QA) with optional protection rules. An approval gate is a protection rule that requires one or more designated reviewers to approve a deployment before it executes. Environments also support secret shielding, branch restrictions, and deployment branch policies.

Think of it like a submarine hatch: the crew can open the inner door from either side (free movement between CI and staging), but the outer door has a double-lock system (approval gate) that requires two authorized officers to turn their keys simultaneously before the hatch opens to the ocean (production deployment).

## Core Environments YAML Keys

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `environment` | Target deployment environment name | Yes | `environment: production` |
| `environment.name` | Environment name matching GitHub UI setup | Yes | `name: production` |
| `environment.url` | Deployment URL shown in the GitHub UI | No | `url: https://your-app.io` |
| `environment.environment_url` | Alternate URL field (deprecated) | No | `environment_url: https://your-app.io` |
| `secrets:inherit` | Allow job access to environment secrets | No | `secrets: inherit` |
| `concurrency` | Deployment group for this environment | No | `concurrency: production` |
| `cancel-in-progress` | Cancel pending for same concurrency group | No | `cancel-in-progress: true` |

## Approval Gate YAML Keys (UI Configuration — Not in YAML)

GitHub Environments are configured in Settings > Environments, not in YAML. The YAML references the environment name; the protection rules are set in the GitHub UI:

| Protection Rule | Purpose | Configured In |
|-----------------|---------|---------------|
| Required reviewers | 1–6 people who must approve before deploy runs | GitHub UI > Environment |
| Wait timer | Delay before deploy executes (minutes) | GitHub UI > Environment |
| Deployment branches | Restrict which branches can deploy | GitHub UI > Environment |
| Secret shielding | Environment secrets not exposed to protected branches | GitHub UI > Environment |

## How Environments and Approval Gates Actually Work — Step by Step

### Step 1: Reference the Environment in Your Workflow

You declare the environment in the `environment` key of any job. This is what links the YAML to the protection rules configured in GitHub:

```yaml
deploy-production:
  needs: deploy-staging
  runs-on: ubuntu-latest
  environment:
    name: production
    url: https://your-app.io
  steps:
    - run: ./deploy.sh production
```

The `name` must exactly match the environment name you configured in GitHub Settings > Environments. If the environment does not exist, GitHub creates it on first run (without protection rules).

### Step 2: GitHub Checks Protection Rules

Before the job starts, GitHub evaluates the environment's protection rules. If any rules are configured and the current run does not satisfy them, the job enters a "waiting" state:

- **Required reviewers:** The job sits at "Waiting for approval from reviewers." An allowed reviewer (or team member) gets a notification. They navigate to the Actions tab and click "Review deployments."
- **Wait timer:** The job pauses for the configured duration (e.g., 5 minutes). This gives a human a window to abort if something is obviously wrong.
- **Deployment branch restriction:** If the workflow branch is not in the allowed list, the job fails immediately with "Environment protection rule not met."

### Step 3: Reviewer Approves or Rejects

The reviewer clicks "Review deployments," sees the requested environment, and either approves or rejects:

```
Review pending deployments for production:
  Branch: main @ abc1234
  Requested by: jane.doe

  [Approve and deploy]  [Reject]
```

On **Approve**: The waiting job immediately unblocks. GitHub provisions a runner and the `deploy-production` job starts.

On **Reject**: The job is marked as skipped with the reason recorded in the run log. Downstream jobs (if any) are cancelled.

The reviewer does not need to understand the deployment process — they just need to authorize it. The entire deployment procedure runs automatically after approval.

### Step 4: Environment Secrets Are Exposed

GitHub Environments can store secrets scoped to that environment. These secrets are only injected into jobs that reference the matching environment name:

```yaml
deploy-production:
  environment: production
  steps:
    - run: ./deploy.sh
      env:
        PROD_API_KEY: ${{ secrets.PROD_DEPLOY_KEY }}
```

This is critical for security: production credentials are never exposed to CI, staging, or any job that does not explicitly reference `environment: production`. Additionally, if you enable "Secret shielding" in the environment settings, these secrets are not injected into workflows triggered by pull requests from forks — preventing a malicious fork PR from exfiltrating production keys.

### Step 6: Protect with Concurrency Groups

Always combine environments with `concurrency` to prevent deployment race conditions:

```yaml
concurrency:
  group: production
  cancel-in-progress: false
```

`cancel-in-progress: false` (the default) means if a production deploy is in progress and a new one is triggered, the new one waits behind the current one. Use this over `true` because cancelling an in-progress production deploy — especially one running database migrations — can leave the environment in an inconsistent state.

## When to Use Approval Gates / When Not To

**Use approval gates when:**

- Production deployments affect paying customers — a bad deploy means revenue loss or SLA violation
- You have compliance requirements (SOC 2, PCI-DSS, SOX) mandating separation of duties between deployer and approver
- You are deploying to a shared integration environment where multiple teams deploy — approval prevents unintended overwrites
- The deployment involves irreversible database changes — someone should review the migration before it runs

**Consider skipping approval gates when:**

- Your staging environment gives you sufficient confidence and you deploy to production less than once per day — the approval overhead slows you down without adding value
- You have a fully automated canary or blue-green deploy with automated rollback — the deployment itself is self-healing
- Your team operates on trunk-based development with feature flags — the code is already in production but gated by a flag that the deploy cannot affect

The rule: if a bad deploy costs more than the time spent waiting for approval, add a gate. If waiting for approval costs more than the occasional bad deploy, skip it. Most teams with paying customers benefit from at least one reviewer on production deploys.

## Full Environments Example

```yaml
name: CI/CD with Environments

on:
  push:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: ci
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.your-app.io
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://your-app.io
    concurrency:
      group: production-deploy
      cancel-in-progress: false
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

To enable approval gates for this workflow, go to GitHub.com > your repo > Settings > Environments > production > Add protection rules > Required reviewers > add 1 reviewer. The next workflow run will pause at `deploy-production` until someone approves.
