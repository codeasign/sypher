These exercises build on the walkthrough. Create a fresh repository for each exercise to avoid carry-over state.

## Tier 1: Beginner

### Exercise 1: Unstage a Staged File

You staged a file by accident and want to unstage it without losing your edits.

```bash
mkdir ex1 && cd ex1
git init
echo "data" > important.txt
git add important.txt
echo "more data" >> important.txt
```

The file `important.txt` is staged with its first line, but you edited it again afterwards. Unstage it so the index matches HEAD, but keep both lines in the working directory.

<details>
<summary>Hint</summary>

Use `git reset` with just the file path. The default `--mixed` mode when given a path only affects the index.

</details>

<details>
<summary>Solution</summary>

```bash
git reset important.txt
```

This copies HEAD's version of `important.txt` into the index, but the working directory still has your edits. Run `git status` — `important.txt` now appears as "Changes not staged for commit."

</details>

### Exercise 2: Undo the Last Commit But Keep Changes

You just made a commit with message "WIP" and want to undo it, but keep all the changes in your working directory exactly as they are.

```bash
mkdir ex2 && cd ex2
git init
echo "alpha" > file1.txt && git add file1.txt && git commit -m "First"
echo "beta" > file2.txt && git add file2.txt && git commit -m "WIP"
```

Undo the "WIP" commit and leave `file2.txt` in your working directory as an untracked file.

<details>
<summary>Hint</summary>

Use `git reset --mixed HEAD~1`. The `--mixed` flag resets the index but leaves the working directory intact.

</details>

<details>
<summary>Solution</summary>

```bash
git reset --mixed HEAD~1
```

The "WIP" commit is undone. HEAD points to "First." The index is reset to match "First." The `file2.txt` still exists in the working directory but is now untracked:

```
Untracked files:
  file2.txt
```

</details>

### Exercise 3: Discard Unstaged Changes in One File

You have two modified files but only want to discard one.

```bash
mkdir ex3 && cd ex3
git init
echo "keep me" > safe.txt
echo "lose me" > risky.txt
git add . && git commit -m "Setup"
echo "new safe content" > safe.txt
echo "new risky content" > risky.txt
```

Discard the changes in `risky.txt` without affecting `safe.txt`.

<details>
<summary>Hint</summary>

Use `git checkout` with the `--` separator and the file path, without a commit reference.

</details>

<details>
<summary>Solution</summary>

```bash
git checkout -- risky.txt
```

This copies the index version of `risky.txt` (which matches HEAD) into the working directory. `safe.txt` still shows as modified. Verify with `git status`.

</details>

### Exercise 4: Revert a Commit Without Creating a New Commit

You want to see what the revert diff looks like before committing it.

```bash
mkdir ex4 && cd ex4
git init
echo "first version" > doc.txt
git add doc.txt && git commit -m "Initial"
echo "second version" > doc.txt
git add doc.txt && git commit -m "Second update"
echo "third version" > doc.txt
git add doc.txt && git commit -m "Third update"
```

Revert the "Second update" commit, but do not create a new commit yet. Look at the diff, then discard the revert.

<details>
<summary>Hint</summary>

Use `git revert --no-commit <hash>`. This stages the inverse changes without committing. Use `git reset --mixed` to discard the staged revert.

</details>

<details>
<summary>Solution</summary>

```bash
# Find the hash of "Second update"
git log --oneline

# Assume the hash is abc123
git revert --no-commit abc123
git diff --cached

# This shows the inverse diff: "third version" becomes "first version"
# Discard the revert
git reset --mixed
```

`git reset --mixed` without a commit argument resets the index to HEAD, discarding the staged revert changes. The working directory still has "third version" in `doc.txt`.

</details>

## Tier 2: Intermediate

### Exercise 5: Split a Commit into Two

You made a commit that combines an API endpoint and its test file. Split it into two commits.

```bash
mkdir ex5 && cd ex5
git init
echo "def get_user(): pass" > api.py
echo "def test_get_user(): pass" > test_api.py
git add . && git commit -m "First"
echo "def create_user(): pass" >> api.py
echo "def test_create_user(): pass" >> test_api.py
git add . && git commit -m "Add user creation endpoint and tests"
```

Split the last commit so that `api.py` changes are in one commit and `test_api.py` changes are in a separate commit.

<details>
<summary>Hint</summary>

Use `git reset --mixed HEAD~1` to undo the commit while keeping the working directory. Then stage each file separately and commit.

</details>

<details>
<summary>Solution</summary>

```bash
git reset --mixed HEAD~1

# api.py and test_api.py still have the new lines but are unstaged
git add api.py
git commit -m "Add user creation endpoint"

git add test_api.py
git commit -m "Add tests for user creation endpoint"

git log --oneline
```

You now have three commits: "First", "Add user creation endpoint", and "Add tests for user creation endpoint." Each file change is in its own logical commit.

</details>

### Exercise 6: Recover from a Hard Reset Using Reflog

You accidentally ran `git reset --hard` and lost two commits. Recover them.

```bash
mkdir ex6 && cd ex6
git init
echo "1" > nums.txt && git add nums.txt && git commit -m "Commit 1"
echo "2" >> nums.txt && git add nums.txt && git commit -m "Commit 2"
echo "3" >> nums.txt && git add nums.txt && git commit -m "Commit 3"
echo "4" >> nums.txt && git add nums.txt && git commit -m "Commit 4"
```

Now simulate the accident:

```bash
git reset --hard HEAD~2
```

The log shows only "Commit 1" and "Commit 2." Recover commits 3 and 4.

<details>
<summary>Hint</summary>

Run `git reflog` to find the commit hash that was HEAD before the reset. Then use `git reset --hard` to restore it.

</details>

<details>
<summary>Solution</summary>

```bash
git reflog

# Look for the entry that says "HEAD@{N}: commit: Commit 4"
# The hash is the first column on that line

git reset --hard <hash_of_commit_4>
git log --oneline
```

All 4 commits are restored. The reflog recorded every HEAD movement, including the hard reset. The commit you reset to still exists in the object store — the reflog kept a reference to it.

</details>

### Exercise 7: Restore a Deleted File from an Older Commit

You deleted a file and committed the deletion. Restore it without reverting the entire commit.

```bash
mkdir ex7 && cd ex7
git init
echo "important data" > critical.txt
echo "other data" > other.txt
git add . && git commit -m "Initial"
rm critical.txt
git add . && git commit -m "Remove critical file"
```

Restore `critical.txt` without undoing the deletion of the file in a way that affects `other.txt`.

<details>
<summary>Hint</summary>

Use `git checkout <commit> -- critical.txt` where `<commit>` is the commit before the deletion.

</details>

<details>
<summary>Solution</summary>

```bash
# Find the commit hash of "Initial"
git log --oneline

# Assume the initial commit hash is a1b2c3d
git checkout a1b2c3d -- critical.txt
cat critical.txt
git status
```

`critical.txt` is restored in both the index and the working directory. `other.txt` is untouched. The "Remove critical file" commit still exists in the history, but now the working tree has the file back.

</details>

## Tier 3: Advanced

### Exercise 8: You Pushed a Secret — Now What?

You accidentally committed and pushed an API key to the `main` branch. Three other developers have pulled the commit. Your team uses `main` for production deployments.

The commit looks like:

```bash
git log --oneline origin/main

abc123 (HEAD -> main, origin/main) Add API key in config
def456 Previous commit
```

The `config.py` contains `API_KEY="sk-abc123"`. You need to remove the key from the codebase while preserving the rest of the work.

<details>
<summary>Hint</summary>

You cannot use `git reset` because the commit is already pushed and pulled by others. You need `git revert` to undo the changes in a new commit, but that leaves the secret in the history. If the secret is truly sensitive (production credential), you must also rotate the key and consider a forced rewrite of the branch with a `git filter-branch` or `git filter-repo` cleanup. For non-sensitive data, revert is enough.

</details>

<details>
<summary>Solution</summary>

Step 1 — Revert the commit immediately to remove the key from the working code:

```bash
git revert --no-edit abc123
git push origin main
```

This creates commit `def456` that removes the API key. The code is now safe, but the key still exists in the git history at commit `abc123`.

Step 2 — Rotate the API key. The old key is compromised because it exists in the git history. Generate a new key and update any services.

Step 3 — For sensitive credentials, purge the key from history so future clones do not contain it:

```bash
# Install git-filter-repo first
git filter-repo --path config.py --invert-paths --force
```

This rewrites history, removing `config.py` from every commit. It changes every commit hash. Everyone on the team must re-clone after this.

Step 4 — Force push the cleaned history:

```bash
git remote add origin <url>
git push origin --force --all
```

Coordinate with your team to re-clone. This approach has a high coordination cost — only use it for genuinely sensitive data like production database passwords or signing keys.

</details>

### Exercise 9: Detached HEAD Recovery

You ran `git checkout <hash>` (without a file path) and then made three commits. Now you realize you are in detached HEAD state and want to keep your work on a branch.

```bash
mkdir ex9 && cd ex9
git init
echo "data" > data.txt && git add data.txt && git commit -m "Base"
echo "v1" > data.txt && git add data.txt && git commit -m "Feature start"

# Detach HEAD
git checkout HEAD~1
```

Now you are on a detached HEAD at the "Base" commit. Make two more commits:

```bash
echo "experiment 1" > exp1.txt && git add exp1.txt && git commit -m "Exp 1"
echo "experiment 2" > exp2.txt && git add exp2.txt && git commit -m "Exp 2"
```

You want to keep these commits and merge them back onto the main branch.

<details>
<summary>Hint</summary>

Create a branch at the current HEAD position with `git branch <name>`, then switch to the main branch and merge.

</details>

<details>
<summary>Solution</summary>

Step 1 — Create a branch at the current location:

```bash
git branch experiment
```

This creates a branch pointer at the current HEAD (the "Exp 2" commit). The branch name anchors those commits so they will not be garbage collected.

Step 2 — Switch to the main branch:

```bash
git checkout main
```

If you do not know the main branch name, run `git branch` to list all branches. The default is `main` or `master`.

Step 3 — Merge the experiment branch:

```bash
git merge experiment
```

All commits are preserved on the `experiment` branch and the changes are now part of `main`. The three-branch approach (or even just creating the branch and staying there) is also valid — you do not have to merge if you prefer the experiment to live on its own branch.

</details>

### Exercise 10: The Case of the Ambiguous Stash

You have been working on two unrelated features in the same repository. You made changes, stashed them, switched branches, came back, and now you have three stashes. You need to selectively apply one stash item and discard the rest, but you cannot tell which stash contains what.

```bash
mkdir ex10 && cd ex10
git init
echo "base" > base.txt && git add base.txt && git commit -m "Base"

# Work on feature A
echo "feature A work" > feature-a.txt
git stash push -m "Feature A WIP"

# Work on feature B
echo "feature B work" > feature-b.txt
git stash push -m "Feature B WIP"

# Work on feature C
echo "feature C work" > feature-c.txt
git stash push -m "Feature C WIP"

git stash list
```

You need to restore "Feature B" and delete stashes A and C. You want to use checkout or reset operations to manage the files properly after applying the wrong stash.

<details>
<summary>Hint</summary>

Use `git stash show -p <stash>` to inspect each stash's contents before applying. If you apply the wrong stash, use `git checkout -- <file>` or `git reset --mixed` to clean up the working directory and index without losing the stash.

</details>

<details>
<summary>Solution</summary>

Step 1 — Inspect each stash before applying:

```bash
git stash show -p stash@{2}
git stash show -p stash@{1}
git stash show -p stash@{0}
```

Identify that `stash@{1}` contains `feature-b.txt`.

Step 2 — Apply the correct stash:

```bash
git stash pop stash@{1}
```

This applies "Feature B" and removes it from the stash list.

Step 3 — Remove the remaining stashes:

```bash
git stash drop stash@{1}  # This is now the old Feature C, since stash@{1} was popped
git stash drop stash@{0}  # Feature A
```

Alternatively, clear all remaining stashes:

```bash
git stash drop stash@{0}
git stash drop stash@{0}
```

Or use `git stash clear` to remove everything, but this is irreversible.

Step 4 — If you accidentally applied the wrong stash, discard the working directory changes and re-apply the correct one:

```bash
# Wrong stash applied, discard files
git checkout -- .
git stash list  # The wrong stash is still there (pop removed it, apply keeps it)

# Oops, already popped — it is gone. But the data is still in the object store.
# Find the hash: git fsck --lost-found | grep commit
# Then git show <hash> -- feature-b.txt to extract the content
```

This exercise shows that `git stash pop` is destructive (it removes the stash entry). Use `git stash apply` instead of `pop` if you want to examine the changes before committing to keeping them.

</details>
