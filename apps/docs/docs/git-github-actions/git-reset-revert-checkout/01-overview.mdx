You have been working on a feature branch for three days. You made a commit that broke the build, then made two more commits on top trying to fix it. Now you need to undo the damage without losing the good work in those two fix commits. Or maybe you staged a file by accident. Or you want to look at a file from an older commit without switching branches. Git reset, revert, and checkout are the three tools that handle all of these scenarios, and each one operates on a different combination of Git's three internal trees.

## The Concept

Git tracks your project state in three trees: HEAD (the last commit on the current branch), the index (the staging area), and the working directory (the files you edit on disk). Reset, revert, and checkout move data between these trees in different ways — reset rewrites history by moving the branch pointer, revert creates a new commit that undoes an old one, and checkout restores files from any tree to your working directory.

Think of your project as a notebook with three layers: a published table of contents (HEAD), a draft page you are filling in (the index), and your scratch paper (working directory). Reset lets you tear out pages from the published table of contents, revert adds a new page that says "ignore what page 7 said," and checkout copies text from an older page onto your scratch paper without changing the table of contents.

## Command Reference

| Command | Trees Affected | Typical Use | Example |
|---------|---------------|-------------|---------|
| `git reset --soft <commit>` | HEAD only | Uncommit but keep changes staged | `git reset --soft HEAD~1` |
| `git reset --mixed <commit>` | HEAD, index | Uncommit and unstage changes | `git reset --mixed HEAD~2` |
| `git reset --hard <commit>` | HEAD, index, working dir | Discard all changes since a commit | `git reset --hard abc123` |
| `git revert <commit>` | HEAD (new commit) | Safely undo a pushed commit | `git revert abc123` |
| `git checkout <commit> -- <file>` | Working directory only | Restore a file from an older commit | `git checkout HEAD~1 -- main.py` |
| `git checkout -- <file>` | Working directory only | Discard unstaged changes in a file | `git checkout -- config.json` |
| `git reset <file>` | Index only | Unstage a file (safe unstage) | `git reset README.md` |
| `git revert --no-commit <commit>` | HEAD, index | Apply revert changes without committing | `git revert --no-commit abc123` |
| `git reset --keep <commit>` | HEAD, index, working dir | Reset but keep unstaged changes if possible | `git reset --keep HEAD~2` |
| `git checkout <branch>` | HEAD, index, working dir | Switch branches (safe) | `git checkout feature/login` |

On Windows (cmd.exe or PowerShell), use the same syntax. On macOS and Linux, the commands are identical. File paths with spaces need quotes on all platforms: `git checkout "my file.txt"`. On Windows, avoid using angled bracket paths — use quotes around paths that contain spaces.

## How It Actually Works

Git internally uses three trees to track state. The `HEAD` pointer references the current branch's latest commit. The index (staging area) holds the content you plan to commit next. The working directory is your actual filesystem. Every reset, revert, or checkout operation moves data between these three layers.

When all three trees are in sync, your working directory is clean and there are no staged or unstaged changes. This is the baseline state. From here, you can experiment with reset, revert, and checkout without worrying about losing uncommitted work.

You edit `main.py` and stage it with `git add`. The index tree now differs from HEAD and the working directory.

The index now holds `main.py v4`, which is the same content as the working directory because you staged it after editing. If you now run `git reset --mixed HEAD` (the default), Git copies HEAD's `main.py v3` back into the index, leaving the working directory untouched.

Git moved the branch pointer from `c4` back to `c2`, but the index and working directory still contain the content from `c4`. The commit `c4` still exists in the repository but is no longer reachable from any branch. It will be garbage collected eventually unless you create a branch or tag pointing to it.

The `--mixed` mode is the default. It moves HEAD and resets the index to match, but leaves the working directory alone. Your changes are now unstaged. Run `git status` to see them as "Changes not staged for commit."

`--hard` is the nuclear option. It overwrites your working directory with the content from the target commit. Any uncommitted changes are lost. There is no recovery if you did not first commit or stash them. Only use `--hard` when you are certain you want to discard everything.

### Step 6: Git Revert in Action

Revert does not move HEAD backward. It creates a new commit that applies the inverse of the changes from a target commit.

│                                                                      │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────────┐     │
│  │    HEAD      │     │    Index     │     │ Working Directory │     │
│  │  commit c4   │     │  commit c4   │     │   commit c4      │     │
│  │  main.py v2  │     │  main.py v2  │     │  main.py v2      │     │
│  │  utils.py v2 │     │  utils.py v2 │     │  utils.py v2     │     │
│  │  config.py v1│     │  config.py v1│     │  config.py v1    │     │
│  └──────────────┘     └──────────────┘     └──────────────────┘     │
│                                                                      │
│  c4 is a new commit that undoes c3's changes.                       │
│  History is linear and additive — no rewriting.                     │
│  Safe to push because you never rewrote shared history.             │
└──────────────────────────────────────────────────────────────────────┘
`} />

Revert reads the diff of the target commit, computes the inverse patch, applies it, and creates a new commit. This is safe for commits that have already been pushed to a shared branch because it does not rewrite history — it only adds to it.

### Step 7: Git Checkout for Files

Unlike `git checkout <branch>`, checking out a file does not change HEAD. It copies content from the index or a specific commit into the working directory.

When you run `git checkout <commit> -- <file>`, Git reads the file's blob from the target commit, writes it to the index, and then copies it to the working directory. This is different from `git checkout -- <file>` (without a commit), which only copies from the index to the working directory, discarding unstaged changes.

## When to Use This

Use `git reset --soft` when you committed too early and want to amend the commit message or add more related changes. It keeps everything staged so you can re-commit immediately.

Use `git reset --mixed` (default) when you committed something that should be split across multiple commits. The changes remain in your working directory but unstaged, so you can add them in smaller chunks.

Use `git reset --hard` only when you are absolutely sure you want to discard all uncommitted changes and go back to a known good state. Example: a failed experiment you want to erase completely.

Use `git revert` when the commit you need to undo has already been pushed to a shared branch. Revert is the only safe option for public history because it does not rewrite commits that others may have based work on.

Use `git checkout -- <file>` to discard unstaged changes in a single file without affecting other files or the staging area.

Use `git checkout <commit> -- <file>` to restore a file to a specific version from an older commit without changing the current branch.

## When Not to Use This

Do not use `git reset --hard` on a branch that others are working on. If you force-push a rewritten branch, other contributors will have merge conflicts or lose work.

Do not use `git reset` at all on commits that have been pushed to a shared branch and are visible to your team. Use `git revert` instead.

Do not use `git checkout -- <file>` if you have important unstaged changes you want to keep — you will lose them. Use `git stash` first to save them.

Do not use `git checkout <branch>` with a file path argument expecting to switch branches. If you write `git checkout main -- config.py`, Git interprets `main` as a commit reference, not a branch name, and restores `config.py` from the tip of `main` without switching branches.

Do not use `git reset HEAD~1` after pushing the commit to a shared branch. The commit you thought you removed is still in the remote history, and a `git pull` will bring it back.
