In this walkthrough, you will build a Git history step by step, then practice reset, revert, and checkout operations on it. You will see how each command affects the three trees and learn to recover from mistakes.

## Setup

Open a terminal in a directory where you can create a test project. Run these commands to create a repository with a known history:

```bash
mkdir git-reset-lab
cd git-reset-lab
git init
```

Create a file called `app.py` with an initial version:

```bash
echo "print('hello')" > app.py
git add app.py
git commit -m "Initial commit"
```

Now add a configuration file:

```bash
echo "DEBUG=true" > config.py
git add config.py
git commit -m "Add config"
```

Add a feature and a test file:

```bash
echo "def greet(name):\n    return f'Hello {name}'" > app.py
echo "def test_greet():\n    assert greet('World') == 'Hello World'" > test_app.py
git add app.py test_app.py
git commit -m "Add greet function with test"
```

Finally, add a README:

```bash
echo "# Git Reset Lab" > README.md
git add README.md
git commit -m "Add README"
```

On Windows, use `notepad` or your editor to create the files instead of `echo` with `\n`. The file contents are the only thing that matters.

Verify your log:

```bash
git log --oneline
```

You should see 4 commits. The most recent is "Add README" at the top.

## The Walkthrough

### Step 1: Make and Stage Changes, Then Unstage

Make a change to `app.py` and stage it, then unstage it using `git reset`.

```bash
echo "print('hello world')" > app.py
git add app.py
git status
```

You will see that `app.py` is staged as a new change. Now unstage it:

```bash
git reset app.py
git status
```

`app.py` now appears as "Changes not staged for commit." The `git reset` command without a mode flag defaults to `--mixed`, but when given a file path, it only affects the index — it copies HEAD's version of that file into the index, leaving the working directory unchanged.

**Checkpoint:** Confirm `git status` shows unstaged changes to `app.py`.

### Step 2: Discard Unstaged Changes with Checkout

Now discard the unstaged change in `app.py`:

```bash
git checkout -- app.py
git status
git diff app.py
```

The working directory is clean again. The `git checkout -- app.py` command copied the index version of `app.py` (which matches HEAD) into the working directory, overwriting your edit.

**Checkpoint:** `git status` shows no changes. `cat app.py` shows `print('hello')` from the first commit.

### Step 3: Create a Commit to Undo Later

Make a deliberate change and commit it:

```bash
echo "print('this is a mistake')" > app.py
git add app.py
git commit -m "Add mistake"
```

Verify the commit exists:

```bash
git log --oneline
```

You now have 5 commits. The newest one is "Add mistake."

**Checkpoint:** `git log --oneline` shows 5 commits. The newest is "Add mistake."

### Step 4: Revert the Mistake (Safe Undo)

Since this commit exists only in your local repository, you could use reset. But practice the safe approach that works even on shared branches:

```bash
git revert HEAD --no-edit
```

Git opens the revert commit message automatically. With `--no-edit`, it accepts the default message. Your log now shows:

```bash
git log --oneline --all
```

You will see a new commit at the top: "Revert 'Add mistake'." The original "Add mistake" commit is still in the history. The `app.py` file should now contain the content from before the mistake.

### Step 5: Soft Reset to Edit a Commit Message

Make a normal commit, then realize you want to change the message:

```bash
echo "VERSION=2" > version.py
git add version.py
git commit -m "Add version"
```

Now you decide the message should be "Add version configuration file." Use `--soft` reset:

```bash
git reset --soft HEAD~1
git status
```

Everything is still staged — the index was not touched. Only HEAD moved back one commit.

```bash
git commit -m "Add version configuration file"
```

**Checkpoint:** `git log --oneline` shows the new message. The commit contains the same `version.py` file.

### Step 6: Mixed Reset to Split a Commit

You made a commit that contains two unrelated changes. Use `--mixed` to split them:

```bash
echo "x = 1" > math.py
echo "y = 2" > calc.py
git add math.py calc.py
git commit -m "Add math and calc modules"
```

Now you realize these should be two separate commits. First, find the commit hash:

```bash
git log --oneline
```

Take note of the hash of the commit just before "Add math and calc modules." Then:

```bash
git reset --mixed HEAD~1
git status
```

Both `math.py` and `calc.py` appear as "Untracked files" — the reset moved HEAD back and cleared the index, so these files are no longer tracked. But they still exist in your working directory.

Stage and commit them separately:

```bash
git add math.py
git commit -m "Add math module"
git add calc.py
git commit -m "Add calc module"
```

**Checkpoint:** `git log --oneline` shows two separate commits for the two files.

### Step 7: Checkout a File from an Older Commit

You want to see what `app.py` looked like in the very first commit:

```bash
git log --oneline
```

Find the hash of the first commit (the one with message "Initial commit"). Then:

```bash
git checkout <hash> -- app.py
cat app.py
```

The file now contains `print('hello')` from the first commit. Your HEAD and branch pointer have not moved — only the working directory and index for `app.py` changed.

### Step 8: Hard Reset to a Known Good State

You have been experimenting and want to completely reset to the state before the "Add math and calc modules" split. This is dangerous, so be careful.

```bash
git reset --hard HEAD~3
git log --oneline
```

Your log now shows only the revert commit and the version configuration commit. All commits after that are gone, and the working directory matches HEAD exactly.

If you want to get back to where you were, you can use `git reflog`:

```bash
git reflog
```

Find the commit hash that was your previous HEAD (the one before the hard reset). Restore it:

```bash
git reset --hard <hash>
```

The reflog is your safety net. Every time HEAD moves — even during a hard reset — Git records the previous position in the reflog. The reflog is local to your repository and is never pushed.

**Checkpoint:** You can recover from a hard reset as long as you act before the reflog entries expire (default 90 days).

## Common Errors at This Stage

### Error 1: "Cannot do hard reset with paths"

```
fatal: Cannot do hard reset with paths.
```

You ran `git reset --hard` with a file path, like `git reset --hard HEAD~1 -- app.py`. The `--hard` flag is incompatible with file paths. Use `git checkout <commit> -- <file>` instead.

**Fix:** `git checkout HEAD~1 -- app.py`

### Error 2: "You are in 'detached HEAD' state"

```
Note: switching to 'abc123f'.
You are in 'detached HEAD' state.
```

You ran `git checkout <hash>` without a file path, which switched HEAD to a detached state. This is not an error, but it is confusing. You are no longer on a branch, so any new commits will be orphaned when you switch away.

**Fix:** If you meant to restore a file, add `-- <file>`: `git checkout abc123f -- app.py`. If you meant to switch to a branch, use `git checkout <branchname>`.

### Error 3: "Would lose changes"

```
error: Your local changes to the following files would be overwritten by checkout
```

You have unstaged changes in a file that `git checkout` or `git reset --hard` would overwrite. Git refuses to overwrite your work.

**Fix:** Stash your changes first: `git stash push -m "save before reset"`. Then run the reset or checkout. Restore later with `git stash pop`.

### Error 4: "Reflog shows nothing after reset"

Running `git reflog` shows no entries for the commit you want to recover. This happens if you created a shallow clone or if the reflog entries expired.

**Fix:** If `git reflog` does not show the commit, try `git fsck --lost-found` to find dangling commits. Git will list commits that are not reachable from any branch or tag. You can then inspect them and cherry-pick or reset to them.

### Error 5: "Merge conflict during revert"

```
error: could not revert abc123f... Add mistake
hint: After resolving the conflicts, mark them with
hint: "git add <file>" and run "git revert --continue"
```

Revert detected a conflict because the changes in the target commit overlap with changes made in later commits.

**Fix:** Resolve the conflicts in your editor, then `git add <file>` and `git revert --continue`. Git will create the revert commit with the resolved content.

## What You Just Learned

- `git reset --soft` moves HEAD backward without changing the index or working directory, keeping your changes staged.
- `git reset --mixed` (the default) moves HEAD backward and clears the index, but leaves the working directory intact.
- `git reset --hard` moves HEAD backward, clears the index, and overwrites the working directory — use with extreme caution.
- `git revert` creates a new commit that undoes a previous commit without rewriting history, making it safe for shared branches.
- `git checkout -- <file>` discards unstaged changes by copying the index version to the working directory.
- `git checkout <commit> -- <file>` restores a file from a specific commit without changing HEAD or switching branches.
- `git reflog` is your safety net — it records every HEAD movement and lets you recover from accidental resets.
