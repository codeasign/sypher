## Tier 1 — Beginner

### Exercise 1: View Last 5 Commits in Compact Form

Use a single `git log` command to show only the most recent 5 commits, each on one line with abbreviated hashes.

<details>
<summary>Hint</summary>

Combine `-n` (or `--max-count`) with `--oneline`.

```bash
git log --oneline -n 5
```
</details>

<details>
<summary>Solution</summary>

```bash
git log --oneline -5
```

This limits output to the 5 most recent commits. The `-5` flag is shorthand for `--max-count=5`. Combine it with any other flag — for example `git log --oneline --graph -5` to visualize the last 5 commits with branch structure.
</details>

### Exercise 2: Find Commits from This Week

Show all commits made in the last 7 days, with full details (not compact).

<details>
<summary>Hint</summary>

Use `--since` with a relative date string.

```bash
git log --since="7 days ago"
```
</details>

<details>
<summary>Solution</summary>

```bash
git log --since="7 days ago"
```

Git parses `"7 days ago"` into a Unix timestamp and compares it against each commit's author date. You can use shorter windows: `"1 day ago"`, `"2 hours ago"`, or fixed dates: `--since="2026-06-01"`. For a specific end date, add `--until="2026-06-30"`.
</details>

### Exercise 3: See What Changed in a Single File

Show the commit history that affected only `README.md`, with one line per commit.

<details>
<summary>Hint</summary>

Pass the file path at the end of `git log --oneline`.

```bash
git log --oneline -- README.md
```
</details>

<details>
<summary>Solution</summary>

```bash
git log --oneline -- README.md
```

The `--` separates refs from paths. It tells Git that `README.md` is a file path, not a branch name. This is critical when a file name matches a branch (e.g., `main`). Without `--`, Git would interpret `main` as a branch reference and show unrelated history. You can also add `-p` to see the actual changes to that file: `git log -p -- README.md`.
</details>

### Exercise 4: Count Your Commits

Count how many commits exist in the repository without piping to another command.

<details>
<summary>Hint</summary>

Use `git rev-list` with `--count` or `git log` with `--oneline` piped to a counter. But there is a direct flag.

```bash
git rev-list --count HEAD
```
</details>

<details>
<summary>Solution</summary>

```bash
git rev-list --count HEAD
```

`git rev-list` is the plumbing command that `git log` uses internally. The `--count` flag outputs a single integer. This is faster than `git log --oneline | wc -l` (or `Measure-Object` on PowerShell) because it does not need to format any output — it only counts objects traversed.
</details>

## Tier 2 — Intermediate

### Exercise 1: Find All Commits by Two Authors on a Specific Date

Show all commits made by either Alice or Bob on June 30, 2026, using the author filter and a date range.

<details>
<summary>Hint</summary>

You can pass `--author` multiple times in Git 2.18+ or use the `--all-match` flag. Combine with `--since` and `--until`.

```bash
git log --oneline --author="Alice" --author="Bob" --since="2026-06-30" --until="2026-06-31"
```

Or use a regex pattern in the author name.
</details>

<details>
<summary>Solution</summary>

```bash
# Option A: Multiple --author flags (Git 2.18+)
git log --oneline --author="Alice" --author="Bob" --since="2026-06-30" --until="2026-07-01"

# Option B: Regex pattern (works on all Git versions)
git log --oneline --author="Alice\|Bob" --since="2026-06-30" --until="2026-07-01"

# Option C: Multiple calls + sort -u
{
  git log --oneline --author="Alice" --since="2026-06-30" --until="2026-07-01"
  git log --oneline --author="Bob" --since="2026-06-30" --until="2026-07-01"
} | sort -u
```

Option A is the most readable. Note that `--until="2026-07-01"` is exclusive (up to but not including July 1), so a full day query requires the next day's start.
</details>

### Exercise 2: Generate a Changelog Between Two Tags

Assume your repository has a tag `v1.0` on the initial commit and you want to show all commits from that tag up to (but not including) the current HEAD. Output should be formatted as a changelog with commit hash, author, and message.

<details>
<summary>Hint</summary>

Use Git's `--format` flag to define the output. The range syntax `tag..HEAD` shows commits reachable from HEAD but not from the tag.

```bash
git log --oneline v1.0..HEAD
```

For a formatted changelog:
</details>

<details>
<summary>Solution</summary>

```bash
# If the tag does not exist, create one for practice
git tag v1.0 HEAD~10 2>/dev/null

# Generate the changelog
git log --format="%h | %an | %s" v1.0..HEAD
```

The range `v1.0..HEAD` means "commits reachable from HEAD minus commits reachable from v1.0." The `--format` flag accepts placeholders:
- `%h` — abbreviated commit hash
- `%an` — author name
- `%ae` — author email
- `%s` — subject line
- `%b` — body
- `%ci` — committer date (ISO format)

For a Markdown changelog:
```bash
git log --format="* **%h** by %an — %s" v1.0..HEAD
```
</details>

### Exercise 3: Track When a Function Was Removed

You have a legacy codebase. A function `validateEmail` existed at some point but is now gone. Find every commit that ever touched this function, including commits that added, modified, or removed it.

<details>
<summary>Hint</summary>

The pickaxe option `-S` finds commits where the number of occurrences of a string changed. Unlike `--grep`, it searches code content, not commit messages.

```bash
git log -S"validateEmail" --oneline --all
```

Add `-p` to see the actual changes.
</details>

<details>
<summary>Solution</summary>

```bash
git log -S"validateEmail" --oneline --all -p
```

The `-S` flag instructs Git to examine every commit's diff and count occurrences of `"validateEmail"`. If the count differs from the parent commit, that commit is included. The `--all` flag ensures the search covers all branches and not just the current one — critical if the function existed on a now-deleted or merged branch.

For more detailed output showing how many occurrences changed:
```bash
git log -S"validateEmail" --oneline --all --stat
```

A related flag `-G"regex"` accepts a regular expression and matches commits where the pattern was added or removed (even if the total count stays the same), which is useful for tracking variable renames or refactors.
</details>

### Exercise 4: Create a Branch Point Report

Find the exact commit where a feature branch diverged from `main`. You know the branch was merged, so you need to trace back from the merge commit or use the merge base.

<details>
<summary>Hint</summary>

Use `git merge-base` to find the common ancestor of two branches. Then compare the history from there.

```bash
git merge-base main feature/dashboard
git log --oneline <base-hash>..feature/dashboard
```
</details>

<details>
<summary>Solution</summary>

```bash
# Find the merge base (common ancestor)
BASE=$(git merge-base main feature/dashboard)
echo "Branch point: $BASE"
git show --oneline $BASE

# Show all commits on the feature branch since it diverged
git log --oneline $BASE..feature/dashboard

# One-liner: find the divergence point with graph context
git log --oneline --graph --all --decorate \
  $(git merge-base --all main feature/dashboard)
```

`git merge-base` finds the best common ancestor commit. If there are multiple merge bases (criss-cross merge), `--all` shows all of them. The range `BASE..feature/dashboard` shows commits on the feature branch that are not in main, effectively showing the branch's unique work.
</details>

## Tier 3 — Advanced

### Exercise 1: Debug a Regression Across 100 Commits

Your application worked 100 commits ago but is now broken. You have narrowed the symptoms: a file `src/payments/processor.js` is throwing an error that did not exist before. You need to find the exact commit that introduced the regression, using only CLI tools (no GUI).

<details>
<summary>Hint</summary>

This is a binary search problem, not a linear scan. Use `git bisect` to perform a divide-and-conquer search across the commit range. First, mark a known-good commit and a known-bad commit.

```bash
git bisect start
git bisect bad HEAD
git bisect good HEAD~100
```
</details>

<details>
<summary>Solution</summary>

```bash
# Start the bisect session
git bisect start

# Mark current HEAD as bad
git bisect bad HEAD

# Mark the earlier commit as good
git bisect good HEAD~100

# Git checks out a commit halfway between, then you test:
# Run your test command
python -c "from src.payments.processor import process; print(process())"

# On success, mark the current commit as good
git bisect good

# On failure, mark as bad
git bisect bad

# Repeat until Git reports the first bad commit
# Git needs approximately log2(100) ≈ 7 iterations

# When done, view the result:
git bisect log

# Clean up:
git bisect reset
```

For an automated search (if you have a reliable test script):
```bash
git bisect start HEAD HEAD~100
git bisect run python -m pytest tests/test_processor.py
```

To supplement the bisect, use `git log` to investigate the found commit:
```bash
# Show the offending commit in detail
git log -p $(git bisect visualize --oneline | head -1)

# See all files changed in that commit
git show --stat $(git bisect visualize --oneline | awk '{print $1}')
```

**Why this works:** `git bisect` performs a binary search, which requires `log2(N)` steps to find a regression across N commits. For 100 commits, that is 7 steps instead of potentially 100 with a linear search. The key insight is that you do not read every commit — you test each half-interval and let the algorithm eliminate half the remaining candidates each time.

**Other approaches exist:** You could also use `git log -S"error_string" -- src/payments/processor.js -p` if you know the exact error string, or `git log -L1,50:src/payments/processor.js --oneline` to trace line-by-line history. The bisect approach is best when you cannot identify a specific search term.
</details>

### Exercise 2: Reconstruct a Lost Commit Message

You accidentally squashed or rebased a branch and lost the original commit messages. The branch had 6 commits that were combined into 1. The code is correct, but you need the original commit messages for an audit trail. You only have the hash of the pre-squash branch tip from a stale reflog entry.

<details>
<summary>Hint</summary>

Git's reflog records every movement of HEAD and branch pointers. Check the reflog for the branch before the squash. Use `git reflog show feature/dashboard` to find old positions, then `git log` to read the original commits.

```bash
git reflog show feature/dashboard
```
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Find the reflog entries for the branch
git reflog show feature/dashboard

# Look for an entry like:
# a1b2c3d feature/dashboard@{1}: (before squash) checkout: moving from feature/dashboard

# Step 2: Inspect the commits at that earlier position
git log --oneline feature/dashboard@{1}

# Step 3: Preserve the messages while keeping the squashed state
git log --format="* %h %s (%an, %ar)" feature/dashboard@{1} > original-messages.txt

# Step 4: Create a new annotated tag pointing to the pre-squash state
git tag audit/pre-squash feature/dashboard@{1}

# Now you have the messages in the file AND a tag pointing to the original history.
# The original commits are still in the object store as long as the reflog entry exists
# (typically 90 days before garbage collection).
```

**Why this works:** The reflog is a local-only safety net that records every ref update. `feature/dashboard@{1}` means "the previous position of feature/dashboard before the most recent update." The `@{N}` notation works for any ref (branches, HEAD, remote-tracking branches). Reflog entries expire by default after 90 days for reachable commits and 30 days for unreachable ones.

**Important caveats:**
- Reflog entries are local — they do not exist in clones or on remotes
- Running `git gc` or `git prune` may remove the underlying commit objects
- `@{N}` counts change with each operation — verify with `git reflog show` first

**Other approaches exist:** If the squash was done via `git merge --squash` on a branch that still exists, the original commits are on that branch — simply check it out and run `git log`. If the squash was done via interactive rebase, use `git fsck --lost-found` to find dangling commits by traversing unreachable objects.
</details>

### Exercise 3: Audit All Changes to a Sensitive File Across All Branches

Your security team needs to find every change ever made to `src/config/secrets.js` across every branch in the repository, including deleted branches. They need to know who changed what, when, and — most critically — whether secrets (API keys, passwords) were ever committed in the file.

<details>
<summary>Hint</summary>

By default, `git log` only follows the current branch's history. To find changes on deleted branches, use `--all` to traverse every ref, and add `--reflog` to include commits referenced only in the reflog. For content searches, combine with `-S` or `-G`.

```bash
git log --all --reflog --oneline -p -- src/config/secrets.js
```
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Comprehensive history of the file across all branches and reflog
git log --all --reflog --oneline -p -- src/config/secrets.js

# Step 2: Search for common secret patterns in the file's history
git log --all -S"api_key" --oneline -p -- src/config/secrets.js
git log --all -S"password" --oneline -p -- src/config/secrets.js
git log --all -S"secret" --oneline -p -- src/config/secrets.js
git log --all -G"(AKIA|AAAA|SKIA)[A-Z0-9]{16}" --oneline -- src/config/secrets.js

# Step 3: List every distinct version (blob) of the file that existed
git log --all --pretty=format:"%H %ai" -- src/config/secrets.js

# Step 4: For each version, extract and scan the content
# This creates a temporary directory with every version of the file
git log --all --diff-filter=ACDMR --name-only --format="" -- src/config/secrets.js | \
  while read commit; do
    git show $commit:src/config/secrets.js 2>/dev/null | \
      grep -n "api_key\|password\|secret\|token\|credential" && \
      echo "--- Found in commit $commit ---"
  done

# Step 5: Check if the file still exists in any branch
git branch -a --contains $(git log --all --oneline -- src/config/secrets.js | tail -1 | cut -d' ' -f1)
```

**Why this approach is comprehensive:**
- `--all` includes all local branches, remote-tracking branches, and tags
- `--reflog` includes commits that are no longer referenced by any branch but still exist in the reflog
- `-S"string"` finds commits that changed the number of occurrences of a string — critical for identifying when secrets were added and later removed
- `--diff-filter=ACDMR` shows only commits where the file was Added, Copied, Deleted, Renamed, or Modified

**Important considerations:**
- Commits that have been garbage collected (reflog expired AND no branch reference) are permanently gone from the default object store. To maximize coverage, run this audit before `git gc` and encourage developers to never force-push sensitive branches
- If the repository was ever public or shared, assume secrets in the history are compromised regardless of later deletion
- The `git filter-branch` or `git filter-repo` tools can purge secrets from history, but they rewrite commit hashes and require coordination with all contributors

**Other approaches exist:** For large repositories, consider using dedicated secret-scanning tools (GitHub secret scanning, `truffleHog`, `git-secrets`) which automate pattern detection across the full history. The CLI approach shown here gives you full control and transparency for audit purposes.
</details>

## Bonus: Quick Reference Card

```bash
# Everyday log
git log --oneline -10                    # Last 10 commits
git log --oneline --graph --all          # Full DAG visualization

# Filtering
git log --author="name"                  # By author
git log --since="7 days ago"             # By date range
git log --grep="fix" -i                  # By keyword in message
git log -S"functionName"                 # By code content (pickaxe)

# Detail level
git log -p                               # With patch diffs
git log --stat                           # File change stats
git log --format="%h %an %s"             # Custom format

# Comparisons
git show HEAD~3                          # Single commit details
git diff main..feature                   # Compare branches
git diff --staged                        # Compare staged vs HEAD

# Find the divergence point
git merge-base main feature              # Common ancestor

# Binary search for regressions
git bisect start HEAD good-commit
git bisect run ./test-script.sh

# Reflog
git reflog                               # Recent operations
git log --reflog                         # Include reflog-only commits
```
