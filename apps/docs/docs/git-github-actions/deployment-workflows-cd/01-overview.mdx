Your team finishes a feature on Friday. Someone manually SSHes into the production server, pulls the latest code, restarts the service, and crosses their fingers. On Monday, you discover the deploy broke because they forgot to run migrations first. Continuous Delivery (CD) automates this: every commit that passes CI is automatically deployed to a staging environment, and a one-click approval promotes it to production — no SSH, no forgotten steps, no Friday-night anxiety.

## The Concept

Continuous Delivery is a pipeline extension after CI that automatically builds deployable artifacts and deploys them to environments (staging, production) through an automated, repeatable process. Continuous Deployment takes this further by deploying every green build to production without human intervention. CD is the difference between "someone deploys" and "the pipeline deploys."

Think of it like package delivery sorting: your local post office receives packages (CI build), sorts them onto the correct truck (staging deploy), and then delivers them to the final address (production deploy). No one decides route-by-route whether the truck should drive — the process is defined and runs automatically.

## Core CD Workflow YAML Keys

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `environment` | Target deployment environment with approval rules | No | `environment: production` |
| `deployment` | Deployment metadata (URL, description, environment) | No | `deployment: { url: ${{ steps.deploy.outputs.url }}, environment: production }` |
| `needs` | Job dependency — CD always depends on CI | Yes (for CD) | `needs: [build, test]` |
| `if` | Conditional execution (e.g., only deploy from main) | No | `if: github.ref == 'refs/heads/main'` |
| `concurrency` | Prevent concurrent deployments to same environment | No | `concurrency: production` |
| `cancel-in-progress` | Cancel in-progress deployment when a new one triggers | No | `cancel-in-progress: true` |
| `secrets` | Pass repository secrets to the deployment job | No | `secrets: { DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }} }` |
| `runs-on` | Runner for deployment (often self-hosted for prod) | Yes (per job) | `runs-on: [self-hosted, production]` |

## How CD Pipelines Actually Work — Step by Step

### Step 1: CI Passes → CD Begins

When CI completes successfully on a branch you deploy from (typically `main` or `master`), the CD job starts. The `needs` keyword enforces this dependency:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to staging
        run: ./deploy.sh staging
```

The `deploy-staging` job only starts after `build` succeeds. If `build` fails, `deploy-staging` is skipped automatically — you never deploy a broken build.

### Step 2: Build the Deployable Artifact

Before deploying, you typically create a deployable artifact — a compiled binary, a Docker image, or a bundled tarball. This artifact is uploaded as a workflow artifact so it can be shared across jobs:

```yaml
deploy-staging:
  needs: build
  steps:
    - uses: actions/download-artifact@v4
      with:
        name: build-output
    - name: Push to staging server
      run: |
        rsync -avz ./dist/ user@staging-server:/var/www/app/
```

Using artifacts ensures the exact same code that passed CI is what gets deployed — not whatever happens to be on `main` at deploy time.

### Step 3: Deploy to Staging (Automated)

The staging deploy runs automatically on every green commit. Staging is a production-like environment that lets you verify the deployment process, test database migrations, and check environment-specific configuration:

```yaml
deploy-staging:
  environment:
    name: staging
    url: https://staging.your-app.io
  steps:
    - name: Deploy
      run: ./deploy.sh staging
    - name: Smoke test
      run: ./smoke-test.sh https://staging.your-app.io
```

Adding a smoke test step after deploy validates the deployment worked. A smoke test runs a few critical end-to-end checks — "is the home page loading?", "can you log in?", "is the database connected?" — and fails the job if anything is wrong.

### Step 4: Production Deploy (Manual Approval or Automatic)

For production, you typically add a manual approval step using GitHub Environments, or you automate with a conditional:

```yaml
deploy-production:
  needs: deploy-staging
  if: github.ref == 'refs/heads/main'
  environment:
    name: production
    url: https://your-app.io
  steps:
    - name: Deploy to production
      run: ./deploy.sh production
```

With GitHub Environments configured, you can require reviewers to approve before this job runs. Without that, the `if` condition gates deployment to the `main` branch, but every green commit to `main` deploys automatically.

### Step 5: Deployment Tracking and Rollback

A production deployment should always be reversible. Your deploy script should include a rollback strategy:

```yaml
steps:
  - name: Deploy with rollback
    run: |
      if ./deploy.sh production; then
        echo "Deploy succeeded"
      else
        echo "Deploy failed — rolling back"
        ./rollback.sh production
        exit 1
      fi
```

Common rollback strategies include:
- **Blue-green deploy:** Keep the previous version running on separate infrastructure. Switch the load balancer back on failure.
- **Canary deploy:** Route 10 percent of traffic to the new version. If error rates stay low, route 100 percent. If errors spike, redirect traffic to the old version.
- **Versioned deploy:** Keep the last N versions of the artifact in cloud storage. Rollback = redeploy the previous version.

### Step 6: Concurrency Control

Use `concurrency` to prevent two deployments from running to the same environment at once:

```yaml
concurrency:
  group: production-deploy
  cancel-in-progress: true
```

Without this, if you push twice in rapid succession, two deployment jobs run concurrently and race each other — one might overwrite the other mid-deploy, leaving the environment in an undefined state. `cancel-in-progress: true` cancels the in-progress deploy when a new one starts, ensuring only the latest deployment applies.

## When to Use CD / When Not To

**Use CD when:**

- You deploy to production at least once per week — automation pays back the setup time within a month
- You have a staging environment that mirrors production — CD is useless if staging and production differ significantly
- Your deploy process involves more than 2 steps (build, test, migrate, deploy, smoke test) — automation eliminates forgotten steps

**Consider not automating deployment when:**

- You deploy less than once per month — the setup time may not justify the benefit, though you should question why you deploy so infrequently
- Your deploy requires manual database migrations or data backfills that cannot be scripted — automate everything you can, but keep the manual data step explicit
- You have regulatory constraints that require a person to type the deploy command (PCI-DSS, SOC 2) — CD can still stage the artifact; the person just clicks "Approve" in the UI rather than SSHing

The rule: automate the decision to deploy. Manual the execution only where regulation requires it. Every scripted step is a step that cannot be forgotten.

## Full CD Pipeline Example

```yaml
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy-staging:
    needs: build-and-test
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.your-app.io
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        run: ./deploy.sh staging
      - name: Smoke test
        run: ./smoke-test.sh https://staging.your-app.io

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://your-app.io
    concurrency:
      group: production-deploy
      cancel-in-progress: true
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        run: ./deploy.sh production
```

This pipeline builds and tests, then deploys to staging automatically. After staging succeeds, it deploys to production. Only one production deploy runs at a time. All steps use `main` branch code and pass through the full CI → staging → production flow.
