Your CI pipeline has a `lint` job and a `test` job. You expected them to run in parallel, but `test` is sitting there waiting for `lint` to finish. Meanwhile, the `test` job on Windows is passing but the same job on macOS is failing, and you cannot figure out why because the runner images are different. Understanding how jobs, steps, and runners interact is the key to designing fast, reliable pipelines.

## The Concept

A **job** is a unit of work that runs on a single **runner** (a virtual machine or container). Each job contains an ordered list of **steps** — individual commands or actions that execute sequentially. Jobs within a workflow can run in parallel (default) or sequentially (via `needs`), and can share data through outputs, artifacts, or caches.

Think of it like a restaurant kitchen. The **runner** is a chef's station with its own tools and ingredients. A **job** is a recipe card — it stays at one station from start to finish. **Steps** are the individual actions: chop, sauté, plate. Multiple recipes (jobs) can be prepared simultaneously at different stations. If one recipe depends on another (e.g., sauce must be ready before plating), the `needs` keyword ensures the right order.

## Key Properties

| Property | Scope | Purpose |
|----------|-------|---------|
| `runs-on` | Job | Runner type or label (e.g., `ubuntu-latest`, `windows-latest`) |
| `steps` | Job | Ordered list of steps to execute |
| `needs` | Job | List of jobs that must complete before this one starts |
| `if` | Job/Step | Condition for running the job or step |
| `continue-on-error` | Job/Step | If true, a failure does not mark the job as failed |
| `timeout-minutes` | Job | Maximum time before the job is cancelled (default 360) |
| `strategy` | Job | Matrix or fail-fast configuration |
| `strategy.matrix` | Job | Variables that create multiple job variants |
| `strategy.fail-fast` | Job | Stop all matrix jobs when one fails |
| `services` | Job | Docker containers to run alongside the job |
| `container` | Job | Docker container to run the job inside |
| `outputs` | Job | Named values passed to downstream jobs |
| `env` | Job/Step | Environment variables |
| `working-directory` | Step | Override the working directory for a `run` step |
| `shell` | Step | Shell to use for a `run` step (bash, pwsh, python, etc.) |

## How It Actually Works — Step by Step

### Step 1: GitHub schedules jobs onto runners

When a workflow run starts, GitHub reads the `jobs` map and identifies which jobs have no `needs` dependencies. Those jobs are immediately eligible for scheduling. Jobs with `needs` are added to the queue and start when all their dependencies have completed successfully.

### Step 2: A runner is provisioned for each job

Each job gets its own runner instance. GitHub maintains pools of runners for each OS type:

- `ubuntu-latest` / `ubuntu-22.04` / `ubuntu-20.04`
- `windows-latest` / `windows-2022` / `windows-2019`
- `macos-latest` / `macos-14` / `macos-13`

The runner is provisioned from GitHub's managed pool (free tier: up to 2,000 minutes/month for private repos, unlimited for public repos). Self-hosted runners can also be registered.

### Step 3: Steps execute sequentially in the job context

Within a job, steps run in the order they are defined. Each step has access to:
- The working directory (shared across steps — files created in step 1 are available in step 2)
- Environment variables set at the workflow, job, or step level
- The `$GITHUB_ENV` file for setting new env vars that persist to subsequent steps
- The `$GITHUB_OUTPUT` file for setting step outputs that are consumed by other jobs

### Step 4: Each step is either a `run` command or a `uses` action

- **`run`**: Executes the command string in the default shell (bash on Linux/macOS, PowerShell on Windows)
- **`uses`**: Downloads and runs a GitHub Action (a reusable unit of automation from the Marketplace or a repository)

Both can be combined in the same step:
```yaml
- name: Build and test
  uses: actions/setup-node@v4
  with:
    node-version: 20
- run: npm ci && npm test
```

### Step 5: The matrix strategy creates multiple job variants

When you define a `strategy.matrix`, GitHub creates one job instance per combination of matrix variables. Each variant is independent — they run on separate runners in parallel.

```yaml
jobs:
  test:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
```

This produces 4 job instances: `test (ubuntu-latest, 18)`, `test (ubuntu-latest, 20)`, `test (windows-latest, 18)`, `test (windows-latest, 20)`.

### Step 6: Job outputs flow between dependent jobs

A job can declare `outputs` that downstream jobs (via `needs`) can read:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.get-version.outputs.version }}
    steps:
      - id: get-version
        run: echo "version=1.2.3" >> $GITHUB_OUTPUT

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying version ${{ needs.build.outputs.version }}"
```

## When to Use Jobs / Steps / Runners

### Use multiple jobs when:
- You need to run tasks on different OSes (matrix strategy)
- Tasks have different resource requirements (e.g., lint on a small runner, deploy on a large one)
- You want to parallelize independent work to reduce wall-clock time
- You need to gate a task on the success of previous tasks (deploy after tests pass)

### Use a single job with multiple steps when:
- Tasks share files or state (steps share the working directory)
- Tasks must run sequentially on the same machine
- You want to keep the job structure simple

### Use a specific runner type when:
- **Linux**: General CI/CD, Docker containers, most tooling (fastest boot time)
- **Windows**: .NET, Windows-specific builds, WPF/WinForms apps
- **macOS**: iOS/macOS builds, Xcode, Swift

### Do not use a matrix when:
- The matrix would produce fewer than 2 variants — just use a single job
- The variants are fundamentally different operations (put them in separate jobs)
- You need to run different steps per variant — use `if: matrix.os == 'ubuntu-latest'` to conditionally execute steps
