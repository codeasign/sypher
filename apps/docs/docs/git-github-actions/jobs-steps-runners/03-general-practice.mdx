## Tier 1 — Beginner

### Exercise 1: Add a third job

**Task:** Take this workflow and add a `security-scan` job that runs after `test` but before `deploy`:

```yaml
name: Add Security
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Tests passed"
  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying"
```

<details>
<summary>Solution</summary>

```yaml
name: Add Security
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Tests passed"
  security-scan:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Security scan completed"
  deploy:
    needs: [test, security-scan]
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying"
```

Both `security-scan` and `deploy` wait for `test`. The `deploy` job also waits for `security-scan`.
</details>

### Exercise 2: Run steps conditionally

**Task:** Modify the workflow below so that the "Deploy" step only runs when the `github.ref_name` is `main`, but the "Cleanup" step always runs.

```yaml
name: Conditional
on: workflow_dispatch

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - name: Test
        run: echo "Testing..."
      - name: Deploy
        run: echo "Deploying..."
      - name: Cleanup
        run: echo "Cleaning up..."
```

<details>
<summary>Solution</summary>

```yaml
name: Conditional
on: workflow_dispatch

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - name: Test
        run: echo "Testing..."
      - name: Deploy
        if: github.ref_name == 'main'
        run: echo "Deploying..."
      - name: Cleanup
        if: always()
        run: echo "Cleaning up..."
```

The `if: always()` on Cleanup ensures it runs even if Deploy is skipped or fails.
</details>

### Exercise 3: Change the runner OS

**Task:** Change the workflow below to run on `windows-latest` instead of `ubuntu-latest`. Add a step that prints the OS name using `${{ runner.os }}`.

```yaml
name: Runner OS
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."
```

<details>
<summary>Solution</summary>

```yaml
name: Runner OS
on: workflow_dispatch

jobs:
  build:
    runs-on: windows-latest
    steps:
      - run: echo "Building on ${{ runner.os }}"
      - run: echo "Windows-specific commands go here"
        shell: pwsh
```

On `windows-latest`, the default shell is PowerShell (pwsh). The `${{ runner.os }}` expression evaluates to "Windows."
</details>

## Tier 2 — Intermediate

### Exercise 4: Use a container for the job

**Task:** Create a workflow where the `build` job runs inside a Python 3.11 Docker container instead of using `actions/setup-python`. The job should print the Python version and run a simple script.

**Hint:** Use the `container` key at the job level.

<details>
<summary>Solution</summary>

```yaml
name: Container Job
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    container:
      image: python:3.11-slim
    steps:
      - uses: actions/checkout@v4
      - run: python --version
      - run: python -c "print('Hello from inside a Docker container!')"
```

The `container` key makes the job run inside the specified Docker image. All steps execute inside the container.
</details>

### Exercise 5: Use a service container

**Task:** Create a workflow that runs tests against a PostgreSQL database. Use a service container to provide the database and set the `DATABASE_URL` environment variable.

**Hint:** Use the `services` key with a `postgres` image. The service is reachable at `localhost` from the job.

<details>
<summary>Solution</summary>

```yaml
name: Database Test
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: testpass
          POSTGRES_DB: testdb
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
    env:
      DATABASE_URL: postgresql://postgres:testpass@localhost:5432/testdb
    steps:
      - uses: actions/checkout@v4
      - run: echo "Connecting to $DATABASE_URL"
      - run: pip install psycopg2-binary
      - run: python -c "
import psycopg2
conn = psycopg2.connect('$DATABASE_URL')
cur = conn.cursor()
cur.execute('SELECT 1')
print('Connected successfully:', cur.fetchone())
"
```

The service container starts before the job's steps and is stopped after the job completes.
</details>

### Exercise 6: Parallel matrix with exclusive steps

**Task:** Create a matrix workflow that runs on `ubuntu-latest`, `macos-latest`, and `windows-latest`. On Windows, the workflow should run a PowerShell script. On Linux and macOS, it should run a bash script. All three should print the OS name.

**Hint:** Use `if: runner.os == 'Windows'` or `if: matrix.os == 'windows-latest'` to conditionally execute steps.

<details>
<summary>Solution</summary>

```yaml
name: OS-Specific Steps
on: workflow_dispatch

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    steps:
      - uses: actions/checkout@v4
      - name: Common step
        run: echo "Running on ${{ runner.os }}"
      - name: Linux/macOS specific
        if: runner.os != 'Windows'
        run: |
          echo "Unix-specific commands"
          uname -a
      - name: Windows specific
        if: runner.os == 'Windows'
        shell: pwsh
        run: |
          Write-Host "Windows-specific commands"
          Get-ComputerInfo | Select-Object OsName, OsVersion
```

Each OS variant runs the common step, then only the steps matching its OS.
</details>

## Tier 3 — Advanced

### Exercise 7: Fix a broken multi-job workflow

**Scenario:** This workflow is supposed to:
1. Build the project on 3 OSes
2. Run integration tests only on the Linux build
3. Deploy only if all builds and integration tests pass
4. Send a notification regardless of outcome

Find and fix the issues:

```yaml
name: Broken Pipeline
on: push

jobs:
  build:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    steps:
      - uses: actions/checkout@v4
      - run: echo "Building on ${{ runner.os }}"
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-${{ runner.os }}
          path: .

  integration-test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running integration tests..."

  deploy:
    needs: [build, integration-test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying..."

  notify:
    needs: [build, integration-test, deploy]
    runs-on: ubuntu-latest
    steps:
      - run: echo "Sending notification..."
```

<details>
<summary>Solution</summary>

There are at least 3 issues:

1. **Matrix job outputs are hard to reference**: The `build` job is a matrix with 3 variants. When `integration-test` says `needs: build`, it actually depends on ALL 3 build variants. This is correct for the `needs` dependency, but the `deploy` job's artifact download would need to handle multiple artifacts.

2. **No `fail-fast` control**: If one OS build fails, the other matrix variants are cancelled by default (`fail-fast: true` is the default). This may or may not be desired.

3. **Notification does not run if deploy fails**: The `notify` job has `needs: [build, integration-test, deploy]`. If `deploy` fails, `notify` is skipped. It should use `if: always()` to always run.

Fixed version:

```yaml
name: Fixed Pipeline
on: [push, workflow_dispatch]

jobs:
  build:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
      fail-fast: false
    steps:
      - uses: actions/checkout@v4
      - run: echo "Building on ${{ runner.os }}"
      - uses: actions/upload-artifact@v4
        with:
          name: build-${{ runner.os }}
          path: .

  integration-test:
    needs: build
    runs-on: ubuntu-latest
    if: always()
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running integration tests on Linux..."

  deploy:
    needs: [build, integration-test]
    if: success()
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying..."

  notify:
    if: always()
    needs: [build, integration-test, deploy]
    runs-on: ubuntu-latest
    steps:
      - run: echo "Sending notification — pipeline completed"
```

Key fixes:
- `fail-fast: false` prevents one OS failure from cancelling others
- `if: success()` on deploy ensures it only runs when all dependencies passed
- `if: always()` on notify ensures it runs regardless of outcome
</details>

### Exercise 8: Design a build matrix with include/exclude

**Scenario:** Design a matrix strategy that:
- Tests Node.js versions 16, 18, and 20
- Tests on Ubuntu, Windows, and macOS
- Excludes Node 16 on Windows (known incompatibility)
- Adds an extra step for Node 20 on Ubuntu that runs a linting check
- Does not use `fail-fast` (all variants should run independently)

<details>
<summary>Solution</summary>

```yaml
name: Smart Matrix
on: workflow_dispatch

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        node: [16, 18, 20]
        os: [ubuntu-latest, windows-latest, macos-latest]
        exclude:
          - node: 16
            os: windows-latest
        include:
          - node: 20
            os: ubuntu-latest
            lint: true
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: node --version
      - run: npm test
      - name: Lint (Node 20 + Ubuntu only)
        if: matrix.lint
        run: npm run lint
        continue-on-error: true
```

This produces 8 job variants (3x3=9 minus 1 excluded = 8), with only the Ubuntu+Node 20 variant running the lint step.
</details>

### Exercise 9: Optimize a slow pipeline

**Scenario:** Your pipeline takes 18 minutes because it runs everything sequentially:

```yaml
name: Slow Pipeline
on: push

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install flake8 && flake8 .  # 2 min
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest  # 8 min
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: python -m build  # 5 min
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying..."  # 1 min
  notify:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Done"  # 1 min
```

**Constraint:** You cannot change the underlying tools or their runtime. Your redesign must produce the same end result faster through parallelism and job structure.

<details>
<summary>Solution</summary>

The key insight: `lint` has no dependency on `test` or `build`, and `test` and `build` could share the same dependency installation step. But since each job gets its own runner, the real optimization is parallelism:

```yaml
name: Fast Pipeline
on: push

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install flake8 && flake8 .  # 2 min — runs in parallel

  test:
    needs: lint  # Optional — could run in parallel too
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest  # 8 min

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: python -m build  # 5 min — runs in parallel with test

  deploy:
    needs: [test, build]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying..."  # 1 min

  notify:
    if: always()
    needs: [deploy]
    runs-on: ubuntu-latest
    steps:
      - run: echo "Done"  # 1 min
```

By removing `needs: lint` from `test` and `needs: test` from `build`, the wall-clock time drops from 18 min to ~9 min (the max of the parallel paths: `lint` (2 min) + `test` (8 min) = 8 min, `build` (5 min) runs in parallel, then `deploy` (1 min) + `notify` (1 min) = 10 min total).

Further optimization: adding a matrix with OS variants or using `actions/cache` for pip dependencies would reduce the 8 min `test` time further.

Other valid approaches include: running `lint` and `test` in parallel, running `build` on a separate OS matrix, or moving `deploy` into a separate workflow triggered by `workflow_run`.
</details>
