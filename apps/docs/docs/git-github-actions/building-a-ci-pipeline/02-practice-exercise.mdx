You will create a real CI pipeline in a GitHub repository, push changes, and watch the pipeline run in the Actions tab. By the end of this exercise, you will have a working CI pipeline that runs tests and reports status on every push.

## Setup

Before starting, you need:

1. A GitHub account with a repository called `ci-pipeline-practice`. If you do not have one, create it at github.com/new — make it public so you can see the Actions tab (free accounts get unlimited public-action minutes).
2. Git installed and configured (see the Installing Git topic if you need help).
3. Node.js 18+ installed locally (or Python if you prefer — you can adapt the commands).

Clone the repository locally:

```bash
git clone https://github.com/YOUR-USERNAME/ci-pipeline-practice.git
cd ci-pipeline-practice
```

Create a simple project to test:

```bash
# Initialize a Node.js project
npm init -y

# Create a simple test file
mkdir -p .github/workflows
```

Now create `index.js`:

```javascript
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports = { add, subtract };
```

Create `index.test.js`:

```javascript
const { add, subtract } = require('./index');

test('adds 1 + 2 equals 3', () => {
  expect(add(1, 2)).toBe(3);
});

test('subtracts 5 - 3 equals 2', () => {
  expect(subtract(5, 3)).toBe(2);
});
```

Install the test runner:

```bash
npm install --save-dev jest
```

Update `package.json` to add a test script:

```json
"scripts": {
  "test": "jest"
}
```

Verify the tests pass locally:

```bash
npm test
```

Expected output:

```
PASS  ./index.test.js
  ✓ adds 1 + 2 equals 3 (2 ms)
  ✓ subtracts 5 - 3 equals 2 (1 ms)

Tests:        2 passed, 2 total
```

## The Walkthrough

### Step 1: Create the Workflow File

Create `.github/workflows/ci.yml` with the following content:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      - run: npm test
```

This workflow triggers on pushes and pull requests to `main`. It checks out code, installs Node 20, installs dependencies from `package-lock.json`, and runs the test suite.

**Checkpoint:** Verify the file exists:

```bash
ls .github/workflows/
```

Output:
```
ci.yml
```

### Step 2: Commit and Push the Workflow

```bash
git add .
git commit -m "Add CI pipeline with Jest tests"
git push origin main
```

After pushing, navigate to your repository on GitHub and click the "Actions" tab. You should see a workflow run titled "CI" triggered by your commit.

### Step 3: Watch the Pipeline Run

Click on the workflow run to see the live logs. You will see:

```
● build-and-test
  ↓ Post Checkout
  ↓ Setup node
  ↓ Run npm ci
  ↓ Run npm test
```

Each step expands to show its output. The `npm test` step should show:

```
PASS  ./index.test.js
  ✓ adds 1 + 2 equals 3
  ✓ subtracts 5 - 3 equals 2

Tests: 2 passed, 2 total
```

**What happened under the hood:**

1. GitHub detected the `ci.yml` file in `.github/workflows/` on the `main` branch
2. A Ubuntu VM was provisioned from the GitHub-hosted runner pool
3. `actions/checkout@v4` cloned your repository at the exact commit SHA
4. `actions/setup-node@v4` downloaded and installed Node.js 20, adding it to the runner's PATH
5. `npm ci` read `package-lock.json` and installed the exact dependency versions
6. `npm test` ran Jest, which passed all tests, returning exit code 0

### Step 4: Verify the Green Checkmark

After the workflow completes, go back to your repository's main page. Next to your latest commit, you should see a green ✅ checkmark. This is the commit status set by the CI pipeline.

Click the checkmark to see a details panel:

```
CI / build-and-test (ubuntu-latest) — 2m 14s — ✅ Passed
```

This is the same status that appears on pull requests. If you opened a PR, you would see "All checks have passed" before merging.

### Step 5: Make CI Fail (Intentionally)

Now add a failing test to see what CI looks like when it breaks. Add this test to `index.test.js`:

```javascript
test('fails intentionally', () => {
  expect(add(1, 1)).toBe(3);  // 1 + 1 = 2, not 3
});
```

Commit and push:

```bash
git add index.test.js
git commit -m "Add intentionally failing test"
git push origin main
```

Go back to the Actions tab. A new run should be in progress. Wait for it to complete.

Expected output in the `npm test` step:

```
FAIL  ./index.test.js
  ✓ adds 1 + 2 equals 3
  ✓ subtracts 5 - 3 equals 2
  ✕ fails intentionally (2 ms)

  ● fails intentionally

    expect(received).toBe(expected)
    Expected: 3
    Received: 2

Tests: 1 failed, 2 passed, 3 total
```

The workflow run shows a red ❌ status. The commit on the main page also shows a red ✗. No one would merge a PR into main with this status.

### Step 6: Fix the Failing Test and Observe the Green Status

Remove the failing test or fix the assertion:

```javascript
test('fails intentionally', () => {
  expect(add(1, 1)).toBe(2);  // fixed: 1 + 1 = 2
});
```

Commit and push:

```bash
git add index.test.js
git commit -m "Fix intentionally failing test"
git push origin main
```

The pipeline runs again. This time it passes, and the commit status returns to green ✅.

### Step 7: Add a Lint Step

Add a linting step to your CI pipeline. First install ESLint locally:

```bash
npm install --save-dev eslint
npx eslint --init
```

Then update `.github/workflows/ci.yml`:

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20
  - run: npm ci
  - run: npm run build --if-present
  - run: npm run lint  # new step
  - run: npm test
```

Add a lint script to `package.json`:

```json
"scripts": {
  "test": "jest",
  "lint": "eslint index.js"
}
```

Commit and push:

```bash
git add -A
git commit -m "Add linting step to CI pipeline"
git push origin main
```

The pipeline now runs four steps: checkout, setup, lint, and test. Each step is reported independently in the Actions UI.

### Step 8: Add a Timeout and Notification

As a final improvement, add a timeout and a conditional failure notification. Update `ci.yml`:

```yaml
jobs:
  build-and-test:
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint
      - run: npm test

      - name: Notify failure
        if: failure()
        run: |
          echo "Workflow failed — check the logs at:"
          echo "https://github.com/${{ github.repository }}/actions/runs/${{ github.run_id }}"
```

The `if: failure()` condition runs only when a previous step fails. The `github.run_id` variable links directly to the failed run.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add timeout and failure notification step"
git push origin main
```

## Common Errors at This Stage

### Error 1: `npm ci` fails with "package-lock.json not found"

```
npm ERR! cipm can only install packages with an existing package-lock.json or npm-shrinkwrap.json
```

**Why it happens:** `npm ci` requires a `package-lock.json` file. If you ran `npm init -y` but never ran `npm install`, the lockfile does not exist.

**Fix:** Run `npm install` locally to generate the lockfile, then commit and push it:

```bash
npm install
git add package-lock.json
git commit -m "Add package-lock.json"
git push origin main
```

### Error 2: Workflow file is in the wrong directory

```
No workflow files found.
```

**Why it happens:** The workflow file must be at `.github/workflows/ci.yml` — not `.github/workflow/ci.yml` or `github/workflows/ci.yml`.

**Fix:** Move the file to the correct location:

```bash
mkdir -p .github/workflows
mv ci.yml .github/workflows/ci.yml  # if you created it in the wrong place
```

### Error 3: Tests pass locally but fail in CI

```
FAIL  ./index.test.js
  ● Test suite failed to run

    Jest: 'test' is not defined
```

**Why it happens:** CI runs with a clean environment. If you installed Jest globally but forgot to add it as a dev dependency, CI will not find it.

**Fix:** Install Jest as a dev dependency locally, commit the updated `package.json` and `package-lock.json`:

```bash
npm install --save-dev jest
git add package.json package-lock.json
git commit -m "Add jest as dev dependency"
```

### Error 4: "Permission denied" when running scripts

```
+ npm ci
npm ERR! code EACCES
npm ERR! syscall mkdir
npm ERR! path /home/runner/.npm/_cacache
```

**Why it happens:** This is rare on GitHub-hosted runners but can happen when permissions are misconfigured. The runner runs as the `runner` user, which should have write access to its home directory.

**Fix:** Add a step to fix permissions before npm ci:

```yaml
- run: npm cache clean --force
  if: runner.os == 'Linux'
```

- You created a `.github/workflows/ci.yml` file from scratch
- You configured `on: [push, pull_request]` triggers to run on every commit

- You ran `npm ci` and `npm test` as CI steps and saw the live output in the Actions tab
- You intentionally broke the build and observed the red ✗ status on the commit

- You learned how to diagnose and fix common CI pipeline errors
