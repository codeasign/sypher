You just pushed a feature branch to GitHub. Your reviewer pulls it down, runs `npm test`, and finds three failing tests you never saw because you forgot to run the test suite before pushing. Now you wait for their feedback, fix the tests, push again, and wait again. Every round-trip costs 15 minutes of context switching. A CI pipeline catches this before anyone reviews: push triggers an automated build and test run, and the PR shows a red ✗ status within 2 minutes. Your reviewer never sees a broken build because you fix it before requesting review.

## The Concept

A CI pipeline is an automated workflow that builds your code, runs tests, and reports results every time you push changes — without anyone typing a single command. It is the quality gate between "it works on my machine" and "it works in the repository."

Think of it like a factory assembly-line inspection station: every part that comes off the line automatically passes through quality checks before reaching the next station. The CI pipeline is that inspection station for every commit — it runs the same checks, the same way, every single time.

## Core CI Pipeline YAML Keys

Every CI pipeline workflow uses these keys:

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `name` | Workflow display name in the Actions tab | No | `name: CI` |
| `on` | Event trigger that starts the workflow | Yes | `on: [push, pull_request]` |
| `jobs` | One or more jobs that run in parallel or sequence | Yes | `jobs: { build: {...}, test: {...} }` |
| `runs-on` | Runner OS image for the job | Yes per job | `runs-on: ubuntu-latest` |
| `steps` | Sequential commands or actions inside a job | Yes per job | `steps: [uses: actions/checkout@v4, run: npm test]` |
| `uses` | Reference a GitHub Action from the marketplace | No | `uses: actions/setup-node@v4` |
| `with` | Input parameters passed to an action | No | `with: { node-version: 20 }` |
| `run` | Run a shell command directly | No | `run: npm run build` |
| `needs` | Declare job dependency ordering | No | `needs: build` |
| `if` | Conditional execution based on status or expressions | No | `if: failure()` |
| `continue-on-error` | Allow a step to fail without failing the job | No | `continue-on-error: true` |
| `timeout-minutes` | Maximum time a job can run before auto-cancel | No | `timeout-minutes: 10` |
| `strategy` | Matrix or fail-fast configuration | No | `strategy: { matrix: { node: [18, 20] } }` |

## How a CI Pipeline Actually Works — Step by Step

### Step 1: Push Triggers the Workflow

You run `git push origin main`. GitHub receives the push and evaluates every workflow file in `.github/workflows/` against the pushed branch. If a workflow's `on:` trigger matches the event (push, tag, or pull_request), GitHub creates a new workflow run and assigns it a unique run ID.

### Step 2: GitHub Provisions a Runner

GitHub assigns the workflow run to a runner — either a GitHub-hosted virtual machine (Ubuntu, Windows, or macOS) or a self-hosted runner you manage. The runner spins up a clean environment with nothing but the OS pre-installed tools. No cached dependencies, no previous build artifacts — every run starts from a known baseline.

### Step 3: Checkout Action Retrieves Your Code

The first step in almost every CI job is `uses: actions/checkout@v4`. This action clones your repository at the exact commit SHA that triggered the run, checks out the branch, and leaves the working directory ready for subsequent steps.

```yaml
steps:
  - uses: actions/checkout@v4
```

Without this step, your runner has no code to build. Every subsequent command would fail with "no such file or directory."

### Step 4: Install Dependencies

After checkout, you install project dependencies. For a Node.js project, that means `npm ci` (preferred over `npm install` because it uses `package-lock.json` for deterministic installs):

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20
  - run: npm ci
```

`actions/setup-node` installs the specified Node.js version and adds it to `PATH`. Then `npm ci` reads `package-lock.json` and installs exact versions — no version resolution, no differences between your machine and CI.

### Step 5: Run the Build and Tests

With dependencies installed, you run your build command and your test suite:

```yaml
steps:
  - run: npm run build
  - run: npm test
```

If any step returns a non-zero exit code, the runner immediately stops, marks the job as failed, and reports the failure to GitHub. The PR or commit gets a red ✗ status. You can also add a `timeout-minutes` at the job level to prevent a hung test suite from running indefinitely and wasting runner minutes.

### Step 6: Status Reported to GitHub

After all jobs complete, GitHub aggregates their statuses (success, failure, cancelled, or skipped) and reports them to the pull request or commit. If the pipeline configured required status checks under branch protection rules, a failed pipeline blocks merging entirely — the merge button stays gray until CI passes.

## When to Use a CI Pipeline / When Not To

**Use a CI pipeline when:**

- Your project has more than one contributor — consistency across machines is essential
- You have automated tests of any kind — even 5 tests are worth running automatically
- You practice Trunk-Based Development or short-lived feature branches — fast feedback is critical
- You need branch protection (required status checks) to enforce quality gates
- You deploy to production — CI is the prerequisite for CD; green CI is the condition for deploy

**Consider a simpler approach when:**

- You are working on a solo project with no external consumers — local `npm test` before push might be sufficient overhead for a one-person team
- The project is purely documentation or static assets — basic Markdown linting may be all you need, not a full build-and-test pipeline
- Your test suite takes over 30 minutes and rarely fails — in that case, split into fast (gate) and slow (nightly) pipelines rather than having developers wait for the full suite on every push

The rule: if a broken commit would waste someone's time (your future self, a reviewer, or a deploy), add a CI pipeline. The setup cost is one file commit, and the payoff is every broken build caught before it reaches anyone else.

## Full Minimal CI Pipeline Example

Here is the complete workflow file you would place at `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      - run: npm run build --if-present

      - run: npm test
```

This file defines one job (`build-and-test`) that runs on Ubuntu, checks out code, installs Node 20, installs dependencies with `npm ci`, builds if a build script exists, and runs the test suite. It triggers on every push to `main` and every pull request targeting `main`. The entire pipeline must complete in under 10 minutes or GitHub cancels it.
