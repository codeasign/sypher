## Tier 1 — Beginner

### Exercise 1

**Task:** Create a PR from a branch with a single commit. Use `gh pr view` to display the PR details. Then close the PR without merging using `gh pr close`.

<details>
<summary>Hint</summary>
Use `gh pr create` to create, `gh pr view <number>` to view, `gh pr close <number>` to close.
</details>

### Solution

```bash
mkdir -p ~/review-beginner-1 && cd ~/review-beginner-1
git init && echo "init" > f.txt && git add f.txt && git commit -m "Init"
# Create a GitHub repo and push first, then:
git switch -c feature-branch
echo "feature" >> f.txt && git add f.txt && git commit -m "Feature"
git push origin feature-branch
gh pr create --title "Feature" --body "Adds feature" --base main
gh pr view 1
gh pr close 1
```

### Exercise 2

**Task:** Create a PR, submit a review requesting changes, then approve it. Use the `gh pr review` command for both.

<details>
<summary>Hint</summary>
`gh pr review <number> --request-changes --body "..."` then `gh pr review <number> --approve --body "..."`.
</details>

### Solution

```bash
cd ~/review-beginner-1
git switch main
git switch -c fix-branch
echo "fix" >> f.txt && git add f.txt && git commit -m "Fix"
git push origin fix-branch
gh pr create --title "Fix" --body "A fix" --base main
gh pr review 2 --request-changes --body "Please add tests"
# Make changes
echo "tests" > test.txt && git add test.txt && git commit -m "Add tests"
git push origin fix-branch
gh pr review 2 --approve --body "Tests added, approved"
```

### Exercise 3

**Task:** Configure branch protection requiring 1 approval using `gh api`. Verify the setting by attempting to push directly to main (it should be blocked by the branch protection rule if you are not an admin).

<details>
<summary>Hint</summary>
Use `gh api repos/:owner/:repo/branches/main/protection --method PUT` with the JSON body requiring reviews.
</details>

### Solution

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 1
  },
  "enforce_admins": true
}
EOF
# Try to push directly to main:
git checkout main
echo "direct" >> f.txt && git add f.txt && git commit -m "Direct push"
git push origin main  # Should be rejected
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a PR with multiple commits. Use `gh pr diff` to view the full diff, then use `git log main..HEAD` to show the individual commits. Review the diff and identify the commit that introduced a specific change.

<details>
<summary>Hint</summary>
Create two commits on the branch. Use `gh pr diff` for the combined diff. Use `git log --oneline main..HEAD` to see individual commits. Use `git show <hash>` on each commit to see per-commit diffs.
</details>

### Solution

```bash
mkdir -p ~/review-inter-1 && cd ~/review-inter-1
git init && echo "init" > app.py && git add app.py && git commit -m "Init"
# Push to GitHub and create repo
git switch -c multi-commit
echo "def add(a, b): return a + b" >> app.py
git add app.py && git commit -m "Add add function"
echo "def subtract(a, b): return a - b" >> app.py
git add app.py && git commit -m "Add subtract function"
git push origin multi-commit
gh pr create --title "Multi-commit" --body "Multiple commits"
gh pr diff 1
git log main..HEAD --oneline
git show HEAD~1  # Show the first feature commit
```

### Exercise 2

**Task:** Create a PR that has a merge conflict with the base branch. Resolve the conflict locally and push the resolution. Verify the PR becomes mergeable.

<details>
<summary>Hint</summary>
Make a change to `main` after creating the PR branch. Then in the PR branch, `git fetch origin main && git rebase origin/main`, resolve conflicts, `git add`, `git rebase --continue`, `git push --force-with-lease`.
</details>

### Solution

```bash
mkdir -p ~/review-inter-2 && cd ~/review-inter-2
git init && echo "line1" > app.py && git add app.py && git commit -m "Init"
# Push to GitHub
git switch -c conflict-branch
sed -i 's/line1/line1 - branch/' app.py && git add app.py && git commit -m "Branch change"
git push origin conflict-branch
# Now main gets a conflicting change
git switch main
sed -i 's/line1/line1 - main/' app.py && git add app.py && git commit -m "Main change"
git push origin main
# Open PR
gh pr create --title "Conflict PR" --base main
# Resolve conflict
git switch conflict-branch
git fetch origin main
git rebase origin/main
# Resolve conflict in app.py, then:
git add app.py && git rebase --continue
git push origin conflict-branch --force-with-lease
```

### Exercise 3

**Task:** Use `gh pr status` to view all PRs relevant to you (author, reviewer, or mentions). Then use `gh pr list --search` to find PRs by label or state.

<details>
<summary>Hint</summary>
`gh pr status` shows PRs grouped by your role. `gh pr list --search "label:bug"` searches by label. `gh pr list --state merged` shows merged PRs.
</details>

### Solution

```bash
gh pr status
gh pr list --author @me
gh pr list --review-requested @me
gh pr list --search "label:bug"
gh pr list --state merged --limit 5
gh pr list --search "is:open sort:created-desc"
```

## Tier 3 — Advanced

### Exercise 1: Design a CODEOWNERS-based review policy

**Scenario:** Your repository has three teams: `@frontend-team`, `@backend-team`, and `@devops-team`. You want:
- Any changes to `src/frontend/` require approval from `@frontend-team`
- Changes to `src/backend/` require approval from `@backend-team`
- Changes to `deploy/` or `Dockerfile` require approval from `@devops-team`
- Changes to `src/shared/` require approval from both `@frontend-team` and `@backend-team`
- Changes to `README.md` require no review

**Task:** Write the `CODEOWNERS` file that implements this policy. Create the directory structure and files, then verify with `gh api` that the CODEOWNERS file is correctly parsed.

<details>
<summary>Hint</summary>
Create a `.github/CODEOWNERS` file. Use `*` for default owners, directory paths for specific owners. Multiple owners on one line means any of them can approve. Multiple owners across lines means all must approve.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/review-adv-1 && cd ~/review-adv-1
git init
mkdir -p src/frontend src/backend src/shared deploy .github
echo "frontend code" > src/frontend/app.js
echo "backend code" > src/backend/api.py
echo "shared code" > src/shared/utils.py
echo "version: '3'" > deploy/docker-compose.yml
touch Dockerfile README.md
```

Create the CODEOWNERS file:

```bash
cat > .github/CODEOWNERS << 'EOF'
# Default owner for everything
* @core-team

# Frontend changes need frontend team
/src/frontend/ @frontend-team

# Backend changes need backend team
/src/backend/ @backend-team

# DevOps changes need devops team
/deploy/ @devops-team
/Dockerfile @devops-team

# Shared changes need both teams
/src/shared/ @frontend-team @backend-team

# README changes need no review
/README.md
EOF
git add . && git commit -m "Add CODEOWNERS"
```

**Alternative approach:** Use `*` without a team to require no default owner, then add specific paths. Some teams use `* @core-team` as a fallback so nothing is unreviewed.

### Exercise 2: Enforce a squash-merge-only policy with a script

**Scenario:** Your team wants all PRs to be squash-merged to keep a linear history. Not everyone remembers to use `--squash`. You want to automate this by creating a script that checks the merge method and rejects non-squash merges.

**Task:** Write a script that, given a PR number, checks whether the PR can be squash-merged (no conflicts, approved, checks passing). The script should output the mergeability status and the recommended action.

<details>
<summary>Hint</summary>
Use `gh pr view <number> --json mergeable,reviews,statusCheckRollup` to get the PR's state. Check that `mergeable` is `MERGEABLE`, reviews include `APPROVED`, and all status checks are `SUCCESS`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/review-adv-2 && cd ~/review-adv-2
cat > check-pr.sh << 'EOF'
#!/bin/bash
PR_NUM=${1:?Usage: $0 <PR-number>}
echo "Checking PR #$PR_NUM..."
DATA=$(gh pr view $PR_NUM --json mergeable,reviews,statusCheckRollup,title,author 2>/dev/null)
if [ $? -ne 0 ]; then echo "PR #$PR_NUM not found"; exit 1; fi
MERGEABLE=$(echo "$DATA" | jq -r '.mergeable')
APPROVED=$(echo "$DATA" | jq -r '.reviews | map(select(.state == "APPROVED")) | length')
CHECKS=$(echo "$DATA" | jq -r '.statusCheckRollup | map(select(.conclusion != "SUCCESS" and .conclusion != "NEUTRAL")) | length')
echo "Mergeable: $MERGEABLE"
echo "Approvals: $APPROVED"
echo "Failing checks: $CHECKS"
if [ "$MERGEABLE" = "MERGEABLE" ] && [ "$APPROVED" -ge 1 ] && [ "$CHECKS" -eq 0 ]; then
  echo "✓ PR #$PR_NUM is ready for squash-merge"
  echo "Run: gh pr merge $PR_NUM --squash"
else
  echo "✗ PR #$PR_NUM is NOT ready"
fi
EOF
chmod +x check-pr.sh
./check-pr.sh 1
```

**Alternative approach:** Use GitHub Actions to enforce the merge strategy automatically. A workflow can check the merge method and post a status check that blocks merging if the policy is violated.

### Exercise 3: Automate review assignment with a round-robin script

**Scenario:** Your team has 5 reviewers. You want to distribute PR review load evenly — each new PR should be assigned to the reviewer who has the fewest open review requests. You want to automate this.

**Task:** Write a script that:
1. Lists all open PRs
2. For each reviewer, counts how many open review requests they have
3. Assigns the next PR to the reviewer with the fewest open requests
4. Accepts a PR number as an argument and assigns it

<details>
<summary>Hint</summary>
Use `gh pr list --state open --json reviewRequests,number` to get open PRs with review requests. Count per reviewer. Use `gh pr edit <number> --add-reviewer <user>` to assign.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/review-adv-3 && cd ~/review-adv-3
cat > assign-reviewer.sh << 'EOF'
#!/bin/bash
PR_NUM=${1:?Usage: $0 <PR-number>}
REVIEWERS=("reviewer1" "reviewer2" "reviewer3" "reviewer4" "reviewer5")
echo "Finding least-loaded reviewer..."
# Count open review requests per reviewer
declare -A COUNTS
for r in "${REVIEWERS[@]}"; do COUNTS[$r]=0; done
OPEN_PRS=$(gh pr list --state open --json reviewRequests,number --jq '.[] | .reviewRequests[].slug // empty' 2>/dev/null)
for r in "${REVIEWERS[@]}"; do
  COUNT=$(echo "$OPEN_PRS" | grep -c "$r" || true)
  COUNTS[$r]=$COUNT
done
# Find the reviewer with fewest open requests
MIN_COUNT=9999
MIN_REVIEWER=""
for r in "${REVIEWERS[@]}"; do
  echo "  $r: ${COUNTS[$r]} open requests"
  if [ ${COUNTS[$r]} -lt $MIN_COUNT ]; then
    MIN_COUNT=${COUNTS[$r]}
    MIN_REVIEWER=$r
  fi
done
echo "Assigning PR #$PR_NUM to $MIN_REVIEWER"
gh pr edit $PR_NUM --add-reviewer "$MIN_REVIEWER"
EOF
chmod +x assign-reviewer.sh
./assign-reviewer.sh 1
```

**Alternative approach:** Use GitHub Actions with a `pull_request_target` trigger to auto-assign reviewers based on file paths (CODEOWNERS) or a round-robin pool. The script approach is simpler for manual triage; the Actions approach is fully automated.
