You will set up a CODEOWNERS file from scratch, create pull requests that trigger automatic reviewer assignment, and enforce required code owner reviews through branch protection. Every command works on Windows, macOS, and Linux.

## Setup

### Prerequisites

- A GitHub account and a repository you can push to
- GitHub CLI installed: `gh auth login` (run once)
- A second GitHub account or a team you can create for testing (optional -- you can test with a single account using the self-review pattern)

If you do not have a GitHub repo, create one:

```bash
# Create a repo on GitHub (replace 'your-username' with your username)
gh repo create codeowners-lab --public --clone
cd codeowners-lab
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

### Create the directory structure

Create the directories that represent different team ownership areas:

```bash
mkdir -p src/frontend src/backend docs deploy .github
```

Create some starter files so the directories are not empty:

```bash
echo "console.log('hello frontend');" > src/frontend/app.js
echo "def api_handler(): pass" > src/backend/api.py
echo "# Deployment Configuration" > deploy/docker-compose.yml
echo "# Project Documentation" > docs/README.md
echo "version: 1" > .github/settings.yml
```

Commit the initial setup:

```bash
git add .
git commit -m "Initial repo structure with team directories"
git push origin main
```

## The Walkthrough

### Step 1: Create a second branch that changes backend files only

```bash
git switch -c update-backend-api
```

Modify the backend file:

```bash
echo "def api_handler(request): return {'status': 'ok'}" > src/backend/api.py
git add src/backend/api.py
git commit -m "Update backend API handler"
git push origin update-backend-api
```

Do not create a PR yet. You will create the CODEOWNERS file first, then open the PR.

### Step 2: Create the CODEOWNERS file

Now create the `.github/CODEOWNERS` file. This is the core of the exercise:

```bash
cat > .github/CODEOWNERS << 'EOF'
# Default owner for all files -- catch-all rule
* @core-team

# Frontend code belongs to the frontend team
/src/frontend/ @frontend-team

# Backend code belongs to the backend team
/src/backend/ @backend-team

# Deployment configuration belongs to the devops team
/deploy/ @devops-team

# Documentation belongs to the docs team
/docs/ @docs-team

# Specific files can have individual owners
/.github/CODEOWNERS @core-team
EOF
git add .github/CODEOWNERS
git commit -m "Add CODEOWNERS file with per-directory ownership"
git push origin main
```

Note: The CODEOWNERS file was committed to `main`, not to the `update-backend-api` branch. GitHub reads CODEOWNERS from the base branch, so this is the correct approach.

### Step 3: Create a PR that changes backend files only

Push the backend branch and open a PR:

```bash
git switch update-backend-api
git push origin update-backend-api
gh pr create \
  --title "Update backend API handler" \
  --body "Updates the API handler function with request parameter" \
  --base main
```

Now check the reviewers that GitHub automatically assigned:

```bash
gh pr view 1 --json reviewRequests
```

Expected output -- the backend team should be requested as a reviewer:

```json
{
  "reviewRequests": [
    {
      "requestedReviewer": {
        "__typename": "Team",
        "name": "backend-team"
      }
    }
  ]
}
```

If you have not created the `@backend-team` team in your GitHub organisation, you will see an error about the team not being found. This is expected -- GitHub validates that the referenced users and teams exist. For a single-user test, replace `@backend-team` with your own username:

```bash
# Update the CODEOWNERS file to use your username instead of teams
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/frontend/ @your-username
/src/backend/ @your-username
/deploy/ @your-username
EOF
git add .github/CODEOWNERS
git commit -m "Use individual username for CODEOWNERS"
git push origin main
```

### Step 4: Create a PR that changes files in multiple directories

Now create a branch that touches files in multiple ownership areas:

```bash
git switch -c cross-team-change
```

Modify a backend file and a deploy file:

```bash
echo "def new_endpoint(): pass" >> src/backend/api.py
echo "services: app: build: ." > deploy/docker-compose.yml
git add src/backend/api.py deploy/docker-compose.yml
git commit -m "Add new endpoint and update deploy config"
git push origin cross-team-change
```

Open the PR:

```bash
gh pr create \
  --title "Cross-team change: backend + deploy" \
  --body "Adds a new backend endpoint and updates the Docker Compose config" \
  --base main
```

Check the reviewers:

```bash
gh pr view 2 --json reviewRequests
```

Expected output -- both the backend team and the devops team should be requested:

```json
{
  "reviewRequests": [
    {
      "requestedReviewer": {
        "__typename": "Team",
        "name": "backend-team"
      }
    },
    {
      "requestedReviewer": {
        "__typename": "Team",
        "name": "devops-team"
      }
    }
  ]
}
```

This is the power of CODEOWNERS: multiple teams are automatically requested based on the files that changed.

### Step 5: Enable require_code_owner_reviews in branch protection

Now enforce that code owners must approve before merging. Use the GitHub API to set up branch protection:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "require_code_owner_reviews": true,
    "dismiss_stale_reviews": true
  },
  "enforce_admins": true
}
EOF
```

This configures the `main` branch to:
- Require at least one approving review
- Require that the approval comes from a code owner
- Dismiss approvals when new commits are pushed
- Apply these rules to administrators too

You can verify the settings:

```bash
gh api repos/:owner/:repo/branches/main/protection --jq '.required_pull_request_reviews'
```

Expected output:

```json
{
  "required_approving_review_count": 1,
  "require_code_owner_reviews": true,
  "dismiss_stale_reviews": true
}
```

### Step 6: Test that a PR cannot merge without code owner approval

Go back to PR #1 (the backend-only change) and attempt to merge without approving:

```bash
gh pr merge 1 --squash
```

Expected output -- the merge is rejected:

```
X Pull request #1 is not in a mergeable state. Required status check "CODEOWNERS" is failing.
```

Now approve the PR as if you were a code owner. If you are using your own username as the code owner, approve from the same account:

```bash
gh pr review 1 --approve --body "Approved as backend code owner"
```

Now try to merge again:

```bash
gh pr merge 1 --squash --delete-branch
```

Expected output -- the merge succeeds:

```
✓ Merged pull request #1 (update-backend-api)
✓ Deleted branch update-backend-api
```

### Step 7: Add a default owner for everything else

You already have a default `* @core-team` rule in the CODEOWNERS file. But what about files that do not match any specific directory rule? For example, a file in the repository root like `README.md` or a new file in a directory not covered by any rule.

Add a default rule that catches everything not explicitly owned:

```bash
git switch main
cat > .github/CODEOWNERS << 'EOF'
# Default owner for everything
* @core-team

# Frontend code
/src/frontend/ @frontend-team

# Backend code
/src/backend/ @backend-team

# Deployment configuration
/deploy/ @devops-team

# Documentation
/docs/ @docs-team

# Configuration files at root
/*.yml @devops-team
/*.json @devops-team

# The CODEOWNERS file itself
/.github/CODEOWNERS @core-team
EOF
git add .github/CODEOWNERS
git commit -m "Add default owner and root config rules"
git push origin main
```

### Step 8: Test with a PR that changes an unowned file

Create a branch that adds a new type of file not covered by any specific directory rule:

```bash
git switch -c new-config-file
echo "debug: true" > .lintrc.yml
echo "node_modules/" > .gitignore
git add .lintrc.yml .gitignore
git commit -m "Add lint config and gitignore"
git push origin new-config-file
```

Open the PR:

```bash
gh pr create \
  --title "Add new config files" \
  --body "Adds .lintrc.yml and .gitignore at root" \
  --base main
```

Check the reviewers:

```bash
gh pr view 3 --json reviewRequests
```

The `.lintrc.yml` file matches `/*.yml @devops-team` because it is a `.yml` file at the root of the repository. The `.gitignore` file does not match any specific rule, so it falls through to the default `* @core-team` rule. This means the `@devops-team` and `@core-team` are both requested.

If you had not included the `* @core-team` default rule, the `.gitignore` file would have no owner and would not trigger any review request. This is why the default catch-all rule is important -- it ensures no file goes unreviewed.

## Common Errors at This Stage

### Error 1: CODEOWNERS file not found

You see this error when opening a PR:

```
No CODEOWNERS file found on the base branch. No code owner reviews will be requested.
```

**Cause:** The CODEOWNERS file does not exist, or it exists in a location GitHub does not check. GitHub looks in three locations in order: `.github/CODEOWNERS`, `docs/CODEOWNERS`, and `CODEOWNERS` at the root.

**Fix:** Verify the file exists and is in the correct location:

```bash
ls -la .github/CODEOWNERS
```

If the file exists but is in a different location (e.g., `docs/CODEOWNERS`), move it to `.github/CODEOWNERS`:

```bash
mv docs/CODEOWNERS .github/CODEOWNERS
git add .github/CODEOWNERS
git commit -m "Move CODEOWNERS to .github directory"
git push origin main
```

### Error 2: Invalid owner pattern

You see this error when committing the CODEOWNERS file:

```
Invalid CODEOWNERS pattern: "src/backend/ some-team". Owner "some-team" is not a valid user or team reference.
```

**Cause:** The owner reference does not start with `@` and does not match a valid GitHub user or team.

**Fix:** Ensure every owner reference starts with `@` and uses the correct format:

```bash
# Correct -- individual user
/src/backend/ @alice

# Correct -- team in an organisation
/src/backend/ @my-org/backend-team

# Incorrect -- missing @
/src/backend/ backend-team

# Incorrect -- no organisation prefix for team
/src/backend/ @backend-team
```

Teams must be prefixed with the organisation name if the repository belongs to an organisation. For user-owned repositories, only individual usernames are valid.

### Error 3: CODEOWNERS must be in a valid branch

You push a PR that includes a CODEOWNERS file, but the owners are not requested:

```
CODEOWNERS file on the head branch is ignored. The file on the base branch is used instead.
```

**Cause:** GitHub reads CODEOWNERS from the **base branch** (the branch you are merging into, typically `main`), not from the PR branch. This is a security measure -- it prevents a PR author from changing the CODEOWNERS file in their branch to bypass review requirements.

**Fix:** Commit the CODEOWNERS file to the base branch directly:

```bash
git switch main
# Add or update CODEOWNERS here
git add .github/CODEOWNERS
git commit -m "Add CODEOWNERS"
git push origin main
```

Then switch back to your feature branch and rebase if needed.

### Error 4: User or team not found

You see this error when GitHub tries to parse the CODEOWNERS file:

```
Unknown owner on line 3: "@my-org/nonexistent-team". Owner not found.
```

**Cause:** The referenced user or team does not exist on GitHub, or the team exists but does not have access to the repository.

**Fix:** Verify the user or team exists and has access:

```bash
# Check if a team exists (replace my-org and team-name)
gh api orgs/my-org/teams/team-name

# Check if a user exists
gh api users/username
```

Teams must have at least read-level access to the repository for GitHub to be able to request reviews from them. Add the team to the repository:

```bash
gh api repos/:owner/:repo/collaborators/team-name \
  --method PUT \
  --field permission="pull"
```

## What You Just Learned

- The CODEOWNERS file maps file paths to users or teams who are responsible for those files
- GitHub reads CODEOWNERS from the base branch (not the PR branch) for security reasons
- The longest-match-wins algorithm ensures that the most specific rule is applied to each file
- Multiple teams can be requested on a single PR when the change spans multiple ownership areas
- `require_code_owner_reviews` in branch protection enforces that code owners must approve before merging
- The `*` default owner rule acts as a catch-all so no file is left unreviewed
- Teams must be referenced with the organisation prefix (`@org/team-name`) and must have repository access
- When a PR changes files in multiple directories, all matched owners are automatically requested as reviewers
