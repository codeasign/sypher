You push a change to a shared repository. A frontend developer on your team accidentally merges a pull request that modifies the database migration directory without a single DBA review. No one caught the missing foreign key constraint. Three hours later, production goes down. Schema corrupted. Rollback takes forty minutes. The post-mortem reveals the obvious gap: the team relied on manual reviewer assignment, and the DBA was never asked to review.

CODEOWNERS and required reviewers exist to prevent exactly this scenario. They automate the assignment of reviewers based on what files a change touches, and they enforce that those reviewers actually approve before the merge button becomes clickable.

## The Concept

A CODEOWNERS file is a plain text file stored in your repository that maps file paths to individuals or teams who are responsible for those files. When someone opens a pull request that changes files covered by CODEOWNERS rules, GitHub automatically requests reviews from the listed owners. Combined with the `require_code_owner_reviews` branch protection setting, the PR cannot merge until at least one owner from each matched rule has approved.

Think of CODEOWNERS as a guard roster for a building with multiple secure zones. The lobby (root directory) has a general security guard. The server room (deploy directory) requires the sysadmin. The accounting office (finance code) requires the finance team lead. When someone enters a zone, the system checks whether they are authorised for that zone. If they enter multiple zones, the guards for each zone must sign off. The building has a rule: you cannot leave (merge) without every relevant guard's signature.

The required reviewers setting is the lock on the door. CODEOWNERS assigns the guards. But without the lock -- the `require_code_owner_reviews` branch protection rule -- the guards can be bypassed. An administrator could merge without approval. The required reviewers setting ensures that at least one owner from every matched CODEOWNERS rule must approve before the PR can merge.

## Command / Syntax Reference

### CODEOWNERS File Syntax

The CODEOWNERS file must be named `CODEOWNERS` (all caps, no extension) and placed in one of three locations:

| Location | Description |
|---|---|
| `.github/CODEOWNERS` | Recommended location |
| `docs/CODEOWNERS` | Useful for documentation-only repos |
| `CODEOWNERS` (root) | Simplest but can clutter root directory |

Only one CODEOWNERS file is used -- if multiple exist, GitHub looks in the order above and uses the first one found.

### Pattern Rules

| Pattern | Meaning | Example |
|---|---|---|
| `* @team-lead` | Default owner for all files not matched by a more specific rule | `* @core-team` |
| `src/backend/ @backend-team` | All files in `src/backend/` and subdirectories | `/src/frontend/` |
| `docs/*.md @docs-team` | All markdown files directly in `docs/` (not recursive) |  |
| `*.config.* @devops-team` | Any config file anywhere in the repo |  |
| `**/secrets.* @security-team` | Any file named `secrets.*` at any depth |  |
| `/Dockerfile @devops-team` | Single file at root only | `/docker-compose.yml` |
| `*.js @frontend-team` | All JavaScript files at any depth |  |
| `@org/team-name` | Team reference using the `@org/` prefix | `@my-org/backend-devs` |
| `/src/` | Directory-only -- no trailing pattern needed for directory ownership |  |

### Wildcard Reference

| Wildcard | Meaning | Matches |
|---|---|---|
| `*` | Any single file or directory name | `docs/*.md` matches `docs/readme.md` but not `docs/sub/readme.md` |
| `**` | Any number of nested directories | `**/logs` matches `logs/`, `app/logs/`, `src/app/logs/` |
| `?` | Any single character | `config?.json` matches `config1.json` and `config2.json` |
| `[abc]` | Character class | `[Rr]eadme.md` matches both `Readme.md` and `readme.md` |

### Negation Patterns

Prefix a pattern with `!` to exclude files from ownership:

```gitignore
# Everything in docs/ is owned by the docs team
docs/ @docs-team

# Except the release notes, which the product team owns
!/docs/release-notes.md
/docs/release-notes.md @product-team
```

### Comments

Lines starting with `#` are comments:

```gitignore
# Frontend code
/src/frontend/ @frontend-team

# DevOps configuration
/deploy/ @devops-team
```

### Branch Protection Settings for Required Reviewers

| Setting | API key | What it does |
|---|---|---|
| Require pull request reviews | `required_pull_request_reviews` | Enable review requirement on the branch |
| Required approving reviewers | `required_approving_review_count` | Set how many approvals are needed (default 1) |
| Require review from code owners | `require_code_owner_reviews` | Reviews from CODEOWNERS are mandatory |
| Dismiss stale reviews | `dismiss_stale_reviews` | New commits invalidate existing approvals |
| Require approval of the most recent reviewable push | `require_last_push_approval` | Only the most recent commit push is reviewed |
| Include administrators | `enforce_admins` | Apply rules to repository admins too |

## How It Actually Works -- Step by Step

### Step 1: A pull request is opened

When a developer opens (or pushes to) a pull request, GitHub's internal webhook system fires a `pull_request` event. This event contains the full list of files that were added, modified, or deleted in the PR's diff.

GitHub then locates the CODEOWNERS file by checking three locations in order: `.github/CODEOWNERS`, `docs/CODEOWNERS`, and `CODEOWNERS` at the repository root. The file is read from the base branch (usually `main`), not from the PR's head branch. This is intentional -- it prevents a malicious PR author from modifying the CODEOWNERS file in their branch to exempt themselves from review.

### Step 2: Pattern matching algorithm runs

GitHub parses each rule in the CODEOWNERS file from top to bottom. For each file in the PR diff, it checks all patterns and selects the matching rule with the **longest path prefix**. This "longest match wins" algorithm means that a more specific rule like `/src/backend/api/auth.py` overrides a broader rule like `/src/backend/`.

If multiple patterns match the same file at the same specificity level, the last matching rule in the file wins. This means you can place a general catch-all at the top and override specific exceptions lower down.

The algorithm collects the set of all owners from every matched rule across every changed file. A file that matches `/src/backend/ @backend-team` and also `*.py @python-team` (if both rules are active) would require approval from both teams.

### Step 3: Review requests are created automatically

After pattern matching completes, GitHub automatically submits review requests to the matched owners. Each matched user or team appears as a `review requested` notification on the PR.

For team-based ownership (e.g., `@org/backend-team`), GitHub requests review from the team as a whole. Any member of the team can satisfy the review requirement by approving. The team lead sees the request and can assign specific members.

If the same person matches multiple ownership rules (e.g., a developer who is a member of both `@backend-team` and `@core-team`), GitHub only requires one approval from that person -- their single approval counts for all rules they cover.

### Step 4: Branch protection evaluates required reviewers

If branch protection is enabled on the target branch with `require_code_owner_reviews: true`, GitHub adds an additional check to the merge gate: **every code owner whose rule matched at least one changed file must approve the PR**.

This is different from a standard review requirement. Standard review requirements only ask for N approvals from anyone on the team or in the repository. CODEOWNERS-based review requires approvals from **specific** people -- the owners of the files that changed.

The status check `CODEOWNERS review required` appears in the PR's checks section. It remains in a pending state until all matched owners have approved. If no CODEOWNERS rules matched any changed file, this check passes automatically.

The `required_approving_review_count` setting interacts with CODEOWNERS as follows:

| Scenario | Effect |
|---|---|
| `require_code_owner_reviews: false` | CODEOWNERS are auto-requested but non-owners can satisfy the review count |
| `require_code_owner_reviews: true`, count: 1 | One code owner must approve for each matched rule |
| `require_code_owner_reviews: true`, count: 2 | Two approvals needed from code owners of each matched rule |
| No CODEOWNERS file at all | The rule has no effect -- standard review requirement applies |

### Step 5: Approval and merge gate

When a code owner reviews and approves the PR, GitHub checks two things:
1. Does the reviewer belong to at least one of the matched owner teams or match one of the matched owner individuals?
2. Has this reviewer already approved? (Duplicates are not counted twice.)

If the reviewer satisfies the ownership requirement, their approval counts toward the `required_approving_review_count` for each matched rule they cover. Once every matched rule has met its approval threshold, the CODEOWNERS check passes and the PR transitions to `mergeable` status.

If a new commit is pushed to the PR branch and `dismiss_stale_reviews` is enabled, all previous approvals are invalidated. The CODEOWNERS check resets to `pending` and the owners must re-review.

## When to Use This / When Not To

**Use CODEOWNERS and required reviewers when:**

- Your repository has multiple teams or individuals responsible for different areas of the codebase
- You have a monorepo with frontend, backend, infrastructure, and documentation living side by side
- Specific files (secrets, deployment configs, database schemas) require specialised knowledge to review safely
- You want to enforce a least-privilege review model where only the right people can approve changes to sensitive files
- Compliance or regulatory requirements mandate that certain changes get specific sign-off
- Your team is distributed across time zones and manual reviewer assignment leads to delays or missed notifications

**Do not use CODEOWNERS and required reviewers when:**

- Your repository has a single owner or a single team that handles everything -- the overhead of maintaining the file is not worth it
- Your project is in early prototyping or exploration mode where speed matters more than process
- Your team has fewer than three contributors -- the review process may slow everyone down unnecessarily
- The CODEOWNERS file would be too complex to maintain -- if your pattern list exceeds 50 rules, consider splitting the monorepo into smaller repositories

**Best practices:**

- Start with broad rules at the top for the default case, then add specific overrides below
- Use team references (`@org/team-name`) rather than individual usernames so ownership does not break when people leave
- Include a default `* @some-team` rule at the top so no file is left unowned
- Test CODEOWNERS rules by opening a draft PR and checking the reviewers section
- Keep the CODEOWNERS file in version control and review changes to it like any other code change
- Document why each rule exists with comments -- the person who understands today's ownership decisions may not be around tomorrow
- Combine CODEOWNERS with branch protection rules that require status checks for a complete gated delivery pipeline
