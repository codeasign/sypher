## Tier 1 — Beginner

### Exercise 1

**Task:** Create a simulated upstream repo (bare), clone it as a "fork", add the original as the `upstream` remote, and verify both remotes point to the correct paths.

<details>
<summary>Hint</summary>
Create a bare repo with `git init --bare`, clone it to a working copy, make a commit, push. Then clone it again as "your-fork" and add the first location as the `upstream` remote.
</details>

### Solution

```bash
mkdir -p ~/fork-beginner-1 && cd ~/fork-beginner-1
git init --bare upstream.git
git clone upstream.git working && cd working
echo "init" > f.txt && git add f.txt && git commit -m "Init"
git push origin main
cd ..
git clone upstream.git my-fork && cd my-fork
git remote add upstream ../upstream.git
git remote -v
```

Output shows `origin` and `upstream` remotes with the correct paths.

### Exercise 2

**Task:** Create a feature branch in your fork, make a commit, push it. Verify the branch exists on your fork but not on the upstream.

<details>
<summary>Hint</summary>
After pushing, use `git branch -a` to list all branches. Use `git ls-remote origin` to see what is on the remote.
</details>

### Solution

```bash
cd ~/fork-beginner-1/my-fork
git switch -c new-feature
echo "feature" >> f.txt && git add f.txt && git commit -m "New feature"
git push origin new-feature
git ls-remote origin refs/heads/
git ls-remote ../upstream.git refs/heads/
```

`origin` shows both `main` and `new-feature`. `upstream` shows only `main`.

### Exercise 3

**Task:** Simulate upstream accepting a PR by merging the feature branch from your fork into the upstream. Then sync your fork's main with the upstream.

<details>
<summary>Hint</summary>
In the upstream working copy, `git fetch origin` then `git merge origin/new-feature` and push. Then in your fork, `git fetch upstream` + `git merge upstream/main` and push.
</details>

### Solution

```bash
cd ~/fork-beginner-1/working
git fetch origin
git merge origin/new-feature
git push origin main
cd ../my-fork
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
git branch -d new-feature
git push origin --delete new-feature
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Simulate a situation where upstream advances while you are working on a fork feature branch. Rebase your fork branch onto the new upstream main. Push with force-lease.

<details>
<summary>Hint</summary>
In the upstream repo, add a commit to main while your fork has a feature branch. In your fork, `git fetch upstream`, then `git rebase upstream/main` on the feature branch, then `git push origin <branch> --force-with-lease`.
</details>

### Solution

```bash
mkdir -p ~/fork-inter-1 && cd ~/fork-inter-1
git init --bare upstream.git
git clone upstream.git dev && cd dev
echo "C1" > f.txt && git add f.txt && git commit -m "C1"
git push origin main
cd ..
git clone upstream.git fork && cd fork
git remote add upstream ../upstream.git
git switch -c feature
echo "F1" >> f.txt && git add f.txt && git commit -m "Feature"
# Meanwhile, upstream advances
cd ../dev
echo "C2" >> f.txt && git add f.txt && git commit -m "C2"
git push origin main
# Rebase in fork
cd ../fork
git fetch upstream
git rebase upstream/main
git push origin feature --force-with-lease
git log --oneline
```

The feature commit now appears after C2 in the log.

### Exercise 2

**Task:** Create two forks from the same upstream, each with different feature branches. Simulate both submitting PRs. The upstream merges the first PR, then you need to update the second PR's branch to account for the first PR's changes.

<details>
<summary>Hint</summary>
Create the upstream, then two clones as "fork-a" and "fork-b". Each creates a different feature. Upstream merges fork-a's PR. Fork-b then rebases onto upstream/main to incorporate fork-a's changes.
</details>

### Solution

```bash
mkdir -p ~/fork-inter-2 && cd ~/fork-inter-2
git init --bare upstream.git
git clone upstream.git dev && cd dev
echo "base" > f.txt && git add f.txt && git commit -m "Base"
git push origin main
cd ..
git clone upstream.git fork-a && cd fork-a
git remote add upstream ../upstream.git
git switch -c feature-a && echo "A" > a.txt && git add a.txt && git commit -m "Feature A"
git push origin feature-a
cd ..
git clone upstream.git fork-b && cd fork-b
git remote add upstream ../upstream.git
git switch -c feature-b && echo "B" > b.txt && git add b.txt && git commit -m "Feature B"
git push origin feature-b
# Merge A into upstream
cd ../dev && git merge fork-a/feature-a && git push origin main
# Fork B rebases
cd ../fork-b && git fetch upstream && git rebase upstream/main
git log --oneline
```

Fork B's feature commit now sits on top of the merged Feature A.

### Exercise 3

**Task:** Use `gh` CLI (if available) or simulate: create a pull request description file that follows a template with sections for changes, testing, and breaking changes. Write a script that generates this template and saves it.

<details>
<summary>Hint</summary>
Create a `PULL_REQUEST_TEMPLATE.md` with sections. Write a script that takes a branch name and generates the PR body.
</details>

### Solution

```bash
mkdir -p ~/fork-inter-3 && cd ~/fork-inter-3
cat > PULL_REQUEST_TEMPLATE.md << 'EOF'
## Description
<!-- What does this PR do? -->

## Changes
<!-- List the key changes -->

## Testing
<!-- How did you test these changes? -->

## Breaking Changes
<!-- Yes/No — if yes, describe migration path -->

## Related Issues
<!-- Closes #N, Fixes #N -->
EOF
cat > create-pr-body.sh << 'EOF'
#!/bin/bash
TITLE="${1:-Untitled}"
BRANCH=$(git rev-parse --abbrev-ref HEAD)
cat << PRBODY
## Title: $TITLE
## Branch: $BRANCH

**Changes:**
- $(git log --oneline main..HEAD 2>/dev/null || echo "List your changes here")

**Testing:**
- Describe how you tested

**Checklist:**
- [ ] Code compiles
- [ ] Tests pass
- [ ] Linter passes
PRBODY
EOF
chmod +x create-pr-body.sh
./create-pr-body.sh "Fix bugs"
```

## Tier 3 — Advanced

### Exercise 1: Manage multiple upstreams in a fork workflow

**Scenario:** You forked a project. The original upstream has two active branches: `main` (stable) and `develop` (next release). Your feature depends on changes in `develop` that are not yet in `main`. You need to base your feature branch on `develop`, submit a PR to `develop`, and keep your fork synced with both upstream branches.

**Task:** Simulate this multi-branch upstream. Your upstream has `main` and `develop`. Create a feature branch based on `develop`, make a commit, and keep both upstream branches synced in your fork. Do not use `main` as the base for your feature.

<details>
<summary>Hint</summary>
Add both `main` and `develop` branches to the upstream. In your fork, fetch both: `git fetch upstream main develop`. Create your feature from `upstream/develop`: `git switch -c my-feature upstream/develop`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/fork-adv-1 && cd ~/fork-adv-1
git init --bare upstream.git
git clone upstream.git dev && cd dev
echo "stable" > app.txt && git add app.txt && git commit -m "Main init"
git push origin main
git switch -c develop
echo "in progress" > app.txt && git add app.txt && git commit -m "Develop init"
git push origin develop
cd ..
git clone upstream.git fork && cd fork
git remote add upstream ../upstream.git
git fetch upstream main develop
git switch -c my-feature upstream/develop
echo "feature work" >> app.txt && git add app.txt && git commit -m "Feature"
git push origin my-feature
```

The PR base should be `develop`, not `main`. On GitHub, when creating the PR, select `develop` as the base branch.

**Alternative approach:** Use `git branch --set-upstream-to=upstream/develop` to track the correct upstream branch. Then `git pull` will sync with `develop` instead of `main`.

### Exercise 2: Recover from a messy fork with unrelated history

**Scenario:** You accidentally initialized a new repository inside your fork directory, committed, and now your fork has unrelated history to the upstream. When you try to open a PR, GitHub shows thousands of conflicting files.

**Task:** Simulate this scenario. Your fork has a completely separate initial commit with no common ancestor to the upstream. Fix it by either:
1. Resetting your fork to match upstream, then re-applying your changes
2. Using `git rebase --onto` or `git merge --allow-unrelated-histories`

<details>
<summary>Hint</summary>
Use `git log --oneline` to see if the first commit hashes differ. If the histories are unrelated, `git rebase upstream/main` will refuse. Use `git merge --allow-unrelated-histories` to combine them, or start fresh by cloning upstream again and cherry-picking your changes.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/fork-adv-2 && cd ~/fork-adv-2
git init --bare upstream.git
git clone upstream.git dev && cd dev
echo "upstream code" > app.py && git add app.py && git commit -m "Upstream init"
git push origin main
cd ..
# Simulate a broken fork with unrelated history
git init fork && cd fork
git remote add origin ../upstream.git
echo "completely different" > app.py && git add app.py && git commit -m "My init"
git fetch origin
git log --oneline --all
# Histories are unrelated
```

Fix option 1 — merge with unrelated histories:

```bash
git merge origin/main --allow-unrelated-histories
# Resolve conflicts, git add, git commit
git push origin main --force-with-lease
```

Fix option 2 — start fresh and cherry-pick:

```bash
cd ..
rm -rf fork
git clone upstream.git fork && cd fork
# Find your changes from the broken fork's reflog
# Or re-apply manually
```

**Alternative approach:** Use `git rebase --root --onto origin/main` to rebase the entire unrelated history onto the upstream. This is complex but preserves the original commits.

### Exercise 3: Automate fork sync with a script

**Scenario:** You maintain 5 forks of active open-source projects. Each morning, you want to sync all of them with their upstreams in one command. Manually running `git fetch upstream && git merge upstream/main && git push origin main` for each repo is tedious.

**Task:** Write a script that takes a directory containing multiple fork repositories and syncs each one with its upstream. The script should:
1. Find all directories that have a `upstream` remote
2. For each, checkout main, fetch upstream, merge upstream/main, push to origin
3. Report which repos succeeded and which failed
4. Accept a `--dry-run` flag that shows what it would do without executing

<details>
<summary>Hint</summary>
Use a `for d in */; do (cd "$d" && ...)` loop. Check for the `upstream` remote with `git remote get-url upstream 2>/dev/null`. Use `--dry-run` flag with `if [ "$1" = "--dry-run" ]; then echo "Would sync $d"; fi`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/fork-adv-3 && cd ~/fork-adv-3
# Create two test fork repos
for repo in fork-a fork-b; do
  git init --bare upstream-$repo.git
  git clone upstream-$repo.git $repo
  cd $repo
  echo "content" > f.txt && git add f.txt && git commit -m "Init"
  git remote add upstream ../upstream-$repo.git
  cd ..
done
```

The sync script:

```bash
cat > sync-forks.sh << 'EOF'
#!/bin/bash
DRY_RUN=false
if [ "$1" = "--dry-run" ]; then DRY_RUN=true; fi
for dir in */; do
  if [ ! -d "$dir/.git" ] && [ ! -f "$dir/.git" ]; then continue; fi
  echo "=== $dir ==="
  cd "$dir"
  if ! git remote get-url upstream &>/dev/null; then
    echo "  No upstream remote, skipping"
    cd ..; continue
  fi
  echo "  Syncing with upstream..."
  if $DRY_RUN; then
    echo "  [DRY RUN] git fetch upstream"
    echo "  [DRY RUN] git merge upstream/main"
    echo "  [DRY RUN] git push origin main"
  else
    git checkout main 2>/dev/null || git switch main 2>/dev/null
    git fetch upstream && echo "  Fetch OK" || { echo "  Fetch FAILED"; cd ..; continue; }
    git merge upstream/main && echo "  Merge OK" || { echo "  Merge FAILED"; cd ..; continue; }
    git push origin main && echo "  Push OK" || echo "  Push FAILED"
  fi
  cd ..
done
EOF
chmod +x sync-forks.sh
```

Test with dry-run:

```bash
./sync-forks.sh --dry-run
```

**Alternative approach:** Use `gh repo sync <repo>` if you prefer GitHub CLI. The CLI handles the sync for a single fork but does not batch across multiple repos. The script approach works for any number of forks and gives you control over the merge strategy.
