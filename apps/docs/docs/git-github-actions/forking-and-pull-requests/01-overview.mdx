You find an open-source library on GitHub that almost does what you need, but it has a bug in one function. You cannot push a fix directly — you do not have write access to the repository. You need to make a copy of the project under your own account, fix the bug, and request that the original project maintainers pull your changes in. Forking creates your personal copy, and a pull request sends your changes back.

## The Concept

A fork is a server-side copy of a repository under a different owner. You clone your fork, make changes, push them to your fork, and then open a pull request (PR) asking the original repository to pull your changes. The PR is a proposal to merge changes from one branch (your fork) into another branch (the original repo).

Think of forking like photocopying a page from a shared notebook. You write your corrections on your copy (fork), then slide it across the table to the notebook owner saying "here is my fix — would you like to add it to the original?" The owner can accept it as-is, ask for changes, or decline it. Your copy remains yours either way.

## Command / Syntax Reference

| Command / Action | What it does | Example |
|---|---|---|
| `gh repo fork <repo>` | Fork a repository to your GitHub account (GitHub CLI). | `gh repo fork octocat/Spoon-Knife` |
| (GitHub UI) Fork button | Fork a repo via the GitHub web interface. | Click "Fork" at the top-right of any public repo |
| `git remote add <name> <url>` | Add the original repo as an upstream remote. | `git remote add upstream https://github.com/original/repo.git` |
| `git fetch upstream` | Fetch branches and commits from the original repo. | `git fetch upstream` |
| `git merge upstream/main` | Merge the original repo's main branch into your local branch. | `git merge upstream/main` |
| `git rebase upstream/main` | Rebase your branch onto the original repo's latest main. | `git rebase upstream/main` |
| `gh pr create` | Create a pull request from your fork to the original repo. | `gh pr create --title "Fix login bug" --body "..."` |
| `gh pr checkout <number>` | Check out a pull request locally for review. | `gh pr checkout 42` |
| `gh pr list` | List open pull requests. | `gh pr list --state open` |
| `gh pr review <number>` | Submit a review for a pull request. | `gh pr review 42 --approve` |
| `git push origin <branch>` | Push your changes to your fork. | `git push origin fix-login-bug` |

**Windows note:** The GitHub CLI (`gh`) works identically on all platforms. Install it from https://cli.github.com/. On Windows, use the PowerShell or cmd terminal. The `gh` commands for fork, PR create, and PR review are the same across all OSes.

## How It Actually Works — Step by Step

### Step 1: Fork the repository on GitHub

You click the "Fork" button on the original repository's page. GitHub creates a copy of the repository under your account. The fork includes all branches, tags, and commit history. GitHub records the fork relationship so the original repo appears under the "forks" counter.

```
  GitHub.com:
  Original: octocat/repo ──▶ YourFork/repo  (server-side copy)
```

### Step 2: Clone your fork locally

```bash
git clone https://github.com/your-username/repo.git
cd repo
git remote -v
origin  https://github.com/your-username/repo.git (fetch)
origin  https://github.com/your-username/repo.git (push)
```

Your fork is the `origin` remote — where you have push access.

### Step 3: Add the original repo as upstream

```bash
git remote add upstream https://github.com/original-owner/repo.git
git remote -v
origin    https://github.com/you/repo.git (fetch/push)
upstream  https://github.com/original/repo.git (fetch)
upstream  https://github.com/original/repo.git (push — will fail without access)
```

You now have two remotes. `origin` is your fork (you can push). `upstream` is the original (read-only unless you are a maintainer).

### Step 4: Sync your fork's main with upstream

```bash
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
```

Your local main and your fork's main now match the upstream's main. This is important before creating a feature branch so your changes start from the latest upstream code.

```
  upstream/main ──▶ C0 ──▶ C1 ──▶ C2  ← your local main
                                       ← origin/main
```

### Step 5: Create a feature branch and make changes

```bash
git switch -c fix-login-bug
# make changes
git add .
git commit -m "Fix login timeout bug"
git push origin fix-login-bug
```

Your fork now has a branch `fix-login-bug` that the upstream does not have.

```
  origin/fix-login-bug ──▶ F0
  origin/main ───────────▶ C2
```

### Step 6: Open a pull request

Via GitHub CLI:

```bash
gh pr create --title "Fix login timeout bug" --body "Changed the timeout from 30s to 60s to handle slow responses."
```

Or via the GitHub web UI: navigate to your fork, click "Contribute" → "Open Pull Request", select your branch, fill in the title and description, and create the PR.

GitHub presents a comparison view showing:
- Base repository: `original/repo`, branch: `main`
- Head repository: `your-fork/repo`, branch: `fix-login-bug`
- The diff of commits `F0` that are in your branch but not in the base

### Step 7: Code review and merge

Maintainers review your PR. They may request changes. You push more commits to the same branch on your fork — the PR updates automatically.

```bash
# Make requested changes
git add .
git commit -m "Address review: increase timeout to 90s"
git push origin fix-login-bug
```

When approved, a maintainer merges the PR. The commits become part of the upstream repository's main branch.

```
  upstream/main ──▶ C0 ──▶ C1 ──▶ C2 ──▶ F0  (merged PR)
```

Your fork's `fix-login-bug` branch still exists. Delete it after the merge:

```bash
git switch main
git branch -d fix-login-bug
git push origin --delete fix-login-bug
```

## When to Use This / When Not To

**Use fork + PR when:**
- You are contributing to an open-source project where you do not have write access
- Your team uses a fork-based workflow for code review (each developer works from their own fork)
- You want to experiment with changes without affecting the original repository
- You need to contribute across organization boundaries

**Do not use fork + PR when:**
- You have direct write access to the repository — work on a branch in the shared repo instead
- The change is a one-line typo — some projects accept quick fixes via the GitHub web editor directly
- You are the sole maintainer and do not need code review — commit directly to main (or use branches)

**Fork vs. branch:** In a team with shared repo access, working on a branch in the shared repo is simpler than maintaining a fork. Forking adds overhead: you must sync your fork, manage two remotes, and coordinate with upstream. Only fork when you do not have access to the upstream repository. Some large projects (Kubernetes, TensorFlow) require forking regardless of access to maintain the separation between contributor and maintainer roles.
