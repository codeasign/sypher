You will create a repository with multiple workflow files, each triggered by a different event type. By the end you will have workflows that fire on push, pull request, schedule, manual dispatch, and in response to another workflow completing.

## Setup

```bash
# Create a new repository for this practice
gh repo create triggers-practice --public --clone
```

```bash
# Create the workflows directory
mkdir -p .github/workflows
```

```bash
# Create a simple application file
cat > app.py << 'EOF'
def main():
    print("App is running")

if __name__ == "__main__":
    main()
EOF

# Create a test file
cat > test_app.py << 'EOF'
from app import main

def test_main():
    main()

def test_always_passes():
    assert 1 + 1 == 2
EOF
```

```bash
# Create a requirements file
echo "pytest>=7.0" > requirements.txt
```

## The Walkthrough

### Step 1: Push-triggered workflow

Create `.github/workflows/on-push.yml`:

```yaml
name: Push Trigger
on:
  push:
    branches:
      - main
      - develop
    paths:
      - '**.py'
      - 'requirements.txt'

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: python -m pytest test_app.py -v
```

This workflow triggers only when you push Python files or requirements.txt to `main` or `develop`.

**Checkpoint:**

```bash
git add .github/workflows/on-push.yml app.py test_app.py requirements.txt
git commit -m "Add push-triggered workflow"
git push origin main
```

Check the Actions tab. You should see the workflow running within 30 seconds.

### Step 2: Pull request trigger

Create `.github/workflows/on-pr.yml`:

```yaml
name: PR Trigger
on:
  pull_request:
    types:
      - opened
      - synchronize
      - reopened
    branches:
      - main

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install flake8
      - run: flake8 . --exit-zero
```

Now create a branch, push some changes, and open a PR:

```bash
git checkout -b feature/test-pr
```

```bash
echo "# Extra comment" >> app.py
```

```bash
git add app.py
git commit -m "Add comment to app"
```

```bash
git push origin feature/test-pr
```

```bash
# Open a pull request via CLI
gh pr create --base main --head feature/test-pr --title "Test PR" --body "Testing the PR trigger"
```

Check the Actions tab. The "PR Trigger" workflow should be running.

### Step 3: Manual trigger with inputs

Create `.github/workflows/on-dispatch.yml`:

```yaml
name: Manual Deploy
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment target'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production
      version:
        description: 'Version tag to deploy'
        required: true
        default: 'latest'
      dry-run:
        description: 'Dry run (no actual deploy)'
        required: false
        default: false
        type: boolean

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Show deployment parameters
        run: |
          echo "Environment: ${{ github.event.inputs.environment }}"
          echo "Version: ${{ github.event.inputs.version }}"
          echo "Dry run: ${{ github.event.inputs.dry-run }}"
      - name: Simulate deployment
        run: |
          echo "Deploying ${{ github.event.inputs.version }} to ${{ github.event.inputs.environment }}..."
          if [ "${{ github.event.inputs.dry-run }}" = "true" ]; then
            echo "DRY RUN — no actual deployment performed"
          else
            echo "Deployment complete!"
          fi
```

Commit and push:

```bash
git add .github/workflows/on-dispatch.yml
git commit -m "Add manual dispatch workflow"
git push origin feature/test-pr
```

Now go to the Actions tab, select "Manual Deploy," click "Run workflow," choose "production," enter version "v2.1.0," and click "Run workflow."

**Expected output:**

```
Environment: production
Version: v2.1.0
Dry run: false
Deploying v2.1.0 to production...
Deployment complete!
```

### Step 4: Scheduled (cron) trigger

Create `.github/workflows/on-schedule.yml`:

```yaml
name: Nightly Cleanup
on:
  schedule:
    - cron: '0 4 * * *'

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Simulate cleanup
        run: |
          echo "Nightly cleanup started at $(date)"
          echo "Removing stale temp files..."
          echo "Cleanup complete"
```

Cron format: minute, hour, day-of-month, month, day-of-week. `'0 4 * * *'` means "every day at 4:00 AM UTC."

To test a scheduled workflow without waiting for the cron, you can also add `workflow_dispatch` as a fallback:

```yaml
name: Nightly Cleanup
on:
  schedule:
    - cron: '0 4 * * *'
  workflow_dispatch:
```

**Checkpoint:** Manually trigger the workflow via "Run workflow" to verify it works.

### Step 5: Workflow-run trigger (chained workflows)

Create `.github/workflows/on-completion.yml`:

```yaml
name: On CI Complete
on:
  workflow_run:
    workflows:
      - Push Trigger
    types:
      - completed

jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - name: Check predecessor status
        run: |
          echo "Push Trigger workflow completed with status: ${{ github.event.workflow_run.conclusion }}"
          echo "Branch: ${{ github.event.workflow_run.head_branch }}"
          echo "Commit: ${{ github.event.workflow_run.head_sha }}"
```

This workflow runs after the "Push Trigger" workflow completes, regardless of success or failure.

### Step 6: Merge the PR and observe triggers

```bash
git checkout main
git merge feature/test-pr
git push origin main
```

Check the Actions tab. You should see:
- "Push Trigger" runs (because you pushed to main)
- "On CI Complete" runs (because Push Trigger completed)
- The PR is merged, so the PR trigger no longer applies

## Common Errors at This Stage

### Error 1: "No matching workflow trigger"
**Message:** Actions tab shows "This workflow has been triggered but skipped."

**Why:** The event fired, but the filter (branch, path, or type) did not match. For example, you pushed to `feature/x` but the workflow only triggers on `push: branches: [main]`.

**Fix:** Widen the trigger or check the branch name:
```yaml
on:
  push:
    branches:
      - main
      - 'feature/**'  # Wildcard matches feature/test-pr
```

### Error 2: "Scheduled workflow did not run"
**Message:** It is 4:05 AM and your scheduled workflow has not started.

**Why:** Scheduled workflows may be delayed by up to 30 minutes if GitHub's scheduler is busy. They also only run on the default branch.

**Fix:** Add `workflow_dispatch` as a fallback for manual triggering during testing. Check that the cron expression is correct (UTC timezone, not your local time).

### Error 3: `workflow_dispatch` inputs not showing up
**Message:** The inputs form is empty or missing your defined inputs.

**Why:** The workflow file has a YAML syntax error. GitHub could not parse the `workflow_dispatch` block, so it fell back to the default (no inputs).

**Fix:** Validate your YAML indentation. Common mistake:
```yaml
# Bad — inputs is misaligned
on:
  workflow_dispatch:
  inputs:  # Wrong: should be under workflow_dispatch, not at same level

# Good
on:
  workflow_dispatch:
    inputs:
      version:
        ...
```

### Error 4: `pull_request` workflow uses GITHUB_TOKEN to write
**Message:** Step fails with "Resource not accessible by integration."

**Why:** `pull_request` workflows from forks run with a read-only token by default. You need `pull_request_target` to grant write access — but be cautious, as `pull_request_target` runs in the context of the base branch.

**Fix:** For safe PR operations like labeling or commenting, use `pull_request_target`:
```yaml
on:
  pull_request_target:
    types: [opened, labeled]
```

## What You Just Learned

- `push` triggers on git pushes with optional branch and path filters
- `pull_request` triggers on PR lifecycle events with `types` filtering
- `workflow_dispatch` lets you manually trigger workflows with custom inputs
- `schedule` uses cron syntax for time-based triggers
- `workflow_run` chains workflows together
- Filters like `branches`, `paths`, and `types` prevent workflows from running unnecessarily
- The event payload is available via `${{ github.event }}`
