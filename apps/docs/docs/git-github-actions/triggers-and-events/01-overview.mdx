You push a commit to a feature branch. Nothing happens. The CI pipeline is configured to only run on `pull_request`, but you forgot to open a PR. You push again to a different branch вҖ” still nothing. You waste an hour before realizing you typed `branches: [main]` instead of `branches: [main, develop]`. The `on:` key in your workflow file controls when your automation runs, and getting it wrong means your workflow either runs too often or never runs at all.

## The Concept

Triggers (defined by the `on:` key) are GitHub events that cause a workflow run to start. Each event carries a payload of data about what happened вҖ” which branch was pushed, which issue was created, which PR was opened вҖ” that your workflow can access via the `github.event` context.

Think of triggers as the doorbell to your automation house. Different visitors ring the bell in different ways: a push rings once, a pull request rings twice (opened, then synchronized), a scheduled trigger is the mail carrier arriving at the same time every day. Your workflow is the person inside who decides whether to answer based on who is at the door.

## Event Types Reference

| Event | Trigger | Common Filters | Typical Use |
|-------|---------|----------------|-------------|
| `push` | Git push to any branch or tag | `branches`, `tags`, `paths` | CI on every commit |
| `pull_request` | PR opened, synchronized, reopened, etc. | `branches`, `types`, `paths` | PR quality checks |
| `pull_request_target` | PR with write token and base context | `branches`, `types` | Safe PR labeling |
| `workflow_dispatch` | Manual trigger via UI or API | `inputs` | Manual deploy, ad-hoc tasks |
| `schedule` | Cron-based time trigger | None (cron in value) | Nightly builds, cleanup |
| `repository_dispatch` | Custom webhook via API | `types` | External system integration |
| `issue_comment` | Comment on issue or PR | `types` | ChatOps, `/deploy` commands |
| `release` | Release published, created, edited | `types` | Deploy on release |
| `page_build` | GitHub Pages build | None | Deploy Pages site |
| `watch` | Repository starred | `types` | Welcome new stargazers |
| `workflow_run` | Another workflow completes | `workflows`, `types` | Chained workflows |
| `deployment` | Deployment created | None | Track deployments |
| `registry_package` | Package published | None | Package CI |

## How It Actually Works вҖ” Step by Step

### Step 1: An event occurs in the repository

Some action happens in your GitHub repository вҖ” a push, a PR open, a comment. GitHub's internal event system captures this action and creates an event object with a type (e.g., `push`) and a payload (the commit details, branch name, author, etc.).

### Step 2: GitHub checks the event against all workflow files

The event system scans every `.yml` file in `.github/workflows/` of the default branch. For each file, it checks:

1. Does the `on:` key include this event type?
2. If the event has filters (branches, paths, types), does the event match?

For example, if the workflow has `on: push: branches: [main]` and the push is to `feature/x`, the workflow is skipped.

### Step 3: Matching workflows are queued

For each workflow that matches the event, GitHub creates a workflow run record. The run is queued with a status of `pending`. If the event fired multiple workflows (e.g., you have `ci.yml` and `lint.yml` both triggered by push), each one gets its own queued run.

### Step 4: Filter evaluation for complex triggers

For events with `paths` or `paths-ignore` filters, GitHub checks the actual files changed in the push or PR. If the push changed only documentation files and the workflow has `paths-ignore: ['*.md']`, the run is cancelled before it starts.

### Step 5: The `workflow_dispatch` manual trigger

When you click "Run workflow" in the Actions tab, GitHub creates a `workflow_dispatch` event. If the workflow defines `inputs`, the UI shows a form where you can enter values:

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production
      debug:
        description: 'Enable debug mode'
        required: false
        default: false
        type: boolean
```

These inputs are accessible as `${{ github.event.inputs.environment }}` and `${{ github.event.inputs.debug }}`.

### Step 6: The `schedule` trigger

The `schedule` trigger uses POSIX cron syntax with five fields: minute, hour, day-of-month, month, day-of-week. GitHub's scheduler checks every minute and queues runs when the cron expression matches.

```yaml
on:
  schedule:
    - cron: '0 6 * * 1'    # Every Monday at 6:00 AM UTC
    - cron: '0 0 1 * *'    # First day of every month at midnight UTC
```

Scheduled workflows run on the default branch only. They do not run on forks. If a scheduled run is delayed (queued backlog), the delay is shown in the run log.

## When to Use Each Trigger / When Not To

### Use `push` when:
- You want to run tests on every commit
- You need to deploy every time code is pushed to a specific branch

### Do not use `push` when:
- You want to run checks before code is merged вҖ” use `pull_request` instead
- You only want to run on reviewed, approved code вҖ” use `pull_request` with `types: [review_requested]`

### Use `pull_request` when:
- You want to validate PRs before merging
- You need to enforce status checks as branch protection rules

### Do not use `pull_request` when:
- The workflow needs write access to the repository вҖ” use `pull_request_target` instead
- You want to run on every push, not just PRs вҖ” use `push`

### Use `workflow_dispatch` when:
- You want to trigger deployments manually
- You need to pass runtime parameters to a workflow
- You want to run ad-hoc maintenance tasks

### Use `schedule` when:
- You need nightly builds or daily dependency updates
- You want to run cleanup tasks on a regular interval

### Do not use `schedule` when:
- You need exact-second precision вҖ” cron runs are approximate and may be delayed
- You want to trigger on external events вҖ” use `repository_dispatch` instead
