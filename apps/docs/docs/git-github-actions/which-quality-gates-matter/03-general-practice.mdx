## Tier 1 -- Beginner

### Exercise 1: Identify the cheapest gate

You have a pipeline with four possible gates: a code coverage check that runs for thirty seconds, a linting check that runs for eight seconds, an integration test suite that runs for four minutes, and an end-to-end test suite that runs for fifteen minutes.

**Task:** Order these gates from cheapest to most expensive. Which one should run first in the pipeline?

<details>
<summary>Hint</summary>

Think about cost in terms of both time and likelihood of failure. The gate that takes the least time and catches the most common defects should run first. False positives matter too -- a gate that fails for reasons unrelated to your code change is more expensive than its runtime suggests.
</details>

<details>
<summary>Solution</summary>

The correct order from cheapest to most expensive is:
1. Linting -- 8 seconds, catches syntax errors and formatting issues
2. Code coverage -- 30 seconds, but should run after tests
3. Integration tests -- 4 minutes, catches contract violations
4. End-to-end tests -- 15 minutes, highest cost and most flaky

Linting should run first because it catches defects at the lowest cost. If the code has syntax errors, there is no point running any other gate.
</details>

### Exercise 2: Read a pipeline and identify the gate tiers

Look at the following workflow YAML and identify which jobs are tier 1 (fast blocking), tier 2 (medium blocking), and tier 3 (slow informational).

```yaml
name: CI
on: [pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint
  test:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test
  integration:
    needs: [test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:integration
  security:
    needs: [lint]
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm audit
  e2e:
    needs: [integration]
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:e2e
```

<details>
<summary>Hint</summary>

Look at the `needs` chains to see which jobs block others. Look at `continue-on-error` to see which jobs are informational. The jobs without `continue-on-error` are blocking gates.
</details>

<details>
<summary>Solution</summary>

Tier 1 -- Fast blocking: `lint` (runs first, takes seconds, blocks everything)

Tier 2 -- Medium blocking: `test` and `integration` (run after lint, take minutes, block downstream jobs)

Tier 3 -- Slow informational: `security` and `e2e` (have `continue-on-error: true`, do not block the merge)

Note that `security` runs in parallel with `test` because it only needs `lint`, while `integration` runs after `test` because it needs the tests to pass first. The `e2e` gate is the most expensive and has the least blocking power.
</details>

### Exercise 3: Add a `paths` filter to reduce wasted runs

You have a monorepo with `frontend/`, `backend/`, and `docs/` directories. The CI pipeline runs all gates on every pull request, but changes to `docs/` do not need any of the gates.

**Task:** Modify the `on.pull_request` trigger to only run when `frontend/` or `backend/` files change.

<details>
<summary>Hint</summary>

Use the `paths` key under `on.pull_request` to filter which files trigger the workflow. The value is an array of glob patterns.
</details>

<details>
<summary>Solution</summary>

```yaml
on:
  pull_request:
    paths:
      - 'frontend/**'
      - 'backend/**'
      - '.github/workflows/**'
```

This configuration ensures the pipeline runs only when frontend or backend source files change, or when the workflow files themselves change. Changes to `docs/` do not trigger the pipeline.
</details>

### Exercise 4: Identify gate bloat

A team has the following checks in their pipeline: linting, type checking, unit tests, code coverage threshold (80 percent), integration tests, end-to-end tests, a spell-checker for commit messages, a license header check, a Dockerfile linter, a YAML linter, a Markdown link checker, and a performance benchmark.

**Task:** Which three checks would you remove first to reduce gate bloat? Justify each removal.

<details>
<summary>Hint</summary>

Ask yourself for each check: How often does this catch a defect that would reach production? How often does it fail for reasons unrelated to code correctness? How much CI time does it cost?
</details>

<details>
<summary>Solution</summary>

The three checks to remove first:

1. **Spell-checker for commit messages** -- Commit messages are not part of the shipped product. A spelling error in a commit message has zero impact on users or production stability. This gate catches nothing that affects quality and adds noise.

2. **Markdown link checker** -- While broken documentation links are annoying, they are not a quality gate in the CI/CD sense. They do not block shipping a working feature. Move this to a weekly scheduled workflow instead of running it on every PR.

3. **License header check** -- Unless your project is an open-source library with strict license compliance requirements, this check catches nothing related to code correctness. It adds two to three minutes of CI time for zero defect-prevention value.
</details>

## Tier 2 -- Intermediate

### Exercise 1: Design a gating pipeline for a three-service microservices project

You have three microservices: `auth`, `api`, and `worker`. Each service has its own test suite and build step. The `api` service depends on the `auth` service at runtime. The `worker` service is independent. You want a pipeline that:

- Runs linting and type checking for all three services first (cheapest gates)
- Runs tests for each service in parallel after linting passes
- Builds Docker images for each service after tests pass
- Pushes images to a registry only for `auth` and `api` (worker is internal)
- Runs integration tests after `auth` and `api` images are pushed

**Task:** Write the YAML structure with appropriate `needs` chains. Do not write the full step details -- just the job names and their `needs` dependencies.

<details>
<summary>Hint</summary>

Map out the dependency graph on paper first. Jobs that are independent should run in parallel. Jobs that depend on each other should use `needs`. The integration tests need both `auth` and `api` images to be built first.
</details>

<details>
<summary>Solution</summary>

```yaml
jobs:
  lint-all:
    # Runs linting and type check for all three services

  test-auth:
    needs: [lint-all]

  test-api:
    needs: [lint-all]

  test-worker:
    needs: [lint-all]

  build-auth:
    needs: [test-auth]

  build-api:
    needs: [test-api]

  build-worker:
    needs: [test-worker]

  push-auth:
    needs: [build-auth]

  push-api:
    needs: [build-api]
    # No push-worker -- it is internal only

  integration-tests:
    needs: [push-auth, push-api]
```

The key insight is that `test-auth`, `test-api`, and `test-worker` run in parallel after `lint-all`. The `integration-tests` job waits for both `push-auth` and `push-api` to finish. The `build-worker` job does not have a corresponding push job.
</details>

### Exercise 2: Convert a blocking gate to an informational gate

You have a security scan that takes eight minutes and fails about twenty percent of the time due to false positives. Currently, a failure blocks the PR from merging. Your team wants to keep the scan running but stop letting it block merges.

**Task:** Write the YAML change needed to convert this gate to informational. Also write the new `if` condition that makes the downstream deploy job run regardless of the security scan result.

<details>
<summary>Hint</summary>

Two changes are needed: add `continue-on-error: true` to the security scan job, and change the downstream job's `if` condition so it does not require the security scan to succeed.
</details>

<details>
<summary>Solution</summary>

Add `continue-on-error: true` to the security scan job:

```yaml
  security-scan:
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - run: npm audit
```

For the downstream deploy job, change the `if` condition to check for success of the blocking gates only, not the informational gate. The safest approach is to check the blocking gates explicitly:

```yaml
  deploy:
    needs: [lint, type-check, unit-tests, integration-tests, security-scan]
    if: |
      always() &&
      needs.lint.result == 'success' &&
      needs.type-check.result == 'success' &&
      needs.unit-tests.result == 'success' &&
      needs.integration-tests.result == 'success'
    steps:
      - run: deploy.sh
```

This `if` condition ignores the `security-scan` result entirely. If the security scan fails, the deploy job still runs.
</details>

### Exercise 3: Prioritize gates for a hotfix pipeline

Your team has a normal CI pipeline with all gates. For production hotfixes, you want a fast-track pipeline that skips the expensive gates and runs only linting, type checking, and unit tests.

**Task:** Write a workflow that uses `on.workflow_dispatch` with an input parameter called `fast-track`. When `fast-track` is `true`, only the three fast gates run. When it is `false`, all gates run normally.

<details>
<summary>Hint</summary>

Use `workflow_dispatch` inputs to pass the `fast-track` flag. Then use `jobs.<id>.if` conditions to skip expensive jobs when `fast-track` is true. You can also skip concurrency cancellation for hotfixes by conditionally setting the concurrency group.
</details>

<details>
<summary>Solution</summary>

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [main]
  workflow_dispatch:
    inputs:
      fast-track:
        description: 'Skip expensive gates for hotfix'
        required: true
        default: false
        type: boolean

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint

  type-check:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run type-check

  unit-tests:
    needs: [type-check]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  integration-tests:
    needs: [unit-tests]
    if: github.event_name == 'pull_request' || !github.event.inputs.fast-track
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:integration

  e2e-tests:
    needs: [integration-tests]
    if: github.event_name == 'pull_request' || !github.event.inputs.fast-track
    continue-on-error: true
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:e2e
```

When a developer triggers the workflow manually with `fast-track: true`, only the first three jobs run. The integration and E2E tests are skipped entirely. When the workflow runs on a pull request, all gates run normally.
</details>

## Tier 3 -- Advanced

### Exercise 1: Design a pipeline with adaptive gate selection based on change type

You have a monorepo with three areas: `frontend/`, `backend/`, and `infrastructure/`. Each area needs different gates:

- Changes to `frontend/` need linting, type checking, unit tests, and visual regression tests
- Changes to `backend/` need linting, type checking, unit tests, integration tests, and security scanning
- Changes to `infrastructure/` need only a Terraform plan validation and a security scan

A single change may touch multiple areas. If a change touches `frontend/` and `infrastructure/`, both sets of gates should run.

**Task:** Design the `needs` chains and `if` conditions that implement this adaptive gate strategy. Do not write the full YAML -- write only the `jobs` section with the `if` conditions that control job execution.

<details>
<summary>Hint</summary>

Use GitHub Actions `changed-files` to detect which files were modified in a pull request. You can use the `dorny/paths-filter` action or the `tj-actions/changed-files` action to get a list of changed files and set output variables. Then use `jobs.<id>.if` to check those variables.

For the visual regression tests, run them only on frontend changes. For integration tests, run them only on backend changes. Some gates like linting and type checking should run for both frontend and backend changes.
</details>

<details>
<summary>Solution</summary>

```yaml
jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.filter.outputs.frontend }}
      backend: ${{ steps.filter.outputs.backend }}
      infrastructure: ${{ steps.filter.outputs.infrastructure }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            frontend:
              - 'frontend/**'
            backend:
              - 'backend/**'
            infrastructure:
              - 'infrastructure/**'

  lint:
    needs: [detect-changes]
    if: needs.detect-changes.outputs.frontend == 'true' ||
        needs.detect-changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint

  type-check:
    needs: [detect-changes, lint]
    if: needs.detect-changes.outputs.frontend == 'true' ||
        needs.detect-changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run type-check

  unit-tests:
    needs: [detect-changes, type-check]
    if: needs.detect-changes.outputs.frontend == 'true' ||
        needs.detect-changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  visual-regression:
    needs: [detect-changes, unit-tests]
    if: needs.detect-changes.outputs.frontend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:visual

  integration-tests:
    needs: [detect-changes, unit-tests]
    if: needs.detect-changes.outputs.backend == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run test:integration

  security-scan:
    needs: [detect-changes]
    if: needs.detect-changes.outputs.backend == 'true' ||
        needs.detect-changes.outputs.infrastructure == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit

  terraform-plan:
    needs: [detect-changes]
    if: needs.detect-changes.outputs.infrastructure == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: terraform plan
```

This design uses a `detect-changes` job that runs a path filter and exposes the results as outputs. Each subsequent job checks whether it needs to run based on the changed paths. The `security-scan` runs for both backend and infrastructure changes. The `visual-regression` runs only for frontend changes. The `terraform-plan` runs only for infrastructure changes.
</details>

### Exercise 2: Analyze the cost-benefit ratio of a flaky gate

Your team has an end-to-end test suite that runs for twelve minutes and has a ten percent flake rate. It catches a production-critical regression about once every fifty runs.

**Task:** Calculate the expected CI time wasted per regression caught. Then decide whether this gate should be blocking or informational, and justify your decision with concrete numbers.

<details>
<summary>Hint</summary>

Calculate the total CI time spent waiting for this gate over fifty runs. Then calculate the cost of a false positive that reaches production. Compare the two numbers. A flake rate means that one in ten runs fails for non-code reasons.
</details>

<details>
<summary>Solution</summary>

Calculation:

- One regression caught every 50 runs = 1 valuable outcome per 50 executions
- Time per execution: 12 minutes
- Total time spent for 50 runs: 50 times 12 = 600 minutes
- Time per regression caught: 600 minutes divided by 1 = 600 minutes
- Flake rate: 10 percent = 5 flaky failures per 50 runs
- Time wasted on flakes: 5 times 12 = 60 minutes (plus re-run time)
- Total time spent on false positives: 60 minutes per regression caught
- Signal-to-noise ratio: 1 valuable catch to 5 flaky failures

Verdict: This gate should be `continue-on-error: true` unless the cost of a regression reaching production is astronomically high (for example, a medical device or payment system). For most projects, the 600 minutes of CI time per regression caught is not worth the blocking power. Make the gate informational, alert on failure, and investigate only when it consistently fails rather than flaking.

If the team reduces the flake rate from 10 percent to 2 percent, the calculation changes:
- Flaky failures per 50 runs: 1
- Signal-to-noise ratio: 1 valuable catch to 1 flaky failure
- At this improved ratio, the gate could be promoted back to blocking status

This exercise demonstrates why false-positive rate is as important as detection rate when designing quality gates.
</details>

### Exercise 3: Migrate a pipeline from all-sequential to tiered gating

Your team has an existing pipeline where every job runs after the previous one, in a single long chain:

```
lint -> type-check -> unit-tests -> integration-tests -> security-scan -> e2e-tests -> coverage-check
```

Total time: 38 minutes. The team wants to reduce it to under 15 minutes by introducing parallelism where possible, while keeping the same gates.

**Task:** Redesign the job dependency graph. Show the new `needs` chains and calculate the expected wall time for each tier. Assume the following individual job times: lint 8s, type-check 12s, unit-tests 45s, integration-tests 3m, security-scan 5m, e2e-tests 22m, coverage-check 30s.

<details>
<summary>Hint</summary>

Identify which jobs can run in parallel. Security scanning and integration tests do not depend on each other -- they both need the unit tests to pass. Coverage check can run alongside E2E tests. The goal is to maximize parallel execution while respecting true dependencies.
</details>

<details>
<summary>Solution</summary>

New dependency graph:

```yaml
jobs:
  tier1-fast:
    # lint (8s) -> type-check (12s) -> unit-tests (45s)
    # Total: 65 seconds

  tier2-medium-a:
    needs: [tier1-fast]
    # integration-tests (3m)
    # Runs in parallel with tier2-medium-b

  tier2-medium-b:
    needs: [tier1-fast]
    # security-scan (5m)
    # Runs in parallel with tier2-medium-a

  tier3-slow-a:
    needs: [tier2-medium-a, tier2-medium-b]
    # e2e-tests (22m)
    # coverage-check (30s) runs inside same job

  tier3-slow-b:
    needs: [tier2-medium-a, tier2-medium-b]
    # coverage-check if separated (30s)
```

Expected wall time:
- Tier 1: 65 seconds (lint, type-check, unit-tests)
- Tier 2: 5 minutes (integration at 3m and security at 5m run in parallel)
- Tier 3: 22 minutes (E2E tests, coverage check at 30s negligible)

Total expected wall time: 65 seconds + 5 minutes + 22 minutes = approximately 28 minutes if E2E is blocking.

If E2E is made informational (`continue-on-error: true`), the blocking time drops to approximately 6 minutes. The E2E tests continue to run in the background but do not block the merge. This gives the team a 6-minute blocking pipeline compared to the original 38 minutes -- an 84 percent reduction in blocking time.

The key takeaway: making the single most expensive gate (E2E, 22 minutes) informational is the highest-impact change you can make. The tier 1 and tier 2 optimizations save about 2 minutes combined, but the tier 3 change saves 22 minutes.
</details>
