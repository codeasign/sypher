You will create a repository with two divergent branches, perform a fast-forward merge, then perform a true three-way merge with a merge commit. Every command is identical on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/git-merge-lab
cd ~/git-merge-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the starting structure:

```bash
echo "Chapter 1" > report.md
echo "Chapter 2" >> report.md
git add report.md
git commit -m "Initial report"
```

Expected output:

```
[main (root-commit) a1b2c3d] Initial report
 1 file changed, 2 insertions(+)
 create mode 100644 report.md
```

## The Walkthrough

### Step 1: Create a feature branch and make diverging commits

```bash
git switch -c feature-stats
echo "Chart: revenue grew 40 percent in Q2" >> report.md
git add report.md
git commit -m "Add revenue chart"
```

Expected output:

```
[feature-stats c3d4e5f] Add revenue chart
 1 file changed, 1 insertion(+)
```

Now switch back to `main` and make a different change:

```bash
git switch main
echo "Author: Jane Doe" > metadata.txt
git add metadata.txt
git commit -m "Add author metadata"
```

Expected output:

```
[main d4e5f6a] Add author metadata
 1 file changed, 1 insertion(+)
 create mode 100644 metadata.txt
```

Your history now diverges:

```
  ((a1b2c3d)) ──▶ ((d4e5f6a))  ← main
       │                      ← HEAD
       └──▶ ((c3d4e5f))  ← feature-stats
```

### Step 2: Fast-forward merge (main has NOT diverged)

First, let's see what a fast-forward looks like. Create a second branch that has not diverged from `main`:

```bash
git switch -c hotfix
echo "Urgent fix" >> report.md
git add report.md
git commit -m "Hotfix applied"
git log --oneline --graph --all
```

Expected output:

```
* e5f6a7b (HEAD -> hotfix) Hotfix applied
* d4e5f6a (main, feature-stats) Add author metadata
* a1b2c3d Initial report
```

`hotfix` is one commit ahead of `main` with no divergence. Now merge it:

```bash
git switch main
git merge hotfix
```

Expected output:

```
Updating d4e5f6a..e5f6a7b
Fast-forward
 report.md | 1 +
 1 file changed, 1 insertion(+)
```

Git says "Fast-forward" — it simply moved the `main` pointer forward. No merge commit was created.

```
  ((a1b2c3d)) ──▶ ((d4e5f6a)) ──▶ ((e5f6a7b))  ← main, hotfix
                                                ← HEAD
```

**Checkpoint:** Run `git log --oneline` — you should see a linear chain with no merge commit.

### Step 3: Create true divergence

Delete the `hotfix` branch (it is now merged) and make a new commit on `main` to create real divergence from `feature-stats`:

```bash
git branch -d hotfix
echo "Conclusion" >> report.md
git add report.md
git commit -m "Add conclusion"
git log --oneline --graph --all
```

Expected output:

```
* f6a7b8c (HEAD -> main) Add conclusion
* e5f6a7b Hotfix applied
* d4e5f6a Add author metadata
| * c3d4e5f (feature-stats) Add revenue chart
|/
* a1b2c3d Initial report
```

The fork is visible — `main` and `feature-stats` have diverged from `d4e5f6a`. Neither is a direct ancestor of the other.

```
  ((a1b2c3d)) ──▶ ((d4e5f6a)) ──▶ ((e5f6a7b)) ──▶ ((f6a7b8c))  ← main
       │                                                       ← HEAD
       └──▶ ((c3d4e5f))  ← feature-stats
```

### Step 4: Perform a true three-way merge

Now merge `feature-stats` into `main`:

```bash
git merge feature-stats
```

Expected output:

```
Merge made by the 'ort' strategy.
 report.md | 1 +
 1 file changed, 1 insertion(+)
```

(Note: `ort` is Git's modern merge strategy, replacing the old `recursive` strategy.)

This creates a merge commit. Verify it:

```bash
git log --oneline --graph --all
```

Expected output:

```
*   g7h8i9j (HEAD -> main) Merge branch 'feature-stats'
|\
| * c3d4e5f (feature-stats) Add revenue chart
* | f6a7b8c Add conclusion
* | e5f6a7b Hotfix applied
* | d4e5f6a Add author metadata
|/
* a1b2c3d Initial report
```

```
  ((a1b2c3d)) ──▶ ((d4e5f6a)) ──▶ ((e5f6a7b)) ──▶ ((f6a7b8c)) ──▶ ((g7h8i9j))  ← main
       │                                                /            ← HEAD
       └──▶ ((c3d4e5f)) ──────────────────────────────
```

The merge commit `g7h8i9j` has two parents: `f6a7b8c` (the previous `main` tip) and `c3d4e5f` (the `feature-stats` tip). Its snapshot includes both the conclusion added on `main` and the revenue chart added on `feature-stats`.

**Checkpoint:** Look at the merged file:

```bash
cat report.md
```

Expected output:

```
Chapter 1
Chapter 2
Urgent fix
Conclusion
Chart: revenue grew 40 percent in Q2
```

The content from both branches is present.

### Step 5: Force a merge commit (no fast-forward)

Even when a fast-forward is possible, you can force a merge commit:

```bash
git switch -c no-ff-test
echo "Test line" >> report.md
git add report.md
git commit -m "Test line"
git switch main
git merge --no-ff no-ff-test
```

Expected output:

```
Merge made by the 'ort' strategy.
 report.md | 1 +
 1 file changed, 1 insertion(+)
```

Notice Git created a merge commit even though a fast-forward was technically possible:

```bash
git log --oneline --graph -3
```

Expected output:

```
*   h8i9j0k (HEAD -> main) Merge branch 'no-ff-test'
|\
| * i9j0k1l (no-ff-test) Test line
|/
*   g7h8i9j Merge branch 'feature-stats'
```

`--no-ff` is useful when you want to mark clearly in the history that a feature was merged, preserving the branch structure even for single-commit branches.

### Step 6: Squash merge

Sometimes you want the feature's changes without importing all its commits:

```bash
git switch -c squash-test
echo "Squash line" >> report.md
echo "Another line" >> notes.txt
git add report.md notes.txt
git commit -m "Squash first change"
echo "Third line" >> notes.txt
git add notes.txt
git commit -m "Squash second change"
git switch main
git merge --squash squash-test
```

Expected output:

```
Automatic merge went well; stopped before committing as requested.
Squash commit -- not updating HEAD
```

The squash staged all changes from `squash-test`'s two commits as a single set of changes in the staging area:

```bash
git status
git diff --cached
git commit -m "Add all squash-test changes in one commit"
```

Now `main` has one commit containing all the changes from `squash-test`'s two commits, with no merge commit and no branch history imported.

## Common Errors at This Stage

### Error 1: `fatal: You have not concluded your merge (MERGE_HEAD exists)`

You tried to run a Git command while a merge was in progress (possibly with conflicts). Git prevents any other operation until you resolve or abort the merge.

**Fix:** Either resolve the conflicts, stage the resolved files, and run `git merge --continue`, or abort with `git merge --abort` to return to the pre-merge state.

### Error 2: `fatal: refusing to merge unrelated histories`

You tried to merge two branches that do not share a common ancestor (e.g., merging from two different repositories).

**Fix:** If you intentionally want to merge unrelated histories, add `--allow-unrelated-histories` to the merge command. This is rare — only use it when you know the branches were meant to be combined.

### Error 3: `Auto-merging <file> CONFLICT (content): Merge conflict in <file>`

Git could not automatically merge changes because both branches modified the same lines in the same file.

**Fix:** Do not panic. Edit the conflicted file to resolve the markers (`<<<<<<<`, `=======`, `>>>>>>>`), stage it with `git add`, and continue with `git merge --continue`.

### Error 4: Merge succeeds but tests fail

The merge completed without textual conflicts, but the combined code does not work correctly. This is a semantic conflict — Git cannot detect logic bugs.

**Fix:** Always run your test suite after a merge. If tests fail, fix the code on `main` and commit the fix. The merge commit already exists; the fix commit will be a child of it.

## What You Just Learned

- A fast-forward merge moves the branch pointer forward with no new commit
- A true merge creates a commit with two parents after a three-way comparison
- `git merge --no-ff` forces a merge commit even when fast-forward is possible
- `git merge --squash` stages all source commits as one diff without merging history
- The merge base is the most recent common ancestor of two branches
- Three-way merge compares base, ours, and theirs to decide each file's result
- The `ort` strategy is Git's modern default merge engine
