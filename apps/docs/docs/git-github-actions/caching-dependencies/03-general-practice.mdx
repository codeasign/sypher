Complete these exercises on your own to prove you can configure, tune, and debug dependency caching in CI pipelines.

## Tier 1 — Beginner

### Exercise 1: Cache pip Dependencies for a Python Project

**Task:** Create a CI pipeline for a Python project that caches pip-installed packages between runs. Use the Python cache path `~/.cache/pip` and key it on `requirements.txt`.

**Hint:** Use `actions/cache@v4` with `path: ~/.cache/pip` and `key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}`.

**Solution:**

```yaml
name: Python CI with Cache

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Cache pip
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - run: pip install -r requirements.txt
      - run: pytest
```

The pip cache is stored at `~/.cache/pip`. When `requirements.txt` has not changed, the cache is restored and pip installs from the local cache rather than downloading from PyPI. This typically reduces install time from 60 seconds to 10 seconds.

jobs:

      - name: Cache Go modules

## Tier 2 — Intermediate

### Exercise 1: Cache with restore-keys for Partial Hit

**Task:** Create a CI pipeline for a Node.js project that uses both an exact cache key on `package-lock.json` and a `restore-keys` fallback to `npm-${{ runner.os }}-` for partial cache hits.

**Hint:** Add `restore-keys: | ${{ runner.os }}-npm-` to the cache step.

**Solution:**

```yaml
name: Node CI with Partial Cache

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Restore npm cache
        id: cache
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-npm-

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - if: steps.cache.outputs.cache-hit != 'true'
        run: npm ci

      - run: npm test
```

When the exact key misses (e.g., you added a single dependency), the `restore-keys` fallback matches the most recent cache starting with `linux-npm-`. This restores all previously cached packages, and `npm ci` only downloads the new dependency. Without `restore-keys`, `npm ci` would re-download all 200 packages.

### Exercise 2: Cache Multiple Paths for a Mixed Project

**Task:** Create a CI pipeline for a project that uses both npm and pip (e.g., a Node.js frontend with a Python backend). Cache both `~/.npm` and `~/.cache/pip` in the same workflow.

**Hint:** Use a multi-line `path` with `|` and separate cache keys for each ecosystem.

**Solution:**

```yaml
name: Mixed Project Cache

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Cache npm and pip
        id: cache
        uses: actions/cache@v4
        with:
          path: |
            ~/.npm
            ~/.cache/pip
          key: ${{ runner.os }}-deps-${{ hashFiles('package-lock.json', 'requirements.txt') }}

      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - if: steps.cache.outputs.cache-hit != 'true'
        run: |
          npm ci
          pip install -r requirements.txt

      - run: npm test
      - run: pytest
```

Multiple paths are specified with the YAML block scalar `|`. The cache key is computed from both `package-lock.json` and `requirements.txt` — if either changes, the key changes. Both directories are cached in a single cache entry.

### Exercise 3: Use `save-always` to Cache Despite Test Failures

**Task:** Create a CI pipeline that caches dependencies even when the test step fails. The cache should be saved regardless of the job's final status.

**Hint:** Add `save-always: true` to the cache step.

**Solution:**

```yaml
name: Resilient Cache

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Cache dependencies
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
          save-always: true

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci
      - run: npm test  # may fail, but cache is still saved
```

Without `save-always`, the cache is only saved when the job succeeds. If the test step fails, the cache is discarded and the next run starts from scratch. With `save-always: true`, the dependency cache survives test failures, so the next run (after fixing the test) benefits from the saved cache.

## Tier 3 — Advanced

### Exercise 1: Debug a Cache Misconfiguration

**Scenario:** Your team has this caching workflow. It is supposed to speed up builds, but the cache is never hit — every run is a cache miss. Diagnose and fix.

```yaml
name: CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/cache@v4
        with:
          path: ~/.npm
          key: npm-cache
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
```

**Issues to identify:**

1. **Static cache key:** The key `npm-cache` is a literal string. It never changes, so every run should use the same cache. But the cache was never saved because the cache step runs before the code is checked out — `npm ci` is not in the cache path, so the cache is always empty.
2. **Cache step before checkout:** `actions/cache@v4` runs before `actions/checkout@v4`. The cache step will restore an empty cache (or no cache) because the dependencies have not been installed yet. The cache step should run after setup, but before the install step.
3. **No hash of lockfile:** The key should include `hashFiles('package-lock.json')` to create a new cache entry when dependencies change.

**One valid fix:**

```yaml
name: CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache npm
        id: cache
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-npm-

      - if: steps.cache.outputs.cache-hit != 'true'
        run: npm ci

      - run: npm test
```

### Exercise 2: Design a Caching Strategy for a Monorepo with NPM Workspaces

**Scenario:** You have a monorepo with 10 npm workspaces (each in its own directory). The root `package-lock.json` covers all workspaces. `npm ci` takes 4 minutes. Design a caching strategy that minimizes install time while handling the fact that any workspace's `package.json` can change independently.

**Approach:**

There is no single correct answer, but a strong approach uses the root lockfile for the primary cache key and adds a composite hash of all workspace `package.json` files:

```yaml
name: Monorepo CI

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Compute workspace hash
        id: hash
        run: |
          # Hash all workspace package.json files + root lockfile
          cat packages/*/package.json package-lock.json | sha256sum > /tmp/deps-hash
          echo "key=$(cat /tmp/deps-hash)" >> $GITHUB_OUTPUT

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache node_modules
        uses: actions/cache@v4
        with:
          path: node_modules
          key: ${{ runner.os }}-monorepo-${{ steps.hash.outputs.key }}
          restore-keys: |
            ${{ runner.os }}-monorepo-

      - run: npm ci
      - run: npm test
```

Alternative valid approaches include:
- Caching `~/.npm` (smaller, more portable, but `npm ci` still runs linking step)
- Using `actions/cache` with `restore-keys` for each workspace individually
- Using a tool like `nx` or `turborepo` that has built-in remote caching

The key trade-off is cache granularity vs. complexity. A single cache for the entire monorepo is simple but invalidated by any workspace change. Per-workspace caching is more efficient but significantly more complex to set up.

### Exercise 3: Cache Across Branches with a Shared Fallback

**Scenario:** You have a CI pipeline that runs on every branch. Each branch has its own `package-lock.json` that may differ slightly. Design a caching strategy where:
- Each branch gets its own cache key (fastest for that branch)
- If a branch's cache misses, it falls back to the `main` branch's cache
- The `main` branch cache is always the most up-to-date

**One valid approach:**

```yaml
name: CI with Cross-Branch Cache

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache with branch-aware key
        id: cache
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}-${{ github.ref_name }}
          restore-keys: |
            ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}-
            ${{ runner.os }}-npm-

      - if: steps.cache.outputs.cache-hit != 'true'
        run: npm ci

      - run: npm test
```

The cache key includes the branch name. The first `restore-keys` tries to match without the branch name (matching any branch with the same lockfile hash). The second `restore-keys` matches any lockfile on the same OS. This ensures that even if a feature branch has never been cached, it benefits from the `main` branch's cache (or any other branch's cache with the same dependencies).
