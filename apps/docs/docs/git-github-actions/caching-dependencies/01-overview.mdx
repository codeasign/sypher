Your CI pipeline runs `npm ci` on every push, downloading 150 MB of dependencies from scratch each time. A push-to-fix-the-typo cycle that takes 45 seconds of actual work wastes 2 minutes waiting for `npm install`. Over 20 pushes a day, that is 40 minutes of total CI wait time — just on installing the same modules that did not change. Caching dependencies eliminates this redundant download and cuts CI execution time by 50–80 percent on most projects.

## The Concept

Caching stores downloaded or compiled dependencies between workflow runs so that subsequent runs reuse them instead of re-downloading or re-compiling from scratch. A cache is identified by a key — when the key changes (because your lockfile changed), a new cache is created and the old one is eventually evicted.

Think of it like a local grocery store's delivery schedule: if you order the same staples every week (milk, eggs, bread), the store pre-packs your standard order and you pick it up in 2 minutes. When you change your order (switch to oat milk), the first delivery takes the full 10 minutes because the store needs to pick the new items. Your next week is fast again.

## Core Caching YAML Keys

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `actions/cache@v4` | Action that saves and restores cache entries | Yes | `uses: actions/cache@v4` |
| `key` | Unique cache identifier, typically derived from lockfile hash | Yes | `key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}` |
| `path` | Directory to cache (relative to workspace or absolute) | Yes | `path: ~/.npm` |
| `restore-keys` | Fallback keys to try when the exact key misses | No | `restore-keys: npm-${{ runner.os }}-` |
| `lookup-only` | Check cache existence without restoring | No | `lookup-only: true` |
| `save-always` | Save cache even if the job fails | No | `save-always: true` |
| `enableCrossOsArchive` | Allow cache to work across different OS runners | No | `enableCrossOsArchive: true` |

## How Caching Actually Works — Step by Step

### Step 1: Cache Key Computation

When the workflow reaches the `actions/cache@v4` step with a `restore` action (the default on first usage), GitHub computes the cache key by evaluating all `${{ }}` expressions. The key is a string like `linux-npm-abc123def456`.

The most important expression for dependency caching is `hashFiles()`, which produces an SHA-256 hash of one or more files:

```yaml
- uses: actions/cache@v4
  id: npm-cache
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
```

If `package-lock.json` has not changed since the last run, `hashFiles()` produces the same hash, and the computed key matches a stored cache entry. If the lockfile changed (new dependency added), the hash changes and the key is different — producing a cache miss.

### Step 2: Cache Restore (Hit or Miss)

GitHub checks its global cache store for an entry matching the computed key. The cache store is regional (us-east-1 by default for most repos) and has a max total size of 10 GB per repository.

- **Hit:** The cache is downloaded and extracted to the path specified (`~/.npm`). Subsequent `npm ci` finds the cached packages and skips the network download entirely.
- **Miss:** The step completes without restoring anything. You can check the outcome with the action's output:

```yaml
- name: Cache check
  id: npm-cache
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}

- name: Install dependencies
  run: npm ci
  if: steps.npm-cache.outputs.cache-hit != 'true'
```

Using `cache-hit` output lets you skip `npm ci` entirely when the cache restores successfully — nothing to install because everything is already in `~/.npm`.

### Step 3: `npm ci` with Cached Packages

When `~/.npm` is populated from cache, `npm ci` still runs its full algorithm (resolving the dependency tree, building the `node_modules` directory structure) — but it skips the actual download of every package. For a project with 200 dependencies, this reduces install time from ~90 seconds to ~10 seconds on a typical runner.

The key difference between `npm ci` with and without cache:

```
Without cache:  npm ci  90.3s  (downloads 200 packages from registry)
With cache:     npm ci  10.1s  (copies 200 packages from ~/.npm cache)
```

### Step 4: Cache Save (Post-Job)

After the job completes successfully, GitHub runs a post-job action that archives the specified `path` into a compressed tarball, uploads it to the global cache store, and associates it with the computed key. This tarball is stored for up to 7 days since last access. If the repository reaches the 10 GB cache limit, the least-recently-used entries are evicted.

```yaml
- uses: actions/cache@v4
  with:
    path: |
      ~/.npm
      node_modules
    key: npm-${{ hashFiles('package-lock.json') }}
    save-always: true
```

`save-always: true` forces the cache to save even if a later step in the job fails. This is useful for long-running builds where you want to salvage the dependency cache even when the test step fails.

### Step 5: Restore Keys for Partial Matches

Sometimes the exact cache key misses (because the lockfile changed), but you still want to reuse part of the old cache. `restore-keys` provides fallback patterns:

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: npm-${{ hashFiles('package-lock.json') }}
    restore-keys: |
      npm-${{ runner.os }}-
```

When the exact key misses, GitHub tries each restore key in order. `restore-keys` matches the longest prefix. For example, if the key is `npm-Linux-abcd` and no cache with that key exists, GitHub looks for any cache starting with `npm-Linux-` and restores the most recent match. This is useful when rebuilding the full cache is expensive but a stale cache is still faster than downloading everything fresh.

## When to Use Caching / When Not To

**Use caching when:**

- Your dependency install takes more than 10 seconds and the lockfile changes infrequently
- You have multiple CI jobs that share the same dependencies (same language ecosystem)
- Your build produces intermediate artifacts that are expensive to rebuild (compiled C++ extensions, Docker layers)

**Consider skipping caching when:**

- Your total dependency install time is under 5 seconds — the cache lookup overhead (~1–2 seconds) may not be worth it
- Your lockfile changes on every commit (like a monorepo with daily dependency updates) — the cache hit rate will be near zero
- You are already using Docker layer caching for your build — Docker's own layer cache may handle dependency caching more efficiently for containerized builds

The rule: cache when the lockfile changes less often than commits. A 90 percent cache hit rate saves significant time; a 10 percent hit rate adds complexity for marginal gain.

## Full Caching Example

```yaml
name: CI with Caching

on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache npm dependencies
        id: npm-cache
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-npm-

      - if: steps.npm-cache.outputs.cache-hit != 'true'
        run: npm ci

      - run: npm test
```

This workflow caches `~/.npm` based on a hash of `package-lock.json`. On a cache hit, `npm ci` is skipped entirely. On a miss (or when the lockfile changes), `npm ci` runs fully and the new cache is saved for the next run. A `restore-keys` fallback ensures that even a stale cache is better than no cache when the lockfile increments a minor version.
