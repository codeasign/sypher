You will set up a CI pipeline with dependency caching, measure the speed difference between cached and uncached runs, and learn to fine-tune cache keys for maximum hit rate.

## Setup

Create a repository on GitHub called `caching-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/caching-practice.git
cd caching-practice
```

Set up a project with enough dependencies to make caching worthwhile:

```bash
npm init -y
mkdir -p .github/workflows
```

Create a `package.json` with several dependencies:

```bash
npm install express lodash axios dayjs chalk
npm install --save-dev jest eslint prettier typescript ts-jest @types/node
```

This installs enough packages (~50 MB) that the difference between cached and uncached installs will be noticeable.

Create a minimal test file `index.test.js`:

```javascript
test('environment is ready', () => {
  expect(true).toBe(true);
});
```

Update `package.json`:

```json
"scripts": {
  "test": "jest"
}
```

Verify tests pass:

```bash
npm test
```

## The Walkthrough

### Step 1: Create a CI Pipeline Without Caching

Create `.github/workflows/ci.yml`:

```yaml
name: CI No Cache

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
```

Commit and push:

```bash
git add .
git commit -m "Add CI without caching"
git push origin main
```

Note the total run time. It will be approximately 2–3 minutes, with most of that time spent on `npm ci`. Remember this number — you will compare it with the cached version.

### Step 2: Add a Cache Step

Now add caching for `~/.npm` (the npm global cache directory):

```yaml
name: CI with Caching

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Cache npm dependencies
        id: npm-cache
        uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - if: steps.npm-cache.outputs.cache-hit != 'true'
        run: npm ci

      - run: npm test
```

The key uses `hashFiles('package-lock.json')` — a hash of the lockfile contents. When the lockfile changes (new dependency added), the hash changes, producing a cache miss. The `if` condition on `npm ci` skips the install entirely when the cache is restored successfully.

**Checkpoint:** Verify the workflow file exists:

```bash
cat .github/workflows/ci.yml
```

### Step 3: Measure the Cached Run

Commit and push the caching workflow:

```bash
git add .github/workflows/ci.yml
git commit -m "Add npm dependency caching"
git push origin main
```

The first run after adding caching will be a **cache miss** — the cache does not exist yet, so `npm ci` runs fully. The cache is saved after the job completes.

Push again to trigger a **cache hit**:

```bash
git commit --allow-empty -m "Trigger cached run"
git push origin main
```

Compare the run times:

| Run | npm ci time | Total job time |
|-----|-------------|----------------|
| No cache (Step 1) | ~90s | ~120s |
| First cache (miss) | ~90s | ~122s |
| Second cache (hit) | ~0s (skipped) | ~30s |

The cache hit cuts total job time by approximately 75 percent.

### Step 4: Verify Cache Hit Output

Click on the cached run in the Actions tab and expand the "Cache npm dependencies" step. The output should show:

```
Cache restored successfully
Cache restored from key: linux-npm-abc123def456
```

The `npm ci` step shows:

```
Cache hit occurred — skipping npm ci
```

This confirms the cache was restored and `npm ci` was skipped.

### Step 5: Add restore-keys for Partial Cache Hits

When the lockfile changes, the exact cache key misses. In that case, you want to fall back to the previous cache even though it is slightly stale, because installing from a partial cache is faster than installing from scratch.

Update the workflow:

```yaml
- name: Cache npm dependencies
  id: npm-cache
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-
```

Now, if the exact key misses (e.g., you added one dependency), GitHub looks for any cache starting with `linux-npm-` and restores the most recent match. `npm ci` then downloads only the new package instead of all 200 packages.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add restore-keys for partial cache hits"
git push origin main
```

### Step 6: Simulate a Lockfile Change

Add a new dependency to trigger a lockfile change:

```bash
npm install uuid
```

Commit and push:

```bash
git add package.json package-lock.json
git commit -m "Add uuid dependency"
git push origin main
```

The cache key now hashes the new lockfile, which results in a different hash. The exact key misses. But `restore-keys` finds the previous cache and restores it. `npm ci` runs but only downloads the `uuid` package — the other 200 packages come from the restored cache.

Expected output:

```
Cache miss — exact key not found
Restored from: linux-npm-def789ghi012  (partial match via restore-keys)
```

The `npm ci` step will be faster than a full install but slower than a complete cache hit:

```
Without restore-keys: npm ci  90s  (full download)
With restore-keys:    npm ci  5s   (only uuid downloaded)
```

### Step 7: Cache Multiple Paths

Some projects need to cache more than `~/.npm`. For example, you can cache `node_modules` directly (though this is less common for npm because `~/.npm` is more portable):

```yaml
- name: Cache npm and node_modules
  id: npm-cache
  uses: actions/cache@v4
  with:
    path: |
      ~/.npm
      node_modules
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
```

Caching `node_modules` directly can speed up the post-install step even further because `npm ci` does not need to link modules from the cache — they are already in place. However, it is more fragile: OS-specific or architecture-specific binaries in `node_modules` can cause issues when the cache is restored on a different runner.

### Step 8: Add save-always for Resilient Caching

By default, the cache is only saved when the entire job succeeds. If the test step fails, the cache is not saved — meaning the next run starts from scratch even though the dependencies installed fine.

Add `save-always: true` to save the cache regardless of test results:

```yaml
- name: Cache npm dependencies
  id: npm-cache
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-
    save-always: true
```

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add save-always for resilient caching"
git push origin main
```

## Common Errors at This Stage

### Error 1: Cache save fails with "Archive path too long"

```
Error: Failed to save: Unable to archive /home/runner/.npm with the following error: The path is too long
```

**Why it happens:** Some npm packages install deeply nested directory structures that exceed the archive tool's path length limit on Windows.

**Fix:** Use a shorter cache path. Instead of caching the full `~/.npm`, cache a subdirectory:

```yaml
path: ~/.npm/_cacache
```

### Error 2: Cache hit but npm ci still downloads packages

```
Cache restored successfully
npm ci  — 90s (downloading all packages)
```

**Why it happens:** The npm cache (`~/.npm`) stores compressed package tarballs, not the extracted `node_modules`. `npm ci` still runs its full resolution algorithm, but it reads from the local cache instead of the network. This is normal — the install time should drop from ~90s to ~10s. If it is still 90s, the cache may not have been restored to the correct path.

**Fix:** Verify the cache path is correct for your ecosystem:

| Package manager | Default cache path |
|-----------------|--------------------|
| npm             | `~/.npm` |
| pip             | `~/.cache/pip` |
| Maven           | `~/.m2/repository` |
| Gradle          | `~/.gradle/caches` |
| Go modules      | `~/go/pkg/mod` |
| NuGet           | `~/.nuget/packages` |

### Error 3: Cache size exceeds 10 GB limit

```
Warning: Cache size of 12 GB exceeds the 10 GB limit for this repository.
```

**Why it happens:** The repository's total cache (across all keys) has exceeded 10 GB. GitHub evicts the least-recently-used entries.

**Fix:** Reduce what you cache. Cache only the package manager's global cache directory, not `node_modules`. Use `restore-keys` to share cache entries, reducing the number of unique keys.

### Error 4: Cache key is the same for every branch

```
Cache restored from key: linux-npm-abc123 (same for main and feature branches)
```

**Why it happens:** `hashFiles('package-lock.json')` produces the same hash on every branch that has the same lockfile. This is actually correct behavior — the cache is shared across branches. But it can cause confusion when a feature branch overwrites the cache.

**Fix:** Add the branch name to the cache key if you want branch-specific caches:

```yaml
key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}-${{ github.ref }}
```

## What You Just Learned

- You added `actions/cache@v4` to cache the npm global directory (`~/.npm`)
- You used `hashFiles()` to generate a cache key from the lockfile
- You used `cache-hit` output to conditionally skip `npm ci` on cache hits
- You measured the runtime difference between uncached (~90s) and cached (~10s) installs
- You added `restore-keys` for partial cache hits when the lockfile changes
- You added `save-always: true` to save the cache even when tests fail
- You learned which cache paths to use for different package managers
