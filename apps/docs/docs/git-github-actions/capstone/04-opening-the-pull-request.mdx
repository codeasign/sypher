# Opening the Pull Request

You have code on a branch and you want to bring it into `main`. The mechanism is a pull request.

## Method 1: Via the GitHub UI

1. Open your browser and go to:
   `https://github.com/YOUR-USERNAME/urlcheck/pull/new/feature/add-urlcheck-cli`
2. The page shows a **Comparing changes** view with your feature branch on the right and `main` on the left.
3. Verify the base is `main` and the compare is `feature/add-urlcheck-cli`.
4. Click **Create pull request**.

## Method 2: Via gh CLI

```bash
gh pr create --base main --head feature/add-urlcheck-cli --title "Add urlcheck CLI tool" --body ""

# Alternatively, let the CLI open your default editor
gh pr create
```

## Writing a Real PR Description

A good PR description answers three questions: *what changed, why, and how to test it.* Here is the template you will use for this capstone:

```
## What
Adds the urlcheck CLI tool -- a Python utility that checks HTTP URL
reachability via HEAD requests. Supports single URL mode and batch mode
from a JSON config file.

## Why
This is the initial feature for the urlcheck project, providing the core
functionality that the CI/CD pipeline will validate.

## How to Test
- Single URL:  python urlcheck.py https://example.com
- Batch mode:  python urlcheck.py --batch urls.json
- No args:     python urlcheck.py (should print help and exit 1)

## Checklist
- [x] Code follows project conventions
- [x] Tests pass locally (`pytest test_urlcheck.py`)
- [x] No lint errors
```

Copy this template into the PR description body, replacing the placeholder text from the `gh pr create` command.

## Link the PR to an Issue

If you created a GitHub issue to track this feature, you can link it in the PR description or body:

```
Closes #1
```

When the PR merges, GitHub automatically closes the issue. This connects your tracker to your code.

## What Happens Automatically

The moment you click **Create pull request** (or run `gh pr create`), several things happen:

1. **GitHub registers the PR** -- it appears in the repo's Pull Requests tab with a unique number (e.g., `#1`).
2. **GitHub checks for merge conflicts** -- it shows a green checkmark (no conflicts) or a red warning.
3. **GitHub Actions triggers** -- if any workflows are configured for the `pull_request` event, they start running immediately. On this page there is no CI workflow yet, so nothing runs.
4. **GitHub notifies reviewers** -- if you set reviewers, they receive a notification.
5. **The PR is ready for review** -- even without CI, someone can review the code.

The next page will add the CI workflow so that the moment a PR opens, the pipeline starts.

## What You Should See

On the PR page, you should see:

- A green check: **This branch has no conflicts with the base branch**
- A grey area that says **No status checks** (because no CI workflow exists yet)
- The description you wrote rendered below the title
- A button: **Merge pull request** (greyed out if you set branch protection, or green if you have not yet)

Leave the PR open. Do not merge yet. The next step is to add the CI pipeline that will run against this PR.
