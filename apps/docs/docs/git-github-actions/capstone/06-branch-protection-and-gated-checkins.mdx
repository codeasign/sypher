# Branch Protection and Gated Check-ins

CI reports status, but it does not enforce anything by itself. Branch protection is what makes CI mandatory. Without branch protection, someone can click **Merge** even when tests are red. With branch protection, the merge button stays disabled until every required condition is met.

## What a Gated Check-in Means

A gated check-in is a quality gate on the `main` branch: code cannot enter `main` until specific conditions are satisfied. These conditions are not suggestions -- they are enforced by GitHub's branch protection rules. The merge button is physically disabled until every gate is green.

Gates answer specific questions:

| Gate | Question it answers |
|------|-------------------|
| Required pull request | Was this change submitted through a PR, not pushed directly? |
| Required status checks | Did CI pass? |
| Required approving review | Did someone else look at the code? |
| Up-to-date branch | Is the feature branch based on the latest `main`? |

## Configuring Branch Protection on main

### Step 1: Navigate to Branch Protection Settings

1. Go to your repo on GitHub: `https://github.com/YOUR-USERNAME/urlcheck`
2. Click **Settings** (the tab near the top right).
3. In the left sidebar, click **Branches**.
4. Under **Branch protection rules**, click **Add rule**.

### Step 2: Configure the Rule

Set the following fields exactly:

| Field | Value |
|-------|-------|
| **Branch name pattern** | `main` |

Under **Protect matching branches**, check these boxes and fill these values:

**Require a pull request before merging**
- [x] Require a pull request before merging
- [x] Require approvals: **1**
- [x] Dismiss stale pull request approvals when new commits are pushed
- [x] Require review from Code Owners

**Require status checks before merging**
- [x] Require status checks to pass before merging
- [x] Require branches to be up to date before merging
- Search for and select these status checks:
  - `lint (ubuntu-latest)`
  - `test (3.11)`
  - `test (3.12)`
  - `build (ubuntu-latest)`

**Require conversation resolution before merging**
- [x] Require conversation resolution before merging

**Require signed commits**
- Leave unchecked (this is optional and requires GPG setup, which is outside this capstone)

**Do not allow bypassing the above settings**
- [x] Do not allow bypassing the above settings

**Status check requirements**
- Leave **Allow specified actors to bypass required pull requests** empty (you are not using this)

### Step 3: Save the Rule

Click **Create** or **Save changes** at the bottom.

## How to Find the Exact Status Check Names

When you configure the **Require status checks to pass before merging** setting, GitHub shows a list of checkboxes. Each checkbox corresponds to a status check that has run on your repo recently. If no CI runs have completed on this repo, the list is empty -- you must push a commit to trigger CI first.

To find the exact names of your status checks:

1. Go to your PR page.
2. Scroll to the **Checks** section at the bottom (or click the **Checks** tab).
3. Each job appears as a separate check:
   - `lint (ubuntu-latest)`
   - `test (3.11)`
   - `test (3.12)`
   - `build (ubuntu-latest)`

These names match the `jobs.<job_id>.name` fields in your CI workflow YAML. If you named a job differently (e.g., `linting` instead of `lint`), the check name changes accordingly. Select the names that match your workflow.

## Testing the Gate

### Test 1: Attempt Merge with Failing Check

1. If the PR currently has a green CI run, break a test again (repeat the deliberate break from the CI pipeline page).
2. Push the broken test.
3. Wait for CI to run and fail.
4. Navigate to the PR page.
5. The merge button shows **Merge** with the text **Required statuses must pass before merging** in red. The button is not clickable.
6. Below the merge button, GitHub shows which checks are failing and links to the logs.

### Test 2: Everything Green But No Review

1. Fix the test and push.
2. CI passes -- checks are green.
3. Do not approve the PR yet.
4. The merge button shows **Merge** with the text **At least 1 approving review is required by people with write access.** The button is not clickable.

### Test 3: Everything Green Plus Approval

1. Approve the PR yourself (since you own the repo, you can approve your own PR in a solo repo, or add a collaborator).
2. To self-approve: go to the **Files changed** tab, click **Review changes**, select **Approve**, leave a comment, and submit.
3. The merge button now shows **Merge pull request** in green and is clickable.

## Adding a CODEOWNERS File

CODEOWNERS lets you require specific people or teams to review changes to specific paths. Even in a solo project, creating a CODEOWNERS file is good practice -- it documents ownership and prepares the repo for future collaborators.

### Step 1: Create the File

Create `.github/CODEOWNERS`:

```
# Each line: <path> <owner>
# Owners are GitHub usernames or team names

# Default owner for everything
* @YOUR-GITHUB-USERNAME

# Core source requires senior review
urlcheck.py @YOUR-GITHUB-USERNAME

# Workflow changes need pipeline owner review
.github/workflows/ @YOUR-GITHUB-USERNAME
```

Replace `@YOUR-GITHUB-USERNAME` with your actual GitHub username (e.g., `@jane-doe`).

### Step 2: Commit and Push

```bash
git add .github/CODEOWNERS
git commit -m "chore: add CODEOWNERS for path-based review requirements"
git push origin feature/add-urlcheck-cli
```

### Step 3: Verify

Go to the PR page. Under **Reviewers**, you should see that the CODEOWNERS rule automatically requests your review for the changed files.

Because you configured branch protection to require review from Code Owners, you now get two layers of review: the general approval requirement and the CODEOWNERS requirement.

## Why Gated Check-ins Matter at Scale

At a single-person project, gated check-ins prevent you from merging broken code to main when you are tired or rushing. At a team of 10 or 100, they prevent a single mistaken commit from breaking every developer's workflow.

Without gated check-ins:
- Anyone can merge code that does not compile.
- Anyone can merge code without review.
- Main branch breaks multiple times per week.
- Every developer wastes time figuring out who broke the build.

With gated check-ins:
- Main is always deployable.
- Code reviews are mandatory, not optional.
- CI failures are caught before they affect anyone else.
- The pipeline becomes a source of truth, not a suggestion.

**Gated check-ins are not bureaucracy.** They are the single most effective practice for keeping a shared codebase healthy. In production environments, a main branch with no protection is considered a critical incident waiting to happen.

## What You Should Have Now

- Branch protection configured on `main` requiring PR, CI checks, and review
- The merge button disabled (greyed out) on the PR because not all conditions are met
- A CODEOWNERS file requiring your review on all files
- Knowledge of exactly how to test each gate condition
