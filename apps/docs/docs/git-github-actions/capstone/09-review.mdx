## Checklist: What Your Repo Should Now Contain

Verify each item against your repository:

- [ ] `urlcheck.py` -- the CLI tool
- [ ] `test_urlcheck.py` -- unit tests
- [ ] `requirements.txt` -- project dependencies
- [ ] `.github/workflows/ci.yml` -- CI workflow triggered on pull request
- [ ] `.github/workflows/cd.yml` -- CD workflow triggered on push to main
- [ ] `.github/CODEOWNERS` -- path-based code ownership
- [ ] Branch protection configured on `main`:
  - [ ] Requires pull request before merging
  - [ ] Requires CI status checks to pass
  - [ ] Requires approving review (at least 1)
  - [ ] Requires branches to be up to date
  - [ ] CODEOWNERS review required
- [ ] `production` environment configured with required reviewer
- [ ] At least one PR that went through the full lifecycle
- [ ] A PR with a red CI run (deliberate break) visible in the history
- [ ] A squashed merge commit on `main`
- [ ] The feature branch deleted after merge
- [ ] At least one successful CD deployment run

## Interview Readiness: Self-Assessment

You should be able to answer the following questions without notes:

1. **What is trunk-based development and why use it over Git Flow for a small project?**
2. **How does branch protection enforce gated check-ins? What specific rules did you configure?**
3. **What status checks did you require in branch protection and how did you find their exact names?**
4. **Describe the difference between a CI trigger (`pull_request`) and a CD trigger (`push: branches: [main]`). Why must they be different?**
5. **What is a GitHub Environment and how does an approval gate work in the deployment flow?**
6. **What is a CODEOWNERS file and what does it enforce when combined with branch protection?**
7. **Compare squash merge, merge commit, and rebase merge. When would you use each?**
8. **Describe the complete flow from a developer checking out a feature branch to code running in production. Name every gate that must pass.**

If you can answer all eight without looking back at the pages, you have a working understanding of git-based CI/CD that will carry you through technical interviews.

## What to Extend Next

Your repo is now a foundation. Here are the next steps to turn it into a more complete pipeline:

### 1. Add a Security Scan

Add a new job to the CI workflow that runs `pip audit` or `safety check` on your dependencies. In branch protection, add the new job as an additional required status check.

### 2. Add a Staging Environment

Create a second GitHub Environment called `staging`. Update the CD workflow to deploy to staging first, run smoke tests, then deploy to production after a manual promotion.

### 3. Add a Matrix Build Across Multiple Python Versions

The CI workflow already runs tests on Python 3.11 and 3.12. The CD workflow currently runs on a single version. Add a matrix to the CD workflow to verify the build against multiple interpreters before the deploy step.

### 4. Replace the Simulated Deploy with a Real One

Pick a free hosting provider and add real deployment steps:

| Provider | Free tier | Deploy method |
|----------|-----------|---------------|
| GitHub Pages | Unlimited public sites | `peaceiris/actions-gh-pages` action |
| Render.com | 500 build minutes/mo | Deploy via API hook |
| Fly.io | 3 shared VMs | `superfly/flyctl-actions` action |
