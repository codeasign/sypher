# The Brief

## The Project

You will build a small Python CLI tool called **urlcheck**. It takes a URL as an argument, sends an HTTP HEAD request, and reports whether the URL is reachable along with its HTTP status code. The project also includes a batch mode that reads URLs from a JSON config file and checks them all at once.

This is a deliberately small application. The code is not the point -- the pipeline that delivers it is. You should be able to write the entire feature in under 100 lines of Python across two files.

### What urlcheck Does

```bash
python urlcheck.py https://example.com
# Output: https://example.com -- 200 OK

python urlcheck.py --batch urls.json
# Output:
# https://example.com -- 200 OK
# https://httpstat.us/404 -- 404 Not Found
# https://httpstat.us/500 -- 500 Internal Server Error (FAIL)
```

### Project Files You Will Create

| File | Purpose |
|------|---------|
| `urlcheck.py` | CLI entry point and HTTP checking logic |
| `requirements.txt` | Dependencies (only `requests`) |
| `urls.json` | Sample batch config for testing |
| `test_urlcheck.py` | Tests using `pytest` |
| `.github/workflows/ci.yml` | CI pipeline (will build step by step) |
| `.github/workflows/cd.yml` | CD pipeline (will build step by step) |

## The Constraints

- The `main` branch must never be broken -- nothing merges without passing CI.
- Every change goes through a pull request, never a direct push to `main`.
- At least one required status check must block the merge until it passes.
- Deployment (to a free target -- we will use a mock deploy that echoes success) only happens after merge to `main`, never from a feature branch.

## What You Will Build Across This Capstone

| Page | What You Will Do |
|------|-----------------|
| Repo and Branch Strategy | Create the GitHub repo, set up the initial project structure, choose a branching model |
| Feature Branch and Commits | Create a branch, write the urlcheck code, commit atomically, push |
| Opening the PR | Open a pull request with a real description linking to an issue |
| CI Pipeline | Write a real GitHub Actions CI workflow, watch it run, break it intentionally, fix it |
| Branch Protection and Gated Check-ins | Configure branch protection so nothing merges without passing CI and review |
| CD Pipeline | Write a CD workflow that deploys after merge, with an environment approval gate |
| Review and Merge | Get a review, merge, watch CD fire, clean up |
| Review | Assess what you built, checklist your repo, self-test for interviews |

## How to Use This Capstone

This is not a read-along tutorial. You must actually create the GitHub repo, run every command, open every pull request, and break things on purpose to see what failure looks like. Reading without doing defeats the purpose.

Each page tells you exactly what to type and where to click. Follow along in your own terminal and browser. By the end, you will have a public GitHub repository that demonstrates a complete, real-world gated delivery pipeline.
