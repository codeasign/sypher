You commit a file containing a debug `print()` statement and a hardcoded API key. The commit passes your local review but gets flagged by the CI pipeline 10 minutes later. You push another commit to remove the secret, but the original commit is already in the shared history. You need a gate that catches these mistakes before the commit is recorded — not after. Git hooks run custom scripts at key points in Git's lifecycle to enforce policies locally.

## The Concept

A Git hook is a script that Git executes automatically before or after a specific event: committing, pushing, merging, rebasing, or receiving updates. Hooks live in `.git/hooks/` and are local to each clone — they are not pushed to the remote.

Think of hooks like airport security checks at different gates. Before you board (commit), a screener checks your ID and bags (pre-commit hook checks code quality and secrets). Before you enter the jet bridge (push), another screener verifies your boarding pass (pre-push hook runs tests). If you fail any check, you are turned back. The airline does not need to know about each airport's procedures — each airport runs its own security.

## Command / Syntax Reference

| Hook name | When it runs | Exit code 0 = | Exit code non-0 = |
|---|---|---|---|
| `pre-commit` | Before `git commit` creates the commit | Commit proceeds | Commit is aborted |
| `prepare-commit-msg` | After the default message is created, before the editor opens | Uses the modified message | Commit is aborted |
| `commit-msg` | After the commit message is edited | Commit proceeds | Commit is aborted |
| `post-commit` | After the commit is created | Hook runs silently | Hook runs, commit already done |
| `pre-rebase` | Before `git rebase` begins | Rebase proceeds | Rebase is aborted |
| `post-checkout` | After `git checkout` or `git switch` completes | No effect on the operation | Hook errors shown, operation done |
| `pre-push` | Before `git push` sends data to the remote | Push proceeds | Push is aborted |
| `post-merge` | After `git merge` completes | No effect on the merge | Hook errors shown |
| `pre-auto-gc` | Before `git gc --auto` runs | Garbage collection proceeds | GC is skipped |
| `post-rewrite` | After `git commit --amend` or `git rebase` | Hook runs silently | Hook errors shown |

**Windows note:** Hook scripts can be written in any scripting language. On Windows, you can use PowerShell (`.ps1`), batch (`.bat`), or any interpreter. The hook file must be executable and have the correct shebang line. On Windows with Git Bash, bash scripts work natively. On cmd or PowerShell, use a `.bat` file: the hook file name is `pre-commit.bat` or `pre-commit` with a PowerShell launcher.

**Shared hooks (Git 2.9+):**

| Command | What it does | Example |
|---|---|---|
| `git config core.hooksPath <path>` | Set a custom hooks directory outside `.git/hooks/`. | `git config core.hooksPath .githooks` |
| `git config --global core.hooksPath <path>` | Set hooks globally for all repositories. | `git config --global core.hooksPath ~/.githooks` |

## How It Actually Works — Step by Step

### Step 1: Create a hook script

Hook scripts are executable files in `.git/hooks/` with specific names. Git ships with example hooks in `.git/hooks/examples/`.

```bash
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
echo "Running pre-commit checks..."
if grep -n "print(" *.py; then
  echo "WARNING: Debug print statements found"
  exit 1
fi
echo "All checks passed"
EOF
chmod +x .git/hooks/pre-commit
```

For PowerShell on Windows:

```powershell
# .git/hooks/pre-commit (PowerShell)
Write-Host "Running pre-commit checks..."
if (Select-String -Path *.py -Pattern "print\(") {
  Write-Host "ERROR: Debug print statements found"
  exit 1
}
Write-Host "All checks passed"
```

### Step 2: Git executes the hook before the commit

When you run `git commit`:

1. Git checks if `.git/hooks/pre-commit` exists and is executable
2. Git runs the script
3. If the script exits with 0, the commit proceeds
4. If the script exits with non-0, the commit is aborted

```
  $ git commit -m "Add feature"
  Running pre-commit checks...
  src/app.py:10:    print("debug")
  ERROR: Debug print statements found
  $  ← Commit was aborted. No commit was created.
```

The commit is not recorded. The staged changes remain in the staging area — you can fix the issue and retry.

### Step 3: Pre-push hook — run before remote push

Create a pre-push hook that rejects branches with forbidden names:

```bash
cat > .git/hooks/pre-push << 'EOF'
#!/bin/sh
# Reject pushes to main (require PR workflow)
protected_branches="main master"
while read local_ref local_oid remote_ref remote_oid; do
  branch=${remote_ref#refs/heads/}
  for protected in $protected_branches; do
    if [ "$branch" = "$protected" ]; then
      echo "ERROR: Direct push to $branch is not allowed."
      echo "Use a pull request instead."
      exit 1
    fi
  done
done
exit 0
EOF
chmod +x .git/hooks/pre-push
```

The pre-push hook uses stdin to receive the list of refs being pushed. Each line has four fields: `local_ref local_oid remote_ref remote_oid`.

### Step 4: Share hooks via the project

Since `.git/hooks/` is not committed, teams need a way to share hooks:

```bash
mkdir -p .githooks
cp .git/hooks/pre-commit .githooks/
git add .githooks/
git commit -m "Add shared pre-commit hook"
git push origin main
```

Each developer must configure Git to use the shared hooks directory:

```bash
git config core.hooksPath .githooks
```

Or clone new developers can run a setup script:

```bash
cat > setup-hooks.sh << 'EOF'
#!/bin/bash
git config core.hooksPath .githooks
echo "Hooks configured from .githooks/"
EOF
chmod +x setup-hooks.sh
```

### Step 5: Use a hook manager (Husky for Node.js)

For Node.js projects, Husky manages hooks automatically:

```bash
npx husky init
echo "npx lint-staged" > .husky/pre-commit
git add .husky/
git commit -m "Configure Husky with lint-staged"
```

Husky stores hooks in `.husky/` and installs them during `npm install` or `git config core.hooksPath` automatically. This eliminates the manual setup step.

## When to Use This / When Not To

**Use hooks when:**
- You want to enforce code style locally before code reaches CI
- You need to prevent secrets, debug statements, or large files from being committed
- You want to automate version bumping or changelog generation
- You want to reject direct pushes to protected branches at the client level

**Do not use hooks when:**
- You need a hard security boundary — hooks are client-side and can be bypassed (git commit --no-verify). Use server-side hooks (GitHub Actions, server-side pre-receive hooks) for enforcement
- The check takes more than a few seconds — developers will disable hooks if they slow down commits. Move slow checks to CI
- You are not sure your team can install and configure the hooks — use CI for enforcement instead
- The hook depends on tools not in every developer's PATH — document prerequisites clearly

Hooks are a developer convenience, not a security mechanism. `git commit --no-verify` (or `-n`) bypasses all hooks. Use CI/CD pipelines for hard enforcement.
