# What Is an Agent?

*Reading time: 5 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Define an AI agent in terms of perception, reasoning, action, and memory
- Distinguish an agent from a plain LLM call
- Identify the three essential properties that make a system agentic
- Recognize the right and wrong use cases for an agent architecture

## Prerequisites

You should already understand:

- How LLMs generate text from prompts (covered in [LLM Fundamentals](../llm-fundamentals/overview))
- How system prompts set persistent behavior (covered in [System Prompts](../system-prompts/overview))
- How structured output constrains model responses (covered in [Structured Output Prompting](../structured-output-prompting/overview))

## Why This Matters

The term "agent" is used loosely across the industry — every demo that chains two API calls gets called an agent. Understanding what actually makes a system agentic is crucial because agent architectures introduce complexity, cost, and failure modes that simpler systems do not have. If you do not know what an agent is, you cannot know when you need one — and building one when you do not is a common and expensive mistake.

Companies that deploy agents in production — coding assistants, customer support bots, research assistants — use well-defined boundaries around agency. They know when the model acts and when it just responds. This distinction determines everything from the API design to the safety infrastructure to the cost model.

## The Problem

You write a prompt that asks an LLM to "look up the weather, check the user's calendar, and suggest a meeting time." The model returns text that says "I would look up the weather, then check the calendar, then suggest a time." But it did not actually run any lookups — it generated text about running lookups.

A plain LLM call produces text. An agent produces action. The gap between saying and doing is the core problem that agent architectures solve. Your system needs to actually call APIs, query databases, and take actions in the world — not just talk about them.

## The Concept

An AI agent is a system in which an LLM drives a loop of perception, reasoning, action, and memory update. The agent does not just respond to a single prompt — it perceives its environment (user input, tool results, system state), reasons about what to do next, takes an action (calls a tool, produces output), and incorporates the result back into its context.

An agent has three essential properties:

**Autonomy**. Once launched, the agent decides its own steps within the boundaries you set. You do not micromanage each tool call — the agent chooses which tools to call in what order based on the current state.

**Tool use**. The agent can call external functions, APIs, and services. This is what turns text generation into action. Without tools, an LLM is a text generator. With tools, it becomes an actor.

**Memory**. The agent maintains and updates its understanding across steps. It remembers what it has done, what results it got, and what still needs to be done. Without memory, each step starts from zero.
