# Avoid Mistakes: Filesystem MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A filesystem MCP server gives an agent read and write access to files. Do not use it when:

- **You only need to read specific data** — prefer a resource-based server that serves structured data (CSV, JSON, config) instead of raw file access
- **The agent runs on untrusted input** — if users can craft prompts that include file paths (e.g. "read this file: `/etc/passwd`"), a filesystem server amplifies prompt injection risks
- **You need database-level consistency** — use a database MCP server instead; filesystem writes are not transactional
- **The files are binary or extremely large** — reading a 500 MB log file through MCP will exhaust the agent's context window and spam the transport

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Sandboxed file access through discrete MCP tools |
| **When to use** | Agent needs to read, write, or search project files |
| **When NOT to use** | Binary data, untrusted prompts, database workloads |
| **Mental model** | A restricted `cat`/`echo` with directory jail |

## Common Mistakes

### Mistake 1: Setting the Sandbox Too Broadly

**Mistake:** Configuring the server with a root or home directory sandbox, allowing the agent to read SSH keys, `.env` files, or system configuration.

**Why it happens:** It is easy to type `npx .../server-filesystem /` during testing and leave the configuration in place.

**Symptoms:** The agent unexpectedly reads files outside the project, or a prompt injection could leak sensitive data.

**Fix:** Always use the most restrictive sandbox that covers your use case. Use a project-specific subdirectory, not the repo root even, if the agent only needs one folder.

### Mistake 2: Forgetting to Handle Symlinks

**Mistake:** Writing a path validation check that only compares string prefixes without resolving symlinks.

**Why it happens:** A naive `path.startswith(sandbox)` check passes for a symlink that points outside.

**Symptoms:** The agent accesses files outside the sandbox when the path traverses a symlink.

**Fix:** Always call `os.path.realpath()` or `Path.resolve()` before checking the sandbox boundary, as shown in the custom server example.

### Mistake 3: Not Setting File Size Limits

**Mistake:** Letting the agent read arbitrarily large files unchecked.

**Why it happens:** The server reads the whole file into memory and sends it over the transport. A 2 GB log file causes an out-of-memory crash or a multi-minute hang.

**Symptoms:** Agent requests time out, the MCP client shows "response too large" errors, or the server process runs out of memory.

**Fix:** Add a `st_size` check before reading, and either truncate, reject, or stream the file.

### Mistake 4: Writing Files Without UTF-8 Encoding

**Mistake:** Writing binary or non-UTF-8 encoded data through `write_file`.

**Why it happens:** The LLM may generate text that includes null bytes, control characters, or non-UTF-8 sequences. The server blindly writes it to disk.

**Symptoms:** Files appear corrupted when opened in an editor, or downstream tools crash on encoding errors.

**Fix:** Always specify `encoding="utf-8"` and either reject or encode non-text content before writing.

### Mistake 5: Running Multiple Servers on the Same Path

**Mistake:** Starting two filesystem server instances pointing at the same directory simultaneously.

**Why it happens:** Multiple agents or sessions each start their own server.

**Symptoms:** One agent reads a file that another agent is in the middle of writing, getting partial or inconsistent content.

**Fix:** Use a single shared server instance, or make writes atomic by writing to a temp file and renaming.
