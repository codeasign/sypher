# Shared Memory and State

*Reading time: 6 minutes*

**Difficulty:** Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Design a shared state store that multiple agents read from and write to
- Implement read and write isolation to prevent agents from corrupting each other's data
- Choose between centralized state, distributed state, and agent-local state
- Handle race conditions in multi-agent state access

## Prerequisites

You should already understand:

- How agents maintain internal memory (covered in [Agent Memory](../agent-memory/overview))
- How orchestrators delegate work to sub-agents (covered in [Orchestrator and Supervisor Patterns](../orchestrator-and-supervisor-patterns/overview))
- Basic understanding of concurrent state access and race conditions

## Why This Matters

When you have multiple agents working on related tasks, each agent needs access to context that other agents have created or modified. The research agent updates the "customer profile" document. The writing agent needs to read that update. The review agent needs to see what both produced. Without shared state, every agent works with stale or incomplete information.

In production multi-agent systems, shared state is the most common source of subtle bugs. An agent reads a value that another agent is simultaneously updating. Two agents write conflicting values to the same field. An agent's internal memory contradicts the shared state because it never refreshed its cache. These bugs are hard to reproduce and harder to diagnose because they depend on timing.

The industry has converged on three patterns for shared state: a centralized store (single source of truth), a distributed log (event-sourced state), and agent-local state (no sharing, explicit handoffs). Each pattern makes different trade-offs between consistency, latency, and complexity.

## The Problem

Your research agent queries the database and writes "total revenue: 4.2 million" to the shared state. Simultaneously, your data agent runs a more precise query and writes "total revenue: 4.18 million." The writing agent reads the state and sees the first value because it read a few milliseconds before the data agent's write completed. The report goes out with the wrong number.

This is a classic read-write race condition, but in agent systems it is worse than in traditional distributed systems because agents also have their own internal reasoning state. An agent might have already committed to a course of action based on a stale read, so even correcting the shared state does not fix the downstream work already in progress.

## The Concept

Shared state in multi-agent systems is any data that more than one agent reads or writes. It can be as simple as a shared dictionary or as sophisticated as an event-sourced database. The key design decision is which consistency model you need:

**Strong consistency** — every read sees the latest write. This is what you get from a single database with transactions. It is the safest model but the slowest, because every read and write goes through a central coordinator.

**Eventual consistency** — writes propagate to all readers within a bounded time. Agents may see stale data for a brief window, but the system converges. This model is faster but requires agents to tolerate temporary inconsistency.

**Per-agent snapshots** — each agent gets a read-only snapshot of the shared state at the start of its task. The agent works against that snapshot and writes its results back. This eliminates race conditions within a single task but means agents never see each other's in-progress changes.
