# Structured Output Prompting — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Structured output is not always the right approach:

- **Human-facing responses**: If the output goes directly to a user interface and is not processed programmatically, let the model write natural language. Forcing structured output adds unnecessary complexity.
- **Creative or open-ended tasks**: Structured constraints fight against creative generation. Use loose format guidance instead of strict schemas.
- **Prototyping**: During early exploration, structured output adds overhead. Let the model produce free text and iterate on the interaction design first.

## The Pattern to Remember

**Pattern**: Constrain output structure at the API level when possible (tool use), at the prompt level when necessary (JSON mode), and always validate after generation.

When to use:
- Output must be consumed by another system (database, API, UI component)
- You need to parse the output programmatically
- You need to validate the output against a schema

When NOT to use:
- Output is displayed to humans directly
- You are exploring a problem and do not know the schema yet
- Creative or open-ended tasks

**Mental model**: Structured output is an interface contract between the model and your code. Define the contract clearly, validate every response, and handle violations gracefully.

## Common Mistakes

### Mistake 1: Relying Only on Prompt Instructions for JSON Format

**Why it happens**: You write "Return as JSON" and the model obliges most of the time.

**Symptoms**: About 5-15% of responses include markdown fences, explanatory text before or after the JSON, or slightly wrong key names. Your parser fails randomly.

**Fix**: Use validation and cleaning logic to handle common output variations. Or use JSON mode / tool use for guaranteed syntax.

### Mistake 2: Not Handling the "Explanatory Prefix" Problem

**Why it happens**: Even with "Return ONLY JSON," the model sometimes adds "Here is the JSON:" before the object.

**Symptoms**: `json.loads()` fails because the string starts with non-JSON text.

**Fix**: Strip non-JSON prefixes before parsing. Search for the first `{` or `[` character and parse from there.

### Mistake 3: Schema Drift

**Why it happens**: You update the schema but forget to update the prompt. Or you change the prompt but existing post-processing code expects old key names.

**Symptoms**: The output is valid JSON but has different keys, additional fields, or different types than expected.

**Fix**: Version your schemas. Store the schema alongside the template. Validate the output against the expected schema after parsing.

### Mistake 4: Confusing JSON Syntax with JSON Schema

**Why it happens**: You provide an example JSON object and expect the model to infer the schema from it.

**Symptoms**: The model produces output that follows the example's structure approximately — different nesting, optional fields omitted, extra fields added.

**Fix**: Provide the schema explicitly (field names, types, and constraints) rather than just an example. Examples help with format; schemas help with structure.

### Mistake 5: Assuming Parsed JSON Is Correct Data

**Why it happens**: The JSON parses successfully, so you assume the data is correct.

**Symptoms**: A JSON object with `{"urgency": "LOW"}` parses fine, but the correct label is actually `"URGENT"`. The model used a value not in your allowed set.

**Fix**: After parsing JSON, validate the values against your allowed sets. Check that enums, numbers, and dates are within expected ranges.

```python
VALID_URGENCIES = {"HIGH", "MEDIUM", "LOW", "CRITICAL"}

def validate_extraction(data: dict) -> list[str]:
    """Validate extracted data and return a list of validation errors."""
    errors = []

    if data.get("urgency") not in VALID_URGENCIES:
        errors.append(f"Invalid urgency: {data.get('urgency')}")

    if not data.get("email") or "@" not in data["email"]:
        errors.append(f"Invalid email: {data.get('email')}")

    return errors
```
