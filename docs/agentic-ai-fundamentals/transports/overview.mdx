# Transports

*Reading time: 6 minutes*

**Difficulty:** Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Explain the difference between stdio and SSE transport in MCP
- Choose the right transport for a given deployment scenario
- Implement a client that connects over either transport
- Handle transport-level errors and reconnection

## Prerequisites

You should already understand:

- The MCP client-server architecture (covered in [MCP Architecture](../mcp-architecture/overview))
- How JSON-RPC messages are structured (covered in [What Is MCP?](../what-is-mcp/overview))
- Basic understanding of subprocess management and HTTP

## Why This Matters

The transport layer is the most deployment-specific part of MCP. A local file server should run over stdio — it is a child process that communicates through stdin and stdout. A remote database server should run over SSE — it is a network service that multiple clients connect to over HTTP. Choosing the wrong transport means your server either cannot reach its data sources or introduces unnecessary latency.

In production, the transport choice determines everything about your deployment topology. Stdio servers must run on the same machine as the client — they are child processes. SSE servers can run anywhere reachable by HTTP — they are network services. Stdio servers have lower latency because there is no network round trip. SSE servers scale horizontally because they are stateless HTTP services.

MCPs architecture intentionally abstracts transport behind the same JSON-RPC message format. The same server logic works with either transport; only the I/O layer changes. But the transport-specific details — process management, HTTP headers, reconnection — require different code and different error handling.

## The Problem

You build an MCP server that provides database query tools. In development, it runs on your laptop as a child process of your agent — stdio transport. It works perfectly. You deploy it to a server so your team can share it. Now multiple agents need to connect to the same server. Stdio cannot help — it is one-to-one, one process per client. You need to switch to SSE.

The server logic — tool definitions, query execution — is identical. But the transport layer is completely different. The stdio version reads from stdin and writes to stdout. The SSE version handles HTTP requests and manages Server-Sent Events streams. If you did not design for transport abstraction from the start, you are rewriting the server.

The transport choice also affects error handling. If a stdio server crashes, the client's subprocess pipe breaks and you get an immediate error. If an SSE server crashes, the client's HTTP connection times out after 30 seconds. These are different failure modes that require different recovery strategies.

## The Concept

MCP defines two transport protocols:

**stdio** — the client spawns the server as a child process. Messages are JSON-RPC lines on stdin (requests) and stdout (responses). Stderr is reserved for logging. This transport is local, low-latency, and one-to-one. It is ideal for file servers, local database servers, and any tool that runs on the same machine.

**SSE (Server-Sent Events)** — the client connects to an HTTP endpoint. The server pushes events to the client over a long-lived SSE stream. The client sends JSON-RPC requests via HTTP POST to a separate endpoint. This transport is remote, supports multiple clients, and works across machines. It is ideal for shared services, team deployments, and cloud-hosted tools.

Both transports carry the same JSON-RPC messages. The client and server do not need to change their business logic when switching transports — only the I/O adapter changes.
