# Agent vs. Chatbot vs. Workflow — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

The architectural comparison itself is a decision framework, not a solution. Some common misapplications:

- **Over-engineering**: Building an agent for a task a simple if/else workflow handles perfectly.
- **Under-engineering**: Using a plain chatbot for tasks that require database lookups or external actions.
- **Mixing without boundaries**: Blending agent and workflow patterns without clear separation, producing a system that is neither predictable nor flexible.
- **Assuming agents are always better**: Agents cost more, run slower, and fail more interestingly than workflows.

## The Pattern to Remember

**Pattern**: Choose the architecture by the task's need for autonomy.

When to use each:
- **Chatbot**: Conversational Q&A, no external actions needed
- **Workflow**: Fixed process with known steps
- **Agent**: Variable process where the model must decide the steps

**Mental model**: The architecture determines who decides what happens next. Chatbot: the user decides by typing the next message. Workflow: the developer decided at build time. Agent: the model decides at runtime.

## Common Mistakes

### Mistake 1: Building an Agent When a Workflow Works Better

**Why it happens**: Agents are exciting. Teams reach for the most sophisticated architecture first.

**Symptoms**: The agent works, but it is slower, more expensive, and less reliable than a workflow would be. Debugging takes longer because the path varies each time.

**Fix**: Start with the simplest architecture that can plausibly work. Only add agent capabilities when the simpler approach fails.

### Mistake 2: Adding Tools to a Chatbot without a Loop

**Why it happens**: You add tool definitions to the API call but do not implement the loop that feeds results back to the model.

**Symptoms**: The model requests a tool call, but the response contains a description of what it would do rather than an actual result. The system cannot execute tool calls.

**Fix**: Tools require a loop. The model requests a tool, your code executes it, and the result goes back to the model. Without the loop, tools are decoration.

### Mistake 3: Making a Workflow Too Rigid

**Why it happens**: You define every possible branch at design time, making the workflow enormous.

**Symptoms**: The workflow has hundreds of branches. Adding a new input type requires modifying the branching logic. The workflow becomes unmaintainable.

**Fix**: If your workflow has more than 10-15 branches, consider whether an agent with a few general-purpose tools would be simpler.

### Mistake 4: Ignoring Failure Modes

**Why it happens**: Each architecture fails differently, but you test only the happy path.

**Symptoms**: The chatbot hallucinates data. The workflow crashes on unexpected input. The agent enters an infinite loop. The failure mode is discovered in production.

**Fix**: Test each architecture against its characteristic failure modes. Chatbots: hallucination. Workflows: unexpected input. Agents: infinite loops.

### Mistake 5: Not Having an Escape Hatch

**Why it happens**: You commit to one architecture and never plan for escalation.

**Symptoms**: When the chatbot cannot answer, it keeps apologizing. When the workflow hits an unknown case, it returns a generic error. When the agent fails, the entire process stops.

**Fix**: Every architecture needs an escalation path: "I cannot handle this, here is a human." Build the escape hatch before you need it.
