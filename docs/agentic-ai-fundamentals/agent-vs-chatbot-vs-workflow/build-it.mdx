# Agent vs. Chatbot vs. Workflow — Build It

*Reading time: 14 minutes*

## Breaking It Down

### 1. The Task: Customer Order Lookup

You will implement the same task — looking up a customer order and returning its status — in three different architectures. This will make the trade-offs concrete.

```python
# Shared data for all three implementations
ORDERS = {
    "ORD-1001": {"status": "shipped", "eta": "2026-07-03", "total": 149.99},
    "ORD-1002": {"status": "processing", "eta": None, "total": 299.50},
    "ORD-1003": {"status": "delivered", "eta": "2026-06-28", "total": 79.99},
}
```

### 2. Chatbot Implementation

A chatbot receives the prompt and generates a response. It has no tools, no lookup capability. It answers from whatever the LLM knows.

```python
import anthropic

client = anthropic.Anthropic()


def chatbot_response(user_input: str) -> str:
    """Pure chatbot — no tools, no data access."""
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=500,
        temperature=0,
        system="You are a customer support bot. Answer questions about orders.",
        messages=[{"role": "user", "content": user_input}]
    )
    return response.content[0].text


# The chatbot may hallucinate order data it cannot actually look up
print(chatbot_response("What is the status of order ORD-1001?"))
# Likely output: "I'd need to look that up in our system..."
# The model knows it cannot access a real database
```

The chatbot cannot actually look up data. If the system prompt tells it to "use its training data" it will invent plausible-sounding but false order information. This is the core limitation of a pure chatbot — it generates text about the world but cannot query the world.

### 3. Workflow Implementation

A workflow defines a fixed sequence of steps. The order lookup follows a predetermined path.

```python
def workflow_order_status(order_id: str) -> str:
    """Deterministic workflow — fixed steps, no LLM autonomy."""
    # Step 1: Validate the order ID format
    if not order_id.startswith("ORD-"):
        return "Invalid order ID format. Orders start with ORD-"

    # Step 2: Look up the order in the database
    order = ORDERS.get(order_id)
    if not order:
        return f"Order {order_id} not found."

    # Step 3: Format the response
    status = order["status"]
    eta = order["eta"] or "not yet available"
    total = order["total"]
    return (
        f"Order {order_id} is currently {status}.\n"
        f"Total: ${total:.2f}\n"
        f"Estimated delivery: {eta}"
    )


print(workflow_order_status("ORD-1002"))
# Output is deterministic and always correct
```

The workflow is fast, predictable, and testable. But it is brittle. To handle a different type of order query — "How many orders did I place this month?" — you need to add a new workflow.

### 4. Agent Implementation

The agent uses the LLM to decide which tool to call. It can handle a wider range of requests with the same tool set.

```python
def agent_order_response(user_input: str) -> str:
    """Agent — LLM decides which tools to call."""
    messages = [{"role": "user", "content": user_input}]

    tools = [
        {
            "name": "lookup_order",
            "description": "Look up an order by its ID and return its status, ETA, and total",
            "input_schema": {
                "type": "object",
                "properties": {
                    "order_id": {
                        "type": "string",
                        "description": "Order ID starting with ORD-"
                    }
                },
                "required": ["order_id"]
            }
        }
    ]

    for turn in range(5):
        response = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=500,
            temperature=0,
            system="You are a helpful order lookup assistant.",
            messages=messages,
            tools=tools,
        )

        content = response.content[0]

        if content.type == "tool_use":
            order = ORDERS.get(content.input["order_id"])
            if not order:
                result = f"Order {content.input['order_id']} not found."
            else:
                result = (
                    f"Status: {order['status']}, "
                    f"Total: ${order['total']:.2f}, "
                    f"ETA: {order['eta'] or 'not available'}"
                )
            messages.append({"role": "assistant", "content": content})
            messages.append({
                "role": "user",
                "content": [
                    {"type": "tool_result", "tool_use_id": content.id, "content": result}
                ]
            })
        else:
            return content.text

    return "No final response within turn limit."


# The agent handles varied phrasing naturally
print(agent_order_response("Did my order ORD-1003 arrive yet?"))
print(agent_order_response("Where is ORD-1002?"))
```

The agent handles multiple phrasings of the same request because the LLM maps the user's words to the correct tool. The workflow would fail on "Did my order arrive yet?" if the input format was rigid.

## How It Works Internally

The three architectures differ in where the decision logic lives:

- **Chatbot**: All decisions happen inside the LLM response. The LLM decides what to say — but cannot take any action. There is no external control flow.
- **Workflow**: All decisions happen in your code. If-else branches, switch statements, and function calls determine the path. The LLM may be used within individual steps, but the overall path is hardcoded.
- **Agent**: Decisions are shared. The LLM decides which tools to call and in what order. Your code provides the tools and the loop structure. The LLM fills in the control flow at runtime.

## Build Something Real

Build a triage system that classifies a customer inquiry and routes it to the correct handler — chatbot, workflow, or agent.

```python
import anthropic
from typing import Literal

client = anthropic.Anthropic()


class TriageSystem:
    """A system that routes requests to chatbot, workflow, or agent."""

    def __init__(self):
        self.orders = ORDERS
        self.refund_threshold = 500.0

    def classify_request(self, user_input: str) -> Literal["simple", "order", "complex"]:
        """Use an LLM to classify the request type quickly."""
        response = client.messages.create(
            model="claude-3-haiku-20240307",
            max_tokens=50,
            temperature=0,
            messages=[{
                "role": "user",
                "content": (
                    f"Classify this customer request into one of: "
                    f"simple (greeting, FAQ, thanks), "
                    f"order (order status, tracking, delivery), "
                    f"complex (refund, complaint, multiple issues).\n\n"
                    f"Request: {user_input}\n\n"
                    f"Respond with only the category word: simple, order, or complex."
                )
            }]
        )
        return response.content[0].text.strip().lower()

    def handle_simple(self, user_input: str) -> str:
        """Chatbot path — simple greeting or FAQ."""
        response = client.messages.create(
            model="claude-3-haiku-20240307",
            max_tokens=200,
            temperature=0,
            system="You are a friendly customer service bot. Keep responses brief.",
            messages=[{"role": "user", "content": user_input}]
        )
        return response.content[0].text

    def handle_order(self, order_id: str) -> str:
        """Workflow path — deterministic order lookup."""
        order = self.orders.get(order_id)
        if not order:
            return f"Order {order_id} not found."
        return (
            f"Order {order_id}: {order['status']}. "
            f"Total: ${order['total']:.2f}. "
            f"ETA: {order['eta'] or 'pending'}"
        )

    def handle_complex(self, user_input: str) -> str:
        """Agent path — full agent loop for complex requests."""
        messages = [{"role": "user", "content": user_input}]
        tools = [
            {
                "name": "lookup_order",
                "description": "Look up order details by order ID",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "order_id": {"type": "string"}
                    },
                    "required": ["order_id"]
                }
            },
            {
                "name": "calculate_refund",
                "description": "Calculate a refund amount for an order",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "order_id": {"type": "string"},
                        "reason": {"type": "string"}
                    },
                    "required": ["order_id", "reason"]
                }
            },
        ]

        for turn in range(10):
            response = client.messages.create(
                model="claude-3-5-sonnet-20241022",
                max_tokens=1000,
                temperature=0,
                system=(
                    "You are a senior support agent. Handle refunds, complaints, "
                    "and multi-issue requests. Be helpful and accurate."
                ),
                messages=messages,
                tools=tools,
            )

            content = response.content[0]
            if content.type == "tool_use":
                name = content.name
                inp = content.input
                if name == "lookup_order":
                    order = self.orders.get(inp["order_id"])
                    result = (f"Order not found." if not order
                              else f"Order {inp['order_id']}: {order}")
                elif name == "calculate_refund":
                    order = self.orders.get(inp["order_id"])
                    amount = order["total"] if order else 0
                    if amount > self.refund_threshold:
                        result = (
                            f"Refund ${amount:.2f} requires manager approval "
                            f"(over ${self.refund_threshold:.2f})."
                        )
                    else:
                        result = f"Refund of ${amount:.2f} approved."
                else:
                    result = f"Unknown tool: {name}"

                messages.append({"role": "assistant", "content": content})
                messages.append({
                    "role": "user",
                    "content": [
                        {"type": "tool_result", "tool_use_id": content.id, "content": result}
                    ]
                })
            else:
                return content.text

        return "Unable to resolve within turn limit."

    def route(self, user_input: str) -> str:
        """Route a request to the appropriate handler."""
        # Check if the input contains a known order ID pattern
        import re
        match = re.search(r"ORD-\d+", user_input)

        classification = self.classify_request(user_input)

        if classification == "simple":
            return self.handle_simple(user_input)
        elif classification == "order" and match:
            return self.handle_order(match.group())
        else:
            return self.handle_complex(user_input)


system = TriageSystem()

# Test each path
print("=== SIMPLE ===")
print(system.route("Thanks, you guys are great!"))

print("\n=== ORDER (workflow) ===")
print(system.route("Where is my order ORD-1001?"))

print("\n=== COMPLEX (agent) ===")
print(system.route("I want a refund for order ORD-1002. It arrived damaged."))
```

This triage system demonstrates the practical wisdom of the lesson: use the simplest architecture that meets the need, and escalate to more complex architectures only when necessary.

## Performance Considerations

The triage classification adds one fast model call (Haiku) before deciding the path. This overhead is worth it — it prevents the expensive agent path from running on simple requests. Measure your request distribution. If 70% of requests are simple greetings, the triage saves significant cost.

## Security Considerations

When routing to workflows, validate that the input does not contain injection-like patterns that could trick the classification into routing to the wrong handler. The agent path should be the default for ambiguous inputs — it gives the LLM the most control and is safest for unexpected requests.
