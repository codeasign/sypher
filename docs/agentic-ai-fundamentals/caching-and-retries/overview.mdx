# Caching and Retries

⏱️ 15 minutes · Difficulty: Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Cache LLM responses to reduce API costs and cut latency by 50-80 percent on repeated queries
- Implement retry strategies for tool calls and API calls that handle transient failures without compounding them
- Design cache invalidation policies that keep agent outputs fresh without trashing cache hit rates
- Apply exponential backoff and circuit breaker patterns so retries do not overwhelm downstream services

## Prerequisites

You should already understand:

- How HTTP caching works — cache keys, TTLs, and cache invalidation basics
- Retry logic fundamentals — what makes a retry safe versus dangerous
- Agent tool-calling patterns — how agents call tools and process results through the observe-think-act loop (covered in [Observe-Think-Act Loop](../observe-think-act-loop/overview))
- Basic Python — function decorators, dictionaries, hashing

## Why This Matters

Every LLM API call costs money and takes seconds. A single agent session that makes 15 tool calls also makes 16 LLM calls, each consuming thousands of tokens. At scale — thousands of sessions per day — these costs add up to hundreds or thousands of dollars per month. The majority of these calls are repetitive: the same system prompt, the same task descriptions, the same tool definitions sent in every turn. Caching identical or semantically similar requests can save 40-60 percent on API costs and reduce average response latency by an order of magnitude.

Retries are equally critical for production reliability. LLM APIs and external tool endpoints experience transient failures — network timeouts, HTTP 429 rate limits, service degradation, brief outages. Without retries, a single transient error fails the entire agent task. With naive retries — retrying immediately, retrying indefinitely, retrying without backoff — you risk making the problem worse by flooding the failing service with requests. A systematic retry strategy with exponential backoff and jitter turns transient failures into blips the agent never notices.

## The Problem

Your agent calls the same LLM with the same system prompt and tool definitions 10 times in an hour — once for each user session. Each call costs $0.03 and takes 4 seconds. That is $0.30 and 40 seconds of cumulative latency per hour for exactly the same payload. Multiply by 20 tool-using sessions and the waste is obvious: you are paying the same cost and waiting the same time for results you could have served from a cache in milliseconds.

Worse, a transient API error kills a 15-step agent task that was 14 steps complete. The database tool returns HTTP 503 during a brief deployment window. The agent crashes, the session is lost, and the user must restart from scratch. The failure was temporary — the database recovered in 3 seconds — but the agent had no retry logic, so a 3-second blip destroyed 14 steps of progress. With exponential backoff, the agent would have waited 1 second, then 2 seconds, and succeeded on the second retry.

## The Concept

Caching and retries together form the reliability and efficiency foundation for production agents. They address two separate problems with a shared design philosophy: make the system resilient to predictable failure modes.

**Response caching with semantic keys.** A cache stores LLM responses keyed by a hash of the input. Exact-match caching maps identical prompts to cached responses. Semantic caching maps semantically similar prompts — for example, two users asking "what is the refund policy?" and "tell me about returns" — to the same cached answer. The cache key includes the system prompt, user message, tool definitions, and model parameters so that different configurations produce different cache entries.

**Retry with exponential backoff.** When a tool call or API call fails, the agent waits and retries. The delay grows exponentially: 1 second, 2 seconds, 4 seconds, 8 seconds. Jitter (randomizing the delay slightly) prevents multiple agents from retrying in lockstep, which would create a retry storm. A maximum retry count (typically 3) prevents infinite loops.

**Circuit breaker pattern.** When failures exceed a threshold — for example, 5 consecutive failures or a 50 percent error rate in the last 10 calls — the circuit breaker trips. No further calls are attempted until a recovery timeout elapses. This prevents retries from overwhelming a downstream service that is already failing.

**Cache invalidation for agent outputs.** Agent outputs can become stale. A cached result from yesterday may reference outdated data. Invalidation strategies include: time-to-live (TTL) based expiration, invalidation on data mutation (for example, after a write tool executes), and version-based invalidation where a new prompt version invalidates all caches created under the old version.
