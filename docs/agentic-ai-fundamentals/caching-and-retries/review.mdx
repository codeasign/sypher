# Review: Caching and Retries

⏱️ 5 minutes

## Key Takeaways

1. **Cache LLM responses aggressively** — identical or semantically similar queries can save 40-60 percent on API costs and reduce latency from seconds to milliseconds. Always set a TTL to prevent stale data.

2. **Retry with exponential backoff and jitter** — transient failures are inevitable. Exponential backoff gives the downstream service time to recover, and jitter prevents retry storms when multiple agents fail simultaneously.

3. **Distinguish retryable from non-retryable errors** — only retry 5xx, timeout, and rate-limit errors. Never retry 4xx, auth failures, or validation errors — they will never succeed and waste resources.

4. **Use circuit breakers for failing services** — when a tool or API is consistently failing, the circuit breaker stops all calls and gives the service time to recover. This prevents retries from compounding the failure.

5. **Validate responses before caching** — a single bad LLM response can poison the cache and serve incorrect answers to every matching query until the TTL expires. Always check response quality before storing it.

## Recap in 60 Seconds

1. **Cache first, ask second** — check the cache before every LLM call. Use exact-match keys for correctness and semantic keys for higher hit rates. Never cache sensitive or user-specific data without including user identity in the key.

2. **Backoff every retry** — add exponential delay (1s, 2s, 4s) plus jitter (randomize by 50 percent) to every retry cycle. Cap at a reasonable maximum (3-5 attempts) and stop for non-retryable errors.

3. **Circuit breakers stop the bleed** — after N consecutive failures, trip the breaker to OPEN. Use a recovery timeout to transition to HALF-OPEN for testing. Success in HALF_OPEN resets everything; failure restarts the timeout.
