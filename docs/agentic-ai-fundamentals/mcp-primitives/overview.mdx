## Why This Matters

An MCP server can expose three different kinds of capabilities: Tools, Resources, and Prompts. Choosing the wrong primitive for a given job leads to awkward integrations — exposing a write operation as a Resource, or exposing static data as a Tool that takes no arguments. Understanding the distinction between primitives is essential for designing servers that are intuitive to use and efficient to operate.

The three primitives map to three fundamental operations an agent might perform: **call a function** (Tool), **read data** (Resource), or **use a structured prompt** (Prompt). Many real-world servers expose all three, but each primitive has its own protocol methods, return types, and best practices.

When you design an MCP server, you are designing the API contract for every agent that connects to it. Getting the primitive choice right makes your server natural to use. Getting it wrong means every client needs workarounds.

## The Problem

You want to build an MCP server that provides access to a company's internal documentation. Should the documentation be exposed as a Tool or a Resource? A Tool that takes a search query and returns a document is how most developers think about it — "I call the tool with a question and it returns an answer." But a Resource is read-only data, and documentation is inherently read-only data. A Tool implies action. A Resource implies data.

If you expose documentation as a Tool, the client treats it as a callable function. Every documentation request goes through the tool-calling pipeline with tool selection, argument validation, and result processing. If you expose it as a Resource, the client can treat it as data — the agent can read it directly without going through the tool-selection decision process.

The distinction matters because it affects how agents use your server. A Tool must be selected by the model among all other tools. A Resource can be fetched directly. The wrong choice means the agent either struggles to find the documentation tool among 50 other tools, or cannot use the documentation because it was exposed as a function that requires arguments it does not have.

## The Concept

**Tools** are functions the agent can call. They accept arguments, perform actions, and return results. Tools are for operations that have side effects or require computation — sending a message, querying a database, transforming data. Each tool has a name, a description, and an input schema (JSON Schema format). Tools are discovered via `tools/list` and called via `tools/call`.

**Resources** are data objects the agent can read. They are identified by a URI and have a MIME type. Resources are for exposing structured or unstructured data — files, API responses, configuration objects. Resources are discovered via `resources/list` and read via `resources/read`. Unlike tools, resources do not take arguments beyond the URI itself.

**Prompts** are reusable prompt templates stored on the server. They accept arguments and produce a rendered prompt. Prompts are for capturing prompt engineering expertise — a well-crafted prompt for analyzing log output, a structured template for generating reports. Prompts are discovered via `prompts/list` and retrieved via `prompts/get`.
