# MCP Primitives — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Each primitive has specific use cases where it is not the right choice:

- **Tools for read-only data**: If your operation has no side effects and returns the same data every time, it should be a Resource, not a Tool. Tools imply action and go through model selection, adding latency.
- **Resources for operations**: If reading a Resource triggers a database query, email send, or file write, it should be a Tool. Resources must be read-only.
- **Prompts for single-use logic**: If a prompt template is only used once, inline it in the client rather than hosting it on an MCP server.
- **All three when one suffices**: Not every server needs all three primitives. A file server may only need Resources. A compute server may only need Tools.

## The Pattern to Remember

**Pattern**: Tools = actions with arguments and side effects; Resources = read-only data with URIs; Prompts = reusable templates that produce structured messages.

When to use Tools:
- Operations with side effects (send, write, modify)
- Operations requiring arguments for computation
- Model-mediated actions where the model decides

When to use Resources:
- Static or computed read-only data
- Configuration, files, API responses
- Data the client should fetch directly

When to use Prompts:
- Reusable prompt patterns
- Domain-specific analysis templates
- Standardized report generators

**Mental model**: A Tool does something, a Resource is something, and a Prompt frames something.

## Common Mistakes

### Mistake 1: Exposing Static Data as a Tool

**Why it happens**: The developer thinks of every server capability as a "function" and writes it as a Tool.

**Symptoms**: Tools with no arguments that return the same result every time. The model must decide whether to call a "static data" tool, adding latency and decision overhead.

**Fix**: Expose static data as a Resource. The client can read it directly without model involvement.

### Mistake 2: Using Resources for Write Operations

**Why it happens**: Resources look like simple data endpoints, so the developer adds write functionality to a resource URI.

**Symptoms**: A `resources/read` call triggers side effects — modifying state, sending messages, or writing to disk. This violates the MCP spec and confuses clients.

**Fix**: Keep Resources strictly read-only. If the operation has side effects, it must be a Tool.

### Mistake 3: Prompt Templates That Are Too Generic

**Why it happens**: You write a prompt template that is just "Answer the user's question" with no domain expertise.

**Symptoms**: The rendered prompt is no better than the client's default system prompt. The server adds latency without adding value.

**Fix**: A prompt should encode expertise that the client does not have — analysis frameworks, domain-specific algorithms, structured output formats. If it is just "answer a question," skip it.

### Mistake 4: Ignoring Resource Templates

**Why it happens**: You only register static resources with fixed URIs and do not support URI templates.

**Symptoms**: The server can only serve the exact resources it knew about at startup. Dynamic data (user files, query results) cannot be accessed through Resources.

**Fix**: Use resource templates (`uriTemplate`) to describe dynamic resources. The client substitutes parameters into the URI template to access specific resources.

### Mistake 5: Tool Descriptions That Do Not Differentiate

**Why it happens**: Multiple tools have similar descriptions because they do conceptually related things.

**Symptoms**: The model frequently calls the wrong tool because it cannot distinguish between "search_docs" and "search_knowledge_base" based on their descriptions.

**Fix**: Write tool descriptions that explicitly say when to use this tool vs. another. Use phrases like "Use this when..." and "Do not use for..."
