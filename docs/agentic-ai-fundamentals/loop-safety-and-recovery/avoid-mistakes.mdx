# Loop Safety and Recovery — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Full safety systems are not always necessary:

- **Prototypes**: Safety layers add complexity. Prototypes do not need circuit breakers.
- **Read-only agents**: If the agent never writes or sends data, the risk profile is much lower.
- **Human-supervised agents**: If a human watches every action, automated safety is less critical.
- **Short-lived agents**: Agents that run for 1-2 turns and exit are unlikely to hit limits.

## The Pattern to Remember

**Pattern**: Hard limits (unconditional stops) + recovery strategies (retry, fallback) + circuit breaker (pattern-based halt).

When to use full safety:
- Production deployments
- Agents with write/send/delete tools
- Long-running or unattended agents

When NOT to use:
- Quick prototypes
- Read-only agents
- Fully supervised operation

**Mental model**: Safety is like a building's fire safety system. Hard limits are the sprinklers that go off automatically. Recovery strategies are the fire extinguishers. The circuit breaker is the fire alarm that detects the pattern of smoke.

## Common Mistakes

### Mistake 1: Limits That Are Too Generous

**Why it happens**: You set max_turns to 1000 and max_time to 1 hour "just to be safe."

**Symptoms**: An agent that gets stuck runs for 45 minutes before the time limit stops it. You pay for 1000 turns of useless tool calls.

**Fix**: Set limits based on real task requirements. If the task typically takes 5 turns, set the limit to 15 (3x headroom). Monitor actual usage and tighten limits over time.

### Mistake 2: Retry Storm

**Why it happens**: Multiple agents share the same failing service. Each one retries simultaneously, overwhelming the service further.

**Symptoms**: A database outage causes all agents to retry at the same intervals. The retry storm prevents the database from recovering.

**Fix**: Use jitter in retry delays. Instead of retrying at 1s, 2s, 4s, use random delays: 0.8-1.2s, 1.6-2.4s, 3.2-4.8s. This spreads retries across time.

### Mistake 3: Circuit Breaker That Never Closes

**Why it happens**: The recovery timeout is too short. The breaker transitions to HALF-OPEN before the underlying issue is resolved.

**Symptoms**: The breaker oscillates between OPEN and HALF-OPEN, blocking requests in OPEN and failing them in HALF-OPEN.

**Fix**: Make the recovery timeout proportional to the failure duration. If failures persisted for 5 minutes, the recovery timeout should be at least 5 minutes.

### Mistake 4: Silent Limit Breach

**Why it happens**: When a limit is reached, the agent stops silently or returns an uninformative message.

**Symptoms**: The user sees "Agent stopped" with no reason. They do not know whether the task is complete, failed, or needs action.

**Fix**: Always return a structured error message when limits are hit: which limit was hit, what was accomplished, and what the user should do next.

### Mistake 5: Not Testing Safety Systems

**Why it happens**: Safety systems are tested in "happy path" scenarios where nothing goes wrong.

**Symptoms**: When the real failure happens — the database times out, the model returns garbage — the safety system has bugs that were never exercised.

**Fix**: Write tests that specifically trigger each safety system. Force tool failures, slow responses, and high error rates in test environments.
