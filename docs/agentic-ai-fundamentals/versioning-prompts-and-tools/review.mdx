# Review: Versioning Prompts and Tools

⏱️ 5 minutes

## Key Takeaways

1. Every prompt and tool definition should carry a version identifier — use content hashes for prompts and semantic version strings for tool schemas — so you can always reconstruct what an agent session was running
2. Registry-based versioning decouples prompt deployment from code deployment, letting you update prompts without rebuilding containers, but requires an audit trail and rollback support to manage the risk of uncoordinated changes
3. Immutable session pinning ensures that a running agent session continues with the prompt and tool versions it started with, even if the deployment tag moves to a newer version mid-session
4. Tool description text is prompt content, not implementation detail — if you version your system prompt but not your tool descriptions, you have an incomplete picture of what the model saw
5. Every versioned system needs a rollback procedure that works in seconds, not hours, and a dry-run mode so you can preview the impact of a rollback before executing it

## Recap in 60 Seconds

If you only remember three things:

1. **Tag it, hash it, pin it** — use semantic tags for readability, content hashes for uniqueness, and session pinning for runtime stability
2. **Version everything the model sees** — system prompt, tool names, tool descriptions, and parameter schemas all influence model behavior and all change independently
3. **Roll forward or back with confidence** — a registry with audit history and dry-run rollback means you never have to guess what happened or hope a revert works
