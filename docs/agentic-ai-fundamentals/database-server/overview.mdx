# Building a Database MCP Server

⏱️ 15 minutes · Difficulty: Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Configure the SQLite and PostgreSQL MCP servers
- Let your agent query, insert, and update database records through tools
- Restrict queries to read-only mode for safe exploration
- Implement schema introspection tools that let the agent understand your database structure

## Prerequisites

You should already understand:

- How MCP tools work (see [Agent Tools and Tool Calling](../../agent-tools-and-tool-calling/overview))
- Basic SQL syntax — SELECT, INSERT, CREATE TABLE
- How database connections and connections pools work
- SQL injection and why parameterized queries are essential

## Why This Matters

Your application's data lives in a database. When an agent can query that database directly, it stops guessing about data and starts reasoning from facts. Instead of asking "how many users signed up last week?" and waiting for a developer to run a query, the agent runs the query itself and gives you the answer immediately.

Customer support teams use database-connected agents to look up user accounts, check order status, and investigate billing issues without waiting for engineering. Data analysts use them to explore schemas and generate reports. Engineering teams use them to debug production issues by inspecting recent records, checking error logs, and validating data migrations.

The challenge is safety. You cannot give an LLM carte blanche to run SQL on your production database. One off-by-one in a prompt could produce `DELETE FROM users` instead of `SELECT * FROM users WHERE id = 123`. A database MCP server solves this by wrapping database access in safe, bounded tools — read-only by default, write enabled only when explicitly configured, with query-timeout guards and row limits on every result set.

## The Problem

A developer debugging a production issue in a customer chat says "check the orders table for user 456 in the last hour." The agent needs to connect to a database, explore what tables exist, understand the schema, run the query, and return the results.

Without a database MCP server, you would need to give the agent a generic SQL execution tool. That tool would need connection string management, credential handling, query timeout enforcement, result truncation, and SQL injection prevention — all while the agent might produce a malformed query that locks a table or drops a column.

Worse, if the agent can run arbitrary SQL, it can run destructive commands. A database MCP server solves this by offering discrete tools — `read_query`, `write_query`, `list_tables`, `describe_table` — each with its own guardrails. `read_query` never executes INSERT, UPDATE, DELETE, or DDL statements, even if the agent tries.

## The Concept

A database MCP server sits between the agent and the database, translating MCP tool calls into parameterized SQL queries. The server knows the database schema and can introspect tables, columns, indexes, and foreign keys. When the agent calls `describe_table("orders")`, the server queries `INFORMATION_SCHEMA` (or the database-specific equivalent) and returns the column names, types, and constraints.

Each query runs through a query parser that classifies it as read-only or write. Read-only queries go to the database immediately. Write queries are either rejected (if read-only mode) or require an explicit confirmation step. Every query has a configurable timeout (default 30 seconds) and a row limit (default 1000 rows) to prevent runaway queries from consuming resources.
