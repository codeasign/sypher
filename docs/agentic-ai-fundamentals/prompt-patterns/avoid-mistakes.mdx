# Prompt Patterns — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Prompt patterns are tools, not rules. They are not always the right approach:

- **Very simple tasks**: "Translate this to French" does not need a pattern. Use a direct instruction.
- **Tool/function calling**: When the API enforces a schema via tool definitions, adding an extraction pattern on top is redundant.
- **Creative writing**: Patterns constrain output, which is the opposite of what you want for free-form creative tasks. Use a persona or style instruction instead.
- **Single-use prompts**: If you will only use a prompt once, spend your time on the task itself rather than designing a reusable pattern.

## The Pattern to Remember

**Pattern**: Recognize the task type, select the matching pattern, tailor it to your data.

When to use:
- Tasks that follow a known shape (classification, extraction, comparison)
- Prompts that will be reused across multiple inputs
- Code that other developers will maintain

When NOT to use:
- Novel, creative, or highly unstructured tasks
- Tasks where the API already enforces the structure via tool schemas

**Mental model**: Prompt patterns are design patterns for LLM interactions. Just as you would not write a web server without using the request-response pattern, do not write a classification prompt without using the label-constraint pattern.

## Common Mistakes

### Mistake 1: Using the Wrong Pattern for the Task

**Why it happens**: You learn one pattern and apply it to everything.

**Symptoms**: The persona pattern is used for classification, producing long role-played outputs instead of labels. The template pattern is used for creative writing, producing stiff, formulaic content.

**Fix**: Match the pattern to the task. Classification needs tight label constraints. Creative writing needs persona or style guidance. Extraction needs structured templates.

### Mistake 2: Combining Too Many Patterns in One Prompt

**Why it happens**: You try to cover every edge case by layering persona + label-constraint + template + comparison + chain-of-thought.

**Symptoms**: The model gets confused about which pattern to follow. The output sometimes follows one pattern, sometimes another.

**Fix**: Use one primary pattern per task. If you need multiple patterns, split the work into multiple API calls — one for each pattern — and combine the results in your application code.

### Mistake 3: Forgetting to Validate Pattern Output

**Why it happens**: The pattern seems to work, so you assume the output always matches.

**Symptoms**: The label-constraint pattern occasionally returns a label not in your set. The template pattern occasionally changes a field name. These edge cases break downstream code.

**Fix**: Always validate pattern outputs against your expected schema. If the label is not in your set, retry or default to a fallback.

### Mistake 4: Making Patterns Too Rigid

**Why it happens**: You specify every detail of the output format, leaving no room for the model to handle variations.

**Symptoms**: The model fails on inputs that do not exactly match the expected pattern. Small formatting differences cause the model to produce unusable output.

**Fix**: Build tolerance into your pattern handling. Use post-processing to normalize output rather than requiring the model to be pixel-perfect.

### Mistake 5: Not Testing Patterns Across Diverse Inputs

**Why it happens**: You test with three examples that all work perfectly.

**Symptoms**: In production, the pattern fails on edge cases you never considered — input that is too short, too long, multilingual, or in an unexpected format.

**Fix**: Test patterns across the full range of expected inputs, including edge cases. A pattern that fails on 5% of production traffic is not production-ready.
