# Observe-Think-Act Loop — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

The observe-think-act loop is the universal agent pattern, but not every system needs a loop:

- **Single-shot tasks**: If you need one answer from one API call, a loop adds unnecessary complexity.
- **Streaming responses**: For real-time streaming, the loop model (request-response) is replaced by event-driven processing.
- **Background batch processing**: For batch jobs, use a queue-based architecture instead of a live loop.

## The Pattern to Remember

**Pattern**: Three phases — observe (receive input), think (decide), act (execute) — repeated until done or halted.

When to use:
- Any multi-step agent task
- Systems that interact with external tools
- Tasks requiring iterative refinement

When NOT to use:
- Single API call tasks
- Pure streaming pipelines
- Scheduled batch jobs

**Mental model**: The loop is like a while loop in programming — keep going until the condition changes. The condition is "task not done, no errors, and limits not exceeded."

## Common Mistakes

### Mistake 1: No Termination Condition

**Why it happens**: You expect the model to always produce a final answer after some number of turns.

**Symptoms**: The agent runs forever, calling tools and generating text but never reaching a clear stopping point. Token costs grow without bound.

**Fix**: Always set at least a `max_turns` limit. Also check for stall conditions: if the last N turns produced the same tool call with the same arguments, stop.

### Mistake 2: Mixing Observation and Action

**Why it happens**: You append the tool result and the next user message in the same turn, confusing the observation with a new request.

**Symptoms**: The model treats tool results as new user instructions. It acts on data it should only observe.

**Fix**: Keep a strict separation: observations go into `tool_result` blocks, new actions go as separate `user` messages. Never merge them.

### Mistake 3: Premature Loop Exit

**Why it happens**: The model produces a text response that sounds like a final answer, but the task is not complete.

**Symptoms**: The agent produces an intermediate result and stops. The user gets a partial answer.

**Fix**: Check whether all required actions have been taken before treating text output as final. For complex tasks, have the model explicitly signal completion.

### Mistake 4: State Corruption in the Message List

**Why it happens**: You modify message content in place, append out-of-order, or reuse message IDs incorrectly.

**Symptoms**: The model gets confused about the conversation state. Tool results appear before the tool call. The model repeats actions or loses context.

**Fix**: Append messages in strict order: user → assistant (tool_use) → user (tool_result). Never modify existing messages. Always use fresh `tool_use_id` values.

### Mistake 5: No Error Recovery in the Loop

**Why it happens**: The loop assumes every tool call succeeds and every model response is valid.

**Symptoms**: A single tool failure crashes the entire loop. A malformed model response (not a tool call, not text) causes an unhandled exception.

**Fix**: Wrap every tool execution in try/except. Handle unexpected response types gracefully. Log errors and continue rather than crashing.
