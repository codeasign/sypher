# Agent Memory — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Memory strategies beyond the basic sliding window are not always needed:

- **Stateless agents**: If each request is independent and no context carries over, no memory system is needed.
- **Very short interactions**: A 2-3 turn conversation fits easily in any model's context window.
- **One-shot tasks**: The agent runs once, produces output, and exits. Memory resets each time.

## The Pattern to Remember

**Pattern**: Use short-term memory for the current conversation window, long-term memory for persistent facts, and episodic memory for learning from mistakes.

When to use:
- Multi-turn conversations where context accumulates
- Agents that run across multiple sessions
- Tasks where the agent should learn from past errors

When NOT to use:
- Single-turn stateless agents
- Very simple Q&A (sliding window is sufficient)
- Prototypes before memory is a bottleneck

**Mental model**: Short-term memory is your desk — what you are working on right now. Long-term memory is your filing cabinet. Episodic memory is your notebook of lessons learned.

## Common Mistakes

### Mistake 1: Including Everything in Context

**Why it happens**: You are afraid of losing information, so you keep every message.

**Symptoms**: The context window fills up after 10-15 turns. Token costs increase linearly with conversation length. The model becomes slower and less focused.

**Fix**: Be selective. Use sliding windows for routine conversations. Summarize or archive old turns. Not everything needs to stay in context.

### Mistake 2: Summarization Losing Critical Facts

**Why it happens**: A summarization call compresses a 10-turn conversation into 2 sentences, dropping specific numbers, names, or instructions.

**Symptoms**: The agent forgets the user's name, account number, or specific preferences after a summarization step.

**Fix**: Store critical facts (names, IDs, preferences) in a separate key-value store that is not subject to summarization. Inject these facts back into the context explicitly.

### Mistake 3: Retrieval That Returns Irrelevant Results

**Why it happens**: The retrieval system uses simple keyword matching that returns loosely related documents.

**Symptoms**: The context is polluted with irrelevant memories. The model wastes tokens reasoning about why unrelated information was injected.

**Fix**: Use embedding-based retrieval with a relevance threshold. Only inject memories above the threshold. If using keyword retrieval, require multiple keyword matches.

### Mistake 4: Not Handling Memory at Scale

**Why it happens**: Your retrieval memory works in development with 50 entries, but fails in production with 50,000.

**Symptoms**: Retrieval latency spikes. Relevance scores drop. The vector store becomes slow to query.

**Fix**: Design for scale from the start. Use proper vector databases (Pinecone, Weaviate, Chroma) rather than in-memory stores for production.

### Mistake 5: Race Conditions in Memory Updates

**Why it happens**: Multiple agent turns or multiple agents write to the same memory store concurrently.

**Symptoms**: Memory state becomes inconsistent. A fact that was just stored is not found on retrieval. The agent sees stale data.

**Fix**: Use write locks or atomic operations for memory updates. For single-agent systems, sequential processing avoids this naturally.
