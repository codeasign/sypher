# MCP Architecture — Review

*Reading time: 3 minutes*

## Key Takeaways

- MCP follows a strict lifecycle: transport connect, initialize request/response, initialized notification, operation, transport close
- Servers must reject operations received before initialization completes — use an initialized flag to enforce this
- Capability negotiation happens during initialization: both sides declare what primitives they support
- The `id` field in JSON-RPC messages is essential for matching responses to requests, especially over SSE transport
- Transport (stdio vs. SSE) is an implementation detail — the same handler logic works for both

## Recap in 60 Seconds

1. The MCP lifecycle has four phases: connect, initialize (with capability negotiation), operate, and close — skip none of them
2. Servers must guard against premature operations by checking an initialized flag before handling tool calls or resource reads
3. Always correlate responses to requests by the JSON-RPC `id` field — this works across both stdio and SSE transports
