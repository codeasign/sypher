# MCP with Claude Code — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Configuring MCP servers in Claude Code is not always the right approach:

- **One-time tool use**: If you only need a tool for a single task, running the server manually and piping input is faster than configuring settings.json.
- **Sensitive credentials in shared config**: If your MCP server needs API keys or database credentials, use `.claude/settings.local.json` (gitignored), not `.claude/settings.json` which is checked into version control.
- **Server with GUI or interactive input**: MCP servers in Claude Code run as headless child processes over stdio. If your server needs a GUI or interactive terminal, it will not work.
- **Server listening on HTTP**: Claude Code uses stdio transport for MCP. SSE/HTTP servers require a custom MCP client — they cannot be configured directly in `mcp_servers`.

## The Pattern to Remember

**Pattern**: Configure MCP servers in `mcp_servers` within Claude Code's settings files. Each entry has `command`, optional `args`, and optional `env`. Claude Code spawns the server, initializes it, and merges its tools.

When to use:
- Adding custom tools to every Claude Code session
- Sharing tool configurations with your team via `.claude/settings.json`
- Integrating private databases, APIs, or file systems

When NOT to use:
- One-off tool calls
- Servers needing GUI or interactive input
- HTTP/SSE-based servers (Claude Code uses stdio transport)

**Mental model**: `mcp_servers` in settings.json is a plugin system for Claude Code. Each server adds tools to the model's toolkit.

## Common Mistakes

### Mistake 1: Incorrect JSON in settings.json

**Why it happens**: A trailing comma, missing quote, or extra brace in the settings file.

**Symptoms**: Claude Code silently ignores the invalid JSON and loads no MCP servers. No error message is shown.

**Fix**: Validate your settings file with `python -c "import json; json.load(open('path/to/settings.json'))"` after every edit.

### Mistake 2: Wrong Path in args

**Why it happens**: You use a relative path for the server script, but Claude Code's working directory is different from what you expect.

**Symptoms**: The server command fails with "file not found" or "module not found." Claude Code starts without the server's tools.

**Fix**: Use absolute paths for server scripts, or paths relative to the project root where Claude Code is started. Test the server command independently in the same directory before adding it to settings.

### Mistake 3: Server Does Not Follow MCP Protocol

**Why it happens**: You build a server that does not implement the full MCP lifecycle — missing `notifications/initialized`, incorrect JSON-RPC format, or missing capability declarations.

**Symptoms**: Claude Code spawns the server, the initialization handshake fails silently, and the server's tools never appear.

**Fix**: Test your server with the MCP Inspector before adding it to Claude Code. The Inspector shows you exactly what the server sends and receives, making protocol violations visible.

### Mistake 4: Server Runs Forever on Startup

**Why it happens**: Your server script has a bug that causes it to hang during startup — for example, waiting for input that never arrives.

**Symptoms**: Claude Code starts but is slow or unresponsive. Multiple hung server processes accumulate in the process list.

**Fix**: Add a startup timeout to your server. If initialization takes more than 10 seconds, log an error and exit. Test the server independently before configuring it.

### Mistake 5: Conflicting Tool Names

**Why it happens**: Two MCP servers define a tool with the same name (e.g., both have a `search` tool).

**Symptoms**: Claude Code merges both tools, but the model may call the wrong one. The behavior depends on which server's tool definition appears first in the merged list.

**Fix**: Give your tools unique, descriptive names that include the domain (e.g., `filesystem_search` vs. `database_search` instead of `search`).
