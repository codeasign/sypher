# Tokens and Context Windows — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Token optimization strategies are important, but not every system needs aggressive trimming:

- **Short, single-turn interactions**: If users send one message and get one response, context window management is trivial — your prompt always fits
- **Prototypes and demos**: Prematurely optimizing token usage adds complexity without benefit. Get it working first, then optimize
- **Models with very large windows**: A 200K token window is so large that most conversations fit without any trimming strategy
- **Stateless operations**: If every request is self-contained (no conversation history), context windows are not a constraint

## The Pattern to Remember

**Pattern**: Budget tokens before calling — always know how many tokens your prompt uses and whether the response fits.

When to use:
- Multi-turn conversations where context accumulates
- RAG systems where you inject many document chunks
- Any system where prompts vary in size dynamically
- Systems serving thousands of requests per day (cost optimization)

When NOT to use:
- Single-turn, fixed-prompt systems
- Prototypes and proof-of-concept work

**Mental model**: The context window is a fixed-size suitcase. Pack the essentials first (system prompt, current query), then add what fits (recent history, relevant documents), and leave space for the reply.

## Common Mistakes

### Mistake 1: Assuming One Token Equals One Word

**Why it happens**: It is an easy mental shortcut, and the model returns whole words, so it seems intuitive.

**Symptoms**: Your cost estimates are consistently underestimated by 20-40%.

**Fix**: Use the 1.3 tokens-per-word heuristic for English text. For code or non-English text, use an actual tokenizer to verify.

### Mistake 2: Not Budgeting for the Response

**Why it happens**: You calculate whether the prompt fits but forget that the response also consumes context window space.

**Symptoms**: Calls that should fit fail with context length errors. The model produces truncated responses.

**Fix**: Always add `input_tokens + max_tokens ≤ context_window` as your budget check. Never compare only input against the window.

### Mistake 3: Appending Everything to the Conversation Forever

**Why it happens**: The API accepts everything, so there is no immediate error. Developers do not realize the conversation is growing until it breaks.

**Symptoms**: Costs increase linearly with time. Eventually the API starts returning errors on long conversations.

**Fix**: Set a maximum number of conversation turns and a maximum token budget for history. Trim, summarize, or drop old turns when exceeded.

### Mistake 4: Ignoring Token Costs at Scale

**Why it happens**: A few test calls cost pennies. The cost model does not sink in until the bill arrives.

**Symptoms**: Production bills are 10x what was expected. The team scrambles to optimize under pressure.

**Fix**: Before deploying, calculate your cost per request and multiply by expected monthly volume. Set budget alerts from day one.

### Mistake 5: Using the Same Context Strategy for All Models

**Why it happens**: Engineers develop a working strategy with one model and apply it universally.

**Symptoms**: On a model with a 200K window, you over-trim and lose useful context. On an 8K window model, your trimming strategy is not aggressive enough.

**Fix**: Parameterize your context management by model. Each model has a different window size and different cost per token. Adjust your strategy accordingly.

```python
MODEL_CONFIGS = {
    "claude-3-5-sonnet-20241022": {"window": 200000, "cost_input": 3.0, "cost_output": 15.0},
    "claude-3-haiku-20240307": {"window": 200000, "cost_input": 0.25, "cost_output": 1.25},
}


def get_strategy(model: str, conversation: list[dict]) -> dict:
    """Return the context management strategy for a specific model."""
    config = MODEL_CONFIGS.get(model)
    if not config:
        raise ValueError(f"Unknown model: {model}")

    # Reserve 20% of the window for the response
    response_budget = int(config["window"] * 0.2)
    prompt_budget = config["window"] - response_budget

    return {
        "model": model,
        "window": config["window"],
        "prompt_budget": prompt_budget,
        "response_budget": response_budget,
        "cost_per_1k_input": config["cost_input"] / 1000,
    }
```
