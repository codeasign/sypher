# LLM APIs in Practice — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Building a production API client wrapper is essential for most applications, but not every scenario needs one:

- **Prototyping and exploration**: Use the raw SDK for quick experiments. A wrapper adds complexity you do not need yet.
- **Serverless functions with minimal traffic**: If your function runs once per invocation with no retry concerns, a simple SDK call is sufficient.
- **Batch jobs with built-in retry**: Some orchestration frameworks (Airflow, Prefect) handle retries at the task level — duplicating retry in the API client adds no value.

## The Pattern to Remember

**Pattern**: Client wrapper with retry, cost tracking, and structured responses.

When to use:
- Any service that will handle production traffic
- Systems where cost monitoring matters
- Code that will be maintained by other developers

When NOT to use:
- Prototypes and one-off scripts
- Systems where the orchestration layer already handles retries

**Mental model**: Your API client is the network boundary of your application. Treat it like any other I/O boundary — wrap it in an abstraction, handle errors explicitly, log everything, and never let raw exceptions bubble up to the user.

## Common Mistakes

### Mistake 1: Not Handling Streaming and Non-Streaming Differently

**Why it happens**: Developers treat streaming as an optional flag without understanding that the response API differs.

**Symptoms**: Code that parses `response.content[0].text` fails silently on streaming calls because the streaming response object is different.

**Fix**: Always check whether the response is a streaming or non-streaming result. Use the SDK's context manager pattern for streams (`with client.messages.stream(...) as stream`) and the direct call for non-streaming.

### Mistake 2: Retrying Everything the Same Way

**Why it happens**: A single `try/except` catches all errors and retries them identically.

**Symptoms**: Authentication errors get retried (wasting time), while transient server errors do not get enough retries (causing unnecessary failures).

**Fix**: Classify errors. Rate limits deserve backoff. Server errors (5xx) deserve retry. Auth errors (401) and bad request errors (400) should fail immediately.

### Mistake 3: Forgetting to Track Token Usage

**Why it happens**: The response works and the text is correct, so developers do not inspect metadata.

**Symptoms**: No visibility into costs until the bill arrives. No data to optimize prompt lengths.

**Fix**: Log `response.usage.input_tokens` and `response.usage.output_tokens` on every API call. Calculate and aggregate costs.

### Mistake 4: Hardcoding Model Names

**Why it happens**: The model name is a string passed to the API call, and it works, so it stays as a string literal.

**Symptoms**: Upgrading models requires searching through code for every hardcoded string. You accidentally use different models in different parts of the system.

**Fix**: Define model constants or configuration at the application level. Reference the config everywhere.

### Mistake 5: Ignoring Timeout Configuration

**Why it happens**: The default SDK timeout (usually 60-120 seconds) works most of the time.

**Symptoms**: Long prompts or complex generation tasks hit the timeout. The request fails even though the model would have completed given more time.

**Fix**: Set the timeout explicitly based on your expected token count. As a rule of thumb, allow 1-2 seconds per 100 output tokens plus buffer for the prompt.

```python
import anthropic

# Estimate required timeout
expected_input_tokens = 5000
expected_output_tokens = 2000
# Rough: ~100 tokens/sec output, add 10s buffer for input processing
estimated_timeout = 10 + (expected_output_tokens / 100) * 2

client = anthropic.Anthropic(timeout=estimated_timeout)
```
