# Configuration Management

⏱️ 15 minutes · Difficulty: Advanced

## What You Will Learn

By the end of this lesson you will be able to:

- Manage agent system configuration across multiple environments (development, staging, production)
- Apply configuration-as-code patterns with layered merging and validation
- Implement runtime configuration reload without redeploying your agent
- Handle secrets management securely alongside regular configuration

## Prerequisites

You should already understand:

- How environment variables work and how to set them in your shell and deployment platform
- Basic YAML and JSON syntax for configuration files
- The anatomy of an agent — system prompts, tool definitions, model parameters (see [Anatomy of an Agent](../anatomy-of-an-agent/overview))
- How agents are deployed and run in production (see [Deployment Patterns](../deployment-patterns/overview))

## Why This Matters

An agent's behavior is defined by its configuration. Change the system prompt and the agent responds differently. Change the model temperature and reasoning depth shifts. Change the tool list and the agent gains or loses capabilities. Every one of these parameters is a configuration value, and every one of them differs between development, staging, and production. A single agent deployment may have dozens of configurable knobs — model selection, tool definitions, rate limits, memory settings, timeout durations, logging levels, and feature flags. Managing these consistently across environments is the difference between a reliable production system and one that breaks unexpectedly when you deploy.

Configuration errors are a top cause of production incidents in agent systems. A misconfigured environment variable can silently switch an agent to the wrong model, point it at the wrong database, or disable critical safety features. Unlike code bugs, configuration errors often go unnoticed by tests because tests typically run with their own test configuration. They surface only in production, where they can cause data corruption, security breaches, or extended outages. A systematic approach to configuration management prevents these failures and gives your team confidence that every environment is running the right settings.

## The Problem

Hardcoding configuration works when you have one environment running one agent. You write the system prompt as a constant, set the model name in a string literal, and define tool parameters inline. Everything works on your laptop.

When you move to staging, you copy the code and change a few values. When you add production, you copy again. Now you have three copies of the same configuration logic, each with slightly different values. You forget to update one when you add a new tool. The staging agent has a different timeout than production. Someone commits the production API key to the repository by accident.

Environment variables help, but they grow unmanageable. A production agent can easily require 20-30 environment variables. Nobody documents them all. New team members miss half of them in their local setup. The CI pipeline uses different variable names than the deployment. A missing variable causes a silent fallback to defaults, and no one notices until the agent behaves wrong in production.

This is config drift — the gradual divergence of configuration across environments. It is one of the hardest problems to debug because the code is identical. The behavior difference is invisible in the codebase. You can read every line and never see the bug, because the bug is in a YAML file on the server that differs from the one in your repository.

## The Concept

Configuration management for agent systems uses three complementary channels, each with a specific role:

- **Configuration files** — YAML or JSON files checked into version control, one per environment. These contain non-sensitive settings that differ between environments: model names, tool lists, logging levels, timeout values, system prompt templates. They give you a single source of truth that is reviewed, versioned, and deployable.

- **Environment variables** — Runtime values injected by the deployment platform. These are used for deployment-specific settings like database URLs, service endpoints, and instance identifiers. They bridge the gap between your config files and the runtime environment.

- **Secrets management** — API keys, authentication tokens, and database credentials are never stored in config files or environment variables directly. They are fetched from a secrets store (HashiCorp Vault, AWS Secrets Manager, or a local `.env` file in development) and injected at runtime.

Configuration is resolved by layering these sources. Base defaults provide safe fallback values. Environment-specific YAML files override those defaults. Secrets are injected separately. Runtime overrides — such as feature flags or A/B test parameters — apply last. Each layer only specifies the values that differ from the layer below, keeping configuration minimal and auditable.

Feature flags extend this model. They let you toggle agent behaviors — enable a new tool, switch to a different reasoning strategy, activate a safety check — without changing configuration files or redeploying. Feature flags are typically stored in a database or external service and checked at runtime, giving you dynamic control over agent behavior while the agent is running.
