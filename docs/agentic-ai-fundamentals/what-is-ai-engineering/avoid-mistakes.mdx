# What is AI Engineering — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

AI engineering — integrating LLMs into production systems — is not always the right approach. Consider alternatives in these situations:

**Deterministic problems**: If your task can be solved with a lookup table, regular expression, or decision tree, do that instead. An LLM adds latency, cost, and nondeterministic behavior where none is needed. Validating a credit card number or parsing a date does not require a language model.

**High-volume, low-latency paths**: LLM inference takes hundreds of milliseconds at minimum. If you need sub-10ms responses for thousands of requests per second, traditional algorithms or smaller embedding models are better suited.

**Regulated environments without guardrails**: If you cannot tolerate the model occasionally producing incorrect output (healthcare diagnoses, financial calculations), you need deterministic verification layers before the LLM output reaches users. The LLM alone is insufficient.

**Data sensitivity**: When using third-party APIs, your prompts are sent to external servers. If your data cannot leave your infrastructure, consider self-hosted models or on-premise deployments.

## The Pattern to Remember

**Pattern**: LLM-as-component — treat the model as a probabilistic service that you integrate behind an abstraction layer, never scattered directly through your codebase.

When to use:
- Natural language understanding or generation is core to the feature
- The problem has high variability in inputs
- You need to handle edge cases you cannot enumerate

When NOT to use:
- The problem has a known algorithmic solution
- You need guaranteed correctness
- Latency or cost budgets are extremely tight

**Mental model**: Think of an LLM as a junior developer who is extremely knowledgeable but occasionally makes up facts. You would never let a junior developer push code without a review — similarly, never pass LLM output to users without validation.

## Common Mistakes

### Mistake 1: Hardcoding the API Key

**Why it happens**: You want to get something working quickly and paste the key directly into your script.

**Symptoms**: The key gets committed to version control, exposed in screenshots, or leaked through CI logs.

**Fix**: Always use environment variables or a secrets manager. The Anthropic SDK reads `ANTHROPIC_API_KEY` from the environment automatically.

### Mistake 2: Ignoring Token Usage

**Why it happens**: The first few calls are cheap, so you do not track token counts.

**Symptoms**: Your bill surprises you at the end of the month. A production loop that retries failed calls multiplies costs unexpectedly.

**Fix**: Log `response.usage.input_tokens` and `response.usage.output_tokens` on every call. Set budget alerts in your provider dashboard.

### Mistake 3: Sending the Entire Conversation on Every Turn Without Truncation

**Why it happens**: Each call needs the full history, so you keep appending to an ever-growing messages array.

**Symptoms**: You hit the context window limit on long conversations. Costs increase linearly with conversation length.

**Fix**: Implement a sliding window — keep the system prompt, the most recent N turns, and optionally summarize older context.

### Mistake 4: Assuming Output Is Deterministic

**Why it happens**: The same prompt sometimes gives different answers, and you did not expect variance.

**Symptoms**: Tests fail intermittently. Users report inconsistent responses.

**Fix**: Set `temperature=0` for maximum determinism. For critical outputs, validate the response before using it (check format, values, constraints).

### Mistake 5: Not Handling API Errors

**Why it happens**: You test locally with a working key and assume the API will always respond.

**Symptoms**: Production crashes when rate limits are hit, the API is temporarily down, or your key expires.

**Fix**: Wrap API calls in try/except blocks with proper error handling and retries with exponential backoff.

```python
import anthropic
import time

client = anthropic.Anthropic()
max_retries = 3

for attempt in range(max_retries):
    try:
        response = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=256,
            messages=[{"role": "user", "content": "Hello"}]
        )
        break
    except anthropic.RateLimitError:
        if attempt < max_retries - 1:
            time.sleep(2 ** attempt)  # Exponential backoff
    except anthropic.APIError as e:
        print(f"API error: {e}")
        raise
```
