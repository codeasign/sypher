# Prompt Engineering Basics

*Reading time: 5 minutes*

**Difficulty:** Beginner

## What You Will Learn

By the end of this lesson you will be able to:

- Identify the essential elements of an effective prompt
- Structure prompts using role, task, format, and constraints
- Distinguish between good and bad prompts systematically
- Debug a prompt that is producing poor results

## Prerequisites

You should already understand:

- How LLMs predict tokens one at a time (covered in [LLM Fundamentals](../llm-fundamentals/overview))
- How inference parameters like temperature affect output (covered in [LLM Fundamentals](../llm-fundamentals/build-it))
- What tokens are and how they are counted (covered in [Tokens and Context Windows](../tokens-and-context-windows/overview))

## Why This Matters

Prompt engineering is the primary way you control an LLM's behavior. Unlike traditional programming where you write logic that executes deterministically, prompt engineering is about designing input text that guides a probabilistic model toward your desired output. It is not "magic" — it is a systematic skill with principles, patterns, and debugging techniques.

The difference between a junior and senior AI engineer is visible in their prompts. The junior writes vague instructions and blames the model when the output is wrong. The senior writes precise prompts with explicit constraints, validates the output, and iterates based on observed failures. Prompt quality directly determines output quality, which makes it the highest-leverage skill in the AI engineering toolkit.

## The Problem

You ask the model: "Tell me about machine learning." It returns a 500-word essay on the history of AI, which is not what you wanted. You try: "List 3 types of machine learning." It returns "1. Supervised, 2. Unsupervised, 3. Reinforcement learning" — better, but it is a bare list with no context. You wanted a comparison table with examples.

The model did exactly what you asked, but you did not specify the format, scope, audience, or constraints. This is not the model being bad — it is the prompt being underspecified. In software engineering, an underspecified function raises a type error. In prompt engineering, underspecification produces output that looks plausible but is wrong for your use case.

## The Concept

A well-structured prompt contains several elements that work together:

**Role** defines who the model should be. "You are a senior data scientist" produces different output than "You are a 5th-grade teacher" for the same task.

**Task** states what you want done. Be specific: "Create a comparison table" is clearer than "Tell me about X."

**Context** gives background information. Include relevant details the model needs to produce accurate output.

**Format** specifies the structure of the response. "Return as JSON with keys: name, description, example" is far more reliable than "List them out."

**Constraints** set boundaries. "Use no more than 50 words per entry" or "Cite specific statistics" narrows the output space.

**Examples** (few-shot) show the model what you want. A single example is often enough to establish the pattern.

The most important rule: if the model produces wrong output, the bug is almost always in your prompt, not in the model. Before blaming the model, examine your prompt for missing elements, ambiguous language, or contradictory instructions.
