# Building an MCP Server — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Building a full MCP server is not always the best approach:

- **Single-use tool**: If a tool is only used by one agent in one workflow, a hard-coded function in the agent code is simpler.
- **Rapid prototyping**: When iterating on tool design, test the tool logic directly before wrapping it in MCP.
- **Extremely high throughput**: If a tool handles thousands of calls per second, the JSON-RPC serialization and transport overhead of MCP may not be suitable. Consider a direct API.
- **Tools with no discoverability need**: If every client already knows exactly what tools are available and how to call them, MCP's discovery mechanism adds no value.

## The Pattern to Remember

**Pattern**: An MCP server has three layers — transport (stdio/SSE), protocol (JSON-RPC lifecycle), and business logic (tool/resource/prompt handlers). The MCP SDK handles the first two; you implement the third.

When to use:
- Tools that multiple agents or teams will use
- Tools that need service boundaries (auth, rate limiting, monitoring)
- Data sources that should be read-only and mediated

When NOT to use:
- Single-use, single-agent tools
- Prototypes where the tool design is still changing
- Sub-millisecond latency requirements

**Mental model**: The MCP SDK handles plumbing; you write business logic. The server is a thin wrapper around your tool implementations.

## Common Mistakes

### Mistake 1: SDK Version Mismatch

**Why it happens**: You install an older version of the MCP SDK and write code against the newer spec.

**Symptoms**: Methods like `list_tools()` or `call_tool()` do not exist, have different signatures, or return different types. The server fails to start.

**Fix**: Pin your MCP SDK version in requirements.txt and verify compatibility with the MCP spec version you target. The SDK version should match the `protocolVersion` you declare in initialization.

### Mistake 2: Missing Capability Declaration

**Why it happens**: You add tool handlers to the server but forget to declare the `tools` capability in the initialization options.

**Symptoms**: The client initializes successfully but `tools/list` returns an empty list. The server never advertises its tools.

**Fix**: Ensure your `InitializationOptions` includes the capabilities your server actually provides. The SDK's `server.get_capabilities()` method reads the registered handlers and builds the capability object automatically.

### Mistake 3: Synchronous Blocking in Async Handlers

**Why it happens**: You use `time.sleep()`, `requests.get()`, or other synchronous blocking calls inside an async handler.

**Symptoms**: The server becomes unresponsive. Tool calls from other clients queue up while the blocking call is in progress.

**Fix**: Use `asyncio.sleep()`, `httpx.AsyncClient`, and other async equivalents. If you must call a synchronous library, use `asyncio.to_thread()` to run it in a thread pool.

### Mistake 4: Path Traversal in Resource URIs

**Why it happens**: You build a resource handler that accepts a file path from the URI and reads it without validation.

**Symptoms**: A client requests `file://../../etc/passwd` and gains access to files outside the workspace.

**Fix**: Always resolve the requested path to an absolute path and verify it starts with the allowed workspace directory. Use `Path.resolve()` and `str.startswith()` for the check.

### Mistake 5: Not Handling the `isError` Flag

**Why it happens**: When a tool fails, you return an error message as text content with no indication that it is an error.

**Symptoms**: The client treats the error message as a successful tool result. The model may try to act on the error text rather than recognizing it as a failure.

**Fix**: Set `isError: true` on tool error responses. The MCP SDK's `TextContent` and other content types support an `isError` boolean field that tells the client the call failed.
