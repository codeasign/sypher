# Build It: Designing an Agent Evaluation Framework

⏱️ 25 minutes

## Breaking It Down

### 1. Defining Your Evaluation Tasks

Start by writing 10-20 evaluation tasks that represent your agent's core use cases. Each task should have:

- A natural language prompt (what the user says)
- The expected outcome (what success looks like)
- The expected tool call sequence (optional, for step-level eval)
- Constraints (e.g., "must not call the delete tool")

Here is an example evaluation dataset for a customer support agent:

```python
# eval_tasks.py
EVAL_TASKS = [
    {
        "id": "refund-status",
        "prompt": "Check the refund status for order ORD-12345",
        "expected_outcome": "Refund is in processing, estimated completion in 3-5 business days",
        "expected_tools": ["read_query", "read_query"],
        "must_not_call": ["refund_order", "send_email"],
        "success_criteria": "Returns the correct status and estimated time",
    },
    {
        "id": "cancel-order-within-window",
        "prompt": "Cancel order ORD-67890 — it was placed 30 minutes ago",
        "expected_outcome": "Order ORD-67890 cancelled successfully",
        "expected_tools": ["read_query", "cancel_order", "send_email"],
        "must_not_call": [],
        "success_criteria": "Order cancelled, customer notified",
    },
    {
        "id": "cancel-order-outside-window",
        "prompt": "Cancel order ORD-11111 — it was placed 3 days ago",
        "expected_outcome": "Cannot cancel: order is outside the 24-hour cancellation window",
```

### 2. Building an Evaluation Runner

Create a script that runs the agent against each task and records the results:

```python
# eval_runner.py
import json
import time

def run_evaluation(tasks, agent_function):
    """Run an agent against a set of evaluation tasks."""
    results = []
    for task in tasks:
        start_time = time.time()
        try:
            output, tool_calls = agent_function(task["prompt"])
            elapsed = time.time() - start_time
            results.append({
                "task_id": task["id"],
                "prompt": task["prompt"],
                "success": evaluate_outcome(output, task),
                "tool_calls": tool_calls,
                "latency_seconds": round(elapsed, 2),
                "tool_call_count": len(tool_calls),
                "errors": [],
            })
        except Exception as e:
            elapsed = time.time() - start_time
            results.append({
                "task_id": task["id"],
                "prompt": task["prompt"],
                "success": False,
                "tool_calls": [],
                "latency_seconds": round(elapsed, 2),
                "tool_call_count": 0,
                "errors": [str(e)],
            })
    return results

def evaluate_outcome(output: str, task: dict) -> bool:
    """Check if the agent's output meets the success criteria."""
    # In production, use an LLM-as-judge or keyword matching
    keywords = task["success_criteria"].lower().split()
    output_lower = output.lower()
    return any(kw in output_lower for kw in keywords)
```

### 3. Calculating Key Metrics

From the evaluation results, compute aggregate metrics:

| Metric | Formula | What It Measures |
|---|---|---|
| **Task Success Rate** | `successful / total` | Overall goal achievement |
| **Tool Call Efficiency** | `avg(tool_calls_per_task)` | How directly the agent solves tasks |
| **Average Latency** | `avg(latency_seconds)` | User-facing response time |
| **Error Rate** | `tasks_with_errors / total` | How often the agent crashes |
| **Safety Violation Rate** | `must_not_call_violations / total` | How often the agent breaks constraints |

```python
def compute_metrics(results):
    total = len(results)
    successful = sum(1 for r in results if r["success"])
    total_tool_calls = sum(r["tool_call_count"] for r in results)
    total_latency = sum(r["latency_seconds"] for r in results)
    error_count = sum(1 for r in results if r["errors"])

    return {
        "task_success_rate": round(successful / total * 100, 1),
        "avg_tool_calls": round(total_tool_calls / total, 1),
        "avg_latency_seconds": round(total_latency / total, 1),
        "error_rate": round(error_count / total * 100, 1),
        "total_tasks": total,
        "successful_tasks": successful,
    }
```

### 4. Using an LLM-as-Judge for Sophisticated Evaluation

Keyword matching is brittle. A more robust approach is using an LLM-as-Judge — a separate LLM that evaluates the agent's output:

```python
import openai

def llm_judge(output: str, task: dict) -> dict:
    """Use an LLM to evaluate whether the agent's output satisfies the task."""
    prompt = f"""
You are evaluating an AI agent's response to a user request.

USER REQUEST: {task["prompt"]}

AGENT RESPONSE: {output}

SUCCESS CRITERIA: {task["success_criteria"]}

Evaluate whether the agent's response meets the success criteria.
Respond with a JSON object:
- "pass": true or false
- "reasoning": one-sentence explanation
- "score": 1-5 (5 = perfect)
"""
    response = openai.chat.completions.create(
        model="claude-sonnet-4-6",
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"},
    )
    return json.loads(response.choices[0].message.content)
```

LLM-as-judge evaluation correlates well with human evaluation (typically 80-90% agreement) and catches subtle failures that keyword matching misses, such as fabricated data or plausible-sounding but incorrect reasoning.

## How It Works Internally

The evaluation process runs in four phases:

1. **Task execution** — the agent runs against each task in isolation. Each run is a fresh session with a cleared conversation history, preventing cross-task contamination.
2. **Logging** — every tool call, model response, latency measurement, and error is recorded in a structured log. This includes the full input and output of each tool call.
3. **Scoring** — each task is scored on three axes: outcome success (did the end result match?), safety compliance (did the agent avoid prohibited actions?), and efficiency (was the tool call count within expected range?).
4. **Aggregation** — individual scores are rolled up into per-task and per-category metrics. A regression report highlights tasks that regressed compared to the previous evaluation run.

## Build Something Real

Set up a nightly evaluation pipeline using GitHub Actions. Define 20 tasks for your agent, run them every night, and have the results posted to a Slack channel. When a task regresses (passes last night but fails tonight), the pipeline flags it for investigation:

```yaml
# .github/workflows/agent-eval.yml
name: Agent Evaluation
on:
  schedule:
    - cron: '0 6 * * *'  # Run daily at 6 AM UTC
  workflow_dispatch:       # Allow manual trigger

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run agent evaluation
        run: |
          python eval_runner.py --output eval-results.json
      - name: Post results to Slack
        run: |
          python post_to_slack.py eval-results.json
      - name: Check for regressions
        run: |
          python check_regressions.py eval-results.json previous-results.json
```
