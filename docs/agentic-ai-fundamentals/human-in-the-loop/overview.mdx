# Human in the Loop

*Reading time: 5 minutes*

**Difficulty:** Intermediate

## Why This Matters

The most dangerous agent is one that can take irreversible actions without oversight. Sending an email, deleting a record, approving a refund, deploying code — these actions need human approval before execution. Human-in-the-loop (HITL) design is the practice of inserting approval gates at critical decision points.

Production agent systems at companies like Klarna and Stripe use HITL extensively. Klarna's customer service agent can resolve most issues autonomously, but refunds over a certain threshold require human approval. The agent gathers all the context needed for the decision and presents it in a structured format, so the human can approve or reject with one click.

## The Problem

Your support agent has access to a `process_refund` tool. It handles most refund requests correctly. But one day, a user convinces the agent to process a $5,000 refund for a $50 item through social engineering. The agent approves because the user's argument sounds plausible to the model.

The agent cannot reliably distinguish legitimate large refunds from fraudulent ones. This is not a flaw in the model — it is a design problem. The system should have an approval gate on refunds over a threshold. The agent gathers the information, presents it to a human, and waits for approval before executing.

## The Concept

Human involvement in agent systems falls on a spectrum:

**Human-in-the-loop (HITL)** : The agent pauses at key decision points and waits for human input. The human must approve or reject each gated action. This is the safest mode.

**Human-on-the-loop (HOTL)** : The agent acts autonomously but streams its actions to a human monitor. The human can intervene at any time but does not need to approve every action. This is a balance of speed and safety.

**Fully autonomous**: The agent acts independently. No human approval is required. This is the fastest mode but has the highest risk.

Approval gates are inserted in the agent loop. When the model requests a gated tool, the orchestrator checks whether the action needs approval. If yes, it pauses, presents the decision context to the human, and waits.
