# Human in the Loop — Avoid Mistakes

*Reading time: 5 minutes*

## When NOT to Use This

Human-in-the-loop is not always the right choice:

- **Fully autonomous tasks**: If the task has no risk of harm and errors are cheap, HITL adds unnecessary friction.
- **Speed-critical systems**: Waiting for human approval adds seconds, minutes, or hours to response time.
- **24/7 operations**: If no human is available at 3 AM, the agent stops dead waiting for approval.
- **High-volume operations**: Every action needing human approval does not scale. Design for autonomy first.

## The Pattern to Remember

**Pattern**: Approval gates block risky tool calls. Escalation paths route uncertain tasks to humans. The agent gathers context so the human can decide quickly.

When to use:
- Irreversible actions (send, delete, approve payment)
- High-value decisions (large refunds, account changes)
- Tasks requiring human judgment

When NOT to use:
- Reversible actions
- Low-risk operations
- 24/7 systems with no human coverage

**Mental model**: HITL is like a two-person rule for sensitive operations. The agent prepares the paperwork; the human signs off.

## Common Mistakes

### Mistake 1: Approval Fatigue

**Why it happens**: Every minor action requires human approval. The human is asked to approve 50+ actions per hour.

**Symptoms**: The human starts approving everything without reading. The approval gate becomes a rubber stamp, defeating its purpose.

**Fix**: Only gate high-risk actions. Auto-approve low-risk actions. Use risk tiers: green (auto), yellow (notify), red (approve required).

### Mistake 2: No Timeout on Human Approval

**Why it happens**: The agent pauses indefinitely waiting for human input.

**Symptoms**: The agent is stuck for hours. The user who requested the task waits forever. The task context expires.

**Fix**: Set a timeout on each approval request. If the human does not respond within the timeout, use a fallback: reject, auto-approve with warning, or escalate to a different human.

### Mistake 3: Insufficient Context for the Human

**Why it happens**: The agent presents a tool call without explaining why it wants to make it.

**Symptoms**: The human sees "approve: send_email(to=user@example.com)" with no context about why the email is being sent or what it contains.

**Fix**: Present the decision context alongside the tool call: the original user request, what has been done so far, and why this action is needed.

### Mistake 4: Gate Configuration in Code

**Why it happens**: Approval thresholds are hardcoded (e.g., "refunds over $100 need approval").

**Symptoms**: Changing a threshold requires a code deploy. Different customers or regions need different thresholds.

**Fix**: Make gates configurable via external configuration or a database. Allow per-customer, per-region overrides.

### Mistake 5: No Audit Trail for Human Decisions

**Why it happens**: Human approvals and rejections are not logged.

**Symptoms**: A week later, no one knows who approved a refund, when, or for what reason. Auditing is impossible.

**Fix**: Log every approval decision: who approved, what was approved, the context presented, the time, and the outcome. Store this in a tamper-evident audit log.
