# Agent Tools and Tool Calling — Avoid Mistakes

*Reading time: 6 minutes*

## When NOT to Use This

Tool calling is the right mechanism for most agent actions, but it is not universal:

- **Purely internal calculations**: If a step does not interact with the outside world, a direct function call in your code is simpler.
- **Read-only data access that is always needed**: If every request needs the same data, inject it in the system prompt rather than making a tool call.
- **Operations that must be atomic and reliable**: Tool calls can fail, be called multiple times, or return unexpected results. For critical operations, consider a workflow instead.

## The Pattern to Remember

**Pattern**: Each tool gets a name, a description, a JSON schema, and an implementation. The model requests tool calls; your code executes them and returns results.

When to use:
- Any action the agent takes in the external world
- Data lookups the model cannot do from its training data
- Operations that have side effects (write, send, create)

When NOT to use:
- Simple calculations your code can do directly
- Static information that never changes
- Operations requiring reliability guarantees

**Mental model**: Tools are the agent's hands. Without them, the agent can see and talk but cannot touch anything.

## Common Mistakes

### Mistake 1: Vague Descriptions That Confuse Similar Tools

**Why it happens**: Two tools do related things, and their descriptions use similar language.

**Symptoms**: The model calls `search_products` when it should have called `search_inventory`. The results are wrong but structurally valid.

**Fix**: Make tool descriptions distinct. Use contrasting language: "Search the customer-facing product catalog for items available to purchase" vs. "Search the internal inventory system for warehouse stock levels."

### Mistake 2: Not Validating Arguments Before Execution

**Why it happens**: You trust the model to provide valid arguments matching the schema.

**Symptoms**: The model passes an out-of-range value, a string where a number is expected, or SQL injection-like input.

**Fix**: Validate every argument against the schema before passing it to the implementation. Use schema validators like Pydantic or JSON Schema validators.

### Mistake 3: Over-Parameterized Tools

**Why it happens**: You create a tool with 10+ optional parameters "just in case."

**Symptoms**: The model rarely fills in parameters correctly. It misses required combinations. Error rates are high.

**Fix**: Keep tools focused. If a tool needs more than 5 parameters, split it into multiple tools or use a simpler input format. Prefer `enum` over free-text string parameters.

### Mistake 4: Silent Tool Failures

**Why it happens**: The tool implementation catches exceptions but returns generic "error occurred" messages.

**Symptoms**: The model receives an error it cannot act on because the message does not explain what went wrong.

**Fix**: Return descriptive error messages: what was attempted, what went wrong, and what the model could try instead. A good error message is "File 'report.txt' not found. Available files are: notes.txt, tasks.txt."

### Mistake 5: Not Handling Idempotency

**Why it happens**: The model may request the same tool call multiple times (due to retries or confusion).

**Symptoms**: The same database insert runs twice. The same email is sent twice. The same file is created twice.

**Fix**: Tools that create side effects should be idempotent. Use unique request IDs, check for duplicates before executing, or use "create if not exists" semantics.
