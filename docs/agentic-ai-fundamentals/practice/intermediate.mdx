## Exercise 1: Agent with Memory and Tool Orchestration

**Problem:** Build an agent that maintains conversation memory across turns and uses at least two different tools to answer a user's multi-step request. The agent should:

1. Maintain a short-term memory of the conversation (list of messages, updated each turn)
2. Use a `search_documents` tool (mock) and a `summarize_text` tool (mock)
3. Decide which tool to call based on the user's request
4. Handle the case where the first tool call produces no results — try an alternative approach

The user's request is: `"Find the Q4 2025 sales report and give me a one-paragraph summary of it."`

**Tools to implement:**
```python

    """Mock: search indexed documents by keyword."""
    pass

def summarize_text(text: str, max_words: int = 100) -> str:
    """Mock: summarise a block of text."""
    pass
```

**Expected behavior:**
1. Agent receives the user request, stores it in memory
2. Agent calls `search_documents("Q4 2025 sales report")`
3. If no results, agent tries `search_documents("Q4 2025 sales")` or `search_documents("2025 sales report")`
4. Agent passes the found document text to `summarize_text`
5. Agent returns the summary to the user, with memory updated

## Exercise 2: MCP Server with Authentication and Observability

**Problem:** Build a minimal MCP server that provides a `read_customer` tool and a `create_order` tool, each of which requires authentication. The server must also log every request with timing and expose a `_health` endpoint.

**Specifications:**

**Server setup:**
- Use the FastMCP Python SDK.
- Define two tools: `read_customer(customer_id: str)` and `create_order(customer_id: str, product: str, quantity: int)`.
- `read_customer` requires a valid API key in the request metadata.
- `create_order` requires a valid API key AND a specific `admin` scope.

**Authentication:**
- Valid API keys: `key_user` (scope: read), `key_admin` (scope: read, write).
- Return an error tool response if authentication fails: `{"error": "unauthorized", "detail": "..."}`.
- Log every authentication attempt (success or failure) with a timestamp.

**Observability:**
- Each tool call should log: `tool_name`, `arguments`, `timestamp`, `duration_ms`, `auth_status`.
- Provide a `_health` tool (not exposed to users as a regular tool, or available to admins only) that returns: `{"status": "ok", "uptime": "...", "tool_call_count": N, "failed_auth_count": M}`.

**Expected behavior:**
```python
# With key_user (read-only):
read_customer("C12345")  # Returns customer data
create_order("C12345", "Widget", 5)  # Returns auth error

# With key_admin:
read_customer("C12345")  # Returns customer data
create_order("C12345", "Widget", 5)  # Returns order confirmation
```

**Hints:**
- Use a dictionary to store valid keys and scopes.
- The auth check happens at the start of each tool function.
- Use `time.perf_counter()` for duration measurement.
- Store call count and failed auth count in module-level variables.

**Constraints:** Must use FastMCP syntax. No external API dependencies (mock customer data from a dict). Must log every call. Must support scope-based access control.
