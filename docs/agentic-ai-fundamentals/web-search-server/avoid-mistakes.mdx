# Avoid Mistakes: Web Search MCP Server Pitfalls

⏱️ 10 minutes

## When NOT to Use This

A web search MCP server fetches live data from the internet. Do not use it when:

- **You need deterministic, verifiable answers** — search results change over time; the same query tomorrow may return different results
- **The agent runs offline** — the server cannot cache results indefinitely; design your workflows to tolerate transient failures
- **You query internal knowledge bases** — use a dedicated documentation or database MCP server instead; web search exposes your query patterns to an external provider
- **You need real-time streaming data** — MCP is request-response; for live stock prices or sports scores, use a push-based system like WebSockets

## The Pattern to Remember

| Dimension | Guideline |
|---|---|
| **Pattern** | Search API calls wrapped as MCP tools with result filtering |
| **When to use** | Agent needs current information, live docs, or news |
| **When NOT to use** | Deterministic answers, offline environments, internal knowledge |
| **Mental model** | A programmable search bar with configurable scope and source filtering |

## Common Mistakes

### Mistake 1: Ignoring Result Freshness

**Mistake:** Searching with default parameters and getting stale results.

**Why it happens:** The Brave Search API returns the "best" results by default, which may be old but highly ranked pages. The agent does not know to request recent results.

**Symptoms:** The agent cites outdated information — last year's API docs, old pricing pages, or superseded blog posts.

**Fix:** Specify the `freshness` parameter explicitly when asking about time-sensitive topics. The agent should be prompted to include time qualifiers in its search query, or use the news-specific tool.

### Mistake 2: Wasting API Quota on Repeated Queries

**Mistake:** The agent searches for the same term multiple times in one session.

**Why it happens:** The agent does not remember that it already searched for "pandas DataFrame merge" two turns ago.

**Symptoms:** API quota is exhausted quickly for simple repetitive queries.

**Fix:** Most MCP clients cache tool results, so repeated calls with identical parameters return cached results without hitting the API. Check that your client has caching enabled.

### Mistake 3: Trusting Search Snippets as Complete Answers

**Mistake:** The agent uses only the search result snippet instead of fetching the full page.

**Why it happens:** The snippet looks sufficient. But snippets are truncated, often out of context, and sometimes misleading.

**Symptoms:** The agent gives an answer that sounds authoritative but contradicts the actual article content.

**Fix:** Prompt the agent to use `fetch_page` on the top result when it needs detailed information. Many agents do this automatically for results with high relevance scores.

### Mistake 4: Fetching Pages That Are Unnecessarily Large

**Mistake:** Using `fetch_page` on a multi-megabyte documentation page.

**Why it happens:** The agent does not know the page size in advance.

**Symptoms:** Slow response times, high transfer costs, and context windows filled with boilerplate navigation HTML.

**Fix:** Set a content size limit in `fetch_page` (e.g., 10 KB minimum, 100 KB maximum for useful content). Many servers also strip navigation, headers, and footers before returning the text.

### Mistake 5: Exposing the API Key in Error Messages

**Mistake:** When the Brave Search API returns a 401 error, the server echoes the full request, including the `X-Subscription-Token` header.

**Why it happens:** Naive error handling returns `str(resp)` or `await resp.text()` directly to the agent.

**Symptoms:** The API key is printed in the agent's response, visible to the user.

**Fix:** Strip sensitive headers from error responses before returning them:

```python
@server.tool("web_search")
async def web_search(query: str) -> TextContent:
    try:
        resp = await client.get(...)
        resp.raise_for_status()
    except httpx.HTTPStatusError as e:
        return TextContent(
            type="text",
            text=f"Search failed with status {e.response.status_code}"
        )
```
