# Review: Scaling Agent Systems

*Reading time: 5 minutes*

## Key Takeaways

- A single agent instance handles one conversation at a time. To serve multiple concurrent users, you must deploy multiple stateless instances behind a load balancer with a shared session store.
- Stateless scaling requires externalizing session state — agent memory, conversation history, and tool results — into a shared store like Redis. Every instance loads and saves state from the same source of truth.
- LLM rate limits are a shared resource across all instances. Use a distributed token bucket algorithm backed by Redis to prevent any single instance from exhausting the API quota.
- Queue-based consumption with a message broker (Celery, Redis Streams, SQS) decouples request acceptance from processing, smoothing traffic spikes and preventing the LLM API from being overwhelmed.
- Auto-scaling based on queue depth matches resource consumption to demand, but requires monitoring end-to-end latency and planning for cold starts to avoid degrading the user experience during traffic spikes.

## Recap in 60 Seconds

1. Make agents stateless by moving session state out of the process and into a shared Redis or database store — any instance can then handle any session.
2. Implement distributed rate limiting and queue-based consumption so multiple instances share a single API key fairly and do not overwhelm the LLM provider under load.
3. Monitor p95 latency end-to-end and auto-scale based on leading indicators like queue depth and request rate, not just as a reaction to errors.
