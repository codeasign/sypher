# Requirements

## Functional requirements

1. **Natural-language question input.** The agent accepts a free-form research question (typed or read from a file) and returns a structured report. Example input: *"What are the top 5 open-source AI agent frameworks on GitHub as of June 2026, how do they compare in stars, license, and last commit date, and what are the latest news articles about each one?"*

2. **Multi-step planning.** Before executing any tool, the agent generates a structured plan (a numbered or JSON list of steps). The plan is stored as a mutable document outside the message list and updated as steps complete, fail, or are revised.

3. **Dynamic replanning.** If a step fails (tool error, empty results, timeout), the agent revises its plan from the current state rather than starting over or crashing. The modified plan is logged and clearly visible in the trace.

4. **MCP server integration.** The agent connects to at least three MCP servers:
   - **Web Search** — search the live web and fetch page content for current information
   - **GitHub** — search repositories, read issues, examine pull requests, look up repository metadata
   - **Database** — store conversation history, agent logs, cached results, and report metadata in a queryable data store
   You may add a custom MCP server for any additional capability (PDF parsing, news aggregation, citation formatting).

5. **Conversation memory.** The agent remembers previous questions and findings across turns within the same session. Short-term memory (sliding window of recent turns) and long-term memory (external store with retrieval) are both implemented.

6. **Retry with exponential backoff.** When a tool call fails with a retriable error (network timeout, 429 rate limit, 5xx server error), the agent retries with exponential backoff (at least 3 attempts) before classifying the step as failed.

7. **Reflection and self-correction.** Before finalizing its answer, the agent reflects on its own output. It checks for missing citations, contradictory claims, incomplete sections, and low-confidence assertions. If issues are found, it revises.

8. **Per-claim confidence scoring.** Every factual claim in the output is accompanied by a numeric confidence score (0.0–1.0) and a brief justification. Scores below a configurable threshold (default 0.6) are flagged for human review.

9. **Citations.** Every claim is sourced. Citations include the URL, title, and date retrieved for web sources, or the database query and table name for data sources. The report footer contains a full citation list.

10. **Professional report export.** The agent produces a final report in at least two formats (Markdown and PDF, or Markdown and HTML). The report includes: executive summary, methodology, findings with per-claim scores and citations, confidence summary, and limitations.

11. **Structured logging and tracing.** Every agent action — plan generation, tool call, tool result, retry, reflection outcome, final output — is logged as a structured JSON record. Every session has a unique trace ID that connects all logs, tool calls, and LLM requests into a single view.

12. **Self-evaluation.** After generating the report, the agent evaluates its own output against at least three metrics: completeness (did it cover all plan steps), accuracy (are claims internally consistent and sourced), and efficiency (tool call count vs. expected). The evaluation is included in the report appendix.

## Non-functional requirements

- **Performance.** A single research question should complete within 180 seconds. The export step may add up to 60 seconds.
- **Error handling.** Every tool call has a timeout (default 30 seconds). The agent never crashes on a tool error — it retries, skips, or escalates.
- **Configuration.** All environment-specific values (API keys, MCP server commands, model selection, confidence threshold, log level) are managed through a layered configuration system with defaults, environment files, and runtime overrides.
- **Safety.** The agent applies defense-in-depth against prompt injection: input sanitization, output validation before export, tool-level permission checks on write operations, and a human approval gate before the final report is exported to disk.
- **Observability.** Every session produces a trace file (`traces/<trace-id>.json`) viewable in any JSON viewer or trace-compatible tool.
- **Cost tracking.** Every session records total token usage, cost estimate, and latency breakdown.

## Constraints

- You may only use Python standard library, the Anthropic SDK, the MCP Python SDK, and libraries already introduced in this course (SQLite for the database, httpx for HTTP, pytest for testing, etc.). No framework that hides the agent loop (LangChain, AutoGen, CrewAI, etc.).
- The agent loop must be your own implementation — not a framework's built-in agent executor.
- MCP servers must be configured via the MCP protocol (stdio transport), not hardcoded integrations.
- The project must include automated tests that verify at least the core agent loop, tool dispatch, and report generation.
- All code must be typed (Python type hints) and include docstrings.

## What's explicitly out of scope

- A web UI or chat interface — the agent runs from the command line or as a Python script
- Real-time streaming of token-by-token output to a browser
- OAuth login flows for end users
- Multi-user session management or authentication
- Deployment to a cloud platform (Dockerizing is optional)
- Integration with more than 5 MCP servers (3 is required, 5 is the recommended maximum)

## Suggested tech stack

- **Python 3.11+** with `asyncio` for async agent execution
- **Anthropic SDK** for LLM access (Claude Sonnet for reasoning, Claude Haiku for execution if you separate them)
- **MCP Python SDK** (`mcp`) for building and connecting to MCP servers
- **SQLite** (via `sqlite3` or `aiosqlite`) for the database server and long-term memory store
- **httpx** for any additional HTTP calls
- **fpdf2** or **weasyprint** for PDF report export (optional — Markdown + HTML is sufficient)
- **json** standard library for structured logging
- **pytest** for automated tests
- **Pydantic** (optional, recommended) for configuration models and structured output schemas
