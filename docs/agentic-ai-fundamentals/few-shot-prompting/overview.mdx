# Few-Shot Prompting

*Reading time: 5 minutes*

**Difficulty:** Beginner

## Why This Matters

Few-shot prompting — providing examples in the prompt — is the most reliable way to show the model exactly what you want. Describing the desired output format in words is useful, but showing the model three examples of the exact format, tone, and structure you expect is far more effective. The model can infer the pattern from the examples even when your verbal description is incomplete.

When you have a specific output format (a particular JSON schema, a specific writing style, a classification system with subtle rules), few-shot prompting consistently outperforms instruction-only prompts. The examples ground the model's output in concrete patterns rather than abstract rules.

## The Problem

You need the model to classify customer feedback into nuanced categories: BUG, PERFORMANCE_ISSUE, UX_CONCERN, FEATURE_REQUEST, or OTHER. The categories overlap — a slow page load could be PERFORMANCE_ISSUE or BUG depending on the root cause. You write detailed category definitions in the system prompt, but the model still misclassifies: it labels everything with a "slow" mention as PERFORMANCE_ISSUE, even when the real issue is a missing feature that happens to load slowly.

Your definitions are clear, but the model's training data does not share your category boundaries. The definitions alone are not enough to teach the model the subtle distinctions you need.

## The Concept

Few-shot prompting is the technique of including input-output examples in your prompt to teach the model the desired behavior through pattern matching. It relies on the model's ability to perform in-context learning: given a sequence of examples, the model infers the underlying pattern and applies it to new inputs.

In-context learning is an emergent property of large language models. The model does not update its weights when you provide examples — it uses the examples as context to shape its output distribution. The examples prime the model toward the statistical patterns present in the examples.

Key variables in few-shot prompting:

**Number of examples**: 1-5 examples is typical. More examples improve reliability but increase token costs. Diminishing returns set in after 5-10 examples.

**Example selection**: Examples should cover the range of inputs the model will encounter. Include edge cases and boundary examples, not just the most common cases.

**Example ordering**: The order of examples matters. Models tend to put more weight on examples at the end of the sequence (recency bias). Put your most representative or challenging example last.

**Example quality**: Every example must be correct. A single wrong example can teach the model the wrong pattern.
