# Milestones

Suggested checkpoints. These are not mandatory steps — they exist so you have a way to know if you are on track. Skip ahead if you prefer.

---

## Milestone 1 — Project Scaffold and Package Structure

**Goal:** Set up the project directory following the standard structure taught in this course. The package is installable, the config system works, and a health endpoint responds.

**You will know this works when:** `pip install -e ".[dev]"` succeeds, `python -c "from research_assistant.config import load_config; print(load_config())"` prints your default config, and a `uvicorn` dev server responds to `GET /health` with `{"status": "ok"}`.

**Concrete steps:**

1. Create the directory skeleton: `src/research_assistant/`, `tests/`, `configs/`, `data/`, `scripts/`, `notebooks/`.
2. Write `pyproject.toml` with dependencies (fastapi, uvicorn, anthropic, openai, pydantic, chromadb, httpx, diskcache, pyyaml, python-dotenv, tiktoken, numpy) and dev extras (pytest, pytest-cov, ruff, mypy).
3. Create `src/research_assistant/config.py` — load YAML config with sensible defaults for chunk size (500), overlap (100), top-K (3), cache TTL (3600), model names, and auth token.
4. Create `configs/default.yaml` with those defaults.
5. Create `src/research_assistant/main.py` with a minimal FastAPI app and a `GET /health` endpoint.
6. Create `scripts/run.py` as the entry point that starts Uvicorn.
7. Verify: `pip install -e ".[dev]"` → `uvicorn research_assistant.main:app --reload` → `curl http://localhost:8000/health`

**Relevant lessons:** [Project Structure](../project-structure/overview), [Packaging](../packaging/overview), [Modules](../modules/overview), [JSON](../json/overview)

## Milestone 2 — Document Ingestion Pipeline

**Goal:** Build the ingestion pipeline that chunks documents, generates embeddings, and stores them in Chroma.

**You will know this works when:** You send a document to `POST /ingest`, the pipeline splits it into overlapping chunks, embeds each chunk via the OpenAI API, stores them in Chroma, and a test query retrieves the expected chunk.

**Concrete steps:**

1. Create `src/research_assistant/ingestion/chunker.py` — the `chunk_text()` function from the RAG lesson (fixed-size with overlap, using `tiktoken`).
2. Create `src/research_assistant/ingestion/pipeline.py` — an `IngestionPipeline` class that chunks, embeds via OpenAI, and stores in Chroma's `PersistentClient`. Use the pattern from the Embeddings lesson's `PersistentVectorSearch`.
3. Create Pydantic models for ingest requests in `src/research_assistant/models.py`: `IngestRequest(doc_id, text, source, metadata?)`.
4. Add `POST /ingest` to FastAPI that accepts the request, calls the pipeline, and returns the chunk count.
5. Create a Chroma collection with cosine similarity (`hnsw:space = "cosine"`).
6. Write a quick sanity test: ingest a known document, then query the collection directly and confirm the chunk is retrievable.

**Relevant lessons:** [RAG](../rag/overview) (chunking, ingestion), [Embeddings](../embeddings/overview) (batch embedding, Chroma), [Structured Output](../structured-output/overview) (Pydantic models), [Dataclasses](../dataclasses/overview), [Type Hints](../type-hints/overview)

## Milestone 3 — Retrieval and Answer Generation (RAG)

**Goal:** When a question arrives, the service retrieves relevant chunks and generates a grounded answer with structured citations.

**You will know this works when:** `POST /ask` with a question about an ingested document returns `{"answer": "...", "citations": [{"source_doc_id": "...", "relevance_score": 0.95}], "latency_ms": 3400}`. The answer is factually correct and cites the source document.

**Concrete steps:**

1. Create `src/research_assistant/rag/retriever.py` — a `Retriever` class that embeds the query and fetches top-K chunks from Chroma (following the RAG lesson's `Retriever`).
2. Define a Pydantic `Citation(BaseModel)` with `source_doc_id: str` and `relevance_score: float`.
3. Define a Pydantic `AnswerResponse(BaseModel)` with `answer: str`, `citations: list[Citation]`, `latency_ms: int`.
4. Create `src/research_assistant/rag/generator.py` — a prompt template that takes the question and retrieved context, and uses the Anthropic SDK with a tool-defined structured output schema to produce a JSON response with `answer` and `citations` fields. Use the structured output patterns from the Structured Output lesson.
5. Create `src/research_assistant/rag/pipeline.py` — a `RAGPipeline` class that orchestrates: embed query → retrieve chunks → build prompt → call LLM → parse structured response.
6. Add error handling: if retrieval returns zero chunks, the pipeline returns a message stating no relevant documents were found (using the pattern from the RAG lesson's `RAGPipeline.answer()`).
7. Wire `POST /ask` in FastAPI: validate the input question, call the RAG pipeline, return the structured response with latency.

**Relevant lessons:** [RAG](../rag/overview), [Structured Output](../structured-output/overview), [LLM API Basics](../llm-api-basics/overview), [Prompt Engineering](../prompt-engineering/overview), [Error Handling](../error-handling/overview)

## Milestone 4 — Authentication and Error Middleware

**Goal:** Secure `/ask` and `/ingest` with bearer token auth. Add global error handling that returns structured JSON for every failure mode.

**You will know this works when:** A request to `POST /ask` without an `Authorization` header returns 401. An invalid token returns 403. A request with a valid token works. An LLM timeout returns 408 with `{"error": "LLM request timed out"}`. A malformed question (empty string) returns 422.

**Concrete steps:**

1. Create `src/research_assistant/auth.py` — a FastAPI dependency that reads the `Authorization` header, extracts the bearer token, and compares it against the configured token from environment/config. Use the API Authentication lesson's bearer token pattern.
2. Apply the dependency to `/ask` and `/ingest` routes. Exclude `/health` and `/docs`.
3. Create `src/research_assistant/exceptions.py` — custom exception classes: `ServiceAuthError`, `LLMTimeoutError`, `RateLimitError`, `ValidationError`.
4. Register FastAPI exception handlers that catch each custom exception and return structured JSON error bodies with appropriate HTTP status codes.
5. Add `except` blocks in the RAG pipeline that catch transient errors (timeout, connection error, rate limit) and raise your custom exceptions. Use the retry-with-backoff pattern from the LLM Client Wrapper in the LLM API Basics lesson.
6. Verify: `curl -X POST http://localhost:8000/ask -H "Content-Type: application/json" -d '{"question":""}'` returns 422 JSON. `curl` without auth returns 401. Valid auth + valid question returns 200.

**Relevant lessons:** [API Authentication](../api-authentication/overview), [Error Handling](../error-handling/overview), [HTTPX](../httpx/overview), [Decorators](../decorators/overview)

## Milestone 5 — Observability and Caching

**Goal:** Add structured logging that traces every request through the pipeline. Add disk-persistent caching so repeated questions return instantly.

**You will know this works when:** A first question takes 3–5 seconds and appears in the logs with INFO level showing latency, token count, and chunk count. The same question repeated returns in under 50ms with a "(cached)" log entry. File logs contain DEBUG-level detail including the raw LLM response. Cache persists across a service restart.

**Concrete steps:**

1. Create `src/research_assistant/logging_config.py` — configure a logger with two handlers: console (INFO, concise format) and file (DEBUG, detailed format with timestamps and source lines). Follow the `PipelineLogger` pattern from the Logging lesson.
2. Wrap the RAG pipeline stages with the `@pipeline_stage` decorator from the AI Pipelines lesson — add timing, retry, and logging per stage.
3. Create `src/research_assistant/cache.py` — a `ResponseCache` class using `diskcache.Cache` with a configurable TTL. Key = SHA256 hash of the question. Value = serialized `AnswerResponse`.
4. Add a cache check at the start of `/ask`: if the question hash is in cache, return the cached response immediately (with a `"X-Cache-Hit": "true"` response header).
5. Store the response in cache after a successful LLM call.
6. Add a `contextmanager.CacheTimer` context manager (from the Context Managers lesson) that measures and logs total request latency.
7. Verify: send the same question twice. First call is slow, second is instant. Restart the service — the third call is still instant (disk cache). Log files show the full pipeline trace.

**Relevant lessons:** [Logging](../logging/overview), [Performance](../performance/overview) (caching), [Decorators](../decorators/overview), [Context Managers](../context-managers/overview), [AI Pipelines](../ai-pipelines/overview) (pipeline_stage decorator), [Async Python](../async-python/overview)

## Milestone 6 — Docker, Tests, and CI/CD

**Goal:** Containerise the service, write comprehensive tests with mocked external APIs, and set up a GitHub Actions pipeline.

**You will know this works when:** `docker build -t research-assistant .` produces an image under 300MB. `docker run` starts the service. `pytest --cov --cov-fail-under=80` passes with all tests green. A GitHub Actions workflow runs lint, test, and build on every push.

**Concrete steps:**

1. Write a multi-stage `Dockerfile` using `python:3.12-slim`:
   - Stage 1 (builder): install dependencies
   - Stage 2 (runtime): copy installed packages, add a non-root user, copy application code, set `HEALTHCHECK`, `EXPOSE 8000`, `CMD`
   - Follow the Docker lesson's production-ready pattern exactly
2. Add `.dockerignore` excluding cache, `.env`, `chroma_db/`, `__pycache__`, etc.
3. Write `docker-compose.yml` with the app service and a Chroma service (or mount the host `chroma_db` directory as a volume).
4. Write tests in `tests/`:
   - `test_auth.py` — test valid token, invalid token, missing token
   - `test_ingestion.py` — test chunking logic, ingestion with mocked embeddings API
   - `test_retriever.py` — test retrieval with mocked Chroma
   - `test_generator.py` — test structured output parsing with mocked Anthropic
   - `test_pipeline.py` — test the full RAG pipeline with mocked dependencies using pytest fixtures and parametrize (follow the test patterns from the Pytest lesson)
   - `test_cache.py` — test cache hit, cache miss, cache expiry
5. Create `.github/workflows/ci.yml`:
   - **test** job: Ruff lint → mypy type check → pytest with coverage (`--cov-fail-under=80`)
   - **build** job (main branch only): Docker build and push to ghcr.io
   - Follow the CI/CD lesson's pattern for the workflow structure
6. Verify end-to-end: `pytest -v` passes with 80%+ coverage, `docker build` succeeds.

**Relevant lessons:** [Pytest](../pytest/overview) (fixtures, parametrize, mocks), [Docker](../docker/overview) (multi-stage build, health check, non-root), [CI/CD](../ci-cd/overview) (GitHub Actions workflow), [Clean Code](../clean-code/overview), [Project Structure](../project-structure/overview)
