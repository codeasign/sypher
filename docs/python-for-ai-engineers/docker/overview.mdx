## Why This Matters

Python AI applications are notoriously fragile across environments. The code works on your laptop with Python 3.12, a specific NumPy version, and a GPU driver. It fails on your teammate's Mac with Python 3.11 and a newer pip. It fails again in production on a Linux server with different system libraries.

Docker packages your application with its entire runtime: Python version, every pip package, system libraries, and configuration files. The same image runs identically on your laptop, your teammate's machine, your CI server, and your production cluster. This is how every production AI application is deployed.

## The Problem

You built an AI pipeline that uses Chroma, Anthropic SDK, Pandas, and NumPy. You send it to a colleague. They install Python 3.13, but your code needs Python 3.12. NumPy 2.0 is incompatible with your code. The system lacks `libgomp` for Chroma's HNSW index. Debugging takes two hours.

This is the "works on my machine" problem. Docker eliminates it by freezing the entire environment into a single artifact: the image. If the image works in one place, it works everywhere.

## The Concept

A Docker image is a layered filesystem snapshot. Each instruction in a Dockerfile creates one layer:

1. **Base image** — the OS and Python runtime (e.g., `python:3.12-slim`)
2. **Dependencies** — system libraries and pip packages
3. **Application code** — your Python modules and scripts
4. **Configuration** — entrypoint, default commands, exposed ports

Each layer is cached. When you change your code, only the layers above it are rebuilt. This makes rebuilds fast.
