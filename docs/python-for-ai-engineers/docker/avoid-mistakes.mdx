## When NOT to Use Docker

- **Local development of library packages.** If you are developing a library (not an application), Docker adds overhead without benefit. Use virtual environments instead.
- **Short-lived scripts.** A one-off data cleaning script does not need containerisation. Docker is valuable when you need reproducibility across environments or deployments.
- **GPU-heavy workloads without proper setup.** Running CUDA-based ML training in Docker requires matching the CUDA version in the base image to the host driver. This is fragile â€” consider Conda environments for local GPU development.

## The Pattern to Remember

- **Pattern:** Multi-stage Dockerfile (builder + runtime), `.dockerignore`, non-root user, environment variables for secrets, Docker Compose for multi-service orchestration.
- **When to use:** Any application that needs to run in more than one environment (your laptop, CI, production).
- **When NOT to use:** Local library development, one-off scripts, or GPU training without careful CUDA version management.
- **Mental model:** A Docker image is a frozen filesystem. Whatever runs inside the image is independent of the host â€” Python version, system libraries, every dependency.

## Common Mistakes

### Mistake 1: Bloated Images

Using `python:3.12` (the full image, ~900MB) instead of `python:3.12-slim` (~120MB) or `python:3.12-alpine` (~50MB) adds hundreds of megabytes of unused build tools.

**Symptoms:** Large images take minutes to push and pull. CI builds are slow. Storage costs are higher.

**Fix:** Use slim or alpine variants. Use multi-stage builds so build tools (compilers, headers) are not in the final image.

### Mistake 2: Building with Latest Tags

`FROM python:latest` pulls whatever `latest` points to today. Tomorrow it could be Python 3.14 with breaking changes.

**Symptoms:** The build works on Monday and fails on Wednesday. No dependency changed.

**Fix:** Pin the minor version: `FROM python:3.12-slim`. Update the version deliberately and test before deploying.

### Mistake 3: Running as Root

The container runs as root by default. An attacker who exploits your application gains root access to the container.

**Symptoms:** Security scan flags the container. Penetration testing report includes "Container runs as root."

**Fix:** Always create and switch to a non-root user:

```dockerfile
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser
```

### Mistake 4: No .dockerignore

Without a `.dockerignore`, the entire project directory â€” including `__pycache__`, `.git`, `.env`, and `node_modules` â€” is sent to the Docker daemon as build context.

**Symptoms:** Building an image takes 30 seconds just to transfer the build context. The `.env` file (with API keys) is accidentally baked into the image.

**Fix:** Always add a `.dockerignore` file with at minimum `__pycache__`, `*.pyc`, `.env`, `.git`, `.gitignore`, `*.md`, `rag_db/`, `chroma_db/`.

### Mistake 5: Architecture Mismatch

Building on an ARM Mac (Apple Silicon) and deploying on an AMD64 Linux server can produce images that use the wrong architecture.

**Symptoms:** The container crashes with "Exec format error" or segfaults on native extension imports.

**Fix:** Use `docker buildx` to build for the target architecture:

```bash
# Build for AMD64 even on ARM Mac
docker buildx build --platform linux/amd64 -t ai-app .

# Or build multi-architecture
docker buildx build --platform linux/amd64,linux/arm64 -t ai-app:latest .
```
