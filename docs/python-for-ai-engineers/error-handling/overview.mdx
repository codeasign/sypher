# Error Handling

⏱ 10 minutes · Difficulty: Intermediate

## What You Will Learn

By the end of this lesson you will be able to:

- Use `try`/`except`/`else`/`finally` to handle exceptions
- Raise exceptions with `raise` and create custom exception classes
- Understand the exception hierarchy and when to catch specific types
- Use `try`/`except` to retry operations and degrade gracefully
- Write exception-safe code that cleans up resources

## Prerequisites

You should already understand:

- [Control flow](../control-flow/overview) — exceptions alter the normal flow
- [Functions](../functions/overview) — exceptions propagate through call stacks
- [Context managers](../context-managers/overview) — `with` is the preferred way to handle resource cleanup

## Why This Matters

In AI engineering, errors are not exceptional — they are expected. API calls time out, models return malformed responses, files are missing, data is corrupt, GPU memory runs out. The difference between a robust system and a fragile one is how it handles these failures.

Good error handling means: retry transient failures, fail fast on permanent ones, log enough context to diagnose problems, and never crash without reporting why. Every function in a production pipeline needs to consider "what if this fails?" — and error handling is how you answer that question.

## The Problem

You call an external LLM API. It can fail in several ways: network timeout, rate limit (429), server error (500), or invalid request (400). Each needs different handling:

- Timeout → retry with backoff.
- Rate limit → wait and retry.
- Server error → retry, but alert if persistent.
- Bad request → do not retry; the caller needs to fix their input.

Without structured error handling, you either write deeply nested checks or ignore failures entirely and crash.

## The Concept

When an error occurs, Python raises an **exception** — an object that represents the error. Exceptions propagate up the call stack until a matching `except` block catches them. If no block catches the exception, the program terminates and prints a traceback.

The `try` statement gives you four blocks: `try` (the risky code), `except` (what to do on failure), `else` (runs only if no exception), and `finally` (always runs, for cleanup).
