# Error Handling — Common Mistakes

⏱ 6 minutes

## When NOT to Use try/except

- **Preflighting:** Do not use try/except to check if a value exists or a key is in a dict. Use `if` checks instead. Exceptions are for exceptional conditions, not control flow.
- **Silent catching:** Catching `Exception` (the base of almost everything) and doing nothing hides bugs. Never write `except: pass` without a very specific reason.
- **Normal control flow:** In Python, `for` loops use `StopIteration` internally, but this is handled for you. Do not catch or raise `StopIteration` in normal code.

## The Pattern to Remember

- **Pattern:** `try` risky code → `except SpecificError` handle → `else` on success → `finally` cleanup.
- **When to use:** Operations that can fail — API calls, file I/O, user input parsing, resource allocation.
- **When NOT to use:** Checking types, checking existence, normal control flow.
- **Mental model:** Exceptions are for situations your code cannot prevent. Validate what you can (inputs, types); handle what you cannot (network, disk, external services).

## Common Mistakes

### Mistake 1: Bare `except:` Catching Everything

```python
try:
    result = api_call()
except:  # Catches SystemExit, KeyboardInterrupt, everything!
    print("Something went wrong")
```

This catches `SystemExit` (making `Ctrl+C` or `sys.exit()` fail) and `KeyboardInterrupt` (making `Ctrl+C` not work). Always catch specific exceptions:

```python
except Exception as e:  # Catches most programming errors
except (ValueError, KeyError) as e:  # Even better — be specific
```

### Mistake 2: Swallowing Exceptions Silently

```python
try:
    process(data)
except Exception:
    pass  # No logging, no re-raise — bug hidden!
```

**Fix:** At minimum log the error. If you cannot handle it, let it propagate.

### Mistake 3: Catching Too Broadly and Missing the Real Bug

```python
try:
    value = int(user_input)
    result = 100 / value
except Exception as e:
    print(f"Error: {e}")
    result = 0
```

If `user_input` is `"abc"`, you get `ValueError` → prints "Error" → returns 0. This is fine. But if `value` is 0, `ZeroDivisionError` is also caught and returns 0 — also fine. But what if someone renamed `value` to `val` and `NameError` fires? That is also caught and masked. Be specific:

```python
except (ValueError, ZeroDivisionError) as e:
```

### Mistake 4: Not Using `else` When You Should

```python
try:
    data = load_file()
except FileNotFoundError:
    data = {}
    logging.warning("File not found")

# Without else, this line runs even if FileNotFoundError fired!
process(data)  # data might be {}
```

**Fix:** Put code that should only run on success in `else`:

```python
try:
    data = load_file()
except FileNotFoundError:
    data = {}
    logging.warning("File not found")
else:
    process(data)  # Only runs if no exception
```

### Mistake 5: Raising the Wrong Exception Type

```python
def get_user(user_id):
    if not isinstance(user_id, int):
        raise Exception("user_id must be an int")  # Too generic!
```

**Fix:** Use specific exception types:

```python
def get_user(user_id):
    if not isinstance(user_id, int):
        raise TypeError(f"user_id must be int, got {type(user_id).__name__}")
```
