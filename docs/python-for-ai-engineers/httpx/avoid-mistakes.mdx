# HTTPX — Common Mistakes

⏱ 5 minutes

## When NOT to Use httpx

- **Existing codebase using requests:** If you are not using async, `requests` has the same sync API and is more widely used. Do not switch just for the sake of it.
- **Web scraping with JavaScript:** httpx does not execute JavaScript — use Playwright or Selenium.
- **Very old Python versions:** httpx requires Python 3.8+. Use `requests` for older versions.
- **Environments without async support:** Some embedded or restricted Python environments may not support asyncio. Use the sync client in those cases.

## The Pattern to Remember

- **Pattern sync:** `with httpx.Client(timeout=10) as client: response = client.get(url)`.
- **Pattern async:** `async with httpx.AsyncClient() as client: response = await client.get(url)`.
- **Concurrent batch:** Create a list of async tasks, use `asyncio.gather()`, limit concurrency with `asyncio.Semaphore`.
- **When to use:** Async HTTP calls, concurrent API requests, streaming responses, HTTP/2, granular timeouts.
- **When NOT to use:** Simple single requests (requests is simpler), JS-heavy web pages, Python < 3.8.

## Common Mistakes

### Mistake 1: Using Async Client Without `await`

```python
import httpx
import asyncio

async def main():
    client = httpx.AsyncClient()
    # Wrong — missing await
    response = client.get("https://httpbin.org/get")
    # This returns a Coroutine, not a Response!

    # Correct
    response = await client.get("https://httpbin.org/get")
    await client.aclose()

asyncio.run(main())
```

**Why it happens:** `AsyncClient` methods are coroutines and must be `await`ed. Forgetting `await` returns an unexecuted coroutine object instead of the response.

**Symptoms:** You get a `coroutine` object instead of a `Response`. The program may not crash but the value is useless. Or you see `RuntimeWarning: coroutine was never awaited`.

**Fix:** Always `await` async methods. Use a linter that catches missing await on coroutines.

### Mistake 2: Not Closing the Async Client

```python
import httpx
import asyncio

async def main():
    client = httpx.AsyncClient()
    await client.get("https://httpbin.org/get")
    # Client is never closed — connections leak!

asyncio.run(main())
```

**Why it happens:** Unlike the sync client (which works as a context manager), the async client left unclosed keeps connections open until the garbage collector runs or the event loop shuts down.

**Symptoms:** Warnings about unclosed connections. Resource exhaustion under load.

**Fix:** Always use `async with`:

```python
async with httpx.AsyncClient() as client:
    response = await client.get("https://httpbin.org/get")
```

### Mistake 3: Mixing Sync and Async Code Incorrectly

```python
import httpx

# Calling an async function from sync code without an event loop
async_client = httpx.AsyncClient()
# This raises: RuntimeError: asyncio.run() cannot be called from a running event loop
```

**Why it happens:** `AsyncClient` requires an event loop. Calling it from synchronous code or from within a running event loop (like a Jupyter notebook) requires careful handling.

**Symptoms:** `RuntimeError: There is no current event loop in thread 'MainThread'` in scripts. `RuntimeError: This event loop is already running` in notebooks.

**Fix:** Use `httpx.Client()` (sync) from sync code. Use `asyncio.run(main())` as the entry point for async code. In Jupyter, the event loop is already running, so `await` works directly but `asyncio.run()` does not.

### Mistake 4: Overloading the Server with Unlimited Concurrency

```python
import httpx
import asyncio

async def main():
    async with httpx.AsyncClient() as client:
        # Sends 1000 requests at once — likely to overwhelm both client and server
        tasks = [client.get("https://api.example.com") for _ in range(1000)]
        results = await asyncio.gather(*tasks)
```

**Why it happens:** `asyncio.gather()` runs all tasks concurrently. Without a concurrency limit, you may exhaust local connection pools, hit OS file descriptor limits, or overwhelm the remote API.

**Symptoms:** Connection errors, `OSError: Too many open files`, 429 or 503 responses from the server.

**Fix:** Limit concurrency with `asyncio.Semaphore`:

```python
sem = asyncio.Semaphore(10)  # Max 10 concurrent requests

async def limited_request(client, url):
    async with sem:
        return await client.get(url)
```

### Mistake 5: Ignoring httpx's Different `raise_for_status` Location

```python
import httpx

with httpx.Client() as client:
    response = client.get("https://httpbin.org/status/404")
    # Both work — httpx has the same API as requests here
    response.raise_for_status()  # Raises httpx.HTTPStatusError
```

Unlike some newer HTTP clients, httpx keeps `raise_for_status()` as a method on the response object — same as requests. The mistake is forgetting to call it at all (same as Mistake 3 in the requests lesson).

**Fix:** Call `response.raise_for_status()` after every request, in both sync and async code.
