# Structured Output — Common Mistakes

⏱ 5 minutes

## When NOT to Use Structured Output

- **Free-form generation.** If you want a creative story, a poem, or open-ended dialogue, structured output constrains the model too much. Let the model generate naturally and optionally parse the result.
- **Simple classifications.** If you need a single yes/no answer, structured output adds schema overhead. A prompt constraint with temperature 0.0 is simpler and cheaper.
- **Streaming UI.** Structured output is not compatible with streaming in most providers. If you need real-time token-by-token display, stream free text and validate after the fact.

## The Pattern to Remember

- **Pattern:** Define a Pydantic model per output type, pass its JSON schema to the API, validate the response, and handle `ValidationError` with a fallback.
- **When to use:** Any time the LLM output is consumed by another system (database, API, frontend form) that expects a specific shape.
- **When NOT to use:** Free-form generation, simple binary classification, or streaming scenarios.
- **Mental model:** Structured output is a contract between your code and the LLM. The schema is the contract. If either side violates it, the pipeline fails early instead of propagating bad data.

## Common Mistakes

### Mistake 1: Overly Complex Schemas

A schema with 10 nested objects and arrays of 20 fields each is hard for the model to fill correctly and expensive in tokens. The model may skip fields or populate them with placeholder values.

**Symptoms:** Frequent validation errors, high token usage from schema overhead, slow responses.

**Fix:** Flatten where possible. Split large schemas into multiple simpler extractions. Use `Optional` fields to avoid requiring data the model cannot reliably extract.

### Mistake 2: Not Validating the Response

Structured output reduces parsing errors but does not eliminate them. The API may return a tool use with missing fields, type mismatches, or values outside allowed ranges (e.g., confidence of 2.5).

**Symptoms:** Downstream code crashes because a field is missing or has the wrong type. The bug is intermittent and hard to reproduce.

**Fix:** Always validate the response through your Pydantic constructor. Wrap in a `try/except ValidationError` and log the raw output for debugging.

### Mistake 3: Using Non-Deterministic Fields as Unique Identifiers

If you ask the model to generate an `id` field, you get a different value every call — even for the same input. The model is not a hash function.

**Symptoms:** Duplicate records in your database because the same entity got different IDs.

**Fix:** Generate IDs on your side (UUID, hash of content). Never let the model create identifiers.

### Mistake 4: Forgetting Optional Fields

Required fields with no default cause the entire extraction to fail if the model omits them. Documents often lack information you expect.

**Symptoms:** Extraction fails on a large percentage of inputs because the model could not find an author name or date.

**Fix:** Make realistically-missing fields `Optional` with a default of `None`.

```python
class Article(BaseModel):
    title: str
    author: Optional[str] = None  # Not all articles have authors
    date: Optional[str] = None    # Not all documents have dates
```

### Mistake 5: Ignoring Enum Mismatches

If your schema has an enum field (e.g., `sentiment: Literal["POSITIVE", "NEGATIVE", "MIXED"]`), the model may return "NEUTRAL" or "positive" (wrong case). The API may not catch this if it uses string matching.

**Symptoms:** Validation errors on enum fields even though the model's output looks reasonable.

**Fix:** Use `Literal` types and add a case-normalisation step. Or accept a broader string type and normalise after extraction.

```python
from typing import Literal

# Strict — may fail
class StrictResult(BaseModel):
    sentiment: Literal["POSITIVE", "NEGATIVE", "MIXED"]

# Lenient with post-processing
class LenientResult(BaseModel):
    sentiment: str

def normalise_sentiment(raw: str) -> str:
    mapping = {"positive": "POSITIVE", "negative": "NEGATIVE",
               "mixed": "MIXED", "neutral": "NEGATIVE"}
    return mapping.get(raw.strip().lower(), "NEGATIVE")
```
