# Structured Output — Hands On

⏱ 15 minutes

## Breaking It Down

### Step 1: Install Dependencies

```bash
pip install anthropic pydantic
```

Pydantic provides the model classes you use to define your output schema. The Anthropic SDK can accept a Pydantic model directly.

### Step 2: Define Your Output Schema

Define what you want the model to return as a Pydantic model:

```python
# schemas.py
from pydantic import BaseModel


class Person(BaseModel):
    """Schema for person extraction."""
    name: str
    age: int
    occupation: str
    email: str | None = None  # Optional field


class SentimentResult(BaseModel):
    """Schema for sentiment analysis."""
    sentiment: str  # POSITIVE, NEGATIVE, MIXED, NEUTRAL
    confidence: float  # 0.0 to 1.0
    key_phrases: list[str]
```

Each field has a type and optionally a default. Pydantic validates types at construction time — if the model returns `"thirty"` for an `int` field, it will raise a `ValidationError`.

### Step 3: Call the API with Structured Output

Anthropic supports structured output through tool use with `tool_choice` set to force a specific tool:

```python
# structured_call.py
import os
from anthropic import Anthropic
from schemas import Person

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": "Alice Johnson is a 32-year-old software engineer from San Francisco. "
                   "Her email is alice@example.com."
    }],
    tools=[
        {
            "name": "extract_person",
            "description": "Extract person details from text",
            "input_schema": Person.model_json_schema(),
        }
    ],
    tool_choice={"type": "tool", "name": "extract_person"},
)

# The response contains a tool use block
for block in response.content:
    if block.type == "tool_use":
        person = Person(**block.input)
        print(person)
        # Person(name='Alice Johnson', age=32, occupation='software engineer',
        #        email='alice@example.com')
```

With OpenAI, use `response_format` with a JSON schema:

```python
from openai import OpenAI
from schemas import Person

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

completion = client.beta.chat.completions.parse(
    model="gpt-4o",
    messages=[{
        "role": "user",
        "content": "Alice Johnson is a 32-year-old software engineer from San Francisco. "
                   "Her email is alice@example.com."
    }],
    response_format=Person,
)

person = completion.choices[0].message.parsed
print(person)
# Person(name='Alice Johnson', age=32, occupation='software engineer',
#        email='alice@example.com')
```

### Step 4: Handling Validation Errors

Even with structured output, things can go wrong — the model may omit a required field or the API may fall back to free text:

```python
# validation_handling.py
from pydantic import BaseModel, ValidationError


class ExtractionResult(BaseModel):
    topic: str
    summary: str
    key_points: list[str]


def safe_parse(data: dict, model: type[BaseModel]) -> dict | None:
    """Try to parse dict into Pydantic model, return None on failure."""
    try:
        instance = model(**data)
        return instance.model_dump()
    except ValidationError as e:
        print(f"Validation failed: {e.errors()}")
        return None


# Simulate an LLM response that has issues
llm_output = {"topic": "AI", "summary": "AI is...", "key_points": "not a list"}
result = safe_parse(llm_output, ExtractionResult)
# Validation failed: [{'type': 'string_type', 'loc': ('key_points',), ...}]
print(result)  # None
```

### Step 5: Multi-Field Extraction with Nested Schemas

Real-world extraction often requires nested objects:

```python
# nested_schema.py
from pydantic import BaseModel
from typing import Optional


class Address(BaseModel):
    street: str
    city: str
    country: str


class Product(BaseModel):
    product_id: str
    name: str
    price: float
    in_stock: bool


class Invoice(BaseModel):
    invoice_number: str
    vendor: str
    customer: str
    billing_address: Address
    items: list[Product]
    total: float
    notes: Optional[str] = None


# Send this schema to the API to parse an invoice from raw text
print(Invoice.model_json_schema())
# Generates a full JSON Schema with nested objects and arrays
```

## Build Something Real — Document Extraction Pipeline

The following pipeline extracts structured data from raw documents, validates every field, and saves the results as clean JSON:

```python
# extraction_pipeline.py
import os
import json
import logging
from pathlib import Path
from typing import Optional

from pydantic import BaseModel, ValidationError
from anthropic import Anthropic

logger = logging.getLogger(__name__)


class DocumentSummary(BaseModel):
    title: str
    author: Optional[str] = None
    date: Optional[str] = None
    main_topics: list[str]
    key_findings: list[str]
    word_count_estimate: int
    sentiment: str


class ExtractionPipeline:
    """Extracts structured data from documents using structured output."""

    def __init__(self):
        self.client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

    def extract(self, text: str) -> DocumentSummary | None:
        """Extract structured summary from raw text.

        Args:
            text: Raw document text (up to ~50K tokens).

        Returns:
            DocumentSummary if extraction succeeds, None otherwise.
        """
        try:
            response = self.client.messages.create(
                model="claude-sonnet-4-20250514",
                max_tokens=1024,
                messages=[{
                    "role": "user",
                    "content": f"Extract structured information from this document:\n\n{text[:10000]}"
                }],
                tools=[{
                    "name": "summarize_document",
                    "description": "Extract structured document summary",
                    "input_schema": DocumentSummary.model_json_schema(),
                }],
                tool_choice={"type": "tool", "name": "summarize_document"},
            )

            for block in response.content:
                if block.type == "tool_use":
                    return DocumentSummary(**block.input)

            logger.warning("No tool use in response")
            return None

        except (ValidationError, Exception) as e:
            logger.error("Extraction failed: %s", e)
            return None


# Run on multiple documents
pipeline = ExtractionPipeline()

documents = [
    "Machine learning is transforming healthcare. A new study published in Nature "
    "shows that deep learning models can detect lung cancer from CT scans with 94% accuracy. "
    "The research team, led by Dr. Smith at Stanford, analysed 10,000 patient scans.",

    "Python 3.13 introduces several performance improvements. The new JIT compiler "
    "shows a 20-30% speedup on CPU-bound workloads. Memory usage is also reduced "
    "by approximately 15% compared to Python 3.12.",
]

results = []
for doc in documents:
    result = pipeline.extract(doc)
    if result:
        results.append(result.model_dump())
        logger.info("Extracted: %s", result.title)

# Save as JSON
output_path = Path("extracted_documents.json")
output_path.write_text(json.dumps(results, indent=2))
print(f"Saved {len(results)} extracted documents to {output_path}")
```

## Performance Considerations

- **Schema size adds tokens.** A large JSON schema with many fields and nested objects consumes prompt tokens. Keep schemas as lean as the task allows. A schema with 50 fields costs more per call than one with 5.
- **Structured output can be slower.** Constraining token generation to match a schema adds overhead. For simple extractions, the difference is negligible. For complex nested schemas with arrays, expect 10-30% higher latency.
- **Validation is cheap.** Pydantic validation takes microseconds. Always validate — do not skip it for performance.

## Security Considerations

- **Schema injection.** If you dynamically generate schemas from user input (e.g., "extract any fields the user specifies"), a malicious user can craft a schema that causes excessive token generation or expensive validation. Never let users define arbitrary schemas.
- **Field-level access control.** If your schema exposes fields like `salary` or `ssn`, ensure the pipeline calling the LLM has the authority to extract those fields. Do not send sensitive schemas to the model if the user should not see that data.
