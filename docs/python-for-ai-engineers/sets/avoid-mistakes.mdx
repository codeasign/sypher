# Sets — Common Mistakes

⏱ 5 minutes

## When NOT to Use Sets

Sets are specialised for unordered uniqueness. Do not use them for:

- **Ordered data:** Sets have no concept of position. If you need to process elements in insertion order, use a list.
- **Duplicate-tracking with counts:** If you need to know how many times each element appears, use `collections.Counter`.
- **Mutable elements:** Sets cannot contain lists, dicts, or other sets. Use frozensets for nested set structures.
- **JSON serialisation:** Sets are not JSON-serializable by default. Convert to a list: `json.dumps(list(my_set))`.

## The Pattern to Remember

- **Pattern:** Create with `set()` or `{a, b, c}` (not `{}`), add with `.add()`, check membership with `in`, deduplicate with `set(list)`.
- **When to use:** Fast membership checks, deduplication, mathematical set operations.
- **When NOT to use:** Ordered data, counting with tallies, mutable collections.
- **Mental model:** A set is a bag of unique items — you can check if something is in the bag instantly, but you cannot ask "what is at position 3?"

## Common Mistakes

### Mistake 1: Creating an Empty Set with `{}`

```python
wrong = {}      # This is an empty dictionary!
correct = set()  # This is an empty set
```

**Why it happens:** Curly braces were historically used for dictionaries. Python uses `{}` for empty dicts and requires `set()` for empty sets.

**Symptoms:** When you try to add elements with `.add()`, you get `AttributeError: 'dict' object has no attribute 'add'`.

**Fix:** Always use `set()` for an empty set. Only use `{a, b, c}` when you have elements to include.

### Mistake 2: Assuming Sets Have Order

```python
words = {"cat", "dog", "emu"}
print(words)  # {'dog', 'cat', 'emu'} — maybe not the order you specified
```

**Why it happens:** Sets are unordered by design. The internal hash table ordering is an implementation detail that varies between runs and Python versions.

**Symptoms:** Your code works in testing but produces different results on a different machine or Python version. Assertions that check element order fail.

**Fix:** Never rely on set ordering. If you need a specific order, use `sorted()` or convert to a list and sort it.

### Mistake 3: Modifying a Set While Iterating Over It

```python
items = {1, 2, 3, 4, 5}
for item in items:
    if item % 2 == 0:
        items.remove(item)  # RuntimeError: set changed size during iteration
```

**Why it happens:** Same rule as dictionaries — you cannot modify a collection's size while iterating over it.

**Symptoms:** `RuntimeError`.

**Fix:** Iterate over a copy:

```python
for item in items.copy():
    if item % 2 == 0:
        items.remove(item)
```

### Mistake 4: Using a Set with Mutable Elements

```python
# Trying to put lists in a set
data = {[1, 2], [3, 4]}  # TypeError: unhashable type: 'list'
```

**Why it happens:** Sets require hashable elements. Lists, dicts, and sets are mutable, hence not hashable.

**Symptoms:** `TypeError: unhashable type`.

**Fix:** Convert lists to tuples:

```python
data = {(1, 2), (3, 4)}  # Tuples are hashable
```

Or use `frozenset` if you need a set of sets.

### Mistake 5: Forgetting That Sets Deduplicate Automatically

```python
expected = [1, 2, 3]  # You want exactly three elements
data = list(set([1, 2, 2, 3, 3]))  # [1, 2, 3] — fine for dedup
```

But when you need duplicates preserved:

```python
scores = [85, 90, 85, 92, 90, 90]
unique_scores = list(set(scores))
print(unique_scores)  # {90, 85, 92} — order not preserved, duplicates gone
```

**Why it happens:** You use `set()` to filter something and inadvertently lose duplicates or ordering that mattered.

**Symptoms:** Data loss — fewer results than expected, or results in an arbitrary order.

**Fix:** Only use `set()` when you explicitly want unique elements and do not need ordering. When you need both uniqueness and order, use a loop with a seen-set:

```python
seen = set()
ordered_unique = []
for x in scores:
    if x not in seen:
        seen.add(x)
        ordered_unique.append(x)
```
