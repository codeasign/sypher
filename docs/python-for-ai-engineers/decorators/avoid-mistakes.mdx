# Decorators — Common Mistakes

⏱ 6 minutes

## When NOT to Use Decorators

- **Single-use wrapping:** If you only wrap one function, a decorator adds ceremony without benefit — just add the logic inline.
- **Simple behaviour that could be a helper function:** If the wrapper is just calling another function, refactor instead.
- **Decorators that change the function signature:** Wrapping a function and changing its return type or argument expectations confuses callers. If you do this, document it explicitly.

## The Pattern to Remember

- **Pattern:** `def decorator(func):` → `@functools.wraps(func)` → `def wrapper(*args, **kwargs):` → `return func(*args, **kwargs)` → `return wrapper`.
- **When to use:** Cross-cutting concerns that apply to many functions — logging, timing, retry, caching, validation.
- **When NOT to use:** Single-use behaviour, simple logic better expressed as a helper function.
- **Mental model:** A decorator is a wrapper layer that adds behaviour around a function without modifying the function itself.

## Common Mistakes

### Mistake 1: Forgetting `@functools.wraps`

```python
def bad_decorator(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@bad_decorator
def train():
    """Train the model."""
    pass

print(train.__name__)  # 'wrapper' — not 'train'!
print(train.__doc__)   # None — docstring lost!
```

This breaks introspection: debuggers, documentation tools, and libraries like Flask or Click that inspect function metadata will not work correctly.

**Fix:** Always decorate the inner wrapper with `@functools.wraps(func)`.

### Mistake 2: Decorator With vs. Without Arguments

A decorator without arguments receives the function directly. A decorator with arguments needs an extra nesting level:

```python
# Without arguments
@timer
def f(): ...

# With arguments
@retry(max_attempts=3)
def g(): ...
```

The `retry` decorator is actually `retry(max_attempts=3)` which returns a decorator which then receives `g`. Forgetting this extra level causes `TypeError: 'int' object is not callable`.

**Fix:** Recognise the pattern: a decorator with arguments has three nested functions: `outer(args) -> decorator(func) -> wrapper(*args, **kwargs)`.

### Mistake 3: The Wrapper Does Not Return the Function's Result

```python
def log_call(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        func(*args, **kwargs)  # Missing return!
    return wrapper
```

**Fix:** Always return the result: `return func(*args, **kwargs)`.

### Mistake 4: Decorator Order Matters

With stacked decorators, the order in which they wrap the function matters:

```python
@log_call   # Applied second
@timer      # Applied first
def f(): ...

# Equivalent to: f = log_call(timer(f))
# timer runs first (inner), log_call runs second (outer)
```

If you swap them, the time measurement includes the logging overhead, and the logging sees a different execution time.

### Mistake 5: Modifying Mutable Defaults in a Decorator

```python
shared_cache = {}

def cache_result(func):
    def wrapper(*args):
        if args in shared_cache:
            return shared_cache[args]
        result = func(*args)
        shared_cache[args] = result
        return result
    return wrapper
```

Two problems: the cache never clears, and it is global. For real caching, use `functools.lru_cache` instead of writing your own.
