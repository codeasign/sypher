# Control Flow — Common Mistakes

⏱ 6 minutes

## When NOT to Use if/elif/else Chains

Conditional chains are the right tool for most branching. But for some cases, other approaches are better:

- **Lookup tables:** If you are mapping fixed values to fixed outcomes (e.g. HTTP status code to label), use a dictionary instead of a 20-branch if/elif chain.
- **Polymorphism:** If different types need different behaviour, use classes and method dispatch rather than type-checking conditionals.
- **Strategy pattern:** If you select an algorithm at runtime, store callables in a dict rather than writing a large conditional.

## The Pattern to Remember

- **Pattern:** `if condition:` → `elif condition:` → `else:` — most specific conditions first.
- **When to use:** Any time your program must choose between code paths based on a condition.
- **When NOT to use:** Fixed-value lookups (use dict), type-based dispatch (use classes).
- **Mental model:** A conditional is a funnel — the most specific test goes first; everything else falls through to the bottom.

## Common Mistakes

### Mistake 1: Using `=` Instead of `==` in Conditions

```python
if score = 0.5:  # SyntaxError — assignment in condition
```

**Why it happens:** C and Java allow assignment as an expression inside `if`. Python does not — `=` is a statement, not an expression.

**Symptoms:** `SyntaxError: invalid syntax. Maybe you meant '=='?`

**Fix:** Use `==` for comparison. If you intentionally want to assign and test in one expression (Python 3.8+), use the walrus operator `:=`:

```python
if (result := api_call()) is not None:
    print(result)
```

### Mistake 2: Ordering Conditions Wrong (Unreachable Branch)

```python
def describe(confidence):
    if confidence > 0.3:
        return "medium"
    elif confidence > 0.6:  # This branch can never execute!
        return "high"
    else:
        return "low"

print(describe(0.8))  # "medium" — not "high"!
```

**Why it happens:** You put a broad condition before a narrow one. Any value above 0.3 matches the first branch and short-circuits.

**Symptoms:** The more specific branch never executes, but you do not notice because the code does not crash.

**Fix:** Order conditions from most specific to most general:

```python
if confidence > 0.6:
    return "high"
elif confidence > 0.3:
    return "medium"
else:
    return "low"
```

### Mistake 3: Forgetting Parentheses in Compound Conditions

```python
# What you wrote:
if status_code == 401 or 403:  # ❌ Bug!
    print("auth error")

# Python sees: (status_code == 401) or (403 — which is truthy!)
# This condition is ALWAYS True!
```

**Why it happens:** `or` separates two conditions. `status_code == 401 or 403` is parsed as `(status_code == 401) or 403`. Since `403` is truthy, the whole condition is always `True`.

**Symptoms:** The branch always executes, regardless of the actual status code.

**Fix:** Explicitly compare each value:

```python
if status_code == 401 or status_code == 403:
    print("auth error")
```

Or use `in`:

```python
if status_code in (401, 403):
    print("auth error")
```

### Mistake 4: Deep Nesting (Arrow Code)

```python
# Hard to read and debug
if response["status"] == 200:
    if response["body"]:
        if response["body"]["confidence"] > 0.5:
            if response["body"]["confidence"] > 0.9:
                label = "very confident"
            else:
                label = "confident"
        else:
            label = "uncertain"
    else:
        label = "empty"
else:
    label = "error"
```

**Why it happens:** You add conditions incrementally without refactoring.

**Symptoms:** Code that looks like an arrow shape from the indentation. Hard to trace all paths.

**Fix:** Use early returns or guard clauses to flatten:

```python
if response["status"] != 200:
    return "error"
if not response["body"]:
    return "empty"

confidence = response["body"]["confidence"]
if confidence > 0.9:
    return "very confident"
if confidence > 0.5:
    return "confident"
return "uncertain"
```

### Mistake 5: Chaining Comparisons Incorrectly

Python's chained comparisons look intuitive, but you can accidentally write a condition that is always `False`:

```python
score = 0.45

# Bug: can any number be both > 0.6 AND < 0.3?
if 0.6 < score < 0.3:  # Always False — no number satisfies both
    print("in range")   # This never executes
```

**Why it happens:** `0.6 < score < 0.3` is equivalent to `(0.6 < score) and (score < 0.3)`. For a score of 0.45, the first part is `False`, so the whole expression short-circuits to `False`. For score = 0.8, the first part is `True` but the second is `False`. No value of `score` can satisfy both simultaneously when the bounds are reversed. Python does not warn about this — it is mathematically well-defined, just logically impossible.

**Symptoms:** The branch never executes, but the code compiles without errors. You may only notice during testing or when a user reports missing behaviour.

**Fix:** Write chained comparisons with the lower bound on the left and the upper bound on the right:

```python
if 0.3 < score <= 0.6:   # Correct — score between 0.3 (exclusive) and 0.6 (inclusive)
    print("in range")     # Executes for score = 0.45
```
