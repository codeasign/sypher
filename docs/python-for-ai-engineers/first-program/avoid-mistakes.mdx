# Your First Program — Common Mistakes

⏱ 5 minutes

## When NOT to Use This Pattern

The script structure shown here — a single file, top-to-bottom execution with `input()` or `sys.argv` — is for entry-point scripts that you run directly. For **reusable libraries**, **web applications**, or **multi-file projects**, you will use a different pattern: functions, classes, and `if __name__ == "__main__"` (covered in the Functions and Modules lessons). Use this linear script pattern only for quick utilities, prototypes, and entry points.

## The Pattern to Remember

- **Pattern:** Read input → process → print output. A script is a sequence of statements executed top to bottom.
- **When to use:** Small utilities, data-processing scripts, API call wrappers, and any script you invoke from the command line.
- **When NOT to use:** Libraries, web apps, long-running services, or any code that another module imports — those need functions and classes.
- **Mental model:** A script is a recipe — each line is one step. Python follows them in order.

## Common Mistakes

### Mistake 1: Running the Wrong File

**Mistake:** You edit the file but `python3 greet.py` still runs the old version.

**Why it happens:** You forgot to save the file before running it. The editor buffer has your changes but the disk still has the old content.

**Symptoms:** Changes do not appear when you run the script. You edit, run, edit, run — no difference.

**Fix:** Save the file (`Ctrl+S` in most editors) before running. Some editors auto-save on focus loss — enable that setting.

### Mistake 2: Forgetting Quotes Around Strings

**Mistake:** Writing `print(Hello)` instead of `print("Hello")`.

**Why it happens:** In natural language, you write `Hello` directly. Python interprets bare words as variable names, not string literals.

**Symptoms:** `NameError: name 'Hello' is not defined`.

**Fix:** String literals must be enclosed in quotes — single or double. Both `'Hello'` and `"Hello"` work, but choose one style and be consistent.

### Mistake 3: IndentationErrors from Mixing Tabs and Spaces

**Mistake:** Using a mix of tabs and spaces for indentation.

**Why it happens:** The editor inserts tabs by default while you press the spacebar elsewhere. Python 3 treats tabs and spaces as distinct but uses an 8-space tab stop internally, so the visual alignment in your editor may not match Python's interpretation.

**Symptoms:** `TabError: inconsistent use of tabs and spaces in indentation`.

**Fix:** Configure your editor to insert spaces when you press Tab. In VS Code, set `"editor.insertSpaces": true` and `"editor.tabSize": 4` in settings.json.

### Mistake 4: Assuming input() Returns a Number

**Mistake:** Writing `age = input("Age: ")` then using `age + 1`.

**Why it happens:** `input()` always returns a string. If the user types `25`, `age` is `"25"` (a string), not `25` (an integer).

**Symptoms:** `TypeError: can only concatenate str (not "int") to str`, or you get `"251"` instead of `26`.

**Fix:** Convert explicitly with `int()` or `float()`: `age = int(input("Age: "))`. If the user enters non-numeric input, this raises a `ValueError` — handle it with a `try/except` block (covered in Error Handling).

### Mistake 5: Misunderstanding sys.argv Indexing

**Mistake:** Accessing `sys.argv[0]` thinking it is the first argument.

**Why it happens:** You expect `python3 greet.py Jenny` to put `"Jenny"` in `sys.argv[0]`.

**Symptoms:** You pass arguments but the script uses the script name instead of the argument value.

**Fix:** `sys.argv[0]` is always the script name. The first actual argument is `sys.argv[1]`. The length `len(sys.argv)` includes the script name — `["greet.py", "Jenny"]` has length 2, with index 0 being the script name and index 1 being the first argument.
