# Embeddings — Common Mistakes

⏱ 5 minutes

## When NOT to Use Embeddings

- **Exact keyword matching.** If you need to find documents containing an exact phrase or specific ID, use a traditional search index (Elasticsearch, SQL LIKE). Embeddings are fuzzy by nature.
- **Small, static datasets.** For fewer than 100 documents, a simple keyword search or even a grep is simpler and equally effective.
- **Real-time classification at high throughput.** If you need to classify thousands of short texts per second (e.g., spam filtering), a fine-tuned classifier is faster and cheaper than embedding + nearest neighbour.

## The Pattern to Remember

- **Pattern:** Embed all documents once (offline), embed the query at search time, compute cosine similarity, return top-K results by score.
- **When to use:** Semantic search, RAG, clustering, recommendation systems, deduplication, and any task where meaning matters more than exact keywords.
- **When NOT to use:** Exact keyword matching, very small datasets (fewer than 100 documents), or high-throughput real-time classification (more than 10K QPS).
- **Mental model:** Embeddings turn text into coordinates in semantic space. Similar meaning = nearby coordinates. Search is a nearest-neighbour problem.

## Common Mistakes

### Mistake 1: Using Different Models for Documents and Queries

The embedding from `text-embedding-3-small` and `text-embedding-3-large` live in different vector spaces. Comparing them is meaningless.

**Symptoms:** Search quality is terrible. Similar texts get low similarity scores.

**Fix:** Always use the same embedding model for both documents and queries. Store the model name alongside each embedding so you can verify consistency at search time.

### Mistake 2: Not Normalising Vectors

Cosine similarity expects unit-length vectors. If you ever compute dot product directly instead of cosine similarity, unnormalised vectors give incorrect results.

```python
# Wrong — assumes vectors are normalised
score = np.dot(a, b)

# Right — normalise before dot product
score = np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Or just normalise once after embedding
a = a / np.linalg.norm(a)
b = b / np.linalg.norm(b)
score = np.dot(a, b)  # Now safe
```

**Fix:** Normalise vectors once after embedding. Many ANN libraries expect normalised vectors and will ignore the length component.

### Mistake 3: Ignoring Dimension Reduction

`text-embedding-3-large` returns 3072 dimensions. That much storage and compute is wasted if your task does not need it. OpenAI supports a `dimensions` parameter to truncate the embedding to fewer dimensions.

**Symptoms:** Slow search, high memory usage, no improvement over the smaller model.

**Fix:** Use the `dimensions` parameter. For most tasks, 256-512 dimensions capture enough signal.

```python
response = client.embeddings.create(
    model="text-embedding-3-large",
    input="Some text",
    dimensions=256,  # Truncate to 256 dimensions
)
```

### Mistake 4: Embedding Entire Documents Instead of Chunks

A 100-page PDF gets one vector. A search for a specific detail buried in that PDF returns the document but not the relevant passage.

**Symptoms:** Search returns the right document but you still have to scan it to find the specific answer.

**Fix:** Chunk documents into passages (200-500 tokens each) and embed each chunk separately. The next lesson on RAG covers chunking strategies in depth.

### Mistake 5: Forgetting That Embedding APIs Have Rate Limits

Sending 10,000 documents in a single batch or sending 100 batches per second hits OpenAI rate limits.

**Symptoms:** HTTP 429 errors, failed embeddings, incomplete document collections.

**Fix:** Batch in sizes of 100-500 texts per request. Implement retry logic with exponential backoff. Use `tiktoken` to count tokens and stay under the model's input limit (8192 tokens for `text-embedding-3-small`).
