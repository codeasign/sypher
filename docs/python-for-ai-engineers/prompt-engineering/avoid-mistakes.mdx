# Prompt Engineering — Common Mistakes

⏱ 5 minutes

## When NOT to Use Prompt Engineering

- **When fine-tuning is better.** If your task requires a specific behaviour for thousands of inputs (e.g., classifying legal documents into 50 categories), prompt engineering adds latency and token cost to every call. A fine-tuned small model is faster and cheaper at scale.
- **When you need deterministic guarantees.** Prompts give probabilistic behaviour, not guarantees. If a mistake costs real money (financial calculations, medical advice), do not rely on prompting alone — add validation layers.
- **When simpler solutions exist.** If you can solve the problem with a lookup table, a regex, or 10 lines of Python, do that instead of prompting an LLM.

## The Pattern to Remember

- **Pattern:** System prompt to set role and constraints, 2-3 few-shot examples to anchor format, clear instruction at the end. Temperature 0.0 for deterministic tasks, 0.7-0.9 for creative tasks.
- **When to use:** Any task where you need natural language understanding, generation, or reasoning and cannot use a simpler deterministic alternative.
- **When NOT to use:** High-throughput classification (fine-tune instead), deterministic logic (write code instead), or tasks where mistakes are unacceptable (add validation).
- **Mental model:** Prompting is interactive constraint design. Each failed attempt tells you which constraint is missing — add an example, tighten the format, or lower the temperature.

## Common Mistakes

### Mistake 1: Vague System Prompts

"You are a helpful assistant" is the default — it adds almost no useful constraint. The model behaves the same with or without it.

**Why it happens:** Beginners treat the system prompt as a greeting rather than a behaviour specification.

**Symptoms:** Outputs are inconsistent, drift between responses, and ignore formatting instructions.

**Fix:** Specify role, tone, format, constraints, and what NOT to do in concrete terms.

```python
# Weak — almost no effect
system = "You are a helpful assistant."

# Strong — specific constraints
system = (
    "You are a data extraction pipeline. Respond ONLY with valid JSON. "
    "Do not include explanations, greetings, or markdown. "
    "If a field is missing, use null, not 'N/A' or 'missing'."
)
```

### Mistake 2: Overloading the Prompt

One prompt that tries to do everything — extract, summarise, translate, and format — confuses the model and degrades quality on every axis.

**Symptoms:** The model partially completes the task, mixes up formats, or omits parts of the instruction.

**Fix:** Split into discrete steps. One prompt for extraction, another for summarisation. Chain the outputs.

### Mistake 3: Forgetting Recency Bias

The model pays most attention to the end of the prompt. If you bury the actual instruction after paragraphs of context, the model may miss or ignore it.

**Symptoms:** The model responds to the middle of the context rather than the explicit task at the end.

**Fix:** Put the concrete instruction immediately before where you expect the model to start generating. Context goes before the instruction, not after.

### Mistake 4: Not Testing at Temperature 0.0

If your prompt design relies on getting consistent output, you must set temperature to 0.0. At 0.7, the same prompt can produce different outputs every call — making debugging nearly impossible.

**Symptoms:** The prompt works in testing but produces different (sometimes wrong) results in production. You cannot reproduce the problem.

**Fix:** Use temperature 0.0 for all extraction, classification, and structured output tasks. Reserve higher temperatures only for creative generation.

### Mistake 5: Using a Single Example When You Need Five

If one example does not cover an edge case, the model will invent its own behaviour for that case. For multi-class classification, include at least one example per class, including edge cases.

**Symptoms:** The model handles the common cases correctly but fails on less frequent inputs that fall between classes.

**Fix:** Audit your few-shot examples against the full distribution of inputs you expect. Add examples for boundary cases, not just the typical ones.
