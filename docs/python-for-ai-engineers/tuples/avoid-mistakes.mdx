# Tuples — Common Mistakes

⏱ 5 minutes

## When NOT to Use Tuples

Tuples are for **fixed-length records** that should not change. Do not use them for:

- **Homogeneous collections of unknown length** — use a list instead. A list of 100 prediction scores is a list, not a tuple.
- **Data you need to modify** — by definition, if you need to add or remove elements, you need a list.
- **Large sequences with frequent membership checks** — use a set (though sets are not ordered, so only if order does not matter).

## The Pattern to Remember

- **Pattern:** Create with `(a, b, c)` or `a, b, c`, unpack with `x, y, z = tuple`, use as dict keys.
- **When to use:** Fixed records (return values, coordinates, config pairs), dict keys, data that should not change.
- **When NOT to use:** Mutable data, homogeneous collections of unknown size.
- **Mental model:** A tuple is a sealed envelope — the contents are fixed, but you can still look at them.

## Common Mistakes

### Mistake 1: Forgetting the Trailing Comma for Single-Element Tuples

```python
not_a_tuple = (42)
print(type(not_a_tuple))  # <class 'int'>

actually_a_tuple = (42,)
print(type(actually_a_tuple))  # <class 'tuple'>
```

**Why it happens:** Parentheses are used for grouping expressions. `(42)` is just the integer `42` in parentheses. The comma is what makes it a tuple.

**Symptoms:** You pass what you think is a tuple of one element, but downstream code iterates over the integer and gets `TypeError: 'int' object is not iterable`, or the length is wrong.

**Fix:** Always include the trailing comma for single-element tuples: `(42,)`. This also applies when building tuples in function arguments: `f((42,))` not `f((42))`.

### Mistake 2: Trying to Modify a Tuple

```python
t = (1, 2, 3)
t[0] = 99  # TypeError: 'tuple' object does not support item assignment
t.append(4)  # AttributeError: 'tuple' object has no attribute 'append'
```

**Why it happens:** You are used to lists being mutable and expect tuples to behave the same way.

**Symptoms:** `TypeError` or `AttributeError` at runtime.

**Fix:** If you need to modify the data, use a list. If you thought you needed a tuple but need to modify it, convert to a list first: `mutable = list(t)`, modify, then convert back if needed: `t = tuple(mutable)`.

### Mistake 3: Unpacking Mismatch — Wrong Number of Values

```python
record = ("gpt-4", 2026, 0.95, "production")
model, year, score = record  # ValueError: too many values to unpack
```

**Why it happens:** The number of variables on the left does not match the number of elements on the right.

**Symptoms:** `ValueError: too many values to unpack` or `not enough values to unpack`.

**Fix:** Use `_` for values you do not need, or use extended unpacking with `*`:

```python
model, year, score, _ = record  # OK — ignore "production"
model, year, *rest = record     # OK — rest = [0.95, "production"]
```

Or ensure your assignments match the tuple length.

### Mistake 4: Confusing Tuples with Parenthesised Expressions

```python
result = (5 + 3) * 2    # 16 — parentheses for grouping, not a tuple
result2 = (5 + 3,) * 2  # (8, 8) — tuple with one element, repeated
```

**Why it happens:** Parentheses pull double duty — they group expressions and create tuples. Without a comma, Python treats `(...)` as grouping.

**Symptoms:** You get a scalar value instead of a tuple, or an unexpected multiplication result.

**Fix:** When in doubt, add the trailing comma for tuples, and avoid ambiguous expressions. Use `tuple()` constructor for clarity: `tuple([42])`.

### Mistake 5: Using a List Where a Tuple Is More Appropriate

```python
# Over-use of lists — this should be a tuple
def get_config():
    return ["gpt-4", 2026, True]  # Returns a mutable list for a fixed record

config = get_config()
config.append("extra")  # This should not be possible for a fixed record
```

**Why it happens:** Lists are the default collection in many developers' muscle memory. You use a list for everything without thinking about whether mutability is needed.

**Symptoms:** The code works, but other developers cannot tell whether the return value is meant to be modified. Bugs arise when someone mutates what should be a fixed value.

**Fix:** Use tuples for fixed records. Treat the presence of `append()`/`pop()` calls as a signal: if you never mutate, use a tuple.
