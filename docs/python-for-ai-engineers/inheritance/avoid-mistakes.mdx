# Inheritance — Common Mistakes

⏱ 6 minutes

## When NOT to Use Inheritance

- **Has-a relationships:** If class A *has* a class B (e.g., a Model *has* a Tokeniser), use composition, not inheritance. Inheritance should model is-a.
- **Just to reuse one method:** If you only want to reuse a single method from another class, extract it into a utility function or use composition.
- **Deep hierarchies:** More than 2-3 levels of inheritance makes code hard to understand and debug. Prefer composition over deep inheritance.
- **Frameworks that expect inheritance:** Frameworks like PyTorch require inheritance from their base classes — that is appropriate. But creating your own deep hierarchies in application code is rarely the right choice.

## The Pattern to Remember

- **Pattern:** Base class defines the interface with abstract methods → subclasses implement specifics → `super()` calls the parent when extending.
- **When to use:** Is-a relationships, shared interface with multiple implementations, framework-required inheritance.
- **When NOT to use:** Has-a relationships (use composition), single-method reuse (use a function), deep hierarchies (prefer composition).
- **Mental model:** Inheritance says "I am a kind of." Composition says "I have a." Prefer composition.

## Common Mistakes

### Mistake 1: Forgetting to Call `super().__init__()`

```python
class Child(Base):
    def __init__(self, x, y):
        self.x = x
        # Forgot: super().__init__(y)
        # Base's attributes are never initialised
```

**Fix:** Call `super().__init__()` at the start of the child's `__init__` if the parent needs initialisation.

### Mistake 2: Deep Inheritance Hierarchies

```python
class A: pass
class B(A): pass
class C(B): pass
class D(C): pass
class E(D): pass  # 4 levels deep — hard to trace
```

Beyond 2-3 levels, the chain of behaviour becomes difficult to trace.

### Mistake 3: Diamond Problem Confusion

```python
class A:
    def method(self): print("A")

class B(A):
    def method(self): print("B")

class C(A):
    def method(self): print("C")

class D(B, C): pass  # Which method() does D use?

d = D()
d.method()  # B — Python uses MRO (left-to-right, depth-first)
```

Python's MRO is well-defined (C3 linearisation), but multiple inheritance with the same method in multiple parents is confusing. Use mixins for small, orthogonal behaviours instead.

### Mistake 4: Overriding Instead of Extending

```python
class Base:
    def setup(self):
        print("Base setup")

class Child(Base):
    def setup(self):
        print("Child setup")  # Overrides — Base.setup never runs
```

If you intend to extend the parent's behaviour, call `super().setup()`:

```python
class Child(Base):
    def setup(self):
        super().setup()     # Runs Base.setup
        print("Child setup")  # Then adds child-specific logic
```

### Mistake 5: Using Inheritance for Code Reuse When Composition Is Better

```python
# Inheritance — wrong: a Model is not a kind of Logger
class Model(Logger):
    def predict(self, text):
        self.log("Predicting...")

# Composition — right: a Model has a Logger
class Model:
    def __init__(self, logger):
        self.logger = logger

    def predict(self, text):
        self.logger.log("Predicting...")
```
