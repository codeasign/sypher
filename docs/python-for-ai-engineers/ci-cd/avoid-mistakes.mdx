# CI/CD — Common Mistakes

⏱ 5 minutes

## When NOT to Use CI/CD

- **Personal projects with no users.** CI/CD adds setup overhead. If you are the only user and deploy manually, skip it until you have collaborators or users depending on uptime.
- **Exploratory research code.** Jupyter notebooks, quick experiments, and throwaway scripts do not need automated pipelines. CI/CD is for production code.
- **Prototypes before product-market fit.** Premature CI/CD slows iteration. Wait until you have users depending on the service.

## The Pattern to Remember

- **Pattern:** Separate CI jobs (quality, test, build) that run in parallel, followed by a CD job (deploy) that only runs on main. Cache dependencies. Use environment variables for secrets. Set coverage thresholds.
- **When to use:** Any production application with multiple contributors or users who depend on uptime.
- **When NOT to use:** Personal projects, research code, or early prototypes.
- **Mental model:** CI is a checklist that runs on every push. CD is the automation that deploys when the checklist passes. Both are scripted in YAML and live in version control.

## Common Mistakes

### Mistake 1: Flaky Tests in CI

A test that passes locally and fails in CI 30% of the time destroys trust in the pipeline. Developers start ignoring CI failures.

**Why it happens:** Tests depend on network calls, wall-clock timing, or shared state between test runs. CI runs on different hardware with different timing.

**Fix:** Mock all network calls. Use fixed seeds for randomness. Run flaky tests in isolation with `pytest --flake-finder` to identify them, then fix or quarantine them.

### Mistake 2: Hardcoded Secrets in Workflow Files

Committing a file with `${{ secrets.API_KEY }}` is safe — GitHub evaluates the expression. But committing a workflow that uses `${{ secrets.ANTHROPIC_API_KEY }}` and forgetting to set the secret in the repo causes a silent failure or a cryptic error.

**Symptoms:** The workflow runs but the secret evaluates to empty. Tests fail because the API key is blank.

**Fix:** Add a validation step at the start of the workflow:

```yaml
- name: Validate secrets
  run: |
    if [ -z "${{ secrets.ANTHROPIC_API_KEY }}" ]; then
      echo "ERROR: ANTHROPIC_API_KEY is not set"
      exit 1
    fi
```

### Mistake 3: No Cache for Dependencies

Without caching, every CI run re-downloads all pip packages. A 30-second install becomes 3 minutes.

**Symptoms:** CI runs are dominated by pip install. Developers wait longer for results.

**Fix:** Use `actions/cache` to cache `~/.cache/pip`. Cache the `.venv` directory if using a virtual environment. Invalidate the cache when `requirements.txt` changes.

### Mistake 4: Deploying Without Health Checks

The pipeline deploys successfully — the container started. But the application is crashing on startup because of a missing environment variable. The health check would fail, but nobody checks.

**Symptoms:** The deploy reports "success" but the service is down. The error only surfaces when users report it.

**Fix:** Add health check polling to the deploy step. Wait for the service to return 200 before marking the deploy as successful. Rollback automatically if the health check fails within 5 minutes.

### Mistake 5: Environment Drift Between CI and Production

CI runs Python 3.12 on Ubuntu 22.04. Production runs on Debian 11 with Python 3.11. The code works in CI but fails in production due to a library incompatibility.

**Symptoms:** Deploys fail intermittently with native extension compilation errors. The root cause is a system library version mismatch.

**Fix:** Use the same base image in the Dockerfile for both CI (if you test inside Docker) and production. Or run CI inside the same Docker image that production uses.

```yaml
- name: Test inside production-like environment
  run: |
    docker build -t test-image .
    docker run test-image pytest --cov=. -v
```
