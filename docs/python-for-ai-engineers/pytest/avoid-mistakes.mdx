# Pytest — Common Mistakes

⏱ 5 minutes

## When NOT to Use Pytest

- **Testing external API behaviour.** Do not test whether Anthropic or OpenAI return correct completions — that is their responsibility. Test that your code handles their responses correctly.
- **Covering trivial code.** Getter/setter properties, dataclass definitions with no custom logic, and constants do not need tests. Coverage should focus on business logic.
- **Testing implementation details.** Test observable behaviour (inputs and outputs), not which internal method was called or how many times. Implementation tests break on every refactor.

## The Pattern to Remember

- **Pattern:** One test file per module, one test function per behaviour, fixtures for shared data, parametrize for edge cases, mock for external calls.
- **When to use:** Every Python module with non-trivial logic. Every refactor. Every PR before merge.
- **When NOT to use:** External API behaviour you cannot control (mock it instead), trivial code, or implementation details.
- **Mental model:** A test is a runnable specification. If someone asks "what does this function do?" they should be able to read the tests.

## Common Mistakes

### Mistake 1: Fixture Scope Confusion

By default, fixtures are created fresh for every test function (`scope="function"`). This is correct for mutable state but wasteful for expensive setup.

**Why it happens:** Creating an LLM client fixture without setting `scope="session"` creates a new client for every test, wasting time.

**Fix:** Use `scope` appropriately:

```python
@pytest.fixture(scope="session")
def llm_client():
    """Created once per test session, reused by all tests."""
    return Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

@pytest.fixture(scope="function")
def email_data():
    """Created fresh for each test — mutable state lives here."""
    return {"id": "1", "body": "Test email"}
```

### Mistake 2: Shared Mutable Fixtures

A fixture returns a list. One test modifies it. Another test relying on the same fixture instance sees the modified list.

**Symptoms:** Tests pass when run individually but fail when run together. The failure depends on test order.

**Fix:** Create fresh data inside each test, or use factory fixtures that return new objects:

```python
@pytest.fixture
def fresh_emails():
    return [{"id": str(i), "body": f"Email {i}"} for i in range(3)]
```

### Mistake 3: Testing Real API Calls

A test that calls the real Anthropic API every time is slow, non-deterministic, costs money, and breaks when the API is down.

**Symptoms:** Test suite takes minutes instead of seconds. Tests fail when the network is unavailable or rate limits are hit.

**Fix:** Always mock external API calls. Use `pytest-mock` to patch at the module boundary.

### Mistake 4: Flaky Assertions on LLM Output

Testing that an LLM response contains an exact string when the model's wording varies produces tests that pass sometimes and fail other times.

**Symptoms:** The same test passes on one run and fails on the next with no code change.

**Fix:** Test with mocked responses (return fixed text) or use fuzzy matching (substring, regex, key-phrase presence).

### Mistake 5: Not Testing Error Paths

Happy-path tests pass, but the code crashes when the API returns a 429, the input is malformed, or the JSON is invalid.

**Symptoms:** Production errors all come from cases that "would never happen" — but they do.

**Fix:** Write tests for every error path: empty inputs, rate-limit responses, malformed JSON, timeout exceptions, missing fields.

```python
def test_empty_input_returns_fallback():
    pipeline = RAGPipeline()
    result = pipeline.generate_answer("", [])
    assert result == "No relevant documents found."

def test_malformed_api_response_raises_validation_error():
    # Simulate malformed API response
    mock_client = MagicMock()
    mock_client.messages.create.side_effect = ValueError("Invalid response format")
    pipeline = RAGPipeline(llm_client=mock_client)
    with pytest.raises(ValueError):
        pipeline.generate_answer("Hello", ["context"])
```
