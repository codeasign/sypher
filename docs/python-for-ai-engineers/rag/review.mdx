# RAG — Review

⏱ 3 minutes

## Key Takeaways

- RAG grounds LLM answers in your own documents by retrieving relevant chunks before generation.
- Chunk documents into overlapping passages (500 tokens with 100-token overlap is a good starting point), embed each chunk, and store in a vector DB.
- At query time, embed the question, retrieve the top-K chunks, and insert them as context in the prompt.
- Tune chunk size and top-K together — small chunks lose context, large chunks waste tokens, too-few chunks miss information.
- Filter retrieved chunks by metadata for language, date, and access control — embeddings do not understand these attributes.

## Recap in 60 Seconds

If you only remember three things:

1. RAG = retrieve relevant document chunks + insert them as context into the LLM prompt.
2. Chunk quality determines answer quality — chunk semantically (by paragraphs or sections), not by arbitrary token counts alone.
3. If no relevant chunks are retrieved, say "I do not know" rather than letting the LLM guess from its training data.
