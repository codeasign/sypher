# RAG — Common Mistakes

⏱ 5 minutes

## When NOT to Use RAG

- **The LLM already knows the answer reliably.** For well-known facts (historical dates, common knowledge), RAG adds latency and complexity for no benefit. Only retrieve when the information is private, recent, or niche.
- **You need exact answers from structured data.** If your data is in SQL tables, use text-to-SQL — convert the question to a SQL query and run it against the database. RAG on text serialisations of tables is lossy and error-prone.
- **Real-time updates are not needed.** If your knowledge base changes monthly, fine-tuning may be a simpler alternative. RAG shines when data changes daily or hourly.

## The Pattern to Remember

- **Pattern:** Chunk documents → embed chunks → store in vector DB → embed query → retrieve top-K chunks → build prompt with context → generate answer.
- **When to use:** Any scenario where you need an LLM to answer questions about private, recent, or specialised information that is not in its training data.
- **When NOT to use:** Well-known facts the LLM handles reliably, structured querying of databases, or static knowledge that rarely changes (fine-tune instead).
- **Mental model:** RAG is open-book exam for LLMs. You give the model the relevant pages before asking the question. The quality of the answer depends on how well you chose those pages.

## Common Mistakes

### Mistake 1: Poor Chunking Strategy

One chunk size does not fit all. Legal contracts need different chunking than code snippets or chat logs.

**Why it happens:** Using a fixed token count without considering document structure.

**Symptoms:** Retrieval returns chunks that start or end mid-sentence, or chunks that mix unrelated topics.

**Fix:** Use semantic chunking — split at paragraph boundaries, markdown headers, or code function boundaries. Combine with token limits as a safety net.

### Mistake 2: Retrieving Too Few or Too Many Chunks

Top-1 is too few — you miss relevant context. Top-10 is too many — you waste tokens and confuse the model with irrelevant text.

**Symptoms:** With too few, answers are incomplete. With too many, answers hallucinate from irrelevant context.

**Fix:** Start with top-3 to top-5. Tune by measuring retrieval precision on a validation set. Adjust chunk size and top-K together.

### Mistake 3: Not Handling Empty Retrieval

When no relevant chunks are found, the RAG pipeline should say "I do not know." Instead, many pipelines pass the question to the LLM with empty context — which causes the LLM to fall back to its training data or hallucinate.

**Symptoms:** The model answers confidently but incorrectly about topics outside the document collection.

**Fix:** Check if any chunks were retrieved. If not, return a "not found" response instead of sending a prompt with no context.

### Mistake 4: Ignoring Metadata Filtering

If your document collection contains both English and Spanish documents, a Spanish query may retrieve English chunks that happen to be semantically close. Without metadata filtering, retrieval is blind to language, date, source, or access level.

**Symptoms:** Answers mix languages, outdated information appears alongside current information, or users see data from documents they should not access.

**Fix:** Store metadata (language, date, source, access level) with every chunk. Filter by metadata before or after retrieval.

```python
# Filter by metadata before retrieval
results = collection.query(
    query_embeddings=[query_emb],
    n_results=5,
    where={"language": "en", "access_level": "public"},
)
```

### Mistake 5: Context Window Overflow

The retrieved chunks may exceed the LLM's context window when combined with the system prompt and conversation history.

**Symptoms:** The API returns an error about exceeding the maximum token limit, or the model silently truncates the beginning of the context (missing the most important retrieved chunks because the system prompt paged them out).

**Fix:** Calculate the total tokens before sending. If the context plus chunks exceeds the limit, reduce `top_k` or use a smaller chunk size.

```python
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")
max_allowed = 100_000  # Leave room for response tokens

total_tokens = len(enc.encode(system_prompt + context + question))
if total_tokens > max_allowed:
    # Reduce chunks or trim context
    context = truncate_to_token_budget(context, max_allowed - len(enc.encode(system_prompt + question)))
```
