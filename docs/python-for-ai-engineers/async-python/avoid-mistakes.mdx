# Async Python — Common Mistakes

⏱ 6 minutes

## When NOT to Use Async

- **CPU-bound workloads:** Async does not make computation faster. Use `multiprocessing` or `threading` for CPU-intensive tasks.
- **Simple scripts:** If you are making one API call, the overhead of async is unnecessary. Sync is simpler and fast enough.
- **Already synchronous libraries:** Not all libraries support async. Using `requests` inside an async function blocks the event loop — use `httpx.AsyncClient` instead.
- **Very short tasks (microseconds):** The overhead of context-switching between coroutines can exceed the runtime of the task itself.

## The Pattern to Remember

- **Pattern:** Define with `async def`, call with `await`, run with `asyncio.run()`. For concurrent work, create tasks and `await asyncio.gather(*tasks)`. Limit concurrency with `asyncio.Semaphore`.
- **When to use:** I/O-bound workloads — API calls, file reads, database queries, network operations.
- **When NOT to use:** CPU-bound computation, one-off scripts, synchronous-only libraries, sub-microsecond tasks.
- **Mental model:** `await` is a yield point — the coroutine pauses and the event loop runs other coroutines. Async is concurrency on one thread, not parallelism.

## Common Mistakes

### Mistake 1: Blocking the Event Loop with Sync Calls

```python
import asyncio
import time

async def main():
    # This blocks the ENTIRE event loop — no other coroutine can run
    time.sleep(2)  # Blocking call!
    print("Done")

# Compare with:
async def main():
    await asyncio.sleep(2)  # Non-blocking — yields to event loop
    print("Done")
```

**Why it happens:** `time.sleep()`, `requests.get()`, `file.read()` (blocking) — these are synchronous calls that block the current thread. In async code, the event loop runs on one thread, so blocking it stops all coroutines.

**Symptoms:** Other coroutines do not make progress. The program appears to freeze during synchronous operations. `asyncio.gather()` takes as long as the sum of all sync calls.

**Fix:** Use async equivalents: `asyncio.sleep()` not `time.sleep()`, `httpx.AsyncClient` not `requests`, `aiofiles` for file I/O. If you must call a sync function, run it in a thread pool with `asyncio.to_thread()`:

```python
result = await asyncio.to_thread(sync_function, arg1, arg2)
```

### Mistake 2: Forgetting to `await` a Coroutine

```python
import asyncio

async def fetch_data():
    await asyncio.sleep(1)
    return "data"

async def main():
    # Wrong — creates a coroutine object but never runs it
    result = fetch_data()
    print(result)  # <coroutine object fetch_data at 0x...>

    # Correct
    result = await fetch_data()
    print(result)  # data

asyncio.run(main())
```

**Why it happens:** Calling an `async def` function returns a coroutine object — it does not execute the function. Only `await` (or `asyncio.create_task()`) starts execution.

**Symptoms:** Functions appear to return nothing. The program prints coroutine object addresses instead of results. A `RuntimeWarning: coroutine was never awaited` appears.

**Fix:** Use a linter that catches missing `await`. When in doubt, check if the return value is a coroutine.

### Mistake 3: Using `asyncio.run()` in a Jupyter Notebook

```python
import asyncio

# In Jupyter — this raises RuntimeError
asyncio.run(some_coroutine())
```

**Why it happens:** Jupyter already runs an event loop. `asyncio.run()` creates a new one, which fails because a loop is already running.

**Symptoms:** `RuntimeError: asyncio.run() cannot be called from a running event loop`

**Fix:** In Jupyter, use `await` directly at the cell level:

```python
await some_coroutine()
```

For scripts, always use `asyncio.run(main())`.

### Mistake 4: Not Handling Exceptions in `gather` Tasks

```python
import asyncio

async def might_fail(item_id: int):
    if item_id == 2:
        raise ValueError(f"Item {item_id} failed")
    await asyncio.sleep(0.5)
    return f"Item {item_id} OK"

async def main():
    # If any task raises, gather propagates the first exception
    results = await asyncio.gather(
        might_fail(1),
        might_fail(2),  # This raises
        might_fail(3),
    )
    # The other results are lost!

asyncio.run(main())  # ValueError: Item 2 failed
```

**Why it happens:** `asyncio.gather()` raises the first exception from any task, and other results are not available.

**Symptoms:** One failing task cancels the entire batch. Successful results from other tasks are inaccessible.

**Fix:** Use `return_exceptions=True` to capture exceptions as values, or wrap each task in a try/except:

```python
results = await asyncio.gather(
    might_fail(1),
    might_fail(2),
    might_fail(3),
    return_exceptions=True,
)
# results = ["Item 1 OK", ValueError("Item 2 failed"), "Item 3 OK"]
for r in results:
    if isinstance(r, Exception):
        print(f"Task failed: {r}")
    else:
        print(f"Success: {r}")
```

### Mistake 5: Creating Tasks Without Awaiting Them

```python
import asyncio

async def worker():
    await asyncio.sleep(1)
    print("Worker done")

async def main():
    # Task is created but never awaited — it may not complete
    asyncio.create_task(worker())

    # Without await, the event loop shuts down immediately
    # "Worker done" may not print

asyncio.run(main())
```

**Why it happens:** `create_task()` schedules the coroutine to run, but if the main coroutine finishes before the task completes, the event loop shuts down and pending tasks are cancelled.

**Symptoms:** Background tasks do not complete. Log messages from tasks are missing. The program exits before async work finishes.

**Fix:** Always await your tasks before the event loop shuts down:

```python
async def main():
    task = asyncio.create_task(worker())
    # ... do other work ...
    await task  # Wait for the task to finish
```
