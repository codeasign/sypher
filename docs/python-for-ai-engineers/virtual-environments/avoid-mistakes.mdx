# Virtual Environments — Common Mistakes

⏱ 6 minutes

## When NOT to Use `venv`

`venv` is the right tool for standard Python projects where you use `pip`. However:

- **Conda projects:** If your team uses conda (common in data science with its own package manager and non-Python dependencies), use `conda create -n myenv` instead of `venv`. Mixing `venv` and `conda` in the same project causes confusion.
- **Docker containers:** Inside a Docker image, you typically install packages globally into the image's Python — the container itself provides isolation.
- **Poetry / Pipenv:** Some projects use higher-level dependency managers that manage virtual environments automatically. Those are fine too — the concept is the same.

## The Pattern to Remember

- **Pattern:** `python3 -m venv venv` → `source venv/bin/activate` → `pip install` → `pip freeze > requirements.txt` → `deactivate`
- **When to use:** Every Python development project, always.
- **When NOT to use:** Conda environments, Docker containers, or when the project already uses Poetry/Pipenv.
- **Mental model:** A virtual environment is a clean room for each project — its packages cannot interfere with any other project.

## Common Mistakes

### Mistake 1: Forgetting to Activate the Environment

**Mistake:** You create a virtual environment but install packages without activating it.

**Why it happens:** It is easy to run `pip install` right after `python3 -m venv venv` without the activation step in between. The environment exists on disk but is not yet active.

**Symptoms:** Packages install without error, but `pip list` later shows nothing. Your script cannot import them.

**Fix:** Always run `source venv/bin/activate` before the first `pip install`. Check that your shell prompt shows `(venv)` or run `which pip` to verify it points inside the environment.

### Mistake 2: Committing the venv Directory to Git

**Mistake:** Running `git add .` without a `.gitignore` file.

**Why it happens:** The `venv/` directory sits inside your project folder, so it is easy to accidentally track.

**Symptoms:** Repository size grows to hundreds of megabytes. Teammates get merge conflicts on binary `.pyc` files. CI checks take forever because they download your entire environment.

**Fix:** Add `venv/` to `.gitignore` before your first commit. Also add `.env/`, `.venv/`, `env/` — common naming variations. Always use `pip freeze > requirements.txt` to share dependencies instead of sharing the environment directory.

### Mistake 3: Using pip Instead of pip3 (or Vice Versa) Inside the Environment

**Mistake:** Inside the activated environment, running `pip install` when `pip` is a Python 2 binary, or `pip3` resolves outside the environment.

**Why it happens:** The activation script prepends the environment's `bin/` to PATH, but if you have multiple Python installations, the wrong `pip` may shadow the environment's copy.

**Symptoms:** `pip list` shows packages installed outside the environment. `which pip` does not point to `venv/bin/pip`.

**Fix:** Within the activated environment, always use `pip` (not `pip3`) — the environment's `bin/` should be first in PATH. Verify by checking `which pip` after activation. If it does not resolve to the environment, your PATH order has a problem.

### Mistake 4: Not Having a requirements.txt

**Mistake:** Installing packages manually, keeping no record of what was installed.

**Why it happens:** You install a few packages, they work, and you move on. You do not think about reproducibility until someone else needs to run your code.

**Symptoms:** Someone clones your repo, creates a fresh environment, and your script fails with `ModuleNotFoundError`. You do not remember which packages or versions you installed.

**Fix:** Run `pip freeze > requirements.txt` immediately after getting your project working. Update it whenever you add or remove a dependency. Treat `requirements.txt` as a source-controlled file alongside your code.

### Mistake 5: Deleting or Moving the venv Directory

**Mistake:** Moving your project folder to a new location without recreating the virtual environment.

**Why it happens:** The environment contains absolute paths to the Python interpreter. If you move the project, those paths become invalid.

**Symptoms:** Activation fails with `bad interpreter: No such file or directory`. Or activation succeeds but `python3` resolves to the system Python.

**Fix:** Do not move `venv/`. If you need to relocate the project, recreate the environment at the new location: delete the old `venv/`, run `python3 -m venv venv` again, activate, and `pip install -r requirements.txt`. Since `requirements.txt` captures everything, this takes seconds.
