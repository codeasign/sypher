# Type Hints — Common Mistakes

⏱ 5 minutes

## When NOT to Use Type Hints

- **Quick scripts:** For a 20-line script you will run once, annotations add ceremony without benefit.
- **Dynamic code:** Code that intentionally handles many types (e.g., a generic serialiser) may be hard to annotate. Use `Any` sparingly in these cases.
- **Heavily dynamic patterns:** Monkey-patching, `setattr`, and metaprogramming are difficult to type-check. Use judiciously.

## The Pattern to Remember

- **Pattern:** Annotate function signatures always, annotate variable assignments when the type is not obvious from context.
- **When to use:** Every function in every file that is part of a reusable library, module, or long-lived project.
- **When NOT to use:** One-off scripts, exploratory code in a notebook.
- **Mental model:** Type hints are contracts documented in code. They say "I promise this function takes X and returns Y" — and mypy holds you to that promise.

## Common Mistakes

### Mistake 1: Confusing `Optional[X]` with `X | None`

These are equivalent:

```python
from typing import Optional
x: Optional[str] = None  # str | None
y: str | None = None     # Python 3.10+ — same thing
```

`Optional` does not mean "a value that might or might not be present" — it strictly means "the type or `None`."

### Mistake 2: Using `List` and `Dict` Instead of `list` and `dict`

Python 3.9+ supports built-in generics:

```python
# Old (still works but deprecated from typing module)
from typing import List, Dict
x: List[str] = ["a"]
y: Dict[str, int] = {"a": 1}

# New — use built-in types
x: list[str] = ["a"]
y: dict[str, int] = {"a": 1}
```

Use the built-in forms unless you need to support Python 3.8 or earlier.

### Mistake 3: Adding Type Hints That Lie

```python
def get_data() -> str:
    """Return data, but sometimes returns None — annotation is wrong."""
    if random.random() < 0.5:
        return None  # mypy flags this as an error
```

**Fix:** Annotate accurately: `def get_data() -> str | None:`

### Mistake 4: Over-Using `Any`

```python
def process(data: Any) -> Any:  # Effectively untyped — defeats the purpose
```

`Any` disables type checking entirely for that expression. Use it only at boundaries where types are genuinely unknown (e.g., deserialised JSON).

### Mistake 5: Expecting Runtime Type Enforcement

```python
def greet(name: str) -> str:
    return f"Hello, {name}"

greet(42)  # No error at runtime — only mypy flags this
```

Type hints are not runtime assertions. If you need runtime type checking, add explicit guards:

```python
def greet(name: str) -> str:
    if not isinstance(name, str):
        raise TypeError(f"Expected str, got {type(name).__name__}")
    return f"Hello, {name}"
```
