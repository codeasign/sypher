# Pandas — Common Mistakes

⏱ 6 minutes

## When NOT to Use Pandas

- **Very large datasets (100 GB+):** Pandas keeps data in RAM. For datasets that exceed available memory, use Dask, Polars, or a database with chunked reading.
- **Deep learning data loading:** Use PyTorch DataLoader or TensorFlow tf.data — they support GPU transfer, augmentation, and shuffling natively.
- **Highly structured numerical computation:** If all your data is numeric and you only need matrix operations, NumPy is faster and lighter.
- **Real-time streaming:** Pandas is designed for batch analysis, not per-event processing. Use streaming frameworks like Kafka or Flink.

## The Pattern to Remember

- **Pattern:** Load → inspect (`head()`, `info()`, `describe()`) → clean (`dropna()`, `fillna()`) → transform (new columns, filters) → group/aggregate → export.
- **When to use:** Tabular data analysis, CSV/JSON/Excel processing, experiment log analysis, data cleaning, feature engineering.
- **When NOT to use:** Data too large for RAM, deep learning training loops, real-time streaming, pure numerical linear algebra.
- **Mental model:** A DataFrame is a dict of aligned Series — each operation returns a new DataFrame, and you chain them together.

## Common Mistakes

### Mistake 1: Chained Indexing (SettingWithCopyWarning)

```python
import pandas as pd

df = pd.DataFrame({"model": ["gpt-4", "claude-3"], "accuracy": [0.94, 0.93]})

# Chained indexing — may modify a copy, not the original
subset = df[df["accuracy"] > 0.9]  # Returns a copy (usually)
subset["model"] = subset["model"].str.upper()
# SettingWithCopyWarning: A value is trying to be set on a copy of a slice
```

**Why it happens:** `df[df["accuracy"] > 0.9]` may return a copy or a view depending on the internal structure. Modifying the result may or may not affect `df`. Pandas warns you about this ambiguity.

**Symptoms:** `SettingWithCopyWarning` appears. Changes you make to a filtered DataFrame may be lost.

**Fix:** Use `.loc` for all assignments, or use `.copy()` explicitly:

```python
# Works — single assignment with .loc
df.loc[df["accuracy"] > 0.9, "model"] = df.loc[df["accuracy"] > 0.9, "model"].str.upper()

# Or be explicit about the copy
subset = df[df["accuracy"] > 0.9].copy()
subset["model"] = subset["model"].str.upper()
```

### Mistake 2: Using `and` / `or` Instead of `&` / `|`

```python
# Wrong — raises ValueError
df[(df["accuracy"] > 0.9) and (df["latency_ms"] < 1500)]

# Correct
df[(df["accuracy"] > 0.9) & (df["latency_ms"] < 1500)]
```

**Why it happens:** `and` and `or` are Python keywords that work on single booleans. `&` and `|` are bitwise operators that work element-wise on boolean Series. Pandas overloads `&` and `|` for Series, but `and` / `or` cannot be overloaded.

**Symptoms:** `ValueError: The truth value of a Series is ambiguous. Use a.empty, a.bool(), a.item(), a.any() or a.all().`

**Fix:** Always use `&` for AND, `|` for OR, and `~` for NOT. Wrap each condition in parentheses — operator precedence requires it.

### Mistake 3: Ignoring the Index

```python
import pandas as pd

df = pd.DataFrame({"accuracy": [0.94, 0.93, 0.89]}, index=["run1", "run2", "run3"])
# Filter
filtered = df[df["accuracy"] > 0.9]
print(filtered.index)  # Index(['run1', 'run2'], dtype='object')

# After operations, the index carries over
# When merging, index misalignment can cause NaN values
```

**Why it happens:** The index is a fundamental part of a DataFrame, not just decorative. Operations like `concat()`, `merge()`, and `groupby()` use the index for alignment. If you filter, sort, or reorder rows, the index stays with the original labels unless you reset it.

**Symptoms:** Merging produces unexpected `NaN` values. Filtered data keeps old index labels. Joining DataFrames by position fails.

**Fix:** Use `reset_index(drop=True)` after filtering operations that change row order. Use `set_index("col")` to promote a column to the index when you need label-based access.

### Mistake 4: Modifying a DataFrame While Iterating Over It

```python
import pandas as pd

df = pd.DataFrame({
    "model": ["gpt-4", "claude-3", "llama-3"],
    "accuracy": [0.94, 0.93, 0.89],
})

# Slow and fragile — modifying while iterating
for idx, row in df.iterrows():
    if row["accuracy"] < 0.9:
        df.drop(idx, inplace=True)  # Modifies size during iteration
```

**Why it happens:** `iterrows()` yields copies of each row, not views. Modifying the DataFrame inside the loop (dropping rows, adding columns) creates undefined behaviour.

**Symptoms:** Rows are skipped, indices get out of sync, or the loop raises a `RuntimeError`.

**Fix:** Use vectorized operations instead of loops:

```python
# Vectorized — fast and safe
df = df[df["accuracy"] >= 0.9]  # Filter instead of dropping in a loop

# Or collect indices first, then drop
drop_idx = df[df["accuracy"] < 0.9].index
df.drop(drop_idx, inplace=True)
```

### Mistake 5: Forgetting That `groupby` Requires an Aggregation

```python
import pandas as pd

df = pd.DataFrame({
    "model": ["gpt-4", "gpt-4", "claude-3"],
    "accuracy": [0.94, 0.95, 0.93],
})

# This returns a GroupBy object, not a DataFrame
grouped = df.groupby("model")
print(grouped)  # <pandas.core.groupby.DataFrameGroupBy object — not useful!

# You MUST call an aggregation
result = grouped["accuracy"].mean()
print(result)
# model
# claude-3    0.93
# gpt-4       0.945
```

**Why it happens:** `groupby()` is lazy — it only records the grouping specification. No computation happens until you call `.agg()`, `.mean()`, `.sum()`, `.count()`, etc.

**Symptoms:** You print the result of `groupby()` and see a memory address. Or you try to index into it directly and get a `KeyError`.

**Fix:** Always follow `groupby()` with an aggregation method. Common ones: `.mean()`, `.sum()`, `.count()`, `.std()`, `.agg(["mean", "std"])`, `.agg({"col": "mean", "col2": "sum"})`.
