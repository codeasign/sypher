# Variables — Common Mistakes

⏱ 6 minutes

## When NOT to Use Dynamic Typing

Dynamic typing is a feature of Python, but that does not mean you should change a variable's type mid-function. Code that does this is harder to read, harder to debug, and more likely to hide bugs. Exceptions exist — the `process_prediction` function in Build It deliberately handles multiple input types at a boundary — but inside your logic, keep each variable's type stable.

## The Pattern to Remember

- **Pattern:** Assign with `=`, name with `snake_case`, convert explicitly with `int()`/`str()`/`float()`/`bool()`.
- **When to use:** Every time you store data that you need to reference later.
- **When NOT to use:** Do not reuse a variable to hold a different type mid-function — create a new variable with a clear name instead.
- **Mental model:** A variable is a sticky label on an object, not a box. Changing what the label points to is fine; changing the type of object it points to within the same logical use is not.

## Common Mistakes

### Mistake 1: Using an Unassigned Variable

**Mistake:** Referencing a variable before assigning a value to it.

```python
# Bug: count is used before assignment
print(count)  # NameError: name 'count' is not defined
count = 10
```

**Why it happens:** You assume a variable exists because it is used elsewhere in the file, but the definition has not executed yet.

**Symptoms:** `NameError` at runtime.

**Fix:** Define variables at the top of the scope where they are used. If the value depends on a condition, give the variable a default first:

```python
count = 0  # Default
if some_condition:
    count = fetch_data()
print(count)  # Always safe
```

### Mistake 2: Assuming Division Returns an Integer

**Mistake:** Writing `result = 5 / 2` and expecting `2`.

**Why it happens:** In Python 2, `/` performed integer division when both operands were integers. Python 3 changed this.

**Symptoms:** `result` is `2.5` when you expected `2`.

**Fix:** Use `//` for integer (floor) division: `5 // 2` → `2`. Use `/` for float division: `5 / 2` → `2.5`. If you need both the quotient and remainder, use `divmod(5, 2)` → `(2, 1)`.

### Mistake 3: Accidentally Shadowing a Built-in

**Mistake:** Using `list`, `str`, `int`, `dict`, `type`, `input`, `print`, `len`, etc. as a variable name.

```python
list = [1, 2, 3]        # Shadows the built-in list()
print(list([4, 5, 6]))  # TypeError: 'list' object is not callable
```

**Why it happens:** These names are natural choices for variables. Python does not prevent you from using them.

**Symptoms:** Code that worked before breaks when you add `list = [...]` somewhere above it. The error message points to a line using `list()` — now a variable, not the built-in.

**Fix:** Avoid using any name from the [list of Python built-in functions](https://docs.python.org/3/library/functions.html) as a variable name. Use `names_list`, `item_list`, or similar instead of bare `list`.

### Mistake 4: Confusing `=` (Assignment) with `==` (Equality)

**Mistake:** Writing `if x = 5:` instead of `if x == 5:`.

**Why it happens:** In other languages (C, Java), assignment is an expression that can appear inside `if`. In Python, assignment is a statement, not an expression — it cannot go inside `if`.

**Symptoms:** `SyntaxError: invalid syntax. Maybe you meant '=='` — Python explicitly tells you.

**Fix:** Use `==` for comparison. Python 3.8+ also supports the walrus operator `:=` for assignment expressions, but stick with `==` in conditions until you are comfortable with the distinction.

### Mistake 5: Assuming `is` Checks Value Equality

**Mistake:** Writing `x is 5` or `x is "hello"` instead of `x == 5` or `x == "hello"`.

**Why it happens:** `is` checks **object identity** — whether two variables point to the exact same object in memory — not whether their values are equal. Python caches small integers and some strings, so `x is 5` may appear to work for `x = 5` due to interning, but it fails for larger integers and dynamically created strings.

**Symptoms:** The comparison works in simple test cases but fails unexpectedly with larger values or different string construction methods.

**Fix:** Use `==` for value equality. Reserve `is` for comparing against `None` (`x is None` or `x is not None`) and for boolean constants (`x is True`).
