# Dataclasses — Common Mistakes

⏱ 5 minutes

## When NOT to Use Dataclasses

- **Behaviour-heavy classes:** If a class has methods that compute or transform data significantly, it may be better as a plain class with a clear purpose. Dataclasses are optimised for data storage.
- **Inheritance beyond one level:** Deep dataclass inheritance hierarchies become confusing because field ordering rules are complex.
- **Simple dicts for one-off use:** If you only pass data once within a function, a dict may be simpler.

## The Pattern to Remember

- **Pattern:** `@dataclass` → type-annotated fields → `__post_init__` for validation → `asdict()` for serialisation.
- **When to use:** Structured data containers with named fields — model predictions, configs, experiment results, API response wrappers.
- **When NOT to use:** Behaviour-heavy classes, deep inheritance, single-use dicts.
- **Mental model:** A dataclass is a struct with benefits — named fields, defaults, validation, and auto-generated methods.

## Common Mistakes

### Mistake 1: Using Mutable Defaults Without `field(default_factory=...)`

```python
@dataclass
class Bad:
    tags: list[str] = []  # ❌ Shared mutable default!

a = Bad()
b = Bad()
a.tags.append("bug")
print(b.tags)  # ['bug'] — shared between instances!
```

**Fix:** Always use `field(default_factory=list)` for mutable defaults.

### Mistake 2: Ordering Fields Incorrectly

Fields with defaults must come before fields without defaults:

```python
@dataclass
class Config:
    name: str          # No default
    model: str         # No default
    temperature: float = 0.7  # Has default — must be after non-default fields
```

Reversing this order causes `SyntaxError` or `TypeError`.

### Mistake 3: Forgetting That Fields Are Included in `__eq__` by Default

```python
@dataclass
class Prediction:
    text: str
    label: str
    confidence: float
    id: str = ""

a = Prediction("Great", "positive", 0.95, id="abc")
b = Prediction("Great", "positive", 0.95, id="xyz")
print(a == b)  # False — id differs, even though you may not want id in equality
```

**Fix:** Use `field(compare=False)` to exclude fields from equality checks.

### Mistake 4: Using `@dataclass` on a Class That Already Has a Custom `__init__`

If you define `__init__` manually on a `@dataclass`, it is overridden. Use `__post_init__` for custom initialisation logic instead.

### Mistake 5: Expecting `@dataclass(frozen=True)` to Make Nested Objects Immutable

`frozen=True` only prevents reassignment of fields on the dataclass itself, not mutation of the objects those fields reference:

```python
@dataclass(frozen=True)
class Config:
    tags: list[str]

c = Config(tags=["a"])
# c.tags = ["b"]  # FrozenInstanceError — good
c.tags.append("b")  # Works! The list is mutated, not reassigned
```

For true deep immutability, use tuples or `dataclasses.field` with `deepfrozen` alternatives from third-party libraries.
