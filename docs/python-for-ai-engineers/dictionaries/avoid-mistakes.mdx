# Dictionaries — Common Mistakes

⏱ 6 minutes

## When NOT to Use Dictionaries

Dictionaries are powerful but not universal:

- **Sequential integer keys:** If your keys are `0, 1, 2, 3...`, use a list — it is faster and more memory-efficient.
- **Ordered iteration with changes:** From Python 3.7+, dicts preserve insertion order. If you need order but also modify frequently, consider `collections.OrderedDict` for clarity.
- **Numeric computation on values:** If you have 10,000 key-value pairs of numbers and need matrix operations, use a numpy array with an index mapping.
- **Simple data transfer:** For simple named records, consider `dataclasses` or `NamedTuple` — they provide better autocomplete and attribute access.

## The Pattern to Remember

- **Pattern:** Create with `{}`, access with `[]` or `.get()`, iterate with `.items()`, handle missing keys with `defaultdict`.
- **When to use:** Any time you need to look up a value by a meaningful name rather than a numeric index.
- **When NOT to use:** Sequential keys (use list), fixed named records (use dataclass), matrix data (use numpy).
- **Mental model:** A dictionary is a labelled filing cabinet — open the drawer labelled with your key, find the value inside.

## Common Mistakes

### Mistake 1: Accessing a Missing Key with Brackets

```python
config = {"model": "gpt-4"}
print(config["temperature"])  # KeyError: 'temperature'
```

**Why it happens:** You assume the key exists. In configuration dicts with defaults, you may forget which keys are guaranteed.

**Symptoms:** Unhandled `KeyError` crashes your program at runtime.

**Fix:** Use `.get()` with a sensible default:

```python
temperature = config.get("temperature", 0.7)
```

Or set defaults upfront with `config.setdefault("temperature", 0.7)`.

### Mistake 2: Modifying a Dict While Iterating Over It

```python
config = {"model": "gpt-4", "temperature": 0.7, "max_tokens": 2048}
for key in config:
    if key.startswith("max_"):
        del config[key]  # RuntimeError: dictionary changed size during iteration
```

**Why it happens:** Python dicts track iteration state and raise an error if you add or delete keys during iteration.

**Symptoms:** `RuntimeError: dictionary changed size during iteration`.

**Fix:** Iterate over a copy of the keys:

```python
for key in list(config.keys()):
    if key.startswith("max_"):
        del config[key]
```

### Mistake 3: Using a Mutable Object as a Key

```python
data = {}
key = [1, 2, 3]  # List — mutable
data[key] = "value"  # TypeError: unhashable type: 'list'
```

**Why it happens:** Dictionary keys must be hashable. Lists, sets, and other mutable types are not hashable because their value can change, which would break the hash table.

**Symptoms:** `TypeError: unhashable type`.

**Fix:** Use a tuple instead of a list if you need a compound key:

```python
key = (1, 2, 3)  # Tuple — immutable, hashable
data[key] = "value"
```

### Mistake 4: Confusing `=` and `==` in Dict Literals

**Mistake:** Writing `{"key" = "value"}` or `dict(key = value)` incorrectly:

```python
# ❌ SyntaxError — dict literal uses : not =
# {"model" = "gpt-4"}

# ✅ Correct
{"model": "gpt-4"}

# dict() constructor with keyword args — keys are strings without quotes
dict(model="gpt-4", temperature=0.7)
```

**Why it happens:** Dict literals use `:` to separate key from value, while function keyword arguments use `=`. Beginners mix them up.

**Symptoms:** `SyntaxError`.

**Fix:** Use `:` in `{}` literals and `=` in the `dict()` constructor. The `dict()` constructor only works for string keys that are valid Python identifiers.

### Mistake 5: Assuming Dicts Are Ordered Before Python 3.7

```python
# Pre-3.7: order was NOT guaranteed
# 3.7+: insertion order IS preserved
config = {"z": 1, "a": 2, "m": 3}
print(list(config.keys()))  # ['z', 'a', 'm'] — insertion order
```

**Why it happens:** Before CPython 3.6 and Python 3.7 (language spec), dict order was an implementation detail you could not rely on. Many developers still carry this assumption.

**Symptoms:** Code that relies on insertion order runs fine in modern Python but may fail if backported or run on an alternative interpreter like PyPy.

**Fix:** For explicit ordering, use `collections.OrderedDict`. For modern Python (3.7+), standard dicts preserve insertion order and are safe to depend on.
