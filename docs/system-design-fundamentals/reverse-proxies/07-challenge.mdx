## Prove It

### Tier 1 — Foundation

Each challenge is designed to be completed in 5 minutes. Write your answer, then check against the model answer.

**Challenge 1.1: Explain TLS Termination to a Non-Technical Product Manager**

Scenario: Your product manager asks why the engineering team is spending two sprints adding a reverse proxy. They understand the basic idea ("it sits in front of servers") but do not understand why TLS termination matters. They suggest: "Why not just have each server handle its own HTTPS?"

Your task: Write a 3-sentence explanation of TLS termination that a non-technical stakeholder can understand. Use an analogy (not "asymmetric cryptography" or "certificate chain"). Then add one sentence about why offloading TLS to the proxy saves money.

**Evaluation criteria:**
- No jargon without explanation
- Clear cause-and-effect: "doing X here means Y benefits there"
- Concrete benefit mentioned (cost, speed, or security)

**Model answer:**

"Every HTTPS connection requires an expensive cryptographic handshake — similar to a security guard checking every visitor's ID before letting them in. If every backend server runs its own ID check, you need guards on every door, and each request takes longer. By putting one security checkpoint at the proxy, only the proxy bears that cost, and backend servers focus entirely on processing requests. This means we need fewer backend servers overall — a saving of roughly 30-50% on compute costs at our current scale."

**Challenge 1.2: Identify the Health Check Failure**

Scenario: You have two Nginx proxy instances behind an AWS Network Load Balancer. Each Nginx proxies to a pool of 4 backend servers. The NLB health check hits `GET /health` on each Nginx instance every 5 seconds.

This week, the /health endpoint on Nginx is configured as:

```
location /health {
    proxy_pass http://backend_pool/health;
    proxy_read_timeout 3s;
}
```

The backend /health endpoint makes a database connection to verify the DB is reachable. Yesterday, the database had a 10-second failover during maintenance. When the NLB hit the health check during those 10 seconds, the backend /health timed out (3 seconds > timeout). Nginx returned 502. The NLB marked both Nginx instances as unhealthy and removed them from rotation. All 50,000 active connections were dropped.

Identify the design flaw and propose a fix.

**Evaluation criteria:**
- Correctly identifies that the health check is too deep (tests the full stack instead of the proxy itself)
- Proposes a specific fix (change the health check endpoint)
- Explains why the original design seemed reasonable

**Model answer:**

The design flaw is that the NLB health check tests the full application stack through the proxy, not the proxy itself. The health check should verify Nginx is running and accepting connections — not that the database is reachable. When the database had a 10-second failover, the health check failed, NLB removed the Nginx instances, and 50,000 active connections were dropped. The database failure was minor (10 seconds, self-healing) but the health check design amplified it into a full service outage.

Fix: Change the Nginx health check to test a static file or a simple Nginx status endpoint:

```
location /health {
    add_header Content-Type text/plain;
    return 200 "healthy\n";
}
```

This checks only that Nginx is running and accepting requests. Database health should be monitored by a separate system (e.g., a monitoring agent that alerts but does not remove proxy instances from rotation). The original design seemed reasonable because testing the full stack catches more failures — but the cost (dropping all connections for a transient DB blip) far exceeds the benefit.

**Challenge 1.3: Calculate the Connection Pool**

Scenario: Your reverse proxy has 4 workers. Each worker maintains a keepalive pool of 32 idle connections to each of 3 backend upstreams. The backend servers can handle 200 concurrent connections each. There are 3 backend servers.

Calculate:
1. Total idle connections the proxy maintains across all workers and all backends
2. Whether the keepalive pool can saturate the backends (if all idle connections became active simultaneously)
3. Memory used by idle connections (kernel socket overhead: 3 KB per idle TCP connection)

**Evaluation criteria:**
- Correct multiplication and units
- Correct comparison between pool capacity and backend capacity
- Recognizes that idle connections are not a problem but active connections are

**Model answer:**

1. Total idle connections: 4 workers x 32 keepalive x 3 upstreams = 384 idle connections total. Per backend server: 384 / 3 = 128 idle connections.

2. If all idle connections became active simultaneously, each backend server would receive 128 active connections from the proxy pool. With a backend capacity of 200 concurrent connections each, the pool uses 128 of 200 available connections (64% utilization). The backends have headroom for additional direct connections or requests from other sources. The pool cannot saturate the backends under this configuration.

3. Memory from idle connections: 384 idle connections x 3 KB = 1,152 KB = 1.125 MB. This is negligible compared to the proxy's total memory (typically 500 MB to 2 GB). The concern with a large keepalive pool is not memory but the potential to overwhelm backends during traffic shifts. At 128 active connections per backend out of 200 capacity, the margin of safety is 72 connections (36%). This is acceptable but should be monitored.

### Tier 2 — Design

Each challenge is designed to be completed in 15 minutes. Write a structured answer with architecture diagram (your choice of format), scale justification, and trade-off analysis.

**Challenge 2.1: Design a Proxy Layer for a Video Upload Platform**

Requirements: Your platform allows users to upload videos (100 MB to 2 GB each). 1 million daily active uploaders. Peak uploads: 5,000 concurrent uploads. Each upload takes 30-120 seconds. Uploads must be encrypted in transit. The platform serves the videos for playback via a CDN.

Design the reverse proxy layer for this upload pipeline.

**Rubric:**
- Identifies that upload traffic and playback traffic have different proxy requirements (scoring: creates separate upload vs. playback proxy configurations)
- Addresses large request body handling (client_max_body_size, streaming vs. buffering) (scoring: mentions disabling proxy buffering for uploads or using chunked transfer encoding)
- Accounts for TLS cost on long-lived upload connections (scoring: mentions TLS session reuse or considers TLS overhead per byte)
- Addresses rate limiting to prevent upload abuse (scoring: proposes per-IP or per-user upload rate limits)
- Considers upload progress and resume (scoring: mentions how the proxy handles chunked uploads or range requests)
- Mentions quota management at the proxy level (scoring: distinguishes between rate limiting and bandwidth quotas)

**Model answer:**

The upload flow and playback flow have fundamentally different characteristics, so they should use separate proxy configurations — possibly separate proxy instances.

**Upload proxy configuration:**

```
# Upload-specific proxy (separate port/domain)
server {
    listen 443 ssl;
    client_max_body_size 3G;  # Allow up to 3GB uploads
    client_body_timeout 300s;  # 5 minutes for slow uploads
    
    # Critical: disable buffering for uploads
    # Otherwise the proxy buffers the entire file before sending to backend
    proxy_buffering off;
    proxy_request_buffering off;
    
    # Long timeouts for upload duration
    proxy_read_timeout 300s;
    proxy_connect_timeout 10s;
    
    # Connection pool: uploads hold connections longer
    # so we need a larger pool
    upstream upload_backend {
        server 10.0.0.1:8080;
        keepalive 128;  # Larger pool for long-lived upload connections
    }
    
    location /upload {
        # Rate limit: 2 uploads per minute per IP (prevent abuse)
        limit_req zone=upload_limit burst=5 nodelay;
        proxy_pass http://upload_backend;
    }
}
```

Key decisions:
- Disable proxy_buffering and proxy_request_buffering: for 2 GB uploads, buffering at the proxy would either require 2 GB of RAM (impossible) or significant disk I/O (slow). Stream directly to the backend.
- client_max_body_size 3G: sufficiently above the 2 GB max upload size. Large body limit is safe because rate limiting prevents abuse.
- keepalive 128: uploads hold connections for 30-120 seconds. With 5,000 concurrent uploads, we need enough pooled connections. 128 per upstream with 4 workers = 512 per backend. Each backend must handle 512 concurrent upload connections.
- Rate limit: 2 uploads/minute per IP with burst 5 prevents a single client from saturating the upload pipeline.

**Playback proxy (separate):** Simple configuration that caches frequently-watched videos (or just proxies to the CDN). Caches video metadata (title, description, thumbnail) with 5-minute TTL. No buffering for video streaming.

**Challenge 2.2: Design Gradual Proxy Deployment**

Scenario: You are migrating from an AWS ALB to a self-managed Nginx reverse proxy to reduce costs. The ALB currently handles all production traffic (50,000 requests/second). You need to migrate with zero downtime and the ability to roll back in under 60 seconds.

Design the migration strategy, including DNS, health checks, traffic shifting mechanism, and rollback plan.

**Rubric:**
- Proposes a gradual traffic shift (10% -> 50% -> 100%), not a cutover (scoring: mentions canary or weighted routing)
- Uses ALB as the front-end during migration (scoring: Nginx instances sit behind ALB, not replace it directly)
- Includes monitoring gates between traffic increases (scoring: checks error rates, latency, cache hit rate before increasing)
- Has a specific rollback trigger (scoring: "roll back if 5xx rate exceeds 1%" not "roll back if something goes wrong")
- Addresses cache warm-up (scoring: mentions cold cache causing increased backend load during initial traffic)

**Model answer:**

**Phase 1: Deploy Nginx behind ALB (0% traffic).** Provision 2 Nginx instances in a new target group. Register them with the ALB as a second target group with weight 0. Run synthetic traffic against the Nginx instances directly (using `/etc/hosts` or a test client) to verify TLS termination, routing, caching, and health checks work correctly.

**Phase 2: Cache warm-up (24 hours).** Set ALB to forward 5% of traffic to the Nginx target group (weight 5 on Nginx, weight 95 on original ALB target group). This warms the Nginx cache without noticeable impact if something goes wrong. Monitor: Nginx error rate (target: under 0.1%), latency p50/p99 (target: within 5% of ALB-only baseline), backend error rate (must not increase). Run cache-warming scripts during low-traffic hours to pre-populate hot keys.

**Phase 3: Scale up (50%).** Increase Nginx weight to 50. Add 2 more Nginx instances (total 4) to handle increased load. Monitor for 1 hour during peak traffic. Check: Nginx CPU under 60%, cache hit rate above 70%, no 502/504 increase.

**Phase 4: Full migration (100%).** Increase Nginx weight to 100. Reduce original ALB target group weight to 0 (keep the target group registered but zero weight for rollback). Monitor for 24 hours.

**Rollback plan:** If at any phase the error rate exceeds baseline by 2x for 2 consecutive minutes, set the Nginx target group weight to 0 and ALB target group weight back to 100. The ALB target group (original backends) remains registered and ready throughout the migration. Rollback is instantaneous (ALB weight change propagates within seconds). Keep the old ALB and target group for 72 hours after full migration completes.

**Key consideration:** Cache warm-up. Without phase 2, the Nginx cache starts cold, so the 50% traffic step would send 25,000 requests/second directly to backends (instead of the usual 7,500 after caching). This 3x backend load spike could overload backends. Phase 2 reduces this risk by warming the cache over 24 hours at low traffic.

**Challenge 2.3: Design a Cache Invalidation System**

Scenario: Your social media platform uses Nginx with Redis as a shared cache. You cache user profile data (name, bio, avatar URL) with a TTL of 300 seconds. When a user updates their profile, the backend writes to the database and returns the updated data. But the proxy continues serving the old data from cache for up to 300 seconds (5 minutes). Users report: "I changed my name and it still shows the old name for 5 minutes."

Design a cache invalidation system that reduces the stale window to under 1 second. The platform handles 10,000 profile updates per minute and 500,000 profile reads per minute.

**Rubric:**
- Identifies that passive TTL-based invalidation is insufficient (scoring: proposes explicit invalidation)
- Chooses an invalidation strategy (purge on write, ETag revalidation, versioned keys) (scoring: compares at least 2 strategies)
- Addresses race conditions (update in flight while invalidation arrives) (scoring: mentions locking or version counters)
- Provides concrete implementation (API endpoint, header, or Redis command) (scoring: names specific commands: PURGE, DEL, or SET)
- Considers cache stampede (scoring: mentions what happens if many requests arrive after invalidation but before cache repopulation)

**Model answer:**

Design: Use the **cache-aside with explicit purge** pattern. When a user updates their profile, the backend sends a cache purge request to the proxy before returning the response. The proxy deletes the cached entry from Redis, so the next read fetches fresh data.

**Implementation:**

1. Define a cache tag per user: `profile:user:{user_id}`.
2. On profile update (POST /api/user/profile), the backend:
   a. Writes the new profile data to the database
   b. Sends a PURGE request to Nginx: `PURGE /api/user/profile`
   c. Nginx deletes the Redis key: `DEL profile:user:123`
   d. Returns 200 to the client
3. On next GET /api/user/profile, the cache misses (key deleted), the proxy fetches from backend, caches it, and returns fresh data.

**Nginx configuration:**

```
map $request_method $purge_method {
    PURGE 1;
    default 0;
}

server {
    location /api/user/profile {
        # Normal caching
        proxy_cache my_cache;
        proxy_cache_key "profile:user:$arg_user_id";
        proxy_cache_valid 200 300s;
        proxy_pass http://backend;
        
        # Purge support
        if ($purge_method) {
            proxy_cache_purge my_cache "$scheme$request_method$host$request_uri";
        }
    }
}
```

**Race condition handling:** There is a race where a read request arrives between steps (a) and (b) — the read fetches old data from the database and caches it, overwriting the purge. To handle this, use a version field in the user profile. The backend increments a version counter on each update. The cache key includes the version: `profile:user:123:v{version}`. The proxy reads the current version from a low-latency source (Redis atomic counter) before constructing the cache key. Reads after a write always get the new version key. The old version key expires naturally via TTL.

Simpler alternative: Set a much shorter TTL for user-specific data (15 seconds instead of 300 seconds). The stale window drops to 15 seconds without explicit invalidation. This increases backend load (profile reads go from 1/300 to 1/15 of traffic hitting backends), but at 500,000 reads/minute, the increase is manageable: from 1,667 reads/minute to 33,333 reads/minute. At 10,000 updates/minute, the explicit purge approach handles 10,000 purge requests/minute on top of 500,000 reads. The 15-second TTL approach has no purge cost. Choose based on whether purge infrastructure complexity is worth the backend load savings.

### Tier 3 — Expert

Each challenge is designed to be completed in 30 minutes. These challenges have no single correct answer. The evaluation focuses on the depth of your reasoning, your awareness of trade-offs, and your ability to defend your decisions.

**Challenge 3.1: The Proxy as a Product — Design the Proxy Strategy for a Cloud Provider**

Scenario: You are the principal engineer at a new cloud provider building a managed reverse proxy service (like Cloudflare, but as a cloud provider). Your customers are small-to-medium businesses who want DDoS protection, TLS termination, caching, and a WAF — all managed, no configuration burden.

Design the proxy architecture. But here is the constraint: your cloud provider has only 10 data centers (not 310 like Cloudflare). The business team has committed to a price point of $50/month per customer, and the target is 100,000 customers in the first year.

**Discussion points to cover in your answer:**
- How do you handle DDoS without 310+ data centers? (Scrubbing centers, Anycast with 10 PoPs, sinkholing, or upstream DDoS protection from transit providers)
- How do you manage 100,000 customer TLS certificates across 10 data centers? (Certificate distribution, auto-renewal, SNI-based routing)
- How do you handle the multi-tenant cache? (Partitioned by customer, shared with per-customer quotas, or dedicated per-customer cache nodes?)
- What is the cost model per customer at $50/month? (Proxy compute, bandwidth, certificate management, storage — can you make a profit?)
- What features differentiate you from Cloudflare when you have 1/30th the PoPs? (SIEM integration, dedicated support, Kubernetes-native proxy, compliance certifications)

**Evaluation:** No single correct answer. A strong answer defines a clear set of trade-offs, acknowledges where the architecture cannot compete with Cloudflare, and identifies a niche (e.g., compliance-focused customers who need data to stay in specific geographic boundaries, supported by your 10 data centers in specific countries). A weak answer tries to match Cloudflare feature-for-feature without addressing the 10-PoP constraint.

**Challenge 3.2: Diagnose the Phantom Cache Eviction**

Scenario: You operate a reverse proxy with a 64 GB Redis cache cluster (4 x 16 GB nodes). Your application has 500 GB of hot data (frequently accessed content). The cache hit rate should be approximately 80% (64 GB / 500 GB hit rate adjusted for access frequency distribution = approximately 80%).

One morning, the cache hit rate drops from 80% to 25% over 10 minutes. It stays at 25% for 2 hours, then recovers back to 80% over 30 minutes. There is no deployment, no backend change, and no traffic spike during that window. Redis CPU and memory are normal (40% CPU, 12 GB used out of 16 GB per node). The proxy logs show no errors.

Your task: Investigate and determine the root cause. Then propose a fix.

**Surface-level hypotheses (all should be considered and ruled out):**
- Network issue between proxy and Redis? (No — Redis CPU/memory are normal, and proxy logs show no connection errors.)
- Backend response time increase causing cache write delays? (No — backend logs are normal.)
- Cache key namespace collision? (Possible but unlikely to produce an overnight step change.)

**Root cause pattern (what likely happened):**

The most likely root cause: **mass key expiration.** If a large fraction of cache keys share the same TTL and were written around the same time (e.g., a cache-warming script runs at midnight, sets 80% of keys with TTL of 5 hours), they all expire simultaneously at 5 AM. The cache hit rate drops from 80% to near-zero instantly. Redis eviction policy (LRU or allkeys-lru) becomes irrelevant because the keys expired naturally, not through memory pressure. Over the next 2 hours, new requests repopulate the cache key by key as users access different content. The hit rate gradually climbs back to 80%.

The recovery curve (30 minutes from 25% back to 80%) matches the time needed to repopulate 64 GB of cache at typical request rates. If the system processes 100,000 reads/minute at 2 KB per response, it fills 200 MB/minute, reaching 64 GB in about 5 hours. But the hit rate recovers faster because (a) the most popular content is requested more frequently, so it gets cached first, and (b) the access pattern is Zipfian — 20% of keys handle 80% of requests, so the hit rate recovers to 80% once those top 20% of keys (approximately 12.8 GB) are repopulated. With 200 MB/minute fill rate, 12.8 GB takes about 64 minutes — consistent with the observed 30-minute to 2-hour window.

**Fix:** Add jitter to cache TTLs. Instead of setting a single TTL for all items from a cache-warming script, randomize each key's TTL by +/- 20% of the configured value. For a 5-hour TTL (18,000 seconds), randomize to 14,400-21,600 seconds. This spreads expiration across the window, preventing mass simultaneous expiration. Additionally, implement stale-while-revalidate on cache misses: serve the stale value (0-60 seconds old) while asynchronously refetching, masking the expiration event from users.

**Alternative pattern:** Something else could cause this: a Redis cluster resharding event (moving keys between nodes takes them temporarily unavailable), a Redis failover (promoting a replica with cold cache), or a network partition that caused the proxy to fall back to per-worker memory cache (much smaller, lower hit rate). Each has distinct diagnostic signatures (Redis logs show failover, proxy logs show connection shifts to different Redis nodes). A strong answer enumerates and rules out alternatives systematically.
