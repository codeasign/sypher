## Naive Implementation — Single Region

### 1,000 Users: Single Region, Single Instance

A single-region deployment with one application server and one database serves every user. DNS points to the single load balancer IP. The application reads and writes to the same database. At 1,000 users and 100 requests per second, this works flawlessly. Latency for local users is under 10 milliseconds. Users in other countries wait 100 to 300 milliseconds, but the traffic is low enough that no one notices.

### 100,000 Users: Single Region, Scaled Up

The application grows. Multiple application servers behind the load balancer handle the load. The database is scaled up to a larger instance with read replicas within the same region. The architecture is robust within us-east-1 but still has a single geographic point of failure. A region-wide outage takes everything down.

### 10 Million Users: Multi-Region Becomes Mandatory

At 10 million users, the single-region architecture breaks. Regional latency makes the application unusable for users far from us-east-1. Read replicas in other regions reduce read latency, but writes still travel to the primary region. The architecture must extend to multiple regions, adding geo-DNS, cross-region replication, and failover capabilities.

## Production-Grade Architecture

A production-grade multi-region architecture at 10 million users requires two or more regions with active-active or active-passive configuration, automated failover, and comprehensive observability.

### Key Design Decisions

**Region count**: Three regions are the minimum for true multi-region resilience. Two regions can experience a split-brain scenario where neither can determine which should be primary. Three regions provide a quorum: if one region loses connectivity, the other two maintain quorum and continue operating. The third region also provides a failover target if the primary region fails and the second region needs to remain as a backup.

**Database topology**: The architecture uses an active-passive database topology with a single primary in us-east-1 and read replicas in eu-west-1 and ap-southeast-1. This is the safest approach for transactional workloads. The read replicas serve local reads and can be promoted to primary if us-east-1 fails. The replication lag is monitored and alerted on.

**Caching strategy**: Each region has its own local cache cluster. Cache data is not replicated across regions because caching is latency-sensitive and the cache hit rate benefits from local traffic patterns. Cache invalidation is handled by setting short TTLs (60 to 300 seconds) rather than cross-region invalidation messages. For globally invalidated data (deleted items, policy changes), an invalidation message is published to an SNS topic that each region subscribes to.

**Static assets**: S3 buckets with cross-region replication store static assets. CloudFront CDN caches assets at edge locations worldwide, providing sub-10ms delivery for most users regardless of the origin region.

## Integration Patterns

### Pattern 1: Global HTTP Session Sharing

Applications that require sticky sessions across regions need a global session store. The session is written to the local region's cache and also replicated to a global cache layer.

```yaml
# session-store-config.yaml
session:
  local_cache:
    engine: redis
    ttl: 3600
    region: auto  # uses local region's ElastiCache
  global_replication:
    engine: redis_global_datastore
    replication_type: active_active
    regions:
      - us-east-1
      - eu-west-1
      - ap-southeast-1
  consistency:
    read: local_first  # try local, fall back to global if miss
    write: local_and_global  # write to both, async global sync
```

The session data is small (under 4 KB per session), so the replication overhead is minimal. Redis Global Datastore provides active-active replication with sub-second propagation. Session reads are local in under 1 millisecond; writes are local plus async global. The fallback from local cache miss to global read adds approximately 200 milliseconds for cross-region reads but affects only cold sessions or sessions that migrated between regions.

### Pattern 2: Global Queue for Cross-Region Workflow

Workflows that span regions — processing orders placed in one region and fulfilled in another — use a global queue service.

```
Order placed in ap-southeast-1
         │
         ▼
  ┌──────────────┐     ┌──────────────┐
  │ SQS (local)  │────▶│ Order        │  Validate in local region
  │ ap-southeast-1│    │ Processor    │  (check inventory, auth)
  └──────────────┘     └──────┬───────┘
                              │
                    Order requires fulfillment from
                    us-east-1 warehouse
                              │
                              ▼
  ┌──────────────┐     ┌──────────────┐
  │ SQS (global) │────▶│ Fulfillment  │  Process in us-east-1
  │ us-east-1    │     │ Worker       │  (pick, pack, ship)
  └──────────────┘     └──────────────┘
```

The global queue is implemented using a region-specific queue that the processing logic in each region reads from. The order message includes a region attribute that indicates where the fulfillment should happen. The fulfillment worker in the target region picks up the message and processes it. This decouples the regions and allows each region to operate independently.

### Pattern 3: Global Read After Local Write

A user writes data in their local region. The write is forwarded to the primary region and acknowledged. The response includes a timestamp or version number. When the user immediately reads the data, the application includes this timestamp in the read request. The local replica compares the timestamp against its replication watermark. If the replica is behind, the read is redirected to the primary region.

```python
def read_with_consistency(user_id, record_id, last_write_timestamp):
    replica_lag = get_replication_lag(local_region())
    current_time = current_timestamp()
    estimated_replica_time = current_time - replica_lag

    if last_write_timestamp and last_write_timestamp > estimated_replica_time:
        # Replica may not have the write yet - read from primary
        return read_from_primary(record_id)
    else:
        # Replica is guaranteed to have the write
        return read_from_local_replica(record_id)
```

This pattern gives users read-after-write consistency for their own data without reading all data from the primary. It requires reasonably accurate clock synchronization between regions (within 100 milliseconds, achievable with NTP).

### Pattern 4: Regional Failover with Canary Traffic

Before a full failover, route a small percentage of traffic to the standby region and verify that the system handles it correctly.

```yaml
# failover-canary-config.yaml
failover:
  canary:
    enabled: true
    initial_traffic_percentage: 1
    steps:
      - percentage: 1
        duration: 300s  # 5 minutes
        verification: [error_rate < 0.1%, latency_p99 < 500ms]
      - percentage: 5
        duration: 300s
        verification: [error_rate < 0.1%, latency_p99 < 500ms]
      - percentage: 25
        duration: 600s  # 10 minutes
        verification: [error_rate < 0.1%, latency_p99 < 500ms]
      - percentage: 100
        duration: 0  # complete failover
    rollback:
      trigger: any_verification_fails
      action: immediate_rollback_to_previous_region
```

The canary pattern catches issues like database capacity, cache miss storms, and configuration drift before they affect all users. If the canary detects elevated error rates, the failover is rolled back automatically.

## Multi-Region and Global Scale

### Global Traffic Routing Strategies

Beyond simple geo-DNS, global traffic routing can use latency-based routing, weighted routing, or proximity routing:

| Strategy | Mechanism | Use Case | Failover Speed |
|----------|-----------|----------|----------------|
| Geo-DNS | IP geolocation table | Compliance, content localization | 60-300s (DNS TTL) |
| Latency-based | Health check probes measure latency | Performance optimization | 60s (health check interval) |
| Weighted | Configurable percentage distribution | Canary deployments, A/B testing | Instant (config change) |
| Anycast | BGP routing to nearest PoP | CDN, DNS, global load balancing | Sub-second (BGP convergence) |
| Client-side | Application code selects region | SDK-based routing, custom logic | Instant (application logic) |

### Data Residency and Compliance

Multi-region architecture must respect data residency laws. GDPR requires that EU personal data stays in EU regions. The China Cybersecurity Law requires that Chinese user data stays in China. These constraints mean that certain users must be routed to specific regions regardless of latency.

Implementation approach: The geo-DNS layer checks the user's geolocation. If the user is in the EU, DNS returns the IP of the EU region's load balancer. If the EU region is unhealthy but the user is in the EU, the system must either fail over to another EU region (if available) or deny service rather than route the user to a non-EU region. The application layer also enforces data residency: the EU application instances are configured to never write EU user data to non-EU databases.

### Compute Estimation

For a global application with 10 million daily active users, each user making 20 requests per day, with 30 percent of traffic during peak hour:

```
Total daily requests: 10,000,000 × 20 = 200,000,000
Peak hour requests: 200,000,000 × 0.3 = 60,000,000
Peak requests per second: 60,000,000 / 3600 = 16,667 RPS

Per-region RPS (active-active, 50/50 split):
  Region A: 8,333 RPS
  Region B: 8,334 RPS

Per-region compute (each app instance handles 500 RPS):
  Instances per region: 8,334 / 500 = 17 instances
  Buffer for failover (50%): 17 × 1.5 = 26 instances
  Auto-scaling range: 10 to 40 instances per region
```

### Database Estimation

For a write-heavy workload where 10 percent of requests are writes:

```
Writes per second (global): 16,667 × 0.1 = 1,667 writes/s
Primary region writes (active-passive): 1,667 writes/s
Replication throughput: 1,667 writes/s × 2 KB per write = 3.3 MB/s
Database storage growth: 1,667 × 2 KB × 86,400 = 288 GB/day
Annual growth: 288 GB × 365 = 105 TB/year
```

### Network Bandwidth

```
Cross-region replication traffic: 3.3 MB/s = 26 Mbps
Cross-region request forwarding (writes): 1,667 RPS × 10 KB = 16.7 MB/s = 133 Mbps
CDN egress: 10,000,000 users × 1 MB per user per day = 10 TB/day = 93 Mbps average
  Peak CDN egress: 93 Mbps × 3 = 279 Mbps
Total cross-region bandwidth: 26 + 133 = 159 Mbps
Total CDN bandwidth: 279 Mbps
```

### Cost Breakdown

```
Monthly compute (2 regions, 26 instances × 2 = 52 instances, r5.xlarge at $0.252/hr):
  52 × $0.252 × 730 = $9,565

Monthly database (1 primary db.r5.xlarge + 2 read replicas db.r5.xlarge):
  ($1.17 × 730) × 3 = $2,562

Monthly cache (2 Redis clusters, cache.r5.large):
  ($0.295 × 730) × 2 = $431

Monthly data transfer (cross-region replication):
  159 Mbps × 730 hours × 30 days / 8 = 4.5 PB → approximately $4,500
```

This is approximately 3x the cost of a single-region deployment at the same scale. The premium covers the infrastructure duplication, cross-region data transfer, and the additional complexity of maintaining multiple environments.

### Deployment Strategy

Deploying to multiple regions requires a phased rollout to prevent global outages:

1. **Deploy to region 1 (us-east-1)**: Apply infrastructure changes, deploy application code. Run smoke tests against the updated region. Monitor for 10 minutes.

2. **Verify region 1**: Check error rates, latency, database connections, and cache hit rates. If any metric deviates from baseline by more than 10 percent, abort the rollout.

3. **Deploy to region 2 (eu-west-1)**: Repeat the deployment process. Run smoke tests. Monitor for 10 minutes.

4. **Deploy to region 3 (ap-southeast-1)**: Repeat the deployment process. This is the last region. If the deployment causes issues, roll back region 3 first, then region 2.

5. **Post-deployment verification**: Run a global health check suite that tests all regions. Verify that cross-region replication is still below the latency threshold.

### Database Schema Migrations

Multi-region database migrations require careful sequencing:

| Migration Type | Approach | Risk |
|---------------|----------|------|
| Add column (nullable) | Apply to all regions simultaneously | Low — old columns remain functional |
| Add column (non-nullable) | Add as nullable first, populate data, then add NOT NULL constraint | Medium — requires backfill |
| Rename column | Add new column, dual-write, migrate reads, drop old column | High — long transition window |
| Remove column | Mark as deprecated, stop writing, verify no reads, then drop | Medium — monitoring required |
| Add index | Apply to primary, wait for replication to catch up, then apply to replicas | Low — may cause replication lag |

### Operational Runbooks

**Failover test**: Every month, the on-call engineer initiates a planned failover. The standby region is promoted to primary. All traffic is routed to the new primary for 24 hours. Then the original primary region is promoted back. This ensures that the failover process works and that the team is familiar with the procedure.

**Recovery test**: Every quarter, the team simulates a full region outage. The region is taken offline. The failover process is triggered. The team measures the actual RTO and RPO and compares against the SLO targets. Any gaps are documented and addressed.

**Replication lag incident**: When replication lag exceeds 5 seconds, the on-call engineer investigates the cause. Common causes include large batch operations on the primary, network congestion, or insufficient bandwidth on the replication channel. The engineer either throttles the cause or scales up the replication channel.

**DNS propagation monitoring**: After any failover, the team monitors DNS propagation globally using a distributed health check service. If DNS propagation is incomplete after 5 minutes, the TTL-forcing mechanism is triggered (where the application sends a response header instructing clients to use the new DNS record).
