## Company Case Studies

### Netflix — Global Streaming at 200+ Regions

Netflix operates in over 200 countries and territories through a custom multi-region architecture built on AWS. Rather than deploying application stacks in every AWS region, Netflix uses a combination of three primary AWS regions (us-east-1, eu-west-1, ap-southeast-1) for control plane operations and a distributed CDN for content delivery.

The core insight: Netflix separates the **control plane** (user signup, billing, recommendations, search) from the **data plane** (video streaming). The control plane runs in three active-passive regions. The data plane uses a custom CDN called Open Connect, which places caching appliances directly inside ISP networks worldwide. Over 95 percent of Netflix traffic is served from these Open Connect appliances, meaning multi-region architecture applies to the management interface, not the video bits themselves.

**Key numbers**: Netflix serves 260 million subscribers. The control plane handles approximately 10,000 requests per second during peak hours. Cassandra is used for cross-region data replication with tunable consistency. Zookeeper manages cluster coordination. The Chaos Monkey tool regularly kills infrastructure components to verify resilience.

**Lessons learned**: Automate everything. Netflix's Spinnaker CI/CD pipeline deploys to multiple regions with canary analysis. The Eureka service registry enables regional failover at the service discovery level. Every component is designed with graceful degradation — if the recommendation service in a region fails, users still get a curated homepage, just without personalization.

### Uber — Regional Availability Zones as Regions

Uber operates a global ride-sharing platform. Their multi-region story is unique because each geographic area has specific regulatory and operational requirements. Rather than replicating global state, Uber uses a regional architecture where each city or metro area operates as a semi-independent unit.

Uber's core data is highly location-specific. A trip in London involves London drivers, London pricing rules, London regulatory requirements, and a London database. There is no reason to replicate London trip data to Singapore. Uber's architecture is essentially regional sharding with a global overlay for user accounts and payment profiles.

**Key numbers**: Uber operates in over 10,000 cities across 70 countries. Each city runs its own ringpop-based application cluster. The global services (user accounts, payments) run in three AWS regions with active-passive failover. Regional services run in the cloud region closest to the city, typically collocated with the city's compute.

**Lessons learned**: Not all data needs to be global. Uber's region-specific sharding means no cross-region replication for trip data, eliminating consistency concerns for the most latency-sensitive operations. The trade-off is that a user traveling from London to Tokyo cannot see their London trip history while in Tokyo — it requires a cross-region query. This is acceptable for Uber because the primary use case is current trips, not historical lookups.

### Discord — From Single Region to Global Real-Time

Discord started as a single-region application on GCP. As the platform grew to hundreds of millions of users, the latency for non-US users became a critical problem. Discord's multi-region journey is documented in detailed engineering blog posts.

Discord's architecture uses an active-passive approach with a single primary region and read replicas in multiple regions. The critical design decision was their data model: Discord uses Cassandra for message storage, which has built-in multi-region support. Each message is stored with a timestamp and channel ID as the primary key, enabling efficient range queries regardless of region.

### Cloudflare — Global Anycast Network

Cloudflare operates a truly global infrastructure that spans over 330 cities in 120 countries. Their architecture is fundamentally multi-region by design — every data center runs the same software stack and can serve any request. This is achieved through an anycast network where the same IP address is advertised from every data center, and BGP routing directs each user to the nearest available location.

Cloudflare's multi-region insight: by making every location stateless (or at least able to serve any request from any location), failover becomes automatic and instantaneous at the BGP routing level. There is no failover — the traffic simply routes to the next closest data center.

**Key numbers**: Cloudflare's network processes an average of 60 million HTTP requests per second. Durable Objects provide strongly consistent, globally distributed state. R2 storage replicates data across data centers with automatic failover.

**Lessons learned**: Anycast routing is the most elegant multi-region approach because failover happens at the network layer without application involvement. The cost is that every data center must be able to handle any request, requiring either stateless design or a low-latency global state layer. Cloudflare's approach is ideal for a reverse-proxy/CDN use case but harder to apply to general-purpose application hosting.

## Open Source Implementations

| Technology | Multi-Region Approach | Replication | Consistency Model |
|------------|----------------------|-------------|-------------------|
| Cassandra | Active-active across any number of regions | Gossip-based, configurable replication factor per DC | Tunable (ONE, QUORUM, ALL per operation) |
| CockroachDB | Active-active with automatic failover | Synchronous via Raft consensus | Strongly consistent (serializable) |
| Spanner (CP) | Active-active with external clock | TrueTime + Paxos | Strongly consistent (external consistency) |
| Redis Enterprise | Active-active with CRDT-based conflict resolution | CRDT-based synchronization | Eventually consistent (CRDT guarantee) |
| Kafka MirrorMaker | Active-passive cross-region topic replication | Async, offset-based | At-least-once, bounded lag |
| Patroni (PostgreSQL) | Active-passive with automated failover | Streaming replication | Strongly consistent per region, eventual cross-region |

### Cassandra — Open Source Active-Active Pioneer

Cassandra was designed from the ground up for multi-datacenter deployment. Each data center (region) operates independently with its own replication factor. Writes are acknowledged when the local data center's replication factor is satisfied. Cross-datacenter replication happens asynchronously via hinted handoff and gossip. The trade-off is eventual consistency across regions, but within a region, consistency is tunable per query.

### CockroachDB — Strong Consistency Across Regions

CockroachDB takes the opposite approach: strong consistency everywhere through synchronous Raft consensus that spans regions. A write is acknowledged only after a majority of replicas across regions have committed it. This provides serializable isolation, the strongest consistency level, but adds 80 to 200 milliseconds of write latency for accessing replicas in other regions. The latency cost is the price of strong consistency.

### Patroni for PostgreSQL — Widespread Active-Passive

Patroni is the most common open-source tool for PostgreSQL high availability and multi-region failover. It manages leader election, replication, and automated failover using a DCS (Distributed Consensus Store) like etcd or Consul. A typical deployment has a primary in one region, a synchronous replica in a different availability zone within the same region, and an asynchronous replica in a second region for disaster recovery.

## Cloud Provider Implementations

### AWS

AWS offers the most comprehensive multi-region toolset. Route53 provides geo-DNS and latency-based routing with health checks. AWS Global Accelerator provides anycast IP addresses with traffic routing to the nearest healthy region. DynamoDB Global Tables provide active-active replication across up to 12 regions with last-writer-wins conflict resolution. Aurora Global Database provides active-passive replication across up to 5 regions with under 1 second replication lag.

The AWS multi-region pattern typically follows: Route53 for DNS, Global Accelerator for network path optimization, DynamoDB Global Tables or Aurora Global Database for data, and S3 Cross-Region Replication for object storage.

### GCP

GCP's global infrastructure is built on a single private fiber network that connects all regions. This means inter-region latency is typically 30 to 50 percent lower than AWS for the same geographic distances. Cloud Load Balancing is a global anycast load balancer that routes traffic to the nearest healthy backend. Cloud Spanner provides strongly consistent global database. Bigtable provides multi-region replication for analytical workloads. Firestore provides multi-region document storage with strong consistency.

### Azure

Azure's multi-region approach centers on Azure Traffic Manager for DNS-based routing, Azure Front Door for global HTTP load balancing with WAF, and Cosmos DB for multi-region database with multiple consistency levels. Cosmos DB is unique in offering five consistency levels from strong to eventual, each with specific latency and throughput characteristics. Azure also offers SQL Database geo-replication for relational workloads.

## Lessons Learned

**Start with read replicas, not active-active.** Active-active sounds appealing because it eliminates failover time, but the conflict resolution complexity is underestimated. Every engineering team that has attempted active-active for transactional workloads reports that conflict resolution consumed far more development time than anticipated. Start with read replicas in multiple regions. Add active-active only for specific data categories where zero failover time is business-critical.

**Replication lag is the single most important metric.** Monitor it from every secondary region to the primary. Track it as p50, p99, and p999. Alert on sustained increases. A lag spike of more than 5 seconds during normal operations indicates a systemic issue — under-provisioned replication bandwidth, large batch operations, or query contention on the primary. Fix these before they cause a data loss incident during failover.

**Test failover monthly, not quarterly.** The failover process that worked three months ago may not work today because of configuration changes, schema migrations, or network topology updates. Automated failover testing eliminates the fear of the procedure. The team that fails over every month treats it as routine. The team that fails over once a year treats it as a crisis — and makes mistakes under pressure.

**Multi-region does not replace multi-AZ.** Running in three regions with a single availability zone in each is less resilient than running in one region with three AZs. AZ failures are more common than region failures. A proper architecture uses both: multiple AZs within each region for intra-region resilience and multiple regions for geographic disaster recovery.

**Accept the cost of data duplication.** Multi-region architecture inherently requires 2x to 3x infrastructure cost. Attempting to optimize this away — by running standby regions at 10 percent capacity, or sharing resources between environments — undermines the resilience benefits. Budget for the full cost. If the budget does not support it, multi-region is not the right choice.

**Document the data flow for every data type.** For user sessions: stored in local cache, TTL of 30 minutes, lost on failover (users re-authenticate). For product catalog: replicated via S3 cross-region, 5 minute synchronization window, stale reads acceptable. For financial transactions: always routed to primary region, strong consistency required, idempotency keys enforced. This documentation becomes the source of truth for understanding what breaks during each failure scenario.
