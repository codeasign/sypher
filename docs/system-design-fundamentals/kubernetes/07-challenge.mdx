## Tier 1: Foundation

### Challenge 1: Write a Deployment Manifest

**Prompt**: Write a Kubernetes Deployment YAML that runs three replicas of an nginx container serving on port 80. The deployment must perform a rolling update with one unavailable pod at any time. Each pod must request 256 MB of memory and limit to 512 MB. Add an environment variable `NGINX_HOST` with value `example.com`. The deployment must include a liveness probe that checks the HTTP endpoint `/healthz` every 15 seconds with a timeout of 5 seconds and a readiness probe that checks the same endpoint every 10 seconds.

**Model Answer**:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
        env:
        - name: NGINX_HOST
          value: "example.com"
        resources:
          requests:
            memory: "256Mi"
          limits:
            memory: "512Mi"
        livenessProbe:
          httpGet:
            path: /healthz
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 15
          timeoutSeconds: 5
          failureThreshold: 3
        readinessProbe:
          httpGet:
            path: /healthz
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 10
          timeoutSeconds: 3
          successThreshold: 1
          failureThreshold: 3
```

**Key points**: The `maxUnavailable: 1` ensures at least 2 pods are always serving. The deployment uses RollingUpdate strategy (default) but explicitly sets the parameters. Resource requests and limits are properly scoped. Probes use different periodSeconds for appropriate detection frequency. The liveness probe has a longer timeout (5 seconds) to avoid false positives from transient slowness.

### Challenge 2: Expose the Deployment with a Service

**Prompt**: Write a Kubernetes Service YAML that exposes the nginx deployment from Challenge 1 internally within the cluster on port 80, and externally through a cloud load balancer on port 443 (with TLS termination at the load balancer). The Service should use TCP protocol and target container port 80. Describe how a client inside the cluster discovers and connects to this Service.

**Model Answer**:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-ssl-cert: "arn:aws:acm:us-east-1:123456789012:certificate/abc-123"
    service.beta.kubernetes.io/aws-load-balancer-ssl-ports: "443"
spec:
  type: LoadBalancer
  ports:
  - name: http
    port: 80
    targetPort: 80
    protocol: TCP
  - name: https
    port: 443
    targetPort: 80
    protocol: TCP
  selector:
    app: nginx
```

**How internal clients discover the Service**: Clients inside the cluster use DNS resolution through CoreDNS. The DNS name follows the pattern `<service-name>.<namespace>.svc.cluster.local`. For this example, `nginx-service.default.svc.cluster.local` resolves to the Service's ClusterIP (e.g., 10.96.0.50). The kube-proxy on each node configures iptables rules that DNAT traffic destined for the ClusterIP to a randomly selected healthy pod IP. The EndpointSlice controller continuously updates the list of healthy pod IPs based on readiness probe results.

**Key points**: The Service type `LoadBalancer` creates both a ClusterIP and an external load balancer. The annotation configures TLS termination at the load balancer level. The selector must match the pod labels from the Deployment. The targetPort maps the Service port to the container port.

### Challenge 3: Configure a ConfigMap and Mount It as a File

**Prompt**: Write a ConfigMap that stores an nginx configuration file (`nginx.conf`) with a custom server block listening on port 80, proxying requests to a backend service at `backend-service:8080`. Then write a Deployment that mounts this ConfigMap as a file at `/etc/nginx/nginx.conf`. Explain how ConfigMap updates propagate to running pods.

**Model Answer**:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    server {
        listen 80;
        server_name example.com;

        location / {
            proxy_pass http://backend-service:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-custom
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-custom
  template:
    metadata:
      labels:
        app: nginx-custom
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        volumeMounts:
        - name: config-volume
          mountPath: /etc/nginx/nginx.conf
          subPath: nginx.conf
      volumes:
      - name: config-volume
        configMap:
          name: nginx-config
```

**How ConfigMap updates propagate**: When a ConfigMap is updated, the kubelet detects the change through its watch on ConfigMap resources. For ConfigMaps mounted as volumes, the kubelet updates the symlink target in the pod's filesystem (the ConfigMap volume uses a symlink-based atomic swap). The update propagation time is approximately 60-90 seconds due to the kubelet sync period and the watch event delivery delay. Applications do not automatically reload their configuration when the file changes, so you need a mechanism (sidecar reloader, SIGHUP signal, or pod restart) to apply the new configuration.

**Key points**: The `subPath` field mounts a single key from the ConfigMap instead of creating a directory with all keys. ConfigMap updates are eventually consistent and may take up to 2 minutes to propagate. For critical configuration changes, use a rolling update of the Deployment rather than relying on in-place ConfigMap updates.

## Tier 2: Design

### Challenge 4: Design a Multi-Tenant Cluster

**Prompt**: Design a Kubernetes cluster for 10 engineering teams, each deploying 5-20 services. The teams must be isolated from each other: Team A should not be able to read or modify Team B's resources. Each team should have a resource quota of 16 CPU cores and 64 GB memory. Define the RBAC roles, namespace structure, network policies, and resource quotas. Explain how a new team member gains access to their team's namespace.

**Rubric**:

| Criterion | Pass | Exemplary |
|---|---|---|
| Namespace isolation | Creates one namespace per team | Creates separate namespaces with labels for policy targeting |
| RBAC structure | Defines one Role per namespace | Defines Role (namespace-scoped) and ClusterRole (read-only access to cluster resources) |
| Network policy | Mentions network policies but no default-deny | Defines default-deny ingress, then allows intra-team traffic and denies cross-team traffic |
| Resource quota | Sets CPU and memory quotas per namespace | Sets CPU, memory, persistent storage, and object count quotas with scopes |
| Onboarding process | Manual kubectl apply for each user | Describes automated onboarding through a service account + group-based RoleBinding |

**Model Answer**: Create 10 namespaces: `team-a`, `team-b`, etc. Each namespace has a `ResourceQuota` limiting CPU to 16 cores, memory to 64 GB, pods to 50, and services to 20. A `NetworkPolicy` with `podSelector: {}` and `policyTypes: [Ingress]` denies all ingress traffic by default for each namespace. A second `NetworkPolicy` allows ingress from pods within the same namespace. For each team, create a `Role` with `get, list, watch, create, update, patch, delete` permissions on pods, services, deployments, configmaps, and ingresses. Bind the role to a team-specific group (e.g., `team-a-developers`) via a `RoleBinding`. A new team member is added to the group in the identity provider (Azure AD, Okta) and automatically has access through the existing RoleBinding.

### Challenge 5: Design a Canary Deployment Strategy

**Prompt**: Design a Kubernetes-based canary deployment strategy for a service called `checkout-api`. The service runs 10 production replicas. You want to deploy version 2.0 with the following rollout plan: send 5 percent of traffic to v2.0 for 30 minutes, then 25 percent for 1 hour, then 50 percent for 2 hours, then 100 percent. Implement rollback criteria: if error rate exceeds 0.1 percent or p99 latency exceeds 500 ms for 2 consecutive minutes, roll back immediately. Describe your implementation and how you measure rollback conditions.

**Rubric**:

| Criterion | Pass | Exemplary |
|---|---|---|
| Traffic splitting | Uses two Services with selectors for each version | Uses a single Service with label weighting or a service mesh (Istio VirtualService) |
| Monitoring | Manual observation of metrics | Automated metrics collection with alerting rules |
| Rollback trigger | Manual rollback when someone notices errors | Automated rollback with a controller or progressive delivery tool |
| Rollout timeline | Mentions gradual traffic increase | Specifies exact traffic percentages, duration, and verification gates for each step |
| Tooling | Describes manual kubectl commands | References Argo Rollouts, Flagger, or Istio for progressive delivery |

**Model Answer**: Use Argo Rollouts for progressive delivery. The rollout creates a canary ReplicaSet with 1 replica (5 percent of 10 + 1 surge). An Istio VirtualService splits traffic 5/95 between v2.0 and v1.0. The rollout pauses for 30 minutes while monitoring error rate and latency from Prometheus. If metrics stay within thresholds, Argo promotes to 25 percent (3 canary replicas) for 1 hour. If metrics exceed thresholds at any point, Argo rolls back by redirecting all traffic to v1.0 and scaling down the canary. After 50 percent for 2 hours, the rollout promotes to 100 percent and scales down the old ReplicaSet. The rollback is automatic through Argo's analysis template that queries Prometheus and evaluates the success condition.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: checkout-api
spec:
  replicas: 10
  strategy:
    canary:
      steps:
      - setWeight: 5
      - pause: {duration: 30m}
      - setWeight: 25
      - pause: {duration: 1h}
      - setWeight: 50
      - pause: {duration: 2h}
      - setWeight: 100
      analysis:
        templates:
        - templateName: error-rate-check
        - templateName: latency-check
```

### Challenge 6: Design a Stateful Database Deployment

**Prompt**: Design a Kubernetes deployment for PostgreSQL that runs 3 replicas in a primary-replica configuration. The primary handles writes, the two replicas handle read-only queries. Each instance requires 100 GB of persistent storage on SSDs. The deployment must survive the loss of any single node without data loss. Describe how failover works, how replicas discover the primary, and how you handle backups. What happens when the primary node fails?

**Rubric**:

| Criterion | Pass | Exemplary |
|---|---|---|
| Kubernetes resources | Creates a StatefulSet and Service | Creates StatefulSet with headless Service, plus a separate read-only Service for replicas |
| Storage | Mentions PersistentVolumeClaims | Uses StorageClass with SSD provisioner, volumeClaimTemplates for automatic PVC creation |
| Failover | Manual failover procedure | Automated failover through an Operator (Crunchy Data, Zalando, CloudNativePG) |
| Backup | Mentions pg_dump | Describes WAL archiving to S3 with Point-in-Time Recovery |
| Failure scenario | Says "pod restarts" | Describes the complete sequence: node failure → pod eviction after 5 min → StatefulSet recreates pod with same PVC → pod starts with existing data |

**Model Answer**: Use the CloudNative PostgreSQL Operator. The operator manages the StatefulSet with 3 replicas, a headless Service for stable DNS names (`pg-0`, `pg-1`, `pg-2`), and a separate read-only Service. Each pod has a 100 GB PVC using a SSD storage class (e.g., `gp3` on AWS EKS). The operator handles automated failover: if `pg-0` (primary) becomes unreachable, the operator promotes `pg-1` to primary. The other replicas update their replication configuration to follow the new primary. Applications write to the primary Service (`pg-rw`) and read from the read-only Service (`pg-ro`). WAL archiving runs as a sidecar container, streaming WAL segments to S3 every 5 minutes. Point-in-Time Recovery restores from the S3 archive with a specified timestamp. When the original primary node fails, Kubernetes evicts pods after 5 minutes. The StatefulSet recreates `pg-0` on a healthy node with its original PVC (preserving the WAL). The new `pg-0` connects as a replica of the current primary.

## Tier 3: Expert

### Challenge 7: Design a Control Plane Failure Recovery Procedure

**Prompt**: You are the platform engineer on call. All three control plane nodes in your production Kubernetes cluster become unresponsive simultaneously due to a regional network outage that lasted 15 minutes. When the network recovers, the control plane nodes come back but etcd refuses to start, showing the error `etcdserver: mvcc: database space exceeded`. The etcd database is 9.2 GB (exceeding the 8 GB default quota). The last etcd backup is 4 hours old. Your cluster runs 200 production services serving 5 million users. Design the recovery procedure step by step.

**Model Answer**: The recovery proceeds in 5 phases:

**Phase 1: Restore etcd from backup (30 minutes)**

1. Stop all control plane components on all three nodes.
2. On the node that will become the restored leader, restore the latest etcd snapshot: `ETCDCTL_API=3 etcdctl snapshot restore /backups/etcd-snapshot.db --data-dir=/var/lib/etcd-restore --initial-cluster-token=etcd-restore-token`.
3. Configure the restored node as a single-member cluster: `--initial-cluster=default=http://localhost:2380 --initial-advertise-peer-urls=http://localhost:2380`.
4. Start etcd on this node. Verify it is healthy: `ETCDCTL_API=3 etcdctl endpoint health`.
5. Start the API server, scheduler, and controller manager on this node, pointing to the local etcd.
6. Verify the API server responds: `kubectl get nodes`.

**Phase 2: Add the other two etcd members (15 minutes)**

7. Add the second etcd member as a learner: `etcdctl member add node2 --learner --peer-urls=http://node2:2380`.
8. Start etcd on node2 with the same `--initial-cluster-token`. It joins as a learner and catches up.
9. Promote the learner to a voting member: `etcdctl member promote node2`.
10. Repeat for node3.
11. Verify the cluster has a healthy 3-node majority.

**Phase 3: Recover control plane components (15 minutes)**

12. Start the API server on all three control plane nodes, each pointing to the local etcd instance.
13. Place the control plane nodes behind the load balancer.
14. Verify the API server responds to health checks.
15. Start the scheduler and controller manager on all three nodes.

**Phase 4: Verify cluster health (15 minutes)**

16. Verify all nodes are present: `kubectl get nodes`. If nodes show as `NotReady`, wait for the kubelet heartbeats to resume.
17. Verify all pods are running: `kubectl get pods --all-namespaces`. Expect some pods to be in `Unknown` state (from the backup being 4 hours stale).
18. Verify critical services: CoreDNS, kube-proxy, CNI DaemonSet, Ingress controller.

**Phase 5: Reconcile stale state (60 minutes)**

19. The backup is 4 hours old. Any Deployments, ConfigMaps, Secrets, or Services created or updated in the last 4 hours are missing. Coordinate with service owners to re-apply recent changes.
20. Pods that were created in the last 4 hours are now orphaned (they run but etcd does not know about them). The kubelet on each node detects the orphaned pods (pods running locally but not in the API server) and terminates them. The controllers recreate the pods.
21. Restart etcd backups with a shorter interval (every 30 minutes instead of 4 hours) and enable alerting for disk usage at 70 percent.

**Key decisions and why**: The 4-hour-old backup means 4 hours of cluster state changes are lost. There is no way to recover in-flight configuration changes. The priority is restoring service to the 5 million users as quickly as possible, then reconciling the missing state. The single-member etcd approach avoids quorum issues during recovery. The 8 GB quota exceeded is the root cause — the etcd database grew beyond the limit, etcd stopped accepting writes, and the compaction could not run because writes were blocked.

### Challenge 8: Debug a Performance Degradation

**Prompt**: Your Kubernetes cluster has been running smoothly for 6 months. Suddenly, deployment times increase from 2 minutes to 15 minutes. Pod creation takes 30 seconds longer than usual. The API server CPU usage is at 90 percent. etcd shows increased write latency (p99: 800 ms, up from 20 ms). The cluster has 150 nodes and 3,000 pods running 500 services. Diagnose the problem and propose a resolution.

**Model Answer**: The symptoms point to a control plane bottleneck, most likely in etcd. Here is the systematic diagnosis:

**Step 1: Check etcd health**

Run `etcdctl endpoint health --cluster` and `etcdctl endpoint status --cluster --write-out=table`. Look for a high number of Raft proposals waiting (`raft.quota-backend-bytes`), frequent leader elections, or a large database size (`db size`).

**Step 2: Check etcd database size and compaction**

If the database size exceeds 4 GB and compaction is not keeping up, the etcd write path slows down significantly. Check `DB_METRICS` via the etcd `/metrics` endpoint for `etcd_mvcc_db_total_size_in_bytes`. If the size is over 6 GB, the compaction process struggles to keep up with writes.

**Step 3: Identify the source of write amplification**

The most common cause of etcd write latency spikes at this scale is a resource with frequent updates. Check the API server audit log for the most frequent `PATCH` requests. Common culprits:

- **Pod status updates**: If a pod updates its status every 5 seconds (e.g., a health check that creates status transitions), each update goes to etcd. 3,000 pods × 1 status update per 10 seconds = 300 writes/second just for status.
- **Endpoints changes**: If a Service has many pod IPs that change frequently, EndpointSlice updates create write traffic.
- **Lease objects**: Node heartbeats create lease updates. 150 nodes × 1 update per 10 seconds = 15 writes/second.

**Step 4: Check for a specific misbehaving component**

Use `apiserver_request_duration_seconds` metric to find the slowest API calls. Filter by `verb`, `resource`, and `client`. Look for a controller or operator that is making excessive updates to a resource.

**Most likely diagnosis**: A recently deployed operator or controller is thrashing updates. For example, a custom operator that reconciles a resource every 2 seconds instead of using a watch-based approach. Each reconciliation creates a PUT or PATCH request to the API server, which propagates to etcd. The operator's update loop combined with normal cluster activity pushes etcd write throughput beyond 1,000 writes/second.

**Resolution**:

1. Identify the misbehaving controller through API server metrics: sort by `request_count`, filter to `PUT/PATCH` verbs. The controller with the highest request rate is the culprit.
2. Check the controller's reconciliation logic. Look for missing rate limiting (`workqueue.RateLimitingInterface`), missing informer caching (the controller makes API calls instead of reading from cache), or missing filters (the controller watches all namespaces when it should watch only its own).
3. Fix the controller: add rate limiting to the work queue, use the informer cache for reads, and reduce the reconciliation interval from 2 seconds to 30 seconds.
4. After the fix, verify etcd write latency returns to baseline within 5 minutes.
5. If the operator is a third-party component without source code access, contact the vendor or limit its scope (reduce the namespaces it watches) to reduce write pressure.

**Fallback**: If the operator cannot be fixed immediately, increase etcd's `--quota-backend-bytes` to 12 GB to handle the increased write throughput temporarily. This is a band-aid, not a fix, because the underlying write amplification problem remains.

## Additional Expert Challenges

### Challenge 9: Design a GitOps-Based Deployment Pipeline

**Prompt**: Design a GitOps deployment pipeline for a financial services company that requires deployment audit trails, signed commits, and approval gates. The pipeline must support 100 deployments per day across 3 environments (dev, staging, production) and 5 regional clusters. Every deployment must be traceable to a specific Git commit, approved by 2 reviewers, and signed by a CI system. Describe the architecture, the tools, and how you handle a failed deployment that must be rolled back in under 60 seconds.

**Discussion**:

The GitOps pipeline uses ArgoCD as the deployment controller. The architecture has three Git repositories:

1. **Application code repository**: Contains the source code and CI pipeline definition. Each commit triggers a build that produces a container image with an immutable tag (the Git commit SHA). The CI pipeline also runs unit tests, integration tests, and vulnerability scans.

2. **Deployment configuration repository**: Contains the Kubernetes manifests for each service in each environment. Manifests are structured as: `deployments/<service>/<environment>/kustomization.yaml`. A pull request to this repository triggers a CI pipeline that validates the manifests (kubectl validate, kubeconform, OPA policy checks) and generates a preview of the changes.

3. **Cluster state repository**: Contains the actual state of each cluster, synced by ArgoCD. This is the audit trail — every change to the cluster is recorded as a Git commit signed by ArgoCD's GPG key.

**Deployment workflow**:

1. Developer merges a PR to the application code repository.
2. CI builds the container image, tags it with the commit SHA (`sha-a1b2c3d4`), and pushes it to the container registry.
3. Developer opens a PR in the deployment configuration repository, changing the image tag for their service in the staging environment.
4. The PR CI pipeline validates the manifests. Two reviewers approve the PR.
5. The PR is merged to the main branch. CI signs the merge commit with the CI system's GPG key.
6. ArgoCD detects the change in the deployment configuration repository and syncs it to the staging cluster.
7. After verification in staging, a similar PR is created for the production environment.
8. The production PR requires approval from a senior engineer and a security team member.
9. After merge, ArgoCD syncs to all 5 production clusters using a sync wave (cluster by cluster, pausing 2 minutes between each).

**Rollback procedure**:

If a deployment fails in production, the rollback must happen in under 60 seconds. The rollback uses Git revert:

1. Engineer runs `git revert HEAD` on the deployment configuration repository.
2. The revert commit triggers ArgoCD to sync to the previous state.
3. ArgoCD detects the change and reverts the Deployment to the previous image tag.
4. The entire process takes 30-45 seconds (Git revert + ArgoCD sync + pod rolling update).

**Key design decisions**:

- Immutable image tags (Git commit SHA) ensure that a rollback deploys the exact same binary that was previously running.
- Separate repositories for application code and deployment configuration provide a clean separation of concerns and audit trail.
- Environment-specific branches or directories in the deployment configuration repository enable staged rollouts.
- ArgoCD's sync waves control the order of resource creation across clusters.
- Signed commits in the cluster state repository provide cryptographic proof of every change.

### Challenge 10: Design a Pod Security and Compliance Framework

**Prompt**: Your company must comply with SOC 2 Type II and PCI-DSS. Design a Kubernetes security framework that prevents non-compliant workloads from running. The framework must enforce: all containers must run as non-root, all images must be scanned for vulnerabilities, no privileged containers, no host network access, and all network traffic must be encrypted in transit. Describe how you enforce these policies at admission time and how you monitor compliance at runtime.

**Discussion**:

The framework has three layers: admission control (prevent), runtime enforcement (detect), and compliance auditing (report).

**Layer 1: Admission Control (Prevent)**

Use OPA Gatekeeper (or Kyverno) as a validating admission webhook. Define constraint templates that enforce the required policies:

```yaml
# Constraint: containers must run as non-root
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: non-root-container
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
  parameters:
    message: "All containers must run as non-root user"
    rules:
    - name: must_run_as_non_root
      check:
        securityContext:
          runAsNonRoot: true
```

```yaml
# Constraint: no privileged containers
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: no-privileged-containers
spec:
  parameters:
    message: "Privileged containers are not allowed"
    check:
      securityContext:
        privileged: false
```

**Layer 2: Runtime Enforcement (Detect)**

Use Falco as a runtime security monitoring tool. Falco runs as a DaemonSet and monitors system calls on every node. It detects violations at runtime:

- Container attempting to run as root (despite admission control bypass)
- Unexpected network connections to external IPs
- Mount of sensitive host paths (/etc/shadow, /var/run/docker.sock)
- Execution of shell commands inside containers (potential compromise)

Falco alerts are sent to a centralized SIEM (Splunk, Elasticsearch) for analysis and alerting.

**Layer 3: Compliance Auditing (Report)**

A compliance operator runs periodic scans of the cluster and generates compliance reports:

1. **Image scanning**: Trivy or Aqua scans all container images in the registry. Images with critical vulnerabilities (CVSS 9.0+) are blocked by the admission controller. A weekly report lists all images with vulnerabilities and their severity.

2. **Network policy audit**: A custom controller checks that every namespace has a default-deny network policy. Namespaces without network policies are flagged for remediation.

3. **RBAC audit**: A periodic job checks for over-privileged service accounts. Service accounts with `cluster-admin` role but no documented justification are flagged.

4. **Encryption audit**: The controller checks that all Ingress resources have TLS configured. A report lists all services that accept traffic without TLS.

**Dashboards**:

- Compliance scorecard: percentage of pods passing each security policy
- Vulnerability trend: number of critical/high/medium/low vulnerabilities over time
- Policy violations by namespace: which teams have the most violations
- Runtime alerts: Falco alerts by severity and type

**Emergency override**:

A security team member can use a temporary exemption for critical workloads. The exemption is a custom resource that expires after 72 hours:

```yaml
apiVersion: security.company.com/v1
kind: PolicyExemption
metadata:
  name: database-backup-2024-07-15
spec:
  policy: no-privileged-containers
  reason: "Legacy database backup tool requires root for file system access"
  expires: "2024-07-18T00:00:00Z"
  approvedBy: "security-team-lead"
  ticket: "SEC-12345"
```

The exemption controller monitors expirations and alerts 24 hours before expiration. Expired exemptions are automatically revoked, and the pod is rejected by the admission controller.

### Challenge 11: Design a Kubernetes Cluster Autoscaler for a Multi-Cloud Environment

**Prompt**: Design a cluster autoscaler that works across AWS and GCP for a multi-cloud Kubernetes deployment. The autoscaler must handle nodes from both cloud providers in the same cluster, support spot/preemptible instances, and respect node taints and tolerations. Describe how the autoscaler decides to scale up, which node group to add, and how it handles scale-down during a multi-cloud node failure.

**Discussion**:

A multi-cloud cluster autoscaler presents unique challenges because each cloud provider has different instance types, pricing models, and API rate limits.

**Architecture**: The cluster autoscaler runs as a Deployment in the control plane. It uses a pluggable cloud provider interface. For AWS, it uses the AWS cloud provider (EC2 Auto Scaling Groups). For GCP, it uses the GCE cloud provider (Managed Instance Groups). Both providers are registered as node group providers.

**Scale-up decision**: When pods are unschedulable (no node has enough resources), the autoscaler checks if any node group can accommodate the pending pods. It evaluates each node group:

1. Does the node group's instance type have enough CPU, memory, and GPU for the pending pods?
2. Do the node group's taints match the pod's tolerations?
3. Is the node group within its maximum size limit?
4. Is the cloud provider API available (not rate-limited)?

The autoscaler selects the cheapest node group that can accommodate the pods. For AWS, spot instances are 60-70 percent cheaper than on-demand. For GCP, preemptible instances are 60-80 percent cheaper than regular instances. The autoscaler prefers spot/preemptible instances for fault-tolerant workloads and on-demand for critical workloads.

**Scale-down decision**: The autoscaler periodically checks for underutilized nodes (CPU and memory utilization below 50 percent for 10 minutes). It considers:

1. Can the pods on the underutilized node be rescheduled to other nodes?
2. Would rescheduling violate any PodDisruptionBudget?
3. Is the node running a stateful workload (avoid scaling down)?
4. Is the node a spot/preemptible instance (prefer to scale down)?

The autoscaler drains the node by cordoning it (no new pods) and evicting pods. PodDisruptionBudgets are respected. If a pod cannot be evicted, the node is not scaled down.

**Multi-cloud failure handling**: When a cloud provider has an outage, its nodes become unreachable. The autoscaler:

1. Detects unreachable nodes through the node controller (node status = Unknown).
2. After 5 minutes, the node controller evicts pods from the unreachable nodes.
3. The autoscaler checks if the remaining cloud provider has capacity for the evicted pods.
4. If the remaining cloud provider does not have capacity, the autoscaler provisions new nodes from the available provider.
5. This cross-provider failover ensures that a single cloud provider outage does not cause a cluster-wide capacity shortage.

**Cost optimization**: The autoscaler tracks the cost of each node group in real-time. It maintains a cost model:

```text
AWS us-east-1, m5.large, on-demand: $0.096/hour
AWS us-east-1, m5.large, spot: $0.029/hour (70% savings)
GCP us-central1, n1-standard-2, regular: $0.095/hour
GCP us-central1, n1-standard-2, preemptible: $0.021/hour (78% savings)
```

The autoscaler uses cost-weighted scoring when multiple node groups can accommodate the pending pods. It prefers the cheapest option first, falling back to more expensive options only when the cheapest is unavailable (capacity, API limits, or taint requirements).
