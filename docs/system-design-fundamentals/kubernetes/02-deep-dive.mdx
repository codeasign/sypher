## Internal Architecture

A Kubernetes cluster consists of two distinct layers: the control plane, which makes global decisions about the cluster, and the worker nodes, which run container workloads. Each layer contains multiple components that communicate through the kube-apiserver, the central hub of all cluster operations.

### Control Plane Components

The control plane runs on three to five dedicated nodes for high availability. Each control plane node runs all four control plane components, with etcd forming a separate Raft cluster for consistency.

**kube-apiserver**: The front-end of the Kubernetes control plane. It exposes the Kubernetes REST API, handles authentication and authorization, processes admission controller webhooks, and persists all cluster state to etcd. Every interaction between components goes through the API server. No component reads etcd directly.

**etcd**: A distributed key-value store that holds the complete state of the cluster. etcd uses the Raft consensus algorithm to replicate writes across a cluster of three or five members. etcd stores all Kubernetes objects as JSON-serialized blobs under a hierarchical key structure. The key prefix `/registry/` contains all cluster resources.

**kube-scheduler**: Watches the API server for newly created pods with no node assignment and selects the optimal node for each pod. The scheduler runs a filter-and-score algorithm against every node in the cluster, then binds the pod to the highest-scoring node.

**kube-controller-manager**: A collection of controller processes bundled into a single binary. Each controller watches a specific resource type through the API server and reconciles the current state toward the desired state. Major controllers include the Node Controller, ReplicaSet Controller, Deployment Controller, Endpoint Controller, ServiceAccount Controller, and Namespace Controller.

### Worker Node Components

**kubelet**: The primary node agent that registers the node with the cluster, watches for pods assigned to its node, creates and manages pod containers, runs health probes, and reports node and pod status to the API server. The kubelet uses the container runtime interface (CRI) to interact with the container runtime (containerd, CRI-O, or Docker).

**kube-proxy**: A network proxy that runs on each node and maintains network rules. kube-proxy watches the API server for Service and EndpointSlice changes, then translates them into iptables (or IPVS) rules that route traffic to the correct pods.

**Container runtime**: The software that runs containers. containerd is the most common runtime, used by both Docker and Kubernetes. CRI-O is an alternative runtime designed specifically for Kubernetes. Both implement the CRI (Container Runtime Interface) gRPC protocol.

## Data Structures and Algorithms

### Algorithm 1: The Scheduler's Filter and Score Pipeline

The scheduler runs a two-phase algorithm for every pending pod. The algorithm is pluggable through the scheduling framework, allowing custom filter and score plugins.

**Filter phase (Predicates)**: The scheduler evaluates each node against a set of predicates. A node passes the filter phase only if all predicates pass.

Core predicates:
- `PodFitsResources`: Sum of pod resource requests (CPU, memory, ephemeral storage) must not exceed node allocatable capacity.
- `PodMatchNodeSelector`: The pod's `nodeSelector` labels must be a subset of the node's labels.
- `PodFitsHostPorts`: The pod's requested host ports must not conflict with ports already in use on the node.
- `PodToleratesNodeTaints`: The pod must tolerate all node taints with `NoSchedule` effect.
- `CheckNodeUnschedulable`: The node must not have the `unschedulable` condition.
- `CheckVolumeBinding`: The pod's PersistentVolumeClaims must be bindable to an available PersistentVolume.
- `NodeAffinity`: The node must satisfy the pod's node affinity rules (required, not preferred).

**Score phase (Priorities)**: Each node that passed the filter phase receives a score from each priority function. Scores are normalized to a 0-100 range and summed (optionally weighted).

Core priority functions:
- `BalancedResourceAllocation`: Favors nodes with balanced CPU and memory utilization.
- `LeastRequestedPriority`: Favors nodes with the least requested resources (spreads pods across nodes).
- `MostRequestedPriority`: Favors nodes with the most requested resources (packs pods tightly).
- `SelectorSpreadPriority`: Spreads pods belonging to the same Service, ReplicaSet, or StatefulSet across nodes for fault tolerance.
- `ImageLocalityPriority`: Favors nodes that already have the required container image cached locally.
- `NodeAffinityPriority`: Weights nodes higher when pod node affinity rules are satisfied.

The scheduler selects the node with the highest total score. If multiple nodes have the same score, the scheduler picks one randomly for load distribution.

### Data Structure 2: The ReplicaSet and Pod Ownership Tree

Kubernetes uses owner references to build a tree of resource ownership. A Deployment owns a ReplicaSet. The ReplicaSet owns Pods. Each object has an `metadata.ownerReferences` field that points to its parent.

The ownership tree enables cascading deletion: when you delete a Deployment, all owned ReplicaSets and Pods are garbage collected. The garbage collector watches the API server for objects with owner references. When an owner object is deleted, the garbage collector deletes the owned objects.

The ReplicaSet controller uses a label selector to identify the pods it owns. The controller lists all pods in the namespace, filters by the ReplicaSet's label selector, and counts the number of matching pods. If the count does not match the desired replica count, the controller creates or deletes pods to reach the desired count.

```yaml
# Owner reference chain: Deployment → ReplicaSet → Pod
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-6b8d9f
  ownerReferences:
  - apiVersion: apps/v1
    kind: Deployment
    name: myapp
    uid: a1b2c3d4-...
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      version: v2
  template:
    metadata:
      labels:
        app: myapp
        version: v2
    spec:
      containers:
      - name: app
        image: myapp:v2
```

### Data Structure 3: Service and EndpointSlice

A Service defines a logical set of pods and a policy for accessing them. The Service's `spec.selector` defines which pods belong to the service. The EndpointSlice controller watches both Services and Pods. When a pod matches a Service's selector and the pod's readiness probe succeeds, the EndpointSlice controller adds the pod's IP and port to the corresponding EndpointSlice resource.

Each EndpointSlice stores up to 100 pod endpoints. If a Service has more than 100 pod endpoints, the controller creates multiple EndpointSlice resources, each with a unique name. The EndpointSlice controller adds a label to each EndpointSlice pointing to the owning Service.

```yaml
apiVersion: discovery.k8s.io/v1
kind: EndpointSlice
metadata:
  name: myapp-abc123
  labels:
    kubernetes.io/service-name: my-service
addressType: IPv4
endpoints:
- addresses:
  - "10.1.0.5"
  conditions:
    ready: true
  nodeName: node-a
  targetRef:
    kind: Pod
    name: myapp-6b8d9f-x1y2z
- addresses:
  - "10.1.1.3"
  conditions:
    ready: true
  nodeName: node-b
  targetRef:
    kind: Pod
    name: myapp-6b8d9f-a3b4c
ports:
- name: http
  port: 8080
  protocol: TCP
```

## Request Lifecycle: Pod Creation End-to-End

The following sequence traces the complete lifecycle of creating a pod in a Kubernetes cluster, from the user's `kubectl run` command to the running container.

### Phase 1: kubectl to API Server

You run `kubectl run nginx --image=nginx:1.25 --replicas=3`. kubectl reads your kubeconfig file to find the cluster's API server address, your credentials (client certificate, token, or OIDC token), and any proxy configuration. kubectl constructs an HTTP POST request to the API server:

```
POST /apis/apps/v1/namespaces/default/deployments
Content-Type: application/json
Authorization: Bearer <token>

{
  "apiVersion": "apps/v1",
  "kind": "Deployment",
  "metadata": { "name": "nginx" },
  "spec": {
    "replicas": 3,
    "selector": { "matchLabels": { "app": "nginx" } },
    "template": {
      "metadata": { "labels": { "app": "nginx" } },
      "spec": {
        "containers": [{
          "name": "nginx",
          "image": "nginx:1.25",
          "ports": [{ "containerPort": 80 }]
        }]
      }
    }
  }
}
```

### Phase 2: API Server Processing

The API server receives the request and runs it through a processing pipeline:

1. **Authentication**: The API server validates the bearer token (or client certificate). If invalid, returns 401 Unauthorized.
2. **Authorization**: The API server evaluates RBAC rules. Does the user have `create` permission on `deployments` in the `default` namespace? If not, returns 403 Forbidden.
3. **Admission**: The API server runs mutating admission webhooks (which can modify the resource), then validating admission webhooks (which can reject the resource). Common mutating webhooks inject sidecar containers, set default resource limits, or add labels. Common validating webhooks enforce pod security policies or validate resource quotas.
4. **Validation**: The API server validates the object against the OpenAPI schema for Deployment resources. Ensures required fields are present, field types are correct, and semantic rules are satisfied (for example, container names must be unique within a pod).
5. **Storage**: The API server serializes the validated Deployment object to JSON, assigns a UID and resource version, and writes it to etcd under the key `/registry/deployments/default/nginx`.

### Phase 3: Controller Reconciliation

The deployment controller in the kube-controller-manager watches for Deployment changes. It detects the new Deployment and begins reconciliation:

1. The deployment controller reads the Deployment spec: `replicas: 3`, template with `nginx:1.25`.
2. The controller checks for an existing ReplicaSet owned by this Deployment. None exists, so it creates one.
3. The controller POSTs a new ReplicaSet object to the API server with `replicas: 3` and the pod template from the Deployment.

The ReplicaSet controller watches for ReplicaSet changes. It detects the new ReplicaSet and begins its own reconciliation:

1. The ReplicaSet controller lists all pods in the namespace that match the ReplicaSet's label selector.
2. Zero pods match. The desired count is 3.
3. The controller creates 3 Pod objects by POSTing to the API server. Each pod has an empty `spec.nodeName` field, meaning it is unscheduled.

### Phase 4: Scheduling

The scheduler watches for new pods with empty `nodeName`. It detects the three new pods and runs the filter and score algorithm for each:

For each pod, the scheduler evaluates all nodes in the cluster. With a 10-node cluster, the scheduler evaluates:

1. **Filter**: 10 nodes. 2 nodes fail the filter (one has insufficient CPU, one has a taint the pod does not tolerate). 8 nodes pass.
2. **Score**: Each of the 8 nodes gets a score. Node scores: NodeA=95, NodeB=87, NodeC=82, NodeD=78, NodeE=76, NodeF=70, NodeG=65, NodeH=60.
3. **Bind**: The scheduler picks NodeA (score 95) and sends a PATCH request to the API server, setting the pod's `nodeName` to `NodeA`.

The scheduler processes all three pods. With the `SelectorSpreadPriority`, it spreads the pods across different nodes. The three pods are assigned to NodeA, NodeB, and NodeC.

### Phase 5: Kubelet Creates the Pod

The kubelet on NodeA watches for pods assigned to its node. It detects the new pod and begins the pod creation sequence:

1. **Pod status**: The kubelet updates the pod's status to `Pending` through the API server.
2. **Image pull**: The kubelet checks if the `nginx:1.25` image is cached locally. If not, it instructs containerd to pull the image from Docker Hub. The pull takes 5-15 seconds depending on image size and network speed.
3. **Pod sandbox**: The kubelet creates the pod sandbox (the CRI sandbox) with its own network namespace, cgroup, and loopback interface.
4. **Volume setup**: The kubelet mounts any volumes. For this simple nginx pod, no volumes are needed.
5. **Container start**: The kubelet starts the nginx container inside the pod sandbox. containerd creates the container and starts the nginx process.
6. **Readiness probe**: The kubelet begins executing readiness probes. If the nginx container exposes port 80 and a readiness probe is configured, the kubelet performs HTTP GET requests to `http://podIP:80/` every 10 seconds. When the probe succeeds, the kubelet marks the pod as `Ready`.
7. **Endpoint update**: The EndpointSlice controller detects the ready pod and adds `10.1.0.5:80` to the Service's EndpointSlice. The Service can now route traffic to this pod.

## Consistency and State Management

Kubernetes achieves consistency through etcd's Raft consensus algorithm. All cluster state flows through the API server, which provides linearizable reads and writes on top of etcd.

### How etcd Ensures Consistency

etcd uses the Raft consensus algorithm to maintain a consistent replicated log across all cluster members. Every write must be acknowledged by a majority of etcd members (for a 3-node etcd cluster, 2 out of 3 nodes must acknowledge). This ensures that even if one etcd node fails, the cluster retains a consistent view of state.

The API server uses etcd's revision mechanism to implement optimistic concurrency control. Every object stored in etcd has a `resourceVersion` field. When you update an object, you must include its current `resourceVersion`. If another client modified the object in the meantime, the `resourceVersion` in etcd is higher than the version you provided, and the write fails with a 409 Conflict error. Your client must re-read the object and retry.

### Failure Scenario 1: Network Partition Splitting the Control Plane

A network partition divides a three-node control plane into two partitions: two nodes in Partition A and one node in Partition B.

**What happens**: The etcd leader is in Partition A. Partition B's etcd node cannot reach the leader. After the election timeout (default: 1 second), Partition B's etcd node attempts to start a new election. However, it cannot form a majority (a majority of 3 is 2, and Partition B has only 1 node). The election fails. Partition B's etcd node remains a follower.

Meanwhile, the kube-apiserver in Partition B tries to read and write to its local etcd. Both fail because the local etcd node is not the leader and cannot accept writes. The API server in Partition B returns errors to all clients.

**Impact**: The API server in Partition A continues to serve requests normally because its etcd can reach a majority. The API server in Partition B serves read requests from its etcd cache but returns errors for writes. Existing pods continue running on worker nodes because the kubelet's local state is unaffected. New deployments, scaling events, and configuration changes are accepted only by Partition A's API server.

**Recovery**: When the network partition heals, Partition B's etcd node rejoins the majority. Its stale data is overwritten by the leader's log. The API server in Partition B re-establishes its watch and receives the full state of the cluster. Worker nodes reassigned to Partition B's API server begin receiving updates again.

### Failure Scenario 2: Worker Node Failure

A worker node experiences a hardware failure (power supply, motherboard, disk). The kubelet stops sending heartbeats to the API server.

**What happens**: The node controller in the kube-controller-manager watches for node heartbeats (node status updates). The kubelet sends heartbeats every 10 seconds by default via the `--node-status-update-frequency` flag. The node controller checks for stale heartbeats every 5 seconds (configured via `--node-monitor-period`).

After 40 seconds (default `--node-monitor-grace-period`), the node controller marks the node as `Unknown` (the heartbeat update timestamp is older than the grace period). The node controller can take no further action for 5 minutes. This 5-minute delay (configured via `--pod-eviction-timeout`) gives the node a chance to recover.

If the node remains unreachable for 5 minutes, the node controller evicts all pods from the failed node. The eviction process updates each pod's status to show the node failure and the pod's nodeName is cleared. The scheduler detects the evicted pods (now unscheduled) and assigns them to healthy nodes.

**Impact**: Pods on the failed node become unavailable immediately. The deployment controller detects that the replica count is below the desired count and creates replacement pods (this happens in parallel with the eviction process). Total recovery time: 40 seconds (node detection) + 5 minutes (eviction timeout) + 15 seconds (scheduling and pod startup) = approximately 5 minutes and 55 seconds for the replacement pods to become ready.

### Failure Scenario 3: Pod Crash Loop

A deployment runs a Java application that leaks memory. After processing 10,000 requests, the JVM hits its heap limit and crashes with an OutOfMemoryError.

**What happens**: The kubelet detects the container exit with exit code 137 (SIGKILL from OOM killer). The kubelet restarts the container according to the restart policy.

**Crash loop backoff**: The kubelet applies exponential backoff to restart attempts. First restart: immediate. Second: 10 seconds. Third: 20 seconds. Fourth: 40 seconds. Subsequent restarts: 80 seconds (capped at 300 seconds by default). The counter resets after the container stays healthy for 10 minutes (configured via kubelet's `CrashLoopBackOff` reset period).

During the backoff, the readiness probe fails. The pod is removed from all Service endpoints. Clients see errors or increased latency as traffic is redirected to remaining healthy replicas.

**Recovery**: An operator must fix the application to free memory or increase the heap limit. The deployment controller creates a new pod on each iteration, but each new pod also crashes. The crash loop continues until the application code is fixed. The Horizontal Pod Autoscaler may scale up replicas in an attempt to meet demand, but the new pods also crash, so scaling does not help.

## Performance Characteristics

Kubernetes performance is measured at three levels: API server latency, scheduling throughput, and pod startup time.

### API Server Performance

The kube-apiserver handles all cluster operations. Its performance depends on the number of nodes, the rate of watch events, and the complexity of admission webhooks.

| Operation | p50 latency | p99 latency | p999 latency | Max throughput |
|---|---|---|---|---|
| Create a pod | 50 ms | 150 ms | 500 ms | 300 pods/second |
| Read a pod | 2 ms | 10 ms | 50 ms | 10,000 reads/second |
| List pods (1000 pods) | 100 ms | 500 ms | 2,000 ms | 500 lists/second |
| Update deployment | 30 ms | 100 ms | 400 ms | 100 updates/second |
| Watch event delivery | 50 ms | 200 ms | 1,000 ms | 100,000 events/second |

The API server caches list responses in memory and serves reads from the cache. Write operations go directly to etcd, which is the bottleneck for write-heavy workloads.

### etcd Performance

etcd performance is critical for cluster responsiveness. etcd performance degrades when the key space grows beyond 10 GB or the number of keys exceeds 1 million.

| Metric | Value | Bottleneck |
|---|---|---|
| Write latency (p50) | 5 ms | Disk fsync (SSD) |
| Write latency (p99) | 20 ms | Disk fsync (HDD → 100 ms) |
| Read latency (p50) | 1 ms | In-memory read |
| Max write throughput | 1,000 writes/second | Single leader bottleneck |
| Max key-value pairs | 1 million | Memory and compaction overhead |
| Max database size | 8 GB | Default --quota-backend-bytes |

etcd performance degrades significantly on HDDs due to fsync latency. Always use SSDs and dedicated network interfaces for etcd nodes.

### Scheduler Throughput

The scheduler's throughput depends on the number of nodes and the number of pending pods.

| Cluster size | Pods/second (filter only) | Pods/second (filter + score) |
|---|---|---|
| 100 nodes | 200 pods/second | 150 pods/second |
| 1,000 nodes | 100 pods/second | 50 pods/second |
| 5,000 nodes | 30 pods/second | 10 pods/second |

The scheduler scales sub-linearly with cluster size because filtering every pod against every node creates an O(N×M) workload. Large clusters use scheduler profiling to run multiple scheduler instances or tune the scheduling queue.

### Pod Startup Time

Pod startup time is the interval between pod creation (stored in etcd) and the pod becoming ready (readiness probe passes).

| Component | Time | Notes |
|---|---|---|
| etcd write | 10 ms | 3-node cluster with SSD |
| Scheduler binding | 50 ms | For a 100-node cluster |
| kubelet detection | 100 ms | Watch event delivery |
| Image pull | 5-15 seconds | For nginx:alpine (25 MB) |
| Image pull | 60-300 seconds | For large Java images (1 GB+) |
| Container start | 1-3 seconds | containerd starts the process |
| Readiness probe | 10 seconds | 1 probe interval at 10 seconds |
| Endpoint update | 100 ms | EndpointSlice controller |

A small image (nginx:alpine, 25 MB) on a warm node (image already cached) starts in under 15 seconds. A large Java image (1.5 GB) on a cold node requires 60-120 seconds for the image pull. Image caching reduces cold starts significantly.

## Watch and Informer Pattern

Kubernetes controllers use the watch API and informer pattern for efficient state synchronization.

### How Watches Work

The API server supports HTTP long-polling for resource changes. A controller creates a watch by sending a GET request to the API server with the `watch=true` parameter and a `resourceVersion` parameter indicating the last known state:

```
GET /api/v1/namespaces/default/pods?watch=true&resourceVersion=12345
```

The API server holds the HTTP connection open. When a pod is created, modified, or deleted, the API server sends a JSON event to the connection. The event contains the event type (ADDED, MODIFIED, DELETED), the object, and the new resourceVersion.

### Informer Pattern

The client-go library implements the informer pattern, which wraps watches with a local cache and event handlers:

1. **List**: On startup, the informer lists all objects of the watched type (e.g., all pods) and stores them in a local cache indexed by namespace/name.
2. **Watch**: The informer starts a watch from the last resourceVersion. Events update the local cache and trigger registered event handlers.
3. **Resync**: The informer periodically re-lists all objects to handle missed events (default: 10 minutes).

The informer pattern offloads reads from the API server. Controllers read from the local cache instead of making API calls. Writes still go through the API server.

### Watch Performance Considerations

At scale, watches can consume significant API server resources:

- Each watch adds an HTTP connection to the API server. With 5,000 nodes, each running 10 controllers, that is 50,000 watch connections.
- The API server maintains a change history in etcd. If a controller's watch falls behind (missed events due to slow processing), the API server sends a 410 Gone response. The controller must re-list all objects.
- Watch event batching reduces the overhead. The API server batches events within a 100ms window and sends them in a single HTTP response chunk.

### Event-Driven Architecture in Kubernetes

Kubernetes is fundamentally an event-driven system. Every state change in the cluster generates events that propagate through the watch mechanism. The event flow follows a specific path:

1. **State change**: A user or controller creates, updates, or deletes a resource through the API server.
2. **etcd store**: The API server persists the change to etcd, which generates a new revision.
3. **Watch notification**: etcd sends the change event to the API server's watch store.
4. **Event dispatch**: The API server dispatches the event to all active watchers for that resource type.
5. **Controller processing**: Watcher receives the event and enqueues it in the controller's work queue.
6. **Reconciliation**: The controller processes the event by reading the current state, comparing to the desired state, and taking corrective action.

This event-driven architecture ensures that controllers react promptly to state changes without polling. The API server acts as the event bus, connecting state producers (users, controllers) to state consumers (controllers, kubelets, monitoring systems).

## Resource Quotas and Limit Ranges

### Resource Quota

A ResourceQuota constrains aggregate resource consumption in a namespace. It prevents any single team or application from consuming all cluster resources.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-a-quota
  namespace: team-a
spec:
  hard:
    requests.cpu: "16"
    requests.memory: "64Gi"
    limits.cpu: "32"
    limits.memory: "128Gi"
    persistentvolumeclaims: "10"
    pods: "50"
    services: "20"
    configmaps: "30"
    secrets: "30"
```

When a ResourceQuota is applied, all pods in the namespace must specify resource requests. If a pod does not specify requests, the API server rejects it with a validation error. The admission controller checks the cumulative resource consumption against the quota before allowing any new resource creation.

### Limit Range

A LimitRange sets default resource requests and limits for pods and containers in a namespace. It also enforces minimum and maximum resource constraints.

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: team-a-limits
  namespace: team-a
spec:
  limits:
  - max:
      cpu: "4"
      memory: "8Gi"
    min:
      cpu: "100m"
      memory: "128Mi"
    default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "200m"
      memory: "256Mi"
    type: Container
```

The LimitRange admission controller applies defaults to containers that do not specify resource requirements. The min/max constraints reject pods that request too few or too many resources.

## Pod Priority and Preemption

Kubernetes supports pod priority classes that determine scheduling and eviction order. Higher-priority pods are scheduled before lower-priority pods. When cluster resources are insufficient, the scheduler may preempt (evict) lower-priority pods to make room for higher-priority pods.

### Priority Class Definition

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: critical-production
value: 1000000
globalDefault: false
description: "Critical production workloads that must never be preempted"
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: batch-low
value: 100
globalDefault: false
description: "Low-priority batch jobs that can be preempted"
```

### How Preemption Works

1. A high-priority pod (value 1000000) is created but cannot be scheduled because all nodes are at capacity.
2. The scheduler identifies nodes where preempting lower-priority pods would free enough resources.
3. The scheduler chooses the node with the lowest-priority victim pods that meet the resource requirement.
4. The scheduler evicts the victim pods by deleting them from the API server. The victims receive a deletion timestamp and termination begins.
5. The high-priority pod is scheduled on the freed node.
6. The victim pods are rescheduled on other nodes (if resources are available) or remain pending.

Preemption is a last resort. The scheduler first tries to find a node with available capacity. Preemption only occurs when no node can accommodate the pod without evicting existing pods.

## Taints and Tolerations in Depth

Taints and tolerations work together to ensure pods are placed on appropriate nodes. A taint on a node repels all pods that do not tolerate the taint.

### Taint Effects

Three taint effects control how pods are repelled:

**NoSchedule**: The scheduler does not place new pods on the node. Existing pods continue running.

**PreferNoSchedule**: The scheduler tries to avoid placing pods on the node but places them if no other node is available. This is a soft constraint.

**NoExecute**: The scheduler does not place new pods on the node. Existing pods that do not tolerate the taint are evicted immediately. This is the strongest effect.

### Use Cases

```yaml
# Dedicated GPU nodes
apiVersion: v1
kind: Node
metadata:
  name: gpu-node-1
spec:
  taints:
  - key: "gpu"
    value: "true"
    effect: "NoSchedule"

# Pod that tolerates the GPU taint
spec:
  tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"
```

Common taint patterns:
- **Dedicated node pool**: `workload=critical:NoSchedule` ensures only critical workloads use specific nodes.
- **Spot instances**: `spot=true:NoSchedule` limits spot instances to fault-tolerant workloads.
- **Operating system**: `os=windows:NoSchedule` ensures Linux pods are not scheduled on Windows nodes.
- **Node maintenance**: A node operator adds `maintenance=true:NoExecute` to drain pods before rebooting a node.
