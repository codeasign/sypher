## The Problem Story

A team of fifteen engineers manages a microservices platform running 200 services across 40 virtual machines. Each service requires a specific runtime environment: Service A needs Python 3.11 with OpenSSL 1.1.1, Service B needs Python 3.9 with OpenSSL 3.0, Service C needs a specific JVM version with custom JVM flags. The operations team manages deployment through SSH and shell scripts. A developer pushes a configuration change that accidentally upgrades OpenSSL across all services. Service A crashes. Service B experiences random TLS handshake failures. It takes six hours to identify the root cause and another four hours to roll back. The company loses $120,000 in revenue during the ten-hour outage.

A second team deploys a new version of their payment service. The deployment script copies the new binary to each server sequentially. Server three of twelve has a full disk. The script continues deploying to servers four through twelve. Seven servers run the new version, five run the old version. Traffic hits all twelve servers through a round-robin load balancer. Customers see inconsistent behavior: some get the new checkout flow, others get the old flow. Payment failures spike. The team discovers the issue when support tickets reach five hundred in an hour. Rolling back requires another sequential script run. Total recovery time: three hours.

A third team runs a batch processing job every night. The job processes 500 GB of data and takes four hours to complete. One night, the job runs thirty percent slower than usual. The operations team investigates and finds that another team's machine learning training job is consuming all available CPU on the shared server. The two teams share the same infrastructure pool but have no resource isolation. The batch job misses its SLA deadline. The machine learning team loses three days of training progress because the OOM killer terminated their process. No team owns the infrastructure contention problem.

All three scenarios share the same root cause: **managing containerized applications across a cluster of machines without a unified orchestration platform**. Teams need a system that handles deployment, scaling, resource management, and self-healing automatically, so developers focus on application logic instead of infrastructure operations.

## The Insight

Kubernetes is a container orchestration platform that automates deployment, scaling, and management of containerized applications across a cluster of machines. It provides a declarative model where you describe the desired state of your system, and Kubernetes continuously drives the actual state toward the desired state.

Think of Kubernetes as an air traffic control system for a major airport. Planes (containers) arrive and depart constantly. Each plane has a specific destination, fuel requirements, and crew. Without air traffic control, planes would collide on runways, circle endlessly waiting for gates, or land at the wrong airport. Air traffic control tracks every plane, assigns runways and gates, sequences takeoffs and landings, and reroutes around weather or congestion. The pilots do not need to negotiate with each other — they communicate only with the tower. Kubernetes plays the same role for containers: it tracks every container, assigns it to a machine with sufficient resources, restarts it if it crashes, scales it when demand increases, and routes traffic to healthy instances. The containers do not need to know about each other — they communicate through the orchestration layer.

## Core Vocabulary

**Pod** — The smallest deployable unit in Kubernetes. A pod wraps one or more containers that share a network namespace, storage volumes, and lifecycle. Pods are ephemeral — they can be created, destroyed, and rescheduled at any time. Most pods run a single container, but sidecar patterns use multiple containers per pod for logging, proxying, or monitoring.

**Node** — A worker machine in the Kubernetes cluster, either a physical server or a virtual machine. Each node runs a container runtime (like containerd), a kubelet agent that communicates with the control plane, and kube-proxy for network rules. A production cluster typically has 10 to 5,000 nodes.

**Control Plane** — The brain of the cluster, running the API server, scheduler, controller manager, and etcd database. The control plane makes global decisions about scheduling, scaling, and responding to failures. If the control plane goes down, existing workloads continue running but no new deployments or scaling events can occur.

**Deployment** — A Kubernetes resource that manages a replicated set of identical pods. A deployment defines the container image, replica count, update strategy (rolling update or recreate), and health check probes. The deployment controller ensures the specified number of pods are always running, replacing any that fail.

**Service** — A stable network endpoint that provides load-balanced access to a set of pods. Services have a fixed virtual IP address and DNS name that remain constant even as pods are created and destroyed. Services decouple clients from the ephemeral nature of pods.

**ConfigMap and Secret** — Resources that separate configuration from application code. ConfigMaps store non-sensitive configuration (environment variables, configuration files). Secrets store sensitive data (API keys, database passwords, TLS certificates) with base64 encoding and optional encryption at rest.

**Namespace** — A virtual cluster boundary within a physical Kubernetes cluster. Namespaces isolate resources, manage access control, and apply resource quotas. Teams, environments (dev, staging, prod), or application boundaries each get their own namespace.

**Volume** — A storage abstraction that provides persistent data storage for containers. Containers are ephemeral — all data written to a container filesystem is lost when the container restarts. Volumes mount external storage (host disk, cloud disk, network filesystem) into pods, surviving pod restarts and rescheduling.

**Ingress** — A Kubernetes resource that manages external HTTP and HTTPS traffic to services within the cluster. An ingress controller (like NGINX or AWS ALB) implements the ingress rules, handling TLS termination, path-based routing, and virtual hosting.

**Horizontal Pod Autoscaler (HPA)** — A controller that automatically scales the number of pod replicas based on observed CPU utilization, memory utilization, or custom metrics. The HPA continuously monitors metrics and adjusts the replica count of a deployment to match the target utilization.

**Label and Selector** — Key-value pairs attached to Kubernetes objects that organize and select subsets of objects. Labels are the fundamental grouping mechanism in Kubernetes. Selectors use label queries to identify which pods belong to a service, which pods a deployment manages, or which nodes can run a workload.

**etcd** — A distributed key-value store that serves as Kubernetes' single source of truth. All cluster state — pod definitions, service configurations, deployment specs, node status — is stored in etcd. The API server reads from and writes to etcd for every cluster operation.

## How It Actually Works — Step by Step

The lifecycle of a Kubernetes deployment follows a precise sequence of operations across control plane and worker node components.

### Step 1: User Submits a Declarative Specification

You write a YAML or JSON file describing the desired state of your application. The file specifies the container image, replica count, resource requests and limits, health check probes, environment variables, and storage requirements. This declarative approach means you tell Kubernetes what you want, not how to achieve it.

Example fields in a Deployment specification include: `apiVersion`, `kind` (Deployment), `metadata.name`, `spec.replicas`, `spec.template.spec.containers[].image`, `spec.template.spec.containers[].ports`, `spec.template.spec.containers[].resources`.

### Step 2: API Server Validates and Stores the Resource

The API server receives the HTTP request (typically a POST or PUT on the Kubernetes REST API). It validates the resource against its OpenAPI schema, performs admission control checks (authentication, authorization, mutating and validating webhooks), and stores the resulting object in etcd. The API server is the only component that interacts with etcd — no other component reads or writes the store directly.

Admission controllers can modify or reject the request before it is persisted. For example, the `PodSecurityPolicy` admission controller enforces security constraints, the `ResourceQuota` controller validates that the namespace has capacity, and a custom `MutatingAdmissionWebhook` might inject a sidecar proxy into every pod.

### Step 3: Controller Watches for Changes

The deployment controller, running inside the kube-controller-manager, watches the API server for changes to Deployment resources using a watch API. When it detects a new Deployment or an update to an existing one, it reads the current state of the cluster (existing pods, ReplicaSets) and compares it to the desired state defined in the Deployment.

The deployment controller creates or updates a ReplicaSet object that defines the desired pod template and replica count. If the deployment uses a rolling update strategy, the controller creates a new ReplicaSet with the new pod template and gradually scales down the old ReplicaSet while scaling up the new one.

### Step 4: Scheduler Assigns Pods to Nodes

The scheduler watches the API server for unscheduled pods (pods with an empty `nodeName` field). For each unscheduled pod, the scheduler runs a two-phase algorithm: filtering and scoring.

In the filtering phase, the scheduler identifies nodes that can run the pod based on resource availability, node selector constraints, affinity and anti-affinity rules, taints and tolerations, and port conflicts. Nodes that pass filtering enter the scoring phase.

In the scoring phase, the scheduler assigns a score to each eligible node based on resource utilization (spreading pods across nodes for high availability or packing pods tightly for cost optimization), the locality of data volumes, and the proximity of pods with inter-pod affinity.

The scheduler updates the pod object in the API server with the chosen node name. The scheduler does not physically start the pod — it merely binds the pod to a node.

### Step 5: Kubelet Creates the Pod on the Assigned Node

The kubelet agent runs on every worker node. It watches the API server for pods assigned to its node. When it detects a new pod bound to its node, the kubelet pulls the container image from the specified registry, creates the pod's sandbox environment (cgroups, namespaces), and starts the containers in the pod.

The kubelet performs the following sequence:

1. **Pull images**: Downloads the container image (from Docker Hub, a private registry, or a cloud container registry).
2. **Create pod sandbox**: Sets up the network namespace, cgroup limits, and loopback interface.
3. **Mount volumes**: Attaches ConfigMaps, Secrets, persistent volumes, and emptyDir volumes to the pod's filesystem.
4. **Start init containers**: Runs any init containers (used for setup tasks like database migrations or permission changes).
5. **Start main containers**: Runs the application containers specified in the pod spec.
6. **Execute startup probes**: Runs the startup probe to determine when the container has started.
7. **Report status**: Updates the API server with the pod's status.

### Step 6: Kube-Proxy Configures Network Rules

After the pod starts, the kube-proxy component on the worker node configures network rules so that traffic can reach the pod. Kube-proxy watches the API server for Service and EndpointSlice resources and updates the node's iptables (or IPVS or eBPF) rules to route traffic to the correct pods.

When a client sends a request to a Service's ClusterIP, the node's iptables rules redirect the packet to a randomly selected backend pod in round-robin fashion. If a pod becomes unhealthy, the endpoint controller removes it from the Service's endpoint list, and kube-proxy updates the iptables rules to stop routing traffic to the failed pod.

### Step 7: Health Probes Detect and Replace Failures

The kubelet continuously monitors the health of running containers using three types of probes:

**Liveness probe**: Determines if a container is still running. If the liveness probe fails, the kubelet kills the container and restarts it. Use liveness probes for applications that deadlock or enter an unrecoverable state.

**Readiness probe**: Determines if a container is ready to serve traffic. If the readiness probe fails, the kubelet removes the pod from all Service endpoints. Traffic is not routed to the pod until the readiness probe succeeds again.

**Startup probe**: Determines if a container has started. Used for slow-starting containers. The startup probe runs before the liveness probe, giving the container time to initialize.

When a pod fails its liveness probe, the kubelet restarts the container. If the container repeatedly crashes (configured by `restartPolicy` and `backoffLimit`), the kubelet backs off exponentially. When a pod is permanently unhealthy, the deployment controller creates a replacement pod on a different node, ensuring the desired replica count is maintained.

### Step 8: Horizontal Autoscaler Adjusts Replicas

The Horizontal Pod Autoscaler (HPA) periodically checks the metrics of a deployment or statefulset. If the observed metrics exceed the target threshold, the HPA increases the replica count. If the metrics fall below the target, the HPA decreases the replica count (subject to cooldown periods to avoid thrashing).

The HPA calculates the desired replica count using the formula:

```
desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue))
```

For example, if there are 5 current replicas, the current CPU utilization is 80 percent, and the target CPU utilization is 50 percent, the HPA calculates `ceil(5 * (80 / 50)) = ceil(8) = 8` replicas.

### Step 9: Rolling Update Replaces Pod Versions

When you update a Deployment's pod template (for example, changing the container image tag from v1.0 to v2.0), the deployment controller performs a rolling update. The controller creates a new ReplicaSet with the new pod template and gradually scales down the old ReplicaSet while scaling up the new one.

The rolling update respects two parameters:

**maxUnavailable**: The maximum number of pods that can be unavailable during the update (absolute count or percentage). Default: 25 percent.

**maxSurge**: The maximum number of pods that can be created above the desired replica count (absolute count or percentage). Default: 25 percent.

For a Deployment with 10 replicas, maxUnavailable=1, and maxSurge=1, the rolling update proceeds as: the controller creates 1 new pod (surge), waits for it to become ready, then terminates 1 old pod. This cycle repeats until all 10 pods run the new version. At no point are fewer than 9 pods (minAvailable) serving traffic.

If the new pod fails its readiness probe, the rolling update pauses. You can inspect the failed pods, roll back by reverting the Deployment to a previous revision (`kubectl rollout undo`), or fix the issue and resume the update.

### Step 10: Self-Healing Maintains Desired State

Every controller in Kubernetes runs a reconciliation loop that continuously compares the observed state of the system to the desired state. If a discrepancy is found, the controller takes corrective action.

If a worker node fails (the kubelet stops sending heartbeats), the node controller marks the node as `NotReady` after a configurable timeout (default: 40 seconds). After another timeout (default: 5 minutes), the controller evicts all pods from the failed node. The scheduler assigns the evicted pods to healthy nodes.

If a pod is deleted manually, the ReplicaSet controller detects the missing pod and creates a replacement. If a pod's container exits with a non-zero exit code, the kubelet restarts it according to the restart policy.

This self-healing behavior is automatic — no human intervention is required for routine failures.

## Variants and Flavors

### Variant 1: Managed Kubernetes (EKS, AKS, GKE)

Cloud providers offer managed Kubernetes services that handle control plane management, etcd backups, node upgrades, and integration with cloud-native services. The cloud provider manages the control plane's availability, scaling, and security patches. You manage the worker nodes and workloads.

**Amazon EKS**: Runs the Kubernetes control plane across three Availability Zones. Each API server endpoint is fronted by an AWS Network Load Balancer. EKS integrates with IAM for authentication, ALB for ingress, EBS/EFS for storage, and VPC CNI for pod networking.

**Google GKE**: Offers node auto-repair, auto-upgrade, and cluster autoscaling. GKE's VPC-native networking assigns IP addresses directly from the VPC range. GKE Autopilot runs a fully managed cluster where you only define workloads.

**Azure AKS**: Provides integrated Azure Active Directory authentication, Azure Policy for governance, and Azure Container Registry integration. AKS supports virtual nodes powered by ACI for burst scaling.

### Variant 2: Self-Managed Kubernetes (kubeadm, K3s, MicroK8s)

For on-premises deployments, edge computing, or environments with specific compliance requirements, you can deploy and manage Kubernetes yourself.

**kubeadm**: The standard tool for bootstrapping production-grade Kubernetes clusters on any infrastructure. kubeadm initializes the control plane, generates TLS certificates, configures the kubeconfig file, and joins worker nodes. It does not manage the underlying infrastructure (VMs, networking) — you provision those separately.

**K3s**: A certified Kubernetes distribution for resource-constrained environments (Raspberry Pi, edge devices, IoT). K3s is packaged as a single binary under 100 MB, replaces etcd with an embedded SQLite by default (optional etcd), and simplifies the control plane by combining components into a single process.

**MicroK8s**: A snap-based Kubernetes distribution from Canonical. MicroK8s runs a single-node cluster with a single command and can be extended to multi-node for production use. It includes add-ons for Istio, Knative, and GPU support.

### Variant 3: Specialized Kubernetes Distributions

**OpenShift (Red Hat)**: An enterprise Kubernetes platform built on Kubernetes with additional features: the Source-to-Image (S2I) build system, integrated container registry, built-in CI/CD with Tekton, and a web console for management. OpenShift uses Security Context Constraints (SCCs) for fine-grained pod security.

**VMware Tanzu**: A Kubernetes platform for running containers on vSphere infrastructure. Tanzu integrates with existing VMware networking, storage, and management tools. It provides namespace-as-a-service, where platform engineers create namespaces and developers deploy workloads without managing cluster infrastructure.

**Rancher**: A Kubernetes management platform that can deploy and manage clusters on any infrastructure (on-premises, cloud, edge). Rancher provides a unified UI for managing multiple clusters, centralized authentication (LDAP, SAML, OIDC), and policy management across clusters.

### Variant 4: Edge and Lightweight Kubernetes

Edge computing requires Kubernetes distributions that run on constrained hardware (1-4 CPU cores, 512 MB-4 GB RAM) and can operate offline or with intermittent connectivity.

**K3s**: Optimized for ARM and low-power x86 processors. K3s replaces etcd with embedded SQLite by default, reducing the memory footprint from 1 GB to 200 MB. A K3s cluster can run on two Raspberry Pi 4 devices with 4 GB RAM each. K3s is the foundation for edge deployments at companies like Verizon and Siemens.

**MicroK8s**: Canonical's snap-based distribution runs on Ubuntu Core and supports over-the-air updates. MicroK8s includes a built-in registry, dashboard, and Istio add-on. It supports clustering through a "cluster add" command that joins nodes over the local network.

**k0s**: A zero-friction Kubernetes distribution that packages everything into a single binary. k0s does not require a container runtime to be pre-installed — it bundles containerd. It supports air-gapped installations by allowing pre-loading of container images.

### Variant 5: Windows Container Support

Kubernetes supports Windows worker nodes for running Windows Server containers alongside Linux containers. Windows nodes run a different kubelet binary, use Windows networking (HNS instead of iptables), and support Windows-specific storage drivers.

**Challenges**: Windows containers have larger image sizes (5-15 GB versus 50-500 MB for Linux). Windows nodes cannot use Calico or Flannel for networking — they require the Windows-specific WinOverlay plugin. Windows pods cannot use hostNetwork mode. The combination of Linux control plane nodes and Windows worker nodes is the standard architecture.

**Use cases**: Organizations migrating Windows Server workloads to containers run mixed clusters. A cluster might have 100 Linux nodes for stateless web services and 20 Windows nodes for legacy .NET Framework applications that have not been ported to .NET Core.

### Variant 6: Virtual Cluster with vCluster

vCluster creates virtual Kubernetes clusters inside a physical cluster. Each virtual cluster has its own API server, scheduler, and controller manager, running as pods in the host cluster. Virtual clusters provide stronger isolation than namespaces (separate API servers, separate RBAC) with lower overhead than separate physical clusters.

**Use cases**: Multi-tenant SaaS platforms where each tenant gets a full Kubernetes API experience. Development environments where each developer gets a virtual cluster for testing. CI/CD pipelines that need isolated clusters for each build.

### Additional Mental Model Check Questions

**Question 6**: You have a pod that runs two containers: a web server and a log sidecar. The web server crashes with a segmentation fault. What happens to the sidecar container?

**Answer**: Both containers share the same pod lifecycle. The kubelet detects the web server's exit (non-zero exit code) and restarts only the web server container according to the restart policy. The sidecar container continues running uninterrupted — it shares the pod's network namespace and volumes but has its own process namespace. If the web server repeatedly crashes, the kubelet applies exponential backoff only to the crashing container. The sidecar container keeps running and collecting logs during the backoff period.

**Question 7**: Your cluster runs 100 nodes. You notice that one node has 100 pods running while another node has only 5 pods. The scheduler uses the default configuration. Why are pods distributed unevenly?

**Answer**: The default scheduler uses `LeastRequestedPriority` with `BalancedResourceAllocation` as the primary scoring plugins. Pods are placed based on resource utilization, not pod count. If the node with 100 pods runs many small pods (each requesting 10m CPU and 10 MiB memory), its total resource allocation might be lower than the node with 5 pods (each requesting 2 CPU and 8 GB memory). The scheduler places new pods on the node with the lowest resource utilization, regardless of pod count. The node with 100 small pods may still have more allocatable resources than the node with 5 large pods.

**Question 8**: You delete a namespace that contains running Deployments, Services, and ConfigMaps. What happens to the pods?

**Answer**: When a namespace is deleted, the namespace controller initiates graceful termination of all resources in the namespace. The deletion is not instantaneous. First, the API server marks the namespace as `Terminating`. The garbage collector deletes all objects with owner references in a cascading manner: Deployments → ReplicaSets → Pods. The kubelet detects the pod deletion and sends SIGTERM to the containers. After the `terminationGracePeriodSeconds` (default: 30 seconds), the kubelet sends SIGKILL. The namespace is permanently removed only after all resources are deleted. During this process, the API server rejects new resource creation in the terminating namespace.

**Question 1**: You deploy 5 replicas of a web application. One of the pods crashes due to an out-of-memory error. What happens next?

**Answer**: The kubelet detects the container exit (non-zero exit code) and restarts the container according to the restart policy (default: Always). If the container repeatedly crashes, the kubelet applies an exponential backoff (10 seconds, 20 seconds, 40 seconds, up to 5 minutes). Meanwhile, the ReplicaSet controller detects that only 4 pods are healthy and creates a new replacement pod. Kubernetes handles both approaches simultaneously — the kubelet restarts the local container while the controller creates a new pod to maintain the replica count.

**Question 2**: You update your deployment's container image from nginx:1.21 to nginx:1.25 using a rolling update strategy with maxUnavailable=2 and maxSurge=1. You have 10 replicas. What is the minimum number of available pods during the update?

**Answer**: The minimum available pods is `desiredReplicas - maxUnavailable = 10 - 2 = 8`. At any point during the rolling update, at least 8 pods must be serving traffic. The deployment controller creates up to 1 surge pod (beyond the 10 desired), so the maximum total pods at any time is 11. The controller terminates pods in batches of 2 (maxUnavailable), ensuring the availability guarantee is met.

**Question 3**: A client inside the cluster sends a request to `http://my-service:8080`. How does the request reach a specific pod?

**Answer**: DNS resolution resolves `my-service` to the Service's ClusterIP (a virtual IP, e.g., 10.96.0.1). The client's network stack sends the packet to this ClusterIP. On the node, kube-proxy has configured iptables rules that match packets destined for 10.96.0.1:8080. The iptables rules perform DNAT (Destination Network Address Translation), rewriting the destination IP to a randomly selected pod IP (e.g., 10.1.0.5:8080). The pod receives the request. The response follows the same path in reverse, with the pod's response appearing to come from the ClusterIP.

**Question 4**: Your application requires 2 CPU cores and 4 GB of memory to run. A worker node has 4 CPU cores and 8 GB total capacity, but 2.5 CPU cores and 5 GB are already allocated to other pods. Can the scheduler place your pod on this node?

**Answer**: No. The scheduler's filtering phase checks resource availability based on pod resource requests (not actual usage). The node has 4 CPU total and 2.5 CPU already requested, leaving 1.5 CPU available. Your pod requests 2 CPU, which exceeds the available CPU. The node fails the filter phase and is excluded from scheduling consideration. The scheduler looks for another node with sufficient unreserved resources.

**Question 5**: Your cluster uses Horizontal Pod Autoscaling with a CPU target of 60 percent. Current load shows average CPU utilization of 45 percent across 8 replicas. A traffic spike pushes CPU to 85 percent. How many replicas does the HPA target? What happens when the spike subsides and CPU returns to 30 percent?

**Answer**: During the spike: `desiredReplicas = ceil(8 * (85 / 60)) = ceil(11.33) = 12` replicas. The HPA updates the deployment's replica count to 12. Pods are created, and traffic distributes across them. After the spike, with CPU at 30 percent: `desiredReplicas = ceil(12 * (30 / 60)) = ceil(6) = 6` replicas. However, the HPA has a cooldown period (default: 5 minutes for scale-down, 3 minutes for scale-up) to prevent thrashing. After the cooldown, replicas scale down to 6. The scale-down process terminates pods gradually, with one pod terminated per interval in the default configuration.

### Common Debugging Patterns

**Debugging a CrashLoopBackOff**: Run `kubectl logs pod-name --previous` to see the logs from the previous (crashed) container instance. The previous logs are preserved even after a container restart. Combine with `kubectl describe pod pod-name` to see the exit code and restart count.

**Debugging a Pending pod**: Run `kubectl describe pod pod-name` and look at the Events section. The `FailedScheduling` reason includes the specific constraint that cannot be satisfied. Common causes: insufficient CPU, insufficient memory, node selector mismatch, taint intolerance, or unbound PersistentVolumeClaim.

**Debugging a NotReady node**: Run `kubectl describe node node-name` and look at the Conditions section. The `Ready` condition shows the status and the message. Common causes: kubelet not running, disk pressure, memory pressure, or network connectivity issues between the kubelet and API server.

**Debugging a service that cannot route traffic**: Run `kubectl get endpoints service-name` to see if the Service has healthy endpoints. If the endpoint list is empty, the pod selector does not match any running pods, or the pods are not passing readiness probes. Run `kubectl get pods --selector=app=myapp` to verify the selector matches.

## Kubernetes Resource Types Overview

Kubernetes resources are organized by API group and version. The core group (no API group prefix) includes the most commonly used resources:

**Workload resources** (apps/v1): Deployment, ReplicaSet, StatefulSet, DaemonSet. These manage the lifecycle of containerized applications.

**Service resources** (core/v1): Service, Endpoints, EndpointSlice. These provide network access to pods.

**Configuration resources** (core/v1): ConfigMap, Secret. These separate configuration from application code.

**Storage resources** (core/v1): PersistentVolume, PersistentVolumeClaim, StorageClass. These provide persistent storage for stateful workloads.

**Access control resources** (rbac.authorization.k8s.io/v1): Role, ClusterRole, RoleBinding, ClusterRoleBinding. These control who can do what in the cluster.

**Network resources** (networking.k8s.io/v1): NetworkPolicy, Ingress, IngressClass. These control traffic flow and external access.

**Autoscaling resources** (autoscaling/v2): HorizontalPodAutoscaler, VerticalPodAutoscaler. These automatically adjust resource allocation based on demand.

**Batch resources** (batch/v1): Job, CronJob. These manage run-to-completion and scheduled workloads.

Each resource type is represented as a YAML or JSON object with a standard structure: `apiVersion`, `kind`, `metadata`, `spec`, and `status`. The `metadata` section contains identifying information (name, namespace, labels, annotations). The `spec` section describes the desired state. The `status` section contains the current state (read-only, updated by controllers).

## Kubernetes Architecture Patterns

### Pattern 1: Three-Tier Web Application

A standard web application in Kubernetes has three tiers: frontend, backend API, and database. Each tier is a separate Deployment with its own Service.

**Frontend**: Stateless web server (nginx, React, Vue). Exposed through an Ingress with TLS termination. HPA scales based on CPU utilization. No persistent storage needed.

**Backend API**: Stateless application server (Node.js, Python, Java). Exposed through a ClusterIP Service (internal only). HPA scales based on CPU utilization and request latency. Connects to the database through a database Service.

**Database**: Stateful PostgreSQL or MySQL. Runs as a StatefulSet with persistent storage. Exposed through a ClusterIP Service. No HPA (scale manually or use a database operator).

```
Request flow:
  User ──> DNS ──> Ingress (TLS) ──> Frontend Service ──> Frontend Pod
            └──> Frontend Pod calls Backend Service ──> Backend Pod
                  └──> Backend Pod calls Database Service ──> Database Pod
```

### Pattern 2: Event-Driven Microservices

Microservices communicate through an event broker (Kafka, RabbitMQ, NATS). Each microservice is a separate Deployment with HPA based on queue depth.

**Producer service**: Creates events and publishes them to the broker. Stateless, scales based on API request rate.

**Consumer service**: Reads events from the broker and processes them. Stateless, scales based on queue depth (custom HPA metric from Prometheus).

**Broker**: Kafka or RabbitMQ. Runs as a StatefulSet with persistent storage. Managed by an operator (Strimzi for Kafka, RabbitMQ Cluster Operator).

### Pattern 3: Batch Processing Pipeline

A batch processing pipeline processes data in stages. Each stage is a Kubernetes Job or a Deployment with a queue.

**Stage 1**: CronJob that runs every hour, pulling data from an external source and writing to a staging area.

**Stage 2**: A Work Queue pattern where a Deployment reads from a queue, processes data, and writes results.

**Stage 3**: A final Job that aggregates results and loads them into a database or data warehouse.

```
CronJob (hourly) ──> Queue ──> Worker Deployment (autoscaling) ──> Output
```

### Pattern 4: Sidecar Proxy with Service Mesh

Every pod includes an Envoy or Linkerd proxy sidecar that handles traffic management, security, and observability.

**Traffic flow**: Pod traffic → Sidecar proxy (mTLS, routing, metrics) → Destination pod → Sidecar proxy → Response.

**Benefits**: Zero-trust security (all traffic encrypted), automatic retries and circuit breaking, distributed tracing without application changes, and fine-grained traffic splitting for canary deployments.

## Container Runtime Interface (CRI)

The CRI is a gRPC protocol that allows the kubelet to interact with different container runtimes without being tightly coupled to any specific runtime. The kubelet sends CRI requests (create container, start container, stop container, remove container) to the CRI runtime shim.

**containerd**: The most widely used CRI runtime. Originally part of Docker, now a standalone CNCF project. containerd handles image management (pull, push, store), container execution, and storage management. containerd communicates with runc (the low-level container runtime) through the containerd-shim process.

**CRI-O**: A lightweight CRI runtime designed specifically for Kubernetes. CRI-O is optimized for Kubernetes workloads and has a smaller footprint than containerd. It is the default runtime in OpenShift and some enterprise Kubernetes distributions.

**Docker through cri-dockerd**: Docker does not implement the CRI natively. The cri-dockerd shim translates CRI requests to Docker API calls. This adds latency and complexity. Kubernetes deprecated Docker as a runtime starting in v1.24, recommending containerd directly.
