# Relational Databases — Under the Hood

In the previous lesson, you learned what relational databases are and why they dominate the storage landscape. Now you will open the hood and see what actually happens when you run a query.

This lesson covers the internal architecture of a relational database, the data structures and algorithms that make it fast, the lifecycle of a query from client to disk and back, the consistency model that keeps data safe, the performance characteristics you can expect, and the configuration knobs you can turn to make it all go faster.

---

## Internal Architecture

A relational database is not a monolith. It is a collection of collaborating subsystems, each with a specific job. When you connect to a database and send a query, your request passes through several of these subsystems before returning a result.

### The Major Subsystems

**Connection Manager** — Accepts client connections, authenticates users, and manages a pool of backend processes or threads. Each client connection is typically bound to a dedicated backend process (PostgreSQL model) or a thread from a pool (MySQL InnoDB model).

**Query Parser** — Takes the raw SQL text and turns it into a parse tree. It checks syntax and produces an internal representation the database can work with. If your SQL has a typo, the parser is where you get the error.

**Query Planner / Optimizer** — Takes the parse tree and produces one or more query plans, then estimates the cost of each plan and picks the cheapest one. The planner uses table statistics (row counts, data distribution, index selectivity) to make these estimates.

**Executor** — Takes the chosen plan and executes it, pulling rows from storage, applying filters, performing joins, and returning results. The executor is a pipeline: each node in the plan tree produces rows that the parent node consumes.

**Storage Engine** — Manages how data is laid out on disk and in memory. It handles pages (the unit of I/O), the buffer pool (in-memory cache of pages), the write-ahead log (WAL) for crash safety, and the access methods (sequential scan, index scan) for reading data.

**Transaction Manager / Lock Manager** — Enforces concurrency control. It manages locks on rows, pages, and tables, ensures that transactions are isolated from each other, and coordinates the commit or rollback of transactions.

**Buffer Pool** — The in-memory cache of data pages. When you read a row, the storage engine checks the buffer pool first. On a hit, no disk I/O is needed. On a miss, the page is read from disk into the pool, possibly evicting a less-used page.

**Write-Ahead Log (WAL)** — Every write is first appended to the WAL before the data page is modified. This ensures that if the database crashes, it can replay the WAL to recover committed transactions. This is the "W" in ACID durability.

### How These Subsystems Work Together

When you run `SELECT * FROM orders WHERE user_id = 42`, here is what each subsystem contributes:

1. The **Connection Manager** accepts your TCP connection and authenticates you.
2. The **Parser** converts the SQL text into a parse tree and checks for syntax errors.
3. The **Planner** looks at the `orders` table statistics, sees that there is an index on `user_id`, estimates that an index scan will read 10 pages vs. a sequential scan that would read 10,000 pages, and picks the index scan plan.
4. The **Executor** runs the plan: it asks the storage engine to open the index on `user_id`, seek to the value 42, and fetch matching rows from the heap.
5. The **Storage Engine** checks the buffer pool for the relevant index and data pages, loads missing pages from disk, and returns rows through the executor and connection manager back to you.

Each subsystem is independently designed, tested, and tuned. When you change a configuration parameter like `shared_buffers` or `work_mem`, you are influencing the behavior of one or more of these subsystems.

## Data Structures and Algorithms

### B-Tree Indexes

The B-tree is the most important data structure in relational databases. Almost every index is a B-tree (or its variant, the B+ tree).

A B-tree is a balanced tree structure where each node can hold multiple keys and pointers to child nodes. The tree maintains sorted order, supports efficient insertion, deletion, and lookup, and keeps the depth low (typically 3-5 levels for billions of rows).

Key properties:

- **Self-balancing** — All leaf nodes are at the same depth, guaranteeing O(log n) lookup time.
- **High fanout** — Each node holds hundreds of keys, so the tree is short and wide, not tall and narrow.
- **Sequential access** — Leaf nodes are often linked together, allowing efficient range scans without jumping back up the tree.

```sql
-- Creating a B-tree index explicitly (default index type in PostgreSQL)
CREATE INDEX idx_orders_user_id ON orders (user_id);

-- This makes the following query much faster:
SELECT * FROM orders WHERE user_id = 42;

-- Range queries also benefit:
SELECT * FROM orders WHERE created_at BETWEEN '2025-01-01' AND '2025-01-31';
```

**When B-tree indexes help:**
- Point lookups by a specific value
- Range scans on ordered data
- `ORDER BY` on indexed columns (avoids an explicit sort)
- `JOIN` conditions on indexed columns

**When they do not help:**
- Columns with very few distinct values (low selectivity), like a boolean column
- Full-text search (use a dedicated inverted index for that)
- Wildcard patterns like `LIKE '%xyz'` (cannot use the tree ordering)

### Hash Indexes

Hash indexes map keys directly to bucket locations using a hash function. They give O(1) lookup time for equality conditions but cannot support range queries.

```sql
-- PostgreSQL supports hash indexes (though B-tree is often just as good for equality)
CREATE INDEX idx_orders_order_id_hash ON orders USING hash (order_id);

-- Good for:
SELECT * FROM orders WHERE order_id = 12345;

-- Useless for:
SELECT * FROM orders WHERE order_id > 10000;  -- range not supported
```

Hash indexes are less common than B-trees in practice. PostgreSQL strongly recommends B-trees for almost all use cases. MySQL's InnoDB uses B+ trees as its clustered index and does not natively support hash indexes (though it has an adaptive hash index feature built into the buffer pool).

### Heap Pages

The heap is where the actual row data lives. A heap is an unordered collection of pages, each typically 8 KB (PostgreSQL) or 16 KB (MySQL). Pages contain a header with metadata and an array of row slots.

```
┌──────────────────────────────────────────┐
│  PAGE HEADER (24 bytes)                   │
│  - LSN (last log sequence number)         │
│  - number of tuples                       │
│  - free space pointer                     │
│  - flags (page type, checksum)            │
├──────────────────────────────────────────┤
│  ROW SLOT 1                               │
│  ROW SLOT 2                               │
│  ROW SLOT 3                               │
│  ...                                       │
│  FREE SPACE  ████████████████████████     │
│  ...                                       │
│  ROW SLOT N  (added from bottom)          │
│  ROW DATA AREA (variable length)          │
└──────────────────────────────────────────┘
```

When you update a row in PostgreSQL, the old version stays in the heap and a new row version is inserted (MVCC). The old version becomes dead and is eventually cleaned up by `VACUUM`. In MySQL InnoDB, similar undo log records preserve the old version.

### Write-Ahead Log (WAL)

The WAL is a sequentially-written, append-only log on disk. Before any data page is modified, the change is recorded in the WAL. This guarantees durability: if the database crashes, it can replay the WAL from the last checkpoint to recover all committed transactions.

```
┌─────────────────────────────────────────────────────────┐
│                    WAL SEGMENTS                           │
├──────────┬──────────┬──────────┬──────────┬─────────────┤
│ WAL 0001 │ WAL 0002 │ WAL 0003 │ WAL 0004 │ WAL 0005    │
│          │          │          │          │ (current)   │
├──────────┴──────────┴──────────┴──────────┴─────────────┤
│ Each segment is 16 MB (PostgreSQL default)               │
│ Records are LSN-ordered for sequential replay            │
│ Segments are recycled, not deleted, to avoid fsync       │
└─────────────────────────────────────────────────────────┘
```

Each WAL record contains:

- **LSN** (Log Sequence Number) — A monotonically increasing byte offset that identifies the record's position in the log.
- **Transaction ID** — The transaction that made the change.
- **Page ID** — Which data page was modified.
- **Delta or full-page image** — The actual change data (or the entire page during a full-page write).
- **Checksum** — For integrity verification during recovery.

The WAL is written to disk synchronously at transaction commit. PostgreSQL's `synchronous_commit` setting controls this behavior:

| Setting | Behavior | Risk |
|---------|----------|------|
| `on` | Flush WAL to disk on every commit | Maximum durability |
| `remote_write` | Flush WAL to OS on commit, disk on receipt | One crash between commit and disk flush |
| `local` | Flush WAL to local disk only | Replica may lose data on failover |
| `off` | Keep WAL in buffers, flush once per second | Lose up to 1 second of commits on crash |

### MVCC (Multi-Version Concurrency Control)

MVCC is the mechanism that allows concurrent readers and writers without blocking each other. Instead of locking rows for reading, MVCC presents each transaction with a snapshot of the database as of a specific point in time.

Here is how it works:

- Every row has two hidden columns: `xmin` (the transaction that created this version) and `xmax` (the transaction that deleted or superseded this version).
- When a transaction reads, it sees only row versions where `xmin` was committed before the transaction started and `xmax` is either null or from a transaction that started after the reading transaction.
- When a transaction updates a row, it marks the old version as deleted (sets `xmax`) and inserts a new version with its own transaction ID as `xmin`.
- Dead row versions accumulate and must be cleaned up by `VACUUM`.

```sql
-- Check dead tuple count in PostgreSQL
SELECT relname, n_dead_tup, n_live_tup, last_vacuum, last_autovacuum
FROM pg_stat_user_tables WHERE relname = 'orders';

-- Manual vacuum to reclaim space
VACUUM ANALYZE orders;
```

### Page Layout Details

Each page in the database follows a specific layout. Understanding this layout helps you reason about storage efficiency and performance.

```
┌──────────────────────────────────────────────────────────────┐
│  PAGE HEADER (24 bytes)                                      │
│  pd_lsn     - LSN of last change to this page                │
│  pd_checksum - optional page checksum                        │
│  pd_flags   - page type flags                                │
│  pd_lower   - offset to start of row pointers                │
│  pd_upper   - offset to end of free space                    │
│  pd_special - offset for index-specific data                 │
│  pd_pagesize_version - page size and version                 │
├──────────────────────────────────────────────────────────────┤
│  ROW POINTERS (grows downward from header)                    │
│  ┌─────┬─────┬─────┬─────┬─────┐                            │
│  │ P1  │ P2  │ P3  │ P4  │     │  ← free slot              │
│  └─────┴─────┴─────┴─────┴─────┘                            │
│                                                              │
│  FREE SPACE                                                   │
│  ████████████████████████████████████████████████████████████ │
│                                                              │
│  ROW DATA (grows upward from bottom)                         │
│  ┌──────────────────────────────────────────────────────┐    │
│  │ Row 4 data (most recently inserted)                  │    │
│  │ Row 3 data                                           │    │
│  │ Row 2 data                                           │    │
│  │ Row 1 data (first inserted, possibly dead)           │    │
│  └──────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
```

Page fill factor controls how much of each page is reserved for future updates. A fill factor of 70 percent leaves 30 percent of each page empty, so an update that expands a row (e.g., changing a short string to a longer one) can fit on the same page without forcing a page split.

```sql
-- Set fill factor on a table (PostgreSQL)
ALTER TABLE orders SET (fillfactor = 70);
```

## Request / Write / Read Lifecycle

When you run a query, a carefully orchestrated sequence of events unfolds inside the database. This section walks through each type of operation step by step.

### Read Lifecycle

A read query like `SELECT status FROM orders WHERE order_id = 42` goes through these steps:

1. **Connection** — The client opens a TCP connection to the database, which authenticates the user and allocates a backend process.

2. **Parse** — The SQL text is tokenized and parsed into an internal parse tree. Syntax errors are reported here.

3. **Analyze** — The parse tree is analyzed against the system catalog. The database checks that the `orders` table exists, that `order_id` is a valid column, and that the user has permission to read it.

4. **Plan** — The planner receives the analyzed tree and generates alternative plans. For this query, the alternatives might be a sequential scan of `orders` or an index scan on the `order_id` index. The planner estimates the cost of each using table statistics.

5. **Execute** — The executor starts the chosen plan:
   a. Open the B-tree index on `order_id`.
   b. Traverse from root to leaf to find the entry for 42.
   c. Read the row pointer (TID — tuple identifier) from the leaf.
   d. Read the heap page at the given offset to fetch the row.
   e. Apply MVCC visibility: check whether this row version is visible to the current transaction.
   f. Project the `status` column and return the row.

### Write Lifecycle

A write query like `UPDATE orders SET status = 'shipped' WHERE order_id = 42` goes through these steps:

1. **Parse and Plan** — Same as reads. The planner generates a plan that includes the index seek and the heap update.

2. **Lock** — The transaction manager acquires a row-level lock (or page-level lock, depending on the database) on the target row to prevent concurrent writes.

3. **Read the current row version** — The executor fetches the current row as it would for a read.

4. **Write the WAL record** — Before modifying the page, a WAL record is written describing the change. This record includes the LSN, transaction ID, page ID, and the before-and-after image of the row.

5. **Modify the heap page** — The storage engine marks the old row version as dead (sets `xmax` to the current transaction ID) and inserts a new row version. This happens in the buffer pool; the page is not yet written to disk.

6. **Write the index entry** — If the indexed column changed, the index is also updated (delete old entry, insert new one). If the indexed column is unchanged, many databases (like PostgreSQL) use a "heap-only tuple" optimization that avoids touching the index.

7. **Commit** — At commit time, the WAL is flushed to disk synchronously. The commit record is the last WAL entry. After the flush, the transaction is guaranteed durable even if the database crashes immediately.

8. **Return** — The database sends the number of affected rows back to the client. The transaction is still holding locks until it commits or rolls back.

### Transaction Lifecycle with Savepoints

You can mark points within a transaction that allow partial rollback. This is useful for batch operations where you want to preserve work done before an error.

```sql
BEGIN;
  INSERT INTO audit_log (event) VALUES ('start_batch');
  SAVEPOINT sp1;
  INSERT INTO orders (id, user_id, total) VALUES (1001, 42, 59.99);
  -- Something goes wrong, but you want to keep the audit entry
  ROLLBACK TO SAVEPOINT sp1;
  -- The orders insert is rolled back, the audit entry remains
  INSERT INTO orders (id, user_id, total) VALUES (1001, 42, 59.99);
COMMIT;
```

## Consistency and State

### ACID Properties

ACID is the contract that relational databases offer to application developers. Each letter represents a guarantee:

**Atomicity** — A transaction is all-or-nothing. If the database crashes mid-transaction, the effects of any partially-executed transaction are fully rolled back. The WAL is crucial here: committed transactions have a COMMIT record; uncommitted ones do not, and their changes are not recovered.

**Consistency** — A transaction brings the database from one valid state to another. Constraints (foreign keys, unique constraints, check constraints) are enforced at transaction boundaries. Consistency is more of a correctness property that you define through constraints; the database enforces it.

**Isolation** — Concurrent transactions do not interfere with each other. The database provides isolation levels that trade strictness for performance. The ANSI SQL standard defines four levels, described below.

**Durability** — Once a transaction commits, its changes persist even if the database crashes immediately after. Durability comes from the WAL: the commit record is flushed to disk before the database acknowledges the commit to the client.

### Isolation Levels

The ANSI SQL standard defines four isolation levels, each preventing a specific set of concurrency anomalies:

| Level | Dirty Read | Non-repeatable Read | Phantom Read |
|-------|-----------|---------------------|--------------|
| Read Uncommitted | Possible | Possible | Possible |
| Read Committed | Prevented | Possible | Possible |
| Repeatable Read | Prevented | Prevented | Possible |
| Serializable | Prevented | Prevented | Prevented |

**Read Uncommitted** — A transaction can see uncommitted changes from other transactions. Rarely used in practice because dirty reads are almost always undesirable. PostgreSQL does not actually support this level; it treats it as Read Committed.

**Read Committed** (default in PostgreSQL and most databases) — A transaction sees only committed changes. If you run the same query twice in the same transaction, you might get different results (non-repeatable read), because another transaction may have committed between the two queries.

**Repeatable Read** — A transaction sees a consistent snapshot taken at the start of the transaction. Running the same query twice returns the same results. Phantom reads (new rows appearing in a range query) were originally allowed under this level, but PostgreSQL and InnoDB prevent them through their MVCC implementations.

**Serializable** — Transactions are executed as if they ran one after another. The database detects conflicts and aborts transactions when serialization cannot be guaranteed. This is the safest level but also the most expensive, as it increases the rate of transaction aborts under contention.

```sql
-- Set isolation level for a transaction
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM orders WHERE user_id = 42;
-- ... more queries, same snapshot ...
COMMIT;
```

### How Isolation Levels Map to MVCC

Each isolation level corresponds to a different snapshot behavior:

- **Read Committed** — A new snapshot is taken for each statement in the transaction.
- **Repeatable Read** — A single snapshot is taken at the start of the transaction and reused for all statements.
- **Serializable** — Same snapshot as Repeatable Read, but the database also tracks predicate-based conflicts and adds runtime conflict detection.

### Crash Recovery

When a database starts after a crash, it goes through three phases:

1. **Redo Phase** — Starting from the last checkpoint, the database replays every WAL record forward. Any committed transaction that modified a page not yet flushed to disk has its changes reapplied. This guarantees durability.

2. **Undo Phase** — Any transaction that was active at the time of the crash (no COMMIT record) has its changes rolled back. In PostgreSQL, this is done lazily: the changes remain on disk but are marked as invisible to all transactions. The next `VACUUM` cleans them up.

3. **Recovery Complete** — The database opens for business. The WAL is truncated to the last checkpoint, and normal operation resumes.

### Deadlock Detection

When two transactions each hold a lock that the other needs, a deadlock occurs. The database detects this cycle and aborts one of the transactions, allowing the other to proceed.

```
Transaction A                     Transaction B
─────────────────                 ─────────────────
BEGIN;                            BEGIN;
UPDATE accounts SET              UPDATE orders SET
  balance = balance - 100          status = 'shipped'
  WHERE id = 1;                    WHERE id = 42;
                                  UPDATE accounts SET
UPDATE orders SET                    balance = balance - 100
  status = 'paid'                    WHERE id = 1;
  WHERE id = 42;               ← DEADLOCK! A waits for B's
                                  lock on accounts(1),
DEADLOCK detected!                 B waits for A's lock on
A is chosen as victim              orders(42).
ROLLBACK;                        COMMIT; -- B succeeds
```

The deadlock detector runs periodically (every 1 second in PostgreSQL by default, configurable via `deadlock_timeout`). When it finds a cycle, it chooses the transaction that is cheapest to roll back (based on the accumulated work) and aborts it with a `40001` error code.

```sql
-- Your application must handle deadlock errors with retry logic
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE orders SET status = 'paid' WHERE id = 42;
COMMIT;
-- If you get ERROR: deadlock detected, retry the entire transaction
```

## Performance Characteristics

### Latency

Latency in a database system comes from multiple sources. Understanding where time goes helps you diagnose slow queries.

| Operation | p50 | p99 | p99.9 |
|-----------|-----|-----|-------|
| Simple point lookup (PK, cached) | 0.2 ms | 0.5 ms | 2 ms |
| Simple point lookup (PK, uncached) | 3 ms | 10 ms | 50 ms |
| Range scan (100 rows, cached) | 1 ms | 5 ms | 20 ms |
| Range scan (100 rows, uncached) | 10 ms | 30 ms | 100 ms |
| Two-table join (indexed) | 2 ms | 8 ms | 40 ms |
| Two-table join (sequential scan) | 50 ms | 200 ms | 2000 ms |
| INSERT (single row) | 1 ms | 5 ms | 20 ms |
| INSERT (batch of 1000) | 20 ms | 50 ms | 200 ms |
| UPDATE with index maintenance | 2 ms | 10 ms | 50 ms |

These numbers assume PostgreSQL 15 on an SSD-backed instance with 16 GB RAM. Multiply by 2-3 for MySQL on equivalent hardware without connection pooling. Multiply by 5-10 for spinning disks.

The gap between p50 and p99.9 tells you about tail latency. A query that takes 0.2 ms in the median but 2 ms at the 99.9th percentile is likely hitting buffer pool misses or context switches. If the p99.9 jumps to 50 ms, check for lock contention or checkpoint stalls.

### Throughput

A single PostgreSQL instance on moderate hardware (8 vCPUs, 32 GB RAM, SSD) handles roughly:

- **Reads**: 50,000-100,000 simple point lookups per second
- **Writes**: 10,000-25,000 single-row inserts per second
- **Mixed**: 5,000-15,000 transactions per second (TPC-C-like workload)

Throughput is highly workload-dependent. Analytical queries scanning millions of rows saturate disk I/O at far lower throughput. Insert-heavy workloads hit WAL throughput first.

### Memory Footprint

The buffer pool is the single largest memory consumer. As a rule of thumb, allocate 25 percent of total RAM to the buffer pool (PostgreSQL's `shared_buffers`) and another 25 percent to the OS page cache. If your working set fits in this combined cache, most reads will be cache hits. If the working set is larger than the cache, expect frequent page evictions and a sharp drop in performance.

Other memory consumers:

- **WAL buffers**: 16-64 MB (small but critical for write throughput)
- **work_mem**: per-operation, per-query memory for sorts and hash joins. A query with 3 sort operations uses 3 x work_mem.
- **maintenance_work_mem**: used by VACUUM, CREATE INDEX, and other maintenance operations
- **Connection overhead**: each connection uses 2-10 MB for its backend process

### Cache Hit Ratio

The cache hit ratio is the percentage of page requests satisfied from the buffer pool without going to disk. For most OLTP workloads, a rate above 99 percent is expected.

```sql
-- Check cache hit ratio in PostgreSQL
SELECT
  'buffer hit ratio' AS metric,
  (SUM(heap_blks_hit) * 100.0 / NULLIF(SUM(heap_blks_hit + heap_blks_read), 0)) AS value
FROM pg_statio_user_tables;
```

If your cache hit ratio drops below 95 percent, your working set does not fit in the buffer pool. Options include adding more RAM, increasing `shared_buffers`, or reducing the working set by archiving old data.

## Configuration and Tuning

### Key Parameters in PostgreSQL

**shared_buffers** — Size of the buffer pool. Default: 128 MB. Suggested: 25 percent of RAM. Going too high (over 40 percent) causes contention on the buffer pool management structures and can actually reduce performance.

**effective_cache_size** — An estimate of how much memory the OS page cache has available. Default: 4 GB. Suggested: 75 percent of RAM. This does not allocate memory; it tells the planner how likely it is that a page is cached, influencing the planner toward index scans over sequential scans.

**work_mem** — Memory per operation (sort, hash join) per query. Default: 4 MB. Each sort operation in a query gets its own allocation. If you have 100 concurrent queries each sorting, total memory used for work_mem = 100 x work_mem. Increase cautiously.

**wal_buffers** — Memory for WAL buffers before flushing to disk. Default: 16 MB. Suggested: 64 MB for write-heavy workloads.

**maintenance_work_mem** — Memory for VACUUM, CREATE INDEX, and other maintenance. Default: 64 MB. Suggested: 1 GB for large databases with regular vacuuming.

**max_connections** — Default: 100. Each connection consumes about 2-10 MB. For high concurrency, use a connection pooler (PgBouncer, Pgpool-II) instead of raising this past 500.

### Aggressive Settings for Write-Heavy Workloads

For a database handling 50,000 writes per second on dedicated hardware:

```ini
shared_buffers = 8GB         -- on 32 GB machine
wal_buffers = 128MB
work_mem = 16MB
maintenance_work_mem = 1GB
synchronous_commit = off     -- faster but loses up to 1 WAL flush on crash
checkpoint_timeout = 15min
max_wal_size = 32GB
```

With `synchronous_commit = off`, a crash can lose the last few milliseconds of writes. Use this only when the application can tolerate some data loss (analytics pipelines, append-only logs).

### Checkpoint Configuration

Checkpoints are periods when all dirty pages in the buffer pool are flushed to disk. Frequent checkpoints use more I/O bandwidth but reduce crash recovery time. Infrequent checkpoints use less I/O during normal operation but take longer to recover from a crash.

```ini
checkpoint_timeout = 5min    -- default; shorter = more frequent
checkpoint_timeout = 30min   -- less I/O load but longer recovery
max_wal_size = 1GB           -- default; triggers checkpoint when full
max_wal_size = 16GB          -- fewer checkpoints on write-heavy workloads
```

The tradeoff is between steady-state I/O and crash recovery time. Each checkpoint writes all dirty buffers, which can cause a spike in disk writes. If your application is latency-sensitive, you may want to spread checkpoint writes over time:

```ini
checkpoint_completion_target = 0.9   -- spread writes over 90% of the checkpoint interval
```

### Observability Queries

PostgreSQL's `pg_stat_statements` extension tracks cumulative execution statistics for every query: total time, calls, rows returned, shared block hits and reads. This is the single most useful tool for identifying slow queries:

```sql
SELECT query, calls, total_exec_time / calls AS avg_ms,
       shared_blks_hit, shared_blks_read
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

A query with a high ratio of `shared_blks_read` to `shared_blks_hit` is scanning disk pages that should be cached or indexed. That is the classic sign of a missing index.

### Monitoring Locks and Waits

```sql
-- Find blocked queries
SELECT blocked.pid AS blocked_pid,
       blocked.query AS blocked_query,
       blocking.pid AS blocking_pid,
       blocking.query AS blocking_query
FROM pg_locks blocked
JOIN pg_stat_activity blocked_act ON blocked.pid = blocked_act.pid
JOIN pg_locks blocking ON blocked.pid != blocking.pid
  AND blocked.granted = false
  AND blocking.granted = true
  AND blocked.relation = blocking.relation
JOIN pg_stat_activity blocking_act ON blocking.pid = blocking_act.pid;
```

This query shows which transactions are waiting for locks and which transactions are holding those locks. Long wait chains suggest contention on a hot row or table.

### Indexing Strategy Guidelines

Getting indexes right is the single highest-impact tuning lever for query performance. Follow these guidelines:

**Index columns used in WHERE clauses** — Equality conditions benefit from simple B-tree indexes. Range conditions (greater than, less than, BETWEEN) also benefit but may need careful ordering of composite indexes.

**Index join columns** — Columns used in JOIN conditions (foreign keys) should be indexed. Every `orders.user_id` referencing `users.id` should have an index on `orders.user_id`.

**Composite indexes** — For queries filtering on multiple columns, a composite index can be more efficient than separate indexes:

```sql
-- For this query:
SELECT * FROM orders WHERE user_id = 42 AND status = 'pending';

-- A composite index on (user_id, status) is better than two single-column indexes
CREATE INDEX idx_orders_user_status ON orders (user_id, status);

-- Column order matters: put high-selectivity columns first
-- user_id (high selectivity) before status (low selectivity)
```

**Partial indexes** — When only a subset of rows is queried frequently, a partial index saves space and improves performance:

```sql
-- Index only pending orders (90% of orders are 'completed' and rarely queried)
CREATE INDEX idx_orders_pending ON orders (user_id) WHERE status = 'pending';
```

**Covering indexes** — If the index contains all the columns needed by a query, the database never needs to touch the heap:

```sql
-- If you only select status and created_at for pending orders:
CREATE INDEX idx_orders_pending_covering ON orders (user_id)
  INCLUDE (status, created_at)
  WHERE status = 'pending';
```

### Common Configuration Mistakes

**Setting shared_buffers too high** — On many operating systems, allocating more than 8-10 GB to the buffer pool causes the kernel to manage shared memory inefficiently. The OS page cache is often more efficient for large sequential scans. Do not exceed 40 percent of total RAM.

**Not adjusting effective_cache_size** — The default is 4 GB. On a 64 GB machine, the planner thinks only 4 GB of page cache is available and may prefer sequential scans over index scans because it underestimates the cache hit rate. Set this to 75 percent of RAM.

**Ignoring connection pool limits** — With max_connections = 500 and work_mem = 64 MB, a query that sorts could consume 32 GB of memory (500 x 64 MB). Use a connection pooler to cap active connections to the number of CPU cores (8-16 for most workloads).

**Omitting VACUUM and ANALYZE** — Dead tuples accumulate, bloating tables and degrading index performance. Autovacuum should be left enabled and tuned to run frequently enough for your write rate.

## Summary

You have now covered the internal architecture of a relational database in depth. The key takeaways:

- A database is composed of specialized subsystems: connection manager, parser, planner, executor, storage engine, lock manager, buffer pool, and WAL.
- B-trees provide O(log n) lookups and efficient range scans. Hash indexes give O(1) equality lookups but cannot handle ranges.
- The WAL is the foundation of durability: every write is logged before the data page is modified.
- MVCC allows concurrent reads and writes without blocking by keeping multiple row versions visible per transaction.
- ACID properties provide strong guarantees: atomicity through transaction logs, consistency through constraints, isolation through MVCC and locking, and durability through the WAL.
- Cache hit ratio above 99 percent is normal for OLTP workloads. Below 95 percent indicates the working set exceeds the buffer pool.
- Configuration tuning is about tradeoffs: conservative settings give reliable but modest performance; aggressive settings give higher throughput at the cost of longer crash recovery and higher memory usage.
