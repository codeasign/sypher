# Understanding Relational Databases

## The Startup Pain

Picture this: you are building a food delivery app. In week one, you store orders in a JSON file on your laptop. It works. In month three, you have 200 customers and the JSON file has grown to 50 MB. You start writing helper functions to search, filter, and sort вҖ” fragile code that breaks when the file format changes. By month six, two people modified the same order at the same time, and one update silently overwrote the other. A customer's dinner never arrived. The business lost that customer forever.

This is the moment every growing system hits the data wall. A flat file cannot handle concurrent writes. It cannot enforce that every order has a valid customer ID. It cannot guarantee that when you read the latest status, you actually get the latest status. The app needs a structured, reliable, concurrent-safe place to store and query data. It needs a relational database.

A relational database is not a magic black box. It is a purpose-built engine with specific data structures, algorithms, and guarantees that solve exactly these problems. By the end of this page you will know what those guarantees are, how the engine delivers them, and why this particular approach has dominated production systems for over four decades.

## The Insight

Think of a relational database as a self-serve library with a strict cataloging system. The library has thousands of books (rows), each on a specific shelf (table), organized by a universal filing code (primary key). When you want a book, you do not wander the aisles вҖ” you ask the librarian (the query engine), who consults the card catalog (the index), walks directly to the correct shelf, and hands you the book in under a second. If you ask for "all books by a particular author published after 2020", the librarian combines two card catalogs (joins two indexes) and returns only matching books. While you read, a security guard (the lock manager) prevents anyone else from reshelving that book until you put it back (transaction isolation).

The genius of the relational model is that it separates **what data you want** from **how to find it**. You write a declarative query (SQL) describing the result set, and the database figures out the most efficient path вҖ” even if the data organization changes underneath. This separation is what makes relational databases adaptable to workloads their creators never imagined.

## Core Vocabulary

**SQL (Structured Query Language)** вҖ” The standard language for defining, querying, and manipulating relational data. SQL is declarative: you say what you want, not how to get it.

**Table** вҖ” A collection of related data organized into rows and columns. Each table represents one entity type: customers, orders, products.

**Row (Tuple/Record)** вҖ” A single entry in a table representing one instance of the entity. One row in the orders table = one order.

**Column (Attribute/Field)** вҖ” A named property of the entity. Columns have a data type (integer, text, timestamp, boolean).

**Primary Key** вҖ” A column or combination of columns that uniquely identifies each row. No two rows can share the same primary key. Every table should have one.

**Foreign Key** вҖ” A column that references the primary key of another table, creating a relationship between rows. This is how you link an order to the customer who placed it.

**Index** вҖ” A separate data structure (usually a B-tree) that speeds up lookups on specific columns. Indexes make reads fast but slow down writes because the index must be updated on every insert.

**ACID** вҖ” The four guarantees a relational database makes about transactions: Atomicity (all or nothing), Consistency (data always satisfies constraints), Isolation (concurrent transactions do not interfere), Durability (committed data survives crashes).

**Transaction** вҖ” A group of SQL operations executed as a single unit. Either all operations succeed (commit) or none of them take effect (rollback).

**Normalization** вҖ” The process of organizing columns and tables to reduce redundancy and avoid anomalies. First, second, and third normal forms are the most common levels.

**Join** вҖ” An operation that combines rows from two or more tables based on a related column. Joins are the mechanism that reconstructs complex objects from normalized storage.

**Schema** вҖ” The formal definition of tables, columns, constraints, indexes, and relationships in a database. A schema is the blueprint for your data.

**Query Planner** вҖ” The component that translates a SQL statement into an execution plan: which indexes to use, which join algorithm to apply, and in which order to access tables.

**B-tree** вҖ” A self-balancing tree data structure used for indexes. B-trees keep data sorted and allow searches, sequential access, inserts, and deletes in logarithmic time.

**MVCC (Multi-Version Concurrency Control)** вҖ” The mechanism that allows multiple readers and writers to access the database simultaneously without blocking each other. Each transaction sees a snapshot of data as of a point in time.

The diagram above shows the main components. The client sends a SQL query over a network connection. The **Parser** checks syntax and builds a parse tree. The **Query Planner** converts that tree into an execution plan by considering available indexes, table sizes, and join strategies. The **Executor** runs the plan, fetching data from the **Storage Engine**, which manages pages on disk. The **Buffer Pool** caches frequently-used pages in memory to avoid slow disk reads. The **Lock Manager** ensures concurrent transactions do not corrupt data. The **Write-Ahead Log (WAL)** records every change so committed data survives a crash.

If the SQL has a syntax error, the parser returns an error immediately вҖ” no query plan is generated.

### Step 3: The Rewriter Applies Rules

Before planning, the rewrite system applies semantic transforms: expanding views into their underlying queries, resolving table aliases to real schema names, and folding constant expressions. After rewriting, the query is semantically equivalent to the original but contains only base-table references.

The planner uses table statistics вҖ” number of rows, number of distinct values per column, most-common values вҖ” to estimate how many rows each operation will return. A sequential scan of a 10-million-row table with a selective filter costs far more than an index scan. The planner picks the plan with the lowest estimated total cost.

### Step 5: The Executor Runs the Plan

The executor takes the chosen plan and iterates through the nodes. Each node is an iterator: it produces rows on demand, one at a time, pulling from child nodes as needed. This "volcano" model minimizes memory usage вҖ” the database does not materialize the entire result set before sending the first row.

### Step 6: The Buffer Pool Handles Pages

When the executor needs a row, the storage engine first checks the buffer pool. If the page containing that row is already in memory (a cache hit), it returns immediately вҖ” an operation that takes under 100 microseconds. If not (a cache miss), the engine reads the page from disk, which takes 1-10 milliseconds.

### Step 7: The Lock Manager Ensures Isolation

If another transaction is modifying data your query needs, the lock manager decides whether to block you (pessimistic locking) or let you see a consistent snapshot from the past (MVCC, optimistic). PostgreSQL uses MVCC: each row carries visibility metadata so readers never block writers and writers never block readers.

## Variants and Flavors

### Row-Oriented vs. Column-Oriented Storage

Row-oriented databases (PostgreSQL, MySQL, Oracle) store all columns of a row together on disk. This is ideal for transactional workloads where you frequently read or write entire rows. Column-oriented databases (Amazon Redshift, ClickHouse) store all values of a column together, which is better for analytical queries that scan millions of rows but only read a few columns.

### Single-Master vs. Multi-Master Replication

In single-master (most common), only one node accepts writes. Read replicas asynchronously replicate from the primary. In multi-master (CockroachDB, MySQL Group Replication), any node can accept writes, but conflict resolution becomes complex.

### Managed Cloud Databases

AWS RDS, Google Cloud SQL, and Azure Database handle backups, replication, patching, and failover for you. Serverless options (Aurora Serverless, Cloud SQL Serverless) scale compute and storage independently and can pause when idle.

## Mental Model Check

**Question 1:** Why does adding an index speed up reads but slow down writes?

An index is a separate data structure that must be kept in sync with the table. On every insert, update, or delete, the database must update both the table and every index on that table. This adds write overhead. But for reads, the index reduces the search from a full table scan (linear in the number of rows) to a B-tree lookup (logarithmic). On a table with one million rows, a full scan reads all one million entries; a B-tree index reads about three to four levels.

**Question 2:** What happens if the database crashes in the middle of a transaction?

The Write-Ahead Log (WAL) contains a record of every change before it is applied to the data files. On recovery, the database replays committed transactions from the WAL (redo) and rolls back uncommitted ones (undo). This guarantees durability: once the database says "COMMIT", the data survives a power failure.

**Question 3:** Can two transactions read the same row at the same time?

Under MVCC, yes. Each transaction sees a snapshot of the database as of its start time. Readers never block writers and writers never block readers. However, if two transactions try to update the same row, one will have to wait or will receive a serialization error depending on the isolation level.

**Question 4:** What is the difference between a clustered and a non-clustered index?

A clustered index determines the physical order of rows on disk. A table can have only one clustered index because the rows cannot be physically sorted two ways. A non-clustered index is a separate structure that contains a copy of the indexed columns plus a pointer to the actual row. Lookups via a non-clustered index require two steps: find the entry in the index, then fetch the row from the table.

**Question 5:** Why do you normalize a database?

Normalization eliminates redundant data and prevents update anomalies. Without normalization, the same customer name might be stored in a hundred order rows. If the customer changes their name, you must update all one hundred rows. Normalization stores the customer name once in a customers table and references it by ID from orders. This also saves storage space and makes queries more predictable.
