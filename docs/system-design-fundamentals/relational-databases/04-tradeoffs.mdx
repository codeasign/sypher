# Relational Databases — Trade-offs & Failure Modes

Every engineering decision involves trade-offs. Choosing a relational database means
accepting certain constraints in exchange for guarantees. This page covers the core
tensions you navigate when designing with a relational store, the failure modes that
emerge in production, and the anti-patterns that turn a well-designed system into a
maintenance burden.

---

## Core Tension — Consistency vs. Performance

The central trade-off in every relational database is between **consistency guarantees**
and **raw throughput**. You cannot maximize both simultaneously.

### What consistency costs

When you INSERT a row and immediate COMMIT, the database must:

1. Write the row to a data page in the shared buffer pool.
2. Write a Write-Ahead Log (WAL) record to disk (fsync).
3. Update every index that covers a changed column.
4. Hold locks on affected rows (and possibly gap locks) until commit.
5. Make the row visible to concurrent transactions based on the isolation level.

Each of these steps adds latency. The strongest isolation levels (Serializable) use
predicate locks or serialization conflict detection, which adds more overhead and
increases the chance of serialization failures that force retries.

### What sacrificing consistency buys

When you relax durability or isolation, you reduce per-write work:

- **Asynchronous commit** skips the WAL fsync on every write. If the server crashes,
  the last N milliseconds of writes are lost. Throughput increases by 2x-5x.
- **Read Committed** instead of Repeatable Read avoids maintaining multiple row
  versions for the full transaction duration. Reads see the latest committed value
  even if another transaction is in progress.
- **Synchronous replication** waits for a replica to confirm the WAL write before
  acknowledging the client. This doubles the write latency (at minimum) but guarantees
  zero data loss on primary failure.

### How you choose

Ask yourself these questions to find the right point on the spectrum:

- **Can you tolerate data loss on crash?** If yes, asynchronous commit gives the
  biggest performance win with no schema changes. If no, you must accept the fsync
  cost.
- **Can you tolerate stale reads during replication lag?** If yes, asynchronous
  replication keeps primary throughput high. If no, synchronous replication halves
  write capacity.
- **Do concurrent transactions conflict frequently?** If yes, Serializable isolation
  will cause many retries and you should redesign to reduce contention instead of
  relying on isolation. If no, Read Committed with row-level locks is sufficient.

In practice, most production systems run at **Read Committed** with **asynchronous
replication** and **synchronous commit** (fsync on). This gives a reasonable balance:
durability for the primary, high throughput, and stale reads that are acceptable for
most use cases. Only financial, inventory, and critical ledger systems move right on
the spectrum.

### How to read the matrix

For each dimension, identify where your workload falls:

1. **Normalization**: If you write more than you read and data integrity is critical
   (ledgers, compliance records), stay at 3NF. If you read 10x more than you write and
   query speed is the bottleneck, introduce denormalization (computed summary columns,
   pre-joined tables).

2. **Indexing strategy**: If you run ad-hoc analytical queries where any column could
   be a filter, consider a columnar store instead. For transactional workloads with
   known query patterns, index only the columns that appear in WHERE clauses with high
   selectivity. Every extra index slows writes by the cost of updating one more B-tree.

3. **Connection mode**: Direct connections are simpler to debug but do not scale. A
   pooler like PgBouncer in transaction mode can handle thousands of application
   connections with a few dozen database connections. Use the pooler from the start;
   retrofitting it requires code changes.

4. **Replication mode**: Synchronous replication is mandatory when your RPO (recovery
   point objective) is zero. For everything else, asynchronous replication gives you
   read scaling without write latency penalties. Semi-sync (wait for at least one
   replica) is a middle ground that most teams should use.

5. **Isolation level**: Start at Read Committed. Only move to Repeatable Read or
   Serializable when you have a proven phantom-read or serialization anomaly. Most
   teams never need stronger isolation.

6. **Sharding**: Do not shard until you have exhausted vertical scaling, read replicas,
   connection pooling, and query optimization. Sharding adds application complexity
   that dwarfs the operational cost of a larger instance.

7. **Backup strategy**: Continuous WAL archiving is the only safe choice for any
   database that cannot lose more than a few minutes of data. Daily snapshots without
   WAL archiving lose up to 24 hours of data on a failure.

## CAP Theorem Positioning

Relational databases are **CP systems** in the CAP theorem: they choose **Consistency**
and **Partition tolerance** over **Availability** during a network partition.

When a network cut separates database nodes, the system must either:

- Reject writes on the minority side (sacrifice availability), or
- Accept writes on both sides and reconcile later (sacrifice consistency).

Relational databases choose option one. During a partition, the minority side stops
accepting writes. The majority side continues to serve reads and writes.

### What this means in practice

When a partition occurs between the primary and a replica, the primary continues
serving writes. The replica stops receiving WAL updates and serves increasingly stale
data. How the system handles this depends on configuration:

- **PostgreSQL**: The `max_standby_streaming_delay` parameter controls how long a
  replica keeps serving queries when it cannot apply WAL. After the delay, the replica
  can either kill conflicting queries or stop serving reads. The default is 30 seconds,
  after which long-running queries on the replica are terminated.
- **MySQL**: Replicas continue serving stale data indefinitely. There is no built-in
  mechanism to reject reads based on lag. You must monitor lag externally.
- **Synchronous replication**: If no replica acknowledges the WAL write within the
  configured timeout, the primary degrades to asynchronous mode (or rejects writes,
  depending on `synchronous_commit` level). This trades durability for availability
  during replica failures.

The practical conclusion: relational databases are reliable when partitions are rare.
If your network fabric is unreliable and you need both high write availability and
strong consistency, relational databases will frustrate you. Consider a NewSQL
database (CockroachDB, Spanner) that uses consensus-based replication instead.

## Failure Modes

Production relational databases fail in predictable patterns. Understanding these
failure modes helps you design defenses before they cause outages.

### Failure Mode 1: Connection Pool Exhaustion

The application opens connections faster than they are closed. The database reaches
`max_connections`. New connections are rejected with "too many clients" errors.
Existing queries that need a new connection for a subquery also fail. The application
starts returning 500 errors for every request that touches the database.

**Root causes**:
- Connection leak in application code — a connection is opened but never closed
  (especially in error-handling paths).
- Traffic spike that exceeds the pool size.
- Background job that opens hundreds of connections without using the pooler.
- ORM that creates a new connection per request instead of using the pool.

**Symptoms**:
- Application logs show `FATAL: remaining connection slots are reserved for
  non-replication superuser connections`.
- Monitoring shows connection count hitting `max_connections` and then flatlining.
- Database CPU drops because the database is idle-waiting, not processing queries.

**Prevention**:
1. Use a connection pooler (PgBouncer, ProxySQL) in transaction mode.
2. Set a hard pool limit at the pooler (not just at the database).
3. Configure connection timeout at the application layer — refuse to wait more than
   5 seconds for a connection.
4. Alert when pool utilization exceeds 80% for more than 2 minutes.
5. Test error-handling paths in CI to catch connection leaks.

**Recovery**:
1. Kill idle connections: `SELECT pg_terminate_backend(pid) FROM pg_stat_activity
   WHERE state = 'idle' AND state_change < now() - interval '5 minutes';`
2. Restart the pooler, which drops all pooled connections cleanly.
3. Scale application instances and increase pool size.

### Failure Mode 2: Replication Lag Cascade

A replica falls behind because a long-running query blocks WAL replay. The replica's
data becomes minutes or hours stale. When the primary fails and the replica is
promoted, all updates from the lag period are lost.

**Root causes**:
- A long-running analytical query on the replica holds a snapshot that conflicts with
  WAL replay. In PostgreSQL, `hot_standby_feedback` prevents the primary from cleaning
  up dead rows that the replica still needs. This causes bloat on the primary too.
- The replica has insufficient IOPS to replay WAL at the rate the primary generates it.
  This is common when the primary uses NVMe and the replica uses network-attached
  storage with lower IOPS.
- Network latency between primary and replica adds baseline lag that grows during
  write bursts.

**Prevention**:
1. Ensure replicas have equal or better IO capacity than the primary. Do not put a
   replica on cheaper hardware unless you accept lag risk.
2. Set `idle_in_transaction_session_timeout` to 30 seconds on replicas to prevent
   long-running idle transactions from blocking WAL replay.
3. Monitor replication lag in bytes and seconds. Alert at P99 of 5 seconds or 1 GB
   of WAL pending.
4. Use `pg_stat_replication` in PostgreSQL or `SHOW SLAVE STATUS` in MySQL to track
   lag programmatically.
5. For critical systems, use synchronous replication to force the primary to wait for
   at least one replica before acknowledging writes.

**Recovery**:
1. Terminate conflicting queries on the replica.
2. If lag is extreme, rebuild the replica from a fresh basebackup rather than waiting
   for WAL replay to catch up.
3. If primary fails while lag is large, accept data loss or failover to a different
   replica with lower lag.

### Failure Mode 3: Table Bloat and Autovacuum Storms

In PostgreSQL, deleted or updated rows leave behind dead tuples. Autovacuum reclaims
this space. On a busy system, autovacuum may not keep up, causing table bloat. A
100 GB table can grow to 200 GB or more due to dead tuples. Indexes bloat too. Query
performance degrades because the database must scan through dead rows in the index
and the heap.

When autovacuum finally catches up, it consumes significant CPU and IO, competing with
regular queries. This is called an autovacuum storm.

**Root causes**:
- High-volume UPDATE or DELETE workload that generates dead tuples faster than
  autovacuum can clean them.
- Default autovacuum settings that are too conservative for modern hardware. The
  default `autovacuum_vacuum_scale_factor` of 0.2 means a 100 GB table waits until
  20 GB of rows change before vacuuming.
- Long-running transactions that prevent vacuum from cleaning dead tuples
  (transaction ID wraparound protection can force a vacuum even if it hurts).

**Prevention**:
1. Tune `autovacuum_vacuum_scale_factor` to 0.01 (vacuum when 1% of rows change,
   not the default 20%). For large tables, use `autovacuum_vacuum_threshold` with
   a fixed row count instead of a percentage.
2. Set `autovacuum_vacuum_cost_limit` to at least 2000 (default is 200). On SSDs,
   the cost-based vacuum delay is unnecessary.
3. For tables with high write churn, partition by time so you can TRUNCATE old
   partitions instead of DELETing rows.
4. Monitor `pg_stat_user_tables.n_dead_tup` and alert when dead tuples exceed 10%
   of live tuples.
5. Use `pgstattuple` extension to measure actual bloat percentage.

### Failure Mode 4: Checkpoint Storm

When the checkpointer flushes all dirty pages from the buffer pool to disk, it can
saturate disk IO for tens of seconds. Queries that need to read pages from disk are
starved because the IO subsystem is overwhelmed by the checkpoint write flood.

**Root causes**:
- `max_wal_size` is too small, forcing frequent checkpoints. Each checkpoint writes
  a burst of dirty pages calculated as `(shared_buffers * dirty_ratio)` to disk.
- The database has large `shared_buffers` (16 GB or more) and a high `dirty_ratio`,
  so each checkpoint flushes multiple gigabytes of data in one burst.
- The storage cannot sustain the write throughput needed for checkpoint bursts. HDDs
  and network-attached storage with burst credits are especially vulnerable.

**Prevention**:
1. Set `max_wal_size` high enough that checkpoints happen at most every 15-30 minutes.
   On a system with 16 GB of WAL traffic per hour, `max_wal_size` should be at least
   8 GB.
2. Increase `checkpoint_completion_target` to 0.9 to spread the checkpoint write over
   90% of the checkpoint interval instead of flushing everything at once.
3. Use SSDs with sufficient write throughput for the checkpoint burst. A single NVMe
   drive handles this better than a RAID of spinning disks.
4. If checkpoints still cause latency spikes, reduce `shared_buffers` to limit the
   amount of dirty data the checkpointer must flush.

### Failure Mode 5: Deadlock Cascade

A deadlock occurs when two transactions hold locks that the other needs. The database
kills one transaction to break the deadlock. If the application retries the killed
transaction immediately, it may immediately re-enter the deadlock with the surviving
transaction. This can cascade across many transactions, causing a spike in deadlock
errors and a sudden drop in throughput.

**Root causes**:
- Transactions that lock tables or rows in different orders. Transaction A locks
  Table Customers then Table Orders. Transaction B locks Orders then Customers.
- A hot row that multiple transactions try to update simultaneously, especially in
  Repeatable Read or Serializable isolation where gap locks come into play.
- Application retry logic that retries on deadlock error without exponential backoff.

**Prevention**:
1. Enforce a consistent lock ordering across all transactions. Document the order
  (e.g., always lock Users before Accounts before Orders) and enforce it in code
  reviews.
2. Keep transactions short. The fewer locks held at once, the lower the probability
  of deadlock.
3. Use Read Committed isolation where possible. Serializable isolation increases
  the lock surface area.
4. Implement retry with exponential backoff. On deadlock error, wait 100ms and retry
  up to 3 times, doubling the delay each time.
5. For high-contention rows (counters, balances), consider using atomic UPDATE
  statements that do not require a separate SELECT, or use optimistic locking with
  a version column.

### Failure Mode 6: Long-Running Query Starvation

A single analytical query consumes all available CPU or IO, starving short
transactional queries. Users experience timeouts on simple lookups while the
analytical query runs for minutes.

**Root causes**:
- An unoptimized query with a full table scan on a large table (missing index, wrong
  join order, or bad selectivity estimate).
- A reporting query run against the primary instead of a read replica.
- Parallel query execution that saturates all worker processes.
- A query that triggers a sort or hash join spilling to disk, saturating IO.

**Prevention**:
1. Set `statement_timeout` (PostgreSQL) or `max_execution_time` (MySQL) to kill
  queries that run longer than a threshold. For example, set 30 seconds for
  transactional queries and 5 minutes for analytical queries on replicas.
2. Route reporting and analytical queries to read replicas, never the primary.
3. Use `pg_stat_activity` monitoring to detect long-running queries and send alerts.
4. In PostgreSQL, set `idle_in_transaction_session_timeout` to prevent connections
   from holding open transactions indefinitely.
5. Use `pg_stat_statements` to identify the worst-performing queries by total time,
   then optimize or cache their results.

## Anti-Patterns Hall of Shame

These patterns appear in production databases everywhere. Each one seemed like a good
idea at the time. None of them aged well.

### Anti-Pattern 1: One Table to Rule Them All

Storing multiple entity types in a single table with nullable columns and a "type"
discriminator column. This destroys the referential integrity that makes relational
databases valuable. The table grows without bounds, indexes become less effective, and
every query filters by a non-selective type column.

```sql
-- Bad: Polymorphic table with nullable columns
CREATE TABLE entities (
    id SERIAL PRIMARY KEY,
    entity_type VARCHAR(20),     -- 'user', 'order', 'product'
    name VARCHAR(255),
    email VARCHAR(255),          -- NULL for products
    price DECIMAL(10,2),         -- NULL for users
    order_date TIMESTAMP,         -- NULL for users and products
    shipping_address TEXT         -- NULL for products
);

-- Every query includes WHERE entity_type = 'user'
-- Indexes on email are useless for non-user rows
-- referential integrity is impossible
```

**Solution**: one table per entity type. Use foreign keys to relate them. Each table
has only the columns it needs, no NULLs, and proper constraints.

### Anti-Pattern 2: Using the Database as a Job Queue

Polling a table every few seconds for rows with `status = 'pending'` and updating them
to `processing` creates contention on the index and rows. The UPDATE must acquire
locks, blocking other transactions. At high frequency, this becomes a bottleneck.

```sql
-- Bad: Polling query every 2 seconds
SELECT * FROM jobs WHERE status = 'pending' ORDER BY created_at LIMIT 10;

-- Then: UPDATE jobs SET status = 'processing', started_at = NOW()
--       WHERE id IN (...)
```

This pattern causes contention because:
- Every poll scans the same index leaf pages for `status = 'pending'`.
- The UPDATE locks the selected rows. If two workers poll simultaneously, they see
  the same pending rows and try to lock them. One worker waits.
- Under load, the table accumulates dead rows from frequent status changes, causing
  bloat.

**Solution**: use a dedicated message queue (Redis Streams, RabbitMQ, Kafka) for job
distribution. Use the database only for persistent state and audit trails. If you
must use the database, use `SELECT ... FOR UPDATE SKIP LOCKED` (PostgreSQL 9.5+) to
atomically claim rows without blocking other workers.

### Anti-Pattern 3: The Catch-All JSON Column

Storing everything as a JSON column because "the schema might change." This bypasses
type checking, constraint enforcement, and indexing. Queries that filter by JSON fields
cannot use standard B-tree indexes — they need GIN indexes on JSON paths, which are
slower and larger.

```sql
-- Bad: Everything in one JSONB column
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    data JSONB NOT NULL
);

-- data contains: user_id, items, total, shipping_address, payment_info
-- Querying: SELECT * FROM orders WHERE data->>'user_id' = '42'
-- No type checking, no constraint, no standard index
```

**Solution**: define real columns for fields you query. Use JSON only for truly
unstructured metadata that you never filter, sort, or join on. When you need to search
by a JSON field, add a generated column with a proper index.

```sql
-- Better: Real columns for queried fields, JSON for the rest
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id),
    total DECIMAL(10,2) NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    metadata JSONB               -- only for data you never filter on
);

CREATE INDEX idx_orders_user_id ON orders(user_id);
```

### Anti-Pattern 4: SELECT * Everywhere

Fetching all columns when you need only one or two. This wastes network bandwidth,
fills the buffer pool with unused columns, and prevents index-only scans — where the
index contains all needed data so the database never touches the table heap.

```sql
-- Bad: Fetching 20 columns when you only need the name
SELECT * FROM users WHERE email = 'alice@example.com';

-- The database must read the heap page even if the email index
-- could satisfy the query with an index-only scan
```

An index-only scan works when the index contains all columns referenced in SELECT,
WHERE, and ORDER BY. If you SELECT *, the index never satisfies the query — the
database always reads the heap, which means more IO and more buffer pool pressure.

**Solution**: list only the columns you need in SELECT queries. This is especially
important for wide tables (50+ columns) and queries that return many rows.

```sql
-- Better: Fetch only the column you need
SELECT name FROM users WHERE email = 'alice@example.com';
```

### Anti-Pattern 5: N+1 Queries in a Loop

Fetching a list of parent entities and then, for each parent, fetching child entities
in a separate query. For 100 parent rows, this generates 101 database round trips.

```python
# Bad: N+1 queries
users = db.execute("SELECT * FROM users")  # 1 query
for user in users:
    orders = db.execute(                   # N queries
        "SELECT * FROM orders WHERE user_id = ?",
        (user['id'],)
    )
    # process orders
```

Each round trip adds network latency (1-10 ms in the same datacenter, 10-100 ms
cross-region). For 100 users, this adds 1-10 seconds of latency just for the network
overhead, not counting query execution time.

**Solution**: use a JOIN or a batch query with IN clause.

```python
# Better: Single JOIN query
rows = db.execute("""
    SELECT u.*, o.id as order_id, o.total, o.status
    FROM users u
    LEFT JOIN orders o ON o.user_id = u.id
    WHERE u.id IN (SELECT id FROM users WHERE ...)
""")
```

Most ORMs have features to eager-load related data. Learn your ORM's eager loading
API before writing production code.

### Anti-Pattern 6: Indexing Everything

Creating indexes on every column "just in case." Each index adds write overhead — the
index B-tree must be updated on every INSERT, UPDATE, and DELETE. On a table with 10
indexes, a single INSERT updates 10 B-trees plus the heap. Index maintenance can
consume more resources than the actual data modification.

```sql
-- Bad: Indexing every column
CREATE INDEX idx_users_name ON users(name);
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_status ON users(status);
CREATE INDEX idx_users_created ON users(created_at);
CREATE INDEX idx_users_updated ON users(updated_at);
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_department ON users(department);
CREATE INDEX idx_users_last_login ON users(last_login);
-- ... and so on for every column
```

Each index:
- Adds write amplification: every INSERT writes to all indexes.
- Consumes disk space. An index on a UUID column can be larger than the column itself.
- Slows down autovacuum because more index pages need cleaning.
- Must be maintained during VACUUM, increasing bloat.

**Solution**: create indexes only for query patterns you actually use. Remove unused
indexes. In PostgreSQL, check `pg_stat_user_indexes`:

```sql
-- Find indexes with zero scans (candidates for removal)
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
AND indexrelid NOT IN (
    SELECT indexrelid FROM pg_constraint WHERE contype = 'p'
)
ORDER BY tablename, indexname;
```

A zero-index scan count means the index has never been used for an index scan since
the stats were reset. It is pure overhead.

### Anti-Pattern 7: The God Transaction

One transaction that does everything — validates input, checks inventory, charges
the payment processor, updates inventory, sends a confirmation email, writes audit
logs, and refreshes a materialized view. This transaction holds locks for seconds
or minutes, blocking every other operation on the affected rows.

**Root problems**:
- Long-held locks increase deadlock probability.
- The transaction includes external calls (payment processor, email) that can hang,
  leaving the transaction open indefinitely.
- Rollback is expensive because the database must undo all changes.

**Solution**: separate operations that do not need atomicity into independent
transactions. Use an orchestration pattern: charge the card in one transaction, create
the order in another, send the email asynchronously. If the email fails, the order is
already committed and you can retry the email.

## When NOT to Use This

A relational database is the wrong choice when your problem fits one of the following
profiles. Picking the right tool for the job is a sign of engineering maturity, not
weakness.

### Schema-less or Evolving Data

If your data has no fixed schema — documents, logs, event streams, user-generated
content with varying fields — a relational database forces you to either use nullable
columns for every possible field or shove everything into JSON columns. Neither is
satisfying.

**Better choice**: a document store like MongoDB or a flexible schema store like
DynamoDB. These let you store differently-shaped documents in the same collection
without migration overhead.

### Extreme Write Throughput

If you need single-digit-millisecond writes at 100,000 or more per second, a single
relational primary cannot sustain this. You would need to shard aggressively, and
application-level sharding on a relational database is painful.

**Better choice**: a purpose-built database for your write pattern. Cassandra for
write-heavy workloads with its log-structured merge tree storage. Redis for
in-memory write throughput in the millions per second. TimescaleDB for time-series
ingestion at high velocity.

### Graph Traversal

If your data is a graph — social networks, recommendation engines, network maps,
hierarchical org charts — a relational database uses recursive CTEs for traversal,
which are slow and cumbersome for deep relationships.

**Better choice**: a graph database like Neo4j or ArangoDB. Graph databases store
relationships as first-class citizens. Traversing a friendship or recommendation
graph is orders of magnitude faster than the equivalent recursive SQL query.

### Time-Series Data

Metrics, logs, and sensor readings append at high velocity and are queried by time
range. A relational database stores each row as a separate heap entry with full MVCC
overhead. Old rows must be purged with DELETE, which generates dead tuples and
requires VACUUM.

**Better choice**: a time-series database. TimescaleDB extends PostgreSQL with
automatic partitioning (hypertables), continuous aggregates, and retention policies.
InfluxDB and ClickHouse offer columnar compression that reduces storage by 5-10x
compared to a row-oriented relational store.

### Full-Text Search at Scale

If full-text search is your primary feature, a relational database will disappoint at
scale. PostgreSQL has good full-text search for basic use cases, but it lacks
relevance scoring sophistication, faceted search, typo tolerance, and query-time
boosting.

**Better choice**: Elasticsearch or Meilisearch. These are built on inverted indexes
designed for search, not B-trees designed for range scans. They handle relevance
ranking, partial matching, synonyms, and highlight fragments natively.

### Ephemeral or Cache Data

If data has a short TTL and you never query it after expiration (sessions, rate-limit
counters, temporary tokens), a relational database adds durability overhead for no
benefit.

**Better choice**: Redis or Memcached. These store data in memory with optional
persistence. TTL-based expiration is a first-class feature. Throughput is measured in
hundreds of thousands of operations per second per node.

### Blob Storage

If you are storing binary files (images, videos, PDFs) in database columns or as
base64-encoded strings, you are paying for full ACID guarantees on data that has no
relational structure.

**Better choice**: object storage (S3, GCS, Azure Blob) with the file path stored
in the database as a VARCHAR column. Object storage costs pennies per GB per month.
Database storage costs dollars per GB per month. The database column is just a
pointer.

## Summary

Relational databases are the most mature and trusted data storage technology in
existence. Their strengths are well-documented: ACID transactions, rich query
capabilities, referential integrity, and decades of operational tooling.

The trade-offs you make when choosing a relational database are:

- **Consistency costs throughput**. The stronger your isolation and durability
  guarantees, the fewer writes per second you can sustain.
- **Normalization saves storage but costs queries**. Joins are expensive. Denormalize
  strategically for hot read paths.
- **Indexes speed reads but slow writes**. Every index is a write-time tax you pay
  for read-time speed. Do not create indexes you do not need.
- **Single-primary is a write bottleneck**. If you need more write throughput than
  one server can provide, you must shard, which adds application complexity.
- **Not every data model is relational**. Graph, time-series, document, and search
  workloads benefit from purpose-built databases.

The key insight: use a relational database for what it is good at — structured data
with clear relationships and ACID requirements. For everything else, use the right
tool and reference the data by key in your relational store. Your future self will
thank you.
