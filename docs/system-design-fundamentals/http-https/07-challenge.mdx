## Tier 1 — Foundation

These challenges take approximately 5 minutes each. They test your understanding of core HTTP/HTTPS concepts without any design or system context.

### Challenge 1.1: Trace the HTTPS Request

**Prompt**: You type https://api.example.com/v2/users/42 into a browser. Walk through every step from keystroke to the first byte of the response arriving at the browser. Draw a diagram showing each layer and protocol involved.

**Time**: 5 minutes

**Evaluation criteria**:
- DNS resolution (cache check at browser/OS/router/resolver levels, recursive resolution via root/TLD/authoritative nameservers)
- TCP three-way handshake (SYN with ISN, SYN-ACK with server ISN, ACK — each with sequence numbers showing the increment)
- TLS 1.3 handshake (ClientHello with key_share, ServerHello + Certificate + CertificateVerify + Finished, client verifies cert chain, sends Finished)
- HTTP request construction (method, URL, headers — Host, Accept, User-Agent, Connection)
- Network transit through routers and load balancers
- Server-side processing (request parsing, TLS decryption, route dispatch, handler execution, response construction)
- Response transit back through TLS encryption, TCP segmentation
- Time estimate for the full round trip

**Model answer**:

The URL is parsed: scheme=https, host=api.example.com, path=/v2/users/42.

The browser checks its DNS cache. Miss (TTL expired or never visited). It queries the OS cache. Miss. It queries the configured DNS resolver (8.8.8.8 or ISP-provided). The resolver queries a root nameserver for the .com TLD, then the .com TLD nameserver for example.com, then the authoritative nameserver for api.example.com. The authoritative server returns 203.0.113.42. Total: approximately 35ms.

The browser opens a TCP connection to 203.0.113.42:443. Three-way handshake: SYN (seq=1000) → SYN-ACK (seq=5000, ack=1001) → ACK (seq=1001, ack=5001). RTT: approximately 40ms.

TLS 1.3 handshake: ClientHello (TLS 1.3 version, supported cipher suites including TLS_AES_128_GCM_SHA256, ECDHE key share for key agreement). Server responds with ServerHello (chosen TLS 1.3, cipher TLS_AES_128_GCM_SHA256, its ECDHE key share), the server's certificate chain (leaf → intermediate → root), CertificateVerify (the server signs the handshake transcript with its private key to prove it owns the certificate), and Finished (encrypted — contains MAC of handshake). The client verifies each certificate in the chain against its trust store, checks the Common Name/Subject Alternative Name matches api.example.com, checks the notBefore/notAfter dates, and optionally checks revocation via OCSP. Both sides derive the same symmetric key using ECDHE. Client sends its Finished message. RTT: approximately 40ms.

HTTP request: GET /v2/users/42 HTTP/1.1 with Host: api.example.com, Accept: application/json, User-Agent: Browser/1.0, Connection: keep-alive. The request is encrypted with the negotiated TLS key and sent over the TCP connection.

The request travels across the internet through approximately 15 router hops. Each router reads the destination IP and forwards the packet. The total network transit time is approximately 40ms (same as RTT since this is a small request that fits in one round).

The server's network interface receives the packet. The kernel TCP stack reassembles the segments and passes them to the TLS layer (OpenSSL or BoringSSL). TLS decrypts the records and passes plaintext bytes to the HTTP parser (Nginx, Envoy, or application server). The parser reads the request line and headers, producing a parsed request object. The router matches the URL path /v2/users/42 to the user lookup handler. The handler authenticates the request (verifies the session token), queries the database (SELECT * FROM users WHERE id = 42), serializes the result to JSON, and constructs the HTTP response with status 200, headers (Content-Type: application/json, Content-Length: 256), and the JSON body. The total server processing time is approximately 15ms.

The response is encrypted by TLS, segmented by TCP, and sent back across the network. Transit time: approximately 40ms.

Total time: approximately 35ms (DNS) + 40ms (TCP) + 40ms (TLS) + 40ms (request transit) + 15ms (server processing) + 40ms (response transit) = 210ms for first request. With keep-alive, subsequent requests skip TCP and TLS, taking approximately 35ms (request transit + server + response transit) = 90ms.

### Challenge 1.2: Distinguish Status Codes

**Prompt**: For each scenario below, state the correct HTTP status code and explain why:

1. A client sends a POST to create a user, but the request body is missing the required email field.
2. A client sends a request with an expired OAuth token.
3. A client sends a GET request for a resource that used to exist at this URL but now lives at a different URL.
4. A client sends a request and the server cannot process it because an upstream database is unreachable.
5. A client sends a GET request and the server has the resource, but wants to tell the client to use its cached copy.

**Time**: 5 minutes

**Model answer**:

1. **400 Bad Request**. The request is malformed from the client's perspective — required fields are missing. The response body should describe which field is missing and the expected format. The client should not retry the same request without modification. Never use 500 for this — the server did not fail, the client sent a bad request. Distinguish from 422 Unprocessable Entity (used when the request syntax is correct but the semantics are wrong, e.g., an email address that does not exist).

2. **401 Unauthorized**. The client did not provide valid authentication credentials. The response must include a WWW-Authenticate header describing how to obtain a valid token (e.g., WWW-Authenticate: Bearer realm="api.example.com"). The client should obtain a new token and retry. Note: 403 Forbidden means the client IS authenticated (the token is valid) but lacks permission for this specific resource — a different scenario entirely.

3. **301 Moved Permanently** (or 308 if the client must keep the same HTTP method). The response must include a Location header with the new URL. The client should update its bookmarks and internal references. For temporary moves (maintenance window, A/B testing, content relocation), use 302 Found or 307 Temporary Redirect instead.

4. **503 Service Unavailable**. The server is temporarily unable to handle the request due to an upstream dependency failure. The response should include a Retry-After header (e.g., Retry-After: 30). The client should retry after the specified duration. Do not use 500 (generic server error) when the specific cause is known to be an upstream dependency.

5. **304 Not Modified**. The client sent an If-None-Match header (with the ETag of its cached copy) or If-Modified-Since header (with the timestamp of its cached copy). The server checks whether the resource has changed. If unchanged, it returns 304 with no body and no Content-Type. The client uses its cached copy. This is how browser caching and CDN revalidation work in practice.

### Challenge 1.3: Explain HTTP Caching

**Prompt**: Explain the difference between these three Cache-Control header values. In what scenario would you use each? Draw a timeline of client, cache, and origin interactions for each:

- Cache-Control: public, max-age=3600
- Cache-Control: no-cache
- Cache-Control: no-store

**Time**: 5 minutes

**Model answer**:

**Cache-Control: public, max-age=3600**: The response can be cached by any cache (browser, CDN, proxy, intermediate). The cache can serve the response without revalidating for 3,600 seconds (1 hour). Use for static assets (CSS, JS, images with versioned URLs like style.v2.css) and public API responses that change infrequently (country lists, product categories). Risk: clients see data up to 1 hour stale. For content-addressed URLs (hash in filename), use max-age=31536000 (1 year) since the URL changes when the content changes.

**Cache-Control: no-cache**: The response CAN be cached, but the cache MUST revalidate with the origin before serving. The cache sends If-None-Match with the stored ETag. If the origin returns 304 Not Modified, the cache serves the stored copy (saving bandwidth — the body is not re-downloaded). If the origin returns 200, the cache updates and serves the new copy. Use for dynamic API responses where freshness matters but the origin can handle validation requests quickly (stock prices, order status, user profiles, news articles). Cost: every cache hit requires a round trip to the origin for validation, typically 20-80ms of additional latency, but the body (usually 1-50KB) is only transferred when it changes.

**Cache-Control: no-store**: The response must NOT be cached by any cache at any layer. Every request goes to the origin server. Use for sensitive data (auth tokens, payment information, PII, session IDs) and responses that should never be served from cache under any circumstances. Cost: every request hits the origin server, increasing server load and latency.

Timeline for each scenario:

**max-age=3600**: Client → Cache (miss) → Origin (200, max-age=3600). Cache stores. 1 minute later: Client → Cache (fresh, no origin call). Client ← Cache (200). 61 minutes later: Client → Cache (stale, TTL expired) → Origin (If-None-Match). Origin returns 304. Client ← Cache (304, uses cached body, no body re-download). Total: 1 origin call per hour per cache.

**no-cache**: Client → Cache (miss) → Origin (200, no-cache, ETag). Cache stores. Next request: Client → Cache → Origin (If-None-Match). Origin (304, no body). Client ← Cache (304). Every request triggers a revalidation round trip, but the body is not re-downloaded until the content changes.

**no-store**: Client → Cache → Origin (200, no-store). Cache must NOT store. Next request: Client → Cache → Origin (200, no-store). Every request hits the origin. Cache never has a copy to serve.

## Tier 2 — Design

These challenges take approximately 15 minutes each. They require designing a system or protocol around HTTP/HTTPS constraints.

### Challenge 2.1: Design a Global CDN

**Prompt**: Design a CDN (Content Delivery Network) that serves static assets (images, CSS, JS) over HTTPS to users in 200 countries. The origin server is in us-east-1. Users in Australia experience 200ms latency for uncached assets.

- 10 million unique assets, totaling 50TB
- 100,000 requests per second at peak
- 99.5% cache hit ratio required for static assets
- Zero-downtime TLS certificate rotation required

**Time**: 15 minutes

**Rubric**:

| Criterion | Strong (3) | Adequate (2) | Weak (1) |
|-----------|------------|--------------|----------|
| TLS architecture | Edge termination + Let's Encrypt automated renewal + certificate monitoring with 30-day alerts | Simple TLS termination at edge | No TLS mention |
| Cache hierarchy | Multi-tier: browser → edge PoP → regional → origin with stale-while-revalidate for origin outage protection | Single-tier CDN cache with origin fallback | No cache hierarchy, no stale-while-revalidate |
| Protocol selection | HTTP/3 for end users, HTTP/2 to origin, HTTP/1.1 fallback for legacy clients | HTTP/2 only | No protocol version discussion |
| Certificate management | ACME DNS-01 automated at each PoP, 90-day certs, 30/14/7-day pre-expiry alerts | Single certificate at origin, manual renewal | No certificate management plan |
| Failure handling | Stale content served during origin outage, health-based failover, stale-if-error and stale-while-revalidate headers | Basic retry on origin failure | No failure handling |

**Model answer**:

**TLS architecture**: TLS is terminated at each edge Point of Presence (PoP). Each PoP uses Let's Encrypt with DNS-01 challenge for certificate issuance. A certificate is issued per customer domain per PoP. For a CDN serving 33,000 domains across 330 PoPs, that is 10.9 million certificate validations per renewal cycle. Automate with cert-manager or a custom ACME client integrated with the CDN control plane. Use wildcard certificates (*.cdn.example.com) where possible to reduce the number of certificates needed.

**Cache hierarchy**: Three-tier cache. Tier 1: Browser cache (Cache-Control: public, max-age=31536000, immutable for content-addressed assets where the filename includes a content hash). Tier 2: Edge PoP cache (RAM-based, 500GB per PoP, LRU eviction, 99% hit rate). Tier 3: Regional hub cache (NVMe SSD-based, 20TB per region, 99.5% aggregate hit rate). Origin fallback: us-east-1. Use stale-while-revalidate=86400 so the edge serves stale content for 24 hours while fetching fresh content in the background. This protects against origin outages entirely — users never see an error because the edge always has something to serve.

**Protocol selection**: Clients connect to edge PoPs over HTTP/3 (QUIC) for best lossy-network performance on cellular and satellite connections. HTTP/2 between edge PoPs and regional hubs (internal network, low loss — TCP is fine). HTTP/1.1 or HTTP/2 between regional hubs and origin (depending on origin server capability). Fallback chain: HTTP/3 → HTTP/2 → HTTP/1.1 based on client capability negotiation. The CDN indicates supported protocols via the Alt-Svc header.

**Certificate management**: Let's Encrypt with DNS-01 wildcard certificates. 90-day validity. Automated renewal at 60, 30, and 7 days before expiry. Monitoring alerts if renewal fails for any reason (DNS zone locked, ACME rate limit exceeded, certificate validation failure). Certificates distributed to all PoPs via the control plane config push within 30 seconds of issuance. Use OCSP stapling to improve TLS handshake performance — the server includes a signed OCSP response in the TLS handshake, so the client does not need to contact the CA for revocation checking.

**Failure handling**: When the origin is unreachable, edge PoPs serve stale content from cache for up to 7 days (stale-if-error=604800). Health checks run every 5 seconds from each PoP to the origin. If 3 consecutive health checks fail from a PoP, that PoP marks the origin as degraded and exclusively serves stale content. Users see a warning header (X-CDN-Info: stale-due-to-origin-error) for transparency. When the origin recovers, PoPs revalidate their cached content against fresh origin data.

**Latency improvement**: Before CDN: 200ms RTT to us-east-1. After CDN with a PoP in Sydney: 10ms RTT to CDN edge. If the asset is cached at the edge: 10ms total. If the edge misses and the regional hub in Singapore has it: 40ms total. If the regional hub misses and we must go to origin: 210ms total (200ms origin + 10ms edge). The 99.5% cache hit rate means 99.5% of requests are under 10-40ms.

### Challenge 2.2: Design an HTTP Rate Limiter

**Prompt**: Design a rate limiter for a public HTTP API that serves 10,000 requests per second. Must support:

- Per-API-key rate limits (100 req/s standard, 1,000 req/s for premium)
- Per-IP rate limits (50 req/s per IP, regardless of API key)
- Burst allowance (2x the rate for up to 5 seconds)
- Distributed across 10 API server instances (rate limit state must be shared)
- Return appropriate HTTP headers so clients can self-regulate

**Time**: 15 minutes

**Model answer**:

**Algorithm**: Token bucket with Redis as the distributed state store. Each (api_key, ip) pair has a token bucket key in Redis. The bucket has a capacity equal to rate * burst_multiplier and refills at rate tokens per second. The token bucket algorithm allows bursts (up to capacity) while enforcing the average rate over time, making it ideal for API rate limiting.

**Data model in Redis**:

```
Key: ratelimit:apikey:<api_key>
Value: {tokens: 100, last_refill: 1719829200.000}
TTL: 60 seconds (auto-expire for inactive keys, preventing memory waste)

Key: ratelimit:ip:<ip_address>
Value: {tokens: 50, last_refill: 1719829200.000}
TTL: 60 seconds
```

**Lua script for atomic operations** (avoids race conditions between read and write):

```lua
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local cost = tonumber(ARGV[4])

local bucket = redis.call('GET', key)
if not bucket then
    -- First request: initialize full bucket
    redis.call('SET', key, capacity .. ':' .. now)
    redis.call('EXPIRE', key, 60)
    return {1, capacity - cost, capacity}
end

local parts = {}
for token in string.gmatch(bucket, '([^:]+)') do
    table.insert(parts, token)
end
local tokens = tonumber(parts[1])
local last_refill = tonumber(parts[2])

-- Refill tokens based on elapsed time
local elapsed = now - last_refill
tokens = math.min(capacity, tokens + elapsed * rate)
tokens = tokens - cost

if tokens < 0 then
    return {0, 0, capacity}  -- Rate limited
end

redis.call('SET', key, tokens .. ':' .. now)
redis.call('EXPIRE', key, 60)
return {1, tokens, capacity}
```

**HTTP headers returned**:

```
RateLimit-Limit: 100           # Max requests per window
RateLimit-Remaining: 87        # Tokens remaining
RateLimit-Reset: 1719829205    # When the bucket will be full (UNIX timestamp)

# When rate limited (429):
Retry-After: 2                  # Seconds to wait (integer, not float)
RateLimit-Limit: 100
RateLimit-Remaining: 0
X-RateLimit-Type: api_key       # Which limit was exceeded (useful for debugging)
```

**Architecture**: Each of the 10 API servers runs a local rate limiter middleware. The middleware calls Redis with the Lua script using EVALSHA (cached script, avoids re-sending the Lua code on every request). Redis is deployed as a 3-node cluster with 2 replicas for high availability. If Redis is unreachable, each server falls back to a local in-memory rate limiter with relaxed limits (allows 2x the configured rate). This degrades gracefully rather than failing closed — a rate limiter that refuses all requests during a Redis outage is a denial-of-service vulnerability.

### Challenge 2.3: Design a Retry and Circuit Breaker Strategy

**Prompt**: You have three downstream HTTP services. Service A is a critical payment processor (POST requests, idempotency key required). Service B is a notification service (POST requests, no idempotency). Service C is a user profile service (GET requests, rarely changes). Design a retry and circuit breaker strategy for each. Include the exact backoff values, retry counts, circuit breaker thresholds, and how you handle partial failures.

**Time**: 15 minutes

**Model answer**:

**Service A — Payment Processor (POST, idempotent with key)**:
- Retries: 3 maximum. First retry at 200ms, second at 800ms, third at 3.2s. Jitter: +/- 25%. Total retry window: approximately 4.2 seconds.
- Each retry includes the same Idempotency-Key header. The server deduplicates at the storage layer.
- Circuit breaker: After 5 consecutive failures (5xx or timeout), open circuit for 30 seconds. While open, all requests fail fast with a 503 to the upstream caller. After 30s, allow 1 probe request. If probe succeeds, close circuit. If probe fails, keep open for another 60s (exponential backoff on circuit cooldown).
- Timeout: 5 seconds per attempt. A payment should not take longer than 5s — longer suggests a problem with the processor.

**Service B — Notification Service (POST, not idempotent)**:
- Retries: 1 maximum. Single retry at 500ms with jitter. A second retry could cause duplicate notifications.
- Since there is no idempotency key, the guarantee is "at most once" delivery: if the first attempt gets a 5xx, retry once. If that also fails, fail the entire request and log the notification to a dead-letter queue for manual review by the operations team.
- Circuit breaker: After 3 consecutive failures, open circuit for 10 seconds. The shorter cooldown reflects that notifications are usually fast but occasional noise spikes cause transient failures.
- Timeout: 2 seconds per attempt. Notifications are cheap operations.

**Service C — User Profile (GET, idempotent, rarely changes)**:
- Retries: 5 maximum. Backoff: 100ms, 400ms, 1.6s, 6.4s, 25.6s. Jitter: +/- 50%. Total retry window: approximately 34 seconds.
- Aggressive retry is safe because GET is idempotent and the data rarely changes. The cost of an extra retry is low (small response).
- Circuit breaker: After 10 consecutive failures, open circuit for 60 seconds. High threshold because transient network issues are common and this service should be highly available. While circuit is open, return cached profile data (stale for up to 5 minutes) instead of failing. This ensures users always see a profile, even if slightly outdated.
- Timeout: 3 seconds per attempt.
- Cache layer: In-memory cache with TTL of 300 seconds. Checked before any network call. If the cache has a recent entry (under 60s old) and the circuit is open, serve from cache without attempting the network call.

**Partial failure handling**: When a single request to one downstream service fails but others succeed, return a 200 with a warning header: X-Partial-Failure: payment-service=503, retry-id=key-abc. The client can check this header and decide whether to retry the entire operation or handle the partial failure. Do not return a 5xx for the entire request just because one of three downstream calls failed.

## Tier 3 — Expert

These challenges take approximately 30 minutes each. They are open-ended with no single correct answer. The goal is discussion, exploration, and depth.

### Challenge 3.1: Migrate a Legacy HTTP/1.1 API to HTTP/3

**Prompt**: Your company has a monolith API serving 500 million requests per day over HTTP/1.1 with keep-alive. The infrastructure is 200 Nginx servers running on bare metal in three data centers. 70% of traffic comes from mobile apps on cellular networks. Over the past year, the operations team has reported increasing latency for mobile users — p99 has grown from 400ms to 1.2s.

Your VP of Engineering asks you to plan the migration of this API to HTTP/3 over QUIC. Consider: client compatibility (many users have phones as old as 2018), infrastructure changes needed, monitoring and debugging tooling gaps, rollback strategy, cost implications, and the migration timeline. There is no single right answer — the quality of your analysis and trade-off reasoning matters more than your final design.

**Discussion points**:

- **Client compatibility**: HTTP/3 requires QUIC support in the client library. For mobile apps using OkHttp (Android), Cronet (cross-platform), or NSURLSession (iOS), HTTP/3 support was added in 2019-2020, but many users on older OS versions lack it. Strategy: negotiate to HTTP/3 with a graceful fallback chain (HTTP/3 → HTTP/2 → HTTP/1.1). Phase the rollout by client OS version: start with 5% of clients on the latest OS, ramp to 100% over 3 months as old clients upgrade organically.

- **Infrastructure changes**: Nginx added QUIC support in version 1.25 (2023). All 200 bare-metal servers must be upgraded to Nginx 1.25+ and OpenSSL 3.x. QUIC requires UDP port 443 in addition to TCP port 443. Firewall rules must be updated for UDP traffic. Load balancers (hardware F5) must support UDP load balancing or be replaced with software LBs.

- **Monitoring and debugging gaps**: TCP connections are transparent — you can capture with tcpdump and inspect with Wireshark. QUIC encrypts transport-layer metadata, so existing packet capture tools cannot inspect individual streams. Build QUIC-specific monitoring: quic loss rate, connection migration events, 0-RTT success rate, UDP packet drop rate. Use qlog and qvis for QUIC-specific debugging.

- **Rollback strategy**: Serve HTTP/3 (UDP:443) and HTTP/2 (TCP:443) simultaneously. Clients prefer HTTP/3 over QUIC but fall back to HTTP/2 if QUIC fails. Rollback is instantaneous — disable HTTP/3 at the Nginx layer and clients switch to HTTP/2 on their next request.

- **Cost implications**: QUIC requires more CPU per connection (userspace packet processing). Estimate 15-20% more CPU on Nginx servers. However, CDN egress cost may decrease because QUIC's better loss recovery reduces retransmission. Net infrastructure cost may increase 5-10% initially.

### Challenge 3.2: Design an HTTP-Based Event Delivery System

**Prompt**: Design a system that delivers real-time events from a server to 10 million connected clients over HTTP. Each client maintains a persistent connection to receive events. Events include chat messages, stock price updates, and system alerts. Delivery latency must be under 500ms p99 for chat messages and under 5 seconds p99 for alerts.

Clients connect from browsers (JavaScript), mobile apps (iOS, Android), and IoT devices. Some clients are behind corporate firewalls that block WebSocket connections and long-lived TCP connections. The system must handle 1 million events per second at peak with no message loss.

Compare at least three HTTP-based approaches for delivering events to clients.

**Discussion points**:

**Approach 1 — Long Polling**:
The client sends a GET request. The server holds the request open (up to 30 seconds) until an event is available, then responds with the event. The client immediately sends a new request upon receiving the response.
- Pros: Works through any firewall. Simple to implement. Easy to debug. No special protocol support needed.
- Cons: Each request-response cycle re-establishes headers and cookies, incurring overhead. If the server has no events for a client, it wastes server resources holding idle connections. Overhead: each empty response is approximately 800 bytes (headers). With 10 million clients polling every 30 seconds: 333,333 req/s of overhead traffic. Wastes bandwidth when events are infrequent.

**Approach 2 — Server-Sent Events (SSE)**:
The client opens a single HTTP connection. The server sends events as text/event-stream data frames over the same connection. The connection stays open indefinitely.
- Pros: Native browser support (EventSource API). Over HTTP/2, SSE streams are multiplexed with other requests. Simple text-based protocol. Automatic reconnection built into the EventSource API.
- Cons: Browser limitation of 6 SSE connections per origin under HTTP/1.1. No binary messaging (text-only). No client-to-server messaging over the same connection. Some corporate proxies buffer responses, causing SSE to fail.

**Approach 3 — WebSocket over HTTPS**:
The client sends an HTTP Upgrade request (101 Switching Protocols). The connection is upgraded from HTTP to WebSocket protocol. Full-duplex communication.
- Pros: Lowest overhead (2 bytes per message after initial upgrade). Binary and text messaging. Full-duplex. Native browser support. Widely supported.
- Cons: Some corporate proxies block WebSocket Upgrade. Requires HTTP/1.1 for the upgrade path. More complex reconnection logic than SSE. Must implement heartbeat/ping to detect dead connections.

**Recommended**: Hybrid strategy. WebSocket as primary, SSE as fallback for restrictive firewalls, long polling as last resort for extremely restricted environments.
